synchronized 与 volatile 基础

并发的一切麻烦都源于三个词：可见性、原子性、有序性——这两个关键字就是 JDK 给你的第一副解药。

一、先搞清楚：并发到底难在哪

单线程程序里，代码"从上往下、说一是一"，你写什么就执行什么。可一旦多个线程同时跑，三个反直觉的问题就冒出来了。它们是理解 synchronized 与 volatile 的唯一前提——这两个关键字存在的意义，就是分别修补这三个漏洞。

问题它是什么生活类比
可见性A 线程改了变量，B 线程却看不到（读到旧值）你在共享文档改了字，同事的缓存页面还是老版本
原子性一个"看起来一步"的操作，其实由多步组成，中途会被打断转账"扣款+入账"被打断，钱扣了却没到账
有序性编译器/CPU 为了性能会重排指令，顺序和你写的不一样快递员为了顺路，不按你下单的顺序送货
为什么会有"可见性"这种怪事？根源在硬件。现代 CPU 每个核心都有自己的高速缓存（L1/L2），比主内存快上百倍。线程读写变量时，为了性能会先在缓存里操作，不会每次都同步回主内存。于是 A 核心改了值还留在自己缓存里，B 核心从主内存或自己的旧缓存读，自然看到旧值。Java 把这套硬件现实抽象成了 JMM（Java 内存模型）。

二、JMM：主内存与工作内存

JMM 规定：所有共享变量都存在主内存；每个线程有自己的工作内存（对应 CPU 缓存+寄存器），线程只能操作自己工作内存里的副本，改完再择机写回主内存。可见性问题就发生在"择机"二字上——JVM 不保证你什么时候写回、别人什么时候重读。

┌─────────────── 主内存 (Main Memory) ───────────────┐ │ int flag = 0; │ └──────▲───────────────────────────────▲─────────────┘ │ read / write │ read / write ┌──────┴───────┐ ┌──────┴───────┐ │ 线程 A │ │ 线程 B │ │ 工作内存 │ │ 工作内存 │ │ flag = 1 ✍ │ │ flag = 0 ？ │ <- 看不到 A 的修改 └──────────────┘ └──────────────┘ volatile: 强制每次读都从主内存拉、每次写都刷回主内存 (可见性) synchronized: 进入锁时清工作内存、退出锁时刷回主内存 (可见性 + 互斥)

下面这段经典代码，几乎是可见性问题的"教科书现场"：主线程改了 running，工作线程却可能永远停不下来。

StopThread.java没有 volatile → 可能死循环
class Worker extends Thread {
    private boolean running = true;   // ← 没加 volatile

    public void run() {
        while (running) { /* 干活 */ }
        // JIT 可能把 running 缓存进寄存器，甚至优化成 while(true)
    }
    public void stop_() { running = false; }  // 主线程调用，工作线程却看不到
}
初学者最常掉的坑：觉得"我明明把 running 设成 false 了，循环怎么不停？"——这不是玄学，是 JMM + JIT 的真实行为。JIT 编译器看到循环体内 running 没被修改，会把它提升到寄存器缓存起来（叫做循环不变量外提），于是工作线程读的永远是那个缓存的 true。给 running 加上 volatile，问题立刻消失。

三、volatile：轻量级的"可见性 + 有序性"

volatile 做两件事，而且只做这两件：

  1. 保证可见性：写 volatile 变量会立即刷回主内存；读 volatile 变量会强制从主内存重新加载。
  2. 禁止指令重排序：通过插入内存屏障（Memory Barrier），保证 volatile 写之前的操作、读之后的操作不会被重排到屏障另一侧。

它不做的那件事，恰恰是最多人误解的：volatile 不保证原子性。看这个反例——

✗ Bad · 以为 volatile 能保护 count++
volatile int count = 0;

void incr() {
    count++;   // 三步！读-改-写
}
// 1000 线程各 +1000 次
// 结果 < 100 万，丢更新 💥
// volatile 保证不了"读改写"不被打断
✓ Good · 用原子类或加锁
AtomicInteger count =
        new AtomicInteger();

void incr() {
    count.incrementAndGet();  // CAS 原子
}
// 或: synchronized void incr(){count++;}
// 结果稳定 = 100 万 ✅

count++ 在字节码里是 getfield → iadd → putfield 三条指令。volatile 能保证每一步读到/写回的是最新值，但拦不住两个线程在"读"和"写"之间交错，导致其中一次自增被覆盖。

什么时候用 volatile 才对？记住一个判据：写入不依赖当前值（不是"读-改-写"），或者只有一个线程写、多个线程读。最典型的两个正确场景：① 状态标志位（如上面的 running、开关量）；② 双重检查锁定（DCL）里的单例引用（下一节讲）。

四、synchronized：原子性 + 可见性 + 有序性一把梭

如果说 volatile 是"精准修补"，synchronized 就是"重锤"——它通过互斥（一次只让一个线程进临界区），一次性解决三大问题。它有三种用法，锁的对象各不相同，这是高频考点：

三种写法，锁的是谁？看清 monitor 挂在哪个对象上
class Counter {
    // ① 同步实例方法 → 锁 this（当前实例）
    public synchronized void incr() { ... }

    // ② 同步静态方法 → 锁 Counter.class（类对象，全局唯一）
    public static synchronized void reset() { ... }

    // ③ 同步代码块 → 锁你指定的任意对象（推荐，粒度可控）
    private final Object lock = new Object();
    public void add() {
        synchronized (lock) { ... }
    }
}
面试高频陷阱：实例锁和类锁互不干扰。①和② 锁的是不同对象（一个是实例，一个是 Counter.class），因此一个线程持有实例锁执行 incr()，不会阻塞另一个线程执行静态的 reset()。另一个坑：用 synchronized(new Object()) 或锁 String 常量、Integer 缓存对象，等于没锁或锁错——锁必须是被多个线程共享、且引用稳定的对象，所以用 private final 锁对象最稳。

底层：monitor 与对象头 Mark Word

synchronized 靠对象内置的 monitor（监视器锁）实现。编译后，同步代码块对应字节码里的 monitorenter / monitorexit 指令；同步方法则在方法表加一个 ACC_SYNCHRONIZED 标志。每个 Java 对象的对象头里有一块 Mark Word，记录当前的锁状态和持有者。

线程进入 synchronized 块: monitorenter ──► 争抢对象的 monitor ├─ 抢到: Mark Word 记下"我持有"，进入临界区 └─ 没抢到: 阻塞，进入 EntryList 等待队列 线程离开: monitorexit ──► 释放 monitor，唤醒等待队列里的线程 (异常退出时也会自动 monitorexit，锁不会泄漏)
synchronized 并不总是"重量级"。JDK 6 起对它做了大量优化，引入了锁升级机制：无锁 → 偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁。只有真正发生激烈竞争时才会膨胀为重量级锁（依赖操作系统的 mutex，涉及用户态/内核态切换才慢）。竞争不激烈时，轻量级锁靠 CAS 自旋就搞定，开销很小。所以"synchronized 一定慢"是过时的老黄历。（注：偏向锁在 JDK 15 起已默认关闭并被弃用，见 JEP 374，现代 JVM 主要在轻量级与重量级之间升级。）

五、happens-before：可见性的"官方保证书"

你可能会问：JMM 那么自由，我怎么知道"哪次写一定对哪次读可见"？答案是 happens-before 原则——它是 JMM 给程序员的可见性契约：若操作 A happens-before 操作 B，那么 A 的结果对 B 一定可见，且 A 排在 B 前面。你不用去理解底层内存屏障，记住这几条规则即可：

规则含义
程序顺序同一线程内，前面的操作 happens-before 后面的操作
锁规则对一个锁的 解锁 happens-before 后续对同一锁的加锁
volatile 规则对 volatile 变量的写 happens-before 后续对它的读
线程启动Thread.start() happens-before 该线程的所有动作
线程终止线程的所有动作 happens-before 其他线程 join() 成功返回
传递性A hb B 且 B hb C，则 A hb C
volatile 写 happens-before volatile 读　 ≈ 　解锁 happens-before 加锁
为什么 synchronized 能保证可见性？正是靠"锁规则 + 传递性"。线程 A 在锁内修改的变量，退出锁时刷回主内存；线程 B 加同一把锁时清空工作内存、重读主内存。于是 A 锁内的写 happens-before B 锁内的读——可见性就这么"顺带"实现了。这解释了一个常被忽略的点：synchronized 不只是互斥，它同时提供可见性，所以被锁保护的普通变量不需要再加 volatile。

六、经典应用：双重检查锁定（DCL）单例

DCL 是 volatile 和 synchronized 配合的最经典案例，也几乎是必考题。目标：懒加载单例，且只有第一次创建时才加锁（避免每次 getInstance() 都同步的性能损耗）。

✗ Bad · 少了 volatile
class Singleton {
  private static Singleton I;
  static Singleton get() {
    if (I == null)          // ①
      synchronized(Singleton.class){
        if (I == null)      // ②
          I = new Singleton();
      }
    return I;
  }
}
// new 分三步且可能重排:
// 1.分配 2.初始化 3.赋值引用
// 若 3 先于 2，别的线程在①
// 拿到未初始化完的对象 💥
✓ Good · 加 volatile 禁重排
class Singleton {
  private static volatile
          Singleton I;       // ★
  static Singleton get() {
    if (I == null)
      synchronized(Singleton.class){
        if (I == null)
          I = new Singleton();
      }
    return I;
  }
}
// volatile 禁止 2、3 重排，
// 引用可见即对象已初始化完 ✅
DCL 里 volatile 不能省。很多人以为有了双重检查和 synchronized 就够了，其实 I = new Singleton() 的"分配-初始化-赋值"三步可能被重排成"分配-赋值-初始化"。一旦引用先被赋值，另一个线程在第①处判断 I != null 直接返回了一个半成品对象，访问其字段就是脏数据。volatile 的"禁止重排序"正是为堵这个漏洞而生。（当然，用 静态内部类 或 enum 实现单例可以完全绕开这套复杂度，更推荐。）

七、volatile vs synchronized 怎么选

维度volatilesynchronized
可见性✓✓
有序性（禁重排）✓✓
原子性✗ 仅单次读/写✓ 整个临界区
是否阻塞不阻塞（无锁）会阻塞（互斥）
作用对象只能修饰变量方法 / 代码块
开销很轻较重（竞争激烈时）
典型场景状态标志、DCL 引用、一写多读复合操作、"读-改-写"、保护多个变量
还有一条选择：java.util.concurrent 原子类。对"读-改-写"型计数场景，AtomicInteger / AtomicLong 用 CAS（比较并交换）实现无锁原子操作，比 synchronized 更高效——它底层是 Unsafe.compareAndSwap，最终对应一条 CPU 的 lock cmpxchg 指令。它兼具 volatile 的可见性 和 单操作的原子性，可以视作"能做加减的 volatile"。这条线索会在后续线程池与并发工具章节继续展开。

八、面试速记清单

背诵要点：
· 并发三问题：可见性 / 原子性 / 有序性，根源是 CPU 缓存 + 指令重排，JMM 是其抽象。
· volatile = 可见性 + 禁重排，不保证原子性（i++ 会丢更新）。
· synchronized = 互斥（原子性）+ 可见性 + 有序性；靠 monitor 实现，字节码是 monitorenter/exit。
· 锁的对象：实例方法锁 this、静态方法锁 Class、代码块锁指定对象；实例锁与类锁互不阻塞。
· JDK 6+ 锁升级：无锁 → (偏向锁) → 轻量级锁(CAS) → 重量级锁；synchronized 已不再"一定慢"。
· 可见性的理论基础是 happens-before：解锁 hb 加锁、volatile 写 hb 读。
· DCL 单例必须给引用加 volatile，防"分配-赋值-初始化"重排暴露半成品对象。
一句话收束：并发的三座大山是可见性、原子性、有序性。volatile 是一把"轻刀"——只保证可见性与有序性，代价小、不阻塞，适合状态标志和一写多读；synchronized 是一把"重锤"——用互斥同时锁住三者，代价是阻塞。选型只看一个问题：你要保护的是"单个变量的读或写"，还是"一连串必须不可分割的操作"？前者上 volatile（或原子类），后者上 synchronized（或 Lock）。理解了 JMM 与 happens-before，这两个关键字就不再是魔法，而是可推导的工程结论。
