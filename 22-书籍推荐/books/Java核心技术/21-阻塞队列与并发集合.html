阻塞队列与并发集合

把线程之间"交接工作"这件危险的手工活，封装成一个线程安全、满了会等、空了也会等的队列——从此告别裸写 wait/notify。

一、要解决的问题：生产者-消费者

并发编程里最常见的协作模式是生产者-消费者：一批线程负责生产任务（下载的网页、待写库的日志、上传的图片），另一批线程负责消费处理。两边速度往往不匹配——生产快了，任务堆积撑爆内存；消费快了，消费者只能空转浪费 CPU。中间需要一个缓冲区来削峰填谷。

用普通 ArrayList 当缓冲区行不行？不行。多线程同时读写会数据错乱（非线程安全）；就算加了锁，还有两个头疼问题：缓冲区满了生产者该怎么办？缓冲区空了消费者又该怎么办？答案是"等待"——但手写"等待/唤醒"极易出错。阻塞队列（BlockingQueue）就是把这三件事（线程安全 + 满了阻塞 + 空了阻塞）一次性打包好的标准件。

生产者线程 阻塞队列 (缓冲区) 消费者线程 ┌────────┐ put() ┌──┬──┬──┬──┬──┐ take() ┌────────┐ │ P1 ────┼──────────────>│t5│t4│t3│t2│t1│─────────────>│ C1 │ │ P2 ────┼──────────────>│ │ │ │ │ │─────────────>│ C2 │ └────────┘ └──┴──┴──┴──┴──┘ └────────┘ 队列满 → put 阻塞 容量有界 队列空 → take 阻塞 (天然背压 / 限流) FIFO 先进先出 (不空转, 让出 CPU)
核心直觉——快递驿站：快递员（生产者）把包裹放进驿站货架（队列），收件人（消费者）自己来取。货架满了快递员就得在门口等（put 阻塞）；货架空了收件人白跑一趟就在门口等来货（take 阻塞）。驿站解耦了双方，谁也不用直接等谁、当面交接。这就是解耦 + 缓冲 + 背压三合一。

二、BlockingQueue 的四组方法

BlockingQueue 继承自 Queue，最精妙的设计是：同一个操作（入队/出队）提供了四种"失败处理策略"，让你按场景选。这是初学者最容易混淆、也是面试爱考的点。

操作抛异常返回特殊值一直阻塞超时退出
入队add(e)offer(e)put(e)offer(e,t,u)
出队remove()poll()take()poll(t,u)
查看队首element()peek()——
  • 抛异常组：队列满时 add 抛 IllegalStateException，队列空时 remove 抛 NoSuchElementException。
  • 返回特殊值组：offer 满了返回 false，poll 空了返回 null——不阻塞、不抛异常，最"温和"。
  • 阻塞组：put/take 是生产者-消费者的主角，满/空就一直等，直到条件满足或被中断。
  • 超时组：等一段时间还不行就放弃，返回布尔或 null，兼顾"等待"与"不死等"。
高频陷阱——别用 add 往有界队列里硬塞。很多人下意识写 queue.add(task)，队列一满就抛 IllegalStateException 把线程炸掉。生产者-消费者场景想要"满了就等"用 put；想要"满了就丢/走降级"用 offer 判返回值。add 只适合你确信不会满的无界队列。同理 poll 空了返回 null，若不判空直接用会导致后续 NPE。

三、七种实现该怎么选

JDK 在 java.util.concurrent 下提供了一整个家族，各有取舍。选错会直接影响吞吐、内存甚至安全：

经典 OOM 事故——Executors.newFixedThreadPool() 内部用的就是无界 LinkedBlockingQueue。一旦任务提交速度长期超过处理速度，任务会在队列里无限堆积，最终 OutOfMemoryError。阿里《Java 开发手册》因此强制要求：不要用 Executors 快捷方法，改用 new ThreadPoolExecutor(...) 显式传有界队列 + 明确的拒绝策略。这也是下一篇线程池要展开的重点。

四、原理：阻塞是怎么实现的

阻塞队列的"满了等、空了等"不是靠自旋空转（那样浪费 CPU），而是基于 ReentrantLock + Condition（条件变量）。以 ArrayBlockingQueue 为例，它内部维护一把锁和两个"等待室"：notEmpty（消费者在这等队列变非空）和 notFull（生产者在这等队列变不满）。

ArrayBlockingQueue.java (JDK 源码节选)核心字段 + put
final Object[] items;         // 环形数组：用取模复用空间，不用真的搬数据
int takeIndex, putIndex, count;  // 队头下标 / 队尾下标 / 当前元素数
final ReentrantLock lock;    // 一把锁保护整个队列
private final Condition notEmpty;  // 消费者的等待队列
private final Condition notFull;   // 生产者的等待队列

public void put(E e) throws InterruptedException {
    Objects.requireNonNull(e);         // 不允许 null 元素
    final ReentrantLock lock = this.lock;
    lock.lockInterruptibly();          // 可被中断的加锁
    try {
        while (count == items.length)  // ★ 必须是 while 不是 if！见下方陷阱
            notFull.await();             // 队列满 → 生产者进 notFull 等待室并释放锁
        enqueue(e);                    // 放入元素，内部会 notEmpty.signal() 唤醒消费者
    } finally {
        lock.unlock();
    }
}

配对的 take() 逻辑对称：队列空时在 notEmpty 上 await()，取出元素后 notFull.signal() 唤醒一个正在等待的生产者。await() 的关键在于它会原子地释放锁并让线程休眠，把 CPU 让给别人；被 signal() 唤醒后再重新竞争锁——这正是操作系统层面"阻塞而非忙等"的思想，避免无谓的 CPU 消耗。

put 满时: 拿锁 → count==length? → notFull.await() ─┐ 释放锁并休眠 │ take 取走一个 → count-- → notFull.signal() ─────────┘ 唤醒生产者 生产者被唤醒 → 重新抢锁 → 再次检查 while → 条件满足 → 入队
面试必考——为什么条件判断要用 while 而不是 if？两个原因：① 虚假唤醒（spurious wakeup），线程可能没被 signal 也醒来，这是操作系统允许的行为；② 多消费者竞争，A 被唤醒后到真正抢到锁之间，元素可能已被 B 抢走，此时队列又空了。用 if 只判一次就会带着"队列非空"的错误假设去操作，导致取到 null 或数组越界。while 保证被唤醒后再确认一遍条件，这是使用条件变量的铁律——Object.wait() 同理。

五、实战：三行代码搞定生产者-消费者

有了 BlockingQueue，你完全不需要碰锁和 wait/notify——所有同步细节都被封装在 put/take 里。对比一下"裸手写"和"用队列"的差距：

✗ 裸写 wait/notify（暗藏 bug）
Object lock = new Object();
Queue<Task> q = new LinkedList<>();

// 消费者
synchronized (lock) {
  if (q.isEmpty())   // BUG: 该用 while
    lock.wait();
  Task t = q.remove();
  // 多消费者时 remove 可能抛异常
}
// 还要自己处理满、中断、
// signal 时机... 极易写错
✓ 用 BlockingQueue（省心正确）
BlockingQueue<Task> q =
    new ArrayBlockingQueue<>(100);

// 生产者
q.put(task);   // 满了自动阻塞

// 消费者
Task t = q.take();  // 空了自动阻塞
handle(t);

// 锁、条件、唤醒、中断
// 全由 JDK 保证正确
完整的生产者-消费者骨架含"毒丸(poison pill)"优雅关闭
BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);
final String POISON = "__END__";   // 约定一个终止信号

Runnable producer = () -> {
    try {
        for (String task : tasks) queue.put(task);
        queue.put(POISON);            // 生产完投毒丸，通知消费者收工
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt(); // 恢复中断标志，别吞掉
    }
};

Runnable consumer = () -> {
    try {
        while (true) {
            String task = queue.take();   // 空则阻塞，不空转
            if (POISON.equals(task)) break; // 收到毒丸就退出循环
            handle(task);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
};
联系线程池：你几乎每天都在间接用阻塞队列。ThreadPoolExecutor 内部就是一个大号的生产者-消费者——你 submit() 任务是"生产"，工作线程从 workQueue（一个 BlockingQueue）里 take() 任务执行是"消费"。换用不同队列会直接改变线程池性格：SynchronousQueue → 来一个任务直接开线程（newCachedThreadPool）；无界 LinkedBlockingQueue → 任务无限排队（newFixedThreadPool）；有界 ArrayBlockingQueue → 排满就触发拒绝策略。这是下一篇的核心伏笔。

六、并发集合：不止队列

阻塞队列解决"线程间传递数据"，而并发集合解决"线程间共享数据结构"。老一辈的 Vector/Hashtable 靠给每个方法加 synchronized 实现线程安全——全表一把大锁，任何时刻只有一个线程能操作，高并发下就是性能瓶颈。java.util.concurrent 提供了更聪明的替代品：

并发集合替代谁核心技巧
ConcurrentHashMapHashtableCAS + 锁单个桶（Java 8），读几乎无锁，并发度极高
CopyOnWriteArrayListVector写时复制整个数组，读完全无锁
ConcurrentSkipListMapTreeMap跳表实现的有序 Map，无锁并发
ConcurrentLinkedQueue—CAS 实现的无锁非阻塞队列（不阻塞，队空返回 null）

CopyOnWrite：读多写极少的利器

CopyOnWriteArrayList 的思路很极端：读操作完全不加锁（因此飞快），写操作则先把整个底层数组复制一份，在副本上改，改完用新数组原子替换旧引用。由于读的永远是一个"不会再变"的快照，读写互不干扰。

CopyOnWriteArrayList.javaadd 的写时复制
public boolean add(E e) {
    synchronized (lock) {                    // 写之间才互斥，读不受影响
        Object[] es = getArray();
        int len = es.length;
        es = Arrays.copyOf(es, len + 1);   // ★ 复制一份新数组
        es[len] = e;
        setArray(es);                        // 原子替换引用，读线程下一秒就看到
        return true;
    }
}
别把 CopyOnWrite 当银弹。每次写都全量复制数组，写多的场景性能灾难、内存翻倍；而且读到的是"旧快照"，写入后其他线程的迭代器不会立刻看到新元素（弱一致性）。它只适合读远多于写的场景，最典型的就是"事件监听器列表""白名单配置"这类初始化后极少改动、却被高频遍历的数据。Spring 的事件监听器容器、很多框架的 Listener 列表内部就用它。

七、fail-fast vs 弱一致性迭代器

为什么并发集合遍历时不像 ArrayList 那样抛 ConcurrentModificationException？因为它们用的是不同的迭代器策略，这是理解并发集合"安全边界"的关键对比：

维度fail-fast（普通集合）弱一致性（并发集合）
代表ArrayList / HashMapConcurrentHashMap / CopyOnWriteArrayList
遍历时被修改立刻抛 ConcurrentModificationException不抛异常，正常遍历完
能否看到新改动—（直接失败）可能看到、也可能看不到，不保证
设计目的尽早暴露"边遍历边改"的 bug并发下的可用性优先于强一致
为什么并发集合选择"弱一致"？因为在真正的多线程环境下，"遍历的同时数据被别的线程改了"是常态而非 bug，若像 fail-fast 那样动不动抛异常，程序根本没法用。弱一致性迭代器基于某个时刻的数据视图工作，宁可"看不到最新修改"也不崩溃——这是用一致性换可用性的经典权衡，和分布式系统里的 CAP 取舍是同一种哲学。

八、性能直觉：锁粒度决定并发度

这一整套并发容器进化的主线，其实就一句话：不断缩小锁的粒度。锁得越细，能同时干活的线程越多。

吞吐 ≈ 可并行操作数 ∝ 1 / 锁粒度
Hashtable 全表锁 ArrayBlockingQueue 单锁 LinkedBlockingQueue 双锁 ConcurrentHashMap 桶锁+CAS

从"整张表一把锁"（Hashtable），到"队列一把锁"（ArrayBlockingQueue），到"入队出队各一把锁"（LinkedBlockingQueue），再到"每个桶一把锁 + 无锁 CAS"（ConcurrentHashMap），最后到"完全无锁"（ConcurrentLinkedQueue 基于 CAS）——每一步都在把"必须串行"的范围压小，让更多线程真正并行。理解了这条主线，你就能预判任何一个并发容器大概快在哪、慢在哪。

一句话收束：阻塞队列 = 线程安全的缓冲区 + 满/空自动阻塞，把生产者-消费者从"裸写 wait/notify 的地狱"简化成 put/take 两个方法；它的底层是 ReentrantLock + Condition，条件判断永远用 while。选型三条铁律：① 生产者-消费者优先有界的 ArrayBlockingQueue，绝不裸用无界 LinkedBlockingQueue（会 OOM）；② 满了用 put 阻塞、用 offer 降级，别用会抛异常的 add；③ 共享数据结构选并发集合而非 Vector/Hashtable，读多写少用 CopyOnWriteArrayList，通用 Map 用 ConcurrentHashMap。它们的迭代器是弱一致的，不抛 ConcurrentModificationException——这不是 bug，是并发下"可用性优先"的刻意设计。
