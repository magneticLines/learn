Object 类：equals、hashCode、toString

每个 Java 对象都继承的三个方法，看似不起眼，却藏着一套必须严守的"契约"——违反它，HashMap 会当场把你的对象弄丢。

一、Object：所有类的共同祖先

在 Java 里，只要你没写 extends，编译器就自动帮你 extends Object。所以任何一个类、任何一个对象，天生就带着 Object 的一整套方法。这是一种"最大公约数"设计：JVM 需要一个能装下"任意对象"的类型（比如老式集合的 Object[]），也需要给所有对象一些通用能力（打印、比较、求哈希、加锁）。

java.lang.Object <-- 一切类的根，隐式的父类 | +-------------+-------------------------------+ | | | | toString() equals(o) hashCode() getClass() 「怎么打印」 「是否相等」 「哈希桶编号」 「运行时类型」 | | | +-- 默认实现都基于"对象身份(identity)"，即 this==o --+ | 想按"内容相等"就必须自己 override

其中 getClass()、wait/notify 被 final 修饰不能覆盖；而 toString、equals、hashCode 三兄弟是留给你重写的——它们的默认实现只认"是不是同一个对象"，一旦你想表达"内容相等"，就得亲自动手，而且必须成套地、守契约地改。这正是本篇的主题。

二、toString：对象的自我介绍

Object.toString() 的默认实现是这样的（JDK 源码原文）：

java.lang.Object默认 toString 源码
public String toString() {
    return getClass().getName() + "@" + Integer.toHexString(hashCode());
}
// 打印出来长这样： com.demo.User@1b6d3586
// 全限定类名 + "@" + hashCode 的十六进制
误区提醒：那串 @1b6d3586 不是内存地址！它只是 hashCode() 的十六进制表示。很多人以为"打印对象看到的就是地址"，其实 Java 从不向应用层暴露真实地址（GC 会移动对象，地址根本不稳定）。这个值只是身份哈希，下一节细说。

默认输出对人类毫无意义。所以只要这个类的对象会被打印、会进日志、会在调试器里被观察，就应该重写 toString。它是"程序员友好"的方法——System.out.println(obj)、字符串拼接 "" + obj、日志框架的 {} 占位符，底层全都在悄悄调用它。

实战建议：用 IDE 一键生成，或直接用 Lombok @ToString / record（后面讲）。手写时把关键字段都带上，例如 return "User{id=" + id + ", name='" + name + "'}";。但注意：别在 toString 里打印密码、token 等敏感字段，日志泄密是真实事故。

三、== 与 equals：一字之差，天壤之别

初学者最爱踩的第一个坑。先建立直觉：== 比的是"变量里存的值"，对引用类型来说，存的是对象的地址，所以 == 比的是"是不是同一个对象"；而 equals 想表达的是"内容是否相等"——但前提是这个类重写了它。

对比维度==equals()
本质运算符，比较栈上的值方法，可被重写
基本类型（int 等）比较数值大小不适用（基本类型没方法）
引用类型比较是否同一对象（地址）默认同 ==；重写后比内容
能否处理 nulla == null 安全a.equals(b) 若 a 为 null 会 NPE
经典面试陷阱 —— String 常量池：
String a = "hi"; String b = "hi"; → a == b 为 true（都指向常量池同一个对象）。
String c = new String("hi"); → a == c 为 false（new 强制在堆上造新对象），但 a.equals(c) 为 true。
结论：比较字符串/包装类等对象内容，永远用 equals，绝不用 ==。而 Integer 还有 -128~127 缓存池的坑，同样只有 equals 才靠谱。

四、equals 的五条契约

为什么 JDK 要在 equals 的 Javadoc 里郑重写下"契约"？因为 HashMap、HashSet、List.contains、List.indexOf……无数集合方法都假定你的 equals 满足这些性质。一旦违反，行为就变得诡异且难以调试。五条契约（对非 null 引用 x、y、z）：

契约含义直觉
自反性x.equals(x) 恒为 true自己等于自己
对称性x.equals(y) ⇔ y.equals(x)相等是双向的
传递性x=y 且 y=z ⇒ x=z相等可传导
一致性对象不变时，多次调用结果一致别依赖随机/时间等易变量
非空性x.equals(null) 恒为 false任何对象都不等于 null

一份满足契约的标准写法（Objects.equals 帮你处理字段为 null 的情况）：

User.java标准 equals 写法
@Override
public boolean equals(Object o) {
    if (this == o) return true;              // ① 同一对象，短路，满足自反性&性能
    if (o == null || getClass() != o.getClass()) // ② null 判断 + 类型判断
        return false;
    User user = (User) o;                     // ③ 安全强转
    return id == user.id                        // ④ 逐字段比：基本类型用 ==
        && Objects.equals(name, user.name);  //    引用类型用 Objects.equals(防 NPE)
}
getClass() 还是 instanceof？这是《Effective Java》里争论多年的话题。
· 用 instanceof：允许子类和父类"相等"，但一旦子类加了新字段还想参与比较，对称性极易被破坏（父类.equals(子类)=true，反过来却=false）。
· 用 getClass()：要求"两边必须是完全同一个类"，天然对称、传递，代价是违反里氏替换（子类实例永远不等于父类实例）。
对普通业务实体，推荐 getClass() 更安全；确需跨层次比较时，优先"组合优于继承"来规避，而不是硬写 instanceof。
对称性翻车实录：经典的 Point / ColorPoint 例子——若 ColorPoint 用 instanceof 写 equals，会出现 p.equals(cp)==true（Point 只比坐标）但 cp.equals(p)==false（ColorPoint 还比颜色）。此时把它们塞进同一个 List 再 contains，结果取决于谁先谁后，是极难排查的 bug。

五、hashCode：与 equals 绑死的契约

hashCode 返回一个 int，用途是给基于哈希的容器（HashMap/HashSet）快速定位"桶（bucket）"。它和 equals 之间有一条单向但必须遵守的铁律：

a.equals(b) == true ⇒ a.hashCode() == b.hashCode()

注意方向：相等 → 哈希必相等；反过来不成立（哈希相同不代表相等，这叫哈希冲突，是允许的）。所以规则记两句就够：

  • 只要 equals 认为两个对象相等，它俩的 hashCode 必须相等。
  • 同一对象在内容不变的前提下，多次调用 hashCode 应返回同一个值。

为什么 HashMap 死磕这条契约？看它 get 的流程就懂了：先用 hashCode 找桶，再在桶内用 equals 逐个比对。如果两个"相等"的对象 hashCode 不同，它们会被分到不同的桶——put 进去后再用另一个"相等"的 key 去 get，永远找不到那个桶，返回 null。

map.put(userA, "VIP"); hashCode=17 -> 桶[1] 存入 map.get(userB); // userB.equals(userA)==true，本该拿到 "VIP" 若 hashCode 也一致： userB.hashCode()=17 -> 桶[1] -> 桶内 equals 命中 -> 返回 "VIP" ✔ 若只重写了 equals、忘了 hashCode： userB.hashCode()=93 -> 桶[5] -> 桶里空的 -> 返回 null X 丢了！

这就是那句名言的由来：重写 equals 就必须重写 hashCode，两者是一对不可拆分的孪生方法。反例与正例对照：

✗ 反例：只改 equals
// 只重写 equals，hashCode
// 用 Object 默认(基于身份)
@Override
public boolean equals(Object o){
   ... 按 id 比较 ...
}
// 两个 id 相同的 User：
// equals -> true
// 但 hashCode 各不相同！

Set<User> s = new HashSet<>();
s.add(new User(1));
s.add(new User(1));
// s.size() == 2  去重失效！
✓ 正例：成套重写
@Override
public boolean equals(Object o){
   ... 按 id、name 比较 ...
}
@Override
public int hashCode(){
   // 字段与 equals 保持一致
   return Objects.hash(id, name);
}

Set<User> s = new HashSet<>();
s.add(new User(1));
s.add(new User(1));
// s.size() == 1  正确去重
身份哈希是怎么来的？没重写时的默认 hashCode 是 native 方法。在 HotSpot 里，它不是内存地址，而是首次调用时用一个（默认基于 xorshift 的）伪随机算法算出、然后缓存进对象头 Mark Word 里的一个值——后续调用直接读缓存，保证"一致性"。这也解释了：为何身份哈希稳定，即使 GC 移动了对象。（顺带一提：一旦对象因 hashCode 占用了 Mark Word，偏向锁等就会受影响，这是并发篇的伏笔。）

六、现代写法：让工具替你守契约

手写这俩方法样板代码多、易错。工程里几乎不再手写，而是靠工具生成——关键是保证 equals 和 hashCode 用的是同一组字段，工具天然做到这点。

方案怎么用取舍
IDE 生成Alt+Insert → equals()/hashCode()零依赖，字段增改后要记得重新生成
Lombok@EqualsAndHashCode / @Data最省代码；注意继承时 callSuper 与 JPA 实体的坑
recordJava 16+，record User(int id, String name){}自动生成三方法且不可变，DTO/值对象首选
Objects 工具Objects.hash(...) / Objects.equals(a,b)手写时的标配，帮你处理 null 与拼哈希
Point.javarecord：一行搞定三方法
// record 会自动生成：全字段构造器、equals、hashCode、toString、访问器
public record Point(int x, int y) {}

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);
p1.equals(p2);      // true  —— 按分量比
p1.hashCode() == p2.hashCode();  // true —— 契约自动满足
p1.toString();     // "Point[x=1, y=2]"
最隐蔽的陷阱 —— 可变对象当 HashMap 的 key：如果把对象放进 HashSet/HashMap 后，又修改了参与 hashCode 计算的字段，它的哈希值就变了，但它还待在按旧哈希算出的旧桶里。此后无论用 get 还是 contains 都可能找不到它——对象"人间蒸发"。原则：哈希容器的 key 应当是不可变的（如 String、record、或只用不可变字段算哈希）。这也是为什么 String 被设计成 immutable、还缓存了 hash 值。
一句话收束：toString 是给人看的、可选；equals 与 hashCode 是给集合看的、必须成对出现且共用同一组字段。铁律只有一条方向——equals 相等 ⇒ hashCode 必相等；守住它，HashMap 才不会把你的对象弄丢。能用 record / Lombok 就别手写，把守契约的活交给编译器。
