Java 核心 · 第4-5章·面向对象

继承、多态、super

让"新类型复用旧类型、又能表现出自己脾气"——面向对象把变化关进笼子的核心机制。

一、继承要解决的真问题：消除重复 + 面向未来

假设你在写一套员工系统，先有了 Employee（有姓名、薪水、算奖金 getBonus()）。现在要加一种"经理"，经理除了普通员工的一切，还多一个"管理津贴 bonus"。笨办法是把 Employee 的字段和方法整个复制一份到 Manager——一旦哪天薪水计算规则改了，你得记得改两处，漏一处就是 bug。

继承（inheritance）的动机正是消灭这种复制：子类自动拥有父类的全部字段和方法，只需写出"多出来的"和"要改动的"部分。它表达的是一种 "is-a"（是一种） 关系——经理"是一种"员工。

生活类比：父类像一张"通用简历模板"，写好了所有人共有的栏目（姓名、联系方式）。子类不是重新写简历，而是"在模板基础上补充自己的专业技能栏"。模板改了版式，所有基于它的简历自动跟着变——这就是继承省下的维护成本。
Employee.java / Manager.javaextends 的最小骨架
public class Employee {
    protected String name;
    protected double salary;

    public Employee(String name, double salary) {
        this.name = name;
        this.salary = salary;
    }
    public double getSalary() { return salary; }
}

public class Manager extends Employee {   // 一个 extends，白拿父类全部
    private double bonus;              // 只写"多出来的"字段

    public Manager(String name, double salary, double bonus) {
        super(name, salary);       // 必须先把父类那半个身子造好
        this.bonus = bonus;
    }

    @Override
    public double getSalary() {       // 只写"要改动的"行为
        return super.getSalary() + bonus;  // 复用父类逻辑再叠加
    }
}

二、super 的两副面孔：调构造器、调父类方法

super 常被误当成"父类对象"，其实它不是一个对象引用（不能 Employee e = super;），而是编译器提供的一个关键字指令，只有两种合法用法：

  • super(...)：在子类构造器里调用父类构造器，且必须是构造器的第一条语句；
  • super.方法()：在被覆盖的场景下，穿透"就近覆盖"直接调用父类版本，如上面 getSalary() 复用父类逻辑。
高频陷阱一：构造器不被继承，且父类先造。子类构造器若不写 super(...)，编译器会自动插入一个无参 super()。因此——如果父类只定义了带参构造器（没有无参构造器），子类构造器又忘了显式 super(参数)，会直接编译报错。这是初学者最常见的"莫名其妙编译不过"。
联系底层：为什么"父类先造"是硬规则？因为子类对象在内存里是"父类那部分字段 + 子类新增字段"拼在一起的一整块。父类的构造逻辑（可能给字段设了默认合法值、建立了不变式）必须先跑完，子类才能在一个"合法的父类地基"上继续盖楼。所以 super(...) 必须排在最前面——你不能在地基还没打好时就装修二楼。

super vs this 速查

维度thissuper
指向当前对象自己指令：访问父类那一层的成员/构造器
调构造器this(...) 调本类其他构造器super(...) 调父类构造器
调方法this.m() 走动态分派（可能落到子类）super.m() 强制静态绑定到父类版本
构造器中位置必须第一条必须第一条
推论：this(...) 和 super(...) 都要求"必须是第一条语句"，所以二者不能在同一个构造器里同时出现。常见写法是子类多个构造器用 this(...) 互相转发，最终由某一个统一 super(...)。

三、覆盖(override) ≠ 重载(overload)

两个词长得像，是完全不同的机制，面试必考、初学者必混。

覆盖 Override重载 Overload
发生在父类与子类之间同一个类内（也可跨父子）
方法签名方法名+参数完全相同方法名相同、参数列表不同
绑定时机运行期动态分派编译期静态决定
返回值相同或协变（子类型）不参与区分，可不同
关键字建议加 @Override无
务必写 @Override 注解。它不是装饰，而是让编译器帮你校验"我真的覆盖到了父类方法"。一旦你把参数或方法名写错（本意是覆盖，结果变成了新方法/重载），有注解就会立刻编译报错，而不是留下一个"永远不会被调用的僵尸方法"这种极难排查的暗坑。经典翻车：想覆盖 Object.equals(Object)，却写成 equals(Employee)——那是重载，不是覆盖。
✗ Bad · 以为在覆盖 equals
class Point {
  int x, y;
  // 参数写成了 Point，编译器不报错
  public boolean equals(Point o) {
    return x==o.x && y==o.y;
  }
}
// HashSet/Map 内部调的是
// equals(Object)，永远命中不到它 ->
// 去重、查找全部失效！
✓ Good · 真正覆盖，签名对齐
class Point {
  int x, y;
  @Override
  public boolean equals(Object o) {
    if (!(o instanceof Point p))
      return false;
    return x==p.x && y==p.y;
  }
  // 加了 @Override，写错立刻红
}

四、多态：一句 e.getSalary() 背后的动态分派

多态（polymorphism）是这一切的高潮：用父类类型的变量去调方法，实际执行的是"变量当前真正指向的那个对象"的版本。这叫动态绑定 / 动态分派（dynamic dispatch）。

多态的威力一段循环处理所有子类型
Employee[] staff = {
    new Employee("Carl", 8000),
    new Manager("Harry", 10000, 5000)   // Manager 也 is-a Employee
};
for (Employee e : staff) {
    // 声明类型都是 Employee，但每个对象跑自己的 getSalary()
    System.out.println(e.getSalary());
}
// 输出 8000.0 和 15000.0 —— 调用方一行 if/switch 都没写

威力在于：新增一种员工类型（比如 Executive），这段循环代码一个字都不用改。这正是"对扩展开放、对修改关闭"（开闭原则）的落地，也是策略模式、模板方法模式、Spring 里注入不同实现的底层同一个原理。

它是怎么做到的：方法表(vtable)与内存视图

e.getSalary() 编译时并不知道 e 到底是 Employee 还是 Manager。JVM 在运行时顺着对象头找到它真实类的方法表（虚方法表 vtable），查到 getSalary 那一格里存的方法地址，再跳转执行——地址指向谁，就执行谁的版本。

引用变量 e (声明类型 Employee) │ 运行时指向 ↓ ▼ ┌──────────────── 堆上的 Manager 对象 ───────────────┐ │ 对象头 ──> Class 指针 ──> Manager 的方法表(vtable) │ │ name="Harry" salary=10000 bonus=5000 │ └────────────────────────────────────────────────────┘ │ Manager vtable ▼ ┌───────────────────────────────┐ getSalary │ &Manager.getSalary ← 被覆盖 │ ← 查这一格 getName │ &Employee.getName ← 未覆盖 │ └───────────────────────────────┘ 声明类型只决定"编译期能调哪些方法(能看到哪些格子)" 真实类型决定"运行期格子里填的是谁的实现"
调用结果 = 方法名(编译期定) × 真实对象(运行期定)
联系底层：静态 vs 动态。并非所有调用都动态分派。private、static、final 方法以及构造器都是静态绑定（编译期定死，不进 vtable）——因为它们本就不能被覆盖，无需运行时查表，这也带来轻微性能优势。这解释了为什么 super.m() 是静态绑定：它明确要"父类那个版本"，绕开了动态查表。JIT 还会对"实际只有一种实现"的虚调用做去虚化(devirtualization)内联优化。

五、向上转型与向下转型

向上转型（子→父）永远安全、自动完成：Employee e = new Manager(...);。向下转型（父→子）是危险动作，必须显式写、且运行时可能抛 ClassCastException。

✗ Bad · 盲目向下转型
Employee e = new Employee("Carl", 8000);
Manager m = (Manager) e;
// e 其实不是 Manager ->
// 运行时炸 ClassCastException
m.setBonus(1000);
✓ Good · instanceof 先探路
Employee e = ...;
// Java 16+ 模式匹配，判断并绑定
if (e instanceof Manager m) {
  m.setBonus(1000);  // 安全
}
// null instanceof X 恒为 false，
// 天然免了空指针判断
坏味道提醒：代码里频繁出现 instanceof + 向下转型，往往是"多态没用好"的信号——本该由子类覆盖方法自动分派的逻辑，被你用 if (x instanceof A) ... else if (x instanceof B) 手写了一遍。此时应回头把差异行为下沉成子类覆盖的方法。真正需要转型的场景其实不多（多为与旧 API、Object 容器打交道时）。

六、继承 vs 组合：最重要的设计取舍

继承虽好，却是 Java 里耦合最强的一种复用方式：子类能看到父类的 protected 成员，父类内部实现一改，可能悄悄弄坏所有子类（脆弱基类问题）。业界共识（《Effective Java》第 18 条）是：优先用组合，谨慎用继承。

✗ Bad · 为复用而滥用继承
// 想复用 ArrayList 的能力就去继承它
class Stack<E> extends ArrayList<E> {
  void push(E e){ add(e); }
  E pop(){ return remove(size()-1); }
}
// 灾难：栈却暴露了 get(i)、
// add(i,e)、clear() ... 破坏"栈"语义
// 这就是 java.util.Stack 的历史包袱
✓ Good · 组合，只暴露该暴露的
class Stack<E> {
  private final List<E> list
      = new ArrayList<>();  // has-a
  void push(E e){ list.add(e); }
  E pop(){ return list.remove(list.size()-1); }
  // 内部换成 LinkedList 也不影响调用方
  // 对外只有 push/pop，语义干净
}
判断准则（is-a 测试）：只有当"子类确实是一种父类，且在任何用到父类的地方都能拿子类顶替而不出错"（里氏替换原则 LSP）时，才用继承。若你只是"想借用它的几个方法"，那是 has-a，用组合（把它作为一个字段持有）。经典反例：正方形继承长方形——数学上正方形是长方形，但 setWidth 会连带改高，违反长方形的行为契约，替换就出错。
联系生态：Java 顶层还有一条隐藏规则——每个类若没写 extends，都隐式继承 Object，且是单继承（一个类只能有一个直接父类，避免 C++ 多继承的"菱形歧义"）。想要"多重能力"，靠实现多个interface。Spring 的 JdbcTemplate、Servlet 的 HttpServlet（继承 GenericServlet 并让你覆盖 doGet/doPost）都是"父类搭好模板骨架、你覆盖钩子方法"的模板方法模式典范。

七、final 与抽象：给继承装上闸门

  • final 修饰类：禁止被继承（如 String、Integer——为了安全与不可变性）。
  • final 修饰方法：禁止被覆盖（锁死行为，也享受静态绑定）。
  • abstract 类：不能被 new，专门当父类；里面可有 abstract 方法（只有签名没有实现），强制子类必须覆盖实现。
陷阱：构造器里别调可被覆盖的方法。父类构造器执行时，子类字段还没初始化。若父类构造器调用了一个被子类覆盖的方法，跑的是子类版本，却会读到子类那些"尚未赋值、还是默认 0/null"的字段，导致极隐蔽的 bug。规则：构造器中只调用 private、static 或 final 方法（这些不会被覆盖）。
一句话收束：继承用 extends 表达 is-a 并复用父类、消除重复；super 只有 super(...) 调父类构造器与 super.m() 调父类方法两副面孔，且"父类先造、super 排第一"；多态是覆盖(override) + 运行时动态分派(vtable)的合力，让一段面向父类的代码自动适配所有子类，是开闭原则与设计模式的地基。但继承耦合强、易生脆弱基类，能用组合就别用继承，只在通过 is-a / 里氏替换测试时才动用它。
