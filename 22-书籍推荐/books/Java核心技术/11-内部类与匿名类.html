内部类与匿名类

把一个类"塞进"另一个类里，究竟解决了什么问题——顺着"闭包与变量捕获"这条线，看清 Java 是怎么伪造出闭包的。

一、为什么要把类塞进类里

面向对象最初的模型很朴素：一个 .java 文件放一个顶层类，类之间靠字段互相引用。但很快出现两种尴尬：其一，某个"小类"只服务于某个"大类"，单独暴露成顶层类既污染包命名空间，又割裂了它俩的强关联；其二，某个类型天生就要访问外层对象的私有状态（比如链表的迭代器要摸链表的内部节点），做成顶层类就得把私有字段开成 public，破坏封装。

内部类（nested class）就是为这两点而生：它是"逻辑上从属、且需要贴身访问外层"的类的容身之所。Java 把它分成四种，动机各不相同：

种类写在哪是否持有外层实例典型用途
静态嵌套类
static nested		类内，带 static否（独立）Builder、Map.Entry、纯粹的逻辑分组
成员内部类
inner		类内，无 static是（隐式 Outer.this）迭代器、需要读写外层字段的辅助类
局部内部类
local		方法体内是（若在实例方法中）只在一个方法里用一次的类
匿名内部类
anonymous		表达式里 new是（若在实例上下文）回调、事件监听、一次性接口实现
先建立直觉：把外层类想成一栋"公司大楼"。静态嵌套类是挂靠在公司名下、但独立运营的子品牌（不依赖任何一位具体员工）；成员内部类则是某位具体员工的"贴身秘书"——它离不开"某一个"外层对象，天生揣着老板的门禁卡（能进老板的私人办公室，读老板的私有字段）。这张"隐形门禁卡"就是理解内部类一切行为的钥匙。

二、成员内部类：那张隐形的 Outer.this 门禁卡

成员内部类最反直觉、也最强大的一点：它能直接读写外层对象的私有成员，仿佛那些字段是自己的。

Counter.java内部类穿透私有边界
public class Counter {
    private int count = 0;   // 私有，外部谁都碰不到

    // 成员内部类：没有 static
    public class Ticker {
        public void tick() {
            count++;            // 直接改外层的私有字段，编译器不拦
            System.out.println("count = " + count);
        }
    }

    public static void main(String[] args) {
        Counter c = new Counter();
        // 注意语法：内部类实例必须"依附"在某个外层实例上
        Counter.Ticker t = c.new Ticker();
        t.tick();   // count = 1
    }
}

它凭什么能碰 private count？因为编译器在背后偷偷给内部类加了一个字段，指向创建它的那个外层实例。反编译 Counter$Ticker.class 你会看到一个合成字段 final Counter this$0;，构造器也被塞进了一个 Counter 参数。所谓"直接访问 count"，实际被编译成 this$0.count。

Counter 对象 c (堆) Counter$Ticker 对象 t (堆) ┌──────────────────┐ ┌──────────────────────────┐ │ count = 1 │◀───────────│ this$0 ──▶ (指回 c) │ └──────────────────┘ 合成引用 │ (Ticker 自己的字段...) │ ▲ └──────────────────────────┘ │ c.new Ticker() 时，编译器把 c 塞进 this$0 └── 所以每个 Ticker 都"绑定"在某个具体 Counter 上 访问 count ≡ 编译器改写成 this$0.count
高频陷阱一：成员内部类不能有 static 成员（Java 16 前）。因为它天生依附外层实例才能存在，而 static 成员属于"类级别、无需实例"——两者语义冲突，编译器直接报错。所以想在内部类里放 static 常量/方法，就得把它声明为 static 嵌套类。（Java 16+ 放宽了这条限制，允许成员内部类拥有 static 成员，但老代码与面试题里这仍是经典考点。）
高频陷阱二：内部类导致外层对象无法被 GC。只要那张 this$0 门禁卡还被人握着（比如一个 Ticker 被放进了长生命周期的集合、或注册成了监听器），它就死死拽住整个外层 Counter 不放。这是 Android/Swing 里"Activity/窗口泄漏"的头号原因——一个内部类监听器活着，整个界面对象都没法回收。规则：凡是生命周期可能超过外层对象的内部类，优先做成 static 嵌套类，斩断这条隐式引用。

三、静态嵌套类 vs 成员内部类：差一个 static，差一条命脉

这两者长得像，本质天差地别，取舍只看一句话：它到底需不需要访问外层对象的实例字段？

✗ 不需要外层实例，却用了内部类
class Cache {
  private Map data;
  // Node 只是个纯数据容器，
  // 根本用不到 Cache 的实例字段
  class Node {          // 成员内部类
    Object key, value;
  }
  // 后果：每个 Node 都白白多背
  // 一个 this$0 引用，且拖住 Cache
  // 不被 GC —— 纯属浪费+泄漏风险
}
✓ 加 static，切断隐式引用
class Cache {
  private Map data;
  static class Node {   // 静态嵌套类
    Object key, value;
  }
  // Node 独立自足，不含 this$0
  // 更省内存，也不拖住 Cache
}
// 这正是 HashMap.Node、
// ConcurrentHashMap 里节点的做法
源码印证：翻开 JDK 的 java.util.HashMap，它的 static class Node<K,V>、static class TreeNode 全是静态嵌套类——因为一个哈希桶节点是纯数据，不需要回头找"我属于哪个 HashMap 实例"。而 HashMap 的迭代器 HashIterator 却是非静态成员内部类，因为它必须实时访问外层 table 数组、modCount 来做遍历和 fail-fast 检查。一静一动，教科书级的取舍示范。

四、匿名内部类：声明与实例化二合一

当你只需要"某个接口/抽象类的一次性实现"，为它单独写个具名类太啰嗦。匿名内部类让你在 new 的现场直接写出类体，声明和创建对象一步到位。

匿名类的经典场景：回调Comparator / Runnable
// 给字符串按长度排序：现场造一个 Comparator 实现
Comparator<String> byLen = new Comparator<>() {
    @Override
    public int compare(String a, String b) {
        return Integer.compare(a.length(), b.length());
    }
};  // ← 分号！这整个是一条赋值语句

// 起一个线程：现场造一个 Runnable
new Thread(new Runnable() {
    @Override public void run() {
        System.out.println("在新线程里跑");
    }
}).start();

匿名类没有名字，所以不能写构造器（构造器名要和类名一致，而它没名字）。它只能"就地实现一个接口"或"就地继承一个类"，二选一。编译后你会得到一个 Outer$1.class、Outer$2.class 这样以数字编号的类文件——这就是它"匿名"的物理证据。

陷阱：匿名类里的 this 指自己，不是外层。在匿名类方法体内写 this，指的是这个匿名对象本身；要引用外层对象，得写 Outer.this。这和 Lambda 恰好相反（见第七节），是两者一个关键区别，也是把匿名类改写成 Lambda 时最容易踩的语义变化。

五、闭包线索：什么是变量捕获

现在进入本页的主线。看这段代码——一个方法返回了一个"记住了局部变量的对象"：

Adder.java局部变量被"带出"了方法
interface IntOp { int apply(int x); }

static IntOp makeAdder(int delta) {   // delta 是局部变量
    return new IntOp() {
        public int apply(int x) {
            return x + delta;   // 匿名类"记住"了 delta
        }
    };
}

IntOp add10 = makeAdder(10);
System.out.println(add10.apply(5));  // 15，此时 makeAdder 早已返回

makeAdder 执行完就返回了，它的栈帧被销毁，局部变量 delta 本该"人死账消"。可返回的那个对象在方法结束后还能用出 delta 的值——这种"函数 + 它捕获的外部变量"打包在一起的东西，就是闭包（closure）。被带走的 delta 叫做被捕获的变量（captured variable）。

闭包 = 代码逻辑 + 被捕获的环境变量

Java 是怎么实现的？它没有真正的"变量共享闭包"，而是用了一个朴素办法：捕获 = 值拷贝。编译时，编译器给匿名类合成一个字段（比如 final int val$delta;），并在构造它时把 delta 当前的值拷进去。所谓"记住 delta"，其实是"把 delta 复制了一份存进对象里"。

makeAdder(10) 执行中 栈帧销毁后 ┌─────────────────┐ ┌─────────────────┐ │ 栈: delta = 10 │ 拷贝 │ (栈帧没了) │ │ │ │────────▶│ │ └────────┼────────┘ └─────────────────┘ ▼ new 匿名对象时 ▲ ┌──────────────────────────┐ │ 对象独立存活于堆， │ 匿名对象 (堆) │ │ 自带 delta 的副本， │ val$delta = 10 ◀──────┼──────────────┘ 不再依赖那个栈变量 └──────────────────────────┘ 结论：捕获的是"值的快照"，不是"变量本身的地址"

六、为什么捕获的变量必须是"事实 final"

你一定遇到过这个编译错误："local variables referenced from an inner class must be final or effectively final"。为什么 Java 逼着被捕获的局部变量不能再被修改？

答案就藏在上一节的"值拷贝"里。既然内部类拿到的是 delta 的一份副本，那就产生了一个危险的"两地分居"：方法栈里有一个 delta，内部类对象里有一个 val$delta。如果允许方法里再改 delta，两份就会不一致——你以为改了，内部类却看不见；反之亦然。这种"看起来是同一个变量、实则各改各的"是极其隐蔽的 bug 源头。

Java 的处理简单粗暴：干脆禁止修改，从根上消灭不一致。只要变量"初始化后再没被赋过值"（即"事实 final"，effectively final，哪怕你没写 final 关键字），编译器就放行捕获。

✗ Bad · 捕获后又修改
int sum = 0;
list.forEach(x -> {
  sum += x;   // 编译错误！
  // sum 被 lambda 捕获，
  // 就不许在外面再改它
});
✓ Good · 用不可变引用/可变容器
// 办法一：引用本身 final，改其内容
int[] sum = {0};
list.forEach(x -> sum[0] += x);
// sum 这个引用没变(仍指同一数组)
// 变的是数组内容 -> 合法

// 办法二(更推荐)：用 Stream 归约
int total = list.stream()
    .mapToInt(Integer::intValue).sum();
联系内存模型 & 对比其他语言："办法一"能绕过限制，是因为 final 约束的是引用不可重新指向，而不是对象内容不可变——数组引用没变，改数组元素当然合法。但要警惕：这样做等于自己捅破了 Java 替你堵上的漏洞，多线程下这个数组就成了共享可变状态，得自己加同步，否则有可见性/竞态问题。相比之下，JavaScript / Python 的闭包捕获的是"变量本身"（词法环境的引用），改了外面里面立刻能看见；Java 选择"拷贝值 + 禁止改"，是一种用更强约束换取更少并发陷阱的设计取舍。
面试高频：为什么捕获实例字段却能随便改？因为内部类捕获实例字段时，捕获的是 this$0（外层对象引用），访问 count 走的是 this$0.count——始终是"同一个对象的同一个字段"，不存在两份副本，自然没有一致性问题，所以不受 final 限制。"必须 final"只针对局部变量的捕获。这也解释了为何有人把要改的值提升为字段来"绕过 final 限制"——本质是把"值捕获"换成了"引用捕获"。

七、匿名内部类 vs Lambda：该用哪个

Java 8 的 Lambda（见上一篇）大量取代了匿名内部类的场景。但它俩不是同一种东西，理解差异才能选对：

维度匿名内部类Lambda 表达式
能实现的类型任意接口 或抽象类，方法数不限仅函数式接口（恰好一个抽象方法）
this 指向匿名对象自己外层实例（词法作用域）
能否有字段/额外方法可以不可以
编译产物独立的 $1.class 文件无额外类文件（invokedynamic 运行时生成）
变量捕获规则完全相同：都要求局部变量 final / 事实 final
取舍准则：能用 Lambda 就用 Lambda——更短、无额外 class 文件、this 语义更符合直觉。只有在这三种情况回退到匿名类：① 要实现的接口有多个抽象方法（非函数式接口）；② 需要在实现体里存状态字段或写额外辅助方法；③ 需要在方法体里用 this 指代这个实现对象自己（例如把自己注册/反注册出去）。
联系底层：Lambda 为什么不生成 class 文件？匿名类在编译期就把 $1.class 定死了，类一多，class 文件爆炸、启动变慢。Lambda 改用 invokedynamic 字节码指令，把"生成实现类"这件事推迟到运行时首次执行，由 LambdaMetafactory 动态合成。这既减少了 class 文件数量，也给 JVM 留了未来优化的空间。所以严格说：Lambda 不是"匿名类的语法糖"，它俩底层实现路径完全不同。

八、局部内部类：几乎被遗忘的一种

为了完整，提一下第四种——写在方法体内部、有名字的局部内部类。它能力和匿名类几乎一样（同样捕获事实 final 的局部变量），但因为有名字，可以被多次实例化、也能写构造器。实践中很少见，多数场景要么用匿名类要么用 Lambda 就够了。它的存在主要是语言完备性的体现。

局部内部类方法内定义、方法内使用
static Iterator<Integer> range(int n) {
    // Seq 只在 range 内部有意义，就地定义、就地封装
    class Seq implements Iterator<Integer> {
        private int i = 0;
        public boolean hasNext() { return i < n; }  // 捕获 n
        public Integer next() { return i++; }
    }
    return new Seq();
}
一句话收束：内部类的一切行为都从"那张隐形门禁卡 Outer.this"推导而来——非静态内部类持有它，故能穿透私有边界，也因此拖住外层对象不被 GC（泄漏根源）；不需要它就该加 static。匿名类是"声明+实例化二合一"的一次性实现。而闭包在 Java 里是"值拷贝式捕获"：编译器把局部变量复制进合成字段，正因为是副本，才逼着被捕获的局部变量必须 final / 事实 final，从根上杜绝"两份不一致"。Lambda 与匿名类共享这套捕获规则，但 this 语义相反、底层用 invokedynamic 实现——能用 Lambda 就用 Lambda，需要状态、多方法或自引用时才退回匿名类。
