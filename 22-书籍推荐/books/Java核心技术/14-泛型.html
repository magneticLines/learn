泛型：类型擦除、桥接方法、通配符

泛型只活在编译期——理解"编译后它就消失了"，是看懂这三个高频考点的唯一钥匙。

一、泛型到底解决了什么问题

Java 5 之前的集合只能装 Object。取东西时你必须手动强制转换，而编译器帮不上任何忙——一旦你往一个"本该装 String"的列表里塞进一个 Integer，编译一路绿灯，直到运行时取出来 cast 才轰然崩溃。这就是典型的"错误发现得太晚"。

✗ Bad · 泛型出现前（Java 1.4）
List list = new ArrayList();
list.add("hello");
list.add(42);          // 编译通过！没人拦你
String s = (String) list.get(1);
// 运行时 ClassCastException 💥
✓ Good · 有了泛型
List<String> list = new ArrayList<>();
list.add("hello");
list.add(42);          // 编译期直接报错 ✅
String s = list.get(0);
// 无需 cast，编译器替你保证类型

所以泛型的两大价值：把类型错误从运行时提前到编译期，以及消除到处都是的强制转换。记住这个本质，下面所有"奇怪"的规则都能被解释。

泛型 = 编译期 类型检查 + 自动 插入 cast ，运行时 一无所知

二、类型擦除：泛型只活在编译期

这是理解泛型的第一原理：Java 的泛型是"编译期的语法糖"。编译器做完类型检查后，会把所有类型参数擦掉，字节码里根本没有 <String> 这种信息。这个过程叫类型擦除（Type Erasure）。

Pair.java你写的 vs 编译后 JVM 眼里的
// —— 你写的（编译期视角，有类型参数 T）——
class Pair<T> {
    private T first;
    public T getFirst() { return first; }
    public void setFirst(T v) { first = v; }
}
Pair<String> p = new Pair<>();
String s = p.getFirst();

// —— 擦除后（运行时视角，T 全部变 Object）——
class Pair {
    private Object first;
    public Object getFirst() { return first; }
    public void setFirst(Object v) { first = v; }
}
Pair p = new Pair();
String s = (String) p.getFirst();  // ← 编译器悄悄替你加的 cast

擦除时"擦成什么"有明确规则，取决于类型参数有没有上界（bound）：

类型参数声明擦除后变成说明
<T>（无界）Object最常见，如 ArrayList<E> 内部是 Object[]
<T extends Comparable>Comparable擦成上界类型
<T extends Number & Comparable>Number多个上界时，擦成 第一个（故应把主类型写在最前）
为什么用"擦除"而不像 C# 那样"具体化（reification）"？一个字：兼容。Java 5 引入泛型时，世界上已有海量基于原始类型 List 的 Java 1.4 代码。擦除让泛型代码编译后与老代码"长得一模一样"，新旧可以无缝互操作（迁移兼容性），也不会像 C# 那样为每个类型参数生成一份专门的类而膨胀。代价就是——运行时丢失了泛型信息，后面所有"陷阱"都源于此。

三、擦除的连锁反应：那些"不能做"的事

既然运行时 T 已不存在，一切依赖"运行时知道 T 是什么"的操作统统失效。这些几乎每条都是面试题：

擦除导致的五条禁令每条都能追溯到"运行时没有 T"
// ① 不能用基本类型做类型参数（Object 装不下 int）
List<int> a;                 // ✗ 只能 List<Integer>，靠自动装箱

// ② 不能对带具体类型参数的泛型做 instanceof（运行时没有 T）
if (obj instanceof List<String>)  // ✗ 编译错误
if (obj instanceof List<?>)       // ✓ 只能问到"是不是 List"

// ③ 不能 new T() / new T[]（拿不到运行时类型，无从分配）
T t = new T();              // ✗
T[] arr = new T[10];       // ✗ 变通：Array.newInstance(clazz, 10)

// ④ 静态字段/静态方法不能引用类的类型参数（T 属于实例，不属于类）
class Box<T> { static T shared; }  // ✗

// ⑤ 两个"擦除后签名相同"的方法会冲突
void f(List<String> x) {}
void f(List<Integer> x) {}   // ✗ 擦除后都是 f(List)，name clash
高频陷阱：list.getClass() 骗不了你。List<String> 和 List<Integer> 在运行时是同一个类——new ArrayList<String>().getClass() == new ArrayList<Integer>().getClass() 的结果是 true。因此别指望用反射拿到"这个 List 装的是 String"；泛型信息只有在字段声明、方法签名等处以元数据形式保留（可通过 getGenericType() 读到），而不在对象实例上。

四、桥接方法：擦除与多态的和解

擦除有个"副作用"会破坏多态。看这个例子：DateInterval 继承 Pair<LocalDate> 并覆盖 setFirst：

DateInterval.java本想 @Override，擦除后却"对不上"
class DateInterval extends Pair<LocalDate> {
    @Override public void setFirst(LocalDate v) { /* 校验日期区间 */ }
}

矛盾在于：父类 Pair 擦除后只有 setFirst(Object)，而子类写的是 setFirst(LocalDate)——签名不同，从 JVM 角度这是重载而非覆盖！那么下面这段多态调用该走哪个方法？

Pair<LocalDate> p = new DateInterval(); p.setFirst(someDate); // 静态类型是 Pair，编译期绑定到 setFirst(Object) // 但动态分派要落到 DateInterval 上…… // 可 DateInterval 里只有 setFirst(LocalDate)！对不上 ✗

为了让"覆盖"名副其实，编译器会在子类里自动合成一个"桥接方法（bridge method）"——签名与父类擦除后一致，内部转调你写的那个真正方法：

$ javap DateInterval编译器悄悄多生成了一个方法
class DateInterval extends Pair {
    public void setFirst(LocalDate v);   // 你写的那个
    public void setFirst(Object v);      // ★ 桥接方法：编译器合成
    // 桥接方法的方法体等价于：
    //   void setFirst(Object v) { setFirst((LocalDate) v); }
}

这样多态调用 setFirst(Object) 时，动态分派命中子类的桥接方法，桥接方法再向下转调你写的 setFirst(LocalDate)——覆盖语义恢复了。

协变返回值也靠桥接方法实现。当子类把返回类型收窄（如 Object clone() 覆盖为 DateInterval clone()），编译器同样生成一个返回 Object 的桥接方法转调它。有趣的是：桥接方法可以出现"仅返回类型不同"的两个同名方法——这在 Java 源码里是非法的，但在字节码层面完全合法（JVM 用 名字+参数+返回类型 共同标识方法）。这也解释了 @Override 注解为何重要：它让编译器帮你确认"真的覆盖成功了"。

五、为什么 List<String> 不是 List<Object>（型变）

直觉上 String 是 Object 的子类，那 List<String> 应该也是 List<Object> 的子类吧？错。泛型是"不型变"（invariant）的。假如允许，就能撕开一个类型安全的大口子：

✗ 数组：协变，把错误留到运行时
Object[] arr = new String[3];
// ↑ 编译通过（数组是协变的）
arr[0] = 42;
// ↑ 编译也通过，但运行时
//   ArrayStoreException 💥
✓ 泛型：不型变，编译期就拦住
List<Object> lo = new ArrayList<String>();
// ↑ 编译错误！根本编不过
// 因为若允许，接下来就能
//   lo.add(42); 污染这个 String 列表
// 泛型选择把错误提前到编译期

这是 Java 设计上一个刻意的取舍：数组协变 + 运行时检查 vs 泛型不型变 + 编译期检查。泛型选了后者——更安全，但也更"死板"。而"死板"带来了新问题：一个 double sum(List<Number>) 竟然不能接收 List<Integer>！通配符就是来解开这道枷锁的。

六、通配符与 PECS 法则

通配符 ? 用来表达"某种未知但受限的类型"，恢复被"不型变"锁死的灵活性。它有三种形态：

写法含义能读 get能写 add角色
<? extends T>上界，协变✓ 返回 T✗（除 null）生产者（你从里取）
<? super T>下界，逆变只能返回 Object✓ 加 T 及其子类消费者（你往里塞）
<?>无界✓ 返回 Object✗（除 null）只读/不关心类型

为什么 ? extends T 只能读不能写？因为 List<? extends Number> 可能真身是 List<Integer>，也可能是 List<Double>——编译器不知道到底是哪个，所以禁止你 add 任何东西（塞 Double 进 Integer 列表就完了）；但无论真身是谁，取出来的东西一定是 Number，读取绝对安全。? super T 反过来：真身是 Integer / Number / Object 之一，它们都装得下 Integer，故 add Integer 安全；但取出来只能保证是 Object。

Producer-Extends ，Consumer-Super　→　PECS

这就是 Joshua Bloch 在《Effective Java》里的 PECS 法则：你从中取数据的是生产者，用 extends；你往里存数据的是消费者，用 super。JDK 源码里到处是它的身影：

java.util.CollectionsPECS 的教科书级示范
// 从 src 读（生产者→extends），往 dest 写（消费者→super）
public static <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (int i = 0; i < src.size(); i++)
        dest.set(i, src.get(i));
}

// 只从集合里挑最大值 = 纯生产者 → extends
public static <T extends Comparable<? super T>> T max(Collection<? extends T> c) { ... }
联系已学：Comparator 与 Spring 里也随处可见。比如给 List<Integer> 排序，可以用一个 Comparator<? super Integer>（如 Comparator<Number>）——因为比较器是"消费"元素的。Spring 的 ApplicationListener<? extends ApplicationEvent>、各种 Collection<? extends BeanDefinition> 参数，本质都是 PECS 在框架 API 上的应用：让调用方能传入更宽或更窄的类型，同时保持类型安全。

七、List<?>、List<Object> 与原始类型 List

这三者长得像，语义天差地别，是又一个高频辨析点：

类型能否 add(new Object())类型安全定位
List<Object>✓ 可以✓ 编译期检查"我确实是一个装 Object 的列表"
List<?>✗ 不行（除 null）✓ 编译期检查"某种未知类型的列表"，只读安全
List（原始类型）✓ 可以✗ 只有 unchecked 警告兼容老代码的历史遗留，别用
别把 List<?> 当 List 原始类型用。两者都"不挑元素类型"，但 List<?> 是类型安全的——编译器禁止你往里 add（因为不知道真实类型），从根上杜绝污染；而原始类型 List 完全绕过泛型检查，只会给你一句黄色的 unchecked 警告，把风险重新丢回运行时。能用 List<?> 表达"我不关心元素类型、只想遍历/求 size"的意图时，永远优先用它。
类比记忆：把 List<? extends Number> 想成一个"只出不进的自动售货机"——你能取出商品（一定是 Number），但投币口焊死了，塞不进任何东西。List<? super Integer> 则像"只进不出的回收箱"——你能投进 Integer（及子类），但伸手去掏，掏出来的只能当"某个 Object"看待。

八、高频面试速记

一图流背诵清单：
· 类型擦除：泛型是编译期语法糖，编译后 T 擦成 Object（有界则擦成第一个上界）；选擦除是为了兼容 Java 1.4 老代码。
· List<String> 与 List<Integer> 运行时同一个 Class，getClass() 相等。
· 不能：基本类型做参数、new T()/new T[]、静态字段用 T、对 List<String> 做 instanceof、擦除后同签名重载。
· 桥接方法：擦除破坏了覆盖语义，编译器合成签名与父类一致的方法转调，保住多态与协变返回；桥接方法可"仅返回类型不同"。
· 泛型不型变；PECS：生产者 extends（只读）、消费者 super（只写）。
· List<?> 类型安全不可写，原始类型 List 不安全——别混用。
一句话收束：Java 泛型的一切"怪脾气"都源于同一件事——它只活在编译期，运行时被擦除得干干净净。擦除换来了与老代码的兼容，代价是运行时丢失类型信息（于是有了那一串"不能做"的禁令）；擦除破坏了覆盖语义，编译器用桥接方法把多态补回来；擦除让泛型选择了"不型变"以保证编译期安全，而通配符 + PECS 又在安全的前提下把灵活性还给你。抓住"擦除"这条主线，泛型的三大考点就串成了一条逻辑链。
