类、对象、封装

把数据和操作数据的代码"锁进同一个盒子"——面向对象的第一块基石。

一、为什么要有"类"：从过程到对象

在没有类的年代（比如纯 C 语言），数据和操作数据的函数是分开的：一堆 struct Employee 散落在内存里，一堆 raiseSalary(emp, pct) 函数散落在各个文件里。谁都能直接改结构体的字段，改错了也没人拦。程序规模一大，"哪些函数会碰这块数据"就成了灾难。

面向对象的核心动机不是"模拟现实世界"这种口号，而是管理复杂度：把"一份数据"和"合法操作这份数据的行为"绑在一起，形成一个自我负责的单元——这就是类（class）。类是模板/蓝图，按模板造出来的具体实例就是对象（object）。

生活类比：类是"月饼模具"，对象是"用模具压出来的每一块月饼"。模具只有一个（描述形状），月饼可以压很多块，每块的馅料（字段值）可以不同，但形状（结构与行为）完全一致。你手里拿的、能吃的，永远是月饼（对象），不是模具（类）。
对象 = 状态(字段) + 行为(方法) + 身份(identity)

这三者缺一不可：状态是它"记住了什么"（字段/实例变量）；行为是它"能做什么"（方法）；身份是"即使两个对象状态完全一样，它们仍是两个不同的东西"（这正是后面 == 与 equals 的区别根源）。

二、类与对象在内存里长什么样

写下 Employee harry = new Employee(); 时发生了两件事，很多初学者把它们混为一谈：

  • new Employee() 在堆（heap）上真正开辟一块内存，放对象的字段，返回它的地址；
  • harry 是一个放在栈（stack）上的引用变量，里面存的只是那个地址，它本身不是对象。
栈 (Stack) 堆 (Heap) ┌────────────────┐ ┌─────────────────────────┐ │ harry ●──────┼────────>│ Employee@1a2b │ ├────────────────┤ │ name = "Harry" │ │ boss ●──────┼────┐ │ salary = 8000.0 │ └────────────────┘ │ └─────────────────────────┘ │ ┌─────────────────────────┐ └───>│ Employee@3c4d │ 引用只存"地址" │ name = "Carl" │ 对象实体都在堆里 │ salary = 9500.0 │ └─────────────────────────┘

理解这一点能帮你想通两件事：其一，Employee boss = harry; 只是复制了地址，两个变量指向同一个对象，改 boss 就是改 harry；其二，对象由垃圾回收器（GC）在"没有任何引用指向它"时回收——这与 C++ 里必须手动 delete 截然不同，也是 Java "一次编写，安心运行"的重要一环。

高频陷阱：引用 ≠ 对象。Java 里没有 C++ 那种"栈上对象"。所有用 new 造的对象都在堆上，变量永远只是"遥控器"。所以 a == b 比较的是两个遥控器是否指向同一台电视（地址相同），而不是"两台电视内容是否一样"。要比内容，得用 equals（后续章节详解）。

三、封装：把字段藏起来，只留一扇受控的门

封装（encapsulation）= 数据私有（private）+ 通过公开方法受控访问。它要解决的根本问题是：不让外界随意破坏对象的内部一致性。

Counter.java最小封装示例
public class Counter {
    private int count;            // 状态被藏起来，外界摸不到

    public void increment() {   // 只暴露"受控的行为"
        count++;
    }
    public int getCount() {   // 只读窗口，没有 setCount，count 不可被随意赋值
        return count;
    }
}

关键不在于"给字段加 private 然后机械生成 getter/setter"（那是很多人对封装的误解），而在于你有权决定哪扇门开、开多大。上面的 Counter 故意不提供 setCount，于是"计数只能一步步加、不能被人直接改成负数"这条规则被牢牢锁死。这条"对象必须始终满足的规则"，术语叫不变式（invariant）。

反例 → 正例：公有字段 vs 受控方法

✗ Bad · 裸奔的公有字段
class Account {
  public double balance;
}

// 任何人都能：
acc.balance = -99999;
// 余额变负数，没人拦得住，
// 非法状态直接进入系统
✓ Good · 用方法守住不变式
class Account {
  private double balance;

  public void deposit(double amt) {
    if (amt <= 0)
      throw new IllegalArgumentException();
    balance += amt;   // 只增不减负
  }
  public double getBalance() {
    return balance;
  }
}
封装的真正回报是"未来自由"。因为外界只依赖方法签名、看不到字段，所以有一天你把 double balance 换成 BigDecimal（避免浮点精度坑），或改成"实时从数据库查"，只要 getBalance() 签名不变，所有调用方一行都不用改。这就是"实现细节可以自由演化"——Spring 的 Bean、JDK 的 String（内部从 char[] 改成 byte[]，见 JDK 9 Compact Strings）都靠这一点悄悄升级而不惊动使用者。

四、最隐蔽的封装陷阱：访问器泄露了可变引用

你以为字段设成 private 就安全了？看这个 Core Java 反复强调的经典坑：访问器方法直接把内部可变对象的引用返回出去，等于把私有字段的钥匙拱手送人。

✗ Bad · 返回内部可变对象
class Employee {
  private Date hireDay;

  public Date getHireDay() {
    return hireDay; // 泄露引用！
  }
}

// 外部一行就击穿封装：
Date d = harry.getHireDay();
d.setTime(0); // harry 的入职日
              // 被偷偷改到 1970 年
✓ Good · 返回防御性拷贝
class Employee {
  private Date hireDay;

  public Date getHireDay() {
    return (Date) hireDay.clone();
  }
}

// 或者从根上用不可变类型：
// private final LocalDate hireDay;
// LocalDate 不可变，可直接返回
面试高频 + 实战高危：只要访问器返回的是可变对象（Date、ArrayList、数组、自定义可变类……），就必须返回防御性拷贝，否则 private 形同虚设。这也是为什么 Java 8 之后强烈推荐用不可变的 LocalDate/LocalDateTime 取代可变的 java.util.Date——不可变对象天生免疫这个问题，可以放心共享。

五、this：当前对象的自我指认

方法内部的 this 指向"调用这个方法的那个对象"。两个高频用途：消除字段与参数的同名歧义、返回自身实现链式调用（fluent API）。

Point.javathis 的两种用法
public class Point {
    private int x, y;

    public Point moveTo(int x, int y) {
        this.x = x;       // this.x 是字段，x 是参数，不加 this 就自己赋值给自己
        this.y = y;
        return this;          // 返回自身 -> 支持 p.moveTo(1,2).moveTo(3,4)
    }
}
联系底层：实例方法里的 this 并非凭空出现。字节码层面，编译器把 void moveTo(int,int) 悄悄编译成"第 0 个隐藏参数是 this"的方法（这也是 static 方法里不能用 this 的原因——它压根没有那个隐藏参数，没有"当前对象"可指）。链式调用则是 StringBuilder（append().append()）、Stream、以及无数 Builder 模式的基础套路。

六、访问修饰符：封装的"门禁级别"

Java 用四级可见性控制"谁能推开哪扇门"。字段几乎总是 private，方法按需公开；理解 default（包级私有）和 protected 的边界是常见易错点。

修饰符本类同包子类(跨包)其他典型用途
private✓✗✗✗字段、内部辅助方法（首选）
default(不写)✓✓✗✗包内协作类，对外隐藏
protected✓✓✓✗留给子类扩展的钩子
public✓✓✓✓对外 API、方法接口
易错点：不写修饰符不是 public，而是 default（包级私有）——这是初学者最常踩的默认值误解。另外 protected 的可见范围比 default 更宽（多了"跨包子类"），别被"protected 听起来更严格"的字面直觉骗了。

七、封装的边界：不要过度封装

封装是手段不是宗教。有两个反面极端要警惕：

  • "贫血 + 全套 getter/setter"：给每个字段无脑配一对 getter/setter，等于把字段又变相公开了，不变式毫无保护——这和公有字段几乎没区别，只是多敲了代码。
  • 纯数据载体过度封装：像 DTO、配置对象这种"只是搬运数据、没有不变式要守"的类，Java 16+ 的 record 才是更好的答案，一行搞定且天生不可变。
判断准则：写 setter 前先问自己——"这个字段被外界随意改写，会破坏对象的某条规则吗？"如果会，就别给 setter，或在 setter 里加校验；如果这个类根本没有任何规则要守（就是个数据袋），考虑用 record。封装的目的是保护不变式，不是"凑齐 get/set"。
一句话收束：类是模板、对象是堆上的实例、变量只是指向它的引用；封装的本质是用 private 藏起状态、只经由受控方法暴露行为，从而守住对象的不变式、并把实现细节的修改权留给自己。真正的封装不是机械生成 getter/setter，而是"想清楚哪扇门该开、返回可变对象时记得防御性拷贝"。
