抽象类、接口、sealed 类

从"强制子类补全"到"纯契约"，再到 Java 17 的"受控继承"——抽象能力的三种刻度。

一、抽象到底在解决什么问题

写业务代码时你会不断遇到一类东西："我知道它必须能做某件事，但此刻我说不清、也不该说清它具体怎么做。"比如"所有员工都能算工资"，但经理和小时工的算法完全不同；"所有图形都能算面积"，但圆和矩形的公式毫不相干。

如果硬要在父类里写一个"通用实现"，你要么写出一堆 if (是圆) ... else if (是矩形) ... 的丑陋分支，要么随便返回个 0 当占位符——而占位符最大的危险是它是合法的，子类忘了覆盖也不会报错，Bug 就这样悄悄溜进生产环境。抽象机制的核心价值就是：把"必须有、但现在给不出"的行为，变成一个编译期强制约束。

abstract 类 = 部分实现 + 强制补全  ·  interface = 纯契约  ·  sealed = 受控的继承边界
生活类比：抽象类像一份"填空题试卷"——大部分内容（表头、题干、评分逻辑）都印好了，只留几个空格强制你填（抽象方法）。接口像一份"岗位职责说明书"——只规定"你必须能干这些活"，一个字都不告诉你怎么干。sealed 则像"内部会议的受邀名单"——不是谁想来就能来，主人明确列出了"只允许这几位"。

二、抽象类：给我一半，逼你补另一半

抽象类用 abstract 修饰，它不能被 new 实例化（因为它本就"不完整"）。它可以有构造器、字段、普通方法，也可以有只声明不实现的抽象方法。只要一个类含有抽象方法，这个类就必须是抽象类；反过来，抽象类里可以一个抽象方法都没有。

Shape.java抽象类 + 模板方法模式
public abstract class Shape {
    private final String name;

    protected Shape(String name) {   // 抽象类可以有构造器：给子类共享字段用
        this.name = name;
    }

    public abstract double area();        // 抽象方法：只有签名，怎么算交给子类

    // 模板方法：复用逻辑写在父类，可变部分回调抽象方法（这就是"骨架已定，细节留空"）
    public final String describe() {
        return name + " 面积=" + area();  // 调用尚未实现的 area()，运行时按实际子类分派
    }
}

public class Circle extends Shape {
    private final double r;
    public Circle(double r) { super("圆"); this.r = r; }
    @Override public double area() { return Math.PI * r * r; }
}
这就是"模板方法模式"（Template Method）。describe() 把稳定的流程骨架定死在父类（还加了 final 禁止子类篡改流程），只把会变的一步（area()）留给子类。JDK 里 AbstractList、InputStream，Spring 里 AbstractApplicationContext.refresh() 全是这个套路——框架定流程，你填业务。它的底层依托正是上一节讲的多态：父类里调 area()，运行时会分派到 Circle 的实现。

三、接口：不谈实现，只立契约

接口（interface）是更彻底的抽象：它不关心"你是什么"，只关心"你能做什么"。一个类可以 implements 任意多个接口，但只能 extends 一个类——这是 Java "单继承类、多实现接口"的根本设计。

几个初学者常忽略的隐式规则：接口里的字段默认就是 public static final（是常量，不是实例字段）；方法在 Java 8 前默认是 public abstract。所以下面两种写法完全等价：

接口的隐式修饰符写与不写等价
public interface Comparable<T> {
    int MAX = 100;              // 实为 public static final int MAX
    int compareTo(T o);            // 实为 public abstract int compareTo(T)
}

为什么要"面向接口编程"？因为接口把"能力"和"实现"解耦。你的代码依赖 List 这个接口，而不是 ArrayList 这个具体类，将来想换成 LinkedList 或 CopyOnWriteArrayList，调用方一行都不用改。这正是 Spring 依赖注入的哲学基础——注入的字段类型永远写接口（UserRepository），运行时才决定塞哪个实现（JDK 动态代理生成的实现类）。

Java 8+ 的三次进化：接口不再"纯"了

  • Java 8 · default 方法：接口可以带默认实现。动机是向后兼容——Collection 想加 stream()，但全世界几百万个已有实现类不可能一夜之间都改，于是给它一个默认实现，老代码不改也能跑。
  • Java 8 · static 方法：把工厂/工具方法收进接口自身，如 Comparator.comparing(...)。
  • Java 9 · private 方法：给多个 default 方法抽取公共逻辑，又不想暴露出去。
陷阱：default 方法不是"接口有状态了"。接口永远不能有实例字段（那个 int MAX 是静态常量，不是每个对象一份）。default 方法只能操作参数和调用其它接口方法，不能读写"对象的字段"——因为接口根本不知道实现类会有哪些字段。把接口当"带状态的抽象类"用，是对 default 方法最常见的误解。

四、抽象类 vs 接口：到底选哪个

自从接口有了 default 方法，二者边界变模糊，但设计意图截然不同。一句话：抽象类回答"我是什么"（is-a，共享状态与骨架），接口回答"我能做什么"（can-do，能力契约）。

数量限制只能继承 1 个可实现 多个
实例字段（状态）✓ 可有，可被子类共享✗ 只能有 static final 常量

经验法则：优先用接口。接口更灵活（可多实现、不占用唯一的继承名额），耦合更松。只有当你需要在多个子类间共享字段/构造逻辑，或要做模板方法时，才动用抽象类。很多成熟框架采用"接口 + 骨架抽象类"的组合拳：接口 List 定契约，抽象类 AbstractList 帮你实现大部分方法，你继承它只需填几个空——鱼和熊掌兼得。

五、多接口的"菱形冲突"与消歧规则

C++ 的多重继承因"菱形问题"（同一祖先的状态被继承两份）而声名狼藉。Java 用"只单继承类"绕开了状态冲突，但 default 方法带来了行为冲突：两个接口给了同名 default 方法，实现类该听谁的？

interface A interface B default greet() default greet() \ / \ / \ / +------ class C ----+ greet() ?? <- 编译器：说不清，报错！ 你必须显式 @Override 消歧

Java 的裁决顺序有明确规则，记住这三条优先级：

  1. 类优先于接口：父类的具体方法总是压过接口的 default 方法（"具体的赢过默认的"）。
  2. 更具体的子接口优先：若 B extends A 且都定义了 greet()，B 的赢。
  3. 否则编译报错，必须自己 @Override 手动选边，用 接口名.super.方法() 指定。
DiamondDemo.java用 A.super 显式消歧
interface Walk { default String move() { return "走"; } }
interface Swim { default String move() { return "游"; } }

class Duck implements Walk, Swim {
    @Override public String move() {          // 不写就编译不过
        return Walk.super.move() + "也能" + Swim.super.move();
    }
}

六、sealed 类：给继承划一条明确的边界（Java 17+）

问题来了：public 类和接口是"敞开大门"的——任何人、任何模块都能继承/实现它。但很多时候你希望继承是封闭的：一个 Shape 就应该只有 Circle / Rectangle / Triangle 这三种，绝不允许别人偷偷加个 Blob。以前你只能靠"把构造器设 private + 文档写注释"这种脆弱手段。

Java 17 正式落地的 sealed（JEP 409）让这个约束写进语言、由编译器强制执行。密封类用 permits 明确列出"允许的子类白名单"：

Shape.javasealed 接口 + record 实现
public sealed interface Shape
        permits Circle, Rectangle, Triangle { }

// 三个允许的子类型，每个都必须声明"封口方式"：
public record Circle(double r)             implements Shape {}  // record 隐式 final
public record Rectangle(double w, double h) implements Shape {}
public record Triangle(double b, double h)  implements Shape {}
三条硬规则（编译器会逐条检查）：
① 每个 permits 列出的子类，必须显式声明为以下三者之一：final（到此为止，不再有子类）、sealed（继续密封，再列白名单）、或 non-sealed（在这里"开封"，恢复成普通可任意继承）。
② 子类必须与密封类在同一模块（无模块时须同一包）——防止外人从别的 jar 里塞进来。
③ record 天生 final，所以做密封子类最省事，一个字都不用加。

七、sealed 的杀手锏：让 switch 变"穷尽"

光是"限制继承"还不足以让 sealed 火起来。它真正的威力，是和 switch 模式匹配（Java 21 JEP 441 转正）联手，实现代数数据类型（ADT）式的"穷尽性检查"。

编译器知道 Shape 只有三种可能，于是你的 switch 只要覆盖了这三种，就不需要写 default 分支；反过来，哪天你给 Shape 加了第四种 Pentagon，所有没更新的 switch 会当场编译报错，逼你去补——漏处理分支的 Bug 在编译期就被逮住了。

✗ Bad · instanceof 长链
double area(Shape s) {
  if (s instanceof Circle c)
    return Math.PI*c.r()*c.r();
  else if (s instanceof Rectangle r)
    return r.w()*r.h();
  // 忘了 Triangle... 编译器不管，
  // 运行到三角形直接抛异常 or 返回错值
  throw new IllegalStateException();
}
✓ Good · sealed + switch 穷尽
double area(Shape s) {
  return switch (s) {        // 无 default
    case Circle c    -> Math.PI*c.r()*c.r();
    case Rectangle r -> r.w()*r.h();
    case Triangle t  -> 0.5*t.b()*t.h();
    // 少写一种 -> 编译期直接报错
  };
}
为什么这是"函数式的胜利"。sealed + record + 模式匹配 = Java 版的"代数数据类型"（类似 Kotlin 的 sealed class、Rust 的 enum、Scala 的 sealed trait）。它让"一个类型只有有限几种形态"这件事被类型系统知晓并强制，把大量运行时错误提前到编译期。这也是近年 Java 向"数据导向编程"演进的关键拼图。

八、抽象三兄弟的能力刻度

把三者放到"继承开放程度"这条轴上看，脉络就清晰了——从完全敞开到彻底封死：

开放继承 <──────────────────────────────────────> 禁止继承 普通 class abstract class sealed class final class 任何人可继承 任何人可继承 仅白名单可继承 谁都不能继承 (默认) (且必须补全空缺) (permits 列出) (String/record)
易错点合集：
· 抽象类不能 new，但可以有构造器（供子类 super() 调用）——两件事不矛盾。
· 接口里写的 int X = 1; 是常量不是字段，别当成"接口有状态"。
· sealed 的子类必须三选一标注 final/sealed/non-sealed，漏标编译不过；record 自带 final 可省。
· sealed 是 Java 17 转正（15/16 为预览），面试问"哪个版本"别答错。
· 两个接口有同名 default 方法时，实现类必须手动 @Override 消歧，编译器不会替你选。
一句话收束：抽象类是"给一半强制补另一半"，靠共享状态和模板方法为 is-a 服务；接口是"只立契约不谈实现"，靠多实现和 default 方法为 can-do 服务，优先选它；sealed 则给继承划了一条编译器强制的白名单边界，与 record + switch 模式匹配组队，把"类型只有有限几种"这件事变成编译期可穷尽的保证——这正是 Java 17+ 迈向数据导向编程的核心一步。
