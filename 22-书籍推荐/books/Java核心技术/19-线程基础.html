线程基础：Thread、Runnable、线程状态

从"一个 main 到底"到"多条执行流并行"——先认清线程是什么、怎么起、活在哪几种状态里。

一、为什么需要线程：从进程说起

你写的每个 Java 程序，启动时操作系统都会为它开一个进程（process）——一块独立的地址空间、一份文件句柄表、一套资源账本。但进程内部真正"跑代码"的执行单元是线程（thread）：程序计数器 PC、一个调用栈、寄存器现场，这些才是线程私有的。一个进程默认就有一条线程（跑 main 方法的那条），需要"同时做几件事"时再开更多线程。

为什么不多开几个进程反而要在一个进程里开多线程？因为同进程的线程共享堆内存和方法区，通信只需读写同一个对象，几乎零成本；而进程间通信要走管道、共享内存、socket，昂贵得多。代价是：共享带来了"数据竞争"，这正是后面 synchronized、volatile 要解决的问题。

生活类比：进程像一家公司（有自己的办公楼、账户、资产），线程像公司里的员工。同一家公司的员工共用一间办公室、一台打印机（共享内存），协作方便；但两人同时改同一份文档就会冲突（数据竞争）。不同公司之间要合作，只能发邮件、走合同（进程间通信），慢且麻烦。
进程 = 资源分配的基本单位  ·  线程 = CPU 调度的基本单位

二、Java 里怎么"起"一条线程：三种写法

Java 把"一段要被独立执行的逻辑"抽象成 Runnable 接口（只有一个 run() 方法），把"一条真正的执行流"抽象成 Thread 类。有意思的是——Thread 自己就实现了 Runnable，所以它既是"执行流"又可以是"任务"。于是有三种起线程的写法：

ThreeWays.java继承 / 实现 / lambda
// 方式一：继承 Thread，重写 run（不推荐，理由见下一节）
class MyThread extends Thread {
    @Override public void run() {
        System.out.println("run on: " + getName());
    }
}

// 方式二：实现 Runnable，把"任务"和"线程"解耦（推荐）
class MyTask implements Runnable {
    @Override public void run() {
        System.out.println("run on: " + Thread.currentThread().getName());
    }
}

public class ThreeWays {
    public static void main(String[] args) {
        new MyThread().start();                       // 继承方式
        new Thread(new MyTask()).start();            // Runnable 方式
        new Thread(() -> System.out.println("lambda task")).start();
        // Runnable 是函数式接口(单抽象方法)，所以能直接写成 lambda
    }
}
联系已学：还记得 lambda 那一节说的"函数式接口"吗？Runnable 只有一个抽象方法 run()，正是天生的函数式接口（源码上标了 @FunctionalInterface），所以 () -> {...} 能无缝当成一个 Runnable 传给 Thread 构造器。这也是为什么第三种写法最简洁。

三、继承 Thread vs 实现 Runnable：为什么首选后者

两种写法都能跑，但工程上几乎一致推荐 Runnable。核心原因是"任务(what to do)"和"执行机制(how to run)"应该分离——继承 Thread 把这两者焊死在一个类里，既浪费了 Java 唯一的单继承名额，又让任务无法交给线程池复用。

✗ Bad · extends Thread
class Job extends Thread {
  public void run() { ... }
}
// 1. 用光了唯一的继承名额
//    (以后想 extends 别的类没门)
// 2. 任务与线程绑死，无法复用
// 3. 不能直接交给线程池
new Job().start();
✓ Good · implements Runnable
class Job implements Runnable {
  public void run() { ... }
}
// 1. 还能再 extends 别的基类
// 2. 任务(what)与线程(how)解耦
// 3. 可直接丢进线程池执行
Runnable job = new Job();
pool.submit(job);   // 复用线程
维度extends Threadimplements Runnable
继承限制占用单继承名额不占，还能实现多接口
任务复用每次都要 new 一个 Thread 子类同一 Runnable 可交给多个线程/线程池
关注点分离任务和线程耦合任务是纯逻辑，线程是执行器
能否配合线程池不方便天然契合 ExecutorService
有返回值/抛异常不支持升级用 Callable（后续章节）
翻 JDK 源码看清关系：public class Thread implements Runnable。Thread 内部持有一个成员 private Runnable target;。它的 run() 默认实现是 if (target != null) target.run();——所以当你传入 Runnable 时，线程跑的是你的 target.run()；当你继承并重写 run() 时，跑的是你重写的版本。这就是两种写法在底层"殊途同归"的原因。

四、最经典的坑：start() 不是 run()

初学者第一个必踩的坑：想启动线程却调了 run()。结果程序照常输出，但根本没有新线程产生——因为 run() 只是个普通方法，直接调用就是在当前线程里同步执行。只有 start() 才会请求 JVM 去向操作系统申请一条新线程，再由那条新线程回调你的 run()。

✗ Bad · 直接调 run()
Thread t = new Thread(task);
t.run();   // 只是普通方法调用
// 在 main 线程里同步执行，
// 没有任何新线程被创建

// task 里打印当前线程名 ->
//   "main"
✓ Good · 调 start()
Thread t = new Thread(task);
t.start(); // JVM 创建新线程,
// 由新线程异步回调 run()

// task 里打印当前线程名 ->
//   "Thread-0"
t.start(); // 再调一次 -> 抛异常
 IllegalThreadStateException
三个高频陷阱一次记牢：
① run() ≠ 启动线程：直接调 run() 是同步执行，不会并发，也享受不到多核。
② start() 只能调一次：一个 Thread 对象生命周期结束（TERMINATED）后不能"复活"，再 start() 会抛 IllegalThreadStateException。要重复执行请用线程池。
③ 线程执行顺序不确定：start() 只是让线程"就绪"，何时真正跑由 OS 调度器决定，别假设"先 start 的先跑"。

五、Java 线程与操作系统线程的关系

你 new 出来的 Thread 对象只是 Java 层的"句柄"，调用 start() 时，HotSpot JVM 会通过系统调用（Linux 上是 pthread_create）向内核申请一条真正的内核线程，两者一对一（1:1）绑定。这就是为什么线程不是免费的：每条线程默认要预留约 1MB 的栈空间，且创建/销毁、上下文切换都要陷入内核，开销可观——这也正是"池化线程"的动机。

联系前沿：正因为"1:1 + 每线程 1MB 栈"限制了并发规模（几千条就吃紧），JDK 21 正式引入了虚拟线程（Virtual Threads，Project Loom）：由 JVM 在少量内核线程上"多路复用"海量虚拟线程（M:N 模型），单机可轻松跑百万级。但它的编程模型仍然是本节的 Thread/Runnable——先把基础打牢，虚拟线程只是执行载体的升级。

六、线程的六种状态（Thread.State）

Java 用枚举 Thread.State 定义了六种状态，可通过 t.getState() 拿到。要特别注意：Java 的 RUNNABLE 把操作系统层面的"就绪(ready)"和"运行(running)"合并成了一个状态——因为在 Java 视角里"能被调度"和"正在被调度"没必要区分，真正在哪个 CPU 上跑是 OS 的事。

NEW已创建，还没 startnew Thread(...) 之后
RUNNABLE就绪或正在运行（含等 I/O）调用 start() 后
BLOCKED等待获取 synchronized 监视器锁进不去被别的线程占着的同步块
WAITING无限期等待，被唤醒才走wait() / join() / LockSupport.park()
TIMED_WAITING限时等待，超时自动醒sleep(t) / wait(t) / join(t)

面试高频辨析：BLOCKED vs WAITING 不是一回事。
· BLOCKED：被动地卡在"抢 synchronized 锁"这一步，一旦锁释放就自动去竞争，无需谁来唤醒。
· WAITING：主动调用了 wait()/join() 交出执行权，必须等 notify()/notifyAll() 或目标线程结束来唤醒，否则永远醒不过来。
还有个易错点：线程等待 I/O（如读 socket）时状态仍是 RUNNABLE，不是任何 waiting——因为在 JVM 看来它只是"没拿到 CPU 时间片"，阻塞在内核里而已。

七、sleep / wait / yield / join：别用混

这几个方法都能让线程"暂停"，但归属、是否释放锁、进入的状态完全不同，是并发面试的重灾区。关键分水岭是"会不会释放已持有的锁"：

方法所属释放锁?进入状态怎么醒
Thread.sleep(t)Thread 静态否（抱着锁睡）TIMED_WAITING睡够 t 毫秒
obj.wait()Object 实例是（放锁）WAITINGnotify/notifyAll
obj.wait(t)Object 实例是TIMED_WAITING被唤醒或超时
Thread.yield()Thread 静态否仍 RUNNABLE只是"让一让"，可能立刻又跑
t.join()Thread 实例底层是 waitWAITING目标线程 t 终止
为什么 wait/notify 定义在 Object 而不是 Thread？因为 Java 的锁（监视器 monitor）是绑在每个对象上的，wait/notify 操作的是"某个对象上的等待队列"，而不是某条线程。任何对象都能当锁，所以这套方法只能放在所有类的祖先 Object 里。这也解释了下一条铁律——它们必须在 synchronized 块内调用，否则你根本没持有那个对象的锁。

下面用"抱着锁睡"的反例说明 sleep 与 wait 在协作时的天壤之别：

✗ Bad · 在同步块里 sleep
synchronized (lock) {
  while (!ready) {
    Thread.sleep(1000); // 抱着锁睡!
    // 别的线程进不来改 ready,
    // 只能干等 -> 可能死等
  }
}
✓ Good · wait 让出锁
synchronized (lock) {
  while (!ready) {
    lock.wait();  // 放锁并挂起
    // 别的线程能进来改 ready
    // 并 lock.notifyAll() 唤醒我
  }
}
// 注意: 用 while 而非 if 防"虚假唤醒"
再点两个坑：
① 停线程别用 stop()/suspend()：这些方法已废弃，会在任意点强杀线程导致锁不释放、状态被破坏。正确姿势是协作式中断：调 t.interrupt() 设置中断标志，线程内部轮询 Thread.currentThread().isInterrupted() 或捕获 InterruptedException 自行收尾退出。
② wait/notify 唤醒后要用 while 复查条件，不能用 if——因为存在"虚假唤醒(spurious wakeup)"，且被 notifyAll 唤醒的多个线程里可能只有一个该继续。

八、守护线程：一个容易被忽略的开关

线程分两类：用户线程和守护线程（daemon）。JVM 的退出规则是——只要还有一条用户线程活着，JVM 就不退出；当最后一条用户线程结束，JVM 直接退出，不管守护线程还在不在跑。垃圾回收线程就是典型的守护线程，它默默兜底，没必要为它拖住整个程序退出。

DaemonDemo.javasetDaemon 必须在 start 前调
Thread gc = new Thread(() -> {
    while (true) { /* 后台清理，永不主动结束 */ }
});
gc.setDaemon(true);   // 标记为守护线程；放到 start() 之后调会抛 IllegalThreadStateException
gc.start();
// main(用户线程)结束后，即使 gc 还在死循环，JVM 也会退出
一句话收束：Runnable 描述"要干的活"，Thread 描述"干活的执行流"，二者优先用组合（implements Runnable）而非继承。启动线程只认 start()（run() 是同步调用的假动作，且 start() 只能调一次）。Java 平台线程与内核线程 1:1 绑定，所以它不便宜，为后面的线程池埋下伏笔。生命周期就 NEW → RUNNABLE →（BLOCKED / WAITING / TIMED_WAITING）→ TERMINATED 六态；分清"抢锁被动卡住的 BLOCKED"和"主动交权等唤醒的 WAITING"，记牢"sleep 抱锁、wait 放锁"。基础立稳，下一节的 synchronized 与 volatile 才好谈"怎么让共享数据别出错"。
