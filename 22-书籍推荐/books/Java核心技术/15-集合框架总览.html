集合框架总览：Collection 与 Map 体系

一张接口地图，看清 List / Set / Queue / Map 各归其位——先建立骨架，再谈实现取舍。

一、为什么需要"集合框架"这套东西

数组（int[]、String[]）是 Java 与生俱来的容器，但它有两条硬伤：长度一旦创建就固定，和只提供最原始的"按下标读写"。现实里你要的往往是"能自动扩容的列表""能自动去重的集合""能按 key 查值的字典""先进先出的排队"——如果每个项目都各写一套，世界上就会有一万种互不兼容的 MyList。

Java 集合框架（Java Collections Framework，JDK 1.2 引入，主体在 java.util 包）的价值，是把这些数据结构抽象成一组统一的接口，再给出多种实现。你面向 List 接口编程，底层换 ArrayList 还是 LinkedList，调用方一行都不用改——这正是上一章"泛型"和之前"面向接口编程"思想的集中兑现。

生活类比：集合框架像一套标准化的收纳系统。Collection 是"一堆东西"的总称；List 是带编号的书架（有顺序、可重复、能按第几格取）；Set 是集邮册（同一张邮票不收第二遍）；Queue 是排队的队伍（讲究谁先谁后）；Map 则是字典（用词条 key 查释义 value）。它们共享同一套"收纳规范"，所以你换一种收纳盒，取放东西的手法不变。

二、两条主线：Collection 与 Map 为何分家

初学者最容易问的一句话是："Map 为什么不属于 Collection？"答案在于它们建模的东西根本不同：Collection 装的是"一个个元素"，Map 装的是"一对对键值映射"。让 Map 去实现 Collection<E> 里的 add(E)、iterator() 语义会非常别扭——一个"键值对"到底算一个元素还是两个？于是设计者把它们拆成两棵独立的接口树。

Collection<E> = 元素的集合  ·  Map<K,V> = 键到值的映射
Iterable<E> (根接口：一切能被 for-each 遍历的东西) │ Collection<E> Map<K,V> (独立的另一棵树) ┌────────┼─────────┐ ┌─────┴──────┐ List Set Queue HashMap SortedMap │ ┌─┴──┐ │ LinkedHashMap │ ArrayList Set SortedSet Deque TreeMap ◄────────┘ LinkedList │ │ Hashtable(遗老) ... HashSet ArrayDeque TreeSet LinkedList LinkedHashSet PriorityQueue

注意一个关键事实：Collection 继承自 Iterable，所以任何 Collection 都能用 for-each 遍历；而 Map 不继承 Iterable，你不能直接 for (x : map)，得先取出它的 entrySet() / keySet() / values()（这三个"视图"才是 Collection）再遍历。

三、Collection 三大子接口的语义分工

同样是"一堆元素"，为什么还要分 List / Set / Queue？因为它们对"顺序"和"重复"的承诺不同——接口本身就是一份语义契约。

接口有序?可重复?核心语义典型实现
List有索引顺序✓ 可重复按位置访问的序列ArrayList / LinkedList
Set一般无序*✗ 不重复数学意义的"集合"HashSet / TreeSet / LinkedHashSet
Queue按处理顺序✓ 可重复先进先出/优先级排队ArrayDeque / PriorityQueue

* Set 是否"有序"取决于实现：HashSet 无序、LinkedHashSet 保插入序、TreeSet 保排序序——这也说明"接口定契约、实现定策略"的分层威力。

去重的本质：Set 靠 equals() + hashCode() 判重。HashSet 用 hashCode() 定位桶、再用 equals() 确认相等；TreeSet 则完全靠 compareTo()/Comparator 的"比较为 0 即相等"。这就是为什么把自定义对象放进 Set 前，必须正确重写 equals/hashCode——否则两个"内容相同"的对象会被当成两个，去重失效（这一点在讲 Object 类那节埋过伏笔）。

四、一段代码看清"面向接口编程"的红利

集合框架最实用的习惯：声明用接口，new 用实现。变量类型写成 List/Map，让实现类只出现在 new 那一刻，将来替换实现的成本降到最低。

CollectionOverview.java四条主线各来一例
import java.util.*;

public class CollectionOverview {
    public static void main(String[] args) {
        // List：有序可重复，声明用接口 List，实现随时可换
        List<String> names = new ArrayList<>();
        names.add("Tom"); names.add("Tom");   // 允许重复 -> [Tom, Tom]

        // Set：自动去重（HashSet 底层就是一个 HashMap）
        Set<String> uniq = new HashSet<>(names);  // -> [Tom] 只剩一个

        // Queue：Deque 双端队列，offer 入队 / poll 出队（FIFO）
        Deque<Integer> q = new ArrayDeque<>();
        q.offer(1); q.offer(2);
        Integer head = q.poll();          // -> 1（先进先出）

        // Map：键值映射，不属于 Collection，需通过视图遍历
        Map<String, Integer> age = new HashMap<>();
        age.put("Tom", 18);
        for (Map.Entry<String, Integer> e : age.entrySet())  // entrySet 才是 Collection
            System.out.println(e.getKey() + "=" + e.getValue());
    }
}
一个反直觉但重要的事实：HashSet 内部就是一个 HashMap。翻开 JDK 源码你会看到 HashSet 里藏着 private transient HashMap<E,Object> map;，每个元素被当作 key 存进去，value 是一个共享的常量 PRESENT。所以"集合去重"复用的正是"字典 key 唯一"的能力——Map 才是集合框架里最底层的基础设施之一。

五、Map 家族：无序、保序、有序三选一

Map 这棵树上最常用的三兄弟，差别不在"能不能存"，而在遍历时 key 以什么顺序出现，这背后是完全不同的数据结构：

实现底层结构key 顺序get/put 复杂度额外能力
HashMap数组+链表/红黑树无序（哈希打散）O(1) 均摊最快，允许 1 个 null key
LinkedHashMapHashMap+双向链表插入序（或访问序）O(1) 均摊可做 LRU 缓存
TreeMap红黑树按 key 排序O(log n)范围查询 subMap 等
联系底层：JDK 8 起 HashMap 的桶"链表转红黑树"。当单个桶内链表长度 ≥ 8 且表容量 ≥ 64 时，链表会树化成红黑树，把最坏情况下的查找从 O(n) 降到 O(log n)，防御哈希碰撞攻击。这是"数据结构随规模自适应切换"的经典工程案例，也是高频面试题——下一节我们会展开 HashMap 的扩容与树化细节。

六、易踩的坑：接口选错 / 遍历删除 / null 假设

四个高频陷阱：
① 把 Map 当 Collection 遍历：Map 不实现 Iterable，不能直接 for-each，要经 entrySet()（比先 keySet() 再 get() 少一次哈希查找，更高效）。
② foreach 中直接 remove：会触发 ConcurrentModificationException（fail-fast 机制），必须用 Iterator.remove() 或 removeIf()。
③ 误以为 Set/Map 有序：HashSet/HashMap 的遍历顺序不保证、且可能随版本变化，别依赖它；要顺序请选 LinkedHashXxx 或 TreeXxx。
④ 对 null 的假设不一致：HashMap 允许 null key/value，但 TreeMap（要比较）和 ConcurrentHashMap（并发歧义）都不允许 null key，塞进去会抛 NullPointerException。

其中"遍历删除"最坑，务必用迭代器的删除口子，而不是集合本身的：

✗ Bad · 遍历中直接删
List<String> list = ...;
for (String s : list) {
  if (s.isEmpty())
    list.remove(s);   // modCount 变了
}
// -> ConcurrentModificationException
//    (fail-fast 在下次迭代时爆)
✓ Good · Iterator / removeIf
Iterator<String> it = list.iterator();
while (it.hasNext()) {
  if (it.next().isEmpty())
    it.remove();       // 同步 modCount，安全
}
// 或一行搞定：
list.removeIf(String::isEmpty);
fail-fast 原理：集合内部维护一个 modCount（结构性修改计数器）。Iterator 创建时记下 expectedModCount，每次 next() 都校验二者是否相等；你绕过迭代器直接 list.remove() 会让 modCount 变化而 expectedModCount 没变，于是抛异常。这是一种"尽早暴露 bug"的防御设计，不是线程安全保证——多线程下即使用迭代器删也可能出错，那要用并发集合。

七、并发场景：别再用 Vector / Hashtable

早期的 Vector、Hashtable（JDK 1.0 遗留）给每个方法都加了 synchronized，看似线程安全，实则粒度太粗、性能差，且"单个方法安全"不等于"复合操作安全"。现代做法是用 java.util.concurrent 里的专用并发容器。

✗ Bad · 遗老容器
Map<String,Integer> m
    = new Hashtable<>();
// 全表锁，读也要抢锁，高并发瓶颈
// 复合操作仍需自己加锁：
if (!m.containsKey(k))  // 两步之间
  m.put(k, 0);          // 可能被插队
✓ Good · 并发容器
Map<String,Integer> m
    = new ConcurrentHashMap<>();
// 分段/CAS，读几乎无锁，吞吐高
m.putIfAbsent(k, 0);    // 复合操作原子化
m.merge(k, 1, Integer::sum); // 计数
选型速记：并发字典用 ConcurrentHashMap；读多写极少可用 CopyOnWriteArrayList；线程间传递任务用 BlockingQueue（如 ArrayBlockingQueue），线程池的任务队列正是它。这些留待并发章节深入，此处只需记住：需要线程安全时，去 java.util.concurrent 找专用实现，而非给普通集合套锁。

一句话收束：Java 集合框架是一棵"接口定契约、实现定策略"的两棵树——Collection（继承 Iterable）管"一堆元素"，下分List(有序可重复)/Set(去重)/Queue(排队)；Map 独立成树管"键值映射"。记住三条主心骨：声明用接口、new 用实现换起来零成本；去重和字典都建立在 equals/hashCode 之上；需要顺序找 Linked/Tree、需要并发找 concurrent 包。骨架立住了，下一节的 ArrayList vs LinkedList、HashMap vs TreeMap 才是往这张地图里填性能细节。
