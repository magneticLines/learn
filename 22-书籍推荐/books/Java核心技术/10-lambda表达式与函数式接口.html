lambda 表达式与函数式接口

让"一段行为"像数据一样被传递——Java 8 把函数式编程塞进面向对象语言的那把钥匙。

一、它要解决的真问题：传"行为"太啰嗦

在 Java 8 之前，Java 只能传"对象"和"数据"，不能直接传"一段代码/行为"。可现实里我们经常想说："给你一个列表，请按我给的规则排序""线程启动后，请执行我这段逻辑"。规则和逻辑本身就是行为。旧办法只能把行为包进一个匿名内部类的对象里传过去，样板代码淹没了真正想表达的那一行。

✗ Bad · 匿名内部类的样板税
list.sort(new Comparator<String>() {
  @Override
  public int compare(String a, String b) {
    return a.length() - b.length();
  }
});
// 5 行里只有 1 行是我真正关心的逻辑
// 其余全是"仪式感"噪音
✓ Good · lambda 直击本质
list.sort((a, b) -> a.length() - b.length());

// 只保留"输入 -> 怎么算"这一句
// 甚至能用方法引用再瘦身:
list.sort(Comparator.comparingInt(String::length));
核心思想叫"行为参数化"（behavior parameterization）。方法的参数过去只能接收"数据"，现在能接收"一段代码"。sort 负责通用的排序骨架，把"两个元素谁大"这个会变的部分抽成参数交给调用方——这正是策略模式的极致轻量化，也是 Stream、事件回调、Spring 函数式路由的共同地基。

二、lambda 的语法：三段式与省略规则

一个 lambda 表达式本质就是"参数列表 + 箭头 + 函数体"三部分。Java 在能推断的地方大量允许省略，所以它有很多变体，但都是同一个骨架：

lambda 的各种形态从完整到极简
// 完整形态：显式类型 + 花括号 + return
(int x, int y) -> { return x + y; }

// 省略参数类型（由目标类型推断）+ 单表达式省略 {} 和 return
(x, y) -> x + y

// 恰好一个参数：连括号都能省
s -> s.isEmpty()

// 零参数：必须保留空括号
() -> System.out.println("run!")

// Java 11+ 可用 var（一般用于配合注解）
(var x, var y) -> x + y
陷阱：lambda 没有"独立类型"。你不能写 var f = (x, y) -> x + y;——编译器无法凭空知道这是 Comparator 还是别的什么。lambda 的类型来自它被赋值/传参时的目标类型（target type）：Comparator<Integer> c = (x, y) -> x + y; 才行。同一段 () -> 42，赋给 Supplier<Integer> 和赋给你自定义的接口，是两种完全不同的类型。

三、函数式接口：lambda 的"落脚点"

lambda 不是凭空存在的，它必须被赋给一个函数式接口（Functional Interface）——即"有且仅有一个抽象方法"的接口，业界简称 SAM（Single Abstract Method）。编译器把 lambda 的参数、返回值对上那个唯一抽象方法的签名，lambda 就成了这个接口的一个实现实例。

函数式接口 = 恰好 1 个抽象方法 (SAM)
@FunctionalInterface让编译器替你把关
@FunctionalInterface              // 非必需，但强烈建议加
interface Calculator {
    int apply(int a, int b);   // 唯一抽象方法

    default int twice(int a) {       // default 方法不算，可以有多个
        return apply(a, a);
    }
    static Calculator plus() {          // static 方法也不算
        return (a, b) -> a + b;
    }
}

Calculator add = (a, b) -> a + b;  // lambda 对上 apply(int,int)
System.out.println(add.twice(5));       // 10，default 方法照常可用
为什么 default / static / Object 的方法不破坏"单抽象"？因为 lambda 只需要"知道去实现哪一个方法"。default/static 已有实现，不需要 lambda 提供；而 equals、hashCode、toString 这些来自 Object 的方法，任何对象天生就有，也不计入抽象方法数。所以 Comparator 里显式声明了 equals(Object) 仍是合法的函数式接口。加上 @FunctionalInterface 注解后，一旦你不小心加了第二个抽象方法，编译器会直接报错——把约束前移到编译期。

四、别重复造轮子：java.util.function 四大金刚

大多数 lambda 场景无非"要不要入参、要不要返回值"。JDK 在 java.util.function 包里预置了几十个通用函数式接口，日常 90% 用得上的就四类，务必记牢它们的"抽象方法名"（写 lambda 时用不到，但看 API、写方法引用时天天见）：

接口抽象方法语义（入 → 出）典型场景
Supplier<T>T get()无入参 → 产出 T延迟求值、工厂、Optional.orElseGet
Consumer<T>void accept(T)吃进 T → 无返回forEach、打印、写库
Function<T,R>R apply(T)T → Rmap 转换、类型映射
Predicate<T>boolean test(T)T → booleanfilter 判断、条件筛选
UnaryOperator<T>T apply(T)T → T（同类型）原地转换、replaceAll
BiFunction<T,U,R>R apply(T,U)(T,U) → R两参合并、Map.merge
联系性能：为什么还有 IntFunction、ToIntFunction、IntPredicate 一大堆带 Int/Long/Double 前缀的变体？因为泛型 Function<Integer,…> 会触发自动装箱——每个 int 都包成一个 Integer 对象，在 Stream 处理百万数据时会制造海量垃圾对象、拖垮 GC。基本类型特化接口（如 IntStream.map(IntUnaryOperator)）让整条链路都走原始 int，避免装箱。这也是 IntStream/LongStream 单独存在的原因。

五、方法引用：当 lambda 只是"转手调一下"

如果 lambda 的函数体只是"拿参数去调用某个已有方法"，如 s -> s.length()、x -> System.out.println(x)，那连 lambda 都嫌啰嗦，可用方法引用（::）直接指名那个方法。它只是 lambda 的语法糖，编译后完全等价。四种形态：

形态写法等价 lambda
静态方法Integer::parseInts -> Integer.parseInt(s)
特定对象的实例方法System.out::printlnx -> System.out.println(x)
某类型任意对象的实例方法String::lengths -> s.length()
构造器ArrayList::new() -> new ArrayList<>()
易混点：String::length 这种"未绑定实例方法引用"。它看着像静态引用，其实第一个参数会被当作"调用该方法的接收者"。所以 Function<String,Integer> f = String::length; 里，f.apply("abc") 实际执行 "abc".length()——那个 String 入参悄悄变成了 this。初学者常纠结"参数怎么少了一个"，症结就在这。

六、变量捕获：为什么必须 effectively final

lambda 可以引用外层方法的局部变量，这叫捕获（capture）。但有条硬限制：被捕获的局部变量必须是 final 或 effectively final（事实最终——赋值后再没被改过）。

✗ Bad · 捕获后又修改
int count = 0;
Runnable r = () -> System.out.println(count);
count++;   // 编译错误！
// count 不再 effectively final，
// lambda 引用它就不合法了
✓ Good · 用不变量 / 数组或字段绕行
int count = readCount();  // 之后不改
Runnable r = () -> System.out.println(count);

// 真要"可变累加"，用字段或包装：
int[] box = {0};
list.forEach(x -> box[0] += x); // 捕获引用，改内容
联系内存模型：这不是 Java 故意刁难，而是不得不为。局部变量存在栈帧里，方法一返回栈帧就销毁。可 lambda（尤其作为回调/丢进线程）的生命周期往往比创建它的方法更长。JVM 的做法是把被捕获的变量值拷贝一份存进 lambda 实例。既然是拷贝，如果允许外层继续改原变量，就会出现"外面改了、lambda 里还是旧值"的诡异不一致。强制 effectively final，等于强制"值语义"，从根上消灭这种二义性。对比：实例字段/静态字段存在堆上，不受此限，能在 lambda 里随意读写——这也是上面 box[0] 能绕过限制的原因（数组对象在堆上，改的是它的内容而非那个 final 引用）。

七、lambda ≠ 匿名内部类：this 与编译产物

初学者常以为"lambda 就是匿名内部类的简写"。语义上像，但两处关键区别是高频面试点：

维度匿名内部类lambda
this 指向匿名类实例自己外层类实例（词法作用域）
能否有字段/状态可以，是个完整的类不能，只是一段行为
编译产物生成独立 Outer$1.class不生成额外 class 文件
底层机制编译期定死 new 一个类运行期 invokedynamic 动态生成
命名冲突/遮蔽可遮蔽外层变量名不能与外层局部变量同名
最坑的差异是 this。在匿名内部类里 this 指匿名类那个对象；在 lambda 里 this 指写下这段 lambda 的那个外层类实例。想在回调里访问外层对象，匿名类要写 Outer.this.field，lambda 直接写 this.field 或 field 即可。若你把一段依赖 Outer.this 的匿名类"顺手改成 lambda"，语义可能悄悄变化。

八、底层揭秘：invokedynamic 与 LambdaMetafactory

既然 lambda 不生成匿名内部类文件，编译器到底把它编译成了什么？答案是 Java 7 引入、Java 8 首次大规模使用的 invokedynamic 指令。流程如下：

编译期（javac） ┌────────────────────────────────────────────────┐ │ lambda 体 (a,b) -> a+b │ │ ①抽成一个 private 合成方法 lambda$main$0() │ │ ②在用它的地方生成一条 invokedynamic 指令 │ │ 引导方法 = LambdaMetafactory.metafactory │ └────────────────────────────────────────────────┘ │ 首次执行到这条指令时 ↓ ▼ 运行期（JVM） ┌────────────────────────────────────────────────┐ │ 引导方法调用 InnerClassLambdaMetafactory │ │ 动态"纺出(spin)"一个实现 Calculator 的类 │ │ 它的 apply() 转调 lambda$main$0() │ │ 生成好的调用点被链接并缓存 │ └────────────────────────────────────────────────┘ │ 之后再执行 ↓ ▼ 直接命中缓存的调用点，几乎零额外开销 无捕获的 lambda 还会复用同一个单例实例
为什么费这么大劲用 invokedynamic，而不直接生成匿名类？①避免类爆炸：一个大项目成千上万个 lambda，若各生成一个 .class，会显著拖慢类加载、撑大 Jar。②把"怎么实现"的决策推迟到运行期：具体用什么策略（生成类？还是未来某种更轻的方式）由 LambdaMetafactory 决定，语言规范不写死，给 JVM 留了优化空间。③无状态 lambda 可复用单例：不捕获任何变量的 lambda（如 x -> x*2）整个 JVM 只需一个实例，比每次 new 匿名对象省内存。这就是为什么把无捕获 lambda 放循环里也不用担心反复建对象。

九、串起来看：Stream 与 Spring 里的 lambda

lambda 单独看是把小刀，真正的威力在它成为整个函数式 API 的"通用插槽"。你以前学过的循环+条件，现在能声明式地拼起来：

四大金刚齐上阵Stream 流水线
List<String> names = people.stream()
    .filter(p -> p.getAge() >= 18)   // Predicate：筛成年人
    .map(Person::getName)          // Function：取名字（方法引用）
    .sorted()
    .collect(Collectors.toList());

names.forEach(System.out::println);   // Consumer：逐个打印

在 Spring 生态里 lambda 同样无处不在：jdbcTemplate.query(sql, (rs, rowNum) -> ...) 的 RowMapper、restTemplate 回调、WebFlux 的函数式路由 route().GET("/hi", req -> ...)、Optional.map/orElseGet——它们的参数几乎都是函数式接口，lambda 让"配置行为"这件事从写一堆匿名类变成写一行表达式。

陷阱：lambda 里不要偷偷改共享可变状态。Stream 尤其是并行流，多个线程会并发跑同一段 lambda。若你在 lambda 里对外部集合 add、对计数器 ++（哪怕用字段绕过了 effectively final），就会产生竞态、丢数据。函数式的正道是无副作用：用 collect/reduce 收集结果，而不是在 forEach 里手动往外部容器塞。这与并发章节的"共享可变状态是万恶之源"是同一条铁律。
一句话收束：lambda 让"行为"成为一等公民、能被当参数传递，本质是目标类型为某个"单抽象方法（SAM）"函数式接口的实现；@FunctionalInterface 把"唯一抽象方法"的约束交给编译器把关，java.util.function 提供 Supplier/Consumer/Function/Predicate 等现成插槽，方法引用是它更简的语法糖。它不是匿名内部类的语法糖——this 指外层、不生成额外 class、靠 invokedynamic + LambdaMetafactory 在运行期动态生成并缓存实现。用好它要守两条纪律：捕获的局部变量必须 effectively final、lambda 体保持无副作用。
