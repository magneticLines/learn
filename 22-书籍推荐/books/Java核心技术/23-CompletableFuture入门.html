CompletableFuture 入门

Future 只能"埋头等结果"，CompletableFuture 让你"约定好结果来了做什么"——把阻塞式的取值，变成可编排的异步流水线。

一、先看痛点：Future 的三宗罪

上一节我们用 ExecutorService.submit() 拿到了 Future。它确实能代表一个"未来才有的结果"，但用起来处处别扭。问题的根源是：Future 只给了你一个"取货凭证"，却没给你"货到通知我"的能力。

Future 的短板具体表现
只能阻塞取值get() 会把当前线程卡死直到结果就绪，异步的意义荡然无存；isDone() 轮询又浪费 CPU
无法链式编排"A 算完把结果喂给 B"只能先 get() 出 A 再手动提交 B，退化成串行
无法组合与兜底不能"等 A、B 都完成再汇总"，也没有统一的异常回调，出错只能靠 get() 抛出 ExecutionException 后手动 catch

Java 8 引入的 CompletableFuture 就是来补齐这三块的。它同时实现了两个接口：

CompletableFuture = Future（可取值） + CompletionStage（可编排）
类比 JavaScript 的 Promise：如果你写过前端，CompletableFuture 几乎就是 Java 版的 Promise。thenApply ≈ .then()、exceptionally ≈ .catch()、allOf ≈ Promise.all()、anyOf ≈ Promise.race()。核心思想都一样：不要主动去等结果，而是提前登记好"结果来了之后该干什么"的回调，这就是"响应式/声明式"编程的雏形。

二、创建异步任务：supplyAsync 与 runAsync

最常用的两个静态工厂方法，区别只在于"有没有返回值"——正好对应 Supplier 和 Runnable 两个函数式接口（还记得第 10 章的 lambda 吗）。

CompletableFuture 的两种启动方式有返回值 vs 无返回值
// 有返回值：Supplier<T> → CompletableFuture<T>
CompletableFuture<String> cf1 = CompletableFuture.supplyAsync(() -> {
    // 这段代码在“别的线程”里跑，主线程不会被卡住
    return queryUserName(42);   // 假设是一次耗时的 DB 查询
});

// 无返回值：Runnable → CompletableFuture<Void>
CompletableFuture<Void> cf2 = CompletableFuture.runAsync(() -> {
    sendEmail();               // 只干活，不产出结果
});

// 手动“完成”一个已知值——这就是名字里 Completable 的由来
CompletableFuture<String> cf3 = CompletableFuture.completedFuture("cached");
"Completable"到底是什么意思？普通 Future 的结果只能由执行任务的线程写入，外界无法插手。而 CompletableFuture 暴露了 complete(value) / completeExceptionally(ex)，允许你从任意线程手动把它标记为完成。这在桥接老式回调 API（如 Netty 的 ChannelFuture、各种基于监听器的 SDK）时极其有用：在回调里调 cf.complete(result)，就把"回调地狱"转成了"链式流水线"。

三、结果来了做什么：then 家族三兄弟

登记回调是 CompletableFuture 的灵魂。根据"你要不要上一步的结果、要不要产出新结果"，分成三个方法，别记混：

方法入参（函数式接口）返回语义
thenApplyFunction<T,U>CF<U>拿结果、转换出新结果（像 Stream 的 map）
thenAcceptConsumer<T>CF<Void>拿结果、消费掉，不产出
thenRunRunnableCF<Void>不关心结果，只收尾做点事
一条声明式流水线没有一次显式 get()
CompletableFuture.supplyAsync(() -> queryUser(42))   // CF<User>
    .thenApply(User::getName)                     // CF<String> 转换
    .thenApply(String::toUpperCase)                // CF<String> 再转换
    .thenAccept(name -> log.info("got {}", name)) // CF<Void> 消费
    .thenRun(() -> log.info("done"));            // CF<Void> 收尾

四、线程从哪来：commonPool 的隐形陷阱

你可能好奇：supplyAsync 里的 lambda 到底跑在哪个线程？答案是——如果不显式传 Executor，默认用 ForkJoinPool.commonPool()，这是整个 JVM 共享的一个全局池。这里藏着新手最常踩的坑。

supplyAsync(fn) supplyAsync(fn, myPool) │ │ ▼ ▼ ForkJoinPool.commonPool() 你自己的线程池 ├ 全 JVM 共享（并行流也用它） ├ 独立、可控、可命名 ├ 线程数 ≈ CPU核数 - 1 ├ 大小按 IO/CPU 特性定制 └ 都是 daemon 守护线程 └ 出问题只影响自己
陷阱一：别拿默认 commonPool 跑阻塞 IO。commonPool 的线程数默认约等于 CPU 核数 - 1，它是为CPU 密集型的短任务设计的。一旦你把"查数据库、调 HTTP、读文件"这类会阻塞的 IO 任务丢进去，几个慢请求就能把这寥寥几个线程全占满，导致同一 JVM 里其他用 commonPool 的代码（包括 parallelStream()）集体饿死。生产环境务必给 IO 任务传自定义 Executor。
陷阱二：commonPool 是守护线程。如果 main 方法直接结束，JVM 不会等这些 daemon 线程跑完就退出，你的异步任务可能悄无声息地被腰斩。写 main 里的 demo 时，记得在结尾 get()/join() 一下等它完成。

五、*Async 后缀：到底谁在哪个线程跑

几乎每个 then 方法都有三个重载，这是面试高频区，务必搞清"回调在哪个线程执行"：

写法回调执行线程
thenApply(fn)由完成上一步的那个线程顺手执行；若上一步已完成，则由当前调用线程执行
thenApplyAsync(fn)重新丢回 ForkJoinPool.commonPool() 执行
thenApplyAsync(fn, executor)丢到你指定的 executor 执行
怎么选？如果回调很轻（一次字段映射），用不带 Async 的版本，省去一次线程切换开销；如果回调本身较重或会阻塞，用 xxxAsync(fn, 专用池) 把它隔离出去。滥用 Async 会让任务在不同线程间反复"乒乓"，反而拖慢速度。

六、组合多个任务：Compose vs Combine

真正的价值在于把多个异步任务编排到一起。两个最容易混淆的方法——thenCompose 和 thenCombine——区别在于任务之间"有没有依赖"。

方法适用场景类比 Stream
thenComposeB 依赖 A 的结果（前后串联，且 B 本身也返回一个 CF）flatMap
thenCombineA、B 互不依赖，各自跑完后汇总两者结果无（是"合流"）
thenCompose：为什么不用 thenApply避免嵌套 CF<CF<T>>
// 场景：先查用户，再用 userId 查他的订单（第二步也是异步）

// ✗ 用 thenApply 会套娃：结果类型变成 CF<CF<Order>>
CompletableFuture<CompletableFuture<Order>> bad =
    supplyAsync(() -> queryUser(42))
        .thenApply(u -> queryOrderAsync(u.getId()));

// ✓ 用 thenCompose 把内层“摊平”：干净的 CF<Order>
CompletableFuture<Order> good =
    supplyAsync(() -> queryUser(42))
        .thenCompose(u -> queryOrderAsync(u.getId()));
thenCombine：两个独立任务并行后汇总这才是异步的甜点
CompletableFuture<Integer> price  = supplyAsync(() -> queryPrice(sku));
CompletableFuture<Integer> stock  = supplyAsync(() -> queryStock(sku));

// 两个查询各跑各的，都完成后用 BiFunction 合并
CompletableFuture<String> result = price.thenCombine(stock,
    (p, s) -> "价格=" + p + " 库存=" + s);

并行组合的收益可以量化——串行等待是把耗时相加，并行编排是取最大值：

串行 Σ(t₁+t₂+t₃)   →   并行 max(t₁,t₂,t₃)

七、等待一批任务：allOf 与 anyOf

要"等一组任务全部完成"或"等最快的那个"，用这两个静态方法：

方法完成时机返回
allOf(cf...)所有都完成才完成（≈ Promise.all）CF<Void>（不带结果，需回原 CF 取值）
anyOf(cf...)任一完成即完成（≈ Promise.race）CF<Object>（最先完成者的结果）
陷阱三：allOf 返回的是 CompletableFuture<Void>，它不帮你收集结果。很多人以为 allOf(a,b,c).get() 能拿到列表，其实拿到的是 null。正确做法是 allOf(...) 之后，再对原来的每个 CF 分别 join()（此时它们都已完成，join 不会阻塞）来取值。
allOf 的正确收集姿势批量并发 + 汇总
List<CompletableFuture<String>> futures = ids.stream()
    .map(id -> supplyAsync(() -> query(id), pool))
    .toList();

CompletableFuture.allOf(futures.toArray(CompletableFuture[]::new))
    .thenApply(v -> futures.stream()
        .map(CompletableFuture::join)  // 此刻都完成了，join 不阻塞
        .toList())
    .thenAccept(System.out::println);

八、异常处理：exceptionally / handle / whenComplete

异步任务里抛出的异常，不会在原地被 catch 到——它会"沿着链条向下顺流"，直到遇见一个异常处理节点。三种处理方式各有分工：

方法何时触发能否改结果用途
exceptionally仅出异常时能（给个兜底值）像 catch，只管出错的分支
handle成功/失败都触发能（拿到 result 或 ex）统一收口，二选一处理
whenComplete成功/失败都触发否（只能"观察"）像 finally，记日志/清理，原样透传结果
三者组合使用异常沿链下沉，被 handler 拦截
supplyAsync(() -> {
        if (down) throw new RuntimeException("服务不可用");
        return query();
    })
    .thenApply(Result::parse)              // 上游异常时，这一步被“跳过”
    .exceptionally(ex -> Result.EMPTY)   // 拦截异常，给兜底值，链条恢复正常
    .whenComplete((res, ex) ->               // finally：无论如何都记一笔
        log.info("finished, ex={}", ex));
陷阱四：吞掉的异常。如果一个 CompletableFuture 失败了，而你从不对它调用 get()/join()，也没挂 exceptionally/handle，那么这个异常会被彻底吞掉，控制台一片安静——不像同步代码会打印堆栈。这是异步排障最坑的地方。规矩：每条链的末端都要有异常出口。
陷阱五：get() 与 join() 的差异 + 异常被包一层。get() 抛受检异常（InterruptedException、ExecutionException），必须显式 try-catch；join() 抛非受检的 CompletionException，写在 lambda / Stream 里更顺手。另外注意：你原本抛的 RuntimeException 传到下游时会被包裹进 CompletionException（或 get 的 ExecutionException），取真正病因要用 ex.getCause()。

九、反例 → 正例：告别回调地狱与阻塞

把前面的知识落到一个真实场景：查用户、并行查其画像和积分、汇总、出错兜底。对比"老式 Future + 阻塞"和"CompletableFuture 编排"两种写法。

✗ Future：串行阻塞 + 手动 catch
Future<User> fu = pool.submit(() -> queryUser(id));
User u = fu.get();          // 阻塞①

Future<Profile> fp = pool.submit(
    () -> queryProfile(u.getId()));
Future<Integer> fs = pool.submit(
    () -> queryScore(u.getId()));
Profile p = fp.get();       // 阻塞②
Integer s = fs.get();       // 阻塞③
// 三次 get 把线程钉死；
// 每个 get 都要 try/catch ExecutionException；
// 出错兜底逻辑散落各处
✓ CompletableFuture：编排 + 统一兜底
supplyAsync(() -> queryUser(id), pool)
  .thenCompose(u -> {          // 依赖 user
    var p = supplyAsync(
        () -> queryProfile(u.getId()), pool);
    var s = supplyAsync(
        () -> queryScore(u.getId()), pool);
    return p.thenCombine(s,     // p、s 并行
        (pr, sc) -> merge(u, pr, sc));
  })
  .exceptionally(ex -> View.fallback()) // 统一兜底
  .thenAccept(view::render);   // 全程不阻塞主线程

十、与生态的联系 & Java 9 增强

  • Spring：@Async 方法可以直接返回 CompletableFuture<T>，Spring MVC 也支持它作为 Controller 返回值实现异步请求处理，释放 Servlet 容器线程。
  • Reactor / WebFlux：CompletableFuture 是"一次性单值"的异步（一个 Promise 只完成一次），而 Mono/Flux 是"可背压的响应式流"；两者可通过 Mono.fromFuture() 互转。理解了前者，学响应式会顺很多。
  • Java 9+ 补强：新增 orTimeout(t, unit)（超时自动异常）、completeOnTimeout(v, t, unit)（超时给默认值）、delayedExecutor(...)（延迟执行），补上了"没有超时控制"这一早期短板。
心智模型：把 CompletableFuture 想成一条传送带。supplyAsync 把原料放上带子；每个 then* 是带子上的一个加工工位；thenCombine/allOf 是多条带子的合流口；exceptionally/handle 是次品分拣口。你只负责布置工位，不负责站在出口干等——这就是从"命令式阻塞"到"声明式编排"的思维转变。
一句话收束：CompletableFuture = Future 的取值能力 + CompletionStage 的编排能力。用 supplyAsync/runAsync 启动，用 thenApply/thenCompose/thenCombine/allOf 把任务串成并行流水线，用 exceptionally/handle/whenComplete 收口异常。五条铁律记牢：①IO 任务必须传自定义线程池，别用默认 commonPool；②Async 后缀决定回调在哪个线程跑，别滥用；③thenCompose 摊平嵌套、thenCombine 汇总独立任务；④allOf 不收集结果，要回原 CF 取值；⑤每条链末端都要有异常出口，否则异常被静默吞掉。
