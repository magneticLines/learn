输入输出基础：Scanner 与格式化输出

从键盘的一串字节，到程序里的 int、double、String——中间那层"翻译官"到底做了什么？

1. 从 System.in 说起：为什么需要 Scanner

程序要"读用户输入"，本质是从操作系统提供的标准输入里拿数据。在 Java 里这个入口就是 System.in。但当你看它的真实类型，会发现它只是一个 InputStream——一个字节流，一次只吐一个 byte。

System.in 裸读到的是什么？ 键盘输入 "42⏎" │ ▼ ┌──────────────┐ │ System.in │ 类型: InputStream（字节流） │ .read() │ 一次返回一个 byte（0~255，读到末尾返回 -1） └──────────────┘ │ ▼ 52, 50, 10 ← '4' '2' '\n' 的 ASCII 码 你拿到的是字节 52，而不是整数 42！

所以裸用 System.in 得自己干三件苦力活：把字节按编码拼成字符、把字符按空白切成 token、再把 token 字符串解析成数字。Scanner 就是把这三步打包好的"翻译官"——它是一个建立在流之上的解析器（parser），而不是流本身。

装饰者模式的影子。Java I/O 到处是"包一层加一个能力"的设计：InputStream（字节）→ InputStreamReader（按字符集转成字符）→ BufferedReader（加缓冲、按行读）。Scanner 走的是另一条路——它不改变流，而是在流之上叠加正则分词 + 类型转换的语义。理解这条"能力叠加链"，后面学 IO/NIO 会顺很多。

2. Scanner 基础：token 与整行是两套逻辑

Scanner 有两类读法，初学者最容易混。next 系列（next() / nextInt() / nextDouble()）按分隔符切 token，默认分隔符是"任意空白"（空格、Tab、换行都算）；nextLine() 则读到行尾，返回不含换行符的一整行。

ScannerBasics.javatoken 读取
import java.util.Scanner;

public class ScannerBasics {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);

        System.out.print("姓名和年龄（空格分隔）: ");
        String name = sc.next();       // 读一个 token，遇空白即停
        int age = sc.nextInt();        // 只吃掉数字本身，把后面的换行留在缓冲里

        System.out.printf("%s 今年 %d 岁%n", name, age);
        sc.close();              // 关闭 Scanner 会连带关闭 System.in（见下方陷阱）
    }
}
用 hasNextXxx 做防御式读取。nextInt() 遇到非数字会直接抛 InputMismatchException。健壮的写法是先问再拿：while (sc.hasNextInt()) total += sc.nextInt();。hasNext* 只"看一眼"不消费 token，特别适合读到文件末尾（EOF）或不确定长度的输入。

3. 头号陷阱：nextInt() 之后的 nextLine() 读到空行

这是几乎每个人都会踩的坑，也是笔试面试常考的"你真懂 Scanner 吗"。原因就藏在上一节：nextInt() 只消费数字，不消费数字后面的换行符。紧接着的 nextLine() 会从"当前这个残留的换行符"读起，于是立刻遇到行尾——返回一个空字符串。

缓冲区里的字符（用户先输入 42⏎，再输入 Tom⏎） [ 4 ][ 2 ][\n][ T ][ o ][ m ][\n] └──┬──┘ │ nextInt() 读走 "42"，光标停在这里 ↑ │ └─ 残留的 \n 还在！ nextLine() 从这里读起 → 立刻碰到 \n → 返回 ""（空串）
✘ 反例：city 永远是空串
int age = sc.nextInt();
// 想读下一行的城市名
String city = sc.nextLine();
// city == ""  （吃到的是 42 后面的换行）
String real = sc.nextLine();
// 真正的城市名跑到了这里！
✔ 正例：先消费掉残留换行
int age = sc.nextInt();
sc.nextLine();            // 关键：吃掉残留的 \n
String city = sc.nextLine();
// city 正确拿到整行

// 或统一用 nextLine() + 手动解析：
int age2 = Integer.parseInt(
        sc.nextLine().trim());
记住这条铁律：只要你在 nextInt() / nextDouble() 之后想用 nextLine()，中间必须补一句 sc.nextLine() 清掉行尾。更省心的做法是——整个程序统一只用 nextLine() 读整行，再自己 Integer.parseInt / split，从根上避免两套光标语义打架。
另一个隐蔽坑：sc.close() 会关掉 System.in。Scanner 实现了 Closeable，关闭它会连带关闭底层的 System.in。若之后再 new Scanner(System.in) 读取，会直接抛 NoSuchElementException。整个程序对 System.in 只建一个 Scanner，且通常不必手动 close。（对文件/网络流则相反，务必 try-with-resources 关。）

4. Scanner 的分隔符原理与性能取舍

为什么 next() 能自动跳过空格？因为 Scanner 内部维护了一个分隔符正则，默认是 \p{javaWhitespace}+。它每次用这个正则在缓冲区里找下一个 token 边界——所以 Scanner 本质是一台正则驱动的分词机，功能强但每次读取都要跑正则匹配，开销不小。

CustomDelimiter.java自定义分隔符
// 读 CSV 那样的逗号分隔：把分隔符换成逗号（可带空白）
Scanner sc = new Scanner("tom, 42, beijing");
sc.useDelimiter("\\s*,\\s*");   // 分隔符 = 逗号及其两侧空白
while (sc.hasNext()) System.out.println(sc.next());
// 输出: tom / 42 / beijing

当输入量很大（几十万行、竞赛/日志处理），Scanner 的正则开销会成为瓶颈。此时应换成 BufferedReader：它按行整块读、几乎不做解析，速度快一个数量级，代价是你要自己 split 和 parseInt。

维度ScannerBufferedReader
定位解析器：分词 + 类型转换缓冲字符流：按行读原始文本
类型转换内置 nextInt/nextDouble…无，需自己 parseInt
底层机制正则匹配分隔符（慢）char 缓冲区批量拷贝（快）
缓冲区较小，默认约 1KB较大，默认 8KB
线程安全方法非同步，非线程安全readLine 同步
适合场景交互式、少量输入、要按类型读大批量、按行处理、追求性能
联系 Spring / 实战：真正的后端项目里几乎不用 Scanner——HTTP 请求体由框架（Jackson/Spring MVC）反序列化，文件用 Files.lines() / NIO。Scanner 主要活在命令行小工具、教学 demo、算法竞赛的本地调试里。理解它，是为了理解"字节流→字符→token→类型"这条通用管线，而不是把它当生产工具。

5. 格式化输出：printf 与 String.format

输入的对偶是输出。System.out.print 只会调 toString，控制不了小数位、对齐、进制。printf（借鉴自 C）用格式字符串 + 转换说明符精确排版；String.format 参数完全一样，只是返回字符串而不是直接打印——底层都走同一个 java.util.Formatter。

%[argIndex$] [flags] [width] [.precision] conversion

方括号部分可选；conversion（如 d/f/s）必填。例如 %-8.2f = 左对齐、宽 8、2 位小数、浮点。

FormatDemo.javaprintf 排版
System.out.printf("%-8s|%5d|%8.2f%n", "苹果", 3, 12.5);
System.out.printf("%-8s|%5d|%8.2f%n", "西瓜", 12, 6.0);
// 苹果    |    3|   12.50   （%-8 左对齐，%5/%8 右对齐补空格）
// 西瓜    |   12|    6.00

// 千分位 + 正号 + 十六进制 + 参数复用
System.out.printf("%,d  %+d  %#x%n", 1000000, 42, 255);
// 1,000,000  +42  0xff
System.out.printf("%1$s-%1$s%n", "重复");  // 1$ = 复用第 1 个参数 → 重复-重复

// String.format：不打印，拿到字符串（记日志、拼消息常用）
String line = String.format("订单 #%05d 金额 ¥%.2f", 7, 99.5);
// 订单 #00007 金额 ¥99.50  （%05d 补零到 5 位）
说明符含义示例（值 → 输出）
%d十进制整数42 → 42
%f浮点，默认 6 位小数3.14 → 3.140000
%.2f浮点，保留 2 位（四舍五入）3.146 → 3.15
%e科学计数法12345.0 → 1.234500e+04
%s字符串（对任意对象调 toString）"hi" → hi
%b布尔true → true
%x / %o十六 / 八进制255 → ff / 377
%,d加千分位分组1000000 → 1,000,000
%-10s宽 10、左对齐"hi" → hi········
%n平台换行符Win 输出 \r\n、Unix 输出 \n
%%一个字面百分号→ %

6. 格式化输出的三个高频陷阱

陷阱一：%n vs \n。\n 是硬编码的 LF；%n 由 Formatter 按操作系统的 line.separator 生成。在 Windows 上写日志/生成文本文件时，用 %n 才能得到 \r\n，避免文件被某些工具当成"没有换行"。跨平台一律优先 %n。
陷阱二：类型对不上直接抛异常。printf("%d", 3.5) 不会自动取整，而是抛 IllegalFormatConversionException；%d 只认整型。同理给 %f 传 int 也会炸。这类错误编译期查不出（格式串是运行期解析的），只能靠测试暴露——这正是它危险的地方。
陷阱三：locale 会改变小数点/分组符。Formatter 默认用 JVM 的默认 Locale。在某些欧洲区域，%,.2f 会把 1234.5 输出成 1.234,50（点作分组、逗号作小数点）。对机器可读的输出（写 CSV、拼 SQL、生成 JSON），务必显式指定 String.format(Locale.ROOT, ...)，别让运行环境的地区设置污染数据。
✘ 依赖默认 Locale，输出不稳定
// 部署到不同区域的机器，结果不同
String s = String.format(
        "%,.2f", 1234.5);
// 可能是 "1,234.50"
// 也可能是 "1.234,50"
✔ 显式 Locale.ROOT，结果确定
String s = String.format(
        Locale.ROOT, "%,.2f", 1234.5);
// 永远是 "1,234.50"
// 给人看的界面才用本地 Locale

7. 现代写法：formatted 与文本块

Java 15 起，String 多了实例方法 formatted(...)，等价于 String.format(this, ...)，配合文本块（三引号）写多行模板更顺眼：

Modern.javaJava 15+
String report = """
        用户: %s
        余额: ¥%,.2f
        """.formatted("Tom", 12345.6);
System.out.print(report);
// 用户: Tom
// 余额: ¥12,345.60
一页话记住： ① System.in 是字节流，Scanner 是叠在它上面的正则分词 + 类型转换解析器； ② nextInt() 不吃换行，紧跟 nextLine() 必踩空行坑——统一用 nextLine() 或补一句清行； ③ 大数据量换 BufferedReader，Scanner 只适合交互/小输入； ④ 格式化记住结构 %[flags][width][.precision]conversion，换行用 %n，机器可读输出加 Locale.ROOT； ⑤ printf 直接打印、String.format/formatted 返回串，底层同为 Formatter。
