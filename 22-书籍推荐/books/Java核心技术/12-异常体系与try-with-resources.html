异常体系与 try-with-resources

异常是把"正常业务"与"出错处理"分成两条轨道的语言机制；而 try-with-resources 则是 Java 对"忘记关资源"这一经典 bug 的一次语言级根治。

一、为什么需要异常？返回码的困境

在没有异常的年代（比如 C），函数出错只能靠返回一个特殊值（-1、NULL、errno）告诉调用方。问题是：错误处理代码和正常逻辑缠在一起，而且返回码可以被悄悄忽略——你不检查，程序照样往下跑，错误被吞掉，直到很久之后在别处爆炸。

✗ 返回码风格：正常逻辑被淹没
int fd = open(path);
if (fd < 0) return ERR_OPEN;
int n = read(fd, buf);
if (n < 0) { close(fd); return ERR_READ; }
if (parse(buf) != 0) { close(fd); return ERR_PARSE; }
close(fd);
// 每一步都要手动判、手动清理
// 漏一个 if，错误就被吞
✓ 异常风格：两条轨道分离
try {
    var buf = read(path);   // 正常逻辑
    parse(buf);             // 一气呵成
} catch (IOException e) {
    // 出错处理集中在一处
    log.error("读取失败", e);
}
// 忽略错误？编译器不让你(受检异常)

异常的核心设计思想就藏在这里：把"正常控制流"和"异常控制流"拆成两条独立的轨道。正常路径读起来一目了然，出错路径被集中收拢，而且异常一旦抛出无法被无声无息地忽略——不处理，它就沿调用栈一路上抛，直到线程终止并打印栈轨迹。这是一种"不许你装看不见"的强约束。

二、Throwable 家族树

Java 里所有能被 throw 和 catch 的东西，都必须继承自 java.lang.Throwable。它分成两条大支：Error 和 Exception，而 Exception 下面又切出一支特殊的 RuntimeException。这棵树的形状，直接决定了编译器怎么管你。

Throwable <-- 唯一能被 throw/catch 的根 | +---------------+----------------+ | | Error Exception 「JVM 层面的灾难」 「程序可预期的问题」 OutOfMemoryError | StackOverflowError +-----------+-------------+ NoClassDefFoundError | | → 别 catch，救不了 RuntimeException 其它 Exception 「编程 bug」 「受检异常 checked」 NullPointerException IOException ArrayIndexOutOfBounds SQLException IllegalArgumentException FileNotFoundException ClassCastException → 编译器强制处理 「unchecked 免检」
为什么 Error 不该 catch？Error 代表 JVM 自身或环境层面的严重故障——堆内存耗尽（OutOfMemoryError）、栈溢出（StackOverflowError）、类加载失败（NoClassDefFoundError）。这些问题不是你的业务代码能恢复的：内存都满了，你 catch 住又能做什么？强行 catch 往往只会掩盖真相、让程序在半死不活的状态下继续糟蹋数据。正确姿势是让它把线程/进程干掉，然后靠监控告警和重启来处理。

三、受检 vs 非受检：这是全篇最关键的分野

编译器把异常分成两类，用完全不同的态度对待。判断标准很简单：凡是 RuntimeException 及其子类、以及 Error 及其子类，都是"非受检（unchecked）"；其余 Exception 的子类都是"受检（checked）"。

维度受检异常 checked非受检异常 unchecked
顶层父类Exception（非 RuntimeException 支）RuntimeException / Error
编译器态度强制：要么 catch，要么方法签名 throws不管，爱处理不处理
语义可预料、可恢复的外部问题编程 bug / JVM 灾难
典型代表IOException、SQLExceptionNullPointerException、IllegalArgumentException
处理直觉"网络可能断、文件可能没" → 该重试/降级"你代码写错了" → 该改代码，不是 catch

背后的设计哲学是：受检异常说的是"环境靠不住"——文件可能不存在、网络可能超时，这些是调用方理应预见并制定对策的，所以编译器逼你正视它。非受检异常说的是"你写错了"——传了 null、下标越界、类型转错，这些是 bug，正确的做法是修 bug，而不是到处 try-catch 遮丑。

高频陷阱 —— 空 catch 块吞异常：下面这段代码是灾难之源。异常被"吃掉"后既不记录也不上抛，出了问题现场一片祥和，你却要熬夜排查一个永远找不到栈轨迹的诡异 bug。
try { risky(); } catch (Exception e) { } ← 空 catch，绝对禁止
最起码也要 log.error("...", e)。捕获了就必须处理或重新抛出，沉默是最坏的选择。
Spring 的选择很说明问题：Spring 把 JDBC 那套受检的 SQLException 全部包装成非受检的 DataAccessException 体系（@Transactional 也默认只对 RuntimeException 回滚）。因为绝大多数数据库错误业务代码根本无从恢复，强制 throws SQLException 只会污染每一层方法签名。这反映了业界共识：受检异常虽严谨，却容易造成"签名传染"和满屏样板 catch，现代框架普遍倾向非受检。

四、try-catch-finally 的执行顺序

finally 的承诺是：无论 try 块正常结束、抛异常、还是中途 return，finally 里的代码几乎总会执行。它天生是用来做"收尾清理"的——关文件、还连接、释放锁。执行流程如下：

进入 try | +-- 无异常 --> try 跑完 --> finally --> 方法返回 | +-- 抛异常 --> 匹配的 catch 执行 --> finally --> 继续上抛/返回 | +-- try 里 return --> 先算好返回值 --> 执行 finally --> 才真正返回 唯一逃过 finally 的情况：System.exit() 或 JVM/线程被强杀
魔鬼陷阱 —— 在 finally 里 return / throw：它会覆盖掉 try 或 catch 中原本要返回的值或要抛出的异常，让异常凭空消失。这是面试常考的"坑题"，实战中更是一类极隐蔽的 bug。
FinallyTrap.javafinally 里 return 吞掉一切
int bad() {
    try {
        return 1;          // 返回值 1 已算好，暂存
    } finally {
        return 2;          // finally 的 return 把 1 顶掉！
    }
    // 实际返回 2 —— try 的 return 形同虚设
}

int worse() {
    try {
        throw new RuntimeException("重要错误");
    } finally {
        return 0;          // 异常被 finally 的 return 无声吞掉，调用方毫不知情
    }
}

结论很直接：finally 块里只放清理代码，永远不要在 finally 里写 return 或 throw。它的职责是"善后"，不是"改变结局"。

五、老式关资源写法有多丑

在 Java 7 之前，正确地关闭资源需要一坨令人窒息的嵌套 try-finally。原因有二：close() 本身可能抛异常；而且多个资源要各自保证关闭。稍不留神就会资源泄漏。

OldStyle.javaJava 6 时代：正确但丑陋
InputStream in = null;
try {
    in = new FileInputStream(path);
    // ... 使用 in ...
} finally {
    if (in != null) {     // 必须判空：构造可能就失败了
        try {
            in.close();     // close 自己也可能抛异常，得再包一层
        } catch (IOException ignore) { }
    }
}
// 两个资源？再套一层，缩进能顶到屏幕右边
更阴险的"异常屏蔽（exception masking）"：如果 try 块里先抛了一个业务异常 A，接着 finally 里 close() 又抛了异常 B，那么B 会覆盖 A——真正的病因 A 被丢弃，你看到的栈轨迹却是无关紧要的 B。这是老式写法最坑爹的地方，也是 try-with-resources 要解决的核心痛点之一。

六、try-with-resources：语言级的根治

Java 7 引入 try-with-resources：把资源声明写进 try(...) 的括号里，只要资源实现了 AutoCloseable 接口，编译器就自动、保证、逆序地帮你调用 close()。它不是运行时魔法，而是语法糖——编译器在字节码里帮你展开成正确的 try-finally。

✗ 手动 finally
BufferedReader br = null;
try {
  br = new BufferedReader(
        new FileReader(path));
  return br.readLine();
} finally {
  if (br != null) br.close();
}
// close 抛异常会屏蔽真正的异常
✓ try-with-resources
try (var br = new BufferedReader(
        new FileReader(path))) {
  return br.readLine();
}
// close 自动调用、逆序调用
// close 的异常被"抑制"不会屏蔽主异常
// 代码短了一半，且没有泄漏可能
MultiResource.java多资源：逆序关闭 + 抑制异常
// 多个资源用分号隔开，声明顺序打开，逆序关闭（后开的先关）
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql);
     ResultSet rs = ps.executeQuery()) {

    while (rs.next()) {
        process(rs.getString("name"));
    }
}
// 关闭顺序：rs.close() → ps.close() → conn.close()
// 逆序是关键：Connection 最先建、最后关，符合依赖方向
逆序关闭为什么重要？资源之间常有依赖：ResultSet 建立在 Statement 之上，Statement 又依附于 Connection。就像穿衣服——先穿内衣再穿外套，脱的时候必须先脱外套。若先关了 Connection，再去关 ResultSet 就可能操作一个已失效的资源。try-with-resources 用"栈"式的逆序关闭天然保证了这个顺序。

七、AutoCloseable 与自定义资源

任何类只要实现 AutoCloseable（只有一个 close() 方法），就能放进 try-with-resources。Closeable 是它的子接口（专用于 IO，close() 只抛 IOException）。你自己的类也能享受这套机制：

MyResource.java让自定义类支持自动关闭
public class DistributedLock implements AutoCloseable {
    private final String key;

    public DistributedLock(String key) {
        this.key = key;
        redis.acquire(key);        // 构造即加锁
    }

    @Override
    public void close() {         // try 块结束自动调用 → 自动解锁
        redis.release(key);
    }
}

// 用起来像 RAII：作用域一结束，锁必被释放，绝不会忘
try (DistributedLock lock = new DistributedLock("order:1001")) {
    processOrder();
}
与 C++ RAII / Python with 的联系：这套"作用域结束即释放"的思想，在 C++ 里叫 RAII（析构函数自动释放），在 Python 里是 with 语句配合 __enter__/__exit__。三者殊途同归，都是为了把"资源生命周期"绑定到"代码作用域"上，从根本上消灭"忘记释放"这类 bug。Java 因为有 GC，反而更需要它——GC 只管内存，不管文件句柄、socket、锁这些非内存资源，这些必须由你显式关闭。

八、被抑制的异常（Suppressed Exception）

还记得第五节的"异常屏蔽"吗？try-with-resources 用一个精妙的设计根治了它：如果 try 块抛出主异常，同时 close() 也抛异常，那么主异常照常上抛，而 close() 的异常被"抑制"并挂在主异常的 suppressed 列表里——两个都保留，一个都不丢。

主异常上抛  +  close 异常 → getSuppressed()
Suppressed.java取回被抑制的异常
try (MyResource r = new MyResource()) {
    throw new RuntimeException("主异常：业务失败");
    // 假设 r.close() 又抛了 "关闭失败"
} catch (Exception e) {
    System.out.println(e.getMessage());        // 主异常：业务失败
    for (Throwable s : e.getSuppressed()) {   // 遍历被抑制的异常
        System.out.println("  被抑制: " + s.getMessage()); // 关闭失败
    }
}
// 栈轨迹里会以 "Suppressed:" 前缀打印被抑制的异常，病因一个不丢
对比老式 finally：老式手写 finally 中 close() 抛的异常会直接顶掉主异常（后抛的异常胜出）；而 try-with-resources 是主异常胜出、close() 异常降级为 suppressed。前者丢信息，后者全保留——这正是语言级方案优于手写的价值所在。Throwable.addSuppressed() 就是编译器为此生成的字节码在背后调用的。

九、异常链：别丢了根因

实际工程里，我们常需要把底层异常"翻译"成本层的语义异常再上抛（比如把 SQLException 包成 UserNotFoundException）。此时务必把原始异常作为 cause 传进去，否则原始栈轨迹丢失，排查时你只能看到表层症状。

✗ 丢失根因
catch (SQLException e) {
  // 只抛新异常，e 被丢弃
  throw new ServiceException(
      "查询用户失败");
  // 栈轨迹里再也看不到
  // 到底是哪条 SQL 出的错
}
✓ 保留异常链
catch (SQLException e) {
  // 把 e 作为 cause 传入
  throw new ServiceException(
      "查询用户失败", e);
  // 栈轨迹打印 "Caused by: ..."
  // 根因清清楚楚
}
面试高频 —— 自定义异常的构造器：写自定义异常时，一定要提供带 Throwable cause 参数的构造器并 super(message, cause)，否则你的异常类根本无法承载异常链。另外别忘了：e.printStackTrace() 只该出现在最顶层或调试时，业务代码里应统一用日志框架 log.error("msg", e)，让它带着完整的 "Caused by" 链落到日志系统。

十、异常处理的最佳实践清单

原则说明
早抛出，晚捕获发现非法参数立即抛（fail-fast）；catch 放在有能力处理的那一层
捕获具体类型别用 catch (Exception e) 一网打尽，会连编程 bug 一起吞
不要吞异常catch 里至少记日志或重新抛出，空 catch 是重罪
关资源用 twr凡是 AutoCloseable，一律 try-with-resources，别手写 finally
异常不做流程控制用异常控制正常逻辑（如"抛异常表示循环结束"）极慢且反直觉
参数校验用非受检非法入参抛 IllegalArgumentException，null 抛 NullPointerException
性能冷知识 —— 别把异常当返回值：抛异常时 JVM 需要填充栈轨迹（fillInStackTrace），这一步要遍历整个调用栈，代价远高于普通 return。所以用异常做正常流程控制（比如遍历到末尾抛异常来跳出）会带来巨大性能损失。异常就该用在异常的场景。（题外话：如果确实需要高频轻量异常，可重写 fillInStackTrace 返回 this 来省掉栈填充，但这属于特殊优化，慎用。）
一句话收束：异常体系的精髓是用继承树把"你写错了（unchecked）"和"环境靠不住（checked）"分开，让编译器只在后者上强制你面对；Error 别碰、空 catch 别写、finally 别 return、异常链别丢。而 try-with-resources 是对"忘记关资源 + 异常屏蔽"这两个经典 bug 的语言级根治——只要实现 AutoCloseable，编译器就保证逆序关闭、并用 suppressed 机制让每一个异常都不丢失。能用 twr 就绝不手写 finally。
