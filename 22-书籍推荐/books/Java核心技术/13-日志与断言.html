日志与断言

两件都在"记录程序运行状态"的事：日志陪你走完生产环境的每一天，断言只在开发期替你抓住"绝不该发生"的假设。用对了，排障如破案；用错了，反而埋雷。

一、先问一句：为什么不用 System.out.println？

几乎每个人的第一次调试都是 System.out.println(x)。它能用，但一旦项目上了规模，问题就全冒出来了。日志框架存在的意义，正是逐条解决这些痛点——理解"它解决了什么"，比背 API 重要得多。

痛点System.out.println日志框架
能否分级不能，全都一股脑打印DEBUG / INFO / WARN / ERROR 分级，可按级过滤
上线后能否关要删代码、重新编译改配置文件即可，无需动代码
输出到哪只能到控制台文件、滚动文件、网络、数据库…可同时多路
有没有上下文只有你手写的那点信息自动带时间、线程、类名、行号
性能System.out 内部同步、还拖慢 IO可异步、级别不满足时几乎零开销
本质区别：println 是"写死"的输出——写什么、去哪、要不要，全在编译期定死。日志框架把这几件事变成运行期可配置：代码里只声明"我要在 DEBUG 级别记这句话"，至于它到底显不显示、去文件还是控制台，交给外部配置决定。这是"策略与机制分离"设计思想在日常开发中的一次体现。

二、JDK 自带的 java.util.logging（JUL）

《Java 核心技术》讲的是 JDK 内置的 java.util.logging（简称 JUL，Java 1.4 引入）。它不需要任何第三方依赖，理解它能帮你看清所有日志框架的公共骨架：Logger（记录器）、Level（级别）、Handler（输出目的地）、Formatter（格式化器）。

JulDemo.javajava.util.logging 最小示例
import java.util.logging.*;

public class JulDemo {
    // 惯例：用类的全限定名当 logger 名字，天然形成命名层级
    private static final Logger log =
        Logger.getLogger(JulDemo.class.getName());

    public static void main(String[] args) {
        log.info("服务启动");                   // INFO，默认可见
        log.warning("磁盘剩余不足 10%");         // WARNING
        log.fine("命中缓存 key=42");           // FINE，默认级别 INFO 下不显示

        try {
            Integer.parseInt("abc");
        } catch (Exception e) {
            // 记录异常要连同堆栈：把 Throwable 作为最后一个参数传进去
            log.log(Level.SEVERE, "解析失败", e);
        }
    }
}

JUL 的 Logger 按名字里的"点"组成一棵树：com.demo.User 的父 logger 是 com.demo，再往上是 com，根是名字为空串的 root logger。子 logger 没配自己的级别 / handler 时，会沿着树向上继承。这套设计让你能"给整个 com.demo 包统一调级别"，而不必逐个类去设。

"" (root logger, 默认 Level=INFO, 挂 ConsoleHandler) | com | com.demo <-- 在这一层 setLevel(FINE) | +--------+--------+ | | com.demo.User com.demo.Order <-- 未单独配置, 继承上层的 FINE | log.fine(...) 先过自己的级别门槛 -> 再交给继承来的 Handler 输出
新手第一坑：调了级别却没输出。JUL 有两道闸门：Logger 的 Level 和 Handler 的 Level。就算 logger.setLevel(Level.FINE) 放行了 FINE，默认那个 ConsoleHandler 自己的级别仍是 INFO，会把 FINE 再挡回去。必须两道都调（handler 也 setLevel(FINE)）才看得到。这正是"日志消息要闯过一连串过滤器"模型的直接后果。

三、日志级别：一根轴上的取舍

级别的核心思想是：给每条日志标一个"重要度"，运行期设一个阈值，只放行 ≥ 阈值的消息。开发时把阈值调低（看到全部细节），生产时调高（只留关键信息，省 IO、省磁盘）。JUL 与业界主流（SLF4J / Logback）用词不同，但轴是一样的：

JUL 级别数值SLF4J / Logback 对应什么时候用
SEVERE1000ERROR功能失败、异常，需要人来处理
WARNING900WARN不正常但还能撑，潜在风险
INFO800INFO关键业务节点（启动、下单成功）
CONFIG700（无，并入 INFO/DEBUG）配置信息
FINE / FINER / FINEST500 / 400 / 300DEBUG / TRACE调试细节、方法进出、变量值

再加上两个"边界值"：Level.ALL（放行一切）和 Level.OFF（关闭一切）。判断一条消息是否输出，就是一个数值比较：

msg.level.value ≥ logger.level.value ⇒ 放行
怎么定级别，记住一句话："这条日志半夜三点会不会把人吵醒？"会 → ERROR；"要不要有人第二天看一眼？"要 → WARN；"平时不用看，出事时用来还原现场"→ DEBUG。生产环境阈值通常设在 INFO，把海量 DEBUG 挡在门外。

四、现代生态：为什么人人都用 SLF4J？

虽然 JDK 自带 JUL，但真实项目里几乎没人直接用它。主流是 SLF4J（门面）+ Logback 或 Log4j2（实现）。要理解这个格局，得先懂"门面（facade）"这个词。

历史上日志框架百花齐放：Log4j、JUL、Commons Logging、Logback……如果你的代码直接调 Log4j 的 API，那引入的第三方库若用的是 JUL，最后一个应用里就有好几套日志系统各配各的，配置乱成一锅粥。SLF4J 的解法是：让所有代码只面向一套统一 API（门面）编程，运行期再"绑定"到某个具体实现。

你的代码 / 各种第三方库 | 只依赖 org.slf4j.Logger (统一 API, 门面) v +----------------------------+ | SLF4J API | <-- 编译期只认它 +----------------------------+ | 运行期挑一个 binding 绑定 v +----------+ +----------+ +----------+ | Logback | | Log4j 2 | | JUL | <-- 三选一, 真正干活 +----------+ +----------+ +----------+ | 输出到 控制台 / 文件 / 滚动文件 / ELK ...
类比 JDBC 你就秒懂了：SLF4J 之于日志，就像 java.sql（JDBC 接口）之于数据库。你写代码只调 Connection、Statement，至于底下是 MySQL 还是 PostgreSQL，换个驱动 jar 就行，业务代码一行不改。SLF4J 也一样：换日志实现只换依赖，log.info(...) 照旧。题外话：SLF4J、Log4j、Logback 三者的作者是同一人 Ceki Gülcü——他先造 Log4j，再造门面 SLF4J，最后造出 SLF4J "原生实现" Logback。
方案定位取舍
SLF4J门面（只有 API）必选。让代码与具体实现解耦
Logback实现，SLF4J 原生Spring Boot 默认；配置友好、无桥接开销
Log4j 2实现异步性能强；配 SLF4J 需 log4j-slf4j2-impl 桥
JULJDK 自带实现零依赖；功能与性能较弱，罕用于新项目
联系 Spring：你 new 一个 Spring Boot 项目，spring-boot-starter 里就已经带了 spring-boot-starter-logging = SLF4J + Logback。所以你在 Spring 里写的 private static final Logger log = LoggerFactory.getLogger(Xxx.class);，走的正是"SLF4J 门面 + Logback 实现"这条链路——你早就在用了，只是没意识到。

五、性能陷阱：字符串拼接 vs 占位符

这是 SLF4J 相对 JUL 最实用的改进，也是最高频的代码审查点。看这两种写法的差别——它决定了在日志级别关闭时，会不会有一堆无用功白白烧掉 CPU。

✗ 反例：字符串拼接
// 无论 debug 开不开，右边的
// 拼接和 toString() 都会先执行！
log.debug(
   "user=" + user
   + " order=" + order.toJson());

// 若 DEBUG 被关闭：
// - 字符串照样拼好
// - order.toJson() 照样跑一遍
// - 结果算完却被直接丢弃
// 高频路径上就是纯浪费
✓ 正例：占位符 {}
// 参数按需惰性求值：
// 只有确定要输出时才拼接
log.debug(
   "user={} order={}",
   user, order.toJson());

// 若 DEBUG 被关闭：
// - 方法内部先判级别
// - 不满足就直接 return
// - {} 从不做拼接
// 注意: order.toJson() 作为实参
// 仍会被求值 -> 见下方提示
占位符不是万能的。log.debug("{}", expensiveCall()) 里，expensiveCall() 是方法实参，Java 是严格求值——不管日志级别开没开，它都会先被算出来再传进去。占位符省掉的只是"字符串拼接 / toString()"的开销。若参数本身计算很贵，要么用 if (log.isDebugEnabled()) 包起来，要么用 SLF4J 2.0 的 log.atDebug().addArgument(() -> expensiveCall()).log("{}") 传 Supplier 惰性求值。
小结（日志部分）：面向 SLF4J 门面 编程，实现交给 Logback；用 占位符 {} 而非 + 拼接；记异常时把 Throwable 作为最后一个参数传进去以保留堆栈（log.error("msg", e)），千万别写成 log.error("msg" + e)（那样只有一行 message，堆栈全丢）。

六、断言 assert：给"绝不可能"上把锁

断言是一种内部一致性检查：你在代码里断定"到这一行，某个条件一定成立"。如果它居然不成立，说明程序逻辑本身出了 bug，抛出 AssertionError 让你当场发现。语法只有两种：

AssertDemo.javaassert 的两种语法
// 形式一：assert 布尔表达式;
assert list.size() > 0;

// 形式二：assert 布尔表达式 : 描述信息;   （信息进入 AssertionError）
int balance = account.getBalance();
assert balance >= 0 : "余额不可能为负: " + balance;

// 典型用法：断定 switch 不会走到 default（枚举穷尽性）
switch (color) {
    case RED -> handleRed();
    case GREEN -> handleGreen();
    default -> throw new AssertionError("未知颜色: " + color);
}

最反直觉的一点：断言默认是关闭的。直接 java AssertDemo 运行，所有 assert 语句形同不存在；必须加 JVM 参数 -ea（-enableassertions）才生效，-da 关闭。这背后是编译器的一个巧妙实现：

字节码视角assert 编译后的真面目
// 你写的：
assert x > 0 : "x must be positive";

// javac 实际生成的等价逻辑（每个类有个合成的静态字段 $assertionsDisabled）：
if (!AssertDemo.$assertionsDisabled && !(x > 0))
    throw new AssertionError("x must be positive");

// $assertionsDisabled 在类加载时由 desiredAssertionStatus() 决定。
// 未加 -ea 时它为 true，那个 && 短路 -> 检查开销几乎为零。
为什么这么设计？断言是给开发和测试期用的自检工具，不该拖累生产性能，也不该在生产环境改变程序行为。默认关闭 + 短路判断，让你能在代码里到处撒断言当"活文档"，上线后又不付出任何运行代价。补充：AssertionError 继承自 Error 而非 Exception——它表达的是"程序进入了不可能状态"，本就不该被 catch 恢复。（回顾上一篇的异常体系：Error 属于不应捕获的一类。）

七、断言 vs 异常：别用错工具

初学者最容易混淆这两者。一句话区分：异常处理"预期内会发生的坏情况"（外部输入非法、文件不存在、网络断），断言检查"我认为绝不会发生的情况"（内部逻辑不变量）。前者面向"环境不可控"，后者面向"代码自己别写错"。

维度断言 assert异常 throw
检查什么程序内部的不变量、"不可能"的状态预期内的错误、非法输入、外部故障
面向对象程序员（发现 bug）调用方 / 用户（可恢复的错误）
生产环境默认关闭，通常不开始终生效
能否被捕获恢复是 Error，不该 catchException，应按需处理
校验公共 API 入参不该用该用（IllegalArgumentException 等）

下面这个"校验公共方法参数"的例子，是最经典的误用：

✗ 反例：用断言校验入参
public void transfer(int amount){
   // 公共方法, amount 来自调用方
   assert amount > 0;
   // 生产环境断言默认关闭!
   // 于是这行等于不存在,
   // 负数金额畅通无阻 -> 资损
   doTransfer(amount);
}
✓ 正例：入参用异常，内部才用断言
public void transfer(int amount){
   if (amount <= 0)          // 参数校验:始终生效
      throw new IllegalArgumentException(
         "amount must > 0: " + amount);
   int newBalance = doTransfer(amount);
   // 内部不变量:相信自己的算法
   assert newBalance >= 0
      : "余额算成负数, 逻辑有 bug";
}
断言的另外两个坑：
1. 断言表达式里绝不能有副作用。写 assert list.remove(x);，开 -ea 时会删元素、关掉时不会——程序行为随开关变化，这是灾难。
2. 别拿断言当参数校验的偷懒替身。上面那个 transfer 就是活例子：生产默认关断言，你以为的"保护"其实根本没执行。校验外部输入永远用显式的 if + throw。
一句话收束：日志面向运维、贯穿生产全程——用 SLF4J 门面、分级别、占位符、留堆栈；断言面向开发、只活在测试期——用来固化"绝不可能"的内部不变量，默认关闭、零成本、无副作用。判据很简单："坏情况来自外部且预期会发生" → 异常；"坏情况意味着我代码写错了" → 断言。选错工具，轻则日志刷屏，重则生产资损。
