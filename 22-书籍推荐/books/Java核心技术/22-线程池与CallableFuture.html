线程池与 Callable、Future

别再为每个任务 new 一个线程——把线程当成昂贵的"班组"复用起来，再用 Future 把异步的结果"寄存单"拿回来。

一、为什么需要线程池：线程是昂贵的操作系统资源

初学并发时最自然的写法是"来一个任务，new Thread 一个线程去跑"。这在演示代码里没问题，但放到真实服务里是灾难。原因要回到线程的本质：一个 Java 平台线程（platform thread）在底层一对一映射到一个操作系统内核线程，创建它需要向 OS 申请内核数据结构、分配默认约 1MB 的栈空间，并伴随一次系统调用。

为什么这么设计？——池化(pooling)是对"昂贵资源反复创建/销毁"的通用解法。数据库连接昂贵，于是有连接池；线程昂贵，于是有线程池。它们的核心思想一模一样：预先创建一批资源、任务来了借一个用、用完还回去而不是销毁。这样把"创建成本"从每个任务身上摊掉，只在池初始化时付一次。同时池还给了你一个限流的闸门——线程数有上限，不会因为瞬时涌入十万请求就 new 出十万线程把机器拖垮。

用一个生活类比：new Thread 每次都现招一个临时工、干完就辞退——招聘和解雇的成本比干活本身还高；线程池则是养了一个固定班组，任务贴到工单板上，谁空了谁来揭榜，干完回去待命。三个直接收益：

  • 降低资源消耗：复用已创建的线程，省掉反复创建/销毁的开销。
  • 提高响应速度：任务到达时线程已经就绪，无需等待线程创建。
  • 可管、可控：统一分配、调优、监控，并通过队列 + 最大线程数做限流，避免资源耗尽。
✗ Bad · 每个任务一个线程
while (req = accept()) {
  // 每来一个请求就现造线程
  new Thread(() -> handle(req))
      .start();
}
// 洪峰下线程数无上限
// -> 内存被栈吃光、CPU 疯狂切换
// -> OutOfMemoryError / 系统假死
✓ Good · 提交给线程池复用
ExecutorService pool = /*见下*/;
while (req = accept()) {
  // 提交到固定班组，超载自动排队
  pool.execute(() -> handle(req));
}
// 线程数封顶、任务排队缓冲
// -> 平稳、可预测、不 OOM

二、ThreadPoolExecutor 的七个参数与调度流程

Java 的线程池核心实现是 java.util.concurrent.ThreadPoolExecutor（JUC 由 Doug Lea 设计）。理解它的关键，是把它的七个构造参数和一个任务进来后的调度顺序吃透——面试几乎必问。

参数含义类比 / 要点
corePoolSize核心线程数常驻的"正式工"，默认不因空闲被回收
maximumPoolSize最大线程数核心 + 应急"临时工"的上限
keepAliveTime空闲存活时间临时工空闲超时后被裁掉，退回到核心数
workQueue任务阻塞队列忙不过来时的"工单缓冲区"，务必有界
threadFactory线程工厂给线程起名字，出问题时日志/dump 能定位
handler拒绝策略队列满 + 线程满时的兜底动作
unit时间单位配合 keepAliveTime，如 TimeUnit.SECONDS

当你调用 execute(task)，线程池不是"有空闲就跑、没空闲就排队"这么简单。它遵循一个固定的、优先"攒够核心线程再排队、排满队再扩容、扩到顶才拒绝"的顺序：

execute(task) │ ▼ 运行线程数 < corePoolSize ? ── 是 ──▶ 新建核心线程立即执行 │ 否 ▼ 工作队列 workQueue 没满 ? ──── 是 ──▶ 任务入队，等空闲线程来取 │ 否(队列满) ▼ 运行线程数 < maximumPoolSize ? ─ 是 ─▶ 新建"临时工"线程救急 │ 否(线程也满) ▼ 触发拒绝策略 RejectedExecutionHandler
最反直觉的一点：队列没满前，线程池不会扩到 maximumPoolSize。很多人以为"任务多了就会一路开到最大线程数"，其实只有队列被塞满之后才会去创建非核心线程。这也解释了一个经典坑：如果你用无界队列（如默认的 LinkedBlockingQueue），队列永远填不满，于是 maximumPoolSize 形同虚设、拒绝策略永远不触发——任务只会在队列里无限堆积，最终 OOM。所以队列必须有界。

三、别用 Executors 快捷工厂：藏着 OOM 地雷

Executors 工具类提供了 newFixedThreadPool / newCachedThreadPool 等一行搞定的工厂方法，看着方便，但《阿里巴巴 Java 开发手册》明令禁止用它们创建线程池。原因就在上一节的坑里——它们的默认参数暗藏无界资源：

工厂方法参数真相地雷
newFixedThreadPool(n)core = max = n，队列 LinkedBlockingQueue队列无界(容量 Integer.MAX_VALUE)，任务堆积 → OOM
newSingleThreadExecutor()单线程，无界队列同上，队列堆积 → OOM
newCachedThreadPool()core=0，max=Integer.MAX_VALUE，SynchronousQueue线程数无上限，任务多 → 线程爆炸 → OOM
newScheduledThreadPool(n)定时/周期任务，延迟队列队列无界，同样有堆积风险
✗ Bad · 图省事用快捷工厂
// 队列无界，10 万任务全堆进去
ExecutorService p =
  Executors.newFixedThreadPool(10);

// 或：线程数无上限
ExecutorService q =
  Executors.newCachedThreadPool();
// 参数被藏起来，出事才发现
✓ Good · 手动 new，参数透明可控
ThreadPoolExecutor p =
  new ThreadPoolExecutor(
    10, 20,
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200),
    namedFactory,
    new ThreadPoolExecutor.CallerRunsPolicy());
// 队列有界 + 拒绝策略兜底
PoolConfig.java手动构建，掌控每个参数
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

// 给线程起有意义的名字，是生产排障的救命细节
ThreadFactory factory = new ThreadFactory() {
    private final AtomicInteger n = new AtomicInteger(1);
    public Thread newThread(Runnable r) {
        return new Thread(r, "order-worker-" + n.getAndIncrement());
    }
};

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    4,                             // corePoolSize：常驻核心线程
    8,                             // maximumPoolSize：救急上限
    60L, TimeUnit.SECONDS,         // keepAliveTime：临时工空闲存活时间
    new ArrayBlockingQueue<>(100), // 有界队列！防止无限堆积
    factory,
    new ThreadPoolExecutor.CallerRunsPolicy()); // 满了让提交者自己跑 = 背压

四、四种拒绝策略：满载时的兜底动作

当队列满了、线程也到 maximumPoolSize 了，新任务无处安放，就由 RejectedExecutionHandler 决定怎么办。JDK 内置四种，取舍在"丢数据"和"降速度"之间：

策略行为适用场景
AbortPolicy（默认）抛 RejectedExecutionException要求任务不能悄悄丢，调用方需感知失败
CallerRunsPolicy让提交任务的线程自己执行它天然背压(backpressure)：提交方被拖慢，从源头降速
DiscardPolicy静默丢弃新任务，不报错可容忍丢失的场景（如打点、监控采样）
DiscardOldestPolicy丢掉队列最老的任务，再重试提交只关心最新数据（如实时行情覆盖旧值）
为什么 CallerRunsPolicy 常是好默认？它把"过载"这个信号反向传导给了任务来源：主线程被迫亲自处理一个任务，就没法马上提交下一个，提交速率自然下降，给线程池腾出喘息时间——这正是背压的精髓。相比直接丢数据或直接抛异常，它更平滑。当然前提是任务能容忍被同步执行、且不会造成主线程长时间阻塞。

五、Runnable 交不出结果，于是有了 Callable + Future

线程池能跑任务，但 Runnable 有两个先天缺陷：run() 方法没有返回值，而且不能抛受检异常（签名里没有 throws）。可现实中我们常需要"异步算一个结果拿回来"——比如并行查三个下游服务再汇总。Java 5 为此引入了 Callable 和 Future。

对比RunnableCallable<V>
核心方法void run()V call() throws Exception
返回值无有，泛型 V
受检异常不能抛可以抛
提交方式execute() 或 submit()只能 submit()

Future 是对"一个还没算完的结果"的抽象——像去干洗店拿到的一张取衣寄存单：衣服还没洗好，但单子先给你；等你想要时凭单子去取（get()），没好就在柜台等（阻塞），也可以中途问洗好没（isDone()），甚至反悔取消（cancel()）。

FutureDemo.java提交 Callable，凭 Future 取结果
// Callable 有返回值，还能抛受检异常
Callable<Integer> task = () -> {
    Thread.sleep(1000);       // 假装在做耗时计算/远程调用
    return 6 * 7;
};

Future<Integer> future = pool.submit(task);  // 立即返回"寄存单"，不阻塞

// ...这中间主线程可以先干别的、或再提交别的任务并行...

try {
    Integer r = future.get(2, TimeUnit.SECONDS); // 凭单取货：阻塞等结果，带超时
    System.out.println(r);                    // 42
} catch (TimeoutException e) {
    future.cancel(true);       // 超时了：尝试中断正在跑的任务
} catch (ExecutionException e) {
    Throwable cause = e.getCause(); // 任务内部抛的异常被包在这里
}
联系 JDK 源码：submit 到底做了什么？无论你提交的是 Runnable 还是 Callable，submit 内部都会把它包装成一个 FutureTask——它同时实现了 Runnable（能被线程池跑）和 Future（能取结果），是"任务 + 结果容器"的合体。call() 跑完后，结果或异常被存进 FutureTask 的一个字段；你的 get() 就是去读这个字段，读不到（还没算完）就把当前线程挂起等待，算完后由执行线程唤醒。ThreadPoolExecutor 内部的工作线程封装为 Worker，它继承 AQS 实现了一把不可重入锁，用来精细控制"正在执行任务的线程"能否被中断。

六、两个致命陷阱：吞掉的异常与阻塞的 get()

Callable/Future 用起来爽，但两处坑让无数人栽跟头。

陷阱一：submit 的任务抛异常，你可能永远发现不了

用 execute() 提交时，任务里抛出的未捕获异常会一路冒泡到线程的 UncaughtExceptionHandler，通常打印到控制台，你能看见。但用 submit() 提交时，异常被捕获并封存进了 Future——如果你从不调用 get()，这个异常就被彻底吞掉，任务像是"静默失败"了，排查时一头雾水。

✗ Bad · submit 后不管 Future
pool.submit(() -> {
  int x = 1 / 0;   // 抛 ArithmeticException
});
// 没人调 get()，异常被封进 Future
// -> 控制台一片安静，任务静默死掉
// -> 线上问题极难定位
✓ Good · 取结果或换 execute
Future<?> f = pool.submit(() -> {
  int x = 1 / 0;
});
f.get();  // 异常在此以 ExecutionException 抛出

// 或：纯副作用任务用 execute，
// 异常会走 UncaughtExceptionHandler
pool.execute(() -> riskyWork());

陷阱二：get() 是阻塞的，用不好等于把异步写成了同步

future.get() 会阻塞当前线程直到结果就绪。如果你提交完一个任务，紧接着就 get()，那这段代码和直接同步调用没有任何区别——异步的意义被你亲手抹掉了。正确姿势是：先把所有并行任务都 submit 出去拿到一批 Future，最后再统一 get，这样任务才是真正并行的。另外，永远优先用带超时的 get(timeout, unit)：无参 get() 会无限期等待，一旦某个下游卡死，你的线程就被永久占用，可能引发线程池耗尽的连锁雪崩。
ParallelFanout.java正确的并行姿势：先全提交，再统一取
List<Callable<Integer>> tasks = List.of(this::queryA, this::queryB, this::queryC);

// 一次性全部提交出去 —— 三个任务此刻已在并行跑
List<Future<Integer>> futures = new ArrayList<>();
for (Callable<Integer> t : tasks) futures.add(pool.submit(t));

// 全提交完之后，才逐个取结果并汇总
int sum = 0;
for (Future<Integer> f : futures) sum += f.get(3, TimeUnit.SECONDS);
// 总耗时 ≈ max(A,B,C)，而不是 A+B+C

七、如何优雅关闭线程池

线程池里的核心线程默认永不退出，如果不主动关闭，JVM 会因为还有非守护线程存活而无法退出。关闭有两个档位：

  • shutdown()：温和关闭。不再接收新任务，但会把已提交的任务（包括队列里排队的）全部执行完，然后各线程自然退出。
  • shutdownNow()：强硬关闭。尝试interrupt正在执行的线程，清空并返回队列中尚未执行的任务列表。注意它只是"尝试中断"，任务若不响应中断则停不下来。
  • awaitTermination(timeout, unit)：在 shutdown() 之后阻塞等待，给正在跑的任务一段收尾时间，返回是否在超时内全部结束。
GracefulShutdown.java推荐的两段式优雅关闭
pool.shutdown();                       // 1. 停止接单，让存量任务跑完
try {
    if (!pool.awaitTermination(30, TimeUnit.SECONDS)) {
        pool.shutdownNow();              // 2. 超时还没跑完，强制中断
    }
} catch (InterruptedException e) {
    pool.shutdownNow();
    Thread.currentThread().interrupt(); // 恢复中断标志，别把中断吞掉
}

八、线程池该开多大：一条经验公式

核心线程数不是拍脑袋定的。它取决于任务是"算得多"还是"等得多"——即 CPU 密集型 vs IO 密集型。经典参考公式（出自《Java 并发编程实战》）：

Nthreads = Ncpu × Ucpu × (1 + W / C)

其中 N_cpu 是核数，U_cpu 是目标 CPU 利用率（0~1），W/C 是等待时间 / 计算时间之比。直觉很简单：

  • CPU 密集型（加解密、压缩、纯计算）：W≈0，线程数 ≈ 核数，多开也只是徒增上下文切换。经验值 N_cpu + 1（+1 是为了在偶发缺页/中断时补位）。
  • IO 密集型（远程调用、读数据库、读文件）：线程大部分时间在等，W 远大于 C，可以远超核数，让 CPU 在某个线程等 IO 时去跑别的线程，把 CPU"喂饱"。
公式是起点不是终点。真实业务里 W/C 很难精确测量，任务也常是混合型。正确做法是：以公式估个初值，然后压测 + 监控（关注 CPU 利用率、队列长度、任务排队时长、GC），再迭代调优。切忌只背公式不做实测。另外，不同性质的任务不要共用一个线程池——把 CPU 密集和 IO 密集混在一个池里，参数怎么调都别扭，应按业务隔离成多个池，也能防止一类任务拖垮另一类（舱壁隔离）。

九、串起来：Spring、虚拟线程与 Future 的局限

你在业务里几乎不会裸用 ThreadPoolExecutor——Spring 已经封装好了。@Async 注解 + ThreadPoolTaskExecutor（本质就是对 ThreadPoolExecutor 的封装，参数一一对应）让你一个注解就把方法异步化；@Scheduled 背后则是 ScheduledThreadPoolExecutor。理解了本节的七参数与调度流程，你就能看懂 Spring 那些 corePoolSize/queueCapacity 配置到底在配什么。

Java 21 的新变量：虚拟线程(Virtual Thread)。Project Loom 带来的虚拟线程极其轻量（不再 1:1 绑定 OS 线程，栈按需增长），Executors.newVirtualThreadPerTaskExecutor() 可以"每个任务一个虚拟线程"而不用担心创建成本——这在过去是禁忌，现在对 IO 密集型任务反而是推荐写法。但注意：虚拟线程不需要池化（它便宜到无需复用），而 CPU 密集型任务仍应交给传统的平台线程池。池化的思想没变，只是"什么资源昂贵到值得池化"的答案变了。

最后留一个悬念：Future 虽好，却有硬伤——get() 只能阻塞等待，无法注册"算完了自动回调";多个 Future 也无法方便地编排（比如"A 和 B 都完成后再跑 C"只能靠手动 get 拼凑，又退回阻塞)。这正是下一节 CompletableFuture 要解决的问题——它让异步结果可以链式组合、非阻塞回调。

一句话收束：线程是昂贵的 OS 资源，所以要用线程池复用与限流——核心是 ThreadPoolExecutor 的七参数与"核心满→入队→队满扩容到 max→再满则拒绝"的调度顺序；队列务必有界、拒绝策略要选对，别用藏着无界地雷的 Executors 快捷工厂。要拿异步结果就用 Callable(有返回值、可抛受检异常) + Future(取货寄存单)，但要警惕submit 吞异常和get() 阻塞两大坑，且优先 get(timeout)、先全提交再统一取。用完记得优雅 shutdown。而 Future 无法回调与编排的短板，正是下一站 CompletableFuture 的舞台。
