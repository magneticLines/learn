HashMap 的实现原理

用一个数组、一个哈希函数，把"按 key 找 value"从遍历 O(n) 压到平均 O(1)——这是数据结构里最经典的一次空间换时间。

一、要解决的问题：如何做到 O(1) 查找

想象你有一万个用户，要按用户名查手机号。如果放在数组里挨个比对，最坏要比一万次（O(n)）；如果用有序结构做二分（TreeMap），也要 O(log n)。HashMap 的野心是：不管有多少数据，一次定位就找到——平均 O(1)。

它的思路借鉴了数组"按下标随机访问"的天赋：数组之所以快，是因为 arr[i] 能通过 基地址 + i × 元素大小 一步算出内存地址。HashMap 就想：能不能把任意 key（字符串、对象……）也算成一个数组下标？这个"把 key 变成下标"的函数，就是哈希函数。

下标 index = hash(key) & (数组长度 - 1)
核心直觉——哈希 = 建索引：就像图书馆按书名首字母分区，你找《Java》直接走到"J"区，而不是从头翻。哈希函数就是那个"分区规则"，把 key 直接映射到它该待的"桶（bucket）"。区分得越均匀，每个区书越少，找得越快。

二、底层数据结构：数组 + 链表 + 红黑树

HashMap 的骨架是一个数组，源码里叫 table，数组元素类型是 Node<K,V>（Java 8 命名，Java 7 里叫 Entry）。每个数组位置称为一个桶（bucket）。当多个 key 算出同一个下标时（哈希冲突），它们就在同一个桶里用链表串起来；链表太长（Java 8 起）会转成红黑树。

table (Node[] 数组) ┌────┐ │ 0 │──> null ├────┤ │ 1 │──> (k1,v1) ──> (k9,v9) ──> null ← 链表：桶内冲突 ├────┤ │ 2 │──> null ├────┤ │ 3 │──> (k3,v3) ──> null ├────┤ │ 4 │──> ●红黑树根 ← 链表过长转红黑树 │ │ / \ │ │ node node ... (>=8 个且容量>=64) └────┘ ↑ 每个格子叫一个"桶 bucket"；数组长度必是 2 的幂
HashMap.java (JDK 源码节选)核心字段与 Node 结构
transient Node<K,V>[] table;        // 桶数组，第一次 put 时才懒初始化
transient int size;                   // 实际键值对数量
int threshold;                        // 扩容阈值 = 容量 × 负载因子
final float loadFactor;               // 负载因子，默认 0.75

static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;   // 缓存的 hash 值，避免每次重算（key 不变，hash 不变）
    final K key;
    V value;
    Node<K,V> next;    // 指向同桶下一个节点 —— 这就是链表
}
为什么把 hash 缓存进 Node？因为 key 一旦放进去就不该变（见第八节），它的 hash 也就固定了。缓存下来，扩容重新分桶、查找比对时都能直接复用，省掉大量重复的 hashCode() 调用——这是"用一点空间换很多时间"的典型微优化。

三、哈希函数：为什么要把高位"搅拌"进来

你可能以为下标就是 key.hashCode() % 数组长度。但 JDK 做了两处关键改造。第一，用位与代替取模：当数组长度 n 是 2 的幂时，hash % n 完全等价于 hash & (n-1)，而位运算比除法快得多。

HashMap.java扰动函数 hash()
static final int hash(Object key) {
    int h;
    // key 为 null 时 hash 恒为 0 —— 所以 null 永远落在 0 号桶
    return (key == null) ? 0
         : (h = key.hashCode()) ^ (h >>> 16);
    // 把高 16 位异或到低 16 位：让高位也参与最终下标运算
}

第二处改造就是那句 (h = key.hashCode()) ^ (h >>> 16)，称为扰动（perturbation）。原因藏在下标公式里：hash & (n-1)。数组初期很小（默认 16），n-1 = 15 = 0b1111，位与只会保留 hash 的最低 4 位，高位再怎么不同也被丢弃。如果两个 key 的 hashCode 只是高位不同、低位相同，它们就会撞进同一个桶。

hashCode : 1111 1111 1111 0101 1010 0000 0000 0001 h >>> 16 : 0000 0000 0000 0000 1111 1111 1111 0101 异或结果 : 1111 1111 1111 0101 0101 1111 1111 0100 ← 高位信息"混"进了低位 ↑↑↑↑ & (16-1) 时，这几位才是真正决定下标的位
一句话概括扰动的价值：让 hashCode 的高位也对最终下标产生影响，从而在数组还很小的时候就把元素撒得更均匀，减少冲突。这是一次成本极低（一次异或、一次移位）却收益明显的工程折中。

四、为什么容量必须是 2 的幂

这个约束贯穿整个 HashMap，背后有三个环环相扣的理由：

好处原理
取模变位与仅当 n 为 2 的幂，hash % n 才等于 hash & (n-1)，用位运算提速
下标分布均匀n-1 的二进制是连续的 1（如 1111），位与能保留 hash 低位的全部信息；若 n 非 2 的幂，n-1 中有 0 位，那些位永远取不到，桶被浪费
扩容重哈希极快容量翻倍后，元素要么留在原位 i，要么移到 i + oldCap，只需看 hash 多出来的那一位是 0 还是 1，无需重算 hash（见第六节）
陷阱：new HashMap<>(1000) 并不会得到容量 1000。你传入的初始容量会被 tableSizeFor() 向上取整到最近的 2 的幂——传 1000 实际得到 1024，传 17 得到 32。所以别以为传个数字就精确控制了容量。想装 1000 个元素又不想扩容，正确算法是 (int)(1000 / 0.75) + 1 再交给它取幂。

五、put 的完整流程

把这条主干走一遍，HashMap 的绝大多数细节就串起来了：

put(key, value) │ ├─ 1. 算 hash = hash(key) // 扰动后的 hash ├─ 2. 若 table 为空 → resize() 懒初始化 // 第一次 put 才建数组 ├─ 3. 算下标 i = (n-1) & hash │ ├─ 4. table[i] 为空？ │ 是 → 直接放入新 Node，结束 │ 否 → 发生冲突，进入桶内查找 ↓ │ ├─ 5. 遍历桶内节点，逐个比较： │ hash 相同 && (key == k || key.equals(k)) ? │ 是 → 找到同一个 key，覆盖旧 value（返回旧值） │ 否 → 继续找下一个 │ ├─ 6. 走到链表尾仍没找到 → 尾插一个新 Node │ 若链表长度 >= 8 且容量 >= 64 → 转红黑树 │ └─ 7. ++size > threshold ? → resize() 扩容
get 的对称逻辑先定位桶，再桶内比对
public V get(Object key) {
    int hash = hash(key);
    Node<K,V> e = table[(n - 1) & hash];  // 一步定位到桶
    while (e != null) {                       // 桶内可能有多个（冲突）
        // 先比 hash（int 比较极快，快速排除）
        // 再用 == 比引用，最后才 equals —— 短路优化
        if (e.hash == hash &&
            (e.key == key || (key != null && key.equals(e.key))))
            return e.value;
        e = e.next;
    }
    return null;   // 没找到返回 null（注意：也可能是存了 null 值，见陷阱）
}
高频面试点——为什么比较条件是 hash 相同 && (== 或 equals)？因为不同的 key 可能算出相同的下标（冲突），甚至相同的 hash 值，所以定位到桶只是第一步，桶里可能有好几个 key，必须逐个用 equals 确认到底是不是你要的那个。先比 hash 是为了快速剪枝——hash 都不同的连 equals 都省了。

六、负载因子与扩容（resize）

数组总会装满。HashMap 用负载因子（loadFactor，默认 0.75）来决定"多满就扩容"：当 size > 容量 × 0.75 时，触发 resize()，容量翻倍，并把所有元素重新分配到新数组——这个过程叫rehash（重哈希）。

threshold = capacity × loadFactor   (16 × 0.75 = 12)

0.75 是空间与时间的折中：太大（如 1.0）桶容易挤满、冲突链变长、查询变慢；太小（如 0.5）数组还很空就扩容，浪费内存且频繁 rehash。0.75 让"每个桶平均元素数"维持在较低水平，冲突概率与内存占用达到较好平衡。

Java 8 扩容的神来之笔——不用重算 hash：容量从 16 翻到 32，n-1 从 0b01111 变成 0b11111，只多了最高一位。一个元素的新位置只取决于它 hash 在那一位上是 0 还是 1：是 0 就留在原下标 i，是 1 就移到 i + oldCap。于是整条链被一拆为二，无需对每个元素重新 hashCode() 和取模，扩容成本大幅下降。
扩容前 n=16，桶 i=1 的链： A(...0 0001) → B(...1 0001) ↑ ↑ 多出的那一位: 0 1 扩容后 n=32 拆成两条链： 桶 1 (原位 i) : A ← 多出位为 0，留下 桶 17 (i + oldCap=1+16) : B ← 多出位为 1，前移 16

七、Java 7 vs Java 8：一次影响深远的重构

HashMap 在 Java 8 经历了大改，这些差异是面试的重灾区，也解释了很多"历史 bug"的由来：

维度Java 7Java 8+
桶结构数组 + 链表数组 + 链表 + 红黑树
冲突最坏查询O(n)（链表退化）O(log n)（树化后）
链表插入位置头插法尾插法
扩容时重算下标、可能倒置链表低位/高位拆链，保持相对顺序
并发扩容风险可能形成环形链表导致 CPU 100%不会成环（但仍非线程安全，会丢数据）
树化的三个阈值（记牢）：链表长度达到 TREEIFY_THRESHOLD = 8 且 数组容量 >= MIN_TREEIFY_CAPACITY = 64 时，链表转红黑树；若容量不足 64，会优先扩容而非树化（因为扩容能分散冲突，成本更低）。当树节点因删除/扩容降到 UNTREEIFY_THRESHOLD = 6 时，退化回链表。为什么是 8？在 hash 分布良好时，一个桶里堆到 8 个的概率按泊松分布已低到约千万分之六，正常几乎不会树化——树化是给"极端糟糕的 hash"兜底的保险，不是常态。

八、equals 与 hashCode 契约：最经典的坑

HashMap 定位靠 hashCode，桶内确认靠 equals。所以自定义类做 key 时，必须同时正确重写这两个方法，否则 HashMap 会"抽风"。契约铁律：equals 相等的两个对象，hashCode 必须相等（反之不要求）。

✗ 只重写 equals，忘了 hashCode
class Point {
  int x, y;
  // 重写了 equals 判断 x,y 相等
  // 却用了 Object 默认 hashCode
}
Map<Point,String> m = new HashMap<>();
m.put(new Point(1,2), "A");
m.get(new Point(1,2));  // 返回 null !!
// 两个"相等"的 Point 算出不同 hash，
// 落到不同桶，永远找不到
✓ 两者同步重写
class Point {
  int x, y;
  public boolean equals(Object o){...}
  public int hashCode() {
    return Objects.hash(x, y);
  }
}
// 或直接用 record（自动生成两者）
record Point(int x, int y) {}
// equals 相等 → hashCode 必相等
// get 能稳定命中
可变对象做 key 是隐形炸弹：如果把一个对象放进 HashMap 后又修改了它参与 hashCode 的字段，它算出的桶下标就变了，但它物理上还躺在旧桶里。之后你再拿"新状态"去 get，会去新桶找，自然找不到——数据"人间蒸发"。所以 HashMap 的 key 应当是不可变的（这也是 String 设计成不可变、并缓存 hash 的重要原因，你的 key 用 String 天生安全）。

九、线程安全：HashMap 不是并发容器

HashMap 完全没有做同步。多线程并发 put 时，可能同时命中同一个桶互相覆盖导致数据丢失；并发扩容在 Java 7 甚至会形成环形链表让 get 陷入死循环、CPU 打满。它的迭代器还是 fail-fast 的：迭代过程中被结构性修改会抛 ConcurrentModificationException。

方案做法 / 特点适用
HashMap无同步，最快单线程 / 局部变量 / 只读共享
Hashtable每个方法 synchronized，全表锁，已过时基本不用（遗留代码）
Collections.
synchronizedMap		包一层全表锁，粒度粗低并发、图省事
ConcurrentHashMapJava 8 用 CAS + synchronized 锁单个桶，并发度高高并发首选
联系 Spring：Spring 内部大量用 ConcurrentHashMap 做缓存（如单例 Bean 的 singletonObjects、各种 @Cacheable 后端），因为容器初始化后是"读多写少 + 多线程访问"的典型场景，正是 ConcurrentHashMap 的主场。而方法内部临时用的 Map 用普通 HashMap 就好——不共享就没有并发问题，别为不存在的问题付同步的代价。

一句话收束：HashMap 的本质是「数组做索引骨架 + 哈希函数把 key 变成下标 + 链表/红黑树兜底冲突」。它用扰动函数让高位参与运算、用2 的幂容量把取模变位与并让扩容只需拆链、用0.75 负载因子平衡空间与冲突、用Java 8 红黑树为坏 hash 兜底。三条铁律务必记牢：key 要不可变、equals 与 hashCode 必须成对重写、多线程请换 ConcurrentHashMap。理解了这套设计，你就理解了几乎所有哈希表的共同套路。
