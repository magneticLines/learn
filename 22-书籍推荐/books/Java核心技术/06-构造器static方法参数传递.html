构造器、static、方法参数传递

对象如何诞生、类级别的数据存在哪里，以及那个让无数人栽跟头的结论——Java 永远只有值传递。

一、构造器：对象出生时的"入职登记"

上一篇讲封装时，我们把字段设成 private，靠 setter 一个个赋值。但这样有个漏洞：一个对象可能在"信息还没填全"的半成品状态就被人拿去用了。构造器（constructor）的使命，就是保证一个对象从 new 出来的那一刻起，就处于合法、可用的状态。

类比：构造器就像新员工入职当天必须填完的登记表——没填完工号、部门、姓名，系统里就不允许生成这个"员工对象"。它有三条和普通方法不同的硬规则：

  • 方法名必须和类同名（连大小写都要一致）；
  • 没有返回类型——连 void 都不能写（写了 void 它就退化成一个普通方法，是初学者的高频坑）；
  • 只能由 new 触发调用，新对象的引用由 new 表达式返回。
Account.java构造器与 this() 复用
public class Account {
    private String owner;      // 实例字段：每个对象各有一份
    private double balance;

    // 有参构造器：一出生就带上合法状态
    public Account(String owner, double balance) {
        this.owner = owner;      // this 区分"字段"与"同名形参"
        this.balance = balance;
    }

    // 无参构造器：用 this(...) 复用上面的逻辑，给默认值
    public Account(String owner) {
        this(owner, 0.0);       // 必须是构造器体的第一条语句！
    }
}

// 使用：new 表达式返回的就是这个新对象的引用
Account a = new Account("Alice", 100.0);
为什么构造器没有返回类型？因为它的"返回值"永远是刚创建的那个对象本身，这件事由 new 关键字负责，不需要（也不允许）你在方法签名里再声明一遍。理解这点就能解释一类怪现象："我明明写了个和类同名的方法，怎么 new 的时候没被调用"——多半是手滑加了返回类型，它变成普通方法了。

二、默认构造器的陷阱 & 重载

如果你一个构造器都不写，编译器会偷偷送你一个无参构造器（default constructor），它把字段初始化为默认值（数值 0、boolean false、引用 null）。这就是为什么空类也能 new。

高频编译错误：一旦你手写了任意一个构造器（哪怕只是有参的那个），编译器就不再赠送无参构造器。此时再写 new Account() 会直接编译失败："no suitable constructor found"。这也是 Spring / JPA / Jackson 反序列化反复要求你"补一个无参构造器"的根本原因——很多框架靠反射调用无参构造器来创建对象。

多个构造器同名、靠参数列表不同来区分，这就是构造器重载（overloading）。用 this(...) 让一个构造器调用另一个，可以把初始化逻辑收敛到一处，避免复制粘贴——JDK 里 String、ArrayList 拥有一大堆构造器却不重复写逻辑，用的正是这招。

三、初始化顺序：谁先谁后

一个类里可能同时存在：静态字段、静态初始化块、实例字段、实例初始化块、构造器。它们的执行顺序是面试常考点，本质由"类加载 vs 对象创建"这两个阶段决定：

类第一次被使用（类加载阶段 · 整个 JVM 生命周期仅一次） └─> ① 静态字段赋值 + 静态初始化块 static { ... } 每次 new 一个对象（对象创建阶段 · 每 new 一次都走一遍） ├─> ② 实例字段赋值 + 实例初始化块 { ... } └─> ③ 构造器方法体 Account(...) { ... } 口诀：静态最早且只跑一次，然后"字段/实例块"，最后"构造器体"
联系已学：静态块只在类加载时执行一次，这个特性常被用来做"只做一次"的初始化，比如加载 JDBC 驱动 Class.forName("com.mysql.cj.jdbc.Driver")——驱动类的静态块里会把自己注册进 DriverManager。等学到并发时你还会看到，靠"静态内部类 + 类加载只发生一次"实现的线程安全懒加载单例（Holder 模式）。

四、static：属于"类"而不属于"对象"

static 修饰的成员属于类本身，而不是某个具体对象。类比：一个班级里，"班级人数"这块黑板全班共用、只有一份（static 字段）；而每个学生自己的"学号"写在各自的笔记本上、人手一份（实例字段）。

Robot.java共享计数器 · 静态方法
public class Robot {
    private static int count = 0;   // 类级别：所有 Robot 共享同一份
    private final int id;           // 实例级别：每个机器人各不相同

    public Robot() {
        count++;                    // 每 new 一个，共享计数器 +1
        this.id = count;
    }

    public static int total() {     // 静态方法：用 Robot.total() 调用
        return count;               // 只能碰 static 成员
        // return id;  ✗ 编译错误：非静态字段无法在静态上下文中访问
    }
}
最常见的编译报错：Cannot make a static reference to the non-static field/method。原因很简单：静态方法没有 this——它不依附于任何对象，自然找不到"哪个对象的 id"。所以 main（它是 static 的）里不能直接调用非静态方法，必须先 new 一个对象再通过它调用。反过来，非静态方法里可以随意访问静态成员。

和内存模型串联：JDK 7 及以前，静态变量和类元信息放在"永久代（PermGen）"；JDK 8 起类元信息迁移到元空间（Metaspace，直接用本地内存），而静态变量本身则随 Class 对象放到了堆里。无论怎么演进，核心不变：static 数据的生命周期跟着"类"，不跟着"对象"。JDK 里典型例子是 Integer.valueOf 用一个静态缓存数组缓存了 -128~127 的 Integer，以及全是静态方法的工具类 Math、Collections。

五、方法参数传递：Java 只有值传递

先立结论，再拆解：Java 中方法传参永远是值传递（pass by value），没有引用传递。这句话之所以让人困惑，是因为要先分清两个概念：

  • 值传递：把实参的"值"复制一份交给形参，方法内改形参不影响实参。
  • 引用传递：把实参变量本身（那个存储单元）交给方法，方法内重新赋值会直接影响实参。C++ 的 & 引用、C# 的 ref 才是真引用传递。Java 没有这东西。
形参 = copy( 实参的值 )

关键在于"值"到底是什么：基本类型的值就是那个数字本身；引用类型的值是"引用"（可理解为对象在堆中的地址），而不是对象本身。所以传引用类型时，复制的是那张"藏宝图"，不是宝藏本身——两张一样的藏宝图，指向同一个宝藏。

PassByValue.java基本类型 vs 引用类型
static void tryReset(int n) { n = 0; }            // 改的是副本
static void fill(int[] arr) { arr[0] = 99; }      // 改的是同一个数组的内部
static void reassign(int[] arr) { arr = new int[]{7}; } // 只改副本指向，无效

public static void main(String[] args) {
    int x = 5;
    tryReset(x);
    System.out.println(x);       // 5   基本类型：改副本，原值不变

    int[] data = {1, 2, 3};
    fill(data);
    System.out.println(data[0]);  // 99  副本引用与原引用指向同一数组

    reassign(data);
    System.out.println(data[0]);  // 99  形参改指向不影响实参，仍是老数组
}

为什么 fill 能改成功、reassign 却无效？看内存布局就懂了：

调用 fill(data) 时： 栈 (Stack) 堆 (Heap) ┌────────────────┐ ┌──────────────────┐ │ main: │ │ int[] { … } │ │ data = 0x100 ┼───────────>│ [0] = 1 => 99 │ ├────────────────┤ ┌───>│ [1] = 2 │ │ fill: │ │ │ [2] = 3 │ │ arr = 0x100 ┼───────┘ └──────────────────┘ └────────────────┘ 两个变量各存一份"地址副本"(0x100)，却指向同一个堆对象； 所以 arr[0]=99 改到了原对象。但若 arr = new int[]{7}， 只是让 arr 这份副本改指向新对象，main 里的 data 纹丝不动。
误解：以为是引用传递
static void swap(Account a, Account b) {
    Account t = a;
    a = b;
    b = t;   // 以为能换掉外面的 x, y
}
// 调用后 x、y 毫无变化！
// 换的只是两份"引用副本"，
// 方法一返回它们就被丢弃了。
真相：改对象内部才有效
static void deposit(Account a, double m) {
    a.add(m);  // 顺着副本引用
              // 找到同一个堆对象改它
}
// 调用后原对象余额真的变了。
// 关键：不是"重新赋值形参"，
// 而是"操作形参指向的对象"。
面试杀手锏问题："String / Integer 作参数能被方法改掉吗？" 不能。String、Integer 等都是不可变（immutable）对象，方法内任何"修改"都会生成新对象并让形参副本改指向，原实参毫发无损。想让方法"输出"多个结果，别指望改形参，应当返回对象、或传入一个可变容器（如数组、集合、自定义 DTO）再改它的内部。
一句话收束：Java 只有值传递。传基本类型，复制的是数值；传对象，复制的是引用（地址）而非对象。因此——顺着形参去改它指向的对象内部，外面看得见；给形参重新赋值换个对象，外面看不见。记住这条，swap 失效、"对象参数为什么有时改得动有时改不动"这类问题就再也不会绕晕你。
