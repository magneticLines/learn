基本语法：变量、运算符、控制流

从一块内存到一条指令流：搞懂变量存什么、运算符怎么算、程序如何做决定。

一、变量：给内存里的一块空间起名字

计算机的内存本质上是一排排编了号（地址）的格子。让人去记 0x7ffee3 这种地址显然反人类，于是高级语言引入了变量——用一个可读的名字，绑定到一块能存放特定类型数据的内存空间。Java 是静态强类型语言：变量在声明时就必须确定类型，编译期就把类型钉死，运行期不能随便变。

Variables.java变量声明与初始化
int age = 25;          // 声明 + 初始化，一步到位
double salary;            // 先声明
salary = 12000.5;         // 再赋值
final double PI = 3.14159; // final = 常量，只能赋一次

var name = "Tom";        // Java 10+ 局部变量类型推断，实际类型仍是 String
为什么要有类型？ 类型告诉编译器三件事：这块内存占多大（int 4 字节、long 8 字节）、怎么解读这些二进制位（是补码整数还是 IEEE 754 浮点）、能做哪些操作（字符串能 + 拼接，布尔不能做减法）。有了类型信息，很多错误在编译期就被拦下，而不是等到线上崩溃。

八种基本类型（primitive types）

Java 的数据分两大阵营：基本类型直接把值存在栈上（或对象内部），引用类型（对象）在栈里存一个指向堆的地址。基本类型只有固定的 8 种，且它们的大小与平台无关——这正是「一次编写，到处运行」在数据层面的体现，不像 C 里 int 可能 16/32/64 位随编译器变。

陷阱：默认值只对成员变量生效。 类的字段（实例/静态变量）没赋值会自动初始化为上表默认值；但局部变量没有默认值，声明后不赋值就使用，编译直接报 variable might not have been initialized。这是 Java 刻意的安全设计——逼你显式初始化，避免读到内存里的垃圾值。

二、运算符：编译期就定好的计算规则

运算符看着简单，坑最多的是类型提升与精度。Java 做算术时会先把小类型「提升」到大类型再算：两个 int 相除结果还是 int（直接截断小数），只要有一个操作数是 double，整个表达式就升为 double。

反例 · 整数除法丢精度
double avg = 5 / 2;
// 5、2 都是 int
// 先算 int 除法 = 2
// 再赋给 double = 2.0
// 结果 2.0，不是 2.5！
正例 · 让操作数变浮点
double avg = 5.0 / 2;
// 有一个 double
// 整个升为 double 运算
// 结果 2.5 ✓
// 或 (double) a / b
浮点数不能用 == 判等，也别用来算钱。 0.1 + 0.2 == 0.3 结果是 false——因为 IEEE 754 用二进制表示十进制小数是无限循环的，只能存近似值。金额、税率这类场景要用 java.math.BigDecimal（且必须用字符串构造 new BigDecimal("0.1")，传 double 进去等于把误差也带进来了）。

短路求值：&& 和 || 的「偷懒」

逻辑运算符 &&/|| 是短路的：&& 左边为 false 就不再算右边，|| 左边为 true 就不再算右边。这不只是性能优化，更是一种常用的防御性写法。

ShortCircuit.java短路的实战价值
// 若 user 为 null，短路让右边不执行，避免 NPE
if (user != null && user.isActive()) { doSomething(); }

// 对比：& 和 | 不短路，两边都会求值 —— 这里会抛 NullPointerException
if (user != null & user.isActive()) { crash(); }
类别运算符易踩的坑
算术+ - * / %% 结果符号跟被除数走：-7 % 3 = -1
自增/自减++ --i++（先用后加）vs ++i（先加后用）
关系== != > <对象用 == 比的是地址，不是内容（要用 equals）
逻辑&& || !短路 vs 非短路 & |
位运算& | ^ ~ << >> >>>>> 补符号位，>>> 补 0（无符号右移）
位运算不是「炫技」，JDK 源码里到处都是。 HashMap 用 (n - 1) & hash 代替 hash % n 求桶位（要求容量是 2 的幂，位与比取模快得多）；ArrayList 扩容用 oldCap >> 1 表示「乘以 1.5」里的那半个。理解位运算能帮你读懂集合框架的底层。

三、控制流：让程序「做决定」和「重复干活」

没有控制流，程序只能从上到下一条道走到黑。if/else 提供分支，循环提供重复——这两样加上顺序执行，就是结构化编程的全部积木（图灵完备只需要它们）。

switch：从旧写法到「箭头表达式」

传统 switch 有个经典大坑——忘记 break 会「贯穿」（fall-through）到下一个 case。Java 14 起转正的 switch 表达式（箭头语法）根治了这个问题：不贯穿、能直接返回值、编译器还会检查是否覆盖所有分支。

反例 · 旧 switch 漏 break
switch (day) {
  case 1:
    type = "工作日";
    // 忘了 break！
  case 6:
  case 7:
    type = "周末"; // 被覆盖
    break;
}
// day=1 得到 "周末" ✗
正例 · switch 表达式（14+）
String type = switch (day) {
  case 1, 2, 3, 4, 5 -> "工作日";
  case 6, 7          -> "周末";
  default            -> "非法";
};
// 箭头不贯穿
// 直接把结果赋给 type ✓

三种循环 & break / continue

Loops.javafor / for-each / while
int[] nums = {3, 7, 9, 2};

// 需要索引时用传统 for
for (int i = 0; i < nums.length; i++) {
    if (nums[i] == 9) break;       // 跳出整个循环
}

// 只关心元素、不关心下标 —— 首选 for-each，更不易越界
int sum = 0;
for (int n : nums) {
    if (n % 2 == 0) continue;  // 跳过本轮，进入下一轮
    sum += n;
}

// 次数未知、靠条件驱动 —— 用 while
while (sum > 0) sum -= 1;

该用哪种循环？

场景推荐理由
需要下标 / 反向遍历 / 步长非 1for三段式把控制逻辑集中在一行，一目了然
遍历数组或集合的每个元素for-each无索引即无越界；底层对集合走 Iterator
循环次数取决于运行时条件while如「读到文件末尾为止」，先判断后执行
循环体至少要执行一次do-while先执行后判断，如菜单「输入直到合法」
陷阱：for-each 里不能安全地增删集合元素。 for (String s : list) list.remove(s); 会抛 ConcurrentModificationException——因为 for-each 底层是 Iterator，它靠 modCount 检测结构被改动。要边遍历边删，得显式用 iterator.remove()，或用 list.removeIf(...)。这是新手到中级最高频的报错之一。

四、把三者串起来：一个最小完整例子

变量存状态、运算符做计算、控制流做决策——三者协作就能表达任意逻辑。下面用「判断闰年」把它们组合起来：

闰年 = (year % 4 == 0 && year % 100 != 0) || year % 400 == 0
LeapYear.java变量 + 运算符 + 控制流
int year = 2000;                          // 变量：存状态
boolean isLeap = (year % 4 == 0 && year % 100 != 0)
                 || year % 400 == 0;       // 运算符：算逻辑

if (isLeap) {                             // 控制流：做决策
    System.out.println(year + " 是闰年（2 月 29 天）");
} else {
    System.out.println(year + " 是平年");
}
一句话收束： Java 用静态类型的变量把内存管起来（8 种基本类型大小固定、跨平台一致）；用运算符做计算，牢记「整数除法截断、浮点不能 ==、&&/|| 短路」这三条；用 if/switch/循环组织控制流，优先用不易出错的 switch 表达式 和 for-each。这三块是所有后续知识（对象、集合、并发）的语法地基。
