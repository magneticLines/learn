Collections 工具类与不可变集合

一个静态工具箱，一组"改不了"的集合——看懂"只读视图"和"真不可变"的一字之差，是写出线程安全代码的第一步。

一、Collections 不是 Collection：一字之差的工具类

初学者最容易混的一对名字：Collection（单数，接口，上一节讲过的"一堆元素"的根接口）和 Collections（复数，类）。后者是 java.util 里的一个 工具类——构造器私有、成员全是 static 方法，你永远不会 new Collections()，只会 Collections.sort(...) 这样静态调用。它和 Arrays（数组工具类）、Objects（对象工具类）是同一路数。

为什么要单独造一个工具类？Java 8 之前接口不能有方法实现（没有 default 方法）。如果把 sort、shuffle 这些"对集合做的算法"直接塞进 List 接口，每个实现类都得重写一遍——重复且易错。于是设计者把这些"通用算法"抽到一个静态工具类里，面向 List/Collection 接口编程，一次实现、处处可用。这正是"能力（算法）与数据（容器）分离"的经典手法。

Java 8 有了 default 方法后，一部分能力"回迁"到了接口上：list.sort(cmp)、list.removeIf(...)、coll.forEach(...) 都是接口默认方法。但 Collections 里的 binarySearch、shuffle、frequency、各种包装方法仍然只此一家，工具类并未退场。

二、算法方法：排序、查找、洗牌、统计

把 Collections 当成"集合上的常用算法合集"来记最省事。它们绝大多数面向 List 或 Collection 接口，因此对任意实现都通用。

方法作用关键点 / 复杂度
sort(list) / sort(list, cmp)原地排序底层 TimSort，稳定排序，O(n log n)
binarySearch(list, key)二分查找要求已排序；RandomAccess 表 O(log n)
reverse / shuffle / swap反转 / 随机打乱 / 交换shuffle 用 Fisher-Yates 洗牌算法
max(coll) / min(coll)取最值按自然序或传入的 Comparator
frequency(coll, o)统计某元素出现次数依赖 equals()
nCopies(n, o) / fill生成 n 个副本 / 整表填充nCopies 返回不可变列表
CollectionsAlgo.java算法方法一览
import java.util.*;

List<Integer> nums = new ArrayList<>(List.of(3, 1, 4, 1, 5, 9));

Collections.sort(nums);                 // [1, 1, 3, 4, 5, 9]
int pos = Collections.binarySearch(nums, 4);  // 3（必须先排序！）
int c = Collections.frequency(nums, 1);    // 2（元素 1 出现两次）
int hi = Collections.max(nums);          // 9

Collections.reverse(nums);              // [9, 5, 4, 3, 1, 1]
Collections.shuffle(nums);              // 随机打乱（可传 Random 复现）

// 逆序比较器：常配合 sort 用
Collections.sort(nums, Collections.reverseOrder());  // 降序
陷阱：binarySearch 必须先排序，否则结果无意义。二分查找的前提是"数据有序"，对未排序列表调用它，返回值是不确定的（负数/正数都可能），且不会报错——这种"静默错误"比抛异常更难查。另外若元素不存在，返回的是 -(应插入点) - 1（一个负数），可用 -(ret)-1 反推插入位置。

三、包装方法：给普通集合套一层"能力外壳"

除了算法，Collections 还有一批 xxxList/Set/Map(...) 方法，它们不改变数据，只是给已有集合套一层装饰器（Decorator 模式），返回一个行为被增强/限制的"视图"。三类最常用：

包装方法套上什么外壳典型用途
synchronizedList/Map/Set每个方法加 synchronized老式线程安全（现多被 concurrent 包取代）
unmodifiableList/Map/Set所有写方法抛异常对外暴露"只读视图"，防外部篡改
checkedList/Map/Set插入时做运行时类型检查抓泛型擦除后混入的错误类型
emptyList() / singletonList(x)空 / 单元素不可变集合返回共享常量，零分配
synchronizedList 的隐藏坑：迭代时仍要手动加锁。它只保证单个方法（add、get）原子，但 for-each / 迭代器是"多次调用的复合操作"，多线程下会抛 ConcurrentModificationException。正确写法是遍历时用 synchronized(list){ ... } 把整个迭代包起来。这也是为什么现代并发首选 ConcurrentHashMap、CopyOnWriteArrayList——它们的迭代器天生安全（弱一致性 / 快照）。

四、"只读视图" unmodifiableXxx：一层单向玻璃

Collections.unmodifiableList(src) 返回的不是一份拷贝，而是原集合的一个"只读视图"。它像贴在窗户上的单向玻璃：透过它只能看（读操作转发给原集合），任何写操作（add/remove/set）都会抛 UnsupportedOperationException。

但关键在于——玻璃管得住外人，管不住原主人。视图内部只是持有原集合的引用；只要有人还握着原始集合的引用去修改它，改动会穿透到只读视图上：

UnmodifiableView.java只读视图会"穿透"
List<String> src = new ArrayList<>(List.of("a", "b"));
List<String> ro  = Collections.unmodifiableList(src);

ro.add("c");   // 抛 UnsupportedOperationException —— 视图挡住了写

src.add("c");  // 但从原始引用改，完全合法……
System.out.println(ro); // -> [a, b, c]  改动穿透到了"只读"视图！
面试高频：unmodifiableXxx 只是"不可修改（unmodifiable）"，不是"不可变（immutable）"。要真正安全地对外暴露一份"谁也改不了"的数据，必须做防御性拷贝——先复制一份新集合、再包只读视图（unmodifiableList(new ArrayList<>(src))），或者干脆用下一节的 List.copyOf(src)，它内部会自动拷贝。
只读视图 (view) 不可变拷贝 (copy) Collections.unmodifiableList(src) List.of(...) / List.copyOf(src) │ │ 只读包装(转发) 全新独立对象 │ │ ▼ ▼ ┌──────────────┐ src.add() 穿透 ┌──────────────┐ │ 原始 src │ ◄─────────────── │ 内部私有数组 │ 没有任何外部 │ (仍可被改) │ 视图跟着变 │ (无外部引用) │ 引用能碰它 └──────────────┘ └──────────────┘

五、真正的不可变：Java 9 的 List.of / Set.of / Map.of

Java 9 引入的静态工厂方法 List.of()、Set.of()、Map.of() 才是"从出生就不可变"的集合：它们自带数据、没有任何外部引用能改到内部，因此没有"穿透"问题。而且实现上做了极致优化——元素直接以字段形式内联存储（List0/List12/ListN 等紧凑类），比 ArrayList 更省内存。

ImmutableFactory.javaJava 9+ 不可变工厂
List<String> list = List.of("a", "b", "c");
Set<Integer> set  = Set.of(1, 2, 3);
Map<String, Integer> map = Map.of("x", 1, "y", 2);

// 多于 10 个键值对时用 ofEntries + Map.entry
Map<String, Integer> big = Map.ofEntries(
    Map.entry("a", 1), Map.entry("b", 2));

list.add("d");       // UnsupportedOperationException

// Java 10：从任意集合拷贝出不可变副本（自动防御性拷贝）
List<String> copy = List.copyOf(someMutableList);
三个"想当然"会踩的坑：
① 不接受 null：List.of(null)、Map.of("k", null) 直接抛 NullPointerException（而 ArrayList、HashMap 允许 null）。
② 元素/键不能重复：Set.of(1, 1)、Map.of("k",1,"k",2) 抛 IllegalArgumentException（不是静默去重）。
③ Set.of/Map.of 的遍历顺序不保证，甚至每次 JVM 启动都不同——底层用一个随机 SALT 打散顺序，就是逼你别依赖遍历顺序。要顺序请另选 LinkedHashXxx。

六、Arrays.asList 的三个坑

Arrays.asList(...) 常被误当成"快速建列表"的银弹，但它返回的是 Arrays 内部的一个私有类 ArrayList（不是 java.util.ArrayList！），是一个定长、由原数组支撑的视图，坑不少：

✗ Bad · 误用 Arrays.asList
List<String> l = Arrays.asList("a","b");
l.add("c");   // 定长！抛 UnsupportedOp

// 更隐蔽：primitive 数组
int[] arr = {1, 2, 3};
List<Integer> bad
    = Arrays.asList(arr);  // 编译错/
// 实为 List<int[]>，size==1 !
✓ Good · 按意图选对方法
// 要可变列表：包一层真 ArrayList
List<String> m
  = new ArrayList<>(Arrays.asList("a","b"));
m.add("c");   // OK

// 要不可变列表：直接 List.of
List<String> im = List.of("a","b");

// primitive -> 装箱列表用 stream
List<Integer> boxed
  = Arrays.stream(arr).boxed().toList();
第三个坑（视图穿透）：Arrays.asList(arr) 和原数组共享同一块存储——改列表的元素（list.set(0, x)）会改到原数组，反之亦然。它的定位是"把数组适配成 List 接口"，而非"复制成列表"。想要独立副本，仍是 new ArrayList<>(Arrays.asList(arr))。

七、五种"只读/不可变"手段怎么选

市面上"看起来改不了"的集合有好几种，语义差别很大，选错就埋雷。一张表对齐：

手段可否 add/remove可否 set(改元素)会否被源头穿透允许 null
new ArrayList<>()可可——允许
Arrays.asList(...)否（定长）可（写回数组）会（共享数组）允许
unmodifiableList(src)否否会（视图）随 src
List.of(...)否否不会不允许
List.copyOf(src)否否不会（已拷贝）不允许
Stream 侧的对应物：用流收集时，collect(Collectors.toList()) 返回的实现不保证可变性（JDK 未承诺）；Java 16 的 stream.toList() 明确返回不可变列表；要可变请用 Collectors.toCollection(ArrayList::new)；要明确不可变用 Collectors.toUnmodifiableList()。选 API 时先问自己一句：我需要它可变吗？

八、为什么要"不可变"：贯穿全栈的一条主线

不可变（immutable）不是集合独有的花招，而是一条贯穿 Java 设计的核心思想：String、包装类 Integer、LocalDate、记录 record……全是不可变的。它到底换来了什么？

不可变 = 构造后状态永不变 ⟹ 线程安全 + 可自由共享 + hashCode 可缓存
  • 天生线程安全：状态不会变，就不存在"写-写"或"读-写"冲突，多线程共享无需任何锁。这与内存模型直接相关——只要通过 final 字段发布，JMM 保证其他线程一定能看到构造完成后的正确值（安全发布），无需额外同步。
  • 可放心共享，无需防御性拷贝：把不可变集合当参数传出去/返回出去，不怕对方偷改你的内部状态。可变集合则必须在"进"和"出"的边界都拷贝一次，既啰嗦又耗性能。
  • 可安全做 Map 的 key / Set 的元素：可变对象一旦入了 HashMap 又被改动，其 hashCode 就变了，会导致"存进去却再也 get 不到"的诡异 bug；不可变对象没这个风险，还能缓存 hashCode（String 正是这么干的）。

所以对外暴露集合时的黄金准则，是"能不可变就不可变"。下面是最典型的防御性设计反例与正例：

✗ Bad · 直接返回内部集合
class Team {
  private final List<String> members
      = new ArrayList<>();

  List<String> getMembers() {
    return members;  // 引用泄漏！
  }
}
// 外部：team.getMembers().clear();
// 内部状态被外人清空，封装形同虚设
✓ Good · 返回不可变视图
class Team {
  private final List<String> members
      = new ArrayList<>();

  List<String> getMembers() {
    // 只读视图，外人改不了
    return Collections
        .unmodifiableList(members);
  }
}
// 想要彻底隔离用 List.copyOf(members)
联系 Spring / 实战：大量框架把配置、常量表建成不可变集合对外发布——比如枚举映射、白名单、状态机的合法迁移表。Guava 更早提供了 ImmutableList/Map（真拷贝、真不可变），Java 9 的 List.of 系列本质是把这套思想收编进了标准库。见到"配置类里 static final Map"，优先用 Map.of 而不是静态块里 put。
一句话收束：Collections（复数，工具类）提供 sort/binarySearch/shuffle 等算法与 synchronized*/unmodifiable* 等包装；而"改不了"要分清两层——unmodifiableXxx 和 Arrays.asList 只是视图，会被源头穿透、只挡外人不挡原主人；List.of/Set.of/Map.of（Java 9）与 List.copyOf（Java 10）才是真不可变，自带数据、拒绝 null、遍历序不保证。记住黄金准则：对外暴露集合，能不可变就不可变——它换来的是天生的线程安全、免拷贝的安全共享，和作为 key 的稳定性，这条主线一路通到并发章节。
