ArrayList vs LinkedList、HashMap vs TreeMap

同一个接口的两种实现，取舍全在底层数据结构上——选错了，代码能跑但性能差几个数量级。

一、先建立坐标：它们都是"接口的实现"

初学者最大的误区，是把 ArrayList、LinkedList 当成"两种不同的东西"来记。其实它们是同一份契约（接口）的两种实现——就像"交通工具"这个抽象概念，可以用汽车实现，也可以用自行车实现。你面向的应该是接口，选择哪个实现只是一个性能与场景的工程决策。

List<E>（有序、可重复、按下标访问） Map<K,V>（键值对、键唯一） | | +-------+--------+ +-------+--------+ | | | | ArrayList LinkedList HashMap TreeMap 底层: 动态数组 底层: 双向链表 底层: 数组+链表/红黑树 底层: 红黑树 连续内存 节点散落堆上 哈希桶 有序平衡树 随机访问 O(1) 顺序访问，头尾 O(1) 平均 O(1) 无序 O(log n) 有序
为什么"面向接口"这么重要？因为需求会变。今天你以为要频繁头部插入而选了 LinkedList，明天需求改成随机读为主。如果你全程写的是 List<User> list = ...，那换实现只需改一行；如果你到处写死 LinkedList<User> 并调用了它特有的 addFirst()，就得改一大片。这正是 Spring 里方法参数、字段几乎清一色声明为 List/Map 接口的原因。

二、ArrayList vs LinkedList：底层结构决定一切

要理解它们的性能差异，先得看清内存里到底躺着什么。ArrayList 内部是一块连续的数组（Object[] elementData）；LinkedList 则是一串散落在堆各处、靠指针串起来的节点（双向链表）。

ArrayList —— 一整块连续内存，下标即偏移量 ┌────┬────┬────┬────┬────┬────┐ │ e0 │ e1 │ e2 │ e3 │ e4 │ .. │ get(3) = 基址 + 3×8字节，一步到位 O(1) └────┴────┴────┴────┴────┴────┘ LinkedList —— 节点散落堆上，靠 prev/next 指针串联 first last │ │ ┌─┴─┐ ┌───┐ ┌───┐ ┌───┐ ┌─┴─┐ │e0 │⇄ │e1 │⇄ │e2 │⇄ │e3 │⇄ │e4 │ get(3) 必须从头(或尾)一个个跳 O(n) └───┘ └───┘ └───┘ └───┘ └───┘

这张图几乎解释了后面所有的复杂度差异：数组能"算地址"，链表只能"走指针"。数组的第 i 个元素地址 = 首地址 + i × 元素宽度，一次乘加就到；链表要拿第 i 个元素，只能从头（或尾）沿着 next 一个一个数过去。

LinkedList.java（JDK 源码节选）节点结构 + 定位算法
// 每个节点是一个对象：存值 + 两个指针，内存开销远大于数组
private static class Node<E> {
    E item;          // 元素本身
    Node<E> next;    // 指向后一个节点
    Node<E> prev;    // 指向前一个节点
}

// get(index) 背后：这已经是"优化过"的实现了——折半选方向
Node<E> node(int index) {
    if (index < (size >> 1)) {      // 在前半段 → 从 first 往后走
        Node<E> x = first;
        for (int i = 0; i < index; i++) x = x.next;
        return x;
    } else {                        // 在后半段 → 从 last 往前走
        Node<E> x = last;
        for (int i = size - 1; i > index; i--) x = x.prev;
        return x;
    }
}
// 即便折半，仍是 O(n)——最坏在中间，要走 n/2 步

三、复杂度对决（面试必背）

最反直觉的陷阱 —— "中间插入 LinkedList 更快"是半句真话：教科书说链表中间插入 O(1)，那只在你已经拿着那个节点（比如通过 Iterator）时成立。如果你写 list.add(5000, x)，LinkedList 得先花 O(n) 走到第 5000 个节点，再 O(1) 改指针——总成本仍是 O(n)，而且这个 O(n) 是"指针跳跃 + 缓存不友好"，实测往往比 ArrayList 的数组搬移还慢。

四、为什么实践中 ArrayList 几乎总赢？

Big-O 只数"操作次数"，却忽略了每次操作的真实成本。真正让 ArrayList 在实战里碾压 LinkedList 的，是CPU 缓存局部性（cache locality）。

CPU 读内存不是一个字节一个字节读的，而是一次拉一整条缓存行（cache line，通常 64 字节）。ArrayList 的元素在内存里紧挨着，读 e0 时 e1 e2 e3 已被顺带拉进 L1 缓存，遍历时几乎全是缓存命中；LinkedList 的节点由 new 分散分配在堆各处，每跳一个 next 就极可能是一次缓存未命中（cache miss），要停下来等主存——一次 miss 的代价相当于上百次算术运算。

连 JDK 作者都"劝退"LinkedList：LinkedList 的作者 Josh Bloch 本人曾公开说他几乎从不用它。现代工程共识是：默认就用 ArrayList。只有在明确"以头尾频繁增删为主、且几乎不随机访问"（典型是当队列/双端队列用）时才考虑 LinkedList——但即便这种场景，ArrayDeque 通常还比 LinkedList 更快、更省内存。
✗ 反例：拿 LinkedList 做随机访问
List<Integer> list =
    new LinkedList<>();
// ...塞入 10 万个元素...

// 灾难：每次 get 都 O(n)
// 整个循环退化成 O(n²)
for (int i = 0; i < list.size(); i++) {
    process(list.get(i));
}
// 10 万元素 ≈ 50 亿次指针跳转
✓ 正例：ArrayList + 增强 for
List<Integer> list =
    new ArrayList<>();
// ...塞入 10 万个元素...

// get(i) 是 O(1)，整体 O(n)
// 且连续内存，缓存命中率极高
for (Integer x : list) {   // 走迭代器
    process(x);
}
// 遍历链表也该用增强 for，
// 别用下标 get(i)！

五、ArrayList 的扩容：均摊 O(1) 从哪来

数组长度固定，那 ArrayList 怎么"动态"增长？答案是满了就换一个更大的数组，把旧数据拷过去。关键在增长策略：JDK 采用1.5 倍扩容。

newCap = oldCap + (oldCap >> 1)  ≈  1.5 × oldCap
ArrayList.java（JDK 源码节选）grow() 扩容逻辑
private Object[] grow(int minCapacity) {
    int oldCapacity = elementData.length;
    int newCapacity = oldCapacity + (oldCapacity >> 1);  // 位运算 ×1.5
    // ...边界校验略...
    return elementData = Arrays.copyOf(elementData, newCapacity);  // 拷贝旧数组
}
// 默认初始容量 10，但注意：new ArrayList() 时数组还是空的，
// 直到第一次 add 才懒加载分配为 10 —— 省掉"建了不用"的浪费

为什么是 1.5 倍而不是 2 倍？因为倍数扩容能让扩容成本被"均摊"：n 次 add 触发的总拷贝量是等比数列，加起来仍是 O(n)，平摊到每次就是 O(1)。而 1.5 相比 2 更温和，能减少内存浪费、且更容易复用之前释放的内存块。

性能陷阱 —— 已知大小却不预分配：如果你要往 ArrayList 里塞 100 万条数据却用默认构造，它会经历十几次"扩容 + 全量拷贝"，白白搬运大量数据。已知规模时务必指定初始容量：new ArrayList<>(1_000_000)，一次到位，杜绝反复扩容。这也是 HashMap 同理的高频优化点。

六、HashMap vs TreeMap：无序哈希 vs 有序红黑树

换到 Map 这边，对立的核心是"用哈希换 O(1) 但放弃顺序" 还是 "用平衡树保 O(log n) 但换来有序"。

  • HashMap：底层是数组 + 链表 / 红黑树。用 key 的 hashCode() 算出落在哪个桶（bucket），理想情况下一步定位，平均 O(1)。代价是完全无序——遍历顺序和插入顺序、大小顺序都无关。
  • TreeMap：底层是红黑树（自平衡二叉搜索树）。所有 key 按排序规则组织，任何操作都是 O(log n)。代价是慢一点，但换来始终有序，还能做范围查询。
维度HashMapTreeMap
底层结构数组+链表/红黑树（哈希表）红黑树
get/put/remove平均 O(1)，最坏 O(log n)稳定 O(log n)
顺序无序（不保证任何顺序）按 key 排序
对 key 的要求正确实现 hashCode + equals可 Comparable 或传 Comparator
null key允许 1 个 null key不允许（自然排序时抛 NPE）
额外能力——范围查询 NavigableMap
典型场景绝大多数"按键查值"需要有序/范围/最近邻
还有个"中间派"：LinkedHashMap。如果你既想要 HashMap 的 O(1)，又想保留"插入顺序"（或"访问顺序"），用 LinkedHashMap——它在哈希表基础上用一条双向链表串起所有 entry。注意区别：LinkedHashMap 保的是插入顺序，TreeMap 保的是大小排序，两者不是一回事。用访问顺序模式（accessOrder=true）还能几行代码实现一个 LRU 缓存。

七、TreeMap 的杀手锏：范围查询

如果只是"按 key 查 value"，TreeMap 没理由比 HashMap 快。它真正无可替代的是有序性带来的范围/最近邻查询（NavigableMap 接口），这些是 HashMap 根本做不到的。

RangeQuery.javaTreeMap 的有序能力
TreeMap<Integer, String> scores = new TreeMap<>();
scores.put(60, "及格线");
scores.put(85, "良好");
scores.put(95, "优秀");

scores.firstKey();           // 60  —— 最小的 key
scores.lastKey();            // 95  —— 最大的 key
scores.floorKey(88);         // 85  —— ≤88 中最大的（"向下取"）
scores.ceilingKey(88);       // 95  —— ≥88 中最小的（"向上取"）

// 范围子视图：[60, 90) 之间的所有 entry，且仍然有序
scores.subMap(60, 90);        // {60=及格线, 85=良好}

// 应用：阶梯计费/评级、时间区间查询、IP 段归属……
// 凡是"落在哪个区间"的问题，TreeMap 一行 floorEntry 搞定
建立联系 —— 数据库索引也是这套思路：MySQL InnoDB 的主键索引用的是 B+ 树，和 TreeMap 的红黑树一样是"有序平衡树家族"。这就是为什么数据库 范围查询（WHERE age BETWEEN 20 AND 30）走索引很快，而哈希索引（类似 HashMap）只能做等值查询、做不了范围。有序结构换来的是范围能力，这个取舍在内存和磁盘上是相通的。

八、两个 Map 各自的经典陷阱

HashMap 陷阱 —— 破坏 hashCode / equals 契约：把自定义对象当 key 时，必须同时正确重写 hashCode() 和 equals()。若只重写 equals 不重写 hashCode，两个"相等"的对象可能算出不同哈希、落到不同桶，导致你 put 进去后 get 不出来。契约铁律：equals 相等 → hashCode 必须相等。另外别用可变对象做 key——put 之后再改动它的字段会让哈希值变化，从此永远找不回。
TreeMap 陷阱 —— 排序与 equals 不一致 / null key：TreeMap 判断"两个 key 是否相同"靠的是 compareTo/Comparator 返回 0，而不是 equals。如果你的比较器只比较对象的某个字段（比如只按年龄），那么两个"年龄相同但其实不同"的 key 会被当成同一个，后者覆盖前者。还有：TreeMap 用自然排序时 put(null, v) 会直接抛 NullPointerException——因为它要拿 null 去调 compareTo。这点和"HashMap 允许一个 null key"正好相反。
✗ 只重写 equals，忘了 hashCode
class Point {
  int x, y;
  // 只重写 equals
  @Override public boolean
  equals(Object o) { /* 比 x,y */ }
  // hashCode 用了 Object 默认实现
}

map.put(new Point(1,2), "A");
map.get(new Point(1,2));
// → null! 两次算出的 hash 不同，
//   落到不同桶，根本找不到
✓ 两者成对重写
class Point {
  int x, y;
  @Override public boolean
  equals(Object o) { /* 比 x,y */ }
  @Override public int hashCode() {
    return Objects.hash(x, y);
  }
  // equals 相等 ⇒ hashCode 相等
}

map.put(new Point(1,2), "A");
map.get(new Point(1,2)); // → "A"
// 或直接用 record，自动生成两者

九、如何选型：一张决策表

需求选它理由
大多数 List 场景（读多、按下标）ArrayList随机访问 O(1)、缓存友好、省内存
频繁头尾增删、当队列/栈用ArrayDeque比 LinkedList 更快更省，优先考虑
大多数 Map 场景（按键查值）HashMap平均 O(1)，无脑默认

一句话收束：四个类的差异，本质是底层数据结构在"随机访问、增删位置、有序性、内存"四个维度上的取舍。ArrayList（连续数组）赢在随机访问与缓存局部性，是 List 的默认答案；LinkedList（双向链表）只在头尾增删且不随机访问时才有意义，多数时候还打不过 ArrayDeque。HashMap（哈希表）用无序换来平均 O(1)，是 Map 的默认答案；TreeMap（红黑树）用 O(log n) 换来有序与范围查询——当且仅当你真的需要顺序时才用它。记住原理，而不是背结论：给定操作，先想"它在这个结构上要走几步"，答案自然浮现。
