字符串与 StringBuilder

一个被设计成"永远无法修改"的对象，为什么反而是 Java 里最灵活、最安全的类型？

一、String 为什么是"不可变"的

不可变（immutable）意思是：一个 String 对象一旦创建，它内部的字符序列就永远不会改变。你以为在"修改"字符串的每一个方法（toUpperCase、substring、replace……）其实都在返回一个全新的对象，原对象纹丝不动。

Immutable.java初学者最常踩的第一个坑
String s = "hello";
s.toUpperCase();              // 返回了新对象, 但没人接收, 白算了
System.out.println(s);       // 仍然是 hello —— s 从未被改变

s = s.toUpperCase();          // 必须重新赋值, 让 s 指向新对象
System.out.println(s);       // HELLO

这不是 API 设计上的偶然，而是从底层数据结构就锁死的。看一眼 JDK 里 String 的骨架：

java.lang.String（简化）JDK 源码
public final class String {          // final: 类不能被继承, 防止子类破坏不可变契约
    private final byte[] value;      // final: 引用不能改指向 (Java 9+ 用 byte[] 存, 紧凑字符串)
    private int hash;                // 缓存 hashCode, 只算一次
}
三重保险锁死可变性：① class 是 final，没人能写一个"可变的子类 String"冒充它；② value 数组引用是 final，构造后不能再指向别的数组；③ 这个数组是 private 的，且所有方法都不把它暴露出去。三者缺一，不可变都会被击穿。

为什么值得设计成不可变？它解决了什么问题

好处背后的原理 / 场景
可安全共享不可变对象天然线程安全，多个线程读同一个 String 不需要加锁，也就有了下文的"常量池"复用。
可缓存 hashCode内容永不变 → hash 只需算一次存进 hash 字段。这正是 String 成为 HashMap 头号 key 类型的原因。
安全性类名、文件路径、网络地址、数据库连接串常以 String 传递。若可变，就可能出现"检查时是安全路径、使用时被改成 /etc/passwd"的 TOCTOU 攻击。
陷阱：不可变 ≠ 引用不可变。String s = "a"; s = "b"; 完全合法——变了的是变量 s 指向谁，不是那个 "a" 对象本身。final String s 才锁住引用，但即便不加 final，那个字符串对象的内容也永远不会变。分清"对象不可变"和"变量不可重新赋值"是两回事。

二、字符串常量池：== 陷阱的根源

因为 String 不可变、可安全共享，JVM 就干脆维护一个字符串常量池（String Pool）：所有源码里写死的字符串字面量 "java" 都指向池中同一个对象，能省内存就省。而 new String("java") 明确要求"给我造个新的"，于是在堆上另开一块。

栈(Stack) 堆(Heap) a ───────────┐ b ───────────┼───▶ ┌───────────────────────────┐ └───▶ │ "java" ← 字符串常量池 │ (a、b 共享同一个) └───────────────────────────┘ c ─────────────────▶ ┌───────────────────────────┐ │ new String("java") │ (内容相同, 地址不同) └───────────────────────────┘
PoolTrap.java面试高频
String a = "java";
String b = "java";
String c = new String("java");

System.out.println(a == b);          // true  —— 指向常量池同一对象
System.out.println(a == c);          // false —— c 是堆上的新对象, 地址不同
System.out.println(a.equals(c));     // true  —— equals 比较内容
System.out.println(a == c.intern()); // true  —— intern() 把 c 换回池中规范对象
永远别用 == 比字符串内容！== 比的是引用地址，只有恰好指向同一对象时才为 true。看似能过的 == 往往是被常量池"骗"了——一旦其中一个来自 new、拼接、或 IO 读入，就翻车。比内容一律用 equals；防 null 时把常量写前面："OK".equals(status)。
编译期折叠（constant folding）：如果拼接的两端都是编译期常量，javac 会在编译时就算好并放进常量池。所以 "ja" + "va" == "java" 结果是 true；但 "ja" + new String("va") == "java" 是 false——只要掺入一个运行期才确定的值，就不再折叠，而是走下一节的运行时拼接。
联系底层：常量池在 Java 7 之前放在方法区（PermGen），Java 7 起挪进了堆，这样它才能被 GC 正常回收——否则大量 intern() 的字符串会撑爆 PermGen。intern() 好用但别滥用：手动把海量运行期字符串塞进池反而增加 GC 压力和查表成本。

三、拼接性能：+ 号在循环里的灾难

正因为 String 不可变，"拼接"本质上是"造一个更大的新字符串再把旧内容拷进去"。单次拼接没问题，但放进循环就会退化成灾难级复杂度。

先看 javac 对 a + b 做了什么：Java 8 及之前，它会脱糖成 new StringBuilder().append(a).append(b).toString()；Java 9+ 改用 invokedynamic 调用 StringConcatFactory 动态生成拼接逻辑。关键在于：这个优化只作用于"一条语句内"。循环里每一次 result += x 都是独立的一条语句，编译器无法把一个 StringBuilder 跨迭代复用——于是每轮都新建一个 StringBuilder、把已有结果整个拷进去。

✗ 反例 · O(n²)
String result = "";
for (int i = 0; i < n; i++) {
    result += arr[i];
    // 每轮都 new 一个 StringBuilder,
    // 并把已累积的内容整段拷贝一遍
}
// 第 i 轮拷 i 个字符 → 1+2+…+n
✓ 正例 · O(n)
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) {
    sb.append(arr[i]);
    // 复用同一块内部缓冲区,
    // 满了才扩容(容量翻倍, 均摊 O(1))
}
String result = sb.toString();
String += : 1+2+…+n ≈ n²/2  →  O(n²)

StringBuilder : n 次均摊拷贝  →  O(n)

拼接 1 万个字符串时，两者差距不是"快一点"，而是几百上千倍量级的鸿沟：

String +=  ≈ 5000万次字符拷贝 SB
StringBuilder 内部机制：默认初始容量 16 个字符，底层就是一个可变的 byte[]。append 时若容量不够，扩容公式约为 newCapacity = (oldCapacity << 1) + 2（翻倍再加 2），并把旧内容拷到新数组。翻倍策略保证 n 次 append 的总拷贝量是 O(n)（均摊 O(1)）。若能预估长度，用 new StringBuilder(expectedSize) 一次性给够容量，可完全避免扩容拷贝。
别矫枉过正：不是所有 + 都要换成 StringBuilder！单条语句里的 "user=" + id + ", age=" + age 编译器已优化成一次拼接，手写 StringBuilder 反而更啰嗦、可读性更差。只有在循环 / 递归里反复累加时才必须用 StringBuilder。这也是初学者矫枉过正的高频面试反问点。

四、String / StringBuilder / StringBuffer 怎么选

维度StringStringBuilderStringBuffer
可变性不可变可变可变
线程安全安全（因不可变）不安全安全（方法 synchronized）
性能拼接慢最快较慢（有锁开销）
登场时间始祖Java 5Java 1.0（元老）
适用场景常量、key、只读文本单线程内大量拼接多线程共享的可变缓冲（罕见）
StringBuffer 基本已被时代淘汰。它 1.0 就有，Java 5 才补上不加锁的 StringBuilder。真实开发中一个 StringBuilder 极少被多线程共享——通常是方法内局部变量。与其为"万一"付出每次 append 都加锁的代价，不如让它做局部变量天然隔离。所以默认用 StringBuilder，只有确实要跨线程共享同一个可变缓冲时才考虑 StringBuffer。这和 ArrayList（不同步）取代 Vector（同步）是同一套设计哲学。
联系 Spring / 工具库：你很少直接写常量池细节，但天天在享受它。Spring 的 StringUtils、日志框架的消息格式化、JSON 序列化内部大量使用 StringBuilder 累积输出；而配置 key、Bean 名称等被频繁比较的字符串，正是靠不可变 + 缓存 hashCode 才能高效地做 Map 查找。
一句话收束：String 的不可变换来了常量池共享、线程安全、hashCode 缓存三大红利，代价是"拼接即新建"；因此比内容永远用 equals（== 只是被常量池骗）、循环拼接永远用 StringBuilder（把 O(n²) 拉回 O(n)），而单线程默认选 StringBuilder、几乎不再用 StringBuffer。
