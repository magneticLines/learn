死锁、活锁、饥饿

加锁能保证“坏事不发生”，却可能让“好事永远不发生”——这三种活跃度故障，就是锁的账单。

一、先分清：安全性 vs 活跃度

并发程序的正确性有两条相互拉扯的主线。安全性（Safety）关心“坏事永远不会发生”——数据不被写坏、不变式不被破坏；活跃度（Liveness）关心“好事最终一定会发生”——线程能推进、请求能被响应、程序不会永远卡住。

为什么把它们放在一起讲？因为二者是此消彼长的：为了安全我们加锁串行化，而锁一旦用得过火，就会引出这一章的三个活跃度故障——死锁（deadlock）、活锁（livelock）、饥饿（starvation）。它们的共同点是“程序没崩、没抛异常，但就是不干活了”，因此格外难排查，也是面试里区分“会用锁”和“懂并发”的分水岭。

一句话对照： 死锁——大家都在等对方手里的锁，谁也不动（线程 BLOCKED，CPU 空转为 0）；活锁——大家都在动，却都在做无用功、始终不前进（线程 RUNNABLE，CPU 反而很高）；饥饿——某个线程总是抢不到 CPU 或锁，被“饿着”（其他线程正常，唯独它进展不了）。

二、死锁：互相攥着对方要的钥匙

死锁是一组线程互相持有对方所需的资源并循环等待，导致全体永久阻塞。最经典的类比是哲学家就餐问题：5 个哲学家围坐一桌，每人左右各一根筷子，都遵循“先拿左筷、再拿右筷”的规则。若同一瞬间所有人都拿起了左筷，就人人都在等右筷——而右筷正是邻座的左筷，谁都不肯放，于是永远吃不上饭。

映射到代码里，最简单的形式是顺序不一致的锁获取：线程 A 按 left → right 加锁，线程 B 按 right → left 加锁，只要两者交错运行就可能死锁。

LeftRightDeadlock.java静态锁顺序死锁
public class LeftRightDeadlock {
    private final Object left  = new Object();
    private final Object right = new Object();

    public void leftRight() {          // 线程 A 调用
        synchronized (left) {
            synchronized (right) {   // 先 left 再 right
                doSomething();
            }
        }
    }
    public void rightLeft() {          // 线程 B 调用
        synchronized (right) {
            synchronized (left) {    // 先 right 再 left —— 顺序相反！
                doSomethingElse();
            }
        }
    }
}

用资源分配图看，死锁的本质就是图里出现了一个环：

线程 A 持有 left，等待 right 线程 B 持有 right，等待 left ┌──────────┐ 持有 ┌───────┐ │ 线程 A │ ───────▶ │ left │ └──────────┘ └───────┘ ▲ │ 等待 │ │ 想要（被 A 持有） │ ▼ ┌───────┐ ┌──────────┐ │ right │ ◀─────── │ 线程 B │ └───────┘ 持有 └──────────┘ A → right → B → left → A ← 形成闭环 = 死锁
如何确诊： 生产环境里对进程执行 jstack <pid>（或在 IDE 里 dump 线程），JVM 会自动做死锁检测并打印 Found one Java-level deadlock:，直接列出互相等待的线程与锁对象。这是排查“系统假死”最快的一招——比盲猜代码高效得多。检测与预防的完整手段（锁顺序、定时锁、tryLock）留到下一篇专门展开。

三、动态锁顺序死锁：更隐蔽的转账 bug

上面的例子锁顺序是写死在代码里的，肉眼可查。真正阴险的是锁顺序由参数决定的情况——代码里明明只有一处加锁顺序，却依然会死锁。银行转账是《Java 并发编程实战》里的经典案例：

Bad · 锁顺序随参数漂移
void transfer(Account from,
              Account to, int amt) {
    synchronized (from) {
        synchronized (to) {
            from.debit(amt);
            to.credit(amt);
        }
    }
}
// 线程1: transfer(A, B, 10)
// 线程2: transfer(B, A, 20)
// → 一个先锁A等B，一个先锁B等A
// → 同样的方法，照样死锁！
Good · 用哈希诱导全局顺序
int fh = System.identityHashCode(from);
int th = System.identityHashCode(to);
if (fh < th)
    lock(from, to, amt);   // 恒定顺序
else if (fh > th)
    lock(to, from, amt);
else synchronized (tieLock) { // 哈希撞车
    lock(from, to, amt);   // 加时锁兜底
}
// 所有线程都按 hash 大小加锁
// → 不可能形成循环等待
高频面试陷阱： 很多人以为“我全程只按对象出现的先后顺序加锁，不会死锁”。错！只要加锁顺序依赖于外部传入的对象，两次调用传参顺序相反就会构成相反的锁序。解法是给对象定义一个与调用无关的全局序——用 System.identityHashCode() 或一个唯一 ID 排序后再加锁；哈希极小概率相同时，用一把额外的“加时锁（tie-breaking lock）”保证唯一顺序。这正是把“循环等待”这个必要条件从根上打破。

四、死锁的四个必要条件（Coffman 条件）

操作系统课本告诉我们：死锁的发生必须同时满足以下四个条件，缺一不可。这也意味着——只要破坏其中任意一个，死锁就无法形成。这是所有防死锁策略的理论地基。

死锁 = 互斥 ∧ 持有并等待 ∧ 不可抢占 ∧ 循环等待
条件含义如何打破（Java 手段）
互斥
Mutual Exclusion		资源同一时刻只能被一个线程占用难破坏（锁的本质就是互斥）；可改用无锁 CAS / 不可变对象绕开
持有并等待
Hold & Wait		已持有资源的线程又去请求新资源，且不释放已有的一次性申请所有锁；或拿不到就tryLock失败后释放全部重试
不可抢占
No Preemption		资源只能由持有者主动释放，不能被强夺用lock.tryLock(timeout)定时锁，超时主动放弃（相当于自我抢占）
循环等待
Circular Wait		存在“线程等资源”的闭环链条最常用：给所有锁定义全局顺序，一律按序申请
为什么工程上首选“破坏循环等待”？ 因为“互斥”是锁的立身之本，动不得；“持有并等待”和“不可抢占”的破坏方案（一次抢全部 / 定时放弃）要么降低并发度、要么带来重试开销。而统一锁顺序只是一个编码约定，几乎零运行时代价，却能彻底根除锁顺序死锁——这就是上一节转账例子的思路来源。

五、活锁：一直在动，就是没进展

活锁和死锁的“症状”相反：线程没有阻塞，全都在 RUNNABLE 状态忙个不停、CPU 甚至打满，但因为不断重复同一个会失败的动作，谁都没能真正前进。生活里的类比是：两个人在走廊迎面相遇，你往左他也往左，你往右他也往右，反复“礼让”却始终撞在一起。

工程中最典型的是消息处理的“毒丸”回滚：一条无法处理的消息在事务失败后被放回队列头部重试，下次取出还是它、还是失败……处理线程永远在“取出→失败→回滚→再取出”里空转。另一类是无退避的冲突重试：两个线程检测到冲突就同时回退再同时重试，步调完全一致，于是永远撞车。

Bad · 固定步调重试 → 活锁
while (!tryCommit()) {
    rollback();
    // 两个线程都立刻重试
    // 冲突 → 回退 → 再冲突……
    // CPU 100%，事务永不成功
}
Good · 随机化退避打破同步
int attempt = 0;
while (!tryCommit()) {
    rollback();
    // 随机等待，错开彼此步调
    long back = random.nextInt(
        1 << Math.min(attempt++, 6));
    Thread.sleep(back);
}
解法直觉： 活锁的根子在于“所有参与者做了完全相同的决定”。只要引入一点随机性（随机退避时间、随机重试间隔），就能打破对称、错开步调——以太网的 CSMA/CD 二进制指数退避、分布式系统里的 jittered backoff 用的都是同一招。对“毒丸消息”，则应设置最大重试次数，超限就投递到死信队列（DLQ）而非无限回滚。

六、饥饿：轮不到我

饥饿是指某个线程始终得不到它需要的资源（CPU 时间片或锁），因而无法推进，尽管系统整体在正常运转。它不像死锁那样全盘停摆，而是“个别倒霉蛋被长期忽视”。常见成因有三类：

  • 线程优先级误用：把某些线程设成高优先级，低优先级线程可能长期抢不到 CPU。
  • 非公平锁的“插队”：非公平锁允许活跃线程连续抢占，排队线程可能一直轮不上。
  • 持锁时间过长 / 死循环占用：某线程拿着锁做耗时操作（甚至在锁内 sleep、做 I/O），其余线程集体挨饿。
别依赖 Thread.setPriority()！ Java 的线程优先级（1~10）只是给操作系统调度器的一个建议，最终如何映射由 OS 决定：Linux 上默认几乎被无视，Windows 上只有寥寥几档生效。所以用优先级来“保证”某线程多跑或防饥饿是不可靠的，反而容易在某些平台上亲手制造饥饿。想控制执行秩序，应当用显式的并发工具（公平锁、有界队列、Semaphore），而不是拨弄优先级。
联系上一章： 上一节我们讲 ReentrantLock / Semaphore 的公平模式时说过：非公平锁吞吐更高但有饥饿风险，公平锁按 FIFO 发放、杜绝饥饿但吞吐略低。这正是“防饥饿”与“性能”的经典取舍——绝大多数场景选非公平（饥饿概率极低且短暂），只有对响应时间可预测性有硬性要求时才付出公平锁的代价。

一句话记忆： 死锁是“都不动”，活锁是“白忙活”，饥饿是“没我份”。面试被追问“活锁和死锁的区别”时，答题要点就是线程状态与 CPU 占用——死锁线程阻塞、CPU 空闲；活锁线程运行、CPU 繁忙却零进展。这一句往往就能把回答者和背概念的人区分开。
核心收束： 三者都是活跃度故障——程序没崩，但“好事不再发生”。死锁由四个必要条件（互斥 · 持有并等待 · 不可抢占 · 循环等待）共同促成，破坏任意一个即可预防，工程上首选统一全局锁顺序打破循环等待；活锁源于所有参与者步调一致地重复失败动作，靠随机退避 + 重试上限破除对称；饥饿是资源分配长期不公，用公平锁、限时持锁解决，切忌依赖不可靠的线程优先级。记住：加锁是为安全付的费，而这三种故障就是没算清楚的活跃度账单。
