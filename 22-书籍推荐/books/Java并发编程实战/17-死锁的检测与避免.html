死锁的检测与避免

四个必要条件、锁顺序、定时锁与线程转储——把偶发的“程序卡死”变成可预防、可诊断的工程问题。

一、死锁到底是怎么形成的

死锁不是玄学。它有精确的定义：一组线程互相持有对方需要的资源，形成环路等待，谁都不肯先松手，于是集体永久阻塞。操作系统课上给出了著名的四个必要条件，四者同时成立才可能死锁——这也意味着，只要打破其中任意一个，死锁就不可能发生。这是后面所有“避免”手段的理论根基。

必要条件含义Java 里的体现
互斥资源同一时刻只能被一个线程占有synchronized / Lock 本质就是互斥，很难打破
持有并等待持有一把锁的同时去申请另一把嵌套加锁：synchronized(A){ synchronized(B){} }
不可剥夺锁不能被强行抢走，只能主动释放synchronized 天生不可中断，只能等它自己退出
循环等待存在线程等待环 T1→T2→…→T1两个线程以相反顺序获取同两把锁
类比：两个人过独木桥，A 从左端上、B 从右端上，走到中间面对面。桥太窄（互斥），谁都站在桥上不退（持有并等待 + 不可剥夺），A 等 B 退、B 等 A 退（循环等待）——于是永远僵在那里。工程上的解法往往不是“加宽桥”，而是规定“永远先让编号小的人上桥”，从根上消灭循环等待。

二、最经典的死锁：锁顺序不一致

《Java 并发编程实战》第 10 章反复强调：绝大多数死锁都源于“两个线程以不同的顺序获取相同的锁”。经典案例是转账——从账户 A 转钱到账户 B，需要同时锁住两个账户。

DeadlockTransfer.java反例
public void transfer(Account from, Account to, int amount) {
    synchronized (from) {              // 先锁转出方
        synchronized (to) {            // 再锁转入方
            from.debit(amount);
            to.credit(amount);
        }
    }
}

看起来无懈可击，但只要有两个线程反向操作，灾难就发生：

线程 T1: transfer(账户X, 账户Y) 线程 T2: transfer(账户Y, 账户X) ────────────────────────── ────────────────────────── 1. lock(X) ✔ 拿到 1. lock(Y) ✔ 拿到 2. lock(Y) ✘ 等 T2 释放 2. lock(X) ✘ 等 T1 释放 T1 持有 X → 等 Y T2 持有 Y → 等 X └───────── 环路形成 = 死锁 ─────────┘
陷阱：这类死锁靠测试极难复现。 它依赖两个线程恰好在“各拿一把、去抢另一把”的窗口相遇，概率可能是万分之一。开发机上跑一万次不出问题，上了生产、流量一大就偶发卡死。不要用“测过没事”来判断没有死锁——要靠代码审查确认加锁顺序全局一致。

三、避免手段一：全局锁顺序

既然死锁来自“顺序不一致”，那就强制所有线程按同一顺序申请锁，循环等待条件被打破。问题在于：from、to 是运行时才确定的，怎么定义“谁先谁后”？答案是给每把锁一个全序——用对象的 System.identityHashCode() 或业务主键排序，永远先锁 hash 小的那个。

Bad · 顺序取决于参数
synchronized (from) {
  synchronized (to) {
    ...
  }
}
// T1: from=X,to=Y → 锁 X,Y
// T2: from=Y,to=X → 锁 Y,X
// 顺序相反 → 死锁
Good · 按 hash 定序
int h1 = identityHashCode(from);
int h2 = identityHashCode(to);
Object first  = h1 < h2 ? from : to;
Object second = h1 < h2 ? to : from;
synchronized (first) {
  synchronized (second) {
    ...   // 全局同序，无环
OrderedTransfer.java正例 · 完整版
private static final Object tieLock = new Object();  // 处理 hash 冲突的兜底锁

public void transfer(Account from, Account to, int amount) {
    int hFrom = System.identityHashCode(from);
    int hTo   = System.identityHashCode(to);

    if (hFrom < hTo) {
        synchronized (from) { synchronized (to)   { doTransfer(from, to, amount); } }
    } else if (hFrom > hTo) {
        synchronized (to)   { synchronized (from) { doTransfer(from, to, amount); } }
    } else {                                   // 极少数：两对象 hash 相同
        synchronized (tieLock) {                  // 先抢全局兜底锁，再定序
            synchronized (from) { synchronized (to) { doTransfer(from, to, amount); } }
        }
    }
}
联系 JDK / 数据库： 这个“按固定顺序加锁”的思想在数据库里同样是金科玉律——多条事务若都按主键升序更新行，就不会出现 MySQL 那种 Deadlock found; try restarting transaction。此外 Java 里 identityHashCode 一旦生成便缓存在对象头（Mark Word）中，同一对象多次调用返回值稳定，所以能作为可靠的全序依据。

四、避免手段二：定时锁与可中断锁

全局锁顺序在“锁数量有限、能事先排序”时最优。但现实中锁可能来自不同模块、无法统一定序。这时换个思路——打破“不可剥夺”条件：不再无限期死等，而是用 ReentrantLock.tryLock(timeout) 定时申请，超时就主动放弃已持有的锁、退避后重试。这正是 synchronized 永远做不到、而显式 Lock 存在的核心理由之一。

能力synchronizedReentrantLock
限时获取（避免死等）✘ 不支持✔ tryLock(t, unit)
可中断等待✘ 阻塞不可中断✔ lockInterruptibly()
非阻塞尝试✘✔ tryLock() 立即返回
释放责任自动（出块即释放）手动，必须 finally { unlock() }
TimedTransfer.java定时锁 · 退避重试
public boolean transfer(Account from, Account to, int amount) throws InterruptedException {
    long stop = System.nanoTime() + TimeUnit.SECONDS.toNanos(3);
    while (System.nanoTime() < stop) {
        if (from.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
            try {
                if (to.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                    try { from.debit(amount); to.credit(amount); return true; }
                    finally { to.lock.unlock(); }
                }
            } finally { from.lock.unlock(); }   // 拿不到第二把 → 释放第一把，破环
        }
        Thread.sleep(random(10, 50));          // 随机退避，避免活锁式对拍
    }
    return false;                                // 超时放弃，交给上层重试/告警
}
陷阱：定时锁不是免费午餐。 ① 拿不到锁时必须释放已持有的全部锁，否则退避重试只是把死锁变成慢速死锁；② 退避一定要随机化，否则两个线程同步重试、同步失败，会退化成活锁；③ ReentrantLock 忘写 finally{unlock()} 会导致锁永不释放，比死锁更隐蔽。
可发生死锁 ⇔ 互斥 ∧ 持有并等待 ∧ 不可剥夺 ∧ 循环等待

锁顺序打破「循环等待」；定时锁打破「不可剥夺」——破一个即可。

五、检测：线程转储读环路

预防再周全，线上仍可能出现死锁。此时 JVM 是你的救命工具：jstack <pid> 或 jcmd <pid> Thread.print 打印线程转储，JVM 会主动分析锁的 owner/waiter 关系，直接标出 “Found one Java-level deadlock” 并把环路列给你。这背后就是操作系统里“资源分配图找环”的算法。

Found one Java-level deadlock: ============================= "T1": waiting to lock monitor 0x... (object 0x...Account Y), which is held by "T2" "T2": waiting to lock monitor 0x... (object 0x...Account X), which is held by "T1" ← owner/waiter 成环，铁证 Java stack information for the threads listed above: "T1": at Bank.transfer(Bank.java:14) - waiting to lock <Y> - locked <X>
工具全景： jstack（命令行抓转储）→ jconsole / VisualVM 有 “检测死锁” 按钮 → 生产环境常配合 APM（Arthas 的 thread -b 一键找出阻塞其他线程的“罪魁”）。注意：JVM 的自动检测只覆盖内置锁 synchronized 与 java.util.concurrent 的 Lock；纯靠 Semaphore、CountDownLatch 等资源造成的等待环，jstack 可能不会标注为 deadlock，需要你自己读栈判断。

预防 vs 检测：定位不同

维度预防（锁顺序 / 定时锁）检测（jstack / 图找环）
时机编码期，从设计上杜绝故障后，事后诊断
成本需要全局约定、代码纪律零日常成本，出事再用
能否自动恢复定时锁可自愈只能看，不能自动解开
适用业务代码首选兜底 + 排查第三方库

六、其他实战守则

  • 缩小锁范围，用开放调用（open call）： 持有锁时不要调用外部/未知方法（回调、监听器、Spring Bean 的其它方法），因为你不知道它内部又会去抢什么锁。把外部调用移到 synchronized 块之外。
  • 能不共享就不共享： 用不可变对象、ThreadLocal、无锁的 ConcurrentHashMap，从源头减少需要同时持有的锁数量。
  • 警惕隐式锁顺序： Spring 的单例 Bean 若在方法里 synchronized(this) 又互相调用，容易埋下你看不见的锁环——审查时把“谁在锁里调了谁”画成有向图。
一句话收束： 死锁 = 四条件同时成立，破一即解。工程首选全局锁顺序（破循环等待）；无法定序时用 tryLock 定时锁 + 随机退避（破不可剥夺，还能自愈）；上线后靠 jstack 线程转储读 owner/waiter 环路兜底诊断。预防在设计，检测在事后——两手都要有。
