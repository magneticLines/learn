并发陷阱：指令重排与内存屏障

你写下的代码顺序，从来不是它真正执行的顺序——搞懂谁在偷偷重排、为什么允许重排、以及内存屏障如何按住它，并发的"灵异事件"才会现出原形。

一、为什么会有指令重排

先建立一个反直觉的认知：你写的 Java 源码顺序，只是给单线程读者看的"意图"，不是 CPU 真正执行的顺序。从源码到真正落地执行，至少有三层都在偷偷调整顺序，目的只有一个——榨干硬件性能。

层级谁在重排为什么这么做
编译器javac / JIT（C2）把不依赖的指令挪开，隐藏内存访问延迟、复用寄存器、做循环优化
CPU乱序执行 + 指令流水线一条指令等内存时，让后面无依赖的指令先跑，别让执行单元闲着
内存系统Store Buffer / 失效队列写操作先塞进写缓冲区异步刷回，读操作从缓存拿——导致"写后读"看起来被换位
为什么允许它们乱来？因为访问主内存要上百个时钟周期，而一次寄存器运算只要一两个。如果 CPU 老老实实"取数→算→存→取下一个数"，绝大部分时间都在干等内存。重排的本质，是让"能立刻算的先算，等内存的往后靠",把等待的空窗填满。这和你做饭时"烧水的间隙先切菜"是同一个直觉——顺序无关的事，凭什么要排队？

二、护身符：as-if-serial 与数据依赖

既然大家都在重排，为什么单线程程序从来没出过问题？因为所有重排都必须遵守一条铁律——as-if-serial（貌似串行）语义：不管怎么重排，单线程的执行结果必须和"按代码顺序跑"一模一样。要守住这条，重排就不能打破数据依赖。

Reorder.java数据依赖决定谁能被重排
int a = 1;        // ①
int b = 2;        // ②  ① 与 ② 无依赖 → 可以互换顺序
int c = a + b;    // ③  依赖 ①②的结果 → 必须排在它们之后，不可提前

①②之间没有依赖，编译器 / CPU 尽可随意换序；但③读了 a 和 b，存在真实数据依赖，永远不会被提到①②之前。单线程里，as-if-serial 帮你把"重排"这件事完全屏蔽了，你根本感知不到。

陷阱一：as-if-serial 只对"单线程"负责，它压根不看别的线程。①②可以换序，在单线程里毫无区别；可一旦另一个线程正盯着 a 和 b，它就可能先看到 b == 2 却还看到 a == 0——因为对它而言②被"提前"了。重排在跨线程时不再透明，这正是几乎所有可见性 / 有序性 bug 的总根源。

三、亲眼看见重排：经典的 0,0 现象

这是 JMM 教科书级的例子（Doug Lea 用它做过大量实测）。两个线程、四个赋值，直觉告诉你结果不可能是 (x=0, y=0)——但它真的会发生。

初始：a = 0, b = 0 线程 1 线程 2 ───────── ───────── a = 1; ① b = 1; ③ x = b; ② y = a; ④ 你以为的所有可能：(x,y) ∈ { (0,1), (1,0), (1,1) } 实际可能出现： (x,y) = (0,0) ← 灵异！ 原因：①② 无数据依赖，可被重排成 ②① ；③④ 同理。 若两线程都先执行了"读"（② 与 ④），再执行"写"， 就出现 x=0 且 y=0。这正是 x86 唯一允许的 StoreLoad 重排（写后读换位）。

这不是理论臆想。在真实 x86（强内存模型 TSO）上，用 上一节的 jcstress 跑上千万次，就能稳定观测到一小撮 (0,0)。能被工具复现出来，才叫真陷阱，而不是"理论上存在"。

联系上一篇《并发测试》：为什么普通 JUnit 测不出这种 bug？因为它出现概率极低，且强依赖硬件与 JIT 优化时机。jcstress 之所以专门存在，就是用海量迭代 + 内存打乱 + 多平台，把这类"重排 / 可见性"缺陷从概率的阴影里逼出来。指令重排是"病",jcstress 是"CT 机"——这也是本章把测试和 JMM 放在一起讲的原因。

四、内存屏障：按住重排的手

要禁止某处重排，靠的是内存屏障（Memory Barrier / Fence）——一条特殊 CPU 指令，像马路上的硬隔离栏：栏两侧的车（指令）绝不许越界互换。JSR-133 把它抽象成四种，按"前操作—后操作"的类型组合命名：

屏障含义（X 屏障 Y：X 不能重排到 Y 之后）成本（x86）
LoadLoad读1 → 读2：保证读1先于读2完成几乎免费
StoreStore写1 → 写2：保证写1先刷回，再做写2几乎免费
LoadStore读 → 写：读先完成，写才能动几乎免费
StoreLoad写 → 读：写彻底落地、清空 store buffer 后，读才能进行昂贵（唯一在 x86 有真实开销）

volatile 正是靠这四种屏障实现的（上一章《volatile 与 happens-before》已详解插入策略）。而它们真正的物理落地，取决于 CPU 的内存模型强弱：

x86 / TSO（强模型）
只允许一种重排：
  写后读（StoreLoad）

→ LoadLoad/StoreStore/LoadStore
  基本是空操作
→ 只有 StoreLoad 要真屏障
  （lock 前缀 / mfence）

代价：屏障便宜，但"看起来正常"
的代码在这里 bug 反而藏得更深
ARM / Power（弱模型）
四种重排几乎都允许

→ 每种屏障都要真指令
  （dmb ish 等）

代价：屏障更贵，但也逼你
把同步写对——少了 volatile
的代码在这里立刻暴雷

启示：x86 上"能跑"≠ 正确
陷阱二："我在自己电脑上跑了一万次都没问题" 不能证明代码正确。你大概率用的是 x86，它是四种重排里最保守的强模型，很多缺同步的代码在 x86 上碰巧不出错。同一段代码换到手机、Mac 的 ARM 芯片（弱模型）上，重排更激进，立刻翻车。并发正确性要靠 JMM 规则（happens-before）证明，而不是靠"在某台机器上测没崩"。

五、最出名的重排陷阱：DCL 单例

双重检查锁（Double-Checked Locking）是重排"坑人"的头号案例，几乎每个 Java 面试都会问。先看那个看似正确、实则致命的版本：

✗ 少了 volatile
class Singleton {
  private static Singleton inst;
  static Singleton get() {
    if (inst == null) {          // ①
      synchronized (Singleton.class) {
        if (inst == null)        // ②
          inst = new Singleton();// ③ 危险
      }
    }
    return inst;                  // ④
  }
}
✓ 加上 volatile
class Singleton {
  private static volatile Singleton
      inst;                       // 关键字
  static Singleton get() {
    if (inst == null) {
      synchronized (Singleton.class) {
        if (inst == null)
          inst = new Singleton();
      }
    }
    return inst;
  }
}

问题出在第③行 inst = new Singleton()。它不是一条原子指令，而是三步，且第2、3步之间可以被重排：

inst = new Singleton() 实际展开为三步： 1. memory = allocate(); // 分配内存，得到地址 2. ctor(memory); // 调用构造函数，初始化对象 3. inst = memory; // 把 inst 指向这块地址（此刻 inst != null） 编译器 / CPU 允许把顺序重排为 1 → 3 → 2 ： 1. memory = allocate(); 3. inst = memory; // inst 已经 != null，但对象还没构造完！ 2. ctor(memory); // 姗姗来迟 ┌─ 线程 A 正卡在 3 和 2 之间 ─┐ │ 线程 B 执行 ① if(inst==null) 判为 false， │ │ 直接 return 一个"半初始化"对象 → 拿去用就爆炸 │ └──────────────────────────────────────────────┘
volatile 是怎么修好它的？给 inst 加 volatile 后，写 inst = memory 前会插入 StoreStore 屏障，逼着"构造对象（第2步）"必须先于"发布引用（第3步）"完成；读 inst 后的 LoadLoad 屏障，保证别的线程读到非 null 引用时，对象一定已构造完整。这与上一章"搭便车（piggyback）安全发布"是同一个机制——用一次 volatile 写把它之前的一切修改一起、有序地发布出去。这也是 Java 5+ 修订 JMM 后，DCL 必须配 volatile 才成立的根本原因。
更省心的替代方案：静态内部类（Holder 模式）。与其和 volatile + DCL 较劲，不如利用类初始化的天然线程安全——JVM 保证一个类的 <clinit> 只被执行一次且对所有线程可见（靠类加载锁）。private static class Holder { static final Singleton INST = new Singleton(); }，外部 return Holder.INST; 触发懒加载。既懒汉、又线程安全、还没有屏障心智负担——《Effective Java》推荐的正是它和枚举单例。

六、另一个隐蔽陷阱：long / double 的非原子写

重排之外，JMM 还留了一个冷门但致命的坑。JLS 规定：对 long / double 这两个 64 位类型的普通读写，不保证原子性——JVM 允许把它拆成两个 32 位操作。

WordTearing.java64 位写可能被撕裂
long value;   // 非 volatile

// 线程 A
value = 0xFFFFFFFF_FFFFFFFFL;   // 写全 1，可能拆成 高32位 + 低32位 两步

// 线程 B 恰好在两步之间读，可能读到"半新半旧"：
// 0xFFFFFFFF_00000000  —— 一个从未被任何线程写入过的诡异值（字撕裂 word tearing）
陷阱三：共享的 long / double 字段不加 volatile，可能读到"拼接出来的幽灵值"。在 64 位 JVM 上现代实现通常已保证其原子性，所以你平时"碰巧没事"；但 JLS 并不强制，32 位平台或某些实现就可能撕裂。只要一个 long/double 会被多线程共享，就给它加 volatile——volatile 的 64 位读写 JLS 明确要求原子。（注意：这只解决"单次读写"的原子与可见，复合运算 value++ 仍要 AtomicLong。）

七、四把"锁不住乱序"的钥匙怎么选

禁止有害重排、建立正确的内存顺序，Java 给了你几件工具，能力与代价各不相同：

手段禁止重排的范围额外能力代价
volatile屏障两侧的读/写不越界可见性 + 有序性低（读几乎零成本）
final构造期写 → 引用发布 不重排安全发布不可变对象零（无运行期开销）
synchronized/Lock整个临界区不外泄原子性 + 可见 + 有序较高（可能阻塞）
VarHandle/CAS可精确指定 acquire/release/opaque无锁原子 + 细粒度顺序低，但心智门槛高
被忽视的 final：很多人只知道 final 表示"不可变",却不知它有专属的内存语义。JMM 保证：只要对象正确构造（this 没在构造期逸出），那么其他线程即便通过普通引用（非 volatile）拿到它，也一定能看到 final 字段被正确初始化后的值——JVM 在构造函数末尾对 final 字段插入了 StoreStore 屏障。这就是 String、Integer 等不可变类不用任何同步就能安全共享的底层依据，也是上一章"不变性"能成为并发利器的原因。

八、把重排"钉死"的形式化契约

逐个屏障去推理太累。JMM 最终把这一切抽象成一个你只需记住的判据——只要在写与读之间建立起 happens-before 关系，就不允许出现"读看不到写"或"重排导致错乱"：

写 happens-before 读   ⟹   无有害重排 ∧ 结果可见

建立这条边的方式，就是上一章那几条规则：volatile 写→读、锁的 unlock→lock、Thread.start()、join()、final 字段构造语义，再用传递性串成链。你不必关心底层插了哪种屏障——那是 JVM 按平台去兑现契约的实现细节。

一句话收束：指令重排不是 bug，而是编译器 / CPU / 内存系统为压榨性能付出的"合法作弊"，它被 as-if-serial 关在单线程的笼子里——可一旦跨线程共享数据，笼子就消失了，于是有了 (0,0) 的灵异、DCL 的半初始化对象、long 的字撕裂。内存屏障是按住乱序的物理手段，而 happens-before 是你不必碰屏障、就能推理正确性的高层契约。所以并发正确性的黄金法则是：不要靠"在我机器上测着没崩"来判断，而要靠 volatile / final / 锁 / CAS 显式建立 happens-before——这，也正是本章 JMM 与并发测试合并讲解想留给你的最终直觉。
