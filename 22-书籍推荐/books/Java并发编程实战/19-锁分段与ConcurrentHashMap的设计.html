锁分段与 ConcurrentHashMap 的设计

降低锁竞争的三板斧里，最锋利的一招是"把一把热锁拆成 N 把凉锁"——这就是锁分段，也是 ConcurrentHashMap 高吞吐的灵魂。

一、承上：锁竞争是可伸缩性的头号敌人

上一篇《性能权衡》讲了并发的三大隐性开销：上下文切换、内存同步、锁竞争。其中锁竞争最致命——因为它直接把程序里"本可以并行"的部分强行拉回"串行"。回忆 Amdahl 定律：加速比受限于串行占比 F。一把被所有线程争抢的独占锁，就是一段谁也绕不过的串行区间，你加再多核也白搭。

Speedup ≤ 1 / ( F + (1 − F) / N )　　F = 串行占比

那么"锁竞争"到底由什么决定？《Java 并发编程实战》给出一个朴素但极其有用的判据：

锁竞争程度 ∝ 请求该锁的频率 × 持有该锁的时长
这个乘积告诉我们降低竞争只有三条路。要么减少持锁时长（缩小锁的范围，锁外的活儿别占着锁干）；要么降低请求频率——而"降低频率"又分两种玩法：把一把大锁按无关状态拆成几把锁（锁分解 lock splitting），或把一把锁按同类状态的分片拆成一组锁（锁分段 lock striping）。本篇聚焦最后一招，它正是 ConcurrentHashMap 可伸缩性的来源。

二、第一招：缩小锁的范围（先把能省的省掉）

在动用花哨的拆锁技巧前，最廉价的优化是让锁只保护真正需要保护的那几行。持锁时间越短，其他线程等待的窗口就越小，竞争概率随之下降。下面是本书里的经典反例：把一堆与共享状态无关的运算（正则匹配、日志、耗时计算）也圈进了 synchronized 块。

✗ 锁保护范围过大
public synchronized void
        addUser(String u) {
    // ↓ 与共享状态无关，却占着锁
    String key = normalize(u); // 耗时
    boolean ok = PATTERN
        .matcher(key).matches();
    users.add(key);            // 只有这行需锁
    log.info("added " + key);  // 又占着锁做 IO
}
✓ 锁只圈住共享状态
public void addUser(String u) {
    String key = normalize(u); // 锁外做
    boolean ok = PATTERN
        .matcher(key).matches();
    synchronized (this) {
        users.add(key);        // ★ 只锁这里
    }
    log.info("added " + key);  // 锁外做 IO
}
陷阱：别把"缩小范围"做成"频繁进出锁"。反复地"获取—释放—再获取"同一把锁，会引入额外的加解锁开销（尤其锁升级抖动）。如果一段逻辑必须整体原子，硬拆成好几个小 synchronized 块反而更糟——既破坏原子性，又增加请求频率。缩范围的正确姿势是：把与锁无关的代码移出去，而不是把需要原子的代码切碎。

三、第二招 vs 第三招：锁分解 与 锁分段

缩范围之后如果还热，就该拆锁了。这里有两个极易混淆的术语，面试常考它们的区别：

维度锁分解 Lock Splitting锁分段 Lock Striping
拆什么一把锁保护多个互相独立的状态变量，按变量拆一把锁保护一组同类数据（如数组/表），按分片拆
锁的数量固定、少量（几把）可以很多（一组，如 16 把、N 把）
典型例子一个统计类里，用户锁保护 users、查询锁保护 queriesConcurrentHashMap、本书的 StripedMap
关系—锁分段可看作锁分解的推广：分解到"一组数量可变的锁"

锁分解：把互不相干的状态解绑

如果一把锁同时守着 users 和 queries 两个毫无关系的集合，那么"改 users 的线程"和"改 queries 的线程"会无谓地互相阻塞。分解很直白——给它们各配一把锁：

ServerStatus.java把一把锁分解成两把独立锁
public class ServerStatus {
    private final Set<String> users;      // 状态 A
    private final Set<String> queries;    // 状态 B，与 A 无关

    // ✗ 老写法：addUser 和 addQuery 抢同一把 this 锁，互相拖累
    // ✓ 分解后：各自锁各自的集合，两类操作可并行
    public void addUser(String u) {
        synchronized (users) { users.add(u); }
    }
    public void addQuery(String q) {
        synchronized (queries) { queries.add(q); }   // 不再等 users 的锁
    }
}
建立联系。这其实就是我们在《组合设计线程安全的类》里讲的把同步策略下放：与其用一把外层大锁笼罩整个对象，不如让每个独立状态各自持有守护它的锁。锁分解的前提是——被拆开的状态之间确实没有一致性约束。若 users 和 queries 之间有"必须同增同减"的不变式，就绝不能拆，否则会破坏原子性。

四、锁分段：把一把热锁拆成 N 把凉锁

锁分解拆的是"不同种类"的状态；但像 HashMap 的桶数组，成千上万个桶是同一种状态，没法按语义分解。锁分段的思路是按哈希分片切：准备 N 把锁（N 远小于桶数），第 i 个桶由第 i % N 把锁守护。这样访问不同分片的线程各持不同的锁，天然并行。

锁分段结构：16 把锁守护整张桶数组（N_LOCKS = 16） buckets[]: [0][1][2][3] ... [16][17] ... 共 numBuckets 个桶 │ │ │ │ │ │ locks[16]: L0 L1 L2 L3 ... L0 L1 ... 桶 i 由 locks[i % 16] 守护 ▲ ▲ └── 桶0、桶16、桶32…共用 L0（同分段互斥） 不同分段（L0 vs L1）→ 完全并行写 竞争概率 ≈ 原来的 1/16 （请求频率被摊到 16 把锁上）

下面是《Java 并发编程实战》第 11 章的原始 StripedMap 示例（略作精简）。它不是 ConcurrentHashMap，而是把锁分段这招剥离出来的最小教学实现——看懂它，就看懂了 CHM 的底层直觉。

StripedMap.java（JCiP 清单 11.8 精简）锁分段的最小实现
// 同步策略：buckets[n] 由 locks[n % N_LOCKS] 守护
public class StripedMap {
    private static final int N_LOCKS = 16;
    private final Node[] buckets;
    private final Object[] locks;      // ★ 一组锁，而不是一把

    public StripedMap(int numBuckets) {
        buckets = new Node[numBuckets];
        locks = new Object[N_LOCKS];
        for (int i = 0; i < N_LOCKS; i++)
            locks[i] = new Object();   // 16 个互不相干的锁对象
    }

    private int hash(Object key) {
        return Math.abs(key.hashCode() % buckets.length);
    }

    public Object get(Object key) {
        int hash = hash(key);
        synchronized (locks[hash % N_LOCKS]) {   // ★ 只锁 key 所在的那个分段
            for (Node m = buckets[hash]; m != null; m = m.next)
                if (m.key.equals(key)) return m.value;
        }
        return null;
    }
}
为什么 N 把锁而不是"一桶一锁"？理论上桶越多、锁越多，并发度越高。但每把锁都是一个对象、要占内存、还要被 JVM 逐一管理；桶动辄上万，一桶一锁会让锁本身的开销压过收益。16 把锁是工程上"够用又不浪费"的经验值——它把竞争降到约原来的 1/16，同时只多花 16 个轻量对象。这也是 JDK 7 ConcurrentHashMap 默认 concurrencyLevel = 16 的由来。

五、锁分段的代价：独占整张表变贵了

天下没有免费的午餐。锁分段让"单点访问"飞快，却让"需要独占整个集合"的操作变得昂贵——因为你得依次拿到所有分段的锁才能安全地重排整张表。StripedMap.clear() 就得逐段加锁；ConcurrentHashMap 的 size()、扩容 resize 也同理。

StripedMap.javaclear：独占操作要"集齐"所有分段锁
public void clear() {
    for (int i = 0; i < buckets.length; i++) {
        synchronized (locks[i % N_LOCKS]) {   // 每个分段都要锁一遍
            buckets[i] = null;
        }
    }
    // 扩容(rehash)更麻烦：必须同时握住全部 N 把锁，
    // 期间整张表对写入是"冻结"的 —— 这是分段方案最贵的时刻
}
陷阱：size() 在分段容器里天生就是"估计值"。要得到精确大小，就得同时锁住所有分段，那等于把整张表退化成"一把大锁"，前面拆锁的努力全白费。所以 ConcurrentHashMap 索性放弃精确——JDK 8 用类似 LongAdder 的分片计数（多个 counterCell 分散热点），size() 只在求和瞬间近似。永远不要写 for (int i = 0; i < map.size(); i++)，也别拿 size()==0 当严格空判断。这正是"用一致性换吞吐"的取舍。（关于弱一致性与原子复合方法的用法细节，见 并发容器 一篇。）

六、ConcurrentHashMap：锁分段思想的两代实现

ConcurrentHashMap 是锁分段最著名的落地，但它的实现随 JDK 演进：

JDK 7：Segment 分段锁JDK 8：桶级 CAS + synchronized
锁粒度固定 16 段（concurrencyLevel），一段一把 ReentrantLock单个桶（数组的一个槽），粒度随容量增长
并发度创建后写死 = 段数，最多 16 线程并行写≈ 桶数，容量越大能并行的写越多
结构两层哈希：先定位 Segment，再定位桶一层数组 + 链表/红黑树（同 HashMap）
空桶写入加段锁后写CAS 无锁写入，抢失败重试
扩容段内独立扩容多线程协同搬迁（helpTransfer）
JDK 8 是"锁分段思想"的胜利，而非"Segment 实现"的延续。它把 Segment 这层中间商彻底删掉，锁粒度从"16 段"细化到"每个桶"——本质上是把分段数从固定 16 推向了"和桶数一样多"的极限，再叠加 CAS 让空桶写入连锁都省了。Segment 的类还在源码里，但只为反序列化旧数据保留，早已不参与主逻辑。所以面试若问"CHM 还用分段锁吗"，答案是：分段的思想在，Segment 的实现没了。

把三种 Map 放在一起看可伸缩性（线程数增加时的相对吞吐，示意）：

Hashtable synchronizedMap CHM (JDK7 分段)
ConcurrentHashMap (JDK8 桶级 CAS)

↑ 越长表示线程增多时吞吐扩展得越好。整表大锁的 Hashtable 几乎不随核数提升；桶级方案则接近线性扩展。

七、什么时候值得动手拆锁？

别为了拆锁而拆锁——先测量，后优化。锁分解/分段会让代码更复杂、更容易写错（尤其是那些需要独占全表的操作）。只有当性能剖析（profiler / jstack 抓到大量 BLOCKED 线程）证明某把锁确实是热点时，拆它才有意义。对一把竞争本就轻微的锁做分段，只会白白增加内存与复杂度，还可能因为多把锁的缓存行争用（false sharing）而变慢。这与上一篇《性能权衡》的主旨一致：过早优化是万恶之源。
一句话收束：降低锁竞争 = 减少持锁时长（缩小锁范围） + 降低请求频率（拆锁）。拆锁有两式——锁分解按"互不相干的状态"拆成几把固定的锁；锁分段按"同类数据的哈希分片"拆成一组锁，把一把热锁的竞争摊到 N 把凉锁上，竞争概率降到约 1/N。ConcurrentHashMap 正是锁分段的巅峰：JDK 7 用 16 个 Segment，JDK 8 把粒度推到每个桶并叠加 CAS。但分段有代价——任何需要独占整张表的操作（size、clear、resize）都得集齐所有分段锁，因此 size() 只能是估计值。最后记住：拆锁前先测量，别为一把凉锁徒增复杂度。
