同步工具：CountDownLatch、CyclicBarrier、Semaphore

不再手写 wait/notify——用三个现成的“交通信号灯”协调线程的开始、汇合与准入。

为什么需要“同步工具类”

前面我们用 synchronized、wait/notify 做过线程协作，但那套 API 极易写错：忘了在循环里检查条件、丢失唤醒、虚假唤醒……同步工具类（Synchronizer）把这些常见的协作模式封装成对象，你只需要“描述意图”，不必再操心底层的等待/唤醒细节。

它们的共同本质是：根据自身状态，决定到达的线程是“放行”还是“阻塞”。这三个类都建立在 AbstractQueuedSynchronizer（AQS）之上——AQS 用一个 volatile int state 表示同步状态，用一个 CLH 变体的等待队列挂起线程。理解这一点，就理解了它们“为什么长得像亲戚”。

一句话区分： CountDownLatch 是“一次性发令枪”（等 N 件事做完再放行）；CyclicBarrier 是“可循环的集合点”（N 个线程互相等，凑齐再一起走）；Semaphore 是“许可证发放机”（同时最多 N 个人进场）。

CountDownLatch：等 N 件事做完

CountDownLatch 用一个计数器初始化。线程调用 await() 会阻塞，直到计数器被 countDown() 减到 0，此时所有等待线程一起被唤醒。它是单向、一次性的——归零后就永久失效，不能重置。

典型用途有两类：①开关（主线程持有一个 count=1 的门闩，所有工作线程 await() 在起跑线上，主线程一 countDown() 就同时起跑，用于测并发性能）；②等待完成（count=N，每个任务干完 countDown() 一次，主线程 await() 等全部结束）。

CountDownLatch.java等待所有子任务完成
public class DataLoader {
    public void loadAll(List<String> shards) throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(shards.size());  // 计数 = 分片数
        ExecutorService pool = Executors.newFixedThreadPool(4);
        for (String shard : shards) {
            pool.submit(() -> {
                try {
                    loadShard(shard);          // 真正干活
                } finally {
                    latch.countDown();       // 关键：放 finally，异常也要减，否则永久阻塞
                }
            });
        }
        latch.await();                     // 阻塞直到计数归零
        System.out.println("全部分片加载完成");
    }
}
高频陷阱： countDown() 一定要放进 finally！如果任务抛异常直接跳过了 countDown()，计数器永远归不了零，await() 的线程就会永久挂起——这类“任务少减一次导致主线程卡死”的 bug 在生产环境非常隐蔽。另外别忘了 await() 有带超时的重载 await(timeout, unit)，生产代码几乎都应该用它兜底。
联系 JDK 源码： CountDownLatch 内部用 AQS 的共享模式。构造时把计数值写进 state；await() 调 acquireSharedInterruptibly，只要 state != 0 就入队挂起；countDown() 调 releaseShared，用 CAS 把 state 减 1，减到 0 时唤醒队列里所有共享等待者——这就是“一次放行全部”的实现原理。

CyclicBarrier：N 个线程互相等待

CyclicBarrier（栅栏）让一组线程彼此等待，都到达同一个“集合点”后再一起继续。和门闩最大的不同：它可以重复使用（Cyclic = 循环），凑齐一批放行后自动重置，用于下一轮。它还能带一个 barrierAction——最后一个到达的线程会先执行它，再释放大家，天然适合“汇总本轮结果”。

CountDownLatch（发令枪，一次性） CyclicBarrier（集合点，可循环） 主线程 ──countDown──▶ [ 门闩 ] T1 ─await─▶┐ 工作线程 A ─await─▶ ⌛ T2 ─await─▶┤ 凑齐3个 工作线程 B ─await─▶ ⌛ count==0 T3 ─await─▶┘ ──▶ [barrierAction] 工作线程 C ─await─▶ ⌛ ──▶ 全部放行 └──▶ 一起进入下一轮 （谁减到 0 都行） （必须互相等齐）
CyclicBarrier.java多线程分步迭代计算
int workers = 3;
// barrierAction：最后一个到达者执行，用来做每轮的汇总
CyclicBarrier barrier = new CyclicBarrier(workers,
        () -> System.out.println("本轮结束，合并结果"));

for (int i = 0; i < workers; i++) {
    new Thread(() -> {
        for (int round = 0; round < 5; round++) {
            computeSubGrid(round);       // 各自算自己那块
            try {
                barrier.await();        // 等所有人算完本轮，再进下一轮
            } catch (InterruptedException | BrokenBarrierException e) {
                return;                    // 栅栏被打破，退出
            }
        }
    }).start();
}
BrokenBarrier 陷阱： 如果一个线程在 await() 时被中断、超时，或 barrierAction 抛异常，整个栅栏会进入破损（broken）状态——此刻其余所有 await() 的线程会立刻抛 BrokenBarrierException 而不会继续等待。这是“全体成败与共”的设计：一个掉队，全队重来。想复用得手动 barrier.reset()。

Latch vs Barrier：别用错

两者最容易混。核心差异是“谁等谁”和“能不能重来”：

维度CountDownLatchCyclicBarrier
等待关系A 组线程等 B 组事件（可以不是同一批线程）同一组线程互相等待
是否可重用否，归零即失效是，放行后自动重置
触发条件countDown() 被调用足够次数足够多线程到达 await()
汇合动作无支持 barrierAction
底层实现AQS 共享模式ReentrantLock + Condition
计数方向递减到 0递增到目标数（内部 count 递减）
记忆锚点： Latch 像裁判鸣枪（一个信号，全体出发，用完扔掉）；Barrier 像登山队约定的休息点（每个人到了都得等最慢的，凑齐再一起走下一段，下一个休息点还接着用）。

Semaphore：限制并发访问数

Semaphore（信号量）维护一组许可（permit）。acquire() 拿走一个许可（没有就阻塞），release() 归还一个。它把“同时最多允许 N 个线程访问某资源”这件事直接对象化——本质是把 synchronized 的“互斥（最多 1 个）”推广成“最多 N 个”。当 permit = 1 时，它退化成一个可跨线程释放的互斥锁。

经典场景：数据库连接池、限流、有界资源池。想象一个只有 3 个车位的停车场：

Semaphore.java限流：最多 3 个并发请求外部服务
public class RateLimitedClient {
    // 3 张许可 = 同一时刻最多 3 个线程能进去；true 表示公平模式
    private final Semaphore semaphore = new Semaphore(3, true);

    public String call(String url) throws InterruptedException {
        semaphore.acquire();              // 拿车位，没位就排队
        try {
            return doRemoteCall(url);       // 访问受保护资源
        } finally {
            semaphore.release();          // 关键：必须在 finally 归还，否则许可越用越少
        }
    }
}

反例 → 正例：用 Semaphore 做有界资源池

Bad · 忘记归还许可
sem.acquire();
Result r = doRemoteCall(url);
// doRemoteCall 抛异常时
// 下面这行永远执行不到
sem.release();
// → 许可泄漏，permit 逐渐耗尽
// → 最终所有线程卡在 acquire()
Good · try/finally 保证归还
if (!sem.tryAcquire(2, SECONDS)) {
    throw new RejectedException(); // 快速失败
}
try {
    return doRemoteCall(url);
} finally {
    sem.release(); // 异常也归还
}
两个易错点：（1）acquire/release 不要求成对同线程——A 线程 acquire、B 线程 release 完全合法，这也是它能当“生产-消费令牌”的原因；正因为如此，编译器不会帮你检查配对，泄漏全靠自觉。（2）你甚至可以 release() 出比初始更多的许可来动态扩容资源池，但这也意味着过度 release 会悄悄突破你设定的并发上限，务必小心。
联系底层： Semaphore 就是操作系统教科书里 Dijkstra 提出的信号量（P/V 操作，即 acquire/release）在 JVM 层的实现，底层同样基于 AQS 共享模式：state 就是当前剩余许可数，acquire 用 CAS 把 state 减 1（减到负数就入队等待），release 把 state 加 1 并唤醒等待者。它和 synchronized 的区别在于：synchronized 靠 monitor 强制“同一线程加锁解锁”，而信号量的许可是“无主”的。

公平 vs 非公平

Semaphore（以及 ReentrantLock）的构造函数有个 fair 参数。这背后是吞吐量与公平性的经典权衡：

非公平（默认）公平（fair=true）
抢占方式新来的线程可“插队”直接拿许可严格按入队顺序（FIFO）发放
吞吐量高（减少线程切换）较低（唤醒排队者有开销）
饥饿风险有（倒霉线程可能长期抢不到）无
适用绝大多数高并发场景要求响应时间可预测、防饥饿
为什么默认非公平？ 因为线程被唤醒到真正运行有延迟，公平锁在“唤醒队首”期间，一个正在运行的线程往往能瞬间抢到并用完再释放——非公平模式利用这个时间窗把许可交给活跃线程，避免频繁的上下文切换，整体吞吐更高。这与 CLAUDE.md 里强调的“方案取舍”一致：默认值往往是为通用场景优化的。

延伸对比： 若你要“提交任务并拿回结果”，用 Future/CompletableFuture 而不是门闩；若要“等一批任务全部完成再继续”，CompletableFuture.allOf(...) 往往比手写 CountDownLatch 更表意。同步工具解决的是线程协作时机，不是结果传递——别拿门闩去当返回值容器。
核心收束： 三者都建立在 AQS 之上，区别只在“状态如何决定放行”。CountDownLatch——一次性、单向、等事件完成，谁 countDown 都行；CyclicBarrier——可循环、双向、同组线程互等，一人掉队全队 broken；Semaphore——控并发数，许可无主、可跨线程借还。三者的 acquire / countDown / release 都要用 try/finally 兜底，并优先用带超时的重载防止永久阻塞。
