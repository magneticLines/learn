组合设计线程安全的类

线程安全不必每次从锁写起——把已经安全的零件封起来、拼起来，才是工程师真正该做的事；但零件本身安全，拼出来的整体未必安全。

一、设计线程安全类，先回答三个问题

前几节我们从"底层机制"（synchronized、volatile、happens-before）一路讲到"别把对象发布错"。现在把镜头拉高：面对一个要在多线程下用的类，该从哪里下手？《Java 并发编程实战》给了一套朴素但极其可靠的流程，本质是三个问题：

设计线程安全类的三步（JCIP 第 4.1 节）：
① 找出构成对象状态的所有变量 —— 对象状态就是它所有实例字段与静态字段组成的集合（还包括它所引用对象的字段）。
② 找出约束状态变量的不变式（invariant） —— 比如"lower ≤ upper"、"计数器非负"、"缓存值必须与被缓存的输入匹配"。不变式限定了哪些状态是合法的、哪些状态转换是允许的。
③ 建立管理并发访问的同步策略（synchronization policy） —— 用什么锁、保护哪些变量、哪些操作必须原子。
线程安全类 = 状态变量 + 不变式约束 + 同步策略

为什么这三步的顺序不能乱？因为不变式决定了原子性的粒度：只有先搞清楚"哪些变量之间存在约束关系"，你才知道"哪几个操作必须捆成一个原子块"。跳过第②步直接给每个字段加锁，正是本节后面要拆的那类经典 bug。

二、实例封闭：把状态关进笼子，再配一把锁

最基础的线程安全构建手段叫实例封闭（instance confinement）：把一个非线程安全的对象封装进另一个对象里，并保证所有访问它的路径都持有同一把锁。这样即使内部零件不安全，外壳也能安全。

实例封闭：把非线程安全的 HashSet 关起来PersonSet
public class PersonSet {
    // HashSet 本身不是线程安全的，但它被封闭在 PersonSet 内部，
    // 且只能通过下面两个 synchronized 方法访问 —— 由 this 这把锁守护
    private final Set<Person> mySet = new HashSet<>();

    public synchronized void addPerson(Person p) {
        mySet.add(p);
    }
    public synchronized boolean containsPerson(Person p) {
        return mySet.contains(p);
    }
}
你其实天天在用它：Collections.synchronizedList(...) / synchronizedMap(...) 就是实例封闭的现成实现——装饰器（Decorator）把原始集合包起来，每个方法都 synchronized 后再转发。它把"非线程安全的 ArrayList"和"一把守护它的锁"绑成一个整体。理解了这一点，你就明白为什么拿到包装后的引用还得小心（后面第六节会看到坑）。

三、Java 监视器模式，与"私有锁"的关键取舍

把状态全部封闭在对象内部、并用对象自己的内置锁（即 this）守护全部状态——这个最常见的模式叫 Java 监视器模式（Java monitor pattern）。前面的 Counter、PersonSet 都是它的实例。它的好处是简单；但用 this 当锁有个隐患：这把锁是"公开"的，任何拿到你对象引用的客户端代码都能 synchronized(yourObject) 把它锁住。

✗ 用 this 当锁：锁是公开的
public class Counter {
  private long value = 0;
  public synchronized long inc() {
    return ++value;   // 锁的是 this
  }
}

// 恶意/无意的客户端:
Counter c = ...;
synchronized (c) {
  // 长时间持有 c 的锁不放
  Thread.sleep(60_000);
}
// 期间所有人的 inc() 全被阻塞
// → 拒绝服务 / 意外死锁
✓ 私有锁对象：锁被封起来
public class Counter {
  private final Object lock
      = new Object();          // 外部拿不到
  private long value = 0;
  public long inc() {
    synchronized (lock) {
      return ++value;
    }
  }
}

// 客户端只能拿到 Counter 引用，
// 拿不到 lock，无法干扰你的同步策略
// → 封装更彻底，可独立演进
陷阱一：把 this 当锁，等于把"锁协议"也暴露给了外界。客户端一旦参与进你的锁（哪怕是好意地做客户端加锁），你就再也无法安全地更换同步策略——比如某天想从"一把大锁"改成"分段锁"，会悄悄破坏所有依赖 this 锁的外部代码。私有锁对象（private final Object lock）把同步策略变成了纯粹的实现细节，这也是很多 JDK 类（如早期的部分内部实现）和高质量库偏爱私有锁的原因。代价是：客户端无法做客户端加锁来扩展原子操作（见第六节，那时反而要用 this 或组合）。

四、委托：把线程安全"外包"给现成的安全组件

监视器模式什么都自己扛，但现代 JDK 已经给了我们大量本身就线程安全的构建块（ConcurrentHashMap、AtomicInteger、CopyOnWriteArrayList…）。如果一个类的状态完全由这些安全组件承载，那它可以把线程安全委托（delegation）给它们，自己一行 synchronized 都不用写：

委托给 ConcurrentHashMap（Point 为不可变类）DelegatingVehicleTracker
public class DelegatingVehicleTracker {
    private final ConcurrentMap<String, Point> locations;
    private final Map<String, Point> unmodifiableMap;

    public DelegatingVehicleTracker(Map<String, Point> points) {
        locations = new ConcurrentHashMap<>(points);
        unmodifiableMap = Collections.unmodifiableMap(locations);
    }

    // 没有一个 synchronized：所有线程安全都由 ConcurrentMap 承担
    public Point getLocation(String id) { return locations.get(id); }

    public void setLocation(String id, int x, int y) {
        if (locations.replace(id, new Point(x, y)) == null)
            throw new IllegalArgumentException("invalid vehicle: " + id);
    }
}

这里有两个值得点出的设计细节：Point 必须是不可变的——否则把它 get 出去就等于发布了可变状态（回顾上一节的"逸出"）；而 setLocation 里用 replace 而非 put，是借 ConcurrentMap 的原子方法维持"只能更新已存在车辆"这条约束。

五、委托的边界：零件都安全，拼起来却翻车

委托很香，但它有一条致命前提，也是本节最高频的面试点与线上事故源：当类的不变式跨越了多个状态变量时，委托就会失效。看这个"数值范围"类——它用两个 AtomicInteger 分别存下界和上界，每个都线程安全，但整体不线程安全：

反例：不变式 lower ≤ upper 跨越两个变量NumberRange —— 有 bug！
public class NumberRange {
    // 不变式：lower <= upper
    private final AtomicInteger lower = new AtomicInteger(0);
    private final AtomicInteger upper = new AtomicInteger(0);

    public void setLower(int i) {
        // 警告：不安全的 check-then-act（先检查后执行）
        if (i > upper.get())
            throw new IllegalArgumentException("lower > upper");
        lower.set(i);
    }
    public void setUpper(int i) {
        if (i < lower.get())
            throw new IllegalArgumentException("upper < lower");
        upper.set(i);
    }
}

每个 AtomicInteger 的 get/set 都是原子的，可 setLower 内部"先读 upper，再写 lower"这两步中间没有任何东西拦得住另一个线程插进来。两个线程同时改，就能把范围搅成非法状态：

初始范围 (lower=0, upper=10) 不变式要求 lower <= upper 线程 A: setLower(5) 线程 B: setUpper(4) │ 检查 5 > upper(10) ? 否 ✓ │ 检查 4 < lower(0) ? 否 ✓ │ (各自的检查此刻都合法) │ ▼ ▼ lower.set(5) upper.set(4) 最终范围 (lower=5, upper=4) → lower > upper，不变式被击穿！ 两个 AtomicInteger 各自线程安全，组合起来却不安全。
陷阱二：能不能委托，取决于"不变式关不关联多个变量"。如果状态变量彼此独立（一个变量的合法取值不依赖另一个），可以放心把线程安全委托给多个独立的安全变量。可一旦存在 NumberRange 这种跨变量的复合不变式，委托就必须让位给"用同一把锁把相关操作捆成原子块"——把 setLower/setUpper 加上 synchronized，或把 lower/upper 换成一个不可变的 Range 对象整体替换（写时复制）。把这道题记牢：面试官问"两个 AtomicInteger 组合还线程安全吗"，答案是否定的。
什么时候可以安全地把状态变量直接发布出去（getter 返回内部变量）？JCIP 给了三个必须同时满足的条件：① 该变量本身是线程安全的；② 它不参与任何约束其取值的不变式；③ 它没有任何被禁止的状态转换。上面 NumberRange 的 lower 就违反了②，所以既不能委托、也不能公开发布。这条判据把"委托"和"发布"串成了同一套逻辑。

六、给已有线程安全类"加操作"：三条路，两个坑

常见需求：List 已经是线程安全的（Collections.synchronizedList），但我想要一个原子的 put-if-absent（不存在才添加）。它是 contains + add 的 check-then-act 复合操作，现成 API 没有。怎么原子地加上去？有三条路：

路 A：继承（扩展）—— 最脆弱

直接 extends 那个类，加一个 synchronized putIfAbsent。问题：你的锁能不能守住基类的操作，取决于基类用哪把锁做同步。基类哪天改了同步策略（换成私有锁），你的扩展就会静默失效。继承把你和基类的实现细节死死绑在一起。

路 B：客户端加锁 —— 必须锁"同一把锁"

不改类，在调用方用外部代码把复合操作锁起来。关键陷阱在于：你必须锁与被扩展对象内部使用的同一把锁，否则只是"看起来加了锁"：

✗ 锁错了对象（锁的是 this）
public class ListHelper<E> {
  public List<E> list =
    Collections.synchronizedList(
        new ArrayList<>());

  // synchronized 锁的是 ListHelper 实例
  public synchronized boolean
      putIfAbsent(E x) {
    boolean absent = !list.contains(x);
    if (absent) list.add(x);
    return absent;
  }
}
// 但 list 用的是"它自己"这把锁！
// 别的线程直接 list.add(x) 不受你阻拦
// → check-then-act 依然会竞态
✓ 锁 list 本身（同一把锁）
public class ListHelper<E> {
  public List<E> list =
    Collections.synchronizedList(
        new ArrayList<>());

  public boolean putIfAbsent(E x) {
    synchronized (list) {   // 关键
      boolean absent =
          !list.contains(x);
      if (absent) list.add(x);
      return absent;
    }
  }
}
// synchronizedList 内部正是用
// "包装对象自身"做互斥锁，对上了
陷阱三：客户端加锁最容易锁错锁。Collections.synchronizedXxx 返回的包装器，其内部同步用的锁就是包装器对象本身（即你拿到的那个 list 引用）。所以要让你的 putIfAbsent 和内部的 add/contains 互斥，就必须 synchronized (list)。用 synchronized this 是最常见的错误——它锁的是 ListHelper，和 list 内部那把锁是两把无关的锁，等于没锁。这也解释了第三节的取舍：要支持客户端加锁，锁就得是"可被外界锁到"的——这与"私有锁封装更好"是一对天然矛盾。

路 C：组合（Composition）—— 最健壮

不去猜别人的锁，而是自己再套一层：把 List 作为私有成员持有，用自己的内置锁守护所有转发方法。这样无论底层 list 是否线程安全、用哪把锁，都不影响正确性——因为外界只能透过你的 synchronized 方法碰它。

组合：多加一层锁，但彻底不依赖底层的同步策略ImprovedList
public class ImprovedList<T> implements List<T> {
    private final List<T> list;   // 被封闭，外界拿不到

    public ImprovedList(List<T> list) { this.list = list; }

    // 用 ImprovedList 自己的锁守护复合操作，与底层用什么锁无关
    public synchronized boolean putIfAbsent(T x) {
        boolean contains = list.contains(x);
        if (!contains) list.add(x);
        return !contains;
    }
    // 其余 List 方法全部同样 synchronized 后转发
    public synchronized boolean add(T e)      { return list.add(e); }
    public synchronized boolean contains(Object o) { return list.contains(o); }
    // ... clear() / remove() / size() ... 略
}

三条路横向对比，选型一目了然：

方式健壮性依赖什么代价 / 适用
A 继承扩展低（脆弱）依赖基类的具体锁与同步策略基类改同步策略即失效，仅在你掌控基类时用
B 客户端加锁中依赖被扩展对象公开、稳定的锁协议不改原类；极易锁错锁，破坏封装
C 组合高（最健壮）只依赖自己的锁，不假设底层安全多一层锁开销；首选方案
联系 Spring / JDK：Spring 大量用"组合"而非"继承"来叠加行为（如各种 Decorator、Wrapper、AOP 代理），背后正是同一哲学——组合优于继承（这条也是《Effective Java》的核心条目，作者 Joshua Bloch 同是本书作者之一）。JDK 的 Collections.synchronizedXxx 是"客户端加锁需要公开锁"的活标本；而 ConcurrentHashMap 则代表另一条路：把复合操作（putIfAbsent、computeIfAbsent）直接做进类里，从根上免去客户端拼装原子操作的麻烦。能用后者就别自己拼。
一句话收束：设计线程安全类，永远先走"状态变量 → 不变式 → 同步策略"三步，其中不变式决定原子性的粒度。构建手段由易到难有四层——实例封闭（把非安全对象关进笼子加锁，如 synchronizedList）、Java 监视器模式（全部状态用一把锁守护，优先选私有锁对象把同步策略藏成实现细节）、委托（把安全外包给 ConcurrentHashMap/Atomic 这类现成组件）。委托的红线是：一旦不变式跨越多个变量（如 NumberRange 的 lower ≤ upper），两个各自线程安全的变量组合起来就不再安全，必须回到"同一把锁捆成原子块"。给已有安全类加原子操作有继承/客户端加锁/组合三条路，组合最健壮——它只依赖自己的锁，不去赌别人的同步策略；而客户端加锁一定要锁"同一把锁"，否则形同虚设。记住这套地图，后面第 5-7 章的并发容器与同步工具，你就是在"挑现成的安全零件来委托"。
