Future、Callable 与异常处理

submit 之后拿到的那张“提货凭据”，如何取结果、控超时、接异常、喊停任务。

Runnable 的两处硬伤

把任务丢进线程池，最朴素的接口是 Runnable。但它有两个从 JDK 1.0 就带着的先天缺陷：run() 返回 void，拿不回结果；run() 的签名没有 throws，不能抛受检异常。这不是疏忽，而是当年设计 Thread 时的取舍——线程的“执行体”被定义成一个纯粹的动作，谁去接收结果、谁去处理异常，语言当时没给答案。

于是你只能靠共享变量 + 手写同步把结果“捎”出来，靠 try/catch 把受检异常在 run() 内部就地咽下——两种做法都又丑又易错。JDK 5 的 java.util.concurrent 给出了正解：Callable 补上返回值和异常，Future 补上“凭据式”的结果领取。

维度RunnableCallable<V>
方法签名void run()V call() throws Exception
返回值无，只能靠副作用返回 V
受检异常不能抛，必须内部消化可以直接抛 Exception
引入版本JDK 1.0JDK 5（并发包）
典型入口Thread、execute()ExecutorService.submit()
为什么是两个接口而不是给 Runnable 加返回值？ 因为 Runnable 早已被无数 API 依赖（Thread、事件回调、GUI……），改签名会破坏二进制兼容。新增 Callable 是最小侵入的演进——这也是 JDK 演化里反复出现的模式：不改老接口，加新接口 + 适配器（后面的 FutureTask 就是那个适配器）。

Future：一张“提货凭据”

当你 submit 一个 Callable，线程池立刻返回一个 Future——它不是结果本身，而是一张“凭据”：任务还在后台跑，凭据先给你，等你需要结果时拿它去“提货”。这套模型让提交和取结果在时间上解耦，主线程提交后可以继续干别的，真正需要时再 get()。

submit(task) ⇒ Future 凭据 ⇒ get() 兑现「结果 | 异常」

Future 接口只有 5 个方法，但每一个都埋着坑：

方法作用关键点 / 陷阱
get()阻塞直到任务完成，返回结果无超时，任务卡住则永久阻塞
get(t, unit)限时取结果超时抛 TimeoutException，但不取消任务
cancel(mayInterrupt)尝试取消任务协作式；任务已完成返回 false
isDone()是否已结束正常/异常/取消都算 done
isCancelled()是否被取消与 isDone 区分“怎么结束的”
FutureBasic.java提交 Callable、限时取结果、分类接异常
ExecutorService pool = Executors.newFixedThreadPool(2);

// Callable<V>：有返回值、可抛受检异常，正好补上 Runnable 的两处短板
Callable<Integer> task = () -> {
    int price = fetchPriceFromRemote();   // 可能抛 IOException（受检）
    return price * 100;                  // 有返回值
};

Future<Integer> future = pool.submit(task);  // 立即返回凭据，不阻塞

// …… 主线程可以继续做别的事 ……

try {
    Integer result = future.get(2, TimeUnit.SECONDS);  // 最多等 2s
    System.out.println("结果 = " + result);
} catch (ExecutionException e) {
    // 任务内部抛的异常被“包装”在这里，真正的异常要看 getCause()
    Throwable real = e.getCause();
    log.error("任务执行失败", real);
} catch (TimeoutException e) {
    future.cancel(true);   // 超时不会自动取消，必须手动打断
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复中断状态，别把它吞掉
}
头号陷阱：无参 get() 会永久阻塞。 只要任务不结束（死循环、下游服务挂起、网络无响应），get() 就会把调用线程一直挂在那里。生产代码几乎永远应该用带超时的 get(timeout, unit)——这和上一节同步工具里“优先用带超时重载”的原则完全一致。更隐蔽的是：如果这个阻塞发生在 Web 容器的请求线程里，会把整个线程池的工作线程逐个耗光，最终拖垮服务。

异常去哪了：ExecutionException 的包装

这是 Future 最反直觉、也是面试最爱问的一点：任务内部抛出的任何异常，都不会在任务那一刻“炸”给你，而是被 Future 捕获、存起来，等你调用 get() 时，再包装成 ExecutionException 重新抛出。

为什么要包装？因为任务运行在另一个线程里，它抛的异常无法穿越线程边界直接落到你的 catch。Future 的做法是：把异常当成一种“结果”存下来，在 get() 这个跨线程的“交汇点”统一抛出。所以真正的业务异常永远藏在 e.getCause() 里——直接打印 ExecutionException 你只会看到一层没用的外壳。

任务线程 调用 get() 的线程 ────────── ───────────────── call() 抛 IOException │ 异常被 FutureTask 捕获 ▼ 存进 outcome 字段（不外抛） [ state: EXCEPTIONAL ] ── happens-before ──▶ get() │ 发现 state==EXCEPTIONAL ▼ report() 重新包装 throw new ExecutionException(ioe) │ e.getCause() ← 真正的 IOException
四种 get() 可能抛的异常，别搞混：
异常何时抛处理要点
ExecutionException任务内部抛了异常受检；getCause() 取真身
CancellationException任务被取消后又去 get运行时异常，无需强制捕获
InterruptedException等待中的当前线程被中断恢复中断位或向上抛
TimeoutException限时 get 到点还没完成任务仍在跑，需自行 cancel

execute 还是 submit：异常的两种命运

同一个会抛异常的任务，用 execute() 和 submit() 提交，异常的下场完全不同，这是无数“任务悄悄失败、日志里啥都没有”的根因。

Bad · submit 后从不 get
// 用 submit 提交
pool.submit(() -> {
    doWork();      // 若这里抛 RuntimeException
});
// 异常被吞进 Future 对象，
// 你从不 get()，就永远看不到，
// 任务“静默失败”，无任何日志
// → 排查时一脸懵
Good · 消费异常 或 用 execute
// 方案A：既然 submit，就消费 Future
Future<?> f = pool.submit(() -> doWork());
try { f.get(); }
catch (ExecutionException e) {
    handle(e.getCause());
}

// 方案B：不要结果就用 execute()，
// 未捕获异常会走线程的
// UncaughtExceptionHandler，能打日志
记牢这条差异： execute(Runnable) 里抛出的未捕获异常，会传播到该线程的 UncaughtExceptionHandler（默认打印到 stderr）；而 submit(...) 会把异常吞进 Future，只有你调 get() 时才现身。也就是说：submit 提交却不 get，等于给异常盖了块布。 原理很简单——submit 内部把你的任务包成 FutureTask，FutureTask.run() 用 try/catch 兜住了所有 Throwable 并存进 outcome，异常根本没机会冒泡到线程顶层。

FutureTask：Runnable 与 Future 的“转接头”

前面反复提到的 FutureTask 到底是什么？它是 JDK 里连接两个世界的适配器——同时实现了 Runnable（能被线程/线程池执行）和 Future（能被取结果）。ExecutorService.submit(callable) 内部做的，正是 new FutureTask<>(callable) 然后 execute 它，再把这个 FutureTask 当 Future 返回给你。

FutureTaskBridge.java一个对象，两种身份
// FutureTask 同时是 Runnable 和 Future，是二者之间的“转接头”
FutureTask<String> ft = new FutureTask<>(() -> {
    Thread.sleep(500);
    return "done";
});

new Thread(ft).start();   // 当 Runnable 用：交给线程去跑
String r = ft.get();        // 当 Future 用：阻塞取结果

它的内部是一个用 CAS 驱动的状态机，一个 volatile int state 记录任务生命周期。理解这张图，就理解了“为什么 get 有时抛 ExecutionException、有时抛 CancellationException”：

FutureTask 状态机（volatile int state，靠 CAS 流转，只进不退） run() 正常结束 NEW ─────────────────▶ COMPLETING ──▶ NORMAL get() 返回结果 │ run() 抛异常 ├─────────────────▶ COMPLETING ──▶ EXCEPTIONAL get() 抛 ExecutionException │ cancel(false) ├──────────────────────────────▶ CANCELLED get() 抛 CancellationException │ cancel(true) └──────────▶ INTERRUPTING ──▶ INTERRUPTED （中断运行中的线程后落定） get() 时若 state <= COMPLETING（还没出结果）： → LockSupport.park() 把当前线程挂起，塞进一个等待者链表（Treiber 栈） run()/cancel() 落定后： → finishCompletion() 逐个 unpark 所有等待者，大家各自读 outcome
串联内存模型： 你可能担心——任务线程写的结果，get 线程读得到吗？会不会有可见性问题？JDK 保证了：Callable 中的所有动作，happens-before 另一个线程从对应 Future 的 get() 成功返回。实现上，state 是 volatile，run() 结束前对 state 的写、get() 读到终态 state 之间构成 happens-before（这就是我们在《volatile 与 happens-before》里讲的规则），因此 outcome 里的结果对读取方一定可见，无需额外加锁。

取消：一场需要双方配合的“喊停”

cancel(mayInterruptIfRunning) 的名字很有迷惑性——它不是“强制杀死任务”。Java 里没有安全地强杀线程的手段（Thread.stop() 早已废弃，因为会让对象处于不一致的中间态）。取消是协作式的：

  • 任务还没开始跑：直接把它从队列里摘掉，标记 CANCELLED，稳妥。
  • 任务正在跑 + mayInterrupt=true：给执行线程发一个中断信号，但任务必须自己响应中断（检查 Thread.interrupted() 或让阻塞方法抛 InterruptedException）才会停。
  • 任务已经完成：cancel 返回 false，什么也没发生。
Bad · 不响应中断，cancel 形同虚设
Callable<Void> task = () -> {
    while (true) {      // 死循环
        crunch();       // 只顾算，
    }                   // 从不看中断标志
};
// future.cancel(true) 发了中断，
// 但任务根本不理会 → 永远停不下来
Good · 主动检查中断，及时退出
Callable<Void> task = () -> {
    while (!Thread.currentThread()
                  .isInterrupted()) {
        crunch();       // 每轮检查中断标志
    }
    return null;        // 被 cancel(true) 时干净退出
};
超时 ≠ 取消。 get(t, unit) 超时抛出 TimeoutException 后，后台任务仍在继续跑、继续占资源——TimeoutException 只是告诉“我不等了”，并不会替你喊停。想真正止损，必须在 catch (TimeoutException) 里显式调用 future.cancel(true)。忘了这一步，就是典型的“慢任务越堆越多把线程池撑爆”。

Future 的天花板，与 CompletableFuture 的接棒

Future 解决了“提交—取结果—接异常”，但它的短板同样明显，这也是为什么 JDK 8 要推出 CompletableFuture：

诉求Future（JDK 5）CompletableFuture（JDK 8）
取结果只能阻塞 get() 或轮询 isDone()注册回调，完成时自动触发
任务编排无，多任务只能手动串thenApply/thenCompose 链式组合
合并多个手写等待逻辑allOf/anyOf 一行搞定
异常处理get 时集中 try/catchexceptionally/handle 声明式
手动完成不支持complete(v) 可外部赋值
生活类比： Future 像取快递的取件码——你得亲自跑去驿站、排队等它到货（阻塞 get）。CompletableFuture 像开了到货短信通知：货一到就主动推给你，你还能预约“到了先送去代收点、再转寄给朋友”（回调编排）。同样是等结果，一个是“拉”（pull/poll），一个是“推”（push/callback）。
联系 Spring： Spring 的 @Async 方法既可以返回 Future（老式，用 AsyncResult 包装），也可以返回 CompletableFuture（推荐）。返回 Future 时同样受限于“只能阻塞取结果”；而返回 CompletableFuture 才能配合 Spring MVC 的异步请求、WebFlux 做真正的非阻塞编排。选型时优先 CompletableFuture——这正是下一节的主角。
核心收束： Callable 补齐 Runnable 缺失的“返回值 + 受检异常”；Future 是提交后拿到的“提货凭据”，把提交与取结果解耦。牢记四条铁律：① 生产代码用带超时的 get，无参 get 会永久阻塞；② 任务异常被包装成 ExecutionException，真身在 getCause()；③ submit 会吞异常、execute 会走 UncaughtExceptionHandler，submit 后务必 get 消费；④ cancel 是协作式的、且超时不等于取消，任务必须响应中断才能真正停下。Future 只会“拉”，要“推”和编排，请看 CompletableFuture。
