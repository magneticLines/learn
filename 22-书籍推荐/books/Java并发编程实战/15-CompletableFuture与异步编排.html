CompletableFuture 与异步编排

把“等结果”变成“接着做什么”——用声明式的流水线取代阻塞的 get()。

Future 到底差在哪

上一节我们用 Future 拿到了异步任务的“提货单”。但真正用起来，你会发现 Future 有三个硬伤，它更像一个“只读的结果占位符”，而不是一个能编排的工具：

  • 只能阻塞取结果。future.get() 会把当前线程死死钉住，异步执行的意义大打折扣——你无非是把“同步等”换了个地方等。
  • 无法注册回调。你不能说“算完后自动接着做 X”，只能自己轮询 isDone() 或阻塞 get()。
  • 无法组合。两个 Future 都完成后合并结果、一个的输出喂给下一个……全得手写线程协调，代码迅速滑向回调地狱。

Java 8 的 CompletableFuture 就是为解决这三点而生。名字里的 Completable（可完成）是关键：它的结果不仅能被后台任务填入，也能被外部手动 complete(value) 填入——这让它既能当异步任务的句柄，也能当跨线程传递结果的“信箱”。

类型血统： CompletableFuture<T> 同时实现了 Future<T>（保留 get() 兼容旧代码）和 CompletionStage<T>（提供 thenApply/thenCompose/exceptionally 等 50 多个编排方法）。可以说：Future 负责“拿结果”，CompletionStage 负责“编排下一步”。这与前面学的 Optional、Stream 是同一套“声明式链式”思想——把“对值做什么”写成一条流水线。
CompletableFuture = Future(可取结果) + CompletionStage(可编排下一步)

创建异步任务：谁在哪个线程跑

两个工厂方法起步：supplyAsync(Supplier) 用于“有返回值”的任务，runAsync(Runnable) 用于“无返回值”的任务。不显式传线程池时，任务默认跑在 ForkJoinPool.commonPool() 上。

Create.java两种起步方式与默认线程池
// 有返回值：Supplier 的返回值成为 future 的结果
CompletableFuture<String> cf1 = CompletableFuture.supplyAsync(() -> queryUser(42));

// 无返回值：CompletableFuture<Void>
CompletableFuture<Void> cf2 = CompletableFuture.runAsync(() -> warmCache());

// 生产环境强烈建议：显式传自己的线程池，别用 commonPool
ExecutorService io = Executors.newFixedThreadPool(16);
CompletableFuture<String> cf3 = CompletableFuture.supplyAsync(() -> queryUser(42), io);

// “手动完成”：结果不来自后台任务，而是外部塞进去（当信箱用）
CompletableFuture<String> box = new CompletableFuture<>();
box.complete("done");   // 任意线程可调用，等待者立即被唤醒
头号生产事故：别把阻塞任务丢进 commonPool。 ForkJoinPool.commonPool() 是全 JVM 共享的，默认并行度只有 CPU核数 - 1。如果你在里面跑数据库查询、HTTP 调用这类阻塞 IO，几个慢请求就能占满它，导致整个应用（包括并行 Stream、其他 CompletableFuture）一起卡死。规则：CPU 密集可用默认池；IO 密集必须传专用线程池，并为它设置合理大小与有界队列。

链式编排：thenApply / thenAccept / thenRun

拿到一个 CompletableFuture 后，用 then* 方法声明“完成后接着做什么”。三个基础变体的区别只在于——是否消费上一步结果、是否产出新结果：

方法入参返回类比 Stream
thenApply(fn)接收上一步结果 T新结果 Umap
thenAccept(c)接收结果 TVoid（只消费）forEach
thenRun(r)不接收结果Void收尾动作

每个方法都有 xxxAsync 变体，差别在“下一步在哪个线程跑”：

  • thenApply：可能由完成上一步的那个线程直接继续跑；若上一步在调用时已完成，甚至由调用线程同步跑。适合极轻量的转换。
  • thenApplyAsync：把下一步重新提交到线程池（默认 commonPool，或你传的 executor）异步执行。适合较重、或不想占用上游线程的操作。
为什么要区分同步/异步续接？ 因为“调度到另一个线程”本身有上下文切换成本。一次简单的 x -> x + 1 转换，直接在原线程跑（thenApply）比再排一次队（thenApplyAsync）划算得多。反过来，如果续接动作会阻塞或耗时，就该用 Async 把它挪走，别拖住负责回调的那个线程。这正是 CLAUDE.md 强调的“方案取舍”——没有银弹，看动作的轻重。

thenApply vs thenCompose：别嵌套 Future

当续接函数本身又返回一个 CompletableFuture（比如第一步查到用户 id，第二步再异步查该用户的订单），用 thenApply 会得到套娃类型 CompletableFuture<CompletableFuture<Order>>。thenCompose 会把它“拍平”，正是 Stream/Optional 里 flatMap 之于 map 的关系。

Bad · thenApply 套娃
CompletableFuture<
    CompletableFuture<Order>> nested =
  findUserId(name)              // CF<Long>
    .thenApply(id ->
        loadOrderAsync(id));    // 又返回 CF<Order>
// 想拿 Order 还得再 get 一层，噩梦
Order o = nested.join().join();
Good · thenCompose 拍平
CompletableFuture<Order> flat =
  findUserId(name)              // CF<Long>
    .thenCompose(id ->
        loadOrderAsync(id));    // 拍平为 CF<Order>
// 一层，直接拿结果
Order o = flat.join();
一句话记忆： 续接函数返回普通值用 thenApply（map）；返回另一个 CompletableFuture 用 thenCompose（flatMap）。判据永远是“函数的返回类型是不是又包了一层 Future”。

组合多个任务：combine / allOf / anyOf

真实业务常常是“并行发几个请求，等它们凑齐再汇总”。CompletableFuture 把这类扇出/扇入（fan-out / fan-in）也变成了声明式：

扇出并行 + 扇入汇总（allOf） ┌─ supplyAsync(查库存) ─┐ │ │ ├─ supplyAsync(查价格) ─┤──▶ allOf(...).thenRun(汇总) │ │ └─ supplyAsync(查评价) ─┘ 三个请求并行发出 三者全部完成后才触发 thenCombine（两两合并） anyOf（谁先好用谁） A ─┐ A ─┐ ├─(a,b)->merge──▶ 结果 ├──▶ 取最先完成的那个 B ─┘ B ─┘
Aggregate.java并行调用 3 个服务并汇总
CompletableFuture<Integer> stock = CompletableFuture.supplyAsync(() -> queryStock(id), io);
CompletableFuture<Double>  price = CompletableFuture.supplyAsync(() -> queryPrice(id), io);
CompletableFuture<Double>  score = CompletableFuture.supplyAsync(() -> queryScore(id), io);

// thenCombine：两个独立结果都好后合并
CompletableFuture<String> card = stock.thenCombine(price,
        (s, p) -> "库存" + s + " 单价" + p);

// allOf：等全部完成（返回 Void，结果得各自 join 取回）
CompletableFuture.allOf(stock, price, score)
    .thenRun(() -> {
        // 到这里三个都已完成，join 不会再阻塞
        render(stock.join(), price.join(), score.join());
    });
allOf 的坑： allOf(...) 的返回类型是 CompletableFuture<Void>——它不帮你收集结果，只告诉你“都好了”。取值仍要对每个原始 future 调 join()（此时不会阻塞）。另外 anyOf 返回 CompletableFuture<Object>（因入参类型可能不同），拿到值往往得强转，注意别踩类型陷阱。

异常处理：让流水线不“断电”

同步代码靠 try/catch，异步流水线里异常会沿链向下游传播：某一步抛异常，它后面的 thenApply 等正常续接会被整体跳过，直到遇到一个专门处理异常的节点。三个处理方法要分清：

方法正常时异常时能否改结果
exceptionally(fn)透传原值，不触发用 fallback 值兜底只在异常时改
handle(bi)(结果, null) 都进来(null, 异常) 都进来两种情况都可改
whenComplete(bi)(结果, null)(null, 异常)否，只“旁观”不改结果
ErrorHandling.java兜底、观测、超时
CompletableFuture.supplyAsync(() -> callFlakyService(), io)
    .thenApply(Response::body)                 // 抛异常则跳过这步
    .exceptionally(ex -> {                   // 只在链上出异常时进入
        log.warn("降级返回缓存", ex);
        return fromCache();                     // 提供兜底值，链恢复正常
    })
    .whenComplete((v, ex) -> recordMetric(v, ex))  // 旁观：埋点，不改结果
    .orTimeout(2, TimeUnit.SECONDS);       // Java 9+：超时则以 TimeoutException 完成
异常包装陷阱： 链上抛出的异常在下游会被包成 CompletionException，通过 join() 抛出的也是它；而 get() 抛的是 ExecutionException——两者都套了一层，真正的原因要 getCause() 才拿得到。别直接对着 wrapper 的类型做 instanceof 判断业务异常。

更隐蔽的： 如果你只创建了 future 却从不 join / 不注册 exceptionally，任务里抛的异常会被悄悄吞掉，既不打印也不上报——这是异步代码最常见的“幽灵 bug”。

反例 → 正例：回调地狱 vs 声明式流水线

“查用户 → 用用户 id 查订单 → 汇总发货”这条链，用裸线程 + Future 会写成层层嵌套；用 CompletableFuture 则是一条平坦的流水线：

Bad · 阻塞 + 嵌套
Future<User> uf =
    pool.submit(() -> findUser(name));
User u = uf.get();          // 阻塞①
Future<Order> of =
    pool.submit(() -> loadOrder(u.id()));
Order o = of.get();         // 阻塞②
Future<Ship> sf =
    pool.submit(() -> ship(o));
Ship s = sf.get();          // 阻塞③
// 三次阻塞、线程全程被占、异常
// 靠散落的 try/catch 兜
Good · 非阻塞 + 链式
CompletableFuture
  .supplyAsync(() -> findUser(name), io)
  .thenCompose(u ->
      supplyAsync(() -> loadOrder(u.id()), io))
  .thenCompose(o ->
      supplyAsync(() -> ship(o), io))
  .exceptionally(ex -> fallbackShip(ex))
  .thenAccept(this::notifyUser);
// 全程不阻塞调用线程，异常
// 一处统一兜底，可读性接近同步

Future vs CompletableFuture 全景对比

能力Future (JDK 5)CompletableFuture (JDK 8)
取结果只能阻塞 get()阻塞 get/join 或注册回调
回调/续接不支持thenApply/thenAccept/...
任务组合手写协调thenCompose/combine/allOf/anyOf
异常处理get 抛 ExecutionExceptionexceptionally/handle/whenComplete
手动完成不支持complete/completeExceptionally
超时仅 get(timeout)（只是等超时）orTimeout/completeOnTimeout(9+)
取消cancel(true) 可中断任务cancel 不保证中断底层线程
联系生态： Spring 的 @Async 方法可直接返回 CompletableFuture，配合自定义 ThreadPoolTaskExecutor 就是生产级异步；WebFlux 的 Mono/Flux 是它的“流式加强版”（支持背压与多值）。而 Java 21 的虚拟线程又带来新思路：IO 阻塞变得极廉价，很多原本靠 CompletableFuture 编排来“躲避阻塞”的场景，如今可以用直白的同步代码 + 虚拟线程写出，可读性更好。选型时记住：CompletableFuture 擅长“组合与编排多个异步阶段”，虚拟线程擅长“让海量阻塞调用不吃线程”，两者互补而非互斥。
核心收束： CompletableFuture = Future（可取结果）+ CompletionStage（可编排下一步），把“阻塞等结果”翻转成“声明接着做什么”。记牢四条：① 创建用 supplyAsync/runAsync，IO 任务务必传专用线程池，别污染 commonPool；② 续接返回普通值用 thenApply、返回 Future 用 thenCompose（map vs flatMap）；③ 扇入用 allOf（记得它返回 Void，结果各自 join）、两两合并用 thenCombine；④ 异常沿链传播，用 exceptionally/handle 统一兜底，否则未消费的异常会被静默吞没——注意 join/get 抛的是被包装过的异常。
