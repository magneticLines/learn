Atomic 类与 CAS 原理

不用锁也能安全地"读—改—写"——CAS 把并发的胜负交给一条 CPU 指令，Atomic 类则把它包成你顺手就能用的原子变量。

一、锁太重了：我们到底想要什么

上一节的 AQS 让我们看清了锁的本质：抢不到就挂起（阻塞），醒过来还要重新调度。对一个简单到不能再简单的 count++ 而言，用 synchronized 保护它，就像为了给一张纸盖个章，先把整个房间锁上、把别人赶到门外排队睡觉——安全，但代价高得离谱：上下文切换、线程挂起唤醒、内核态与用户态往返。

我们真正想要的是：让 count++ 这个复合操作（读→改→写三步）变成一个不可分割的原子动作，同时不阻塞任何线程。这就是 无锁（lock-free） 的诉求，而实现它的地基，是一条硬件提供的原子指令——CAS。

回顾第 2 章的痛点：count++ 之所以线程不安全，是因为它是"读 count → 加 1 → 写回"三步，两个线程可能都读到旧值 10，各自 +1 写回 11，丢了一次更新。volatile（第 4 章）能保证可见性却挡不住这种复合竞态。锁能解决，但太重。CAS 给了第三条路：把"检查旧值 + 写入新值"合并成一步硬件原子操作。

二、CAS 是什么：一条"比较并交换"指令

CAS 全称 Compare-And-Swap（比较并交换）。它接受三个参数，语义可以用一句话概括：

CAS(V, A, B)  =  if (V == A) V = B, return true  else  return false

其中 V 是内存里的变量（value），A 是你预期它现在应该是的旧值（expected），B 是你想写入的新值（new）。关键在于："比较 V 是否等于 A"和"把 B 写进 V"这两个动作，由 CPU 保证是一个原子步骤，中间不可能被其他线程插入。如果比较失败（说明有人抢先改了 V），CAS 什么都不做，返回 false，由调用方决定"要不要再试一次"。

生活类比——抢共享文档的"版本号"：你打开一份在线文档，看到它是"第 10 版"（预期值 A=10）。你在本地改好，点保存时系统做一次 CAS："服务器上现在还是第 10 版吗？是→存成第 11 版；不是→说明别人已经改过，拒绝你的保存，让你刷新后重来"。这正是乐观锁的思路：不提前上锁，先干活，提交时才检查有没有冲突。数据库的乐观锁 UPDATE ... WHERE version = ? 就是同一个套路。

三、自旋：CAS 失败了怎么办

CAS 只做"一次尝试"。要把它变成"一定成功的原子自增"，需要在外面套一个循环——失败就重读最新值、重算、再 CAS，直到成功。这个"失败重试"的循环就叫 自旋（spin）。用伪代码表达一次 getAndIncrement：

┌─────────────────────────────────────┐ │ do { │ 读旧值 │ int old = value; (volatile 读)│ │ int next = old + 1; │ │ } while ( !CAS(value, old, next) ) │◄── 失败：说明别人抢先改了 │ 比较并交换 ──── 成功 ──► │ old 已过期，回到循环顶重读 └─────────────────────────────────────┘ 线程A: old=10 ─CAS(10→11)─► 成功✔ 线程B: old=10 ─CAS(10→11)─► V已经是11≠10，失败✗ ─► 重读 old=11 ─CAS(11→12)─► 成功✔ （B 没有被挂起，只是多转了一圈）

对比锁：锁竞争失败会让线程挂起（park），交出 CPU；而 CAS 失败只是空转一圈重试，线程始终持有 CPU、从不进入内核阻塞。在竞争不激烈、临界区极短时，这种"忙等一小会儿"远比"睡眠再唤醒"划算——这就是无锁算法的性能来源。

四、Atomic 类：把自旋 CAS 包成顺手的工具

你不用自己写自旋循环。java.util.concurrent.atomic 包把这套逻辑封装成了一族开箱即用的原子变量。按用途分四类：

类别代表类用途
基本类型AtomicInteger AtomicLong AtomicBoolean原子计数器、序号生成、开关标志
引用类型AtomicReference<T>原子地替换一个对象引用（如无锁栈的栈顶）
带版本/标记AtomicStampedReference AtomicMarkableReference解决 ABA 问题（见第六节）
字段更新器AtomicIntegerFieldUpdater 等对已有类的 volatile 字段做原子更新，省一层包装对象

最常见的 AtomicInteger 用法，就是把那个会翻车的 count++ 换成一个方法调用：

✗ synchronized 保护自增
class Counter {
  private int count = 0;
  // 每次自增都要抢锁、可能挂起
  public synchronized void inc() {
    count++;
  }
  public synchronized int get() {
    return count;
  }
}
// 正确，但高频调用下锁竞争严重
✓ AtomicInteger 无锁自增
class Counter {
  private final AtomicInteger count
      = new AtomicInteger(0);
  // 无锁，底层自旋 CAS
  public void inc() {
    count.incrementAndGet();
  }
  public int get() {
    return count.get();
  }
}
// 同样正确，无阻塞、无上下文切换

五、扒开 JDK 源码：CAS 到底怎么落到硬件

AtomicInteger.getAndIncrement() 顺着调下去，会走到 Unsafe.getAndAddInt——JDK 里那个"绕过 Java 安全检查直接操作内存"的后门类。这段循环正是我们前面画的自旋 CAS：

sun.misc.Unsafe（JDK 8 源码节选）自旋 + CAS
// AtomicInteger 内部
public final int getAndIncrement() {
    return unsafe.getAndAddInt(this, valueOffset, 1);
}

// Unsafe.getAndAddInt：一直重试到成功
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);        // 每轮都重新读最新值（volatile 读）
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS 失败就再来
    return v;                                // 返回旧值
}

// 这是个 native 方法，最终由 JVM 编译成一条 CPU 原子指令
public final native boolean compareAndSwapInt(
        Object o, long offset, int expected, int x);
联系操作系统 / 硬件：compareAndSwapInt 是 native 的，HotSpot 会把它编译成一条带 lock 前缀的 cmpxchg（compare-exchange）指令（x86 架构）。lock 前缀会锁定缓存行（现代 CPU 用缓存锁定而非古老的锁总线），并通过第 4 章讲过的 MESI 缓存一致性协议让其他核的副本失效——所以 CAS 天然带有 volatile 级别的可见性保证。ARM 上则用 LDREX/STREX（load/store exclusive）指令对来实现同样的语义。一句话：原子性不是 Java 变的魔法，是 CPU 指令集直接提供的能力。
JDK 9+ 的演进：Unsafe 是不对外的内部 API。从 Java 9 起官方推荐用 VarHandle（java.lang.invoke）来做 CAS 与各种内存访问模式，它是 Unsafe 的合法、安全、跨平台替代品。AtomicInteger 等类的内部实现也逐步迁移到了 VarHandle。你自己写并发代码时，需要底层 CAS 就用 VarHandle，别再直接摸 Unsafe。

六、CAS 最大的坑：ABA 问题

CAS 只关心"现在的值等不等于我预期的旧值"。但"值相等"不代表"这中间没被人动过"。设想一个变量从 A 被改成 B、又被改回 A：你手里的预期值还是 A，CAS 一看"当前是 A，符合预期"，就放心地成功了——可它完全没察觉这中间发生过两次修改。这就是臭名昭著的 ABA 问题。

时间轴 ─────────────────────────────────────────► 线程1: 读到 stack.top = A ......(被抢占，挂起一会儿)...... CAS(top, A→X) 成功✔ 但栈已被破坏 线程2: pop A → pop B → push A (top 又变回 A，但 A.next 已不是当初那个) 线程1 的 CAS 看到 top 仍是 A，误以为"没人动过"， 结果把一个已经过时的结构接了回去 ── 经典的无锁栈丢节点 / 悬挂指针。
陷阱与面试高频点：ABA 何时"真的有害"？对 AtomicInteger 这类只看最终数值的计数器，ABA 通常无害——值是几就是几，中间怎么变的不影响结果。但对引用类型（如无锁栈/队列的节点指针、对象池的复用对象），A 虽然"看起来是同一个引用"，其内部状态或它在数据结构中的语义可能已经天翻地覆，此时 ABA 会导致数据结构损坏。别一听 ABA 就恐慌，也别以为它永远无所谓——关键看你 CAS 的对象"值相等"是否真的等价于"状态没变"。

解法很直观：给值加一个只增不减的版本号 / 时间戳，CAS 时连版本一起比较。AtomicStampedReference 就是干这个的——它把"引用 + int 戳记"打包，只有引用和戳记都匹配才交换成功。这样 A→B→A 的过程会把戳记从 1 推到 3，预期戳记 1 不再匹配，CAS 失败，ABA 被识破。

AtomicStampedReference 破解 ABA引用 + 版本戳一起 CAS
AtomicStampedReference<String> ref =
        new AtomicStampedReference<>("A", 0);   // 初始值 A，戳记 0

int stamp = ref.getStamp();                 // 先记下当前戳记 = 0
String cur = ref.getReference();           // 当前引用 = "A"

// 期间别的线程做了 A→B→A，戳记已被推到 2

boolean ok = ref.compareAndSet(
        cur,          // 预期引用 "A" —— 值确实还是 A
        "X",          // 新引用
        stamp,        // 预期戳记 0
        stamp + 1);   // 新戳记
// ok == false！因为当前戳记是 2，不等于预期的 0 —— ABA 被识别

七、CAS 的另外两个短板

ABA 之外，CAS 还有两个绕不开的代价，正好对应它"无锁"的另一面：

短板一：高竞争下自旋空转烧 CPU。CAS 失败重试是靠"占着 CPU 转圈"实现的。当竞争激烈、大量线程同时抢一个变量时，绝大多数 CAS 都失败，线程们疯狂空转却没几个能真正前进，CPU 利用率飙高而吞吐量反而下降——这时候"忙等"就不再比"挂起"划算了。短板二：只能保证一个变量的原子性。CAS 天生只针对单个内存位置。要原子地更新多个变量（维持一组不变式），CAS 无能为力，只能退回到锁，或把多个变量封装进一个对象再用 AtomicReference 整体替换。

针对"高竞争计数"这个最典型的痛点，JDK 8 给出了 LongAdder / DoubleAdder。它的思路是分而治之：不再让所有线程抢同一个 value，而是内部维护一个 Cell[] 数组，不同线程散列到不同的 Cell 上各自累加，读取时再把所有 Cell 求和。竞争被分散到多个热点，CAS 冲突大幅下降。

维度AtomicLongLongAdder
核心结构单个 volatile longbase + Cell[] 分段
低并发快，且省内存没必要，略费内存
高并发写CAS 冲突剧烈，慢冲突分散，快得多
读取实时性get() 精确即时sum() 是弱一致的近似汇总
适用需要精确读、或写不密集只增不太读的高频计数（QPS、监控埋点）
选型口诀：写少或需要精确当前值 → AtomicLong；写多读少、只要最终统计（如接口调用次数、限流计数）→ LongAdder。Spring Boot Actuator / Micrometer 的计数指标底层用的正是 LongAdder，就是看中它在高并发埋点下的吞吐优势。

八、CAS vs 锁：一张表理清取舍

维度CAS / 无锁synchronized / Lock
阻塞否，线程不挂起（自旋）是，抢不到就 park
思路乐观锁：先干再验冲突悲观锁：先独占再干
粒度单个变量任意大小的临界区、多变量
低竞争性能极好（无切换开销）一般（有锁开销）
高竞争性能可能退化（自旋空转）相对稳定（挂起让出 CPU）
典型坑ABA、自旋消耗死锁、上下文切换
代表场景计数器、标志、无锁数据结构复合业务逻辑、多变量一致性
建立联系——CAS 是整个 JUC 的基石：你上一节学的 AQS，其 volatile int state 的获取/释放全靠 CAS（compareAndSetState）；ConcurrentHashMap 在桶为空时用 CAS 无锁地放入首节点（casTabAt），只在冲突时才退化到 synchronized；ReentrantLock、Semaphore、线程池的 ctl 状态字……几乎所有 JUC 组件都把 CAS 当作"不阻塞快路径"。看懂 CAS，你就拿到了理解整个并发包的钥匙。
一句话收束：CAS 是一条 "比较预期旧值，相符才写入"的 CPU 原子指令（x86 上是 lock cmpxchg），它用乐观锁 + 自旋重试替代了锁的悲观阻塞，让"读—改—写"复合操作在不挂起线程的前提下保持原子。Atomic 类把这套自旋 CAS 封装成顺手的原子变量。记牢三条边界：①ABA——值相等不等于没被改过，引用类型用 AtomicStampedReference 加版本戳；②高竞争下自旋空转——热点计数改用分段的 LongAdder；③只能护一个变量——多变量一致性仍需上锁。CAS 不是锁的全面替代品，而是"临界区极短、竞争不激烈"时更聪明的那把刀；它更是 AQS、ConcurrentHashMap 乃至整个 JUC 的共同地基。下一篇《Java 内存模型（JMM）》，我们将从底层回答一个更根本的问题：这些可见性与有序性的保证，究竟由谁、按什么规则兑现。
