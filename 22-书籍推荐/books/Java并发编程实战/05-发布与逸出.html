发布与逸出

你以为对象只在类内部悄悄用着，可它早就被一条你没注意到的暗道递到了外面——最阴险的一条，是构造函数还没跑完，this 就已经溜了出去。

一、先把"发布"和"逸出"两个词分清

上一章我们反复强调：线程安全的病根是共享的可变状态。可一个对象是怎么从"只有我知道"变成"大家都能碰"的？答案就是发布（publish）。而逸出（escape），是发布出了岔子——本不该被外界看到的对象，被意外发布了出去。

两个定义（JCIP 第 3.2 节）：
发布：使对象能够在当前作用域之外的代码中使用。比如把引用存进 public 字段、从非私有方法 return 出去、或作为参数传给别的方法。
逸出：当某个不应该被发布的对象被发布时，就称它逸出了。逸出 = 发布，只是这次发布不是你的本意。
逸出 = 发布 − 你的本意

为什么要这么较真地区分？因为发布本身是中性的——你需要发布对象，程序才能协作。危险的从来不是"发布"这个动作，而是"发布了不该发布的东西"，或者"在不该发布的时机发布"。理解了这一点，你就抓住了本节所有陷阱的共同结构。

为什么它属于"共享对象"这一章？一个对象一旦逸出，就等于凭空多了一个"共享点"：外部代码持有了它的引用，可能在任意线程里读写它。你精心设计的封装、你对"这个字段只有我改"的假设，全部作废。逸出的本质，是不受控制地制造了共享可变状态——这正是并发 bug 的温床。

二、逸出的几条"暗道"

对象逸出，说到底就那么几种途径。把它们列成一张地图，以后 code review 时就能条件反射地警觉：

一个对象"逸出"当前作用域，通常只有这几条暗道： ① 存入 public / public static 字段 → 任何代码都能拿到并修改 ② 从非私有方法 return 内部可变对象的引用 → 调用方能直接改你的"内脏" ③ 把内部对象作为参数传给外部(alien)方法 → 对方可能偷偷把它存起来 ④ 发布一个内部类实例 → 隐式捎带走了外围类的 this 一句话：只要外部代码能"摸到"这个引用，它就逸出了。

其中 ① 太直白（谁都知道别乱开 public 字段），真正常被忽视的是 ②③④。下面逐条拆。

三、暗道 ②：把内部可变状态直接 return 出去

这是最常见、也最容易被当成"正常写法"的逸出。你有一个私有数组，写了个 getter 返回它——看起来天经地义，实则把内部状态的可写引用拱手送人：

✗ 逸出：外人能改你的内脏
class UnsafeStates {
  private String[] states =
      { "AK", "AL", "AR" };

  // 直接把内部数组引用返回
  public String[] getStates() {
    return states;
  }
}

// 调用方:
var s = new UnsafeStates();
s.getStates()[0] = "XX";
// 内部 states[0] 也变成了 "XX"!
// 封装被彻底击穿
✓ 返回副本 / 不可变视图
class SafeStates {
  private final String[] states =
      { "AK", "AL", "AR" };

  // 返回防御性拷贝
  public String[] getStates() {
    return states.clone();
  }
  // 或返回只读视图:
  // List.of(states) / 不可变集合
}

// 外人改的是副本,
// 内部 states 岿然不动
陷阱一：Collections.unmodifiableList(list) 只是"只读视图"，不是拷贝。它包了一层禁止 add/remove 的壳，但底层还是同一个 list。如果你后续还持有并修改原 list，外部拿到的视图会跟着变——这依然是一种逸出。要彻底切断，用 List.copyOf(list)（JDK 10+，真拷贝且不可变）或先 new ArrayList<>(list) 再包一层。

四、暗道 ③④：alien method 与内部类的 this

暗道 ③ 是把内部对象传给一个你无法控制其行为的方法——书里叫它 alien method（外部方法）：可能是别的类的方法、可以被子类覆写的方法、或者一个回调接口。你不知道它会不会把这个引用偷偷存进某个静态集合、或另起线程去访问它。

暗道 ④ 更隐蔽，是本节的重头戏。任何内部类（包括匿名内部类、Lambda 捕获外围实例时）都隐式持有一个指向外围对象的 this 引用。你发布这个内部类实例，就等于顺手把外围的 this 也发布了出去——你自己往往完全没意识到。

内部类隐式持有外围 this反编译能看到 this$0 字段
class Outer {
    private int secret = 42;

    Runnable makeTask() {
        // 匿名内部类里能直接读 secret，靠的就是
        // 编译器塞进来的隐藏字段 Outer.this（this$0）
        return () -> System.out.println(secret);
    }
}
// 你发布的是 Runnable，实际上 Outer 实例也随之被引用住了
// —— 既可能导致内存泄漏，也可能导致 this 逸出

五、最致命的一种：this 引用在构造期逸出

把暗道 ④ 和"时机"结合起来，就是本节的核心线索：this 引用逸出（this escape）——在构造函数还没执行完的时候，就让 this 溜到了外部。经典翻车现场是"在构造器里注册监听器"：

✗ this 在构造期逸出
class ThisEscape {
  public ThisEscape(EventSource src) {
    // 匿名类隐式持有 ThisEscape.this
    src.registerListener(
      new EventListener() {
        public void onEvent(Event e) {
          doSomething(e);   // 用到外围 this
        }
      });
    // ↑ 构造器还没 return，
    //   this 已经被 src 发布出去了
  }
}
✓ 工厂方法：先造好再发布
class SafeListener {
  private final EventListener listener;

  private SafeListener() {           // 私有
    listener = new EventListener() {
      public void onEvent(Event e) {
        doSomething(e);
      }
    };
  }

  public static SafeListener
      newInstance(EventSource src) {
    SafeListener safe = new SafeListener();
    src.registerListener(safe.listener);
    return safe;   // 构造完成后才发布
  }
}

左边为什么危险？registerListener 一旦被调用，这个 listener（连带外围 this）就登记进了事件源。如果此时事件立刻到来（尤其在多线程环境下，事件线程和构造线程是两条线），onEvent 就会在 ThisEscape 的构造函数尚未跑完时被回调——它访问到的，是一个字段可能还没初始化、还是"半成品"的对象。

时间线：构造器还没跑完，别的线程已经能看到 this 构造线程: new ThisEscape(src) │ ├─ registerListener(listener) │ │ └── listener 隐式持有 ThisEscape.this ──┐ │ │ │ this 已逸出 │ ├─ (其余字段还没赋值) │ │ └─ 构造器尚未 return │ ▼ ▼ 事件线程: 事件到来 → 回调 onEvent() → 访问到"半成品" this (字段可能还是默认值 0 / null)
陷阱二：别在构造函数里 new Thread(...).start()。这是 this 逸出的另一张常见面孔。如果线程的 Runnable 是内部类，它就带走了外围 this；而 start() 会让新线程立刻可能开始运行，此时构造器还没结束。正确做法：构造器里只 创建线程（this.thread = new Thread(...)），把 start() 挪到一个单独的 start() / init() 方法，等对象完全构造好后再由调用方触发。

六、为什么"构造期逸出"格外要命：JMM 视角

普通的逸出顶多是破坏封装；而构造期逸出会撞上 Java 内存模型（JMM）最微妙的一条规则——final 字段的初始化安全保证，只在构造函数正常返回之后才生效。

联系上一节的 happens-before：JMM 承诺，只要对象被安全发布（且构造函数已返回），其他线程就能看到所有 final 字段被正确初始化后的值——这是"不可变对象天生线程安全"的底层依据。但这个承诺有个前提：this 不能在构造期间逸出。一旦逸出，其他线程可能在 final 字段写入之前就拿到了引用，读到的是"半初始化"状态。逸出直接把 JMM 给你的安全保证作废了。

换句话说，即便你把所有字段都声明成 final、把对象设计成"不可变"，只要 this 在构造期逸出，这个"不可变"的护身符也不管用。这也是为什么本书把"发布/逸出"安排在"不可变对象"和"安全发布"之前讲——它是后两者能成立的前提。

七、正确的发布：安全发布 vs 逸出

把"逸出"（错误的发布）和"安全发布"（正确的发布）并排对照，选型时就有了标尺：

方式是发布还是逸出安全吗说明
构造器里 registerListener / start()this 逸出✗ 危险构造未完成即暴露 this，读到半成品
非私有方法 return 内部可变对象逸出✗ 危险击穿封装，需返回拷贝或只读视图
公有 public 字段直接暴露引用逸出✗ 危险谁都能改，几乎没有合法用途
工厂方法：造好后才发布安全发布✓把 this 逸出改成"构造后发布"
用 volatile / final 字段保存引用安全发布✓借 happens-before 保证可见性
放进线程安全容器（如 ConcurrentHashMap）安全发布✓容器内部的同步顺带完成安全发布
联系 Spring：这也是为什么 Spring 推荐把"启动线程、订阅消息、注册回调"这类会发布 this 的逻辑放进 @PostConstruct 方法或 InitializingBean.afterPropertiesSet()，而不是塞进构造函数——框架保证这些回调在Bean 完全构造并注入完依赖之后才执行，天然规避了 this 逸出。你手写代码时的"工厂方法 + 独立 start()"，本质和 Spring 的生命周期钩子是同一套思路。
一句话收束：发布是让对象在当前作用域之外可见，逸出是这种发布违背了本意——本质都是"不受控地制造共享可变状态"。逸出主要走四条暗道：public 字段、return 内部可变对象、传给 alien 方法、发布内部类实例（隐式带走外围 this）。其中最致命的是 this 引用在构造期逸出：它让别的线程在构造函数返回前就摸到"半成品"对象，直接作废了 JMM 对 final 字段的初始化安全保证，连"不可变"都救不了。铁律两条——永远不要在构造函数里让 this 逃出去（注册监听、启动线程一律挪到工厂方法或独立的 start()），永远不要把内部可变状态的可写引用直接交出去（返回拷贝或不可变视图）。把这两条刻进肌肉记忆，下一节的"线程封闭"和后面的"安全发布四方式"才站得住脚。
