安全发布的四种方式

上一节说不可变对象是天然线程安全的；可对象自己再安全，如果"交到别人手里"这一步做错，另一个线程照样会拿到一个残缺不全的它。

一、先厘清：什么叫"发布"，什么叫"安全"发布

发布（publish）就是把一个对象的引用暴露到当前作用域之外，让别的代码（尤其是别的线程）能拿到它——赋值给公共字段、放进集合、作为返回值、传给回调，都是发布。前面《发布与逸出》讲的是"不该发布的别发布"（逸出）；这一节的问题反过来：该发布的，怎样才能发布得对。

直觉上你会觉得："不就是 obj = new Foo() 一行赋值吗，能有什么坑？"坑在于——发布一个引用，包含两件必须同时可见的事：

  • 引用本身：别的线程要能看到这个字段不再是 null，而是指向新对象。
  • 对象的内部状态：别的线程通过这个引用去读字段时，要能看到构造函数写完之后的值，而不是默认值（0/null）或半成品。

安全发布的定义就一句话：让发布对象的线程与读取对象的线程之间，对"引用 + 状态"建立起可见性保证——用内存模型的术语说，就是建立一条 happens-before 边。做不到这一点的，叫不安全发布（unsafe publication）。

二、反例：不安全发布会发生什么

《Java 并发编程实战》给了一个让人后背发凉的例子。Holder 看起来平平无奇，assertSanity() 里那句 n != n 按单线程直觉永远为假——可在不安全发布下，它真的可能抛异常。

不安全发布 · 经典翻车现场n != n 竟然可能成立
// 一个普通字段：既非 volatile、也非 final、读写都没有锁
public Holder holder;

public void initialize() {
    holder = new Holder(42);   // 发布：把引用写进共享字段
}

public class Holder {
    private int n;
    public Holder(int n) { this.n = n; }

    public void assertSanity() {
        if (n != n)                 // 单线程下"不可能"成立
            throw new AssertionError("看到了没构造完的对象！");
    }
}

不安全发布会引发两种故障，第二种尤其反直觉：

线程 A 线程 B ───────────────────────────────────────────────────── initialize(): ① 分配 Holder 内存 ② 构造函数写 n = 42 ③ holder = <地址> ──────► 读 holder 故障 1（引用不可见）： B 读到 holder 仍是 null → 空指针 故障 2（状态不可见 / 重排序）： B 看到 ③ 却看不到 ② → 读到 n == 0（默认值） assertSanity 两次读 n：先 0 后 42 → n != n 成立，抛错 💥
为什么"两次读同一个字段"会得到不同的值？因为 JMM 允许编译器/CPU 对没有 happens-before 约束的操作重排序，也不保证一个线程的写立刻对另一线程可见。第一次读 n 命中的可能是尚未刷新的默认值 0，第二次读又碰巧读到了刷新后的 42。这不是玄学，是缺少内存屏障的真实后果。记住：构造函数里的写，和"把引用赋给字段"这两步，在没有安全发布时对别的线程是无序的。

三、原理：安全发布 = 建立一条 happens-before 边

四种方式看起来花样不同，本质却是同一个：都在"发布线程写完对象状态"和"读取线程读到引用"之间，插入一条 happens-before 边。有了这条边，构造期的所有写入，对随后拿到引用的线程就一定可见——两种故障同时被堵死。

构造期的写入 happens-before 其他线程对该引用的读取
建立联系（对接内存模型）：happens-before 不是"时间上先发生"，而是"前者的结果对后者保证可见且有序"。安全发布无非是复用了 JMM 已有的几条 happens-before 规则——volatile 规则、锁规则、final 字段规则、类初始化规则——把它们借来给"发布"这件事兜底。所以学过 happens-before，再看这四种方式就是"同一招的四种打法"。

四、四种安全发布方式

《Java 并发编程实战》原文给出的四种手段，以及它们各自"借"的那条规则：

方式怎么做背后的 happens-before 规则
① 静态初始化器用 static 字段 / static{} 块初始化类初始化在 JVM 的初始化锁内完成，结果对所有线程可见
② volatile / 原子引用写入 volatile 字段或 AtomicReferencevolatile 写 happens-before 后续 volatile 读
③ final 字段存进"正确构造"对象的 final 字段final 字段的构造期 freeze 语义
④ 锁保护 / 线程安全容器用同一把锁守护读写；或放入并发/同步容器解锁 happens-before 后续对同一锁的加锁

① 静态初始化器

JVM 保证类的初始化由一把初始化锁串行完成，且完成后对所有线程可见。所以静态字段的初始化天然就是安全发布——这也是"按需加载的静态内部类单例（Holder Idiom）"能不加锁却线程安全的根本原因。

静态内部类单例 · Initialization-on-demand Holder无锁却安全
public class Singleton {
    private Singleton() {}

    // Holder 类只有在首次调用 getInstance 时才被加载
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;   // 类初始化保证安全发布
    }
}

② volatile 字段 / AtomicReference

把引用写进 volatile 字段：volatile 写会插入 StoreStore + StoreLoad 屏障，构造期的写不会被重排到发布之后，且写立即对读可见。这正是双重检查锁（DCL）里 instance 必须 volatile 的原因。

volatile 安全发布 · 也是 DCL 的地基只改一个关键字
private volatile Holder holder;   // ★ 关键：volatile

public void initialize() {
    holder = new Holder(42);   // 读到该引用的线程，一定看到 n==42
}

③ final 字段

JMM 对 final 字段有一条特殊承诺：只要对象正确构造（构造期间 this 没有逸出），构造函数结束时会有一个 freeze 动作，其他线程不用任何同步就能看到 final 字段的正确值。这就是上一节"不可变对象天然线程安全"的底层保证——不变性和安全发布，在 final 这里合流了。

final 字段安全发布 · 不变性的地基无需 volatile / 锁
public final class ImmutablePoint {
    private final int x, y;   // ★ final
    public ImmutablePoint(int x, int y) {
        this.x = x; this.y = y;
    }
    // 构造完成后，任意线程读到的 x/y 必是构造时写入的值
}

④ 锁保护的字段 / 线程安全容器

把读、写都放进用同一把锁保护的临界区：解锁 happens-before 下一次加锁，于是写入对读取可见。更常见的做法是直接扔进现成的线程安全容器——它们内部已经用锁或 volatile 替你完成了安全发布。

锁 / 线程安全容器安全发布日常最高频的一种
// 方式 A：读写用同一把锁守护
private Holder holder;                        // 普通字段即可
public synchronized void set(Holder h) { holder = h; }
public synchronized Holder get() { return holder; }

// 方式 B：放入线程安全容器，取出即安全发布
Map<String, Holder> cache = new ConcurrentHashMap<>();
cache.put("h", new Holder(42));       // 别的线程 get 出来的一定完整
哪些容器算"帮你安全发布"？把元素放进 Hashtable、ConcurrentHashMap、Collections.synchronizedMap，或 Vector、CopyOnWriteArrayList、synchronizedList，或 BlockingQueue、ConcurrentLinkedQueue——都能安全发布该元素。这正是生产者-消费者模式（下一章）天然安全的原因：把对象 put 进阻塞队列的动作，本身就完成了向消费者线程的安全发布。

五、对比：不安全发布 vs 安全发布（同一段代码）

下面发布一个"事实不可变"的配置对象。差别只在字段修饰符一个词，行为却天壤之别。

✗ Bad · 普通字段发布
public Config config;  // 裸字段

void load() {
  config = parse(file);
}
// 别的线程可能读到 null，
// 或读到字段还没填好的半成品 💥
✓ Good · volatile 发布
public volatile Config config;

void load() {
  config = parse(file);
}
// 只要读到非 null，
// 就一定是完整的 Config ✅

六、发布要求由"可变性"决定

要不要安全发布、发布后能不能直接共享，取决于对象的可变性。这是《Java 并发编程实战》最实用的一张决策表：

对象类型发布要求发布后能否直接并发访问
不可变（所有字段 final 且状态永不变）任意方式，即便"不安全发布"也 OK可任意线程并发读，无需额外同步
事实不可变（构造后不再改，但没写 final）必须安全发布发布后当只读用，无需额外同步
可变（发布后还会改）必须安全发布每次访问仍需同步：对象自身线程安全，或统一用锁守护
那个反直觉的结论："不可变对象怎么发布都安全"。因为 final 字段语义兜底：另一个线程只要能看到这个引用，就必然看到正确的 final 字段值——顶多是暂时看不到引用（读到 null），但绝不会看到"半成品"。所以真正 100% 不可变（字段全 final、构造期 this 不逸出）的对象，即使用最原始的普通字段发布，也不会重演第二节的翻车。这也是为什么把对象设计成不可变是最省心的并发策略。

七、四个必踩的坑

坑 1 · 安全发布 ≠ 之后一直线程安全。安全发布只保证"发布那一刻"的状态可见。如果对象是可变的、发布后还继续改，别的线程不保证看到后续修改——那需要对象本身线程安全，或读写继续用锁。volatile 发布的是"引用"这一层，管不到对象内部字段后续的可见性。
坑 2 · final 保证只对"正确构造"的对象有效。若构造函数里 this 逸出（把 this 注册进监听器、启动内部线程、传给外部集合），别的线程可能在 freeze 之前就拿到引用，final 的可见性承诺当场失效。呼应《发布与逸出》：构造期间绝不能让 this 逸出。
坑 3 · 只锁写、不锁读 = 半个安全发布。用锁做安全发布时，读和写必须是同一把锁。只给 set 加 synchronized 而 get 裸读，读线程照样可能看到旧引用或旧状态。
坑 4 · "有序发布"不等于"发布了正确的东西"。安全发布解决的是可见性/有序性，不解决逻辑正确性。若你发布的本就是个共享可变对象，多个线程拿到后并发乱改，安全发布救不了你——那是"组合设计线程安全"（下一节）的范畴。

八、建立联系：框架里到处是安全发布

Spring 单例 Bean。Spring 的 DefaultSingletonBeanRegistry 把单例存进一个 ConcurrentHashMap（singletonObjects）——正是"放入线程安全容器"这种安全发布。容器启动期创建并 put，运行期各线程从 map 取出即安全可见。而 @ConfigurationProperties 绑定的配置多为事实不可变对象，启动写、运行只读，完美契合上表第二行。
DCL 与 String。双重检查锁的 instance 字段必须 volatile，就是方式 ② 的直接应用（第 16 章还会细讲）；String 内部的 value、hash 靠 final/不可变实现方式 ③，所以字符串能被无数线程共享而不用同步。你会发现：并发库里"能安全共享"的类，几乎都能在这四种方式里对号入座。

九、面试速记清单

背诵要点：
· 安全发布要同时让"引用 + 对象状态"对读线程可见，本质是建立 happens-before 边。
· 不安全发布两大故障：读到 null（引用不可见）、读到默认值/半成品（状态不可见，如经典 n != n）。
· 四种方式：静态初始化器 / volatile·AtomicReference / final 字段 / 锁守护·线程安全容器。
· 各借一条规则：类初始化锁 / volatile 规则 / final freeze / 锁规则。
· 发布要求看可变性：不可变→随便发；事实不可变→必须安全发布后只读；可变→安全发布 + 持续同步。
· 三个必记陷阱：安全发布只管发布那一刻；构造期 this 逸出毁掉 final 保证；读写必须同一把锁。
一句话收束：发布一个对象，等于要同时把引用和它的完整状态交到别的线程手里；只有在两者之间建立 happens-before 边，才叫安全发布。Java 给了四条现成通道——静态初始化器、volatile/AtomicReference、final 字段、锁或线程安全容器——分别复用类初始化锁、volatile 规则、final freeze 与锁规则。至于发布后能否放心共享，则由对象的可变性拍板：不可变最省心，事实不可变要安全发布后只读，可变对象还得靠下一节的"组合"继续把它裹成线程安全的类。
