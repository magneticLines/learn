不变性：不可变对象是天然的线程安全

不改变的东西，就没有"改坏"的窗口——为什么"永不变"是并发里最省心的一招。

一、一个"反直觉却最省心"的结论

前几节我们一直在跟"共享的可变状态"搏斗：竞态、加锁、原子变量……根源都在那个"可变"上。那能不能干脆釜底抽薪——让状态压根没法变？这就是本节的主角：不可变对象（immutable object）。

无可变状态  ⇒  无竞态  ⇒  无需同步  =  天然线程安全

回想上一节我们反复给复合操作找"原子性"：竞态之所以存在，是因为"某个线程读到的值，可能被另一个线程改掉"。如果一个对象从生下来到死都不会变，那"被改坏"这件事就从物理上不可能发生——所有线程任何时刻看到的都是同一份、且是正确的状态。不需要锁，不需要 volatile，不需要 Atomic。

可变对象：一条不断被改写的时间线 t1 Account.balance = 100 t2 线程 B 读到 balance = 100 ← 脏读/竞态都藏在这些缝隙里 t3 Account.balance = 80 → 谁都可能撞见"改到一半"的中间态 不可变对象：一张拍好就不再动的快照 Money(100) ── new 出来那一刻，字段就被"冻结" 想"改成 80"？ 只能 new 一个新的 Money(80) → 旧快照永远是 100，N 个线程同看一张快照，看到的永远一致
生活类比：可变对象像一块白板，谁都能上去擦了重写，你回头看时可能正赶上别人擦到一半；不可变对象像一张已经冲印出来的照片，内容定格了，你要"新版本"就再拍一张，老照片永远不会在你手里变样。多人同时看同一张照片，绝不会有人看到"半张脸"。

二、到底什么才算"不可变"——三个必要条件

很多人以为"给字段加个 final 就不可变了"，这是最大的误会。书里给出的定义是三条缺一不可的条件：

  1. 对象创建后，其状态就不能再被修改（没有任何 setter，也没有任何方法会改动内部字段）；
  2. 所有域都是 final 类型（这不仅是"防手滑"，更是为了拿到内存模型给 final 域的特殊保证，下一节细说）；
  3. 对象是被"正确创建"的——构造期间 this 引用没有逸出（没有在构造函数里把自己提前塞给别的线程）。

注意第一条的微妙之处："状态不可变" ≠ "内部不能持有可变对象"。只要那个可变对象被严严实实地封在里面、永不逸出，从外部看这个对象依然是铁板一块。看书里这个经典例子——它内部用的是一个可变的 HashSet，但整体却是不可变的：

ThreeStooges.javaJCP §3.4 · 内含可变 Set 却依然不可变
public final class ThreeStooges {
    // final：引用不可换；Set 本身可变，但它被彻底封死在内部
    private final Set<String> stooges = new HashSet<>();

    public ThreeStooges() {
        stooges.add("Moe");
        stooges.add("Larry");
        stooges.add("Curly");
        // 构造完成后，再没有任何路径能改动这个 Set
    }

    public boolean isStooge(String name) {
        return stooges.contains(name);  // 只读，绝不把 stooges 引用泄露出去
    }
}

关键就在于：这个类没有任何修改方法，也从不把 stooges 的引用暴露给外界。可变的 HashSet 被"关进笼子"，外部无从下手，于是整体对外呈现为不可变。

三、final 的陷阱：它锁的是"引用"，不是"内容"

这是初学者和面试里最容易翻车的一点。final 修饰一个引用类型字段时，它保证的是"这根指针不能再指向别的对象"，而完全管不住被指向的那个对象内部会不会变。

FinalTrap.javafinal ≠ immutable
final List<String> list = new ArrayList<>();

list.add("我照样能被加进去！"); // ✅ 合法：改的是"内容"
list = new ArrayList<>();          // ❌ 编译错误：改的是"引用"
高频面试陷阱：final 的对象不等于不可变的对象。final 只冻结"引用绑定"，不冻结"对象状态"。一个 final Date、final ArrayList、final int[] 里的内容都能被随意修改——它们统统不是不可变对象。想让对象真正不可变，"所有域 final"只是必要条件之一，还必须保证这些域指向的东西本身也不可变，或被安全封装、防御性拷贝（见第五节）。
为什么定义里非要"所有域都 final"？——牵出内存模型。JMM 对 final 域有一条特殊保证（"final 字段语义"/初始化安全性）：只要对象正确构造（构造期间 this 没逸出），那么任何线程哪怕通过数据竞争拿到这个对象的引用，也一定能看到 final 域被正确初始化后的值。原理是编译器会在构造函数结尾插入一道 StoreStore 屏障，禁止把"写 final 域"重排到"发布 this 引用"之后。正因如此，不可变对象可以被任意方式安全发布——这正是下一节《安全发布》的铺垫。

四、String：不可变的教科书

你天天在用的 String 就是最标准的不可变对象：类被 final（不能被继承来搞破坏），核心字节数组被 private final 封死，所有"修改"方法（substring、concat、replace……）都返回一个新 String，绝不改动原对象。设计者为什么甘愿承受"每次改都新建对象"的开销也要让它不可变？因为不可变带来了一连串"白捡"的好处：

不可变带来的红利具体是怎么受益的
天然线程安全多线程共享同一个 String 无需任何同步
可自由缓存 / 共享字符串常量池（String Pool）能把相同字面量复用成同一个对象
hashCode 可缓存内容永不变 ⇒ 哈希值算一次就能存起来，HashMap 用 String 当 key 极快
可安全当 Map 的 key放进去后 key 不会"背着你变身"，不会破坏哈希桶的定位
安全类加载路径、数据库 URL、文件名等即使被恶意代码拿到引用也改不了
一个"教科书里没写"的彩蛋：String 有个域不是 final。看 JDK 源码，String 缓存哈希值的字段是 private int hash;（并非 final，首次调用 hashCode() 时才惰性计算并写入）。这不违反不可变吗？——不违反。因为这是一场良性数据竞争（benign data race）：不同线程即便同时计算 hash，算出来的结果完全相同，谁写进去都一样，最坏不过是多算几遍。所以书里的三条件里"所有域 final"是强烈推荐的稳妥做法，而非绝对铁律——但除非你像 JDK 作者那样能证明竞争是良性的，否则老老实实全 final。

五、当类必须持有可变组件：防御性拷贝

现实中你常要封装像 Date、数组这种本身可变的类型。这时想保持不可变，就要在入口和出口两道关都做防御性拷贝（defensive copy）：构造时拷贝一份存起来，返回时再拷贝一份给出去。否则外部握着的引用就是一把能捅进你内部的刀。

✗ Bad · 直接存/直接给引用
public final class Period {
  private final Date start;
  public Period(Date s) {
    this.start = s;   // 存了外部引用
  }
  public Date getStart() {
    return start;      // 又把引用漏出去
  }
}
// 调用方:
Date d = new Date();
Period p = new Period(d);
d.setTime(0);          // 内部被从外面改了！
✓ Good · 入口出口各拷一份
public final class Period {
  private final Date start;
  public Period(Date s) {
    // 入口拷贝：切断外部引用
    this.start = new Date(s.getTime());
  }
  public Date getStart() {
    // 出口拷贝：不泄露内部引用
    return new Date(start.getTime());
  }
}
// 外部怎么改自己那份都伤不到内部
易错点：Collections.unmodifiableList() 只是"只读视图"，不是拷贝。它返回的是原 List 的一层包装：你若还攥着原始 List 的引用，照样能改，改动会"透"到这个只读视图上。而且它是浅只读——List 里装的元素对象本身若可变，依然能被改。真要不可变集合，用 List.copyOf(...)（Java 10+，会真拷贝）或 Guava 的 ImmutableList。

六、volatile + 不可变对象：不加锁的"整体替换"

上一节我们见过一个反例 UnsafeCachingFactorizer：它用两个独立的原子变量缓存"上次算的数"和"它的因数"，结果因为跨变量的不变式（"factors 必须是 number 的因数"）没被原子地维护而出错。不可变对象给了它一个优雅解法——把彼此相关的多个变量打包进一个不可变对象，用一个 volatile 引用"整体替换"：

OneValueCache.javaJCP 清单 3.11 · 把 number 与 factors 冻成一个原子整体
class OneValueCache {              // 不可变的"结果快照"
    private final BigInteger   lastNumber;
    private final BigInteger[] lastFactors;

    public OneValueCache(BigInteger n, BigInteger[] factors) {
        lastNumber  = n;
        lastFactors = Arrays.copyOf(factors, factors.length); // 防御性拷贝
    }

    public BigInteger[] getFactors(BigInteger n) {
        if (lastNumber == null || !lastNumber.equals(n))
            return null;                            // number 对不上，未命中
        return Arrays.copyOf(lastFactors, lastFactors.length);
    }
}
VolatileCachedFactorizer.javaJCP 清单 3.12 · 一个 volatile 引用搞定，无需锁
public class VolatileCachedFactorizer {
    // volatile 保证"换出的新快照"对其他线程立刻可见
    private volatile OneValueCache cache = new OneValueCache(null, null);

    public BigInteger[] service(BigInteger i) {
        BigInteger[] factors = cache.getFactors(i); // 读到的一定是配套的一对
        if (factors == null) {
            factors = factor(i);
            cache = new OneValueCache(i, factors); // 整体替换，一步到位
        }
        return factors;
    }
}

妙在哪？number 和 factors 被封进同一个不可变对象，它俩要么一起旧、要么一起新，绝无"新 number 配旧 factors"的中间态。更新时只是让 volatile 引用从"旧快照"整体跳到"新快照"——换引用本身是原子的，加上 volatile 保证可见性，于是整个方法一把锁都不用，既线程安全又高并发。这正是"用不可变性替代加锁"的典范。

为什么这里 volatile 就够了、不用 Atomic？因为这里的操作是"读一个引用 → 判断 → 直接覆盖写一个新引用"，我们并不依赖写之前的旧值（大不了两个线程各算一遍、各覆盖一次，结果都正确），所以不构成"读-改-写"复合操作，无需 CAS。volatile 提供的可见性恰好是这里唯一缺的那块拼图。

七、怎样亲手打造一个不可变类 + 生态一览

把前面的要点收拢成一张"施工清单"：

  • 类声明为 final（或用私有构造 + 工厂方法），防止子类破坏不可变性；
  • 所有字段 private final，且只在构造函数里赋值一次；
  • 不提供任何 setter，也不提供任何会改动字段的方法；
  • 对持有的可变组件（Date、数组、集合）做入口 + 出口双重防御性拷贝；
  • 构造期间不让 this 逸出（别在构造函数里注册监听器、启动线程、把 this 塞进静态容器）。
方案是否真不可变取舍 / 坑
String / BigInteger / BigDecimal / LocalDate是（JDK 内建）标准不可变类型，可放心跨线程共享
record（Java 14+）浅不可变自动生成 final 字段与访问器，最省事；但组件若是可变对象（如 List），仍需自己在紧凑构造器里做防御性拷贝
List.copyOf / Map.copyOf（Java 10+）是真正拷贝出不可变集合；试图 add 会抛 UnsupportedOperationException
Collections.unmodifiableList否（只读视图）不拷贝，仍持有原引用者可改；且是浅只读——见第五节陷阱
Guava ImmutableList / ImmutableMap是真不可变、构造即拷贝、性能好，第三方依赖
Lombok @Value浅不可变一键生成 final 字段 + 全参构造 + getter，语义同 record
别被 record / @Value 的"不可变"骗了——它们只是浅不可变。record Point(int x, int y) 确实铁板一块；但 record Team(String name, List<String> members) 里那个 List 引用虽是 final，List 内容照样能被改。record 不会替你做防御性拷贝。要么传入前就用 List.copyOf，要么在紧凑构造器里 members = List.copyOf(members)。

不可变 vs 可变 + 同步：怎么选

维度不可变对象可变对象 + 锁/Atomic
线程安全天然安全，零同步代码要靠正确加锁，易漏、易死锁
共享 / 缓存可随意共享、缓存、当 Map key共享需谨慎，当 key 会出问题
推理难度低：状态定格，没有中间态高：要盯着每个临界区
"修改"成本每次变更都要 new 新对象，产生垃圾原地改，无额外分配
适用场景值对象、配置、事件、DTO、缓存快照大对象频繁增量更新、高频热点计数
一句话收束：不可变对象把"并发难题"从根上消掉了——不能变，就没有需要保护的可变状态，也就没有竞态、没有可见性问题。它的三条件是"状态创建后不可改 + 所有域 final + this 不逸出"；记牢 final 只锁引用不锁内容，持有可变组件时要做防御性拷贝，而 Collections.unmodifiableXxx 和 record 都只是浅不可变。当你需要"共享一组相关状态"时，优先考虑把它们打包成不可变对象，用一个 volatile 引用整体替换——这往往比加锁更简单、更快、更不容易错。而"不可变对象能被任意方式安全发布"背后的 final 域内存语义，正是下一节《安全发布》要展开的话题。
