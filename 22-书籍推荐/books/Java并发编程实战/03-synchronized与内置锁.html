synchronized 与内置锁

上一节我们看到复合操作会撕裂成竞态；这一节，Java 给每个对象都内置了一把锁，用互斥把这条裂缝重新缝合。

一、承上：为什么"加锁"是复合操作的解药

上一节留下一个悬案：check-then-act（检查后执行）、read-modify-write（读改写）这类复合操作之所以不安全，是因为它由多条指令组成，两条指令之间能被其他线程插进来。要修复它，只需要一件事：让这一串指令不可分割地整体执行——这就是原子性。

经典竞态：put-if-absent两步之间是可以被打断的缝隙
// 想实现"不存在才放入"，但 get 和 put 是两步
if (map.get(key) == null) {   // ← ①检查（此刻可能被切走）
    map.put(key, value);        // ← ②执行（回来时前提已失效）
}
// 两个线程都在①看到 null，于是都执行②，后者覆盖前者 💥

Java 提供的最基本的加锁工具，就是关键字 synchronized。它保护的那把锁，不需要你手动 new 出来——每个 Java 对象天生自带一把锁，这就是"内置锁"的由来。

二、内置锁（Intrinsic Lock）到底是什么

《Java 并发编程实战》给它起了个正式名字：内置锁（Intrinsic Lock），也叫监视器锁（Monitor Lock）。理解它，抓住三个关键事实：

  • 它挂在对象上：每个 Java 对象都可以充当一把锁。synchronized 块必须指定"以哪个对象为锁"。
  • 它是互斥的：同一时刻最多只有一个线程能持有某个对象的内置锁，其余线程阻塞等待。
  • 它是自动的：进入 synchronized 块自动加锁，退出（包括正常返回和抛异常）自动释放——你不会忘记解锁。
为什么叫"监视器"（Monitor）？这是并发理论里的经典概念（Hoare 于 1974 年提出）。监视器 = 一段被"互斥门禁"保护的临界区 + 一套等待/通知机制。Java 把这个学术模型直接编进了语言：synchronized 提供互斥门禁，Object 的 wait() / notify() 提供等待通知。所以你在 Object 类里看到 wait/notify 是"天经地义"的——因为每个对象都是一个潜在的监视器。

编译后，synchronized 代码块对应字节码里成对的 monitorenter / monitorexit 指令；同步方法则在方法的访问标志上打一个 ACC_SYNCHRONIZED 标记，由 JVM 在调用前后隐式加解锁。锁的状态（谁持有、重入了几次）记录在对象头的 Mark Word 里。

三、三种写法，锁的到底是哪个对象

这是面试高频送分题，也是最容易写错的地方。核心只有一句话：看清 monitor 挂在哪个对象上。

Counter.java三种用法 = 三个不同的锁对象
class Counter {
    // ① 同步实例方法 → 锁 this（当前实例，每个对象一把）
    public synchronized void incr() { ... }

    // ② 同步静态方法 → 锁 Counter.class（类对象，全 JVM 唯一）
    public static synchronized void reset() { ... }

    // ③ 同步代码块 → 锁你指定的任意对象（推荐：粒度可控）
    private final Object lock = new Object();
    public void add() {
        synchronized (lock) { ... }   // synchronized(this) 等价于①
    }
}
写法锁的对象作用范围
实例方法 synchronizedthis（当前实例）同一实例的所有同步实例方法互斥
静态方法 synchronized类名.class（类对象）全 JVM 内该类的所有同步静态方法互斥
同步代码块括号里指定的对象只锁需要保护的那几行，粒度最小
两个必踩的坑：
· 实例锁和类锁互不干扰。①锁 this、②锁 Counter.class，是两个不同的对象。所以线程 A 在执行 incr() 时，不会阻塞线程 B 执行静态的 reset()——如果它们改的是同一份数据，就仍然是竞态。
· 锁一个会变的引用，等于没锁。用 synchronized(new Object())（每次新对象，永远抢得到）、或锁一个可变字段、或锁 String 常量与被缓存的 Integer（-128~127 会与别处意外共享同一实例），都会锁错或锁空。所以约定俗成：用 private final Object lock 作专用锁对象——引用稳定、外部拿不到、不会被误用来做别的锁。

四、互斥：竞态是怎么被"缝合"的

把上一节的坏例子用同一把锁包起来，两步就变成了"不可分割的一步"：别的线程要么完整地看到"检查+执行"之前，要么看到之后，永远看不到中间态。

✗ Bad · 复合操作裸奔
class Seq {
  private int value;
  public int next() {
    return value++;  // 读-改-写
  }
}
// 两线程同拿到 5，都返回 5，
// 序号重复 + 丢更新 💥
✓ Good · 内置锁保证原子
class Seq {
  private int value;
  public synchronized int next() {
    return value++;  // 整体不可分割
  }
}
// 一次只有一个线程进临界区，
// 序号严格递增 ✅
时间 ──► 线程 A: monitorenter ─[ 读5 · 改6 · 写回 ]─ monitorexit 线程 B: 阻塞……等 A 释放…… monitorenter ─[ 读6 · 改7 · 写回 ]─ exit ▲ 互斥：B 绝不会在 A 的临界区中间挤进来

五、可重入（Reentrancy）：锁认的是"线程"不是"次数"

内置锁有一个极其重要、却常被忽略的性质：可重入。意思是——持有锁的线程可以再次进入被同一把锁保护的代码，而不会把自己锁死。它的实现原理很朴素：每把锁关联一个持有线程和一个重入计数器。

线程 T 调用 LoggingWidget.doSomething() ── synchronized 锁的是 this ┌ 外层 synchronized(this) │ owner == null ? 是 → owner = T, count: 0 → 1 （抢到锁） │ └─ super.doSomething() 也是 synchronized(this) │ owner == T ? 是 → 不阻塞自己！ count: 1 → 2 （重入，无需再竞争） │ ...执行父类逻辑... │ 退出 super → count: 2 → 1 └ 退出外层 → count: 1 → 0 → owner 置空，真正释放锁

如果内置锁不可重入，下面这段极普通的继承代码会直接死锁——子类方法已经持有 this 锁，调用 super.doSomething() 时又要获取 this 锁，结果自己等自己永远等不到。这正是《Java 并发编程实战》举的经典例子。

为什么锁必须可重入否则子类调父类的同步方法就死锁
class Widget {
    public synchronized void doSomething() { ... }
}

class LoggingWidget extends Widget {
    public synchronized void doSomething() {
        System.out.println(toString() + ": calling doSomething");
        super.doSomething();   // 再次请求 this 锁——可重入才不会死锁
    }
}
建立联系：可重入不是内置锁独有的。后面章节要讲的显式锁 ReentrantLock，名字里的 "Reentrant" 就是"可重入"——它同样靠"持有线程 + 计数器"实现（内部由 AQS 的 state 字段承担计数）。区别只在于 ReentrantLock 需要你手动 lock()/unlock() 且成对配平，而 synchronized 的加解锁由 JVM 托管，更不容易出错。

六、JCiP 的核心律条：用锁保护"状态"，而不是"方法"

新手常把 synchronized 理解成"给方法上锁"，这是本末倒置。《Java 并发编程实战》反复强调的原则是——锁保护的是数据（共享可变状态），不是代码。由此推出一条铁律：

每个共享可变变量，都只能由 同一把锁 守护全部访问

关键词有三个：共享（会被多个线程访问）、可变（值会改变）、同一把锁（读和写、以及所有相关变量，用的必须是同一个锁对象）。缺任何一个，锁都白加。下面是最典型的两种"同步不足"翻车现场。

✗ Bad · 只锁写、不锁读
class Point {
  private int x, y;
  synchronized void set(int a,int b){
    x = a; y = b;
  }
  int getX() { return x; }  // 没锁！
  int getY() { return y; }  // 没锁！
}
// 读方读到旧值 / 半更新，
// 且无可见性保证 💥
✓ Good · 读写用同一把锁
class Point {
  private int x, y;
  synchronized void set(int a,int b){
    x = a; y = b;
  }
  synchronized int getX(){ return x; }
  synchronized int getY(){ return y; }
}
// 读写互斥 + 可见性齐活，
// x/y 始终成对一致 ✅
面试高频陷阱：多个同步方法拼起来，不等于线程安全。即使 contains() 和 add() 各自都是 synchronized 的，下面这段"如果不存在就添加"仍是竞态——因为两次调用之间锁已经被释放了，别的线程能趁虚而入。这说明：原子性的边界，是你划定的临界区，而不是单个方法。正确做法是把复合动作整体放进一个同步块。
✗ Bad · 原子方法 + 非原子组合
if (!list.contains(x))   // 同步，但…
    list.add(x);          // …两步之间有缝
// 两线程都判"不含"，重复添加 💥
✓ Good · 把复合动作圈进临界区
synchronized (list) {
    if (!list.contains(x))
        list.add(x);
}
// 检查+添加 一气呵成，原子 ✅

七、锁的粒度：性能与安全的拉锯

既然一把大锁能保证安全，为什么不把整个方法都锁上？因为锁会串行化执行——临界区越大，能并行的部分越少，还容易把耗时的 I/O、网络调用也一并锁进去，拖垮吞吐量。但缩得太小又会切断复合操作的原子性。这中间有个权衡：

JCiP 的实践建议：把不影响共享状态的耗时操作（如日志、网络请求、复杂计算）移出同步块；同步块里只保留"读写共享状态"这几行。原则是——同步块要够大以保证不变性，又够小以不必要地阻塞别人。
✗ Bad · 锁里干耗时活
synchronized void handle(Req r){
  count++;                // 需要锁
  byte[] resp = io(r);   // 慢！却也被锁住
  cache.put(r, resp);
}
// 所有线程排队等 io()，形同单线程
✓ Good · 只锁必须同步的部分
void handle(Req r){
  synchronized(lock){ count++; }
  byte[] resp = io(r);   // 锁外并行
  synchronized(lock){ cache.put(r,resp); }
}
// io() 可并发，吞吐大幅提升
synchronized 并不总是"重量级"。JDK 6 起引入了锁升级：无锁 → 轻量级锁（CAS 自旋）→ 重量级锁。只有在真正激烈竞争时才膨胀为依赖操作系统 mutex 的重量级锁（涉及用户态/内核态切换才慢）。竞争不激烈时开销很小。所以"synchronized 一定慢"是老黄历。（注：偏向锁在 JDK 15 起默认关闭并被弃用，见 JEP 374。）

八、别忘了：内置锁还顺带解决了"可见性"

很多人只记得 synchronized 提供互斥（原子性），却忽略了它同时保证内存可见性：线程释放锁时会把工作内存刷回主内存，获取同一把锁时会清空工作内存、重新从主内存读取。所以被同一把锁保护的普通变量，天然可见，不需要再额外加 volatile。

synchronized = 互斥（原子性） +  进出锁的内存同步（可见性）
埋个伏笔：这个"释放锁 → 获取锁"之间的可见性保证，背后是 Java 内存模型的 happens-before 规则中的"锁规则"：对一个锁的解锁，happens-before 于后续对同一把锁的加锁。这也是为什么互斥和可见性总是被一把锁一起兑现。这条线索，正是下一节 volatile 与 happens-before 的主角。

九、对比：synchronized vs 显式锁 ReentrantLock

内置锁足够好用，但它有天花板：不能中断等待、不能设超时、不能实现公平锁、只能有一个等待队列。这些正是显式锁 ReentrantLock（第 10-11 章）诞生的理由。先建立一个对比框架：

维度synchronized（内置锁）ReentrantLock（显式锁）
加解锁JVM 自动，出块/异常自动释放手动 lock()/unlock()，须放 finally
可重入✓✓
可中断等待✗ 只能死等✓ lockInterruptibly()
可超时/尝试✗✓ tryLock(timeout)
公平锁✗ 非公平✓ 可选公平/非公平
等待通知wait/notify，单一队列多个 Condition，可分组唤醒
忘记释放风险无（自动）有（漏写 unlock 会泄漏）
那到底用哪个？《Java 并发编程实战》的建议很务实：默认优先用 synchronized——它更简单、不会忘记释放、且是内建语义，工具和 JVM 对它优化最好。只有当你确实需要可中断、可定时、公平锁、或多个条件队列这些高级能力时，才换成 ReentrantLock。不要为了"听说 Lock 更快"而无脑替换——现代 JVM 里两者性能已非常接近。

十、面试速记清单

背诵要点：
· 内置锁=每个对象自带的监视器锁；字节码 monitorenter/monitorexit，状态记在对象头 Mark Word。
· 三种用法锁的对象：实例方法锁 this、静态方法锁 Class、代码块锁指定对象；实例锁与类锁互不阻塞。
· 可重入：靠"持有线程 + 计数器"，同一线程可重复进入；否则子类调 super 同步方法会死锁。
· 核心律条：每个共享可变变量由同一把锁保护全部访问；只锁写不锁读 = 半个安全。
· 原子性边界是临界区不是方法：多个同步方法拼接≠线程安全（如 if(!contains)add）。
· synchronized 同时提供互斥 + 可见性；被同一把锁保护的变量无需再加 volatile。
· 锁粒度权衡：耗时操作移出同步块；synchronized 有锁升级，早已不是"一定慢"。
一句话收束：synchronized 借助每个对象内置的监视器锁，用互斥把复合操作重新变回原子的一步，同时靠"进出锁的内存同步"顺带保证可见性；它可重入，所以继承与自调用不会自锁。用好它只需记住一条心法——锁保护的是数据不是代码，每个共享可变状态都必须由同一把锁守护它的每一次读写。当你需要可中断、可超时、公平或多条件队列时，再升级到显式的 ReentrantLock。
