安全初始化与双重检查锁（DCL）

DCL 是被写进面试题的"经典反面教材又经典正面教材"——搞懂它为什么必须加 volatile，你就真正看懂了 JMM 里"半初始化对象"的幽灵。

一、问题的起点：延迟初始化

很多对象昂贵又不一定用得上——一个几十 MB 的缓存、一条数据库连接池、一个重量级的解析器。我们希望它第一次被用到时才创建，这就是延迟初始化（lazy initialization）。它省内存、加快启动，代价是把"创建"这个动作从确定的类加载期，挪到了不确定的运行期——于是多个线程可能同时撞上"还没创建"这个窗口。

先厘清两条路：如果对象一定会用、且创建不贵，直接用饿汉式（static final 字段，类加载即创建）——它天然线程安全，根本不需要 DCL。只有当你确实需要又懒又线程安全时，才会走进 DCL 或 Holder 这些讨论。别一上来就套 DCL，那是把简单问题复杂化。

二、朴素懒汉：一个教科书级的竞态

最直觉的写法，恰恰是错的。它踩中了我们在《什么是线程安全性》里讲过的先检查后执行（check-then-act）竞态：

✗ 朴素懒汉，非线程安全
class Singleton {
  private static Singleton inst;

  static Singleton get() {
    if (inst == null) {       // 检查
      inst = new Singleton(); // 执行
    }
    return inst;
  }
}
// 两个线程同时看到 inst==null
// → 各 new 一次，单例被破坏
✓ 同步方法，正确但慢
class Singleton {
  private static Singleton inst;

  static synchronized
  Singleton get() {          // 整个方法加锁
    if (inst == null) {
      inst = new Singleton();
    }
    return inst;
  }
}
// 正确，但每次读都要抢锁

右边的 synchronized 方法确实正确，但它的代价刺眼：单例只在第一次需要初始化，之后的成千上万次调用都只是"取一个已存在的引用"，却仍要各自抢一次锁、走一次内存屏障。热点路径上被一把用不上的锁拖慢——这正是 DCL 想解决的痛点。

三、双重检查锁：给"第一次"单独加锁

DCL（Double-Checked Locking）的思路很聪明：先无锁地检查一次，只有当对象确实还没创建时才加锁；进了锁再检查第二次，防止两个线程都通过了第一次检查后重复创建。这样一旦对象建好，后续调用在第一次检查处就返回了，完全不碰锁。

Singleton.java正确的 DCL —— 注意 volatile
class Singleton {
    // ★ volatile 不可省略，原因见第四节
    private static volatile Singleton instance;

    private Singleton() { /* 昂贵的初始化 */ }

    static Singleton getInstance() {
        if (instance == null) {              // ① 第一次检查（无锁，快速路径）
            synchronized (Singleton.class) {
                if (instance == null) {      // ② 第二次检查（持锁，防重复创建）
                    instance = new Singleton();  // ③ 发布引用
                }
            }
        }
        return instance;
    }
}
两次检查各司其职：第 ① 次检查是性能优化——绝大多数调用发生在对象已就绪之后，让它们无锁直接返回。第 ② 次检查是正确性保证——线程 A、B 可能都通过了 ① 号检查排队进锁，A 建好后 B 拿到锁，若不再查一次，B 会再 new 一个。少了任何一次检查，DCL 都不成立。

四、致命细节：为什么必须 volatile？

这是 DCL 全部精髓所在，也是面试必问。很多人写 DCL 时把 volatile 去掉，程序在测试里跑得好好的，上了生产才偶发崩溃——因为祸根是一个只在特定 CPU / JIT 优化下才显形的重排序问题。

看似原子的一行 instance = new Singleton()，编译成字节码后其实是三步（对应 new / invokespecial <init> / putstatic）：

instance = new Singleton() 实际拆成三步： ① allocate 分配内存，字段先置零值（此时对象"存在"但没初始化） ② <init> 执行构造函数，把字段设为真正的初始值 ③ publish 把 instance 引用指向这块内存 正常顺序： ① → ② → ③ （构造完再发布，安全） 重排之后： ① → ③ → ② （先发布再构造！JMM 允许，因为对单线程结果无影响） ▲ └── 危险窗口：instance 已非 null，但对象还没构造完

JMM 允许把 ② 和 ③ 重排——在单线程视角里，先赋引用还是先构造，最终结果一样，所以编译器 / CPU 有权这么优化。但在多线程下就出事了：

时间线（无 volatile，发生了 ①→③→② 重排）： 线程 A 线程 B ───────────────────── ───────────────────── ① 分配内存 ③ instance = 那块地址 ──────► ① if (instance == null) → false！ （看到非 null，跳过锁） ② 执行构造函数... return instance; ↓ 使用一个"半初始化"对象 字段还是零值 → NPE / 脏数据
陷阱一：没有 volatile 的 DCL 会返回"半初始化对象"。线程 B 在第一次检查里看到 instance != null 就直接返回并使用它，可此刻线程 A 的构造函数还没跑完，对象的字段仍是默认零值（引用为 null、int 为 0）。B 拿着这个"空壳"去调方法，轻则读到脏字段，重则 NullPointerException。这种 bug极难复现——它依赖重排序真实发生，只在高并发、特定 JIT/CPU 下偶发。

加上 volatile 后，问题从两个层面被堵死：

  • 禁止重排（有序性）：对 volatile 变量 instance 的写（③）之前会插入 StoreStore 屏障，逼着构造函数里对字段的所有普通写（②）必须先于 ③ 完成刷回主内存——② 再也不能被挪到 ③ 之后。
  • 建立 happens-before（可见性）：线程 A 对 volatile 的写 hb 线程 B 对它的读；由传递性，A 在构造函数里对字段的初始化，对读到非空引用的 B 完全可见。这正是上一篇讲的 volatile "搭便车"发布对象。
② 构造 hb ③ volatile写 hb ① volatile读 hb 使用
构造完成 → 对读到引用的线程完全可见
陷阱二：Java 5 之前，加了 volatile 的 DCL 也是坏的。网上那篇著名的《Double-Checked Locking is Broken》正是老内存模型时代的产物。JSR-133（Java 5）重写了内存模型之前，旧 volatile 语义很弱——它不禁止 volatile 写与其前面普通写之间的重排，所以修不好半初始化问题。只有 Java 5 及以后，volatile 有了完整的 acquire/release 屏障语义，volatile 版 DCL 才真正成立。今天用 JDK 8/11/17，放心用；但要知道这段历史，别在旧 JDK 上照搬。

五、更优雅的替代：静态内部类 Holder

DCL 能用，但它把 JMM 的复杂度暴露给了每个读者——一不小心漏个 volatile 就埋雷。有没有既懒加载、又线程安全、还不用自己操心屏障的写法？有，而且更简单：静态内部类持有者（Initialization-on-demand Holder）。

Singleton.javaHolder 模式 —— 把线程安全外包给 JVM
class Singleton {
    private Singleton() {}

    // 内部类不会随外部类加载而加载
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    static Singleton getInstance() {
        return Holder.INSTANCE;  // 首次访问才触发 Holder 初始化
    }
}
它凭什么线程安全又懒加载？靠的是 JVM 的类初始化机制（JLS §12.4）。Holder 作为一个独立的类，只有在第一次被主动使用（这里是读 Holder.INSTANCE）时才会被加载和初始化——这就是"懒"。而 JVM 规范强制要求：类的初始化过程（执行 <clinit>）由一把初始化锁保护，保证在多线程下 <clinit> 只会被执行恰好一次，其余线程会阻塞等待完成。线程安全由 JVM 用类初始化锁替你实现了，你一行同步代码、一个 volatile 都不用写——把复杂度外包给了语言规范。

六、终极简洁：枚举单例

《Effective Java》第 3 条明确推荐：单元素枚举是实现单例的最佳方式。它不仅线程安全（同样靠类初始化锁），还免费解决了 DCL / Holder 都要额外操心的两个问题——反射攻击和序列化破坏。

Singleton.java枚举单例 —— 天然防反射 / 防序列化
enum Singleton {
    INSTANCE;

    public void doWork() { /* 业务方法 */ }
}

// 使用：Singleton.INSTANCE.doWork();
为什么枚举能防反射和序列化？JVM 从底层禁止用反射 newInstance() 创建枚举实例（会抛 IllegalArgumentException）；而枚举的序列化是特殊处理的——反序列化时按名字查找已有实例，而不是 new 一个新的。DCL 和 Holder 若不额外重写 readResolve()、加防护，都可能被反射或反序列化"造出第二个实例"，破坏单例。枚举把这些坑一次性堵上了。

选型建议：需要懒加载 → 首选 静态内部类 Holder（简单、无 volatile 心智负担）；不在乎懒加载、追求最健壮 → 用枚举；只有当你要延迟初始化一个实例字段（不是静态单例），比如某个对象内部的懒加载缓存，Holder 用不上时，才轮到 DCL——这也是《JCIP》里 DCL 唯一还值得用的场景。作者原话是：DCL 曾是个流行技巧，但如今"已经被广泛废弃"，因为其收益（避免几次无竞争的锁开销）已经很微小。

八、串联：JDK 与 Spring 里的影子

安全初始化不是屠龙之技，你天天用的框架里全是它的身影：

  • JDK 里的 DCL：String.hashCode() 缓存了哈希值到一个非 volatile 的 hash 字段——它用的是"无害竞态"变体：即使重复计算也没关系，且 int 写本身原子，故不需要 volatile。这是与 DCL 相对的另一种思路：当重复初始化无害且结果是原子值时，连锁都可以省。
  • Spring 单例：Bean 的单例语义由容器保证，而非你写 DCL。DefaultSingletonBeanRegistry.getSingleton() 内部用 synchronized (this.singletonObjects) 保护创建过程，并通过"三级缓存"里的早期引用来处理循环依赖——本质仍是"检查—创建—发布"，只是搬到了容器层。
  • final 字段的安全发布：如果单例内部字段都是 final，JMM 给了一条特殊保证——构造函数中对 final 字段的写，在对象引用发布给其他线程时一定可见，无需额外同步。这是《不变性》一篇里不可变对象能被自由共享的根基，也和这里"如何把一个构造好的对象安全交出去"是同一个主题。
一句话收束：DCL 的全部难点浓缩成一句——instance = new Singleton() 不是原子的，它的"构造对象"与"发布引用"两步可能被重排，导致别的线程拿到非 null 但没构造完的半成品；volatile 用内存屏障禁止这次重排、并借 happens-before 把构造结果安全发布出去，所以在 DCL 里不可省略（且仅在 Java 5+ 才有效）。但更重要的是记住：大多数时候你根本不该手写 DCL——需要懒加载用静态内部类 Holder，追求健壮用枚举，把线程安全这件苦差事外包给 JVM 的类初始化锁，远比自己和 JMM 的重排序幽灵搏斗更靠谱。这也把《JMM 与 happens-before》的理论，落到了"如何安全地把一个对象交给多个线程"这个最实际的问题上。
