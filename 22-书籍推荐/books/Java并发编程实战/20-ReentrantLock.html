ReentrantLock：可中断、定时、公平

当内置锁"只能死等、不能反悔、无法排队"的天花板挡住你时，把加锁从关键字变成一个可编程的对象。

一、承上：synchronized 撞到了哪三堵墙

上一章我们已经确认：synchronized 简单、自动、不会忘记释放，是加锁的默认首选。既然如此，JDK 5 为什么还要在 java.util.concurrent.locks 里再造一把 ReentrantLock？答案不是"更快"（现代 JVM 两者性能已非常接近），而是内置锁有三件事根本做不到：

synchronized 的三堵墙 ───────────────────────────── ① 拿不到锁只能"死等" ── 无法响应中断，线程卡死无法取消 / 优雅关停 ② 要么阻塞、要么立刻 ── 不能"等 2 秒还拿不到就放弃"，无法探测/规避死锁 ③ 抢锁靠运气 ── 无法选择公平排队，也只有一个 wait 队列 ──────────────────────────────────────────────── ReentrantLock 把加锁做成"方法调用" → 三堵墙全部可编程

核心思想一句话：synchronized 是语法（由 JVM 托管），ReentrantLock 是一个普通对象（由你调用方法控制）。加锁一旦变成方法调用，就能给它加参数（超时时间）、加返回值（能否加锁）、加可中断语义——这正是"可中断、定时、公平"三大能力的来源。

二、基本用法：为什么 unlock() 必须写在 finally

灵活性是有代价的：内置锁进出块自动加解锁，而 ReentrantLock 把释放锁的责任交还给你。忘记释放，或在临界区抛异常后跳过了 unlock()，这把锁就永远泄漏——后续所有线程全部卡死。所以有一条铁律：

lock() 之后立刻 try，unlock() 永远放进 finally
✗ Bad · unlock 可能被跳过
lock.lock();
doWork();        // 若这里抛异常…
lock.unlock();   // …这行永远执行不到
// 锁泄漏，所有等待者永久阻塞 💥
✓ Good · finally 保证释放
lock.lock();
try {
    doWork();
} finally {
    lock.unlock();  // 无论正常/异常都释放
}
三个高频翻车点：
· lock() 不能写在 try 里面。要写在 try 之前——如果 lock() 本身抛异常（几乎不会，但语义上），你并没有持有锁，finally 里的 unlock() 反而会抛 IllegalMonitorStateException。
· lock() / unlock() 必须一一配平。因为它可重入，加了几次就要解几次；解多了直接 IllegalMonitorStateException。
· 不同线程不能替对方解锁。锁认的是"持有线程"，A 加的锁只能 A 来解。
建立联系：这正是 synchronized 的价值所在——它把"finally 里释放锁"这件极易出错的事编进了 JVM，出块或抛异常自动 monitorexit。所以书里反复强调：不需要那三大高级能力时，就别用 ReentrantLock，多写的每一个 try-finally 都是一次潜在的泄漏风险。

三、能力一 · 可中断：lockInterruptibly()

synchronized 等锁时是"聋子"——你对它 interrupt() 毫无作用，线程会一直阻塞到天荒地老。这在需要优雅取消 / 超时关停的场景（线程池 shutdown、用户点了"取消"）是致命的：一个卡在锁上的任务永远无法退出。

方法拿不到锁时的行为能否响应中断
lock()阻塞直到拿到（死等）✗ 不响应
lockInterruptibly()阻塞，但可被打断✓ 抛 InterruptedException
可中断地获取锁等锁期间被 interrupt() 就抛出，得以退出
public void doTask() throws InterruptedException {
    lock.lockInterruptibly();   // 等锁时若被中断 → 抛异常，不再傻等
    try {
        // ... 临界区 ...
    } finally {
        lock.unlock();
    }
}
为什么这很关键？中断是 Java 里线程间"请你停下来"的协作式信号（不是强杀）。一个无法响应中断的阻塞点，就是取消链条上的一个"黑洞"。lockInterruptibly() 让"等待获取锁"这个阻塞点也变得可取消，与 Thread.sleep()、BlockingQueue.take()、Future.get() 等其他可中断阻塞点拉齐，让整个任务能被一致地、干净地取消。

四、能力二 · 定时与轮询：tryLock()

tryLock() 把"要么拿到、要么阻塞"这个二选一，变成了带返回值的可判断动作：拿到返回 true，拿不到返回 false——决定权回到你手里。

调用形式语义典型用途
tryLock()立刻尝试，拿不到马上返回 false（不阻塞）抢不到就干别的，绝不排队
tryLock(t, unit)在超时时间内轮询等待，超时返回 false限时等待，可响应中断
一个反直觉的坑：tryLock() 无参版本会"插队"。即使你构造的是公平锁，无参 tryLock() 也会无视队列直接抢——只要锁当前空闲就拿走。这是 JDK 有意为之（源码注释里明确说 barging 是可接受的）。要保持公平语义，应使用 tryLock(0, TimeUnit.SECONDS)，它会遵守排队规则。

定时锁最经典的用武之地：规避死锁

上一节《死锁的检测与避免》讲过，两个线程以相反顺序获取两把锁就会抱死。用 synchronized 你只能靠"固定加锁顺序"来预防；而 tryLock 提供了另一条路——拿不到第二把锁就退避、释放已持有的锁、稍后重试，从机制上让死锁无法形成。

✗ Bad · 嵌套死等，顺序相反即抱死
// 线程1: transfer(A, B)
synchronized (a) {
  synchronized (b) { move(a, b); }
}
// 线程2: transfer(B, A) 顺序相反
// 1 持 a 等 b，2 持 b 等 a → 死锁 💥
✓ Good · tryLock 退避重试
while (true) {
  if (a.tryLock()) {
    try {
      if (b.tryLock()) {
        try { move(a, b); return; }
        finally { b.unlock(); }
      }
    } finally { a.unlock(); }
  }
  Thread.sleep(rnd()); // 退避，避免活锁
}
别把死锁换成活锁。如果所有线程"拿不到就立刻同步重试"，它们可能一直互相谦让、谁也不前进——这就是活锁。补救办法：重试前随机退避一小段时间（上例的 sleep(rnd())），打破对称性。这也是网络里 CSMA/CD 处理冲突的同一套思路。

五、能力三 · 公平锁：排队 vs 插队

构造函数 new ReentrantLock(true) 得到公平锁，false（默认）是非公平锁。区别只在一件事：当锁被释放的一瞬间，是让"排队最久的线程"拿到，还是允许"刚好路过、还没进队列"的新线程直接抢走。

非公平锁（默认） ── 锁一释放，谁先 CAS 抢到归谁 队列: [T2 等待中][T3 等待中] 新来的 T4 ──► 直接抢！ 可能插队成功 ↑ 排最久的 T2 可能反被 T4 抢先（吞吐高，但有饥饿风险） 公平锁（true） ── 锁一释放，先看队列里有没有人排队 队列: [T2 等待中][T3 等待中] 新来的 T4 ──► 乖乖去队尾排队 ↑ 严格 FIFO，T2 必然先拿到（无饥饿，但吞吐低）

为什么默认是非公平？因为公平的代价很高。当锁释放时，公平锁必须唤醒队首那个正在挂起的线程——而唤醒一个线程涉及上下文切换，是重量级操作。在这段唤醒的空档里，如果一个新线程恰好跑来且锁空闲，非公平锁让它直接拿走并用完，避免了那次昂贵的唤醒，吞吐量因此高得多。直观感受一下差距：

什么时候才真该用公平锁？只有当"持锁时间较长"或"请求间隔较大"、且你无法容忍某个线程被长期饿死时（例如需要保证响应时间的公平性）。绝大多数场景下非公平锁是更好的默认——这也是 ReentrantLock 和 synchronized（本身就是非公平的）不约而同的选择。

六、原理揭秘：一切都建立在 AQS 之上

把上面三大能力串起来的，是同一个引擎：AQS（AbstractQueuedSynchronizer，抽象队列同步器）。ReentrantLock 内部有一个继承自 AQS 的 Sync，几乎所有工作都委托给它。AQS 的模型极其简洁：

AQS 的两个核心零件 ──────────────────────────────── ① 一个 volatile int state ── 表示"锁被重入了几次" 0 = 没人持有；n = 持有线程重入了 n 次 ② 一个 FIFO 等待队列(CLH变体) ── 抢不到锁的线程在此排队挂起(park) 加锁 lock(): CAS 把 state 从 0 改成 1 成功 → 记下 exclusiveOwnerThread 若 owner 就是自己 → state++ (可重入！) 否则 → 进队列，LockSupport.park() 挂起 解锁 unlock(): state--，减到 0 才真正释放，唤醒队首线程 unpark() ─────────────────────────────────────────────────

看懂了这个模型，三大能力就全部"自然长出来"了，你会发现它们不是硬塞的特性，而是把 AQS 的能力暴露成了不同的 API：

  • 可重入：owner == 当前线程时 state++，与上一章内置锁"持有线程 + 计数器"完全同构——只是计数器从对象头 Mark Word 换成了 AQS 的 state 字段。
  • 可中断：排队挂起用的是 LockSupport.park()，它本就能被中断唤醒；lockInterruptibly 只是在被唤醒后检查中断标志并抛出异常。
  • 定时：tryLock(t) = 带 deadline 的 park（parkNanos）+ 到点就退出队列。
  • 公平/非公平：只差一行判断——加锁前是否调用 hasQueuedPredecessors()（前面是否有人排队）。公平锁问一句"有人排队吗"，非公平锁直接 CAS 抢。
建立联系：AQS 是整个 java.util.concurrent 的地基。ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock、ThreadPoolExecutor 里的 Worker……全都靠 AQS 的 state + 队列实现，区别只在于对 state 的解释不同（锁的重入次数 / 信号量的许可数 / 闭锁的计数）。这就是模板方法模式的教科书级应用：AQS 写好排队挂起的骨架，子类只填 tryAcquire/tryRelease。下一节的 ReadWriteLock 更巧妙——它把一个 32 位的 state 拆成高 16 位记读锁、低 16 位记写锁。

七、终极对比：到底该用哪把锁

把上一章的表格补完整。记住选择的顺序是：能不用锁就不用（用不可变对象 / volatile / 原子类）→ 需要互斥优先 synchronized → 确实需要下面这些高级能力，才升级到 ReentrantLock。

维度synchronizedReentrantLock
加解锁JVM 自动，出块/异常自动释放手动，必须 try...finally 配平
可重入✓（Mark Word 计数）✓（AQS state 计数）
可中断等待✗ 只能死等✓ lockInterruptibly()
定时/非阻塞尝试✗✓ tryLock() / tryLock(t)
公平锁✗ 只有非公平✓ 构造参数可选
条件队列单一 wait/notify多个 Condition，可分组唤醒
可否查询状态✗✓ isLocked() / getHoldCount() / getQueueLength()
忘记释放风险无有（漏 unlock 即泄漏）
性能有锁升级优化，现代 JVM 已很接近高竞争下略优，差距通常可忽略
面试高频陷阱汇总：
· "ReentrantLock 一定比 synchronized 快"是错的。JDK 6 起 synchronized 有偏向/轻量级锁升级，低竞争下几乎零开销；选 Lock 是为了功能，不是为了性能。
· 无参 tryLock() 会破坏公平性（barging），要公平请用 tryLock(0, unit)。
· Condition 的 await/signal 前必须先持有该 Lock，否则 IllegalMonitorStateException——和 wait/notify 必须在同步块里同理。
· unlock() 只能由持有线程调用，且加几次解几次。
一句话收束：ReentrantLock 的本质，是把"加锁"从 JVM 托管的关键字降维成一个由你调用方法的普通对象——正因为变成了方法调用，才能给它加上可中断（lockInterruptibly，让等锁的线程能被取消）、可定时（tryLock，拿不到就退避，用来规避死锁）、可公平（构造参数，用排队换取无饥饿）这三种内置锁给不了的能力；而这三者背后是同一个引擎 AQS——一个 volatile state 计数 + 一条 FIFO 等待队列。代价是你必须亲手把 unlock() 放进 finally。所以准则始终是：默认用 synchronized，只有真正需要这三大能力时，才升级到 ReentrantLock。
