ReadWriteLock 与 StampedLock

读操作明明互不影响，为什么还要排队？读写锁把"读读并行"这块被浪费的并发度还给了你，StampedLock 更进一步——连读锁都不想加。

一、为什么需要读写锁：普通互斥锁把"读读"也串行了

上一篇的 ReentrantLock（以及 synchronized）是独占锁：任何时刻只有一个线程能持有它，不区分你是读还是写。可现实里的很多共享数据——配置表、路由表、缓存、字典——是读多写少的：一天写一次，一秒读一万次。用独占锁保护它，就等于让那一万次互不干扰的读也乖乖排队，白白浪费了 CPU 的并行能力。

一个生活类比：独占锁像一间单人试衣间——不管你是进去换衣服（写）还是只想进去照个镜子（读），都得等前一个人出来。但"照镜子"这件事，十个人一起照本来毫无冲突。读写锁相当于把规则改成："只照镜子的可以一起进；只要有人换衣服，其他人（无论照镜子还是换衣服）都得在门外等。"——读与读共享，写与任何操作互斥，这就是读写锁的全部语义。

Java 用接口 java.util.concurrent.locks.ReadWriteLock 抽象这个概念，它只有两个方法，返回两把关联的锁：

ReadWriteLock.java一对读锁 / 写锁
public interface ReadWriteLock {
    Lock readLock();   // 读锁：可被多个读线程同时持有（共享锁）
    Lock writeLock();  // 写锁：同一时刻只能一个线程持有（独占锁）
}

标准实现是 ReentrantReadWriteLock。三条规则记牢，其余都是推论：

  • 读读共享：没有写线程时，多个读线程可同时持有读锁，完全并行。
  • 写写互斥：写锁是独占的，同一时刻只有一个写线程。
  • 读写互斥：只要有线程持有写锁，读锁就拿不到；只要有线程持有读锁，写锁也拿不到。
锁的兼容性矩阵（√ 可同时持有 / ✗ 互斥，必须等）： 别的线程持有 读锁 别的线程持有 写锁 我要拿读锁 √ 一起读 ✗ 等它写完 我要拿写锁 ✗ 等读都散了 ✗ 等它写完 一句话：只有"读+读"能并行，其余任意组合都互斥。

二、最小示例：一个读多写少的缓存

看一个典型场景——线程安全的缓存，读取命中率极高，偶尔刷新。用读写锁可以让所有读并行，只有刷新那一瞬间才独占。

ConfigCache.java读并行 / 写独占
class ConfigCache {
    private final Map<String,String> map = new HashMap<>();
    private final ReadWriteLock rw = new ReentrantReadWriteLock();
    private final Lock r = rw.readLock();
    private final Lock w = rw.writeLock();

    String get(String key) {
        r.lock();                 // 多个 get 可同时进来，不互相阻塞
        try { return map.get(key); }
        finally { r.unlock(); } // 解锁永远放 finally，异常也要释放（上一篇的铁律）
    }

    void put(String key, String val) {
        w.lock();                 // 写独占：此刻所有读都被挡在门外
        try { map.put(key, val); }
        finally { w.unlock(); }
    }
}
底层其实还是一把 AQS 锁。ReentrantReadWriteLock 内部只有一个 AQS 同步器（回忆将要学的 AQS：它用一个 int state 表示同步状态）。这里的巧思是把 32 位的 state 一劈两半：高 16 位记录读锁被持有的总次数，低 16 位记录写锁的重入次数。读锁用 AQS 的共享模式（acquireShared），写锁用独占模式（acquire）。所以读锁最多 65535 个持有者、写锁最多重入 65535 次——超了会抛 Error。
state(32位) = 高16位: 读锁持有总数 │ 低16位: 写锁重入数

三、两个必考特性：锁降级可以，锁升级会死锁

读写锁最容易翻车、也最爱考的地方，是"读锁和写锁之间能不能互相转换"。答案是不对称的：

  • 锁降级（write → read）：支持。持有写锁的线程，可以在还没释放写锁前先获取读锁，然后再释放写锁——平滑地从"写"过渡到"读"。
  • 锁升级（read → write）：不支持，会死锁。持有读锁的线程直接去申请写锁，会永远阻塞。
陷阱：为什么锁升级必然死锁？写锁要求"当前没有任何读锁被持有"。可如果两个线程 A、B 都持有读锁，然后都想升级成写锁——A 要等 B 放掉读锁、B 要等 A 放掉读锁，互相死等，谁也升不上去。即便只有一个线程持有读锁去申请写锁，它也在等"读锁数归零"，而那个读锁正攥在它自己手里、又不肯放，等于自己等自己，同样卡死。所以 ReentrantReadWriteLock 干脆禁止升级：持读锁时调 writeLock().lock() 直接挂起。
✗ 锁升级：持读锁直接抢写锁 → 卡死
r.lock();                 // 持有读锁
try {
    if (needUpdate) {
        w.lock();         // ★ 死锁！写锁在等
                          //   读锁归零，而读锁在自己手里
        // ...永远走不到这里
    }
} finally { r.unlock(); }
✓ 先放读锁，再抢写锁，拿到后可降级
r.lock();
if (needUpdate) {
    r.unlock();           // ① 必须先彻底释放读锁
    w.lock();             // ② 再申请写锁
    try {
        // 重新检查！释放到拿到之间别人可能已更新
        if (needUpdate) { /* 写入 */ }
        r.lock();         // ③ 降级：释放写锁前先抓读锁
    } finally { w.unlock(); }  // 释放写锁，仍持读锁
}
try { /* 读数据 */ } finally { r.unlock(); }
锁降级到底解决了什么？这段"先拿读锁 → 发现要更新 → 换写锁改 → 改完降回读锁 → 用数据"的写法，正是 JDK 官方 CachedData 例子的模板。降级的价值在于：写完后不必完全释放锁再重新抢读锁，避免了"写锁一放，别的写线程立刻插进来又把数据改了"的窗口——降级保证了"我刚写的值，我自己接着读时一定还在"。注意 ③ 中"释放读锁到拿到写锁之间"要重新检查条件（double-check），因为这段缝隙里别的线程可能已经把活干了。

四、公平性与写饥饿：读写锁没那么完美

ReentrantReadWriteLock 和 ReentrantLock 一样，构造时可选公平 / 非公平（默认非公平）。但读写锁多了一个独占锁没有的隐患——写线程饥饿。

陷阱：非公平模式下，源源不断的读线程会饿死写线程。设想读请求像潮水一样涌来，读锁一直有人持有、读计数从没归零，那么排队等写锁的线程就永远等不到"读锁数为 0"的那一刻。JDK 为此在非公平实现里加了一个小心机：当队列头部正好是一个等待中的写线程时，后来的读线程不再"插队"抢读锁（readerShouldBlock 会返回 true），给写线程一点机会。即便如此，写饥饿在读极密集时依然可能发生。公平模式能杜绝饥饿，但吞吐更低。这也正是 StampedLock 登场的动机之一。
陷阱：读多写少不代表读写锁一定更快。读写锁的读锁本身也是要抢的——每个读线程都要对同一个 AQS 的 state 做 CAS 修改读计数。读线程越多，这个共享计数就成了缓存行争抢的热点（多核 CPU 反复使对方缓存失效）。当临界区极短、读并发极高时，读写锁在读锁上的簿记开销可能抵消甚至超过并行带来的收益，反而比一把简单的 synchronized 还慢。永远用压测数据说话，别想当然。

五、StampedLock：连读锁都懒得加的"乐观读"

Java 8 引入 StampedLock，正是为了治读写锁的两个病：写饥饿 和 读锁的共享计数开销。它的核心武器是一种全新姿势——乐观读（optimistic read）。

它提供三种访问模式，每次加锁都返回一个 long 类型的"邮戳 / 版本号"（stamp），解锁时要把这个 stamp 交回去：

模式获取方式本质会阻塞谁
写锁writeLock()独占锁（悲观）阻塞其他所有写和悲观读
悲观读锁readLock()共享锁（悲观）与写互斥，读读共享
乐观读tryOptimisticRead()根本不加锁，只拿版本号不阻塞任何人，包括写
"乐观读"到底是什么？它压根不获取锁——只向 StampedLock 要一个当前的版本邮戳，然后你放心大胆地去读共享字段。读完，再问一句 validate(stamp)："我读这段时间里，有没有写操作发生过？" 没有（版本号没变）→ 恭喜，你这次读到的是一致的快照，零锁开销完成；有 → 说明数据可能被改脏了，回退到老老实实的悲观读锁再读一遍。这和 CAS、乐观锁、数据库的 MVCC 版本号是同一种思想：先假设不冲突、干完再验证，冲突了才补偿。读多写极少时，validate 几乎总是成功，读操作快到接近无锁。
乐观读的一次典型流程： long stamp = sl.tryOptimisticRead(); ← 拿版本号，不加锁（不阻塞任何写） x' = x; y' = y; ← 把共享字段快照到局部变量 ┌─────────────────────────────────────────────┐ │ 期间若有写线程 writeLock() 改了数据，版本号会变 │ └─────────────────────────────────────────────┘ if (sl.validate(stamp)) { → true : 没人写过，x'/y' 就是一致快照，直接用 ✔ → false: 有人写过，快照可能不一致 stamp = sl.readLock(); ← 回退：改用悲观读锁，阻塞等写完，重读一遍 ...重新读 x' y'... sl.unlockRead(stamp); }
Point.java（改编自 StampedLock JavaDoc）乐观读的标准姿势
class Point {
    private double x, y;
    private final StampedLock sl = new StampedLock();

    void move(double dx, double dy) {   // 写：独占
        long stamp = sl.writeLock();
        try { x += dx; y += dy; }
        finally { sl.unlockWrite(stamp); }
    }

    double distanceFromOrigin() {         // 读：乐观
        long stamp = sl.tryOptimisticRead();  // ① 拿版本号，不加锁
        double curX = x, curY = y;          // ② 先把字段拷进局部变量（关键！）
        if (!sl.validate(stamp)) {          // ③ 校验：这期间有写发生吗？
            stamp = sl.readLock();          // ④ 有→回退成悲观读锁
            try { curX = x; curY = y; }      //    重新读一次（此刻保证一致）
            finally { sl.unlockRead(stamp); }
        }
        return Math.sqrt(curX * curX + curY * curY);
    }
}
陷阱：乐观读必须"先拷贝到局部变量，再 validate"，顺序不能反。为什么②要把 x、y 先读进 curX、curY？因为乐观读不阻止写线程，如果你直接 return Math.sqrt(x*x + y*y) 边算边读共享字段，在算 x*x 和读 y 之间写线程可能改了 y，你就会得到一个"半旧半新"的撕裂快照（x 是旧的、y 是新的），validate 也救不了——因为你根本没留住那一刻的值。正确姿势永远是：先把需要的字段一次性拷进局部变量 → 再 validate → validate 通过才使用这些局部变量。validate 失败就整段丢弃、走悲观路径重来。

六、StampedLock 的三大硬约束（用之前必须知道）

陷阱一：不可重入！这是与 ReentrantReadWriteLock 最扎眼的区别。同一个线程已经持有 StampedLock 的写锁，再调一次 writeLock() 会自己把自己锁死。因为它不记录"锁的持有者是谁、重入了几层"，只认版本号。所以严禁在持锁期间调用可能再次加锁的方法（比如递归、回调）。
陷阱二：不支持 Condition，且对中断不友好。StampedLock 没有 newCondition()，无法做"等待/通知"式的条件协作（需要 Condition 请用 ReentrantLock，正是下一篇的主题）。此外它默认的 readLock()/writeLock() 不响应中断——线程在等锁时被 interrupt() 也不会醒来。要可中断，得用 readLockInterruptibly() / writeLockInterruptibly()。更坑的是：在获取悲观锁的过程中若线程被中断，某些版本可能导致 CPU 空转，官方文档明确警告了这点。
陷阱三：它没实现 Lock/ReadWriteLock 接口，也不能当 synchronized 那样跨方法随意用。StampedLock 是个独立的类，靠传递 long stamp 工作，不能直接塞进期望 Lock 类型的 API。若确实需要接口兼容，它提供了 asReadLock() / asWriteLock() / asReadWriteLock() 三个视图适配器（但这些视图同样不可重入）。它还支持锁模式转换：tryConvertToWriteLock / tryConvertToReadLock / tryConvertToOptimisticRead，可用来实现"读着读着发现要写"的原子升级尝试。

建立联系：一条清晰的"降低读成本"演进线。从 synchronized/ReentrantLock（读也独占）→ ReadWriteLock（读读共享，但读锁仍要抢共享计数）→ StampedLock（乐观读连计数都不抢，退化成一次版本号比较）。每一步都在把读操作的成本往下压，代价是使用约束越来越多、越来越难写对。这与并发容器篇的思路一脉相承——CopyOnWriteArrayList 让读变成一次纯 volatile 读、ConcurrentHashMap 用 CAS 缩小锁粒度，本质都是"为读多写少的现实，把读路径优化到极致"。乐观读与 CAS、数据库乐观锁（version 字段）、MVCC 更是同一套"先干后验"的世界观。
一句话收束：读写锁把独占锁浪费掉的"读读并行"还了回来——核心语义只有一条：读读共享，读写/写写互斥；底层是一个 AQS 把 32 位 state 劈成"高16位读计数 + 低16位写重入"。两条铁律必背：锁降级（write→read）可以，锁升级（read→write）会死锁。它的软肋是非公平下写饥饿与读锁共享计数的争抢开销——于是 Java 8 用 StampedLock 的乐观读回应：不加锁、只拿版本号，读完 validate，没被写过就白赚一次无锁读，被写过再回退悲观读锁。但乐观读要牢记"先拷贝局部变量，后 validate"，且 StampedLock 不可重入、无 Condition、默认不可中断、只有非公平。选型口诀：需要条件协作或写为主 → ReentrantLock；读多写少、临界区不算太短 → ReentrantReadWriteLock；读极多写极少、临界区短且不需重入 → StampedLock。任何选择，最终都要用压测验证。
