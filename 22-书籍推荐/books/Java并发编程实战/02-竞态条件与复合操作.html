竞态条件与复合操作

count++ 为什么会"丢更新"？——从"一步代码其实是三步"讲透竞态的根源与修复。

一、先看一个会"闹鬼"的计数器

下面这个类只有一个字段和一个方法，看起来朴实无害：每调一次 getNext() 就返回一个"下一个"的序号，本该是 0、1、2、3……全局唯一、单调递增。

UnsafeSequence.javaJCP 清单 2.1
public class UnsafeSequence {
    private int value;

    /** 期望：每次调用返回一个唯一的、递增的序号 */
    public int getNext() {
        return value++;   // 看似一步，实为"读-改-写"三步
    }
}

单线程下它永远正确。可一旦多个线程同时调 getNext()，你会偶尔看到两个线程拿到同一个序号，或者最终计数比调用次数少。诡异的是：它不是每次都错，而是时灵时不灵——这恰恰是并发 Bug 最折磨人的特征。这种"正确与否取决于运气"的现象，就叫竞态条件（race condition）。

竞态条件  =  正确结果 依赖于 多个线程执行的 相对时序
生活类比：两个人共用一个记账本，都想"在余额上记一笔"。正确的做法是"看一眼余额 → 加钱 → 写回"这一整套动作独占账本。但如果甲刚看完余额（100）还没落笔，乙也看了同一个 100 并抢先写下 150，等甲回过神按自己记的 100 写下 120——乙的那笔就凭空蒸发了。谁先谁后纯看巧合，这就是竞态。

二、复合操作：value++ 其实是三步

问题的根子在于：value++ 在源码里是一个字符，在 CPU 眼里却是三个独立步骤。它是典型的"读-改-写"（read-modify-write）操作。反编译成字节码就一目了然：

value++ 编译后其实是三条独立指令： getfield value ① 读：把 value 从堆内存读进操作数栈 iadd (+ 1) ② 改：在栈上算出新值 (旧值 + 1) putfield value ③ 写：把新值写回堆内存 ↑ 三步之间的任意缝隙，线程都可能被 OS 抢占、切走

由多个独立操作组成、必须整体不可分割地执行才能保持正确的操作序列，书里称为复合操作（compound action）。"读-改-写"就是最经典的复合操作。既然它不是一步完成的，操作系统的抢占式调度就可能在任意两步之间把 CPU 切给别的线程。于是两个线程的三步交错在一起：

时刻 线程 A 线程 B value 内存 t1 读 value → 9 9 t2 读 value → 9 9 t3 加 1 → 10 9 t4 加 1 → 10 9 t5 写回 value = 10 10 t6 写回 value = 10 10 两个线程各自都执行了一次 +1，结果却只 +1 —— 丢失了一次更新！
面试高频陷阱：i = i + 1 和 i++ 没有本质区别。很多人以为 ++ 是"自增指令"所以原子，其实两者都会被编译成"读-改-写"三步。更进一步：即便给字段加上 volatile（下一节讲），count++ 依然是竞态——volatile 只保证"每次读到的是最新值"（可见性），却管不住"读完到写回之间被别人插队"（原子性）。可见性 ≠ 原子性，这是初学者最容易混淆的一刀。

三、竞态条件的两大家族

书里把竞态条件归为两类最常见的模式。上面的计数器属于第一类"读-改-写"；还有一类更隐蔽，叫"先检查后执行"（check-then-act）：你先观察到某个条件成立，再基于这个"过时的观察"去行动，而在观察和行动之间条件早已改变。

家族行为模式典型代码出问题的窗口
读-改-写
read-modify-write		读出旧值 → 基于旧值算新值 → 写回count++、i = i + 1、balance -= amount读与写之间被别的线程插入了写
先检查后执行
check-then-act		观察一个条件 → 基于观察去动作延迟初始化、"不存在才创建"、单例、手写的 putIfAbsent检查与执行之间条件被改变

"先检查后执行"的经典受害者是延迟初始化（lazy initialization）——想省内存，等第一次用到才创建对象：

LazyInitRace.javaJCP 清单 2.2 · 先检查后执行的竞态
public class LazyInitRace {
    private ExpensiveObject instance = null;

    public ExpensiveObject getInstance() {
        if (instance == null)                  // ① 检查：此刻确实是 null
            instance = new ExpensiveObject(); // ② 执行：两个线程可能都走到这
        return instance;                        // 结果：可能返回两个不同实例！
    }
}

若线程 A、B 几乎同时进来，都在 A 完成赋值前读到 instance == null，就会各自 new 一个对象。对于"应当全局唯一"的单例，这是灾难；即便不要求唯一，两个调用方各自持有不同实例，也会让"共享缓存"之类的语义彻底失效。这也正是后面章节要讲"双重检查锁（DCL）"和"安全初始化"的动机来源。

与 Spring 串联：Spring 的 Bean 默认是单例作用域（singleton）——整个容器只有一个实例，被所有并发请求共享。如果你在一个单例 Bean 里放一个可变字段并做 this.count++ 或"检查再赋值"，那就是把上面的竞态直接搬进了生产环境。这就是为什么"Spring 单例 Bean 要保持无状态"是一条铁律：无共享可变状态，自然无竞态。

四、怎么修：让复合操作重新"合"成一步

竞态的本质是"该原子的一组操作被拆开了"。修复思路只有一条：把复合操作重新变成不可分割的原子操作。当只有单个变量需要维护时，最轻量的方案是用 java.util.concurrent.atomic 里的原子变量，而不用加锁：

✗ Bad · 裸露的读-改-写
private long value;

public long getNext() {
    return value++;
    // 三步可被打断 → 丢更新
}
✓ Good · AtomicLong 原子自增
private final AtomicLong value
        = new AtomicLong(0);

public long getNext() {
    return value.getAndIncrement();
    // CAS 保证读-改-写不可分
}
原子变量凭什么"一步到位"？AtomicLong.getAndIncrement() 底层是 CAS（Compare-And-Swap）：它带着"我读到的旧值"去写，只有当内存仍然是那个旧值时才写成功，否则重试。这条"比较并交换"由 CPU 单条指令（x86 的 lock cmpxchg）保证原子，硬件层面就杜绝了"读完被插队"。JDK 通过 Unsafe / VarHandle 调到这条指令——这也是后面"Atomic 类与 CAS 原理"一节的伏笔。

如果只是单个变量，也可以用 synchronized 把三步锁成一段临界区（下一节详解）；但要在只需保护一个计数器时用锁，就有点杀鸡用牛刀——原子变量更轻、无阻塞、吞吐更高。选型对照：

方案适用场景代价 / 取舍
AtomicLong / AtomicInteger只维护单个数值型状态无锁、非阻塞，高竞争下靠自旋重试，最轻量
synchronized / Lock需要把多个相关变量作为整体保护能覆盖任意复合逻辑，但有阻塞与上下文切换开销
ConcurrentHashMap.putIfAbsent / computeIfAbsent"不存在才放入"这类 check-then-act把复合操作打包成库提供的单个原子方法，最省心

五、更隐蔽的陷阱：单个变量原子 ≠ 整体不变式原子

学会用原子变量后，很多人会踩下一个坑："我每个变量都用了 Atomic，为什么还是错？" 看这个缓存"上一次分解质因数结果"的例子——它用了两个原子引用，每个单独看都无懈可击：

UnsafeCachingFactorizer.javaJCP 清单 2.6 · 两个原子变量仍不安全
public class UnsafeCachingFactorizer {
    private final AtomicReference<BigInteger>   lastNumber  = new AtomicReference<>();
    private final AtomicReference<BigInteger[]> lastFactors = new AtomicReference<>();

    public BigInteger[] service(BigInteger i) {
        if (i.equals(lastNumber.get()))
            return lastFactors.get();       // 命中缓存：但拿到的可能是"别人的"因数
        BigInteger[] factors = factor(i);
        lastNumber.set(i);              // ← 这两次 set 之间，线程可能被切走，
        lastFactors.set(factors);       //   导致 number 与 factors 对不上号
        return factors;
    }
}

这里真正要守护的不变式（invariant）是："lastFactors 必须是 lastNumber 的质因数。" 这个约束横跨两个变量。当线程 A 刚 set(lastNumber) 还没 set(lastFactors) 时被切走，线程 B 进来看到新的 number 却配着旧的 factors——校验通过，返回一组张冠李戴的错误结果。每个变量各自的更新都是原子的，可它们之间的"关系"没被原子地维护。

核心误区：把"每个变量原子"当成"整体原子"。当一个不变式涉及多个变量时，这些变量不是彼此独立的——更新一个而不更新另一个，就会出现"中间态"被别的线程看到。用一堆独立的原子变量拼不出跨变量的原子性。这里正确的做法是给整个 service 方法加锁（或用不可变对象把 number+factors 打包成"一个"状态一起替换），让相关状态在同一个原子操作里一起变。
书里的黄金准则："要保持状态一致性，就要在单个原子操作中更新所有相关的状态变量。" 判断标准是"不变式牵扯了哪些变量"——凡是被同一个不变式绑在一起的变量，必须被同一把锁、同一个原子步骤统一保护，而不能各管各的。
一句话收束：竞态条件的本质是"正确性依赖线程交替的时序"，而它的根源几乎总是复合操作被拆开执行——无论是"读-改-写"（count++）还是"先检查后执行"（延迟初始化）。修复的唯一思路是让这组操作重新原子化：单个变量用 Atomic（CAS），跨变量的不变式则用锁或不可变对象把相关状态打包成一步一起更新——记住，"每个变量都原子"绝不等于"整体原子"。
