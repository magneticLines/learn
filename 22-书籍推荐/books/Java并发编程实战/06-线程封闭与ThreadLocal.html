线程封闭与 ThreadLocal

对付并发最省力的一招，不是把锁加得更严，而是干脆让数据"不共享"——没有共享，就没有竞争。

一、换个思路：与其保护共享，不如取消共享

前面几篇一直在讲"共享的可变状态"如何用同步来保护。但还有一条更彻底的路：让数据根本不被多个线程访问。数据只被单个线程独占，即使它可变，也不需要任何同步——这就是线程封闭（Thread Confinement）。

一个生活类比：公司会议室（共享资源）要排队预定、加锁使用；但你自己工位上的水杯（线程封闭）从来不用跟别人协调——因为它只属于你。线程封闭就是把"公用水杯"换成"人手一个私人水杯"：代价是每人多一个杯子（内存），换来的是彻底不用排队（无锁）。

回忆上一篇「发布与逸出」：线程安全问题的前提是对象被发布给了其他线程。线程封闭的做法恰恰相反——刻意不发布，把对象牢牢圈在一个线程内部。这也是 GUI 框架（Swing/JavaFX 的事件分发线程 EDT）、以及绝大多数数据库连接池管理连接的核心手段。

线程安全 = 消灭共享 | 消灭可变 | 正确同步

三条路里，"消灭共享"（线程封闭）和"消灭可变"（不可变对象，下一篇）都能绕开锁；只有实在做不到时，才退回到"正确同步"。优先级从左到右递减。

二、线程封闭的三种境界

Brian Goetz 把线程封闭分成三档，从"靠自觉"到"有语言机制兜底"，可靠性递增：

类型怎么做到封闭可靠性典型场景
Ad-hoc
（临时/约定式）		纯靠程序员自觉，没有任何机制强制，全凭"我保证不把它传出去"最脆弱，一次疏忽就破功用 volatile 保证只有单线程写、其他线程只读
栈封闭
（Stack Confinement）		对象只存在于局部变量里，天然被限制在当前线程的调用栈中较可靠，只要引用不逸出方法内 new 出来、用完就丢的临时对象
ThreadLocal由 JDK 类库提供机制，为每个线程维护一份独立副本最可靠，有语言级支撑连接、事务上下文、SimpleDateFormat

栈封闭：其实你天天在用

局部变量为什么天生线程安全？因为每个线程有自己独立的 JVM 栈，方法的每次调用都会压入一个新的栈帧，局部变量表就在栈帧里。不同线程调用同一个方法，各自的局部变量互不相干——这正是操作系统"每线程一个栈"在 JVM 里的映射。

✗ 引用逸出，栈封闭被破坏
class Stats {
  private List<Integer> cache; // 成员字段

  void collect() {
    // 本想局部使用，却存进了字段
    cache = new ArrayList<>();
    for (int i = 0; i < 100; i++)
      cache.add(i);
    // cache 现在被多线程共享 → 危险
  }
}
✓ 只用局部变量，封闭在栈上
class Stats {
  int collect() {
    // list 是局部变量，never 逸出
    List<Integer> list =
        new ArrayList<>();
    for (int i = 0; i < 100; i++)
      list.add(i);
    // ArrayList 非线程安全也没关系，
    // 它只活在当前线程的栈帧里
    return list.size();
  }
}
陷阱一：基本类型天然栈封闭，对象引用不一定。int、long 这类基本类型局部变量的值直接躺在栈里，别人绝无办法拿到。但对象引用是局部的、对象本身却在堆上——只要你一不小心把这个引用赋给字段、塞进静态集合、或作为返回值/回调传出去，栈封闭立刻失效。"逸出"（escape）就是这么发生的（见上一篇）。守则：非线程安全对象要留在方法内部，绝不让它的引用跑出去。

三、ThreadLocal：给每个线程发一份私有副本

栈封闭的局限是"活不过一次方法调用"。如果我想要一个变量在整个线程生命周期内都存在、但又不和别的线程共享呢？比如"当前登录用户""当前数据库连接"——这就是 ThreadLocal 的舞台。

心智模型（务必建立正确直觉）：很多人误以为 ThreadLocal 是"一个容器，里面装了 N 份数据，按线程分发"。恰恰相反。真正装数据的是每个 Thread 对象自己身上的一张 Map；ThreadLocal 只是这张 Map 的key。你调 get()，它拿"当前线程"这个 Map，再用"自己"当 key 去取值。所以同一个 ThreadLocal 实例在不同线程里取到不同的值——因为 Map 根本不是同一张。
ThreadLocalDemo.java三个 API：set / get / remove
public class UserContext {
    // 通常声明为 static final：一个 ThreadLocal 实例服务所有线程
    private static final ThreadLocal<String> CURRENT_USER =
            new ThreadLocal<>();

    // 也可用 withInitial 提供默认值（惰性初始化，每线程首次 get 时调用）
    private static final ThreadLocal<SimpleDateFormat> SDF =
            ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

    public static void login(String user) { CURRENT_USER.set(user); }  // 存入当前线程
    public static String get()  { return CURRENT_USER.get(); }        // 取当前线程的那份
    public static void clear() { CURRENT_USER.remove(); }              // 用完必须清（见陷阱）
}
为什么它能"跨方法传参"却不用改签名？ThreadLocal 的杀手锏是隐式传参。想象一个 Web 请求：Controller → Service → DAO 层层调用，如果每层方法都要多带一个 currentUser 参数，签名会被污染得一塌糊涂。把它放进 ThreadLocal，任何一层需要时直接 get()——因为 Servlet 容器是"一个请求一个线程"处理的，整条调用链跑在同一个线程上，自然读到同一份数据。这就是 Spring 事务、Spring Security 上下文的底层套路。

四、揭开源码：数据其实存在 Thread 身上

光说不练不踏实。翻开 JDK 源码你会发现，Thread 类里有个包私有字段：

JDK 源码（简化）Thread.java / ThreadLocal.java
// —— Thread.java ——
class Thread {
    // 每个线程自带一张 map，key 是 ThreadLocal，value 是你存的值
    ThreadLocal.ThreadLocalMap threadLocals = null;
}

// —— ThreadLocal.java ——
public T get() {
    Thread t = Thread.currentThread();     // ① 拿当前线程
    ThreadLocalMap map = getMap(t);          // ② 取它身上的 map（就是 t.threadLocals）
    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this);  // ③ 用 this(当前 ThreadLocal) 当 key 取值
        if (e != null) return (T) e.value;
    }
    return setInitialValue();               // 没有则走 initialValue()
}

ThreadLocalMap getMap(Thread t) { return t.threadLocals; }
每个 Thread 对象内部 ↓ ThreadLocal 实例（只是 key，不存数据） ┌──────────────────────────────┐ │ Thread-1 │ ┌─────────────────┐ │ threadLocals ──► Map { │ │ tlUser (key A) │───┐ │ keyA(弱) → "Alice" │◄───────┤ tlConn (key B) │ │ 同一个 ThreadLocal │ keyB(弱) → conn#1 │ └─────────────────┘ │ 实例被所有线程共用 │ } │ │ 做 key └──────────────────────────────┘ │ ┌──────────────────────────────┐ │ │ Thread-2 │ │ │ threadLocals ──► Map { │◄─────────────────────────────┘ │ keyA(弱) → "Bob" │ ← 同一个 keyA，在 T2 的 map 里对应 "Bob" │ } │ 两张 map 相互独立，所以数据不共享 └──────────────────────────────┘ Entry extends WeakReference<ThreadLocal> ：key 是弱引用，value 是强引用（关键！见陷阱二）
两个源码彩蛋。其一，ThreadLocalMap 不是 HashMap，而是自己实现的一张 开放寻址（线性探测） 哈希表，冲突时往后找下一个空槽，而非拉链。其二，它的哈希用了一个魔数 0x61C88647（黄金分割数对应的 32 位整数），每加一个 ThreadLocal 就把哈希值累加这个常量——这能让多个 ThreadLocal 的 key 在表里近乎均匀地散开，几乎不碰撞。这是"斐波那契散列"的经典应用，比通用哈希更贴合"key 数量少而固定"的场景。

五、经典实战：驯服 SimpleDateFormat

SimpleDateFormat 是面试和线上事故的常客：它非线程安全（内部用了一个可变的 Calendar 字段做中间状态），多线程共享同一个实例会解析出错乱的日期、甚至抛 NumberFormatException。三种解法对比：

✗ 静态共享一个实例
// 多线程共用，内部 Calendar
// 状态被并发践踏 → 结果错乱
static final SimpleDateFormat SDF =
    new SimpleDateFormat("yyyy-MM-dd");

String fmt(Date d) {
    return SDF.format(d); // 线程不安全!
}
✓ ThreadLocal 每线程一份
static final ThreadLocal<SimpleDateFormat>
  SDF = ThreadLocal.withInitial(
      () -> new SimpleDateFormat(
                "yyyy-MM-dd"));

String fmt(Date d) {
    // 复用本线程实例，无锁无竞争
    return SDF.get().format(d);
}
方案线程安全性能点评
共享一个 SimpleDateFormat✗ 错误—典型 bug，禁用
每次 new 一个✓差（频繁创建/GC）简单但浪费
加锁共享一个✓差（串行化）把并发变成了串行
ThreadLocal 缓存✓好（复用+无锁）空间换安全的经典范式
DateTimeFormatter（JDK8+）✓ 本身不可变最好新代码首选，直接免疫
建立联系：ThreadLocal 的本质是用空间换取"不共享"——每线程一份副本，牺牲内存，换来彻底的无锁。而 JDK8 的 DateTimeFormatter 走的是另一条路：把对象做成不可变的（下一篇「不变性」的主题），不可变对象天生可共享。同一个问题，"线程封闭"和"不可变"是两把方向相反却都有效的钥匙。能用不可变就别用 ThreadLocal，因为后者有内存泄漏的坑——这就引出下一节。

六、陷阱二：内存泄漏——弱引用的 key 与强引用的 value

回看上面那张图的最后一行：Entry 的 key 是弱引用，value 是强引用。这个设计既是"防泄漏的努力"，又是"泄漏的根源"，必须讲透。

泄漏是怎么发生的（尤其在线程池里）： 栈上强引用 ThreadLocal实例 Thread(存活很久) ──► ThreadLocalMap X ───────► tlObj ◄╌╌╌弱引用╌╌╌ Entry.key │ (方法结束，X 消失) Entry.value ────────► 大对象(强引用) ▲ ① tlObj 只被 Entry.key 弱引用 → 下次 GC 时 key 被回收 → key = null │ ② 但 value 仍被 Entry 强引用，且 Thread 一直活着(池化线程不销毁) │ ③ 于是出现一条"够不着 key、却回收不掉 value"的僵尸链 ──────────────┘ → value 永远留在内存里 = 内存泄漏

为什么 JDK 要把 key 设成弱引用？这其实是一种补救：如果 key 是强引用，那么只要 Thread 活着，ThreadLocal 实例和它的 value 就都别想被回收，泄漏更严重。设成弱引用后，至少 ThreadLocal 实例本身能被 GC。而且 ThreadLocal 在 get/set/remove 时会顺手清理一些 key 已为 null 的"脏 Entry"（源码里的 expungeStaleEntry）。但这只是"顺手"——如果之后你再也不碰这个 ThreadLocal，脏 Entry 的 value 就永远清不掉。

陷阱（高频面试题）：线程池 + ThreadLocal 不 remove = 定时炸弹。普通线程用完就死，map 随之被回收，泄漏有限。但线程池里的线程会被复用、几乎永不销毁，它身上的 threadLocals map 会一直累积。两个后果：（1）内存泄漏——旧 value 堆积；（2）脏数据——上一个任务 set 的值，下一个复用同一线程的任务 get() 会读到（比如串了别人的登录态！这是安全事故）。铁律：ThreadLocal 一定在 finally 里 remove()。
✗ 忘记 remove（线程池中）
void handle(Request req) {
  CURRENT_USER.set(req.user());
  service.process();
  // 方法结束但线程回池，
  // value 还挂在 map 上：
  //  · 内存泄漏
  //  · 下个任务可能读到 req.user()
}
✓ try / finally 兜底清理
void handle(Request req) {
  CURRENT_USER.set(req.user());
  try {
    service.process();
  } finally {
    CURRENT_USER.remove(); // 必须!
    // 无论是否异常都清掉当前线程副本
  }
}
Spring 是怎么做的？它把 remove 藏在了框架里，你平时感觉不到。DispatcherServlet 处理完请求会在 finally 里调 RequestContextHolder.resetRequestAttributes() 清掉 RequestAttributes 的 ThreadLocal；Spring 事务在提交/回滚后由 TransactionSynchronizationManager 解绑线程上的连接；Spring Security 的过滤器链末尾会 SecurityContextHolder.clearContext()。它们都遵守同一条纪律：谁 set，谁负责在请求边界 remove。你自己写 ThreadLocal 时，也要模仿这种"边界清理"的模式。

七、陷阱三：ThreadLocal 传不到子线程 / 线程池

ThreadLocal 是线程私有的，所以父线程 set 的值，子线程 get() 不到。想传给子线程，得用 InheritableThreadLocal——它在创建子线程时（new Thread()）把父线程的值拷贝过去。

但它对线程池无效！InheritableThreadLocal 只在线程被创建的那一刻继承。线程池的线程是预先创建、反复复用的：它诞生时的"父线程"是初始化池子的那个线程，跟你后来提交任务的线程八竿子打不着。于是你在业务线程 set 的值，池里的工作线程根本继承不到。要在线程池里做上下文传递（如全链路 traceId），得用阿里开源的 TransmittableThreadLocal（TTL）——它在任务提交时而非线程创建时抓取快照，靠包装 Runnable/线程池来把上下文"带过去"。
方案数据隔离能传子线程兼容线程池
ThreadLocal每线程独立✗ 不能✓（但须手动 remove）
InheritableThreadLocal创建时拷贝父值✓ 新建线程时✗ 复用线程会失效
TransmittableThreadLocal（TTL，阿里）提交任务时快照✓✓ 专为线程池设计

八、什么时候该用 ThreadLocal

ThreadLocal 不是银弹，用错地方会变成"隐式全局变量"，让代码难以追踪。一条决策链帮你判断：

这份数据需要跨方法/跨层传递吗？ │ ├─ 否 → 用局部变量（栈封闭）就够了，别上 ThreadLocal │ └─ 是 → 它必须"每线程一份、互不干扰"吗？ │ ├─ 否，数据不可变 → 直接共享(不可变对象)，更简单 │ └─ 是 → 用 ThreadLocal，且牢记： · static final 声明 · 请求/任务边界 finally remove() · 跨线程池传递 → 换 TransmittableThreadLocal
一句话收束：线程封闭是对付并发的"上策"——没有共享，就没有竞争，也就不需要锁。它有三层：靠自觉的 Ad-hoc、天然可靠的栈封闭（局部变量，源于每线程独立的栈）、以及机制兜底的 ThreadLocal（用空间换隔离，本质是"数据存在每个 Thread 自己的 ThreadLocalMap 里，ThreadLocal 只是 key"）。但 ThreadLocal 有两颗地雷：value 强引用 + 线程池复用会导致内存泄漏和脏数据，务必在 finally 里 remove()；它传不到线程池的工作线程，跨线程池场景要用 TransmittableThreadLocal。能用不可变对象解决的，就别用 ThreadLocal——这正是下一篇「不变性」要展开的另一把钥匙。
