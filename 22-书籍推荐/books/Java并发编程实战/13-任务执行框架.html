任务执行框架：Executor 与 ThreadPoolExecutor

把"线程的生命周期管理"从"任务的业务逻辑"里彻底剥离——这是 java.util.concurrent 送给你的第一份大礼。

一、为什么不能"每个任务开一个线程"

刚学多线程时，最直觉的写法是：来一个请求，new Thread(task).start()。它在玩具程序里能跑，可一旦放到真实服务器上，就是一颗定时炸弹。问题不在"用了线程"，而在于把"要做什么"（任务）和"用什么资源去做"（线程）死死绑在了一起。

✗ Bad · thread-per-task
while (true) {
  Socket conn = socket.accept();
  // 每来一个连接就 new 一个线程
  new Thread(() -> handle(conn))
        .start();
}
// 高峰期瞬间创建上万线程：
// 内存爆掉 / 疯狂上下文切换 / OOM
✓ Good · 交给线程池
ExecutorService pool =
    Executors.newFixedThreadPool(100);
while (true) {
  Socket conn = socket.accept();
  // 只提交"任务"，线程复用交给池
  pool.execute(() -> handle(conn));
}
// 线程数被封顶，超出的排队，平稳降级

"每任务每线程"的三宗罪，正是线程池要解决的问题：

  • 创建/销毁开销大：起一个线程要向 OS 申请栈内存（默认约 1MB）、注册内核线程，纳秒级的任务配上毫秒级的建线程成本，得不偿失；
  • 资源无上限：线程数随请求量线性膨胀，几千个线程就能把内存和 CPU 调度拖垮，最终 OutOfMemoryError 或整机卡死；
  • 缺乏管理：没法统一控制并发度、没法优雅关闭、没法监控、没法复用。
类比：餐厅雇服务员。"每桌客人临时雇一个服务员，客人走了就辞退"——招聘和辞退的成本比端菜还高，人一多老板也管不过来。合理的做法是固定雇 N 个服务员（线程），客人（任务）来了排号，谁空了谁去接。这个"固定的一批服务员 + 排队叫号"的机制，就是线程池。

二、Executor：一行接口，一次思想解耦

Executor 是整个框架的地基，而它简单到只有一个方法：

java.util.concurrent.ExecutorJDK 源码 · 整个框架的原点
public interface Executor {
    void execute(Runnable command);
}

别小看这一行。它把"提交任务"和"如何执行任务"划成两个世界：调用方只管 execute(task)，至于用新线程、用池里的旧线程、还是干脆在当前线程里同步跑，全由 Executor 的实现说了算。这正是生产者—消费者模式的教科书应用：提交任务的代码是生产者，执行任务的线程是消费者，中间用一个队列解耦。想调优？换个 Executor 实现即可，业务代码一行不动。

Executor 框架的分层（从抽象到具体） ┌──────────────────────────────────────────────┐ │ Executor void execute(Runnable) │ ← 最小抽象：只管"执行" └───────────────▲──────────────────────────────┘ │ extends ┌───────────────┴──────────────────────────────┐ │ ExecutorService submit / shutdown / │ ← 加了生命周期 + 返回 Future │ invokeAll / awaitTermination └───────────────▲──────────────────────────────┘ │ implements（核心实现） ┌───────────────┴──────────────────────────────┐ │ ThreadPoolExecutor ← 本节主角，真正的线程池 │ │ ScheduledThreadPoolExecutor ← 定时/周期任务 │ └──────────────────────────────────────────────┘ Executors（工具类）= 一堆 static 工厂方法，帮你 new 好上面这些
分清三个名字，面试常混。Executor 是接口（只有 execute）；ExecutorService 是子接口（多了 submit/shutdown 等生命周期管理）；Executors（带 s）是工具类，里面全是 newFixedThreadPool() 这类静态工厂方法。真正干活的线程池实现类是 ThreadPoolExecutor。

三、ThreadPoolExecutor 的七个构造参数

所有 Executors.newXxx() 工厂方法，底层不过是用不同参数 new ThreadPoolExecutor(...)。吃透这七个参数，你就掌握了线程池的全部行为：

ThreadPoolExecutor 完整构造器七大金刚
public ThreadPoolExecutor(
    int      corePoolSize,       // 1. 核心线程数：常驻员工
    int      maximumPoolSize,    // 2. 最大线程数：员工上限（含临时工）
    long     keepAliveTime,      // 3. 临时工空闲多久被辞退
    TimeUnit unit,              // 4. 上面时间的单位
    BlockingQueue<Runnable> workQueue,   // 5. 任务排队的队列
    ThreadFactory threadFactory,          // 6. 怎么造线程（命名/守护/优先级）
    RejectedExecutionHandler handler)       // 7. 满了之后的拒绝策略
参数作用踩坑提示
corePoolSize长期存活的线程数，默认即使空闲也不回收可用 allowCoreThreadTimeOut(true) 让核心线程也超时回收
maximumPoolSize线程数的绝对上限（core + 临时线程）只有当队列满了才会创建临时线程——用无界队列时它永远是摆设
keepAliveTime临时线程（超出 core 的部分）空闲多久后被销毁过小则线程反复创建销毁，失去池化意义
workQueuecore 满了之后，任务在此排队选无界还是有界，决定了整个池的"性格"，见下节
threadFactory定制线程名、是否守护、优先级务必自定义线程名，否则出问题时线程栈全是 pool-1-thread-3，无法定位
handler池和队列都满时，如何处置新任务默认 AbortPolicy 直接抛异常，需按业务选择

四、任务提交的决策流程（最核心的机制）

当你 execute(task) 时，线程池内部按一条固定优先级做决策。这套逻辑是所有关于线程池的问题的根，务必刻进脑子：

execute(task) 的四级台阶（顺序不可颠倒） ① 当前线程数 < corePoolSize ? └─ 是 → 直接新建【核心线程】执行 task ← 优先造人，哪怕有空闲也先补到 core ② 否则，尝试把 task 塞进 workQueue └─ 入队成功 → 等空闲线程来取 ← 优先排队，而不是急着造临时工 ③ 队列满了（入队失败）&& 线程数 < maximumPoolSize ? └─ 是 → 新建【临时线程】执行 task ← 加班：拉临时工顶上 ④ 队列满 && 线程数已达 maximumPoolSize └─ 触发【拒绝策略】handler.rejectedExecution(...) 记忆口诀：核心线程 → 排队 → 临时线程 → 拒绝
最反直觉的一点：队列的优先级高于"创建临时线程"。很多人以为"任务多了就该多开线程直到 max"，其实只要队列还没满，线程池宁可让任务排队，也不会去创建超过 core 的线程。这就解释了下一节那个著名的坑——用无界队列时队列永远不会满，于是 maximumPoolSize 形同虚设，池子永远只有 core 个线程在干活。

五、workQueue 的选择：决定线程池的"性格"

队列选得对不对，几乎决定了线程池会不会出事。三种最常见的选择：

队列特性配合的典型池风险
LinkedBlockingQueue（无界）容量 Integer.MAX_VALUE，几乎永不满FixedThreadPool / SingleThreadExecutor任务堆积 → 内存耗尽 OOM；max 参数失效
ArrayBlockingQueue（有界）固定容量，满了才会触发扩容到 max / 拒绝需要背压的生产环境需要自己权衡容量与 max，配置稍复杂
SynchronousQueue（不存储）不缓存任务，每个 put 必须等一个 takeCachedThreadPool来一个任务就可能造一个线程 → 线程数爆炸 OOM
为什么有界队列才是"生产级"选择？因为它带来了背压（back pressure）：队列一满就走拒绝策略，逼着系统正视"我处理不过来了"这个事实——要么丢弃、要么降级、要么让调用方慢下来。无界队列则相反，它把"处理不过来"藏起来，任务默默堆在内存里，直到某一刻突然 OOM 整个进程崩掉。把问题暴露在可控的拒绝点，好过把它藏到不可控的崩溃点。

六、别用 Executors 工厂方法（阿里规约红线）

Executors 的工厂方法用起来很爽，但它们藏着上面说的两类 OOM 雷。看它们的真实底层参数：

工厂方法等价参数为什么危险
newFixedThreadPool(n)core=max=n，队列 = 无界 LinkedBlockingQueue请求积压时任务无限入队 → OOM
newSingleThreadExecutor()core=max=1，队列 = 无界同上，单线程 + 无界队列
newCachedThreadPool()core=0，max=Integer.MAX_VALUE，SynchronousQueue任务洪峰时线程无限增长 → OOM
《阿里巴巴 Java 开发手册》强制条款：线程池不允许使用 Executors 创建，必须通过 new ThreadPoolExecutor 显式指定参数。理由正是上表——工厂方法要么用无界队列、要么用无界线程数，都可能在流量高峰把内存打爆。这是后端面试关于线程池的必考题，记住结论和原因。
✗ Bad · 图省事用工厂方法
// 无界队列，堆积到 OOM 你都不知道
ExecutorService pool =
  Executors.newFixedThreadPool(10);

// 洪峰时线程数无上限
ExecutorService pool2 =
  Executors.newCachedThreadPool();
✓ Good · 显式七参数 + 有界队列
ExecutorService pool = new ThreadPoolExecutor(
    10, 20,                 // core, max
    60L, TimeUnit.SECONDS,   // keepAlive
    new ArrayBlockingQueue<>(500),  // 有界
    new ThreadFactoryBuilder()          // 命名
        .setNameFormat("order-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy());

七、四种拒绝策略

当队列满且线程达到 max，新任务将被 handler 处置。JDK 内置四种：

策略行为适用场景
AbortPolicy（默认）直接抛 RejectedExecutionException需要让调用方明确感知失败
CallerRunsPolicy由提交任务的线程自己去执行该任务想要"天然限流/背压"，让提交方慢下来
DiscardPolicy悄悄丢弃新任务，不抛异常允许丢失的可有可无任务（如埋点）
DiscardOldestPolicy丢掉队首最老的任务，再尝试提交新任务只关心最新数据（如行情推送）
CallerRunsPolicy 是个被低估的妙招。它不丢任务也不抛异常，而是"谁提交谁执行"。效果是：主线程一旦被拉去干活，就没空往池子里塞新任务了，提交速率被自动拽慢，给线程池喘息时间。这是一种优雅的削峰/背压机制，比直接丢弃或抛异常更平滑。当然，如果任务耗时很长，会拖慢主流程，需要评估。

八、如何确定线程池大小

线程不是越多越好——超过某个点，收益被上下文切换、内存竞争吃光。《Java 并发编程实战》给出的经典公式（作者 Brian Goetz）：

Nthreads = Ncpu × Ucpu × (1 + W/C)

其中 Ncpu = CPU 核数（Runtime.getRuntime().availableProcessors()），Ucpu = 目标 CPU 利用率（0~1），W/C = 任务的等待时间 / 计算时间之比。由此可以推出两条广为流传的经验法则：

CPU 密集 ≈ Ncpu + 1 IO 密集 ≈ Ncpu × (1 + W/C)，往往是核数的数倍
  • CPU 密集型（加解密、压缩、纯计算）：线程一直在烧 CPU，W/C≈0，线程数略大于核数即可（+1 是为了在偶发缺页/中断时补位），再多只会徒增切换开销；
  • IO 密集型（读数据库、调远程接口）：线程大部分时间在等，等待时线程被挂起不占 CPU，因此可以开远多于核数的线程来"填满"等待的空档。
公式是起点，压测才是终点。真实任务往往混合了计算与 IO，W/C 很难精确估算。工程上正确的姿势是：用公式估一个初始值，然后在接近生产的负载下压测，观察吞吐量、响应时间（P99）、CPU 利用率与队列长度，反复微调。切忌拍脑袋写个 200。

九、生命周期：优雅关闭线程池

线程池创建了就必须负责关闭，否则里面的非守护线程会一直存活，导致 JVM 无法退出。ExecutorService 提供两把"关门"的方法，区别是面试高频点：

方法是否接收新任务已提交任务怎么办返回值
shutdown()否队列中和正在跑的全部执行完（平缓）void
shutdownNow()否尝试 interrupt() 正在跑的线程，清空队列并返回未执行的任务List<Runnable>
优雅关闭的标准姿势shutdown + awaitTermination + 兜底
void gracefulShutdown(ExecutorService pool) {
    pool.shutdown();                     // 1. 不再接新活，让存量任务跑完
    try {
        // 2. 最多等 60 秒
        if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
            pool.shutdownNow();          // 3. 超时了就强制中断
        }
    } catch (InterruptedException e) {
        pool.shutdownNow();              // 4. 自己被中断，也强制关闭
        Thread.currentThread().interrupt(); // 恢复中断状态
    }
}
陷阱：shutdownNow() 只是"尝试"停止，不保证立刻停。它靠的是给线程发 interrupt() 信号。如果你的任务里根本不响应中断（比如死循环里没检查 Thread.interrupted()，或吞掉了 InterruptedException），线程照样停不下来。能否被优雅关闭，取决于任务本身是否正确响应中断——这一点会在下一节《Future、Callable 与异常处理》里进一步展开。

十、与 Spring 生态的联系

你在 Spring 里写的 @Async 方法，背后就是一个线程池。默认情况下 Spring Boot 用的是 ThreadPoolTaskExecutor——它其实是对 ThreadPoolExecutor 的一层封装，参数名换成了更友好的 corePoolSize / maxPoolSize / queueCapacity。

串联已学与生态：Spring 的 ThreadPoolTaskExecutor 内部 createExecutor() 最终 new 的就是 ThreadPoolExecutor；它默认的 queueCapacity 是 Integer.MAX_VALUE（又是无界队列！），所以生产环境用 @Async 时同样要显式配置有界的 queueCapacity 和拒绝策略，否则会踩到和裸用 newFixedThreadPool 一模一样的 OOM 坑。原理是相通的：换了层马甲，七大参数的账依然要算。
一句话收束：Executor 用一行 execute(Runnable) 把"任务"与"执行任务的线程"解耦，本质是生产者—消费者模式；真正的引擎 ThreadPoolExecutor 靠七大参数驱动，其行为核心是那条"核心线程 → 排队 → 临时线程 → 拒绝"的四级台阶。记牢两条底线：① 别用 Executors 工厂方法（无界队列或无界线程数都会 OOM），一律 new ThreadPoolExecutor 显式配有界队列 + 合理 max + 拒绝策略 + 线程命名；② 用完必须 shutdown() + awaitTermination() 优雅关闭，而能否真正关掉取决于任务是否响应中断。至于任务的返回值与异常怎么拿——那是下一节 Future / Callable 的主场。
