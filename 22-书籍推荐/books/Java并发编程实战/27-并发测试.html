并发测试：正确性测试与性能测试

并发 bug 藏在不确定的线程交错里——测试的全部艺术，就是把"偶发"逼成"必现"，再用数据说清"快"与"稳"。

一、为什么并发测试这么难

写并发代码难，测并发代码更难。单线程测试的黄金假设是"确定性"：相同输入 → 相同输出 → 断言稳定通过或稳定失败。而并发程序的行为取决于线程的交错顺序（interleaving），这个顺序由 OS 调度器决定，你无法控制、也无法复现。于是并发缺陷有了三个折磨人的特征：

  • 概率性：竞态窗口可能只有几纳秒，成千上万次运行才踩中一次。
  • 不可复现：这次挂了，再跑一遍又好了——调试时最抓狂的场景。
  • 海森堡 bug（Heisenbug）：你一加日志 / 一挂调试器，时序被改变，bug 就"躲"起来了——观测行为本身改变了被观测对象。
陷阱：测试通过 ≠ 没有并发 bug。并发测试和普通测试的证明能力天差地别——它只能提高你对正确性的信心，永远无法证明不存在数据竞争。一个跑了一百万次都绿的测试，也可能在生产环境第一天就崩。所以并发正确性的第一道防线永远是正确的设计（不可变、线程封闭、正确同步），测试是第二道防线，不是替代品。这也是本书把"测试"放在"设计"章节之后的深意。

二、先分清测什么：正确性 vs 性能

并发测试大致分两大阵营，目标、手段、成功标准完全不同，动手前必须想清楚自己在测哪一类：

维度正确性测试（Safety / Liveness）性能测试（Performance）
问的问题"在任何交错下结果都对吗？会死锁吗？""够快吗？加线程能等比例变快吗？"
关注指标数据一致性、无丢失/重复、无死锁活锁吞吐量、响应时间/延迟、可伸缩性
核心手段多线程压测 + 校验和、屏障制造真并发、超时检测死锁基准测试（JMH）、预热、多次采样取分布
成功标准大量交错下断言从不失败数据稳定、可解释、能对比方案优劣
最大天敌竞态窗口太窄，测不出来JIT/GC/死代码消除让测量失真
建立联系。正确性测试是在验证前面所学的《什么是线程安全性》《竞态条件》《安全发布》有没有落到实处；性能测试则是在给《性能权衡》那一篇的三笔"隐藏税"和 Amdahl 定律做实证。两者都绕不开一个前提：真正让多个线程同时冲进临界区，否则你测的根本不是并发。

三、正确性测试（一）：先当单线程测基本属性

再复杂的并发类，也应先用单线程验证它的基本契约——这些 bug 和并发无关，却常被忽略。例如一个有界缓冲区 BoundedBuffer：新建时应 isEmpty()==true 且 isFull()==false；放满 N 个后应 isFull()==true。这些确定性断言能先排除一大批低级错误，别急着上多线程。

方法论：从确定性到非确定性，层层加压。顺序是 —— ① 单线程测基本功能 → ② 单线程测边界（空取、满放的阻塞语义）→ ③ 多线程测安全性 → ④ 多线程测阻塞与中断 → ⑤ 长时间压测抓偶发。每一层都建立在上一层通过的基础上，出问题时更容易定位到底是"逻辑错"还是"并发错"。

四、正确性测试（二）：用校验和逼出安全性 bug

这是本章最经典的技术，来自书中的 PutTakeTest。要检验"生产者放进去的东西，消费者一个不多、一个不少、一个不错地取出来"，难点是：多线程下你根本无法逐一比对。解法是顺序无关的校验和（order-insensitive checksum）——用异或/加法把所有元素揉成一个数：

Σput 全部生产的校验和 == Σtake 全部消费的校验和

只要两边总和相等，就说明没有元素丢失、重复或损坏——而加法/异或天然满足交换律，与元素被取出的顺序完全无关，正好适配并发的不确定性。

关键陷阱：校验和的输入必须"不可预测"，否则会被优化掉。如果生产的数据是 0,1,2,3... 这种可预测序列，聪明的 JIT 编译器可能提前算出总和甚至把整段循环消掉，测试就形同虚设。正确做法是用一个廉价但结果不可预知的伪随机数发生器（书中的 xorShift）现场生成数据，让编译器无法预测——这个细节同样是后面性能测试要反复对付的"死代码消除"。
PutTakeTest 核心：屏障 + 校验和生产/消费总和相等 ⇒ 无丢失无重复
public class PutTakeTest {
    private final BoundedBuffer<Integer> bb;
    private final CyclicBarrier barrier;   // 让所有线程"同时起跑/同时收工"
    private final AtomicInteger putSum  = new AtomicInteger(0);
    private final AtomicInteger takeSum = new AtomicInteger(0);
    private final int nTrials, nPairs;

    // 廉价的伪随机：结果不可预测 → 阻止 JIT 提前算出校验和
    static int xorShift(int y) {
        y ^= (y << 6); y ^= (y >>> 21); y ^= (y << 7);
        return y;
    }

    class Producer implements Runnable {
        public void run() {
          try {
            int seed = (this.hashCode() ^ (int) System.nanoTime());
            int sum = 0;
            barrier.await();                 // ① 等齐所有线程，一起冲
            for (int i = nTrials; i > 0; i--) {
                bb.put(seed);
                sum += seed;                // 本线程放入值的累加
                seed = xorShift(seed);
            }
            putSum.getAndAdd(sum);          // 汇总到全局 putSum
            barrier.await();                 // ② 等齐所有线程，一起收
          } catch (Exception e) { throw new RuntimeException(e); }
        }
    }
    // Consumer 对称：take() 后 takeSum.getAndAdd(...)

    void test() {
        ExecutorService pool = Executors.newCachedThreadPool();
        for (int i = 0; i < nPairs; i++) {
            pool.execute(new Producer());
            pool.execute(new Consumer());
        }
        // barrier 的屏障动作里可断言：缓冲区已空
        assertEquals(putSum.get(), takeSum.get());  // 核心断言
    }
}
为什么一定要用 CyclicBarrier？如果直接 start() 一堆线程就完事，第一个线程可能早已跑完，最后一个才刚创建——它们根本没有真正重叠，等于退化成串行，竞态窗口一次都没打开。CyclicBarrier 让所有线程在起跑线（第一个 await）集结、一声令下同时冲，又在终点线（第二个 await）集合，把并发的"密度"拉到最大。这正是《同步工具》里 barrier 的教科书用法。
没有屏障 vs 有屏障：并发"密度"的区别 ✗ 直接 start： ✓ CyclicBarrier 集结起跑： T1 ├──run──┤ await ↓ 一声令下 T2 ├──run──┤ T1 ····│├───run───┤│···· T3 ├──run──┤ T2 ····│├───run───┤│···· T4 ├──run┤ T3 ····│├───run───┤│···· 几乎没有重叠 └ 真正同时冲进临界区 ┘ 竞态窗口 ≈ 0，测了个寂寞 竞态窗口最大化，bug 无处遁形

五、正确性测试（三）：阻塞、中断与资源泄漏

并发类还有一类"活跃度"契约要测：take() 在空缓冲区上必须阻塞，且能被 interrupt() 唤醒。测法是：起一个线程去 take()，主线程 sleep 一小会儿确认它没结束（说明确实阻塞了），再 interrupt 它，并断言它抛出 InterruptedException 后干净退出。

陷阱：用 Thread.getState() 断言线程"处于 BLOCKED/WAITING"是不可靠的。线程状态是给调试和监控用的，规范并不保证时序，JVM 可能用自旋等待而根本不进入 WAITING 状态。正确做法是靠"线程是否已终止"这种确定性事实来间接判断阻塞，而不是去窥探它的内部状态。这是初学者写并发测试最常见的误区之一。

资源泄漏（尤其是内存泄漏）也要测：往缓冲区放入大对象、取出后，被取走的对象应能被 GC 回收。可以持有一个 WeakReference，在若干次 System.gc() 后断言它被清空，验证缓冲区没有意外地长期持有已消费元素的强引用（这类"忘了置 null"的坑在自研容器里很常见）。

六、制造更多交错：把偶发变必现

竞态窗口太窄怎么办？主动"搅乱"调度，让不同的交错更容易出现：

  • 线程数远超 CPU 核数：制造频繁的上下文切换，交错更丰富。
  • 在关键点插入 Thread.yield() 或极短的随机 sleep：主动交出 CPU，增大"别的线程正好插进来"的概率。
  • 反复运行（迭代成千上万次）：靠次数堆概率。

这背后有个朴素但有力的概率模型：设单次运行触发某个竞态的概率是 p（哪怕小到 0.001），跑 N 次，至少命中一次的概率是：

P(至少触发一次) = 1 − (1 − p)N

当 p=0.001、N=10000 时，这个概率已高达约 99.995% —— 这就是"迭代压测 + yield 加噪"能把偶发 bug 逼出来的数学依据。yield 提升的正是那个 p，而循环提升的是 N，两者相乘威力惊人。

建立联系：现代工具已把这套思路工程化。OpenJDK 官方的 jcstress 就是专门做并发正确性压测的框架——它自动生成海量交错、内置 CPU 亲和/噪声注入、把"内存模型层面"的重排序都翻出来，比手写 yield 循环靠谱得多。测底层无锁算法（CAS、volatile 可见性、《并发陷阱》要讲的指令重排）时，jcstress 几乎是标配。手写 PutTakeTest 用来理解原理，实战抓 JMM 级 bug 请上 jcstress。

七、性能测试：三个不同的"快"

切到性能测试，先明确你要的是哪个"快"——它们经常互相矛盾（回顾《性能权衡》）：

指标含义典型场景
吞吐量
throughput		单位时间完成多少任务（TPS/QPS）批处理、消息队列消费
响应时间/延迟
latency		单个请求从发出到完成多久（常看 P99）在线接口、交互式服务
可伸缩性
scalability		加 CPU/线程，处理能力能否等比例上涨评估架构能否横向扩展
别只看平均值，要看分布。平均响应时间是最会骗人的指标：平均 20ms 可能藏着 1% 的请求卡了 2 秒。用户感受由尾延迟（P95/P99/P999）决定。性能测试要采集大量样本看分布直方图，而不是report一个均值了事——这与正确性测试"多次运行"的思路一脉相承：并发世界里，单次采样毫无意义。

八、性能测试的五大陷阱

JVM 是个"活的"运行时——它会即时编译、垃圾回收、内联优化。天真地"记个时间戳、跑个循环、再记个时间戳"测出来的数几乎全是错的。书中列出五大陷阱：

陷阱它如何污染你的数据对策
① 动态编译（JIT 预热）前几千次是解释执行的"慢速版"，之后才被编译成机器码，混在一起测数据偏慢充分预热（warmup）后再计时
② 垃圾回收测量窗口里恰好触发一次 Full GC，数据凭空暴涨要么测足够久让 GC 摊平，要么用 -verbose:gc 剔除受影响的批次
③ 死代码消除结果没被使用，JIT 把整段计算删掉，你测了个空循环消费掉结果（返回/累加/黑洞）
④ 代码路径不真实单测只走了热路径，JIT 按单一场景激进优化，上线后走别的分支性能崩模拟真实的输入分布与分支比例
⑤ 竞争程度不真实基准里线程只做同步、不做别的事，竞争被人为放大或缩小让线程也做与生产相当的"真实工作量"
最阴险的是"死代码消除"和"常量折叠"。如果你的基准循环算出的结果没人用，JIT 完全有权把它当死代码删光——于是你以为"这算法快如闪电"，其实它根本没运行。同理，若输入是编译期常量，JIT 会常量折叠提前算好答案。判断基准是否被"优化没了"有个土办法：把问题规模翻倍，耗时若没接近翻倍，基本可以断定代码被删了。

九、别手写计时循环，用 JMH

要正确对付上面五个陷阱，几乎不可能靠手写代码全兼顾。业界标准答案是 JMH（Java Microbenchmark Harness）——由 JDK 团队打造、和 JVM 优化"斗智斗勇"的官方基准框架。对比一下天真写法与 JMH：

✗ 手写计时：几乎必错
long t0 = System.currentTimeMillis();
long sum = 0;
for (int i = 0; i < N; i++) {
    sum += compute(i);   // 结果可能被
}                        // 死代码消除删掉
long t1 = System.currentTimeMillis();
System.out.println(t1 - t0);
// ① 没预热，混进解释执行
// ② 没处理 GC 抖动
// ③ sum 没被消费 → 可能整段被删
// ④ 只测了一次，没有分布
// 结论：这个数字毫无意义
✓ JMH：把陷阱都替你处理了
@Benchmark
@BenchmarkMode(Mode.Throughput)
@Warmup(iterations = 5)      // 强制预热
@Measurement(iterations = 10)// 多次采样
@Fork(2)                     // 独立 JVM,隔离
public void bench(Blackhole bh) {
    long sum = 0;
    for (int i = 0; i < N; i++)
        sum += compute(i);
    bh.consume(sum);         // 黑洞:防止
}                            // 死代码消除
// 预热/多轮/独立进程/黑洞
// 全帮你做好了,数据可信可对比
Blackhole 是精髓。JMH 的"黑洞"专门用来消费结果——把计算结果喂给它，JVM 就无法证明该结果"没被使用"，于是不敢删掉你的代码。这正是针对陷阱③的对症下药。此外 @Fork 在独立 JVM 里跑，避免上一个基准的 JIT profile 污染下一个；@State 管理测试数据的作用域来模拟真实竞争。测并发数据结构性能（比如对比 synchronized vs ReentrantLock vs LongAdder）时，JMH 的 @Threads 注解能一键拉起多线程压测。

十、测试之外：三道互补防线

测试永远无法穷尽所有交错，所以必须配合其他手段，形成纵深防御：

手段抓什么工具
代码审查人脑能发现测试盖不到的设计缺陷，性价比极高结对/评审，对照本书的同步策略清单
静态分析不运行代码就扫出可疑模式：未同步字段、双重检查锁、wait 不在循环里、锁顺序不一致SpotBugs（原 FindBugs）、ErrorProne
剖析与监控运行时抓锁竞争、线程阻塞、GC、CPU 热点JFR / async-profiler / jstack
面试高频：并发测试的经典误区盘点。① 认为"测试通过就没有并发 bug"——它只能增强信心；② 不用屏障就 start() 一堆线程，线程根本没重叠，测了个假并发；③ 手写计时循环不预热、不消费结果，测出来的数被 JIT 优化污染；④ 用 Thread.getState() 断言阻塞状态；⑤ 只看平均延迟不看 P99；⑥ 校验和用可预测数据，被编译器提前算掉。这六条几乎覆盖了工程里所有"自以为测过了"的翻车现场。
一句话收束：并发测试的本质是"对抗不确定性"。正确性测试要主动制造并发——用 CyclicBarrier 让线程同时起跑把竞态窗口拉满，用顺序无关的校验和验证不丢不重不错，用迭代 + yield 加噪把偶发 bug 按 1−(1−p)ᴺ 的概率逼成必现，用超时和终止事实（而非线程状态）验证阻塞与中断；性能测试则要对抗 JVM 的"活性"——警惕动态编译、GC、死代码消除、路径与竞争不真实五大陷阱，绝不手写计时循环，一律交给 JMH（@Warmup 预热、@Fork 隔离、Blackhole 防删），并看分布而非均值。最后牢记：测试提高信心，但正确的设计（不可变/线程封闭/正确同步）才是根本，代码审查与静态分析是不可或缺的第二、第三道防线。
