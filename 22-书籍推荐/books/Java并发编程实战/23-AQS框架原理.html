AQS 框架原理

一个 volatile int + 一条 CLH 队列，撑起了 ReentrantLock、Semaphore、CountDownLatch 半个 JUC——读懂 AQS，锁的世界就没有黑盒。

一、AQS 解决了什么问题

AbstractQueuedSynchronizer（AQS，抽象队列同步器）是 Doug Lea 在 JUC 里埋下的发动机。ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock、ThreadPoolExecutor 里的 Worker、甚至 CompletableFuture 的等待节点——它们的底层几乎都是同一套代码。

为什么要抽出这么一层？因为任何一个“阻塞式同步器”都要重复解决三件极难写对的事：①用什么表示“同步状态”（锁被占了几次？还剩几个许可？）、②抢不到时线程怎么排队阻塞、③状态释放时怎么公平地唤醒下一个。这三件事里，队列管理和 park/unpark 的时序，是并发 bug 的重灾区。AQS 把这些共性一次性写对、封装好，只把差异——“状态怎么算作获取成功”——用几个钩子方法留给子类。

一句话定位： AQS 用模板方法模式（Template Method）把“排队 + 阻塞 + 唤醒”这套骨架固化在父类，把“状态语义”这个变量下沉给子类实现。你只要回答“此刻算不算获取成功”，剩下线程怎么挂起、怎么排队、怎么被叫醒，AQS 全包了。这正是《设计模式》里模板方法“不变的算法骨架 + 可变的步骤”的教科书级应用。

二、核心三件套：state + CLH 队列 + 模板方法

AQS 的全部机关，浓缩成三样东西。理解了它们的配合，源码就不再吓人。

组件实现职责
同步状态private volatile int state用一个整数表达“资源还剩多少”。锁=持有次数，信号量=许可数，闭锁=倒数计数
CLH 变体队列双向链表 Node head/tail抢不到的线程在此 FIFO 排队并 park 挂起，释放时按序唤醒
模板方法tryAcquire / tryRelease 等子类重写，定义“对 state 做什么运算才算获取/释放成功”

关键在于 state 是 volatile 的，而所有修改都通过 getState() / setState() / compareAndSetState() 三个方法进行——最后一个是基于 Unsafe.compareAndSwapInt 的 CAS 操作。这把我们之前学的 volatile 保可见性、CAS 保原子性 两块拼图，在这里合体成了无锁的状态机。

联系已学： volatile 保证一个线程改了 state，其他线程立刻看得见（可见性 + 禁止重排）；但 state++ 这种读-改-写不是原子的，所以要用 compareAndSetState 这个 CAS 来“比较并交换”。volatile 负责“看得见”，CAS 负责“改得对”——这也是下一篇《Atomic 类与 CAS 原理》的核心，AQS 是它最重量级的用户。

三、CLH 队列长什么样

抢不到锁的线程被包装成一个 Node，用 CAS 挂到队尾（tail），然后调用 LockSupport.park() 把自己挂起。CLH 是一种基于双向链表的公平 FIFO 队列（AQS 用的是它的变体：显式维护 prev/next 指针，并用 waitStatus 传递唤醒信号）。

┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ head→ │ Node │ ←──→ │ Node │ ←──→ │ Node │ ←──→ │ Node │ ←tail │ (哨兵)│ │ T1 │ │ T2 │ │ T3 │ │ ws=-1│ │ ws=-1│ │ ws=-1│ │ ws= 0│ └──────┘ └──────┘ └──────┘ └──────┘ 持锁线程 第1个等待 第2个等待 刚入队 (parked) (parked) (parked) ws = waitStatus：SIGNAL(-1) 表示“我释放后有责任唤醒后继” head 是不含线程的哨兵节点；真正的锁持有者的“影子”即 head 释放锁 → 找到 head 的后继 → LockSupport.unpark(T1) 叫醒队首

每个 Node 有一个 waitStatus 字段，最常见的是 SIGNAL(-1)：它标记在前驱节点上，含义是“我（前驱）在释放/取消时，有义务 unpark 我的后继”。这套“前驱负责叫醒后继”的设计，避免了每个线程醒来空转检查，是 park/unpark 时序正确的关键。

waitStatus值含义
SIGNAL-1后继节点已 park，本节点释放时须唤醒它
CANCELLED1线程因超时/中断放弃，节点作废，会被跳过清理
CONDITION-2节点在 Condition 条件队列上等待（非同步队列）
PROPAGATE-3共享模式下，唤醒需向后传播
默认0初始状态

四、独占 vs 共享：两种获取模式

AQS 支持两大类同步器，区别在于“state 是否允许多个线程同时获取”。这决定了子类该重写哪组模板方法。

Exclusive · 独占
同一时刻只有一个线程能获取
state 常表示“持有/重入次数”

要重写：
  tryAcquire(int)
  tryRelease(int)
  isHeldExclusively()

代表：ReentrantLock
      WriteLock
Shared · 共享
多个线程可同时获取
state 常表示“剩余许可/计数”

要重写：
  tryAcquireShared(int)
  tryReleaseShared(int)

代表：Semaphore
      CountDownLatch
      ReadLock
陷阱：模板方法默认抛异常，不是空实现。 AQS 里 tryAcquire/tryRelease/tryAcquireShared/tryReleaseShared 的默认实现都是 throw new UnsupportedOperationException()。这是刻意为之——强迫你按模式只重写该重写的那一组：写独占同步器却手贱重写了 tryAcquireShared，或该重写的忘了写，一运行就抛异常暴露，而不是静默错误。另外注意 tryAcquireShared 返回的是 int 而非 boolean：负数=失败，0=成功但后续无余量，正数=成功且还有剩余可继续唤醒后继。

五、acquire 源码走读（独占）

以最经典的 acquire(int) 为例，这是 ReentrantLock.lock() 的骨架。全流程只有短短几行，却精确编排了“先自旋抢一把、抢不到才排队挂起”的逻辑。

AbstractQueuedSynchronizer.java（简化）JDK 源码
public final void acquire(int arg) {
    // 1) 先试着直接抢（tryAcquire 由子类实现）；抢到就返回，连队列都不进
    if (!tryAcquire(arg) &&
        // 2) 抢不到 → 包装成独占节点入队 → 在队列里自旋+park 直到拿到锁
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();   // 3) 排队期间被中断过，这里补一次中断标志
}

final boolean acquireQueued(final Node node, int arg) {
    boolean interrupted = false;
    for (;;) {                              // 死循环 = 醒来后重新竞争
        final Node p = node.predecessor();
        if (p == head && tryAcquire(arg)) { // 只有“队首的下一个”才有资格抢
            setHead(node);                   // 抢到 → 自己变哨兵头，脱离等待
            p.next = null;                    // 帮助 GC
            return interrupted;
        }
        // 没轮到 / 没抢到 → 该不该 park？该就 LockSupport.park() 挂起
        if (shouldParkAfterFailedAcquire(p, node) && parkAndCheckInterrupt())
            interrupted = true;
    }
}
acquire(1) 流程： tryAcquire ── 成功 ──────────────────────────→ 拿到锁，返回 │ 失败 ↓ addWaiter：CAS 把当前线程节点接到 tail ↓ acquireQueued 死循环： ├─ 我的前驱是 head？→ 再 tryAcquire 一次 │ └─ 成功 → setHead，出队，返回 └─ 否则 → 把前驱 ws 置 SIGNAL → park() 挂起 ↑ 别人 release → unparkSuccessor → 我被唤醒，回到循环顶重试

释放侧对称而简洁：release() 调用子类 tryRelease() 把 state 减到位后，若 head 存在且 waitStatus != 0，就 unparkSuccessor(head) 唤醒队首后继。被唤醒的线程从 park() 处恢复，回到 for(;;) 顶端重新 tryAcquire——注意是重新竞争，不是直接给它，这给了非公平锁“插队”的可能。

联系操作系统： LockSupport.park()/unpark() 底层是 Unsafe.park，在 Linux 上对应 pthread 的条件变量/futex——线程真正被内核挂起，不占 CPU，这与自旋锁（忙等烧 CPU）本质不同。park 还有个妙处：unpark 可以先于 park 发生（许可可预存一个），从而避免了 wait/notify 那种“通知丢失”的经典竞态。

六、手写一个同步器：Mutex

百闻不如一见。用 AQS 造一把最简单的不可重入互斥锁，你会发现自己几乎不用碰任何线程排队代码——只需回答“state 0→1 算加锁成功，1→0 算释放”。这正是 AQS 官方 Javadoc 的示例。

Mutex.java正例 · 基于 AQS 自定义锁
public class Mutex {
    // 把同步逻辑委托给一个继承 AQS 的内部类（组合优于继承的实践）
    private static class Sync extends AbstractQueuedSynchronizer {

        // state==1 表示已被某线程独占持有
        protected boolean isHeldExclusively() { return getState() == 1; }

        // 唯一要动脑的地方：CAS 把 0 改成 1 就算抢到锁
        protected boolean tryAcquire(int ignore) {
            if (compareAndSetState(0, 1)) {
                setExclusiveOwnerThread(Thread.currentThread());
                return true;
            }
            return false;                       // 抢不到 → 交给 AQS 去排队/park
        }

        protected boolean tryRelease(int ignore) {
            if (getState() == 0)
                throw new IllegalMonitorStateException();
            setExclusiveOwnerThread(null);
            setState(0);                       // 释放：置回 0，AQS 会唤醒后继
            return true;
        }
        Condition newCondition() { return new ConditionObject(); }
    }

    private final Sync sync = new Sync();
    public void lock()    { sync.acquire(1); }     // 复用 AQS 的排队+挂起
    public void unlock()  { sync.release(1); }    // 复用 AQS 的唤醒
    public boolean tryLock() { return sync.tryAcquire(1); }
}
体会“不变 vs 可变”的边界： 上面这 30 行里，你没写一行“如何入队、如何 park、如何唤醒下一个”——那些复杂、易错的骨架全在 acquire()/release() 里由 AQS 提供。你只实现了 tryAcquire/tryRelease 这几个纯状态判断。这就是模板方法模式为你省下的心智负担。

七、AQS 如何撑起半个 JUC

同一套 AQS，靠对 state 赋予不同语义，就变出了性格迥异的同步工具。看清这张“语义对照表”，JUC 的许多类你就能秒懂其原理。

同步器模式state 的含义核心逻辑
ReentrantLock独占重入持有次数0→1 加锁；同线程再进则 +1；释放 -1，减到 0 才真正放锁
Semaphore共享剩余许可数acquire 时 CAS 减 1（减到负数则排队）；release 加 1 并唤醒
CountDownLatch共享剩余计数countDown 减 1；await 在 state≠0 时阻塞，归 0 时全部放行
ReentrantReadWriteLock混合高16位=读锁数
低16位=写锁数		一个 int 拆两半，同时管读写两种状态
ThreadPoolExecutor.Worker独占0/1 运行标记用极简 AQS 表达“任务执行中不可被 interrupt 打断”
ReentrantReadWriteLock 的位运算巧思： 一个 32 位 int 装不下两个独立计数？拆成两半即可——state >>> 16 是读锁持有总数，state & 0xFFFF 是写锁重入数。CountDownLatch 则更极简：tryAcquireShared 只判断 getState()==0 ? 1 : -1，tryReleaseShared 循环 CAS 把计数减 1。同一引擎，配不同的“state 解释器”，就是完全不同的工具。

八、公平锁 vs 非公平锁

ReentrantLock 的公平与否，差别仅仅一行：抢锁前要不要检查队列里有没有人排在你前面。这也解释了为什么默认是非公平——吞吐更高。

NonfairSync · 非公平（默认）
protected boolean tryAcquire(int acq) {
  if (getState() == 0) {
    // 不看队列，上来就抢！
    if (compareAndSetState(0, acq)) {
      setExclusiveOwnerThread(cur);
      return true;
    }
  }
  ...重入判断...
}
// 新来的可能插队 → 吞吐高
// 但排队者可能饥饿
FairSync · 公平
protected boolean tryAcquire(int acq) {
  if (getState() == 0) {
    // 先问：队列里有前辈吗？
    if (!hasQueuedPredecessors()
        && compareAndSetState(0, acq)) {
      setExclusiveOwnerThread(cur);
      return true;
    }
  }
  ...重入判断...
}
// 严格 FIFO → 无饥饿
// 频繁挂起/唤醒 → 吞吐低
陷阱 & 高频面试点： ① 非公平锁并非“完全无序”——只有刚好在锁空闲瞬间到达的新线程能插队；一旦进了队列，仍是 FIFO。② 默认非公平的真实理由：公平锁每次都要唤醒队首线程再让它上下文切换，而非公平允许“正在运行的线程”直接续上，省掉一次昂贵的 park/unpark，吞吐通常显著更高。③ hasQueuedPredecessors() 是公平性的唯一开关，别小看这一个方法。

九、Condition：AQS 自带的等待队列

还记得 synchronized 配 wait/notify 吗？显式锁的对应物是 Lock.newCondition() 返回的 Condition，其实现 ConditionObject 正是 AQS 的内部类。它维护一条独立于同步队列的“条件队列”：await() 把线程节点从同步队列移到条件队列并释放锁；signal() 再把节点从条件队列迁回同步队列去重新抢锁。

同步队列（争锁） 条件队列（等条件） head→A→B→C firstWaiter→X→Y X.await() ：X 释放锁，从"同步队列"移入"条件队列"，park signal() ：把 X 从"条件队列"摘出，重新 enq 回"同步队列"队尾 ↑ 注意：signal 只是"移队"，X 还得重新排队抢锁才能醒
为什么比 wait/notify 强： 一把 Lock 可以 newCondition() 出多个独立 Condition（如“非空”和“非满”两个条件，见 ArrayBlockingQueue），signal 时能精准唤醒等待特定条件的线程；而 synchronized 只有一个隐式等待集，notify 只能盲目唤醒。这就是 Condition 独立成篇的价值——详见上一节。
同步器 = AQS(排队 + park/unpark) + 子类(state 语义)

十、避坑清单与面试锦囊

  • 别直接继承 AQS 暴露给外部。 规范做法是像 ReentrantLock 那样，用一个 private static class Sync extends AQS 内部类持有同步逻辑，外层类组合它——避免把 acquire/release 这些底层方法泄露给使用者。
  • tryAcquireShared 返回 int，不是 boolean。 语义是“剩余资源量”，返回值 ≥0 才算成功，且正数会触发向后传播唤醒。写共享同步器时最容易在这里翻车。
  • CAS 一定放在自旋里。 单次 compareAndSetState 可能因竞争失败，同步器代码几乎总是 for(;;) 包着 CAS 重试，切勿写成“失败就返回”。
  • 面试必答： AQS 用什么模式？模板方法。队列是什么？CLH 变体的双向 FIFO。靠什么挂起线程？LockSupport.park/unpark。state 怎么保证线程安全？volatile 可见性 + CAS 原子更新。公平锁差在哪？tryAcquire 里多一个 hasQueuedPredecessors 判断。
一句话收束： AQS 是 JUC 的地基发动机——它用 模板方法模式把“入队 → park 挂起 → 释放时唤醒后继”这套最难写对的骨架封在父类，只把“对 volatile 的 state 做什么 CAS 才算获取成功”留给子类。ReentrantLock、Semaphore、CountDownLatch 只是给同一个 state 赋予了不同解释。看懂它，你就同时看穿了半个 java.util.concurrent。
