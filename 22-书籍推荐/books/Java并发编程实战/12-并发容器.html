并发容器：ConcurrentHashMap、CopyOnWriteArrayList

把整张表锁死，还是只锁需要动的那一格？并发容器的全部智慧，都藏在"缩小锁的粒度"和"读写分离"这两句话里。

一、为什么需要它：同步容器把并发退化成了串行

在并发容器出现之前，想要一个线程安全的 Map，标准做法是 Hashtable 或 Collections.synchronizedMap(new HashMap<>())。它们的实现思路简单粗暴：给每个方法都加上 synchronized，用同一把锁（对象自己）保护整个容器。

这确实安全，但代价是——无论多少线程、无论操作的是不是同一个 key，都得排队抢这一把大锁。10 个线程读 10 个互不相干的桶，本可以完全并行，却被强行串行化。这在高并发下就是灾难。

一个生活类比：同步容器像一间只有一把总钥匙的图书馆——哪怕你只想看 A 区的书、我只想看 Z 区的书，也得轮流拿那把总钥匙进门。ConcurrentHashMap 则像给每个书架单独配了锁：你动 A 架、我动 Z 架，互不打扰，只有抢同一个架子时才需要等。锁的粒度从"整栋楼"缩小到了"一个架子"，并发度自然上来了。
陷阱：同步容器"每个方法安全" ≠ "复合操作安全"。这是最高频的误区。synchronizedMap 保证单次 get/put 原子，但"先查再放"这种复合操作横跨两次调用，两次之间锁已经释放，别的线程能插进来。下面这段"如果不存在就放入"的代码在并发下会重复放入或覆盖——这正是上一章讲的竞态条件（check-then-act）。
✗ 复合操作，两次调用之间有缝隙
Map<String,User> m =
    Collections.synchronizedMap(new HashMap<>());

// check-then-act：非原子!
if (!m.containsKey(id)) { // 线程A、B 可能同时判为 false
    m.put(id, load(id));  // 于是 load 两次、put 两次
}
✓ 用容器提供的原子复合方法
ConcurrentHashMap<String,User> m =
    new ConcurrentHashMap<>();

// 整个"不存在才计算并放入"是一个原子操作
User u = m.computeIfAbsent(id,
        k -> load(k)); // load 只会被调用一次

二、ConcurrentHashMap 的进化：从分段锁到 CAS + synchronized

要理解它今天的样子，得先看它当年怎么想的。JDK 7 和 JDK 8 是两套完全不同的设计，面试几乎必问它们的区别。

JDK 7：Segment 分段锁（把大锁拆成 16 把小锁）

JDK 7 的核心思想是把一整个 Map 切成若干段（Segment），每段一把锁。默认 16 段，意味着理论上可以有 16 个线程同时写而互不阻塞。Segment 继承自 ReentrantLock，本质是"16 个各自带锁的小 HashMap"拼在一起。

JDK 7 结构：两层哈希（先定位 Segment，再定位桶） ConcurrentHashMap ├─ Segment[0] (extends ReentrantLock) ─► HashEntry[] ─► 链表 ├─ Segment[1] (一把独立的锁) ─► HashEntry[] ─► 链表 ├─ ... └─ Segment[15] ─► HashEntry[] ─► 链表 写 key 时： hash(key) 高位 → 选中某个 Segment → 只锁这一段 不同 Segment 的写操作 → 完全并行 并发度(concurrencyLevel) = Segment 个数 = 16 (默认，创建后不可变)
分段锁的精妙与局限。精妙在于：把"一把锁"变成"16 把锁"，锁竞争的概率直接降到 1/16。局限在于：并发度被 Segment 数量写死了——16 段就最多 16 线程并行，再多也没用；而且两层哈希多绕了一次，定位一个元素要先算 Segment 再算桶，还多占一层对象的内存。JDK 8 决心把这层"中间商"干掉。

JDK 8：直接锁桶（数组 + 链表/红黑树 + CAS + synchronized）

JDK 8 彻底重写：抛弃 Segment，回到和 HashMap 一样的"一个大数组"结构，锁的粒度进一步缩小到单个桶（数组的一个槽 / 一条链表的头节点）。这样并发度不再是固定的 16，而是等于桶的数量——数组越大，能并行的写线程越多，粒度天然随容量伸缩。

锁粒度：整表 > Segment(16) > 单个桶(随容量增长)

它把两种利器用在了不同场景，堪称"能不加锁就不加锁"的教科书：

  • CAS（无锁）：当某个桶是空的，用 compareAndSet 直接把新节点塞进去。这一步连锁都不用——乐观地假设没人跟我抢，抢输了就重试。
  • synchronized（有锁）：当桶非空（发生哈希冲突，要挂到链表/红黑树上）时，才对这个桶的头节点 synchronized。只锁一个桶，其他桶照常并行。
为什么 JDK 8 敢把 ReentrantLock 换回 synchronized？很多人以为 synchronized 一定比 ReentrantLock 慢——那是 JDK 6 之前的老黄历。JDK 6 起 synchronized 引入了偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁的锁升级机制，在竞争不激烈时几乎零成本；而这里锁的是单个桶头节点，竞争本就轻微，绝大多数情况停在轻量级锁阶段。加上 synchronized 由 JVM 原生支持、无需像 ReentrantLock 那样额外维护 AQS 队列对象，更省内存、更贴近底层。这是"选对场景"胜过"选贵的工具"的经典案例。
ConcurrentHashMap.java (JDK 8 简化)putVal 的核心分支
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 计算哈希，注意 key/value 不能为 null（见下文陷阱）
    int hash = spread(key.hashCode());
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();                       // 懒初始化，也靠 CAS 保证只初始化一次
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // ① 桶为空：直接 CAS 放入，无锁！抢失败就 for 循环重试
            if (casTabAt(tab, i, null, new Node<>(hash, key, value)))
                break;
        }
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);             // 正在扩容？当前线程"帮忙"搬迁
        else {
            // ② 桶非空（哈希冲突）：只锁这个桶的头节点 f
            synchronized (f) {
                // 遍历链表或红黑树，插入/覆盖……
            }
        }
    }
    addCount(1L, binCount);                     // 用类似 LongAdder 的分片计数（见 size 陷阱）
    return null;
}
两个 JDK 8 彩蛋。其一，链表长度超过 8 且数组容量 ≥ 64 时，链表会树化成红黑树，把最坏查找从 O(n) 降到 O(log n)——这和 HashMap JDK 8 的优化同源。其二，扩容时支持多线程协同搬迁（helpTransfer）：一个线程发现 map 正在扩容，不会干等，而是"搭把手"一起搬运桶，搬完的桶用一个特殊的 ForwardingNode（hash = MOVED）标记，读操作遇到它会自动转到新表去查。并发扩容，这是 JDK 7 做不到的。

三、弱一致性：迭代器不抛 ConcurrentModificationException

用过普通 HashMap 的人都被 ConcurrentModificationException（并发修改异常）坑过：一边遍历一边改，立刻抛异常。这叫 fail-fast（快速失败）——靠一个 modCount 计数器发现结构变了就报错，宁可崩也不给你脏数据。

ConcurrentHashMap 走的是相反的 fail-safe / 弱一致性（weakly consistent）路线：迭代器基于创建那一刻的结构遍历，遍历期间别的线程增删，它不抛异常，但你可能看到、也可能看不到那些改动。

陷阱：弱一致性下，聚合方法的返回值只是"估计值"。size() 和 isEmpty() 不加全局锁，返回的是某一瞬间的近似值——你拿到 100 的下一刻它可能已是 101。所以绝不能写 for (int i = 0; i < map.size(); i++) 这种依赖精确 size 的循环，也不能用 size()==0 当作严格的空判断。JDK 8 还新增了 mappingCount() 返回 long，因为元素数可能超过 Integer.MAX_VALUE——但它同样是估计值。需要精确计数？说明你可能选错了数据结构。
为什么甘愿牺牲一致性？这是一次刻意的取舍：要让 size() 精确，就得在统计时锁住整个 map，那又回到了"整表大锁"的老路，前面所有缩小粒度的努力就白费了。并发容器的设计哲学是——用"稍微不那么精确"换取"高吞吐、不阻塞读写"。这也解释了它内部为什么用类似 LongAdder 的分片计数（多个 counterCell 分散热点，求和时才相加）来统计元素数，而不是一个全局的 volatile int——后者会成为所有写线程争抢的热点。
陷阱：ConcurrentHashMap 不允许 null 键或 null 值（HashMap 却允许）。put 一个 null value 会直接 NullPointerException。原因是并发下的二义性：map.get(k) 返回 null 时，你无法区分"这个 key 不存在"还是"这个 key 存了 null 值"——单线程里可以再用 containsKey 补一刀判断，但并发环境下这两步之间值可能被改，判断本身就不可靠。索性从源头禁止 null，让 get()==null 唯一地表示"不存在"。

四、用原子复合方法代替"查—改—写"

并发容器最大的价值不只是"线程安全的 get/put"，更是它把常见的复合操作打包成了单个原子方法，帮你消灭 check-then-act 竞态。这些方法你必须烂熟：

方法语义典型用途
putIfAbsent(k,v)key 不存在才放入 v，返回旧值（或 null）去重注册、只初始化一次
computeIfAbsent(k, fn)key 不存在才用 fn 计算并放入，fn 只跑一次懒加载缓存（最常用）
compute(k, fn)基于旧值算新值并放入原子更新对象
merge(k, v, fn)无则放 v，有则用 fn 合并新旧值原子累加计数
remove(k, v)key 的当前值等于 v 才删（CAS 语义）安全删除
✗ 并发计数丢更新
ConcurrentHashMap<String,Long> cnt
    = new ConcurrentHashMap<>();

// get / put 两步，中间会被打断
Long old = cnt.get(word);   // 读到 5
cnt.put(word, (old==null?0:old)+1);
// 两个线程都读到 5，都写 6 → 少加一次
✓ merge 一步到位，原子累加
ConcurrentHashMap<String,Long> cnt
    = new ConcurrentHashMap<>();

// 无则置 1，有则旧值 + 1，整体原子
cnt.merge(word, 1L, Long::sum);

// 计数场景更推荐 LongAdder 作为 value：
// map.computeIfAbsent(k,
//     x->new LongAdder()).increment();
陷阱：computeIfAbsent 的映射函数里，别再动同一个 map。它执行 fn 时持有当前桶的锁，如果你在 fn 内部对同一个 key（甚至映射到同一桶的另一个 key）再做 put/computeIfAbsent，JDK 8 可能死循环或抛异常（这是官方修复过的知名 bug）。守则：映射函数应当短小、无副作用，只负责"算出这个 key 的值"，别在里面折腾容器本身。

五、CopyOnWriteArrayList：读多写极少的另一条路

ConcurrentHashMap 走的是"缩小锁粒度"；CopyOnWriteArrayList（写时复制）则走另一个极端——读操作完全不加锁，代价全压给写操作。它的名字就是全部秘密：每次写（add/set/remove），都把整个底层数组复制一份，在副本上改，改完再把内部引用"原子地"指向新数组。

写时复制（Copy-On-Write）的一次 add： 写前: array ──► [A, B, C] ← 读线程正遍历这个旧数组，安全 （它读到的始终是一个不变的快照） 写时: 1) 加 ReentrantLock（写与写互斥） 2) 复制: newArray = [A, B, C] → [A, B, C, D] 3) 在副本上改，旧数组一个字节都没动 4) array = newArray ← volatile 引用赋值，一步切换（原子） 写后: array ──► [A, B, C, D] ← 新读线程看到新数组 关键: 读路径上没有任何锁、没有 CAS，就是一次 volatile 读，快到极致
为什么读可以完全无锁？因为写操作从不修改正在被读的那个数组——它另起炉灶复制一份改。旧数组是事实不可变（effectively immutable）的，而不可变对象天生线程安全（回忆「不变性」那一篇）。读线程要么看到旧数组、要么看到新数组，绝不会看到"改了一半"的中间态。内部数组引用用 volatile 修饰，保证写线程 array = newArray 后，读线程立刻可见——这正是happens-before 里 volatile 写/读建立的可见性保证。
迭代器是"快照"，永不抛 ConcurrentModificationException。调用 iterator() 那一刻，它把当时的数组引用抓在手里，之后无论别人怎么写（写的是新数组），这个迭代器始终遍历它抓到的旧快照。好处是遍历绝对安全、无需加锁；代价是遍历期间的新增改动你看不到（弱一致性），而且迭代器不支持 remove()/set()/add()——因为它面对的是一份只读快照，改了也没意义，会直接抛 UnsupportedOperationException。
CopyOnWriteArrayList.java (JDK 简化)add 加锁复制，get 直接读
public boolean add(E e) {
    final ReentrantLock lock = this.lock;
    lock.lock();                              // 写与写互斥（同一时刻只允许一个写）
    try {
        Object[] es = getArray();
        int len = es.length;
        es = Arrays.copyOf(es, len + 1);  // ★ 复制整个数组，O(n)
        es[len] = e;
        setArray(es);                        // volatile 写，原子切换引用
        return true;
    } finally {
        lock.unlock();
    }
}

public E get(int index) {
    return (E) getArray()[index];        // 读：无锁！就是一次 volatile 读 + 取下标
}
陷阱：写操作是 O(n) 且极耗内存，绝不能用于写频繁的场景。每次 add 都复制整个数组：往一个 10000 元素的列表里循环 add 一万次，就是复制一万次、总共搬运约 5000 万次引用，还在瞬间产生大量待回收的旧数组，直接把 GC 压垮。它只适合"读远多于写、且集合不大"的场景——最经典的就是事件监听器列表（注册一次、之后疯狂遍历触发）、配置/白名单快照。需要频繁写？请回到 ConcurrentHashMap（用 key 集合模拟 Set）或加锁的普通结构。

建立联系：三条完全不同的技术路线。并发安全其实就三大流派——①分段/分桶加锁（ConcurrentHashMap：缩小锁粒度，读写都快）；②写时复制（CopyOnWriteArrayList：读无锁，写复制，赌"读远多于写"）；③无锁 CAS（ConcurrentLinkedQueue：靠 CAS 自旋，一把锁都不用）。它们没有优劣，只有是否匹配你的读写比例。选容器的第一问永远是：我的读写比例、数据规模、是否需要有序，到底是什么样？

七、终极陷阱：容器安全 ≠ 你的业务逻辑安全

用了 ConcurrentHashMap 就高枕无忧了？不。它只保证单个方法调用原子，你的业务往往是多次调用的组合——组合起来照样有竞态。这是初学者最容易踩空的地方。

✗ 两次原子调用 ≠ 组合原子
ConcurrentHashMap<String,Integer> m
    = new ConcurrentHashMap<>();

// get 原子、put 原子，但合起来不是!
Integer v = m.get("k");        // 读到 3
m.put("k", v + 1);             // 别人也读到 3
// 结果 4，本该是 5 → 丢更新
✓ 让容器帮你把组合变原子
ConcurrentHashMap<String,Integer> m
    = new ConcurrentHashMap<>();

// compute 内部持桶锁，读改写一体原子
m.compute("k",
    (k, v) -> (v == null ? 0 : v) + 1);
// 或 m.merge("k", 1, Integer::sum);
一句话收束：并发容器的全部智慧就两条主线——缩小锁的粒度与读写分离。ConcurrentHashMap 从 JDK 7 的分段锁进化到 JDK 8 的桶级 CAS + synchronized + 红黑树，把锁粒度压到单个桶，读几乎无锁；CopyOnWriteArrayList 则用写时复制把读变成一次纯 volatile 读，代价是写 O(n)、只配读多写少。它们都以弱一致性（size()/迭代器是估计值/快照、不抛并发修改异常、不许 null）换取高吞吐。但请牢牢记住最后一颗地雷：容器保证单次调用原子，不保证你把多次调用拼起来的复合操作原子——需要"读—改—写"时，务必用 computeIfAbsent / merge / compute 这类原子方法，而不是自己 get 完再 put。选容器前先问清楚：读写比例、数据规模、是否需要有序。
