volatile 与 happens-before

volatile 是"最轻量的同步"，而 happens-before 是 JMM 给你的一纸契约——看懂这纸契约，可见性与有序性才不再是玄学。

一、volatile 到底承诺了什么

上一节我们把线程安全拆成了原子性、可见性、有序性三根支柱。volatile 是一个只对其中两根负责的关键字：它保证可见性和有序性，唯独不保证原子性。把这句话钉死在脑子里，你就绕过了初学者一半的坑。

volatile 的两条承诺：①可见性——对一个 volatile 变量的写，会立刻刷回主内存，并让其他 CPU 核的缓存副本失效；后续对它的读，一定读到最新值。②有序性——JMM 通过内存屏障禁止把 volatile 读写与它前后的普通读写做特定方向的重排序。它没有承诺的是原子性：volatile 只保证"单次读"或"单次写"这一个动作是完整的，管不了"读—改—写"这种复合操作。
陷阱一：volatile int count; count++; 依然会丢更新。这是最高频的误解。count++ 是读→改→写三步，volatile 只保证每一步各自读到 / 写出的是新值，却挡不住两个线程同时读到同一个旧值再各自 +1。想让计数器正确，请用 AtomicInteger（CAS）或 synchronized——把 volatile 当原子变量用，是并发代码里最常见的翻车姿势。
✗ 误把 volatile 当原子
volatile int count = 0;

// 两个线程各 +10000
void inc() {
  count++;   // 读+改+写，非原子
}
// 结果常常 < 20000
// volatile 救不了它
✓ 复合操作用 CAS
final AtomicInteger count
    = new AtomicInteger();

void inc() {
  count.incrementAndGet();
  // CAS 自旋，硬件级原子
}
// 结果稳定 == 20000
那 volatile 适合什么？一句话：一个线程写、多个线程读，且写入不依赖当前值的场景。最典型的就是状态标志位（volatile boolean running）、一次性发布的配置 / 单例引用、以及"写完就不改"的一写多读数据。只要写操作是"直接赋一个不依赖旧值的新值"，volatile 就够用且比锁便宜得多。

二、看得见的底层：内存屏障

volatile 不是魔法，它编译后会在字节码 / 机器码里插入内存屏障（Memory Barrier / Fence）——一种告诉 CPU 和编译器"这里不许重排、缓存必须同步"的特殊指令。JSR-133（Java 5 重写内存模型）规定的保守插入策略是：

volatile 写： │ volatile 读： │ [普通写 a = 1] │ ┌─────────────────────┐ ┌──── StoreStore ───┐ │ │ volatile 读 v │ │ volatile 写 v = 1 │ │ └─────────────────────┘ └──── StoreLoad ────┘ │ ┌──── LoadLoad ───────┐ │ ├──── LoadStore ──────┤ │ │ 后续普通读/写 │ │ └─────────────────────┘ 含义： StoreStore → 保证 volatile 写之前的普通写，都先刷回主内存 StoreLoad → 保证 volatile 写"落地"后，后面的读才能进行（最贵的屏障） LoadLoad → 保证 volatile 读之后的读，不会被提前 LoadStore → 保证 volatile 读之后的写，不会被提前

这套屏障恰好构成一道"单向栅栏"：volatile 写之前的所有操作不能挪到写之后，volatile 读之后的所有操作不能挪到读之前。这正是下一节"搭便车"能成立的物理基础。

联系操作系统 / 硬件：可见性问题的根源是 CPU 每个核有自己的缓存和写缓冲区（store buffer）。缓存之间靠 MESI 缓存一致性协议互相通知，但 store buffer 里的写还没广播出去时，别的核就看不到。volatile 写在 x86 上通常编译成一条带 lock 前缀的指令（如 lock addl），它会把 store buffer 刷干净并让其他核的对应缓存行失效——这就是 StoreLoad 屏障在真实硬件上的落地。x86 属于强内存模型（TSO），只允许"写后读"重排，所以四种屏障里只有 StoreLoad 真正有开销，其余在 x86 上几乎是空操作。

三、happens-before：JMM 给你的契约语言

如果每次判断"这个修改另一个线程看不看得到"都要去推理内存屏障、缓存协议，那并发编程就没法写了。所以 JMM 抽象出一套更高层、与硬件无关的规则——happens-before。它是你和 JVM 之间的契约：只要你按规则建立起 happens-before 关系，JVM 就保证可见性和有序性，至于底层用屏障还是别的手段实现，你不用关心。

陷阱二：happens-before ≠ "时间上先发生"。这个名字极具误导性。A happens-before B 的真正含义是：A 的执行结果对 B 可见，并且 A 排在 B 之前（不会被重排到 B 之后）。它是一种可见性 + 有序性的保证，不是墙上时钟的先后。反过来也成立：即使 A 在时间上确实早于 B 执行，只要它们之间没有 happens-before 关系，B 也可能看不到 A 的修改。面试问"happens-before 是不是先后顺序"，标准答案是"不是时间顺序，是可见性顺序"。

更妙的是它的传递性：A hb B 且 B hb C，则 A hb C。这让你能把一串规则串成链条，把可见性从一个变量"传导"到一堆变量上。

A hb B   ∧   B hb C   ⟹   A hb C

四、happens-before 的核心规则

《JCIP》第 16 章列出了这几条你必须记住的规则。绝大多数并发正确性推理，都是这几条 + 传递性拼出来的：

规则内容典型用途
程序顺序同一个线程内，前面的操作 hb 后面的操作（单线程语义）串起同线程内的其他规则
volatile 规则对 volatile 变量的写，hb 后续对同一变量的读状态发布、标志位
监视器锁规则对一把锁的 unlock，hb 后续对同一把锁的 locksynchronized 的可见性来源
线程启动规则Thread.start() hb 新线程内的所有操作主线程准备的数据子线程可见
线程终止规则线程内所有操作 hb 其他线程从 join() 成功返回收集子线程的计算结果
中断规则interrupt() hb 被中断线程检测到中断取消 / 中断协作
传递性A hb B、B hb C ⟹ A hb C把上面的规则连成链
联系 JDK 源码：这些规则不是纸上谈兵。AbstractQueuedSynchronizer（AQS，ReentrantLock 的底层）用一个 volatile int state 承担同步状态的可见性；ConcurrentHashMap 的 Node.val / Node.next 是 volatile；线程池 ThreadPoolExecutor 靠 lock 的 unlock→lock 规则让 worker 之间看到彼此对任务队列的修改。你平时用的并发工具，正确性都建立在这几条 happens-before 之上。

五、volatile 的杀手锏：搭便车

volatile 最被低估的能力，不是"让这一个变量可见"，而是用一次 volatile 写，把它之前的一大堆普通变量修改"一起"发布出去。这个技巧叫 piggyback（搭便车），是安全发布对象的关键。看这段代码：

SafePublish.java用 volatile 写"发布"一整个对象
class SafePublish {
    private Config config;              // 普通字段，非 volatile
    private volatile boolean ready = false;  // 唯一的 volatile

    // 线程 A：初始化并发布
    void init() {
        config = new Config(8080, "prod");  // ① 普通写
        ready = true;                       // ② volatile 写（栅栏）
    }

    // 线程 B：读取
    void use() {
        if (ready) {                        // ③ volatile 读
            config.connect();               // ④ 此刻 config 一定已完整初始化
        }
    }
}

为什么线程 B 一定能看到完整初始化好的 config，即使 config 自己不是 volatile？靠的正是 happens-before 的链式推理：

① config = new Config(...) [线程A, 普通写] │ 程序顺序规则 (① hb ②) ▼ ② ready = true [线程A, volatile 写] │ volatile 规则 (② hb ③) ← 关键的一跳，跨线程 ▼ ③ if (ready) [线程B, volatile 读] │ 程序顺序规则 (③ hb ④) ▼ ④ config.connect() [线程B, 普通读] 传递性把四步连成：① hb ④ ⟹ 线程A 对 config 的初始化，对线程B 完全可见
为什么能"搭便车"？回到第二节的屏障：volatile 写前的 StoreStore 屏障，逼着 config = ... 这个普通写必须先于 ready = true 刷回主内存；volatile 读后的 LoadLoad/LoadStore 屏障，又保证读到 ready == true 后，才去读 config。于是普通变量 config "免费"搭上了 volatile 的可见性顺风车。这就是 DCL 单例里给 instance 加 volatile 能修好"半初始化对象"问题的根本原因——它建立了构造完成到读取之间的 happens-before 边。

六、经典用武之地：可停止的线程

volatile 最教科书的正例，是用一个标志位优雅地停止线程——它完美契合"一写多读、写不依赖旧值"的条件：

✗ 普通 boolean
class Worker implements Runnable {
  private boolean stop = false;
  public void run() {
    while (!stop) {
      doWork();
    }  // JIT 可能把 stop 提到
       // 寄存器里，改了也看不见
  }
  void shutdown() {
    stop = true;  // 可能永不生效
  }
}
✓ volatile boolean
class Worker implements Runnable {
  private volatile boolean stop
      = false;
  public void run() {
    while (!stop) {   // 每轮从主内存读
      doWork();
    }
  }
  void shutdown() {
    stop = true;  // 立即对 run() 可见
  }
}
陷阱三：volatile 对"引用"生效，对"引用指向的内容"不生效。volatile int[] arr 保证的是 arr 这个引用的可见性，而不是 arr[0] 这些元素的可见性——写 arr[0] = 1 不会触发 volatile 语义。同理，把一个 可变对象（如 volatile HashMap map）用 volatile 发布，只保证你拿到的是最新的 map 引用，之后往里 put 的内容依然没有可见性保证。要按下标做原子可见访问，得用 AtomicIntegerArray / VarHandle；要共享可变集合，请用 ConcurrentHashMap。

七、volatile vs synchronized：怎么选

两者都能提供可见性与有序性，但代价和能力边界差别很大。选型口诀：能用 volatile 解决的可见性问题，别动用锁。

维度volatilesynchronized / Lock
原子性✗ 仅单次读或写✓ 整个临界区
可见性✓✓（unlock→lock 规则）
有序性✓（内存屏障）✓
是否阻塞否，无锁无上下文切换是，可能挂起线程
能否协调多变量✗ 保证不了多个变量的一致性✓ 可保护一组不变式
开销极低（读几乎零成本）较高（锁竞争 / 唤醒）
适用状态标志、安全发布、一写多读复合操作、多变量一致、读改写
陷阱四：volatile 无法保护"多个变量之间的不变式"。比如 volatile int lower, upper; 想维持 lower <= upper 这个约束，两个字段各自 volatile 也没用——线程可能在你改完 lower、还没改 upper 时插进来读到不一致的中间态。只要正确性依赖多个变量的联合状态，或依赖变量的当前值去算新值，就必须上锁，volatile 一定不够。
一句话收束：volatile 是"半个同步"——它用内存屏障买来可见性与有序性，却买不来原子性，所以只配用在"一写多读、写不依赖旧值、无需多变量一致"的场景（标志位、安全发布、单例引用）。而它之所以能成立，是因为 JMM 给了你一纸叫 happens-before 的契约：A hb B 意味着 A 的结果对 B 可见且排在 B 之前，而非时间上先发生。掌握"程序顺序 + volatile 规则 + 锁规则 + 传递性"这几条，你就能像推公式一样证明一段并发代码的可见性——这正是从"背 API"跨到"懂并发"的分水岭，也是下一篇《发布与逸出》讨论对象如何被安全交出去的理论地基。
