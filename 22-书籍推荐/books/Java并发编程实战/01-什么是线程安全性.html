什么是线程安全性

并发编程的全部难题，最后都能归到一句话：多个线程访问同一份可变状态时，程序还能不能给出正确的结果。

一、先给一个能落地的定义

很多人对"线程安全"的第一反应是"加了锁就安全"，这其实把因果搞反了。Brian Goetz 在书里给出的定义不谈任何实现手段，只谈行为：

线程安全的定义（JCIP 第 2 章）：当多个线程访问某个类时，不管运行时环境采用何种调度方式、也不管这些线程将如何交替执行，并且在主调代码中不需要任何额外的同步或协同，这个类都能表现出正确的行为，那么这个类就是线程安全的。

这个定义有三个关键词值得咀嚼："正确"指行为与其规约（specification）一致——你说好返回计数值，就得返回真实的计数值；"任何调度"意味着你不能假设线程会以某种"友好"的顺序执行，JVM 和 OS 有权在任意两条字节码之间切走线程；"不需要额外同步"是说责任在类内部——把线程安全的重担甩给每一个调用方，等于没有线程安全。

为什么"无状态"对象天生线程安全？线程安全问题的根源永远是共享的可变状态（shared mutable state）。一个没有任何字段、只依赖方法入参和局部变量的对象（如很多无状态的 Spring @Service），每个线程都有自己的一份栈帧和局部变量，彼此隔离，根本不存在"共享"，自然不用同步。这正是 Spring 默认把 Bean 设成单例还敢多线程共享的底气——只要你别在里面加可变字段。

二、三条主线：原子性 / 可见性 / 有序性

"正确"二字听着抽象，落到 JVM 层面，一段并发代码要正确，必须同时守住三道关。它们是本书后续所有工具（synchronized、volatile、锁、原子类）真正在解决的底层问题：

特性它保证什么被破坏时的现象典型武器
原子性
Atomicity		一组操作要么全做完、要么全不做，中间不被别的线程插入更新丢失（如计数器少算）synchronized、Lock、Atomic 类（CAS）
可见性
Visibility		一个线程改了共享变量，别的线程能立刻看到新值读到旧值、循环停不下来volatile、synchronized
有序性
Ordering		程序执行的顺序符合预期，不被重排序坑到对象"半初始化"、DCL 单例出 bugvolatile、synchronized、happens-before
线程安全 = 原子性 + 可见性 + 有序性
建立联系：这三条不是 Java 发明的概念，而是所有共享内存并发模型都要面对的。原子性来自"操作可被中断"的硬件事实；可见性来自 CPU 有多级缓存、写入不会瞬间对所有核可见；有序性来自编译器与 CPU 为了性能会重排指令。Java 用内存模型（JMM）把这三件事统一抽象成一套规则，让你不用关心 x86 还是 ARM 也能写出可移植的并发代码。

三、原子性：i++ 的三副面孔

最经典的反例就是一个共享计数器被多线程 ++。你眼里 count++ 是一行代码，CPU 眼里它是读—改—写三步，任意两步之间都可能被切走：

count++ 实际被拆成三步（对应字节码 getfield / iadd / putfield）： ① read : 从主内存读出 count 当前值到工作内存 ② modify : 在工作内存里 +1 ③ write : 把结果写回主内存 两个线程交错时会"更新丢失"： count = 5 线程A ① 读到 5 ────────────┐ 线程B ① 读到 5 ──┐ │ 线程B ②③ 写回 6 ┘ │ 线程A ②③ 写回 6 ←──────────┘ ← A 覆盖了 B，本该是 7，结果是 6
✗ 非原子：会少算
class Counter {
  private long count = 0;
  // count++ = 读+改+写，三步非原子
  public void inc() { count++; }
  public long get() { return count; }
}
// 两个线程各 +10000 次，
// 结果常常 < 20000
✓ 用 CAS 原子类
class Counter {
  private final AtomicLong count
      = new AtomicLong(0);
  // incrementAndGet 底层是
  // CAS 自旋，硬件级原子
  public void inc() {
    count.incrementAndGet();
  }
  public long get() {
    return count.get();
  }
}
陷阱一：volatile 不能救原子性。初学者常以为给 count 加个 volatile 就万事大吉——错。volatile 只保证可见性和有序性，管不了"读—改—写"这个复合操作的原子性。两个线程照样能同时读到同一个旧值再各自 +1。计数场景请用 AtomicLong（CAS）或 synchronized，别指望 volatile。
联系 JDK 源码：AtomicLong.incrementAndGet() 底层走的是 Unsafe.getAndAddLong，本质是一个 compareAndSwap（CAS）自旋循环——"我读到的还是旧值吗？是就把它换成新值，不是就重来"。它把"读改写"压缩成一条硬件原子指令（x86 上是带 lock 前缀的 cmpxchg），既拿到原子性又避免了锁的阻塞开销。这就是后面「Atomic 类与 CAS 原理」一节的引子。

四、可见性：那个永远停不下来的循环

可见性问题比原子性更隐蔽，因为它平时"看起来"没事。下面这段代码里，主线程把 stop 改成 true，工作线程却可能永远看不到，一直空转：

NoVisibility.java缺乏可见性 → 死循环
class NoVisibility {
    private boolean stop = false;   // 没有 volatile

    void runWorker() {
        new Thread(() -> {
            while (!stop) {           // 可能一直读到缓存里的旧值 false
                // 空转……主线程改了也看不见
            }
        }).start();
    }

    void shutdown() { stop = true; }  // 写入可能只停在自己的工作内存
}

为什么会这样？现代 CPU 每个核心都有自己的高速缓存，JMM 把它抽象成"每个线程有一份工作内存，共享变量存在主内存"。线程读变量时可能读的是工作内存里的副本，写变量时也可能先攒在工作内存里不立刻刷回。没有可见性保证，一个线程的修改对另一个线程就是"透明"的。

┌──────────── 主内存 (Main Memory) ────────────┐ │ stop = true │ └───────▲───────────────────────────▲──────────┘ │ 写回(可能延迟) │ 读取(可能读旧副本) ┌─────────┴────────┐ ┌─────────┴────────┐ │ 线程1 工作内存 │ │ 线程2 工作内存 │ │ stop = true ✔ │ │ stop = false ✘ │ ← 看不到别人的修改 └──────────────────┘ └──────────────────┘ volatile 的作用：写立即刷回主内存 + 读强制从主内存拉最新值 （底层是插入内存屏障，并使其他核缓存失效）
正解：把字段声明为 private volatile boolean stop; 即可。volatile 写会强制刷回主内存并让其他 CPU 缓存行失效，volatile 读会强制从主内存拉取最新值，从而保证可见性。synchronized 也能达到同样效果——退出同步块时把工作内存刷回主内存，进入时清空工作内存重新读。所以"用锁保护的变量"天然具备可见性。
陷阱二：这个 bug 在你本机可能"复现不出来"。可见性问题依赖硬件、JIT 优化和运行时机——debug 模式、加了打印语句、CPU 核数不同，都可能让它"碰巧正常"。并发 bug 最大的恶意就是不稳定复现。所以正确性不能靠"跑了几次没事"来判断，必须靠对 JMM 规则的推理来保证。别把"测不出"当成"没问题"。

五、有序性：编译器和 CPU 会偷偷重排

为了榨干性能，编译器、JIT 和 CPU 都可能在不改变单线程执行结果的前提下调整指令顺序（这叫 as-if-serial：单线程内怎么重排你都察觉不到）。但一旦多线程共享变量，重排就可能暴露出来：

Reordering.java重排序导致的诡异结果
int a = 0, b = 0;
int x = 0, y = 0;

// 线程1                 // 线程2
a = 1;                  b = 1;
x = b;                  y = a;

// 直觉上 x==0 && y==0 不可能出现，
// 但由于两行之间没有依赖，CPU 可能把 x=b 排到 a=1 之前，
// 于是 (x=0, y=0) 真的会偶发出现

有序性被破坏最著名的翻车现场是双重检查锁（DCL）单例。instance = new Singleton() 看似一步，实则三步：① 分配内存 → ② 调用构造器初始化 → ③ 把引用赋给 instance。如果 ②③ 被重排成 ①③②，另一个线程就可能拿到一个已非 null 但还没初始化完的"半成品"对象：

✗ 无 volatile 的 DCL
class Singleton {
  private static Singleton
      instance;   // 无 volatile
  static Singleton get() {
    if (instance == null) {
      synchronized(Singleton.class){
        if (instance == null)
          // 可能被重排，别的线程
          // 拿到半初始化对象
          instance = new Singleton();
      }
    }
    return instance;
  }
}
✓ 加 volatile 禁止重排
class Singleton {
  private static volatile
      Singleton instance;
  static Singleton get() {
    if (instance == null) {
      synchronized(Singleton.class){
        if (instance == null)
          // volatile 插入内存屏障，
          // 保证 ①②③ 不被重排
          instance = new Singleton();
      }
    }
    return instance;
  }
}
happens-before：把有序性讲清楚的那把尺子。JMM 不会禁止所有重排（那样太慢），而是给出一组 happens-before 规则：只要 A happens-before B，A 的结果就一定对 B 可见、且不会被重排到 B 之后。常用的几条——程序顺序规则、volatile 写先于后续读、unlock 先于后续 lock、线程 start() 先于线程内所有操作。它同时管住了可见性和有序性，是后面「JMM: happens-before 规则」的核心，先记住这个名字。

六、别把"线程安全"想窄了

不同手段覆盖的特性并不一样，选型时要对号入座，而不是无脑加 synchronized：

手段原子性可见性有序性会阻塞吗
volatile✗ 单读/单写才算✓✓否
Atomic 类 (CAS)✓✓✓否（自旋）
synchronized / Lock✓✓✓是
final 字段—✓（安全发布）✓（构造完成前禁重排）否
不可变对象 / 无状态✓（无需修改）✓✓否
陷阱三：线程安全的类拼在一起，整体不一定线程安全。ConcurrentHashMap 的每个方法都线程安全，但 if (!map.containsKey(k)) map.put(k, v); 这两步之间仍会被别的线程插入——这是一个复合操作，需要用 putIfAbsent 这样的原子方法，或额外加锁。线程安全不能"叠加"，它是针对具体操作序列而言的。这正是下一篇「竞态条件与复合操作」要展开的主题。

七、什么时候才需要关心它

把判断流程记成一条决策链，能帮你避免"过度加锁"这种常见的性能自残：

这个变量/对象会被多个线程访问吗？ │ ├─ 否 → 不用管（局部变量、线程封闭） ← 最省心，优先设计成这样 │ └─ 是 → 它是可变的吗？ │ ├─ 否（不可变/final）→ 天然安全，无需同步 │ └─ 是 → 必须同步： · 只读写单个变量 → volatile / Atomic · 读改写 or 多变量一致 → 锁 / synchronized
一句话收束：线程安全的本质，是让一段代码在任意线程调度下、无需调用方额外协同都能给出正确结果；而"正确"在 JVM 层面就落在三根支柱上——原子性（操作不被打断）、可见性（改动能被看见）、有序性（顺序不被坑）。它们的共同病根是共享的可变状态，所以第一优先级永远是消灭共享或消灭可变（无状态、线程封闭、不可变对象），实在要共享可变，再用 volatile / Atomic / 锁按需补齐这三条。记住：并发 bug 不稳定复现，正确性靠对 JMM 的推理，而不是靠"多跑几遍没崩"。
