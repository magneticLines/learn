性能权衡：上下文切换、内存同步、锁竞争

多线程从来不是"免费加速"——它先向你收三笔隐藏税：切换、同步、竞争。看懂这本账，才知道该往哪里砍。

一、先分清两件事：性能 ≠ 可伸缩性

很多人一提"多线程"就等同于"更快"，这是最大的误解。用线程的本意是提升性能，但线程本身也要花钱：创建、调度、协调、内存同步都有开销。一旦这些开销超过了并行带来的收益，多线程反而更慢。要谈清楚，得先拆开两个常被混为一谈的目标。

目标问的问题怎么衡量优化手段
性能
（performance）		"做同一件事有多快 / 用多少资源"服务时间、延迟、单位资源吞吐算法、少做重复工作、减少开销
可伸缩性
（scalability）		"加机器 / 加 CPU，处理能力能不能等比例涨"吞吐量、单位时间完成量减少串行部分、降低共享争用
它们经常互相打架。为了单次调用更快，我们常做缓存、复用对象、把工作合并——这些手段往往引入更多共享状态和协调，反而拖累可伸缩性。反过来，为了可伸缩，我们把大任务拆成许多小任务并行，拆分与协调本身又增加了单次工作的开销。所以第一课是：先想清楚你要优化哪一个。服务端应用通常优先可伸缩性——单个请求慢一点没关系，能扛住十倍流量才要命。
陷阱：先测量，再优化，别凭感觉。并发性能问题极其反直觉——你以为的瓶颈九成不是真瓶颈。"避免不成熟的优化，首先使程序正确，然后再加快——如果它还不够快的话"（JCP 原话）。用 profiler、vmstat（看上下文切换次数）、JMH 基准测试拿数据说话；任何"我觉得加锁很慢所以改成无锁"的改动，不带 benchmark 就是耍流氓。

二、Amdahl 定律：串行部分是并行的天花板

为什么加了 8 个核，速度却远远到不了 8 倍？因为几乎任何程序都有无法并行、必须串行执行的部分（比如从共享队列里取任务、写同一份结果、加锁临界区）。Amdahl 定律给出了理论上限：

Speedup ≤ 1 / ( F + (1 − F) / N )

其中 F 是必须串行的比例，N 是处理器数量。让 N → ∞，加速比的极限就是 1/F——只要有 5% 串行，哪怕给你无限核，最多也就快 20 倍。串行部分才是真正的敌人。

直观感受：串行占比 5% vs 25%（假设 N=8）

F=5% → 加速 ≈ 5.9× 被串行拖住
F=25% → 加速 ≈ 2.9× 被串行拖住
建立联系：这解释了上一篇《并发容器》的所有努力。Hashtable 用一把大锁保护整表 —— 整个 put/get 都是串行的，F 很大，加再多线程也上不去。ConcurrentHashMap 把锁粒度缩到"单个桶"，绝大多数写操作互不冲突、可以并行，大幅压低了 F，于是可伸缩性一个数量级地领先。所有降低锁竞争的技巧，本质都是在做一件事：把 Amdahl 公式里的 F 往下压。

三、线程要收的三笔"隐藏税"

引入线程能并行干活，但调度和协调线程本身要消耗资源。主要有三笔开销，下面逐一拆解：

开销它是什么什么时候被触发
① 上下文切换CPU 从一个线程切到另一个，要保存/恢复现场，还伴随缓存失效可运行线程数 > CPU 数；线程阻塞被挂起时
② 内存同步synchronized/volatile 插入的内存屏障，刷新缓存、禁止重排序每次进出同步块、读写 volatile 变量
③ 阻塞（锁竞争）抢不到锁的线程被挂起，涉及一次或多次上下文切换 + OS 调度多个线程争抢同一把锁（竞争激烈时）

四、上下文切换：调度不是免费的午餐

单个 CPU 核心同一时刻只能跑一个线程。当"可运行线程"多于核心数，OS 就得轮流给它们分配时间片。切换时，内核要保存当前线程的寄存器、程序计数器、栈指针等执行现场，再加载下一个线程的现场——这段纯管理开销叫上下文切换（context switch）。

一次上下文切换发生了什么： 线程 A 运行中 ──► 时间片用完 / A 阻塞在锁上 │ ├─ ① 保存 A 的现场（寄存器、PC、栈指针）到内核 ├─ ② 内核调度器挑选下一个线程 B ├─ ③ 加载 B 的现场，切换到 B 的地址空间 └─ ④ B 开始跑，但 CPU 缓存里全是 A 的数据…… └─► ⑤ B 频繁 cache miss，要从主存重新加载 ← 真正的大头 直接成本：现场保存/恢复 + 调度 ≈ 几千~上万时钟周期（几微秒） 间接成本：缓存被冲刷，新线程"冷启动" ← 常常比直接成本更贵
真正贵的不是"切换"本身，而是"缓存失效"。直接的切换成本约 5000~10000 个时钟周期（几微秒）。但更隐蔽的代价是：线程 A 跑的时候，L1/L2 缓存里装满了它的热数据；切到 B 后，这些缓存对 B 毫无用处，B 要一次次 cache miss 去主存里捞数据（一次主存访问就是上百个周期）。所以刚被换上的线程，头一小段时间是"缓慢的冷启动"。这也是为什么 频繁切换 比"少数几次长任务"更伤性能。
陷阱：线程数不是越多越好，"过度切换"会让你越加线程越慢。很多人以为线程越多吞吐越高，于是开几百个线程。结果 CPU 大量时间耗在切换和缓存冷启动上，真正干活的时间反而变少——这叫thrashing（颠簸）。经验法则：CPU 密集型任务，线程数 ≈ 核心数 + 1；IO 密集型任务因为线程常在等待，可以适当多开。在 Linux 上用 vmstat 看 cs（context switch）列，若它高得离谱，就是信号。这也串起了后面《线程池》为什么要精心设定 corePoolSize。

五、内存同步：屏障的开销，和它没那么可怕的真相

synchronized 和 volatile 的性能开销，来自它们插入的内存屏障（memory barrier）。屏障有两个作用（回忆《volatile 与 happens-before》那一篇）：一是刷新缓存 / 使缓存失效，保证可见性；二是禁止特定的指令重排序，保证有序性。这两件事都会限制 CPU 和编译器的优化空间，所以确实有成本。

但请记住一个反直觉的事实：无竞争的同步，几乎是免费的。初学者常谈"锁"色变，其实现代 JVM 对没有竞争的同步做了极其激进的优化。真正贵的是"竞争"（第六节），不是"同步"这个动作本身。JVM 会用下面这些手段把无谓的同步成本抹掉：
JVM 优化做了什么例子
锁消除
lock elision		逃逸分析发现锁对象是线程私有的，根本不可能被别的线程访问，直接把锁去掉方法内 new StringBuffer() 的 append（自带 synchronized）会被消除
锁粗化
lock coarsening		把相邻的、对同一把锁的多次加解锁合并成一次，减少反复进出屏障循环里连续 append，合并成锁一次
偏向/轻量级锁无竞争时用 CAS 标记而非重量级 OS 互斥量，几乎零成本；竞争出现才升级见《synchronized 与内置锁》的锁升级
逃逸分析 → 锁消除这段的 synchronized 会被 JIT 抹掉
public String concat(String a, String b) {
    // StringBuffer.append 是 synchronized 方法
    StringBuffer sb = new StringBuffer();  // sb 是局部变量，不会逃逸出本方法
    sb.append(a);   // JIT：既然没人能拿到 sb，加锁毫无意义
    sb.append(b);   // → 锁被"消除"，这几次同步的开销归零
    return sb.toString();
}
// 结论：不要因为"怕慢"就手动把 StringBuffer 换成 StringBuilder 来
// 规避同步——在这种局部场景，JVM 已经帮你消除了。真正该换的理由是语义清晰。
陷阱：volatile 不是"轻量级 synchronized"，它也有内存屏障成本，只是更小。volatile 读的开销接近普通读，但 volatile 写 要插入 StoreLoad 屏障刷新，比普通写贵。更关键的是——volatile 只保证可见性和有序性，不保证原子性。volatile int i; i++; 依然是"读-改-写"三步，并发下照样丢更新（这是《竞态条件》的老问题）。别为了"省一点同步开销"把该用 synchronized/AtomicInteger 的地方换成 volatile，那不是优化，是引入 bug。

六、阻塞与锁竞争：性能的真正黑洞

前面说无竞争同步很便宜；那贵在哪？贵在竞争（contention）。当线程 B 想要的锁正被 A 持有，B 就得等。JVM 有两种等法：

  • 自旋等待（spin-waiting）：B 在一个循环里反复试探锁——不放弃 CPU。适合锁马上就会释放的场景（等待时间 < 一次上下文切换成本）。
  • 挂起（suspension）：B 被 OS 挂起、让出 CPU，等锁释放时再被唤醒。适合要等很久的场景，但唤醒/挂起各带一次上下文切换，还要经过内核调度。
锁竞争把"并行"退化成了"串行 + 排队开销"： 无竞争： A ─临界区─► B ─临界区─► (各干各的，快) 有竞争： A ─────临界区─────►│ └─ B 被挂起（切换出去）… 等 A 解锁 └─ B 被唤醒（再切换回来）─临界区─► ▲ ▲ 上下文切换×2 串行执行，N 核也没用 竞争越激烈 → 挂起/唤醒越频繁 → 上下文切换越多 → Amdahl 里的 F 越大
看穿本质：锁竞争同时点燃了前面全部三笔开销。一次激烈的锁争用，会让线程阻塞挂起（触发上下文切换）、让临界区串行化（撑大 Amdahl 的 F）、还伴随缓存在多核间来回失效同步。所以"降低锁竞争"是并发性能优化里回报率最高的一件事——它一箭三雕。这也正是下一篇《锁分段与 ConcurrentHashMap 的设计》要深入的主题。

七、降低锁竞争的三板斧

锁上的竞争程度，由两个因素决定：锁被请求的频率 × 每次持有锁的时间。想减少竞争，就从这两处下手，共三种手段：

① 缩短持有锁的时间（把锁的范围收窄）

只把真正访问共享状态的那几行放进同步块，与共享状态无关的计算、IO、日志统统挪到锁外面。锁持有时间越短，别人排队越少。

✗ 整个方法加锁，无关操作也占着锁
public synchronized
        boolean userMatches(String id) {
    // 下面这些都不碰共享状态，
    // 却全程霸占着锁！
    String prefix = expensiveParse(id);
    log.info("checking " + prefix);
    return map.get(prefix)      // ← 只有这行真的读共享状态
              .contains(id);
}
✓ 只锁真正访问共享数据的那一行
public boolean userMatches(String id) {
    // 无关的重活放在锁外，谁都不用等
    String prefix = expensiveParse(id);
    log.info("checking " + prefix);
    Set<String> s;
    synchronized (this) {        // 临界区极小
        s = map.get(prefix);
    }
    return s.contains(id);      // 读的是局部引用，安全
}

② 降低请求锁的频率（锁分解 / 锁分段）

如果一把锁保护了多个互相独立的变量，那它就成了瓶颈——访问 A 的线程和访问 B 的线程被同一把锁互相拖住。把它拆成多把锁，各管各的，冲突概率立降。

✗ 一把锁保护两份不相关的状态
public class ServerStatus {
    private final Set<String> users;
    private final Set<String> queries;

    // 加/删 user 和加/删 query 抢同一把锁 this
    public synchronized void addUser(String u)  { users.add(u); }
    public synchronized void addQuery(String q) { queries.add(q); }
    // 明明互不相干，却被迫排队
}
✓ 锁分解：一份状态一把锁
public class ServerStatus {
    private final Set<String> users;
    private final Set<String> queries;

    public void addUser(String u) {
        synchronized (users)  { users.add(u); }
    }
    public void addQuery(String q) {
        synchronized (queries){ queries.add(q); }
    }
    // 操作 users 与操作 queries 现在可以并行
}
锁分解 → 锁分段（lock striping）。锁分解是把一把锁拆成"两三把、每把管一块固定字段"。当一把锁保护的是一个可无限增长的容器（如一整张哈希表）时，可以更进一步：把数据切成 N 份，用 N 把锁，用元素的 hash 决定落到哪把锁——这就是锁分段。这正是 JDK 7 ConcurrentHashMap 的 Segment 设计（默认 16 段 = 16 把锁，理论上 16 个写线程并行）。代价是：需要"锁全表"的操作（如精确 size()、扩容）反而变复杂、变慢——又是一次典型取舍。下一篇细讲。

③ 用"能并发的协调机制"替代独占锁

最彻底的一招：干脆别用独占锁。根据场景换成允许更高并发度的工具——很多时候能把竞争降到零。

用它替代 synchronized适用场景为什么更好
ReadWriteLock读多写少读读不互斥，多个读线程可同时进
Atomic 类（CAS）单个计数器 / 引用无锁，靠 CAS 自旋，无挂起开销
LongAdder高并发计数（热点）分片累加，把单点竞争摊到多个 cell
不可变对象 / ThreadLocal状态可拆分或只读压根没有共享，也就没有竞争
并发容器共享集合内部已用分段锁 / CAS，比外部加锁强
把独占锁计数器换成无锁 Atomic高竞争下差距悬殊
// 反例：每次自增都要抢锁、可能挂起
private long count = 0;
public synchronized void inc() { count++; }

// 正例：CAS 自旋，无锁，无上下文切换
private final AtomicLong count = new AtomicLong();
public void inc() { count.incrementAndGet(); }

// 更强：竞争极高时 AtomicLong 的 CAS 也会频繁自旋失败
// LongAdder 把计数分散到多个 cell，各线程更新不同 cell，几乎不冲突
private final LongAdder count = new LongAdder();
public void inc() { count.increment(); }   // sum() 时才把各 cell 相加
陷阱：小心"伪竞争"和"过度优化"两个反方向的坑。其一，别看到"竞争"就无脑上无锁 CAS——竞争极其激烈时，CAS 会不停自旋失败重试，白烧 CPU，此时反而可能不如让线程挂起（这正是 LongAdder 存在的理由）。其二，别为了减少上下文切换而搞"忙等待"自旋占满 CPU。任何优化都要拿 benchmark 验证，因为并发性能高度依赖具体的竞争程度、核数、JVM 版本——脱离测量的优化，往往把简单代码改复杂了，性能还没变好。
一句话收束：多线程不是免费加速，它先收三笔税——上下文切换（切换直接成本 + 缓存冷启动，别开太多线程）、内存同步（屏障成本，但无竞争的同步已被 JVM 用锁消除/锁粗化/偏向锁抹得几乎为零）、阻塞与锁竞争（真正的性能黑洞，一次激烈争用同时点燃前两笔开销并撑大 Amdahl 的串行比例 F）。Amdahl 定律 Speedup ≤ 1/(F + (1−F)/N) 告诉我们：串行部分才是并行的天花板，一切优化的终极目标就是压低 F。压 F 的三板斧是——缩短锁持有时间、锁分解/锁分段降低请求频率、用读写锁/原子变量/不可变对象/并发容器替代独占锁。而所有这些的前提只有一条铁律：先测量，用数据说话，再优化。
