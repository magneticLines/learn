Java 内存模型（JMM）：happens-before 规则

JMM 不是硬件缓存的说明书，而是 JVM 与你签的一份"可见性合同"——happens-before 就是这份合同的全部条款。

一、为什么需要一个"内存模型"

先问一个反直觉的问题：下面这段代码，线程 2 有没有可能打印出 x = 0？

Reordering.java看似不可能，实则合法
int x = 0;
boolean ready = false;

// 线程 1
x = 42;          // ①
ready = true;      // ②

// 线程 2
if (ready) {         // ③
    print(x);       // ④  竟可能打印 0！
}

答案是：会。原因有二：其一，编译器 / CPU 可能把 ① ② 重排序（它们在单线程里无依赖，交换顺序不影响单线程结果）；其二，即使不重排，线程 1 写的 x=42 也可能还躺在它那颗 CPU 核的写缓冲区里，没刷回主内存，线程 2 读到的是旧值 0。

三重"捣乱者"：一次 x = 42 从源码到真正被别的线程看见，要过三道会打乱顺序的关卡——①编译器重排序（JIT 为优化调整指令顺序）；②CPU 指令级重排序（乱序执行、流水线）；③内存系统重排序（store buffer、缓存未同步，写"延迟可见"）。这三者叠加，多线程下"所见顺序"可以千奇百怪。JMM 存在的意义，就是在这片混沌之上，划出一条你能依赖的确定性边界。

关键矛盾在于：这些重排序在单线程下是安全的——编译器 / CPU 会遵守 as-if-serial（貌似串行） 语义，保证单线程程序的运行结果和顺序执行一模一样。可一旦有第二个线程来观察，它看到的中间状态就可能"穿帮"。JMM 要回答的正是这个问题：一个线程的写，到底在什么条件下，对另一个线程保证可见？

二、JMM 的抽象模型：主内存与工作内存

JMM 用一个抽象模型来描述多线程下的内存交互：所有共享变量存在主内存（main memory）；每个线程有自己的工作内存（working memory），里面是它用到的共享变量的副本。线程只能操作自己的副本，不能直接读写别人的工作内存。

┌───────────────── 主内存 (Main Memory) ─────────────────┐ │ 共享变量 x=42 ready=true │ └───────────────────────────────────────────────────────┘ ▲ 刷回 / 读取 ▲ 刷回 / 读取 │ (何时同步？由 JMM 规则决定) │ ┌────────┴─────────┐ ┌────────┴─────────┐ │ 线程1 工作内存 │ │ 线程2 工作内存 │ │ x'=42 ready'=? │ │ x'=? ready'=? │ └──────────────────┘ └──────────────────┘ 线程1 线程2 注意：主内存/工作内存是【逻辑抽象】，不等于 RAM/L1缓存/寄存器， 只是用来【定义可见性规则】的概念模型。
陷阱一：别把 JMM 的"工作内存"当成真实硬件。很多人以为工作内存 = CPU 的 L1/L2 缓存、主内存 = 物理 RAM，这是错的。JMM 是一层规范（specification），它只规定"程序的执行结果必须满足哪些约束"，至于 JVM 用寄存器、缓存还是别的手段去实现，它一概不管。用硬件去"具象"JMM 只能帮助建立直觉，真正做正确性推理时，请只依据 happens-before，而不是脑补缓存刷新时机。

三、happens-before：整套模型的核心契约

如果每写一段并发代码都要去推演编译器、CPU、缓存三层重排序，人根本扛不住。所以 JSR-133（Java 5 起重写的内存模型）把这一切抽象成一个与硬件无关的偏序关系——happens-before。它是你和 JVM 之间的合同：只要你建立起 A happens-before B，JVM 就保证 A 的结果对 B 可见，且 A 不会被重排到 B 之后。

陷阱二（高频面试）：happens-before ≠ 时间上"先发生"。这个名字极具误导性。A hb B 的真正含义是"A 的结果对 B 可见，且在语义上 A 排在 B 前"，是一种可见性 + 有序性保证，不是墙上时钟的先后。反例：即便 A 在时间上确实早于 B 执行，只要两者间没有 happens-before 关系，B 依然可能看不到 A 的修改（就像第一节那段代码）。面试标准答话："happens-before 描述的是可见性顺序，不是发生时间顺序。"

它最强大的性质是传递性：能把一条条规则串成链，把可见性从一个变量"接力"传导到一批变量上。这正是安全发布得以成立的地基。

A hb B   ∧   B hb C   ⟹   A hb C

四、happens-before 的全部规则

《JCIP》第 16 章列出了 JMM 内建的这几条规则。几乎所有并发正确性推理，都是它们 + 传递性拼出来的。记住这张表，你就掌握了 JMM 的"语法"：

规则内容典型用途
程序顺序同一线程内，书写在前的操作 hb 书写在后的操作串起同线程内其他规则
监视器锁对一把锁的 unlock hb 后续对同一把锁的 locksynchronized 的可见性来源
volatile对 volatile 变量的写 hb 后续对同一变量的读状态标志、安全发布
线程启动Thread.start() hb 新线程内的任何操作主线程备好的数据子线程可见
线程终止线程内所有操作 hb 其他线程从 join() 成功返回 / 检测到它已终止收集子线程计算结果
中断interrupt() hb 被中断线程发现中断（抛异常或查到中断位）取消 / 中断协作
终结器对象构造函数结束 hb 该对象 finalize() 的开始GC 回收前的清理
传递性A hb B、B hb C ⟹ A hb C把上面规则连成链
联系 JDK 源码：这些规则不是纸上谈兵。AbstractQueuedSynchronizer（AQS，ReentrantLock / CountDownLatch 的底层）用一个 volatile int state 承担同步状态的可见性；ConcurrentHashMap 的 Node.val / Node.next 都是 volatile；Thread.join() 内部靠对象锁 + isAlive() 实现终止规则。你日常用的每一个并发工具，正确性都建立在这几条 happens-before 之上——它们把"底层怎么插屏障"这件苦活封装掉了，只把契约暴露给你。

五、用 happens-before 破解开篇的 bug

回到第一节那段会打印 0 的代码。它错在线程 1 的写和线程 2 的读之间没有任何 happens-before 边——程序顺序规则只在单个线程内有效，跨不过线程边界。修法：给 ready 加 volatile，用 volatile 规则搭一座跨线程的桥。

✗ 无 hb 边，可能读到 x=0
int x = 0;
boolean ready = false;

// 线程1
x = 42;         // ①
ready = true;   // ② 可与①重排

// 线程2
if (ready)      // ③
  print(x);     // ④ 可能是 0
// ①与④之间没有 hb 关系
// → 结果未定义
✓ volatile 搭出 hb 链
int x = 0;
volatile boolean ready = false;

// 线程1
x = 42;         // ①
ready = true;   // ② volatile 写

// 线程2
if (ready)      // ③ volatile 读
  print(x);     // ④ 一定是 42
// ①hb②(程序序) hb③(volatile)
//   hb④(程序序) → 传递: ①hb④
① x = 42 [线程1, 普通写] │ 程序顺序规则 (① hb ②) ▼ ② ready = true [线程1, volatile 写] │ volatile 规则 (② hb ③) ← 关键一跳，跨线程 ▼ ③ if (ready) [线程2, volatile 读] │ 程序顺序规则 (③ hb ④) ▼ ④ print(x) [线程2, 普通读] 传递性把四步连成：① hb ④ ⟹ 线程1 对 x 的写，对线程2 完全可见（打印 42）
"搭便车（piggyback）"：注意 x 自己并不是 volatile，却借着 ready 这次 volatile 写，把可见性"免费"传导了出去。原理是 volatile 写前有 StoreStore 屏障（逼 x=42 先刷回），volatile 读后有 LoadLoad/LoadStore 屏障（逼读到 ready 后才读 x）。这正是 DCL 单例给 instance 加 volatile 能修好"半初始化对象"的根本原因——下一篇会专门拆解。

六、数据竞争：JMM 的"未定义行为"红线

JMM 用 happens-before 精确定义了什么叫数据竞争（data race）：两个线程访问同一个非 volatile 变量、至少一个是写、且这两个访问之间没有 happens-before 关系——这就是一次数据竞争。含数据竞争的程序，JMM不保证任何结果（类似 C++ 的 undefined behavior）。

race = 同址 ∧ 至少一写 ∧ 无 hb 序

反过来，如果一个程序正确同步（correctly synchronized）——即所有对共享数据的冲突访问都被 happens-before 排好序——那么 JMM 保证它表现得像顺序一致（sequentially consistent）：所有线程看到的操作顺序一致，仿佛没有任何重排。这条被称为 DRF-SC 保证（Data-Race-Free ⟹ Sequential Consistency），是现代内存模型的黄金定律。

一句话记住：你不需要理解重排序的每个细节。只要消除数据竞争（给共享访问建立 happens-before），JMM 就还你一个"没有重排、顺序一致"的世界。这就是为什么"该同步的地方都同步了"之后，你可以放心地用直觉去推理并发代码。

七、被忽视的第三种手段：final 字段语义

建立可见性的手段，除了 volatile 和锁，还有第三种常被忽略的——final 字段。JSR-133 给 final 加了一条特殊保证：只要对象在构造期间没有发生 this 逸出，那么其他线程即使通过数据竞争拿到该对象引用，也一定能看到构造函数中对 final 字段写入的正确值，无需任何额外同步。

FinalFieldSafe.javafinal 保证构造完成后的可见性
class Holder {
    private final int[] data;      // final：构造后不可变引用

    Holder() {
        data = new int[]{1, 2, 3};  // 构造函数内写 final
    }                                // ← 构造结束时插入 freeze 屏障

    int first() { return data[0]; }   // 任何线程读到都是 1，无需同步
}

这条语义正是不可变对象（Immutable Object）天然线程安全的底层依据（呼应第 7 篇《不变性》）：String、Integer、BigDecimal 全靠 final 字段做到"发布即安全"。它也是"安全发布四种方式"里"用 final 域"那一种的原理。

陷阱三：final 的保证有前提——构造期间 this 不能逸出。如果你在构造函数里把 this 注册到某个监听器、丢进静态集合、或启动一个引用了 this 的线程，别的线程就可能拿到一个 final 字段还没写完 的半成品对象，final 的可见性保证当场失效。所以构造函数里永远不要让 this 提前跑出去（this-escape）。

八、三种"同步手段"横向对比

volatile、锁、final 都能建立 happens-before 边，但能力边界与代价差别很大。选型口诀：能用 final 就别用 volatile，能用 volatile 就别上锁。

手段建立的 hb 边原子性适用场景代价
final构造结束 hb 引用可见后的读—（只读）不可变对象、安全发布几乎为零
volatile写 hb 后续读✗ 仅单次读/写状态标志、一写多读低（读近乎免费）
synchronized / Lockunlock hb 后续 lock✓ 整个临界区复合操作、多变量一致较高（竞争/挂起）
陷阱四：happens-before 只约束"有依赖关系"的操作，不是全局排序。初学者常误以为"加了 volatile / 锁，所有代码就都按写的顺序跑了"。其实 JMM 只保证被 happens-before 串起来的那些操作有序、可见；毫无关系的两个操作，JVM 依然可以自由重排。它是一张偏序（partial order）网，不是全序（total order）——这也是为什么它能在保证正确性的同时，把优化空间尽量留给编译器和 CPU。
一句话收束：JMM 是一份规范而非硬件说明书，它把"编译器 + CPU + 缓存"三层重排序的混沌，收敛成一个你能依赖的抽象——happens-before。记住三件事就够用：①A hb B 意味着 A 的结果对 B 可见且排在 B 前，而非时间上先发生；②所有可见性都来自那几条规则（程序顺序 / volatile / 锁 / start / join / final）+ 传递性拼成的链；③只要消灭数据竞争，DRF-SC 就还你一个顺序一致的世界。掌握这套"契约语言"，你就能像推公式一样证明并发代码的正确性——这正是从"背 API"迈向"懂并发"的分水岭，也是下一篇拆解双重检查锁（DCL）为何非 volatile 不可的理论地基。
