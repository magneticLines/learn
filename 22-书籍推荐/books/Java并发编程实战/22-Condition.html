Condition：比 wait/notify 更灵活的等待通知

一把锁挂多个等待队列——把「谁该醒」这件事讲清楚，从此告别 notifyAll 的惊群与白醒。

一、先看 wait/notify 的痛：一间屋子，两种人在等

内置锁 synchronized 配套的 wait/notify 有一个结构性限制：每个对象只有一个隐式的「条件队列」。也就是说，无论线程是因为「缓冲区满了」在等，还是因为「缓冲区空了」在等，它们全都睡在同一个队列里。

后果很直接——你想唤醒「等不满的生产者」，却没办法只叫醒他们；notify() 随机挑一个（可能挑到不该醒的消费者），notifyAll() 则把所有人全叫醒，让它们抢锁、发现条件还不满足、再睡回去。这就是惊群（thundering herd）与白醒（无谓唤醒）。

生活类比：一家银行只有一个叫号大厅（单条件队列），办「存款」和办「取款」的人混坐在一起。柜员每处理完一笔，就冲大厅喊一嗓子「下一位！」（notifyAll）——结果 50 个人全站起来挤到窗口，49 个发现叫的不是自己的业务，又坐回去。Condition 做的事，就是给存款、取款各设一个叫号厅，需要谁就精准喊哪个厅。

二、Condition 是什么：Lock 的「可拆分等待室」

Condition 是 java.util.concurrent.locks 包里的接口，由 Lock.newCondition() 创建。它把 wait/notify/notifyAll 三件套原样搬到了显式锁上，方法名对应为 await/signal/signalAll。关键差异：一把 Lock 可以 newCondition() 出任意多个 Condition，每个 Condition 都是一条独立的等待队列。

内置锁（synchronized）显式锁（Lock + Condition）说明
obj.wait()cond.await()释放锁并挂起，被唤醒后重新竞争锁
obj.wait(ms)await(t, unit) / awaitNanos / awaitUntilCondition 的限时等待更丰富、返回值更好用
obj.notify()cond.signal()唤醒该条件队列的一个等待者
obj.notifyAll()cond.signalAll()唤醒该条件队列的全部等待者
一个对象 = 一个条件队列一个 Lock = N 个 Condition这是 Condition 存在的根本理由
wait 可被中断额外提供 awaitUninterruptibly()可选择「不响应中断」地等待，wait 做不到
1 个 synchronized → 1 条隐式队列 ┃ 1 个 Lock → N 个 Condition

三、经典战场：有界缓冲区（生产者-消费者）

这是《Java 并发编程实战》第 14 章的招牌案例。缓冲区有两个截然不同的等待原因：放不进去（满）和取不出来（空）。用 Condition 正好把它们拆成两个队列。先看用单条件 wait/notify 的写法为什么别扭，再看双 Condition 如何一击即中。

Bad · 单条件 + notifyAll
synchronized void put(T x) throws ... {
  while (count == cap)
    wait();            // 生产者、消费者
                       // 睡在同一队列
  doPut(x); count++;
  notifyAll();         // 只能全叫醒：
                       // 醒来的可能又是
                       // 生产者 → 白醒
}
synchronized T take() throws ... {
  while (count == 0)
    wait();
  T x = doTake(); count--;
  notifyAll();         // 惊群：N 醒 1 用
  return x;
}
Good · 双 Condition + signal
void put(T x) throws ... {
  lock.lock();
  try {
    while (count == cap)
      notFull.await();   // 只等"不满"
    doPut(x); count++;
    notEmpty.signal();   // 精准叫消费者
  } finally { lock.unlock(); }
}
void take() throws ... {
  lock.lock();
  try {
    while (count == 0)
      notEmpty.await();  // 只等"不空"
    doTake(); count--;
    notFull.signal();    // 精准叫生产者
  } finally { lock.unlock(); }
}
BoundedBuffer.java正例 · 完整版
public class BoundedBuffer<T> {
    private final Lock lock      = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();  // 队列①：等"不满"的生产者
    private final Condition notEmpty = lock.newCondition();  // 队列②：等"不空"的消费者
    private final Object[] items = new Object[100];
    private int count, putIdx, takeIdx;

    public void put(T x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length)     // ← while 而非 if！防虚假唤醒
                notFull.await();               // 释放锁 + 挂起；返回时已重新持有锁
            items[putIdx] = x;
            if (++putIdx == items.length) putIdx = 0;
            count++;
            notEmpty.signal();              // 只唤醒 notEmpty 队列上的一个消费者
        } finally {
            lock.unlock();                    // 必须放 finally，否则异常时锁永不释放
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await();
            Object x = items[takeIdx];
            if (++takeIdx == items.length) takeIdx = 0;
            count--;
            notFull.signal();               // 只唤醒 notFull 队列上的一个生产者
            return (T) x;
        } finally {
            lock.unlock();
        }
    }
}
它就在 JDK 源码里： 上面这个双 Condition 结构，几乎逐字复刻了 ArrayBlockingQueue 的实现——打开源码你会看到同一把 ReentrantLock 派生出的 notEmpty 与 notFull。这也解释了「为什么 put/take 阻塞时能被精准唤醒而不惊群」。学会 Condition，你就读懂了半个 java.util.concurrent。

四、原理：AQS 的「同步队列」与「条件队列」两套链表

要真正讲透，得看 ReentrantLock 背后的 AQS（AbstractQueuedSynchronizer）。Condition 的实现类是 AQS 的内部类 ConditionObject。这里有一个精妙的双队列模型：争锁靠「同步队列」，等条件靠「条件队列」，await/signal 做的就是节点在两条队列间的搬运。

ReentrantLock (AQS 内部) ┌───────────────────────────────────────────────────────────┐ │ 同步队列 (sync queue, CLH) —— 争"锁"失败的线程在这排队 │ │ head → [n1] → [n2] → tail │ │ │ │ 条件队列 (每个 Condition 各一条) │ │ notFull : firstWaiter → [w1] → [w2] → lastWaiter │ │ notEmpty : firstWaiter → [c1] → lastWaiter │ └───────────────────────────────────────────────────────────┘ await() : ①完全释放锁 ②把当前线程包成 Node 挂到"条件队列"尾 ③LockSupport.park() 挂起 （人离开窗口去等待室） signal() : ①把"条件队列"队首 Node 摘下 ②转移(transfer)到"同步队列"尾部 ③它在那儿排队，轮到时被 unpark 重新拿锁 （叫回窗口排队）

看懂这张图，几个「反直觉」瞬间就通了：

  • 为什么 await() 会释放锁？ 因为它要让出临界区给别人干活，否则「等条件成立」永远等不到——谁来改变条件呢？这一点和 Object.wait() 完全一致。
  • 为什么 signal() 后被唤醒线程不会立刻执行？ 因为它只是从条件队列被转移到了同步队列，还得重新排队抢锁，抢到才继续。所以 signal() 并不释放锁，唤醒的线程要等 signal 方最终 unlock() 后才有机会。
  • 为什么 await/signal 必须在锁内调用？ 因为它们操作的是 AQS 的共享链表状态，靠「当前线程持有锁」来保证串行安全。没持锁就调，直接抛 IllegalMonitorStateException。
串联操作系统： 这套机制不是 Java 发明的——它对应操作系统里的条件变量（condition variable），即 POSIX 的 pthread_cond_wait / pthread_cond_signal。C 里条件变量必须搭配一把 mutex，pthread_cond_wait 会「原子地释放锁并睡眠」；Java 的 Condition 必须搭配 Lock、await 会「释放锁并挂起」，是同一个思想的对象化封装。底层的 park/unpark 最终落到 futex 等内核原语。

五、陷阱：新手最容易踩的四个坑

坑 ①：用 if 判断条件，而不是 while。 必须写 while (条件不满足) cond.await();。原因有二：一是虚假唤醒（spurious wakeup），线程可能没被 signal 也醒来；二是被唤醒 ≠ 条件仍成立——你从条件队列转到同步队列、再抢到锁，这中间可能别的线程又把资源抢光了。if 只判一次就往下走，会在错误状态上操作。「循环里重检条件」是等待通知的铁律。
坑 ②：signal 用错了 Condition，或该用 signalAll 却用 signal。 若多个不同条件共用一个 Condition，signal 只叫醒一个、且可能叫醒「条件并不满足的那个」，导致信号丢失、系统卡死。安全法则：拿不准就用 signalAll；只有当「所有等待者等的是同一个条件、且一次只能让一个前进」时，用 signal 才安全（这也是有界缓冲区能用 signal 的前提——notFull 队列里全是等「不满」的生产者）。
坑 ③：忘了 finally { unlock() }。 与 synchronized 出块自动释放不同，Lock 必须手动解锁。await() 内部会临时全释放锁、返回时重新加锁；但如果你的业务代码在持锁期间抛异常又没 finally，锁就永久泄漏，其他线程全卡死——这比死锁更难查。
坑 ④：在没持有该 Lock 时调用 await/signal。 直接抛 IllegalMonitorStateException（和 synchronized 块外调 wait 报的是同一个异常）。另外要分清：lock.newCondition() 产出的 Condition 必须配那把 lock 使用，别张冠李戴。

六、对比与选型：什么时候真的需要 Condition

维度wait / notify（内置锁）Condition（显式锁）
等待队列数量每对象仅 1 条每 Lock 可多条，按语义拆分
唤醒精度只能 notify/notifyAll，易惊群可对特定队列 signal，精准
限时/可中断/不可中断等待仅 wait(ms)，且必被中断await 系列齐全，含 awaitUninterruptibly
公平性不可选随 ReentrantLock(true) 可公平
释放锁责任自动（出 synchronized 块）手动 finally{unlock()}
复杂度 / 出错概率低，够用就别上高，用错反而更危险
务实选型： 如果只有一种等待条件（比如「等某个标志位变 true」），synchronized + wait/notifyAll 更简单、更不容易写错，优先用它。只有当同一把锁上存在两种及以上、需要分别精准唤醒的等待条件时（生产者/消费者、读者/写者），Condition 的多队列能力才真正物有所值。更上层：能用 ArrayBlockingQueue、LinkedBlockingQueue 这些现成并发容器，就别自己撸 Condition——它们已经替你踩平了所有坑。
一句话收束： Condition 的核心价值是把「一把锁上的多种等待原因」拆成多条独立队列，从而实现精准唤醒、消灭惊群与白醒。它是 wait/notify 在显式锁上的进化版：底层靠 AQS 的「同步队列 ⇄ 条件队列」节点搬运实现，await 释放锁并挂到条件队列，signal 把节点转回同步队列重新抢锁。用它时牢记三条铁律——while 循环重检条件、拿不准用 signalAll、unlock 放 finally。
