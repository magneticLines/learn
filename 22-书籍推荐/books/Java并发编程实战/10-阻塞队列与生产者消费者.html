阻塞队列与生产者消费者

一条会"堵"的传送带：满了就让上料的人歇一会儿，空了就让取料的人等一等——它把两拨线程之间那点最难写对的握手协议，悄悄替你办完了。

一、先想清楚：生产者消费者到底解决什么问题

假设你在写一个日志系统。业务线程负责生成日志（生产者），另一个线程负责把日志刷到磁盘（消费者）。最朴素的写法是业务线程自己写盘——但磁盘 IO 慢，会拖垮业务响应。自然的想法是：业务线程只管把日志"扔到一个中间地带"，写盘线程从中间地带慢慢取。这个"中间地带"就是队列，这套协作模式就是生产者-消费者（Producer-Consumer）。

它的核心价值是"解耦（decoupling）"。生产者不需要知道消费者是谁、有几个、处理多快；消费者也不关心数据从哪来。双方只通过一个队列打交道。这带来三个红利：① 平滑削峰——突发流量先堆进队列，消费者按自己的节奏消化；② 独立伸缩——想提速就多加几个消费者线程，生产者代码一行不用改；③ 关注点分离——生产逻辑和消费逻辑各自独立演进、独立测试。

难点在哪？在于那个队列必须被多线程同时读写，而且要处理两个尴尬的边界：队列空了消费者该怎么办？队列满了生产者该怎么办？如果用忙等（busy-wait）不停轮询，会白白烧 CPU；如果处理不好线程间的等待/唤醒，又极易写出竞态和死锁。阻塞队列（BlockingQueue）就是把这套"等待-唤醒"协议封装到位的现成构件。

二、BlockingQueue：会"堵"的队列

java.util.concurrent.BlockingQueue 在普通 Queue 的基础上，多了两个会"阻塞"的核心方法。这就是它名字的由来：

生产者 阻塞队列(容量有限) 消费者 ┌──────────┐ ┌───┬───┬───┬───┬───┐ ┌──────────┐ │ put(x) │ ─────► │ D │ C │ B │ A │ │ ─────► │ take() │ └──────────┘ 尾入 └───┴───┴───┴───┴───┘ 头出 └──────────┘ ▲ ▲ 队列满 → put 阻塞 队列空 → take 阻塞 (生产者被迫等待) (消费者被迫等待) put() : 队列满时，生产者线程挂起，直到有空位才放入并返回 take(): 队列空时，消费者线程挂起，直到有元素才取出并返回

之所以说它"优雅"，是因为它把流量控制（backpressure，背压/反压）天然内建了进去：当消费者跟不上时，队列会满，put() 自动把生产者顶住，逼它慢下来。这是有界队列送给你的一份大礼——它让系统的负载有了上限，而不是无限堆积直到内存耗尽。

建立直觉：把它想成餐厅的"出餐口"。厨师（生产者）把做好的菜放到出餐台（队列），服务员（消费者）来端走。出餐台就那么大：菜堆满了，厨师就得停手等台子空出来（put 阻塞）；台子空了，服务员就在旁边候着（take 阻塞）。谁也不用一直盯着对方，出餐台自己就把节奏协调好了。

陷阱一：别再用 add() 往有界队列硬塞。很多人从 ArrayList 的习惯出发直接 add()，一旦队列满就抛 IllegalStateException("Queue full")，把线程搞挂。生产者要么用 put()（想让它阻塞、形成背压），要么用带超时的 offer()（想快速失败、丢弃或降级）。同理，消费者别用 poll() 后不判 null 就直接用——空队列返回的是 null，一个 NPE 在等着你。

三、手写 wait/notify vs 直接用 BlockingQueue

在 BlockingQueue 出现之前（JDK 1.5 前），生产者-消费者要靠 synchronized + wait()/notifyAll() 手写。这段代码是并发里最容易踩坑的经典场景之一，我们把"反例的坑"和"正例的清爽"摆在一起看：

✗ 手写 wait/notify，坑点密集
// 共享缓冲区
final Queue<Task> buf = new LinkedList<>();
final int CAP = 10;

// 生产者
synchronized (buf) {
  // ① 必须用 while 不能用 if!
  while (buf.size() == CAP) {
    buf.wait();     // 释放锁并挂起
  }
  buf.add(task);
  buf.notifyAll();  // ② 只能 notifyAll,
}                   //   用 notify 可能唤错线程

// 消费者
synchronized (buf) {
  while (buf.isEmpty()) {   // 又是 while
    buf.wait();
  }
  Task t = buf.poll();
  buf.notifyAll();
}
// 坑: 忘了 while → 虚假唤醒;
//     用了 notify → 丢信号/死锁;
//     忘了 notify → 对方永远醒不来
✓ 用 BlockingQueue，协议全内建
BlockingQueue<Task> buf =
    new ArrayBlockingQueue<>(10);

// 生产者
buf.put(task);   // 满了自动阻塞

// 消费者
Task t = buf.take();  // 空了自动阻塞

// 就这些。
// 等待/唤醒、虚假唤醒、
// 满/空判断,全部由队列内部
// 用 ReentrantLock + Condition
// 帮你处理好了
陷阱二：wait() 永远要放在 while 循环里，不能用 if。因为存在虚假唤醒（spurious wakeup）——线程可能在没有 notify 的情况下自己醒来；而且 notifyAll() 会唤醒所有等待者，但只有一个能拿到资源，其余的醒来后发现条件仍不满足，必须重新检查并再次等待。用 if 只判一次就往下走，会直接操作一个仍处于满/空状态的队列。这正是 BlockingQueue 帮你规避的头号大坑。

四、六种实现，怎么选

BlockingQueue 是接口，JUC 给了多个实现。选型的关键维度是：有界还是无界、底层结构、是否排序、锁的粒度。

实现结构 / 边界特点与典型场景
ArrayBlockingQueue数组 / 有界（必须指定容量）单把锁 + 两个 Condition。容量固定、内存可控，最常用的有界选择
LinkedBlockingQueue链表 / 可选有界（默认 Integer.MAX_VALUE ≈ 无界）双锁（putLock / takeLock）读写分离，吞吐更高；不指定容量则近乎无界，慎用
PriorityBlockingQueue堆 / 无界按优先级出队而非 FIFO；只有 take 会阻塞，put 永不阻塞（会扩容）
SynchronousQueue无容量 / 无缓冲不存元素，一个 put 必须等一个 take 当面"交接"；用于线程间直接传递
DelayQueue堆 / 无界元素到期才能取出；定时任务、缓存过期、延迟重试的利器
LinkedTransferQueue链表 / 无界SynchronousQueue 与 LinkedBlockingQueue 的融合，transfer() 可等消费者接手
ArrayBlockingQueue vs LinkedBlockingQueue，面试常问的取舍：
· 锁粒度：ArrayBQ 全程一把 ReentrantLock，入队和出队互斥；LinkedBQ 把锁拆成 putLock 和 takeLock，生产者和消费者可以并行操作两端，高并发下吞吐更好。
· 内存：ArrayBQ 创建时一次性分配数组，无额外节点对象，GC 压力小、内存可预测；LinkedBQ 每个元素都要 new Node，产生更多小对象。
· 边界：ArrayBQ 强制有界；LinkedBQ 不传容量就是"伪无界"（上限 Integer.MAX_VALUE），生产快于消费时会悄悄堆到 OOM。

五、一个能跑的例子：多消费者 + 毒丸关停

光看 API 没感觉，来一段完整的生产者-消费者。重点看两处：多个消费者共享一个队列天然做到负载均衡；以及如何"优雅关停"——用毒丸（poison pill）这个经典技巧。

ProducerConsumerDemo.java1 生产者 · 3 消费者 · 毒丸关停
import java.util.concurrent.*;

public class ProducerConsumerDemo {
    // 有界队列：容量 10，天然形成背压
    static final BlockingQueue<String> queue = new ArrayBlockingQueue<>(10);
    // 毒丸：一个约定好的"结束"标记对象
    static final String POISON = "__EOF__";
    static final int CONSUMERS = 3;

    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(CONSUMERS + 1);

        // —— 消费者：循环 take，取到毒丸就退出 ——
        for (int i = 0; i < CONSUMERS; i++) {
            pool.submit(() -> {
                try {
                    while (true) {
                        String item = queue.take();     // 空则阻塞
                        if (item.equals(POISON)) {
                            queue.put(POISON);          // 关键：把毒丸再放回去,
                            break;                       // 让其他消费者也能收到
                        }
                        System.out.println(Thread.currentThread().getName() + " 处理 " + item);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt(); // 恢复中断状态，别吞掉
                }
            });
        }

        // —— 生产者：生产 20 个任务，最后投一颗毒丸 ——
        pool.submit(() -> {
            try {
                for (int i = 1; i <= 20; i++) queue.put("task-" + i); // 满则阻塞
                queue.put(POISON);   // 只投一颗，靠消费者接力传递
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        pool.shutdown();
    }
}
毒丸（poison pill）为什么优雅？关停多线程消费者是个老大难：直接 interrupt 可能打断正在处理的任务，加个 volatile boolean running 标志又会遇到"消费者正卡在 take() 上根本检查不到标志"的问题。毒丸把"结束信号"当成队列里的一个普通元素顺着数据流走——它保证消费者是在处理完前面所有真实任务后才收到结束通知，不会丢数据。上面的写法只投一颗毒丸、消费者取到后再放回队列，像接力棒一样把关停信号传遍所有消费者。

六、掀开盖子：ArrayBlockingQueue 的源码骨架

阻塞队列不是魔法。它内部就是我们上一章学的 ReentrantLock + Condition——把手写 wait/notify 那套逻辑封装了起来。看一眼 ArrayBlockingQueue 的核心字段和 put 就全懂了：

java.util.concurrent.ArrayBlockingQueue（节选）JDK 源码骨架
final Object[] items;          // 底层数组，容量创建时定死
final ReentrantLock lock;      // 全局一把锁：入队出队互斥
private final Condition notEmpty; // 条件：队列非空（唤醒消费者）
private final Condition notFull;  // 条件：队列未满（唤醒生产者）

public void put(E e) throws InterruptedException {
    Objects.requireNonNull(e);
    final ReentrantLock lock = this.lock;
    lock.lockInterruptibly();          // 可中断地加锁
    try {
        while (count == items.length) // ← 注意：还是 while！源码也防虚假唤醒
            notFull.await();             // 队列满，生产者在 notFull 上等待（并释放锁）
        enqueue(e);                     // 放入元素，count++
    } finally {
        lock.unlock();
    }
}
// enqueue() 末尾会调用 notEmpty.signal()，精准唤醒一个等待的消费者
和手写 wait/notify 一一对应，但更聪明：手写只有一个条件队列（对象的 monitor），生产者和消费者混在一起等，只能 notifyAll() 全部叫醒、大家再抢。而 Condition 让 ArrayBQ 拆出 notFull 和 notEmpty 两个独立的等待集：入队后只 signal notEmpty（只叫消费者），出队后只 signal notFull（只叫生产者）。精准唤醒避免了无谓的上下文切换——这正是 Condition 相对 Object.wait/notify 的核心优势，也解释了"一个 Lock 可绑定多个 Condition"的设计意图。
陷阱三：LinkedBlockingQueue 的双锁不是免费午餐。它用 putLock 和 takeLock 让两端并行，吞吐更高，但代价是：跨越两把锁的操作（如 size()、remove(o)、迭代器）需要同时获取两把锁，成本更高；而且它用了一个 AtomicInteger count 来在两把锁之间同步元素计数。别以为链表队列一定比数组队列快——在低竞争、小容量场景，ArrayBQ 反而因为单锁简单、无节点分配而更快。

七、和线程池的血缘：它就是池的"腰部"

你几乎每天都在间接使用阻塞队列——只要用了线程池。ThreadPoolExecutor 内部就靠一个 BlockingQueue<Runnable> workQueue 缓存待执行任务，核心线程当消费者、提交任务的代码当生产者。工作队列的选型直接决定线程池的行为：

提交任务 execute(task) ───► ThreadPoolExecutor │ ┌──────────────────────────────┼───────────────────────────────┐ ▼ ▼ ▼ 核心线程没满? workQueue 没满? 还能开新线程? → 直接开核心线程跑 → 塞进 BlockingQueue 排队 → 开非核心线程救急 (队列就是这里!) → 否则触发拒绝策略 队列类型决定池的"性格": · LinkedBlockingQueue(无界) → FixedThreadPool：任务无限排队，永不触发拒绝(可能 OOM) · SynchronousQueue → CachedThreadPool：不排队，来一个开一个线程 · ArrayBlockingQueue(有界) → 队列满且线程满 → 触发拒绝策略(推荐生产使用)
陷阱四：Executors.newFixedThreadPool() / newSingleThreadExecutor() 用的是无界 LinkedBlockingQueue。这是《阿里巴巴 Java 开发手册》明令禁止用 Executors 工厂的根本原因：任务生产速度一旦持续高于消费速度，队列会无限堆积，maximumPoolSize 形同虚设，最终 OutOfMemoryError。同理，newCachedThreadPool 用 SynchronousQueue + 近乎无限的线程数，突发流量会瞬间创建海量线程打爆系统。生产环境请手动 new ThreadPoolExecutor(...)，显式传有界队列 + 合适的拒绝策略。

八、双端队列与工作窃取（Deque）

普通阻塞队列是单一入口/出口，所有消费者抢同一个队尾——竞争激烈时锁会成为瓶颈。BlockingDeque（如 LinkedBlockingDeque）两端都能进出，催生了一个更高级的模式：工作窃取（work stealing）。

传统: 所有消费者抢一个共享队列的头 工作窃取: 每个线程有自己的双端队列 ┌──────────────┐ Worker1: [■■■■]◄─自己从头取(LIFO) │ 共享 Queue │◄── W1 W2 W3 抢! Worker2: [■■] └──────────────┘ (队头是热点，锁竞争) Worker3: [ ] ──空了！从别人队尾"偷"─►┐ ◄──────────────────┘ 各干各的队列(无竞争)，闲了才去偷(尾端， 与队主的头端错开，进一步降低冲突)
联系 Fork/Join 与并行流：Java 7 的 ForkJoinPool 正是工作窃取的经典落地——每个 worker 线程维护自己的双端队列，从头部(LIFO，缓存局部性好)取自己拆分出的子任务，空闲时从其他线程队列的尾部(FIFO)偷任务。你用的 Stream.parallel()、CompletableFuture 默认线程池（common pool）背后都是它。工作窃取让大量小任务在多核上尽量少地互相踩锁，负载自动均衡——这是"减少共享、错开竞争"思想的又一次体现。

九、几个方案的取舍速查

把常见的"协作方式"横向排一下，直观感受阻塞队列在光谱上的位置：

忙等轮询 差 手写wait/notify 易错 BlockingQueue 推荐 Deque窃取 高阶
方案CPU 开销正确性风险背压能力适用
忙等 / while 轮询高（空转烧 CPU）低无几乎不用
synchronized+wait/notify低高（虚假唤醒/丢信号）需手写学习原理；不建议手写上线
BlockingQueue低低（协议内建）有界队列天然背压绝大多数生产者-消费者
BlockingDeque 工作窃取低低好大量细粒度任务、Fork/Join
一句话收束：生产者-消费者的本质是用一个队列解耦生产与消费，从而削峰、独立伸缩、关注点分离；而 BlockingQueue 把最难写对的"队空/队满时的等待-唤醒"协议封装成了 put()/take()——底层不过是 ReentrantLock + notFull/notEmpty 两个 Condition，帮你规避了虚假唤醒、丢信号、忙等三大坑。选型抓两条主线：要背压保护就必须用有界队列（ArrayBlockingQueue 或指定容量的 LinkedBlockingQueue），关停多消费者优先用毒丸。再往上，线程池的 workQueue、Fork/Join 的工作窃取双端队列，都是这套构件的延伸——记住"无界队列 = 潜在 OOM"，别再用 Executors 的默认工厂，这一节的价值就落地了。
