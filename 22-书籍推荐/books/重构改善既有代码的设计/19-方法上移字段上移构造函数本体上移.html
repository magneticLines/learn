方法上移、字段上移、构造函数本体上移

兄弟姐妹身上长着同一样东西,就把它搬到父母那儿去——继承体系里消除重复的三把上移手术刀。

一、一个共同的直觉:重复往"上"提

这三个手法(Pull Up Method / Field / Constructor Body)是同一件事的三个变体:当几个子类身上出现完全相同(或几乎相同)的成员时,把它移到共同的父类里,只留一份。 它是继承体系里对付《重复代码》这条坏味道的专用武器。

为什么"上移"是对的方向?因为继承的语义就是"子类是一种父类"——凡是所有子类都共有的东西,本质上属于"这一类事物"的公共特征,理应放在描述"这一类"的父类里。生活类比:如果家里三个孩子都姓"张",那"姓张"这件事不该分别写在三个孩子的档案上,而应写在"张家"这个家族条目里。

重构前:重复散落在子类 重构后:抽取到父类一份 ───────────────────── ───────────────────── Employee Employee (什么都没) + name ← 字段上移 ╱ ╲ + getName() ← 方法上移 Engineer Salesman + Employee(name){...} ← 构造本体上移 + name + name ==> ╱ ╲ + getName + getName Engineer Salesman (两份一样) (两份一样) (只留差异) (只留差异)
一句话抓住本质: 上移手法把子类间的重复"折叠"成父类里的一份。它和「提炼函数」的动机完全一致——消除重复、建立单一真相——只不过舞台从一个类内部换到了一条继承链上。改动方向永远是从叶子往根,从具体往抽象。

二、方法上移(Pull Up Method)

最常见、也最安全的一个。当两个子类有函数体完全相同的方法,把它提到父类,子类里的副本删掉。

Bad · 子类各存一份相同方法
abstract class Employee {
    protected String name;
}

class Engineer extends Employee {
    String getName() {      // 重复
        return name;
    }
}
class Salesman extends Employee {
    String getName() {      // 一模一样
        return name;
    }
}
// 想给 getName 加日志?改两处
Good · 提到父类,只留一份
abstract class Employee {
    protected String name;

    String getName() {      // 上移到此
        return name;
    }
}

class Engineer extends Employee { }
class Salesman extends Employee { }
// 单一入口,改一处即可
关键前提:先统一签名,再上移。 如果两个方法只是"做的事一样"但参数名/返回类型/方法名不同,直接上移会编译失败。正确顺序是先用上一章的「改变函数声明」把它们调成完全一致,再上移。这也是 Fowler 反复强调的"小步前进":每一步都保持编译通过、测试通过。
串联设计模式:模板方法(Template Method)。 方法上移常常引出一个更漂亮的结构:如果两个子类的方法大体相同、只有几步有差异,可以把相同的骨架上移到父类,把差异步骤留成父类里的抽象方法交给子类实现——这就是模板方法模式。JDK 里 java.util.AbstractList 就是典范:它把 indexOf、equals、iterator 等通用逻辑写死在父类,只把 get(int)、size() 留给 ArrayList / Vector 去填。"上移"正是通往这类抽象基类的第一步。

三、字段上移(Pull Up Field)

如果多个子类都声明了用途相同的字段,把它移到父类。它比方法上移多一层坑:字段名可能不一样,但语义相同。

Bad · 同一概念,各叫各的名
class Engineer extends Employee {
    private String name;   // 员工姓名
}
class Salesman extends Employee {
    private String empName; // 也是姓名!
}
// 名字不同,肉眼难看出是重复
// 报表代码得为两种字段各写一套
Good · 先统一命名,再上移
abstract class Employee {
    protected String name;  // 上移的公共字段
}

class Engineer extends Employee { }
class Salesman extends Employee { }
// 统一为 name,子类共享同一份定义
陷阱一:字段名不同会掩盖重复。 字段上移前,务必先用「神秘命名」把 name / empName / userName 这类"同物异名"统一。
陷阱二:别为了上移滥用 protected。 上移后的字段通常要放宽到 protected 让子类可见,但 protected 字段等于向所有子类敞开了内部状态,破坏封装、埋下"子类随意改父类字段"的隐患。更稳妥的做法是字段仍设 private,通过 protected 的 getter/setter 暴露——这与「封装变量」是同一条心法。

四、构造函数本体上移(Pull Up Constructor Body)

前两个手法几乎可以机械照搬,唯独构造函数不能直接上移——因为 Java 对构造函数有硬约束:子类构造函数中,super(...) 必须是第一条语句。所以我们没法像普通方法那样把整段构造逻辑挪走,而是要把"公共初始化"抽成父类构造函数,子类改用 super(...) 委托给它。

子类构造 = super(公共部分) + 各自差异部分
Pull Up Constructor BodyJava · super() 必须第一句
// —— 重构前:两个子类构造函数里塞着相同的初始化 ——
class Manager extends Employee {
    Manager(String name, String id, int grade) {
        this.name = name;   // ┐
        this.id   = id;     // ┘ 与 Engineer 重复
        this.grade = grade;
    }
}

// —— 重构后:公共初始化上移进父类构造函数 ——
abstract class Employee {
    protected final String name;
    protected final String id;

    protected Employee(String name, String id) {  // 抽出公共部分
        this.name = name;
        this.id   = id;
    }
}

class Manager extends Employee {
    private final int grade;
    Manager(String name, String id, int grade) {
        super(name, id);   // 必须是第一句!委托父类做公共初始化
        this.grade = grade;  // 只保留自己独有的部分
    }
}
高频面试陷阱:构造函数里别调用可被覆写的方法! 构造函数本体上移后,父类构造函数若调用了一个被子类覆写的方法,会踩到 Java 的初始化顺序坑:父类构造函数先于子类字段初始化执行。此时虚方法分派会调到子类的覆写版本,而子类的字段还没赋值(全是默认值),轻则逻辑错乱,重则 NullPointerException。这是 Effective Java "构造函数不要调用可覆写方法"条目的经典考点,上移构造逻辑时尤其容易撞上。
反面教材:父类构造调用被覆写的方法Java · 初始化顺序陷阱
class Base {
    Base() { init(); }          // 父类构造里调用可覆写方法
    void init() { }
}
class Sub extends Base {
    private String msg = "hi";
    @Override void init() {
        System.out.println(msg.length());  // NPE! 此刻 msg 还是 null
    }
}
// new Sub(): Base() 先跑 → init() 分派到 Sub 版 → msg 尚未初始化 → 抛 NPE

五、三者对比:同源,不同约束

手法上移对象特有约束 / 注意点
方法上移函数体相同的方法先统一签名;可顺势演化为模板方法模式
字段上移语义相同的字段先统一命名;慎用 protected,优先封装访问
构造本体上移构造函数里的公共初始化不能整体挪,须抽父类构造 + super() 第一句;防止调用可覆写方法
顺序有讲究。 Fowler 建议的落地次序通常是:先字段上移 → 再构造本体上移 → 最后方法上移。因为方法往往依赖字段,字段先就位,方法上移才不会引用到子类里还没搬走的成员;而构造函数初始化的正是这些字段,自然夹在中间。逆着来会不断遇到"父类找不到某成员"的编译错误。

六、方向反过来?——上移不是唯一答案

上移的前提是"重复真的属于所有子类"。如果一段逻辑只被部分子类共享,或者你发现越来越多东西必须靠 instanceof / 条件判断来区分子类,那可能说明继承这条路本身走偏了。这时该考虑的不是继续上移,而是它的反向与替代手法:

信号更合适的手法
成员只被部分子类共用「函数下移 / 字段下移」推回具体子类
子类只是想复用父类实现,并非"is-a"「以委托取代子类 / 取代超类」(见下一节)
继承层级已臃肿、多处靠类型判断「以委托取代继承」,改组合
Spring 生态的印证。 Spring 早期大量使用抽象基类(JdbcDaoSupport、WebMvcConfigurerAdapter)把公共逻辑"上移"给用户继承;但后来的趋势是用委托与组合替代继承——比如 WebMvcConfigurerAdapter 被废弃,改成让你实现 WebMvcConfigurer 接口(Java 8 默认方法)。这印证了"上移"是把双刃剑:它消除重复,却也把子类和父类死死绑在一起。共享逻辑,不必然要靠继承。
核心结论: 方法上移、字段上移、构造本体上移是同一动作的三个变体——把子类间的重复折叠成父类里的一份,建立单一真相。方法上移最简单(先统一签名),字段上移要防"同物异名"与滥用 protected,构造本体上移受 super() 必须第一句的约束、还要提防"构造期调用可覆写方法"的初始化陷阱。落地记住两点:方向从叶到根、小步且每步可编译可测;上移只在"重复真属于全体子类"时才成立——否则,该考虑的是下移,或干脆用委托取代继承。
