依恋情结：一个方法过分依赖另一个类

一个方法总在翻别人家的抽屉——那它八成该搬到别人家里去。

什么是依恋情结（Feature Envy）

Martin Fowler 给它下的定义只有一句话："一个函数跟另一个模块中的数据或函数交流，比跟自己所处模块内部的交流还要频繁。" 说白了，这个方法身在曹营心在汉——它写在 A 类里，心思却全在 B 类身上，不停地调用 B 的 getter、拿 B 的字段来算东西，却几乎不碰自己所在类 A 的任何数据。

核心原理：行为应当和它所操作的数据待在一起。面向对象最根本的一条组织原则，就是把「数据」和「操作数据的行为」封装进同一个类。依恋情结恰恰违背了这一条——行为放在 A，数据却在 B，于是"改数据结构"和"改处理逻辑"被拆到了两处，每次改动都得两头同步。它本质上是低内聚、高耦合：A 和 B 之间被一根看不见的绳子绑死了。

一个生活类比

想象公司里有位员工，工位挂在"市场部"，可他每天一睁眼就跑去"财务部"办公室，整天翻财务的账本、借财务的文件、用财务的电脑。所有人都看得出来：他其实是财务部的人，只是工位放错了地方。合理做法不是让他天天两头跑（沟通成本高），而是把工位挪进财务部。依恋情结的解药——移动函数（Move Function）——就是这一步"调岗"。

识别它：数一数"访问谁"

判断一个方法是否"依恋"别人，有个非常朴素的可操作办法：数它访问了几次别人的数据、几次自己的数据。

依恋度 = 访问「别的类」字段次数 / 访问「自己」字段次数

当这个比值明显 > 1（尤其分母趋近于 0）时，这个方法几乎可以断定"站错了队"。下图把一个典型的依恋方法画出来——它对外张牙舞爪，对内一片空白：

OrderService (方法所在处) Customer (被依恋的类) +----------------------+ +--------------------+ | discount(Customer c) | | loyaltyPoints | | |- c.getLoyalty() -|-------->| vip | | |- c.isVip() ------|-------->| level | | |- c.getLevel() ---|-------->| | +----------------------+ +--------------------+ 对 Customer：访问 3 次 对自己：访问 0 次 结论：discount() 该住进 Customer，而不是 OrderService

解药：Move Function（移动函数）

既然行为依恋的是 Customer 的数据，那就把它搬到 Customer 里去。搬过去之后，方法访问的全是自己的字段，绳子解开了，Customer 也从"只有数据的躯壳"变成了"有血有肉的对象"。

Bad · 行为在 Service，依恋 Customer
class Customer {   // 贫血：只有数据
  int loyaltyPoints;
  boolean vip;
  String level;
  // 全是 getter …
}

class OrderService {
  double discount(Customer c) {
    double d = 0;
    if (c.getLoyalty() > 1000) d += .05;
    if (c.isVip())            d += .10;
    if ("GOLD".equals(c.getLevel()))
                              d += .03;
    return d;  // 全程在读 c 的数据
  }
}
Good · 行为搬回数据所在处
class Customer {   // 数据 + 行为
  int loyaltyPoints;
  boolean vip;
  String level;

  double discount() {  // 访问的全是自己
    double d = 0;
    if (loyaltyPoints > 1000) d += .05;
    if (vip)                  d += .10;
    if ("GOLD".equals(level)) d += .03;
    return d;
  }
}
// 调用方：c.discount()  ——— Tell, Don't Ask
"Tell, Don't Ask"（告诉，别询问）。好代码不该"把对象的数据掏出来、在外面算好、再决定怎么办"，而应直接命令对象"你自己算"。c.discount() 就是"告诉"；service.discount(c) 里一堆 c.getXxx() 就是"询问"。依恋情结几乎总是"询问式代码"的产物——把询问改成告诉，坏味道自然消散。

只有"部分"依恋怎么办：先提炼，再移动

现实里更常见的情况是：一个方法只有一部分逻辑依恋别的类，另一部分确实用的是自己的数据。此时不能整个搬走，而要先用 Extract Function（提炼函数）把"依恋段"切出来，再对切出来的小函数做 Move Function。这也是 Fowler 反复强调的："先决定这个函数的哪一部分该移动过去"。

Refactor.javaExtract → Move
// 重构前：finalPrice 里只有"折扣段"依恋 Customer
double finalPrice(Order o) {
    double base = o.amount();          // 用自己的数据 ✔
    Customer c = o.customer();
    double d = 0;                      // ↓ 这几行全在依恋 Customer
    if (c.getLoyalty() > 1000) d += 0.05;
    if (c.isVip())            d += 0.10;
    return base * (1 - d);
}

// 步骤①：Extract Function，把依恋段提成 discountFor(c)
// 步骤②：Move Function，把 discountFor 挪进 Customer → c.discount()
// 重构后：一行说清意图，各归各位
double finalPrice(Order o) {
    return o.amount() * (1 - o.customer().discount());
}
陷阱一：贫血领域模型（Anemic Domain Model）是依恋情结的"工业化量产"。很多 Spring 项目里，@Entity / DTO 只有一堆 getter/setter，全部业务逻辑塞进 @Service。于是每个 Service 方法都在疯狂 entity.getXxx()——这正是成规模的依恋情结。Fowler 把贫血模型明确称为反模式（anti-pattern）：它挂着面向对象的名号，干的却是面向过程的事——数据是数据、逻辑是逻辑，OO 的封装红利一分没拿到。DDD 提倡的"充血/富领域模型"，本质就是把这些依恋方法搬回实体。
陷阱二：火车残骸（Train Wreck）与迪米特法则。依恋情结常伴随 order.getCustomer().getAddress().getCity() 这种一长串点号调用（被戏称为"火车残骸"）。它违反迪米特法则（Law of Demeter，最少知识原则）：一个对象不该了解另一个对象的内部结构。链条越长，耦合越深——中间任何一环改字段，这行都得跟着改。解法同样是把逻辑"告诉"给最里层的对象自己做。

不是所有"分离"都是坏味道

这里有个高频误区：并非"行为和数据不在一起"就一定要合并。有些设计模式刻意把行为与数据分开，那是深思熟虑的取舍，不能当依恋情结去"治"。

场景 / 模式行为为何与数据分离该不该消除
策略模式 Strategy把可替换的算法族抽成独立对象，便于运行期切换、单测合理，别硬合
访问者模式 Visitor数据结构稳定、却要不断新增操作，把操作外置更易扩展合理，别硬合
Spring Service 编排跨多个聚合的事务/流程编排，本就不属于任何单个实体合理（但别把纯实体逻辑也塞进来）
真·依恋情结只是随手把"某实体自己的逻辑"写进了 Service / 工具类消除：Move Function
怎么区分？问一句："这段逻辑是某一个对象自己的事，还是协调多个对象的事？"折扣算法只关乎单个 Customer——那是它自己的事，搬进去；下单流程要动库存、账户、物流三个聚合——那是编排，留在 Service 才对。依恋情结判的是"错位"，不是"分离"本身。

连接已学：与其他坏味道的关系

依恋情结很少单独出现，它常和相邻章节的坏味道互为因果：

  • 数据类（Data Class）：一个只有字段和 getter/setter、没有任何行为的类，几乎必然导致别处出现依恋它的方法——两者是同一枚硬币的两面。
  • 过长函数：一个大函数里往往藏着若干"依恋不同类"的片段，提炼函数正是发现依恋情结的时机。
  • 发散式变化 / 霰弹式修改：逻辑与数据错位，改一处需求就得改两个类，这正是上一节"霰弹式修改"的温床。
行为在 A · 数据在 B（依恋，来回拉扯） Move Function 后：行为与数据同处一室
一句话收束：依恋情结的病根是行为和它操作的数据被放错了地方——方法"人在 A、心在 B"。解药是 Move Function（部分依恋则"先 Extract 再 Move"），把行为搬回数据所在的类，让代码从"询问"变成"告诉"。它是贫血模型、火车残骸、迪米特法则违背的共同源头；但也记住——策略、访问者等模式的"刻意分离"不是病，别把有意的设计当坏味道去合并。判的是错位，不是分离。
