过长函数与过长参数列表

函数越短越好、参数越少越好——把"活得太久、要得太多"的函数拆开、瘦身，是重构里回报率最高的一招。

为什么"短函数"是重构第一守则

几乎所有资深工程师最后都会得出同一个结论：一个函数活得越久，就应该越短。Martin Fowler 在书里说得很直白——他和 Kent Beck 写代码时，"只要感觉需要用注释来说明点什么，我们就把需要说明的东西写进一个独立函数中，并以其用途（而非实现手法）命名"。这句话是理解本章的钥匙。

核心原理：短函数的价值不在"行数少"，而在于它把"做什么（意图）"和"怎么做（实现）"分离开。当你读一个由一串命名良好的小函数组成的大函数时，就像在读一份目录——每一步意图一目了然，只有真正关心细节时才需要点进去。这正是抽象的本质。

这背后其实是人脑工作记忆的物理限制：心理学上著名的"神奇的数字 7±2"告诉我们，人一次只能在脑子里同时追踪 5~9 个信息块。一个 200 行、嵌套 4 层的函数，需要你同时记住十几个局部变量的状态、几个循环的当前位置、若干标志位——这远远超出了工作记忆的容量，于是你只能反复上下翻页、丢失上下文。短函数把每一块逻辑塞进一个有名字的"信息块"，等于给大脑做了压缩。

过长函数的典型信号

  • 需要写注释来分段（"// 下面是校验"、"// 计算折扣"）——注释就是拆函数的路标
  • 大量临时变量和参数贯穿始终，读到后半段已经忘了前面变量是干嘛的
  • 深层嵌套的 if / for，缩进像楼梯一样往右滑
  • 你想给它起个准确的名字，却发现"它干的事太多，一个名字概括不了"

手法一：Extract Function（提炼函数）

对付过长函数，99% 的场景第一招就是提炼函数：把一段有内聚意图的代码切出去，变成一个命名良好的新函数。下面是一个订单打印逻辑的"反例 → 正例"。

Bad · 一个函数啥都干
void printOwing(Invoice inv) {
  printBanner();
  // 计算欠款
  double outstanding = 0;
  for (Order o : inv.orders)
    outstanding += o.amount;
  // 打印明细
  System.out.println(
    "name: " + inv.name);
  System.out.println(
    "amount: " + outstanding);
}
Good · 主函数只读意图
void printOwing(Invoice inv) {
  printBanner();
  double outstanding =
    calcOutstanding(inv);
  printDetails(inv, outstanding);
}
// 每个小函数名 = 那段注释
double calcOutstanding(Invoice inv){…}
void printDetails(
  Invoice inv, double amt){…}
命名口诀：提炼出的函数按"做什么"命名，不按"怎么做"命名。叫 calcOutstanding（意图）而不是 loopOrdersAndSum（实现）。哪怕新函数只有一行、名字比函数体还长也值得——因为读代码的次数远多于写代码的次数，你在用一个好名字替换掉一次"读懂实现"的成本。
陷阱：不要因为"性能"而拒绝拆函数。很多初学者担心多一层方法调用会变慢。事实上 JVM 的 JIT 编译器（HotSpot C2）有强大的方法内联（inlining）优化：小而热的方法（默认 -XX:MaxInlineSize=35 字节码、热点方法 FreqInlineSize=325）会在运行期被直接"摊平"回调用处，调用开销归零。也就是说——你写的短函数，运行时和一大坨代码几乎一样快，可读性却天差地别。别做过早优化。

过长参数列表：另一种"藏起来的耦合"

参数列表过长（Long Parameter List）常和过长函数结伴出现。一个方法要 6、7 个参数，调用方每次都要费劲拼凑，还极容易传错顺序——尤其是连续几个同类型参数时，createUser(true, false, true) 这种调用鬼才看得懂。

面试高频 & 实战大坑：布尔参数（Flag Argument）。形如 render(true) 的调用，读代码的人根本不知道 true 是什么意思，必须跳进定义去看。参数越多、同类型越多，"传错位置"的 bug 越隐蔽——而且编译器不会报错。这类错误在生产环境极难排查。

三种"瘦身"手法怎么选

手法适用场景怎么做
Replace Parameter with Query
（以查询取代参数）		某参数能从另一个参数推导出来去掉它，在函数内部现算。如同时传 quantity 和 price 又传 total，删掉 total
Preserve Whole Object
（保持对象完整）		多个参数取自同一个对象直接传整个对象，如 f(p.low, p.high) → f(p)
Introduce Parameter Object
（引入参数对象）		几个参数总是一起出现（数据泥团）打包成一个新的值对象/DTO，如 DateRange
Bad · 7 个参数 + 布尔标志
createOrder(
  "张三", "010-123",
  "北京", "朝阳",
  100, true, false);
// true/false 是啥？谁记得住
// 地址三个字段散落各处
Good · 参数对象 + 枚举
createOrder(
  new Customer("张三", "010-123"),
  new Address("北京", "朝阳"),
  100,
  ShipType.EXPRESS);
// 语义自解释，顺序错了编译报错
为什么引入参数对象是"化学反应"级的重构？把 startDate / endDate 打包成 DateRange 后，你会发现原本散落在各处的"日期区间"逻辑（比较、判断包含、算天数）自然而然地被吸附到这个新类里。这就是 Fowler 说的——引入参数对象常常是"更深层结构涌现"的起点，坏味道消除会连锁触发更好的设计。这和面向对象的高内聚原则完全一致。
DateRange.java参数对象吸附行为
// 原本散落在多个方法里的"区间"逻辑，现在有了归宿
public record DateRange(LocalDate start, LocalDate end) {
    public DateRange {                       // 紧凑构造器：把校验也收进来
        if (end.isBefore(start))
            throw new IllegalArgumentException("end 早于 start");
    }
    public boolean includes(LocalDate d) {   // 逻辑被"吸"进来了
        return !d.isBefore(start) && !d.isAfter(end);
    }
    public long days() {
        return ChronoUnit.DAYS.between(start, end) + 1;
    }
}
// 调用方从 "f(start, end)" 变成 "f(range)"，且再也不用重复写校验

联系已学：JDK 与 Spring 里的正反面教材

标准库和框架其实到处是这套思想的印证——好的、坏的都有：

正面：Builder 模式治过长参数。当参数实在多（尤其可选参数多）时，构造器会爆炸成一长串。JDK 的 StringBuilder、Spring 的 UriComponentsBuilder、OkHttp 的 Request.Builder 都用建造者模式把"一长串参数"变成"一串带名字的链式调用"：builder.host("x").port(8080).path("/a").build()——本质就是"引入参数对象"的加强版，用方法名给每个参数打上标签。
反面教材：Thread 的构造器家族。java.lang.Thread 有个 Thread(ThreadGroup group, Runnable target, String name, long stackSize) 的重载，四个参数里 group 和 stackSize 大多数人根本用不到，还得记住顺序。这正是"过长参数列表"在权威源码里的真实存在——连 JDK 也难免。后来 Java 21 的虚拟线程改用 Thread.ofVirtual().name("x").start(r) 的 Builder 风格，就是一次教科书式的"参数列表瘦身"。

拆到什么程度才算够

"函数该多长"没有硬性行数，但有一个可操作的直觉标准，来自 Robert C. Martin 的《代码整洁之道》与 Fowler 的实践共识：

意图 和 实现 之间的语义距离 ≈ 0

也就是：当你读一个函数体，如果每一行都在同一个抽象层级上、都在直接表达函数名承诺的那件事，就不用再拆了；一旦某一行"降到了更细的实现细节"（比如函数叫"结账"却突然出现一段字符串拼接），那段细节就该被提炼出去。下面这个"拆分收益"直觉图能帮你把握火候：

1 个 200 行巨函数 拆成 8 个命名良好的小函数 拆成 40 个碎片(过度)

拆分不是越碎越好：拆到"每个函数是一个完整意图"就停手；拆到"读代码要在几十个一行函数间反复跳转"就过头了，那是另一种坏味道（Middle Man / 过度中间层）。

维度过长函数过度拆分（矫枉过正）
阅读体验翻页、丢上下文反复跳转、迷失在调用链
命名一个名字概括不了硬凑名字，语义重复
信号需要注释分段函数比它的名字还短、只被调一次且无意义
解药Extract FunctionInline Function（内联函数）
一句话收束：过长函数的病根是混淆了"意图"与"实现"，解药是 Extract Function——用函数名承载意图；过长参数列表的病根是把耦合藏在了签名里，解药是 以查询取代参数 / 保持对象完整 / 引入参数对象。两者共同的哲学只有一句：让代码在"该被读的抽象层级"上说话。而 JIT 内联保证了你可以放心地为可读性而拆——运行时几乎不付出代价。
