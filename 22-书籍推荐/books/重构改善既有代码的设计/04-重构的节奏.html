重构的节奏：小步修改加频繁测试

重构不是一次憋大招的手术，而是一连串"改一点、测一下、存个档"的小循环——慢就是快，稳才是真的快。

一、什么是"重构的节奏"

很多人对重构的想象是：关起门来啃三天，把一坨烂代码"一次性"改漂亮。Fowler 在书里反复强调的恰恰相反——重构真正的功夫，在于把一个大改动拆成一串小到几乎不可能出错的步子，每走一步就编译并运行测试，绿了就继续、红了就立刻退回。他自己的原话是："我做重构的步子小到别人觉得可笑，但正是这份小，让我几乎不用花时间调试。"

核心定义：重构的节奏 = 小步修改（small steps） + 频繁测试（frequent testing） + 及时提交（commit as checkpoints）。每一步都保证程序始终处于可运行、测试全绿的状态；任何时刻停下来，代码都是能交付的，而不是一堆改了一半、编译都过不了的残骸。

这里的关键词是"始终可用"。传统开发常常有一段"施工期"——代码被拆得七零八落，几个小时甚至几天里跑都跑不起来。重构刻意消灭这段施工期：它要求你像走钢丝一样，每一小步落脚点都是稳的。

二、为什么要小步？——把"调试"变成"必然定位"

小步的价值不是"稳重"这种玄学，而是一条冷冰冰的工程收益：它把 bug 的搜索范围压缩到了一次改动之内。

假设你测试从绿变红，说明是你刚刚那次改动引入的问题。如果这次改动只有 5 行，你的排查范围就是这 5 行；如果你憋了两小时改了 500 行才第一次跑测试，那 bug 藏在这 500 行的任意角落，你得靠 debug 一点点缩小。前者是"看一眼就知道"，后者是"折磨一下午"。

测试从绿变红时，bug 的可能藏身范围： 小步（改 5 行就测） [■■□□□□□□□□□□□□□□□□□□] ← 范围 ≈ 你刚写的这几行，抬眼即定位 大步（改 500 行才测） [■■■■■■■■■■■■■■■■■■■■] ← 范围 = 整段改动，只能 debug 二分排查
建立联系：这就是 git bisect 的思想被前置到了写代码阶段。git bisect 靠二分法在历史提交里定位是哪一次引入了 bug——提交越小、越频繁，它定位得越精准。小步重构相当于你一边写就一边把提交粒度做到最细，于是根本不需要事后二分，出错点自动就摆在你眼前。频繁的小 commit，就是给未来的自己铺好的排查台阶。

一个生活类比：下楼梯 vs 跳楼梯

小步重构像一级一级下楼梯：每一级都踩实了再迈下一步，就算脚滑也只退回上一级。大步重构像从楼梯口一跃而下：运气好省了几秒，运气不好就是骨折——而且你根本说不清是哪一下摔的。攀岩里有个"三点固定"原则（手脚四个支点始终留三个在岩壁上），重构的小步就是代码世界的三点固定：永远保留一个"随时能退回去的稳定态"。

三、频繁测试：把反馈循环压到秒级

小步之所以能成立，前提是测试要快、要能随手就跑。这也是为什么上一篇《重构的前提》强调"自测试代码（self-testing code）"是重构的地基——没有一套秒级返回的测试，你根本没法频繁测，小步也就失去了意义。

反馈延迟 ↓　→　测试频率 ↑　→　可承受的步子 ↓　→　调试成本 ↓

这是一条正反馈链：测试跑得越快，你就越舍得频繁跑；跑得越频繁，你就越敢把步子迈小；步子越小，出错定位越准。反过来，如果一次测试要跑十分钟，你自然舍不得频繁跑，只能被迫攒大步——节奏就崩了。

陷阱一：拿"慢测试"当频繁测试的挡箭牌。很多团队的单元测试里塞满了真实数据库连接、网络调用、Thread.sleep，跑一轮好几分钟，于是没人愿意频繁跑。这不是"重构不适合我们"，而是测试设计出了问题。真正的单元测试应当是内存里跑的、毫秒级的（依赖用 Mock/Stub 隔离掉）。先治好慢测试，重构的节奏才谈得上。

四、把大改动拆成小步：一个真实示例

看一个具体动作："把一段散落在方法里的金额计算逻辑提炼成独立函数"。新手往往一把梭：剪切、粘贴、改名、顺手把逻辑也优化了，改完一大坨才第一次编译。下面对比"一把梭"和"拆成小步"两种做法。

✗ 一把梭：混着改
// 一次性：提炼 + 改名 + 改逻辑
double calc(Order o) {
  // 直接删掉旧代码，
  // 敲一个全新的 amountFor()，
  // 顺手把 int 改成 BigDecimal，
  // 还把变量 p 改名成 price……
  return amountFor(o);
}
// 改完 60 行才第一次编译
// 测试红了：是提炼错了？
// 改名漏了？还是逻辑动了？
// 三种嫌疑，只能 debug
✓ 拆成小步：每步一测
// 步骤①：只复制，不删原代码
double amountFor(Order o) {
  return o.qty * o.price; // 原样搬
}
// ← 编译 + 测试：绿。commit

// 步骤②：原处改成调用它
double calc(Order o) {
  return amountFor(o);
}
// ← 编译 + 测试：绿。commit

// 步骤③（换顶帽子）再谈优化
// 比如把 double 换成 BigDecimal
// ← 单独一步、单独一测、单独提交

右边每一步都极其"无聊"——无聊到你几乎确信它不可能出错。这正是目的：把风险摊薄到每一步都趋近于零。而且注意步骤③——优化逻辑（改行为）被明确推迟到重构完成之后单独做。这就引出下一个关键纪律。

五、两顶帽子：绝不同时改结构和改行为

Kent Beck 提出的"两顶帽子（Two Hats）"是维持节奏的灵魂纪律。你写代码时永远只戴一顶帽子：

帽子你在做什么行为会变吗测试应如何
重构帽只调整内部结构：改名、提炼、搬移不变（外部行为完全一致）已有测试应始终全绿，不新增测试
加功能帽新增能力、修改业务逻辑会变（这就是目的）先写会红的新测试，再写代码让它变绿
陷阱二：两顶帽子一起戴，是重构翻车的头号原因。如果你一边提炼函数（改结构）一边顺手改了业务逻辑（改行为），当测试变红时你会陷入致命的模糊：到底是重构手滑改坏了行为，还是新逻辑本身有 bug？这两种错误的排查方式完全不同，混在一起就是灾难。纪律是：戴重构帽时，测试红了 = 你一定改错了结构，退回去就对了——因为行为本就该保持不变。这份"红就是我错了"的确定性，只有分开戴帽子才拿得到。
联系 TDD 的红-绿-重构循环：Kent Beck 的 TDD 三步曲——红（写个失败测试）→ 绿（写最脏的代码让它通过）→ 重构（在测试保护下清理代码）——正是"两顶帽子"的时间编排。红和绿两步戴"加功能帽"，第三步戴"重构帽"。重构的节奏，本质上就是把 TDD 里那个"重构"步骤，独立出来放大成一套完整的工作方式。

六、commit 就是你的存档点

小步 + 频繁测试还差最后一块拼图：每走完一个绿色的小步，就 git commit 一次。这些提交是你的"游戏存档点"——一旦某一步玩崩了，一个 git reset --hard 就能瞬间退回最近的稳定态，而不是手忙脚乱地手动撤销那些你已经记不清的改动。

refactor-rhythm.sh重构循环的命令级写照
# 一个重构小步的标准循环
# 1. 做一个小到不可能错的改动（改名 / 提炼 / 搬移）……

# 2. 编译 + 跑测试（要求秒级返回）
mvn -q test-compile test

# 3a. 绿了 → 存档，进入下一小步
git commit -am "refactor: extract amountFor()"

# 3b. 红了 → 别 debug，直接退回最近的绿
git reset --hard HEAD   # 撤销这一步，换更小的步子重来

# 全部改完、确认整体绿之后，再一次性 squash 成一个干净提交
git rebase -i HEAD~8
为什么"红了别 debug、直接退回"往往更划算？直觉上你想"再看看哪错了"，但既然上一步还是绿的、这一步只改了几行，那么丢弃重来的成本常常低于调试的成本。退回去、把步子拆得更小再走一遍，通常几分钟就绕过了原来卡你半天的坑。这也是小步的隐藏福利：每一步都小到"丢了也不心疼"。

七、慢就是快：节奏感的经济账

初学者最大的心理障碍是："这么小步，得多慢啊？"事实恰恰相反。开发时间的大头从来不是敲代码，而是调试。小步几乎消灭了调试，所以总时长反而更短。用一条"耗时构成"直观感受一下：

大步重构的时间构成

写代码 25% 调试排查 60% 返工 15%

小步重构的时间构成

写代码+测试 70% 调试 12% 跑测试等待 18%

小步看起来"多花了"跑测试的时间，但它把那块吓人的红色调试区几乎抹平了。这就是 Fowler 说的"慢就是快（slow is smooth, smooth is fast）"——单步慢一点，全程快很多，而且过程可预测、不会突然掉进调试黑洞。

维度大步重构（憋大招）小步重构（勤测勤提交）
出错定位范围大，靠 debug 二分范围=一步，抬眼即知
回退成本手动撤销，易撤错撤漏git reset 秒回存档点
中途可交付性施工期代码跑不起来任意时刻都全绿可发布
心理负担越改越怕，不敢停每步都稳，随时能停
总耗时看着快，实则被调试拖长看着慢，实则更短更稳
陷阱三：把"重构"和"重写"混为一谈，于是放弃了节奏。一旦你想"这破代码没救了，我推倒重来"，就等于给自己开了一张"长期不可运行"的通行证——这正是小步重构要极力避免的状态。除非旧代码已烂到无法在其上小步前进，否则优先选择在能运行的代码上一小步一小步地改进。重写是高风险豪赌，重构是稳定复利。（这一取舍，正是本章"重构 vs 重写"那一节的主题。）
联系工程实践：小步节奏和 CI/持续集成是同构的。持续集成鼓励"小批量、频繁合并"，道理和小步重构一模一样——批量越小，集成冲突和缺陷定位就越简单。你在本地做的"改一点就测一下、绿了就 commit"，放大到团队尺度就是"每人每天多次向主干提交小改动 + 流水线自动跑测试"。同一种"缩小批量、加密反馈"的智慧，贯穿了从个人编辑器到团队流水线的每一层。
一句话收束：重构的节奏，就是把一个吓人的大改动，拆成一串小到不可能出错的动作，每动一下就编译并跑一次秒级测试，绿了就 git commit 存档、红了就直接退回上一个存档。这套节奏之所以成立，靠三根支柱：快速自测试（让你敢频繁测）、两顶帽子（让"红了=我改错了结构"这条判断永远成立）、频繁提交（让回退成本趋近于零）。它换来的是把开发中最贵的"调试"几乎归零——所以别怕步子小，慢就是快，稳才是真的快。
