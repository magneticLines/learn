什么是重构：不改变外部行为改善内部结构

在不改变程序"能做什么"的前提下，改善它"内部长什么样"——这是每个工程师都该有的日常肌肉记忆。

一句话看懂：重构改了什么、没改什么

"重构"（Refactoring）这个词被滥用得很厉害——不少人把"我把这块代码推倒重写了一遍"也叫重构。Martin Fowler 在书里给了一个非常严格、克制的定义，正是这份克制让它有价值：

重构（名词）：对软件内部结构的一种调整，目的是在不改变软件可观察行为的前提下，提高其可理解性、降低其修改成本。
重构（动词）：使用一系列重构手法，在不改变软件可观察行为的前提下，调整其结构。

注意两次都出现的核心限定语——不改变可观察行为（observable behavior）。所谓"可观察行为"，指从外部看得见的一切：函数的返回值、抛出的异常、写进数据库的数据、打到屏幕上的输出……只要调用方察觉不到区别，内部你怎么翻天覆地地改都算重构。

重构 = 改变内部结构  ∧  保持外部行为不变

两个正交的维度：功能 vs 结构

理解重构最好的心智模型，是把编程活动拆成两个互相垂直的维度：一个是"程序做什么"（功能/行为），一个是"代码长什么样"（结构/设计）。重构只在结构轴上移动，功能轴纹丝不动。

功能轴（外部行为）→ 加需求 / 改逻辑 / 修 bug ▲ 加功能 ●──┼──────────────● 又加功能又改结构 │ （危险区：出错分不清谁的锅） │ ───────────┼──────────────▶ 结构轴（内部设计） │ 起点 ●───┼──────────────● 纯重构：行为不变，结构变好 │ Kent Beck「两顶帽子」：任何时刻只戴一顶 ┌──────────────┐ ┌──────────────┐ │ 加功能帽 │ │ 重构帽 │ │ 只沿功能轴走 │ │ 只沿结构轴走 │ └──────────────┘ └──────────────┘

这就是 Kent Beck 著名的「两顶帽子」（Two Hats）比喻：写代码时你要么在加功能，要么在重构，两者交替进行，但任何一个瞬间只能戴一顶帽子。加功能时不碰既有结构，重构时不加任何新功能。混着来，一旦测试挂了，你根本分不清是"新逻辑写错了"还是"结构改坏了"。

为什么"只戴一顶帽子"如此关键？这本质上是把一个大而模糊的变更拆成两类小而明确的变更，从而缩小排错的搜索空间。重构那顶帽子下的每一步都有测试兜底、行为不变；加功能那顶帽子下你才引入真正的行为变化。这和调试时"一次只改一个变量"是同一种科学方法。

最小例子：一次真实的提炼函数

光看定义没有直觉。来看一个最典型的重构手法——提炼函数（Extract Function）。下面是一段"能跑但读不懂"的打印客户欠款代码，输入输出完全不变，我们只改它的组织方式。

✗ BAD · 一坨到底，靠注释分段
void printOwing(Invoice inv) {
  double outstanding = 0;

  // 打印表头
  System.out.println("***** 账单 *****");

  // 计算欠款
  for (Order o : inv.orders)
    outstanding += o.amount;

  // 打印明细
  System.out.println(
    inv.customer + ": " + outstanding);
}
✓ GOOD · 提炼出意图清晰的小函数
void printOwing(Invoice inv) {
  printBanner();
  double outstanding = calcOutstanding(inv);
  printDetails(inv, outstanding);
}

void printBanner() {
  System.out.println("***** 账单 *****");
}
double calcOutstanding(Invoice inv) {
  double sum = 0;
  for (Order o : inv.orders) sum += o.amount;
  return sum;
}

关键观察：两段代码对同一个 Invoice 输入，会打印一模一样的内容。可观察行为零变化——所以这就是重构。变的只是：注释被"函数名"取代（// 计算欠款 变成了 calcOutstanding），阅读者不必再逐行读循环就能看懂主流程。

另一个更"轻"的重构手法是变量改名（Rename Variable）——它连结构都几乎没动，只把名字改得能表达意图。别小看它，Fowler 说这是他用得最频繁的重构：

Java改名：外部行为不变，可读性飙升
// 重构前：a、d 是什么？读者只能靠猜
double price(Order o) {
    double a = o.qty * o.unitPrice;          // a = 基础价
    double d = Math.max(0, o.qty - 500) * o.unitPrice * 0.05;  // d = 折扣
    return a - d;
}

// 重构后：名字自解释，计算逻辑一字未改 → 返回值完全相同
double price(Order o) {
    double basePrice = o.qty * o.unitPrice;
    double volumeDiscount = Math.max(0, o.qty - 500) * o.unitPrice * 0.05;
    return basePrice - volumeDiscount;
}
联系 JDK 与你每天用的框架：你以为这是玩具例子？ArrayList.add() 里那个 grow() 扩容方法、Spring AbstractApplicationContext.refresh() 里排成一列的 prepareBeanFactory() / invokeBeanFactoryPostProcessors() / finishRefresh()——都是"把一大段逻辑提炼成一串自解释的方法调用"的成果。refresh() 本身像一张目录，每个子方法是一个章节，这正是提炼函数在大型源码里的样子。

重构 ≠ 这些容易混淆的活动

初学者最容易把重构和几个近义词搅在一起。它们经常同时发生，但目标和"是否改变行为"这条红线完全不同：

活动改变外部行为？核心目标典型场景
重构否（红线）让代码更好懂、更好改提炼函数、改名、搬移字段
加功能是满足新需求加一个新接口、新分支
修 Bug是（把错的改对）修正错误行为改判空、修边界
性能优化否（结果同，只求更快）提升速度/省内存加缓存、换算法

有意思的是重构和性能优化：两者都"保持行为不变"，看起来像亲兄弟。但方向相反——性能优化常常牺牲可读性换速度（比如手动内联、位运算技巧），而重构牺牲一点点性能换可读性。Fowler 的观点是：先重构让代码清晰，清晰的代码更容易被精准地优化；瞎优化一团乱麻，往往优化错了地方。

新手高频陷阱 & 面试考点：
  • 把"重写"当"重构"：推倒重来、行为可能变、没有测试兜底的大改，那叫重写（rewrite），不叫重构。重构的精髓是小步 + 行为不变 + 随时可停。（下一节专门讲这个取舍）
  • 「顺手改一下」偷偷夹带功能：一边提炼函数一边"顺便"修了个 bug 或调了个逻辑——这就是同时戴两顶帽子。出了问题 code review 的人和你自己都定位不到。
  • 没测试就重构：重构的安全网是测试。没有测试，你根本无法证明"行为没变"，那是在裸奔改代码，不是重构。（"重构的前提：好的测试"一节详述）
  • 面试话术：被问"什么是重构"，务必点出"不改变外部可观察行为"这条红线，并能举出提炼函数、改名等具体手法，而不是笼统答"把代码改得更好"。

为什么要专门发明"重构"这件事？

你可能会问：代码能跑就行，干嘛没事找事去改结构？答案藏在软件成本的分布里。

写代码 ~20% 读代码 / 维护 / 改代码 ~80%

一段代码的生命里，被"读"的次数远多于被"写"的次数。Fowler 有句名言："任何傻瓜都能写出计算机能懂的代码；好的程序员写出人能懂的代码。" 重构就是持续偿还技术债（technical debt）的过程——债不还，利息（每次改动的痛苦）会越滚越大，直到某天"这块谁都不敢动"。

类比 GC 与内存整理：把代码库想成 JVM 堆。不断加功能就像不断分配对象，久了会产生"内存碎片"（散落的重复逻辑、纠缠的依赖）。重构就像 GC 的标记-整理（Mark-Compact）——不改变存活对象（外部行为），只是把它们挪到更紧凑、更连续的位置，让后续分配（加功能）更快更顺。二者都在"不改变可见结果的前提下优化内部布局"。

小结

一图流记住"什么是重构"：
  • 定义：在不改变外部可观察行为的前提下，调整内部结构，让代码更易读、更易改。这条红线是判断"是不是重构"的唯一标准。
  • 两顶帽子：加功能 vs 重构，任何时刻只戴一顶，绝不混着来——这样测试挂了才能秒级定位原因。
  • 手段：提炼函数、改名、搬移等一系列有名字、有套路的小步手法（第 6-12 章的目录）。
  • 价值：代码读 >> 写，重构是持续偿还技术债、对抗"腐化"的日常动作，如同堆内存的标记-整理。
  • 前提：必须有测试兜底，否则无法证明"行为没变"，那不是重构而是赌博。
