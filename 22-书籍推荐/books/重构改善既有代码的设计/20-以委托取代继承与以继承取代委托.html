以委托取代继承与以继承取代委托

继承是最诱人也最容易被滥用的复用手段——这对镜像手法教你在 is-a 与 has-a 之间来回权衡。

又是一对镜像手法

和上一章"隐藏委托 / 移除中间人"一样，本节也是一对互逆的手法：Replace Inheritance with Delegation（以委托取代继承）把 extends 拆成"持有一个字段"，往组合的方向走；Replace Delegation with Inheritance（以继承取代委托）反过来，把满屏的转发方法收敛成一句 extends。它们回答同一个设计问题：两个类之间的关系，到底该用"是一个（is-a）"还是"有一个（has-a）"来表达？

底层原理：白盒复用 vs 黑盒复用。继承是白盒复用——子类能看见、能覆写父类的内部实现，父类的一举一动（尤其是"自调用"）都会漏进子类，耦合极紧。委托/组合是黑盒复用——你只能透过被委托对象的公开接口使用它，看不见它内部怎么实现。GoF《设计模式》开篇的第一条原则就是"优先使用对象组合，而非类继承（Favor object composition over class inheritance）"，讲的正是这个取舍。

继承为什么会"变味"

继承本身不是错，错在为了复用代码而继承一个语义上并非父类的类。一旦继承下来，你就被迫吞下父类全部的公开方法——哪怕其中一大半根本不该出现在子类上。这就是第 3 章列过的坏味道：被拒绝的遗赠（Refused Bequest）。JDK 里就有个人尽皆知的反面教材：

StackVector.javaJDK 的历史失误
// java.util.Stack 竟然 extends Vector —— 一个被官方承认的设计失误
Stack<String> stack = new Stack<>();
stack.push("a");
stack.push("b");

// 灾难：Vector 的方法全被继承下来,可以从"栈底"任意插入,
// 彻底破坏 LIFO(后进先出)不变式
stack.add(0, "sneaky");        // 继承自 Vector,栈根本不该有这操作
stack.insertElementAt("x", 0); // 同样是"被拒绝的遗赠"
stack.remove(1);                // 从中间抽走一个元素,栈的语义荡然无存

因为 Stack 是一个 Vector（is-a），它就没法阻止别人把它当普通列表乱用。栈的不变式（只能从顶端进出）被继承来的方法轻易击穿。正确做法是让 Stack 持有一个 Vector（has-a），只对外转发 push/pop/peek——这正是"以委托取代继承"。

【继承 is-a · 白盒 · 编译期静态绑定】 Stack ──extends──▶ Vector │ 自动继承 Vector 全部方法(含 add(i)/insertElementAt/remove(i)...) └─ 外界能把 Stack 当 Vector 乱用,不变式无法守住 【委托 has-a · 黑盒 · 运行期组合】 Stack ──holds(private)──▶ Vector │ 只对外暴露 push/pop/peek,其余方法外界根本看不见 └─ 只依赖 Vector 的公开接口,栈的不变式牢牢锁死
官方"认错"的证据。JDK 文档在 Stack 上直接写着："A more complete and consistent set of LIFO stack operations is provided by the Deque interface... Deque<Integer> stack = new ArrayDeque<>();"。也就是说连 JDK 官方都推荐弃用 Stack、改用 ArrayDeque。ArrayDeque 只 implements Deque（面向接口），不去继承一个不该继承的实现类——这本身就是"优先组合/接口而非继承"的示范。

更隐蔽的坑：脆弱基类与"自调用"

Stack 的问题还只是"暴露了不该暴露的方法"。继承更阴险的一面是脆弱基类问题（Fragile Base Class）：父类内部方法之间的自调用会悄悄穿过你覆写的方法，让子类行为出错。这是 Josh Bloch《Effective Java》第 18 条"组合优先于继承"的经典案例——一个只想统计"共加入过多少元素"的 Set：

Bad · 用继承"增强" HashSet
class InstrumentedHashSet<E>
        extends HashSet<E> {
  private int addCount = 0;

  @Override public boolean add(E e) {
    addCount++;
    return super.add(e);
  }
  @Override public boolean
      addAll(Collection<? extends E> c) {
    addCount += c.size();
    // HashSet.addAll 内部又会调 add()!
    return super.addAll(c);
  }
}
// addAll(List.of("a","b","c"))
// 期望 addCount=3, 实际=6 (各计两次)
Good · 组合 + 转发类
// 可复用的转发外壳:持有(has-a)一个 Set
class ForwardingSet<E> implements Set<E> {
  private final Set<E> s;
  ForwardingSet(Set<E> s){ this.s = s; }
  public boolean add(E e){ return s.add(e); }
  public boolean
    addAll(Collection<? extends E> c){
      return s.addAll(c); }
  // ...其余方法一律转发给 s
}
class InstrumentedSet<E>
        extends ForwardingSet<E> {
  private int addCount = 0;
  InstrumentedSet(Set<E> s){ super(s); }
  @Override public boolean
    addAll(Collection<? extends E> c){
      addCount += c.size();
      return super.addAll(c); }
  // addCount 恰好 = 3
}

为什么正例对了？关键在自调用被"隔断"了。反例里 super.addAll(c) 落进 HashSet.addAll，而它内部又回调 this.add()——由于多态，this 是子类，于是每个元素又被 InstrumentedHashSet.add 加计一次。正例里 super.addAll 落进 ForwardingSet.addAll，它转发给内部真实的 s.addAll；那个 s 是原生 HashSet，它的 addAll 只会调它自己的 add，压根不知道外面套了统计层。组合把父类的实现细节挡在了黑盒之外。

陷阱一：继承会把父类的"实现细节"变成子类的"隐藏契约"。反例的 bug 根源是子类依赖了 HashSet.addAll "内部会自调用 add" 这一未公开、随版本可能变化的实现细节。哪天 JDK 改了 addAll 的实现，你的计数就又对了或又错了——你的正确性系于父类的私房事。只有当你能确定 X 真的 is-a Y、且愿意与 Y 的实现共命运时，才用继承；否则一律优先组合。

以委托取代继承 · 小步做法（Mechanics）

  1. 在子类里新建一个字段，指向父类类型（或其接口）的实例——通常先指向 this 自己，保证行为不变。
  2. 把子类中每个"用到父类功能"的方法，改成通过这个字段来委托调用，逐个改、逐个测。
  3. 去掉 extends，改由字段真正 new 出一个独立的委托对象。
  4. 若客户端此前依赖"子类是父类子类型"这一事实，就让子类改为 implements 父类抽取出的接口，只对外转发需要的方法。全绿收工。

反向：以继承取代委托

天平也会朝另一边倒。如果你发现自己写了一个类，它几乎把每个方法都原样转发给某个被委托对象——这就落回上一章的"中间人（Middle Man）"坏味道了。当满足严格条件时，可以反向操作，用继承一举消灭所有转发样板。

DelegationToInheritance.javaReplace Delegation with Inheritance
// 重构前:List 沦为纯转发的"中间人",一堆一行方法
class CatalogList {
    private ArrayList<Item> items = new ArrayList<>();
    public void    add(Item i)   { items.add(i); }
    public Item    get(int i)     { return items.get(i); }
    public int     size()         { return items.size(); }
    public boolean isEmpty()      { return items.isEmpty(); }
    // ...把 List 的方法几乎抄了一遍,自己没别的行为
}

// 重构后:直接继承,转发样板全部消失
class CatalogList extends ArrayList<Item> {
    // 什么都不用写,add/get/size/isEmpty 全都白拿
}
陷阱二：反向操作有苛刻的前提，别乱用。Fowler 明确列了几条硬性条件：（1）你用到了被委托对象的全部（或绝大部分）接口——只用一半就继承，会立刻掉进"被拒绝的遗赠"；（2）这个被委托对象不与其他对象共享——因为继承后"你"和"它"合体成一个身份，无法再被别人共享引用；（3）委托对象的类型在对象生命周期里不会变。上面的 CatalogList extends ArrayList 其实就踩了第 (1) 条的雷——继承 ArrayList 会连 remove(int)、clear() 也一并暴露；若目录不该被随意清空，就还是老实用组合。

里氏替换：判断该不该继承的硬尺子

"到底是不是真 is-a"不能凭语感，有一条形式化的判据——里氏替换原则（LSP, Liskov Substitution Principle）：凡是父类对象能出现的地方，换成子类对象，程序的行为都必须依然正确。

∀ s:Sub, ∀ 使用 Super 的程序 P  ⟹  P(s) 行为不变

拿它套 Stack extends Vector：任何"期待一个 Vector"的代码，传进一个 Stack 后仍应正常工作——这一条技术上成立，所以编译器不拦你。但反过来（把 Stack 当 Stack 用时能否守住栈语义）却守不住。真正的 is-a 要求的是行为可替换，而不仅仅是"能通过编译"。经典的 LSP 反例还有"正方形 is-a 长方形吗"：数学上是，但给正方形设 setWidth 会连带改 height，违反了"长方形宽高可独立设置"的行为契约。

决策口诀。问三个问题：①它真的 is-a 吗（LSP 是否成立）？②我要用父类的全部接口，还是只想蹭其中几个（会不会造成被拒绝的遗赠）？③我需要在运行期切换行为、或复用于多种实现吗（组合能给你运行期灵活性，继承是编译期焊死的）？三个问题里只要有一个偏向"否 / 只想蹭 / 需要灵活"，就选委托。

连接生态：框架早就替你做了选择

场景用了哪种为什么
Spring AOP 默认代理（CGLIB/JDK）委托为主JDK 动态代理靠"实现同一接口 + 持有目标对象转发"，正是委托；只有无接口时才退化为 CGLIB 子类继承
Collections.synchronizedList委托（装饰器）包一层转发给底层 List，转发时套 synchronized——组合才能给任意 List 实现加锁
Servlet 的 HttpServletRequestWrapper委托（装饰器）持有原始 request、逐方法转发，让你只覆写想改的几个——JDK/Servlet 给你现成的 ForwardingXxx
AbstractList / AbstractMap继承（模板方法）你的类确实 is-a List，只需填几个抽象方法，模板方法模式让继承在此处恰如其分
Lombok @Delegate委托自动生成转发方法，解决"委托样板太多"的痛点——等于把 Replace Inheritance with Delegation 的代价降到零
把这对手法和"装饰器/代理"串起来。上一章说"转发 + 增值 = 装饰器"。本章告诉你：装饰器之所以能存在，正是因为它选了委托而非继承——用组合持有被装饰对象，才能在运行期任意套壳、层层叠加。假如装饰器改用继承，你就得为每种"底层实现 × 每种增强"的组合都写一个子类，类会爆炸（这正是 GoF 讲装饰器时批判的"继承导致子类数量爆炸"）。组合优先，本质是为了对抗继承的组合爆炸与刚性。
一句话收束："以委托取代继承"与"以继承取代委托"是同一根 is-a / has-a 天平的两端。默认选委托（组合优先于继承）——它是黑盒复用、运行期灵活、只暴露该暴露的；只有当三条都满足时才反向用继承收敛样板：真 is-a（LSP 成立）、要用对方全部接口、且不与他人共享。判断继承是否变味的两把尺子是被拒绝的遗赠（继承了用不上的方法，如 Stack extends Vector）与脆弱基类（父类自调用穿透覆写方法，如 InstrumentedHashSet）。记住：重构让你随时能改主意，天平本就该随系统演化来回滑动。
