搬移函数、搬移字段、提炼类、内联类

这四招回答同一个问题：一段行为、一个字段、一整个类，到底该住在哪？——重新划分职责的边界。

一条主线：让数据和用它的行为"团聚"

前面几节（提炼函数、改变函数声明）都在一个类内部做手术；从第 8 章开始，Fowler 把镜头拉远到类与类之间——一个元素放错了类，就像把菜刀塞进卧室柜子：不是刀有问题，而是它放错了房间。这四招做的都是同一件事：把元素搬到它该在的地方。

核心洞察：面向对象最根本的组织原则，是把数据和"操作这些数据的行为"放在一起（高内聚）。当一个函数总在读别的类的数据、一个字段总被别的类使用、一个类塞了两三种毫不相干的职责时，边界就画歪了。搬移函数/字段是成员级的微调，提炼类/内联类是类级的重划——粒度不同，方向一致：向"高内聚、低耦合"收敛。

手法针对的坏味道做什么逆操作
搬移函数依恋情结 Feature Envy把方法挪到它最常访问数据的那个类搬移函数（本身可逆）
搬移字段字段用错地方 / 数据泥团把字段挪到最常读写它的那个类搬移字段
提炼类过大的类 / 职责过多拆出一个新类承担一部分职责内联类
内联类无所事事的类 / 冗余间接层把不再"挣自己身价"的类折回宿主提炼类

搬移函数：治"依恋情结"的正药

还记得依恋情结那节吗？——一个方法对别的类的数据兴趣，远大于对自己类的数据。搬移函数就是它的解药：把方法搬到它"眼馋"的那个类里，让行为和数据重新住到一起。判断信号很直接：数一数方法体里，访问自己字段多，还是访问别人字段多。

Bad · 折扣逻辑长在 Order 里，却全在读 Customer
class Order {
  private Customer customer;
  private double amount;

  double finalPrice() {
    double d = 0;
    // 全程盯着 customer 的数据
    if (customer.getLevel() == VIP)
      d = customer.getPoints() > 1000
          ? 0.2 : 0.1;
    return amount * (1 - d);
  }
}
Good · 折扣规则回到 Customer，行为与数据团聚
class Customer {
  private Level level;
  private int points;

  // 数据的主人自己算折扣
  double discountRate() {
    if (level != VIP) return 0;
    return points > 1000 ? 0.2 : 0.1;
  }
}
class Order {
  double finalPrice() {
    return amount *
      (1 - customer.discountRate());
  }
}
迪米特法则的味道：搬移后 Order 不再连环追问 customer.getLevel()、customer.getPoints()——它只说一句"给我你的折扣率"。这正是"告诉，别询问"（Tell, Don't Ask）：别把对象的内部数据掏出来在外面算，而是把算法送进对象里去。耦合从"依赖 Customer 的两个字段"降到"依赖 Customer 的一个方法"。
陷阱一：不是所有"访问别人"都要搬。如果方法同时重度使用两个类的数据，就要判断"它更想和谁在一起"，而不是机械搬家。陷阱二：小心搬出循环依赖。把 A 的方法搬进 B 后，若 B 反过来又要回调 A，就形成 A⇄B 双向依赖——这比原来更糟。搬移前先问：搬完之后箭头是不是只指向一个方向？

搬移字段：字段的归属由"谁最常用它"决定

字段放错家比函数更隐蔽，因为它不"报错"，只是让每次使用都要绕远路。经典信号：一个字段的值总是由另一个类决定，或总是和另一个类的字段成对出现。比如"利率"——同一类型的账户利率相同，那它本就属于"账户类型"，而不是每个账户实例。

MoveField.javainterestRate 从 Account 搬到 AccountType
// Before：利率由账户"类型"决定，却挂在每个 Account 实例上
class Account {
    private AccountType type;
    private double interestRate;      // ← 放错了家：同类型账户会各存一份
    double interestFor(double days) {
        return interestRate * days;
    }
}

// After：字段回到 AccountType，所有同类型账户共享同一个真相来源
class AccountType {
    private double interestRate;      // ← 回到该在的地方
    double getInterestRate() { return interestRate; }
}
class Account {
    private AccountType type;
    double interestFor(double days) {
        return type.getInterestRate() * days;   // 改一次利率，全类型账户同步生效
    }
}
意外的性能红利（内存局部性）：搬移字段把"总是被一起访问"的数据收拢进同一个对象后，它们在堆上更可能落进相邻内存、甚至同一条 CPU 缓存行（cache line，通常 64 字节）。遍历时一次加载就能命中多份数据，cache miss 更少。这是"高内聚"在硬件层面额外送你的性能——软件设计原则和 CPU 缓存机制在这里罕见地站在了同一边。
陷阱：字段搬家必须一次改全所有引用点。字段往往被四处读写，手工搬极易漏改某个角落，编译期还未必报错（比如通过反射、序列化访问）。务必用 IDE 的 Move / Change Signature 自动重构（IDEA 选中字段 F6），它会连引用一起改。另外注意：若字段参与了 equals/hashCode、JSON 序列化、JPA 映射，搬家会改变对外的数据结构和持久化格式，要同步评估兼容性。

提炼类：一个类扛了两份工，就拆开

类会"长胖"：需求一波波来，字段和方法不断往一个类里堆，直到它同时管着好几摊互不相干的事——这就是坏味道"过大的类（Large Class）"。判据是单一职责原则（SRP）：一个类应该只有一个引起它变化的原因。若你能说出"这些字段/方法是为 X 服务的，那些是为 Y 服务的"，X 和 Y 就该是两个类。

Bad · Person 里混进了"电话号码"这套子概念
class Person {
  private String name;
  // 下面三样其实是"一个电话号码"
  private String areaCode;
  private String number;
  String telephone() {
    return "(" + areaCode
         + ") " + number;
  }
}
Good · 提炼出 TelephoneNumber，各自内聚
class TelephoneNumber {
  private String areaCode, number;
  @Override
  public String toString() {
    return "(" + areaCode
         + ") " + number;
  }
}
class Person {
  private String name;
  private TelephoneNumber phone;
}
提炼类 (Extract Class) ⇄ 内联类 (Inline Class) 互为逆操作 ┌─────────────── Person（臃肿）───────────────┐ │ name │ │ areaCode number telephone() ← 电话概念 │ 一个类扛两种职责 └──────────────────────────────────────────────┘ │ 提炼类 ▼ ┌──── Person ────┐ ┌──── TelephoneNumber ────┐ │ name │ 持有 │ areaCode number │ │ phone ─────────┼──────▶ │ toString() │ └────────────────┘ └───────────────────────────┘ ▲ 内联类：若 TelephoneNumber 后来变得无足轻重，再折回 Person
你天天在用的"架构级提炼类"：Spring 三层。@Controller 管 HTTP 请求解析、@Service 管业务规则、@Repository 管数据访问——这正是把"一个大 Servlet 里三种职责"提炼成三个类。反过来，当你看到一个 XxxService 长到三千行、既拼 SQL 又发邮件又算报表，那就是"过大的类"在向你发出提炼类的信号。JDK 里也有教科书级案例：老 java.util.Date 把"时间点 + 时区 + 格式化"糅成一团，Java 8 的 java.time 把它提炼成 LocalDateTime / ZoneId / DateTimeFormatter 各司其职。

内联类：当一个类不再"挣自己的身价"

内联类是提炼类的逆操作，用在一个类退化到几乎没有独立价值时——它可能是过去某次提炼留下的产物，如今职责被搬空，只剩一两个字段做转发，反而成了阅读时的额外跳转。此时把它的字段和方法折回宿主类，减少一层无谓的间接。

高频面试对比：提炼类 ↔ 内联类不是"谁对谁错"，是火候。提炼过头会走向另一个极端——贫血类 / 中间人：一堆只有 getter/setter、没有行为的小类，读一条业务要在七八个类之间反复横跳，和"过大的类"一样痛苦。判据只有一句：这个类是否还挣得起自己的身价（pull its weight）？挣得起就留着，挣不起就内联掉。和上一章"提炼函数 ↔ 内联函数"完全同构——都是"先按当前理解划边界，边界不对了就反手调整"。
搬移的方向 = 数据 与 用它的行为 的团聚
过大的类(职责糊成一团) 恰当拆分 · 各司其职 拆过头(贫血类/中间人泛滥)

火候：拆到"每个类都有清晰的一句话职责"就停手；一旦拆出的类只剩转发、没有自己的行为，就该反手内联。

共同底线：这仍是"行为保持"的等价变换

和所有重构一样，这四招都必须不改变程序的外部可观察行为。你只是把同样的逻辑换了个"住址"，输入输出必须一字不差。正因为是等价变换，IDE 才能把它们做成自动化菜单——IDEA 里 F6（Move）、Ctrl+Alt+M 后再 Move、以及 Refactor 菜单里的 Extract Delegate（提炼类）都会替你自动改掉所有引用。

最容易翻车的地方：搬移会改变封装与可见性。把 private 方法搬到另一个类，它可能被迫变成 public，无意中扩大了 API 暴露面；搬字段则可能破坏原类维护的不变量（invariant，比如"这两个字段必须同增同减"）。所以铁律不变——搬移前先有测试托底。没有测试的跨类搬移，等于蒙眼重排承重墙。
一句话收束：这四招是用不同粒度回答同一个问题——"这段行为 / 这个字段 / 这个类，到底该住在哪"。判据始终是那句"让数据和操作它的行为团聚"（高内聚、低耦合、单一职责）：函数眼馋别人的数据就搬移函数，字段总被别人用就搬移字段，一个类扛了两份工就提炼类，一个类挣不起身价就内联类。它们都是行为保持的等价变换，因此能被 IDE 自动化、也因此必须有测试托底——从 Spring 分层到 java.time，你每天都在使用它们的成果。
