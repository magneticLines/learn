改变函数声明与引入参数对象

函数的声明就是它对外的"契约"——改好名字与参数，等于重新划定模块之间的边界。

为什么函数声明是"最重要的接口"

Fowler 说：函数是软件里最基本的组合单元，而函数声明（名字 + 参数列表）就是它对外的契约。名字告诉读者"这是干什么的"，参数列表决定"它需要从外界拿什么、和谁耦合"。这两样但凡有一样起错了，整段代码的可读性和可维护性都会打折。

函数声明 = 名字〈意图〉 + 参数列表〈依赖边界〉
一个视角：名字是给人看的抽象——好名字让人不必读实现就懂；参数列表是给机器与模块关系看的边界——它决定这个函数"知道"多少外部世界。改变函数声明这一手法，本质就是在调整这两条维度：改名字（对人更友好）、加/减/换参数（对耦合更精准）。它的英文别名很多——Rename Function、Add/Remove Parameter、Change Signature——但都归到同一招下。

看一个"参数选得不好导致过度耦合"的经典例子：一个判断客户是否在新英格兰地区的函数，它其实只需要一个州代码，却把整个 Customer 拖了进来。

before · 耦合过重
boolean inNewEngland(Customer c) {
  return Set.of("MA","CT","ME",
      "VT","NH","RI")
    .contains(c.address().state());
}
// 只为读一个 state,
// 却依赖了 Customer 的
// 整个内部结构
after · 只依赖真正需要的
boolean inNewEngland(String state) {
  return Set.of("MA","CT","ME",
      "VT","NH","RI")
    .contains(state);
}
// 现在既好测(传字符串即可),
// 又可复用于任何有 state
// 的场景

调用点相应改为 inNewEngland(c.address().state())。把"从 Customer 里掏 state"的职责从被调用方挪回调用方，函数的依赖面立刻收窄——这正是"改变参数"背后的设计动机：让函数只知道它非知道不可的东西。

两种实施机制：简单机制 vs 迁移机制

改声明这件事看着简单，难点全在"有多少人在调用它"。Fowler 给了两套机制，选哪套只取决于一个问题：你能不能一次性把所有调用点都改掉？

维度简单机制 Simple迁移机制 Migration
适用调用点你全都能改（私有方法、团队内代码，IDE 可全局改名）调用点你控制不了（已发布 API、跨团队 / 跨仓库、开源库）
步骤直接改声明，一次性改完所有调用提炼新函数 → 旧函数转发 → 逐步迁移 → 删旧函数
风险改动一步到位，全靠测试兜底每一步都能独立提交 / 回滚，风险摊到最小
生活类比一次性换掉整段水管先接旁路，逐段切换，最后再拆旧管

迁移机制的精髓是"让旧函数临时转发到新函数"——这样每一步的外部行为都不变，测试始终全绿，你可以从容地一个调用点一个调用点地迁移，甚至跨多次提交：

ChangeDeclaration.java · 迁移机制四步Java
// ① 用"新签名"提炼出新函数（若只是改名，先起个临时名）
boolean inNewEngland(String state) {
    return Set.of("MA","CT","ME","VT","NH","RI").contains(state);
}

// ② 旧函数改为"转发"到新函数 —— 此刻行为完全不变，测试全绿
boolean inNewEngland(Customer c) {
    return inNewEngland(c.address().state());   // 委托，不重复逻辑
}

// ③ 把调用点一处一处从旧签名迁到新签名（可分多次提交、边迁边发布）
// ④ 旧函数归零调用后，安全删除它
Java 特有的坑：上例里新旧两版能同时存在，是因为参数类型不同（String vs Customer），构成了方法重载。但如果你只是改名字而参数不变，Java 不允许两个签名完全相同的方法共存——迁移期必须给新函数起个临时名（如 inNewEngland2），迁完再一次性改回。别指望"同名不同实现"。
旧签名 (仍被 N 处调用) │ ① 用新签名提炼出函数体 ▼ 新函数 (新名 / 新参数) ▲ ② 旧函数改为「转发」→ 行为不变、测试全绿 │ ③ 调用点逐个迁移 (可跨多次提交，随时可停) │ ④ 旧函数归零调用 → 删除 ▽ 只剩新签名 ← 全程外部行为零变化

陷阱：标志参数（Flag Argument）

"改变函数声明"最常见的一个改进目标，就是干掉布尔标志参数。当一个 boolean 参数被用来在函数内部"选择走哪条分支"时，调用点会变得完全读不懂——book(customer, true) 里的 true 是啥？必须点进函数体才知道。这是 Effective Java Item 51「谨慎设计方法签名」明确点名的反例。

bad · 标志参数
book(customer, true);
// true 是什么?调用点读不出

void book(Customer c,
          boolean isPremium) {
  if (isPremium) { /*...*/ }
  else           { /*...*/ }
}
good · 拆成两个明确的函数
premiumBook(customer);
regularBook(customer);
// 意图一目了然、可各自测试

void premiumBook(Customer c){/*..*/}
void regularBook(Customer c){/*..*/}
还有一类"沉默的 bug"：相邻的同类型参数容易被调错顺序。createUser("shea", "s3cr3t")——哪个是用户名、哪个是密码？写反了照样编译通过、运行出错。这正是下一节「引入参数对象」要治的病：用有名字的字段代替靠位置区分的参数。

引入参数对象：消灭"数据泥团"

当同一组数据总是成群结队地出现——一个函数的参数列表里有它们，另一个函数又有它们，字段声明里还有它们——这就是代码坏味道里的数据泥团（Data Clump）。最典型的就是 min / max、start / end、x / y 这种"其实是一个整体、却被拆成好几个参数到处传"的组合。

「引入参数对象」就是把这群总在一起的数据，打包成一个对象来传递。它的价值远不止"参数变少"：

  • 让隐含关系显式化：min 和 max 本是一个"范围"，包成 NumberRange 后，这层关系写进了类型里。
  • 统一命名、减少调错顺序：所有用到范围的地方都用同一套字段名。
  • 最关键——它是"深层重构的催化剂"：一旦有了这个类，你会发现原本散落各处的行为（如"判断某值是否落在范围内"）可以被"吸附"到它身上，一个真正的领域对象就此诞生。
IntroduceParameterObject.javaJava 16+ record
// ── before：min / max 数据泥团，反复成对出现 ──
List<Reading> readingsOutsideRange(Station s, int min, int max) {
    return s.readings().stream()
        .filter(r -> r.temp() < min || r.temp() > max)
        .toList();
}
// 别处又出现同样的 min/max……散落的比较逻辑到处重复

// ── after：引入参数对象，并把行为吸附过去 ──
record NumberRange(int min, int max) {   // record 天生就是参数对象
    boolean contains(int v) {                 // ★ 逻辑收敛到数据旁边
        return v >= min && v <= max;
    }
}

List<Reading> readingsOutsideRange(Station s, NumberRange range) {
    return s.readings().stream()
        .filter(r -> !range.contains(r.temp()))   // 语义清晰、逻辑唯一
        .toList();
}
为什么用 record？Java 16 起的 record 自动生成构造器、getter、equals/hashCode/toString，且字段不可变——正是参数对象最理想的载体：无样板、值语义、天然线程安全。这也解决了上一节"同类型参数调错顺序"的问题：new NumberRange(min, max) 里字段有名字，比裸传两个 int 安全得多。

横向对比：参数对象不是唯一解

"参数太多"有好几种治法，别一上来就无脑套参数对象。关键看这些参数是不是一个语义整体：

方案治的是什么适用局限 / 注意
引入参数对象反复同行的数据泥团一组参数语义上是一个整体（range、money、坐标）要新建类型；只当数据袋、不搬行为则收益有限
Builder 建造者可选参数多、伸缩构造器难读参数多且多为可选，想链式 + 不可变样板多（Lombok @Builder / record 可缓解）
方法重载参数个数 / 类型的变体少量、语义相同的调用形态编译期静态分派，自动装箱易踩坑
可变参数 varargs同类型、个数不定数量不定的同质参数只能放最后一个；装箱 / 建数组有开销
高频面试点——重载的静态分派陷阱：List<Integer> list 上调 list.remove(1)，走的是 remove(int index)（按下标删），而不是 remove(Object)（按值删）！因为重载在编译期就按静态类型选定了方法，1 是 int，优先匹配不需装箱的版本。要删值得写 list.remove(Integer.valueOf(1))。这提醒我们：给一个方法新增重载本身就是一次"改变函数声明"，可能悄悄改变已有调用点的分派结果——务必谨慎。

和 Java 生态的联系

这两招在 JDK、Spring 里到处是现成的样板：

  • JDK 的值对象即参数对象：java.time.Period（年月日）、Duration、LocalDate 本质上都是把过去用一堆 int 传的时间"泥团"包成了带行为的类型；java.awt.Point / Rectangle 同理。Guava 的 Range<T> 正是我们上面 NumberRange 的工业级版本。
  • Spring 的 Pageable / PageRequest：把 page + size + sort 这组永远一起出现的分页参数打包成一个对象传给 Repository——教科书级的引入参数对象。RequestEntity / ResponseEntity 把状态码 + 头 + 体打包，也是同一思路。
  • @ConfigurationProperties：把 application.yml 里一整组配置项绑定到一个 POJO，等于在"配置层"做引入参数对象——总比一个个 @Value 注入几十个字段清爽。
  • 迁移机制 + @Deprecated：JDK 里 new Date(year, month, day) 被标记废弃、引导你改用 LocalDate；Spring Security 的 WebSecurityConfigurerAdapter 废弃后引导改用组件式配置——都是"旧签名先保留转发、给足迁移期、再删除"的活样本。对外发布的 API 绝不能用简单机制硬改签名，那是破坏性变更。
串联记忆：上一节的「提炼函数」是把一段逻辑命名成函数；本节的「改变函数声明」则是在维护这个名字与它的参数边界；而「引入参数对象」是当参数本身开始"结块"时，把它们提炼成一个新名字（类型）。三者一脉相承，都是"用好名字管理复杂度"，只是作用在不同粒度上。
一句话收束：函数声明是模块对外的契约——名字管意图、参数管耦合。改声明时，能一次改完就用简单机制，改不完（尤其是公开 API）就用"旧函数转发"的迁移机制小步前进；当一组参数总是结伴出现（数据泥团），就用「引入参数对象」把它们打包，并趁机把散落的行为吸附到新类型上——参数对象的真正威力，不在于少传几个参数，而在于它常常是一个领域对象破土而出的起点。
