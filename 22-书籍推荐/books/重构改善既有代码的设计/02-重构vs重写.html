重构 vs 重写：什么时候该重构，什么时候该重写

"这代码烂透了，不如重写！"——这句话背后，藏着软件工程史上最贵的几个坑。

一、先把两个词的定义掰开

几乎每个程序员接手一坨"祖传代码"时都动过一个念头：与其在这堆屎山上打补丁，不如推倒重来一把梭。但在做这个决定之前，你得先分清两条路的本质区别——它们不是"程度不同"，而是安全模型完全相反的两件事。

维度重构 Refactoring重写 Rewrite
外部行为严格不变（这是定义的一部分）目标"等价"，但无从保证
步子大小一连串小步，每步都可编译、可测试一个大跳跃，中间态往往跑不起来
风险曲线随时可停、可回退，风险被摊平风险堆在最后"切换"那一刻，集中爆发
已有知识原地保留（代码里的历史修复都还在）大量隐性知识随旧代码一起被丢弃
可交付性任何时刻停下都是"更好一点的版本"不写完就是零，中途停 = 全亏
一句话辨析：重构是改善内部结构、行为不变（详见上一篇《什么是重构》）；重写是丢掉旧实现、从头再造一个功能等价物。前者像给行驶中的汽车换轮胎（还得保证一直在跑），后者像另造一辆新车再把乘客搬过去——听起来后者更彻底，但你有没有想过：旧车上那些叮叮当当的修补，恰恰是它十年里撞过的每一堵墙的记录。

二、为什么"重写"看着爽、实际是个大坑

软件史上最著名的反面教材，是 Netscape。2000 年前后它决定把浏览器代码推倒重写，结果耗时近三年、几乎没有新版本可发，市场份额被 IE 一路碾平。Joel Spolsky 为此写下名篇《Things You Should Never Do》，结论只有一句：把工作代码扔掉从头写，是一家软件公司能犯的最严重的战略错误。

为什么？因为你以为在扔"烂代码"，其实在扔知识。看这段谁都写过的丑陋逻辑：

每一个"莫名其妙的 if"，都是一次线上事故的墓碑Java
public BigDecimal calcTax(Order o) {
    BigDecimal tax = o.getAmount().multiply(RATE);

    // FIX #1832：某些历史订单 region 为 null，NPE 炸过一次生产
    if (o.getRegion() == null) return BigDecimal.ZERO;

    // FIX #2109：2019 前的合同免税，财务合规要求，不能删
    if (o.getSignDate().isBefore(TAX_REFORM_DATE)) return BigDecimal.ZERO;

    // FIX #2530：分转元的精度坑，必须 HALF_EVEN，否则对账差 1 分
    return tax.setScale(2, RoundingMode.HALF_EVEN);
}
陷阱：把"读不懂"误当成"没必要"。 一段代码"丑"，大概率不是因为原作者蠢，而是因为它吸收了三年里所有的边界情况。你重写时看不懂那几个 if，于是理直气壮地删掉——然后你会用未来一整年，把这三个 bug 一模一样地再踩一遍。这正是软件里的"契斯特顿栅栏（Chesterton's Fence）"：在你搞清楚一道篱笆为什么立在这儿之前，别急着拆它。

重写的时间到底花在了哪

人们对重写的期待是"腾出手来做新功能"，但真实的时间分布往往是这样——绝大部分预算被用来把旧系统已有的能力再实现一遍，真正的新价值少得可怜：

重新实现已有功能 60% 补回历史踩坑/边界 28% 新价值 12%

更糟的是 Fred Brooks 在《人月神话》里点破的 "第二系统效应（Second-system effect）"：人在重写时会把上次没来得及做的花哨设计一股脑塞进去，导致第二个系统过度设计、比第一个还臃肿。旧系统的缺点你记得清清楚楚，旧系统的优点你却浑然不觉——直到重写版把它们弄丢。

三、那到底什么时候该重写？

重写不是永远错误，它是一张高风险、偶尔必要的牌。判断的关键不是"代码丑不丑"（丑可以重构），而是下面这几条结构性绝症是否成立：

信号倾向重构倾向重写
技术栈存活度语言/框架仍在维护，能招到人建立在已死的技术上（停维护的框架、买不到的中间件、无人会的语言）
架构能否演进能在现有骨架上小步改良核心假设根本错了（如把多租户塞进单租户内核，怎么改都别扭）
测试安全网有测试，或能补出特征测试行为已由外部契约/规范定义，且旧代码已无人能懂
变更成本趋势加功能还算线性每个小改动都引发霰弹式连锁 bug，成本指数上升
规模大到重写风险不可控小到几周能验证等价（小工具、边缘服务）
该重写? = 技术栈已死 ∧ 架构假设错误 ∧ 规模可控 ∧ 行为可验证
决策口诀：先问"能不能重构到目标"。 90% 你想重写的场景，其实是"局部结构烂 + 没有测试保护"，那是重构要解决的问题，不是重写。真正该重写的，是那种"连方向盘都装在错误位置、且这辆车用的燃料已经停产"的系统——此时在旧骨架上重构，等于在流沙上盖楼。

四、同一段坏代码：重构路线 vs 重写路线

假设你接手一个 800 行的 OrderService.process()，逻辑缠成一团。两种做法的操作方式和风险天差地别：

Bad · Big-Bang 重写
// 停掉旧代码，另起一个新类
class OrderServiceV2 {
  void process(Order o) {
    // 凭"我理解的业务"从零写
    // ...三个月后...
  }
}
// 痛点：
// · 三个月不能上线，业务停摆
// · 旧版那些 if 我没抄 → 老 bug
//   全部复活
// · "切换日"当天一次性替换，
//   出事无法回退，只能连夜回滚
// · 期间旧系统还在改 → 追不上
Good · 小步重构
// 原地动手，每步都能编译+测试
void process(Order o) {
  validate(o);      // 提炼函数
  var tax = calcTax(o);   // 保留旧 if
  var bill = buildBill(o, tax);
  persist(bill);    // 一次搬一小块
}
// 收益：
// · 任何一步停下都是可上线版本
// · 历史修复原样保留，不复活老 bug
// · 出问题只回退最近一小步
// · 边重构边正常发新需求
高频面试陷阱：被问"如何治理这个上千行的祖传方法"时，答"重写"会被扣分。面试官想听的是——先补特征测试（Characterization Test）锁住当前行为 → 再用"提炼函数/搬移"小步拆解 → 每步跑测试。这套"有网 + 小步"的节奏，正是本书第 2 章反复强调的重构前提（下一篇细讲）。

五、第三条路：绞杀者模式（Strangler Fig）

"要么原地重构、要么推倒重写"其实是个假两难。Martin Fowler 提出的 绞杀者无花果（Strangler Fig） 给了第三条路：用重构的节奏（小步、可回退）去达成重写的目标（换掉整套实现）。名字来自热带的绞杀榕——它沿着老树生长，慢慢包裹，等长成后老树枯死、新树已然成形，全程森林没有一刻是空的。

工程上的做法是：立一个外观（Facade）挡在调用方前面，接口保持不变；内部把请求一个方法、一个方法地从旧实现切到新实现，用开关灰度、随时可切回：

绞杀者外观：新旧共存，逐个迁移，灰度可回退Java
public class PaymentFacade {
    private final LegacyPayment legacy;   // 老实现，暂时还在跑
    private final ModernPayment modern;   // 新实现，一次迁一个方法
    private final FeatureFlag flag;

    public Receipt charge(Order o) {
        // charge 已迁移：按流量灰度切到新实现，出问题一键关开关切回
        if (flag.isOn("payment.charge.v2")) {
            return modern.charge(o);
        }
        return legacy.charge(o);
    }

    // refund 尚未迁移：继续走旧实现。新旧共存，绝不 big-bang
    public void refund(String id) { legacy.refund(id); }
}

迁移过程用一张图看清——老树被逐步"绞杀"，但系统从头到尾都在正常服务：

Big-Bang 重写（危险） Strangler Fig（安全） ───────────────────── ───────────────────────────── 调用方 ──► 旧系统 (跑着) 调用方 ──► Facade（接口不变） ┊ ├─ charge ──► 新实现 ✅ 已迁移 [ 三个月黑箱开发 ] ├─ refund ──► 旧实现 ⏳ 待迁移 ┊ └─ query ──► 旧实现 ⏳ 待迁移 调用方 ──► 新系统 (切换日一把梭) 每迁完一个方法就上线一次，旧实现最终枯死 ▲ 出事无法回退，全盘皆输 ▲ 任何一步可回退，风险摊平到每一天
与你已学知识串联：Strangler 的外观层就是代理/外观模式；灰度开关是策略模式的运行时切换；"接口不变、实现替换"正是面向接口编程 + 里氏替换的直接应用。放大到架构层，Spring 里你也见过同款思路——@Primary / @ConditionalOnProperty 让新老 Bean 共存、按配置切换；数据库迁移里的"双写 + 影子读"校验新旧一致，本质都是同一招：用可回退的小步，完成不可回退的大事。

六、别被"沉没成本"和"手痒"带偏

陷阱一：沉没成本谬误（两个方向）。一个方向是"都写这么久了不能停"，硬着头皮把烂重写做完；另一个方向是"旧代码写了五年不能扔"，抱着真该淘汰的死技术栈不放。过去投入多少与现在该怎么选无关——只看"从今天起，哪条路的总成本更低"。
陷阱二：把"我不熟"当成"它该重写"。新人接手老代码最常见的冲动就是重写，因为读别人的代码比写自己的代码难得多（Joel 定律）。但难读 ≠ 该扔。先花两周读懂、补测试、小步重构，往往就够了。
一句话收束：重构是默认选项，重写是最后手段。先问"能不能重构到目标"——90% 的答案是能；只有当技术栈已死、核心假设错误、且规模可控可验证时，重写才成立。而即便决定重写，也优先用绞杀者模式把它拆成一连串"可回退的小步"，让系统始终在线。记住那条铁律：能工作的代码里，藏着你看不见的知识；扔掉它之前，先确认你真的读懂了它为什么这样写。
