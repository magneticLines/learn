拆分循环、合并条件表达式、以卫语句取代嵌套条件

三个"整理逻辑"的小手法：让一个循环只干一件事、让一堆 if 说清楚"到底在判断什么"、让嵌套的箭头拉平成一条主线。

这一节把三个看似零碎、其实同源的重构放在一起讲。它们都不改变程序行为，只做一件事：把"混在一起的意图"拆开、把"其实是一回事的判断"合并、把"层层缩进的条件"拉平。三者共同指向一个理念——代码首先是写给人读的，其次才是给机器跑的。

一、拆分循环（Split Loop）

坏味道：一个循环里同时干了好几件不相干的事——一边求总薪水，一边找最年轻员工。为了"少遍历一次"，很多人把它们塞进同一个 for。问题是：当你想修改其中一件事时，必须先看懂另一件；想复用其中一段逻辑时，又拆不出来。

Bad · 一个循环干两件事
int youngest = people.get(0).age;
int totalSalary = 0;
for (Person p : people) {
  if (p.age < youngest)
    youngest = p.age;   // 关注点 A
  totalSalary += p.salary; // 关注点 B
}
return "最年轻 " + youngest
     + "，总薪 " + totalSalary;
Good · 一个循环只干一件事
int totalSalary = 0;
for (Person p : people)
  totalSalary += p.salary;

int youngest = people.get(0).age;
for (Person p : people)
  if (p.age < youngest)
    youngest = p.age;

return "最年轻 " + youngest
     + "，总薪 " + totalSalary;

拆开后，每个循环都是一个可以独立命名、独立提炼的"意图单元"。下一步几乎总是配合提炼函数——把每个循环抽成 totalSalary(people) 和 youngestAge(people)，调用处就变成两行自解释的表达式。

初学者最大的心理障碍：性能。"拆成两个循环不就遍历两遍了吗？慢一倍！"—— 这是典型的过早优化。第一，绝大多数循环规模下，遍历两遍的开销小到测不出来；第二，先让代码清晰，再用 profiler 定位真正的热点。真到了性能瓶颈，把两个循环合回去只是几分钟的事，但一份清晰的代码能省下你几个月的维护成本。Martin Fowler 的原话："如果拆分让性能出了问题，合并回去很容易。"
联系 Java Stream：拆分循环的思想在函数式 API 里被推到了极致。people.stream().mapToInt(p->p.salary).sum() 与 people.stream().mapToInt(p->p.age).min() 本就是两条独立的流水线，各自表达一个意图。Stream 用惰性求值 + 短路把"多次声明"的可读性和"单次遍历"的性能在一定程度上兼顾了——但它的本质仍是"一条管道只做一种聚合"。

二、合并条件表达式（Consolidate Conditional Expression）

坏味道：一连串判断条件不同、但返回/执行的动作完全相同的 if。它们其实在表达同一个概念，却被写成了散装的三四个判断。

Bad · 散装的相同动作
double disabilityAmount(Emp e) {
  if (e.seniority < 2)
    return 0;
  if (e.monthsDisabled > 12)
    return 0;
  if (e.isPartTime)
    return 0;
  // 真正的计算...
}
Good · 合并 + 提炼命名
double disabilityAmount(Emp e) {
  if (isNotEligible(e))
    return 0;
  // 真正的计算...
}
boolean isNotEligible(Emp e) {
  return e.seniority < 2
      || e.monthsDisabled > 12
      || e.isPartTime;
}

合并的价值有两层：一是用一个 ||（或 &&）把"其实是一回事"的判断显式地连起来，读者一眼看出"这几个条件是同一组资格校验"；二是合并后往往能提炼出一个有意义的布尔函数名（isNotEligible），把"怎么判断"藏进函数、把"判断什么"留在主流程。命名，才是这次重构真正的产出。

条件之间的关系合并用短路语义（重要）
任一满足即同一结果（并列的"或"）||左边为 true 立即返回，右侧不再求值
层层嵌套、须全部满足才继续&&左边为 false 立即短路，右侧不再求值
合并前必须过的两道检查，否则改变行为：
① 副作用——如果某个条件里藏着 x++、方法调用改了状态，合并后短路可能让它不再执行。先把副作用挪出去再合并。
② 动作必须真的相同——只有"不同条件 → 完全一致的结果"才能合并。若各分支返回不同值，这是另一件事，别硬凑。
③ 注意 ||（短路）与 |（按位/非短路）的区别：后者两侧都会求值，用错会踩到本该被短路掉的空指针或异常。

三、以卫语句取代嵌套条件（Replace Nested Conditional with Guard Clauses）

坏味道："箭头形代码"（arrow code）——一层层 if 向右缩进，真正的核心逻辑被挤到最里面，读者得先在脑子里压一个"如果…那么…"的栈，读到最深处再一层层弹出来。

嵌套条件（箭头形） 卫语句（拉平主线） ────────────────── ────────────────── if (a) { if (!a) return X; ◄ 异常/边界先走 if (b) { if (!b) return Y; ◄ 一条条挡在门口 core(); ◄ 主线被埋 core(); ◄ 主线在最外层 } else return Y; return Z; } else return X; ▲ 缩进越深，认知负担越重 ▲ 缩进恒定，从上往下读一遍就懂

核心思想的转变：嵌套 if 把"正常路径"和"异常路径"当成对等的两支（if…else…）；而卫语句认为它们不对等——异常/边界情况应当尽早判断、尽早返回，把它们"挡在门口"，让函数主体只关心那条唯一的、最重要的正常路径。

PayCalculator.javaBad → 嵌套条件
double getPayAmount(Emp e) {
  double result;
  if (e.isSeparated) {          // 离职
    result = separatedAmount();
  } else {
    if (e.isRetired) {          // 退休
      result = retiredAmount();
    } else {
      result = normalPayAmount(); // 真正的主线，被埋在两层里
    }
  }
  return result;
}
PayCalculator.javaGood → 卫语句
double getPayAmount(Emp e) {
  if (e.isSeparated) return separatedAmount(); // 卫语句 1
  if (e.isRetired)   return retiredAmount();   // 卫语句 2
  return normalPayAmount();                    // 主线，一目了然
}

临时变量 result 也顺手消失了——这是卫语句常见的附带收益。整个函数从"需要在脑里维护一个 if 栈 + 一个可变变量"，变成"从上往下扫一遍，遇到特殊情况就地折返"。

什么时候用卫语句、什么时候用 if-else？判断标准是两个分支是否"地位对等"：
· 若两条路径同等重要、都是正常业务（比如"会员价 vs 非会员价"），用 if-else 让对称性显式化。
· 若一条是"正常主线"、另一条是"边界 / 异常 / 提前退出"（空值、无权限、已停用），用卫语句把噪音挡在门口。
卫语句本质是在向读者喊话："这不是正常情况，先处理掉，别让它干扰主线。"
联系你天天在写的 Spring / JDK 源码：卫语句是工业级代码里最常见的结构。Objects.requireNonNull(x)、Spring 的 Assert.notNull(arg, "...")、参数校验 if (bean == null) throw ...、集合的 if (isEmpty()) return Collections.emptyList(); —— 全是"先把不合法/边界情况挡掉，再执行主逻辑"。你读框架源码时之所以觉得"方法开头总有一串 if return"，正是卫语句风格的体现。

三者的共同底色：意图 vs 实现的分离

把三个手法放到一起看，会发现它们是同一枚硬币的三面——都在把"代码在做什么（意图）"从"代码怎么做（实现细节）"里剥离出来。

手法针对的坏味道做的动作常搭配
拆分循环一个循环多个关注点按意图拆开提炼函数
合并条件表达式散装的相同动作判断按意图合并提炼函数（命名布尔）
以卫语句取代嵌套条件箭头形代码按重要性拉平内联/消除临时变量
可读性 = 一个单元只表达一个意图
共同的高频面试/评审陷阱——反射式的"single return"教条。很多人被"一个函数只能有一个 return"束缚，于是宁可写三层嵌套也不肯早返回，把卫语句彻底废掉。这条老规矩来自 C 时代手动管理资源（怕多个 return 漏掉 free）；在有 try-with-resources、GC、finally 的 Java 里，多个早返回反而更清晰、更安全。别为了教条牺牲可读性——现代主流风格（含 Google Java Style）都拥抱卫语句式的早返回。
一句话收束：这三个重构不追求"更聪明的代码"，而追求"更诚实的代码"——让循环只说一件事、让条件说清它在判断什么、让主线浮到最外层。它们几乎不涉及类结构调整，成本极低、收益立竿见影，是你在任何一次日常改动中都能顺手施展的"清理动作"。记住优先级：先清晰，再谈性能；先意图，再谈技巧。
