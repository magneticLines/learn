提炼函数、内联函数、提炼变量

整本《重构》里最常用的三招——用名字给代码"贴标签"，让意图浮出水面，也知道什么时候该把标签撕掉。

三招是一体两面：给代码"命名"的艺术

Fowler 在第 6 章把这几个手法排在最前面，不是偶然——它们是使用频率最高、几乎每次重构都会用到的基本功。它们看似朴素，本质却只做一件事：用一个好名字，替换掉一段需要费脑子读懂的代码。

核心洞察：提炼函数、提炼变量的产物，都是一个名字。名字是最廉价、最强大的抽象工具——它让读者不必读实现就知道"这是什么"。三招的差别只在"贴标签的粒度"：提炼变量给一个表达式贴标签，提炼函数给一段语句贴标签；而内联是反向操作——当名字反而成了噪音时，把标签撕掉、露出本体。

提炼函数（Extract Function）

这是整本书出现频率最高的手法。信号很简单：当你看着一段代码，需要花一点脑力去"理解它在干什么"，就把它提出去，用一个描述"干什么"的名字命名。Fowler 的原话是——只要需要写注释来解释一段代码，就把它提炼成函数，用函数名代替那句注释。

Bad · 意图埋在实现里
void printOwing(Invoice inv) {
  printBanner();
  // 计算欠款
  double outstanding = 0;
  for (Order o : inv.orders)
    outstanding += o.amount;
  // 打印详情
  System.out.println(
    "欠款: " + outstanding);
}
Good · 主函数只读意图
void printOwing(Invoice inv) {
  printBanner();
  double outstanding =
    calcOutstanding(inv);
  printDetails(inv, outstanding);
}
// 函数名 = 原来那句注释
double calcOutstanding(Invoice i){…}
void printDetails(
  Invoice i,double amt){…}
命名口诀：按"做什么（意图）"命名，不按"怎么做（实现）"命名。叫 calcOutstanding 而不是 loopAndSum。哪怕新函数只有一行、名字比函数体还长也值得——因为代码被读的次数远多于被写的次数。这正和上一节"过长函数"里的短函数原则一脉相承。

提炼时的三种变量处理，是它唯一的难点

提炼函数本身不难，难的是如何处理跨越"提炼边界"的局部变量。这是面试与实战都容易翻车的地方：

变量角色含义处理方式
只读不改提炼段只读取它作为参数传入
被赋值 & 之后还用提炼段改了它，外面还依赖作为返回值带回来（一个变量对应一个返回值最干净）
被赋值但外面不再用提炼段的临时变量直接下沉进新函数，成为其局部变量
陷阱：一次提炼想带回多个被修改的变量。如果一段代码修改了 2 个以上、且外部都还要用的变量，Java 没有多返回值，你会被迫返回数组/对象、或退回用可变参数对象——这是代码在提示你：这里逻辑纠缠太深，本就不该硬提。正确做法是先用拆分变量、移动语句把纠缠解开，再提炼。别用 int[] 硬凑返回值。

提炼变量（Extract Variable）

当一个表达式复杂到"读一遍看不懂它算的是啥"，就把它拆出来赋给一个命名良好的局部变量。它把表达式的"意义"显式写进代码，而不是让读者在脑子里做一次逆向工程。

Bad · 一坨表达式
return order.quantity * order.price -
  Math.max(0, order.quantity - 500)
    * order.price * 0.05
  + Math.min(order.quantity
    * order.price * 0.1, 100);
// 三个月后的你：这算的啥？
Good · 每步都有名字
double base =
  order.quantity * order.price;
double volDiscount = Math.max(0,
  order.quantity - 500)
  * order.price * 0.05;
double shipping = Math.min(
  base * 0.1, 100);
return base - volDiscount + shipping;
额外红利：断点调试。把长表达式拆成几个具名变量后，Debug 时每一步中间值都能在 Watch 窗口直接看到，不用再在表达式中间打断点或写临时 System.out.println。这也是它在实战中被高频使用的隐藏理由。
提炼变量 vs 提炼函数怎么选？看作用域：如果这个名字只在当前函数里有意义，用提炼变量；如果它表达的概念在这个类的其它地方也用得上，就直接提炼成函数（在 JS 里是取 getter）。Fowler 本人越来越偏爱直接提炼成函数——因为函数能被复用，而局部变量的名字被"锁死"在一个函数内。

内联：反向手法，撕掉没价值的标签

提炼不是越多越好。内联函数（Inline Function）和内联变量（Inline Variable）是提炼的逆操作，用在"这个名字没有比它的本体讲出更多信息"时。

Bad · 名字是纯噪音
boolean moreThanFive(int n){
  return n > 5;
}
// 函数名没比 n>5 多说任何东西
int getRating(Driver d){
  return moreThanFive(d.trips)
    ? 2 : 1;
}
Good · 内联掉，更直接
int getRating(Driver d){
  return d.trips > 5
    ? 2 : 1;
}
// 一眼看懂，少一层跳转
面试高频对比：内联 vs 提炼不是"谁对谁错"。过度提炼会得另一种坏味道——Middle Man / 间接层泛滥，读代码要在几十个一行函数间反复横跳，同样痛苦。内联还有个经典用法：当一堆函数组织混乱、互相委托绕圈时，先把它们全部内联回一个大函数，看清全貌，再重新按更合理的边界提炼出去。先合并、再拆分，是重构老手的常规套路。

联系底层：这里的"内联"和 JVM 的"内联"是两码事

初学者极易混淆两个"inline"。它们名字相同、方向类似（都是"把调用摊平回本体"），但发生在完全不同的层面——一个是你手动改源码，一个是 JVM 运行期自动做的优化。

维度内联函数（重构手法）方法内联（JIT 优化）
谁做程序员手动改源码HotSpot JIT（C2）运行期自动
目的提升可读性提升运行性能（消除调用开销）
改变的是源代码文本机器码，源码不变
触发条件你判断"名字是噪音"热点方法 & 方法体够小
-XX:MaxInlineSize=35 字节码
这个对比给你吃下定心丸：你为了可读性大量提炼出的小函数，运行时会被 JIT 内联回去，调用开销几乎归零。所以——你在源码层面追求可读性，JVM 在机器码层面负责性能，两者不冲突。别再用"多一层方法调用会变慢"当拒绝拆函数的借口。
JitInlineDemo.java你写的小函数，运行时被摊平
// 你为了可读性提炼出的小方法
private int square(int x) { return x * x; }

public long sumSquares(int n) {
    long s = 0;
    for (int i = 0; i < n; i++)
        s += square(i);          // 循环里被反复调用 → 变成热点
    return s;
}
// 跑够多次后，C2 把 square(i) 内联成 i*i，等价于：
//   for (...) s += i * i;   ← 调用开销消失，源码却依然清晰
// 可用 -XX:+PrintInlining 亲眼看到 "inline (hot)" 日志
再串一条线：Spring 的 @Transactional 反教材。Spring 事务靠 AOP 动态代理实现——只有从代理对象外部进入的方法调用才会被增强。如果你把一段逻辑提炼成同类的 private 方法再自调用（this.xxx()），调用不经过代理，@Transactional 直接失效。这提醒你：提炼函数虽是纯粹的"不改变外部行为"重构，但一旦被提炼的方法上挂着代理型注解（事务、缓存、异步），提炼的边界就可能踩中框架的语义陷阱。

它们都遵守同一条铁律：不改变外部行为

三招（及其逆操作）都属于行为保持（behavior-preserving）的等价变换。这也是"重构"的定义本身：

重构前的输出 ≡ 重构后的输出

正因为是等价变换，现代 IDE（IntelliJ IDEA / Eclipse）才能把它们做成自动化菜单：选中一段代码 → Ctrl+Alt+M（Extract Method）、Ctrl+Alt+V（Extract Variable）、Ctrl+Alt+N（Inline）。IDE 会自动分析变量作用域、生成参数与返回值。强烈建议用 IDE 快捷键而非手敲——机器做等价变换不会漏改引用、不会打错字。

陷阱：手动提炼时最容易漏掉的"副作用"。如果被提炼的表达式里含有 i++、方法调用带 I/O、或读取会变化的字段，那它就不是纯表达式，提炼位置若变了求值时机，行为就变了。所以"提炼前必须有测试托底"——这正是前面章节反复强调"重构的前提是好的测试"的原因。没有测试的重构是在裸奔。
一坨看不懂的代码 提炼出恰当命名的函数/变量 过度提炼(间接层泛滥)

火候：拆到"每个名字都在帮读者理解"就停手；一旦名字开始变成"要跳转才能确认它就是字面意思"的噪音，就该反手内联。

一句话收束：提炼函数 / 提炼变量的本质，是用一个好名字把"意图"从"实现"里解放出来；内联则是当名字沦为噪音时的反向清理。三招都是行为保持的等价变换，因此可被 IDE 自动化、也因此必须有测试托底。记住两条底层保证：你为可读性提炼的小函数，JIT 会替你内联回去（性能不亏）；但只要方法挂着代理型注解，提炼边界就可能踩中框架陷阱（语义要留神）。
