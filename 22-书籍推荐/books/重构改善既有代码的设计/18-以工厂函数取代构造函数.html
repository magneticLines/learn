以工厂函数取代构造函数

构造函数戴着三副天生的枷锁：名字被锁死、只能返回自己、每次必须 new。工厂函数把它们一一解开。

一句话：把 new 包一层

这个手法本身极其朴素——不再让调用方直接 new，而是提供一个普通函数（Java 里通常是静态工厂方法），由它内部去 new。表面上只是多包了一层，实质上却挣脱了构造函数与生俱来的三道限制。

命名小历史：本手法在《重构》第 1 版叫「以工厂方法取代构造函数」（Replace Constructor with Factory Method），第 2 版 Fowler 把示例语言换成 JavaScript，改称「工厂函数」（Factory Function）。名字变了，思想没变。在 Java 世界，它落地的最常见形态就是 Effective Java 第 1 条所说的静态工厂方法。注意：它和 GoF 的「工厂方法模式」不是一回事，后文会专门辨析。

为什么要取代？构造函数的三道天生枷锁

要理解这个重构解决了什么问题，得先看清构造函数被语言强加了哪些限制。这三条不是"写得不好"，而是 new 这个关键字在语法层面就锁死的：

构造函数的三道天生枷锁 ───────────────────────────────────────────── ① 名字锁死 new Employee(...) 方法名只能叫类名 → 无法表达"我造的到底是哪一种员工" → 参数列表相同时，无法再区分两种构造语义 ② 返回自己 new X() 语法上必然得到 X 本身 → 无法返回它的子类 / 代理 / 包装对象 ③ 必然新建 每次调用都分配一个全新对象 → 无法复用、无法做单例 / 享元 / 缓存 ───────────────────────────────────────────── 工厂函数 = 一个普通函数，内部照样 new， 但以上三条一律不受限。
直觉类比：构造函数像"必须自己去后厨现炒一盘同名的菜"；工厂函数像"你对服务员点单"——你说「来份工程师」，至于后厨是现做、从冰箱里端出预制的（缓存），还是端来个"高级工程师"（子类），你都不用管。服务员这一层间接，就是所有灵活性的来源。

最小例子：从魔法字符串到自解释的名字

Fowler 的经典示例：Employee 带一个类型码字段。用构造函数时，调用方被迫记住 "E"、"M" 各代表什么；换成工厂函数后，意图直接写在名字里。

Bad · 魔法字符串 + 构造函数
// 调用方到处硬记类型码含义
Employee e1 =
  new Employee("小明", "E"); // E=工程师?
Employee e2 =
  new Employee("小红", "M"); // M=经理?

// 构造函数名只能叫 Employee，
// 类型码校验散落在每个调用点
Good · 名字自解释的工厂
Employee e1 =
  Employee.createEngineer("小明");
Employee e2 =
  Employee.createManager("小红");

// 构造函数收成 private，
// 类型码与校验集中在工厂里
private Employee(String n, String t){…}
Employee.java工厂函数内部照样 new，只是把选择权收拢
public class Employee {
    private final String name;
    private final String type;

    // ① 构造函数私有：外部无法再直接 new，只能走工厂
    private Employee(String name, String type) {
        this.name = name;
        this.type = type;
    }

    // ② 名字表达意图，参数更少，语义更清晰
    public static Employee createEngineer(String name) {
        return new Employee(name, "E");
    }
    public static Employee createManager(String name) {
        return new Employee(name, "M");
    }
}
重构节奏别贪快：正确的步骤是——先建工厂函数（函数体就是调一次构造函数）→ 把调用点逐个从 new X() 换成工厂 → 全部换完、测试通过后，才把构造函数改 private。如果一上来就把构造函数私有化，还没替换完的调用点会立刻编译报错，你会陷入"一次改一大片"的危险大跃进，违背重构"小步快跑、随时可编译"的铁律。

在 Java 里，它就是"静态工厂方法"

Effective Java 开篇第 1 条就是「考虑用静态工厂方法代替构造函数」。Bloch 列出的优势，正好逐条对应上面三道枷锁的解开：

能力构造函数 new静态工厂方法
有意义的名字✗ 只能叫类名✓ of/valueOf/parse…
可不新建对象（缓存/单例）✗ 必然 new✓ Integer.valueOf 缓存
能返回子类型✗ 只能返回本类✓ EnumSet.of 返回子类
返回类型随参数动态变✗✓ 依入参选实现类
返回类"写代码时还不存在"✗✓ SPI：JDBC / ServiceLoader
能被子类继承✓✗ 构造私有时不能被继承
在 API 文档中显眼易发现✓ 单独列出✗ 混在方法里，需靠命名约定
记住这套命名约定（面试会问）：静态工厂不像构造函数在文档里独立成组，读者容易漏看，所以社区约定了一套"一看就知道是工厂"的名字：from（类型转换）、of（聚合多个参数）、valueOf、instance/getInstance、create/newInstance、getType/newType。见到它们，基本可断定背后是工厂而非 new。

枷锁二解开：装箱背后的缓存工厂（== 陷阱）

"每次必 new"被解开的最著名例子，就藏在 Java 自动装箱里。Integer a = 127; 编译器并不会 new Integer，而是插入 Integer.valueOf(127)——一个会命中缓存的工厂方法。

java.lang.Integer装箱其实走的是工厂方法，不是 new
public static Integer valueOf(int i) {
    // 默认缓存区间 -128 ~ 127（上界可用 JVM 参数调）
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)]; // 命中：复用同一对象
    return new Integer(i);   // 越界才真正新建
}
AutoboxTrap.java经典 == 陷阱：同样的写法，结果却不同
Integer a = 127, b = 127;   // → valueOf(127)，命中缓存
System.out.println(a == b);       // true  ← 是同一个缓存对象

Integer c = 128, d = 128;   // → 128 越界，各自 new
System.out.println(c == d);       // false ← 两个不同对象！
System.out.println(c.equals(d));  // true  ← 比"值"才对
高频面试陷阱：为什么 127 相等、128 不等？根因就是 valueOf 这个工厂对小整数做了缓存复用（享元模式），而 == 比的是对象引用。包装类型永远用 equals 比较值。同理 Boolean.valueOf 从不新建对象（永远返回 Boolean.TRUE/FALSE 两个常量）。这正是"工厂能不 new"带来的能力——构造函数 new Integer(127) 每次都是新对象，永远 == 不相等，所以 new Integer 早已被标记为过时。

枷锁三解开：返回子类型，与多态串起来

上一节讲的以多态取代条件表达式，落地时几乎离不开工厂——因为要让调用方"只认抽象、不认具体子类"，就得由工厂来决定实例化哪个子类。EnumSet 是 JDK 里最漂亮的示范：

java.util.EnumSet工厂按输入规模，返回不同实现子类
public static <E extends Enum<E>> EnumSet<E> noneOf(Class<E> elementType) {
    Enum<?>[] universe = getUniverse(elementType);
    if (universe.length <= 64)
        return new RegularEnumSet<>(elementType, universe); // ≤64 个：单个 long 位向量，最省
    else
        return new JumboEnumSet<>(elementType, universe);   // >64 个：long[] 位向量
}
// 调用方永远只写 EnumSet.of(...)，从不 new 具体实现——
// 将来 JDK 想换第三种实现，调用方一行都不用改。
这解释了"面向接口编程"为什么必须搭配工厂：你把变量声明成抽象类型 EnumSet，就不能再 new EnumSet(...)（抽象类无法实例化，且你也不想在调用点写死是 Regular 还是 Jumbo）。是工厂替你把"选哪个实现"这个决定封装了起来——抽象靠接口，选择靠工厂，二者天生一对。Collections.unmodifiableList、List.of、Optional.empty 全是同一套路。

辨析：静态工厂方法 ≠ 工厂方法模式

这是初学者最容易混的一组概念。"以工厂函数取代构造函数"产出的是静态工厂方法（一种编码惯用法），它不是 GoF 设计模式；而"工厂方法模式"是靠继承 + 多态实现的正经设计模式。三者区别如下：

名称是 GoF 模式吗核心机制典型代表
静态工厂方法
（本手法/EJ 第1条）		否，是惯用法类内加静态方法，替代/包裹构造函数Integer.valueOf
简单工厂否一个方法内 switch 造不同产品按类型码 return
工厂方法模式是父类声明抽象工厂方法，子类覆写决定造哪个产品ArrayList.iterator()
抽象工厂是一个接口创建一族相关产品DocumentBuilderFactory
Spring 里再多两个易混概念：BeanFactory vs FactoryBean。BeanFactory 是 IoC 容器的顶层接口（整个容器本身就是个大工厂）；FactoryBean<T> 则是一个特殊的 Bean，它实现 getObject() 来生产别的对象——当你按名字 getBean("foo") 取一个 FactoryBean 时，拿到的是它 getObject() 造出来的产品；想拿 FactoryBean 本身，得加前缀 getBean("&foo")。此外，配置类里的 @Bean 方法，本质就是一个个"以工厂函数取代 new"的静态/实例工厂——你让 Spring 通过工厂而非直接 new 来装配对象，正是这个重构手法在框架层面的规模化应用。

取舍：什么时候别用工厂

任何手法都有代价。工厂不是越多越好，下面这些情形要克制：

new X(...)  ⟶  X.ofXxx(...)
灵活性 ↑    可发现性 ↓    可被继承性 ↓
  • 类需要被继承时慎用私有构造。若把构造函数私有、只留静态工厂，子类就无法调用父类构造函数，等于堵死了继承。JDK 的 Collections（工具类，本就不该被继承）用私有构造正合适；但一个业务领域基类若还想被扩展，就别一刀切私有化。
  • 没有额外收益就别包。如果只是把 new Point(x,y) 改成 Point.of(x,y)、内部啥也没多做，那只是徒增一层、还牺牲了构造函数在文档里的可发现性——这属于过度设计（YAGNI）。等到真正需要"命名/缓存/返回子类"其中之一时，再重构不迟。
  • DI/框架环境下别绕过容器。在 Spring 里对着受管 Bean 手写 new 固然不对，但自己再造一个静态工厂去 new，同样绕过了容器的依赖注入与代理增强。此时"取代构造函数"的正解是交给容器（@Bean/@Component），而非手搓工厂。
到处 new + 魔法参数 恰当的工厂函数（命名/缓存/返回子类） 为包而包(过度设计)

火候：当"命名、缓存复用、返回子类型、按参数选实现"这四条里至少中一条时，工厂就物有所值；一条都不中，就让 new 保持素颜。

一句话收束：「以工厂函数取代构造函数」= 用一层普通函数把 new 包起来，换取构造函数三道枷锁的解放——名字能表意（createEngineer）、对象可复用（Integer.valueOf 缓存）、返回可为子类（EnumSet.of）。它在 Java 里的形态就是 Effective Java 第 1 条的静态工厂方法，注意别和 GoF 的工厂方法模式混为一谈。但要克制：需要继承、或没有任何额外收益时，一个朴素的 new 反而更诚实。
