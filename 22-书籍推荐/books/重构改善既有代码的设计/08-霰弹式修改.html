霰弹式修改：一个改动要改很多类

改一个需求，却要在十几个文件里东补一枪西补一枪——散弹枪打出的一片弹孔，正是"该在一起的东西被拆散了"的报警灯。

一句话看懂：什么是霰弹式修改

Martin Fowler 的定义非常形象：每遇到某种变化，你都必须在许多不同的类里做出许多小修改（Shotgun Surgery）。名字来自散弹枪——扣一下扳机，弹丸打出一大片分散的弹孔。对应到代码，就是"一个逻辑上单一的改动，需要你在四处零零散散地各改一点点"。

核心原理：霰弹式修改的病根是「同一个变化原因」被散落在了多个模块里。理想的代码结构应该让"因为同一个原因而变化的东西"聚在一起——这样一次需求改动只落在一个地方。当它们被打散，你就被迫做"考古式修改"：先满世界找出所有相关点，再一个个改，还极容易漏改一处（编译器不会提醒你"还有一枪没补"）。

生活类比

想象你家的电灯开关：如果"客厅灯的开关"被拆成了三段，一段在客厅、一段在卧室、一段在阳台——每次想调客厅灯，都得跑三个房间各拨一下，还常常忘了阳台那一段。霰弹式修改就是这种感觉：一个概念的控制权被物理地分散了。

它和"发散式变化"是一对孪生兄弟

上一篇讲的发散式变化（Divergent Change）和本篇的霰弹式修改，是同一枚硬币的两面，也是本章最容易混淆的高频考点。二者都在描述"变化原因"与"代码模块"之间的错配，但方向正好相反：

发散式变化 (Divergent Change) 霰弹式修改 (Shotgun Surgery) 原因A ─┐ 原因A ──┬──> 类① 原因B ─┼──> 一个类 (改这一个) ├──> 类② 原因C ─┘ ├──> 类③ └──> 类④ "一个类，被多个原因反复揉搓" "一个原因，散弹式打中多个类" 一处代码 ← 多个变化 一个变化 → 多处代码
维度发散式变化霰弹式修改
症状打开一个类，为了各种不相干的原因反复改它为了一个原因，被迫打开一大堆类各改一点
错配方向一个模块承担了多个职责一个职责被拆散进多个模块
违反原则低内聚（职责混杂）高散度 / 高耦合（职责分散）
主要解药拆分：Extract Class / Extract Function聚合：Move Function/Field、Combine Functions into Class
方向记忆把大杂烩拆开把散落物收拢
一招记牢：看你打开了几个文件。改需求时如果反复回到同一个文件、却为了不同原因，是发散式变化（该拆）；如果一次改动要来回切换一堆文件，是霰弹式修改（该合）。前者"分不清"，后者"合不拢"。

病根：违反了"单一职责原则"

为什么"该在一起的东西要在一起"？这不是玄学，而是 Robert C. Martin 对单一职责原则（SRP）最本质的表述——它常被误解成"一个类只做一件事"，其真正含义是：

因同一原因而变化的东西 聚在一起
因不同原因而变化的东西 彼此分开

把这句话代进来就通透了：霰弹式修改 = "因同一原因而变化的东西"没有聚在一起。所以它的解药永远指向"聚合"——用 Move Function（搬移函数）、Move Field（搬移字段） 把散落各处的相关行为搬到同一个模块，理想情况下让一个模块专门吸收这一类未来的变化。这也印证了软件设计的一条黄金法则："一起变化的代码，应该放在一起"（things that change together belong together）。

反例 → 正例：一次改动打中三个类

设想一个电商系统，"税率如何计算"这段逻辑被复制粘贴进了下单、退款、报表三个服务。当国家调整税率、或新增一档"数字商品免税"规则时，你必须同时改三处——这就是标准的霰弹式修改。

Bad · 税率逻辑散落三处
class OrderService {
  double total(Order o){
    return o.amt * 1.13; // 税率
  }
}
class RefundService {
  double refund(Order o){
    return o.amt * 1.13; // 又一处
  }
}
class ReportService {
  double tax(Order o){
    return o.amt * 0.13; // 还漏改成 0.13!
  }
}
Good · 收拢到一个 TaxPolicy
class TaxPolicy {
  static final double RATE = 0.13;
  double taxOf(double amt){
    return amt * RATE;
  }
  double withTax(double amt){
    return amt + taxOf(amt);
  }
}
// 三个 Service 都委托它
// 调税率？只改这一个类
陷阱：复制粘贴是霰弹式修改的头号温床。注意上面 Bad 版第三处偷偷写成了 0.13（少乘了个 1）——这正是散弹式改动最阴险的地方：你以为改全了，其实漏了一枪，而且各副本还悄悄"变异"了。这类 bug 编译器查不出、测试若不全也测不到，往往在生产环境的对账差异里才暴露。所以看到"同一段逻辑在多个类里出现"，几乎可以直接判定为潜在的霰弹式修改。

另一种典型信号：满地的 switch / if-else

当"新增一种类型"要求你去好几个 switch 里各加一个 case 时，也是霰弹式修改。这在面试里常以"如何用多态消除 switch"的形式出现。

ShotgunSignal.java散弹信号
// 新增一种支付方式 WECHAT，要改下面每一处 switch —— 霰弹式
double fee(PayType t){ switch(t){ case ALIPAY: … case CARD: … } }
String icon(PayType t){ switch(t){ case ALIPAY: … case CARD: … } }
boolean refundable(PayType t){ switch(t){ case ALIPAY: … } }

// 正解：把三段行为收进 enum / 子类，加类型只动一处
enum PayType {
  ALIPAY { double fee(){…} String icon(){…} },
  CARD   { double fee(){…} String icon(){…} };
  abstract double fee();  abstract String icon();
}
为什么多态能治它？把分散的 switch 收进带方法体的枚举（或子类）后，"某个类型的所有行为"被物理地聚合在一起。将来新增 WECHAT，你只在一个地方补齐它的 fee/icon/refundable，编译器还会强制你把抽象方法实现全——漏改直接编译失败。这就是"用编译期约束替代人肉搜索"，把霰弹式修改从"跑遍全场"降级为"补齐一处"。

解药手册：三种"收拢"手法

手法适用场景做法
Move Function / Move Field
（搬移函数 / 字段）		相关行为/数据散落在多个类把它们逐个搬到最该拥有它的那个类，向"一个模块吸收一类变化"靠拢
Combine Functions into Class
（函数组合成类）		一组函数总是操作同一批数据、总是一起改连同数据打包成一个新类，让它成为这类变化的唯一落点
Inline Function / Inline Class
（内联函数 / 内联类）		逻辑被过度切碎、散在多处中间层先内联合并回一处，再重新按职责切一刀（"先聚拢再重分"）
实战节奏：治霰弹式修改常用"先内联、再提炼"的组合拳——先用 Inline 把散落的碎片暂时揉回一坨（哪怕短暂制造一个过长函数/过大类），看清全貌后，再用 Extract 按正确的职责边界重新切分。这比"隔着好几个文件盲目搬运"稳得多，也更容易保持每一步都可编译、可测试。

联系 Spring：AOP 就是为消灭霰弹式修改而生

霰弹式修改最经典的现实场景是横切关注点（cross-cutting concern）：日志、事务、权限、缓存这类逻辑，天然会"横穿"业务代码，散落在几百个方法里。想统一改一下"事务的传播行为"？——不用 AOP 的话，就是一次超大规模的霰弹式修改。

Bad · 事务代码散落每个方法
void placeOrder(){
  tx.begin();
  try { …业务… tx.commit(); }
  catch(Exception e){ tx.rollback(); }
}
// 每个方法都抄一遍这坨模板
// 想改回滚策略？改 N 个方法
Good · AOP 把它收进一处切面
@Transactional
void placeOrder(){ …纯业务… }

// 事务模板被 Spring 的
// TransactionInterceptor 统一织入
// 改策略只动一个切面/配置
本质对应：Spring AOP 用动态代理（JDK Proxy / CGLIB）在运行期把 @Transactional、@Cacheable 这些横切逻辑"织入"到目标方法周围。它做的事，正是把一个原本会散弹式打中所有方法的关注点，收拢到一个切面里集中管理——AOP 之于横切关注点，就是重构之于霰弹式修改，同一种"聚合"思想在框架层面的产品化。你之前学的代理模式、@Transactional 失效场景（自调用不走代理），根子都在这里。
别把钟摆甩到另一头。治霰弹式修改是"把散的收拢"，但收得过猛，就会把八竿子打不着的职责硬塞进一个类，反而制造出发散式变化（一个类被各种原因反复改）。二者是一对需要平衡的张力：目标不是"越集中越好"，而是让"变化原因"与"模块"尽量一一对应。判断标准始终是那句话——因同一原因变化的放一起，因不同原因变化的分开。
一句话收束：霰弹式修改的病根是「一个变化原因被拆散进了多个模块」，与发散式变化（一个模块被多个原因揉搓）正好互为镜像。解药是用 Move Function/Field、Combine Functions into Class 把散落的相关逻辑收拢到一处，让一次需求改动只落在一个地方。它背后是同一条设计公理：因同一原因而变化的代码，就该住在一起——Spring AOP 消灭横切关注点，用的正是这条公理。
