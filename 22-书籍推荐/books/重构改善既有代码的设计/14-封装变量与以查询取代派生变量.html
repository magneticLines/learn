封装变量与以查询取代派生变量

给数据装上"门",让所有访问都从门口过;再把能算出来的东西还原成"算",别让它变成第二份真相。

一、为什么数据比函数更难改

Fowler 在书里说过一句耐人寻味的话:重构函数很容易,重构数据却很难。原因在于——函数天然有"调用点"这个拦截层:你想改一个函数的行为,只要改函数体,所有调用它的地方自动生效。但裸露的数据没有这层拦截:一个 public 字段散落在几十处被直接读写,你想给它加一条校验、一行日志、一把锁,得挨个改这几十处。

一句话抓住本质: 封装变量(Encapsulate Variable)做的事,就是给数据人为地造出一个"调用点"——把字段藏进 private,逼所有访问都经过 getter/setter 这道窄门。于是"改数据的行为"退化成"改一个函数",难题就被拉回到我们最擅长的战场。
裸变量 (public field) 封装后 (private + accessor) ───────────────────── ─────────────────────────── order.total = 100 ┐ order.setTotal(100) ┐ order.total = -5 ┤ N 个写入点 order.setTotal(-5) ┤ 仍是 N 个调用 order.total += tax ┘ 分散各处 order.setTotal(...) ┘ │ │ ▼ ▼ 无处可拦截 setTotal(v){ /* 单一入口 */ } 想加校验? 改 N 处 想加校验 / 日志 / 事件 / 锁? 出了 bug? 全库是嫌疑人 只改这 1 处

这正是我们在《全局数据》一节里对付"全世界都能改的库存"用的同一招:封装不是为了"面向对象的仪式感",而是为了把数据变更的入口收窄到一处,让影响半径可控。

二、封装变量的手法(反例 → 正例)

手法本身极其朴素,但价值巨大。步骤:① 新建取值/设值函数;② 逐个把直接访问改成调用函数(小步、频繁测试);③ 把字段设为 private,让编译器帮你揪出漏网之鱼。

Bad · 裸露公有字段
public class Order {
    // 谁都能直接读写,无拦截点
    public double total;
}

// 调用处散落各地:
order.total = 100;
order.total = -5;      // 负数?没人拦
order.total += tax;

// 想给 total 加"不能为负"的校验
// 得翻遍全库改每一处赋值
Good · 封装 + 单一入口
public class Order {
    private double total;   // 藏起来

    public double getTotal() {
        return total;
    }
    public void setTotal(double v) {
        if (v < 0)          // 想加校验/日志
            throw new IllegalArgumentException();
        this.total = v;     // 只需守住这一处
    }
}
// 编译器逼所有外部访问走窄门
串联 Spring / JavaBean 规范: 为什么 Java 生态几乎所有框架(Spring、Jackson、MyBatis、Lombok)都围着 getXxx/setXxx 转?正因为封装变量把"访问数据"统一成了"调用方法",框架才有机会在这道门上做文章——Spring AOP 能在 setter 上织入事务/日志,Jackson 靠 getter 序列化,JPA 靠 setter 回填。访问器就是数据的"扩展点"。这也是 Fowler 建议"哪怕现在 getter 里啥都没干,也值得先封装"的原因:门先立在那,将来才有地方挂东西。
陷阱:封装 ≠ 无脑生成 getter/setter。 初学者常把封装理解成"给每个字段配一对 get/set",结果 setter 全裸——外部照样能随意改内部状态,封装形同虚设。真正的封装要问:这个字段到底该不该允许外部改? 不该改的就只给 getter、不给 setter(甚至设成 final);返回集合时用 Collections.unmodifiableList(...) 或返回副本,防止内部可变对象"引用逃逸"。全套 setter 只是把 public 字段换了层皮。

三、以查询取代派生变量:消灭"第二份真相"

封装解决的是"数据被谁改"的问题;以查询取代派生变量(Replace Derived Variable with Query) 解决的是另一个更隐蔽的病:同一个事实被存了两份。

所谓派生变量,就是一个能从别的数据算出来的值。购物车的 totalPrice 可以由 items 算出来;矩形的 area 可以由 width * height 算出来。当你把这种"算得出来的值"额外存成一个字段,就等于让系统里出现了两份需要彼此吻合的真相——而只要有两份真相,它们迟早会打架。

派生数据 = f( 源数据 )　　⇒　　该算,不该存
Bad · 冗余存储派生字段
class Cart {
  private List<Item> items = new ArrayList<>();
  private double totalPrice;  // 派生:被冗余存储

  void add(Item it) {
    items.add(it);
    totalPrice += it.price(); // 记得手动同步
  }
  void remove(Item it) {
    items.remove(it);
    // 忘了减 totalPrice!
    // → items 与 total 不一致的 bug
  }
  double total() { return totalPrice; }
}
Good · 每次查询算出来
class Cart {
  private final List<Item> items = new ArrayList<>();

  void add(Item it)    { items.add(it); }
  void remove(Item it) { items.remove(it); }

  // items 是唯一真相,total 现算现得
  double total() {
    return items.stream()
                .mapToDouble(Item::price)
                .sum();
  }
}
// 不可能不一致:根本没有第二份数据

左边的 bug 几乎是必然的:任何一处修改 items 却忘了同步 totalPrice,数据就对不上了。而且这种 bug 极难排查——因为不一致是"存"出来的,不是"算"错的。右边把派生字段删掉、改成一个查询方法后,items 成了唯一数据源(single source of truth),不一致的可能性从根上被消灭了。

核心结论: 冗余的可变状态是 bug 的温床。凡是能算出来的,优先算,别存——你少维护一个字段,就少一处"忘记同步"的入口。这与函数式编程"减少可变状态"、数据库设计"消除冗余"是同一条心法的不同投影。

四、跨层的同一心法:范式、派生状态、缓存

"以查询取代派生变量"绝不只是个 OO 小技巧,它在整个工程栈里反复出现:

层次"存派生"的坏味道"以查询取代"的对应做法
对象内部字段 totalPrice 与 items 冗余删字段,改成 total() 现算
数据库(3NF)order 表存 total 列,与明细行冗余用视图 / SUM() 查询算出,消除传递依赖
前端(React)把算得出的值塞进 useState,再手动 setState 同步渲染时直接由 props/state 计算(derived value)
缓存层缓存与 DB 双写、难保一致能重算的就设 TTL 让它过期重算
为什么数据库范式(3NF)反对冗余? 第三范式要求"消除对主键的传递依赖",本质就是禁止存派生列。订单表若同时存了 total 和明细,某天改了一条明细的单价却忘了更新 total,财务对账就会对不上——和购物车的 bug 一模一样。这就是为什么老练的 DBA 宁可查询时用 SUM() 现算,也不轻易存冗余汇总列;实在为了性能要存,也会用触发器/物化视图把"同步"这件事自动化,而不是交给业务代码去记。

五、什么时候"该存"?——缓存是有正当理由的例外

把话说满就会翻车:并不是所有派生值都该现算。当计算代价高且结果被频繁读取时,存一份缓存是合理的性能优化。关键区别在于:安全的缓存,前提是源数据不可变或缓存有失效机制。JDK 里就有教科书级的例子——String.hashCode():

java.lang.String(简化)JDK · 派生值的安全缓存
private int hash;              // 缓存的 hashCode,派生自 value[]

public int hashCode() {
    int h = hash;
    if (h == 0 && value.length > 0) {   // 只有没算过才算
        h = computeHash(value);
        hash = h;                // 算一次,缓存起来
    }
    return h;
}

String 是不可变的,底层 value[] 永不改变,所以 hash 这份"派生缓存"永远不会过期(stale)。这正是缓存派生值的黄金条件:源不变 ⇒ 派生也不变 ⇒ 缓存永远正确。反过来看购物车——items 随时会变,所以缓存 totalPrice 才那么危险。同理 ArrayList 用一个 size 字段缓存元素个数(而非每次数一遍),换来 size() 的 O(1),代价是每次增删都必须同步维护 size——这是它主动接受的一笔"一致性债务"。

维度存储派生(缓存字段)以查询取代(现算)
数据一致性差:需手动同步,易出 stale bug好:唯一真相,不可能不一致
读性能快:O(1) 直接取可能慢:每次重算
写复杂度高:每处改源都要记得同步低:只改源,派生自动跟上
适用场景源不可变 / 计算昂贵且高频读默认首选,尤其源频繁变化
高频陷阱与面试点:
① 过早为性能存派生——先写成查询,等 profiler 真的指出它是瓶颈,再引入带失效机制的缓存(memoize)。别拿"怕慢"当冗余状态的借口,这是典型的 premature optimization。
② 缓存了可变源的派生值又忘了失效——线上最常见的一类"数据对不上"事故。缓存的难点从来不是"存",而是"什么时候让它失效"(cache invalidation)。
③ 把副作用藏进派生 getter——查询方法应当是纯的(幂等、无副作用)。如果 getTotal() 里偷偷改了别的字段,调用者多调一次就出 bug,直接违背"以查询取代"的初衷。
两招合起来记: 封装变量是给数据"立一道门",把散乱的访问收拢成单一入口,让加校验、加日志、加锁都只改一处;以查询取代派生变量是把"存出来的第二份真相"还原成"算出来的一份真相",从根上消灭不一致。前者管"谁能改",后者管"该不该存"——两者都指向同一个更高目标:把可变状态压到最小,让代码的行为可推理、影响半径可控。
