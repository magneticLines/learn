重构的前提：好的测试

没有测试的重构，本质上只是"在悬崖边上闭着眼睛跳舞"——你改的到底是结构还是行为，全凭运气。

一、为什么"没测试就别重构"是一条铁律

重构的定义（见上一节）是：在不改变外部可观测行为的前提下，改善内部结构。这句定义里藏着一个致命的隐含要求——你必须有办法证明"外部行为没变"。否则你嘴上说着"我只是重构"，手上却可能悄悄引入了 bug 而毫不知情。

Martin Fowler 在书里说得极其直白："如果你想重构，前提是你得有一套可靠的测试。" 这不是一句励志口号，而是一条工程上的因果链：

重构 = 改结构 ∧ 行为不变　⟹　需要一台"行为不变探测器"

这台"探测器"就是自动化测试。它的作用不是帮你写出好代码，而是在你每一次小改动之后，几秒钟内回答一个问题："我刚才有没有把东西改坏？" 有了它，重构才从"提心吊胆的手术"变成"有安全网的杂技"。

换个视角：测试是重构的"回归基线"。 重构前，测试全绿——这是你的"存档点"。重构中，你频繁运行测试；一旦某个测试变红，你立刻知道"就是刚才那几行改错了"，问题范围被死死锁定在最近这一小步里。这和 Git 的作用异曲同工：Git 让你敢删代码（因为能回滚），测试让你敢改代码（因为能验证）。

二、"自测试代码"（Self-Testing Code）

Fowler 把这套实践提炼成一个概念——自测试代码：代码库里内建一套能自动运行、自动判定对错的测试，一条命令跑完就告诉你"全绿 / 有红"，无需人肉点点点去验证。这是 Kent Beck 在极限编程（XP）里最重要的贡献之一，也是 JUnit 之所以诞生的原因。

历史彩蛋： JUnit 正是 Kent Beck 与 Erich Gamma（《设计模式》GoF 四人组之一）在一次飞机航程上写出来的。它把"写测试"这件苦差事的成本压到极低——一个 @Test 注解、一个 assertEquals，让"自测试代码"从理念变成了整个 Java 生态的肌肉记忆。你今天在 Spring Boot 里随手写的 @SpringBootTest，血脉源头就在这里。

一段最小的"自测试代码"长这样——注意它是如何把"预期行为"钉死成可执行的断言的：

被测代码 + 它的"行为快照"Java · JUnit 5
// 被测：一个计算购物车折扣的方法（重构目标）
public class PriceCalculator {
    public double finalPrice(double price, int qty) {
        double total = price * qty;
        if (total > 1000) total *= 0.9;   // 满 1000 打 9 折
        return total;
    }
}

// 测试：把"外部可观测行为"固化成可执行的断言
class PriceCalculatorTest {
    PriceCalculator calc = new PriceCalculator();

    @Test void noDiscountBelowThreshold() {
        assertEquals(200.0, calc.finalPrice(100, 2), 1e-9);
    }
    @Test void applyDiscountOverThreshold() {
        // 1200 > 1000，应打 9 折 = 1080
        assertEquals(1080.0, calc.finalPrice(600, 2), 1e-9);
    }
    @Test void boundaryExactly1000_noDiscount() {
        // 边界：等于 1000 时【不】打折（> 而非 >=）
        assertEquals(1000.0, calc.finalPrice(500, 2), 1e-9);
    }
}

现在这三条测试就是 finalPrice 的"行为契约"。无论你之后怎么重构它的内部——提炼函数、以多态取代 if、引入策略对象——只要这三条依然全绿，你就有底气说"行为没变"。

三、重构与测试的节奏：红绿交替的心跳

重构不是"改一大堆然后跑一次测试"，而是改一小步 → 跑测试 → 再改一小步的高频循环。测试就是这个循环的"心跳监护仪"：

┌─────────────────────────────────────────────────────┐ │ ① 跑测试 → 全绿 ✔ （确认起点是干净的"存档点"） │ │ │ │ │ ▼ │ │ ② 做一次【极小】的重构（提炼一个变量 / 改个名） │ │ │ │ │ ▼ │ │ ③ 跑测试 ──┬── 全绿 ✔ ─→ 回到 ②，继续下一小步 │ │ │ │ │ └── 变红 ✘ ─→ 立刻 undo / 修（问题只在这一步）│ └─────────────────────────────────────────────────────┘ 测试跑得越快，这个心跳循环就能转得越快
陷阱：把"测试慢"当成不跑测试的借口。 如果测试要跑 10 分钟，你就不会频繁跑，重构循环被拉长成"改半天跑一次"——一旦变红，问题范围是"这半天的所有改动"，锁不定。这就是为什么单元测试必须快（毫秒级）：速度直接决定你敢不敢小步走。反过来，一堆连数据库、连网络的"慢测试"，对重构的支撑作用会大打折扣。

四、什么才算"好的测试"？

"有测试"和"有好测试"是两回事。一套拖后腿的测试甚至比没有更糟——它会让你误以为有安全网，或者每次重构都要陪着改一大堆测试。好的重构安全网，追求以下特质：

特质含义为什么对重构关键
快（Fast）整套单元测试秒级跑完快才敢频繁跑，才敢小步走
测行为，不测实现断言"给什么输入得什么输出"，而非"内部调了哪个私有方法"重构就是改实现，测试若绑死实现，一重构就集体变红
独立可重复不依赖执行顺序、不依赖上次残留的数据否则红/绿是随机的，安全网变成"薛定谔的网"
自验证结果只有"通过/失败"，不需要人眼比对日志人肉判断慢且易错，无法支撑高频循环
覆盖到位正常路径 + 边界 + 异常分支都有没覆盖到的分支，重构改错了测试也不会报警
覆盖率的正确用法： 行覆盖率（line coverage）是个反向指标——低覆盖率一定有问题（大片代码没测），但高覆盖率不保证测试好（可能只调用了代码却没断言任何东西）。别把 100% 覆盖率当 KPI 去刷。Fowler 的建议朴素而实用："当你不再害怕会有 bug 溜过测试时，覆盖率就够了。"

五、反例 → 正例：脆弱的测试 vs 抗重构的测试

同一个需求，测试写法不同，对重构的支撑天差地别。核心分水岭：你在测"它做了什么"（行为），还是测"它怎么做的"（实现细节）？

Bad · 绑死实现细节
// 用 mock 断言"内部调用了哪些方法"
@Test void save() {
  service.register(user);

  // 一重构（比如合并两次DB调用），
  // 这些验证就全红——哪怕行为没变
  verify(dao).insert(user);
  verify(dao).updateIndex(user);
  verify(cache).evict("users");
}
// 痛点：
// · 测的是"过程"不是"结果"
// · 重构=改过程，测试必然连坐
// · 沦为"改代码就得改测试"的负担
Good · 只断言可观测行为
// 只关心"注册后能不能查到这个用户"
@Test void register_thenQueryable() {
  service.register(user);

  User found =
      service.findByEmail(user.email());
  assertEquals(user.name(),
               found.name());
}
// 收益：
// · 内部怎么存、存几次、清不清缓存
//   都是实现细节，随便重构
// · 只要"注册完能查到"这一行为不变
//   测试就一直绿——真正的安全网
过度 mock 是重构的头号敌人。 用 verify() 去校验"内部调用序列"，等于把测试焊死在当前实现上。Fowler 和 Kent Beck 都强调：测试应该像客户端一样使用你的代码——客户端不关心你内部调了几次 DAO，只关心结果对不对。Mock 应留给"真的难以在测试里构造的外部依赖"（如第三方支付网关），而不是拿来监视自家代码的一举一动。

六、没有测试的遗留代码怎么办？——先补网，再动手

现实里最扎心的场景：你面对的偏偏是一坨又乱又没测试的祖传代码——正因为没测试，它才烂成这样；正因为烂，你才想重构。这是个"先有鸡还是先有蛋"的死结。

解法（出自 Michael Feathers《修改代码的艺术》，与本书互补）： 先写特征测试（Characterization Test）。它不问"代码应该输出什么"，而是先跑一遍，把它当前实际输出的结果原样固化成断言——哪怕这个结果本身是错的。目的不是判对错，而是给"当前行为"拍一张快照，作为重构前后的对比基线。织好这张网，才敢下刀。
特征测试：给遗留代码"当前行为"拍快照Java
@Test void characterize_legacyBehavior() {
    // 不知道它"该"返回啥，先跑一次看实际返回什么
    String actual = legacy.process("2026-07-12", 3);
    // 把跑出来的结果原样钉死——这就是"当前行为"的快照
    assertEquals("OK#7#DELTA", actual);
    // 之后任何重构都不许改变这个输出；等真正理解了逻辑
    // 再决定这行为是否是 bug、要不要单独修
}

七、串联：从 JUnit 到 Spring 到 CI

"好的测试是重构前提"这条原则，早已被整个 Java 生态制度化，你每天都在用它：

  • JUnit / AssertJ：提供 @Test、assertThat(...)，让"自测试代码"的编写成本趋近于零。
  • Spring Test：@SpringBootTest、@DataJpaTest、MockMvc 让你能在贴近真实运行环境下测行为——重构 Service 层时，一整套集成测试就是你的安全网。
  • Testcontainers：用真实的 MySQL/Redis 容器跑测试，避免"mock 出来的假 DB 掩盖了真 bug"，让"测行为不测实现"落到实处。
  • CI（如 GitHub Actions）：每次 push 自动全量跑测试。这是"自测试代码"的团队级放大——把个人重构的安全网，扩展成整个团队协作重构的护栏。你项目里 main 分支推送即部署，同理，测试全绿才敢部署。
面试高频串联： 被问"你怎么保证重构不引入 bug？"——标准答案就是这条链：先有覆盖行为的测试（红绿基线）→ 小步重构 → 每步跑测试 → CI 兜底。若再追问"TDD 和重构什么关系？"：TDD 的"红-绿-重构"三步循环里，第三步就是本书讲的重构；TDD 天然产出了"好的测试"，因此和重构是绝配——测试先行，安全网就已经在那儿了。

八、常见误区一次说清

误区一："我改得很小，不用跑测试。" 恰恰相反，正因为改得小，跑测试的成本才低、定位才准。人的直觉判断在"这么点改动不可能出错"上失手过无数次——空指针、边界、精度、副作用，全藏在"看起来没问题"里。
误区二："先重构，回头再补测试。" 顺序反了。没有基线测试，你根本无法证明重构过程中行为没变——等你重构完，早已分不清"这个诡异表现是我改出来的，还是它本来就这样"。测试必须先于重构存在。
误区三："测试和被测代码一起改，还叫安全网吗？" 这是最隐蔽的陷阱。重构时若发现测试变红就顺手改测试让它变绿，安全网就形同虚设。铁律：重构阶段只允许改产品代码，绝不同时改测试的断言；测试要变，得在独立的、非重构的步骤里、并且有充分理由地改。
一句话收束： 重构的胆量不是来自"我很仔细"，而是来自"我有一张随时能兜住我的网"。好的测试 = 一套快、测行为不测实现、可自动重复运行的安全网；它把"改代码"从赌博变成了可验证的工程动作。所以本书真正的第一步从来不是背重构手法，而是：先确认（或补齐）你的测试。没有绿灯，就不要开始重构。
