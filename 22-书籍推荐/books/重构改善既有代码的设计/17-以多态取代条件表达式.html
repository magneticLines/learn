以多态取代条件表达式

当同一个 switch 在代码里反复出现，就别再靠 case 分派了——让每个类型自己回答"我该怎么做"。

它到底在解决什么问题

一段 switch / if-else 本身没有错。真正的坏味道是：同一套"按类型分支"的逻辑，在系统里被复制、散落到好几个方法里。每加一种新类型，你就得回去把每一处 switch 都补一个 case——漏掉一个就是线上 bug。这正是上一节讲的霰弹式修改与基本类型偏执 / switch 滥用的合流。

「以多态取代条件表达式」（Replace Conditional with Polymorphism）的思路是：把"根据类型走不同分支"这件事，从调用方的 switch，下沉成各个子类对同一个方法的不同实现。分派工作交给语言的动态绑定去做，你的代码里就再也看不到那个 switch 了。

一句话直觉：switch 是"外人拿着一张清单，挨个问'你是谁？那你该干嘛'"；多态是"直接把活派给对象，它自己知道该怎么干"。前者是过程式思维，后者是面向对象的本质——数据和处理数据的行为绑在一起。这也是 OOP 相对于面向过程最核心的价值点。

反例 → 正例：员工薪酬计算

经典场景：不同工种的月薪算法不同。用类型码 + switch 是最容易写、也最容易腐烂的写法。

Bad · 类型码 + switch
class Employee {
  static final int ENGINEER=0,
    SALESMAN=1, MANAGER=2;
  int type;
  int salary, commission, bonus;

  int payAmount() {
    switch (type) {
      case ENGINEER:
        return salary;
      case SALESMAN:
        return salary + commission;
      case MANAGER:
        return salary + bonus;
      default:
        throw new IllegalStateException();
    }
  }
  // getLeaveDays()、getTitle()…
  // 每个都得再 switch 一遍 type
}
Good · 让子类自己回答
abstract class Employee {
  int salary;
  abstract int payAmount();
}
class Engineer extends Employee {
  int payAmount() {
    return salary;
  }
}
class Salesman extends Employee {
  int commission;
  int payAmount() {
    return salary + commission;
  }
}
class Manager extends Employee {
  int bonus;
  int payAmount() {
    return salary + bonus;
  }
}

调用方 emp.payAmount() 一个字没改，但"新增一个工种"从"改 N 处 switch"变成了"加一个新类，老代码一处都不用碰"。这就是开闭原则落到实处的样子。

对扩展 开放 · 对修改 关闭

加新类型 = 加新类（扩展），而不是回去改老代码里的每个 switch（修改）。多态是实现 OCP 最直接的语言级手段。

底层：switch 靠跳表，多态靠虚方法表

为什么说"分派交给语言去做"？因为多态并不是魔法，JVM 用一张虚方法表（vtable，藏在每个类的 klass 元数据里）把它实现得又快又稳。对象头里的类型指针指向它的 klass，klass 里有一张方法表；执行 invokevirtual 时，按方法在表中的固定槽位取出真正要调用的实现地址即可——这是一次常数级的指针跳转，和写多少个子类无关。

条件分发 (switch) 多态分发 (invokevirtual) ────────────────── ───────────────────────── emp ─> [对象头 | klass ptr] switch(type){ │ case 0: A(); ▼ case 1: B(); Engineer klass ─ vtable case 2: C(); [3] payAmount ─> Engineer::payAmount } Salesman klass ─ vtable ▲ 逻辑写在调用方 [3] payAmount ─> Salesman::payAmount ▲ 新增类型要改这里 ▲ 逻辑写在各自类里 ▲ 新增类型 = 新增一张表，调用方不动
性能别多虑：很多人担心"虚方法调用比 switch 慢"。实际上单态 / 双态的虚调用会被 JIT 用内联缓存（inline cache）优化到接近直接调用，热点甚至被彻底内联。可读性、可扩展性带来的收益，远大于这点微乎其微的分派开销。别拿"多态慢"当拒绝重构的借口。

第二种形态：基础逻辑 + 变体覆写

Fowler 特别指出这个手法有两种用法。上面那种是"每个类型行为完全不同"；还有一种更微妙——大部分逻辑是共通的，只有个别步骤随类型变化。这时把"通用骨架"放进父类，只让子类覆写"变化的那一步"，也就是模板方法模式。

RatingEngine.java保险评级：主干在父类，风险加成子类覆写
abstract class Rating {
    // 通用骨架：所有险种都一样，只在父类写一次
    final int value(Voyage v, History h) {
        return baseRating(v, h) - voyageRisk(v)
             + captainRisk(h);
    }
    int baseRating(Voyage v, History h) { return 70; }
    // 只有这一步随险种变化 → 交给子类覆写
    abstract int voyageRisk(Voyage v);
    int captainRisk(History h) { return 0; }
}

class ExperiencedRating extends Rating {
    @Override int voyageRisk(Voyage v) {
        return v.length() > 100 ? 2 : 0;   // 只改变化点
    }
}
陷阱：不是所有 if 都该变多态。初学者读完这一招容易"逢 if 必拆类"，结果制造出一堆只有一个方法、彼此几乎相同的子类，反而更难读。请守住两条前提：① 这套类型分支在多处重复出现（一处孤立的 switch 不值得动）；② 分支是围绕稳定的"类型/种类"展开，而不是围绕"金额区间""是否为空"这类临时条件。后者用合并条件表达式、卫语句更合适（后续章节）。

横向对比：几种"消灭 switch"的方案

方案适用取舍
保留 switch分支只有一处、且很少变最简单，无需引入类层次；一旦分支扩散就腐烂
多态（子类覆写）同一套类型分派在多处重复扩展性最好，符合 OCP；但类会变多，且需要工厂来创建对象
策略 / Map 分派行为可插拔、运行期可切换比继承更灵活（组合优于继承）；把 switch 换成一次 map.get(type)
枚举 + 抽象方法类型固定且不多Java 特有，把多态收进一个 enum 文件里，紧凑又类型安全
Java 独门写法——枚举携带行为。Java 的 enum 允许每个常量覆写抽象方法，等于把"多态子类"塞进一个枚举里，既消灭了 switch 又不用建一堆类文件。java.util.concurrent.TimeUnit 就是活教材：SECONDS.toMillis()、MINUTES.toMillis() 各常量各自实现换算，源码里根本没有 switch(unit)。

串起来看：这招在源码与框架里无处不在

  • JDK：InputStream.read() 是抽象方法，FileInputStream / ByteArrayInputStream 各自实现。你调用时从不 switch 流的种类——这正是"以多态取代条件表达式"在标准库里的默认姿态。
  • Spring：把一堆 Strategy 实现注入成 Map<String, Handler>，用 map.get(type).handle() 取代巨型 if-else，是消灭"业务分派 switch"的经典手法（多态 + 策略的组合）。
  • 设计模式：本手法是策略模式、状态模式、模板方法的共同底座——它们本质都是"用对象的多态替代显式的条件判断"。
高频面试点：多态 ≠ 消灭了所有条件判断。对象总得先被创建出来——"根据类型码 new 出对应子类"这件事，还是需要一个 switch / Map。诀窍是把这唯一的分支收敛到工厂里，只此一处，其余业务代码全部走多态。所以下一节紧接着讲以工厂函数取代构造函数——两招是天生一对。
switch 散落各处 多态分派 + 单一工厂建对象 过度拆类(空壳子类泛滥)

火候：把重复的类型分派收进多态、把"选择子类"收进一个工厂就停手；别为一处孤立的 if 硬造类层次。

一句话收束：「以多态取代条件表达式」的本质，是把散落、重复的"按类型分派"从调用方剥离，交还给对象自己——用语言内建的动态绑定（vtable / invokevirtual）替代手写的 switch，从而满足开闭原则。用它的前提是：分派逻辑在多处重复、且围绕稳定的类型展开；用它的代价是：类变多、且"选谁"这一步要收进工厂集中管理。孤立的、临时的条件，交给卫语句和合并条件更划算。
