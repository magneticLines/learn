全局数据、可变数据、发散式变化

三种坏味道，一个共同的病根——它们都在偷偷放大"改一处、崩一片"的作用半径。

一、为什么把这三个放在一起讲

《重构》第3章列了 20 多种坏味道。本节挑出的这三种——全局数据（Global Data）、可变数据（Mutable Data）、发散式变化（Divergent Change）——看似分散，其实指向同一个核心指标：一次改动的"爆炸半径"有多大。

软件里最贵的东西不是写代码，而是改代码。而改动之所以昂贵，是因为你永远无法确定"我改这里，会不会连累到那里"。这三种坏味道，正是从三个不同角度把这种"连累"放到最大：

谁能看见/改到它？ 它会不会被偷偷改？ 为什么而改？ （空间维度） （时间维度） （原因维度） │ │ │ ▼ ▼ ▼ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ 全局数据 │ │ 可变数据 │ │ 发散式变化 │ │ 作用域 = 全世界│ │ 状态随时被写 │ │ 一个模块因多 │ │ 谁都能改 │ │ 难以推理 │ │ 种原因而改动 │ └──────────────┘ └──────────────┘ └──────────────┘ └────────────┬───────────┴───────────┬───────────┘ ▼ ▼ 共同后果：改一处 → 无法预测会崩哪里（爆炸半径失控）
一句话抓住本质： 好代码的秘诀是把"变化"关进小笼子——让每个改动只影响一小块、可预测的范围。这三种坏味道，恰恰是三种不同形状的"笼子破洞"。理解它们，本质上是在训练一种直觉：时刻警惕"这个改动的影响范围到底有多大"。

二、全局数据（Global Data）：谁都能改的"公共黑板"

Fowler 说全局数据是他心中"最刺鼻的坏味道"。典型形态：public static 变量、单例里的可写字段、跨模块共享的配置对象。它的问题不在"全局"这个作用域本身，而在于——你根本无法知道是"谁"在"什么时候"改了它。

打个生活类比：全局变量就像办公室里一块谁都能写、谁都不署名的公共白板。今天你在上面记了个电话号码，明天发现被人擦掉改成了别的——你连问谁都不知道从何问起。代码里的全局数据就是这样：出了 bug，嫌疑人是整个代码库。

陷阱：把"单例 + 可变字段"当成方便的全局收纳箱。 初学者最爱写一个 GlobalConfig.getInstance().setX(...) 到处塞状态。这在单线程 demo 里很爽，一旦上了多线程（比如 Spring 的单例 Bean 被多个请求线程共享），可变的全局字段就成了数据竞争的重灾区——这正好把下一节的"可变数据"叠加进来，两种坏味道复利式爆炸。
Bad · 全局可写状态
public class Inventory {
  // 全世界都能读写的库存
  public static int stock = 100;
}

// A 模块：下单
Inventory.stock -= qty;

// B 模块：某处"顺手"也改了它
Inventory.stock = reload();

// 半年后线上库存对不上：
// 到底谁改的？全代码库都是嫌疑人
// 无法加断点、无法收窄排查范围
Good · 封装 + 收窄入口
public class Inventory {
  private int stock = 100;   // 藏起来

  // 所有变更必须走这一个门
  public synchronized void deduct(int q) {
    if (q > stock)
      throw new IllegalStateException();
    stock -= q;   // 想加日志/断点/校验
  }                // 只需守住这一处
  public int current() { return stock; }
}
// 修改入口唯一 → 排查范围 = 1 个方法

上面的重构手法叫 封装变量（Encapsulate Variable）：把裸露的数据藏到函数背后，逼所有访问都走"取值/设值"的窄门。这样做的收益是决定性的——任何时候你想给这个数据加日志、加校验、加锁、加断点，都只有一个地方要动。这就把"全代码库都是嫌疑人"收窄成"只看这一个方法"。

串联 JDK： Java 生态早就用制度对抗全局可变状态。System.out 虽是全局的，但被设成 final；想替换得走 System.setOut(...) 这个受控入口（还要过 SecurityManager）。再看 Collections.emptyList()、Optional.empty()——这些"全局共享"的对象之所以能安全共享，前提是它们不可变。可见对付全局数据的终极答案，往往是让它变成下一节的"不可变数据"。

三、可变数据（Mutable Data）：状态是 bug 最爱藏身的地方

可变数据的坏味道是：一个变量在生命周期里被反复赋值/修改，导致你读代码时无法只看一处就断定它此刻的值——你得在脑子里"模拟执行"整段代码，追踪它一路被改了几次。人脑的这种"状态追踪"能力极其有限，这正是大量 bug 的温床。

为什么函数式编程能减少 bug？ 答案就藏在这里。函数式的核心信条是不可变（immutability）——数据一旦创建就永不改变，需要"变"就返回一个新副本。这样每个值都是"定值"，你看到 x 在哪定义就知道它永远是那个值，无需追踪。Fowler 本人后期越来越推崇这种风格，《重构》第2版大量重构手法（如"将查询与修改分离""移除设值函数")都在往"减少可变性"的方向走。

3.1 共享可变状态 + 并发 = 灾难

可变数据在单线程里已经够烦，一旦叠加并发，就直接踩中 Java 内存模型（JMM）的雷区。看这个经典反例——它同时违反了原子性和可见性：

共享可变字段的并发陷阱Java · JMM
class Counter {
    private int count = 0;          // 共享的可变状态

    public void increment() {
        count++;                    // 看似一行，实为 读→改→写 三步
    }                              // 多线程下会丢失更新（非原子）
}

// 两个线程各 +10000 次，结果几乎永远 < 20000
// 因为 t1 读到 5，t2 也读到 5，各自 +1 写回 6 —— 少加了一次

修复它有两条路，正好体现"处理可变数据"的两种哲学：

思路A · 保留可变，加锁保护
// 用同步/原子类"看守"可变状态
class Counter {
  private final AtomicInteger
      count = new AtomicInteger();
  public void increment() {
    count.incrementAndGet(); // CAS 原子
  }
}
// 有效，但状态仍可变：
// 一旦别处忘了走同步入口就出错
// 心智负担仍在（要记得处处加锁）
思路B · 干脆消灭可变状态
// 不可变对象：改 = 造新对象
record Counter(int count) {
  Counter increment() {
    return new Counter(count + 1);
  }
}
// 天然线程安全：没有共享可写状态
// 就没有数据竞争。可自由发布、缓存、
// 当 Map 的 key —— 这正是 String、
// Integer 不可变的根本原因
高频面试点：为什么 String 设计成不可变？ 把答案往可变数据这条线上收：① 线程安全——不可变对象天生可被多线程共享，无需加锁；② 可安全做 HashMap 的 key——若 String 可变，改了内容后 hashCode 变了，就再也找不回原来那个桶；③ 字符串常量池可以放心复用——多个引用指向同一个字面量对象也不怕被谁改坏。Integer、LocalDate、BigDecimal 全是同一套设计哲学。

3.2 更隐蔽的坑：可变数据"逃逸"出封装

就算你把字段设成 private，如果 getter 直接把内部的可变集合"原样递出去"，封装就形同虚设——外部拿到引用后照样能改你的内部状态。这叫引用逃逸（reference escape）：

Bad · getter 泄漏内部引用
class Team {
  private List<String> members
      = new ArrayList<>();
  public List<String> getMembers() {
    return members;   // 把内部引用交出去
  }
}
// 外部一行就击穿了封装：
team.getMembers().clear();  // 内部被清空
// private 形同虚设
Good · 返回不可变视图/副本
class Team {
  private final List<String> members
      = new ArrayList<>();
  public List<String> getMembers() {
    return Collections
        .unmodifiableList(members);
  }                    // 只读视图，改就抛异常
  public void add(String m) {
    members.add(m);    // 变更只能走这个门
  }
}

四、发散式变化（Divergent Change）：一个模块被多种理由反复撕扯

前两种坏味道关注"数据"，这一种关注"变化的方向"。发散式变化的定义是：每当因为某种原因要改动系统时，你总要改动同一个模块里的很多不同地方。

Fowler 给的判断句极其好用——如果你能对着一个类说出这样的话，它就得了发散式变化：

"每次接入新支付渠道，我要改这个类；
每次换数据库，我又要改这个类。"

这里的病根是一个模块承担了多个"变化的原因"——它既管支付、又管持久化、又管发通知。这三件事本该因为三种完全不同的外部原因而变化，却被塞进同一个类里，于是这个类变成了"每个方向的风都要吹到它身上"的受气包。

它就是"单一职责原则"（SRP）的反面。 SRP 常被误解成"一个类只做一件事"，Robert C. Martin 的精确表述是："一个模块应该有且只有一个引起它变化的原因（reason to change）。" 发散式变化，正是"一个模块有多个变化原因"的症状。所以 SRP 不是按"功能"切，而是按"谁会要求你改它"（变化的来源/责任方）来切。
反例：一个类，三个变化方向Java
class OrderService {
    void checkout(Order o) {
        // —— 变化原因①：支付渠道升级时要改这里 ——
        if (o.payType() == ALIPAY) { /* 调支付宝 SDK */ }
        else if (o.payType() == WECHAT) { /* 调微信 SDK */ }

        // —— 变化原因②：换 DB / 改表结构时要改这里 ——
        String sql = "INSERT INTO orders ...";
        jdbc.execute(sql);

        // —— 变化原因③：改通知文案/渠道时要改这里 ——
        emailClient.send(o.userEmail(), "下单成功");
    }
}  // 三种毫不相干的外部原因，全砸在同一个方法/类上

重构方向：按"变化的原因"把它拆分成互不相干的模块（手法：提炼类 / 搬移函数）。支付归 PaymentGateway，持久化归 OrderRepository，通知归 Notifier。之后"换支付渠道"只碰支付模块，"换数据库"只碰仓储模块，井水不犯河水：

重构前（发散式变化） 重构后（各自独立） ┌───────────────────┐ ┌──────────────────┐ │ OrderService │ │ OrderService │ ← 只做编排 │ ├─ 支付逻辑 ⟵① │ ──▶ │ ├─ PaymentGateway ⟵① │ ├─ 持久化 ⟵② │ │ ├─ OrderRepository⟵② │ └─ 发通知 ⟵③ │ │ └─ Notifier ⟵③ └───────────────────┘ └──────────────────┘ ①②③ 三种原因都改它 改 ① 只碰支付模块，互不牵连
串联 Spring： 你天天用的 Controller → Service → Repository 分层，本质就是为了对抗发散式变化——把"HTTP 协议变化""业务规则变化""持久化变化"三个方向的改动，分别隔离到三层里。Spring 的 @Repository/@Service 分层、PaymentGateway 接口 + 多实现（策略模式），都是把"变化原因"拆开、让每个方向各有归宿的工程化手段。

五、发散式变化 vs 霰弹式修改：一枚硬币的两面

下一节要讲的霰弹式修改（Shotgun Surgery）常和发散式变化搞混，其实它俩正好相反，是"内聚度失衡"的两个方向。记住这张对比表，面试和实战都用得上：

维度发散式变化 Divergent Change霰弹式修改 Shotgun Surgery
症状一个模块，因多种原因反复被改一种改动，要动多个模块
数量关系1 个类 ← 多个变化原因1 个变化原因 → 散落多处
比喻一个受气包，八方来风都打它改一个需求，满地捡碎片
失衡方向职责过度集中（内聚太杂）职责过度分散（该聚没聚）
解药拆分：提炼类 / 搬移函数聚合：把散落的逻辑搬到一处
陷阱：过度拆分会从一个坑跳进另一个坑。 治发散式变化靠"拆"，但拆得太碎，一个改动就会散落到十几个小类里——反而变成了霰弹式修改。二者是天平的两端，目标不是无脑拆或无脑合，而是让"因同一原因而变的东西聚在一起，因不同原因而变的东西分开"（这正是"高内聚、低耦合"的另一种表述）。

六、三者的内在关联：都是在控制"变化的作用半径"

回到开头。这三种坏味道其实是从三个坐标轴，度量同一件事——一个改动的影响范围（爆炸半径）：

全局数据 · 空间半径 可变数据 · 时间半径 发散式变化 · 原因半径
  • 全局数据放大了空间半径：数据可见/可改的范围 = 整个程序。
  • 可变数据放大了时间半径：一个值在时间线上被改了多少次，你就要追踪多少次。
  • 发散式变化放大了原因半径：一个模块因多少种原因而改，它就多脆弱。

对付它们的手法虽各不相同（封装变量 / 走向不可变 / 提炼类），但内在目标高度一致：缩小每一次改动能波及的范围，让"改这里"和"崩那里"之间的因果链变短、变可预测。

一句话收束： 全局数据、可变数据、发散式变化，是从空间、时间、原因三个维度失控的"变化爆炸半径"。全局数据要封装（收窄谁能改），可变数据要尽量不可变（收窄何时改），发散式变化要按变化原因拆分（收窄为何改）。 三者共同的解药，都是同一句古老箴言的具体化——高内聚、低耦合，把变化关进小笼子。
