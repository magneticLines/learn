重复代码与夸夸其谈通用性

一个是"写得太少"（同一份知识抄了三遍），一个是"写得太多"（为幻想中的需求提前搭好脚手架）——这两种相反的坏味道，其实是同一枚硬币的两面。

一枚硬币的两面：写太少 vs 写太多

把这两个坏味道放在一起讲，是因为它们恰好站在"抽象光谱"的两个极端。重复代码（Duplicated Code）是抽象不足——本该只写一次的知识，被复制粘贴散落到多处；夸夸其谈通用性（Speculative Generality）是抽象过度——为了"以后也许用得上"，提前造了一堆没人真正需要的接口、参数和继承层级。它们分别由软件工程里两条最著名的原则来治：DRY 与 YAGNI。

写得太少 <──────────────── 恰到好处 ────────────────> 写得太多 重复代码 单一职责 / 单一知识 夸夸其谈通用性 (Duplicated Code) (Single Source of Truth) (Speculative Generality) │ │ │ DRY 治它 "此刻真实的需求" YAGNI 治它 │ │ 改一处 → 漏改另几处 正好够用，不多不少 为幽灵需求预付"复杂度利息"
为什么值得对照着学？新手最常见的成长路径是：先犯"重复代码"（懒得抽象，Ctrl+C / Ctrl+V），被坑几次后矫枉过正，转而犯"夸夸其谈通用性"（凡事先抽个接口、先留个扩展点）。真正的功力，是知道在哪一刻抽象、抽象到什么粒度——既不早也不晚。这一课就是帮你校准这个"火候"。

重复代码：坏味道之首

Martin Fowler 在书里说得毫不含糊："如果你在一个以上的地点看到相同的代码结构，那么可以肯定：设法将它们合而为一，程序会变得更好。"重复代码之所以是头号坏味道，不是因为它"多打了几个字"，而是因为它埋下了一颗定时炸弹：当这段逻辑要改时，你必须找到每一个副本、逐一修改，而且不能漏。

维护成本 ∝ 重复副本的数量

漏改一处，就是一个 bug。更阴险的是：随着时间推移，几个副本会各自"漂移"——有人改了 A 处忘了 B 处，两份代码看起来一样却行为不同，后来者根本分不清哪个是对的。这就是重复代码真正的代价：它把"一处知识"变成了"多处需要同步维护的真相"。

DRY 原则的精确表述（来自《程序员修炼之道》）："系统中的每一处知识，都必须有单一、无歧义、权威的表示（Every piece of knowledge must have a single, unambiguous, authoritative representation within a system）。"注意关键词是"知识"而不是"代码字符"——DRY 反对的是知识的重复，不是长得像的字符串重复。这个区别下面会反复用到。

反例 → 正例：用提炼函数消灭重复

Bad · 同一段逻辑抄两遍
if (isVip) {
  double base = qty * price;
  double t = base * 1.13;   // 含税
  audit(t);
  total = t * 0.9;          // VIP 9 折
} else {
  double base = qty * price;
  double t = base * 1.13;   // 又抄一遍
  audit(t);
  total = t;
}
// 税率要改？两处都得改，漏一处就出错
Good · 知识只存在一处
double t = taxedAmount(qty, price);
total = isVip ? t * 0.9 : t;

// 权威定义：只有这里知道税率
double taxedAmount(int qty,
                    double price) {
  double t = qty * price * 1.13;
  audit(t);
  return t;
}
// 改税率只改一行，天然不会漏

重复的三种形态，对应三种手法

Fowler 把重复代码按"出现的位置"分成三类，每一类有它最顺手的重构手法：

重复出现在哪典型手法要点
同一个函数/类内，两段代码几乎一样Extract Function（提炼函数）把公共段切成一个命名良好的方法，两处都调它
两段代码相似但不完全相同，只是次序不一致Slide Statements（移动语句）+ 提炼先把相同的语句挪到一起，让重复"对齐"，再提炼
兄弟子类里各写了一份一样的方法Pull Up Method（函数上移）把它提到共同父类；若只是骨架相同、细节不同，用 Form Template Method
不相关的类里出现同一段逻辑Extract Class（提炼类）抽出一个新类承载这份知识，双方共用

兄弟子类里的重复尤其常见。下面用 Pull Up Method 把两个子类里一模一样的方法提到父类——这也是 Spring 的 JdbcDaoSupport 等基类抽公共行为的思路：

PullUpMethod.java兄弟子类重复 → 上移到父类
// Bad：两个子类各写一份完全相同的 fullName()
class Employee extends Party {
    String fullName() { return firstName + " " + lastName; } // 重复
}
class Customer extends Party {
    String fullName() { return firstName + " " + lastName; } // 又一份，会各自漂移
}

// Good：提到共同父类 Party，子类共享唯一权威实现
abstract class Party {
    protected String firstName, lastName;
    String fullName() {                    // 只此一处，改一次全生效
        return firstName + " " + lastName;
    }
}
class Employee extends Party { /* 不再重写 fullName */ }
class Customer extends Party { /* 不再重写 fullName */ }
Rule of Three（三次法则）——什么时候动手抽象？Fowler 引用 Don Roberts 的经验法则："第一次做某件事只管去做；第二次做类似的事，你会对重复皱皱眉，但还是照抄；到第三次再出现，就该重构了。"为什么不是第二次就抽？因为只有两个样本时，你还看不清"什么是共性、什么是偶然的巧合"，过早抽象很可能抽错。等第三次，共性的轮廓才清晰。
1 次 → 写它    2 次 → 忍它    3 次 → 重构它
陷阱：别被"长得像"骗了，那可能不是真重复。两段代码字面一样，不代表它们表达同一份知识。经典例子：订单里的"数量下限校验 qty >= 1"和用户表单里的"年龄下限校验 age >= 1"，代码长得一模一样，但它们是两条会独立演化的业务规则——今天你把它们合并成一个 checkMin()，明天年龄下限改成 18、订单下限还是 1，你就得被迫拆开。DRY 是"知识不重复"，不是"代码不重复"。合并前先问：这两处将来会不会因为同一个原因一起改？

夸夸其谈通用性：为不存在的需求买单

这个略带调侃的名字（Speculative Generality，由 Brian Foote 命名）指的是：你为了"以后也许会有这种需求"而写下的一切——用不上的抽象类、只有一个实现的接口、永远走同一条分支的钩子、没人传的参数、为了"通用"而绕的弯路。Fowler 的判据一针见血："如果所有装置都被用到，就值得；如果用不到，就不值得。用不上的装置只会挡你的路，把它们移走。"

它的克星是 YAGNI。YAGNI = "You Aren't Gonna Need It"（你不会需要它），是极限编程（XP）的核心信条之一：不要为"预测中的未来需求"编写代码，只为"此刻真实存在的需求"编写。因为对未来的预测大概率是错的——你精心预留的扩展点，未来的真实需求往往从另一个方向袭来，你留的口子一个都用不上，反而每个读代码的人都要为这些"以防万一"的复杂度付出理解成本。

反例 → 正例：一个实现的接口就是过度设计

Bad · "以后可能有别的策略"
// 三层结构，只为了一个默认实现
interface DiscountStrategy {
  double apply(double amt);
}
class DefaultDiscount
    implements DiscountStrategy {
  public double apply(double a){
    return a;   // 啥也没打折
  }
}
class Checkout {
  DiscountStrategy s =
    new DefaultDiscount();
  double total(double a){
    return s.apply(a);
  }
}
Good · 需要时再抽象
class Checkout {
  double total(double amt){
    return amt;
  }
}

夸夸其谈通用性的常见"病灶"与解药

病灶（你会看到）解药手法
抽象类只有一个子类；继承层级毫无必要Collapse Hierarchy（折叠继承体系）
函数/类只是一层没意义的转发（Middle Man）Inline Function / Inline Class（内联）
参数是"为将来预留的"，目前所有调用方都传同一个值或压根没用Change Function Declaration（移除参数）
方法名里带 foo/doStuff 这类"通用占位"色彩，或用泛型硬撑其实只处理一种类型Rename + 具体化，去掉不必要的类型参数
整段无人调用的"以后可能用"的代码Remove Dead Code（删除死代码）
面试高频 & 实战大坑：过度设计比重复更难被发现。重复代码肉眼可见（复制粘贴的痕迹很明显），而夸夸其谈通用性常常披着"设计良好"的外衣——满屏接口、工厂、策略、泛型，看起来很"专业"，实则大多是自嗨。更麻烦的是它有很强的心理惯性："这是我精心设计的扩展点，删掉多可惜"。记住：没被用到的灵活性，不是资产，是负债。删掉它，需要时 Git 历史还能找回来。

联系已学：框架源码里的正反两面

正面（消灭重复）：Spring 的 JdbcTemplate。裸写 JDBC 时，每个查询都要重复"获取连接 → 创建 Statement → 执行 → 遍历 ResultSet → 在 finally 里逐个 close"这一大坨样板，而且极易漏关导致连接泄漏。JdbcTemplate 用模板方法 + 回调把这段不变的骨架收敛到一处，你只需通过 RowMapper 提供"怎么把一行映射成对象"这个变化点。这正是 Form Template Method 手法在真实框架里的大规模落地——把成千上万处重复的资源管理，收拢成"单一权威表示"。
反面（夸夸其谈）：被滥用的 AbstractSingletonProxyFactoryBean 之类。企业级 Java 曾以又长又"通用"的抽象类名著称（"AbstractXXXFactoryBean"梗由此而来）。很多这类抽象层是为"万一将来要换实现"预留的，结果绝大多数项目从头到尾只用一种实现。对照 JDK 的正向演进：早年 java.util.Date 想用一个类"通用地"表达时刻、日期、时区，结果什么都做不好；Java 8 的 java.time 反其道而行，拆成 LocalDate / LocalTime / Instant 等一组各司其职的具体类——用恰当的具体取代了含糊的通用。

DRY 与 YAGNI 的张力：错误的抽象比重复更贵

初学者会问：既然重复要消除、过度设计也要避免，那到底听谁的？答案是——它们并不矛盾，而是互为刹车。DRY 拦住你"抄代码"，YAGNI 拦住你"过早抽象把它们强行合并"。把两者结合，就得到 Sandi Metz 那句流传甚广的箴言：

"重复，远比错误的抽象便宜（duplication is far cheaper than the wrong abstraction）。" —— Sandi Metz
当你还看不清两段相似代码是否表达同一份知识时，宁可暂时容忍重复，也不要急着抽出一个错误的抽象。因为重复只是"多处要一起改"，而错误的抽象会用一堆参数和分支把不该耦合的东西焊死，未来每个人都得绕着它走，拆解它的代价远高于当初消除重复。
维度重复代码夸夸其谈通用性
本质同一份知识散落多处（抽象不足）为不存在的需求预留结构（抽象过度）
违反原则DRYYAGNI / KISS
症状改一处要同步改多处，容易漏接口/参数/继承层级没人用、读代码要绕路
代价维护成本随副本数线性上升持续的认知负担 + 维护死代码
解药提炼函数 / 上移方法 / 提炼类折叠继承 / 内联 / 移除参数 / 删除死代码
动手时机第三次出现（Rule of Three）发现"这份灵活性从没被真正用过"
抽象不足 = 重复 刚好够用 抽象过度 = 夸夸其谈

好设计不在光谱的某个固定点，而是随需求演化不断微调——今天"刚好够用"的抽象，明天需求变了可能就重复了或过度了。重构就是让代码持续贴着"真实需求"这条中线滑动的能力。

一句话收束：重复代码是抽象不足，病根在"同一份知识没有单一权威表示"，解药是 DRY + 提炼，动手时机看 Rule of Three；夸夸其谈通用性是抽象过度，病根在"为幻想的未来预付复杂度利息"，解药是 YAGNI + 内联/删除。二者共同的智慧只有一句：让抽象由"已经发生的真实需求"来驱动，而不是由"对未来的猜测"来驱动——看不清时，宁可容忍重复，也别造出一个错误的抽象。
