隐藏委托关系与移除中间人

一对互为镜像的手法——封装该藏就藏、该露就露，答案不在教条里，而在天平上。

先看一对镜像手法

这一节的两个手法是互逆的：Hide Delegate（隐藏委托关系）往"多封装"的方向走，Remove Middle Man（移除中间人）往"少封装"的方向退。它们共同回答的是同一个设计问题：一个类到底该替它的合作者"挡"多少细节？

底层原理：封装（Encapsulation）与信息隐藏。好的模块化设计的第一要义，是让模块尽量少暴露自己的内部构造——这正是 David Parnas 提出的"信息隐藏"原则：把最可能变化的设计决策藏进模块内部。当客户端 A 要通过 B 去够到 C（a.getB().getC()），A 就被迫知道了"B 内部有个 C"这件本该保密的事。隐藏委托，就是把这条"通往 C 的路"封死在 B 内部。

隐藏委托关系（Hide Delegate）

Fowler 的经典例子：一个 Person 属于某个 Department，部门有一位经理。客户端想拿到某人的经理，写成了：

manager = person.getDepartment().getManager();

问题在于：客户端知道得太多了。它不仅依赖 Person，还依赖"Person 里有个 Department"、"Department 里有个 manager"这两条内部结构。哪天经理不再挂在部门上、而是直接挂在人身上，所有这么写的客户端都得改。下图是两种拿法的对比：

【隐藏前】客户端亲自"走"对象图，知道全程路径 Client ──▶ Person.getDepartment() ──▶ Department.getManager() ──▶ Manager client 依赖：Person + Department 的内部结构（2 层知识） 【隐藏后】客户端只按一次门铃，路怎么走是 Person 家的事 Client ──▶ Person.getManager() ┄内部委托┄▶ department.getManager() client 依赖：只有 Person（1 层知识，Department 被藏起来）
Bad · 委托关系暴露（火车残骸）
class Person {
  private Department department;
  // 把内部结构直接漏出去
  public Department getDepartment() {
    return department;
  }
}
class Department {
  private Person manager;
  public Person getManager() {
    return manager;
  }
}

// 客户端：一长串点号 = 火车残骸
Person mgr =
    john.getDepartment().getManager();
Good · 委托被封在 Person 内
class Person {
  private Department department;

  // 只暴露"结果",不暴露"路径"
  public Person getManager() {
    return department.getManager();
  }
  // getDepartment() 若无人再用即可删
}

// 客户端：只认识 Person
Person mgr = john.getManager();
// Department 的存在被彻底隐藏
它是迪米特法则（Law of Demeter）的标准解法。迪米特法则又叫"最少知识原则"：一个方法只应该跟直接朋友说话——自己、自己的字段、方法的参数、自己创建的对象——而不该跟"朋友的朋友"说话。a.getB().getC() 就是在跟陌生人 C 搭话（俗称"火车残骸 / train wreck"）。Hide Delegate 把这条链子的后半截塞进 Person 内部，客户端就只和"直接朋友"打交道了。

小步做法（Mechanics）

重构讲究小步、可回退。隐藏委托的标准步骤是：

  1. 在服务对象（Person）上，为客户端用到的每个委托方法建一个转发方法（getManager()）。
  2. 把客户端逐个改成调用服务对象的新方法，改一个测一次。
  3. 当再没有客户端访问委托对象（Department）时，删掉服务对象上暴露委托的访问器（getDepartment()）。
  4. 全绿收工。

移除中间人（Remove Middle Man）

但故事会反转。随着需求增长，客户端要用的 Department 功能越来越多——经理、成本中心编码、人头数、地址……于是 Person 上转发方法越堆越多，最后变成一个几乎什么都不干、只会转手的"中间人"。这就是第 3 章列过的坏味道中间人（Middle Man）。此时该反向操作：把委托对象重新暴露出来，让客户端直接对话。

MiddleMan.javaRemove Middle Man
// 重构前：Person 沦为纯转发的"中间人"
class Person {
    private Department department;
    public Person   getManager()    { return department.getManager(); }
    public String   getChargeCode() { return department.getChargeCode(); }
    public int      getHeadCount()  { return department.getHeadCount(); }
    public String   getLocation()   { return department.getLocation(); }
    // ……全是一行转发，Person 自己没有任何价值
}

// 重构后：把委托对象暴露出来,让客户端直接用
class Person {
    private Department department;
    public Department getDepartment() { return department; }
}
// 客户端改为:
Person mgr = john.getDepartment().getManager();
陷阱一：把"有价值的委托"当成中间人删掉。中间人坏味道的判据不是"转发得多"，而是转发之外几乎不做任何事。装饰器（Decorator）、代理（Proxy）、适配器（Adapter）也大量转发，但它们在转发时加了料——加缓冲、加同步、加权限、转协议。判断口诀："拿掉这层，会不会丢东西？" 丢东西的是合法委托，不丢东西的才是该删的中间人。

天平的两端：到底藏还是露？

这才是本节真正的灵魂——没有标准答案。Fowler 说得很直白："你不必第一次就做对；重构的意义就在于你可以随时改主意。"两个手法是同一个天平的两端，随着"客户端用到委托的多寡"来回滑动：

← 用得少：Hide Delegate（多封装，降耦合） 用得多：Remove Middle Man（少转发，减样板）→
维度Hide Delegate 隐藏委托Remove Middle Man 移除中间人
方向多封装（往里藏）少封装（往外露）
客户端认识谁只认识服务对象直接认识委托对象
耦合客户端与委托解耦客户端与委托直连
代价服务对象堆积转发方法暴露内部结构、违反迪米特
适用信号委托只被用到一两处转发方法多到服务对象没别的事干
迪米特法则遵守放宽

那"转发方法多到什么程度"算中间人？没有硬阈值，但有个常被引用的经验比值可帮你嗅味道：

中间人味道 ≈ 委托方法数 / 该类方法总数 > 50%
陷阱二：迪米特法则会"反噬"成中间人。如果把迪米特法则当成不可违逆的"法律"去死守，你会为每一层委托都写一堆小小的包装方法，最终亲手制造出一堆中间人——这正是 Remove Middle Man 要解决的问题。所以 Fowler 干脆调侃它应该叫"偶尔有用的迪米特建议（occasionally useful suggestion of Demeter）"，而不是"法则"。封装是手段不是目的，别为封装而封装。

连接 Java 生态：处处都是这对手法

把视野从两个类放大到子系统，你会发现整个 Java 世界都在这条天平上做取舍：

例子它在做什么是味道还是设计？
门面模式 Facade用一个简单接口挡住复杂子系统——本质就是"子系统级的 Hide Delegate"设计：加了"简化"的价值
Spring JdbcTemplate / RestTemplate把 JDBC/HTTP 的繁琐样板隐藏在模板方法后设计：加了资源管理、异常转译
Spring AOP 代理代理转发调用给目标 Bean，顺手织入事务/日志设计：加了横切关注点
Collections.synchronizedList转发所有方法给底层 List，但套上 synchronized设计：加了线程安全
纯转发的 XxxService / XxxManager方法体只有一行 return repo.xxx()，别无他物味道：Remove Middle Man
和 java.io 的装饰器串一下。new BufferedReader(new InputStreamReader(in)) 是层层委托——每层 read() 最终都转发给最内层的 in。它看起来"中间人一大堆"，但绝不是坏味道：InputStreamReader 转发时做了字节→字符的解码，BufferedReader 转发时加了缓冲。转发 + 增值 = 装饰器；只转发不增值 = 中间人。这也呼应了上一节"提炼类/内联类"：中间人本质是一个"没被内联掉的空壳类"。
一句话收束：隐藏委托与移除中间人是同一个封装天平的两端——前者把 a.getB().getC() 收进服务对象内部以降低耦合、遵守迪米特法则；后者在转发方法泛滥、服务对象沦为空壳时反向拆掉封装。关键不是选对哪一个，而是理解"该藏多少"会随系统演化而变——重构让你可以随时改主意。判断中间人的唯一尺子是"拿掉这层会不会丢价值"：门面、代理、装饰器转发时都增了值，是设计；只会转手的空壳才是要移除的坏味道。
