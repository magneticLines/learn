神秘命名：名字看不出意图

代码是写一次、读一万次的东西——一个看不懂的名字，会被后来人反复解码一万次。

一句话看懂：什么是"神秘命名"

神秘命名（Mysterious Name）是 Fowler 在第 3 章罗列的第一个坏味道——不是巧合，而是刻意。它排在"过长函数""重复代码"这些更"显眼"的坏味道前面，因为它出现频率最高、修复成本最低、收益却最大。所谓神秘命名，就是一个名字（变量、函数、类、参数）无法凭字面告诉你它是什么、做什么、返回什么，逼着读代码的人去翻实现、翻上下文、甚至翻数据库表结构才能猜出意图。

Fowler 的原话："整洁代码最重要的一环就是好的名字，所以我们会深思熟虑如何给函数、模块、变量和类命名，使它们能清晰地表明自己的功能和用法。" —— 命名不是"顺手起个名"，而是一次设计决策。你被迫想清楚"这东西到底是什么"，才可能给出好名字。取不出好名字，往往说明你还没想清楚职责。

业界有句被引用到烂的名言，最能说明它为什么难：

Phil Karlton："计算机科学中只有两件难事：缓存失效和命名。"（There are only two hard things in Computer Science: cache invalidation and naming things.）—— 命名难，是因为它要求你用一两个词，精确压缩一段逻辑的全部意图。

为什么它是"最该先修"的坏味道：读 >> 写

要理解命名的价值，得先接受一个反直觉的事实：代码被阅读的次数，远远超过被编写的次数。你写一个方法花 5 分钟，但它会在未来两年里被你、被同事、被值班救火的人读上几百上千次。命名成本是"一次投入，长期分红"。

维护成本 ≈ 命名质量 × 阅读次数(N)  ,  N → ∞

一个坏名字的伤害不是一次性的，而是乘以 N。每一次阅读，大脑都要重新"解码"一次这个神秘符号——这就是所谓的认知负荷（cognitive load）：

读到坏名字： list1 ──► 大脑：「list1 是啥?」──► 翻上下文/翻实现 ──► 「哦，是被筛出来的格子」──► 继续 ↑ │ └────────── 下次再读到 list1，重来一遍 ◄──────────┘ 读到好名字： flaggedCells ──► 大脑：直接懂 ──► 继续读 结论：坏名字把「解码成本」摊到每一次阅读里，好名字把它一次性消灭在命名时。
联系操作系统 / CPU 缓存的类比：好名字就像给数据加了一层"语义缓存"——读代码的人第一次理解后，名字本身就是缓存命中（cache hit），无需再去"内存"（实现细节）里取。坏名字则是每次都 cache miss，每次都要走一遍慢速的"翻源码"通路。命名，本质上是在优化人脑的读取延迟。

神秘命名的五宗罪

神秘不是单一形态，它有几张常见的脸。识别它们，是"闻到坏味道"的第一步：

症状反面例子为什么坏改进方向
无意义名tmp data obj flag list1等于没起名，用途全靠猜用领域词汇：flaggedCells
缩写 / 黑话genymdhms calcTx2需要脑内解压缩拼全，或只用公认缩写
名不副实（撒谎）getUser() 里却发了邮件制造"惊讶"，最危险名字 = 真实行为
类型冗余（匈牙利）strName iCount lstUser类型一改名字就骗人去掉类型前缀
万能筐命名Manager Processor Info Util什么都能装=职责不清按真实职责命名

反例 → 正例：一段"神秘"代码的现形记

这是 Robert Martin 在《Clean Code》里的经典例子，也完美对应 Fowler 的神秘命名。同一段逻辑，只改名字（行为完全不变，这正是重构的定义），可读性天差地别：

Bad · 神秘命名
List<int[]> getThem() {
  List<int[]> list1 = new ArrayList<>();
  for (int[] x : theList)
    if (x[0] == 4)
      list1.add(x);
  return list1;
}
// 疑问：theList 是啥?
// x[0] 是啥? 4 又代表什么?
// 返回的 list1 里装的到底是什么?
Good · 意图清晰
List<Cell> getFlaggedCells() {
  List<Cell> flaggedCells = new ArrayList<>();
  for (Cell cell : gameBoard)
    if (cell.isFlagged())
      flaggedCells.add(cell);
  return flaggedCells;
}
// 一眼看懂：从棋盘里
// 挑出所有「被标记」的格子
// 连注释都不需要了
注意：右边一行逻辑都没删。循环、判断、结构完全一样，只是把 theList→gameBoard、x[0]==4→cell.isFlagged()、list1→flaggedCells。这就是"改名"作为重构手法的威力——零行为改变，纯收益。魔法数字 4 被封装进 isFlagged()，顺带消灭了另一个坏味道。

对应的重构手法：三个"改名"

Fowler 给每个坏味道都配了对应的重构手法（"药方"）。神秘命名的药方朴素得可爱，就是"改名"：

  • 改变函数声明（Change Function Declaration）——给函数改名、调整参数，让签名自解释
  • 变量改名（Rename Variable）
  • 字段改名（Rename Field）

它们看似简单，却是最考验"重构节奏"的手法：改名会牵动所有调用点。因此务必依赖 IDE 的自动重构（Rename / Refactor） 和测试，而不是全局文本替换——后者会误伤字符串、注释、无关的同名符号。

Rename.java · 改名要顺着"意图"走Java
// Bad：名字在描述「实现/类型」，而不是「意图」
boolean flag;                 // flag 什么? 谁知道
int d;                        // d = day? distance? delta?
void handle(Order o) { ... }  // handle 是个万能词，等于没说

// Good：名字描述「它是什么 / 做什么」
boolean isPaymentOverdue;       // boolean 用 is/has/can 前缀（JavaBeans 规范）
int elapsedDaysSinceOrder;     // 带单位、带语境
void refundExpiredOrder(Order order) { ... }

建立联系：好名字长什么样，看 JDK 和 Spring

你每天用的类库，就是命名的活教材。它们之所以"不用看文档也能猜对用法"，靠的正是命名纪律：

来源好命名它凭什么好
JDK 集合list.add() map.getOrDefault() Optional.orElseGet()动词精确描述行为，一读就懂副作用
JDK 并发AtomicInteger.getAndIncrement()连"先取值后自增"的时序都写进了名字
Spring DatafindByEmailAndStatus(...)方法名 就是 查询契约，框架据此生成 SQL
Spring 分层UserService UserRepository UserController后缀即职责，团队心照不宣
陷阱：连 JDK 也有"神秘命名"的历史包袱。java.util.Date 的 getYear() 返回的是"年份 - 1900"，getMonth() 从 0 开始计月——名字说 year/month，行为却名不副实，坑了整整一代人，最终被 java.time（LocalDate.getYear()）彻底重做。教训：一旦名字撒谎并被公开，代价是几乎无法回头的兼容性债务。

常见陷阱与面试高频点

1. 匈牙利命名法（strName/iAge）是反模式。在强类型语言 + 现代 IDE 里，类型信息鼠标一悬停就有；把类型编进名字，一旦重构改了类型，名字立刻变成谎言。
2. 别用注释去补救坏名字。int d; // elapsed time in days —— 既然注释都写清楚了，为什么不直接叫 elapsedTimeInDays？好名字让注释变得多余。
3. 循环里的 i/j/k 不算坏味道。作用域极小、约定俗成的短名反而更清晰——命名长度应与作用域大小成正比：作用域越大，名字越要具体。
4. 一致性 > 个人偏好。同一代码库里，取数别一会儿 get、一会儿 fetch、一会儿 retrieve，读者会以为它们有语义差别（其实没有）。

取数动词到底怎么选：同类方案取舍

动词约定俗成的含义典型场景
get轻量、内存内、几乎无代价getter、访问已有字段
find / query可能找不到，返回可空 / Optional数据库、集合查找
fetch / load有明显 I/O 代价（网络、磁盘）远程调用、懒加载
compute / calculate每次都现算，可能耗时派生值、聚合统计

关键不在于"哪个词最对"，而在于在一个团队/代码库里保持一致，让动词本身成为一种关于"代价与副作用"的暗号。

一页话收束：神秘命名之所以排在坏味道之首，是因为命名是一次设计、却要被阅读一万次——坏名字的成本要乘以 N。它的药方（改名）几乎零风险、纯收益：好名字描述意图而非实现，用领域词汇而非泛化黑话，让行为与名字一致而不撒谎，并在整个代码库里保持一致。当你发现"取不出好名字"时，别硬凑——那通常是代码在提醒你：这段逻辑的职责本身还没想清楚。
