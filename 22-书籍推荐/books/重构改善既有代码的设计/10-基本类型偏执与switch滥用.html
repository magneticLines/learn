基本类型偏执与 switch 滥用

用 String/int 硬扛一切概念，用 switch 到处分类型——这是把"领域知识"从代码里活活抽走的两种典型病。

什么是基本类型偏执（Primitive Obsession）

"基本类型偏执"指的是过度使用语言内置的基本类型（int、double、String、boolean）去表达本应有自己类型的领域概念。金额用 double、电话号码用 String、订单状态用 int、坐标用两个 double……表面上"能跑就行"，实则把这些概念的规则和行为全都从类型系统里剥离了出去。

核心原理：类型是编译期就在替你干活的"文档 + 校验器"。当你写 Money 而不是 double，编译器就永远不会让你把"金额"和"重量"相加、不会让你把"人民币"和"美元"混算；你把领域规则（币种、精度、非负）一次性焊进类型，之后所有调用点自动享受这份保护。基本类型偏执的本质，是放弃了类型系统这个免费的守卫，改用一堆散落的 if 和注释去人肉维持不变量。

一个生活类比

基本类型就像超市里"未分装的散装货"：一袋白色粉末，可能是糖、盐、也可能是面粉，全靠贴的标签（注释/变量名）来区分——标签掉了就出人命。领域类型则是"带包装带成分表的商品"：包装本身就限定了里面是什么、怎么用、保质期多久。String phone 是散装货，PhoneNumber phone 是有包装的商品。

Bad · 万物皆基本类型
// 金额？分还是元？哪国货币？
double balance = 100.0;
// 任何字符串都能塞进来
String phone = "not-a-phone";
// 1 是啥 2 是啥，魔法数字
int status = 1;

// 调用方可以传反、传错单位
transfer(balance, "USD");
// 0.1+0.2 != 0.3，浮点算钱是灾难
Good · 领域概念各归其类
Money balance =
    Money.of("100.00", Currency.CNY);
PhoneNumber phone =
    PhoneNumber.parse("010-12345678");
OrderStatus status =
    OrderStatus.PAID;

// 币种不匹配 -> 运行期立刻抛错
account.transfer(balance);
// 非法号码 -> parse 阶段就拦住
高频大坑：用 double/float 表示金额。浮点数是二进制近似，0.1 + 0.2 == 0.30000000000000004，对账时一分钱都对不上。正确做法是用 BigDecimal 或"以最小单位（分）存 long"，并封进 Money 值对象。这几乎是每个电商/金融新人踩过的第一个生产事故——它同时是"基本类型偏执"和"浮点陷阱"两个坑的叠加。

把散装的基本类型升级为值对象，对应的重构手法叫以对象取代基本类型（Replace Primitive with Object）；把 int status = 1 这类"类型码"升级为枚举，叫以子类/枚举取代类型码（Replace Type Code with Subclasses / Enum）。

Money.java值对象：行为跟着数据一起搬进来
// record 天然不可变、自带 equals/hashCode，非常适合做值对象
public record Money(BigDecimal amount, Currency currency) {
    public Money {                              // 紧凑构造器：不变量一次焊死
        Objects.requireNonNull(currency);
        if (amount.signum() < 0)
            throw new IllegalArgumentException("金额不能为负");
        amount = amount.setScale(2, RoundingMode.HALF_UP);
    }
    public Money plus(Money other) {
        if (!currency.equals(other.currency))   // 币种混算 -> 立刻爆
            throw new IllegalStateException("币种不一致");
        return new Money(amount.add(other.amount), currency);
    }
}

switch 滥用：把类型判断散落一地

第二个坏味道是 switch（以及等价的 if-else 链）滥用。Fowler 特别强调：孤立的一个 switch 本身不可怕，可怕的是同一套"按类型分支"的逻辑在代码里反复出现。一旦你在 N 个地方都写了 switch(shape.type)，那么每新增一种形状，你就得记得去改这 N 个地方——漏掉一个就是 bug。这正好和上一节的霰弹式修改是一枚硬币的两面。

Bad · 类型码 + 到处 switch
double area(Shape s) {
  switch (s.type) {
    case CIRCLE:
      return Math.PI*s.r*s.r;
    case RECT:
      return s.w*s.h;
  }
  throw new AssertionError();
}
// perimeter() 里又一个一模一样
// 的 switch... 新增三角形要改两处
Good · 多态，行为归位
sealed interface Shape
    permits Circle, Rect {
  double area();
  double perimeter();
}
record Circle(double r)
    implements Shape {
  public double area(){return Math.PI*r*r;}
  public double perimeter(){return 2*Math.PI*r;}
}
// 新增 Triangle：只加一个类，编译器逼你实现全部方法

这个手法叫以多态取代条件表达式（Replace Conditional with Polymorphism），是本书最"化学反应"级的重构之一。它把"根据类型选行为"这件事，从运行时的手动 dispatch（switch）交还给了语言内置的虚方法分派（vtable）——JVM 每次调用 shape.area() 时，本就会根据对象的实际类型自动跳到正确实现，这是免费且不会漏分支的。

改动一种新类型时，两种设计的"波及面"对比 ┌─ 基本类型偏执 + switch 滥用 ─────────────┐ │ 新增 TRIANGLE： │ │ area() switch <- 手动加 case │ │ perimeter() switch <- 手动加 case │ │ render() switch <- 手动加 case │ │ toJson() switch <- 忘了改 -> BUG │ │ 改动量 = O(switch 出现次数) │ └──────────────────────────────────────────┘ ┌─ 多态（sealed + 子类）─────────────────────┐ │ 新增 Triangle： │ │ class Triangle implements Shape { … } │ │ 编译器：未实现 area/perimeter -> 报错 │ │ 改动量 = O(1)，且漏实现编译期就挡住 │ └──────────────────────────────────────────┘

switch 一定是坏味道吗？现代 Java 的正名

这里要给初学者纠一个偏——switch 本身不是原罪。Fowler 反对的是"用类型码 + 散落的 switch 去模拟多态"。有两类场景，switch（尤其是 Java 14+ 的模式匹配 switch）反而是更好的选择：

  • 行为不属于某个对象：比如你在做序列化、渲染，逻辑天然属于"访问者"而非数据本身，硬塞进领域类反而污染了它（这就是访问者模式要解决的问题）。
  • 封闭的、确定的分支集合：配合 sealed 接口，switch 能做到穷尽性检查（exhaustiveness）——少写一个分支编译不过，等于把"漏 case"这个经典 bug 消灭在编译期。
Java 21 · sealed + pattern matching switch穷尽性由编译器保证
// sealed 封死子类集合，switch 无需 default 也能编译
static double area(Shape s) {
    return switch (s) {                       // 表达式 switch，箭头无 fall-through
        case Circle c   -> Math.PI * c.r() * c.r();
        case Rect r     -> r.w() * r.h();
        // 若漏掉某个 permits 的子类，此处直接编译报错
    };
}
怎么选，一句话：行为"跟着数据走"就用多态（数据的一部分，如 area/save/validate）；行为"跟着操作走"、且分支集合封闭就用 sealed + 模式匹配 switch（外部操作，如渲染成不同格式）。判据是"以后更常新增哪个维度"——常加类型选多态，常加操作选 switch。这其实就是著名的表达式问题（Expression Problem）。

两种方案的权衡

维度类型码 + switch（坏味道）多态（子类分派）sealed + 模式匹配 switch
新增一种类型改 N 处 switch，易漏加一个类，O(1)改每个 switch，但编译器强制补全
新增一种操作加一个 switch改每个类，散得慌加一个 switch，集中
漏分支后果运行期悄悄出错不会漏（虚方法必实现）编译期报错
行为归属散落各处内聚在类里内聚在 switch 里

联系已学：JDK 与 Spring 里的印证

JDK 冷知识：switch 在字节码层面怎么实现？switch(int/enum) 会被 javac 编译成 tableswitch（连续值，O(1) 跳表）或 lookupswitch（稀疏值，二分）指令，比一长串 if-else 快。而 switch(String) 更巧妙——它先按 hashCode() 做一次 lookupswitch，命中后再用 equals() 二次确认（防哈希碰撞），本质是"两级 switch"。对 enum switch，javac 还会偷偷生成一个 $SwitchMap$ 合成数组把常量映射到序号——这也是为什么 enum 的 switch 不受常量重排序影响。了解这些能帮你看懂反编译产物，也是面试加分项。
Spring 里"消灭 switch"的经典范式：策略 + Map 注入。与其写 switch(payType){ case ALIPAY: … case WECHAT: … }，Spring 允许你把所有 PayStrategy 实现注入成一个 Map<String, PayStrategy>（key 为 bean name），用 map.get(payType).pay(order) 直接分派。新增一种支付方式 = 新增一个 @Component，零改动已有代码——这是"以多态取代条件"在框架层面的工业级落地，也完美符合开闭原则（OCP）。
Bad · Service 里长 switch
void pay(Order o, String type) {
  switch (type) {
    case "ALIPAY": /*…*/ break;
    case "WECHAT": /*…*/ break;
    // 新增银联要回来改这里
  }
}
Good · 策略 Map 自动分派
private final Map<String,PayStrategy> strategies;
// 构造注入：Spring 自动收集所有实现

void pay(Order o, String type) {
  strategies.get(type).pay(o);
}
// 加银联 = 加一个 @Component 类

把改动量算出来

两个坏味道殊途同归，都会让"一个概念的知识"散落到系统各处，于是"加一个新种类"的成本从常数变成了线性。用一个公式概括这次重构的收益：

新增一种类型的改动量：O(N 处 switch) ⟶ O(1)
基本类型偏执 switch 散落各处 值对象 + 多态 = 知识内聚

注意火候：不必把每个 int 都包成类（Age、Count 这种无额外规则的场景过度封装反成噪音）。判据是这个基本类型是否携带了规则或行为——有规则（金额、币种、状态机、带校验的字符串）就升级，纯计数就放过。

陷阱：别把"用了 enum"当成"已经消灭了坏味道"。很多人把 int status 换成 enum Status 就收工了，结果 switch(status) 依然散落十几处。枚举只解决了"类型码"，没解决"散落的分支"。如果每个状态的行为差异大，应进一步用状态模式或给枚举常量各自实现抽象方法（Java 枚举支持 { ABSTRACT_METHOD() } 常量体），让行为真正归位。
一句话收束：基本类型偏执的病根是"该有类型的概念没有类型"——领域规则被逐出类型系统，只能靠散落的 if 和注释人肉维持；switch 滥用的病根是"同一套类型判断在多处重复"——每加一种类型就要霰弹式地改多处。两者的解药是同一个方向：让知识内聚——用值对象/枚举给概念一个家，用多态或 sealed 穷尽 switch把分支收束成一处。判断该不该重构，就问一句："再加一种，我要改几个地方？"
