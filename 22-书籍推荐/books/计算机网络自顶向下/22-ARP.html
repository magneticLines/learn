ARP：地址解析协议

IP 地址负责"从哪到哪"，MAC 地址负责"下一跳交给谁"——ARP 就是把 IP 翻译成 MAC 的那本"本地电话簿"。

一、为什么需要 ARP：两套地址的鸿沟

前面几章我们一直在讲 IP：数据报靠 32 位 IP 地址在全网被路由，从源主机一跳跳到目的主机。但真正把比特送上物理线路的是链路层，而链路层认的不是 IP，而是网卡上烧死的 48 位 MAC 地址（也叫物理地址 / 硬件地址）。于是同一台主机同时背着两个身份：

维度IP 地址（网络层）MAC 地址（链路层）
长度32 位（IPv4）48 位
由谁分配逻辑分配，随所在网络而变（DHCP / 静态）出厂烧录在网卡，全球唯一（理论上）
作用范围端到端：贯穿整条路径，标识"最终目的"逐跳：只在同一链路 / 同一子网内有效
换网后是否变会变（换到别的子网就换 IP）不变（跟着网卡走）
一个生活类比： IP 像是"收件人姓名 + 城市地址"（谁、最终送到哪），MAC 像是"这一段路由哪辆快递车装车"。快递从北京发往上海，收件人不变（IP 端到端不变），但北京仓 → 分拣中心 → 上海仓，每一段装车的车辆都在换（MAC 逐跳改写）。ARP 回答的就是："下一站要把包裹交给哪辆车？"

问题来了：主机 A 要给同一子网里的主机 B 发一个 IP 数据报，它知道 B 的 IP（比如 192.168.1.5），可链路层要求它填上 B 的 MAC 地址才能封装成帧。A 怎么根据一个 IP 查到对应的 MAC？这正是 ARP（Address Resolution Protocol，地址解析协议，RFC 826）要解决的唯一问题。

二、ARP 的核心机制：一次广播问，一次单播答

ARP 的思路简单到近乎"原始"：既然不知道谁是谁，那就在整个局域网里吼一嗓子。它靠两种报文完成解析：

  • ARP 请求（Request）：以广播方式发出，目的 MAC 填 FF:FF:FF:FF:FF:FF，子网内所有主机都会收到。内容是"谁的 IP 是 192.168.1.5？请把你的 MAC 告诉我。"
  • ARP 应答（Reply）：只有 IP 匹配的那台主机才回应，以单播直接发回给请求者，内容是"192.168.1.5 在我这儿，我的 MAC 是 xx:xx:...。"
主机 A (192.168.1.2 / MAC aa:aa) 子网内其它主机 想发数据给 192.168.1.5，但不知道它的 MAC ① ARP Request ──广播 FF:FF:FF:FF:FF:FF──► 所有主机都收到 "谁是 192.168.1.5？请告诉 aa:aa" ├─ 192.168.1.3 (不是我) 丢弃 ├─ 192.168.1.4 (不是我) 丢弃 └─ 192.168.1.5 (是我!) 回应 ② ARP Reply ◄──单播 直接发回 aa:aa──── "192.168.1.5 的 MAC 是 bb:bb" ③ A 把 (192.168.1.5 -> bb:bb) 写进 ARP 缓存 以后再发就不用广播了，直接查表
为什么请求用广播、应答用单播？ 请求时 A 还不知道目标的 MAC，只能广播"群发"才能触达那台未知主机；而应答时，B 已经从请求报文里读到了 A 的 MAC（请求里带着发送方的 IP+MAC），所以能精准单播回去，不必再打扰其他主机。这是一个"用一次广播换取此后长期单播"的经典权衡。

ARP 缓存：一张会过期的本地电话簿

如果每发一个包都广播一次，局域网早被 ARP 淹没了。所以每台主机 / 路由器都维护一张 ARP 缓存表（ARP cache），把解析结果缓存下来。表项通常带一个 TTL（生存时间，典型约 20 分钟），到期自动删除——这是为了应对"IP 换了网卡 / MAC 变了"的情况，避免旧映射永远错下去。

查看本机 ARP 缓存Shell
# Linux / macOS
$ arp -a
? (192.168.1.1) at 34:60:f9:1a:2b:3c [ether] on eth0   # 网关
? (192.168.1.5) at 00:1c:42:aa:bb:cc [ether] on eth0

# 现代 Linux 更推荐 iproute2：
$ ip neigh show
192.168.1.1 dev eth0 lladdr 34:60:f9:1a:2b:3c REACHABLE
192.168.1.5 dev eth0 lladdr 00:1c:42:aa:bb:cc STALE   # STALE=表项老化，下次用会重新校验
联系 DNS 与你已学的缓存思想： ARP 之于"IP→MAC"，几乎就是 DNS 之于"域名→IP"的链路层翻版——都是把一次昂贵的查询结果缓存起来、加 TTL 兜底一致性。这跟 CPU 的 TLB（缓存"虚拟页→物理页"）、JVM 的方法内联缓存本质同构：用一点点"可能过期"的风险，换绝大多数情况下的零查询开销。缓存无处不在，套路都一样。

三、ARP 报文长什么样

ARP 报文直接封装在以太网帧里（以太网类型字段 0x0806 标识"这是 ARP"，而 IP 是 0x0800）。它的关键字段：

以太网帧头 | ARP 报文（28 字节，用于 IPv4/以太网） ──────────────────────┼─────────────────────────────────────────── 目的 MAC FF:FF:...:FF | 硬件类型 = 1 (以太网) 源 MAC aa:aa:... | 协议类型 = 0x0800 (IPv4) 类型 0x0806(ARP) | 硬件长度 = 6 协议长度 = 4 | 操作码 op = 1(请求) / 2(应答) <── 请求还是应答看这里 | 发送方 MAC (Sender HA) | 发送方 IP (Sender PA) <── 请求里也带着自己，方便对方回填缓存 | 目标 MAC (Target HA) <── 请求时此处全 0（正是要问的） | 目标 IP (Target PA) <── "我要找谁"
易错点：ARP 到底是哪一层的协议？ 这是高频面试坑。ARP 不封装在 IP 里（它没有 IP 头），而是直接坐在以太网帧上，因此它不属于严格的网络层之上；但它又服务于网络层（帮 IP 找 MAC）。所以常见说法是："ARP 是介于网络层与链路层之间的辅助协议"。切勿答"ARP 跑在 IP 之上"——恰恰相反，是 IP 的数据链路封装依赖 ARP。

四、跨子网怎么办？ARP 只解析"下一跳"

最容易被误解的一点：ARP 从不为"远端目的主机"解析 MAC。如果目的 IP 不在本子网（比如你要访问 8.8.8.8），主机会先比对子网掩码，发现"不在本网段"，于是它要做的不是去 ARP 那个遥远的 IP，而是把包交给默认网关——所以它 ARP 的是网关的 IP，拿到网关的 MAC，把帧发给网关，剩下的交给路由。

目的 IP 在本子网？ │ ├─ 是 ──► ARP 目的主机的 IP，拿它的 MAC，直接封帧发过去 │ └─ 否 ──► ARP 默认网关的 IP，拿网关的 MAC 帧的目的MAC = 网关MAC，但 IP 头里的目的IP 仍是最终目的(8.8.8.8) ↓ 网关收到后，再 ARP 下一跳……如此逐跳，MAC 每跳改写、IP 端到端不变
抓住这条主线： 一次跨网通信中，IP 头里的源/目的 IP 自始至终不变（端到端寻址），而每一跳的以太网帧头里的源/目的 MAC 都在被改写（逐跳转发）。ARP 就是每一跳用来填写"下一跳 MAC"的那个动作。理解了"IP 管全程、MAC 管一跳、ARP 管翻译"，链路层转发的画面就通了。

五、代码视角：Java 里能不能拿到 MAC？

ARP 发生在操作系统内核协议栈里，应用层通常感知不到。但 JDK 提供了读取本机网卡 MAC 的 API——注意，只能读本机的，读不到别人的（读别人的 MAC 属于内核 ARP 缓存的活儿）。

读取本机网卡的 MAC 地址Java
import java.net.*;

public class MacReader {
    public static void main(String[] args) throws Exception {
        InetAddress local = InetAddress.getLocalHost();
        NetworkInterface ni = NetworkInterface.getByInetAddress(local);
        byte[] mac = ni.getHardwareAddress();  // 拿到 48 位 MAC 的 6 个字节
        if (mac != null) {
            StringBuilder sb = new StringBuilder();
            for (int i = 0; i < mac.length; i++) {
                // %02X: 每个字节转两位十六进制，用冒号连成 aa:bb:cc:dd:ee:ff
                sb.append(String.format("%02X%s", mac[i],
                        (i < mac.length - 1) ? ":" : ""));
            }
            System.out.println("本机 MAC = " + sb);
        }
        // 想查"别人"的 MAC？JDK 没有 API——那是内核 ARP 表的职责，
        // 只能读 /proc/net/arp 或调 `ip neigh`，或用 pcap 发原始 ARP 帧。
    }
}
为什么 getHardwareAddress() 可能返回 null？ 回环接口（lo / 127.0.0.1）没有物理网卡，自然没有 MAC；某些虚拟接口、或权限不足时也会返回 null。这提醒我们：拿到的 NetworkInterface 不一定有硬件地址，取值前务必判空——和处理 getLocalHost() 在多网卡机器上"选错网卡"一样，都是网络编程里常被忽视的边界。

六、ARP 欺骗：一个"没有认证"的古老协议之痛

ARP 诞生于 1982 年，那是一个"局域网里都是自己人"的年代，所以它完全没有认证机制：谁都能发 ARP 应答，收到应答的主机无条件相信并更新缓存——哪怕它根本没发过对应的请求（这叫"免费/无端 ARP"，Gratuitous ARP）。这就埋下了 ARP 欺骗 / ARP 投毒（ARP Spoofing / Poisoning）的祸根。

攻击 — ARP 投毒中间人
攻击者 M 持续伪造 ARP 应答：

对受害者 A 说:
  "网关 192.168.1.1
   的 MAC 是 我(M) 的 MAC"
对网关 G 说:
  "主机 192.168.1.2(A)
   的 MAC 是 我(M) 的 MAC"

→ A 发往网关的所有流量
  实际先到 M，M 再转给 G
→ M 静默窃听/篡改
  (中间人 MITM)
防御 — 分层加固
1. 静态 ARP 绑定
   关键设备写死 IP↔MAC
2. 交换机 DAI
   (Dynamic ARP Inspection)
   + DHCP Snooping 校验
3. 划分 VLAN 缩小广播域
4. 端到端加密兜底
   HTTPS/TLS/SSH 即便被
   中间人也无法解密内容
5. arpwatch 监控 MAC 突变
面试高频 & 认知纠偏： ①ARP 欺骗只能在同一广播域（同一局域网）内实施，因为 ARP 报文出不了本子网——所以它常见于公共 WiFi、公司内网，而非跨公网攻击。②很多人以为"投毒后攻击者就能看到我的密码"，但只要你用的是 HTTPS / TLS，攻击者即便截获流量也只是密文，看不到明文——这正是传输层加密作为"不信任下层网络"的最后防线的价值。③别把 ARP 欺骗和 DNS 欺骗、DHCP 欺骗搞混：它们都利用"无认证"的老协议，但作用在不同层，投毒的表也不同（ARP 表 / DNS 缓存 / 默认网关配置）。

七、ARP vs 其它"地址映射"方案

ARP 是"IP→MAC"的正向解析，围绕它还有一族相关协议，理解它们的分工能把知识串成网：

协议解决什么方向 / 备注
ARP已知同网段 IP，求 MACIP → MAC，广播请求、单播应答，IPv4 专用
RARP（已淘汰）无盘工作站开机时，已知自己 MAC，求 IPMAC → IP，功能已被 DHCP 全面取代
Gratuitous ARP主动广播"我的 IP↔MAC"，用于检测 IP 冲突 / 通告 MAC 变更（如 VIP 漂移）无人请求的自发 ARP，也是投毒的工具
NDP（邻居发现）IPv6 世界里 ARP 的替代者基于 ICMPv6，用多播而非广播，功能更强（含 SLAAC 自动配置）
为什么 IPv6 抛弃了 ARP，改用 NDP？ 一是 IPv6 干脆取消了"广播"这一寻址方式（广播会打扰全网所有主机，太吵），改用更精准的多播——只有加入了特定"被请求节点多播组"的主机才会被打扰。二是 NDP 顺手把地址自动配置（SLAAC）、路由器发现、重复地址检测都收编进来，比"ARP + DHCP + 一堆补丁"整合得更干净。这体现了协议演进的一贯思路：把散落的、事后打补丁的机制，在新一代里做成内建的一等公民。

八、一次完整通信里 ARP 出场的时机

把本页串成一个真实场景——你在同子网内 ping 192.168.1.5，底层依次发生：

  1. 应用发起 ping，内核构造 ICMP + IP 数据报，目的 IP = 192.168.1.5。
  2. 比对子网掩码：目的在本网段 → 需要目的主机的 MAC 才能封成以太网帧。
  3. 查 ARP 缓存：命中则直接用；未命中则先广播 ARP 请求，此时 ICMP 包被暂存，等 ARP 应答回来。
  4. 收到应答，写入缓存，才把 ICMP 帧真正发出去。
  5. 之后 20 分钟内的所有通信都直接查缓存，不再广播——直到表项老化。
观测小实验： 先 ip -s neigh flush all（或 arp -d）清空 ARP 缓存，再开 tcpdump -n arp 抓包，然后 ping 一个从没通信过的同网段 IP——你会清清楚楚看到"先冒出一对 ARP request/reply，之后才是 ICMP"。这是把抽象协议变成肉眼可见现象的最好方式。
一句话收束： ARP = IP 世界通往 MAC 世界的翻译官。它只在同一子网内工作，靠"广播问一次、单播答一次、缓存记 20 分钟"把 IP 解析成 MAC；跨网时它解析的是下一跳网关而非最终目的。记住三条主线：IP 管全程、MAC 管一跳、ARP 管翻译；以及它的阿喀琉斯之踵——无认证，由此衍生出 ARP 欺骗，而 IPv6 用 NDP、内网用 DAI、上层用 TLS 分别从不同层面把这个 1982 年的老协议"补"到今天还能用。
