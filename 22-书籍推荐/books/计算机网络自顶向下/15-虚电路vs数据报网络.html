虚电路 vs 数据报网络

网络层给上层提供的是"有连接"还是"无连接"的服务？这个二选一决定了路由器要不要记住每一条连接。互联网选了后者——本页讲清这个选择，以及它为什么把"连接"这件事甩给了两端。

一、同一个问题：网络核心该不该"记住"一条连接？

上一章的传输层给了我们两种服务：TCP（面向连接）和 UDP（无连接）。到了网络层，同样存在这道选择题，但含义完全不同——它问的是：网络核心里的路由器，要不要为每一条端到端的通信维护一份状态？

  • 虚电路网络（Virtual Circuit）：要。通信前先建连，沿途每台路由器为这条连接记一行状态，之后的分组沿这条"逻辑电路"走。代表：ATM、帧中继、X.25，以及现代的 MPLS。
  • 数据报网络（Datagram）：不要。没有建连，每个分组各自带着完整目的地址独立上路，路由器只按目的地址查表转发，转完就忘。代表：互联网（IP）。
生活类比先建立直觉：
· 虚电路像坐高铁——出发前买票订好车次座位，全程走同一条"预定路线"，检票系统一路认得你这张票（沿途有状态）。
· 数据报像寄明信片——每张明信片自己写着完整收件地址，扔进邮筒；每个邮局只看地址决定往哪个方向递，不记得你之前寄过什么，不同明信片甚至可能走不同路线。

二、虚电路：网络核心维护连接状态

一条虚电路（VC）的一生分三个阶段，和 TCP 的连接管理形似，但发生在网络层、发生在路由器里：

① 建立 setup 发送方请求 → 沿途每台路由器插入一行转发状态、预留资源(可选) ② 数据传输 分组头只带"本段 VC 号"，路由器查表转发 ③ 拆除 teardown 逐跳删除状态，释放资源

精髓：VC 号是"逐跳改写"的本地编号

初学者最意外的一点：一条虚电路在每一段链路上的编号是不一样的，路由器每转发一跳就把分组头里的 VC 号改写成下一段的编号。路由器的转发表把 (入接口, 入VC号) 映射到 (出接口, 出VC号)。

路径 A ── 接口1 ── R1 ── 接口2 │ 接口3 ── R2 ── 接口1 ── B A ──────────► R1 ──────────► R2 ──────────► B 头部 VC=12 头部 VC=22 头部 VC=32 │ │ │ R1 查表: (入接口1, 12) → (出接口2, 22) R2 查表: (入接口3, 22) → (出接口1, 32)

R1 的转发表里，这条 VC 对应这样一行：

入接口入 VC 号出接口出 VC 号
112222
............
为什么不干脆让 VC 号全程不变？因为那样每条 VC 都要一个全局唯一的号，建连时所有路由器得协商出一个大家都空闲的号，既麻烦又浪费号段。改成"逐跳本地编号"后，每台路由器只需保证本机某条链路上的 VC 号不重复即可，建连时随手挑一个本段空闲号写进表就行——把"全局唯一"这个强约束降级成了"链路内唯一"。这和数据库里"全局自增 ID vs 每表本地自增 ID"是同一种降耦合思路。

三、数据报网络：无连接，每个分组自带地址

数据报网络里没有建连、没有 VC 号。每个分组头都带着完整的目的 IP 地址，路由器收到后只做一件事：拿目的地址去转发表里查一条出接口，然后转出去。没有"这条连接"的概念，路由器是无状态的。

但转发表不可能为每个可能的 IP 地址（IPv4 有 43 亿个）都存一行。真实做法是按地址前缀（网段）聚合存储，查表时用最长前缀匹配（Longest Prefix Match, LPM）：哪条前缀匹配得最长，就用哪条对应的出接口。

ForwardingTable.java数据报转发 · 最长前缀匹配
// 数据报路由器的核心逻辑：只看目的 IP，无连接状态
class ForwardingTable {
    // 一条表项：网络前缀 + 前缀长度 → 出接口
    record Entry(int prefix, int maskLen, int outIf) {}
    List<Entry> table;

    int lookup(int dstIp) {
        int bestIf = -1, bestLen = -1;
        for (Entry e : table) {
            // 由前缀长度算出子网掩码，比较网络号是否相同
            int mask = e.maskLen == 0 ? 0 : (-1 << (32 - e.maskLen));
            boolean hit = (dstIp & mask) == (e.prefix & mask);
            if (hit && e.maskLen > bestLen) {
                bestIf = e.outIf;
                bestLen = e.maskLen;   // 关键：前缀越长越具体，优先级越高
            }
        }
        return bestIf;   // 命中最长前缀的那条出接口
    }
}
为什么是"最长"前缀优先？因为长前缀代表更具体的路由。比如 10.0.0.0/8 走接口 A，而更具体的 10.1.0.0/16 走接口 B；一个发往 10.1.2.3 的分组两条都匹配，选更具体的 /16。这让"默认路由 0.0.0.0/0 兜底 + 特例覆盖"成为可能，是 CIDR 编址的地基（下一篇 IP 协议会展开）。

四、正面对比：一张表看清取舍

维度虚电路（VC）数据报（Datagram）
连接建立通信前必须建连（有建立时延）无需建连，分组随到随发
路由器状态有状态：每条 VC 一行表项无状态：只按目的地址前缀存表
分组头携带短小的本段 VC 号完整目的地址
转发依据(入接口, VC号) 查表并改写目的 IP 的最长前缀匹配
同一"连接"的分组严格走同一条路径、保序可能走不同路径、可能乱序
故障恢复路径中断 → 整条 VC 断，需重建路由器可独立改路，绕过故障点
复杂度落点网络核心复杂（要管连接）核心简单，复杂度推给端系统
典型代表ATM / 帧中继 / X.25 / MPLS互联网 IP
核心状态量的量级差异（一句话记住取舍）：
核心状态 ∝ 活跃连接数（VC，可能爆炸）  vs  目的前缀数（数据报，相对稳定）

一台骨干路由器每秒可能有百万级并发连接穿过，但全球的路由前缀只有百万条左右且增长缓慢。VC 让状态随连接数膨胀，数据报让状态只随网络拓扑规模变化——这是数据报能撑起互联网体量的根本原因之一。

五、互联网为什么选数据报？端到端原则

ATM 曾被寄予厚望要统一网络，但互联网最终跑的是数据报模型。原因回到互联网的设计哲学——端到端原则（end-to-end principle）：把复杂功能（可靠、有序、连接状态）放到端系统，让网络核心尽量简单、傻、快。

  • 健壮性：源于冷战时期"可生存网络"的诉求（Baran 的分布式网络思想）。核心无状态，某个路由器炸了，分组换条路继续走，不需要重建谁的连接。
  • 异构性：互联网要把千奇百怪的底层链路（以太网、Wi-Fi、光纤、卫星）粘在一起。数据报只要求底层"能尽力把一个分组送到下一跳"，门槛极低。
  • 可演进：核心不理解"连接"和"应用"，所以新应用（Web、视频、P2P）无需改路由器就能跑起来——创新发生在边缘。
并非非黑即白：MPLS 是 VC 思想的现代回归。运营商骨干网在 IP 之上叠了 MPLS，用一个短标签（Label）逐跳交换转发——本质就是虚电路那套"建立标签路径 + 逐跳改写标签"。它换来的是流量工程（能指定路径、做 QoS 和快速重路由），这正是纯数据报"每包自由选路"给不了的确定性。所以现实是数据报打底、VC 补足确定性的混合。

六、三大高频陷阱

面试与理解的高频坑，务必分清：
① "TCP 连接就是虚电路"——错，这是最大的误区。 TCP 是传输层的连接，状态只存在于通信两端的主机里；中间路由器对你的 TCP "连接"一无所知，它们跑的是数据报转发。虚电路的状态是存在网络层路由器里的。IP 网络是数据报网络，TCP 只是在两端"假装"有连接。
② "虚电路 = 电路交换"——错。 虚电路仍然是分组交换（存储转发、统计复用），只是多了"连接状态"这层逻辑；它默认不像电路交换那样独占预留带宽（是否预留资源是可选项）。别把上一章的"电路交换 vs 分组交换"和这里的"虚电路 vs 数据报"混为一谈——前者是资源怎么分，后者是核心记不记状态。
③ "同一条连接的分组一定按序、按同一路径到达"——只在 VC 成立。 数据报网络里，同一对主机的分组可能因路由变化走不同路径、发生乱序；保序是 TCP 在端上用序号重排出来的，不是网络给的。

七、串联软件世界：有状态 vs 无状态

"核心要不要记住连接"这道题，你在设计后端服务时天天在做。VC = 有状态节点，数据报 = 无状态节点，取舍一模一样：

类·虚电路 — 会话状态存本地
// 会话状态存在这台机器内存里
// 像 VC 把连接状态钉在某台路由器
Map<String,Session> local =
    new HashMap<>();

// 后果：请求必须回到同一台(粘性会话)
// 这台一挂，会话全丢；扩容/迁移都难
Session s = local.get(sessionId);
类·数据报 — 无状态 + 自带信息
// 每个请求自带全部凭证(如 JWT)
// 像数据报每个分组自带完整地址
Claims c = Jwt.verify(token);

// 任意一台节点都能独立处理
// 随便挂/随便加机器，水平扩展无痛
handle(c.getUserId());
把点连成线：
· REST 要求无状态：每个 HTTP 请求自带全部上下文、服务器不保存会话——这就是数据报哲学在应用层的翻版，也是它能被负载均衡随意打散、水平扩展的原因。
· 粘性会话（sticky session）/ 长连接网关：则是把"连接状态"留在某个节点上，像 VC 一样换来了亲和性与效率，代价是故障转移和扩容变难。
· 同一句箴言：把状态推到边缘、让核心保持无状态，系统就更容易横向扩展和容错——互联网用数据报证明了这一点，你的微服务架构也在复用它。
一句话收束：虚电路是"先建连、核心记状态、逐跳改写 VC 号"——用有状态换来路径确定与保序；数据报是"无连接、每包自带目的地址、最长前缀匹配转发"——用无状态换来简单、健壮、可演进。互联网基于端到端原则选择了数据报：把连接、可靠、有序这些复杂事甩给两端，让网络核心保持又傻又快。记住三组关键词：有状态 vs 无状态、VC 号逐跳改写 vs 最长前缀匹配、TCP 连接 ≠ 虚电路。
