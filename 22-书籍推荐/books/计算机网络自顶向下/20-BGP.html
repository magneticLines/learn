BGP：路径向量与策略路由

把七万多个自治系统粘成"一个互联网"的胶水协议。它选路的第一依据不是"哪条最快"，而是"谁给我付钱"——本页讲清路径向量为什么能防环，以及策略为什么压过距离。

一、为什么需要 BGP：一个算法撑不起整个互联网

上一篇的路由算法（Dijkstra 的链路状态、Bellman-Ford 的距离向量）都有一个隐含前提：所有路由器彼此信任、目标一致，都想找"最短路径"。这在一个公司、一所大学内部成立，但放到全球互联网就崩了——中国电信凭什么把自己内部的完整拓扑广播给美国的某个 ISP？它又凭什么愿意免费替别人家的流量做中转？

于是互联网被切成了两层。互联网不是一张扁平的图，而是由约 7 万多个自治系统（Autonomous System, AS） 拼成的"网络的网络"。每个 AS 是一家机构统一管理的路由器集合，有一个全球唯一的 AS 号（ASN），比如 AS4134（中国电信）、AS15169（Google）。

┌─────────────── 域内 (Intra-AS) ───────────────┐ AS 内部： OSPF / IS-IS / RIP 目标=最短路径，信任内部拓扑 └────────────────────────────────────────────────┘ ┌─────────────── 域间 (Inter-AS) ───────────────┐ AS 之间： BGP (唯一选择) 目标=遵守商业策略，互不信任 └────────────────────────────────────────────────┘ [AS1 电信]───BGP───[AS2 联通]───BGP───[AS3 Google] │OSPF │IS-IS │内部协议 路由器们 路由器们 路由器们
两个术语先分清（面试常混）：
· IGP（内部网关协议）：AS 内部用的，如 OSPF、IS-IS、RIP。目标单一——找最短/最优路径。
· EGP（外部网关协议）：AS 之间用的。今天 EGP 事实上只剩 BGP 一种（当前版本 BGP-4，RFC 4271）。
一句话：BGP 是把所有 AS 缝在一起、让互联网成其为"互联"网的唯一胶水。

二、为什么是"路径向量"：既不是距离向量，也不是链路状态

BGP 属于 路径向量协议（Path-Vector Protocol），它是距离向量的一种改造。要理解它，先看它没选另外两条路的原因：

方案核心思想为什么不适合域间
链路状态
(OSPF)		每个节点广播完整拓扑，各自算 Dijkstra要把全网拓扑泄露出去（隐私/安全不可接受），且全球拓扑太大算不动
距离向量
(RIP)		只告诉邻居"我到某地距离是几"经典无穷计数问题：链路断了要收敛很久，且"距离"无法表达商业策略
路径向量
(BGP)		不报距离，报完整的 AS 路径：到 X 要依次经过 AS7、AS12、AS3—— 正是这个改造解决了上面两者的痛点

关键升级：距离向量宣告的是"我到 X 的代价是 4"，而路径向量宣告的是"我到 X 要走 AS7→AS12→AS3 这条完整路径"。多带上这一串 AS 列表（AS-PATH 属性），换来两个决定性的好处——天然防环与策略可判定。

路径向量如何"看一眼就防环"

距离向量之所以有无穷计数，是因为一个节点收到"距离=N"时，看不出这条路是不是绕回了自己。路径向量把整条路摊开了：一个 AS 收到一条路由，只要发现 AS-PATH 里已经包含自己的 AS 号，就知道"这是我自己发出去又绕回来的",直接丢弃。防环从"靠时间慢慢收敛"变成了"接收即刻判定"。

PathVector.java路径向量 · 接收路由时的防环判定
// BGP 路由器收到邻居宣告的一条路由时的核心逻辑（简化）
class Route {
    String prefix;         // 目的网段，如 "203.0.113.0/24"
    List<Integer> asPath;  // 路径向量：这条路依次经过的 AS 号
}

boolean accept(Route r, int myAsn) {
    // —— 环路检测：整个 BGP 稳定性的基石 ——
    if (r.asPath.contains(myAsn)) {
        return false;   // AS-PATH 里已有我 → 是回环，立即丢弃
    }
    // 通过检测 → 把自己"前置"进路径，再转告下游邻居
    r.asPath.add(0, myAsn);   // prepend，路径永远反映真实经过顺序
    return true;
}
把它和 JVM 类加载的"双亲委派"放一起体会：两者都靠"把来路记在身上"来防止重复/成环。类加载器向上委派时，靠 parent 引用链保证一个类不会被同一体系加载两次；BGP 靠 AS-PATH 这条"血统链"保证一条路由不会在 AS 之间兜圈子。记录路径本身，就是最朴素的防环手段。

三、eBGP 与 iBGP：跨门与串内

BGP 跑在 TCP 179 端口之上（这点很重要：路由协议自己却依赖传输层的可靠连接，把重传/有序交给了 TCP，自己专注选路）。按会话双方是否同属一个 AS，分两种：

  • eBGP（external）：不同 AS 的边界路由器之间。用来学习"外面世界怎么去"。
  • iBGP（internal）：同一 AS 内部的 BGP 路由器之间。用来把从 eBGP 学到的外部路由,同步给本 AS 内其他边界路由器,让整个 AS 对外表现一致。
AS100 AS200 ┌──────────────────┐ ┌──────────────────┐ │ Ra ──iBGP── Rb │ │ Rc │ │ │ │ │ │ │ │ └──┼───────────┼───┘ └────────┼─────────┘ │ └────────eBGP───────────────┘ (OSPF 跨 AS 学路由 在内部 保证 Ra/Rb 可达)
高频坑：iBGP 不是"AS 内部的路由协议"，别拿它替代 OSPF。iBGP 只负责在 AS 内传播外部学到的路由信息，它不负责让 AS 内两台路由器彼此可达——那是 IGP（OSPF/IS-IS）的活。另一个经典坑：iBGP 不做水平分割式的再传播——从一个 iBGP 邻居学到的路由默认不会再转发给另一个 iBGP 邻居（因为 iBGP 之间穿越同一 AS，AS-PATH 不变，无法靠它防环）。所以经典组网要求 iBGP 全互联（full-mesh），N 台需要 N(N-1)/2 条会话，规模一大就得靠 路由反射器(RR) 或 联盟(Confederation) 来破解。

四、策略路由：为什么"最短"让位给"最省钱"

这是 BGP 最反直觉、也最本质的地方：BGP 选路的首要依据是商业策略，不是路径长短。因为 AS 之间是有钱银关系的，主要三种：

关系含义谁付钱
Provider ↔ Customer
(上游/客户)		你向上游 ISP 买流量出口Customer 付钱给 Provider
Peer ↔ Peer
(对等互联)		两个对等 AS 互相免费交换各自及其客户的流量互不付费（settlement-free）
Sibling
(兄弟/同属)		同一集团下的关联 AS自由互通

由此推出著名的 Gao-Rexford 出口策略——一个 AS 决定"把某条路由宣告给谁"时遵守的经济学铁律：

宣告规则（把哪条路告诉谁，本质是"愿意替谁运流量"）： · 从 Customer 学到的路由 → 告诉所有人 (Provider/Peer/Customer) ✔ 帮客户带货能赚钱 · 从 Provider 学到的路由 → 只告诉自己的 Customer ✘ 不帮 Provider/Peer 白干 · 从 Peer 学到的路由 → 只告诉自己的 Customer ✘ 同上 选路偏好（自己出流量时优先走哪条，靠 LOCAL_PREF 实现）： Customer 路径 > Peer 路径 > Provider 路径 （走客户方向=有人给我付钱；走 Provider=我要给别人付钱，最不划算）
一句话抓住本质："帮客户运流量我能收钱，帮上游运流量我得付钱"。所以宁可绕远也不替 Provider/Peer 白当免费中转——这就是为什么你 ping 一个地理上很近的服务器，路由可能绕了一大圈：路由跟着钱走，不跟着距离走。

五、BGP 的选路决策：一串有序的 tie-breaker

当去往同一个前缀有多条候选路由时，BGP 按一串严格有序的属性逐条比较，前一条分不出胜负才看下一条。核心几步（各厂实现略有出入，但顺序理念一致）：

#属性比较规则本质
1LOCAL_PREF越大越优（AS 内部一致，仅 iBGP 传播）本 AS 的策略意志：想走哪条我说了算
2AS-PATH 长度越短越优近似"跨越 AS 越少"，才轮到"距离"登场
3Origin 类型IGP < EGP < Incomplete路由来源越可信越优
4MED越小越优（同一邻居 AS 的多入口间比较）邻居对我说"从这个口进我更划算"
5eBGP > iBGP优选 eBGP 学来的外部直达优于绕内部
6IGP 到 NEXT_HOP 代价越小越优（热土豆路由）尽快把流量甩出本 AS
选路优先级：LOCAL_PREF > AS-PATH长度 > Origin > MED > eBGP>iBGP > IGP代价
"热土豆路由（hot-potato routing）"是第 6 步的灵魂。当策略和路径都分不出高下时，路由器选择"IGP 代价最小的下一跳"，也就是用最短的内部路径把这个包尽快扔出自己的 AS——像手里拿着烫手的土豆，恨不得马上甩给邻居 AS 去处理。因为流量在自己网里跑得越久越费自己的资源，能早点交给下家最省成本。这解释了为什么去程和回程路径常常不对称。

六、四个高频陷阱

务必分清，全是面试与排障高频点：
① "BGP 选最短路径"——错。 LOCAL_PREF（策略）排在 AS-PATH 长度（近似距离）之前，策略永远压过距离。BGP 是策略驱动协议，不是最短路径协议。
② "AS-PATH 长度 = 路由器跳数"——错。 它数的是经过多少个 AS，不是多少台路由器。一个 AS 内部可能有几十跳，在 AS-PATH 里只算 1。
③ "BGP 收敛很快"——错，恰恰相反。 全球 BGP 收敛可能长达分钟级，且存在路径抖动（route flapping）。为此有 MRAI 定时器、路由抑制（route damping）等机制来抑制震荡。
④ "AS 号是 16 位的"——过时了。 早期 ASN 是 16 位（0–65535，已耗尽），2007 年起启用 32 位 ASN（RFC 4893/6793）。别在设计里写死 short。

七、劫持与信任：BGP 的阿喀琉斯之踵

BGP 诞生于一个"大家都是熟人、彼此信任"的年代，它默认相信邻居宣告的任何路由。这埋下了互联网最大的系统性隐患之一——BGP 前缀劫持（prefix hijacking）：

攻击/事故 — 谎报前缀
// 某 AS 宣告了本不属于它的前缀
// 甚至用更长(更具体)的前缀
prefix : "8.8.8.0/24"
origin : AS_攻击者

// 最长前缀匹配 → 全世界流量
// 被吸到错误的 AS
// 2008 巴基斯坦封 YouTube
→ 意外把全球 YouTube 引流黑洞
防御 — RPKI + 起源验证
// RPKI: 用证书把"前缀↔合法AS"
// 绑定成可验证的 ROA
ROA{ prefix:"8.8.8.0/24",
      maxLen:24, asn:15169 }

// 路由器校验起源 AS 是否合法
if (!rpki.valid(route))
    drop(route);  // 拒绝非法起源
联系你熟悉的安全模型：BGP 的信任问题和早期 SMTP 无认证导致垃圾邮件、DNS 无签名导致缓存投毒（后有 DNSSEC）是同一类病根——互联网早期协议普遍"默认信任"，安全是后来打的补丁。RPKI 之于 BGP，正如 DNSSEC 之于 DNS：用一套 PKI 证书链事后给"信任"补上密码学根基。

八、把点连成线：BGP 与你写的分布式系统

BGP 表面是网络协议，内核是一套大规模、去中心、带策略的分布式状态传播系统。它的每一个设计取舍，你在做微服务时都会重逢：

反例 — 全局最优的幻想
// 幻想：有个中心大脑掌握全网
// 拓扑，算出"全局最短/最优"
Graph world = fetchWholeInternet();
Path best = dijkstra(world, dst);

// 现实崩塌：
// 1.没人愿交出内部拓扑(隐私)
// 2.规模太大算不动/不收敛
// 3.各家利益不一致,没有"全局最优"
正例 — 局部自治 + 策略暴露
// 每个节点只维护本地视图
// 只暴露"愿意公开"的信息
LocalPolicy p = this.policy;
Route r = bestByPolicy(candidates, p);

// 对外只宣告选中的一条
// 内部实现/拓扑完全隐藏
advertise(r, neighbors);
// = 服务发现里"只暴露契约,藏实现"
三条可迁移的架构直觉：
· 放弃全局最优，追求局部自治：BGP 不追求全网最短，只让每个 AS 按自己策略做局部最优决策——和微服务里"每个服务自治、别搞上帝式全局编排"如出一辙。
· 信息隐藏是可扩展性的前提：AS 只对外宣告一条选中的路由、藏起内部拓扑，才让全球路由表规模可控。这就是接口/契约暴露、实现隐藏。
· 最终一致而非强一致：BGP 收敛是最终一致的（收敛期间会有短暂路由不一致/黑洞），换来了海量规模下的可用性——和 AP 系统、Eureka 式服务注册中心的取舍是同一个 CAP 权衡。
一句话收束：BGP 是缝合 7 万多个自治系统的唯一域间协议，跑在 TCP 179 上。它用路径向量（携带完整 AS-PATH）换来两件事——看一眼路径里有没有自己就能防环，以及能表达并执行商业策略。它选路的铁律是"策略（LOCAL_PREF）压过距离（AS-PATH 长度）"，路由跟着钱走（Gao-Rexford + 热土豆）。记住四组关键词：路径向量防环、eBGP 跨门 / iBGP 串内且需 full-mesh、策略优先于最短、默认信任 → 前缀劫持 → RPKI 补救。
