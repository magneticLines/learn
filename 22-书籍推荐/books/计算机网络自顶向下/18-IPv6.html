IPv6：动机、头部对比、过渡

IPv4 的 32 位地址快用光了。IPv6 不只是"把地址加宽到 128 位"，它借机把头部重新设计得更精简、让路由器转得更快——但换代不能一夜之间完成，于是有了双栈与隧道这套"两代共存"的过渡工程。

一、动机：为什么非换 IPv6 不可

IPv4 地址是 32 位，理论上限约 43 亿个。上世纪 90 年代初，随着互联网爆发式增长，人们就已经预见到"地址迟早不够分"。这不是危言耸听：2011 年 2 月，IANA（顶层地址分配机构）把最后几个 /8 地址块分完；此后各区域注册机构（RIR）也陆续见底。IPv6 就是为了从根上解决这个问题而设计的下一代 IP 协议。

IPv4：2³² ≈ 43 亿   →   IPv6：2¹²⁸ ≈ 3.4 × 10³⁸

3.4×10³⁸ 是什么概念？给地球上每一粒沙子分配上万亿个地址都绰绰有余——从此"地址稀缺"这件事在可预见的未来彻底消失。但设计者的野心不止于此，他们借这次不兼容的大版本升级，顺手修掉了 IPv4 头部里那些"拖慢路由器"的历史包袱（见第二节）。

联系上一节：NAT 是"止痛药"，IPv6 才是"根治"。你可能会问：既然 NAT 已经让一个公网 IP 背后藏成千上万台内网设备，地址不就够用了吗？没错——正是 NAT 的普及，大大延缓了 IPv6 的落地进度。但 NAT 破坏了 IP"端到端"的初衷（内网设备无法被外部直接寻址，P2P、VoIP 都得靠打洞），还给路由器增加了维护映射表的负担。IPv6 地址管够，理论上可以让 NAT 彻底退休，恢复端到端连通性。

二、IPv6 头部：为"快"而生的定长 40 字节

IPv6 头部最反直觉的一点：地址长度翻了 4 倍（从 4 字节到 16 字节），可整个基本头部却是固定的 40 字节，字段数量反而比 IPv4 更少。这不是巧合，而是刻意的"减法设计"——目标是让核心路由器转发得更快。

0 4 12 31 ← 位 +-------+--------------+----------------------------------+ |版本(4)| 流量类别(8) | 流标签 Flow Label (20) | +-------+--------------+-------------+--------------------+ | 载荷长度 (16) | 下一个头部(8)|跳限(8)| +------------------------------------+-------------+------+ | | | 源地址 Source Address (128 位 / 16 字节) | | | +---------------------------------------------------------+ | | | 目的地址 Destination Address (128 位 / 16 字节)| | | +---------------------------------------------------------+ 固定 40 字节；可选内容全部移到后面的"扩展头部"链

三个设计要点，每一个都值得记住"为什么"：

  • 定长头部 → 无需解析首部长度：IPv4 头部长度可变（20~60 字节），路由器得先读"首部长度字段"才知道数据从哪开始。IPv6 恒为 40 字节，路由器可以硬编码偏移，处理路径更简单、更适合硬件流水线。
  • 路由器不再分片 → 拿掉分片字段：IPv4 中间路由器遇到过大的包会就地分片，很耗资源。IPv6 规定只有源主机能分片；中间路由器若发现包超过链路 MTU，直接丢弃并回一个 ICMPv6"包太大"，由源端做路径 MTU 发现（PMTUD）后自己切小。
  • 去掉首部校验和 → 每跳少算一次：IPv4 每经过一跳，TTL 减 1 就得重算一遍首部校验和。IPv6 认为链路层（如以太网 CRC）和传输层（TCP/UDP 校验和）已经兜底，网络层再算是浪费，干脆删掉。
"下一个头部（Next Header）"是 IPv6 的精髓之一。它一字段两用：既可能指向上层协议（如 6=TCP、17=UDP，和 IPv4 的"协议"字段含义相同），也可能指向一个扩展头部（Extension Header）。分片、逐跳选项、路由、认证等原本塞在 IPv4 头里的"选项"，在 IPv6 里都变成独立的扩展头，像链表一样一节接一节挂在基本头后面。这样核心路由器只看它关心的头，其余整体跳过——把复杂度从"每个包都解析"挪到了"只有需要的端点才解析"。

三、头部字段逐个对比：IPv4 里的字段去哪了

理解 IPv6 头部最好的方式，是拿它和你上一章学过的 IPv4 头部逐个对照，看每个字段的"命运"：

IPv4 字段在 IPv6 里的命运为什么这么改
版本 Version保留（值=6）用来区分 v4/v6
首部长度 IHL删除头部定长 40 字节，无需再标
服务类型 ToS改名 流量类别做 QoS / 区分服务，含义类似
总长度改为 载荷长度只算载荷，不含 40 字节头
标识 / 标志 / 片偏移删除（移到扩展头）路由器不再分片，日常路径不需要
TTL改名 跳限 Hop Limit名字更诚实：它本就是"剩余跳数"
协议 Protocol改为 下一个头部既指上层协议，也指扩展头
首部校验和删除链路层 + 传输层已兜底，省每跳开销
选项 Options删除（改扩展头）保证基本头定长
—（新增）流标签 Flow Label标识同一数据流，便于路由器统一特殊处理（如低时延）
高频面试陷阱：IPv6 头部更长，但字段更少、更简单。很多人一看"40 字节 > 20 字节"就以为 IPv6 头更复杂、更慢。恰恰相反——多出来的字节几乎全是加长的地址；字段数量从 IPv4 的十几个降到 8 个，还去掉了变长选项、逐跳校验和、中途分片这三大"耗时活"。所以更长的头 ≠ 更慢的转发，这是考察你是否真懂设计意图的经典点。

四、地址表示法：128 位怎么写才不崩溃

128 位如果写成 IPv4 那样的点分十进制会有 16 段，太长。IPv6 改用冒分十六进制：8 组、每组 16 位（4 个十六进制数字）、用冒号分隔。再配两条压缩规则让它变短：

原始（8 组，每组 4 位十六进制）： 2001:0db8:0000:0000:0000:ff00:0042:8329 规则① 每组的前导零可省略： 2001: db8: 0: 0: 0:ff00: 42:8329 规则② 一段连续全 0 组可用 "::" 折叠（整个地址只能用一次！）： 2001:db8::ff00:42:8329 └──────┘ 这里的 :: 代表 3 组连续的 0 常见特殊地址： ::1 环回地址（等价 IPv4 的 127.0.0.1） :: 未指定地址（全 0，等价 0.0.0.0）
两个必踩坑。其一：:: 在一个地址里只能出现一次——否则 2001::25::1 无法确定每个 :: 各代表几组 0，属于非法地址。其二（后端最常见）：URL 里写 IPv6 地址必须用方括号括起来，因为冒号会和"端口分隔符"冲突。正确写法是 http://[::1]:8080/api，而不是 http://::1:8080/api——后者浏览器和很多 HTTP 客户端会解析失败。
顺带一提：IPv6 没有"广播"了。IPv4 的广播（如 ARP 全网喊话）在 IPv6 里被多播（multicast）和任播（anycast）取代。原本靠 ARP 做的"IP → MAC"解析，改由基于 ICMPv6 的 邻居发现协议（NDP）完成；主机还能通过路由器通告的前缀 + 自己的接口标识自动配好地址（SLAAC，无状态自动配置），很多场景下连 DHCP 都省了。

五、过渡：两代协议如何"边跑边换轮子"

最棘手的现实问题是：全球有海量 IPv4 设备，不可能选一个"割接日"让全世界同时切到 IPv6。而且 IPv6 和 IPv4 报文格式不兼容——一台纯 IPv4 的路由器根本看不懂 IPv6 包。于是工程上采用"共存 + 逐步迁移"的三类手段：

手段做法适用 / 代价
双栈 Dual Stack主机 / 路由器同时装 IPv4 和 IPv6 两套栈，靠 DNS（A 记录 vs AAAA 记录）决定用哪套最平滑；但只要路径上有一段是纯 IPv4，IPv6 独有字段（如流标签）就会丢失
隧道 Tunneling把整个 IPv6 包当"货物"塞进 IPv4 包里，穿过纯 IPv4 区域，出口再拆开Kurose 重点；能让两片 IPv6"孤岛"跨越 IPv4 海洋互通
翻译 Translation网关在 v4/v6 头部之间做转换（如 NAT64 + DNS64）用于纯 IPv6 端要访问纯 IPv4 服务器；有状态、易成瓶颈

隧道（Tunneling）：把 IPv6 包"套娃"进 IPv4 包

隧道是《自顶向下》着墨最多、实践中也最主流的方案。核心思想一句话：把 IPv6 数据报整体当作 IPv4 数据报的载荷。入口路由器负责封装，出口路由器负责解封装，中间那片纯 IPv4 网络对里面装的是什么一无所知——它只是在转发一个普通 IPv4 包（协议号 41 标明"里面是 IPv6"）。

IPv6 岛 A ─[IPv6包]─► R1 ══════ 纯 IPv4 网络（隧道）══════ R2 ─[IPv6包]─► IPv6 岛 B │ │ 封装(encapsulate) 解封装(decapsulate) 把整个 IPv6 包塞进 剥掉外层 IPv4 头， IPv4 包的数据区 还原出原封不动的 IPv6 包 隧道内 R1→R2 线上真正传输的字节： ┌────────────┬────────────────────────────────────────┐ │ IPv4 头 │ IPv6 头 | IPv6 载荷（如 TCP 段） │ └────────────┴────────────────────────────────────────┘ └─外层─┘ └────────── 内层：原始 IPv6 包 ──────────┘
生活类比：跨国邮寄一封"外文信"。你的 IPv6 包好比一封只有 IPv6 邮局看得懂的信。要经过一个只认 IPv4 信封的国家，就把这封信整个塞进一个标准 IPv4 信封（封装），贴上 IPv4 地址寄出；沿途邮局按 IPv4 规则转发，根本不拆开看里面；到了对面 IPv6 邮局（出口路由器）再拆掉外层信封（解封装），取出原信继续按 IPv6 投递。隧道 = 用一种协议的包裹，护送另一种协议的包裹过境。6to4、Teredo、ISATAP 都是这套思路的具体实现。

双栈的取舍：一个隐蔽的"字段丢失"陷阱

双栈的坑 — IPv6 字段在 v4 段"蒸发"
A(v6) ─► R(v6) ─► S(仅v4) ─► D(v6) ...

R 想把 IPv6 包发给纯 v4 的 S，
只能把它"降级"翻译成 IPv4 包，
IPv6 独有的【流标签】无处安放 → 丢弃。
即便后面 D 又是 v6，也补不回来 ✗
隧道 — IPv6 包全程原样保留
A(v6) ─► R1 ══隧道══ R2 ─► D(v6)

R1 把整个 IPv6 包封进 IPv4 包，
穿过 v4 区域时内层字节【原封不动】，
R2 解封装后 D 拿到的还是
一模一样的 IPv6 包，流标签都在 ✓
这解释了为什么"隧道"比"双栈翻译"更受青睐。双栈虽然部署简单，但一旦路径中夹了纯 IPv4 节点，就不得不把 IPv6 包翻译成 IPv4——IPv6 独有的信息（流标签等）会被永久丢弃，无法在后段恢复。隧道则把 IPv6 包当成不透明货物护送过境，全程逐字节保真，这也是它成为主流过渡手段的根本原因。

六、落到代码：Java / 操作系统里的 IPv6

好消息是，JDK 早就把双栈处理得相当透明：InetAddress 是抽象父类，具体是 Inet4Address 还是 Inet6Address 由解析结果决定。一个域名可能同时拥有 A（IPv4）和 AAAA（IPv6）记录：

InetAddress 同时拿到 IPv4 / IPv6 地址Java
// 一个域名可能同时挂着 A(IPv4) 和 AAAA(IPv6) 记录
InetAddress[] addrs = InetAddress.getAllByName("www.example.com");
for (InetAddress a : addrs) {
    if (a instanceof Inet6Address) {
        System.out.println("IPv6 → " + a.getHostAddress()); // 2606:2800:...
    } else if (a instanceof Inet4Address) {
        System.out.println("IPv4 → " + a.getHostAddress()); // 93.184.x.x
    }
}

// JVM 启动参数控制双栈偏好（老系统 / 容器里很常见）：
//   -Djava.net.preferIPv4Stack=true      只用 IPv4，彻底不碰 v6
//   -Djava.net.preferIPv6Addresses=true  解析时优先返回 IPv6

真正会翻车的，往往是应用层自己"手搓"的地址校验逻辑——默认全世界都是点分十进制 IPv4：

反例 — 正则硬校验点分十进制
// 假设地址一定是 a.b.c.d
boolean ok = ip.matches(
    "\\d{1,3}(\\.\\d{1,3}){3}");
if (ok) connect(ip);
// "::1"、"2001:db8::1" 全被判非法 ✗
// v6 时代这段代码直接拒绝合法地址
正例 — 交给 JDK 解析，v4/v6 通吃
try {
    InetAddress addr =
        InetAddress.getByName(ip);
    connect(addr);
    // "::1"、"2001:db8::1" 都能识别 ✓
} catch (UnknownHostException e) {
    // 真正非法才走到这里
}
迁移体检清单（面试也爱问"你的服务支持 IPv6 吗"）：① 日志 / 数据库存 IP 的字段够不够长（IPv6 文本最长 45 字符，别再用 varchar(15)）；② 有没有到处写正则校验点分十进制；③ 拼 URL 时 IPv6 有没有加方括号 [...]；④ 服务端 bind 时是绑到 0.0.0.0（仅 v4）还是 ::（双栈）；⑤ 防火墙 / 安全组规则是不是只配了 IPv4——很多"IPv6 走后门绕过限流"的事故就出在这里。

七、收束

一句话收束：IPv6 的起点是地址耗尽（32 位 → 128 位，2¹²⁸ 个地址一劳永逸），但它的精髓是为转发提速的头部重构——定长 40 字节（免解析首部长度）、砍掉逐跳校验和、路由器不再分片（改由源端做 PMTUD）、变长选项下放为扩展头部链（靠"下一个头部"串联）。地址用冒分十六进制 + :: 压缩（只能折叠一次），URL 里要加方括号。因为无法一夜割接，过渡靠双栈（简单但会丢 IPv6 字段）、隧道（把 IPv6 包套进 IPv4 包全程保真，主流方案）、翻译（NAT64）三件套。落到代码：别手写正则校验 IP，交给 InetAddress；数据库字段留够长度、URL 记得加中括号、bind 到 :: 开双栈——这些才是你迁移时真正会踩的坑。
