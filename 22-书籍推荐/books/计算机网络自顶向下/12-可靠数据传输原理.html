可靠数据传输原理：rdt 协议家族

底层信道会出错、会丢包、会乱序——可靠数据传输要做的，就是在这样一条"不靠谱的管道"上，一个字节不差、不重、不乱地把数据送到对面。本节用五步 FSM 从零造出一个可靠协议，这正是 TCP 的思想内核。

一、问题定义：可靠传输到底要保证什么

可靠数据传输（rdt, reliable data transfer）是传输层要向上层应用兑现的一份"服务契约"：应用把数据交给传输层，传输层负责在一条不可靠的底层信道上，把数据完好无损地交到对端应用手里。所谓"完好无损"，具体是三条承诺：

  • 不出错：交上去的每一位都和发出去的一致（对抗比特差错）。
  • 不丢失：发了 N 个分组，对端应用最终收到 N 个（对抗丢包）。
  • 不乱序、不重复：按发送顺序、且每个分组只交付一次（对抗乱序与重复交付）。
┌──────────────┐ ┌──────────────┐ │ 发送方应用 │ │ 接收方应用 │ └──────┬───────┘ └──────▲───────┘ rdt_send() │ 可靠数据传输(rdt)提供的"契约" │ deliver_data() ┌──────▼───────┐ ~~~ 可靠信道(抽象) ~~~ ┌──────┴───────┐ │ rdt 发送方 │ ───────────────────────► │ rdt 接收方 │ └──────┬───────┘ └──────▲───────┘ udt_send() │ 不可靠信道(unreliable) │ rdt_rcv() ┌──────▼─────────────────────────────────────┴───────┐ │ 会出比特差错 / 会丢包 / 时延不定的真实网络 │ └────────────────────────────────────────────────────┘
关键抽象：把"可靠"这层做成一个协议，扣在不可靠信道之上。上层完全感知不到底层的丢包和差错——就像你写 list.get(0) 时不用关心数组扩容一样。Kurose 用有限状态机（FSM, Finite State Machine）来描述这个协议：发送方 / 接收方各是一台状态机，"收到什么事件 → 做什么动作 → 跳到哪个状态"。下面我们就用"逐步加码信道难度"的方式，从最简单的 rdt1.0 一路进化到 rdt3.0。

二、rdt1.0 → rdt2.0：先解决"会出错"

rdt1.0：完全可靠的信道（基线）

假设底层信道百分百可靠——不出错也不丢包。那协议什么都不用做：发送方来数据就发，接收方来分组就收。这是我们的基线，也说明"复杂度是被信道的缺陷逼出来的"。

rdt2.0：信道会翻转比特（引入 ARQ）

现在信道会把 1 变 0、0 变 1（比特差错），但还不会丢包。要在这条信道上可靠传输，得引入三样东西，它们合起来就是ARQ（Automatic Repeat reQuest，自动重传请求）协议的雏形：

机制作用现实对照
校验和 checksum接收方靠它检测分组是否出错UDP / TCP 首部的校验和字段
确认 ACK / 否认 NAK接收方反馈："收对了(ACK)"或"收坏了(NAK)"打电话时的"嗯""啥？再说一遍"
重传 retransmission发送方收到 NAK 就重发刚才那个分组没听清就再说一遍
生活类比：电话里念一串验证码。你念一段（发分组），对方复述"对，继续"（ACK）或"没听清"（NAK）。听到"没听清"你就重念一遍。这种"发一个、停下来等确认、确认了再发下一个"的工作方式，叫停等协议（stop-and-wait）——rdt2.0 / 2.1 / 2.2 / 3.0 全是停等协议，它们的区别只在于"如何应对确认本身也可能出问题"。

三、rdt2.0 的致命缺陷 → rdt2.1 引入序号

rdt2.0 有个要命的漏洞：它默认 ACK/NAK 一定不会损坏。可 ACK/NAK 也是走同一条会出错的信道！如果发送方收到一个被弄坏、看不清是 ACK 还是 NAK 的反馈，它该怎么办？

选项一：当作 NAK 直接重发。问题来了——如果对方其实早就正确收到了（发的是 ACK 只是路上坏了），这次重发就制造了一个重复分组（duplicate）。而 rdt2.0 的接收方无法分辨手里这个是新分组还是上一个的重传，就会把同一份数据交付两次。这违反了"不重复"的承诺。

反例 · rdt2.0：无序号，重复无法识别
发送方: 发分组 D0
接收方: 收到 D0，交付，回 ACK
   ~~~ ACK 在路上被弄坏 ~~~
发送方: 看不懂反馈，只好重发 D0
接收方: 又收到一个 D0……
        这是新数据？还是重传？
        无从判断 → 重复交付 D0 !
        （数据被应用处理了两次）
正例 · rdt2.1：加 1 比特序号
发送方: 发 [seq=0] D0
接收方: 收 seq=0，交付，回 ACK
   ~~~ ACK 被弄坏 ~~~
发送方: 看不懂，重发 [seq=0] D0
接收方: 咦，还是 seq=0 ?
        我上次已收过 seq=0 →
        判定为重传，丢弃不交付，
        只重回一个 ACK。不重复 !

这就是 rdt2.1 的核心改进：给每个分组打上序号（sequence number）。因为停等协议里"空中"最多只有一个未确认的分组，所以一个比特（0 和 1 交替）就够用——所以它又叫交替比特协议（alternating-bit protocol）。接收方靠序号判断"这是我期待的新分组，还是上一个的重传"。

面试高频陷阱：序号解决的是"重复"，不是"丢失"。很多人把序号的作用记成"防丢包"，错。在 rdt2.1 里，序号纯粹是让接收方能识别重复（以及让发送方能识别收到的是不是当前分组的 ACK）。真正对抗丢包的机制（定时器）要到 rdt3.0 才登场。把"序号→去重、定时器→抗丢"这两件事分开记，你就不会在协议设计题里张冠李戴。

rdt2.2：一个不用 NAK 的等价设计

rdt2.2 是 rdt2.1 的等价变体，只做一件事：废掉 NAK。做法是让接收方在回 ACK 时带上它最后一次正确收到的分组的序号。发送方如果收到的是"对上一个分组的 ACK"（即序号不对），就等价于收到了一个 NAK，于是重发当前分组。

为什么要费劲取消 NAK？为 TCP 铺路。"用带序号的重复 ACK 代替 NAK"这个设计被 TCP 完整继承了——TCP 里根本没有 NAK 字段，接收方永远只回 ACK；发送方连续收到3 个重复 ACK（duplicate ACK），就当作"你要的那个包丢了"的信号，触发快速重传。你现在理解的 rdt2.2，就是后面"TCP 快速重传"那一节的地基。

四、rdt3.0：直面丢包——定时器登场

现在给信道加上最后一种麻烦：分组会整个丢失（数据分组丢、或 ACK 丢都算）。前面所有机制在丢包面前都失灵了——因为分组一旦丢了，接收方压根不会反馈任何东西，发送方就永远死等下去，协议卡死。

怎么办？发送方无法区分"分组丢了""ACK 丢了""网络只是慢"这三种情况，于是采用一个朴素而有效的办法：发出分组后启动一个倒计数定时器，等待一个"合理的"时间；到点还没收到 ACK，就直接重发。这就是超时重传（timeout & retransmit），rdt3.0 由此诞生。

rdt3.0 停等 + 超时重传：一次"数据分组丢失"的时序 发送方 接收方 │ 发 [seq=0] ──────────✗ (分组丢失，接收方啥也没收到) │ ⏱ 启动定时器… │ (死等 → 定时器超时!) │ ⏱ TIMEOUT → 重发 [seq=0] ───────────────► 收到 seq=0 │ ◄──────────────────────────────── 回 ACK0 交付数据 │ 收到 ACK0，翻转序号 → 准备发 seq=1 │ 发 [seq=1] ─────────────────────────────► 收到 seq=1 │ ◄──────────────────────────────── 回 ACK1 交付数据
陷阱：超时会带来"没必要的重复"，所以序号绝不能省。如果只是网络慢、ACK 还在路上，定时器就提前超时（premature timeout）了，发送方会重发，从而在网络里制造出重复分组。正因为丢包机制天然会产生重复，rdt3.0 才必须保留 rdt2.1 的序号去重。这也解释了为什么 TCP 的超时时间要基于 RTT 动态估算（用 EWMA 指数加权移动平均去平滑抖动）——超时设太短会疯狂误重传拖垮网络，设太长则丢包后恢复太慢。定时器长度的选择本身就是一门权衡。

用代码把 rdt3.0 发送方逻辑写清楚

下面这段是 rdt3.0 发送方的核心循环。注意它把前面所有机制都拧在了一起：校验、序号、ACK 判断、超时重发。

Rdt3Sender.java（停等 · 交替比特 · 超时重传）伪代码化的核心逻辑
int seq = 0;                              // 交替比特：只在 0 / 1 之间翻转
while (hasData()) {
    Packet pkt = make(seq, data, checksum(data));  // 打上序号 + 校验和
    send(pkt);                                // udt_send：交给不可靠信道
    startTimer(TIMEOUT);                       // rdt3.0 的灵魂：发完就上闹钟

    Ack ack;
    while ((ack = waitAck()) == null       // null 表示定时器超时了
           || corrupt(ack)                    // ACK 本身损坏，看不清
           || ack.seq != seq) {               // 收到的是"上一个"的 ACK(rdt2.2 思想)
        send(pkt);                            // 以上任一情况：一律重发当前分组
        restartTimer(TIMEOUT);                // 重发后重置闹钟
    }
    stopTimer();                              // 收到了正确的 ACK
    seq ^= 1;                               // 翻转序号，准备发下一个
}
把概念对回代码：checksum 是 rdt2.0 的检错、seq 是 rdt2.1 的去重、ack.seq != seq 判断是 rdt2.2 的"用 ACK 代替 NAK"、startTimer / waitAck()==null 是 rdt3.0 的抗丢包。四层信道难度、四个机制，一段循环全兜住了。

五、停等的性能之殇：信道被白白闲置

rdt3.0 功能上正确了，但性能糟糕——因为它"发一个就停下来等一个 RTT"。用一个 Kurose 书里的经典算例感受一下：一条 1 Gbps 的链路、单向传播时延 15ms（RTT = 30ms）、分组大小 8000 bit（1KB）。

先算把一个分组"推上链路"的发送时延 L/R = 8000 bit / 10⁹ bps = 0.008 ms。发送方真正"忙"的时间只有 0.008ms，然后要干等一整个 RTT 才能发下一个。它的信道利用率（发送方有效工作时间占比）是：

U = (L/R) / (RTT + L/R) = 0.008 / (30 + 0.008) ≈ 0.00027

也就是 0.027%！一条 1 Gbps 的链路，实际有效吞吐只有约 267 kbps——99.97% 的时间信道在空转。画成时间轴一目了然：

发 0.008ms 空等 RTT ≈ 30ms（信道彻底闲置）
根因：停等协议没让"传播时延"这段时间干活。数据在光纤里飞的那 15ms、ACK 飞回来的 15ms，发送方本可以继续塞新分组进管道，却在傻等。解决思路很直接——别停下来等，一口气发多个再一起等确认。这就是流水线（pipelining），也是所有现代可靠传输协议的性能基石。

六、流水线：Go-Back-N vs 选择重传

流水线允许发送方在收到确认前连续发出多个分组，用一个"窗口"限制未确认分组的数量。窗口开 N 个，利用率就大致乘以 N（直到把管道填满）：

U流水线 ≈ N × (L/R) / (RTT + L/R)

但流水线带来一个新问题：管道里某个分组出错或丢了，怎么办？只重那一个，还是把它之后的全重来？这里分出了两条经典路线——回退 N（Go-Back-N, GBN）和选择重传（Selective Repeat, SR）。

Go-Back-N（累积确认 + 出错就"回退 N 步"全部重发） 已确认 │◄── 发送窗口 N=4 ──►│ 还不能发 ┌──┬──┬──┬──┬──┬──┬──┬──┐ │0 │1 │2 │3 │4 │5 │6 │7 │ 假设 pkt2 丢失… └──┴──┴──┴──┴──┴──┴──┴──┘ 接收方对 3、4、5 一律回 "ACK1"(我只连续收到了1)，并丢弃 3/4/5 发送方超时 → 从 2 开始，把 2、3、4、5 全部重发（回退 N）

GBN 的接收方不缓存失序分组，只认"下一个按序到达的"，用累积确认（cumulative ACK）——ACKn 表示"n 及之前全收到了"。代价是一个包丢了，它后面已经正确到达的包会被全部丢弃、全部重发，浪费带宽。SR 则相反：接收方缓存失序但正确的分组，发送方只重传真正出问题的那一个，每个分组单独确认。

对比维度Go-Back-N (GBN)选择重传 (SR)
确认方式累积确认（ACKn = n 及以前全收到）逐个单独确认
接收方缓存失序包不缓存，直接丢弃缓存，等缺口补齐再上交
丢一个包的代价重发它及其后全部分组只重发丢的那一个
接收方复杂度低（无需缓冲区、无需乱序管理）高（要缓冲区 + 逐包定时器）
带宽利用高丢包率下浪费大高效，但实现复杂
序号空间约束窗口 ≤ 2ᵏ − 1窗口 ≤ 2ᵏ / 2（序号空间的一半）
最经典的考试陷阱：SR 的窗口大小必须 ≤ 序号空间的一半。假设序号只有 2 比特（0,1,2,3 共 4 个），若把 SR 窗口开到 3：发送方发 0,1,2 全部成功，但三个 ACK 全丢，于是超时重发 0；而此时接收方的窗口已经滑动到期待 3,0,1，它无法分辨刚收到的 0 是"重传的旧 0"还是"全新一轮的 0"——二义性导致数据被错误接收。所以 SR 必须 窗口 ≤ 序号空间 / 2；GBN 因为累积确认，约束宽松些是 窗口 ≤ 序号空间 − 1。这两个数记反是失分重灾区。

七、串联到 TCP 与工程实践

rdt 家族不是屠龙之技——它是 TCP 的设计蓝本。TCP 把上面所有零件融合成了一个"混血协议"：

rdt 中的原理在 TCP 里的落地
校验和检错TCP 首部 16 位校验和（与 UDP 同源）
序号去重、按序交付基于字节的序号（不是分组号）
rdt2.2：无 NAK、用重复 ACK3 个 duplicate ACK → 快速重传
rdt3.0：超时重传基于 RTT 估算（EWMA）的自适应超时
流水线 + 窗口滑动窗口（叠加流量控制 / 拥塞控制）
GBN 的累积确认TCP 基础用累积确认，靠 SACK 选项补上 SR 式的选择性重传
建立联系：rdt2.1 的"序号去重" = 分布式系统的"幂等性"。重传机制天然带来"至少一次（at-least-once）"投递，也就必然产生重复。接收方靠序号去重，这和你在后端设计里给消息队列（Kafka / RabbitMQ）消费者做幂等键（idempotency key）去重、给支付接口加去重号是同一个问题的同一种解法——"网络会重发 → 消费端必须能识别并丢弃重复"。你在传输层学的这套东西，会在应用层的每一个可靠投递场景里再遇见一次。
一句话收束：可靠传输是被信道的缺陷一步步逼出来的——校验和对付出错、ACK+序号对付重复、定时器超时重传对付丢包，四步走完就是功能正确的停等协议 rdt3.0；但停等让信道空转 99%，于是用流水线 + 窗口提性能，并在"出错怎么重"上分化出GBN（累积确认、回退全重）与SR（逐个确认、只重丢的）两条路。把这三组机制和两种流水线记牢，你就拿到了读懂 TCP 序号、确认、超时、快速重传、SACK 的全部钥匙。
