HTTP：持久连接、Cookie、缓存、条件 GET

HTTP 天生"无状态、每次都要重新拉"。这四个机制，正是围绕"少建连接、记住你是谁、别重复传"三件事，把一个朴素协议打磨成撑起整个 Web 的工程杰作。

一、先立地基：HTTP 是无状态的请求-响应协议

HTTP 建立在 TCP 之上，模型极简：客户端发一个请求报文，服务器回一个响应报文，一来一回。它最重要的性格是无状态（stateless）——服务器不会为"你上一个请求是什么"保留任何记忆。每个请求都像陌生人第一次上门。

为什么故意设计成无状态？这是一个刻意的取舍。有状态的协议（如 FTP 要记住当前目录）需要服务器为每个客户端维护会话状态，一旦并发量上万，内存和复杂度都会爆炸，且任一请求出错就要考虑状态回滚。无状态换来的是极致的可伸缩性与容错性：任何一台服务器都能处理任何一个请求，宕机重启也无所谓——这正是 Web 能水平扩展到十亿级的根基。代价是"记住用户"这件事得另想办法（就是后面的 Cookie）。

但无状态带来两个现实痛点，本页四个机制正是逐一破解它们：

  • 连接开销：每次请求都重开 TCP 连接太慢 → 持久连接解决。
  • 无法识别用户：服务器认不出回头客 → Cookie 解决。
  • 重复传输：同一张图反复从源站拉，慢又费带宽 → Web 缓存 + 条件 GET 解决。

二、持久连接：别为每张图都重新握手

一个网页往往由 1 个 HTML + 几十个对象（图片、CSS、JS）组成。问题在于：每建立一次 TCP 连接，都要先花一个 RTT 做三次握手。若每个对象都新开一条连接（非持久连接 / non-persistent，HTTP/1.0 默认），代价惊人。

用 RTT 算一笔账

设一个 RTT 为往返时延。非持久连接下，取一个对象要花：1 个 RTT 建 TCP 连接 + 1 个 RTT 发请求收响应 = 2 个 RTT（外加对象本身的传输时间）。

非持久：总时延 ≈ (握手 + 请求) × N = 2·RTT·N

持久连接（persistent，HTTP/1.1 默认）只在第一次握手一次，之后所有对象复用同一条 TCP 连接，一个接一个（甚至流水线并发）地传，省掉了 N-1 次握手：

持久：总时延 ≈ 1·RTT(握手) + RTT·N(取对象)
非持久（HTTP/1.0）：每个对象都要重新握手 对象1: [握手 RTT][请求 RTT] 对象2: [握手 RTT][请求 RTT] 对象3: [握手 RTT][请求 RTT] ... 持久（HTTP/1.1）：握手一次，连接复用 握手一次: [握手 RTT] 对象1..N: [请求][请求][请求]... 复用同一条 TCP ▲ 省掉 N-1 次握手，还避开 TCP 慢启动重头再来
持久连接不只省一个 RTT。更深一层：TCP 有慢启动（slow start），新连接的拥塞窗口从很小开始，要慢慢"热身"才能跑满带宽。每次新建连接都要重头慢启动，等于每次都从低速起步。复用同一条已经"热"起来的连接，能持续享受较大的拥塞窗口——这是持久连接被低估的性能收益（联系后面传输层的拥塞控制章节）。

连接管理的演进对比

方案连接策略关键问题
HTTP/1.0 非持久一对象一连接，收完即关每对象 2 RTT + 反复慢启动，最慢
HTTP/1.1 持久一条连接串行复用，可流水线队头阻塞：前一个响应没回完，后面的排队干等
HTTP/2 多路复用一条 TCP 上多个流并发交错解决应用层队头阻塞，但仍受 TCP 层丢包阻塞
HTTP/3 (QUIC)基于 UDP，流之间彻底独立连 TCP 层队头阻塞也消除，0-RTT 恢复
常见误区：把"持久连接"和"HTTP/2 多路复用"混为一谈。HTTP/1.1 的持久连接是串行的——同一条连接上请求必须排队，前一个响应堵住后面所有请求（队头阻塞 HOL blocking）。浏览器当年被迫开 6 条并行连接来缓解。真正的"一条连接多请求并发交错"是 HTTP/2 的多路复用才做到的。面试若问"HTTP/1.1 持久连接能并发吗"，答案是不能并发，只能复用+排队。

三、Cookie：给无状态协议装上"记忆"

无状态的服务器认不出回头客。Cookie 的思路很朴素：既然服务器不记，那就让客户端替它记一张"号码牌"，每次请求都把号码牌带上，服务器凭号码牌去自己的数据库里查你是谁。

Cookie 四步曲（生活类比：超市寄存柜给你一张牌） ① 首次访问 浏览器 ──────── GET / ───────────► 服务器 ② 发号码牌 浏览器 ◄── Set-Cookie: id=1678 ─── 服务器(在DB建条目 1678→用户) ③ 之后每次 浏览器 ──── Cookie: id=1678 ──────► 服务器 ④ 认出你 浏览器 ◄──── "欢迎回来" ────────── 服务器(拿 1678 查DB)

它靠四个部件协作：响应头 Set-Cookie、请求头 Cookie、浏览器本地的 Cookie 文件、以及服务器后端数据库。真正的用户数据在服务器 DB，Cookie 里往往只是一个不透明的标识符。

报文示例 · Set-Cookie / Cookie 往返HTTP
# ① 服务器在响应里下发号码牌
HTTP/1.1 200 OK
Set-Cookie: sessionId=a3f9x; Max-Age=3600; HttpOnly; Secure; SameSite=Lax

# ③ 浏览器之后每次自动带上（同域名下）
GET /cart HTTP/1.1
Host: shop.example.com
Cookie: sessionId=a3f9x
安全陷阱，面试高频：
· HttpOnly：禁止 JS 通过 document.cookie 读取，防 XSS 窃取会话。
· Secure：只在 HTTPS 下发送，防中间人明文嗅探。
· SameSite：限制跨站请求携带 Cookie，是防 CSRF 的关键。
别把敏感信息（密码、余额）直接塞进 Cookie 明文——Cookie 存在客户端，用户可查看/篡改。正确做法是只存一个随机 sessionId，真实数据留在服务器。
串联你已会的：Session vs Token。上面这套"服务器存状态、Cookie 只存 id"就是传统 Session 机制（Java 里对应 HttpSession，Tomcat 用 JSESSIONID 这个 Cookie 来关联）。它的软肋是服务器要存状态，多台服务器就得共享 Session（Spring Session + Redis）。于是有了 JWT Token 方案：把用户信息签名后整个塞给客户端，服务器验签即可、不存状态——用"无状态"换"可水平扩展"，正好呼应第一节 HTTP 无状态的设计哲学。

四、Web 缓存：让内容离用户更近

Web 缓存（代理服务器 / CDN 的雏形）是一台站在客户端和源服务器之间的中间机器。浏览器把请求发给缓存，命中就直接返回，未命中才去源站取回并留一份副本。它同时扮演客户端（对源站）和服务器（对浏览器）两个角色。

命中(hit)：本地直接返回，不打扰源站 浏览器 ──► Web缓存 ══════════════════════► 浏览器 ✔ 快 │ │ 未命中(miss)：回源取一份，存下来再给你 └──► 源服务器 ──► 缓存(存副本) ──► 浏览器

它带来三重收益：降低响应时延（副本离用户近）、省接入链路带宽（机构出口那条昂贵的链路少跑流量）、减轻源站负载。《自顶向下》里有个经典量化：机构接入链路一旦利用率逼近 100%，排队时延会趋于无穷大；装上缓存把命中的流量拦下，接入链路利用率骤降，总时延大幅改善——比花钱升级链路带宽划算得多。

缓存无处不在，是一种通用思想。Web 缓存之于源站，正如 CPU 的 L1/L2/L3 缓存之于内存、Redis 之于 MySQL、MyBatis 二级缓存之于数据库查询。核心公式都一样：用"离得近、命中率高"的一层，挡住"离得远、代价高"的一层。判断缓存是否值得，看的都是平均访问时延 = 命中率×近端时延 + (1−命中率)×远端时延。

五、条件 GET：缓存的副本会过期，怎么保证不是旧货？

缓存最棘手的问题：源站的对象更新了，缓存里那份副本还是旧的怎么办？要么每次都回源验证（那缓存就白装了），要么直接返回可能过期的旧内容（用户看到脏数据）。条件 GET（Conditional GET）给出第三条路：只在对象真的变了时才传输内容，没变就只回一个空壳。

机制靠一对头字段配合：源站给对象打上"版本标记"，缓存下次带着这个标记去问"我这份还新鲜吗？"。

方向头字段含义
源站 → 缓存Last-Modified对象最后修改时间（打标记）
源站 → 缓存ETag对象内容指纹/版本号（更精确）
缓存 → 源站If-Modified-Since"这个时间之后改过吗？"
缓存 → 源站If-None-Match"内容指纹还是这个吗？"
条件 GET · 一问一答HTTP
# 缓存拿着上次记下的时间去"问"源站
GET /logo.png HTTP/1.1
Host: example.com
If-Modified-Since: Wed, 09 Jul 2025 10:00:00 GMT

# 情况A：没改过 → 源站只回状态行，不带 body（省带宽！）
HTTP/1.1 304 Not Modified

# 情况B：改过了 → 正常 200，带上新内容和新的 Last-Modified
HTTP/1.1 200 OK
Last-Modified: Fri, 11 Jul 2025 08:30:00 GMT
...(新的图片字节)
为什么 304 是省钱的关键？304 Not Modified 的报文没有 body，只有几十字节的头。一张 500KB 的图，若没变，条件 GET 只花几十字节确认"没变"，浏览器直接用本地副本渲染。这就是"协议缓存"：不是不通信，而是用极小的验证成本，避免昂贵的重复传输。
初学者最易混：强缓存 vs 协商缓存。条件 GET（304 / Last-Modified / ETag）属于协商缓存——每次仍要发一个请求去问。而 Cache-Control: max-age=600 是强缓存——在有效期内浏览器连请求都不发，直接用本地副本（网络面板显示 200 (from disk cache)）。二者配合：先看强缓存是否过期，过期了再用条件 GET 去协商。别把"发了 304 请求"当成"没缓存"——它恰恰是缓存在起作用。

另外，ETag（内容指纹）比 Last-Modified（时间戳）更可靠：时间戳精度只到秒，1 秒内多次修改会漏判；文件内容没变只是重新生成，时间戳也会变导致误判。ETag 基于内容哈希，能精确判断"内容是否真的变了"。

六、反例 → 正例：Spring 里到底该怎么用缓存头

后端最常见的坏味道，是把动态、私密的数据也无脑允许缓存，或干脆不做任何协商缓存导致带宽浪费。看一组对比：

反例 — 私密数据被公共缓存
// 用户账单接口，却没设缓存策略
// 中间代理/CDN 可能把 A 的账单
// 缓存后返回给 B → 严重越权！
@GetMapping("/api/bill")
public Bill bill() {
    return service.current();
    // 无 Cache-Control，行为不可控
}
正例 — 显式声明缓存意图
// 私密数据：禁止任何共享缓存
@GetMapping("/api/bill")
public ResponseEntity<Bill> bill() {
  return ResponseEntity.ok()
    .cacheControl(CacheControl
        .noStore())      // 谁都别存
    .body(service.current());
}
// 静态资源：走协商缓存，交给 ETag
// Spring 的 ShallowEtagHeaderFilter
// 自动算 ETag 并回 304
联系框架源码：Spring MVC 内置 ShallowEtagHeaderFilter，它会对响应体做 MD5 生成 ETag，并自动比对请求里的 If-None-Match，命中就替你返回 304——你写的 Controller 甚至不用感知。而 Cache-Control 的语义就来自本页的 HTTP 规范（RFC 9111）。看，协议不是纸上概念，它直接决定了你 ResponseEntity 上该调哪个方法。

七、四机制协同：一次真实的图片请求

浏览器第二次打开同一页面，四机制如何联手： ┌─ 强缓存未过期? ── 是 ─► 直接用本地副本 (200 from cache，零请求) │ │否 │ ▼ │ 复用【持久连接】── 带上【Cookie】识别身份 │ │ │ ▼ 发【条件 GET】(If-None-Match: 上次的 ETag) │ │ │ ┌────┴─────┐ │ ▼ ▼ │ 304 未变 200 已变 │ (用副本) (传新内容+新ETag) └── 全程经过【Web 缓存/CDN】就近命中
一句话收束：HTTP 无状态是为可伸缩而生的取舍，四个机制各补一块短板——持久连接复用 TCP、省握手与慢启动；Cookie 用客户端号码牌给协议装记忆；Web 缓存把内容推到离用户更近处；条件 GET（304）用极小验证成本避免重复传输。记住三条判别线：持久连接 ≠ 多路复用（1.1 只复用不并发）、强缓存 ≠ 协商缓存（前者不发请求、后者发请求收 304）、Cookie 只存标识、真实状态在服务器。这四点是理解 CDN、Session/JWT、HTTP/2/3 的地基。
