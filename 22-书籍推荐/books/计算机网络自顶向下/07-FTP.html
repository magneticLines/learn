FTP：控制连接与数据连接分离

大多数应用层协议都在一条 TCP 连接上"边指挥边搬货"，FTP 偏偏拆成两条：一条专门发指令，一条专门传文件。这个"带外控制"的老设计，恰恰是理解主动/被动模式、NAT 穿透、有状态协议的最佳切入点。

一、FTP 是什么，以及那个反常识的设计

FTP（File Transfer Protocol，文件传输协议）是最早期的互联网应用层协议之一（1971 年就有雏形，RFC 959 定稿于 1985 年），干的活很朴素：让你在本地主机和远程主机之间上传 / 下载文件。它跑在 TCP 之上（要求可靠传输，一个字节都不能错）。

它最出名、也最"反常识"的一点是：一次 FTP 会话会同时用到两条独立的 TCP 连接——

  • 控制连接（control connection）：走服务器 21 端口。只跑命令和应答（登录、切目录、"我要下这个文件"），全程保持不断开。
  • 数据连接（data connection）：另开一条 TCP，只跑文件字节流本身。传完一个文件就关闭，下一个文件再新开一条。
生活类比：餐厅点菜 vs 上菜。你和服务员对话点单（"来份宫保鸡丁""结账")——这是控制连接，一直开着、信息量小、你一句我一句。而后厨把菜从传菜口一盘盘端出来——这是数据连接，只负责搬运实物，一道菜一趟。点菜的嘴和上菜的手是分开的，这就是 FTP 的核心思想：指挥通道和货运通道分离。

二、带外控制（out-of-band）：FTP 与 HTTP 的分水岭

FTP 把命令放在独立于数据之外的连接上传输，这叫带外控制（out-of-band）。与之相对，HTTP 把请求行、首部（命令性信息）和响应体（数据）塞在同一条连接里，叫带内控制（in-band）。这是两者最本质的区别之一。

FTP：两条 TCP 连接（带外控制） 端口 21（控制连接·全程不断） ┌─────────┐ ───── USER / PASS / RETR / QUIT ─────► ┌─────────┐ │ FTP │ ◄──── 220 / 331 / 150 / 226 应答码 ──── │ FTP │ │ 客户端 │ │ 服务器 │ │ │ ═════ 文件字节流（每传一个文件新开）═════ │ │ └─────────┘ 数据连接（端口 20 或协商端口） └─────────┘ HTTP：一条 TCP 连接（带内控制） ┌─────────┐ ── GET /a.jpg + 首部 + 响应体 全在一条连接 ── ┌─────────┐ │ 浏览器 │ ◄══════════════════════════════════════════ │ Web服务 │ └─────────┘ └─────────┘
为什么当年要这么设计？好处很实在：控制连接不被大文件的数据流"堵死"，所以传输途中你随时能在控制连接上发命令——比如发 ABOR 中止一个正在进行的大文件下载，或用 STAT 查询进度，而不必等它传完。指挥通道永远畅通，这是带外控制的最大红利。

三、一次真实的 FTP 会话长什么样

控制连接上跑的是可读的 ASCII 文本：客户端发命令（大写，如 USER、RETR），服务器回三位数字应答码（和 HTTP 状态码同源的设计思想）。下面是下载一个文件的完整对话，C: 是客户端发的，S: 是服务器回的：

控制连接（端口 21）上的对话ASCII 命令 / 应答码
// 建立控制连接后，服务器先打招呼
S: 220 Welcome to Example FTP
C: USER alice          // 报用户名
S: 331 Password required for alice
C: PASS s3cret         // 明文密码！（见后面陷阱）
S: 230 User alice logged in
C: TYPE I              // I=Image=二进制模式，防止文件被改坏
S: 200 Type set to I
C: PASV                // 请求进入"被动模式"（见第四节）
S: 227 Entering Passive Mode (192,168,1,10,195,80)
C: RETR book.pdf       // 我要下载这个文件
S: 150 Opening BINARY mode data connection
     ┈┈┈ 文件字节流此刻正通过"另一条"数据连接传输 ┈┈┈
S: 226 Transfer complete   // 数据连接关闭，传输完成
C: QUIT
S: 221 Goodbye

注意 227 那行的 (192,168,1,10,195,80)：前四个数是服务器 IP，后两个数编码了数据连接要连的端口。端口不是直接给一个数字，而是拆成高低两个字节：

数据端口 = p1 × 256 + p2 = 195 × 256 + 80 = 50000

客户端解析出 192.168.1.10:50000，然后主动去连这个地址建立数据连接。这个"用逗号分隔、端口拆两字节"的怪异格式，正是 FTP 历史悠久、贴着 8 位字节时代设计的痕迹。

四、主动模式 vs 被动模式：数据连接到底谁发起？

控制连接永远是客户端主动连服务器 21 端口，没有悬念。真正的高频面试点在数据连接由谁发起——这就是主动模式（PORT / Active）和被动模式（PASV / Passive）的分野。

客户端 → 控制连接发 PORT，

两条连接的方向可以一图看清（下图为被动模式，两条都由客户端向外发起）：

面试高频 + 初学者踩坑：为什么被动模式成了默认？
关键在 NAT / 防火墙对连接方向的态度是不对称的：防火墙默认放行内部主动发起的出站连接，但拦截外部主动发起的入站连接。主动模式里服务器要"反向连"客户端，这是一条入站连接，正好撞在客户端 NAT/防火墙的枪口上——于是"能登录、能看目录（走控制连接），一 ls / 下载就卡死"成了 FTP 最经典的故障现象。被动模式把两条连接都改成客户端主动向外发起，绕开了这个问题。

五、FTP 是"有状态"的，HTTP 是"无状态"的

因为控制连接全程不断，服务器会为每个会话维护状态：你是谁（已登录的账号）、当前在哪个目录（CWD 切过之后服务器记得）、传输模式是文本还是二进制。这叫有状态协议（stateful）。而 HTTP 刻意做成无状态（stateless）——每个请求都得自带全部信息，服务器不替你记上一次。

维度FTPHTTP
连接结构控制 + 数据，两条 TCP一条 TCP（可复用）
控制方式带外（out-of-band）带内（in-band）
状态有状态（记登录/当前目录）无状态（靠 Cookie/Token 补）
命令格式ASCII 命令 + 三位应答码方法 + 首部 + 状态码
默认端口21（控制）/ 20（数据）80 / 443
典型用途批量文件搬运网页 / API / 一切
串联你已经会的：无状态才好扩展。HTTP 选无状态不是偷懒，而是为了水平扩展——正因为服务器不记状态，同一个用户的两个请求可以被负载均衡打到不同机器上，随便加机器就能扩容（这也是为什么会话状态要外置到 Redis）。FTP 的有状态恰恰相反：一个会话的两条连接必须由同一台服务器处理，天然难以负载均衡。这就是"有状态 vs 无状态"这一对经典权衡在协议层面的体现，和你在设计后端服务时反复面对的取舍是同一个问题。

六、更多陷阱：明文、ASCII 模式与端口混淆

三个必须记住的坑：
① FTP 是明文协议，绝不能用于公网敏感传输。上面对话里 PASS s3cret 是明文走线路的，账号密码、文件内容都能被抓包看到。要加密请用 FTPS（FTP over TLS）或干脆用 SFTP（注意：SFTP 是 SSH 的子系统，和 FTP 毫无血缘关系，只是名字像）。
② ASCII 模式会悄悄改坏二进制文件。FTP 有 TYPE A（文本）和 TYPE I（二进制）两种模式。文本模式会自动转换换行符（\n ↔ \r\n）。用它传 .zip / .jpg / .pdf，文件会被"顺手"改几个字节，下载后打不开——这是新手最难查的诡异 bug。传非文本一律先 TYPE I。
③ 别把 20 和 21 记反。21 是控制连接（命令），20 是主动模式下数据连接的服务器源端口。被动模式下数据端口是动态协商的，不是 20。

七、Java 代码：用 Apache Commons Net 走一遍

JDK 自带的 sun.net.ftp 是内部 API 不推荐直接用，业界事实标准是 Apache Commons Net 的 FTPClient。看它如何把上面所有概念落到代码里——注意注释里标出的两个"救命"设置：

FtpDownloadDemo.javaorg.apache.commons.net.ftp.FTPClient
FTPClient ftp = new FTPClient();
ftp.connect("ftp.example.com", 21);   // ① 建立控制连接：连 21 端口

// FTP 应答码 < 400 表示成功；这里体现了"三位应答码"的判定
if (!FTPReply.isPositiveCompletion(ftp.getReplyCode())) {
    ftp.disconnect();
    throw new IOException("FTP server refused connection");
}

ftp.login("alice", "s3cret");          // ② 控制连接上发 USER/PASS（明文！）
ftp.enterLocalPassiveMode();               // ③ 救命设置 1：用被动模式穿 NAT/防火墙
ftp.setFileType(FTP.BINARY_FILE_TYPE); // ④ 救命设置 2：TYPE I，别让文本模式改坏文件

// ⑤ retrieveFile 内部会新建一条数据连接，传完这一个文件后自动关闭
try (OutputStream out = new FileOutputStream("book.pdf")) {
    boolean ok = ftp.retrieveFile("/pub/book.pdf", out);  // 对应 RETR 命令
    System.out.println(ok ? "下载完成" : "下载失败");
}

ftp.logout();      // 对应 QUIT
ftp.disconnect();  // 关闭控制连接
把概念对回代码：connect(21) 建控制连接、login 走控制连接发命令、enterLocalPassiveMode() 决定数据连接方向、retrieveFile 触发一次数据连接的新建与销毁。真实项目里 90% 的"FTP 传大文件卡住"报障，最后都归结到没开被动模式或没设二进制模式这两行——你现在已经知道为什么了。
一句话收束：FTP 的灵魂是控制连接与数据连接分离（带外控制）——一条 21 端口的控制连接全程发命令、保持有状态；文件本身走独立的数据连接，一文件一开合。数据连接由谁发起，决定了主动（PORT，服务器反连，NAT 下易卡）还是被动（PASV，客户端主动连，现代默认）。记住三组对照：带外 vs 带内（对 HTTP）、有状态 vs 无状态、主动 vs 被动——它们不只是 FTP 的知识点，更是你理解一切网络协议设计取舍的通用坐标系。
