HTTPS / TLS 握手

HTTPS 不是"另一个协议"，而是给明文的 HTTP 垫了一层 TLS。看懂那几个来回的握手报文，你就同时看懂了对称加密、非对称加密、数字证书三件密码学积木如何在一次连接里协同起来。

一、先想清楚：HTTP 明文到底怕什么

HTTP 报文在网线上是明文裸奔的。你和银行之间隔着无数台路由器、WiFi 热点、运营商设备，任何一跳都能看到、甚至改写你的报文。具体有三种攻击，正好对应密码学要守住的三个目标：

攻击后果要守住的目标靠什么解决
窃听 (sniffing)WiFi 抓包看到你的密码机密性 Confidentiality加密
篡改 (tampering)页面被注入广告/木马完整性 Integrity消息认证码 MAC / AEAD
冒充 (spoofing)假冒银行网站钓鱼身份认证 Authentication数字证书 + CA
为什么"加密"一件事解决不了全部？初学者常以为"加密了就安全了"。但假设你和一个冒充银行的中间人建立了加密通道——通道是加密的，可对面根本不是银行，你还是被骗了。所以 机密性 ≠ 安全。TLS 之所以要引入证书，正是因为"加密"只解决"别人看不懂"，解决不了"我在跟谁说话"。这三件事必须一起做，缺一不可。

二、三块积木：对称、非对称、数字证书

TLS 不是发明了新密码学，而是把三件成熟的工具拼起来。先分别理解它们的性格与短板。

1. 对称加密：快，但密钥怎么给对方？

对称加密（AES）用同一把钥匙加密和解密，速度极快、适合传大量数据。致命短板是密钥分发：你要先把这把钥匙安全地交给对方——可网线本身就不安全，你怎么在不安全的网线上安全地传一把钥匙？这就是著名的"先有鸡还是先有蛋"困境。

2. 非对称加密：解决了密钥分发，但太慢

非对称加密（RSA / ECC）有一对钥匙：公钥公开、私钥自己藏。公钥加密的只有私钥能解。于是密钥分发困境迎刃而解：对方把公钥大大方方发给你，你用公钥加密一个秘密发回去，只有握着私钥的它能解开。中间人抄到公钥也没用——公钥只能加密不能解密。代价是：非对称运算比对称慢成百上千倍，不可能拿来加密整个网页流量。

维度对称加密 (AES)非对称加密 (RSA/ECC)
密钥一把，双方共享一对，公钥/私钥
速度快（有硬件指令 AES-NI）慢 100~1000 倍
核心难题密钥如何安全分发性能差，不适合大数据
TLS 里的角色握手后加密应用数据握手时协商出对称密钥
混合加密（hybrid）是 TLS 的灵魂：用慢但能解决密钥分发的非对称加密，只做一件小事——在握手阶段安全地协商出一把对称会话密钥；之后海量的应用数据全部用快的对称加密传输。一句话：非对称负责"交换钥匙"，对称负责"干活"。这就是"为什么 TLS 既要 RSA 又要 AES"的根本答案。

3. 数字证书：怎么确认这个公钥真是银行的？

还差最后一环。中间人可以在你和银行之间，把银行的公钥换成自己的公钥——你以为在用银行的公钥，其实在用攻击者的。这叫中间人攻击（MITM）。解决办法：引入一个双方都信任的权威第三方 CA（Certificate Authority），由它给"公钥属于谁"这件事盖章背书。这个盖了章的文件就是数字证书。

面试高频陷阱：数字证书里到底装了什么？"数字签名"又是怎么防伪的？
证书 = 域名 + 服务器公钥 + 有效期 + CA 用自己私钥对上述内容哈希后的签名。验证时，浏览器用内置的 CA 公钥去验签：把证书内容重新哈希，再用 CA 公钥解开签名，两者一致就说明"这份证书确实是该 CA 签发、且没被改过"。攻击者拿不到 CA 私钥，伪造不出合法签名，也就没法把公钥偷换成自己的。注意方向：签名用私钥、验签用公钥，和"加密用公钥、解密用私钥"正好相反——很多人在这里绕晕。
证书信任链（浏览器为什么信你的证书） 根 CA 证书 ── 内置在操作系统/浏览器的"信任库"里（自签名，信任锚） │ 用根私钥签 ▼ 中间 CA 证书 ── DigiCert / Let's Encrypt 等 │ 用中间私钥签 ▼ 你的服务器证书 (leaf) ── CN/SAN = www.example.com 校验：leaf ← 中间CA ← 根CA，逐级验签直到某一环在本地信任库里 任一环签名对不上、或域名不匹配、或过期 → 浏览器报"不安全"
串联 JDK：这个"信任库"在 Java 里就是 cacerts。路径 $JAVA_HOME/lib/security/cacerts，是一个内置了几百个根 CA 公钥的 KeyStore（默认密码 changeit）。你调 HttpsURLConnection 或用 RestTemplate 访问 HTTPS 站点时，JSSE（Java Secure Socket Extension）就是拿服务器证书去 cacerts 里逐级验链。企业内网自签证书报 PKIX path building failed，本质就是这条链的根不在 cacerts 里——正确做法是 keytool -importcert 把内网根导进去，而不是关掉校验。

三、TLS 握手全过程（以 TLS 1.2 为例）

三块积木就位，现在看它们怎么在一次连接里协同。注意：TLS 跑在 TCP 之上——必须先完成 TCP 三次握手，才开始 TLS 握手。下面是 TLS 1.2 完整握手（ECDHE 密钥交换）：

TCP 三次握手 (1 RTT) ── 完成后 ↓ 才开始 TLS 客户端 服务器 │ ──── ClientHello ──────────────────────────► │ ① 我支持的 TLS 版本 / 密码套件列表 │ (随机数 ClientRandom, 支持的 cipher) │ + 一个客户端随机数 │ │ │ ◄──── ServerHello ─────────────────────────── │ ② 敲定用哪个密码套件 + 服务器随机数 │ ◄──── Certificate ─────────────────────────── │ ③ 把证书链发来（含服务器公钥） │ ◄──── ServerKeyExchange ───────────────────── │ ④ ECDHE 参数 + 用私钥签名（证明我持有私钥） │ ◄──── ServerHelloDone ─────────────────────── │ │ │ │ 【本地】验证书链 → 生成 ECDHE 密钥对 │ │ 双方各自算出相同的 pre-master → master secret │ │ │ │ ──── ClientKeyExchange ────────────────────► │ ⑤ 发客户端的 ECDHE 公开参数 │ ──── ChangeCipherSpec ─────────────────────► │ ⑥ "接下来我改用对称加密了" │ ──── Finished (加密) ──────────────────────► │ ⑦ 用会话密钥加密的握手摘要，校验没被篡改 │ │ │ ◄──── ChangeCipherSpec ────────────────────── │ ⑧ 服务器也切换 │ ◄──── Finished (加密) ─────────────────────── │ │ │ │ ═══════ 之后：对称加密的应用数据 (HTTP) ══════ │

关键在于会话密钥怎么算出来的：双方交换 ClientRandom、ServerRandom 和 ECDHE 的公开参数，各自在本地用 DH 算法算出同一个 pre-master secret，再和两个随机数一起推导出 master secret，最终派生出对称会话密钥。妙处在于：真正的密钥从没在网线上传输过，中间人抄到所有公开参数也算不出私有的那部分。

那两个 Finished 报文常被忽略，其实是完整性的最后一道锁。它把之前所有握手报文做一次摘要，用刚协商好的会话密钥加密后发给对方校验。如果中间人偷偷改过 ClientHello 里的密码套件列表（想把你降级到弱加密，即降级攻击），双方算出的握手摘要就对不上，握手直接失败。所以握手过程自身也是被保护的。

读懂那串"天书"：密码套件名到底在说什么

ServerHello 里敲定的密码套件（cipher suite），其实是把这次握手要用的每种算法都写进了名字。用 openssl s_client 抓一次握手，就能看到它：

命令行 · 用 openssl 观察真实握手shell
# 连上目标站点，看协商结果与证书链
openssl s_client -connect example.com:443 -servername example.com

# 输出里会看到类似这一行 —— 拆开读：
# TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
#  │    │     │        │       │    └─ SHA256：派生密钥/摘要用的哈希
#  │    │     │        │       └────── GCM   ：AEAD 模式，加密同时保完整性
#  │    │     │        └────────────── AES-128：真正加密应用数据的对称算法
#  │    │     └─────────────────────── RSA  ：服务器用 RSA 私钥签名证明身份
#  │    └───────────────────────────── ECDHE：临时 DH 密钥交换 → 前向保密
#  └────────────────────────────────── TLS  ：协议族
看懂这一行，等于看懂了三块积木如何分工。同一串名字里，ECDHE 管交换密钥、RSA 管身份签名、AES-128-GCM 管高速加密数据、SHA256 管派生与校验——正是前两节讲的对称 + 非对称 + 完整性各就各位。TLS 1.3 把这串精简成如 TLS_AES_128_GCM_SHA256（密钥交换固定 ECDHE、签名算法另行协商，所以名字里不再写）。

四、TLS 1.2 vs 1.3：为什么新版本快一半

上面 TLS 1.2 的完整握手要两个来回（2 RTT）才能开始传数据，加上 TCP 的 1 RTT，打开一个 HTTPS 页面前先花 3 个 RTT。TLS 1.3 把握手压到 1 RTT，还支持会话恢复时的 0-RTT。

首字节时延 = TCP(1) + TLS握手 + 请求(1)
1.2: 1+2+1 ≈ 4·RTT  |  1.3: 1+1+1 ≈ 3·RTT
维度TLS 1.2TLS 1.3
完整握手 RTT2 RTT1 RTT
会话恢复Session ID / Ticket，1 RTT0-RTT（首个数据随 ClientHello 发出）
密钥交换RSA 或 ECDHE 都允许只保留 ECDHE（强制前向保密）
密码套件几百种，含不安全的老算法精简到 5 种，全是 AEAD
降级/弱算法RC4、SHA1、CBC 等历史包袱一律移除

TLS 1.3 的核心提速技巧：客户端在 ClientHello 里就把 ECDHE 公开参数（key_share）一起带上，赌服务器会选它支持的曲线；服务器回 ServerHello 时也带上自己的 key_share。这样一个来回双方就都凑齐了算密钥的材料，省掉了 1.2 里专门交换密钥的那个 RTT。

0-RTT 虽快，却有"重放攻击"隐患，别乱用。0-RTT 数据在握手完成前就发出，攻击者可以录下这段密文原样重发。若这条请求是"转账 100 元"这种非幂等操作，重放会造成重复扣款。所以规范要求 0-RTT 只用于幂等的 GET，写操作必须走完整握手——这和你在 HTTP/RESTful 里学的"GET 幂等、POST 非幂等"是同一套安全直觉。

五、前向保密（PFS）：为什么 1.3 干脆抛弃 RSA 密钥交换

TLS 1.2 允许一种老式的 RSA 密钥交换：客户端直接用服务器证书里的公钥加密 pre-master secret 发过去。它有个致命隐患：假设攻击者今天把你的加密流量全录下来，几年后设法拿到了服务器私钥（比如服务器被入侵、或 Heartbleed 这类漏洞），他就能回溯解密所有历史流量——因为所有会话密钥都是用那把长期私钥保护的。

ECDHE（临时椭圆曲线 DH）解决了这个问题：每次握手都临时生成一对用完即弃的密钥对，会话密钥由临时密钥算出，握手结束就销毁。服务器的长期私钥只用来签名（证明身份），不参与算会话密钥。于是即便长期私钥日后泄露，也倒推不出任何一次历史会话密钥。这就是前向保密（Forward Secrecy）。

无前向保密 — RSA 密钥交换
会话密钥 = f(pre_master)
pre_master 用长期私钥保护

私钥一旦泄露
 → 录下的所有历史流量
   全部可被回溯解密 ✗
有前向保密 — ECDHE
每次握手临时生成 DH 密钥对
会话密钥 = f(临时私钥)
用完即销毁

长期私钥只用于签名
 → 泄露也解不了历史流量 ✓
联系操作系统与 JDK：临时密钥的"随机"从哪来？ECDHE 每次要生成不可预测的临时私钥，依赖高质量的随机数。Java 里是 SecureRandom，底层在 Linux 上取自内核熵池 /dev/urandom（由硬件中断时序、时钟抖动等熵源喂养）。历史上有 CVE 就是因为随机数可预测导致密钥被猜出。这也是为什么早年容器/虚拟机里熵不足会让 SecureRandom.getInstanceStrong() 阻塞卡住——TLS 的安全性最终扎根在操作系统的随机性上。

六、代码实战：Java 里正确 vs 错误地做 TLS 校验

后端调第三方 HTTPS 接口时，最臭名昭著的坏味道就是：一遇到证书报错，就上网抄一段"信任所有证书"的 TrustManager 糊上去。这等于亲手把 TLS 的身份认证关掉，加密还在，但中间人可以随意冒充——机密性看似还在，安全性已归零。

反例 — 信任一切，等于裸奔
// 抄来的"解决证书报错"代码
TrustManager[] all = { new X509TrustManager(){
  public void checkServerTrusted(
      X509Certificate[] c, String a){
    // 空实现 = 任何证书都放行 ✗
  }
  // getAcceptedIssuers 返回 null...
}};
ctx.init(null, all, new SecureRandom());
// MITM 可任意冒充，钓鱼站畅通无阻
正例 — 用默认信任库，需要时导入
// 直接用 JDK 默认校验（读 cacerts）
HttpClient client = HttpClient.newHttpClient();
HttpResponse<String> r = client.send(
    HttpRequest.newBuilder()
      .uri(URI.create("https://api.bank.com"))
      .build(),
    BodyHandlers.ofString());
// 内网自签证书？把根证书导入信任库：
// keytool -importcert -alias corp \
//   -file corp-root.crt -keystore cacerts
串联 Spring：给自己的服务开 HTTPS 只需几行配置。Spring Boot 里 server.ssl.key-store=classpath:keystore.p12、server.ssl.key-store-type=PKCS12、server.ssl.key-store-password=... 即可，底层由内嵌 Tomcat 通过 JSSE 完成 TLS 握手。生产环境更常见的是把 TLS 卸载（offload）在 Nginx / 负载均衡 / 网关上——外网到网关走 HTTPS，网关到内部服务走明文 HTTP，减轻应用的握手开销。这也解释了为什么后端拿到的 request.getScheme() 常是 http，要靠 X-Forwarded-Proto 头还原真实协议。

七、呼应第 8 章：HTTPS 只是密码学的一个应用

本页看似讲"网络"，其实每一步都在用第 8 章网络安全的原语。把它们对齐一下，你会发现 TLS 就是密码学的一次"综合应用题"：

第 8 章密码学原语在 TLS 握手里的位置
对称加密（AES-GCM）握手后加密应用数据，AEAD 同时保完整性
非对称加密 / 签名（RSA、ECDSA）证书验签、ServerKeyExchange 签名
Diffie-Hellman 密钥交换（ECDHE）不传密钥就协商出会话密钥、实现前向保密
哈希 / MAC（SHA-256、HMAC）证书指纹、Finished 摘要、密钥派生
数字证书 / PKI 信任链把公钥绑定到域名、抵御中间人
一句话收束：HTTPS = HTTP + TLS，而 TLS 握手的全部意义，就是在不安全的信道上安全地协商出一把对称会话密钥。它用三块积木各司其职——非对称加密/DH 负责在明网上交换密钥、数字证书 负责确认"对面真是它"、对称加密 负责之后高速传数据。记住三条判别线：加密解决机密性，证书解决身份认证，二者缺一不可；加密用公钥、验签用公钥而签名用私钥（方向别搞反）；ECDHE 带来前向保密，所以 TLS 1.3 抛弃了 RSA 密钥交换。把这张握手图刻进脑子，第 8 章的密码学就有了落地的抓手。
