应用层架构：CS 与 P2P

写一个网络应用，第一个要拍板的不是用什么协议，而是"进程之间怎么摆位置"——谁固定在那儿等着被找、谁负责去找别人。这就是应用架构，两条主线：C/S 与 P2P。

一、先分清：应用架构 ≠ 网络核心架构

应用层架构（application architecture）说的是应用程序自己怎么组织，是应用开发者说了算的东西——它决定"程序运行在哪些端系统上、这些进程之间怎么分工通信"。它不是网络核心（路由器/链路）的结构，那部分对应用是透明的、由 ISP 固定好了，你改不了。

先建立直觉（生活类比）：
· C/S（客户-服务器）像去饭店吃饭——饭店有固定地址、常年营业（永远在线、有固定 IP），你（客户）主动上门点单，饭店做好端给你。客人之间从不直接交易，全都通过饭店。
· P2P（对等）像小区邻里换闲置——没有中心店铺，每家既是买家也是卖家，谁有东西谁就直接给需要的人，人越多，能互相调剂的东西也越多。

无论哪种架构，通信的本质都一样：是进程与进程在通信，靠 Socket 收发报文（下一节讲）。C/S 和 P2P 的区别，只是"哪个进程发起、哪个进程常驻等待"的角色分工不同。

二、C/S 架构：一个永远在线的中心

客户-服务器（Client-Server）架构里有一台（或一组）服务器，它有三个硬性特征：永远在线、有固定且众所周知的 IP 地址、被动等待连接。客户端们主动发起请求，客户之间从不直接通信——所有交互都经服务器中转。你熟悉的 Web、邮件、Spring Boot 后端、数据库，全是 C/S。

C/S（星型）：所有客户端都指向中心服务器 C1 ┌─ 特征 ─────────────────┐ \ │ · 服务器永远在线 │ C2 ─── [ Server ] ─── C3 │ · 固定/公开 IP │ / \ │ · 客户之间不直接通信 │ C4 C5 │ · 负载全压在服务器上行 │ ↑ 瓶颈：服务器的上行带宽 u_s └────────────────────────┘
高频面试陷阱：C/S 不等于"单台机器"。一台服务器扛不住海量请求（想想双十一），所以现实中的"服务器"往往是一个塞满成千上万台主机的数据中心（data center），前面挂负载均衡把请求分发下去。但从架构上看它仍是 C/S——对客户端而言，背后是 1 台还是 1 万台是透明的，客户之间依旧不直接对话。别把"物理上一台机器"和"逻辑上的服务器角色"混为一谈。

最小可读示例：一个 Java C/S 回声服务器

用 JDK 的 ServerSocket/Socket 就能看清 C/S 的角色分工：服务器 bind 到固定端口后 accept() 阻塞等待（"永远在线、被动等"），客户端主动 connect（"发起方"）。

EchoServer.java — 服务器：被动等待Java
public class EchoServer {
    public static void main(String[] args) throws IOException {
        // 绑定到众所周知的端口，这就是"固定地址"的体现
        ServerSocket server = new ServerSocket(9000);
        ExecutorService pool = Executors.newFixedThreadPool(16);
        while (true) {
            // 永远在线：阻塞等待客户主动上门，绝不主动找客户
            Socket client = server.accept();
            pool.submit(() -> echo(client)); // 一台服务器服务多个客户
        }
    }
}
EchoClient.java — 客户端：主动发起Java
// 客户端必须知道服务器的 IP+端口（众所周知）才能连
try (Socket s = new Socket("api.example.com", 9000)) {
    s.getOutputStream().write("hello".getBytes());
    // 客户端可以没有固定 IP、可以随时下线，服务器却不行
}
和你已学的连起来：服务器里那个 newFixedThreadPool(16) + accept 循环，正是"少量资源统计复用给大量请求"——和上一章"分组交换/线程池"是同一套思想。Spring MVC 的 DispatcherServlet、Tomcat 的连接器，本质就是把这段 accept→线程池→handle 的骨架产品化了。

三、P2P 架构：没有中心，人人都是服务器

对等（Peer-to-Peer）架构不依赖（或极少依赖）永远在线的中心服务器。任意成对的、间歇连接的主机（称为 peer / 对等方）直接通信。每个 peer 既向别人要数据（当客户端），又给别人供数据（当服务器）——身份是双重的。BitTorrent、区块链的节点网络、早期的电驴/迅雷，都是 P2P。

P2P 最迷人的性质：自扩展性（self-scalability）。每新增一个 peer，它在增加下载需求的同时，也贡献出自己的上传能力。用户越多，系统的总服务能力反而跟着一起涨——这是 C/S 做不到的：C/S 里用户越多，只会把中心服务器压得越狠。

用一个公式看穿"自扩展性"

《自顶向下》给了一个经典模型：一个大小为 F 的文件，要分发给 N 个用户。服务器上行速率 u_s，第 i 个用户的下载速率 d_i、上传速率 u_i，记 d_min 为最慢的下载速率。理论最短分发时间：

DC/S = max( N·F / u_s , F / d_min )

C/S 里，分子上那个 N·F/u_s 随 N 线性增长——服务器要把整个文件"亲手"发 N 份，N 翻倍，时间就翻倍，中心上行带宽是死结。再看 P2P：

DP2P = max( F / u_s , F / d_min , N·F / (u_s + Σu_i) )

关键在第三项的分母：总供给不再只有 u_s，而是 u_s + Σu_i——把所有 peer 的上传能力都加了进来。N 增大时，分子 N·F 变大，但分母里的 Σu_i 也同时变大，两者抵消，分发时间不会像 C/S 那样被 N 拖垮。这就是自扩展性的数学本质。

直觉验证：假设 100 个人下同一个 10GB 游戏。C/S 下，服务器得吐出 100×10GB=1000GB，纯靠它的上行硬扛。P2P 下，前面下好片段的人立刻变成"临时服务器"帮后来者供货，"发货窗口"从 1 个变成上百个——这正是游戏平台、Linux 发行版镜像早年爱用 BT 分发的原因。

四、正面对比：一张表看清取舍

维度C/S（客户-服务器）P2P（对等）
是否需要常驻中心必须，服务器永远在线不需要（或仅少量辅助节点）
进程角色角色固定：client / server 分明角色双重：既是 client 又是 server
扩展性差，用户越多中心越吃紧好，自扩展，越多越强
基础设施成本高，需买/租大量服务器与带宽低，蹭用户自己的资源
可管理性 / 安全易，集中管控、易审计难，节点分散、难监管、安全风险高
可用性中心是单点故障（SPOF）无单点，抗毁性强，但节点飘忽
NAT / 连通性简单，客户只需能出网难，peer 多在 NAT 后，需打洞
典型代表Web、邮件、Spring Boot、数据库BitTorrent、区块链、迅雷

看清取舍后你会发现：这不是"谁更先进"，而是集中的可控性 vs 分散的可扩展性之间的权衡。要强一致、要审计、要按用户收费 → C/S；要海量分发、要抗审查、要压成本 → P2P。

五、设计取舍的代码化：负载压向哪里

把两种架构的"心智模型"落到代码/伪码上，取舍一目了然——它们的差别就是"负载压在中心，还是摊给每个节点"。

纯 C/S 心智 — 中心扛全部
// N 个下载者全向 server 要完整文件
// server 上行成瓶颈，N↑ → 时间线性↑
for (Peer u : users) {          // N 份
    server.upload(fullFile, u);   // 全从中心出
}
// 供给能力恒为 u_s，与 N 无关 → 不扩展
P2P 心智 — 人人贡献上行
// 文件切块，下好一块就转发给别人
for (Chunk c : file.chunks()) {
    Peer src = swarm.whoHas(c); // 任意持有者
    me.download(c, src);
    swarm.announce(me, c);      // 我也成了货源
}
// 供给 = u_s + Σu_i，随 N 一起长 → 自扩展
别把 P2P 当银弹（常见误区）：
① P2P ≠ 完全无服务器。 BitTorrent 仍需 Tracker（或 DHT）帮 peer 互相发现，早期 Skype 也有超级节点做协调——纯粹"零中心"极难落地。
② 上行往往是短板。 家用宽带上行远小于下行，Σu_i 没想象中大；BT 靠"多下必须多传"的激励机制（tit-for-tat）逼大家上传，否则人人只下不传，系统就垮了。
③ NAT 穿透是硬骨头。 大多数 peer 藏在家用路由器 NAT 后没有公网 IP，两个 peer 直连要"打洞"（STUN/TURN），这也是 WebRTC 要费大力气解决的问题。

六、现实里几乎都是"混合架构"

纯 C/S 和纯 P2P 是两个极端，工业界更常见的是混合（hybrid）：用一个轻量中心做"发现/信令/元数据"，用 P2P 做"重活/数据传输"。

  • 即时通讯：用中心服务器管理"谁在线、谁的地址"（目录服务），但两人的语音/视频尽量走 P2P 直连省服务器带宽。
  • BitTorrent：Tracker（中心，轻）负责撮合，文件块传输（重）走 peer 之间。
  • CDN：可看作 C/S 的"分布式加强版"——不是 P2P，但同样在用"把内容推到离用户更近的边缘、分散中心压力"的思路。
串一条大线："预留 vs 共享""集中 vs 分散"这对张力贯穿全栈——网络核心的电路交换 vs 分组交换、这里的 C/S vs P2P、后端的单体 vs 微服务、存储的主从 vs 去中心账本。学架构，学的就是在这条谱系上，为你的约束（成本/一致性/扩展性/可控性）挑一个落点。
一句话收束：C/S = 有永远在线、地址固定的中心，客户只跟中心说话——好管控、易一致，但中心是瓶颈也是单点，用户越多越吃紧；P2P = 无中心，peer 身份双重、直接互通，靠 u_s + Σu_i 实现自扩展——省成本、抗毁强，但难管控、受上行与 NAT 拖累。记住三把钥匙：角色分工（谁常驻）、自扩展性（供给随 N 增长）、混合架构（中心做发现、P2P 做传输）。
