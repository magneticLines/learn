TCP：头部、序号确认、流量控制、连接管理

上一节的 rdt 只是"原理机"。TCP 把这些原理落到一张 20 字节的头部里，靠序号/确认号在字节流上做可靠传输，用接收窗口做流量控制，再用三次握手、四次挥手把连接稳稳地开与关。

一、先立人设：TCP 到底是一条什么样的"管道"

上一节我们从零推导出 rdt 协议家族（校验和、序号、ACK、定时器、滑动窗口）。TCP 就是把这些原理工程化落地的成品。在拆头部之前，先记住 TCP 的五个基本性格，后面每个机制都是为了兑现这些承诺：

性格含义由谁兑现
面向连接收发数据前必须先握手建立状态三次握手 / 四次挥手
可靠有序不丢、不重、不乱序，按发送顺序交付序号 + 累积确认 + 重传
字节流不保留"消息边界"，只是一串有序字节序号按字节编号
全双工同一连接双向同时收发每方向各有独立 seq/ack
流量 / 拥塞控制不压垮接收方，也不压垮网络rwnd（本节） / cwnd（下节）
为什么强调"字节流"？这是 TCP 和 UDP 最本质的分野，也是无数 Java 网络 bug 的源头。UDP 是数据报——你 send 一次，对方就 recv 到完整一坨，边界天然对齐。TCP 眼里根本没有"一条消息"这个概念，它只管把你交下来的字节按顺序、编上号、可靠地送到对面，至于"哪几个字节算一条业务消息"是应用层自己的事。这个设计换来了极高的灵活性和吞吐，代价就是后面第四节的"粘包/半包"陷阱。

二、TCP 段结构：一张 20 字节的头部撑起所有能力

TCP 报文段 = 首部（至少 20 字节）+ 数据。不用背每一位（书里也反复说"这不是 RFC 手册"），但要看懂每个字段解决什么问题——它们和上一节的 rdt 要素一一对应：

0 15 16 31 ← 位偏移 +---------------------+---------------------+ | 源端口 (16) | 目的端口 (16) | ← 多路复用/分解(上上节) +---------------------------------------------+ | 序号 seq (32) | ← 本段首字节的字节流编号 +---------------------------------------------+ | 确认号 ack (32) | ← 期望收到的下一个字节号 +----+-----+----------+---------------------+ |首长|保留 |U A P R S F| 接收窗口 rwnd (16) | ← 流量控制 +----+-----+----------+---------------------+ | 校验和 (16) | 紧急指针 (16) | ← 差错检测 +---------------------+---------------------+ | 选项 (变长, 如 MSS/时间戳/窗口扩大) | +---------------------------------------------+ | 数据 (变长) | +---------------------------------------------+ 标志位: URG ACK PSH RST SYN FIN

几个字段值得单独点名，因为它们直接决定连接的行为：

字段作用面试/实战要点
seq 序号本段第一个字节在整条字节流中的编号不是"第几个报文段"，是字节编号
ack 确认号接收方期望收到的下一个字节的序号累积确认；只在 ACK=1 时有效
首部长度首部有多少个 4 字节字（因选项变长）4 位 → 首部最大 15×4=60 字节
rwnd接收方还能收多少字节（剩余缓冲）流量控制核心，仅 16 位 → 需窗口扩大选项
标志位SYN 建连、FIN 拆连、RST 强制复位、ACK 确认有效、PSH 尽快上交、URG 紧急SYN/FIN 各占 1 个序号！
MSS(选项)本方向单段最大数据字节数握手时协商，常见 1460 = 1500−20(IP)−20(TCP)
陷阱：MSS ≠ MTU，SYN/FIN 会"吃掉"一个序号。MTU 是链路层一帧能装的最大字节数（以太网 1500），MSS 只算 TCP 数据部分（1500 减去 IP 头 20、TCP 头 20 = 1460）。另外一个最容易错的点：SYN 和 FIN 虽然不带数据，却各自消耗一个序号——这就是为什么握手时对方回的 ack 是 client_isn + 1 而不是 client_isn。这个 "+1" 几乎是三次握手题的必考点。

三、序号与确认号：把可靠性建立在"字节流"上

rdt 里序号是给"分组"编的，而 TCP 给每一个字节编号。一个报文段的 seq = 它携带的第一个字节的编号；ack = 我已经全部收齐、正等着的下一个字节的编号。起始编号 ISN 不是从 0 开始，而是随机选取（防止旧连接的残包被误认，也提高安全性）。

某段 seq = 本段首字节序号    ack = 已收齐字节数 + 起始序号

用一个经典例子建立直觉

《自顶向下》里的 Telnet 例子最能说明"确认号"的含义。主机 A 敲一个字符 'C'，B 回显这个 'C'，全程都是捎带确认（piggyback）——把 ACK 塞进反方向的数据段里，省一个报文：

主机A(seq起点42) 主机B(seq起点79) │ seq=42, ack=79, data='C' (发1字节) │ ├───────────────────────────────────────►│ 收到第42字节 │ │ │ seq=79, ack=43, data='C' (回显+确认) │ ← ack=43: "42收到,等43" │◄───────────────────────────────────────┤ │ │ │ seq=43, ack=80 (纯ACK) │ ← ack=80: "79收到,等80" ├───────────────────────────────────────►│

注意第二段：B 既回显数据（seq=79），又确认了 A 的字节（ack=43），一石二鸟。这就是全双工 + 捎带确认。

累积确认（cumulative ACK）是 TCP 的灵魂。ack=n 的真正含义是"第 n 之前的所有字节我都收齐了"，而不仅仅是"收到了第 n−1 个"。好处：即使中间几个 ACK 丢了，只要后一个更大的 ACK 到达，发送方就知道前面全都到了——ACK 天然带"补丁"。这也解释了下一节"快速重传"里 3 个重复 ACK 的现象：接收方每收到一个失序段，就重复回同一个 ack（还在等那个缺口），发送方连着收到 3 个重复 ack 就明白"那个字节丢了"，不等超时立刻重传。
高频错点：ack 是"期望下一个"，不是"最后收到的"。很多人把 ack 写成"已收到的最后一个字节号"，差一位。正确记法：ack = 我下一步想要的字节。收到了第 42 字节（1 字节数据），就回 ack=43。三次握手里收到 SYN(seq=x) 回 ack=x+1 也是同一个逻辑（SYN 占 1 号）。

四、字节流的代价：粘包 / 半包（Java 必踩坑）

既然 TCP 不保留消息边界，那么"你写了几次"和"对方读到几次"完全没有对应关系。发送方 write 两条消息，可能被合并进一个段发出（粘包）；一条大消息也可能被拆到两个段（半包）。这不是 bug，是 TCP 的本性——它只保证字节有序，不保证"按你写的次数切好"。

反例 — 以为一次 read = 一条消息
InputStream in = socket.getInputStream();
byte[] buf = new byte[1024];
int n = in.read(buf);
// n 可能只是半条,也可能粘了两条
String msg = new String(buf, 0, n);
handle(msg);   // 消息边界全乱 ✗
正例 — 自定义"长度前缀"帧
DataInputStream in = new DataInputStream(
        socket.getInputStream());
int len = in.readInt();   // 先读4字节长度
byte[] body = new byte[len];
in.readFully(body);       // 读满一整帧
handle(new String(body)); // 边界清晰 ✓
串联 Netty：帧解码器就是干这个的。手写 readInt + readFully 太原始，实际生产用 Netty 的 LengthFieldBasedFrameDecoder（长度字段法）或 LineBasedFrameDecoder（分隔符法）来自动"切帧"。它们把"从无边界字节流里还原业务消息"这件事沉淀成了可复用组件——你在本项目 Netty 模块学到的 ByteToMessageDecoder，本质就是在对抗这一节讲的"TCP 是字节流"。三种切帧方案的取舍：固定长度（简单但浪费）、分隔符（如 \n，但内容不能含分隔符）、长度前缀（最通用，HTTP/2、Dubbo、gRPC 都用它）。

五、流量控制：别把接收方的缓冲区撑爆

接收方内核为每条连接维护一个接收缓冲区（RcvBuffer）。数据先进缓冲区，等应用层调 read() 才被取走。如果发送方发得太快、应用层读得太慢，缓冲区就会溢出、丢数据。流量控制就是让发送方"看接收方脸色行事"的机制——速度匹配服务（speed-matching）。

做法很朴素：接收方在每个段的 rwnd 字段里实时广播"我还能收多少"，发送方保证"已发出但没被确认的数据量"不超过这个数。

rwnd = RcvBuffer − (LastByteRcvd − LastByteRead)
发送方约束：LastByteSent − LastByteAcked ≤ rwnd
接收方缓冲区 (RcvBuffer) ┌───────────────┬──────────────┬───────────────┐ │ 已被应用读走 │ 已收未读(排队) │ 空闲 = rwnd │ └───────────────┴──────────────┴───────────────┘ LastByteRead LastByteRcvd 应用read()慢 → 中间段变大 → rwnd变小 → 广播给发送方 应用read()快 → 空闲变大 → rwnd变大 → 发送方可多发
经典陷阱：rwnd=0 后的死锁，靠"持续计时器"破解。当接收方缓冲满了，广播 rwnd=0，发送方停发。问题来了：此后接收方即使把数据读走、腾出了空间，它没有数据要发（比如是纯下载场景），也就没机会捎带一个"rwnd 变大了"的通知——双方就此僵死。TCP 的解法：发送方在收到 rwnd=0 后启动 持续计时器（persist timer），定期发送只含 1 字节的零窗口探测（window probe）去戳接收方，逼它回一个带最新 rwnd 的 ACK。这题很爱考。
别把流量控制和拥塞控制搞混（下一节主角）。两者都限制发送速率，但"为谁踩刹车"完全不同：
维度流量控制（本节）拥塞控制（下节）
防止压垮谁接收方的缓冲区网络（中间路由器队列）
限制变量接收窗口 rwnd拥塞窗口 cwnd
信息来源接收方在头部里明说发送方靠丢包/RTT推测
是端到端还是全局纯两端协商所有连接共同影响

实际发送方的窗口是两者取小：发送窗口 = min(rwnd, cwnd)。谁更紧张听谁的。

六、连接管理（上）：三次握手为什么是"三"次

建连的目标：让双方都确认对方能收能发，并交换各自的初始序号 ISN（后续所有 seq/ack 都以它为基准）。三次握手正好是达成这个目标的最小步数：

客户端 服务端 CLOSED LISTEN │ ① SYN=1, seq=x │ ├──────────────────────────────────────────►│ SYN_RCVD SYN_SENT │ │ ② SYN=1, ACK=1, seq=y, ack=x+1 │ │◄──────────────────────────────────────────┤ ESTABLISHED │ │ ③ ACK=1, seq=x+1, ack=y+1 (可带数据) │ ├──────────────────────────────────────────►│ ESTABLISHED │ === 连接建立,开始传数据 === │

关键观察：② 的 ack=x+1 证明"服务端收到了客户端的 SYN"（还证明服务端能收）；③ 的 ack=y+1 证明"客户端收到了服务端的 SYN"。到第三步，双向的收发能力和双向的 ISN 才全部被确认。

为什么不能两次握手？核心是防止历史失效连接（旧的重复 SYN）建立假连接。设想客户端一个很早发出、在网络里滞留很久的旧 SYN，如果只要两次握手（服务端收到 SYN 就直接建连并开始发数据），这个旧 SYN 到达时服务端会误建一条连接、白白占资源。有了第三次握手，服务端发完 SYN-ACK 后要等客户端最终确认——客户端发现这是过期的旧连接，会回 RST 拒绝，连接建不起来。一句话：第三次握手是为了让客户端有机会否决一个过期的连接请求。
SYN Flood 攻击 —— 高频安全考点。攻击者伪造大量源 IP 只发 ① SYN，服务端回了 ② SYN-ACK 后进入 SYN_RCVD，把连接放进半连接队列等第三次 ACK——但第三次永远不来。队列被塞满后，正常用户的 SYN 就被丢弃（拒绝服务）。防御：SYN Cookies——服务端在 SYN_RCVD 阶段不分配资源，而是把连接状态编码进 SYN-ACK 的初始序号里，等真正的第三次 ACK 回来再校验并建连。

在 Java 里，这一切被 Socket/ServerSocket 封装得几乎透明——但要知道每行代码触发了哪一步握手：

Java Socket 与三次握手的对应Java
// 服务端: bind + listen, 进入 LISTEN 态
ServerSocket server = new ServerSocket(8080, 128);
//                          端口↑        ↑backlog=已完成握手队列长度

// accept() 从"已完成三次握手"的队列里取一条连接
// 若队列空则阻塞;握手是内核在后台完成的,不是 accept 触发的
Socket conn = server.accept();

// 客户端: new Socket() 内部就在做①②③三次握手
// 构造函数返回时,连接已 ESTABLISHED
Socket s = new Socket("10.0.0.5", 8080);
s.getOutputStream().write(data);  // 此时才真正传业务数据
联系操作系统：backlog 其实是"两个队列"。内核为每个监听 socket 维护半连接队列（SYN 队列，收到 ① 后）和全连接队列（accept 队列，完成 ③ 后）。new ServerSocket(port, backlog) 的 backlog 影响的是全连接队列。如果应用 accept() 太慢，全连接队列被塞满，新完成握手的连接会被丢弃或触发重传——线上高并发服务如果看到大量连接超时，backlog 和 accept 处理速度是必查项。

七、连接管理（下）：四次挥手与 TIME_WAIT

关闭比建立多一步，因为 TCP 是全双工：每个方向都要单独关闭。一方发 FIN 只表示"我这边没数据要发了"，但它仍可继续接收对方的数据（半关闭）。于是 SYN 能合并成三次，FIN 却要四次：

主动关闭方(如客户端) 被动关闭方(如服务端) ESTABLISHED ESTABLISHED │ ① FIN=1, seq=u │ ├──────────────────────────────────────►│ CLOSE_WAIT ← 收到FIN,通知应用 FIN_WAIT_1 │ (对方可能还有数据要发完) │ ② ACK, ack=u+1 │ │◄──────────────────────────────────────┤ FIN_WAIT_2 │ │ ③ FIN=1, seq=w (应用调close后) │ │◄──────────────────────────────────────┤ LAST_ACK TIME_WAIT │ │ ④ ACK, ack=w+1 │ ├──────────────────────────────────────►│ CLOSED │ 等待 2·MSL 后 → CLOSED │

为什么 ② 和 ③ 不能合并成一步（像握手那样）？因为收到对方 FIN 时，被动方自己可能还有数据没发完——它得先立刻 ACK"我知道你要关了"，等自己的数据发完、应用主动调 close()，才发自己的 FIN。这中间的 CLOSE_WAIT 状态正是留给它"把话说完"的窗口。

生产事故 TOP 级：CLOSE_WAIT 堆积 = 你的代码忘了 close()。如果对端已关闭（你收到了 FIN、连接进入 CLOSE_WAIT），但你的应用迟迟不调 close()，第三次 FIN 就发不出去，连接会永久卡在 CLOSE_WAIT，持续占用文件描述符，最终 Too many open files 打爆服务。排查口诀：netstat 里大量 CLOSE_WAIT → 一定是本机应用没关连接（漏了 finally 里的 close）；大量 TIME_WAIT → 是本机作为主动关闭方、且短连接太多。前者是 bug，后者多是设计问题。
TIME_WAIT 为什么要等 2·MSL（MSL=报文最大存活时间）？两个原因：
① 保证最后那个 ACK（④）能到达对方：万一 ④ 丢了，对方会重发 FIN（③），此时我还在 TIME_WAIT，能再补一个 ACK。如果我直接 CLOSED 走人，对方就会一直重发 FIN、最后收到 RST 而异常关闭。
② 让本连接的旧报文在网络中彻底消亡：等 2·MSL（一去一回），确保没有"迟到的老报文"会污染下一条复用相同四元组的新连接。
用 netstat 观察 TCP 状态 + SO_REUSEADDRShell / Java
# 统计各状态连接数(排查利器)
$ netstat -an | awk '/^tcp/ {s[$6]++} END{for(k in s)print k,s[k]}'
ESTABLISHED 812
TIME_WAIT   3049      # 主动关闭方,短连接多 → 考虑长连接/连接池
CLOSE_WAIT  57        # 危险!应用漏了 close()

// Java: 服务重启时报 "Address already in use"?
// 是上次的连接还在 TIME_WAIT 占着端口。开启地址复用即可绕过:
ServerSocket server = new ServerSocket();
server.setReuseAddress(true);   // SO_REUSEADDR,必须在 bind 之前设
server.bind(new InetSocketAddress(8080));

八、收束：一图看懂 TCP 状态机主干

[主动] [被动] CLOSED ──connect─► SYN_SENT ──收SYN-ACK/发ACK─► ESTABLISHED ▲ │ │ │ listen │ │ 收FIN/发ACK └── LISTEN ──收SYN/发SYN-ACK─► SYN_RCVD ────────┘ ▼ CLOSE_WAIT 主动关闭链: │ 应用close/发FIN ESTABLISHED ─close/发FIN─► FIN_WAIT_1 ─收ACK─► ▼ FIN_WAIT_2 ─收FIN/发ACK─► TIME_WAIT ─2MSL─► LAST_ACK │ │ 收ACK └──► CLOSED ◄──┘
一句话收束：TCP 把上一节的 rdt 原理压进一张 20 字节头部——序号/确认号让可靠性建立在"字节流"上（累积确认 ack=期望的下一个字节，SYN/FIN 各占 1 号）；rwnd 做流量控制，防止压垮接收方（下节 cwnd 防止压垮网络，发送窗口取二者小值）；三次握手用最小步数交换双向 ISN 并否决过期连接，四次挥手 + TIME_WAIT(2MSL) 因全双工而拆两半、并为可靠关闭兜底。记牢三条实战线：字节流 → 粘包要自己切帧、CLOSE_WAIT 堆积是你没 close()、流量控制≠拥塞控制。这三点是理解 Netty、Socket 编程和线上网络故障的地基。
