分组交换 vs 电路交换

同一根链路，如何让成千上万人共享？两种答案代表了两种世界观——"先占座后开跑"的电路交换，和"随到随发、按需拼车"的分组交换。互联网选了后者，本页讲清这个选择背后的取舍。

一、它们要解决的同一个问题：如何共享链路

网络核心是一堆链路被一堆交换设备（路由器 / 交换机）串起来。核心问题只有一个：一条物理链路带宽有限，怎么让多个用户/会话共用它？"交换方式"就是对这个问题的两种根本不同的回答。

先建立直觉（生活类比）：
· 电路交换像打传统电话 / 包场餐厅——通话前先建立一条专属线路，从头到尾这条线路只归你，哪怕你俩都在沉默不说话，线路也被你占着，别人用不了。
· 分组交换像寄快递 / 拼车走高速——你把货拆成一箱箱（分组）贴上地址扔进物流网，路上和别人的包裹混着走、共用车道，谁的货多谁那段就多占点，没有"专线"这回事。

二、电路交换：先预留资源，再通信

电路交换（Circuit Switching）的核心动作是：通信前，沿途每一段链路都为这次会话预留一份固定的带宽，建立一条端到端的"电路"。会话期间这份带宽专属独享，用不用都锁定；会话结束才释放。这是传统电话网（PSTN）的工作方式。

一条链路怎么"切"成多份预留给不同电路？两种复用方式：

复用方式怎么分类比
FDM 频分复用把链路总频带切成若干频段，每条电路独占一个频段，同时并行传广播电台：不同电台占不同频率，同时播
TDM 时分复用把时间切成周期性的时隙，每条电路每个周期分到固定的一格轮流发言：每人每轮固定一个时间窗
电路交换：通信前先"三次握手式"地建立专属电路，全程独占 预留的专属带宽(全程锁定) 主机A ──[预留]── R1 ──[预留]── R2 ──[预留]── 主机B │ │ ├─ ① 建立电路(沿途预留带宽) ────────────►│ │◄──────────── ② 电路就绪 ──────────────┤ ├─ ③ 传数据(独占,时延稳定) ─────────────►│ └─ ④ 拆除电路(释放带宽) ────────────────►│
电路交换的最大优点：确定性。带宽被预留，所以只要电路建成，传输速率恒定、有保证，时延稳定、不会因为别人而变慢，也不会丢包。这对实时语音这类"宁可占着也不能卡"的业务非常合适——这也是它当年统治电话网的原因。

三、分组交换：拆成分组，存储转发，统计复用

分组交换（Packet Switching）反其道而行：不预留任何东西。发送方把报文（message）拆成一个个分组（packet），每个分组带上地址独立地丢进网络；沿途路由器对每个分组做存储转发（store-and-forward）——先把整个分组完整收下来，再根据地址转发到下一段链路。链路空着就发，谁有数据谁就用，这叫统计复用（statistical multiplexing）。

存储转发带来的时延：为什么是"整包收完再转"

路由器必须先收完整个分组才能开始往下一跳转发（要校验、要看完整的头部）。所以一个 L 比特的分组，经过速率 R 的一段链路，光"推上链路"就要花 L/R 秒。经过 N 段链路（无排队、无其他时延的理想情况）：

端到端时延 ≈ N × (L / R)  (N 段链路 · 存储转发)

举个具体的数：分组 L = 1000 比特，每段链路 R = 1 Mbps（即 10⁶ bit/s），传输时延 L/R = 1 ms；经过源→R1→R2→目的共 3 段链路，理想端到端就是 3 × 1 ms = 3 ms。注意这只是"传输时延"，真实网络还要叠加排队、处理、传播时延——下一节会专门拆解。

存储转发：整个分组"收齐"才能往下一跳吐 时间 → 链路1 [==P整包收进R1==] 链路2 [==P整包收进R2==] 链路3 [==P到达目的==] ↑收完才转发 ↑收完才转发 (这就是为什么 N 段要 N×L/R，而不是一段的时间)

杀手锏：统计复用带来的"超卖"红利

为什么互联网选分组交换？因为真实流量是突发（bursty）的——你刷网页时，大部分时间在读页面（链路空闲），只有点开一瞬间才猛发数据。电路交换给每个人预留固定带宽，等于"人走了座位还空着占位"，利用率极低。分组交换让大家共享，用统计规律"超卖"，把闲置带宽榨出来。看《自顶向下》里的经典算例：

经典算例（Kurose 教材）：一条 1 Mbps 的共享链路，每个用户活跃时占 100 kbps，且平均只有 10% 的时间在活跃。
· 电路交换：必须按峰值预留，1 Mbps ÷ 100 kbps = 最多只能支持 10 个用户，多一个都不行。
· 分组交换：可以接进 35 个用户。同时有 11 个及以上用户活跃（才会超过 1 Mbps 造成拥塞）的概率不到 0.0004。也就是说，用几乎可忽略的拥塞概率，换来了 3.5 倍的用户容量。

这就是统计复用的本质：不是所有人同时活跃，就不必给所有人同时预留。这是一种概率上的"超卖"，和航空公司超售机票、银行不为每个储户备足现金、线程池不为每个请求开一个真线程，是同一套思想。

四、代价：排队与丢包（分组交换的"阴暗面"）

天下没有免费的午餐。既然不预留，就可能出现多个分组同时想用同一条出链路的情况。路由器只能让它们在输出队列（缓冲区）里排队——这就是排队时延；如果流量持续超过链路速率，缓冲区被塞满，后来的分组只能被丢弃（丢包）。

正常转发 (队列有空) 排队等待 缓冲满→丢包
初学者高频误区，务必分清：
① "分组交换没有时延波动"——错。 恰恰相反，排队时延随负载剧烈波动，这正是分组交换换来高利用率所付的代价；电路交换才是时延稳定的那个。
② "丢包是链路出了物理故障"——多数时候不是。 互联网里绝大部分丢包是路由器缓冲区溢出造成的，属于正常拥塞现象。TCP 正是靠"检测到丢包"来判断网络拥塞并减速的（后面拥塞控制章节详讲）。
③ "存储转发 = 存到硬盘"——错。 是暂存在路由器内存缓冲区，"存储"指的是"整包收进内存再转发"，不是持久化。

五、正面对比：一张表看清取舍

维度电路交换（Circuit）分组交换（Packet）
资源分配通信前预留专属带宽不预留，按需抢占共享
复用方式FDM / TDM（固定切分）统计复用（谁有数据谁用）
建立阶段需要先建立电路（有建立时延）无需建立，分组随到随发
带宽利用率低（沉默也占着）高（榨干空闲带宽）
时延特性稳定、有保证波动大（含排队时延）
拥塞 / 丢包建成后不会拥塞、不丢包可能排队、可能丢包
容纳用户数受峰值预留限制，上限硬可"超卖"，上限软
典型代表传统电话网 PSTN互联网 / IP 网络

从设计取舍看，这是一对经典权衡：确定性 vs 利用率。电路交换用低利用率换来了确定的性能保证；分组交换用不确定的性能（可能排队丢包）换来了极高的利用率和弹性。互联网面对的是海量、突发、异构的数据流量，"利用率和弹性"压倒了"确定性"，所以选了分组交换。

六、串联你已经会的：这套思想在软件里到处都是

"预留 vs 按需共享"根本不是网络独有的。你在写 Java 后端时天天在做同一个选择：

类·电路交换 — 每连接一线程
// 为每个请求"预留"一个专属线程
// 连接空闲时线程也被占着 → 像
// 电路沉默也锁带宽，资源浪费
while (true) {
    Socket s = server.accept();
    new Thread(() -> handle(s))
        .start(); // C10K 直接压垮
}
类·分组交换 — 线程池统计复用
// 固定几个线程"共享"给大量请求
// 按需分配、忙时排队 → 正是
// 统计复用 + 排队缓冲的思想
ExecutorService pool =
    Executors.newFixedThreadPool(16);
while (true) {
    Socket s = server.accept();
    pool.submit(() -> handle(s));
}
把点连成线：
· 线程池的任务队列 = 路由器的输出缓冲区；队列满了拒绝任务（RejectedExecutionException）就是丢包；核心线程数就是"链路速率"。
· 数据库连接池（HikariCP）：不给每个请求预留一条连接，而是少量连接被大量请求统计复用——和 35 个用户共享 1 Mbps 一模一样。
· TCP 的"连接"是个错觉：TCP 让你感觉像电路交换（有连接、可靠有序），但它底下跑的仍是分组交换——所谓"连接"只是通信两端维护的一份状态，网络核心的路由器根本不知道你这条"连接"的存在（这是后续传输层的核心主题）。
一句话收束：电路交换是"先预留、后独占"——用低利用率换确定的性能；分组交换是"拆分组、存储转发、统计复用"——用可能的排队与丢包换极高的利用率和弹性。互联网因为流量突发、规模巨大，选择了分组交换。记住三个关键词：存储转发（N×L/R）、统计复用（超卖红利）、排队/丢包（红利的代价）——它们是理解后面所有时延、TCP、拥塞控制的地基。
