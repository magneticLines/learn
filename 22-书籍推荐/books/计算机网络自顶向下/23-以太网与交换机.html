以太网与交换机：自学习与 VLAN

一台交换机开箱即插即用、不用任何配置就能把帧送到对的口——它是怎么"自己学会"网络拓扑的？又如何用 VLAN 把一台物理交换机切成互不打扰的多个逻辑网络？

一、以太网帧：链路层的"信封"长什么样

以太网（Ethernet）是今天有线局域网事实上的唯一标准。它工作在链路层，负责把上层交下来的 IP 数据报，封装成能在一段物理链路上传输的帧（frame）。理解交换机之前，先看清帧的结构——尤其是那两个 MAC 地址字段，它们是交换机做一切决策的唯一依据。

┌──────────┬─────┬──────────┬──────────┬──────┬──────────────┬──────┐ │ 前导码 │ SFD │ 目的 MAC │ 源 MAC │ 类型 │ 数据载荷 │ FCS │ │ 7 字节 │ 1B │ 6 字节 │ 6 字节 │ 2B │ 46~1500 字节 │ 4B │ └──────────┴─────┴──────────┴──────────┴──────┴──────────────┴──────┘ └─ 物理层同步用 ─┘ └───────── 帧头（14 字节）─────────┘ └ CRC 校验
  • 目的 / 源 MAC：各 48 位（6 字节）。MAC 地址由网卡出厂烧录，前 24 位是 OUI（IEEE 分配给厂商的编号），后 24 位由厂商自己保证唯一——所以理论上全球网卡地址不重复。
  • 类型（Type）：标明载荷是什么上层协议，如 0x0800=IPv4、0x0806=ARP、0x86DD=IPv6。这就是上一节 ARP 能和 IP 共用一根网线的原因——靠这个字段分流。
  • FCS：4 字节 CRC 校验码，接收方重算比对，发现错误直接丢弃（不重传）——以太网只做检错，可靠性交给上层的 TCP。
MAC 地址不是随便一串数。首字节最低位是 I/G 位：0=单播，1=多播；全 1 的 FF:FF:FF:FF:FF:FF 是广播地址。ARP 请求正是发到这个广播 MAC，才能让同一网段所有主机都收到。记住这个细节，下面讲"泛洪"时你会更有画面感。

二、集线器 → 交换机：从"一起吼"到"精准送达"

要体会交换机的价值，得先看它取代的东西——集线器（Hub）。集线器是纯物理层设备，它根本不看帧内容，收到的比特无脑往所有其他口复制。结果就是：所有主机共享一个冲突域，同一时刻只能有一台在发，别人发就撞车（CSMA/CD 检测冲突后退避重发）。交换机则工作在链路层，会读MAC 地址、只往正确的口转发。

集线器 Hub — 物理层"广播喇叭"
A 发给 B 的帧：
  Hub 无脑复制到 C、D、E...

× 所有口共享 1 个冲突域
× 同一时刻只能 1 人发，撞了重来
× 半双工，带宽被全员瓜分
× C/D 能嗅探到 A→B 的所有流量
→ 8 口百兆 Hub，人均远达不到 100M
交换机 Switch — 链路层"智能邮差"
A 发给 B 的帧：
  Switch 查表，只从 B 所在口发出

√ 每个口是独立冲突域（几乎无冲突）
√ 多对主机可同时通信（A→B、C→D 并行）
√ 全双工，每口独享线速
√ C/D 默认收不到 A→B 的单播流量
→ 8 口千兆交换机，每口都是满 1G
一句话记牢分层："交换机分割冲突域，路由器分割广播域。"交换机让每个端口成为独立冲突域，从根上消灭了 Hub 时代的碰撞问题；但一台普通交换机下的所有主机仍处在同一个广播域——一个广播帧会传遍全交换机。想切分广播域，要么上路由器，要么用本节后面讲的 VLAN。

三、自学习：交换机怎么"无师自通"

交换机最妙的地方是即插即用（plug-and-play）：你不用告诉它"哪台主机接在哪个口"，它自己就能学会。秘密在于内部维护的一张 MAC 地址表（业界也叫 CAM 表），每条记录是 (MAC 地址, 端口, 时间戳)。学习规则只有一句话：帧从哪个口进来，它的"源 MAC"就一定在那个口的方向。

A(MAC-a) C(MAC-c) │端口1 │端口3 ┌──┴─────────────────────┴──┐ │ 交换机 S │ └──┬─────────────────────┬──┘ │端口2 │端口4 B(MAC-b) D(MAC-d) ① A→B 发帧：S 从"端口1"收到，记下 MAC-a→端口1； 此刻还不知道 B 在哪 → 泛洪(除端口1外全发) ② B→A 回帧：S 从"端口2"收到，记下 MAC-b→端口2； 查表知道 MAC-a 在端口1 → 只从端口1转发（不再泛洪） 学习后的 MAC 地址表： ┌──────────┬───────┬────────┐ │ MAC 地址 │ 端口 │ 时间戳 │ ├──────────┼───────┼────────┤ │ MAC-a │ 1 │ 09:32 │ │ MAC-b │ 2 │ 09:33 │ └──────────┴───────┴────────┘

拿到目的 MAC 后，交换机的转发决策只有三种结果，务必记牢这张表：

查目的 MAC 的结果动作术语
表中查不到该 MAC从除入端口外的所有端口发出泛洪 Flooding
查到，且记录端口 == 入端口直接丢弃（目的和源在同侧，无需过网）过滤 Filtering
查到，且记录端口 != 入端口只从记录的那个端口发出转发 Forwarding

把这套逻辑写成伪代码，你会发现它本质就是一个"带过期时间的哈希表 + 三分支判断"——像极了 Java 里的一个 ConcurrentHashMap：

交换机收到一帧时的核心处理逻辑伪代码 · Java 风格
// 转发表：key=目的MAC, value=(端口, 最后活跃时间)
Map<Mac, Entry> table = new ConcurrentHashMap<>();

void onFrameReceived(Frame f, Port inPort) {
    // ① 自学习：源 MAC 必在 inPort 一侧，覆盖并刷新时间戳
    table.put(f.srcMac, new Entry(inPort, now()));

    // ② 转发决策：查目的 MAC
    Entry e = table.get(f.dstMac);
    if (e == null || f.dstMac.isBroadcast()) {
        flood(f, inPort);        // 泛洪：除入端口外全发（广播也走这里）
    } else if (e.port == inPort) {
        drop(f);                 // 过滤：目的就在来的这侧，扔掉
    } else {
        forward(f, e.port);       // 转发：只从记录端口发出
    }
}

// ③ 后台老化线程：超过 aging time（Cisco 默认 300s）未刷新就删除
//    这样主机换口/关机后，陈旧表项会自动清理，避免误转发
高频面试陷阱：为什么第一个包总要泛洪？因为交换机只从"源 MAC"学习，从不主动探测。一台从没发过帧的主机（比如刚开机、纯监听），交换机根本不知道它在哪个口，发给它的帧只能泛洪。另一个坑：ARP 请求本身是广播帧，必然泛洪到全网段——这正是 ARP 能工作的前提，别把它和"交换机没学到"混为一谈。还有个易错点：交换机转发时既不修改 MAC 也不减 TTL，它对主机完全透明；会减 TTL、会换 MAC 的是路由器（三层设备）。

四、交换机 vs 路由器：别把二层和三层搞混

初学者最常混淆的就是这俩。它们都能"转发数据"，但工作在不同层、依据不同地址、解决不同问题：

维度交换机 Switch（二层）路由器 Router（三层）
工作层链路层网络层
转发依据目的 MAC 地址目的 IP 地址
转发表怎么来自学习（看源 MAC）路由协议 / 静态配置计算
是否改帧/包不改 MAC、不减 TTL（透明）重写二层 MAC、TTL−1、重算校验和
广播域默认所有口同一广播域每个接口分割广播域
典型场景连接同一子网内的主机连接不同子网 / 接入互联网
串起前几节：一次访问其实是"接力"。你 ping 一个外网 IP，主机先用 ARP 查到"默认网关"的 MAC，把帧发给网关——这一跳靠交换机按 MAC 转发；网关（路由器）拆开看 IP，决定下一跳，重写 MAC 再发出去——这一跳靠路由器按 IP 转发。二层管"同一网段内的最后一公里"，三层管"跨网段的长途路由"，二者交替接力，你在Netty 里 write 出去的每个字节都要走这一整套。

五、VLAN：把一台交换机切成多个"隔离网"

问题来了：一台 48 口交换机接了研发和财务两个部门，它们在同一广播域里——财务发个 ARP 广播，研发全体都被打扰；更糟的是任何人插上网线就能嗅探到广播、试图访问对方主机。物理上再买一台交换机隔离太浪费。VLAN（Virtual LAN，虚拟局域网）就是软件层面的解法：把交换机的端口逻辑分组，每组是一个独立广播域，组间默认二层不通。

VLAN 10（研发） VLAN 20（财务） ┌────┬────┐ ┌────┬────┐ │ P1 │ P2 │ 交换机 S │ P3 │ P4 │ └─┬──┴──┬─┘ └─┬──┴──┬─┘ A B C D · P1、P2 划入 VLAN 10；P3、P4 划入 VLAN 20 · A 发广播帧 → 只在 VLAN 10 内泛洪，C/D 完全收不到 · A 想访问 C（跨 VLAN）→ 二层不通，必须经过路由器/三层交换机 → 一台物理交换机 = 两个互不干扰的广播域

那两台交换机之间怎么传多个 VLAN 的流量？总不能每个 VLAN 拉一根线。答案是 Trunk（干道）链路 + IEEE 802.1Q 标记：在帧里插一个 4 字节的 VLAN Tag，标明"我属于哪个 VLAN"，一根线就能承载所有 VLAN。

带 802.1Q 标记的以太帧（VLAN Tag 插在 源MAC 与 类型 之间）： ┌──────────┬──────────┬────────────────────────┬──────┬──────┐ │ 目的 MAC │ 源 MAC │ 802.1Q Tag（4 字节） │ 类型 │ 数据 │ └──────────┴──────────┴────────────────────────┴──────┴──────┘ │ TPID=0x8100 │ PCP DEI VID(12位) │ └──── 2B ─────┴──────── 2B ────────┘
VID 占 12 位  →  2¹² = 4096 个 VLAN  (去掉保留的 0/4095 → 可用 4094)

802.1Q 帧比普通帧长 4 字节，且插入位置在源 MAC 之后——这也解释了为什么带 Tag 的最大帧会略超标准 1518 字节（即所谓 baby giant 帧），需要交换机端口支持。Access 口连主机、只属于一个 VLAN、进出都不带 Tag；Trunk 口连交换机、承载多 VLAN、帧带 Tag（本征 VLAN/native VLAN 例外，不打 Tag）。

六、安全视角：交换机会被怎么攻击

本节归在"链路层与安全"章节，正因为交换机的自学习和泛洪机制天生有可乘之机。理解攻击，才能真正理解防御为什么那么设计。

反例 — 默认无防护，MAC 泛洪攻击
攻击者伪造海量随机源 MAC 猛发帧：

MAC 地址表容量有限（几千~几万条）
→ 被垃圾表项灌满、正常表项被挤掉
→ 交换机查不到目的 MAC，只能泛洪
→ 退化成 Hub！攻击者坐等嗅探全网流量
（这就是 CAM Table Overflow / MAC Flooding）
正例 — 端口安全 + VLAN 隔离
交换机开启 Port Security：

限制每个端口可学习的 MAC 数量
→ 超限即告警/关闭端口，表项灌不满
配合 VLAN 划分：
→ 即便某段被攻破，广播域被限在本 VLAN
→ 影响面被"舱壁"隔离，不会波及全网
再加 DHCP Snooping / 动态 ARP 检测兜底
面试常问："交换机安全吗？"标准答案是"默认不安全，靠配置加固"。除了上面的 MAC 泛洪，还有 VLAN 跳跃攻击（VLAN Hopping）——利用 native VLAN 不打 Tag 的特性或伪造 Trunk 协商，让流量跨越 VLAN 边界。防御要点：把闲置口划入无用 VLAN 并关掉、native VLAN 改成一个没人用的专用 ID、关闭端口的自动 Trunk 协商。记住一个安全直觉：凡是"自动、无验证"的便利机制（自学习、ARP、Trunk 自协商），都是攻击面。

七、落到代码：在 Java / 系统里看 MAC 与网卡

虽然交换机对应用透明，但你在 Java 里随时能读到本机网卡的 MAC——比如生成设备指纹、老式软件授权绑定网卡时会用到 NetworkInterface：

读取本机网卡的 48 位 MAC 地址Java
NetworkInterface ni = NetworkInterface.getByName("eth0");
byte[] mac = ni.getHardwareAddress();   // 6 字节 = 48 位；无网卡时返回 null

StringBuilder sb = new StringBuilder();
for (int i = 0; i < mac.length; i++) {
    // %02X：不足两位补零、大写十六进制——MAC 的标准写法
    sb.append(String.format("%02X%s", mac[i], (i < mac.length - 1) ? ":" : ""));
}
System.out.println(sb);   // e.g. 3C:22:FB:8A:1D:07（前 3 字节 3C:22:FB 是 OUI）
陷阱提醒：别拿 MAC 当"可信身份"。getHardwareAddress() 拿到的只是当前接口的 MAC，而现代操作系统普遍支持 MAC 随机化（手机连 Wi-Fi 时尤其如此，隐私保护），且 MAC 可被软件任意伪造（ip link set eth0 address ... 一行搞定）。所以用 MAC 做授权绑定、访问控制都不可靠——这也正是上一节反复强调"ARP/MAC 无认证"的延续。真要做身份，用证书（下一节 HTTPS 的思路）。

八、收束

一句话收束：以太网帧靠 48 位 MAC 寻址，交换机是二层设备，凭一句"源 MAC 必在入端口一侧"就能自学习出 MAC 地址表，从而做出泛洪 / 过滤 / 转发三种决策——查不到就泛洪、同侧就丢弃、异侧才精准转发，表项还会按 aging time 老化。它分割冲突域但不分割广播域，对主机完全透明（不改 MAC、不减 TTL），这是它和按 IP 转发、会重写帧的路由器的本质区别。要在一台物理交换机上隔离广播域，用 VLAN（12 位 VID，802.1Q Tag，Trunk 口跨机承载多 VLAN，跨 VLAN 必须过三层）。而这些"自动、无验证"的便利（自学习、Trunk 自协商）也正是 MAC 泛洪、VLAN 跳跃等攻击的入口，防御靠端口安全 + VLAN 隔离 + 关闭自动协商。落到代码：NetworkInterface.getHardwareAddress() 能读 MAC，但 MAC 可伪造、会随机化，永远别拿它当可信身份。
