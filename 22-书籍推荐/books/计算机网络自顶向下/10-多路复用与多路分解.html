多路复用与多路分解

IP 只负责把数据送到"这台主机"，可主机上跑着几十个进程。传输层如何把一条到达主机的数据，精准塞进正确那个进程的信箱？答案就藏在报文段的两个端口字段里。

一、问题的起点：网络层只送到"主机"，谁来送到"进程"？

先厘清一条分界线：网络层（IP）提供的是主机到主机（host-to-host）的交付——它靠目的 IP 地址把分组送到正确的那台机器，仅此而已。但你的电脑此刻同时开着浏览器、微信、网易云、一个后台的 Java 服务……数据到了机器门口，该交给哪个进程？IP 不管这事。

补上这块拼图、把交付从"主机到主机"延伸到进程到进程（process-to-process）的，正是传输层。而它完成这件事的两个动作，就叫多路复用（multiplexing）和多路分解（demultiplexing）。

《自顶向下》的经典类比：分信的老大。想象两户人家：Ann 家和 Bill 家。Ann 家有 4 个孩子在外地上学，天天写信；这些信统一投进邮局（网络层）。信送到 Bill 家门口时，是 Bill 家的老大负责把每封信按收信人名字分发到弟弟妹妹手里。这里——每个孩子 = 一个进程（socket），信封上的名字 = 端口号，负责分信的老大 = 传输层的分解逻辑，邮局 = 只管送到门口的网络层。多路复用就是发信端老大把各孩子的信塞进统一信封投递；多路分解就是收信端老大按名字拆分。

二、两个动作到底谁在做什么

关键是分清发送端和接收端各干一半的活，别把两个词记混：

发送主机（多路复用 multiplexing） 接收主机（多路分解 demultiplexing） ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ │S1 │ │S2 │ │S3 │ ← 多个 socket │S1 │ │S2 │ │S3 │ └─┬─┘ └─┬─┘ └─┬─┘ └─▲─┘ └─▲─┘ └─▲─┘ │ │ │ │ │ │ └──┐ │ ┌──┘ 各自的数据 + 端口首部 └──┐ │ ┌──┘ 按首部端口分发 ▼ ▼ ▼ │ │ │ ┌────────────┐ 汇成一股交给网络层 ┌────────────┐ │ 传输层复用 │ ───────► IP ─── 网络 ─── IP ──►│ 传输层分解 │ └────────────┘ └────────────┘
  • 多路复用（发送端）：从多个 socket 收集数据，给每块数据封上首部信息（含源/目的端口），再汇聚成报文段交给网络层。"多路"进、"一路"出。
  • 多路分解（接收端）：把网络层递上来的报文段，依据首部里的端口信息，投递到正确的那个 socket。"一路"进、"多路"出。
记忆锚点：复用 = 汇聚（many→one），发生在发送方；分解 = 分发（one→many），发生在接收方。二者是同一枚硬币的两面，一次通信里两端各做一件。"分解"这个中文译名有点拗口，理解成"拆信分发"就顺了。

三、靠什么分解？报文段首部里的端口号

多路分解要能工作，报文段里就必须带上"这封信给谁"的线索。UDP 和 TCP 的首部头 4 个字节完全一样：源端口号和目的端口号，各 16 比特。

UDP / TCP 报文段首部的第一个 32-bit 字（两者相同） 0 15 16 31 ┌──────────────────────┬──────────────────────┐ │ 源端口号 (16 bit) │ 目的端口号 (16 bit) │ └──────────────────────┴──────────────────────┘ ▲ 我是谁发的（回信地址） ▲ 交给本机哪个 socket

16 比特意味着端口号范围是 0 ~ 65535。它被划分成几档，这是面试常问的常识：

范围名称说明
0 – 1023周知端口（well-known）保留给标准服务：HTTP 80、HTTPS 443、SSH 22、DNS 53。Linux 下绑定需 root 权限。
1024 – 49151注册端口分配给特定应用：MySQL 3306、Redis 6379、Tomcat 8080。
49152 – 65535动态/临时端口（ephemeral）客户端发起连接时由 OS 临时分配的源端口。
陷阱：端口不是 socket，端口号也不"属于"某个进程独占。初学者常说"这个端口被 Chrome 占了"。更准确的表述是：一个监听 socket 绑定了某端口。同一端口在 TCP 里可以同时服务成千上万个连接（见第五节四元组），所以"端口 = 一个连接"是错的。真正标识"一路数据"的是socket，而 socket 由一组地址元组唯一确定——UDP 和 TCP 的元组还不一样，这正是下面两节的分水岭。

四、UDP 的分解：只看"二元组"

UDP 的 socket 靠一个二元组来识别：（目的 IP 地址，目的端口号）。

UDP socket = (目的IP, 目的端口)

这带来一个反直觉但重要的结论：只要目的 IP 和目的端口相同，两个源头完全不同的报文段，会被投递到同一个 UDP socket。源 IP、源端口在分解时不参与匹配——它们只是被当作"回信地址"存起来，供服务器回复时使用。

两个客户端发给同一个 UDP 服务端口 6789： 客户端A (IP=10.0.0.1:5500) ──┐ ├──► 服务器 (IP=203.0.0.9 : 6789) 同一个 socket！ 客户端B (IP=10.0.0.2:7700) ──┘ 分解只看 (203.0.0.9, 6789) ▲ 源地址不同，但都进同一个信箱；服务器靠 recvfrom 拿到的源地址来区分是谁
Java · 一个 DatagramSocket 服务所有客户端UDP
DatagramSocket socket = new DatagramSocket(6789); // 只此一个 socket
byte[] buf = new byte[1024];
DatagramPacket pkt = new DatagramPacket(buf, buf.length);

while (true) {
    socket.receive(pkt);                 // A、B 的包都从这里出来
    InetAddress from = pkt.getAddress();  // ← 源 IP，用来区分/回信
    int fromPort = pkt.getPort();       // ← 源端口
    // 想回信必须显式带上对方地址，因为 socket 本身"记不住"谁是谁
    DatagramPacket reply = new DatagramPacket(data, data.length, from, fromPort);
    socket.send(reply);
}
为什么这样设计？——因为 UDP 无连接。UDP 不维护任何连接状态，服务器脑子里没有"和 A 的会话""和 B 的会话"这种概念，自然没必要用源地址去区分 socket。所有人共用一个信箱、来信统一收，谁是谁靠每封信自带的回信地址临时判断。这份"轻"正是 UDP 的卖点：省去连接管理，换来低开销与低时延（DNS、视频会议、游戏都爱用它，下一篇会展开）。

五、TCP 的分解：必须看"四元组"

TCP 的 socket 靠一个四元组唯一识别：

TCP socket = (源IP, 源端口, 目的IP, 目的端口)

和 UDP 的最大区别：源 IP 和源端口也参与匹配。这意味着，即便两个报文段的目的 IP 和目的端口一模一样（都发给服务器的 80 端口），只要源不同，就会被分解到两个不同的 socket。

这正是"一台 Web 服务器只监听 80 端口，却能同时服务上万个客户端"的原理——也是面试高频题。

Nginx 监听 203.0.0.9:80，三个连接同时在线： 连接① (10.0.0.1 : 5501, 203.0.0.9 : 80) ─► socket #1 连接② (10.0.0.2 : 6602, 203.0.0.9 : 80) ─► socket #2 目的端口都是 80， 连接③ (10.0.0.1 : 5502, 203.0.0.9 : 80) ─► socket #3 但四元组各不相同！ ▲ 注意①③同一台主机同一 IP，靠不同源端口区分
欢迎 socket vs 连接 socket。TCP 服务器其实有两类 socket：一个长期存在的欢迎/监听 socket（welcoming socket）专门"接客"；每当三次握手完成，服务器就为这条连接新建一个连接 socket。之后该客户端的所有报文段，凭四元组精确分解到它专属的连接 socket。在 Java 里，ServerSocket 就是欢迎 socket，accept() 返回的 Socket 就是连接 socket。
Java · accept() 每来一个连接就产出一个专属 socketTCP
ServerSocket welcome = new ServerSocket(80); // 欢迎 socket，只此一个

while (true) {
    Socket conn = welcome.accept();      // 握手完成 → 诞生一个连接 socket
    // conn 内部就绑定了完整四元组：
    // (conn.getInetAddress()/getPort() 是对端, getLocalAddress()/getLocalPort() 是本端)
    new Thread(() -> handle(conn)).start(); // 每连接一线程（或交给线程池/NIO）
}
// 内核里维护着一张四元组 → socket 的哈希表，收到段就查表分解
串联操作系统：socket 本质是文件描述符。在 Linux 里，每个 socket 对应一个文件描述符（fd），内核用一张哈希表把四元组映射到对应的 fd/socket 结构体。收到一个 TCP 段，内核提取四元组去查表，命中就把数据放进该 socket 的接收缓冲区——这就是分解的真实落地。也正因为每个连接吃一个 fd，才有了"单机最大连接数受 ulimit -n 限制""C10K 问题"这些工程话题。Java NIO 的 Selector/epoll 正是为了让一个线程管理海量 socket 而生（联系你学过的 IO 多路复用，注意此"多路复用"是 IO 层面的复用，与本页传输层的 multiplexing 名同实异）。

七、反例 → 正例：别把 UDP 当 TCP 那样"一客户端一 socket"

理解了二元组 vs 四元组，就能避开一个真实坑：用 UDP 时试图给每个客户端单开一个 socket 去"隔离"，或回信时忘了带对方地址。UDP 的 socket 不记得对端，所有客户端天然共享一个信箱。

反例 — 误以为 UDP 也一客户端一 socket
// 想"每个客户端独占一个 socket"
Map<Client,DatagramSocket> pool;
DatagramSocket s = new DatagramSocket(6789);
pool.put(clientA, s);
pool.put(clientB, s); // 端口只能绑一次！
// 且回信直接 send，没带对方地址：
s.send(new DatagramPacket(data, len));
// → 不知发给谁 / 端口冲突，逻辑全错
正例 — 一 socket 收全部，回信带上源地址
DatagramSocket s = new DatagramSocket(6789);
while (true) {
  s.receive(pkt);              // 所有客户端都从这收
  InetAddress ip = pkt.getAddress();
  int port = pkt.getPort();     // 从包里取回信地址
  DatagramPacket reply =
    new DatagramPacket(data, len, ip, port);
  s.send(reply);              // 精确回给来信者
}
另一个高频误区：以为"端口耗尽"是服务端问题。常有人担心"Web 服务器只有一个 80 端口，连接多了会不会把端口用完？"——不会，服务端所有连接都复用 80，区分靠四元组。真正可能耗尽临时端口的是客户端：一台机器主动向同一个目标 (目的IP, 目的端口) 发起大量连接时，可用的源端口（约 2.8 万个临时端口）会被占满，报 Cannot assign requested address。压测工具、连接池打满时就会撞上，解法是扩源 IP、复用连接或调大端口范围。

八、把整条链路串起来

一次 HTTP 请求，多路复用/分解各出场一次： 浏览器进程 ─┐ 复用: 数据 + (源端口 5501, 目的端口 80) 封成段 ▼ [传输层]──► [IP: 加目的IP]──► 网络 ──► [到达服务器主机] │ [传输层] 分解: 查四元组哈希表 (10.0.0.1:5501, 203.0.0.9:80) ▼ 命中 Nginx 的连接 socket ✔
一句话收束：多路复用/分解，是传输层把 IP 的"主机到主机"延伸为"进程到进程"的那层关键胶水——复用在发送端汇聚（many→one）、分解在接收端分发（one→many），都靠首部里的源/目的端口。两条判别线务必刻进脑子：UDP 用二元组（目的IP+目的端口，不同源进同一 socket）、TCP 用四元组（含源地址，一连接一 socket，这正是单端口撑起海量并发的根本）；还要记住端口 ≠ socket，真正被内核哈希表索引、对应一个文件描述符的，是那组地址元组。理解了它，你才真正看懂 Web 服务器的并发模型、C10K、以及为什么 UDP 服务器回信非得自己带地址。
