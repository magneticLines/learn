UDP：无连接与校验和

把传输层剥到最薄的一层：只做复用/分解和一次差错检测，剩下的可靠、拥塞、有序，统统交还给应用自己决定要不要。

一、UDP 是什么：几乎"什么都不做"的传输层

上一节我们看到，传输层最基本的职责是多路复用/分解——把主机与主机之间的传输，细分到进程与进程之间。UDP（User Datagram Protocol，用户数据报协议，RFC 768）做的事，几乎就只有这一件：给应用数据加上源端口、目的端口，再加一个校验和做差错检测，然后甩给网络层（IP）就不管了。

正因为它"什么都不做"，理解 UDP 的正确姿势不是"UDP 有哪些功能"，而是"相比 TCP，UDP 砍掉了什么"。它砍掉了：建立连接的握手、可靠交付的确认重传、保证顺序的序号、以及流量/拥塞控制。留下的，是一个无连接（connectionless）、尽力而为（best-effort）的通道。

为什么要保留这么"简陋"的一层？网络层 IP 提供的就是"主机到主机、尽力而为、不可靠"的服务。UDP 做的，本质是把 IP 的服务从"主机到主机"延伸到"进程到进程"，并顺手加一道校验。它是最贴近 IP 原貌的传输层——你想要极致的控制权和最低的开销时，UDP 把方向盘完整交回你手里，而不是像 TCP 那样替你把一切都安排好。

二、"无连接"到底意味着什么

"无连接"常被误解成"不需要网络就能发"。它真正的含义是：发送方和接收方之间不事先建立任何会话状态，发数据前不握手，发完也不挥手。每一个 UDP 数据报都是独立的、自带完整地址信息的"信封"，彼此之间毫无关系。

TCP（有连接）：先建立关系，再对话，最后正式告别 客户端 ──SYN──► 握手 3 次，双方各建一份连接状态(TCB) 客户端 ◄SYN/ACK─ 客户端 ──ACK──► ✔ 连接建立后才能发数据 …… 结束再 4 次挥手 UDP（无连接）：不打招呼，直接扔信封 客户端 ──[信封: 目的IP+端口 | 数据]──► 接收方 客户端 ──[信封: 目的IP+端口 | 数据]──► 接收方 ← 每封独立，可发给不同人 ▲ 无握手、无连接状态、无挥手
生活类比：寄明信片 vs 打电话。TCP 像打电话：要先拨号、等对方接通（握手），确认"喂听得到吗"才开始说，讲完还要说"再见"挂断。UDP 像寄明信片：写上地址直接投进邮筒，不确认对方在不在家、不保证一定送到、也不保证先寄的先到。但明信片快、便宜、可以同时给很多人群发——这正是 UDP 的性格。

"无连接"带来一个 TCP 给不了的能力：UDP 天然支持一对多。因为没有"连接"这个概念绑定通信双方，同一个 UDP 套接字可以往不同目的地发送数据报——这是广播（broadcast）和多播（multicast）只能建在 UDP 上、而 TCP 做不到的根本原因。

三、UDP 报文段结构：只有 8 字节头部

UDP 的极简，直接体现在报文格式上。它的首部只有4 个字段、共 8 字节，而 TCP 首部至少 20 字节。头部越小，有效载荷占比越高，开销越低。

0 16 31 ← 比特位 +-------------------+-------------------+ | 源端口(16) | 目的端口(16) | ← 复用/分解靠这两个 +-------------------+-------------------+ | 长度(16) | 校验和(16) | ← 长度=首部+数据；校验和查错 +-------------------+-------------------+ | 应用层数据 | | (payload) | +---------------------------------------+ 每格 16 位，共 4 格 = 8 字节首部
字段长度作用
源端口16 位标识发送进程，供接收方回信用（可为 0 表示不需回复）
目的端口16 位分解的关键：内核据此把数据交给对应进程的套接字
长度16 位首部 + 数据的总字节数（最小为 8，即空数据报）
校验和16 位差错检测，覆盖首部、数据及一段 IP 伪首部
陷阱：端口是 16 位，所以范围是 0~65535——这对 TCP/UDP 都成立。另外注意"长度"字段最大 65535 字节，减去 8 字节首部和 IP 首部，单个 UDP 数据报的理论载荷上限约 65507 字节。但实际远小于此：一旦超过链路 MTU（以太网通常 1500 字节），IP 层就得分片（fragmentation），任何一个分片丢失，整个数据报都得丢弃——UDP 又不重传，于是全丢。所以工程上单个 UDP 报文常控制在不触发分片的大小（如 1472 字节以内）。

四、校验和：UDP 唯一的"良心"

UDP 放弃了可靠性，却保留了一个校验和（checksum）字段。它不能纠错、也不能重传，唯一能做的是检测：这份数据在传输途中是否发生了比特翻转。检测到出错，接收方默认的处理就是直接丢弃（部分实现会附带一个警告），应用层根本收不到。

怎么算的：16 位反码求和

发送方把报文段（含伪首部）按 16 位为一组切开，全部相加；求和时若最高位产生进位（carry），要把这个进位回卷（wrap around）加回到最低位；最后对结果取反码（每一位取反），填入校验和字段。

checksum = ~( Σ 16位分组 + 回卷进位 )

接收方的验证更巧妙：它把所有 16 位分组连同收到的校验和一起再求和。如果传输无误，结果应当是全 1（即 0xFFFF）；只要出现任何一个 0 位，就说明数据被篡改过。

发送方：假设有 3 个 16 位字（示例简化） 0110011001100000 0101010101010101 + 1000111100001100 ────────────────── 相加可能溢出16位 → 把进位回卷加回最低位 得到 sum，再逐位取反(~sum) = 校验和，放进报文一起发出 接收方：把 3 个字 + 收到的校验和 全部相加 若结果 = 1111111111111111 (0xFFFF) → 认为没错，接收 若出现任何一个 0 → 检测到差错，丢弃
为什么用"反码求和"而不是直接异或或普通加法？反码和有一个漂亮性质：与字节序无关、可增量更新、且交换律成立，路由器在修改报文（如 NAT 改端口）后能只对变化部分做增量计算，无需重算整包——这在早期低算力路由器上极其重要。它换来的是实现简单、速度快，代价是检错能力有限：偶数个位翻转、或两处误差恰好相互抵消时会漏检。所以它是"廉价的健康检查"，不是"密码学级的完整性保证"。

陷阱：校验和其实是"可选"的，且有个 0 的特例

面试高频细节：在 IPv4 下 UDP 校验和是可选的——发送方可填 0 表示"我没算，别校验"。而 IPv6 因为网络层自己不再做首部校验，强制要求 UDP 必须计算校验和。
另一个坑：如果真算出来的校验和是 0x0000，必须把它写成 0xFFFF（反码里 +0 和 −0 等价，二者数值相同）。因为 0x0000 被保留用来表示"未启用校验"，两者不能混淆。

五、为什么明明有链路层校验，UDP 还要自己再校验一次?

很多人疑惑：以太网帧尾有 CRC 校验、很多链路层也有纠错，为什么传输层还要多此一举？答案是计算机网络里最重要的思想之一——端到端原则（End-to-End Argument）。

端到端原则：某些功能只有在通信的两个端点完整实现才真正可靠，中间环节的保障只是优化、不能替代。链路层校验只能保证某一段链路传输无误，但数据在路由器内存里排队、在被拷贝、在跨越无校验的内部总线时同样可能出错；而且数据要经过多段异构链路，无法假设每一段都有校验。只有发送进程和接收进程各自校验一次，才能覆盖全程。UDP 校验和正是这一原则在传输层的最小落地。

六、UDP vs TCP：一张表看清取舍

选 UDP 还是 TCP，本质是回答一个问题："我愿不愿意为可靠性支付延迟和开销的代价？"

维度UDPTCP
连接无连接，发前不握手面向连接，三次握手 / 四次挥手
可靠性尽力而为，可能丢/乱序/重复可靠交付，确认 + 重传保证到达
顺序不保证顺序字节流按序交付
首部开销仅 8 字节至少 20 字节
拥塞控制无（可打满带宽，也可能加剧拥塞）有（慢启动 / 拥塞避免等）
通信模式支持一对一 / 一对多（广播、多播）仅一对一
典型场景DNS、DHCP、实时音视频、游戏、QUIC/HTTP3HTTP/1&2、FTP、SMTP、文件传输
为什么 DNS 用 UDP？一次 DNS 查询往往就一个小请求 + 一个小响应。若用 TCP，光三次握手就要多花 1 个 RTT，代价比查询本身还大。用 UDP 一来一回即可，丢了就重发一次——把"可靠性"这点小事交给应用层自己兜底，比背上 TCP 全套机制划算得多。这就是"砍掉功能反而更优"的经典案例。（注：当响应过大或做区域传送时，DNS 会切换到 TCP。）

七、代码：Java 里用 UDP 收发数据报

Java 用 DatagramSocket + DatagramPacket 操作 UDP。注意它的 API 直接暴露了"无连接"的本质：每个 packet 自带目的地址，socket 本身不绑定对端。

UDP 客户端 · DatagramSocketJava
import java.net.*;

public class UdpClient {
    public static void main(String[] args) throws Exception {
        // 不 connect、不握手：创建即可发送
        DatagramSocket socket = new DatagramSocket();
        InetAddress server = InetAddress.getByName("127.0.0.1");

        byte[] data = "hello udp".getBytes();
        // 关键：目的地址写在 packet 上，不在 socket 上
        DatagramPacket out = new DatagramPacket(
                data, data.length, server, 9999);
        socket.send(out);   // 扔出去就返回，不等任何确认

        // 收回信：buffer 要预先给够，收不下的字节会被悄悄丢弃！
        byte[] buf = new byte[1024];
        DatagramPacket in = new DatagramPacket(buf, buf.length);
        socket.receive(in);  // 阻塞直到收到一个数据报
        System.out.println(new String(
                in.getData(), 0, in.getLength()));
        socket.close();
    }
}
UDP 编程三大新手坑：
· 接收缓冲区太小：receive() 只填满你给的 buffer，超出部分被静默丢弃且不报错，不像 TCP 是字节流可以再读。
· 把 UDP 当流用：UDP 是面向报文（datagram）的——发一次 = 收一次，边界清晰；不存在 TCP 那种"粘包/拆包"。但也意味着一次 send 就是一个不可分割的报文。
· 误以为 send 成功 = 对方收到：send() 返回只代表数据交给了内核/网卡，不代表送达。要可靠就得自己在应用层加确认与超时重传。

八、坏味道 → 正确姿势：别拿 UDP 当可靠通道

最常见的误用，是想当然地认为"局域网很稳，UDP 发出去肯定到"，于是把重要指令用裸 UDP 发送、不做任何补偿。看一组对比：

反例 — 裸 UDP 发关键指令
// 发一条"扣款"指令就撒手不管
socket.send(packet);
// 假设一定送达 → 丢包时
// 扣款指令凭空消失，无人知晓
// 也没有序号，重复到达时
// 可能被执行两次
正例 — 应用层补上最小可靠性
// 给每条指令加序号 + ACK + 超时重传
int seq = nextSeq();
sendWithSeq(seq, cmd);
while (!gotAck(seq, timeout)) {
    sendWithSeq(seq, cmd);  // 重传
}
// 接收端按 seq 去重，保证幂等
// —— 这正是下一节 rdt 的雏形
联系下一节与现实：如果你发现自己在 UDP 上手写"序号 + 确认 + 重传 + 排序"，恭喜——你正在重新发明 TCP，这也正是下一节《可靠数据传输原理》要系统讲的 rdt 协议。现实中 QUIC（HTTP/3 的底座）就是这么干的：它建在 UDP 之上，在用户态实现了可靠性、多路复用和拥塞控制，从而绕开内核 TCP 的僵化、实现更快的连接建立与演进。选 UDP 不是"放弃可靠"，而是"把可靠性的实现权夺回到应用层"。
一句话收束：UDP = 复用/分解 + 一次校验和，此外几乎什么都不做。无连接意味着不握手、无连接状态、天然支持一对多；8 字节首部换来最低开销与最小延迟；校验和只检错不纠错，出错即丢，且体现了贯穿全书的端到端原则。记住三条判别线：UDP 面向报文、TCP 面向字节流；send 成功 ≠ 送达；需要可靠就在应用层自建 rdt（或直接用 TCP/QUIC）。这一层的"简陋"，恰恰是它作为 DNS、实时音视频、QUIC 基石的力量所在。
