Socket 编程：UDP 与 TCP Socket

协议画得再漂亮，最终都要落到一行 socket.send(...) 上。Socket 就是应用进程伸进传输层的那只手——本节把 UDP 与 TCP 两套写法拆开对照，看清"要不要建连接"这一个决定，如何一路决定了你后面所有的代码。

一、Socket 是什么：应用与传输层之间的那扇门

《自顶向下》把 Socket 定义成进程与网络之间的"门"（door）。进程好比一栋房子，Socket 就是它的大门：应用要发数据，只管把报文推出门外；剩下的运输（怎么分段、怎么找路、要不要重传）全交给门外的传输层基础设施。门这一侧归应用开发者管，门那一侧的运输网络你无权干预——你唯一的旋钮，就是"选哪种门"：UDP 门还是 TCP 门。

和操作系统串起来：Socket 本质是一个文件描述符（fd）。在 Linux 里 socket() 返回一个整数 fd，和 open() 打开文件拿到的 fd 是同一套东西——所以你能用 read()/write() 收发网络数据，"一切皆文件"在这里体现得淋漓尽致。Java 的 Socket.getInputStream() 底层就是把这个 fd 包成了 InputStream；DatagramSocket 里的 fd 字段（java.net.AbstractPlainDatagramSocketImpl）也正是它。

两种门对应传输层的两种服务，本书前面已经埋过伏笔：UDP 提供不可靠、无连接的数据报服务；TCP 提供可靠、面向连接的字节流服务。选门这一步，决定了你后面要不要自己处理丢包、要不要自己切分消息。

二、UDP Socket：把信丢进邮筒，发完就走

UDP 是无连接的：发送前不握手，两端不维护任何连接状态。写代码时你会强烈感觉到这一点——每个数据报（datagram）都必须自带完整的目的地址，就像寄平信，每封信封上都要重写收件人地址。JDK 里对应 DatagramSocket（那只邮筒）和 DatagramPacket（那封信）。

UDP：一个 socket 服务所有人，地址随包携带 客户端 服务器 DatagramSocket() ──包①(含目的IP:Port)──▶ DatagramSocket(9877) (OS 分配临时端口) receive() 阻塞等信 ◀──包②(回信地址从包①现取)─ send() · 没有 connect / accept · 一个 socket 收所有客户端的包 · 发完不等确认，可能丢/乱序/重复 · 每个包独立，天生有"消息边界"
UDPServer.java — 一个 socket 收发所有人Java
public class UDPServer {
    public static void main(String[] args) throws IOException {
        // 只是"绑个端口开个邮筒"，没有任何连接概念
        DatagramSocket socket = new DatagramSocket(9877);
        byte[] buf = new byte[1024];
        while (true) {
            DatagramPacket req = new DatagramPacket(buf, buf.length);
            socket.receive(req);            // 阻塞，收到一个包就填充 req
            // 关键：必须用 getLength()，不能用 buf.length，否则会读到脏数据
            String msg = new String(req.getData(), 0, req.getLength());
            byte[] out = msg.toUpperCase().getBytes();
            // 回信地址不是配置来的，是从收到的包里"现取"的
            DatagramPacket resp = new DatagramPacket(
                    out, out.length, req.getAddress(), req.getPort());
            socket.send(resp);
        }
    }
}
UDPClient.java — 每个包都写全目的地址Java
DatagramSocket socket = new DatagramSocket();   // OS 分配一个临时源端口
InetAddress server = InetAddress.getByName("localhost");
byte[] data = "hello".getBytes();
// 目的 IP+端口写在包上，不是写在 socket 上
DatagramPacket req = new DatagramPacket(data, data.length, server, 9877);
socket.send(req);                             // 发完即走，不等任何确认

byte[] buf = new byte[1024];
DatagramPacket resp = new DatagramPacket(buf, buf.length);
socket.receive(resp);                          // 可能永远收不到（丢包了也没人告诉你）
System.out.println(new String(resp.getData(), 0, resp.getLength()));
socket.close();
UDP 三大新手坑，全是"无连接"的副作用：
① 收缓冲区太小会静默丢字节。若 DatagramPacket 的 buffer 比实际数据报短，Java 会悄悄丢弃放不下的部分，没有异常。
② 读数据必须用 getLength()。buffer 是复用的，上一个包的残留还在里面；用 buf.length 会把脏数据一起读出来。
③ 丢包/乱序/重复都不报错。UDP 不重传，receive() 收不到就一直阻塞。要可靠，得你自己在应用层补——这正是 QUIC / 视频会议在 UDP 上做的事。

三、TCP Socket：先打通电话，再开始说话

TCP 是面向连接的：真正收发数据前，客户端和服务器要先完成三次握手建立连接。连接建好后，双方各得到一根字节流管道——你写进去的字节会原样、有序、不丢地从另一头流出。代价是：这根管道没有"消息边界"（后面第五节的大坑）。

Kurose 特别强调 TCP 服务器有两种 Socket，Java 的 API 把它体现得非常清楚：

TCP：欢迎套接字负责迎宾，连接套接字负责服务 客户端 服务器 new Socket(host,9878) ─握手─▶ ServerSocket(9878) ← 欢迎套接字(welcoming) │ 只干一件事：accept() 迎客 三次握手完成 ✔ ▼ ◀━━━━━━ 字节流管道 ━━━━━━▶ accept() 返回 Socket ← 连接套接字(connection) 每个客户端一个，专供它服务
为什么要两种 socket？欢迎套接字（ServerSocket）是"总机/前台"，永远守在众所周知的端口上迎客；每来一个客户端握手成功，accept() 就新造一个连接套接字（Socket）专门伺候它。这样服务器能同时和成千上万个客户端各聊各的、互不串线——这也为下一节"多路复用与多路分解"埋下伏笔：靠什么把回来的报文分派给正确的那根管道？答案就在下面的四元组。
TCPServer.java — 欢迎套接字 + 连接套接字Java
public class TCPServer {
    public static void main(String[] args) throws IOException {
        ServerSocket welcome = new ServerSocket(9878); // 欢迎套接字：只迎宾
        while (true) {
            Socket conn = welcome.accept();          // 握手完成才返回连接套接字
            try (conn) {
                var in  = new BufferedReader(
                        new InputStreamReader(conn.getInputStream()));
                var out = new DataOutputStream(conn.getOutputStream());
                String line = in.readLine();            // 面向字节流，靠 '\n' 断句
                out.writeBytes(line.toUpperCase() + "\n");
            }
        }
    }
}
TCPClient.java — new Socket 就是主动握手Java
// 构造 Socket 的那一刻，三次握手就已经在发生了
try (Socket socket = new Socket("localhost", 9878)) {
    var out = new DataOutputStream(socket.getOutputStream());
    var in  = new BufferedReader(
            new InputStreamReader(socket.getInputStream()));
    out.writeBytes("hello\n");            // 写进管道，TCP 保证有序不丢
    System.out.println(in.readLine());
}                                            // try-with-resources 自动 close，触发四次挥手
和 Spring / Tomcat 串起来：你几乎不会亲手写 accept() 循环——因为 Tomcat 的 Acceptor 线程就是这段代码的产品化版本：它守着 ServerSocket 不停 accept，把每个连接套接字丢进线程池（或 NIO 的 Poller）里处理，最后调到你的 @RestController。理解了这 10 行，你就理解了整个 Servlet 容器的入口骨架。

四、谁是谁：socket 的身份靠几元组

一个到达的报文，操作系统凭什么知道该交给哪个 socket？答案藏在 socket 的"身份标识"里，而 UDP 和 TCP 的身份规则不同——这正是它们行为差异的根源：

UDP socket  ≡  ( 目的IP , 目的端口 )  二元组
TCP socket  ≡  ( 源IP , 源端口 , 目的IP , 目的端口 )  四元组

因为 UDP 只认二元组，所以两个不同客户端发往同一个 目的IP:端口 的包，会被导向同一个 UDP socket——这就是为什么服务器一个 DatagramSocket 能接待所有人，也是为什么你必须从每个包里现取来源地址才能回信。而 TCP 认四元组，只要源不同就是不同 socket，于是服务器能为每个客户端单独开一个连接套接字。这套"分派"逻辑，就是下一节多路分解（demultiplexing）的全部秘密。

五、面试必考大坑：TCP "粘包/拆包"

这是 TCP 编程最高频的翻车点：新手默认"我 write 一次，对方就能 read 到一条完整消息"。错。TCP 是字节流，不是报文流——它只保证字节的顺序，完全不保证你的应用消息边界。发送方两次 write 可能被合并成一次 read（粘包），一次 write 也可能被拆成多次 read（拆包），这跟 Nagle 算法、MSS 分段、网络缓冲都有关。

反例 — 假设"一次读=一条消息"
// 发送端连发两条
out.write("PING".getBytes());
out.write("PONG".getBytes());

// 接收端天真地读一次
byte[] b = new byte[1024];
int n = in.read(b);
// 可能读到 "PINGPONG"（粘包）
// 也可能只读到 "PIN"（拆包）
// 字节流里根本没有"边界"这回事
正例 — 长度前缀，自己造边界
// 发送：先写 4 字节长度，再写内容
var dout = new DataOutputStream(out);
byte[] body = "PING".getBytes();
dout.writeInt(body.length);   // 帧头
dout.write(body);

// 接收：先读长度，再精确读满
var din = new DataInputStream(in);
int len = din.readInt();
byte[] frame = new byte[len];
din.readFully(frame);         // 不多不少
记牢：解决粘包只有两条正道 —— 定界（framing）。要么像上面用长度前缀（先发 4 字节长度再发正文），要么用分隔符（如 HTTP 用 \r\n、Redis 协议用 \r\n）。UDP 天生没这个问题，因为它保留报文边界——一个 send 对应一个 receive。这也解释了为什么讲究"消息感"的协议（DNS 查询、直播弹幕）偏爱 UDP。
本仓库刚学过的 Netty 就是干这个的：你不用手写上面的 readInt + readFully，Netty 提供了现成的 LengthFieldBasedFrameDecoder（长度前缀方案）和 LineBasedFrameDecoder（分隔符方案）。理解了粘包的成因，你才知道这些 Decoder 到底在替你解决什么问题，而不是当黑盒背 API。

六、UDP vs TCP：一张表定选型

维度UDP SocketTCP Socket
连接无连接，发前不握手面向连接，先三次握手
可靠性不可靠，丢/乱序/重复都不管可靠，有序、不丢、不重
数据模型数据报，保留消息边界字节流，无边界（会粘包）
核心 APIDatagramSocket / DatagramPacketServerSocket / Socket
服务器 socket 数一个 socket 收所有人欢迎 1 个 + 每客户 1 个连接套接字
身份标识二元组（目的 IP:端口）四元组（源+目的 IP:端口）
开销 / 时延小，无握手无状态，首字节快大，握手 1 个 RTT + 连接状态
拥塞/流量控制无（要靠应用自己做）内置，会主动降速让路
典型场景DNS、视频/语音、游戏、QUIC/HTTP3HTTP、邮件、文件传输、数据库

选型的心法很简单：要"稳"（一个字节都不能错）选 TCP；要"快"、能容忍偶尔丢一点、且自己想掌控重传节奏选 UDP。视频通话丢一帧无所谓、但迟到一帧就卡顿，所以宁可用 UDP 丢掉迟到的包；转账少一分钱都是事故，所以必须 TCP。

一句话收束：Socket 是应用进程通往传输层的那扇门，"选哪种门"是你唯一也是最重要的决定。UDP 门——无连接、二元组标识、一个 socket 接待所有人、保留消息边界但不可靠，写法是"每封信都写全地址、发完就走"；TCP 门——面向连接、四元组标识、欢迎套接字迎宾+连接套接字服务、可靠字节流但会粘包，写法是"先握手打通管道、再靠定界自己断句"。记住三把钥匙：连接与否（要不要握手）、身份几元组（决定分派与并发模型）、有无消息边界（TCP 必须自己 framing）。
