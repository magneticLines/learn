互联网的构成：边缘、核心、接入网

互联网不是一团乱麻——它有清晰的三层结构：程序运行在边缘，分组穿行于核心，而接入网是连接两者的"最后一公里"。抓住这三个词，整本书的地图就有了。

一、两种看互联网的视角

Kurose 在书的开篇提出一个很妙的问题：互联网是什么？有两种回答方式。

  • "零件"视角（nuts and bolts）：互联网是由无数硬件和软件拼起来的——主机、路由器、光纤、无线电波、协议……
  • "服务"视角：互联网是一个为分布式应用提供服务的基础设施——它让你的浏览器能拿到远端服务器的网页。

本页走"零件"视角，但只需记住一件事：再复杂的零件，都能归入三大类——网络边缘、网络核心、接入网。这套划分不是死记的名词，而是理解后面所有章节（应用层跑在边缘、传输/网络层在核心里穿行）的骨架。

生活类比（贯穿全页）：把互联网想成高速公路系统。你家和目的地是网络边缘（端系统）；家门口到高速入口的那段辅路、匝道是接入网（最后一公里）；而由无数立交、枢纽、主干道织成的高速网本身，就是网络核心——车（分组）在这里被一个个路口（路由器）转发到下一段路。
┌──────────────┐ 接入网 ┌───────────────────────┐ 接入网 ┌──────────────┐ │ 网络边缘 │ (最后一公里) │ 网络核心 │ (光纤/ │ 网络边缘 │ │ │ │ │ 以太网) │ │ │ 端系统: │ WiFi/光纤/DSL/4G │ 分组交换机(路由器) │ │ 端系统: │ │ 你的浏览器 ●─┼───────────────────►│ 组成的网状网 │◄────────────┼─● Web 服务器 │ │ 手机 App │ [边缘路由器] │ [R]──[R]──[R] │ [边缘路由器]│ (数据中心) │ │ │ │ │ ╳ │ │ │ │ │ (运行程序!) │ │ [R]──[R]──[R] │ │ (运行程序!) │ └──────────────┘ │ 只转发分组,不跑用户程序│ └──────────────┘ └───────────────────────┘

二、网络边缘：程序真正运行的地方

网络边缘由端系统（end system）组成，端系统又叫主机（host）——因为它"托管"（host）着应用程序。你的笔记本、手机、智能音箱、机房里的服务器，都是端系统。它们按角色分两类：

角色是什么典型例子
客户机 client主动发起请求的一方浏览器、手机 App、curl
服务器 server被动等待、响应请求的一方；如今大多集中在数据中心Web 服务器、数据库、你写的 Spring Boot 应用

这里有个对初学者极其关键、却常被忽略的点：应用程序只跑在端系统（边缘）上，网络核心里的路由器/交换机不运行任何用户程序。你写的每一行业务代码，本质上都活在"边缘"。用一段最小的 Java Socket 代码就能看清这条边界：

服务器端 —— 跑在一台"端系统"上（比如数据中心的一台主机）
// server.accept() 之前，这台机器只是边缘上的一个普通主机
ServerSocket server = new ServerSocket(8080);   // 在 8080 端口监听
Socket conn = server.accept();                    // 阻塞,等客户端接入
BufferedReader in = new BufferedReader(
        new InputStreamReader(conn.getInputStream()));
System.out.println("收到请求: " + in.readLine());
客户机端 —— 跑在另一台"端系统"上（比如你的笔记本）
Socket sock = new Socket("93.184.216.34", 8080);  // 连到服务器 IP:端口
PrintWriter out = new PrintWriter(sock.getOutputStream(), true);
out.println("GET /index.html");
// 报文交给操作系统内核 → 接入网 → 核心的一台台路由器接力转发
// 沿途的路由器完全"看不懂"这行业务语义,它们只按目的地址转发分组
联系已学 / 操作系统：这两段代码就是后面「Socket 编程」章节的雏形。注意 Socket 是操作系统内核暴露给应用的接口——应用程序把数据写进 Socket，剩下的分段、寻址、发到网卡全由内核网络协议栈接管。所以"端系统运行程序"更精确的说法是：用户进程在边缘产生数据，OS 内核把它送上网络。这也解释了为什么路由器不跑你的程序：它们根本没有为你的应用启动进程，只做 L3 转发。

三、接入网：连接边缘与核心的"最后一公里"

端系统不会直接插进网络核心，中间隔着接入网——把主机物理连接到"边缘路由器"（进入核心的第一跳）的那段网络。这就是运营商反复念叨的"最后一公里"。常见接入方式对比如下：

接入技术物理介质典型下行速率共享 / 专用上下行
DSL（电话线宽带）电话双绞铜线几~几十 Mbps专用（到 DSLAM 独享）不对称
电缆（HFC 有线电视）同轴 + 光纤混合几十~几百 Mbps共享（整片小区共用）不对称
FTTH（光纤到户）光纤几百 Mbps ~ 1Gbps+专用可对称
以太网（企业/校园）双绞线 / 光纤1 / 10 Gbps交换机后专用对称
WiFi（无线局域网）无线电波几十~几百 Mbps共享（同一 AP 内争用）半双工
4G / 5G（广域蜂窝）无线电波十几 M ~ 几百 M（5G 更高）共享（同一基站扇区）不对称

表里最值得咀嚼的一列是"共享 vs 专用"。它直接决定了你实际能用到多少带宽：

专用介质：实测速率 = R　　　　共享介质：实测速率 ≈ R / N

其中 R 是链路标称总速率，N 是同时活跃的用户数。DSL/FTTH 是你独享一条线（专用），标了多少大致就有多少；而电缆 HFC、WiFi、蜂窝是一群人共用一段介质（共享），邻居一起看高清直播，你的网速就被摊薄——这就是为什么"晚高峰卡顿"在有线电视宽带上格外明显。

三个高频陷阱：
① "办了 1Gbps 宽带 = 下载永远 1Gbps"——错。接入网速率只是你到运营商这一段的上限；端到端速率还受核心拥塞、对端服务器、路径上最慢链路（瓶颈链路）共同限制。接入网快，不代表访问大洋彼岸的服务器就快。
② 混淆共享与专用。HFC/WiFi/蜂窝是共享介质，标称速率是"理论峰值"，人多必掉速；DSL/FTTH 才是独享。
③ 忽视上下行不对称。家用宽带上行往往远小于下行（下载快、上传慢）——做直播推流、上传大文件、自建家庭服务器时会被上行卡死。

再往下一层，所有接入技术无非用两类物理介质传输比特，二者的取舍值得对比：

导引型介质 (Guided)
信号被"约束"在固体介质里传播
• 双绞线：电话线/网线,便宜,距离近
• 同轴电缆：有线电视,抗干扰较好
• 光纤：光脉冲,超高带宽/超低损耗,
        长途骨干的绝对主力

特点：稳定、抗干扰强、可控
代价：要铺线,部署贵、不灵活
非导引型介质 (Unguided)
信号在自由空间(空气)里自由传播
• WiFi：室内无线局域网
• 4G/5G：广域蜂窝
• 卫星：偏远/海上覆盖

特点：无需布线、移动性强
代价：受干扰/遮挡/距离影响大,
      是共享介质,多设备要争用频谱

四、网络核心：由分组交换机织成的网状网

网络核心是一张由分组交换机（packet switch）和链路互联而成的网状网。互联网核心里最常见的分组交换机就是路由器（router）。核心做的事只有一件：把分组从一条入链路，转发到一条出链路，一跳一跳接力送到目的地。（分组交换的具体机制——存储转发、排队——留到下一页细讲。）

这里必须厘清两个极易混淆、又是面试高频的概念——转发（forwarding） vs 路由（routing）：

维度转发 forwarding路由 routing
作用范围单台路由器局部动作全网级别决策
做什么分组进来，查转发表，决定从哪个出口送出去运行路由算法，计算出"到各目的地该走哪条路"，填出转发表
时间尺度纳秒级，每个分组都做秒/分钟级，拓扑变化时才重算
属于数据平面（data plane）控制平面（control plane，见第 5 章）
一句话记住：路由是"画地图、定路线"（planning），转发是"到了路口按指示牌拐弯"（doing）。转发靠的是路由算法预先算好、填进每台路由器的"转发表"。二者是"规划"与"执行"的关系。

互联网是"网络的网络"（network of networks）

网络核心并不是一整块，而是由成千上万个独立的 ISP（互联网服务提供商）网络互联而成——这正是"Inter-net（网络间的网络）"这个名字的由来。它们大致是分层的：

┌───────────────────────┐ │ 第一层 ISP (骨干网) │ 全球性,彼此免费对等(peering) └───────────┬───────────┘ ┌─────────────────────┼──────────────────────┐ ┌─────┴──────┐ ┌──────┴──────┐ ┌───────┴────────┐ │ 区域 ISP │ │ IXP │◄──────►│ 内容提供商网络 │ └─────┬──────┘ │ (互联网交换点)│ │ (Google/CDN 自建)│ ┌─────┴──────┐ └─────────────┘ └────────────────┘ │ 接入 ISP │ ← 电信/联通/校园网,直接连到你家 └─────┬──────┘ │ 你家 / 公司 (端系统 —— 又回到了网络边缘)
为什么要有 IXP 和内容提供商自建网？层层向上级 ISP 买流量既贵又绕远。IXP（互联网交换点）让多个 ISP 在一个物理机房里直接互联，抄近路省钱；而 Google、Netflix、阿里这类内容提供商干脆自建全球网络和 CDN，把内容缓存到离用户最近的接入 ISP 旁边——绕开拥塞的公共核心。这就是为什么访问大厂网站往往又快又稳。

五、把三者串起来：一次网页请求的旅程

现在用一个完整例子把边缘、接入网、核心缝合起来。你在笔记本上打开 example.com：

  1. 边缘（客户机）：浏览器（运行在你的端系统上）通过 Socket 把 HTTP 请求交给 OS 内核。
  2. 接入网：请求经家庭 WiFi → 光猫 → 接入 ISP，到达第一台边缘路由器。
  3. 核心：分组在若干 ISP 的路由器间被逐跳转发（每台按转发表选出口），可能经 IXP 抄近路。
  4. 边缘（服务器）：分组抵达数据中心的 Web 服务器（另一个端系统），服务器程序处理后原路返回响应。
想亲眼看到这段旅程？在终端敲 tracert example.com（Windows）或 traceroute example.com（Linux/macOS），每一行就是核心里被你的分组经过的一台路由器——你会真切看到"一跳一跳接力转发"，以及从接入 ISP 逐步汇入骨干的过程。这是本页最好的动手实验。

六、小结

一张地图记住整章：互联网 = 网络边缘（端系统/主机，客户机与服务器，你写的程序只活在这里）+ 接入网（最后一公里，分 DSL/HFC/FTTH/以太网/WiFi/蜂窝，关键区分共享 vs 专用、上下行是否对称）+ 网络核心（分组交换机/路由器织成的网状网，靠转发逐跳接力、靠路由规划路线，整体是"网络的网络"）。后面每一章都可以定位到这张图上：应用层在边缘、传输层与网络层在核心里穿行、链路层管好每一段接入与链路。
