DNS：层次命名、递归迭代查询、缓存

互联网的电话簿——把人类记得住的名字，翻译成机器认得的地址，还要撑住每秒千万次查询而不崩。

一、DNS 解决了什么问题：一张全世界共用的电话簿

机器之间靠 IP 地址 通信（如 142.250.72.14），但人类记不住一串数字，只想记 www.google.com。DNS（Domain Name System，域名系统） 干的就是这件事：把域名翻译成IP 地址。它是一个运行在应用层、由无数服务器协作构成的分布式数据库，同时也是一个查询这个数据库的应用层协议。

历史小考据：为什么不用一个大文件？ARPANET 早期，全网主机名到 IP 的映射就存在一个 HOSTS.TXT 文件里，由斯坦福统一维护，各主机定期下载。主机一多，这套集中式方案立刻崩了：单点故障、维护量爆炸、命名冲突、更新延迟。DNS 就是它的"分布式 + 层次化"替代品——没有任何一台服务器掌握全部映射。（你今天电脑里的 /etc/hosts 正是那份古董文件的遗骸。）
建立联系：DNS 是"典型的关键基础服务"。它不像 HTTP 那样直接面向用户，却被几乎所有应用间接依赖——你 new Socket("api.xx.com", 443)、浏览器打开网页、微服务互相调用，第一步几乎都是 DNS 解析。所以后面会看到：DNS 慢，一切都慢；DNS 挂，半个互联网跟着挂（2016 年 Dyn 被 DDoS，Twitter/GitHub 集体宕机就是例证）。

二、层次命名：域名是一棵"从右往左"读的树

DNS 的第一个精妙设计是层次化命名空间。把域名想象成一棵倒挂的树，最右边的点 . 是根（root），越往左越具体。www.cs.umass.edu 读作：根 → edu 顶级域 → umass → cs → 主机 www。

. (根 root，逻辑上唯一) ┌──────────────┼───────────────┐ com edu cn ← 顶级域 TLD ┌──┴───┐ │ ┌───┴───┐ google amazon umass edu com ← 二级域(可注册) │ │ │ www cs … tsinghua │ www → 完整域名 www.cs.umass.edu

为什么要分层？因为分层就能分权、分而治之：根只需要知道"各个 TLD 归谁管"，.edu 的服务器只需知道"umass.edu 归谁管"，而 umass 自己维护 cs.umass.edu、www 这些主机记录。每一级只对下一级负责，管理权层层下放——这和协议分层、和公司的组织架构、和文件系统的目录树，是同一种治理复杂度的思想。

对应的服务器也是四层结构

服务器角色职责由谁运营
根 DNS 服务器不知道具体域名的 IP，只告诉你"某个 TLD 该去问谁"13 个逻辑根（a~m.root-servers.net），实际由 anycast 部署成上千台物理机
TLD 服务器管理某个顶级域（com/org/cn…）下所有二级域的权威服务器地址如 Verisign 管 .com
权威 DNS 服务器存放某组织自己域名的最终记录（真正的 IP）组织自建，或托管给云厂商（Route 53、Cloudflare 等）
本地 DNS 服务器
(递归解析器)		不在层次树里，是主机的"代理人"，替你跑完整个查询流程并缓存结果ISP / 公司，或公共 DNS（8.8.8.8、114.114.114.114）
13 个根服务器 = 只有 13 台机器？错。13 是"逻辑标识"数量（受早期 UDP 512 字节响应上限约束）。每一个都通过 Anycast（任播） 复制成全球成百上千个物理节点，路由会把你的请求送到最近的那台。所以"打掉 13 台就断网"是彻底的误解。

三、递归查询 vs 迭代查询：谁替谁跑腿？

这是本节最核心、也是面试最爱问的一点。DNS 查询有两种"责任模式"：

  • 递归查询（recursive）：我把问题全权委托给你，你不查到最终答案不许回来。"帮我查到底。"
  • 迭代查询（iterative）：我问你，你不知道就把线索给我，我自己再去问下一个。"我不管，但你可以去问它。"

现实中最典型的组合是：主机 → 本地 DNS 用递归（你只想要答案，不想自己跑腿），本地 DNS → 根/TLD/权威 用迭代。下面用课本经典例子走一遍：主机 cis.poly.edu 想解析 gaia.cs.umass.edu。

┌───────────────┐ ② 迭代 ─────▶│ 根 DNS 服务器 │ ┌───────────────── │ "去问 edu" │◀── ③ │ ③ 返回 edu TLD 地址└───────────────┘ │ ┌───────────────┐ ┌──────────┐ │ ④ 迭代 ──▶│ TLD 服务器 edu │ │ 主机 │ ▼ ⑤ 返回umass权威 │ "去问 umass" │◀── ⑤ │cis.poly │ ① 递归 ┌──────────────┐ └───────────────┘ │ .edu │────────▶│ 本地 DNS │ ┌───────────────┐ │ │◀────────│ (递归解析器) │──▶│ 权威 DNS 服务器 │ ⑥迭代 └──────────┘ ⑧ 返回IP└──────────────┘◀──│ dns.umass.edu │ ▲ └───────────────┘ ⑦返回 └── ⑦ gaia = 128.119.245.12 ──┘

整个过程一共 8 条报文：① 主机把"帮我查到底"的递归请求丢给本地 DNS；② ~ ⑦ 本地 DNS 拿着这个任务，迭代地依次问根、TLD、权威，每一步只拿到"下一跳该问谁"，直到权威服务器吐出真正的 IP；⑧ 本地 DNS 把答案回给主机。

陷阱 / 高频考点：为什么本地 DNS 对上游用迭代，而不是让根服务器也递归帮忙查到底？因为如果每台本地 DNS 都要求根服务器"递归查到底"，根服务器就得替全世界跑完整个流程，瞬间被压垮。迭代把跑腿的活留给数量众多、分散的本地 DNS，让顶层服务器只做"指路"这一件轻量的事。这是一个用架构分摊负载的经典权衡，不是技术限制。协议本身两种模式都支持，选哪种是"策略"问题。
递归查询（主机 → 本地 DNS）
问："www.a.com 的 IP？"
答："176.32.103.205"   ← 直接给终值
· 客户端省心，一问一答
· 跑腿压力全压在被问方
· 用于 主机↔本地DNS
迭代查询（本地 DNS → 上游）
问 根 ："www.a.com 的 IP？"
答    ："我不知道，去问 .com"  ← 只给线索
问 TLD："…？" 答："去问 a.com 权威"
问 权威："…？" 答："176.32.103.205"
· 分散负载，保护顶层服务器

四、缓存：DNS 能扛住互联网流量的真正秘密

如果每次解析都从根跑一遍，根服务器和你的体验都会崩。缓存（caching） 才是让 DNS 实际可用的关键：任何 DNS 服务器只要拿到一条映射，就会把它暂存一段时间，下次直接命中，不再向上查询。

缓存多久？由记录自带的 TTL（Time To Live，生存时间，单位秒） 决定。TTL 一到，缓存条目就被丢弃。绝大多数热门域名的记录，在本地 DNS 里几乎永远是热的——所以你日常访问网站，DNS 这一步通常只花几毫秒。

未命中：解析时延 ≈ RTTlocal + RTTroot + RTTTLD + RTT权威
命 中：解析时延 ≈ RTTlocal （其余全省）
命中 ~2ms 未命中 ~60ms+（跨越根/TLD/权威多次 RTT）
缓存也缓存"坏消息"——负缓存（negative caching）。如果一个域名不存在（NXDOMAIN），这个"查无此名"的结果也会被缓存一小段时间，避免有人反复查一个错域名把上游打爆。这在防御拼写错误型攻击、以及排查"域名明明建好了却还解析不到"时很关键。
陷阱：TTL 是把双刃剑，切换服务器前务必提前调低。TTL 设得大（如 86400 秒 = 1 天）→ 缓存命中率高、上游压力小，但你一旦改了 IP，老 IP 会被全世界缓存整整一天，用户仍打到已下线的旧机器。TTL 设得小（如 60 秒）→ 切换快、故障转移灵敏，但查询量激增。正确做法：计划迁移前几天先把 TTL 调低，迁移完成、观察稳定后再调回。

五、常见资源记录（RR）：DNS 里到底存了什么

DNS 数据库的每一条记录叫 资源记录（Resource Record），格式是四元组 (Name, Value, Type, TTL)。Type 决定了 Name 和 Value 该怎么解读——这是理解 DNS 的最后一块拼图。

类型Name → Value 含义典型用途
A主机名 → IPv4 地址最常见，域名解析到 IPv4
AAAA主机名 → IPv6 地址解析到 IPv6
NS域 → 该域权威服务器的主机名层次委派的"指针"，迭代查询靠它
CNAME别名 → 规范名（canonical name）把多个名字指向同一个真实域名，便于统一改 IP
MX域 → 邮件服务器的规范名邮件路由（SMTP 靠它找收信服务器）
TXT域 → 任意文本域名归属验证、SPF/DKIM 反垃圾邮件
陷阱：根域名（apex/裸域）上不能放 CNAME。RFC 规定 CNAME 记录不能与同名的其它记录（如 SOA、NS）共存，而裸域 example.com 必然带有 SOA/NS。所以你没法把 example.com 直接 CNAME 到 xxx.cloudfront.net——只能给 www.example.com 这种子域用 CNAME，或改用云厂商的 ALIAS/ANAME 这类私有扩展。很多人第一次配 CDN 都栽在这。

六、协议细节 + 代码视角

DNS 报文默认跑在 UDP 端口 53 上——因为一次查询就是"小请求 + 小响应"，用 UDP 省去 TCP 三次握手，快。但当响应过大（超过 512 字节、DNSSEC、或做区域传送 zone transfer）时，会回退到 TCP。这是"UDP 为主、TCP 兜底"的经典设计。

命令行：用 dig 观察一次真实解析shell
# +trace 让 dig 模拟本地 DNS，从根开始逐级迭代查询
$ dig www.cs.umass.edu

;; ANSWER SECTION:
www.cs.umass.edu.   21599  IN  A   128.119.240.84
#                   ↑TTL(秒)     ↑Type    ↑Value(IP)
# 再查一次，TTL 会变小 —— 说明命中了缓存、且在倒计时

作为 Java 程序员，你几乎从不手写 DNS 报文——java.net.InetAddress 封装了一切。当你解析域名时，JVM 委托操作系统 resolver 完成上面整套流程：

DnsDemo.java — InetAddress 背后就是一次 DNS 解析Java
InetAddress addr = InetAddress.getByName("www.cs.umass.edu");
System.out.println(addr.getHostAddress());  // 128.119.240.84

// 一个域名可能有多条 A 记录（DNS 轮询做负载均衡）
for (InetAddress a : InetAddress.getAllByName("www.google.com")) {
    System.out.println(a.getHostAddress());
}
JDK 高频大坑：JVM 自己还有一层 DNS 缓存，且默认策略常年坑人。InetAddress 在进程内缓存解析结果，由 java.security 里的 networkaddress.cache.ttl 控制。历史上，一旦安装了 SecurityManager，默认值是 -1（永久缓存，进程内绝不刷新）。后果：如果你连的是"域名背后 IP 会漂移"的服务（AWS RDS 故障转移、ELB、K8s Service），JVM 会死抱着一个已经失效的旧 IP 直到重启——即使 DNS 早就更新了。排查这类"重启就好"的诡异连接故障，先看这个属性。
✗ 反例：放任 JVM 无限缓存旧 IP
// 什么都不配，靠默认
// SecurityManager 下 ttl = -1 → 永久缓存
InetAddress.getByName("db.rds.aws.com");
// RDS 主备切换后 IP 变了
// 应用仍连旧 IP → 连接全部失败
// "重启 JVM 才恢复" ✗ 生产事故
✓ 正例：显式设定一个合理的正/负缓存 TTL
// 启动早期设置（或写进 java.security）
Security.setProperty(
    "networkaddress.cache.ttl", "30");        // 正缓存 30s
Security.setProperty(
    "networkaddress.cache.negative.ttl", "5");// 负缓存 5s
// 30s 后重新解析 → 自动跟上 IP 漂移 ✓
// 既不打爆 DNS，又能及时故障转移
一句话收束：DNS 靠三招撑起了整个互联网的命名。① 层次命名——把一棵域名树拆给根/TLD/权威分级管理，谁都不必掌握全局；② 递归+迭代——主机把活递归甩给本地 DNS，本地 DNS 再迭代地"逐级问路"，把负载从顶层服务器分摊开；③ 缓存 + TTL——绝大多数查询命中缓存、只花几毫秒，TTL 则在"新鲜度"和"查询压力/切换速度"之间做权衡。记住这三点，再叠加"UDP:53 兜底 TCP""JVM 自带一层缓存要小心"两个工程细节，DNS 就通了。
