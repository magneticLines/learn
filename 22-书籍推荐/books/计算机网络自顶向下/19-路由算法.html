路由算法：Dijkstra、距离向量、RIP、OSPF

数据平面负责"照着转发表把包扔出去"，那张转发表从哪来？控制平面用路由算法在一张加权图上求最短路——两大流派（集中式 Dijkstra / 分布式距离向量）分别长成了 OSPF 和 RIP。

一、先分清：数据平面 vs 控制平面

第 4 章我们讲的是数据平面——路由器收到一个包，查转发表（forwarding table），按目的 IP 的最长前缀匹配决定从哪个端口发出去。这一步极快、纯本地、按包发生。但转发表里那些"到 10.2.0.0/16 走 3 号端口"的条目是谁算出来的？这就是控制平面的活：运行路由算法，在全网拓扑上算出每个目的地的最短路径，再把结果"编译"成每台路由器的转发表。

一句话类比：数据平面像司机看导航照着开（毫秒级、每个路口都在做），控制平面像导航后台算路线（分钟级、拓扑变了才重算）。路由算法算出的"最短路"是结果，转发表是把结果拆成"在每个路口该往哪拐"的本地指令。别把二者混为一谈——这是第 4、5 章的分界线。

把网络抽象成一张图，路由问题就变成了教科书里的最短路径问题：

图 G = (N, E)　节点 N=路由器　边 E=链路　c(x,y) = 链路代价

代价 c(x,y) 可以是传播时延、也可以是"1/带宽"、还可以人为配置成 1（那最短路就退化成最少跳数）。算法不关心代价的物理含义，只管在图上求 源到所有点 的最小代价路径。按"谁掌握拓扑信息"分成两大流派：

流派信息掌握算法内核代表协议
链路状态 LS（集中式）每个节点知道全局拓扑（靠泛洪广播自己的链路）DijkstraOSPF
距离向量 DV（分布式）每个节点只知道到邻居的代价，靠交换"距离表"逐步收敛Bellman-FordRIP、BGP

二、Dijkstra（链路状态）：手握全局地图求最短路

链路状态算法的前提：每台路由器都把"我到各个邻居的链路代价"泛洪给全网，于是人人手里都有一张完整的地图。有了完整地图，就能各自独立跑 Dijkstra 算出到所有目的地的最短路——这也是它叫"链路状态"的原因：广播的是链路状态，不是算好的距离。

Dijkstra 是贪心算法：维护一个"已确定最短路的节点集合 N′"，每一轮从"尚未确定"的节点里挑当前距离最小的那个加入 N′（因为边权非负，它的距离不可能再变小），然后用它去松弛（relax）邻居。

无向带权图 G = (N, E) N = {u, v, w, x, y, z} 链路 代价 c() 我们从源点 u 出发求最短路 u — v 2 u — w 5 u — x 1 v — w 3 v — x 2 w — x 3 w — y 1 w — z 5 x — y 1 y — z 2

把 Dijkstra 一步步跑出来（这是《自顶向下》图 5.4 的经典表，D(),p() 表示当前已知最短距离,前驱节点）：

步N′（已定）D(v),pD(w),pD(x),pD(y),pD(z),p
0u2,u5,u1,u∞∞
1ux2,u4,x—2,x∞
2uxy2,u3,y——4,y
3uxyv—3,y——4,y
4uxyvw————4,y
5uxyvwz全部确定，结束

每步"加粗"的就是被选进 N′ 的节点。注意第 1 步：把 x（距离 1）纳入后，用它松弛 w：原来 D(w)=5（u→w 直连），经 x 是 1+3=4 更短，于是 D(w) 更新为 4,x；y 从 ∞ 变成 2,x。跑完得到从 u 出发的最短路径树：

从 u 出发的最短路径树（括号内为到 u 的总代价） (u) / \ 1/ \2 / \ (x)=1 (v)=2 | 1| | (y)=2 / \ 1/ \2 / \ (w)=3 (z)=4 u→z 最短路 = u-x-y-z = 1+1+2 = 4
陷阱：Dijkstra 要求边权非负。贪心的正确性建立在"一旦某节点被选入 N′，它的距离就永不再变小"——这依赖非负边权。路由代价（时延、1/带宽）天然非负，所以没问题；但如果面试官问"能有负权吗"，答案是不能，负权要用 Bellman-Ford。另一个易错点：选最小时的平局（第 1 步后 v 和 y 都是 2）可以任选，最终各点最短距离一致，但最短路径树的形状可能不同。

用 Java 写一遍 Dijkstra（连接 JDK：PriorityQueue 就是二叉堆）

朴素 Dijkstra 每轮"线性扫描找最小"是 O(V²)；工程实现用最小堆把"取最小"降到 O(log V)。Java 里现成的堆就是 java.util.PriorityQueue——它底层是一个二叉小根堆（数组实现的完全二叉树），offer/poll 都是 O(log n)。这正好是我们在数据结构里学的堆的最经典应用：

链路状态算法核心：堆优化 DijkstraJava
// graph[u] 里每个元素是 {邻居v, 边权w}；返回 src 到各点的最短距离
int[] dijkstra(int src, List<int[]>[] graph, int n) {
    int[] dist = new int[n];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[src] = 0;
    boolean[] done = new boolean[n];        // done[] 就是集合 N′

    // 堆里放 {到源点距离, 节点}，按距离升序出堆
    PriorityQueue<int[]> pq =
        new PriorityQueue<>((a, b) -> Integer.compare(a[0], b[0]));
    pq.offer(new int[]{0, src});

    while (!pq.isEmpty()) {
        int u = pq.poll()[1];               // 贪心：全局最近的未确定节点
        if (done[u]) continue;               // 惰性删除：跳过堆里的陈旧副本
        done[u] = true;                       // 纳入 N′，其最短路已定
        for (int[] e : graph[u]) {          // 松弛 u 的每个邻居
            int v = e[0], w = e[1];
            if (!done[v] && dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;      // relax：发现更短路
                pq.offer(new int[]{dist[v], v});
            }
        }
    }
    return dist;
}

为什么"取最小"用堆而不是每轮遍历全部节点？看这组对比——稀疏图上（真实网络链路数 E 远小于 V²）差一个数量级：

反例 — 线性扫描选最小 O(V²)
while (还有未确定节点) {
  int u = -1, best = INF;
  for (int i=0; i<n; i++)   // 每步都
    if (!done[i] && dist[i]<best)
      { best=dist[i]; u=i; } // 扫全表
  done[u]=true;
  // relax(u)...
}
// V 轮 × 每轮扫 V = O(V²)
// 稀疏图(E≈V)白白扫，浪费
正例 — 最小堆维护候选 O(E log V)
PriorityQueue<int[]> pq = ...;
while (!pq.isEmpty()) {
  int u = pq.poll()[1];  // O(log V)
  if (done[u]) continue;
  done[u]=true;
  // 每条边至多入堆一次
  // relax → offer O(log V)
}
// 总 O(E log V)，稀疏图快一档
为什么可以"惰性删除"而不用 decreaseKey？JDK 的 PriorityQueue 没有高效的"减小某元素优先级"操作。工程上不去改堆里的旧值，而是直接再 offer 一个更小的新副本；旧的大副本 poll 出来时，发现 done[u] 已是 true，一句 continue 跳过即可。代价是堆里可能有 O(E) 个副本，但复杂度仍是 O(E log V)，实现却简单得多——这是竞赛/生产里最常见的写法。

三、距离向量（Bellman-Ford）：只跟邻居聊天也能收敛

链路状态要求"人人有全局地图"，代价是全网泛洪。距离向量反其道而行：每个节点只跟直连邻居交换"我到各目的地的距离估计"，没有任何节点掌握全局拓扑，却能靠迭代收敛出正确的最短路。它的数学基石是Bellman-Ford 方程：

dx(y) = minv { c(x,v) + dv(y) }

翻译成大白话：我 x 到目的地 y 的最短距离，等于"我到某个邻居 v 的代价 + 邻居 v 号称它到 y 的距离"，在所有邻居 v 里取最小。这是一种动态规划——每个节点把邻居报来的距离向量代进方程重算一遍自己的距离向量，如果结果变了就再广播给邻居，如此往复直到没人再变（收敛）。

类比：问路接力。你不知道去火车站怎么走，就问路口每个方向的邻居"从你那到火车站还要多久"，再加上"我到你那要多久"，取最快的一条。邻居也是这么问它的邻居的。一开始大家估计都很粗糙，但每轮交换后信息就多传播一跳，网络直径是几跳，最多几轮就收敛。这就是 DV"分布式、迭代、异步"的精髓——没有上帝视角，靠口口相传。
维度链路状态 LS（Dijkstra）距离向量 DV（Bellman-Ford）
掌握信息全局拓扑（泛洪链路状态）仅到邻居代价 + 邻居报的距离
消息内容广播链路状态给全网只把距离向量发给邻居
消息复杂度O(|N|·|E|)，泛洪较重只跟邻居交换，本地开销小
收敛速度快、可预期（各自算一遍）慢、可能震荡（依赖迭代）
故障影响算错只影响自己的表错误距离会扩散给邻居
代表协议OSPF、IS-ISRIP、BGP（路径向量）

DV 的致命陷阱：无穷计数（count-to-infinity）

DV 有个著名毛病：好消息传得快，坏消息传得慢。链路代价变小时几轮就传遍；但某条链路变大或断掉时，节点之间会互相引用对方那条已经过时的路径，形成路由环路，距离一点一点往上加，一直"数到无穷"。

三节点例子： y ——4—— x c(y,x)=4, c(z,x)=50, c(y,z)=1 \ / 此时 d_y(x)=4(直连), d_z(x)=5(经 y) 1\ /50 \ / (z) ╭─ 链路 y–x 代价 4 → 60（变差） ─────────────────────────────╮ │ y 重算: d_y(x)=min{60, 1+d_z(x)=1+5=6} = 6 ← 竟然"经 z 更近"│ │ （错！z 的那个 5 本来就是绕经 y 算出来的 → 成环） │ │ y 告诉 z: d_y(x)=6 → z 重算 d_z(x)=1+6=7 → 告诉 y → 8 ... │ │ 就这样每轮 +1，慢慢数到 50，z 才改用直连 c(z,x)=50 收敛 │ ╰──────────────────────────────────────────────────────────╯
面试高频：无穷计数怎么缓解？——毒性逆转（poisoned reverse）。既然环路源于"z 把'经 y 到 x'这条路又报给了 y"，那就让 z 对 y 撒个谎：如果 z 去 x 的下一跳是 y，z 就向 y 谎报 d_z(x) = ∞。这样 y 永远不会傻到"经 z 到 x"，两节点环路被打破。但要点睛一句：毒性逆转只能解决两节点之间的环路，涉及三个及以上节点的环仍可能计数到无穷——这正是 RIP 要把"无穷"定义成一个很小的数（16）来兜底的原因。

四、RIP：距离向量的"极简实现"

RIP（Routing Information Protocol）是距离向量思想最朴素的落地，也是最早的内部网关协议之一。它把代价简化到极致——度量就是跳数（hop count），每经过一台路由器 +1，跟带宽、时延统统无关。

  • 度量 = 跳数，最大 15；16 表示不可达（无穷）——这就是上面无穷计数的"封顶值"，把"数到无穷"限制在 16 步内结束。代价是 RIP 只能用于直径 ≤ 15 跳的小网络。
  • 周期性广播：每 30 秒把整张路由表（RIP Response 报文）发给邻居；若 180 秒没收到某邻居的更新，就认为它挂了、把相关路由标记为不可达。
  • 跑在 UDP 之上，端口 520——注意 RIP 是应用层进程（如 Unix 的 routed 守护进程）在维护路由，而不是内核直接干，这点很反直觉。
  • 用水平分割 + 毒性逆转缓解环路；RIPv2 增加了子网掩码（支持 CIDR）和认证，RIPng 支持 IPv6。
直觉记法：RIP 就是"数格子"。它不管哪条路更宽更快，只管"少走几个路口"。所以一条 1Gbps 但多一跳的路，RIP 会嫌它"远"而不选——这在真实网络里往往是错的。度量过于粗糙 + 收敛慢 + 15 跳上限，让 RIP 如今基本退居教学和超小型网络，生产环境早被 OSPF 取代。

五、OSPF：链路状态的"工业级实现"

OSPF（Open Shortest Path First）是链路状态算法的主力实现，也是当今企业/运营商内部网络的事实标准。每台 OSPF 路由器周期性地（以及拓扑变化时触发式）向整个区域泛洪自己的链路状态通告（LSA），于是区域内人人都拼出完整拓扑图，再各自跑 Dijkstra 算最短路。

  • 度量可配，通常按 参考带宽 / 接口带宽 计算——带宽越高代价越低，天然偏好高速链路（这正是 RIP 做不到的）。
  • 直接跑在 IP 之上，协议号 89（不经 TCP/UDP），自己实现可靠泛洪与确认。
  • 分层区域（area）：大网络切成若干区域，所有区域都连到骨干区域 area 0，区域边界路由器（ABR）之间只交换汇总信息。这让 OSPF 能扩展到大规模网络——泛洪被限制在区域内，不会全网风暴。
  • 内建认证（明文/MD5），防止伪造 LSA 投毒；支持等价多路径（ECMP）做负载均衡。

RIP 和 OSPF 都是自治系统内部的路由协议（IGP），但取舍截然不同——这张表是高频考点：

别混：IGP vs EGP。RIP 和 OSPF 都是内部网关协议（IGP）——只在一个自治系统（AS）内部算路，追求的是"技术最优（最短/最快）"。而AS 与 AS 之间的路由是另一套逻辑，用的是BGP（下一节的主角），它是路径向量协议，且更看重商业策略而非最短路径。把"公司内部找最优路"和"运营商之间按合同选路"分开，是理解整个互联网路由体系的关键。

六、串联操作系统：算完的路去哪了？

路由进程（如 ospfd/routed）算出最短路后，会把结果写进操作系统内核的路由表（FIB），真正转发时内核按最长前缀匹配（第 4 章）查这张表。你在 Linux 上看到的正是这张被路由算法"喂"出来的表：

控制平面(算路) → 内核路由表 → 数据平面(转发)Shell
$ ip route
default via 192.168.1.1 dev eth0            # 默认路由(0.0.0.0/0)
10.2.0.0/16 via 10.0.0.9 dev eth1 metric 20  # ← OSPF 算出的下一跳
192.168.1.0/24 dev eth0 scope link         # 直连网段

# metric 就是路由代价；有多条路时内核选 metric 最小的
# 这些条目 = 路由算法(控制平面)算完写进来的"结果"
# 每个进来的包按目的 IP 做最长前缀匹配 = 数据平面转发
回收前文，形成闭环：Dijkstra 在图上求出"u→z 最短路是 u-x-y-z"，落到 u 的转发表就只剩一条本地指令——"去 z 的包，从连着 x 的那个端口发出"。路由算法算的是完整路径，转发表存的只是下一跳。为什么只存下一跳？因为最短路径树有个漂亮性质：最短路的子路径也是最短路，所以每台路由器只需各自把"下一跳"填对，包就会沿着全局最短路一站站被接力送达——这与我们在 Bellman-Ford 里看到的最优子结构是同一件事。
一句话收束：路由算法在加权图上求最短路，分两大流派——链路状态 LS 泛洪拓扑、人人有全局地图、各自跑 Dijkstra（贪心 + 非负边权 + 堆优化 O(E log V)），实现即 OSPF（度量按带宽、分区域、over IP 协议号 89、触发式泛洪）；距离向量 DV 只跟邻居交换距离、靠 Bellman-Ford 迭代收敛，实现即 RIP（度量=跳数、上限 15、16=无穷、UDP 520、30s 周期广播），它的软肋是无穷计数，靠毒性逆转与"16 封顶"缓解。记牢三条主线：控制平面算路 → 内核路由表 → 数据平面按最长前缀转发；LS 快而重、DV 轻而慢；RIP/OSPF 是 AS 内的 IGP，AS 之间才轮到 BGP。
