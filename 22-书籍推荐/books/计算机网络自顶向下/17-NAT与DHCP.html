NAT 与 DHCP

两个协议撑起了几乎每一个家庭和办公室网络：DHCP 让你插上网线就能上网，NAT 让一整栋楼共享同一个公网 IP。

一、两个协议，回答两个"痛点"

上一节我们讲清了 IP 编址：每台主机要通信，必须有一个 IP 地址、一个子网掩码、还得知道默认网关和 DNS。问题随之而来——这些地址从哪来？谁负责发？ 更棘手的是，全球 IPv4 地址只有 2^32 ≈ 43 亿 个，早在 2011 年就已分配殆尽，可全世界的设备远不止 43 亿台。

DHCP 和 NAT 就是对这两个痛点的直接回应，它们分工明确、又常常配对出现在同一台家用路由器里：

协议解决的问题一句话本质
DHCP
动态主机配置		手动给每台设备配 IP / 掩码 / 网关 / DNS 太痛苦，还容易冲突局域网里的"户籍科"：设备一接入就自动租给它一整套上网参数
NAT
网络地址转换		公网 IPv4 地址不够用，一个家庭 / 公司只买得起一个公网 IP局域网出口的"总机前台"：让一大群私有 IP 共用一个公网 IP 对外通信
为什么放在一起讲？ 它们是"即插即用网络"的两块基石，且都工作在你家那台无线路由器上。DHCP 负责对内——给屋里每台设备发私有地址；NAT 负责对外——把这些私有地址翻译成运营商给你的那唯一一个公网地址。DHCP 让"插上就能用"，NAT 让"一个够全家用"。

二、DHCP：局域网里的"自动户籍科"

DHCP（Dynamic Host Configuration Protocol，RFC 2131）是一个应用层协议，跑在 UDP 之上（服务器端口 67，客户端端口 68）。它的目标是：新设备接入网络时，不需要人工配置任何东西，就能自动获得一套完整的上网参数——IP 地址、子网掩码、默认网关、DNS 服务器地址。

核心流程：DORA 四步握手

这个过程被戏称为 DORA（Discover → Offer → Request → Ack）。关键难点在于：客户端此刻还没有 IP 地址，也不知道 DHCP 服务器在哪，所以只能靠广播喊话。

客户端(还没有IP) DHCP 服务器 src 0.0.0.0:68 (端口 67) ① DHCP Discover ──广播──► 255.255.255.255 "有人能给我个地址吗？" ② DHCP Offer ◄────── "192.168.1.23 给你，租期24h， 网关=192.168.1.1，DNS=192.168.1.1" ③ DHCP Request ──广播──► "我要 192.168.1.23 这个！"（广播=顺带谢绝其他服务器） ④ DHCP Ack ◄────── "成交，绑定登记完毕" ✔ 现在可以上网了
为什么是"两个来回"而不是"发现即分配"？ 因为一个网络里可能有多台 DHCP 服务器，它们都会响应 Discover 各发一个 Offer。第 ③ 步的 Request 用广播发出，作用有二：一是正式向选中的服务器"确认下单"，二是让没被选中的服务器听到、从而收回它们预留的地址。这是一个典型的"提议—确认"两阶段设计，避免多台服务器把同一批地址锁死。
联系上一节 UDP：为什么 DHCP 只能用 UDP 而不能用 TCP？ 想想 TCP 需要三次握手，而握手要求双方都有明确的 IP 地址——可 DHCP 客户端此刻恰恰还没有 IP！而且它连服务器地址都不知道，只能往 255.255.255.255 广播。而 UDP 页讲过：广播 / 多播只能建在 UDP 上，TCP 天生做不到一对多。所以 DHCP 用 UDP 是被"无地址 + 需广播"这两个硬约束逼出来的必然选择。

租约（Lease）：地址是"租"来的，不是"送"的

DHCP 分配的地址带一个租期。这不是抠门，而是为了回收：笔记本、手机来来去去，若地址只发不收，很快就会耗尽。客户端通常在租期过半（T1 时刻）就用单播向原服务器续租，续租失败再到 T2（约 87.5%）广播寻求任意服务器 rebinding。

初学者高频误区 & 面试点：
· "DHCP 只发 IP 地址"——错。 它同时下发子网掩码、默认网关、DNS 服务器，这四样缺一不可，少了网关你连不出局域网，少了 DNS 你打不开任何域名。
· Discover 是广播，广播出不了本网段。 大企业不可能每个子网都放一台 DHCP 服务器，靠的是DHCP 中继代理（relay agent）：它把本网段的广播请求转成单播，转发给中心 DHCP 服务器，并在报文里填上 giaddr 告诉服务器"该从哪个子网的地址池里分配"。
· DHCP 无认证，存在欺骗风险。 恶意主机伪装成 DHCP 服务器抢先应答（DHCP spoofing），把网关 / DNS 指向自己，即可中间人劫持。交换机上的 DHCP Snooping 就是为此而生。

三、NAT：一个公网 IP 背后藏着整个局域网

DHCP 发给你屋里设备的地址，长得都像 192.168.x.x 或 10.x.x.x——这些是私有地址（RFC 1918），被约定"只在局域网内部有效、绝不会出现在公网路由表里"，因此全世界的局域网可以重复使用同一批私有地址而互不干扰。

私有地址段CIDR地址数量常见于
10.0.0.0 – 10.255.255.25510.0.0.0/8约 1678 万大型企业 / 云 VPC
172.16.0.0 – 172.31.255.255172.16.0.0/12约 104 万Docker 默认网段
192.168.0.0 – 192.168.255.255192.168.0.0/1665536家用路由器

可私有地址无法在公网被路由，那屋里的设备怎么访问外网？答案是出口路由器上的 NAT（Network Address Translation）。现实中家用路由器用的其实是 NAT 的加强版——NAPT（网络地址端口转换，也叫 PAT / IP 伪装）：它不仅改 IP，还改端口，从而让成百上千台内网设备复用同一个公网 IP。

它是怎么"翻译"的：一张 NAT 转换表

NAT 的全部秘密，就在路由器内存里维护的一张转换表。出方向改写源地址并登记，入方向查表反向还原：

内网主机 NAT 路由器(公网IP 138.76.29.7) 公网服务器 192.168.1.10:3345 ──出──► 改写源地址: 192.168.1.10:3345 ➜ 138.76.29.7:5001 ──► Web 服务器:80 并登记转换表 ┐ ▼ ┌─────────── NAT 转换表 ───────────┐ │ 内网侧 公网侧 │ │ 192.168.1.10:3345 ⇄ 138.76.29.7:5001 │ │ 192.168.1.10:3346 ⇄ 138.76.29.7:5002 │ │ 192.168.1.11:3345 ⇄ 138.76.29.7:5003 │ ← 注意内网端口相同也不冲突 └──────────────────────────────────┘ ▲ 回包目的地= 138.76.29.7:5001 ◄─入── 查表反向改写: 138.76.29.7:5001 ➜ 192.168.1.10:3345 ◄── Web 服务器:80

注意上表第 1 行和第 3 行：两台不同内网主机恰好用了相同的源端口 3345，NAT 靠给它们分配不同的公网端口（5001 / 5003）来区分。这正是"端口"作为复用手段的又一次登场——和传输层多路复用同源。

NAT 破坏了分层原则，但很实用。 严格说，IP 是网络层设备，本不该关心传输层的端口；但 NAPT 为了复用，硬是伸手去改写 TCP/UDP 头部里的端口号——这是一次教科书级的"跨层侵入"。它换来了 IPv4 的续命，代价是让"端口"这个传输层概念泄露到了网络层设备中。

一个公网 IP 到底能扛多少并发？

既然靠端口区分连接，而端口是 16 位，那么单个公网 IP 上的 NAT 映射数就有一个理论上限：

并发映射上限 = 216 − 保留端口 ≈ 65535 条 / 每个公网 IP

《自顶向下方法》里给出的经典数字是"一个 WAN 侧地址可支持约 6 万多个并发连接"。这对一个家庭绰绰有余，但对运营商级 NAT（CGNAT，成千上万用户共享一个公网 IP）就可能成为瓶颈——端口耗尽时，新连接会直接建不起来。

四、NAT 的"原罪"：它打破了端到端原则

NAT 极其成功，却也是网络工程界争议最大的东西之一。核心矛盾在于——它违背了我们在 UDP 一节反复强调的端到端原则。

回顾端到端原则： 理想的互联网中，任意两台主机应能直接、对等地互相寻址通信。但 NAT 后面的主机没有公网身份——它只有一个私有地址，对外界"隐身"。外界无法主动发起一个连接到 NAT 内部的主机，因为转换表里还没有对应的条目。于是 NAT 把互联网从"人人平等"悄悄改成了"内网只能主动出击，不能被动接客"。这直接冲击了 P2P、在线游戏、VoIP、点对点文件共享这类需要被连接的场景。

怎么"穿透"NAT？

手段原理适用场景

(Port Forwarding)		在路由器手工静态配置"公网某端口 → 内网某主机:端口"自建家用服务器、NAS

(Hole Punching)		双方各自先向对方的公网映射发包，"骗"两端 NAT 各建一条出方向条目，从而互相可达P2P 直连
TURN 中继打洞失败时，找一台公网服务器做数据中转（牺牲直连，保证连通）ICE 框架的兜底方案
陷阱：某些协议把 IP 地址写进了"数据里"，NAT 改不到。 比如 FTP 主动模式会在应用层报文中携带自己的私有 IP 和端口，让服务器回连——可 NAT 只改了 IP 头和 TCP 头，没改载荷里的那个私有地址，于是服务器拿着一个内网地址回连，必然失败。SIP（VoIP 信令）也有同样的毛病。为此路由器不得不引入 ALG（应用层网关）去深入解析并改写载荷——这又是一次无奈的跨层妥协。

五、代码：从 JDK 视角看 DHCP 与 NAT

把抽象拉回代码。当你在 Java 里调 InetAddress.getLocalHost()，拿到的其实就是开机时 DHCP 分配、被操作系统协议栈记录下来的那个私有地址——而你永远看不到自己的公网 IP，因为 NAT 映射发生在网关上，主机本身对此一无所知。

本机能看到的只有私有地址Java
import java.net.*;
import java.util.Enumeration;

public class WhoAmI {
    public static void main(String[] args) throws Exception {
        // DHCP 早在开机时就把地址塞给了网卡，
        // JDK 只是从操作系统协议栈把它读出来
        InetAddress local = InetAddress.getLocalHost();
        System.out.println("内网 IP(DHCP 分配): " + local.getHostAddress());
        // 典型输出: 192.168.1.23  —— 一个 RFC1918 私有地址

        // 遍历所有网卡，看到的同样全是私有地址：
        // NAT 的公网映射在网关路由器上，主机根本无从得知
        Enumeration<NetworkInterface> nis = NetworkInterface.getNetworkInterfaces();
        while (nis.hasMoreElements()) {
            NetworkInterface ni = nis.nextElement();
            for (InterfaceAddress ia : ni.getInterfaceAddresses()) {
                System.out.println(ni.getName() + " -> "
                        + ia.getAddress().getHostAddress());
            }
        }
        // 想知道自己的公网 IP？只能去问外部服务器"你看到的源地址是多少"：
        //   curl ifconfig.me  →  返回的才是 NAT 改写后的公网 IP
    }
}

坏味道 → 正确姿势：别把私有 IP 当成对外可达地址

这是分布式 / 微服务里极其真实的坑。服务向注册中心（Eureka / Nacos）上报自己的地址时，若直接用 getLocalHost()，报的是 192.168.x.x；一旦调用方在另一个 NAT / 网段 / 公网，就永远连不上它。

反例 — 上报私有地址
// 服务注册时直接用本机地址
String host = InetAddress
    .getLocalHost()
    .getHostAddress();
registry.register("order-svc",
        host, 8080);
// 上报了 192.168.1.23:8080
// 跨 NAT 的调用方拿到后
// 连的是它自己网段的 .1.23，
// 要么连错机器、要么超时
正例 — 显式声明对外地址
// 用配置 / 环境变量注入外部地址
// (公网 IP 或经 STUN 探测的映射)
String host = env.get(
    "SERVICE_ADVERTISED_HOST");
registry.register("order-svc",
        host, 8080);
// 对应 Spring Cloud 里的
//   eureka.instance.ip-address
//   / prefer-ip-address
// 让"广播出去的地址"= 别人真能连的地址
同一个坑的另一副面孔——反向代理 / 负载均衡后的"真实客户端 IP"。 NAT 和代理都会改写源地址，于是 Spring MVC 里 request.getRemoteAddr() 拿到的往往是代理 / 网关的地址，而非真实用户。解决办法是让代理带上 X-Forwarded-For 头，再用 Spring 的 ForwardedHeaderFilter 还原——本质上和 NAT 转换表"记录并还原真实来源"是同一个思路。

六、DHCP vs 静态配置 · NAT vs 直连公网

DHCP 动态分配
+ 即插即用，零配置
+ 地址可回收、不易冲突
+ 集中管理网关 / DNS
- 地址可能变（对服务器不友好）
- 依赖 DHCP 服务器可用性
静态手工配置
+ 地址固定（服务器 / 打印机适用）
- 逐台手配，易配错 / 冲突
- 换网段要全部重配
※ 折中：DHCP "地址保留"
  (按 MAC 绑定固定 IP)
取舍结论： 终端设备用 DHCP（省心、可回收）；服务器 / 网关 / 打印机用静态或 DHCP 保留（地址必须稳定）。同理，公网直连（每台设备一个公网 IP）虽最符合端到端理想，但 IPv4 地址不够、且会把内网设备直接暴露在公网攻击面下；NAT 在"省地址"的同时，客观上还提供了一层默认的入站隔离——这也是它虽被诟病却长盛不衰的原因之一。真正的终极解药是下一节的 IPv6：地址多到人手无数个，从根上让 NAT 变得不再必要。
一句话收束： DHCP = 局域网自动户籍科，靠 UDP 广播 + DORA 四步，一次性把 IP / 掩码 / 网关 / DNS 租给新设备；NAT（NAPT） = 出口总机前台，靠一张转换表改写"IP + 端口"，让整个内网复用一个公网 IP，上限约 6 万并发。二者一对内、一对外，共同实现"即插即用、一个够用"。但记住 NAT 的两面性：它续命了 IPv4、破坏了端到端原则，逼出了 STUN / 打洞 / TURN 等一整套穿透术，也让"私有 IP 别当公网地址用"成为分布式系统里的经典陷阱。而这一切的根本解，是下一节要讲的 IPv6。
