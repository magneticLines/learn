TCP 拥塞控制：AIMD、慢启动、快速重传、快速恢复

没有交警的十字路口，几十亿条 TCP 连接却能自发不堵死——靠的是每个发送方都遵守同一套"探测—退让"的默契。

一、先分清：流量控制 ≠ 拥塞控制

上一节讲 TCP 时提到了流量控制（flow control）：接收方用 rwnd（接收窗口）告诉发送方"我的缓冲区还剩多少"，防止发送方淹没接收方。这是一个端到端两个人之间的事。

但网络里真正的瓶颈往往不在两端，而在中间的路由器：成千上万条连接的分组挤在同一台路由器的输出队列里，队列排满就开始丢包，排队时延也急剧上升。这叫网络拥塞（congestion）。接收方缓冲区再大也管不了中间的路由器有没有堵——所以 TCP 又引入了第二个窗口 cwnd（拥塞窗口）来应对它。

面试高频第一坑：把流量控制和拥塞控制混为一谈。记死这张对照——
· 流量控制：保护接收方，信号是 rwnd（接收方显式告知），针对一条连接的对端。
· 拥塞控制：保护网络（中间路由器），信号是丢包/时延（网络不会显式通知，只能靠发送方自己"推断"），针对全网共享资源。
发送方每一刻能发多少，取决于两者的下限。
发送窗口 = min( cwnd , rwnd )

其中 rwnd 是接收方明确告诉你的死数字，而 cwnd 完全由发送方自己维护、自己猜测——网络从不会发一封"我堵了"的信。拥塞控制的全部精妙，就在于如何仅凭 ACK 和丢包，反推出网络此刻能容纳多少。

二、为什么拥塞如此可怕：崩溃是"雪崩式"的

直觉上你可能觉得"堵一点无所谓，慢就慢点"。但拥塞的危害是非线性、正反馈的：路由器丢包 → TCP 超时重传 → 重传又灌入更多分组 → 队列更满 → 丢更多 → 重传更多……这就是历史上真实发生过的拥塞崩溃（congestion collapse）：1986 年,美国的一段网络吞吐量从 32 kbps 暴跌到 40 bps，跌了近千倍。

联系操作系统里的"抖动（thrashing）"。这和内存不足时频繁换页导致 CPU 全耗在换页、有效工作量趋近于零，是同一类现象：资源过载后，系统把大部分精力花在"重试/搬运"上，而非有效工作。解法思路也一致——不是硬扛，而是主动降低负载。TCP 的选择就是：一旦嗅到拥塞，发送方主动踩刹车。

Van Jacobson 在 1988 年的论文里给出的方案，就是今天我们要讲的这套算法。它没有任何中心调度、也不需要路由器配合（早期路由器就是简单丢包），全靠每个端主机自觉遵守同一套规则——这是一种精彩的分布式协同。

三、AIMD：整套算法的灵魂

AIMD = Additive Increase, Multiplicative Decrease（加性增、乘性减）。它是拥塞控制的核心思想，其余机制（慢启动、快恢复）都只是它的"启动加速"或"细节修补"。规则只有两条：

  • 加性增（AI）：没丢包 → 每经过一个 RTT，cwnd 增加 1 个 MSS。小步试探，缓慢地问"还能再快点吗？"
  • 乘性减（MD）：一丢包 → cwnd 立即砍半。踩重刹，因为拥塞的代价是雪崩式的，退让必须果断。
AI: cwnd += 1·MSS / RTT　　MD: cwnd ← cwnd / 2

画出来，cwnd 随时间呈现标志性的锯齿波（sawtooth）：缓慢爬升，触顶丢包后腰斩，再缓慢爬升……周而复始，让连接始终在"网络容量"上下小幅震荡，既尽量用满带宽，又不长期把队列压爆。

cwnd │ /| /| │ / | / | ← 加性增：+1 MSS/RTT，缓慢爬坡 │ / | / | │ / | / | │ / | / | │/ |/ | ← 乘性减：丢包瞬间腰斩 └──────────────────────────────────► 时间 每次触顶(丢包) → cwnd 减半 → 重新爬升 = 锯齿
为什么偏偏是"加法增、乘法减"，而不是别的组合？这是可证明的最优结构。用一张"公平性图"想象两条共享瓶颈的连接：
· 乘性减沿着"指向原点的直线"退让——按比例缩小，能拉近两条连接的差距，趋向公平。
· 加性增沿着"45°方向"一起前进——增量相同，不改变两者差距，只逼近总容量。
两者交替，轨迹会收敛到"公平且高效"的那个点。反过来若用"乘性增（MIMD）"，差距会被不断放大，谁快谁恒快，永远不公平。这就是为什么必须是 AI + MD。

四、慢启动：为什么开局要"指数狂飙"

AIMD 的加性增太慢了：假设 cwnd 要从 1 爬到 1000 个 MSS，每个 RTT 只 +1，得等约 1000 个 RTT——对一条刚建立、根本不知道网络有多宽的连接，这种龟速开局是巨大的浪费。

于是 TCP 连接一开始并不走 AIMD 的线性增，而是走"慢启动（slow start）"：cwnd 初始为 1 个 MSS（现代 RFC 6928 放宽到约 10 个 MSS），此后每收到一个 ACK 就 +1 MSS——效果是每个 RTT，cwnd 翻倍，呈指数增长。

慢启动：每 RTT 翻倍（指数级快速探测网络容量） RTT 0 : cwnd = 1 ██ RTT 1 : cwnd = 2 ████ RTT 2 : cwnd = 4 ████████ RTT 3 : cwnd = 8 ████████████████ RTT 4 : cwnd = 16 ████████████████████████████████ ↑ 一旦 cwnd ≥ ssthresh，切换成"拥塞避免"的线性增（AIMD）
命名陷阱："慢启动"其实一点都不慢——它是全程增长最快的阶段。"慢"是相对于早期 TCP "一上来就把接收窗口全发出去" 的暴力开局而言：慢启动是从 1 开始试探性地起步，而非相对于速率慢。面试被问到千万别答反。

指数增长不能无限持续，否则很快又会撞爆网络。这里引入一个门限 ssthresh（slow start threshold，慢启动门限）：当 cwnd 达到 ssthresh，就从"指数增"切换为"线性增"（进入拥塞避免 / AIMD 阶段）。而 ssthresh 从哪来？它记录的是上一次发生拥塞时 cwnd 的一半——相当于"上次是在这个高度出的事，这次快到那儿了就悠着点"。

五、如何"感知"拥塞：两种丢包信号，两种态度

网络不会通知你堵了，TCP 只能把"丢包"当作拥塞的信号。而丢包有两种被察觉的方式，TCP 对它们的"恐慌程度"完全不同：

信号含义推断严重程度反应
超时（Timeout）连 ACK 都等不到，网络可能严重拥塞甚至中断重度cwnd 直接砸回 1 MSS，重走慢启动
3 个重复 ACK后续分组还能到达（对方在催缺的那个），网络只是轻微丢了一个轻度cwnd 减半即可，无需清零

关键在于理解3 个重复 ACK 为什么代表"网络还算通畅"：TCP 的 ACK 是累积确认的，接收方每收到一个失序的分组，就会重发一次"我还在等 N 号"的 ACK。发送方连收到 3 个内容相同的 ACK，说明 N 号后面的好几个分组都顺利到达了——路还是通的，只是中间漏了一个。这种情况没必要恐慌到把窗口清零。

为什么阈值是"3"个而不是 1 个或 2 个？因为网络本身会乱序（reordering）：一个分组走了条稍慢的路径，晚到几十毫秒，会天然产生 1~2 个重复 ACK，但它并没有真正丢。若见到 1 个重复 ACK 就重传，会把大量"只是乱序"的正常情况误判为丢包。取 3 是工程经验值：在"尽早发现真丢包"和"不被乱序误导"之间的平衡点。

六、快速重传 + 快速恢复：不必傻等超时

快速重传（Fast Retransmit）

没有它时，发送方要一直等到超时计时器（RTO）到期才敢重传丢失的分组，而 RTO 通常远大于 RTT（为避免误判留了余量），白白空等一大段时间。快速重传说：既然收到 3 个重复 ACK 已经强烈暗示某个分组丢了，那就立刻重传，不等超时。

快速恢复（Fast Recovery）

快速重传解决了"何时重传"，快速恢复解决了"重传后 cwnd 怎么走"。它的态度是：既然网络只是轻微丢包（还在收到重复 ACK，说明分组仍在流动），就不要把 cwnd 清零、不要退回慢启动，而是：

  • ssthresh = cwnd / 2（记住这次的"出事高度"）；
  • cwnd = ssthresh + 3·MSS（"+3" 是因为那 3 个重复 ACK 证明有 3 个分组已离开网络、被对方收下）；
  • 之后每再收到一个重复 ACK，cwnd 再 +1 MSS（继续把已离网的分组"充值"回窗口）；
  • 等收到新数据的 ACK（丢的那个补上了），cwnd 落到 ssthresh，直接进入拥塞避免（线性增），不重走慢启动。
一句话抓住快恢复的精神：轻伤别下火线。超时 = 重伤（送回慢启动、cwnd 清 1）；3 个重复 ACK = 轻伤（cwnd 减半后原地接着跑）。快速恢复让连接在轻度丢包时维持较高的吞吐，避免每次小磕碰都从 1 重新爬坡——这正是 TCP Reno 相比更早的 TCP Tahoe 的核心改进。

七、三个状态的完整状态机

把前面所有片段拼起来，TCP 拥塞控制就是一个在三个状态间跳转的状态机（以经典的 TCP Reno 为例）：

cwnd 达到 ssthresh ┌───────────────┐ ────────────────► ┌────────────────┐ │ 慢启动 │ │ 拥塞避免 │ │ Slow Start │ │ Congestion Avoid│ │ cwnd 每 RTT×2 │ ◄──── 新 ACK ──── │ cwnd 每 RTT +1 │ └───────────────┘ └────────────────┘ ▲ ▲ │ │ │ │ 超时: ssthresh=cwnd/2 │ │ 超时: ssthresh=cwnd/2 │ └───────── cwnd=1 ───────────────┘ │ cwnd=1 │ │ │ 新 ACK: cwnd=ssthresh │ 3 个重复 ACK: │ ▼ ssthresh=cwnd/2 ┌────────────────────────────────────────────────────┐ │ 快速恢复 Fast Recovery cwnd = ssthresh + 3·MSS │ │ 每个重复 ACK: cwnd += 1 ; 收到新 ACK → 回拥塞避免 │ └────────────────────────────────────────────────────┘
面试必背的三条"事件 → 动作"映射，别记混：
· 超时（任意状态）→ ssthresh = cwnd/2；cwnd = 1 MSS；进慢启动。
· 3 个重复 ACK → ssthresh = cwnd/2；cwnd = ssthresh + 3；进快速恢复（这是 Reno；老的 Tahoe 在这里也是 cwnd=1 回慢启动）。
· cwnd ≥ ssthresh → 慢启动切拥塞避免。

八、Tahoe vs Reno vs 现代算法

教科书讲的是 Tahoe / Reno，但你今天用的 Linux/Windows 早已换了更聪明的算法。理解这条演进线，才算真正"讲透"：

算法遇到 3 个重复 ACK拥塞信号今天的地位
TCP Tahoe (1988)cwnd 清 1，回慢启动丢包历史，已淘汰
TCP Reno快速恢复，cwnd 减半丢包教科书标准模型
TCP CUBIC用三次函数曲线增长丢包Linux 默认，高带宽长距离更优
BBR (Google 2016)基本不理会丢包带宽 & RTT 建模YouTube/QUIC 大规模使用
为什么会出现 BBR 这类"叛逆者"？Reno/CUBIC 都把丢包当作拥塞的唯一信号，这在两种场景下会误判：
① 无线网络：丢包常因信号干扰（随机丢包），并非拥塞，但 Reno 照样减半，白白降速。
② Bufferbloat（缓冲区膨胀）：现代路由器缓冲区做得很大，队列排很长也不丢包，于是 Reno 一直不减速、持续灌入，导致时延飙到几百毫秒（打游戏"网络卡顿"的元凶之一）。
BBR 换了思路：主动测量瓶颈带宽和最小 RTT，直接把发送速率控制在"填满管道但不塞满队列"的甜点上，不再依赖丢包报警。

九、把速率算出来：cwnd 到底等于多大带宽

cwnd 是"一个 RTT 内允许在途（未确认）的字节数"，所以平均发送速率约为：

吞吐率 ≈ cwnd / RTT

这个简单公式藏着一个重要推论：RTT 越大，同样的 cwnd 吞吐越低。跨洲连接（RTT 大）想跑满带宽，就需要非常大的 cwnd，也因此更依赖 CUBIC/BBR 这类为"高带宽·大时延（长肥管道 LFN）"优化的算法。进一步地，把 Reno 锯齿的平均行为做数学推导，可得经典的 Mathis 吞吐率公式（p 为丢包率）：

吞吐率 ≈ 1.22 · MSS / (RTT · √p)
这个公式的直觉暴击：吞吐与 √丢包率 成反比、与 RTT 成反比。它解释了为什么"下载慢"时，跨国链路（RTT 大）+ 轻微丢包（p 稍大）会叠加成灾难；也解释了为什么 CDN 把服务器放到离你更近的地方（压 RTT）如此有效——不只是传播时延短，更因为小 RTT 直接抬高了单条 TCP 的吞吐上限。

十、用代码把状态机跑起来

应用层的 Java 代码看不到也控制不了 cwnd——它被封装在操作系统内核的 TCP 协议栈里（Java 的 Socket 只暴露到 setSendBufferSize 这类粗粒度旋钮）。但我们可以写一段小模拟，把慢启动 + AIMD + 两种丢包反应的骨架跑出来，帮助建立直觉：

拥塞控制状态机模拟 · TCP Reno 骨架Java
class RenoCongestionCtrl {
    private double cwnd     = 1;     // 拥塞窗口，单位 MSS
    private double ssthresh = 16;    // 慢启动门限

    // 每收到一个"新数据"的 ACK 时调用
    void onNewAck() {
        if (cwnd < ssthresh) {
            cwnd += 1;                 // 慢启动：每 ACK +1 → 每 RTT 翻倍
        } else {
            cwnd += 1.0 / cwnd;         // 拥塞避免：累积一个 RTT 才 +1 MSS（AI）
        }
    }

    // 收到 3 个重复 ACK：轻伤，快速重传 + 快速恢复
    void onTripleDupAck() {
        ssthresh = cwnd / 2;            // MD：乘性减，砍半
        cwnd     = ssthresh + 3;         // +3 是已离网的那 3 个分组
        fastRetransmit();               // 立即重传，不等 RTO 超时
    }

    // 超时：重伤，回到起点
    void onTimeout() {
        ssthresh = cwnd / 2;            // 门限同样减半
        cwnd     = 1;                  // cwnd 清 1，重走慢启动（指数增）
    }
}
真想观察真实的 cwnd？在 Linux 上用 ss -ti 可以看到每条连接的 cwnd、ssthresh、rtt；用 sysctl net.ipv4.tcp_congestion_control 查看/切换算法（如 cubic、bbr）。这也印证了：拥塞控制是内核的职责，跨语言、跨应用共享同一套实现——你写的 Java、Go、Nginx 用的是同一个内核算法。

十一、坏味道 → 正解：别想"绕过"拥塞控制

一个真实且高频的反模式：为了"下载更快"，有人开一堆并行 TCP 连接。因为 N 条连接就有 N 个各自独立的 cwnd，在共享瓶颈上能抢到大约 N 倍的带宽份额——这本质是用连接数作弊，破坏了 AIMD 精心维护的公平性，会挤占别人的份额、加剧拥塞。

反例 — 开 N 条连接强抢带宽
// "下载加速器"式思路
for (int i = 0; i < 32; i++) {
    new Thread(() ->
        openConn().download(part[i])
    ).start();
}
// 32 个独立 cwnd 各跑各的 AIMD
// → 抢到 ~32× 份额，挤垮邻居
// → 瓶颈更堵，整体丢包上升
// → 对所有人（含自己）都变差
正解 — 单连接多路复用，信任内核
// HTTP/2 / HTTP/3：一条连接跑多流
HttpClient c = HttpClient.newBuilder()
    .version(Version.HTTP_2)
    .build();
// 多个请求复用同一 TCP/QUIC 连接
// → 共享一个 cwnd，公平且省握手
// 需要更高吞吐？换 BBR 拥塞算法：
// sysctl net.ipv4.tcp_congestion_control=bbr
// 让内核算法去优化，别在应用层作弊
顺带一个吞吐陷阱：把 SendBufferSize 调得巨大并不能直接提速——真正的在途数据量受 min(cwnd, rwnd) 约束。缓冲区只是"能排队的上限"，拥塞窗口才是"实际放行的闸门"。盲目调大缓冲区反而可能加剧 bufferbloat，把时延搞坏。
一句话收束：TCP 拥塞控制解决的是"没人指挥、也不能问网络"的分布式限速难题。核心是 AIMD——加性增小步试探、乘性减果断退让，收敛到公平且高效；慢启动让新连接指数级快速探到网络容量（"慢"是起点低，不是速率慢），到 ssthresh 转为线性的拥塞避免；丢包即拥塞信号，但超时=重伤（cwnd 清 1 回慢启动）、3 个重复 ACK=轻伤（快速重传不等超时、快速恢复只减半不清零）。牢记两条线：发送量 = min(cwnd, rwnd)；吞吐 ≈ cwnd / RTT。而 CUBIC/BBR 的出现提醒我们：把丢包当唯一信号已经过时，现代拥塞控制正在转向对带宽与时延的直接建模。
