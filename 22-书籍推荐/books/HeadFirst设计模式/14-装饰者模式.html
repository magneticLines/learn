装饰者模式：动态扩展行为

与其为每种组合造一个子类，不如把功能像"套娃"一层层裹上去——运行期想加什么就加什么。

一、从一杯咖啡说起：子类爆炸

Head First 的经典场景：星巴兹（Starbuzz）咖啡店要给点单系统建模。饮料有浓缩（Espresso）、深焙（DarkRoast）、低咖啡因（Decaf）几种，还能加各种调料——牛奶、豆浆、摩卡、奶泡。每种饮料 + 每种调料组合都要算价钱。

最直觉的做法是用继承：为每种组合建一个子类。于是你得到 DarkRoastWithMocha、DarkRoastWithMochaAndWhip、DarkRoastWithDoubleMocha……调料一多，类的数量呈组合爆炸。

继承方案：每加一种调料/组合就多一堆子类 Beverage ┌───────────┼───────────┬─────────── ... DarkRoast DarkRoastMocha DarkRoastMochaWhip Espresso EspressoMocha EspressoSoyMocha Decaf DecafWhip DecafSoyMochaWhip ▲ 类的数量 ≈ 饮料种类 × 调料的所有子集 → 爆炸
坏味道预警：类爆炸 + 违反开闭原则。星巴兹哪天新增一种调料"焦糖"，你就得为每一种已有饮料再补一批子类；调料涨价，你要翻遍所有含该调料的类去改价格。这正是用继承做功能叠加的死穴——继承是编译期就写死的静态关系，无法在运行期灵活组合。

二、换个思路：把调料"包裹"在饮料外面

《Head First》的破局点：调料不是一种饮料的"子类"，而是对饮料的一层"包装"。一杯"加了摩卡和奶泡的深焙"，本质是——深焙咖啡，被摩卡包了一层，再被奶泡包了一层。每一层都能回答"我总共多少钱"：先问被包住的对象多少钱，再加上自己这层的钱。

生活类比：礼物套娃 / 穿衣服。你（本体）先穿衬衫，外面套毛衣，再套外套。每加一件都不改变"你还是你"，但整体的保暖度累加了。别人问你"多重"，外套会说"我自己的重量 + 里面那一层的重量"——一层层往里问。装饰者就是这种层层包裹、逐层累加的结构。

关键在于：装饰者和被装饰者必须是同一个类型。奶泡包着的东西"看起来"仍然是一杯 Beverage，所以奶泡外面还能再包一层，也能被任何"接受 Beverage 的地方"直接使用。这就是装饰者模式与"针对接口编程"的呼应。

装饰者 is-a 组件　(同一超类型，能被继续包)
装饰者 has-a 组件　(持有内层引用，把调用委托进去)

三、模式定义与角色结构

GoF 定义：动态地给一个对象添加一些额外的职责。就增加功能而言，装饰者比生成子类更为灵活。注意"动态"二字——继承是编译期的，装饰是运行期的。

Component（抽象组件，定义统一接口） + cost(): double + getDescription(): String ▲ ▲ ┌───────────┘ └───────────┐ ConcreteComponent Decorator（抽象装饰者，也 is-a Component） （被装饰的本体） - Component wrapped; ← has-a 内层引用 DarkRoast / Espresso ▲ ┌──────────┴──────────┐ MochaDecorator WhipDecorator cost()=wrapped.cost()+调料价
  • Component（抽象组件）：本体和装饰者的共同超类型，定义被装饰的接口，如 cost()。
  • ConcreteComponent（具体组件）：被包裹的核心对象，如 DarkRoast。
  • Decorator（抽象装饰者）：同样继承 Component，且内部持有一个 Component 引用（那个"被包住的内层"）。
  • ConcreteDecorator（具体装饰者）：如 Mocha，在委托内层的基础上叠加自己的行为。

四、代码实现：星巴兹咖啡

Beverage.java抽象组件 + 抽象装饰者
// ① 抽象组件：本体和装饰者共同的超类型
abstract class Beverage {
  protected String description = "未知饮料";
  public String getDescription() { return description; }
  public abstract double cost();
}

// ② 具体组件：被装饰的本体，DarkRoast 同理
class Espresso extends Beverage {
  public Espresso() { description = "浓缩咖啡"; }
  @Override public double cost() { return 1.99; }
}

// ③ 抽象装饰者：关键——它 is-a Beverage，又 has-a Beverage
abstract class CondimentDecorator extends Beverage {
  protected Beverage beverage;   // 被包住的"内层"
  public CondimentDecorator(Beverage beverage) { this.beverage = beverage; }
  @Override public abstract String getDescription();  // 强制子类重写描述
}
Mocha.java具体装饰者：委托内层 + 叠加自己
class Mocha extends CondimentDecorator {
  public Mocha(Beverage beverage) { super(beverage); }

  @Override public String getDescription() {
    return beverage.getDescription() + " + 摩卡";   // 先问内层，再拼自己
  }
  @Override public double cost() {
    return beverage.cost() + 0.20;   // 委托内层算钱，再加自己这层 0.20
  }
}

// —— 客户端：像套娃一样一层层包 ——
Beverage b = new Espresso();          // 本体 1.99
b = new Mocha(b);                       // 裹一层摩卡 +0.20
b = new Mocha(b);                       // 再来一份摩卡 +0.20（同种调料随意叠！）
b = new Whip(b);                        // 再裹一层奶泡
System.out.println(b.getDescription()); // 浓缩咖啡 + 摩卡 + 摩卡 + 奶泡
System.out.println(b.cost());           // 1.99+0.20+0.20+0.10 = 2.49

调 b.cost() 时发生的是一条递归委托链：Whip 问 Mocha，Mocha 问 Mocha，最里层 Mocha 问 Espresso，Espresso 返回 1.99，再逐层加回来。这就是"层层往里问、层层往外加"。

五、反例 → 正例：继承 vs 装饰

Bad · 继承做叠加
// 每种组合一个子类
class DarkRoast extends Beverage {...}
class DarkRoastMocha
      extends DarkRoast {...}
class DarkRoastMochaWhip
      extends DarkRoastMocha {...}
class DarkRoastDoubleMocha
      extends DarkRoast {...}
// 新增"焦糖"？
// → 每种饮料再补一批子类
// 摩卡改价？→ 翻遍所有含摩卡的类
// 组合在编译期写死，无法运行期变
Good · 装饰做叠加
// 调料就是一层包装，随意组合
Beverage b = new DarkRoast();
b = new Mocha(b);
b = new Whip(b);
// 新增"焦糖"？只加一个
class Caramel extends
      CondimentDecorator {...}
// 摩卡改价？只改 Mocha.cost() 一处
// 组合在运行期任意拼装
// 类数量 = 饮料种数 + 调料种数
//   （加法，不再是乘法/幂）
继承：类数量 ≈ 饮料 × 调料子集（组合爆炸） 装饰：饮料 + 调料
这正是"开闭原则"的教科书示范。新增调料 = 新增一个装饰类，完全不碰已有的 Beverage / Espresso / Mocha 代码——对扩展开放；老代码一行不改就继续正常工作——对修改关闭。装饰者是"组合优先于继承（HAS-A 优于 IS-A）"这条原则最漂亮的落地。

六、你天天在用它：java.io 流

装饰者模式最著名的现实案例，就是 JDK 的 I/O 库。为什么 Java 读文件要写成"一个流套一个流"？因为整套 InputStream 体系就是按装饰者模式设计的。

IoDecorator.javajava.io 就是装饰者套娃
InputStream in =
    new LineNumberInputStream(          // 装饰3：加行号
      new BufferedInputStream(          // 装饰2：加缓冲，减少系统调用
        new FileInputStream("a.txt")));// 本体：真正从磁盘读字节的具体组件

// FileInputStream          = ConcreteComponent（真干活的本体）
// FilterInputStream        = 抽象装饰者（内部持有 protected InputStream in）
// BufferedInputStream 等   = ConcreteDecorator（加缓冲/行号等功能）

对照角色一目了然：InputStream 是抽象组件；FileInputStream/ByteArrayInputStream 是具体组件；FilterInputStream 是抽象装饰者（它内部就藏着一个 protected volatile InputStream in）；BufferedInputStream、DataInputStream、GZIPInputStream 全是具体装饰者。字符流那侧的 BufferedReader(new InputStreamReader(...)) 同理。

不止 I/O。Servlet 的 HttpServletRequestWrapper / HttpServletResponseWrapper 就是给你的现成装饰者基类，用来包裹请求/响应做改写（如 XSS 过滤、Body 可重复读）。Spring Security 的 filter 链、MyBatis 的 Cache 层层包装（LruCache、SynchronizedCache 等）、Java NIO 里的 collections 包装（Collections.synchronizedList 也是"包一层加同步"的思路）——装饰者无处不在。

七、和"近亲"模式的区别（高频面试）

结构型模式里，装饰者、代理、适配器、外观长得都像"包一层"，考点就在于分清意图。它们的 UML 可以几乎一样，区别不在结构而在为什么要包。

模式意图（为什么包）接口变化
装饰者增强功能，动态叠加职责接口不变（甚至可加新方法），持同一超类型
代理控制访问：懒加载、权限、远程、计数接口不变，但通常不改变核心行为，只是"挡在前面"
适配器转换接口：让不兼容的两方能协作接口改变（A 接口 → B 接口）
外观简化：给复杂子系统一个高层入口接口简化/收窄，通常包多个对象
一句话记牢：装饰者"加料"（同接口、功能变强），代理"把关"（同接口、控制访问），适配器"转插头"（换接口），外观"做前台"（简化多个子系统）。装饰者和代理最像，差别是：装饰者的目的是叠加功能且可多层，代理的目的是控制对某个特定对象的访问（通常一层，且往往自己负责创建被代理对象）。

八、陷阱与代价

① 小类泛滥。装饰者用"很多小类"换来"灵活组合"。类一多，光看类型很难看出一个对象到底被包了哪几层，调试时栈会很深。
② 顺序有语义！new Whip(new Mocha(b)) 和 new Mocha(new Whip(b)) 算钱结果一样，但换成"先压缩再加密"vs"先加密再压缩"的流，结果天差地别——装饰顺序会影响行为，别以为随便套。
③ 别用 instanceof / 依赖具体类型判断。一旦被包了几层，obj instanceof DarkRoast 会返回 false（最外层是 Whip），依赖具体类型的代码会被装饰者悄悄破坏。这也是《Head First》反复强调"针对接口编程"的原因。
④ 对内层的引用相等性敏感的场景要小心。装饰后 wrapped != decorated，凡是用 == 或做 Map key 认对象身份的逻辑都可能出问题。
JDK 自己都栽过。java.io 这套装饰者被公认设计过度：类太多、组合心智负担重，且 LineNumberInputStream 等已被废弃。这说明装饰者不是越多越好——当维度组合真的爆炸、且需要运行期动态叠加时才值得用；如果组合固定、就那么两三种，直接写清楚反而更好读。

九、一句话收束

装饰者模式 = 让功能像套娃一样在运行期一层层包上去，用组合代替继承来叠加职责。核心记三条：① 装饰者与被装饰者共享同一超类型（is-a），所以能被继续包、能替换本体；② 装饰者内部持有一个内层引用（has-a），调用时"先委托内层、再叠加自己"；③ 它是开闭原则与组合优于继承的最佳范例，代价是类变多、顺序有语义、破坏 instanceof。看懂它，你就真正读懂了 new BufferedReader(new InputStreamReader(...)) 为什么要这么写。
