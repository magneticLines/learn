享元模式：共享细粒度对象

当同一种对象要造几百万个时，别一个个 new——把"长得一样"的部分抽出来共享，内存瞬间瘦身。

一、从一片森林说起

你在做一款游戏，地图上要渲染一片百万级的森林。最直觉的写法：每棵树 new Tree()，每个 Tree 对象里存着自己的贴图（mesh，几 MB）、颜色、纹理，还有它在地图上的坐标 (x, y)。

问题来了：一百万棵松树，本质上贴图、颜色都一模一样，唯一不同的只是"长在哪"。可你却把那几 MB 的贴图数据在内存里复制了一百万份——几个 T 的内存瞬间被撑爆。

一句话定义：享元模式（Flyweight）通过共享大量细粒度对象中"可复用的那部分状态"，来减少对象数量、节省内存。核心动作只有一个——把对象里"人人都一样"的数据抽出来，全局只留一份，大家共用。

"Flyweight"原意是拳击里的"最轻量级"。这个名字点题：模式的目标就是让每个逻辑对象变得尽可能轻，轻到你能造几百万个也不心疼内存。

二、关键拆分：内蕴状态 vs 外蕴状态

享元模式能成立，全靠一个洞察：一个对象的状态可以拆成两半。想吃透这个模式，就必须先把这两个词分清楚——这也是面试高频考点。

维度内蕴状态 (Intrinsic)外蕴状态 (Extrinsic)
含义与具体使用场景无关、可被多个对象共享的部分随场景变化、无法共享、需由客户端传入的部分
树的例子贴图、颜色、纹理（松树都长一样）坐标 (x, y)、生长高度
存在哪存在享元对象内部，只造一份不存在享元内部，调用时作参数传入
可变性创建后不可变（保证共享安全）由外部管理，可变

所以享元对象的方法签名往往长这样：把外蕴状态当参数收进来，而不是当字段存起来。

享元方法：外蕴状态靠参数传入Java
// 内蕴状态（贴图/颜色）是字段，只存一份；坐标是外蕴状态，靠参数传
void draw(Canvas canvas, int x, int y) {
    canvas.render(this.mesh, this.texture, x, y);
}

三、反例 → 正例：森林渲染

先看"每棵树都背着一份贴图"的坏味道，再看抽出享元后的写法。

反例 · 每个对象自带全部数据
class Tree {
  // 每个实例都复制一份贴图！
  byte[] mesh;      // ~2MB
  String texture;   // 大纹理
  int x, y;         // 位置
  Tree(byte[] m, String t,
       int x, int y){...}
}

// 一百万棵 = 一百万份 mesh
List<Tree> forest = ...;
for (100万次)
  forest.add(new Tree(
    松树mesh, 松树纹理, x, y));
// 内存 ≈ 100万 × 2MB 💥
正例 · 抽出享元 + 工厂共享
// 享元：只放内蕴状态（可共享）
class TreeType {           // 不可变
  final byte[] mesh;
  final String texture;
  void draw(Canvas c,
            int x,int y){...}
}
// 逻辑对象：只放外蕴状态
class Tree {
  final TreeType type;     // 共享引用
  final int x, y;
}
// 100万棵树 → 只有 1 个 TreeType
// 内存 ≈ 1 × 2MB + 100万个坐标 ✅

关键角色是享元工厂（FlyweightFactory）：它用一个 Map 做缓存，相同 key 只创建一次，之后直接复用。这也是享元模式与"手动 new"最大的区别——客户端不能直接 new 享元，必须走工厂。

享元工厂：相同 key 只造一份Java
public class TreeFactory {
    // 缓存已创建的享元；key 是内蕴状态的标识
    private static final Map<String, TreeType> pool = new HashMap<>();

    public static TreeType get(String name, String texture) {
        // 有就复用，没有才创建 —— 这一步是享元的灵魂
        return pool.computeIfAbsent(name,
                k -> new TreeType(name, texture));
    }
}

// 客户端：坐标各不相同，但 TreeType 被共享
TreeType pine = TreeFactory.get("pine", "pine.png");
for (int i = 0; i < 1_000_000; i++)
    forest.add(new Tree(pine, randX(), randY())); // pine 只有一个实例

四、省了多少内存？

假设有 N 个逻辑对象，内蕴状态每份占 I 字节、外蕴状态每份占 E 字节，且这些对象只有 k 种不同的内蕴状态（k 远小于 N）。内存对比：

不共享 N×(I+E)  →  共享后 k×I + N×E

森林例子里 I（贴图 2MB）远大于 E（两个 int）。N=100 万、k=1 时，内蕴部分从 2TB 压到 2MB。下面这条"内存占比条"直观展示了差距（左红=内蕴总量，右绿=外蕴总量）：

不共享：内蕴数据几乎吃满内存

内蕴 ×100万份 (≈2TB) 外蕴

共享后：内蕴塌缩成一份，几乎不可见

内蕴 ×1 外蕴 ×100万 (坐标)
什么时候值得用：三个条件同时满足才划算 —— ① 系统里有海量相似对象；② 对象的大部分开销来自可共享的内蕴状态；③ 内蕴状态种类不多（k 小）。如果对象没几个，或每个对象几乎全是外蕴状态，享元只会白白增加 Map 查找和代码复杂度。

五、你早就用过享元：JDK 里的享元

享元不是花架子，JDK 自己就大量在用。理解这些能帮你看穿一些经典"坑"。

场景享元池说明
Integer.valueOf(int)IntegerCache缓存 -128~127（上界可调），区间内返回同一对象
Long / Short / Byte / Character各自的 Cache同样缓存小范围数值；Character 缓存 0~127
Boolean.valueOfTRUE / FALSE全局就两个实例
字符串字面量字符串常量池"abc" 与 String.intern() 共享同一份

其中最出名的是 Integer 的自动装箱缓存。它就是一个货真价实的享元工厂——这也解释了下面这道经典面试题：

Integer 缓存：== 的经典陷阱Java
Integer a = 100, b = 100;   // 走 valueOf，命中缓存 → 同一对象
Integer c = 200, d = 200;   // 超出 127，各自 new 一个新对象

System.out.println(a == b);   // true  —— 共享了缓存里的享元
System.out.println(c == d);   // false —— 两个不同实例
陷阱 1：== 比较包装类型。正因为享元缓存的存在，Integer 的 == 结果会随数值大小"薛定谔式"变化。永远用 .equals() 比较包装类型的值，别把业务正确性押在缓存边界上。new Integer(100) 会绕过缓存、强制新建，恒返回 false——所以也别用 new 造包装类型。

六、三个致命陷阱

陷阱 2：享元必须不可变（线程安全）。享元被成千上万个对象、多个线程共享。如果它的内蕴状态可变，一个线程改了字段，所有持有它的地方都被"殃及"，还会引发并发可见性问题。把内蕴字段设为 final，创建后绝不修改——这与我们讲 不可变对象 天生线程安全是同一个道理。
陷阱 3：别把外蕴状态误塞进享元。一旦你为了"方便"把坐标 x/y 也存进 TreeType，共享就彻底破产——每个位置都得造一个新对象，模式退化回反例。判断标准：这个字段在不同使用场景里会变吗？会变就是外蕴，必须外置。
陷阱 4：享元池的内存泄漏。用静态 Map 做池时，享元对象的生命周期被拉长到与池同寿。如果 key 无限增长（比如把用户输入当 key），池会越撑越大回收不掉。种类可控时才用强引用池；否则考虑 WeakHashMap 或设置容量上限。

七、和几个"近亲"的对比

享元、单例、对象池、原型很容易混。它们都跟"对象的创建/复用"有关，但意图完全不同。

模式意图实例数量与享元的界线
享元共享内蕴状态，省内存按内蕴种类，k 个—
单例全局唯一访问点永远 1 个单例是"1 种就 1 个"，享元是"k 种就 k 个"，且强调状态拆分
对象池复用可变的昂贵对象（如连接）池大小固定池里对象可变、借出后归还、同一时刻独占；享元不可变、可同时被无数方共享
原型克隆现有对象快速造新的越造越多原型是"复制出更多"，享元是"复用同一个"，方向相反
与 Spring 的联系：Spring 默认的 singleton Bean（一个容器内每个 Bean 定义只有一个实例，被众多组件共享）在效果上很像享元——共享同一个无状态对象。但 Spring 单例的意图是"管理生命周期与依赖注入"，并非以省内存为目的；且它是"每个定义一个"而非"按状态种类分桶"。这也是为什么 Spring 强调 无状态 Bean——共享对象一旦带上可变状态，就会重蹈享元"陷阱 2"的覆辙。
一句话收束：享元模式 = 把对象状态切成"可共享的内蕴"和"随场景变的外蕴"，内蕴交给工厂缓存、全局只留一份并保持不可变，外蕴由客户端在调用时传入。它以少量的查找/传参成本，换取海量相似对象场景下的巨大内存节省。用之前先问自己那三个前提（海量、开销集中在内蕴、种类少）是否成立——不成立就别为了模式而模式。
