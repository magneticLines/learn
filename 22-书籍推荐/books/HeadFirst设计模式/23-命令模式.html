命令模式：请求封装为对象

把“做一件事”这个动词，塞进一个名词里——请求一旦变成对象，就能被传递、排队、记录、撤销。

一、从餐厅点餐建立直觉

你在餐厅点了一份牛排。你（客户）并没有冲进后厨，也不认识哪位厨师会做牛排——你只是把需求写在订单上，交给服务员。服务员把订单挂到出餐口，厨师取单、按订单上的内容动手。这里的关键是：“点餐的人”和“做菜的人”被一张订单彻底解耦了。

请把三个角色对号入座：你是发起请求的客户；服务员只负责把订单送出去（他不做菜，也不关心怎么做）；订单把“要做什么”封装成了一个独立的东西；厨师才是真正干活的人。命令模式做的事情，就是把“一个方法调用”打包成订单这样的对象。

核心洞察：请求本身也可以是对象。平时我们调用 chef.cookSteak()，这个“调用动作”是转瞬即逝的——它发生完就没了，你既不能把它存起来，也没法晚点再执行、更没法撤销。命令模式把这个动作物化（reify）成一个 Command 对象：动词变成了名词。一旦成为对象，它就能像数据一样被传参、入队列、写日志、压进撤销栈。这正是它解决的根本问题。

二、模式定义与四个角色

GoF 定义：将请求封装成对象，以便用不同的请求、队列或日志来参数化其他对象，并支持可撤销的操作。拆开看是四个角色（外加发起者 Client）：

Client ── 创建并配置 ──▶ ConcreteCommand │ 持有 ▼ Invoker ──▶ Command Receiver （服务员） <<interface>> （厨师·真正干活） holds cmd + execute() + action() + call() { ▲ cmd.execute(); ───────────┐ │ } │ │ ConcreteCommand.execute() { receiver.action(); ← 把调用委托给接收者 }
  • Command（命令接口）：只声明一个 execute()（常再加 undo()）。它是发起方与执行方之间唯一的约定。
  • ConcreteCommand（具体命令）：持有一个 Receiver 引用，execute() 里把动作委托给接收者。它就是那张“订单”。
  • Receiver（接收者）：真正知道怎么干活的对象（厨师）。命令不做业务，只负责“喊”接收者做。
  • Invoker（调用者）：持有命令并在某个时机触发 execute()（服务员），它完全不知道命令背后是谁、做什么。
Invoker → command.execute() → receiver.action()
调用者只认 Command 接口，接收者被牢牢藏在命令内部

三、没有命令模式时有多难受

想象一个万能遥控器，7 个插槽，每个能控制不同电器（灯、风扇、车库门、音响……）。如果让遥控器直接认识每一种电器，代码会长成这样——每加一个新电器，就得改遥控器：

Bad · 调用者硬编码接收者
void onButtonPressed(int slot) {
  if (slot == 0) {
    light.on();          // 直接依赖灯
  } else if (slot == 1) {
    fan.high();          // 直接依赖风扇
  } else if (slot == 2) {
    garage.up();         // 直接依赖车库门
  }
  // 新增电器 → 再加一段 if
  // 遥控器和每个电器强耦合
  // 违反开闭原则，if 越堆越长
}
Good · 插槽只认 Command
Command[] slots;  // 只依赖抽象

void onButtonPressed(int slot) {
  slots[slot].execute();   // 不问细节
}

// 配置阶段把命令塞进插槽
slots[0] = new LightOnCommand(light);
slots[1] = new FanHighCommand(fan);
// 新增电器？写个新 Command，
// 遥控器一行都不用改 ← 开闭原则

坏味道的本质是：调用者被迫认识所有接收者，请求逻辑用 if/else 焊死在调用者里。命令模式把每一种“按下按钮要干的事”抽成对象，遥控器从此只依赖 Command 接口——这就是你已学的“针对接口编程”“开闭原则”在行为型模式里的又一次落地。

四、最小实现：遥控器 + 灯

把餐厅那套映射到代码：Light 是接收者，LightOnCommand 是订单，RemoteControl 是服务员。

Command.java请求被封装成对象
// ① 命令接口：发起方与执行方唯一的约定
interface Command {
  void execute();
  void undo();               // 支持撤销——命令模式的招牌能力
}

// ② 接收者：真正知道怎么干活
class Light {
  private final String loc;
  Light(String loc) { this.loc = loc; }
  void on()  { System.out.println(loc + " 灯亮"); }
  void off() { System.out.println(loc + " 灯灭"); }
}

// ③ 具体命令：持有接收者，把动作委托出去
class LightOnCommand implements Command {
  private final Light light;             // 组合，不是继承
  LightOnCommand(Light light) { this.light = light; }
  public void execute() { light.on();  }  // 委托给接收者
  public void undo()    { light.off(); }  // 撤销 = 反向操作
}

// ④ 调用者：只认 Command，对接收者一无所知
class RemoteControl {
  private Command slot;
  private Command last;                    // 记住上一条，供撤销
  void setCommand(Command c) { this.slot = c; }
  void pressButton() { slot.execute(); last = slot; }
  void pressUndo()   { if (last != null) last.undo(); }
}

// —— 组装（Client 的活）——
Light kitchen = new Light("厨房");
RemoteControl remote = new RemoteControl();
remote.setCommand(new LightOnCommand(kitchen));
remote.pressButton();  // 厨房 灯亮
remote.pressUndo();    // 厨房 灯灭
注意“组合优先于继承”。命令持有接收者引用（组合），而不是继承它。所以同一个 Light 可以配上 LightOnCommand、LightOffCommand、DimCommand 等多个命令；同一段“撤销栈”逻辑也能作用于任意 Command。调用者、命令、接收者三者可以自由排列组合，这正是行为型模式追求的灵活性。

五、为什么值得？请求变对象后能干四件大事

把请求物化成对象，绝不只是“为了解耦而解耦”。它解锁的是一批用普通方法调用根本做不到的能力：

能力原理现实落地
撤销 / 重做命令记录执行前的状态或反向操作，压入 undo 栈编辑器 Ctrl+Z、Photoshop 历史记录
请求排队命令是对象 → 能放进队列，由线程池慢慢消费线程池任务、消息队列、工作队列
日志 / 事务把执行过的命令序列化落盘，崩溃后重放恢复数据库 redo log、事件溯源 Event Sourcing
延迟 / 定时先把命令存着，到点或满足条件再 execute()定时任务、宏命令、批处理脚本

直观看一下“请求排队”带来的解耦收益——发起请求和执行请求在时间上被彻底分开：

生产者：把命令扔进队列就返回 消费者线程：稍后取出 execute()

六、宏命令：命令还能组合命令

“一键回家”场景：按一个键要同时开灯、开空调、放音乐。用宏命令（MacroCommand）把多个命令打包成一个命令即可——它本身也实现 Command，内部持有一组子命令，这其实是命令模式与组合模式的联手。

MacroCommand.java命令的组合
class MacroCommand implements Command {
  private final Command[] cmds;
  MacroCommand(Command... cmds) { this.cmds = cmds; }
  public void execute() {
    for (Command c : cmds) c.execute();       // 顺序执行
  }
  public void undo() {
    for (int i = cmds.length - 1; i >= 0; i--)
      cmds[i].undo();                          // 撤销要逆序！后做的先撤
  }
}

Command goHome = new MacroCommand(
    new LightOnCommand(livingRoom),
    new AcOnCommand(ac),
    new MusicPlayCommand(speaker));
goHome.execute();   // 一键触发全部
陷阱：撤销必须逆序。上面 undo() 里的 for 是倒着走的。若你正序撤销，逻辑上可能出错——比如“先转账、再扣手续费”，撤销时必须“先退手续费、再退转账”。这和栈（后进先出）、事务回滚、递归调用栈的展开顺序是同一个道理：操作有依赖时，撤销要按相反顺序进行。面试问“如何实现多步撤销”，答“用一个 Deque<Command> 当撤销栈，pop 出来调 undo()”即可。

七、Lambda 是命令模式的语法糖

命令接口往往只有一个抽象方法——这是天然的函数式接口。JDK 8 起，凡是 @FunctionalInterface 都能用 Lambda 或方法引用直接写命令，省掉一堆 ConcreteCommand 类：

LambdaCommand.java方法引用即命令
Light light = new Light("客厅");

// 不再需要 LightOnCommand 类，方法引用就是命令
remote.setCommand(light::on);   // () -> light.on()
remote.pressButton();          // 客厅 灯亮

// 若 Command 只有 execute()，它就是 Runnable 的近亲
Runnable task = light::off;
task.run();                    // 客厅 灯灭
联系 JDK：Runnable 就是命令模式。看 Thread / ThreadPoolExecutor：你把 Runnable（命令）交给 executor.execute(task)（调用者），线程池把它放进 BlockingQueue（请求排队），再由某个工作线程（接收者的执行环境）择机 run()。“提交任务的人”和“执行任务的线程”被 Runnable 这个命令对象完全解耦——这正是命令模式“参数化 + 排队”能力的教科书级应用。Callable、Swing 的 Action、Spring 的 @Async 提交，本质都是这一套。

八、易混辨析：命令 vs 策略

两者都“把一段行为封进对象、用组合注入”，结构极像，是面试高频辨析题。差别全在意图：

维度命令模式策略模式
关注点做什么——把一个动作/请求物化怎么做——同一件事的不同算法
是否知道接收者命令持有接收者，负责“喊谁去做”策略通常自己就是算法，无接收者概念
典型能力撤销、排队、日志、宏命令运行期替换算法，客户端择一而用
调用方视角“把请求存起来，回头/交给别人执行”“当场用选中的算法算出结果”
类比餐厅订单（可攒、可撤、可排队）导航里的“最短/最快/避高速”
一句话记忆：策略是“换一种算法把这件事算出来”，命令是“把这件事打包成对象，方便攒着、撤着、排着队执行”。判断技巧：如果你需要撤销、排队、记录、延迟其中任何一样，几乎一定是命令模式；如果只是想在几种做法里切换，那是策略模式。

九、代价与陷阱

陷阱一：类爆炸。每种请求都要一个命令类，系统里请求一多，命令类可能几十上百个。缓解办法：只有一个动作的命令直接用 Lambda/方法引用；把状态相同的命令做成参数化命令（构造器传入差异）。别为“只调一次、也不需要撤销/排队”的简单调用硬套命令模式——那是过度设计。
陷阱二：undo 状态没存对。“撤销”不总能靠反向操作实现。比如 DimCommand 把灯从 70% 调到 20%，撤销得回到 70% 而不是某个固定值——命令必须在 execute() 时先记住旧状态（prevLevel = light.getLevel();），undo() 再恢复它。这与数据库 undo log 记录“修改前镜像”是同一思想。忘记存旧值，是新手实现撤销时最常见的 bug。
联系：命令模式是“事件溯源 / WAL”的思想源头。数据库先写 redo log 再改数据、Redis 的 AOF、消息队列持久化，本质都是“把每一条改动请求当成命令记录下来”，崩溃后按日志重放命令序列即可恢复到最新状态。请求一旦是对象，就能被序列化、持久化、跨网络传输——这是命令模式在架构层面的深远回响。

十、一句话收束

命令模式 = 把“一次方法调用”封装成对象（Command），让调用者（Invoker）与接收者（Receiver）彻底解耦。请记牢三点：① 四角色——命令持有接收者、调用者只认命令接口、Client 负责组装；② 请求变对象后解锁撤销、排队、日志、宏命令四大能力，这才是它存在的理由；③ 单方法命令用 Lambda 简化，Runnable/线程池就是它的活体标本。和策略模式撞脸时，看你要不要撤销/排队——要，就是命令。
