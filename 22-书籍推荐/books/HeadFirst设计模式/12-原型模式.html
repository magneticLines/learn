原型模式：通过克隆创建对象

与其从零 new 一个昂贵对象，不如拿一个现成的“样板”复印一份——这就是原型模式。

一、一个“造起来很贵”的场景

想象你要批量生成游戏地图里的怪物。每只怪物的初始化都很重：要读配置文件、算属性、加载 3D 模型、绑定 AI 行为树。如果每刷一只都 new Monster() 从头初始化一遍，帧率会被拖垮。

但仔细想：这一波刷出来的 100 只哥布林，属性几乎一模一样，只是坐标不同。既然有一只已经初始化好了，为什么不直接“复印”它 99 份，再各自改一下坐标？这就是原型模式（Prototype）的核心直觉。

生活类比：复印机 vs 重新誊写。你要 100 份同样的合同，不会拿钢笔一字一句抄 100 遍（new + 逐字段赋值），而是把第一份塞进复印机按 99 下（clone()）。原型模式就是把“复印”这个动作封装成对象自己的能力——对象自我复制。

二、模式定义与角色

GoF 的定义：用原型实例指定要创建对象的种类，并通过拷贝这个原型来创建新对象。它属于创建型模式——和工厂方法、构建器一样，都在解决“怎么造对象”，区别是原型模式不通过 new，而是通过复制一个既有实例。

<<interface>> Prototype + clone(): Prototype ▲ ┌───────┴────────┐ ConcreteGoblin ConcreteDragon + clone() + clone() （返回自身副本） （返回自身副本） Client ──持有──▶ Prototype 只调 clone() 拿副本，不认识具体怪物类 也不用知道“初始化一只怪有多复杂”
  • Prototype（原型接口）：声明 clone()，宣布“我能复制自己”。
  • ConcretePrototype（具体原型）：实现 clone()，返回自身的一份拷贝。
  • Client（客户端）：手里攥着一个原型实例，需要新对象时调 clone()，而不是 new 具体类。
副本 = 原型.clone()  ≠  new 类型(...)
复制既有状态（跳过重初始化）　　从零重新构造

三、Java 的天然支持：Cloneable 与 clone()

Java 在语言层面就为原型模式留了后门：Object 上有个 protected 的 clone()，配合标记接口 Cloneable 使用。

Monster.java用 JDK 内建 clone 实现原型
class Monster implements Cloneable {   // ① 不实现这个标记接口，clone() 会抛异常
  String name;
  int x, y;

  @Override
  public Monster clone() {          // ② 协变返回：返回 Monster 而非 Object（JDK5+）
    try {
      return (Monster) super.clone();  // ③ Object.clone：逐字段“按位”拷贝
    } catch (CloneNotSupportedException e) {
      throw new AssertionError(e);  // 实现了 Cloneable 就绝不会走到这里
    }
  }
}

三个关键点必须记牢：

  1. Object.clone() 是 protected native 的，由 JVM 底层做逐字段的浅拷贝，不会调用构造器——它直接复制内存里的字段值。
  2. Cloneable 是个空接口（没有任何方法）。它不是“提供 clone 能力”，而是一个开关：Object.clone() 会检查“你实现了 Cloneable 吗？没实现就抛 CloneNotSupportedException”。
  3. 覆盖时用协变返回类型把返回值收窄成具体类，调用方就不用强转了。
陷阱一（高频面试点）：clone() 跳过构造器。因为是按位复制内存，构造器里的校验、计数、日志统统不执行。如果你在构造器里维护了“已创建实例数”这类不变式，或给 final 字段赋值——clone 会绕过它们，甚至让 final 引用字段无法在 clone 里被重新赋值（这正是深克隆时的一大麻烦）。
陷阱二：Cloneable 的设计被公认为“失败案例”。它是标记接口，却去改变父类 Object 中一个 protected 方法的行为——接口本该声明“能做什么”，这里却在偷偷控制“基类方法怎么走”，违反直觉。《Effective Java》第 13 条专门吐槽了它。

四、浅克隆 vs 深克隆（最容易踩的坑）

这是原型模式的命根子问题。Object.clone() 只做浅克隆：基本类型的值被复制，但引用类型只复制“引用”本身——副本和原型的那个引用，指向的是同一个堆对象。

浅克隆（super.clone 默认行为） 原型 ─────▶ ┌────────┐ │Address │ ◀───── 副本 ← 两者共享同一个 Address！ └────────┘ 改副本的地址 = 改原型的地址（Bug） 深克隆（手动复制可变引用字段） 原型 ─────▶ ┌────────┐ │Address │ └────────┘ 副本 ─────▶ ┌────────┐ │Address'│ ← 各持一份独立副本，互不影响 └────────┘
Bad · 浅克隆埋雷
class Person implements Cloneable {
  String name;
  Address address;   // 可变引用字段

  public Person clone() {
    try {
      // super.clone 只拷了 address
      // 的“引用”，没拷对象本身
      return (Person) super.clone();
    } catch (CloneNotSupportedException e) {
      throw new AssertionError(e);
    }
  }
}
// 后果：
Person p2 = p1.clone();
p2.address.city = "上海";
// p1.address.city 也变成了"上海"！
Good · 深克隆隔离状态
class Person implements Cloneable {
  String name;          // 不可变，共享无妨
  Address address;

  public Person clone() {
    try {
      Person copy =
          (Person) super.clone();
      // 关键：可变引用字段单独深拷
      copy.address = this.address.clone();
      return copy;
    } catch (CloneNotSupportedException e) {
      throw new AssertionError(e);
    }
  }
}
// p2.address 与 p1.address 各自独立
判断口诀：String、包装类、以及所有不可变对象（immutable）即使被“共享”也毫无风险——反正谁都改不了它。所以深克隆只需要处理“可变的引用字段”：集合、数组、可变自定义对象。这也解释了为什么“不可变对象”能省掉一大堆克隆烦恼——和你已学的“封装变化 / 少暴露可变状态”一脉相承。

五、实现深克隆的几条路子

手写递归 clone() 只是最基础的一种。对象图一深，手写就变成体力活且易漏。工程上还有几种常见做法：

方式怎么做取舍
手动递归 clone对每个可变引用字段调它自己的 clone()最快、无额外依赖；但对象图深时繁琐、易漏字段
拷贝构造 / 静态工厂new Person(other) 在构造器里逐字段深拷《Effective Java》首推：走构造器、无异常、无强转、更清晰
序列化往返对象写进 ByteArrayOutputStream 再读回来一行搞定任意深度；但要求全链路 Serializable、慢、绕不过 transient
JSON 往返Jackson/Gson 序列化成串再反序列化直观通用；性能一般，且丢类型信息 / 循环引用要小心
DeepCopy.java序列化实现“一把梭”深克隆
// 要求：整条引用链上的类都实现 Serializable
static <T extends Serializable> T deepCopy(T obj) throws Exception {
  var bos = new ByteArrayOutputStream();
  try (var out = new ObjectOutputStream(bos)) {
    out.writeObject(obj);              // 把整棵对象树序列化成字节
  }
  var bis = new ByteArrayInputStream(bos.toByteArray());
  try (var in = new ObjectInputStream(bis)) {
    return (T) in.readObject();          // 读回来即是一份全新深拷贝
  }
}
// Apache Commons 直接封装好了：SerializationUtils.clone(obj)
《Effective Java》第 13 条的忠告：慎用 Cloneable，优先“拷贝构造器 / 拷贝工厂”。原因：clone() 强转丑陋、要处理 CloneNotSupportedException（还是个受检异常）、跳过构造器导致 final 字段难处理、且继承体系里正确实现极难。而拷贝构造器 Person(Person p) 没有任何这些毛病。结论：能用拷贝构造器就别碰 Cloneable。Cloneable 主要留给你读别人代码时认得出来。

六、什么时候真该用原型模式？

原型模式不是“替代 new 的银弹”，它有明确的适用场景。对比一下 new 与 clone：

维度new 构造clone 复制
是否走构造器走，执行全部初始化逻辑不走（native 拷贝内存）
初始化昂贵时每次都付一遍代价只付一次，之后复制现成状态
客户端是否知道具体类要写死具体类名只依赖原型接口，运行期决定
动态配置的对象难：配置要重新组装易：配好一个当模板，随时复制

当“初始化很重”时，克隆的收益直接体现在耗时上：

new HeavyObject()：读配置 + 建连接 + 复杂计算 clone()：复制内存

典型的用武之地：

  • 创建成本高：对象初始化要 IO、网络、大量计算，而你需要很多“状态相近”的实例。
  • 运行期动态配置：用户在界面上拼好一个对象当“模板”，之后不断复制它（如画图软件里“复制图形”、Photoshop 图层复制）。
  • 要屏蔽具体类：客户端只想要“一个和这个长得一样的新对象”，不关心它到底是哪个类——原型注册表（Prototype Registry）就是干这个的。
原型注册表：用一个 Map<String, Prototype> 预先缓存各种配好的原型，客户端拿 key 取出并 clone() 一份。这等于把“创建对象”变成“查表 + 复印”，新增种类只需往表里 put 一个原型，无需改客户端代码——又一次呼应了开闭原则。

七、串联 JDK / Spring：它到底藏在哪

  • ArrayList、HashMap、Date、Calendar 都实现了 Cloneable 并覆盖了 clone()——但注意它们多是浅克隆（ArrayList.clone() 复制底层数组，但数组里装的元素引用仍是共享的）。
  • 数组的 arr.clone() 是语言内建的复制手段，同样是浅拷贝。
  • Spring 的 @Scope("prototype") 虽叫 “prototype”，但它不是本模式——它每次请求都重新实例化（走构造器/反射），而非克隆某个既有实例。同名不同物，别混淆。
和内存模型的连线：浅克隆之所以是“共享引用”，本质是 JVM 堆里对象的引用字段存的是地址，按位复制自然把地址也一起复制了，于是两个引用指向同一块堆内存。理解了“引用即地址”，浅 / 深克隆的区别就不再需要死记——它只是“复制不复制地址背后那块内存”而已。

八、一句话收束

原型模式 = 让对象学会“自我复印”，用复制既有实例代替昂贵的从零构造。它的价值在创建成本高或需要动态配置模板时最明显。落地记住三条：① 客户端只依赖原型接口、不认具体类；② 可变引用字段必须深克隆，否则浅克隆会让副本与原型偷偷共享状态；③ Java 里 Cloneable 设计有缺陷（空标记接口 + 改父类行为 + 跳过构造器），能用拷贝构造器 / 拷贝工厂就优先用它，把 Cloneable 当作“看得懂就行”的历史遗产。
