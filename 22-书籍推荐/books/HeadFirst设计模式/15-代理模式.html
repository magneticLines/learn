代理模式：控制对象访问

给真实对象派一个“替身”，所有访问先过替身的手——它可以放行、拦截、延迟、记账，而调用方浑然不觉。

一、一个“不能让你直接碰”的场景

你想联系一位明星，但你打不通他的电话——你只能联系他的经纪人。经纪人对外暴露的接口和明星一样（“接洽演出”），但你每次的请求都要先过他的手：档期不合适他直接回绝（访问控制）、小活动他自己就处理了（延迟惊动本人）、每笔合作他都记账（附加逻辑）。你以为在和明星打交道，其实全程只在跟经纪人对话。

这就是代理模式（Proxy）的全部直觉：不让你直接访问真实对象，而是塞给你一个“替身”，由替身来控制对真实对象的访问。替身和本尊长着同一副面孔（实现同一接口），所以你根本分不清手里拿的是谁。

为什么不直接访问？因为“访问真实对象”这件事本身有成本或有风险：对象创建很贵（先给个占位，用到再造）、身处远程（本地要个代表）、需要鉴权（不是谁都能调）、要埋点/缓存/加锁（横切逻辑不该塞进业务对象）。代理把这些“访问前后要做的事”从真实对象里剥出来，单独封装——本质是关注点分离。

二、模式定义与三个角色

GoF 定义：为其他对象提供一种代理，以控制对这个对象的访问。核心是三个角色，且代理与真实对象实现同一个接口——这是代理能“以假乱真”的前提。

<<interface>> Subject + request() ▲ ┌──────┴───────┐ RealSubject Proxy + request() - real : RealSubject ← 持有真实对象引用 （真正干活） + request() { ①前置控制：鉴权/懒加载/加锁 real.request(); ← ②委托给真身 ③后置控制：缓存/记日志 } Client ─────▶ Subject （拿到的其实是 Proxy，但它只认接口，毫不知情）
  • Subject（抽象主题）：真实对象与代理的共同接口。有了它，客户端才能对二者一视同仁。
  • RealSubject（真实主题）：真正干活的对象，专注业务，不掺杂访问控制逻辑。
  • Proxy（代理）：持有 RealSubject 的引用，实现同一接口；在转发请求前后插入控制逻辑。
Client → Proxy.request() → 前置控制 → RealSubject.request() → 后置控制
调用方只看到左边这一步，右边发生了什么它并不知道

三、最小静态代理：给业务加一层访问控制

先看手写的“静态代理”——代理类在编译期就由程序员写死。假设有个昂贵资源，我们想在访问前做权限校验、访问后记日志，又不想污染业务类：

StaticProxy.java代理与真身实现同一接口
interface FileService {                 // ① 共同接口 Subject
  String read(String path);
}

class RealFileService implements FileService {   // ② 真身，只管读文件
  public String read(String path) {
    return "文件内容:" + path;
  }
}

class FileServiceProxy implements FileService {  // ③ 代理，同一副面孔
  private final FileService real;
  private final String role;
  FileServiceProxy(FileService real, String role) {
    this.real = real; this.role = role;
  }
  public String read(String path) {
    if (!"admin".equals(role))         // 前置：保护代理，鉴权
      throw new SecurityException("无权访问");
    String data = real.read(path);    // 委托给真身
    System.out.println("[LOG] read " + path); // 后置：审计
    return data;
  }
}

// 客户端只依赖接口，感知不到中间隔了一层
FileService fs = new FileServiceProxy(new RealFileService(), "admin");
fs.read("/etc/hosts");
注意这里的“组合优先于继承”。代理持有真身的引用（组合），而不是继承真身。这样代理只依赖 FileService 接口，可以代理任何实现——正是你已学的“多用组合，少用继承”“针对接口编程”两条原则在结构型模式里的落地。

四、代理有很多“变种”——按控制目的分类

代理模式是一个“家族”，结构都一样（替身 + 转发），区别只在“替身在转发前后到底做了什么控制”。这也是面试爱问的“代理有哪几种”：

类型控制什么现实原型
虚拟代理
Virtual		延迟创建昂贵对象，用到时才真正实例化（懒加载）网页图片占位符、Hibernate/JPA 懒加载、ImageIcon
保护代理
Protection		按角色/权限决定放行还是拒绝Spring Security 方法级鉴权、上例的 role 校验
远程代理
Remote		本地持有一个“本地代表”，方法调用被转成网络请求Java RMI 的 Stub、各类 RPC 客户端桩
缓存代理
Cache		缓存真身的结果，命中就不再打扰真身Spring @Cacheable、MyBatis 二级缓存
智能引用
Smart Ref		访问时附带簿记：引用计数、加锁、懒初始化写时复制、@Transactional 开关事务

以虚拟代理为例，它的收益全在“把昂贵的创建推迟到真正需要时”——首屏就不必为暂时看不到的对象买单：

无代理：启动即加载全部大对象 虚拟代理：先给占位，用到再造

五、静态代理的痛点 → 动态代理

静态代理有个致命缺点：一个接口就得手写一个代理类，横切逻辑（日志、鉴权）在每个代理里重复抄一遍。系统里有 50 个 Service，就要写 50 个代理类，日志代码抄 50 遍。这直接违反 DRY，也让“加一行日志”变成改 50 个文件。

Bad · 静态代理：类爆炸
class UserServiceProxy
      implements UserService {
  // 日志 + 委托……
}
class OrderServiceProxy
      implements OrderService {
  // 同样的日志，再抄一遍
}
class PayServiceProxy
      implements PayService {
  // 还是同样的日志……
}
// N 个接口 → N 个代理类
// 横切逻辑到处复制，改一处要改 N 处
Good · 动态代理：一个处理器通吃
class LogHandler
      implements InvocationHandler {
  // 日志逻辑只写这一份！
  // 运行期为任意接口
  // 生成代理，自动套用
}
// 1 个 Handler → 代理无数接口
// 加日志？只改 LogHandler 一处

动态代理把代理类的生成从“程序员编译期手写”变成“运行期由 JVM/字节码库自动生成”。Java 里有两条主流路子：JDK 动态代理 和 CGLIB。

六、JDK 动态代理：基于接口，运行期造类

JDK 自带的动态代理靠两个东西：java.lang.reflect.Proxy（在运行期生成一个实现了指定接口的类，通常叫 $Proxy0）和 InvocationHandler（所有方法调用都被“漏斗”进它的 invoke）。

JdkProxy.java一个 Handler 代理所有接口方法
import java.lang.reflect.*;

class LogHandler implements InvocationHandler {
  private final Object target;             // 被代理的真身
  LogHandler(Object target) { this.target = target; }

  @Override
  public Object invoke(Object proxy, Method m, Object[] args)
      throws Throwable {
    System.out.println("before " + m.getName()); // 前置
    Object result = m.invoke(target, args);    // 反射委托给真身
    System.out.println("after " + m.getName());  // 后置
    return result;
  }
}

// 运行期生成代理：不需要任何手写的 XxxProxy 类
UserService target = new UserServiceImpl();
UserService proxy = (UserService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),      // ① 用哪个类加载器装载 $Proxy0
    target.getClass().getInterfaces(),      // ② 要实现哪些接口 ← 关键：必须有接口
    new LogHandler(target));                 // ③ 方法调用都进这个 handler
proxy.save();  // 实际调用 $Proxy0.save() → handler.invoke(...) → target.save()
陷阱（高频面试点）：JDK 动态代理只能代理“接口”。因为它生成的 $Proxy0 必须 extends Proxy implements 你的接口——Java 单继承已经被 Proxy 占了，只能靠实现接口来“变身”。所以目标类没有实现任何接口时，JDK 动态代理无能为力，这时才需要 CGLIB。另外别把 Proxy.newProxyInstance 的第二个参数写成 target.getClass()——它要的是接口数组，传错会直接抛 IllegalArgumentException。

七、JDK 动态代理 vs CGLIB

CGLIB（Code Generation Library，底层是 ASM 操作字节码）走的是另一条路：为目标类动态生成一个子类，通过覆盖方法来插入逻辑。它不要求接口，但因为靠继承，就受继承的限制。

维度JDK 动态代理CGLIB
实现机制运行期生成实现接口的类运行期生成目标类的子类
前提条件目标必须有接口无需接口，直接代理类
不能代理什么没接口的类、接口里没有的方法final 类、final/private/static 方法（无法被子类覆盖）
依赖JDK 内置，无需额外库需引入 cglib（Spring 已内嵌）
典型使用者有接口的 Spring Bean（旧默认）无接口的 Bean；Spring Boot 现默认
Spring AOP 怎么选？历史上 Spring 的规则是：目标类实现了接口就用 JDK 动态代理，没有接口就用 CGLIB。但从 Spring Boot 2.x 起，默认把 spring.aop.proxy-target-class 设为 true，也就是一律用 CGLIB（避免“注入接口类型 vs 实现类型”不一致带来的困惑）。@Transactional、@Async、@Cacheable 本质都是 Spring 给你的 Bean 套了一层代理。
陷阱：Spring AOP 的“自调用（self-invocation）失效”。代理只拦截“从外部经过代理对象”的调用。若在同一个类里用 this.otherMethod() 调用另一个带 @Transactional 的方法，走的是真身自己的 this 引用，绕过了代理，注解直接失效。同理 private/final 方法上的 AOP 注解也不生效（CGLIB 覆盖不了）。这是新手 Debug “事务不回滚”的头号元凶。

八、别搞混：代理 vs 装饰者 vs 适配器

三者结构上都是“包一层、持有被包对象、转发”，长得几乎一样，区别全在意图——这是面试必问的辨析题：

模式意图（为什么包）接口是否变典型特征
适配器接口转换：把已有接口转成客户端期望的另一种变（换一副面孔）解决“接口不兼容”
装饰者增强功能：给对象叠加新职责不变可多层堆叠，客户端主动组合
代理控制访问：管住“能不能访问 / 何时访问”不变常自己创建/管理真身，客户端无感
一句话记忆：适配器改“形”（换接口），装饰者加“料”（加功能），代理设“关”（控访问）。装饰者和代理接口都不变，最大差别在意图与控制权：装饰者是“我想给它加点东西”，通常由使用者层层包；代理是“我替它挡在门口”，往往连真身什么时候创建都由它说了算（虚拟代理就是典型）。

九、串联 JDK 与框架：它无处不在

  • Java RMI：客户端拿到的 Stub 就是远程代理，你调本地方法，它悄悄把参数序列化、发网络、等结果——Head First 正是用 RMI 讲远程代理。
  • Spring AOP / 事务：@Transactional 的开启提交回滚，就是代理在方法前后包了 begin/commit/rollback——智能引用型代理。
  • MyBatis Mapper：你只写了 interface UserMapper 接口，没有实现类，运行期由 MapperProxy（JDK 动态代理）把方法调用翻译成 SQL 执行。
  • Hibernate 懒加载：getOne() 返回的其实是个虚拟代理，真正访问字段时才发 SQL 加载——这也是为什么脱离 Session 后会抛 LazyInitializationException。

十、一句话收束

代理模式 = 给真实对象派一个“同接口的替身”，把对它的访问控制（鉴权、懒加载、远程、缓存、事务）从业务对象里剥离到替身中。落地记牢三条：① 代理与真身实现同一接口，客户端才会分不清、才能无感替换；② 静态代理会类爆炸，工程上用动态代理——JDK（基于接口）与 CGLIB（基于子类，怕 final）；③ 它和装饰者/适配器结构相似但意图迥异——代理管“控制访问”。Spring 里 @Transactional 失效？先想想是不是踩了自调用绕过代理的坑。
