依赖倒置：依赖抽象，不依赖具体类

让高层策略不再被低层细节绑架——把依赖的箭头，统统指向抽象。

一、先看一个被具体类绑架的场景

假设我们开一家披萨店，PizzaStore 负责点餐流程（这是高层业务策略），下面挂着一堆具体披萨类 CheesePizza、VeggiePizza（这是低层实现细节）。最朴素的写法是这样：

PizzaStore.java坏味道：高层直接 new 低层
class PizzaStore {
  Pizza orderPizza(String type) {
    Pizza pizza;
    // 高层 PizzaStore 亲手 new 了一堆具体类
    if (type.equals("cheese")) pizza = new CheesePizza();
    else if (type.equals("veggie")) pizza = new VeggiePizza();
    else if (type.equals("clam"))   pizza = new ClamPizza();
    else throw new IllegalArgumentException(type);

    pizza.prepare();
    pizza.bake();
    return pizza;
  }
}

问题不在于代码跑不起来——它跑得挺好。问题在于依赖方向反了：定义业务规则的高层模块，反过来去依赖那些会频繁变动的低层具体类。每加一种新披萨，你都得回来修改 orderPizza（这同时也违反了上一节的开闭原则）。

「高层 / 低层」到底谁高谁低？不是看代码写在上面还是下面，而是看谁定义策略、谁提供细节。PizzaStore 表达“先做后烤再装盒”的业务流程，它是策略，是高层；CheesePizza 只是流程里的一个可替换零件，是细节，是低层。经验之谈：越接近“需求 / 用例”的越高层，越接近“数据库 / 网络 / 具体算法”的越低层。

二、原则定义：两句话

依赖倒置原则（Dependency Inversion Principle，DIP，SOLID 里的 D）由 Robert C. Martin 提出，Head First 把它浓缩成一句口号：「要依赖抽象，不要依赖具体类」。展开是两条：

高层模块 ✗ 依赖 低层模块
高层、低层 都 → 依赖 抽象
  1. 高层模块不应该依赖低层模块，二者都应该依赖抽象。
  2. 抽象不应该依赖细节，细节应该依赖抽象。

换句话说：在高层和低层之间，插一层接口 / 抽象类做“合同”。高层只对着合同编程，低层则去实现这份合同。谁都不直接抓着对方的实现类不放。

三、「倒置」倒的是什么？

关键词是 Inversion（倒置）。“倒”的是源码依赖的箭头方向。传统的过程式设计里，高层调用低层，源码依赖自然从上往下指；DIP 通过引入抽象，让低层的依赖箭头反转向上，指向抽象：

传统（依赖细节） 倒置后（依赖抽象） PizzaStore PizzaStore ──依赖──┐ │ 依赖 ▼ ▼ ┌──────────┐ CheesePizza │ Pizza │ <<接口/抽象>> VeggiePizza └──────────┘ ClamPizza ▲ │ 实现（箭头“倒”过来向上） 高层被低层的细节绑架 CheesePizza / VeggiePizza / ClamPizza 低层反过来依赖抽象，高层不再认识它们
直觉记忆：倒置前，抽象“属于”高层还是低层含糊不清；倒置后，抽象由高层来拥有和定义（谁用它、谁定义它），低层来适配它。这就是 Martin 说的“接口所有权反转”——控制权从低层实现方，交回到高层策略方手里。

四、反例 → 正例：把 new 赶出高层

再看一个更贴近业务的例子：订单服务要持久化订单。坏写法让高层 OrderService 直接 new 出具体的数据库实现；好写法让它只依赖 OrderRepo 抽象，具体实现由外部注入。

Bad · 焊死在实现上
class OrderService {
  // 直接 new 具体类：
  // 想换成 Mongo？改源码。
  // 想写单测？连不上真库。
  private MySqlOrderRepo repo
      = new MySqlOrderRepo();

  void save(Order o) {
    repo.insert(o);
  }
}
Good · 依赖抽象 + 注入
interface OrderRepo {
  void insert(Order o);
}
class OrderService {
  private final OrderRepo repo;
  // 依赖抽象，实现由外部塞进来
  OrderService(OrderRepo repo) {
    this.repo = repo;
  }
  void save(Order o) { repo.insert(o); }
}

右边有两个立竿见影的好处：可替换（MySqlOrderRepo / MongoOrderRepo 随便换，OrderService 一行不改）与 可测试（单测里塞一个内存假实现 InMemoryOrderRepo 或 Mockito 的 mock，不用起数据库）。这就是为什么 DIP 是“可测试代码”的地基。

OrderServiceTest.java依赖抽象带来的可测试性
@Test
void save_writesThroughRepo() {
  // 不碰真实数据库，注入一个假仓库
  OrderRepo fake = new InMemoryOrderRepo();
  OrderService svc = new OrderService(fake);

  svc.save(new Order(1));

  assertEquals(1, fake.count()); // 只验行为，不依赖细节
}

五、Head First 的三条“可依赖”指导方针

Head First 给了三条努力方向（注意是理想目标，不是铁律，做不到全部也没关系）：

指导方针含义现实中的松动
变量不要持有具体类的引用用 new 就是在抓具体类，改用工厂来创建对象创建对象总得有人 new，把这些 new 集中到工厂 / 容器里
不要让类派生自具体类继承要基于接口或抽象类，别继承具体实现继承稳定的框架基类（如 AbstractList）通常可接受
不要覆盖基类已实现的方法基类若已实现方法，说明它不是纯抽象，子类覆盖会破坏共享模板方法里覆盖的是抽象的钩子方法，不算违反
陷阱一：把方针当教条。“不许出现 new”是不可能的——总得有地方创建对象。方针的真正意图是不要让高层策略代码里散落 new，把创建职责收拢到工厂 / DI 容器这种“低层边缘”去。String s = new String() 这种依赖稳定、几乎不会变的类，纠结它反而是过度设计。
陷阱二：为抽象而抽象。给每个类都配一个只有单一实现、且永远不会有第二个实现的接口（FooImpl implements Foo），只是徒增跳转成本。只在“变化点 / 需要替换或测试的边界”上抽象——这与第一节“封装变化”一脉相承。

六、和 Spring / IoC / DI 的连线

你每天用的 Spring，本质就是 DIP 的工业化落地。@Autowired 一个接口字段时，Spring 在运行期把某个实现类的 Bean 注进来——高层代码从头到尾只认识接口：

Spring 场景DIP 的运行期兑现
@Service
class OrderService {
  private final OrderRepo repo; // 只依赖抽象

  // 构造器注入：控制权交给容器（控制反转）
  OrderService(OrderRepo repo) { this.repo = repo; }
}

@Repository
class MySqlOrderRepo implements OrderRepo { /* 细节依赖抽象 */ }

这里有三个常被混淆的名词，必须理清：

概念是什么关系
DIP 依赖倒置原则一条设计原则：面向抽象、让依赖箭头指向抽象目标 / “要什么”
IoC 控制反转一种思想：把“创建 / 组装对象”的控制权从代码交给框架（好莱坞原则：别调用我，我会调用你）实现 DIP 的手段
DI 依赖注入IoC 的一种具体实现方式：通过构造器 / setter / 字段把依赖“喂”进来实现 IoC 的具体技术
一句话串起来：为了满足 DIP（原则），我们采用 IoC（思想） 把创建权交出去，具体靠 DI（技术） 把抽象的实现注入进来。Spring 容器就是那个替你 new 并组装的“大工厂”。

七、DIP 在 JDK 里随处可见

不用 Spring，标准库本身就浸透了 DIP。你写业务时依赖的几乎全是接口，具体实现在运行期才确定：

  • List<String> list = new ArrayList<>(); —— 高层声明用 List 抽象，哪天换成 LinkedList 上层无感。
  • JDBC 的 Connection / Statement 全是接口，MySQL、PostgreSQL 驱动各自实现——你的 DAO 代码不认识任何具体驱动类。
  • SLF4J 是一层日志抽象门面，底层可切 Logback / Log4j2；应用代码只依赖 org.slf4j.Logger。
  • Comparator、Runnable、Iterator——把“怎么做”的细节以抽象形式交给调用方注入。
和已学知识的联系：“针对接口编程，不针对实现编程”是局部写法层面的建议（一个变量声明成接口类型）；依赖倒置是把它上升到模块架构层面的原则（整个高层不依赖低层，依赖箭头统一指向抽象）。前者是砖，后者是承重墙的排布方式。

八、一句话收束

依赖倒置 = 把依赖的箭头统一指向抽象。高层定义并拥有抽象（接口），低层去实现它，谁都别直接抓着对方的具体类。做到这一点，你就同时收获了可替换、可测试、抗变化三样东西——这也是 Spring IoC 容器、JDBC、SLF4J 之所以能“换底层不动上层”的根本原因。落地手段记住：把 new 赶出高层、只在变化点抽象、用工厂或 DI 容器接管对象创建。
