单例模式：全局唯一实例

确保一个类只有一个实例，并提供一个全局访问点——听起来简单，却是并发与面试的重灾区。

一、它到底解决什么问题

有些对象，全局有且只应有一个：线程池、配置管理器、日志器、数据库连接池、缓存、注册表。理由通常是两类——要么它持有全局共享状态（多个实例会让状态互相打架），要么它创建代价昂贵（重复 new 是浪费）。

你也许会说：那我用静态全局变量不就行了？确实能存一份，但有个致命缺陷：静态字段在类加载时就会初始化，哪怕你整个程序压根没用到它，也白白付出了创建成本。单例模式想同时拿到两点：唯一性 + 可延迟到第一次使用才创建（懒加载）。

生活类比：一个国家同一时刻只有一位在任总统。无论你在新闻、外交场合还是签署文件里提到“总统”，指的都是同一个人——不会因为你换了个场合就冒出第二位。单例就是把“同一个身份，全局共享”这件事在代码里固化下来。

二、经典结构：私有构造器 + 静态访问点

单例的三板斧，缺一不可：①私有构造器（堵死外部 new）、②一个持有唯一实例的静态字段、③一个静态的全局访问方法。Head First 的教科书写法（懒加载版）是这样：

Singleton.java教科书懒汉式（单线程 OK，多线程有坑）
public class Singleton {
  // 静态字段：全类共享的唯一引用，先留空
  private static Singleton instance;

  // 私有构造器：外部无法 new，唯一入口被封死
  private Singleton() {}

  // 全局访问点：第一次调用才创建（lazy）
  public static Singleton getInstance() {
    if (instance == null) {          // (1) 检查
      instance = new Singleton(); // (2) 创建
    }
    return instance;
  }
}
┌─────────────────────────────┐ │ Singleton │ ├─────────────────────────────┤ │ - static instance : Singleton│ <── 唯一实例藏在这 ├─────────────────────────────┤ │ - Singleton() │ <── 构造器私有 │ + static getInstance() │ <── 全局访问点 └─────────────────────────────┘ ▲ 调用方 ──────┘ 只能通过 getInstance() 拿实例

单线程下它完美无缺。但那句 if (instance == null) 在多线程下就是一颗定时炸弹——这也正是面试官最爱追问的地方。

三、多线程翻车现场（面试高频）

上面第 (1)(2) 两步不是原子操作。想象两个线程几乎同时进入 getInstance()：

时间轴 ─────────────────────────────────────────► 线程 A: 读到 instance==null ┐ │(此刻还没来得及 new) 线程 B: 读到 instance==null ┘ 线程 A: new Singleton() → 得到对象 #1 线程 B: new Singleton() → 得到对象 #2 ← 出现了两个实例! 结果: A 和 B 各自持有不同对象，单例被打破
面试陷阱：为什么“懒汉式”默认不是线程安全的？因为“检查 null”和“赋值创建”之间存在竞态窗口（race condition）。两个线程可能都通过了 null 检查，于是各 new 一次，全局唯一性直接崩塌。而“饿汉式”（下文）之所以天生安全，是因为把创建这一步交给了 JVM 的类加载机制——类初始化本身由 JVM 保证只执行一次、且带同步。

四、五种实现方式全景对比

解决线程安全有一整条演进路线，各有取舍。先上全景表，后面逐个拆：

方式线程安全懒加载要点 / 代价
懒汉式（无同步）❌ 否✅ 是教科书写法，多线程下会产生多实例，别用
synchronized 方法✅ 是✅ 是正确但每次调用都加锁，高并发下性能差
饿汉式（静态字段）✅ 是❌ 否类加载即创建，简单可靠；无法延迟，无法拿构造参数
双重检查锁 DCL✅ 是✅ 是高性能懒加载，但必须配 volatile，易写错
静态内部类 Holder✅ 是✅ 是兼顾懒加载与无锁，推荐；靠 JVM 类初始化保证
枚举 enum✅ 是❌ 否Effective Java 首选；天然防反射/反序列化攻击

五、双重检查锁 DCL：为什么非要 volatile

synchronized 整个方法虽然安全，但“只有第一次创建才需要同步，之后每次读都加锁”太浪费。双重检查锁（Double-Checked Locking）的思路是：只在实例还没创建时才进同步块。

Bad · 少了 volatile
class Singleton {
  // 少了 volatile!
  private static Singleton inst;
  private Singleton() {}

  static Singleton getInstance() {
    if (inst == null) {
      synchronized (Singleton.class) {
        if (inst == null)
          inst = new Singleton();
      }
    }
    return inst; // 可能拿到"半成品"
  }
}
Good · 加上 volatile
class Singleton {
  // volatile 禁止指令重排
  private static volatile Singleton inst;
  private Singleton() {}

  static Singleton getInstance() {
    if (inst == null) {              // 第一次检查(无锁)
      synchronized (Singleton.class) {
        if (inst == null)            // 第二次检查(持锁)
          inst = new Singleton();
      }
    }
    return inst;
  }
}

关键在于 inst = new Singleton() 这一行不是一条指令，而是三步：

① 分配内存  →  ② 调用构造器初始化  →  ③ 把引用指向内存

JVM 允许在不改变单线程结果的前提下重排序，于是它可能变成 ① → ③ → ②。这时：线程 A 已经执行完 ③（inst 不再为 null），但 ② 还没跑（对象没初始化完）。线程 B 在第一次检查时看到 inst != null，直接 return 了一个没构造完的“半成品”对象，一用就 NPE 或数据错乱。

volatile 干了两件事：①保证可见性（一个线程的写立刻对其他线程可见）；②禁止指令重排序——它插入内存屏障，强制 ② 必须排在 ③ 之前。这正是 Java 内存模型（JMM）里 happens-before 规则的体现：对 volatile 变量的写，happens-before 后续对它的读。所以“为什么 DCL 要 volatile”这道面试题，本质考的是 JMM + 指令重排。

六、更优雅的两种：Holder 与 enum

静态内部类 Holder（推荐日常使用）

它巧妙利用了一条 JVM 规则：外部类加载时，静态内部类并不会被加载；只有第一次真正用到它时才加载并初始化。而类初始化由 JVM 用锁保证“恰好一次”，天然线程安全，且完全无需自己写 synchronized。

Singleton.java静态内部类 Holder：懒加载 + 无锁 + 线程安全
public class Singleton {
  private Singleton() {}

  // 加载 Singleton 时，Holder 不会被加载
  private static class Holder {
    // 只有首次访问 Holder.INSTANCE 才触发初始化
    static final Singleton INSTANCE = new Singleton();
  }

  public static Singleton getInstance() {
    return Holder.INSTANCE; // JVM 保证类初始化只发生一次
  }
}

枚举 enum（Effective Java 首选）

Joshua Bloch 在《Effective Java》里明确推荐用单元素枚举实现单例。它不仅代码最短，还免费获得了两项别的写法要额外费劲才能防住的能力：防反射攻击和防反序列化生成新实例。

Elvis.java枚举单例：最简且最健壮
public enum Elvis {
  INSTANCE; // 唯一实例，JVM 保证

  private final String name = "The King";

  public void sing() {
    System.out.println(name + " is singing");
  }
}
// 使用：Elvis.INSTANCE.sing();
面试陷阱：其他写法能被“攻破”，枚举不能。普通单例有三条后门可以造出第二个实例：
① 反射：setAccessible(true) 强行调用私有构造器；
② 反序列化：默认反序列化会 new 一个新对象（需实现 readResolve() 返回单例才能堵住）；
③ 克隆：若实现了 Cloneable 会复制出新对象。
而 enum 从语言层面禁止反射创建枚举实例（newInstance 会抛 IllegalArgumentException），序列化也由 JVM 特殊处理保证单例。这就是它“最健壮”的原因。

七、和 JDK / Spring 的连线

单例不是纸上概念，你天天在用：

  • Runtime.getRuntime()——JDK 里教科书级的饿汉式单例，整个 JVM 一个 Runtime。
  • Desktop.getDesktop()、Collections.emptyList() 背后的空集合常量，也是单例思想。
  • Spring 的 Bean 默认就是单例（scope="singleton"），一个容器里一份。
高频面试题：Spring 的单例 和 GoF 单例是一回事吗？——不是！
  • 作用域不同：GoF 单例是每个 ClassLoader / JVM 一个实例（靠私有构造器强制）；Spring 单例是每个 IoC 容器一个实例（靠容器缓存 Map 管理），构造器通常是 public 的，你完全可以自己再 new 一个。
  • 保证方式不同：GoF 靠代码结构（私有构造器）；Spring 靠容器行为（DefaultSingletonBeanRegistry 里的 singletonObjects 缓存）。
  • 范围不同：同一个 JVM 里起两个 Spring 容器，就会有两个“单例”Bean。
类加载器带来的隐藏坑：GoF 单例的“唯一”其实是每个 ClassLoader 唯一。如果同一个类被两个不同的 ClassLoader 分别加载（Web 容器多应用、OSGi、热部署等场景常见），会产生两份 Class 对象，也就有了两个“单例”。这解释了很多“我明明写了单例，怎么会有两个实例”的诡异现象。

八、别滥用：单例的代价

单例好用，但它常被批评为“反模式”，因为它本质是受控的全局变量，会悄悄带来耦合：

问题说明
隐藏依赖方法内部直接 Xxx.getInstance()，从签名上看不出它依赖了谁，破坏了依赖倒置
难以测试全局状态在测试间互相污染；无法轻易替换成 mock（不像构造器注入那样可替换）
违反单一职责单例类同时管着“自己的业务”和“控制自己的实例数”两件事
并发陷阱全局共享可变状态，若内部字段可变，又回到多线程同步问题
实践建议：在用 Spring 的项目里，别自己手写 GoF 单例——直接把类交给容器管理成单例 Bean，用构造器注入拿到它。这样既有“全局一份”的效果，又保留了可替换、可测试的好处（测试时注入假实现即可）。手写单例更适合没有容器的纯工具类、或框架底层设施。

九、一句话收束

单例 = 私有构造器 + 静态唯一实例 + 全局访问点。核心难点全在线程安全：懒汉式默认不安全，synchronized 方法安全但慢，DCL 要配 volatile 防指令重排，日常首选静态内部类 Holder（懒加载 + 无锁），要防反射/反序列化就用 enum。同时记住两个边界：GoF 单例 ≠ Spring 单例（一个每 JVM/ClassLoader 一份，一个每容器一份），且单例是“受控的全局变量”，能交给容器就别手写——它省的是内存，赔的可能是可测试性与解耦。
