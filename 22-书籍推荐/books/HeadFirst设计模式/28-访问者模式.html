访问者模式：操作与数据结构分离

数据结构相对稳定、操作却层出不穷时，把“操作”抽出来做成访问者，让新功能不再侵入每一个节点类。

一、从“体检”建立直觉

把公司里的每个人想象成一个稳定的数据结构：员工、经理、实习生。他们的“类型”几乎不会变。但每年要对这群人做的操作却花样翻新——今年是体检，明年是薪资审计，后年是安全培训统计。

如果每来一个新操作，你就得跑去改Employee、Manager、Intern每一个类，给它们塞一个新方法，那太痛苦了。访问者模式换了个思路：让一个访问者（Visitor）“上门拜访”每个人。体检医生是一个访问者，薪资审计员是另一个访问者。人（数据结构）只需要配合一个动作——“开门迎客”，剩下的事全交给访问者去做。

核心洞察：把“操作”从“数据结构”里抽出来，变成独立对象。传统 OOP 把方法写在对象自己身上（employee.checkHealth()）。访问者模式反过来——数据结构只暴露一个统一入口 accept(visitor)，具体要干什么由传进来的访问者决定。于是新增一种操作 = 新写一个访问者类，一个节点类都不用动。这就是“操作与数据结构分离”这句话的全部含义。

二、它到底解决什么问题？先看痛点

假设你在做一个文件系统扫描器，节点有TextFile、ImageFile、Directory三种。产品今天要“算总大小”，明天要“统计文件数”，后天要“导出成 XML”。若把每个操作都写进节点类：

Bad · 操作塞进每个节点类
class TextFile {
  long size();
  int  count();      // 新操作 → 改这里
  String toXml();    // 又一个 → 又改
}
class ImageFile {
  long size();
  int  count();      // 每个类都要
  String toXml();    // 重复添加
}
class Directory { ... 同样一堆 ... }

// 加一个操作 = 改 N 个节点类
// 一个操作的逻辑散落在 N 处
// 违反单一职责 & 开闭原则
Good · 操作收进访问者
interface Visitor {
  void visit(TextFile f);
  void visit(ImageFile f);
  void visit(Directory d);
}
// 一个操作 = 一个访问者类，
// 该操作的全部逻辑聚在一处
class SizeVisitor  implements Visitor {...}
class CountVisitor implements Visitor {...}
class XmlVisitor   implements Visitor {...}

// 新增操作：只写一个新 Visitor
// 节点类一行都不用改 ← 开闭原则

坏味道的本质是：一个“操作”的逻辑被拆散到了每个节点类里——想读懂“算总大小”得翻遍三个文件。访问者把同一操作的三段逻辑聚拢到一个SizeVisitor里，按操作组织代码，而不是按数据类型组织。这与你已学的单一职责、开闭原则一脉相承。

三、模式结构与五个角色

GoF 定义：表示一个作用于某对象结构中各元素的操作，它让你在不改变各元素类的前提下定义作用于这些元素的新操作。

Client │ 创建 ▼ Visitor <<interface>> Element <<interface>> + visit(TextFile) + accept(Visitor v) + visit(ImageFile) ▲ + visit(Directory) │ implements ▲ ┌───────┼───────┐ │ implements TextFile ImageFile Directory ┌────┴────┐ accept(v){ v.visit(this); } SizeVisitor XmlVisitor └─── 回调 ───┘ element.accept(v) → v.visit(element) （this 的真实类型在此刻确定）
  • Visitor（访问者接口）：为每一种具体元素声明一个 visit(具体元素) 重载。
  • ConcreteVisitor（具体访问者）：一个操作对应一个类，把该操作在各元素上的逻辑集中实现。
  • Element（元素接口）：只声明一个 accept(Visitor)，是数据结构对外的唯一“开门”动作。
  • ConcreteElement（具体元素）：accept 里回调 v.visit(this)——注意这个 this 是关键。
  • ObjectStructure（对象结构）：持有元素集合，能遍历它们并逐个 accept（如 Directory 或一个 List）。
陷阱预警：为什么非要那句 accept 回调？新手最大的困惑是——“我直接 visitor.visit(element) 不就行了，绕一圈调 element.accept(visitor) 再回调 visitor.visit(this) 图啥？”答案藏在下一节：这一去一回，是为了破解 Java 只有单分派的限制。看不懂这节，就没真懂访问者模式。

四、精髓：用“双分派”破解单分派

Java 的方法调用是单分派（single dispatch）：运行时只根据调用者（receiver）的真实类型决定调哪个重写方法；而方法参数选哪个重载，是编译期按静态类型定死的。这会出问题：

SingleDispatch.java为什么直接 visit 不行
Element e = new TextFile("a.txt");   // 静态类型 Element，动态类型 TextFile
Visitor v = new SizeVisitor();

// ✗ 想直接这样调？编译器按 e 的【静态类型 Element】选重载
//   但 Visitor 里根本没有 visit(Element)，编译不过；
//   就算加一个 visit(Element)，运行期也永远进不到 visit(TextFile)
v.visit(e);   // 参数重载在编译期定死 → 无法按 e 的真实类型分派

访问者用一次巧妙的“回踢”解决了它——双分派（double dispatch）：先靠虚方法确定元素的真实类型，再靠重载确定访问者的操作。两次分派，两个维度都被精确锁定：

DoubleDispatch.javaaccept 回调如何锁定类型
class TextFile implements Element {
  public void accept(Visitor v) {
    v.visit(this);   // ★ this 在这里的编译期静态类型就是 TextFile
  }                       //   → 编译器稳稳选中 visit(TextFile)
}

// 调用现场：
Element e = new TextFile("a.txt");
e.accept(v);
// 第 1 次分派（虚方法/动态）：e 真实是 TextFile → 进 TextFile.accept()
// 第 2 次分派（重载/静态）：this 是 TextFile → 进 v.visit(TextFile)
e.accept(v)  ⇒  TextFile.accept  ⇒  v.visit(TextFile)
第 1 跳锁定「元素类型」，第 2 跳锁定「操作」= 双分派
一句话记住双分派：“元素在 accept 里出卖了自己的真实类型”。因为 accept 是被重写的虚方法，运行期一定进到 TextFile.accept；而进去后 this 的静态类型就是 TextFile，于是重载解析在编译期就能挑中 visit(TextFile)。Java 天生只有单分派，访问者用“虚方法 + 重载”手工拼出了双分派——这是本模式最值得玩味的地方。

五、完整最小实现：文件系统扫描

把体检那套映射到代码：文件节点是元素，SizeVisitor 是“算总大小”这个操作的访问者。

Visitor.java操作与数据结构彻底分家
// ① 访问者接口：为每种元素声明一个重载
interface Visitor {
  void visit(TextFile f);
  void visit(ImageFile f);
  void visit(Directory d);
}

// ② 元素接口：只暴露一个“开门”动作
interface Element { void accept(Visitor v); }

// ③ 具体元素：accept 里回踢 visit(this)，交出真实类型
class TextFile implements Element {
  final String name; final long bytes;
  TextFile(String n, long b) { name = n; bytes = b; }
  public void accept(Visitor v) { v.visit(this); }
}
class ImageFile implements Element {
  final String name; final long bytes; final int w, h;
  ImageFile(String n, long b, int w, int h){name=n;bytes=b;this.w=w;this.h=h;}
  public void accept(Visitor v) { v.visit(this); }
}

// ④ 对象结构：持有子元素，遍历时逐个 accept
class Directory implements Element {
  final String name;
  final List<Element> children = new ArrayList<>();
  Directory(String n) { name = n; }
  Directory add(Element e) { children.add(e); return this; }
  public void accept(Visitor v) {
    v.visit(this);                       // 先让访问者处理目录自身
    for (Element c : children) c.accept(v);  // 再递归下钻（遍历逻辑在结构侧）
  }
}
SizeVisitor.java一个操作 = 一个访问者，逻辑聚一处
// ⑤ 具体访问者：把“算总大小”的三段逻辑集中在一起
class SizeVisitor implements Visitor {
  private long total = 0;              // 访问者可携带累积状态
  public void visit(TextFile f)  { total += f.bytes; }
  public void visit(ImageFile f) { total += f.bytes; }
  public void visit(Directory d) { /* 目录本身不计大小 */ }
  long total() { return total; }
}

// —— 组装并跑一遍（Client 的活）——
Directory root = new Directory("root")
    .add(new TextFile("a.txt", 100))
    .add(new ImageFile("b.png", 900, 64, 64));

SizeVisitor sv = new SizeVisitor();
root.accept(sv);                     // 一次遍历，收集结果
System.out.println(sv.total());       // 1000

// 想要“统计文件数”？只写一个 CountVisitor，节点类零改动 ↑
看清收益：新增操作只碰一个文件。要加“导出 XML”，你只需再写一个 XmlVisitor implements Visitor，TextFile/ImageFile/Directory 一行都不改。而且“算总大小”这件事的全部代码都在 SizeVisitor 里——想读懂它不用翻三个节点类。这就是“按操作聚合代码”相对“按数据类型聚合”的核心优势。

六、致命权衡：它把开闭原则“掰弯”了

访问者不是银弹，它的适用性极度依赖“哪个维度会变”。一个数据结构有两个变化维度：元素种类（横向）和操作种类（纵向）。访问者对二者的态度截然相反：

变化方向不用访问者（方法写在节点里）用访问者
新增一种操作痛：改遍每个节点类爽：只写一个新 Visitor
新增一种元素爽：新节点类自带方法即可痛：所有 Visitor 都要加 visit(新元素)
逻辑内聚性按数据类型聚合按操作聚合
对开闭原则对“加元素”开放对“加操作”开放
元素稳定 + 操作多变  →  用访问者
元素多变 + 操作稳定  →  别用访问者
陷阱：元素频繁变动时，访问者是灾难。一旦你新增一种元素 VideoFile，就必须给每一个 Visitor 的接口加 visit(VideoFile)，然后所有实现类被迫全部改一遍——这正好把开闭原则反过来砸在脸上。所以决定用不用访问者前，先问自己：“未来是操作变得多，还是元素类型变得多？”只有前者，访问者才划算。这是它最大的适用性陷阱，也是面试高频追问点。

七、串联：编译器、JDK 与 Spring 里的访问者

访问者在“树形结构 + 多种遍历操作”的场景里几乎是标配，你早就用过它了：

  • 编译器 / AST：语法树节点（IfNode、BinaryOpNode…）相对稳定，但要对它做的操作层出不穷——类型检查、常量折叠、代码生成、格式化。每个都是一个访问者。这是访问者模式最经典、最贴切的舞台。
  • JDK 的 FileVisitor：Files.walkFileTree(path, visitor) 遍历目录树，对每个文件回调 visitFile、进出目录回调 preVisitDirectory / postVisitDirectory。文件树是稳定结构，你想干什么（删除、统计、拷贝）写进 SimpleFileVisitor 子类即可。
  • Java 注解处理 ElementVisitor：javax.lang.model.element.ElementVisitor 及其 visitType/visitExecutable/visitVariable，就是标准库里教科书级的访问者，配合 APT 在编译期扫描程序元素。
  • ASM 字节码框架：ClassVisitor、MethodVisitor、FieldVisitor 一整套访问者管线，逐个“拜访”字节码结构以做插桩/改写。
联系已学：访问者常与组合模式（第 17 篇）联手。上面 Directory 递归 accept 子节点，正是组合模式的树形结构；访问者负责“对这棵树做点什么”。组合搭骨架、访问者填操作，是处理层级数据的黄金组合。而当遍历顺序本身也需要灵活切换时，还能再叠一层迭代器模式把遍历逻辑抽出去。

八、访问者 vs 其他“回调型”模式

访问者、策略、命令都“把行为装进对象再传进来”，容易混。差别在于它们各自要解决的核心矛盾不同：

维度访问者策略命令
要解决给稳定结构加新操作同一件事换算法把请求物化以便攒/撤/排队
关键机制双分派（accept 回调）接口 + 组合注入命令持有接收者
作用对象一整个对象结构（树/集合）单个上下文单个请求
擅长的扩展加操作易、加元素难加算法易加命令易
陷阱：别为“操作只有一两个、以后也不会加”的场景硬套访问者。它引入了 accept 回调、双分派、Visitor 接口一整套间接层，理解与维护成本都不低。如果操作稀少且稳定，把方法直接写在节点类里反而更清晰——这是过度设计的典型来源。记住模式是为“变化”服务的，没有对应的变化就没必要引入。

九、进阶取舍：反射式访问者能救“加元素难”吗？

为了缓解“新增元素要改所有访问者”的痛，有人用反射（instanceof 链或 getClass() 反射查表）做一个“通用 visit(Element)”，绕开重载。但这是有代价的：

Bad · 反射/instanceof 分派
void visit(Element e) {
  if (e instanceof TextFile) {...}
  else if (e instanceof ImageFile){...}
  else throw new Error("未知元素");
  // 编译期不再检查“漏了哪种元素”
  // 加元素忘改这里 → 运行期才炸
  // 反射有性能开销
Good · 重载式双分派
void visit(TextFile f)  {...}
void visit(ImageFile f) {...}
// 加了 VideoFile 却没实现 visit
// → 编译期直接报错，逼你补齐
// 无反射开销，JIT 可内联
// 类型安全由编译器兜底
取舍结论：经典重载式访问者用“加元素麻烦”换来了编译期类型安全——漏处理某种元素编译就过不了。反射式虽然加元素时不用动接口，却把错误推迟到运行期，还牺牲性能。除非元素种类确实频繁变动，否则优先用经典写法，让编译器替你守住底线。

十、一句话收束

访问者模式 = 把“操作”从“数据结构”里剥离出来，用一次 accept 回调实现双分派，从而在不改节点类的前提下自由新增操作。记牢三点：① 五角色——元素只管 accept(v){ v.visit(this) }，访问者按操作聚合逻辑；② 精髓是双分派，用“虚方法定元素类型 + 重载定操作”手工拼出 Java 缺失的二维分派；③ 它对加操作友好、对加元素敌对——只有“元素稳定、操作多变”（如 AST、文件树、字节码）才该用它。撞脸策略/命令时，看你是不是要“给一整个稳定结构不断加新操作”——是，就是访问者。
