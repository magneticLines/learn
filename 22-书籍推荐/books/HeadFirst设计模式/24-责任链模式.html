责任链模式：请求沿链传递

把一堆处理者串成一条链，请求丢进链头顺着往下传，谁能处理谁接手——发送者不必知道最终是谁在干活。

一、一个“层层上报”的场景

公司报销：你交一张单子，组长能批 1000 以内，超了转给经理（批 1 万以内），再超转给总监（批 10 万以内），还超就上CEO。你作为发起人，根本不用关心到底谁批的——你只管把单子递给组长，剩下的沿着这条“审批链”自己往上流，直到有人拍板。

这就是责任链模式（Chain of Responsibility）的全部直觉：把多个处理者连成一条链，请求沿链传递，每个处理者要么自己处理掉，要么甩给下一个。请求的发送者与接收者被彻底解耦——发送者只认识链头，压根不知道链上有几个人、最后落到谁头上。

它到底解决了什么问题？“一个请求可能由多个对象之一处理，但具体由谁处理在编译期无法确定”。若不用责任链，你就得在发送方写一大坨 if (amount < 1000) 组长; else if (amount < 10000) 经理; else ...——发送方被迫认识所有处理者，还把分派逻辑和业务逻辑搅在一起。责任链把“谁来处理”的决策分散到各处理者自己身上，发送方就此解放。

二、模式定义与角色

GoF 定义：使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。将这些对象连成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。

  • Handler（抽象处理者）：定义处理请求的接口，并持有一个指向下一个处理者的引用（next）。这个 next 就是“链”的物理载体。
  • ConcreteHandler（具体处理者）：判断自己能否处理；能就处理，不能就调用 next 把请求转下去。
  • Client（客户端）：组装链、把请求丢给链头。
<<abstract>> Handler - next : Handler ←── 指向下一环，这就是“链” + setNext(h) + handle(request) ▲ ┌──────┼───────┐ 组长 经理 总监 (都继承 Handler) handle() handle() handle() Client ──▶ 组长.handle(单子1200) │ 我只能批1000，超了 → next.handle() ▼ 经理.handle(单子1200) │ 我能批1万，搞定！ (链在此终止，不再往下) ▼ [ 后面的总监/CEO 根本没被打扰 ]
request → H1 → H2 → H3 → … → null
每一环：能处理就拦下，否则 next.handle(request) 往下抛

三、最小实现：报销审批链

核心是抽象类里那个 next 字段和一个统一的 handle 模板：本环处理不了就委托给下一环。注意每个具体处理者只关心“我这一档的规则”，对链的整体结构一无所知。

ApprovalChain.javaHandler 持有 next，请求顺链下沉
abstract class Approver {              // 抽象处理者
  protected Approver next;              // ← 指向下一环，构成链
  protected final String name;

  Approver(String name) { this.name = name; }

  Approver setNext(Approver next) {   // 返回 next，方便链式拼接
    this.next = next;
    return next;
  }

  public final void handle(double amount) {  // 模板方法：定死“处理不了就往下”的骨架
    if (canApprove(amount)) {
      System.out.println(name + " 批准了 " + amount);
    } else if (next != null) {
      next.handle(amount);           // ← 关键：甩锅给下一环
    } else {
      System.out.println("没人能批 " + amount + "，请走特批流程");
    }
  }
  protected abstract boolean canApprove(double amount); // 各档自定规则
}

class TeamLead extends Approver {
  TeamLead(String n) { super(n); }
  protected boolean canApprove(double a) { return a <= 1_000; }
}
class Manager extends Approver {
  Manager(String n) { super(n); }
  protected boolean canApprove(double a) { return a <= 10_000; }
}
class Director extends Approver {
  Director(String n) { super(n); }
  protected boolean canApprove(double a) { return a <= 100_000; }
}

// 客户端：组装链，只把请求丢给链头
Approver lead = new TeamLead("组长");
lead.setNext(new Manager("经理"))
    .setNext(new Director("总监"));   // 组长 → 经理 → 总监

lead.handle(1_200);   // 组长批不了 → 经理批准了 1200.0
lead.handle(80_000);  // 组长→经理都不行 → 总监批准了 80000.0
发现了吗？这里悄悄用了“模板方法模式”。父类 handle 用 final 锁死了“能处理就处理、否则往下传”的骨架，把“能不能处理”这个变化点抽象成 canApprove 让子类填。责任链天然和模板方法是好搭档——这正是 CLAUDE.md 强调的建立联系：模式很少单独出现，它们互相咬合。

四、反例 → 正例：干掉 if-else 巨兽

责任链最经典的价值，就是把一坨随需求膨胀的 if-else / switch 拆成一个个可插拔的处理者。看对比：

Bad · if-else 集中分派
double approve(double amt) {
  if (amt <= 1000) {
    // 组长逻辑
  } else if (amt <= 10000) {
    // 经理逻辑
  } else if (amt <= 100000) {
    // 总监逻辑
  } else if (...) { // CEO...
  }
  // 加一档 → 改这个方法（违反开闭）
  // 分派 + 业务 全糊在一起
  // 发送方认识所有处理者
}
Good · 责任链拆分
// 每档 = 一个独立 Handler
// 顺序 = 链的拼接方式
lead.setNext(mgr)
    .setNext(director);

lead.handle(amt); // 发送方只认链头

// 加一档？ new 一个 Handler
//   插进链里，源码零改动
// 调顺序？ 换 setNext 的连法
// 处理逻辑各自内聚、可单测
本质是“把分支拆成对象”。if-else 把所有分支硬编码在一个方法里，链却把每个分支变成一等公民（对象）——可以单独测试、复用、动态增删、运行期改顺序。这就是行为型模式反复在做的事：把“行为/流程”从写死的控制流里提出来，变成可组合的对象。策略模式提炼“单个算法”，责任链提炼“一串带顺序的处理步骤”。

五、两种“味道”：纯的 vs 不纯的责任链

责任链在工程里其实有两种截然不同的用法，面试常拿来区分：

维度纯的责任链（Pure）不纯的责任链（Impure）
谁处理恰好一个接手，接手即终止每一环都插一手，处理后继续传
典型语义“找到合适的人”——审批、异常上抛“流水线加工”——层层包裹/校验
请求走向命中即 return，短路贯穿整条链，常有“去程 + 回程”
现实例子报销审批、try/catch 匹配Servlet Filter、拦截器、Netty Pipeline

“不纯的链”里，每一环常常在 next 调用的前后都做事——请求进来时走一遍（去程），结果返回时再走一遍（回程），形成“洋葱模型”。这跟你学过的装饰者味道很像：

请求 → Filter1 → Filter2 → Filter3 → Servlet │pre │pre │pre │ │ │ │ ▼(真正处理) │post ◄ │post ◄ │post ◄ ┘ 响应 ◄───┘ └──────────┘ 每一环 = chain.doFilter() 前做“去程”，后做“回程” (洋葱模型)

六、Java 世界里到处是它

责任链是最“实战”的模式之一，你天天在用只是没察觉：

  • Servlet Filter：doFilter(req, resp, chain) 里调 chain.doFilter(...) 就是“传给下一环”。这是不纯链的教科书实现（ApplicationFilterChain 内部用数组 + 指针推进）。
  • Spring MVC HandlerInterceptor、Spring Security FilterChain：认证、鉴权、日志一环接一环。
  • Netty ChannelPipeline：一串 ChannelHandler，入站事件顺链传、出站逆链传——典型双向责任链。
  • OkHttp Interceptor：chain.proceed(request) 层层下探，重试、缓存、网络各占一环，回程再层层加工响应。
  • Log4j / SLF4J 的 Logger 层级：日志事件沿 Logger 父子链上传，additivity 决定是否继续往父级传——这是纯度介于两者间的变体。
  • JS 事件冒泡：DOM 事件从目标元素沿父节点往上冒，任一层 stopPropagation() 就截断——浏览器内建的责任链。
联系底层：它和异常处理是一回事。Java 的 try/catch 本质就是责任链——异常沿调用栈向上抛，逐层寻找能 catch 住它的处理者，找到就停，找不到就一路抛到 JVM 顶层。栈帧序列就是那条“链”，catch 块就是 canApprove 的判断。理解了责任链，你就理解了异常机制的设计哲学。

七、常见陷阱（面试高频）

坑一：忘了调 next，请求“断链”。不纯链里最容易犯——你在 doFilter 里写完前置逻辑，却忘了 chain.doFilter(req, resp)，后面所有 Filter 和真正的业务全被你吃掉了，页面莫名 404 / 白屏。写 Filter/拦截器时，“是否放行、在哪放行”必须想清楚。
坑二：链没有兜底，请求“掉地上”。纯链若走到链尾还没人处理，又没写默认分支，请求就被静默丢弃。务必给链尾一个兜底 Handler 或抛异常。
坑三：环形链导致死循环。setNext 连错把链接成了环，请求转圈直到 StackOverflowError。
坑四：链太长带来性能与调试成本。每个请求都要走完可能很长的链，且调用栈很深，出错时堆栈难读——不要为了“优雅”把本该一个 if 解决的事拆成十环。

八、别搞混：责任链 vs 装饰者 vs 命令

三者都可能“一环套一环”，但意图不同——这是行为型/结构型交界处的经典辨析：

模式核心意图关键区别
责任链让请求在处理者链上流动，找到合适的处理者每环可选择终止或转发；发送者不知道谁最终处理
装饰者层层增强同一个对象的功能每层必然调用内层，接口不变，为的是“加料”而非“分派”
命令把“请求”封装成对象，以便排队/记录/撤销关注“请求本身对象化”，不关注谁沿链处理
一句话辨析：装饰者是“每层都得干活并往里传”（洋葱，不可短路）；责任链是“谁想干谁干，能截就截”（可短路）。所以“不纯的责任链”和装饰者结构极像，差别在能否中途终止：Filter 里你可以不调 chain.doFilter 直接返回（短路），这在装饰者语义里是不允许的。

九、什么时候用 / 不用

适合：多个处理者、顺序敏感、动态增删、发送方不该认识接收方 不适合：只有一两个固定分支、追求极致性能
  • 该用：处理步骤会随需求增删（过滤器、审批流、规则引擎）；发送者与接收者需要解耦；处理顺序需要灵活配置。
  • 别硬套：分支就俩且永远不变，写个 if 更清楚；对性能极敏感的热路径，别为“优雅”付出长链的调用开销。

十、一句话收束

责任链模式 = 把处理者串成一条链（每人持有 next），请求丢进链头顺流而下，谁能处理谁接手，从而彻底解耦请求的发送者与接收者。记牢四点：① 物理载体就是每个 Handler 里的 next 引用；② 分两种味道——纯链命中即短路（审批、异常匹配），不纯链层层贯穿呈洋葱模型（Filter、拦截器、Netty Pipeline）；③ 它把膨胀的 if-else 拆成可插拔对象，践行开闭原则；④ 头号大坑是忘调 next 导致断链、以及链无兜底让请求掉地上。看懂它，你也就看懂了 try/catch 和 Servlet 过滤器的底层设计。
