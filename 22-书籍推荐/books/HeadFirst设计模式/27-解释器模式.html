解释器模式：定义语言语法

为一种"小语言"定义文法，再把每一条文法规则做成一个类，让对象树自己解释自己。

一、从一个反复出现的需求说起

做业务系统时，你迟早会遇到这类需求：运营想自己配优惠规则——"会员 且 金额大于 100"、"新用户 或 周末下单"；风控想配拦截规则；报表想配筛选条件。这些规则天天变，如果每来一条就发一次版、写一个 if，程序员会被逼疯。

本质上，你需要的是一门只解决这一类问题的"小语言"（DSL，领域特定语言）：让用户用文本写规则，程序把文本读进来当场执行。解释器模式就是回答一个问题——当你已经有了一门小语言，如何用面向对象的方式把它"跑起来"。

一句话定义：给定一门语言，解释器模式为它的文法（Grammar）定义一套类表示，并给出一个解释器，用这套表示来解释语言中的句子。核心洞察：文法规则里的每一条产生式，都对应一个类。规则递归，类也递归；句子被解析成一棵抽象语法树（AST），解释 = 遍历这棵树。

二、先厘清"文法"这个词

别被"文法"吓到。它就是用有限的规则描述"什么样的句子是合法的"。以布尔规则语言为例，用 BNF（巴科斯范式）写出来只有四行：

expression ::= orExpression orExpression ::= andExpression ('OR' andExpression)* andExpression ::= primary ('AND' primary)* primary ::= variable | 'NOT' primary | '(' expression ')'

观察这份文法，两类符号泾渭分明，它们直接决定了你要写哪些类：

文法概念含义对应到解释器模式的角色
终结符
Terminal		不能再拆的最小单元，如 variable（是否会员）、字面量 trueTerminalExpression
树的叶子节点
非终结符
Non-terminal		由别的符号组合而成，如 AND、OR、NOTNonterminalExpression
树的内部节点，持有子表达式
关键类比：解释器模式其实是组合模式（第 17 章）在"语言"场景下的一次专门应用。终结符 = 叶子，非终结符 = 组合节点，两者共享同一个 interpret() 接口——所以你能像操作一个对象一样操作整棵表达式树。前面学的组合模式在这里"变现"了。

三、四个角色 + 一棵树

把上面的文法翻译成 UML 角色，只有四个：

┌─────────────────────────┐ │ Expression (抽象) │ AbstractExpression │ + interpret(ctx): bool │ 所有节点的公共接口 └────────────▲────────────┘ │ 都实现 interpret ┌──────────────┼───────────────┐ │ │ │ ┌──────────────┐ ┌──────────┐ ┌──────────────┐ │ Variable │ │ AndExpr │ │ OrExpr / Not │ │ (终结符/叶子) │ │(非终结符) │ │ (非终结符) │ └──────────────┘ └────┬─────┘ └──────────────┘ │ 持有 left / right 子表达式（递归组合） ▼ 句子 "member AND (amount_gt_100 OR weekend)" 被解析成： AndExpr / \ Variable OrExpr (member) / \ Var(amount) Var(weekend) Context（上下文）：{ member:true, amount:false, weekend:true } 从根节点调用 interpret(ctx)，递归向下求值 → 结果 true
  • AbstractExpression：声明 interpret(Context)，是树上所有节点的统一契约。
  • TerminalExpression：终结符，直接从 Context 取值返回，是递归的出口。
  • NonterminalExpression：非终结符，先递归解释子节点，再组合结果（AND/OR/NOT）。
  • Context：解释时的"全局环境"，存放变量到值的映射（这里是"哪些标签为真"）。
句子 解析→ AST 遍历→ 递归 interpret() → 结果

四、最小可运行示例：会员优惠规则引擎

下面这段代码可以直接跑。注意每个类如何精确对应文法里的一条规则。

BooleanRuleInterpreter.javaJava 17
// ① Context：解释时的环境，记录每个标签是否成立
record Context(Map<String, Boolean> flags) {
    boolean lookup(String name) { return flags.getOrDefault(name, false); }
}

// ② AbstractExpression：全树统一契约（对应组合模式的 Component）
interface Expr { boolean interpret(Context ctx); }

// ③ TerminalExpression：终结符 = 叶子，递归出口
record Variable(String name) implements Expr {
    public boolean interpret(Context ctx) { return ctx.lookup(name); }
}

// ④ NonterminalExpression：非终结符，持有子表达式，递归组合
record And(Expr left, Expr right) implements Expr {
    public boolean interpret(Context ctx) {
        return left.interpret(ctx) && right.interpret(ctx); // 先解释左右子树
    }
}
record Or(Expr left, Expr right) implements Expr {
    public boolean interpret(Context ctx) {
        return left.interpret(ctx) || right.interpret(ctx);
    }
}
record Not(Expr inner) implements Expr {
    public boolean interpret(Context ctx) { return !inner.interpret(ctx); }
}

// ⑤ 客户端：手工搭出 AST（真实项目里由 Parser 从文本生成）
// 规则："会员 且 (金额>100 或 周末)"
Expr rule = new And(
    new Variable("member"),
    new Or(new Variable("amount_gt_100"), new Variable("weekend")));

Context ctx = new Context(Map.of("member", true, "amount_gt_100", false, "weekend", true));
System.out.println(rule.interpret(ctx)); // true：member=true 且 weekend=true
解释器 ≠ 解析器。解释器模式只负责"给我一棵树，我来求值"（图中的 ⑤ 是手搭的树）。把文本 "member AND (...)" 变成树，是解析（Parsing）的活，属于另一个话题（词法分析 + 语法分析）。GoF 明确说：本模式不涉及如何创建 AST。真实项目里往往用 ANTLR、JavaCC 生成解析器，或手写递归下降解析器，产物才是喂给解释器的树。

五、为什么这么设计？对比"一坨 if-else"

不用解释器，你多半会写一个巨型 evaluate 方法，把规则硬编码进 if。它的致命伤是：规则一变就要改代码、重新发版，且无法让运营自助配置。

反例 · 硬编码规则
boolean canDiscount(Order o) {
  // 规则揉进代码，每加一条就改这里
  if (o.isMember()
       && (o.amount() > 100
           || o.isWeekend())) {
    return true;
  }
  // 下周产品又要加"或首单"...
  // → 改代码 + 测试 + 发版
  return false;
}
正例 · 解释器 + 可配置文法
// 规则以文本/配置存在 DB
String dsl = ruleRepo.load("discount");
Expr ast  = parser.parse(dsl);
boolean ok = ast.interpret(ctx);

// 产品要加"或首单"？
// 运营改一行配置：
//   ... OR first_order
// 无需改代码、无需发版
取舍的本质：解释器把"规则"从编译期硬编码迁移到了运行期数据。代价是要额外维护文法、解析器与一批表达式类；收益是规则可扩展、可配置、可组合。只有当"同一类语句反复出现且频繁变化"时，这笔投资才划算。

七、陷阱与面试高频点

陷阱一：滥用——类爆炸。文法每多一条规则就多一个类。语法一旦复杂（几十条产生式），类的数量和维护成本会失控，AST 的递归也难调试。复杂语言请直接上 ANTLR / JavaCC 等成熟工具，不要手写解释器。GoF 明确定位：本模式只适合简单、稳定、频繁复用的小文法。
陷阱二：把"解析"当成模式的一部分。面试常问"解释器模式怎么把字符串变成树？"——正确回答是它不管解析，解析是词法/语法分析的职责，模式只负责树的求值。混淆这两步是最典型的理解偏差。
陷阱三：性能。每次调用都递归遍历整棵树，是解释执行，天然比编译执行慢。热点规则可考虑缓存 AST（别每次重新 parse）、或把 AST 编译成字节码/表达式（如 Spring SpEL 的 compilerMode、Janino 动态编译）来提速。

一句话收束：解释器模式 = 文法产生式 → 类，句子 → 抽象语法树，解释 → 递归遍历这棵树。它本质是组合模式在"语言"领域的落地，让你能用一门可配置的小语言把易变规则从代码里解放出来。用它的前提是文法简单且稳定——一旦复杂，交给 ANTLR，别自己造轮子。
