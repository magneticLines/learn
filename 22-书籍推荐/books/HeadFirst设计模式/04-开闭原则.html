开闭原则：对扩展开放，对修改关闭

加新功能时，你应该"写新代码"而不是"改旧代码"——这是几乎所有设计模式的共同追求。

一句话看懂：开什么，闭什么

开闭原则（Open-Closed Principle, OCP）由 Bertrand Meyer 于 1988 年在《面向对象软件构造》中提出，后被 Robert Martin（Uncle Bob）纳入 SOLID 五大原则，是其中的第一条。完整表述是：

OCP 定义：软件实体（类、模块、函数）应当对扩展开放（Open for extension），对修改关闭（Closed for modification）。
翻译成人话：当需求变化、要加新功能时，你应该通过新增代码来满足它，而不是回头去改动那些已经写好、已经测过、已经上线的旧代码。

初学者第一次听会觉得矛盾：既然不让改，功能怎么加？ 关键在于"开"和"闭"针对的是不同的东西——对行为开放（可以有新行为出现），对源码关闭（旧文件一行不动）。

加功能 = 新增一个类 ✅  ≠  改动旧的类 ❌

为什么这样设计？"改代码"的隐藏代价

你可能想：需求变了改一行代码怎么了？问题在于，"改"是有传染性的、有风险的。想象一台已经封装好、贴了质检合格标签的机器——每次扩展都去撬开它，迟早出事。

生活类比：墙上插座 vs 焊死的电器 ┌───────────────────────────────────────────┐ │ 违反 OCP：想接新电器 → 撬开墙面重新焊线 │ │ 焊错一根 → 全屋跳闸（改旧代码引入回归 bug） │ └───────────────────────────────────────────┘ ↑ 每次扩展都动"内脏" ┌───────────────────────────────────────────┐ │ 遵守 OCP：墙上预留插座（稳定的抽象接口） │ │ 新电器直接插上 → 墙里的线一根不动 │ └───────────────────────────────────────────┘ ↑ 扩展点稳定，内部封闭

改动旧代码至少带来三重代价：

  • 回归风险：动了 A 功能可能悄悄弄坏 B 功能，旧的测试、旧的调用方全部需要重新验证。
  • 协作冲突：一个方法被十个功能塞满 if-else，十个人改同一个文件，天天 merge 冲突。
  • 违反"稳定依赖"：越核心、被依赖越多的代码越应该稳定；频繁修改核心类等于在地基上施工。

坏味道：不断膨胀的 if / switch

识别是否违反 OCP，最简单的信号：每来一个新需求，你都要回到同一个方法里加一个 else if 或 case。看一个计算图形面积的例子——需求从"圆形+矩形"扩展到"再加三角形"时会发生什么。

✗ BAD · 每加图形就改这个方法
class AreaCalculator {
  double area(Object shape) {
    if (shape instanceof Circle c)
      return Math.PI * c.r * c.r;
    else if (shape instanceof Rect r)
      return r.w * r.h;
    // 加三角形？回来再塞一个 else if
    // 这个类永远"改不完"
    return 0;
  }
}
✓ GOOD · 面积算法内聚到各自的类
interface Shape { double area(); }

class Circle implements Shape {
  double r;
  public double area() { return Math.PI*r*r; }
}
class Rect implements Shape {
  double w, h;
  public double area() { return w*h; }
}
// 加三角形：只写一个 new class，
// AreaCalculator 一行都不用改

正例里，AreaCalculator 只依赖稳定的抽象 Shape，具体算法被"封"进各个实现类。新增图形 = 新增一个类：

Java对扩展开放：新增而非修改
// 需求：再支持三角形。既有代码全部不动，只新增下面这个类
class Triangle implements Shape {
    double base, height;
    public double area() { return 0.5 * base * height; }
}

class AreaCalculator {
    // 依赖抽象 List<Shape>，对任何 Shape 实现都成立
    double total(List<Shape> shapes) {
        return shapes.stream()
                     .mapToDouble(Shape::area)
                     .sum();
    }
}
抓住本质：OCP 不是"禁止一切修改"，而是要求你提前识别出"会变的部分"，把它抽象成一个稳定的扩展点（接口 / 抽象类）。之后变化都发生在扩展点"之外"的新实现里，核心逻辑对这些变化保持关闭。这正是前一节"封装变化 + 针对接口编程"的直接延续。

靠什么实现 OCP？三根支柱

OCP 不是凭空实现的，它落地依赖三个面向对象机制。缺了抽象与多态，OCP 就是一句空话。

机制作用对应模式举例
抽象（接口/抽象类）定义稳定的"扩展点"，是"闭"的边界策略、模板方法
多态运行时替换实现，让新行为"插进来"策略、状态
组合 + 依赖注入把具体实现从外部传入，而非硬编码 new装饰者、依赖注入

反过来看：几乎每一个设计模式，都是"用某种结构去满足 OCP"的固定套路。策略模式让算法可扩展、装饰者模式让功能可叠加、观察者模式让订阅方可增减——它们都在回答同一个问题：怎样加功能才不用改老代码？

它在 JDK 与 Spring 里无处不在

OCP 不是教科书概念，你每天用的框架都建立在它之上：

JavaCollections.sort：排序规则对扩展开放
// JDK 的排序算法（TimSort）早已写死、封闭，绝不因你的需求改动。
// 你只需"新增"一个 Comparator 实现，就能注入任意排序规则：
List<User> users = getUsers();

// 按年龄升序
users.sort(Comparator.comparingInt(User::getAge));

// 需求变成"按名字倒序"？不改 JDK、不改上面这行，再传一个新的比较器
users.sort(Comparator.comparing(User::getName).reversed());
  • JDK：Comparator / Comparable（排序规则可扩展）、InputStream 家族（装饰者，BufferedInputStream 包裹增强而非改原类）、Thread 接受 Runnable。
  • Spring：Bean 生命周期扩展点 BeanPostProcessor——想在 Bean 初始化前后插逻辑，实现这个接口注册进去即可，Spring 容器核心代码从不为你改动；此外还有 ApplicationListener（事件监听）、HandlerInterceptor（MVC 拦截器）。整个 Spring 就是"用扩展点欢迎你，用 final/private 保护核心"的典范。
联系操作系统：Linux 的 可加载内核模块（LKM）、VFS（虚拟文件系统）也是 OCP 的体现——内核定义好 file_operations 这组函数指针接口，新增一种文件系统（ext4、NTFS）只需实现这组接口并挂载，内核主干代码无需重新编译。抽象接口 + 多态派发，在 C 语言里用函数指针，在 Java 里用虚方法表，本质是同一件事。

陷阱：别把 OCP 变成"过度设计"

高频面试追问 & 常见误区：
  • 为抽象而抽象：只有一种实现、且看不到变化趋势时，就急着抽接口、加工厂，制造出一堆只有单个实现的接口。YAGNI（You Aren't Gonna Need It）优先于 OCP——变化没出现前，过早抽象是负债。
  • 误以为"绝不能改代码"：OCP 是理想，不是铁律。抽象本身设计错了就必须改；正确姿势是把修改集中、控制在"扩展点定义"这一处，而不是散落在业务逻辑里。
  • "对扩展开放"≠"随便继承"：暴露的扩展点应是接口/抽象方法，而不是让人 extends 你的具体类去 override 一堆细节——那会违反里氏替换、破坏封装（对应上一节"多用组合，少用继承"）。
  • 面试话术：被问"如何理解 OCP"时，务必点出"识别变化点 → 抽象出稳定接口 → 用多态承接扩展"这条主线，并能举出 Comparator 或策略模式作为落地例子，而不是只背定义。

与相邻原则的取舍对比

维度开闭原则 OCPKISS / YAGNI
核心诉求为"预期的变化"预留扩展点，降低改动成本先做最简单能用的，别为"想象的变化"埋伏笔
过度时的后果过度抽象：接口满天飞，代码绕、难读过度简化：变化真来时被迫大改，回归风险高
何时优先变化点已明确 / 已经改过第二次时需求早期、变化方向不明时
实战经验（三次法则）：不必第一次写就上抽象。第一次直接写；第二次复制时忍住；第三次再出现类似变化，就该抽象出扩展点了。此时变化方向已清晰，OCP 的收益才真正大于成本。

小结

一图流记住 OCP：
  • 目标：加功能靠"新增类"，而不是"修改已上线的旧类"，从而隔离回归风险。
  • 手段：识别变化点 → 抽象成稳定接口 → 用多态 / 组合承接新实现。
  • 地位：它是设计模式的"总纲"，策略、装饰者、观察者等模式都是达成 OCP 的具体套路。
  • 边界：OCP 服务于"应对变化"，不为想象中的变化过度设计——先 YAGNI，变化露头（第三次）再抽象。
