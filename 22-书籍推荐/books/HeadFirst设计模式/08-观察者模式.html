观察者模式：一对多依赖，主题通知观察者

就像订报纸——出版社不认识每一个订户，但报纸一出刊，所有订户的信箱都会自动收到。一变多知，松耦合是灵魂。

一、开场：气象站的"一变全知"难题

这是《Head First 设计模式》第 2 章的招牌案例。你要做一个气象站软件：有一个 WeatherData 对象，它从物理传感器拿到温度、湿度、气压三个读数；每当读数更新，就要刷新三块布告板——当前状况显示、气象统计显示、天气预报显示。而且甲方明确说了："以后还会有第三方接入自己的布告板，接口要能随便扩展。"

新手最直觉的写法，是在 WeatherData 里塞一个"测量更新"方法，手动挨个通知每块布告板：

反直觉的"硬编码通知"——问题的根源Java
public class WeatherData {
    public void measurementsChanged() {
        float temp = getTemperature();
        float humidity = getHumidity();
        float pressure = getPressure();

        // ↓↓↓ 坏味道：针对【具体实现】编程，且是硬编码
        currentDisplay.update(temp, humidity, pressure);
        statisticsDisplay.update(temp, humidity, pressure);
        forecastDisplay.update(temp, humidity, pressure);
    }
}
这段代码违反了你已学的两条原则。 其一，它针对具体类编程——每加/删一块布告板，都得回来改 measurementsChanged() 的源码，第三方根本无法在运行时接入，违反开闭原则。其二，会变的部分（"有哪些观察者、怎么通知"）和不变的部分（"取三个读数"）搅在了一起，违反封装变化。真正的痛点是：WeatherData 和三块布告板互相知道对方是谁，耦合被焊死了。

二、生活类比：报纸订阅

《Head First》用了一个绝妙的类比——报纸/杂志订阅，它几乎就是观察者模式的字面翻译：

出版社(Subject / 主题) 订户(Observer / 观察者) ───────────────────── ────────────────────── ① 你去出版社【订阅】 ◄────subscribe──── 张三、李四、王五 ② 只要出新刊 ──────notify──────► 信箱自动收到，无需你去问 ③ 不想看了【退订】 ◄──unsubscribe─── 王五退订，其他人照收 ④ 新人随时可【加入】 ◄────subscribe──── 赵六今天才订，明天就收到 关键：出版社【不关心】订户具体是谁、有多少、住哪， 只认"订户名单"这一层抽象；订户来去自由，双方都不必改对方。

把这套关系抽象出来，就是模式的两个角色：主题（Subject / Observable）维护一份观察者名单，提供注册/注销/通知的能力；观察者（Observer）实现一个统一的回调接口，被通知时更新自己。二者之间由"一对多"的依赖连接——一个主题状态一变，所有登记在册的观察者都会被自动通知。

1 个 Subject  ──状态改变──►  通知 N 个 Observer

三、模式定义与松耦合的设计哲学

观察者模式（Observer Pattern）定义
定义对象之间的一对多依赖关系，这样一来，当一个对象（主题）改变状态时，它的所有依赖者（观察者）都会收到通知并自动更新。
Defines a one-to-many dependency between objects so that when one object changes state, all of its dependents are notified and updated automatically.

它背后站着一条核心设计原则——为交互对象之间的松耦合设计而努力（Strive for loosely coupled designs between objects that interact）。"松耦合"具体松在哪几处，值得逐条拆开看，这是理解本模式的钥匙：

松耦合体现说明
主题只知道观察者实现了某接口它持有的是 List<Observer>，从不关心具体是哪个类，符合"针对接口编程"
可随时增删观察者运行时 register / remove，主题代码一行不改——满足开闭原则
新增观察者类型无需改主题第三方写个新 Observer 实现，注册进来即可，双方独立演化
双方可独立复用主题不依赖具体观察者，观察者也不依赖具体主题，各自能被单独测试和替换
与已学知识串联： 松耦合的实现手段，正是你在前几节学到的组合拳——主题组合（HAS-A）了一份观察者列表（多用组合少用继承），依赖的是 Observer 接口而非实现（针对接口编程），从而让高层的 WeatherData 和低层的布告板都依赖抽象（依赖倒置）。观察者模式不是凭空出现的新东西，而是这些原则在"事件通知"这一场景下的组合成品。

四、动手实现气象站

先定义两个接口——这就是"变化"与"稳定"之间的那堵墙：

Subject 与 Observer 接口：契约层Java
public interface Subject {
    void registerObserver(Observer o);   // 订阅
    void removeObserver(Observer o);     // 退订
    void notifyObservers();              // 出新刊，广播
}

public interface Observer {
    // 主题把新数据"推"给观察者（push 模型，稍后讨论 pull）
    void update(float temp, float humidity, float pressure);
}

主题的实现是模式的心脏——它维护名单，并在状态变化时遍历通知：

WeatherData：具体主题Java
public class WeatherData implements Subject {
    // 名单：只认 Observer 接口，不认任何具体布告板
    private final List<Observer> observers = new ArrayList<>();
    private float temperature, humidity, pressure;

    public void registerObserver(Observer o) { observers.add(o); }
    public void removeObserver(Observer o)   { observers.remove(o); }

    public void notifyObservers() {
        for (Observer o : observers)          // 一对多广播
            o.update(temperature, humidity, pressure);
    }

    // 传感器新数据进来时被调用 → 触发广播
    public void measurementsChanged() { notifyObservers(); }

    public void setMeasurements(float t, float h, float p) {
        this.temperature = t; this.humidity = h; this.pressure = p;
        measurementsChanged();
    }
}

观察者实现回调，被通知时刷新自己；构造时顺手完成订阅：

CurrentConditionsDisplay：具体观察者 + 客户端Java
public class CurrentConditionsDisplay implements Observer {
    private final Subject weatherData;   // 持有主题引用，为的是能退订

    public CurrentConditionsDisplay(Subject wd) {
        this.weatherData = wd;
        wd.registerObserver(this);       // 自我登记，主动订阅
    }
    public void update(float t, float h, float p) {
        System.out.println("当前温度：" + t + "℃，湿度：" + h + "%");
    }
}

// —— 客户端：注意加一块新布告板，WeatherData 一行都不用改 ——
WeatherData wd = new WeatherData();
new CurrentConditionsDisplay(wd);       // 订阅
new ForecastDisplay(wd);                // 订阅（第三方也这么接入）
wd.setMeasurements(26, 65, 1013);       // 一次调用，全部布告板自动刷新
看清主动权在哪一侧。 是布告板主动去 registerObserver(this) 订阅，而不是气象站去"收集"布告板。这就是"好莱坞原则"——"别调用我们，我们会调用你（Don't call us, we'll call you）"：观察者向主题登记后就被动等待回调。这种控制反转（IoC）正是后面 Spring 容器、GUI 事件、回调式框架的思想内核。

五、Push vs Pull：两种数据传递取舍

上面 update(temp, humidity, pressure) 是 推模型（push）——主题把所有数据主动塞给观察者。但如果某块布告板只关心气压，它也被迫接收全部三个参数；将来主题多一个"风速"，接口签名一改，所有观察者都得跟着改。于是有了 拉模型（pull）：主题只通知"我变了"，观察者拿着主题引用自己去取需要的字段。

Push · 主题推全量数据
void update(float t, float h, float p);
// 主题：
o.update(temp, humidity, pressure);

// 痛点：
// · 参数写死，加字段=改所有观察者
// · 观察者被迫接收用不到的数据
// · 接口与具体数据结构强耦合
Pull · 观察者按需自取
void update(Subject src);   // 只告知"谁变了"
// 观察者：
void update(Subject src) {
  WeatherData wd = (WeatherData) src;
  float p = wd.getPressure(); // 只取我要的
}
// 收益：主题加字段，观察者可不改；
//      各观察者只依赖自己关心的 getter
JDK 里的官方选择： 早已废弃的 java.util.Observable 的 update(Observable o, Object arg) 同时支持两种——arg 非空是 push，观察者也可通过 o 拉数据。《Head First》作者也明说：大多数情况下 pull 更"正确"，因为它把"要什么数据"的决定权交回给观察者，耦合更松。但 push 在数据简单、性能敏感时更直接。没有绝对优劣，看数据量与变更频率权衡。

六、别自己造轮子？聊聊 java.util.Observable

JDK 从 1.0 就内置了 java.util.Observable（主题）和 java.util.Observer（观察者）接口。看起来能省事，但《Head First》专门用一节泼了冷水，Java 9 更是直接把它俩标记为 @Deprecated。原因很硬核：

缺陷后果
Observable 是类不是接口你的主题必须 extends Observable，Java 单继承，一旦继承它就没法再继承别的类，扩展性被锁死——这正是"多用组合少用继承"的反面教材
setChanged() 是 protected只能在子类内部调用，无法用组合方式把 Observable 当成员委托，逼你走继承这条死路
不支持序列化、线程安全弱内部状态无法可靠保存/传输，且通知顺序、并发保证都不明确，官方文档直接建议改用 java.beans.PropertyChangeSupport 或 java.util.concurrent.Flow（响应式流）
面试高频点：Observable 为什么被废弃？ 一句话——它把"主题"设计成了一个必须被继承的类，而非接口，从根上违背了面向对象设计原则。这也反向印证了《Head First》让你"亲手撸一遍接口版观察者"的良苦用心：理解模式本质，比记住某个现成 API 重要得多。今天真要用现成方案，选 PropertyChangeListener（JavaBeans）、Spring 事件、或 Flow.Publisher（Reactive Streams / Java 9+）。

七、观察者模式其实无处不在

一旦你识别出"一个变，多个自动响应"的结构，就会发现整个生态都是观察者：

场景Subject（主题）Observer（观察者）
Swing / AWT GUIJButtonActionListener——按钮被点，所有监听器的 actionPerformed 被回调
Spring 事件机制ApplicationEventPublisher@EventListener / ApplicationListener——发布 ApplicationEvent，容器广播给所有监听 Bean
JavaBeansPropertyChangeSupportPropertyChangeListener——属性值一变就通知
响应式编程 (Java 9+ / RxJava)Flow.Publisher / ObservableFlow.Subscriber——观察者模式加上背压、异步、操作符的强化版
消息中间件 (Kafka/MQ)Topic / 生产者消费者组——发布-订阅是观察者模式在分布式下的放大版
与操作系统 / 底层串联： 往下看，Linux 的 epoll、inotify 文件监听、信号（signal）机制，本质都是"事件源 + 回调"的观察者思想；往上看，前端的 addEventListener、Vue/MobX 的响应式依赖收集，也都是它。区别只在于：观察者模式是同步、进程内、一对多；而消息队列的发布-订阅引入了中间的 broker，做到了异步与跨进程解耦——可以理解为"观察者模式 + 一个中间人"。

八、四个必踩的坑

坑一：忘记退订 → 内存泄漏（Lapsed Listener）。 主题持有观察者的强引用。如果观察者生命周期比主题短（如一个短命的 UI 面板订阅了长寿的数据源），用完却没 removeObserver，它就永远无法被 GC 回收，还会持续收到无用通知。这是 GUI/安卓开发最经典的内存泄漏源。对策：显式退订，或让主题持有 WeakReference（弱引用）。
坑二：遍历通知时增删观察者 → ConcurrentModificationException。 若某观察者的 update() 里又调用了 removeObserver（比如"收到一次通知后就退订"），正在 for 遍历的那个 ArrayList 会被结构性修改，直接抛异常。对策：遍历前复制一份快照（new ArrayList<>(observers)）再遍历，或使用 CopyOnWriteArrayList——这也是 Swing、Spring 事件源码里的标准做法。
坑三：不要依赖通知顺序。 模式定义里从未承诺"按注册先后通知"。观察者之间必须相互独立，任何"A 一定在 B 之前收到"的假设都是脆弱的。若真有顺序需求，说明这里可能不该用观察者，或该引入显式的优先级机制（如 Spring 的 @Order）。
坑四：观察者里做重活 / 抛异常。 同步观察者模式下，notifyObservers() 是串行调用每个 update() 的。某个观察者里做了耗时 IO，会阻塞后面所有观察者；某个 update() 抛了异常且未捕获，通知链会中断，后面的观察者收不到通知。对策：观察者内部要快、要吞掉自己的异常，或改用异步/独立线程分发。

九、与相似模式的界限

观察者常和几个"看起来也在传消息"的模式混淆，划清边界能帮你在面试里答到点上：

模式关系形态核心区别
观察者一对多，主题→观察者状态变化时广播给所有登记者，观察者被动响应
发布-订阅 (Pub/Sub)多对多，中间隔 broker发布者与订阅者互不知道对方，靠消息通道/主题解耦，通常异步跨进程
中介者 (Mediator)多对多，星型汇聚对象间通信全走中介者，目的是消除对象间的网状引用，而非"一变多知"
责任链 (Chain)一条链，顺序传递请求依次传给处理者，通常只被一个处理掉，不是广播
一个判别口诀： 看到"一个状态源，多个方要同步跟着变，且它们来去自由"，就是观察者；如果发送方和接收方之间需要完全互不感知并跨系统，那是发布-订阅（观察者 + broker）。
一句话收束： 观察者模式用一份"观察者名单"，把"谁在关心我"这件会变的事从主题里彻底封装出去，换来主题与观察者之间的松耦合——主题只管在状态变化时对着接口喊一嗓子，剩下的"谁收、怎么处理、来去增删"全交给观察者自理。它是事件驱动、GUI、Spring 事件、响应式编程共同的地基。记住它的灵魂只有一句：为交互对象之间的松耦合而设计。
