适配器模式：接口转换

把不兼容的接口"翻译"成客户端期望的样子——就像出国旅行时的电源转换插头。

一、从一个真实的痛点说起

你有一台国内买的笔记本，插头是两脚扁头；到了英国，墙上的插座是三脚方孔。两者都能供电，但物理接口对不上。你不会因此换掉笔记本，也不可能砸墙改插座——你买一个转换插头：一头插进英标插座，另一头露出你熟悉的两脚孔。

这就是适配器模式的全部精神：接口不兼容，但双方各自都不能改，于是引入一个中间层做转换。

一句话定义：适配器模式（Adapter）将一个类的接口转换成客户端期望的另一个接口，使原本因接口不兼容而无法协作的类能够一起工作。它的关键词是"转换"，不新增功能，只做翻译。

二、为什么需要它？解决了什么问题

软件里最常见的三种"接口对不上"场景：

  • 老代码 + 新系统：遗留模块的接口是 getXml()，新系统统一按 getJson() 调用。
  • 第三方库：你依赖的日志框架方法叫 writeLog()，而你项目里到处用的是 log(msg)。
  • 两套已定型的接口：都不能改（老代码怕出 bug，第三方库你无源码），只能加一层。
核心价值：适配器让你在不修改现有代码的前提下复用它。这直接呼应"开闭原则"（对扩展开放、对修改关闭）——你通过新增一个适配器类来扩展兼容性，而不是改动已经稳定的老类。

三、角色与结构

适配器模式里只有三个角色，务必记牢它们的关系：

角色说明插头类比
Target（目标接口）客户端期望调用的接口你笔记本的两脚插头形状
Adaptee（被适配者）已存在、接口不兼容的类英标三脚方孔插座
Adapter（适配器）实现 Target，内部委托给 Adaptee转换插头本身
客户端 Client │ 只认识 Target 接口 ▼ ┌───────────────────┐ ┌────────────────────┐ │ Target │ │ Adaptee（老类） │ │ + request() │ │ + specificRequest()│ └─────────▲─────────┘ └──────────▲─────────┘ │ implements │ 内部持有并调用 │ │ ┌─────────┴───────────────────────────┴──┐ │ Adapter │ │ - adaptee : Adaptee │ │ + request() { adaptee.specificRequest();}│ └──────────────────────────────────────────┘ Client → Target.request() → Adapter 转译 → Adaptee.specificRequest()
Client ──调用──▶ Target ◀实现── Adapter ──委托──▶ Adaptee

四、最小代码示例：鸭子与火鸡

《Head First》的经典例子：客户端只会用Duck（会 quack() 呱呱叫、fly() 飞）。现在手上只有Turkey（会 gobble() 咯咯叫、飞得很近）。我们要让火鸡假装成鸭子被使用。

Duck.java / Turkey.java被适配的两套接口
// Target：客户端期望的接口
interface Duck {
    void quack();
    void fly();
}

// Adaptee：已存在、接口不同的类（假设来自第三方，不能改）
interface Turkey {
    void gobble();   // 火鸡咯咯叫
    void fly();      // 火鸡也能飞，但飞不远
}
TurkeyAdapter.java核心：对象组合 + 委托
// Adapter：实现 Target，内部"包住"一个 Adaptee
public class TurkeyAdapter implements Duck {
    private final Turkey turkey;   // 组合，而非继承

    public TurkeyAdapter(Turkey turkey) {
        this.turkey = turkey;
    }

    @Override
    public void quack() {
        turkey.gobble();       // 把 quack 翻译成 gobble
    }

    @Override
    public void fly() {
        // 火鸡飞不远，调 5 次凑数——转换不总是 1:1
        for (int i = 0; i < 5; i++) turkey.fly();
    }
}

// 客户端全程只认识 Duck，对 Turkey 一无所知
Duck duck = new TurkeyAdapter(new WildTurkey());
duck.quack();  // 实际执行 gobble()
关键点：适配器实现目标接口 Duck，组合被适配对象 Turkey。客户端拿到的是一个货真价实的 Duck，它对 Turkey 的存在毫不知情——这正是"面向接口编程"的威力。

五、对象适配器 vs 类适配器

适配器有两种实现方式，取决于你用"组合"还是"继承"接住 Adaptee：

维度对象适配器（组合）类适配器（多继承）
实现手段持有 Adaptee 实例，委托调用同时继承 Adaptee 和实现 Target
Java 可行性✅ 推荐⚠️ 受限，Java 单继承，Adaptee 须为接口
能否适配子类✅ 可适配 Adaptee 及其所有子类❌ 只能适配某个具体类
耦合度低（依赖抽象）高（绑死具体类）
符合原则合成复用原则（组合优于继承）违背"少用继承"的建议
为什么 Java 里几乎只用对象适配器？ C++ 支持多重继承，可以让 Adapter 同时 public 继承 Target 和 Adaptee，这就是类适配器。但 Java 类只能单继承，只有当 Adaptee 恰好是接口时才能勉强凑出类适配器。所以在 Java 生态里，对象适配器（组合）是绝对主流。

六、坏味道对比：不用适配器会怎样

如果不引入适配器，直接在客户端里"就地改造"火鸡，会发生什么？

Bad · 到处 if-else 硬判断
void makeItQuack(Object o) {
  if (o instanceof Duck) {
    ((Duck) o).quack();
  } else if (o instanceof Turkey) {
    // 客户端被迫知道 Turkey
    ((Turkey) o).gobble();
  }
  // 每来一种新动物，
  // 这里就得再加一个分支……
}
Good · 适配器统一接口
void makeItQuack(Duck duck) {
  duck.quack();  // 只认 Duck
}

// 火鸡？包一层就行
makeItQuack(
  new TurkeyAdapter(new WildTurkey())
);
// 客户端零改动，
// 完全不知道 Turkey 存在
陷阱提醒：坏例里的 instanceof + 强转会让客户端耦合每一种被适配类，违背开闭原则，且类型判断散落各处难以维护。适配器把这些脏活集中到一个类里，客户端保持干净。看到大片 instanceof 分支时，就该想想是不是缺一个适配器。

七、JDK 与 Spring 中的真实身影

适配器不是教科书里的玩具，JDK 和主流框架里到处都是：

出处如何体现适配器
java.io.InputStreamReader把字节流 InputStream（Adaptee）适配成字符流 Reader（Target），内部靠 StreamDecoder 做编码转换
java.util.Arrays.asList()把数组（Adaptee）适配成 List 接口（Target）
Collections.enumeration()新老遍历接口互转：Iterator ↔ Enumeration
Spring MVC HandlerAdapter把千奇百怪的 Controller（@Controller/HttpRequestHandler/Servlet）统一适配成 DispatcherServlet 能调用的形式
Spring AOP AdvisorAdapter把不同类型的 Advice 适配成统一的 MethodInterceptor
面试高频：Spring MVC 的 HandlerAdapter 是最爱被问的例子。DispatcherServlet 拿到 handler 后不直接调用，而是找到匹配的 HandlerAdapter，由它统一以 handle(request, response, handler) 调用——这样新增一种 Controller 写法，只要加一个 Adapter，DispatcherServlet 主流程一行都不用改。请务必能说出"为什么不用 if-else 判断 handler 类型"（答：开闭原则 + 消除耦合）。

八、和"长得像"的模式划清界限

适配器、装饰者、外观、代理都是"包一层"，极易混淆。看意图区分：

模式意图接口是否变化
适配器转换接口，让不兼容的能协作✅ 变（A → B）
装饰者不改接口，动态增强功能❌ 不变（A → A，功能更强）
外观（Facade）为一组复杂子系统提供简化入口✅ 变（多 → 一，简化）
代理接口不变，控制访问（权限/延迟/远程）❌ 不变（A → A，加管控）
易错点：别把"接口变没变、功能变没变"记混。适配器改接口不改功能；装饰者改功能不改接口——这一句是区分二者的黄金标准。而外观是"多个接口 → 一个简化接口"，重点在简化而非转换。

九、双向适配器与常见误区

如果适配器同时实现 Target 和 Adaptee 两个接口，让两边都能把对方当成自己人使用，就是双向适配器（如 Duck 能当 Turkey 用，Turkey 也能当 Duck 用）。用得少，了解即可。

三个初学者常踩的坑：
1. 在适配器里塞新业务逻辑——适配器只做"翻译"，加功能是装饰者的活。
2. 误以为适配器一定 1:1 映射方法——如上面 fly() 循环 5 次，转换可以是多对一、带逻辑的。
3. 过度设计——只有当接口确实不兼容且双方都不能改时才用；能直接改接口就别硬加一层。
收束：适配器模式 = 组合被适配对象 + 实现目标接口 + 在方法内做翻译委托。它的价值在于不改动任何一方的前提下让两个不兼容的接口协作，是"开闭原则"和"面向接口编程"的直接落地。记住那个电源转换插头，你就抓住了它的全部。
