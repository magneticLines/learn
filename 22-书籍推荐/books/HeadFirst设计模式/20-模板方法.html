模板方法：算法骨架，子类实现细节

父类把流程的骨架焊死，只在几个位置留出空槽让子类去填——流程不变，细节可换。

一、正式定义：把"流程"和"步骤"拆开

先看一个每个人都熟悉的场景：泡咖啡和泡茶。它俩的流程几乎一模一样——烧水 → 冲泡 → 倒进杯子 → 加料，只有"冲泡"和"加料"两步不同（咖啡用热水冲咖啡粉、加糖和奶；茶用热水泡茶叶、加柠檬）。如果分别写两个类，"烧水""倒杯子"这些相同代码就会被复制两份——这就是重复。模板方法要解决的正是这类"流程相同、个别步骤不同"的重复。

模板方法模式（Template Method Pattern）
在一个方法中定义一个算法的骨架，而将某些步骤延迟到子类中实现。模板方法使得子类可以在不改变算法结构的前提下，重新定义算法中的某些步骤。
Define the skeleton of an algorithm in a method, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.

拆开这个定义，只有三个关键词：

  • 算法骨架（skeleton）——把"步骤的先后顺序"固定在父类的一个方法里，这个方法就叫模板方法。它是"总指挥"，规定了先做什么后做什么。
  • 延迟到子类（defer to subclasses）——骨架里那些"每个子类做法都不同"的步骤，父类只声明为抽象方法，具体怎么做交给子类填。
  • 不改变结构（without changing structure）——子类只能填空，不能改流程的顺序。骨架牢牢握在父类手里，这正是它区别于"复制粘贴改几行"的关键。

二、结构：一个模板方法 + 三类步骤

模板方法的类图比策略模式还简单——它甚至只需要继承，一个接口都不用：

┌──────────────────────────────────────┐ │ abstract CaffeineBeverage 抽象父类 │ │────────────────────────────────────── │ │ final prepareRecipe() ← 模板方法(骨架) │ │ ├─ boilWater() ← 具体步骤(共用) │ │ ├─ brew() ← 抽象步骤(必填) │ │ ├─ pourInCup() ← 具体步骤(共用) │ │ └─ if wantsCondiments() ← 钩子(可选) │ │ addCondiments() ← 抽象步骤 │ └────────────────────┬─────────────────────┘ │ 继承 (IS-A) + 覆盖空槽 ┌──────────────┴──────────────┐ ▼ ▼ ┌──────────┐ ┌──────────┐ │ Coffee │ │ Tea │ │ brew() │ │ brew() │ │ addCond()│ │ addCond()│ └──────────┘ └──────────┘ 调用: new Coffee().prepareRecipe(); ← 骨架在父类跑，遇到空槽回调子类

骨架方法里出现的步骤，按"谁负责实现"分成三类，这是理解模板方法的核心：

步骤类型由谁实现作用
模板方法 prepareRecipe()父类，且通常 final定义并锁死步骤顺序，是唯一对外的入口
具体方法 boilWater()父类实现，子类共用各子类完全相同的步骤，避免重复
抽象方法 brew()父类只声明，子类必须实现各子类做法不同、且缺一不可的步骤
钩子方法 customerWantsCondiments()父类给默认实现，子类可选覆盖让子类"插一脚"控制流程、或对某步骤做可选扩展

三、代码实战：从重复到模板

先看"坏味道"——咖啡和茶各写一个类，流程复制两份，`boilWater`/`pourInCup` 一字不差地重复：

Bad · 骨架被复制两份
class Coffee {
  void prepare() {
    boilWater();          // 重复①
    brewCoffeeGrinds();
    pourInCup();          // 重复②
    addSugarAndMilk();
  }
  void boilWater(){...}   // 复制
  void pourInCup(){...}   // 复制
}
class Tea {
  void prepare() {
    boilWater();          // 又抄一遍①
    steepTeaBag();
    pourInCup();          // 又抄一遍②
    addLemon();
  }
  void boilWater(){...}   // 又复制
  void pourInCup(){...}   // 又复制
}
// 痛点:
// · 流程改一步(如加"预热杯子")要改两处
// · 谁也没规定 prepare 的步骤顺序,
//   新人写第 3 个饮料可能顺序都写错
Good · 骨架抽到父类
abstract class Beverage {
  // 模板方法:骨架只此一份,顺序被锁死
  final void prepare() {
    boilWater();
    brew();               // 空槽
    pourInCup();
    addCondiments();      // 空槽
  }
  void boilWater(){...}   // 共用,只此一份
  void pourInCup(){...}   // 共用,只此一份
  abstract void brew();
  abstract void addCondiments();
}
class Coffee extends Beverage {
  void brew(){ /*冲咖啡粉*/ }
  void addCondiments(){ /*加糖和奶*/ }
}
class Tea extends Beverage {
  void brew(){ /*泡茶叶*/ }
  void addCondiments(){ /*加柠檬*/ }
}
// 收益: 流程唯一 + 顺序不可篡改 + 子类只管填空

再看带钩子的完整版——钩子让子类可以"投票"决定某个可选步骤要不要执行：

钩子方法：给子类一个可选的"插脚"Java
abstract class CaffeineBeverage {

    // 模板方法：final 是关键——子类能填空，但绝不能改流程
    final void prepareRecipe() {
        boilWater();
        brew();
        pourInCup();
        if (customerWantsCondiments()) {   // 钩子控制是否走这一步
            addCondiments();
        }
    }

    void boilWater() { System.out.println("把水烧开"); }
    void pourInCup() { System.out.println("倒进杯子"); }

    abstract void brew();            // 抽象步骤：子类必须实现
    abstract void addCondiments();

    // 钩子：父类给默认实现，子类想干预流程就覆盖它
    boolean customerWantsCondiments() { return true; }
}

class Coffee extends CaffeineBeverage {
    void brew() { System.out.println("用热水冲咖啡粉"); }
    void addCondiments() { System.out.println("加糖和奶"); }

    @Override                              // 覆盖钩子：这位客人不要加料
    boolean customerWantsCondiments() { return false; }
}
抽象方法 vs 钩子方法，一句话记牢： 抽象方法是"必答题"——子类不实现就编译不过，用于流程里"每个子类都得给出自己做法"的必需步骤；钩子是"选答题"——父类已给默认值，子类觉得需要才覆盖，用于"可选步骤"或"让子类对流程投票"。设计时的取舍：能用钩子就别用抽象方法，抽象方法越多，子类被强制填的空越多，越难写。

四、灵魂：好莱坞原则（控制反转）

模板方法背后藏着一条重要设计原则，Head First 起了个好记的名字——好莱坞原则（Hollywood Principle）："别调用我们，我们会调用你（Don't call us, we'll call you）"。这句话精确描述了模板方法的调用方向：

普通继承：子类 → 调用 → 父类工具方法
模板方法：父类骨架 → 回调 → 子类步骤

平常我们写继承，是子类主动去 super.xxx() 调父类。模板方法把这个方向反转了：高层的父类掌控流程，在需要的时刻"回头"调用低层子类填的空。子类不主动发起流程，只是被动地"被点名"。这就是控制反转（IoC）——和 Spring 容器"你别自己 new 依赖，我来把依赖递给你"是同一个思想内核。

为什么这么设计？ 因为"流程顺序"往往是最不该被随意改动、最需要统一的东西（想想事务的"开始→执行→提交/回滚"、Servlet 的"接收→分发→响应"）。把它焊死在父类，就杜绝了子类把顺序写错、漏掉某一步的可能。子类被限制得越死，误用的空间越小——好的框架都在用这招"用约束换安全"。

五、无处不在：JDK 与 Spring 里的模板方法

一旦认出它的形状（"一个方法定义好流程，中间几步调用了 abstract/protected 的方法交给子类"），你会发现它是框架设计的头号常客：

出处模板方法（骨架）留给子类填的步骤
java.util.AbstractList迭代、indexOf、equals 等get(i)、size()——只填这两个就能用
java.io.InputStreamread(byte[], off, len) 循环调用抽象的 read()——读一个字节
HttpServletservice() 按方法分发doGet() / doPost()
AbstractQueuedSynchronizeracquire() 排队/阻塞/唤醒的骨架tryAcquire()——是否抢锁成功
Spring JdbcTemplate拿连接→建 Statement→执行→处理结果→关资源RowMapper / ResultSetExtractor
Spring AbstractApplicationContextrefresh()——容器启动的 12 步骨架onRefresh() 等钩子
重点看 AQS——并发包的地基。 ReentrantLock、Semaphore、CountDownLatch 全都靠它。AbstractQueuedSynchronizer.acquire() 是模板方法，它把"尝试获取失败就入队、park 挂起、被唤醒后重试"这套复杂又易错的骨架写死；子类（各种锁的 Sync）只需覆盖 tryAcquire(int) 回答一句"现在能不能拿到锁"。把最难写、最该统一的排队/阻塞逻辑收归父类，把最易变的"抢锁规则"下放给子类——这就是模板方法在 JDK 里最教科书的应用。
你其实早就在用它了。 每次继承 HttpServlet 只重写 doGet/doPost、继承 Spring 的 OncePerRequestFilter 只实现 doFilterInternal、继承 TestCase 写 setUp/tearDown——你都是在"往框架预留的空槽里填代码"，框架的骨架方法会在恰当时机回调你。这正是好莱坞原则："别来调框架，框架会调你。"

六、对比：模板方法 vs 策略模式

这是最高频的面试对比。两者都在解决"算法可变"，但一个走继承，一个走组合，方向正好相反：

模板方法 = 继承(IS-A，编译期填空槽)  vs  策略 = 组合(HAS-A，运行时换整体)
维度模板方法 Template Method策略模式 Strategy
复用手段继承——子类复用父类骨架组合——Context 持有策略对象
可变粒度只换流程里的某几步换掉整个算法
绑定时机编译期（继承关系一旦确定就固定）运行时（可随时 setStrategy 热插拔）
控制方向父类控流程，回调子类（好莱坞原则）客户端控制，主动选并注入策略
灵活性低，一个子类绑死一套步骤组合高，可自由组合/替换
骨架复用强，公共流程只写一份弱，Context 不负责固定流程
选型口诀： "只想改流程里的某几步、且流程本身要被统一固定"→ 模板方法；"想在运行时换掉一整套做法、要灵活热插拔"→ 策略。它们正是"多用组合少用继承"原则在两个模式上的分野：模板方法是继承派的代表，策略是组合派的代表。

七、常见陷阱与高频面试点

陷阱一：模板方法忘了加 final。 模板方法的价值就是"骨架不可篡改"。如果不加 final，子类完全可以覆盖 prepareRecipe() 把步骤顺序改乱，甚至漏掉关键步骤（比如漏掉"关资源"）——你精心设计的约束瞬间失效。骨架方法几乎总该是 final；反过来，被回调的抽象/钩子方法则不能是 final，否则子类没法填空。
陷阱二：继承带来的强耦合与"脆弱基类"。 模板方法靠继承，子类和父类死死绑在一起：父类骨架一改，所有子类都受影响；而且 Java 单继承，子类的"继承名额"被这个抽象父类占掉了，没法再继承别的类。步骤组合一多还会引发类爆炸（3 种冲泡 × 3 种加料 = 9 个子类）。这种情况就该改用组合——把可变步骤抽成策略对象注入，而不是靠开子类。
陷阱三：抽象方法开太多，子类沦为"苦力"。 骨架里塞了七八个 abstract 方法，等于逼着每个子类填一堆空，既啰嗦又容易漏。原则：骨架里尽量多用具体方法和钩子，抽象方法只留给"真正因子类而异且必需"的步骤。步骤越少、粒度越大，子类越好写。
陷阱四：在钩子/回调方法里调用未初始化的字段（父类构造期回调）。 如果父类构造函数里调用了模板方法，而模板方法又回调了子类覆盖的方法，此时子类自己的字段还没初始化（子类构造体尚未执行），会读到默认值 null/0。这是"构造期调用可覆盖方法"的经典坑——别在构造函数里调模板方法。

面试延伸：Java 8 之后模板方法可以用 default 方法实现吗？ 可以且更轻。接口的 default 方法能写骨架，调用接口里其他抽象方法，实现类填空——这样就绕开了"单继承名额被占"的限制。但要注意接口没有实例字段、default 方法里也不该依赖状态；需要共享状态的骨架仍然更适合抽象类。这也是"抽象类 vs 接口"取舍的一个现实切面。

八、什么时候该用它

  • 多个类有几乎相同的流程，只有个别步骤实现不同（导入/导出、编解码、生命周期回调、各种"处理流水线"）。
  • 你想把流程顺序牢牢固定下来，不允许使用者改动或写错（框架/基础库的典型诉求）。
  • 想消除子类间的重复代码，把公共部分上提到父类，只把差异下沉到子类。
  • 想给使用者预留可选的扩展点（钩子），让他们在不碰主流程的前提下微调行为。
一句话收束： 模板方法 = 父类定骨架、子类填细节，用一个 final 方法把"流程"和"步骤"焊接又分离。它是"继承式复用"最优雅的形态，灵魂是好莱坞原则——控制权在父类，子类被回调。认出它的信号永远是同一个："这几段代码流程一模一样，就中间几步不同"。当你又要复制粘贴一整套流程只改几行时，停下来问：这是不是又一个模板方法？
