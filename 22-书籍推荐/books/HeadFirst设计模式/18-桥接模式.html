桥接模式：抽象与实现解耦

当一个类沿着两个维度同时变化，别让它们绑死在继承树上——搭一座桥，让维度各自独立生长。

一、从一个"类爆炸"的痛点讲起

假设你要做一个跨平台的遥控器（Remote），要控制不同品牌的电视（Sony、TCL）。最直觉的做法是用继承：一个平台/品牌组合一个类。

需求刚开始只有 SonyRemote、TCLRemote，很舒服。可当"遥控器种类"也开始变化——除了基础遥控器，还要高级遥控器（带静音记忆）、触屏遥控器——你会发现类的数量开始失控。

继承方案：两个维度纠缠在一张继承树上 Remote │ ┌─────────────┼─────────────┐ BasicRemote AdvancedRemote TouchRemote ← 遥控器种类维度 │ │ │ ┌────┴────┐ ┌────┴────┐ ┌────┴────┐ Sony TCL Sony TCL Sony TCL ← 品牌维度 种类(m) × 品牌(n) = m × n 个类 ← 组合爆炸
陷阱：把"多个变化维度"塞进一棵继承树。 继承是"编译期"绑定的强关系——一旦 SonyRemote 继承了 BasicRemote，它就同时被"Sony"和"Basic"两个维度锁死。新增一个品牌，要为所有种类各写一个子类；新增一种遥控器，又要为所有品牌各写一个。类数量按 乘法增长，这就是典型的类爆炸（Class Explosion）。

二、桥接模式的核心思想：用组合替代继承

桥接模式（Bridge）的定义是：将抽象部分与它的实现部分分离，使它们都可以独立地变化。 这句话很拗口，我们把它翻译成人话：

当一个事物有两个（或多个）独立变化的维度时，不要用继承把它们纠缠在一起，而是把其中一个维度抽出来做成独立的接口层，然后用一个字段（引用）把两层"桥接"起来。乘法关系 m × n 就变成了加法关系 m + n。

类数量：m × n  →  m + n

这里的两个关键词，正是模式名字里两个易混的概念：

  • 抽象（Abstraction）：面向使用者的高层控制逻辑。注意，这里的"抽象"不是指 Java 的 abstract 关键字，而是指"遥控器"这一层——它定义"要做什么"（开机、调台、静音）。
  • 实现（Implementor）：底层的具体动作接口。这里指"电视"这一层——它定义"具体怎么做"（Sony 怎么开机、TCL 怎么调台）。
"桥"在哪里？ 就是抽象类里持有实现接口的那个字段：protected Device device;。这个引用把两棵独立的继承树连起来，运行时才决定"哪个遥控器 + 哪台电视"的组合。所以桥接是运行期组合，而继承是编译期绑定——这正是它灵活的根源。
桥接方案：两个维度拆成两棵独立的树，用一条"桥"相连 抽象层 (Abstraction) 实现层 (Implementor) ┌────────────────┐ bridge ┌────────────────┐ │ RemoteControl │ ──── device ───────▶ │ Device │ <<interface>> └────────────────┘ └────────────────┘ △ △ ┌─────┴──────┐ ┌──────┴──────┐ BasicRemote AdvancedRemote SonyTV TclTV (m 种遥控器，独立增长) (n 个品牌，独立增长) 任意 RemoteControl 可搭配 任意 Device → m + n 个类

三、代码实现：遥控器与电视

1. 实现层：Device 接口 + 具体电视

Device.java — 实现层接口Java
// 实现层：只关心"底层动作怎么做"，完全不知道遥控器长什么样
public interface Device {
    boolean isEnabled();
    void enable();
    void disable();
    int getVolume();
    void setVolume(int percent);
    int getChannel();
    void setChannel(int channel);
}

public class SonyTV implements Device {
    private boolean on = false;
    private int volume = 30, channel = 1;

    public void enable()  { on = true;  System.out.println("[Sony] 开机"); }
    public void disable() { on = false; System.out.println("[Sony] 关机"); }
    public void setVolume(int v) { volume = Math.max(0, Math.min(100, v)); }
    // ... 其余 getter/setter 略
}

2. 抽象层：RemoteControl 持有 Device 引用（这就是"桥"）

RemoteControl.java — 抽象层Java
// 抽象层：面向用户的高层逻辑，通过 device 字段"桥接"到实现层
public class RemoteControl {
    protected Device device;   // ← 桥！运行期注入具体实现，而非编译期继承

    public RemoteControl(Device device) { this.device = device; }

    public void togglePower() {
        if (device.isEnabled()) device.disable();
        else device.enable();
    }
    public void volumeUp()   { device.setVolume(device.getVolume() + 10); }
    public void channelUp()  { device.setChannel(device.getChannel() + 1); }
}

// 精化抽象（Refined Abstraction）：扩展遥控器功能，与品牌完全无关
public class AdvancedRemote extends RemoteControl {
    public AdvancedRemote(Device device) { super(device); }

    public void mute() {
        device.setVolume(0);
        System.out.println("静音（高级遥控器专属）");
    }
}

3. 客户端：任意组合，自由搭配

Client.javaJava
Device sony = new SonyTV();
RemoteControl basic = new RemoteControl(sony);   // 基础遥控器 + Sony
basic.togglePower();

AdvancedRemote adv = new AdvancedRemote(new TclTV()); // 高级遥控器 + TCL
adv.mute();

// 想加"华为电视"？只写一个 HuaweiTV，两种遥控器立刻都能用它。
// 想加"触屏遥控器"？只写一个 TouchRemote，所有品牌立刻都能被它控制。
识别桥接的信号词。 当需求描述里出现"任意 A 都能搭配任意 B""A 和 B 各自会独立增加"时，几乎就是桥接的用武之地：图形（形状 × 渲染引擎）、支付（业务场景 × 支付渠道）、消息（消息类型 × 发送通道 短信/邮件/推送）、日志（日志级别 × 输出目的地）。

四、反例 → 正例：从继承纠缠到组合解耦

Bad · 继承纠缠
// 两个维度锁死在一棵树上
class SonyBasicRemote { ... }
class SonyAdvancedRemote { ... }
class TclBasicRemote { ... }
class TclAdvancedRemote { ... }

// 加个"触屏遥控器"？
// 必须新增：
//   SonyTouchRemote
//   TclTouchRemote
// 加个"华为电视"？
// 又要新增：
//   HuaweiBasicRemote
//   HuaweiAdvancedRemote ...
// → m×n，改一处牵动全身
Good · 桥接组合
// 两个维度各自独立
interface Device { ... }        // 实现层
  class SonyTV implements Device
  class TclTV  implements Device

class RemoteControl {           // 抽象层
    Device device;              // 桥
}
class AdvancedRemote extends RemoteControl

// 加"触屏遥控器"：只写 TouchRemote
// 加"华为电视"：只写 HuaweiTV
// 组合在运行期决定：
new AdvancedRemote(new HuaweiTV());
// → m+n，各自演化互不影响

五、与相近模式的对比

桥接的结构（一个对象持有另一个对象的引用）和适配器、策略、装饰器长得很像，面试常拿来辨析。区别不在"长相"，而在意图：

模式意图（解决什么）关键区别
桥接让两个维度预先解耦，各自独立扩展设计之初就规划的两层结构，两边都会变化
适配器让已有的、不兼容的接口能协作事后补救，接口已定死无法改
策略封装一族可互换的算法只关注"行为/算法"一个维度的替换
装饰器动态给对象叠加职责装饰者与被装饰者是同一接口，可层层包裹
高频面试坑：桥接 vs 策略。 两者代码骨架几乎一样（都持有一个接口引用）。差别是意图与稳定性：策略针对"某个行为"提供多种可替换实现，抽象层通常是稳定的、单一的；桥接则是抽象和实现两边都是一棵会生长的继承树，是为了应对"双维度爆炸"而在架构层面做的分离。一句话：策略换的是"算法"，桥接分的是"两个变化轴"。

六、串联：你早就用过桥接了

桥接不是屠龙之术，JDK 和框架里到处是它的影子：

场景抽象层实现层（桥接到）
JDBCjava.sql.Driver/Connection（规范 API）各厂商 Driver 实现（MySQL、PG、Oracle）
SLF4J 日志门面Logger（统一门面 API）Logback / Log4j2 具体实现
AWT 图形Component（组件抽象）ComponentPeer（各 OS 的对等实现）
Spring ResourceResource 接口ClassPathResource / UrlResource / FileSystemResource
JDBC 是教科书级桥接。 DriverManager 面向 java.sql 的抽象 API（Connection、Statement），而 MySQL、PostgreSQL 各自提供 Driver 实现。你的业务代码（抽象层的使用者）永远只写标准 SQL API，换数据库只需换一个 jar 包和 URL——抽象与实现独立变化，正是桥接的精神。这也解释了为什么面向 java.sql.* 接口编程、而非面向具体驱动类，是 JDBC 的铁律。

七、什么时候不该用

  • 只有一个变化维度：那用策略或简单继承就够了，硬拆两层只是徒增抽象，违背 KISS。
  • 维度虽多但组合很少且不会增长：比如永远只有 2×2 且业务已冻结，直接写 4 个类反而更直白。
  • 过早抽象：桥接的成本是多一层间接、多一批接口。在维度是否真的独立变化尚不明朗时就上桥接，是典型的过度设计（over-engineering）。
一句话收束： 桥接模式的本质是"优先使用组合而非继承"这条设计原则在"双维度变化"场景下的具体落地——把 m × n 的继承爆炸拆成 m + n 的两棵独立树，用一个引用字段（桥）在运行期把它们组合起来。判断要不要用，只问一句：这里是不是有两个会各自独立生长的变化维度？ 是，就搭桥；不是，就别过度设计。
