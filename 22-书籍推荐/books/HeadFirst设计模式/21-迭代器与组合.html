迭代器与组合：统一遍历方式

把"怎么遍历"从集合里剥离出来，再让树形结构和单个元素长得一模一样。

一、从一个真实痛点说起

Head First 里的经典例子是两家餐厅合并菜单：煎饼屋用 ArrayList 存菜品，餐厅用定长数组存。现在要做一个女招待 Waitress，能打印所有菜单项。你会发现——只要底层集合类型不同，遍历代码就得写两遍：

反例 · 遍历方式泄漏
List<Item> a = pancake.getItems();
for (int i=0; i<a.size(); i++) {...}

Item[] b = diner.getItems();
for (int i=0; i<b.length; i++) {...}

// 再来一家用 HashMap 的？
// Waitress 又得改一次！
正例 · 统一接口
Iterator it1 = pancake.iterator();
Iterator it2 = diner.iterator();

print(it1);
print(it2);   // 同一个方法搞定

// 新菜单只要能给出 iterator()
// Waitress 一行都不用动
坏味道：遍历逻辑到处复制。 女招待被迫知道"煎饼屋用 List、餐厅用数组"这些实现细节。这违反了最少知识原则，也让 Waitress 和具体集合紧耦合——集合一换，客户端全崩。

二、迭代器模式：把遍历封装成对象

迭代器模式提供一种方法顺序访问一个聚合对象中的各元素，而不暴露其内部表示。核心就是把"游标"这个概念对象化：集合负责"存"，迭代器负责"走"。

<<interface>> <<interface>> Aggregate Iterator + createIterator() ──创建──> + hasNext(): boolean △ + next(): E │ 实现 △ │ │ 实现 PancakeMenu PancakeMenuIterator (内部是 ArrayList) (持有 List + 游标 position) DinerMenu DinerMenuIterator (内部是 Item[]) (持有数组 + 游标 position)
为什么迭代器要独立成对象，而不是给集合加一堆 getFirst/getNext 方法？ 因为遍历状态（当前走到哪个 position）不属于集合，属于"这一次遍历"。独立成对象后，同一个集合可以同时开多个迭代器各走各的，互不干扰——这就是"单一职责"：集合管存储，迭代器管遍历。

手写一个迭代器（餐厅数组版）

DinerMenuIterator.javaJava
public class DinerMenuIterator implements Iterator<MenuItem> {
    private MenuItem[] items;
    private int position = 0;   // 遍历状态封装在迭代器内部

    public DinerMenuIterator(MenuItem[] items) {
        this.items = items;
    }

    public boolean hasNext() {
        // 数组是定长的，可能有 null 空位，要跳过
        return position < items.length && items[position] != null;
    }

    public MenuItem next() {
        return items[position++];   // 取当前，游标后移
    }
}

客户端 Waitress 从此只依赖 Iterator 接口，完全不知道背后是数组还是 List：

Waitress.javaJava
private void printMenu(Iterator<MenuItem> it) {
    while (it.hasNext()) {
        MenuItem item = it.next();
        System.out.println(item.getName() + " $" + item.getPrice());
    }
}

三、JDK 里的迭代器：你天天在用

Java 集合框架就是迭代器模式的教科书级实现。你写的每一个增强 for 循环，编译器都会翻译成 Iterator 调用：

你写的
for (String s : list) {
    use(s);
}
编译器展开后
Iterator<String> it =
    list.iterator();
while (it.hasNext()) {
    String s = it.next();
    use(s);
}

关键设计：Iterable 是"能给出迭代器"的能力接口，Iterator 才是遍历器本身。二者对应 Head First 里的 Aggregate 与 Iterator。

角色Head First 术语JDK 对应关键方法
聚合接口AggregateIterable<T>iterator()
迭代器接口IteratorIterator<T>hasNext() / next() / remove()
具体聚合DinerMenuArrayList / HashMap各自内部类实现
具体迭代器DinerMenuIteratorArrayList$Itr持有 cursor + expectedModCount
高频面试点：ConcurrentModificationException（快速失败 fail-fast）。 遍历 ArrayList 时若直接调 list.remove(x)，会抛此异常。原因：ArrayList$Itr 里存了一个 expectedModCount，每次 next() 都会检查它是否等于集合的 modCount；你绕过迭代器改结构，两者对不上就抛异常。正确做法是调 迭代器自己的 it.remove()，它会同步更新 expectedModCount。这正是"遍历状态应由迭代器统一管理"的血泪教训。
安全删除元素Java
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().isEmpty()) {
        it.remove();   // ✅ 用迭代器删，不会 fail-fast
        // list.remove(...) ❌ 会抛 ConcurrentModificationException
    }
}

四、当菜单里还有"子菜单"：组合模式登场

迭代器解决了"扁平集合怎么遍历"，但现实更麻烦：餐厅菜单里可能嵌一个"甜点子菜单"，子菜单里又可能有别的菜单……这是一棵树。如果还想用一个迭代器走完整棵树，就需要组合模式（Composite）先把结构统一了。

组合模式的核心思想：让"单个对象"和"对象组合"实现同一个接口，客户端可以一致地对待它们——你不需要判断"这是一道菜还是一个子菜单"。

<<abstract>> MenuComponent + print() + add(MenuComponent) ← 组合才有意义 + getChild(int) △ ┌───────────┴────────────┐ │ │ MenuItem (叶子) Menu (组合节点) 一道菜，没有孩子 持有 List<MenuComponent> print() 打印自己 print() 递归打印所有孩子
为什么叶子和组合共享同一个接口，哪怕 MenuItem 根本没有孩子？ 这是组合模式最反直觉、也最关键的取舍：为了让客户端透明地一致处理，把 add()/getChild() 提到公共接口。代价是叶子被迫拥有"无意义"的方法（默认抛 UnsupportedOperationException）。这是拿"安全性"换"透明性"——GoF 明确指出这是一个刻意的权衡。
Menu.java（组合节点）Java
public class Menu extends MenuComponent {
    private List<MenuComponent> children = new ArrayList<>();
    private String name;

    public void add(MenuComponent c) { children.add(c); }

    // 递归：打印自己，再让每个孩子打印自己——叶子和组合同样对待
    public void print() {
        System.out.println("== " + name + " ==");
        for (MenuComponent c : children) {
            c.print();   // 无需 instanceof 判断！递归自然展开整棵树
        }
    }
}

五、两个模式如何"统一遍历方式"

把二者合起来看，就是本节标题的含义。递归遍历有两种写法，Head First 特意演示了用组合内部迭代器组合出一个能穿透整棵树的外部迭代器（CompositeIterator）：

维度内部迭代（递归 print）外部迭代（CompositeIterator）
控制权在集合自己手里（它决定怎么走）在客户端手里（客户端调 next()）
暂停/中途退出难，得靠标志位或异常容易，随时停下
实现复杂度简单，一个递归方法搞定较复杂，要用栈保存遍历路径
典型场景全量打印、全量计算需要逐个消费、条件过滤（如只找素食）
组合模式 统一结构  +  迭代器模式 统一遍历 = 客户端只面对一个 Iterator

六、生态里的组合 + 迭代器

  • 文件系统：目录（组合）和文件（叶子）都是"节点"，find /path 递归遍历就是外部迭代——操作系统层面天然的组合结构。
  • DOM / 前端虚拟 DOM：一个 Node 既可能是元素（有子节点）也可能是文本节点，childNodes 遍历就是组合 + 迭代。
  • Spring：CompositeCacheManager、Servlet 的 CompositeFilter 都是组合模式；Iterable 让 @Autowired List<Bar> 能被 for-each 统一消费。
  • AST 编译器：抽象语法树的表达式节点（组合）与字面量节点（叶子），访问者遍历前先靠组合统一结构。
识别信号：什么时候该想到这两个模式？ 看到"部分-整体""树形层级""菜单里套菜单""想对单个和一组做一样的操作"——用组合；看到"我想遍历它但不想知道它内部是数组还是链表还是哈希表"——用迭代器。二者常常结伴出现。
别硬套组合模式。 只有当结构确实是递归的树、且客户端确实需要"一致对待单体与组合"时才值得。若层级固定只有两层、或叶子和组合的操作差异巨大，强行统一接口反而制造出一堆抛异常的空方法，得不偿失。这也是"透明 vs 安全"权衡的反面——过度追求透明会牺牲类型安全。
一句话收束： 迭代器把"遍历"从集合中抽出来变成对象，让客户端用统一的 hasNext()/next() 面对任何集合、屏蔽内部结构；组合把"部分-整体"的树抹平成同一个接口，让单个元素和一组元素被一致对待。合起来，客户端最终只需要面对一个迭代器，就能遍历任意复杂的层级结构——这就是"统一遍历方式"的全部含义。
