策略模式：定义算法族，分别封装

把一族"做同一件事的不同做法"各自装箱、贴上统一接口的标签，让调用者可以像换插头一样在运行时替换算法。

一、正式定义：给"封装变化"一个名字

第 1 章我们用鸭子例子推导出了封装变化 + 组合委托，其实那时候你已经在不知不觉中写出了策略模式。现在给它一个正式定义（GoF 原话）：

策略模式（Strategy Pattern）
定义一族算法（算法族），把每个算法分别封装起来，并让它们之间可以互相替换。策略模式让算法的变化独立于使用算法的客户。
Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.

拆开这三个关键词，正好对应三个动作：

  • 算法族（family of algorithms）——一组"目标相同、做法不同"的行为。比如"计算运费"：普通/顺丰/自提是三种算法；"排序"：升序/降序/按销量是三种算法。
  • 分别封装（encapsulate each）——每种做法各自关进一个类，实现同一个接口。这就是第 1 章的 FlyWithWings / FlyNoWay。
  • 互相替换（interchangeable）——因为它们长着同一张接口脸，调用者可以在运行时拿一个换掉另一个，而自己一行都不用改。
为什么叫"策略"？ 想象你在下棋，"开局策略""残局策略"是可以随时切换的整套打法；又或者电商结账时的"满减策略""打折策略""无优惠策略"。它们都是解决同一个问题的可互换方案。策略模式就是把"方案"提升为一等公民——变成一个可以被赋值、被传递、被替换的对象。

二、结构：三个角色 + 一根委托箭头

策略模式的类图极其简单，只有三个角色。记住这张图，90% 的策略模式变体都是它的皮肤：

┌─────────────────────┐ <<interface>> │ Context 上下文 │────────► Strategy 策略接口 │─────────────────────│ 持有 ──────────────── │ - strategy: Strategy │ (HAS-A) │ + execute() │ │─────────────────────│ └───────┬────────┘ │ + setStrategy(s) │ 实现 │ △ │ + doSomething() │ ┌──────────────┼──────────────┐ └─────────────────────┘ │ │ │ │ 委托 ▼ ▼ ▼ └──────────► ConcreteStrategyA ...B ...C (算法1) (算法2) (算法3) 客户端: context.setStrategy(new ConcreteStrategyB()); ← 运行时选择/替换 context.doSomething(); → 内部委托给 strategy.execute()
角色职责鸭子例子里的对应
Strategy（策略接口）声明算法族的统一"长相"，一个方法即可FlyBehavior / QuackBehavior
ConcreteStrategy（具体策略）各自实现一种算法，彼此可替换FlyWithWings、FlyNoWay...
Context（上下文）持有一个策略引用，把请求委托给它；对外屏蔽算法细节Duck（performFly() 委托给 flyBehavior）
它就是"封装变化 + 针对接口编程 + 多用组合少用继承"三条原则的合体。 Context 通过组合（HAS-A）而非继承持有策略；它只依赖 Strategy 接口而不认识任何具体实现；变化的算法被封装在接口后端。前几节学的原则，到这里凝结成了一个可复用的套路——这正是"模式是原则的具体化"的最佳注脚。

三、一个真实场景：电商运费计算

抛开鸭子，看个业务里天天遇到的例子：结算时按不同物流方式算运费。先看几乎每个新人都写过的"坏味道"版本——一长串 if-else：

Bad · if-else 分支洪流
double calcFee(String type, double w) {
  if (type.equals("STANDARD")) {
    return w * 5;
  } else if (type.equals("SF")) {
    return w * 12 + 8;   // 顺丰含首重
  } else if (type.equals("PICKUP")) {
    return 0;             // 自提免运费
  } else if (type.equals("EMS")) {
    return w * 8;         // 新增：又要动这里
  }
  throw new IllegalArgumentException(type);
}
// 痛点:
// · 每加一种物流 = 改这个方法(违反开闭)
// · 算法全揉在一起,无法单独测试/复用
// · type 是字符串,拼错了编译期发现不了
Good · 策略模式
interface ShippingStrategy {       // 算法族接口
  double calc(double weight);
}
class Standard implements ShippingStrategy {
  public double calc(double w){ return w*5; }
}
class SF implements ShippingStrategy {
  public double calc(double w){ return w*12+8; }
}
class Pickup implements ShippingStrategy {
  public double calc(double w){ return 0; }
}
// 新增 EMS: 只 new 一个类, 老代码一行不改
// 收益:
// · 对扩展开放,对修改关闭
// · 每个算法可独立单测/复用
// · 用枚举或Map选策略,类型安全

Context 只负责持有策略并委托，完全不关心运费到底怎么算：

Context + 用 Map 干掉 switch，实现真正的开闭Java
public class Order {                               // Context 上下文
    private ShippingStrategy strategy;          // 组合：持有算法，而非继承它

    public void setStrategy(ShippingStrategy s) { this.strategy = s; }

    public double checkout(double weight) {
        return strategy.calc(weight);          // 委托：把"怎么算"甩给策略对象
    }
}

// 用一张注册表把 "选哪个策略" 也变成开闭的：新增策略只往 Map 里 put
Map<String, ShippingStrategy> registry = Map.of(
    "STANDARD", new Standard(),
    "SF",       new SF(),
    "PICKUP",   new Pickup()
);

Order order = new Order();
order.setStrategy(registry.get("SF"));       // 运行时根据用户选择装配策略
System.out.println(order.checkout(2.0));    // 32.0
陷阱：策略模式并不会"消灭"分支，它只是把分支挪到了"选策略"这一步。 很多人以为用了策略就再也没有 if-else 了——错。总得有个地方决定"这次用哪个策略"。策略模式的价值在于把这个"选择"从"算法实现"里剥离出来，集中到一处（如上面的 registry 或工厂/Spring 容器），于是算法本身满足开闭。如果你在 calc() 内部还写 if-else 判断类型，那就白封装了。

四、别再手写策略类：Lambda 才是它的现代形态

策略接口往往只有一个方法——这正是函数式接口（@FunctionalInterface）。Java 8 之后，写一堆 class Standard implements... 显得很笨重，一个 Lambda 就是一个"匿名策略对象"：

Lambda = 策略模式的语法糖Java 8+
// 策略接口天然就是函数式接口
@FunctionalInterface
interface ShippingStrategy { double calc(double w); }

// 每个 Lambda 就是一个 ConcreteStrategy，连类名都省了
order.setStrategy(w -> w * 5);              // Standard
order.setStrategy(w -> w * 12 + 8);         // SF

// JDK 内置的函数式接口本身就是"通用策略接口"
Comparator<Order> byWeight = Comparator.comparingDouble(Order::getWeight);
list.sort(byWeight);   // sort 是 Context，Comparator 是可替换的比较策略
与 JDK / 内存模型串联： Lambda 底层不是匿名内部类，而是 invokedynamic 指令 + LambdaMetafactory 在运行时生成实现类，无状态的 Lambda 还会被缓存复用（不像匿名类每次 new 一个对象）。所以"用 Lambda 当策略"不仅代码短，通常还更省内存。这也解释了为什么 Comparator、Runnable、Function 这些 JDK 里最常见的"传行为进去"的接口，本质都是策略模式——Lambda 让策略从"重量级的类"退化成了"轻量级的函数值"。

五、无处不在：JDK 与 Spring 里的策略模式

一旦你认出策略模式的形状（"传一个接口进来，行为随之改变"），会发现它遍地都是：

场景Context（用算法的人）Strategy（被替换的算法）
集合排序Collections.sort(list, c) / list.sort(c)Comparator——怎么比大小
线程执行Thread / 线程池Runnable——线程跑什么逻辑
线程池拒绝ThreadPoolExecutorRejectedExecutionHandler——队列满了怎么办
Spring 事务TransactionTemplateTransactionCallback——事务里执行什么
Spring 资源解析ResourceLoader不同 Resource 加载策略（classpath/file/url）
Spring 类型转换ConversionService一堆 Converter——每种类型转换是一个策略
Spring 的经典用法：用容器帮你收集所有策略。 把策略接口的所有实现都交给 Spring 管理，然后注入 Map<String, ShippingStrategy> 或 List<ShippingStrategy>，Spring 会自动把全部实现类装进来（key 是 bean 名）。于是"选策略"和"注册策略"都不用你手写 new——这就是"策略模式 + 依赖注入"的黄金组合，也是企业代码里替代 if-else 的标准姿势。

六、常见陷阱与高频面试点

陷阱一：策略膨胀（Strategy Explosion）。 只有 2~3 个分支、而且几乎不会再变的逻辑，硬套策略模式会凭空多出一堆小类，反而更难读。判断标准：算法族是否会持续增长？是否需要运行时切换？是否要独立测试/复用？都否，就老实用 if-else 或 enum。别为了用模式而用模式（呼应第 1 章的 YAGNI）。
陷阱二：无状态策略应当共享，不要每次 new。 大多数策略是无状态的纯算法，完全可以做成单例（或用 enum 实现策略，天然单例且线程安全）。给每个订单都 new SF() 是浪费——这也是 Comparator.naturalOrder() 返回单例的原因。反过来，如果策略有可变状态，多线程共享一个实例就会踩到并发坑，必须小心。
陷阱三：Context 与 Strategy 的数据传递。 策略要算东西，就得拿到数据。两种传法：① 把需要的参数作为方法入参传给 execute(data)（推荐，策略保持无状态）；② 把整个 Context 传进去让策略回调取数据（耦合更强）。别图省事把 Context 的一堆字段设成 public 给策略乱读。

再对比一个近亲：模板方法。 两者都在处理"算法可变"，但手段相反：

策略 = 组合(HAS-A，运行时换整个算法)  vs  模板方法 = 继承(IS-A，编译期填算法空槽)
一句话记牢取舍： 模板方法把"算法骨架"定死在父类、让子类只覆盖几个钩子步骤，是继承路线，灵活性低但复用骨架强；策略模式把整个算法抽成对象，是组合路线，能运行时热插拔。当你需要"换掉整套做法"选策略，当你只想"改流程里的某几步"选模板方法。这正是"多用组合少用继承"原则在两个模式上的分野。

七、什么时候该用它

  • 一个功能有多种可切换的实现，且未来还会新增（运费/支付/优惠/排序/加密/序列化格式……）。
  • 代码里出现了基于类型的一大坨 if-else 或 switch，且每个分支都是一段独立算法。
  • 需要在运行时根据配置、用户选择、AB 实验动态换算法。
  • 想把算法独立出来单元测试，或在多处复用同一段算法。
一句话收束： 策略模式 = 把"怎么做"抽成一个可替换的对象。它是"封装变化"最直接、最纯粹的落地：变化的算法被关进一族实现类，稳定的 Context 只依赖接口并做委托。认出它的信号永远是同一个——"这里有一族做同一件事的不同做法，而且还会变"。当你下次又想写第 4 个 else if 时，停下来问一句：这是不是又一个策略？
