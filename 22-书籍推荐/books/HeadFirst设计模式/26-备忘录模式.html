备忘录模式：对象状态保存与恢复

给对象拍一张"快照"存起来，需要时原样还原——在不破坏封装的前提下实现撤销、回滚与存档。

一、从 Ctrl+Z 说起

你在编辑器里敲错了一段，按下 Ctrl+Z，文字回到了上一步；游戏打到一半存档，读档后从存档点复活;数据库事务出错 ROLLBACK，数据退回事务开始前。这三件事背后是同一个诉求：把某一时刻的对象状态"冻结"下来，将来能原样恢复。备忘录模式（Memento，又译"快照模式")就是把这件事标准化的方案。

它要解决的真正难题不是"怎么存"——一个 copy 谁都会写——而是"怎么存得干净"：既要拿到对象的内部状态，又不能把对象的私有字段暴露给外部，否则封装（Encapsulation）就被撕开了口子。备忘录模式的全部精妙，都在于这一点平衡。

GoF 定义：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后可以将对象恢复到原先保存的状态。关键词是"在对象之外保存"和"不破坏封装"——这正是它区别于"随便拷贝一份"的地方。

二、三个角色：谁拍照、照片是啥、谁保管相册

备忘录模式只有三个角色，用"拍照存档"这个类比一秒记住：

角色职责类比
Originator
（发起人）		业务对象本身。负责createMemento() 生成快照、restore() 从快照恢复。只有它能读写备忘录里的内容。被拍照的人
Memento
（备忘录）		存储发起人某一时刻状态的不可变值对象。对外只是个"黑盒句柄"，对发起人才敞开内部。一张照片
Caretaker
（管理者）		负责保管备忘录（通常用栈/列表存历史），但绝不查看也不修改其内容。它只管"存"和"取"。相册 / 保险柜
┌───────────────┐ createMemento() ┌──────────────┐ push/pop ┌──────────────┐ │ Originator │─────────────────▶│ Memento │◀────────────│ Caretaker │ │ (Editor) │ │ (不可变快照) │ 只存不看 │ (History 栈) │ │ - state │◀─────────────────│ - state │ │ Stack<Memento>│ │ + restore(m) │ restore(m) └──────────────┘ └──────────────┘ └───────────────┘ ▲ 只有 Originator 能读/写 Memento 的内容；Caretaker 只当"快递箱"传递

三者的分工暗含一条信息边界：Memento 对 Originator 是"宽接口"（能看到全部字段），对 Caretaker 是"窄接口"（只是个不透明句柄)。这就是备忘录能"保存状态却不泄露封装"的核心机关，后面第五节会专门拆解。

三、反例 → 正例：为什么不能直接暴露字段

最"省事"的做法是把发起人的字段全 public，或提供一堆 getter/setter，让外部自己读出来存、再写回去恢复。看起来能用，实则埋雷：

Bad · 暴露内部让外部代存
class Editor {
  public String content;   // 全开放
  public int cursor;
}

// 调用方自己"扒"状态存起来
Editor e = new Editor();
String bakC = e.content;   // 手动读
int bakP = e.cursor;
// ... 一堆修改 ...
e.content = bakC;          // 手动写回
e.cursor  = bakP;

// 问题：
// 1. 封装被彻底打破，谁都能改
// 2. 新增字段 => 所有备份点都要改
// 3. 恢复逻辑散落在每个调用处
Good · 备忘录封装状态
class Editor {
  private String content;  // 依旧私有
  private int cursor;

  Memento save() {         // 自己拍照
    return new Memento(content, cursor);
  }
  void restore(Memento m) {// 自己还原
    this.content = m.content();
    this.cursor  = m.cursor();
  }
}

// 调用方只拿到不透明句柄
Memento snap = e.save();
// ... 一堆修改 ...
e.restore(snap);
// 字段永远私有，恢复逻辑内聚在 Editor
陷阱：把备份职责交给外部 = 封装崩塌。一旦调用方能直接读写 content/cursor，你就再也无法保证这两个字段的一致性约束（比如 cursor 不能越界）。更糟的是，将来 Editor 加一个 selectionRange 字段，所有散落在业务代码里的手动备份点都会"漏存"，导致恢复后状态残缺——这是典型的散弹式修改坏味道。

四、完整实现：一个可撤销的文本编辑器

下面用最经典的"编辑器 + 撤销栈"把三个角色写全。先看发起人 Editor 和它的内部备忘录。

Editor.java · 发起人 Originator + 备忘录 MementoJava
public class Editor {
    private String content = "";
    private int cursor  = 0;

    public void type(String text) {
        content = content.substring(0, cursor) + text
                + content.substring(cursor);
        cursor += text.length();
    }

    // —— 拍快照：把当前状态打包进一个不可变 Memento ——
    public Memento save() {
        return new Memento(content, cursor);
    }

    // —— 从快照恢复：只有 Editor 自己能读 Memento 的内部 ——
    public void restore(Memento m) {
        this.content = m.content;
        this.cursor  = m.cursor;
    }

    public String getContent() { return content; }

    // 静态内部类：天生能访问外围类私有成员，且对外只是"黑盒"
    public static class Memento {
        private final String content;   // final → 快照不可变
        private final int cursor;
        private Memento(String content, int cursor) {
            this.content = content;   // 构造器私有：外界无法伪造/篡改
            this.cursor  = cursor;
        }
    }
}

再看管理者 History——它用一个栈保管快照，只调 push/pop，从不碰 Memento 内部：

History.java · 管理者 Caretaker + 客户端Java
public class History {
    // Caretaker 只知道 Memento 的存在，看不到里面是什么
    private final Deque<Editor.Memento> stack = new ArrayDeque<>();

    public void push(Editor.Memento m) { stack.push(m); }
    public Editor.Memento pop()        { return stack.pop(); }
    public boolean isEmpty()          { return stack.isEmpty(); }
}

// —— 客户端：编辑 + 撤销 ——
Editor editor = new Editor();
History history = new History();

editor.type("Hello");
history.push(editor.save());   // 存档点 1: "Hello"

editor.type(" World");
history.push(editor.save());   // 存档点 2: "Hello World"

editor.type("!!!");           // 现在是 "Hello World!!!"

editor.restore(history.pop()); // 撤销 → "Hello World"
System.out.println(editor.getContent()); // Hello World
为什么用静态内部类？Java 里把 Memento 写成 Editor 的静态内部类是最地道的实现：内部类天生能访问外围类的私有成员（编译器会放行），因此 Editor.restore() 能直接读 m.content；而 Memento 的构造器是 private 的，外部（包括 Caretaker）根本 new 不出来，只能通过 editor.save() 拿到——封装就这样被守住了。

五、点睛之笔：宽接口与窄接口

备忘录最反直觉、也最常被面试追问的一点是：同一个 Memento 对象，要对两方呈现两副面孔。

宽接口 (Wide) 窄接口 (Narrow) ┌─────────────────────────┐ ┌─────────────────────────┐ │ Originator 看到的 Memento│ │ Caretaker 看到的 Memento │ │ · content / cursor 全可读│ │ · 什么都看不到，只是句柄 │ │ · 能读能写，用于恢复 │ │ · 只能存进栈、取出来 │ └─────────────────────────┘ └─────────────────────────┘ ▲ ▲ └──────── 同一个对象 ────────────────┘

如何用 Java 表达这两副面孔？除了上面的"静态内部类 + private 构造器"，还有一个更显式的手法：让 Memento 实现一个空的标记接口，Caretaker 只依赖这个空接口。

标记接口实现窄接口Java
// 窄接口：空的，Caretaker 只认识它，什么都拿不到
public interface Snapshot { }

public class Editor {
    // 具体 Memento 实现窄接口；对 Editor 是宽接口，对外只是 Snapshot
    private static class Memento implements Snapshot {
        final String content;
        Memento(String c) { this.content = c; }
    }
    public Snapshot save() { return new Memento(content); }
    public void restore(Snapshot s) {
        this.content = ((Memento) s).content;  // 只有 Editor 知道真身，向下转型取值
    }
}
建立联系："宽/窄两副接口"和你学过的代理模式、迭代器里"对外只暴露必要方法"是同一种接口隔离思想。JDK 里 Collections.unmodifiableList() 返回的也是"窄化"后的视图——同一份数据，给不同调用者不同的可见能力。这本质是最小知识/最小权限原则在 API 层的体现。

六、与 JDK / Spring / 数据库 / OS 的联系

备忘录不是一个"课本里才有"的模式，它在真实系统里无处不在，只是常常换了名字叫"快照""存档""检查点""版本"：

  • JDK · Swing 撤销框架：javax.swing.undo 包里的 UndoableEdit / UndoManager 就是备忘录 + 命令的组合，UndoManager 正是一个专业的 Caretaker。
  • JDK · 序列化：java.io.Serializable 把对象状态"外化"成字节流，是备忘录思想的通用化——用序列化做深拷贝快照非常常见（但性能差、需处理 transient）。
  • 数据库 · 事务与保存点：事务日志（undo log）就是数据库版的备忘录，SAVEPOINT + ROLLBACK TO 让你回到某个中间快照——和撤销栈同构。
  • Spring · @Transactional 回滚：方法抛异常时把数据恢复到方法进入前，语义上就是"恢复到进入事务时的备忘录"。
  • OS / 中间件：Redis 的 RDB 快照、虚拟机快照、Git 的每次 commit（一棵完整目录树的不可变快照）、浏览器的历史后退——全是备忘录的放大版。
高频面试点：备忘录 vs 原型（Prototype）。两者都"复制状态"，但意图不同：原型是为了造一个新对象（clone 出可独立使用的副本），备忘录是为了存档以便日后恢复原对象，副本只当"底片"、不参与业务。此外备忘录强调不破坏封装、有专门的 Caretaker 管历史，而 clone 出的对象通常直接交给业务用。

七、横向对比：几种"恢复"方案的取舍

方案怎么存封装典型代价 / 适用
备忘录模式发起人自造快照对象不破坏状态复杂、需干净撤销/存档；要留意内存
直接序列化整对象转字节/JSON基本不破坏通用但慢、体积大；跨进程/持久化时用
原型/clone深拷贝一份新对象取决于 clone 实现易踩浅拷贝坑；意图是造副本而非存档
命令模式反向操作记录"怎么改"而非"改成啥"不破坏省内存（只存增量），但每步都要能写逆操作
备忘录 vs 命令，实现撤销的两条路：备忘录是状态快照法——存"结果"，恢复=整体替换，简单但费内存;命令模式是反向操作法——存"动作"，撤销=执行逆操作（如 delete 的逆是 insert），省内存但要求每个操作可逆。工业级编辑器（如 IDEA）常两者混用：小改动记增量命令，大改动或难以求逆的操作存快照。

八、陷阱与取舍

三个必须警惕的坑：
1. 内存开销：每存一个快照就多一份状态副本。无限撤销 = 内存无限增长。务必给历史栈设上限（如只保留最近 50 步），或改用命令模式存增量。
2. 浅拷贝陷阱：若状态里含可变引用（如 List、自定义对象），Memento 只存引用等于没存——原对象一改，快照跟着变。必须深拷贝或存不可变副本，这和 原型模式的浅/深拷贝是同一个坑。
3. 频繁快照的性能：大对象每次操作都全量快照会拖垮性能。可采用"节流"（每 N 步存一次）或写时复制（copy-on-write）优化。

用一条"公式"收束备忘录与它的两种典型用法：

撤销 = 恢复(历史栈.pop())  ·  存档 = 历史.push(save())
核心结论：备忘录模式 = 把"某一刻的状态"打包成一个不可变黑盒，交给第三方保管，日后原样还原。它的灵魂不是"能存状态"（谁都能拷贝），而是"存状态却不泄露封装"——靠 Originator 独占读写、Caretaker 只当快递、Memento 双接口这三条纪律实现。凡是遇到撤销/回滚/存档/检查点，而你又不想把对象内部暴露给外界，就该想到它;但一定要为"内存"和"深拷贝"这两件事付出设计代价。
