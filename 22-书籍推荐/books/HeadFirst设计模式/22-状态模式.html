状态模式：状态机行为封装

让对象在内部状态改变时改变它的行为，看起来就像换了一个类——把散落各处的 if/else 状态机装进一组状态对象里。

一、从一台会"抽风"的糖果机说起

《Head First 设计模式》用一台糖果机贯穿全章。它有四个状态：没有硬币（NoQuarter）、有硬币（HasQuarter）、售出糖果（Sold）、糖果售罄（SoldOut）。用户能做四个动作：投币、退币、转动曲柄、发放糖果。

问题在于：同一个动作，在不同状态下结果完全不同。"投币"在"没有硬币"时应该接受，在"有硬币"时应该拒绝（"你不能再投一枚了"）。这正是有限状态机（FSM, Finite State Machine）的典型场景。

状态转移图（× 表示该动作在此状态下被拒绝） 投币 转动曲柄 发放 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ NoQuarter│─▶│ HasQuarter│─▶│ Sold │──┐ └──────────┘ └──────────┘ └──────────┘ │ 发放成功 ▲ ▲ │ 退币 │ │ │ └──────────┘ │ 售罄 ▼ │ ▼ 回到 NoQuarter └────────────────────── ┌──────────┐ (还有糖) 没糖了 │ SoldOut │ └──────────┘
什么是状态模式？GoF 定义：允许一个对象在其内部状态改变时改变它的行为，对象看起来好像修改了它的类。核心手法是——把每一个状态抽象成一个类，把"在该状态下各动作的行为"作为方法封装进去；上下文（Context）持有"当前状态"引用，动作调用直接委托给当前状态对象。

二、反例：if/else 状态机为什么会腐烂

最直觉的写法是用整型常量表示状态，每个动作方法里用一串 if/else 判断当前状态。糖果机只有 4 个状态时还能看，一旦老板要求"加一个抽奖状态：10% 概率转一次曲柄掉两颗糖"，你就得回到每一个方法里去插新分支——这直接违反开闭原则。

Bad · if/else 巨型状态机
final int SOLD_OUT=0, NO_QUARTER=1,
          HAS_QUARTER=2, SOLD=3;
int state = NO_QUARTER;

void insertQuarter() {
  if (state == HAS_QUARTER)
    print("不能再投币");
  else if (state == NO_QUARTER) {
    state = HAS_QUARTER;
    print("已收到硬币");
  } else if (state == SOLD_OUT)
    print("售罄，退币");
  else if (state == SOLD)
    print("请稍候");
}
// turnCrank / dispense / ejectQuarter
// 每个方法都要重复这套 if/else...
// 加一个状态 = 改所有方法
Good · 每个状态一个类
interface State {
  void insertQuarter();
  void turnCrank();
  void dispense();
  void ejectQuarter();
}
// 各状态各自实现，行为就地封装
class NoQuarterState implements State {
  void insertQuarter() {
    machine.setState(
        machine.getHasQuarterState());
    print("已收到硬币");
  }
  // 其余动作各自处理...
}
// 加状态 = 新增一个 State 类
// 老代码一行不改
坏味道信号：当你发现多个方法里都出现结构相同的 switch(state) / if(state==...)，且新增一个枚举值要同步改多处——这就是状态模式的"召唤咒语"。这类代码的圈复杂度随"状态数 × 动作数"爆炸，测试用例组合也随之爆炸。

三、正例：完整的状态模式实现

先看上下文 GumballMachine。它不再关心"现在是什么状态该怎么做"，只负责持有所有状态实例、记录当前状态，并把动作原样委托出去。

GumballMachine.java · 上下文（Context）Java
public class GumballMachine {
    // 预先创建好所有状态对象（无状态可共享，见后文陷阱）
    private final State noQuarter   = new NoQuarterState(this);
    private final State hasQuarter  = new HasQuarterState(this);
    private final State sold       = new SoldState(this);
    private final State soldOut    = new SoldOutState(this);

    private State state;      // 当前状态：整台机器行为的"总开关"
    private int count;        // 剩余糖果数

    public GumballMachine(int count) {
        this.count = count;
        this.state = count > 0 ? noQuarter : soldOut;
    }

    // 关键：动作全部委托给"当前状态对象"，Context 自己不做 if 判断
    public void insertQuarter() { state.insertQuarter(); }
    public void ejectQuarter()  { state.ejectQuarter();  }
    public void turnCrank() {
        state.turnCrank();
        state.dispense();   // 转完曲柄紧接着发糖
    }

    // 状态切换的唯一入口，供各 State 回调
    public void setState(State s) { this.state = s; }
    public void releaseBall() {
        if (count > 0) { System.out.println("糖果滚出..."); count--; }
    }
    // getter：让状态对象拿到彼此的引用 & 库存
    State getNoQuarter()  { return noQuarter;  }
    State getHasQuarter() { return hasQuarter; }
    State getSold()       { return sold;       }
    State getSoldOut()    { return soldOut;    }
    int getCount()      { return count;      }
}

再看两个典型状态类。注意"下一个状态去哪儿"这条边，被写死在当前状态类里——这就是状态转移逻辑的封装位置。

State 接口 + 两个具体状态Java
public interface State {
    void insertQuarter();
    void ejectQuarter();
    void turnCrank();
    void dispense();
}

public class NoQuarterState implements State {
    private final GumballMachine m;
    public NoQuarterState(GumballMachine m) { this.m = m; }

    public void insertQuarter() {
        System.out.println("收到硬币");
        m.setState(m.getHasQuarter());   // 状态转移：NoQuarter → HasQuarter
    }
    public void ejectQuarter() { System.out.println("你还没投币"); }
    public void turnCrank()    { System.out.println("先投币再转");   }
    public void dispense()     { System.out.println("未付款");       }
}

public class SoldState implements State {
    private final GumballMachine m;
    public SoldState(GumballMachine m) { this.m = m; }

    public void dispense() {
        m.releaseBall();
        // 根据库存决定下一个状态——转移逻辑就近内聚
        m.setState(m.getCount() > 0 ? m.getNoQuarter() : m.getSoldOut());
    }
    public void insertQuarter() { System.out.println("请稍候"); }
    public void ejectQuarter()  { System.out.println("已在出糖"); }
    public void turnCrank()     { System.out.println("别转两次"); }
}
现在加"抽奖状态"多简单：只需新写一个 WinnerState 类，再在 HasQuarterState.turnCrank() 里按概率切到它。已有的四个状态类一行都不用动——这正是"对扩展开放、对修改关闭"的落地。

三·五、UML 结构与角色

┌──────────────┐ 持有 current ┌───────────────┐ │ Context │────────────────▶│ <<interface>> │ │(GumballMachine) │ State │ │ + request() ─┼── 委托 ───────▶ │ + handle() │ │ + setState() │ └───────┬───────┘ └──────────────┘ │ implements ▲ 回调 setState 切换状态 ├──────────┬──────────┐ └──────────────────────────┐ ┌──┴───┐ ┌────┴───┐ ┌────┴────┐ └──│StateA│ │ StateB │ │ StateC │ └──────┘ └────────┘ └─────────┘
  • Context（上下文）：持有一个 State 引用，把请求委托给它；对外暴露 setState() 供状态切换。
  • State（抽象状态）：声明所有"动作"方法。
  • ConcreteState（具体状态）：实现本状态下各动作的行为，并决定下一步转移到哪个状态。

四、状态模式 vs 策略模式：一对"双胞胎"

两者的类图几乎一模一样（都是 Context 委托给一个接口的实现），这是面试高频陷阱。区别不在结构，在意图。

维度状态模式 State策略模式 Strategy
意图随内部状态自动改变行为让客户端选择一种算法
切换者状态对象/Context 自己切换由客户端从外部设定
对象间关系状态之间彼此知晓、会互相转移各策略互不知晓、平行独立
数量与顺序有限且有转移顺序（状态图）一组可互换、无先后
类比红绿灯：绿→黄→红自动流转导航"最快/最短/避堵"三选一
一句话记忆：策略模式是"我用哪种算法由外部一次性决定"；状态模式是"我表现出哪种行为由自己当前处境决定，且会自动演进到下一处境"。同样是委托，一个是横向可替换，一个是纵向会流转。

五、与 JDK / Spring / 底层的联系

状态机是计算机世界的通用骨架，你早就在用它：

Thread.State 状态机（JDK 线程状态，本质就是一台 FSM） NEW ──start()──▶ RUNNABLE ◀──────┐ notify()/锁获得 │ │ │ wait()/sleep │ │ 抢到 CPU │ ▼ │ │ WAITING/TIMED_WAITING─┘ │ synchronized 竞争失败 ▼ BLOCKED ──▶ RUNNABLE ──run 结束──▶ TERMINATED
  • JDK：java.lang.Thread.State 就是六状态有限状态机（NEW / RUNNABLE / BLOCKED / WAITING / TIMED_WAITING / TERMINATED）。JVM 内部依据锁、wait/notify 等事件驱动状态转移，与糖果机同构。
  • JDK：TCP 协议栈（CLOSED→LISTEN→SYN_RCVD→ESTABLISHED…）是操作系统里最经典的状态机；AbstractQueuedSynchronizer（AQS）用一个 volatile int state 表达同步状态，也是状态思想。
  • Spring：Spring StateMachine 是把状态模式做成的完整框架，声明式配置状态、事件、转移与守卫（guard）——适合订单流转、审批流这类复杂 FSM。
  • Spring：Bean 生命周期（实例化→属性填充→初始化→就绪→销毁）本质也是状态推进。
  • 生活/前端：视频播放器（播放/暂停/缓冲/结束）、订单（待付款/已付款/已发货/已完成/已取消），前端 XState 库就是状态机的直接产品化。
核心结论：状态模式 = 用多态替换条件分支。把"状态 × 动作"的二维表格，从散落在多个方法里的 if/else，重构为"一状态一类、行为就地封装、转移边写在状态内部"。它把圈复杂度转化为类的数量——牺牲一点类膨胀，换来开闭、可测试与可读性。凡是"同一动作在不同状态下行为不同、且状态会按规则自动流转"的场景，都该想到它。

六、陷阱与取舍

三个常见坑：
1. 状态对象能否共享？若状态类不持有实例字段（行为只依赖传入的 Context），可像本例一样每台机器复用单例状态对象，省内存；一旦状态类里存了可变数据，就绝不能共享，否则并发下数据串台。
2. 转移逻辑放哪？把"下一状态"写死在具体状态类里，会让状态之间产生耦合（StateA 需 import StateB）；若转移规则频繁变动，可考虑把转移表抽到 Context 或独立的转移表/配置里集中管理。
3. 并发安全：多线程同时驱动同一个 Context 时，setState() 与读 state 存在竞态（这正是 Java 内存模型里的可见性/原子性问题）。需要对状态切换加锁或用 AtomicReference 做 CAS，否则可能出现"投两次币都成功"的错乱。
if/else 状态机状态模式
新增状态改所有动作方法（违反开闭）加一个类，老代码不动
类的数量少（就一个类）多（状态数 + 1）
行为内聚同一状态的逻辑散落各方法同一状态的逻辑聚在一个类
可测试性需构造整机 + 复杂前置可单独 new 一个状态测
适用规模状态/动作都很少且稳定状态多、动作多、常扩展

转移逻辑的选择可用一句"公式"概括：

行为 = 当前状态(动作)  而非  if(状态) { 动作 }
落地建议：不要一上来就套状态模式。先用枚举 + switch 顶着，当你第二次因为加状态而不得不改多个方法、或 switch 分支超过 3~4 个且还在长，再重构为状态模式——这符合"先让它工作，再让它优雅"的节奏，也避免过度设计。
