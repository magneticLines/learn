组合模式：树形结构的一致处理

让“一个对象”和“一堆对象组成的树”长同一副面孔——客户端从此不必区分叶子与枝干。

一、痛点：调用方被“单个 vs 一组”的判断淹没

想象你在做一个餐厅点菜系统。菜单里有菜品（一份宫保鸡丁），也有子菜单（“甜点”这一整栏，里面又装着若干菜品）。老板还想在“甜点”里再塞一个“进口冰淇淋”子菜单——菜单可以无限套娃。

如果不加设计，客户端打印整份菜单时就会写成这样：拿到一个元素，先 instanceof 判断“你是菜品还是菜单？”，是菜品就直接打印，是菜单就递归遍历它的孩子。每加一个新操作（计价、导出 JSON、统计辣度）都要把这套 if-else 判断重抄一遍。

生活类比：文件系统。Windows 里一个“文件”和一个“文件夹”，你都能对它们右键“属性 → 大小”。文件夹的大小其实是把里面所有文件（和子文件夹）的大小递归加起来。但你操作时感觉不到区别——都是右键、都是“大小”。组合模式就是把“文件”和“文件夹”统一成同一种东西：File，让你用同一套动作对待单个和整棵树。

二、模式定义与三个角色

GoF 定义：将对象组合成树形结构以表示“部分-整体”的层次关系，使得客户端对单个对象（叶子）和组合对象（容器）的使用具有一致性。它是结构型模式——和适配器、装饰器一样，都在解决“对象怎么拼装成更大结构”，而组合专攻递归的树。

┌──────────────────────────┐ │ Component（抽象） │ ← 叶子和容器共同的“超类型” │ + print() │ │ + add(Component) * │ ← 容器才真正需要的操作 │ + getChild(int) * │ 也提在这里（透明式） └────────────┬─────────────┘ ▲ ▲ ┌───────┘ └────────────┐ ┌─────────────┐ ┌──────────────────────┐ │ MenuItem │ │ Menu │ │ （Leaf 叶子）│ │ （Composite 容器） │ │ print() │ │ List<Component> kids │ │ │ │ add / print(递归) │ └─────────────┘ └──────────────────────┘ Client ──▶ 只持有 Component，调 print() 即可 不关心手里是一道菜还是一整棵菜单树
  • Component（抽象构件）：叶子与容器的共同超类型，声明所有人都要响应的操作（如 print()）。这是“一致处理”的地基。
  • Leaf（叶子）：树的末梢，没有子节点（一道菜品）。它实现自己的业务操作。
  • Composite（容器）：持有一个 List<Component> 装孩子，它的操作通常是“先处理自己，再递归委派给每个孩子”。
Composite.print() = 自己的信息 + Σ child.print()
容器的行为 = 对所有孩子做同样的行为（递归的自相似）
为什么天然是递归的？因为 Composite 的孩子类型是 Component，而 Composite 自己也是 Component——所以容器里可以再装容器，无限嵌套。这个“整体和部分是同一类型”的自相似结构，正是递归数据结构（树）的本质，也是组合模式区别于普通聚合的关键。

三、最小可运行示例：统一的菜单树

先定义共同超类型，再让叶子和容器各自实现。注意客户端代码有多干净。

MenuComponent.java抽象构件：叶子与容器的共同面孔
abstract class MenuComponent {
  // 所有节点都要能打印自己（可能带缩进表示层级）
  abstract void print(String indent);

  // ↓ 只有容器才真正需要的操作，抽象类给个默认实现（透明式设计）
  void add(MenuComponent c)    { throw new UnsupportedOperationException(); }
  MenuComponent getChild(int i) { throw new UnsupportedOperationException(); }
}
MenuItem.java / Menu.java叶子 + 容器
// ===== 叶子：一道菜，没有孩子 =====
class MenuItem extends MenuComponent {
  private final String name;
  MenuItem(String name) { this.name = name; }

  @Override void print(String indent) {
    System.out.println(indent + "- " + name);   // 叶子：只画自己
  }
}

// ===== 容器：一栏菜单，装着若干 MenuComponent =====
class Menu extends MenuComponent {
  private final String name;
  private final List<MenuComponent> kids = new ArrayList<>();

  Menu(String name) { this.name = name; }

  @Override void add(MenuComponent c) { kids.add(c); }
  @Override MenuComponent getChild(int i) { return kids.get(i); }

  @Override void print(String indent) {
    System.out.println(indent + "[" + name + "]");       // ① 先处理自己
    for (MenuComponent c : kids) c.print(indent + "  ");  // ② 再递归委派给每个孩子
  }
}
Client.java客户端：对叶子和树一视同仁
Menu root = new Menu("全部菜单");
Menu dessert = new Menu("甜点");
dessert.add(new MenuItem("提拉米苏"));
dessert.add(new MenuItem("进口冰淇淋"));

root.add(new MenuItem("宫保鸡丁"));  // 直接挂一片叶子
root.add(dessert);                    // 挂一整棵子树，方法调用完全相同

// 客户端只认 MenuComponent，不写一个 instanceof、不关心层级深度
root.print("");

关键在最后一行：root.print("") 一句话就打印了整棵树。客户端没有任何 if 去区分“这是菜还是菜单”——多态和递归替它扛下了所有分支。这就是“一致处理”的全部含金量。

四、有它 vs 没它：客户端代码的天壤之别

Bad · 无组合模式：类型判断满天飞
void print(Object node, String pad) {
  // 每个操作都要重写这套判断
  if (node instanceof Dish d) {
    System.out.println(pad + d.name);
  } else if (node instanceof Category c) {
    System.out.println(pad + c.title);
    for (Object child : c.items) {
      // 递归，还得再判断一次类型
      print(child, pad + "  ");
    }
  }
  // 新增“文件夹套文件夹”？
  // 新增 calcPrice()？统计辣度？
  // → 每个方法都复制粘贴这坨 if-else
}
Good · 组合模式：判断消失了
// 判断逻辑被“下沉”到各自的类里
// 客户端只面向 Component 编程
component.print("");

// 叶子的 print 只画自己
// 容器的 print 递归画孩子
// —— 多态自动分派，调用方零分支

// 新增操作？给 Component 加一个
// 抽象方法，叶子/容器各实现一遍
// 客户端代码一行都不用改
本质：把“类型判断”从调用方转移到了类型系统。instanceof + if-else 是把分支写在调用处，一处新增就要处处修改；组合模式借助多态，把“我该怎么响应”封进每个类自己身上——这正是“用多态消灭条件分支”这条老箴言在树形结构上的落地，也直接呼应了你学过的开闭原则。

五、透明式 vs 安全式：本模式最经典的取舍

那些“只有容器才有意义”的方法（add/remove/getChild）到底放哪？这是组合模式绕不开的设计抉择，也是面试高频考点。

维度透明式（Transparent）安全式（Safe）
把 add/remove 放哪放在 Component 顶层，叶子也继承到只放在 Composite 里，叶子没有
叶子与容器接口完全一致，客户端无需区分接口不同，操作子节点前要先转型
代价对叶子调 add() 能编译通过，运行期才抛异常类型安全，但要 instanceof + 强转，破坏透明性
GoF 立场更推崇（一致性优先）更安全但牺牲统一接口

上面的示例走的是透明式：add() 提到了 MenuComponent，代价是 MenuItem 也“继承”到了一个它并不支持的 add()，只能靠抛 UnsupportedOperationException 兜底。

陷阱一：透明式把“不该有的能力”暴露给了叶子。客户端写 menuItem.add(x) 编译器不拦，直到运行时才炸——这是用运行期异常换编译期透明的妥协。JDK 里 Collections.unmodifiableList() 返回的列表调 add() 抛 UnsupportedOperationException，是同一套思路，也同样被诟病“不够类型安全”。
陷阱二：别在 equals/hashCode 或 toString 里无脑递归。如果树被意外构造成有环（把祖先节点 add 回子节点），递归遍历会栈溢出 / 死循环。组合模式默认假设结构是无环的树（DAG 的特例），必要时要在 add() 里做环检测。
陷阱三：父指针与一致性。若为了“向上遍历”给每个节点加 parent 引用，务必在 add/remove 时同步维护它，否则父子关系会对不上。

六、和迭代器、访问者的配合

组合模式很少单打独斗，它常和这几个模式咬合在一起：

  • + 迭代器模式：想“遍历整棵树的所有叶子”而不暴露内部结构？给 Composite 实现一个组合迭代器（内部用栈把递归展平），客户端就能 for-each 整棵树。
  • + 访问者模式：树结构稳定、但操作频繁新增（打印、计价、导出）时，把操作抽成 Visitor，避免每加一个操作就往 Component 塞一个新方法。
  • + 装饰器模式：两者结构图很像（都递归持有同类型），但意图相反——组合表达“部分-整体”的一对多包含；装饰表达“层层加料”的一对一包裹。别混淆。

七、串联 JDK / Spring / 前端：它无处不在

出处ComponentLeaf / Composite
AWT / Swingjava.awt.ComponentButton(叶) / Container、JPanel(容器，可装组件)
DOMNode文本节点(叶) / Element(可 appendChild)
文件系统java.io.File / Path文件(叶) / 目录(容器)；Files.walk 递归遍历树
Spring—CompositeCacheManager、CompositeUserDetailsService 等 Composite*：把多个同接口实现当成一个用
React / VueVNode / 组件叶组件 / 容器组件（children 递归渲染）——整个虚拟 DOM 树就是组合
Swing 的活教材：Container.add(Component c) 里，参数和容器自身都是 Component——所以 JPanel 里能放 JButton，也能放另一个 JPanel，无限嵌套。当你调 panel.paint(g)，它会画自己再递归 paint 每个孩子。这就是本节 print() 示例的工业级版本，你早就用过组合模式，只是没意识到。
Spring 的 Composite* 家族体现了组合模式的另一面：不只用于“树”，也用于“把 N 个同接口的实现聚合成 1 个”。例如 CompositeCacheManager 持有多个 CacheManager，对外仍是一个 CacheManager——调用方拿到缓存的过程被统一，底层是遍历所有子 manager。这与“客户端一致处理”的内核完全一致。

八、什么时候该用、什么时候别硬套

组合模式的适用信号很明确：你的数据天然是“部分-整体”的树，且希望对单体和整体施加统一操作。反之，如果结构是扁平的、或者叶子与容器的行为差异巨大到根本无法抽象出共同接口，硬凑一个 Component 只会让抽象变得空洞。

直观感受一下“新增一种操作”的成本差异：

无组合：每个操作重写 instanceof 判断 + 改动散落各处 组合：加一个方法
别过度设计。只有两三个固定层级、且永远不会嵌套的结构（比如“订单 → 订单项”，订单项里绝不会再套订单），用组合模式反而是杀鸡用牛刀——直接 List<OrderItem> 更清晰。组合模式的收益来自“不确定深度的递归嵌套”；没有递归嵌套，就没必要请出它。

九、一句话收束

组合模式 = 让叶子和容器共享同一个超类型，把“部分-整体”的树封装成客户端可以一致处理的整体。它的价值在不确定深度的递归树上最亮眼。落地记住三条：① 定义共同的 Component 超类型，容器的操作 = 处理自己 + 递归委派给孩子，客户端从此告别 instanceof；② 透明式（统一接口、叶子对不支持的操作抛异常）vs 安全式（类型安全但要转型）是核心取舍，GoF 偏向透明式；③ 警惕环导致的栈溢出、以及给叶子暴露 add() 的运行期风险。看懂它，你就看懂了 Swing、DOM、文件系统和虚拟 DOM 的共同骨架。
