中介者模式：多对象交互集中管理

把"谁跟谁说话"从一堆对象里抽出来，交给一个中间人统一调度，让网状依赖塌缩成星形。

一、从一个失控的智能家居说起

Head First 里的场景是一套"傻瓜之家"：闹钟、咖啡壶、日历、洒水器互相认识。周末闹钟不该响；咖啡壶要在闹钟响后 15 分钟才开始煮；下雨天洒水器要看日历有没有安排户外活动……于是每个对象里都塞满了"如果别的对象怎样，我就怎样"的判断。

问题的本质是：N 个对象两两直接引用，交互规则分散在每个对象内部。任何一条规则变化，都要打开好几个类去改。这是"紧耦合"的教科书式反例。

网状耦合（无中介者） 星形调度（有中介者） 闹钟 ───────── 咖啡壶 闹钟 咖啡壶 │ ╲ ╱ │ ╲ ╱ │ ╲ ╱ │ ╲ ╱ │ ╲ ╱ │ ┌───────────┐ 日历 ──╳───── 洒水器 │ 中介者 │ │ ╱ ╲ │ └───────────┘ │ ╱ ╲ │ ╱ ╲ 连接数 ≈ N(N-1)/2 ╱ ╲ 日历 洒水器 连接数 = N

核心的定量差异，可以用连接数来量化——这正是中介者要打的那一仗：

网状：N·(N-1)/2 条依赖  →  星形：N 条依赖

10 个对象两两相连是 45 条边，加一个对象要接 10 条新边（O(N²) 增长）；改成中介者后，加一个对象只需接 1 条边到中介者（O(N) 增长）。这就是"多对象交互集中管理"的价值来源。

二、定义与角色

意图（GoF）：用一个中介对象来封装一系列对象之间的交互。中介者使各对象不需要显式地相互引用，从而使其松耦合，而且可以独立地改变它们之间的交互。

拆成四个角色，对照上面的家居例子：

角色职责家居例子
Mediator中介者接口，声明同事对象与自己通信的方法Mediator 接口
ConcreteMediator持有所有同事引用，实现协调逻辑（规则都在这里）SmartHomeHub
Colleague同事基类，只持有中介者引用，不认识其他同事各设备的基类
ConcreteColleague发生状态变化时通知中介者，由中介者决定后续闹钟、咖啡壶、洒水器…
关键心智模型：同事对象只会说两句话——"我出事了，中介者你看着办"（changed()），以及被动接受"中介者让我做某事"。同事之间永远不直接对话。

三、反例 → 正例：智能家居

先看没有中介者时的"设备互相引用"，再看引入中介者后的样子。注意 bad 里每个设备都持有其它设备的引用，规则写死在设备内部。

反例 · 同事互相引用
class Alarm {
  // 闹钟居然要认识咖啡壶、洒水器
  CoffeePot pot;
  Sprinkler sprinkler;

  void onTimeReached() {
    if (isWeekend()) return;   // 规则散落
    ring();
    pot.startBrewing();        // 直接指挥别人
    sprinkler.checkAndWater(); // 加一个设备
                               // 就要改这里
  }
}
正例 · 只认识中介者
class Alarm extends Device {
  Alarm(Mediator m) { super(m); }

  void onTimeReached() {
    // 我不判断、不指挥，只上报
    mediator.deviceChanged(this,
        "ALARM_RANG");
  }
}
// 加设备/改规则 → 只动 Hub，
// Alarm 一个字都不用改

下面是完整的正例骨架，注意规则如何被集中到 SmartHomeHub：

SmartHomeHub.javaMediator Pattern
// 1) 中介者接口：同事通过它上报事件
interface Mediator {
    void deviceChanged(Device source, String event);
}

// 2) 同事基类：只持有 mediator，彼此不相识
abstract class Device {
    protected final Mediator mediator;
    protected Device(Mediator m) { this.mediator = m; }
}

class Alarm extends Device {
    Alarm(Mediator m) { super(m); }
    void timeReached() {
        mediator.deviceChanged(this, "ALARM_RANG"); // 只上报
    }
}
class CoffeePot extends Device {
    CoffeePot(Mediator m) { super(m); }
    void brew() { System.out.println("开始煮咖啡"); }
}

// 3) 具体中介者：所有协调规则集中于此
class SmartHomeHub implements Mediator {
    private CoffeePot pot;
    public void register(CoffeePot p) { this.pot = p; }

    public void deviceChanged(Device src, String event) {
        if ("ALARM_RANG".equals(event)) {
            // 规则 = 谁触发 + 触发什么，都写在中介者里
            pot.brew();
        }
    }
}
陷阱一：中介者变成"上帝对象"。规则全塞进 ConcreteMediator，它会随着同事和规则增长而膨胀成一个几千行、谁都不敢碰的巨类。中介者本质上是把"分散的复杂度"换成了"集中的复杂度"——它没有消灭复杂度，只是搬了家。对策：当规则过多时，按业务域拆分多个中介者，或让中介者内部委托给策略对象/规则引擎，而不是一路 if-else 堆到底。

四、和"长得像"的模式划清界限

中介者、观察者、外观、责任链在"解耦"这件事上气质相近，初学者极易混淆。用一张表钉死区别：

模式解决的核心问题通信方向谁知道谁
中介者多个对象双向、多对多交互的混乱星形，多对多同事只知中介者
观察者一个主题变化要通知一群观察者单向广播，一对多主题持有观察者列表
外观给复杂子系统一个简化入口单向，外→内外部→子系统（子系统不知外观）
责任链请求沿一条链逐个尝试处理单向，链式传递每个节点知道下一个
最容易搞混的是「中介者 vs 观察者」。观察者是"一个说，多个听"的单向广播（主题不关心谁听、听了做什么）；中介者是"多个互相说、由中间人裁决"的双向协调（规则的判断权在中介者手里）。实践中两者常配合使用：同事用观察者机制把事件喂给中介者，中介者再决定指挥谁。

五、你天天在用，只是没意识到

中介者是那种"框架帮你实现了、你只管享用"的模式。它无处不在：

  • Spring 的 ApplicationContext / IoC 容器：Bean 之间不 new 彼此，而是向容器声明依赖，由容器注入并管理生命周期。容器就是所有 Bean 的中介者——这也是"控制反转"的一种体现。
  • Spring MVC 的 DispatcherServlet：它是前端控制器，也是 HandlerMapping、Controller、ViewResolver 之间的中介者；这些组件互不引用，全部由 DispatcherServlet 编排。
  • JVM 层面的 java.util.Timer / 线程池 ThreadPoolExecutor：提交任务的线程与执行任务的 worker 线程不直接交互，靠中间的任务队列 + 调度器解耦——思想同源。
  • 消息中间件（Kafka / RabbitMQ）：生产者与消费者互不相识，Broker 就是分布式规模上的中介者。这说明该模式可以跨越"一个进程内"直接放大到"整个系统架构"。
  • GUI：Head First 用了对话框控件的例子。JavaFX/Swing 里各控件的联动（选了 A 就禁用 B）交给控制器/中介者，而不是控件互相引用。
陷阱二：把中介者用成了"消息总线穿透一切"。如果两个对象关系简单、只是 A 调一下 B，直接引用反而更清晰。中介者的成本（多一层间接、可读性下降、调试要多跳一跳）只有在"交互确实是网状且易变"时才划算。不要为了解耦而解耦——过度使用会让调用链变成"到处发消息，没人知道谁会响应"。

六、优缺点权衡

收益
+ 依赖 O(N²) → O(N)，
  同事之间彻底解耦
+ 交互规则集中，单一位置
  修改（符合单一职责）
+ 同事可独立复用/替换
+ 新增同事不影响旧同事
代价
- 中介者易膨胀成上帝对象
- 复杂度只是转移，未消灭
- 多一层间接，调试要多跳
- 用错场景反而更绕
  （简单交互别硬套）
一句话收束：中介者模式用"多一个中间人"换掉"多对象之间的两两耦合"——把散落在各处的交互规则收进一个可控的地方，让依赖从 O(N²) 的网 塌缩成 O(N) 的星。它的适用边界也很清楚：当且仅当对象间是网状、双向、易变的交互时才用它；否则你只是给简单调用套上了一层没必要的间接，还顺手养出一个上帝对象。
