封装变化

软件唯一不变的就是"会变"。把变化的部分揪出来单独关进笼子，剩下的稳定代码才能睡个好觉。

一、开场：一只会"飞"的橡皮鸭

这是《Head First 设计模式》第 1 章的经典案例。你在做一个鸭子池塘模拟器 SimUDuck：一开始只有一个 Duck 超类，子类继承它——绿头鸭、红头鸭都会 quack()（叫）、swim()（游）、display()（长相各异）。设计得很"面向对象"，皆大欢喜。

直到产品经理说："给鸭子加个 fly() 吧，让它们飞起来更好玩。"你顺手把 fly() 加到超类 Duck 里——毕竟"所有鸭子都能飞"嘛。演示当天，屏幕上一群橡皮鸭（RubberDuck）在天上乱飞。橡皮鸭不会飞，它只会"吱吱"响。

这不是一个"低级 bug"，而是设计的必然。 你把一个只对部分子类成立的行为放进了超类，继承就会无差别地把它"传染"给所有子类。生活类比：你给"交通工具"基类加了个 加油() 方法，结果电动车、自行车都被迫"加油"了。问题不在这一行代码，而在于你没有区分"哪些是共性、哪些是会变的差异"。

二、两次"打补丁"，两次踩坑

面对满天飞的橡皮鸭，直觉上有两条路，书里都走了一遍，也都撞了墙：

方案做法为什么不行
① 继承里覆盖橡皮鸭 override fly() 让它"什么都不做"；木头诱饵鸭 DecoyDuck 把 fly() 和 quack() 都覆盖成空每加一个子类都要人肉检查"它该不该飞/该不该叫"，一旦漏改就出 bug。行为分散在几十个子类里，无法在运行时改变，改一处规则要翻遍整棵继承树
② 抽成接口定义 Flyable、Quackable 接口，会飞的鸭子才 implements Flyable接口只有方法签名、没有实现（本书基于 Java 5，接口无默认方法）。于是每个会飞的鸭子都要各自重写一遍一模一样的 fly()，代码复用被彻底摧毁——改飞行方式要改 N 个类
陷阱：把"继承"当成复用的唯一锤子。 初学者常有一种执念——看到相似行为就往超类塞。但继承是编译期绑定的强耦合：超类打个喷嚏，全体子类感冒。而"接口 + 各自实现"又走到另一个极端，为了灵活性丢掉了复用。这两次失败共同指向一个更本质的问题：fly / quack 这两个行为本身就是"会变的"，它们不该被硬编码在鸭子的类型体系里。

三、原则登场：封装变化

《Head First》给出的第一条设计原则，一句话点破迷雾：

设计原则 · 封装变化
找出应用中可能需要变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起。
Identify the aspects of your application that vary and separate them from what stays the same.

翻译成工程语言：先做一次"体检"，找到系统里变化的轴（axis of change），把它从稳定内核里"剥离"出来，用接口把变化的那一面封装起来，让稳定代码只依赖这层接口。这样一来——需求变了，你只改被隔离的那一小块，稳定内核纹丝不动。

系统 = 稳定内核（不变）  ⊕  可插拔策略（变化）

在鸭子例子里，体检结论很清楚：display() 各不相同但那是"长相"，真正让人头疼的是 fly 和 quack 这两族行为在不同鸭子间反复横跳。于是把它们抽成两个"行为家族"，从 Duck 里彻底搬出去：

把"变化"封装成行为接口 + 若干实现Java
// 变化点 1：飞行行为——独立成一个"家族"，各成员是可互换的算法
public interface FlyBehavior {
    void fly();
}
public class FlyWithWings implements FlyBehavior {
    public void fly() { System.out.println("我扇动翅膀飞起来了！"); }
}
public class FlyNoWay implements FlyBehavior {          // 橡皮鸭 / 诱饵鸭用它
    public void fly() { System.out.println("我不会飞。"); }
}

// 变化点 2：叫声行为——同理封装（Quack / Squeak / MuteQuack ...）
public interface QuackBehavior { void quack(); }

Duck 不再自己实现飞和叫，而是持有一个 FlyBehavior 和 QuackBehavior，把动作委托（delegate）出去。这就从"鸭子 是 会飞的"（IS-A，继承）变成了"鸭子 有 一个飞行行为"（HAS-A，组合）：

稳定内核：Duck 只负责"委托"，并开放运行时替换Java
public abstract class Duck {
    FlyBehavior flyBehavior;       // 组合：把会变的部分"抱"进来，而不是继承下来
    QuackBehavior quackBehavior;

    public void performFly() {
        flyBehavior.fly();            // 委托：自己不实现，交给持有的行为对象
    }
    public void performQuack() { quackBehavior.quack(); }

    // setter 让行为可在【运行时】热替换——继承永远做不到这一点
    public void setFlyBehavior(FlyBehavior fb) { this.flyBehavior = fb; }

    public void swim() { System.out.println("所有鸭子都会游泳。"); } // 不变的共性，留在这
    public abstract void display();
}

public class ModelDuck extends Duck {
    public ModelDuck() {
        flyBehavior = new FlyNoWay();      // 模型鸭出厂不会飞
        quackBehavior = new Quack();
    }
    public void display() { System.out.println("我是一只模型鸭。"); }
}

// 客户端：运行时给模型鸭装上"火箭引擎"，无需改任何已有类
Duck model = new ModelDuck();
model.performFly();                          // 输出：我不会飞。
model.setFlyBehavior(new FlyRocketPowered());  // ← 热插拔
model.performFly();                          // 输出：火箭推进，起飞！
你刚刚"发明"了策略模式。 把一族可互换的算法（FlyWithWings / FlyNoWay / FlyRocketPowered）各自封装、让它们的使用者（Duck）可以独立于算法而变化——这正是策略模式（Strategy）的定义。所以"封装变化"不是某一个模式，而是几乎所有设计模式背后共同的引擎：装饰者封装的是"附加职责"的变化，工厂封装的是"创建逻辑"的变化，观察者封装的是"谁在监听"的变化……记住原则，模式自然浮现。

四、反例 → 正例：继承覆盖 vs 组合委托

Bad · 继承里硬覆盖
class RubberDuck extends Duck {
  // 为了"不飞"，被迫覆盖成空
  @Override void fly() { /* 啥也不干 */ }
  @Override void quack(){ squeak(); }
}
// 痛点：
// · 行为写死在类型里，运行时改不了
// · 每加一个子类都要人肉判断
//   "它到底该不该 override"
// · 改飞行规则 → 翻遍整棵继承树
Good · 组合 + 委托
class RubberDuck extends Duck {
  RubberDuck() {
    flyBehavior  = new FlyNoWay();
    quackBehavior= new Squeak();
  }
}
// 收益：
// · 行为是"零件"，随时 set 热替换
// · 新增 FlyRocketPowered 不动老代码
//   （对扩展开放，对修改关闭）
// · FlyNoWay 可被 N 种鸭子复用

用一张图看清"变化"是怎么被从继承树里"抽走"、挂到一根独立的接口轴上的：

继承方案（变化混在类型里） 封装变化方案（变化挂在独立轴上） ────────────────────────── ────────────────────────────── Duck Duck ──HAS-A──► FlyBehavior «接口» fly() quack() ← 到处覆盖 performFly() ▲ ┌────┼────┐ performQuack() ├─ FlyWithWings Mallard Rubber Decoy │ ├─ FlyNoWay (改一处规则 = 全树体操) swim()(不变) └─ FlyRocketPowered ← 新增只加不改 ▲ Duck ──HAS-A──► QuackBehavior «接口»

五、这条原则你其实天天在用

"封装变化"绝不是教科书里的玩具，它渗透在你每天调用的 JDK 与框架中。凡是你看到"传一个接口/函数进去、行为随之改变"的地方，背后都是这条原则：

场景不变的内核被封装出去的"变化"
JDK 排序 Collections.sort(list, cmp)归并/TimSort 排序算法本身Comparator——"怎么比大小"随业务变，作为参数传入
线程池 ThreadPoolExecutor取任务、调度线程的主循环RejectedExecutionHandler——队列满了怎么办（丢弃/抛异常/调用者跑）
Java 8 Runnable / Function + Lambda调用方框架（如 stream().map()）Lambda 就是"把一段会变的行为当对象传进去"，是策略模式的语法糖
Spring 依赖注入业务 Service 只依赖接口具体实现（MySQLUserDao / MockUserDao）由容器注入，配置一改就换
操作系统设备驱动内核只认统一的"驱动接口"千差万别的硬件（网卡/显卡），厂商各写各的驱动去适配
与你已学知识串联： 这和面向对象的"多态"是同一枚硬币的两面——多态是机制（一个接口引用指向不同实现，运行时动态分派），封装变化是指导思想（该把哪些东西做成可多态的）。往上一层，它直接催生了后面几条原则：因为要隔离变化，所以要"针对接口编程"；因为组合比继承更能容纳变化，所以要"多用组合少用继承"；因为变化被隔离到接口后端，稳定内核不用改，所以自然满足"开闭原则"。这五条原则是一条藤上的瓜。

六、别把原则用成教条

陷阱一：过度封装 / 预测未来（YAGNI）。 "万一以后要变呢"是过度设计的头号借口。封装变化的前提是识别出真实存在或高度可能的变化，而不是给每个类都套上接口、给每个常量都做成可配置。为想象中的变化提前抽象，只会换来一堆用不上的间接层，让代码更难读。原则是"封装变化"，不是"封装一切"。
陷阱二：没找准"变化轴"。 一个类如果有多个变化原因（既会因飞行规则变、又会因计费规则变），把它们塞进同一个接口，等于没隔离。这正呼应单一职责原则（SRP）：一个类/接口应该只有一个引起它变化的原因。封装变化的第一步永远是"先看清到底是哪个维度在变"。

三种思路的最终取舍，一表看清：

维度继承覆盖接口(无默认实现)封装变化 + 组合
代码复用高（但常复用错东西）差（各实现各写一遍）高（行为对象可跨类复用）
运行时可变不可不可可热替换
新增行为动继承树，易漏改改所有实现类只加新类，不改旧类
耦合度强（编译期绑定）弱但重复弱（只依赖接口）
一句话收束： 软件唯一确定的就是"会变"。封装变化 = 找到会变的那一小块，用接口把它和稳定内核之间砌一堵墙，让"变"发生在墙的一侧、"不变"安睡在另一侧。这是《Head First》全书的第一块基石，也是从"能跑的代码"迈向"抗得住需求变更的设计"的分水岭——后续每一个设计模式，本质都是这条原则在某个具体场景下的定制化答案。
