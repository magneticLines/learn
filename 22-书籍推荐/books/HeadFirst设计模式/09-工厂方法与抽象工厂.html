工厂方法与抽象工厂：把创建对象延迟到子类

一个 new 就把你和具体实现焊死了。工厂模式做的事，就是把"造哪个对象"的决定权从高层逻辑里剥出来。

一、罪魁祸首：一个不起眼的 new

这是《Head First 设计模式》第 4 章的披萨店案例。你在写一个点单流程 orderPizza(type)：备料、烘烤、切块、装盒——这套流程是稳定的，无论哪种披萨都一样。可"到底做哪种披萨"却在不停变，于是代码长成了这样：

坏味道：高层流程里直接 new 具体类Java
Pizza orderPizza(String type) {
    Pizza pizza;
    // ↓ 这一坨 if-else 就是"会变"的部分，却和稳定流程搅在一起
    if      (type.equals("cheese")) pizza = new CheesePizza();
    else if (type.equals("veggie")) pizza = new VeggiePizza();
    else if (type.equals("clam"))   pizza = new ClamPizza();   // 每加一种，就得回来改这里

    // ↓ 这套流程永远不变
    pizza.prepare(); pizza.bake(); pizza.cut(); pizza.box();
    return pizza;
}
陷阱：把 new 当成理所当然。 new CheesePizza() 看起来人畜无害，但它是一次对具体类的硬编码依赖——你在"针对实现编程"，而不是"针对接口编程"。后果是：每当披萨种类增减，你都要回来动 orderPizza 这段本应稳定的代码。这直接违反了前面学过的开闭原则（对扩展开放、对修改关闭）。new 不是错，把一堆易变的 new 和稳定逻辑焊在一起才是错。

二、第一步：把 new 挪个窝（简单工厂）

最直觉的做法：把那坨 if-else 搬到一个专门的类里，让它专职"造披萨"。这就是简单工厂（Simple Factory）：

简单工厂：把创建逻辑收拢到一处Java
public class SimplePizzaFactory {
    public Pizza createPizza(String type) {
        if      (type.equals("cheese")) return new CheesePizza();
        else if (type.equals("veggie")) return new VeggiePizza();
        return null;
    }
}
// PizzaStore 组合一个工厂，把 new 委托出去；自己只管流程
Pizza pizza = factory.createPizza(type);
先破除一个误区：简单工厂不是 GoF 的 23 种设计模式之一。 它只是一个把创建逻辑集中管理的编程习惯。它的价值在于：改动创建逻辑时只需改一个类，且多个客户端能复用同一个工厂。但它没解决根本问题——if-else 只是搬了个家，新增产品仍要修改 createPizza，依旧不满足开闭原则。这为下一步"真正的模式"埋下伏笔。

三、工厂方法：把"造什么"交给子类拍板

披萨店要开加盟连锁了：纽约店做纽约风味，芝加哥店做芝加哥风味。点单流程要全国统一（备料/烘烤/切块/装盒不能各搞各的），但用哪种披萨料由各地分店说了算。这正是工厂方法的用武之地——把 orderPizza 里的创建那一步，抽象成一个由子类实现的方法：

工厂方法模式：createPizza 成为抽象方法Java
public abstract class PizzaStore {
    // 工厂方法：只声明"我要一个 Pizza"，把"造哪个"的决定权下放给子类
    protected abstract Pizza createPizza(String type);

    // 稳定的高层流程：依赖抽象的 createPizza，永远不用改（开闭原则）
    public final Pizza orderPizza(String type) {
        Pizza pizza = createPizza(type);   // ← 不再 new 具体类，交给子类填空
        pizza.prepare(); pizza.bake(); pizza.cut(); pizza.box();
        return pizza;
    }
}

// 纽约加盟店：只需回答一个问题——"我这儿造哪些披萨？"
public class NYPizzaStore extends PizzaStore {
    protected Pizza createPizza(String type) {   // 子类决定实例化哪个具体类
        if (type.equals("cheese")) return new NYStyleCheesePizza();
        if (type.equals("clam"))   return new NYStyleClamPizza();
        return null;
    }
}
// 用法：new NYPizzaStore().orderPizza("cheese") → 得到纽约风味芝士披萨

关键的反转在这里：父类 PizzaStore 定义了算法骨架（orderPizza 里的固定步骤），但把"造哪个对象"这一步挖了个洞，留给子类去填。这就叫"把实例化延迟到子类"——父类写代码时根本不知道会 new 出什么具体类，它只依赖抽象的 Pizza。

Creator «abstract» Product «abstract» ┌───────────────────────┐ ┌──────────────────┐ │ orderPizza(type) │──调用──────► │ prepare()/bake() │ │ createPizza(type) «抽象工厂方法» │ cut()/box() │ └───────────▲───────────┘ └────────▲─────────┘ │ 子类覆写 = 决定 new 谁 │ is-a ┌───────────┴───────────┐ │ NYPizzaStore ChicagoPizzaStore │ createPizza() ── new ──► NYStyleCheesePizza ──────────┘ createPizza() ── new ──► ChicagoStyleCheesePizza ─────┘
本质：工厂方法就是"封装变化 + 依赖倒置"的合体。 变化的"创建逻辑"被封装进 createPizza 这个可覆写的洞里；高层 orderPizza 不再依赖具体类，而是依赖抽象方法——这正是依赖倒置原则说的"依赖抽象，不依赖具体类"。同时它和模板方法模式是近亲：orderPizza 就是模板方法（定义骨架），createPizza 是被下放给子类的钩子。
依赖倒置：高层 → 抽象工厂方法 ← 子类填空   （而非 高层 → new 具体类）

四、抽象工厂：不止造一个，而是造"一整族"

披萨的问题升级了：纽约风味不只是"某种披萨"，而是一整套配料——薄脆面团 + 大蒜番茄酱 + 顶级干酪；芝加哥则是厚饼皮 + 李子番茄酱 + 马苏里拉。这些配料必须成套搭配，绝不能纽约面团配芝加哥酱。当你要创建的是"相互关联、必须配套的产品家族"时，工厂方法就不够用了，需要抽象工厂（Abstract Factory）：

抽象工厂：一个接口生产"一族"产品Java
// 抽象工厂：声明"创建一整族相关原料"的接口
public interface PizzaIngredientFactory {
    Dough  createDough();
    Sauce  createSauce();
    Cheese createCheese();
}

// 纽约这一族原料，由一个具体工厂保证彼此"配套"
public class NYPizzaIngredientFactory implements PizzaIngredientFactory {
    public Dough  createDough()  { return new ThinCrustDough(); }
    public Sauce  createSauce()  { return new MarinaraSauce(); }
    public Cheese createCheese() { return new ReggianoCheese(); }
}

// Pizza 用【组合】持有工厂，用它备齐原料——不关心具体是哪一族
public class CheesePizza extends Pizza {
    private PizzaIngredientFactory factory;
    public CheesePizza(PizzaIngredientFactory factory) { this.factory = factory; }
    public void prepare() {
        dough  = factory.createDough();    // 换一个工厂 → 整套原料全换，且保证配套
        sauce  = factory.createSauce();
        cheese = factory.createCheese();
    }
}
为什么它保证"配套"？ 因为一个具体工厂（NYPizzaIngredientFactory）的所有 createXxx 方法返回的都是同一族产品。客户端拿到工厂后，只要一路 factory.createXxx()，就不可能混入别族的零件——这是抽象工厂最核心的价值：用"选哪个工厂"这一个决定，锁定"一整套产品"的一致性。GUI 里换肤（Windows 风格按钮 + 滚动条 + 窗口 vs macOS 风格）是同一个道理。

五、工厂方法 vs 抽象工厂：一张表理清

两者名字像、都为了解耦创建，但手段和适用场景不同。最容易混的一点：工厂方法靠继承，抽象工厂靠组合。

维度工厂方法（Factory Method）抽象工厂（Abstract Factory）
意图创建一个产品，把"造哪个"延迟到子类创建一族相关/配套的产品
实现手段继承：子类覆写工厂方法组合：客户端持有一个工厂对象
核心方法数通常一个抽象方法多个（每种产品对应一个方法）
新增一个产品族加一个 PizzaStore 子类加一个工厂实现类 ✅ 满足开闭
新增一种产品改动对应工厂方法即可要改工厂接口 + 全部实现类 ⚠️ 违反开闭
关系抽象工厂内部常常用工厂方法来实现每个 createXxx——两者可以叠加使用
面试高频陷阱：抽象工厂对"新增产品种类"是封闭的。 如果哪天要给披萨加一味 createVeggies()，你得改 PizzaIngredientFactory 接口，然后所有实现类被迫跟着改——这违反开闭原则，是抽象工厂公认的软肋。所以选型口诀：产品维度稳定、只是切换整族 → 抽象工厂；产品种类会频繁增加 → 别硬套抽象工厂。

六、反例 → 正例：直接 new vs 工厂方法

Bad · 到处 new 具体类
class ReportService {
  Exporter get(String fmt) {
    if (fmt.equals("pdf"))
      return new PdfExporter();
    if (fmt.equals("excel"))
      return new ExcelExporter();
    // 新增 CSV → 回来改这里
    return null;
  }
}
// 痛点：
// · 高层逻辑耦合全部具体类
// · 每加一种格式都要改这个类
//   → 违反开闭原则
// · 单测想 mock 导出器都插不进去
Good · 工厂方法 + 子类填空
abstract class ReportService {
  abstract Exporter create();   // 洞
  void run() {                  // 稳定骨架
    Exporter e = create();
    e.export(load());
  }
}
class PdfReport extends ReportService {
  Exporter create(){ return new PdfExporter(); }
}
// 收益：
// · run() 只依赖抽象，永不改动
// · 加 CsvReport 只新增类，不改旧码
// · 测试可注入 FakeExporter 子类

七、你早就在用的工厂

工厂模式绝非教科书玩具，JDK 和 Spring 里遍地都是。看到"不 new、改调一个 getXxx()/createXxx()"的地方，八成就是它：

出处类型说明
Collection.iterator()工厂方法每个集合子类（ArrayList/HashSet）决定 new 哪种 Iterator，客户端只认 Iterator 接口
Calendar.getInstance()（静态）工厂按 Locale/时区返回 GregorianCalendar 等具体子类，调用方不感知
DocumentBuilderFactory / SAXParserFactory抽象工厂创建一整套 XML 解析相关对象，可整族替换底层实现
java.sql.Connection抽象工厂由它 createStatement()/prepareStatement() 产出同一数据库驱动族的对象
Spring FactoryBean<T>工厂方法getObject() 就是工厂方法，把复杂 Bean 的创建逻辑交给你实现（如 SqlSessionFactoryBean）
Spring BeanFactory / @Bean 方法工厂整个 IoC 容器就是个超级工厂，把 new 从业务代码里彻底赶走
别把"静态工厂方法"和"工厂方法模式"搞混（Effective Java 高频考点）。 Integer.valueOf(1)、List.of(...) 这种叫静态工厂方法，是《Effective Java》第 1 条讲的一种命名/编码技巧（比构造器可读、可缓存、可返回子类型）。而 GoF 的工厂方法模式强调的是"通过继承让子类决定实例化"——一个是语言层面的写法，一个是类结构层面的设计模式，名字像但不是一回事。
一句话收束： 工厂模式的全部意义，就是把"决定造哪个具体对象"这个易变点，从稳定的高层逻辑里剥离出来——工厂方法用继承把这个决定下放给子类，抽象工厂用组合把"一整族产品的一致性"锁进一个可替换的工厂对象。它们让高层代码只依赖抽象（依赖倒置），新增产品时只加类不改旧码（开闭原则），本质仍是第 1 章"封装变化"那条原则，在"对象创建"这个具体场景下的定制答案。
