多用组合，少用继承

继承是"生来就是"，组合是"运行时装配"——把 is-a 换成 has-a，你才能在不改动旧代码的前提下随意换零件。

一、先看继承"复用"是怎么翻车的

Head First 全书用一个"鸭子模拟器"开场。一开始很自然：所有鸭子都会 quack()（叫）、swim()（游）、display()（外观），于是抽一个父类 Duck，各种鸭子 extends Duck 复用这些方法。皆大欢喜——直到产品经理说："给鸭子加个飞。"

你把 fly() 加进 Duck 父类。看似一处修改让所有鸭子白拿了飞行能力，可灾难随之而来：橡皮鸭（RubberDuck）也"继承"了 fly()，于是橡皮鸭满屏乱飞；木头诱饵鸭（DecoyDuck）既不会飞也不会叫，你只能在每个"不该有该行为"的子类里覆盖方法把它关掉。每加一种鸭、每改一个行为，都要翻遍所有子类逐个检查——这就是为了复用而滥用继承的典型代价。

问题的根源：继承把"行为"焊死在了"类型"上。子类在编译期就把父类的全部方法一并继承下来，无法选择"我只要 swim、不要 fly"。而现实里，"会不会飞"是一种会频繁变化、且在不同子类间横向不同的行为——把它塞进继承层级，等于用一棵僵硬的树去表达一张灵活的网。
✗ Bad · 行为写进父类，靠继承传播
class Duck {
  void quack(){ ... }
  void swim(){ ... }
  void fly(){ ... }   // 后加的
}
class RubberDuck extends Duck {
  // 白拿了 fly() —— 可橡皮鸭不会飞!
  @Override void fly(){ /*啥都不做,硬关*/ }
  @Override void quack(){ squeak(); }
}
// 每加一种鸭 / 每改一个行为,
// 都要翻遍所有子类逐个覆盖或关闭
✓ Good · 行为抽成接口,让鸭子"持有"
interface FlyBehavior { void fly(); }
class FlyWithWings implements FlyBehavior{
  public void fly(){ /*真的飞*/ }
}
class FlyNoWay implements FlyBehavior {
  public void fly(){ /*不会飞,空实现*/ }
}
class Duck {
  FlyBehavior fly;   // has-a: 持有行为
  void performFly(){ fly.fly(); }
}

二、组合是什么：把 is-a 换成 has-a

继承表达的是 "is-a"（是一种）：RubberDuck 是一种 Duck。组合表达的是 "has-a"（拥有一个）：一只 Duck 拥有一个飞行行为、拥有一个叫声行为。把会变化的部分从"我天生是什么"改成"我手里拿着什么零件"，就获得了两个继承给不了的能力：

  • 运行时可替换：duck.setFly(new FlyRocketPowered()) 能在程序运行中给鸭子换上火箭动力；继承来的方法一旦编译就定死了。
  • 行为可跨类型复用：FlyWithWings 既能装在鸭子上，也能装在飞机、鸟类上，不受继承树限制。
继承(is-a): 用一棵树穷举所有组合 —— 类会爆炸 Duck ┌──────────┼───────────┬─────────────┐ 会飞会叫 会飞哑巴 不飞会叫 不飞哑巴... (每多一维行为, 叶子数就再乘一遍 → 指数膨胀) 组合(has-a): 行为拆成可插拔零件 —— 线性增长 ┌─ Duck ────────────────────────┐ │ FlyBehavior fly; ← 一个插槽 │ │ QuackBehavior quack; ← 一个插槽│ └───────────────────────────────┘ △ 装配 △ 装配 [FlyWithWings] [Quack] [FlyNoWay ] [Squeak] [FlyRocket ] [MuteQuack] 任意组合 = 插槽里换零件, 无需新增子类
继承穷举: 子类数 = 飞法 × 叫法 × 外观 …  (相乘·爆炸)
组合装配: 零件数 = 飞法 + 叫法 + 外观 …  (相加·线性)
为什么这条原则如此重要？它其实是"封装变化"和"针对接口编程"两条前置原则的合体落地：把变化的行为封装成独立的类（封装变化），让主体持有它们的接口类型（针对接口编程），持有的方式就是组合。三条原则拧成一股，直接催生了策略模式（Strategy）。

三、经典事故：为复用而继承，反被内部实现反咬

《Effective Java》第 18 条"组合优先于继承"给了一个比鸭子更扎心的例子：想给 HashSet 加一个计数功能（统计一共 add 了多少元素），最直觉的做法是继承它。结果这段代码算出来的数是真实值的两倍。

✗ Bad · 继承 HashSet,被父类内部调用坑
class CountingSet<E> extends HashSet<E>{
  int count = 0;
  @Override public boolean add(E e){
    count++;
    return super.add(e);
  }
  @Override public boolean addAll(
        Collection<? extends E> c){
    count += c.size();
    return super.addAll(c);
  }
}
// addAll 加 3 个 → count 变 6 !
// 因为 HashSet.addAll 内部又逐个调
// 了 add(), 触发被覆盖的 add,重复计数
✓ Good · 组合 + 委托(转发),黑盒复用
class CountingSet<E> implements Set<E>{
  private final Set<E> s;  // has-a
  int count = 0;
  CountingSet(Set<E> s){ this.s = s; }

  public boolean add(E e){
    count++;  return s.add(e);
  }
  public boolean addAll(
        Collection<? extends E> c){
    count += c.size();
    return s.addAll(c); // 委托,不管它内部
  }        // s 内部怎么实现,与我无关
}
脆弱基类问题（Fragile Base Class）——高频面试点。继承是白盒复用：子类必须了解并依赖父类"方法之间怎么互相调用"这种内部实现细节。HashSet.addAll 恰好用 add 实现，这是它的私有约定、文档里根本没承诺，却让子类的覆盖被"意外触发"。一旦父类某个版本改了内部实现（比如 addAll 改成不再调用 add），你的子类就会莫名其妙地坏掉——父类作者一次看似无害的重构，能同时打碎所有子类。组合是黑盒复用：只依赖对方公开的接口契约，天然免疫这类问题。

四、组合 + 委托 = 策略模式的骨架

上面 Good 版里那句 return s.addAll(c) 叫委托（delegation）：自己不干活，把请求转交给持有的对象。"组合一个接口 + 把活委托给它 + 运行时可替换实现"这套组合拳，正是策略模式的定义。回到鸭子，完整装配长这样：

Duck.java · 组合装配行为策略模式雏形
abstract class Duck {
    protected FlyBehavior   flyBehavior;   // has-a,面向接口而非具体类
    protected QuackBehavior quackBehavior;

    public void performFly()   { flyBehavior.fly();   } // 委托,不写 if/switch
    public void performQuack() { quackBehavior.quack(); }

    // 运行时热插拔:给这只鸭子换一套飞行零件
    public void setFlyBehavior(FlyBehavior fb) { this.flyBehavior = fb; }
}

class ModelDuck extends Duck {
    public ModelDuck() {
        flyBehavior   = new FlyNoWay();    // 模型鸭默认不会飞
        quackBehavior = new Quack();
    }
}

// —— 使用:运行中给模型鸭装上火箭,立刻会飞,一行 Duck 代码没改 ——
Duck model = new ModelDuck();
model.performFly();                       // 不会飞
model.setFlyBehavior(new FlyRocketPowered()); // 换零件
model.performFly();                       // 现在飞起来了!
注意这里"继承仍然存在":ModelDuck extends Duck。原则叫"少用继承"不是"禁用继承"。这里的继承用得恰当——ModelDuck 确实 is-a Duck（共享类型、共享稳定的不变部分）；而会变化的行为用组合。判断标准是：稳定的、真正的"是一种"关系用继承；易变的、"具备某种能力/零件"用组合。

五、继承 vs 组合，全面对比

维度继承（extends）组合（has-a + 委托）
语义关系is-a（是一种）has-a（拥有一个）
复用方式白盒：依赖父类内部实现细节黑盒：只依赖对方公开接口
耦合强度强，编译期绑死父子弱，可面向接口解耦
变更时机编译期静态固定，无法更换运行期可动态替换零件
封装性破坏封装，易生脆弱基类保持封装，改内部不影响外部
数量限制Java 单继承，只能一个父类可同时组合多个对象，无上限
典型代价子类爆炸、误覆盖、父类改动连坐需写转发/委托的样板代码
组合唯一的"缺点"要客观看待。组合确实要多写一层转发方法（如 Set 有十几个方法都得写 return s.xxx()）——这是它换取解耦付出的样板代码成本。但在 IDE 自动生成委托方法、以及 Lombok @Delegate、Kotlin by 关键字面前，这点成本几乎可忽略。不要因为"继承少打几个字"就选继承，那是拿长期维护性换短期省事。

六、什么时候仍然该用继承

继承不是坏东西，它的正确用途是表达稳定的类型关系与多态，而不是"顺手借用几个方法"。用一个测试来判断：

里氏替换测试（LSP）：只有当"子类在任何用到父类的地方都能无痕顶替父类、且不破坏原有行为契约"时，才用继承。经典反例——Stack extends Vector（java.util.Stack 的历史包袱）：栈本该只有 push/pop，却因继承 Vector 而暴露了 get(i)、add(i, e)、remove(i)，任何人都能从栈中间插删元素，彻底破坏了"栈"的语义。正确做法是组合一个 List，只对外暴露 push/pop。
初学者最常踩的坑：把继承当"代码复用工具"。看到 B 类需要 A 类里几个现成方法，就让 B extends A——这几乎总是错的。问自己一句："B 真的是一种 A 吗？"如果答案是"不是，我只是想用它的功能"，那就是 has-a，请用组合。Properties extends Hashtable、Stack extends Vector 都是 JDK 里"为复用而错用继承"留下的、无法回退的教训。

七、串联生态：这条原则无处不在

  • Spring 的依赖注入（DI）本质就是组合。@Autowired 把 UserService 注入 UserController，就是让 Controller 持有（has-a）一个 Service 接口的实现。因为是面向接口的组合，测试时能轻松换成 Mock，生产时能换成不同实现——这是继承给不了的灵活度。
  • Java IO 的装饰者模式（new BufferedReader(new FileReader(...))）：外层 BufferedReader 通过组合持有内层的 Reader 并委托，从而层层叠加缓冲、编码转换等能力，而不是造出 BufferedFileReader、BufferedGzipReader …… 一大堆继承子类。
  • Kotlin 语言级支持：class MyList(list: List<T>) : List<T> by list，一个 by 关键字自动生成所有委托方法——语言设计者把"组合优于继承"直接做进了语法。
  • 反向印证：Java 之所以是单继承、"多能力"要靠实现多个 interface，也是在从语言层面引导你——需要"具备多种能力"时别指望继承，用接口 + 组合。
一句话收束：继承是"编译期焊死的 is-a"，把行为绑在类型上，一改父类全家连坐、子类还会随行为维度组合爆炸；组合是"运行期可插拔的 has-a"，把易变行为封装成独立零件、由主体持有并委托，实现黑盒复用、动态替换与跨类型共享。判定口诀：真正稳定的"是一种"关系用继承求多态；"想用它的能力"就用组合。能用组合就别用继承——这正是策略模式，以及 Spring DI、Java IO 装饰者等无数设计的共同地基。
