构建器模式：分步构造复杂对象

当构造函数长到没人看得懂时，把"怎么装配"从"装配成什么"里拆出来。

一、痛点：一个失控的构造函数

想象要给电商系统建一个 User 对象：必填的只有 id 和 name，可选的却有一大串——邮箱、电话、地址、头像、生日、公司……用最朴素的构造函数写，很快就演变成灾难。

User.java坏味道：可伸缩构造函数（telescoping constructor）
public User(long id, String name) { this(id, name, null, null); }
public User(long id, String name, String email) { this(id, name, email, null); }
public User(long id, String name, String email, String phone) {
  // ...再往下还有 5 个重载，参数越叠越长
}

// 调用方：谁能一眼看出这几个参数是什么？
User u = new User(1001, "Alice", null, "138...", null, "北京", true, 0);
可伸缩构造函数的三宗罪：①可读性——new User(..., null, "138...", null, "北京", true, 0) 里一串 null 和裸值，靠 IDE 提示才知道谁是谁；②易错性——两个相邻的 String 参数写反了，编译器不报错，运行时才炸；③爆炸性——n 个可选参数最多要 2ⁿ 个重载才能覆盖所有组合，根本写不完。

二、模式定义与角色

构建器模式（Builder Pattern）属于创建型模式。GoF 的经典定义是：

「将一个复杂对象的构建过程与它的表示分离，使得同样的构建过程可以创建出不同的表示。」换句话说：把"分几步、每步做什么"（过程）交给 Builder，把"最终长什么样"（表示）交给 Product，两者解耦。

GoF 原始形态有四个角色，但在 Java 实战里最常见的是 Effective Java 提出的简化版（静态内部类 Builder）。先看角色对照：

GoF 角色职责Effective Java 简化版对应
Product被构造的复杂对象外层类 User（不可变）
Builder声明构建各部件的抽象接口常省略，直接用具体 Builder
ConcreteBuilder实现构建步骤、持有中间状态静态内部类 User.Builder
Director（指挥者）按固定顺序调用构建步骤常省略，由调用方链式调用充当
GoF 完整版（可换表示） Effective Java 简化版（最常用） Director ──构建流程──> Builder 调用方 ──链式 setter──> User.Builder △ │ ┌────────────┴──────┐ build() │ 校验 + 构造 TextBuilder HtmlBuilder ▼ （同一流程，不同产物） User（不可变）

三、正例：Effective Java 风格的静态内部类 Builder

这是面试和生产代码里的"标准答案"。核心三点：外层类字段 final、构造器私有、通过 Builder 一步步收集参数，最后 build() 里做集中校验再一次性构造。

User.java不可变对象 + 静态内部类 Builder
public final class User {
  private final long id;      // 必填
  private final String name;   // 必填
  private final String email;  // 可选
  private final String phone;  // 可选

  // 私有构造：只能被 Builder 调用，外界无法绕过
  private User(Builder b) {
    this.id = b.id; this.name = b.name;
    this.email = b.email; this.phone = b.phone;
  }

  public static class Builder {
    private final long id;       // 必填项放构造器，强制传入
    private final String name;
    private String email;         // 可选项给默认值
    private String phone;

    public Builder(long id, String name) {
      this.id = id; this.name = name;
    }
    // 每个 setter 返回 this —— 这是链式调用的关键
    public Builder email(String v) { this.email = v; return this; }
    public Builder phone(String v) { this.phone = v; return this; }

    public User build() {
      // 集中校验：跨字段的约束在这里一次性检查
      if (name == null || name.isBlank())
        throw new IllegalStateException("name 必填");
      return new User(this); // 校验通过才构造出合法对象
    }
  }
}
调用方自解释的链式调用
User u = new User.Builder(1001, "Alice")
    .email("alice@x.com")   // 每一步都写明"我在设置什么"
    .phone("138...")
    .build();               // 只需要一个可选项？中间几步随便省
为什么每个 setter 要 return this？因为返回的是 Builder 自己，下一个 .email(...) 就能继续接着调——这叫流式接口（Fluent Interface）。它把"一堆参数"变成了"一串带名字的赋值语句"，可读性直接拉满，而且可选项想设几个设几个、顺序无所谓。

四、反例 → 正例：和 JavaBean setter 方式对比

有人会说："我用无参构造 + 一堆 setter 不也能分步设置吗？"能，但那是可变对象，会引入两个隐患：对象在"设置到一半"时就能被别人读到（不一致状态），且天生非线程安全。

Bad · JavaBean setter
User u = new User();
u.setId(1001);
u.setName("Alice");
// 危险窗口：此刻 u 已"半成品"
// 别的线程读到的是不完整对象
u.setEmail("a@x.com");
// 字段非 final，随时能被改
// 也无法在"完工"时统一校验
Good · Builder
User u = new User.Builder(1001, "Alice")
    .email("a@x.com")
    .build();
// build() 之前对象根本不存在
// 一旦 build 出来即完整、不可变
// 天然线程安全，可安全共享
// 跨字段校验集中在 build() 里

核心差异在于"对象什么时候进入合法状态"：setter 方式下对象从 new 那刻起就存在，却要靠调用方"记得"把每个字段都设对；Builder 方式下真正的对象只在 build() 那一瞬诞生，且诞生即完整。

五、三种"分步构造"方案的取舍

维度可伸缩构造函数JavaBean setterBuilder
可读性差（一串裸值）好好（带字段名）
不可变 / 线程安全可以（final）否（可变）可以（final）
半成品风险无有无
可选参数处理差（组合爆炸）好好
集中校验散在各构造器难（没有终点）好（build 里）
额外开销无无多一个 Builder 对象
参数少且都必填 → 构造函数
参数多 / 多可选 / 要不可变 → Builder
高频面试陷阱：Builder ≠ 万能。它的代价是每次创建都多 new 一个 Builder 对象、代码量翻倍。若一个类只有 2~3 个参数且都必填，直接用构造函数更清爽——为它套 Builder 就是过度设计。经验阈值：可选参数达到 4 个以上，或明确要求对象不可变时，才值得上 Builder。

六、和 JDK / Spring / 生态的连线

Builder 不是教科书里的摆设，它在你天天用的库里无处不在——认出它们能让"分步构造"的直觉更牢：

  • StringBuilder —— 最亲民的例子。sb.append("a").append("b") 就是链式的分步构造，最后 toString() 相当于 build()。
  • StringBuilder 之外，Stream.Builder、Calendar.Builder（JDK 8）也是标准库的 Builder。
  • Lombok 的 @Builder 注解——一个注解自动生成上面那整套静态内部类模板，是它在项目里最常见的落地方式。
  • Spring：UriComponentsBuilder、BeanDefinitionBuilder、MockMvcRequestBuilders.get(...) 全是 Builder。
  • OkHttp / Retrofit：new OkHttpClient.Builder().connectTimeout(...).build() 是 Android/服务端网络层的标配。
建立联系：Builder 与"不可变对象"是黄金搭档。不可变对象（如 String、Integer、LocalDate）字段全 final，一旦创建不能改——这正好要求"所有字段在构造那一刻就备齐"。Builder 就是那个"备料台"：先在 Builder 这个可变的临时容器里慢慢攒齐参数，攒好了再一次性浇筑成一个不可变的成品。JDK 8 的 LocalDateTime、DateTimeFormatterBuilder 正是这套思路。

七、进阶陷阱：泛型 Builder 与继承

当 Builder 遇上继承（父类 Builder、子类 Builder），return this 会返回父类 Builder 类型，导致子类的链式调用断链。Effective Java 给的解法是递归泛型（self-type idiom）：

泛型 Builder 骨架解决继承下的链式断链
abstract class Pizza {
  // T 是"自身类型"，让 self() 返回真正的子类 Builder
  abstract static class Builder<T extends Builder<T>> {
    protected abstract T self();       // 子类返回 this
    abstract Pizza build();
    public T addTopping(String t) {
      // ...记录配料
      return self();  // 返回子类型，链式不断
    }
  }
}
更隐蔽的坑：build() 里做防御性拷贝。如果 Builder 收集的是可变集合（如 List），build() 直接把引用交给 Product，那么调用方 build() 之后继续改那个 List，就能偷偷篡改"不可变"对象。正确做法是在构造 Product 时对集合做 List.copyOf(...) 或 Collections.unmodifiableList(new ArrayList<>(...))。这也是"不可变"真正难做对的地方。

八、一句话收束

构建器模式 = 把"怎么一步步装配"和"最终装配成什么"分开。它专治可选参数多、构造函数爆炸、又想要不可变对象这三个联合症状：用一个可变的 Builder 当"备料台"分步攒参数，build() 时集中校验并一次性浇筑出完整、不可变、线程安全的成品。记住取舍——参数少且必填用构造函数，可选参数 ≥4 或要不可变才上 Builder；落地首选 Effective Java 的静态内部类写法或 Lombok 的 @Builder，别忘了对可变集合做防御性拷贝。
