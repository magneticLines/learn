针对接口编程，不针对实现编程

让调用方只认识"能做什么"（抽象），不认识"具体是谁在做"（实现）——把变化关进实现里，把稳定留给接口。

一句话先立住：这里的"接口"到底指什么

初学者最大的误解，是把这条原则读成"必须写 interface 关键字"。不是的。这里的"接口（interface）"是一个更宽泛的概念——它指超类型（supertype）：任何能对上层隐藏具体类的抽象都算，包括 Java 的 interface，也包括 abstract class，甚至一个普通父类。

原则真正的意思是：变量声明的类型、方法参数/返回值的类型，应该是抽象超类型，而不是某个具体实现类。这样调用方拿到的是"契约"，而不是"某个牌子的零件"。

声明类型(抽象·稳定)  ≠  运行类型(具体·易变)
直觉类比：墙上的插座是"接口"（220V 两孔标准），电视、充电器、台灯是"实现"。你装修布线时只针对插座标准，绝不会把某台特定电视焊死进墙里。将来换任何符合标准的电器都能插——这就是"针对接口编程"给你的自由。焊死电视，就是"针对实现编程"。

反例 → 正例：把类型写死的代价

看最经典的例子。假设有一个 Animal 超类型，子类有 Dog、Cat。

✗ 针对实现编程
Dog d = new Dog();
d.makeSound();
// 变量类型 = 具体类 Dog
// 想换成 Cat？调用点、
// 参数、返回值都得跟着改
// 编译期就把自己焊死了
✓ 针对接口编程
Animal a = new Dog();
a.makeSound();
// 变量类型 = 超类型 Animal
// 换实现只动 "new" 那一行
// 甚至交给工厂/容器去决定
// 调用点纹丝不动

差别看似只是变量声明左边那个类型，但它决定了调用方"认识"谁。左边写 Dog，你的代码就和 Dog 这个具体类耦死了；写 Animal，你只依赖那份"会叫"的契约，谁来实现都行。

陷阱 1：以为只是局部变量写死没关系？真正致命的不是局部变量，而是方法签名。若 public void feed(Dog dog) 或 public ArrayList<User> findAll() 把具体类暴露在 API 边界上，调用方就被你的实现"传染"了——将来想把返回类型从 ArrayList 换成 LinkedList 都会引发一片编译错误。参数/返回类型尽量用抽象（List 而非 ArrayList），实现类只在"创建那一刻"出现。

为什么这么设计：它到底解决了什么问题

三个字：解耦。展开说，是把系统里"会变的部分"和"不变的部分"用一条抽象缝隙隔开，让改动被局限在缝隙的一侧。

┌───────────────────────────┐ │ 高层调用方 (Client) │ 只认识抽象，稳定 └─────────────┬─────────────┘ │ 依赖 ▼ ╔═══════════════════╗ ║ «interface» ║ 契约 = 稳定的缝隙 ║ Animal ║ ║ + makeSound() ║ ╚═════════▲═════════╝ │ 实现 (implements) ┌───────────────┼───────────────┐ │ │ │ ┌────┴────┐ ┌────┴────┐ ┌────┴────┐ │ Dog │ │ Cat │ │ Duck │ 易变，随需扩展 └─────────┘ └─────────┘ └─────────┘

调用方箭头永远指向那个方框（抽象），永远不指向下面的具体类。于是：新增一个 Duck 实现，调用方一行都不用改（对扩展开放）；测试时能塞一个假的 MockAnimal，不必依赖真实实现（可测试）；实现的内部怎么改都是它自己的事（封装）。这三点，正是下一节要讲的多用组合、少用继承和后面开闭原则的地基。

Head First 的鸭子：把"变化的行为"提成接口

本书第 1 章用鸭子模拟器讲这条原则：如果把 fly() 直接写死在 Duck 里，橡皮鸭也会飞——灾难。正解是把"飞行行为"抽成一个接口 FlyBehavior，Duck 只持有这个接口类型的引用，运行时再决定注入哪种实现。

Duck.java针对 FlyBehavior 接口编程
// 1) 把"会变的行为"定义成接口（超类型）——这就是要针对的"接口"
public interface FlyBehavior {
    void fly();
}

class FlyWithWings implements FlyBehavior {
    public void fly() { System.out.println("我用翅膀飞"); }
}
class FlyNoWay implements FlyBehavior {
    public void fly() { /* 橡皮鸭：什么都不做 */ }
}

public abstract class Duck {
    // 关键：字段类型是接口 FlyBehavior，不是某个具体实现类
    protected FlyBehavior flyBehavior;

    public void performFly() {
        flyBehavior.fly();       // 委托给接口，多态分派，调用点不认识具体类
    }
    public void setFlyBehavior(FlyBehavior fb) {
        this.flyBehavior = fb;  // 运行时动态替换实现——针对实现编程做不到这点
    }
}
体会一下自由度：rubberDuck.setFlyBehavior(new FlyNoWay()) 一行就能让某只鸭子"不会飞"，而 Duck 类源码一个字都不用改。行为的变化被完全隔离在 FlyBehavior 的实现里——这正是"策略模式"，而它的骨架就是这条原则。

你天天在用它，只是没意识到：JDK 与 Spring

这条原则不是理论洁癖，标准库里到处都是它的身影。左边是"针对接口"的惯用写法，注意变量类型全是抽象。

场景针对接口（推荐）接口背后的实现可换成
集合List<T> l = new ArrayList<>();LinkedList / CopyOnWriteArrayList
映射Map<K,V> m = new HashMap<>();TreeMap / LinkedHashMap / ConcurrentHashMap
线程池ExecutorService es = Executors.newFixedThreadPool(4);任意 ExecutorService 实现
JDBCConnection c = ds.getConnection();MySQL / PostgreSQL 各自的驱动实现
日志Logger log = ...;（SLF4J 门面）Logback / Log4j2 任选其一

为什么老手一律写 List 而不是 ArrayList？因为声明成接口后，将来在方法内部把 ArrayList 换成 LinkedList，方法签名和所有调用方都不受影响。JDBC 更是教科书级示范：java.sql 只定义 Connection、Statement 等接口，各数据库厂商去实现，你的业务代码换数据库只需换驱动、改连接串。

Spring 的依赖注入，本质就是这条原则的工业化：你写 @Autowired private UserService userService;（接口类型），至于注入的是 UserServiceImpl 还是测试里的 mock，由容器在运行时决定。"针对接口编程"负责解耦，"控制反转/DI"负责由谁来 new——两者一配合，把 new 具体类这件"脏活"从业务代码里彻底赶了出去。

和"依赖倒置原则(DIP)"什么关系？（面试高频）

面试常问二者区别。它们是同一思想的不同视角，别背混：

针对接口编程依赖倒置原则 (DIP)
关注点调用方"看到"的类型是抽象模块之间的依赖方向指向抽象
一句话用抽象类型声明变量/参数/返回值高层不依赖低层，两者都依赖抽象
层次更偏编码习惯（写代码时）更偏架构方向（分层设计时）
关系DIP 是"针对接口编程"上升到模块/架构层面的推广；遵守前者是落实后者的基本手段

别矫枉过正：什么时候不该造接口

陷阱 2：为每个类都配一个接口（过度抽象）。如果一个实现永远只有一个、且看不到任何变化的迹象，硬造 XxxService + XxxServiceImpl 只会平添跳转成本和心智负担，违背 YAGNI（You Aren't Gonna Need It）。抽象是有成本的——它值不值，取决于"这里真的会变吗"。这也呼应上一节封装变化：只把真正会变的点提成接口，不变的部分保持简单。判断标准：出现第 2 个实现、或需要 mock 来测试时，再提接口也不迟。
陷阱 3：接口在，却仍在别处 new 具体类。写了 List 却在方法里到处 new ArrayList()，实现依赖并没消失，只是从"变量类型"挪到了"创建语句"。这正是工厂模式和依赖注入要接手的活：把"选哪个实现"这个决定集中到一处，其余代码彻底只面向接口。原则本身只解决"声明"，"由谁创建"要靠创建型模式补齐。
一句话收束："针对接口编程"= 让代码依赖抽象类型，而非具体类。它把"会变的实现"关进接口背后，换实现只动创建点、不动调用点，从而换来可替换、可扩展、可测试。记住三件事：① 这里的"接口"泛指超类型（含抽象类）；② 主战场是方法签名与字段类型，不只是局部变量；③ 只为真正会变的点抽象，配合工厂/DI 把 new 收拢到一处，才算真正落地。
