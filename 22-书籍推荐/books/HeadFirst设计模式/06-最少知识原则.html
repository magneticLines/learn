最少知识原则：只和你的朋友交谈

一个对象知道得越少，能拖它下水的改动就越少——减少对象之间的"认识范围"，就是减少耦合。

一句话看懂：只和"直接朋友"说话

最少知识原则（Least Knowledge Principle）在学术界更常被叫作迪米特法则（Law of Demeter, LoD），1987 年由美国东北大学 Demeter 项目组提出。名字听起来玄，核心却极朴素：

LoD 定义：一个对象应当对其它对象保持最少的了解。在一个方法里，你只应该调用"直接朋友"的方法，而不要去调用"朋友的朋友"的方法。
翻译成人话：别顺着一长串 . 去够到很远的对象——你认识的类越少，将来别人重构时能连累到你的地方就越少。

它的落脚点和前几节是一脉相承的：封装变化、针对接口编程、依赖倒置都在降低耦合，而 LoD 从"调用链"这个角度补了一刀——限制一个方法能"顺藤摸瓜"够到多少陌生对象。

a.getB().getC().doX() ❌  →  a.doX() ✅

生活类比：报童该不该翻顾客的钱包

经典例子来自 David Bock 的"报童与钱包"。报童来收报费，如果他直接从顾客兜里掏出钱包、自己数钱、抽走两块，会怎样？

违反 LoD：报童伸手翻顾客的内部结构 ┌──────────────────────────────────────────────────┐ │ paperboy → customer.getWallet().getMoney(2.0) │ │ │ │ │ │ │ └ 陌生人:Wallet 的方法 │ │ └ 朋友:Customer 返回的"内脏" │ │ 报童被迫知道了两件本不该知道的事： │ │ 1) 顾客"有钱包" 2) 钱包"怎么取钱" │ │ → 顾客改用手机支付(换掉 Wallet)，报童代码立刻崩 │ └──────────────────────────────────────────────────┘ 遵守 LoD：报童只对顾客开口 ┌──────────────────────────────────────────────────┐ │ paperboy → customer.pay(2.0) │ │ 钱怎么掏、从钱包还是从手机，是顾客自己的私事 │ │ → 报童只认识 customer 一个"朋友"，顾客随便改 │ └──────────────────────────────────────────────────┘

违反 LoD 的代码会形成一种坏味道，业界形象地叫它"火车残骸（train wreck）"——一长串 getX().getY().getZ() 像一节节车厢连着出轨。它的真正问题不是"难看"，而是把中间每一层的内部结构都变成了你的依赖：链条上任何一环换了实现，你的代码都要跟着改。

规则：谁才算你的"朋友"

Head First 给出了可操作的判定——在对象 O 的某个方法里，你只应该调用以下四类对象的方法：

#可以调用谁的方法为什么它是"朋友"
①对象本身（this）自己的方法，天然可信
②方法的参数是调用方主动交给你的，你有权用
③方法内 new 出来的对象你亲手造的，生命周期归你管
④对象的成员（组件）HAS-A 的直接持有者，是你的"家人"
唯一的红线：不要调用"其它方法返回值"上的方法。也就是说，getX() 给你返回的那个对象，是从别处"捡来的陌生人"——你可以拿着它，但不该继续对它 . 下去调它的方法。
Java四类"朋友" & 一条红线
class Station {                       // 对象 O
    private Thermometer thermometer;   // ④ 成员(组件)——朋友

    float report(Display display) {     // ② 参数 display——朋友
        selfCheck();                     // ① 调自己的方法——朋友
        Logger log = new Logger();      // ③ 自己 new 的——朋友
        log.write("reported");

        display.refresh();               // ✓ 参数的方法，合法
        return thermometer.getTemp();      // ✓ 成员的方法，合法

        // ✗ 红线：对"别人返回的对象"继续点下去
        // getCloud().getSensor().read();  // getSensor() 返回陌生人
    }
}
与"Tell, Don't Ask"互为表里：LoD 的正向说法是"命令，而不是询问"——不要问对象要出它的内部零件、再自己动手（getWallet().getMoney()），而是直接命令它做完整件事（pay(2.0)）。把"怎么做"留在数据所在的那个对象里，这也正是高内聚的体现。

坏味道对照：火车残骸 → 委托转发

用 Head First 里的气象站/温度计场景：Display 想拿到温度。

✗ BAD · 火车残骸
class Display {
  float getTemp(Station s) {
    // 顺着链条够到远处的对象
    return s.getThermometer()
            .getTemperature();
  }
}
// Display 现在"认识"了 Thermometer。
// 哪天 Station 换掉温度计的实现，
// 甚至改用远程 API 取温度——
// Display 被迫跟着改。耦合到了"孙子"辈。
✓ GOOD · 委托(delegate)
class Station {
  private Thermometer t;
  float getTemperature() {  // 门面/转发
    return t.getTemperature();
  }
}
class Display {
  float getTemp(Station s) {
    return s.getTemperature();  // 只认识 s
  }
}
// 温度从哪来，是 Station 的私事。
代价看得见：你会发现 Station 多写了一个转发方法（wrapper / delegate method）。这正是 LoD 的成本——为了不让外界够到内部，包装者要多写一层"传话"方法，类会变多、调用会多一跳。所以 LoD 是用少量样板代码换取解耦的一笔交易，不是免费午餐。

它在框架里的名字：门面模式与 Spring

把"多写转发方法来屏蔽内部"这件事系统化，就得到了门面模式（Facade）——用一个门面类把子系统里一堆零件包起来，客户端只跟门面这一个"朋友"打交道。可以说，门面模式就是 LoD 的工程化产物。

  • Spring JdbcTemplate：原生 JDBC 是典型的火车残骸——conn.createStatement().executeQuery(sql)，再从 ResultSet 一路 getXxx。JdbcTemplate 把 Connection → Statement → ResultSet 这条链全藏进内部，你只对 jdbcTemplate 一个对象说 query(...)。这既是门面，也是 LoD。
  • SLF4J 的 LoggerFactory.getLogger()：你不会去碰底下的 Logback/Log4j 的实现类，只握着 Logger 这个门面接口。
  • 操作系统：系统调用（syscall）就是内核给用户态的"门面"。应用程序不能顺着指针去访问内核的内部数据结构，只能通过 read/write/open 这组有限接口"交谈"——用户态对内核内部保持最少知识，这也是权限隔离与稳定性的基石。

陷阱：别把 LoD 用成"教条"

高频误区 & 面试追问：
  • "一个点就合法，两个点就违规"是错的。LoD 数的不是 . 的个数，而是你调用的方法属于谁。this.a.b.doX() 全是自己的成员链，并不违反；而 getX().doY() 只有一次调用却踩了红线。
  • 流式 API / Builder 不是火车残骸。StringBuilder.append(..).append(..)、Stream.filter().map().collect() 看着链很长，但它们每次返回的是同一个对象或同一"管道"自身（Fluent Interface），并没有把你引向陌生的内部结构，所以是安全的。区别在于：链条是否让你依赖了"别人的内部零件"。
  • DTO / 纯数据结构例外。LoD 主要约束有行为的对象。对付纯数据载体（如从数据库映射出来的 DTO、配置树、JSON 节点），点链取值 dto.getUser().getName() 通常是可以接受的——它本就没有要保护的行为。
  • 别为消灭点链而制造上帝对象。过度包装会让门面类堆满转发方法，退化成什么都管的"上帝类"，反而降低内聚。解耦是目的，包装只是手段。

取舍对比：遵守 LoD 的收益与成本

维度遵守最少知识原则直接暴露内部对象（点链穿透）
耦合度低：只依赖直接朋友，改动被隔离高：依赖到"孙子"辈，牵一发动全身
可维护性好：内部结构可自由重构差：任一中间层换实现就连锁改
代码量多：要写一层转发/门面方法少：直接够过去，一行搞定
运行开销略高：多一跳方法调用（通常可忽略）略低：少一层间接
适用场景有行为、需长期演进的领域对象一次性脚本、纯数据 DTO 取值
落地判据：当你写出 a.getB().getC()... 时，问自己一句——"我真的需要认识 C 吗？还是只想让 B 帮我把事办了？" 如果是后者，就在 B 上加一个方法把 C 藏起来。让每个对象都"守着自己的一亩三分地"，别越俎代庖。

小结

一图流记住最少知识原则：
  • 目标：让每个对象"认识"尽量少的其它类，从而把改动的涟漪控制在最小范围。
  • 规则：方法里只调用四类朋友——自己、参数、方法内 new 的、自己的成员；不要对"别人方法的返回值"继续点下去。
  • 手段：给持有者加转发方法 / 门面（Facade），把内部结构藏起来；配合"Tell, Don't Ask"，命令而非询问。
  • 边界：不数点的个数、不管纯数据 DTO、不误伤 Fluent API；解耦是目的，别为包装而包装。
