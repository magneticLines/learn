Spring Data MongoDB 概览（进阶）

换一个完全不同的存储引擎，代码风格却几乎不用变——这就是 Spring Data 家族"统一编程模型"想要达成的效果。

一句话讲清楚

Spring Data 不只是"Spring Data JPA"，它是一整个家族：JPA、MongoDB、Redis、Elasticsearch 等各种存储都有对应的 Spring Data 子项目，它们共享同一套编程模型——Repository 接口 + 方法命名查询 + @Query。 这意味着你学会了 Spring Data JPA 的用法，切换到 MongoDB 时学习成本极低，因为"定义接口、继承 XxxRepository、写方法名或注解查询"这套心智模型是通用的。

为什么能做到统一？Spring Data 把"仓库模式"抽象成一层通用基础设施（RepositoryFactorySupport、方法名解析器等），各个子项目只需要实现"如何把解析出来的查询条件翻译成本数据库的查询语句"这一层适配逻辑。这和 JDBC 通过统一接口屏蔽不同数据库驱动差异，是同一种"面向抽象、隔离实现"的设计思路。

MongoRepository 基本用法

MongoDB 是文档型数据库，一条记录对应一个 JSON 风格的文档（BSON），不需要预先定义表结构。

Product.java（MongoDB 文档实体）
@Document(collection = "products")  // 对应 MongoDB 的一个集合（collection），类似关系型的表
public class Product {

    @Id
    private String id;            // MongoDB 主键默认是 ObjectId 字符串

    private String name;
    private BigDecimal price;

    // 文档型数据库天然支持嵌套结构，不需要像关系型那样拆成关联表
    private List<Spec> specs;   // 比如商品规格列表，直接内嵌在文档里
}
ProductRepository.java（和 JpaRepository 用法几乎一致）
public interface ProductRepository extends MongoRepository<Product, String> {

    // 方法命名查询同样适用
    List<Product> findByPriceLessThan(BigDecimal price);

    // 也可以直接写 MongoDB 原生查询语法（JSON 风格）
    @Query("{ 'specs.color': ?0 }")
    List<Product> findByColor(String color);
}

对比 JpaRepository 的写法几乎一样：接口继承、方法命名、@Query 自定义查询，唯一的区别是 @Query 里的语法从 JPQL 换成了 MongoDB 自己的查询 JSON 语法。

什么场景适合用 MongoDB 而不是关系型数据库

维度关系型数据库（MySQL/PostgreSQL）MongoDB
Schema 固定表结构，改字段需要 ALTER TABLE 无固定 Schema，同一集合里的文档字段可以不完全一致，迭代快
层级 / 嵌套数据 需要拆成多张关联表 + JOIN 才能表达"一对多"嵌套关系 天然支持嵌套文档和数组，一次查询就能拿到完整的树状结构
事务 / 一致性 成熟的 ACID 事务，多表关联的强一致场景优势明显 单文档操作是原子的，多文档事务支持较晚加入且性能开销更大
典型场景 订单、支付、库存等要求强一致性的核心交易数据 商品详情页（规格、属性差异极大）、日志、内容管理、物联网时序数据
面试高频："什么时候该用 MongoDB？"——不要因为"NoSQL 听起来更现代"就选它。核心判断标准是数据结构是否稳定、是否有强关联事务需求。像商品详情这种"不同品类字段差异巨大、经常嵌套"的数据放 MongoDB 很合适；而订单、账户余额这类需要强一致性的核心交易数据，仍然应该优先选关系型数据库。

核心价值总结

  • Spring Data 的"统一编程模型"让你在不同存储之间迁移的学习成本大幅降低，接口定义方式高度一致。
  • MongoRepository 的用法几乎是 JpaRepository 的翻版，差异主要在 @Query 里的查询语法和实体注解（@Document 而非 @Entity）。
  • 选型的关键是数据形状和一致性要求，而不是趋势——灵活 schema、深层嵌套用 MongoDB，强一致事务用关系型。
