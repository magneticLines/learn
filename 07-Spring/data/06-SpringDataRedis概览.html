Spring Data Redis 概览（进阶）

给方法加一个注解，方法的返回值就自动被缓存了——这背后又是 AOP 代理机制的一次巧妙应用，和 @Transactional 是同一套底层逻辑。

一句话讲清楚

Spring Data Redis 提供了 RedisTemplate 这样的手动操作模板，也提供了 @Cacheable/@CacheEvict 这样的声明式缓存注解——前者是"手动挡"，后者是"自动挡"，两者底层都是通过 Redis 客户端和数据库交互，只是抽象层次不同。

RedisTemplate / StringRedisTemplate 基本用法

基本读写操作（手动挡）
@Service
@RequiredArgsConstructor
public class TokenService {

    // StringRedisTemplate 是 RedisTemplate<String, String> 的简化版，key/value 都按字符串处理
    private final StringRedisTemplate redisTemplate;

    public void saveToken(String userId, String token) {
        // 设置带过期时间的 key，30 分钟后自动失效，避免手动清理
        redisTemplate.opsForValue()
                     .set("token:" + userId, token, Duration.ofMinutes(30));
    }

    public String getToken(String userId) {
        return redisTemplate.opsForValue().get("token:" + userId);
    }
}

RedisTemplate 支持 key/value 泛型自定义，需要自己配置序列化方式（默认 JDK 序列化会把对象存成不可读的字节流，生产环境通常改成 Jackson2JsonRedisSerializer，存成可读的 JSON）；StringRedisTemplate 则默认按字符串处理，是最常用、最不容易踩序列化坑的选择。

声明式缓存：@Cacheable / @CacheEvict

如果只是想给一个查询方法加缓存，不想手写"先查缓存、没命中再查库、再写回缓存"这套模板代码，可以用声明式缓存注解。

ProductService.java（自动挡 — 声明式缓存）
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;

    // 方法结果按 key（这里是 id）缓存到名为 "products" 的缓存区
    // 第二次调用相同 id 时，方法体根本不会执行，直接从 Redis 返回结果
    @Cacheable(value = "products", key = "#id")
    public Product getProduct(Long id) {
        return productRepository.findById(id).orElseThrow();
    }

    // 更新商品后必须清掉对应缓存，否则读到的是过期数据
    @CacheEvict(value = "products", key = "#product.id")
    public void updateProduct(Product product) {
        productRepository.save(product);
    }
}
原理和 @Transactional 一模一样：容器为 ProductService 生成一个 AOP 代理，代理在方法执行前先根据 key 表达式（SpEL）算出缓存键，去 Redis 查有没有命中；命中直接返回缓存值，方法体根本不执行；没命中才放行执行方法体，并把返回值写回 Redis。这也解释了为什么 @Cacheable 同样会踩"同类自调用失效"的坑——原理和 @Transactional 完全一致，绕开代理调用，缓存逻辑同样不会生效。
生产常见坑：忘记在写操作方法上加 @CacheEvict（或 @CachePut 更新缓存），会导致数据库已经更新，但 Redis 里还是旧值——读写不同步是声明式缓存最容易踩的坑，一定要保证"改数据的方法"和"读数据的方法"缓存键（key 表达式）完全对应。

核心价值总结

  • RedisTemplate 提供细粒度控制，适合复杂数据结构操作（如排行榜用 ZSet、计数器用 incr）。
  • @Cacheable/@CacheEvict 提供声明式的开发体验，本质仍是 AOP 代理拦截方法调用，和 @Transactional 是同一套机制的不同应用场景。
  • 声明式缓存最大的坑是"读写不同步"，写操作务必配对 @CacheEvict 或 @CachePut。

与其他知识点的联系

  • AOP 代理机制：@Cacheable/@CacheEvict 和 @Transactional 走的是完全同一套底层机制——容器为目标 Bean 生成 AOP 代理，代理在方法调用前后插入缓存读写逻辑，两者本质都是"方法前后插入横切逻辑"的具体应用，只是关注点不同：一个管缓存，一个管事务。理解了 Advisor/Pointcut/Advice 的分工（详见 internals/06-AdvisorAdvicePointcut的关系.html），就能理解为什么 @Cacheable 同样会踩"同类自调用失效"这个坑——绕开代理调用，缓存逻辑同样不会触发。
  • 事务管理 @Transactional：Spring 的 @Transactional 默认只管理关系型数据库这类支持 PlatformTransactionManager 的资源，对 Redis 操作默认不纳入事务范围——Redis 自身的 MULTI/EXEC 事务模型和关系型数据库的 ACID 事务在语义上也不同（Redis 事务不支持失败后自动回滚，只是把命令打包顺序执行）。如果业务需要"数据库和 Redis 同时成功/失败"，必须显式配置 RedisTransactionManager 并接入 Spring 的事务同步机制，否则容易出现"数据库已回滚、Redis 缓存却是脏数据"的不一致，详见 data/04-事务管理Transactional.html。
  • Repository 方法命名查询：Spring Data 家族提供了 RedisRepository，同样支持类似 JPA 的方法命名查询（如 findByUsername），底层同样是运行时动态代理生成实现类；但由于 Redis 本质是键值/哈希结构而不是关系表，能表达的查询条件远比 JPQL 弱（一般依赖二级索引配置才能支持非主键字段查询），这是两种 Repository 在"约定优于配置"这同一设计思想下、因存储模型不同而产生的能力差异，详见 data/01-Repository接口与方法命名查询.html。
  • 连接池 HikariCP：RedisTemplate 内部同样依赖一个连接池（Lettuce/Jedis 各自的连接池实现）来复用与 Redis 服务器的网络连接，避免每次读写都重新建立 TCP 连接——这与 HikariCP 管理数据库连接是同一种"连接复用"思想的不同实现，只是应用场景从关系型数据库换成了 Redis。
