N+1 查询问题回顾（承接 04-工程实战）

原理已经在工程实战篇讲透了，这里不再重复"为什么会发生"，只聚焦一件事：在 Spring Data JPA 里，到底该用哪个注解去解决它。

核心结论回顾

N+1 查询问题的本质，是 ORM 把"访问关联对象属性"这个看似廉价的对象导航操作，悄悄伪装成了一次远程数据库查询——一个无害的 for 循环访问懒加载集合，会在背后触发 1 次主查询 + N 次关联查询，这正是本篇 @Entity 映射篇里提到的 FetchType.LAZY 默认行为在循环场景下的放大效应。 完整的原理分析、SQL 日志示例、四种通用解法的详细对比，已经在 04-工程实战/db/01-N+1查询问题.html 讲过，这里不再重复展开。本篇聚焦一个更具体的问题：在 Spring Data JPA 里，具体该怎么写代码解决它。

方案一：@EntityGraph（声明式，最适合复用 Repository 方法）

OrderRepository.java（@EntityGraph）
public interface OrderRepository extends JpaRepository<Order, Long> {

    // 声明这次查询要一并加载 items 关联，不用改动方法签名和返回类型
    @EntityGraph(attributePaths = {"items"})
    List<Order> findAll();
}

方案二：JOIN FETCH（JPQL 显式声明，不分页场景首选）

OrderRepository.java（JOIN FETCH）
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();

一条 SQL 用 JOIN 把关联数据一次性拉回来，DISTINCT 用于去掉一对多 JOIN 产生的重复主对象行。注意：不能和分页一起使用——Hibernate 会把全表数据拉进内存再分页，数据量大时直接 OOM。

方案三：@BatchSize（折中方案，兼容分页，推荐做全局兜底）

Order.java（@BatchSize）
@OneToMany(mappedBy = "order")
@BatchSize(size = 100)                // 关键：把 N 次单条查询合并成 IN 查询
private List<OrderItem> items = new ArrayList<>();
// 也可以在 application.yml 里配置全局默认值，兜底所有没显式处理的懒加载集合：
// spring.jpa.properties.hibernate.default_batch_fetch_size=100

不消灭懒加载本身，而是把"访问时触发的 N 条单查"合并成"1 条 IN 查询"——原本 1 + N 条 SQL 降到 1 + ⌈N/size⌉ 条，天然兼容分页，也不会产生 JOIN FETCH 的笛卡尔积问题。

三种方案取舍对比

方案SQL 数量是否兼容分页笛卡尔积适用场景
@EntityGraph 1 ❌ 失效（同 JOIN 一次性拉全部数据） ⚠️ 一对多时存在，需注意 想复用已有 Repository 方法，不改查询语句本身
JOIN FETCH 1 ❌ 失效 ⚠️ 需要 DISTINCT 确定不分页的小结果集，一次性把关联全部拉出来
@BatchSize 1 + ⌈N/size⌉ ✅ 完全兼容 ✅ 无 需要分页、或作为全局默认兜底策略，推荐优先考虑
常见坑：@EntityGraph 和 JOIN FETCH 本质都是"一次性 JOIN 拉全部数据"，一旦这条查询后面还接了分页参数（Pageable），Hibernate 就没法在 SQL 层做 LIMIT/OFFSET，只能先把全部数据加载进内存再做分页，并打印一条容易被忽视的警告日志。只要业务接口涉及分页，就应该优先用 @BatchSize，而不是 @EntityGraph/JOIN FETCH。

核心价值总结

  • N+1 的原理不再重复——它是懒加载在循环场景下的放大效应，详见 04-工程实战/db/01-N+1查询问题.html。
  • @EntityGraph 和 JOIN FETCH 适合不分页的场景，一次性把关联数据拉全，但要小心笛卡尔积和分页失效。
  • @BatchSize 是最稳妥的默认选择，兼容分页、无笛卡尔积风险，建议在项目里配置全局默认值做兜底。
  • 判断标准很简单：这个查询要不要分页？要分页选 @BatchSize，不分页且结果集小可以考虑 JOIN FETCH/@EntityGraph。
