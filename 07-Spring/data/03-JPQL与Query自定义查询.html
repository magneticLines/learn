JPQL 与 @Query 自定义查询

方法命名查询很优雅，但一旦条件超过三四个，方法名就会变成一行无法直视的天书。是时候祭出更强力的武器：直接写查询语句。

一句话讲清楚

@Query 注解允许你在 Repository 方法上直接写查询语句，绕开方法命名规则的解析，换来更强的表达能力。 默认写的是 JPQL（Java Persistence Query Language）——一种语法像 SQL、但操作对象是实体和实体属性而不是表和列的查询语言；如果需要用到数据库特有的函数或极致的性能优化，也可以加 nativeQuery = true 直接写原生 SQL。

问题：方法命名查询失控

当查询条件变多，方法命名查询会变成这样：

UserRepository.java（反例 — 方法名已经失控）
public interface UserRepository extends JpaRepository<User, Long> {

    // 5 个条件叠加，方法名已经没人愿意读第二遍了
    List<User> findByStatusAndDepartmentIdAndCreatedAtBetweenAndNicknameContainingOrderByCreatedAtDesc(
        Status status, Long deptId, LocalDateTime start, LocalDateTime end, String keyword);
}

正例：@Query 直接写 JPQL

UserRepository.java（正例 — JPQL 清晰可读）
public interface UserRepository extends JpaRepository<User, Long> {

    @Query("SELECT u FROM User u " +
           "WHERE u.status = :status AND u.department.id = :deptId " +
           "AND u.createdAt BETWEEN :start AND :end " +
           "AND u.nickname LIKE %:keyword% " +
           "ORDER BY u.createdAt DESC")
    List<User> searchUsers(
        @Param("status") Status status,
        @Param("deptId") Long deptId,
        @Param("start") LocalDateTime start,
        @Param("end") LocalDateTime end,
        @Param("keyword") String keyword);

    // 更新 / 删除语句需要额外加 @Modifying，并且通常要在事务方法里调用
    @Modifying
    @Query("UPDATE User u SET u.status = :status WHERE u.id = :id")
    int updateStatus(@Param("id") Long id, @Param("status") Status status);
}
常见坑：忘记加 @Modifying。普通 @Query 默认只能执行查询语句，如果用 UPDATE/DELETE，必须搭配 @Modifying 注解，否则会抛 InvalidDataAccessApiUsageException。另外，这类批量更新会绕过持久化上下文的一级缓存，执行后建议用 clearAutomatically = true 清空缓存，避免读到内存中的旧数据。

需要数据库特性时：原生 SQL

UserRepository.java（nativeQuery — 用到 MySQL 特有函数）
@Query(value = "SELECT * FROM t_user WHERE MATCH(nickname) AGAINST(:keyword IN BOOLEAN MODE)",
       nativeQuery = true)
List<User> fullTextSearch(@Param("keyword") String keyword);

JPQL 与 SQL 的区别

维度JPQL原生 SQL
操作对象 实体和属性（User、u.email） 表和列（t_user、email）
数据库依赖 数据库无关，由 Hibernate 翻译成目标方言 SQL 强绑定具体数据库方言（MySQL 函数换到 Oracle 可能直接报错）
关联查询写法 JOIN u.department（走对象引用，不用手写外键条件） JOIN department d ON u.dept_id = d.id（手写外键关联）
返回结果 直接映射成实体对象 需要 nativeQuery=true 手动指定返回实体，或返回 Object[]/投影接口
适用场景 绝大多数业务查询，享受面向对象和数据库无关性 复杂报表、全文检索、窗口函数等 ORM 表达力不够的场景
面试高频："JPQL 和 SQL 的本质区别是什么？"——JPQL 面向的是持久化的对象模型，你写的 SELECT u FROM User u WHERE u.department.name = 'RD' 中的 User、department、name 都是 Java 类和属性名，Hibernate 在执行前会把它们翻译成实际的表名和列名。这也是 JPQL 能做到"数据库无关"的原因——换数据库时只需要换 Hibernate 的方言配置，业务查询语句一行不用改。

核心价值总结

  • 方法命名查询处理简单条件，@Query 处理复杂查询——不是替代关系，而是互补，简单场景没必要为了"统一风格"强行写 JPQL。
  • JPQL 操作实体属性，SQL 操作表列，这是理解两者所有差异的根源。
  • 更新删除记得加 @Modifying，并且要在事务方法内调用，否则不会生效或抛异常。
  • 原生 SQL 是最后的逃生舱：用于 JPQL 表达力不够的场景，但代价是牺牲数据库无关性，谨慎使用。
