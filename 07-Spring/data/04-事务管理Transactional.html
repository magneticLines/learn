事务管理 @Transactional

一个注解就能让一段业务逻辑"要么全部成功，要么全部回滚"——但它不是魔法关键字，而是 AOP 代理机制的一次具体应用。不理解代理原理，就一定会在某天踩中"自调用失效"这个经典陷阱。

一句话讲清楚

@Transactional 本身只是一个标记注解，真正干活的是 Spring AOP 生成的代理对象：容器发现某个 Bean 的方法上有这个注解，就会在启动时为这个 Bean 生成一个代理，代理在方法执行前开启事务，方法正常返回后提交事务，方法抛出符合条件的异常时回滚事务。

这与 04-工程实战/mechanism/02-AOP.html 讲的"面向切面编程"是同一套底层机制——事务管理正是 Spring 内置的一个最典型的 AOP 切面（TransactionInterceptor）。理解了 AOP 代理的本质，@Transactional 的所有"诡异行为"都会变得可以预期。

原理走一遍：代理做了什么

① 容器启动时
  BeanPostProcessor（InfrastructureAdvisorAutoProxyCreator）扫描所有 Bean
  发现方法或类上有 @Transactional
  → 为这个 Bean 生成一个代理对象（JDK 动态代理或 CGLIB），放入容器替代原始对象

② 方法被调用时（通过代理）
  代理拦截方法调用，交给 TransactionInterceptor
  开启事务 → 绑定数据库连接到当前线程（ThreadLocal）
  → 调用真正的业务方法

③ 方法执行完毕
  正常返回 → 提交事务（connection.commit()）
  抛出 RuntimeException/Error → 回滚事务（connection.rollback()）
  → 释放数据库连接归还连接池
关键点：代理是"外面套了一层"，不是原对象本身。只有通过代理对象调用方法，事务逻辑才会被执行。如果调用没有经过代理，@Transactional 就形同虚设——这正是下面这个经典陷阱的根源。

经典陷阱：同类内部自调用导致事务失效

OrderService.java（反例 — this.otherMethod() 绕过了代理）
@Service
public class OrderService {

    public void createOrder(OrderDTO dto) {
        // 校验逻辑……
        this.saveOrder(dto);   // 危险：this 调用，没有经过代理对象！
    }

    @Transactional
    public void saveOrder(OrderDTO dto) {
        // 期望：这里出异常应该整体回滚
        // 实际：事务根本没开启，因为方法是被 this 直接调用的，而不是代理调用的
        orderRepository.save(toEntity(dto));
        inventoryService.deduct(dto);  // 如果这里抛异常，前面的 save 不会回滚！
    }
}
为什么会失效：Spring 默认用的是 AOP 代理模式而不是编译期织入（AspectJ 是另一种方式，能规避此问题但项目里很少用）。容器里放的是"代理后的 OrderService"，但方法内部写 this.saveOrder(dto) 时，this 指向的是原始对象自身，不是代理对象——调用完全没有经过 TransactionInterceptor，自然不会开启事务。这是 Spring 面试最高频的坑之一。

正例：拆分到另一个 Bean，让调用经过代理

正例 — 把带事务的方法拆到另一个 Bean
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderWriter orderWriter;  // 注入另一个 Bean

    public void createOrder(OrderDTO dto) {
        // 通过容器注入的代理对象调用，事务逻辑正常生效
        orderWriter.saveOrder(dto);
    }
}

@Service
public class OrderWriter {
    @Transactional
    public void saveOrder(OrderDTO dto) { ... }
}

如果实在不想拆 Bean，也可以注入 ApplicationContext 或开启 @EnableAspectJAutoProxy(exposeProxy = true) 后用 ((OrderService) AopContext.currentProxy()).saveOrder(dto) 拿到代理对象再调用——但这种写法侵入性强，拆分职责到独立 Bean 永远是更干净的方案。

事务传播行为：REQUIRED vs REQUIRES_NEW

传播行为（Propagation）决定"当一个事务方法调用另一个事务方法时，两者的事务边界该怎么处理"。

传播行为行为典型场景
REQUIRED（默认） 有事务就加入当前事务，没有就新建一个。子方法和外层方法共用同一个事务，任何一方回滚，整体回滚。 绝大多数业务场景——一次业务操作要么全部成功要么全部失败
REQUIRES_NEW 无论当前有没有事务，都挂起外层事务，开启一个全新的独立事务。新事务提交/回滚不受外层事务影响，反之亦然。 需要"无论外层最终成不成功，这段日志/审计记录都必须落库"的场景
REQUIRES_NEW 典型用法 — 记录操作日志不受业务回滚影响
@Service
public class AuditLogService {

    // 无论调用方所在的业务事务最终是否回滚，日志都要独立提交，落库成功
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void record(String action) {
        auditLogRepository.save(new AuditLog(action));
    }
}
底层原理：REQUIRES_NEW 会挂起（suspend）当前线程绑定的数据库连接，重新从连接池获取一个新连接开启独立事务，执行完再恢复原连接。这意味着同一个线程内会短暂持有两个数据库连接——如果连接池的 maximum-pool-size 配置得太小，高并发下容易把连接池耗尽（详见本组第 7 篇「连接池 HikariCP」）。

常见坑：rollbackFor 与受检异常

@Transactional 默认只回滚 RuntimeException 和 Error，不回滚受检异常（checked exception，如 IOException）。 如果业务方法抛出的是自定义受检异常，事务不会自动回滚，数据会"部分提交"，这是新手很容易忽略的坑。解决方式是显式声明：@Transactional(rollbackFor = Exception.class)，让所有异常都触发回滚，这也是大多数团队规约里的强制要求。

核心价值总结

  • @Transactional 的本质是 AOP 代理，理解了这一点，"自调用失效"就不再是一个诡异的黑魔法，而是代理机制的必然结果。
  • 自调用陷阱的唯一根治办法是拆分职责到独立 Bean，让方法调用必须经过容器管理的代理对象。
  • REQUIRED 是默认且最常用的选择，只有明确需要"子操作独立于外层事务成败"时才用 REQUIRES_NEW，并注意它会额外占用一个数据库连接。
  • 生产项目建议统一加 rollbackFor = Exception.class，避免受检异常导致的静默数据不一致。

与其他知识点的联系

  • AOP 的 Advisor/Advice/Pointcut：@Transactional 的事务逻辑不是靠反射"识别注解"这么简单，而是一个标准的 AOP 切面——TransactionAttributeSourceAdvisor 的 Pointcut 负责匹配"哪些方法带有 @Transactional 注解"，它的 Advice 是 TransactionInterceptor，在方法调用前后完成开启/提交/回滚事务的逻辑。也就是说事务管理本质上就是 Spring 内置的一个 Advisor，具体三者如何协作见 internals/06-AdvisorAdvicePointcut的关系.html。
  • BeanPostProcessor 扩展点：代理的生成时机由 InfrastructureAdvisorAutoProxyCreator（本质是一个 BeanPostProcessor）在 Bean 初始化完成后的 postProcessAfterInitialization 阶段触发——这解释了为什么事务代理一定包裹的是"完整初始化好"的 Bean，而不是半成品，详见 internals/04-BeanPostProcessor扩展点.html。
  • 三级缓存与循环依赖：如果一个需要事务代理的 Bean 恰好参与了循环依赖，代理就必须提前生成——三级缓存里的 ObjectFactory 在被其他 Bean 提前引用时会调用 getEarlyBeanReference()，把代理生成的时机从"初始化后"提前到"属性填充阶段"，否则最终会出现"容器里是代理对象、但循环依赖的另一方持有的是原始对象"的错乱，详见 internals/03-三级缓存解决循环依赖.html。
  • 连接池 HikariCP：事务的边界直接决定了数据库连接的占用时长——事务开始时通过 DataSourceUtils/TransactionSynchronizationManager 把一个从连接池借来的 Connection 绑定到当前线程的 ThreadLocal，事务提交或回滚之后才归还连接池；如果事务开得过大（比如在事务方法里调用了慢的外部接口），会导致连接被长时间占用，高并发下容易把连接池打满，详见 data/07-连接池HikariCP.html。
  • Repository 方法命名查询：Repository 生成的动态代理和事务代理是两层独立的代理，但它们经常作用在同一次调用链上——Service 方法上的 @Transactional 保证一次业务操作里多次 Repository 调用共用同一个连接和事务，而 Repository 代理本身只负责把方法名翻译成查询逻辑，并不关心事务边界，两者职责互补。
  • Spring Data Redis：@Transactional 默认只管理支持 PlatformTransactionManager 的资源（如 JDBC 连接），对 Redis 操作默认不生效——除非显式配置 RedisTransactionManager 并让 Redis 连接参与 Spring 的事务同步机制，否则一个方法里既操作数据库又操作 Redis，Redis 那部分不会随数据库事务一起回滚，这是实际项目中"缓存和数据库不一致"问题的一个常见根源，详见 data/06-SpringDataRedis概览.html。
