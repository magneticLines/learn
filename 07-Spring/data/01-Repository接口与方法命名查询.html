Repository 接口与方法命名查询

你只写了一个接口，一行实现代码都没写，Spring Data JPA 却能凭空变出一个能查数据库的对象——这不是魔法，是"约定优于配置"思想在数据访问层的又一次落地。

一句话讲清楚

Spring Data JPA 的核心把戏：你只需要定义一个继承 JpaRepository 的接口，Spring 会在运行时用动态代理为这个接口生成一个实现类。 方法名本身就是查询的"声明式描述"——比如 findByEmailAndStatus，Spring 通过反射解析方法名，把 findBy / And / Status 这些关键字拆出来，自动拼出对应的 JPQL 语句去执行。你不用写一行 SQL，也不用写一行实现代码。

类比自动配置：这和 Spring Boot 的自动配置（@ConditionalOnClass 探测到某个类存在就自动装配对应 Bean）是同一种设计哲学的两种表现形式——约定优于配置。自动配置是"约定包名/类名 → 自动装配 Bean"，方法命名查询是"约定方法名 → 自动生成查询逻辑"。框架帮你把重复劳动标准化，你只需要在"声明"这一层工作。

反例：传统 DAO 层的样板代码

在 Spring Data JPA 出现之前，哪怕是最简单的"按邮箱查用户"，也要手写一整套 DAO 实现。

UserDaoImpl.java（反例 — 手写实现，全是样板）
public class UserDaoImpl implements UserDao {

    private final EntityManager em;

    @Override
    public Optional<User> findByEmailAndStatus(String email, Status status) {
        // 每一个查询都要手写 JPQL、绑定参数、处理结果为空的情况
        TypedQuery<User> query = em.createQuery(
            "SELECT u FROM User u WHERE u.email = :email AND u.status = :status", User.class);
        query.setParameter("email", email);
        query.setParameter("status", status);
        try {
            return Optional.of(query.getSingleResult());
        } catch (NoResultException e) {
            return Optional.empty();
        }
    }
    // 每加一个查询条件，就要再复制一遍这套模板……
}

正例：一个接口，零实现代码

用 Spring Data JPA 重写，UserRepository 只是一个接口，不需要 implements 任何类。

UserRepository.java（正例 — 方法命名查询）
public interface UserRepository extends JpaRepository<User, Long> {

    // 方法名本身就是查询条件的声明，Spring 在运行时解析出实现
    Optional<User> findByEmailAndStatus(String email, Status status);

    List<User> findByNicknameContainingOrderByCreatedAtDesc(String keyword);

    long countByStatus(Status status);
}

就这样，接口写完，UserRepository 就能直接被 @Autowired 进 Service 使用，findByEmailAndStatus("a@x.com", ACTIVE) 直接可以调用。

Spring 在背后做了什么？（原理走一遍）

① 扫描接口（Bean 定义阶段）
  @EnableJpaRepositories（Spring Boot 自动配置里已默认开启）
  扫描所有继承 Repository / JpaRepository 的接口
  → 为每个接口注册一个 FactoryBean（负责生成代理实现）

② 生成动态代理（容器启动时）
  用 JDK 动态代理为接口生成一个实现类
  通用方法（save/findById/deleteById...）直接委托给 SimpleJpaRepository
  → 生成一个可以放进容器的 代理 Bean

③ 方法调用时（运行时解析）
  不是通用方法？交给 PartTreeQuery 解析器
  把方法名按 findBy / And / Or / OrderBy 等关键字切分成"属性树"
  → 拼出对应的 JPQL，缓存解析结果，之后直接复用
这套解析逻辑只在方法第一次被调用时执行一次（结果会被缓存成 PartTree 对象），之后每次调用都直接复用解析好的查询，并不会每次都重新拆方法名字符串，所以性能上不用担心。

方法命名查询关键字速查

关键字示例生成的语义
findBy / getBy / queryByfindByEmail按条件查询，返回结果
existsByexistsByEmail返回 boolean，只判断是否存在
countBycountByStatus返回 long，统计数量
deleteBy / removeBydeleteByStatus按条件删除
And / OrfindByEmailAndStatus组合多个条件
Between / LessThan / GreaterThanfindByAgeBetween范围比较
Like / Containing / StartingWithfindByNicknameContaining模糊匹配
OrderByfindByStatusOrderByCreatedAtDesc排序
Top / FirstfindTop10ByStatus限制返回条数
常见坑：方法命名查询的解析基于实体的属性名，不是数据库列名。如果 User 实体里字段叫 createdAt，方法必须写 OrderByCreatedAtDesc，写错大小写或用了数据库列名 create_time 会直接抛 PropertyReferenceException，而且这个异常只有在应用启动时（Spring 会提前校验方法名合法性）才会暴露，属于典型的"约定"失效场景。

概念辨析：几个 Repository 接口的关系

接口提供的能力
Repository<T, ID>顶层标记接口，本身不含任何方法，只是让 Spring Data 识别这是一个仓库
CrudRepository<T, ID>提供基础增删改查：save / findById / deleteById / findAll
PagingAndSortingRepository<T, ID>在 CRUD 基础上增加分页与排序能力
JpaRepository<T, ID>面向 JPA 的专属扩展，增加批量操作、flush()、saveAndFlush() 等，日常开发几乎都直接继承它

核心价值总结

  • 零实现代码：基础 CRUD 和大多数查询不需要写一行 SQL 或实现类，接口即文档。
  • 约定优于配置：方法名本身承载查询语义，是这一设计思想在数据访问层的具体体现，和自动配置的思路一脉相承。
  • 本质是动态代理：理解了"接口在运行时被代理成实现类"，就不会对"接口怎么能直接用"这件事感到困惑，也为理解后面 @Transactional 的代理机制打下基础。
  • 有边界：条件一复杂，方法名会变得又臭又长，这就是下一篇 @Query 要解决的问题。

与其他知识点的联系

  • 动态代理机制：Repository 接口能直接被注入使用，靠的是 Spring 在容器启动时用 JDK 动态代理为接口生成实现类——这和 internals/06-AdvisorAdvicePointcut的关系.html 里讲的 AOP 代理是同一种底层技术（运行时生成一个实现了目标接口/继承目标类的新类），但目的完全不同：Repository 代理是为了把方法名解析成查询逻辑并委托给 SimpleJpaRepository，AOP 代理是为了在方法调用前后插入事务、日志等横切逻辑。理解了"接口/类在运行时被包了一层"这个共同套路，就不会对两者的"魔法感"感到困惑。
  • 事务管理 @Transactional：Repository 方法（如 save）本身默认没有事务边界，实际项目里 Service 层方法上的 @Transactional 才是真正的事务边界——一次业务操作里调用多个 Repository 方法，必须依赖 Service 层的事务代理把它们纳入同一个数据库连接、同一个事务，否则每次 Repository 调用都可能各自开关事务，破坏原子性，详见 data/04-事务管理Transactional.html。
  • 连接池 HikariCP：不管方法名解析出的是简单查询还是复杂查询，最终执行 JPQL 都需要从连接池借一个物理连接——Repository 只是帮你省去了手写 JPQL 和绑定参数的过程，并没有绕开"连接从哪来"这个底层问题，具体如何借还连接见 data/07-连接池HikariCP.html。
  • Spring Data Redis：Spring Data 家族里的 Repository 抽象不止服务于 JPA，Spring Data Redis 也提供了 CrudRepository 风格的 RedisRepository，同样支持方法命名查询，但底层存储结构是 Redis 的 Hash 而不是关系表，查询能力也弱得多（不支持 JOIN、复杂条件组合有限），这体现了 Spring Data 用统一编程模型抹平不同存储技术差异的设计理念，具体差异见 data/06-SpringDataRedis概览.html。
