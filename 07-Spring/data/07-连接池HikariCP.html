数据访问

连接池 HikariCP

每次查询都新建一个数据库连接，就像每次打电话都要先办一张新手机卡——开销大到离谱。连接池的存在，就是为了把"办卡"这件事的成本从每次请求里挪走。

一句话讲清楚

建立一个数据库连接需要经历 TCP 三次握手、（可能的）TLS 协商、数据库身份认证等多个网络往返，耗时通常在几十到上百毫秒；连接池的思路是：启动时预先建立一批连接放着，业务代码用完即还，不用即销，从而把"建连接"这个昂贵操作从"每次请求"降低到"应用启动时"发生一次。 这和线程池"复用线程，避免每次任务都创建销毁线程"是完全相同的设计思路。

没有连接池会发生什么

每次请求都新建连接的开销
  ① TCP 三次握手 —— 客户端与数据库服务器建立网络连接
  ② （可能）TLS 握手 —— 协商加密参数，多几个网络往返
  ③ 数据库身份认证 —— 校验用户名密码、分配权限、初始化会话
  → 一套流程下来，耗时可能远超实际那条 SQL 的执行时间

连接池的做法
  应用启动时：预先建立 N 个连接，放进一个"连接池"里
  业务需要查询：从池子里借一个空闲连接，用完还回去（不真正关闭 TCP 连接）
  → 上面①②③ 这套开销只在连接创建时发生一次，之后反复复用

为什么是 HikariCP，而不是 c3p0 / Druid

Spring Boot 2.0 开始把 HikariCP 设为默认连接池（在此之前是 Tomcat JDBC Pool），原因很直接：性能测试碾压同类竞品。

连接池特点
HikariCP 字节码级优化（用 javassist 生成代理，避免反射开销）；自研 FastList 替代 ArrayList 减少边界检查；大量使用无锁/低锁并发数据结构；代码量极小（核心代码仅几千行），Bug 面小
Druid 阿里出品，监控能力最强（内置 SQL 执行统计、慢查询日志、防 SQL 注入墙），但性能略逊于 HikariCP，更适合对可观测性要求高的场景
c3p0 / DBCP 历史悠久但性能落后，锁竞争严重，高并发下吞吐量明显低于 HikariCP，目前已较少用于新项目
连接池本身是一个共享资源池，天然涉及多线程并发访问——多个业务线程同时向池子"借连接""还连接"，如果用普通的 synchronized 粗粒度加锁，高并发下会成为性能瓶颈。HikariCP 内部大量使用 ConcurrentBag（自研的无锁并发容器）来管理连接的借还，这正是并发编程里"减少锁粒度、优先使用无锁数据结构"这一设计思路的工程实践。

关键配置项与调优思路

application.yml（HikariCP 常用配置）
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        # 连接池最大连接数
      minimum-idle: 10            # 最小空闲连接数，低于此值会补充新连接
      connection-timeout: 30000   # 借连接的最长等待时间（毫秒），超时抛异常
      idle-timeout: 600000       # 空闲连接超过此时间会被回收（仅当连接数 > minimum-idle 时生效）
      max-lifetime: 1800000      # 连接最大存活时间，到期强制重建，避免数据库单方面断开的僵尸连接
配置项调优思路
maximum-pool-size 不是越大越好。HikariCP 官方建议参考公式 连接数 = ((核心数 × 2) + 有效磁盘数)，因为数据库本身的 CPU/IO 处理能力有限，连接数过多反而会因为数据库端锁竞争、上下文切换而拖慢整体吞吐。需结合数据库自身的 max_connections 上限，以及应用实例数一起评估（多个应用实例共享同一个数据库连接数上限）。
connection-timeout 业务能接受的最大排队等待时间。设太短，高峰期容易因为瞬时连接数不够而抛超时异常；设太长，会让请求积压、级联拖慢上游服务。
max-lifetime 务必设置成比数据库或网络设备（如云厂商 LB）的空闲连接超时时间更短，否则连接池以为连接还活着，实际已被数据库/网络设备单方面断开，业务代码拿到一个"死连接"直接报错。
常见坑：maximum-pool-size 设置过小，遇到本组第 4 篇讲的 REQUIRES_NEW 传播行为时容易连接池耗尽。 REQUIRES_NEW 会挂起当前连接、额外借用一个新连接开启独立事务——如果同一时刻大量请求都触发了这种"双连接占用"，实际消耗的连接数会是平时的两倍，连接池配置偏小时很容易被打满，后续请求全部因为 connection-timeout 超时而失败。

核心价值总结

  • 连接池存在的意义：把"建立连接"这个高开销操作从"每次请求"降低到"预先建立、重复复用"，本质和线程池是同一种资源复用思想。
  • HikariCP 成为默认选择的原因是性能：字节码优化、无锁并发数据结构、代码精简，在几乎所有 benchmark 中领先同类竞品。
  • 连接池是共享资源，涉及并发安全：这也是为什么 HikariCP 内部要用无锁数据结构（ConcurrentBag）来管理连接借还。
  • 调优核心不是"调大池子"，而是匹配数据库真实承载能力，同时要留意 REQUIRES_NEW 等会额外占用连接的场景。

与其他知识点的联系

  • 事务管理 @Transactional：事务的边界本质上就是"一个连接被占用多长时间"——事务开始时通过 DataSourceUtils 从 HikariCP 借出一个连接并绑定到当前线程的 ThreadLocal，事务提交或回滚之后才归还连接池，所以事务方法写得越"重"（比如内部调用慢的外部 HTTP 接口），连接被占用的时间就越长，高并发下更容易把 maximum-pool-size 打满，详见 data/04-事务管理Transactional.html。
  • REQUIRES_NEW 传播行为：这是连接池被过度消耗的一个典型触发点——REQUIRES_NEW 会挂起当前事务、额外借用一个新连接开启独立事务，意味着同一个线程在这段时间里同时占用两个连接。如果大量请求同时触发这种"双连接占用"，实际连接消耗会是平时的两倍，是连接池耗尽最常见的隐藏原因之一。
  • Repository 方法命名查询：不管查询是手写 JPQL 还是通过方法命名自动生成，最终执行时都要从 HikariCP 借一个物理连接——Repository 层的动态代理只是省去了写实现代码的过程，并没有改变"连接从连接池借出、用完归还"这个底层执行模型，详见 data/01-Repository接口与方法命名查询.html。
  • Spring Data Redis：Redis 客户端（Lettuce/Jedis）也有自己的连接池概念，作用和 HikariCP 完全类似——把"建立到 Redis 服务器的网络连接"这个开销从每次读写降低到预先建立、重复复用，只是管理的资源从数据库连接换成了 Redis 连接，两者是同一种资源复用思想在不同中间件上的落地。
