@Entity 映射与关系（一对多 / 多对多）

Java 对象和数据库表本是两种不同的世界模型——JPA 用一组注解把它们缝在一起，但这条缝合线也是新手踩坑最多的地方，尤其是懒加载。

一句话讲清楚

@Entity 系列注解的本质，是告诉 JPA 提供商（Hibernate）如何把一个 Java 对象的字段和方法，翻译成数据库表的列和外键约束。 关系映射（@OneToMany/@ManyToOne/@ManyToMany）解决的是"对象引用"和"外键关联"之间的转换问题——Java 里用一个字段引用另一个对象，数据库里只能用外键 ID。

基础映射注解

User.java（基础实体映射）
@Entity                                   // 声明这是一个 JPA 管理的实体
@Table(name = "t_user")                   // 指定映射的表名，不写则默认用类名
public class User {

    @Id                                     // 声明主键字段
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // 主键由数据库自增生成
    private Long id;

    @Column(name = "email", nullable = false, unique = true)
    private String email;
}
面试高频：GenerationType 有 IDENTITY（依赖数据库自增列，简单但不能批量插入优化）、SEQUENCE（依赖数据库序列，可以预先批量拿号，性能更好，但 MySQL 不支持）、AUTO（交给 Hibernate 自行判断）几种策略，实际项目中 MySQL 常用 IDENTITY，PostgreSQL/Oracle 常用 SEQUENCE。

关系映射：一对多与双向维护

假设一个订单（Order）包含多个订单明细（OrderItem），这是最常见的一对多关系。

Order.java / OrderItem.java（一对多双向关联）
@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;

    // "一"的一方：mappedBy 指向对方维护关系的字段名，自己不建外键
    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    private List<OrderItem> items = new ArrayList<>();
}

@Entity
public class OrderItem {
    @Id @GeneratedValue
    private Long id;

    // "多"的一方：真正拥有外键 order_id，是关系的维护方（owning side）
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;
}
为什么需要 mappedBy？数据库里外键只能存在于一张表（order_item.order_id），但 Java 对象里 Order 和 OrderItem 都可以互相持有引用。JPA 必须知道"谁才是关系的真正拥有者（owning side）"——mappedBy 标在非拥有方（Order）上，告诉 JPA："我这边的 items 字段只是镜像，真正的外键由 OrderItem.order 字段维护，别给我建表/建列。"如果两边都不写 mappedBy，Hibernate 会误以为这是两个独立的单向关系，多建出一张不必要的中间表。

多对多关系

Student.java / Course.java（多对多，经中间表）
@Entity
public class Student {
    @Id @GeneratedValue
    private Long id;

    @ManyToMany
    @JoinTable(
        name = "student_course",                     // 中间表名
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses = new HashSet<>();
}

多对多关系底层必须依赖一张中间表存储两边的外键对；Course 一侧同样用 @ManyToMany(mappedBy = "courses") 声明为非拥有方，避免重复建表。

陷阱：懒加载与 LazyInitializationException

@OneToMany/@ManyToMany 的默认 FetchType 是 LAZY（懒加载），@ManyToOne/@OneToOne 默认是 EAGER（立即加载）。懒加载字段在真正被访问的那一刻才会触发 SQL 查询——这依赖一个前提：当前必须处于 Hibernate Session（持久化上下文）打开的状态。

OrderController.java（反例 — Controller 层访问懒加载集合）
@RestController
public class OrderController {

    private final OrderRepository orderRepository;

    @GetMapping("/orders/{id}")
    public Order getOrder(@PathVariable Long id) {
        Order order = orderRepository.findById(id).orElseThrow();
        // Repository 方法返回后事务已提交、Session 已关闭
        // 这里访问 items 会触发懒加载，但 Session 早就没了 → 抛异常！
        order.getItems().size();
        return order;  // 序列化成 JSON 时同样会访问 items，同样报错
    }
}
新手高频坑：LazyInitializationException: could not initialize proxy - no Session。 根本原因是：Order 从 Repository 查出来时，事务方法一结束（默认没开事务的话 Session 也随之关闭），items 字段还只是一个"占位代理对象"，没有真正查过数据库。之后在 Service 之外（Controller、返回给前端做 JSON 序列化时）再访问 items，Hibernate 想去数据库补查，却发现 Session 已经关闭，直接抛异常。

正例：在 Service 层内完成懒加载访问

OrderService.java（正例 — 事务边界内完成访问 + 转换 DTO）
@Service
public class OrderService {

    private final OrderRepository orderRepository;

    @Transactional(readOnly = true)  // 保证方法执行期间 Session 一直开着
    public OrderDTO getOrder(Long id) {
        Order order = orderRepository.findById(id).orElseThrow();
        // 在事务边界内访问懒加载集合，此时 Session 还开着，能正常查询
        return OrderDTO.from(order);  // 转换成 DTO 后再返回给外层，彻底摆脱懒加载代理
    }
}

这也是为什么生产项目里几乎不会直接把 @Entity 对象序列化给前端——除了懒加载问题，DTO 转换还能避免暴露内部字段、循环引用等一系列问题。

EAGER vs LAZY 对比

策略默认适用注解行为风险
FetchType.EAGER @ManyToOne / @OneToOne 默认 查主对象时立即 JOIN 查出关联对象 用不到也会查，容易引发隐藏的 N+1（多层 EAGER 嵌套时）
FetchType.LAZY @OneToMany / @ManyToMany 默认 返回代理对象，真正访问字段时才查库 访问时机脱离 Session 生命周期会抛 LazyInitializationException
工程实践建议：把所有关联都显式声明为 LAZY（包括 @ManyToOne），需要关联数据时用 JOIN FETCH 或 @EntityGraph 显式指定要加载什么——这样查询行为完全可预期，不会因为默认的 EAGER 在你没注意的地方悄悄拖慢接口。这个话题会在本组第 8 篇「N+1 查询问题回顾」里进一步展开。

核心价值总结

  • 基础映射注解解决的是"字段 ↔ 列"的对应关系，@GeneratedValue 的策略选择要结合具体数据库。
  • 关系映射的核心是找准"拥有方"：谁持有外键字段，谁就不用写 mappedBy；非拥有方靠 mappedBy 声明"我只是镜像"。
  • 懒加载不是 bug，是设计：它避免了不必要的查询开销，但要求你必须在 Session 存活期间（通常是 @Transactional 方法内）完成所有懒加载字段的访问。
  • DTO 转换是最稳妥的解法：在 Service 层把实体转换成 DTO 再返回，从根源上避免懒加载异常和序列化陷阱。
