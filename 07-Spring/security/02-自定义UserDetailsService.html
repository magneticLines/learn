自定义 UserDetailsService

上一篇的内存用户只能应付本地测试——生产环境的用户信息必须来自数据库，UserDetailsService 就是"去哪查用户"这一步的可插拔接口。

一句话讲清楚

UserDetailsService 是 Spring Security 认证流程里唯一负责"根据用户名去查用户"的接口， 它只有一个方法：UserDetails loadUserByUsername(String username)。 默认实现（InMemoryUserDetailsManager）把用户写死在内存里，生产环境要自己实现这个接口，从数据库查出用户名、密码哈希、角色列表，包装成 UserDetails 交还给框架。

这一步只负责"查到用户是谁、他有什么权限"，不负责比对密码是否正确——密码比对是 DaoAuthenticationProvider 拿到 UserDetails 后，配合下一篇要讲的 PasswordEncoder 完成的。把"查用户"和"验密码"拆成两个独立职责，正是接口抽象的意义：换数据源不影响密码校验逻辑，反之亦然。

UserDetails 接口封装了什么

loadUserByUsername 返回的 UserDetails 不是简单的用户 DTO，而是 Spring Security 内部认证流程需要的"标准用户视图"，主要包含四类信息：

方法含义典型用途
getUsername()用户名身份标识，和请求携带的用户名比对
getPassword()密码哈希（不是明文）交给 PasswordEncoder 比对
getAuthorities()权限 / 角色集合后续 hasRole/hasAuthority 授权判断的数据来源
isAccountNonExpired() 等四个布尔标志位账号是否过期 / 锁定 / 凭证过期 / 是否启用账号封禁、密码过期强制改密等业务场景

正例：从数据库查询用户

CustomUserDetailsService.java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    // 复用自己的用户仓储，和业务代码共享同一张 user 表
    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        AppUser user = userRepository.findByUsername(username)
                .orElseThrow(() ->
                    new UsernameNotFoundException("用户不存在: " + username));

        // 把角色转换成 Spring Security 认识的 GrantedAuthority
        List<GrantedAuthority> authorities = user.getRoles().stream()
                .map(role -> new SimpleGrantedAuthority("ROLE_" + role))
                .collect(Collectors.toList());

        // org.springframework.security.core.userdetails.User 是官方内置的 UserDetails 实现
        return new org.springframework.security.core.userdetails.User(
                user.getUsername(),
                user.getPasswordHash(),       // 数据库里存的是哈希，不是明文
                user.isEnabled(),            // enabled
                true,                            // accountNonExpired
                true,                            // credentialsNonExpired
                !user.isLocked(),          // accountNonLocked
                authorities);
    }
}

只要容器里存在一个实现了 UserDetailsService 的 @Service Bean，Spring Boot 的自动配置就会检测到它， 不再生成上一篇讲的那个内存随机密码用户——这同样是 @ConditionalOnMissingBean 在起作用。

原理走一遍：认证流程里它处在哪一步

① 用户提交用户名密码
  封装成 UsernamePasswordAuthenticationToken

② AuthenticationManager 委派给 DaoAuthenticationProvider
  调用 UserDetailsService.loadUserByUsername(username)
  → 只根据用户名查库，此时还没碰密码校验

③ 拿到 UserDetails 后校验密码
  PasswordEncoder.matches(用户输入的密码, UserDetails.getPassword())
  → 匹配成功才生成已认证的 Authentication 对象
常见误区：不少新手会在 loadUserByUsername 里顺手做密码比对，然后自己抛异常判断"密码错误"。 这违背了接口的单一职责——密码校验应该完全交给 PasswordEncoder，UserDetailsService 只管"查到用户就返回，查不到就抛 UsernameNotFoundException"。
面试高频："UserDetailsService 和 UserDetailsManager 的区别？" —— UserDetailsManager 是 UserDetailsService 的子接口，多了 createUser/updateUser/deleteUser 等写操作，InMemoryUserDetailsManager 就是它的一个实现。生产环境自定义时通常只需要实现只读的 UserDetailsService，用户的增删改由自己的业务代码管理。

核心价值总结

  • 职责单一：UserDetailsService 只负责"根据用户名查用户"，密码比对是下一层的事。
  • 可替换：换成从数据库、LDAP、第三方系统查用户，只需重新实现这一个接口，认证流程的其余部分完全不用动。
  • UserDetails 是标准契约：用户名、密码哈希、权限集合、四个状态标志位，构成了框架能理解的"用户视图"。

与其他知识点的联系

  • Spring Security 自动配置默认行为（上一篇）：本篇实现的 CustomUserDetailsService 一旦注册为 Bean，就会让上一篇讲的"内存随机密码用户"兜底配置失效——同样是 @ConditionalOnMissingBean 在起作用，说明整套自动配置体系是靠"有没有你自己的 Bean"这一个统一规则串起来的，不是零散的特例。
  • 密码编码 PasswordEncoder（下一篇）：本篇反复强调 UserDetailsService 只管查用户、不管验密码，这个职责边界正是为了让 PasswordEncoder 能独立演进——DaoAuthenticationProvider 拿到 UserDetails 后单独调用 PasswordEncoder.matches()，两个接口各自替换互不影响，这是"单一职责"在框架设计里的具体体现。
  • 方法级安全 @PreAuthorize：本篇 loadUserByUsername 返回的 UserDetails 会被塞进 Authentication.getPrincipal()，之后 @PreAuthorize("authentication.principal.id == #userId") 这类表达式才能直接引用到它——如果自定义的 UserDetails 实现不暴露业务字段（比如用户主键），方法级安全的很多精细判断根本没法写。
  • 无状态会话与 JWT 整合：JWT 架构下依然要用到 UserDetailsService——JwtAuthenticationFilter 解析出 Token 里的用户名后，还是要调用 loadUserByUsername 把用户名还原成完整的 UserDetails（带上权限集合），才能构造出框架认识的 Authentication 对象，说明这个接口不只服务于 Session 认证，是整个认证体系通用的"查用户"入口。
  • DispatcherServlet 请求处理流程：UserDetailsService.loadUserByUsername 的调用发生在 Security Filter 链内部，早于请求真正到达 DispatcherServlet——这意味着 Controller 方法执行时，认证早已完成，Authentication 已经可以通过 SecurityContextHolder 直接取到，不需要再查一次数据库。
