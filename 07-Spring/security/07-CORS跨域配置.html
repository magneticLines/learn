CORS 跨域配置

前端 localhost:5173 调用后端 localhost:8080 的接口，浏览器控制台报一片红色的 CORS 错误——这是前后端分离项目几乎人人都会遇到的第一个坑。

先搞清楚：什么是"同源"

浏览器的同源策略（Same-Origin Policy）规定：只有协议、域名、端口三者完全相同，才算"同源"。 http://localhost:5173 和 http://localhost:8080 域名相同、协议相同，但端口不同——只要有一项不同，就是跨域。 浏览器默认禁止网页脚本读取跨域请求返回的数据，这是浏览器自己内置的安全策略，不需要任何服务端配合就会生效。

常见误解纠正："跨域"不是一种安全漏洞，恰恰相反，它是浏览器主动做的一种保护机制，目的是防止恶意网站的脚本偷偷用你的登录态去读取另一个网站的数据（比如你登录了银行网站后，打开一个恶意页面，这个页面偷偷用 fetch 请求银行接口窃取你的余额信息）。CORS 配置的作用不是"消除风险"，而是"服务端主动告诉浏览器：这个特定的源，我允许你放行"。

一句话讲清楚

CORS（Cross-Origin Resource Sharing，跨域资源共享）的本质是服务端在 HTTP 响应头里显式"授权"哪些源可以访问自己—— 比如响应头里带上 Access-Control-Allow-Origin: http://localhost:5173，浏览器看到这个头，才会把响应结果交给前端 JS 代码，否则直接在浏览器层面拦截，请求可能已经真正发到了服务器（服务器该执行的逻辑已经执行了），只是浏览器不把结果给前端脚本看。

预检请求（Preflight）是怎么回事

对于"非简单请求"（比如带自定义请求头 Authorization、Content-Type 是 application/json、或者用了 PUT/DELETE 方法），浏览器会先自动发送一个 OPTIONS 方法的"预检请求"， 询问服务端"我接下来要发一个这样的请求，你允许吗"，服务端在预检响应里明确允许了，浏览器才会真正发出业务请求。这也是为什么跨域配置不当时，Network 面板里能看到一个先失败的 OPTIONS 请求。

正例：全局 CorsConfigurationSource 配置

CorsConfig.java
@Configuration
public class CorsConfig {

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();

        // 只允许指定的前端域名访问，生产环境切忌写 "*"
        config.setAllowedOrigins(List.of("http://localhost:5173"));
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
        // 允许前端携带 Authorization 头（JWT 就放在这里）
        config.setAllowedHeaders(List.of("Authorization", "Content-Type"));
        // 是否允许携带 Cookie（跨域场景下 Cookie 认证要额外开启，见下一篇 CSRF）
        config.setAllowCredentials(true);
        // 预检请求结果缓存时间，减少重复的 OPTIONS 请求
        config.setMaxAge(3600L);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);  // 对所有路径生效
        return source;
    }
}

光定义这个 Bean 还不够，还需要在 SecurityFilterChain 里显式启用它，否则 Spring Security 的过滤器链会先于 CORS 处理拦截请求：

SecurityConfig.java（接入 CORS）
http.cors(Customizer.withDefaults())  // 使用容器里已注册的 CorsConfigurationSource Bean
    .authorizeHttpRequests(auth -> ...);
带自定义头的跨域请求完整流程
浏览器发现是跨域 + 非简单请求 → 先自动发 OPTIONS 预检请求
→ 服务端 CorsFilter 检查 Origin 是否在白名单 → 是：响应头带上 Access-Control-Allow-*
→ 浏览器看到预检通过 → 才真正发出 GET/POST 等业务请求，带上 Authorization 头
→ 服务端正常处理业务逻辑，响应头依然带上 Access-Control-Allow-Origin
→ 浏览器允许前端 JS 读取响应内容
生产环境常见坑：
  • setAllowedOrigins(List.of("*")) 配合 setAllowCredentials(true) 会直接报错——规范禁止"允许任意源 + 允许携带凭证"这种危险组合同时出现，必须精确列出允许的源。
  • 只在 Controller 上加了 @CrossOrigin 注解，却忘了 Spring Security 的过滤器链是在 DispatcherServlet 之前执行的，请求可能在到达 Controller 之前就被安全过滤器拦下，@CrossOrigin 根本来不及生效——正确做法是像上面这样在 SecurityFilterChain 里统一配置。
  • 本地开发图省事把 allowedOrigins 写死成生产域名，导致本地联调一直失败。
面试高频："CORS 报错说明请求没有发到服务器吗？" —— 不一定。对于"简单请求"（比如没有自定义头的 GET），浏览器会先把请求正常发出去，服务器该处理的逻辑（比如写入数据库）已经真实执行了，只是浏览器在收到响应后，发现没有 Access-Control-Allow-Origin 头，选择不把响应内容交给前端脚本。这也是为什么"看起来请求失败了，但数据库里数据却真的多了一条"的现象会发生。

核心价值总结

  • 同源策略是浏览器的保护机制，不是漏洞：CORS 配置是服务端主动开的"白名单"，不是在"修复"什么安全问题。
  • 本质是响应头授权：Access-Control-Allow-Origin 等响应头才是 CORS 真正生效的载体。
  • Spring Security 场景下要单独启用：光定义 CorsConfigurationSource Bean 不够，还要在 SecurityFilterChain 里调用 .cors()，否则会被安全过滤器提前拦截。
