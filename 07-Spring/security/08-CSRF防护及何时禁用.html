CSRF 防护及何时禁用

上一篇整合 JWT 时代码里悄悄写了一行 csrf(AbstractHttpConfigurer::disable)——这一篇讲清楚 CSRF 防护是什么，以及为什么在 JWT 架构下关掉它反而是对的。

攻击原理：借用户的手打枪

CSRF（Cross-Site Request Forgery，跨站请求伪造）的核心利用点是：浏览器发请求时会自动携带目标网站的 Cookie，而攻击者根本不需要知道这个 Cookie 的内容，只需要"诱导"用户的浏览器发出请求即可。

典型攻击场景
① 用户正常登录了 bank.com，浏览器保存了登录 Cookie
② 用户在没有登出的情况下，访问了攻击者精心构造的恶意页面 evil.com
③ evil.com 页面里藏着一个自动提交的表单：
  <form action="https://bank.com/transfer" method="POST">
    <input name="to" value="attacker-account">
    <input name="amount" value="10000">
  </form><script>document.forms[0].submit()</script>
④ 浏览器提交这个表单时，自动带上了 bank.com 的登录 Cookie
⑤ bank.com 服务端一看 Cookie 合法，以为是用户本人发起的转账请求，直接执行

整个过程中，攻击者完全不需要偷到用户的 Cookie，只是"借用"了浏览器自动携带 Cookie 这个机制，伪造了一个看起来"来自合法用户"的请求。

Spring Security 默认的 CSRF Token 机制

Spring Security 默认对所有非 GET 请求（POST/PUT/DELETE 等）开启 CSRF 防护：服务端生成一个随机的 CSRF Token， 要求每次提交表单/发起写请求时都必须在请求参数或请求头里带上这个 Token。因为这个 Token 存在页面里（不是 Cookie），恶意网站的表单没有渠道能拿到它，构造出来的伪造请求会因为缺少合法 Token 被服务端拒绝。

一句话讲清楚：为什么 JWT 架构下通常可以禁用 CSRF

CSRF 攻击的前提是"浏览器自动携带凭证"，而这个前提对 JWT 不成立。 JWT 通常存放在前端的内存变量或 localStorage 里，每次请求需要前端 JS 显式把它放进 Authorization 请求头——浏览器不会像对待 Cookie 那样自动附加这个头。 攻击者构造的恶意页面即使能诱导用户浏览器发出请求，也没办法让浏览器"自动"带上存在别的网站 localStorage 里的 JWT（何况跨域也读不到），所以这类请求到达服务端时根本没有合法的 Authorization 头，会被认证层直接拒绝，天然免疫 CSRF。

SecurityConfig.java（纯 JWT 架构下禁用 CSRF）
http
    .csrf(AbstractHttpConfigurer::disable)  // JWT 不走 Cookie，天然免疫 CSRF，无需再加一层防护
    .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
关键前提，容易被忽略：上面这个结论只在"纯前后端分离 + JWT 放在请求头"的架构下成立。 如果项目里用 Cookie 存储 Session ID（传统的服务端渲染，或者把 JWT 存进 HttpOnly Cookie 里给浏览器自动携带）， 那么浏览器一样会自动带上这个 Cookie，CSRF 攻击的前提重新成立，这种情况下必须保留 CSRF 防护，禁用会直接留下真实漏洞。
架构凭证存储方式浏览器是否自动携带CSRF 要不要开启
传统服务端渲染 Session ID 存 Cookie 是（浏览器自动带） 必须开启
前后端分离 + JWT 存请求头 JWT 存内存/localStorage，手动放进 Authorization 头 否（需要 JS 显式设置） 可以安全禁用
前后端分离 + JWT 存 HttpOnly Cookie JWT 存 Cookie（为了防 XSS 读取） 是（浏览器自动带） 必须开启（或用 SameSite=Strict 等额外手段替代）
面试高频："前后端分离项目为什么可以关闭 CSRF？" —— 不要死记"前后端分离就可以关"，正确的回答逻辑是：判断依据是"凭证是否被浏览器自动携带"，如果 Token 存在需要 JS 显式处理的地方（请求头/内存），CSRF 的攻击前提就不成立；反之只要用了 Cookie 自动携带凭证，不管是不是前后端分离架构，CSRF 防护都不能省。这也侧面呼应了 04-工程实战/auth 里讲过的 AAA 基础——认证凭证的存储方式，直接决定了要防御哪一类攻击。

核心价值总结

  • CSRF 利用的是"自动携带"：攻击者不需要偷到 Cookie，只需要借用浏览器自动附带 Cookie 这个默认行为。
  • Token 机制的本质：要求请求携带一个恶意页面拿不到的额外凭证，打破"仅凭 Cookie 就能伪造合法请求"的前提。
  • 禁用与否看凭证存储方式，不看是否前后端分离：JWT 放请求头可以安全禁用；一旦凭证进了 Cookie，不管什么架构都必须开启（或采取等效防护）。

与其他知识点的联系

  • 无状态会话与 JWT 整合：本篇的结论直接建立在 JWT 篇的架构选择之上——正是因为那一篇选择把 JWT 放进 Authorization 请求头而不是 Cookie，浏览器"自动携带凭证"这个 CSRF 攻击的必要前提才不成立；如果哪天改成把 JWT 存进 Cookie（哪怕是 HttpOnly 防 XSS），CSRF 防护立刻要重新打开，两篇的配置是联动的，不能孤立看待某一行 .disable()。
  • CORS 跨域配置（上一篇）：CorsConfiguration.setAllowCredentials(true) 允许跨域请求携带 Cookie，这一开关和本篇"判断 CSRF 能否禁用"用的是同一个依据——凭证是否走 Cookie 通道；两篇合起来才能回答"前后端分离到底要不要防 CSRF"这个高频面试题，单看任何一篇都只是半个答案。
  • 密码编码 PasswordEncoder：CSRF Token 和密码哈希是同一种防御哲学的两次应用——都是"给攻击者设一个拿不到/算不出的门槛"，密码哈希防的是拖库后反查原文，CSRF Token 防的是伪造看起来合法的请求，本质都不是消灭攻击面，而是把攻击成本抬到不可行的程度。
  • DispatcherServlet 请求处理流程：CSRF 校验和 CORS 处理一样发生在 Security Filter 链、也就是 DispatcherServlet 之前——一个缺少合法 Token 的写请求，会在到达 Controller 之前就被拦下并返回 403，Controller 方法体完全不会被执行，这也是为什么关闭/开启 CSRF 这行配置要写在 SecurityFilterChain 里而不是业务代码里。
