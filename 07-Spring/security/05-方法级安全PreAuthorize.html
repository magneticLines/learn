方法级安全 @PreAuthorize（进阶）

URL 级别的角色控制只能挡住"整个接口"，挡不住"同一个接口里，用户只能改自己的订单，不能改别人的"——这需要精确到参数级别的判断。

一句话讲清楚

@PreAuthorize 是标注在 Service 方法（也可以是 Controller 方法）上的注解， 在方法真正执行之前用 SpEL 表达式做一次权限判断，表达式返回 false 就直接抛出 AccessDeniedException，方法体一行代码都不会执行。 相比上一篇的 URL 级别控制，它的粒度可以精确到"某个方法 + 方法参数 + 当前登录用户"的组合判断。

先决条件：必须在配置类上加 @EnableMethodSecurity（Spring Security 6 的新注解，替代了旧版的 @EnableGlobalMethodSecurity），否则 @PreAuthorize 完全不生效，方法照常执行，这是最容易被忽略的一步。
MethodSecurityConfig.java
@Configuration
@EnableMethodSecurity  // 开启 @PreAuthorize / @PostAuthorize / @Secured 支持
public class MethodSecurityConfig {
}

正例：参数级别的权限判断

OrderService.java
@Service
public class OrderService {

    // 粗粒度：URL 级别做不到的角色判断，方法级别可以直接写
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteAnyOrder(Long orderId) { ... }

    // 细粒度：只能修改自己的订单——#userId 引用方法参数，authentication.principal 是当前登录用户
    @PreAuthorize("#userId == authentication.principal.id")
    public void updateOwnOrder(Long userId, OrderDTO dto) { ... }

    // 也可以组合：管理员放行，或者是本人操作也放行
    @PreAuthorize("hasRole('ADMIN') or #userId == authentication.principal.id")
    public Order getOrderDetail(Long userId, Long orderId) { ... }
}

authentication.principal 就是上一篇讲过的 UserDetails 对象——如果你自定义的 UserDetails 实现类里暴露了 getId() 方法，SpEL 表达式就能直接调用 authentication.principal.id 拿到当前登录用户的主键。

原理走一遍：同样是 AOP 代理机制

@PreAuthorize 能生效，靠的不是什么"运行时反射扫描注解"，而是和事务管理 @Transactional 完全一样的 AOP 代理机制—— 容器给标了 @PreAuthorize 方法所在的 Bean 生成一个代理对象，方法调用先被代理对象拦截，代理逻辑里执行 SpEL 表达式判断权限，通过了才调用真正的目标方法。

外部调用 orderService.updateOwnOrder(...)
  → 实际拿到的是 代理对象，不是 OrderService 原始实例
  → 代理对象里的 MethodSecurityInterceptor 先执行
    解析 @PreAuthorize 的 SpEL 表达式
    → 通过：放行，调用真正的 updateOwnOrder()
    → 不通过：抛 AccessDeniedException，目标方法根本不会被执行
AOP 自调用失效陷阱，这里同样适用： 在 data 组讲事务管理时提到过，@Transactional 方法如果被同一个类里的另一个方法直接 this.xxx() 调用，会绕开代理直接调用原始对象，导致事务失效。 @PreAuthorize 是同一套 AOP 机制，踩的坑一模一样：
自调用导致 @PreAuthorize 失效
@Service
public class OrderService {

    @PreAuthorize("#userId == authentication.principal.id")
    public void updateOwnOrder(Long userId, OrderDTO dto) { ... }

    public void batchUpdate(List<OrderDTO> dtos) {
        for (OrderDTO dto : dtos) {
            // this.updateOwnOrder() 是类内部直接调用，绕过代理对象，@PreAuthorize 完全不会执行！
            this.updateOwnOrder(dto.getUserId(), dto);
        }
    }
}

解决方式和事务失效问题一样：把被调用方法拆到另一个 Bean 里，或者注入自身的代理对象（AopContext.currentProxy()）再调用。

维度URL 级别（authorizeHttpRequests）方法级别（@PreAuthorize）
粒度接口路径方法 + 参数 + 当前用户
判断能力只能判断角色/权限字符串SpEL 表达式，能引用方法参数和返回值
生效前提SecurityFilterChain 生效即可必须额外加 @EnableMethodSecurity
典型场景整个模块只有管理员能访问用户只能操作自己的数据
面试高频："@PreAuthorize 和 @PostAuthorize 的区别？" —— @PreAuthorize 在方法执行前判断（判断依据只能是参数），@PostAuthorize 在方法执行后判断（可以用 returnObject 引用方法的返回值做判断，比如"只有订单归属人才能看到查询结果"，但代价是即便权限不通过，方法体也已经执行完了，不适合有副作用的写操作）。

核心价值总结

  • 细粒度授权：SpEL 表达式能引用方法参数和当前登录用户，弥补 URL 级别控制做不到的"数据归属"判断。
  • 前提条件：必须加 @EnableMethodSecurity 才生效，这是最容易漏掉的一步。
  • 本质是 AOP 代理：和 @Transactional 一样存在自调用失效问题，排查思路可以完全复用。
