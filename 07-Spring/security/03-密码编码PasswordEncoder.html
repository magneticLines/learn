密码编码 PasswordEncoder

上一篇的 UserDetails 里放的是密码"哈希"而不是明文——数据库里存明文密码是所有安全事故里最不该发生、却最常发生的一种。

为什么数据库绝对不能存明文密码

历史上多次大规模数据泄露事故（撞库攻击的源头）都源于同一个原因：某个网站的数据库被拖库后，发现密码字段是明文或者用 MD5 简单加密后存储的，攻击者拿到后要么直接可用，要么用彩虹表几秒钟就能反查出原文。 更严重的是，很多用户在不同网站复用同一套密码——一个网站泄露，其他网站的账号跟着一起沦陷，这就是"撞库"。

常见错误做法：自己用 MessageDigest.getInstance("MD5") 对密码做一次哈希就存库。问题有两个：① MD5 设计初衷是校验数据完整性，运算速度极快，正好方便攻击者用 GPU 暴力破解或查彩虹表；② 没有加盐（salt），相同密码的用户会算出完全相同的哈希值，一旦某个常见密码的哈希被破解，所有用这个密码的用户全部沦陷。

一句话讲清楚

BCryptPasswordEncoder 是 Spring Security 默认推荐的密码编码器，它对密码做单向哈希 + 自动加盐： "单向"意味着不可逆——没有任何函数能把哈希值还原成原始密码；验证密码时不是"解密后比对"，而是把用户输入的密码重新按同样的算法哈希一遍，再和数据库里存的哈希做比对。

类型可逆性典型算法能不能用来存密码
对称加密 可逆（同一把密钥加密解密） AES 不行——密钥一旦泄露，所有密码全部可解密还原
非对称加密 可逆（公钥加密私钥解密） RSA 不行——同样存在私钥泄露即全盘沦陷的风险，且性能开销大
哈希 / 摘要 不可逆 BCrypt、Argon2、MD5（不推荐） BCrypt/Argon2 可以；MD5/SHA-1 不推荐（太快、易碰撞攻击）

BCrypt 为什么比 MD5 更适合存密码

  • 加盐（salt）：BCrypt 每次编码会自动生成一个随机盐值并拼接进最终结果里存储，同一个密码每次编码得到的哈希字符串都不同，彻底防住彩虹表攻击。
  • 刻意设计得慢：BCrypt 内置一个"强度因子"（cost factor），可以调节计算成本，人为拖慢哈希运算速度——对正常登录这点延迟无感（几十毫秒），但对攻击者暴力破解是致命的减速。
  • 自带版本与参数：BCrypt 输出的字符串本身就编码了算法版本、强度因子、盐值和哈希结果，例如 $2a$10$N9qo8uLOickgx2ZMRZoMy...，验证时直接从字符串里解析出这些参数，不需要额外存一列"盐值"。

正例：注册时编码、登录时比对

PasswordEncoderConfig.java
@Configuration
public class PasswordEncoderConfig {

    // 交给容器管理，全局复用同一个编码器实例
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();  // 默认强度因子 10，够用且不太慢
    }
}
UserRegisterService.java（注册：编码后存库）
@Service
@RequiredArgsConstructor
public class UserRegisterService {

    private final PasswordEncoder passwordEncoder;
    private final UserRepository userRepository;

    public void register(String username, String rawPassword) {
        // 数据库里只存 encode() 之后的哈希，rawPassword 用完即弃，绝不落库
        String hashed = passwordEncoder.encode(rawPassword);
        userRepository.save(new AppUser(username, hashed));
    }
}

登录时怎么验证？不需要你手写比对逻辑——DaoAuthenticationProvider 内部会自动调用 passwordEncoder.matches(用户输入的原始密码, 数据库里的哈希)。 matches() 的实现原理就是"把第一个参数按同样算法（含盐值）重新哈希一遍，再对比字符串是否相等"，全程不存在"解密"这个动作。

注册流程
  原始密码 "123456" → encode() → "$2a$10$...哈希结果..." → 存入数据库

登录流程
  用户输入 "123456" → matches("123456", 数据库哈希)
  → 内部：用数据库哈希里的盐值，重新对 "123456" 哈希一遍 → 比对两个哈希字符串是否相等
  → 全程原始密码只出现在内存里，从未被"解密"过，因为根本不可解
生产环境常见坑：
  • 用 NoOpPasswordEncoder（明文比对，仅用于快速原型/教学）忘记在上线前换掉。
  • 自己实现"加盐 MD5"造轮子，盐值又固定写死在代码里，等于没加盐。
  • 忘记数据库迁移/初始化脚本里插入的测试账号密码也要走 encode()，直接明文写 SQL 插入语句。
面试高频："为什么密码要用哈希而不是加密存储？" —— 加密的设计目的是"能被合法方解密拿回原文"，一旦密钥泄露就全盘沦陷；密码验证的需求其实只是"确认两次输入是否一致"，根本不需要拿回原文，哈希的不可逆性刚好精确匹配这个需求，这是"选对工具解决对的问题"的经典案例。

核心价值总结

  • 不可逆：哈希和加密的根本区别——哈希没有解密函数，泄露了哈希值也无法直接还原密码。
  • 加盐防彩虹表：BCrypt 自动加盐，相同密码每次编码结果都不同。
  • 刻意慢：强度因子让暴力破解的成本指数级上升，是"用性能换安全"的典型权衡。
  • 验证靠重算不靠解密：登录校验的本质是"重新哈希再比对字符串"，理解这一点才能真正理解为什么密码不可能被"找回"，只能"重置"。
