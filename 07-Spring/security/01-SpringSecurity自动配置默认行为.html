Spring Security 自动配置的默认行为

只加了一个依赖，什么都没配置，全部接口突然要登录——这不是 Bug，是 Spring Boot 刻意设计的"惊吓"。

一个新手常见的"惊吓时刻"

在 pom.xml 里加一行 spring-boot-starter-security，什么代码都不写，重启项目后会发生两件事：

  • 原本能直接访问的接口，现在浏览器打开全部跳转到一个 Spring 自带的登录页，或者返回 401 Unauthorized。
  • 控制台打印一行类似 Using generated security password: 8e557245-73e2-4286-969a-ff57fe326336 的日志。

默认用户名是 user，密码就是控制台打印的这一串随机 UUID，每次重启都会变。很多新手第一反应是"我是不是把配置写错了"——其实这正是框架故意设计成的默认行为。

一句话讲清楚

Spring Boot 的安全自动配置遵循"安全优先、默认拒绝"（secure by default）的设计哲学： 只要classpath 里有 Spring Security，且你没有自己定义任何 SecurityFilterChain Bean，Spring Boot 就会用最严格的兜底配置保护你的应用——所有请求都要认证，用一个内存里的随机密码用户。

这和之前 core 组讲过的自动配置原理是同一套机制：SecurityAutoConfiguration 内部的默认配置类上标了 @ConditionalOnMissingBean(SecurityFilterChain.class)——翻译过来就是"如果容器里没有别人定义的 SecurityFilterChain，我才生效"。这是典型的"约定优于配置 + 条件装配"组合拳：框架给你一个能跑起来的默认值，但只要你自己动手，默认值立刻让位。

为什么不是"默认放行"，而是"默认拒绝"？

试想一下反过来的设计：只要引入依赖，默认所有接口都能匿名访问，直到你手动加限制。这种设计下，如果开发者忘记配置某个新加的接口，它就会被裸奔到公网上——这是真实发生过的大量数据泄露事故的根源（忘配、漏配比错配更常见）。 Spring Security 选择"默认拒绝"，逼着你必须显式地"开一个口子"才能放行，出错的方向是"多了一层认证麻烦"而不是"泄露了敏感数据"，代价小得多。

原理走一遍：自动配置怎么兜底的

① 检测依赖
  classpath 中发现 spring-security-web、spring-security-config
  → 触发 SecurityAutoConfiguration 生效

② 检查用户是否自定义
  容器中是否已存在开发者自己声明的 SecurityFilterChain Bean？
  → 有：用户配置优先，跳过默认配置
  → 没有：SpringBootWebSecurityConfiguration 兜底生效

③ 生成默认防线
  默认 SecurityFilterChain：所有请求都要 authenticated()
  开启 formLogin() + httpBasic()（自带一个登录页）
  UserDetailsServiceAutoConfiguration 生成一个内存用户 user/随机密码
  → 应用"安全但不可用"，逼着开发者主动配置

怎么覆盖默认行为：自定义 SecurityFilterChain

Spring Security 5.7+（Spring Security 6 延续这个写法）已经废弃了 WebSecurityConfigurerAdapter 的继承式写法，官方推荐用 组件式（Bean 风格） 配置——直接声明一个 SecurityFilterChain 类型的 @Bean，交给容器管理，风格上和 Spring Boot 3 全面拥抱"函数式配置"的思路一致。

SecurityConfig.java（Spring Security 6 标准写法）
@Configuration
public class SecurityConfig {

    // 声明 SecurityFilterChain Bean，容器里一旦存在，默认兜底配置自动失效
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                // 显式声明哪些路径放行，其余仍然默认拒绝
                .requestMatchers("/public/**", "/login").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults());

        return http.build();
    }
}

只要这个 Bean 存在，@ConditionalOnMissingBean 条件不再满足，Spring Boot 的默认随机密码兜底配置就彻底不生效了。

场景是否有 SecurityFilterChain Bean实际生效的配置
只加依赖，不写配置 否 兜底配置：所有请求需认证，内存用户 user + 随机密码
自己写了 SecurityConfig 是 完全由开发者的配置决定，兜底配置整体让位
旧代码继承 WebSecurityConfigurerAdapter —（该类已废弃） Spring Security 5.7+ / 6.x 编译报错或不推荐使用，需迁移到 SecurityFilterChain Bean 写法
生产环境常见坑：
  • 把控制台打印的默认密码当成"正式密码"记下来用——它每次重启都会变，根本不该出现在生产环境。
  • 本地测试图省事一直没配置 SecurityFilterChain，上线后才发现所有接口都要登录，前端联调全部失败。
  • 误以为"没配置 = 没有安全限制"，实际恰恰相反：没配置时防线最严，配置了才可能"松口子"。
面试高频："为什么加了 spring-boot-starter-security 之后所有接口都要登录？" 考察的就是这条"默认拒绝 + @ConditionalOnMissingBean 兜底"的自动配置链路，能答出这个机制比死记"要写 SecurityFilterChain"更能体现理解深度。

核心价值总结

  • 安全优先：默认拒绝一切，逼着开发者显式声明放行规则，避免"忘配置"导致的裸奔风险。
  • 约定优于配置：没有自定义时给一个能跑起来的默认值，写了自定义后立刻让位，两者通过 @ConditionalOnMissingBean 无缝切换。
  • 新写法统一：Spring Security 6 全面转向 SecurityFilterChain Bean 风格，WebSecurityConfigurerAdapter 已废弃，不要再用继承式写法。
