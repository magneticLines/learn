无状态会话与 JWT 整合（承接 04-工程实战）

前面几篇讲的 formLogin + Session 是有状态认证——真正的前后端分离项目通常要换成 JWT，这一篇聚焦"在 Spring Security 里具体怎么接入"。

承接：JWT 原理回顾

JWT（JSON Web Token）本身的结构、签名原理、为什么能做到"无状态"，在 04-工程实战/auth/01-JWT.html 已经详细讲过，这里不重复展开。 核心结论回顾一句话：JWT 把用户身份信息和签名一起编码进一个字符串里，服务端不需要存 Session，只要验证签名没被篡改，就能相信 Token 里的内容——这正是"无状态"三个字的含义：服务端不再需要为每个登录用户保存一份会话状态。

这一篇要解决的问题是：Spring Security 默认的认证机制是围绕 Session 设计的（登录成功后把 Authentication 存进 HttpSession，后续请求靠 JSESSIONID Cookie 识别身份）。要换成 JWT，需要关掉这套默认机制，自己在过滤器链里插入一段"解析 JWT、还原登录状态"的逻辑。

第一步：关闭 Session 机制

SecurityConfig.java（声明无状态）
@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http,
                                           JwtAuthenticationFilter jwtFilter) throws Exception {
        http
            // 关键一：关闭 CSRF（下一篇详细讲原因），JWT 场景下不需要
            .csrf(AbstractHttpConfigurer::disable)
            // 关键二：告诉容器不要创建、也不要使用 HttpSession
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/login").permitAll()
                .anyRequest().authenticated())
            // 关键三：把自定义的 JWT 过滤器插入到框架默认认证过滤器之前
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}

SessionCreationPolicy.STATELESS 意味着 Spring Security 永远不会调用 request.getSession() 去创建 Session，也不会去读已存在的 Session——每一次请求都被当成"全新的、未知身份"，必须靠请求本身携带的凭证（也就是 JWT）重新证明身份。

第二步：自定义过滤器解析 JWT

OncePerRequestFilter 是 Spring 提供的一个基类，保证同一个请求在一次转发链路里只会被这个过滤器处理一次（避免 forward/include 场景下重复执行）。 在这里解析请求头里的 JWT，验证签名通过后，手动构造一个 Authentication 对象塞进 SecurityContext——效果等价于"告诉框架：这个请求已经登录过了，登录的人是谁"。

JwtAuthenticationFilter.java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtTokenProvider jwtTokenProvider;       // 校验签名、解析 claims，复用 JWT 篇讲过的工具类
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {

        String header = request.getHeader("Authorization");

        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);

            if (jwtTokenProvider.validate(token)) {
                String username = jwtTokenProvider.getUsername(token);
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);

                // 手动构造已认证的 Authentication，塞进当前线程的 SecurityContext
                UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());

                SecurityContextHolder.getContext().setAuthentication(authentication);
            }
        }

        chain.doFilter(request, response);  // 无论是否认证成功都要放行给下一个过滤器
    }
}
STATELESS 模式下每个请求的处理流程
请求到达 → JwtAuthenticationFilter 解析 Authorization 头
  → Token 合法：SecurityContextHolder 里放入 Authentication（当前请求"临时登录"）
  → Token 缺失/非法：SecurityContext 里没有 Authentication，继续走后续流程
→ authorizeHttpRequests 检查 SecurityContext 里有没有已认证的用户
  → 没有 → 401；有 → 放行进入 Controller
→ 请求处理完毕，SecurityContext 随线程结束自动清空，下一个请求重新来过
常见坑：忘记在 authorizeHttpRequests 里放行登录接口本身（比如 /api/auth/login），导致"要登录才能访问登录接口"的死循环； 或者过滤器抛出的异常没有被正确处理，导致 Token 过期时返回的是 500 而不是清晰的 401，前端无法据此触发重新登录逻辑。

核心价值总结

  • STATELESS 是开关：一行 sessionCreationPolicy(STATELESS) 就让 Spring Security 彻底放弃 Session 机制。
  • 自定义过滤器是桥梁：JWT 校验的产出（用户名/权限）通过手动构造 Authentication 对象，"翻译"成框架认识的格式放进 SecurityContext。
  • 过滤器顺序很关键：addFilterBefore 必须插在 UsernamePasswordAuthenticationFilter 之前，保证后续的授权检查能读到已经填好的登录状态。
