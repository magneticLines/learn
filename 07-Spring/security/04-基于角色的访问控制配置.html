基于角色的访问控制配置

知道"用户是谁"（认证）只是第一步，"这个用户能不能访问这个接口"（授权）才是权限系统的核心——RBAC 模型在 Spring Security 里怎么落地。

承接：RBAC 模型回顾

RBAC（Role-Based Access Control，基于角色的访问控制）的核心思想此前在 04-工程实战 里已经讲过： 不给用户直接分配权限，而是"用户 → 角色 → 权限"两层间接映射——角色是权限的集合，用户通过被赋予角色来获得一批权限。 这样调整权限时只需要改角色和权限的对应关系，不需要挨个改每个用户，管理成本大幅降低。这一篇不重复展开模型本身，聚焦在"这套模型在 Spring Security 里具体怎么写代码"。

一句话讲清楚

在 SecurityFilterChain 里用 authorizeHttpRequests 声明一组"URL 匹配规则 + 所需角色/权限"， Spring Security 会在每个请求进入业务代码之前，先经过一个过滤器检查当前登录用户的权限集合（也就是上一篇 UserDetails.getAuthorities() 返回的那份数据）是否满足规则，不满足直接返回 403 Forbidden。

SecurityConfig.java（URL 级别的角色控制）
@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth
            // 规则按顺序匹配，越具体的路径要写在越前面
            .requestMatchers("/api/public/**").permitAll()
            .requestMatchers("/api/admin/**").hasRole("ADMIN")
            .requestMatchers("/api/orders/**").hasAnyRole("USER", "ADMIN")
            .requestMatchers(HttpMethod.DELETE, "/api/orders/**").hasAuthority("ORDER_DELETE")
            .anyRequest().authenticated()
        )
        .formLogin(Customizer.withDefaults());

        return http.build();
    }
}

hasRole 和 hasAuthority 的区别

这是新手最容易踩的一个坑：两者看起来都是"检查有没有某个权限字符串"，但 hasRole("ADMIN") 会自动在字符串前面拼接 "ROLE_" 前缀， 实际比对的是 ROLE_ADMIN；而 hasAuthority("ADMIN") 是原样比对，不会加前缀。

方法实际比对的字符串语义定位
hasRole("ADMIN") ROLE_ADMIN（自动加前缀） 粗粒度"角色"，通常一个用户只有少数几个角色
hasAuthority("ORDER_DELETE") ORDER_DELETE（原样比对） 细粒度"权限点"，一个角色可以聚合很多个权限点
新手常见错误：在 UserDetailsService 里把角色存成 new SimpleGrantedAuthority("ADMIN")（没加 ROLE_ 前缀）， 却在配置里用 hasRole("ADMIN") 判断——这样永远匹配不上，因为 hasRole 实际找的是 ROLE_ADMIN。 要么在存储角色时统一加上 ROLE_ 前缀再配合 hasRole，要么干脆全部使用 hasAuthority 并自己控制前缀命名。

RBAC 模型到 Spring Security 配置的映射关系

RBAC 概念   →   Spring Security 落地方式

用户（User）   →   UserDetailsService 查出的 UserDetails
角色（Role）   →   GrantedAuthority，约定加 ROLE_ 前缀，配合 hasRole 使用
权限点（Permission）   →   GrantedAuthority，不加前缀，配合 hasAuthority 使用
用户-角色映射表   →   数据库里 user_role 关联表，UserDetailsService 查询时 JOIN 出来
访问控制点   →   authorizeHttpRequests 里按 URL 声明的规则

规则顺序很重要

authorizeHttpRequests 里的规则是按声明顺序依次匹配，命中第一条就停止，不是"匹配最精确的那条"。 所以更具体的路径规则必须写在更宽泛的规则前面，例如上面例子里 /api/admin/** 必须写在 anyRequest() 之前，否则永远会被 anyRequest() 先拦截命中。

面试高频："如何在 Spring Security 里实现细粒度的按钮级权限控制？" —— URL 级别的 authorizeHttpRequests 只能做到接口粒度，无法区分"同一个接口内部不同业务分支需要不同权限"这种场景，这正是下一篇 @PreAuthorize 方法级安全要解决的问题。

核心价值总结

  • URL 级别的粗粒度控制：authorizeHttpRequests 是 RBAC 模型在接口层面最直接的落地方式。
  • hasRole 自动加前缀，hasAuthority 不加：两者本质都是比对 GrantedAuthority 字符串，区别只在于是否拼接 ROLE_。
  • 规则顺序决定优先级：具体规则要写在宽泛规则之前，这是配置类里最容易踩的坑之一。
