三级缓存解决循环依赖

上一篇提到"属性填充"和"实例化"分两步走，正是为了给循环依赖留一个"缓兵之计"的窗口。这一篇就来看 Spring 具体是怎么利用这个窗口，用三级缓存把循环依赖拆解开的。

问题：A 依赖 B，B 又依赖 A

循环依赖示例
@Component
public class A {
    @Autowired
    private B b;
}

@Component
public class B {
    @Autowired
    private A a;
}

创建 A 需要先注入 B，而创建 B 又需要先注入 A——如果严格按"创建完整对象后才能被别人引用"的规则，两边会互相等待，谁都无法完成。

类比：两个人交换礼物，都要求"你先把礼物包装完整给我，我才愿意包装我的礼物"，结果谁都不动手。Spring 的解法是：先给对方一张"提货凭证"（引用一个还没装满东西的空盒子），凭证可以先用来完成自己的包装，等真正需要打开盒子的时候，东西早已经放进去了——因为 Java 对象是引用传递，只要拿到的是同一个对象的引用，之后往里面塞多少属性，外部持有的引用看到的都是最新状态。

三级缓存分别是什么

三级缓存都在 DefaultSingletonBeanRegistry 里，本质是三个 Map：

层级字段名存的是什么作用
一级缓存 singletonObjects 完全初始化好的单例 Bean 容器对外提供 Bean 的最终来源
二级缓存 earlySingletonObjects 提前暴露的"半成品"Bean（已实例化，属性还没填充完） 让其他 Bean 能提前拿到引用，打破循环等待
三级缓存 singletonFactories ObjectFactory（能生产 Bean 或其代理对象的工厂） 延迟决定"到底暴露原始对象还是代理对象"

创建流程：A 和 B 如何互相拿到对方

1. 容器要创建 A，调用构造器完成 实例化（此时 A 是个空对象）
2. 实例化完成后，立刻把一个能生产 A（或 A 的代理）的 ObjectFactory 放进三级缓存
3. 开始给 A 填充属性，发现需要注入 B，于是转去创建 B
4. B 实例化完成，同样把 ObjectFactory 放进三级缓存，开始填充 B 的属性
5. B 发现需要注入 A，去缓存里找 A：
    ├─ 一级缓存没有（A 还没创建完）
    ├─ 二级缓存没有
    └─ 三级缓存命中：调用 ObjectFactory.getObject()，拿到 A 的早期引用（如果 A 需要被代理，此时就生成代理），
        把结果提升到二级缓存，同时从三级缓存移除
6. B 拿着这个引用完成属性填充，顺利完成初始化，被放入一级缓存
7. 回到 A 的属性填充，此时 B 已经在一级缓存里，A 直接拿到完整的 B，完成初始化
8. A 也被放入一级缓存，循环依赖问题解决

为什么两级缓存不够，非要第三级

如果只有一级 + 二级缓存（提前暴露原始对象），会遇到一个致命问题：AOP 代理是在初始化完成之后（postProcessAfterInitialization）才生成的， 但循环依赖要求在"属性填充阶段"就把引用提前暴露出去。如果这时候暴露的是原始对象，等 A 真正初始化完成、该生成代理时， 已经有 B 持有了 A 的原始对象引用——最终容器里是代理对象，B 手里却是原始对象，两者不是同一个实例，破坏了"单例"的语义。

三级缓存存的不是对象，而是一个 ObjectFactory（工厂），把"到底返回原始对象还是代理对象"这个决定延迟到真正被取用的那一刻再做： 如果该 Bean 需要 AOP 代理，工厂内部会调用 getEarlyBeanReference() 提前触发代理逻辑，保证不管是谁提前拿到的引用，都和最终放入一级缓存的对象是同一个。

面试高频："为什么要三级缓存，两级不行吗？" 标准答案：如果不涉及 AOP，两级缓存确实够用；三级缓存存在的唯一原因是要兼容 AOP 代理——用工厂模式把"生成代理"这个动作延迟到真正发生循环依赖、需要提前暴露引用的那一刻才执行，避免了"同一个 Bean 出现两个不同实例"的问题。

构造器循环依赖为什么无解

上面的方案能成立，前提是"实例化"和"属性填充"是分开的两步——实例化一结束，就能立刻拿到一个空对象的引用放进缓存。 但如果 A 和 B 都是通过构造器注入互相依赖（比如 A(B b) 和 B(A a)），那么实例化这一步本身就无法完成： 创建 A 需要先有一个完整的 B 作为构造参数，创建 B 又需要先有一个完整的 A——在对象连"空壳"都造不出来的阶段，压根没有引用可以提前暴露，三级缓存无从下手。

结论：Spring 只能解决 setter/字段注入的循环依赖，构造器注入的循环依赖会直接抛出 BeanCurrentlyInCreationException。这也是为什么强依赖推荐用构造器注入（快速失败、暴露设计问题），而不是简单地为了"绕开循环依赖报错"就无脑改成字段注入掩盖问题。

常见误区

误区：以为三级缓存能解决"所有"循环依赖场景。实际上它只覆盖单例 + setter/字段注入这一种情况；prototype（原型）作用域的 Bean 每次都要创建新实例，Spring 压根不会缓存，一旦出现循环依赖会直接报错。

核心价值总结

  • 问题的本质：循环依赖能被打破，靠的是 Java 对象"先分配内存、后填充状态"的特性——只要能提前拿到引用，之后填充什么内容外部都能感知到。
  • 三级缓存的分工：一级放成品、二级放半成品原始引用、三级放"决定要不要代理"的工厂，本质是用空间换时间，把"是否要代理"这个决策推迟到刚好够用的最后时刻。
  • 边界很清晰：只能救 setter/字段注入的单例循环依赖，构造器注入和原型作用域都救不了——理解这一点比背诵缓存层级更重要。

与其他知识点的联系

  • Bean 的完整生命周期：三级缓存不是一个独立设计，而是完全建立在上一篇"实例化和属性填充是两个分开的步骤"这一前提之上——正是因为实例化一结束就能拿到一个空对象的引用，才有机会把它（或它的工厂）放进缓存，供其他 Bean 在属性填充阶段提前引用，详见 internals/02-Bean的完整生命周期.html。
  • BeanPostProcessor 扩展点：三级缓存里存的 ObjectFactory 之所以不能简化成直接存原始对象，根源在于 AOP 代理的生成时机——正常情况下代理是在 postProcessAfterInitialization 才生成的，但循环依赖要求提前暴露引用，所以 ObjectFactory.getObject() 内部会调用 getEarlyBeanReference()，实质是提前触发了 AnnotationAwareAspectJAutoProxyCreator（一个 BeanPostProcessor）的代理逻辑，把"要不要代理"这个判断从"初始化后"提前搬到"属性填充时"执行，两篇讲的是同一个扩展点在两种不同触发时机下的行为，详见 internals/04-BeanPostProcessor扩展点.html。
  • 事务管理 @Transactional：如果一个需要事务代理的 Service Bean 恰好陷入循环依赖，就必须依赖三级缓存提前生成代理，否则会出现"容器最终持有的是代理对象，但循环依赖的另一方在属性填充时拿到的是原始对象"这种错乱——即注入进去的 Bean 不是真正生效事务的那个对象，调用它的事务方法会静默失效，这是三级缓存和 AOP、事务机制之间最容易被忽视但后果很严重的联系，详见 data/04-事务管理Transactional.html。
