BeanDefinition 与容器启动流程

04-工程实战 讲过"IoC 是什么、为什么要用它"，这里往下钻一层：Spring 容器启动时到底做了什么？答案的核心只有一句话——先造元数据，再造对象。

为什么不能"读到 Bean 定义就直接 new"

最朴素的想法是：解析 XML/注解，读到一个 Bean 的配置，立刻 new 出来放进容器。但这样做不通，原因是 Bean 之间有依赖关系，而配置的加载顺序是不可控的。 如果 A 依赖 B，但 A 的配置先被解析到，此时 B 还没被创建，A 就无法完成实例化。

类比：盖房子之前，工地不会一边挖地基一边现场设计图纸——而是先把整栋楼的设计图纸（图纸上标注每个房间的用途、面积、装修材料）画完、评审完，再统一开工。图纸阶段和施工阶段是分离的：图纸可以互相引用（"客厅的门通向厨房"），施工时才按依赖顺序把墙砌起来。

Spring 的解法是把"配置解析"和"实例化"拆成两个独立阶段，中间插入一层元数据——BeanDefinition。

BeanDefinition：Bean 的"设计图纸"

BeanDefinition 是一个接口，描述了创建一个 Bean 所需要的全部元信息，但它本身不是 Bean 实例，只是一份"待办清单"：

  • beanClassName —— 这个 Bean 是哪个类
  • scope —— 单例（singleton）还是原型（prototype）
  • propertyValues —— 属性要注入什么值（依赖哪些其他 Bean）
  • constructorArgumentValues —— 构造器参数
  • lazyInit —— 是否懒加载
  • initMethodName / destroyMethodName —— 自定义初始化/销毁方法名

不管你写的是 XML 的 <bean> 标签、@Component 注解，还是 @Bean 方法，Spring 解析配置的第一步，都是把它们统一"翻译"成 BeanDefinition 对象， 塞进一个叫 BeanDefinitionRegistry 的注册表里（本质是一个 Map<String, BeanDefinition>，key 是 beanName）。

BeanDefinition 核心方法（简化）
public interface BeanDefinition {
    String getBeanClassName();
    String getScope();          // singleton / prototype
    boolean isLazyInit();
    ConstructorArgumentValues getConstructorArgumentValues();
    MutablePropertyValues getPropertyValues();
    String getInitMethodName();
    String getDestroyMethodName();
}

容器启动的三个阶段

以 AnnotationConfigApplicationContext / ClassPathXmlApplicationContext 为例，启动过程可以概括为三步：

① 加载配置 → 生成 BeanDefinition
  ├─ XML：BeanDefinitionReader 解析 <bean> 标签
  ├─ 注解：ClassPathBeanDefinitionScanner 扫描 @Component/@Service 等
  ├─ Java Config：ConfigurationClassPostProcessor 解析 @Configuration/@Bean
  └─ 统一注册进 BeanDefinitionRegistry（DefaultListableBeanFactory 内部的 Map）
② 执行 BeanFactoryPostProcessor → 修改 BeanDefinition
  └─ 容器在实例化任何 Bean 之前，把已注册的 BeanDefinition 全部暴露出来，
      允许开发者在这一步"手术"式地修改元数据（比如替换占位符 ${}）
③ 实例化单例 Bean → BeanDefinition 变成真正的对象
  └─ preInstantiateSingletons()：遍历所有非懒加载单例的 BeanDefinition，
      逐个调用 getBean() 触发真正的实例化（下一篇细讲这条链）
核心结论（面试高频）：Spring 容器的启动过程本质上是"两阶段"的——第一阶段只造元数据（BeanDefinition），不涉及任何业务类的构造器调用；第二阶段才真正 new 对象。 这也是为什么 BeanFactoryPostProcessor（操作 BeanDefinition）永远先于 BeanPostProcessor（操作 Bean 实例）执行——它们分别工作在两个不同的阶段。

常见误区

误区一：以为 @Component 标注的类在"被扫描到"的那一刻就已经被实例化了。实际上扫描阶段只生成 BeanDefinition 并注册，真正的 new 要等到容器刷新流程走到"实例化单例"这一步（甚至更晚，如果是懒加载或原型 Bean）。
误区二：混淆 BeanFactoryPostProcessor 和 BeanPostProcessor。前者操作的是 BeanDefinition（配置元数据），在所有 Bean 实例化之前执行一次；后者操作的是 Bean 实例本身，在每个 Bean 的生命周期中穿插执行。名字像双胞胎，作用对象完全不同，是非常经典的面试题。

核心价值总结

  • 元数据与实例分离：BeanDefinition 让 Spring 可以在真正创建对象之前，统一处理依赖关系、修改配置，而不用担心"创建顺序"带来的死锁式依赖问题。
  • 可扩展的处理管道：BeanDefinitionRegistryPostProcessor → BeanFactoryPostProcessor → 实例化，每一步都是可插拔的扩展点，这也是 Spring 生态里各种"魔法"（如 @ConfigurationProperties、条件装配）得以实现的地基。
