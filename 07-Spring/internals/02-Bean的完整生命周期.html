Bean 的完整生命周期

上一篇讲到"实例化单例 Bean"是容器启动的最后一步——这一步展开来看，是一条环环相扣的链条，链条上的每一环都是 Spring 留给开发者的扩展点。

为什么生命周期要设计得这么长

如果 Spring 只是简单地 new 一个对象然后塞进容器，那开发者就没有机会在"对象造好之后、真正对外提供服务之前"做点什么——比如注入依赖、校验配置、包装代理。 Spring 的做法是把"造一个可用的 Bean"拆解成多个明确的阶段，每个阶段之间都开放回调接口，让框架和开发者都能在合适的时机介入。

类比：新员工入职不是"签完合同就直接上岗"，而是：入职登记（实例化）→ 领工位电脑（属性填充）→ 被系统识别工号（Aware 回调）→ 入职培训前的材料准备（前置处理）→ 正式培训并考核合格（初始化）→ 培训后颁发工牌（后置处理，AOP 代理就发生在这里）→ 正式上岗工作（使用）→ 离职交接（销毁）。

完整链条

1. 实例化（Instantiation） —— 调用构造器，此时对象内存已分配，但属性全是默认值
2. 属性填充（Populate） —— 依赖注入，把 @Autowired / <property> 指定的依赖设置进去
3. Aware 接口回调 —— BeanNameAware / BeanFactoryAware / ApplicationContextAware ...
    让 Bean 有机会拿到自己在容器中的名字、甚至容器本身的引用
4. BeanPostProcessor.postProcessBeforeInitialization() —— 初始化前置处理
5. 初始化（Initialization）
    ├─ @PostConstruct 标注的方法
    ├─ InitializingBean.afterPropertiesSet()
    └─ 自定义 init-method
6. BeanPostProcessor.postProcessAfterInitialization() —— 初始化后置处理（AOP 代理在这里生成）
7. 使用（In Use） —— Bean 正式对外提供服务，放入单例池
8. 销毁（Destruction） —— 容器关闭时触发
    ├─ @PreDestroy 标注的方法
    ├─ DisposableBean.destroy()
    └─ 自定义 destroy-method

为什么"属性填充"和"实例化"必须分开两步

构造器只负责"分配内存 + 执行构造代码"，但 Java 语法要求构造器参数必须在调用时就确定——而 Spring 的依赖注入经常涉及循环引用（A 依赖 B、B 又依赖 A）， 如果所有依赖都必须通过构造器注入，循环依赖就无解（下一篇会详细展开）。所以 Spring 默认走的是"先造出一个光秃秃的空对象，再通过 setter/字段反射把依赖灌进去"的两阶段方式，这也是三级缓存能生效的前提。

AOP 代理为什么恰好在 postProcessAfterInitialization 生成

代理的本质是"包一层"——原始对象必须先完整地初始化好（属性齐全、afterPropertiesSet() 跑完），代理才有意义。 如果在实例化阶段就生成代理，代理内部包裹的其实是个半成品对象。所以 Spring 选择在初始化完全结束之后的这个后置钩子里，判断该 Bean 是否命中某个切面的 Pointcut，命中则用代理对象替换原始对象放入容器（细节见第 4 篇）。

BeanPostProcessor 接口
public interface BeanPostProcessor {
    Object postProcessBeforeInitialization(Object bean, String beanName);
    Object postProcessAfterInitialization(Object bean, String beanName);
    // 返回值可以是原对象，也可以替换成一个全新的对象（代理正是靠这一点实现）
}

常见误区

误区一：以为 @PostConstruct 和 InitializingBean.afterPropertiesSet() 是互斥的两种方案随便选一个。实际上如果同时存在，@PostConstruct 先执行，afterPropertiesSet() 后执行，最后才是自定义 init-method——三者可以同时生效，顺序是固定的。
误区二：在构造函数或字段初始化时就想拿到被 @Autowired 注入的其他 Bean，结果拿到 null。原因是构造阶段属性还没填充，正确做法是在实现了 InitializingBean 或标注 @PostConstruct 的方法里使用注入的依赖。
面试高频："Spring Bean 的生命周期有哪些阶段？" 按"实例化 → 属性填充 → Aware 回调 → 初始化前置处理 → 初始化 → 初始化后置处理 → 销毁"这条主线回答，并强调 AOP 代理生成于 postProcessAfterInitialization，这是加分点。

核心价值总结

  • 每一环都是扩展点：Aware 接口让 Bean 感知容器，BeanPostProcessor 让框架/开发者能介入任意 Bean 的初始化前后，InitializingBean 让 Bean 自己声明"准备好了"的时机。
  • 顺序不是随意的：先填充属性再初始化，是为了保证初始化逻辑能安全使用依赖；先初始化完成再生成代理，是为了保证代理包裹的是"成品"而非"半成品"。
  • 为后两篇做铺垫：属性填充阶段的循环依赖问题（第 3 篇）、以及 postProcessAfterInitialization 里生成 AOP 代理的具体机制（第 4 篇），都是在这条链条基础上的深入展开。

与其他知识点的联系

  • BeanDefinition 与容器启动流程：本篇的整条生命周期链，本质就是上一篇"实例化单例 Bean"这一步的展开——容器按 BeanDefinition 里记录的类名、属性值、初始化方法名等元信息，依次驱动实例化、属性填充、初始化，BeanDefinition 是"图纸"，本篇讲的是"照图施工"的具体过程，详见 internals/01-BeanDefinition与容器启动流程.html。
  • 三级缓存解决循环依赖：本篇强调"实例化"和"属性填充"必须分成两个独立步骤，正是三级缓存能够生效的根本前提——实例化刚结束、属性还没填充完的这个"空窗期"，恰好是三级缓存往外提前暴露引用的时机；如果这两步合并成一步，循环依赖就完全无解，详见 internals/03-三级缓存解决循环依赖.html。
  • BeanPostProcessor 扩展点：本篇列出的"初始化前置处理"和"初始化后置处理"两个节点，正是 BeanPostProcessor 接口的两个方法——AOP 代理为什么恰好在 postProcessAfterInitialization 生成，本篇已经给出了原理（代理要包裹的必须是"成品"而非"半成品"），第 4 篇会展开讲这个扩展点本身的设计和更多用途，详见 internals/04-BeanPostProcessor扩展点.html。
  • 事务管理 @Transactional：@Transactional 背后的事务代理，正是通过本篇讲的 postProcessAfterInitialization 钩子生成的——容器判断某个 Bean 的方法命中了事务相关的 Pointcut，就在这个阶段用代理对象替换掉原始对象再放入容器，这也是为什么"事务方法必须通过容器管理的代理对象调用才会生效"，详见 data/04-事务管理Transactional.html。
