Advisor / Advice / Pointcut 的关系

前两篇讲了"代理怎么造出来"，这一篇讲代理内部真正执行的增强逻辑从哪来、怎么判断该不该拦截某个方法——这是 Spring AOP 里最基础也最容易混淆概念的三个术语。

先拆分成两个独立的问题

要在方法调用前后插入自定义逻辑，天然要回答两个互相独立的问题：

  • 在哪里切入？——哪些类的哪些方法需要被拦截（比如"所有 @Service 类里以 save 开头的方法"）
  • 切入之后做什么？——具体的增强逻辑是什么（比如"记录方法执行耗时"）

Spring AOP 把这两个问题分别抽象成两个独立接口——Pointcut（在哪）和 Advice（做什么），然后用 Advisor 把两者绑定在一起，形成一个完整可用的"切面单元"。

类比：公司规定"财务审批"要有额外流程。Pointcut 相当于规则说明"金额超过 5000 元的报销单"（在哪里生效）；Advice 相当于具体动作"打印一份审批单、发邮件给主管"（做什么）；Advisor 则是把这两者装订成一份正式的公司制度文件——"金额超过 5000 元的报销单，需要打印审批单并通知主管"，这份文件才是真正能被执行、被审计的东西。

三者的接口定义

Pointcut：定义"在哪里"
public interface Pointcut {
    ClassFilter getClassFilter();     // 匹配哪些类
    MethodMatcher getMethodMatcher(); // 匹配哪些方法
}
Advice：定义"做什么"（顶层是个标记接口，实际有多种子类型）
public interface Advice {}

// 常见子接口，对应 @Before / @AfterReturning / @Around 等注解
public interface MethodBeforeAdvice extends Advice {
    void before(Method method, Object[] args, Object target);
}
public interface MethodInterceptor extends Advice {   // 对应 @Around，功能最强
    Object invoke(MethodInvocation invocation) throws Throwable;
}
Advisor：Pointcut + Advice 的组合
public interface Advisor {
    Advice getAdvice();   // 做什么
}

public interface PointcutAdvisor extends Advisor {
    Pointcut getPointcut();  // 在哪里，两者合体才是一个完整的切面单元
}

三者如何在自动代理流程里协作

回顾上一篇提到的自动代理创建器逻辑，可以把 Pointcut / Advice / Advisor 的分工放进整条链路里看：

容器启动时： 所有 @Aspect 类里的通知方法，被解析成一个个 Advisor 对象，注册到容器中
每个 Advisor 内部：
  ├─ Pointcut —— 记录这条通知匹配哪些类、哪些方法（比如"execution(* com.xx.service..*(..))"）
  └─ Advice —— 记录具体执行逻辑（比如 @Around 方法里的那段代码，被包装成 MethodInterceptor）
postProcessAfterInitialization 阶段：
  对每个新创建的 Bean，遍历容器里所有 Advisor，用其 Pointcut 逐个匹配 —— 只看"在哪"，不关心"做什么"
  匹配上的 Advisor 会被收集起来，成为这个代理对象的"拦截器链"
方法真正被调用时：
  代理内部按顺序执行拦截器链上每个 Advisor 的 Advice —— 这时才真正执行"做什么"的逻辑

@Aspect 注解和底层 Advisor 的对应关系

日常开发几乎不会手写 Advisor/Pointcut/Advice，而是用更友好的注解语法：

注解写法 vs 底层概念
@Aspect
@Component
public class LogAspect {

    @Pointcut("execution(* com.example.service..*(..))")  // ← 对应 Pointcut
    public void serviceLayer() {}

    @Around("serviceLayer()")  // ← @Around 方法体本身对应 Advice
    public Object logAround(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed();  // 放行，执行目标方法
        System.out.println("耗时: " + (System.currentTimeMillis() - start));
        return result;
    }
}

这段代码在容器启动时，会被 AnnotationAwareAspectJAutoProxyCreator 内部的解析逻辑翻译成一个标准的 Advisor 对象： @Pointcut 表达式被解析成 AspectJExpressionPointcut，@Around 方法体被包装成一个 MethodInterceptor（Advice）， 两者再组合成一个 InstantiationModelAwarePointcutAdvisor。注解只是更友好的语法糖，Spring AOP 运行时能理解的、真正参与匹配和执行的，始终是底层的 Advisor/Pointcut/Advice 体系。

常见误区

误区一：以为 @Pointcut 和 @Around 是同一层次的概念，可以互换。实际上 @Pointcut 只负责"匹配规则"，本身不包含任何业务逻辑；@Around（连同它标注的方法体）才是真正的"做什么"。两者职责不同，只是经常写在一起容易被误认为是一回事。
误区二：认为一个 Advisor 只能对应一种通知类型。实际上 @Before/@After/@AfterReturning/@AfterThrowing/@Around 各自都会被解析成独立的 Advisor（各自持有自己的 Advice 实现），一个 @Aspect 类里写多个通知方法，最终会生成多个 Advisor，它们各自独立地参与匹配和拦截器链的组装。
面试高频："Pointcut、Advice、Advisor 分别是什么，什么关系？" 标准答案：Pointcut 定义拦截的位置（在哪），Advice 定义拦截后执行的逻辑（做什么），Advisor 是两者的组合，是 Spring AOP 真正参与运行时匹配和执行的最小单元；日常写的 @Aspect/@Pointcut/@Around 只是更符合人类阅读习惯的注解形式，底层都会被统一解析成 Advisor。

核心价值总结

  • 关注点分离：把"在哪匹配"和"做什么"拆成两个独立接口，使得同一个 Advice 可以被不同的 Pointcut 复用，同一个 Pointcut 也可以搭配不同的 Advice，组合的灵活性大大提升。
  • Advisor 是运行时的最小执行单元：自动代理创建器遍历的、拦截器链上挂载的，都是 Advisor，而不是零散的 Pointcut 或 Advice。
  • 注解是语法糖，不是新概念：@Aspect 体系提供的只是更符合直觉的书写方式，理解了 Advisor/Pointcut/Advice 三者的关系，就理解了 Spring AOP 运行时真正依赖的核心模型——这也是这个系列 6 篇文章想要建立的完整认知链条：从容器启动、Bean 生命周期、循环依赖、扩展点，到代理技术选型，最终落到 AOP 运行时的核心抽象。

与其他知识点的联系

  • BeanPostProcessor 扩展点：本篇讲的 Advisor 匹配逻辑，正是第 4 篇 postProcessAfterInitialization 内部"遍历容器中所有 Advisor，用其 Pointcut 逐个匹配当前 Bean"这一步的具体展开——BeanPostProcessor 回答的是"代理在哪个时机生成"，本篇回答的是"生成代理时到底依据什么规则判断该不该代理、代理里执行的是什么逻辑"，两篇合起来才是 Spring AOP 自动代理机制的完整闭环，详见 internals/04-BeanPostProcessor扩展点.html。
  • 三级缓存解决循环依赖：如果一个存在循环依赖的 Bean 同时命中了某个 Advisor 的 Pointcut，三级缓存里的 ObjectFactory 在被提前引用时就必须提前完成本篇讲的"Pointcut 匹配 + 生成代理"这套流程，否则会出现"提前暴露的是原始对象、但最终容器里是代理对象"的两个实例不一致问题，这正是三级缓存需要第三级（而非两级）的根本原因，详见 internals/03-三级缓存解决循环依赖.html。
  • 事务管理 @Transactional：事务管理是本篇讲的 Advisor/Advice/Pointcut 模型最典型的内置应用——@Transactional 背后对应一个 TransactionAttributeSourceAdvisor，它的 Pointcut 匹配"带 @Transactional 注解的方法"，它的 Advice 是 TransactionInterceptor，在方法调用前后完成开启、提交、回滚事务的具体逻辑；理解了本篇的三元关系，就能理解事务本质上"只是一种特殊的切面增强"，并不是脱离 AOP 体系之外的独立机制，详见 data/04-事务管理Transactional.html。
  • Spring Data Redis：@Cacheable/@CacheEvict 同样是本篇模型的一次应用——对应的 Advisor 用 Pointcut 匹配带缓存注解的方法，Advice 负责在方法执行前查缓存、执行后写缓存，和事务管理是同一套底层机制在不同场景下的具体实例，详见 data/06-SpringDataRedis概览.html。
  • Repository 方法命名查询：Repository 接口的动态代理和本篇讲的 AOP 代理都属于"运行时生成代理对象"这一大类技术，但两者不共享 Advisor/Pointcut/Advice 这套模型——Repository 代理的目标是把方法名解析成查询逻辑并委托执行，不涉及"匹配已有对象的方法并插入横切逻辑"，理解这个边界能帮助区分"动态代理"这个通用技术手段和"AOP"这个具体应用场景的关系，详见 data/01-Repository接口与方法命名查询.html。
