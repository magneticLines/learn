JDK 动态代理与 CGLIB 字节码代理的实现差异

上一篇讲到自动代理创建器命中 Pointcut 后会调用 createProxy()——这一步具体是怎么"造出"一个代理对象的？Spring AOP 底层其实有两条完全不同的技术路线。

代理的本质：换一个"接头人"

代理要解决的问题很简单：在不修改原始类代码的前提下，让方法调用先经过一层"拦截"逻辑，再决定要不要真正执行原方法。 实现方式无非两种思路：一是"造一个实现相同接口的新类"，二是"造一个继承目标类的子类"——这正好对应 JDK 动态代理和 CGLIB 两条路线。

类比：你想找人代收快递又不想暴露自己的联系方式。方案一（JDK 动态代理）：找一个"和你签了同一份委托协议"的代收点，快递员只认协议，不关心背后具体是谁在处理——前提是必须有这份协议（接口）存在。方案二（CGLIB）：直接"克隆"你的身份信息造一个几乎一样的分身，分身长得和你一模一样（继承你的类），能处理你能处理的所有事情，但如果你是"绝密身份、禁止复制"（final 类），这个方案就行不通。

JDK 动态代理：基于接口

JDK 动态代理要求目标类必须实现至少一个接口，代理类在运行时动态生成，实现和目标类相同的接口，内部持有一个 InvocationHandler，所有接口方法的调用都会被转发到 invoke() 方法。

JDK 动态代理示例
UserService target = new UserServiceImpl();

UserService proxy = (UserService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),   // 关键：要求目标类实现了接口
    (proxyObj, method, args) -> {
        // 前置增强逻辑
        Object result = method.invoke(target, args);  // 反射调用真实方法
        // 后置增强逻辑
        return result;
    }
);

生成的代理类和 target 是平级兄弟关系——两者都实现 UserService 接口，代理类本身并不是 UserServiceImpl 的子类。这就意味着：如果代码里用 UserServiceImpl（具体类）而不是 UserService（接口）来接收这个代理对象，会直接抛 ClassCastException。

CGLIB：基于继承的字节码增强

CGLIB（Code Generation Library）不依赖接口，而是用 ASM 字节码技术，在运行时动态生成一个目标类的子类，重写所有可以被重写的方法， 在重写的方法里插入拦截逻辑（MethodInterceptor），再调用父类（也就是原始目标类）的方法。

CGLIB 示例
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserServiceImpl.class);  // 关键：以继承方式生成子类
enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
    // 前置增强逻辑
    Object result = proxy.invokeSuper(obj, args);  // 调用父类（原始类）的方法
    // 后置增强逻辑
    return result;
});
UserServiceImpl proxy = (UserServiceImpl) enhancer.create();

正因为是继承关系，目标类不能是 final（final 类不能被继承），目标方法也不能是 final 或 private（无法被子类重写），否则该方法不会被增强，调用时会绕过代理逻辑直接执行原始实现——这是一个非常容易踩的坑。

两者对比

维度JDK 动态代理CGLIB
实现方式实现同样的接口生成目标类的子类（继承）
前提条件目标类必须实现至少一个接口目标类不能是 final，方法不能是 final/private
依赖JDK 原生 java.lang.reflect.Proxy，无需额外依赖需要引入 CGLIB 库（Spring 已内置）
性能（历史印象）早期版本创建代理较快，调用稍慢早期版本创建代理较慢，调用较快
类型转换只能转换成接口类型，不能转换成实现类可以直接转换成目标类的类型（因为是子类）
关于性能的误区：网上很多资料还在说"CGLIB 创建代理慢、调用快，JDK 相反"，这是基于很老的 JDK/CGLIB 版本得出的结论。现代 JDK（尤其是 JDK 8+ 之后）动态代理的性能已经大幅优化，两者性能差异在实际业务场景中基本可以忽略，不应该把性能作为选择依据，选择的关键还是"有没有接口"和团队的一致性偏好。

Spring AOP 的选择策略

Spring 在 DefaultAopProxyFactory 里决定用哪种方式创建代理，核心逻辑（简化）：

if (目标类实现了至少一个接口 && 没有强制要求使用 CGLIB) {
  使用 JDK 动态代理
} else {
  使用 CGLIB（目标类没有接口，或显式配置了 proxyTargetClass=true）
}

这是传统 Spring（Spring MVC 时代）的默认策略：有接口优先用 JDK 动态代理。但从 Spring Boot 2.x 开始，默认把 spring.aop.proxy-target-class 设为 true， 也就是说不管目标类有没有接口，一律使用 CGLIB。这么改的原因是为了行为一致性：如果同一个项目里，有接口的类走 JDK 代理、没接口的类走 CGLIB 代理， 会导致"能不能强转成实现类"这种细节行为在不同 Bean 上表现不一致，增加了心智负担；统一用 CGLIB 可以消除这种不一致，代价是牺牲了一点"面向接口编程"的纯粹性。

该行为可以通过配置显式关闭，回退到"有接口用 JDK、无接口用 CGLIB"的传统策略：

application.yml
spring:
  aop:
    proxy-target-class: false  # 关闭后恢复"有接口优先 JDK 代理"的旧策略

常见误区

误区一：在 @Transactional、@Async 等注解生效失败时，没意识到很可能是"同类内部方法互相调用"——因为代理对象是在外部包了一层，类内部 this.method() 的调用不会经过代理，不管是 JDK 代理还是 CGLIB 都存在这个问题，是 AOP 领域公认的高频坑。
面试高频："JDK 动态代理和 CGLIB 的区别？" 从三个维度答：实现原理（接口 vs 继承）、使用前提（必须有接口 vs 不能是 final）、Spring 的默认策略（Spring Boot 2.x 后默认统一用 CGLIB）。这道题几乎是 Spring 面试必考。

核心价值总结

  • 两条路线本质是"接口 vs 继承"的取舍：JDK 代理轻量但要求接口，CGLIB 灵活但受 final 限制，没有绝对的优劣，只有适用场景的不同。
  • Spring Boot 默认统一 CGLIB 是工程权衡：为了消除"不同 Bean 代理行为不一致"带来的心智负担，宁可牺牲一点点性能和"纯接口编程"的洁癖。
