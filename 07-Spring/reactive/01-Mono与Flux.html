响应式编程

Mono 与 Flux

为什么一个已经用了十几年、性能足够好的 Spring MVC 之外，还需要一整套新的响应式技术栈？先搞清楚这个根本问题，再看 Reactor 提供的两个核心类型。

一句话讲清楚

响应式编程（Reactive Programming）解决的是"高并发下线程资源不够用"的问题。 传统同步阻塞模型下，一个请求占用一个线程，线程在等待 I/O（查数据库、调下游接口）时什么都不做，纯粹在"占着茅坑不拉屎"； 响应式模型让少量线程通过事件循环（Event Loop）处理海量并发连接——发起 I/O 后线程立刻被释放去处理别的请求，I/O 完成时通过回调通知，再把结果继续往下传。

类比：传统模型像餐厅里"一个服务员从点菜守到上菜全程盯着一桌客人"，客人在厨房等菜的这段时间服务员完全闲置却不能去服务别桌； 响应式模型像"一个服务员同时观察十桌客人，谁的菜好了就去谁那上菜，中间空闲时间用来招呼别的桌"——同样一个服务员，吞吐量高了一个数量级。

和你已经学过的线程模型对比

在 02-并发模块你学过：Java 线程是操作系统线程的封装，创建和上下文切换都有真实开销（栈内存、内核调度）。 传统 Tomcat（Spring MVC 用的默认容器）是"一个请求一个线程"（Thread-per-Request）模型：

Spring MVC（阻塞模型）
  请求进来 → 从线程池取一个线程 → 该线程独占处理整个请求
  遇到查数据库 → 线程阻塞等待，什么都不做
  数据库返回 → 线程继续执行 → 返回响应 → 线程归还线程池
  并发数 = 线程池大小，线程池撑满了新请求只能排队

WebFlux（响应式 / 事件循环模型）
  请求进来 → 少量固定线程（通常等于 CPU 核数）轮询事件
  遇到查数据库 → 注册回调，线程立刻释放去处理别的事件
  数据库返回数据 → 触发回调 → 换一个空闲线程继续处理
  并发数不受线程数限制，只受内存和连接数限制

这和 Node.js 单线程却能扛住海量并发的哲学完全一致：不是用更多线程堆吞吐量，而是让每个线程永远不空闲等待。 Netty（WebFlux 默认的服务器）本质上就是 Java 版的事件循环实现。

Reactor 的两个核心类型

Spring WebFlux 底层用的响应式库是 Project Reactor（Spring 官方主导），它只提供两个核心类型，比 RxJava 简单得多：

类型含义类比
Mono<T> 异步返回 0 或 1 个元素的序列 类似 CompletableFuture<T>，"未来某个时刻会给我一个结果（或没有结果、或报错）"
Flux<T> 异步返回 0 到 N 个元素的序列 类似异步版的 Stream<T>，"未来会陆续给我一批结果"

两者都是"发布者"（Publisher）：在没有人订阅（subscribe）之前，它们只是一份"待执行的计划"，什么都不会发生——这叫惰性求值（Lazy），是响应式编程的一大特点，也是初学者最容易忽略的一点。

陷阱：创建了 Mono/Flux 但没有调用 subscribe()（或被框架自动订阅，比如作为 Controller 返回值），整条链路根本不会执行。 新手常见的"我这段代码怎么没反应"，十有八九是漏了订阅。

创建 Mono / Flux

MonoFluxDemo.java（创建）
// Mono：0-1 个元素
Mono<String> mono1 = Mono.just("hello");          // 直接包装一个值
Mono<String> mono2 = Mono.empty();               // 0 个元素
Mono<String> mono3 = Mono.fromCallable(() -> queryFromDb()); // 包装一个耗时调用

// Flux：0-N 个元素
Flux<Integer> flux1 = Flux.just(1, 2, 3);
Flux<Integer> flux2 = Flux.fromIterable(list);
Flux<Long>    flux3 = Flux.interval(Duration.ofSeconds(1)); // 每秒发一个递增数字，无限流

常用操作符：map / filter / flatMap

MonoFluxDemo.java（操作符）
Flux.just(1, 2, 3, 4)
    .filter(n -> n % 2 == 0)          // 同步转换：偶数过滤
    .map(n -> n * 10)              // 同步转换：值本身做变换
    .flatMap(n -> callRemoteApi(n)) // 异步转换：每个元素换成一个新的 Mono/Flux 并"拍平"
    .subscribe(result -> System.out.println(result)); // 订阅后整条链才会真正执行
map vs flatMap：map 是"同步变换值"（T -> R），flatMap 是"异步变换并展开"（T -> Mono<R> / T -> Flux<R>）。 这和 Java Stream 里 map/flatMap 的关系是同一套思路，唯一区别是这里每一步都可能是异步的。面试高频：能不能说清楚这两者的区别，是判断是否真正理解响应式操作符链的分水岭。

核心价值总结

  • 响应式编程的本质：用少量线程通过事件循环处理海量并发连接，遇到 I/O 不阻塞而是注册回调，把"等待"的时间让给别的任务。
  • Mono 是 0-1 个元素的异步序列（类似 CompletableFuture），Flux 是 0-N 个元素的异步序列（类似异步 Stream）。
  • Mono/Flux 是惰性的，不订阅就不会执行，这是和普通对象/集合最大的思维差异。
  • map 做同步值变换，flatMap 做异步变换并展开，是最常用也最容易混淆的两个操作符。
