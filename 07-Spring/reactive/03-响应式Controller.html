响应式编程

响应式 Controller

知道了 Mono/Flux 是什么之后，落地到 Spring WebFlux 里其实非常简单——Controller 方法不再直接返回结果，而是返回"一份稍后会给出结果的承诺"。

一句话讲清楚

Spring WebFlux 的 Controller 写法和 Spring MVC 几乎一模一样，唯一的关键区别是：方法返回值从"直接返回结果对象"变成"返回 Mono<T>/Flux<T>"。 容器（Netty）拿到这个 Mono/Flux 之后，会异步订阅它，等数据真正就绪时才写回响应，处理请求的线程在这期间可以立刻去处理别的请求。

对比：Spring MVC vs Spring WebFlux

UserController.java（Spring MVC — 传统同步阻塞）
@RestController
@RequestMapping("/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        // 方法在这里"卡住"，直到数据库查询完成才能返回
        // 这段时间内，这个线程被 id 这一个请求完全占用
        return userService.findById(id);
    }
}
UserController.java（Spring WebFlux — 响应式非阻塞）
@RestController
@RequestMapping("/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    @GetMapping("/{id}")
    public Mono<User> getUser(@PathVariable Long id) {
        // 立刻返回一个 Mono，方法本身不阻塞
        // 容器会在数据真正查出来时才把响应写回给客户端
        return userService.findById(id);
    }

    @GetMapping
    public Flux<User> listUsers() {
        // 返回一批数据用 Flux，客户端可以边收边处理（流式响应）
        return userService.findAll();
    }
}
表面上只是把返回类型从 User 换成 Mono<User>，但背后的执行模型完全不同：MVC 版本执行 getUser 方法的线程会一直卡在 findById 里；WebFlux 版本执行 getUser 方法的线程立刻返回（返回的只是一个"计划"），真正查数据库的动作是异步发起的，另一个线程在结果就绪时才去写响应。

函数式路由：另一种写法风格

除了熟悉的注解式 Controller（@RestController + @GetMapping），Spring WebFlux 还提供了一套函数式编程风格的路由方式：RouterFunction + HandlerFunction。 两者功能等价，只是编码风格不同——注解式是声明式（"这个路径归这个方法管"），函数式是组合式（把路由规则像搭积木一样拼出来）。

UserRouter.java（函数式路由）
@Configuration
public class UserRouter {

    @Bean
    public RouterFunction<ServerResponse> userRoutes(UserHandler handler) {
        // 把 URL 匹配规则和处理函数显式组合在一起，而不是靠注解隐式关联
        return RouterFunctions.route()
                .GET("/users/{id}", handler::getUser)
                .GET("/users",      handler::listUsers)
                .build();
    }
}

@Component
@RequiredArgsConstructor
class UserHandler {

    private final UserService userService;

    public Mono<ServerResponse> getUser(ServerRequest req) {
        Long id = Long.valueOf(req.pathVariable("id"));
        return userService.findById(id)
                .flatMap(user -> ServerResponse.ok().bodyValue(user))
                .switchIfEmpty(ServerResponse.notFound().build());
    }
}
生产实践中注解式 Controller 仍是绝对主流，函数式路由更多用在极致轻量场景（比如 Serverless 冷启动优化）或者团队偏好函数式编程风格的项目。作为响应式编程的一部分了解即可，不需要都精通。

Spring MVC vs Spring WebFlux 取舍对比

维度Spring MVCSpring WebFlux
底层服务器 Servlet 容器（Tomcat / Jetty），线程池模型 默认 Netty，事件循环模型
并发模型 一个请求一个线程，阻塞等待 I/O 少量线程处理海量连接，I/O 非阻塞
编程心智负担 低，同步代码符合直觉，易调试（堆栈清晰） 高，链式操作符 + 异步回调，调试和排错更难
适用场景 绝大多数中低并发的常规业务系统 高并发网关/代理、大量下游调用聚合、长连接推送场景
生态成熟度 非常成熟，几乎所有库都原生支持 要求全链路非阻塞，混用阻塞库会前功尽弃（见下一篇陷阱）
不要为了"响应式"而响应式：如果你的系统并发量不高、瓶颈也不在 I/O 等待，硬切 WebFlux 只会徒增复杂度而拿不到收益。响应式编程是应对"高并发 + I/O 密集"场景的工具，不是万能升级。

核心价值总结

  • WebFlux 的 Controller 写法和 MVC 几乎一致，核心区别是返回值从直接结果变成 Mono<T>/Flux<T>。
  • 函数式路由（RouterFunction + HandlerFunction）是注解式 Controller 的等价替代风格，日常了解即可。
  • Spring MVC 简单直观、生态成熟，适合大多数业务；WebFlux 换来高并发吞吐量，代价是更陡的学习曲线和更难调试。
  • 技术选型要基于真实的并发/I/O 瓶颈判断，而不是盲目追新。
