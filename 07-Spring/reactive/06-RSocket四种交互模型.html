RSocket 四种交互模型（进阶）

本组最后一篇，介绍一个相对小众但值得知道存在的技术：HTTP 的"请求-响应"模型并不是唯一选择，RSocket 天生支持四种交互方式。

一句话讲清楚

HTTP 协议的通信模式本质上只有一种：请求-响应（Request-Response）。 客户端发一个请求，服务端必须返回恰好一个响应，这个假设在很多场景下并不贴合实际需求——比如你只是想"通知一下"不需要回应，或者你想"订阅一个会持续更新的数据流"。 工程师们过去只能在 HTTP 之上"曲线救国"（轮询、长轮询、WebSocket、SSE 分别应对不同场景，彼此互不兼容）。

RSocket 是一个应用层协议，从设计之初就原生支持四种交互模型，并且天然内置背压支持（呼应第 2 篇讲的背压概念），Spring Framework 提供了对它的原生支持。

四种交互模型

模型行为适用场景
Request-Response
请求-响应		 发一个请求，等一个响应 和 HTTP 语义等价，常规的"查一条数据"场景
Fire-and-Forget
发了不管		 发一个请求，不等待也不关心响应 日志上报、埋点、通知类场景——发送方不需要知道对方是否处理完成
Request-Stream
请求-流		 发一次请求，换回一个持续的数据流 订阅股票行情、订阅日志流——一次订阅，服务端持续推送
Channel
双向通道		 双方都能持续互相发送数据流，全双工 实时协作场景，比如双向语音/视频信令交换、多人协同编辑
HTTP：只有 Request-Response 一种模式
  想做流式推送 → 只能用 WebSocket / SSE 这类"补丁方案"，协议层面各自独立

RSocket：Request-Response / Fire-and-Forget / Request-Stream / Channel
  四种模型统一在同一套协议、同一条连接之上，按场景直接选用对应模型

Spring 中的基本用法（了解即可）

RSocketController.java（示意，四种模型对应的注解）
@Controller
public class RSocketController {

    // Request-Response：一次请求换一次响应，和 @GetMapping 语义类似
    @MessageMapping("request-response")
    public Mono<String> requestResponse(String request) { ... }

    // Fire-and-Forget：返回 Mono<Void>，调用方发完即走
    @MessageMapping("fire-and-forget")
    public Mono<Void> fireAndForget(String request) { ... }

    // Request-Stream：一次请求，返回持续的 Flux 数据流
    @MessageMapping("request-stream")
    public Flux<String> requestStream(String request) { ... }

    // Channel：入参和返回值都是 Flux，双向持续通信
    @MessageMapping("channel")
    public Flux<String> channel(Flux<String> requests) { ... }
}
了解到什么程度就够了：RSocket 是一个相对新颖、相对小众的技术，绝大多数业务系统用 HTTP + WebSocket 的组合已经能覆盖所有场景。 这一篇的目标只是知道"RSocket 解决了 HTTP 交互模型太单一这个问题，原生支持四种交互模型且内置背压"——面试或技术选型讨论时能说清楚这个概念即可，不需要深入实践或在生产项目里落地。

核心价值总结

  • HTTP 天生只有请求-响应一种交互模式，流式推送等场景只能靠 WebSocket/SSE 这类"外挂方案"打补丁。
  • RSocket 原生支持 Request-Response / Fire-and-Forget / Request-Stream / Channel 四种交互模型，统一在同一套协议之上。
  • RSocket 内置背压支持，理念上和 Reactor/Reactive Streams 是同一套哲学的延伸。
  • 这是一个进阶、小众的知识点，掌握"它解决了什么问题"这个概念层面即可，不必深入实践。
