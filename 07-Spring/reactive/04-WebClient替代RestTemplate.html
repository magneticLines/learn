WebClient 替代 RestTemplate

调第三方接口时线程该不该"傻等"？这一篇讲清楚 RestTemplate 的历史局限，以及新手最容易在 WebClient 上犯的一个致命错误。

一句话讲清楚

RestTemplate 是同步阻塞式 HTTP 客户端：调用它发起请求后，当前线程会一直卡在那里，直到对方接口返回响应才能继续往下执行。 Spring 官方从 5.0 起已将 RestTemplate 标记为维护模式（maintenance mode）——不再新增功能，只做必要修复——推荐用 WebClient 作为替代品：非阻塞、响应式，同时也能兼容传统同步调用方式。

类比上一篇：如果说 Controller 层从阻塞变非阻塞是"接单员不再傻等厨房出餐"，那么这一篇讲的是"接单员打电话问供应商要货时，是守着电话等对方说完，还是挂了电话该干嘛干嘛、等对方回电"。RestTemplate 是前者，WebClient 是后者。

WebClient 的两种用法

WebClient 本质上返回的也是 Mono/Flux，怎么用取决于你调用它的位置在不在响应式链路里。

用法一：.block() 同步获取（在非响应式代码中使用）
WebClient client = WebClient.create("https://api.example.com");

// block() 会把异步结果强行转成同步——当前线程在这里等待，直到拿到结果
// 只应该在普通同步方法里、明确不在响应式调用链上时使用
User user = client.get()
        .uri("/users/{id}", id)
        .retrieve()
        .bodyToMono(User.class)
        .block(); // 阻塞等待，效果等同于旧的 RestTemplate
用法二：整条响应式链路，配合 Controller 一路异步到底
@GetMapping("/orders/{id}/user")
public Mono<User> getOrderUser(@PathVariable Long id) {
    // 不调用 block()，而是把 Mono 继续往下传
    // 线程发起请求后立刻释放，数据到达时由事件循环触发后续处理
    return client.get()
            .uri("/users/{id}", id)
            .retrieve()
            .bodyToMono(User.class); // 直接返回 Mono，交给 WebFlux 容器订阅
}

最容易犯的错误：在响应式链里调用 .block()

反例：在 WebFlux Controller 里混用阻塞调用
@GetMapping("/orders/{id}/user")
public Mono<User> getOrderUser(@PathVariable Long id) {
    // 危险！这里的 block() 会让当前的事件循环线程真正"卡住"
    // 而事件循环线程数量极少（通常 = CPU 核数），一旦被占住
    // 这个线程能处理的其它所有并发请求全部跟着被拖慢
    User user = client.get()
            .uri("/users/{id}", id)
            .retrieve()
            .bodyToMono(User.class)
            .block();
    return Mono.just(user);
}
陷阱（面试高频）：Spring MVC 里线程池有几十上百个线程，一个线程阻塞几十毫秒无伤大雅；但 WebFlux 的事件循环线程通常只有几个（等于 CPU 核数）， 阻塞其中一个，就等于让一部分并发请求集体排队等待。这相当于开着跑车却把手刹一直拉着——响应式带来的全部吞吐量优势被这一次 block() 抵消得一干二净。 Reactor 甚至会在检测到某些场景下的阻塞调用时主动抛出 BlockingOperationError 来提醒你。

原则：一旦进入响应式调用链（Controller 返回 Mono/Flux），链路上的每一步都应该保持非阻塞，用 map/flatMap 组合下去，而不是中途调用 .block() 把异步"拉回"同步。.block() 只应该出现在你确定自己身处普通同步方法、并且能接受这次调用付出线程等待代价的地方（比如程序 main 方法、定时任务里的一次性调用）。

RestTemplate vs WebClient

维度RestTemplateWebClient
调用模型 同步阻塞 非阻塞，返回 Mono/Flux；也可通过 .block() 兼容同步用法
官方状态 维护模式，不再新增功能 官方推荐，持续迭代
底层依赖 阻塞式 HTTP 客户端（如 JDK HttpURLConnection / Apache HttpClient） 默认基于 Reactor Netty，也可切换其它响应式 HTTP 客户端
适用场景 传统 Spring MVC 项目里的存量代码 新项目首选；尤其是 WebFlux 项目里必须用它，否则前功尽弃

核心价值总结

  • RestTemplate 已进入维护模式，新项目应默认选择 WebClient。
  • WebClient 同时支持 .block()（同步获取，兼容传统写法）和 subscribe()/继续链式组合（异步非阻塞）两种用法。
  • 最容易踩的坑：在响应式 Controller 内部调用 .block()，会阻塞宝贵的事件循环线程，抵消响应式编程的全部收益。
  • 原则：一旦身处响应式链路，就把非阻塞坚持到底；.block() 只用在确定处于普通同步上下文的地方。
