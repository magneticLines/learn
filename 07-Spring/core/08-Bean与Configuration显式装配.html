IoC 核心与自动配置

@Bean 与 @Configuration 显式装配

组件扫描不是万能的——第三方库的类你没法在它源码上加注解。这时候需要换一种装配方式：自己写一个"配置类"，手动声明 Bean。

一句话讲清楚

组件扫描（@Component 系列）适用于"你自己写的类"，而 @Configuration + @Bean 适用于"你没法修改源码的类"（第三方库的类）或者 "需要根据条件动态决定怎么创建"的场景。前者是"贴标签让容器自己发现"，后者是"你亲手写一个工厂方法告诉容器怎么造"。

什么时候组件扫描用不了

典型场景：想把某个第三方库的类（比如一个 HTTP 客户端、一个第三方 SDK 的 Client）交给容器管理，但你根本没法在它的源码上加 @Service。

ThirdPartyConfig.java（@Configuration + @Bean 手动装配）
@Configuration
public class ThirdPartyConfig {

    // 第三方类 OkHttpClient，没法在它源码上加注解，只能用 @Bean 手动声明
    @Bean
    public OkHttpClient okHttpClient() {
        return new OkHttpClient.Builder()
                .connectTimeout(5, TimeUnit.SECONDS)
                .build();
    }

    // 依赖上面的 OkHttpClient Bean，参数直接由容器注入
    @Bean
    public PaymentGatewayClient paymentGatewayClient(OkHttpClient client) {
        return new PaymentGatewayClient(client, "https://api.pay.example.com");
    }
}

方法上标 @Bean 就相当于告诉容器："调用这个方法拿到的返回值，注册为一个 Bean"。方法名默认就是 Bean 名称，方法参数会自动从容器里按类型注入（上例中 client 参数会拿到 okHttpClient() 那个 Bean）。

新手困惑：@Configuration 类里方法互相调用，会创建两个实例吗

如果在同一个 @Configuration 类里，一个 @Bean 方法内部直接调用另一个 @Bean 方法（而不是通过参数注入），会不会变成"手动 new"从而绕过容器？

方法内部直接调用另一个 @Bean 方法
@Configuration
public class AppConfig {

    @Bean
    public ConnectionPool connectionPool() {
        return new ConnectionPool();
    }

    @Bean
    public OrderRepository orderRepository() {
        // 直接调用方法而不是靠参数注入——这样会创建两个 ConnectionPool 吗？
        return new OrderRepository(connectionPool());
    }

    @Bean
    public PaymentRepository paymentRepository() {
        return new PaymentRepository(connectionPool()); // 也调用了一次
    }
}

答案：不会，两处调用拿到的是同一个 ConnectionPool 单例。原因是 @Configuration 类会被 Spring 用 CGLIB 生成一个子类代理，重写了所有 @Bean 方法：

调用 connectionPool() 时，实际执行的是 CGLIB 代理后的逻辑：
  ① 检查容器的单例池里是否已经有名为 "connectionPool" 的 Bean
  ② 有 → 直接返回缓存的实例，不会重新执行方法体里的 new
  ③ 没有 → 才真正执行方法体创建新实例，并存入单例池
这就是为什么普通类调用自己的方法只是"走一遍方法体"，而 @Configuration 类里调用 @Bean 方法却带有"单例拦截"效果——本质是方法调用被 CGLIB 代理拦截了，这也是 AOP 动态代理思想的一次具体应用（承接 02-AOP.html 里讲过的代理机制）。
前提条件：这个"自动去重"只在 full 模式的 @Configuration（默认，proxyBeanMethods = true）下成立。 如果显式写成 @Configuration(proxyBeanMethods = false)（lite 模式，性能更好，Spring Boot 自动配置类大量使用）， 方法内部互相调用就会退化成普通方法调用，确实会创建出两个不同的实例——这是容易踩的坑，写 lite 模式配置类时应该始终通过方法参数注入依赖，而不是直接调用另一个 @Bean 方法。

组件扫描 vs @Bean 显式装配，怎么选

场景推荐方式
自己写的业务类（Service/Repository/Controller）组件扫描（@Service 等）
第三方库的类，没法加注解@Configuration + @Bean
创建过程需要复杂逻辑（比如根据配置动态选择实现、构建 Builder 链）@Configuration + @Bean
Spring Boot 自动配置类内部（追求启动性能）@Configuration(proxyBeanMethods = false) + 参数注入

核心价值总结

  • 组件扫描管"自己的类"，@Configuration + @Bean 管"没法加注解的类"或"需要手写创建逻辑的场景"。
  • @Bean 方法参数会自动从容器按类型注入，这是获取其他 Bean 依赖的推荐方式。
  • full 模式（默认）的 @Configuration 类会被 CGLIB 代理，方法内部互相调用也能保证单例；lite 模式（proxyBeanMethods = false）没有这个保证。
  • 这背后是 AOP 动态代理思想的具体应用，和 02-AOP.html 讲的代理机制是同一套底层原理。

与其他知识点的联系

  • 组件扫描注解（第 6 篇）：本篇一开始就点明了组件扫描的边界——只能管"自己写的类"；@Configuration + @Bean 正是补上这个边界之外的空白（第三方类、需要复杂创建逻辑的场景）。两者共同构成了 Spring 装配 Bean 的完整两条路线，理解各自的适用边界比死记语法更重要。
  • @EnableAutoConfiguration 与条件化装配（第 10 篇）：本篇讲的 lite 模式（proxyBeanMethods = false）并不是个孤立的性能选项——第 10 篇会看到 Spring Boot 官方的上百个 XxxAutoConfiguration 类几乎全部用 lite 模式，因为容器启动时要扫描解析这些类，去掉 CGLIB 代理能显著减少启动开销，这也是"为什么普通业务配置类不用太纠结这个参数，但框架级自动配置类必须用"的原因。
  • @ConditionalOnXxx 系列注解（第 11 篇）：本篇的 @Bean 方法在没有条件注解时是"无条件生效"的；第 11 篇会在 @Bean 方法上叠加 @ConditionalOnMissingBean 等条件注解，让"要不要执行这个 @Bean 方法"从"写了就一定生效"变成"满足条件才生效"——这是从"显式装配"到"条件化装配"的自然演进。
  • IoC 容器底层（CGLIB 代理与单例保证）：本篇讲的"full 模式下 @Configuration 类被 CGLIB 代理、方法内部调用会被拦截并查单例池"，本质上是 AOP 动态代理思想在容器自身实现里的一次具体落地——这也提示了一个更普遍的原理：Spring 容器为了保证"同一个 Bean 名称只对应一个实例"这条核心承诺，会在必要的地方主动插入代理层，而不是依赖开发者自觉遵守约定。
