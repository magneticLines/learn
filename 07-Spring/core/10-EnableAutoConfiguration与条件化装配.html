IoC 核心与自动配置

@EnableAutoConfiguration 与条件化装配

本组的重点：拆开"约定优于配置"这句口号背后的真实机制——Spring Boot 到底怎么知道该帮你配哪些 Bean。

一句话讲清楚

自动配置的本质是：容器启动时批量加载一份"候选配置类清单"，然后逐个用条件注解判断"要不要真正生效"。 这份清单是静态的、写死在 jar 包里的文本文件；真正决定某个配置类生不生效的，是运行时的条件判断（下一篇细讲）。

触发入口：@EnableAutoConfiguration

第 5 篇讲过，@SpringBootApplication 内部组合了 @EnableAutoConfiguration。这个注解真正做的事情，是注册了一个特殊的组件 AutoConfigurationImportSelector，容器启动扫描到它时，会触发"加载自动配置候选类清单"的逻辑。

清单存在哪：AutoConfiguration.imports

Spring Boot 3 开始，这份清单的位置是每个 jar 包里的固定路径：

META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（片段）
org.springframework.boot.autoconfigure.web.servlet.DispatcherServletAutoConfiguration
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.jackson.JacksonAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration
... 还有上百个 XxxAutoConfiguration

这就是一份纯文本文件，每行一个全限定类名。容器启动时会扫描 classpath 下所有 jar 包里符合这个路径的文件，把内容全部汇总成一份完整的候选清单—— 这也是为什么"只要引入某个 Starter 依赖，相关的自动配置就自动生效"：Starter 依赖的那个 jar 包里带着这份清单，清单里的类被自动加载了。

版本坑（面试常问）：Spring Boot 2.x 及更早版本，这份清单写在 META-INF/spring.factories 文件里， 格式是 key=value（key 固定是 org.springframework.boot.autoconfigure.EnableAutoConfiguration）。 Spring Boot 3 把它拆分成了独立的 AutoConfiguration.imports 文件，好处是职责更单一、加载解析更快。 如果你在网上搜到的教程还在讲 spring.factories，那大概率是针对 Spring Boot 2.x 的旧文章。

光"加载"还不够：为什么需要"条件"

上百个 XxxAutoConfiguration 类会被全部加载进候选清单，但显然不能全部生效——没有引入数据库驱动的项目， 没道理去创建一个 DataSource Bean。这就是为什么每个自动配置类上都密密麻麻标注了条件注解：

DataSourceAutoConfiguration（简化示意）
@AutoConfiguration
@ConditionalOnClass(DataSource.class)          // classpath 里得有 DataSource 这个类
@ConditionalOnMissingBean(DataSource.class)   // 用户没有自己定义 DataSource 时才生效
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource(DataSourceProperties props) { ... }
}

整个流程可以画成一张图：

① 扫描清单：读取 classpath 下所有 jar 包的 AutoConfiguration.imports
  → 汇总成一份包含上百个 XxxAutoConfiguration 类名的候选列表

② 条件过滤：逐个类检查其上的 @ConditionalOnXxx 注解
  classpath 没相关依赖 → 跳过
  用户已自定义同类型 Bean → 跳过
  配置文件没打开对应开关 → 跳过

③ 真正生效：条件全部满足的配置类，其中的 @Bean 方法才会被执行
  → 汇入普通的 IoC 容器 Bean 创建流程（复用 04-工程实战讲过的"扫描 → 实例化注入 → 初始化"三阶段）
"约定优于配置"的技术本质到这里就讲清楚了：不是 Spring Boot 有什么"智能推断"的魔法，而是把所有可能用到的配置都提前写好、放进候选清单， 再用一堆条件注解做"运行时筛选"——本质上是一次性穷举 + 条件裁剪，而不是某种动态推理。理解了这点，遇到自动配置没生效的问题时， 第一反应就应该是"去看这个 AutoConfiguration 类上标了什么条件，是不是有条件没满足"，而不是当成黑盒排查。

核心价值总结

  • 自动配置分两步：先加载候选清单（AutoConfiguration.imports），再用条件注解逐个筛选是否生效。
  • Spring Boot 3 用 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports，取代了 2.x 的 spring.factories。
  • "约定优于配置"不是玄学，是"穷举所有可能配置 + 条件注解运行时裁剪"这套朴素机制的效果。
  • 下一篇具体展开 @ConditionalOnClass / @ConditionalOnMissingBean / @ConditionalOnProperty 这几个最常用的条件注解。

与其他知识点的联系

  • 起步依赖 Starter 机制（第 4 篇）：本篇讲的"AutoConfiguration.imports 清单为什么会出现在候选列表里"，答案就在第 4 篇——Starter 依赖链把对应的 autoconfigure jar 包拉进了 classpath，这份 jar 包里自带的清单文件才有机会被容器扫描到。没有第 4 篇的 Starter 引入，本篇讲的整套自动配置清单加载机制根本没有触发的物质基础。
  • 主启动类与 @SpringBootApplication（第 5 篇）：第 5 篇只交代了 @SpringBootApplication 里组合了 @EnableAutoConfiguration 这一件事，本篇把这个注解背后"注册 AutoConfigurationImportSelector → 扫描清单 → 条件过滤"的完整链路展开——两篇是"知道有这个开关"和"搞懂这个开关怎么工作"的递进关系。
  • @ConditionalOnXxx 系列注解（第 11 篇）：本篇留了个悬念——"条件注解决定配置类要不要生效"，第 11 篇正是逐个拆解 @ConditionalOnClass / @ConditionalOnMissingBean / @ConditionalOnProperty 这几种最常用的条件实现。本篇讲的是"筛选流程"，第 11 篇讲的是"筛选依据"，缺了任何一篇都无法完整理解自动配置。
  • IoC 容器底层（BeanFactoryPostProcessor 与容器扩展点）：AutoConfigurationImportSelector 本质上是 Spring 提供的一种"在容器刷新过程中动态决定要注册哪些配置类"的扩展机制，它工作在常规组件扫描之后、Bean 实例化之前的阶段——理解这个时机，才能明白为什么用户手写的 @Configuration 类总是"先于"自动配置类被处理，进而支撑第 11 篇 @ConditionalOnMissingBean 让用户配置优先的效果。
