Spring 生态全景（Framework / Boot / Cloud 的关系）

新手学 Spring 最常见的困惑：Spring、Spring Boot、Spring Cloud 到底是什么关系？是三个框架，还是一个东西的三个名字？先把这张地图画清楚，后面每一篇才知道自己在地图的哪个位置。

一句话讲清楚

这三者不是并列关系，而是一层套一层的"地基 → 装修 → 联排别墅小区"关系： Spring Framework 是最底层的地基——IoC 容器 + AOP + 事务等一整套基础设施； Spring Boot 在地基上做"精装修"——自动配置 + 起步依赖，让你不用手工搭脚手架就能拎包入住； Spring Cloud 是在装修好的一栋栋房子（微服务）基础上，规划整个小区的水电煤气道路——服务发现、配置中心、网关这些跨服务的协调能力。

Spring Cloud（微服务治理：服务发现 / 配置中心 / 网关 / 熔断）

Spring Boot（自动配置 + 起步依赖 + 内嵌容器，解决"配置地狱"）

Spring Framework：地基

这是 2004 年就存在的"老家底"，核心是 IoC 容器（管理对象的创建和依赖关系）和 AOP（横切关注点织入）， 这两个概念在 04-工程实战/mechanism/01-IoC-DI.html 和 02-AOP.html 里已经讲透了，这里不再重复。 Spring Framework 本身还包含 Spring MVC、Spring Data、事务抽象等一大堆模块，但它们都需要手工配置才能用起来——这正是 Spring Boot 要解决的问题。

Spring Boot：精装修

在 Spring Boot 出现之前（约 2010 年代早期），搭一个 Spring MVC 项目要手写一堆 XML 配置：DispatcherServlet 怎么配、视图解析器怎么配、 数据源怎么配、事务管理器怎么配……新项目的前半天时间基本都在"配置地狱"里打转。

Spring Boot 的核心贡献是两样东西：起步依赖（Starter）（一个依赖引入一整套配套的包，版本还互相兼容，见下一篇）和 自动配置（Auto-Configuration）（根据 classpath 里有什么类，自动帮你把该配的 Bean 都配好，见本组第 10 篇）。 再加上内嵌 Tomcat，一个 main 方法就能启动一个完整的 Web 应用，这就是"开箱即用"的由来。

记住这句话：Spring Boot 不是一个新框架，它没有发明新的 IoC 容器，本质是"用一堆预设配置 + 依赖打包，把 Spring Framework 用起来的门槛降到最低"。

Spring Cloud：小区规划

当系统从单体拆成多个微服务后，会冒出一批新问题：服务之间怎么找到彼此（服务发现）、配置怎么统一管理（配置中心）、 请求怎么统一入口分发（网关）、某个服务挂了怎么不拖垮全局（熔断/限流）。这些问题跟单个应用内部的 IoC/AOP 已经没关系了， 是"多个 Spring Boot 应用之间"的协调问题，这就是 Spring Cloud 要解决的层面。没有 Spring Boot 也可以有 Spring Cloud 的思想（比如 Netflix 生态）， 但 Spring Cloud 的具体实现是构建在 Spring Boot 之上的。

"学 Spring 到底在学什么"

很多新手一上来就直接学 Spring Boot 用法（加注解、跑起来），跳过了 Spring Framework 的原理，结果遇到问题就懵——比如为什么循环依赖会报错、 为什么事务不生效、为什么某个 Bean 注入不进去。这些问题的答案都在 Spring Framework 这层地基里。

你在学本质是在学关键词
@Service / @Autowired 怎么用Spring Framework 的 IoC 容器BeanDefinition、依赖注入
为什么加个依赖就自动能用Spring Boot 的自动配置机制@ConditionalOnXxx、AutoConfiguration.imports
怎么让多个服务互相发现、统一配置Spring Cloud 的微服务治理组件注册中心、配置中心、网关

本组文章（IoC 核心与自动配置）聚焦在第一、第二层：先回顾 IoC/DI 的核心结论，再深入 Spring Boot 具体怎么在这套容器上做自动配置。Spring Cloud 不在本组范围内。

核心价值总结

  • Spring Framework 是地基：IoC 容器 + AOP，解决"对象怎么创建、怎么解耦"的问题。
  • Spring Boot 是精装修：自动配置 + 起步依赖，解决"怎么把 Spring Framework 快速用起来"的问题。
  • Spring Cloud 是小区规划：构建在 Spring Boot 之上，解决"多个微服务之间怎么协调"的问题。
  • 三者是依赖关系而非替代关系，遇到问题要先判断"这是哪一层的问题"，再去对应的知识体系里找答案。
