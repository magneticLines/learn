IoC / DI 的原理已经在工程实战模块讲透了，这里不重复展开，只做一次结论性回顾，然后正式转入本模块的主题：Spring 具体怎么实现这套容器。

已经讲过的结论（直接复用）

完整的原理、反例/正例代码、三种注入方式对比、循环依赖坑等内容都在 04-工程实战/mechanism/01-IoC-DI.html 里，这里只提炼出后续要反复用到的三条结论：

  • IoC 是原则，DI 是实现：对象不再自己 new 依赖，而是声明"我需要什么"，由容器负责创建和装配。
  • 容器启动分三步：扫描生成 BeanDefinition → 实例化并注入依赖 → 执行初始化回调，最终所有单例 Bean 进入单例池。
  • 构造器注入是官方推荐方式：依赖不可变、可写 final、循环依赖能在启动期尽早暴露、测试时不需要启动容器。
如果这三条结论有任何一条觉得陌生，建议先回去读完 01-IoC-DI.html 再继续本模块——本模块默认你已经理解"为什么要有 IoC 容器"这个前提。

这个模块接下来要讲什么

工程实战模块讲的是 IoC/DI 作为设计思想 解决了什么工程问题；本模块（IoC 核心与自动配置）要往下钻一层，讲 Spring 具体怎么实现 这套容器，以及更重要的—— Spring Boot 怎么在这套容器基础上做"自动配置"，让你几乎不用手写 @Bean 就能拿到一整套开箱即用的组件（数据源、消息队列客户端、Web 服务器……）。

与其他知识点的联系

  • Spring 生态全景：上一篇讲的"Spring Framework 是地基"，这个地基具体指的就是本篇回顾的 IoC 容器——没有 IoC/DI 这套机制，Spring Boot 的自动配置、Spring Cloud 的服务治理都无从谈起，三层地图的最底层就是这里。
  • Spring Initializr 创建项目：下一篇动手创建项目后，生成的 DemoApplication 主启动类上的 @SpringBootApplication 注解会触发组件扫描，本质就是本篇讲的"容器启动第一步：扫描生成 BeanDefinition"，理解了这一步才能理解为什么加个 @Component 注解就能被自动装配。
  • @ConfigurationProperties 配置绑定：配置类本质上也是一个受 IoC 容器管理的普通 Bean，只是在实例化之后多了一步"从配置源绑定字段值"，这一整套流程依然遵循本篇总结的"扫描 → 实例化并注入依赖 → 初始化回调"三步骤。
  • @Async / @Scheduled 与 AOP 代理：本篇提到构造器注入是官方推荐方式，而 @Async、@Transactional 等注解依赖的是容器为 Bean 生成的代理对象——只有经过代理的调用才能被拦截，这也是为什么理解"容器管理的是代理对象而非原始对象"对排查这类注解失效问题至关重要。
  • 消息队列监听器（@RabbitListener / @KafkaListener）：这些监听器所在的类同样是被 IoC 容器创建和管理的 Bean，容器负责把 RabbitTemplate/KafkaTemplate 等依赖注入进去，业务代码只需要声明"我需要什么"，这正是本篇强调的"IoC 是原则，DI 是实现"在消息驱动场景下的具体应用。
