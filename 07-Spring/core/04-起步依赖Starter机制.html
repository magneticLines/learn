IoC 核心与自动配置

起步依赖（Starter）机制

为什么勾一个 "Spring Web" 就能同时拿到 Spring MVC、内嵌 Tomcat、Jackson，而且它们的版本刚好都兼容？答案是 Starter——一个"没有代码、只管拉依赖"的胖依赖。

一句话讲清楚

Starter 本质是一个空壳 Maven 模块：它自己不包含任何业务代码，只在 pom.xml 里声明了一组"配套齐全、版本互相兼容"的依赖。 引入一个 Starter，就等于间接引入了它背后的一整条依赖链。

反例：没有 Starter 之前要怎么做

如果没有 spring-boot-starter-web，想手搭一个 Spring MVC + 内嵌 Tomcat 的项目，得自己一个个引入并对齐版本：

pom.xml（反例 — 手动拼凑一堆零散依赖）
<dependency> spring-webmvc <!-- 版本？ --> </dependency>
<dependency> tomcat-embed-core <!-- 版本要匹配 Spring 版本 --> </dependency>
<dependency> jackson-databind <!-- 版本要匹配 Jackson 全家桶 --> </dependency>
<dependency> validation-api <!-- 还要再手动引一堆... --> </dependency>

手工维护版本兼容矩阵，一旦升级 Spring 版本，这几个依赖的版本号要联动调整，稍不留神就是 NoSuchMethodError 或 ClassNotFoundException。

正例：一个 Starter 搞定

pom.xml（正例 — 一个 Starter）
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!-- 不写版本号，由 spring-boot-starter-parent 统一决定 -->
</dependency>

这一个坐标背后传递引入了 Spring MVC、内嵌 Tomcat、Jackson、Hibernate Validator 等一整套配套依赖，并且都是经过 Spring Boot 团队测试过的兼容版本组合。

Starter 内部长什么样

打开 spring-boot-starter-web 的 pom.xml 就会发现——它自己没有 src/main/java，只有一份 <dependencies> 列表：

spring-boot-starter-web （空壳，只有 pom.xml）
├── 依赖 → spring-boot-starter（基础 Starter，含自动配置核心 + 日志）
├── 依赖 → spring-boot-starter-json（Jackson 序列化）
├── 依赖 → spring-boot-starter-tomcat（内嵌 Tomcat）
└── 依赖 → spring-web / spring-webmvc（Spring MVC 核心）
命名规律：官方 Starter 统一叫 spring-boot-starter-xxx（如 -web、-data-jpa、-security）； 第三方或社区 Starter 反过来叫 xxx-spring-boot-starter（如 mybatis-spring-boot-starter），这是一条不成文但被广泛遵守的命名约定，见第 12 篇自定义 Starter。

Starter 只解决"依赖引入"，不解决"自动装配"

容易混淆的一点：Starter 本身不包含任何自动配置逻辑，它只是把该有的 jar 包拉齐。真正"引入依赖后自动帮你配好 Bean"这件事， 是由该依赖对应的 xxx-autoconfigure 模块（比如 spring-boot-autoconfigure 里的 WebMvcAutoConfiguration）在容器启动时完成的，这部分原理见第 10 篇 @EnableAutoConfiguration。

可以这样理解分工：Starter 负责"把菜买齐"，AutoConfiguration 负责"照着菜谱把菜做好摆上桌"——两者配合才是"引入依赖就能用"体验的完整链路。

核心价值总结

  • Starter 是一个只含依赖声明、不含代码的聚合包，解决"手动拼凑一堆零散依赖及其版本匹配"的痛点。
  • Starter 的版本由 spring-boot-starter-parent（或对应 BOM）统一管理，跟着 Spring Boot 版本联动。
  • 官方 Starter 命名为 spring-boot-starter-xxx，第三方 Starter 命名为 xxx-spring-boot-starter。
  • Starter 只管"引依赖"，真正"自动配好 Bean"靠的是配套的 AutoConfiguration 模块，两者要分开理解。

与其他知识点的联系

  • @SpringBootApplication / @EnableAutoConfiguration（第 5、10 篇）：Starter 只负责把 jar 包拉进 classpath，真正"发现并生效"这些依赖背后的自动配置类，靠的是 @EnableAutoConfiguration 触发的清单扫描机制——Starter 是"原材料入库"，@EnableAutoConfiguration 是"验货上架"，两者是先后接力的关系，不理解这一点很容易把"引了依赖"和"配置已生效"混为一谈。
  • @ConditionalOnClass（第 11 篇）：Starter 决定了 classpath 里有没有某个类，而 @ConditionalOnClass 正是通过检测"classpath 里存不存在这个类"来判断某个自动配置要不要生效。换句话说，Starter 引入依赖的动作，本身就是在给 @ConditionalOnClass 的判断结果"投票"——这也是为什么不同 Starter 组合会激活不同的自动配置集合。
  • 自定义 Starter（第 12 篇）：本篇讲的"Starter 空壳 + 命名规律"是第 12 篇动手写 Starter 的直接理论基础——第 12 篇里"Starter 模块只放依赖声明、Autoconfigure 模块放真正逻辑"的分层惯例，正是从官方 Starter（如 spring-boot-starter-web 拆分出 spring-boot-autoconfigure）身上总结出来的经验，本篇是那次实战的铺垫。
  • IoC 容器底层（BeanDefinition 的来源）：Starter 引入的依赖 jar 包里往往还带着大量可被扫描或被自动配置类声明的类，这些类最终都要在容器启动时被解析成 BeanDefinition 才能变成真正的 Bean。可以说 Starter 决定了"容器有哪些候选原材料可用"，但它本身不直接产生任何 BeanDefinition——这条边界感是理解"Starter 不等于自动装配"的关键。
