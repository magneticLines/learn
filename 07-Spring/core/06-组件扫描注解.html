IoC 核心与自动配置

@Component / @Service / @Repository / @Controller

四个注解，容器眼里几乎是同一件事。搞清楚"语义分层"和"技术强制"的区别，才不会在选择贴哪个注解上纠结。

一句话讲清楚

@Service、@Repository、@Controller 本质上都是被 @Component 元注解标注的"语义化别名"—— 对 Spring 容器来说，它们做的事情几乎一样：把这个类注册成一个 Bean，加入组件扫描的候选范围。区别不在于容器处理逻辑有多大差异，而在于"给人看的语义"。

四个注解的源码关系（简化）
@Component                 // 最底层，通用组件
public @interface Component {}

@Component                 // @Service 元注解上标了 @Component
public @interface Service {}

@Component
public @interface Repository {}

@Component
public @interface Controller {}

组件扫描时，Spring 判断"这是不是一个可注册的组件"，用的是"这个注解上是否间接标注了 @Component"，而不是精确匹配某一个具体注解名——所以这四个注解在能不能被扫描到这件事上完全等价。

那区别到底在哪

注解语义额外的技术行为
@Component通用组件，不属于任何特定分层无
@Service业务逻辑层无额外行为，纯语义标记
@Controller表现层（配合 @RequestMapping 处理请求）会被 Spring MVC 识别为可能包含请求映射方法的类，用于扫描 @RequestMapping
@Repository数据访问层唯一有实质技术差异：会被 PersistenceExceptionTranslationPostProcessor 代理，把各种数据库驱动抛出的原生异常统一转换成 Spring 的 DataAccessException 体系
@Repository 的异常转换解决的问题：不同数据库驱动抛出的异常五花八门（JDBC 的 SQLException、JPA 的 PersistenceException……），业务代码如果直接捕获这些原生异常，就和具体的持久化技术耦合死了。 统一转换成 DataAccessException 后，上层业务代码只需要处理一套异常体系，换底层实现（MySQL 换 PostgreSQL，JPA 换 MyBatis）也不用改异常处理逻辑。

"分层注解是约定，不是语法要求"

从技术上讲，把一个业务逻辑类标成 @Repository，或者把一个数据访问类标成 @Service，代码照样能跑通（除了失去异常转换这一项能力）。 Spring 不会因为你"分层用错了"而报错。这正说明：@Service/@Repository/@Controller 的分层含义完全靠团队约定维护，编译器不会替你把关。

团队实践建议：严格按照架构分层贴注解（Controller 处理请求参数校验和响应封装、Service 写业务逻辑、Repository 只做数据访问）， 不是因为 Spring 要求，而是因为这能让新人一眼看懂类的职责，Code Review 时也更容易发现"Controller 里直接写 SQL"这类架构腐化的苗头。

核心价值总结

  • @Service/@Repository/@Controller 都是 @Component 的语义化别名，容器对它们的扫描注册逻辑几乎相同。
  • 唯一有实质技术差异的是 @Repository 的数据库异常转换能力。
  • 分层注解的核心价值是"给人看的架构语义"，靠团队约定和 Code Review 维护，不是靠编译器强制。
  • 面试常问"这几个注解有什么区别"，标准答案就是这张表：语义分层 + @Repository 的异常转换。

与其他知识点的联系

  • 主启动类与 @SpringBootApplication（第 5 篇）：本篇讲的四个注解要能被容器发现，前提是它们所在的类落在 @ComponentScan 的扫描范围内，而这个范围由第 5 篇讲的主启动类包路径决定——组件扫描注解是"被扫描的标签"，主启动类的位置是"扫描的起点和边界"，两者缺一不可。
  • @Autowired 注入方式（第 7 篇）：组件扫描解决的是"把哪些类注册成 Bean"，而 @Autowired 解决的是"这些 Bean 之间怎么互相拿到依赖"——先有本篇的扫描注册，Bean 才存在于容器里，第 7 篇的注入才有对象可注入，二者是注册和消费的先后关系。
  • @Bean 与 @Configuration 显式装配（第 8 篇）：本篇的四个注解只适用于"自己能改源码的类"；第 8 篇专门解决组件扫描的覆盖盲区——第三方库的类没法贴注解。理解这一点，才能明白为什么 Spring 需要两条并行的装配路线，而不是只靠组件扫描一条路走到底。
  • IoC 容器底层（BeanDefinition 的注册）：组件扫描的本质，是容器启动时按 @ComponentScan 指定的路径遍历 classpath 下的 .class 文件，把带有 @Component（及其派生注解）标记的类解析成 BeanDefinition 并注册进 BeanFactory——本篇讲的"四个注解在扫描判定上完全等价"，根源就在于这一步只认元注解 @Component，不区分具体是哪个派生注解。
