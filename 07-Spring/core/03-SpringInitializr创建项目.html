Spring Initializr 创建项目

动手写代码之前，先搞清楚"项目骨架从哪来、pom.xml 里那个 parent 到底是什么"——这是每个 Spring Boot 项目共同的起点。

一句话讲清楚

Spring Initializr（start.spring.io）是官方提供的项目脚手架生成器： 你在网页上勾选构建工具、语言、Spring Boot 版本、需要的依赖，它就生成一个可以直接导入 IDE 运行的最小项目骨架。 本质上它省掉的是"手写 pom.xml 依赖坐标 + 手建目录结构"这些体力活。

实际操作：填哪些选项

选项常见选择说明
ProjectMaven国内团队 Maven 更常见；Gradle 构建更快、DSL 更灵活，团队约定优先
LanguageJava本系列教程统一用 Java
Spring Boot 版本选最新的非 SNAPSHOT 正式版避免用还在孵化的预览版本，生产项目更是如此
PackagingJar内嵌 Tomcat，java -jar 直接跑；War 包只在必须部署到外部容器时才用
Java 版本17 或 21（LTS）Spring Boot 3 要求 Java 17+，选长期支持版本更稳
Dependencies按需勾选，如 Spring Web每勾一个就是引入一个 Starter，见下一篇

生成的骨架长什么样

demo/
├── src/main/java/com/example/demo/
│ └── DemoApplication.java　　　　　　　　　←主启动类，见第 5 篇
├── src/main/resources/
│ ├── application.properties　　　　　　　　　←集中配置文件
│ └── static / templates　　　　　　　　　　　←静态资源 / 视图模板目录（预留）
├── src/test/java/...　　　　　　　　　　　　　←测试代码，包结构与 main 对称
└── pom.xml　　　　　　　　　　　　　　　　　←依赖与构建配置

pom.xml 里的 parent 在做什么

打开生成的 pom.xml，最先看到的是这一段：

pom.xml（片段）
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.3.0</version>
</parent>

spring-boot-starter-parent 本身也是一个 Maven 项目，它的作用是集中管理一份"经过官方测试、互相兼容"的依赖版本清单（背后是 spring-boot-dependencies BOM）。 继承了这个 parent 之后，你在自己的 pom.xml 里引入 Spring 相关依赖不需要写版本号——版本由 parent 统一决定，只要 Spring Boot 版本一致，整套依赖之间就不会打架。

常见坑：如果公司项目已经有自己的父 POM（比如统一管理内部组件版本），不能简单再继承 spring-boot-starter-parent（Maven 一个模块只能有一个 parent）。 这种情况改用 <dependencyManagement> 里 import spring-boot-dependencies 这个 BOM，效果等价但可以和自己的父 POM 共存。

为什么这样设计

没有统一版本管理之前，Spring MVC + Jackson + Tomcat + Hibernate Validator 各自独立发版，版本组合爆炸，稍微手动升级一个依赖就可能触发一连串不兼容问题（俗称"依赖地狱"）。 starter-parent 把"哪些版本互相兼容"这件事从每个项目里挪到 Spring Boot 团队统一维护，你只需要升级一个 Spring Boot 版本号，一整套依赖版本跟着联动升级，这是"约定优于配置"思想在依赖管理层面的体现。

核心价值总结

  • Spring Initializr 生成的是标准项目骨架，省去手写目录结构和依赖坐标的体力活。
  • spring-boot-starter-parent 的核心价值是统一管理依赖版本，避免"依赖地狱"。
  • 公司已有父 POM 时用 dependencyManagement + BOM 导入的方式替代直接继承。
  • Java 版本选 LTS（17/21），Spring Boot 版本选最新正式版，这是团队协作中最省心的默认选择。

与其他知识点的联系

  • 控制反转与依赖注入回顾：Initializr 生成的 DemoApplication.java 上的 @SpringBootApplication 注解会触发组件扫描，这正是上一篇回顾的"容器启动第一步"，本篇搭出的骨架就是 IoC 容器接下来要管理的起点。
  • application.properties / yml：生成的骨架里 src/main/resources/application.properties 是空的，但这正是后面配置组要重点展开的外部化配置入口——数据库连接、消息队列地址等所有环境相关的值都会写进这个文件。
  • Profile 多环境配置：骨架里只有一份 application.properties，实际项目会在此基础上补充 application-dev.properties、application-prod.properties 等按环境拆分的文件，这是配置组后续要讲的内容，本篇的骨架是它们的起点。
  • 依赖版本统一管理与消息队列集成：本篇讲的 spring-boot-starter-parent 统一管理依赖版本的思路，同样适用于后面要引入的 spring-boot-starter-amqp（RabbitMQ）、spring-kafka 等 Starter——只要 Spring Boot 版本一致，这些中间件客户端的版本也不会互相打架。
  • Java 版本与 @Async/@Scheduled 等特性：选择 Java 17/21 这类 LTS 版本不仅是"更稳"，也是因为 Spring Boot 3.x 的许多新特性（包括虚拟线程对 @Async 场景的优化）都要求较新的 JDK 版本，项目骨架阶段的这个选择会影响后面能用到的并发编程能力。
