Bean 作用域（singleton / prototype 等）

容器管理的每个 Bean 并不都是"独一份"——作用域决定了 getBean 每次拿到的是同一个实例，还是全新的实例。

一句话讲清楚

作用域（Scope）定义了容器对一个 Bean 实例的"生产和复用策略"。默认是 singleton（单例）——整个容器里这个 Bean 只有一个实例， 所有注入点共享同一份；而 prototype（原型）则相反——每次向容器请求这个 Bean，都会创建一个全新实例。

作用域生命周期典型场景
singleton（默认）容器启动时创建一次，跟随容器销毁无状态的 Service、Repository——绝大多数 Bean
prototype每次 getBean() 都新建，容器不负责销毁有状态、非线程安全的对象，比如一次性使用的上下文/构建器
request（Web）每个 HTTP 请求一个实例，请求结束即销毁需要在一次请求内共享、但不同请求间隔离的数据
session（Web）每个会话一个实例购物车、用户会话级别的状态
为什么默认是 singleton：大多数 Service/Repository 都是无状态的（不持有可变的实例字段，只是一堆方法的集合）， 多个请求共享同一个实例完全没有线程安全问题，还能节省对象创建开销。只有当 Bean 确实持有跟"一次使用"绑定的可变状态时，才需要考虑非单例作用域。

陷阱：prototype 注入到 singleton 里会失效

04-工程实战/mechanism/01-IoC-DI.html 里已经点出过这个坑的现象：把一个 prototype Bean 通过字段注入到一个 singleton Bean 里， 本意是"每次用都是新实例"，但实际上注入只发生一次（容器创建 singleton Bean 时注入的那个实例，会被一直复用），完全违背了 prototype 的初衷。

ReportService.java（陷阱复现）
@Service                              // singleton，容器只创建一次
public class ReportService {

    @Autowired
    private ReportContext ctx;       // ReportContext 声明为 @Scope("prototype")

    // ReportService 只在启动时被创建一次，ctx 也只在那一刻被注入一次
    // 之后 generate() 被调用一万次，拿到的都是同一个 ctx —— prototype 名存实亡
    public void generate() { ctx.doWork(); }
}

正确做法：用 ObjectProvider 延迟获取

真正想要"每次调用方法都拿到一个新的 prototype 实例"，不能在字段注入阶段就把依赖"定死"，而要在每次方法调用时，主动向容器再要一次。 Spring 提供的标准方式是注入 ObjectProvider<T>（比早期直接注入 ApplicationContext 手动 getBean() 更轻量、更符合依赖注入的语义）：

ReportService.java（正例 — ObjectProvider 按需获取新实例）
@Service
@RequiredArgsConstructor
public class ReportService {

    // 注入的不是 ReportContext 本身，而是"获取 ReportContext 的能力"
    private final ObjectProvider<ReportContext> ctxProvider;

    public void generate() {
        // 每次调用方法时才向容器要一个新实例，这才是 prototype 该有的效果
        ReportContext ctx = ctxProvider.getObject();
        ctx.doWork();
    }
}
ObjectProvider 相比直接注入 ApplicationContext 的优势：类型安全（不需要 getBean(ReportContext.class) 强转）、 还提供 getIfAvailable()/getIfUnique() 等方法优雅处理"Bean 不存在"或"有多个候选"的情况，代码意图比裸用 ApplicationContext 更清晰，也更容易 Mock 测试。

核心价值总结

  • 作用域决定容器对 Bean 的"生产和复用策略"，默认 singleton 适合绝大多数无状态 Bean。
  • prototype 注入到 singleton 里会失效——注入只发生一次，后续复用的还是那一个实例。
  • 正确做法是注入 ObjectProvider<T>，在真正需要的时刻再向容器请求新实例。
  • Web 场景下的 request/session 作用域本质是同一套思路的延伸：把 Bean 的生命周期和某种边界（请求/会话）绑定。
