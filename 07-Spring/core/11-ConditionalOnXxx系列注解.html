IoC 核心与自动配置

@ConditionalOnXxx 系列注解

上一篇留了个悬念——"条件注解决定配置类要不要生效"。这篇把最常用的几个条件注解逐个拆开，也顺手回答"为什么我自己声明一个 Bean 就能覆盖 Spring Boot 的默认值"。

一句话讲清楚

@ConditionalOnXxx 系列注解都是贴在配置类或 @Bean 方法上的"生效开关"：容器在决定要不要执行这个配置/创建这个 Bean 之前， 会先检查这些条件是否满足，条件不满足就直接跳过，仿佛这段配置从未存在过。

@ConditionalOnClass：classpath 里有没有这个类

典型用法
@AutoConfiguration
@ConditionalOnClass(RedisTemplate.class)  // classpath 里得有 Redis 客户端相关的类
public class RedisAutoConfiguration { ... }

如果项目没有引入 spring-boot-starter-data-redis，classpath 里根本没有 RedisTemplate 这个类，这个配置类整个被跳过—— 这就是为什么"不同项目引入不同 Starter，最终生效的自动配置也完全不同"：@ConditionalOnClass 是自动配置和 Starter 联动的关键纽带。

@ConditionalOnMissingBean：用户没自定义时才用默认的

解释"为什么自己声明一个 Bean 就能覆盖默认值"
@AutoConfiguration
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(DataSource.class)  // 容器里已经有 DataSource 了？那就不用我这个默认的
    public DataSource dataSource(DataSourceProperties props) { ... }
}
你自己的配置类（会被优先采用）
@Configuration
public class MyDataSourceConfig {

    @Bean
    public DataSource dataSource() {
        // 你自己手动配了一个连接池，比如切换成 HikariCP 定制参数
        return new HikariDataSource(customConfig);
    }
}

容器加载配置类是有先后顺序的，用户自己写的 @Configuration 类会先于自动配置类被处理。等轮到 DataSourceAutoConfiguration 执行时， 容器里已经存在一个 DataSource 类型的 Bean 了，@ConditionalOnMissingBean 条件不满足，Spring Boot 的默认实现就被自动"让路"。 这正是"约定优于配置，但用户配置优先"这条设计原则的具体实现。

面试高频："你自己声明一个 DataSource Bean，为什么能覆盖 Spring Boot 的默认数据源？"——标准答案就是这个 @ConditionalOnMissingBean 机制：自动配置类里的默认 Bean 都套了这个条件，用户的自定义配置一旦存在，默认值自动让位，不需要任何额外的排除配置。

@ConditionalOnProperty：根据配置文件的某个值

根据配置文件开关决定是否生效
@Configuration
@ConditionalOnProperty(name = "app.mock-payment.enabled", havingValue = "true")
public class MockPaymentConfig {
    @Bean
    public PaymentService paymentService() {
        return new MockPaymentService();  // 测试环境用假实现，不真的调支付网关
    }
}

这就是 01-IoC-DI.html 里"切换支付渠道只改配置，不改代码"那个例子背后用的正是同一个注解。 典型用法：本地/测试环境打开某个开关启用 Mock 实现，生产环境关闭该开关自动切回真实实现。

三个注解速查对比

注解判断依据典型场景
@ConditionalOnClassclasspath 里存不存在指定的类Starter 联动自动配置的关键纽带
@ConditionalOnMissingBean容器里存不存在指定类型/名称的 Bean让用户自定义配置优先于默认配置
@ConditionalOnProperty配置文件里某个属性的值功能开关、环境差异化配置
这几个注解底层都实现了同一个 Condition 接口（matches() 方法返回 true/false），Spring 允许你自己实现 Condition 接口并配合 @Conditional(MyCondition.class) 写出完全自定义的条件逻辑——官方这些 @ConditionalOnXxx 只是最常用的几种"预置实现"。

核心价值总结

  • @ConditionalOnClass 检查 classpath，是 Starter 触发自动配置的关键纽带。
  • @ConditionalOnMissingBean 让"用户自定义优先于框架默认"成为可能，不需要任何额外排除配置。
  • @ConditionalOnProperty 用配置文件的值控制功能开关，典型场景是环境差异化配置。
  • 这些条件注解都基于统一的 Condition 接口，理解了这一点就能看懂甚至自己扩展条件逻辑。
