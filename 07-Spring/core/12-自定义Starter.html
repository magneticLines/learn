IoC 核心与自动配置

自定义 Starter

前面十一篇讲的都是"怎么用" Spring Boot 的机制，这一篇反过来：把学到的 Starter + 自动配置原理组装起来，自己造一个团队内部可复用的 Starter。

一句话讲清楚

自定义 Starter 就是把前面几篇的知识串起来的一次实战：用 Starter 模块负责"声明依赖"，用 AutoConfigure 模块负责"写自动配置类 + 条件注解 + AutoConfiguration.imports 清单"， 这是团队内部沉淀公共基础设施（统一日志、统一签名验证、统一异常处理等）的标准做法——一次开发，所有微服务引入一个依赖就能用。

官方的分层惯例：为什么拆成两个模块

观察官方 Starter 会发现一个规律：spring-boot-starter-web（空壳，只有依赖声明）背后真正干活的是 spring-boot-autoconfigure 里的 WebMvcAutoConfiguration 等类。这种"Starter 模块 + Autoconfigure 模块"分层不是必须的，但已经成为业界惯例，好处是：

  • 职责单一：Starter 模块只管依赖坐标，Autoconfigure 模块只管配置逻辑，谁该改哪个文件一目了然。
  • 可以被单独复用：如果别的项目只想要自动配置逻辑，不想要 Starter 附带的那一整套依赖，可以只依赖 Autoconfigure 模块。

最小可用示例：统一请求日志 Starter

假设团队想要一个"引入依赖就自动打印请求耗时日志"的公共组件，命名遵循第三方 Starter 惯例：xxx-spring-boot-starter。

log-trace-spring-boot-starter/　　　　　　　　　　　←空壳模块，只有依赖声明
└── pom.xml　　（依赖 log-trace-spring-boot-autoconfigure）

log-trace-spring-boot-autoconfigure/　　　　　　　←真正干活的模块
├── src/main/java/.../LogTraceAutoConfiguration.java
├── src/main/java/.../LogTraceProperties.java　　　←配置属性绑定类
└── src/main/resources/META-INF/spring/
    └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
LogTraceProperties.java（读取 application.yml 里的 log-trace.* 配置）
@ConfigurationProperties(prefix = "log-trace")
public class LogTraceProperties {
    private boolean enabled = true;  // 默认开启，用户可在配置文件里关闭
    // getter/setter 省略
}
LogTraceAutoConfiguration.java（自动配置类，串联前面讲过的条件注解）
@AutoConfiguration
@ConditionalOnProperty(prefix = "log-trace", name = "enabled", matchIfMissing = true)
@EnableConfigurationProperties(LogTraceProperties.class)
public class LogTraceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // 用户想自己实现拦截器时可以覆盖
    public LogTraceInterceptor logTraceInterceptor() {
        return new LogTraceInterceptor();  // 记录请求耗时的拦截器实现，代码略
    }
}
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.company.logtrace.autoconfigure.LogTraceAutoConfiguration

这份 imports 文件就是让容器"认识"这个自动配置类的钥匙——只要业务项目的 pom.xml 里引入了 log-trace-spring-boot-starter，这个 jar 包就会随之进入 classpath，容器启动扫描时自然会读到这份清单，加载并按条件判断是否生效，跟第 10 篇讲的官方自动配置机制完全是同一套流程。

预留开关是好习惯：给自定义 Starter 加一个 @ConditionalOnProperty 总开关（如上例的 log-trace.enabled）， 能让接入方在个别环境（比如压测环境不想要日志开销）临时关闭功能，而不需要移除依赖重新打包。这是公共组件设计的一条经验法则。

核心价值总结

  • 自定义 Starter 遵循"Starter 只放依赖声明、Autoconfigure 放真正逻辑"的分层惯例，职责单一、可独立复用。
  • 核心三件套：自动配置类（@AutoConfiguration + 条件注解）、属性绑定类（@ConfigurationProperties）、AutoConfiguration.imports 清单文件。
  • 给自定义 Starter 预留一个总开关（@ConditionalOnProperty），方便接入方按环境灵活控制。
  • 这是本组"IoC 核心与自动配置"的收尾：从"用容器"到"懂容器"，再到"能自己扩展容器生态"，闭环完成。
