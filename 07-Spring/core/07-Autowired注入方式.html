@Autowired 注入方式（构造器 / Setter / 字段）

三种注入方式的优劣对比已经在工程实战模块讲过，这篇聚焦几个实操细节：什么时候能省略 @Autowired、多个同类型 Bean 怎么消歧义。

构造器注入 / Setter 注入 / 字段注入的优缺点对比表，以及"为什么构造器注入是官方推荐方式"，已经在 04-工程实战/mechanism/01-IoC-DI.html 的"三种注入方式对比"一节讲透了，这里不再重复画表格。

实操细节 1：单构造器可以省略 @Autowired

Spring 4.3 之后有一个很实用的简化：如果一个类只有一个构造器，Spring 会自动把它当作注入点使用，不需要再显式标注 @Autowired。 如果有多个构造器，则必须在其中一个上标 @Autowired 明确告诉容器用哪个。

OrderService.java（单构造器，省略 @Autowired）
@Service
public class OrderService {

    private final OrderRepository orderRepo;

    // 只有一个构造器，Spring 4.3+ 自动识别为注入点，@Autowired 可省略
    public OrderService(OrderRepository orderRepo) {
        this.orderRepo = orderRepo;
    }
}

这也是配合 Lombok @RequiredArgsConstructor 的写法在实际项目里几乎看不到 @Autowired 字样的原因——单构造器场景下它已经变成了可选项。

实操细节 2：required = false

@Autowired 默认要求依赖必须存在，找不到就直接启动失败。如果这个依赖是可选的，用 required = false 声明：

可选依赖
@Autowired(required = false)
private MetricsCollector metrics;  // 容器里没有这个 Bean 也能正常启动，metrics 为 null
用了 required = false 就必须在使用前做判空处理，否则一旦这个可选依赖真的缺失，NPE 会在运行时才暴露，比启动期失败更难排查。这是新手常忽略的连带责任。

实操细节 3：多个同类型 Bean 的歧义问题

当容器里有多个同类型的 Bean（比如 PaymentService 的多个实现），直接 @Autowired 会因为"不知道注入哪一个"而启动失败，报 NoUniqueBeanDefinitionException。

反例：歧义注入，启动报错
@Service class AlipayPaymentService implements PaymentService { ... }
@Service class WechatPaymentService implements PaymentService { ... }

@Service
class OrderService {
    @Autowired
    private PaymentService payService; // 报错：两个实现，容器不知道选哪个
}

两种解决方式，各自适用场景不同：

方式用法适用场景
@Qualifier 按 Bean 名称精确指定要注入哪一个 需要明确区分"就是要这一个"，比如按支付渠道分别注入不同实现
@Primary 标在某个实现类上，声明它是"默认优先"的那个 大多数地方都用同一个默认实现，只有极少数地方需要例外指定（用 @Qualifier 覆盖）
正例：@Qualifier + @Primary 配合使用
@Service
@Primary  // 大多数场景默认用支付宝，作为兜底选择
class AlipayPaymentService implements PaymentService { ... }

@Service("wechatPay")  // 显式指定 Bean 名称
class WechatPaymentService implements PaymentService { ... }

@Service
class OrderService {
    @Autowired
    private PaymentService defaultPay; // 无歧义时自动选 @Primary 的 AlipayPaymentService

    @Autowired @Qualifier("wechatPay")
    private PaymentService wechatPay; // 显式指定，绕过 @Primary
}
面试高频："@Autowired 找到多个同类型 Bean 怎么办？"——按优先级依次是：先看有没有 @Qualifier 精确指定名称；没有的话看是否有 @Primary 标记的默认实现；都没有就直接抛 NoUniqueBeanDefinitionException。 另外如果注入类型是 List<PaymentService>，则不存在歧义问题——Spring 会把所有实现都收集进列表，这是策略模式常用写法。

核心价值总结

  • 三种注入方式的优劣对比复用 04-工程实战/mechanism/01-IoC-DI.html 的结论，不重复展开。
  • 单构造器场景下 @Autowired 可省略（Spring 4.3+），多构造器必须显式标注。
  • required = false 声明可选依赖，但必须配合判空处理，否则风险从启动期推迟到运行期。
  • 多个同类型 Bean 的歧义问题：@Qualifier 精确指定，@Primary 设默认优先，两者可以配合使用。

与其他知识点的联系

  • 组件扫描注解（第 6 篇）：@Autowired 能注入成功的前提，是目标类型已经通过第 6 篇的 @Component 系列注解（或第 8 篇的 @Bean）注册成了 Bean——@Autowired 本身不创造 Bean，只是"按类型/名称去容器里找一个已经存在的 Bean"，找不到就是本篇讲的启动失败或 required=false 兜底。
  • @Bean 与 @Configuration 显式装配（第 8 篇）：本篇"多个同类型 Bean 用 @Qualifier/@Primary 消歧义"的问题，同样会出现在 @Bean 方法场景——如果一个接口有多个 @Bean 方法产出不同实现，第 8 篇的方法参数自动注入同样会踩到本篇讲的 NoUniqueBeanDefinitionException，两篇讲的是同一个歧义问题在"扫描注入"和"显式装配"两种场景下的共同表现。
  • Bean 作用域（第 9 篇）：本篇默认讨论的都是单例 Bean 之间的注入，一旦涉及把 prototype 作用域的 Bean 注入到 singleton Bean 里，就会遇到第 9 篇讲的"注入只发生一次、Bean 失去作用域语义"的陷阱——@Autowired 的注入时机（容器创建期）和作用域的生效时机（每次 getBean）之间的错配，是理解那个陷阱的关键前提。
  • IoC 容器底层（依赖解析与循环依赖）：@Autowired 的底层是 AutowiredAnnotationBeanPostProcessor 在 Bean 实例化后的属性填充阶段介入，按类型在 BeanFactory 里查找候选 Bean 完成赋值——这也是为什么"构造器注入无法解决循环依赖，而字段/Setter 注入可以通过三级缓存曲线救国"，本篇三种注入方式的选择本质上也是在选择"依赖解析发生的时机"。
