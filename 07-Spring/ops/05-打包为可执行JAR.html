部署与运维

打包为可执行 JAR

"部署一个 Java Web 应用"曾经意味着装 Tomcat、配环境、传 WAR 包——Spring Boot 把这一切压缩成了一条 java -jar 命令。

一句话讲清楚

Spring Boot 应用默认打包成一个内嵌 Tomcat（或 Jetty/Undertow）的可执行 JAR：这个 JAR 里不仅有你的业务代码， 还打包了运行它所需要的全部依赖，甚至连 Web 服务器本身都在里面。结果就是：拿到这一个文件，任何装了 JRE 的机器上一条 java -jar app.jar 就能跑起来， 不需要额外安装部署任何中间件。

反例：传统 WAR 包部署方式

在 Spring Boot 出现之前，一个典型的 Java Web 应用部署流程是这样的：

  • 先在目标服务器上单独安装、配置一个 Tomcat（版本要和开发/测试环境一致，否则可能出现"我本地能跑"的问题）。
  • 把项目打成 .war 包，手动或用脚本拷贝到 Tomcat 的 webapps/ 目录下。
  • Tomcat 解压 WAR、加载应用，中间任何一步配置错了（端口冲突、JVM 参数、Tomcat 版本不兼容）都要登录服务器排查。
  • 一台服务器上如果要同时跑多个不同版本 Spring / 不同 Servlet 容器版本要求的应用，容器共享会互相牵制，版本升级投鼠忌器。
这种"应用和容器分离部署"的模式，把"运行环境"变成了一个需要单独维护、且容易和应用版本产生耦合冲突的外部依赖——这正是 Spring Boot 要解决的核心痛点之一。

正例：内嵌容器 + 可执行 JAR

pom.xml
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
        </plugin>
    </plugins>
</build>
终端
# mvn package 会自动触发 repackage，生成可执行 jar
$ mvn clean package

# 任何装了 JRE 的机器，一条命令直接启动，Tomcat 已经在 jar 里了
$ java -jar target/order-service-1.0.0.jar

repackage 到底做了什么（原理走一遍）

mvn package 本身只会生成一个"瘦" JAR（只含你自己的类，不含依赖）——这是 Maven 的标准行为，不是 Spring Boot 独有的。 spring-boot-maven-plugin 在这一步之后追加了一个 repackage 目标，把它重新加工成"胖" JAR（fat jar / uber jar）：

① mvn package 阶段
  编译代码，生成一个只含 target/classes 内容的普通 jar

② spring-boot-maven-plugin:repackage
  把上一步的瘦 jar 重命名为 xxx.jar.original（保留原始产物）
  把所有第三方依赖 jar（含内嵌 Tomcat）整体塞进 BOOT-INF/lib/
  把你自己的 class 文件放进 BOOT-INF/classes/
  生成一个特殊的 MANIFEST.MF，指定 Main-Class 为 Spring Boot 自带的
  JarLauncher（而不是你写的 @SpringBootApplication 类）

③ java -jar 执行
  JVM 先加载 JarLauncher，它负责把 BOOT-INF/lib 下所有依赖
  加入类加载器的 classpath，再反射调用你真正的启动类的 main 方法
普通 JAR 的 MANIFEST.MF 里 Class-Path 属性并不支持指向 jar 包内部的其他 jar（嵌套 jar）， 这是标准 JAR 格式本身的限制。Spring Boot 的 JarLauncher 用自定义类加载器（LaunchedURLClassLoader）绕开了这个限制， 这也是为什么可执行 JAR 不能简单地用 unzip 展开后直接用普通方式运行——它依赖这套自定义启动机制。

对比：WAR 外部容器 vs 可执行 JAR

维度WAR + 外部 Tomcat可执行 JAR（内嵌容器）
部署步骤装容器 → 配置 → 拷贝 WAR → 重启容器一条 java -jar 命令
版本一致性容器版本需要单独维护，易出现环境差异容器版本随应用一起打包，环境即代码
多应用共存共享一个容器，互相牵制每个应用独立进程，互不影响
容器化适配度需要额外镜像分层安装 Tomcat天然契合"一个容器一个进程"，见下一节
适用场景需要在同一容器里跑多个遗留 WAR 应用微服务、云原生、CI/CD 自动化部署
如果确实需要兼容传统的外部容器部署方式，Spring Boot 也支持打成 WAR：让启动类继承 SpringBootServletInitializer， 并把内嵌容器依赖 scope 改成 provided。但如无历史包袱，官方和社区都推荐直接用默认的可执行 JAR 方式。

核心价值总结

  • 部署即代码：运行环境（Web 容器版本）被固化进构建产物，彻底消灭"环境不一致"这一类问题。
  • repackage 的本质：把瘦 jar + 依赖 + 自定义启动器打包成一个能被 java -jar 直接执行的 fat jar，核心是自定义类加载器解决了嵌套 jar 加载的限制。
  • 为容器化铺路：一个自包含、可独立运行的可执行 JAR，正是下一节 Docker 容器化部署最理想的打包形态。

与其他知识点的联系

  • Docker 容器化部署：本节讲的"自包含、内嵌容器、无外部依赖"正是下一节 Docker 化能一步到位的前提——Dockerfile 里只需要 COPY 这个 fat jar 再 ENTRYPOINT 启动，不需要额外安装任何中间件。
  • Kubernetes 与云原生：K8s 调度的最小单元是容器，容器里跑的就是本节打包出来的可执行 JAR；如果没有"一个文件独立运行"的能力，K8s 的弹性伸缩、滚动更新就无从谈起。
  • Spring Boot Admin：Admin Server 和它监控的业务应用本质上都是这里讲的可执行 JAR，理解 repackage 打包机制有助于理解为什么这些应用能作为独立进程互相注册、互相探测。
  • 类加载机制：本节提到的 JarLauncher 用自定义类加载器（LaunchedURLClassLoader）加载嵌套 jar，这与 JVM 基础中"类加载器可以自定义、可以打破双亲委派限制"是同一套底层机制的具体应用。
