健康检查与自定义指标

数据库连着、磁盘没满，不代表你的业务是健康的——真正的健康检查要问到业务自己头上。

一句话讲清楚

默认的 /actuator/health 只知道检查它认识的组件（数据源、磁盘空间、Redis 连接池），但完全不了解你的业务逻辑。 Spring Boot 允许你实现 HealthIndicator 接口，把任意自定义的健康判断逻辑（比如"依赖的第三方支付网关是否可达"）接入到同一个 /actuator/health 端点里，让"健康"这个概念真正覆盖到业务层面。

类比：体检报告默认只查血压、心率这些通用指标，但如果你是运动员，还需要额外查肺活量、关节柔韧度——这些是"业务特有"的健康指标，需要专门加检查项。

默认健康检查的边界

假设一个订单服务的核心链路依赖一个外部风控服务：数据库、Redis 全部正常，但风控服务挂了，订单流程实际上完全走不通。 这种情况下默认的 health 端点会天真地返回 UP，K8s 探针看到"健康"，继续把流量导进来——用户持续下单失败，而运维毫无察觉，这就是只依赖默认健康检查的典型陷阱。

自定义 HealthIndicator：把业务健康状态纳入检查

RiskControlHealthIndicator.java
@Component
public class RiskControlHealthIndicator implements HealthIndicator {

    private final RiskControlClient riskClient;

    public RiskControlHealthIndicator(RiskControlClient riskClient) {
        this.riskClient = riskClient;
    }

    @Override
    public Health health() {
        try {
            boolean reachable = riskClient.ping();
            if (reachable) {
                // withDetail 附加的信息会一并出现在 /actuator/health 的 details 里
                return Health.up().withDetail("riskControl", "reachable").build();
            }
            return Health.down().withDetail("riskControl", "unreachable").build();
        } catch (Exception e) {
            // 检查过程本身出异常，也要算作 DOWN，不能让健康检查自己成为故障点
            return Health.down(e).build();
        }
    }
}

Spring Boot 会自动发现所有 HealthIndicator Bean，把名字（去掉 HealthIndicator 后缀，即 riskControl）作为 key 合并进 /actuator/health 的返回结构里，任意一个子项 DOWN，整体状态也会变成 DOWN。

陷阱：不要在 health() 方法里做重量级操作（比如真的下一笔测试订单），健康检查通常被高频轮询（K8s 默认几秒一次）， 过重的检查逻辑会反过来拖垮被检查的下游服务。轻量级的连通性探测（ping、心跳接口）就够了。

用 Micrometer 埋点自定义业务指标

健康检查回答的是"行不行"（布尔），而指标（Metrics）回答的是"怎么样"（数值/趋势）——这就是可观测性（Observability）的另一个维度。 Spring Boot Actuator 底层使用 Micrometer 作为指标门面（类比 SLF4J 之于日志框架），业务代码只需注入 MeterRegistry 打点， 底层可以无缝切换到 Prometheus、Datadog 等不同的监控后端。

OrderMetrics.java
@Component
public class OrderMetrics {

    private final Counter orderCreatedCounter;
    private final Timer   orderProcessTimer;

    // MeterRegistry 由 Micrometer 自动装配，业务代码不关心具体接的是哪个监控后端
    public OrderMetrics(MeterRegistry registry) {
        this.orderCreatedCounter = Counter.builder("order.created")
                .description("累计创建订单数").register(registry);
        this.orderProcessTimer = Timer.builder("order.process.time")
                .description("订单处理耗时").register(registry);
    }

    public void recordOrderCreated() { orderCreatedCounter.increment(); }

    public <T> T timeProcess(Supplier<T> work) {
        return orderProcessTimer.record(work);
    }
}
指标类型语义典型用途
Counter只增不减的累计值累计创建订单数、累计报错次数
Gauge某一时刻的瞬时值，可增可减当前线程池活跃线程数、当前队列长度
Timer耗时的次数分布统计接口响应时间的 P50/P99
引入 micrometer-registry-prometheus 依赖后，Actuator 会自动多出一个 /actuator/prometheus 端点，输出 Prometheus 抓取格式的文本， Prometheus Server 定时来拉取即可，无需业务再额外开发上报逻辑——这也是本模块前面讲的"约定优于配置"思想在运维领域的延伸。

概念辨析：健康检查 vs 指标监控

维度健康检查（Health）指标监控（Metrics）
回答的问题现在能不能正常工作？（是/否）现在运行得怎么样？（数值/趋势）
使用者负载均衡器、K8s 探针——决定是否路由流量Prometheus/Grafana——用于告警、容量规划、性能分析
时效性关注当下这一刻关注一段时间的变化趋势
面试高频："如何给 Spring Boot 应用接入监控？"——标准答案链路是：业务代码用 Micrometer 打点 → Actuator 暴露 /actuator/prometheus → Prometheus 定时抓取 → Grafana 可视化 → Alertmanager 告警。理解这条链路比记住某个 API 更重要。

核心价值总结

  • 健康检查要贴近业务：默认组件检查只是及格线，关键业务依赖必须自定义 HealthIndicator。
  • 指标是趋势，健康是快照：两者互补，共同构成可观测性的基础三支柱之一（另外两个是日志和链路追踪）。
  • Micrometer 是门面：业务代码只面向 MeterRegistry 编程，底层监控后端可以随时替换而不改业务代码，这是"面向接口编程"思想的又一次实践。
