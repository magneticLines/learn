容器化部署 Docker

上一节让应用变成了"一个能独立跑起来的文件"，这一节让它变成"一个能在任何地方原样跑起来的镜像"。

一句话讲清楚

Docker 的核心理念是"一个容器只跑一个进程"，把应用和它运行所需的一切（JRE、系统库、配置）打包进一个不可变的镜像， 保证"在我机器上能跑"和"在生产环境能跑"是同一件事。而 Spring Boot 应用天生就是一个独立可执行的 JAR、自带内嵌容器、不依赖外部安装的中间件—— 这与 Docker "一个容器一个进程"的理念几乎是为彼此量身定做的。

回想上一节的对比：如果还是 WAR + 外部 Tomcat 的部署方式，镜像里就要同时装 Tomcat、配置好版本、再把 WAR 扔进去， 一个镜像里混杂着"容器软件"和"业务代码"两种生命周期不同的东西，Docker 化的意义会被削弱。可执行 JAR 消灭了这层依赖，Docker 化才真正顺理成章。

一个基础 Dockerfile

Dockerfile
# 使用只含 JRE（不含开发工具）的精简基础镜像，减小体积、缩小攻击面
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

# 把上一节 mvn package 产出的可执行 jar 拷进镜像
COPY target/order-service-1.0.0.jar app.jar

# 声明容器对外监听的端口（仅文档作用，真正映射靠 docker run -p）
EXPOSE 8080

# 容器启动时执行的命令——和上一节本地启动完全一样
ENTRYPOINT ["java", "-jar", "app.jar"]
终端
# 构建镜像
$ docker build -t order-service:1.0.0 .

# 运行容器，把容器内 8080 映射到宿主机 8080
$ docker run -p 8080:8080 order-service:1.0.0

可以看到，Dockerfile 里唯一和 Spring Boot 相关的就是那一条 COPY + ENTRYPOINT——正因为可执行 JAR 足够"自包含"，容器化的门槛才这么低。

朴素做法的问题：每次改一行代码都要重建整个层

上面的 Dockerfile 把整个 fat jar 当成一个整体 COPY 进去。Docker 镜像是分层存储的，只要某一层的内容变了，这一层及其之后的所有层都要重新构建。 而 fat jar 里 90% 以上的体积通常是不常变化的第三方依赖（Spring 自身、数据库驱动等），只有一小部分是业务代码。 只要改一行业务代码重新打包，整个 fat jar 的内容哈希就变了，导致 CI/CD 每次都要重新上传/下载整个大文件，构建和部署都很慢。

Spring Boot 3 的分层打包（Layered Jar）

Spring Boot 从 2.3 版本起、在 Spring Boot 3 中进一步完善了分层 JAR（layered jar）能力：把 fat jar 内部按"变化频率"拆分成多个逻辑层 （依赖层、Spring Boot loader 层、快照依赖层、业务代码层），构建镜像时对应生成多个 Docker 层。

分层策略（从不常变 → 常变）
  dependencies　　　　—— 第三方依赖 jar（体积最大，改动最少）
  spring-boot-loader　—— Spring Boot 启动器相关类
  snapshot-dependencies —— SNAPSHOT 快照依赖（变化频率中等）
  application　　　　　—— 你自己的业务代码（体积最小，改动最频繁）
Dockerfile（分层优化版，多阶段构建）
FROM eclipse-temurin:21-jre-alpine AS builder
WORKDIR /app
COPY target/order-service-1.0.0.jar app.jar
# 把 fat jar 拆解成 dependencies / spring-boot-loader / application 等目录
RUN java -Djarmode=layertools -jar app.jar extract

FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
# 按"变化频率从低到高"的顺序逐层 COPY —— 越靠前的层缓存命中率越高
COPY --from=builder /app/dependencies/ ./
COPY --from=builder /app/spring-boot-loader/ ./
COPY --from=builder /app/snapshot-dependencies/ ./
COPY --from=builder /app/application/ ./
ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]
效果：只改业务代码时，前三层（体积占绝大部分的依赖层）命中 Docker 构建缓存，完全不需要重新构建或推送； 只有最后的 application 层需要重新构建和传输。在依赖体积远大于业务代码体积的典型项目里，这能把 CI/CD 的镜像构建、推送时间从分钟级压缩到秒级。

对比：整体打包 vs 分层打包

维度整体 COPY fat jar分层打包（Layered Jar）
Docker 层数1 层（jar 作为整体）4 层，按变化频率拆分
改一行业务代码的构建成本整个 jar 对应的层全部失效重建只有最小的 application 层重建
实现复杂度简单，一行 COPY需要多阶段构建 + layertools 提取，配置稍复杂
适用场景个人学习、简单项目、构建频率低团队协作、高频 CI/CD、镜像仓库带宽敏感
生产环境常见坑：直接把开发机上的 JDK 完整镜像（如 openjdk:21，体积可能超过 500MB）当作运行时基础镜像， 而不是用精简的 JRE-only 镜像（如 eclipse-temurin:21-jre-alpine，体积可以做到 100MB 以内）。 生产环境运行阶段根本不需要 javac 等开发工具，多余的体积既拖慢分发速度，也扩大了潜在的安全攻击面。

核心价值总结

  • 可执行 JAR 是容器化的理想载体：自包含、无外部依赖，天然契合"一个容器一个进程"的设计理念。
  • 镜像分层意识：把"不常变的依赖"和"常变的业务代码"分层存放，是缩短 CI/CD 构建时间的关键工程实践，这个思路不止适用于 Java，对任何语言的容器化都成立。
  • 基础镜像要精简：运行时只需要 JRE，不需要完整 JDK，体积和安全都要纳入考量。
  • 容器解决了"打包一个应用"的问题，但生产环境往往有成百上千个容器实例需要调度——这正是下一节 Kubernetes 要解决的问题。

与其他知识点的联系

  • 打包为可执行 JAR：本节 Dockerfile 里的 COPY 对象、分层打包里 application 层的内容，都直接来自上一节 mvn package 的产物，没有可执行 JAR 的自包含特性，容器化就要多背负一整套中间件安装逻辑。
  • Kubernetes 与云原生：本节构建出的镜像正是 K8s 调度的基本单元，下一节的 livenessProbe/readinessProbe 要探测的就是本节容器里跑起来的这个 Spring Boot 进程。
  • Spring Boot Actuator 端点：容器化之后应用变成黑盒，只能通过网络请求了解内部状态，这正是 Actuator 的 /actuator/health 端点在容器化场景下变得不可或缺的原因——本节和 Actuator 一节合起来才是完整的"可观测的容器化部署"。
  • JMX 管理 Bean：本节提到 JMX 的 RMI 端口在容器网络（NAT、随机端口）里几乎无法正常暴露，这是"JMX 让位于 HTTP 方案"这一趋势在容器化场景下最直接的体现。
  • WebClient 替代 RestTemplate：容器化环境下服务间调用常常要应对瞬时的网络抖动和更高的并发密度，响应式的 WebClient 基于 Netty 的少量线程模型比 RestTemplate 的阻塞式连接池更适配容器资源受限（CPU/内存 limit）的场景。
