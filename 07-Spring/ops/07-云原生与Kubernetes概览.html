云原生与 Kubernetes 概览（进阶）

Docker 解决了"打包一个应用"，但生产环境从来不是只有一个容器——这一节是整个 Spring 模块的收尾篇。

一句话讲清楚

当一个系统从"几个容器"变成"成百上千个容器实例"，人工管理（启动哪个、挂了要不要重启、流量该分给谁、要不要多开几个实例）就完全不现实了—— 这就是容器编排（Container Orchestration）要解决的问题，Kubernetes（K8s）是目前事实上的标准方案： 它自动完成调度（把容器放到合适的机器上）、故障自愈（容器挂了自动重启/替换）、水平扩缩容（根据负载自动增减实例数）。

类比：上一节的 Docker 相当于给每个演员准备好了独立的化妆间和道具（打包好、随时能上场）， 而 K8s 是那个统筹调度全场演出的导演——谁什么时候上场、场地怎么分配、演员突然请假了怎么临时补位，全部由它统一指挥。

Actuator 健康检查如何对接 K8s 探针

K8s 要自动完成"故障自愈"和"流量调度"，前提是它得知道每个容器实例的真实状态——这正好接回本组第一、二篇讲过的 Actuator 健康检查。 K8s 定义了两种探针，分别对应不同的判断目的：

探针回答的问题失败后 K8s 的动作
存活探针
Liveness Probe		 这个进程是不是"卡死"了，需不需要重启？ 杀掉容器，重新拉起一个新实例
就绪探针
Readiness Probe		 这个实例现在能不能正常处理请求？ 暂时把它从负载均衡的可用列表里摘掉，但不重启，等恢复后再重新纳入

Spring Boot Actuator 从 2.3 版本起内置了对这两种探针语义的原生支持，暴露独立的健康分组端点：

application.yml
management:
  endpoint:
    health:
      probes:
        # 开启后自动暴露 /actuator/health/liveness 与 /actuator/health/readiness
        enabled: true
deployment.yaml（K8s 部署清单片段）
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 20
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
常见坑：如果把自定义 HealthIndicator（比如检查第三方支付网关是否可达）也计入 liveness， 一旦第三方服务临时抖动，K8s 会误判"应用本身"挂了，反复重启原本健康的实例——这只会让故障雪上加霜。 正确做法：liveness 只反映"进程自身是否卡死"（比如死锁），readiness 才纳入外部依赖是否可用的判断，两者语义不能混用。

ConfigMap / Secret：外部化配置在云原生环境的延伸

本模块前面 config 组讲过"配置和代码分离"的外部化配置思想——同一个 JAR 包，通过不同的 application-{profile}.yml 或环境变量， 就能跑出开发、测试、生产等不同的行为。K8s 把这个思想进一步产品化，提供了两种专门的资源对象：

K8s 资源对应配置管理场景与 Spring Boot 的衔接方式
ConfigMap普通配置项（如日志级别、超时时间）挂载为容器内的配置文件，或注入为环境变量，Spring Boot 自动读取
Secret敏感配置项（数据库密码、API Key）同样可挂载/注入，K8s 对其做了额外的存储加密和访问控制

本质上，这依然是"配置外部于代码"这一条思路的延伸：只是把配置的存放和分发从"配置中心/环境变量"进一步下沉到了"编排平台原生支持"这一层， 让配置管理和容器调度整合在同一套体系里。

面试高频："Spring Boot 应用怎么部署到 K8s？"——标准链路是：可执行 JAR（上上节）→ Docker 镜像（上一节）→ K8s Deployment 定义副本数和探针 → Service 做负载均衡和服务发现 → ConfigMap/Secret 挂载配置。能完整讲出这条链路，比记住某个 YAML 字段更重要。

模块总结：从写代码到跑在生产环境

到这里，整个 Spring 模块的学习内容全部完成。回顾这条从"写代码"到"跑在生产环境"的完整链路，会发现每一组内容都在解决上一组遗留下来的问题：

IoC 核心与自动配置 —— 对象怎么创建、怎么组装、怎么按需自动配置
  ↓
Web / REST —— 把业务能力通过 HTTP 接口暴露给外部调用方
  ↓
数据访问 —— 业务数据如何持久化、如何高效查询
  ↓
安全 —— 谁能访问这些接口和数据，怎么鉴权和授权
  ↓
配置与集成 —— 同一份代码如何适配不同环境、对接外部系统
  ↓
响应式编程 —— 面对高并发、流式数据时的另一套编程模型
  ↓
部署与运维（本组） —— 应用如何被监控、打包、容器化，最终稳定跑在生产环境

这条链路本质上回答的是同一个问题在不同阶段的具体形态："如何让代码可靠地服务于真实用户"。IoC 解决的是代码内部的组织问题， Web/数据/安全解决的是业务能力的对外呈现问题，配置与集成解决的是"同一份代码换个环境还能不能跑"的问题，响应式编程解决的是极端并发下的性能问题， 而部署与运维解决的是"代码跑起来之后，如何被看见、被管理、被大规模复制"的问题。

下一步可以深入的方向：当一个单体应用被拆分成多个微服务后，会出现新的问题——服务之间怎么发现彼此、 怎么做熔断降级、怎么做统一的分布式配置和链路追踪。这些正是 Spring Cloud 生态要系统性解决的问题， 是本模块之后自然的进阶方向。

核心价值总结

  • K8s 解决的是编排问题：在 Docker 打包好单个应用的基础上，自动化地完成调度、故障自愈、弹性伸缩。
  • 探针语义要分清：liveness 判断"要不要重启"，readiness 判断"要不要摘流量"，两者不能混用同一套健康检查逻辑。
  • ConfigMap/Secret 是外部化配置思想的云原生延伸：配置与代码分离的核心思路没变，只是承载配置的基础设施从应用层下沉到了编排平台层。
  • 整个 Spring 模块串成了一条完整链路：从对象怎么创建，到接口怎么暴露、数据怎么存取、访问怎么鉴权、配置怎么适配环境、高并发怎么应对，最终落到应用怎么被监控和部署——这是"写代码"到"跑在生产环境"的完整闭环，Spring Cloud / 微服务体系是下一步值得深入的方向。
