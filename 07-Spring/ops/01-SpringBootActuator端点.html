Spring Boot Actuator 端点

应用上线之后，"能跑起来"和"能被安心运维"是两回事——Actuator 就是 Spring Boot 给生产环境开的那扇"体检窗口"。

一句话讲清楚

Actuator 是 Spring Boot 提供的生产就绪（production-ready）特性模块：只要引入一个依赖，就能自动暴露一组 HTTP 端点， 对外回答"应用活不活着""依赖的数据库/消息队列通不通""内存和线程用了多少"这些运维最关心的问题，不用自己写一行监控代码。

类比：汽车仪表盘。发动机是否正常、油量还剩多少、胎压够不够，司机不需要拆开引擎盖检查，仪表盘上的几个指示灯和数字就够了。 Actuator 就是给你的 Spring 应用装上这套仪表盘。

没有 Actuator 之前：靠"猜"和"看日志"运维

假设一个订单服务部署到了生产环境，运维/SRE 想知道它是否健康，通常只能：

  • 用 curl 随便打一个业务接口，"能返回 200 就当活着"——但如果这个接口本身依赖了数据库，数据库挂了它也会超时，判断不出问题出在哪一层。
  • 登录服务器 tail -f 日志，靠肉眼扫错误堆栈——效率低，而且日志里往往看不出"内存是不是快满了"这种趋势性问题。
  • 没有统一的健康检查协议，K8s 的存活探针、负载均衡器的健康检查各写各的判断逻辑，标准不统一。

这正是本模块前面 config 组讲过的"外部化配置"要解决的姊妹问题——配置管理让应用"跑在哪里都一样"，Actuator 则让应用"跑起来之后能被看见"。两者共同构成生产就绪应用的基础设施。

引入依赖，立即获得一批端点

pom.xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

启动应用后，Actuator 会根据类路径上存在的组件（数据源、消息队列、磁盘）自动装配一批端点，常用的有：

端点作用典型信息
/actuator/health健康检查UP / DOWN，各组件（db、diskSpace、redis...）细分状态
/actuator/info应用元信息版本号、Git commit（需配合插件）
/actuator/metrics运行时指标JVM 内存、GC 次数、HTTP 请求耗时分布
/actuator/env环境变量与配置当前生效的所有 Property Source（含敏感信息！）
/actuator/beans容器里的 Bean 清单与前面 IoC 章节的单例池概念直接对应
/actuator/loggers运行时查看/修改日志级别无需重启即可临时把某个包调成 DEBUG 排查线上问题
/actuator/threaddump线程快照排查死锁、线程池耗尽

默认只暴露 health 和 info——这是故意的

直接启动一个新项目访问 /actuator，你会发现能看到的端点少得可怜。这不是 bug，是 Spring Boot 团队刻意做的安全默认值：凡是可能泄露内部信息或允许修改运行时状态的端点，一律不默认暴露，需要显式开启。

application.yml
management:
  endpoints:
    web:
      exposure:
        # 按需开启，而不是无脑写 include: "*"
        include: health,info,metrics,loggers
  endpoint:
    health:
      # 让 health 端点返回详细的组件明细，而不是只有一个 UP/DOWN
      show-details: when-authorized

生产环境常见坑

千万不要 include: "*" 无脑暴露到公网。 /actuator/env 会把当前所有生效配置（包括数据库密码、第三方 API Key，只要没做脱敏）原样打印出来； /actuator/heapdump 能直接把整个堆内存导出，内存里可能驻留着用户的敏感数据； /actuator/shutdown（默认关闭但一旦手滑打开）可以被任何人一个 POST 请求关掉你的服务。 正确做法：只暴露必要端点，且把 /actuator/** 放到独立端口或内网网段，配合 Spring Security 做认证。
application.yml（生产推荐配置）
management:
  server:
    # 让 Actuator 端点跑在独立端口，与业务端口物理隔离
    port: 9001
  endpoints:
    web:
      exposure:
        include: health,metrics,prometheus
      exclude: env,heapdump,shutdown
面试高频："Actuator 端点默认暴露哪些？为什么？"——默认只暴露 health 和 info，因为安全优先于便利；其余端点需要显式加入 management.endpoints.web.exposure.include，且生产环境要结合独立端口 + 鉴权 + 白名单三重防护。

核心价值总结

  • 开箱即用：一个依赖就能获得健康检查、指标采集、运行时管理一整套运维能力，不用自己造轮子。
  • 标准化：K8s 探针、负载均衡健康检查、监控系统采集都可以统一对接同一套 /actuator/* 协议。
  • 安全默认值：默认最小暴露面，敏感端点需要显式开启，这是"安全前置"设计思想的体现。
  • 这一节是后面健康检查自定义、Prometheus 指标接入、K8s 探针的基础，务必先分清"默认开箱即用"和"需要主动配置"的边界。
