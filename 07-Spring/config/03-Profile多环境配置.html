Profile 多环境配置

开发环境连本地 MySQL，生产环境连集群数据库——同一套代码，不同环境需要不同的配置甚至不同的 Bean，Profile 就是 Spring 给出的答案。

一句话讲清楚

Profile（环境标签）是 Spring 提供的一套机制，允许你为不同的运行环境（开发 dev、测试 test、生产 prod）准备不同的配置文件和不同的 Bean 实现， 启动时通过一个开关（spring.profiles.active）决定"此刻我是谁"，Spring 就只加载对应环境的那一份。

类比：就像手机有"飞行模式""省电模式"几套预设，你不需要每次都手动把所有开关调一遍，切换一个模式标签，一组相关设置就整体生效了。

按环境拆分配置文件

约定是：application.yml 放所有环境通用的配置，环境专属的差异部分放进 application-{profile}.yml， 两者会被合并加载，同名配置项以环境专属文件为准（呼应上一篇讲的优先级规则）。

src/main/resources/
  application.yml —— 通用配置：应用名、日志格式、公共超时时间...
  application-dev.yml —— 本地开发：内存数据源、日志级别 DEBUG、关闭限流
  application-test.yml —— 测试环境：连测试库、Mock 第三方支付
  application-prod.yml —— 生产环境：连生产集群、日志级别 WARN、开启限流熔断
application-dev.yml
spring:
  datasource:
    url: jdbc:h2:mem:testdb   # 开发环境用内存数据库，不依赖外部 MySQL
logging:
  level:
    com.supernovacompanies: DEBUG
application-prod.yml
spring:
  datasource:
    url: jdbc:mysql://prod-cluster:3306/ccc
logging:
  level:
    com.supernovacompanies: WARN

启动时指定激活哪个环境

指定 Profile 的几种方式
# 方式一：命令行参数（最常用，优先级最高）
java -jar app.jar --spring.profiles.active=prod

# 方式二：application.yml 里写默认值（本地开发常用）
spring:
  profiles:
    active: dev

# 方式三：环境变量
export SPRING_PROFILES_ACTIVE=prod

@Profile：让 Bean 只在特定环境下生效

光切配置文件还不够——有些场景需要整个 Bean 的实现都不一样。比如开发环境用一个内存 Mock 数据源避免依赖外部服务， 生产环境用真实的数据库连接池。这时候用 @Profile 标注 Bean，容器启动时只会实例化匹配当前激活 Profile 的那个。

DataSourceConfig.java（按环境切换 Bean 实现）
public interface UserRepository {
    User findById(Long id);
}

@Repository
@Profile("dev")  // 只有 dev 环境才会创建这个 Bean
public class MockUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        return new User(id, "mock-user");  // 不需要连真实数据库就能跑通业务逻辑
    }
}

@Repository
@Profile("prod")  // 只有 prod 环境才会创建这个 Bean
public class JpaUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        // 真实走 JPA 查询生产数据库
        return jpaRepository.findById(id).orElseThrow();
    }
}

业务代码里只依赖 UserRepository 接口（这正是上一模块 IoC/DI 讲过的"依赖接口不依赖实现"）， 到底注入哪个实现，完全由当前激活的 Profile 决定，切换环境时业务代码零改动。

常见坑：同一个接口在多个 Profile 下都没匹配到实现。如果忘了给某个 Bean 加 @Profile，或者激活的 Profile 名字拼错（比如配置文件是 application-prod.yml， 但 @Profile 里写成了 "production"），容器启动时会因为找不到可注入的 Bean 而报 NoSuchBeanDefinitionException。 建议把 Profile 名字定义成常量集中管理，避免散落各处手写字符串导致拼写不一致。

组合 Profile：可以同时激活多个

spring.profiles.active 支持用逗号分隔同时激活多个 Profile，常见于"环境 + 功能开关"的组合：

组合激活示例
# 同时激活 prod 环境配置 和 metrics 监控功能
java -jar app.jar --spring.profiles.active=prod,metrics

配置文件级 Profile vs Bean 级 @Profile

维度application-{profile}.yml@Profile 注解
作用对象 配置项的值 整个 Bean 是否被创建
典型场景 数据库地址、日志级别、超时时间等"值不同但结构相同"的配置 需要切换完全不同实现的场景，如 Mock 数据源 vs 真实数据源
粒度 整份配置文件 可以精细到单个 Bean 甚至单个 @Configuration 类
面试高频："Profile 和 @ConditionalOnProperty 有什么区别？"——两者都能按条件控制 Bean 是否生效，但语义不同： @Profile 面向的是"当前处于哪个部署环境"这种粗粒度、全局性的开关；@ConditionalOnProperty（自动配置组讲过）更适合"某个具体配置项开启/关闭某个功能"这种细粒度开关。 实际项目里两者经常配合使用。

核心价值总结

  • 环境隔离：不同环境的配置差异集中管理在各自的 application-{profile}.yml，避免"改了本地忘改生产"的低级错误。
  • Bean 级别切换：@Profile 让整个实现都能按环境切换，而不仅仅是配置值。
  • 启动时一键切换：通过 spring.profiles.active 一个开关决定整套环境配置，符合"一次构建，到处部署"的理念，与上一篇的配置优先级机制天然配合。
  • 与自动配置呼应：@Profile 本质上也是一种 Spring 的条件装配机制，和 core 组讲过的 @Conditional 系列注解同源。

与其他知识点的联系

  • application.properties / yml：Profile 机制直接建立在第一篇讲的"配置优先级"规则之上——application-{profile}.yml 之所以能覆盖 application.yml 里的同名配置，就是因为它在优先级链条里排在更靠后（更晚加载、优先级更高）的位置。
  • @ConfigurationProperties：如果配置类绑定了某个前缀下的字段（比如 OssProperties.bucketName），不同 Profile 下同一个字段可以拿到不同的值，配置类代码本身完全不用改，这正是"批量绑定 + 环境隔离"两个机制叠加使用的典型场景。
  • IoC/DI 与 @Conditional 系列注解：@Profile 本身底层是 Spring 的条件装配机制（@Conditional 的一种特例），容器在扫描 BeanDefinition 阶段就会判断当前激活的 Profile 是否匹配，不匹配的 Bean 定义直接不会进入容器，这与 core 组讲的 IoC 容器启动流程是同一套机制在起作用。
  • 消息队列 RabbitMQ/Kafka 集成：实际项目里开发环境和生产环境往往连接不同的 MQ 集群（比如开发用本地单机 RabbitMQ，生产用高可用集群），这类连接地址的切换就是通过 application-{profile}.yml 天然实现的，不需要在代码里写 if-else 判断环境。
  • @Scheduled 定时任务：某些定时任务只应该在生产环境跑（比如凌晨清理任务），开发环境频繁重启反而会被误触发，可以用 @Profile 标注定时任务所在的 Bean，让它只在特定环境下被创建和注册。
