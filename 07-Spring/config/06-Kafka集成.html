Kafka 集成（进阶）

RabbitMQ 和 Kafka 都叫"消息队列"，但设计目标完全不同——一个是精细控制的邮差，一个是永不停歇的日志流水线。

一句话讲清楚

Kafka 本质上是一个分布式的、可持久化的"提交日志"（Commit Log）系统，而不是传统意义上的消息队列。 它的设计目标是高吞吐量地顺序写入和读取海量事件流（比如埋点日志、订单流水、监控指标）， 而 RabbitMQ 的设计目标是灵活、可靠地投递单条消息，更偏向传统消息队列语义（一条消息处理完就该从队列消失）。

关键差异的根源：RabbitMQ 的队列是"一次性消费"模型——消息被消费确认后就从队列删除。 Kafka 的 Topic 更像一份只能追加写入的日志文件，消息被消费后依然保留（按配置的保留策略，比如 7 天）， 多个消费者组可以各自独立地从任意位置重新读取同一份日志。这个"日志"设计决定了 Kafka 天生适合"消息重放""多消费者独立消费同一份数据"这类场景。

Kafka 核心概念速览

Topic（主题） —— 逻辑上的消息分类，类似 RabbitMQ 的 Queue，但消息不会因消费而删除
  ↓ 物理上被拆分成多个
Partition（分区） —— 每个分区内部消息严格有序，分区之间并行读写，是 Kafka 高吞吐的关键
  ↓ 每条消息在分区内有一个递增的
Offset（偏移量） —— 消费者通过记录自己消费到哪个 offset，决定下次从哪里继续读

Consumer Group（消费者组） —— 同一个组内的消费者分摊消费一个 Topic 的不同分区（负载均衡）
  不同消费者组之间互不影响，各自独立维护自己的 offset（这就是"多消费者独立消费同一份数据"的实现原理）

KafkaTemplate 生产者 与 @KafkaListener 消费者

application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: order-service-group
      auto-offset-reset: earliest  # 该组首次消费时从最早的消息开始读
OrderEventProducer.java（生产者）
@Service
@RequiredArgsConstructor
public class OrderEventProducer {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public void publish(OrderEvent event) {
        // key 用 orderId，保证同一订单的所有事件落到同一分区，从而保持顺序
        kafkaTemplate.send("order-events", event.getOrderId(), event);
    }
}
OrderEventConsumer.java（消费者）
@Component
public class OrderEventConsumer {

    @KafkaListener(topics = "order-events", groupId = "order-service-group")
    public void onEvent(OrderEvent event) {
        // 处理事件，比如写入数据仓库做后续分析
        handle(event);
    }
}
为什么按 key 分区很重要：Kafka 只保证单个分区内消息有序，不保证跨分区的全局有序。 如果不指定 key（或 key 相同的消息落到不同分区），同一个订单的"创建"和"支付"两个事件可能被并发消费、乱序处理。 实践中通常用业务主键（如 orderId）作为 key，让相关消息始终落入同一分区，从而在业务层面获得"局部有序"的保证。

RabbitMQ vs Kafka：该怎么选

维度RabbitMQKafka
吞吐量 万级/秒，适合中等流量 百万级/秒，为高吞吐设计
消息顺序 单队列内严格有序 仅分区内有序，跨分区无法保证全局顺序
消息保留 消费确认后即删除（可配置死信队列兜底） 按时间/大小策略保留，消费后依然存在，支持重放
路由灵活性 Exchange 支持 Direct/Fanout/Topic 多种复杂路由 路由能力简单，主要靠 Topic + Partition
典型场景 业务解耦、任务队列、延迟消息、RPC 式异步通信 日志采集、埋点上报、事件溯源、大数据流式处理管道
运维复杂度 相对简单，单机也能跑 依赖 ZooKeeper/KRaft 做集群协调，分布式部署更重
选型误区：不要单纯因为"Kafka 吞吐量更高"就在所有场景都选 Kafka。如果业务需要复杂的路由规则、需要严格的单条消息确认与重试语义、 或者团队没有运维 Kafka 集群的经验，RabbitMQ 往往是更务实的选择。Kafka 更适合"数据量大、顺序要求局限在分区级别、需要多方各自独立消费同一份数据流"的场景， 比如构建数据管道，而不是单纯替代业务系统里的任务队列。

核心价值总结

  • 本质定位不同：Kafka 是持久化的分布式日志系统，RabbitMQ 是传统消息队列，二者不是同一赛道的竞品，选型要看场景而非单纯比吞吐量。
  • 分区决定并行度与有序性：Kafka 的高吞吐来自分区并行，但代价是只能保证分区内有序，需要用 key 路由来获得业务层面的局部顺序保证。
  • 消费者组实现独立消费：Kafka 允许多个消费者组各自独立、重复地消费同一份数据，这是 RabbitMQ 队列模型很难做到的。
  • 编程模型高度一致：KafkaTemplate/@KafkaListener 和前一篇的 RabbitTemplate/@RabbitListener 用法几乎对称，切换心智负担不大，难的是理解背后的设计差异。
