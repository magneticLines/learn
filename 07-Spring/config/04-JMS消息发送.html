JMS 消息发送

用户下单后要发短信、发邮件、扣库存、通知物流——如果全部同步执行，用户要等好几秒才能看到"下单成功"，消息队列就是用来解决这个问题的。

一句话讲清楚

消息队列（Message Queue）的核心价值是异步解耦：把"必须立刻做完"和"可以晚点再做"的事情分开—— 主流程只需要把一条消息丢进队列就立刻返回，真正耗时的操作（发短信、写日志、数据同步）由另一个消费者在后台异步处理。 JMS（Java Message Service）则是 Java 官方定义的一套消息服务标准接口（不是具体实现）， 规定了"什么是消息""什么是队列/主题""怎么发送/接收"这些概念和 API，ActiveMQ 等是 JMS 规范的具体实现产品。

消息队列解决了什么工程问题

问题一：异步解耦，避免主流程被拖慢

OrderService.java（反例 — 同步调用拖慢主流程）
@Service
@RequiredArgsConstructor
public class OrderService {

    private final SmsService smsService;

    public void createOrder(OrderDTO dto) {
        orderRepo.save(dto);
        // 短信网关响应慢（几百毫秒到几秒），下单接口被迫等它返回才能给用户响应
        smsService.sendOrderConfirmSms(dto.getPhone());
        // 如果短信网关这时候超时或抖动，整个下单请求直接卡住甚至失败
    }
}

用消息队列改造后，createOrder 只需要把"发送订单确认短信"这个意图作为一条消息丢进队列，立刻返回给用户"下单成功"， 短信服务是否临时故障、响应多慢，都不再影响下单主流程。

问题二：削峰填谷

秒杀场景下，瞬间涌入的下单请求可能是数据库处理能力的几十倍。如果每个请求都直接写库，数据库会被打爆。 做法是先把请求丢进消息队列缓冲，再由消费者按数据库能承受的速率匀速处理，把"瞬时高峰"削平成"持续的稳定流量"。

建立联系：这和前面 core 组讲的"IoC 解耦对象依赖"是同一种设计哲学在不同层面的体现——IoC 解耦的是对象与对象之间的创建关系， 消息队列解耦的是服务与服务之间的调用时序，两者都是用"引入一个中间层"的方式换取更低的耦合度和更高的可维护性。

JMS：标准接口而非具体实现

JMS 只是一套 Java 规范（类似 JDBC 之于数据库驱动），定义了 ConnectionFactory、Destination、MessageProducer、MessageConsumer 等核心接口。 ActiveMQ、Artemis 是 JMS 规范的具体实现产品。而 RabbitMQ 和 Kafka 都不是 JMS 的实现——它们有自己的协议和客户端 API， 这也是为什么 Spring 没有把 RabbitMQ/Kafka 硬塞进 JMS 体系，而是分别提供了独立的 spring-boot-starter-amqp 和 spring-kafka（下两篇会详细展开）。

Spring 消息编程模型的分层关系
  JMS 规范 —— JmsTemplate —— ActiveMQ / Artemis（遵循 JMS 标准）
  AMQP 协议 —— RabbitTemplate —— RabbitMQ（遵循 AMQP 标准）
  Kafka 协议 —— KafkaTemplate —— Kafka（自有协议，非 JMS/AMQP）

三者概念相通（生产者/消费者/队列或主题），但 API 和协议各自独立，不能互相替换底层依赖

JmsTemplate：统一的消息发送模板

Spring 沿用了它一贯的"Template 模式"（呼应你可能接触过的 JdbcTemplate、RestTemplate）——把连接管理、异常转换、资源释放这些样板代码封装起来， 业务代码只需要调用一个 send 方法。

OrderMessageSender.java（用 JmsTemplate 发送消息）
@Service
@RequiredArgsConstructor
public class OrderMessageSender {

    private final JmsTemplate jmsTemplate;  // Spring Boot 自动装配，无需手写工厂

    public void notifyOrderCreated(OrderDTO dto) {
        // 把订单消息发到名为 "order.created" 的目的地（队列）
        jmsTemplate.convertAndSend("order.created", dto);
        // convertAndSend 内部自动完成对象 -> Message 的序列化转换
    }
}
OrderMessageListener.java（用 @JmsListener 消费消息）
@Component
public class OrderMessageListener {

    @JmsListener(destination = "order.created")
    public void onOrderCreated(OrderDTO dto) {
        // 在独立的消费线程中异步执行，不阻塞下单主流程
        sendSms(dto.getPhone());
    }
}
为下文铺垫：JMS 定义的"生产者发消息、消费者监听消息"这套编程范式，会在接下来 RabbitMQ 和 Kafka 两篇里反复出现—— 只是把 JmsTemplate 换成 RabbitTemplate / KafkaTemplate，把 @JmsListener 换成 @RabbitListener / @KafkaListener。 理解了这一篇的编程模型，后面两篇学起来会快很多。

JMS 的两种消息模型

模型特点类比
Point-to-Point（队列） 一条消息只会被一个消费者消费，多个消费者是竞争关系 银行排队叫号，一个号只能被一个窗口处理
Publish/Subscribe（主题） 一条消息会被所有订阅了该主题的消费者各自消费一份 公众号推送，每个订阅者都会收到同一条推送
陷阱：消息队列引入之后，业务从"同步强一致"变成了"异步最终一致"——下单接口返回成功，不代表短信一定发出去了。 初学者容易忽略这一点，误以为消息队列和同步调用在业务语义上完全等价。实际使用中要考虑消息丢失、重复消费、顺序错乱等问题， 这也是为什么下一篇会专门讲 RabbitMQ 的消息确认机制。

核心价值总结

  • 异步解耦：把非核心、耗时的操作从主流程剥离，主流程只负责"发个消息"，响应速度不受下游服务影响。
  • 削峰填谷：用队列做缓冲层，把瞬时高并发压力转换成消费者可以匀速处理的稳定流量。
  • JMS 是标准而非实现：理解 JMS 定义的生产者/消费者/队列/主题概念，是理解后续 RabbitMQ、Kafka 消息模型的基础。
  • Template 编程范式统一：JmsTemplate/RabbitTemplate/KafkaTemplate 三者 API 风格高度一致，学一套能力可以迁移到另外两套。
