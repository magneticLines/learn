任务调度 @Scheduled 与 @Async

定时清理过期数据、异步发送通知——这类"不需要立刻执行"或"需要按周期执行"的任务，Spring 提供了比原生 Java 更简洁的声明式方案。

一句话讲清楚

@Scheduled 让一个方法按固定周期或 cron 表达式自动执行，本质是对 ScheduledExecutorService 的声明式封装； @Async 让一个方法的调用立刻返回、真正的执行放到另一个线程池里异步完成，本质是 Spring AOP 生成的一层代理拦截了方法调用。 两者都是"用注解代替手写线程管理代码"，把并发细节留给框架处理，业务代码只需要声明意图。

@Scheduled：声明式定时任务

Java 原生的 Timer 单线程执行、一个任务抛异常会拖垮整个 Timer；ScheduledExecutorService 改进了这一点但用起来仍然啰嗦， 需要手写线程池、手写周期计算逻辑。Spring 的 @Scheduled 把这些全部封装起来，一个注解就够了。

启动类开启定时任务能力
@SpringBootApplication
@EnableScheduling  // 必须显式开启，否则 @Scheduled 注解不生效
public class Application { ... }
OrderCleanupTask.java（定时清理任务）
@Component
public class OrderCleanupTask {

    // 每天凌晨 2 点执行一次，cron 表达式：秒 分 时 日 月 周
    @Scheduled(cron = "0 0 2 * * ?")
    public void cleanExpiredOrders() {
        orderRepo.deleteByStatusAndCreateTimeBefore(OrderStatus.EXPIRED, daysAgo(30));
    }

    // 上一次执行结束后，固定间隔 60 秒再执行下一次（避免任务执行时间超过间隔导致重叠）
    @Scheduled(fixedDelay = 60_000)
    public void syncInventory() {
        inventoryService.syncFromWarehouse();
    }
}
坑：默认单线程执行所有 @Scheduled 任务。如果不额外配置线程池，Spring 默认只用一个线程顺序执行所有定时任务—— 如果 cleanExpiredOrders 跑了 10 分钟，同一时间点该触发的 syncInventory 会被阻塞，等前一个任务执行完才能开始，看起来像是"漏跑了一次"。 生产环境务必自定义一个专用线程池：
SchedulingConfig.java（自定义调度线程池）
@Configuration
public class SchedulingConfig implements SchedulingConfigurer {
    @Override
    public void configureTasks(ScheduledTaskRegistrar registrar) {
        registrar.setScheduler(Executors.newScheduledThreadPool(10));  // 多个定时任务并行执行，互不阻塞
    }
}

@Async：让方法调用立刻返回

与 @Scheduled 处理"周期性"不同，@Async 处理的是"这个方法很耗时，调用方不想等它执行完"的场景。 比如用户注册后要发欢迎邮件，邮件发送本身可能要几百毫秒，不应该让注册接口等邮件发完才返回。

启动类开启异步能力
@SpringBootApplication
@EnableAsync  // 必须显式开启
public class Application { ... }
NotifyService.java（异步发送邮件）
@Service
public class NotifyService {

    @Async("notifyExecutor")  // 指定用哪个自定义线程池执行
    public void sendWelcomeEmail(String email) {
        // 该方法在独立线程中执行，调用方不会被阻塞
        mailClient.send(email, "欢迎注册");
    }
}

陷阱：@Async 方法必须由其他 Bean 调用才生效

@Async 和 @Transactional 一样，都是靠 Spring AOP 生成代理对象来实现的。只有通过代理对象调用方法，才会经过代理的拦截逻辑（异步调度、事务开启等）； 如果是同一个类内部方法自己调自己（this 调用），走的是原始对象的方法调用，完全绕过了代理，@Async 会静默失效——方法照常同步执行，没有任何报错提示。

反例 — 自调用导致 @Async 失效
@Service
public class UserService {

    @Autowired
    private NotifyService notifyService;

    public void register(UserDTO dto) {
        userRepo.save(dto);
        sendWelcomeEmailInternal(dto.getEmail());  // 同类内自调用，@Async 完全不生效
    }

    @Async  // 加了注解但没用——this.sendWelcomeEmailInternal() 绕过了代理
    public void sendWelcomeEmailInternal(String email) { ... }
}
正例 — 把异步方法拆到另一个 Bean，通过依赖注入调用
@Service
@RequiredArgsConstructor
public class UserService {

    private final NotifyService notifyService;  // 独立的 Bean，调用时经过 Spring 生成的代理

    public void register(UserDTO dto) {
        userRepo.save(dto);
        notifyService.sendWelcomeEmail(dto.getEmail());  // 跨 Bean 调用，@Async 正常生效
    }
}
面试高频，且是老朋友：这和事务管理篇讲过的"@Transactional 自调用失效"是完全同一个根因—— Spring AOP 默认基于 JDK 动态代理或 CGLIB 生成子类代理，代理逻辑只能拦截"从外部经过代理对象发起"的调用， 类内部的 this.xxx() 调用天然绕开代理。凡是靠 AOP 实现的注解（@Transactional、@Async、@Cacheable 等）都有这个共性陷阱，记住一次就能举一反三。

陷阱：默认线程池不满足生产需求

不指定线程池名字时，@Async 使用 Spring 内置的 SimpleAsyncTaskExecutor——它不复用线程，每次调用都新建一个线程， 没有队列缓冲、没有最大线程数限制。高并发下会无限制创建线程，直接把服务器资源耗尽。生产环境必须自定义线程池 Bean。

AsyncConfig.java（自定义线程池）
@Configuration
public class AsyncConfig {

    @Bean("notifyExecutor")
    public Executor notifyExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(20);
        executor.setQueueCapacity(200);  // 超过核心线程数的任务先排队，而不是立刻建新线程
        executor.setThreadNamePrefix("notify-");  // 便于线程 dump 时定位问题
        executor.initialize();
        return executor;
    }
}

@Scheduled vs @Async 对比

维度@Scheduled@Async
触发方式 按周期/cron 自动触发，无需外部调用 由业务代码显式调用方法触发
解决的问题 "这件事需要按时间规律重复做" "这件事很耗时，调用方不想等"
与原生 API 对比 封装 ScheduledExecutorService，省去手写周期计算逻辑 封装线程池提交任务，省去手写 Future/Callback 样板代码
共同点 都需要显式开启（@EnableScheduling / @EnableAsync），都强烈建议自定义线程池而非用默认配置

核心价值总结

  • 声明式替代手写并发代码：@Scheduled/@Async 把线程池管理、周期计算这些样板代码封装起来，业务代码只表达意图。
  • AOP 代理是两者共同的实现基础：理解了这一点，就能理解"自调用失效"这个陷阱为什么会发生，也能推广到 @Transactional、@Cacheable 等所有基于 AOP 的注解。
  • 默认配置只适合演示，不适合生产：默认调度线程池是单线程、默认异步执行器不复用线程，两者都必须在生产环境替换为自定义的、参数明确的线程池。
  • 本组小结：从外部化配置（yml/Profile）到类型安全绑定，再到消息队列异步解耦，最后到定时任务与异步方法——这些都是让应用更好地适配"多环境、高并发、松耦合"生产场景的配置与集成能力。
