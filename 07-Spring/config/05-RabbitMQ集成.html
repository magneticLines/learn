RabbitMQ 集成

生产者从不直接把消息塞进队列，而是发给一个"交换机"，由交换机按规则决定消息该流向哪个队列——理解这一层间接，是用好 RabbitMQ 的关键。

一句话讲清楚

RabbitMQ 是 AMQP（Advanced Message Queuing Protocol）协议最流行的实现之一。它最核心的设计是： 生产者永远不直接往队列发消息，而是发给 Exchange（交换机）；Exchange 根据 Binding（绑定规则）把消息路由到一个或多个 Queue（队列）；消费者只从队列里取消息。 这一层"生产者与队列解耦"的设计，让路由逻辑可以灵活配置而不需要改动生产者代码。

三个核心概念：Exchange / Queue / Binding

消息流转路径
生产者（Producer）
  ↓ 发送消息（不知道、也不关心最终去哪个队列）
Exchange（交换机） —— 只负责路由，不存储消息
  ↓ 按 Binding（绑定规则：routing key 匹配）
Queue（队列） —— 真正存储消息，等待被消费
  ↓
消费者（Consumer）

为什么不让生产者直接发给队列？因为这样生产者就必须知道"这条消息该进哪个队列"，队列一旦拆分或新增，生产者代码就要跟着改。 引入 Exchange 之后，生产者只需要知道"发给哪个交换机、带什么 routing key"，路由规则（Binding）可以独立配置和调整，生产者代码完全不受影响。

Exchange 类型路由规则典型场景
Direct routing key 完全匹配才路由 点对点精确投递，如按订单状态精确路由到对应处理队列
Fanout 忽略 routing key，广播给所有绑定的队列 一条消息需要被多个系统同时感知，如订单创建后同时通知短信、积分、物流三个系统
Topic routing key 支持通配符模糊匹配（* 一个词，# 任意多个词） 按类别灵活订阅，如订阅 order.*.created 匹配所有类型订单的创建事件

生产者：RabbitTemplate 发送消息

RabbitConfig.java（声明 Exchange / Queue / Binding）
@Configuration
public class RabbitConfig {

    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }

    @Bean
    public Queue smsQueue() {
        return new Queue("order.sms.queue", true);  // durable=true，队列在 Broker 重启后依然存在
    }

    @Bean
    public Binding smsBinding(Queue smsQueue, DirectExchange orderExchange) {
        // routing key 为 "order.created" 的消息，路由到 smsQueue
        return BindingBuilder.bind(smsQueue).to(orderExchange).with("order.created");
    }
}
OrderMessageSender.java（生产者发送消息）
@Service
@RequiredArgsConstructor
public class OrderMessageSender {

    private final RabbitTemplate rabbitTemplate;  // Spring Boot 自动装配

    public void notifyOrderCreated(OrderDTO dto) {
        // 发给 order.exchange 交换机，routing key 是 order.created
        // 生产者完全不知道消息最终进了哪个队列，这就是解耦
        rabbitTemplate.convertAndSend("order.exchange", "order.created", dto);
    }
}

消费者：@RabbitListener 监听队列

SmsMessageListener.java（消费者）
@Component
public class SmsMessageListener {

    @RabbitListener(queues = "order.sms.queue")
    public void onOrderCreated(OrderDTO dto) {
        // 在独立的消费线程池中执行，与下单主流程完全异步
        sendSms(dto.getPhone());
    }
}
application.yml（连接配置）
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest

消息确认机制：防止消息丢失

消息在传输过程中有三段旅程：生产者 → Exchange、Exchange → Queue、Queue → 消费者， 每一段都可能因为网络抖动、Broker 故障、消费者处理异常而导致消息丢失。RabbitMQ 提供了对应的确认机制来保障可靠性。

机制保障阶段作用
Publisher Confirm 生产者 → Broker Broker 确认收到消息后回调通知生产者，生产者可以据此重发失败的消息
Mandatory + Return Callback Exchange → Queue 如果消息没有匹配到任何队列（路由失败），触发回调而不是静默丢弃
Consumer Ack（手动确认） Queue → 消费者 消费者处理成功后显式 ack，Broker 才会把消息从队列删除；处理失败可以 nack 让消息重新入队
application.yml（开启手动确认，生产环境推荐）
spring:
  rabbitmq:
    publisher-confirm-type: correlated  # 开启生产者确认
    listener:
      simple:
        acknowledge-mode: manual  # 消费者手动确认，而非默认的自动确认
手动 ack 示例
@RabbitListener(queues = "order.sms.queue")
public void onOrderCreated(OrderDTO dto, Channel channel,
                            @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
    try {
        sendSms(dto.getPhone());
        channel.basicAck(tag, false);  // 处理成功，显式确认，Broker 才会删除这条消息
    } catch (Exception e) {
        channel.basicNack(tag, false, true);  // 处理失败，重新入队，等待重试
    }
}
陷阱：默认的自动确认模式（acknowledge-mode: none）在消息一被消费者接收就立刻从队列删除， 不管消费者代码是否真正处理成功。如果消费逻辑抛异常，消息已经丢了，且没有任何重试机会。 生产环境务必改成手动确认，并且要处理好"重复消费"的幂等性问题——因为 nack 重新入队后，同一条消息可能被处理不止一次。

核心价值总结

  • Exchange 引入的间接层：生产者只关心发给哪个交换机，路由到哪个队列由 Binding 规则决定，两者解耦，方便后续扩展新的消费队列。
  • 三种 Exchange 类型：Direct 精确匹配、Fanout 广播、Topic 模糊匹配，按场景选择合适的路由方式。
  • 可靠性依赖确认机制：Publisher Confirm 保障生产者到 Broker 的可靠投递，Consumer Ack 保障消费成功后才删除消息，二者缺一不可。
  • 幂等性是配套要求：只要用了消息重试机制，就必须假设消息可能被重复消费，业务逻辑要做好幂等处理。

与其他知识点的联系

  • application.properties / yml：RabbitMQ 的连接地址、用户名密码、确认模式这些参数，都是通过 spring.rabbitmq.* 前缀外部化配置注入的，本质上是第一篇讲的机制在中间件客户端上的直接应用，切换开发/生产环境的 Broker 地址不需要改一行 Java 代码。
  • Profile 多环境配置：开发环境通常连本地单机 RabbitMQ（甚至用内嵌的测试 Broker），生产环境连高可用集群，这类差异天然放进 application-{profile}.yml，配合 Profile 切换即可，不需要在业务代码里判断当前环境。
  • Kafka 集成：两者都解决"生产者和消费者解耦"的问题，但设计目标不同——RabbitMQ 面向可靠的单条消息投递，Kafka 面向高吞吐的事件流，选型时要先想清楚"这条消息是一次性任务，还是需要被多方重复消费的事件"。
  • @Async 异步编程：@RabbitListener 消费者方法本身就运行在独立的消费线程池中，与主业务线程天然异步，这和 @Async 解决的"耗时操作不阻塞调用方"是同一类设计目标，只是 MQ 消费额外提供了跨进程、持久化的能力，比单机内的线程池异步更适合"发出去之后即使当前服务重启也不会丢"的场景。
  • 幂等性与事务：手动 Ack + Nack 重试机制意味着同一条消息可能被处理多次，这就要求消费者逻辑具备幂等性（比如用订单号做唯一约束防重复插入），这一设计考量会在后续讲事务和分布式一致性的篇章里反复出现。
