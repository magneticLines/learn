@ConfigurationProperties 类型安全配置

配置项一多，一个个 @Value 读取就变成了体力活——把一组相关配置整体绑定到一个 Java 对象上，才是可维护的做法。

一句话讲清楚

@ConfigurationProperties 能把配置文件里某个前缀下的所有配置项，一次性、按字段名自动映射到一个 Java 对象的属性上， 整个过程自带类型转换（字符串转 int、boolean、List、甚至嵌套对象）和 IDE 智能提示，是 Spring Boot 官方推荐的"批量读配置"方式。

反例：用 @Value 逐个读取

假设要读取一组阿里云 OSS 相关的配置，@Value 写法看起来是这样的：

OssUploader.java（反例 — @Value 逐个注入）
@Component
public class OssUploader {

    @Value("${oss.endpoint}")
    private String endpoint;

    @Value("${oss.access-key}")
    private String accessKey;

    @Value("${oss.secret-key}")
    private String secretKey;

    @Value("${oss.bucket-name}")
    private String bucketName;

    @Value("${oss.max-file-size-mb:10}")  // 手写默认值，容易和别处不一致
    private Integer maxFileSizeMb;
}

这样写的问题

  • 啰嗦重复：每个字段都要重复写一遍完整的 key 路径，配置项一多，字段列表会变得很长。
  • 没有类型安全：如果 oss.max-file-size-mb 在配置文件里被拼错成 oss.max-fileSize-mb，编译期不会报错，只会在运行时该字段永远是 null。
  • 无法复用：如果另一个类也需要 OSS 配置，只能把这几个 @Value 再抄一遍。
  • 不支持嵌套结构：@Value 对付 List、Map 或多级嵌套对象非常吃力。

正例：@ConfigurationProperties 绑定强类型对象

application.yml
oss:
  endpoint: oss-cn-hangzhou.aliyuncs.com
  access-key: xxxxxx
  secret-key: yyyyyy
  bucket-name: ccc-prod
  max-file-size-mb: 10
  allowed-types:
    - jpg
    - png
    - pdf
OssProperties.java（正例 — 强类型配置对象）
@Component
@ConfigurationProperties(prefix = "oss")  // 绑定 oss.* 下的所有配置项
@Data  // Lombok 生成 getter/setter，@ConfigurationProperties 依赖 setter 完成绑定
public class OssProperties {

    private String endpoint;
    private String accessKey;       // yml 里 access-key 自动转驼峰 accessKey
    private String secretKey;
    private String bucketName;
    private Integer maxFileSizeMb = 5;  // 直接用字段默认值，比 @Value 的字符串默认值更自然
    private List<String> allowedTypes;  // 原生支持 List / Map / 嵌套对象
}
OssUploader.java（使用方 — 直接注入整个配置对象）
@Service
@RequiredArgsConstructor
public class OssUploader {

    private final OssProperties props;  // 一个字段拿到所有配置，有 IDE 智能提示

    public void upload(MultipartFile file) {
        if (!props.getAllowedTypes().contains(extOf(file))) {
            throw new IllegalArgumentException("不支持的文件类型");
        }
        // ... 用 props.getEndpoint()、props.getBucketName() 等发起上传
    }
}
命名转换规则：Spring Boot 的 Relaxed Binding（宽松绑定）会自动把 yml 里的短横线命名（kebab-case）转换成 Java 字段的驼峰命名（camelCase）， 所以配置里推荐统一写 max-file-size-mb 这种风格，字段则正常写 Java 驼峰 maxFileSizeMb，不需要额外注解。

两种注册方式：@Component 还是 @EnableConfigurationProperties

上面的例子里在 OssProperties 上直接标了 @Component，让它作为一个普通 Bean 被组件扫描到——这是最简单的方式， 适合业务代码里自己写的配置类。但如果你在开发一个可复用的库或 starter（呼应 core 组讲过的自动配置机制）， 更推荐用 @EnableConfigurationProperties 显式声明，因为库的自动配置类往往不在业务方的组件扫描路径下。

OssProperties.java（去掉 @Component，作为库内配置类）
@ConfigurationProperties(prefix = "oss")
@Data
public class OssProperties { ... }
OssAutoConfiguration.java（自动配置类中显式启用）
@Configuration
@EnableConfigurationProperties(OssProperties.class)  // 显式注册，且不依赖组件扫描路径
public class OssAutoConfiguration {

    @Bean
    public OssUploader ossUploader(OssProperties props) {
        return new OssUploader(props);
    }
}

@Value vs @ConfigurationProperties 对比

维度@Value@ConfigurationProperties
适用场景 零星读取一两个配置项 一组相关配置项批量绑定
类型安全 弱，拼错 key 编译不报错，运行时才发现 强，字段名与配置结构对应，IDE 有自动补全提示
复杂类型（List/Map/嵌套对象） 支持较弱，需要 SpEL 或额外处理 原生支持，包括多层嵌套对象
SpEL 表达式 支持 @Value("#{...}") 不支持
校验 不支持 可配合 @Validated + JSR-303 注解做配置校验
常见坑：@ConfigurationProperties 绑定依赖 setter（或构造器绑定），如果字段没有生成 setter（比如漏加 @Data 或手写时漏写）， 该字段会静默保持 null，不会有任何报错提示。建议给配置类加上 @Validated + @NotNull 等校验注解，让配置缺失在启动阶段就快速失败，而不是运行到用的时候才报空指针。

核心价值总结

  • 批量绑定：一组相关配置一次性映射到强类型对象，避免字段爆炸式重复声明 @Value。
  • 类型安全 + IDE 提示：配置项名和 Java 字段直接对应，重构、补全都更可靠。
  • 支持复杂结构：List、Map、嵌套对象开箱即用，@Value 很难优雅地做到这些。
  • 库开发首选：结合 @EnableConfigurationProperties，是编写 Spring Boot Starter 时暴露可配置项的标准方式。

与其他知识点的联系

  • application.properties / yml：@ConfigurationProperties 绑定的数据来源就是上一篇讲的配置文件，理解了配置的优先级和合并规则，才能理解为什么修改 application-{profile}.yml 里的值会覆盖 application.yml 里绑定到同一个字段的默认值。
  • Profile 多环境配置：同一个 OssProperties 类在不同环境下可以绑定到不同的值（开发环境用测试 Bucket，生产环境用正式 Bucket），配置类本身不用变，靠下一篇的 Profile 机制切换配置来源即可，这正体现了"配置与代码分离"的价值。
  • IoC/DI 依赖注入回顾：OssProperties 本质上就是一个普通的 Spring Bean（无论是 @Component 还是 @EnableConfigurationProperties 注册），它被容器创建、赋值之后，再通过构造器注入交给 OssUploader 使用，走的是 core 组讲过的标准 IoC 容器管理流程，只是多了一步"从配置源绑定字段值"。
  • Spring Boot Starter 开发：@EnableConfigurationProperties 这种注册方式是编写可复用 Starter 时暴露可配置项的标准做法，因为库的自动配置类不在业务方的组件扫描路径下，这一点在后面讲自动配置机制的篇章里会进一步展开。
  • @Validated 配置校验：结合 core 组讲过的 Bean 生命周期，配置类上加 @Validated + JSR-303 注解后，校验会在容器初始化该 Bean 时执行，让"配置缺失/格式错误"这类问题在应用启动阶段就快速失败，而不是留到运行期才暴露成空指针异常。
