配置与集成

application.properties / yml

同一份代码打包一次，靠改配置文件就能适配开发、测试、生产多套环境——这就是"外部化配置"要解决的问题。

一句话讲清楚

外部化配置（Externalized Configuration）指的是：把数据库地址、端口号、密钥这些"会随环境变化"的值从代码里剥离出来，放进独立的配置文件（或环境变量、命令行参数）， 应用启动时读取它们。Spring Boot 约定优先于配置，默认就会读取 src/main/resources 下的 application.properties 或 application.yml，不需要任何额外配置。

为什么这么设计：如果把数据库密码写死在 Java 代码里，切换到生产环境就得改代码、重新编译、重新打包——每个环境都要维护一份不同的 jar 包，版本管理直接失控。 外部化配置让"一次构建，到处部署"（Build Once, Deploy Anywhere）成为可能：同一个 jar 包，换一个配置文件就能跑在不同环境。

properties vs yml：两种格式怎么选

两者本质上是同一份配置数据的两种书写方式，Spring Boot 对两者的支持是完全对等的，选哪个纯粹是团队习惯问题。

application.properties（扁平的 key=value）
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/ccc
spring.datasource.username=root
spring.datasource.password=123456
spring.datasource.hikari.maximum-pool-size=20
application.yml（用缩进表达层级）
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ccc
    username: root
    password: 123456
    hikari:
      maximum-pool-size: 20

可以看到，spring.datasource.hikari.maximum-pool-size 这种多级前缀，properties 里每行都要把完整路径重复一遍； yml 靠缩进天然表达出"层层嵌套"的树形结构，公共前缀只写一次，配置项一多，可读性差距非常明显。

yml 最容易踩的坑：缩进必须用空格，不能用 Tab。而且同一层级的缩进空格数必须完全一致，否则会解析出错或者层级挂错父节点（比如误把 username 挂到了 server 下面）。 IDE（IDEA）通常会对 yml 做语法高亮和校验，但纯文本编辑器很容易踩坑，建议开启"显示空白字符"来肉眼确认缩进。

配置的优先级：谁说了算

实际项目里同一个配置项经常会在多个地方出现（比如本地配置文件写了默认值，运维在 Kubernetes 里又用环境变量覆盖了一遍）。 Spring Boot 定义了一套明确的优先级顺序，优先级越高的来源，越晚被加载，会覆盖前面的值：

优先级从高到低（高的覆盖低的）
① 命令行参数：java -jar app.jar --server.port=9090
② Java 系统属性：-Dserver.port=9090
③ 操作系统环境变量：SERVER_PORT=9090
④ application-{profile}.yml（激活的环境专属配置）
⑤ application.yml（默认配置，jar 包内置）
⑥ @PropertySource 指定的自定义配置文件
⑦ 代码里 @Value 的默认值兜底
实战场景：本地开发用 application.yml 里写死的默认端口 8080；容器化部署时运维不需要改任何文件， 直接在 Kubernetes Deployment 里设置环境变量 SERVER_PORT=80 就能覆盖，这正是利用了环境变量优先级高于配置文件这条规则。 这也是为什么生产环境的密钥、密码往往通过环境变量或配置中心（如 Nacos、Apollo）注入，而不是直接写在 yml 里提交进代码仓库。

properties vs yml 对比总结

维度propertiesyml
层级表达 每行重复完整 key 路径，冗长 缩进表达层级，多级配置更简洁
列表/数组 list[0]=a、list[1]=b，繁琐 - a / - b，原生支持，更直观
易错点 行多了容易看花眼，但格式本身不容易错 对缩进极度敏感，空格数不对就解析失败或挂错层级
多文档支持 不支持在一个文件里定义多个 Profile 支持用 --- 分隔符在一个文件内写多个 Profile 片段

核心价值总结

  • 外部化配置：把易变的环境相关值从代码中剥离，实现"一次构建，多处部署"。
  • yml 更适合复杂配置：层级越深、配置项越多，yml 缩进表达的优势越明显；简单场景两者皆可。
  • 优先级机制：命令行参数、环境变量可以在不改代码不改文件的情况下临时覆盖配置，这是容器化部署的基础能力。
  • 为下一节做铺垫：配置项一多，逐个用 @Value 读取会很啰嗦，下一篇的 @ConfigurationProperties 就是解决这个问题的。

与其他知识点的联系

  • @ConfigurationProperties：本篇讲的配置文件是"原料"，配置项一多，逐个用 @Value 读取会很啰嗦，下一篇的 @ConfigurationProperties 就是把 yml/properties 里某个前缀下的配置整体映射成强类型对象的标准做法。
  • Profile 多环境配置：本篇讲的优先级规则（命令行 > 环境变量 > application-{profile}.yml > application.yml）是 Profile 机制能生效的底层基础——Profile 本质上就是"按环境名加载一份专属的 application-{profile}.yml，再叠加覆盖默认配置"。
  • 消息队列（RabbitMQ/Kafka）连接配置：后面几篇里 spring.rabbitmq.*、spring.kafka.* 这些连接参数，都是通过本篇讲的 application.yml 外部化配置注入的，本质上是同一套机制在不同中间件上的应用。
  • 生产环境敏感信息管理：密码、密钥这类敏感配置不应该直接写进 yml 提交进代码仓库，而是靠本篇讲的"环境变量/命令行参数优先级更高"这条规则，在部署时通过 Kubernetes Secret 或配置中心（Nacos、Apollo）注入，配置文件里只留默认值或占位符。
  • IoC 容器读取配置的时机：Spring 容器启动过程中会先加载并合并所有配置源，之后才会执行 Bean 的实例化和依赖注入（呼应 core 组讲的 IoC 容器启动三步），所以配置源的优先级必须在 Bean 创建之前就确定下来，这也是为什么命令行参数能覆盖配置文件里的默认值。
