@Controller 与 @RequestMapping

上一篇讲了 DispatcherServlet 怎么"找到"Controller 方法，这一篇讲清楚路由是怎么"声明"出来的——以及冗长写法背后藏着的安全隐患。

一句话讲清楚

@RequestMapping 是 Spring MVC 里声明"这个方法处理哪个 URL、哪种 HTTP 方法"的核心注解， HandlerMapping 启动时扫描所有标注了它的方法，构建出前一篇提到的"路由表"。 @GetMapping/@PostMapping/@PutMapping/@DeleteMapping 本质上是 @RequestMapping 的组合注解（Meta-annotation），内部通过 @AliasFor 把 method 属性固定成对应的 HTTP 方法，帮你省掉重复书写。

GetMapping.java（Spring 源码节选，理解组合注解本质）
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@RequestMapping(method = RequestMethod.GET)  // 关键：@GetMapping 本身就是被 @RequestMapping 标注的注解
public @interface GetMapping {

    @AliasFor(annotation = RequestMapping.class)
    String[] value() default {};  // @AliasFor 把这里的值透传给 @RequestMapping.value
}

反例：全部用 @RequestMapping 冗长写法

能用但不推荐——每个方法都要写 value + method 两个属性，代码噪音大，且容易漏写 method。

UserController.java（反例 — 冗长且易漏配 method）
@Controller
@RequestMapping("/api/users")
public class UserController {

    @RequestMapping(value = "/{id}", method = RequestMethod.GET)
    @ResponseBody
    public User getById(@PathVariable Long id) { ... }

    // 危险：漏写 method，这个接口会同时响应 GET/POST/PUT/DELETE ...
    @RequestMapping("/{id}/delete")
    @ResponseBody
    public void delete(@PathVariable Long id) { ... }
}
安全隐患：@RequestMapping 不显式指定 method 时，会匹配所有 HTTP 方法。 上面的 delete 接口本意是走 DELETE，但漏写后一个 GET /api/users/1/delete 就能触发删除—— 浏览器预加载、爬虫、CSRF 攻击都可能顺手把它"点"掉。永远显式声明 HTTP 方法是这里的铁律。

正例：语义化注解 + 完整 CRUD 示例

用 @GetMapping 等语义化注解替代，代码即文档，一眼看出这个方法对应的 HTTP 语义。

UserController.java（正例 — 完整 CRUD）
@RestController
@RequestMapping("/api/users")  // 类级别前缀，所有方法路径都会拼接在它后面
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    // GET /api/users —— 查询列表
    @GetMapping
    public List<User> list() {
        return userService.findAll();
    }

    // GET /api/users/{id} —— 查询单个，{id} 是路径变量
    @GetMapping("/{id}")
    public User getById(@PathVariable Long id) {
        return userService.findById(id);
    }

    // POST /api/users —— 新增
    @PostMapping
    public User create(@RequestBody UserDTO dto) {
        return userService.create(dto);
    }

    // PUT /api/users/{id} —— 整体更新
    @PutMapping("/{id}")
    public User update(@PathVariable Long id, @RequestBody UserDTO dto) {
        return userService.update(id, dto);
    }

    // DELETE /api/users/{id} —— 删除，语义清晰，不会被误触发
    @DeleteMapping("/{id}")
    public void delete(@PathVariable Long id) {
        userService.delete(id);
    }
}

路径拼接规则很简单：最终 URL = 类级别 @RequestMapping 的路径 + 方法级别路径。 例如 getById 的完整路径是 /api/users + /{id} = /api/users/{id}。

陷阱：路径末尾斜杠

Spring Boot 3（Spring Framework 6）默认关闭了末尾斜杠的宽松匹配（useTrailingSlashMatch 从 true 改为 false）。 这意味着 @GetMapping("/users") 只会匹配 /api/users，不会自动匹配 /api/users/—— 这是从 Spring Boot 2 升级到 3 时最容易踩的坑之一，前端如果习惯性带了末尾斜杠，接口会直接 404。

@RequestMapping vs 语义化注解

写法示例优点缺点
@RequestMapping（原始） @RequestMapping(value="/x", method=RequestMethod.GET) 最灵活，可同时匹配多个 method / produces / consumes 条件 冗长；漏写 method 会匹配所有方法，存在安全隐患
语义化注解（推荐） @GetMapping("/x") 简洁；语义清晰即文档；不可能漏写 method 本质仍是 @RequestMapping，理解组合注解原理才不会"知其然不知其所以然"
面试高频："@GetMapping 和 @RequestMapping(method=GET) 有区别吗？"——运行时行为完全等价， 因为前者就是通过 @AliasFor 把属性转发给了后者；区别只在于代码可读性和是否容易漏配 method。

核心价值总结

  • 路由声明即注解：类级别 + 方法级别路径拼接，构成完整 URL，不再需要任何 XML 配置。
  • 语义化优先：@GetMapping 等组合注解是 @RequestMapping 的"语法糖"，生产代码优先用它们。
  • 永远显式声明 HTTP 方法：避免因匹配所有方法而产生的安全隐患，这是最容易被忽视的陷阱。
  • 注意版本差异：Spring Boot 3 默认不再宽松匹配末尾斜杠，升级时要重点排查。

与其他知识点的联系

  • 请求参数绑定（第 3 篇）：本篇只负责把 URL 和 HTTP 方法"路由"到某个方法上，方法参数具体怎么从请求里取值，是下一篇 @RequestParam/@PathVariable/@RequestBody 要解决的问题——路由是入口，参数绑定是入口之后的第一道工序。
  • @RestController 与 @ResponseBody（第 5 篇）：本篇示例里已经用了 @ResponseBody，但没有展开讲它对返回值的影响；等到第 5 篇会看到 @RestController 正是靠组合 @ResponseBody 这个元注解，才让 @GetMapping 等方法的返回值走向序列化而不是视图解析——两者都是"组合注解简化重复写法"的同一种设计思路。
  • 全局异常处理（第 7 篇）：本篇强调"漏写 method 会匹配所有 HTTP 方法"的安全隐患，这类因路由配置疏忽导致的异常（比如误触发了本不该被 GET 访问的删除逻辑）最终也是靠 @ExceptionHandler 统一兜底，避免堆栈信息直接暴露给客户端。
  • DispatcherServlet 请求处理流程（第 1 篇）：@RequestMapping 标注的方法是 HandlerMapping 扫描并注册进"路由表"的原始数据来源，理解组合注解的本质，才能理解上一篇讲的路由匹配是"匹配谁"。
  • HATEOAS 超媒体链接（第 8 篇）：本篇的路径拼接规则（类级别 + 方法级别）正是 WebMvcLinkBuilder.linkTo(methodOn(...)) 能够根据 Controller 方法反向生成 URL 的基础——没有规范的路由声明，就没法用类型安全的方式"反推"出链接。
