视图解析与 Thymeleaf

前几篇讲的都是"返回 JSON"的 REST 接口，这一篇回到传统套路：Controller 返回一个视图名，Spring 帮你渲染出完整的 HTML 页面。

一句话讲清楚

如果 Controller 方法没有标 @ResponseBody（也没用 @RestController）， 它的返回值会被当作视图名，而不是要写入响应体的数据。ViewResolver （在 Thymeleaf 场景下是 ThymeleafViewResolver）根据这个名字找到对应的模板文件， 结合 Controller 放进 Model 的数据，把模板渲染成一份完整的 HTML 字符串，最终写回浏览器。

和上一篇的联系：@RequestBody 讲的是"数据怎么进来"，这一篇讲的是传统场景下"页面怎么出去"—— 两者都是 DispatcherServlet 主干流程（第一篇）里的一环，只是分别对应了流程图里的不同节点。

示例：用户列表页面

UserPageController.java（返回视图名，不加 @ResponseBody）
@Controller  // 注意：这里是 @Controller，不是 @RestController
@RequestMapping("/users")
@RequiredArgsConstructor
public class UserPageController {

    private final UserService userService;

    @GetMapping
    public String listPage(Model model) {
        model.addAttribute("users", userService.findAll());
        return "user-list";  // 视图名，会被解析成 templates/user-list.html
    }
}
templates/user-list.html（Thymeleaf 模板）
<html xmlns:th="http://www.thymeleaf.org">
<body>
    <h1 th:text="'用户列表，共 ' + ${users.size()} + ' 人'"></h1>

    <!-- th:if 条件渲染：列表为空时显示提示 -->
    <p th:if="${users.isEmpty()}">暂无用户</p>

    <!-- th:each 循环渲染列表 -->
    <ul>
        <li th:each="user : ${users}">
            <!-- th:text 输出文本，自动做 HTML 转义防 XSS -->
            <span th:text="${user.name}"></span>

            <!-- th:href 动态拼接链接，支持 @{} URL 表达式 -->
            <a th:href="@{/users/{id}(id=${user.id})}">详情</a>
        </li>
    </ul>
</body>
</html>
Thymeleaf 的设计亮点：模板文件本身是合法的 HTML（th:* 是自定义属性， 不影响浏览器直接打开预览静态效果），这就是它主打的"自然模板（Natural Templates）"——设计师用浏览器直接打开看效果， 不需要启动后端服务，这一点是 JSP 做不到的。

渲染流程走一遍

① Controller 返回视图名字符串 —— 如 "user-list"，方法未标 @ResponseBody

② ViewResolver 定位模板
  ThymeleafViewResolver 按约定拼接路径：
  prefix(classpath:/templates/) + 视图名 + suffix(.html)
  → 找到 templates/user-list.html

③ Thymeleaf 引擎渲染
  把 Controller 放入 Model 的数据（如 users 列表）注入模板上下文
  逐个解析 th:text / th:each / th:if 等属性，替换成真实数据

④ 完整 HTML 字符串返回浏览器 —— 浏览器直接展示，无需再发额外请求拉数据

现代架构下的定位

在前后端分离架构里，前端（React/Vue）独立开发、独立部署，后端只需要通过 @RestController 吐出 JSON 数据，页面渲染完全交给浏览器端的 JS 完成——这已经是目前互联网产品的主流做法，下一篇就会讲 @RestController 与 @ResponseBody。

但 Thymeleaf 这类服务端渲染方式并没有过时：企业内部管理系统、后台运营页面等场景，往往不需要复杂的前端交互， 团队也未必配得起专职前端，服务端渲染开发链路短、部署简单、不需要额外的前端工程，仍然是很实用的选择。

维度服务端渲染（Thymeleaf）前后端分离（REST + React/Vue）
开发效率 单人可全栈搞定简单页面，链路短 前后端并行开发，但需要两套技能栈/两个角色
首屏速度 服务端直出完整 HTML，首屏快 需要先加载 JS 再发请求拉数据，首屏依赖优化（SSR/预渲染）
SEO 天然友好，搜索引擎直接抓到内容 纯客户端渲染对 SEO 不友好，需额外做 SSR
团队分工 前后端耦合在一个工程里，边界模糊 前后端通过 API 契约解耦，职责清晰，可独立迭代部署
适用场景 内部管理后台、传统企业系统 互联网 C 端产品、需要复杂交互的应用

核心价值总结

  • 返回值语义取决于注解：不加 @ResponseBody 时返回值是视图名，加了则是响应体数据本身，这是二者最本质的区别。
  • ViewResolver 按约定定位模板：理解 prefix + 视图名 + suffix 的拼接规则，出现 404 时优先检查模板路径是否匹配。
  • Thymeleaf 是自然模板：模板文件本身合法可预览，比 JSP 更贴近前端协作习惯。
  • 技术选型看场景：没有绝对优劣，内部系统看重开发效率选服务端渲染，用户量大、交互复杂的产品选前后端分离。
