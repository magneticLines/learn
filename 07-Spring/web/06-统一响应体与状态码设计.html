统一响应体与状态码设计

上一篇让 Controller 能返回 JSON 了，但如果每个接口返回的数据结构五花八门，前端会陷入"每个接口写一套处理逻辑"的噩梦。这一篇讲清楚如何设计一份让前端能"一招通吃"的响应结构。

一句话讲清楚

统一响应体的核心目的不是好看，而是让前端可以用一套通用的拦截器逻辑处理所有接口的返回结果——不管调用的是订单接口、用户接口还是商品接口， 前端只需要判断响应体里固定的 code 字段，就能决定走"成功分支"还是"失败分支"，不需要为每个接口单独写不同的判断逻辑。

类比：快递单号查询系统。不管你寄的是文件还是包裹，物流公司都用同一套"已揽收 / 运输中 / 已签收 / 异常"状态码体系。如果每个快递公司自己定义一套状态描述，你的快递查询 App 就要为每家公司单独适配，维护成本爆炸。

反例：每个接口自成一派

OrderController.java（反例 — 响应结构不统一）
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{id}")
    public OrderDTO getOrder(@PathVariable Long id) {
        return orderService.findById(id);
        // 裸露的业务对象，成功时长这样：{"id":1,"amount":99.9}
        // 那失败时呢？订单不存在应该返回什么？前端完全猜不到
    }

    @PostMapping
    public void createOrder(@RequestBody OrderDTO dto) {
        if (dto.getAmount() <= 0) {
            throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "金额非法");
            // 这里抛 400，报错信息格式又和上面 GET 接口不一样
        }
        orderService.create(dto);
        // 成功时啥也不返回，200 空响应体
    }
}

这段代码里，"成功"和"失败"分别长什么样，完全取决于写这个方法的程序员当天心情。前端要处理这个模块的接口，得先逐个读源码或者文档，猜每个接口失败时到底返回什么结构——多人协作的项目里，这种混乱会随着接口数量线性放大。

正例：统一响应包装类

设计一个泛型响应类 Result<T>，所有接口无论成功失败都返回这个结构，用静态工厂方法屏蔽构造细节：

Result.java（通用响应包装）
@Data
public class Result<T> {

    private int code;       // 业务状态码，0 或 200 代表成功，其余代表各类失败
    private String message; // 给人看的提示信息，失败时用于前端弹窗展示
    private T data;         // 真正的业务数据，失败时通常为 null

    private Result(int code, String message, T data) {
        this.code = code;
        this.message = message;
        this.data = data;
    }

    // 静态工厂方法：调用方不需要关心构造细节，语义一目了然
    public static <T> Result<T> success(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> fail(int code, String message) {
        return new Result<>(code, message, null);
    }
}
OrderController.java（正例 — 统一响应结构）
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{id}")
    public Result<OrderDTO> getOrder(@PathVariable Long id) {
        OrderDTO order = orderService.findById(id);
        return Result.success(order);
        // {"code":200,"message":"success","data":{"id":1,"amount":99.9}}
    }

    @PostMapping
    public Result<Void> createOrder(@RequestBody OrderDTO dto) {
        if (dto.getAmount() <= 0) {
            return Result.fail(4001, "金额非法");
            // {"code":4001,"message":"金额非法","data":null}
        }
        orderService.create(dto);
        return Result.success(null);
    }
}
前端拦截器（伪代码 — 一套逻辑处理所有接口）
// axios 响应拦截器，所有接口共用这一份逻辑
axios.interceptors.response.use(response => {
    const { code, message, data } = response.data;
    if (code === 200) { return data; }       // 成功：直接把 data 抛给业务代码
    message.error(message);                    // 失败：统一弹窗提示，业务代码无需关心
    return Promise.reject(message);
});

取舍讨论：HTTP 状态码 vs 自定义业务码

上面的例子里，HTTP 层永远返回 200，真正的成功/失败语义放在响应体的 code 字段里。这是业界一直存在争议的两种流派，没有绝对的对错，只有取舍：

流派做法优点缺点
严格 REST 语义 用标准 HTTP 状态码表达结果：200 成功、400 参数错误、404 资源不存在、500 服务端异常 符合 REST 规范；网关、监控系统、CDN 可以直接基于状态码做熔断/告警/缓存策略；对外开放 API 更专业 前端要针对不同状态码写不同分支处理；浏览器/部分 HTTP 客户端库对非 2xx 响应默认当成"错误"处理，需要额外配置才能拿到响应体里的详细信息
自定义业务码 HTTP 层永远 200，成功/失败靠响应体里的 code 字段区分 前端统一拦截器一套逻辑处理所有接口，业务代码里不需要关心 HTTP 细节；调试时 Network 面板不会出现一堆"报错"的红色请求，视觉上更清爽 对网关层、APM 监控不友好——监控系统只看 HTTP 状态码，会把大量业务失败误判为"全部成功"；不符合 REST 规范，语义上"越权"了 HTTP 协议本该表达的内容
实践建议：国内中后台/移动端项目里"自定义业务码 + HTTP 永远 200"更常见，因为前端团队话语权大、更看重开发效率； 对外开放的公有 API（尤其面向国际开发者）则更倾向遵循标准 HTTP 语义。面试时能说出这个取舍权衡，比死记硬背某一种"标准答案"更加分。 另外注意：即使采用自定义业务码，像 401（未登录）、403（无权限）这类横切关注点，很多项目仍然会用真实的 HTTP 状态码表达，因为网关层的鉴权拦截往往发生在业务代码之前，天然就该用标准状态码。

核心价值总结

  • 统一结构让前端可以复用一套处理逻辑：不用为每个接口单独写成功/失败判断代码。
  • 用静态工厂方法封装构造细节：Result.success() / Result.fail() 让 Controller 代码语义清晰，不暴露内部字段拼装过程。
  • HTTP 状态码 vs 业务码没有标准答案：要结合团队习惯、是否对外开放、监控体系成熟度综合判断，关键是团队内部统一并写进规范。
  • 这只是"手工返回 Result"的阶段：下一篇会看到，真正规模化的项目会把"异常自动转成 Result"这件事交给全局异常处理器统一完成，Controller 里可以直接抛异常。

与其他知识点的联系

  • @RestController 与 @ResponseBody（第 5 篇）：本篇的 Result<T> 能够顺利变成 JSON 返回给前端，前提是 Controller 方法的返回值走的是 @ResponseBody 那条路径——没有上一篇打好的"返回值 = 数据对象"这个基础，统一响应体就无处生根。
  • 全局异常处理（第 7 篇）：本篇末尾已经点破——手工在每个方法里 Result.fail(...) 只是过渡阶段，下一篇会把"异常自动转成 Result"这件事交给全局异常处理器，Controller 从此只管抛异常，不再关心怎么构造失败响应。
  • 参数校验错误处理（第 10 篇）：本篇设计的 Result 结构（code/message/data）正是第 10 篇里 Result.fail(400, "参数校验失败", errors) 直接复用的载体——校验失败信息最终也要塞进这同一套外壳，前端才能用一套拦截器逻辑统一处理成功和失败。
  • HATEOAS 超媒体链接（第 8 篇）：本篇讨论的"响应结构要不要统一"和第 8 篇讨论的"响应体里要不要带 _links"其实是同一类设计决策——都是在权衡"给前端更多约定信息"换来的开发便利，和"响应体膨胀、结构复杂"之间的取舍。
  • Jackson 序列化定制（第 11 篇）：Result 类里 data 字段可能包含日期、枚举等需要特殊格式化的内容，具体怎么让这些字段序列化成前端想要的样子，要靠第 11 篇讲的 @JsonFormat、全局 ObjectMapper 配置来实现。
