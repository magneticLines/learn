DispatcherServlet 请求处理流程

Spring MVC 的所有魔法都始于一个"前端控制器"——搞懂请求从进来到响应返回中间经过了哪些角色，是理解整个 Web 层的地基。

一句话讲清楚

DispatcherServlet 是 Spring MVC 里唯一的 Servlet 入口：所有 HTTP 请求先统一打到它这里， 再由它委托给一系列协作组件（HandlerMapping、HandlerAdapter、ViewResolver）完成"找方法 → 执行方法 → 渲染视图"的全过程。 你写的 @Controller 从来不会被 Servlet 容器直接调用，全部是 DispatcherServlet 在背后转发。

类比：公司前台总机（DispatcherServlet）接到所有外部电话，查花名册找到该转给哪个员工（HandlerMapping 匹配 Controller）， 再让对应部门的人来处理（HandlerAdapter 执行），最后把结果整理成一份文档回复对方（ViewResolver 渲染 / 直接写 JSON）。 你不会指望客户直接拨每个员工的分机——那样公司电话本会无限膨胀，换人还要通知所有客户。

反例：Servlet 时代"一个 URL 一个 Servlet"

在 Spring MVC 出现之前（或者不用框架的原生 Servlet 开发），每新增一个接口就要新建一个 Servlet 类，并在 web.xml 里手工登记映射关系。

web.xml（反例 — 每个接口一条配置，规模变大后难以维护）
<web-app>
    <servlet>
        <servlet-name>getUserServlet</servlet-name>
        <servlet-class>com.demo.GetUserServlet</servlet-class>
    </servlet>
    <servlet-mapping>
        <servlet-name>getUserServlet</servlet-name>
        <url-pattern>/api/users</url-pattern>
    </servlet-mapping>

    <!-- 每加一个接口，就要再复制一份 servlet + servlet-mapping -->
    <servlet>
        <servlet-name>createUserServlet</servlet-name>
        <servlet-class>com.demo.CreateUserServlet</servlet-class>
    </servlet>
    <servlet-mapping>
        <servlet-name>createUserServlet</servlet-name>
        <url-pattern>/api/users/create</url-pattern>
    </servlet-mapping>
</web-app>

这种写法的工程问题

  • 类爆炸：接口数量和 Servlet 类数量 1:1 增长，几百个接口就是几百个类和几百段 XML 配置。
  • 横切逻辑无处安放：登录校验、日志、异常处理这些每个接口都要做的事，只能在每个 Servlet 里重复写，或者手工套一层 Filter 链，非常笨重。
  • 没有统一的参数绑定 / 视图渲染机制：所有请求参数解析、类型转换、JSON 序列化都要自己在 doGet/doPost 里手写，重复劳动。
  • 路由能力弱：url-pattern 只支持简单通配符，做不到 @PathVariable 这种路径变量提取。

正例：中央调度层 —— DispatcherServlet

Spring MVC 反过来做：只注册一个 Servlet（DispatcherServlet），映射 / 或 /api/*， 所有请求都先经过它。它不自己处理业务，而是查询一份"路由表"（由 @RequestMapping 等注解构建）， 找到该由哪个 Controller 的哪个方法来处理，然后把请求"转交"过去。

这就是为什么新增一个接口只需要在 Controller 类里加一个方法、标一个注解，不需要碰任何配置文件—— 因为路由表是应用启动时通过扫描注解自动生成的，而不是手工登记的。

UserController.java（正例 — 只需声明方法，不用配置 Servlet）
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping
    public List<User> list() { ... }

    // 新增一个接口，只是多一个方法——DispatcherServlet 会自动把它纳入路由表
    @PostMapping
    public User create(@RequestBody User user) { ... }
}

一次请求的完整流程（原理走一遍）

① DispatcherServlet 接收请求
  Servlet 容器（Tomcat）把所有匹配 / 的请求交给唯一的 DispatcherServlet

② HandlerMapping 匹配 Controller 方法
  根据请求的 URL + HTTP 方法，在启动时构建好的路由表里查找
  找到对应的 HandlerMethod（Controller 实例 + 目标方法）
  同时找出该请求要经过哪些 拦截器（Interceptor）

③ HandlerAdapter 执行（含拦截器前置处理）
  依次执行 Interceptor.preHandle()（登录校验、日志埋点等横切逻辑）
  HandlerAdapter 负责解析方法参数（@RequestParam / @RequestBody 等）
  反射调用 Controller 方法，拿到返回值
  执行 Interceptor.postHandle()

④ 返回 ModelAndView 或直接写响应体
  若方法标了 @ResponseBody（或类标了 @RestController）→ HttpMessageConverter 直接把返回值序列化成 JSON 写入响应
  否则返回值被包装成 ModelAndView，进入下一步视图解析

⑤ ViewResolver 解析视图（仅当需要渲染页面时）
  根据视图名找到对应模板（如 Thymeleaf 模板文件）
  结合 Model 数据渲染成完整 HTML，写入响应

⑥ Interceptor.afterCompletion() 收尾 → 请求处理完成
关键认知：REST 接口（返回 JSON）走 ①②③④ 就结束了，根本不会经过 ⑤ 视图解析。 只有传统的服务端渲染页面（返回视图名字符串）才会走到 ViewResolver 这一步——这也是后面 @RestController 与 传统 @Controller 的本质区别。

Servlet 原生开发 vs Spring MVC

维度原生 ServletSpring MVC（DispatcherServlet）
路由注册方式 web.xml 手工配置，每接口一条 注解扫描自动生成路由表，零 XML
横切逻辑（登录/日志） 每个 Servlet 重复写，或叠 Filter 链 统一在 Interceptor / AOP 中处理一次
参数绑定 手动 request.getParameter() 逐个解析 HandlerAdapter 自动绑定到方法参数
JSON 序列化 手写 ObjectMapper 转换 HttpMessageConverter 自动完成
扩展点 基本没有，改动即改代码 HandlerInterceptor、HandlerMethodArgumentResolver 等丰富的 SPI
面试高频坑：不少人会误以为"每个 @Controller 都是一个 Servlet"。事实上整个应用生命周期内通常只有一个 DispatcherServlet 实例， 所有 Controller 只是被它管理的普通 Spring Bean，本身并不实现 Servlet 接口。

核心价值总结

  • 统一入口：所有请求收敛到一个 DispatcherServlet，路由逻辑集中管理，不再散落在配置文件里。
  • 职责分离：路由匹配（HandlerMapping）、方法执行（HandlerAdapter）、视图渲染（ViewResolver）各司其职，可以独立扩展替换。
  • 横切能力插拔：Interceptor 提供统一的请求前后置钩子，登录校验、日志、限流等横切逻辑不再侵入业务代码。
  • 为后续注解打基础：理解了这条主干流程，再学 @RequestMapping、参数绑定、视图解析时，就知道它们分别插在流程的哪一环。
