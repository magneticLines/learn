Bean Validation 数据校验（@Valid / @NotNull 等）

参数校验是 Controller 里最容易写成"一坨 if"的地方——用声明式注解把校验规则钉在 DTO 字段上，让校验规则和数据定义长在一起，而不是散落在各个方法体里。

一句话讲清楚

Bean Validation（JSR 380，2.0 版本）是 Java 生态的标准校验规范，Hibernate Validator 是它最主流的实现。 核心思路是：把校验规则以注解的形式声明在 DTO 字段上（这个字段"应该长什么样"），再由 Spring 在参数绑定完成后自动触发校验， 而不是在每个业务方法里手写一堆 if 判断。校验规则和数据结构定义在同一个地方，天然不会脱节。

类比：海关报关单上直接印着"此栏必须填写""此栏为数字格式"的提示文字（声明式规则），而不是让每个海关工作人员凭经验去检查、去记哪张单子该查哪几项（命令式判断）。规则印在单据上，谁来审核都不会漏查。

反例：Controller 里手写校验判断

没有 Bean Validation 之前，校验逻辑往往是这样写的：

UserController.java（反例 — 命令式校验）
@RestController
@RequestMapping("/api/users")
public class UserController {

    @PostMapping
    public Result<Void> createUser(@RequestBody UserCreateDTO dto) {
        // 每个字段手写判断，分散、重复、容易漏
        if (dto.getUsername() == null || dto.getUsername().isBlank()) {
            throw new IllegalArgumentException("用户名不能为空");
        }
        if (dto.getUsername().length() > 20) {
            throw new IllegalArgumentException("用户名长度不能超过 20");
        }
        if (dto.getEmail() == null || !dto.getEmail().matches("^[\\w.-]+@[\\w.-]+\\.\\w+$")) {
            throw new IllegalArgumentException("邮箱格式不正确");
        }
        if (dto.getAge() == null || dto.getAge() < 0 || dto.getAge() > 150) {
            throw new IllegalArgumentException("年龄不合法");
        }
        // ... 业务逻辑真正开始，前面一大段全是校验噪音
        return Result.success();
    }
}

这样写的问题

  • 分散重复：如果十个接口都要接收 UserCreateDTO，这堆判断要在十个方法里各写一遍。
  • 容易遗漏：新增一个字段时，很容易忘记同步加校验，等到线上出脏数据才发现。
  • 规则和数据脱节：校验规则藏在方法体里，看 DTO 定义完全看不出哪些字段是必填、有什么约束。
  • 异常类型随意：各人抛出的异常类型、错误信息格式不统一，前端难以统一处理。

正例：声明式注解 + @Valid

先在 pom.xml 里加上校验依赖——注意：Spring Boot 3 的 spring-boot-starter-web 默认不再自带 Bean Validation 依赖，必须单独引入：

pom.xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>

把校验规则声明在 DTO 字段上（Spring Boot 3 统一使用 jakarta.validation.constraints 包，而不是旧版的 javax.validation）：

UserCreateDTO.java（正例 — 声明式校验）
import jakarta.validation.constraints.*;

public class UserCreateDTO {

    @NotBlank(message = "用户名不能为空")
    @Size(max = 20, message = "用户名长度不能超过 20")
    private String username;

    @NotBlank(message = "邮箱不能为空")
    @Email(message = "邮箱格式不正确")
    private String email;

    @NotNull(message = "年龄不能为空")
    @Min(value = 0, message = "年龄不能小于 0")
    @Max(value = 150, message = "年龄不能大于 150")
    private Integer age;

    // getter / setter 省略
}

Controller 方法只需在参数前加一个 @Valid，一行判断都不用写：

UserController.java（正例 — @Valid 触发校验）
@RestController
@RequestMapping("/api/users")
public class UserController {

    @PostMapping
    public Result<Void> createUser(@Valid @RequestBody UserCreateDTO dto) {
        // 走到这里，dto 已经通过所有校验，业务代码干干净净
        return Result.success();
    }
}
校验规则和字段定义写在一起，看一眼 DTO 就知道每个字段的约束是什么，不需要翻遍 Controller 方法体去找隐藏的判断逻辑。

@Valid 是怎么触发校验的？（原理走一遍）

当 Controller 方法参数同时标注 @RequestBody 和 @Valid 时，Spring MVC 内部大致经历以下流程：

① 参数解析
  RequestResponseBodyMethodProcessor 识别到 @RequestBody 参数
  用 HttpMessageConverter（如 Jackson）把请求体 JSON 反序列化成 DTO 对象

② 校验触发
  发现参数上还标了 @Valid / @Validated
  调用 Validator（Hibernate Validator 实现）对刚绑定好的 DTO 对象执行校验
  逐个字段读取 @NotBlank / @Size 等约束注解并校验

③ 结果处理
  校验全部通过 → 正常进入 Controller 方法体
  校验有失败项 → 抛出 MethodArgumentNotValidException，方法体根本不会被执行
关键点：校验发生在"JSON 绑定完成之后、方法体执行之前"。也就是说，只要 @Valid 校验不通过，你的业务代码一行都不会被执行——这也是为什么校验失败的默认响应看起来是框架层直接抛出来的（下一篇会讲怎么把它接管成友好的错误信息）。

进阶：分组校验（同一个 DTO，不同场景不同规则）

典型场景：UserDTO 在"新增"时 id 必须为空（由数据库自动生成），在"更新"时 id 必须非空（要知道改哪一条）。 如果新增和更新共用一个 DTO，普通的 @NotNull 没法表达"这个字段的必填性取决于场景"——这时就需要分组校验。

分组校验示例
// 1. 定义两个空接口作为分组标识
public interface CreateGroup {}
public interface UpdateGroup {}

// 2. 在字段上通过 groups 指定这条规则属于哪个分组
public class UserDTO {

    @Null(groups = CreateGroup.class, message = "新增时不能指定 id")
    @NotNull(groups = UpdateGroup.class, message = "更新时必须指定 id")
    private Long id;

    // 不指定 groups 的规则，在任何分组下都会生效（默认分组）
    @NotBlank(message = "用户名不能为空")
    private String username;
}

// 3. Controller 方法上用 @Validated 指定本次要按哪个分组校验
@PostMapping
public Result<Void> create(@Validated(CreateGroup.class) @RequestBody UserDTO dto) { ... }

@PutMapping
public Result<Void> update(@Validated(UpdateGroup.class) @RequestBody UserDTO dto) { ... }
分组校验能用的前提是方法上用的是 @Validated 而不是 @Valid——这正是下一节要辨析的两者区别。

@Valid vs @Validated

维度@Valid@Validated
所属规范 JSR 380 标准注解（jakarta.validation） Spring 提供的增强注解（org.springframework.validation.annotation）
分组校验 不支持 支持，@Validated(XxxGroup.class)
使用位置 方法参数、字段、构造器参数（标准范围更广，但功能更基础） 方法参数；还能标在类级别，开启该类所有方法的参数校验（配合方法参数上的约束注解，常用于校验普通方法参数而非只有 @RequestBody）
嵌套对象校验 配合 @Valid 标在嵌套字段上可以级联校验内部对象 同样支持，用法一致
典型场景 不需要分组，简单直接的场景 需要按场景区分校验规则（新增 / 更新）时的首选
面试高频："@Valid 和 @Validated 有什么区别？"——一句话记忆：@Valid 是 Java 标准、功能基础；@Validated 是 Spring 增强、支持分组，还能开在类上做方法级校验。不涉及分组时两者几乎等价，实际项目里很多团队干脆统一用 @Validated 图省心。

核心价值总结

  • 声明式：校验规则和 DTO 字段定义长在一起，看一眼字段就知道约束是什么，不会脱节。
  • 零重复代码：规则只写一次，任何接收该 DTO 的接口都自动生效，不需要每个方法重复判断。
  • 可组合：分组校验让同一个 DTO 在不同业务场景下应用不同规则，避免为新增/更新各写一个几乎重复的 DTO。
  • 为统一错误处理铺路：校验失败统一抛出 MethodArgumentNotValidException，下一篇会讲如何用全局异常处理器把它转成前端友好的响应。

与其他知识点的联系

  • 请求参数绑定（第 3 篇）：本篇讲的 @Valid 校验发生在 @RequestBody 反序列化完成之后、方法体执行之前——第 3 篇讲的是数据怎么"进来"变成 DTO 对象，本篇讲的是这个 DTO 对象进来之后要先经过一道质检关卡，两者在参数解析流程里前后相接。
  • 全局异常处理（第 7 篇）：校验失败抛出的 MethodArgumentNotValidException 本质上也是一种异常，完全可以复用第 7 篇搭好的 @RestControllerAdvice 基础设施来统一处理——本篇末尾也提到了这一点，校验和异常处理共享同一套"关注点分离"的设计思路。
  • 参数校验错误处理（第 10 篇）：本篇只负责"拦下不合法的数据"，但拦下之后返回给前端的默认响应很不友好（暴露堆栈、没有结构化字段信息）——这正是下一篇要解决的问题，是本篇校验能力的直接延伸和收尾。
  • 统一响应体与状态码设计（第 6 篇）：校验失败最终也要被包装进第 6 篇设计的 Result 结构里返回，否则前端还是要为校验失败单独写一套解析逻辑，违背了统一响应体的初衷。
  • Jackson 序列化定制（第 11 篇）：两者都是"用注解在字段上声明规则、让框架自动处理"的同一种声明式设计哲学——@NotBlank/@Email 声明的是校验规则，@JsonFormat/@JsonIgnore 声明的是序列化规则，都不需要在业务代码里手写 if 判断或转换逻辑。
