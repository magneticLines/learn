全局异常处理（@ExceptionHandler / @ControllerAdvice）

上一篇统一了成功/失败的响应结构，但如果每个 Controller 方法都要手写 try-catch 去构造失败的 Result，代码依然又臭又长。这一篇讲清楚如何让异常处理"自动化"。

一句话讲清楚

@ControllerAdvice（Web 场景通常用它的子集 @RestControllerAdvice = @ControllerAdvice + @ResponseBody） 是一个"全局拦截层"，配合 @ExceptionHandler 可以把散落在各个 Controller 里的异常处理逻辑，集中收敛到一个类里统一管理。 Controller 方法本身不再需要 try-catch，只管抛异常，剩下的事情交给全局处理器。

反例：每个方法都写 try-catch

OrderController.java（反例 — 异常处理散落各处）
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{id}")
    public Result<OrderDTO> getOrder(@PathVariable Long id) {
        try {
            return Result.success(orderService.findById(id));
        } catch (OrderNotFoundException e) {
            return Result.fail(4041, e.getMessage());
        } catch (Exception e) {
            return Result.fail(500, "服务器开小差了");
        }
    }

    @PostMapping
    public Result<Void> createOrder(@RequestBody OrderDTO dto) {
        try {
            // 又要重复一遍一模一样的 try-catch 结构……
            orderService.create(dto);
            return Result.success(null);
        } catch (InsufficientStockException e) {
            return Result.fail(4002, e.getMessage());
        } catch (Exception e) {
            return Result.fail(500, "服务器开小差了");
        }
    }
}
  • 代码重复：兜底的 catch (Exception e) 逻辑在每个方法里都要抄一遍。
  • 容易遗漏：新加一个方法时，程序员一忙忘了写 try-catch，异常就直接抛到用户面前，暴露堆栈信息（安全隐患）。
  • 修改成本高：某天想给所有失败响应加一个 traceId 字段方便排查问题，得把每个方法的 catch 块全部改一遍。

正例：全局异常处理器集中兜底

先定义一个业务异常类，带上错误码字段，语义上区别于系统级异常（如 NPE、数据库连接失败）：

BusinessException.java（自定义业务异常）
@Getter
public class BusinessException extends RuntimeException {

    private final int code;  // 携带业务错误码，抛到全局处理器时直接拿来用

    public BusinessException(int code, String message) {
        super(message);       // message 交给父类 Throwable 存储，getMessage() 可直接取
        this.code = code;
    }
}
GlobalExceptionHandler.java（全局异常处理器）
@Slf4j
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody，处理结果直接序列化成 JSON
public class GlobalExceptionHandler {

    // 精确匹配：专门处理业务异常，错误码和信息都来自异常本身
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        log.warn("业务异常: {}", e.getMessage());
        return Result.fail(e.getCode(), e.getMessage());
    }

    // 精确匹配：参数校验失败（配合 Bean Validation，下一篇会展开）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidException(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldError().getDefaultMessage();
        return Result.fail(400, msg);
    }

    // 兜底匹配：其余所有未被上面精确捕获的异常，防止堆栈信息泄露给用户
    @ExceptionHandler(Exception.class)
    public Result<Void> handleUnknown(Exception e) {
        log.error("未捕获的系统异常", e);  // 系统异常必须打全量堆栈，方便排查
        return Result.fail(500, "服务器开小差了");  // 但给用户的提示要脱敏，不能暴露内部细节
    }
}
OrderController.java（正例 — Controller 直接抛异常，不再 try-catch）
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @GetMapping("/{id}")
    public Result<OrderDTO> getOrder(@PathVariable Long id) {
        // 找不到订单就直接抛异常，Controller 方法本身干净得像"正常流程代码"
        return Result.success(orderService.findById(id));
    }
}
OrderService.java（业务层照常抛异常，无需关心怎么响应给前端）
public OrderDTO findById(Long id) {
    return orderRepo.findById(id)
            .orElseThrow(() -> new BusinessException(4041, "订单不存在: " + id));
}

原理走一遍：Spring 如何选中最匹配的处理器方法

① Controller 方法执行时抛出异常
  例如 OrderService 抛出 BusinessException

② DispatcherServlet 捕获异常，交给 HandlerExceptionResolver 处理链
  其中 ExceptionHandlerExceptionResolver 会去找所有 @RestControllerAdvice Bean

③ 按异常类型的继承层级匹配最精确的 @ExceptionHandler
  BusinessException → 精确匹配 handleBusinessException()
  NullPointerException（没有专门处理方法）→ 沿继承链向上找，最终匹配 handleUnknown(Exception e)
  → "最具体优先"原则，和 Java 方法重载解析的直觉一致

④ 处理器方法返回值当作响应体（因为类上有 @ResponseBody）
  序列化成 JSON 写回，前端收到的依然是统一的 Result 结构

生产环境常见坑

陷阱：@ExceptionHandler 只能捕获Controller 方法执行过程中抛出的异常，也就是请求已经进入 DispatcherServlet 分发链路之后发生的异常。 但 Filter（过滤器） 在 DispatcherServlet 之前执行，如果 Filter 里抛异常（比如自定义的 Token 校验 Filter 解析 JWT 失败）， 这个异常压根不会进入 Spring MVC 的异常处理链，@ExceptionHandler 完全捕获不到，最终会变成容器默认的错误页面或裸露的堆栈信息。

解决办法通常有两种：一是在 Filter 内部自己 try-catch，手动构造 Result 写入响应（response.getWriter().write(...)）；二是让 Filter 把异常信息存入 request 属性，转发给一个专门的错误处理 Controller 处理，Spring Boot 默认的 /error 端点就是这个思路的体现。

处理方式执行时机@ExceptionHandler 能否捕获
Filter 抛异常DispatcherServlet 之前不能，需要 Filter 自行处理
拦截器 preHandle 抛异常DispatcherServlet 内部，Controller 方法执行前能，因为已进入 MVC 异常处理链
Controller / Service 方法抛异常DispatcherServlet 内部，请求处理过程中能，最常见的场景
异步线程池内抛异常脱离当前请求线程不能，需要用 @Async 的异常处理器或线程内自行捕获
面试高频："@ControllerAdvice 和 @ExceptionHandler 的关系？"——@ControllerAdvice 标注一个类，声明"这是一个全局增强器"； @ExceptionHandler 标注类里的具体方法，声明"这个方法专门处理某种类型的异常"。两者配合才能生效，@ExceptionHandler 也可以单独写在某个 Controller 内部，只对该 Controller 生效（局部异常处理）。

核心价值总结

  • 关注点分离：Controller/Service 只管抛异常表达"发生了什么"，全局处理器统一决定"怎么把异常翻译成用户能看懂的响应"。
  • 集中管理，一处修改全局生效：想统一加日志、加 traceId、脱敏错误信息，只需要改 GlobalExceptionHandler 一个地方。
  • 按异常类型精确路由：业务异常和系统异常可以给出完全不同粒度的响应和日志级别。
  • 覆盖范围有边界：Filter、异步线程等 DispatcherServlet 之外的异常需要单独处理，不能指望 @ExceptionHandler 包打天下。

与其他知识点的联系

  • 统一响应体与状态码设计（第 6 篇）：本篇的 handleBusinessException/handleUnknown 最终都是调用第 6 篇定义的 Result.fail(...) 来构造响应——全局异常处理器不是重新发明一套响应格式，而是把"手工调用 Result.fail"这件事从业务代码里搬到了统一的拦截层。
  • Bean Validation 数据校验（第 9 篇）：本篇示例里已经预留了 MethodArgumentNotValidException 的处理方法，这正是第 9 篇 @Valid 校验失败时抛出的异常类型——本篇是"接口"，第 9 篇才是真正"产生"这个异常的源头。
  • 参数校验错误处理（第 10 篇）：下一篇会在本篇搭好的 @RestControllerAdvice 基础设施上，把 handleValidException 从"提取第一个字段错误"升级成"遍历所有字段错误组装结构化列表"——是对本篇雏形实现的直接深化，而不是另起一套机制。
  • HATEOAS 超媒体链接（第 8 篇）：本篇强调"@ExceptionHandler 覆盖范围有边界"，这种"客户端需要提前了解系统行为边界"的问题，恰好是 HATEOAS 想通过超媒体链接去缓解的——如果响应能动态告诉客户端"当前状态下可以做什么"，某些因误操作触发的业务异常本可以从源头上被避免。
  • 请求参数绑定（第 3 篇）：本篇提到"Controller/Service 方法抛异常"是最常见能被捕获的场景，而第 3 篇提到的 MethodArgumentTypeMismatchException（参数类型转换失败）正是这类异常里最基础的一种，同样要靠全局异常处理器兜底，避免类型转换失败直接暴露堆栈给用户。
