参数校验错误处理

上一篇让 @Valid 帮我们自动拦下了不合法的数据，但它抛出的异常长什么样？前端能直接用吗？这一篇把校验失败对接到第 7 篇讲过的全局异常处理器上，产出真正对前端友好的错误响应。

问题：校验失败的默认响应很不友好

上一篇讲到，@Valid 校验不通过时 Spring 会抛出 MethodArgumentNotValidException。 如果项目里没有专门处理这个异常，Spring Boot 的默认异常处理机制会接管它，返回类似下面这样的响应：

默认错误响应（未处理时）
{
  "timestamp": "2026-07-03T02:00:00.000+00:00",
  "status": 400,
  "error": "Bad Request",
  "trace": "org.springframework.web.bind.MethodArgumentNotValidException: ...(一大段内部堆栈)...",
  "message": "Validation failed for object='userCreateDTO'. Error count: 2",
  "path": "/api/users"
}

这个响应对前端来说几乎没法用

  • 暴露内部实现：完整的异常堆栈（trace 字段）直接暴露给客户端，是安全隐患。
  • 没有结构化的字段级信息：前端要在表单里把"用户名不能为空"标红显示在用户名输入框下面，但响应里根本没有"哪个字段、什么错误"这种结构化数据，只有一句笼统的英文提示。
  • 响应格式和项目其他接口不一致：正常接口用的是第 6 篇讲过的统一响应体 Result，而这个错误响应完全是另一套格式，前端要为它单独写一套解析逻辑。

正例：用全局异常处理器接管校验异常

思路很直接：复用第 7 篇搭好的 @RestControllerAdvice 全局异常处理器，专门给 MethodArgumentNotValidException 加一个 @ExceptionHandler，把异常里的字段错误信息提取出来，用 Result 统一包装后返回。

GlobalExceptionHandler.java（在第 7 篇基础上新增一个处理方法）
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 第 7 篇已有：处理自定义业务异常
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException ex) {
        return Result.fail(ex.getCode(), ex.getMessage());
    }

    // 新增：专门处理 @Valid / @RequestBody 校验失败
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<List<FieldErrorVO>> handleValidException(MethodArgumentNotValidException ex) {
        // BindingResult 里装着这次校验所有失败的字段信息
        List<FieldErrorVO> errors = ex.getBindingResult()
                .getFieldErrors()   // List<FieldError>，每个元素对应一个校验失败的字段
                .stream()
                .map(fieldError -> new FieldErrorVO(
                        fieldError.getField(),            // 出错的字段名，如 "email"
                        fieldError.getDefaultMessage() // 注解上 message 属性的内容
                ))
                .toList();

        return Result.fail(400, "参数校验失败", errors);
    }
}

// 字段错误信息载体，专门给前端表单定位错误用
public record FieldErrorVO(String field, String message) {}

接管之后，同样的校验失败场景，前端拿到的响应变成了这样——和项目里其他接口用同一套 Result 外壳，还带上了结构化的字段级信息：

处理后的错误响应
{
  "code": 400,
  "message": "参数校验失败",
  "data": [
    { "field": "username", "message": "用户名不能为空" },
    { "field": "email",    "message": "邮箱格式不正确" }
  ]
}

前端拿到 data 数组后，遍历一下就能把每条错误精确标注到对应的输入框下面——这才是校验失败信息真正该有的样子。

整体流程走一遍

① 请求到达 → DTO 绑定 → @Valid 触发校验
② 校验失败 → 抛出 MethodArgumentNotValidException
③ 全局异常处理器拦截 → @RestControllerAdvice 里匹配到对应 @ExceptionHandler
  遍历 BindingResult.getFieldErrors()，组装 FieldErrorVO 列表
④ 统一响应体包装 → Result.fail(400, "参数校验失败", errors)
⑤ 前端收到结构化 JSON → 按 field 精确定位到表单控件展示 message

生产环境常见坑

坑 1：忘记自定义 message，返回一堆英文默认提示

如果注解不写 message 属性，比如只写 @NotBlank 而不是 @NotBlank(message = "用户名不能为空")， getDefaultMessage() 拿到的会是 Hibernate Validator 内置的英文默认提示，例如 "must not be blank"—— 这种提示直接展示给中文用户体验很差。凡是校验注解，都应该显式指定 message。

坑 2：BindingResult 参数 vs 全局异常处理，两种处理方式的取舍

还有一种更"手动"的处理方式：在 Controller 方法参数里，紧跟在 @Valid 校验对象后面直接声明一个 BindingResult 参数。 这样 Spring 不会抛出异常，而是把校验错误收集进这个 BindingResult，由你自己在方法体里手动判断并处理：

手动处理 BindingResult（对比写法）
@PostMapping
public Result<Void> createUser(@Valid @RequestBody UserCreateDTO dto,
                                BindingResult bindingResult) { // 必须紧跟在校验对象后面
    if (bindingResult.hasErrors()) {
        // 需要在这里手动组装错误信息并返回，异常根本不会被抛出
        String msg = bindingResult.getFieldError().getDefaultMessage();
        return Result.fail(400, msg);
    }
    return Result.success();
}
方式触发机制优点缺点
全局异常处理器
（推荐）		 校验失败自动抛异常，全局统一拦截 业务代码零侵入；所有接口错误格式统一；集中维护 需要提前搭好 @RestControllerAdvice 基础设施
BindingResult 手动处理 不抛异常，错误收集进参数对象 可以在方法体里做更灵活的定制判断（比如结合业务状态决定要不要继续走） 每个方法都要重复写 if 判断和错误处理逻辑，容易漏写；违背声明式校验的初衷
面试高频："@Valid 校验失败后怎么统一处理？"——标准答案就是本篇讲的这套：全局异常处理器捕获 MethodArgumentNotValidException， 遍历 BindingResult.getFieldErrors() 组装结构化响应。能进一步说出 BindingResult 手动模式作为对比，会显得对细节掌握更扎实。

核心价值总结

  • 统一出口：所有校验失败都走同一个全局异常处理器，响应格式和其他业务异常保持一致，前端只需一套解析逻辑。
  • 结构化：把字段名和错误信息拆开返回，前端可以精确地把错误定位到具体输入框，而不是弹一个笼统的提示框。
  • 业务代码零侵入：Controller 方法体里不需要任何 if 判断，校验和业务逻辑彻底解耦。
  • message 属性不能省：每个校验注解都应该显式写中文提示，否则用户看到的是 Hibernate Validator 内置的英文默认消息。
