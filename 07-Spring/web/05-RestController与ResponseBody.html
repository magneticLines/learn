Web 开发与 REST

@RestController 与 @ResponseBody

上一篇 Thymeleaf 渲染出的是给浏览器看的 HTML；这一篇开始，Controller 的返回值不再是页面，而是纯数据——这是前后端分离架构的第一块基石。

一句话讲清楚

@RestController = @Controller + @ResponseBody。 这个组合注解改变的不是"能不能处理请求"，而是方法返回值的去向：普通 @Controller 方法返回的字符串会被当成"视图名"，交给 ViewResolver 去找模板文件渲染成 HTML；而标了 @ResponseBody 之后，返回值不再被当成视图名，而是直接交给 HttpMessageConverter 序列化（通常是 JSON），原样写入 HTTP 响应体。

验证方式：打开 @RestController 源码就能看到答案——它上面同时标注了 @Controller 和 @ResponseBody 两个元注解：
RestController.java（Spring 源码节选）
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Controller          // 元注解一：具备 Controller 的能力（能被扫描、能处理请求映射）
@ResponseBody        // 元注解二：类里所有方法的返回值默认都走 HttpMessageConverter
public @interface RestController {
    ...
}

换句话说，@RestController 没有发明新能力，它只是把"@Controller + 每个方法都手动加 @ResponseBody"这个重复劳动，浓缩成一个注解。理解了这一点，下面这个陷阱就不难避免。

对比：同一个类，加不加 @ResponseBody

下面用同一个 UserController 演示两种写法的区别，方便直观感受"返回值去向"的分叉点。

UserPageController.java（传统 @Controller —— 返回视图名）
@Controller
@RequestMapping("/pages/users")
public class UserPageController {

    @GetMapping("/{id}")
    public String detail(@PathVariable Long id, Model model) {
        model.addAttribute("user", userService.findById(id));
        return "user/detail";   // 返回值 = 视图名，ViewResolver 去找 user/detail.html 渲染
    }
}
UserRestController.java（@RestController —— 返回 JSON 数据）
@RestController
@RequestMapping("/api/users")
public class UserRestController {

    @GetMapping("/{id}")
    public UserDTO detail(@PathVariable Long id) {
        return userService.findById(id);
        // 返回值 = 数据对象，HttpMessageConverter（默认 Jackson）序列化成 JSON 写入响应体
        // 响应示例：{"id":1,"name":"张三","email":"zhangsan@example.com"}
    }
}

两个方法长得几乎一样，唯一的区别是类上的注解。前者浏览器直接打开就能看到渲染好的网页；后者浏览器打开看到的是一坨 JSON 文本——这坨 JSON 交给谁去渲染成界面， 完全是前端（React / Vue 等）自己的事情，后端不再关心。

为什么这是前后端分离的基石

传统服务端渲染（上一篇 Thymeleaf）
  浏览器请求 → Controller 返回视图名 → ViewResolver 找模板 → Thymeleaf 渲染 HTML
  → 完整 HTML 页面直接返回浏览器展示
  问题：前端逻辑（模板）和后端逻辑（业务）耦合在同一个工程里，无法独立开发部署，也无法直接给 App / 小程序复用

REST 前后端分离（本篇）
  浏览器 / App / 小程序请求 → @RestController 返回数据对象 → HttpMessageConverter 序列化 JSON
  → 纯数据返回，前端框架各自决定如何渲染
  收益：同一套后端 API 可以同时服务 Web、App、小程序；前后端团队独立开发、独立部署、独立技术选型

后端只做两件事：提供数据和校验/执行业务规则。至于数据在页面上长什么样、用什么组件库、怎么做交互动画，全部交给前端自己决定。这也是为什么现代项目里后端团队和前端团队可以完全解耦排期、分开上线。

生产环境常见坑

陷阱：在 @RestController 标注的类里，如果某个方法想返回一个视图页面（比如想返回字符串 "redirect:/login" 或某个模板名）， 实际效果不会跳转或渲染模板，而是把这段字符串原封不动当成 JSON 字符串写进响应体——因为 @RestController 已经把"返回值走 HttpMessageConverter"这条规则焊死在类级别， 对类里的每一个方法都生效，没有"例外"这一说。

如果一个模块里既要有 REST 接口给前端调用，又要有几个传统页面视图（比如后台管理系统的登录页），正确做法是：

  • 拆分成两个 Controller：一个用 @RestController 专门处理 /api/**，一个用 @Controller 专门处理页面路由。这是最清晰、最推荐的做法。
  • 或者用 @Controller + 局部 @ResponseBody：在类上标 @Controller，只在需要返回 JSON 的个别方法上单独加 @ResponseBody，其余方法保持返回视图名。适合历史项目里页面和接口混杂、暂时不想大改的场景。
写法类注解方法返回值含义适用场景
纯页面 Controller@Controller视图名，走 ViewResolver传统服务端渲染、Thymeleaf 后台管理页
纯 REST Controller@RestController数据对象，走 HttpMessageConverter前后端分离 API，绝大多数新项目的默认选择
混合 Controller@Controller + 方法级 @ResponseBody按方法各自决定历史遗留项目页面/接口混杂，不建议新项目采用
面试高频："@RestController 和 @Controller 的区别？"——本质上不是能力差异，而是返回值处理方式的差异：@RestController 相当于给类里所有方法自动加上了 @ResponseBody。

核心价值总结

  • 本质是组合注解：@RestController = @Controller + @ResponseBody，理解元注解就理解了它的全部行为。
  • 返回值语义彻底不同：视图渲染返回"页面"，REST 返回"数据"，谁来渲染数据是前端的职责。
  • 是前后端分离的起点：同一套 API 可以给 Web、App、小程序等多端复用，团队可以独立开发部署。
  • 类级别的注解会对全部方法生效：不要在 @RestController 类里期望某个方法能返回视图。
