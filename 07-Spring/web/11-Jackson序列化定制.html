Jackson 序列化定制

Controller 返回一个 Java 对象，浏览器收到的却是一段 JSON 文本——这个"变身"过程完全没让你手写代码，都是 Jackson 在背后默默做的。这一篇聊聊怎么按需定制它的行为。

一句话讲清楚

Spring Boot 的 spring-boot-starter-web 会自动引入 spring-boot-starter-json， 底层依赖 Jackson（核心是 jackson-databind 库）。当 Controller 方法标了 @RestController 或返回值标了 @ResponseBody 时，Spring MVC 会用注册好的 MappingJackson2HttpMessageConverter 自动把返回的 Java 对象序列化成 JSON 字符串写入响应体——整个过程你不需要写一行手动转换代码，这就是"约定优于配置"在 Spring Boot 里的又一处体现。

反过来也一样：当 Controller 方法参数标了 @RequestBody 时，Jackson 负责把请求体的 JSON 反序列化成 Java 对象——序列化和反序列化是同一套机制的一体两面。

场景一：@JsonIgnore 忽略敏感字段

最常见的需求：数据库实体里有密码字段，绝不能原样序列化返回给前端。

User.java
public class User {

    private Long id;
    private String username;

    @JsonIgnore // 序列化时直接跳过这个字段，JSON 里根本不会出现 password
    private String password;

    // getter / setter 省略
}
坑：@JsonIgnore 标在实体类字段上会同时影响序列化和反序列化——如果前端注册时确实需要传密码给后端，反序列化也会被一并忽略。 更严谨的做法是分别用 @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)（只能反序列化写入，不能序列化输出）， 或者更推荐的方式：直接用一个不含 password 字段的 DTO 做返回值，从源头上避免实体类直接暴露给前端。

场景二：@JsonFormat 定制日期时间格式

Java 8 引入的 java.time 包（LocalDateTime / LocalDate 等，见 00-Java 基础模块）默认被 Jackson 序列化成数组或 ISO 格式， 不是前端习惯的 yyyy-MM-dd HH:mm:ss 这种可读格式，需要显式指定：

OrderVO.java
public class OrderVO {

    private Long id;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime createTime;

    // getter / setter 省略
}

加上这个注解后，createTime 序列化出来就是 "2026-07-03 10:30:00" 这样的字符串，前端可以直接展示，不需要再做格式转换。

场景三：@JsonProperty 重命名字段

Java 端习惯用驼峰命名（userName），但有的前端团队或第三方接口约定用下划线命名（user_name）。 不想为了适配命名风格而破坏 Java 代码规范，可以用 @JsonProperty 单独指定 JSON 字段名：

UserVO.java
public class UserVO {

    @JsonProperty("user_name") // Java 端仍用驼峰，JSON 输出/解析时用下划线
    private String userName;

    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY) // 只能反序列化写入，序列化时隐藏
    private String internalToken;

    // getter / setter 省略
}

反例：每个字段都重复标注 @JsonFormat

项目里实体类一多，日期字段往往几十上百个，如果每个都手写一遍 @JsonFormat，会出现两个问题：

反例 — 重复且格式不统一
public class OrderVO {
    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime createTime;
}

public class UserVO {
    // 另一个开发者写的，格式变成了 yyyy/MM/dd —— 团队风格不统一
    @JsonFormat(pattern = "yyyy/MM/dd HH:mm")
    private LocalDateTime registerTime;
}

// ... 项目里还有几十个类，每个都要重复写一遍，还容易写漏
  • 啰嗦重复：字段一多，注解写到手软，还容易漏标导致格式不一致。
  • 团队风格不统一：没有强约束的话，不同人写的日期格式五花八门，前端要为每个接口单独适配。

正例：全局配置 ObjectMapper，一处生效

更好的做法是统一在一个地方配置全局日期格式，具体字段不再需要重复标注。Spring Boot 推荐用 Jackson2ObjectMapperBuilderCustomizer：

JacksonConfig.java（正例 — 全局统一配置）
@Configuration
public class JacksonConfig {

    private static final String DATETIME_PATTERN = "yyyy-MM-dd HH:mm:ss";
    private static final String DATE_PATTERN     = "yyyy-MM-dd";

    // Spring Boot 会自动应用这个 Customizer 到全局唯一的 ObjectMapper 上
    @Bean
    public Jackson2ObjectMapperBuilderCustomizer jacksonCustomizer() {
        return builder -> {
            builder.simpleDateFormat(DATETIME_PATTERN);
            // 分别注册 LocalDateTime / LocalDate 的序列化与反序列化规则
            builder.serializers(
                    new LocalDateTimeSerializer(DateTimeFormatter.ofPattern(DATETIME_PATTERN)),
                    new LocalDateSerializer(DateTimeFormatter.ofPattern(DATE_PATTERN))
            );
            builder.deserializers(
                    new LocalDateTimeDeserializer(DateTimeFormatter.ofPattern(DATETIME_PATTERN)),
                    new LocalDateDeserializer(DateTimeFormatter.ofPattern(DATE_PATTERN))
            );
            builder.timeZone(TimeZone.getTimeZone("Asia/Shanghai")); // 顺手统一时区
        };
    }
}

配置好之后，全项目所有 LocalDateTime 字段（只要没有被单独的 @JsonFormat 覆盖）都自动按统一格式序列化， 不需要再逐个字段标注注解。字段级的 @JsonFormat 依然可以对个别字段做特殊覆盖，两者并不冲突——字段级注解优先级更高。

也可以直接自定义 @Bean public ObjectMapper objectMapper() { ... } 完全替换 Spring Boot 自动配置的 ObjectMapper， 但这样容易漏配一些 Spring Boot 默认已经处理好的细节（比如对 Optional、Bean Validation 结合场景的支持）。 官方更推荐用 Jackson2ObjectMapperBuilderCustomizer 这种"在默认配置基础上增量定制"的方式，风险更小。

字段级注解 vs 全局配置

维度字段级注解（@JsonFormat 等）全局 ObjectMapper 配置
生效范围 仅标注的这一个字段 项目内所有同类型字段（除非被字段级注解覆盖）
适用场景 个别字段有特殊需求，比如这一个日期字段就是要用不同于团队规范的格式 统一团队规范，比如"所有日期时间字段都用 yyyy-MM-dd HH:mm:ss"
维护成本 字段越多越啰嗦，容易漏标、写法不统一 一处配置，全局生效，新增字段自动遵循规范，无需额外操作
优先级 更高，会覆盖全局配置 作为兜底默认值
最佳实践：全局配置负责统一大多数字段的默认行为，字段级注解只用来处理少数确实需要"特殊对待"的例外情况——这和很多框架里"约定优于配置、异常再显式覆盖"的设计思路是一致的。

核心价值总结

  • 自动化：Spring Boot 已经把 Jackson 集成到位，Controller 返回对象就自动变成 JSON，不需要手写序列化代码。
  • 字段级注解处理个例：@JsonIgnore 隐藏敏感字段、@JsonFormat 定制单个字段格式、@JsonProperty 重命名字段。
  • 全局配置统一规范：用 Jackson2ObjectMapperBuilderCustomizer 统一日期格式、时区等团队级约定，避免每个字段重复标注、风格不统一。
  • 两者互补：字段级注解优先级更高，可以在全局默认行为之上对个别字段做精确覆盖。
