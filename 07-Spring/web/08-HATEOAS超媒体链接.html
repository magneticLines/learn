HATEOAS 超媒体链接（进阶）

前面几篇讲的 REST 接口，其实严格来说只是"看起来像 REST"。这一篇讲清楚真正完整的 REST 应该是什么样子，以及为什么现实中大家都没有做到那个程度。

一句话讲清楚

HATEOAS（Hypermedia As The Engine Of Application State，超媒体作为应用状态引擎）是 REST 架构风格里最容易被忽略、也最容易被误用的一环： 它要求接口返回的数据里不只包含"数据本身"，还要包含"当前状态下客户端接下来可以做什么操作"的链接，客户端不需要提前知道所有 API 路径，而是像浏览网页一样，点着链接一步步往下走。

REST 成熟度模型（Richardson Maturity Model）

这是业界公认用来衡量一个 API "有多 REST"的分级标准，从第 0 级到第 3 级，越往上越接近 REST 的原始设计理念：

第 0 级：单一 URI + POST（类似 RPC）
  所有操作都打到同一个地址，比如 POST /api/service
  请求体里用一个 action 字段区分"查订单"还是"取消订单"，本质是远程过程调用（RPC），完全没有资源概念

第 1 级：引入资源（Resources）
  不同的东西对应不同的 URI：/api/orders/1、/api/users/1
  但所有操作依然可能只用 POST，URI 只是"分类"，还没用上 HTTP 方法的语义

第 2 级：用 HTTP 方法表达操作语义
  GET /api/orders/1 查询、POST /api/orders 创建、PUT /api/orders/1 更新、DELETE /api/orders/1 删除
  → 绝大多数现实中的"REST API"止步于此，前面几篇讲的都是第 2 级

第 3 级：HATEOAS（超媒体驱动）
  响应体里附带"下一步能做什么"的链接，比如订单详情里带上"取消订单""查看物流"的链接
  客户端不需要硬编码这些 URL，而是从响应里动态发现可用操作，就像人浏览网页点链接一样

Spring HATEOAS 基本用法

Spring 官方提供 spring-hateoas 库，核心是用 EntityModel 包装资源对象，再用 WebMvcLinkBuilder 生成指向其他接口方法的链接：

pom.xml（引入依赖）
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-hateoas</artifactId>
</dependency>
OrderController.java（订单详情附带"下一步操作"链接）
import static org.springframework.hateoas.server.mvc.WebMvcLinkBuilder.*;

@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @GetMapping("/{id}")
    public EntityModel<OrderDTO> getOrder(@PathVariable Long id) {
        OrderDTO order = orderService.findById(id);

        // self：指向自己，客户端可以用这个链接重新拉取最新状态
        Link selfLink = linkTo(methodOn(OrderController.class).getOrder(id)).withSelfRel();

        // cancel：只有订单处于"待支付"状态时才附带这个链接，体现"当前状态下能做什么"
        EntityModel<OrderDTO> model = EntityModel.of(order, selfLink);
        if (order.getStatus() == OrderStatus.PENDING) {
            model.add(linkTo(methodOn(OrderController.class).cancelOrder(id))
                    .withRel("cancel"));
        }
        // track：物流链接，不管什么状态都可以查看
        model.add(linkTo(methodOn(LogisticsController.class).track(id))
                .withRel("track"));

        return model;
    }
}
响应示例（JSON + _links）
{
  "id": 1,
  "amount": 99.9,
  "status": "PENDING",
  "_links": {
    "self":   { "href": "/api/orders/1" },
    "cancel": { "href": "/api/orders/1/cancel" },
    "track":  { "href": "/api/logistics/1" }
  }
}

关键点：cancel 链接是条件性出现的——只有订单处于"待支付"状态才会附带。如果订单已经发货，响应体里根本不会有 cancel 这个字段，前端看到"没有这个链接"就知道当前状态不允许取消操作，不需要自己再写一套状态判断逻辑，也不需要担心用户在错误的状态下误触发操作。

四个成熟度级别对比

级别特征客户端需要知道什么现实采用度
Level 0单一 URI + POST，动作放在请求体所有 action 类型和参数格式，本质是 RPC老旧系统、内部 RPC 接口偶尔可见
Level 1引入资源 URI，但操作方式不规范每个资源的 URI较少见，通常是过渡阶段
Level 2URI + HTTP 方法表达语义每个资源的 URI + 该用哪个 HTTP 方法绝大多数"REST API"止步于此，业界事实标准
Level 3HATEOAS，响应体自带可执行操作链接只需要知道入口 URI，其余靠响应里的链接动态发现非常少，多见于学术讨论、少数开放平台（如 PayPal API 早期）
现实点评：不要误以为 HATEOAS 是"必须掌握的行业标准实践"。现实中绝大部分团队的前端工程师更喜欢直接翻 API 文档、把路径硬编码进代码里调用， 原因很实际：超媒体链接会显著增加响应体体积（每个资源都要带一堆 _links），前端要多写一层"从响应里解析链接再发请求"的逻辑， 团队内部协作时用文档对齐路径反而更简单直接、调试也更直观。真正大规模落地 HATEOAS 的开放平台案例并不多。

这部分内容的定位是：了解即可、面试可能会问到 REST 成熟度模型和 HATEOAS 的概念，但实际项目中很少需要真正引入 spring-hateoas 这个依赖。理解它的价值在于让你明白"REST"这个词在不同语境下含金量差异很大——面试官问"你的接口是 RESTful 的吗"，很可能只是想确认你做到了 Level 2，而不是真的要求 HATEOAS。

核心价值总结

  • REST 成熟度模型是一把度量尺：能帮你准确说清楚"我们的 API 做到了第几级"，而不是笼统地说"这是个 REST 接口"。
  • HATEOAS 的核心思想是"状态驱动可用操作":用响应体里链接的有无，替代客户端自己判断"当前状态能不能做某个操作"的逻辑。
  • Spring HATEOAS 提供 EntityModel + WebMvcLinkBuilder：用类型安全的方式（而不是拼字符串）生成指向 Controller 方法的链接。
  • 务实看待：这是一块面试可能触及、但实际项目投产比例很低的进阶知识，不必投入过多精力钻研落地细节。
