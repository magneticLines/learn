请求参数绑定（@RequestParam / @PathVariable / @RequestBody）

一次 HTTP 请求里的数据分散在 URL 查询串、路径本身、请求体三个地方——Spring 用三个注解分别对应，底层原理和一个致命的"只能读一次"陷阱要搞清楚。

一句话讲清楚

一个 HTTP 请求携带数据的方式无非三种：查询参数/表单参数（?name=tom&age=20）、 URL 路径变量（/users/123 里的 123）、请求体（POST/PUT 时携带的 JSON）。 Spring MVC 对应提供了三个注解：@RequestParam、@PathVariable、@RequestBody， 它们的本质都是告诉 HandlerAdapter："这个方法参数该从请求的哪个部位取值、怎么转换类型"。

三种绑定方式组合使用

OrderController.java（同一方法组合使用三种绑定）
@RestController
@RequestMapping("/api/users/{userId}/orders")
public class OrderController {

    // PUT /api/users/{userId}/orders/{orderId}?remark=xxx
    @PutMapping("/{orderId}")
    public Order updateOrder(
            @PathVariable Long userId,                          // 来自 URL 路径
            @PathVariable Long orderId,                         // 来自 URL 路径
            @RequestParam(required = false) String remark,  // 来自查询参数，可选
            @RequestBody OrderUpdateDTO dto) {              // 来自请求体 JSON

        return orderService.update(userId, orderId, remark, dto);
    }
}

请求长这样：PUT /api/users/1001/orders/88?remark=urgent，请求体是 {"status":"SHIPPED","address":"..."}。HandlerAdapter 会分别从三个不同的位置把值取出来， 转换成对应的方法参数类型后再调用方法——你完全不用手写任何解析代码。

@RequestBody 底层：HttpMessageConverter

请求体本质是一段字节流（InputStream），Spring 并不会自己写 JSON 解析逻辑，而是委托给一组 HttpMessageConverter。当请求头 Content-Type: application/json 时， 容器里注册的 MappingJackson2HttpMessageConverter 会被选中，它内部持有一个 Jackson 的 ObjectMapper，负责把输入流反序列化成目标 Java 对象。

① 请求到达 Content-Type: application/json，Body 是一段 JSON 字节流

② HandlerAdapter 发现方法参数标了 @RequestBody
  遍历已注册的 HttpMessageConverter 列表
  根据 Content-Type 找到能处理它的 Converter → MappingJackson2HttpMessageConverter

③ Converter 读取 InputStream，反序列化
  ObjectMapper.readValue(inputStream, TargetType.class)
  → 生成 Java 对象，赋值给方法参数

④ 方法体拿到的已经是强类型对象，业务代码无需关心序列化细节
反向同理：返回值如果标了 @ResponseBody（或类标了 @RestController）， 走的是同一组 Converter 的反向流程——把 Java 对象序列化成 JSON 写入响应体，用的还是 Jackson。

陷阱：@RequestBody 只能读一次

请求体是一个 InputStream，本质是一次性的数据流。Spring 用 @RequestBody 把它读取并反序列化后，这个流的读指针就已经到了末尾。如果你还想在过滤器 / 拦截器里打印原始请求体日志， 再让 @RequestBody 去读一次，第二次读到的会是空内容，导致反序列化失败或拿到空对象—— 这是很多人调试"为什么日志打印正常但接口收到的对象全是 null"时最容易忽略的原因。
ContentCachingRequestWrapper（正确的日志+绑定共存方案）
// 用可缓存的 Wrapper 包一层，请求体会先被缓存到内存里，可以重复读取
@Component
public class RequestLoggingFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) {
        // 包一层 Wrapper，请求体读取后会被缓存在内部字节数组中
        ContentCachingRequestWrapper wrapped =
                new ContentCachingRequestWrapper(request);

        chain.doFilter(wrapped, response);  // 后续 @RequestBody 正常读取，不受影响

        // 请求处理完后，从缓存里取出 body 字节数组用于日志打印
        String body = new String(wrapped.getContentAsByteArray());
        log(body);
    }
}

三种绑定方式对比

注解对应 HTTP 部分典型场景
@RequestParam URL 查询参数（?k=v）/ 表单参数 分页参数（page、size）、筛选条件、简单表单提交
@PathVariable URL 路径中的占位变量（/users/{id}） REST 资源定位，如资源 ID、层级路径
@RequestBody 请求体（Body） POST/PUT 提交结构化数据（JSON 对象、批量数组）
面试高频："GET 请求能用 @RequestBody 吗？"——技术上大多数 Servlet 容器允许 GET 带 Body， 但 HTTP 规范不建议这么做，很多网关 / 代理会直接丢弃 GET 请求的 Body。查询条件复杂就用 POST + @RequestBody， 不要指望 GET 携带请求体。

核心价值总结

  • 各司其职：三个注解精确对应 HTTP 请求的三个不同部位，不要混用错位（比如把路径变量硬塞进 @RequestParam）。
  • 序列化交给 Converter：@RequestBody 的反序列化由 HttpMessageConverter 统一管理，理解这一点才能定位"参数为什么绑定失败"这类问题。
  • 请求体只能读一次：需要同时打日志和绑定对象时，必须用 ContentCachingRequestWrapper 之类的方案缓存流内容。
  • 类型转换自动完成：字符串到 Long、枚举、日期等基本类型转换由 Spring 内置的 Converter/Formatter 完成，复杂类型转换失败会抛 MethodArgumentTypeMismatchException。
