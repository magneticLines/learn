存储引擎内核

Change Buffer

普通二级索引的写入天然是"随机 I/O 重灾区"——Change Buffer 用"先记账、后对账"的思路，把这部分随机写攒起来批量处理。

要解决的问题：二级索引的随机写代价

一张表通常有一个聚簇索引（主键索引，数据按主键顺序物理存放）和若干二级索引（辅助索引）。当我们按主键递增插入数据时，聚簇索引的写入是顺序的——新行大概率追加在最后一个页；但二级索引的键值往往不按插入顺序排列（比如按 name 建的索引，`name` 的字典序和插入顺序毫无关系），所以对二级索引的写入几乎是随机

随机写的代价在于：如果目标页当前不在 Buffer Pool 里，按照"先读后写"的常规流程，就必须先发起一次随机磁盘 I/O把整页读进内存，才能在内存里完成这次小小的修改——为了改几个字节，付出一次完整的随机 I/O，性价比很低，尤其是在写密集的场景下会显著拖慢吞吐。

解决方案：先记到 Change Buffer，稍后合并

InnoDB 的做法是：如果要修改的二级索引页不在 Buffer Pool 中，就不立即读盘，而是把这次修改（INSERT / UPDATE / DELETE 对应的变更）先记录到一块叫 Change Buffer 的内存结构里（它也会被持久化到系统表空间，防止宕机丢失）。等到将来这个页因为别的原因被读入 Buffer Pool时（比如一次查询命中了它），再把 Change Buffer 里堆积的多条变更一次性合并（merge）到这个页上。

合并的时机主要有三种：(1) 该页被正常的读请求带入 Buffer Pool 时顺带合并；(2) 后台有专门的合并线程定期扫描并合并；(3) MySQL 正常关闭（clean shutdown）时会执行一次合并。这样一来，原本"多次随机读+改"被压缩成了"攒批之后一次性合并"，大幅减少了随机 I/O 次数。

类比：Change Buffer 像是快递驿站的"代收登记簿"。快递员（写请求）不必每次都跑到你家楼上敲门确认（读盘），先在登记簿上记一笔"3 号楼 502 有一件包裹"；等你哪天下楼恰好经过驿站（页面被读入内存），驿站阿姨才把攒的好几笔记录一次性告诉你，你一并领走——比每来一件包裹就上楼跑一趟效率高得多。

为什么唯一索引和聚簇索引不能用 Change Buffer

Change Buffer 只对非唯一的二级索引生效，对唯一索引（含主键/聚簇索引）无效，原因在于唯一性约束的检查方式：

  • 唯一索引在插入或修改时必须保证"这个键值在索引里独一无二"，也就是说必须立刻读取目标页，检查是否已经存在相同键值才能判断这次写入是否合法。这个"读页校验"的动作是没法延后的——一旦延后，就可能出现两个并发事务都以为某个值不存在、都成功写入，造成重复数据。既然唯一性校验已经强制要求立即读盘，Change Buffer"延迟写入、避免读盘"的价值就荡然无存了。
  • 聚簇索引（主键索引）本身通常是顺序写入（自增主键场景下几乎总是追加到最后一页），随机 I/O 问题本来就不严重，用 Change Buffer 收益有限；而且聚簇索引存放的是整行数据，直接关系到主键唯一性检查，同样需要立即读页。
面试高频："为什么唯一索引不能使用 Change Buffer？" 标准答案：唯一索引写入前必须读取目标页判断是否违反唯一约束，这个校验无法推迟，只要读页就已经产生了随机 I/O，Change Buffer 想要"避免立即读页"的初衷就无法达成，因此只有非唯一二级索引才能享受这个优化。

相关参数

参数作用
innodb_change_buffer_max_size Change Buffer 最多可占用 Buffer Pool 的百分比，默认 25，最大可调到 50。设太大会挤占正常的数据缓存空间。
innodb_change_buffering 控制对哪些操作类型启用 Change Buffer，可选 all、inserts、deletes、changes、purges、none 等，默认 all。

适用场景对比

索引类型写入模式能否用 Change Buffer原因
非唯一二级索引 键值分布随机，写入常落在随机页 可以 无需立即校验，可延迟合并，收益最大
唯一二级索引 同样可能随机，但需要唯一性校验 不可以 必须立即读页判重，延迟写入的前提不成立
聚簇索引（主键） 自增场景下多为顺序写 不可以 本身随机 I/O 少，且承载唯一性校验

常见误区

误区："Change Buffer 能让所有写操作都变快。" 实际上它只对目标页不在 Buffer Pool 中的非唯一二级索引写入有效。如果实例内存充足、热数据都常驻在 Buffer Pool（例如整个数据库都是热点数据的中小型库），大部分写入本就命中内存，Change Buffer 的优化空间自然有限；反而在这种场景下 Change Buffer 占用的内存可能不如直接给 Buffer Pool 更划算。

核心价值总结

  • Change Buffer 的本质是"写合并"：把分散的随机写攒批，延迟到页面被读入内存时再合并，减少随机 I/O 次数。
  • 只对非唯一二级索引生效，唯一性校验的强约束决定了它无法适用于唯一索引和聚簇索引。
  • 与 Undo Log 不同，Change Buffer 优化的是"写入路径的 I/O 效率"，而不涉及事务回滚或多版本可见性，二者是完全独立的机制，不要混淆。
