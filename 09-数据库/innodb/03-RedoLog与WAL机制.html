存储引擎内核

Redo Log 与 WAL 机制

这是整个存储引擎内核里最高频的面试考点，也是理解 MySQL 为什么"又快又不丢数据"的关键。核心就八个字：先写日志，后写数据。

Redo Log 是什么：物理日志

Redo Log 记录的不是"SQL 语句"，也不是"这一行数据改成了什么样"，而是对物理页做了什么修改——例如"在 XX 表空间、第 XX 号页、偏移量 XX 处，写入了这几个字节"。 这种"记录页面物理变化"的日志类型称为物理日志（Physical Logging），与之相对的是 binlog 里 statement/row 格式所代表的逻辑日志。 正因为是物理日志，Redo Log 的恢复过程只需要"照着页地址和内容重放"，非常快，且和上层是什么 SQL、什么存储引擎逻辑无关。

WAL：Write-Ahead Logging 原理

如果没有 Redo Log，要保证事务提交后数据不丢，唯一办法是：每次提交都把修改过的数据页同步刷回磁盘。 但数据页在磁盘上的位置是随机的（一次更新可能涉及聚簇索引页、多个二级索引页，散落在文件各处），随机 I/O 极慢；而且一个页 16KB，即使只改了几个字节，也要整页刷盘，代价很不划算。

WAL（Write-Ahead Logging，预写日志）的思路是反过来：先把"修改了什么"顺序追加写到一个专门的日志文件里，数据页本身留在内存（Buffer Pool）里晚点再刷，甚至可以攒着多次修改一起刷。 这样做的关键收益：

  • 顺序写代替随机写：日志文件是追加写入，磁盘顺序 I/O 比随机 I/O 快一到两个数量级（机械盘尤其明显，即使 SSD 也有差距）。
  • 日志远小于数据页：一条 Redo Log 记录只有几十字节，而一个数据页是 16KB，写日志的数据量远小于写整页。
  • 脏页可以延迟、合并刷盘：同一个页可能被多次修改，只要日志保证了持久性，数据页什么时候刷、刷几次都不影响正确性，由后台线程根据负载择机处理（见 07 篇 Checkpoint）。
  • 支持组提交（Group Commit）：多个并发事务的 Redo Log 可以合并成一次 fsync 落盘，进一步摊薄磁盘同步的开销。
类比：WAL 好比餐厅后厨先把"客人点了什么菜"记在小票上（顺序写、飞快），菜不用立刻端上桌摆盘（数据页刷盘可以延后）。哪怕后厨突然断电，只要小票还在，照着小票重新做一遍就能恢复所有订单，不会有客人的菜丢单。如果反过来要求"每接一单就必须立刻把菜摆到桌上才算数"，翻台效率会低得多。

完整写入链路：Buffer → OS Cache → 磁盘

Redo Log 从产生到真正落盘，要经过三层，理解这三层是理解 innodb_flush_log_at_trx_commit 参数的前提：

1. redo log buffer（InnoDB 进程内存，随事务修改实时写入）
    │ write() 系统调用
2. OS page cache（操作系统内核缓冲区，write() 后即可见，但进程崩溃不丢，OS 崩溃/断电会丢）
    │ fsync() 系统调用
3. ib_logfile（磁盘上的 redo log 文件）（fsync 后才是真正落盘，任何故障都不会丢）

write() 只是把数据从进程内存拷贝到内核缓冲区，速度快但不保证掉电不丢；fsync() 才会强制内核把数据真正刷到磁盘介质，是真正意义上的持久化，但代价是一次磁盘同步 I/O 的时间开销。

innodb_flush_log_at_trx_commit：三种策略详解

这个参数决定"事务提交时，Redo Log 写到哪一层"，是性能和持久性之间的经典权衡，也是数据库方向的高频面试题：

取值提交时的行为持久性性能
1（默认） 每次事务提交，都执行 write() + fsync()，立即落盘 最高：只要提交返回成功，即使数据库进程崩溃、操作系统崩溃、服务器断电，事务都不会丢 最慢：每次提交都有一次磁盘同步开销（可靠组提交能部分缓解）
0 提交时不立即写，只由后台线程每秒执行一次 write() + fsync() 最低：数据库进程崩溃或服务器断电，最多丢失约 1 秒内已提交的事务 最快：提交本身几乎没有 I/O 等待
2 每次事务提交立即 write()（写入 OS page cache），但 fsync() 交给后台线程每秒执行一次 中等：只要操作系统不崩溃（仅 MySQL 进程崩溃），数据在 OS cache 里不会丢；但服务器断电/OS 崩溃仍可能丢失约 1 秒数据 较快：省掉了每次提交的 fsync 开销，只有一次系统调用

实践中：核心交易类系统会用 1，用磁盘 I/O 换绝对的持久性；可以容忍秒级数据丢失、追求极致写入吞吐的场景（如日志类、统计类系统）可能选 2 甚至 0。

Redo Log 文件是循环写的

Redo Log 不是无限增长的文件，而是由一组固定大小、数量的文件（如 ib_logfile0、ib_logfile1）构成的环形结构，总大小由 innodb_log_file_size × innodb_log_files_in_group 决定。写指针（write pos）不断向前推进，写到组末尾后会回到组头部覆盖写。 但覆盖的前提是：这部分日志对应的脏页已经真正刷到磁盘了，否则一旦覆盖、又发生崩溃，这部分修改就永久丢失、无法重做。 "已刷盘、可以被覆盖"这个边界正是由 Checkpoint 机制维护和推进的，具体如何计算和推进将在 07 篇展开。

常见误区

误区："只要开着 Redo Log，MySQL 就绝对不会丢数据，跟 innodb_flush_log_at_trx_commit 设成几没关系。" 这是最容易被面试官抓住的误解。Redo Log 提供的持久性保证，强弱完全取决于这个参数设成什么值：设为 1 才是"提交即落盘"的强持久性；设为 0 或 2 时，为了性能都主动放弃了一部分持久性保证，在极端故障场景下（进程崩溃、系统崩溃、断电）会有不同程度的数据丢失窗口。理解这一点，才能在实际项目里根据业务对"丢多少数据可接受"做出正确的权衡选择，而不是死记"开了 Redo Log 就绝对安全"。
面试高频："Redo Log 和 binlog 有什么区别？" 常见对比维度：Redo Log 是 InnoDB 引擎层的物理日志、固定大小循环写、用于崩溃恢复和持久性；binlog 是 MySQL Server 层的逻辑日志（statement/row/mixed）、可以无限追加、用于主从复制和数据归档。二者配合完成"两阶段提交（2PC）"，保证事务提交后 Redo Log 和 binlog 的内容一致，这也是后续可以延伸追问的方向。

与其他知识点的联系

  • Buffer Pool 原理（上一篇）：WAL 的核心价值之一就是"脏页可以延迟、合并刷盘"——这正是依赖 Buffer Pool 的 flush list 来追踪哪些页被修改过、还没落盘，Redo Log 保证了即使脏页迟迟不刷，崩溃后也能靠日志重放恢复，两篇讲的是同一套内存与磁盘协作机制的两个环节。
  • 页结构与记录存储格式（索引 Tab 01）：Redo Log 记录的是"在第 XX 号页、偏移量 XX 处写入了什么字节"，这种物理日志的粒度直接对应第一篇讲的页内布局——不理解页和行记录的物理结构，就无法理解 Redo Log 具体记录的是什么内容。
  • B+ 树索引的原理 / 聚簇索引与二级索引回表（索引 Tab 02-03）：一次更新操作往往会同时修改聚簇索引页和多个二级索引页，这些页在磁盘上物理位置分散——这正是本篇强调"数据页刷盘是随机 I/O、代价高"的原因，也是 WAL 用顺序写日志规避随机写数据页的直接动机。
  • 慢查询定位与优化流程（索引 Tab 11）：该篇陷阱一提到"索引越多，写入维护成本越高"——每多一个索引，一次写入就要多产生对应的 Redo Log 记录，这是索引数量影响写入性能在日志层面的具体体现，加索引的决策不能只看查询收益，也要考虑这部分日志和刷盘开销。
  • 主键设计（design Tab）：主键单调递增时，插入总是发生在聚簇索引最后一页，产生的是连续、可预测的页修改；而 UUID 类无序主键会导致随机位置的页修改（甚至页分裂），产生更零散的 Redo Log 记录和更多脏页，这是主键设计选型影响写入链路开销的另一个角度。

核心价值总结

  • WAL 的本质是把随机写转换成顺序写，用日志的"快"换数据页刷盘的"慢"，是数据库系统里普遍适用的设计思想（不止 InnoDB，几乎所有支持崩溃恢复的存储系统都用这一套）。
  • Redo Log 记录的是物理修改，配合 Checkpoint 机制实现空间循环复用，配合崩溃恢复机制（08 篇）把内存中未落盘的修改重放出来。
  • innodb_flush_log_at_trx_commit 是性能与持久性的调节旋钮，没有绝对正确的取值，只有适合业务场景的取值——这正是"给出对比、权衡取舍"思维在数据库调优中的具体体现。
