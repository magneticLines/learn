Checkpoint 机制

redo log 是一个大小固定的环形文件，它既不能无限写下去，也不能随意覆盖——checkpoint 就是在"写不下"和"覆盖了会丢数据"之间找平衡点的机制。

问题的起点：重启后要重放多久的日志？

回忆 上一篇讲的 WAL 机制：InnoDB 修改数据页时，先把"物理修改记录"顺序写进 redo log， 再由后台线程慢慢把脏页刷到磁盘。如果数据库突然宕机重启，就需要靠 redo log 把脏页丢失的修改重放（redo）回来。

但如果没有 checkpoint，重放该从哪里开始？唯一安全的答案是"从 redo log 文件最开头开始"——因为你不知道哪些修改已经落盘、哪些还没有。 对一个运行了几个月、写入量巨大的生产库来说，这意味着每次崩溃恢复都要把海量历史日志全部重放一遍，耗时可能是灾难性的。

类比：checkpoint 就像你写日记时定期夹的一张书签——"书签之前的内容我已经全部誊抄到正式笔记本上了（落盘），万一日记本丢了几页， 我只需要从书签开始的部分回忆重写，前面的不用管。"

核心概念：Checkpoint LSN 与 Flush List

InnoDB 用 LSN（Log Sequence Number，日志序列号）标记每一条 redo log 记录在整个日志流中的绝对位置，LSN 单调递增。 Checkpoint LSN 是一个承诺："所有 LSN 小于它的修改，对应的数据页都已经安全落盘"。

要知道当前哪些页还没落盘，InnoDB 在 Buffer Pool 中维护一个 flush list：把所有脏页按照它们"第一次被修改时对应的 LSN"从小到大排序。 flush list 表头永远是最老、修改时间最早的脏页——它对应的 LSN 就是决定 checkpoint 能推进到哪里的关键： checkpoint LSN 最多只能推进到 flush list 中最老脏页的 LSN，因为再往后就意味着"承诺已落盘"但实际还没有。

为什么是"模糊检查点"而不是"停机检查点"

最简单粗暴的 checkpoint 方式是：暂停所有写入，把 Buffer Pool 里所有脏页一次性刷完，再把 checkpoint LSN 推到最新——这叫 sharp checkpoint（尖锐检查点）。问题很明显：刷盘期间数据库完全不可写，对在线业务是不可接受的停顿。

InnoDB 采用的是 fuzzy checkpoint（模糊检查点）：后台线程持续、增量地刷一小批脏页，每刷完一批就把 checkpoint LSN 往前推进一点， 整个过程与正常的读写业务并发进行，不需要停机。这也是为什么 InnoDB 会在多种场景下触发局部刷脏，例如：

  • master thread 周期性刷脏：后台线程按一定比例定期刷一批最老的脏页；
  • 脏页比例过高时刷脏：Buffer Pool 中脏页占比超过阈值，主动多刷一些，避免脏页堆积到无法控制；
  • redo log 空间紧张时刷脏：这是本文最重要的一种，见下节。

Checkpoint 与 redo log 复用：谁在等谁

redo log 文件（ib_logfile0、ib_logfile1 ...）是大小固定的循环写文件：写到最后一个文件末尾后， 会绕回第一个文件开头继续写，覆盖旧内容。这就产生一条铁律：

核心规则：redo log 中 checkpoint LSN 之前的部分才允许被新日志覆盖。 因为 checkpoint 之前的修改已经保证落盘到数据页了，这段日志"任务已完成"，可以安全丢弃；而 checkpoint 之后的部分，是这些修改唯一还存在的记录 （数据页本身还没改完/没刷盘），一旦被覆盖，宕机后就再也无法恢复这部分修改，直接导致数据丢失。

由此可以推出一个容易被忽视的生产事故场景：如果写入速度远快于 checkpoint 推进速度（比如刷脏跟不上、或者一个大事务长时间不提交导致脏页无法刷出）， redo log 会写到"追上"checkpoint LSN 的位置——环形缓冲区绕了一圈又追上了自己的尾巴，再往前写就会覆盖还未落盘的修改。 这时 InnoDB 会被迫让新的写事务同步等待，强制推进 checkpoint（同步刷脏），表现为应用侧写入延迟骤增甚至短暂卡顿。

面试高频："redo log 写满了会怎样？" 准确说法不是"写满"，而是"写入追上了 checkpoint"——此时 InnoDB 必须停下来先把老脏页刷盘、 推进 checkpoint 腾出空间，写入线程会被阻塞，这是生产环境常见的"写入毛刺"根因之一。排查思路：观察 innodb_log_file_size 是否过小、 是否有超大事务或刷脏跟不上写入速度。

核心价值总结

  • 缩短崩溃恢复时间：恢复只需从 checkpoint LSN 开始重放，而不是从日志起点全量重放。
  • 释放 redo log 空间：checkpoint 之前的日志空间可以被覆盖复用，环形文件才能持续循环写下去。
  • 模糊检查点是并发与安全的折中：增量刷脏、增量推进，避免停机式刷盘，但也意味着推进速度有上限，写入压力过大时会反过来限流写入。

与其他知识点的联系

  • Redo Log：checkpoint LSN 直接决定了 redo log 环形文件里哪一段可以被新日志覆盖——checkpoint 之前的修改已经保证落盘，对应日志可以安全丢弃；checkpoint 之后的部分是这些修改唯一的存在证据，绝不能被覆盖。可以说 checkpoint 就是 redo log 循环写机制能够"转起来"的调度器，详见 Redo Log 与 WAL 机制。
  • 崩溃恢复流程：崩溃重启时不会从 redo log 起点开始重放，而是从最近一次的 checkpoint LSN 开始——checkpoint 之前的修改已经落盘，重放它们是多余的。checkpoint 位点本质上就是崩溃恢复的起点，checkpoint 推进得越及时，恢复所需重放的日志量就越小、恢复越快，详见 崩溃恢复流程。
  • 两次写机制：checkpoint 触发的刷脏动作，最终落盘时同样要经过 doublewrite 的两步写入（先顺序写副本，再写真实位置），保证被刷的脏页不会因为写入过程中断电而损坏——checkpoint 决定"刷哪个、什么时候刷"，两次写保证"刷的过程是安全的"，两者组合才能构成完整的刷脏链路。
  • Undo Log：undo log 所在的回滚段页面本质上也是普通数据页，其修改同样会产生脏页、被 flush list 追踪、受 checkpoint 调度刷盘，并不存在"undo log 不受 checkpoint 管"的特殊豁免。
