两次写机制（Double Write Buffer）

Redo log 能保证"崩溃后把已提交的修改补回来"，但前提是"页本身是完好的"。如果页在写入时就被写坏了一半，redo log 也无力回天——这正是两次写要解决的问题。

要解决的问题：部分页写入（Partial Page Write）

InnoDB 的页大小默认是 16KB，但操作系统与磁盘的原子写单元往往比这小得多——比如传统机械盘的扇区是 512B，一些文件系统/存储设备的原子写单元是 4KB。这意味着，操作系统一次向磁盘刷新 16KB 的页时，底层实际上是拆成多次更小的写操作完成的。如果这时候系统断电或进程崩溃，就可能出现"这 16KB 里前 4KB 是新数据，后 12KB 还是旧数据"的情况——这种页被称为部分写入页 / "脏页断裂"（torn page），它已经不是一份完整一致的数据，而是新旧数据的拼接体。

为什么 redo log 救不了这种情况：redo log 记录的是"在某个基准页的基础上，某个偏移处的内容改成了什么"，它的前提假设是这个基准页本身必须是完整、未损坏的。如果页已经是新旧数据交错的"缝合怪"，在这个已经损坏的页上再叠加 redo log 描述的增量修改，结果依然是错的——因为损坏的部分根本不在 redo log 的描述范围内。要恢复这种页，必须先有一份完整无损的页副本作为起点，再在其上应用 redo log。

解决方案：先顺序写副本，再落盘到真实位置

Double Write Buffer（两次写缓冲区）的思路是：在把脏页真正写到表空间中它该在的位置之前，先额外写一份完整副本到一块专门、物理连续的缓冲区域，等这份副本安全落盘之后，再把页写到它真正的位置。整个过程分两步：

Step 1：顺序写入 doublewrite 缓冲区
  脏页（16KB）→ 顺序追加写入 doublewrite 缓冲区（连续磁盘空间，顺序 I/O，速度快）
  写完并 fsync 后，这份副本已经安全落盘

Step 2：分散写入表空间真实位置
  脏页（同一份数据）→ 写入 ibd 数据文件中该页真正所在的位置（随机 I/O，分散在文件各处）
  如果这一步中途断电，页可能损坏，但 doublewrite 里还留着一份完好的副本

第一步之所以能做到又快又安全，是因为 doublewrite 缓冲区在磁盘上是物理连续的一段空间，写入是顺序 I/O，代价远低于第二步那种落在表空间各处的随机写；即便顺序写本身也可能遇到断电，但因为一次只覆盖 doublewrite 自己的连续区域，容易通过校验判断这次写是否完整。

在 MySQL 8.0.20 之前，doublewrite 缓冲区位于共享表空间 ibdata1 内；从 MySQL 8.0.20 开始，doublewrite 被拆分到独立的文件中（形如 #ib_16384_0.dblwr），使其管理更灵活，也便于按表空间粒度扩展。

崩溃恢复时如何使用 doublewrite

实例重启时，InnoDB 会对每个数据页做校验和（checksum）校验，用来判断这个页是否是一个"torn page"：

  1. 如果页的 checksum 正常，说明这个页写入是完整的，直接按正常流程用 redo log 前滚即可。
  2. 如果发现某个页 checksum 校验失败（即部分写入导致的损坏），InnoDB 会去 doublewrite 缓冲区里找到该页对应的完好副本，用这份副本整页覆盖恢复该页，使其恢复到写入前那一刻的完整状态。
  3. 页恢复为完整状态后，再在其上应用 redo log 中属于该页、且发生在这次写入之后的修改记录，把该页"前滚"到崩溃前应有的最新状态。
面试高频："两次写机制解决了什么问题？和 redo log 是什么关系？" 标准答案：两次写解决的是"页级别的写入完整性"问题（防止部分页写入导致页损坏），是 redo log 能够正确前滚恢复的前提条件；redo log 假设应用它的目标页是完整的，doublewrite 负责在崩溃后先把损坏的页"修复"成完整页，两者配合才能保证 crash-safe。

与 Redo Log、Change Buffer 的角色对比

机制解决的问题作用层面
Redo Log 已提交事务的修改在崩溃后不丢失（前滚） 逻辑上的"修改是否被持久化"，见 Redo Log 与 WAL 机制
Double Write Buffer 页本身在写入过程中是否完整，防止 torn page 物理层面的"页是否完好"，是 redo log 前滚的前提
Change Buffer 非唯一二级索引的随机写入效率 写入路径的性能优化，与崩溃安全无关，见 Change Buffer

常见误区

误区："开了 doublewrite 就是把每次写入的数据量翻倍，太浪费性能了，应该关掉（innodb_doublewrite=OFF）。" 虽然 doublewrite 确实带来了额外的写入量，但由于第一步是顺序 I/O，实际开销远低于"数据量翻倍"给人的直觉；而如果关闭它，一旦发生部分页写入，几乎没有办法恢复出一份干净的页，代价是数据损坏且难以修复。生产环境一般不建议关闭，除非底层存储本身已经保证了页级别的原子写（比如某些支持 atomic write 的文件系统/云盘），此时 doublewrite 才可以考虑关闭。

核心价值总结

  • 两次写解决的是"页级别的写入原子性"问题：磁盘/文件系统的原子写单元可能小于 InnoDB 页大小，断电会导致 torn page。
  • 先顺序写副本、再分散写真实位置，用一次可控的顺序 I/O 开销换取"崩溃后总能找回一份完整页"的保障。
  • 它是 redo log 能正确前滚的前提：redo log 假设页是完整的，doublewrite 负责在崩溃恢复时先把损坏页修复为完整页，两者配合才构成完整的 crash-safe 链路。
