崩溃恢复流程

redo log、undo log、doublewrite、checkpoint——前面四篇讲的机制，最终都是为了同一个目标服务：数据库进程被强制杀死后，重启时如何把数据恢复到一个正确、一致的状态。

崩溃恢复要解决的两个问题

数据库随时可能因为断电、OOM、kill -9 等原因异常终止。此时内存中的 Buffer Pool 全部丢失，磁盘上的数据页可能停留在"半新半旧"的中间状态， 并且可能存在一些事务已经提交、但修改还没来得及刷盘，也可能存在一些事务修改已经部分落盘、但事务本身并未提交。崩溃恢复要同时解决：

  • 已提交事务的持久性：凭 redo log 把它们"补齐"到磁盘上，保证 ACID 中的 D（持久性）不被破坏；
  • 未提交事务的原子性：凭 undo log 把它们的影响撤销干净，保证 ACID 中的 A（原子性）——要么全做，要么都不做。

第一步：校验页完整性（doublewrite 兜底）

重启时 InnoDB 首先要确认磁盘上的数据页本身没有物理损坏。回忆 两次写机制 一文：如果崩溃恰好发生在某个页刷盘写到一半的时刻， 就会出现"部分写"（partial page write，页内数据前后不一致，magic number/checksum 校验不过）。此时 InnoDB 会去 doublewrite buffer 的共享表空间副本中， 取出这个页最近一次的完整副本覆盖回去，先保证物理页本身是完好的，再进入日志重放阶段。

第二步：Redo 阶段——不区分提交状态，物理重建到崩溃前那一刻

从 上一篇的 checkpoint LSN 开始，顺序扫描并重放 redo log 中记录的每一条物理修改，直到日志末尾。 这里最容易被忽略、也是面试最爱追问的关键点：

关键点：redo 重放不关心事务是否提交，无论这条修改属于已提交事务还是未提交事务，只要 redo log 里记录了， 就无条件重放到对应数据页上。redo 阶段的唯一目标是把数据库的物理状态（所有数据页的内容）精确恢复到"崩溃发生的那一瞬间"， 它不做任何逻辑判断，只是"照着日志把动作再做一遍"。

重放过程中，InnoDB 会同时重建出崩溃时刻的活跃事务列表（哪些事务的 commit 记录没有出现在 redo log 里，说明它们没有提交）， 为下一步 undo 阶段提供依据。

第三步：Undo 阶段——只回滚未提交的事务

redo 全部重放完成后，磁盘上的数据页已经是一个物理上完整、一致的状态（如同崩溃那一刻的快照），但其中混杂着"合法"（已提交）和"不该存在"（未提交）的修改。 这时才轮到 undo log 登场：对第二步识别出的未提交事务，逐条反向应用它们的 undo 记录，把这些事务的修改撤销掉， 数据库最终只保留已提交事务的效果。

为什么必须"先 redo，后 undo"，不能反过来

这个顺序不是随意约定的，而是由依赖关系决定的：

  1. undo log 本身也存放在数据页里（回滚段），如果不先做 redo，这些存放 undo 记录的页本身可能也处于不完整/不一致的物理状态， 根本无法正确读取，更谈不上用它做回滚；
  2. undo 操作依赖"页处于确定状态"这个前提：回滚本质上也是对页做修改（把值改回旧版本），如果页此刻是残缺的、崩溃中途的状态， 在这种不确定基础上再叠加回滚操作，结果无法保证正确；
  3. 反过来，redo 不依赖 undo 是否执行过——它只是机械地把日志物理重放一遍，天然可以先做。

所以整体逻辑是：先用 redo 把物理世界"修复"到一个完整、确定的状态，再在这个确定状态之上，用 undo 做逻辑层面的"撤销"。

示例：事务 A 已提交，事务 B 未提交

假设崩溃前的时间线是这样的：

事务 A： UPDATE t SET balance=90 WHERE id=1 → COMMIT（redo log 中有 A 的 commit 记录）
事务 B： UPDATE t SET balance=50 WHERE id=2 → 尚未 COMMIT，此时进程被 kill

此时内存 Buffer Pool 中 A、B 的修改都可能还没刷盘，但都已写入 redo log（WAL 规则：先写日志再改页面/刷盘）

重启恢复过程：

  • Redo 阶段：顺序重放 redo log，把 id=1 的 balance 改成 90，id=2 的 balance 改成 50——两条记录都会被重放， 因为 redo 不管提交状态，只管把物理页恢复到崩溃前的样子。重放同时发现 redo log 中只有 A 的 commit 标记，B 没有，于是把 B 标记为"待回滚"。
  • Undo 阶段：只针对事务 B，取出它对应的 undo log（记录了 id=2 修改前的旧值），把 balance 从 50 改回崩溃前的原值。 事务 A 因为已提交，不做任何回滚，它的修改被完整保留。

最终结果：A 的修改生效（持久性得到保证），B 的修改被完全撤销（原子性得到保证）——就像 B 这个事务从未发生过一样。

面试高频："MySQL 崩溃后是怎么恢复数据的？" 标准答案框架：先校验并修复物理页（doublewrite）→ 从 checkpoint 开始重放 redo log （对所有事务，无论提交与否，物理重建到崩溃时刻）→ 再用 undo log 回滚崩溃时仍处于活跃、未提交状态的事务。 追问"为什么不能先 undo 再 redo？" 回答核心：undo 依赖页处于完整一致状态，而这个"完整一致"正是 redo 负责重建出来的。

核心价值总结

  • doublewrite 保证页本身没有物理损坏，是后续一切日志重放的前提。
  • redo 阶段只做物理重建，不看事务提交状态，把数据库精确恢复到崩溃那一刻的物理快照。
  • undo 阶段只处理未提交事务的逻辑回滚，建立在 redo 已经完成、页面状态确定的基础之上。
  • 这套机制正是 InnoDB 相比无日志机制的存储引擎（如后面要对比的 MyISAM）能够安全崩溃恢复的根本原因。

与其他知识点的联系

  • 两次写机制：恢复的第一步依赖它——如果某个页在崩溃时正好写到一半（torn page），后面所有基于这个页的 redo 重放都建立在错误的基础上；doublewrite 提供的完整副本是整套恢复流程能够开始的物理前提，详见 两次写机制。
  • Checkpoint 机制：恢复的第二步（redo 重放）不是从 redo log 文件开头扫描，而是从最近一次的 checkpoint LSN 开始——checkpoint 之前的修改已经落盘，不需要重放，checkpoint 推进得越勤，重启后要重放的日志量越小、恢复越快，详见 Checkpoint 机制。
  • Redo Log 与 Undo Log：本文的第二、三步正是这两种日志分工的直接体现——redo 只管物理重建、不看事务提交状态，undo 只处理崩溃时仍活跃的未提交事务的逻辑回滚，二者"先 redo 后 undo"的顺序是因为 undo log 本身依赖 redo 已经把页恢复到确定状态，详见 Undo Log 与回滚。
  • InnoDB 与 MyISAM 对比：本文描述的整套"doublewrite 修页 → redo 前滚 → undo 回滚"流程，正是 InnoDB 具备崩溃安全能力、而 MyISAM 完全不具备（崩溃后需要 myisamchk 手动修复且可能丢数据）的根本原因，详见 InnoDB 与 MyISAM 对比。
