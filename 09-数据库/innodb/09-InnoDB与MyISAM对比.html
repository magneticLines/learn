InnoDB 与 MyISAM 对比

MySQL 5.5 之前默认引擎是 MyISAM，之后换成了 InnoDB——这不是一次随意的换皮，而是"能不能撑起真实在线业务"的路线之争。

两种截然不同的设计哲学

MyISAM 的设计目标是"简单、快、省内存"：不支持事务，不做崩溃安全保证，换取更小的存储开销和某些只读场景下的高性能。 InnoDB 的设计目标是"安全、并发、可靠"：完整支持事务与崩溃恢复（正是前面几篇 redo log、 undo log、崩溃恢复 讲的那一整套机制），代价是更复杂的实现和相对更高的存储/内存开销。

核心维度对比

维度InnoDBMyISAM
事务支持 完整支持 ACID 事务，配合 ACID 四大特性与 MVCC 提供隔离性 不支持事务，每条语句执行完立即生效，无法回滚
锁粒度 基于索引实现行级锁，不同事务可以并发修改同一张表的不同行 只支持表级锁，写操作会锁住整张表，写并发能力弱
外键约束 支持 不支持（语法上可以写，但不会真正生效约束）
崩溃恢复能力 基于 redo log + undo log 的日志式崩溃恢复，可精确恢复到崩溃前状态 没有类似的日志机制，崩溃后极易损坏，需要 myisamchk 手动修复，且可能丢数据
全文索引 MySQL 5.6 才开始支持 FULLTEXT 索引 很早就支持 FULLTEXT 索引，曾是全文检索的唯一选择
数据组织方式 聚簇索引：数据按主键顺序物理存储在 B+ 树叶子节点中，主键查询即数据查询 非聚簇：数据单独存于堆文件（.MYD），所有索引（包括主键）都是指向数据物理行号的引用，没有聚簇概念
COUNT(*) 性能 无条件 COUNT(*) 需要走索引扫描计数，因为 MVCC 下不同事务看到的可见行集合不同，无法简单缓存一个数字，复杂度 O(n) 在表的元数据里缓存了精确行数，无条件 COUNT(*) 直接读取该值，复杂度 O(1)

行级锁 vs 表级锁：并发能力的本质差异

MyISAM 的表锁意味着：只要有一个写事务在改表，其它任何读写都要排队等待，哪怕它们操作的是完全不同的行。这在写多读多的并发场景下会成为严重瓶颈。 InnoDB 的行锁基于索引实现——锁加在索引记录上，只有真正访问到同一行（或同一索引区间，涉及间隙锁时）的事务才会互相阻塞， 不同行之间的修改可以完全并行，这是 OLTP（高并发在线事务处理）场景下 InnoDB 碾压 MyISAM 的关键原因。

常见误区："InnoDB 的行锁是加在行数据上的" ——更准确的说法是行锁通过给索引记录加锁来实现。 如果一条 UPDATE 语句的 WHERE 条件没有命中任何索引，InnoDB 会退化成锁住全表扫描到的所有行，效果上等同于表锁，这也是为什么"锁等待"问题经常和"缺索引"绑在一起排查。

COUNT(*) 差异背后的根本原因：MVCC

这是一道经典面试题，很多人只记住结论"MyISAM 的 COUNT 快"，却说不清原因。本质在于 MVCC： InnoDB 中同一时刻不同事务因为隔离级别和可见性判断，看到的"当前表里有多少行"可能是不同的数字（比如一个事务开始后另一个事务插入了新行， 在可重复读隔离级别下前者可能看不到这行）。因此 InnoDB 无法维护一个所有事务通用的"精确行数"缓存，只能老老实实扫描索引、逐行做可见性判断后计数。 MyISAM 没有事务和多版本并发的概念，任意时刻表的行数是唯一确定的，所以可以直接维护并读取这个缓存值。

为什么 MySQL 5.5 之后默认引擎换成了 InnoDB

MySQL 5.5（2010 年）将默认存储引擎从 MyISAM 切换为 InnoDB，根本原因是绝大多数真实在线业务的核心诉求，MyISAM 从架构上就无法满足：

  • 事务是刚需：转账、下单、库存扣减这类操作必须保证原子性和一致性，MyISAM 完全没有事务能力；
  • 写并发是刚需：互联网应用普遍是高并发读写混合负载，MyISAM 的表锁在写密集场景下几乎无法扩展；
  • 崩溃安全是刚需：生产环境无法接受"服务器一断电，表就可能损坏，需要人工介入修复"这种风险，InnoDB 的日志式恢复机制解决了这个问题。

MyISAM 剩下的优势场景（只读或读远大于写、不需要事务、需要早期版本的全文索引）在今天已经相当边缘——现代 InnoDB 已支持全文索引， 读性能差距也随着版本演进大幅缩小，因此新项目基本不再有理由主动选择 MyISAM。

面试高频："为什么 MySQL 5.5 之后默认引擎换成了 InnoDB？" 参考答案：因为 InnoDB 提供了 MyISAM 不具备的三大能力—— 事务（ACID）、行级锁定的高并发写入、基于 redo/undo log 的崩溃安全恢复，而这三者正是绝大多数生产 OLTP 系统的刚性需求， MyISAM 仅存的性能优势（如 O(1) 的 COUNT(*)）不足以弥补其在可靠性和并发性上的根本缺陷。

核心价值总结

  • 本质区别是"要不要事务、要不要崩溃安全"，其余差异（锁粒度、索引组织方式、COUNT 性能）大多是这个根本选择派生出来的结果。
  • 行级锁 + MVCC 是 InnoDB 支撑高并发的两大支柱，与本 Tab 前面几篇讲的 redo/undo/checkpoint/崩溃恢复共同构成了 InnoDB 的可靠性基石。
  • 选型时不是"哪个更好"，而是"业务要不要事务和高并发写"——现代 MySQL 生态下，InnoDB 几乎是默认且正确的选择。

与其他知识点的联系

  • 崩溃恢复流程：两者崩溃安全能力的差距不是量的差异而是质的差异——InnoDB 靠 doublewrite 修页、redo 前滚、undo 回滚这一整套机制精确恢复到崩溃前状态，MyISAM 完全没有对应机制，崩溃后表可能直接损坏，只能靠 myisamchk 事后修复且不保证数据完整，详见 崩溃恢复流程。
  • Undo Log：MyISAM 不支持事务，也就没有回滚的概念，自然不需要 undo log；InnoDB 的事务能力和行级锁、MVCC 都建立在 undo log 提供的版本链之上，这是两种引擎在设计哲学上分道扬镳的起点，详见 Undo Log 与回滚。
  • Checkpoint / Redo Log：InnoDB 的行级锁、崩溃恢复、COUNT(*) 性能等差异，追根溯源都来自"要不要支持事务与崩溃安全"这一个根本选择，而 redo log 与 checkpoint 正是支撑这个选择落地的具体机制，MyISAM 因为不需要事务语义，天然不需要这套日志体系。
  • 主从复制原理：MySQL 复制依赖的 binlog 是 Server 层日志，与存储引擎无关——哪怕是 MyISAM 表也能参与主从复制；但涉及崩溃恢复、事务一致性的场景（如 InnoDB 与 binlog 之间的两阶段提交）只有 InnoDB 才能提供完整保证，这也是生产环境几乎不会用 MyISAM 做主库的原因之一，详见 主从复制原理。
