Undo Log 与回滚

Redo log 保证的是"已提交的修改不丢"，那"没提交的修改要能撤销"靠谁？答案是 undo log——它不仅是回滚的后悔药，更是 MVCC 能实现的地基。

Undo Log 是什么：逻辑日志，记录"反操作"

Undo log 记录的是修改之前的旧版本数据，本质上是一种逻辑日志（logical log）：它不像 redo log 那样记录"页的哪个字节改成了什么"，而是记录"如果要撤销这次修改，应该执行什么样的反向操作"。

  • INSERT 的 undo：记一条对应的 DELETE（只需要主键值，删除刚插入的这行即可撤销）。
  • DELETE 的 undo：记一条对应的 INSERT（把整行数据记下来，回滚时重新插回去）。实际上 InnoDB 对 DELETE 采用"delete mark"先打删除标记，真正的物理删除交给后台 purge 线程。
  • UPDATE 的 undo：记修改前的旧列值（如果改了主键，会被拆成 DELETE + INSERT 两步处理），回滚时用旧值覆盖回去。
类比：undo log 像是编辑器里的"撤销栈"（Ctrl+Z 历史）。你敲一个字，撤销栈里就压入一条"删除刚才这个字"的记录；你删一个字，撤销栈里压入的是"把这个字加回来"。真正撤销时，不是恢复整个文件快照，而是按记录反着执行一遍。

用途一：事务回滚（ROLLBACK）

这是最直觉的用途：事务执行过程中每一次修改都会先记一条 undo log。如果事务中途执行了 ROLLBACK，InnoDB 就沿着这个事务写下的 undo log 反向执行，把数据恢复到事务开始之前的状态。事务提交后，对应的 undo log 不再需要用于回滚，但不能立刻删除——原因见下一节。

用途二：MVCC 的版本链

这是 undo log 更"隐蔽"但同样关键的用途。InnoDB 每一行记录除了业务列，还有两个隐藏列：trx_id（最后一次修改该行的事务 ID）和 roll_pointer（指向这一行修改前的 undo log 记录）。多个事务先后修改同一行时，这些 undo log 记录会通过 roll_pointer 串成一条版本链，链表头是最新版本，往回追溯就是一次次修改前的旧版本。

当某个事务的 ReadView 判断"当前最新版本对我不可见"时，就沿着这条版本链往回找，直到找到一个对它可见的历史版本为止——这正是 MVCC 能够"读到旧快照"的物理实现基础。ReadView 如何判断可见性、什么是快照读，详见 MVCC 原理；这里只需要记住一点：版本链的载体就是 undo log。

例子：一行数据被连续 UPDATE 三次

当前行（trx_id=103，name='张三丙'）
  └─ roll_pointer → undo(trx_id=102): name='张三乙'
        └─ roll_pointer → undo(trx_id=101): name='张三'
            └─ roll_pointer → NULL（该行最初插入时没有更早版本）

如果事务 101 还没提交时事务 102 就已经修改过这行，那么某个开启较早、ReadView 里 102/103 都不可见的事务，读到"当前行"后会发现 trx_id=103 对它不可见，于是顺着 roll_pointer 找到 undo(102)，再判断还是不可见，继续往前找到 undo(101)，如果这个版本对它可见就返回这份数据——整个过程完全不阻塞其他事务的写入。

Redo Log vs Undo Log

维度Redo LogUndo Log
日志类型 物理日志——记录"某页某偏移改成了什么" 逻辑日志——记录"要撤销这次修改该做什么反操作"
核心用途 崩溃后恢复（前滚已提交但未刷盘的修改） 事务回滚 + MVCC 多版本读
存储方式 固定大小的 redo log 文件组，循环覆盖写（write-ahead log） 存放在 undo tablespace 的 undo segment 中，不是循环覆盖，由独立的 purge 线程按需清理
生命周期 刷盘且对应脏页落盘后即可覆盖 要等到没有任何活跃事务的 ReadView 还需要读到这个版本，才能被 purge 回收
与 WAL 关系 WAL 机制的核心角色，见 Redo Log 与 WAL 机制 不直接参与 WAL，但自身修改也需要写 redo log 保护（undo 页本身也是页，也可能因崩溃丢失）

常见误区与生产陷阱

误区："undo log 只是用来回滚的，提交了就该删。" 这是最常见的误解。实际上事务一旦涉及 MVCC 读，undo log 就必须保留到"没有任何事务可能还需要读这个版本"为止。如果系统里有一个开了很久没提交/没结束的长事务，它的 ReadView 会让很多本该被清理的旧版本一直保留在版本链上，导致 undo log 不断膨胀（俗称 "undo log 堆积"），严重时会撑爆 undo tablespace 磁盘空间，也会拖慢 purge 和查询（版本链变长，扫描更多历史版本）。
面试高频："MVCC 是怎么实现多版本读的？" 标准答案要点：每行有隐藏的 trx_id 和 roll_pointer，roll_pointer 指向 undo log 中该行的旧版本，多次修改形成版本链；ReadView 结合当前活跃事务列表判断某个版本对本事务是否可见，不可见就沿版本链继续往前找。这道题本质上考的就是"undo log 如何支撑 MVCC"。

核心价值总结

  • undo log 是"反操作"的逻辑日志，与 redo log 记录页级物理修改的方式完全不同，二者互补而非替代。
  • 两大用途缺一不可：回滚保证事务原子性（呼应 ACID 特性 中的 A），版本链支撑 MVCC 实现非阻塞的一致性读。
  • 长事务是 undo log 膨胀的元凶，生产环境要监控事务运行时长，避免"一个事务拖垮整个实例的 purge 效率"。
