跨库 Join 与全局表

单库时代一条 JOIN 语句就能搞定的关联查询，分库分表之后突然变得无解——数据库引擎压根不知道另一个分片上还有数据。

为什么分片后无法跨库 JOIN

MySQL 的 JOIN 是在单个数据库实例内部完成的：优化器扫描驱动表，对每一行去被驱动表里查找匹配（联系索引 Tab 讲过的 Nested-Loop Join 原理）。 这一切都建立在"两张表在同一个数据库进程里，可以互相访问"的前提上。

分库之后，order 表可能在 db_0，而 user 表在另一个完全独立的库 user_db， 甚至同一张 order 表本身也被拆成了分布在不同物理库上的 order_0、order_1…… 数据库实例之间没有原生的跨进程 JOIN 能力，一条 SELECT ... FROM order o JOIN user u ON o.user_id = u.id 根本无法直接执行。

类比：单库 JOIN 就像在一个仓库里，管理员可以同时打开两排货架对照着找货；分库之后货物被分散存放在不同城市的仓库里，管理员没法人肉"当场对照"，必须先把其中一批货运到同一个地方，或者提前在每个仓库都放一份小字典方便就地核对。

思路一：业务层聚合查询

把原本一条 SQL 完成的 JOIN，拆成多条独立查询，在应用代码里做内存关联（俗称"服务层 join"）。

业务层聚合示例
// 1. 先查订单分片，拿到 order 列表
List<Order> orders = orderMapper.selectByUserId(userId);

// 2. 再查用户库，拿到关联的用户信息
Set<Long> userIds = orders.stream().map(Order::getUserId).collect(...);
List<User> users = userMapper.selectByIds(userIds);

// 3. 在应用内存里按 id 关联，组装成最终结果
Map<Long, User> userMap = users.stream().collect(toMap(User::getId, u -> u));

优点：不依赖任何中间件，实现直观，适用于任意分片方案。

代价：把数据库的关联能力转移到了应用层，多次网络往返，代码复杂度上升；如果关联的数据量很大（比如上万条订单要关联上万个用户），还得自己处理分批查询，稍不注意就会写出 N+1 查询的低效代码。

思路二：全局表（冗余字典表）

对于数据量小、几乎不变、被频繁关联查询的表（比如省市区字典表、商品类目表、配置表）， 干脆在每个分片里都存一份完整拷贝，这样任何分片上的业务表都能在本地直接 JOIN，不需要跨库。

全局表：城市字典表在每个分片冗余一份
  db_0: order_0 表 + city 表（全量拷贝）
  db_1: order_1 表 + city 表（全量拷贝，与 db_0 内容一致）
  db_2: order_2 表 + city 表（全量拷贝，与 db_0 内容一致）

代价是写多份、维护一致性：字典表一旦更新，需要同步更新到所有分片，通常通过中间件自动广播写入，或者干脆约定这类表极少变更来规避一致性问题。

陷阱：全局表只适合"小表 + 几乎不变"的场景，别把订单表这种大表也搞成全局表——那样每个分片都要存全量数据，直接违背了分片的初衷。

思路三：ER 分片（绑定关系分片）

ER 分片（Entity-Relationship 分片）的思路是：把经常一起查询的表，按同一个分片键分片，让关联的数据始终落在同一个物理分片上，从而在分片内部就能完成 JOIN。

ER 分片示例：订单与订单明细
-- order 表按 user_id 分片
CREATE TABLE order (id, user_id, ...);   -- 分片键 user_id

-- order_item 表也按 order 所属的 user_id 分片（而不是按自己的主键）
CREATE TABLE order_item (id, order_id, user_id, ...); -- 分片键同样是 user_id

-- 结果：同一个用户的 order 和 order_item 永远落在同一个物理分片上
-- 分片内部可以直接 JOIN，无需跨库

这是业界最常用的思路，因为大部分业务场景里"父子表""主表与明细表"天然有一个共同的归属键（比如都归属于某个用户或某个租户）。

三种思路对比

方案核心思想适用场景主要代价
业务层聚合查询 拆分成多次查询，应用内存里关联 任意场景的兜底方案 多次网络往返，代码复杂度上升
全局表 小表在每个分片冗余一份全量拷贝 字典表、配置表等小且几乎不变的数据 写入需要同步到所有分片，只适合小表
ER 分片 关联表按同一分片键分片，保证物理同分片 父子表、主从明细表等有天然归属键的场景 分片键选择被强约束，灵活性下降

常见误区

误区："上了分库分表中间件（如 ShardingSphere），跨库 JOIN 就能像单库一样随便写了。" 中间件确实能帮你自动把跨库 JOIN 拆解成多次查询再合并（下一篇会讲），但这只是把"业务层聚合查询"的脏活自动化了，性能代价依然存在——涉及的分片越多，合并成本越高。设计阶段就应该通过 ER 分片尽量减少真正需要跨库关联的场景，而不是指望中间件"魔法般"解决一切。
面试高频："分库分表之后怎么处理关联查询？" 标准答案要覆盖三点：能否通过 ER 分片让数据物理同分片；小表能否做成全局表冗余；实在不行退化成业务层多次查询再内存聚合。

核心价值总结

  • 分片打破了单库天然支持的 JOIN 能力，本质原因是数据库实例之间没有跨进程关联查询的能力。
  • 全局表用空间换关联能力，ER 分片用分片键设计规避跨库关联，业务层聚合是通用兜底方案。
  • 好的分片键设计应该在项目初期就考虑关联查询模式，而不是分完片才发现处处要跨库 JOIN。
