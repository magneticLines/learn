分库分表的路由策略

决定一行数据"该去哪个分片"的规则，是分库分表方案里最难反悔的一次决定——选错了，扩容那天就是噩梦的开始。

路由策略要解决什么问题

上一篇讲到，水平拆分把一张大表的数据分散到多个分片里。但数据分散之后，应用层写入或查询一行数据时，怎么知道它在哪个分片？ 这就是路由策略要回答的问题：给定一个分片键（sharding key，比如 user_id），用一个确定性的规则算出它落在哪个分片。

类比：路由策略就像图书馆给书编索书号的规则。按书名首字母分区（类似 hash）取书很快但很难"腾出一个新书架"；按入库时间分区（类似 range）方便加新书架，但热门新书全挤在最新那个区，其他区门可罗雀（数据倾斜）。

方案一：hash 路由

对分片键做哈希运算，再对分片数量取模，决定数据落在哪个分片。

hash 路由示例
-- 分片键 user_id，共 4 个分片
shard_index = hash(user_id) % 4

-- user_id = 1001 → hash(1001) % 4 = 1  → 落在 order_1
-- user_id = 1002 → hash(1002) % 4 = 2  → 落在 order_2

优点：数据分布非常均匀，几乎不会出现某个分片数据量远超其他分片的"热点"问题。

致命缺点：取模运算和分片数量强绑定。分片数从 4 扩容到 8，几乎所有数据的 hash % N 结果都会变化， 意味着绝大部分数据都要重新计算位置并迁移——这就是"扩容需要迁移大量数据"痛点的根源。这跟 Java 里 HashMap 扩容时重新计算桶位置、触发大量 rehash 是同一个道理。

方案二：range 路由

按分片键的值区间划分，比如按 user_id 的范围分段，或按时间范围分表（常见于订单表按月分表）。

range 路由示例
-- 按 user_id 区间分片
[0, 1000000)     → shard_0
[1000000, 2000000) → shard_1
[2000000, 3000000) → shard_2
-- 扩容时只需新增一个区间段，例如 [3000000, 4000000) → shard_3
-- 老数据完全不用动，新数据自然落到新分片

优点：扩容极其简单，只需要新增一个区间映射到新分片，历史数据不需要迁移；范围查询（比如按时间区间查订单）效率也很高，因为数据在物理上连续存放。

致命缺点：容易出现数据倾斜——新注册用户、新订单集中在最新的分片，老分片逐渐冷却成"僵尸分片"，新分片则持续高负载，起不到真正分流的作用。电商大促期间新订单暴增更会加剧这种不均衡。

方案三：一致性哈希

一致性哈希是对 hash 路由痛点的改良：把哈希值空间抽象成一个环（0 ~ 2³²-1），每个分片节点在环上占据若干个位置，数据按哈希值顺时针找到最近的节点。

一致性哈希环
  0 ────────────────────── 2³²-1
  │ │ │ │
  node-A node-B node-C (回到 node-A)
  数据落点顺时针找最近的 node，新增 node-D 只影响相邻区间的一小部分数据

核心优势：新增或删除一个节点时，只有该节点相邻区间的数据需要迁移，不会像普通 hash 取模那样引发全量数据搬迁。 为了避免节点分布不均导致的数据倾斜，实践中通常引入虚拟节点（每个物理节点在环上映射成几百个虚拟节点），让数据分布更均匀。

联系已学：一致性哈希不是数据库独有的发明，Redis Cluster、分布式缓存、CDN 节点选择都在用同一套思想解决"增删节点尽量少影响已有数据"的问题。

三种方案对比

路由策略数据均匀性扩容代价范围查询效率典型场景
hash 路由 很均匀 高（几乎全量数据迁移） 差（数据打散，无法利用区间） 分片数量长期固定的场景
range 路由 容易倾斜 低（新增区间即可） 好（区间数据物理连续） 按时间归档的日志表、订单表
一致性哈希 较均匀（配合虚拟节点） 低（只影响相邻区间） 差 节点会动态增减的分布式存储/缓存

常见误区

误区："分片数量定小一点没关系，不够了再加。" 恰恰相反——如果用的是普通 hash 取模路由，分片数量一旦要变化，代价极其高昂（需要停机或双写迁移几乎全部数据）。工程上常见的做法是预分片：一开始就按未来 3-5 年的数据量规划出足够多的逻辑分片（比如 1024 个逻辑分片），再把多个逻辑分片映射到少量物理库上；后续物理扩容时只需要迁移部分逻辑分片，而不用重新计算路由规则。
面试高频："分库分表扩容为什么困难？" 核心原因是路由规则和物理分片数量绑定，一旦分片数变化，路由结果全部失效，需要重新计算并迁移数据。一致性哈希和预分片都是为了缓解这个问题而生的工程手段。

核心价值总结

  • hash 路由换来分布均匀，代价是扩容困难。
  • range 路由换来扩容简单，代价是容易数据倾斜。
  • 一致性哈希是二者的折中，用虚拟节点缓解倾斜，用环形结构缓解扩容代价。
  • 路由策略是分库分表方案里最难事后修改的决定，选型前必须想清楚未来的扩容路径。
