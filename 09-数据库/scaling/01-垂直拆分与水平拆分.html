垂直拆分与水平拆分

单机数据库撑不住了怎么办？先别急着分库分表——搞清楚你的瓶颈到底是"业务太复杂"还是"数据太多"，答案完全不同。

为什么单机数据库会遇到瓶颈

一台数据库服务器的容量是有物理上限的：CPU 核数、内存大小、磁盘 IOPS、网卡带宽都是有限资源。 当业务增长到一定规模，会遇到两类完全不同性质的瓶颈：

  • 业务复杂度瓶颈：一个库里塞了用户、订单、商品、支付、日志几十张毫不相关的表，任何一个模块的变更都可能锁全库、影响其他模块，团队协作也互相踩脚。
  • 数据量/并发瓶颈：单表几亿行，索引树深度增加、Buffer Pool 装不下热数据，查询越来越慢；或者单机连接数、TPS 到顶，加多少索引都无济于事。
类比：垂直拆分好比一家小饭馆生意好了，把"炒菜""收银""洗碗"分给不同的人专门负责（分工提效）；水平拆分则是饭馆一家店坐不下人了，直接开分店，同样的菜单，不同的客人分流到不同的店里（分流量）。这是两个完全不同维度的问题，解法自然也不同。

垂直拆分：按业务/字段拆

垂直拆分是把一个大库按业务模块拆成多个库（比如用户库、订单库、商品库各自独立）， 或者把一张宽表按字段冷热拆成多张表（常用字段一张表，很少查询的大字段/富文本另放一张表）。

垂直拆分示例：宽表拆分
-- 拆分前：一张大宽表，商品详情富文本很少被列表查询用到，却每次 SELECT * 都要带上
CREATE TABLE product (
  id BIGINT, name VARCHAR(255), price DECIMAL,
  description MEDIUMTEXT,  -- 几十 KB 的富文本，占满 Buffer Pool
  detail_images TEXT
);

-- 拆分后：高频字段和低频大字段分离，主表页密度大幅提升
CREATE TABLE product (id BIGINT, name VARCHAR(255), price DECIMAL);
CREATE TABLE product_detail (product_id BIGINT, description MEDIUMTEXT, detail_images TEXT);

垂直拆分解决的是"业务耦合"问题，它带来的好处：

  • 不同业务库可以独立扩容、独立优化、独立运维，互不干扰。
  • 热点字段和冷字段分离后，主表每页能装下更多行，缓存命中率更高（联系索引 Tab 讲过的页结构，一页 16KB，行越窄能塞的行数越多）。
  • 符合微服务"数据库随服务拆分"的天然演进路径。
注意：垂直拆分并不能解决单表数据量过大的问题——拆完之后订单表还是那张订单表，该有几亿行还是几亿行。它只是让"不相关的东西不再互相拖累"，本质上是业务架构层面的解耦，不是容量层面的扩容。

水平拆分：按行拆（分库分表）

水平拆分是把同一张表的数据行按某种规则（通常是某个字段的 hash 或 range）分散到多个库或多张结构相同的表中， 每个分片只存全量数据的一部分。这就是俗称的"分库分表"。

order 表（水平拆分为 4 张分表）
  ├─ order_0（user_id % 4 == 0 的订单）
  ├─ order_1（user_id % 4 == 1 的订单）
  ├─ order_2（user_id % 4 == 2 的订单）
  └─ order_3（user_id % 4 == 3 的订单）

水平拆分解决的是"单表数据量/并发"瓶颈：每张分表的数据量只有原来的 1/N，索引树更浅、查询更快；同时多个分片可以分布在不同物理机器上，并发写入能力也随分片数线性提升。

陷阱：很多人一遇到"表太大就分表"，但没想清楚路由规则怎么定、以后怎么扩容——这些坑会在下一篇"分库分表的路由策略"里展开讲。水平拆分是一把双刃剑，它在解决容量问题的同时，会引入跨库 JOIN、分布式事务、分布式 ID 等一整套新问题（后面几篇会依次讲）。

两者对比：解决的瓶颈不同

维度垂直拆分水平拆分
拆分依据 按业务模块 / 字段冷热 按行（某个字段的值）
解决的瓶颈 业务耦合、团队协作效率、宽表 IO 单表数据量过大、单机并发/连接数瓶颈
拆分后表结构 不同库/表结构不同 各分片表结构完全相同
典型触发条件 一个库塞了几十个不相关业务模块 单表数据量达到千万到亿级、单机 QPS 到顶
引入的新问题 跨库事务（不同业务库间） 跨分片 JOIN、分布式 ID、分布式事务、路由维护

常见误区

误区一："表变大了就应该分库分表。" 实际上应该先看：能不能加索引、能不能优化 SQL、能不能升级硬件（垂直扩容）、能不能读写分离分担查询压力？分库分表是最后一步，因为它的运维复杂度和开发成本极高，一旦分了想合回来非常困难。
误区二：把垂直拆分和水平拆分对立起来。实际工程中两者经常组合使用：先按业务垂直拆成订单库、用户库，订单库量还是太大，再对订单表做水平拆分成多张分表。
面试高频："分库分表的分库和分表有什么区别？" 分表解决单表数据量和索引效率问题；分库解决单机连接数、CPU、磁盘 IO 等物理资源瓶颈。实践中通常"先分库，库内再分表"，因为分库能把压力真正分散到不同物理机器上。

核心价值总结

  • 垂直拆分是业务解耦，按模块或字段冷热拆分，降低系统复杂度，不解决容量问题。
  • 水平拆分是容量扩容，按行拆分到多个分片，是应对海量数据和高并发的终极手段，但代价高昂。
  • 两者常常组合使用，且水平拆分会引出路由策略、分布式 ID、跨库 JOIN 等一系列新问题，这也是本 Tab 接下来几篇的主线。
