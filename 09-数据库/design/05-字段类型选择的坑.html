字段类型选择的坑

字段类型选错，表面上看程序照样能跑，但会在排序、索引、精度、存储空间上埋下隐患，往往等数据量大了才集中爆发。

为什么字段类型不是小事

很多人建表时图省事，能用 VARCHAR 就不纠结用什么类型，能用 TEXT 就不管长度限制。 但字段类型决定了三件事：存储空间占用、比较/排序的语义是否正确、索引能否被高效使用。选错类型不会让 SQL 报错， 却会在数据量增长后逐渐显现为查询变慢、排序结果诡异、金额对不上账这类"隐性 bug"。

坑一：用 VARCHAR 存数字

比如把订单状态码、商品编号定义成 VARCHAR，看似灵活，实际上会导致字符串排序和数值排序语义不一致：

字符串排序 vs 数值排序
-- 字段是 VARCHAR，存的是 '1', '2', '10', '20'
SELECT * FROM orders ORDER BY 排序号;
-- 字符串比较结果：'1' < '10' < '2' < '20'，排序变成 1, 10, 2, 20 —— 不是你想要的数值顺序！

除了排序语义错误，字符串比较、索引查找的效率也明显低于整数比较（字符集、排序规则都会参与比较逻辑，CPU 开销更大）。 只要是参与数值计算、范围查询、排序的字段，一定要用 INT / BIGINT / DECIMAL 等数值类型，不要图方便用字符串。

常见误区："手机号该用什么类型？" 很多人下意识用 INT，但手机号从不参与加减乘除运算，也可能有前导 0（部分国家/地区区号），本质是一串"标识符"而非"数字"，应该用 VARCHAR 存储。判断标准不是"看起来像不像数字"，而是"这个字段是否会参与数值运算或范围比较"。

坑二：滥用 TEXT/BLOB 存本可以定长存储的数据

TEXT/BLOB 类型在 InnoDB 中的存储和访问方式与普通列不同：短字符串可能和行数据一起存在聚簇索引页里， 超过阈值后会被存储到溢出页（Overflow Page）中，访问时需要额外的一次页读取。

常见的滥用场景：

  • 用 TEXT 存储固定格式的状态描述（比如"启用/禁用"），实际上用 ENUM 或 TINYINT 编码即可。
  • 用 TEXT 存储省份/城市名称，实际上是有限枚举集合，VARCHAR(20) 甚至字典表 + 外键 ID 更合适。
  • 用 BLOB 直接在业务库里存图片、文件内容，导致主表体积暴涨，备份、迁移、Buffer Pool 缓存效率都会被拖累（更好的做法是文件存对象存储，数据库只存 URL）。
后果：大字段拖慢的不只是这一列本身的读写，还会拖累整张表的扫描效率——因为 SELECT * 或全表扫描时，这些大字段会占用大量 I/O 带宽和 Buffer Pool 空间，把本该用来缓存热点数据的内存挤占掉。这个问题在最后一篇"反模式"里还会结合"大字段垂直拆分"进一步展开。

坑三：金额用 FLOAT / DOUBLE 导致精度丢失

FLOAT/DOUBLE 是近似浮点数，底层用二进制科学计数法表示，无法精确表达大多数十进制小数（这和 Java 里 0.1 + 0.2 != 0.3 是同一个根源问题）。

浮点数精度丢失示例
SELECT 0.1 + 0.2;
-- 结果可能不是精确的 0.3，而是 0.30000000000000004（取决于具体实现和精度）

-- 如果金额字段用 FLOAT，多次加减后误差累积，账对不上

金额、汇率、税率这类要求精确计算的字段，必须使用 DECIMAL(M, D)（定点数，M 是总位数，D 是小数位数）， 它是按十进制精确存储的，不会有浮点数的舍入误差。

正确做法
订单金额 DECIMAL(10, 2) -- 最多 8 位整数 + 2 位小数，精确到分

另一种业界常见做法是把金额换算成"分"存成整数（金额_分 BIGINT），彻底避开小数运算， 在需要展示时再除以 100 转换成"元"。这也是很多支付系统（如支付宝、微信支付接口）对外部约定金额单位为"分"的原因。

面试高频："数据库里金额字段该用什么类型？为什么不用 FLOAT？" 标准答案：FLOAT/DOUBLE 是二进制近似浮点数，无法精确表示大多数十进制小数，多次运算后误差会累积，导致账务对不上；应该用 DECIMAL 定点数保证精确计算，或者把金额转为最小单位（分）用整数存储。

坑四：时间用字符串存储

有些团队图方便，把时间字段定义成 VARCHAR，存 "2026-07-03 10:30:00" 这样的字符串。这看起来能正常显示，但代价很大：

  • 无法用日期函数计算：DATE_ADD、DATEDIFF 等函数无法直接作用于字符串，要先做类型转换，既麻烦又可能因为格式不统一（有的存 2026/07/03，有的存 2026-07-03）而转换失败。
  • 范围查询无法走索引优化或语义错误：字符串的字典序比较在格式不统一时会给出错误结果，即使格式统一，效率也不如原生时间类型的二进制比较。
  • 存储空间浪费：字符串形式的时间通常需要 19+ 字节，而 DATETIME 只需要 5-8 字节（视精度而定），TIMESTAMP 只需要 4 字节。

正确做法是使用原生时间类型：

类型存储空间取值范围适用场景
DATE 3 字节 1000-01-01 ~ 9999-12-31 只需要日期，不需要时间（如生日、日报日期）
DATETIME 5-8 字节 1000-01-01 ~ 9999-12-31 不受时区影响的时间点，范围宽（如历史记录、日志）
TIMESTAMP 4 字节 1970-01-01 ~ 2038-01-19 需要自动转换时区的时间点，占用空间更小，但范围窄（要注意 2038 问题）

坑五（附带提醒）：字段该不该允许 NULL

这个知识点在"SQL 与关系模型基础" Tab 讲过 NULL 的三值逻辑陷阱，这里从字段设计角度补一句： 尽量给字段设置合理的默认值（如数值型默认 0，字符串默认空字符串），减少不必要的 NULL。 NULL 列会让索引统计和存储结构（NULL 值标记位）复杂化，也容易在业务判断时踩三值逻辑的坑。

核心价值总结

  • 选型原则：该用数值类型的字段不要用字符串（排序/索引效率），该用定长类型的字段不要用大字段类型（I/O 和缓存效率），该用精确类型的字段不要用浮点数（精度），该用原生时间类型的字段不要用字符串（计算和存储效率）。
  • 本质规律：字段类型选择的核心是让"存储的物理表示"匹配"这个字段真实的业务语义"——是数字就该能做数值比较，是钱就该精确无误差，是时间就该能做日期运算。
  • 这些坑往往在数据量小时不会暴露，等到表变大、查询变多才会集中体现为性能问题或数据错误，所以类型选择要在建表阶段就想清楚，而不是等出问题再返工迁移（大表改字段类型的成本极高）。
