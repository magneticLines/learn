索引与执行计划

慢查询定位与优化流程

前面几篇讲的最左前缀、索引失效、filesort，都是"武器库"；这一篇讲的是"打仗流程"——怎么在几百万条 SQL 里揪出真正拖垮系统的那一条，并且验证优化确实生效了。

第一步：打开慢查询日志，让问题"显形"

慢查询优化的前提是"知道慢在哪"，而不是凭感觉猜。MySQL 提供了慢查询日志（slow query log）功能，只要一条 SQL 的执行时间超过阈值，就会被记录下来。

SQL开启慢查询日志
-- 打开慢查询日志开关
SET GLOBAL slow_query_log = 1;

-- 设置慢查询阈值：超过 1 秒的 SQL 才会被记录（单位：秒，支持小数）
SET GLOBAL long_query_time = 1;

-- 查看日志文件路径
SHOW VARIABLES LIKE 'slow_query_log_file';

-- 关键参数：把"没用上索引"的查询也记下来，哪怕它没超过 long_query_time
SET GLOBAL log_queries_not_using_indexes = 1;

log_queries_not_using_indexes 很容易被忽略，但它是提前预警的利器：有些查询单次执行不慢（比如小表全表扫描只要几毫秒），但没走索引意味着数据量涨上来之后一定会变慢，这个参数能帮你在问题还小的时候就发现"没用索引"这个隐患。

常见误区：long_query_time 设得太小（比如默认的 10 秒改成 0.01 秒）会导致日志被大量正常查询淹没，反而找不到真正的重灾区；设得太大又会漏掉中等程度的慢查询。生产环境常见做法是先设一个适中值（如 0.5~1 秒）观察，再结合业务 SLA 微调。

第二步：用 pt-query-digest 找到"最该优化的那条"

慢查询日志文件本身只是流水账，一天下来可能有几万条记录。如果只挑一条看起来"最慢"的 SQL 去优化，很可能优化了一条一天只执行一次的报表查询，却放过了一条每秒执行几百次、单次只慢 200ms 但总耗时惊人的高频查询。 pt-query-digest（Percona Toolkit 里的一个工具）就是解决这个问题的：它会把慢查询日志里的 SQL 按查询指纹（query fingerprint）归类——把 SQL 里的具体参数值（如 WHERE id = 123 里的 123）替换成占位符，变成统一的"模板"， 这样同一类查询无论参数怎么变，都会被聚合到一起统计：出现次数、总耗时、平均耗时、最大耗时等等。

bashpt-query-digest 用法示例
# 分析慢查询日志，按总耗时降序输出 Top SQL 模板
pt-query-digest /var/lib/mysql/slow.log > digest_report.txt

# 报告里每个模板会给出：
#   Count（出现次数）
#   Exec time（总耗时 / 平均耗时 / 占比）
#   对应的 SQL 模板，比如：
#   SELECT * FROM orders WHERE user_id = ? AND status = ?
核心思路：优化的优先级应该是 总耗时 = 出现次数 × 平均耗时 最高的那一类查询，而不是单次执行时间最长的那一条。这才是"把力气花在刀刃上"，避免凭感觉猜哪条 SQL 该优化。

第三步：完整排查流程

找到目标 SQL 之后，一套可复用的排查流程是：

1. 发现慢查询（慢查询日志 + pt-query-digest 定位高优先级 SQL）
  ↓
2. EXPLAIN 分析执行计划（看 type / key / rows / Extra）
  ↓
3. 判断问题类型
    ├─ 缺索引 / 索引设计不合理（最左前缀没用好、索引失效）
    ├─ 触发 filesort（Extra: Using filesort）
    └─ 触发临时表（Extra: Using temporary，常见于 GROUP BY / DISTINCT）
  ↓
4. 加索引 或 改写 SQL（SELECT * 改覆盖索引 / 拆分大 OR 为 UNION）
  ↓
5. 重新 EXPLAIN 验证（type 是否变好、rows 是否大幅下降）
  ↓
6. 上线观察（持续跟踪慢查询日志，确认问题真正消失）

案例：一条全表扫描 SQL 的优化前后对比

假设有订单表，客服系统里有这样一条按用户查询未支付订单的 SQL：

SQL优化前
CREATE TABLE orders (
  id BIGINT PRIMARY KEY,
  user_id BIGINT,
  status TINYINT,
  amount DECIMAL(10,2),
  created_at DATETIME
);

EXPLAIN SELECT * FROM orders
WHERE user_id = 10086 AND status = 0;

-- 优化前 EXPLAIN 结果：
-- type: ALL          （全表扫描，最差的访问类型）
-- key: NULL          （没有用到任何索引）
-- rows: 1200000      （预计要扫描全表 120 万行）
-- Extra: Using where
SQL优化后
-- 分析：user_id 用于等值查询、status 也是等值查询，
-- 按最左前缀原则建联合索引 (user_id, status)
ALTER TABLE orders ADD INDEX idx_user_status (user_id, status);

EXPLAIN SELECT * FROM orders
WHERE user_id = 10086 AND status = 0;

-- 优化后 EXPLAIN 结果：
-- type: ref          （命中索引等值匹配，比 ALL 好得多）
-- key: idx_user_status
-- rows: 3            （精确命中，只需扫描 3 行）
-- Extra: NULL

如果这条查询列很多、又频繁被调用，还可以进一步把索引扩展为覆盖索引（呼应"覆盖索引"知识点），把常用的输出列也加进索引里，避免回表，把 Extra 从 NULL 进一步优化为 Using index。

指标优化前优化后
typeALL（全表扫描）ref（索引等值匹配）
keyNULLidx_user_status
rows（预估扫描行数）1,200,0003
ExtraUsing whereNULL（或 Using index，若为覆盖索引）

常见陷阱：不要无脑加索引

陷阱一：看到慢查询就加索引，是新手最容易犯的错误。索引不是免费的：每次 INSERT / UPDATE / DELETE 都要同步维护所有相关索引（写放大），索引本身还要占用磁盘空间和 Buffer Pool 内存。 加索引前应该先问：这条查询的调用频率高不高？是不是核心链路？如果是一天只跑一次的后台报表，哪怕它是全表扫描，也未必值得为它单独加一个索引，牺牲高频写入路径的性能。
陷阱二：EXPLAIN 里的 rows 是基于表的统计信息估算出来的，不是精确值。如果表刚做过大批量删改、统计信息严重过期，rows 可能被严重高估或低估，导致优化器选错执行计划。这时可以用 ANALYZE TABLE orders; 重新收集统计信息，往往能让优化器"重新想明白"该走哪个索引。

本 Tab 知识地图收尾

"索引与执行计划"这个 Tab 从底层到应用层，串起来的核心逻辑是：

  • 存储结构层：页结构与记录存储格式、B+树叶子节点双向链表——理解索引为什么天然有序、为什么支持范围查询。
  • 索引设计层：最左前缀原则、覆盖索引、索引失效的常见场景——理解怎么设计索引才能被优化器真正用上。
  • 执行计划层：EXPLAIN 的 type / key / rows / Extra——理解怎么读懂 MySQL 到底选了哪条路径。
  • 排序优化层：filesort 的成因与两种排序算法——理解排序为什么有时候能"白嫖"索引，有时候不能。
  • 工程实践层（本篇）：把以上所有知识串成一套可以在生产环境复用的排查流程，并且知道加索引也有成本，不能滥用。
