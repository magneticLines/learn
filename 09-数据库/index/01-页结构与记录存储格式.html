页结构与记录存储格式

在讲 B+ 树之前，必须先搞清楚 InnoDB 存数据的"最小砖块"长什么样——因为 B+ 树的每一个节点，本质上就是一个页。理解页结构，是理解索引一切设计的地基。

为什么按页读写，而不是按行？

磁盘是机械设备（即使是 SSD，也有类似的最小访问粒度），一次 I/O 的时间成本主要花在"寻址"上，真正传输数据反而很快。 如果每次只读一行记录，那访问 100 行数据就要触发 100 次昂贵的磁盘 I/O；但如果一次性读入一大块连续区域（一页）， 大概率会顺带把"相邻"的其他行也读进内存，后续访问这些行时就直接命中内存，不用再打磁盘。

InnoDB 正是基于这个原理，把磁盘和内存之间数据交换的最小单位设计成了页（Page），默认大小 16KB （由参数 innodb_page_size 控制）。不管你只想读一行，还是要读一万行，InnoDB 都是整页整页地往 Buffer Pool 里搬。

类比：这就像网购不会因为你只买一支笔就单独发一个快递，快递公司（磁盘）总是按"箱子"（页）为单位调度——把一箱子东西放的仓库离你近一点（Buffer Pool 缓存整页），比每次单独跑一趟顺丰次日达（单行 I/O）划算得多。

这也是为什么后面要讲的 B+ 树索引和 Buffer Pool 缓存机制都以"页"为基本操作单位——它们都是在"按页读写"这个硬件现实之上搭建的软件抽象。

InnoDB 数据页的七个组成部分

一个 16KB 的页并不是简单地把行记录平铺塞满，而是划分成七个功能区域：

InnoDB Page（16KB）
  ├─ File Header（38 字节：页号、上一页/下一页指针、页类型 ...）
  ├─ Page Header（56 字节：本页记录数、空闲空间起始位置 ...）
  ├─ Infimum + Supremum（两条虚拟记录，代表"最小/最大"边界）
  ├─ User Records（真正存放行记录的区域）
  ├─ Free Space（尚未使用的空闲空间，用链表管理已删除记录留下的空洞）
  ├─ Page Directory（记录指针的稀疏索引，加速页内二分查找）
  └─ File Trailer（8 字节：校验和，用于检测页是否写入完整/损坏）

几个关键点值得展开：

  • File Header 里的双向指针：每个页都记录着"上一页"和"下一页"的页号，这让同一层的所有数据页天然构成一个双向链表——这正是 B+ 树叶子节点能够高效支持范围查询、区间遍历的物理基础，下一篇会详细展开。
  • Infimum / Supremum 是两条固定存在的伪记录，分别表示"比本页任何真实记录都小"和"都大"，作为页内记录链表的头尾哨兵，简化边界判断逻辑。
  • Page Directory 把 User Records 中的记录按分组建立稀疏索引（每组 4~8 条记录一个槽位），页内查找先用二分法定位到大致的槽，再顺着单向链表逐条比较——避免了从第一条记录开始线性扫描整页。

COMPACT 行记录格式：一行数据是怎么存的

页里的 User Records 区域，实际存的就是一条条行记录。MySQL 8.0 默认使用 COMPACT（或其派生的 DYNAMIC）行格式， 每条记录物理上由三部分组成：

一条 COMPACT 记录
  ├─ 变长字段长度列表（倒序存储，仅统计 VARCHAR / TEXT 等变长列）
  ├─ NULL 值列表（bitmap，1 bit 标记一个可为 NULL 的列是否为 NULL）
  ├─ 记录头信息（5 字节：删除标记、记录类型、下一条记录相对偏移 ...）
  └─ 真实数据（各列的值，包括隐藏列 row_id / trx_id / roll_pointer）

为什么变长字段长度列表要"倒序"存储？

记录头信息里有一个指向"下一条记录"的相对偏移量，读取记录时是从记录头开始向后读真实数据的。而变长字段长度列表被存放在记录头的前面， 且是逆序排列（即先存最后一个变长列的长度，再存倒数第二个……）。这样设计是因为 CPU 从记录头向低地址方向读取变长长度列表时， 读到的顺序恰好和列的定义顺序一致，读取和寻址逻辑更简单、CPU 缓存友好，不需要额外跳转。

NULL 值列表为什么用 bitmap

假设一张表有 8 个可为 NULL 的列，NULL 值列表只需要 1 个字节（8 bit）就能表达"每一列是否为 NULL"， 而不是像某些天真的实现那样每列用一个标志字节（那样要花 8 个字节）。位图（bitmap）是一种极致的空间压缩手段：用一个 bit 表达一个布尔状态， 在动辄百万行的表上，这种"抠字节"的设计能显著节省整体存储空间——毕竟省下来的空间意味着一页能装更多行，间接减少了树的高度和 I/O 次数。

一个例子：拆解一条记录的存储

schema.sqlMySQL 8.0
CREATE TABLE user (
  id INT UNSIGNED PRIMARY KEY,
  name VARCHAR(20) NOT NULL,   -- 变长列，会进长度列表
  age TINYINT NULL,                -- 定长列，但可为 NULL，占 NULL 位图 1 bit
  bio TEXT NULL                  -- 变长列 + 可为 NULL
);

-- 插入一行：age 为 NULL，bio 未提供（也是 NULL）
INSERT INTO user(id, name) VALUES (1, 'Tom');

这条记录在磁盘上大致是：变长长度列表记 name=3（'Tom' 占 3 字节，bio 为 NULL 不记入变长列表）； NULL 值列表用 2 bit 标记 age=NULL, bio=NULL；记录头 5 字节；真实数据区只存 id 和 name 的值 （加上 InnoDB 自动追加的隐藏列 row_id、trx_id、roll_pointer，用于聚簇索引和 MVCC，会在后续文章展开）。 可以看到，NULL 值不占用真实数据空间，这也是 InnoDB 存储层面对 NULL 的一种优化。

常见陷阱

陷阱一：建表时可变长字段（VARCHAR/TEXT）过多。 每多一个变长列，就要在变长字段长度列表里多记一份长度信息，而且变长列的存在会让 InnoDB 无法用简单的"列偏移量数组"直接定位某一列 （定长列可以，变长列必须先扫过前面所有变长列的长度才能算出偏移）。过多变长列不仅增加解析开销，还容易导致行长度逼近甚至超过页大小的 一半，从而引发下面的"行溢出"。
陷阱二：行溢出（Row Overflow）。 InnoDB 要求一个页至少能存放 2 条记录（否则 B+ 树就退化成链表，失去树形结构的意义）。 如果一行数据（比如塞了一个几十 KB 的大 TEXT/BLOB）撑爆了这个限制，InnoDB 会把这一列的内容转移到额外分配的 溢出页（Overflow Page）中，原记录处只保留一个指向溢出页的指针（20 字节）。这意味着访问这一行会触发额外的 I/O ——大字段应尽量拆表存放，或用外部存储（如对象存储）+ 数据库只存 URL 的方式规避。
面试高频："MySQL 一行最大能存多少数据？为什么单行长度有 65535 字节的限制？" 这个限制其实是 MySQL Server 层对 "单行所有列定义长度之和"的静态检查上限，与页大小无关；而"行溢出"讨论的是运行时数据实际写入超过页容量后的处理方式，二者是不同层面的问题， 面试时容易被绕进去，回答时要分清楚。

核心价值总结

  • 页是磁盘 I/O 的最小单位：一次 I/O 顺带读入相邻数据，摊薄随机访问的寻址成本，是 Buffer Pool、B+ 树等一切上层设计的物理基础。
  • 页内七大区域各司其职：File Header 的双向指针串联页与页，Page Directory 加速页内查找，File Trailer 保证写入完整性。
  • COMPACT 行格式用变长列表 + NULL 位图做极致的空间压缩，直接影响一页能装多少行，进而影响 B+ 树的高度——这是下一篇的核心话题。
