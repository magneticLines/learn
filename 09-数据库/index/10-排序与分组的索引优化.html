排序与分组的索引优化(filesort)

ORDER BY / GROUP BY 到底能不能"白嫖"索引的有序性？答案取决于索引的物理顺序和你想要的逻辑顺序是否一致——不一致，MySQL 就得掏钱做一次额外排序，这就是 filesort。

为什么索引能省掉排序这一步

上一个知识点（页结构与记录存储格式）讲过：InnoDB 的 B+树索引，叶子节点之间用双向链表连接，而且链表是按索引列的值严格有序排列的——这是 B+树为了支持范围查询而设计的物理结构，不是偶然。 既然叶子节点本身就是一条有序链表，那么如果你要的排序结果，恰好和某个索引的顺序一致，MySQL 完全没必要另起炉灶排序——直接顺着链表往后读就是有序的结果集，一分钱排序成本都不用花。

类比：这就像图书馆的书架已经按书名拼音排好了（索引本身有序）。你要"按书名顺序列出所有书"，直接从书架第一本走到最后一本即可；但如果你想"按借阅次数排序"，书架帮不上忙，你只能把书全搬到桌子上（sort buffer）重新排一遍——这就是 filesort。

什么情况可以避免 filesort

核心条件只有一句话：排序列必须是索引的（前缀）列，且访问索引的顺序与索引本身的物理顺序一致。展开成几个具体场景：

  • 排序列就是单列索引本身，比如索引 idx_age(age)，执行 ORDER BY age。
  • 排序列是联合索引的最左前缀，且排序方向与索引列声明顺序一致（或者全部反向，MySQL 8.0 支持倒序索引和降序扫描优化）。
  • 排序过程中没有被其他条件打断顺序性——这是最容易踩坑的地方，下面详细展开。

例子：联合索引 (a, b) 消除 filesort

SQL可避免 filesort
CREATE TABLE t_order (
  id BIGINT PRIMARY KEY,
  a INT,
  b INT,
  c VARCHAR(32),
  KEY idx_a_b (a, b)
);

-- a、b 都是联合索引 idx_a_b 的列，且顺序与索引声明一致
-- 直接沿着 idx_a_b 叶子链表顺序读，天然有序，无需 filesort
SELECT id, a, b FROM t_order
ORDER BY a, b;

用 EXPLAIN 看这条 SQL，Extra 列不会出现 Using filesort，因为 idx_a_b 的叶子节点本来就是先按 a 排、a 相同再按 b 排，和 ORDER BY a, b 要的结果完全一致。

反例：范围查询打断顺序性

SQL无法避免 filesort
-- a 用了范围查询(>)，联合索引 (a, b) 在 a 值不确定的区间里，
-- b 列并不是整体有序的（只在每个相同 a 值内部按 b 有序）
SELECT id, a, b FROM t_order
WHERE a > 1
ORDER BY b;
-- EXPLAIN 的 Extra 会出现 Using index condition; Using filesort

这正好呼应最左前缀原则的另一面：最左前缀讲的是"WHERE 条件里范围查询会让右边的列失去索引效果"，这里则是同一个原理在排序场景下的体现——范围条件破坏的不仅是等值查找的能力，也破坏了后续列的有序性。 因为 a > 1 覆盖了 a 的一大段取值范围，索引里 b 只是在"每个 a 值内部"有序，跨越多个 a 值之后 b 整体是乱的，MySQL 没法只靠顺序扫描给出结果，只能老老实实 filesort。

常见误区：很多人以为"只要 ORDER BY 的列上有索引就不会 filesort"，这是不严谨的。索引存不存在只是必要条件，访问路径是否保持了索引的物理有序性才是充分条件。如果 WHERE 用了其他索引、或者对索引列做了范围查询、或者排序列不是索引的前导列，都可能导致 filesort 照样发生。

filesort 到底是怎么排的：两种算法

既然叫"file"sort，很容易望文生义以为一定要写磁盘文件，其实不一定——只有当排序数据量超过 sort_buffer_size 时才会用到磁盘临时文件做归并排序。MySQL 内部对 filesort 有两种实现方式：

双路排序（Two-pass，老算法）

第一次扫描：只把 排序字段 + 主键值 读入 sort buffer，在内存里排好序；第二次扫描：按排序好的顺序，用主键回表再取出 SELECT 需要的其他列。 好处是 sort buffer 里只放两列数据，内存占用小；坏处是要扫描两遍数据（一遍排序、一遍回表），如果回表是随机 I/O，代价不小。

单路排序（One-pass，MySQL 4.1 之后默认）

一次性把 SELECT 需要的所有列都读进 sort buffer 排序，排完直接返回，不用二次回表。好处是 I/O 次数少（少了一次回表扫描）； 坏处是如果一行数据很宽（列多、字段大），sort buffer 装不下所有需要排序的行，就只能分成多个批次，每批在内存里排好后写一个临时文件，最后再对多个有序的临时文件做归并排序——这个过程既占用磁盘 I/O，又比双路排序更慢，完全违背了"单路更快"的设计初衷。

对比维度双路排序（Two-pass）单路排序（One-pass）
sort buffer 存什么 排序字段 + 主键 SELECT 需要的全部列
磁盘 I/O 次数 两次全表相关扫描（含一次回表） 一次扫描，但 buffer 不够时要多轮归并临时文件
内存占用 小 大（尤其列多、字段宽时）
最佳场景 sort_buffer_size 有限、SELECT 列很宽 sort_buffer_size 足够大、行数据不算宽
最差场景 回表随机 I/O 代价高 buffer 装不下，触发多次磁盘归并，反而更慢

sort_buffer_size：排序缓冲区大小

这个参数决定了 MySQL 给每个需要排序的连接分配多大的内存空间用来排序。太小会导致排序数据装不下，被迫拆分成多个批次分别排序后再归并（即"多路归并排序"），产生额外的磁盘临时文件读写； 调大能让更多排序在纯内存里一次完成，减少或消除归并过程。但它不是全局共享的，是每个线程独立分配的，一律调得很大反而可能把服务器内存打爆，需要结合并发连接数权衡。

SQL判断是否需要调大 sort_buffer_size
-- 1. 查看当前排序缓冲区大小
SHOW VARIABLES LIKE 'sort_buffer_size';

-- 2. 关键：查看是否发生了"排序合并次数"，大于 0 说明有临时文件归并
SHOW GLOBAL STATUS LIKE 'Sort_merge_passes';

-- 3. 慢查询日志里如果频繁看到 Sort_merge_passes 增长，
--    并且 Extra 里出现 Using filesort，就是调大 sort_buffer_size 的信号
实战判断法：不要拍脑袋调参数。先看慢查询日志 + EXPLAIN 的 Extra: Using filesort，再对比 Sort_merge_passes 这个状态变量在压测前后的增量。如果这个值持续增长，说明排序数据经常超出 sort_buffer_size，产生了多路归并，这时调大 sort_buffer_size 才是对症下药；如果 Sort_merge_passes 几乎不变，盲目调大只是浪费内存。

和前面知识点的联系

  • 呼应"页结构与记录存储格式"：filesort 能否避免的根本原因，就是 B+树叶子节点双向链表天然有序这一物理特性能否被直接利用。
  • 呼应"最左前缀原则"：范围查询打断联合索引的有序性，既会导致索引失效（无法利用后续列做等值过滤），也会导致排序失效（无法利用后续列做排序），是同一个"有序性被破坏"问题的两种表现。
  • 为下一篇"慢查询定位"埋伏笔：看到 Extra: Using filesort 或 Using temporary，是判断一条慢 SQL 需不需要加索引/改写的重要信号之一。

核心价值总结

  • 能不排序就不排序：排序本身是重操作，能靠索引物理有序性省掉这一步，性能收益往往比加机器更明显。
  • filesort 不等于"排序慢"：它只是"MySQL 自己额外做了一次排序"的统称，内存足够时未必慢；真正要警惕的是触发了磁盘临时文件归并的 filesort。
  • 诊断优先于调参：先用 EXPLAIN 和 Sort_merge_passes 确认问题存在，再决定加索引还是调 sort_buffer_size，不要本末倒置。
