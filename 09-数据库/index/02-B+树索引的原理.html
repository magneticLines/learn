B+ 树索引的原理

索引的本质，是用一种额外的数据结构，把"全表扫描"的 O(n) 查找变成 O(log n)。MySQL 选择了 B+ 树，而不是二叉树、B 树或哈希表——这背后是一整套针对磁盘 I/O 特性做出的权衡。

为什么不用二叉查找树？

二叉查找树（BST）每个节点只有两个子节点，如果数据是顺序插入的（比如自增 id），树会退化成一条链表，查找复杂度从 O(log n) 退化到 O(n)。 即使是平衡二叉树（如红黑树），高度依然是 O(log₂ n)——面对百万级数据，树高也要 20 层左右。

问题的关键在于：树的每一层通常对应一次磁盘 I/O（因为节点太大装不进内存，需要从磁盘读取）。20 次随机 I/O 相当于几十毫秒的延迟， 对于要求毫秒级响应的数据库查询来说是不可接受的。所以数据库索引结构的首要目标是压低树的高度，二叉树"一个节点两个分支"的设计天生就很"高瘦"，不适合。

B 树 vs B+ 树：多叉平衡树的两种设计

B 树通过让每个节点存储多个键值、多个子指针（即"多叉"），大幅降低了树高。但 InnoDB 选择的是 B 树的变种——B+ 树， 两者的核心差异在于：

特性B 树B+ 树
数据存放位置 每个节点（包括非叶子节点）都存索引键 + 完整数据 只有叶子节点存完整数据，非叶子节点只存索引键 + 子节点指针
非叶子节点容量 因为要存数据，单个节点能容纳的键值数量较少 不存数据，单个节点能塞下更多键值和指针，树更矮更"胖"
范围查询 需要对树做中序遍历，反复回退父节点，效率低 所有叶子节点通过双向链表相连，范围查询只需定位起点后顺着链表扫
查询稳定性 数据可能落在任意层，查询耗时不稳定 所有数据都在叶子层，每次查询的 I/O 次数几乎相同（等于树高）

正是因为非叶子节点不存数据、只存"路标"，B+ 树的扇出（fan-out，即每个节点的子节点数量）比 B 树大得多， 树高也就压得更低。这一点还与上一篇讲的"页"直接挂钩：InnoDB 把 B+ 树的每一个节点设计成刚好对应一个数据页（16KB）， 页装得下多少个"键值+指针"，就决定了这棵树的扇出有多大。

类比：B 树像一本每页都夹杂着"目录条目 + 正文内容"的书，翻页效率低；B+ 树则像"前面几页纯目录（只有页码索引），后面才是正文，且正文页之间还用书签串成一条线"——查目录能塞下更多条目（矮），翻到正文后还能一直往后翻读完整个区间（范围查询友好）。

为什么不直接用哈希表？

哈希索引的等值查询（WHERE id = 5）确实是 O(1)，比 B+ 树的 O(log n) 更快。但哈希表不保存数据的顺序关系， 无法支持范围查询（WHERE id > 5）、排序（ORDER BY）、前缀匹配（LIKE 'abc%'）——这些都要求数据按序可遍历。 InnoDB 内部的 Adaptive Hash Index 会针对高频等值查询自动在内存里建哈希索引做加速，但这只是 B+ 树之上的一层"缓存优化"，不能替代 B+ 树作为主体结构。

估算一棵三层 B+ 树能存多少数据

以 InnoDB 默认页大小 16KB、主键为 BIGINT（8 字节）、指针占 6 字节为例，简化模型如下：

第 1 层（根节点，1 个页）
  约 16KB / (8+6)字节 ≈ 1170 个指针，指向 1170 个第 2 层节点
第 2 层（非叶子节点，1170 个页）
  每个节点再指向约 1170 个叶子节点 → 共 1170 × 1170 ≈ 137 万个叶子节点
第 3 层（叶子节点，137 万个页）
  每页约能存 16KB / 1KB(假设单行 1KB) ≈ 16 行 → 总数据量 ≈ 137万 × 16 ≈ 2192 万行

这就是为什么常说"一棵三层的 B+ 树足以支撑千万级数据量，且查询一条记录只需要 3 次磁盘 I/O"——而这 3 次 I/O 里， 根节点和第二层节点通常因为被频繁访问，早已被 Buffer Pool 缓存在内存中，实际只有最后一次读叶子节点才真正落盘，效率非常高。

常见陷阱

陷阱一：索引列越长，树越"矮胖"的效果越差。 上面的估算能看到，非叶子节点能容纳的指针数量 = 页大小 / (索引键长度 + 指针长度)。如果索引列是一个 VARCHAR(255) 的长字符串， 单个键值可能就占几十甚至上百字节，一个页能塞的指针数量骤降，扇出变小，树就会变"高瘦"，查询所需的 I/O 次数随之增加。 这也是为什么实践中提倡索引列尽量短、必要时用前缀索引（INDEX(name(10))）。
陷阱二：为什么自增主键比 UUID 更适合做聚簇索引？ InnoDB 的主键索引（聚簇索引，下一篇详细展开）要求叶子节点里的记录按主键值physically排序存放。 自增主键的插入永远发生在最后一页的末尾，属于顺序追加，不会导致页分裂；而 UUID 这类随机值会导致新记录随机插入到已排满的中间页， 频繁触发页分裂（Page Split）——不仅写入变慢，还会产生大量存储碎片，让树的实际利用率下降。
面试高频："为什么 MySQL 索引要用 B+ 树而不是 B 树/红黑树/哈希表？" 标准答案要点：① 非叶子节点不存数据，扇出更大，树更矮，减少磁盘 I/O 次数；② 叶子节点用链表相连，天然支持范围查询和排序；③ 每次查询的 I/O 次数稳定（都要走到叶子层），不像 B 树数据可能分布在任意层导致耗时不稳定。

核心价值总结

  • B+ 树的核心权衡：牺牲非叶子节点的数据存储能力，换取更大的扇出、更矮的树高，本质是在为"减少磁盘 I/O 次数"这个目标服务。
  • 叶子节点的双向链表是范围查询、排序高效的关键，这一点是 B 树和哈希表都不具备的。
  • 索引列的长度直接影响树的高度：列越短，一页能塞的键值越多，树越矮，这是索引设计中"短索引优先"的理论依据。
