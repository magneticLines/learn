联合索引与最左前缀原则

联合索引不是"给每一列都单独建了个索引"，而是把多列拼成一个复合排序键的一棵B+树——理解了这个物理事实，最左前缀原则就不再是需要死记的规则，而是显而易见的推论。

联合索引的物理本质：一棵按复合键排序的 B+树

假设有表 user(id, a, b, c)，建立联合索引 INDEX idx_a_b_c (a, b, c)。这棵 B+树的每个叶子节点存的是 (a, b, c, 主键id) 这样的复合值，排序规则是：先按 a 排序；a 相同的记录，再按 b 排序；a、b 都相同的记录，再按 c 排序。

idx_a_b_c 叶子节点排序示意（按 a,b,c 字典序）
-- (a, b, c) 三元组按字典序排列，注意 b 只在 a 相同时才有序，c 只在 a,b 都相同时才有序
(1, 1, 2)
(1, 2, 1)
(1, 2, 3)
(2, 1, 1)
(2, 1, 5)
(2, 2, 0)

观察上面的数据你会发现一个关键事实：整体上 c 列其实是无序的（比如全表看 c 依次是 2,1,3,1,5,0，毫无规律）。 c 只有在"a 和 b 都已经固定"的小范围内才是有序的。这就是"层层嵌套排序"——每一层排序都是在上一层排序结果相等的前提下才成立的局部有序，而不是全局有序。

生活类比：联合索引就像一本按"姓氏拼音 → 名字拼音 → 出生日期"排序的电话簿。你可以快速翻到"王"姓开头的所有人（用上第一列），也能在"王"姓范围内快速定位"王小明"（用上前两列）；但如果你想直接查所有"出生日期是 5 月 1 日"的人，对不起，这本电话簿对你毫无帮助——因为出生日期只在姓名都锁定后才排序，全局是乱序的。你只能从头翻到尾，逐条比对。

最左前缀原则：为什么必须从最左列开始

B+树的检索依赖"有序"才能进行范围收缩（每一层根据大小关系决定往左还是往右走）。既然 b、c 列的有序性是建立在前面的列已经确定相等的基础上， 那么一旦查询条件跳过了最左列（或最左列不是等值条件），后面的列在树里就退化成无序数据，B+树的比较、收缩逻辑完全用不上，只能退化为全表扫描式的比对。

查询条件能否用上 idx_a_b_c说明
WHERE a = 1 能，用到 a 一列 a 是最左列，直接可以做范围定位
WHERE a = 1 AND b = 2 能，用到 a、b 两列 a 等值锁定范围后，b 在该范围内依然有序
WHERE a = 1 AND b = 2 AND c = 3 能，a、b、c 全部用上 三列均为等值条件，索引查找可以精确定位
WHERE a = 1 AND b > 2 AND c = 3 只能用到 a、b，c 用不上 b 是范围查询，一旦发生范围查询，后面列的有序性立刻中断（见下文重点）
WHERE b = 2 不能，无法使用该索引做定位 跳过最左列 a，b 在全表范围内无序，只能退化为扫描
WHERE c = 3 不能 同上，c 更靠后，全局更加无序
WHERE b = 2 AND c = 3 不能 缺少最左列 a，b、c 组合依然全局无序
重点陷阱：范围查询会中断最左前缀匹配。 WHERE a = 1 AND b > 2 AND c = 3 这个例子极容易被忽视——很多人以为"a、b、c 都在 WHERE 里出现了，索引应该能全用上"。 但事实是：MySQL 在索引树里定位完 a = 1 后，进入 b 这一层做范围扫描（b > 2），此时游标要在满足 a=1 的这一段叶子节点里从 b=2 往后扫描， 但这些被扫描到的记录中 c 已经不是全局有序的了（因为 b 在变化，c 只在 a、b 都固定时才有序）。 所以 c = 3 这个条件无法通过索引定位，只能对扫描到的每一行在索引里额外判断一次（这部分可以靠下一篇讲的"索引下推"优化，但索引本身的有序定位能力到 b 就截止了）。

联合索引列顺序怎么设计

既然最左列决定了索引能否被命中、命中后能收缩多大范围，列的顺序设计就直接影响索引的使用效率，通常遵循两个原则：

  • 区分度（选择性）高的列放前面：比如 status（只有几个取值）和 user_id（几乎每行都不同）组合成索引，user_id 放最左边能更快地把候选行数收缩到很小范围，status 放最左边则收缩效果很差。
  • 常用作等值查询（=）的列放前面，范围查询（>、<、BETWEEN、LIKE 前缀%）的列放后面：因为范围查询一旦出现，后面所有列的有序性都会被打断（如上文重点所述），所以要把范围条件尽量往后放，让更多的等值列排在前面参与索引收缩。
列顺序对比示例
-- 场景：查询 "某用户在某个时间段之后创建的订单，且状态为已支付"
-- 反例：范围列 created_at 放在前面，等值列 status 放在后面用不上
CREATE INDEX idx_bad ON orders(created_at, status, user_id);

-- 正例：等值列 user_id、status 放前面，范围列 created_at 放最后
CREATE INDEX idx_good ON orders(user_id, status, created_at);

SELECT * FROM orders
WHERE user_id = 1001 AND status = 'PAID' AND created_at > '2026-01-01';
-- idx_good：user_id、status 两个等值条件都能参与索引定位，只有 created_at 做范围扫描，收缩效果好
-- idx_bad：created_at 是最左列且是范围查询，后面的 status、user_id 全部用不上索引的有序性

与相邻知识点的联系

最左前缀原则决定了"能不能用上索引、用到哪一列为止"，而索引下推（ICP）（下一篇）解决的是"用不上索引定位、但索引里其实已经有这一列数据"时如何减少不必要的回表—— 就是上面 WHERE a=1 AND b>2 AND c=3 例子里 c 这一列的场景。再往后的覆盖索引（第 6 篇）则是最左前缀原则的延伸应用：即使某些列没法参与索引定位，只要它们存在于索引里，就可能省去回表这一步。

面试高频："最左前缀原则中的'最左'具体指什么？" 很多人会误答成"where 条件里写的顺序"，正确理解是指联合索引定义时列的顺序，与 SQL 里 WHERE 子句写 b=2 AND a=1 还是 a=1 AND b=2 无关——MySQL 优化器会自动调整条件顺序去匹配索引定义顺序。真正决定能否命中的，是索引定义 (a, b, c) 里"从 a 开始连续出现"这几列有没有在 WHERE 条件里都给出。

核心价值总结

  • 联合索引是一棵树，不是多棵单列索引的简单叠加：排序是层层嵌套的，后面列的有序性依赖前面列已经确定。
  • 最左前缀原则的根源是 B+树的物理排序结构，不是人为规定的规则——理解了这一点就不需要死记硬背命中规则表。
  • 范围查询会截断后续列的索引可用性，这是设计联合索引列顺序时最容易被忽视的一点。
