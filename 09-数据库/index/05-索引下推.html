索引与执行计划

索引下推(ICP)

上一篇提到"范围查询会截断最左前缀匹配，后面的列用不上索引定位"——但"用不上定位"不等于"索引里没有这份数据"。索引下推做的事情，就是把这份被浪费的数据利用起来，尽量少走一次回表。

没有 ICP 之前：白白多走的回表

假设有联合索引 INDEX idx_name_age (name, age)，执行查询：

示例查询
SELECT * FROM user
WHERE name LIKE '张%' AND age = 20;

这里 name LIKE '张%' 是一个前缀匹配，本质上是一个范围查询（等价于 name >= '张' AND name < '张'的下一个字符）。 按照上一篇讲的最左前缀原则：a 列（这里是 name）参与范围查询后，后面的 age 列在索引树里已经不再全局有序，无法通过索引直接定位。

在 MySQL 5.6 引入 ICP 之前，存储引擎（InnoDB）的做法很"老实"：只要 name 满足 LIKE '张%'，不管 age 是多少， 先把这条记录对应的主键取出来，回表到聚簇索引查出完整行，再把整行数据交给 Server 层，由 Server 层根据 WHERE 条件里的 age = 20 做二次过滤，不满足的就丢弃。

没有 ICP：
  存储引擎层：遍历 idx_name_age，找到所有 name LIKE '张%' 的记录 → 逐条回表取完整行
  Server 层：拿到所有"姓张"的完整行 → 再用 age = 20 过滤 → 返回结果
  问题：如果姓张的有 10000 人，但只有 50 人是 20 岁，白白多做了 9950 次不必要的回表

有了 ICP 之后：在索引遍历阶段就地过滤

索引下推（Index Condition Pushdown）的核心思路是：age 虽然不能参与索引的"定位"（不能用来缩小 B+树的查找范围），但它的值本来就实实在在地存在于 idx_name_age 这棵索引树的叶子节点里 （联合索引的叶子节点存的是 (name, age, 主键id)）。既然数据已经在手边，为什么不直接在存储引擎层就把 age = 20 这个条件判断掉， 而非要大老远回表取出整行、交给 Server 层才做这件事？

有了 ICP，执行流程变成：存储引擎在遍历 name LIKE '张%' 命中的每一条索引记录时，顺手检查一下该记录里的 age 值是否等于 20， 只有 name、age 都满足条件的记录，才会拿着主键去聚簇索引回表；不满足 age 条件的，直接在索引层跳过，连回表的机会都没有。

有 ICP：
  存储引擎层：遍历 idx_name_age，找到 name LIKE '张%' 的记录 → 就地判断 age = 20 → 只有都满足的才回表
  Server 层：直接拿到已经完全符合条件的行，无需二次过滤
  效果：10000 个姓张的人里最终只回表 50 次，而不是 10000 次
类比：没有 ICP 就像去仓库找"姓张的顾客"的完整档案——保安只看得懂门口登记簿上的姓名栏，凡是姓张的一律把整份档案（回表）搬出来，再由前台工作人员（Server 层）翻开档案看年龄是否等于 20，不符合的扔掉。 有了 ICP，相当于登记簿上除了姓名，年龄也顺手记了一笔——保安直接在登记簿上比对完姓名和年龄，只有两项都符合的人，才值得搬一次档案出来。

如何确认 ICP 生效：EXPLAIN 的 Extra 列

用 EXPLAIN 查看执行计划，如果 Extra 列出现 Using index condition，说明 ICP 被用上了：

EXPLAIN 验证 ICP
EXPLAIN SELECT * FROM user
WHERE name LIKE '张%' AND age = 20;

-- 关键结果列（节选）：
-- key: idx_name_age
-- Extra: Using index condition   -- 说明 age 条件在索引遍历阶段就已经被下推判断

对比一下：如果把 age 换成一个索引里不存在的列（比如 email = 'xxx'），由于 email 不在 idx_name_age 里，存储引擎层无法就地判断， 这个条件就只能等到回表拿到完整行之后，在 Server 层通过 Using where 来过滤——这时 Extra 列会同时出现 Using index condition（针对 age）和 Using where（针对 email 这类索引外的列），两者并不互斥。

ICP 的适用范围

条件说明
只能用在二级索引上 聚簇索引的叶子节点本身就是完整行数据，遍历聚簇索引不存在"要不要回表"的问题，ICP 无从谈起。
只能处理"索引列本身能判断"的条件 下推的条件必须是索引里已经包含的列，且判断逻辑不依赖索引之外的数据（比如不能是需要访问 TEXT/BLOB 大字段之类不在索引里的列）。
需要优化器选择该策略 ICP 是否启用受 optimizer_switch 中的 index_condition_pushdown 开关控制，MySQL 5.6+ 默认开启。
常见误解：不少人以为"只要 WHERE 条件里的列都在联合索引里，就一定能走 ICP 全部命中"。但 ICP 不能让本该无序的部分重新变得有序—— 它优化的是"减少回表次数"，而不是"让索引定位范围变得更精确"。也就是说，即使 ICP 生效，存储引擎依然需要把 name LIKE '张%' 命中的所有记录逐条扫描一遍 （只是省去了逐条回表的开销），扫描的行数并没有减少，减少的只是回表这一步的次数。

与相邻知识点的联系

ICP 直接承接上一篇「联合索引与最左前缀原则」里提到的问题：范围查询打断最左前缀匹配后，后面的列本来就"英雄无用武之地"，ICP 给了它们一个新用途——就地过滤减少回表。 而下一篇要讲的覆盖索引是这一思路的极限形态：如果查询需要的所有列（不只是用来过滤的列）都已经在二级索引里，那么连"回表"这个动作本身都可以彻底省略，比 ICP 更进一步。

核心价值总结

  • ICP 解决的问题：联合索引中最左前缀之后、无法用于定位但仍存在于索引里的列，如何避免"先回表、再过滤"的浪费。
  • 生效标志：EXPLAIN 的 Extra 列出现 Using index condition。
  • 能力边界：只减少回表次数，不改变索引扫描的行数；只能用在二级索引上，且条件必须能通过索引列本身判断。
