索引失效的常见场景

索引建了，但查询该慢还是慢——大多数时候不是索引没建对，而是 SQL 写法让优化器根本用不上索引。理解每种失效场景背后的原理，才能从"背口诀"升级到"能推理"。

为什么索引会"失效"：一句话原理

B+ 树索引能加速查询的根本前提是：索引里存储的键值，是按原始列值排好序的，优化器可以利用这种有序性做二分查找或范围扫描。 一旦 SQL 的写法让"要查找的值"和"索引里存的原始有序值"对不上号（比如对列做了运算、类型不匹配、通配符打乱了前缀顺序）， B+ 树的有序性就用不上了，优化器只能放弃索引，退化为全表扫描（对应 EXPLAIN 中 type=ALL）。

生活类比：索引就像一本按姓氏拼音排好序的通讯录。如果你说"帮我查一个姓名倒过来读是'某某张'的人"，接线员没法直接用这本按拼音排序的册子二分查找，只能从头翻到尾一个个核对——这就是对索引列做"函数运算"之后索引失效的本质。

场景一：对索引列做函数运算

索引存储的是列的原始值，一旦对列施加函数（如 YEAR()、UPPER()、DATE_FORMAT()）， 结果值已经和索引里存的原始有序序列脱钩了，B+ 树没有办法按"变换后的值"做有序定位，只能对每一行都算一次函数值再比较——也就是全表扫描。

SQL · 错误写法（索引失效）
-- create_time 上有索引，但被 YEAR() 包裹后索引失效
SELECT * FROM orders WHERE YEAR(create_time) = 2024;
SQL · 正确写法（索引生效）
-- 改写成范围查询，不对索引列做任何运算
SELECT * FROM orders
WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';

场景二：隐式类型转换

当列的类型是 VARCHAR，但查询条件传入了一个数字，MySQL 会按照类型转换规则，把 字符串转换为数字来比较（而不是把数字转成字符串）。 这本质上等价于对索引列做了 CAST(phone AS SIGNED) 这样的隐式函数运算，同样会导致索引失效。

SQL · 错误写法（索引失效）
-- phone 是 VARCHAR 类型，但传入的是数字字面量
SELECT * FROM users WHERE phone = 13800000000;
SQL · 正确写法（索引生效）
SELECT * FROM users WHERE phone = '13800000000';

场景三：!= 或 <> 导致索引失效

!=（不等于）在语义上意味着"排除一个值，剩下的全都要"，优化器估算这类条件通常会匹配到表中的大部分行， 走索引再逐一回表的代价往往比直接全表扫描更高，因此优化器经常直接放弃索引。这不是"语法不支持"，而是代价评估后的主动选择。

SQL · 容易失效
SELECT * FROM orders WHERE status != 'CANCELLED';
缓解思路：如果确实能预估结果集很小（比如 CANCELLED 占了 99% 的订单），可以改写为等价的 IN 或范围条件，帮助优化器更准确地判断选择性，例如 WHERE status IN ('PENDING', 'PAID', 'SHIPPED')。

场景四：LIKE 前导通配符

这一条呼应"最左前缀匹配"的核心思想：索引的有序性是按字符串从左到右比较建立的。LIKE 'xxx%'（后置通配符）等价于一个范围查询—— "找出所有以 xxx 开头的值"，天然符合前缀有序的特性，可以走索引；而 LIKE '%xxx'（前导通配符）要求"值的末尾是 xxx"， 这和按前缀排序的索引完全对不上，只能全表扫描逐行做字符串匹配。

SQL · 错误写法（索引失效）
SELECT * FROM users WHERE name LIKE '%明';
SQL · 正确写法（索引生效）
SELECT * FROM users WHERE name LIKE '张%';

场景五：OR 连接了没有索引的列

OR 两侧任意一个条件命中的行都要返回。如果其中一侧的列没有索引，优化器就必须对这一侧做全表扫描， 既然都要扫全表了，索引带来的加速对整条查询已经没有意义，优化器索性放弃另一侧本可用的索引，整条语句退化为全表扫描。

SQL · 错误写法（索引失效，假设 email 无索引）
SELECT * FROM users WHERE id = 100 OR email = 'a@x.com';
SQL · 正确写法一（给 email 也建索引）
-- OR 两侧都有索引时，优化器可能使用 index_merge 策略分别走索引再合并结果
ALTER TABLE users ADD INDEX idx_email (email);
SQL · 正确写法二（用 UNION 改写）
SELECT * FROM users WHERE id = 100
UNION
SELECT * FROM users WHERE email = 'a@x.com';

场景六：联合索引未按最左前缀使用

联合索引 (a, b, c) 在 B+ 树里是先按 a 排序，a 相同的情况下再按 b 排序，以此类推—— 这意味着只有查询条件从最左边的 a 开始连续使用，索引的有序性才能被利用。跳过 a 直接用 b、c 查询， 等于让优化器在一堆"不知道从哪开始比较"的数据里查找，索引用不上。

SQL · 错误写法（跳过最左列 a，索引失效）
-- 联合索引 idx_a_b_c (a, b, c)
SELECT * FROM t WHERE b = 2 AND c = 3;
SQL · 正确写法（从最左列开始）
SELECT * FROM t WHERE a = 1 AND b = 2 AND c = 3;

场景七：字符集/排序规则不一致

多表关联时，如果两张表用于 JOIN 的列字符集或排序规则（collation）不一致（例如一个是 utf8mb4_general_ci，另一个是 utf8mb4_bin）， MySQL 需要先做隐式字符集转换才能比较，这同样属于对索引列做了"运算"，会导致相关列上的索引失效。生产实践中建议同一业务库的字符集与排序规则保持统一，避免这类隐性问题。

失效场景速查表

场景失效原因规避方式
索引列做函数运算索引存原始值，运算后值与有序序列脱钩把运算移到条件右侧，改写成范围查询
隐式类型转换等价于对列做了 CAST 运算保证字面量类型与列类型一致（字符串加引号）
!= / <>优化器估算匹配行数过多，代价高于全表扫描能确定小结果集时改写为 IN 或范围条件
LIKE '%xxx'前导通配符破坏了按前缀比较的有序性尽量用 LIKE 'xxx%'，或引入全文索引/搜索引擎
OR 含无索引列其中一侧必须全表扫描，索引失去意义给相关列都建索引，或用 UNION 改写
联合索引未按最左前缀跳过最左列，无法利用树的排序层级调整 WHERE 顺序或索引列顺序，从最左列开始用
字符集/排序规则不一致需要隐式转换字符集再比较统一相关表列的字符集与 collation
陷阱：索引失效不代表 SQL 写错了语法，它完全可以正常返回结果，只是执行效率退化——这也是它更隐蔽、更容易被忽视的原因。定位这类问题的第一反应应该是跑一遍 EXPLAIN，看 type 是否退化为 ALL，key 是否为 NULL。
面试高频："哪些情况会导致索引失效？" 建议按"对索引列做了运算/类型转换" "OR 破坏了可用性" "前导通配符破坏了有序性" "违反最左前缀" 四大类去组织答案，而不是死记零散的规则列表——面试官更看重你是否理解"有序性被破坏"这一根本原理。

核心价值总结

  • 索引失效的本质只有一个：SQL 写法破坏了 B+ 树"按原始值有序排列"这一前提，优化器就没法利用有序性做快速定位。
  • 排查方法统一：先用 EXPLAIN 观察 type 和 key，再对照本文的场景清单逐一排查 SQL 写法。
  • 下一篇会讲：即使索引生效，排序和分组也可能触发 filesort/临时表，这是另一层需要关注的性能问题。
