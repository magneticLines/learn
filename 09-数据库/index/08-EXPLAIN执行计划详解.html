索引与执行计划

EXPLAIN 执行计划详解

SQL 慢不慢，靠猜没有用。EXPLAIN 是优化器"翻译"给你看的执行计划，读懂它的每一列，才能知道一条查询到底走没走索引、走的对不对、扫了多少行。这是索引调优里投入产出比最高的一项技能。

EXPLAIN 是什么：优化器决策的"体检报告"

执行一条 SELECT 语句之前，MySQL 优化器会根据表的统计信息（行数估算、索引基数等）生成若干种可能的执行方案， 并从中挑选一个它认为代价最小的方案去真正执行。EXPLAIN 不会真正执行这条语句（不算 EXPLAIN ANALYZE）， 而是把优化器"打算怎么做"这份计划原样打印出来，让你能审查优化器的决策是否合理。

SQL
EXPLAIN SELECT * FROM employees WHERE emp_no = 10001;
生活类比：EXPLAIN 就像导航软件在你出发前给出的"预计路线"——它会告诉你走高速还是走小路（对应 type）、预计经过多少个红绿灯（对应 rows），但这只是"预案"，真实路况（真实数据分布变化）可能让预计和实际有出入。

输出列总览

列名含义
id查询中每个 SELECT 子句的执行顺序标识
select_type查询类型：简单查询、子查询、联合查询等
table该行数据是关于哪张表的
partitions匹配到的分区（未分区表显示 NULL）
type本次查询的访问类型，性能等级的核心指标
possible_keys优化器认为"可能用得上"的索引候选列表
key优化器实际选择使用的索引
key_len实际使用的索引长度（字节数），可反推用了联合索引的几列
ref与索引比较的是常量还是某个字段（列出该字段）
rows优化器估算本次查询需要扫描的行数
filtered按条件过滤后剩余行数的百分比估算
Extra其他补充信息，很多重要线索都在这一列

id：执行顺序标识

id 相同时，表示这几行是同一个查询块内的多表关联，按从上到下的顺序执行（即书写顺序通常等于驱动表顺序，但最终由优化器决定）； id 不同时，id 值越大的查询越先被执行——这是很多人容易搞反的地方，原因是子查询、派生表必须先算出结果，外层查询才能使用这个结果。

SQL
EXPLAIN SELECT * FROM employees e
WHERE e.dept_no IN (
  SELECT d.dept_no FROM departments d WHERE d.dept_name = 'Sales'
);

-- 输出（简化）：
-- id  select_type      table         type  key         rows
-- 1   PRIMARY          e             ALL   NULL        300000
-- 2   SUBQUERY         d             ref   dept_name   1
-- 说明：id=2 的子查询先执行，结果作为常量传给 id=1 的外层查询使用

select_type：查询的类型

取值含义
SIMPLE不包含子查询或 UNION 的简单查询
PRIMARY包含子查询时，最外层的查询被标记为 PRIMARY
SUBQUERY在 SELECT 或 WHERE 列表中包含的子查询
DERIVEDFROM 子句中的子查询（派生表），MySQL 会先执行并物化成临时表
UNIONUNION 语句中除第一个查询外的后续查询
UNION RESULT从 UNION 临时表中读取结果的查询

type：访问类型（性能等级核心指标）

type 是 EXPLAIN 里最重要的一列，反映了 MySQL 找到目标数据所使用的访问方式。性能从好到差大致排列为： system > const > eq_ref > ref > range > index > ALL。日常优化的目标通常是把 type 优化到 range 及以上级别。

等级含义示例 SQL
system 表中只有一行数据（是 const 的特例），几乎不会在生产表上出现 SELECT * FROM (SELECT 1) t;
const 通过主键或唯一索引，与常量比较，最多匹配一行数据 SELECT * FROM employees WHERE emp_no = 10001;
eq_ref 多表关联时，使用主键或唯一索引作为关联条件，被驱动表每次最多匹配一行 SELECT * FROM orders o JOIN order_detail d ON o.id = d.order_id;（d.order_id 为唯一索引）
ref 使用普通（非唯一）索引做等值查询，可能匹配多行 SELECT * FROM employees WHERE dept_no = 'd001';
range 利用索引做范围查询，如 >、<、BETWEEN、IN SELECT * FROM employees WHERE hire_date BETWEEN '2020-01-01' AND '2020-12-31';
index 遍历整棵索引树（全索引扫描），无需回表但要扫描索引的所有叶子节点 SELECT emp_no FROM employees;（emp_no 是主键，走主键索引全扫描）
ALL 全表扫描，没有用上任何索引，逐行比较 SELECT * FROM employees WHERE YEAR(hire_date) = 2020;（对索引列做了函数运算，索引失效）
陷阱：很多初学者以为 type=index 一定比 type=range 差，其实两者可比性要看具体场景——index 是扫描整棵索引树，range 是索引树里的一段范围。 但通常 range 扫描的数据量更小，所以在多数实际场景中 range 优于 index。真正需要牢记的排序原则是从优化器代价估算的角度：能命中的行数越少、随机 I/O 越少，等级越靠前。

possible_keys 与 key：为什么"可能用的"和"实际用的"不一样

possible_keys 列出的是优化器理论上可能会用到的索引（只要 WHERE 条件里出现的列上有索引，就会被列进来）， 而 key 是优化器结合统计信息（行数估计、索引选择性）后，真正决定采用的那一个。

优化器有时会放弃看起来"更合适"的索引，本质原因是：MySQL 优化器的目标是让总代价（索引扫描代价 + 预计回表代价）最小，而不是简单套用"有索引就用"的规则。 如果某个索引的选择性很差（比如某列 90% 的值都相同），优化器估算走这个索引扫描的行数、再加上大量回表的随机 I/O，代价反而比直接全表扫描更高，于是干脆放弃这个索引。

陷阱："possible_keys 有值，但 key 是 NULL"是一个常见的误解触发点——很多人以为这是 bug，实际上这恰恰说明优化器经过代价评估后认为不走任何索引（全表扫描）反而更快，这在返回结果集占全表比例很大（比如超过 20%~30%，具体阈值取决于统计信息和表结构）时很常见。

key_len：反推联合索引用了几列

key_len 表示实际使用的索引在存储上占用的字节数，可以通过它反推优化器到底用了联合索引的几列、是否用满了整个索引。计算规则大致是：定长类型按类型本身字节数累加， 可为 NULL 的列 +1 字节标记，变长类型（如 VARCHAR）额外 +2 字节记录实际长度。

SQL
-- 假设联合索引 idx_a_b_c (a INT NOT NULL, b INT NULL, c VARCHAR(10))
-- a: 4 字节；b: 4 + 1(NULL标记) = 5 字节；c: 10*字符集字节数 + 2(变长) + 1(NULL标记)

EXPLAIN SELECT * FROM t WHERE a = 1 AND b = 2;
-- key_len = 9  → 说明用到了 a、b 两列（4 + 5），没用到 c

ref：与索引比较的是什么

当 type 是 ref、eq_ref 等类型时，ref 列会显示是与常量比较（显示 const）， 还是与另一张表的某个字段比较（显示 库名.表名.列名），这对排查多表关联的驱动关系很有用。

rows 与 filtered：预估值，不是精确值

rows 是优化器基于索引统计信息（基数估算）预估的需要扫描的行数，filtered 是预估应用 WHERE 条件后剩余行的百分比。 两者相乘大致得到预估的最终参与关联的行数。

陷阱：rows 是估算值，不是精确值！它依赖的统计信息（如 ANALYZE TABLE 生成的索引基数）可能过时或存在误差，尤其在数据分布不均匀、统计信息未及时更新的表上，rows 可能与真实扫描行数相差很大。不能只看这一个数字就断定 SQL 一定快或慢，需要结合 type 和 Extra 综合判断，必要时用 EXPLAIN ANALYZE 拿到真实执行的行数和耗时。

Extra：补充信息（高频考点）

取值含义
Using index 覆盖索引：查询所需的所有列都能在索引中取到，无需回表读聚簇索引
Using where 存储引擎返回记录后，服务层还需要用 WHERE 条件做进一步过滤
Using index condition 使用了索引下推（ICP，Index Condition Pushdown）：将部分 WHERE 条件下推到存储引擎层，在遍历索引时就过滤，减少回表次数
Using filesort 排序不能通过索引顺序直接得到，需要额外的排序操作（内存 quicksort 或磁盘归并排序）
Using temporary 需要借助临时表来完成查询（常见于 GROUP BY、DISTINCT、无法用索引优化的排序）
Using join buffer 关联查询中被驱动表没有可用索引，需要用连接缓冲区做嵌套循环关联（性能较差的信号）
Impossible WHERE WHERE 条件恒为假，优化器判断出该查询不可能返回任何行，直接跳过执行
Using index for group-by GROUP BY 所需的分组信息可以直接从索引中读取，无需额外排序或读取数据行
Select tables optimized away 使用聚合函数（如 MIN/MAX）且该列有索引，优化器可以直接从索引中取得结果，无需真正扫描表

综合示例一：覆盖索引 + 索引下推

SQL
-- 假设联合索引 idx_dept_hire (dept_no, hire_date)
EXPLAIN SELECT dept_no, hire_date FROM employees
WHERE dept_no = 'd005' AND hire_date > '2020-01-01';

-- +----+-------------+-----------+------+---------------+---------------+---------+------+------+--------------------------+
-- | id | select_type | table     | type | possible_keys | key           | key_len | ref  | rows | Extra                    |
-- +----+-------------+-----------+------+---------------+---------------+---------+------+------+--------------------------+
-- |  1 | SIMPLE      | employees | range| idx_dept_hire | idx_dept_hire | 40      | NULL | 120  | Using where; Using index |
-- +----+-------------+-----------+------+---------------+---------------+---------+------+------+--------------------------+

-- 分析：
-- type=range        -> 走了联合索引的范围扫描（hire_date 部分是范围条件）
-- key=idx_dept_hire  -> 用到了联合索引，且 key_len 说明 dept_no、hire_date 两列都用上了
-- Using index        -> SELECT 的列（dept_no, hire_date）恰好都在索引里，无需回表，命中覆盖索引

综合示例二：filesort 与 temporary 的代价

SQL
EXPLAIN SELECT dept_no, COUNT(*) FROM employees
GROUP BY dept_no ORDER BY COUNT(*) DESC;

-- +----+-------------+-----------+------+---------------+------+---------+------+--------+---------------------------------+
-- | id | select_type | table     | type | possible_keys | key  | key_len | ref  | rows   | Extra                           |
-- +----+-------------+-----------+------+---------------+------+---------+------+--------+---------------------------------+
-- |  1 | SIMPLE      | employees | ALL  | NULL          | NULL | NULL    | NULL | 300000 | Using temporary; Using filesort |
-- +----+-------------+-----------+------+---------------+------+---------+------+--------+---------------------------------+

-- 分析：
-- type=ALL          -> dept_no 上没有索引（或未命中），只能全表扫描
-- Using temporary    -> GROUP BY 需要借助临时表汇总各分组的 COUNT 结果
-- Using filesort     -> 按 COUNT(*) 排序，这是聚合结果，无法用索引顺序代替，只能额外排序
-- 优化方向：在 dept_no 上建索引，可以避免全表扫描，并有机会用 "Using index for group-by" 省去临时表

常见误区总结

误区一：只看 rows 数值断定 SQL 性能好坏——它只是统计信息驱动的估算值，可能失真，需结合 type、Extra 综合判断，必要时用 EXPLAIN ANALYZE 或实际执行时间验证。
误区二：认为 type=index 一定比 type=range 好——字面顺序容易让人误解，实际是 range 通常优于 index，因为 index 要扫描整棵索引树。
误区三：看到 possible_keys 有值而 key 为 NULL 就以为索引没生效或有 bug——这其实是优化器主动判断全表扫描代价更低后的正常选择。
面试高频："EXPLAIN 里最重要看哪几列？" 标准答案：type（访问方式等级）、key（实际用的索引）、rows（扫描行数估算）、Extra（是否出现 Using filesort / Using temporary 等性能警示信号），四者结合起来才能判断一条 SQL 的真实执行路径与优化空间。

核心价值总结

  • EXPLAIN 是优化器决策的说明书，不是执行结果：要养成写 SQL 前后都过一遍 EXPLAIN 的习惯。
  • type 是最先要看的等级指标：目标是尽量把 type 稳定在 range 及以上。
  • Extra 里的 filesort / temporary / join buffer 是性能优化的重点排查对象，通常意味着有进一步加索引或改写 SQL 的空间。
