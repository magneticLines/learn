哈希索引与自适应哈希索引

B+ 树几乎是我们对"索引"的默认印象，但还有一种更朴素、更极端的思路——哈希索引。理解它的优劣，才能理解 InnoDB 为什么在幕后偷偷用哈希给你的查询"开小灶"。

哈希索引原理：把"查找"变成"计算"

哈希索引的思路非常直接：对索引列的值计算一个哈希值（hash code），把这个哈希值作为哈希表的 key，指向对应数据行的指针作为 value。 查询时先对查询条件计算哈希值，直接定位到哈希表的桶（bucket），理想情况下一次计算就能找到目标，时间复杂度是 O(1)， 而 B+ 树索引需要从根节点到叶子节点逐层比较，时间复杂度是 O(log n)。

生活类比：B+ 树查找像查字典——按拼音字母顺序一页页翻，不管查"苹果"还是"香蕉"都要经过若干次翻页比较。 哈希索引像超市用会员手机号后四位做柜子编号——你一报手机号，直接算出柜子编号，伸手去拿，不用管其他会员的手机号是大是小。

哈希索引的局限：为什么不能替代 B+ 树

哈希函数的本质是把任意长度的输入"打散"成一个固定长度的值，这个打散过程不保序——也就是说，两个原始值相近（比如 8 和 9）， 它们的哈希值可能天差地别（比如 235 和 71）。这直接决定了哈希索引有三大先天缺陷：

  • 不支持范围查询：`WHERE age > 20` 这种查询，哈希表没法知道哪些哈希值对应"大于 20"的原始值，只能全表扫描。
  • 不支持排序：`ORDER BY age` 需要数据本身有序，哈希值的大小和原始值的大小毫无关系，排序用不上索引。
  • 不支持最左前缀匹配：联合索引 `(a, b)` 做哈希时是把 `a` 和 `b` 拼接后一起计算哈希值，单独查 `WHERE a = 1` 用不上这个哈希值，必须把 `a`、`b` 都传入才能算出正确的哈希桶。

此外还存在哈希冲突问题：不同的原始值可能算出相同的哈希值（哈希碰撞），此时哈希表需要在同一个桶内挂一条链表，逐个比较原始值才能确认，最坏情况会退化成 O(n) 的线性扫描。

InnoDB 的自适应哈希索引（AHI）

既然哈希索引这么多限制，为什么 InnoDB 还要用它？答案是：InnoDB 并不是让你手动建哈希索引，而是自己在内存里"悄悄"建了一层哈希缓存， 这就是 Adaptive Hash Index（自适应哈希索引，AHI）。

InnoDB 会持续监控索引的访问模式：如果发现某些 B+ 树索引页被反复用相同的查询条件访问（典型的热点数据，比如电商大促时的爆款商品 ID）， 就会自动为这部分频繁访问的键值在内存中建立一个哈希表，把"从 B+ 树根节点走到叶子节点的若干次页面访问"， 压缩成"哈希计算一次直接定位"。本质上，AHI 是建立在 B+ 树索引之上的一层缓存优化，而不是一种独立的、可选择的索引类型。

用户发起等值查询 WHERE id = 10086
  ├─ 无 AHI：Root → 中间节点 → 叶子节点（2~4 次页面访问）
  └─ 命中 AHI：哈希计算一次 → 直接定位到数据页（近似 O(1)）

AHI 由参数 innodb_adaptive_hash_index 控制开关（MySQL 5.7+ 默认开启），可以用 SHOW ENGINE INNODB STATUS 查看其命中率。

SQL
SHOW VARIABLES LIKE 'innodb_adaptive_hash_index';

-- 关闭 AHI（需要重启生效验证，也可用 SET GLOBAL 动态调整）
SET GLOBAL innodb_adaptive_hash_index = OFF;
AHI 不是免费的午餐：在高并发写入、访问模式非常分散（没有明显热点）的场景下，InnoDB 需要花费额外的 CPU 和锁开销去维护 AHI（曾经用一把全局的 AHI latch 保护，容易成为并发瓶颈；MySQL 5.7 之后拆分为多个分区锁有所缓解，但仍存在开销）。 如果监控发现维护 AHI 的开销高于它带来的查询加速收益（可通过 SHOW ENGINE INNODB STATUS 中的 btr_search 相关统计观察），可以考虑关闭它。这也是为什么一些高并发压测场景下，DBA 会尝试关闭 AHI 做对比测试。

常见误区

误区："MySQL 支持哈希索引，我可以在建表时给 InnoDB 表的某一列显式指定用哈希索引。" 这是混淆了 MEMORY 引擎和 InnoDB 引擎的差异：MEMORY 引擎（表数据全部存于内存）默认使用哈希索引作为其索引结构，可以用 USING HASH 语法显式声明； 但 InnoDB 不支持用户手动创建哈希索引，唯一和哈希相关的机制就是自动、透明、不可手动指定键值的 AHI，用户唯一能做的只是通过参数整体开关它。
面试高频："InnoDB 支持哈希索引吗？" 标准答法：不支持用户显式创建，但内部有一套自适应哈希索引（AHI）机制，会自动为热点数据在内存中建立哈希表加速等值查询，本质是缓存优化而非索引类型选择。

与其他知识点的联系

  • B+ 树索引的原理（本 Tab 02）：本篇的核心论证是"哈希索引为什么不能替代 B+ 树"，反过来也解释了 B+ 树为什么要牺牲等值查询的极致速度（O(log n) 而非 O(1)），去换取范围查询、排序、最左前缀这些能力——两篇是同一个权衡问题的正反两面。
  • 聚簇索引与二级索引回表（本 Tab 03）：AHI 加速的是"从 B+ 树根节点走到叶子节点"这个过程，无论是聚簇索引还是二级索引的 B+ 树都可能被 AHI 缓存命中；但 AHI 命中后如果查的是二级索引，依然要走一次回表去聚簇索引取完整行，AHI 只优化了"树内查找"这一步，不能消除回表。
  • Buffer Pool 原理（InnoDB Tab）：AHI 本质上是 Buffer Pool 之上再叠加的一层内存缓存优化，二者都是"用内存换磁盘 I/O"思路的具体实现，只是 Buffer Pool 缓存的是页本身，AHI 缓存的是"键值到页内记录位置"的哈希映射，进一步压缩了树内定位的开销。
  • 页结构与记录存储格式（本 Tab 01）：AHI 的哈希表指向的最终还是页内的具体记录位置，页内 Page Directory 做的"分组稀疏索引 + 二分查找"和 AHI 做的"哈希直接定位"，是两种不同粒度、不同代价的页内/树内查找加速手段。
  • 慢查询定位与优化流程（本 Tab 11）：AHI 是否命中、命中率高低，可以通过 SHOW ENGINE INNODB STATUS 观察，这属于排查流程里"判断问题类型"之外的一个深层调优点——当索引设计和 SQL 改写都已经优化到位，AHI 的开关状态也是压测调优时值得对比的一个变量。
