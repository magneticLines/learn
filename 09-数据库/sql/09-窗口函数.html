窗口函数(OVER/PARTITION BY/ROW_NUMBER/RANK)

上一篇讲 GROUP BY 会把多行"压扁"成一行，这在很多场景下其实是个损失——你明明只是想给每一行"贴上一个组内排名标签"，却因为 GROUP BY 丢失了明细数据。窗口函数就是为解决这个矛盾而生的。

核心区别：窗口函数不会减少行数

GROUP BY 是"多对一"的聚合：N 行输入，M 组输出（M ≤ N）。 窗口函数是"多对多"的计算：N 行输入，还是 N 行输出，只是每一行多了一个基于"某个窗口范围"计算出来的附加值。

类比：GROUP BY 好比开班会时把全班学生按小组合影，最后只留下每个小组一张合照（细节被抹掉了）；窗口函数好比给每个学生发一张成绩单，成绩单上除了自己的分数，还额外印上"你在全班排第几名"——每个人的独立记录都保留着，只是多了一个参考了"周围数据"计算出来的字段。

基本语法：OVER 子句定义"窗口"

语法结构
窗口函数() OVER (
  [PARTITION BY 分组列]   -- 把数据划分成多个"窗口"（类似 GROUP BY 但不合并行）
  [ORDER BY 排序列]        -- 决定窗口内的行顺序，影响 ROW_NUMBER/RANK 等排名函数
)

例子：每个专业内，按年龄给学生排名

students 表
id | name | age | major
---+------+-----+-------
1  | 张三 | 22  | CS
2  | 李四 | 22  | CS
3  | 王五 | 20  | CS
4  | 赵六 | 25  | 数学
用 ROW_NUMBER / RANK / DENSE_RANK 对比
SELECT name, major, age,
  ROW_NUMBER() OVER(PARTITION BY major ORDER BY age DESC) AS rn,
  RANK()       OVER(PARTITION BY major ORDER BY age DESC) AS rk,
  DENSE_RANK() OVER(PARTITION BY major ORDER BY age DESC) AS drk
FROM students;

-- 结果（CS 组张三李四同龄并列）：
-- 张三, CS,   22, rn=1, rk=1, drk=1
-- 李四, CS,   22, rn=2, rk=1, drk=1   ← 并列时 ROW_NUMBER 强行区分 1/2，RANK/DENSE_RANK 都算并列第1
-- 王五, CS,   20, rn=3, rk=3, drk=2   ← RANK 并列后跳过 2，DENSE_RANK 不跳号，紧接着排第2
-- 赵六, 数学, 25, rn=1, rk=1, drk=1   ← 换了 PARTITION，重新从第1名开始计

PARTITION BY 把数据切成若干个"窗口"（这里是按 major），但和 GROUP BY 不同的是，每个窗口内的所有原始行都被保留，只是在窗口范围内额外计算了一个排名列。

ROW_NUMBER / RANK / DENSE_RANK 的区别

函数并列处理方式典型场景
ROW_NUMBER()强制唯一编号，即使值相同也分先后（1,2,3,4...）需要绝对唯一序号时，如分页、去重取一条
RANK()并列同名次，但后续名次会跳号（1,1,3,4...）竞赛排名（并列第一后没有第二名）
DENSE_RANK()并列同名次，后续名次不跳号（1,1,2,3...）需要"连续等级"的场景，如价格档位

实战例子：每组取 Top N（分组内排名取值）

需求："查询每个专业年龄最大的 1 名学生"——这是窗口函数最经典的应用场景，GROUP BY 很难优雅地做到。

用窗口函数实现"每组 TopN"
SELECT name, major, age FROM (
  SELECT name, major, age,
    ROW_NUMBER() OVER(PARTITION BY major ORDER BY age DESC) AS rn
  FROM students
) t
WHERE rn = 1;
-- 结果：张三/CS/22（李四同龄但 ROW_NUMBER 只能保留一个）、赵六/数学/25
陷阱：窗口函数不能直接写在 WHERE 里。回顾第 5 篇的执行顺序，WHERE 在 SELECT（窗口函数所在的地方）之前执行，此时窗口函数的结果还没算出来，所以必须像上面例子一样包一层子查询（或用 MySQL 8.0+ 支持的 CTE），在外层再用 WHERE 过滤 rn。这是窗口函数最高频的实战陷阱。

窗口函数 vs GROUP BY 对比总结

维度GROUP BY窗口函数
行数变化多行合并成一行（有损）行数不变，每行附加一个计算列（无损）
能否保留明细不能，明细信息在合并时丢失能，明细数据原样保留
典型场景汇总统计（总数、总和、平均值）排名、累计值、组内对比（如"和上一行的差值"）
MySQL 支持版本一直支持8.0+ 才原生支持窗口函数

面试高频考点

问：窗口函数和 GROUP BY 最本质的区别是什么？ 答：GROUP BY 会把属于同一组的多行合并压缩成一行，丢失明细；窗口函数不会减少行数，只是给每一行附加一个基于窗口范围（PARTITION）计算出来的值，明细数据完整保留。
问：ROW_NUMBER、RANK、DENSE_RANK 有什么区别？ 答：核心看"遇到并列值时怎么处理"——ROW_NUMBER 强制给唯一序号（哪怕值相同）；RANK 允许并列但之后跳号；DENSE_RANK 允许并列且不跳号。

核心价值总结

  • 窗口函数解决了 GROUP BY "有损压缩"的问题：需要"既要看到明细，又要看到组内统计值"时首选窗口函数。
  • PARTITION BY 类似 GROUP BY 但不合并行：只是划定了每个窗口函数计算的"数据范围"。
  • 窗口函数不能直接出现在 WHERE 里：受限于 SELECT 语句的逻辑执行顺序，必须借助子查询或 CTE 在外层过滤。
