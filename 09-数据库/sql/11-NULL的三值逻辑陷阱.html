SQL 与关系模型基础

NULL 的三值逻辑陷阱

SQL 里的布尔逻辑不是"真/假"两值，而是"真/假/未知"三值——这一个设计决定，制造了几乎所有数据库新手都会踩的坑。本篇是本 Tab 最值得反复回顾的一篇。

为什么会有三值逻辑：NULL 代表"未知"，不是"空"或"0"

NULL 表示"这个值不存在/未知"，它不等于空字符串、不等于 0、也不等于任何具体的值——包括它自己。 因为"两个未知的东西是否相等"本身也是未知的，所以关系模型的比较运算不能只返回 TRUE/FALSE，还必须允许第三种结果：UNKNOWN。

表达式结果原因
NULL = NULLUNKNOWN（不是 TRUE！）两个都不知道的值，没法断言它们相等
NULL = 1UNKNOWN不知道 NULL 具体是什么，无法比较
NULL <> 1UNKNOWN同理
1 = 1TRUE正常比较
头号陷阱：NULL = NULL 的结果不是 TRUE，而是 UNKNOWN！ 在 WHERE 子句里，只有结果为 TRUE 的行才会被保留，UNKNOWN 会被当成 FALSE 一样处理（即被过滤掉）。
❌ 常见误区：用 = NULL 判断空值
SELECT * FROM students WHERE major = NULL;   -- 永远返回空结果集，不会报错，但逻辑上完全错误
✅ 正确写法：用 IS NULL / IS NOT NULL
SELECT * FROM students WHERE major IS NULL;
SELECT * FROM students WHERE major IS NOT NULL;

三值逻辑真值表：AND / OR / NOT 遇到 UNKNOWN 怎么算

ABA AND BA OR B
TRUEUNKNOWNUNKNOWNTRUE
FALSEUNKNOWNFALSEUNKNOWN
UNKNOWNUNKNOWNUNKNOWNUNKNOWN

记忆技巧：FALSE 是 AND 的"吸收元素"（FALSE AND 任何值 = FALSE），TRUE 是 OR 的"吸收元素"（TRUE OR 任何值 = TRUE）；除此之外只要沾上 UNKNOWN，结果多半也是 UNKNOWN。 NOT UNKNOWN 的结果仍然是 UNKNOWN（不是 TRUE，这一点极易搞错）。

第二号陷阱：NOT IN 遇到 NULL 会"全军覆没"

enrollments 表里 student_id 有一个 NULL 值
SELECT student_id FROM enrollments;
-- 结果：1, 2, NULL
❌ NOT IN 子查询结果集含 NULL，整体失效
SELECT name FROM students
WHERE id NOT IN (SELECT student_id FROM enrollments);
-- 结果：空集！即使明明有学生的 id 不在 (1, 2) 里

原因：id NOT IN (1, 2, NULL) 在逻辑上等价于 id<>1 AND id<>2 AND id<>NULL。 最后一项 id <> NULL 恒为 UNKNOWN，而 TRUE AND UNKNOWN = UNKNOWN（参考上面真值表）， 所以只要 IN 的集合里混入一个 NULL，整个 NOT IN 表达式对任何行都只能算出 UNKNOWN，等价于全部被过滤掉。这是极其隐蔽、极其高频的线上 bug 来源。

✅ 正确写法一：子查询里过滤掉 NULL
SELECT name FROM students
WHERE id NOT IN (
  SELECT student_id FROM enrollments WHERE student_id IS NOT NULL
);
✅ 正确写法二：改用 NOT EXISTS（推荐，天然不受 NULL 影响）
SELECT name FROM students s
WHERE NOT EXISTS (
  SELECT 1 FROM enrollments e WHERE e.student_id = s.id
);
工程建议：只要不确定子查询结果集是否可能含 NULL，一律优先用 NOT EXISTS 替代 NOT IN，这是团队 SQL 规范里非常常见的强制约定。

第三号陷阱：聚合函数忽略 NULL

第 8 篇提到过，这里再强化一次——聚合函数默认跳过 NULL 值，这会直接影响统计结果的分母/分子：

AVG 的分母是"非 NULL 行数"，不是"总行数"
-- scores 列：[80, 90, NULL, NULL]
SELECT AVG(score) FROM exam_results;
-- 结果 = (80+90)/2 = 85，而不是 (80+90+0+0)/4 = 42.5
-- 如果业务本意是"缺考按0分算平均分"，这里必须先用 COALESCE(score, 0) 处理

第四号陷阱：COUNT(*) 与 COUNT(列) 不是一回事

写法统计口径
COUNT(*)统计所有行数，不关心任何列是否为 NULL
COUNT(1)效果等同于 COUNT(*)，统计所有行数（MySQL 优化器会等价处理，性能无差异）
COUNT(列名)只统计该列非 NULL的行数
COUNT(*) vs COUNT(列) 的差异
-- students 表 4 行，其中 major 列有 1 行是 NULL
SELECT COUNT(*)    FROM students;   -- 4  （统计所有行）
SELECT COUNT(major) FROM students;   -- 3  （NULL 那一行不计入）
面试高频坑："想统计总行数，写 COUNT(某个可能为 NULL 的列)"——这是错误示范，结果会比实际行数少。统计总行数一律用 COUNT(*)，只有明确想统计"某列有多少个非空值"时才用 COUNT(列)。

处理 NULL 的正确工具箱

需求正确写法
判断是否为 NULLIS NULL / IS NOT NULL（禁止用 = NULL）
把 NULL 替换成默认值COALESCE(col, 默认值)（标准 SQL 通用），MySQL 也支持 IFNULL(col, 默认值)
排除子查询里的 NULL子查询加 WHERE col IS NOT NULL，或直接改用 NOT EXISTS
统计总行数COUNT(*)，不要用可能含 NULL 的列

面试高频考点

问：为什么 NULL = NULL 不返回 TRUE？ 答：因为 SQL 采用三值逻辑，NULL 代表"未知"而不是一个具体值，两个未知量无法断言相等，只能返回 UNKNOWN；WHERE 子句只保留结果为 TRUE 的行，所以这类比较永远筛选不出结果，必须用 IS NULL 显式判断。
问：NOT IN 和 NOT EXISTS 哪个更安全？ 答：NOT EXISTS 更安全——它通过关联子查询逐行判断"是否存在"，不会因为子查询结果集里混入 NULL 而导致整体失效；NOT IN 一旦子查询结果含 NULL，会因为三值逻辑变成恒为 UNKNOWN，返回空结果集，是非常隐蔽的 bug 来源。

核心价值总结

  • NULL 是"未知"，比较运算涉及 NULL 时结果是 UNKNOWN，不是 TRUE 也不是 FALSE：这是三值逻辑的根源。
  • NOT IN + 含 NULL 的子查询 = 隐藏地雷：优先用 NOT EXISTS 替代，或提前过滤 NULL。
  • 聚合函数忽略 NULL，COUNT(*) 与 COUNT(列) 语义不同：统计总数一律用 COUNT(*)。
