SQL 与关系模型基础

关系代数基础(选择/投影/连接/并交差)

关系代数是关系模型的"运算法则"——它规定了在"表的集合"上能做哪些合法运算、每种运算的输出还是不是一张表。理解它，你就理解了 SQL 每一条子句的数学原型。

为什么需要关系代数

上一篇讲到，关系模型把数据抽象成了"表 = 元组的集合"。但光有数据存储的抽象还不够，还需要一套运算规则来回答"如何从已有的表产生新的表（也就是查询结果）"。 这套规则就是关系代数（Relational Algebra）——它是一种过程化的查询语言：给定输入关系，通过一系列运算符组合，一步步推导出输出关系。 每个运算符的输入是关系，输出也是关系，这个"运算封闭性"叫闭包性质，正是它能像四则运算一样自由嵌套组合的原因。

类比：关系代数就像是"表格版的四则运算"。加减乘除输入数字、输出数字，可以任意嵌套（如 (a+b)*c）；关系代数的运算符输入表、输出表，同样可以任意嵌套。SQL 里一层套一层的子查询，本质就是关系代数表达式的嵌套。

用一个具体例子贯穿全篇

假设有两张表：

students（学生表）
id | name  | age | major
---+-------+-----+--------
1  | 张三  | 20  | CS
2  | 李四  | 22  | CS
3  | 王五  | 19  | 数学
enrollments（选课表）
student_id | course
-----------+--------
1          | 数据库
1          | 操作系统
2          | 数据库

1. 选择 σ（Selection）—— 挑"行"

选择运算按条件筛选行，不改变列的结构。记法：σ条件(关系)。

σ_age>19(students) 对应的 SQL
SELECT * FROM students WHERE age > 19;

对应关系代数：σage>19(students)。SQL 的 WHERE 子句就是选择运算 σ 的语法糖。

2. 投影 π（Projection）—— 挑"列"

投影运算按列名截取部分属性，并且理论上会去重（因为结果仍是"集合"）。记法：π列1,列2(关系)。

π_name,major(students) 对应的 SQL
SELECT DISTINCT name, major FROM students;

SQL 的 SELECT 列名列表 就是投影运算 π 的语法糖——只是工程实现上默认不去重（返回多重集），需要显式加 DISTINCT 才符合严格的集合语义。

3. 连接 ⋈（Join）—— 把两张表"拼"起来

连接运算把两个关系按某种条件横向拼接，是笛卡尔积 + 选择的组合简写。记法：R ⋈条件 S，等价于 σ条件(R × S)。

students ⋈ students.id=enrollments.student_id enrollments
SELECT s.name, e.course
FROM students s
JOIN enrollments e ON s.id = e.student_id;

结果：(张三, 数据库)、(张三, 操作系统)、(李四, 数据库)。 SQL 的 JOIN ... ON 就是连接运算 ⋈ 的语法糖，本篇先建立直觉，第 6 篇会详细展开 inner/left/right/full/cross 各种连接的差异。

4. 并 ∪ / 交 ∩ / 差 −（Set Operations）

这三个是标准的集合运算，要求参与运算的两个关系结构兼容（列数相同、对应列类型相同），这叫"并相容性（Union Compatibility）"。

运算含义对应 SQL
∪（并）两个结果集合并，自动去重UNION（去重）/ UNION ALL（不去重，性能更好）
∩（交）两个结果集的共有部分INTERSECT（MySQL 8.0.31+ 支持）
−（差）属于第一个但不属于第二个EXCEPT（MySQL 8.0.31+ 支持，早期版本常用 LEFT JOIN ... WHERE ... IS NULL 模拟）
常见误区：UNION 默认会去重（对应集合并运算的语义），这意味着它要对结果做排序/哈希去重，比 UNION ALL 慢。如果业务上已知两个结果集不会重复（比如按不同条件互斥筛选同一张表），应该优先用 UNION ALL，这是非常高频的性能优化点。

面试高频考点

问：SQL 和关系代数是什么关系？ 答：关系代数是理论基础，规定了"表能做哪些合法运算"；SQL 是这套代数运算的声明式语法糖，一条 SQL 语句在数据库内部会被解析成关系代数表达式（或更贴近执行的关系代数树），再交给优化器改写、选择执行计划。下一篇会专门展开这一点。
问：为什么连接可以用"笛卡尔积 + 选择"等价改写？ 答：这正是查询优化器工作的理论基础——数据库不会真的先算出笛卡尔积再过滤（那样开销是 O(n×m)），而是利用关系代数等价变换规则把过滤条件"下推"，用索引直接定位再关联，这是执行计划优化的核心手段之一。

核心价值总结

  • 关系代数是 SQL 的"编译目标"之上再抽象一层的理论模型：写 SQL 时，脑子里默认在做 σ、π、⋈ 的组合。
  • 闭包性质让运算可以自由嵌套：这就是为什么子查询、多层 JOIN 都能写成一条 SQL。
  • 为下一篇"SQL 的声明式本质"打基础：知道了 SQL 子句和代数运算符的对应关系，才能理解"声明式"到底声明的是什么。
