SQL 与关系模型基础

关系模型的抽象(表/元组/属性)

为什么几乎所有数据库教材都从"关系模型"讲起？因为它不是一种具体的数据库产品，而是一套"如何组织数据"的数学抽象——理解了这层抽象，你才能明白 SQL 为什么长这个样子。

关系模型到底"抽象"了什么

关系模型（Relational Model）由 E.F. Codd 在 1970 年提出，核心思想只有一句话： 把所有数据都表示成"关系"（relation），也就是数学意义上的集合。落到具体形式上：

  • 关系（Relation） —— 就是我们平时说的"表"（table）。
  • 元组（Tuple） —— 表中的一"行"（row），是一条完整的记录。
  • 属性（Attribute） —— 表中的一"列"（column），是元组的一个字段。
  • 域（Domain） —— 属性的取值范围，比如"年龄"这个属性的域是正整数。
类比：把一张表想象成一个"学生登记表"的表格模板：表头（属性）固定是"学号、姓名、年龄"，每交上来一张填好的表（元组）就是一条记录。数据库的"关系"不过是这种表格思想的数学严格化版本——它规定了元组是无序的集合，且集合中不允许有两条一模一样的记录。

关键设计选择：为什么是"无序的元组集合"

这是关系模型最容易被忽略、但影响深远的设计决定。它带来两个直接后果：

特性含义对 SQL 的影响
元组无序 表里的行没有"第一行第二行"的固有顺序 不写 ORDER BY 时，返回顺序不保证稳定；想要顺序必须显式声明
元组是集合元素 理论上不允许重复行（集合天然去重） SELECT 默认其实是"允许重复"的（多重集/bag），必须显式写 DISTINCT 才恢复"集合语义"，这是关系代数与工程实现的一个妥协点
常见误区：很多初学者以为数据库里的表是"按插入顺序存的数组"，于是写 SQL 时依赖"最后插入的就在最后一行"这种假设。这是错的——存储引擎完全可能因为索引结构、页分裂等原因重新排布物理存储顺序（后面 索引与执行计划 Tab 会讲 B+ 树如何组织数据）。没有 ORDER BY，顺序就是未定义行为，这是面试和线上事故的高频坑。

为什么关系模型能"赢"：对比层次模型与网状模型

关系模型不是数据库历史上第一个模型。在它之前，1960~70 年代主流是层次模型（Hierarchical Model）和网状模型（Network Model），它们用"指针"把记录像树/图一样连接起来。

模型数据组织方式核心问题
层次模型 树形结构，每个子节点只能有一个父节点（类似文件系统目录） 只能表达"一对多"，多对多关系必须冗余存储；查询路径与物理存储结构强绑定，改一次结构要改所有应用代码
网状模型 图结构，节点间用显式指针连接，允许多对多 比层次模型灵活，但查询必须由程序员手写"指针遍历"逻辑——本质上是命令式的导航式查询，非常依赖存储细节
关系模型 一堆独立的二维表，靠"值"（外键）而非物理指针关联 数据的逻辑结构与物理存储彻底解耦：应用只关心表和列的名字，不关心数据到底怎么存、有没有索引、指针指向哪

这就是关系模型真正解决的问题：数据独立性（Data Independence）。层次/网状模型下，你换一种存储方式几乎要重写所有查询代码；关系模型下，只要表结构（模式）不变，底层想换存储引擎、加索引、搬机器，业务 SQL 完全不用动。这也是为什么下一篇要讲的"关系代数"和后面的"SQL 声明式语言"能够成立的前提——因为数据被抽象成了与物理存储无关的逻辑集合。

用一个具体例子建立直觉

关系：students
CREATE TABLE students (
  id      INT PRIMARY KEY,   -- 属性：id，域是整数
  name    VARCHAR(50),          -- 属性：name，域是长度 ≤50 的字符串
  age     INT,                  -- 属性：age
  major   VARCHAR(50)
);

表 students 就是一个"关系"；(1, '张三', 20, '计算机') 这样一整行数据就是一个"元组"； id、name、age、major 各自是一个"属性"。这套术语看起来是在"炫学术"， 但它的价值在于：关系代数（下一篇要讲的 σ、π、⋈ 等运算）都是定义在"关系"这个抽象之上的，不关心你底层用的是 MySQL 还是 PostgreSQL。

面试高频考点

问：关系数据库中"关系"指的是什么？ 答：不是指表之间的外键关联关系，而是关系代数里的数学概念——一张表本身就是一个"关系"，是元组的集合。这是很多人一开始就理解错的术语。
问：为什么关系模型要求每个属性值是"原子的"（不可再分）？ 答：这叫第一范式（1NF）的雏形要求，如果一个字段里塞一个数组或 JSON 列表，就破坏了"域"的原子性假设，会让关系代数的运算（尤其是选择、连接）语义变得混乱。范式理论会在 数据库设计与范式 Tab 详细展开。

核心价值总结

  • 数据独立性：关系模型把"数据的逻辑视图"和"数据的物理存储"彻底分离，这是它打败层次/网状模型的根本原因。
  • 数学基础扎实：关系是集合，因此可以用集合论/谓词逻辑严格定义"如何查询"，这就是下一篇关系代数的来源。
  • 为 SQL 打地基：理解了"表是关系、行是元组、列是属性"，才能理解 SQL 里 SELECT/WHERE/JOIN 分别对应哪种数学运算。
