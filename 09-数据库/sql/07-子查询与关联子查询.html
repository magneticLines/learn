SQL 与关系模型基础

子查询与关联子查询

子查询是"把一条 SQL 的结果当成另一条 SQL 的输入"——这正是关系代数闭包性质（运算结果还是关系，可以继续参与运算）在 SQL 里最直接的体现。

子查询的分类

按返回结果的形状，子查询大致分三类：

类型返回结果典型用法
标量子查询（Scalar Subquery）单行单列，一个具体值放在 WHERE col = (子查询) 或 SELECT 列表里
行/列子查询单行多列 或 多行单列IN、ANY、ALL
表子查询（Derived Table）多行多列，当成一张临时表放在 FROM 子句里，必须起别名

标量子查询：把查询结果当一个值用

查询年龄大于平均年龄的学生
SELECT name, age
FROM students
WHERE age > (SELECT AVG(age) FROM students);

括号里的 SELECT AVG(age) FROM students 只返回一个值，可以直接和 age 做比较，就像用了一个变量一样。

IN 子查询：把查询结果当一个集合用

查询选过"数据库"这门课的学生
SELECT name
FROM students
WHERE id IN (
  SELECT student_id FROM enrollments WHERE course = '数据库'
);

这里子查询和外层查询之间没有相互依赖——子查询可以独立执行一次，产出一个固定的结果集，再供外层用。这叫非关联子查询。

关联子查询：每一行外层数据都要重新执行一次子查询

与非关联子查询相对，关联子查询（Correlated Subquery）的子查询里引用了外层查询的列，导致子查询不能独立执行—— 它必须针对外层的每一行，代入对应的值重新算一次。最典型的写法是配合 EXISTS：

用 EXISTS 改写上面的需求
SELECT name
FROM students s
WHERE EXISTS (
  SELECT 1 FROM enrollments e
  WHERE e.student_id = s.id -- 引用了外层的 s.id，这就是"关联"
    AND e.course = '数据库'
);

逻辑上可以理解成：对 students 表的每一行 s，都拿它的 s.id 去执行一遍内层查询，看是否存在匹配的选课记录。 子查询里的 SELECT 1 只是占位——EXISTS 只关心"有没有行"，不关心具体返回什么列，因此惯例写 SELECT 1。

子查询 vs JOIN：性能与可读性的取舍

用 JOIN 改写上面 IN 子查询的例子：

用 JOIN 改写
SELECT DISTINCT s.name
FROM students s
JOIN enrollments e ON s.id = e.student_id
WHERE e.course = '数据库';
维度子查询（IN / EXISTS）JOIN
可读性逻辑分层清晰，"先筛选课程，再找学生"的思路更接近自然语言需要理解连接后再过滤的整体结果集，多表关联时容易绕
结果去重IN/EXISTS 天然不会因为右表多行匹配而重复左表的行如果右表对左表是一对多，需要显式 DISTINCT 或调整关联方式，否则行数会膨胀
是否需要用到右表的列只做"存在性判断"，用不到右表的列时很自然如果确实要一起展示右表的列，JOIN 更直接
历史性能差异早期 MySQL（5.5 之前）优化器对某些子查询处理较差，容易出现子查询被重复执行；现代 MySQL 优化器通常能把 IN 子查询改写成半连接（semi-join）执行计划相对直观，容易通过 EXPLAIN 判断走了什么索引
陷阱提醒：关联子查询本质上是"外层每一行都触发一次内层查询"，如果内层查询没有合适的索引，会导致近似 O(n×m) 的开销，比一次性 JOIN 再利用索引关联要慢得多。是否用子查询，最终要靠 EXPLAIN 观察执行计划来判断，不能凭直觉决定——这是 索引与执行计划 Tab 会详细展开的内容。

面试高频考点

问：什么是关联子查询？和普通子查询的区别是什么？ 答：关联子查询在子查询内部引用了外层查询的列，导致它不能独立求值、必须对外层每一行重新执行一次；普通（非关联）子查询可以独立执行一次，产出固定结果供外层使用。
问：NOT IN 子查询有什么隐藏风险？ 答：如果子查询结果集中包含 NULL，整个 NOT IN 会全部返回空结果——这是一个极其隐蔽的陷阱，第 11 篇会专门讲这个 NULL 三值逻辑的坑。

核心价值总结

  • 子查询是关系代数闭包性质的体现：一个查询的结果可以直接作为另一个查询的输入，可以任意嵌套。
  • 关联 vs 非关联的核心区别：子查询是否依赖外层的值决定了它能不能独立求值一次，还是要逐行重新执行。
  • 子查询与 JOIN 没有绝对的优劣：要结合可读性需求和实际执行计划（EXPLAIN）来选择，现代数据库优化器往往能把两者转换成等价的物理执行方式。
