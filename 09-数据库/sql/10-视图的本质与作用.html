视图的本质与作用

视图常被误解成"另一张表"，但它实际上连一份数据都不存——它只是一条被存起来、起了个名字的 SELECT 语句。理解这一点，你才能明白它的能力边界在哪。

视图的本质：存储的查询，不存储数据

创建一个视图
CREATE VIEW cs_students AS
SELECT id, name, age FROM students WHERE major = 'CS';

执行完这条语句后，数据库并没有真的把"CS 专业的学生"复制一份存到磁盘上。它只是把这条 SELECT 语句的定义保存在数据字典（元数据）里，起了个名字叫 cs_students。 之后你查询这个视图：

查询视图
SELECT * FROM cs_students WHERE age > 20;

数据库会在执行时把它原地展开（View Expansion），等价于把视图定义代入替换，变成：

数据库实际执行的等价语句
SELECT * FROM (
  SELECT id, name, age FROM students WHERE major = 'CS'
) AS cs_students
WHERE age > 20;
类比：视图就像 Excel 里的"高级筛选/自定义视图"功能，它不复制一份新表格，只是记住了"筛选条件是什么"，每次打开都用这个条件在原表上重新算一遍结果。数据万一在原表变了，视图看到的立刻就是新数据——因为视图压根没存旧数据。

视图的两大核心作用

1. 简化复杂查询

把一个复杂的多表连接/聚合查询封装成一个视图，之后业务代码只需要 SELECT * FROM 视图名，不用每次都写一遍冗长的 JOIN 逻辑。

把复杂查询封装成视图
CREATE VIEW student_course_summary AS
SELECT s.name, COUNT(e.course) AS course_count
FROM students s
LEFT JOIN enrollments e ON s.id = e.student_id
GROUP BY s.id, s.name;

-- 之后业务方只需要：
SELECT * FROM student_course_summary WHERE course_count > 1;

2. 权限隔离

如果表里有敏感字段（比如身份证号、薪资），可以创建一个只暴露"安全字段"的视图，把权限授予视图而不是原表，实现列级别的权限控制（很多数据库的原生权限系统只能做到表级别）。

用视图做列级权限隔离
CREATE VIEW employee_public AS
SELECT id, name, department FROM employees;  -- 不暴露 salary、id_card 等敏感列

GRANT SELECT ON employee_public TO 'hr_readonly'@'%';
-- 完全不给这个账号访问 employees 原表的权限

视图 vs 物化视图：核心取舍

上面讲的都是普通视图（View），它每次查询都重新执行底层 SQL。而物化视图（Materialized View）则相反——它会真正把查询结果存储成物理数据，查询时直接读存好的结果，不用重新计算。

维度普通视图（View）物化视图（Materialized View）
是否存储数据不存储，只存查询定义存储，是查询结果的物理快照
数据实时性永远实时，因为每次都重新查询原表存在"数据滞后"，需要刷新（Refresh）才能同步最新数据
查询性能取决于底层 SQL 本身的性能，复杂聚合每次都要重算非常快，直接读预计算好的结果，适合复杂聚合的报表场景
写入/维护成本无额外成本需要定期刷新或增量维护，有额外的存储和计算开销
MySQL 原生支持支持（CREATE VIEW）不原生支持，通常用"定时任务 + 真实表"或触发器手动模拟，PostgreSQL/Oracle 原生支持 MATERIALIZED VIEW
陷阱：很多人以为"用视图能提升查询性能"，这是常见的误解——普通视图本身不缓存任何数据，也不会让底层的复杂 JOIN/聚合变快，它带来的收益只是"减少手写重复 SQL、统一权限管理入口"。真正想用空间换时间来加速复杂报表查询，需要的是物化视图或者手动维护的汇总表。

面试高频考点

问：视图能提高查询性能吗？ 答：不能。普通视图只是存储了查询定义，执行时会展开成对原表的查询，本身不做任何缓存，性能取决于展开后 SQL 的执行计划。真正能提速的是物化视图（存储计算结果）或索引优化。
问：什么情况下视图不可更新（不能对视图做 INSERT/UPDATE）？ 答：当视图的 SELECT 中包含聚合函数、GROUP BY、DISTINCT、多表 JOIN、子查询等，数据库无法确定"更新视图的哪一列应该反映回原表的哪一行哪一列"，此时视图是只读的。简单的单表无聚合视图通常可更新。

核心价值总结

  • 视图的本质是"存储的 SELECT 语句"：不存储数据，查询时原地展开为对底层表的查询。
  • 两大作用是简化复杂查询与实现权限隔离：而不是性能优化。
  • 物化视图是"用存储空间换查询速度"的取舍：牺牲实时性，换来复杂聚合查询的性能，MySQL 需要手动模拟。
