聚合函数与 GROUP BY 原理

GROUP BY 做的事情，本质是把"一堆元组的集合"重新划分成"若干个子集合"，再对每个子集合分别计算一个汇总值。理解这个"分组降维"的过程，才能理解 SELECT 列表为什么有严格限制。

聚合函数：把多行压缩成一个值

函数作用是否忽略 NULL
COUNT(*)统计行数不忽略，数所有行（包括含 NULL 的行）
COUNT(列)统计该列非 NULL 的行数忽略 NULL
SUM(列)求和忽略 NULL（相当于 NULL 不参与求和）
AVG(列)求平均值忽略 NULL（分母是非 NULL 的行数，不是总行数）
MAX(列)/MIN(列)求最大/最小值忽略 NULL

第 11 篇会专门讲 COUNT(*) 和 COUNT(列) 的差异陷阱，这里先记住一个核心规律：几乎所有聚合函数（除 COUNT(*) 外）都会自动忽略 NULL 值。

没有 GROUP BY 时，整张表就是"一个组"

不写 GROUP BY，聚合函数作用于整表
SELECT COUNT(*), AVG(age) FROM students;
-- 整张表被当成唯一的一个分组，只返回一行汇总结果

GROUP BY：按列值重新分桶

students 表
id | name | age | major
---+------+-----+-------
1  | 张三 | 20  | CS
2  | 李四 | 22  | CS
3  | 王五 | 19  | 数学
按专业分组统计人数与平均年龄
SELECT major, COUNT(*) AS cnt, AVG(age) AS avg_age
FROM students
GROUP BY major;

-- 结果：
-- CS,   2, 21.0
-- 数学, 1, 19.0

执行过程可以理解成两步：先把 students 按 major 的值切成若干个子集合（CS 组 = {张三,李四}，数学组 = {王五}）， 再对每个子集合分别计算 COUNT 和 AVG。这正是"关系 = 集合"这一抽象在聚合场景下的自然延伸——GROUP BY 是把一个大集合划分成若干个不相交的子集合（这在数学上叫"划分/Partition"）。

核心原理：为什么 SELECT 列表只能出现分组列或聚合函数

❌ 违反 ONLY_FULL_GROUP_BY 的写法
SELECT major, name, AVG(age)  -- name 既不是分组列，也没被聚合函数包裹
FROM students
GROUP BY major;

问题出在：分组之后，CS 这一组里有张三和李四两个人——"这一组的 name 到底该显示哪一个？"这是一个没有唯一答案的问题。 分组运算的本质是"多对一"的降维：一组里可能有多行，但输出只有一行，任何"没有被聚合函数处理、又不是分组依据"的列，在语义上都是模糊不清的（该取哪一行的值？第一行？随便一行？）。

MySQL 5.7 之前默认允许这种"暧昧"的写法（会随机挑一行的值返回，容易造成隐蔽 bug），从 5.7 开始默认开启 ONLY_FULL_GROUP_BY SQL 模式， 强制要求 SELECT 列表中的每一列必须满足：要么出现在 GROUP BY 中，要么被聚合函数包裹，否则直接报错，从语法层面堵住这个语义模糊的坑。

常见误区：很多人升级 MySQL 版本后突然发现老 SQL 报错 this is incompatible with sql_mode=only_full_group_by，第一反应是"关掉这个模式"。但这其实是在掩盖一个真正的设计缺陷——报错的 SQL 本身逻辑就是有歧义的，正确做法是把 name 也纳入 GROUP BY（如果每个 name 唯一对应一个 major，语义就清晰了），或者干脆用聚合函数如 GROUP_CONCAT(name) 把这一组里所有的 name 拼起来显示。

面试高频考点

问：为什么 GROUP BY 之后 SELECT 不能随便写列？ 答：因为分组是"多行压缩成一行"的降维操作，任何没有被聚合、也不是分组依据的列，在一个分组里可能对应多个不同的值，语义上无法确定该显示哪一个。ONLY_FULL_GROUP_BY 模式就是为了防止这种歧义。
问：GROUP BY 和 DISTINCT 有什么关系？ 答：SELECT DISTINCT major FROM students 和 SELECT major FROM students GROUP BY major 在只查分组列、不做聚合计算时结果等价，都是"去重"；但 GROUP BY 的能力更强，它可以同时对每组计算聚合值，这是纯 DISTINCT 做不到的。

核心价值总结

  • GROUP BY 的本质是把集合划分成互不相交的子集合：这是理解一切分组行为的起点。
  • 聚合函数默认忽略 NULL：这会直接影响 AVG 的分母、COUNT(列) 的计数，是常见的数据统计误差来源。
  • ONLY_FULL_GROUP_BY 不是限制，而是在防止歧义：报错时应该反思 SQL 逻辑本身，而不是简单粗暴关掉这个模式。
