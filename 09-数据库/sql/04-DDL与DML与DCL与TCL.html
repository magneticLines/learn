SQL 与关系模型基础

DDL / DML / DCL / TCL 分类

SQL 不是铁板一块的单一语言，而是按"操作的对象层次"分成了四大类。分清楚这四类，你才能理解为什么有的语句会自动提交事务、有的语句需要权限、有的语句改的是"数据"而有的改的是"数据的容器"。

为什么要分类：操作的对象层次不同

SQL 语句按操作对象可以分成四层：数据库的"结构骨架"、结构里装的"数据本身"、谁能访问这些数据的"权限"、以及一组操作如何作为整体生效的"事务边界"。 对应关系数据库理论里"模式（Schema）与实例（Instance）"的区分——DDL 操作的是模式，DML 操作的是实例，DCL/TCL 则是工程落地时必须补充的管理与一致性机制。

DDL —— Data Definition Language（数据定义语言）

操作的对象是表结构本身（模式），而不是表里的数据。

DDL 代表命令
CREATE TABLE students (id INT PRIMARY KEY, name VARCHAR(50));
ALTER TABLE students ADD COLUMN age INT;
DROP TABLE students;
TRUNCATE TABLE students;  -- 清空数据但保留表结构，本质是重建表，也算 DDL
陷阱：在 MySQL（InnoDB）里，DDL 语句会隐式提交当前事务——如果你在一个事务中间执行了 ALTER TABLE，之前未提交的 DML 操作会被强制提交，且无法再 ROLLBACK。这是很多人踩过的坑：以为整个流程在一个事务里，结果一条 DDL 把事务边界"切断"了。

DML —— Data Manipulation Language（数据操纵语言）

操作的对象是表里的数据本身（实例），不改变表结构。

DML 代表命令
SELECT * FROM students;
INSERT INTO students (id, name) VALUES (1, '张三');
UPDATE students SET name = '李四' WHERE id = 1;
DELETE FROM students WHERE id = 1;
严格分类小知识：有些教材把 SELECT 单独归为 DQL（Data Query Language，数据查询语言），因为它只读不写。但工程上通常把它并入 DML 一起讲，因为它同样作用于"数据实例"这一层。

DCL —— Data Control Language（数据控制语言）

操作的对象是权限——谁能对哪些数据做哪些操作。

DCL 代表命令
GRANT SELECT, INSERT ON mydb.students TO 'app_user'@'%';
REVOKE INSERT ON mydb.students FROM 'app_user'@'%';

DCL 是数据库层面实现"最小权限原则"的手段——生产环境里应用账号通常只授予业务必需的 SELECT/INSERT/UPDATE 权限，不给 DROP/GRANT 这类高危权限。

TCL —— Transaction Control Language（事务控制语言）

操作的对象是一组 DML 语句的边界与生效方式，保证 ACID 中的原子性（Atomicity）。

TCL 代表命令
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;      -- 提交，两条 UPDATE 一起生效
-- 或者：
ROLLBACK;    -- 回滚，两条 UPDATE 都撤销，仿佛没发生过
SAVEPOINT sp1; -- 设置保存点，可以只回滚到某个中间状态

TCL 的意义在 事务与并发控制 Tab 会深入展开（ACID、隔离级别、MVCC 等），这里先建立"事务是一组 DML 的原子性打包"这个基础认知。

四类对比一览

分类全称操作对象代表命令是否自动提交
DDLData Definition Language表结构 / 数据库对象CREATE/ALTER/DROP/TRUNCATE是，隐式提交当前事务
DMLData Manipulation Language表中的数据SELECT/INSERT/UPDATE/DELETE否，需要 COMMIT（除非开启了 autocommit）
DCLData Control Language用户权限GRANT/REVOKE是
TCLTransaction Control Language事务边界COMMIT/ROLLBACK/SAVEPOINT—

面试高频考点

问：TRUNCATE 和 DELETE 有什么区别？ 答：DELETE 是 DML，逐行删除、可以加 WHERE、可以回滚、会触发触发器；TRUNCATE 是 DDL，本质是"扔掉整个表重建"，不能加条件、速度快、在多数数据库中不可回滚（或回滚代价很高）、不会逐行触发触发器。这是区分 DDL/DML 分类实用性的经典例子。
问：为什么执行了一条 ALTER TABLE 后，之前事务里的操作突然提交了？ 答：因为 DDL 在 MySQL 里会隐式提交事务（不像标准 SQL 规范中某些数据库支持事务性 DDL）。这解释了为什么线上做表结构变更前，要格外注意当前连接是否还有未提交的业务事务。

与其他知识点的联系

  • SQL 的声明式本质（上一篇）：本篇的四类划分主要是针对 DML 语句（尤其是 SELECT）展开的，DDL 语句本身更接近"声明结构"而不是"声明结果"，理解这个区分有助于明白为什么查询优化器的等价改写、执行计划选择这些"声明式红利"主要发生在 DML 上，DDL 的执行路径相对直接。
  • 数据备份与恢复策略（复制与高可用模块）：逻辑备份（mysqldump）恢复的本质就是把 DDL（CREATE TABLE）和 DML（INSERT）语句重新执行一遍——本篇讲的"DDL 会隐式提交事务"这一特性，正好解释了为什么逻辑备份恢复大表时，表结构会先落地生效，之后的数据插入是分批提交的独立过程，不能指望整个恢复过程回滚。
  • 高可用方案简介（复制与高可用模块）：MHA/MGR 讨论的"事务提交是否需要多数派确认"，本质上是在 TCL 的 COMMIT 语义之上叠加的一层分布式一致性保证——MGR 要求事务提交前先过 Paxos 协商，相当于把单机的 TCL 原子性保证扩展到了多节点场景。
  • SELECT 语句的执行逻辑顺序（下一篇）：本篇把 SELECT 归入 DML 一起讲，下一篇会专门展开 SELECT 内部最复杂、最常考的逻辑执行顺序——这是"先分类、再深入其中最重要的一类"的自然递进关系。

核心价值总结

  • DDL 管结构、DML 管数据：这是最基本的分类线，也是后面很多"为什么会自动提交"疑惑的根源。
  • DCL 是权限最小化的落地手段，TCL 是原子性的落地手段：两者都服务于工程上的安全与一致性目标。
