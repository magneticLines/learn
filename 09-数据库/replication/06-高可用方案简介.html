复制与高可用

高可用方案简介（MHA 与 MGR）

主库一旦挂了，谁来接班、怎么保证数据不丢、怎么让业务无感知切换——这是高可用方案要解决的核心问题。MHA 和 MGR 代表了两代完全不同的设计思路。

一句话讲清楚

高可用（High Availability）方案要解决的核心问题是：主库故障后，如何快速、安全地选出新主库并让从库和应用切过去，尽量少丢数据、少停机。 MHA（Master High Availability）是一套基于传统异步/半同步复制的故障检测 + 自动切换工具； MGR（MySQL Group Replication）则是 MySQL 官方基于 Paxos 协议原生实现的复制组方案，从复制机制的底层就重新设计了一致性保证。

类比：MHA 像是给一个"老板 + 若干分店"的传统架构外挂了一个经验丰富的"调度员"——老板万一出事，调度员负责在几秒内从分店里挑一个数据最全的顶上去当新老板， 再通知其它分店和收银系统认新老板。而 MGR 则是从根上换了一套治理模式：不再有绝对意义上的"唯一老板"，而是几个分店组成一个"合议组"， 每一笔生意都要合议组内多数成员投票通过才算数（Paxos 协议），这样即便原来的带头人挂了，剩下的成员也能立刻自己选出新的带头人，且投票通过的生意保证不会因为换人而丢失。

MHA：基于传统复制的故障转移工具

MHA 不改变 MySQL 原有的主从复制机制（还是异步或半同步复制），而是在这套机制之上加了一层管理程序，核心工作分两部分：

  • MHA Manager：独立部署的监控节点，持续探测主库存活状态；一旦判定主库故障，触发故障转移流程。
  • MHA Node：部署在每个 MySQL 节点上的代理，负责在故障转移时解析和补齐 binlog 差异。

故障转移的核心步骤：

  1. 确认主库真的宕机（避免网络抖动造成的误判，通常会多次探测或结合多个监控点交叉验证）。
  2. 从多个从库中找出relay log 最接近主库最新进度的那个从库（差异最小，意味着数据最全）。
  3. 让其它从库先从这个"数据最全"的从库补齐差异的 relay log（尽量减少数据丢失）。
  4. 把这个从库提升（promote）为新主库。
  5. 让其它从库重新指向新主库，通知应用层/VIP 切换连接到新主库。
常见误区：MHA 能做到"秒级"故障转移，但不能保证零数据丢失——因为它依赖的仍然是异步/半同步复制， 如果主库宕机时有事务的 binlog 还没来得及传给任何从库，这部分数据无论如何补偿都找不回来了。MHA 解决的是"尽快恢复服务、尽量减少数据丢失"，而不是"绝对不丢数据"。

MGR：基于 Paxos 的复制组

MGR 是 MySQL 5.7.17 引入的原生高可用方案，最大的区别在于：它不是在异步复制外面包一层管理工具，而是重新设计了复制协议本身。 MGR 中多个节点组成一个"复制组"，每个事务提交前需要经过组内基于 Paxos 协议的一致性协商，只有得到组内多数节点（Quorum）确认，事务才能真正提交。

MGR 支持两种模式：

  • 单主模式（Single-Primary）：组内自动选出一个主节点接受写入，其它节点只读，类似传统主从但故障转移是自动的且强一致。
  • 多主模式（Multi-Primary）：组内所有节点都可以接受写入，需要业务自行规避多点写入可能产生的冲突（如同一行数据在两个节点同时被修改），复杂度和使用门槛更高，生产环境更常用单主模式。
为什么 MGR 能做到自动选主且一致性更强：因为选主和事务提交的一致性保证都内建在 Paxos 协议里——多数派达成一致才能提交，天然避免了"数据丢失"和"脑裂"（多个节点都认为自己是主）的问题； 而 MHA 的选主是外部工具基于"看起来数据最全的从库"这种启发式判断做出的决策，本质上是"事后补救"，不是协议层面的强保证。

MHA 与 MGR 对比

维度MHAMGR
复制机制 传统异步/半同步复制 + 外部管理工具 原生基于 Paxos 协议的复制组
一致性保证 尽力而为，可能丢失未同步的事务 多数派确认才提交，一致性更强
选主方式 外部监控工具触发，选"数据最全"的从库 协议内建自动选主，故障后组内自动完成
部署复杂度 相对简单，架构改动小，社区经验成熟 较高，网络分区容忍度、组内节点数（推荐奇数，通常3或5）需要仔细规划
写入性能 不受影响（本质还是普通主从） 有一定开销（每个事务都要走组内协商）
成熟度/生态 历史悠久，很多老系统仍在用（官方已停止维护，社区分支维护中） MySQL 官方主推的高可用方向，云厂商（如阿里云 PolarDB-X、AWS Aurora 部分场景）也有借鉴类似思路
面试高频："MHA 和 MGR 怎么选？" 没有绝对答案，要看场景：存量系统、团队对传统复制运维经验丰富、可以容忍极端情况下少量数据丢失，MHA 依然是稳妥选择； 新系统、对一致性要求高、愿意承担一定的写入性能开销和运维学习成本，MGR 是更现代化、官方推荐的方向。核心考察点是能否说清楚两者在"一致性保证来源"上的本质差异——MHA 是外部工具补救，MGR 是协议内建保证。

核心价值总结

  • MHA 是"给传统复制打补丁"的思路，MGR 是"重新设计复制协议"的思路，理解这个根本差异比记住两个名字更重要。
  • Paxos/Raft 这类共识协议是分布式系统实现强一致性的通用武器，MGR 只是把它应用在了 MySQL 复制场景，后面学分布式系统会反复遇到同样的思想。
  • 高可用方案要和前面讲的半同步复制、主从延迟结合起来看——它们共同构成了"MySQL 如何在保证可用性的前提下尽量减少数据风险"这条主线。

与其他知识点的联系

  • 读写分离架构（上一篇）：读写分离里"主库单点故障怎么办"这个悬而未决的问题，正是本篇要补上的答案——MHA/MGR 决定了主库挂掉之后新主库怎么选出来、从库和应用怎么切过去，二者合起来才是一套完整的主从集群方案，读写分离负责日常的读流量分摊，高可用方案负责故障时刻的应急处理。
  • 数据备份与恢复（下一篇）：高可用方案（无论 MHA 还是 MGR）解决的都是"节点故障"场景下的服务连续性问题，但对"数据本身被误删或被脏数据污染"完全无能为力——脏数据会原样同步到所有从库和复制组节点，这时候真正能救命的只有下一篇讲的备份，两者是应对不同风险类型的互补手段，缺一不可。
  • 主从延迟（更早的一篇）：MHA 之所以只能做到"尽力而为、可能丢失部分数据"，根源就在于它依赖的传统复制本身是异步/半同步的，存在延迟窗口；而 MGR 通过 Paxos 协议要求多数派确认后才提交事务，本质上是从协议层消灭了"异步复制窗口"这个延迟问题的根源，理解这一点才能真正理解 MHA 和 MGR 在一致性保证上的本质差异。
  • 分片场景下的分布式事务（扩展性模块）：MGR 用 Paxos 协议在多个节点间对同一份数据达成一致，这与分库分表场景下需要用 Seata 等框架协调多个物理库之间的事务一致性，虽然应用场景不同，但都属于"多节点如何对一份逻辑数据达成共识"这一分布式系统核心问题的具体实例。
