复制与高可用

读写分离架构

数据库扛不住读流量时，最直接的思路就是"写只走一个主库，读分摊到多个从库"——但落地这套架构，绕不开"从库数据不是最新的"这个先天缺陷。

一句话讲清楚

读写分离（Read/Write Splitting）利用主从复制的特性，把写请求（INSERT/UPDATE/DELETE）路由到主库，读请求（SELECT）分摊到一个或多个从库， 从而横向扩展数据库的读吞吐能力。它解决的是"读多写少"场景下单机读性能瓶颈的问题，本质是一种"利用复制的副作用做负载均衡"的架构手段。

类比：就像一家总店生意太好，光靠总店的收银台既要收钱（写）又要回答顾客"这个东西多少钱、有没有货"（读）的咨询，忙不过来。 于是干脆开几家分店专门用来"查询咨询"，所有实际的进货、改价格（写操作）还是必须回总店做，分店的商品信息定期从总店同步过来——但分店的信息可能不是最新的， 总店刚改的价格，分店的价签可能还没来得及换。

典型架构：应用层路由 vs 中间件

方案一：应用层路由

在应用代码或数据访问层（如 MyBatis 拦截器、Spring 的 AbstractRoutingDataSource）根据 SQL 类型或业务显式指定，把请求路由到主库数据源还是从库数据源。

应用层路由架构
  ├─ 应用服务
  │    ├─ 写请求 → 主库数据源（DataSource-Master）
  │    └─ 读请求 → 从库数据源（DataSource-Slave，可配置多个做负载均衡）
  ├─ 优点：架构简单，不引入额外组件，路由逻辑可以按业务灵活定制
  └─ 缺点：路由逻辑侵入业务代码，多语言技术栈需要各自实现一遍

方案二：中间件（如 MyCat、ProxySQL、Atlas）

在应用和数据库之间加一层数据库代理，代理对外表现得像一个单一的 MySQL 实例，应用完全无感知，由中间件负责解析 SQL、判断读写类型、路由到对应节点、 甚至做负载均衡和故障转移。

中间件代理架构
  ├─ 应用服务（像连接单一数据库一样连接中间件）
  ├─ 中间件（ProxySQL / MyCat）
  │    ├─ 解析 SQL，判断是读还是写
  │    ├─ 写 → 路由到主库
  │    └─ 读 → 按负载均衡策略路由到某个从库
  ├─ 优点：对业务代码透明，统一管理路由规则、监控、故障转移
  └─ 缺点：多一层网络跳转有性能损耗，中间件本身也要考虑高可用（避免单点）
维度应用层路由中间件代理
接入成本需要改造代码/配置数据源业务基本无感知
灵活性可按业务定制路由策略（如强制读主）依赖中间件本身支持的规则能力
运维复杂度路由逻辑分散在各服务里集中管理，但多一个需要维护的组件
性能损耗无额外网络跳转多一跳网络代理，有一定延迟

必须面对的问题：读到从库的旧数据

读写分离架构天生要面对上一篇讲过的主从延迟问题：写请求刚提交到主库，紧接着的读请求如果被路由到还没追上进度的从库，就会读到旧数据。 这在很多业务场景下是不可接受的，比如：

  • 用户刚发布一条动态，刷新页面却看不到自己刚发的内容。
  • 下单后立刻查询订单状态，返回"订单不存在"。
  • 修改密码后用新密码登录失败，因为登录校验请求打到了还没同步过来的从库。

常见应对策略（与上一篇主从延迟篇呼应）：

  • 写后读强制走主库：最简单粗暴也最常用的方案，在写请求完成后的一小段时间窗口内（或者同一个用户会话内），强制该用户的读请求路由到主库。
  • 会话粘滞（Session Sticky）：同一个事务/会话内的读写都路由到主库，跨会话的读才走从库，减少"自己写完自己读不到"的体验问题。
  • 基于 GTID/位点等待：中间件记录写请求提交时的位点，后续读请求先等从库追上该位点（或超时后降级读主库）。
  • 业务可接受最终一致：如果业务本身对"稍微旧一点的数据"不敏感（如商品详情页浏览量、历史订单列表查询），可以直接接受从库延迟，不做特殊处理。
常见误区："上了读写分离之后是不是所有 SELECT 都应该走从库？" 不是。哪些查询必须读主库、哪些可以容忍延迟读从库， 需要按业务场景逐个判断——一刀切地把所有读请求都甩给从库，是读写分离踩坑最常见的原因，往往是上线后才发现"用户反馈自己的操作结果看不到"才回头补救。 正确的做法是在设计阶段就明确哪些接口对一致性敏感，提前做好路由策略。
面试高频："读写分离怎么解决数据不一致的问题？" 核心答案：读写分离本身不能消除主从延迟，只能通过"强制读主库"、"会话粘滞"、"等待位点对齐"等策略在特定场景下规避用户能感知到的不一致，本质是在性能收益和一致性体验之间做业务级别的权衡，而不是技术上根治延迟。

核心价值总结

  • 读写分离的收益是横向扩展读吞吐，代价是引入了主从延迟带来的一致性问题，二者是一体两面。
  • 应用层路由灵活但侵入代码，中间件代理透明但多一层依赖，选型要结合团队技术栈和运维能力。
  • 读写分离必须结合前一篇讲的延迟监控和应对策略一起设计，不能只关注"怎么把读流量分出去"而忽视"分出去之后数据可能是旧的"这个核心矛盾。
