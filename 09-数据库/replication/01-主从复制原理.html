复制与高可用

主从复制原理

MySQL 复制的本质，是把主库上发生的"数据变更事件"以日志的形式流式传输给从库，再由从库重放一遍——这套机制撑起了读写分离、异地容灾、数据分析等几乎所有高可用架构。

一句话讲清楚

主从复制不是"主库把数据文件同步给从库"，而是主库把自己执行过的变更写成一份日志（binlog），从库把这份日志原样"重放"一遍， 从而让两边的数据最终保持一致。这是一种基于日志的异步事件流复制，而不是数据文件级别的同步。

类比：主库像一个记账本很勤快的老板，每做一笔生意就把流水记在账本（binlog）上；从库是分店，专门派一个"抄写员"（I/O 线程）盯着总店账本抄新增的流水到本地草稿本（relay log）， 再派一个"记账员"（SQL 线程）照着草稿本把每一笔生意在本地重新做一遍。整个过程分店和总店之间没有直接拷贝数据文件，靠的全是这本流水账。

复制的三个核心角色与流程

MySQL 复制涉及主库 1 个线程 + 从库 2 个线程，一共三个角色协同工作：

Master（主库）
  ├─ 客户端执行写操作（INSERT/UPDATE/DELETE/DDL）
  ├─ 事务提交时，把变更写入 binlog（二进制日志，顺序追加写）
  └─ Binlog Dump 线程：从库来连接后，主库为每个从库启动一个该线程，负责把 binlog 事件推送给从库

Slave（从库）
  ├─ I/O 线程：连接主库，请求 binlog 内容，写入本地的 relay log（中继日志，格式与 binlog 一致）
  └─ SQL 线程（或多线程并行复制）：读取 relay log，在从库上重放这些变更，应用到自己的数据文件

完整流程可以概括为四步：

  1. 主库写 binlog：事务提交（严格说是在 InnoDB 两阶段提交流程中），变更事件被写入 binlog 文件。
  2. 从库 I/O 线程拉取：从库主动向主库发起连接（复制是"拉"模式而不是主库主动"推"给所有从库），请求从某个 binlog 位点（或 GTID）之后的内容，主库的 Binlog Dump 线程把新增内容发过去。
  3. 写入 relay log：从库 I/O 线程收到的 binlog 事件，原样落盘到本地 relay log，这一步只是"抄写"，不做任何数据变更。
  4. SQL 线程重放：SQL 线程按顺序读取 relay log，把里面记录的变更在从库上重新执行一遍，数据才真正落到从库的表里。
常见误区：很多初学者以为主库和从库的数据是"实时同步"的，其实默认的复制模式是异步复制（Asynchronous Replication）—— 主库提交事务后立刻返回给客户端，完全不关心从库有没有收到、有没有回放完，这中间必然存在一个时间差（也就是后面要讲的"主从延迟"）。 这是理解读写分离、高可用容灾方案的前提：默认情况下，从库的数据永远比主库"旧一点"。

binlog 和 redo log 是两回事

这是一个高频面试坑：redo log 是 InnoDB 存储引擎层的日志，只服务于崩溃恢复，保证事务的持久性（D），格式是物理页的变更记录，写满会循环覆盖， 从来不会跨机器传输。binlog 是 MySQL Server 层的日志，与存储引擎无关（哪怕是 MyISAM 也有 binlog）， 专门用来做数据复制和基于时间点的恢复，日志不会被覆盖（除非手动清理或过期删除），可以传给别的实例重放。

面试高频："redo log 和 binlog 有什么区别？为什么两份日志都要写？" 因为它们各司其职：redo log 保证单机崩溃后事务不丢； binlog 让数据可以传播到其它实例。两者通过"两阶段提交（2PC）"机制保证一致——这个细节会在存储引擎内核 Tab 的 Redo Log 篇里详细展开，这里只需要知道二者是完全独立、职责不同的两套日志。

核心价值总结

  • 复制的本质是"日志重放"，而不是文件同步，这决定了它天然是异步、有延迟的。
  • 三个角色（Binlog Dump 线程、I/O 线程、SQL 线程）协作，靠 binlog 和 relay log 两份日志解耦——即便网络短暂中断，从库也能从上次的位点继续拉取，不需要重新全量同步。
  • 默认是异步复制，这是后续讲半同步复制、主从延迟、读写分离的地基，务必先建立这个心智模型。

与其他知识点的联系

  • Redo Log：binlog 和 redo log 是两套独立的日志体系，本文已经详细对比过——redo log 是 InnoDB 引擎层日志，只服务本机崩溃恢复，写满即循环覆盖，从不跨机器传输；binlog 是 Server 层日志，专门为复制而生，持续追加、不会被覆盖。理解这个区分，才能明白"为什么复制传的是 binlog 而不是 redo log"，详见 Redo Log 与 WAL 机制。
  • 半同步复制与全同步复制：本文讲的是默认的异步复制模式——主库提交后完全不等从库确认，这也是"主库崩溃可能丢失已提交但未传给从库的事务"这一风险的来源。半同步复制正是针对这个风险，在异步复制的基础流程上加了一道"至少一个从库确认收到"的等待，是对本文流程的直接增强。
  • 主从延迟问题及优化：本文强调复制默认是异步的，从库数据天然比主库"旧一点"，这个时间差就是主从延迟的定义本身。I/O 线程拉取、SQL 线程重放这两步各自的耗时（网络传输、单线程回放瓶颈）正是延迟的具体成因，下一篇会展开讲如何优化。
  • InnoDB 与 MyISAM 对比：binlog 是 Server 层日志，与存储引擎无关，这意味着无论表用 InnoDB 还是 MyISAM，都可以参与主从复制；但涉及事务一致性的场景（InnoDB 与 binlog 之间靠两阶段提交保证一致）只有 InnoDB 能提供完整保证，这也是生产环境几乎不会用 MyISAM 作为主库存储引擎的原因之一。
