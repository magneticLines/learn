半同步复制与全同步复制

默认的异步复制快但不安全，全同步复制安全但几乎不可用，半同步复制是 MySQL 在"性能"和"数据不丢失"之间找到的一个工程折中点。

一句话讲清楚

三种复制模式的区别在于：主库提交事务前，要不要等从库确认，等多少个从库确认。 异步复制完全不等；全同步复制要等所有从库都确认收到并应用完；半同步复制退一步，只要求至少一个从库确认"收到"了日志即可返回，不要求从库已经应用完。

类比：异步复制像老板记完账直接去干下一件事，完全不管分店有没有收到账本副本； 全同步复制像老板必须等所有分店都确认"账本抄好了、账也记完了"才能继续做生意，只要有一家分店网络卡顿，整个总店都要停摆； 半同步复制则是老板只要等其中一家分店说"我已经把账本内容抄到我这儿了"（不要求分店已经把账记完），就可以接着做下一单——多一次"确认收到"的开销，但不会被单点拖死。

三种模式详解

1. 异步复制（Asynchronous Replication，默认模式）

主库提交事务、写完 binlog 后立即返回给客户端，完全不关心从库是否收到。这是上一篇讲的默认流程。 优点是主库性能不受从库拖累；缺点是如果主库刚提交完事务就宕机，binlog 还没来得及传给任何从库， 那么这个事务在主库上是已提交的，但在所有从库上都不存在——一旦主库不可恢复，这部分数据就永久丢失了。

2. 全同步复制（Fully Synchronous Replication）

主库要等所有从库都执行完这个事务才能给客户端返回成功。这样能保证主从数据强一致， 但代价是：只要有一个从库变慢、宕机或者网络抖动，整个集群的写入都会被阻塞。在实际生产中， MySQL 官方复制机制几乎不提供这种模式的直接支持，因为可用性代价太高，通常只在理论讨论或者特殊的强一致场景（配合 Paxos/Raft 类协议）才会出现。

3. 半同步复制（Semisynchronous Replication，MySQL 5.5+ 通过插件提供，5.7+ 增强为 Loss-Less 半同步）

主库提交事务后，至少要等一个从库的 I/O 线程确认"收到了 relay log"（注意：只是收到写入 relay log，不要求 SQL 线程已经重放完）， 主库才返回成功给客户端。如果等待超时（rpl_semi_sync_master_timeout），会自动降级为异步复制，保证可用性不受影响， 但此时也就失去了数据安全保证，直到有从库重新跟上进度才会恢复半同步模式。

开启半同步复制（简化示例，主库端）
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_timeout = 1000; -- 单位毫秒，超时后自动降级为异步

三种模式对比

模式主库等待条件性能影响数据安全性可用性风险
异步复制 不等，提交即返回 最好 最低：主库宕机可能丢数据 无（从库故障不影响主库）
半同步复制 至少 1 个从库确认收到 relay log 有一定延迟（一次网络往返） 较高：已确认的事务不会因主库宕机丢失 低（超时自动降级为异步，不阻塞写入）
全同步复制 所有从库都应用完事务 最差，受最慢节点拖累 最高：真正的强一致 高：任一从库故障都会阻塞写入
常见误区："开启了半同步复制，是不是从库上就能马上读到最新数据了？" 不是。半同步复制只保证"至少一个从库收到了日志内容"，并不保证这个从库已经把日志应用（重放）完。 读写分离场景下客户端读到的仍然是从库当前已经重放到的数据，半同步复制解决的是数据丢失风险，不是主从延迟问题，这是两个独立的话题（延迟问题下一篇专门讲）。
面试高频："半同步复制解决了什么问题？"标准答案：解决异步复制下主库宕机可能导致已提交事务丢失的问题， 通过"至少一个从库确认收到日志才返回"的机制，在性能损耗可控的前提下提升数据安全性；但它不保证强一致，超时会自动降级为异步以保证可用性， 是在 CAP/可用性与一致性之间做的工程折中，不是"银弹"。

核心价值总结

  • 异步复制快但可能丢数据，全同步复制安全但几乎不可用，半同步复制是中间的工程折中。
  • 半同步复制保证的是"数据不丢"，而不是"数据不延迟"——理解这个边界，才不会在方案选型时张冠李戴。
  • 对数据安全性要求极高的场景（如金融交易），会考虑半同步复制配合多个从库，或者直接使用下面高可用篇要讲的 MGR 方案获得更强的一致性保证。
