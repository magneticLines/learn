binlog 三种格式（statement / row / mixed）

binlog 里到底记的是"我执行了什么 SQL"还是"我改动了哪些数据行"，这个选择直接决定了主从数据能不能保持一致，也是无数生产事故的根源。

一句话讲清楚

binlog 记录变更事件的方式有三种：statement（记录 SQL 语句本身）、row（记录每一行数据实际变化前后的值）、 mixed（混合，MySQL 自动判断用哪种）。三者的核心区别是：statement 记的是"做什么"，row 记的是"结果是什么"。

类比：statement 格式像是把菜谱（"放两勺盐、炒三分钟"）传给分店的厨师，让他自己照着做一遍； row 格式则是直接把炒好的成品菜（"最终这盘菜是什么味道、什么样子"）打包送过去。 如果菜谱里写的是"放当季最新鲜的盐"这种模糊指令（对应 SQL 里的 NOW()、RAND() 这类不确定函数）， 两个厨师在不同时间照做，结果可能完全不同；而直接送成品菜就不存在这个问题。

三种格式的原理

格式记录内容日志量一致性风险
statement 原始 SQL 语句文本 小（一条 UPDATE 影响10万行，也只记一条 SQL） 高：依赖不确定函数、触发器、存储过程等场景可能导致主从数据不一致
row（MySQL 5.7.7+ 默认） 每一行变更前后的实际列值（before-image / after-image） 大（大事务、批量更新时日志膨胀明显） 低：记的是确定性的结果，从库重放不会有歧义
mixed 默认走 statement，遇到不安全语句（如包含 NOW()、UUID()、非确定性 UPDATE 等）自动切换成 row 介于两者之间 较低，但需要理解 MySQL 判定"不安全"的规则，调试成本比纯 row 更高

为什么 statement 格式会导致主从不一致

典型的反例是使用了不确定性函数：

statement 格式的隐患示例
UPDATE orders SET created_at = NOW() WHERE id = 1001;
-- statement 格式：binlog 里原样记录这条 SQL
-- 从库重放时会重新执行一次 NOW()，此时的时间已经和主库执行时不同
-- 主库 created_at = 10:00:00.123，从库重放后 created_at = 10:00:03.456
-- 主从数据出现不一致，且不会报任何错，非常隐蔽

类似的坑还包括：

  • 非确定性函数：NOW()、RAND()、UUID()、CONNECTION_ID() 等，每次执行结果都可能不同。
  • 依赖 AUTO_INCREMENT 的多行插入：并发场景下多个连接交叉插入时，SQL 层面的执行顺序在从库上未必和主库完全一致。
  • 存储过程 / 触发器 / UDF：如果内部使用了不确定性逻辑，同一条调用语句在从库重放时可能产生不同结果。
  • LIMIT 且没有 ORDER BY 的 UPDATE/DELETE：没有确定排序时，"命中哪几行"本身就是不确定的。

而 row 格式记录的是"这一行从值 A 变成了值 B"，不管 NOW() 求值结果是什么，binlog 里落地的都是具体的时间戳， 从库只需要"把这一行改成这个值"，不存在重新计算的过程，天然规避了这类不确定性问题。

常见误区：不要以为"mixed 格式已经把不安全的语句都转成 row 了，所以可以放心用 statement 的日志量优势"。 mixed 模式下的判定规则相当复杂（涉及数百个规则和边界条件，各版本还有变化），排查问题时经常要先确认"这条语句到底被记成了 row 还是 statement"， 调试成本并不比直接用纯 row 格式低多少。目前生产环境的主流选择就是直接使用 row 格式，用日志量换取确定性。

row 格式的代价与应对

row 格式最大的问题是日志量：一条不带 WHERE 的 UPDATE users SET status = 1 如果影响 100 万行， row 格式下 binlog 要记录 100 万条行变更，而 statement 格式只需要记一行 SQL。这直接影响：

  • 磁盘占用和 IO：大事务、批量操作场景下 binlog 文件增长迅速。
  • 主从延迟：从库需要重放这 100 万条行变更，SQL 线程处理时间被拉长（下一篇会展开讲主从延迟的成因）。
  • 网络带宽：binlog 需要通过网络传给从库，日志量大意味着传输时间更长。

工程上常见的缓解手段：把大事务拆成多个小批次执行（如分批 UPDATE ... LIMIT 1000）； 开启 binlog_row_image=MINIMAL（只记录变化的列而不是整行）减少日志量；对于超大表的 DDL 使用 gh-ost/pt-online-schema-change 等工具规避直接产生海量 binlog。

面试高频："binlog 有几种格式，生产环境用哪种，为什么？" 标准答案：三种格式 statement/row/mixed； 生产环境优先用 row，因为它记录的是确定性的行变化结果，能规避 statement 格式在非确定性函数、触发器等场景下的主从数据不一致风险； 代价是日志量变大，需要结合 binlog_row_image 等参数和分批操作来控制。

核心价值总结

  • statement 省空间但不安全，row 安全但费空间，mixed 是过渡期的折中方案，现在已不推荐深度依赖。
  • 判断一致性风险的关键在于"操作是否具有确定性"——只要 SQL 的执行结果可能因为执行时间、执行环境不同而变化，statement 格式就有隐患。
  • 这个知识点直接关联下一篇要讲的半同步/异步复制，以及后面的主从延迟问题——理解 binlog 格式是理解整条复制链路的基础。
