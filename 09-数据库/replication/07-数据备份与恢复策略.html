数据备份与恢复策略

复制解决的是"实时容灾"，但误删数据、程序 bug 写脏数据这类问题复制根本救不了（脏数据会原封不动地同步到所有从库）——这时候能救命的只有备份。

一句话讲清楚

备份分两大类：逻辑备份（把数据导出成一条条可以重新执行的 SQL 语句，代表工具 mysqldump）和 物理备份（直接拷贝数据文件本身，代表工具 Percona Xtrabackup）。 生产环境的标准做法是全量物理备份 + 增量 binlog 组合，而不是单独依赖 mysqldump。

类比：逻辑备份像是把整个图书馆的书目内容重新誊抄一份文字记录（"《高性能MySQL》共400页，第一页写着……"）， 恢复的时候要照着记录把每本书重新印刷装订一遍，速度慢但格式通用、方便挑着恢复某几本书； 物理备份则是直接把图书馆的实体书架原样打包搬走，恢复时直接把书架放回去就行，速度快很多，但打包和摆放过程要格外小心不能有偏差（要保证文件级别的一致性）。

逻辑备份：mysqldump

mysqldump 通过连接数据库执行 SELECT，把表结构和数据转换成一条条 CREATE TABLE / INSERT 语句写入文件， 恢复时就是把这些 SQL 重新执行一遍。

mysqldump 常见用法
# 全库逻辑备份，--single-transaction 依赖 InnoDB 的 MVCC 快照读
# 保证导出期间读到的是一致的快照，而不用对整个库加锁
mysqldump --single-transaction --master-data=2 -u root -p --all-databases > full_backup.sql

# 恢复：把导出的 SQL 重新跑一遍
# 大表恢复时相当于重新逐行 INSERT，速度明显慢于物理备份的恢复方式
mysql -u root -p < full_backup.sql

mysqldump 的问题在大库场景下会被明显放大：

  • 备份耗时随数据量线性增长：本质是在跑大量 SELECT，几十 GB 的表可能要跑几个小时。
  • 恢复更慢：恢复是重新执行 INSERT 语句、重建索引，同样数据量的恢复往往比物理备份慢一个数量级。
  • 对源库有压力：即便用了 --single-transaction 避免长时间锁表，大量的读 IO 依然会挤占 Buffer Pool、影响线上查询性能。
  • 不支持增量：mysqldump 每次都是全量导出，没有原生的增量备份能力。

物理备份：Xtrabackup

Percona Xtrabackup（开源免费）直接在文件系统层面拷贝 InnoDB 的数据文件（.ibd），并且能在不锁表、不阻塞正常读写的情况下完成， 原理是：拷贝数据文件的同时持续追踪拷贝期间产生的 redo log，备份完成后用这部分 redo log 把数据文件"补齐"到一个一致的时间点（类似崩溃恢复的思路，可以回顾存储引擎内核 Tab 的 Redo Log 与崩溃恢复篇加深理解）。

Xtrabackup 全量 + 增量备份示例
# 全量物理备份
xtrabackup --backup --target-dir=/backup/full

# 基于上一次全量备份做增量备份，只拷贝变化的数据页
# 大幅减少后续备份的时间和存储空间
xtrabackup --backup --target-dir=/backup/inc1 --incremental-basedir=/backup/full

# 恢复：先 prepare 把 redo log 应用进数据文件使其一致，再 copy-back 拷回数据目录
xtrabackup --prepare --apply-log-only --target-dir=/backup/full
xtrabackup --prepare --target-dir=/backup/full --incremental-dir=/backup/inc1
xtrabackup --copy-back --target-dir=/backup/full
维度逻辑备份（mysqldump）物理备份（Xtrabackup）
备份内容SQL 语句（CREATE/INSERT）数据文件本身
备份速度慢，随数据量线性增长快，接近文件拷贝速度
恢复速度慢，需要重新执行 SQL、重建索引快，直接使用数据文件
是否支持增量不支持支持（基于 LSN 增量拷贝变化页）
跨版本/跨引擎兼容性好，SQL 文本可读、可编辑、可迁移到不同版本较差，通常要求 MySQL 版本、存储引擎相近
适用场景小库、需要人工可读的备份、跨版本迁移大库、生产环境定期全量+增量备份的主力方案
为什么大库不能只用 mysqldump：假设一个几百 GB 的库，用 mysqldump 全量备份可能要跑数小时，期间对源库的读压力不可忽视； 一旦需要恢复，重新执行几百 GB 的 INSERT 语句、重建所有索引，耗时可能是以"天"为单位——这在任何有 RTO（恢复目标时间）要求的生产环境里都是不可接受的。 Xtrabackup 的物理备份 + 增量备份能把备份时间和恢复时间都压缩到可控范围内，这是大库场景下几乎没有选择余地的方案。

全量备份 + 增量 binlog 的组合策略

生产环境标准做法是：定期全量物理备份（如每周一次）+ 期间持续保留 binlog（相当于全量备份之后的所有增量变更）。 恢复时的流程是：

  1. 用最近一次全量备份恢复出一个基准状态。
  2. 找到该次全量备份对应的 binlog 位点（或 GTID）。
  3. 用 mysqlbinlog 把该位点之后的所有 binlog 重放到恢复出来的实例上，追上到故障发生前的最后一刻（甚至可以精确恢复到"误删除之前的那一秒"）。
基于 binlog 的时间点恢复（Point-in-Time Recovery）
# 从全量备份对应的位点开始，重放到误操作发生前的那一刻，跳过误操作本身
# 这也是为什么 binlog 不能只保留很短时间——它是"全量备份之后所有变更"的唯一凭证
mysqlbinlog --start-position=154 --stop-datetime="2026-07-03 10:29:59" \
  mysql-bin.000123 | mysql -u root -p

这套组合策略正是为什么 binlog 不仅仅是复制链路的基础设施，也是备份体系里不可或缺的一环——它把"两次全量备份之间"这段时间的变更完整记录了下来， 是实现"精确恢复到任意时间点"（Point-in-Time Recovery）的唯一途径。

面试高频："你们的备份策略是什么？" 一个合格的回答结构：全量物理备份（周期性，如每天/每周）用 Xtrabackup 保证备份和恢复速度可控； binlog 持续保留作为增量凭证，支持时间点恢复；定期做恢复演练验证备份可用性（很多事故发生在"以为有备份，真恢复时才发现备份文件损坏或不完整"）； 根据业务的 RPO/RTO 要求决定备份频率和 binlog 保留时长。

核心价值总结

  • 逻辑备份灵活可读但慢，物理备份快但可读性差，大库场景优先选物理备份。
  • 全量备份 + binlog 增量是标准组合，binlog 在这里的角色和它在复制链路里的角色是一致的——都是"完整的变更事件流"。
  • 备份策略不是"配置一次就完事"，一定要定期做恢复演练，否则备份文件本身出问题时根本发现不了，等真正需要恢复才追悔莫及。
