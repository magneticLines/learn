事务与并发控制

行锁 / 表锁 / 间隙锁 / 临键锁

锁的粒度决定了并发度的上限。从表锁到行锁是第一次取舍，从"只锁存在的行"到"连行与行之间的空隙也锁住"是第二次取舍——后者正是 InnoDB 堵住幻读的关键武器。

粒度取舍：表锁 vs 行锁

表锁行锁
锁定范围整张表符合条件的具体行
并发度低——一个事务锁表，其他事务的读写全部等待高——不同行之间互不影响
开销加锁快，内存开销小需要为每一行维护锁信息，开销相对大
适用场景批量 DDL、MyISAM 引擎（只支持表锁）InnoDB 默认的并发写场景
类比：表锁像是"锁整个房间"——谁进这个房间干什么都要排队；行锁像是"只锁房间里那一张具体的桌子"——别人可以同时进来用别的桌子。粒度越细，冲突概率越低，并发度越高，但维护成本也越高。

InnoDB 通过索引来实现行锁——行锁本质是锁在索引记录上的。这意味着如果 WHERE 条件没有用上索引，InnoDB 只能退化成锁全表（扫描到的每一行都加锁，效果等同于表锁）。这是索引设计影响锁粒度的一个重要陷阱。

为什么行锁还不够：幻读的漏洞

回顾上一篇：行锁只能锁住已经存在的行。如果事务 A 执行 SELECT * FROM t WHERE id BETWEEN 10 AND 20 FOR UPDATE， 行锁只会锁住当前查到的那几行——但 id=15 这个位置目前是空的，行锁根本锁不住"空位"。 此时事务 B 完全可以往这个空位插入一条 id=15 的新记录并提交，导致事务 A 再次查询时行数变多，出现幻读。

间隙锁（Gap Lock）：锁住"空隙"本身

间隙锁的思路很直接：不仅锁住已存在的行，还要把行与行之间的"间隙"也锁起来，禁止其他事务在这个间隙里插入新记录。 间隙锁只在可重复读（RR）及以上隔离级别下生效（RC 级别不需要防幻读，所以不加间隙锁）。

间隙锁示例（id 索引上现有值：5, 10, 20）
-- 事务 A：查询 id 在 10 到 20 之间的记录并加锁
SELECT * FROM t WHERE id BETWEEN 10 AND 20 FOR UPDATE;
-- InnoDB 会在 (5,10)、(10,20)、(20,+∞) 这些间隙上加间隙锁
-- 事务 B 此时插入 id=15 会被阻塞，因为 15 落在被锁住的间隙 (10,20) 里
INSERT INTO t (id) VALUES (15); -- 阻塞，直到事务 A 提交或回滚
注意：间隙锁之间是互相兼容的——两个事务都可以同时持有同一个间隙的间隙锁（因为间隙锁的目的只是阻止插入，不是阻止并发读取间隙这个"概念"），这一点和行锁的 X 锁互斥不同，容易搞混。间隙锁真正阻塞的是"往间隙里插入新记录"的动作。

临键锁（Next-Key Lock）：行锁 + 间隙锁的组合拳

临键锁 = 行锁（锁住记录本身） + 间隙锁（锁住该记录前面的间隙），是一个左开右闭的区间。 InnoDB 在 RR 级别下，默认用临键锁给索引记录加锁，这样既防止了别的事务修改已有行（行锁的作用），又防止了别的事务在间隙里插入新行（间隙锁的作用），双重保险堵住幻读。

索引 id 上的现有值: 5 ─── 10 ─── 20 ─── +∞ 临键锁区间划分: (-∞, 5] (5, 10] (10, 20] (20, +∞) 每个临键锁 = "一个间隙" + "间隙右边紧邻的那条记录"

三种锁的对比总结

锁类型锁定对象解决的问题生效隔离级别
行锁（Record Lock）单条已存在的索引记录防止并发修改同一行所有级别
间隙锁（Gap Lock）两条记录之间的空隙（不含记录本身）防止在空隙里插入新记录，堵幻读RR 及以上
临键锁（Next-Key Lock）行锁 + 前面的间隙锁综合防止修改行和插入新行RR 及以上（默认）

常见误区

误区：以为间隙锁和临键锁只在 BETWEEN/范围查询时才会加。实际上只要在 RR 级别下用索引做当前读（UPDATE/DELETE/SELECT...FOR UPDATE），即便是等值查询，只要查询的值不存在，InnoDB 也会在这个值所在的间隙上加间隙锁，防止别人插入这个值。
面试高频："临键锁退化成行锁或间隙锁的场景？" 当等值查询命中一个唯一索引上确实存在的记录时，临键锁会退化为只有行锁（不需要锁间隙，因为这一行已经唯一确定，不存在插入同值记录的问题）；当查询条件在索引最右侧、没有匹配的具体行时，会退化为纯粹的间隙锁（俗称 "supremum" 间隙）。

核心价值总结

  • 行锁只能保护"已存在的行"，无法阻止新行插入，这是幻读的根源。
  • 间隙锁负责封锁空隙，临键锁 = 行锁 + 间隙锁，是 InnoDB 在 RR 级别下默认使用的组合，用来同时防范修改和插入两类幻读诱因。
