事务与并发控制

MVCC 原理（多版本并发控制）

本 Tab 最核心的考点。MVCC 的野心是：让"读"完全不用加锁，也能读到符合隔离级别语义的正确数据，从而实现读写不互相阻塞。拆开看它靠三个零件拼出来的魔法：隐藏列、undo log 版本链、ReadView。

MVCC 要解决的问题

如果读也要加锁（比如串行化级别那样），读写之间会互相阻塞，并发性能大打折扣。 MVCC（Multi-Version Concurrency Control，多版本并发控制）的核心思想是：不修改原数据，而是为每一行保留多个历史版本，让读操作可以选择读某个历史版本，而不必等待写操作释放锁。 这样"读"永远不阻塞"写"，"写"也不阻塞"读"——只有"写"和"写"之间才需要用锁互斥。

类比：MVCC 像是 Git 的版本历史——你在看某个历史 commit 的内容时，完全不影响别人同时往仓库里继续提交新的 commit。每次 UPDATE 就像生成了一个新版本，而旧版本并没有被销毁，只是被链接到了历史记录里，需要时可以"检出"查看。

零件一：两个隐藏列

InnoDB 会给每一行数据额外加上两个用户不可见的隐藏列（此外还有一个 row_id，与 MVCC 无关这里不展开）：

隐藏列含义
trx_id记录最近一次修改（插入/更新）这行数据的事务 ID，是一个全局递增的编号
roll_pointer回滚指针，指向这一行被修改前的旧版本在 undo log 中的位置

零件二：undo log 版本链

每次事务修改一行数据前，都会把旧版本的数据写入 undo log，并让新版本的 roll_pointer 指向这条 undo log 记录。 多次修改之后，这些 undo log 记录会通过 roll_pointer 串成一条版本链，链表头是最新版本，顺着链表往下走能找到任意历史版本。

假设账户余额初始为 1000，先后被事务 100、事务 200 修改： 当前行(最新版本): balance=700, trx_id=200, roll_pointer ──┐ │ ┌───────────────────────────────┘ ▼ undo log 版本: balance=800, trx_id=100, roll_pointer ──┐ │ ┌───────────────────────────────┘ ▼ undo log 版本: balance=1000, trx_id=90(最初插入), roll_pointer = NULL

这条链就是"多版本"里"多版本"的字面意思——同一行数据同时存在好几份历史快照，靠 roll_pointer 串起来。

零件三：ReadView（读视图）的可见性判断

有了版本链，接下来的问题是：一个事务开始查询时，应该在这条链上停在哪个版本？这就是 ReadView 要回答的问题。 ReadView 本质是一个"快照"，记录了生成这个 ReadView 时刻，系统中所有活跃（未提交）事务的信息，包含四个关键字段：

字段含义
m_ids生成 ReadView 时，当前系统中所有活跃（已开始但未提交）事务的 ID 列表
min_trx_idm_ids 中的最小值
max_trx_id系统预分配给下一个事务的 ID（即当前最大事务 ID + 1）
creator_trx_id生成这个 ReadView 的事务自己的 ID

可见性判断规则

对于版本链上某个版本的 trx_id，按下面的规则依次判断是否可见（能不能读到这个版本）：

  1. 如果 trx_id == creator_trx_id：这是自己修改的，可见（自己当然能看到自己的修改）。
  2. 如果 trx_id < min_trx_id：这个版本是在生成 ReadView 之前就已经提交的事务改的，可见。
  3. 如果 trx_id >= max_trx_id：这个版本是 ReadView 生成之后才开始的事务改的，不可见（未来的事务，不该看到）。
  4. 如果 min_trx_id <= trx_id < max_trx_id：说明这个事务在生成 ReadView 时活跃：
    • 如果它在 m_ids 列表里 → 说明还没提交 → 不可见
    • 如果它不在 m_ids 列表里 → 说明在生成 ReadView 之前已经提交了 → 可见

如果当前版本不可见，就顺着 roll_pointer 沿版本链往下找上一个版本，重复上面的判断，直到找到一个可见版本为止（如果全部不可见，说明这行数据对当前事务来说"根本不存在"）。

核心问题：RC 和 RR 的 ReadView 生成时机不同

这是最容易混淆但也是最关键的一点——RC 和 RR 用的是同一套 ReadView 机制，唯一的区别是什么时候生成 ReadView：

隔离级别ReadView 生成时机效果
读已提交（RC） 每次执行 SELECT 语句时都重新生成一份 ReadView 能看到其他事务在此期间刚提交的最新数据 → 同一事务内多次读结果可能不同（不可重复读）
可重复读（RR） 只在事务内第一次执行快照读时生成 ReadView，之后整个事务复用这一份 事务全程看到的都是同一个时间点的快照 → 同一事务内多次读结果一致（可重复读）
这就是"可重复读"名字的由来：不是因为 InnoDB 加了什么特殊的锁让你"重复读到一样的东西"，而是因为 ReadView 只生成一次、全程复用，版本链的可见性判断结果自然就一直不变。这是理解 RC/RR 行为差异最核心的一句话。

为什么 MVCC 能让"读不加锁"

因为普通的 SELECT（快照读）走的是 ReadView + 版本链的逻辑判断，全程不需要对数据行加任何锁——它读的是版本链上某个历史版本的"快照"，而不是当前正在被修改的最新数据，所以完全不会和正在执行 UPDATE 的事务产生锁冲突。 只有当前读（SELECT ... FOR UPDATE、UPDATE、DELETE）才需要读取最新版本并加锁，因为这些操作必须基于最新数据做修改，不能容忍脏数据。

常见误区

误区：以为 MVCC 能完全替代锁，事务并发控制不再需要锁。实际上 MVCC 只解决"读"的并发问题，"写-写冲突"依然必须靠锁（行锁/间隙锁）来解决，MVCC 和锁是配合工作、缺一不可的两套机制。
误区：以为 RR 级别下"事务开始"（BEGIN）那一刻就生成了 ReadView。实际上是"第一条快照读语句执行时"才生成，BEGIN 本身只是标记事务的起点，不触发 ReadView 生成——这个细节在分析边界场景时很容易踩坑。

面试高频考点

问："MVCC 是怎么实现的？"
标准答案分三层：① 每行数据有 trx_id 和 roll_pointer 两个隐藏列；② 通过 roll_pointer 把 undo log 中的历史版本串成版本链；③ 事务通过 ReadView 记录的活跃事务列表，按可见性规则判断版本链上哪个版本对自己可见。RC 和 RR 的区别就在于 ReadView 生成一次还是每次查询都生成。

核心价值总结

  • MVCC 的三大零件：隐藏列（定位版本）、undo log 版本链（保存历史）、ReadView（判断可见性）。
  • MVCC 让读写不互相阻塞，是 InnoDB 高并发性能的核心支柱之一。
  • RC 每次查询生成新 ReadView，RR 整个事务复用同一份，这是二者行为差异的唯一根源，务必吃透。
  • 下一篇讲死锁——当锁和 MVCC 都无法避免多个事务互相等待时，InnoDB 如何检测并打破僵局。

与其他知识点的联系

  • 事务的隔离级别：本篇的核心结论——RC 每次快照读都生成新 ReadView、RR 整个事务复用同一份——直接解释了两个隔离级别行为差异的根源，是理解"隔离级别为什么这样设计"的最后一块拼图。
  • 脏读/不可重复读：脏读靠 ReadView 可见性规则里"只看已提交事务的修改"来避免，不可重复读靠"RR 全程复用同一份 ReadView"来避免，这两种异常本质上都是 MVCC 快照机制的直接产物，而不是靠锁挡住的。
  • 行锁表锁间隙锁临键锁：MVCC 只管"读"这一条腿，"写-写冲突"依然要靠锁解决——本篇强调的"读不加锁"和上一篇的行锁/间隙锁是并发控制的两条并行路径，快照读走 MVCC，当前读走加锁，二者配合才完整覆盖了并发场景。
  • 两阶段锁协议：MVCC 代表的是乐观的多版本并发控制思路（不加锁、靠版本判断可见性），而 2PL 代表的是悲观锁的思路（先加锁再操作），二者是数据库并发控制理论中两条并行的技术路线，各有取舍。
