事务与并发控制

死锁的产生与排查

锁解决了并发安全问题，却也带来了新麻烦：多个事务互相等待对方释放锁，谁都不肯先退让，系统就卡死了。好在 InnoDB 有一套自动检测机制，能主动识破僵局并选择一个"牺牲品"来打破它。

死锁产生的四个必要条件

这是操作系统课程里的经典理论，同样适用于数据库锁场景。四个条件同时满足才会发生死锁，破坏其中任意一个都能避免死锁：

条件含义在数据库锁场景中的体现
互斥条件资源同一时刻只能被一个事务占用X 锁天然互斥，两个事务不能同时持有同一行的 X 锁
请求与保持条件事务已经持有至少一个资源，同时又在请求新的资源事务已经拿到了 A 行的锁，又想申请 B 行的锁
不可剥夺条件已分配的资源不能被其他事务强行抢走，只能自己主动释放InnoDB 的锁只有在事务提交/回滚时才释放，别的事务无法强行剥夺
环路等待条件存在一个事务等待环路：A 等 B，B 等 C，……，最后又回到等 A典型两事务互锁：A 等 B 持有的锁，B 又在等 A 持有的锁

经典死锁场景演示

两个事务以相反顺序更新两行数据，是最常见的死锁诱因：

时间事务 A事务 B
T1BEGIN;BEGIN;
T2UPDATE t SET v=1 WHERE id=1;（获得 id=1 的 X 锁）UPDATE t SET v=2 WHERE id=2;（获得 id=2 的 X 锁）
T3UPDATE t SET v=3 WHERE id=2;（请求 id=2 的锁，被 B 占用，阻塞等待）UPDATE t SET v=4 WHERE id=1;（请求 id=1 的锁，被 A 占用，阻塞等待）
T4A 等 B 释放 id=2，B 等 A 释放 id=1 —— 形成环路等待，死锁产生
常见诱因：业务代码里多个事务对多张表/多行数据的加锁顺序不一致。修复思路很简单——让所有事务都按相同的顺序访问资源（比如都按主键从小到大更新），就能破坏"环路等待"这个条件，从根本上避免死锁。

MySQL 的死锁检测机制：wait-for graph

InnoDB 默认开启 innodb_deadlock_detect，维护一张"等待关系图"（wait-for graph）：每个事务是一个节点，如果事务 A 在等待事务 B 持有的锁，就画一条 A → B 的边。 每当有新的锁等待发生，InnoDB 就会检查这张图里有没有环——一旦出现环，就判定为死锁。

检测到死锁后，InnoDB 不会让所有涉事事务都傻等下去，而是自动选择一个"代价最小"的事务作为牺牲品，强制回滚它（通常是修改行数较少、undo log 量较小的那个），让另一个事务得以继续执行，从而打破僵局。

类比：就像两辆车在窄路上迎面相遇互不相让——与其两车都熄火等到天荒地老，不如交警过来直接让其中一辆倒车让路（回滚），让另一辆先通过。

实战排查：SHOW ENGINE INNODB STATUS

MySQL 8.0
SHOW ENGINE INNODB STATUS\G

输出中的 LATEST DETECTED DEADLOCK 小节记录了最近一次死锁的完整现场，重点关注这几处信息：

  • TRANSACTION：每个涉事事务的 ID、执行了多久、修改了多少行。
  • LOCK WAIT ... lock_mode X locks rec but not gap ...：具体在等待哪张表哪一行的哪种锁，能定位到 SQL 语句本身。
  • WE ROLL BACK TRANSACTION (n)：末尾会明确标注 InnoDB 选择回滚了哪一个事务。
工程实践：生产环境建议开启 innodb_print_all_deadlocks = ON，把每次死锁的详情打到错误日志（SHOW ENGINE INNODB STATUS 默认只保留最近一次），方便事后统一排查高频死锁的 SQL 模式。应用层收到死锁异常（错误码 1213）后，通常的处理方式是捕获异常并重试，因为死锁涉事的事务大概率重试后能成功。

面试高频考点

问："怎么减少死锁？"
从破坏四个必要条件的角度回答最系统：① 让所有事务以一致的顺序访问资源（破坏环路等待）；② 尽量用较低的隔离级别或更精确的索引减少锁的范围（减少互斥资源数量）；③ 缩短事务长度、尽快提交，减少持锁时间（减少请求与保持窗口）；④ 应用层做好死锁重试机制作为兜底。
常见误区：认为死锁是 bug，应该想办法彻底杜绝。实际上在高并发场景下死锁很难 100% 避免，业界的共识是"降低死锁概率 + 死锁发生后能自动恢复（重试）"，而不是追求零死锁。

核心价值总结

  • 死锁四条件：互斥、请求与保持、不可剥夺、环路等待，破坏任意一条即可避免。
  • InnoDB 用 wait-for graph 主动检测环路，自动回滚代价最小的事务，不需要人工干预即可打破僵局。
  • 排查死锁的第一手工具是 SHOW ENGINE INNODB STATUS，结合日志能定位到具体的加锁顺序和 SQL。
