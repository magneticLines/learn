事务的 ACID 特性

ACID 是事务的四条"宪法"，但它们不是四个平级的独立机制——A/I/D 各有专门的底层实现负责兜底，C 只是前三者共同作用后自然得到的结果。理解这一点，是看懂 InnoDB 事务实现的第一把钥匙。

先建立直觉：为什么需要事务

转账场景：把账户 A 的 100 元转给账户 B，本质是两条 SQL —— UPDATE A 减 100 和 UPDATE B 加 100。 如果第一条执行完机器断电，钱就凭空消失了。事务把这一组操作打包成一个不可分割的整体：要么全部生效，要么像什么都没发生过一样。这是 ACID 存在的根本原因。

类比：事务像是"一次性打包发货"——你在电商下单买了 3 件商品要求一起发，仓库要么把 3 件都装箱发出，要么因为缺货全部取消，不会出现"发了 2 件，第 3 件不知道去哪了"的中间状态。

四大特性逐个拆解

A —— 原子性（Atomicity）

事务中的所有操作，要么全部完成，要么全部不做。原子性由 undo log（回滚日志）保证：InnoDB 在修改数据前，会先把修改前的旧值记录到 undo log 中。 一旦事务中途失败或主动 ROLLBACK，就用 undo log 把数据"倒带"回事务开始前的样子。

转账事务的原子性
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1; -- undo log 记下 A 原来的余额
UPDATE account SET balance = balance + 100 WHERE id = 2; -- undo log 记下 B 原来的余额
-- 如果这里断电重启，InnoDB 崩溃恢复时会读 undo log 把两条 UPDATE 都撤销
COMMIT;

D —— 持久性（Durability）

一旦事务提交，修改的数据必须永久保存，即使随后系统崩溃也不会丢失。持久性由 redo log（重做日志）+ WAL（Write-Ahead Logging）机制保证： 事务提交时，InnoDB 不是直接把随机分散在磁盘各处的数据页刷盘（太慢），而是先把"做了什么修改"顺序写入 redo log 并刷盘，数据页本身晚点再由后台线程慢慢刷。 只要 redo log 落盘成功就算提交成功，重启后可以靠 redo log 重放出未落盘的修改。

与后续知识点呼应：redo log 和 WAL 机制、undo log 与回滚的具体实现细节，会在"存储引擎内核"Tab 单独展开讲，这里先建立"A 找 undo，D 找 redo"的对应关系即可。

I —— 隔离性（Isolation）

多个事务并发执行时，彼此的中间状态不应互相干扰，就像每个事务都独占数据库一样。隔离性由锁 + MVCC（多版本并发控制）共同保证： 写操作靠行锁/间隙锁防止相互覆盖，读操作靠 MVCC 生成的历史快照（ReadView）避免读到其他未提交事务的数据，从而做到"读不加锁、写不阻塞读"。 隔离性还分四个强度不同的级别（读未提交/读已提交/可重复读/串行化），级别越高、隔离越彻底，但并发性能越差——这是下一篇的主题。

C —— 一致性（Consistency）

事务执行前后，数据库必须从一个合法状态转换到另一个合法状态（比如转账后两账户余额总和不变，外键约束、唯一约束不被破坏）。 一致性并不是一种独立的实现机制，而是原子性、隔离性、持久性共同作用之后自然得到的结果，再加上应用层自己定义的业务规则（比如"余额不能为负"要在 SQL 里用 CHECK 约束或业务代码兜底）。

常见误区：很多初学者把 C 理解成和 A/I/D 平级的、数据库要专门实现的一种机制，其实不是。数据库只能保证"数据不会因为并发或宕机而损坏"这种技术层面的一致性；业务层面的一致性（比如余额不能为负）必须靠应用代码或约束来保证，数据库无法替你判断业务逻辑对不对。

四大特性与实现机制对照表

特性英文解决的问题底层实现
原子性 Atomicity 事务执行到一半失败，如何撤销已做的修改 undo log
一致性 Consistency 事务前后数据必须保持合法状态 A + I + D 共同作用 + 业务约束
隔离性 Isolation 并发事务之间互不干扰 锁 + MVCC
持久性 Durability 提交后的数据不因宕机丢失 redo log + WAL

面试高频考点

问："MySQL 是怎么保证事务的原子性和持久性的？"
标准答案：原子性靠 undo log，事务失败时用它回滚；持久性靠 redo log 配合 WAL 机制，提交时先顺序写日志再异步刷数据页。二者都是"日志先行"的思路，只是一个用来撤销、一个用来重放。
常见误区：把"隔离性"和"一致性"搞混。隔离性关注的是"并发事务之间看不看得见彼此的中间状态"，一致性关注的是"数据本身是否始终合法"。两者角度完全不同，隔离性是并发控制的手段，一致性是最终追求的目标。

核心价值总结

  • ACID 不是四个孤立的特性，而是有明确分工的一套体系：A 靠 undo log 兜底失败，D 靠 redo log 兜底宕机，I 靠锁 + MVCC 兜底并发，C 是三者共同作用的自然结果。
  • 理解这套分工，后面学 InnoDB 内核会事半功倍：看到 undo log 就想到回滚和 MVCC 版本链，看到 redo log 就想到崩溃恢复。
