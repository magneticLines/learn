事务与并发控制

事务的隔离级别

隔离性不是非黑即白的开关，而是一个"用多少并发性能换多少数据安全感"的滑动刻度。SQL 标准定义了四挡，MySQL 默认停在第三挡——这背后是工程上的权衡，不是拍脑袋决定的。

为什么隔离性需要分级

如果隔离性做到极致（完全串行化，事务一个接一个排队执行），数据绝对安全，但并发度约等于零，性能无法接受。 如果完全不隔离，事务之间互相看得见对方的中间状态，性能最好但数据可能读出脏数据、算错账。 SQL 标准干脆把这个"安全 vs 性能"的取舍点位标准化成四个级别，让开发者按业务场景自己选。

类比：就像图书馆的自习室管理规则——"串行化"是一次只放一个人进去看书，绝对不会有人打扰你；"读未提交"是谁都能随便进出随便翻你摊开的草稿，最快但最容易被打扰。四个级别就是在"绝对安静"和"完全开放"之间取不同的点。

四种隔离级别逐个讲

1. 读未提交（Read Uncommitted）

最低级别：一个事务可以读到另一个事务尚未提交的修改。几乎不提供任何隔离保证，会产生脏读。 实际生产环境几乎不用，性能收益和数据风险完全不成正比。

2. 读已提交（Read Committed，RC）

一个事务只能读到别的事务已经提交的数据。解决了脏读，但同一个事务内两次读同一行，如果中间别的事务提交了修改，两次读到的结果会不一样（不可重复读）。 Oracle、SQL Server 默认用这个级别。RC 下每条 SELECT 语句执行时都会重新生成一份 ReadView，所以能立刻看到别人刚提交的最新数据。

3. 可重复读（Repeatable Read，RR）—— MySQL 默认级别

保证同一个事务内，多次读同一行数据的结果一致，不会因为别的事务提交而改变。InnoDB 在 RR 级别下，事务只在第一次执行快照读时生成 ReadView，之后整个事务复用同一份，这就是"可重复读"名字的由来。 并且 InnoDB 的 RR 级别通过间隙锁（Gap Lock）/ 临键锁（Next-Key Lock），额外解决了标准定义中 RR 应该还会出现的幻读问题——这是 InnoDB 对标准的加强，不是所有数据库的 RR 都能防幻读。

4. 串行化（Serializable）

最高级别：强制所有事务串行执行，读操作也会加锁（相当于把普通 SELECT 变成 SELECT ... LOCK IN SHARE MODE）。 彻底杜绝脏读、不可重复读、幻读，但并发性能最差，一般只在对数据一致性要求极端严苛的场景使用（比如金融对账批处理）。

怎么选隔离级别

  • 不确定就用默认（RR）：MySQL InnoDB 默认 RR 级别在正确性和性能之间已经是一个很均衡的点，且借助间隙锁解决了幻读，多数业务无需调整。
  • 高并发只读报表：可以考虑 RC，减少间隙锁带来的并发阻塞（间隙锁范围更大，容易造成不必要的锁等待）。
  • 极端一致性要求：用串行化，或者干脆用悲观锁 SELECT ... FOR UPDATE 显式控制。

查看与设置隔离级别

MySQL 8.0
-- 查看当前会话隔离级别
SELECT @@transaction_isolation;

-- 设置当前会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别（对新建立的连接生效）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

面试高频考点

常见误区："MySQL 的 RR 级别也会有幻读，因为 SQL 标准是这么定义的。" 这句话只说对了一半——标准定义的 RR 确实允许幻读，但 InnoDB 的具体实现通过间隙锁把这个漏洞补上了。回答面试题时要说清楚"标准定义"和"InnoDB 实现"是两码事。
问："MySQL 默认隔离级别是什么，为什么？"
答：可重复读（RR）。因为 MySQL 最初设计时基于 binlog 的 statement 格式做主从复制，如果用 RC 级别，同一个事务内多次读结果不一致，会导致主从数据不一致；RR 级别配合间隙锁则能保证复制的一致性，这是历史原因和工程考量共同作用的结果。

核心价值总结

  • 隔离级别的本质是"安全性 vs 并发性能"的刻度选择，级别越高读写互相阻塞越多，性能越差。
  • RC 每次快照读都生成新 ReadView，RR 整个事务复用一份 ReadView——这是二者行为差异的根本原因，也是理解 MVCC 的关键前置知识。

与其他知识点的联系

  • 事务的 ACID 特性：隔离级别就是把 ACID 中"I（隔离性）"这一条抽象特性，细化成四个可选的具体强度档位——ACID 只说"隔离性要保证"，隔离级别回答的是"具体保证到什么程度"。
  • 脏读/不可重复读/幻读：四个隔离级别正是按照"能防住哪几种异常现象"来分级定义的，级别越高防住的异常种类越多，下一篇会逐一演示每种异常在哪个级别下出现、哪个级别下消失。
  • MVCC 原理：本篇提到 RC 每次快照读都生成新 ReadView、RR 整个事务复用同一份 ReadView，这正是 MVCC 篇要展开讲的核心机制——隔离级别的行为差异，最终都要落到 ReadView 生成时机这个具体实现细节上才能真正说清楚。
  • 行锁/表锁/间隙锁/临键锁：本篇提到 RR 级别靠间隙锁补上标准定义允许的幻读漏洞，这是隔离级别在"当前读"路径上的加锁策略，间隙锁的具体工作方式会在后面几篇详细拆解。
  • 两阶段锁协议：串行化级别本质上就是让所有事务的加锁调度都满足可串行化——而 2PL 正是证明"遵守它就能达到可串行化"的理论工具，串行化级别可以理解成 2PL 思想的一种极端落地。
