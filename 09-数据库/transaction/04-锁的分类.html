事务与并发控制

锁的分类（共享锁 / 排他锁 / 意向锁）

锁是隔离性的另一半实现（另一半是 MVCC）。先搞懂最基础的共享锁和排他锁，再理解一个容易被忽视但极其巧妙的设计——意向锁，它解决的是"怎么快速判断表上有没有行锁冲突"这个效率问题。

为什么需要锁

MVCC 能让"读"不阻塞"写"，但"写"和"写"之间必须互斥——两个事务同时改同一行数据，必须有个先后顺序，不然数据会错乱（比如经典的库存超卖问题）。 锁就是解决"写-写冲突"和"部分读-写冲突"（当前读）的机制。

共享锁（S 锁）与排他锁（X 锁）

锁类型全称语义加锁方式
S 锁 Shared Lock，共享锁 多个事务可以同时持有同一行的 S 锁（"共享"读权限），但持有 S 锁期间不允许任何事务加 X 锁 SELECT ... LOCK IN SHARE MODE（或 FOR SHARE）
X 锁 Exclusive Lock，排他锁 同一时刻只能有一个事务持有某行的 X 锁，持有期间其他事务不能加 S 锁也不能加 X 锁 UPDATE/DELETE 自动加，或显式 SELECT ... FOR UPDATE
类比：S 锁像图书馆的"阅览室座位"——很多人可以同时坐下来读同一本书的复印件（只读，互不干扰）；X 锁像"独占会议室"——一次只能一组人进去修改白板上的内容，其他人（包括只想看一眼的人）都得在外面等。

兼容关系：S 锁与 S 锁兼容（可以共存），S 锁与 X 锁不兼容，X 锁与 X 锁也不兼容。这是所有锁设计的基本准则——只要涉及"写"，就必须独占。

意向锁（Intention Lock）：为效率而生的巧妙设计

先看它解决的问题

假设事务 A 想给表 account 加一个表级 X 锁（比如 LOCK TABLES account WRITE）， 但此时表里可能有其他事务正持有某几行的行级 X 锁——直接冲突，必须阻塞。问题是：怎么快速知道表上是否存在行锁，而不需要把几百万行数据全部扫一遍去检查？

没有意向锁的笨办法：要给表加锁前，逐行检查是否有行锁——数据量大的表这个检查本身就是灾难性的性能开销。

意向锁的思路：加行锁前先"打个招呼"

InnoDB 的做法是：事务给某一行加 S/X 锁之前，必须先在表级别加上对应的意向锁（IS 或 IX），相当于在表这个更粗粒度的对象上"登记"一下"我在下面某行加了锁"。 这样一来，其他事务想加表级锁时，只需要看这张表上有没有意向锁标记，一次判断就知道表内部有没有行锁冲突，完全不用遍历行。

意向锁全称含义
IS 锁Intention Shared，意向共享锁事务准备给表中某些行加 S 锁，先给表加 IS 锁
IX 锁Intention Exclusive，意向排他锁事务准备给表中某些行加 X 锁，先给表加 IX 锁

意向锁的兼容规则

意向锁之间是互相兼容的（IS 与 IS、IS 与 IX、IX 与 IX 都能共存），因为意向锁本身不锁定任何具体数据，只是一个"声明"。 真正会阻塞的是意向锁和表级 S/X 锁之间：

已持有 \ 请求ISIXS（表级）X（表级）
IS兼容兼容兼容冲突
IX兼容兼容冲突冲突
S（表级）兼容冲突兼容冲突
X（表级）冲突冲突冲突冲突
直觉记忆：意向锁只跟"表级的 S/X 锁"打架，意向锁之间从不打架。因为意向锁的作用就是让"想加表锁的人"一眼看出表内部有没有行锁，而不是控制行与行之间的关系。

常见误区

误区："意向锁会阻塞普通的行级 DML 操作（UPDATE/DELETE）。" 不会。意向锁只影响想要加表级锁的操作（比如 LOCK TABLES、ALTER TABLE），普通的行级 UPDATE/DELETE 之间该怎么并发还是怎么并发，只不过在加行锁前会顺带打一个意向锁的标记。
误区：把意向锁和行锁的粒度搞混，以为意向锁是"细粒度锁"。恰恰相反，意向锁是表级别的粗粒度锁，它存在的意义就是给更粗粒度的操作提供一个"快速探测"的手段。

面试高频考点

问："为什么需要意向锁？"
标准答案：如果没有意向锁，一个事务想给整张表加锁时，必须逐行检查是否存在行锁冲突，时间复杂度是 O(n)。有了意向锁后，只需要检查表级的意向锁标记，就能 O(1) 判断表内是否有行锁在使用，是一种"空间换时间"的效率优化设计。

核心价值总结

  • S 锁和 X 锁是数据层面的互斥控制，解决"谁能读、谁能写"的问题。
  • 意向锁是效率层面的优化设计，让粗粒度锁定（表锁）可以快速探测细粒度锁定（行锁）是否存在冲突，避免全表扫描式检查。

与其他知识点的联系

  • 脏读/不可重复读/幻读：本篇开头提到"锁解决写-写冲突和部分读-写冲突"，正是上一篇讲的三种异常现象里"写"相关部分的解决手段——脏读那种"读到未提交数据"背后就是读操作没有被 S/X 锁约束住。
  • 行锁表锁间隙锁临键锁：本篇讲的 S 锁/X 锁/意向锁是锁的"功能性分类"（共享还是排他、细粒度还是粗粒度探测），下一篇讲的行锁/表锁/间隙锁/临键锁是锁的"粒度分类"，两套分类维度互相独立又互相配合——比如给一行加 X 锁时，究竟锁的是行本身还是连带间隙，就是下一篇的问题。
  • 两阶段锁协议：本篇讲的所有锁类型，都要遵守 2PL 的加锁/解锁两阶段规则——不管是 S 锁、X 锁还是意向锁，一旦事务开始释放锁就不能再申请新锁，这是锁使用规则的通用理论基础。
  • 死锁的产生与排查：意向锁、S 锁、X 锁之间的兼容/冲突关系，正是死锁产生的直接土壤——两个事务如果以不同顺序申请这些互斥的锁，就可能形成环路等待，这是死锁篇要具体演示的场景。
