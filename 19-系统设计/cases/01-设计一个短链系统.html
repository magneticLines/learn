经典案例

设计一个短链系统

把一条很长的 URL 压缩成一个短小的链接（比如 xxx.ly/abc123），是系统设计面试里最经典的入门题——麻雀虽小，五脏俱全，读写分离、缓存、编码算法全都用得上。

第一步：需求澄清

按上一篇讲的套路，先把功能性和非功能性需求过一遍：

类型具体要求
功能性需求 输入一个长 URL，生成一个唯一的短 URL；访问短 URL 时能重定向到原始长 URL
非功能性需求 系统要高可用；短码不能重复；重定向要低延迟；假设读写比例悬殊——生成短链是"写"，访问短链是"读"，读远多于写

第二步：容量估算

假设日增 1000 万条短链，简单估算一下各项指标的数量级：

  • 写 QPS：1000万 ÷ 86400秒 ≈ 116 QPS，考虑峰值系数（比如 5 倍），峰值写 QPS 约 580，压力不大。
  • 读 QPS：假设读写比 10:1（平均每条短链被访问 10 次），读 QPS 约为写 QPS 的 10 倍，峰值接近 5800，这才是系统的主要压力来源——所以"重定向"这条链路必须做优化，缓存是重点。
  • 存储量：假设按 10 年计算，1000万/天 × 365 × 10 ≈ 365 亿条记录。每条记录（短码 + 长 URL + 元信息）估算 500 字节，总存储量约 365亿 × 500B ≈ 17TB，单机数据库存不下，需要考虑分片或者归档冷数据。
估算的意义：这几个数字直接决定了后续设计——读 QPS 比写 QPS 高一个数量级，说明必须加缓存（呼应 12-缓存：读多写少的场景是缓存的典型用武之地）；17TB 的存储量说明单表迟早装不下，需要考虑分表或冷热数据分离。

第三步：高层设计

客户端（提交长 URL / 访问短链）
  ↓
负载均衡 + 应用服务器（分两个接口：生成短链、重定向）
  ↓
缓存（Redis，缓存"短码 → 长URL"的映射，扛住高频重定向读）
  ↓（缓存未命中）
数据库（持久化"短码 → 长URL"映射表）

整个系统对外暴露两个核心接口：

  • POST /shorten：接收长 URL，生成短码，写入数据库，返回短链。
  • GET /{shortCode}：根据短码查出长 URL，做重定向。这个接口会被高频访问，是优化的重点。

第四步：细化设计

1. 短码生成方案

这是本案例的核心难点，常见两种思路：

方案原理优点缺点
自增 ID + 62 进制编码 用一个全局自增 ID（数据库自增主键，或者发号器），把十进制 ID 转换成 62 进制（0-9, a-z, A-Z 共 62 个字符）字符串 天然唯一不会碰撞，实现简单，编码后 6~7 位字符就能表示几十亿的 ID 空间 自增 ID 存在单点问题，且 ID 连续容易被恶意遍历、猜出短链规律
哈希 + 碰撞检测 对长 URL 做哈希（如 MD5/SHA256）取前若干位作为短码 不依赖全局自增，天然分布式友好 哈希存在碰撞概率，需要额外查库检测冲突，冲突后要重新计算或加盐，实现复杂度更高

62 进制编码的直觉：62 个字符，6 位长度能表示 62^6 ≈ 568 亿种组合，远超前面估算的 365 亿总量，留有余量。这也是"先估算容量、再做设计"的具体体现——短码要留多少位，不是拍脑袋决定的，而是由估算出的总数据量反推出来的。

面试高频考点：面试官通常会追问"自增 ID 从哪来？单点怎么办？"——标准答案是用一个独立的发号器服务（比如基于数据库的一段号段预分配，或者类似雪花算法的分布式 ID 生成器），而不是让每个应用实例直接对同一个自增列做 INSERT，那样在高并发写入时会成为瓶颈和单点。

2. 数据库设计

核心表结构很简单：

字段说明
short_code短码，建唯一索引，作为查询的入口
long_url原始长 URL
created_at / expire_at创建时间、过期时间（如果支持短链过期）

因为读远多于写（前面估算读 QPS 是写 QPS 的 10 倍），重定向接口一定要走缓存：先查 Redis，命中直接返回长 URL；未命中再查数据库，并回填缓存。这是 12-缓存 里"Cache-Aside（旁路缓存）"模式的典型应用场景——短码到长 URL 的映射一旦生成基本不会再变，非常适合缓存，几乎不用担心缓存与数据库不一致的问题。

3. 重定向实现：301 还是 302

这是短链系统里一个经常被问到、但很多人答不准确的细节：

状态码含义浏览器行为对系统的影响
301（永久重定向） 告诉浏览器"这个短链永久指向该长 URL" 浏览器会缓存这次跳转结果，之后同一用户再次访问同一短链，不会再请求服务器，直接跳转 大幅减少服务器压力，但因为请求不再到达服务器，无法准确统计点击量
302（临时重定向） 告诉浏览器"这次跳转是临时的，下次还要重新问服务器" 浏览器不缓存跳转结果，每次访问都会重新请求服务器 服务器压力更大，但可以精确统计每一次点击（这对于需要做链接分析、计费的短链服务很重要）
面试高频考点：这是短链系统面试里的经典"送分题"也是"送命题"——很多人只知道"301 是永久、302 是临时"，但答不出为什么短链服务大多默认用 302：因为短链服务的商业模式通常依赖点击统计（分析、广告计费），如果用 301 导致浏览器缓存跳转结果，服务端就完全看不到后续的访问请求了，统计数据会严重失真。所以要不要用 301，本质是"服务器性能"和"数据统计需求"之间的权衡，不是无脑选性能更优的方案。

4. 容量估算的落地验证

回到最初的估算：日增 1000 万短链，10 年 365 亿条数据，17TB 存储量。单张 MySQL 表通常建议不超过千万到亿级行数（受限于索引效率、备份恢复时间等），365 亿显然需要分表。 结合短码本身的哈希特性，可以按 short_code 做哈希取模分表，这样查询时可以直接根据短码定位到具体的表，不需要跨表扫描——这是"先有估算数字，再反推分片方案"的完整闭环。

核心价值总结

  • 短码生成是本案例的灵魂：自增 ID + 62 进制编码是最常见的工业级方案，简单可靠；哈希方案需要额外处理碰撞，复杂度更高。
  • 读多写少的场景，缓存是标配：重定向接口必须走 Cache-Aside 模式，这是对 12-缓存 知识点的直接应用。
  • 301 vs 302 是"性能"和"数据统计"的权衡，没有绝对正确的选择，要结合业务目标决定。
  • 容量估算不是走过场：短码位数、要不要分表，都是由估算出的数据量反推出来的，不是凭感觉决定的。

与其他知识点的联系

  • 关联知识点：缓存 Cache-Aside 模式（12-缓存）：短链系统的读 QPS 是写 QPS 的 10 倍，重定向接口必须走"先查缓存、未命中再查库并回填"的旁路缓存模式。这个案例之所以适合用 Cache-Aside 而不用更复杂的一致性方案，是因为短码到长 URL 的映射一旦生成基本不变——不需要担心"改了数据库、缓存没同步"这类典型缓存一致性难题，是缓存最友好的应用场景之一。
  • 关联知识点：分布式 ID 生成器（14-分布式与微服务）：短码生成方案里提到"自增 ID 不能直接用数据库自增列"，本质原因是单点数据库的自增操作在高并发写入下会成为瓶颈。真正工业级的做法是引入独立的发号器服务（号段模式或雪花算法），这和分布式系统里"去掉单点依赖"的思路是同一件事——短链系统的短码生成，其实是"分布式唯一 ID 生成"这个通用问题在具体业务上的落地。
  • 关联知识点：分库分表（09-数据库·扩展性）：案例第四步用估算出的 17TB / 365 亿条数据反推出"单表存不下，需要按 short_code 哈希分表"，这正是常见的扩展性技术选型清单里"数据库读写分离/分库分表"武器的具体应用——而且因为查询天然按短码定位，分片键选 short_code 之后不会产生跨片查询，是分片设计里"分片键要和主要查询路径对齐"这条原则的典型体现。
  • 关联知识点：容量估算驱动设计（方法论篇）：短码位数（6~7 位）、要不要分表，都不是拍脑袋决定的，而是先算出总数据量再反推参数——这呼应了系统设计通用套路里"容量估算不是走过场，而是后续设计决策的输入"这一核心方法论，本案例是这套方法论最直接的一次实战演练。
