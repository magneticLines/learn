经典案例

设计一个秒杀系统

"1000 件商品，10 万人同时抢购"——秒杀系统的核心挑战不是"数据量大"，而是"极短时间窗口内的超高并发写请求"，这是一道把缓存、消息队列、分布式锁全部串起来的综合题。

第一步：需求澄清

类型具体要求
功能性需求 用户在指定时间点抢购限量商品；库存扣完即结束；不能超卖（卖出数量绝不能超过库存）
非功能性需求 瞬时并发极高（可能是平时流量的成百上千倍）；持续时间短（往往几秒到几分钟内库存售罄）；绝对不能超卖，但允许"少量库存因保护机制而未在第一时间售出"
秒杀 vs 一般电商下单的本质区别：一般电商下单的流量是相对平滑的，而秒杀是"脉冲式"流量——比如平时 QPS 100，秒杀开始瞬间 QPS 冲到 10 万，1 分钟后又回落到平时水平。系统设计的核心目标不是长期支撑 10 万 QPS，而是"扛住这个瞬时脉冲，不被打垮"，这决定了后面所有设计都是围绕"削峰"展开的。

第二步：容量估算

假设某商品秒杀活动预计吸引 100 万用户参与，其中 10% 的用户会在开始的前 1 秒内发起请求，那么瞬时峰值 QPS 可以粗略估算为 10 万。 而实际商品库存可能只有 1000 件——也就是说，99.9% 的请求最终都是"陪跑"，注定失败。这个数字直接决定了架构方向：既然绝大多数请求根本不需要真正走到数据库，那么系统的核心任务就是尽可能早地拦截和过滤掉这些无效请求，只让极少数"真正可能抢到商品"的请求触达最终的库存扣减逻辑。

第三步：高层设计——分层削峰思路

秒杀系统的高层设计可以理解成一个"漏斗"，每一层都在过滤掉一批请求，越往后流量越小：

① 前端限流（按钮防重复点击、验证码增加机器人成本）
  ↓ 过滤掉重复提交和部分机器人流量
② 网关层限流（令牌桶 / 漏桶算法，拒绝超出承载能力的请求）
  ↓ 只放行系统当前处理能力范围内的请求
③ 缓存层扣库存（Redis 预热库存，原子操作 DECR + Lua 脚本防超卖）
  ↓ 只有真正抢到库存名额的请求才能往下走
④ 消息队列异步下单（削峰填谷，异步处理数据库落库）
  ↓
数据库（持久化订单，压力已经被前面几层削掉了绝大部分）

这个漏斗结构直接体现了"分层过滤"的设计哲学：越靠前的层，过滤成本越低但精度也越低；越靠后的层，处理成本越高但必须绝对精确（库存扣减不能有任何差错）。把简单粗暴的拦截放在前面，把精细可靠的操作放在后面，才能用最小的代价扛住最大的流量。

第四步：细化设计

1. 前端限流：第一道减压阀

按钮点击后立即置灰防止重复提交、加验证码增加"机器人刷单"的成本——这一层拦截的是"最低成本、最容易过滤"的无效流量，虽然拦不住真正的技术型攻击，但能显著削减普通用户误操作和低端脚本造成的重复请求。

2. 网关层限流：令牌桶 / 漏桶

进入系统内部之前，网关层要做限流，超出系统承载能力的请求直接拒绝（返回"活动太火爆，请稍后重试"），而不是让请求继续往下游传导压垮数据库。常见的两种限流算法：

算法原理特点
令牌桶（Token Bucket） 系统按固定速率往桶里放令牌，请求要拿到令牌才能通过，桶满了令牌就不再增加 允许一定程度的突发流量（桶里存量令牌可以被瞬间消耗），更贴近秒杀这种脉冲场景
漏桶（Leaky Bucket） 请求先进桶，系统按固定速率从桶里"漏出"处理，桶满则拒绝新请求 严格平滑输出速率，不允许突发，适合要求下游处理速率绝对恒定的场景
选型建议：秒杀场景通常选令牌桶，因为它允许短时间内的突发放行（比如活动开始瞬间攒下的令牌可以一次性用完），更符合"开始瞬间冲一波，然后迅速回落"的流量特征。

3. 库存预热到缓存：避免每次打数据库

活动开始前，把商品库存数量提前加载到 Redis 中（呼应 12-缓存）。这样每个抢购请求只需要跟 Redis 交互，而不是每次都直接查询/更新数据库——Redis 单机可以轻松支撑数万级 QPS，而 MySQL 在同等硬件条件下的写入能力差一个数量级以上，把压力挡在缓存层是撑住瞬时高并发的关键一步。

4. Redis 原子操作 + Lua 脚本：杜绝超卖

这是秒杀系统最核心、面试最爱深挖的部分。先看一个错误的反例：

反例：先 GET 库存判断是否大于 0，再执行 DECR 扣减。这两步操作在高并发下不是原子的——假设库存剩 1，两个请求同时执行到 GET，都读到"库存为 1，大于 0"，于是两边都执行了 DECR，库存变成 -1，发生超卖。这是初学者最容易犯的经典错误，本质是"检查（check）"和"扣减（decrement）"这两步之间存在竞态条件（race condition）。

正例：把"检查库存 + 扣减库存"这两步操作用 Lua 脚本包起来，交给 Redis 单线程原子执行：

seckill.lua
-- KEYS[1]：库存的 key；返回 1 表示扣减成功，返回 0 表示库存不足
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock > 0 then
    redis.call('DECR', KEYS[1])
    return 1
else
    return 0
end

Redis 本身是单线程模型，一个 Lua 脚本在执行期间不会被其他命令打断，这就天然保证了"检查 + 扣减"这个组合操作的原子性——这也是 Redis 处理"读后写"这类竞态场景的标准解法，比在应用层加分布式锁开销更低、实现更简单。

面试高频考点：如果面试官问"能不能不用 Lua 脚本，用应用层的分布式锁（比如 Redisson）实现？"——技术上可行，但分布式锁涉及加锁/解锁的网络往返，并发性能远不如 Redis 原生的单线程原子操作；Lua 脚本方案是秒杀场景的业界标准做法，性能更优。

5. 消息队列异步下单：削峰填谷

Redis 扣减库存成功只代表这个用户"抢到了名额"，紧接着要生成订单、扣减真实数据库库存、锁定优惠券等一系列操作——如果这些操作都同步执行，相当于把 Redis 挡下来的洪峰又原封不动地转嫁给了数据库。 正确做法是：Redis 扣减成功后，立即把"下单请求"作为一条消息丢进消息队列（呼应 13-消息队列），前端先返回"抢购成功，正在生成订单"，后端消费者按数据库能承受的速率慢慢消费这些消息，异步完成下单落库。

类比：这就像热门演唱会的检票口——检票员（Redis 原子扣减）先快速判断"你有没有票"，有票的人先放行进场，但进场之后不是所有人同时冲向座位区的每一个入口，而是排队分批进入，避免所有人同一时刻挤爆一个通道（数据库）。

这一步是"前面学过的多个模块知识点在同一个真实场景里的组合应用"的最好体现：缓存（库存预热、扛住读写）+ 消息队列（削峰、异步解耦）+ 分布式系统的原子性保证（Lua 脚本），三者缺一不可，单独拿出哪一个都无法完整解决秒杀问题。

常见误区：不少初学者认为"加了消息队列就万事大吉"，但要注意消息队列只是把压力从"当下"转移到了"稍后处理"，如果消费速度长期跟不上生产速度，消息会持续堆积，最终还是会影响用户体验（比如订单生成延迟过久）。所以还需要合理评估消费者的处理能力，必要时对消费者也做水平扩容。

核心价值总结

  • 秒杀的本质是"漏斗形"分层过滤：前端限流 → 网关限流 → 缓存扣库存 → 消息队列异步下单，每一层都在过滤流量，越往后精度要求越高。
  • 防超卖的核心是保证"检查库存 + 扣减库存"的原子性，Redis + Lua 脚本是业界标准解法，比应用层分布式锁更轻量高效。
  • 消息队列是"同步压力"转化为"异步慢慢消化"的关键一环，但要注意消费能力要跟得上，避免无限堆积。
  • 这道题串联了缓存、消息队列、限流算法多个模块的知识点，是"综合应用"这个模块名称的最佳注脚——真实系统的复杂问题，往往不是靠单一技术，而是靠多种技术的合理编排解决的。

与其他知识点的联系

  • 关联知识点：限流算法（令牌桶/漏桶）：网关层限流是秒杀漏斗的第二道防线，本质是在"绝对拒绝多余流量"和"数据库承受能力"之间划一条硬边界。选令牌桶而不是漏桶，是因为秒杀的流量特征是脉冲式的——开始瞬间冲高、随后迅速回落，令牌桶允许消耗存量令牌应对突发，这和漏桶追求"绝对平滑输出"的设计目标正好相反，选型必须结合流量形态判断，不能死记"哪个更好"。
  • 关联知识点：缓存的原子操作与竞态条件（12-缓存）：案例里"先 GET 再 DECR"导致超卖的反例，本质是缓存操作中一个经典的 check-then-act 竞态条件，和多线程编程里"先判断再执行"的线程安全问题是同一类 bug 的不同变种。用 Lua 脚本借助 Redis 单线程模型保证原子性，是"给一组操作找一个天然串行的执行环境"这个通用思路的具体实现，比引入分布式锁更轻量。
  • 关联知识点：消息队列削峰与异步解耦（13-消息队列）：Redis 扣减库存成功后不同步写数据库，而是丢进 MQ 异步落库，这是"削峰"在秒杀场景里的直接应用——把瞬时洪峰变成数据库能够匀速消化的持续小流量。同时这一步也是异步解耦的体现：下单接口不必等数据库写完才返回，用户体验和系统吞吐两头兼顾。
  • 关联知识点：扩展性技术选型的组合应用（method/02）：秒杀系统是"多种武器组合登场"的典范——限流挡住多余流量、缓存扛住高频读写、消息队列削峰填谷，三者各自解决不同层面的问题，缺了任何一个都无法完整撑住秒杀场景。这也印证了选型清单里强调的"先定位瓶颈、再选对应武器"，而不是单一技术包打天下。
  • 关联知识点：分布式系统的最终一致性（14-分布式与微服务）：消息队列异步下单意味着"抢购成功"和"订单真正落库"之间存在时间差——用户看到的成功提示,是一种基于最终一致性的承诺,而不是强一致的即时保证。这和分布式事务里"接受短暂不一致，换取系统吞吐和可用性"的取舍思路是一致的。
