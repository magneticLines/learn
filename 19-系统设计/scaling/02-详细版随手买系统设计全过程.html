详细版:"随手买"系统设计全过程

上一篇《从单体到分布式的演化路径回顾》讲清楚了"为什么"要在每个阶段引入对应技术，这一篇往下再深一层—— 从容量估算开始，逐阶段展示代码目录结构如何变化、关键配置和代码长什么样， 让"演化"从一句话变成看得见摸得着的工程细节。

第一步:先算清楚要撑多大的量

系统设计的通用套路要求"先估算容量再画架构"(呼应 方法论 Tab)，这里直接给"随手买"设一组假想但合理的数字，后面每一步的技术选型都基于这组数字推导，而不是凭感觉。

指标数值由此推出的结论
日活用户(DAU)500 万属于中型电商体量，单体撑不住但也用不上超大规模分片
平均 QPS约 2,000单台应用服务器(假设单机承载 500 QPS)至少需要 4~5 台做负载均衡
大促峰值 QPS约 30,000(平时 15 倍)峰值和均值差距巨大 → 需要缓存削峰 + 消息队列削峰，不能按峰值常态化部署
订单表年增量约 3 亿行单表几亿行后索引和写入性能明显下降 → 需要分库分表
商品详情读写比约 200:1典型读多写少 → 缓存收益极高
心法：容量估算不需要精确，重要的是数量级(是"万"还是"亿")。数量级不同，合理的架构完全不同——日活 5 万和日活 5000 万的系统，把对方的架构抄过来往往都是灾难。

阶段 0:单体架构的目录结构

最初的"随手买"是一个 Maven 单模块工程，所有代码在一个 src/main/java 下按"层"分包(controller / service / mapper / entity)，而不是按"业务"分包。这是绝大多数单体应用的起点：

suishoumai/ ├─ src/main/java/com/suishoumai/ │ ├─ controller/ │ │ ├─ UserController.java │ │ ├─ ProductController.java │ │ ├─ OrderController.java // 下单接口在这里 │ │ └─ PaymentController.java │ ├─ service/ │ │ ├─ UserService.java │ │ ├─ ProductService.java │ │ ├─ OrderService.java │ │ └─ PaymentService.java │ ├─ mapper/ // MyBatis Mapper 接口 │ │ ├─ UserMapper.java │ │ ├─ ProductMapper.java │ │ └─ OrderMapper.java │ ├─ entity/ │ └─ config/ ├─ src/main/resources/ │ ├─ application.yml // 一个数据源配置 │ └─ mapper/*.xml └─ pom.xml // 一个可执行 jar

此时下单逻辑非常直接——一个方法里同步完成校验库存、创建订单、扣库存，全部在同一个数据库事务里，靠 @Transactional 保证"扣库存+建订单"这两步要么都成功要么都失败(原子性)：

OrderService.java · 阶段0:有超卖漏洞的写法(反例)
@Transactional
public Order createOrder(Long userId, Long productId, int count) {
    Product product = productMapper.selectById(productId);
    if (product.getStock() < count) {
        throw new BizException("库存不足");
    }
    productMapper.deductStock(productId, count);       // "先查后改"两条语句之间有空隙
    Order order = new Order(userId, productId, count);
    orderMapper.insert(order);                   // 同一个事务，天然原子
    return order;
}
陷阱：@Transactional 解决的是原子性，不是并发安全。上面这版代码有一个经典的超卖(overselling)漏洞：两个并发请求可能同时执行到"查库存"这一步，都看到库存还够，于是都通过校验、都执行扣减——库存被减成负数也不会报错。原子性只保证"这个事务内的多条语句要么都生效要么都不生效"，并不保证"两个并发事务不会读到同一份旧数据"，后者是隔离性要解决的问题，而默认隔离级别下"先查后改"这种模式并不能自动防住它。

正确的写法是把"判断库存够不够"和"扣库存"压缩成一条原子的 UPDATE 语句，让数据库自己保证这一步不会被并发请求穿透：

OrderService.java · 阶段0:正确防超卖写法
@Transactional
public Order createOrder(Long userId, Long productId, int count) {
    // UPDATE product SET stock = stock - #{count} WHERE id = #{id} AND stock >= #{count}
    int affected = productMapper.deductStockIfEnough(productId, count);
    if (affected == 0) {                          // 影响行数为0，说明库存不够，扣减没生效
        throw new BizException("库存不足");
    }
    Order order = new Order(userId, productId, count);
    orderMapper.insert(order);
    return order;
}

这一版代码简单到几乎没有"设计"可言，但已经区分清楚了两件事：原子性由 @Transactional 负责("扣库存+建订单"要么都成功要么都失败)，并发安全由那条"判断+扣减"合一的 UPDATE 语句负责(交给数据库的行锁机制保证)。后面每一次架构演进，本质上都是在分布式环境下，想办法用更复杂的手段补回这两种保证。

阶段 1:读写分离——目录里多了"路由"这一层

引入主从复制后(呼应 09-数据库)，代码目录新增了数据源配置和一个基于 AOP 的路由切面，业务代码本身几乎不用改——这也是"基础设施升级不应该侵入业务代码"这个设计原则的体现：

suishoumai/ ├─ src/main/java/com/suishoumai/ │ ├─ controller/ service/ mapper/ entity/ // 不变 │ ├─ datasource/ // 新增 │ │ ├─ DataSourceConfig.java // 注册主库、从库两个数据源 │ │ ├─ DataSourceRouter.java // 继承 AbstractRoutingDataSource │ │ └─ ReadOnly.java // 自定义注解，标记只读方法 │ └─ config/
ReadOnly 注解 + AOP 切面(简化版)
@Aspect @Component
public class DataSourceAspect {
    @Before("@annotation(com.suishoumai.datasource.ReadOnly)")
    public void routeToSlave() {
        DataSourceContextHolder.setDataSource("slave");  // 标了 @ReadOnly 的方法走从库
    }
}

业务层只需要在查询方法上加一个 @ReadOnly，写操作维持原样默认走主库——这是"新增一层基础设施，业务代码零感知"的典型手法，后面的分库分表、缓存也都遵循同样的思路。

阶段 2:分库分表——中间件接管路由，Mapper 层几乎无感

订单表冲到 3 亿行后，引入 ShardingSphere 这类分库分表中间件(呼应 09-数据库·扩展性)。关键变化在 resources 下多了一份分片规则配置，Mapper 接口和 SQL 完全不用改——中间件在 JDBC 驱动层拦截 SQL，自动重写并路由到正确的物理库表：

sharding-rule.yml(简化版)
rules:
  - !SHARDING
    tables:
      t_order:
        actualDataNodes: ds_${0..3}.t_order_${0..7}   # 4库 × 8表 = 32张物理表
        databaseStrategy:
          shardingColumn: user_id              # 按用户ID哈希分库
        tableStrategy:
          shardingColumn: order_id             # 按订单ID哈希分表
变化点分库分表前分库分表后
OrderMapper.javaSELECT * FROM t_order WHERE id=?代码不变，中间件自动重写成对应物理表的 SQL
分布式主键数据库自增 ID必须换成雪花算法等分布式 ID(自增在多库场景下会冲突)
跨分片查询一个 JOIN 搞定"查某用户所有订单"带上 user_id 只能精准定位到库(databaseStrategy 按它分库)，库内 8 张表仍要按 order_id 扫描合并；如果查询连 user_id 都不带，就要扫全部 4 库 32 表再合并，代价更大

阶段 3:引入缓存——新增 cache 包，Service 层插入旁路逻辑

目录新增一个 cache 包封装 Redis 操作，ProductService 按 Cache-Aside 模式改造(呼应 12-缓存):

suishoumai/ ├─ src/main/java/com/suishoumai/ │ ├─ cache/ │ │ ├─ RedisCacheHelper.java │ │ └─ CacheKeys.java // 统一管理 key 命名规则，避免各处硬编码 │ └─ service/ │ └─ ProductService.java // 改造成 Cache-Aside
ProductService.java · Cache-Aside + 防穿透
public Product getProduct(Long id) {
    String key = CacheKeys.product(id);
    String cached = redis.get(key);
    if (cached != null) {
        return cached.isEmpty() ? null : JSON.parseObject(cached, Product.class);  // 空值也缓存，防穿透
    }
    Product product = productMapper.selectById(id);
    int ttlSeconds = product == null
        ? 60                                              // 空值(防穿透)只缓存很短时间，避免商品上架后长期查不到
        : 300 + new Random().nextInt(60);          // 正常值过期时间加随机数，防雪崩
    redis.setex(key, ttlSeconds, product == null ? "" : JSON.toJSONString(product));
    return product;
}
这版代码只覆盖了穿透和雪崩，防击穿还差一步：如果某个爆款商品的缓存恰好在大促高峰期过期，海量并发请求会同时判断"缓存未命中"然后一起打到数据库(呼应 12-缓存的缓存击穿)。实际生产代码通常还要在"查库存写回缓存"这一段加一把分布式互斥锁(或用逻辑过期时间)，保证同一时刻只有一个请求去查数据库、其余请求等待或返回旧值，这里为了聚焦目录结构演变而简化掉了这一层。

阶段 4:接入消息队列——新增 mq 包，下单逻辑瘦身

下单成功后不再同步调用短信/积分/仓库服务，而是发一条消息(呼应 13-消息队列)，OrderService 反而变得更简单了：

suishoumai/ ├─ src/main/java/com/suishoumai/ │ ├─ mq/ │ │ ├─ producer/OrderEventProducer.java │ │ └─ 消费方在独立的下游服务里，不在本工程
OrderService.java · 引入 MQ 之后
@Transactional
public Order createOrder(Long userId, Long productId, int count) {
    // ... 校验库存、扣库存、写订单，逻辑同阶段0 ...
    orderEventProducer.publish(new OrderCreatedEvent(order.getId()));  // 发消息就返回，不等短信/积分
    return order;
}

短信、积分、仓库通知从"下单接口里的同步依赖"变成了"消息的独立消费者"，任何一个下游抖动都不再拖慢下单主链路。

阶段 5:微服务拆分——一个仓库变成一组仓库

这是目录结构变化最大的一步(呼应 14-分布式与微服务)。原来一个 Maven 工程里的四个 *Controller/*Service，拆成四个独立部署、独立代码仓库的微服务：

拆分前(单体，一个仓库): suishoumai/ └─ com.suishoumai.{controller,service,mapper}.{User,Product,Order,Payment}... 拆分后(微服务，多个仓库): suishoumai-common/ // 共享 DTO / 工具类 / 异常定义 suishoumai-gateway/ // API 网关，统一入口/鉴权/限流 suishoumai-user-service/ // 独立数据库 suishoumai_user ├─ controller/ service/ mapper/ entity/ └─ pom.xml // 独立可执行 jar，独立发布节奏 suishoumai-product-service/ // 独立数据库 suishoumai_product suishoumai-order-service/ // 独立数据库 suishoumai_order(分库分表在这一层) suishoumai-payment-service/ suishoumai-notify-service/ // 专门消费 MQ 消息(短信/积分/仓库通知)
代价立刻出现：拆分前 OrderService 可以直接 productMapper.selectById() 查库存；拆分后订单服务和商品服务是两个进程、两个数据库，只能通过 RPC/HTTP 调用商品服务的接口，一次原本的本地方法调用变成了一次网络调用——这就是为什么"微服务拆分"之后紧跟着就要解决"服务发现"和"分布式事务"两个新问题。

阶段 6:分布式事务——用 TCC 补回原子性

阶段0里一个 @Transactional 搞定的"扣库存+建订单"，现在跨了订单服务和商品服务两个数据库。这里用 TCC(Try-Confirm-Cancel)模式演示柔性事务的骨架(呼应 14-分布式与微服务·事务与一致性)：

ProductTccService.java · Try-Confirm-Cancel 骨架
public interface ProductTccService {
    boolean tryDeduct(Long productId, int count);  // Try: 冻结库存，不真正扣减
    void confirmDeduct(Long txId);            // Confirm: 订单服务确认支付成功 → 真正扣减
    void cancelDeduct(Long txId);              // Cancel: 任一环节失败 → 释放冻结库存
}

核心思路是把"一次跨库的强一致事务"拆成"两阶段的柔性事务"：先冻结资源(Try)，业务全部走完再确认(Confirm)，中途任何一步失败就补偿回滚(Cancel)。代价是每个参与方都要多实现两个接口，复杂度明显上升——这也是为什么不是所有场景都值得上 TCC，很多场景用更简单的"本地消息表+最终一致"就够了。

阶段 7:CDN 与负载均衡——Nginx 配置示例

呼应 10-计算机网络，负载均衡这一层通常不体现在 Java 代码里，而是在网关前的 Nginx/LVS 配置里：

nginx.conf(简化)
upstream gateway_pool {
    server 10.0.1.1:8080 weight=1;
    server 10.0.1.2:8080 weight=1;
    server 10.0.1.3:8080 weight=1;  // 加权轮询，三台网关分摊流量
}
server {
    location /static/ { proxy_pass https://cdn.suishoumai.com; }  // 静态资源走CDN
    location /api/ { proxy_pass http://gateway_pool; }       // 接口走负载均衡池
}

阶段 8:可观测性——日志里多了一个 TraceId

呼应 17-可观测性，微服务拆分之后，排查问题的第一步是把跨服务的日志串起来。做法是在网关入口生成一个 TraceId，透传到下游所有服务，统一打印在日志里：

日志输出对比
// 拆分前(单体，一个进程内，天然连续):
[2024-05-20 10:00:01] OrderService 创建订单 orderId=1001

// 拆分后(没有 TraceId，三份日志散落在三台机器上，靠时间猜测关联):
[gateway] [10:00:01.001] 接收下单请求
[order-service] [10:00:01.020] 创建订单 orderId=1001
[product-service] [10:00:01.035] 扣减库存失败

// 引入 TraceId 之后(可以用同一个ID在日志系统里一键筛出完整链路):
[gateway] [traceId=a1b2c3] 接收下单请求
[order-service] [traceId=a1b2c3] 创建订单 orderId=1001
[product-service] [traceId=a1b2c3] 扣减库存失败

阶段 9:测试与 CI/CD——目录里多了流水线定义

呼应 15-测试 和 16-容器化与CI-CD，每个微服务仓库根目录下新增测试代码和流水线配置：

suishoumai-order-service/ ├─ src/main/java/... ├─ src/test/java/... // 单元测试 + 集成测试 ├─ Dockerfile // 打包成标准化镜像 └─ .github/workflows/ci.yml // 提交即触发:构建→测试→镜像→部署
ci.yml(简化)
jobs:
  build-and-test:
    steps:
      - run: mvn test                     # 单元测试不过，流水线直接红灯终止
      - run: docker build -t order-service:${{ github.sha }} .

阶段 10:安全——落到具体代码行

呼应 18-Web安全，安全不是一个独立模块，而是散布在前面每一层代码里的具体写法：

层不安全的写法安全的写法
Mapper"WHERE id=" + id(字符串拼接)WHERE id=#{id}(参数化查询，防SQL注入)
Controller直接用 URL 里的 orderId 查订单额外校验 order.getUserId() == currentUser.getId()(防越权)
