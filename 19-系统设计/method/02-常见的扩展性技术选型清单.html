常见的扩展性技术选型清单

把前面模块学过的"零件"整理成一张速查表——遇到具体的性能瓶颈时，知道该从工具箱里拿出哪一件武器，而不是每次都从头发明轮子。

为什么需要一张"速查表"

做细化设计（上一篇讲的第四步）时，最容易卡壳的地方不是"不知道有哪些技术"，而是"不知道当前这个瓶颈该用哪个技术解决"。 比如系统读请求量暴增，是该加缓存，还是该做读写分离，还是该上 CDN？答案取决于瓶颈到底出在哪一层。

类比：这就像医生看病，"头疼"这个症状背后可能是感冒、颈椎病、也可能是熬夜——真正专业的地方不是知道所有药名，而是能通过"问诊"准确定位病因，再对症下药。系统设计也一样：先定位瓶颈在哪一层，再从工具箱里挑对应的武器，而不是不管三七二十一先上缓存、上消息队列。

下面把常见的"瓶颈症状 → 对应武器"整理成一张对照表，每一件武器都在前面模块详细学过，这里只梳理"什么时候用、用在哪一层"。

症状 → 武器 速查表

症状对应武器用在哪一层
单机应用服务器扛不住并发连接 负载均衡 客户端与应用服务器集群之间
读多写少，同一份数据被反复查询 缓存 应用服务器与数据库之间
单库存储量或并发写扛不住 数据库读写分离 / 分库分表 数据库层
突发流量远超系统处理能力，但可以稍后处理 消息队列（削峰、异步） 应用服务器与下游系统之间
静态资源(图片/视频/JS/CSS)访问慢，用户地理位置分散 CDN 客户端与源站之间

负载均衡：解决"单机扛不住流量"

负载均衡（Load Balancer）位于客户端和应用服务器集群之间，按某种策略（轮询、最少连接数、一致性哈希等）把请求分发到多台服务器上，本质是把"纵向堆硬件"变成"横向加机器"——这呼应 10-计算机网络 里讲过的四层/七层负载均衡的区别： 四层（如 LVS）工作在传输层，只看 IP/端口做转发，性能高但不感知业务；七层（如 Nginx）能看到 HTTP 内容，可以按 URL 路径路由、做灰度发布，更灵活但开销更大。

什么时候用：只要应用服务器数量 > 1，就需要负载均衡——它是水平扩展的前提。判断依据很简单：单台服务器的 CPU/内存/连接数已经打满，而业务逻辑本身没有明显瓶颈，加机器 + 负载均衡就是最直接的解法。

缓存：解决"读多写少"

缓存把高频访问的数据放在内存里，用空间换时间，避免每次读请求都打到数据库。这是 12-缓存 模块的核心内容，选型时要想清楚几个问题：

  • 加在哪一层：本地缓存（如 Caffeine，进程内，速度最快但多实例间不一致）还是分布式缓存（如 Redis，多实例共享，需要一次网络往返）？
  • 用什么淘汰策略：LRU、LFU 还是设置 TTL 过期？
  • 怎么防止缓存穿透/击穿/雪崩：这是加缓存之后必须同步考虑的副作用，不是"锦上添花"而是"必答题"。
什么时候用：核心判断标准是读写比例——如果一份数据写一次要被读成千上万次（比如短链的长 URL 映射、商品详情页），加缓存的收益极大；反过来如果是写多读少甚至读写均衡的数据（比如日志流水），加缓存的收益有限，还平添了缓存一致性的维护成本。

数据库读写分离与分库分表：解决"单库扛不住"

这是 09-数据库 讲过的扩展手段，本质是把"单点数据库"拆成多个：读写分离解决的是"读并发扛不住"——一主多从，写走主库，读分散到多个从库，代价是主从复制存在延迟，可能读到旧数据； 分库分表解决的是"单库存储量或写并发扛不住"——按某个分片键把数据打散到多个库/表，代价是跨分片查询、分布式事务变得复杂。

面试高频陷阱：很多人一遇到"高并发"就无脑上分库分表，但分库分表是"重武器"，一旦引入就要解决跨库 JOIN、分布式事务、扩容再分片等一系列新问题。正确的顺序应该是：先加索引、再加缓存、再读写分离，实在扛不住存储量或写入量了，才考虑分库分表——这是成本递增的一条路径，不要一步到位。

消息队列：解决"突发流量"与"强耦合"

这是 13-消息队列 的核心应用场景，主要解决两类问题：削峰——把瞬时的高并发写请求先堆积到队列里，后端服务按自己能承受的速度慢慢消费，用"时间"换"当下的处理能力"； 异步解耦——上游发完消息就返回，不必等下游处理完成，两个系统之间不再直接依赖对方的可用性和响应时间。

什么时候用：判断标准是"这个操作能不能延迟处理"。比如下单扣库存必须实时，但下单后发短信通知、更新统计报表可以晚几秒甚至几分钟——凡是能容忍最终一致、不要求实时返回结果的写操作，都是消息队列的用武之地。秒杀系统是这个场景的典型代表，下一篇会具体展开。

CDN：解决"静态资源分发"

CDN（内容分发网络）把静态资源缓存到离用户地理位置更近的边缘节点，这也是呼应 10-计算机网络 里讲过的内容——用户请求图片/视频/JS/CSS 时，DNS 解析会把请求指向就近的边缘节点，而不是每次都跨越大半个地球回源站服务器，既降低了延迟也减轻了源站压力。

什么时候用：只要资源是"静态的、不常变化的、用户分布地理位置分散的"，就适合上 CDN。动态生成、强个性化的内容（比如用户的私人订单列表）不适合缓存到 CDN 边缘节点。

核心价值总结

  • 先定位瓶颈，再选武器：不要看到"高并发"三个字就无脑套用某个技术，先搞清楚瓶颈具体出在哪一层。
  • 成本是递增的：加索引 < 加缓存 < 读写分离 < 分库分表，越靠后的方案引入的复杂度越高，能用简单方案解决就不要上重武器。
  • 这些武器会在后面的案例里组合登场：短链系统会用到缓存，秒杀系统会同时用到缓存 + 消息队列 + 限流，真实系统往往是多种武器的组合拳，而不是单一技术的独角戏。

与其他知识点的联系

  • 关联知识点：短链系统案例（cases/01）：短链系统读 QPS 远高于写 QPS，正好对应本清单"读多写少上缓存"这一条武器；同时它的存储量估算最终导向分库分表，是"数据库读写分离/分库分表"武器在真实案例里的落地。这张速查表和短链案例是"方法论 → 实战"的一组对照。
  • 关联知识点：秒杀系统案例（cases/02）：秒杀系统同时用到了本清单里的三件武器——限流（挡瞬时流量）、缓存（扛高频读写）、消息队列（削峰异步）。它是本清单强调的"真实系统往往是多种武器组合使用"这一观点最典型的证明，单独看清单条目容易觉得每种技术各管一段，但秒杀案例展示了它们如何在同一个系统里协同工作。
  • 关联知识点：缓存穿透/击穿/雪崩（12-缓存）：清单里提到"加缓存之后必须同步考虑防穿透/击穿/雪崩"，这不是缓存的可选项而是必答题——本清单作为速查表只点出"什么时候该上缓存"，具体怎么防止这些副作用需要回到 12-缓存模块看详细方案，两者是"何时用"和"怎么用对"的分工关系。
  • 关联知识点：分布式与微服务的服务治理（14-分布式与微服务）：清单里没有单独列出微服务拆分，但负载均衡、消息队列这些武器在微服务架构下会进一步演化——比如负载均衡要考虑服务发现，消息队列要考虑跨服务的最终一致性。这张清单更多聚焦在"单体或简单分布式场景下的扩展手段"，微服务化之后需要叠加治理层面的新问题。
  • 关联知识点：计算机网络的分层负载均衡（10-计算机网络）：清单里"负载均衡解决单机扛不住流量"这一条，具体展开就是四层/七层负载均衡的选型问题——这提醒我们，速查表给出的只是"该用哪一类武器"的方向，真正落地时每一类武器内部往往还有更细的选型层次，不能止步于表格里的一行结论。
