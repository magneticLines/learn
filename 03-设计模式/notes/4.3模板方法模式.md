# 4.3 模板方法模式 (Template Method Pattern)

---

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"任何技术都是为了解决特定痛点而诞生的。"*

### 痛点分析

想象你在公司负责开发一个**数据导入系统**，需要支持从 Excel、CSV、JSON 三种格式导入数据。你写了三个类，很快发现一个尴尬的事实：

```java
// Excel导入
public void importExcel() {
    validateFile();       // 1. 校验文件 —— 三个类几乎一样
    parseExcel();         // 2. 解析数据 —— 每个类不同！
    transformData();      // 3. 数据转换 —— 三个类几乎一样
    saveToDatabase();     // 4. 存入数据库 —— 三个类完全一样
    sendNotification();   // 5. 发送通知 —— 三个类完全一样
}

// CSV导入
public void importCsv() {
    validateFile();       // 重复！
    parseCsv();           // 只有这里不同
    transformData();      // 重复！
    saveToDatabase();     // 重复！
    sendNotification();   // 重复！
}
```

5 个步骤中有 4 个是重复的，只有"解析"这一步因格式不同而不同。如果有一天老板说"导入完成后要加一步数据审计"，你要**改三个类**。改漏一个？线上 Bug。

**核心矛盾：算法的整体流程是固定的，但其中某些步骤的具体实现是变化的。**

### 核心定义

> **模板方法模式：在父类中定义一个算法的骨架（步骤顺序），将某些步骤的具体实现延迟到子类。子类可以在不改变算法整体结构的情况下，重新定义某些步骤。**

用一句话说：**流程我定，细节你填。**

## 2. 🧩 抽象与心智模型 (The Mental Model)

> **设计视角:** *"不仅要看代码，更要看懂设计。"*

### 生活类比：做菜的标准化流程

想象你是一家连锁餐厅的**总厨**，培训新厨师做菜。你制定了一套标准流程：

```
做一道菜的标准流程（SOP）：
  第一步：备料        ← 每道菜不同（切牛肉 vs 洗青菜）
  第二步：热锅烧油    ← 每道菜都一样
  第三步：烹饪        ← 每道菜不同（爆炒 vs 清蒸）
  第四步：装盘        ← 每道菜都一样
  第五步：撒葱花？    ← 可选步骤，有的菜要，有的菜不要
```

你（父类/总厨）定义了"做菜必须按这个顺序来"，新厨师（子类）只需要填写"备料怎么备"、"烹饪怎么烹"。**流程不能乱，但细节可以变。**

这里的关键角色：
- **模板（SOP 手册）**：固定的流程顺序，谁都不能改
- **抽象步骤（备料、烹饪）**：必须由具体厨师去实现
- **钩子步骤（撒葱花？）**：有默认行为，但可以选择覆盖

### 抽象设计（黑盒模型）

```
┌─────────────────────────────────────────┐
│         AbstractClass（总厨/SOP）          │
│                                         │
│  templateMethod() {     ← final！不可覆盖  │
│      step1();           ← 抽象方法        │
│      step2();           ← 具体方法（已实现） │
│      step3();           ← 抽象方法        │
│      if (hook()) {      ← 钩子方法        │
│          step4();                        │
│      }                                   │
│  }                                       │
└──────────────┬──────────────┬────────────┘
               │              │
    ┌──────────▼──┐    ┌──────▼──────────┐
    │ ConcreteA    │    │ ConcreteB       │
    │ step1() { } │    │ step1() { }    │
    │ step3() { } │    │ step3() { }    │
    │             │    │ hook() = false  │
    └─────────────┘    └────────────────┘
```

三种方法的本质区别：

| 方法类型 | 谁定义 | 谁实现 | 能否覆盖 |
|---------|--------|--------|---------|
| **模板方法** templateMethod() | 父类 | 父类 | ❌ final 禁止 |
| **抽象方法** step1(), step3() | 父类声明 | 子类必须实现 | ✅ 强制覆盖 |
| **钩子方法** hook() | 父类（给默认值） | 子类可选覆盖 | ✅ 可选覆盖 |

## 3. 🏗️ 系统架构与设计 (System & Architecture)

> **架构视角:** *"站得高，看得远。"*

### 在系统中的位置

模板方法模式无处不在，但它最常出现在**框架层（Framework Layer）**：

```
┌─────────────────────────────────────┐
│  你的业务代码（子类）                   │  ← 你写的：具体步骤实现
├─────────────────────────────────────┤
│  框架代码（父类/抽象类）                │  ← 框架写的：流程骨架
│  Spring / Servlet / JUnit / MyBatis  │
├─────────────────────────────────────┤
│  JDK 基础库                          │
└─────────────────────────────────────┘
```

**这就是"好莱坞原则"（Hollywood Principle）：Don't call us, we'll call you.**

传统编程：你的代码调用框架。
模板方法：框架调用你的代码。你只需要"填空"，框架决定"什么时候调用你的填空"。

### 关键组件

1. **AbstractClass（抽象模板）**：定义算法骨架 + 声明抽象步骤
2. **ConcreteClass（具体实现）**：实现抽象步骤的具体逻辑
3. **Template Method（模板方法）**：`final` 修饰，编排步骤顺序，不可被子类篡改
4. **Hook Method（钩子方法）**：提供默认行为，子类可以选择性覆盖来影响流程走向

### 体现的设计原则

- **开闭原则（OCP）**：新增一种导入格式？加一个子类即可，不用改父类
- **依赖倒置（DIP）**：高层模块（算法骨架）不依赖低层模块（具体步骤），都依赖抽象
- **好莱坞原则**：控制反转（IoC）的经典体现——父类控制流程，子类被动响应

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"优雅的代码是清晰思维的体现。"*

### 场景：数据导入系统

```java
/**
 * 抽象模板：数据导入的标准流程
 * 
 * 设计意图：将"导入"这个复杂流程拆解为固定步骤，
 * 把"不变的流程"锁在父类，把"变化的细节"留给子类。
 */
public abstract class AbstractDataImporter {

    /**
     * 模板方法 —— 定义算法骨架
     * 
     * 为什么用 final？因为流程顺序是业务规则，不允许子类打乱。
     * 比如：你不能先存数据库再校验文件，那会导致脏数据入库。
     */
    public final void importData(String filePath) {
        validateFile(filePath);           // 第1步：校验（通用逻辑）
        List<RawRecord> raw = parse(filePath);  // 第2步：解析（子类实现）
        List<Entity> entities = transform(raw); // 第3步：转换（通用逻辑）
        save(entities);                   // 第4步：持久化（通用逻辑）

        // 钩子方法：子类可以决定是否要发通知
        if (shouldNotify()) {
            sendNotification(entities.size());
        }
    }

    // ===== 具体方法：通用逻辑，所有子类共享 =====

    private void validateFile(String filePath) {
        System.out.println("校验文件是否存在、大小是否合规: " + filePath);
        // 实际中：检查文件格式、大小限制、权限等
    }

    private List<Entity> transform(List<RawRecord> rawRecords) {
        System.out.println("将原始记录转换为实体对象，共 " + rawRecords.size() + " 条");
        return rawRecords.stream()
                .map(r -> new Entity(r.getData()))
                .collect(Collectors.toList());
    }

    private void save(List<Entity> entities) {
        System.out.println("批量写入数据库，共 " + entities.size() + " 条");
        // 实际中：调用 Repository.saveAll(entities)
    }

    private void sendNotification(int count) {
        System.out.println("发送通知：成功导入 " + count + " 条数据");
    }

    // ===== 抽象方法：每种格式的解析方式不同，子类必须实现 =====

    /**
     * 为什么这是唯一的抽象方法？
     * 因为在5个步骤中，只有"解析"这一步真正因格式而异。
     * 设计原则：只抽象真正变化的部分，不要过度设计。
     */
    protected abstract List<RawRecord> parse(String filePath);

    // ===== 钩子方法：提供默认行为，子类可选择覆盖 =====

    /**
     * 钩子方法：是否在导入完成后发送通知？
     * 
     * 为什么用钩子而不是抽象方法？
     * 因为"发通知"不是必须由子类决定的核心差异，
     * 大多数情况下都要发，只有少数特殊情况不发。
     * 用钩子 = 给子类一个"可选的控制开关"。
     */
    protected boolean shouldNotify() {
        return true;  // 默认发通知
    }
}
```

```java
/**
 * 具体实现：Excel 导入器
 * 
 * 子类只需要关心"我怎么解析 Excel"这一件事，
 * 其他流程（校验、转换、存储、通知）全部复用父类。
 */
public class ExcelDataImporter extends AbstractDataImporter {

    @Override
    protected List<RawRecord> parse(String filePath) {
        System.out.println("使用 Apache POI 解析 Excel 文件: " + filePath);
        // 实际中：用 POI 读取 .xlsx，逐行转为 RawRecord
        return List.of(new RawRecord("excel-row-1"), new RawRecord("excel-row-2"));
    }
}

/**
 * 具体实现：CSV 导入器
 */
public class CsvDataImporter extends AbstractDataImporter {

    @Override
    protected List<RawRecord> parse(String filePath) {
        System.out.println("使用 OpenCSV 解析 CSV 文件: " + filePath);
        return List.of(new RawRecord("csv-row-1"), new RawRecord("csv-row-2"));
    }

    /**
     * 覆盖钩子：CSV 是内部批量工具，不需要通知
     * 
     * 注意：我们没有修改父类的任何代码，
     * 只是通过覆盖钩子方法来"影响"流程走向。
     * 这就是开闭原则的力量。
     */
    @Override
    protected boolean shouldNotify() {
        return false;
    }
}
```

```java
// 客户端使用
public class Client {
    public static void main(String[] args) {
        AbstractDataImporter importer;

        // 多态：客户端不关心具体是哪种导入器
        importer = new ExcelDataImporter();
        importer.importData("/data/report.xlsx");

        System.out.println("---");

        importer = new CsvDataImporter();
        importer.importData("/data/batch.csv");
    }
}
```

输出：
```
校验文件是否存在、大小是否合规: /data/report.xlsx
使用 Apache POI 解析 Excel 文件: /data/report.xlsx
将原始记录转换为实体对象，共 2 条
批量写入数据库，共 2 条
发送通知：成功导入 2 条数据
---
校验文件是否存在、大小是否合规: /data/batch.csv
使用 OpenCSV 解析 CSV 文件: /data/batch.csv
将原始记录转换为实体对象，共 2 条
批量写入数据库，共 2 条
（没有通知 —— 钩子生效）
```

### Spring 中的真实案例：AbstractApplicationContext.refresh()

这是 Spring 框架中**最经典的模板方法**，每个 Spring 开发者都在间接使用它：

```java
// Spring 源码（简化版）
public abstract class AbstractApplicationContext {

    // 这就是模板方法！Spring 容器启动的13个步骤全在这里
    public void refresh() {
        prepareRefresh();                    // 1. 准备刷新（具体方法）
        ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();
        prepareBeanFactory(beanFactory);     // 3. 配置工厂（具体方法）

        postProcessBeanFactory(beanFactory); // 🪝 钩子！子类可以定制 BeanFactory
        
        invokeBeanFactoryPostProcessors(beanFactory);
        registerBeanPostProcessors(beanFactory);
        initMessageSource();
        initApplicationEventMulticaster();

        onRefresh();                         // 🪝 钩子！SpringBoot 在这里启动内嵌 Tomcat

        registerListeners();
        finishBeanFactoryInitialization(beanFactory);
        finishRefresh();
    }

    // 钩子方法：默认空实现，子类按需覆盖
    protected void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
    }

    protected void onRefresh() {
        // SpringBoot 的 ServletWebServerApplicationContext
        // 在这里覆盖，创建并启动内嵌的 Tomcat/Jetty/Undertow！
    }
}
```

**你每次启动 SpringBoot 应用，`onRefresh()` 这个钩子就在帮你启动 Tomcat。** 你从来没有手动调用过它——这就是好莱坞原则。

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"一切都是权衡 (It depends)。"*

### 优点 (Pros)

| 优点 | 说明 |
|------|------|
| **消除重复代码** | 公共流程只写一次，子类只写差异部分 |
| **流程不可篡改** | `final` 保证算法骨架的稳定性 |
| **扩展性好** | 新增格式只需加子类，符合开闭原则 |
| **框架设计利器** | 框架定骨架，用户填细节——这是所有框架的基本思路 |

### 代价 (Cons)

| 代价 | 说明 |
|------|------|
| **强制继承** | 子类和父类紧耦合，Java 单继承限制意味着你"用掉了"唯一的继承机会 |
| **理解成本** | 阅读代码时要在父类和子类之间跳转，调用链不直观 |
| **违反里氏替换风险** | 如果子类在钩子中做了父类预期之外的事情，可能破坏流程语义 |
| **步骤越多越僵硬** | 如果流程步骤经常变化（加步骤、删步骤），父类会频繁修改 |

### 对比分析：模板方法 vs 策略模式

这是面试高频题，也是理解"继承 vs 组合"的关键：

| 维度 | 模板方法 | 策略模式 |
|------|---------|---------|
| **核心机制** | 继承（is-a） | 组合（has-a） |
| **变化的粒度** | 算法中的**某些步骤** | **整个算法** |
| **控制权** | 父类控制流程，子类填细节 | 客户端选择策略，策略自己执行 |
| **耦合度** | 高（父子类紧耦合） | 低（策略可独立变化） |
| **运行时切换** | ❌ 不能（继承是编译时决定的） | ✅ 可以（组合可以运行时替换） |
| **适用场景** | 流程固定，步骤实现变化 | 多种算法可互相替换 |

**经验法则：**
- 如果你要固定一个**流程骨架**，只让某些步骤变化 → **模板方法**
- 如果你要在运行时**动态切换**整个算法 → **策略模式**
- 如果两者都需要 → 可以组合使用：模板方法定流程，某个步骤内部用策略模式选算法

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"计算机科学是互通的。"*

### 底层原理：控制反转（IoC）的本质

模板方法是**控制反转**最古老、最纯粹的实现形式：

```
传统调用（你控制框架）：
  你的代码 ──调用──> 库的函数

控制反转（框架控制你）：
  框架的模板方法 ──调用──> 你实现的抽象方法
```

Spring 的依赖注入（DI）只是 IoC 的一种形式，而模板方法是另一种。它们的共同本质是：**主控权从应用代码转移到框架代码。**

这也解释了为什么你学 Spring 的时候，总觉得"我的代码什么时候被调用的？"——因为你写的不是完整程序，而是模板方法中的**填空题答案**。

### JDK 中的模板方法

你已经在无意识中用过很多次了：

| 类 | 模板方法 | 你重写的步骤 |
|----|---------|------------|
| `java.io.InputStream` | `read(byte[], int, int)` | `read()` 单字节读取 |
| `java.util.AbstractList` | `get()`, `size()` 已用于 `iterator()` | 你实现 `get()` 和 `size()` |
| `javax.servlet.HttpServlet` | `service()` 根据请求方法分发 | `doGet()`, `doPost()` |
| `java.util.concurrent.AbstractQueuedSynchronizer` | `acquire()`, `release()` | `tryAcquire()`, `tryRelease()` |

最后一个 **AQS** 尤其重要——你之前学并发时接触的 `ReentrantLock`、`CountDownLatch`、`Semaphore`，它们的底层全都是基于 AQS 的模板方法模式！Lock 的 `lock()` 方法最终调用 AQS 的 `acquire()`，而 `acquire()` 内部调用你子类实现的 `tryAcquire()`。

### 💭 思考题

> **你之前学的代理模式中，Spring AOP 用动态代理在方法执行前后插入逻辑（@Before、@After）。模板方法模式也是在固定流程中插入自定义逻辑。那么问题来了：如果让你设计一个"请求处理管道"（类似 Filter 链），你会选择模板方法还是代理模式？还是说，有更合适的模式？**（提示：想想你大纲里第二阶段还没学的那个模式——责任链模式）
