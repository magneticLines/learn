02. 函数（短小 · 单一职责 · 参数）

函数是代码最基本的组织单元。Robert C. Martin 认为：函数写得好不好，决定了整个代码库是否可读、可维护、可测试。

核心主张：函数只做一件事

Uncle Bob 在书中反复强调一句话：

Functions should do one thing. They should do it well. They should do it only.
函数应该只做一件事，做好这件事，只做这件事。

这并非审美偏好，而是有工程逻辑支撑的设计原则：

  • 可读性：短小的函数名即文档，读代码像读散文，不需要滚动屏幕理解一个函数到底干了什么
  • 可测试性：职责单一的函数边界清晰，单元测试极易覆盖，不需要大量 mock
  • 可复用性：只做一件事的函数天然可组合，不会因为隐藏副作用让调用方踩坑
  • 可修改性：改动一件事只影响一个函数，修改范围可预测

判断一个函数是否"只做一件事"，有一个简单的测试：你能否用一句不含"然后（and/then）"的话描述它？ 如果描述里出现了"and"，说明它做了不止一件事。

原则一：函数要短小

Uncle Bob 建议函数通常不超过 20 行，理想状态是 5~10 行。很多人第一反应是"这不现实"，但背后的逻辑是：如果你需要很多行才能说清楚一件事，说明你把多件事混在了一起。

反例：一个做了太多事的函数

Java — 反例
public String renderUserPage(HttpRequest request) {
    // 1. 解析请求参数
    String userId = request.getParam("id");
    if (userId == null || userId.isBlank()) {
        return "<h1>Bad Request</h1>";
    }

    // 2. 查数据库
    User user = db.query("SELECT * FROM users WHERE id = " + userId);
    if (user == null) {
        return "<h1>User Not Found</h1>";
    }

    // 3. 鉴权
    String token = request.getHeader("Authorization");
    if (!isValid(token)) {
        return "<h1>Unauthorized</h1>";
    }

    // 4. 拼 HTML
    StringBuilder sb = new StringBuilder();
    sb.append("<h1>").append(user.getName()).append("</h1>");
    sb.append("<p>").append(user.getEmail()).append("</p>");
    return sb.toString();
}

问题出在哪？ 这个函数干了四件完全不同层次的事：参数解析、数据库查询、权限校验、HTML 渲染。任何一件事需要改动都要修改这整个函数，而且无法对单独的步骤写单元测试。

正例：提炼函数，让每一层只做一件事

Java — 正例
public String renderUserPage(HttpRequest request) {
    // 顶层函数：只负责编排流程，像目录一样可读
    String userId = extractUserId(request);
    validateAuthorization(request);
    User user = findUserOrThrow(userId);
    return renderProfile(user);
}

private String extractUserId(HttpRequest request) {
    String id = request.getParam("id");
    if (id == null || id.isBlank()) {
        throw new BadRequestException("Missing user id");
    }
    return id;
}

private User findUserOrThrow(String userId) {
    User user = userRepository.findById(userId);
    if (user == null) {
        throw new NotFoundException("User not found: " + userId);
    }
    return user;
}

private String renderProfile(User user) {
    return "<h1>" + user.getName() + "</h1><p>" + user.getEmail() + "</p>";
}

现在顶层函数读起来像一段自然语言，每个子函数都可以独立测试，修改某一步骤不影响其他步骤。

原则二：同一函数只保持一个抽象层次

这是一个很容易忽略却极其关键的原则。函数内部的语句应该处于同一抽象层次，不能把"战略决策"和"实现细节"混在同一个函数里。

顶层函数（战略层）
├── renderUserPage() ← 只编排：提取、校验、查找、渲染
│
中间层（战术层）
├── findUserOrThrow() ← 只负责查找并处理缺失
├── validateAuthorization() ← 只负责权限校验逻辑
│
底层函数（实现层）
└── renderProfile() ← 只负责 HTML 字符串拼装
常见陷阱：在高层函数中夹杂低层操作。例如在"处理订单"的函数里直接写 rs.getString("order_status")——这是在把 SQL 字段名泄漏到业务逻辑层，一旦字段改名，业务逻辑也要改。

原则三：参数越少越好

Uncle Bob 给出了一个明确的参数数量建议：

参数个数术语评价
0零参（niladic）最理想
1一参（monadic）很好
2二参（dyadic）可接受，注意顺序
3三参（triadic）尽量避免，需要充分理由
3+多参（polyadic）强烈反对，应该封装成对象

为什么参数多了不好？ 每个参数都是一个需要读者理解的概念，增加了认知负担；测试时参数的组合爆炸；参数顺序容易搞错（特别是同类型的多个参数）。

反例：参数过多

Java — 反例
// 调用方：谁能一眼看出这 5 个参数分别是什么含义？
createUser("Alice", "alice@example.com", "ADMIN", true, 30);

// 函数定义本身也很难读
public User createUser(String name, String email,
                       String role, boolean active, int age) { ... }

正例：用参数对象封装

Java — 正例
// 参数对象：把相关参数聚合成一个有意义的概念
UserCreateRequest req = UserCreateRequest.builder()
    .name("Alice")
    .email("alice@example.com")
    .role(Role.ADMIN)
    .active(true)
    .age(30)
    .build();

createUser(req);   // 调用方清晰，参数含义一目了然

public User createUser(UserCreateRequest req) { ... }

Builder 模式的好处不只是可读性——它还让参数可选、有默认值，避免了"参数顺序写错"这类低级 bug。

特别警告：标志参数（Flag Argument）

标志参数（布尔型参数）是一种常见的坏味道，它直接违反了单一职责原则：向函数传递 boolean，本质上是在告诉函数"根据这个标志，做两件不同的事"。

反例

Java — 反例（标志参数）
// 调用方：true 是什么意思？需要翻函数定义才知道
renderPage(page, true);
renderPage(page, false);

public void renderPage(Page page, boolean isMobile) {
    if (isMobile) {
        renderMobilePage(page);  // 分支一
    } else {
        renderDesktopPage(page); // 分支二
    }
}

正例：拆成两个函数

Java — 正例
// 调用方：意图清晰，一目了然
renderMobilePage(page);
renderDesktopPage(page);

// 每个函数只做一件事，各自独立测试、独立修改
public void renderMobilePage(Page page) { ... }
public void renderDesktopPage(Page page) { ... }

原则四：无副作用（Side-Effect Free）

函数声称只做一件事，却偷偷改了另一件事——这是最危险的代码气味。副作用让代码行为难以预测，是并发 bug 和诡异 bug 的温床。

反例：名字骗人，暗藏副作用

Java — 反例
// 函数名声称"校验密码"，实际上还偷偷初始化了 session
public boolean checkPassword(String username, String password) {
    User user = userRepo.findByUsername(username);
    if (user.passwordMatches(password)) {
        // 隐藏的副作用！调用方完全不知道这里还会初始化 session
        Session.initialize(user);
        return true;
    }
    return false;
}

如果某处只想校验密码而不想启动 session（例如权限检查），调用这个函数就会产生意外的副作用。

正例：职责分离，副作用显式

Java — 正例
public boolean checkPassword(String username, String password) {
    User user = userRepo.findByUsername(username);
    return user.passwordMatches(password);
    // 只做校验，不做任何状态修改
}

public void login(String username, String password) {
    if (!checkPassword(username, password)) {
        throw new AuthException("Invalid credentials");
    }
    Session.initialize(username);  // 副作用放在明确语义的函数里
}

原则五：命令与查询分离（CQS）

函数要么做某件事（命令：修改状态），要么回答某个问题（查询：返回值），不能两者兼顾。这是 Bertrand Meyer 提出的原则，Uncle Bob 在书中也强调了它。

Java — 反例（命令 + 查询混用）
// 既设置属性，又返回是否成功——调用方阅读 if 条件时会困惑
if (setAttribute("username", "Alice")) {
    // 这里 setAttribute 的返回值代表什么？设置成功？还是之前已存在？
}
Java — 正例（命令与查询分离）
// 先查询（问题），再命令（行动）——各司其职
if (!attributeExists("username")) {   // 查询
    setAttribute("username", "Alice");  // 命令
}

实际开发落地技巧

1. "提炼函数"是重构的首选动作

当你看到一段需要注释才能解释的代码块，就是提炼函数的信号。注释的内容正是新函数的名字。 用函数名替代注释，代码变得自解释。

2. 函数越短，命名越重要

短函数只有名字暴露在外，名字必须精准。不要怕名字长，getUserByEmailAndVerifyActive() 比 getUser2() 好一百倍。长名字不是问题，模糊名字才是。

3. 输出参数是坏味道，用返回值替代

Java 中用参数作为输出（out 参数）虽然语言允许，但会让调用方困惑。能用返回值解决的，绝不用修改入参的方式。

Java — 输出参数 vs 返回值
// 坏：通过修改入参传出结果，调用方不知道参数会被改
appendFooter(report);  // report 被修改了？还是被读取了？

// 好：返回新对象，意图清晰，原对象不变
report = withFooter(report);

4. 异常优于错误码

返回错误码（-1、null、false）会迫使调用方在每次调用后做条件判断，把错误处理与正常逻辑混在一起。抛出异常后，正常流程和异常处理可以清晰分开。

Java — 错误码 vs 异常
// 坏：调用方必须检查每一步的返回值，正常逻辑被错误处理淹没
if (deletePage(page) == ErrorCode.OK) {
    if (registry.deleteReference(page.name()) == ErrorCode.OK) { ... }
}

// 好：正常逻辑集中，异常统一捕获处理
try {
    deletePage(page);
    registry.deleteReference(page.name());
} catch (Exception e) {
    logger.error(e.getMessage());
}
进阶技巧： try/catch 块本身就是一种"错误处理"职责，可以把它提炼成独立函数，让正常逻辑和异常处理在代码结构上也保持分离。

一张图总结函数设计原则

函数设计原则
├── 短小 ← 理想 5~10 行，超过 20 行要警惕
├── 单一职责 ← 一句话能描述，且不包含 "and"
├── 同一抽象层次 ← 战略和细节不混在同一函数
├── 参数尽量少 ← 超过 2 个考虑参数对象
├── 无标志参数 ← bool 参数 = 暗示函数做了两件事
├── 无副作用 ← 函数行为要与名字完全一致
├── 命令查询分离 ← 要么修改状态，要么返回结果，不兼顾
└── 异常优于错误码 ← 让正常流程和错误处理各就各位
写函数的正确姿势： 先写出能跑通的"长而乱"版本，然后打磨——提炼函数、改名、消除重复、整理参数。好的函数不是一次写出来的，是重构出来的。
