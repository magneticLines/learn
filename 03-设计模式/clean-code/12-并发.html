12. 并发（Concurrency）

并发是"做什么"与"何时做"的解耦。它提升吞吐量，但也把复杂性引入代码的每一个角落——Robert C. Martin 在本章的核心主张是：把并发当作独立的关注点，用设计将它隔离，而不是让它渗透到业务逻辑中。

为什么并发本质上很难？

并发的难不在于 API，而在于它颠覆了你对"程序执行顺序"的直觉。

  • 时序不确定：两个线程交替执行，结果取决于操作系统调度，而不是你的代码顺序。
  • 共享状态是万恶之源：只要有可变的共享变量，所有读写该变量的代码路径都变成了潜在的竞态条件。
  • Bug 不可复现：并发 Bug 往往在压测时才出现，甚至只在某台服务器的特定时间窗口出现，极难调试。
  • 性能与正确性常常对立：锁越细粒度，吞吐量越高；但代码越复杂，死锁风险越大。
Martin 的原话："并发编程很难。如果你没有把它当作一个独立的关注点来对待，它就会腐蚀你的设计、你的测试，以及你保持代码整洁的能力。"

核心主张：并发防御原则

原则一：单一职责原则（SRP）同样适用于并发

并发相关的代码有自己的开发节奏、自己的失败原因、自己的挑战。把并发代码和业务逻辑代码混在一起是最常见的错误。

反例：业务逻辑与并发耦合在一起Java
// 问题：OrderProcessor 既负责订单业务，又负责并发控制
public class OrderProcessor {
    private final List<Order> pendingOrders = new ArrayList<>();

    // 业务逻辑与同步锁混杂——两种职责，一旦改业务逻辑就要小心锁
    public synchronized void addOrder(Order order) {
        if (order.isValid()) {
            order.applyDiscount();          // 业务
            order.updateInventory();       // 业务
            pendingOrders.add(order);    // 并发保护
        }
    }
}
正例：并发隔离到独立的基础设施层Java
// 业务逻辑类——对线程一无所知，专注订单规则
public class OrderProcessor {
    public Order process(Order order) {
        if (!order.isValid()) throw new InvalidOrderException();
        order.applyDiscount();
        order.updateInventory();
        return order;
    }
}

// 并发容器——只负责线程安全的存储与调度，不含业务逻辑
public class ConcurrentOrderQueue {
    private final BlockingQueue<Order> queue = new LinkedBlockingQueue<>();
    private final OrderProcessor processor;

    public void submit(Order order) {
        queue.put(order);   // 线程安全由 BlockingQueue 保证
    }
}

原则二：限制共享数据的作用域

共享数据越少，竞态条件越少。Martin 建议：尽可能使用局部变量，将共享数据封装在尽量小的范围内，并通过访问器方法统一保护。

反例：共享可变字段暴露给外部Java
public class Counter {
    public int count = 0;  // public 字段，任何线程随时读写，无保护
}

// 两个线程同时执行 counter.count++
// count++ 实际是 read → add 1 → write，三步非原子
// 可能两个线程都读到 0，都写 1，最终 count=1 而非 2
正例：用 AtomicInteger 封装，或用同步访问器Java
public class Counter {
    // AtomicInteger 利用 CAS 指令，无锁且线程安全
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // 原子操作，不可分割
    }

    public int get() {
        return count.get();
    }
}

原则三：优先使用数据副本，而非共享

如果对象不可变（immutable），就不需要同步。如果可以给每个线程发一份数据副本、最后再合并结果，代码简单得多。这正是 Java Stream 并行流、Hadoop MapReduce 的核心思想。

示例：不可变值对象——天然线程安全Java
// final 字段 + 无 setter = 不可变对象
// 任意多个线程可以同时读，没有竞态，不需要锁
public final class Money {
    private final BigDecimal amount;
    private final Currency   currency;

    public Money(BigDecimal amount, Currency currency) {
        this.amount   = Objects.requireNonNull(amount);
        this.currency = Objects.requireNonNull(currency);
    }

    // "修改"时返回新对象，原对象保持不变
    public Money add(Money other) {
        return new Money(this.amount.add(other.amount), this.currency);
    }
}

原则四：线程应该尽量独立

理想情况下，每个线程都像一座孤岛：拥有自己的数据来源，不和其他线程共享任何状态。Web 请求处理就是最好的例子——每个请求分配一个线程，数据来自请求参数，结果写入响应流，线程间互不干涉。

了解常见的并发执行模型

Martin 强调：在没搞清楚问题归属哪种模型之前，不要乱用锁。

模型 核心问题 典型场景 解决工具
生产者-消费者 队列边界同步 异步任务、消息队列 BlockingQueue
读者-写者 读多写少时的吞吐 vs 一致性 缓存、配置热更新 ReadWriteLock
哲学家进餐 多个线程竞争多个资源，易死锁 转账、多资源预订 固定加锁顺序 / tryLock
死锁的四个必要条件（Coffman 条件）：互斥、持有等待、不可抢占、循环等待。只要破坏其中一个就能避免死锁。最实用的手段：固定加锁顺序（破坏循环等待）。

同步方法 vs 同步代码块

Martin 的建议：将同步方法设计为只含同步逻辑，把真正的业务计算移到同步范围之外。锁的持有时间越短，竞争越少，吞吐越高。

反例：锁的粒度太粗，计算逻辑也在锁内Java
public synchronized BigDecimal computeAndStore(List<Order> orders) {
    // 这段聚合计算可能耗时很长，但整个方法都持有锁
    // 其他线程全部阻塞等待——吞吐量大幅下降
    BigDecimal total = orders.stream()
        .map(Order::getAmount)
        .reduce(BigDecimal.ZERO, BigDecimal::add);
    this.cachedTotal = total;
    return total;
}
正例：计算在锁外完成，只在写入时加锁Java
public BigDecimal computeAndStore(List<Order> orders) {
    // 1. 耗时计算在锁外进行，不阻塞其他线程
    BigDecimal total = orders.stream()
        .map(Order::getAmount)
        .reduce(BigDecimal.ZERO, BigDecimal::add);

    // 2. 仅在写入共享状态时加锁，持锁时间极短
    synchronized (this) {
        this.cachedTotal = total;
    }
    return total;
}

如何测试并发代码

Martin 承认：并发 Bug 无法被测试完全消除，但可以通过设计降低其出现概率。几条核心建议：

  • 把并发与非并发代码分离——业务逻辑可以单线程充分测试，并发容器可以单独压测。
  • 让非线程安全代码可以在不同配置下运行：单线程、多线程、随机线程数。
  • 用 Thread.yield()、Thread.sleep() 在关键节点人为引入调度切换，提高并发问题暴露概率。
  • 使用工具辅助：Java 的 java.util.concurrent.atomic、ThreadSanitizer（Go/C++）、FindBugs / SpotBugs 静态分析。
示例：用 CountDownLatch 制造真实的并发竞争Java
@Test
void shouldMaintainCorrectCountUnderConcurrency() throws Exception {
    Counter counter     = new Counter();
    int threadCount     = 100;
    CountDownLatch ready = new CountDownLatch(threadCount);
    CountDownLatch start = new CountDownLatch(1);
    CountDownLatch done  = new CountDownLatch(threadCount);

    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            ready.countDown();   // 告知主线程：我已就位
            start.await();        // 等待发令枪，所有线程同时冲
            counter.increment();
            done.countDown();
        }).start();
    }

    ready.await();   // 等所有线程就位
    start.countDown();  // 同时放行，最大化竞争概率
    done.await();

    assertEquals(threadCount, counter.get());
}

开发者最容易忽略的细节

陷阱一：复合操作不是原子操作

if (map.containsKey(k)) map.get(k) 这两步之间，另一个线程完全可能删掉该 key。使用 ConcurrentHashMap.computeIfAbsent() 替代。

反例 vs 正例Java
// 反例：check-then-act 竞态
if (!cache.containsKey(key)) {
    cache.put(key, expensiveCompute(key));  // 可能被两个线程重复计算
}

// 正例：原子的 computeIfAbsent
cache.computeIfAbsent(key, k -> expensiveCompute(k));

陷阱二：volatile 不能替代 synchronized

volatile 只保证可见性（一个线程的写对其他线程立刻可见），不保证原子性。count++ 加了 volatile 仍然是三步操作，仍然有竞态。

volatile 保证：可见性（happens-before）
volatile 不保证：原子性、有序性（指令重排序仍可能发生）

需要原子性 → AtomicXxx / synchronized / Lock
只需可见性（如标志位 stop） → volatile 足够

陷阱三：使用 java.util.concurrent 而非手写锁

JDK 并发工具经过大量工程实践验证，手写的双重检查锁、读写锁往往存在微妙 Bug。优先顺序：

不可变对象 > java.util.concurrent 容器
> AtomicXxx > synchronized 方法/块
> ReentrantLock / ReadWriteLock
> 手写底层同步（几乎永远不需要）

陷阱四：忽略中断机制

如果线程在 wait()、sleep()、join() 时被中断，会抛出 InterruptedException。不要吞掉它——要么重新设置中断标志，要么向上抛出。

反例 vs 正例Java
// 反例：吞掉中断，调用者无法感知线程被中断
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 啥都不做——中断信号消失了
}

// 正例：恢复中断标志，让上层代码有机会处理
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复中断状态
}

落地技巧速查

场景推荐方案避免
只读共享配置 不可变对象 + volatile 引用 synchronized 读（过度锁）
计数器、序列号 AtomicLong synchronized int
共享缓存 ConcurrentHashMap synchronized HashMap
线程间任务传递 BlockingQueue 共享 List + 手写锁
并发读、偶尔写 ReadWriteLock / StampedLock 全程互斥锁
线程局部状态 ThreadLocal 共享字段 + 线程 ID 区分
Martin 的最终告诫：并发是一个持续演化的领域。语言和框架会提供更好的抽象（如 Java 的虚拟线程、Go 的 goroutine、Kotlin 的协程），但底层竞态和死锁的本质不会变。理解原理，才能正确使用任何工具。

与本书其他章节的联系

  • 单一职责（SRP）：并发代码应该是独立的类/模块，不与业务逻辑混合——正是 SRP 在架构层面的体现。
  • 开放-封闭原则：用 BlockingQueue 等抽象隔离并发，生产者和消费者对彼此封闭，对队列抽象开放。
  • 整洁测试：并发 Bug 的测试需要精心构造时序，测试代码本身也应当整洁，不要用 Thread.sleep 猜测时序。
  • 涌现式设计（上一节）：持续重构并发代码，把共享状态逐步减少，是让设计持续涌现的一部分。
