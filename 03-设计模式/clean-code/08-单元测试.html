08. 单元测试（F.I.R.S.T · 整洁测试）

测试代码与生产代码同等重要。脏测试比没有测试更危险——它让你误以为系统被保护了。

作者的核心主张：测试是第一公民

Robert C. Martin 在本章的核心论点只有一句话：测试代码和生产代码一样需要被认真对待。 很多团队把测试当作"写完功能顺手写一下"的附属品，结果测试越来越难维护，最终被删掉或忽视—— 这比从来没有测试还要危险，因为你会产生虚假的安全感。

Clean Code 给出了两大武器：

  • 整洁测试（Clean Tests）：每个测试只做一件事，可读性第一
  • F.I.R.S.T 原则：好测试必须满足的五个特性
为什么测试如此重要？
测试是让你敢于重构的唯一保障。没有测试的代码，每次修改都是在走钢丝；有了整洁的测试，你可以大胆改动内部实现，只要测试全绿，行为就没有被破坏。

整洁测试的三要素：可读性

Bob 说：测试的可读性比生产代码更重要。测试是活的文档——它描述系统在各种场景下应该做什么。 一个好测试由三个步骤组成，俗称 Build-Operate-Check（构建-操作-检验），也叫 Given-When-Then：

// Given（准备） → 构造测试数据 / 初始化对象
// When（操作） → 调用被测方法
// Then（验证） → 断言结果是否符合预期

反例：难以阅读的测试

Java · 反例❌ 一眼看不出测试意图
@Test
public void testWithdraw() {
    BankAccount a = new BankAccount();
    a.deposit(1000);
    a.deposit(500);
    a.setFee(10);
    a.withdraw(200);
    a.withdraw(300);
    assertTrue(a.getBalance() == 980);
    assertTrue(a.getTransactionCount() == 4);
}

问题在哪里？

  • 测试了多个行为（余额 + 交易次数），失败时不知道是哪个坏了
  • 980 这个数字从哪来？读者需要心算才能验证
  • 命名 testWithdraw 没有描述具体场景，是正常取款还是余额不足？

正例：每个测试只做一件事

Java · 正例✅ Given-When-Then 结构清晰
@Test
public void balance_decreasesByAmountAndFeeAfterWithdraw() {
    // Given：账户有 1500 元，手续费 10 元
    BankAccount account = accountWithBalance(1500).fee(10).build();

    // When：取款 500 元
    account.withdraw(500);

    // Then：余额 = 1500 - 500 - 10 = 990
    assertThat(account.getBalance()).isEqualTo(990);
}

@Test
public void transactionCount_incrementsAfterEachOperation() {
    // Given / When / Then 分别独立，只验证一件事
    BankAccount account = new BankAccount();
    account.deposit(100);
    account.deposit(200);
    assertThat(account.getTransactionCount()).isEqualTo(2);
}
命名即文档：测试方法名用下划线分隔的 被测方法_场景_预期结果 格式（如 withdraw_insufficientBalance_throwsException）， 比 testWithdraw2 有用一百倍。测试失败时，你能从方法名直接读出是什么场景在什么条件下失败了。

F.I.R.S.T 五原则

Bob 把好测试的五个必要特性浓缩为一个缩写。缺少任意一条，测试的价值都会大打折扣：

字母 全称 含义
F Fast（快速） 测试必须跑得快。跑得慢的测试没人愿意频繁执行，它就失去了守护作用。
I Independent（独立） 测试之间不能有顺序依赖。A 测试的结果不应该影响 B 测试。
R Repeatable（可重复） 在任何环境（本地、CI、无网络）重复运行结果都应相同。
S Self-Validating（自我验证） 测试结果只有两种：Pass 或 Fail，不需要人工查日志来判断。
T Timely（及时） 测试应当在生产代码之前或同步编写，而不是事后补写。

F — Fast（快速）

单元测试如果需要连接数据库、启动 Spring 容器或访问网络，一套测试可能要跑几分钟。 开发者就会减少运行频率，测试的"快速反馈"价值归零。

Java · Fast 反例 vs 正例
// ❌ 反例：测试直接调用真实数据库，速度慢、不稳定
@Test
void findUser_returnsCorrectUser() {
    // 连接真实 MySQL，需要环境配置，慢！
    User user = userRepository.findById(1L).orElseThrow();
    assertThat(user.getName()).isEqualTo("Alice");
}

// ✅ 正例：用 Mock 隔离外部依赖，毫秒级完成
@Test
void findUser_returnsCorrectUser() {
    when(userRepository.findById(1L))
        .thenReturn(Optional.of(new User(1L, "Alice")));

    User user = userService.findUser(1L);

    assertThat(user.getName()).isEqualTo("Alice");
}

I — Independent（独立）

测试之间共享状态是最常见的坑。当测试 A 修改了某个静态变量或数据库记录， 测试 B 依赖这个副作用才能通过，那么单独跑 B 就会失败——这让问题定位变得极其困难。

Java · Independent 反例 vs 正例
// ❌ 反例：测试依赖执行顺序（先存再取）
static Map<Long, User> sharedStore = new HashMap<>();

@Test void test1_saveUser() {
    sharedStore.put(1L, new User(1L, "Alice"));
}

@Test void test2_getUser() {
    // 如果 test1 没先跑，这里就是 null！
    assertThat(sharedStore.get(1L).getName()).isEqualTo("Alice");
}

// ✅ 正例：每个测试在 @BeforeEach 中独立初始化自己的数据
Map<Long, User> store;

@BeforeEach
void setUp() {
    store = new HashMap<>(); // 每次测试都是全新状态
}

@Test void getUser_returnsStoredUser() {
    store.put(1L, new User(1L, "Alice"));
    assertThat(store.get(1L).getName()).isEqualTo("Alice");
}

R — Repeatable（可重复）

如果测试在你的机器上通过，在 CI 上失败，那它实际上没有守护任何东西——你只会养成"忽略偶发失败"的坏习惯。 常见的不可重复来源：new Date()、随机数、外部 HTTP 接口、文件系统的绝对路径。

Java · Repeatable 反例 vs 正例
// ❌ 反例：依赖当前时间，每天都可能结果不同
@Test
void order_isExpiredAfterOneDay() {
    Order order = new Order(LocalDate.now().minusDays(2));
    // 如果逻辑依赖 LocalDate.now()，结果随系统时间变化
    assertTrue(order.isExpired());
}

// ✅ 正例：注入固定时间，行为完全受控
@Test
void order_isExpiredAfterOneDay() {
    Clock fixedClock = Clock.fixed(
        Instant.parse("2024-01-10T00:00:00Z"), ZoneOffset.UTC
    );
    // 创建于两天前的订单，固定"今天"是 1-10，那就一定过期
    Order order = new Order(LocalDate.of(2024, 1, 8), fixedClock);
    assertTrue(order.isExpired());
}

S — Self-Validating（自我验证）

测试必须自己能判断对错。如果一个测试"通过了"，但你还需要肉眼看打印输出才能确认结果， 那它就不是真正的自动化测试。这条原则看似简单，但有一个常见的违反形式：

Java · Self-Validating 反例 vs 正例
// ❌ 反例：用打印代替断言，永远是绿色（即使逻辑错了）
@Test
void calculate_printsResult() {
    int result = calculator.add(2, 3);
    System.out.println("result = " + result); // 人工去看！
}

// ✅ 正例：断言即验证，测试框架自动判断
@Test
void add_returnsSumOfTwoNumbers() {
    assertThat(calculator.add(2, 3)).isEqualTo(5);
}

T — Timely（及时）

TDD（测试驱动开发）要求测试先于生产代码编写。即使你不严格践行 TDD， 也应该在写完一个方法后立刻写它的测试，而不是等模块全部完成再回头补。 事后补写的测试有两个问题：

  • 你会不自觉地用测试去"配合"已有实现，而不是去验证行为，测试就失去了独立的检验价值
  • 生产代码写多了，你会发现它对测试不友好（难以 Mock、硬编码依赖），重构成本很高
常见误区：测试覆盖率 ≠ 测试质量
100% 的行覆盖率并不意味着没有 bug。你可以写一堆只有 assertNotNull 或甚至没有断言的测试来刷覆盖率。 真正有价值的测试，是那些能在行为被破坏时立即变红的测试。

双重标准：测试代码可以与生产代码不同

Bob 提出一个有趣的点：测试代码有自己的"双重标准"。 测试中可以出现一些在生产代码里不允许的东西，因为测试跑在受控环境里， 对内存、性能的要求远低于生产系统。

  • 测试里可以用魔法数字（只要它能让断言自解释）
  • 测试里可以用 Builder 模式来构造复杂的测试数据，避免重复代码
  • 测试里可以对 private 方法保持不测（通过公开行为间接验证），不要破坏封装去测内部实现
Java · 测试专用 Builder（提升可读性）
// 测试 DSL：让 Given 阶段表意清晰，不关心无关参数
BankAccount account = BankAccountBuilder.anAccount()
    .withBalance(1000)
    .withFee(5)
    .frozen()          // 语义清晰：这是一个被冻结的账户
    .build();

// 而不是：
BankAccount account = new BankAccount(1000, 5, true, null, "VIP", false);

实际开发落地技巧

1. 测试命名：用句子描述行为

采用 方法名_场景_预期 或 BDD 风格的 should_预期_when_场景，让失败报告一眼就能定位问题。

✅ withdraw_insufficientBalance_throwsInsufficientFundsException
✅ should_return404_when_userNotFound
❌ testWithdraw
❌ test2

2. 一个测试，一个断言概念

"一个测试一个断言"有时过于严格（比如验证一个 VO 对象的多个字段）。 更准确的原则是：一个测试，一个概念。 验证"账户余额"是一个概念，即使需要两行断言（金额 + 货币单位）也可以放在同一个测试里。

3. 测试文件与生产代码镜像

src/main/java/com/example/service/OrderService.java
src/test/java/com/example/service/OrderServiceTest.java

保持相同的包路径，测试可以访问包级可见的成员，也便于快速导航。

4. 警惕测试中的条件逻辑

如果测试里出现了 if、for、try-catch（非异常验证场景）， 往往意味着你在一个测试里塞了太多东西，或者测试本身也需要被测试了。 把条件分支拆成独立的测试用例。

5. 异常测试的正确姿势

Java · 测试异常的三种方式对比
// ❌ 老写法：try-catch，冗长且容易写错
@Test
void withdraw_shouldThrow_whenBalanceInsufficient_old() {
    try {
        account.withdraw(99999);
        fail("应该抛出异常"); // 忘写这行就永远是绿的！
    } catch (InsufficientFundsException e) {
        // pass
    }
}

// ✅ JUnit 5 推荐写法：assertThrows，简洁且安全
@Test
void withdraw_throwsException_whenBalanceInsufficient() {
    assertThrows(
        InsufficientFundsException.class,
        () -> account.withdraw(99999)
    );
}

// ✅ 需要验证异常信息时，用 assertThatThrownBy（AssertJ）
@Test
void withdraw_exceptionMessage_containsAmount() {
    assertThatThrownBy(() -> account.withdraw(99999))
        .isInstanceOf(InsufficientFundsException.class)
        .hasMessageContaining("99999");
}

总结：用一句话记住这一章

核心记忆点：脏测试等于没有测试——它会随着代码演进腐烂，最终成为维护负担并被抛弃。 整洁测试（可读 + F.I.R.S.T）是让团队敢于持续重构的底气，也是系统行为最准确的活文档。
原则一句话核心违反后的代价
整洁 Given-When-Then，一个测试一个概念 测试失败时无法快速定位问题
Fast Mock 外部依赖，毫秒级完成 开发者减少运行频率，失去快速反馈
Independent @BeforeEach 独立初始化，无共享状态 测试顺序依赖，并行执行时随机失败
Repeatable 注入时间/随机数，不依赖外部环境 本地过、CI 挂，养成忽略失败的习惯
Self-Validating 断言而非打印，Pass/Fail 自动判断 测试永远绿色，bug 悄悄溜进生产
Timely 与代码同步写，而非事后补 生产代码对测试不友好，补写成本翻倍
