13. 代码味道与启发

全书的"收口清单"——Martin 把多年重构经验提炼为一张可随时对照的嗅觉地图，帮你在 Code Review 中快速定位腐化点。

核心主张：代码自带气味

Robert C. Martin 在全书最后一章给出了一份「代码味道（Code Smells）」清单，并为每种味道配上对应的「启发（Heuristics）」——即怎样消除它。 这不是规则手册，而是一套嗅觉训练系统：

  • 好的代码读起来像散文，坏的代码让你皱眉。皱眉的那一刻，就是"闻到味道"。
  • 味道背后都有设计原则（SRP、OCP、DRY、最小化依赖…），清除味道 = 让代码更靠近这些原则。
  • 清单的作用是让直觉变得可描述、可沟通，从"感觉不对"变成"这里违反了 X"。
Martin 将味道分为六大类：注释（C）、环境（E）、函数（F）、一般（G）、Java 特有（J）、测试（T）。本文以最常见、跨语言通用的类别为重点展开。

注释味道（C）

C1 不恰当的信息

注释里塞入变更历史、作者、日期——这些信息属于版本控制系统，放在代码里只会制造噪音。

反例 Java
// Author: Zhang San
// Created: 2023-01-05, Modified: 2024-03-12 by Li Si
// v2.3 fix: handle null pointer when user is not logged in
public String getUserName(User user) {
    return user.getName();
}
正例 Java
// 用 git blame / git log 查历史，代码自己说明逻辑
public String getUserName(User user) {
    return user.getName();
}

C2 废弃的注释

注释描述的代码已经不存在了，或者逻辑早已改变，但注释还留着。过时的注释比没有注释更有害——它在积极地误导读者。

C3 多余的注释

注释只是重复了代码本身已经表达清楚的事情。

反例 Java
int age = 0; // 将 age 初始化为 0
i++; // i 自增 1

C4 注释掉的代码

注释掉的代码是代码库的"僵尸"。没人敢删，因为不知道为什么留着；没人知道它还有没有用。直接删掉，版本控制会记住历史。

函数味道（F）

F1 过多的参数

函数参数越多，认知负担越重。三个以上的参数几乎总是意味着函数承担了太多职责，或者某几个参数应该被封装成对象。

反例 Java — 参数爆炸
public Order createOrder(
    String userId, String productId, int quantity,
    String couponCode, String address, boolean isPriority
) { ... }
正例 Java — 引入参数对象
public Order createOrder(CreateOrderRequest request) { ... }

// CreateOrderRequest 将相关字段内聚，还能加校验逻辑
public record CreateOrderRequest(
    String userId, String productId, int quantity,
    String couponCode, String address, boolean isPriority
) {}

F2 输出参数

通过参数"带出"结果而非通过返回值，是反直觉的。调用者必须读函数签名才能知道哪个参数会被修改。

反例 → 正例 Java
// 反例：report 被修改，但调用处看不出来
appendFooter(report);

// 正例：让对象自己负责附加行为
report.appendFooter();

F3 标记参数

布尔参数是"一个函数干两件事"的信号灯。应该拆成两个有名字的函数。

反例 → 正例 Java
// 反例：true/false 含义在调用处完全不透明
render(page, true);

// 正例：两个函数，名字即文档
renderForSuite(page);
renderForSingleTest(page);

F4 死函数

从未被调用的方法应当删除。不要把"也许将来有用"当理由——版本控制会保留历史。

一般味道（G）— 最多也最重要

G1 一个源文件存在多种语言

HTML 里内联大段 JavaScript，Java 文件里用字符串拼 SQL 片段。每种语言都应有自己的归宿，混用让工具链和人都难以理解。

G2 明显的行为未被实现

最小惊讶原则（Principle of Least Surprise）：函数或类的行为应与调用者的预期一致。 如果 Day.fromString("Monday") 不能正确解析大小写混合的字符串，就会让调用者踩坑。

G3 不正确的边界行为

不要依赖直觉，每一个边界条件都要明确测试：空集合、单元素、最大值、最小值、负数、null。直觉经常在边界处失效。

G4 忽视安全

被 // TODO: 以后处理 标注的安全检查，往往永远"以后"。Martin 强调：关闭编译器警告、忽略失败的测试都是危险信号，必须立即处理而不是推迟。

G5 重复（最重要的味道之一）

DRY（Don't Repeat Yourself）。每一处重复都是一个未来需要同步修改的定时炸弹。Martin 识别出几种重复形态：

  • 完全相同的代码片段：提取为函数
  • 相似但有细微差异的 switch/if-else 链：考虑多态替换
  • 算法相似但实现不同：TEMPLATE METHOD 或 STRATEGY 模式
反例 Java — switch 重复
// 在 Employee.calculatePay() 里
switch (type) {
    case COMMISSIONED: return calculateCommissionedPay();
    case HOURLY:       return calculateHourlyPay();
    case SALARIED:     return calculateSalariedPay();
}
// 在 Employee.deliverPay() 里又来一遍相同结构...
正例 Java — 多态消除 switch
// 抽象基类或接口定义契约
public abstract class Employee {
    public abstract Money calculatePay();
    public abstract void  deliverPay(Money pay);
}

// 每种类型只在自己的子类里实现，switch 彻底消失
public class HourlyEmployee extends Employee {
    @Override
    public Money calculatePay() { return hoursWorked.multiply(hourlyRate); }
}

G6 抽象层次错误

高层概念和低层细节不应混居在同一个类或函数里。接口只应出现高层行为，实现细节留给子类。

反例 Java — Stack 接口泄漏了底层细节
public interface Stack {
    void push(Object o);
    Object pop();
    // 这是底层细节，不该出现在接口里
    double percentFull();
}

G7 基类依赖派生类

父类不应该知道子类的存在。一旦父类 import 或引用了某个子类，部署独立性就被破坏了：子类的改动会迫使父类重新编译/部署。

G8 信息过多

良好设计的模块接口非常小，暴露的方法和变量尽量少。一个类暴露的接口越小，耦合度越低，越容易修改。 衡量标准：模块的"扇出"（调用其他模块的数量）要受控。

G9 死代码

永远不会被执行的代码：throw 之后的语句、永假 if 分支、从未被调用的私有方法。死代码会随时间腐化——删掉它。

G10 垂直分隔

变量和函数应该在靠近被使用的地方声明。私有函数应该紧跟在第一次被调用它的函数之后，而不是拉到文件底部。

G11 前后不一致

一旦你决定用某种方式做某件事（命名约定、错误处理方式、返回类型风格），就在整个代码库里保持一致。不一致本身就是噪音，让读者猜测"这里的差异是刻意的吗？"

G12 杂乱

无用的默认构造函数、没有实现的变量、从不用到的导入——这些都是杂乱。保持代码整洁意味着每一行的存在都有理由。

G13 人为耦合

不相互依赖的事物不应该被耦合。通常表现为：把一个枚举定义在了某个不相干的类里，导致所有需要用该枚举的地方都必须引入那个类。

G14 特性依恋（Feature Envy）

一个类的方法对另一个类的数据比对自己的数据更感兴趣，说明该方法应该搬到那个类里。

反例 → 正例 Java
// 反例：HourlyPayCalculator 大量访问 HourlyEmployee 的字段
class HourlyPayCalculator {
    Money calculate(HourlyEmployee e) {
        Money base = e.getRate().times(e.getHours());
        return base.plus(e.getOvertime());
    }
}

// 正例：逻辑移入 HourlyEmployee 自身
class HourlyEmployee {
    Money calculatePay() {
        return rate.times(hours).plus(overtime);
    }
}

G15 选择算法

对于一个明确的问题，应该只有一种"权威算法"。到处散布略有不同的实现，意味着没有人真正理解正确答案。提取并命名它。

G16 晦涩的意图

代码要尽可能表达作者意图。过度的位运算优化、魔法数字、行内表达式紧缩——这些都在牺牲可读性换取微乎其微的性能提升。

反例 → 正例 Java
// 反例：这个 0x0fc0 是什么？读者完全摸不着头脑
return (hours & 0x0fc0) >> 6;

// 正例：用常量和注释表达意图
private static final int OVERTIME_SHIFT_BITS = 6;
private static final int OVERTIME_MASK       = 0x0fc0; // 取 bit 6-11
return (hours & OVERTIME_MASK) >> OVERTIME_SHIFT_BITS;

G17 位置错误的职责

代码应该放在读者最自然的预期位置。PI 常量放在 Math 类而不是 Trigonometry 类，reportHours() 放在 HourlyEmployee 而不是 HRReporter。

G18 不恰当的静态方法

静态方法意味着"不需要任何特定实例的状态"。如果一个方法未来有可能需要多态行为（不同子类有不同实现），就不该定义为静态。当不确定时，选择实例方法。

G19 使用解释性变量

把一个大计算拆成多个步骤，给每个中间结果起有意义的名字，即使不能短，也要能自我解释。

正例 Java
Matcher match = headerPattern.matcher(line);
if (match.find()) {
    String key   = match.group(1);
    String value = match.group(2);
    // key 和 value 让下面的代码不言而喻
    headers.put(key.toLowerCase(), value.trim());
}

G20 函数名应表达其作用

如果你无法用一个精确的名字描述一个函数，说明它做的事情太多了。重命名往往也是一次重构的机会。

G21 理解算法

让代码"能跑"和"真的理解它在做什么"是两回事。如果你不能用自己的话解释一个函数的每一行，就没有真正理解它——也就没有资格声称它是正确的。

G22 用多态替代 if/else 或 switch

这是 G5 重复的延伸：相同结构的条件判断分散在多处，就应该用多态集中处理。规则：对同一个类型判断的 switch 语句，在系统中应该出现不超过一次。

G23 遵循标准惯例

团队应该约定一套编码标准，并且每个成员都遵守它。标准本身不如一致性重要——读者需要能够建立预期。

G24 用命名常量替代魔法数字

反例 → 正例 Java
// 反例：86400 是什么？
if (elapsed > 86400) expireSession();

// 正例：一目了然，修改也只要改一处
private static final long SECONDS_PER_DAY = 86_400L;
if (elapsed > SECONDS_PER_DAY) expireSession();

G25 精确

代码中的决策应该是精确的，而不是"大概是这样"。如果你期望只有一个结果，就用能保证唯一性的数据结构；如果涉及货币，就用 BigDecimal 而不是 double。

G26 结构优于惯例

用设计结构强制约束，比依赖开发者"记住"某个惯例更可靠。枚举 + switch 要求开发者记得处理所有 case；抽象方法则通过编译器强制每个子类必须实现。

G27 封装条件

复杂的布尔条件应该提取成具名函数，让意图清晰。

反例 → 正例 Java
// 反例：读者需要推断这个条件的含义
if (timer.hasExpired() && !timer.isRecurrent()) { ... }

// 正例：意图一读即懂
if (shouldBeDeleted(timer)) { ... }

private boolean shouldBeDeleted(Timer timer) {
    return timer.hasExpired() && !timer.isRecurrent();
}

G28 避免负向条件

否定条件比肯定条件更难理解。在可能的情况下，反转条件让它变成肯定形式。

反例 → 正例 Java
// 反例：双重否定，需要转换才能理解
if (!buffer.isNotEmpty()) { ... }

// 正例
if (buffer.isEmpty()) { ... }

G29 函数只做一件事

这是全书最核心的原则之一。一个函数如果既改变了对象的状态，又返回了对象的信息，它就在做两件事。拆开。

G30 隐藏时序耦合

如果函数 A 必须在函数 B 之前调用，就应该在设计上让这个顺序变得明显，而不是依赖调用者记住文档。

反例 → 正例 Java — 链式传递强制顺序
// 反例：三步顺序隐含在注释里，调用者可能搞错
gradient();     // 必须第一步
reticulateSplines();  // 必须第二步
diveForMoog(); // 必须第三步

// 正例：上一步的结果作为下一步的参数，顺序由签名强制
Gradient        g   = gradient();
SplineReticuler sr  = reticulateSplines(g);
diveForMoog(sr);

G31 不要随意（Be Consistent）

如果你在某个方法里用 variable 命名同类事物，在另一个方法里也要用相同的名字，不要随意换成 instance 或 obj。一致性是读者最大的朋友。

G32 封装边界条件

level + 1 和 level - 1 这类计算散布在代码各处，是边界条件泄漏的信号。提取成一个明确命名的变量。

正例 Java
int nextLevel = level + 1;
if (parts.length > nextLevel)
    addChild(parts[nextLevel]);

G33 函数应该下降一个抽象层次

每个函数内部的语句应该处于同一个抽象层次，并且比函数名低一级。混合高低层次是阅读负担的主要来源之一。

G34 将可配置数据放在高层

如果有一个常量或默认值是系统的"可调旋钮"（端口号、默认超时、文件路径），就把它暴露在高层配置，而不是埋进深层实现。

G35 避免传递性导航（迪米特法则）

不要写 a.getB().getC().doSomething()。这意味着你的模块同时知道 A、B、C 的存在，任何一层改变都会影响你。让直接邻居帮你处理。

反例 → 正例 Java
// 反例：Collaborator 知道了 B 和 C 的存在
ctxt.getOptions().getScratchDir().getAbsolutePath();

// 正例：context 直接提供你真正需要的东西
ctxt.getScratchDirectoryOption().getAbsolutePath();
// 或者更好：
ctxt.createScratchFileStream(filename);

测试味道（T）

味道说明
T1 测试不够充分测试应该覆盖一切可能出错的地方。"没出错过"不是不写测试的理由。
T2 使用覆盖率工具覆盖率工具能找到未被测试的代码区域，但 100% 覆盖率不等于测试充分。
T3 不要跳过简单的测试简单测试的文档价值远超它花费的时间。
T4 被忽视的测试揭示二义性如果你不确定某个行为是否是需求，先写一个 @Ignore 的测试标记它。
T5 测试边界条件边界是 bug 的温床，每一个都要有对应测试。
T6 在 bug 附近做彻底测试发现一个 bug，说明这个模块可能还藏着其他 bug，趁热打铁补测试。
T7 测试失败的模式揭示真相测试失败的规律（哪些输入失败，哪些通过）能快速定位问题根因。
T8 覆盖率模式揭示真相查看未被执行的代码行，往往直接找到 bug 所在。
T9 测试要快慢的测试不会被运行。测试越慢，开发者越倾向于跳过它。

实际开发中的落地技巧

如何在 Code Review 中使用这张清单

不需要背诵全部 66 条启发。建议把清单分成三个优先级来使用：

优先级 1（必须修） ├── G5 重复 ├── G22 switch/if 多态替代 ├── G35 传递性导航（迪米特） ├── F1 过多参数 └── T5 边界条件无测试 优先级 2（应该修） ├── G6 抽象层次错误 ├── G14 特性依恋 ├── G27 条件未封装 ├── G24 魔法数字 └── C4 注释掉的代码 优先级 3（有空修） ├── C1 不恰当的注释信息 ├── G12 杂乱（无用导入等） ├── F4 死函数 └── G1 源文件多种语言

把味道变成 Lint 规则

能自动化的就自动化。以下味道可以直接配置工具检测：

  • 魔法数字：Checkstyle MagicNumber 规则
  • 过多参数：Checkstyle ParameterNumber，默认 7，建议调为 4
  • 死代码：SpotBugs / SonarQube 均有对应检测
  • 重复代码：SonarQube CPD（Copy-Paste Detector）
  • 过长方法：Checkstyle MethodLength

重构时的心理模型

Martin 的本意不是让你一次性消除所有味道，而是形成"童子军规则"的习惯：

每次离开一段代码时，让它比你来时稍微整洁一点点。不需要彻底重构，改一个命名、提一个函数、删一行注释——累积的小改动会让代码库持续向好。

容易忽视的细节

  • G11 不一致是最隐蔽的味道：单个实例看起来无害，但在 10 万行代码库里积累后，会让新成员完全无法建立预期。
  • G30 时序耦合在并发代码中尤其危险：不正确的调用顺序可能导致间歇性 bug，难以复现。
  • G25 精确性在金融业务里是红线：double 的浮点误差在高频计算下会累积成显著错误。
  • 测试速度（T9）在 CI 流水线里是隐形成本：10 分钟的测试套件会让开发者减少提交频率，间接降低代码质量。

总结：味道清单的本质

这份清单不是 Martin 发明的规则，而是他从数十年重构实践中归纳出的经验模式。 每一条味道背后都指向一个更深层的设计原则：

味道类别背后的设计原则
重复（G5、G22）DRY — Don't Repeat Yourself
抽象层次（G6、G33）SRP + 单一抽象层次原则（SLAP）
信息过多 / 传递性导航（G8、G35）迪米特法则 / 最小知识原则
多参数 / 特性依恋（F1、G14）高内聚低耦合
基类依赖派生类（G7）开放-封闭原则（OCP）
边界条件（G3、T5）防御性编程 / 契约式设计
面试高频考点：被问到"你知道哪些代码坏味道"时，不要只列名字——要能解释它违反了哪个设计原则，以及如何重构消除它。将味道与原则关联，才是真正掌握了这章的精华。
