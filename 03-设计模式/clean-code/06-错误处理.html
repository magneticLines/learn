06. 错误处理（用异常而非返回码）

异常不是"高级语法糖"，而是把错误处理从主流程中彻底分离的设计决策。

核心主张：错误处理是独立关注点

Bob 大叔在第七章开门见山：错误处理很重要，但如果它遮蔽了逻辑，那就是写错了。 用返回码处理错误的根本问题在于——它把"检测错误"和"处理业务逻辑"两件事强行混在一起， 逼调用方在每次调用后都要检查返回值，否则错误就被静默吞掉了。

异常机制的设计哲学是：让主流程只描述"事情正确发生时的样子"， 错误路径交由异常体系单独承载。这样阅读代码时，你的注意力不会被 一串 if (result == -1) 打断。

一句话总结：返回码是"内联"的错误处理，异常是"分离"的错误处理。 整洁代码追求关注点分离，因此偏好异常。

反例 → 问题 → 正例

反例：靠返回码传递错误

Java — 返回码风格（Bad）
public class DeviceController {

    public void sendShutdown() {
        // 每一步都要手动检查，主逻辑几乎被淹没
        DeviceHandle handle = getHandle(DEV1);
        if (handle != DeviceHandle.INVALID) {
            retrieveDeviceRecord(handle);
            if (record.getStatus() != DEVICE_SUSPENDED) {
                pauseDevice(handle);
                clearDeviceWorkQueue(handle);
                closeDevice(handle);
            } else {
                logger.log("Device suspended. Unable to shut down");
            }
        } else {
            logger.log("Invalid handle for: " + DEV1.toString());
        }
    }
}

问题出在哪里？

  • 嵌套 if-else 把关机逻辑（暂停、清队列、关闭）淹没在错误检查里。
  • 调用者必须"记得"检查返回值，一旦忘记，错误被静默忽略，产生隐藏 Bug。
  • 新增一步操作就要新增一层 if，维护成本随步骤线性增长。

正例：用异常分离主流程与错误路径

Java — 异常风格（Good）
public class DeviceController {

    public void sendShutdown() {
        try {
            tryToShutDown();        // 主流程：只描述"正确的事"
        } catch (DeviceShutDownException e) {
            logger.log(e);            // 错误路径：集中在一处
        }
    }

    // 主流程方法：一眼看清业务意图，没有任何 if-else 噪声
    private void tryToShutDown() throws DeviceShutDownException {
        DeviceHandle handle = getHandle(DEV1);
        DeviceRecord record  = retrieveDeviceRecord(handle);
        pauseDevice(handle);
        clearDeviceWorkQueue(handle);
        closeDevice(handle);
    }

    private DeviceHandle getHandle(DeviceId id) throws DeviceShutDownException {
        // 若无效，直接抛异常——调用方无需检查返回值
        DeviceHandle handle = deviceRegistry.lookup(id);
        if (handle == DeviceHandle.INVALID)
            throw new DeviceShutDownException("Invalid handle for device: " + id);
        return handle;
    }
}

重构后，tryToShutDown() 方法的每一行都在推进业务—— 阅读主流程时，你的大脑不需要在"正常路径"和"错误路径"之间反复切换。 错误路径被收拢在 catch 块，互不干扰。

先写 try-catch，再填主体逻辑

这是 Clean Code 提出的一个容易被忽视的实战技巧： 当你知道某段代码会抛异常时，先把 try-catch-finally 骨架写出来， 再往里填内容。

原因在于：try 块相当于一个"事务边界"——进入它时系统处于某种状态， 退出时（无论正常还是异常）调用者能对系统状态有确定性预期。 先写骨架，有助于你从"调用者视角"设计接口契约，而不是写完逻辑后再补充错误处理。

Java — TDD 风格：先有 try-catch 骨架
@Test
public void retrieveSectionShouldThrowOnInvalidFileName() {
    sectionStore.retrieveSection("invalid - file"); // 期望抛异常
}

// 先让测试通过（抛异常），再填充正确逻辑
public List<RecordedGrip> retrieveSection(String sectionName) {
    try {
        FileInputStream stream = new FileInputStream(sectionName);
        // ... 后续业务逻辑在这里展开
        stream.close();
    } catch (FileNotFoundException e) {
        throw new StorageException("retrieval error", e);
    }
    return new ArrayList<>();
}

使用非受检异常（Unchecked Exception）

Bob 大叔明确主张：在大多数业务代码中，偏好 RuntimeException（非受检）而非 Checked Exception。

Checked Exception 的问题在于它违反了"开闭原则"： 底层方法新增一个受检异常，调用链上每一层方法签名都要跟着修改（要么 throws，要么 catch）， 造成大范围的级联改动——这是一种隐性的强耦合。

Service → Repository → DB Driver
    ↑ 若 DB Driver 新增 checked exception
     Repository 必须 throws / catch
     Service 必须 throws / catch
     Controller 必须 throws / catch
     ……每一层都被迫感知底层实现细节
维度 Checked Exception Unchecked Exception
编译器强制处理 是 否
修改底层时影响范围 调用链全部层 仅实际 catch 的层
适合场景 IO/网络等可预期且调用者必须处理 业务违规、编程错误等大多数情况
Java 标准库例子 IOException, SQLException IllegalArgumentException, NullPointerException
注意：这不是说 Checked Exception 一无是处。 在库/框架的最底层、或者"调用者必须感知并恢复"的场景（如文件不存在后让用户重新选路径）， Checked Exception 仍是合理选择。重点是：不要因为图方便而把底层受检异常透传给上层。

异常要携带足够的上下文信息

抛出异常时，要让接收者能定位问题，而不只是知道"出错了"。 包含：操作名称、失败对象的标识符、以及原始异常（cause）。

Java — 携带上下文的异常
// Bad：没有上下文，日志里只有一个类名
throw new RuntimeException();

// Bad：有消息，但丢失了原始 cause，排查链路被截断
throw new OrderException("order process failed");

// Good：操作 + 标识符 + 原始异常全部保留
throw new OrderException(
    "Failed to create order for userId=" + userId
        + ", productId=" + productId,
    cause    // 透传原始异常，保留完整 stack trace
);

不要返回 null，也不要传入 null

不要返回 null

返回 null 是比返回错误码更隐蔽的陷阱——调用者如果忘记 null 检查， 会在一个完全不相关的地方触发 NullPointerException，让 Bug 难以定位。

Java — 不返回 null 的替代方案
// Bad：返回 null，调用者必须记得检查
public Employee findEmployee(String name) {
    if (!db.contains(name)) return null;  // 定时炸弹
    return db.get(name);
}

// Good 1：返回 Optional，强制调用者显式处理"无值"情况
public Optional<Employee> findEmployee(String name) {
    return Optional.ofNullable(db.get(name));
}

// Good 2：返回空集合而非 null（集合场景）
public List<Employee> getEmployees() {
    if (db.isEmpty())
        return Collections.emptyList();  // 永远可迭代，不会 NPE
    return db.getAll();
}

不要传入 null

把 null 当参数传入，同样危险——你在把"防御责任"推给被调用方。 大多数方法根本不知道如何合理处理 null 参数。 最好的策略是：在方法入口用断言或 Objects.requireNonNull() 快速失败， 让错误在最靠近根因的地方暴露。

Java — 入口防御
public Metric xProjection(Point p1, Point p2) {
    // 快速失败：比在内部某处 NPE 更容易调试
    Objects.requireNonNull(p1, "p1 must not be null");
    Objects.requireNonNull(p2, "p2 must not be null");
    return new Metric(p2.x - p1.x);
}

落地技巧与常见陷阱

按调用者需求定义异常类

异常类型的划分粒度取决于调用者如何区分处理这些错误， 而不是按底层技术类型划分。如果调用者对所有数据库错误的处理方式都一样（记录日志 + 重试）， 那就用一个 DatabaseException 包装，而不是分别暴露 MySQLException、ConnectionTimeoutException……

Java — 用包装类统一第三方异常
public class LocalPort {
    private ACMEPort innerPort;

    public void open() {
        try {
            innerPort.open();
        } catch (DeviceResponseException e) {
            throw new PortDeviceFailure(e);  // 统一成业务异常
        } catch (ATM1212UnlockedException e) {
            throw new PortDeviceFailure(e);
        } catch (GMXError e) {
            throw new PortDeviceFailure(e);
        }
    }
}

// 调用方现在只需关心一种异常——简洁、可测试
try {
    port.open();
} catch (PortDeviceFailure e) {
    reportError(e);
    logger.log(e.getMessage());
}

特例模式（Special Case Pattern）

有时候"异常情况"其实有默认行为可以内化，不需要调用者处理。 这时可以创建一个"特例对象"——它实现相同接口，但以默认行为应对边界情况。 调用者完全不感知空值或错误的存在。

Java — Special Case Pattern
// Bad：调用方要处理 null
try {
    MealExpenses expenses = expenseRepo.getMealsFor(employee);
    total += expenses.getTotal();
} catch (MealExpensesNotFound e) {
    total += getMealPerDiem();  // 没有餐费记录时使用日补贴
}

// Good：让 getMealsFor 返回特例对象，调用方不再需要 catch
public class PerDiemMealExpenses implements MealExpenses {
    public int getTotal() {
        return MEAL_PER_DIEM();   // 默认行为内化在对象里
    }
}

// 调用方现在干净到不能再干净
MealExpenses expenses = expenseRepo.getMealsFor(employee);
total += expenses.getTotal();
技巧：Special Case Pattern 和 Null Object Pattern 是同一思想的两种叫法。 Spring 中 Collections.emptyList()、Optional.empty() 都是这个模式的具体体现。

总结：错误处理的整洁准则

错误处理整洁准则
├── 用异常代替返回码 — 主流程与错误路径分离
├── 先写 try-catch 骨架 — 从调用者视角设计契约
├── 偏好 Unchecked Exception — 避免级联签名污染
├── 异常携带上下文 — 操作名 + 标识符 + cause
├── 不返回 null — 用 Optional / 空集合 / 特例对象
├── 不传入 null — 入口快速失败
├── 按调用者需求分类异常 — 用包装类隔离第三方
└── 特例模式 — 把边界行为内化到对象，消除 catch
记住核心原则：错误处理不应当掩盖代码逻辑。 当你读完一个方法，脑子里留下的应该是"这个方法做了什么业务"， 而不是"这个方法处理了哪些错误"。 如果错误处理代码比业务逻辑更显眼，说明两者尚未分离干净。

与其他知识点的联系

  • Clean Code《02. 函数》："用异常代替返回码"是函数章节"消除标志参数、追求单一职责"思想在错误处理场景的延伸——返回码把"正常结果"和"错误信号"塞进同一个返回值里，本质上和标志参数一样，是在用一个值表达两种不完全兼容的语义。
  • Clean Code《07. 边界》："按调用者需求定义异常类，用包装类统一第三方异常"和边界章节"用适配器隔离第三方 API"是同一套思路——两者都在解决"外部系统的实现细节不应该泄漏到我方业务代码"的问题，只是边界章节隔离的是接口形状，本章隔离的是异常类型。
  • 责任链模式（Chain of Responsibility）与装饰器模式：Special Case Pattern（特例模式）与 GoF 的 Null Object Pattern 是同一设计的两种叫法，它们都依赖多态——用一个实现相同接口的"默认行为对象"替代 null/异常分支，这正是"针对接口编程，而非针对实现编程"这一 OO 原则的直接应用。
  • Java 异常体系与 Spring @ExceptionHandler：本文"异常携带上下文""统一包装第三方异常"的实践，在 Spring 生态里进一步发展为全局异常处理器（@ControllerAdvice + @ExceptionHandler）——把"错误处理是独立关注点"这一原则从方法级提升到了架构级，是横切关注点（Cross-Cutting Concern）思想的典型例子。
  • Optional 与函数式编程："不要返回 null"一节提到的 Optional，其设计哲学来自函数式编程中对"缺失值"的显式建模（类似 Haskell 的 Maybe）——它强制调用者在编译期就要处理"无值"分支，而不是把这个责任隐性地留给运行期的 null 检查。
