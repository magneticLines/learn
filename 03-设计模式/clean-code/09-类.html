09. 类（内聚 · SRP）

单一职责让类只有一个改变的理由；高内聚让方法与字段紧密咬合——这两条原则共同决定了类的边界在哪里划。

作者的核心主张

Robert C. Martin 在第 10 章开门见山：类应该短小。但"短小"不是靠行数衡量的，而是靠职责数量。一个类只应当有一个改变它的理由，这就是单一职责原则（SRP，Single Responsibility Principle）。

与此紧密配套的是内聚性（Cohesion）：类中每一个方法都应该操作该类的实例变量。方法使用的实例变量越多，它与类的粘合就越强；如果一个方法只用到一两个字段，就要怀疑——它是不是属于另一个类？

两条原则形成闭环：内聚低 → 说明方法在操作"不属于这里"的数据 → 说明类承担了多个职责 → 应当拆分。

为什么这样设计更好？
每次需求变化只触及一个类，改动范围可预测；类足够小，读者能一次性理解完；测试只需覆盖单一职责，用例边界清晰。大类带来的隐患恰恰相反：牵一发而动全身，测试覆盖难以穷举，Code Review 永远看不懂为什么一个类里同时出现"解析 CSV"和"发送邮件"。

类的标准组织顺序

整洁的类有约定俗成的"零件摆放顺序"，读者扫一眼就能找到想看的部分：

ClassName
├── public static 常量
├── private static 变量
├── private 实例变量
├── ── （尽量不出现 public 实例变量）
├── public 构造器
├── public 方法
│ └── 被调用的 private 辅助方法（紧跟在调用者之后）
└── ── 报纸式阅读：由上至下，细节越来越深

这个顺序遵循"报纸隐喻"——重要的放最前，细节依次向后展开，读者可以随时停下来，不必读完整个文件才能理解大意。

反例 → 问题 → 正例

反例：一个类承担三件事

Java — 反例：SuperManager.java
// 这个类同时负责：①读取 CSV 文件 ②解析数据行 ③发送报告邮件
// 三个截然不同的"变化理由"挤在一起
public class SuperManager {

    private String filePath;
    private String smtpHost;
    private String recipient;

    // ① 读文件
    public List<String> readLines() throws IOException {
        return Files.readAllLines(Path.of(filePath));
    }

    // ② 解析行
    public List<Record> parseLines(List<String> lines) {
        return lines.stream()
                     .map(l -> new Record(l.split(",")))
                     .collect(Collectors.toList());
    }

    // ③ 发邮件
    public void sendReport(List<Record> records) {
        // ... 连接 SMTP，拼 HTML，send()
    }
}
问题出在哪里？
1. CSV 格式改变 → 要修改这个类；SMTP 服务器迁移 → 也要修改这个类；Record 结构变化 → 还要修改这个类。三条独立的变化轴全部指向同一个类。
2. 测试时无法单独测"解析逻辑"——必须提供一个真实文件路径，或 mock 掉文件读取逻辑，耦合已经渗透到测试里。
3. 新来的同事看到 SuperManager，完全无法从名字判断它的职责——名字含糊本身就是 SRP 被违反的信号。

正例：按职责拆分成三个类

Java — 正例：三个各司其职的类
// ① 只负责"从哪里读"——I/O 职责
public class CsvReader {
    private final Path path;

    public CsvReader(Path path) { this.path = path; }

    public List<String> readLines() throws IOException {
        return Files.readAllLines(path);
    }
}

// ② 只负责"如何解析"——格式/领域职责
public class RecordParser {
    // 方法只操作入参，不依赖实例状态 → 说明它是无状态的纯函数
    public List<Record> parse(List<String> lines) {
        return lines.stream()
                     .map(l -> new Record(l.split(",")))
                     .collect(Collectors.toList());
    }
}

// ③ 只负责"发给谁"——通知/基础设施职责
public class ReportMailer {
    private final String smtpHost;
    private final String recipient;

    public ReportMailer(String smtpHost, String recipient) {
        this.smtpHost = smtpHost;
        this.recipient = recipient;
    }

    public void send(List<Record> records) {
        // 只操作 smtpHost / recipient，高内聚
    }
}
每个类的每个方法都只操作该类自己的实例变量——这就是高内聚的直观体现。现在可以单独测试解析逻辑，而不需要 mock 任何 I/O；可以替换邮件实现，而不碰 CSV 解析代码。

内聚性：何时该拆分

内聚性可以用一个直观的"方法-变量使用矩阵"来感知：把所有实例变量列成列，所有方法列成行，打勾表示"该方法用到了该变量"。

方法 \ 字段 filePath smtpHost recipient
readLines() ✓ — —
parseLines() — — —
sendReport() — ✓ ✓

矩阵里出现了两个互不相交的"孤岛"，parseLines 甚至不用任何实例变量——这是内聚性极低的警报，说明这些方法本属于不同的类。

Martin 的经验法则：当你发现一个类的某些方法可以被拆出去，并且拆出后的两组方法分别与各自的实例变量保持高内聚，就应该果断拆分。类变多不是问题，只要每个类都只做一件事。

为变化而设计（OCP 的延伸）

SRP 确保了"只有一个理由改变"，而开闭原则（OCP）进一步要求类对扩展开放、对修改关闭。两者在实践中相互强化：

反例：用 if/switch 分支承载变化

Java — 反例：每次新增格式都要改这个类
public class ReportFormatter {
    // 每当产品要求"支持新格式"，就必须修改这个 if-else 链
    public String format(Report report, String type) {
        if ("csv".equals(type))    return toCsv(report);
        if ("json".equals(type))   return toJson(report);
        if ("xml".equals(type))    return toXml(report);
        throw new IllegalArgumentException("unknown type");
    }
}

正例：抽象隔离变化点

Java — 正例：新增格式只需新增实现类，不改已有代码
// 变化点被隔离在接口后面
public interface ReportFormatter {
    String format(Report report);
}

public class CsvFormatter implements ReportFormatter {
    @Override
    public String format(Report r) { /* CSV 逻辑 */ return ""; }
}

public class JsonFormatter implements ReportFormatter {
    @Override
    public String format(Report r) { /* JSON 逻辑 */ return ""; }
}

// 新增 XmlFormatter 时，上面两行代码一个字都不用动

实际开发落地技巧

技巧 1：用"一句话描述"测试 SRP

能否用不含"和"或"或"的一句话描述这个类的职责？如果描述里出现了"并且"、"还可以"，几乎可以确定这个类需要拆分。

  • 好的描述：UserAuthenticator — 负责验证用户凭据
  • 坏的描述：UserManager — 负责验证用户，并且发送欢迎邮件，还可以更新个人资料

技巧 2：关注"改变理由"，而非"代码行数"

不要用行数判断类是否需要拆分。一个 200 行的类如果只做一件事，它是合理的；一个 50 行的类如果职责混乱，它也是问题。Martin 反复强调：改变的理由才是衡量单位。

技巧 3：私有方法过多是拆分信号

当一个类有大量 private 辅助方法，且这些方法只被某几个 public 方法调用时，说明已经形成了"隐形子系统"——可以把这组 public + 配套 private 提取为独立类，原来的 private 变成新类的 public。

技巧 4：构造器参数数量是警示灯

构造器需要 5 个以上参数？大概率是多个职责被注入进来了。用Builder 模式掩盖问题是止痛药，真正的药方是拆分职责、减少依赖数量。

技巧 5：测试难易度是最真实的反馈

如果为一个类写单元测试需要 mock 5 个依赖，或者测一个方法之前需要搭一套复杂的前置状态，这是 SRP 被违反的最直接信号。测试痛苦 = 设计问题，不要用 mock 框架的强大来麻痹自己。

容易忽略的细节

陷阱 1：把"工具类"当垃圾桶
Utils、Helper、Common 这类名字是 SRP 被放弃的重灾区。每加一个"方便"的静态方法，就多一个职责。如果方法数量超过 5 个且没有统一主题，这个类已经是一个杂物间了。
陷阱 2：Service 层的"上帝服务"
在 Spring 项目里，OrderService 动辄 2000 行，注入了 15 个 Repository 和 Service——这是"SRP 失效"的最常见企业级形态。每个 Jira 需求改动都指向同一个文件，是典型的"改变理由过多"。拆分标准：一个 Service 只协调一个聚合根。
细节：拆分不等于分散，要保持关联代码的局部性
按 SRP 拆分后，相关的类应当放在同一个包里（如 report.formatter），而不是把 CsvFormatter 扔到 util、把 JsonFormatter 扔到 service。包是"更大粒度的内聚单元"，同一变化轴的类应聚在同一包下。

面试高频考点

考点答题要点
SRP 的"职责"如何定义？ 职责 = 改变的理由，不是功能点的数量。一个类只能有一个让它被修改的业务原因。
内聚性和 SRP 的关系？ 内聚性是度量工具：方法与实例变量的使用重叠越高，内聚越高；内聚低通常意味着 SRP 被违反。
SRP 和 OCP 如何配合？ SRP 保证每次只有一个理由修改；OCP 保证用扩展代替修改。合用后的效果是：新需求 = 新类，旧代码不动。
大类是否总是错的？ 不是。衡量标准是职责数，不是行数。若一个大类只有一个改变理由，它是合法的；但实践中大类几乎总是职责混杂的产物。
