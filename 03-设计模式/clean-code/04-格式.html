《代码整洁之道》 · 表达与排版

04. 格式（Formatting）

代码是写给人读的，格式就是写给读者的第一印象。统一、整洁的排版能让阅读流畅无阻；混乱的排版会在理解代码之前先消耗读者的认知带宽。

核心主张：格式是职业素养，不是个人风格

Robert C. Martin 在书中开篇就点明：代码格式关乎沟通，沟通是专业开发者的头等大事。 功能实现是暂时的，今天的代码明天就可能被推翻重写；但阅读代码的体验会贯穿整个项目生命周期。

他强调两点：

  • 格式不是装饰，是可读性的基础设施——读者的眼睛会先扫描结构，然后才深入阅读逻辑。
  • 格式规则必须团队统一——个人风格在团队代码库里等于噪音，应通过工具（Checkstyle、Prettier、EditorConfig）强制执行，而非靠"约定"靠自觉。
作者的比喻：好的代码格式就像报纸的排版。大标题让你快速判断值不值得读；段落让你知道在哪里暂停；专栏宽度控制让眼睛不疲劳。你不会把一整版内容堆成一段无缝的文字。

垂直格式：文件就是一篇文章

1. 文件大小：越小越好

Martin 统计了 FitNesse、JUnit 等项目的源文件，发现大多数文件在 200 行以内，极少超过 500 行。 这并非强制规定，而是一个信号：如果一个类已经长达 1000 行，几乎可以确定它承担了过多职责（违反 SRP）。

2. 空行：用来隔离"概念"

空行是代码段落的分隔符。每个空行都在说："上面是一个概念，下面是另一个概念。" 缺少空行会让读者迷失在连续的行中，找不到段落边界。

反例 — 毫无空行，无法感知结构Java
public class ReportGenerator {
    private List<String> lines;
    private String title;
    public ReportGenerator(String title) {
        this.title = title;
        this.lines = new ArrayList<>();
    }
    public void addLine(String line) {
        lines.add(line);
    }
    public String render() {
        StringBuilder sb = new StringBuilder();
        sb.append(title).append("\n");
        for (String line : lines) {
            sb.append(line).append("\n");
        }
        return sb.toString();
    }
}
问题：字段声明、构造函数、方法全部粘连在一起。眼睛扫过去感受到的是"一大块代码"，而不是"三个独立的部分"。识别每个方法的边界需要额外认知负担。
正例 — 空行隔离概念Java
public class ReportGenerator {

    // 字段声明：一组相关的状态，作为一个概念块
    private List<String> lines;
    private String title;

    // 构造函数：初始化职责独立成段
    public ReportGenerator(String title) {
        this.title = title;
        this.lines = new ArrayList<>();
    }

    // 公开方法：各自独立，职责清晰
    public void addLine(String line) {
        lines.add(line);
    }

    public String render() {
        StringBuilder sb = new StringBuilder();
        sb.append(title).append("\n");
        for (String line : lines) {
            sb.append(line).append("\n");
        }
        return sb.toString();
    }
}

3. 垂直紧密：相关代码靠在一起

空行的反面同样重要：密切相关的行不应该被空行割断。 如果两行代码是同一个逻辑步骤的组成部分，它们就应该紧挨着，中间不能有空行干扰。

反例 — 关联代码被不必要的空行割裂Java
public void processOrder(Order order) {

    String customerId = order.getCustomerId();

    String productId = order.getProductId();

    int quantity = order.getQuantity();

    validateCustomer(customerId);

    validateStock(productId, quantity);

    createInvoice(order);
}
正例 — 相关步骤聚合，不同阶段分隔Java
public void processOrder(Order order) {
    // 提取数据：三行是一个整体，紧密聚合
    String customerId = order.getCustomerId();
    String productId  = order.getProductId();
    int    quantity   = order.getQuantity();

    // 校验阶段：与上面提取阶段用空行隔开
    validateCustomer(customerId);
    validateStock(productId, quantity);

    // 执行阶段：独立的业务步骤
    createInvoice(order);
}

4. 垂直距离：相关概念应在物理上接近

Martin 提出"垂直距离"原则：概念上关联的东西，在文件中也应该靠近彼此。 如果你理解一段代码需要反复上下滚动翻找，说明相关代码在垂直空间上离得太远了。

概念关联度 → 垂直距离
├── 局部变量 → 声明尽量靠近第一次使用处
├── 实例变量 → 集中放在类顶部（Java 惯例）
├── 依赖函数 → 调用者在上，被调用者在下
└── 概念相似的函数 → 放在一起，不要散落文件各处
调用者在上，被调用者在下——这是 Martin 最强调的垂直顺序规则。读者阅读一个文件就像读报纸，从上往下扫描。高层函数在前，低层实现细节在后，读者可以选择何时深入细节，而不是被迫先看细节。

水平格式：行宽与对齐

1. 行宽限制：80～120 列

Martin 推荐每行不超过 80～120 个字符。这不是怀旧情怀，而是有实际理由：

  • 超宽行需要水平滚动，打断阅读节奏
  • 并排对比两个文件（diff）时，超宽行会导致折行，干扰比较
  • 代码审查工具、终端默认宽度通常是 80～120 列
  • 过长的行往往暗示表达式太复杂，应拆分

2. 水平空白：表达运算符优先级和语义分组

空格在水平方向上的作用与空行类似——它在视觉上表达"这两者之间有距离"。

反例 — 空格使用混乱，优先级不清Java
double result=(-b+Math.sqrt(b*b-4*a*c))/(2*a);

if(x==0||y==0){
    doSomething();
}
正例 — 空格体现运算结构，条件清晰Java
// 乘法优先级高，不加空格；加减优先级低，加空格——视觉上模拟数学书写
double result = (-b + Math.sqrt(b*b - 4*a*c)) / (2*a);

if (x == 0 || y == 0) {
    doSomething();
}

3. 水平对齐：Martin 的反直觉观点

很多人喜欢对齐赋值号：

"整齐"的对齐写法 — 看起来好看，实则有害Java
private Socket          socket         = null;
private OutputStream    outputStream   = null;
private int             requestsMade  = 0;
private boolean         hasError      = false;
为什么有害？
1. 眼睛会被对齐的列吸引，结果读了一列变量名，然后又读了一列值，跳过了类型——阅读顺序被打乱。
2. 新增一个名字更长的变量时，所有行都需要重新调整空格，造成无意义的 diff 噪音。
Martin 的结论：水平对齐强调了错误的东西，不对齐反而更自然。
正例 — 不对齐，顺序阅读更流畅Java
private Socket socket = null;
private OutputStream outputStream = null;
private int requestsMade = 0;
private boolean hasError = false;

缩进：层次即结构

缩进是代码中最直观的视觉层次表达。每一层缩进代表一层作用域的嵌套——类包含方法，方法包含块，块包含语句。 打破缩进规则会让读者对代码结构产生误判。

不要折叠单行代码破坏缩进感

反例 — 为了"简洁"把块折叠到一行Java
// 表面上省了两行，实际上读者需要二次解析才能理解结构
if (condition) doThis();
else doThat();

while (running) tick();
正例 — 展开块，层次一目了然Java
if (condition) {
    doThis();
} else {
    doThat();
}

while (running) {
    tick();
}
缩进的另一个作用：深度嵌套（超过 3 层）本身就是一个代码异味信号——它通常意味着方法做了太多事，应当提取函数来降低嵌套深度，而不是靠格式化来掩盖。

团队规则：统一高于个人偏好

Martin 用了一整节来强调：格式规则的最终目标是团队的一致性，而不是某个人"更好的风格"。 一个团队就应该只有一套代码格式，任何成员打开任何文件都应该感觉像是同一个人写的。

决策项 常见争议 Martin 的建议
缩进 Tab vs 空格，2 格 vs 4 格 无所谓哪种，团队选一种并统一执行
花括号位置 同行 { vs 换行 { 同上，统一即可，Java 惯例同行
行宽 80 / 100 / 120 列 明确写入 EditorConfig / Checkstyle，CI 检查
空行数量 方法间 1 行 vs 2 行 统一 1 行，用工具 enforce
工具即规则：格式规则不应靠人工 review 来守护，而应该在 pre-commit hook 或 CI 流水线中用工具自动检查和修复。Java 项目用 Checkstyle + google-java-format；前端用 Prettier + ESLint；Go 有内置 gofmt。工具让规则无需讨论，自动执行。

深度理解：格式背后的认知科学

为什么格式能影响可读性？这和人类的视觉认知有关：

  • 预注意属性（Pre-attentive attributes）：空白、缩进、对齐是大脑在意识参与之前就能感知的视觉属性。读者扫描代码时，结构在"读"之前就已经被感知到了。
  • 格式传递了意图：空行说"这里换了一个话题"，缩进说"这里进入了一个子作用域"，紧密聚合说"这几行是一个原子操作"。这些信息是元信息，不在逻辑里，只在格式里。
  • 一致性降低认知负担：格式统一后，读者建立起模式识别，不再需要"解析"格式本身，可以把全部认知资源放在理解逻辑上。

落地技巧：实际开发中容易忽略的细节

1. 方法内的局部变量声明放在使用处附近

Java 早期风格（受 C 影响）喜欢把所有局部变量集中放在方法顶部。Martin 反对这种做法： 变量应在第一次被使用前的最近位置声明，这样读者理解变量时可以立刻看到它的上下文。

反例 — 所有变量堆在顶部（C 风格）Java
public void process() {
    String name;
    int count;
    List<Order> orders;
    boolean valid;

    name = fetchName();
    // ... 20 行其他代码 ...
    orders = fetchOrders(name);
    // ... 再 10 行 ...
    count = orders.size();
}
正例 — 就近声明Java
public void process() {
    String name = fetchName();

    // ... 20 行其他代码 ...

    List<Order> orders = fetchOrders(name);
    int count = orders.size();
}

2. 不要为"节约行数"压缩格式

"代码行数少 = 代码质量高"是一个常见误区。行数不是成本，认知负担才是。 为了减少行数而压缩空行、折叠块、合并语句，往往让代码难以理解。

3. 格式 review 要机械化，不要占用人工 review 资源

代码审查时，reviewer 如果把时间花在指出缩进错误和多余空格上，就是在浪费高价值资源。 配置好自动格式化工具，让人工 review 聚焦在逻辑、设计、命名上。

4. 实例变量顺序：Java 特定惯例

  • 类的静态常量 (static final) 放最前面
  • 静态变量其次
  • 实例变量按重要程度或字母顺序排列
  • 构造函数紧跟在字段之后
  • 公开方法在前，私有方法在后
Java 类的垂直布局惯例
├── static final 常量
├── static 变量
├── 实例变量（private）
├── 构造函数
├── public 方法（按调用顺序，调用者在上）
└── private 辅助方法（在被调用的公开方法下方）

总结

维度核心规则目的
垂直 — 空行 用空行隔离不同概念块 建立视觉段落，帮助扫描
垂直 — 距离 相关代码靠近；调用者在上，被调用者在下 减少上下翻找，保持阅读流畅
垂直 — 文件大小 尽量控制在 200 行内 文件过长往往是 SRP 违反的信号
水平 — 行宽 80～120 列上限 避免折行，兼容 diff 工具
水平 — 空格 用空格表达运算符优先级和语义分组 视觉辅助理解表达式结构
水平 — 对齐 不做列对齐 避免视觉误导和不必要的 diff
缩进 保持展开，不折叠单行块 层次即结构，缩进传递作用域信息
团队 统一规则，工具 enforce 降低全团队认知负担，消除格式争论
一句话记住核心：格式是写给下一个读代码的人的——那个人很可能是三个月后的你自己。
