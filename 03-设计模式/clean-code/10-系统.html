10. 系统（System）

将构造与使用分离，让系统从干净的初态演进，而不是在运行时到处夹带初始化逻辑。

一、核心主张：城市不是一个人建的

Bob 大叔用城市做开篇类比：一座运转良好的城市不会让同一个人既负责建造水管又负责卖水。 建造（Construction）与使用（Use）是两件完全不同的事， 好的系统应当把它们清晰地分离在不同的关注点里。

在代码世界里，"构造"就是创建并装配对象，"使用"就是业务逻辑运行期间调用这些对象。 两者一旦混在一起，系统就会在任意位置埋下初始化副作用，单元测试变得举步维艰， 模块之间产生隐式耦合，架构演进也随之受阻。

一句话原则：将系统的构造与使用分离。 构造阶段完成后，系统应当运行在一个完全可工作、经过验证的依赖图上， 业务代码不应感知任何对象是"怎么来的"。

二、反例：在业务代码里做初始化

最常见的违规模式是在方法内部用 if (field == null) 做懒加载，或者直接 new 依赖对象。

Java — 反例（构造与使用耦合）
public class OrderService {

    private PaymentGateway gateway;

    public Receipt checkout(Cart cart) {
        // 问题1：业务方法里夹带了初始化逻辑
        if (gateway == null) {
            // 问题2：直接 new 具体实现，无法替换为测试桩
            gateway = new StripeGateway(
                Config.get("stripe.key"),
                new HttpClient()   // 问题3：传递性 new，隐式依赖链
            );
        }
        return gateway.charge(cart);
    }
}

这段代码的问题

  • 无法单元测试：checkout 内部硬编码了 StripeGateway，测试时无法注入 Mock，真实网络请求会被触发。
  • 违反单一职责：OrderService 既要知道"如何创建支付网关"，又要负责"执行结账流程"，两个职责挤在一个类里。
  • 全局状态污染：懒初始化的 gateway 字段在多线程下存在竞态条件，且状态在测试用例之间泄漏。
  • 修改传播：每当 StripeGateway 构造函数签名变化，所有类似的初始化点都需要跟着改。

三、分离构造与使用的三种手段

手段 1：将初始化推到 main

最简单的做法：把所有对象的创建集中在 main（或启动入口）， 然后把已构建好的对象传给应用程序。应用程序本身对构造过程一无所知。

main()
├── 创建 HttpClient
├── 创建 StripeGateway(httpClient, apiKey)
├── 创建 OrderService(gateway)
└── 启动应用，OrderService 只管"用"，不管"造"
对应 Spring Boot：@SpringBootApplication 启动时 IoC 容器承担了"main 里的构造工作"， Bean 方法完成装配，业务代码通过 @Autowired 接收成品对象——这正是该手段的框架级实现。

手段 2：工厂（Factory）

有时候对象需要在运行时按需创建（如每次请求创建一个新的 LineItem）， 这时使用抽象工厂：应用代码依赖工厂接口，具体工厂实现留在启动层。 应用代码仍然控制"何时创建"，但不知道"如何创建"。

Java — 工厂接口隔离构造细节
// 应用层只依赖接口，不依赖具体实现
public interface LineItemFactory {
    LineItem create(Product product, int qty);
}

public class OrderService {
    private final LineItemFactory factory;

    // 依赖注入：运行时传入真实工厂，测试时传入 Mock 工厂
    public OrderService(LineItemFactory factory) {
        this.factory = factory;
    }

    public Order addItem(Order order, Product p, int qty) {
        // OrderService 控制"何时"，工厂控制"如何"
        LineItem item = factory.create(p, qty);
        order.add(item);
        return order;
    }
}

手段 3：依赖注入（DI）

依赖注入是"将初始化推到 main"的系统化实现。 对象不自己创建依赖，而是在构造时由外部传入。 DI 容器（如 Spring、Guice）在启动阶段扫描依赖图、完成装配， 之后整个运行期都在使用已构建好的对象图。

Java — 正例（构造器注入 + DI 容器）
@Service
public class OrderService {

    private final PaymentGateway gateway;
    private final InventoryPort inventory;

    // 构造器注入：依赖在对象创建时一次性确定，之后不可变
    public OrderService(PaymentGateway gateway,
                          InventoryPort inventory) {
        this.gateway   = gateway;
        this.inventory = inventory;
    }

    // 业务方法：只关注业务逻辑，完全不感知对象来源
    public Receipt checkout(Cart cart) {
        inventory.reserve(cart);
        return gateway.charge(cart.total());
    }
}

// 单元测试：只需传入 Mock，零框架、零网络
class OrderServiceTest {
    @Test
    void checkout_chargesCorrectAmount() {
        PaymentGateway mockGw  = mock(PaymentGateway.class);
        InventoryPort  mockInv = mock(InventoryPort.class);
        OrderService   svc    = new OrderService(mockGw, mockInv);

        svc.checkout(cartWith(100));
        verify(mockGw).charge(new BigDecimal("100"));
    }
}

四、横切关注点与 AOP：解决"无处不在的代码"

即使做好了 DI，系统中仍然会有一类代码天然地散布在每个模块里： 日志、事务、安全检查、性能监控。 这些被称为横切关注点（Cross-Cutting Concerns）。 如果用普通方法调用处理，它们会让每个业务方法都变成这样：

Java — 横切关注点污染业务代码（反例）
public Receipt checkout(Cart cart) {
    // 与业务无关，却在每个方法里重复
    log.info("checkout start, user={}", ctx.userId());
    txManager.begin();
    checkPermission("ORDER_WRITE");
    try {
        Receipt r = gateway.charge(cart.total());  // 真正的业务：只有这一行
        txManager.commit();
        return r;
    } catch (Exception e) {
        txManager.rollback();
        throw e;
    }
}

解决方案是面向切面编程（AOP）：将横切逻辑从业务代码中抽离， 在"切面"里集中声明"在哪些方法的前后做什么"，框架在运行时（或编译时）织入。 Spring 的 @Transactional、@PreAuthorize、@Async 都是这个思想的体现。

Java — AOP 正例（横切关注点声明式分离）
@Service
public class OrderService {

    // 注解声明"需要事务"，框架负责在外层织入 begin/commit/rollback
    @Transactional
    @PreAuthorize("hasAuthority('ORDER_WRITE')")
    public Receipt checkout(Cart cart) {
        // 方法体只剩纯粹的业务逻辑，可读性与可测试性都大幅提升
        return gateway.charge(cart.total());
    }
}

五、渐进式演进：先让系统能跑，再让它跑得好

Bob 大叔强调：软件系统不像物理建筑，不需要提前把所有承重墙全部设计好。 只要关注点分离得干净，系统完全可以从"今天够用"的简单架构开始， 随着需求增长逐步演进到更复杂的结构，而不需要推倒重来。

这里有一个著名的反模式：EJB 1/2 时代的过度复杂。 为了获得事务和安全，开发者被迫实现大量接口、编写繁琐的 XML 配置， 即便是最简单的业务逻辑也被脚手架代码淹没。 这正是"构造与使用未分离"和"横切关注点侵入业务"的双重惩罚。

警惕 BDUF（Big Design Up Front）：过早优化架构和过度预留扩展点， 往往制造出无人能理解的复杂度。先写能工作的干净代码， 在有真实数据支撑时再做架构升级，这才是演进式设计的正确节奏。

演进的三阶段示意：

阶段 1：单体 + 本地调用（需求初期，快速验证）
└── 所有模块在一个进程，接口清晰，构造在 main/Spring

阶段 2：模块化 + 事件总线（团队扩大，解耦发布节奏）
└── 模块间通过接口/事件通信，各自独立部署窗口

阶段 3：微服务（真正的伸缩瓶颈出现后）
└── 拆分边界已在模块化阶段被验证，迁移成本低

六、对比总结

维度 反例（构造与使用混合） 正例（分离 + DI）
单元测试 依赖真实实现，测试困难 注入 Mock，毫秒级测试
替换实现 散布在各处的 new，改动多 只改 DI 配置或工厂，业务代码零修改
横切关注点 日志/事务散落在每个方法 切面集中管理，业务代码纯粹
架构演进 初始化耦合导致牵一发动全身 接口隔离，可渐进升级
可读性 构造细节淹没业务意图 方法体只剩业务语义

七、实际开发落地技巧

1. 优先使用构造器注入而非字段注入

Spring 的 @Autowired 字段注入很方便，但会隐藏依赖关系， 且无法在不启动 Spring 容器的情况下实例化对象。 构造器注入让依赖一目了然，测试时直接 new 即可。

  • 字段注入：@Autowired private Foo foo; — 隐式依赖，测试需启动容器
  • 构造器注入：public MyService(Foo foo) { this.foo = foo; } — 显式依赖，可直接 new

2. 用接口隔离，而不是依赖具体类

业务代码依赖接口（PaymentGateway），而不是具体实现（StripeGateway）。 这样切换支付渠道、或在测试中注入 Mock，都不需要修改业务代码。

3. 将 ApplicationContext 当作黑盒

不要在业务代码里调用 applicationContext.getBean(...)。 这等价于把 DI 容器当成服务定位器（Service Locator）， 又把构造与使用混在了一起，只是换了一种写法。

常见陷阱：@Autowired ApplicationContext ctx; 后在业务方法里 ctx.getBean(XxxService.class)——这正是把构造逻辑拉回业务代码的反模式。 真正需要动态获取 Bean 时，使用工厂接口或策略模式代替。

4. 横切关注点用注解声明，不要手动调用

事务、缓存、安全、限流等横切需求，优先考虑现有注解（@Transactional、 @Cacheable、@RateLimiter）， 实在没有再自定义切面，但切面逻辑永远不应出现在业务方法体内。

5. 测试即文档：让构造过程成为系统最真实的架构说明

干净的 DI 配置（Spring 的 @Configuration 类、Guice Module） 就是系统依赖图的可执行文档。当新人看到装配代码就能理解整个系统的组件关系， 这比任何架构图都更可靠，因为它会随代码一起编译、一起失效。

与其他知识点的联系

  • Clean Code《07. 边界》：依赖注入是"构造与使用分离"的系统化实现，也是边界隔离能够落地的前提——业务代码依赖 PaymentGateway 接口而不是 StripeGateway 具体类，正是边界章节"面向自定义接口编程"原则在架构层面的延伸，两篇讲的是同一件事在不同粒度上的体现。
  • Clean Code《08. 单元测试》：构造器注入让"传入 Mock 对象"成为可能，这是单元测试 F.I.R.S.T 原则里 Fast 和 Independent 得以实现的架构基础。如果对象在方法内部懒加载/直接 new 依赖，测试就必然要触碰真实实现，本文反例中的 OrderService 正是这种"不可测代码"的典型样本。
  • GoF 工厂模式（Factory）与抽象工厂：手段 2 讲的"工厂接口隔离构造细节"直接对应 GoF 的工厂方法模式——应用代码依赖 LineItemFactory 接口而不感知具体实现，这也是"多用组合，少用继承""针对接口编程"两条 OO 原则的具体应用场景。
  • Spring IoC 容器与 AOP：本文"将初始化推到 main"手段在 Spring 中就是 @SpringBootApplication 启动时的 Bean 装配过程；而横切关注点一节讲的 @Transactional、@PreAuthorize 正是 Spring AOP 基于动态代理（JDK Proxy / CGLIB）实现方法拦截的具体机制，理解代理原理有助于理解为什么"自调用"场景下 @Transactional 会失效。
  • 依赖倒置原则（DIP）与控制反转（IoC）："依赖注入是控制反转的一种实现方式"——传统代码里对象主动创建自己需要的依赖（控制权在对象自己手上），DI 把这个控制权交给外部容器，这正是 SOLID 中依赖倒置原则从"接口抽象"进一步落地到"运行时装配机制"的自然延伸。
