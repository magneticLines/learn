01. 有意义的命名

名字是代码最廉价、最高频的文档——每一个变量、函数、类都在无声地告诉读者"这里发生了什么"。

核心主张：命名即意图表达

Robert C. Martin（Uncle Bob）在第 2 章开篇就断言：程序中到处都是名字，所以命名做好了，代码就已经成功了一半。

他的核心逻辑是：代码被写一次，却被读无数次。读者每次碰到一个模糊的名字都要停下来思考，这种"上下文切换"的成本在大型系统里会累积成真实的生产力损耗。好的名字不需要注释辅助——它本身就是文档。

一句话原则：名字应该完整回答三个问题——它为什么存在？它做什么？它怎么用？如果需要额外注释才能回答这三问，这个名字就不够好。

七条核心规则（精华提炼）

1. 名副其实（Intention-Revealing Names）

变量名应直接表达其用途，而不是迫使读者靠上下文猜。

反例 — 读者无从得知 d 代表什么Java
int d;  // 经过的时间（天）
int elapsed;
int daysSinceModification;
正例 — 不需要注释，名字本身就是文档Java
int daysSinceLastModification;
int fileAgeinDays;

更典型的反例是在集合操作里使用无意义的列表名：

反例 — theList 藏住了业务语义Java
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<>();
    for (int[] x : theList)
        if (x[0] == 4) list1.add(x);
    return list1;
}
正例 — 命名揭示业务含义（扫雷游戏棋盘）Java
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged()) flaggedCells.add(cell);
    return flaggedCells;
}
// 仅靠命名重构，代码意图立刻清晰——一行改动都没有

2. 避免误导（Avoid Disinformation）

误导比模糊更危险。常见陷阱：

  • 用 accountList 命名一个 Map 类型——读者会以为它是 List，调错 API
  • 用仅有一两个字母差异的名字（XYZControllerForEfficientHandlingOfStrings vs XYZControllerForEfficientStorageOfStrings）——代码补全很容易选错
  • 用小写字母 l 和大写字母 O 作变量名——在某些字体下与 1 和 0 无法区分
陷阱：不要用类型名作后缀（如 accountList）除非它真的是 List。更好的做法是直接用 accounts，与具体类型解耦。

3. 做有意义的区分（Make Meaningful Distinctions）

如果两个东西不同，名字就必须能体现这种不同。纯粹为了让编译器不报错而加上数字序号或噪音词，是最差的做法。

反例 — 数字序号和噪音词毫无信息量Java
void copyChars(char[] a1, char[] a2) {
    for (int i = 0; i < a1.length; i++)
        a2[i] = a1[i];
}

// 噪音词：ProductInfo 和 ProductData 有什么区别？
Product theProduct;
Product aProduct;
ProductInfo productInfo;
ProductData productData;
正例 — 名字揭示角色差异Java
void copyChars(char[] source, char[] destination) {
    for (int i = 0; i < source.length; i++)
        destination[i] = source[i];
}

4. 使用可读出来的名字（Use Pronounceable Names）

人类大脑天生是为语言设计的。可以念出来的名字更容易被记住、讨论和搜索。

反例 vs 正例Java
// 反例：没人能优雅地读出 genymdhms
private Date genymdhms;   // generation year, month, day, hour, minute, second
private Date modymdhms;

// 正例：代码审查时可以直接口头交流
private Date generationTimestamp;
private Date modificationTimestamp;

5. 使用可搜索的名字（Use Searchable Names）

单字母变量名和裸数字字面量几乎无法在代码库中搜索。字母 e 会匹配每一个英文单词，数字 7 也会出现在无关字符串里。

反例 — 7 是什么？无法搜索，也无法理解Java
for (int j = 0; j < 34; j++) {
    s += (t[j] * 4) / 5;
}
正例 — 每个名字都是可搜索的具名常量Java
int realDaysPerIdealDay = 4;
final int WORK_DAYS_PER_WEEK = 5;
final int NUMBER_OF_TASKS = 34;

int sum = 0;
for (int j = 0; j < NUMBER_OF_TASKS; j++) {
    int realTaskDays = taskEstimate[j] * realDaysPerIdealDay;
    int realTaskWeeks = realTaskDays / WORK_DAYS_PER_WEEK;
    sum += realTaskWeeks;
}
// 重构前后逻辑完全相同，但后者在代码评审时能立刻看懂业务意图
规则：名字的长度应与其作用域的大小成正比。循环计数器 i 在三行 for 循环里完全合理；但如果它跨越了 50 行逻辑，就必须起描述性名字。

6. 避免编码（Avoid Encodings）

匈牙利命名法（如 strName、iCount）在强类型 IDE 普及前有其历史价值。现代 Java 开发中，IDE 实时显示类型，这类前缀只是噪音。同理，成员变量前缀 m_、接口前缀 I 都属于应当去除的噪音。

旧风格（编码）现代风格（干净）原因
String strNameString name类型由 IDE 和编译器保证
int m_countint count类的作用域已足够区分
interface IShapeFactoryinterface ShapeFactory接口名不应暴露它是接口
class ShapeFactoryImplclass DefaultShapeFactoryImpl 无含义，Default/Cached 更清晰

7. 类、方法和其他命名约定

不同类型的标识符有不同的命名心智模型：

类名：名词或名词短语。Customer、WikiPage、Account、AddressParser
  ✗ 避免：Manager、Processor、Data、Info（意义模糊的万能词）

方法名：动词或动词短语。postPayment、deletePage、save
  访问器：get/set/is 前缀（符合 JavaBean 规范）
  构造器重载：用静态工厂方法代替，名字描述参数含义

布尔型：is/has/can/should 前缀。isEmpty、hasChildren、canWrite

常量：全大写 + 下划线。MAX_RETRY_COUNT、DEFAULT_TIMEOUT_MS
静态工厂方法 vs 构造器重载Java
// 反例：两个构造器，调用方不知道参数语义
Complex fulcrumPoint = new Complex(23.0);

// 正例：静态工厂方法名字即文档
Complex fulcrumPoint = Complex.fromRealNumber(23.0);

一个概念对应一个词（One Word per Concept）

在一个代码库里，同一个抽象概念只用一个词表达。如果 fetch、retrieve、get 在不同类里意思相同，读者就会不断猜测它们是否有细微差别。

同样，不要用同一个词表达不同概念（双关语）。如果 add 在大多数地方表示"把两个值相加"，那么"把元素插入集合"就不该也叫 add，应该用 insert 或 append。

常见双关陷阱：add / append / insert 三者在业务语义上有本质区别（创建新值 vs 追加到序列末尾 vs 在指定位置插入），混用会让调用方猜测副作用。

使用解决方案域 & 问题域名称

代码由程序员读。解决方案域（计算机科学术语）可以直接用：AccountVisitor（Visitor 模式）、JobQueue（队列数据结构）——有经验的读者一眼就懂。

当没有合适的技术术语时，用问题域（业务领域）名称：如果代码与会计系统有关，GrossMargin（毛利率）比 profit1 更清晰，哪怕读者需要去请教领域专家。

添加有意义的语境

孤立的变量名有时需要语境才能被正确理解。最好的语境是一个命名良好的类；其次是在变量名上加前缀；注释是最后手段。

反例 — 单独看 state 不知道是哪个 stateJava
private void printGuessStatistics(char candidate, int count) {
    String number;
    String verb;
    String pluralModifier;
    if (count == 0) {
        number = "no"; verb = "are"; pluralModifier = "s";
    } else if (count == 1) {
        number = "1"; verb = "is"; pluralModifier = "";
    } else {
        number = Integer.toString(count);
        verb = "are"; pluralModifier = "s";
    }
    String guessMessage = String.format(
        "There %s %s %s%s", verb, number, candidate, pluralModifier);
    print(guessMessage);
}
正例 — 提取为类，语境通过类名建立Java
public class GuessStatisticsMessage {
    private String number;
    private String verb;
    private String pluralModifier;

    public String make(char candidate, int count) {
        createPluralDependentMessageParts(count);
        return String.format("There %s %s %s%s",
            verb, number, candidate, pluralModifier);
    }

    private void createPluralDependentMessageParts(int count) {
        if (count == 0)      thereAreNoLetters();
        else if (count == 1) thereIsOneLetter();
        else                  thereAreManyLetters(count);
    }
    // 成员变量现在属于 GuessStatisticsMessage 类，语境自然建立
}

实际开发落地技巧

命名检查清单（每次 Code Review 自查）

  • 能否在不看实现、只看名字的情况下，猜出这个方法/变量的用途？
  • 名字里有没有 Manager、Handler、Processor、Data、Info 这些模糊词？能替换成更精确的吗？
  • 布尔变量/方法是否以 is/has/can 开头，读起来像自然语言断言？
  • 常量是否都有具名的 static final，没有裸魔法数字？
  • 同一概念在整个项目里是否只用了一个词？（搜索一下同义词）

IDE 辅助重命名：零成本改善

IntelliJ IDEA / VS Code 的 Shift+F6（Rename）支持跨文件安全重命名。"命名不好就改"的心理门槛几乎为零——不要让坏名字在代码库里"沉积"。

经验法则：如果你在写注释解释一个变量"是什么"，停下来，把那个解释直接变成变量名。注释可以说"为什么"，但"是什么"应该让名字自己说清楚。

容易被忽略的细节

  • 时态一致性：获取数据用 get，计算/派生用 calculate 或 compute，查询（有副作用判断）用 find，不要混用
  • 复数与单数：集合类型用复数（users），单个对象用单数（user）；一旦混用，循环变量里的 user in users 结构就会消失
  • 时间单位：涉及时间的变量必须在名字里标明单位，如 timeoutMillis、delaySeconds，避免单位混用导致的 Bug（NASA 曾因此损失一艘探测器）
  • 不要为了简短而截断：genAcntNum 并不比 generateAccountNumber 更好——现代代码补全让长名字几乎没有额外输入成本

总结：命名是设计的一部分

好的命名不是文字游戏，它反映了你对问题的理解深度。当你难以为某个类或方法命名时，往往意味着这个类/方法本身的职责不够清晰——命名困难是设计问题的早期预警信号。

Uncle Bob 说："改变名字的恐惧反映了其他恐惧——对于不能重构的恐惧。"持续改善命名，就是持续改善设计。

与后续章节的联系：好的命名与「函数（单一职责）」高度互补——一个职责单一的函数，往往最容易被精准命名；反之，难以命名的函数几乎总是承担了多重职责。
