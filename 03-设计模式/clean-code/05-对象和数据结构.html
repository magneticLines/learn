05. 对象和数据结构

得墨忒耳律（Law of Demeter）、数据抽象与火车失事——搞清楚你在写"对象"还是"结构体"。

核心主张：隐藏数据，暴露行为

Uncle Bob 在本章提出两个相互关联的主张：

  • 数据抽象：不要只把内部变量套上 getter/setter 然后暴露出去，那只是把私有字段做了一层无意义的包装。真正的抽象应该暴露行为，让调用方根本不需要知道你的内部数据结构是什么。
  • 得墨忒耳律（LoD）：一个模块只应该与它的直接朋友交谈，不应该通过一个朋友去认识另一个朋友的朋友。换句话说，不要链式调用穿透多层对象。

这两个主张背后是同一个设计哲学：细节应该对外隐藏，依赖应该只建立在稳定的接口上，而不是脆弱的内部结构上。

得墨忒耳（Demeter）是什么？
得墨忒耳律来自 1987 年 Ian Holland 在东北大学的 Demeter 项目，原始表述是： "只和你的直接朋友说话（Only talk to your immediate friends）。" 也被称为最少知识原则（Principle of Least Knowledge）。

数据抽象：不只是包了一层 getter

反例 — 透明的数据结构（假抽象）

Java — 反例
public class Point {
    public double x;
    public double y;
}

// 加了 getter/setter 本质没变——依然暴露了"我用直角坐标"这个细节
public class Point {
    private double x;
    private double y;
    public double getX() { return x; }
    public double getY() { return y; }
    public void setX(double x) { this.x = x; }
    public void setY(double y) { this.y = y; }
}

问题在哪里？ 调用方代码写死了 getX() / getY()，如果将来把坐标系改成极坐标（r, θ），所有调用方都要跟着改。封装形同虚设。

正例 — 真正的行为抽象

Java — 正例
// 接口只说"我能提供坐标"，完全不泄露内部是直角坐标还是极坐标
public interface Point {
    double getX();
    double getY();
    void setCartesian(double x, double y);  // 整体设置，有业务语义
    double getR();
    double getTheta();
    void setPolar(double r, double theta);
}

// 再比如 Vehicle：不暴露"油箱容量"和"当前油量"两个字段，
// 而是暴露"还剩多少燃料百分比"这个业务语义
public interface Vehicle {
    double getPercentFuelRemaining();
}
判断标准：如果一个类的所有 getter 都与某个具体字段一一对应，那这个"封装"只是语法形式，没有设计价值。真正的抽象应该让调用方用业务语言而不是存储语言和你交流。

对象 vs 数据结构：两种截然不同的设计

Uncle Bob 特别强调，对象和数据结构是两种相反的设计取向，不要混为一谈：

维度 对象（Object） 数据结构（Data Structure）
数据可见性 隐藏，通过行为暴露 公开，直接访问字段
行为 有方法，封装业务逻辑 几乎没有方法，只承载数据
扩展新类型 容易（新增子类即可） 困难（所有处理函数都要改）
扩展新操作 困难（所有子类都要改） 容易（新增一个函数即可）
典型应用 领域模型、策略模式 DTO、数据库记录、JSON 载体
常见陷阱：用面向对象的语法写数据结构（所有字段 private + 全套 getter/setter），然后又在类里堆业务逻辑——这是两者最糟糕的混合，既不清晰又难以扩展。要么做干净的数据结构，要么做真正的对象。

过程式 vs 面向对象的扩展性对比

Java — 过程式（数据结构 + 函数）
// 新增一种形状：只需加一个 class，Geometry 不用改
// 新增一个操作（如 perimeter）：Geometry 要改，所有 Shape 不用改
public class Square { public Point topLeft; public double side; }
public class Circle  { public Point center;  public double radius; }

public class Geometry {
    public double area(Object shape) throws NoSuchShapeException {
        if (shape instanceof Square s) return s.side * s.side;
        if (shape instanceof Circle c) return Math.PI * c.radius * c.radius;
        throw new NoSuchShapeException();
    }
}
Java — 面向对象（多态）
// 新增一种形状：只需新建子类，无需改任何已有代码（开闭原则）
// 新增一个操作（如 perimeter）：所有子类都要实现
public interface Shape {
    double area();
}

public class Square implements Shape {
    private final double side;
    @Override public double area() { return side * side; }
}

public class Circle implements Shape {
    private final double radius;
    @Override public double area() { return Math.PI * radius * radius; }
}

得墨忒耳律（Law of Demeter）

LoD 的正式规则：一个方法 f 属于类 C，它只能调用以下对象的方法：

  • C 自身的方法
  • f 的参数对象的方法
  • f 内部创建的对象的方法
  • C 的实例变量所指向的对象的方法

不能调用：通过上述对象的方法返回的对象的方法——即不能"穿透"。

反例 — 火车失事（Train Wreck）

Java — 反例（违反 LoD）
// 一行代码穿越了 ctxt → options → scratchDir → absolutePath
// 这段代码知道了太多内部细节
String outputDir = ctxt.getOptions().getScratchDir().getAbsolutePath();

这就是所谓的"火车失事"——一列车厢撞上另一节，再撞上另一节。代码耦合了整条链路上的每个结构，任何一节变动（比如 scratchDir 改成 Path 对象）都会引发连锁修改。

过渡方案 — 拆开变量（但本质未变）

Java — 拆链（仍然违反 LoD）
Options  opts       = ctxt.getOptions();
File     scratchDir = opts.getScratchDir();
String   outputDir  = scratchDir.getAbsolutePath();

// 拆成三行只是外观改变——依然知道 ctxt 有 Options，
// Options 有 ScratchDir，ScratchDir 有 absolutePath。
// 知识量没有减少，LoD 依然被违反。

正例 — 让 ctxt 直接提供行为

Java — 正例（遵循 LoD）
// 问自己：我为什么需要 absolutePath？
// 答：为了在 scratchDir 里创建一个临时文件。
// 那就让 ctxt 直接做这件事，把细节封在它自己内部。
BufferedOutputStream bos = ctxt.createScratchFileStream(classFileName);

// ctxt 内部实现（调用方完全不可见）：
public BufferedOutputStream createScratchFileStream(String classFileName) {
    String dir = options.getScratchDir().getAbsolutePath();
    return new BufferedOutputStream(new FileOutputStream(dir + "/" + classFileName));
}
关键思路：发现火车失事的时候，先问"我为什么需要这条链末端的东西？"——把这个意图封装成一个方法，让离你最近的那个对象来提供。

数据传输对象（DTO）是合法的数据结构

LoD 和数据抽象原则都针对对象，不针对纯数据结构。DTO（Data Transfer Object）是合法的使用场景：

  • 在层之间传递数据（Controller → Service → Repository）
  • 与数据库 / 消息队列 / 外部 API 对接
  • 不包含业务逻辑，只承载字段
Java — DTO（合理的数据结构）
// DTO：字段公开或通过 Lombok / record 暴露，没有业务方法，这是合理的
public record AddressDTO(
    String street,
    String city,
    String zipCode
) {}

// Active Record 是 DTO 的变种（带有 save/find 等持久化方法），
// 不要在上面叠加业务逻辑——那会让两种角色混淆。
Active Record 陷阱：有些框架（如 Rails、MyBatis-Plus 的实体类）会在数据结构上加 save()、find() 等方法，这是可以的。但如果你继续在上面加 validateOrder()、calculateDiscount() 等业务方法，就成了"混合怪物"——既是数据结构又是对象，哪种职责都做不纯粹。

实际开发落地技巧

技巧 1：用"Tell, Don't Ask"驱动设计

"不要问对象拿数据然后自己算，直接告诉对象去做某件事。"这是 LoD 的行为版本。

Java — Ask（反例）vs Tell（正例）
// Ask：拿数据自己算折扣 —— 你知道了 Order 的内部结构
if (order.getItems().size() > 10) {
    price = order.getTotalPrice() * 0.9;
}

// Tell：让 Order 自己决定打折逻辑 —— 你只关心结果
price = order.discountedPrice();

技巧 2：识别"特性依恋"代码味道

如果一段方法大量使用另一个类的数据（频繁 getXxx()），说明这段逻辑"依恋"那个类，它本应属于那个类。重构手法：Move Method。

技巧 3：DTO 与领域对象分层放置

src/main/java/com/example/
├── domain/ ← 真正的对象，有行为，隐藏内部状态
│ ├── Order.java
│ └── Customer.java
├── dto/ ← 纯数据结构，字段公开，无业务逻辑
│ ├── OrderDTO.java
│ └── CustomerDTO.java
└── service/
└── OrderService.java ← 协调两者，做映射转换

技巧 4：Stream 流的链式调用是否违反 LoD？

Stream API 的 .filter().map().collect() 是同一个对象（流）的链式变换，返回的始终是流自身的下一状态，不是穿越多个不同对象的边界。这不违反 LoD。 真正的 LoD 违规是穿越不同业务对象的边界，比如 order.getCustomer().getAddress().getCity()。

技巧 5：防御性 DTO 映射，不要让 ORM 实体跨层流动

永远不要把 JPA Entity 直接返回给 Controller，它携带了懒加载代理、事务上下文等内部细节，一旦在 JSON 序列化时触发额外查询，就是典型的"调用方触发了不应知道的内部细节"。始终在 Service 层把 Entity 转成 DTO 后再暴露。

常见误区与面试高频考点

误区 / 考点说明
"加了 getter/setter 就是封装" 封装的本质是隐藏细节，不是加访问方法。Lombok 的 @Data 生成全套 getter/setter 等于把字段全部公开，要谨慎使用。
LoD 与流式 API 冲突 Builder 模式、Stream API 的链式调用是在同一对象上操作，不违反 LoD。
"对象越有行为越好" DTO、Value Object 作为纯数据容器是完全合理的设计，不是坏代码。问题在于角色混淆。
过度拆分导致方法爆炸 遵循 LoD 不代表给每个查询都加一个方法。关键是问"调用方的意图是什么"，按意图抽象，不是每个链节点都转一遍。

一句话总结

对象隐藏数据、暴露行为；数据结构公开数据、没有行为——分清楚你在设计哪种，再用得墨忒耳律约束对象之间的通信深度，代码就会自然地走向低耦合、高内聚。
