03. 注释（好注释 vs 坏注释）

注释不是代码写烂了以后的补救措施——它是无法用代码表达时的最后手段。

作者的核心主张

Robert C. Martin 在《代码整洁之道》第 4 章开门见山：注释是一种失败的补偿。 写注释往往意味着你没能用代码本身把意图说清楚。 每一条注释都是一次"道歉"——你在说"我没能让代码足够清晰，所以我用自然语言再解释一遍"。

这个观点颠覆了很多人的直觉（"注释越多越好"）。作者的逻辑链条非常清晰：

  • 代码会演化，注释往往不会跟着同步更新，于是注释腐烂，变成谎言
  • 过时的注释比没有注释更危险——它误导读者
  • 如果你需要写注释，先问自己：能不能通过改名、抽方法、拆类让代码自我解释？
核心公式：好代码 + 少量必要注释 > 差代码 + 大量注释。 注释的目标不是"让别人看懂差代码"，而是"记录代码无法直接传达的信息"。

坏注释的七宗罪

以下七类注释几乎在每个代码库都能找到，它们是噪音，不是信号。

① 喃喃自语（Mumbling）

写给自己看的注释，读者看完依然一头雾水。

Java — 坏注释mumbling
try {
    loadProperties();
} catch (IOException e) {
    // 加载了默认值   // ← 什么默认值？谁加载的？什么时候加载？
}

这条注释什么信息都没有增加，甚至让人怀疑异常被静默吞掉了。

② 多余的注释（Redundant Comments）

注释只是把代码翻译成中文，代码已经能读懂的东西不需要注释。

Java — 坏注释redundant
// 返回 this 组件的处理器
protected abstract Handler getHandler();

// 如果 closed 为 true，返回 true
public boolean isClosed() {
    return this.closed;
}

这类注释读起来比代码还费时，因为你要先读注释再去核对代码是否一致。

③ 误导性注释（Misleading Comments）

代码改了，注释没改，于是注释变成了谎言——最危险的一类。

Java — 坏注释misleading
// 当 closed 为 true 时立即关闭
public synchronized void close() {
    if (!closed) {
        waitForCloseThread();   // ← 实际上会等待，并非"立即"
        closed = true;
    }
}

④ 规定性注释（Mandated Comments）

规定"每个方法必须写 Javadoc"，结果生成了大量没有意义的模板噪音。

Java — 坏注释mandated
/**
 * @param title       The title of the CD
 * @param author      The author of the CD
 * @param tracks      The number of tracks on the CD
 * @param durationInMinutes The duration of the CD in minutes
 */
public void addCd(String title, String author,
                  int tracks, int durationInMinutes) {
    // ...
}

参数名本身已经说明一切，这些 Javadoc 是零信息密度的填充物。

⑤ 日志式注释（Journal Comments）

在文件顶部记录每次修改历史——这是 Git 的工作，不是注释的工作。

Java — 坏注释journal
// 2019-03-12 张三 修复了空指针 bug
// 2020-07-08 李四 重构了 parseDate 方法
// 2021-11-01 王五 增加了缓存支持
public class DateParser { // ... }

⑥ 注释掉的代码（Commented-Out Code）

被注释的代码是垃圾。其他人不敢删它（万一有用？），于是它永远留在那里腐烂。

Java — 坏注释commented-out code
doSomething();
// doSomethingElse();
// doAnotherThing();
// doSomethingElseAgain();
doMoreStuff();

用 Git 管理历史。不需要的代码，删掉。

⑦ 过多的信息（Too Much Information）

在注释里粘贴 RFC 全文、历史讨论的完整摘录……这些属于文档，不属于代码旁边。

好注释的七种场景

并非所有注释都应该被消灭。以下场景写注释是值得的，甚至是必要的。

① 法律版权声明

公司要求在每个文件头写版权声明——这是合规需求，无法用代码替代。

Java — 好注释license header
// Copyright (C) 2024 Supernova Companies, Inc. All rights reserved.
// Licensed under the Apache License, Version 2.0.

② 提供代码无法直接表达的信息

正则表达式本身极难读，一行注释说明它匹配的模式格式非常有价值。

Java — 好注释informative
// 匹配格式：kk:mm:ss EEE, MMM dd, yyyy
Pattern timeMatcher = Pattern.compile(
    "\\d*:\\d*:\\d* \\w*, \\w* \\d*, \\d*");

③ 解释意图（Explanation of Intent）

代码说明"做了什么"，注释说明"为什么这么做"——这是最有价值的注释类型。

Java — 好注释intent
public int compareTo(Object o) {
    if (o instanceof WikiPagePath) {
        WikiPagePath p = (WikiPagePath) o;
        String compressedName = StringUtil.join(names);
        String compressedArgumentName = StringUtil.join(p.names);
        return compressedName.compareTo(compressedArgumentName);
    }
    // 排在所有其他类型之后，因为 WikiPagePath 在业务中优先级最高
    return -1;
}

④ 阐释晦涩的参数或返回值

当参数是来自第三方库的枚举或常量，无法直接修改命名时，注释可以帮助说清楚语义。

Java — 好注释clarification
assertTrue(a.compareTo(b) == -1);  // a < b
assertTrue(b.compareTo(a) ==  1);  // b > a
assertTrue(a.compareTo(a) ==  0);  // a == a

⑤ 警示后果（Warning of Consequences）

某些操作有非显而易见的副作用，提前警告可以避免协作者踩坑。

Java — 好注释warning
// 不要并发运行！SimpleDateFormat 不是线程安全的。
// 如需并发，请用 ThreadLocal<SimpleDateFormat> 或 DateTimeFormatter。
public static SimpleDateFormat makeStandardHttpDateFormat() {
    SimpleDateFormat df = new SimpleDateFormat("EEE, dd MMM yyyy HH:mm:ss z");
    df.setTimeZone(TimeZone.getTimeZone("GMT"));
    return df;
}

⑥ TODO 注释

用于标记已知的、暂时无法处理的工作项。要有原因，要有明确的计划，不能是无期限的垃圾桶。

Java — 好注释TODO
// TODO: 当 JDK 1.7 发布后，用 Objects.requireNonNull 替换此处
if (value == null)
    throw new IllegalArgumentException("value must not be null");
TODO 注释要定期清理，否则会变成永远不会被执行的"将来某天"计划。在 IDE 中打开 TODO 视图，把它当 backlog 管理。

⑦ 公开 API 的 Javadoc

如果你在写一个公开库，/** 风格的 Javadoc 是必要的——它是 API 契约的一部分， 会被 IDE 工具链、文档生成器直接使用。但私有方法不需要 Javadoc， 内部代码更不需要。

核心对比：用代码替代注释

大多数"需要注释"的地方，都可以通过重构消灭注释需求。以下是两组最典型的对比。

对比一：晦涩条件 → 语义化命名

Java — 坏（依赖注释解释条件）before
// 判断员工是否有资格享受全额福利
if ((employee.flags & HOURLY_FLAG) &&
    (employee.age > 65)) {
    // ...
}
Java — 好（代码自解释）after
if (employee.isEligibleForFullBenefits()) {
    // ...
}

把条件逻辑封装到方法 isEligibleForFullBenefits() 里， 方法名本身就是文档，注释自然消失。

对比二：魔法数字 → 命名常量

Java — 坏（注释解释魔法数字）before
int realDaysPerIdealDay = 4;
final int WORK_DAYS_PER_WEEK = 5;
int sum = 0;
for (int j = 0; j < NUMBER_OF_TASKS; j++) {
    int realTaskDays = taskEstimate[j] * realDaysPerIdealDay;
    int realTaskWeeks = (realDays / WORK_DAYS_PER_WEEK);  // 每周 5 天工作
    sum += realTaskWeeks;
}
Java — 好（常量命名即文档）after
final int WORK_DAYS_PER_WEEK = 5;    // 5 本身已足够清晰，下面无需注释
final int REAL_DAYS_PER_IDEAL_DAY = 4;

for (int j = 0; j < NUMBER_OF_TASKS; j++) {
    int realTaskDays  = taskEstimate[j] * REAL_DAYS_PER_IDEAL_DAY;
    int realTaskWeeks = realTaskDays / WORK_DAYS_PER_WEEK;
    sum += realTaskWeeks;
}

好注释 vs 坏注释 快速对照表

类型 示例 结论
版权声明 // Copyright (C) 2024 ... 好，合规必要
解释意图 // 用素数生成器绕开性能瓶颈 好，无法从代码读出
警示陷阱 // SimpleDateFormat 非线程安全 好，防止协作者踩坑
TODO // TODO: 升级 JDK 后替换 好，有明确计划
公开 API Javadoc /** Returns the user by id */ 好，契约文档
翻译代码 // 返回 i+1 坏，噪音

注释掉的代码 // doOldThing(); 坏，直接删除
过时注释 代码已变，注释未更新 坏，比没有更危险
喃喃自语 // 加载了默认值（语焉不详） 坏，无增量信息

实际开发中的落地技巧

1. 写注释前先问三个问题

我要写一条注释 ├── 能不能改个更好的名字替代它？ → 能 → 改名，不写注释 ├── 能不能抽一个语义化方法替代它？ → 能 → 抽方法，不写注释 └── 以上都不行，这是真正需要注释的地方 → 写好注释

2. 让注释表达"为什么"，而非"是什么"

  • "是什么"：代码本身的责任，注释重复它是噪音
  • "为什么"：业务决策、算法选择、历史约束——这是注释的价值所在
  • 示例：// 用双重检查锁而非 synchronized 是因为此处 QPS > 50k，单锁成为瓶颈

3. 注释要和代码一起维护

  • 修改代码时，第一件事：检查相关注释是否需要同步更新
  • Code Review 时，审查注释的准确性和有效性，不亚于审查逻辑
  • 如果注释和代码矛盾，相信代码，更新或删除注释

4. 关于注释掉的代码——永远直接删除

  • 源代码控制系统（Git）会记住历史，你不需要
  • 注释掉的代码让人困惑：它还有用吗？为什么不删？它是否影响当前逻辑？
  • 怕删错？打个 tag 或者写个 commit message，然后删

5. TODO 注释的正确使用方式

  • 格式统一：// TODO(shea): 等 Spring Boot 3 稳定后迁移——带上负责人
  • 定期清理：每次迭代开始时过一遍 TODO 列表，完成的删掉，过期的删掉
  • 不要用 TODO 掩盖已知 bug——那应该是一个真正的 issue
一个实用习惯：写完一段代码后，通读所有注释，问自己——如果代码改了，这条注释还准确吗？如果不确定，删掉或重写它。宁可没有注释，也不要有错误的注释。

面试高频考点

  • Q：注释越多说明代码质量越好，对吗？
    错。过多注释往往说明代码命名不清晰、职责不单一，靠注释弥补表达力不足。好代码的注释是稀有但精准的。
  • Q：Javadoc 应该写在哪些地方？
    公开 API（public 的类和方法），尤其是对外发布的库。私有方法、内部实现类无需 Javadoc，改善命名和结构即可。
  • Q：怎么处理历史遗留的大量"坏注释"？
    渐进式治理：每次修改一个文件时，顺手清理该文件中的坏注释。不要试图一次性重构全部，优先保证业务功能稳定。
  • Q：代码里有一段复杂算法，要不要写注释？
    要。但优先选择：先用语义化方法名拆解算法步骤；对核心公式或不直观的数学推导，写一行"意图注释"说明为什么用这个算法，以及边界条件。
