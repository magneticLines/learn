07. 边界（Boundaries）

系统总要和外部代码打交道——第三方库、开放 API、遗留模块。边界管理的艺术，在于让外部变化只击中一个点。

核心主张：隔离，而不是依赖

Robert C. Martin 在《代码整洁之道》第 8 章提出一个朴素但深刻的观点： 我们对外部代码（第三方库、外部 API、尚未完成的模块）的了解永远是有限的， 不应该让外部世界的不确定性直接渗透进自己的核心业务逻辑。

外部代码的接口随时可能变化：第三方库升级了方法签名、外部 API 更换了鉴权方式、 同事负责的模块还没写完……如果你的业务代码到处直接调用 thirdPartyLib.doXxx()， 一旦这个库升级，你就要改几十个文件。 边界（Boundary）的核心思路是：用一个薄封装层把外部世界挡在门外， 让内部代码只认识自己定义的接口。

一句话总结：边界不是"多写一层"的形式主义，而是把"变化的方向"集中到一处， 让外部的不稳定只影响边界层，而不扩散到整个代码库。

问题一：直接把第三方类型暴露出去

java.util.Map 是 JDK 自带的，但 Map 同样是一种"边界问题"的典型教材。 Martin 用 Map 举了一个非常经典的例子。

反例：把 Map 直接传来传去

Java — 反例boundary / anti
// 某个传感器管理模块
public class SensorController {

    private Map<String, Sensor> sensors = new HashMap<>();

    // 直接把内部 Map 暴露给调用者
    public Map<String, Sensor> getSensors() {
        return sensors;
    }
}

// 调用方——直接用 Map 的原生方法操作
Sensor s = (Sensor) controller.getSensors().get("sensorId");

问题在哪里？

  • 调用方可以直接对 Map 做任意操作：clear()、remove()，完全绕过你的控制。
  • 如果将来把底层容器从 HashMap 换成 ConcurrentHashMap，或者 Map 的泛型参数改变，所有调用方都要跟着改。
  • (Sensor) map.get(...) 这种强制类型转换，在 Java 5 泛型之前极为常见，极易在运行期抛 ClassCastException。
  • Map 的接口太宽泛，暴露了大量不该暴露的操作能力（这也叫"接口污染"）。

正例：用封装类包裹 Map

Java — 正例boundary / good
// 封装成专属类，只暴露业务需要的方法
public class Sensors {

    private Map<String, Sensor> map = new HashMap<>();

    public Sensor getById(String id) {
        // 类型转换封装在内部，调用方永远不用写强转
        return map.get(id);
    }

    public void add(String id, Sensor sensor) {
        map.put(id, sensor);
    }

    public boolean exists(String id) {
        return map.containsKey(id);
    }

    // 不暴露 clear()、entrySet()、values() 等危险或无关方法
}

// 调用方——类型安全，语义清晰
Sensor s = sensors.getById("sensorId");
设计收益：内部换用 ConcurrentHashMap 或 LinkedHashMap， 调用方零感知。增加校验逻辑（如 id 不能为空），只改一处。

技巧：用"学习性测试"探索边界

引入一个新的第三方库，你如何知道它的行为是否符合预期？ Martin 提出一种被他称为 Learning Tests（学习性测试） 的做法： 不要直接在生产代码里摸索 API，而是在测试文件中编写一批实验代码， 先搞清楚库的行为，再在生产代码里引用。

示例：学习 Jackson 的序列化行为

Java — 学习性测试（JUnit 5）learning test
@Test
void jacksonIgnoresUnknownFieldsByDefault() throws Exception {
    // 探索：JSON 里有额外字段时 Jackson 会不会报错？
    String json = "{\"name\":\"Alice\",\"extra\":\"oops\"}";
    ObjectMapper mapper = new ObjectMapper();

    // 用测试验证行为，而不是靠猜
    User user = mapper.readValue(json, User.class);
    assertEquals("Alice", user.getName());
}

@Test
void jacksonSerializesNullFieldsAsNull() throws Exception {
    // 探索：null 字段序列化后是省略还是写成 null？
    User user = new User("Bob", null);
    String json = new ObjectMapper().writeValueAsString(user);
    assertTrue(json.contains("\"email\":null"));
}

学习性测试有两个额外好处：

  • 升级保护网：当第三方库从 2.x 升到 3.x，跑一遍这批测试，马上知道哪些行为发生了变化。
  • 活文档：新同事接手项目，读这些测试就能快速理解"我们对这个库的依赖假设是什么"。

问题二：依赖尚未完成的模块

现实开发中经常遇到这种情况：你负责的模块 A 需要调用同事负责的模块 B， 但 B 还没写完，甚至接口还没定。怎么办？等是等不起的。

Martin 的建议是：先自己定义一个理想接口（Desired Interface）， 按你希望它应该长什么样来编程，再写一个适配器（Adapter）将来对接真实实现。 这就是"面向接口编程"在边界管理上的具体应用。

场景：我方在写数据分析层，队友在写无线电发送器

我方代码 边界层 队友代码（未完成）
AnalysisService ──→ Transmitter (interface) ──→ FakeTransmitter (stub)
                                               ↓（将来替换为）
                                               TransmitterAdapter → RealTransmitterAPI
Java — 先定义理想接口boundary / adapter pattern
// 第一步：我们自己定义"希望它长这样"的接口
public interface Transmitter {
    void transmit(Frequency frequency, DataStream stream);
}

// 第二步：用 Fake 让我方代码立刻可以跑起来测试
public class FakeTransmitter implements Transmitter {
    @Override
    public void transmit(Frequency frequency, DataStream stream) {
        // 仅用于测试，打印日志即可
        System.out.println("[FAKE] transmit at " + frequency);
    }
}

// 第三步：队友的代码交付后，写一个适配器对接真实 API
public class TransmitterAdapter implements Transmitter {

    // 队友的真实类，接口风格和我们期望的不同
    private final RealTransmitterAPI api;

    public TransmitterAdapter(RealTransmitterAPI api) {
        this.api = api;
    }

    @Override
    public void transmit(Frequency frequency, DataStream stream) {
        // 在这里做参数转换、异常包装，业务代码完全不知道真实 API 长什么样
        api.send(frequency.toHz(), stream.toBytes());
    }
}

// 我方业务代码——完全不依赖真实 API，只认 Transmitter 接口
public class AnalysisService {
    private final Transmitter transmitter;

    public AnalysisService(Transmitter transmitter) {
        this.transmitter = transmitter;
    }

    public void analyze(DataStream stream) {
        // ... 分析逻辑 ...
        transmitter.transmit(Frequency.of(144.39), stream);
    }
}
控制方向翻转了：真实 API 的设计者控制的是 RealTransmitterAPI， 但现在是我方控制 Transmitter 接口的形状。 适配器只存在于边界层，将来对接另一家供应商，只改适配器即可。

整洁边界的三条原则

原则 做法 反面教材
最小化边界接触面 只在少数几个地方直接使用第三方类型 全项目散落几十处 import thirdparty.Foo
封装，不继承第三方类 用组合包裹，而非继承第三方基类 class MyService extends ThirdPartyBase，升级时继承链爆炸
面向自定义接口编程 业务代码只依赖己方接口，适配器负责翻译 业务代码中直接 new ThirdPartyClient(...)

实际开发中的落地技巧

技巧 1：不要把第三方对象放进领域模型

领域对象（Entity、VO）是系统的核心，生命周期最长。 如果在 Order、User 这类类里直接引用 com.stripe.model.Charge，那么换支付供应商时， 你的领域模型也要跟着改，得不偿失。 正确做法是在 Service 层或 Gateway 层做转换，领域模型只含业务字段。

技巧 2：学习性测试也是回归测试

每次升级第三方依赖版本，先跑学习性测试。 如果某个行为的测试挂了，说明库的行为变了， 比翻 changelog 更直接、更可靠。把这批测试放在独立的 test/learning/ 目录， 与业务测试区分，保持整洁。

技巧 3：警惕"幸运的意外相同"

有时第三方接口和你期望的接口碰巧完全一样，于是你直接用了，没有封装。 但"现在一样"不代表"将来一样"。建议至少在模块边界处引入一个类型别名或 简单的 typedefimport，让以后的替换有据可查。

技巧 4：Spring 项目中的边界

Spring 的 @Repository / @Service 分层天然就是边界思想的体现： Service 层不应该直接依赖具体的 JPA 实体类或 MyBatis Mapper， 而应该通过 Repository 接口隔离。 遗憾的是很多项目为了"省事"，在 Service 里直接注入 EntityManager， 这正是边界被突破的典型症状。

Java (Spring) — 边界被突破反例
@Service
public class OrderService {
    @Autowired
    private EntityManager em;  // 直接依赖 JPA 实现层，换 ORM 框架时全改

    public Order findById(Long id) {
        return em.find(Order.class, id);
    }
}
Java (Spring) — 通过 Repository 接口隔离正例
// 边界接口：业务层只认这个
public interface OrderRepository {
    Optional<Order> findById(Long id);
}

@Service
public class OrderService {
    private final OrderRepository orderRepo;  // 只认接口，不认实现

    public Order findById(Long id) {
        return orderRepo.findById(id)
                .orElseThrow(() -> new OrderNotFoundException(id));
    }
}

常见陷阱与面试考点

陷阱：过度封装反而增加复杂度。 边界封装应当止于真正的"系统边界"——第三方库、外部服务、跨团队模块。 对于内部同一模块里的工具类，过度包一层只会增加跳转层数，没有意义。 判断标准：这个依赖有多大概率因外部原因变化？
陷阱：适配器忘记做异常转换。 第三方库往往抛自己的异常类型（如 com.stripe.exception.StripeException）。 如果适配器直接往外抛，调用方又要 import thirdparty.*，边界被突破。 应当在适配器里捕获并包装成自定义的业务异常。
  • 面试题：为什么不直接用 Map<String, Object> 到处传，而要封装成专属类？——回答就是本文第一节的内容。
  • 面试题：Adapter 模式和 Facade 模式的区别是什么？——Adapter 侧重"接口转换（让不兼容的接口配合工作）"，Facade 侧重"简化（隐藏子系统复杂度）"；边界管理两者都会用到。
  • 面试题：学习性测试与普通单元测试有什么区别？——学习性测试的被测对象是第三方库，目的是记录对该库行为的"假设"，而非验证自己的代码逻辑。

小结

边界管理的本质是控制变化的传播范围。 外部世界不可控，但你可以决定"外部的不稳定最多影响哪一层"。 三个核心动作：

  1. 封装外部类型——不把第三方类型直接暴露给调用方（Map 示例）。
  2. 学习性测试——在沙盒中探索第三方行为，并用测试锁定这些行为（升级保护）。
  3. 先定义理想接口，再用适配器对接——让我方代码掌控边界形状（无线电发送器示例）。

做到这三点，外部世界的变化只会"打在"边界层，而不会像涟漪一样扩散到整个代码库。

与其他知识点的联系

  • Clean Code《06. 错误处理》：边界层封装的不只是接口形状，还包括异常类型——适配器捕获第三方异常并包装成业务异常，正是错误处理章节"按调用者需求定义异常类"的具体应用场景之一。两篇合起来才是完整的"边界防御"：接口层面隔离 + 异常层面隔离。
  • GoF 适配器模式（Adapter）与外观模式（Facade）：本文"先定义理想接口，再用适配器对接真实实现"就是标准的适配器模式；而 Spring 的 Repository、Service 分层则更接近外观模式——用一个简化接口隐藏子系统（JPA、MyBatis）的复杂度。文中面试题也专门辨析了两者的差异。
  • 依赖倒置原则（DIP）："控制方向翻转了"这一节点出了边界管理的本质——传统做法是业务代码依赖第三方库的接口形状，而 DIP 要求高层模块定义自己的抽象，底层实现（第三方库）反过来适配这个抽象。这是 SOLID 五原则中最直接支撑边界隔离的一条。
  • Clean Code《08. 单元测试》与学习性测试：Learning Tests 本质上就是单元测试，只是被测对象从"自己的代码"换成了"第三方库的行为"。它同样要满足 F.I.R.S.T 原则（尤其是 Repeatable 和 Self-Validating），才能在库升级时真正起到回归防护的作用。
  • Spring 依赖注入（DI）："面向自定义接口编程"这条原则在 Spring 中的落地就是面向接口注入——Service 依赖 OrderRepository 接口而非具体的 JPA 实现类，Spring IoC 容器在启动时完成"适配器"的装配工作，这也是后文《10. 系统》要讲的"构造与使用分离"的一个具体实例。
