11. 涌现式设计（Emergent Design）

Kent Beck 的简单设计四规则——让良好架构从持续实践中自然生长，而非一次性大设计。

一、Kent Beck 的核心主张

传统软件开发常陷入"大设计在前"（Big Design Up Front，BDUF）的陷阱：在写第一行代码前，先花几周时间设计完整架构。Kent Beck 认为这是一种浪费——需求会变，预测性架构十有八九会被推翻。

他提出的替代方案叫做涌现式设计（Emergent Design）：不预先设计宏大架构，而是通过持续遵守四条简单规则，让良好的设计从代码演进过程中自然涌现出来。

类比：蚂蚁群体没有总指挥，每只蚂蚁遵循几条简单规则，整个蚁巢却涌现出极其复杂的智能行为。代码架构也可以如此——遵守局部规则，全局秩序自然形成。

Kent Beck 将简单设计的四条规则按优先级从高到低排列：

① 运行所有测试（优先级最高）
② 不存在重复代码
③ 代码表达了程序员的意图
④ 尽量减少类和方法的数量（优先级最低）

顺序不可颠倒：先让代码能跑，再消除重复，再提升表达力，最后精简体量。

二、规则一：运行所有测试

一个系统如果无法被验证，就无法被信任。测试迫使你把代码设计成可测试的，而可测试的代码天然具备低耦合、单一职责等优良特质。

为什么测试能驱动设计？
当你发现一个类难以测试时，往往是因为它承担了太多职责，或与外部依赖紧密耦合。为了能写测试，你会被迫把它拆小、注入依赖——这正是好设计的方向。

反例：无法测试的类

Java — 反例：直接 new 依赖，无法隔离测试
public class OrderService {
    private EmailSender sender = new EmailSender();    // 硬编码依赖
    private Database    db     = new Database("prod-host"); // 直连生产库

    public void placeOrder(Order order) {
        db.save(order);
        sender.send(order.getUserEmail(), "下单成功");
    }
}
// 问题：单元测试无法 Mock EmailSender 和 Database，
// 一跑测试就真的发邮件、真的写库。

正例：依赖注入，可测试

Java — 正例：构造器注入，测试时传入 Mock
public class OrderService {
    private final EmailSender sender;
    private final OrderRepository repo;

    // 依赖从外部注入——测试时可换成 Mock
    public OrderService(EmailSender sender, OrderRepository repo) {
        this.sender = sender;
        this.repo   = repo;
    }

    public void placeOrder(Order order) {
        repo.save(order);
        sender.send(order.getUserEmail(), "下单成功");
    }
}
关键洞察：为了可测试而做的设计决策（接口抽象、依赖注入、单一职责），恰好与高质量架构原则完全一致。测试不只是验证代码，它是设计的驱动力。

三、规则二：不存在重复代码（DRY 原则）

重复是一切软件腐化的根源。表面相似的代码复制粘贴在多处，一旦逻辑变更，就必须找到所有副本逐一修改——改漏一处便引入 Bug。

消除重复不只是"提取方法"那么简单。有时重复隐藏在不同的表现形式下，需要识别出相同的概念并抽象统一。

反例：隐式重复（相似实现散落各处）

Java — 反例：两处几乎相同的数据过滤逻辑
// 在 UserService 里
public List<User> getActiveAdmins() {
    return users.stream()
        .filter(u -> u.isActive())
        .filter(u -> u.getRole().equals("ADMIN"))
        .collect(Collectors.toList());
}

// 在 ReportService 里（几乎相同，多了排序）
public List<User> getAdminsForReport() {
    return users.stream()
        .filter(u -> u.isActive())
        .filter(u -> u.getRole().equals("ADMIN"))
        .sorted(Comparator.comparing(User::getName))
        .collect(Collectors.toList());
}
// 当"ADMIN"角色名改为枚举时，需要改两处。

正例：提取共同概念

Java — 正例：统一过滤逻辑，各自在此基础上扩展
// 提取"活跃管理员"这个概念到 User 自身（或 UserSpec）
public class User {
    public boolean isActiveAdmin() {
        return isActive() && Role.ADMIN.equals(role);
    }
}

// UserService
public List<User> getActiveAdmins() {
    return users.stream().filter(User::isActiveAdmin)
               .collect(Collectors.toList());
}

// ReportService
public List<User> getAdminsForReport() {
    return users.stream().filter(User::isActiveAdmin)
               .sorted(Comparator.comparing(User::getName))
               .collect(Collectors.toList());
}
// 角色枚举变更时只改一处：User.isActiveAdmin()
模板方法模式与 DRY：当多个子类有相同的算法骨架、只有某些步骤不同时，提取模板方法到父类，子类只覆写差异部分——这是消除结构性重复的经典手段。

四、规则三：代码表达了程序员的意图

代码被读的次数远多于被写的次数。清晰表达意图，让下一个读代码的人（通常是几个月后的你自己）能立刻理解"这段代码在做什么、为什么这么做"。

表达力的三个层次

层次 做法 例子
命名 用意图揭示型名字 calculateMonthlyInterest() 优于 calc()
函数大小 每个函数只做一件事 20 行以内，能用一句话描述职责
设计模式 在命名中使用模式词汇 PaymentStrategy、UserFactory

反例：神秘的实现细节

Java — 反例：读代码猜不出在做什么
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<>();
    for (int[] x : theList)
        if (x[0] == 4) list1.add(x);
    return list1;
}
// theList 是什么？x[0] 是什么字段？4 代表什么状态？
// 读者完全无从猜测，必须去翻其他文件拼图。

正例：让代码自我解释

Java — 正例：名字揭示意图，常量有名字
// 来自《代码整洁之道》原书同款例子，重命名后立刻清晰
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged())      // 状态封装进 Cell，不再是魔法数字 4
            flaggedCells.add(cell);
    return flaggedCells;
}
// 读者无需上下文：遍历棋盘，收集被标记的格子。一目了然。
设计模式名就是团队词汇表：当你把类命名为 TransactionDecorator 或 ReportBuilderFactory，读者立刻获得了大量隐含信息——结构、职责、扩展方式——这比任何注释都高效。

五、规则四：尽量减少类和方法的数量

这是四条规则里优先级最低的一条，但容易被误读为"越少越好"。它的真实含义是： 在满足前三条的前提下，避免过度设计产生不必要的抽象层。

常见陷阱：为每一个微小操作都创建独立接口、每个数据对象都配一个工厂类——这种"教条主义的设计"让代码类数量爆炸，反而比重复代码更难维护。

反例：过度抽象

Java — 反例：为一个简单行为创造不必要的层级
// 需求：把字符串转成大写。仅此而已。
public interface StringTransformer {
    String transform(String input);
}
public class UpperCaseTransformer implements StringTransformer {
    public String transform(String input) {
        return input.toUpperCase();
    }
}
public class StringTransformerFactory {
    public static StringTransformer createUpperCase() {
        return new UpperCaseTransformer();
    }
}
// 3 个类/接口，目的是调用 String.toUpperCase()。
// 没有任何多态扩展需求，纯属过度设计。

正例：直接使用语言能力

Java — 正例：需求简单就直接写，不强行分层
// 当前没有多态需求时，直接用方法引用或 lambda 即可
String upper = input.toUpperCase();

// 若将来真的需要可替换的转换策略（如大写、小写、驼峰）：
Function<String, String> transformer = String::toUpperCase;
// Java 函数式接口已经是天然的策略抽象，无需自己造轮子
YAGNI（You Aren't Gonna Need It）：不要为"以后可能需要"的场景预先添加抽象。等需求真正出现时，再重构——届时你对需求的理解更准确，设计反而更好。

六、四条规则如何协同产生涌现

四条规则不是孤立的检查项，它们构成一个持续循环：

写代码 → ① 跑测试（验证正确）
→ ② 消除重复（提取抽象）
→ ③ 改善表达力（重命名、拆分）
→ ④ 删除多余的层（精简体量）
→ 再写代码 → 循环

每次小循环后，代码都比之前更清晰、更正确、更精简。随着数十次、数百次这样的循环，一个良好的架构便从代码中自然涌现，而非被预先规划出来。

这就是为什么 Bob 大叔说："遵守这四条规则，设计就会自然成形。" 架构不是设计出来的，是实践出来的。

七、实际开发落地技巧

技巧 1：用 TDD 驱动规则一

先写测试，再写实现（红-绿-重构循环），天然保证每段代码都有测试覆盖，同时被迫设计成可注入、低耦合的结构。

技巧 2："童子军规则"消除重复

每次提交代码时，让代码比拉取时至少干净一点点——哪怕只是提取一个重复了三次的字符串常量。积累效应非常显著。

技巧 3：用 Code Review 提升表达力

让队友读你的代码，问他"你认为这个方法在做什么？"如果答案需要超过一句话，命名或拆分就还不够好。

技巧 4：用"三次法则"判断何时提取抽象

第一次写：直接实现。第二次出现：忍耐，留意。第三次出现：必须提取共同抽象。这样既避免过早抽象，又不让重复蔓延。

技巧 5：重构时遵守优先级顺序

不要跳过"让测试通过"直接去追求"好看的设计"。先绿灯，再重构——带着安全网改动才不会引入隐性错误。

容易忽略的细节：规则四（减少类和方法数量）是最后的约束，不是首要目标。很多人本末倒置：因为"类太多了"而不肯消除重复，导致违反了优先级更高的规则二。
