中介者模式 Mediator

用一个中介对象封装一组对象之间的交互。同事们不再彼此直连，而是只跟中介说话——把"网状耦合"拉直成"星形耦合"。

意图与定义

GoF · Mediator
用一个中介对象来封装一系列对象之间的交互。
Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly.

中介者让各对象不需要显式地相互引用，从而降低耦合，并能独立改变它们之间的交互逻辑。所有交互规则被集中到中介者一处，对象本身只负责自己的核心职责。

解决什么问题，为何这么设计

当一组对象需要频繁、复杂地相互通信时，如果让它们彼此直接持有引用，连接数会随对象数量呈平方级增长：n 个对象最多 n(n-1)/2 条连线。这就是网状耦合，它带来三个致命问题：

  • 任何一个对象改动，可能波及所有与它直连的对象
  • 交互逻辑散落在各个对象内部，没有一处能看清"整体协作规则"
  • 对象无法单独复用——它身上粘着对一堆兄弟的引用

中介者的设计思路：引入一个中心节点，所有交互都经它中转。对象只认识中介者，不认识彼此。连接数从 O(n²) 降到 O(n)，网状变星形。代价是交互复杂度从"分散"转移到了中介者"集中"——这是一次有意识的权衡。

生活类比：机场塔台

如果每架飞机都要直接和其他所有飞机协商起降顺序，机场会瞬间瘫痪——这就是网状通信。现实中，所有飞机只跟塔台（中介者）通话："请求降落"。塔台掌握全局状态，统一调度谁先谁后，再把指令下发给各架飞机。飞机彼此互不通话，只认识塔台。这正是中介者模式：飞机是"同事"，塔台是"中介者"，复杂的协调规则集中在塔台。

反例 → 问题 → 正例

需求：聊天室里，一个用户发消息，其他用户都要收到。

❌ 反例：用户之间互相持有引用，点对点广播
class User {
    private List<User> peers = new ArrayList<>();  // 每个用户都得认识其他所有人
    void addPeer(User u) { peers.add(u); }

    void send(String msg) {
        // 一个用户耦合了"群发逻辑"+"全体成员名单"
        for (User u : peers) u.receive(msg);
    }
    void receive(String msg) { /* 显示 */ }
}
// 新加一个人，要往每个已有用户的 peers 里塞引用——网状维护噩梦

问题：每个 User 都背着全体名单，耦合到所有兄弟；群发规则（屏蔽、私聊、敏感词过滤）只能复制粘贴进每个用户；增删成员要同步维护一堆引用，n 个用户 O(n²) 条连线。

✅ 正例：引入聊天室作为中介者，用户只认识它
// 中介者接口
interface ChatRoom {
    void register(User u);
    void send(String msg, User from);
}

// 同事：只持有中介者引用，不认识任何其他 User
class User {
    private final String name;
    private final ChatRoom room;     // 唯一的对外连接
    User(String name, ChatRoom room) {
        this.name = name; this.room = room; room.register(this);
    }
    void send(String msg) { room.send(msg, this); }  // 交给中介者
    void receive(String msg) { System.out.println(name + " 收到: " + msg); }
    String getName() { return name; }
}

// 具体中介者：所有协作规则集中在此
class ChatRoomImpl implements ChatRoom {
    private final List<User> users = new ArrayList<>();
    public void register(User u) { users.add(u); }
    public void send(String msg, User from) {
        // 广播规则、过滤、私聊……都只改这一个地方
        for (User u : users)
            if (u != from) u.receive(from.getName() + ": " + msg);
    }
}

现在用户之间零耦合；增删成员只跟中介者打交道；所有协作规则集中在一处，O(n) 条连线。User 类干净到可以单独复用。

角色与结构

角色职责本例对应
Mediator定义同事对象通信的接口ChatRoom
ConcreteMediator实现协作逻辑，持有并协调所有同事ChatRoomImpl
Colleague（同事）只持有中介者引用，所有跨对象交互都委托给中介者User

JDK / Spring 中的真实应用

场景说明
Spring MVC 的 DispatcherServlet前端控制器是典型中介者：浏览器请求只到它一处，再由它协调 HandlerMapping、Controller、ViewResolver，这些组件彼此互不直连。
ApplicationEventPublisher / 事件机制发布者与监听者通过 Spring 容器（中介者）解耦，谁都不持有对方引用。
Java Message Service / 消息中间件Broker（如 Kafka、RabbitMQ）是分布式版的中介者，生产者消费者互不可见。
GUI 框架的控制器对话框里诸多控件（按钮、文本框、列表）的联动逻辑集中在控制器/Presenter，控件彼此不直接通信。
java.util.Timer多个 TimerTask 通过 Timer 统一调度，任务之间不互相引用。

常见陷阱

中介者会变成"上帝对象"：所有协作逻辑都往中介者里塞，它会迅速膨胀成一个无所不知、难以维护的庞然大物。耦合没有消失，只是从同事之间转移到了中介者内部。要警惕中介者本身的复杂度。
不是所有交互都该走中介者：若对象间交互简单、固定，引入中介者只会增加一层无意义的间接。中介者的价值在于"复杂且多变的多对多交互"。

易混模式对比

模式意图与中介者的区别
Facade 外观为子系统提供一个简化的统一入口外观是单向的：外部→子系统，子系统不知道外观存在；中介者是双向多对多：同事与中介者互相通信、协同协作。外观简化访问，中介者管理交互。
Observer 观察者一对多的通知发布观察者是单向广播（Subject→Observers）；中介者协调多向、有状态的复杂交互。两者常配合：中介者内部可用观察者实现通知。
Proxy 代理为单个对象提供替身控制访问代理针对一个对象做访问控制；中介者协调一组对象的交互。
面试高频："外观和中介者的区别"。一句话答案——外观对子系统做单向简化封装，子系统组件感知不到外观；中介者管理同事间的双向协作，同事主动持有并调用中介者。

小结

  • 中介者模式：引入中心对象封装一组对象的交互，把网状耦合（O(n²)）拉直成星形（O(n)）。
  • 同事对象只认识中介者，彼此零耦合，可独立复用；协作规则集中在中介者一处，便于修改。
  • 代价：中介者容易膨胀成"上帝对象"，复杂度从分散转向集中——需控制其规模。
  • 真实身影：Spring DispatcherServlet、事件机制、消息中间件、GUI 控制器。
  • 对比要记牢：外观=单向简化入口，中介者=双向交互协调。

与其他知识点的联系

  • 外观模式 Facade：两者都常被拿来对比，因为都"引入一个对象简化复杂关系"。但外观是单向的——外部客户端调用外观简化访问子系统，子系统组件根本不知道外观的存在；中介者是双向多对多的——同事对象主动持有中介者引用、主动调用它来跟其他同事协作。记住这条区分线，是分辨这两个模式的面试标准答案。
  • 观察者模式 Observer：中介者内部经常借助观察者模式来实现"通知"这一环节——比如聊天室中介者可以用发布-订阅的方式通知各个用户，而不是自己遍历用户列表逐个调用。两者常常组合：中介者负责"该不该通知、通知谁"的协调逻辑，观察者负责具体的通知机制。
  • 责任链模式 Chain of Responsibility：两者都试图化解对象间复杂的耦合关系，但拓扑结构不同——责任链是线性的一条链，请求顺序传递、可能被某一环拦截；中介者是星形结构，所有同事都直连一个中心节点，交互可以是任意方向的。
  • Spring DispatcherServlet 与 IoC 容器：本文提到的 DispatcherServlet 是中介者模式在框架层最典型的例子——它协调 HandlerMapping、Controller、ViewResolver 等组件，这些组件彼此互不知道对方的存在。往更底层看，Spring 的 IoC 容器本身也是一种"超级中介者"：Bean 之间通过容器完成依赖注入，而不是互相 new 出对方，这正是控制反转（IoC）思想与中介者模式在解耦目标上的重合之处。
  • 发布-订阅与消息中间件：Kafka、RabbitMQ 等消息中间件本质上是分布式场景下的中介者——生产者和消费者都只跟 Broker 打交道，彼此互不可见，这与本文聊天室例子里用户只认识 ChatRoom 是完全一致的耦合结构，只是把它从单机对象关系放大到了跨进程、跨服务的系统架构层面。
