最少知识原则（迪米特法则）

只和你的直接朋友交谈。一个对象对其他对象知道得越少，被牵连改动的风险就越小。

原则定义

Head First 设计原则 #7
最少知识原则：只和你的密友交谈。
Principle of Least Knowledge — talk only to your immediate friends.

它的另一个名字是迪米特法则（Law of Demeter, LoD）。Head First 在外观模式（Facade，Ch.7）一章引入它——Facade 正是用一个统一入口把客户和复杂子系统隔开，让客户只需和这一个"朋友"打交道。

它解决什么问题

当代码里出现一长串点号——a.getB().getC().doSomething()——你就在悄悄地依赖一整条链上的所有类。这串链子叫"火车失事代码"（train wreck）。它的危险在于：

  • 知道得太多：调用方不只认识 a，还认识 B、C 的内部结构
  • 变化传染：链条中任何一环改了返回类型、改了方法名，调用方就要跟着改
  • 暴露内部：a 把自己的内部对象 B 直接交了出去，破坏封装

最少知识原则的解法是：让对象自己完成请求，不要让调用方穿过它去操作它的成员。你只对直接朋友说话，朋友再去找它的朋友——但那是朋友的事，不是你的事。

谁才算"直接朋友"

书中给出了具体准则：在一个方法里，只应该调用以下范围内对象的方法：

可以交谈的"朋友" 说明
对象自己 调用 this 的方法
方法的参数 作为参数传进来的对象
方法内创建的对象 在方法里 new 出来的对象
对象的直接成员（组件） 该对象持有的实例字段
红线：不要调用"从另一个方法调用返回回来的对象"的方法。getX().doSomething() 里的 doSomething() 就踩线了——getX() 返回的是别人的朋友，不是你的。

生活类比：让狗自己走路

你想让狗走路，正确的做法是对狗说"走"（dog.walk()），由狗自己去协调它的四条腿。荒谬的做法是你伸手去抓狗的腿一条条挪——dog.getLeg(0).move()、dog.getLeg(1).move()。后者意味着你必须知道狗有几条腿、腿怎么动，狗换了走路方式你全得改。对朋友下达意图，而不是替朋友操作它的零件。

反例 → 问题 → 正例

需求：根据温度站读数判断要不要带伞。

❌ 火车失事代码
// 穿过 station 去摸它内部的
// thermometer，再摸 thermometer
// 的读数对象
float t = station
    .getThermometer()
    .getTemperature()
    .getValue();

// 你被迫认识 Thermometer、
// Reading 两个本不该认识的类
✅ 让 station 自己回答
// 只和直接朋友 station 说话
float t = station.getTemperature();

// station 内部怎么拿到温度，
// 是它自己的事；它换了
// thermometer 实现，调用方
// 毫无感知
正例：把"知道内部"的责任收回到对象自身
class WeatherStation {
    private Thermometer thermometer;  // 直接成员，是 station 的朋友

    // 对外只暴露"意图"，不暴露内部对象
    public float getTemperature() {
        // 在自己内部和自己的成员交谈——完全合法
        return thermometer.readValue();
    }
}

// 调用方：只认识 WeatherStation 一个类
float t = station.getTemperature();
本质：最少知识原则把"遍历对象图"的脏活，从调用方移回到了对象内部。调用方少认识了几个类，耦合度随之下降。这正是"为松耦合设计而努力（#4）"的一个可量化、可操作的具体版本。

外观模式：最少知识的标准落地

书中用外观模式（Facade）演示这条原则。家庭影院有投影仪、功放、灯光、爆米花机十几个子系统，看电影要按顺序操作一大堆对象。Facade 提供一个 watchMovie() 方法，把这一堆交互全藏进去：

Client（客户）
只认识 HomeTheaterFacade 一个朋友
↓ facade.watchMovie()
HomeTheaterFacade
替客户和所有子系统交谈
↓
Projector
Amplifier
Lights
PopcornMaker

客户原本要认识四五个子系统类、记住它们的调用顺序；有了 Facade，客户只认识一个朋友。子系统怎么变，客户都不受影响——这就是最少知识原则在架构上的兑现。

与其他原则和模式的关系

原则 / 模式 关系
为松耦合而努力（#4） 最少知识是松耦合的"可执行细则"——少认识几个类，耦合自然降
封装 不暴露内部对象、只暴露行为，正是封装的延伸要求
外观模式（Ch.7） 本原则的代表模式：一个统一入口隔开客户与子系统
中介者模式 同样减少对象间的直接相识，让交互经由中介进行
得墨忒耳（Demeter）项目 原则得名于此项目；故又称迪米特法则

常见误区与陷阱

误区一：把"少点号"当成机械规则。原则关心的是类型上的依赖，不是字面的点号数量。流式 API（如 StringBuilder.append().append() 或 Stream 链）每次返回的是同一类型/同一对象，并没有让你认识新的类，不违反 LoD。别看到链式调用就喊违规。
误区二：为了"消灭点号"给每个对象套包装方法。过度遵守会产生大量"转发方法"（wrapper / delegate），station.getTemperature() 只是原样转调内部对象。包装层一多，代码反而臃肿、间接性上升。LoD 是用来降低耦合的，当转发的成本超过收益时，就是过度了。
务实的判断：问自己"调用方真的需要认识链条中间那些类吗？"。如果中间对象只是个数据载体（DTO/值对象），穿透它通常无害；如果中间对象是有行为、会演化的领域对象，穿透它就埋下了耦合雷。

小结

  • 最少知识原则 / 迪米特法则：只和直接朋友交谈，别穿过朋友去操作它的朋友
  • 直接朋友 = 自己、参数、方法内新建的对象、自己的直接成员
  • 典型坏味道是"火车失事代码" a.getB().getC().doX()，它让你认识一整条依赖链
  • 解法是把遍历内部的责任收回对象自身，对外只暴露意图
  • 外观模式是它的代表落地；本质上是松耦合（#4）的可操作版本
  • 别机械数点号、别滥造转发方法——它服务于降耦合，不是教条
