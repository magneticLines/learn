封装变化

所有设计模式的底层动机。理解了它，你就理解了为什么要有设计模式。

原则定义

Head First 设计原则 #1
找出应用中可能需要变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起。
Encapsulate What Varies

这条原则是所有设计模式的底层动机。GoF（四人帮）在书中反复强调的一句话是：

找到变化，封装变化。这是面向对象设计的第一准则。

为什么需要它？—— 从"鸭子问题"说起

Head First 第一章用一个经典案例引出这条原则：一个鸭子模拟游戏。

最初的设计：所有鸭子都继承自 Duck 基类，基类里有 fly() 和 quack() 方法。

反例：把变化和不变混在一起
abstract class Duck {
    public void swim() {
        // 所有鸭子都会游泳 → 不变的，放基类没问题
    }

    public void fly() {
        // ❌ 问题来了：橡皮鸭不会飞，木头鸭也不会飞
        // 把"飞"的行为放在基类，强迫所有子类继承
    }

    public void quack() {
        // ❌ 橡皮鸭是吱吱叫，木头鸭不出声
        // 子类必须覆盖，但覆盖是否合理？
    }
}

当需求变化时（加入不会飞的鸭子），问题爆发了：

  • 用继承覆盖？每次加新鸭子都要检查哪些方法要覆盖，容易遗漏
  • 用接口拆分 Flyable、Quackable？不同鸭子重复实现同样的飞行逻辑，代码无法复用

根本问题是：变化的行为（飞、叫）和不变的行为（游泳）混在了一起。

怎么解决：把变化独立出来

识别哪些会变化：飞的方式和叫的方式是变化点。把它们抽成独立的行为接口：

正例：封装变化点
// 把"飞"的变化封装成独立的行为族
interface FlyBehavior {
    void fly();
}

class FlyWithWings implements FlyBehavior {
    public void fly() { System.out.println("用翅膀飞！"); }
}

class FlyNoWay implements FlyBehavior {
    public void fly() { System.out.println("不会飞"); }
}

// 把"叫"的变化封装成独立的行为族
interface QuackBehavior {
    void quack();
}

class Quack implements QuackBehavior {
    public void quack() { System.out.println("呱呱叫"); }
}

class Squeak implements QuackBehavior {
    public void quack() { System.out.println("吱吱叫"); }
}

// Duck 基类只保留不变的部分
abstract class Duck {
    FlyBehavior flyBehavior;
    QuackBehavior quackBehavior;

    public void performFly() {
        flyBehavior.fly();   // 委托给行为对象
    }
    public void swim() { ... }   // 不变，留在基类
}

现在加新鸭子类型时，只需要选择行为的组合，不需要修改任何现有代码：

新增橡皮鸭：零修改已有代码
class RubberDuck extends Duck {
    public RubberDuck() {
        flyBehavior   = new FlyNoWay();    // 不会飞
        quackBehavior = new Squeak();      // 吱吱叫
    }
}

如何识别"变化点"

实际项目中，识别变化点的三个问题：

问题 判断依据
这段代码会因为需求变化而修改吗？ 如果答案是"可能会"，就是变化点
不同的子类/对象，这个行为会有差异吗？ 有差异就应该抽出去，而不是让子类覆盖
这个逻辑将来会扩展吗（加新算法/新策略）？ 会扩展就应该设计成可替换的

和其他模式的关系

封装变化是所有模式的动机，几乎每个 GoF 模式都是在回答"把哪部分变化封装起来"：

模式 封装了什么变化
策略模式 算法 / 行为的变化
观察者模式 依赖关系的变化（谁关心状态变化）
装饰者模式 职责的动态扩展
工厂模式 对象创建方式的变化
状态模式 状态转换逻辑的变化

使用它会导致什么问题

封装变化不是免费的，它引入了新的代价：

1. 类的数量爆炸

每个变化点都要抽成接口 + 若干实现类。鸭子案例里，两个变化点就多出了 4 个类（FlyWithWings、FlyNoWay、Quack、Squeak）。系统一大，几十个"行为类"散落各处，新人接手时反而更难理解全貌。

2. 间接层增加，追踪困难

原来一眼能看出"这只鸭子怎么飞"，现在要追到 flyBehavior 字段的运行时赋值，才能知道实际执行哪个类。调试时多了一层间接跳转。

3. 过早抽象的风险

如果你预判了一个"变化点"，但这个变化实际上从没发生，那个接口就成了永久的复杂度负担。书中没有强调这一点，但这是实际项目中最常见的副作用。

关键问题：封装变化的前提是"识别出真正的变化点"。识别错了，等于花代价封装了一个不存在的问题。

如何权衡

书中没有给出一个精确的"何时封装"的公式，但结合实践，有三条可操作的判断标准：

场景 建议
这段逻辑已经出现两种以上不同实现 立刻封装，收益明确
需求文档明确说"未来要支持多种策略" 提前封装，有文档背书
只是"感觉将来可能会变"，没有具体证据 先不封装，等真正变化时再重构（Rule of Three）
Rule of Three（三次法则）：第一次写，直接实现；第二次遇到类似需求，注意到重复；第三次出现，再提取封装。这是避免过早抽象最实用的经验法则。

与开放关闭原则（OCP）的关系

"封装变化"是手段，开放关闭原则（对扩展开放，对修改关闭）是目标。

把变化封装起来之后，新增行为只需要添加新的实现类，不需要修改已有代码——这正是 OCP 的体现。两者是同一设计思想的不同表述角度。
