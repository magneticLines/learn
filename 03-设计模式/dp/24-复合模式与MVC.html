复合模式与 MVC

复合模式（Compound Pattern）是把多个模式组合在一起，解决一个反复出现的大问题。MVC 不是一个模式，而是观察者、策略、组合三种模式协作的经典复合产物。

什么是复合模式

Head First · Ch.12
复合模式：将两个或多个模式组合起来，解决一个一般性的、反复出现的问题。
A compound pattern combines two or more patterns into a solution that solves a recurring or general problem.

注意辨析：复合模式不是简单把几个模式堆在一起，而是它们彼此咬合、协同工作，共同构成一个比单个模式更大的解决方案。MVC 就是被验证过无数次的复合模式。

MVC 解决什么问题、为何这么设计

一个交互式应用里混着三件事：数据与业务规则、界面呈现、用户输入的处理。如果全揉在一起（界面代码里直接改数据、直接写业务），会导致：换个界面要重写业务、改个规则要动界面、根本无法多视图共享同一份数据。

MVC 把这三件事拆开，各司其职：

  • Model（模型）：持有数据和业务逻辑，对界面一无所知。
  • View（视图）：负责把模型的数据呈现出来。
  • Controller（控制器）：接收用户操作，决定调用模型的哪个行为。

生活类比：餐厅的后厨与前厅

厨房（Model）只管做菜、掌握食材和配方，不关心客人坐哪桌、用什么餐具。服务员（Controller）接收客人点单，把请求传给后厨，并协调上菜。餐桌摆盘（View）负责把菜呈现给客人。客人想换个摆盘风格（新视图），后厨配方（业务逻辑）一点不用改——三方解耦，各自能独立演化。

MVC = 观察者 + 策略 + 组合

这是本章的核心：MVC 的优雅来自三个模式的咬合。理解每个模式在哪、解决什么，就真正懂了 MVC。

模式在 MVC 中的位置解决的问题
观察者 ObserverModel 是 Subject，View 是 ObserverModel 数据变化时自动通知所有 View 刷新，而 Model 不需要知道有哪些 View（松耦合）
策略 StrategyView 持有一个 ControllerController 是 View 的"行为策略"，可替换——同一个 View 换个 Controller 就有不同交互行为
组合 CompositeView 内部的 GUI 组件树窗口、面板、按钮组成树形界面，统一 paint() / 事件分发递归处理
三者如何串起来：用户点按钮 → View 把动作交给 Controller（策略） → Controller 调用 Model 改数据 → Model 作为 Subject 通过 观察者通知所有 View → View（本身是 组合的组件树）重新渲染。一个闭环里三个模式各负其责。
Model 用观察者通知 View，自己不认识任何 View
interface ModelObserver { void modelChanged(); }

class BeatModel {  // 书中的节拍器模型
    private List<ModelObserver> observers = new ArrayList<>();
    private int bpm;

    public void setBPM(int bpm) {
        this.bpm = bpm;
        notifyObservers();  // 数据变了，通知所有视图——不知道也不在乎是谁
    }
    private void notifyObservers() {
        for (ModelObserver o : observers) o.modelChanged();
    }
}

// View 持有 Controller（策略），把用户动作委托出去
class DJView implements ModelObserver {
    private ControllerInterface controller;  // ← 策略，可替换
    public void onIncreaseClicked() { controller.increaseBPM(); }
    @Override public void modelChanged() { /* 重绘 */ }
}

MVC 三方职责与依赖方向

角色职责它认识谁
Model数据 + 业务规则；状态变化时发通知不认识 View 和 Controller（只持有 Observer 接口列表）
View渲染模型数据；订阅模型；把用户操作转给控制器认识 Model（读数据）和 Controller（委托行为）
Controller解释用户输入，调用模型的对应行为认识 Model 和 View

Spring MVC 中的映射

Web 时代的 MVC 与桌面 MVC 略有不同（HTTP 无状态，View 难以"订阅" Model），但三方划分思想一脉相承。Spring MVC 的映射如下：

MVC 角色Spring MVC 对应
Controller@Controller / @RestController 里的处理方法，接收请求、调 Service
Model业务对象 + Service 层 + 传给视图的 Model/ModelMap 数据
ViewJSP / Thymeleaf 模板，或 @ResponseBody 直出 JSON
前端控制器DispatcherServlet 统一接收请求，分发给具体 Controller（这本身是前端控制器模式）
区别要点：桌面 MVC 里 Model 用观察者主动推给 View；Web 请求-响应模型里这条"推"链通常不存在——一次请求结束就断开，View 是被动渲染的（除非用 WebSocket/SSE 把观察者推送补回来）。回答面试时点出这个差异是加分项。

常见陷阱

胖控制器 / 贫血视图：把业务逻辑写进 Controller 是最常见的反模式。Controller 应当"薄"——只做请求解析与流程编排，业务规则归 Model/Service。Controller 里出现大段计算就是味道不对。
MVC 不是一个模式：面试若答"MVC 是一种设计模式"不够准确。它是架构模式 / 复合模式，由观察者 + 策略 + 组合等模式协作而成。说清这一点才显出理解深度。
View 直接改 Model 状态：View 应只读 Model 来渲染，写操作必须经 Controller → Model。View 越权直接改数据会破坏单向数据流，导致状态难以追踪。

MVC 与它的变体（易混对比）

架构核心差异
MVCController 处理输入并更新 Model；Model 经观察者通知 View
MVP用 Presenter 取代 Controller，View 与 Model 完全隔离，全部经 Presenter 中转（更易测试）
MVVMViewModel + 数据双向绑定，View 与 ViewModel 自动同步（Vue/WPF 等前端框架）

小结

  • 复合模式是多个模式咬合协作解决一个大问题，不是简单堆砌。
  • MVC = 观察者（Model 通知 View）+ 策略（View 的 Controller 可替换）+ 组合（View 组件树）。
  • 三方依赖单向清晰：Model 不认识 View；View/Controller 认识 Model——这正是松耦合原则的落地。
  • Spring MVC 把它映射到 DispatcherServlet（前端控制器）+ @Controller + Service/Model + 模板视图；Web 场景下观察者推送链通常缺失。
  • 变体 MVP / MVVM 是同一思想在可测试性、数据绑定方向上的演化。

与其他知识点的联系

  • 观察者模式：本文里 Model 通知 View 的机制，正是前面学过的观察者模式在真实架构里的直接应用——Model 是 Subject，View 是 Observer，Model 完全不认识具体是哪些 View 订阅了自己，只对着 ModelObserver 接口广播，这与"发布/订阅解耦"的核心思想一脉相承。
  • 策略模式：View 持有一个可替换的 Controller，这正是策略模式的应用——同一个 View 换一个 Controller 实现，就能获得完全不同的交互行为（比如同一个节拍器视图，换个"节拍加倍"策略的 Controller）。理解这一点后再看 MVC，会发现它不是凭空设计出来的架构，而是模式组合的自然产物。
  • 组合模式：View 内部的 GUI 组件树（窗口 → 面板 → 按钮）是组合模式的应用，`paint()` 或事件分发沿着组件树递归下去，这与本模块前面讲的"容器把操作委托给孩子"完全是同一套机制，只是换了个应用场景。
  • 前端控制器模式与 Spring MVC 的 DispatcherServlet：桌面 MVC 里 Controller 直接对应一个个用户动作处理器，而 Web MVC 引入了"前端控制器"这个额外角色——所有请求先统一进入 DispatcherServlet，再分发给具体的 @Controller，这是 MVC 思想在 Web 架构里演化出的新分工，值得和桌面版 MVC 对比着理解。
  • MVP / MVVM 与前端框架（Vue/React）：正文提到的 MVP 用 Presenter 隔离 View 和 Model 以提升可测试性，MVVM 靠数据双向绑定自动同步——这些都是 MVC 思想在不同技术栈里的变体演化，如果你同时在学前端，会发现 Vue 的响应式系统本质上就是把"观察者通知 View 刷新"这条链路做得更自动化、更细粒度。
