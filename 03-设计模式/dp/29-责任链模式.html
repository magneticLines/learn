责任链模式 Chain of Responsibility

把多个处理者串成一条链，请求沿着链依次传递，每个处理者决定"自己处理"还是"交给下一个"，从而让请求的发送者与接收者解耦。

意图 / 定义

GoF Chain of Responsibility
使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合。
Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request.

发送者不需要知道究竟是链上哪一个处理者会响应，它只管把请求丢进链头。请求顺着链向后传，直到某个处理者把它处理掉（或走到链尾）。

解决什么问题、为何这么设计

当一个请求需要经过"多个可能的处理步骤"，而具体由哪一步处理事先不确定时，朴素写法会堆出一长串 if-else/switch。这种写法的毛病：

  • 所有处理逻辑挤在一个方法里，违反单一职责；
  • 增删或调整处理顺序就要改这坨大方法，违反开放关闭原则；
  • 请求发送方被迫知道全部处理分支——耦合过紧。

责任链把每个处理步骤拆成独立的 Handler，用一根"next 指针"串起来。增加新处理者只需新写一个类并插入链中，调整顺序只需改组装代码，处理逻辑彼此隔离。本质是把"选择哪个分支处理"从编译期写死的 if-else，变成运行时可灵活编排的链。

生活类比：公司报销审批

你提交一张报销单，不需要知道该找谁签字。单子先到组长（1000 元以内他能批）；超额就转给经理（5000 元以内）；再超就转给总监。每一级"能处理就处理，处理不了就往上递"。你（发送者）只把单子交给链头，至于最终谁批的，你不关心。这就是责任链。

反例 → 问题 → 正例

❌ 一长串 if-else：所有审批逻辑挤在一起
void approve(Request req) {
    if (req.amount() <= 1000) {
        // 组长审批逻辑
    } else if (req.amount() <= 5000) {
        // 经理审批逻辑
    } else if (req.amount() <= 20000) {
        // 总监审批逻辑
    } else {
        // CEO 审批逻辑
    }
    // 新增一级审批 / 调整额度 → 必须改这个方法
    // 各级逻辑全耦合在一处，无法单独复用
}

问题所在：所有审批者的逻辑和阈值判断硬编码在一个方法里，每次业务调整都要动这块"地雷代码"，且各级逻辑无法独立测试或复用。

✅ 责任链：每级审批是独立 Handler，串成链
abstract class Approver {
    protected Approver next;            // 指向下一个处理者
    public Approver setNext(Approver n) { this.next = n; return n; }

    public final void handle(Request req) {
        if (canApprove(req)) {
            doApprove(req);          // 我能处理，就处理
        } else if (next != null) {
            next.handle(req);       // 处理不了，传给下一个
        } else {
            throw new IllegalStateException("无人能审批");
        }
    }
    protected abstract boolean canApprove(Request req);
    protected abstract void doApprove(Request req);
}

class Leader extends Approver {
    protected boolean canApprove(Request r) { return r.amount() <= 1000; }
    protected void doApprove(Request r) { /* 组长批 */ }
}
class Manager extends Approver {
    protected boolean canApprove(Request r) { return r.amount() <= 5000; }
    protected void doApprove(Request r) { /* 经理批 */ }
}

// 组装链：顺序、增删都在这里灵活编排，处理者本身不改
Approver chain = new Leader();
chain.setNext(new Manager()).setNext(new Director());
chain.handle(req);   // 发送者只面向链头，不关心谁最终处理

角色 / 结构

角色 职责 对应上例
Handler（处理者） 定义处理请求的接口，持有下一个处理者引用 Approver
ConcreteHandler 处理自己负责的请求，否则转发给后继 Leader、Manager
Client（客户端） 组装链并把请求提交给链头 组装 chain 的代码

两种链：纯链 vs 不纯链

  • 纯责任链：请求被某一个处理者"独占处理"后即终止（如上面的审批，批了就结束）。
  • 不纯责任链：每个处理者都处理一部分，再继续传递给下一个（如下面的 Servlet Filter、Netty Pipeline，人人都过一遍）。

JDK / Spring / Netty 中的真实应用

来源 责任链体现
Servlet Filter + FilterChain 请求经过一串 Filter，每个 Filter 调用 chain.doFilter(req, resp) 把请求传给下一个，处理完再原路返回。典型的不纯责任链
Spring MVC HandlerInterceptor preHandle 返回 true 才继续往下走，返回 false 则中断链。多个拦截器按注册顺序串联
Spring Security FilterChainProxy 认证、授权、CSRF 等十几个安全 Filter 组成的过滤链，是责任链的大型工程实例
Netty ChannelPipeline 入站/出站事件沿 ChannelHandler 链流动，每个 Handler 通过 ctx.fireChannelRead() 传给下一个
OkHttp Interceptor.Chain 请求/响应经过一串拦截器，chain.proceed(request) 驱动链向下
JDK java.util.logging 日志记录沿 Logger 的父级链向上传递（parent handler）

常见陷阱

忘记调用 next（链断了）：不纯责任链里，每个处理者必须显式把请求往下传（如 Filter 里的 chain.doFilter()、Netty 里的 ctx.fireChannelRead()）。忘了调用会导致请求在中途被"吞掉"，后面的处理者全部失效——这是最常见也最难排查的 bug。
请求可能无人处理：如果走到链尾仍没有处理者认领，请求会"掉地上"。设计时要么保证链尾有兜底处理者，要么在链尾抛异常/记录日志，明确这种情况。
链过长的性能与调试成本：链很长时每个请求都要逐个遍历，且调用栈深、不直观——出问题时很难一眼看出请求停在了哪一环。注意控制链长，必要时加日志埋点。

易混模式对比

模式 结构 与责任链的区别
责任链 Chain 请求沿链传递，处理者动态决定是否处理/转发 发送者不知道谁处理；处理者可中断或放行
装饰器 Decorator 层层包装，每层都增强行为后调用内层 结构像链，但目的是"叠加功能"而非"找人处理"，每层都必然执行
命令 Command 把请求封装成对象 关注"请求的封装与排队/撤销"，不涉及多处理者择一
观察者 Observer 一对多广播通知 所有观察者都收到通知（并行扇出）；责任链是顺序传递、可中断
面试高频："举一个责任链在框架里的例子"——首选 Servlet Filter / FilterChain，说清"每个 Filter 调用 chain.doFilter() 传给下一个、处理完原路返回、忘记调用会中断链"，再补一句 Spring 的 HandlerInterceptor 和 Netty 的 ChannelPipeline，就能体现你理解了纯链与不纯链的差别。

小结

  • 责任链模式 = 把多个处理者用 next 指针串成链，请求沿链传递，每个处理者"能处理就处理，否则转发"。
  • 它用一条可灵活编排的链替代了僵硬的 if-else，让发送者与处理者解耦，符合单一职责与开放关闭原则。
  • 分纯链（择一处理后终止）和不纯链（人人都过一遍，如 Filter）。
  • 框架落地极广：Servlet Filter、Spring Interceptor / Security、Netty Pipeline、OkHttp Interceptor。
  • 头号陷阱：不纯链里忘记调用 next 会把请求吞掉，导致后续处理者全失效。

与其他知识点的联系

  • 装饰器模式 Decorator：两者的代码结构极为相似——都是一层包一层、层层调用下一层。区别在于意图：装饰器的每一层都必然执行，目的是叠加增强功能（如给流加缓冲、加压缩）；责任链的每一环则动态决定是处理还是转发，目的是"找到那个该处理请求的人"。分辨"这是装饰器还是责任链"，就看每一层是否都必然参与，还是可能中途被某一环拦下。
  • 观察者模式 Observer：责任链是顺序传递、可中断（像接力棒，传到谁手上谁决定继续传还是叫停）；观察者是一对多广播、并行扇出（像群发通知，所有人都会收到）。责任链关心"谁来处理"，观察者关心"谁都要知道"。
  • 命令模式 Command：命令模式把请求封装成对象以支持排队、撤销、记录日志；责任链关注的是"请求该由链上哪个处理者处理"。两者常配合——比如把命令对象沿着一条责任链传递，链上的每个处理者决定是否拦截或增强这个命令。
  • Servlet Filter 与 AOP：Servlet FilterChain 是责任链在 Web 层的标准实践，而它和 Spring AOP 的环绕通知（@Around）在思想上高度相似——都是"层层包裹、依次放行"，理解责任链有助于理解 Spring AOP 底层通过多个 MethodInterceptor 链式调用 proceed() 的执行机制。
  • 开放关闭原则（OCP）：责任链把原本要写在一个大方法里的 if-else 拆成互相独立的 Handler 类，新增处理逻辑只需新增一个类、插入链中，不用修改已有代码，是 OCP 在行为型模式里的典型示范，和之前学过的策略模式、工厂方法用不同的手法达到同一个目标。
