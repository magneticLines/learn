解释器模式 Interpreter

给定一门语言，为它的文法定义一种表示，并定义一个解释器来解释这门语言中的句子。简单说：把"一句话"翻译成"一棵可执行的对象树"。

意图与定义

GoF · Interpreter
为某门语言定义文法，并构建一个解释器来解释该语言中的句子。
Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language.

解释器模式处理的是一类特殊问题：某种反复出现的规则，可以抽象成一门小语言（DSL）。模式的做法是把语言的每条文法规则映射成一个类，再把这些类组合成一棵抽象语法树（AST），对树做递归求值就完成了"解释"。

解决什么问题，为何这么设计

设想你要支持用户输入数学表达式 3 + 5 - 2，或者权限规则 admin AND (active OR vip)。这些输入有共同特征：

  • 它们遵循一套固定的文法（加减、与或、括号优先级）
  • 同一套文法会被反复套用在无数不同的具体句子上
  • 如果用一堆 if-else 硬解析，规则一变就要大改，没有结构

解释器模式的核心洞察是：文法规则天然是递归的，而递归的数据结构最适合用"组合 + 多态"表达。每个表达式节点只需回答一个问题——"给我一个上下文，我返回我的值"。终结符（数字、变量）是叶子，非终结符（加法、与运算）是内部节点，它们都实现同一个 interpret() 接口。这正是组合模式（Composite）在语法树上的应用。

生活类比：朗读一道竖式

小学老师把 (3 + 5) × 2 写在黑板上，让你算结果。你不会一口气得出答案，而是：先算括号里的 3 + 5 = 8，再算 8 × 2 = 16。每一步你只处理一个运算符，把它的两个"操作数"先各自求出来，再合并。这种"先求子表达式、再合并"的过程，就是解释器对语法树的递归求值。黑板上的算式是"句子"，你脑中的运算规则是"文法"，你本人就是"解释器"。

反例 → 问题 → 正例

需求：对加减表达式（含变量）求值，例如 x + 5 - y。

❌ 反例：一坨字符串解析与硬编码逻辑
int eval(String expr, Map<String,Integer> ctx) {
    String[] tokens = expr.split(" ");
    int result = resolve(tokens[0], ctx);
    for (int i = 1; i < tokens.length; i += 2) {
        String op = tokens[i];
        int val   = resolve(tokens[i + 1], ctx);
        if (op.equals("+"))      result += val;   // 加一种运算就多一个 if
        else if (op.equals("-")) result -= val;   // 没有结构，括号根本无从处理
        // else if ("*") ... else if ("/") ... 无限膨胀
    }
    return result;
}

问题：逻辑全部塞进一个方法，无法表达嵌套/优先级；新增运算符要改这个方法（违反开放关闭原则）；解析和求值耦合在一起，无法复用语法树；测试只能整体黑盒测。

✅ 正例：每条文法规则一个类，组合成语法树递归求值
// 抽象表达式：所有节点的统一契约
interface Expression {
    int interpret(Map<String, Integer> ctx);
}

// 终结符表达式：数字常量（叶子）
class Number implements Expression {
    private final int value;
    Number(int v) { this.value = v; }
    public int interpret(Map<String,Integer> ctx) { return value; }
}

// 终结符表达式：变量（从上下文取值）
class Variable implements Expression {
    private final String name;
    Variable(String n) { this.name = n; }
    public int interpret(Map<String,Integer> ctx) { return ctx.get(name); }
}

// 非终结符表达式：加法（内部节点，持有左右子表达式）
class Add implements Expression {
    private final Expression left, right;
    Add(Expression l, Expression r) { left = l; right = r; }
    // 递归：先解释子节点，再合并——这就是"先算子表达式再合并"
    public int interpret(Map<String,Integer> ctx) {
        return left.interpret(ctx) + right.interpret(ctx);
    }
}

// 客户端：手工搭出 x + 5 这棵语法树（实际由 parser 生成）
Expression expr = new Add(new Variable("x"), new Number(5));
int r = expr.interpret(Map.of("x", 10));  // 15

现在新增"减法"只需写一个 Subtract 类，不动任何已有代码；嵌套和优先级由树的形状自然表达；解析（建树）与求值（遍历树）彻底解耦，语法树可被缓存、复用、做静态分析。

角色与结构

角色职责本例对应
AbstractExpression声明所有节点共同的 interpret() 接口Expression
TerminalExpression终结符，文法中不可再分的最小单元，对应语法树的叶子Number、Variable
NonterminalExpression非终结符，由其他表达式组合而成，递归解释子节点Add、Subtract
Context解释时需要的全局信息（变量绑定等）Map<String,Integer>
Client构建语法树并触发解释（通常借助 parser）构建 Add(...) 的代码

JDK / Spring 中的真实应用

解释器模式在业务代码里很少需要手写——一旦文法复杂，手工维护语法树类会爆炸，通常改用 ANTLR、JavaCC 等解析器生成器。但它的思想在标准库与框架里无处不在：

场景说明
java.util.regex.Pattern正则表达式是一门微型语言，Pattern.compile() 把模式编译成内部节点树，Matcher 是解释器。这是 JDK 里最典型的解释器思想。
Spring 表达式语言（SpEL）ExpressionParser 把 #{user.age > 18} 解析成 Expression 对象树，getValue(context) 对树求值——角色命名几乎与 GoF 一一对应。
java.text.Format 家族MessageFormat、DateFormat 解释格式化模式串。
EL / JPQL / Spring Data 方法名JSP 的 ${}、JPA 的 findByNameAndAgeGreaterThan 都是把约定语法解释成行为。
面试提示：被问"解释器模式 vs 编译原理"时，要点明解释器模式只覆盖编译流程的最后一步——对已建好的 AST 求值；词法分析、语法分析（建树）这两步它并不规定，实践中往往交给生成器或手写 parser。

常见陷阱

文法一复杂就会类爆炸：每条文法规则对应一个类，规则多到几十条时，类的数量和维护成本将失控。这是解释器模式最大的硬伤——它只适合简单且稳定的小语言。
不要用它解析复杂语言：真正的编程语言、复杂查询语言请用 ANTLR / JavaCC。手写解释器去支持运算符优先级、错误恢复、左递归，会让你重新发明一个糟糕的编译器前端。
性能：递归遍历对象树伴随大量虚方法调用与对象分配，对极高频求值的热路径要评估是否改用编译为字节码 / 表驱动的方案。

易混模式对比

模式关注点与解释器的关系 / 区别
Composite 组合把对象组成树形结构，统一处理叶子与容器解释器的语法树就是一棵组合树；解释器是组合模式在"语言文法"这个具体领域的应用
Visitor 访问者在不改节点类的前提下给树加新操作当语法树上要加多种操作（求值、打印、类型检查）时，常用访问者给解释器节点扩展操作
Iterator 迭代器顺序遍历集合元素解释器是递归遍历树并合并结果，迭代器是线性遍历，目的不同
Strategy 策略封装可互换的单个算法策略是"一个算法的多种实现"，解释器是"一套文法的整棵规则树"

小结

  • 解释器模式：为一门小语言定义文法，把每条规则映射成一个类，组合成语法树后递归求值。
  • 本质是组合模式 + 多态：终结符是叶子，非终结符是内部节点，统一实现 interpret(context)。
  • 优点：新增文法规则不改老代码（符合开放关闭原则），结构清晰、可组合可复用。
  • 缺点：文法复杂时类爆炸，性能一般——所以业务中极少手写，复杂场景交给 ANTLR / JavaCC。
  • 真实身影：正则 Pattern、Spring SpEL、各类 EL / 格式化模式串。

与其他知识点的联系

  • 组合模式 Composite：解释器模式的语法树本质上就是一棵组合树——终结符是叶子节点，非终结符是持有子表达式的内部节点，统一实现同一个 interpret() 接口。可以说解释器是组合模式在"语言文法"这个具体领域的应用，理解了组合模式的"统一处理叶子与容器"思想，就能一眼看穿解释器的结构。
  • 访问者模式 Visitor：当语法树建好之后，如果需要对同一棵树做多种不同操作（求值、格式化打印、静态类型检查），比起在每个节点类里堆方法，更好的做法是引入访问者模式，把"操作"从"树结构"里剥离出来。这是解释器模式在实际编译器/DSL 引擎里几乎必然搭配的下一步。
  • 策略模式 Strategy：两者都通过接口+多态实现"可替换的行为"，但粒度不同——策略封装的是一个可互换的算法整体；解释器把一条复杂规则拆解成一整棵由多个节点组合而成的规则树，通过递归而非单次调用完成计算。
  • 正则表达式与 Spring SpEL：本文提到的 Pattern/Matcher 和 Spring 的 ExpressionParser 都是解释器思想在标准库和框架里的真实落地，角色命名（Expression、Context、interpret/getValue）几乎与 GoF 定义一一对应，是把抽象模式和日常写的 @Value("#{...}") 注解联系起来的关键桥梁。
  • 编译原理与递归下降：解释器模式只覆盖"对已经建好的 AST 求值"这一步，词法分析、语法分析（如何把字符串解析成树）并不在模式定义范围内，实践中常配合手写的递归下降解析器或 ANTLR/JavaCC 生成器一起使用——理解这个边界，能避免把解释器模式和整个"写一个编译器"混为一谈。
