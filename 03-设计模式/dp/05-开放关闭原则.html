开放关闭原则（OCP）

类应该对扩展开放，对修改关闭。能在不动一行已有代码的前提下加入新行为——这是装饰者模式背后的核心追求。

原则定义

Head First 设计原则 #5
类应该对扩展开放，对修改关闭。
Classes should be open for extension, but closed for modification.

这条原则最早由 Bertrand Meyer 提出，Head First 在装饰者模式（Ch.3）一章引入它。装饰者模式正是 OCP 的教科书演示：用包装的方式给对象加功能，而不去改对象本身的代码。

它到底在解决什么问题

软件唯一不变的就是变化。需求会增加：星巴克咖啡今天要加摩卡，明天要加豆浆，后天要加双份奶泡。问题不在于"会变"，而在于每次变化都要回去修改一份已经能正常工作、已经过测试、已经上线的旧代码。

修改旧代码的代价是隐蔽而昂贵的：

  • 每改一行，都有引入回归 bug 的风险——之前测试通过的逻辑可能被破坏
  • 旧代码可能被很多地方依赖，改它等于动了一个高扇出的节点
  • 在团队协作里，改公共代码意味着 merge 冲突、code review、重新走整套验证流程

OCP 的解法很巧妙：把"变化"从"修改"转化为"新增"。新增一个类不会破坏任何已有的东西，因为旧代码根本没被碰过。这就是"对修改关闭"的真正含义——不是禁止改，而是让你不必改。

违反 OCP
加功能 = 改旧类
大量 if/switch 判断
每次都动核心代码
部分满足
抽出了接口
但仍有少量改动
扩展点不够干净
满足 OCP
加功能 = 加新类
旧代码零改动
通过抽象点扩展

生活类比：手机壳与电源插座

想给手机加防摔、加支架、加卡包功能，你不会拆开手机改主板（修改），而是套一层手机壳（扩展）。手机本身对你封闭——你碰不到内部；但它对外的卡口是开放的——任何符合规格的壳都能套上去。换壳、叠壳，手机一无所知。

电源插座也一样：墙上的插座规格固定（关闭），但你可以插排插、插转换头、插各种电器（开放）。稳定的接口 + 可插拔的实现，就是 OCP 的物理直觉。

反例 → 问题 → 正例

以"咖啡计价"为例。需求：基础咖啡，可加各种调料，每种调料加价，最后算总价。

❌ 反例：用字段 + if 堆调料，每加一种料就要改 cost()
class Beverage {
    boolean hasMilk, hasMocha, hasSoy;   // 每种调料一个布尔字段

    double cost() {
        double total = 2.0;              // 基础价
        if (hasMilk)  total += 0.3;
        if (hasMocha) total += 0.5;
        if (hasSoy)   total += 0.4;
        return total;                    // 新增"奶泡"？必须回来改这里
    }
}

问题清单：

  • 每新增一种调料，就要修改 Beverage 这个核心类——直接违反"对修改关闭"
  • cost() 方法随调料种类无限膨胀，成了一个谁都不敢碰的大泥球
  • 无法表达"双份摩卡"（布尔字段只有有/无两态）
  • 调料的计价逻辑分散在一个巨型方法里，无法独立复用和测试

用装饰者模式重构：让每种调料成为一个包装器，包住饮料，把自己的价格叠加上去。

正例：装饰者模式——加调料 = 加新类，Beverage 永不修改
// 稳定的抽象：所有饮料和调料都是 Beverage（对修改关闭的核心）
abstract class Beverage {
    abstract double cost();
}

// 一个具体饮料
class Espresso extends Beverage {
    double cost() { return 2.0; }
}

// 调料装饰者基类：它"是一个" Beverage，又"包着一个" Beverage
abstract class CondimentDecorator extends Beverage {
    protected Beverage beverage;   // 持有被包装的对象
}

// 新增"摩卡"：只是新加一个类，不碰任何已有代码 ←── 对扩展开放
class Mocha extends CondimentDecorator {
    Mocha(Beverage b) { this.beverage = b; }
    double cost() { return beverage.cost() + 0.5; }
}

// 使用：层层包装。双份摩卡？再包一层即可，依旧零修改
Beverage b = new Mocha(new Mocha(new Espresso()));
System.out.println(b.cost());  // 2.0 + 0.5 + 0.5 = 3.0
看出门道了吗：抽象类 Beverage 是"关闭"的——它的代码永远不动。而它的子类体系是"开放"的——你随时可以加新饮料、新调料。变化全部发生在新文件里。

关键洞察：扩展点是"设计"出来的，不是天生的

OCP 最容易被误解的地方是：以为它要求"所有地方都对扩展开放"。这是不可能也没必要的——为每个可能的变化都预留抽象，会让系统淹没在过度设计里。

正确的做法是：识别出最可能变化的那个维度，只在那里引入抽象。咖啡店里"调料种类"是高频变化点，所以在那里用装饰者；而"咖啡是热饮还是冷饮"如果几乎不变，就不必为它建抽象。

判断 说明
识别变化轴 哪个维度的需求总在增加？把它找出来——那就是要"对扩展开放"的地方
抽象那个轴 用接口/抽象类把变化点封装起来，让具体实现可插拔
其余保持简单 不变或极少变的部分，不要为了"看起来 OCP"强行抽象

实现 OCP 的常见手段

手段 怎么做到"扩展开放、修改关闭"
继承 / 多态 新增子类覆写行为，父类不变。最基础但要小心继承的脆弱性
装饰者模式 用包装叠加功能，被包装对象零修改（本章主角）
策略模式 把算法抽成接口，新增算法 = 新增策略类，上下文不变
工厂 + 接口 新增产品只需加实现类和注册，调用方面向接口不变
插件 / SPI 机制 Java ServiceLoader、Spring 的扩展点：扔个 jar 进去就能扩展，主程序不重新编译

常见误区与陷阱

误区一：到处预留扩展点。为每个想象中的变化都建抽象，是典型的过度设计（over-engineering）。YAGNI（You Aren't Gonna Need It）告诉你：在变化真正发生第二次之前，先别急着抽象。OCP 是用来应对已经看清的变化轴，不是用来占卜未来。
误区二：用 if/switch 假装满足了 OCP。很多人把"新增分支"当成"扩展"。但 switch(type) 里加一个 case 仍然是在修改原方法，违反"对修改关闭"。真正的 OCP 是连那个 switch 都不用碰——通过多态分发，新类自己接住自己的逻辑。
误区三：以为"修改关闭"= 绝对不能改代码。修复 bug、重构当然要改。OCP 针对的是因新增需求而被迫修改稳定代码这一种情况。理解它的边界，才不会被原则绑架。

小结

  • OCP：对扩展开放，对修改关闭——加新行为靠加新类，而不是改旧类
  • 它解决的核心痛点是：修改已上线、已测试的稳定代码风险高、代价大
  • 实现方式是把最可能变化的维度抽象成稳定接口，让实现可插拔
  • 装饰者模式是它的典型演示；策略、工厂、SPI 都是它的不同落地形态
  • 切忌过度抽象——OCP 服务于已识别的变化，不是给所有代码套壳的借口
