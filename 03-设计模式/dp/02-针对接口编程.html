针对接口编程，不针对实现

让调用方只依赖"能做什么"，而不依赖"谁来做"。这条原则是多态真正发挥价值的前提。

原则定义

Head First 设计原则 #2
针对接口编程，不针对实现编程。
Program to an interface, not an implementation.

书中对这条原则有一个关键补充说明：这里的"接口"不是 Java 的 interface 关键字，而是指"超类型"（supertype）——可以是抽象类，也可以是接口。核心是：变量声明的类型应该是超类型，而不是具体实现类。

为什么有这条原则

早期面向对象代码的一个普遍写法是：需要什么类，就 new 什么类，变量也声明成具体类型。这在系统小、实现单一时没有问题，但一旦需要替换实现，每一处 new 和每一处声明都要跟着改。

这种写法把"我要一个能存数据的东西"和"我要一个 ArrayList"混为一谈——调用方其实只关心前者，却被迫绑定在后者上。

GoF 原书把这条原则表述为："Program to an interface, not an implementation"，并明确说这是面向对象设计中最重要的原则之一，几乎所有设计模式都以它为基础。

它解决了什么问题

问题：调用方与实现类紧耦合

针对实现编程，意味着调用方直接依赖具体类：

❌ 针对实现编程
// 变量类型是具体类
Dog dog = new Dog();
dog.bark();   // 只能是狗叫

// 想换成猫？每处都要改
// Dog dog → Cat cat
// dog.bark() → cat.meow()
✅ 针对接口编程
// 变量类型是超类型
Animal animal = new Dog();
animal.makeSound();  // 只依赖接口

// 想换成猫？只改一处 new
Animal animal = new Cat();
animal.makeSound();  // 调用方不变

更进一步：运行时才决定用谁

针对接口编程的真正威力，是可以把"用哪个实现"的决定推迟到运行时，调用方完全不知道具体类型：

运行时决定实现——调用方零修改
// 工厂方法负责创建，调用方只拿到接口
Animal animal = getAnimal(type);
animal.makeSound();

// 无论 getAnimal() 返回 Dog、Cat 还是 Bird
// 这一行代码永远不需要改

在书的鸭子案例中的体现

Duck 类持有 FlyBehavior 接口而不是具体的飞行类，正是这条原则的直接应用：

鸭子案例：Duck 持有接口而非具体类
abstract class Duck {
    // 声明为接口类型，不是 FlyWithWings 或 FlyNoWay
    FlyBehavior flyBehavior;
    QuackBehavior quackBehavior;

    public void performFly() {
        flyBehavior.fly();   // Duck 不知道也不在乎具体实现
    }

    // 甚至可以在运行时动态替换行为
    public void setFlyBehavior(FlyBehavior fb) {
        this.flyBehavior = fb;
    }
}

这样一来，不仅可以在创建时选择飞行方式，甚至可以运行时切换——把一只普通鸭子变成火箭鸭，调用方代码一行不改。

三个层次的理解

书中展示了这条原则的递进理解方式：

Level 1

变量声明用超类型：List<String> list = new ArrayList<>() 而不是 ArrayList<String> list = ...

Level 2

方法参数用超类型：void process(List<String> list) 而不是 void process(ArrayList<String> list)，调用方传什么实现都行

Level 3

把创建与使用分离：调用方完全不出现 new 具体类()，创建权交给工厂或外部注入，调用方只持有接口引用

大多数人停在 Level 1，而设计模式要求的是 Level 3。

使用它会导致什么问题

1. 接口泛滥，过度抽象

为每个类都配一个接口，系统里充斥着 IUserService、IOrderRepository 这类只有一个实现的接口。这是把原则机械套用的结果——接口的价值在于"有多个实现"或"需要隔离依赖"，如果两者都不满足，接口只是噪音。

2. 可读性下降

读代码时看到 FlyBehavior flyBehavior，你不知道它实际是哪个类，必须去追运行时赋值。对于简单逻辑，直接看到 FlyWithWings 反而更清晰。

3. 增加测试和理解的门槛

新人接手一个到处是接口的系统，需要额外的时间理解"这个接口有哪些实现"、"实际运行时注入的是谁"。IDE 的"Find Implementations"成了必备操作。

如何权衡

场景 建议
有两种以上实现，或实现会变化 针对接口编程，收益清晰
需要单元测试（用 Mock 替换真实实现） 必须针对接口，否则无法 Mock
对外发布的 API 或库 必须针对接口，保持向后兼容
内部工具类，只有一个实现且不会变 直接用具体类，不必强行抽接口
业务逻辑简单，团队规模小 可以先用具体类，等出现第二个实现时再重构
一个实用的判断句式：问自己"如果我想换掉这个实现，需要改多少地方？"如果答案超过一处，就应该针对接口编程。
与依赖注入（DI）的关系：Spring 等框架的依赖注入本质上就是把"针对接口编程"系统化——你声明接口类型的字段，框架负责在运行时注入具体实现。这条原则是理解 Spring IoC 的前提。

与其他原则的关系

原则 / 模式 关系
封装变化（第一条） 把变化点封装成接口，再针对这个接口编程——两条原则配合使用
依赖倒置原则（DIP） DIP 是这条原则的加强版：不只是调用方依赖接口，连高层模块也不能依赖低层模块，都要依赖抽象
策略模式 策略模式的核心就是：Context 持有 Strategy 接口，而不是某个具体策略类
工厂模式 工厂负责创建具体类，让调用方只拿到接口——是实现 Level 3 的标准手段
