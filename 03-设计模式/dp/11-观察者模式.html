观察者模式 Observer

在对象之间定义一对多的依赖关系，当一个对象（Subject）状态改变时，所有依赖它的对象（Observer）都会自动收到通知并更新。

意图与定义

GoF Observer Pattern
一对多自动通知：状态一变，所有订阅者立即收到更新。
Define a one-to-many dependency so that when one object changes state, all its dependents are notified automatically.

观察者模式是出版-订阅（Publish-Subscribe）思想在对象层面的标准实现。它解决"一个对象的状态变化需要联动通知一群对象，但又不想把它们硬绑在一起"的问题，是松耦合原则（第四条设计原则）最经典的演示。

它解决什么问题

当存在"数据源"和"多个依赖该数据的消费者"的关系时，两种朴素做法都有缺陷：

  • 数据源主动调每个消费者：数据源必须持有所有消费者的具体引用，新增/删除消费者都要改数据源代码——紧耦合。
  • 消费者轮询数据源：每个消费者不停问"变了吗"，浪费资源且有延迟。

观察者模式的设计：让 Subject 只维护一个 Observer 接口列表，状态变化时遍历列表统一调用 update()。Subject 不知道也不关心列表里具体是谁，只知道"它们都实现了 Observer 接口"。这样新增观察者无需改 Subject，订阅关系可在运行时动态增删。

生活类比：报社订阅

报社（Subject）有新刊就送到每位订阅者（Observer）手中。报社不需要认识每个读者是谁、住哪儿；读者也不必每天跑去报社问"出新刊了吗"。读者只要"订阅"，报社出刊就自动推送；想退订随时退，报社照常运营。这正是一对多 + 松耦合 + 自动通知。

反例 → 问题 → 正例（气象站案例）

书里的案例：气象数据对象 WeatherData 在测量值更新时，要刷新多个显示面板（当前状况、统计、预报）。

❌ 反例：硬编码调用每个显示器
class WeatherData {
    void measurementsChanged() {
        // 写死了具体显示器，每加一个就要回来改这里
        currentDisplay.update(temp, humidity, pressure);
        statsDisplay.update(temp, humidity, pressure);
        forecastDisplay.update(temp, humidity, pressure);
    }
}

问题在哪：

  • 针对具体实现编程，新增显示器必须修改 measurementsChanged()——违反开放关闭原则
  • 无法在运行时动态增删显示器
  • WeatherData 和显示器之间是 N 条硬连线，任何一端改动都波及对方

正例：引入 Subject 与 Observer 接口，Subject 只面向接口工作。

Subject 与 Observer 接口 + 注册/通知机制
interface Observer { void update(float t, float h, float p); }

interface Subject {
    void registerObserver(Observer o);
    void removeObserver(Observer o);
    void notifyObservers();
}

class WeatherData implements Subject {
    private List<Observer> observers = new ArrayList<>();
    private float temp, humidity, pressure;

    public void registerObserver(Observer o) { observers.add(o); }
    public void removeObserver(Observer o) { observers.remove(o); }

    public void notifyObservers() {
        // 只调接口，不认识对面是谁——这就是松耦合
        for (Observer o : observers) o.update(temp, humidity, pressure);
    }

    public void measurementsChanged() { notifyObservers(); }
}

// 新增显示器：实现接口 + 注册，WeatherData 一行不改
class CurrentConditionsDisplay implements Observer {
    public void update(float t, float h, float p) {
        System.out.println("当前温度: " + t);
    }
}

推模型 vs 拉模型

Subject 把数据传给 Observer，有两种姿势，耦合度不同：

推模型 Push（较紧）
// Subject 主动把数据塞过来
void update(float t,
            float h,
            float p);
// 问题：Observer 被迫接收
// 全部数据；Subject 新增字段
// 接口签名就要改
拉模型 Pull（更松）
// 只通知，Observer 自己来取
void update(Subject s);

float t = s.getTemperature();
// Observer 只取需要的字段
// Subject 加字段不影响别人
取舍：推模型简单直接、适合数据少且固定的场景；拉模型更松、更易扩展，是 Java 标准库（已废弃的 java.util.Observable）和现代事件系统的倾向。

角色与结构

角色职责气象站对应
Subject（主题/被观察者）维护观察者列表，提供注册/注销，状态变化时通知WeatherData
Observer（观察者）定义收到通知后的更新接口Observer 接口
ConcreteObserver实现更新逻辑，从 Subject 获取所需数据CurrentConditionsDisplay 等

JDK 与 Spring 中的真实应用

  • JDK Swing/AWT 事件：button.addActionListener(listener) 就是注册观察者，按钮点击时通知所有 ActionListener。PropertyChangeListener 同理。
  • JDK java.util.Observable / Observer：早期内置实现，但因 Observable 是类（强制继承、无法配合其它继承）等缺陷，已在 Java 9 标记 @Deprecated，官方建议改用 PropertyChangeListener 或响应式流。
  • Spring ApplicationEvent / ApplicationListener：发布 applicationEventPublisher.publishEvent(event)，所有 @EventListener 标注的方法自动收到——这是观察者模式在框架级的标准落地，发布者与监听者完全解耦。
  • 响应式编程：RxJava 的 Observable、Reactor 的 Flux/Mono 是观察者模式的现代加强版（加上背压、操作符链）。

常见误区与陷阱

内存泄漏（最高频）：Observer 忘记 removeObserver() 注销，Subject 一直持有它的引用，导致 Observer 无法被 GC。Android/Swing 里这是经典内存泄漏源。可考虑用弱引用（WeakReference）持有观察者。
通知顺序与重入：不要依赖观察者被通知的先后顺序。若某个 Observer 的 update() 里反过来修改 Subject 状态、触发新一轮通知，可能导致死循环或 ConcurrentModificationException（通知遍历时增删列表）。

与易混模式的对比

对比项观察者模式 Observer发布订阅 Pub/Sub（消息中间件）
耦合Subject 直接持有 Observer 列表，二者还有引用关系发布者与订阅者通过中间的 Broker 隔开，彼此完全不知
同步性通常同步调用 update()通常异步，经消息队列投递
范围进程内对象间常跨进程/跨服务
观察者模式中介者模式 Mediator
一对多：一个 Subject 广播给多个 Observer多对多：用中介者集中协调一群对象的复杂交互
单向通知（Subject → Observer）双向协调，对象都向中介者汇报

小结

  • 观察者模式 = 一对多依赖 + 面向 Observer 接口 + 状态变化自动通知。
  • 它把"谁需要被通知"从 Subject 里解耦出去，新增观察者无需改 Subject，订阅可运行时动态增删——松耦合的典范。
  • 推模型简单、拉模型更松；JDK 事件监听、Spring ApplicationEvent、RxJava 都是它的应用。
  • 最大陷阱是忘记注销导致内存泄漏，以及通知时重入引发的并发问题。
