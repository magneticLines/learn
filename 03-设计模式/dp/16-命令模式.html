命令模式 Command

把"一个请求"封装成一个对象。当请求变成对象，它就能被存储、传递、排队、记录、撤销——动词被名词化，于是拥有了名词的一切能力。

意图与定义

GoF 定义
将一个请求封装成一个对象，从而让你用不同的请求、队列或日志来参数化其他对象，并支持可撤销的操作。
Encapsulate a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support undoable operations.

核心思想一句话：把"调用方法"这个动作，包装成一个对象。原本 light.on() 是一次性的调用，命令模式把它装进 LightOnCommand 对象里，调用被推迟、被保存、被搬运到任何地方再执行。

解决什么问题，为何这么设计

命令模式要解决的是"请求的发出者"与"请求的执行者"之间的解耦。发出请求的人（Invoker，比如遥控器按钮）不应该知道具体要做什么、由谁来做。它只想说"执行"，至于执行的是开灯、关门还是启动音响，它不关心。

为什么要解耦？因为如果 Invoker 直接写死 light.on()，那么：每换一个设备就要改 Invoker；无法在运行时切换按钮功能；更无法做撤销、排队、日志。把请求封装成对象后，Invoker 只持有一个 Command 引用，调用 command.execute() 即可——具体做什么由注入的命令对象决定。

生活类比：餐厅点单

顾客（Client）写下订单（Command），服务员（Invoker）把订单送进厨房并喊"开始"，厨师（Receiver）按订单做菜。服务员不需要会做菜，也不知道怎么做——她只负责传递和触发订单。订单可以排队（订单队列）、可以记录（账单日志）、做错了可以作废（撤销）。这正是命令模式的全部精髓。

反例 → 问题 → 正例

反例：遥控器把所有设备逻辑写死

❌ Invoker 与每个 Receiver 紧耦合
class RemoteControl {
    Light light;
    GarageDoor door;
    Stereo stereo;

    void onButtonPressed(int slot) {
        // 每加一个设备就要改这里，按钮和设备硬绑定
        if (slot == 0) light.on();
        else if (slot == 1) door.up();
        else if (slot == 2) { stereo.on(); stereo.setVolume(11); }
        // 想支持"撤销"？这里完全无从下手
    }
}

问题：遥控器必须认识每一个具体设备的 API；新增设备要改遥控器源码（违反开闭原则）；按钮功能写死，无法运行时重新配置；撤销功能无处安放。

正例：把每个请求封装成 Command 对象

命令接口 + 具体命令 + Invoker，支持撤销
// ① 命令接口：所有命令长一个样
interface Command {
    void execute();
    void undo();   // 支持撤销
}

// ② 具体命令：持有 Receiver，把"做什么"包进来
class LightOnCommand implements Command {
    private final Light light;            // Receiver
    public LightOnCommand(Light l) { this.light = l; }
    public void execute() { light.on();  }
    public void undo()    { light.off(); }
}

// ③ Invoker：只认 Command 接口，不认识任何设备
class RemoteControl {
    private Command[] slots = new Command[7];
    private Command lastCommand;

    public void setCommand(int slot, Command c) { slots[slot] = c; }
    public void press(int slot) {
        slots[slot].execute();
        lastCommand = slots[slot];
    }
    public void pressUndo() { lastCommand.undo(); }  // 撤销上一步
}

现在遥控器彻底解耦：新增设备只需写一个新 Command 类，注入到某个槽位即可，遥控器一行不改。撤销也水到渠成——每个命令自己知道怎么"反向操作"。

结构与角色

角色职责本例
Command声明 execute()（及 undo()）接口Command 接口
ConcreteCommand持有 Receiver，把请求绑定到具体动作LightOnCommand
Receiver真正干活的对象，知道如何执行操作Light
Invoker持有命令并触发 execute()，不关心细节RemoteControl
Client创建命令、绑定 Receiver、配置到 Invoker装配代码

命令模式的延伸能力

  • 宏命令：一个命令内部持有命令数组，execute() 依次执行——"一键全开"
  • 队列：把命令放进线程池/任务队列，异步、串行执行（这正是 Runnable 的本质）
  • 日志与事务：把命令序列化到磁盘，崩溃后重放日志恢复状态（数据库 WAL 思想）
  • 撤销/重做：用栈保存命令历史，撤销弹栈调 undo()

JDK / Spring 中的真实应用

  • java.lang.Runnable 就是最纯粹的命令——run() 即 execute()，线程池把它排队执行，发起者和执行者完全解耦
  • java.util.concurrent.Callable、提交给 ExecutorService.submit() 的任务都是命令对象
  • Swing/AWT 的 Action 接口（AbstractAction）——按钮、菜单共享同一个命令对象
  • Spring 的 TransactionCallback、各类 ...Callback 回调本质都是命令
  • 消息队列（如 RabbitMQ/Kafka 的消费任务）、定时任务调度——命令模式在架构层的放大

常见陷阱

命令类爆炸：每个操作一个命令类，操作多了类会很多。简单场景可用 Lambda 替代——Command c = light::on;（命令接口是函数式接口时）。但要注意：Lambda 难以携带 undo() 状态，需要撤销时仍建议用完整类。
undo 的状态保存：撤销不总是"反向操作"那么简单。比如"设音量为 11"的撤销，需要先记住改之前的旧值，否则无法恢复。命令对象必须在 execute() 时保存足够的回滚状态。
别把命令模式当万能解耦：如果请求种类固定、不需要撤销/队列/日志，直接调用就好。强行套命令模式只会增加一层无意义的间接，让代码更难读。

易混模式对比：命令 vs 策略

两者都"把行为封装成对象注入"，结构相似，但意图完全不同：

对比项命令模式 Command策略模式 Strategy
关注点做什么（一个请求/动作）怎么做（一个算法的不同实现）
对象间关系解耦"发起者"与"执行者"让一个算法可在多个实现间替换
典型能力撤销、队列、日志、宏命令运行时切换算法，无撤销概念
是否持有 Receiver通常持有 Receiver 去调用它策略自己就是算法，不依赖额外 Receiver
类比餐厅订单导航软件选"最快/最短/避高速"路线
一句话区分：策略是"同一件事的不同做法"（多选一），命令是"把一件事打包成对象以便延迟/撤销/排队"（封装请求）。

与其他知识点的联系

  • 策略模式：结构上都是"把行为封装成对象注入使用方"，但意图不同——策略封装的是"同一件事的不同算法实现"（怎么做），命令封装的是"一个具体请求"（做什么），并因为多了这层对象化而额外获得撤销、排队、日志等能力，策略模式没有这些诉求。
  • 观察者模式：命令模式解耦的是"发起者"与"执行者"（Invoker 不知道 Receiver 是谁），观察者模式解耦的是"状态变化的一方"与"关心这个变化的多方"（Subject 不知道 Observer 是谁）——两者都靠一层接口抽象把双方隔开，但命令是一对一的委托链，观察者是一对多的广播。
  • 单一职责与开放关闭原则：把"触发动作"（Invoker）和"知道怎么做"（Receiver）拆成两个职责，新增一种设备只需新增一个 ConcreteCommand 类，RemoteControl 不用改——这正是 SRP 划清职责边界、OCP 靠新增而非修改来扩展的双重体现。
  • JDK Runnable/Callable 与线程池：Runnable.run() 本质就是没有 undo() 的最简命令对象，线程池 ExecutorService.submit() 接收命令对象、排队、异步执行，正是命令模式"队列化"能力在并发编程里最直接的落地，理解了命令模式再看线程池的任务提交机制会更清晰。
  • 数据库 WAL（预写日志）与事务：命令模式"把请求对象化后可以序列化、可以重放"这一特性，和数据库通过预写日志记录操作、崩溃后重放日志恢复数据是同一种思想——一旦一个动作变成了可存储的对象，它就获得了脱离当次调用现场、被重复利用的能力。

小结

  • 命令模式 = 把请求封装成对象，让 Invoker 与 Receiver 解耦
  • 五个角色：Command 接口、ConcreteCommand、Receiver、Invoker、Client
  • 请求一旦对象化，就能排队、记日志、撤销/重做、组合成宏命令
  • JDK 的 Runnable 是命令模式最经典的化身
  • 与策略模式结构像但意图不同：命令封装"请求"，策略封装"算法"
