适配器模式 Adapter

把一个类的接口转换成客户期望的另一个接口，让原本因接口不兼容而无法合作的类能一起工作。它是软件世界里的"电源转换插头"。

意图与定义

GoF 定义
将一个类的接口转换成客户希望的另外一个接口，使原本由于接口不兼容而不能一起工作的那些类可以一起工作。
Convert the interface of a class into another interface clients expect.

适配器夹在客户端和被适配对象之间：客户端只对着它熟悉的目标接口编程；适配器在内部把这些调用翻译成被适配对象能听懂的调用。客户端完全感知不到背后接口长得不一样。

解决什么问题，为何这么设计

现实中你常常拿到一个无法修改的类（第三方库、遗留代码、外部 SDK），它的接口和你系统约定的接口不一致。直接改它的源码不可能，让你系统里所有调用方都去适应它又会污染整个代码库。

适配器的设计精髓是：用一层包装隔离不兼容。你的系统继续面向自己的目标接口编程，适配器承担全部"翻译"工作。这样：① 不改第三方代码；② 不改客户端代码；③ 不兼容性被封装在一个类里，未来换第三方库只需换适配器。这正是开闭原则的体现——对扩展开放，对修改关闭。

生活类比：电源转换插头

你在中国买的笔记本（客户端）插头是两脚扁的，到了欧洲墙上的插座（被适配者）是两脚圆的。你不会改造笔记本，也改不了欧洲的电网，而是买一个转换插头（适配器）——一头接你的插头，一头插进墙。插头本身不发电，它只做"形状转换"。这就是适配器：不增加新功能，只做接口转换。

反例 → 问题 → 正例

场景：客户端要 Duck（quack/fly），手头只有 Turkey（gobble/短距 fly）

❌ 在客户端里到处 if 判断类型并翻译
void useDuck(Object bird) {
    if (bird instanceof Duck) {
        ((Duck) bird).quack();
    } else if (bird instanceof Turkey) {
        // 客户端被迫知道 Turkey 的存在和差异
        ((Turkey) bird).gobble();   // 用 gobble 凑合 quack
    }
    // 每来一种新鸟，客户端就多一个分支——越改越乱
}

问题：客户端被迫认识所有不兼容类型，到处是 instanceof 和翻译逻辑；新增一种鸟就要改客户端；翻译逻辑散落各处无法复用。

正例：对象适配器（组合，推荐）

适配器实现目标接口 Duck，内部持有被适配者 Turkey
interface Duck { void quack(); void fly(); }
interface Turkey { void gobble(); void fly(); }

// 适配器：对客户端"是"一只 Duck，对内"用"一只 Turkey
class TurkeyAdapter implements Duck {
    private final Turkey turkey;              // 组合：持有被适配者
    public TurkeyAdapter(Turkey t) { this.turkey = t; }

    public void quack() {
        turkey.gobble();                     // 把 quack 翻译成 gobble
    }
    public void fly() {
        for (int i = 0; i < 5; i++)     // 火鸡飞不远，连飞 5 次模拟
            turkey.fly();
    }
}

// 客户端：只面向 Duck 编程，对火鸡一无所知
Duck duck = new TurkeyAdapter(new WildTurkey());
duck.quack();  // 客户端照常调用，翻译被封装在适配器里

类适配器（继承，需要多继承能力）

同时继承被适配者、实现目标接口（Java 只能继承一个类，故较少用）
// 类适配器：通过继承获得被适配者的实现，再实现目标接口
class TurkeyClassAdapter extends WildTurkey implements Duck {
    public void quack() { gobble(); }   // 直接调继承来的方法
    // fly() 继承自 WildTurkey，无需重写
}

结构与角色

角色职责本例
Target（目标接口）客户端期望的接口Duck
Adaptee（被适配者）已存在、接口不兼容的类Turkey
Adapter（适配器）实现 Target，内部转调 AdapteeTurkeyAdapter
Client（客户端）只面向 Target 编程调用 duck.quack() 的代码

对象适配器 vs 类适配器

对比项对象适配器（组合）类适配器（继承）
实现方式持有被适配者实例继承被适配者类
能否适配子类能，运行时传任意实现只能适配指定的那个类
Java 可行性无限制（推荐）受单继承限制，且需 Target 是接口
灵活性高，可换被适配对象低，编译期绑定
覆盖被适配者行为不能（只能委托）能（可重写父类方法）
首选对象适配器：遵循"多用组合少用继承"，对象适配器更灵活、不受单继承限制，是 Java 中的主流选择。

JDK / Spring 中的真实应用

  • Arrays.asList(T...)：把数组适配成 List 接口——你能用 List 的 API 操作一个数组（注意：返回的是定长视图，不能 add）
  • InputStreamReader / OutputStreamWriter：把面向字节的 InputStream 适配成面向字符的 Reader，桥接字节流与字符流两套体系
  • java.util.Collections.enumeration()：把新式 Iterator 适配成老式 Enumeration（反向亦有 list()）
  • Spring MVC HandlerAdapter：把不同类型的 Handler（@Controller、Controller 接口、HttpRequestHandler）适配成统一调用方式，让 DispatcherServlet 不必关心 Handler 的具体类型
  • Spring AOP 的各种 AdvisorAdapter、各类日志门面（SLF4J 适配 log4j/logback）

常见陷阱

适配器不应添加新功能：适配器的职责是接口转换，不是增强功能。如果你在适配过程中加了新行为，那已经偏向装饰者模式了。保持适配器纯粹。
Arrays.asList 的坑：它返回的不是 ArrayList，而是 Arrays 内部的固定大小视图。调用 add()/remove() 会抛 UnsupportedOperationException。需要可变集合要 new ArrayList<>(Arrays.asList(...))。
不要过度包装：接口差异很小、或你能直接改被适配类时，没必要硬上适配器。多一层适配器就多一层间接和维护成本。适配器是为"无法修改的不兼容接口"准备的。

易混模式对比

模式意图是否改变接口是否加功能
适配器 Adapter转换接口，让不兼容者协作改变（A→B）不加
装饰者 Decorator动态增加职责不变（同接口）加
外观 Facade简化一组接口为一个新建简化接口不加（只组织）
代理 Proxy控制对对象的访问不变（同接口）控制访问，不改功能语义
一句话区分：适配器"改接口不改功能"；装饰者"不改接口加功能"；外观"把一堆接口收成一个简单接口"；代理"接口不变，控制访问"。四者结构相似，靠意图区分。

小结

  • 适配器 = 接口转换器，让接口不兼容的类能协作，是软件世界的"转换插头"
  • 四角色：Target、Adaptee、Adapter、Client；客户端只面向 Target 编程
  • 对象适配器（组合，推荐） vs 类适配器（继承，受单继承限制）
  • JDK 的 Arrays.asList、InputStreamReader，Spring MVC 的 HandlerAdapter 是典型
  • 记住意图：适配器只转换接口、不增加功能，这是它与装饰者的分界线
