抽象工厂模式 Abstract Factory

提供一个接口，用于创建一系列相关或相互依赖的对象，而无需指定它们的具体类——一次造出一整套"配套家族"。

意图与定义

GoF Abstract Factory Pattern
用一个工厂接口，成套地创建相互配套的产品族。
Provide an interface for creating families of related or dependent objects without specifying their concrete classes.

抽象工厂在工厂方法之上更进一步：工厂方法解决"造一个产品"，抽象工厂解决"造一整族相互搭配的产品"，并保证一次只用某一族的产品，不会混搭。它把"用哪一族"这件事集中到一个工厂对象上，客户只面向抽象工厂和抽象产品编程。

它解决什么问题

当系统需要在多套相互配套的产品族之间整体切换，且要保证同一时刻使用的产品来自同一族（不能混搭）时：

  • 若直接 new 各个具体产品，调用方既要知道每个产品的具体类，还容易把不同族的产品混用（如纽约面团配芝加哥酱料）
  • 切换整套产品族时，要改一大片创建代码

抽象工厂的设计：定义一个工厂接口，里面声明"创建本族每种产品"的一组方法；每个具体工厂负责生产一整族配套产品。客户拿到一个工厂，就能造出整套互相搭配的产品，且想换一族只需换一个工厂实现。

生活类比：家装风格套装

你装修选定"北欧风"（一个具体工厂），那么沙发、桌子、灯具都来自北欧风这一族，风格自然统一；要是改选"工业风"（换一个工厂），整套家具一起换成工业风。你不会从北欧风工厂买沙发、又从工业风工厂买灯——抽象工厂保证了一族产品的内部一致性。

反例 → 问题 → 正例（披萨原料家族）

书里的延伸案例：不同地区的披萨要用本地原料——纽约用 ThinCrustDough + MarinaraSauce，芝加哥用 ThickCrustDough + PlumTomatoSauce。这些原料构成"原料族"。

❌ 反例：在披萨里硬编码各地原料
class CheesePizza extends Pizza {
    void prepare() {
        // 原料散落各处、写死具体类，还可能混搭出错
        dough = new ThinCrustDough();
        sauce = new MarinaraSauce();
        cheese = new ReggianoCheese();
        // 想做芝加哥风味？得复制整段、改成另一批具体类
    }
}

问题在哪：

  • 原料的具体类散落在每个披萨类里，换地区要到处改
  • 无法保证一族原料的一致性——很容易不小心混用了别的地区原料
  • 新增一个地区，要改所有披萨类的 prepare()

正例：定义"原料工厂"接口，每个地区一个具体工厂，成套提供本地原料；披萨只依赖工厂接口。

抽象工厂：声明创建整族产品的方法
// 抽象工厂：一组创建方法，对应原料族里的每种产品
interface PizzaIngredientFactory {
    Dough  createDough();
    Sauce  createSauce();
    Cheese createCheese();
}

// 具体工厂：纽约这一族原料，整套配套
class NYIngredientFactory implements PizzaIngredientFactory {
    public Dough  createDough()  { return new ThinCrustDough();  }
    public Sauce  createSauce()  { return new MarinaraSauce();   }
    public Cheese createCheese() { return new ReggianoCheese();  }
}

// 换芝加哥族只需另一个工厂实现，产品自动配套
class ChicagoIngredientFactory implements PizzaIngredientFactory {
    public Dough  createDough()  { return new ThickCrustDough(); }
    public Sauce  createSauce()  { return new PlumTomatoSauce(); }
    public Cheese createCheese() { return new MozzarellaCheese(); }
}
客户：只依赖抽象工厂，整族原料一次到位
class CheesePizza extends Pizza {
    private PizzaIngredientFactory factory;   // 组合：持有一个工厂

    public CheesePizza(PizzaIngredientFactory factory) {
        this.factory = factory;
    }
    void prepare() {
        // 全程面向抽象，不知道是哪族原料，但保证整套来自同一工厂
        dough  = factory.createDough();
        sauce  = factory.createSauce();
        cheese = factory.createCheese();
    }
}

// 装配：换一族原料只换这一个工厂对象
Pizza p = new CheesePizza(new NYIngredientFactory());
关键价值：抽象工厂把"一族产品"作为一个整体来创建和切换，从根上杜绝了"混搭"。客户只认 PizzaIngredientFactory 接口，换族 = 换一个工厂实现，其余代码不动。

角色与结构

角色职责原料案例对应
AbstractFactory声明创建本族每种产品的一组方法PizzaIngredientFactory
ConcreteFactory生产某一族配套产品NYIngredientFactory / ChicagoIngredientFactory
AbstractProduct每类产品的抽象Dough / Sauce / Cheese
ConcreteProduct某族里的具体产品ThinCrustDough / MarinaraSauce 等
Client只依赖抽象工厂与抽象产品CheesePizza

JDK 与 Spring 中的真实应用

  • JDK DocumentBuilderFactory / TransformerFactory / SAXParserFactory：XML 解析体系里，工厂成套创建相互配套的解析器对象。
  • JDK GUI java.awt.Toolkit、各平台的 LookAndFeel：一个工厂为某个平台/外观主题成套创建窗口部件，保证风格统一——这是抽象工厂的典型动机（同一族 UI 组件）。
  • JDBC 驱动：一个具体驱动成套提供配套的 Connection、Statement、ResultSet 实现，同属一族不混用。
  • Spring BeanFactory：广义上是创建一族 Bean 的工厂抽象（虽然更偏容器，但承载了"用工厂屏蔽具体类创建"的同一思想）。

常见误区与陷阱

难以扩展"新产品种类"。抽象工厂对"新增产品族"友好（加一个工厂实现即可），但对"在产品族里新增一种产品"非常不友好——一旦给工厂接口加一个新方法（如 createVeggies()），所有已有具体工厂都要改。这是抽象工厂最大的硬伤。
不要为单一产品上抽象工厂。如果产品之间没有"配套、成族、不能混搭"的关系，只是单纯要造一个对象，用工厂方法或简单工厂就够了，套抽象工厂只会徒增一堆接口。

重点对比：工厂方法 vs 抽象工厂

这是最容易混淆的一对，务必从"造几个产品 + 用继承还是组合"两个维度区分：

对比维度工厂方法 Factory Method抽象工厂 Abstract Factory
创建什么一个产品一族相关、配套的产品
实现手段继承：子类覆盖工厂方法组合：客户持有一个工厂对象
核心方法数通常一个抽象工厂方法一组工厂方法（每种产品一个）
扩展产品族—加一个工厂实现，容易
扩展产品种类子类自己说了算要改工厂接口 + 所有实现，困难
关系抽象工厂的每个创建方法，内部常常就用工厂方法来实现。可以说抽象工厂是"一组工厂方法"的集合。
一句话记忆：工厂方法用继承造一个，抽象工厂用组合造一族。面试问区别时，先答这一句，再举披萨店（工厂方法）和披萨原料族（抽象工厂）的例子。

小结

  • 抽象工厂 = 一个工厂接口声明一组创建方法，每个具体工厂成套生产一族配套产品。
  • 它保证同族产品的一致性、杜绝混搭，切换整族只需换一个工厂实现。
  • 优点是易于扩展产品族，硬伤是难以在族里新增产品种类（要改所有工厂）。
  • 与工厂方法的区别：组合造一族 vs 继承造一个；抽象工厂常以工厂方法为内部实现。
