四个基础概念

抽象
Abstraction

忽略不重要的细节，只关注对象的本质特征和行为。定义"做什么"，隐藏"怎么做"。

封装
Encapsulation

把数据和操作数据的行为绑定在一起，对外只暴露必要的接口，隐藏内部实现。

继承
Inheritance

子类复用父类的属性和方法，并可以扩展或覆盖。建立"is-a"关系。

多态
Polymorphism

同一个接口，不同的对象有不同的实现。调用者不需要知道具体类型。

抽象：定义"做什么"

抽象是 OO 的出发点。在 Java 中，抽象通过 abstract class 和 interface 表达：

Java · 抽象的两种形式
// 接口：纯粹定义"能做什么"，不关心怎么做
interface Animal {
    void makeSound();   // 只有声明，没有实现
    void move();
}

// 抽象类：定义部分实现，留下一部分让子类完成
abstract class AbstractAnimal {
    public void breathe() {
        System.out.println("呼吸");   // 共同行为，已实现
    }
    public abstract void makeSound();  // 各自不同，留给子类
}

书中强调的关键点：抽象的价值在于它描述的是"概念"，而不是"实现"。 当你写 Animal animal 而不是 Dog dog 时，你的代码就获得了对变化的免疫力。

封装：隐藏实现，暴露接口

封装最常见的误解是"用 private 保护字段"。书中的视角更深：封装是隐藏决策，不是隐藏数据。

❌ 假封装：只是 private + getter
class Order {
  private List<Item> items;

  // 调用者拿到 List，内部结构泄露
  public List<Item> getItems() {
    return items;
  }
}
✅ 真封装：暴露行为，隐藏结构
class Order {
  private List<Item> items;

  // 调用者不知道内部是 List 还是 Set
  public int itemCount() {
    return items.size();
  }
  public Money totalPrice() { ... }
}

判断封装是否到位：如果修改内部实现（比如把 List 换成 Set），调用方需要跟着改吗？ 需要改，说明封装没做好。

继承：复用与扩展

继承建立"is-a"关系，子类可以复用父类行为，也可以通过覆盖改变行为：

Java · 继承与方法覆盖
class Duck {
    public void swim() {
        System.out.println("游泳");
    }
    public void quack() {
        System.out.println("呱呱叫");
    }
}

class RubberDuck extends Duck {
    @Override
    public void quack() {
        System.out.println("吱吱叫");   // 覆盖父类行为
    }
    // swim() 直接继承，不用重写
}
⚠ 书中的警告

Head First 第一章正是因为展示了继承的局限，才引出了设计原则。继承很容易被滥用——当子类越来越多、父类修改影响所有子类、子类需要覆盖大量方法时，继承反而成了负担。这也是为什么书中第一条原则就要讲"多用组合，少用继承"。

多态：同一接口，不同行为

多态让调用者可以面向抽象编程，运行时自动选择正确的实现。这是设计模式能够工作的基础机制：

Java · 多态的威力
interface FlyBehavior {
    void fly();
}

class FlyWithWings implements FlyBehavior {
    public void fly() { System.out.println("扑腾翅膀飞"); }
}

class FlyRocketPowered implements FlyBehavior {
    public void fly() { System.out.println("火箭驱动飞"); }
}

// 调用方只依赖接口，不关心具体类型
void letItFly(FlyBehavior behavior) {
    behavior.fly();   // 运行时决定调用哪个实现
}

// 使用时可以传入任意实现
letItFly(new FlyWithWings());
letItFly(new FlyRocketPowered());

这个例子来自书第一章鸭子案例。letItFly 方法不知道传进来的是哪种飞法——这正是多态的价值：新增一种飞法，调用方不需要修改。

四者的关系

概念 解决什么问题 在书中的作用
抽象 屏蔽细节，聚焦本质 接口和抽象类是所有设计模式的载体
封装 控制变化的扩散范围 "封装变化"原则的实现手段
继承 复用行为 书中主要作为反例——过度继承引发问题
多态 让调用方面向抽象，不面向实现 策略、观察者等模式的运行时基础

使用它们会导致什么问题

这四条是工具，工具用错了照样出问题。书第一章花大量篇幅展示的正是：只靠 OO Basics 写出来的代码，依然可能一团糟。

继承的典型问题：脆弱的基类

继承在父子类之间建立了强耦合。父类一改，所有子类都可能受影响——这叫"脆弱的基类问题"（Fragile Base Class Problem）。

继承导致的连锁反应
class Duck {
    public void fly() { ... }
    public void quack() { ... }
}

// 需求：加入橡皮鸭（不会飞）
class RubberDuck extends Duck {
    @Override
    public void fly() {
        // 只能覆盖成空方法——这是在撒谎
        // 调用者以为它能飞，其实什么都不做
    }
}

// 需求：再加入诱饵鸭（不会飞，也不会叫）
class DecoyDuck extends Duck {
    @Override public void fly() { }    // 又要覆盖
    @Override public void quack() { }  // 又要覆盖
}

// 每加一种鸭子，都要检查父类哪些方法需要覆盖
// 一旦父类新增方法，所有子类都受影响

封装的典型问题：过度设计

封装过头会导致接口爆炸——每个类都有一堆"暴露行为"的方法，调用方反而不知道该用哪个。同时，过度封装往往意味着过早抽象，在需求还不清晰时就做了错误的隐藏决策，后期反而更难改。

抽象的典型问题：抽象泄漏

抽象层定义了"做什么"，但有时底层细节会"渗透"到调用方——这叫抽象泄漏（Leaky Abstraction）。例如一个文件读取接口，调用方本不需要知道它是本地文件还是网络文件，但网络超时的异常却从接口里抛出来了，调用方不得不处理底层细节。

多态的典型问题：隐式逻辑难以追踪

多态让运行时行为难以静态分析。调用 animal.makeSound()，你得知道运行时传进来的是哪个子类，才知道实际执行什么。系统复杂时，"这里到底跑的是哪个实现"会成为调试的噩梦。

如何权衡

概念 用它的时机 该停下的信号
继承 子类和父类是真正的"is-a"关系，且父类行为子类完全适用 子类开始覆盖大量父类方法并留空；继承层超过 2 层
封装 内部实现有可能变化，调用方不需要知道细节 为了封装而封装，强行隐藏本不复杂的逻辑
抽象 有两种以上不同实现，或实现将来会变化 只有一个实现还在抽接口，YAGNI（你不需要它）
多态 消灭 if/else 分支，用类型系统区分行为 类型种类极少且固定，简单 if/else 反而更清晰
书中给出的实用判断：如果你发现自己在不断覆盖父类方法、或者子类行为和父类有本质差异，就应该停下来考虑用组合替代继承。这也是下一篇"封装变化"原则要解决的核心问题。

为什么这四条还不够

书中的核心论点：OO Basics 是工具，不是方法论。

只用这四条工具，完全可以写出这样的代码：

  • 继承层次太深，改父类牵一发动全身
  • 子类为了"不报错"覆盖方法但什么也不做（如橡皮鸭覆盖 fly() 留空）
  • 封装了字段，但内部逻辑仍然到处依赖具体实现
  • 多态用了，但新增类型时仍要修改大量 if/else
书中的结论：设计原则（OO Principles）告诉你如何使用这四个工具。设计模式（Design Patterns）是这些原则在反复出现的问题上的具体解法。

OO Basics → OO Principles → Design Patterns，三层递进，缺一不可。
