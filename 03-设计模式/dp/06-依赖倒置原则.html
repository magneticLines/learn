依赖倒置原则（DIP）

要依赖抽象，不要依赖具体类。让高层和底层都仰望同一个接口——这是工厂模式背后的灵魂。

原则定义

Head First 设计原则 #6
要依赖抽象，不要依赖具体类。
Depend upon abstractions. Do not depend upon concrete classes.

Head First 在工厂模式（Ch.4）一章引入 DIP。工厂模式之所以存在，正是为了把"创建具体对象"这件脏活从业务代码里剥离出去，从而让业务代码只依赖抽象——这就是 DIP 的直接落地。

"倒置"到底倒了什么

"倒置"（Inversion）这个词容易让人困惑。它指的是依赖方向的反转。先看没有 DIP 时的"自然"依赖方向：

PizzaStore（高层策略）
直接 new 各种具体 Pizza
↓ 依赖（向下指向具体类）
CheesePizza
ClamPizza
VeggiePizza

高层模块（业务策略）直接依赖底层模块（具体实现），箭头从上往下。问题是：底层一变，高层就得跟着改。底层细节本该是最易变的部分，却成了高层的主人。这是反的。

DIP 把箭头倒过来：在高层和底层之间插入一个抽象，让两者都依赖这个抽象。

PizzaStore（高层）
↓ 依赖      ↑ 依赖（实现）
抽象：interface Pizza
高层和底层都指向它
CheesePizza
ClamPizza
VeggiePizza

现在底层的具体类实现抽象（箭头朝上），高层使用抽象（箭头朝下）。底层不再是高层的主人，两者平等地服从于同一份契约。这就是"倒置"。

它解决什么问题、为什么这么设计

当高层代码里写满了 new ConcreteClass()，会发生几件坏事：

  • 无法替换实现：想换一个数据库、换一个支付渠道、在测试里换成 mock，都得改高层代码
  • 变化向上传染：底层细节最频繁变动，却把高层拖下水，违反 OCP
  • 无法单元测试：高层和真实的底层硬绑在一起，测高层就必须连数据库、连网络

DIP 的设计哲学是：稳定的东西不应该依赖易变的东西。高层策略（"披萨店怎么营业"）是稳定的，具体披萨种类是易变的。让稳定方依赖易变方是危险的；让两者都依赖一个稳定的抽象，系统就稳了。

生活类比：墙上的电源插座

你家的电器（高层：吹风机、台灯、充电器）从不直接焊死到电网（底层：发电厂的具体线路）。中间隔着一个标准插座（抽象）。电器只认插座规格，发电厂只需保证输出符合规格。于是你能随意换电器，电网也能改造升级，两边互不干扰——因为它们都依赖"插座标准"这个抽象，而不是彼此的具体细节。

反例 → 问题 → 正例

❌ 反例：PizzaStore 直接 new 具体披萨
class PizzaStore {
    Pizza orderPizza(String type) {
        Pizza pizza;
        // 高层业务里硬编码了所有具体类——依赖方向朝下
        if (type.equals("cheese"))  pizza = new CheesePizza();
        else if (type.equals("clam")) pizza = new ClamPizza();
        else if (type.equals("veggie")) pizza = new VeggiePizza();
        // 每加一种披萨，就要回来改这个高层方法
        pizza.prepare(); pizza.bake(); pizza.box();
        return pizza;
    }
}

问题：PizzaStore 这个高层类直接 new 了一堆具体披萨，依赖了所有它本不该关心的细节。加披萨要改它，删披萨要改它，它和底层焊死在一起。

用工厂模式把"创建"这件事抽出去，让高层只依赖抽象：

正例：工厂 + 抽象——PizzaStore 不再认识任何具体披萨
// 高层和底层共同依赖的抽象
interface Pizza { void prepare(); void bake(); void box(); }

// 创建的职责交给工厂，从高层业务里剥离出去
interface PizzaFactory { Pizza create(String type); }

class PizzaStore {
    private final PizzaFactory factory;  // 只依赖抽象
    PizzaStore(PizzaFactory f) { this.factory = f; }

    Pizza orderPizza(String type) {
        Pizza pizza = factory.create(type);  // 不知道也不在乎是哪个具体类
        pizza.prepare(); pizza.bake(); pizza.box();
        return pizza;
    }
}

// 新增披萨：只动工厂（底层），PizzaStore（高层）一行不改
对比关键点：反例里 PizzaStore 指向了 CheesePizza 等具体类；正例里它只指向 Pizza 和 PizzaFactory 两个抽象。依赖方向从"指向细节"变成"指向契约"，这就是依赖倒置。

三条可操作的实用准则

Head First 给出了三条帮助接近 DIP 的指导（注意：是"指导"不是"铁律"，目的是提醒，不是教条）：

准则 含义
变量不要持有具体类的引用 用工厂/依赖注入来避免直接 new，让变量声明为接口类型
不要让类派生自具体类 继承就从抽象类或接口派生，避免依赖某个具体实现
不要覆盖基类中已实现的方法 如果你要覆盖它，说明基类的这个实现不是一个真正适合被继承的抽象

与其他原则和模式的关系

原则 / 模式 与 DIP 的关系
针对接口编程（#2） 几乎是同一句话的不同说法；DIP 强调的是高层和底层的方向关系
开放关闭原则（Ch.3） DIP 提供的抽象层正是 OCP 得以成立的扩展点
工厂模式（Ch.4） DIP 的标准落地——把"创建具体对象"集中到工厂，业务只依赖抽象
控制反转 / 依赖注入（IoC / DI） DIP 是原则，DI 是实现这条原则的机制；Spring 容器就是大规模的依赖注入器
Spring 的 @Autowired 字段声明为接口类型，运行时由容器注入实现——工程界最普及的 DIP 实践
面试高频考点：DIP 和 DI 经常被混为一谈。记住：DIP（依赖倒置原则）是一条设计原则，讲"应该依赖抽象"；DI（依赖注入）是一种实现手段，讲"对象的依赖由外部传进来而不是自己 new"。IoC（控制反转）则是更宽泛的概念，DI 是 IoC 的一种形式。

常见误区与陷阱

误区一：为每个类都建一个接口。看到 UserService 就配一个只有它自己实现的 UserServiceImpl + IUserService，是对 DIP 的形式化误读。如果一个抽象永远只有一个实现、也不需要替换或 mock，这个接口就是纯噪音。DIP 针对的是真有多个实现或需要解耦的边界。
误区二：以为 DIP 就是"加个接口"。关键不在"有没有接口"，而在依赖方向。如果接口和它的实现都属于底层、由底层定义，高层依赖它仍然是在依赖底层。真正的倒置要求抽象由高层的需求来定义（高层说"我需要一个能存数据的东西"），底层去适配它。

小结

  • DIP：依赖抽象，不依赖具体类——高层和底层都仰望同一个接口
  • "倒置"倒的是依赖方向：从"高层指向具体细节"变成"两者都指向抽象"
  • 核心价值：让稳定的高层不被易变的底层牵着走，从而可替换、可测试、可扩展
  • 工厂模式是它的典型实现；DI/IoC 是它在框架层面的大规模落地
  • 别滥用——不需要多实现、不需要解耦的地方，强行加接口只是过度设计

与其他知识点的联系

  • 关联知识点：针对接口编程（原则 #2）：本文已经点出两者近乎同一句话的不同表述，但差别值得强调——针对接口编程只要求"调用方声明接口类型"，DIP 进一步要求这个接口不能是底层定义的（否则高层依赖的仍是底层的东西），而应该由高层的需求驱动产生。DIP 是针对接口编程的方向性精确化。
  • 关联知识点：开放关闭原则（OCP，Ch.3）：OCP 说"新增功能靠新增类，不改旧代码"，但这句话要成立，必须先有一个稳定的抽象点让新类去实现——这个抽象点从哪来？正是 DIP 要求"高层和底层都依赖抽象"提供的。没有 DIP 打好的地基，OCP 会退化成一堆 if/switch 里加分支。
  • 关联知识点：工厂模式（Factory Pattern，Ch.4）：本文把工厂模式当作 DIP 的标准落地：PizzaStore 不再认识任何具体披萨类，创建的职责被剥离到 PizzaFactory 里。工厂模式解决的正是"高层怎么在不认识具体类的前提下获得一个可用实例"这个 DIP 留下的实操问题。
  • 关联知识点：控制反转与依赖注入（IoC / DI）：本文的面试考点部分已经区分了 DIP（原则）和 DI（实现手段）的关系——DIP 说"要依赖抽象"，DI 说"依赖由外部传入而不是自己创建"。Spring 的 @Autowired 把这条原则系统化、自动化了：容器负责在运行时把抽象接口和具体实现连接起来。
  • 关联知识点：Clean Architecture 的依赖规则（Dependency Rule）：Clean Architecture 的核心图景就是 DIP 在整个系统架构上的放大——源码依赖必须指向内层（业务规则），而控制流可以指向外层，通过在边界处插入接口（如 Use Case 的 Port）实现依赖方向和控制流方向的分离。这正是本文"倒置"一词所指的箭头反转，只是发生在架构层而不是单个类之间。
