迭代器模式 Iterator

提供一种方法顺序访问聚合对象中的各个元素，而又不暴露它的内部表示。把"遍历"这件事从集合本身剥离出来，交给一个独立的迭代器对象。

意图 / 定义

GoF 定义
提供一种方法顺序访问一个聚合对象中的各个元素，而又不暴露其内部表示。
Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

迭代器模式的核心是把"怎么遍历"封装进一个迭代器对象。集合（聚合对象）只负责存数据，不负责告诉外界自己是用数组、链表还是哈希表存的；外界拿到一个统一的迭代器，用 hasNext() / next() 就能走完所有元素。

解决什么问题、为何这么设计

想象你要遍历两个不同来源的菜单：一个用 ArrayList 存菜品，另一个用数组存。如果客户端直接面对它们的内部结构，就要写两套遍历逻辑——一套 for-i 加 size()，一套 for-i 加 length。一旦再来第三个用 HashMap 存的菜单，客户端代码又要改。

这违反了两条原则：

  • 单一职责原则：集合类的职责是"管理数据"，"遍历"是另一项职责。把遍历状态（当前走到第几个）放进集合，会让集合因为"遍历方式变了"而被迫修改。迭代器把遍历职责单独抽出。
  • 封装变化：集合的内部数据结构是会变的（数组换链表），迭代器模式让这种变化不泄漏给客户端。

设计上的关键：让所有集合都返回一个实现了相同 Iterator 接口的对象，客户端只面向这个接口编程，于是遍历代码与具体集合彻底解耦。

生活类比：电视遥控器的"频道+/-"

你不需要知道电视内部频道是怎么存的（线性表？跳过空频道的链表？），你只要按"下一个频道"键，电视就帮你跳到下一个有效频道。遥控器的"下一个"按钮就是迭代器的 next()——你不关心内部表示，只要能依次访问。换一台电视，按钮还是那两个，遍历方式一致。

反例 → 问题 → 正例

❌ 反例：客户端硬编码两种遍历
// DinerMenu 用数组，PancakeMenu 用 List
MenuItem[] diner = dinerMenu.getItems();
for (int i = 0; i < diner.length; i++) {
    print(diner[i]);          // 数组式遍历
}

List<MenuItem> pan = pancakeMenu.getItems();
for (int i = 0; i < pan.size(); i++) {
    print(pan.get(i));        // List 式遍历
}
// 再来个 HashMap 菜单？再写第三套
✅ 正例：统一面向 Iterator
// 每个菜单都返回 Iterator
print(dinerMenu.createIterator());
print(pancakeMenu.createIterator());

void print(Iterator<MenuItem> it) {
    while (it.hasNext()) {    // 统一遍历
        print(it.next());
    }
}
// 新菜单只要提供 createIterator()
// 这段 print 一行都不用改

问题出在哪：反例里客户端"知道得太多"——它知道一个菜单是数组、另一个是 List，于是它和这两种数据结构紧紧绑死。正例里客户端只知道"有个迭代器能给我下一个元素"，集合换什么内部结构都与它无关。

给数组菜单实现一个 Iterator
// 统一的迭代器接口（这里直接用 JDK 的 java.util.Iterator）
class DinerMenuIterator implements Iterator<MenuItem> {
    private MenuItem[] items;
    private int position = 0;  // 遍历状态放在迭代器里，不放在菜单里

    public DinerMenuIterator(MenuItem[] items) { this.items = items; }

    @Override
    public boolean hasNext() {
        // 兼容数组中间有 null 的情况
        return position < items.length && items[position] != null;
    }

    @Override
    public MenuItem next() {
        return items[position++];
    }
}

class DinerMenu {
    private MenuItem[] items;
    // 集合只负责暴露一个迭代器，不暴露 items 本身
    public Iterator<MenuItem> createIterator() {
        return new DinerMenuIterator(items);
    }
}

角色 / 结构

角色职责本例对应
Iterator（迭代器接口）定义 hasNext() / next() 等遍历方法java.util.Iterator
ConcreteIterator（具体迭代器）实现遍历逻辑，持有遍历状态（position）DinerMenuIterator
Aggregate（聚合接口）定义 createIterator()，返回迭代器java.lang.Iterable
ConcreteAggregate（具体聚合）实现聚合接口，知道如何创建自己的迭代器DinerMenu / ArrayList
Client只面向 Iterator 接口遍历那个 print(Iterator) 方法

JDK / 框架中的真实应用

迭代器模式是 JDK 集合框架的基石。java.util.Iterable 是聚合接口，java.util.Iterator 是迭代器接口，几乎所有集合都实现了它们。

for-each 的真相：语法糖背后就是迭代器

你写的增强 for 循环，编译器会把它翻译成基于迭代器的循环。这正是为什么任何实现了 Iterable 的类都能用 for-each。

for-each 编译前后对照
// 你写的
for (MenuItem item : menu) {
    print(item);
}

// 编译器实际生成的（要求 menu 是 Iterable）
Iterator<MenuItem> it = menu.iterator();
while (it.hasNext()) {
    MenuItem item = it.next();
    print(item);
}
  • Iterator.remove()：迭代过程中安全删除元素的唯一正确途径。
  • ListIterator：双向迭代器，支持 previous() 和遍历中 add()。
  • Spring：CompositeIterator、各类 *Iterator 工具类；MyBatis 的 Cursor 也是迭代器思想（流式读结果集）。

常见陷阱

fail-fast 与 ConcurrentModificationException：用 for-each（即迭代器）遍历 ArrayList 时，不能通过集合自身的 list.remove(x) 删除元素，否则抛 ConcurrentModificationException。

原理：集合内部有个 modCount 记录结构修改次数，迭代器创建时记下一个快照 expectedModCount。每次 next() 都校验两者是否相等；用 list.remove() 会改 modCount 但不改 expectedModCount，于是下次 next() 校验失败立刻抛异常。这叫 fail-fast——尽早暴露并发/误用问题，而不是默默产生错误结果。

❌ 触发 fail-fast
for (MenuItem m : menu) {
    if (m.isSpicy())
        menu.remove(m);   // 改了 modCount
}                          // 下次 next() 爆炸
✅ 用迭代器自己的 remove
Iterator<MenuItem> it = menu.iterator();
while (it.hasNext()) {
    if (it.next().isSpicy())
        it.remove();      // 同步更新两个 count
}
另一个坑：fail-fast 不保证一定能检测到所有并发修改——它只是"尽力而为"的快速失败机制，不能用它来做线程安全保证。真要并发遍历用 CopyOnWriteArrayList（fail-safe，基于副本迭代）。

面试高频："for-each 底层是什么？" → 迭代器模式，编译器翻译成 Iterator 循环，要求类实现 Iterable。"为什么遍历时不能删？" → fail-fast，modCount != expectedModCount。

小结

  • 迭代器模式把遍历职责从集合中抽离，封装进独立的迭代器对象，遍历状态由迭代器持有。
  • 客户端只面向 Iterator 接口，与集合内部结构（数组/链表/哈希）彻底解耦。
  • JDK 的 Iterable + Iterator 是其标准实现；for-each 就是它的语法糖。
  • 遍历中修改集合会触发 fail-fast（ConcurrentModificationException），删除要用迭代器自己的 remove()。

与其他知识点的联系

  • 组合模式（下一篇）：迭代器和组合是天然搭档——组合模式把数据组织成树形结构，遍历这棵树最自然的方式就是给它配一个迭代器（甚至是能"拉平"多层嵌套的迭代器）。学完迭代器再学组合，能更容易理解"统一遍历接口"和"统一操作接口"是如何配合工作的。
  • 单一职责原则：迭代器模式是"拆分职责"思想的直接应用——集合的职责是"管理数据"，遍历的职责被剥离到独立的迭代器对象里。这与本模块前面讲到的"一个类只负责一件事"是同一条原则在不同场景下的体现。
  • 访问者模式：两者都常用于遍历一组元素，但迭代器只负责"按顺序拿到元素"，不关心拿到后做什么；访问者则是在遍历过程中对每个元素施加不同的操作（通过双分派）。实际实现里，访问者的遍历逻辑内部往往就依赖迭代器。
  • JDK Iterable/Iterator 与 for-each 语法糖：正文已详细讲过——任何实现 Iterable 的类都能用 for-each，这是编译器在背后把语法糖翻译成迭代器循环。理解这一点后，再看 Stream API 的 forEach、Spliterator（并行拆分版的迭代器）会更容易上手，它们是迭代器思想在新 API 里的延伸。
  • fail-fast 与并发容器：本文讲的 ConcurrentModificationException 直接关联到并发编程模块——CopyOnWriteArrayList 之所以能安全地被多线程遍历，正是因为它的迭代器基于"创建时的数组快照"（fail-safe），与 ArrayList 迭代器的 fail-fast 机制形成鲜明对比，是理解并发容器设计取舍的一个重要切入点。
