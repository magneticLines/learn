代理模式 Proxy

为另一个对象提供一个替身或占位符，以控制对这个对象的访问。客户看似在和真实对象打交道，实际经过了一层代理，代理可以决定是否、如何把请求转给真身。

意图 / 定义

GoF 定义
为其他对象提供一种代理以控制对这个对象的访问。
Provide a surrogate or placeholder for another object to control access to it.

代理和真身实现同一个接口，所以客户端用起来感觉不到区别。区别在于：代理在转发请求前后，可以插入"控制"逻辑——延迟创建、权限校验、远程通信、缓存、日志……

解决什么问题、为何这么设计

有时我们不希望客户直接访问真实对象，原因可能是：

  • 对象创建昂贵（如加载一张大图）→ 用虚拟代理延迟到真正需要时才创建（懒加载）。
  • 对象在远程（另一台 JVM/机器）→ 用远程代理把本地调用变成网络调用（RMI）。
  • 访问需要受控（权限/限流）→ 用保护代理在转发前做校验。
  • 想在不改源码的前提下增强行为（事务、日志、缓存）→ 这正是 Spring AOP 的根基。

为什么用"同接口的替身"而不是直接改真实类？因为这样能把控制逻辑与业务逻辑分离，且对客户端完全透明——客户拿到的是接口，给它真身还是代理它无从知晓，符合"针对接口编程"。

生活类比：明星的经纪人

你想找明星合作，但联系不到明星本人，你联系的是经纪人（代理）。经纪人对外提供和明星一样的"接口"（谈合作、定档期），但他会先做控制：审核你的资质（保护代理）、明星没空就先记下需求（延迟）、转达时还能附带条件。你以为在和明星沟通，其实始终隔着经纪人这层代理。

反例 → 问题 → 正例

❌ 反例：把日志/计时硬塞进业务方法
class OrderServiceImpl implements OrderService {
    public void createOrder(Order o) {
        long t = System.currentTimeMillis();      // 计时——非业务
        log.info("createOrder start");            // 日志——非业务
        // ... 真正的下单逻辑 ...
        log.info("cost " + (System.currentTimeMillis() - t));
    }
    // 每个方法都复制这套样板，业务被横切关注点淹没
}

问题：日志、计时这类"横切关注点"和业务逻辑混在一起，每个方法都要复制，改一处要改 N 处。代理模式把它们抽到代理里。

静态代理：手写一个同接口的替身

✅ 正例：静态代理把横切逻辑抽出来
interface OrderService { void createOrder(Order o); }

// 代理与真身实现同一接口
class OrderServiceProxy implements OrderService {
    private OrderService target;  // 持有真身
    public OrderServiceProxy(OrderService t) { this.target = t; }

    public void createOrder(Order o) {
        long t = System.currentTimeMillis();  // 前置增强
        target.createOrder(o);                  // 转发给真身
        log.info("cost " + (System.currentTimeMillis() - t)); // 后置增强
    }
}

静态代理缺点：每个接口都要手写一个代理类，N 个接口就 N 个代理，且每加一个方法都要在代理里同步加。于是有了动态代理——运行时自动生成代理类。

JDK 动态代理：运行时生成代理（要求目标有接口）

JDK 动态代理：Proxy + InvocationHandler
OrderService proxy = (OrderService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),  // 关键：必须有接口
    (p, method, args) -> {          // InvocationHandler
        long t = System.currentTimeMillis();
        Object r = method.invoke(target, args);  // 反射转发到真身
        log.info("cost " + (System.currentTimeMillis() - t));
        return r;
    });

CGLIB：当目标没有接口时，JDK 动态代理用不了，CGLIB 通过生成目标类的子类（字节码增强）来代理，重写父类方法插入增强。因为靠继承，所以 final 类/方法无法被 CGLIB 代理。

角色 / 结构

角色职责本例对应
Subject（抽象主题）真身和代理共同实现的接口OrderService
RealSubject（真实主题）真正干活的对象OrderServiceImpl
Proxy（代理）持有真身引用，控制访问、前后增强后转发OrderServiceProxy / JDK 生成的 $Proxy0

JDK / Spring 中的真实应用

代理模式是 Spring AOP 的底层实现。@Transactional、@Cacheable、@Async 这些注解之所以"加个注解就生效"，是因为 Spring 在运行时给你的 Bean 生成了一个代理：

  • 目标 实现了接口 → 默认用 JDK 动态代理。
  • 目标 没有接口（或配置 proxyTargetClass=true）→ 用 CGLIB 生成子类代理。（Spring Boot 2.x 起默认就用 CGLIB。）
  • 你注入的 OrderService 其实是个代理，调方法时它先开事务/查缓存，再转发给真身。

其他：RMI 的 Stub 是远程代理；Hibernate/MyBatis 的延迟加载（lazy loading）用虚拟代理；java.lang.reflect.Proxy 是 JDK 内建的动态代理工厂。

常见陷阱

Spring AOP 自调用失效（最高频面试题）：同一个类里方法 A 直接调用本类方法 B（this.B()），B 上的 @Transactional / @Cacheable 不会生效。因为 this 指向真身而非代理，绕过了代理层。解决：通过注入自身代理调用，或拆分到另一个 Bean。
CGLIB 不能代理 final：CGLIB 靠生成子类重写方法实现增强，final 类无法被继承、final 方法无法被重写，所以代理对它们无效（增强静默失效，比报错更坑）。
JDK 动态代理只能代理接口方法：它生成的代理实现的是接口，目标类上"没在接口里声明"的 public 方法无法被代理调用到。需要代理类的全部方法时改用 CGLIB。

易混模式对比：代理 vs 装饰者 vs 适配器

三者都"包了一层"、都持有被包对象、外观结构相似，但意图截然不同，这是设计模式里最容易混的一组。

对比项代理 Proxy装饰者 Decorator适配器 Adapter
意图控制对对象的访问增加对象的功能/职责转换接口使不兼容者能合作
对接口的处理接口不变（和真身一致）接口不变（和被装饰者一致）接口改变（A 接口 → B 接口）
是否改变行为一般不改业务，只加控制主动增强、叠加新行为不增不减，只做接口翻译
是否可多层叠加通常一层设计上就是多层链式包裹通常一层
谁创建被包对象代理常自己控制真身的创建由客户端传入要装饰的对象由客户端传入要适配的对象
一句话区分：装饰者关心"给你更多"，代理关心"能不能、要不要给你"，适配器关心"把插头换成另一种形状"。三者代码长得像，意图各管一摊。

小结

  • 代理模式为对象提供一个同接口的替身，以控制访问（延迟、远程、保护、增强）。
  • 静态代理手写、一对一；JDK 动态代理运行时生成、要求接口；CGLIB 生成子类、无需接口但不能代理 final。
  • 它是 Spring AOP（事务/缓存/异步）的底层机制；注意自调用导致增强失效这一经典陷阱。
  • 与装饰者、适配器结构相似但意图不同：代理控制访问、装饰者增加功能、适配器转换接口。
