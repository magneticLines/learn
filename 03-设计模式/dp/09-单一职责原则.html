单一职责原则（SRP）

一个类应该只有一个引起它变化的原因。把不同的变化原因分到不同的类里，才能让每个类高内聚、易维护。

原则定义

Head First 设计原则 #9
一个类应该只有一个引起变化的原因。
A class should have only one reason to change.

Head First 在迭代器与组合模式（Ch.9）一章引入它。讲迭代器时的关键问题是：一个集合类，既要管理元素的存储，又要管理"如何遍历"，这就给了它两个变化的理由。把"遍历"职责抽到独立的 Iterator 里，正是 SRP 的实践。

关键：什么叫"一个引起变化的原因"

这条原则最常被误读成"一个类只做一件事"。更准确的说法是：一个类只应对一类使用者/一个变化来源负责。判断的方法不是看它有几个方法，而是问：会有几种不同的理由让我来改这个类？

如果一个 Employee 类里既有计算工资的逻辑（财务部门要的）、又有生成报表的逻辑（HR 要的）、还有持久化到数据库的逻辑（DBA 要的），那它就有三个变化原因——三个部门任何一个改需求，都会来动这个类。这三类关注点应该拆开。

SRP 的另一面：内聚（Cohesion）

SRP 和"高内聚"是一枚硬币的两面。内聚衡量的是一个类的各部分有多紧密地服务于同一个目的：

低内聚
一个类干杂活
多个变化原因
"上帝类"
中等内聚
职责有点混
方法不全相关
偶尔被多方触动
高内聚 / SRP
围绕单一职责
一个变化原因
方法彼此呼应

高内聚的类，方法都围绕同一职责协作，目的清晰、易于命名、易于测试；低内聚的类像个杂物抽屉，什么都往里塞，谁来都能找到点东西改，于是它永远在变、永远不稳定。

它解决什么问题

当一个类承担多个职责，几个坏处随之而来：

  • 变化互相牵连：改报表格式时不小心碰坏了工资计算——它们本不相关，却被关在同一个类里
  • 难以复用：你只想用它的工资计算逻辑，却被迫把报表、持久化一并拖进来
  • 测试爆炸：一个类做太多事，测试要 mock 一大堆无关依赖
  • 合并冲突：财务、HR、DBA 三拨人改同一个文件，天天冲突

生活类比：瑞士军刀 vs 专业工具

瑞士军刀什么都能干一点，应急方便，但你不会用它的小锯子去锯真正的木头。专业的木工锯只做一件事，却做得极好、也极好维修——锯齿钝了换锯条，不影响刀身。一个类如果是"瑞士军刀"，任何一项功能要升级都得动整把刀；拆成专业工具后，每个职责独立演化、互不干扰。

反例 → 问题 → 正例

❌ 反例：一个集合类既管存储又管遍历
class Menu {
    private MenuItem[] items;          // 职责1：存储元素

    // 职责2：把"如何遍历"也焊死在这里
    // 用数组下标遍历——如果改用 List/Map，遍历方式就得跟着改
    public int getCount() { ... }
    public MenuItem getItemAt(int i) { ... }
    // 客户端被迫知道 Menu 内部是数组，写出 for(i=0; i

问题：Menu 同时为"存储"和"遍历"两件事负责，且把存储结构泄露给了客户端。把"遍历"职责抽成独立的 Iterator：

正例：迭代器模式——遍历职责从集合中分离出去
// 遍历职责独立成一个抽象，客户端只依赖它
interface Iterator<T> {
    boolean hasNext();
    T next();
}

class Menu {
    private List<MenuItem> items = new ArrayList<>();

    // Menu 只负责一件事：管理元素。遍历交给 iterator
    public Iterator<MenuItem> createIterator() {
        return items.iterator();  // 内部结构怎么变都不暴露
    }
}

// 客户端：不知道也不关心 Menu 底层是数组还是 List
Iterator<MenuItem> it = menu.createIterator();
while (it.hasNext()) print(it.next());
职责分离后："存储结构"变了——只改 Menu 内部，客户端无感；"遍历方式"变了（倒序、过滤）——只换一个 Iterator 实现，Menu 不动。两个变化原因被关进了各自独立的盒子。Java 集合框架的 Iterable/Iterator 就是这么设计的。

一个快速自测：描述这个类

有个朴素而有效的检验方法：用一句话描述这个类的职责。如果你的描述里出现了"并且""以及""同时"，往往就是多职责的信号：

  • "这个类解析配置" → 单一职责，OK
  • "这个类解析配置，并且把结果写入数据库，同时发送通知邮件" → 三个职责，该拆

与其他原则和模式的关系

原则 / 模式 关系
高内聚低耦合 SRP 是"高内聚"的具体落地——一个变化原因 = 高度聚焦的单一职责
开放关闭原则（Ch.3） 职责拆清后，每个职责才能独立地"对扩展开放、对修改关闭"
迭代器模式（Ch.9） SRP 的代表落地：把"遍历"职责从集合中分离出来
组合模式（Ch.9） 同章出现；与迭代器配合时也需注意各类职责边界，避免组件类承担过多
Spring 分层架构 Controller / Service / Repository 的分层，本质就是按变化原因拆分职责
面试高频考点：SOLID 里的 S 就是 SRP。回答时务必强调"一个引起变化的原因"而不是"只做一件事"——前者才是 Robert C. Martin 的原意。再补一句"职责的边界由谁会要求改它决定"，就能体现你理解到了演化层面。

常见误区与陷阱

误区一：拆得过细，制造碎片。把 SRP 推到极端，给每个方法都建一个类，会产生大量只有一个方法的"贫血类"，调用关系四处飞，可读性反而崩溃。SRP 的目标是把会因不同原因变化的部分分开，不是把所有东西切成最小颗粒。判断标准始终是"变化原因"，不是"代码行数"。
误区二：误以为"只做一件事"。一个类可以有很多方法、做很多步骤，只要它们都服务于同一个职责、只对同一类变化负责，就符合 SRP。Iterator 有 hasNext() 和 next() 两个方法，但它们共同服务"遍历"这一个职责，完全合规。
务实判断：SRP 是事后识别比事前预测更可靠的原则。一开始不必过度拆分，等到一个类真的因为两类不同需求被反复修改、产生痛感时，再按"变化原因"切开，往往最准。

小结

  • SRP：一个类只应有一个引起它变化的原因——不是"只做一件事"
  • 它是"高内聚"的落地：类的各部分都围绕同一职责协作
  • 判断方法：用一句话描述类的职责，出现"并且/同时"就该拆
  • 迭代器模式是代表落地：把"遍历"从"存储"中分离，各自独立演化
  • 别拆过头——以"变化原因"而非"代码行数"为准；事后按痛感切分更可靠
