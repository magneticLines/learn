单例模式 Singleton

确保一个类只有一个实例，并提供一个全局访问点。听起来最简单，却是最容易写错的模式——线程安全、序列化、反射、类加载，每一关都有坑。

意图与定义

GoF 定义
确保一个类只有一个实例，并提供一个访问它的全局访问点。
Ensure a class only has one instance, and provide a global point of access to it.

单例模式把"只能有一个实例"这件事的责任，从使用方收回到类自身手里。它做两件事：① 控制实例数量为 1；② 暴露统一的获取入口（通常是静态方法 getInstance()）。

解决什么问题，为何这么设计

有些对象天生就该只有一个：线程池、缓存、配置对象、日志对象、注册表、设备驱动。如果不加约束，让任何人都能 new 出多个，会导致：

  • 资源浪费——例如每次 new ConnectionPool() 都开一组数据库连接
  • 状态不一致——两份缓存、两份配置，谁是权威？
  • 行为冲突——两个打印机后台程序同时往同一台打印机送任务

那为什么不直接用全局变量？因为全局变量在程序启动时就被创建，哪怕从没用到也占着内存（无法延迟加载）；而且全局变量无法阻止别人再 new 一个。单例的关键设计是：把构造器设为 private，从语法层面堵死外部 new 的路，再由类自己掌控唯一实例的创建时机。

生活类比：一个国家的政府

一个国家在同一时刻只能有一个合法的中央政府。你不能"再成立一个"——宪法（private 构造器）禁止了。无论从哪个部门、哪个公民去找"政府"，找到的都是同一个。这就是单例：唯一性由系统规则保证，访问入口全局统一。

反例 → 问题 → 正例

反例：天真的懒汉式（线程不安全）

❌ 多线程下可能创建出多个实例
class Singleton {
    private static Singleton instance;
    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {        // 线程 A、B 同时进到这里
            instance = new Singleton(); // 都判断为 null，各自 new 一个
        }
        return instance;
    }
}

问题：if (instance == null) 到 instance = new ... 不是原子操作。线程 A 刚判断完 null、还没赋值，线程 B 也进来判断为 null——结果创建了两个实例，单例失效。

过渡方案：synchronized 整个方法（正确但慢）

加锁能保证安全，但每次 getInstance 都要抢锁
public static synchronized Singleton getInstance() {
    if (instance == null) instance = new Singleton();
    return instance;
}
// 实例只创建一次，但后续每次读取都付出加锁开销——高并发下成瓶颈

正例一：双重检查锁（DCL）+ volatile

✅ 只有第一次创建才加锁，volatile 不能省
class Singleton {
    // volatile 必须有！防止指令重排序导致拿到半成品对象
    private static volatile Singleton instance;
    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                 // 第一次检查：无锁快速路径
            synchronized (Singleton.class) {
                if (instance == null) {         // 第二次检查：拿到锁后再确认
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
为什么 volatile 不能省？instance = new Singleton() 不是一步，它分三步：
① 分配内存 → ② 调用构造器初始化 → ③ 把引用指向内存。JVM 允许把 ②③ 重排序。若执行顺序变成 ①③②，线程 A 刚做完 ③（instance 已非 null）还没做 ②，此时线程 B 在第一次检查时看到 instance 非 null，直接返回了一个构造未完成的半成品对象。volatile 禁止这种重排序，并保证可见性。

正例二：静态内部类（推荐，优雅且高效）

利用类加载机制实现懒加载 + 线程安全，无需任何锁
class Singleton {
    private Singleton() {}

    // 内部类不会随外部类加载而加载，只有首次调用 getInstance 时才加载
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // 触发 Holder 加载
    }
}

JVM 保证类的初始化（<clinit>）是线程安全的（由类加载锁保证），且 Holder 只在被引用时才加载——既是懒加载，又天然线程安全，还没有 DCL 的锁开销。这是大多数场景的首选。

正例三：枚举单例（最安全，Effective Java 推荐）

天生防反射、防序列化破坏，一行搞定
public enum Singleton {
    INSTANCE;
    public void doWork() { /* 业务方法 */ }
}
// 调用：Singleton.INSTANCE.doWork();

结构与角色

角色职责
private 构造器禁止外部 new，是单例成立的语法前提
静态实例字段持有那个唯一的实例引用
静态获取方法 getInstance()全局访问点，负责创建（或返回已创建的）实例

五种写法横向对比

写法懒加载线程安全抗反射/序列化评价
饿汉式否（类加载即创建）是否简单，但可能创建了用不上的实例
懒汉式（不加锁）是否否错误写法，禁用

枚举否是是最安全，需要绝对单例时首选

JDK / Spring 中的真实应用

  • java.lang.Runtime——经典饿汉式单例，Runtime.getRuntime() 返回 JVM 唯一的运行时对象
  • java.awt.Desktop.getDesktop()、System.getSecurityManager()——JDK 中的单例访问点
  • Spring Bean 默认作用域是 singleton——但注意它是"容器内单例"，由 IoC 容器管理，不是 GoF 的 JVM 级单例（同一容器一个实例，不靠 private 构造器，而是靠容器缓存）
  • 日志框架的 LoggerFactory、连接池等基础设施大量使用单例思想
Spring 单例 vs GoF 单例：面试常考。Spring 的 singleton 是"每个 BeanFactory/容器一个实例"，作用域是容器；GoF 单例是"每个 ClassLoader 一个实例"，作用域是 JVM（类加载器）。两者不是一回事。

常见陷阱

序列化破坏单例：普通单例类如果实现了 Serializable，每次反序列化都会通过反射创建新对象，单例失效。解决：提供 private Object readResolve() { return instance; } 方法，让反序列化返回已有实例。枚举单例则天生免疫。
反射破坏单例：通过 constructor.setAccessible(true) 可以强行调用 private 构造器创建新实例。防御办法：在构造器里判断"已存在实例就抛异常"。枚举单例从语言层面禁止反射 newInstance，最彻底。
多个 ClassLoader 破坏单例："单例"的唯一性只在同一个类加载器范围内成立。Web 容器、OSGi 等环境下，同一个类被不同 ClassLoader 各加载一次，就会有多个"单例"。这也是为什么严格说单例是"每 ClassLoader 一个"。
单例是反模式吗？单例引入全局状态和隐藏依赖，让单元测试难以 mock、对象间耦合上升。现代实践更倾向用依赖注入把"单例"交给容器管理，而不是在类内部硬编码 getInstance()。能用 DI 时优先 DI。

小结

  • 单例 = private 构造器 + 静态实例 + 静态获取方法，目标是"全局唯一 + 统一入口"
  • 线程安全是核心难点：天真懒汉式错误；推荐静态内部类（优雅）或枚举（最安全）
  • DCL 必须配 volatile，否则指令重排序会暴露半成品对象
  • 注意三大破坏方式：序列化、反射、多 ClassLoader——枚举对前两者天生免疫
  • 能用依赖注入就别硬编码单例；单例的全局状态会增加耦合、妨碍测试
