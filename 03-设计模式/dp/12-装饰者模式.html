装饰者模式 Decorator

动态地给一个对象添加额外职责，就增加功能而言，装饰者比继承更有弹性——可以层层包裹，运行时自由组合。

意图与定义

GoF Decorator Pattern
用"包裹"代替"继承"，运行时动态叠加职责。
Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing.

装饰者模式体现了一条重要原则：类应该对扩展开放，对修改关闭（OCP）。它让你在不改动原有类、也不靠继承爆炸的前提下，给对象一层层"穿衣服"地加功能。装饰者与被装饰者实现同一接口，所以装饰后的对象能无缝替换原对象。

它解决什么问题

当一个对象需要"可选的、可叠加的"附加功能，且组合数量庞大时，继承会失控：

  • 类爆炸：N 种基础对象 × M 种附加功能的所有组合，要写出 N×M 个子类
  • 静态、无法运行时改变：继承在编译期就定死，运行时不能临时加/减一个功能

装饰者的设计思路：让装饰者持有一个被装饰对象的引用（组合），并和它实现同一接口。装饰者在调用内层对象的方法前后，加上自己的行为。多个装饰者可以嵌套，像俄罗斯套娃一样把功能一层层叠加上去。

生活类比：穿衣服

你（被装饰对象）出门，可以套一件毛衣（装饰者），再套一件外套（再一个装饰者），下雨再披雨衣。每件衣服都"包裹"住里面的你，且每件都让你"还是个能走路的人"（同一接口）。要加要减随时调整，不需要为"穿毛衣的你""穿毛衣加外套的你"各造一个新人——这就是动态叠加职责。

反例 → 问题 → 正例（咖啡案例）

书里的案例：咖啡店要给饮料加各种调料（牛奶、摩卡、豆浆、奶泡），并算出总价。最初想用继承。

❌ 反例：每种组合一个子类
class Espresso extends Beverage { ... }
class EspressoWithMilk extends Beverage { ... }
class EspressoWithMilkAndMocha extends Beverage { ... }
class HouseBlendWithMilk extends Beverage { ... }
class HouseBlendWithMochaAndWhip extends Beverage { ... }
// ... 调料 × 饮料的每种组合都要一个类 → 类爆炸

问题在哪：

  • 调料种类一多，子类数量呈组合爆炸，根本写不完
  • "双份摩卡"这种重复调料无法表达；运行时也不能临时加一份
  • 调料涨价，要追着改一大批子类——违反 OCP

正例：把饮料作为被装饰对象，每种调料做成装饰者，包裹饮料并在 cost() 上叠加价格。

组件接口 + 具体组件 + 装饰者基类
// 组件：饮料与调料共同的抽象
abstract class Beverage {
    public abstract String getDescription();
    public abstract double cost();
}

// 具体组件：被装饰的本体
class Espresso extends Beverage {
    public String getDescription() { return "浓缩咖啡"; }
    public double cost() { return 1.99; }
}

// 装饰者基类：同样 IS-A Beverage，且 HAS-A 一个被包裹的 Beverage
abstract class CondimentDecorator extends Beverage {
    protected Beverage beverage;   // 关键：组合，持有内层对象
}
具体装饰者：包裹内层并叠加自己的行为
class Mocha extends CondimentDecorator {
    public Mocha(Beverage beverage) { this.beverage = beverage; }

    public String getDescription() {
        return beverage.getDescription() + " + 摩卡";   // 先委托内层，再加自己
    }
    public double cost() {
        return beverage.cost() + 0.20;       // 在内层价格上叠加
    }
}

// 使用：层层包裹，运行时自由组合，要双份摩卡就包两次
Beverage b = new Espresso();
b = new Mocha(b);          // 加摩卡
b = new Mocha(b);          // 再加一份摩卡
b = new Whip(b);           // 加奶泡
System.out.println(b.getDescription() + " $" + b.cost());
// 输出：浓缩咖啡 + 摩卡 + 摩卡 + 奶泡 $2.49
核心机制：装饰者和被装饰者是同一类型（都是 Beverage），所以装饰者能替任何 Beverage 出现，也能被另一个装饰者继续包。调用时从最外层一路委托到最内层，再逐层叠加返回——这就是"职责的链式累加"。

角色与结构

角色职责咖啡案例对应
Component（抽象组件）定义被装饰对象与装饰者的公共接口Beverage
ConcreteComponent被装饰的本体Espresso / HouseBlend
Decorator（抽象装饰者）实现组件接口，并持有一个组件引用CondimentDecorator
ConcreteDecorator在委托内层的基础上叠加新职责Mocha / Whip

JDK 与 Spring 中的真实应用

java.io 流是装饰者模式最经典、最教科书的应用，整个 IO 体系就是层层装饰：

java.io：一层层装饰原始流
InputStream in =
    new BufferedInputStream(          // 装饰：加缓冲
        new FileInputStream("a.txt")  // 具体组件：原始文件流
    );
// FileInputStream 是 ConcreteComponent
// BufferedInputStream / DataInputStream 是 ConcreteDecorator
// 它们都 IS-A InputStream，可任意嵌套
  • java.io：FilterInputStream / FilterReader 就是装饰者基类，BufferedReader、DataInputStream、LineNumberReader 都是装饰者。
  • Java Servlet：HttpServletRequestWrapper / HttpServletResponseWrapper 让你包裹原始 request/response 添加功能（如修改参数、包装输出流），是过滤器里改请求的标准手法。
  • Spring：TransactionAwareCacheDecorator、各类 ...Decorator 命名的类；MyBatis 的缓存体系（LruCache、FifoCache 装饰 PerpetualCache）也是装饰者。

常见误区与陷阱

装饰层数过多，调试困难。java.io 的"小类太多、要套好几层才能用"常被诟病。层数多时，一次调用要穿过多层委托，调用栈很深，排查问题成本高。
不要依赖具体类型。装饰之后对象的具体类型变了（外层是装饰者类），如果代码里用 instanceof 判断具体组件类型、或强转回 Espresso，装饰一包就失效。要始终面向 Component 接口编程。装饰者只适合"加行为"，不适合"加客户依赖的新公开方法"。

与易混模式的对比

对比项装饰者 Decorator代理 Proxy适配器 Adapter
意图动态增加职责控制访问（权限/懒加载/远程）转换接口让不兼容的能协作
接口与被包对象相同接口与被代理对象相同接口把一个接口改成另一个
嵌套常多层叠加通常单层通常单层
谁创建内层外部传入，可自由组合代理常自己创建/管理目标外部传入被适配对象

装饰者和代理结构几乎一样（都持有同接口的内层对象），区别在意图：装饰者为了"加功能"，代理为了"控制访问"。

小结

  • 装饰者模式 = 同接口 + 组合持有内层 + 层层包裹叠加职责，是继承的弹性替代。
  • 它践行 OCP：加功能靠"包一层新装饰者"，而不是改原类或造组合子类，避免类爆炸。
  • java.io 流是最经典案例；Servlet 的 Wrapper、MyBatis 缓存也是。
  • 陷阱：层数多导致调试难、小类多；不要在客户端依赖被装饰对象的具体类型。
