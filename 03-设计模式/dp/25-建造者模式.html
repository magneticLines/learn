建造者模式 Builder

把一个复杂对象的"构造过程"和它的"最终表示"分离，让同样的构造步骤可以创建出不同的对象，也让构造代码从一堆参数地狱中解脱出来。

意图 / 定义

GoF Builder Pattern
将一个复杂对象的构建与它的表示分离，使同样的构建过程可以创建不同的表示。
Separate the construction of a complex object from its representation.

建造者模式专门解决"对象的字段很多，且很多是可选的"这一类构造难题。它通过分步骤、链式地设置字段，最后调用 build() 一次性产出不可变对象，让创建过程既清晰又安全。

解决什么问题、为何这么设计

当一个类有大量构造参数时，传统做法会陷入两难：

  • 重叠构造器（telescoping constructor）：为每种参数组合写一个构造器，参数一多就爆炸，调用方还要数第几个参数是什么。
  • JavaBean 模式（无参构造 + setter）：可读性好，但对象在多次 setter 之间处于"半成品"的不一致状态，且无法做成不可变（immutable）对象。

建造者模式取两者之长：用 setter 风格的可读性，分步装配；用 build() 收口，保证产出的对象一旦诞生就是完整且不可变的。这也是它和工厂模式最大的区别——工厂关心"造哪种对象"，建造者关心"怎么一步步把这个对象拼起来"。

生活类比：点一份定制汉堡

在快餐店点餐：你不会一次性把"要牛肉饼、加双层芝士、不要洋葱、加培根、套餐配薯条"塞进一句话（重叠构造器），而是逐项勾选——加这个、去那个，最后说"就这样，做吧"（build()）。店员（Builder）按你的勾选装配，递给你一个成品汉堡（Product）。同样的装配流程，不同的勾选，产出不同的汉堡。

反例 → 问题 → 正例

❌ 重叠构造器：参数地狱
class HttpRequest {
  HttpRequest(String url) {...}
  HttpRequest(String url,
    String method) {...}
  HttpRequest(String url,
    String method,
    Map<String,String> headers) {...}
  HttpRequest(String url,
    String method,
    Map<String,String> headers,
    String body, int timeout,
    boolean retry) {...}
}

// 调用方根本看不懂：
new HttpRequest("u", "GET",
  null, null, 3000, true);
// 第 4 个 null 是啥？timeout?
✅ 建造者：每个字段自带名字
HttpRequest req = HttpRequest
  .builder()
  .url("https://api.x.com")
  .method("GET")
  .header("Auth", "Bearer ..")
  .timeout(3000)
  .retry(true)
  .build();

// 可读、可选参数随意省略、
// 产出对象不可变

问题所在：重叠构造器的参数顺序和含义全靠记忆，null 占位让人无从判断；可选参数一多，构造器数量呈指数膨胀。建造者用"具名方法 + 链式调用"把每个参数的语义写在脸上。

完整的静态内部类 Builder 写法（经典 Effective Java 范式）
public final class HttpRequest {
    // 字段全部 final —— 对象不可变
    private final String url;
    private final String method;
    private final int timeout;
    private final boolean retry;

    // 私有构造器，只能由 Builder 调用
    private HttpRequest(Builder b) {
        this.url = b.url;
        this.method = b.method;
        this.timeout = b.timeout;
        this.retry = b.retry;
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String url;
        private String method = "GET";  // 默认值天然支持
        private int timeout = 5000;
        private boolean retry;

        // 每个 setter 返回 this —— 这就是链式调用的秘密
        public Builder url(String v) { this.url = v; return this; }
        public Builder method(String v) { this.method = v; return this; }
        public Builder timeout(int v) { this.timeout = v; return this; }
        public Builder retry(boolean v) { this.retry = v; return this; }

        public HttpRequest build() {
            if (url == null)            // build() 里集中做校验
                throw new IllegalStateException("url 必填");
            return new HttpRequest(this);
        }
    }
}

角色 / 结构

GoF 原始定义中建造者有四个角色，不过日常 Java 实践（Effective Java 风格）通常简化为 Builder + Product 两个：

角色 职责 对应上例
Product（产品） 最终被构造出来的复杂对象 HttpRequest
Builder（建造者） 提供分步设置字段的方法 + build() HttpRequest.Builder
ConcreteBuilder Builder 的具体实现（简化版里就是 Builder 本身） 静态内部类
Director（指挥者） GoF 原版中封装固定的构造顺序；日常多省略 通常省略

JDK / Spring 中的真实应用

来源 用法
StringBuilder / StringBuffer 最常见的"建造者"：append().append().toString() 分步拼装字符串，toString() 相当于 build()。注意它严格说更接近"可变累加器"，但链式 + 收口的形态与 Builder 一致
Lombok @Builder 在类上加一个注解，编译期自动生成上面那一整套静态 Builder 内部类，省去手写样板。配合 @Value 可生成不可变对象
Stream.Builder Stream.builder().add(a).add(b).build()
Spring UriComponentsBuilder UriComponentsBuilder.fromUriString(..).queryParam(..).build() 分步构造 URI
Spring Security / MockMvc HttpSecurity 的链式配置、MockMvcBuilders 都是 Builder 思想
OkHttp / Retrofit new OkHttpClient.Builder().connectTimeout(..).build() 是业界 Builder 的标杆案例

常见陷阱

必填字段无法在编译期强制：链式 Builder 把所有字段都做成可选方法，build() 里的运行时校验是最后防线，但它发生在运行时。如果某些字段绝对必填，可以把它们放进 Builder 的构造器参数里（如 new Builder(url)），用编译期约束代替运行时检查。
滥用：字段很少也上 Builder。只有两三个参数、且都必填时，直接用构造器更简洁。Builder 引入了一个额外的内部类和一层间接，是为"参数多 + 可选参数多 + 想要不可变"这一组合服务的，不是默认选择。
Builder 复用导致脏数据：把一个 Builder 实例 build() 之后又继续修改再 build() 第二个对象，容易因共享可变引用（如同一个 List）造成两个产品互相污染。一般每次构造用新的 Builder。

易混模式对比

模式 关注点 核心区别
建造者 Builder 如何一步步装配一个对象 强调构造过程；适合字段多、可选多的复杂对象
工厂方法 Factory Method 创建哪一种对象 一步到位返回成品，由子类决定具体类型；不关心装配细节
抽象工厂 Abstract Factory 创建一族相关对象 关注产品族的一致性，同样一步返回，不分步
原型 Prototype 复制一个已有对象 通过克隆产生新对象，不重新装配
面试高频："Builder 和工厂的区别？"标准答案——工厂是"我要个对象，你给我造一个"（一次调用拿成品）；Builder 是"我自己一步步配置这个对象，最后让你封装成不可变成品"。Builder 解决的是多可选参数的构造可读性与不可变性，工厂解决的是类型选择与创建解耦。

小结

  • 建造者模式 = 分步装配 + 链式调用（每个 setter return this）+ build() 收口。
  • 它解决重叠构造器的"参数地狱"和 JavaBean 模式的"半成品状态"，并能产出不可变对象。
  • JDK 的 StringBuilder、Stream.Builder 与 Lombok 的 @Builder 是日常最常见的落地。
  • 用对场景：字段多 + 可选参数多 + 想要不可变。字段少时别过度设计。
  • 和工厂的分界线：工厂管"造哪种"，Builder 管"怎么拼"。
