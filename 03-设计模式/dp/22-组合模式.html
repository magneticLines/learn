组合模式 Composite

将对象组合成树形结构以表示"部分-整体"层次。组合模式让客户以一致的方式对待单个对象（叶子）和对象组合（容器）。

意图 / 定义

GoF 定义
将对象组合成树形结构以表示"部分-整体"的层次结构，使客户对单个对象和组合对象的使用具有一致性。
Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

关键词是"统一对待"：客户端拿着一个引用，不管它是一片叶子还是一整棵子树，调用的方法都一样（比如 print()）。容器把对子节点的操作递归下去，叶子直接处理自己。

解决什么问题、为何这么设计

书中场景：餐厅菜单系统。原本一个菜单（Menu）里装一堆菜品（MenuItem）。后来需求升级——菜单里要能嵌套子菜单（比如"晚餐菜单"下有"甜点子菜单"）。如果叶子（MenuItem）和容器（Menu）是两种完全不同的类型，客户端遍历时就要不停地判断"这是菜品还是子菜单？是子菜单就要再递归进去"。

这种判断会蔓延到所有遍历代码里，且层级一深就难以维护。组合模式的思路是：让叶子和容器实现同一个组件接口。客户端只面对这个接口，调 print() 时：

  • 如果是 MenuItem（叶子），它就打印自己；
  • 如果是 Menu（容器），它就遍历孩子，对每个孩子再调 print()——递归自然展开。

设计代价：为了"统一接口"，组件接口里要同时包含叶子的方法（如 getName()）和容器的方法（如 add()、remove()）。这是组合模式著名的"透明性 vs 安全性"权衡，后面陷阱部分细说。

生活类比：文件系统

文件夹里既能放文件，也能放别的文件夹。当你右键点"计算大小"时，对一个文件就是它自身的大小，对一个文件夹就是递归累加里面所有文件和子文件夹的大小。你不需要关心点的是文件还是文件夹——操作系统对它们一视同仁。这就是组合模式：文件是叶子，文件夹是容器，二者都是"可计算大小的条目"。

反例 → 问题 → 正例

❌ 反例：客户端到处判断"是菜品还是子菜单"
void printMenu(Object node) {
    if (node instanceof MenuItem) {
        print(((MenuItem) node).getName());
    } else if (node instanceof Menu) {
        for (Object child : ((Menu) node).getItems()) {
            printMenu(child);          // 手动递归
        }
    }
    // 再加一种节点类型？所有遍历点都要补 instanceof 分支
}

问题：instanceof + 强转散落在每个要遍历的地方；新增节点类型要全局加分支；递归逻辑由客户端承担，客户端被迫了解树的内部构造。

✅ 正例：叶子和容器共用一个组件抽象
// 统一组件：叶子和容器都继承它
abstract class MenuComponent {
    // 默认实现抛异常，子类按需覆盖（透明式组合）
    public void add(MenuComponent c) { throw new UnsupportedOperationException(); }
    public abstract void print();
}

// 叶子：直接处理自己
class MenuItem extends MenuComponent {
    private String name;
    public void print() { println(name); }
}

// 容器：持有孩子，操作递归下去
class Menu extends MenuComponent {
    private List<MenuComponent> children = new ArrayList<>();
    @Override public void add(MenuComponent c) { children.add(c); }
    @Override public void print() {
        for (MenuComponent c : children) c.print();  // 统一调 print，叶/容自处理
    }
}

// 客户端：完全不需要 instanceof
MenuComponent root = buildMenuTree();
root.print();  // 一行搞定整棵树，递归自然展开

角色 / 结构

角色职责本例对应
Component（组件）叶子和容器的共同抽象，声明统一操作MenuComponent
Leaf（叶子）没有子节点，直接实现操作MenuItem
Composite（容器）持有子组件集合，把操作递归委托给孩子Menu
Client统一通过 Component 操作，不区分叶/容调 root.print() 的代码

JDK / 框架中的真实应用

  • Swing / AWT 容器树：java.awt.Component 是组件，Container 是容器（可 add 子组件），JButton 等是叶子。paint() 会递归画整棵 UI 树——教科书级组合模式。
  • 文件系统 API：java.io.File 既可表示文件（叶子）也可表示目录（容器），listFiles() 递归遍历目录树。
  • DOM 树：HTML/XML 的 Node，元素节点可含子节点，文本节点是叶子，统一的 Node 接口。
  • Spring：CompositeCacheManager、CompositePropertySource 等把多个同类对象组合成一个，对外表现为单个。

常见陷阱

透明性 vs 安全性的两难：
· 透明式（把 add/remove 放在 Component 接口）：叶子和容器接口完全一致，客户端无需区分（更透明）；但叶子也"被迫"拥有 add()，调用它只能抛异常——牺牲了类型安全。
· 安全式（add/remove 只放在 Composite）：叶子上根本没有这些方法，编译期就安全；但客户端必须先判断"这是不是容器"才能 add，又回到了 instanceof。Head First 选择透明式，优先保证统一性。
违反单一职责，是有意为之：组合模式让 Component 同时承担"管理孩子"和"业务操作"两件事，明知违反单一职责原则，换取的是"客户端透明统一"。这是设计中典型的用一个原则换另一个原则的取舍，要理解它的代价。
环引用与遍历：树里若误加成环（A 包含 B，B 又包含 A），递归 print() 会无限循环栈溢出。构建树时要保证它是真正的树（无环）。

易混模式对比

对比项组合 Composite装饰者 Decorator迭代器 Iterator
结构树（一对多，多个孩子）链（一对一，层层包裹）线性遍历，不构树
目的统一对待整体与部分动态给对象加功能顺序访问元素不暴露结构
典型搭配常内置迭代器遍历子节点—常用于遍历组合树
面试高频："组合模式和装饰者都是递归包装，区别？"→ 组合是一对多的树（容器有多个孩子，目的是统一对待整体/部分）；装饰者是一对一的链（每层只包一个对象，目的是叠加功能）。

小结

  • 组合模式用树形结构表示"部分-整体"，让客户端统一对待叶子和容器。
  • 叶子和容器共享同一组件接口，容器把操作递归委托给孩子，消除客户端的 instanceof 判断。
  • 代价是 Component 接口同时含叶子和容器方法，存在透明性 vs 安全性的取舍，并刻意违反了单一职责。
  • JDK 的 Swing 容器树、File 目录树、DOM 树都是经典应用。
