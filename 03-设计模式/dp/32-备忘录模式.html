备忘录模式 Memento

在不破坏封装的前提下，捕获并外部保存一个对象的内部状态，以便之后把对象恢复到这个状态。撤销、存档的底层模式。

意图与定义

GoF · Memento
在不破坏封装的前提下，捕获并外部化对象的内部状态，使对象之后可恢复到此状态。
Without violating encapsulation, capture and externalize an object's internal state so that the object can be restored to this state later.

关键词是"不破坏封装"。备忘录模式让你保存一个对象的快照，却不必把它的私有字段暴露给外界——快照对象只对原对象透明，对其他人是个不可读的黑盒。

解决什么问题，为何这么设计

需要"撤销"、"回退到上一步"、"读档"时，本质上要做两件事：

  • 在某个时刻把对象的内部状态完整保存下来
  • 之后能把对象精确还原到那一刻

最直接的做法是把对象的字段都设成 public，让外部读出来存起来。但这彻底破坏了封装——任何人都能随意改对象内部状态，对象失去了对自身一致性的控制。

备忘录模式的巧妙之处在于角色分工：Originator（原发器）自己生成快照（它当然有权访问自己的私有状态），快照被装进一个 Memento 对象；Caretaker（管理者）只负责保管 Memento，但看不懂里面的内容。这样既能保存/恢复状态，又把状态的细节牢牢锁在原发器内部。这是"窄接口/宽接口"的经典运用：Memento 对 Caretaker 暴露窄接口（几乎什么都没有），对 Originator 暴露宽接口（能取出全部状态）。

生活类比：游戏存档

打 Boss 前你按下"保存"，游戏把你当前的等级、血量、装备、坐标打包成一个存档文件。你不知道、也不需要知道这个文件里的二进制格式（封装没被破坏）。打挂了就读档，游戏自己解析存档把你还原到存档那一刻。你（Caretaker）只是把存档拿在手里、决定何时读，但你看不懂存档内容；游戏（Originator）才是唯一能写入和读取存档的人。

反例 → 问题 → 正例

需求：文本编辑器支持撤销（Ctrl+Z）。

❌ 反例：把内部状态全部暴露，外部直接读写
class Editor {
    public String content;     // 被迫 public，封装尽失
    public int    cursor;
}
// 外部为了"撤销"，自己复制字段保存
Editor e = new Editor();
String savedContent = e.content;   // 谁都能读写 e 的内部
int    savedCursor  = e.cursor;
// ...编辑...
e.content = savedContent;          // 谁都能随意篡改，对象无法保证一致性
e.cursor  = savedCursor;

问题：字段全暴露，任何代码都能破坏编辑器内部一致性；恢复逻辑散落在外部；一旦 Editor 新增字段，所有保存/恢复处都要同步改，极易遗漏。

✅ 正例：Originator 自己打包 / 还原快照，状态对外不可读
// Memento：状态快照。字段私有，只有 Originator 能访问内部
class EditorMemento {
    private final String content;
    private final int cursor;
    // 包级私有/嵌套类：对外（Caretaker）是黑盒
    EditorMemento(String content, int cursor) {
        this.content = content; this.cursor = cursor;
    }
    String getContent() { return content; }  // 仅 Originator 可见
    int getCursor() { return cursor; }
}

// Originator：核心对象，自己负责存档与读档
class Editor {
    private String content = "";
    private int cursor = 0;

    void type(String s) { content += s; cursor = content.length(); }

    // 自己访问私有字段生成快照——封装毫发无损
    EditorMemento save() { return new EditorMemento(content, cursor); }
    // 自己从快照还原
    void restore(EditorMemento m) { content = m.getContent(); cursor = m.getCursor(); }
    String text() { return content; }
}

// Caretaker：只保管 Memento，看不懂里面是什么
class History {
    private final Deque<EditorMemento> stack = new ArrayDeque<>();
    void push(EditorMemento m) { stack.push(m); }
    EditorMemento pop() { return stack.pop(); }
}

// 使用：编辑前存档，撤销时读档
Editor e = new Editor();
History h = new History();
e.type("Hello"); h.push(e.save());   // 存档
e.type(" World");
e.restore(h.pop());            // 撤销 → "Hello"

字段始终私有；只有 Editor 能读写自己的状态；History 拿着快照却看不懂内容。新增字段只改 Editor 和 EditorMemento 一处，外部无感。

角色与结构

角色职责本例对应
Originator（原发器）核心对象，创建快照并能从快照恢复自身状态Editor
Memento（备忘录）保存原发器的内部状态；对原发器宽接口、对外窄接口EditorMemento
Caretaker（管理者）保管 Memento（决定何时存/取），但不读写其内容History
实现技巧：Java 里常把 Memento 写成 Originator 的私有静态内部类，再让 Memento 实现一个空的"标记接口"暴露给 Caretaker。这样只有外层 Originator 能访问 Memento 的真实字段，封装控制达到极致。

JDK / Spring 中的真实应用

场景说明
Java 序列化（Serializable）把对象状态序列化为字节流、之后反序列化还原，是备忘录思想的通用化实现——快照即字节流。
数据库事务与 SavepointJDBC 的 Connection.setSavepoint() / rollback(savepoint) 正是保存与恢复某一时刻状态。
Swing UndoManager / UndoableEdit编辑器撤销栈的标准实现，每个 UndoableEdit 承载一次可逆状态。
Spring Web Flow / 会话快照流程状态在步骤间保存与恢复，便于回退到上一步。
JPA / Hibernate 脏检查持久化上下文保存实体加载时的"快照"，提交时与当前状态比对决定是否 UPDATE。

常见陷阱

内存开销：频繁存档（每次按键一个快照）会保存大量完整状态，内存飙升。优化方向：增量备忘录（只存 diff）、限制历史栈深度、对大对象做写时复制（COW）。
浅拷贝陷阱：如果对象状态包含可变引用（如 List、自定义对象），快照里存的是引用而非副本，后续修改会"穿透"到已保存的快照，恢复时拿到的是被污染的状态。必须做深拷贝。
别让 Caretaker 读 Memento 内容：一旦 Caretaker 开始解析、修改 Memento 内部，封装就被打破，又退化回反例。Caretaker 永远只做"保管"。

易混模式对比

模式意图与备忘录的区别
Command 命令把请求封装成对象，支持撤销/重做命令的撤销靠"反向操作"（记住怎么做的逆动作）；备忘录靠"保存整个状态再还原"。复杂撤销系统常把两者结合：命令负责动作，备忘录负责状态快照。
Prototype 原型通过克隆创建新对象原型用克隆生产新对象；备忘录用快照恢复同一对象的历史状态。两者都涉及拷贝，目的不同。
State 状态对象行为随内部状态改变状态模式管理"当前是哪种状态及其行为"；备忘录管理"状态的保存与回滚"。
面试高频："撤销功能怎么实现？" 答案分两路——状态快照式（备忘录，简单但费内存）与反向操作式（命令模式，省内存但每个操作都要写逆动作）。能说出这两条路线及其取舍即满分。

小结

  • 备忘录模式：在不破坏封装的前提下，把对象内部状态打包成快照外部保存，以便之后恢复。
  • 三角色：Originator 自己存/取快照、Memento 承载状态（对外黑盒）、Caretaker 只保管不解读。
  • 核心价值是"窄/宽接口"——既能保存恢复，又不暴露内部，封装毫发无损。
  • 陷阱：内存开销大（用增量/限深度）、浅拷贝穿透（用深拷贝）、Caretaker 别读内容。
  • 真实身影：序列化、JDBC Savepoint、Swing UndoManager、Hibernate 脏检查快照。

与其他知识点的联系

  • 命令模式的撤销机制 Command：撤销功能有两条实现路线，二者常被拿来对比——命令模式靠"记住怎么做的逆操作"（如插入的逆操作是删除），省内存但每个命令都要写清楚怎么撤销；备忘录靠"保存整个状态快照再整体还原"，实现简单但内存开销大。工业级的撤销系统（如 IDE、图形编辑器）常把两者结合：命令对象记录动作意图，备忘录负责在必要时兜底保存/恢复复杂状态。
  • 原型模式 Prototype：两者都涉及"拷贝对象状态"，容易混淆。原型的克隆是为了产出一个新对象供后续独立使用；备忘录的快照是为了保存历史状态，之后要拿它去恢复同一个原发器，而不是新建一个对象。备忘录里"深拷贝陷阱"的坑，本质上和原型模式里"深克隆 vs 浅克隆"是同一个问题。
  • 状态模式 State：状态模式管理"对象当前处于哪种状态、该状态下如何响应行为"；备忘录管理"如何把状态保存下来、之后如何回滚"。两者关注的都是对象状态，但状态模式是横向的（在多种状态间切换行为），备忘录是纵向的（在时间轴上前进和倒退）。
  • 封装与"宽接口/窄接口"设计原则：备忘录模式是"不破坏封装"这一面向对象核心原则的教科书级案例——它通过让 Memento 对 Originator 暴露宽接口、对 Caretaker 暴露窄接口（甚至用私有内部类+标记接口彻底锁死），示范了如何在"需要导出内部状态"和"不能让外部乱改内部状态"这对矛盾之间找到平衡，这个设计手法在后续接触到的其他需要精细控制可见性的场景中会反复出现。
  • JDBC Savepoint 与 Hibernate 脏检查：本文提到的数据库 Savepoint、Hibernate 加载实体时保存的快照，都是备忘录思想在持久化层的真实落地——事务回滚本质上就是把数据库状态"恢复"到某个 Savepoint 记录的快照，这为后续学习事务管理、乐观锁机制打下了概念基础。
