模板方法模式 Template Method

在父类里定义算法的"骨架"，把某些步骤延迟到子类实现。骨架不变，细节可变——这是"好莱坞原则"（别调用我，我会调用你）的典范。

意图与定义

GoF 定义
在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以在不改变算法结构的前提下，重新定义算法的某些步骤。
Define the skeleton of an algorithm in an operation, deferring some steps to subclasses.

关键词是"骨架"。父类用一个 final 的模板方法把步骤的顺序固定下来，其中可变的步骤声明为抽象方法，交给子类填空。算法的整体流程被父类牢牢掌控，子类只能改细节，改不了流程。

解决什么问题，为何这么设计

当多个类的处理流程大体相同、只在个别步骤有差异时，如果每个类各写一遍完整流程，会产生大量重复代码。一旦流程要调整（比如多加一个校验步骤），所有副本都得改，极易遗漏。

模板方法把公共流程上提到父类，只把差异点留成抽象方法。这样流程只有一份，复用最大化；同时通过把模板方法设为 final，防止子类破坏算法结构——子类只能改你允许它改的步骤。这就是"控制反转"：不是子类调用父类，而是父类的模板在合适时机回调子类。

生活类比：冲泡饮料

冲咖啡和泡茶的流程几乎一样：① 烧水 → ② 冲泡 → ③ 倒进杯子 → ④ 加调料。烧水和倒杯子完全相同（公共步骤，放父类），冲泡（咖啡是冲咖啡粉，茶是泡茶叶）和加调料（咖啡加糖奶，茶加柠檬）不同（抽象步骤，交子类）。模板就是这份"冲泡饮料"的标准作业流程。

反例 → 问题 → 正例

反例：咖啡和茶各写一遍完整流程

❌ 流程重复，烧水/倒杯子两份拷贝
class Coffee {
    void prepare() {
        boilWater();          // 重复
        brewCoffeeGrinds();   // 差异
        pourInCup();          // 重复
        addSugarAndMilk();    // 差异
    }
}
class Tea {
    void prepare() {
        boilWater();          // 又写一遍
        steepTeaBag();        // 差异
        pourInCup();          // 又写一遍
        addLemon();           // 差异
    }
}
// 想给流程加一步"检查水质"？两个类都得改

问题：boilWater() 和 pourInCup() 在两个类里重复；流程顺序在两处各维护一份，容易走样；新增步骤要改所有子类。

正例：父类定义模板方法 + 钩子方法

模板方法（final）固定骨架，抽象步骤交子类，hook 可选覆盖
abstract class CaffeineBeverage {

    // 模板方法：final 防止子类改流程
    public final void prepare() {
        boilWater();
        brew();                       // 抽象步骤，子类必须实现
        pourInCup();
        if (customerWantsCondiments())  // 钩子控制是否执行
            addCondiments();
    }

    // 公共步骤：父类实现，所有子类共享
    private void boilWater() { /* 烧水 */ }
    private void pourInCup() { /* 倒杯 */ }

    // 抽象步骤：差异点，子类必须填空
    protected abstract void brew();
    protected abstract void addCondiments();

    // 钩子方法（hook）：有默认实现，子类可选择覆盖
    protected boolean customerWantsCondiments() { return true; }
}

class Tea extends CaffeineBeverage {
    protected void brew() { /* 泡茶叶 */ }
    protected void addCondiments() { /* 加柠檬 */ }
    // 想要不加料的茶：覆盖钩子返回 false 即可
}

流程只剩一份，写在父类的 prepare() 里；想加"检查水质"，只改父类一处。子类只关心自己的差异步骤。

结构与角色

角色职责
AbstractClass定义 final 的模板方法（算法骨架），声明抽象方法和钩子方法
模板方法 template method按固定顺序调用各步骤，通常设为 final
抽象方法 primitive operation必须由子类实现的差异步骤
钩子方法 hook父类有默认实现，子类可选择性覆盖以影响流程
ConcreteClass实现抽象步骤、按需覆盖钩子

抽象方法 vs 钩子方法

  • 抽象方法：父类无法给出实现，子类必须填空（如"怎么冲泡"）
  • 钩子方法：父类有合理默认，子类可选覆盖——常用于"是否执行某步骤""算法某处的回调通知"，给子类一个介入流程的口子

JDK / Spring 中的真实应用

  • JDK AbstractList / AbstractMap：骨架实现了大部分方法（如 indexOf、contains），只把 get(i)、size() 留作抽象——你继承它只填这两个，整个 List 行为就齐了
  • JDK Arrays.sort / Collections.sort：排序算法是骨架，比较逻辑（compareTo/Comparator）是可变步骤（此处偏向用回调/策略，但思想相通）
  • Spring JdbcTemplate：把"获取连接 → 创建 Statement → 执行 → 处理结果 → 关闭资源 → 异常转换"的繁琐流程固化成模板，你只通过回调提供 SQL 和 RowMapper（结果映射）
  • 同理 RestTemplate、TransactionTemplate、HibernateTemplate——Spring 的 XxxTemplate 家族都是模板方法思想（多用回调实现，本质一致）
  • Servlet 的 HttpServlet.service()：根据请求方法分发到 doGet/doPost，后者由你覆盖

常见陷阱

模板方法要加 final：如果忘了 final，子类可以覆盖整个模板方法，算法骨架就被破坏了——模板方法的全部价值就在于"流程不可被改"。
抽象方法别太多：留给子类填空的步骤越多，子类负担越重、越容易出错。Head First 的建议是尽量减少抽象方法的数量，能用钩子（带默认实现）就别用抽象方法。
继承的硬伤：模板方法靠继承实现，是编译期绑定，一个子类只能继承一个父类的骨架，无法运行时切换算法。若流程本身也要灵活变化，应改用策略模式（组合）。

易混模式对比：模板方法 vs 策略（继承 vs 组合）

两者都用来"封装可变的算法步骤"，但实现机制根本不同——这正是"多用组合少用继承"原则的活教材：

对比项模板方法 Template Method策略 Strategy
实现机制继承（子类覆盖父类步骤）组合（持有策略对象）
绑定时机编译期，写死在子类运行期，可动态 set 替换
复用粒度复用整个算法骨架，只变个别步骤把整个算法作为对象替换
灵活性低（单继承限制）高（可随时换算法）
谁掌控流程父类掌控（好莱坞原则）客户端通过组合掌控
适用流程固定、仅步骤有别整个算法需要互换
好莱坞原则："Don't call us, we'll call you." 高层组件（父类模板）控制流程，在需要时回调低层组件（子类步骤），低层组件不主动调高层。模板方法是这条原则最直接的体现。

小结

  • 模板方法 = 父类定义算法骨架（final 模板方法），子类填空差异步骤
  • 抽象方法（必须实现） vs 钩子方法（可选覆盖，常用于控制流程分支）
  • 核心价值：复用公共流程 + 防止子类破坏算法结构，体现好莱坞原则
  • JDK 的 AbstractList、Spring 的 JdbcTemplate 是经典应用
  • 与策略模式互为镜像：模板方法靠继承（编译期），策略靠组合（运行期）
