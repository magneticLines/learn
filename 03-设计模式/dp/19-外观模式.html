外观模式 Facade

为一个复杂子系统提供一个统一、简化的高层接口。它不隐藏底层能力，只是给客户端递上一个"一键搞定"的入口。

意图与定义

GoF 定义
为子系统中的一组接口提供一个一致的（统一的）界面，外观模式定义了一个高层接口，使得这一子系统更加容易使用。
Provide a unified interface to a set of interfaces in a subsystem.

"Facade"原意是建筑的正立面——你从街上看到的那一面漂亮门脸，背后是错综复杂的管线和房间。外观模式就是给一堆杂乱的子系统装一个漂亮简单的"门脸"，客户端只跟门脸打交道。

解决什么问题，为何这么设计

当一个子系统由很多类组成、彼此还有复杂的调用顺序和依赖时，客户端要完成一件事，往往得自己 new 一堆对象、按正确顺序调一长串方法。这导致：客户端代码冗长易错；客户端与子系统的每个细节紧耦合，子系统一改，客户端全崩。

外观模式的设计思路是：把这段复杂的调用流程封装进一个 Facade 类的简单方法里。客户端只调 facade.watchMovie()，背后的"开投影、降幕布、调音响、放碟"全由 Facade 编排。这样客户端只依赖 Facade 一个类，子系统内部如何重构都不影响它——耦合从"多对多"降为"一对一"。这直接体现了最少知识原则（迪米特法则）。

生活类比：餐厅服务员

你去餐厅吃饭，只对服务员（Facade）说"我要一份套餐"。你不需要直接对接采购、厨师、传菜、收银这一堆部门（子系统）。服务员替你协调所有环节。注意：你仍然可以提特殊要求绕过套餐——外观不剥夺你直接访问底层的能力，只是给了你一条省心的捷径。

反例 → 问题 → 正例

反例：看个电影要手动操作整个家庭影院

❌ 客户端被迫了解并编排所有子系统
// 客户端为了"看电影"，要记住一长串顺序
popper.on();
popper.pop();
lights.dim(10);
screen.down();
projector.on();
projector.setInput(dvd);
projector.wideScreenMode();
amp.on();
amp.setDvd(dvd);
amp.setVolume(5);
dvd.on();
dvd.play(movie);
// 看完关机又要反向来一遍……客户端和每个设备都紧耦合

问题：客户端必须知道 6 个子系统、十几个方法、严格的调用顺序；任何一个设备升级换 API，客户端就要改；这段逻辑无法复用。

正例：用 HomeTheaterFacade 封装编排逻辑

Facade 持有所有子系统，把复杂流程收进一个方法
class HomeTheaterFacade {
    private final Projector projector;
    private final Amplifier amp;
    private final DvdPlayer dvd;
    private final Lights lights;
    private final Screen screen;
    // 构造器注入所有子系统（略）

    // 客户端一句话搞定，编排细节封装在这
    public void watchMovie(String movie) {
        lights.dim(10);
        screen.down();
        projector.on();
        amp.on(); amp.setVolume(5);
        dvd.on(); dvd.play(movie);
    }
    public void endMovie() { /* 反向关闭所有设备 */ }
}

// 客户端：清爽到极致
HomeTheaterFacade home = new HomeTheaterFacade(...);
home.watchMovie("星际穿越");
home.endMovie();

客户端现在只认识 HomeTheaterFacade 一个类。子系统照旧可以单独使用——Facade 不封锁底层，只提供捷径。

结构与角色

角色职责
Facade（外观）知道哪些子系统负责处理请求，把客户端请求委派给恰当的子系统对象，并编排调用顺序
Subsystem（子系统类们）实现真正的功能，处理 Facade 指派的工作；它们不知道 Facade 的存在
Client（客户端）通过 Facade 与子系统交互，无需直接接触子系统细节
外观不是"封死"：Facade 增加便利，但不阻止客户端在需要时绕过它直接访问子系统。这是它与"接口隔离/封装"的关键区别——它降低复杂度，不降低能力。

JDK / Spring 中的真实应用

  • SLF4J 的 LoggerFactory：为 logback、log4j 等多种日志实现提供统一简单的门面 API（日志门面就是 Facade 思想的命名来源之一）
  • JDBC 的 DriverManager：把不同数据库驱动的复杂加载/连接细节收拢成 getConnection()
  • Spring JdbcTemplate：既是模板方法，也充当 JDBC 子系统的外观——把连接、语句、结果集、异常处理一站式封装
  • SpEL、ApplicationContext：作为 Spring 容器众多子系统（Bean 工厂、事件、资源加载、国际化）的统一外观
  • 各种 XxxFacade 命名的 Service 层、微服务的 API 网关（架构级外观）

常见陷阱

上帝对象（God Facade）：不要把所有功能一股脑塞进一个超级 Facade，它会膨胀成无所不知的"上帝类"。一个子系统可以有多个针对不同场景的外观，按职责拆分。
别让 Facade 变成新瓶颈：如果每加一个子系统功能都要改 Facade，且客户端又强依赖它，Facade 反而成了变更集中点。保持 Facade 薄——只做编排转发，不写业务逻辑。
外观不等于隐藏：外观降低复杂度，但子系统应仍可独立访问。如果你的目的是控制/限制访问，那是代理模式；如果是转换接口，那是适配器——别混了意图。

重点对比：适配器 vs 外观 vs 装饰者

这三者结构上都"包了一层别的对象"，极易混淆。区别全在意图和包了几个对象：

对比项适配器 Adapter外观 Facade装饰者 Decorator
核心意图转换接口简化接口增强功能
包装对象数量通常一个一组（整个子系统）一个（可层层叠加）
对接口的影响变成另一个已有接口新建一个更简单的接口保持同一个接口
是否加功能否否（只组织编排）是
客户端目的"我想用不兼容的东西""我嫌这堆东西太难用""我想给它加点料"
类比电源转换插头餐厅服务员 / 总开关给咖啡加奶泡加糖浆
记忆口诀：适配器"改接口"、外观"减接口"、装饰者"同接口加料"。三者都靠组合包装，意图各不相同，这是面试高频区分点。

外观与"最少知识原则"

Head First 设计原则 · 最少知识原则
只和你的密友交谈。
Talk only to your immediate friends. (Law of Demeter / 迪米特法则)

这条原则要求：一个对象应尽量减少它认识的其他对象数量，不要通过"朋友的朋友"层层调用（避免 a.getB().getC().doSomething() 这种火车残骸式调用）。外观模式正是这条原则的直接落地——客户端只认识 Facade 这一个朋友，不再认识子系统里的十几个对象。引入 Facade，客户端的"朋友圈"从 N 个缩到 1 个，依赖大幅降低。

小结

  • 外观 = 给复杂子系统一个统一、简化的高层入口，客户端只认识 Facade 一个类
  • 三角色：Facade（编排转发）、Subsystem（真正干活）、Client（只对 Facade）
  • 外观简化但不封锁底层，客户端仍可绕过它直接访问子系统
  • 是最少知识原则（迪米特法则）的直接体现：把客户端的依赖从多对多降为一对一
  • 与适配器（改接口）、装饰者（加功能）结构像但意图不同——外观是"减少接口数量、降低使用难度"

与其他知识点的联系

  • 适配器模式（上一篇）：结构都是"包一层"，但外观包的是一整个子系统（多个类）、目的是"降低使用难度"；适配器包的是一个不兼容对象、目的是"转换接口让其可用"。两者常被面试题放在一起问区别，关键抓手就是"包几个对象"和"意图"。
  • 代理模式：外观和代理的类图都可能长得像"包一层再转发"，但外观追求简化（子系统本身可被绕过直接访问），代理追求控制访问（通常不允许绕过代理直接访问真实对象，比如权限校验、延迟加载）。区分二者要看"能不能绕过它"。
  • 抽象工厂 / 建造者模式：外观关注"简化一组已有接口的调用方式"，而抽象工厂/建造者关注"简化复杂对象的创建过程"。如果一个 Facade 里主要做的是 new 一堆对象再组装，它就同时兼具了建造者的意味——两者在实践中常常协作出现。
  • Spring JdbcTemplate / ApplicationContext：这两者既是模板方法的应用，也同时是外观模式的应用——JdbcTemplate 把 JDBC 一整套连接/语句/结果集/异常处理子系统收拢成几个简单方法；ApplicationContext 则是 Spring 容器（Bean 工厂、事件、资源、国际化等众多子系统）的统一入口，体现了外观模式在框架设计中"给复杂子系统一个门脸"的核心价值。
  • 分层架构中的 Service 层 / API 网关：后端项目里 Service 层对 Controller 而言就是一层外观——把多个 DAO/Repository 的调用编排收拢成一个业务方法；微服务架构里的 API 网关则是外观模式在架构层面的放大版，同样体现"最少知识原则"。
