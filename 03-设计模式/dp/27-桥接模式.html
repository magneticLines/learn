桥接模式 Bridge

把"抽象"和"实现"分到两个独立的继承体系里，用一座"桥"（组合引用）把它们连起来，让两个维度可以各自独立地变化与扩展。

意图 / 定义

GoF Bridge Pattern
将抽象部分与它的实现部分分离，使它们都可以独立地变化。
Decouple an abstraction from its implementation so that the two can vary independently.

这里的"抽象"和"实现"不是指 Java 的 abstract 和 implements，而是指两个正交的变化维度。桥接的精髓是：当一个事物有两个独立变化的维度时，不要用继承去组合它们（会爆炸），而是用组合把两个维度桥接起来。

解决什么问题、为何这么设计

设想你要画"形状 × 颜色"：形状有圆、方、三角；颜色有红、蓝、绿。如果用继承，你会得到 RedCircle、BlueCircle、RedSquare…… 共 3×3=9 个类。再加一种形状或颜色，类的数量呈笛卡尔积式爆炸（m×n）。

根本原因：继承把"形状"和"颜色"两个独立维度焊死在了一棵继承树上。桥接模式的设计思路是——把其中一个维度抽出去做成接口（实现维度），让另一个维度（抽象维度）通过组合持有它的引用。这样两个维度各自独立扩展，类的数量从 m×n 降到 m+n。这正是"多用组合，少用继承"原则的直接体现。

生活类比：遥控器与电视

遥控器是"抽象"维度（基础遥控、高级遥控、语音遥控），电视是"实现"维度（索尼、三星、小米）。遥控器不通过继承去对应每一台电视，而是持有一个电视接口的引用——任何遥控器都能控制任何电视。新出一款电视，遥控器不用改；新出一种遥控器，电视也不用改。这根"遥控器 → 电视接口"的引用，就是那座桥。

反例 → 问题 → 正例

❌ 继承爆炸：形状 × 颜色 = m×n 个类
abstract class Shape { abstract void draw(); }

class RedCircle    extends Shape { void draw(){ /* 红圆 */ } }
class BlueCircle   extends Shape { void draw(){ /* 蓝圆 */ } }
class GreenCircle  extends Shape { ... }
class RedSquare    extends Shape { ... }
class BlueSquare   extends Shape { ... }
class GreenSquare  extends Shape { ... }
// 加一种颜色 → 要新增"每种形状 × 1"个类
// 加一种形状 → 要新增"每种颜色 × 1"个类  ——爆炸

问题所在：形状和颜色是两个独立变化的维度，继承却把它们捆在一起。任一维度增加，都要按另一维度的规模成倍新增类，且红色的逻辑在每个 RedXxx 里重复。

✅ 桥接：抽象（Shape）持有实现（Color）的引用
// ===== 实现维度：Color 是一个独立的接口体系 =====
interface Color { String fill(); }
class Red   implements Color { public String fill() { return "红色"; } }
class Blue  implements Color { public String fill() { return "蓝色"; } }

// ===== 抽象维度：Shape 通过组合"桥接"到 Color =====
abstract class Shape {
    protected final Color color;     // ← 这就是"桥"：组合引用
    protected Shape(Color color) { this.color = color; }
    abstract String draw();
}

class Circle extends Shape {
    public Circle(Color c) { super(c); }
    String draw() { return "画一个" + color.fill() + "的圆"; }
}
class Square extends Shape {
    public Square(Color c) { super(c); }
    String draw() { return "画一个" + color.fill() + "的方"; }
}

// 自由组合：两个维度独立扩展，类数量 = m + n 而非 m × n
Shape s = new Circle(new Red());   // 红圆
Shape t = new Square(new Blue());  // 蓝方
// 加一种颜色 Green：只写 1 个类，所有形状立刻能用

角色 / 结构

角色 职责 对应上例
Abstraction（抽象） 定义抽象维度的接口，持有 Implementor 的引用（桥） Shape
RefinedAbstraction 抽象维度的具体子类 Circle、Square
Implementor（实现） 定义实现维度的接口 Color
ConcreteImplementor 实现维度的具体类 Red、Blue

JDK / Spring 中的真实应用：JDBC

JDBC 是桥接模式的经典案例。java.sql 包里的 Connection、Statement 是抽象维度（面向应用的统一 API），各数据库厂商的 Driver 实现是实现维度。

JDBC：同一套 API，桥接到不同数据库的 Driver 实现
// 应用代码只面向 java.sql 的抽象接口
Connection conn = DriverManager.getConnection(url, user, pwd);
Statement stmt = conn.createStatement();

// url 决定桥接到哪个厂商的 Driver（Implementor）：
//   jdbc:mysql://...      → MySQL 的 Driver 实现
//   jdbc:postgresql://... → PostgreSQL 的 Driver 实现
// 应用一行代码不改，换数据库只换驱动 jar + url
来源 抽象维度 / 实现维度
JDBC Connection/Statement ↔ 各厂商 Driver
SLF4J 日志门面 API ↔ Logback / Log4j2 等具体实现（绑定即桥）
JCA / JNDI 统一 API ↔ 各服务提供商实现
AWT 图形 Component ↔ 各平台 peer 实现（GoF 书中原例）

常见陷阱

识别"两个独立维度"是前提：桥接只在确实存在两个会各自变化的维度时才有价值。如果只有一个维度变化，硬上桥接只会平白增加一层抽象，得不偿失。
别和"普通的接口编程"混为一谈：桥接不是"随便抽个接口出来"。它的关键特征是抽象维度本身也会扩展出多个子类，并且每个子类都通过组合复用同一个实现维度。只有实现维度变、抽象维度不变，那是策略/适配器，不是桥接。
过度设计：桥接增加了系统的理解成本（要在两棵树之间跳）。维度组合不多、未来也不会增长时，直接继承反而更直白。

易混模式对比

模式 目的 与桥接的区别
桥接 Bridge 让两个维度独立变化 设计之初就主动分离抽象与实现；双方都会扩展子类
策略 Strategy 让一个算法可替换 只有"算法/行为"一个维度在变；上下文不构成独立扩展的继承体系。结构相似（都靠组合），但桥接是双维度
适配器 Adapter 让不兼容的接口能协作 事后补救：包装一个已存在、接口不匹配的类。桥接是事前规划的架构分离
抽象工厂 Abstract Factory 创建产品族 关注对象创建；常与桥接配合（用工厂决定注入哪个 Implementor）
面试高频：桥接 vs 策略——两者代码结构几乎一样（都是组合 + 接口），但意图不同。策略针对"一个行为维度的可替换"；桥接针对"两个独立维度各自演化"，且抽象侧自己也是一棵会长子类的树。一句话：策略是一维，桥接是二维。

小结

  • 桥接模式 = 把两个独立变化的维度拆成两个继承体系，用组合引用（桥）连接。
  • 它把继承的 m×n 类爆炸，降为 m+n，是"多用组合少用继承"的典范。
  • 这里的"抽象/实现"指两个正交维度，不是语法上的 abstract/implements。
  • JDBC（统一 API ↔ 各厂商 Driver）、SLF4J 是经典落地。
  • 与策略的区别是维度数：策略一维、桥接二维；与适配器的区别是时机：适配器事后补救、桥接事前规划。

与其他知识点的联系

  • 策略模式 Strategy：桥接与策略在代码结构上几乎一样——都是"上层持有一个接口引用，运行时注入具体实现"。区别在于维度数：策略只解决一个可替换的算法维度，上下文本身不构成继承体系；桥接解决两个都会独立扩展子类的维度。理解这一区别是判断"我现在写的是策略还是桥接"的关键，也是高频面试题。
  • 适配器模式 Adapter：两者都靠组合而非继承来解耦，但时机不同——适配器是事后补救，用来包装一个接口不兼容、已经存在的类；桥接是事前规划，在设计之初就主动把两个变化维度拆开。判断"这段代码是适配器还是桥接"，关键看是先有了不匹配的接口再补救，还是从头设计时就分离了抽象与实现。
  • 抽象工厂 Abstract Factory：桥接决定"用哪个 Implementor"往往需要一个决策入口，这正是抽象工厂的用场——用工厂根据配置或环境选出该注入哪个 ConcreteImplementor，两者常常配合使用而不是互相替代。
  • 多用组合，少用继承（原则层面）：桥接是这条经典设计原则最直接的教学案例。它和后面会学到的装饰器模式（同样靠组合叠加行为）一起，构成了"当继承会导致类爆炸或耦合过紧时，用组合替代"的两个典型解法，值得放在一起对比记忆。
  • JDBC / SLF4J 与依赖倒置原则：桥接模式的经典落地（JDBC 统一 API ↔ 各厂商 Driver）本质上也是依赖倒置原则（DIP）的体现——上层的 Connection/Statement 依赖的是抽象接口而非具体驱动实现，这与 Spring 依赖注入"面向接口编程"的思想一脉相承。
