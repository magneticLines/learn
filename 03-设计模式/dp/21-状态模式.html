状态模式 State

允许对象在内部状态改变时改变它的行为，对象看起来好像修改了它的类。把每个状态做成一个独立对象，让"当前是什么状态"决定"行为怎么走"，从而消灭一坨 if-else。

意图 / 定义

GoF 定义
允许对象在其内部状态改变时改变它的行为，对象看起来似乎修改了它所属的类。
Allow an object to alter its behavior when its internal state changes. The object will appear to change its class.

状态模式把每一个状态封装成一个类，每个状态类自己知道：处于该状态时，各种动作应该怎么响应，以及响应后该切换到哪个状态。Context（上下文）把行为委托给当前状态对象。

解决什么问题、为何这么设计

书中用一台糖果机举例。糖果机有四个状态：没有25分钱、有25分钱、售出糖果、糖果售罄。客户有四个动作：投币、退币、转动曲柄、发放糖果。这是一个典型的状态机：当前状态 + 动作 → 行为 + 新状态。

朴素实现是：每个动作方法里写一坨 if (state == HAS_QUARTER) ... else if (state == NO_QUARTER) ...。问题：

  • 违反开放关闭原则：新增一个状态（比如"中奖10%送两颗"），要去每个动作方法里加 else if。
  • 状态转移逻辑散落各处：某个状态在某动作下该转去哪，分散在四个方法里，难以读懂整体状态机。
  • 容易漏分支：加状态时漏改一个方法，就是一个隐藏 bug。

设计思路：把"用整数表示状态 + 大量 if 判断"翻转为"把状态变成对象"。每个状态对象封装该状态下的全部行为，Context 只持有"当前状态对象"，动作直接委托过去。新增状态 = 新增一个类，不动老代码。

生活类比：交通信号灯

红灯、绿灯、黄灯就是三个状态。每个灯自己知道"亮够时间后该切到哪个灯"——红灯之后是绿灯，绿灯之后是黄灯，黄灯之后是红灯。路口控制器（Context）不需要写一堆"如果现在红灯就……否则如果绿灯就……"，它只是把"时间到"这个事件转交给当前的灯，由灯自己决定下一个状态。

反例 → 问题 → 正例

❌ 反例：用 int 状态 + 满屏 if-else
class GumballMachine {
    static final int SOLD_OUT = 0, NO_QUARTER = 1,
                     HAS_QUARTER = 2, SOLD = 3;
    int state = SOLD_OUT;

    void insertQuarter() {
        if (state == HAS_QUARTER)      System.out.println("已经投过币了");
        else if (state == NO_QUARTER) { state = HAS_QUARTER; System.out.println("收到25分"); }
        else if (state == SOLD_OUT)    System.out.println("已售罄");
        else if (state == SOLD)        System.out.println("请稍候");
    }
    void turnCrank() {  /* 又是一坨同样结构的 if-else */ }
    void ejectQuarter() { /* 再来一坨 */ }
    // 新增"中奖"状态 WINNER？四个方法全要加 else if
}

问题：每个动作方法都是同一套状态判断的复制；状态转移逻辑被打散到各方法；增加状态要横扫所有方法，极易遗漏。这是状态模式最经典的动机场景。

✅ 正例：把状态对象化
// 每个状态都实现这个接口，方法就是那些"动作"
interface State {
    void insertQuarter();
    void ejectQuarter();
    void turnCrank();
    void dispense();
}

// "没投币"状态：只关心自己这个状态下怎么响应 + 转去哪
class NoQuarterState implements State {
    private GumballMachine machine;
    public NoQuarterState(GumballMachine m) { this.machine = m; }

    public void insertQuarter() {
        println("收到25分");
        machine.setState(machine.getHasQuarterState());  // 状态转移就在这里，集中清晰
    }
    public void turnCrank() { println("先投币"); }
    // ...其余动作在本状态下的合理响应
}

class GumballMachine {
    private State state;  // 只持有"当前状态对象"
    public void insertQuarter() { state.insertQuarter(); }  // 全部委托，无 if
    public void turnCrank() { state.turnCrank(); }
    public void setState(State s) { this.state = s; }
}

现在加"中奖"状态 WinnerState：新建一个类即可，GumballMachine 的动作方法一行都不用改——这就是开放关闭原则落地。

角色 / 结构

角色职责本例对应
Context（上下文）持有当前状态对象，把请求委托给它；提供切换状态的方法GumballMachine
State（状态接口）声明各状态都要响应的动作State 接口
ConcreteState（具体状态）实现本状态下的行为，并负责把 Context 切换到下一状态NoQuarterState / SoldState 等

重点：状态模式 vs 策略模式（结构相同，意图不同）

这两个模式的 UML 几乎一模一样——都是 Context 持有一个接口引用，把行为委托给具体实现。区别完全在意图和使用方式上，这是面试的经典陷阱。

对比项状态模式 State策略模式 Strategy
意图对象的行为随内部状态自动变化让客户选择一种算法，行为可替换
谁切换实现状态对象自己切换到下一个状态（内部驱动）客户端显式设置策略（外部驱动）
实现之间是否知道彼此知道——A 状态决定下一步去 B 状态，存在转移图不知道——各算法相互独立，平行无关
切换频率运行中频繁、按规则自动流转通常设定一次，很少中途变
心智模型"对象有限状态机，行为随状态流转""一组可互换的算法，挑一个用"
一句话记忆：策略模式是"我（客户）决定用哪套算法"；状态模式是"对象根据当前所处状态，自己决定行为并自己切到下一个状态"。结构是孪生兄弟，灵魂完全不同。

JDK / 框架中的真实应用

  • 线程状态：Thread.State（NEW / RUNNABLE / BLOCKED / WAITING / TERMINATED）本质就是一台状态机，线程在动作触发下流转状态。
  • Spring StateMachine：官方提供的状态机框架，把状态、事件、转移配置化，是状态模式的工程化版本。
  • 工作流 / 订单系统：订单"待支付 → 已支付 → 已发货 → 已完成"是状态模式的天然场景，状态转移规则集中管理。
  • TCP 连接：CLOSED / LISTEN / ESTABLISHED 等状态，每个状态对收到的报文响应不同。

常见陷阱

状态类爆炸：状态很多时会产生大量小类。状态数量适中（书中糖果机 4~5 个）才划算；状态极简或几乎不转移时，用 enum + switch 反而更直观，别为了用模式而用模式。
谁负责状态转移要约定清楚：转移逻辑可以放在具体状态类里（状态自治，去耦合 Context，但状态间互相耦合），也可以放在 Context 里（集中管理转移图，但 Context 会膨胀）。两种都行，但必须全项目统一，否则转移逻辑半边在状态、半边在 Context，是维护噩梦。
状态对象共享问题：若状态对象无内部可变字段（无状态），可以做成单例共享；若它持有数据，多个 Context 共享同一个状态对象会串数据。糖果机案例中状态持有 machine 引用，故每台机器各有一套状态对象。

易混模式对比

对比项状态 State策略 Strategy命令 Command
封装的是状态及其行为与转移可互换的算法一个请求 / 动作
驱动方内部状态流转外部选择调用方触发执行
有无"下一个"概念有，状态机转移无无
面试高频："状态模式和策略模式区别？"→ 结构几乎相同；区别在意图：状态强调"随内部状态自动改变行为，状态间会自我转移"，策略强调"客户在一组平行算法中挑选其一"。

小结

  • 状态模式把每个状态封装成对象，Context 把行为委托给当前状态，消灭散落各处的 if-else 状态判断。
  • 新增状态 = 新增一个类，符合开放关闭原则；状态转移逻辑集中可读。
  • 它与策略模式结构相同、意图不同：状态是内部驱动的自动流转，策略是外部选择的算法替换。
  • 适用于状态数量适中、转移规则明确的场景（状态机、订单流、连接管理）；状态过少时用 enum 更简单。
