原型模式 Prototype

不通过 new 调用构造器来创建对象，而是复制一个已有的实例。当对象的创建成本高，或运行时才知道要创建什么时，"克隆一份"往往比"从头造一个"更划算。

意图 / 定义

GoF Prototype Pattern
用原型实例指定要创建对象的种类，通过拷贝这个原型来创建新对象。
Specify the kinds of objects to create using a prototypical instance, and create new objects by copying this prototype.

原型模式把"创建对象"这件事委托给对象自己——每个对象知道如何复制（克隆）自己。客户端拿到一个原型，调用 clone() 即可得到一个内容相同的新对象，无需知道它的具体类。

解决什么问题、为何这么设计

它主要应对两类场景：

  • 创建成本高：对象的初始化要查数据库、读文件、做大量计算。如果反复需要"内容几乎一样"的实例，先造一个原型，之后克隆比重新初始化快得多。
  • 运行时才确定类型：客户端只持有一个 Prototype 引用，不知道也不关心它的具体类，照样能复制出同类对象——把"用哪个构造器"的耦合消除了。

为什么不直接 new？因为 new 要求调用方写死具体类名和构造参数。原型模式让对象自我复制，把这层耦合屏蔽掉——这与工厂的目标一致，只是手段是"复制"而非"构造"。

生活类比：复印一份文件

你手上有一份排版精美、填好了大部分内容的合同模板。需要给新客户签约时，你不会从空白纸重新打一遍（重新 new + 逐字段赋值），而是复印一份，再改几个客户名和金额。复印（clone）比重打（construct）快，且复印件天生带着原件的全部格式。

反例 → 问题 → 正例：深拷贝 vs 浅拷贝

原型模式最大的坑就在"复制到底复制得多深"。Object.clone() 默认是浅拷贝——只复制对象本身的字段值，引用类型字段仍然指向同一个子对象。

❌ 浅拷贝：两个对象共享同一个 List
class Resume implements Cloneable {
  String name;
  List<String> skills;

  // 默认浅拷贝
  public Resume clone() {
    return (Resume)
      super.clone();
  }
}

Resume a = ...; // skills=[Java]
Resume b = a.clone();
b.skills.add("Go");
// a.skills 也变成了 [Java,Go]!
// 因为 a.skills == b.skills
✅ 深拷贝：连引用对象一起复制
public Resume clone() {
  try {
    Resume c =
      (Resume) super.clone();
    // 手动复制可变引用字段
    c.skills =
      new ArrayList<>(this.skills);
    return c;
  } catch (
    CloneNotSupportedException e) {
    throw new AssertionError(e);
  }
}
// 现在 b.skills 是独立副本

问题所在：浅拷贝下，修改副本的可变引用字段会污染原型，制造出极隐蔽的 bug。深拷贝必须为每个可变引用字段手动复制——而且如果字段本身又含引用，要递归处理。

Cloneable 接口的坑

Java 的 Cloneable 是个设计失败的范例，初学者必须了解它的怪异之处：

  • Cloneable 是个空标记接口，里面一个方法都没有。真正的 clone() 定义在 Object 上，且是 protected。
  • 不实现 Cloneable 就调 super.clone() 会抛 CloneNotSupportedException——这个受检异常让代码很啰嗦。
  • clone() 绕过了构造器创建对象，final 字段无法在 clone 中重新赋值，与不可变对象设计冲突。
Effective Java 的建议：尽量不要用 Cloneable/clone()。优先用拷贝构造器（public Foo(Foo other)）或拷贝工厂（static Foo copyOf(Foo other)）实现复制，更直观、可控、不抛受检异常，也不绕过构造器。

用序列化做深拷贝（终极偷懒法）

把对象序列化再反序列化 —— 自动递归深拷贝
public static <T extends Serializable> T deepCopy(T obj) {
    try (
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        ObjectOutputStream oos = new ObjectOutputStream(bos)
    ) {
        oos.writeObject(obj);           // 写入字节流（整棵对象图）
        try (ObjectInputStream ois = new ObjectInputStream(
                new ByteArrayInputStream(bos.toByteArray()))) {
            return (T) ois.readObject(); // 读出来就是全新独立副本
        }
    } catch (Exception e) {
        throw new RuntimeException(e);
    }
}
// 优点：无需逐字段写深拷贝，自动处理嵌套引用
// 缺点：要求整棵对象图都实现 Serializable；序列化开销大；
//      transient 字段会丢失。生产环境常用 JSON/工具库替代

角色 / 结构

角色 职责
Prototype（原型） 声明克隆自己的接口（如 Cloneable + clone()）
ConcretePrototype（具体原型） 实现克隆方法，负责复制自身（含深/浅拷贝策略）
Client（客户端） 持有原型引用，调用 clone() 创建新对象，不依赖具体类

JDK / Spring 中的真实应用

来源 用法
Object.clone() / Cloneable 语言层面的原型支持（虽然设计不佳）
ArrayList.clone()、HashMap.clone() 集合类提供的克隆——注意它们都是浅拷贝，元素仍共享
Arrays.copyOf() 数组的复制本质也是原型思想
Spring @Scope("prototype") 名字虽叫 prototype，含义是"每次获取 Bean 都返回新实例"，与 GoF 的"复制"语义不同，但都体现"每次都要新对象"的需求，别混淆
JSON / BeanUtils 深拷贝 实践中常用 Jackson、Gson 序列化反序列化或 BeanUtils.copyProperties 实现对象复制

常见陷阱

浅拷贝陷阱（头号坑）：super.clone() 只复制一层，所有引用类型字段在原型和副本之间共享。修改副本里的 List/Map/数组会反向污染原型。务必为每个可变引用字段手动深拷贝。
循环引用：对象图里 A 引用 B、B 又引用 A 时，手写递归深拷贝可能无限循环。序列化方式能自动处理引用关系，是它的一个隐藏优势。
clone() 与继承的脆弱性：父类写了 clone()，子类新增了可变字段却忘了重写，就会退化成浅拷贝。clone() 跨继承体系极易出错，这也是推荐用拷贝构造器的原因之一。

易混模式对比

模式 创建新对象的方式 核心区别
原型 Prototype 复制已有实例（clone） 新对象初始状态 = 原型当前状态；适合创建昂贵或运行时定类型
工厂方法 Factory Method 调用构造器从头造 每次都是"全新初始化"的对象，不携带某个已有实例的状态
建造者 Builder 分步装配后构造 关注复杂对象的分步构造过程，不复制
单例 Singleton 全局唯一实例 恰恰相反——单例禁止复制，原型鼓励复制
面试高频："深拷贝和浅拷贝的区别？"——浅拷贝只复制对象本身和基本类型字段，引用字段共享同一子对象；深拷贝连引用指向的子对象也递归复制，副本与原型完全独立。结合 Object.clone() 默认浅拷贝、需手动深拷贝可变字段来回答，再加一句"实践中我倾向用拷贝构造器或序列化避开 Cloneable 的坑"。

小结

  • 原型模式 = 通过复制已有实例来创建新对象，而不是 new。
  • 适用于：创建成本高、需要大量相似对象、运行时才确定类型。
  • 核心难点是深拷贝 vs 浅拷贝：Object.clone() 默认浅拷贝，必须为可变引用字段手动深拷贝。
  • Cloneable 设计糟糕（空标记接口、protected clone、受检异常、绕过构造器），优先用拷贝构造器/拷贝工厂。
  • 序列化反序列化可自动实现深拷贝（含循环引用），代价是性能与 Serializable 约束。
