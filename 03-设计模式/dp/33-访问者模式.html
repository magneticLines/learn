访问者模式 Visitor

表示一个作用于某对象结构中各元素的操作，让你在不修改这些元素类的前提下定义新操作。把"数据结构"和"作用于它的操作"彻底分离。

意图与定义

GoF · Visitor
表示一个作用于某对象结构中各元素的操作，使你可以在不改变元素类的前提下定义作用于这些元素的新操作。
Represent an operation to be performed on the elements of an object structure. Visitor lets you define a new operation without changing the classes of the elements on which it operates.

访问者把"在一组对象上要做的事"抽出来，装进一个独立的 Visitor 对象。元素类只保留数据和一个 accept(visitor) 入口；要加新操作，写一个新 Visitor 即可，元素类一行不改。

解决什么问题，为何这么设计

一个稳定的对象结构（比如 AST、文件系统树、各类商品），需要在它上面执行很多种、且会不断增加的操作：求值、打印、统计、导出、校验……

如果把这些操作都写成元素类的方法，会出现两个麻烦：每加一种操作就要改所有元素类（散落、违反单一职责），而且不相关的操作逻辑全堆在元素里。访问者反其道而行——把"按元素类型分派的操作"集中到一个 Visitor 类里。这就引出了它的技术核心：双分派（double dispatch）。

为什么需要双分派

Java 的方法调用是单分派：调用哪个方法只取决于"接收者的运行时类型"（obj.method() 看 obj 是谁），而方法的参数只按编译时类型选重载。访问者要根据"元素的运行时类型"和"访问者的运行时类型"两者共同决定执行哪段逻辑，单分派办不到。技巧是分两步：

  1. 第一次分派：调用 element.accept(visitor) —— 按 element 的真实类型选中正确的 accept（多态）。
  2. 第二次分派：在 accept 内部回调 visitor.visit(this) —— 此时 this 是具体元素类型，编译期就能选中正确的 visit 重载。

两次多态/重载叠加，效果等价于"按元素类型 × 访问者类型"二维分派。

生活类比：博物馆里的不同访客

博物馆的展品（元素）是固定的：油画、雕塑、文物。不同访客（访问者）会用不同方式"处理"同一批展品：艺术评论家写评论、保险评估师估价、修复师检查保养状况。展品本身不会因为来了新访客而改变；要支持一种新的"处理方式"，只需让一位新访客进馆走一遍。展品提供"接待"（accept）入口，访客提供"如何对待每类展品"（visit）的逻辑。

反例 → 问题 → 正例

需求：购物车里有 Book、Fruit 两类商品，要支持"计算总价"，未来还要加"打印小票""导出 JSON"等操作。

❌ 反例：操作硬塞进元素类，新操作要改所有元素
class Book {
    double price();
    String toReceipt();   // 加打印 → 改这里
    String toJson();      // 加导出 → 又改这里
}
class Fruit {
    double price();
    String toReceipt();   // 同样要改
    String toJson();      // 同样要改
}
// 每加一种"操作"，就要修改 Book、Fruit……所有元素类

问题：操作种类持续增加，每加一种都横扫所有元素类；元素类被各种不相关职责（计价、打印、序列化）撑爆，违反单一职责；同类操作逻辑分散在多个元素里，无法集中维护。

✅ 正例：accept + visit 双分派，操作抽进 Visitor
// 元素接口：只暴露 accept 入口
interface Item {
    <R> R accept(Visitor<R> v);
}

class Book implements Item {
    final double price; final String title;
    Book(String t, double p) { title = t; price = p; }
    // 第一次分派：按 Book 的真实类型选中此 accept
    public <R> R accept(Visitor<R> v) { return v.visit(this); }
    // ↑ this 是 Book 类型 → 第二次分派编译期选中 visit(Book)
}

class Fruit implements Item {
    final double pricePerKg, kg;
    Fruit(double ppk, double kg) { pricePerKg = ppk; this.kg = kg; }
    public <R> R accept(Visitor<R> v) { return v.visit(this); }
}

// 访问者接口：每个元素类型一个 visit 重载
interface Visitor<R> {
    R visit(Book b);
    R visit(Fruit f);
}

// 一个具体操作 = 一个 Visitor，元素类无需改动
class PriceVisitor implements Visitor<Double> {
    public Double visit(Book b) { return b.price; }
    public Double visit(Fruit f) { return f.pricePerKg * f.kg; }
}

// 加"打印小票"只需新写一个 Visitor，Book/Fruit 一行不改
class ReceiptVisitor implements Visitor<String> {
    public String visit(Book b) { return b.title + " ¥" + b.price; }
    public String visit(Fruit f) { return "水果 ¥" + f.pricePerKg * f.kg; }
}

// 使用：遍历结构，对每个元素 accept 一个 visitor
List<Item> cart = List.of(new Book("GoF", 79), new Fruit(6, 2));
double total = cart.stream().mapToDouble(i -> i.accept(new PriceVisitor())).sum();

角色与结构

角色职责本例对应
Visitor为每种具体元素声明一个 visit 重载Visitor<R>
ConcreteVisitor实现一种具体操作（计价 / 打印 / 导出）PriceVisitor、ReceiptVisitor
Element声明 accept(visitor) 入口Item
ConcreteElement实现 accept，回调 visitor.visit(this)（第二次分派）Book、Fruit
ObjectStructure持有元素集合并支持遍历List<Item>

关键权衡：加新操作易，加新元素难

这是访问者模式最重要的取舍，必须理解清楚：

变更类型代价原因
新增一种操作（如导出 XML）✅ 容易只写一个新 ConcreteVisitor，所有元素类与已有 Visitor 完全不动——开放关闭原则在"操作维度"成立
新增一种元素（如 Toy）❌ 困难要在 Visitor 接口加一个 visit(Toy)，所有已有 Visitor 都被迫实现它——一处改动横扫全部访问者
选型准则：当元素类型稳定、操作经常增加时（如编译器 AST 节点固定，但要不断加新分析 pass），访问者是绝配。反之，若元素类型频繁变化，访问者会让你痛苦——此时把操作放回元素类反而更合适。这正是"表达式问题（Expression Problem）"的两难。

JDK / Spring 中的真实应用

场景说明
java.nio.file.FileVisitorFiles.walkFileTree(path, visitor) 遍历文件树，对每个文件/目录回调 visitFile/preVisitDirectory，是 JDK 里最标准的访问者实现。
注解处理器 ElementVisitor / TypeVisitorjavax.lang.model.element 用访问者遍历源码模型的各类元素，APT 编译期处理的基础。
ASM 字节码框架 ClassVisitor / MethodVisitor遍历并改写字节码结构，Spring AOP、CGLIB 底层大量使用。
编译器 / 表达式引擎 AST 遍历ANTLR 生成的 Visitor、各类语言前端用访问者对语法树做求值、类型检查、代码生成等多 pass 处理。
Spring BeanDefinitionVisitor遍历并处理 BeanDefinition 中的属性占位符。

常见陷阱

加新元素的连锁修改：这是访问者最大的硬伤。新增一个元素类型，Visitor 接口和每一个已有 ConcreteVisitor 都要改。若元素体系不稳定，千万别用访问者。
破坏封装：Visitor 要干活，往往需要读取元素的内部状态，迫使元素暴露大量 getter 或 public 字段，封装被削弱。需在"操作外置"和"封装"之间权衡。
别忘了第二次分派：新手常把 accept 写成 v.visit(this) 之外的形式（比如先 instanceof 判断再调用），那就退化成了手写类型分发，丢失了双分派的意义。accept 内部必须是 visit(this)。

易混模式对比

模式意图与访问者的区别
Iterator 迭代器顺序遍历集合，不暴露内部表示迭代器只负责"逐个取出元素"，对元素做什么由客户端定；访问者把"对每类元素做什么"封装进 visitor，常配合迭代器遍历后逐个 accept。
Strategy 策略封装可互换的单一算法策略是"对一种输入的一个算法"；访问者是"对一族不同元素类型的一组操作"（按类型分派）。
Interpreter 解释器对语法树求值解释器把求值逻辑放进节点的 interpret()；当要在 AST 上加多种操作时，常改用访问者把操作外置，避免节点类膨胀。
Composite 组合树形结构统一处理访问者常作用于 Composite 构成的对象结构上，二者是协作关系而非替代关系。
面试高频："什么是双分派？" 标准答：Java 方法调用是单分派（只按接收者运行时类型选方法），访问者通过 element.accept(v)（按元素类型分派）+ 内部 v.visit(this)（按访问者类型 + this 的具体类型分派）两步，模拟出依赖两个对象运行时类型的双分派。

小结

  • 访问者模式：把作用于对象结构的操作抽进独立 Visitor，元素只提供 accept 入口，操作与数据结构分离。
  • 技术核心是双分派：accept（按元素类型）+ visit(this)（按访问者类型）两步，弥补 Java 单分派的不足。
  • 核心权衡：加新操作易（新写一个 Visitor）、加新元素难（所有 Visitor 都要改）——这是表达式问题的体现。
  • 适用：元素类型稳定、操作频繁增加（编译器 AST、文件树遍历、字节码处理）。
  • 真实身影：FileVisitor、APT 的 ElementVisitor、ASM 的 ClassVisitor、ANTLR 生成的 Visitor。
