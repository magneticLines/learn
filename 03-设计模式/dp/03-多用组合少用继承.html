多用组合，少用继承

继承是"是一个"（is-a）关系，组合是"有一个"（has-a）关系。这条原则说的是：优先用"拥有行为"来扩展功能，而不是"成为某种类"。

原则定义

Head First 设计原则 #3
多用组合，少用继承。
Favor composition over inheritance.

这条原则来自 GoF《设计模式》的第一章，也是整本 Head First 设计模式中被反复引用最多的一条。书的第一章花了大量篇幅演示"继承是怎么失控的"，正是为了让你亲身感受这条原则的必要性。

继承是怎么失控的：鸭子故事

书中的例子是一个鸭子模拟游戏。最初设计很自然：所有鸭子继承同一个基类。

第一版：一切看起来很正常
abstract class Duck {
    public void quack() { /* 呱呱叫 */ }
    public void swim() {  /* 游泳 */  }
    public abstract void display();
}

class MallardDuck extends Duck {
    public void display() { /* 显示绿头鸭 */ }
}

需求变化第一波：加飞行能力

产品说："让鸭子会飞！"于是在基类加了 fly()。结果——橡皮鸭也飞起来了，因为它继承了这个方法。

继承的第一个陷阱：超类的修改会影响所有子类，包括不该受影响的那些。你以为是在扩展，实际上是在污染。

补救方案一：在子类里覆盖（override）

橡皮鸭覆盖 fly() 让它什么都不做，诱饵鸭覆盖 quack() 让它不叫。这能解决问题，但代价是：

  • 每增加一种"异类"鸭子，就要检查并覆盖所有不适用的方法
  • 多人协作时极易遗漏，且没有编译器提醒
  • 行为的"是否具备"散落在几十个子类里，没有统一的地方可以查

补救方案二：把行为抽成接口

拆出 Flyable 和 Quackable 接口，让会飞的鸭子实现 Flyable。这解决了"橡皮鸭飞"的问题，但引入了新问题：

接口没有实现，等于把同一段飞行代码复制到每一个"会飞的鸭子"里。一旦飞行逻辑要修改，你得找出所有实现类逐一改——代码复用完全丢失了。

类爆炸：真正的死结

假设鸭子有 5 种叫声 × 4 种飞法 × 3 种游泳方式，要覆盖所有组合：

Duck
├── QuackFlyWithWingsSwimFastDuck
├── QuackFlyWithWingsSwimSlowDuck
├── SilentFlyWithWingsSwimFastDuck
├── SilentNoFlySwimFastDuck
└── ... 共 60 个子类

这就是类爆炸（Class Explosion）——继承在面对多维度变化时的固有缺陷。

组合如何解决这个问题

书中的解法是：把"飞行"和"叫声"从继承层次中剥离出来，变成可以独立替换的对象，由 Duck 持有（has-a）而不是成为（is-a）。

封装变化点：把行为抽成接口 把飞行行为和叫声行为各自抽成一个接口（FlyBehavior、QuackBehavior），每种具体行为是一个独立的类。
Duck 持有接口，而不是实现行为 Duck 里只放两个字段：FlyBehavior flyBehavior 和 QuackBehavior quackBehavior。Duck 自身不知道如何飞，它把这件事委托给持有的对象。
子类在构造时注入行为，也可运行时替换 MallardDuck 构造时传入 FlyWithWings 和 Quack；RubberDuck 传入 FlyNoWay 和 Squeak。行为可以在任何时刻通过 setter 换掉。
组合版本——Duck 委托行为给持有的对象
// 行为接口
interface FlyBehavior   { void fly();   }
interface QuackBehavior { void quack(); }

// 具体行为——各自独立，互不耦合
class FlyWithWings implements FlyBehavior  { public void fly() { /* 扇翅膀 */ } }
class FlyNoWay     implements FlyBehavior  { public void fly() { /* 什么也不做 */ } }
class FlyRocketPowered implements FlyBehavior { public void fly() { /* 火箭推进 */ } }

// Duck：持有行为，而不是实现行为
abstract class Duck {
    FlyBehavior   flyBehavior;
    QuackBehavior quackBehavior;

    // 把行为委托出去——Duck 自己不飞
    public void performFly()   { flyBehavior.fly();   }
    public void performQuack() { quackBehavior.quack(); }

    // 运行时随时可以换行为
    public void setFlyBehavior(FlyBehavior fb) { this.flyBehavior = fb; }
}

// 具体鸭子：只负责装配行为，不实现行为
class MallardDuck extends Duck {
    MallardDuck() {
        flyBehavior   = new FlyWithWings();
        quackBehavior = new Quack();
    }
}

// 使用示例：运行时换成火箭飞
Duck duck = new MallardDuck();
duck.performFly();                               // 扇翅膀
duck.setFlyBehavior(new FlyRocketPowered());
duck.performFly();                               // 火箭推进——不改 Duck 类

现在：5 种叫声 + 4 种飞法 = 9 个行为类，可以自由组合出任意鸭子，而不是 60 个子类。增加一种飞法只需新建一个类，任何已有代码都不动。

组合 vs 继承：核心区别

维度 继承（is-a） 组合（has-a）
复用方式 通过继承超类的实现 通过持有对象并委托调用
耦合程度 强耦合：子类依赖超类内部实现 松耦合：只依赖接口，不依赖实现
变化影响 超类改动可能破坏所有子类 行为类改动不影响持有它的对象
运行时灵活性 编译时决定，无法动态替换 运行时可随时替换持有的对象
多维度变化 导致类爆炸 各维度独立扩展，线性增长
白盒/黑盒 白盒复用：子类暴露在超类内部中 黑盒复用：只通过接口交互
GoF 原书的表述："优先使用对象组合而不是类继承（Favor object composition over class inheritance）。"并将这条原则列为面向对象设计的两条最基本原则之一（另一条就是"针对接口编程"）。

生活类比：员工 vs 工具箱

继承就像雇一个"会开车的会计"——他的所有技能来自他的身份（岗位继承）。如果公司想要一个"会开车的工程师"，就得再招一个人，所有会开车的员工把开车技能各自内化了一份。

组合就像公司采购了一辆公务车——任何人需要开车都借用同一辆车（持有引用）。你不需要为每个会开车的员工单独买一辆，"开车能力"是可复用的独立对象，谁需要谁拿。

这条原则的边界：继承并非总是错的

原则说的是"少用"，不是"不用"。继承有它合适的场合：

  • 子类确实是超类的一种：Cat extends Animal——猫确实是动物，行为上也是严格的特化，没有不适用的继承方法
  • 超类稳定，不会频繁变动：框架层的基类（如 Spring 的 AbstractController）经过充分设计，子类只做少量扩展
  • 复用的是骨架，而不是行为细节：模板方法模式就是合理使用继承——超类固定算法骨架，子类填充具体步骤
判断句式：问自己"子类是超类的一种，还是超类的一个用法？"如果是后者，用组合。比如"鸭子有飞行能力"——它是飞行的一个用户，不是飞行的一个子类型。

这条原则导出了策略模式

书中 Ch.1 的鸭子案例，其实就是策略模式的完整演示。把这个结构提炼一下：

  • Context（Duck）：持有策略接口，不关心具体实现，把行为委托出去
  • Strategy 接口（FlyBehavior）：定义算法族的统一入口
  • ConcreteStrategy（FlyWithWings 等）：封装具体算法，可独立变化、独立测试

所以"多用组合"不是一条孤立的原则，它是策略模式、装饰者模式、状态模式等一大批行为型模式的共同基础。掌握了这条原则，后面的模式学起来就是"哦，原来是这条原则在这个场景下的具体形态"。

在 Java 生态中的体现

地方 怎么用的组合
java.util.Collections synchronizedList(list) 用装饰者（组合）给任意 List 加锁，而不是继承出一个 SynchronizedArrayList
Spring AOP 用动态代理（组合）给 Bean 加事务、日志，而不是让业务类继承框架基类
Comparator 排序行为是独立的策略对象，传给 sort()，而不是让每个类继承"可排序类"
InputStream 装饰链 new BufferedInputStream(new FileInputStream(...))——层层包装，每层加能力，而不是 BufferedFileInputStream 这种组合爆炸的子类
Kotlin 的 by 委托 语言级别的组合语法糖，直接体现"多用组合"

常见误区

误区一："组合就是不用继承"
不是。继承用于建立类型关系（is-a），组合用于复用行为（has-a）。两者不是替代关系，而是用途不同。鸭子继承体系里仍然有 Duck 的继承树——MallardDuck extends Duck 没问题；问题在于把"飞行能力"也塞进继承链。
误区二："用了接口就是组合"
接口只是类型契约，组合的核心是持有对象并委托。如果你只是让类实现接口，行为逻辑还是写在类里，本质上还是继承思路。组合要求把行为独立成对象，让宿主类持有这个对象的引用。
误区三："组合一定比继承复杂"
初期看起来多了几个类，但面对需求变化时，继承方案的修改成本会指数级增加，而组合方案是线性的。在系统稳定时继承看起来更简单，在系统演化时组合更简单。

与其他知识点的联系

  • 关联知识点：针对接口编程（原则 #2）：组合要真正发挥价值，持有的对象必须是接口类型而不是具体类型——否则宿主类和被持有对象之间依然是紧耦合，只是换了一种耦合方式（字段引用而不是继承）。Duck 持有 FlyBehavior 接口而不是 FlyWithWings 具体类，这两条原则在这里是同一份代码的两个侧面。
  • 关联知识点：开放关闭原则（OCP）：组合让"新增功能"变成"新增一个实现类 + 注入"，完全不用修改宿主类的代码——这正是 OCP 所要求的"对扩展开放，对修改关闭"。继承之所以容易违反 OCP，正是因为超类的任何改动都会波及所有子类；组合把这种波及范围缩小到了具体依赖关系上。
  • 关联知识点：策略模式（Strategy Pattern）：本文末尾已经指出，鸭子案例的组合结构本质就是策略模式：Context 持有 Strategy 接口，具体策略可自由替换、独立扩展。策略模式可以理解为"多用组合少用继承"这条原则在"算法/行为可替换"场景下的标准落地形态。
  • 关联知识点：装饰者模式（Decorator Pattern）：装饰者模式是组合思想的另一种典型应用——通过持有并包装一个同类型对象来动态叠加职责，而不是靠继承出一堆"功能组合类"（正如本文提到的 BufferedInputStream 包装 FileInputStream）。装饰者和策略模式都印证了组合能应对继承在多维度变化下的类爆炸问题。
  • 关联知识点：Clean Architecture 的组件依赖原则：Clean Architecture 强调用组件间的接口依赖（组合）而不是继承来搭建系统边界，因为继承会让组件在编译期紧密耦合，而接口 + 运行时注入的组合方式能让各层独立编译、独立部署、独立演化——这是"多用组合少用继承"在架构粒度上的自然延伸。
