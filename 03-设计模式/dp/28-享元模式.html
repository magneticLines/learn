享元模式 Flyweight

当系统中存在大量相似对象时，把它们可共享的部分提取出来只存一份，多个使用场景共享同一个实例，以此大幅节省内存。

意图 / 定义

GoF Flyweight Pattern
运用共享技术，有效支持大量细粒度对象的复用。
Use sharing to support large numbers of fine-grained objects efficiently.

Flyweight 直译"轻量级"，源自拳击的"蝇量级"。核心手段是共享：把对象状态拆成"内部状态（可共享、不变）"和"外部状态（随上下文变化、由客户端传入）"，只为内部状态创建有限的几个共享实例。

解决什么问题、为何这么设计

当程序需要创建海量对象，而这些对象之间有大量重复数据时，内存会被同样的内容反复占用。例如文档里有十万个字符 'a'，如果每个字符都是一个完整对象（含字体、字号、字形数据），内存瞬间爆掉。

享元模式的洞察是：这十万个 'a' 的字形数据是完全一样的（内部状态），变化的只是它们在文档中的位置（外部状态）。于是：

  • 内部状态（Intrinsic）：与上下文无关、可共享、存在享元对象内部。如字符的字形、颜色棋子的形状。
  • 外部状态（Extrinsic）：随场景变化、不可共享、由客户端在调用时传入，不存在享元里。如字符的坐标位置。

只为每种内部状态创建一个共享实例，外部状态作为方法参数传进来。这样 26 个字母只需 26 个享元对象，无论文档多大。

生活类比：围棋棋子

一盘围棋有几百颗棋子，但棋子本身只有"黑"和"白"两种（内部状态——颜色、形状都一样，可共享）。真正区分每颗棋子的是它落在棋盘哪个交叉点（外部状态）。没必要为每个落子位置都造一颗新棋子，只要两个共享的棋子对象，配合棋盘坐标即可。这正是享元的思路。

反例 → 问题 → 正例

❌ 每个棋子都 new 一个完整对象
class Stone {
  String color;   // 重复存储
  int[] shape;    // 大数组，重复!
  int x, y;       // 位置

  Stone(String color,
        int x, int y) {
    this.color = color;
    this.shape = loadShape(color);
    this.x = x; this.y = y;
  }
}
// 300 颗棋子 = 300 份 shape 数组
// 同色棋子的 shape 完全一样，
// 却被重复存了上百遍 —— 浪费内存
✅ 享元：同色共享，位置外置
class Stone {            // 享元
  final String color;   // 内部状态
  final int[] shape;    // 内部状态
  Stone(String c){ color=c;
    shape=loadShape(c); }
  // 外部状态 x,y 作为参数传入
  void drawAt(int x,int y){...}
}
// 工厂保证同色只有一个实例
Stone black = StoneFactory.get("黑");
black.drawAt(3, 4);
black.drawAt(5, 9); // 复用同一对象
// 300 颗棋子只占 2 份 shape

问题所在：反例为每颗棋子都复制了一份不变的 shape 数组，重复数据撑爆内存。正例把不变的内部状态共享、把可变的位置外置，对象总数从"棋子数"降到"颜色数"。

享元工厂：用缓存保证内部状态相同的对象只创建一次
class StoneFactory {
    // 缓存池：key=内部状态，value=共享享元
    private static final Map<String, Stone> pool = new HashMap<>();

    public static Stone get(String color) {
        // 已有就复用，没有才创建 —— 共享的关键
        return pool.computeIfAbsent(color, Stone::new);
    }
}

Stone a = StoneFactory.get("黑");
Stone b = StoneFactory.get("黑");
System.out.println(a == b);  // true —— 是同一个对象！

角色 / 结构

角色 职责 对应上例
Flyweight（享元） 含内部状态，提供接收外部状态的操作方法 Stone
FlyweightFactory（享元工厂） 用池缓存享元，保证内部状态相同的对象只有一个 StoneFactory
Client（客户端） 持有外部状态，调用时传给享元 调用 drawAt(x,y) 的代码
内部状态 / 外部状态 可共享不变 / 随上下文变化由外部传入 color、shape / x、y

JDK 中的真实应用

JDK 里有两个教科书级的享元实例，也是面试常考点：

Integer 缓存：[-128, 127] 的 Integer 被共享
Integer a = 100, b = 100;
System.out.println(a == b);   // true —— 命中 IntegerCache 共享对象

Integer c = 200, d = 200;
System.out.println(c == d);   // false —— 超出缓存范围，各 new 一个
// 自动装箱 Integer.valueOf() 内部就是享元工厂：
// 范围内返回缓存实例，范围外才 new Integer()
来源 享元体现
Integer.valueOf() 缓存 [-128, 127]，是享元工厂的标准实现。Long、Short、Byte、Character 同理
String 常量池 字面量字符串共享同一实例："ab" == "ab" 为 true。String.intern() 手动入池
Boolean.valueOf() 始终返回 Boolean.TRUE/FALSE 两个共享实例
线程池 / 连接池 广义上复用昂贵对象，思想相通（严格说是对象池，但同源）

常见陷阱

享元必须是不可变的（内部状态）：既然多处共享同一个实例，任何一方修改了它的内部状态，就会污染所有使用者。所以享元的内部状态字段应全部 final，不提供 setter。这也是 String、Integer 都不可变的原因。
Integer 缓存导致的 == 陷阱：初学者最易踩——Integer a=127,b=127; a==b 是 true，但 128==128 是 false。原因正是享元缓存只覆盖 [-128,127]。比较包装类的值永远用 equals()，不要用 ==。
不是所有重复对象都该享元：享元用"共享"换"内存"，代价是引入工厂、区分内外部状态的复杂度，以及外部状态每次传参的开销。只有当"相似对象数量巨大 + 重复数据占内存可观"时才划算；对象不多时纯属过度设计。

易混模式对比

模式 目的 与享元的区别
享元 Flyweight 共享细粒度对象省内存 共享不可变实例，区分内外部状态；通常多个共享实例（每种内部状态一个）
单例 Singleton 保证全局唯一实例 只有一个实例且为全局访问；享元是"有限多个、按内部状态分类"
对象池 Object Pool 复用昂贵对象（连接、线程） 池里对象可变、用完归还、状态会被重置；享元对象不可变、永久共享、不归还
原型 Prototype 复制对象 原型为每次需求造新对象；享元恰恰相反，尽量不造、复用同一个
面试高频："Integer a=127,b=127 为什么 a==b 是 true，128 却是 false？"——考的就是享元模式：Integer.valueOf() 对 [-128,127] 做了缓存共享，超出范围则新建对象。引申问题：为什么享元对象必须不可变？因为共享实例一旦被改会污染所有引用方。

小结

  • 享元模式 = 把大量相似对象的可共享内部状态提取出来共用一份，用工厂 + 缓存保证只创建有限实例。
  • 关键是拆分内部状态（可共享、不变、存在享元内）和外部状态（随上下文变、由客户端传入）。
  • 享元对象必须不可变，否则共享会引发污染。
  • JDK 落地：Integer.valueOf() 缓存、String 常量池、Boolean.valueOf()。
  • 记住经典陷阱：Integer 的 == 只在 [-128,127] 成立，比较值要用 equals()。
