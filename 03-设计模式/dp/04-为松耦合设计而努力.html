为松耦合设计而努力

两个对象交互，但彼此对对方知道得越少越好。耦合度决定了系统的弹性上限——耦合越紧，改动波及越广，弹性越低。

原则定义

Head First 设计原则 #4
为交互对象之间的松耦合设计而努力。
Strive for loosely coupled designs between objects that interact.

书中把这条原则放在观察者模式（Ch.2）之后，因为观察者模式是松耦合思想的最典型演示：Subject（主题）和 Observer（观察者）之间交互，却几乎互不了解对方。

什么是耦合

耦合（Coupling）衡量的是：改变 A 时，需要同时改变 B 的程度。耦合越紧，A 和 B 的命运绑定得越死。

紧耦合
A 直接 new B
持有具体类引用
知道 B 的内部状态
中等耦合
A 持有 B 的接口
通过方法调用交互
不知道 B 的实现
松耦合
A 不知道 B 是谁
只知道"有人会响应"
通过事件/回调交互

一个直觉：你改了一行，需要改多少地方？

紧耦合系统里，改一个类名、方法签名或返回类型，往往要追着改几十个调用方。松耦合系统里，变化被隔离在边界内，调用方感知不到。

紧耦合是什么样的

❌ 紧耦合：气象站直接调用各显示设备
class WeatherStation {
    // 直接持有每个显示器的具体引用
    CurrentConditionsDisplay  currentDisplay;
    StatisticsDisplay         statsDisplay;
    ForecastDisplay           forecastDisplay;

    void measurementsChanged() {
        // 每增加一个显示器，就得改这里
        currentDisplay.update(temp, humidity, pressure);
        statsDisplay.update(temp, humidity, pressure);
        forecastDisplay.update(temp, humidity, pressure);
        // 如果某个显示器不存在了，还要手动删这行
    }
}

这段代码的问题：

  • WeatherStation 必须知道所有显示器的具体类型——知道得太多
  • 每增加/删除一个显示器，都要修改 WeatherStation 本身——违反开放关闭原则
  • 显示器无法在运行时动态注册/注销
  • WeatherStation 和显示器之间形成了 N 条硬连线，任何一端修改都可能波及对方

观察者模式：松耦合的教科书案例

书中 Ch.2 的气象站案例就是为了演示这条原则。把 WeatherStation 改成观察者模式之后：

Subject（WeatherData）
只知道：持有一个 Observer 列表
↓ notifyObservers() — 统一调用 update()
CurrentConditionsDisplay
Observer 接口
StatisticsDisplay
Observer 接口
任意新增显示器
只需实现接口
观察者模式骨架——Subject 和 Observer 互不认识对方
// Subject 只依赖这个接口，不认识任何具体 Observer
interface Observer {
    void update(float temp, float humidity, float pressure);
}

class WeatherData {
    private List<Observer> observers = new ArrayList<>();

    // 运行时随意注册/注销，WeatherData 完全不关心是谁
    public void registerObserver(Observer o) { observers.add(o); }
    public void removeObserver(Observer o) { observers.remove(o); }

    private void notifyObservers() {
        // 只调用接口方法，不知道也不在乎对面是谁
        for (Observer o : observers) o.update(temp, humidity, pressure);
    }
}

// 新加一个显示器：只需实现接口，WeatherData 一行不改
class HeatIndexDisplay implements Observer {
    public void update(float t, float h, float p) {
        /* 计算热指数并显示 */
    }
}

现在 Subject 和 Observer 之间只剩一条细线：一个接口。它们互相知道的最少：

  • Subject 只知道：列表里有一堆实现了 Observer 接口的东西
  • Observer 只知道：Subject 会在某个时刻调用 update()
  • 两边都不知道对方的类名、内部实现、生命周期

松耦合的四个具体特征

书中通过观察者模式总结了松耦合系统的四条性质，这四条是判断"是否真的松耦合"的标准：

特征 在观察者模式中的体现
Subject 不知道 Observer 的具体类型 Subject 只持有 Observer 接口列表，CurrentConditionsDisplay 还是 HeatIndexDisplay 它一无所知
可以随时增删 Observer 运行时 register/remove，Subject 行为不受影响
增加新 Observer 不需要修改 Subject 新建一个实现 Observer 的类，注册进去即可——对 Subject 完全透明
Subject 和 Observer 可以独立复用 WeatherData 可以用在完全不同的 UI 场景；同一个显示器可以观察不同的数据源

耦合的三大来源

理解了"什么是松耦合"，反过来认识耦合的根源，才能在设计时主动规避：

耦合来源 表现 松耦合手段
类型耦合 A 直接依赖 B 的具体类（new B()、声明为 B 类型） 针对接口编程；工厂/DI 创建对象
数据耦合 A 知道 B 的内部字段或状态结构，直接读写 封装，只通过方法暴露行为；迪米特法则
时序耦合 A 必须在 B 之前/之后调用，顺序错误就崩 事件驱动、消息队列；Subject 主动推送给 Observer

生活类比：新闻订阅

报社（Subject）出版报纸，不需要知道订阅者是谁、住哪里、每天几点看报。订阅者（Observer）也不需要每小时打电话问"有新闻了吗"。双方通过"订阅关系"这一层接口交互：

  • 报社出版 → 自动推送给所有订阅者（notifyObservers）
  • 新读者订阅：报社照常运营，只是名单里多一人
  • 老读者退订：报社照常运营，只是名单里少一人
  • 报社换了印刷厂：读者感知不到任何变化

这就是松耦合的核心价值：变化被隔离在边界内，不传染给另一侧。

推模型 vs 拉模型：松耦合的两种姿势

书中还讨论了 Observer 模式里数据传递的两种方式，它们在耦合程度上有微妙差别：

推模型（Push）——较紧
// Subject 把数据主动推给 Observer
void update(float temp,
            float humidity,
            float pressure);

// 问题：Observer 不需要的数据
// 也被迫接收；Subject 改了数据
// 结构，接口签名就要变
拉模型（Pull）——更松
// Observer 只收到通知，
// 自己决定要拉什么数据
void update(Subject s);

// Observer 只取自己需要的
float t = s.getTemperature();
// Subject 新增字段不影响
// 不关心的 Observer
Java 标准库的选择：java.util.Observer（已废弃）和现代事件系统都倾向于拉模型——update(Observable o, Object arg)，Observer 自己决定从 o 取什么数据，比推模型耦合更低。

松耦合不是免费的

过度追求松耦合会带来真实的成本，需要权衡：

代价 说明
间接性增加，代码难追 调用链变长：A → 接口 → 谁实现了接口 → B。IDE 的"Go to implementation"成了必备，新人上手成本上升
调试困难 事件/回调系统里，一个通知触发了哪些 Observer 的哪些动作，调用栈不直观
性能开销 Observer 列表遍历、接口调用的间接开销；在高频通知场景（如每帧触发）需要评估
意外副作用 Subject 通知时不知道 Observer 会做什么——某个 Observer 的 update() 里反过来修改 Subject 状态，可能引发循环
实用准则：两个对象如果天然属于同一职责域，紧耦合没有问题（同一个类的方法间）。松耦合的价值在于跨越边界的交互——模块间、层间、服务间。不要为了"看起来松耦合"把同一个东西拆成事件+监听。

与其他原则和模式的关系

原则 / 模式 与松耦合的关系
针对接口编程（#2） 是松耦合最基础的手段——只依赖接口，类型耦合降到最低
多用组合（#3） 组合通过接口连接，天然是松耦合；继承暴露超类内部，天然是紧耦合
最少知识原则（Ch.7） 松耦合的具体量化版本：一个对象只和它最直接的朋友交互，不通过朋友的朋友
观察者模式（Ch.2） 松耦合原则在"一对多通知"场景下的标准实现
Spring 依赖注入 把对象的创建和连接权从代码里移走，由框架在运行时注入——系统级别的松耦合实践
微服务 / 消息队列 同一思想在架构层面的体现：服务间通过消息接口交互，不直接调用对方接口实现
面试高频考点：问"什么是高内聚低耦合"时，很多人只能背定义。结合观察者模式说出"Subject 和 Observer 通过接口交互、运行时动态注册、互不知道对方具体类型"这个结构，才是真正展示了理解。

一个快速自测：你的设计够松吗

改动一个类时，对着这几个问题检查：

  1. 我必须修改哪些不属于这个类职责的地方？（越多 = 越紧耦合）
  2. 这个类的调用方能在不重新编译自己的情况下看到变化生效吗？
  3. 增加一个新实现，需要修改几个已有文件？（理想答案：零个）
  4. 能在不影响其他组件的情况下替换掉这个类吗？

如果四个问题的答案都令人满意，设计的耦合度基本合理。
