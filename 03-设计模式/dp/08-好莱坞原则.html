好莱坞原则

别调用我们，我们会调用你。让高层组件掌握"何时调用"的主动权，低层组件只管准备好被调用。

原则定义

Head First 设计原则 #8
别调用我们，我们会调用你。
The Hollywood Principle — Don't call us, we'll call you.

这句话来自好莱坞试镜的套路：新人面试完想追问进展，得到的回复永远是"别打给我们，有消息我们打给你"。Head First 在模板方法模式（Template Method，Ch.8）一章引入它——模板方法正是这句话的代码化身。

它解决什么问题

当低层组件直接调用高层组件时，容易陷入一种叫"依赖腐烂（dependency rot）"的困境：高层依赖低层，低层又回头依赖高层，旁边的组件还依赖这两者……依赖关系盘根错节，谁也说不清系统到底是怎么运转的，改动一处牵一发动全身。

好莱坞原则给出一个清晰的方向约定：

  • 低层组件可以"挂接"到系统里，但不能主动去调用高层组件
  • 由高层组件决定何时、以何种方式使用低层组件
  • 低层组件只需说"我在这儿，需要我的时候叫我"，绝不反过来戳高层

这样一来，依赖方向被理顺成单向的：高层 → 低层。低层不再回指高层，依赖腐烂被切断。

生活类比：恒温器与空调

你不会让空调每分钟跑去问恒温器"现在该开机了吗"（低层戳高层）。正确的关系是：空调（低层）把自己注册给恒温器（高层），然后安静待命；恒温器检测到温度超标时，主动调用空调的 turnOn()。空调从不主动联系恒温器——"别调用我，我会调用你"。控制权牢牢握在高层手里。

反例 → 问题 → 正例

以"冲泡饮料"为例。咖啡和茶的流程几乎一样（烧水 → 冲泡 → 倒杯 → 加料），只有"冲泡"和"加料"两步不同。

❌ 反例：每个子类各自复制整套流程，还反过来调度
class Coffee {
    void prepare() {                 // 流程被复制
        boilWater();
        brewCoffeeGrinds();          // 不同点
        pourInCup();
        addSugarAndMilk();           // 不同点
    }
}
class Tea {
    void prepare() {                 // 又复制一遍同样骨架
        boilWater();
        steepTeaBag();
        pourInCup();
        addLemon();
    }
}
// 流程顺序散落各处，想统一改流程要改每个子类

问题：流程骨架被重复，且控制权分散在每个子类手里。用模板方法把骨架上移到父类，让父类掌控流程、在需要时"回调"子类的钩子方法：

正例：模板方法——父类定流程，"我会调用你"
abstract class CaffeineBeverage {
    // 模板方法：流程骨架由高层(父类)固定并掌控，final 不许子类改
    final void prepare() {
        boilWater();
        brew();            // ←── 由父类在这一刻"回调"子类
        pourInCup();
        if (customerWantsCondiments())  // 钩子：父类决定是否调用
            addCondiments();
    }

    // 低层子类只负责"被调用时怎么做"，不掌控流程
    abstract void brew();
    abstract void addCondiments();

    // 钩子方法：子类可选地覆盖，但是否触发仍由父类决定
    boolean customerWantsCondiments() { return true; }

    void boilWater() { /* 公共步骤 */ }
    void pourInCup() { /* 公共步骤 */ }
}

class Tea extends CaffeineBeverage {
    void brew() { /* 泡茶包 */ }       // 我准备好了，等父类来调
    void addCondiments() { /* 加柠檬 */ }
}
看控制方向：子类 Tea 从不调用 prepare() 的任何步骤，它只是把自己的实现准备好。是父类的 prepare() 在恰当的时刻回调 brew() 和 addCondiments()。这就是"别调用我，我会调用你"。

与控制反转（IoC）的联系

好莱坞原则其实就是控制反转（Inversion of Control, IoC）在面向对象设计层面的通俗表述。"反转"反的是控制流的发起方：

传统控制流 控制反转 / 好莱坞
你写的代码主动调用库/框架 框架掌握主流程，在合适时回调你写的代码
"我决定什么时候做什么" "框架决定什么时候叫我做什么"
例：你调用 Collections.sort() 例：你实现 Comparator，由 sort 在比较时回调它

所有"框架"都建立在这条原则上：你不调用框架，你把代码挂进框架的扩展点，框架在它的生命周期里调用你。Spring 的 Bean 生命周期回调、Servlet 容器调用你的 doGet()、JUnit 调用你的 @Test 方法——全是好莱坞原则。这正是"库（library)"和"框架（framework)"的本质区别：库被你调用，框架调用你。

与其他原则和模式的关系

原则 / 模式 关系
模板方法模式（Ch.8） 好莱坞原则的代表模式：父类定流程，回调子类的钩子
依赖倒置原则（Ch.4） 两者都在"理顺依赖方向"。DIP 关注依赖谁（依赖抽象），好莱坞关注谁调用谁（高层调低层）。常被一起讨论，但角度不同
观察者模式（Ch.2） Observer 注册后等待 Subject 回调 update()，也是"别调用我，我会调用你"
工厂方法模式 父类的流程在需要对象时回调子类的工厂方法来创建——同样的回调结构
Spring / Servlet / JUnit 框架持有主控制流，在生命周期节点回调你的代码，是好莱坞原则的工程化体现
面试高频考点："模板方法和策略模式有什么区别？" 模板方法用继承，骨架在父类、变化点在子类，符合好莱坞原则（父类回调子类）；策略模式用组合，把整个算法委托给可替换的策略对象。前者控制权在父类，后者在持有策略的上下文。

常见误区与陷阱

误区一：把好莱坞原则等同于依赖倒置原则。它们相关但不相同。DIP 是关于"该依赖抽象还是具体"的依赖关系原则；好莱坞原则是关于"谁来发起调用"的控制流原则。一个低层组件可以同时违反或同时遵守两者，分开理解才不混乱。
误区二：钩子方法里偷偷反调高层。子类的钩子方法里如果又去主动调用父类掌控的流程方法（比如在 brew() 里调 prepare()），就破坏了单向控制流，可能造成递归或状态错乱。低层组件要克制——被调用时老老实实做事，不要反客为主。

小结

  • 好莱坞原则：别调用我，我会调用你——控制权归高层，低层等待被回调
  • 它解决"依赖腐烂"：把高低层双向纠缠的依赖理顺成单向（高层 → 低层）
  • 模板方法模式是它的代表：父类固定流程骨架，在钩子点回调子类
  • 它是控制反转（IoC）在 OO 设计层面的说法，也是"框架 vs 库"的分水岭
  • 与 DIP 相关但不同：DIP 管"依赖谁"，好莱坞管"谁调用谁"
