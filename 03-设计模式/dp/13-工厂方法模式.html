工厂方法模式 Factory Method

定义一个创建对象的接口，但由子类决定要实例化的类是哪一个——工厂方法让类把实例化推迟到子类。

意图与定义

GoF Factory Method Pattern
把"创建哪个具体对象"的决定权下放给子类。
Define an interface for creating an object, but let subclasses decide which class to instantiate.

工厂方法模式遵循依赖倒置原则（DIP）：高层模块不应依赖具体类，二者都应依赖抽象。它把对象的创建（new XXX()）从使用对象的代码里抽出来，封装进一个可被子类覆盖的"工厂方法"，让超类只面向抽象产品工作，由子类决定造出什么具体产品。

它解决什么问题

当一段流程逻辑是固定的，但流程中要创建的具体对象会随场景变化时，直接 new 具体类会让这段流程被钉死在具体类型上：

  • 新增一种产品就要回去改创建代码，违反开放关闭原则
  • 高层流程直接 new 低层具体类，形成自上而下的紧耦合，难以替换、难以测试

工厂方法的设计：超类定义好整套流程（如"做披萨"的步骤），把"造哪种披萨"这一步声明为抽象方法，自己不实现；各子类（各地分店）覆盖这个工厂方法，返回本地风味的产品。流程复用在超类，具体类型决定权在子类。

生活类比：连锁披萨店

总店规定了统一的"订购流程"：和面 → 烘烤 → 切块 → 装盒，这套流程所有分店一样。但"具体做哪种披萨"由各地分店决定——纽约店做纽约风味，芝加哥店做厚底风味。总部不关心分店具体怎么造披萨，只要求"给我一个 Pizza"。分店就是工厂方法的"子类"，它决定实例化哪种 Pizza。

反例 → 问题 → 正例（披萨店案例）

❌ 反例：流程里硬编码 new + 条件分支
class PizzaStore {
    Pizza orderPizza(String type) {
        Pizza pizza;
        // 创建逻辑和流程逻辑混在一起，每加一种披萨都要改这里
        if (type.equals("cheese"))      pizza = new CheesePizza();
        else if (type.equals("veggie")) pizza = new VeggiePizza();
        else if (type.equals("clam"))   pizza = new ClamPizza();

        pizza.prepare(); pizza.bake(); pizza.cut(); pizza.box();
        return pizza;
    }
}

问题在哪：

  • 创建逻辑（if-else new）和不变的流程（prepare/bake/cut/box）耦合在同一个方法
  • 每加一种披萨、纽约店和芝加哥店要造不同风味，都得改这段 if-else——违反 OCP
  • PizzaStore 直接依赖一堆具体 Pizza 类

正例：把流程留在抽象超类，把"造披萨"声明为抽象工厂方法，交给子类实现。

抽象超类：固化流程，下放创建
abstract class PizzaStore {
    // 模板式流程：所有店一样，复用在超类
    public final Pizza orderPizza(String type) {
        Pizza pizza = createPizza(type);   // 工厂方法：造什么交给子类
        pizza.prepare(); pizza.bake();
        pizza.cut(); pizza.box();
        return pizza;
    }

    // ★ 工厂方法：超类不实现，由子类决定实例化哪个具体类
    protected abstract Pizza createPizza(String type);
}
具体子类：决定造哪种产品
class NYPizzaStore extends PizzaStore {
    protected Pizza createPizza(String type) {
        if ("cheese".equals(type)) return new NYStyleCheesePizza();
        if ("clam".equals(type))   return new NYStyleClamPizza();
        return null;
    }
}

class ChicagoPizzaStore extends PizzaStore {
    protected Pizza createPizza(String type) {
        if ("cheese".equals(type)) return new ChicagoStyleCheesePizza();
        return null;
    }
}

// 使用：流程统一，产品风味由具体店决定
PizzaStore store = new NYPizzaStore();
Pizza p = store.orderPizza("cheese");   // 拿到纽约风味芝士披萨
关键：超类 orderPizza() 全程只和抽象 Pizza 打交道，从不知道具体类名。新增一家"加州店"只需写个新子类覆盖 createPizza()，超类一行不动——这就是对扩展开放、对修改关闭。

角色与结构

角色职责披萨案例对应
Product（抽象产品）工厂方法创建的对象的接口Pizza
ConcreteProduct具体产品NYStyleCheesePizza 等
Creator（抽象创建者）声明工厂方法，常含使用产品的业务流程PizzaStore
ConcreteCreator覆盖工厂方法，返回某个具体产品NYPizzaStore / ChicagoPizzaStore

JDK 与 Spring 中的真实应用

  • JDK Collection.iterator()：是工厂方法的范例——每个集合类自己决定返回哪种 Iterator 具体实现，调用方只拿到 Iterator 接口。
  • JDK Calendar.getInstance()、NumberFormat.getInstance()：根据 Locale 等条件返回不同的具体子类实例。
  • JDBC Connection.createStatement()：不同数据库驱动的 Connection 实现返回各自的 Statement 实现。
  • Spring FactoryBean：实现 getObject() 工厂方法来定制复杂 Bean 的创建逻辑，容器拿到的是 getObject() 返回的产品。

常见误区与陷阱

误区一：把"简单工厂"当成工厂方法。简单工厂是用一个静态方法 + if/switch 集中造对象（不是 GoF 模式）；工厂方法是用继承，把创建抽象成方法、由子类覆盖来决定。前者加产品要改工厂代码，后者加产品是加子类。
误区二：为创建而过度抽象。如果产品种类稳定、只有一两种，直接 new 或一个简单工厂就够了。每种产品都强行配一个 Creator 子类，会带来一堆只为"造一个对象"而存在的类，得不偿失。

与其他知识点的联系

  • 依赖倒置原则（DIP）：工厂方法是 DIP 的典型应用——PizzaStore.orderPizza() 全程依赖抽象的 Pizza，从不依赖任何具体披萨类，高层流程和低层实现都倒过来依赖同一个抽象，而不是高层直接依赖低层细节。
  • 抽象工厂模式：二者常被放在一起对比记忆——工厂方法用继承（子类覆盖方法）造一个产品，抽象工厂用组合（持有工厂对象）造一族配套产品；实践中抽象工厂内部的每个创建方法往往就是用工厂方法实现的，二者是嵌套关系而非互斥关系。
  • 模板方法模式：PizzaStore.orderPizza() 本身就是一个模板方法——固定了"造披萨→烤→切→装盒"的流程骨架，只把 createPizza() 这一步留空给子类填——工厂方法模式常常就内嵌在模板方法里，是"变与不变分离"思想的另一处体现。
  • 简单工厂（非 GoF 模式）：简单工厂用一个静态方法加 if/switch 集中创建对象，新增产品要改工厂代码；工厂方法通过继承把决定权下放给子类，新增产品是加子类而不改已有代码——两者常被面试混淆，区分点就在"是否符合 OCP"。
  • Spring FactoryBean 与容器：Spring 允许通过实现 FactoryBean.getObject() 来定制某个 Bean 的创建逻辑，容器拿到的是这个工厂方法的返回值而非直接 new 出来的对象，这与 PizzaStore 只关心 createPizza() 返回的 Pizza、不关心具体类是同一种解耦思路。

小结

  • 工厂方法 = 超类定义流程并声明抽象工厂方法，子类覆盖该方法决定造哪个具体产品。
  • 它解耦了"使用产品的流程"与"创建产品的具体类型"，遵循 DIP 与 OCP，新增产品靠加子类。
  • 与简单工厂的本质区别：靠继承下放决定权 vs 靠条件分支集中创建。
  • JDK 的 iterator()、Calendar.getInstance()、Spring 的 FactoryBean 都是它的体现。
