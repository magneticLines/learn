策略模式 Strategy

定义一族算法，把它们各自封装起来，并让它们可以互相替换；让算法的变化独立于使用算法的客户。

意图与定义

GoF Strategy Pattern
把"会变化的行为"抽成接口，封装成一族可互换的算法。
Define a family of algorithms, encapsulate each one, and make them interchangeable.

策略模式是《Head First 设计模式》开篇第一个模式，因为它最直白地演示了三条核心设计原则：找出变化并封装、针对接口编程、多用组合少用继承。它把"一个对象在不同情况下要表现出不同行为"这件事，从 if-else 和继承里解放出来，变成"持有一个可替换的算法对象"。

它解决什么问题

当一个类的某个行为有多种实现方式，而且这些方式会随时间增加、需要在运行时切换时，如果用继承去覆盖或用一大堆条件分支去判断，会带来两个顽疾：

  • 继承导致行为僵化：行为写死在父类/子类里，运行时无法更换，且每个子类被迫继承自己不需要的行为
  • 条件分支导致修改扩散：每加一种算法就要回去改那段 if-else，违反开放关闭原则

策略模式的设计思路是：把"做什么"（算法）和"谁来用"（上下文）解耦。上下文不关心算法怎么实现，只持有一个算法接口的引用，运行时塞进哪个实现就用哪个。

生活类比：出行方式

你要从家到公司（上下文 Context），可以选择步行、骑车、打车、坐地铁（一族策略 Strategy）。无论用哪种，你的目标"到达公司"不变，变的只是"出行算法"。你不会为了换出行方式而换一个"人"，而是同一个你，今天选这个、明天选那个——这就是组合而非继承、运行时可替换。

反例 → 问题 → 正例（鸭子案例）

书里的经典案例：一个鸭子模拟系统。最初设计者用继承——所有鸭子继承 Duck 超类。

❌ 反例：用继承塞行为
abstract class Duck {
    void quack() { System.out.println("呱呱"); }   // 所有鸭子都会叫
    void fly()   { System.out.println("飞起来"); }   // 灾难：橡皮鸭也会飞了！
    abstract void display();
}

class RubberDuck extends Duck {
    void display() { System.out.println("橡皮鸭"); }
    // 橡皮鸭不会飞，被迫重写 fly() 让它什么都不做
    void fly()   { /* 覆盖成空，治标不治本 */ }
    void quack() { System.out.println("吱吱"); }     // 又得重写
}

问题在哪：

  • 把 fly() 放进超类，导致每个不会飞的子类都要手动覆盖成空——继承让行为强加给了所有子类
  • 新增"诱饵鸭（不会飞、不会叫）"，又要重写两个方法；行为种类越多，重写越泛滥
  • 飞行/叫声的行为无法在运行时改变，也无法在多种鸭子间复用

正例：把"会变化的行为"——飞行与叫声——抽成接口，封装成一族策略，鸭子通过组合持有它们。

策略接口 + 一族算法实现
// ① 把"会变化的行为"抽成策略接口
interface FlyBehavior { void fly(); }
interface QuackBehavior { void quack(); }

// ② 每个算法各自封装，可互换
class FlyWithWings implements FlyBehavior {
    public void fly() { System.out.println("用翅膀飞"); }
}
class FlyNoWay implements FlyBehavior {
    public void fly() { /* 不会飞，什么都不做 */ }
}
Context 通过组合持有策略，并支持运行时替换
abstract class Duck {
    // ③ 组合：持有接口引用，而非把行为写死在继承里
    protected FlyBehavior flyBehavior;
    protected QuackBehavior quackBehavior;

    public void performFly() { flyBehavior.fly(); }   // 委托给策略
    public void performQuack() { quackBehavior.quack(); }

    // ④ 运行时换一套算法——继承做不到
    public void setFlyBehavior(FlyBehavior fb) { this.flyBehavior = fb; }
}

class RubberDuck extends Duck {
    public RubberDuck() {
        flyBehavior = new FlyNoWay();     // 橡皮鸭不会飞，装配即可，无需重写空方法
        quackBehavior = new Squeak();
    }
}
关键转变：从"鸭子是会飞的（IS-A 继承）"变成"鸭子有一个飞行行为（HAS-A 组合）"。HAS-A 比 IS-A 更灵活，因为持有的东西可以换。

角色与结构

角色职责鸭子案例对应
Strategy（抽象策略）定义算法族的公共接口FlyBehavior / QuackBehavior
ConcreteStrategy（具体策略）实现接口，封装某一种算法FlyWithWings / FlyNoWay
Context（上下文）持有策略引用，把请求委托给策略Duck
Client（客户）创建具体策略并装配进 Context构造方法 / setFlyBehavior() 调用方

JDK 与 Spring 中的真实应用

  • JDK Comparator：Collections.sort(list, comparator) 中，排序算法固定，但"如何比较"由传入的 Comparator 决定——比较逻辑就是可替换的策略。Lambda 让策略写起来更轻。
  • JDK ThreadPoolExecutor 的拒绝策略：RejectedExecutionHandler 是接口，AbortPolicy/CallerRunsPolicy/DiscardPolicy 是一族可互换的策略实现。
  • Spring Resource 加载、InstantiationStrategy：Bean 实例化用 SimpleInstantiationStrategy 还是 CglibSubclassingInstantiationStrategy，由策略接口切换。
  • 业务层常见用法：支付方式（微信/支付宝/银行卡）、促销规则、文件解析器，都是策略模式的天然场景。

常见误区与陷阱

陷阱一：策略类爆炸。算法很多时会产生大量小类。对无状态、逻辑简单的策略，Java 8 起可用 Lambda 或方法引用替代具体类；对真正复杂、有状态的算法才独立成类。
陷阱二：客户必须知道所有策略的区别。策略模式把"选哪个算法"的责任推给了客户。如果客户也不该知道，应配合工厂或配置/枚举映射来选择策略，否则只是把 if-else 从 Context 挪到了客户端。

与易混模式的对比

对比项策略模式 Strategy简单工厂 Simple Factory
关注点封装算法/行为，让它们可互换封装对象的创建，让客户不直接 new
类别行为型创建型（严格说不算 GoF 模式）
运行时对象持有策略，可随时替换工厂只负责"生产"，生产完就退场
关系二者常配合：用简单工厂来创建策略，再把策略装配进 Context。一个管"造"，一个管"用"。
策略模式状态模式 State
客户主动选择并设置算法状态随对象内部条件自动流转
各策略之间互不感知各状态常知道下一个状态是谁
解决"同一件事的多种做法"解决"对象在不同状态下行为不同"

小结

  • 策略模式 = 找出变化的行为 + 抽成接口 + 用组合持有 + 运行时可替换。
  • 它用 HAS-A 组合取代 IS-A 继承，避免子类被迫继承不需要的行为，避免条件分支随算法增多而膨胀。
  • JDK 的 Comparator、线程池拒绝策略是教科书级应用。
  • 常与工厂配合：工厂负责造策略，Context 负责用策略。
