07. 信息隐藏与信息泄漏

John Ousterhout 认为信息隐藏是实现深模块最重要的手段——把决策锁在模块内部，让外部世界什么都不需要知道。

作者的核心主张

Ousterhout 在第 5 章开门见山：信息隐藏（Information Hiding）是降低软件复杂性的最有力工具之一。 其本质是：把实现某个功能所需的知识（知识 = 决策 + 假设 + 算法选择）锁进单个模块，不让它渗透到系统其他地方。

为什么这样设计更好？答案来自复杂性的根源——依赖（dependencies）。 当模块 A 知道模块 B 的内部细节时，A 就依赖了 B 的实现；B 一旦改动，A 也得跟着改。 信息隐藏切断这条依赖链：外部只看到稳定的接口，内部细节随时可换，改动不扩散。

一句话记忆：接口是模块向外暴露的"承诺"，实现是模块内部的"秘密"。 好的设计让承诺尽可能小，秘密尽可能多。

信息泄漏是什么

信息泄漏（Information Leakage）是信息隐藏的反面： 某个设计决策被多个模块知晓并依赖，导致一处改动需要同步修改多处。 Ousterhout 把它列为最常见的红色警报之一。

泄漏有两种形态：

  • 显式泄漏（接口泄漏）：把内部细节直接写进方法签名或返回值。调用方被迫了解底层。
  • 隐式泄漏（时序泄漏）：两个方法必须按特定顺序调用才能正确工作，调用方需要了解这个约定。
信息泄漏的典型信号
├── 方法参数里出现实现层的类型（如 FileInputStream、ByteBuffer）
├── 调用方需要判断内部状态（如 isConnected() 后才能 send()）
├── 两个类共享同一个"隐含知识"（如都知道文件格式是 UTF-8）
└── 修改一个实现细节，需要同时改多个文件

反例 → 问题 → 正例

场景：读取配置文件

假设我们要封装"从文件读取应用配置"的功能。

反例 — 信息泄漏Java
// ❌ 接口暴露了"文件用 FileReader 读取"这一实现决策
public class ConfigLoader {

    // 调用方被迫传入 FileReader，意味着调用方必须知道：
    // 1. 配置来自文件（而非数据库、环境变量…）
    // 2. 必须自己负责打开/关闭 FileReader
    public Map<String, String> load(FileReader reader) throws IOException {
        Map<String, String> config = new HashMap<>();
        BufferedReader br = new BufferedReader(reader);
        String line;
        while ((line = br.readLine()) != null) {
            String[] parts = line.split("=", 2);
            if (parts.length == 2) config.put(parts[0].trim(), parts[1].trim());
        }
        return config;
    }
}

// 调用方代码 —— 已经"知道太多了"
FileReader fr = new FileReader("app.properties");
Map<String, String> cfg = new ConfigLoader().load(fr);
fr.close();  // 调用方还要管生命周期！
问题所在：方法签名里的 FileReader 把"从文件读"这个实现决策泄漏给了每一个调用方。 若将来改为从数据库或环境变量读取，所有调用方都得改。 更糟的是，调用方还要自己管 FileReader 的开关，资源泄漏风险暗藏其中。
正例 — 信息隐藏Java
// ✅ 接口只暴露"我需要一个路径"，内部怎么读是秘密
public class ConfigLoader {

    // 对外承诺：给我路径，给你配置 Map。仅此而已。
    public Map<String, String> load(String filePath) {
        Map<String, String> config = new HashMap<>();
        // try-with-resources：资源生命周期是内部秘密，外部无感知
        try (BufferedReader br =
                 Files.newBufferedReader(Path.of(filePath), StandardCharsets.UTF_8)) {
            br.lines()
              .filter(l -> !l.isBlank() && !l.startsWith("#"))
              .map(l -> l.split("=", 2))
              .filter(p -> p.length == 2)
              .forEach(p -> config.put(p[0].trim(), p[1].trim()));
        } catch (IOException e) {
            throw new RuntimeException("Failed to load config: " + filePath, e);
        }
        return config;
    }
}

// 调用方极其简单，对文件读取一无所知
Map<String, String> cfg = new ConfigLoader().load("app.properties");
改变了什么： FileReader、BufferedReader、UTF-8、资源关闭——这四个决策全部锁在了 ConfigLoader 内部。 未来改成读 YAML 或从环境变量合并，调用方代码零修改。

隐式泄漏：时序依赖

接口泄漏容易被发现，时序泄漏（Temporal Leakage）更隐蔽。 两个操作必须按顺序调用，本身就是一种"内部状态"被迫暴露给调用方。

反例 — 时序泄漏Java
// ❌ 调用方必须知道"先 connect，再 send"这条内部约定
public class MessageClient {
    private boolean connected = false;

    public void connect(String host, int port) { ...; connected = true; }

    public void send(String message) {
        // 调用方必须在外部保证先调了 connect()，否则 NPE / 状态错误
        if (!connected) throw new IllegalStateException("Not connected!");
        // ...
    }
}

// 调用方代码
MessageClient client = new MessageClient();
client.connect("localhost", 8080);  // 必须记得先调这一行！
client.send("hello");
正例 — 隐藏时序Java
// ✅ 把"连接"的知识封装进工厂方法，外部只看到"可用的客户端"
public class MessageClient {

    // 构造函数私有，强制通过工厂方法获取"已连接"的实例
    private MessageClient() {}

    public static MessageClient connect(String host, int port) {
        MessageClient c = new MessageClient();
        // 内部完成连接——调用方得到的对象天生就是"已连接"状态
        c.doConnect(host, port);
        return c;
    }

    public void send(String message) { /* 直接用，无需检查 */ }
    private void doConnect(String host, int port) { ... }
}

// 调用方：时序约定已消失，根本没机会"忘记 connect"
MessageClient client = MessageClient.connect("localhost", 8080);
client.send("hello");

关键指标对比

维度 信息隐藏（好） 信息泄漏（坏）
接口大小 最小化，只暴露必要的 暴露内部类型、状态、算法
调用方负担 不需要了解实现细节 需要了解顺序、格式、资源管理
修改影响范围 改动仅限模块内部 改动波及所有调用方
测试难度 只需针对接口编写测试 需要模拟内部细节才能测试
典型代码信号 String path、工厂方法 FileReader、isConnected() 暴露

实际开发落地技巧

技巧 1：用构造函数 / 工厂方法消灭时序泄漏

若一个对象需要初始化步骤，把步骤放进构造函数或静态工厂。 让对象"生来即可用"，调用方不可能拿到"未初始化"的实例。 这也是 Effective Java 条目 1 的背后逻辑。

技巧 2：参数类型越抽象越好

方法参数用 InputStream 而非 FileInputStream， 用 List 而非 ArrayList， 用 Path 而非 File（更现代）。 抽象层越高，暴露的实现细节越少。

技巧 3：同一知识只存一份（Single Source of Truth）

如果"文件编码是 UTF-8"这个知识在 A、B 两个类里都硬编码了，就已经泄漏。 应把它提炼为常量或配置，集中在一处。修改时只改一处，不会漏改。

技巧 4：私有化 + 最小可见性原则

字段默认 private，方法能 private 就不 package-private， 能 package-private 就不 public。 IDE 的"减小访问修饰符"重构是快速执行此原则的捷径。

技巧 5：警惕"过度分解"带来的泄漏

Ousterhout 特别指出：把一个完整操作拆成多个小方法，反而会制造时序泄漏。 readHeader() + readBody() 两个方法，要求调用方知道"先 Header 后 Body"的顺序； 而一个 readMessage() 才是真正的信息隐藏。 不是分得越细越好，而是每个方法完成一个完整的、有意义的操作。

常见陷阱：把"数据容器类"（如 UserDTO）的所有字段都加 getter/setter， 等同于把所有字段都设成了 public。 如果某字段只是内部状态，不提供 getter 才是正确的信息隐藏。

与深模块的联系

信息隐藏是实现深模块（deep module）的核心手段。 回忆深模块的定义：接口简单，功能强大。 接口之所以能简单，正是因为大量的实现知识被隐藏在模块内部。

深模块 = 大接口面积的"功能" / 小接口面积的"暴露"

┌─────────────────────────┐
│ 接口（对外暴露） │ <── 越小越好
├─────────────────────────┤
│ 实现（对外隐藏） │
│ · 算法选择 │
│ · 数据结构 │ <── 越多越好
│ · 错误处理细节 │
│ · 并发策略 │
└─────────────────────────┘

Unix 的 read(fd, buf, count) 系统调用是经典案例： 三个参数的接口背后隐藏了磁盘寻址、缓冲区管理、中断处理、文件系统差异—— 这正是信息隐藏创造出的"深度"。

面试高频考点

  • 封装 vs 信息隐藏的区别？ 封装是机制（访问控制符），信息隐藏是设计目标（减少依赖）。封装是手段，信息隐藏是目的。用了 private 不等于做到了信息隐藏（getter 把一切都暴露出去了）。
  • 如何识别信息泄漏？ 问自己：修改模块 A 的某个实现细节，有哪些模块需要跟着改？需要改的越多，泄漏越严重。
  • 接口 vs 实现，划界标准是什么？ 接口里只放"调用方做决策必须知道的"，其余全部是实现。
  • 什么情况下信息必须"泄漏"？ 性能敏感路径（调用方需要选择缓冲区大小）、安全场景（调用方需要知道加密算法以便审计）。这种情况是合理权衡，要有意识地记录为文档。
