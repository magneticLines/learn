16. 一致性 Consistency

让同一件事在整个代码库中总是以同一种方式呈现——这是减少认知负担最廉价也最持久的手段。

作者的核心主张

Ousterhout 在《软件设计的哲学》第 17 章提出：一致性是一种杠杆。 当开发者在一个地方学会了某种模式，这份认知可以无摩擦地迁移到代码库的其他角落—— 因为"相同的情况总以相同的方式处理"。反过来，不一致会制造认知税： 读者每次遇到偏差都必须停下来问"这里为什么不一样？背后有特殊原因吗？" 即便答案是"没有原因，只是历史遗留"，这份怀疑本身就已经消耗了脑力。

作者把一致性分为几个层次，从低到高：命名惯例、代码风格、接口设计模式、模块间的交互协议。 层次越高，打破一致性的代价越大，也越难被静态检查工具发现。

一句话本质：一致性把"局部知识"变成"全局知识"。 代价是零，收益是每一位后来者的阅读时间。

为什么不一致会伤害可维护性

想象你在接手一个陌生的 Java 服务，下面这棵目录树展示了两种截然不同的文件组织方式并存：

service/
├── UserService.java <-- 按 功能 命名
├── OrderService.java
├── handlePayment.java <-- 突然变成 动词开头？
├── Svc_Inventory.java <-- 又换了前缀风格
└── notification_manager.java <-- 下划线命名法混入

五个文件，三种命名风格。新成员要问：它们之间有无语义差异？ handlePayment 是不是比 OrderService 更底层？ Svc_ 前缀是不是意味着"外部依赖"？这些问题全部是"噪音"—— 它们消耗了本该用于理解业务逻辑的精力。

反例 → 问题 → 正例

场景：Repository 层的错误处理方式

同一个项目里，不同人写了三个 Repository 方法，但错误处理策略各不相同。

反例 — 三种风格并存（Java）
// 方式 A：返回 null 表示"未找到"
public User findById(Long id) {
    return db.query("SELECT * FROM users WHERE id=?", id)
             .firstOrNull();
}

// 方式 B：抛出受检异常
public Order findOrderById(Long id) throws NotFoundException {
    Order o = db.query("SELECT * FROM orders WHERE id=?", id)
               .firstOrNull();
    if (o == null) throw new NotFoundException("Order not found: " + id);
    return o;
}

// 方式 C：返回 Optional
public Optional<Product> getProduct(Long id) {
    return Optional.ofNullable(
        db.query("SELECT * FROM products WHERE id=?", id)
           .firstOrNull()
    );
}
问题清单：
1. 调用方必须记住每个方法的"约定"，不能推断——findById 要判空，findOrderById 要 try-catch，getProduct 要 .isPresent()。
2. 新增方法时，作者凭个人喜好选风格，不一致会继续扩散。
3. 代码审查时无法用"应遵循项目惯例"这条规则来指出问题，因为没有惯例。
正例 — 统一为 Optional（Java）
// 团队约定：Repository.findXxxById 统一返回 Optional，
// 调用方自己决定"找不到"时的业务含义。
// 需要抛异常？在 Service 层做，而不是 Repository 层。

public Optional<User> findById(Long id) {
    return Optional.ofNullable(db.query("...", id).firstOrNull());
}

public Optional<Order> findById(Long id) {
    return Optional.ofNullable(db.query("...", id).firstOrNull());
}

public Optional<Product> findById(Long id) {
    return Optional.ofNullable(db.query("...", id).firstOrNull());
}

// Service 层：在这里决定"找不到"是 404 还是新建
public User getOrThrow(Long id) {
    return userRepo.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("User not found: " + id));
}
收益：读者在任意 Repository 看到 findById， 不需要看文档就知道返回 Optional。认知从"每次查手册"变成"一次学习，处处适用"。

一致性的四个维度

维度具体例子打破时的代价
命名惯例 getUserById vs fetchUser vs queryUser 读者需要记忆多套词汇表，无法靠"猜"找到方法
代码结构 Controller → Service → Repository 的三层结构，某个模块直接在 Controller 查库 破坏心智模型，依赖关系变得不可预测
接口契约 同类方法的参数顺序、返回类型、异常策略 调用方每次都要核实签名，IDE 自动补全失去意义
注释风格 有些方法有 Javadoc，有些没有；有些注释解释"为什么"，有些只是重复代码 读者无法判断"没有注释"是"显而易见"还是"被遗忘了"

一致性 vs. 正确性：哪个更重要？

这是实际开发中最容易引发争论的问题。Ousterhout 的立场很清晰： 如果现有惯例本身是合理的，就应该遵守，哪怕你认为自己的方式"更好"。 理由是：偏离惯例引入的认知成本，往往大于那点"更好"带来的收益。

但如果现有惯例有实质性缺陷（如会引发 bug、存在安全风险）， 则应该系统性地修改整个代码库，而不是在新代码里偷偷用新方式—— 那样只会让不一致的范围更大。

最危险的反模式：在同一个文件里混用新旧两套风格， 并注释 // new style。 这给所有后来者留下了一个永久性的"到底哪套是对的"的疑问。

如何在团队中建立并维护一致性

1. 写下来，而不是靠口口相传

团队约定必须落地为文档（CONTRIBUTING.md、ADR、CLAUDE.md 等）， 否则"一致性"只存在于老成员的记忆中，新人无从遵守。

2. 用工具强制执行，而不是靠 Code Review

能被 linter / formatter 检查的部分，就用工具管：Checkstyle、SpotBugs、ESLint、Prettier。 人工 Review 应聚焦在工具无法判断的语义层面一致性。

示例：用 Checkstyle 强制方法命名前缀（checkstyle.xml）
<!-- 所有 Repository 中的查询方法必须以 find 开头 -->
<module name="MethodName">
  <property name="format" value="^(find|save|delete|exists)[A-Z].*$" />
</module>

3. 新增惯例时先"全量替换"，再提交

如果决定把 null 返回改为 Optional， 那么在同一个 PR 里完成所有受影响文件的改动，不要分批—— 半途而废的迁移比不迁移更糟糕。

4. Code Review 时把"打破一致性"作为独立 checklist 项

不要只问"代码是否正确"，还要问"代码是否符合本模块的惯例"。 这两个问题是独立的，都需要有人负责。

容易忽略的细节

  • 测试代码同样需要一致性。 given/when/then、arrange/act/assert、还是随意组织？ 不一致的测试结构会让人怀疑"这个测试到底在测什么"。
  • 异常消息的格式也是一种惯例。 "User not found: 42" vs "Cannot find user with id=42"—— 如果日志分析脚本依赖固定前缀，风格混乱会导致监控失效。
  • 注释的"有无"本身传递信息。 如果团队约定"复杂逻辑必须有注释"，那么没有注释就意味着"这段逻辑是显而易见的"。 但如果有些人加注释有些人不加，这个信号就失去了意义。
  • 不要为了"保持一致"而复制坏代码。 如果现有惯例有 bug，应该修复根源，而不是"为了一致"继续复制 bug。 一致性是手段，可维护性是目的。
  • 跨语言项目中，每门语言有自己的"语言惯例"。 Java 用 PascalCase 类名，Python 用 snake_case—— 不要强行把 Java 风格带入 Python 模块来"统一"。语言惯例优先于项目惯例。

总结

一致性的价值不在于"代码看起来整齐"，而在于它把局部知识升级为全局知识： 学一次，处处可用；看到偏差，就知道有特殊原因。 这是一种极低成本、高回报的复杂度控制手段。

维护一致性需要主动投资：写下约定、用工具执行、Review 时关注偏差、迁移时彻底完成。 偷懒的结果是"每处都稍微不一样"——而这种模糊感，正是代码库熵增最隐蔽的形式。

与其他原则的关联： 一致性与下一节"显而易见"紧密相连——当代码高度一致时，任何偏离都会立刻变得"显而易见"， 从而降低 Bug 藏身的概率。一致性是让代码"可预测"的基础设施。
