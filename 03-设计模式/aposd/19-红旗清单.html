19. 红旗清单（Red Flags）汇总

全书 18 个「红旗」的完整索引——写代码时如果触发任意一条，请停下来重新思考设计。

什么是「红旗」，为什么要汇总它们

John Ousterhout 在《软件设计的哲学》中把设计坏味道称为 Red Flags（红旗）。 红旗不是「绝对禁止」的规则，而是一种信号： 当你的代码或接口让读者/使用者感到费力，说明复杂度正在悄悄蔓延。

作者的核心主张是：复杂度（Complexity）是软件开发最大的敌人。 复杂度不是一次性爆发的，而是通过无数个小决策逐渐积累起来的。 红旗就是那些「小决策失误」的具体症状，把它们列成清单，就能在日常 Code Review 或自我审查时快速定位问题。

使用方式：把这份清单贴在桌边。每次提 PR 前，扫一遍。 触发任何一条 → 停下来 → 想想是否能用「深模块（Deep Module）」或「信息隐藏」把它消除。

全书 18 个红旗速查表

# 红旗名称 一句话描述 对应章节
1 浅模块
Shallow Module		 接口复杂度 ≥ 实现复杂度，封装没有带来任何简化 Ch.4
2 信息泄漏
Information Leakage		 多个模块都知道同一个实现细节，耦合在这个细节上 Ch.5
3 时序依赖
Temporal Decomposition		 按"先调用 A 再调用 B"的执行顺序来拆分模块，而不是按信息隐藏 Ch.5
4 过度暴露
Overexposure		 把本应是实现细节的 API 暴露给了使用者，迫使他们了解不必要的细节 Ch.5

直通方法		 一个方法只是简单地把参数转发给另一个方法，自身没有增加任何逻辑 Ch.7

Interface Duplication		 多个模块提供功能相似的接口，调用者不知道选哪个 Ch.7
7 装饰器泛滥
Decorator Hell		 为了扩展而叠加大量 Decorator，每层只加一点点逻辑，反而增加理解成本 Ch.7
8 Pass-Through Variable
直通变量		 一个变量被层层传递，每一层都不关心它，只是为了最终层能用到 Ch.7
9 需要了解才能使用
Need-to-know		 使用某个模块前，必须先了解其内部实现，否则无法正确调用 Ch.8

18 模块间的隐式耦合
Conjoined Methods		 两个方法必须成对理解才能分别理解，说明它们共享了本不该共享的状态 Ch.16

五个最高频红旗——代码对比详解

以下五个红旗在日常工作中出现频率最高，每个给出「反例 → 问题 → 正例」的完整对比。

RED FLAG 1浅模块（Shallow Module）

模块提供的抽象价值低于使用它所需学习的成本。极端情况：接口比实现还难理解。

反例——只包一层的"抽象"

Java · 浅模块反例
// 反例：这个类什么都没封装，调用者反而多学了一个类名
public class UserRepository {
    private final UserDao dao;

    public User findById(Long id) {
        return dao.findById(id);   // 直接透传，零附加价值
    }

    public void save(User user) {
        dao.save(user);               // 同上
    }
}

问题：调用者为什么不直接用 UserDao？这一层 Repository 没有隐藏任何信息，没有处理缓存、事务、错误转换，纯粹增加层数。

正例——Repository 真正隐藏信息

Java · 深模块正例
public class UserRepository {
    private final UserDao   dao;
    private final Cache<Long, User> cache;

    /** 先查缓存，miss 后回源并填充缓存；调用者完全不感知 */
    public Optional<User> findById(Long id) {
        return cache.getOrLoad(id, dao::findById);
    }

    /** 写库 + 失效缓存，保证一致性 */
    public void save(User user) {
        dao.save(user);
        cache.invalidate(user.getId());
    }
}
这一层 Repository 隐藏了缓存策略和一致性细节——接口简单（同样两个方法），但实现复杂度被吸收进去了。这就是「深模块」。
RED FLAG 2信息泄漏（Information Leakage）

某个实现细节被多个模块共同知晓。改动这个细节时，需要同时修改多处。

反例——文件格式细节泄漏到调用侧

Java · 信息泄漏反例
// 调用者必须知道"配置文件是 JSON 格式"才能构造正确的 key
String value = configLoader.loadJsonField("config.json", "$.db.host");
// 另一处也知道这个细节：
String port = configLoader.loadJsonField("config.json", "$.db.port");

问题：如果将来配置改成 YAML 或数据库存储，每一个调用侧都要改。

正例——把格式细节封进模块

Java · 信息隐藏正例
// 调用者只知道"我要 db.host 这个配置项"，不关心存在哪、什么格式
String host = config.get("db.host");
String port = config.get("db.port");

// ConfigLoader 内部决定读 JSON / YAML / 环境变量，外部完全不感知
RED FLAG 3直通方法（Pass-Through Method）

一个方法除了把参数原封不动地转发给另一方法，什么也没做。它存在的唯一理由是「层次结构要求有它」。

反例 vs 正例

Java · 直通方法对比
// ---- 反例：Service 只是转发给 Repo，毫无附加价值 ----
class UserService {
    public User getUser(Long id) {
        return userRepository.findById(id);  // 直通，零价值
    }
}

// ---- 正例：Service 层做了业务层面真正有价值的事 ----
class UserService {
    public UserDto getUser(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        checkPermission(user);           // 权限校验——业务逻辑
        return UserDto.from(user);       // DTO 转换——隐藏实体细节
    }
}
如果你发现某个 Service/Facade 方法里全是直通调用，考虑直接删掉这一层，或者把真正的业务逻辑提进来。
RED FLAG 4注释重复代码（Comment Repeats Code）

注释和代码说的是同一件事，注释没有带来任何额外信息。这是注释中最常见的浪费。

Java · 注释对比
// ---- 反例：注释 = 代码的中文翻译，毫无价值 ----
// 将 i 加 1
i++;

// 如果用户未激活，返回 false
if (!user.isActive()) return false;


// ---- 正例：注释解释"为什么"，代码解释"是什么" ----
// 分页索引从 0 开始，但对外 API 从 1 开始，此处做偏移转换
i++;

// 未激活账号不允许登录（安全策略 SEC-42），即使密码正确也拒绝
if (!user.isActive()) return false;
RED FLAG 5模糊名称（Vague Name）+ 难以取名（Hard-to-Pick Name）

名字太宽泛无法传递意图；或者你发现一段代码根本取不出好名字——后者本身就说明这段代码的职责不清晰。

Java · 命名对比
// ---- 反例：模糊名称 ----
void handle(Object data);
List getList();
boolean check(User u);


// ---- 正例：精确名称 ----
void processPaymentCallback(PaymentEvent event);
List<Order> findPendingOrders();
boolean hasPermissionToDeletePost(User user);

// ---- 难以取名 → 设计信号 ----
// 如果你叫不出来，很可能这个方法做了 2 件事，应该拆分：
void validateAndSaveAndNotify(Order order) { /* ← 三件事 */ }

// 正确做法：三个职责，三个方法，各自有清晰的名字
validate(order);
save(order);
notifyBuyer(order);

按「复杂度来源」分类

Ousterhout 认为复杂度有两个来源：变更放大（Change Amplification） 和 认知负担（Cognitive Load）。可以按此对红旗分类：

复杂度来源 ├── 变更放大（改一处必须改多处） │ ├── 信息泄漏 → 实现细节散落在多个模块 │ ├── 时序依赖 → 执行顺序硬编码进多个调用侧 │ ├── 接口重复 → 同功能接口遍布代码库 │ └── 配置分裂 → 参数分散在调用侧与实现侧 │ └── 认知负担（读者需要额外脑力才能理解） ├── 浅模块 → 封装没有减少理解成本 ├── 直通方法/变量 → 增加层次却不减少信息 ├── 过度暴露 → 细节暴露迫使调用者理解内部 ├── 异常过多 → 错误路径比正常路径还复杂 ├── 注释重复/模糊 → 注释无法辅助理解 ├── 模糊名称 → 名字不传递意图 └── 非显而易见 → 行为需要猜测

实际开发中的落地技巧

1. 把红旗清单加入 Code Review Checklist

在 PR 模板里加一个 section：

Markdown · PR 模板片段
## Red Flag 自查
- [ ] 没有浅模块（新增的类/方法，接口比实现简单）
- [ ] 没有信息泄漏（实现细节没有扩散到调用方）
- [ ] 没有直通方法（每一层都有附加价值）
- [ ] 注释解释"为什么"，不只是翻译代码
- [ ] 所有方法/变量名精确、具体，无模糊措辞

2. 「取不出名字」是重构的触发器

当你盯着一段代码超过 30 秒仍然取不出一个好名字，不要强行取名——这是设计信号，说明这段代码的职责不单一，应该先重构职责再命名。

3. 用「最坏情况测试」检验深度

问自己：如果我把这个模块的实现完全换掉（换数据库、换序列化格式、换缓存策略），调用侧需要改几行？ 如果答案不是"零行"，就说明存在信息泄漏或浅模块问题。

4. 异常也是接口的一部分——少即是多

每一个 checked exception 都是对调用者施加的认知负担。Ousterhout 建议： 能在模块内部消化的异常就消化，暴露给调用侧的异常应该是业务层面真正需要区分处理的。

Spring Data JPA 的 findById 返回 Optional 而不是抛 EntityNotFoundException，正是这一原则的体现：把"找不到"这个普通情况纳入正常返回值，不让调用者为此写 try-catch。

5. 警惕「直通变量」——考虑用上下文对象替代

如果你发现某个参数被 A → B → C → D 传递但只有 D 使用，可以考虑把它放入一个上下文对象（如 RequestContext、ThreadLocal），或者重新审视模块边界是否划错了。

红旗与设计原则的对应关系

红旗 违反的正向原则 修复方向
浅模块 深模块（Deep Module） 把更多复杂度吸收进实现，缩小接口
信息泄漏 / 过度暴露 信息隐藏（Information Hiding） 把实现细节封进模块，接口只暴露业务概念
直通方法 / 直通变量 每层有价值（Each Layer Adds Value） 删除无价值层，或把真正的业务逻辑提进来
时序依赖 按信息隐藏拆分，而非按执行顺序 把时序依赖封进一个模块内部
异常过多 错误处理应简单（Define Errors Out of Existence） 在模块内部消化异常，或把边界情况纳入正常返回
注释重复 / 模糊 注释应补充代码未表达的信息 注释写"为什么"、"设计意图"、"非显然约束"
模糊名称 / 难以取名 名称是接口的一部分，应精确表达意图 先厘清职责，再取名；名字不精确则重构职责

总结：用红旗建立设计直觉

这 18 个红旗本质上是同一件事的不同侧面： 复杂度在逃离它本应存在的模块，蔓延到不需要关心它的地方。

作为工程师，建立设计直觉最快的方式是： 在写完代码后，主动用红旗清单扫描一遍自己的实现。 不需要全部修复，但需要有意识地做决策——如果留下某个红旗，你应该能说清楚为什么值得。

终极测试：把你的模块接口（方法签名 + 注释）给从没看过实现的同事看。 如果 5 分钟内他能正确使用它，说明你的接口足够简单、注释足够清晰，没有触发红旗。
