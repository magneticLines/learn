《软件设计的哲学》 · 注释

13. 注释先行

先写注释，再写实现——这不是写文档，而是一种设计思维工具。

核心主张：注释是设计的一部分，不是事后补充

Ousterhout 在《软件设计的哲学》中提出了一个颠覆直觉的主张： 先写注释，后写代码（Comments First，又称 comment-driven design）。 这个做法的本质不是"让文档齐全"，而是把注释当作一把尺子， 在实现之前就检验你的设计是否清晰。

他的逻辑链条很直接：如果你在动手实现前无法用简洁的自然语言描述一个模块的 目的、输入输出、副作用、使用约束，那说明你自己还没想清楚—— 此时写出来的代码，只会把模糊固化下来，越改越乱。

一句话原则：你能不能用两三行注释把这个函数/类的 抽象 说清楚？ 如果不能，说明抽象本身有问题，先停下来重新设计，而不是先写代码再想办法解释它。

为什么顺序很重要

许多开发者的习惯是：写完代码 → 最后加注释（或者根本不加）。这种顺序有一个隐藏的心理陷阱：

  • 沉没成本偏差：代码已经写完了，你会下意识地把注释写成"解释现有实现"，而不是"描述应有的抽象"。
  • 合理化倾向：无论实现多绕，你都会在注释里把它说得好像"本来就该这样"。
  • 掩盖了糟糕的接口设计：真正难以描述的，往往是职责混乱、参数含义模糊的函数——事后注释反而会帮它们续命。

而注释先行则强迫你在 0 行代码的时候就面对"这个模块到底做什么"这个最难的问题。 如果注释写起来很顺畅，设计通常就是好的；如果磕磕绊绊写不下去，那就是设计给你的信号。

类比：好的 API 设计师会先写调用示例（Test-Driven Design 的精神），然后才实现内部。 注释先行做的是同一件事：先站在调用者/读者的视角定义契约，再去实现它。

反例 → 问题 → 正例

反例：先实现，注释是事后解释

Java · 事后注释（坏味道）
// 这段代码先实现，再加注释——注释只是在重复实现细节

public class UserCache {

    private Map<Long, User> cache = new HashMap<>();
    private Map<Long, Long> timestamps = new HashMap<>();
    private static final long TTL = 300_000;

    // 获取用户
    public User get(long id) {
        // 检查时间戳
        Long ts = timestamps.get(id);
        if (ts != null && System.currentTimeMillis() - ts < TTL) {
            return cache.get(id);
        }
        return null;
    }

    // 存用户
    public void put(long id, User user) {
        // 放入 map 并记录时间
        cache.put(id, user);
        timestamps.put(id, System.currentTimeMillis());
    }
}

问题在哪：

  • "获取用户"和"存用户"完全是废话注释——读代码就能知道，注释毫无增量信息。
  • 读者依然不知道：缓存未命中时返回 null 还是抛异常？过期条目什么时候清理？是否线程安全？put(null) 合法吗？
  • 这些关键契约问题，因为"代码已经写完了"，作者根本没意识到自己没有回答它们。

正例：注释先行，先定义契约再写实现

Java · 注释先行（好的做法）
/**
 * 带 TTL 的内存用户缓存（非线程安全，仅供单线程场景使用）。
 *
 * <p>每个条目在首次 put 后 5 分钟内有效。条目过期后 get 返回 null，
 * 但不会被主动清除——调用方应在 get 返回 null 时从数据源重新加载并 put。
 *
 * <p>使用约束：
 * <ul>
 *   <li>user 不能为 null，否则抛出 NullPointerException</li>
 *   <li>同一 id 重复 put 会刷新 TTL</li>
 * </ul>
 */
public class UserCache {

    private static final long TTL_MS = 5 * 60 * 1000;

    private final Map<Long, User>  entries    = new HashMap<>();
    private final Map<Long, Long>   expiresAt  = new HashMap<>();

    /**
     * 返回 id 对应的缓存用户，若不存在或已过期则返回 null。
     * 过期条目会在本次调用时从缓存中删除（懒清理）。
     */
    public User get(long id) {
        Long exp = expiresAt.get(id);
        if (exp == null) return null;
        if (System.currentTimeMillis() > exp) {
            entries.remove(id);
            expiresAt.remove(id);
            return null;
        }
        return entries.get(id);
    }

    /**
     * 将 user 存入缓存并（重）置 TTL。
     *
     * @param id   用户唯一标识，必须为正数
     * @param user 不能为 null
     */
    public void put(long id, User user) {
        Objects.requireNonNull(user, "user must not be null");
        entries.put(id, user);
        expiresAt.put(id, System.currentTimeMillis() + TTL_MS);
    }
}

注意：正例的注释是在一行实现代码都没写之前完成的。 当你写"懒清理"这个词时，你就意识到了需要在 get 里调用 remove—— 这是注释逼你做出的设计决策，而不是实现完之后才想到的。

实际操作：注释先行的三步工作流

Step 1 写类/接口级注释 │ 描述这个模块"是什么"、"解决什么问题"、"边界在哪" │ → 如果你写不出一段连贯的话，说明模块职责不清晰 │ Step 2 写每个公有方法的 Javadoc（只写签名 + 注释，不写方法体） │ 包含：做什么 / 参数约束 / 返回值语义 / 可能的异常 / 线程安全性 │ → 此时就是在设计 API 契约，发现参数歧义立即改签名 │ Step 3 填写实现，并在关键决策处加内联注释 内联注释解释"为什么这样实现"（why），而不是"这行做什么"（what） → 实现与契约不一致时，优先修改实现，而不是修改注释来掩盖

注释先行是设计质量的"晴雨表"

Ousterhout 给出了一个实用法则：如果注释很难写，那几乎一定是设计有问题。 下表列出常见的"注释写不下去"信号及其背后的设计问题：

症状 注释写不下去的表现 真正的设计问题
参数含义模糊 不知道 @param flag 该写什么 用布尔值控制行为，应该拆成两个方法
职责过多 方法注释要写四五个"并且" 方法违反单一职责，应拆分
类的边界不清 类注释写着写着越写越像整个系统的说明 类承担了太多角色，需要拆包
返回值语义不定 @return 要写一段 if/else 解释各种情况 返回值承载了太多隐含状态，考虑用异常或 Optional

落地技巧与容易忽略的细节

技巧 1：写"意图注释"，而不是"实现注释"

Java · 意图 vs 实现注释
// ❌ 实现注释：说的是"这行在做什么"——读代码就能知道，毫无价值
i++; // i 加 1

// ❌ 实现注释：只是代码的自然语言翻译
// 遍历列表，对每个元素调用 process
for (Item item : items) process(item);

// ✅ 意图注释：说的是"为什么这样做"——代码本身读不出来
// 优先处理高优先级队列，确保 SLA 不因低优先级任务积压而被拖慢
for (Item item : highPriorityQueue) process(item);
for (Item item : normalQueue)      process(item);

技巧 2：用"如果注释超过 3 行还说不清楚"作为重构触发器

当你发现一个方法的 Javadoc 需要超过 3 行才能解释清楚参数的各种情况时， 这是一个强烈的信号：这个接口设计得太复杂了。 在动手写实现前，先停下来重新设计签名。

技巧 3：注释和代码不一致时，永远相信注释（然后修复代码）

注释先行的一个隐含原则：注释是"应该的样子"（规约），代码是"现在的实现"。 当两者矛盾时，大概率是实现出了 bug，而不是注释写错了。 许多开发者的错误本能是修改注释来"对齐"有 bug 的实现——这相当于把规格书篡改成了 bug 文档。

技巧 4：私有方法和局部变量不强制 Javadoc，但复杂逻辑必须有内联 why 注释

Java · 私有方法内联注释示例
private int computeBackoff(int attempt) {
    // 指数退避：2^attempt * 100ms，上限 30s
    // 上限避免极端情况下雪崩式重试拖垮下游
    return Math.min((1 << attempt) * 100, 30_000);
}
常见陷阱：注释先行容易退化为"写完代码再把注释补回去"。 建议给自己立一条硬规则：没有类/方法级注释的代码不允许提交 PR。 用 CI 的 Checkstyle / Spotless 强制检查 public API 的 Javadoc 缺失，让规范有牙齿。

注释先行 vs. 测试先行（TDD）

两者精神高度一致：都主张在实现之前先定义契约。

  • TDD 用可执行的测试来定义行为契约，能自动验证正确性。
  • 注释先行用自然语言来定义意图契约，覆盖测试难以表达的非功能约束（并发安全性、边界语义、使用场景）。
  • 两者可以并行：先写注释（定义意图）→ 再写测试（定义行为）→ 最后写实现。

Ousterhout 本人强调，注释先行并非替代 TDD， 而是填补了 TDD 无法覆盖的那部分设计意图表达。

核心记忆点： 注释先行不是"提前写文档"，而是把注释当作设计工具—— 写不出注释就说明设计不清晰，这比等到代码写完再发现问题要便宜得多。 好的注释描述"是什么 + 为什么 + 约束"，而不是翻译代码。

与其他知识点的联系

  • 注释的价值（12）：上一节回答"为什么注释不可替代"，本篇回答"怎么用注释把这份价值榨取到最大"——注释先行把"写注释"从一个事后补充的文档动作，升级成一个前置的设计验证工具，两节是同一个主张的论据与方法论。
  • 深模块（04）与浅模块红旗（05）：本篇"如果注释很难写，那几乎一定是设计有问题"这条法则，本质上是深模块理论的一个可操作检验手段——写不出简洁接口注释，往往对应着 05 节讲的浅模块红旗（接口和实现一样复杂，或者接口本身职责不清），注释先行让这个问题在写代码之前就暴露出来，而不是等代码写完后才被同事在 Code Review 里发现。
  • 复杂性来源：依赖与晦涩（02）：本篇表格列出的"参数含义模糊""职责过多""返回值语义不定"等症状，正是 02 节所说的晦涩在接口设计阶段的早期信号。注释先行相当于把"晦涩检测"提前到了设计阶段，用自然语言的写作难度作为晦涩程度的替代指标。
  • 与 TDD 的关系（本篇第 7 节已展开）：本篇明确将注释先行与测试先行做类比——两者都主张"先定义契约，再写实现"，只是契约的载体不同（自然语言 vs 可执行代码）。这也解释了为什么"测试驱动开发"和"注释先行"在实践中经常被放在一起提，它们覆盖的是契约的两个互补维度：行为正确性与设计意图。
  • 与 Clean Code 的分歧延续：12 节已经指出 APOSD 与 Clean Code 在"是否需要注释"上正面冲突，本篇进一步把这个分歧推向极致——Clean Code 几乎不讨论"写代码前先写什么"，而 APOSD 认为注释应该先于代码存在，因为它是检验接口设计的工具而非事后归档。这种"设计优先于实现"的顺序主张，也与 GoF 设计模式书中"先设计类图和协作关系，再填充具体实现"的思路遥相呼应。
