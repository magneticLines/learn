10. 避免直通方法 Pass-Through Methods

一个只转发调用、不增加任何价值的方法，是浅层类的最典型症状——它暴露了错误的分层设计。

作者的核心主张

John Ousterhout 在《软件设计的哲学》第七章明确指出： 直通方法（pass-through method）是代码设计出问题的信号，不是一种无害的中间层。

所谓直通方法，是指一个方法除了把参数原封不动地传给另一个方法之外，什么也不做。 它的存在意味着两件事同时成立：

  • 调用者需要了解"被穿透"的那一层——分层边界形同虚设。
  • 中间层没有提供任何额外的抽象、变换或保护——层与层之间职责重叠，接口"浅"而"宽"。
一句话核心：每一层都应该有独立存在的理由。如果一层只是把调用转发给下一层，那这一层根本不应该存在，或者应该把职责合并到其中一层。

为什么直通方法是坏味道

Ousterhout 的论证逻辑是：好的分层设计的本质是每一层提供不同的抽象。分层的价值在于隐藏复杂性，让上层调用者不必知道下层细节。直通方法打破了这个承诺：

  • 接口泄漏：调用者事实上绕过了"中间层"，直接依赖下层的接口形状（参数顺序、类型等）。
  • 修改成本翻倍：改下层签名时，中间层也要改，调用方也要改——三处改动，只完成一件事。
  • 认知负担加重：读者必须追踪完整的调用链才能理解行为，而不是在某一层"停下来"就能理解。
  • 掩盖真实依赖：调用者表面上只依赖中间层，实则依赖下层，耦合被掩盖而非消除。

反例 → 问题 → 正例

场景：文本编辑器中的选区管理

假设我们有三个类：Editor（顶层 UI）、TextDocument（中间层）、SelectionManager（底层）。

反例：直通方法

Java — 反例pass-through
// ❌ TextDocument 只是个"传声筒"，没有提供任何额外价值
class TextDocument {
    private SelectionManager selectionManager;

    // 直通方法：原封不动转发给 SelectionManager
    public void setSelection(int start, int end) {
        selectionManager.setSelection(start, end);
    }

    // 直通方法：同上
    public Selection getSelection() {
        return selectionManager.getSelection();
    }

    // 直通方法：继续转发
    public void clearSelection() {
        selectionManager.clearSelection();
    }
}

// 调用方：Editor 调用 TextDocument，但实际在用 SelectionManager 的接口
class Editor {
    private TextDocument doc;

    public void selectWord(int wordStart, int wordEnd) {
        doc.setSelection(wordStart, wordEnd);
    }
}

问题分析

症状一：签名完全相同。 TextDocument.setSelection(int, int) 和 SelectionManager.setSelection(int, int) 一模一样——中间层完全透明，等于不存在。

症状二：Editor 实际上耦合到了 SelectionManager 的接口。 如果 SelectionManager 改成用 Range 对象替代两个 int，三个地方都要改。

症状三：阅读 Editor 时必须穿透两层才能理解行为。 这意味着层次划分没有降低认知复杂度，反而增加了追踪成本。

正例 A：合并层次（消除多余中间层）

如果 TextDocument 对选区没有任何独特的逻辑，最直接的做法是把 SelectionManager 的职责移进来，消除多余的层。

Java — 正例 A：合并remove redundant layer
// ✅ 把选区逻辑直接放在 TextDocument 里，不再有中间转发
class TextDocument {
    private int selectionStart = -1;
    private int selectionEnd   = -1;

    public void setSelection(int start, int end) {
        // 边界校验是真实业务逻辑，不是转发
        if (start < 0 || end > length())
            throw new IllegalArgumentException("Selection out of bounds");
        selectionStart = start;
        selectionEnd   = end;
    }

    public Selection getSelection() {
        return new Selection(selectionStart, selectionEnd);
    }
}

正例 B：让中间层真正提供不同抽象

如果确实需要保留 SelectionManager（例如它还服务于其他模块），那么 TextDocument 就必须在接口层面提供不同的抽象，而不是简单转发。

Java — 正例 B：不同抽象different abstraction per layer
// ✅ TextDocument 提供"按文档语义"操作选区的接口
//    SelectionManager 处理底层字节偏移量
//    两层的抽象粒度不同，都有存在价值
class TextDocument {
    private SelectionManager selectionManager;
    private LineIndex         lineIndex;   // 行列 ↔ 偏移量的转换表

    // 上层接口：行列坐标（对 Editor 友好）
    public void setSelectionByLineCol(
            int startLine, int startCol,
            int endLine,   int endCol) {
        // 做了真实的转换工作：行列 → 字节偏移量
        int start = lineIndex.toOffset(startLine, startCol);
        int end   = lineIndex.toOffset(endLine,   endCol);
        selectionManager.setSelection(start, end);
    }

    // 上层接口：按单词选中（语义更高）
    public void selectWord(int cursorOffset) {
        int[] range = findWordBoundary(cursorOffset);
        selectionManager.setSelection(range[0], range[1]);
    }
}
判断标准：把中间层的方法签名和下层方法签名并排放。如果它们完全相同，中间层就是直通的。如果签名不同（参数类型、语义粒度、抽象层次不同），说明中间层提供了真实价值。

直通方法的常见变体

直通方法并不总是像上面那样赤裸裸，以下几种形式同样是直通的本质，需要警惕：

变体代码特征问题
仅改名转发 doSave() 只调用 persistToDisk() 改名没有提升抽象，只增加了间接层
接口→接口直通 接口 A 的方法与接口 B 完全相同，实现类只转发 两个接口合并成一个即可，人为分裂增加了复杂度
装饰器空壳 Decorator 类覆写了所有方法，但都只调用 super.xxx() 忘记在装饰器中加真实逻辑，写了一个空的中间层
委托链 A.foo() → B.foo() → C.foo()，三层同名方法 应当直接让 A 依赖 C，或在 B 做真实工作

直通方法 vs. 合法委托

Ousterhout 并不反对委托（delegation）本身。委托是健康的分层协作方式。区别在于：

┌────────────────────────────────────────────────┐ │ 直通方法（bad） │ │ A.foo(x, y) ──► B.foo(x, y) │ │ ↑ 签名完全相同，没有变换 │ ├────────────────────────────────────────────────┤ │ 合法委托（good） │ │ A.selectWord(cursor) │ │ │ 做了边界查找、坐标转换等工作 │ │ └──► B.setSelection(start, end) │ │ ↑ 签名/语义/抽象粒度都不同 │ └────────────────────────────────────────────────┘

核心判断问题："调用者绕过我这一层，直接调用下一层，会不会更简单？"如果回答是"是"，这一层就是多余的，或者这个方法是直通的。

与"不同层次应有不同抽象"的联系

上一节讲的是宏观原则：每一层的接口应该提供比下层更高级别的抽象。本节的"避免直通方法"是这个原则的微观检验手段：

  • 如果一个方法只是直通，说明这个方法违反了"不同抽象"原则——它暴露的抽象与下层完全相同。
  • 直通方法是"抽象泄漏（leaky abstraction）"的最简单形式。
  • 修复直通方法，往往意味着要重新思考该层的职责边界，而不只是"删掉一个方法"那么简单。

实际开发中的落地技巧

技巧 1：接口评审时做"签名对比"

在 Code Review 中，将一个类的公开方法列表与它依赖的下层类的方法列表并排对比。如果发现大量同名、同签名的方法，立即追问"这一层到底做了什么"。

技巧 2：用"删除测试"验证必要性

假设把中间层删掉，让调用者直接调用下层。如果调用者的代码几乎不变，说明中间层是多余的，或者这批方法是直通的。

技巧 3：重构方向——三选一

  • 合并：把下层职责吸收进来，消除多余层次（适合下层本就很小）。
  • 下沉：把中间层的调用方改为直接依赖下层（适合中间层真的没有其他价值）。
  • 充实：在中间层加入真实业务逻辑，让它提供更高层次的抽象（适合层次本应存在）。

技巧 4：警惕"接口适配器"写成直通

适配器（Adapter）模式的合理场景是转换接口形状，但如果你写了一个 Adapter，结果所有方法都只是 delegate.sameMethod(sameArgs)，说明根本不需要适配器，两个接口应该统一。

技巧 5：构造函数注入时的警惕点

当一个类在构造函数中接收了某个依赖，然后把这个依赖的所有方法都暴露出去，这个类其实退化成了一个外观（Facade）空壳——它持有了引用，但没有提供真正的封装。此时应该考虑这个外观层是否真的必要。

Spring 项目中的常见陷阱：
Service 层调用 Repository 层时，如果 Service 的每个方法都只是 return repository.xxx()，这个 Service 就是直通的。Service 层应该负责事务管理、业务规则校验、多个 Repository 的编排——这些才是它存在的价值。如果一个 CRUD 功能确实没有业务逻辑，可以考虑让 Controller 直接调用 Repository（或用更薄的 Service 封装一批相关操作），而不是为每个操作单独写一个直通方法。

一层存在的价值 = 它隐藏的信息量。
直通方法隐藏了零信息，是层次存在意义的自我否定。
