06. 复杂性下沉

把难处理的细节埋进模块内部，对外只暴露简单的接口——这才是深模块的精髓。

一、作者的核心主张

John Ousterhout 在《软件设计的哲学》第八章提出了一条简洁的原则： 复杂性应该向下沉入模块内部，而不是向上暴露给调用者。

所谓"复杂性"，包括：边界条件的处理、默认行为的选择、格式转换、错误恢复、并发同步…… 这些细节是真实存在的，无法凭空消除，只能决定由谁来承担。 复杂性下沉的意思是：让模块的实现者来承担，而不是强迫每一位调用者重复处理。

一句话直觉： 如果你的模块让每个使用者都必须先理解五条前提才能正确调用，那复杂性就在上面； 如果调用者传一个参数、得到想要的结果，复杂性就在下面。

这与"深模块"（deep module）的概念直接相连：接口简单 + 功能强大 = 深模块。 浅模块把内部的每一个决策都暴露成参数或前置条件，把负担甩给上层，是反模式。

二、为什么复杂性下沉能让系统更好

2.1 减少认知负担

调用方的工程师只需关心"我想要什么"，不必关心"模块内部怎么实现"。 系统中存在 100 个调用点，每个调用点都少理解一分复杂性，整体认知成本就降低了 100 倍。

2.2 集中修改，降低变更风险

复杂的逻辑只在一处实现。当逻辑需要修改时（如换一套缓存策略、处理新的边界情况）， 只改模块内部，调用方代码无需动。否则分散在 100 个调用点的"重复的复杂性"， 改一处就要改 100 处，漏掉一处就是 Bug。

2.3 更容易正确使用

接口越简单，调用者越难用错。暴露太多控制旋钮，意味着调用者有更多机会做出错误组合。 设计者比调用者更了解内部的约束，理应由设计者把守这些约束。

三、代码对比：反例 → 问题 → 正例

场景：从文件读取文本内容

最常见的场景之一：一个工具方法，从磁盘读取一个文本文件。

反例 — 复杂性浮在上面（Java）bad
// 模块对外暴露的方法：调用者必须自己传入 Charset，自己处理 IOException
public class FileUtils {

    // 返回 null 还是抛异常？调用者必须看实现才知道
    public static String readFile(String path, Charset charset,
                                     boolean throwOnMissing)
            throws IOException {
        File f = new File(path);
        if (!f.exists()) {
            if (throwOnMissing) throw new IOException("File not found: " + path);
            return null;        // 调用者忘了检查 null → NullPointerException
        }
        return new String(Files.readAllBytes(f.toPath()), charset);
    }
}

// 调用方：每处调用都要重复这套仪式
String content = FileUtils.readFile(path, StandardCharsets.UTF_8, true);
// 问题一：charset 由谁定？业务层根本不关心编码细节
// 问题二：throwOnMissing 的语义模糊，true 和 false 的后果截然不同
// 问题三：每个调用点都要写一遍相同的参数，复杂性被复制了 N 份
问题总结
· charset 参数让调用者承担了"该用什么编码"的判断，而 99% 的场景都是 UTF-8。
· throwOnMissing 把异常策略的决策权甩给调用者，导致每处调用都要重新思考。
· 返回 null 的可能性被隐藏在 boolean 参数背后，极易产生 NPE。
正例 — 复杂性下沉到模块内部（Java）good
public class FileUtils {

    /**
     * 读取文本文件内容（UTF-8）。
     * 文件不存在时抛出 FileNotFoundException（是 IOException 子类）。
     * 调用者不需要关心编码，也不需要做 null 检查。
     */
    public static String readFile(String path) throws IOException {
        // 编码决策在这里做一次，永远一致
        return Files.readString(Path.of(path), StandardCharsets.UTF_8);
        // Files.readString 在文件不存在时自动抛出 NoSuchFileException
        // 不存在"返回 null"的歧义路径
    }

    /**
     * 需要自定义编码时，提供明确的重载——但这是例外情况，不是默认要求。
     */
    public static String readFile(String path, Charset charset) throws IOException {
        return Files.readString(Path.of(path), charset);
    }
}

// 调用方：一行，清晰，无歧义
String content = FileUtils.readFile(path);
改进要点
· 编码默认为 UTF-8，隐藏在实现里——这是绝大多数场景的正确答案，无需每次传递。
· 用方法重载替代 boolean 参数，意图更清晰，也不存在"参数顺序记错"的问题。
· 文件不存在时只有一条路：抛异常，不返回 null，消除了 NPE 隐患。

场景二：配置解析中的默认值决策

反例 — 把默认值判断甩给调用者（Java）bad
public class Config {
    // 找不到 key 时返回 null，让调用者自己决定默认值
    public String get(String key) { return map.get(key); }
}

// 调用方：到处散落的相同默认值逻辑
String timeout = config.get("timeout");
if (timeout == null) timeout = "30";  // A 处写了 "30"

String t2 = config.get("timeout");
if (t2 == null) t2 = "60";            // B 处写了 "60"——默认值不一致了！
正例 — 默认值决策下沉到模块内部（Java）good
public class Config {
    private static final Map<String, String> DEFAULTS = Map.of(
        "timeout", "30",
        "retries", "3"
    );

    /**
     * 优先返回配置文件中的值，缺失时使用内置默认值，永远不返回 null。
     * 调用者无需写任何 null 判断。
     */
    public String get(String key) {
        String v = map.get(key);
        return v != null ? v : DEFAULTS.getOrDefault(key, "");
    }
}

// 调用方：干净，无 null 检查，全系统默认值一致
String timeout = config.get("timeout");

四、直觉模型：复杂性在哪一层？

┌─────────────────────────────────┐
│ 调用方（业务层） │
│ FileUtils.readFile(path) ← 简单接口
└────────────────┬────────────────┘
│ 调用
┌────────────────▼────────────────┐
│ 模块实现层（FileUtils） │
│ · 编码决策（UTF-8） │
│ · 路径规范化 │
│ · 文件不存在 → 抛异常，不返回 null │
│ · 缓冲区大小管理 │
└─────────────────────────────────┘
↑ 复杂性全部收口在这里

上图展示了理想状态：调用方只看到一个简单的入口， 所有关于"怎么做"的决策都被压进模块内部。 调用方和模块之间的接口面越窄，复杂性就越靠近底部。

五、与相关原则的关系

原则 侧重点 和复杂性下沉的关系
深模块 接口简单、功能强大 复杂性下沉是实现深模块的核心手段
信息隐藏 内部实现细节不外泄 隐藏的就是"下沉"的那部分复杂性
单一职责 每个类只做一件事 职责收口后，复杂性自然在该模块内集中
最少惊奇原则 行为符合调用者预期 默认行为由模块决定，调用者不被意外行为惊到

六、实际开发中的落地技巧

技巧 1：用方法重载替代 boolean / enum 参数

参数里出现 boolean verbose、Mode mode、boolean strict 之类的控制旗标， 往往意味着模块在把决策权甩给调用者。 优先考虑提供两个名字清晰的重载方法，而不是一个带旗标的方法。

技巧 2：合理选择默认行为而非暴露配置选项

设计时问自己："80% 的调用者会传什么值？" 把这个值内化为默认行为，只对真正需要差异化的 20% 提供重载或配置项。 这不是限制灵活性，而是把灵活性收口在需要它的地方。

技巧 3：不要把异常决策权上推

模块内部遇到错误时，不要返回 null 或错误码让调用者判断—— 在模块内部判断该抛什么异常，直接抛出。 调用者只需在他们关心的地方 catch，其余情况让异常自然向上冒泡。

技巧 4："如果我去掉这个参数，会有什么问题？"

在设计接口时，对每一个参数问这个问题。如果去掉之后模块可以自己做出合理决定， 那这个参数就是不必要的——它是复杂性没有下沉的信号。

技巧 5：警惕"通用化过度"的陷阱

复杂性下沉并不是"把所有配置都做成内置死值"。 当调用者确实有合理的差异化需求时，配置和参数是正确的。 关键是：只暴露调用者真正需要控制的旋钮，其余的全部内化。

容易踩的坑
· 过度下沉：把调用者真正需要控制的行为也藏起来，变成了"不灵活的黑盒"。
· 下沉位置错误：把复杂性下沉到了 util 包，而不是拥有该领域知识的领域模块， 导致 util 变成大杂烩。
· 默认值选择错误：内化了一个错误的默认值，100% 的调用者都要覆盖它—— 这说明你的默认决策没有做好调研。

七、JDK / Spring 中的真实体现

  • Collections.sort(list) 默认使用 Comparable 自然顺序， 不要求调用者传 Comparator——这就是复杂性下沉（排序策略的默认决策在方法内部）。
  • HashMap 的初始容量和负载因子有合理默认值， 99% 的用法是 new HashMap<>()， 不需要 new HashMap<>(16, 0.75f)。
  • Spring 的 @Transactional 默认传播行为是 REQUIRED、 回滚条件是 RuntimeException—— 框架把最常见的事务策略内化为默认值，调用者只在需要定制时才写属性。
  • Optional.orElse("default") 让默认值决策在链式调用中一次性完成， 不用每次都 if (x == null)。
核心记忆点
复杂性是守恒的，它不会消失，只会转移。 好的模块设计者选择把复杂性扛在自己肩上，给调用者一个干净的接口； 差的设计者把复杂性推给调用者，美其名曰"灵活可扩展"。 判断标准很简单：调用方是否因为你的接口而变得更简单了？
