09. 不同层次应有不同抽象

每一层都应该提供与其相邻层截然不同的抽象。若相邻层之间的抽象高度相同，系统便产生了"层次污染"，模块间出现不必要的耦合，复杂度无处可藏。

作者的核心主张

John Ousterhout 在《软件设计的哲学》第 7 章给出了一条对分层架构的诊断标准： 如果你在两个相邻层之间看不到抽象差异，这个层次就没有存在的价值。

他将这一问题命名为 pass-through（直通）现象——底层有什么，上层就暴露什么；底层叫什么，上层就叫什么。 上层并未带来任何新的"视角"，只是在机械转发。这让整个调用链的复杂度与底层持平，调用者却要额外经历两层心智负担。

核心公式： 每升高一层，抽象词汇表就应该替换一次。 低层讲"字节/TCP 包/文件描述符"，高层讲"消息/连接/配置项"，再上层讲"业务规则/用户意图"。 每次替换都在隐藏复杂度，而非转移复杂度。

为什么这样设计更好？因为层次的本质是信息过滤器。 你无需了解 TCP 重传机制就能发一封电子邮件，这是网络协议栈"不同层次不同抽象"带来的礼物。 代码的分层也应如此：每一层把下层的混乱封装成简洁的概念，让上层用更少的认知资源完成更多的事。

反例 → 问题 → 正例

场景：用户服务调用数据访问层

反例：抽象高度相同，上层只是搬运工
Java — 抽象层次相同（反例）
// ===== 数据访问层 =====
public class UserRepository {
    public User findByUsername(String username) { ... }
    public void  saveUser(User user) { ... }
    public void  deleteByUsername(String username) { ... }
}

// ===== 服务层 —— 与 Repository 的词汇表完全相同 =====
public class UserService {
    private final UserRepository repo;

    // 直通：方法名、参数、返回值与 Repository 一模一样
    public User findByUsername(String username) {
        return repo.findByUsername(username);
    }
    public void saveUser(User user) {
        repo.saveUser(user);
    }
    public void deleteByUsername(String username) {
        repo.deleteByUsername(username);
    }
}

问题出在哪里？

  • 词汇表没有升级： Service 层与 Repository 层讲的是同一种语言（"findByUsername / saveUser"），调用者多了一层心智跳转，却没有获得任何新概念。
  • 没有隐藏任何复杂度： 如果底层加了缓存、分库逻辑、审计日志，上层也要跟着改——两层同时承压。
  • 测试意义降低： 对 Service 的单元测试等价于对 Repository 的测试，测试成本翻倍但覆盖率没有提升。
  • 违反"深模块"原则： 接口宽度与 Repository 相同，深度（隐藏的内部逻辑）却为零。
正例：Service 层引入更高抽象，替换词汇表
Java — 抽象层次提升（正例）
// ===== 数据访问层（保持不变）=====
public class UserRepository {
    public Optional<User> findByUsername(String username) { ... }
    public void             save(User user) { ... }
    public void             deleteByUsername(String username) { ... }
}

// ===== 服务层：使用业务词汇，而非数据库词汇 =====
public class UserService {
    private final UserRepository repo;
    private final PasswordEncoder encoder;
    private final AuditLogger     audit;

    /**
     * "注册"是业务概念：包含密码加密、唯一性校验、审计三件事。
     * 调用者不需要知道底层有几张表、用了什么加密算法。
     */
    public void register(String username, String rawPassword) {
        if (repo.findByUsername(username).isPresent())
            throw new DuplicateUsernameException(username);

        User user = User.of(username, encoder.encode(rawPassword));
        repo.save(user);
        audit.log("USER_REGISTERED", username);
    }

    /**
     * "注销账号"是业务操作：先验证权限，再删除，最后失效会话。
     * 三个底层步骤在此被合并为一个有意义的动作。
     */
    public void deactivateAccount(String username, User operator) {
        checkPermission(operator, "DEACTIVATE");
        repo.deleteByUsername(username);
        audit.log("ACCOUNT_DEACTIVATED", username);
    }
}

正例做了什么？

  • register / deactivateAccount 是业务词汇，findByUsername / deleteByUsername 是数据词汇——两层讲不同的语言，这就是"词汇表替换"。
  • Service 层把"密码加密 + 唯一性校验 + 审计"三步封装进 register，调用者的认知负担从 3 件事降为 1 件事。
  • 底层换数据库、换加密算法，只需改 Repository 和 encoder，Service 的接口不变——这才是真正隐藏了复杂度。

直观模型：词汇表替换

可以把每一层想象成一台翻译机，把下层的"方言"翻译成上层能听懂的"标准语"：

Controller 层
    ↑ 讲：用户意图（"注册账号" / "注销账号"）
────────── 抽象边界：业务动词 ──────────
Service 层
    ↑ 讲：业务规则（校验 + 编排 + 审计）
────────── 抽象边界：领域对象 ──────────
Repository 层
    ↑ 讲：数据操作（findBy / save / delete）
────────── 抽象边界：SQL / ORM ──────────
数据库驱动层
    ↑ 讲：字节流 / 连接池 / 事务日志

每穿越一条虚线，词汇就换一套。若两层之间穿越之后词汇没有变化，那条虚线就是多余的——这就是 Ousterhout 所说的"层次没有增加新的抽象"。

相邻层抽象相同的典型表现

症状描述后果
直通方法（pass-through） 上层方法只有一行，直接调用下层同名方法 层次徒增心智负担，无任何额外价值
接口镜像（interface mirroring） 上层接口与下层接口方法列表几乎相同 修改底层时上层必须同步修改，变更成本×2
概念下沉（concept leakage） 上层方法参数/返回值使用底层专有类型（如数据库 ResultSet） 上层被底层实现细节污染，无法独立测试
异常透传（exception tunneling） 上层原样抛出底层异常（如 SQLException）而不转译 调用者被迫理解底层存储细节

实际开发中的落地技巧

技巧 1：用"词汇测试"审查每一层

写完一个新类之后，问自己：这一层使用的词汇，和它调用的那一层的词汇，有多少重叠？ 如果重叠超过 70%，这一层大概率是多余的，或者没有完成应有的抽象工作。

技巧 2：异常在层边界处转译

Java — 异常转译
// 反例：直接把 SQL 异常扔给调用方
public User findUser(String id) throws SQLException {  // ← 底层细节泄漏
    return repo.findById(id);
}

// 正例：在层边界处换"方言"
public User findUser(String id) {
    try {
        return repo.findById(id)
                   .orElseThrow(() -> new UserNotFoundException(id));
    } catch (DataAccessException e) {
        // 把存储异常翻译为业务异常，调用者无需了解数据库
        throw new UserServiceException("无法查询用户: " + id, e);
    }
}

技巧 3：用"能否独立测试"验证抽象是否正确

如果对 Service 层的单元测试需要 mock 十几个 Repository 方法才能运行， 说明 Service 层没有真正封装逻辑——它的测试等价于集成测试。 正确封装的 Service 层只需 mock 少量、高层次的接口，测试用例自然简洁。

技巧 4：DTO 不要直接透传实体

Java — DTO 转换防止概念下沉
// 反例：Controller 直接拿到数据库实体，包含密码哈希等敏感字段
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) {
    return userService.findEntityById(id);  // ← 实体直接暴露给 HTTP 层
}

// 正例：Service 层负责转换，HTTP 层只看到对外契约 DTO
@GetMapping("/users/{id}")
public UserProfileDTO getUser(@PathVariable Long id) {
    return userService.getProfile(id);  // Service 完成实体→DTO 映射
}

技巧 5：层次不是越多越好

"不同层次不同抽象"并不意味着要拆更多层——过度分层会带来另一种复杂度：间接层爆炸。 正确的判断标准是：每新增一层，必须能说清楚"这一层隐藏了什么复杂度、提升了什么抽象"。 说不清楚，就不加这一层。

常见陷阱： 许多团队为了"符合规范"而机械地拆出 Service → Manager → Handler → Helper 四层， 每层方法都是直通的。这不是分层，这是把一个类拆成了四个文件——复杂度没有减少，反而增加了导航成本。

在 Spring 生态中的对应关系

Spring 层次应讲的词汇不该出现的词汇
@Controller / @RestController HTTP 动词、请求参数、响应状态码、DTO SQL、数据库实体、事务注解
@Service 业务规则、用例动词（register / approve / cancel）、领域异常 JDBC、ResultSet、HTTP 状态码
@Repository CRUD 操作、查询条件、分页 业务规则、权限校验、邮件发送

与其他原则的联系

  • 深模块（Deep Module）： 本节是深模块的纵向版本——不是单个类要深，而是整个调用栈的每一层都要深，而非直通。
  • 信息隐藏： 层次边界天然是信息屏障。不同层次的抽象才能让每一层只暴露必要信息，把实现细节关在层内。
  • 通用接口： 上层定义的抽象越通用，底层替换（换数据库、换框架）的成本就越低，这也是"通用比专用更简单"的体现。
  • 单一职责： 若一个类跨越了两个抽象层次（既讲业务又讲 SQL），它就违反了单一职责——本质上是抽象层次混乱的另一种表述。
一句话记忆： 好的分层让你在看某一层代码时，可以完全忘记下面所有层的存在—— 这正是"不同层次应有不同抽象"想要实现的效果。
