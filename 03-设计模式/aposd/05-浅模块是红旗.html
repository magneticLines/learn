05. 浅模块是红旗

接口和实现一样复杂——这是系统腐化的早期信号，也是最容易被"好代码习惯"掩盖的反模式。

核心主张：浅模块不能隐藏复杂性

John Ousterhout 在《软件设计的哲学》第四章给出了一个衡量模块好坏的核心公式： 模块的价值 = 它提供的功能 / 调用者需要了解的知识量。 一个深模块（Deep Module）的接口窄、实现深——调用者只需理解极少的概念就能获得大量能力。 而浅模块（Shallow Module）则恰好相反：接口的复杂度几乎和内部实现持平， 几乎没有任何"信息隐藏"可言，所以它对调用者没有带来任何认知节省。

Ousterhout 明确称浅模块为 "红旗"（Red Flag）——不是说它一定是错的， 而是说每当你看到它，都应该停下来问：这个模块到底隐藏了什么？ 如果答案是"什么都没隐藏"，那它的存在只是在增加代码体量，而没有降低系统复杂度。

原文核心句：「A shallow module is one whose interface is not much simpler than its implementation.」——《A Philosophy of Software Design》, Chapter 4

为什么浅模块有害？

软件复杂性有两个来源：

  • 依赖（dependencies）——修改一处，必须同时修改多处
  • 模糊（obscurity）——不读实现就无法理解如何正确调用

深模块把复杂性包裹在内部，对外只暴露简单的接口，因此减少了依赖的传播面。 浅模块两边都复杂，调用者必须同时理解接口 和 内部实现才能正确使用， 等于把实现细节以另一种名字重新暴露了出来，反而增加了认知负担。

常见误区：把"小方法 = 好设计"当作金科玉律。单一职责原则（SRP）的目标是降低复杂度，而不是强制拆小。如果拆分后每个方法都只有两行且接口和实现一样简单，就已经滑向了浅模块的陷阱。

反例 → 问题 → 正例

场景：读取文件内容

假设团队决定把文件读取封装成"工具类"，下面是一个典型的浅模块写法：

反例 — 浅模块（Java）FileUtil.java
public class FileUtil {

    // 接口：openFile(path) → InputStream
    // 实现：new FileInputStream(path)  ← 几乎等价，零隐藏
    public static InputStream openFile(String path) throws IOException {
        return new FileInputStream(path);
    }

    // 接口：readBytes(stream) → byte[]
    // 实现：stream.readAllBytes()  ← 直接透传，调用者还不如直接调 JDK
    public static byte[] readBytes(InputStream stream) throws IOException {
        return stream.readAllBytes();
    }

    // 接口：closeStream(stream)
    // 实现：stream.close()  ← 这根本不是封装，只是改了个名字
    public static void closeStream(InputStream stream) throws IOException {
        stream.close();
    }
}

调用侧依然需要：

  • 知道要先 openFile，再 readBytes，再 closeStream（顺序依赖暴露给了调用者）
  • 自行处理 IOException 的时机和方式
  • 确保异常时也能 close（资源泄漏风险完全没被隐藏）

这三个方法合在一起的接口复杂度（需要了解的知识量），几乎和直接使用 FileInputStream + try-with-resources 一样高。 封装没有带来任何认知节省，只是多了三个间接层。

调用者需要知道的事（反例）
├── openFile → 内部 = new FileInputStream(path) ← 等价暴露
├── readBytes → 内部 = stream.readAllBytes() ← 等价暴露
├── close → 内部 = stream.close() ← 等价暴露
└── 调用顺序 / 异常安全 ← 仍由调用者承担
正例 — 深模块（Java）FileUtil.java
public class FileUtil {

    /**
     * 读取文件的全部字节内容。
     *
     * 接口极简：只需一个路径，返回 byte[]。
     * 内部隐藏了：资源打开 / 读取 / 关闭 / 异常包装，
     * 调用者完全不需要了解 InputStream 的存在。
     */
    public static byte[] readAllBytes(String path) {
        // try-with-resources 确保任何路径下流都会被关闭
        try (InputStream in = new FileInputStream(path)) {
            return in.readAllBytes();
        } catch (IOException e) {
            // 将受检异常包装成非受检，避免调用者处处 try-catch
            throw new UncheckedIOException("Failed to read: " + path, e);
        }
    }

    /** 同上，返回 UTF-8 字符串，再进一步隐藏编码细节 */
    public static String readText(String path) {
        return new String(readAllBytes(path), StandardCharsets.UTF_8);
    }
}

现在调用侧只需：

调用侧对比
// 反例：调用者需要管 3 件事 + 异常安全
InputStream in = FileUtil.openFile(path);
byte[] data = FileUtil.readBytes(in);
FileUtil.closeStream(in);   // 如果 readBytes 抛异常，这行永远不会执行！

// 正例：调用者只需 1 行，InputStream 不再泄漏到外部
String content = FileUtil.readText(path);
调用者需要知道的事（正例）
└── readText(path) → String ← 唯一需要了解的概念

内部隐藏了
├── FileInputStream 的创建与关闭
├── try-with-resources 保证的资源安全
├── IOException → UncheckedIOException 的转换
└── UTF-8 编码细节

识别浅模块的三个信号

信号 具体表现 说明
接口参数 ≈ 实现步骤数 一个方法 3 行，接口有 3 个参数 接口信息量没有减少，只是换了层名字
方法名是实现动词 openConnection、closeConnection、sendRequest、readResponse 把内部实现流程直接作为接口，调用者仍需了解流程顺序
异常原样透传 内部抛 SQLException，接口 throws SQLException 存储层细节泄漏到业务层，替换数据库时所有调用者都要改

常见的"以浅为深"错觉

错觉 1：方法短 = 职责单一 = 好

Clean Code 提倡"方法要短"，这没有错，但目的是减少单个方法内的复杂度， 不是把一个完整的操作肢解成五个两行方法再全部暴露给调用者。 拆分之后如果调用者仍然需要按固定顺序调用这五个方法，复杂度只是从方法内转移到了调用者那里， 系统整体复杂度没有降低。

错觉 2：Pass-through 方法是分层的体现

一个 Service 方法只是调用一个 Repository 方法，参数一模一样、返回值原样透传—— 这是 Ousterhout 专门批评的 pass-through method，是浅模块的极端形式。 它既没有增加功能，也没有隐藏任何信息，只是在调用链上插了一个无意义的中间层。

pass-through method 典型形态
// Service 层 — 什么都没做，只是转发
public User getUserById(Long id) {
    return userRepository.findById(id).orElse(null);
}

// 如果 Service 层应该存在，它至少应该隐藏一些东西：
public User getUserById(Long id) {
    // 权限校验、缓存、Optional 展开、业务异常转换——这些才是 Service 该做的事
    checkPermission(id);
    return userRepository.findById(id)
        .orElseThrow(() -> new UserNotFoundException(id));
}

实际开发落地技巧

技巧 1：用"调用者视角测试"衡量模块深度

写完一个新方法/类之后，问自己：调用者必须阅读我的实现才能正确使用这个接口吗？ 如果是，说明接口没有足够隐藏实现细节，需要重新设计接口，或者把更多决策移入实现内部。

技巧 2：让异常属于接口层的语义

底层异常（SQLException、IOException）暴露了存储实现， 应该在模块边界处转换为业务语义的异常（UserNotFoundException、StorageException）。 这是深模块最直接的体现之一：调用者不需要知道数据存在哪里。

技巧 3：区分"拆分以降低复杂度"和"拆分以增加层次"

拆分一个方法的合理理由只有一个：拆分之后，每一部分都比整体更容易单独理解。 如果拆分只是为了让每个方法"看起来短"，但调用者仍需了解所有拆出来的部分，就没有收益。

技巧 4：Facade 模式是深模块的标准用法

设计模式中的 Facade 就是在为一组浅的子系统接口构建一个深模块—— 用一个简单的入口把底层多步骤调用封装起来。 当你发现调用者每次都要重复 3 步以上才能完成一件事，那就是建立 Facade 的信号。

面试高频考点：「什么是好的封装？」标准答案不是"方法要短"，而是「调用者需要了解的信息量最小化」。可以用深模块 vs 浅模块的框架来回答，会让面试官印象深刻。

与其他概念的联系

概念与浅模块的关系
信息隐藏（Information Hiding） 浅模块是信息隐藏失败的直接结果——没有隐藏任何实现决策
单一职责原则（SRP） SRP 的目标是减少变更影响范围，不是强制产生浅模块；误用 SRP 会导致大量 pass-through
Facade 模式 Facade 是主动构建深模块的手段，把多个浅接口合并为一个深接口
复杂性下沉（Pull complexity downward） 下一节的主题：主动把复杂决策放进模块内部，是避免浅模块的根本方法
一句话总结：浅模块是红旗，不是因为它"错了"，而是因为它没有完成模块存在的使命——把复杂性收进去，让调用者的世界变得更简单。每看到一个浅模块，就问：我能把什么决策再多吸收进来一点？
