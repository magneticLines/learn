14. 注释不应重复代码

Comments Should Describe Things that Aren't Obvious from the Code

核心主张：注释要提供代码以外的信息

John Ousterhout 在 《A Philosophy of Software Design》 中开宗明义：注释的唯一价值在于给读者带来代码本身无法直接传达的信息。 如果一行注释只是把代码用自然语言"翻译"了一遍，读者需要读两遍才能得到一份信息——这是纯粹的噪音，没有任何增益。

更危险的是，重复性注释会随时间与代码产生漂移：代码被修改了，注释还停留在旧版本，读者就会被误导。 Ousterhout 把这类注释称为"低层次注释"（low-level comments），并认为它们比没有注释更糟糕。

设计哲学一句话：好的注释回答的是"为什么"和"是什么（抽象层次更高的语义）"，而不是"怎么做"——代码本身已经在说怎么做了。

为什么"不重复代码"能让系统更好

这一原则背后有三层更深的工程理由：

  • 降低维护成本。 注释与代码是两份"真相"。重复越多，同步成本越高，漂移风险越大。删掉冗余注释就是消灭潜在的不一致。
  • 迫使作者澄清设计意图。 当你发现"写注释只是在复述代码"时，往往说明代码命名不够好——真正该做的是改进命名，而不是加注释。Ousterhout 认为这是注释倒逼设计质量的"副作用"。
  • 保持读者的注意力。 读者会学着跳过重复注释，最终养成不读注释的习惯——让真正重要的注释也被忽略。

反例 → 问题 → 正例

场景一：逐行翻译型注释

最常见的反模式：注释几乎是代码的中文直译。

Java — 反例（重复代码）
// 将 i 加 1
i++;

// 如果用户未登录，返回 null
if (!user.isLoggedIn()) {
    return null;
}

// 创建一个新的 HashMap
Map<String, Integer> counter = new HashMap<>();
问题： 这三行注释没有传递任何代码之外的信息。删掉它们，代码的可读性分毫未损；保留它们，只增加了后续维护时"需要一起改"的负担。
Java — 正例（删除冗余，或改写为有意义的注释）
i++;  // 注释直接删除，代码已经自解释

// 未认证用户不应触达后续业务逻辑，由调用方负责处理 null
if (!user.isLoggedIn()) {
    return null;
}

// 用 HashMap 而非 TreeMap：此处不需要有序遍历，O(1) 查找优先
Map<String, Integer> counter = new HashMap<>();

修改后的注释回答了"为什么这样做"或"为什么不用另一种方案"——这是代码本身无法直接表达的信息。

场景二：方法签名复述型注释

另一类重灾区：Javadoc/JSDoc 注释只是把参数名和方法名重新排列了一遍。

Java — 反例（Javadoc 重复方法签名）
/**
 * 获取用户
 * @param userId 用户 ID
 * @return 用户对象
 */
public User getUser(long userId) { ... }

/**
 * 添加元素到列表
 * @param element 要添加的元素
 * @param index   插入位置
 */
public void addElement(T element, int index) { ... }
问题： 读者看完注释得到的信息量和只看方法签名完全相同。写注释花了时间，却没有产生任何文档价值。
Java — 正例（Javadoc 补充真正缺失的信息）
/**
 * 通过主键查询用户。
 *
 * <p>结果来自 L2 缓存（TTL 5 min），不保证强一致性。
 * 若需实时数据，请改用 {@link #getUserNoCache(long)}.
 *
 * @param userId 用户主键，必须 > 0
 * @return 用户对象；若不存在返回 {@code null}（而非抛异常）
 */
public User getUser(long userId) { ... }

/**
 * 在指定位置插入元素，后续元素依次右移。
 *
 * <p>与 {@link java.util.List#add(int, Object)} 语义一致，
 * 但针对大批量插入做了批量预分配优化，时间复杂度 O(1) 均摊。
 *
 * @param element 不可为 null
 * @param index   [0, size]，超出范围抛 {@link IndexOutOfBoundsException}
 */
public void addElement(T element, int index) { ... }

正例补充了：缓存策略、一致性语义、null 行为约定、边界条件、与标准库的对比——这些全都是读者需要但代码本身表达不了的信息。

场景三：算法步骤复述型注释

稍微复杂的算法里，很多人习惯在每一步前加注释"解释"这一步在做什么，结果注释和代码几乎同义。

Java — 反例（逐步骤复述算法）
// 遍历列表
for (Order order : orders) {
    // 判断订单是否已支付
    if (order.isPaid()) {
        // 将金额加到总金额上
        total += order.getAmount();
    }
}
问题： 三行注释把代码完整复述了一遍。如果将来改为"只统计本月订单"，注释很可能被遗忘，形成误导性文档。
Java — 正例（用注释说明"为什么"，让代码说明"怎么做"）
// 只统计已支付订单：待支付和已退款状态不计入营收报表
// 参见 PRD §3.2：退款在独立的退款流水中统计，避免重复计算
for (Order order : orders) {
    if (order.isPaid()) {
        total += order.getAmount();
    }
}

一处简洁的注释，交代了业务规则的来源和防止误改的背景知识，比三行废话注释有价值得多。

什么信息值得写进注释

Ousterhout 给出了一个判断标准：注释要提供比代码更高层次的抽象，或者代码结构本身无法表达的背景知识。常见的有价值信息类型：

信息类型 示例 代码能自表达吗？
业务规则 / 需求来源 为什么要过滤掉退款订单 否
设计决策 / 方案取舍 为什么选 HashMap 而非 TreeMap 否
非显然的边界条件 返回 null vs 抛异常的区别 部分（需看实现）
性能 / 并发 / 安全假设 此方法非线程安全，调用方加锁 否
历史遗留原因 由于旧版 API 兼容不得不保留此魔法数字 否
代码功能的中文翻译 i++ → "将 i 加 1" 是（冗余！）

实际开发中的落地技巧

技巧一：写注释前先问自己一个问题

"这条注释能告诉读者什么代码告诉不了他的东西？" 如果答案是"没有"，就不要写。如果你发现很难回答，说明可能需要的是更好的命名，而不是注释。

技巧二：把"为什么不"也写进来

很多设计决策的价值在于排除了其他选项。注释中写出"没有用 X 是因为……"往往比只说"用了 Y"更有价值，可以阻止后来者"聪明地"换回去。

Java — "为什么不"注释示例
// 使用 synchronized 而非 ReentrantLock：
// 该路径是低频冷路径，锁竞争极少，synchronized 的 JIT 偏向锁优化
// 足够应对，引入 ReentrantLock 只会增加可读性负担。
synchronized (this) {
    cache.put(key, value);
}

技巧三：用注释标记"非显然的副作用"

方法签名能表达输入输出，但无法表达副作用。调用某个方法时会发邮件、写数据库、清除缓存——这些都值得在调用点附近用注释标记。

Java — 副作用注释示例
// 注意：此方法会同步清除该用户的所有 Redis 会话缓存，
// 在高并发场景下不应在循环中调用
authService.invalidateUser(userId);

技巧四：接口注释 vs 实现注释分开写

Ousterhout 强调接口注释（方法签名上的 Javadoc）说明使用契约，实现内部注释说明实现细节中不明显的地方。 不要在接口注释里解释实现，也不要把使用契约混进实现注释——读者需要的是不同层次的信息。

注释的信息层次 ├── 接口注释（Javadoc） │ ├── 方法的语义（做什么，不是怎么做） │ ├── 参数约束（null? 范围? 格式?） │ ├── 返回值语义（含异常情况） │ └── 副作用 / 线程安全性 └── 实现注释（行内 // ） ├── 非显然的算法选择原因 ├── 历史遗留原因 / 兼容性妥协 ├── 业务规则来源 └── 性能 / 并发假设

技巧五：命名好了，注释自然少

如果你频繁发现"注释只是在复述代码"，背后的信号是命名还不够好。 getUserById 比 getUser 自解释；isPaidOrder 比 checkOrder 自解释。 先重构命名，再决定是否还需要注释。

容易忽略的细节与常见误区

陷阱一：注释跟着复制粘贴但没有更新
复制一段代码时，注释往往也被复制过来了，但逻辑已经修改。旧注释留在新代码旁边，比没有注释更危险。每次复制代码后，把注释当成第一个需要审查的东西。
陷阱二：TODO 注释无限期存活
// TODO: 临时方案，后续优化 写于三年前，已成永久方案。如果 TODO 没有 issue 编号或截止时间，它大概率永远不会被处理，反而成为干扰。要么关联 issue，要么删掉。
陷阱三：注释写了"是什么"但没写"为什么"
// 乘以 1000 还是重复代码；// 将秒转为毫秒，适配下游 SDK 要求的时间单位 才有价值。同样是一句话，信息量差了一个数量级。
检验标准（Ousterhout 原话改写）：好的注释让一个熟悉编程语言但不熟悉这个模块的人，能够快速理解这段代码的设计意图——而不需要通读实现细节。

小结

"注释不应重复代码"这条原则，本质上是在说：注释和代码应该在不同的抽象层次上工作，互相补充，而不是互相翻译。 代码说"怎么做"，注释说"为什么这么做"、"有什么约定"、"做了哪些取舍"。 两者各司其职，才能构成真正清晰的文档。

删掉冗余注释不是懒惰，而是一种工程纪律：它迫使你把信息表达的责任放回到命名和代码结构上， 同时让真正重要的注释不被淹没在噪音里。
