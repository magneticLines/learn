08. 通用比专用更简单

General-Purpose Modules are Simpler — 为什么设计稍微通用一点的接口，反而能让系统更简单、信息隐藏更彻底

作者的核心主张

John Ousterhout 在《软件设计的哲学》第六章抛出了一个反直觉的论断： 当你在设计一个模块时，与其把接口做得刚好满足当前调用方的需求，不如把它做得稍微通用一些——这样反而会让整个系统更简单。

初听起来很奇怪——"通用"不是意味着要多写代码、多处理边界情况吗？作者的答案是：专用接口把上层的"意图"渗透进了下层的实现，造成信息泄漏；通用接口只暴露能力，把意图留在调用方，信息隐藏更彻底，反而总量更少。

一句话记忆：专用接口 = 为调用方量身定制，下层知道太多上层的事；通用接口 = 只提供能力，上层自己组合，知识边界更清晰。

为什么"专用"反而更复杂

想象一个文本编辑器的 TextBuffer 类。最初开发者只需要实现"删除选中文字"这个功能，于是写了：

❌ 反例 — 专用接口（为 UI 事件量身定制）Java
// 接口名和参数都在描述 UI 层的操作语义
public interface TextBuffer {

    // "删除光标选中区域" — 上层的 UI 语义下沉到了下层接口
    void deleteSelection();

    // "在光标处插入用户输入的字符" — 同样依赖"光标"这个 UI 概念
    void insertAtCursor(String text);

    // "把选中文字高亮" — UI 渲染细节也进来了
    void highlightSelection(Color color);
}

问题在哪里？

  • 信息泄漏：TextBuffer 必须知道"光标"、"选区"等 UI 概念，上层状态渗入下层。
  • 接口爆炸：每当 UI 增加新操作（Ctrl+D 删一行、拖拽移动），都要往 TextBuffer 里加方法。
  • 难以复用：这个 TextBuffer 无法被命令行工具、测试脚本、批处理程序复用——它只服务于这一个 GUI。
  • 测试困难：测试下层逻辑必须先构造出"光标选区"这种 UI 状态，耦合太深。

通用接口：只暴露能力，不暴露意图

把 TextBuffer 改成以"位置索引"为基础的通用接口——它只回答"我能对文本做什么"，而不关心"调用方想在什么场景下做"：

✅ 正例 — 通用接口（只描述文本能力）Java
// 接口只描述文本操作的本质：位置 + 内容，没有任何 UI 概念
public interface TextBuffer {

    // 在任意位置插入文本
    void insert(int position, String text);

    // 删除 [start, end) 范围内的文本
    void delete(int start, int end);

    // 读取 [start, end) 范围内的文本
    String get(int start, int end);

    // 文本总长度
    int length();
}

// ── 上层 UI 层：用通用接口组合出专用行为 ──────────────────
public class EditorController {

    private final TextBuffer buffer;
    private final SelectionModel selection; // 光标/选区归 UI 层管理

    // "删除选中区域"的语义由上层组合实现，下层不需要知道
    public void deleteSelection() {
        buffer.delete(selection.getStart(), selection.getEnd());
    }

    // "在光标处插入" 同理，UI 层自己维护 cursor 位置
    public void insertAtCursor(String text) {
        buffer.insert(selection.getCursorPosition(), text);
    }
}

现在 TextBuffer 变干净了：4 个方法覆盖任意文本操作，UI 层的光标、选区、高亮全部留在了 UI 层。新增 UI 功能不需要改 TextBuffer。

关键洞察：通用接口并没有让下层代码变多——反而更少了。专用接口需要为每种场景写一个方法；通用接口只提供正交的原语，场景由上层组合。正交性带来的是乘法而不是加法。

用"正交原语"理解通用性

通用接口的本质是提供正交的原语（orthogonal primitives）：每个方法只做一件事，方法之间没有语义重叠，组合起来可以表达任意上层意图。

专用接口（加法扩展）
├── deleteSelection() ← 为某个 UI 手势定制
├── insertAtCursor() ← 为某个 UI 手势定制
├── deleteCurrentLine() ← 又一个手势加进来…
├── duplicateLine() ← 继续加…
└── ... 无限膨胀

通用接口（乘法组合）
├── insert(pos, text) ← 原语 A
├── delete(start, end) ← 原语 B
├── get(start, end) ← 原语 C
└── length() ← 原语 D
A×B×C×D 的组合 = 覆盖所有上层场景

"稍微"通用：不是过度设计

作者特别强调，他说的通用是"somewhat general-purpose"（稍微通用一些），不是让你构建一个万能框架。 判断标准只有一个：这个接口是否只服务于当前这一个调用场景？

维度 专用接口（过于具体） 适度通用接口 过度通用（YAGNI 反例）
方法命名 deleteSelection() delete(start, end) applyOperation(Op op)
参数语义 依赖 UI 状态（光标） 依赖本领域概念（索引） 完全抽象，调用方迷失
信息隐藏 下层知道上层意图 下层只知道自己的领域 接口本身也不知道自己在做什么
问题 接口爆炸、耦合深 —— 目标状态 接口晦涩，认知成本高
陷阱：不要把"通用"等同于"增加 Strategy / Template Method 等设计模式"。作者的意思更简单——把接口的参数从"上层意图"替换为"本领域的自然单位"，仅此而已。过早引入抽象框架反而是另一种专用（为未来的猜测量身定制）。

设计时问自己三个问题

Ousterhout 给出了一组自测问题，帮助判断当前接口是否过于专用：

  1. 这个方法的最简单接口是什么？——去掉所有只为当前调用方服务的参数和语义，剩下的就是通用核心。
  2. 在多少种不同场景下我可以使用这个方法？——如果答案是"只有一种"，很可能过于专用。
  3. 这个 API 对用户来说容易使用吗？——通用接口往往更好学，因为它遵循领域自然语言，而不是当前系统的特殊惯例。

实际开发中的落地技巧

技巧 1：把"动词+名词"从上层语境替换为领域语境

方法名中出现 selected、current、active、clicked 等词时，往往是专用的信号——这些词描述的是调用方的状态，不是被调用模块的领域概念。 把它们替换为领域原语（索引、ID、范围、键值）。

技巧 2：让调用方传入"定位信息"，而不是让下层持有上层状态

对比：谁来持有"位置"Java
// ❌ 下层持有光标 — 下层必须知道 UI 的选区概念
class TextBuffer {
    private Cursor cursor;       // UI 状态渗入下层
    void deleteAtCursor() { ... }
}

// ✅ 调用方传入位置 — 下层只关心索引，不关心谁决定了这个索引
class TextBuffer {
    void delete(int start, int end) { ... }  // 纯粹的文本领域概念
}

技巧 3：Repository / DAO 层的常见陷阱

后端开发中，Repository 方法经常因为某个 Service 的需求而"定制化"，最终变成：

专用 vs 通用 Repository 方法Java
// ❌ 专用 — 为某个页面的业务逻辑量身定制
List<Order> findActiveOrdersByCurrentUserAndThisMonth();

// ✅ 通用 — Repository 只暴露查询能力，Service 层传入条件
List<Order> findOrders(OrderQuery query);

// OrderQuery 由 Service 层根据业务逻辑构建
OrderQuery q = OrderQuery.builder()
    .userId(currentUserId)
    .status(OrderStatus.ACTIVE)
    .createdAfter(startOfMonth())
    .build();
repo.findOrders(q);

通用版本的 Repository 方法数量大幅减少，每次新增业务需求只需要在 OrderQuery 里加字段，不需要在 Repository 接口上添加新方法。

技巧 4：识别"接口在向上层要东西"

如果一个方法需要从调用方获取的信息，是调用方的内部状态（不是本模块自然需要的参数），那就是专用的信号。 例如：sendEmail(User loggedInUser, ...) 里的 loggedInUser 如果只是为了获取邮件地址，更通用的签名应该是 sendEmail(String toAddress, ...)——让调用方自己提取，下层只要邮件地址。

与信息隐藏的关系

通用接口是信息隐藏的自然延伸。信息隐藏要求模块只暴露必要的信息；而专用接口恰恰相反——它要求下层必须知道上层的意图，是一种反向的信息泄漏。

信息隐藏的目标
│
├── 隐藏实现细节 ─── 经典信息隐藏（第 07 节）
│
└── 隐藏上层意图 ─── 通用接口解决的问题（本节）
└── 下层不应知道"为什么被调用"，只提供能力

这两个方向共同构成了信息隐藏的完整图景：向下隐藏实现，向上隔离意图。模块只应了解自己领域内的知识，既不暴露内部，也不吸收外部语境。

面试高频考点：问"如何避免接口爆炸（方法数量膨胀）"时，可以回答：将专用方法替换为通用原语 + 调用方组合，同时把决策逻辑（哪些条件、哪些状态）上推到调用层，让下层保持对上层业务意图的无知。

一句话总结

接口的通用性不是为了"以后可能用到"，而是为了现在就把上层意图挡在边界之外。 让每一层只了解自己领域的语言，复杂度自然向上收敛到最合适处理它的地方。
