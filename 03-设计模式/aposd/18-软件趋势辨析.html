18. 软件趋势辨析

OOP、TDD、设计模式、getter-setter —— 流行的就是对的吗？Ousterhout 逐一拆解这些趋势，还原它们与"降低复杂度"目标的真实关系。

作者的核心立场

《软件设计的哲学》第 19–21 章专门审视几种被广泛推崇的"软件趋势"。Ousterhout 的判断标准只有一个：这种实践是否有助于降低复杂度（complexity）、提升深度（depth）、减少认知负担？

他的结论并非"XX 是好的 / 坏的"，而是：每种趋势都有其正当内核，但被过度教条化之后，往往带来相反效果——产生更多浅层模块、暴露更多信息、增加更多文件和类。

判断某个实践好不好，永远要问：它让接口变简单了，还是让实现细节向调用者泄漏了？ 这才是本质问题，而不是"是否符合某种范式"。

面向对象（OOP）：封装是核心，继承要谨慎

OOP 的核心价值是封装——把数据和操作数据的代码捆绑在一起，对外只暴露行为，隐藏实现细节。这与"深模块"高度吻合。

但有两个常见误区：

  • 把每个字段都建一个类：类数量爆炸，每个类只做一件微不足道的事，整体认知成本反而上升。
  • 过度依赖继承：继承在父类和子类之间建立了极强的耦合，父类的任何改动都可能破坏子类。组合（composition）通常比继承更灵活、更局部化。

反例 — 为"数据传输"单独建一堆类

Java — 过度拆分的 OOP
// 问题：UserNameDto、UserEmailDto 都是只有一个字段的"空壳类"
// 调用者需要组装三个对象才能完成一次用户创建，认知成本极高
class UserNameDto {
    private String name;
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}
class UserEmailDto {
    private String email;
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}
// 调用侧：三步才能传一个"用户"
UserNameDto n = new UserNameDto(); n.setName("Alice");
UserEmailDto e = new UserEmailDto(); e.setEmail("a@x.com");
createUser(n, e);

正例 — 合并为有意义的聚合对象

Java — 合理封装
// 一个 UserCreateCmd 包含所有必要信息，接口简洁
record UserCreateCmd(String name, String email) {}

// 调用侧：一行，意图清晰
createUser(new UserCreateCmd("Alice", "a@x.com"));
落地技巧：类的数量不是衡量 OOP 质量的指标。衡量指标是：调用者需要了解多少内部细节才能正确使用这个类？越少越好。

测试驱动开发（TDD）：发现接口问题的工具，而非设计工具

TDD 的正当价值在于：先写测试，迫使你在实现之前思考接口应该如何被调用——这是一种非常有效的"用户视角审查"。如果写测试很痛苦，往往说明接口设计有问题。

但 Ousterhout 指出 TDD 一个危险的副作用：开发者的注意力从"整体设计"转移到"让当前测试通过"。这会导致：

  • 每次只改一点点，缺乏对整体结构的宏观思考；
  • 为了可测试性拆分出大量微型方法，产生浅层模块；
  • 测试本身成为维护负担，阻碍后续重构。

反例 — 为测试而拆分出浅层方法

Java — 为"可测试性"过度拆分
// 原本一行逻辑：price * (1 - discount)
// 为了单独测试每一步，被拆成三个微型方法
public double getBasePrice() { return price; }
public double getDiscountRate() { return discount; }
public double applyDiscount(double base, double rate) {
    return base * (1 - rate);
}
// 问题：调用者被迫理解三步才能算出折扣价，信息泄漏

正例 — 深方法，测试验证行为而非实现

Java — 深方法 + 行为测试
// 封装实现细节，对外只暴露行为
public double discountedPrice() {
    return price * (1 - discount);
}

// 测试验证"行为"，不关心内部拆不拆
@Test
void discountedPrice_shouldApplyRate() {
    Product p = new Product(100, 0.2);
    assertEquals(80.0, p.discountedPrice());
}
陷阱：如果你写测试时发现需要 mock 五六个依赖才能运行，这不是"要更多 mock"的信号，而是"模块边界划错了"的信号。先重新审视设计，再回来写测试。

设计模式：解决真实问题时用，别为用而用

设计模式本身是宝贵的经验结晶，但 Ousterhout 观察到一种反模式：把设计模式当成"显示专业水准"的工具，在不需要的地方强行套用。

每引入一个设计模式，都会引入额外的类、接口和间接层。当问题本身不复杂时，这些间接层只会增加认知负担，而不是减少它。

反例 — 不需要扩展的地方强行用工厂模式

Java — 过度设计的工厂
// 整个系统里只有一种 Logger，却引入了工厂 + 接口
interface Logger { void log(String msg); }
class ConsoleLogger implements Logger {
    public void log(String msg) { System.out.println(msg); }
}
class LoggerFactory {
    public static Logger create() { return new ConsoleLogger(); }
}
// 调用者要读三个文件才能明白"就是打印到控制台"

正例 — 当真正需要扩展时，模式才发挥价值

Java — 有真实扩展需求时的工厂
// 需求：生产环境用 CloudLogger，本地用 ConsoleLogger
// 此时工厂消除了调用者对环境的感知，是真正有价值的抽象
interface Logger { void log(String msg); }

class LoggerFactory {
    public static Logger create(Env env) {
        // 调用者不需要知道有 ConsoleLogger / CloudLogger 的区别
        return env == Env.PROD
            ? new CloudLogger()
            : new ConsoleLogger();
    }
}

判断要不要用某个设计模式，问一个问题：如果不用这个模式，今天的需求能否用更少的代码、更简单的结构同样清晰地表达？如果是，就不要用。

Ousterhout 的措辞是："设计模式本身没问题，问题是把它们当成目标而不是工具。" 模式是解决方案的词汇表，不是设计的起点。

getter / setter：封装的幻觉

这是 Ousterhout 批评最直接的一条。为每个私有字段机械地生成 getter + setter，并不是封装，而是把字段的存在暴露给了所有调用者——只不过加了一层函数调用的包装。

真正的封装是：调用者只能通过"行为"与对象交互，而不是通过"读写字段"。

字段暴露的三种形式（本质相同）：
├── public String name; ← 直接暴露
├── getName() / setName() ← 包装暴露（本质一样）
└── behavior() — 调用者只知道结果 ← 真正封装

反例 — getter-setter 让业务逻辑泄漏到调用者

Java — getter-setter 导致业务逻辑散落
// Order 只是一个数据袋，业务规则散落在调用者中
class Order {
    private double total;
    private String status;
    public double getTotal() { return total; }
    public String getStatus() { return status; }
    public void setStatus(String s) { this.status = s; }
}

// 调用者必须知道：total > 0 时才能支付，支付后状态改为 PAID
// 这个规则该在 Order 里，现在却泄漏到外面
if (order.getTotal() > 0
        && "PENDING".equals(order.getStatus())) {
    order.setStatus("PAID");
}

正例 — 把业务规则内聚到对象本身

Java — 行为封装
class Order {
    private double total;
    private String status = "PENDING";

    // 对外暴露"能不能支付"这个行为，而不是暴露字段
    public void pay() {
        if (total <= 0) throw new IllegalStateException("空订单不能支付");
        if (!"PENDING".equals(status)) throw new IllegalStateException("重复支付");
        this.status = "PAID";
    }

    public boolean isPaid() { return "PAID".equals(status); }
}

// 调用者：完全不需要知道 total、status 字段的存在
order.pay();
常见误区："但是 JavaBean 规范要求有 getter/setter！"——注意区分数据传输对象（DTO）和领域对象。DTO 本就是纯数据载体，getter/setter 完全合理。领域对象应该以行为为中心，不应该把自己的内部状态随意开放。

四种趋势的辨析汇总

趋势 正当内核 过度使用的危害 判断标准
OOP 封装隐藏实现细节，行为聚合 类爆炸，浅层模块，继承耦合 调用者需要了解多少内部信息？
TDD 先写测试迫使思考接口设计 注意力碎片化，为可测性产生浅层方法 测试是在验证行为，还是在绑定实现？
设计模式 解决真实扩展性问题的成熟方案 无谓的间接层，增加认知负担 不用这个模式，今天的需求能否更简单地表达？
getter/setter DTO 序列化、框架绑定 业务规则泄漏到调用者，封装名存实亡 调用者需要读写字段，还是只需要触发行为？

实际开发落地技巧

1. 以"调用者需要知道什么"为尺度审查 API

设计任何方法、类、模块时，默认姿势是：把信息藏起来，直到有充分理由暴露为止——而不是"能暴露就暴露"。这与 Java 的默认访问修饰符建议（字段 private，方法按需 public）一致。

2. 先写测试，但测试的粒度要匹配模块的粒度

不要为每个私有方法写测试。私有方法是实现细节，测试只应覆盖公开行为。如果你发现某个私有方法"感觉需要单独测试"，这往往是提取成独立类的信号，而不是应该改变可见性的信号。

3. 设计模式作为"重构目标"，而非"起点"

先用最简单的实现写出能工作的代码。当你观察到真实的扩展压力（多个 if-else 分支在增长、多处重复逻辑），再引入对应的模式。这比一开始就套模式产生的间接层少得多。

4. 区分 DTO 和领域对象，分别对待

getter/setter 在 DTO（数据传输对象）、数据库映射实体（如 JPA Entity）等场景完全合理，框架本身依赖它们。只需要确保这些类不携带业务逻辑。业务逻辑集中在领域对象（Domain Object）或服务（Service）中，以行为方法对外交互。

一句话原则：每次引入新的类、接口、设计模式或测试时，都问自己——这让整个系统更容易理解了，还是更难理解了？ 如果更难，就是过度设计。

与其他知识点的联系

  • 深模块 / 信息隐藏：本章对 OOP、设计模式、getter/setter 的评判标准，全部收敛到同一个尺度——"调用者需要知道多少内部信息"。这正是深模块理论的直接应用：好的实践应该让接口比实现简单，坏的滥用则会让接口和实现一样复杂甚至更复杂。
  • 17. 代码应显而易见：getter/setter 被批评的根本原因，是它让业务规则从对象内部"泄漏"到调用侧，调用者必须自己拼凑多个字段的组合逻辑才能理解"何时可以支付"——这正是"非显而易见"的典型表现。反过来，把规则封进 pay() 方法后，调用侧一行代码就说明了意图。
  • 19. 红旗清单：本章讨论的四种趋势的"过度使用危害"一栏，几乎可以逐条对应到红旗清单里的具体条目——类爆炸对应"浅模块"，业务规则泄漏对应"信息泄漏"，无谓间接层对应"直通方法"。红旗清单相当于给本章的辨析结论提供了一份可执行的检查表。
  • GoF 设计模式：本章③对设计模式的态度（"解决真实问题时用，别为用而用"）呼应了 GoF 原书序言中的告诫——模式是对已被反复验证的问题的解决方案词汇表,不是设计的起点。判断要不要用某个模式，最终还是要回到"这样做是否真的降低了复杂度"这个本章反复强调的唯一标准。
  • Clean Architecture 的 SOLID 原则：本章对继承和 OOP 的辨析，与 Clean Architecture 中里氏替换原则（LSP）、依赖倒置原则（DIP）讨论的问题高度重叠——继承带来的强耦合正是 LSP 试图约束的对象，而"接口该由谁定义"的问题在两本书里都有专门讨论，可以对照阅读加深理解。

核心主张一句话总结

Ousterhout 对所有软件趋势的评价标准从不改变：一个好的实践应当帮助你隐藏复杂度、深化模块、减少调用者的认知负担。任何趋势，一旦走向了相反的方向——制造更多浅层类、暴露更多细节、增加更多间接层——就应该降低它的优先级，回归简单直接的设计。

流行不等于正确。在你的具体问题面前，问原则，不问潮流。
