02. 复杂性来源：依赖与晦涩

John Ousterhout 认为，软件中所有的复杂性，几乎都能追溯到两个根源：依赖（Dependency） 和 晦涩（Obscurity）。

作者的核心主张

上一节我们看到了复杂性的三大症状（变更放大、认知负荷、未知的未知）。Ousterhout 进一步追问：这些症状是怎么产生的？

他的回答非常精炼：

复杂性的两大根源
· 依赖（Dependency）：某段代码无法被孤立地理解或修改——理解/修改它，必须同时关注其他代码。
· 晦涩（Obscurity）：重要的信息不明显——读者需要付出额外精力才能找到或理解它。

这两者并不是独立的。晦涩往往掩盖了依赖的存在，让依赖变成"隐性依赖"，危害翻倍。

根源一：依赖（Dependency）

什么是依赖？

依赖本身是不可避免的——一个方法调用另一个方法，这是正常的依赖。Ousterhout 真正指的是不必要的依赖，或者说被迫传播的依赖：当你修改 A 时，B、C、D 都要跟着改。

依赖直接导致了"变更放大"症状：本来只需改一处，却牵连多处。

反例：全局状态形成的隐性依赖

反例 — 全局分页大小（Java）bad
// 全局状态：页面大小被多处直接读取
public class Config {
    public static int PAGE_SIZE = 20;
}

public class UserService {
    public List<User> listUsers(int page) {
        int offset = page * Config.PAGE_SIZE;  // 直接依赖全局
        return db.query("SELECT * FROM users LIMIT ? OFFSET ?",
                         Config.PAGE_SIZE, offset);
    }
}

public class OrderService {
    public List<Order> listOrders(int page) {
        int offset = page * Config.PAGE_SIZE;  // 同样依赖全局
        return db.query("SELECT * FROM orders LIMIT ? OFFSET ?",
                         Config.PAGE_SIZE, offset);
    }
}

问题所在：现在产品说"订单列表每页改成 50 条，用户列表保持 20 条"。你发现 Config.PAGE_SIZE 是全局的，改一处就会影响所有调用方。这就是依赖造成的变更放大——被迫去检查、修改所有散落的调用点。

正例 — 将分页大小封装进各自的上下文good
// 每个 Service 自己持有分页大小，互不干扰
public class UserService {
    private static final int PAGE_SIZE = 20;

    public List<User> listUsers(int page) {
        int offset = page * PAGE_SIZE;
        return db.query("SELECT * FROM users LIMIT ? OFFSET ?",
                         PAGE_SIZE, offset);
    }
}

public class OrderService {
    private static final int PAGE_SIZE = 50;  // 独立调整，零影响

    public List<Order> listOrders(int page) {
        int offset = page * PAGE_SIZE;
        return db.query("SELECT * FROM orders LIMIT ? OFFSET ?",
                         PAGE_SIZE, offset);
    }
}
为什么更好？ 每个类只依赖自己内部的常量，修改 OrderService 的分页大小不会对 UserService 产生任何影响。依赖被局部化了。

另一种高危依赖：接口签名的连锁变更

方法参数是显式依赖，但一旦被大量调用方依赖，改签名就是大手术。

反例 — 参数爆炸，调用方强耦合bad
// 随着需求增长，不断往参数列表塞入新字段
public Page<User> searchUsers(
    String name,
    String email,
    Integer status,
    LocalDate startDate,
    LocalDate endDate,
    int page,
    int size
) { ... }

// 调用方被迫传 7 个参数，大多数场景只用到 2-3 个
searchUsers("张三", null, null, null, null, 0, 20);
正例 — 查询对象（Query Object）封装依赖good
// 用查询对象封装所有可选条件，接口签名稳定
public class UserQuery {
    private String name;
    private String email;
    private Integer status;
    private LocalDate startDate;
    private LocalDate endDate;
    private int page = 0;
    private int size = 20;
    // getters / setters / builder 省略
}

// 签名只有一个参数，未来扩展 UserQuery 不需要改调用方
public Page<User> searchUsers(UserQuery query) { ... }

// 调用方只填自己关心的字段
searchUsers(new UserQuery().setName("张三"));

根源二：晦涩（Obscurity）

什么是晦涩？

晦涩是指重要信息不在它该在的地方。读者需要付出额外的侦探工作才能搞清楚：这个变量是什么意思？这个参数合法范围是什么？这个副作用在哪里触发？

晦涩直接导致"认知负荷"和"未知的未知"两个症状——读者不知道自己不知道什么。

反例一：魔法数字与无意义命名

反例 — 数字和名字都没有说明意图bad
public String process(String s, int n) {
    if (s == null || s.length() > 255) {
        throw new IllegalArgumentException("bad");
    }
    if (n == 1) {
        return s.toUpperCase();
    } else if (n == 2) {
        return s.toLowerCase();
    }
    return s;
}

问题所在：s、n、255、1、2 全是谜。读者必须通过猜测或追溯调用处才能理解意图。这就是晦涩——信息就在代码里，但它拒绝把自己说清楚。

正例 — 让意图在代码里自我说明good
public enum CaseMode { UPPER, LOWER, AS_IS }

private static final int MAX_USERNAME_LENGTH = 255;

public String normalizeUsername(String username, CaseMode mode) {
    if (username == null || username.length() > MAX_USERNAME_LENGTH) {
        throw new IllegalArgumentException(
            "用户名不能为空且长度不超过 " + MAX_USERNAME_LENGTH);
    }
    return switch (mode) {
        case UPPER -> username.toUpperCase();
        case LOWER -> username.toLowerCase();
        case AS_IS  -> username;
    };
}

反例二：隐藏的副作用（晦涩掩盖依赖）

这是最危险的晦涩形式——读者以为在调用一个无害的查询，实际上它改了数据库状态。

反例 — 查询方法里藏着写操作bad
// 方法名暗示是"获取"，实际上暗中修改了 lastLoginTime
public User getUser(Long userId) {
    User user = userRepo.findById(userId).orElseThrow();
    // 调用方完全不知道这里有写操作！
    user.setLastLoginTime(LocalDateTime.now());
    userRepo.save(user);
    return user;
}

问题所在：当另一个开发者在单元测试或只读场景下调用 getUser()，会触发意外的数据库写入。这是典型的晦涩掩盖依赖——调用方依赖了一个副作用，却完全不知情。

正例 — 职责分离，副作用显式化good
// 查询就是查询，不做任何修改
public User getUser(Long userId) {
    return userRepo.findById(userId).orElseThrow();
}

// 需要更新登录时间时，显式调用专门的方法
public void recordLogin(Long userId) {
    User user = getUser(userId);
    user.setLastLoginTime(LocalDateTime.now());
    userRepo.save(user);
}
CQS 原则（命令查询分离）：查询方法只返回数据，不改变状态；命令方法只改变状态，不返回业务数据。这个原则直接消灭了"晦涩掩盖依赖"这一类复杂性。

依赖与晦涩的关系图

复杂性的根源
├── 依赖（Dependency）
│ ├── 显式依赖：接口签名、构造函数参数 ← 可接受，易管理
│ └── 隐性依赖：全局状态、隐藏副作用 ← 高危，难以追踪
│
└── 晦涩（Obscurity）
├── 命名晦涩：魔法数字、无意义变量名
├── 结构晦涩：职责混杂、意图不明
└── 行为晦涩：隐藏副作用 ← 同时加剧依赖

注意看：隐藏副作用同时属于晦涩和依赖——晦涩让依赖变成了隐性依赖，这就是两者相互叠加、让复杂性指数级增长的原因。

实际开发中的落地技巧

场景 操作 消除的复杂性根源
全局可变状态 改为构造器注入或方法参数传递 隐性依赖
魔法数字 / 布尔标志参数 提取为具名常量或枚举 命名晦涩
查询方法含写操作 遵循 CQS，拆分为两个方法 行为晦涩 + 隐性依赖
参数列表超过 3 个 封装为 Query Object / DTO 显式依赖过多，接口不稳定
注释解释"这行做什么" 改成让代码自己说明（重命名、拆函数），注释说"为什么" 结构晦涩
跨层传递大对象（如 HttpRequest） 在边界处解构，只向下层传业务字段 层间隐性依赖

容易忽略的细节

时序依赖是最隐蔽的依赖之一。 例如"必须先调用 init() 才能调用 execute()"。如果这个约束没有通过构造器或类型系统强制，仅靠注释说明，就是一个典型的晦涩问题——任何新同事都可能踩坑。解法：用构造器注入确保 init 完成，或通过 Builder 模式在类型层面强制顺序。
boolean 参数是晦涩的重灾区。 sendEmail(user, true) 中的 true 是什么意思？是抄送自己？还是立即发送？阅读这行代码时必须跳转到方法定义才能理解。应改为枚举：sendEmail(user, SendMode.IMMEDIATE)。
Ousterhout 的判断标准： 如果一个模块的行为，光看它的接口（签名 + 注释）就能完整预测，那它就是低晦涩的。如果还需要去看实现细节，晦涩就已经产生了。

小结

  • 依赖让修改变得昂贵——一处变动，多处跟进；根治方法是隔离和局部化依赖。
  • 晦涩让理解变得昂贵——读者需要额外侦探工作；根治方法是让信息就在它该在的地方，并且自我说明。
  • 二者叠加时（晦涩掩盖依赖 = 隐性依赖），复杂性最高、最危险。
  • 每一次对"命名含糊"或"多此一举的全局状态"的妥协，都是在给未来的团队埋雷。复杂性是累积的——下一节会讲这个。
