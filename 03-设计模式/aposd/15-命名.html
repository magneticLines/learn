15. 命名：精确、一致、有信息量

好名字是最廉价的文档——它把意图直接编码进代码，让读者无需翻注释就能建立正确心智模型。

作者的核心主张

John Ousterhout 在《软件设计的哲学》第 14 章专门讨论命名。他的论点可以浓缩为一句话： 名字是一种抽象。好的命名应该让读者在不看实现的情况下，就能在脑海中形成关于这个实体的精确心智模型。

他认为命名的首要目标是精确（precision）——名字必须与实体的真实行为、职责完全匹配，既不能遗漏关键信息，也不能撒谎（名字暗示了实际上并不具备的能力）。

除此之外他还强调两条辅助原则：一致性（同一概念在整个代码库始终用同一个词）和有信息量（名字应当传递读者无法从上下文轻松推断的信息）。

为什么这很重要？
命名是开发者最频繁进行的设计决策。每次读代码时，读者首先看到的是名字，而不是实现。坏名字会让读者建立错误的心智模型，之后每一次错误模型触发的认知修正都是额外的认知负担——这正是 Ousterhout 所说的"认知复杂度"的来源之一。

精确：名字必须与行为完全匹配

反例 → 问题

考虑一个从缓存读取用户数据的方法，它在缓存未命中时会自动查数据库并回填：

Java — 反例：名字撒谎
// 名字暗示"只是从缓存取"，实际上会触发数据库查询 + 写缓存
public User getFromCache(Long userId) {
    User user = cache.get(userId);
    if (user == null) {
        user = db.findById(userId);   // 偷偷查了数据库
        cache.put(userId, user);         // 还写了缓存
    }
    return user;
}

问题：调用方看到 getFromCache 会假设这是一次纯内存操作，不会有网络/磁盘 IO，因此可能在热路径中放心地高频调用。 当缓存未命中时，隐藏的数据库调用会带来意料之外的延迟，更糟糕的是，调用方根本没有意识到这个风险。

正例

Java — 正例：名字描述真实行为
// 名字清晰表达：先查缓存，缓存未命中则查 DB 并回填
public User getUser(Long userId) {
    User user = cache.get(userId);
    if (user == null) {
        user = db.findById(userId);
        cache.put(userId, user);
    }
    return user;
}

// 如果确实需要"只读缓存、不查 DB"的版本，给它一个专属名字
public Optional<User> getCachedUserIfPresent(Long userId) {
    return Optional.ofNullable(cache.get(userId));
}
Ousterhout 的检验标准：能否仅凭名字，在不看实现的情况下，正确预测这个方法在各种边界条件下的行为？如果不能，名字不够精确。

有信息量：不传递无意义信息的名字等于噪音

名字的"信息量"是指：读者从名字中获得的、仅靠上下文无法推断的知识。 冗余信息（类型、作用域等编译器已经表达的东西）不算信息量，反而降低了信号密度。

反例 → 问题

Java — 反例：低信息量命名
public class OrderService {

    // "the"、"data"、"info"、"Manager" 是典型的零信息量词
    private OrderData theOrderData;
    private UserInfo  userInfoData;

    // 变量名 "temp" 传达了什么？什么都没有。
    public BigDecimal calculate(Order o) {
        BigDecimal temp = o.getSubtotal();
        BigDecimal temp2 = temp.multiply(TAX_RATE);
        return temp.add(temp2);
    }
}

读者看到 temp 和 temp2 时，必须手动追踪赋值语句才能理解它们的语义，而这完全可以通过名字直接传达。

正例

Java — 正例：每个名字都传递业务语义
public class OrderService {

    private Order  pendingOrder;   // 说明状态
    private User   buyer;           // 说明角色，不重复类型

    public BigDecimal calculateTotalWithTax(Order order) {
        BigDecimal subtotal = order.getSubtotal();
        BigDecimal taxAmount = subtotal.multiply(TAX_RATE);
        return subtotal.add(taxAmount);
    }
}

一致性：同一概念始终用同一个词

Ousterhout 认为一致性是命名的"第二维度"。它不只关乎单个名字好不好，而是关乎整个代码库是否用统一的词汇表描述相同的概念。 不一致会强迫读者时刻猜测"这个词和那个词是不是同一件事"，造成无谓的认知摩擦。

反例 → 问题

Java — 反例：同一概念用了三个不同词
// UserRepository.java
User fetchById(Long id);

// OrderRepository.java
Order queryByOrderId(Long orderId);

// ProductRepository.java
Product loadProduct(Long productId);

fetch、query、load 在这里表达的是同一个动作——"按 ID 查单条记录"。 新成员加入团队后，会花时间搞清楚这三个词是否有微妙区别，这是不必要的认知成本。

正例

Java — 正例：建立词汇表，全库统一
// 约定：按 ID 查单条记录，统一用 findById（与 Spring Data JPA 保持一致）

// UserRepository.java
User findById(Long id);

// OrderRepository.java
Order findById(Long id);

// ProductRepository.java
Product findById(Long id);
一致性的边界：一致性要求相同的词指代相同的语义，反之亦然——如果两个操作在语义上确实不同（比如 find 返回 null vs get 抛异常），那就该用不同的词，并在整个代码库中严格遵守这个区分。

开发中容易忽略的细节与陷阱

陷阱 1：布尔变量/方法的命名不表达含义

Java — 布尔命名对比
// 反例：flag 传达了什么？
boolean flag = user.check();
if (flag) { ... }

// 正例：名字即文档，读起来像自然语言
boolean isEmailVerified = user.hasVerifiedEmail();
if (isEmailVerified) { ... }

陷阱 2：集合变量只写类型，不写语义

Java — 集合命名对比
// 反例：list 说明了类型，但不知道装的是什么
List<String> list = getItems();

// 正例：说清楚集合里装的是什么
List<String> pendingOrderIds = getPendingOrderIds();

陷阱 3：动词选择不精确（尤其是写操作）

模糊动词为什么模糊精确替换
process 什么叫"处理"？读？写？转换？发消息？ parseAndValidate / transformToDto / publishEvent
handle 同上，万能动词无实际语义 retryOnTimeout / logAndRethrow
manage "管理"覆盖一切，读者无从猜测职责边界 scheduleCleanup / trackActiveSession
update 是 patch 还是 replace？是同步还是异步？ patchUserProfile / replaceShippingAddress

陷阱 4：名字与注释互相矛盾

当方法名是 getUserById，但注释写"返回用户列表"时，读者会优先相信哪个？ 通常相信名字——于是注释被忽略，或者代码被以错误方式调用。名字与注释必须一致，且名字优先级更高。 若两者难以对齐，说明这个方法的抽象本身有问题，应当拆分或重命名。

陷阱 5：重构后忘记同步名字

方法最初只做一件事，名字贴切。后来加了新逻辑，名字却没有更新——这是"名字撒谎"的最常见来源。 每次修改方法行为时，把"更新名字"列入 checklist，与修改测试同等重要。

实际开发中的落地技巧

命名质量自检清单
├── 精确性
│ ├── 仅凭名字，能否准确预测边界行为？
│ ├── 名字有没有"撒谎"（暗示了实际上没有的行为）？
│ └── 名字有没有"沉默"（隐藏了重要的副作用）？
├── 信息量
│ ├── 去掉名字里的冗余词（Data/Info/the/Manager），语义是否完整？
│ ├── 变量名是否说明了"是什么"而非"是什么类型"？
│ └── 布尔名是否以 is/has/can/should 开头？
└── 一致性
├── 同一动作在整个代码库是否用同一个词？
├── 同一领域概念（如 orderId）是否统一拼写？
└── 是否与框架/语言惯例保持一致（如 Spring 的 findBy）？
不要过度设计名字：追求精确不等于名字越长越好。getUserByIdFromDatabaseWithCacheSupport 并不比 getUser 更好——它只是把注释塞进了名字。好名字的标准是：在当前上下文中，传递最多有效信息，同时保持可读性。通常 2-4 个词足够。

在 Code Review 中推行命名规范

  • 评审时提出"我仅凭这个名字，预期它会…，但实现却是…"——这是发现命名问题最自然的方式
  • 团队维护一份"领域词汇表"（Domain Glossary），统一核心概念的英文翻译，例如"优惠券"统一用 coupon 还是 voucher
  • 将命名一致性检查加入 linting（如 ArchUnit 检查命名规范、Checkstyle 检查命名模式）

小结

Ousterhout 把命名视为降低认知复杂度最廉价的手段之一。好名字的三个维度：

  • 精确——名字与实体的真实行为完全匹配，既不撒谎也不沉默
  • 有信息量——传递读者无法从上下文推断的语义，删除冗余词
  • 一致——同一概念全库统一用词，建立并维护领域词汇表

命名是最高频的设计决策，也是最容易被忽视的。将"更新名字"纳入每次代码修改的 checklist，是提升代码库可读性最低成本的工程实践。
