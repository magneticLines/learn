01. 复杂性的本质：三大症状

John Ousterhout 的核心主张：复杂性不是代码行数，而是让修改变得困难、让理解变得痛苦的一切结构特征。

作者的核心主张

《软件设计的哲学》（A Philosophy of Software Design）开篇就给出一个反直觉的定义： 复杂性（Complexity）是任何使软件难以理解或修改的东西。

这个定义的关键在于——它是工程意义上的复杂性，而不是算法或数学意义上的复杂度。 一段只有 20 行的函数，如果别人每次读它都要花 10 分钟才能明白，它就是复杂的； 一个有几千行的框架，如果接口清晰、改动可预测，它就不复杂。

核心公式： C = Σ (cp × tp)
系统整体复杂性 = 各模块复杂性之和，每个模块的贡献 = 其自身复杂度 × 开发者花在它上面的时间比例。

这意味着：不常触碰的复杂模块危害较小；被反复修改的核心模块哪怕有一点复杂性，危害都被放大。

Ousterhout 把复杂性外化为三种可观测的症状，让「复杂」从模糊感受变成可诊断的工程问题。

三大症状总览

软件复杂性的三大症状
├── 症状一：变更放大（Change Amplification）
│ └── 改一处逻辑，要同时修改多处不相关的地方
├── 症状二：认知负担（Cognitive Load）
│ └── 理解一段代码需要记住大量无关上下文
└── 症状三：未知的未知（Unknown Unknowns）
└── 修改时根本不知道还有哪些地方需要一并改动

症状一：变更放大（Change Amplification）

定义：一个看似简单的需求变更，迫使你在代码库中多处进行修改。 每一处额外改动都是潜在的遗漏点和 Bug 来源。

这是复杂性最直观的外在表现。它通常由信息在多个地方重复存放引起——违反 DRY 原则只是其中一种形式， 更深层的原因是模块之间的边界不清晰，导致同一个"概念"散落在多处。

反例：每个页面都硬编码背景色

bad — CSS（样式散落在各组件）
/* PageA.css */
.page-a {
  background-color: #1a1a2e;   /* 手动复制的主色 */
  color: #e2e8f0;
}

/* PageB.css */
.page-b {
  background-color: #1a1a2e;   /* 又一份拷贝 */
  color: #e2e8f0;
}

/* 换主题？需要全局搜索替换 #1a1a2e，漏一处就崩 */
问题：产品要求夜间模式切换主色调，你需要在 30 个文件里逐一替换十六进制值。 改漏一处，页面背景色不一致，而且 Code Review 很难发现。

正例：用 CSS 变量集中定义

good — CSS（单一信息源）
/* tokens.css — 唯一真相来源 */
:root {
  --color-bg:   #1a1a2e;
  --color-text: #e2e8f0;
}

/* PageA.css */
.page-a {
  background-color: var(--color-bg);
  color: var(--color-text);
}

/* PageB.css — 同理，改主题只改 tokens.css 一处 */
.page-b {
  background-color: var(--color-bg);
  color: var(--color-text);
}

同样的道理在 Java 中体现为：别把同一个业务规则（如"订单金额大于 1000 元免运费"） 在 OrderService、CheckoutController、CartSummary 三处各写一遍。 应当封装成一个 ShippingPolicy.isFree(order) 方法，让三处都调用它。

症状二：认知负担（Cognitive Load）

定义：为了完成一个任务，开发者必须在脑子里同时保持大量上下文信息。 认知负担越高，越容易出错，新人上手越慢，Code Review 越难发现问题。

认知负担不等于代码量。一个只有 5 行但充斥隐式全局状态的函数，认知负担可能远超一个 50 行但自洽的函数。 常见成因包括：过深的调用链、函数签名隐藏真实意图、依赖外部可变状态、魔法数字与魔法字符串。

反例：函数依赖隐式外部状态

bad — Java（读者必须追踪 this.discount 从哪来）
public class PriceCalculator {

    private double discount;   // 什么时候被设置？被谁设置？
    private boolean vipMode;  // 又一个隐式开关

    public double calculate(Order order) {
        double base = order.getTotal();
        if (vipMode) base *= 0.9;
        return base * (1 - discount);
        // 调用方看到 calculate(order)，完全猜不到结果还取决于两个字段
    }
}
问题：调用 calculator.calculate(order) 返回了"错误"结果，你不得不往上追溯 discount 和 vipMode 是何时、被谁修改的——隐式状态制造了不必要的认知负担。

正例：把所有输入显式化

good — Java（函数签名即文档）
public class PriceCalculator {

    /**
     * 所有影响结果的因素都在参数里，调用方一眼看清依赖。
     * 函数本身无副作用，单元测试也变得极其简单。
     */
    public double calculate(Order order, double discount, boolean isVip) {
        double base = order.getTotal();
        if (isVip) base *= 0.9;
        return base * (1 - discount);
    }
}

更广义的降低认知负担：使用有意义的命名、把复杂条件提取成布尔变量、 让函数只做一件事（单一职责）、避免全局可变状态。 好的代码不需要读者去"猜"——它把所有需要知道的东西都摆在眼前。

症状三：未知的未知（Unknown Unknowns）

定义：当你做出一个改动时，你甚至不知道还有哪些地方需要一并修改， 或者不知道某个改动会产生什么副作用。

Ousterhout 认为这是三种症状中最危险的一种。 变更放大和认知负担让开发很痛苦，但至少你知道自己不舒服； 未知的未知则会在你以为一切正常时，悄悄埋下 Bug。 它通常由晦涩的接口约定和隐藏的副作用导致。

反例：方法名掩盖了副作用

bad — Java（get 方法偷偷改了状态）
public class UserSession {

    private int accessCount = 0;

    // 调用方以为这只是一个"读"操作
    public User getUser(String id) {
        accessCount++;           // 隐藏的副作用！
        if (accessCount > 100) triggerRateLimit();
        return userRepository.findById(id);
    }
}
问题：某个开发者在循环里调用了 getUser() 做数据聚合， 完全不知道这个"查询"会触发限流逻辑。Bug 出现时， 排查方向首先是业务逻辑，而不是一个命名为 getter 的方法。 这就是「未知的未知」——你不知道你不知道。

正例：副作用显式化，职责分离

good — Java（查询与副作用分离）
public class UserSession {

    private RateLimiter rateLimiter;

    // 纯查询：调用方可以放心地多次调用
    public User getUser(String id) {
        return userRepository.findById(id);
    }

    // 限流检查独立出来，命名明确表达"这会改变状态"
    public void recordAccess(String userId) {
        rateLimiter.increment(userId);
    }
}

这正是 CQS（Command-Query Separation，命令-查询分离）原则的动机所在： 查询方法不应改变系统状态，改变状态的方法不应返回数据。 这样调用方读到方法名，就能知道"这里会不会有我没想到的事发生"。

三大症状对照表

症状 外在表现 根本原因 典型解法
变更放大 改一个需求，要改 N 个文件 信息重复、缺乏单一信息源 DRY、抽取公共模块、配置集中化
认知负担 读懂一段代码需要追踪大量上下文 隐式状态、过深调用链、命名模糊 显式参数、纯函数、有意义命名
未知的未知 改完以为没问题，其实埋了 Bug 隐藏副作用、接口约定不透明 CQS、副作用显式化、文档契约

实际开发落地技巧

如何诊断变更放大

  • 做一次需求变更时，统计实际修改的文件数。如果超过 3 个且原因不是"这个功能本来就分层"，就值得重构。
  • 搜索相同的字面量（魔法字符串、魔法数字），它们往往是变更放大的温床。
  • 在 Java 中：业务规则是否在 Service 和 Controller 各写了一份？常量是否用 public static final 统一管理？

如何诊断认知负担

  • 把代码给一个不熟悉这块业务的同事看 5 分钟，问他"这段代码做什么"——如果他答不上来，认知负担就太高了。
  • 函数参数超过 4 个通常是信号：要么引入 Parameter Object，要么说明这个函数承担了太多职责。
  • 长方法不一定认知负担高，短方法不一定认知负担低——关键是自洽性，而非行数。

如何诊断未知的未知

  • 在做 Code Review 时，主动问自己："这个改动会不会影响到调用方没有意识到的地方？"
  • 副作用一旦存在，必须在方法名或 Javadoc 中明确说明。命名为 getX 的方法不能改变状态。
  • 写单元测试时，如果一个方法很难被独立测试，往往是因为它依赖了隐式外部状态——这是未知的未知的早期预警。
记忆口诀：变更放大 → 信息分散；认知负担 → 上下文隐式；未知的未知 → 副作用隐藏。 三种症状本质上都是"把该放在一起的东西分开了，或者把该显式的东西藏起来了"。

与其他原则的联系

理解了三大症状，很多经典原则就有了更深的动机：

  • DRY（Don't Repeat Yourself）：直接对抗变更放大
  • 单一职责原则（SRP）：减少认知负担，每个类只需要理解一件事
  • CQS（Command-Query Separation）：消灭未知的未知
  • 依赖注入（DI）：把隐式依赖变成显式参数，降低认知负担
  • 深模块（Deep Module）：Ousterhout 的理想——接口简单、实现丰富，对外暴露的复杂性最小
