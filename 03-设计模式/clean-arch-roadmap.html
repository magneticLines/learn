📖 骨架书

《架构整洁之道》(Clean Architecture) — Robert C. Martin（Uncle Bob）

这本书把"软件架构"还原到第一性原理。脉络是： 先从三种编程范式（结构化、面向对象、函数式）讲起——它们各自给程序员"拿走"了一种能力（goto / 函数指针 / 赋值）， 从而约束出更可控的结构；接着用SOLID 五大设计原则规范模块内部的类与依赖； 再用组件的内聚与耦合原则把这些类组织成可独立部署、可演进的组件； 最后揭示架构的本质——划清边界、控制依赖方向、把业务规则（Entity / Use Case）与细节（DB / Web / 框架）彻底分离， 一切汇聚到整洁架构同心圆：依赖只能指向内层的业务核心。

参考书目：Clean Architecture: A Craftsman's Guide to Software Structure and Design · Robert C. Martin 著
前置要求：面向对象基础（多态、接口）· 设计模式中的"依赖倒置 / 面向接口编程"· 理解"为什么要解耦"
  • 从这里开始
    • 全书核心总览核心
      • 一篇读懂全书：终极目标、依赖反转 + 边界、SOLID 速览、整洁架构同心圆

来源：第 3 部分 · 编程范式（Programming Paradigms）

  • 三种范式：每一种都"拿走"一种能力
    • 结构化编程（拿走 goto，约束控制流）P3
      • 可证明性：把程序拆成可推理的小块
    • 面向对象编程（多态是关键）P3
      • 多态让我们安全地反转依赖方向——OO 对架构最大的贡献
    • 函数式编程（不可变性）P3
      • 拿走赋值，消除竞态与并发隐患

来源：第 3 部分 · 设计原则（SOLID）

  • 五大设计原则（类与模块层面）
    • SRP 单一职责原则核心P3
      • 一个模块应只对一个"行为者"负责
    • OCP 开闭原则核心P3
      • 对扩展开放、对修改关闭——靠边界与依赖方向实现
    • LSP 里氏替换原则P3
      • 子类型必须能替换其基类型而不破坏契约
    • ISP 接口隔离原则P3
      • 不要依赖你用不到的东西
    • DIP 依赖倒置原则核心P3
      • 依赖抽象而非具体——整洁架构的发动机

来源：第 4 部分 · 组件构建原则（Component Principles）

  • 组件内聚（哪些类该放进同一个组件）
    • REP 复用/发布等同原则P4
    • CCP 共同闭包原则P4
    • CRP 共同复用原则P4
      • 三者互相拉扯，需要张力平衡
  • 组件耦合（组件之间如何依赖）
    • ADP 无依赖环原则P4
    • SDP 稳定依赖原则P4
    • SAP 稳定抽象原则P4
      • 越稳定的组件越应该抽象，否则难以扩展

来源：第 5 部分 · 软件架构（Architecture）

  • 架构是什么 · 边界 · 业务规则
    • 架构是什么（保留选项、推迟决策）核心P5
    • 划清边界（Boundaries）核心P5
      • 用边界把高层策略和低层细节隔开
    • 业务规则（Entity · Use Case）P5
      • 企业级规则 Entity + 应用级规则 Use Case
    • 整洁架构同心圆核心P5
      • Entities → Use Cases → Adapters → Frameworks，依赖只能向内
    • 尖叫架构（Screaming Architecture）P5
      • 目录结构应"喊出"业务用途，而非框架名
    • 谦卑对象模式（Humble Object）P5
      • 把难测的部分剥离到边界，让核心可测
    • 依赖方向与控制流分离P5
      • 控制流向外，源码依赖向内——靠 DIP 反转

来源：第 6 部分 · 实现细节（Details）

  • 一切皆细节：可被替换、应推迟决策
    • 数据库是细节P6
      • DB 只是存储 I/O，不应渗透进业务规则
    • Web 是细节P6
      • Web 只是一种 I/O 设备/交付机制
    • 框架是细节P6
      • 不要嫁给框架——把它隔在边界之外
