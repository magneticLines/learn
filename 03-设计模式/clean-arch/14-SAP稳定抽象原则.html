14. SAP — 稳定抽象原则

一个组件越稳定，就必须越抽象；越不稳定，就可以越具体。稳定性与抽象性应当同步变化。

原则是什么

稳定抽象原则（SAP，Stable Abstractions Principle） 由 Robert C. Martin 在 《架构整洁之道》"组件耦合"章节提出，其核心主张只有一句话：

一个组件的抽象程度应当与它的稳定程度一致。
换言之：稳定的组件应当是抽象的，不稳定的组件应当是具体的。

SAP 是对上一节 SDP（稳定依赖原则）的延伸与补完。SDP 规定"依赖应该指向稳定的方向"； SAP 则进一步规定"稳定的组件不能被具体实现所占满"，否则整个系统就会变得僵硬—— 你依赖的那个最稳定的组件，一旦需求变化，任何人都改不动它。

为什么要这么设计

问题的根源：稳定 ≠ 不可扩展

日常开发中，我们把"不该轻易改动的组件"做成了稳定组件。 但如果该组件全是具体类，没有任何抽象接口，就会出现矛盾：

  • 其他组件依赖它（SDP 满足）
  • 它本身无法被扩展，只能被修改（OCP 违背）
  • 一旦需求变化，修改代价极高，整个系统冻结

解决方法就是：让稳定组件只暴露抽象（接口、抽象类、协议）， 把"具体怎么做"的细节下沉到不稳定的外层组件里。 这样，稳定组件永远不需要改，扩展需求由新的具体组件承载。

与开闭原则的关系

SAP 本质上是在组件层面落地 OCP（开闭原则）。 OCP 说"对扩展开放、对修改关闭"，SAP 给出了实现路径： 把开放点（抽象）放在稳定组件里，把扩展实现放在可以随时添加或替换的不稳定组件里。

量化：A-I 坐标系

Uncle Bob 定义了两个可测量指标：

指标定义值域
不稳定性 I 扇出依赖 / (扇入依赖 + 扇出依赖)，即"有多少组件依赖它 vs 它依赖了多少组件" 0（极稳定）→ 1（极不稳定）
抽象性 A 组件内抽象类/接口数量 / 组件内总类数量 0（全具体）→ 1（全抽象）

SAP 的理想状态是 A ≈ 1 - I，即两者之和接近 1，落在"主序列"直线上。

A（抽象性） 1 │· 理想区（主序列） │ · │ · │ · 0 └────────── I（不稳定性） 0 1 ▲ 左上角：极稳定 + 全抽象 → 理想（如核心接口层） ▼ 右下角：极不稳定 + 全具体 → 理想（如具体实现层） ⚠ 左下角：极稳定 + 全具体 → "死区" / Zone of Pain ⚠ 右上角：极不稳定 + 全抽象 → "无用区" / Zone of Uselessness

代码示例：反例 → 问题 → 正例

场景：支付系统中的核心组件

我们有一个被多处依赖的 payment-core 组件，它非常稳定（I ≈ 0）。

反例（违反 SAP）

payment-core 里直接是具体的支付实现，A ≈ 0，落入"死区"：

payment-core / WechatPayService.java反例 ❌
// payment-core 组件 —— 被 order、refund、report 三个组件依赖，I = 0（极稳定）
// 问题：里面全是具体类，A = 0，落入 Zone of Pain

public class WechatPayService {

    public String pay(String orderId, BigDecimal amount) {
        // 直接调用微信 SDK，写死了具体实现
        WxPayConfig cfg = new WxPayConfig();
        cfg.setAppId("wx_app_id");
        return WxPayApi.unifiedOrder(cfg, orderId, amount);
    }

    public boolean refund(String transactionId, BigDecimal amount) {
        // 同样写死微信退款逻辑
        return WxPayApi.refund(transactionId, amount);
    }
}
问题暴露：现在需要接入支付宝。
payment-core 极稳定，所有调用方都依赖它，谁都不想它变化； 但它全是具体代码，不改它就无法扩展。 结果：要么在这个"死区"组件里堆 if-else，要么逼着所有依赖方一起改——两种选择都很痛苦。

正例（遵循 SAP）

把抽象留在稳定组件，把具体实现移到不稳定组件：

payment-core / PaymentGateway.java稳定组件只放抽象 ✅
// payment-core 组件：I = 0（极稳定），A = 1（全抽象）→ 落在主序列左上角

/**
 * 支付网关抽象 —— 这是整个支付体系的"稳定核心"。
 * 任何支付方式都必须实现此接口，上层业务只依赖这个抽象。
 */
public interface PaymentGateway {

    /** 发起支付，返回支付流水号 */
    String pay(String orderId, BigDecimal amount);

    /** 退款 */
    boolean refund(String transactionId, BigDecimal amount);

    /** 查询支付状态 */
    PayStatus queryStatus(String transactionId);
}

// payment-core 里同样可以有抽象类，提供部分默认实现（模板方法）
public abstract class AbstractPaymentGateway implements PaymentGateway {

    // 公共逻辑：参数校验、日志埋点——稳定且抽象，不含任何渠道细节
    protected void validate(String orderId, BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0)
            throw new IllegalArgumentException("金额必须为正数");
    }
}
payment-wechat / WechatPayGateway.java不稳定组件放具体实现 ✅
// payment-wechat 组件：I = 1（极不稳定，只被 payment-core 抽象约束），A = 0（全具体）
// → 落在主序列右下角，随时可以替换或新增

@Service
public class WechatPayGateway extends AbstractPaymentGateway {

    private final WxPayConfig config;

    @Override
    public String pay(String orderId, BigDecimal amount) {
        validate(orderId, amount);            // 复用稳定组件的公共校验
        return WxPayApi.unifiedOrder(config, orderId, amount);
    }

    @Override
    public boolean refund(String transactionId, BigDecimal amount) {
        return WxPayApi.refund(transactionId, amount);
    }

    @Override
    public PayStatus queryStatus(String transactionId) {
        WxPayOrderQuery result = WxPayApi.query(transactionId);
        return toPayStatus(result.getTradeState());
    }
}
payment-alipay / AlipayGateway.java新增渠道无需动稳定组件 ✅
// 新增支付宝：新建 payment-alipay 组件即可，payment-core 纹丝不动

@Service
public class AlipayGateway extends AbstractPaymentGateway {

    @Override
    public String pay(String orderId, BigDecimal amount) {
        validate(orderId, amount);
        return AlipayClient.tradePay(orderId, amount);
    }
    // ... 其余实现省略
}

依赖方向总结如下：

order-service ──────────────────────────────┐ refund-service ─────────────────────────────┤──▶ payment-core（接口/抽象类，I=0 A=1） report-service ─────────────────────────────┘ ▲ ▲ │ │ payment-wechat payment-alipay （具体实现，I=1 A=0）

实际落地技巧

技巧 1：用包/模块边界隔离稳定层

在 Maven 多模块项目中，把核心接口单独拆成一个 -api 模块（如 payment-core-api）， 该模块只包含接口、枚举、DTO。具体实现放在 payment-wechat、payment-alipay 等模块。 上层业务只 compile 依赖 -api，实现模块在运行时通过 Spring 注入。

实践检验：如果你能在不改动 -api 模块任何代码的情况下，新增一种支付渠道， 说明你的 SAP 落得不错。

技巧 2：用工具量化 A 和 I

可以借助 JDepend（Java）或 Dependency Cruiser（前端）生成各组件的 A（抽象性）和 I（不稳定性）报告，观察哪些组件偏离主序列， 作为代码评审或架构治理的依据。

技巧 3：识别并治理"死区"组件

"死区"（Zone of Pain）的典型症状：

  • 该组件被大量其他组件依赖（说明稳定），但里面几乎没有接口
  • 每次需求变化都要改这个组件，且改动会引发大规模回归测试
  • 团队成员对"动这个组件"有本能的恐惧

治理方式：将稳定组件中的具体实现逐步提取接口， 再通过依赖注入将实现挪到外层不稳定组件。不必一次完成，按业务变化热点优先排序即可。

技巧 4：警惕"无用区"

"无用区"（Zone of Uselessness）是另一个极端：组件全是抽象，但没有任何其他组件依赖它（I = 1）。 这类组件通常是过度设计的遗留物——有人预先抽象了一套接口，但实际上只有一个实现， 也没有扩展计划。应当合并或删除，避免维护负担。

常见误区 & 面试考点

误区正确理解
把所有类都做成接口就是遵循 SAP SAP 是在组件粒度讨论，要看整个组件的 A 值，而不是每个类都加接口
SAP 和 SDP 是两条独立原则 两者是配套的：SDP 控制依赖方向，SAP 控制被依赖组件的形态。两者共同构成"稳定的核心必须是抽象的"
工具类（Utils）应当抽象化 工具类通常 I 接近 1（没人依赖它，它依赖很多），应是具体的，无需强行抽象
数据库实体类应当放在稳定组件 实体类是具体的，若它同时被大量依赖，会陷入死区。应通过 Repository 接口隔离，接口放稳定层
面试高频：SAP 与 OCP 的关系？
SAP 是 OCP 在组件层面的体现。OCP 要求"对扩展开放、对修改关闭"， SAP 给出了具体实现路径——把开放点（抽象）放进稳定组件，把扩展实现放进不稳定组件， 从而让稳定组件永远不需要修改，新功能通过增加新的不稳定组件来扩展。

与其他原则的联系

SAP 位于组件耦合原则体系的顶层，向下连接多个基础原则：

SAP（稳定抽象原则） ├── 依赖 SDP（稳定依赖原则）：依赖需要指向稳定方向，SAP 规定稳定的才能被依赖 ├── 依赖 OCP（开闭原则）：SAP 是 OCP 的组件级实现 ├── 依赖 DIP（依赖倒置原则）：高层不依赖低层，都依赖抽象——SAP 保证那个"抽象"在稳定组件 └── 关联 CCP（共同封闭原则）：变化原因相同的类放在一起，稳定组件的抽象不该因具体细节变化

一句话记住 SAP

越稳定，越要抽象。
让"被所有人依赖的核心"只提供接口和抽象类，把"会变化的具体实现"放到可以随时替换的外层—— 这样系统既有稳定的骨架，又有灵活扩展的血肉。
