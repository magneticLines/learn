13. SDP — 稳定依赖原则

Stable Dependencies Principle：依赖应该指向更稳定的方向。

核心主张：依赖箭头要指向稳定

Robert C. Martin 的原话是：「依赖应当朝着稳定的方向发展」。 换句话说，当组件 A 依赖组件 B 时，B 必须比 A 更稳定——否则 B 的每一次变动都会 强迫 A 跟着变动，形成"不稳定的地基承重"。

这里的稳定不是"不经常修改"，而是指不容易被迫修改。 衡量标准是：有多少组件依赖它（入向依赖 Fan-in）， 以及它自己依赖了多少组件（出向依赖 Fan-out）。 Fan-in 越高、Fan-out 越低，组件就越稳定。

稳定性公式
I = Fan-out / (Fan-in + Fan-out)
I ∈ [0, 1]。I = 0 表示极度稳定（无依赖出口，被大量依赖）；I = 1 表示极度不稳定（无人依赖它，但它自己依赖很多人）。

SDP 要求：依赖关系图中，箭头指向的组件的 I 值必须小于（或等于）指向方的 I 值。 也就是说，箭头要沿着 I 值从大到小的方向流动。

为什么这样设计代码更好

想象一下，你把一栋楼的承重柱换成了豆腐渣——上层建筑再坚固也白搭。 如果"频繁变更的组件"被大量其他组件依赖，那么每次改动都会引发连锁重编译、 重测试、甚至接口破坏，整个团队陷入"改一处、坏一片"的泥潭。

  • 稳定组件被依赖：变更成本集中在少数几个地方，影响面可控。
  • 不稳定组件不被依赖：它可以自由演化，不拖累其他人。
  • 方向正确才能解耦：依赖方向反过来，就无法独立部署、独立测试。

建立直觉：用组件依赖图来感受稳定性

// I 值示意（越小越稳定） Flexible（I = 1.0，极不稳定） ↓ 依赖 Stable（I = 0.5，中等稳定） ↓ 依赖 Core（I = 0.0，极稳定） // 箭头从高 I 流向低 I —— 这符合 SDP ✓
// 违反 SDP 的反例 Stable（I = 0.1） ↓ 依赖 Volatile（I = 0.9，频繁变更） // 稳定组件依赖了不稳定组件 —— 违反 SDP ✗

反例：稳定组件依赖了不稳定组件

以下场景常见于前期设计粗糙的项目：核心的 OrderService 直接依赖了 一个还在频繁迭代的 PromotionEngine。

反例 · JavaOrderService.java
// ❌ 反例：稳定的订单核心依赖了极不稳定的促销引擎

package com.example.order;

// 促销引擎由运营团队驱动，几乎每周都在改规则
import com.example.promotion.PromotionEngine;

public class OrderService {             // Fan-in = 20，Fan-out = 1 → I ≈ 0.05

    private final PromotionEngine promotionEngine;  // Fan-in = 1，Fan-out = 5 → I ≈ 0.83

    public OrderService(PromotionEngine engine) {
        this.promotionEngine = engine;
    }

    public Order createOrder(Cart cart) {
        // 直接调用不稳定的促销引擎计算折扣
        BigDecimal discount = promotionEngine.calculateDiscount(cart);
        return new Order(cart, discount);
    }
}
问题所在
OrderService 的 I ≈ 0.05（极稳定），PromotionEngine 的 I ≈ 0.83（极不稳定）。 依赖箭头从低 I 指向高 I，违反 SDP。每次促销规则变更都会触发 OrderService 的重编译和回归测试，而 OrderService 本身什么都没变。

正例：引入稳定抽象打断不稳定依赖

解法是在两者之间插入一个稳定的接口（抽象层），让 OrderService 只依赖接口， PromotionEngine 去实现接口。依赖关系的方向因此被"反转"。

正例 · Java（依赖反转）DiscountPolicy.java
// ✅ 第一步：在稳定组件中定义抽象接口
// 接口属于 order 包，由 OrderService 团队负责，极少变动

package com.example.order;

public interface DiscountPolicy {              // Fan-in = 2，Fan-out = 0 → I = 0
    BigDecimal calculateDiscount(Cart cart);
}
正例 · JavaOrderService.java
package com.example.order;

public class OrderService {

    private final DiscountPolicy discountPolicy;  // 只依赖稳定的接口

    public OrderService(DiscountPolicy policy) {
        this.discountPolicy = policy;
    }

    public Order createOrder(Cart cart) {
        BigDecimal discount = discountPolicy.calculateDiscount(cart);
        return new Order(cart, discount);
    }
}
正例 · JavaPromotionEngine.java
// ✅ 不稳定的实现类依赖稳定的接口（方向正确）
package com.example.promotion;

import com.example.order.DiscountPolicy;   // promotion → order（高I → 低I ✓）

public class PromotionEngine implements DiscountPolicy {

    @Override
    public BigDecimal calculateDiscount(Cart cart) {
        // 促销规则随意演化，完全不影响 OrderService
        return cart.totalAmount().multiply(new BigDecimal("0.1"));
    }
}
修复后的依赖方向
PromotionEngine（I≈0.83）→ DiscountPolicy（I=0）← OrderService（I≈0.05）
两个组件都朝着更稳定的 DiscountPolicy 接口依赖，完全符合 SDP。 促销规则随便改，OrderService 无需重编译。

稳定性数值对比

组件 Fan-in Fan-out I 值 稳定性 适合被依赖？
DiscountPolicy（接口） 2 0 0.00 极稳定 ✓ 是
OrderService 20 1 0.05 稳定 ✓ 是
PromotionEngine（修复后） 1 1 0.50 中等 △ 谨慎
PromotionEngine（修复前，被直接依赖） 1 5 0.83 不稳定 ✗ 否

实际开发中的落地技巧

1. 用包/模块边界强制稳定性层次

在 Java 项目中，把接口（契约）单独放在一个 api 或 domain 模块里， 实现放在 infra 或 adapter 模块里。Maven/Gradle 的 module 依赖关系 天然地强制了 SDP：infra 可以依赖 domain，反过来编译直接报错。

project ├── domain (I ≈ 0，核心接口 + 实体，不依赖任何外部框架) │ └── DiscountPolicy.java ├── application (I ≈ 0.2，用例，只依赖 domain) │ └── OrderService.java └── infrastructure (I ≈ 0.9，实现细节，依赖 domain + 第三方库) └── PromotionEngine.java

2. 用 ArchUnit 在 CI 中自动检查依赖方向

ArchUnit 是一个 Java 测试库，可以在单元测试中断言"infrastructure 层不能被 domain 层依赖"。 一旦有人不小心在 domain 里 import 了 infrastructure 的类，CI 立刻失败。

ArchUnit 示例 · JavaArchitectureTest.java
@AnalyzeClasses(packages = "com.example")
class ArchitectureTest {

    @ArchTest
    static final ArchRule domainShouldNotDependOnInfra =
        noClasses()
            .that().resideInAPackage("..domain..")
            .should().dependOnClassesThat()
            .resideInAPackage("..infrastructure..");
}

3. 识别"隐式不稳定"：不只看修改频率

一个组件可能目前很少改动，但如果它的 Fan-out 极高（依赖了大量外部库）， 它的潜在不稳定性也很高——外部库升级、API 变更都可能传导进来。 所以稳定性要同时看 Fan-out 数量和所依赖对象的稳定性。

4. Spring Boot 项目的常见坑

  • 在 @Service（应该稳定）里直接注入 RestTemplate 或 RedisTemplate（基础设施细节），会使业务层的 Fan-out 升高，I 值增大。正确做法是在业务层定义接口，在 infra 层实现并注入。
  • Entity 类混入 DTO 逻辑（或反之），导致两个稳定性不同的组件耦合在一起，任何一方变动都拖累另一方。
  • 在核心领域对象里 import 了 Jackson、Lombok 以外的框架注解，实际上是在稳定层引入了对框架（不稳定外部依赖）的耦合。

SDP 与 DIP 的关系

SDP 告诉你依赖应该指向哪个方向； DIP（依赖倒置原则）告诉你如何在代码层面实现这个方向反转。 两者是一对搭档：当你发现某个依赖违反了 SDP（稳定的东西依赖了不稳定的东西）， DIP 就是修复它的工具——引入接口，让实现去依赖抽象，而不是反过来。

记忆口诀
SDP = 定方向（箭头指向稳定）
DIP = 给工具（用抽象扭转箭头）
SAP = 做保障（稳定的组件应该是抽象的，否则"稳定"会变成枷锁）

常见误区 & 面试高频考点

  • 误区：稳定 = 不经常改。 稳定性是结构性指标（Fan-in/Fan-out），不是历史频率。一个从没改过的组件， 如果 Fan-out 极高，同样是潜在的不稳定源。
  • 误区：只要加了接口就符合 SDP。 接口定义在不稳定模块里，稳定模块去实现它，依赖方向还是错的。 接口必须归属于更稳定的一侧。
  • 面试考点：被问到"为什么要依赖抽象而不是具体实现"时， 不要只说"方便替换"，更深层的理由是：具体实现往往比抽象更不稳定（I 更高）， 依赖它会违反 SDP，导致稳定组件被不稳定组件拖着走。
  • SDP 与独立部署：微服务架构中，若服务 A 依赖服务 B 的某个频繁变更的 API， 每次 B 发版都需要 A 联调验证，这正是 SDP 被违反的体现。 解法是引入稳定的契约（如 OpenAPI Schema 版本化），让 B 的实现自由演化。
