03. 函数式编程（不可变性）

赋值语句是并发 Bug 的温床——函数式编程用"禁止赋值"来从根上消灭竞态条件。

作者的核心主张

Bob 大叔在《架构整洁之道》第四章指出：函数式编程的核心约束只有一条——变量不可变（Immutability）。 其他一切特性（纯函数、高阶函数、惰性求值）都是这条约束自然衍生出来的结果，而不是目的本身。

为什么"禁止赋值"如此重要？因为软件中几乎所有的并发问题，根源都是多个线程/进程同时对同一块可变状态进行读写。 竞态条件（Race Condition）、死锁（Deadlock）、更新丢失（Lost Update）——它们的先决条件都是"某个变量的值会在某个时刻被改写"。 如果变量从创建之日起就永不改变，这些问题便无从发生。

范式的本质是"约束"而非"能力"
结构化编程限制了 goto；面向对象限制了函数指针（用多态封装）；函数式编程限制了赋值语句。 每一种范式都是主动放弃某种"自由"，换取可预测性与可维护性。

不可变性解决了什么问题

可变状态 → 竞态条件（反例）

下面是一个经典的银行转账场景。两个线程同时给同一账户加钱：

Java — 可变状态导致数据丢失反例 ✗
public class BankAccount {
    private int balance = 100;   // 可变字段——危险根源

    public void deposit(int amount) {
        int current = balance;          // Thread-A 读到 100
        // ← Thread-B 也读到 100，然后写入 150
        balance = current + amount;  // Thread-A 写入 150，B 的写入丢失！
    }
}

// 期望：存两次 50，balance = 200
// 实际：两次写入互相覆盖，balance = 150

问题所在

问题不在于"多线程"，而在于 balance 这个字段可以被赋值。 只要一个值能被改写，就必须用锁来保护，而锁本身又带来死锁、优先级反转等新问题。 这是一个无底洞。

不可变值对象（正例）

Java — 不可变 Value Object正例 ✓
public final class Money {
    private final int amount;   // final：创建后永不改变

    public Money(int amount) {
        this.amount = amount;
    }

    // 不修改 this，而是返回新对象——"转换"而非"赋值"
    public Money add(Money other) {
        return new Money(this.amount + other.amount);
    }

    public int getAmount() { return amount; }
}

// 使用：无论多少线程同时调用 add()，Money 对象本身永不改变
Money a = new Money(100);
Money b = a.add(new Money(50));   // a 不变，b 是全新对象 150
Money c = a.add(new Money(50));   // 同样得到 150，天然线程安全

纯函数：不可变性在函数层面的体现

不可变性不只体现在对象字段上，还体现在函数不产生副作用（Side Effect）这一要求上。 一个纯函数（Pure Function）满足两个条件：

  • 相同输入永远产生相同输出（引用透明性）
  • 不修改任何外部状态（无副作用）
Java — 有副作用 vs 纯函数对比
// ——— 反例：依赖外部可变状态 ———
private int taxRate = 13;   // 外部可变字段

public double calcTax(double price) {
    return price * taxRate / 100;  // taxRate 随时可能被别处修改
}
// 问题：同样传入 price=100，今天返回 13，明天可能返回 9
// 测试极难：必须先设置好 taxRate 的状态

// ——— 正例：纯函数，所有依赖都通过参数传入 ———
public static double calcTax(double price, int taxRate) {
    return price * taxRate / 100;
}
// 好处：可单独测试，不依赖任何全局状态，可安全并行调用

不可变性 ≠ 完全没有状态变化

这是初学者最常见的误解。现实世界中状态变化是客观存在的，函数式编程并不否认这一点。 它的处理策略是：把"可变"隔离到系统边界，内部核心逻辑全部用不可变数据流表达。

系统边界（允许可变）
├── 数据库写入　　← I/O 副作用，集中在最外层
├── HTTP 请求/响应　← 同上
└── 内部核心逻辑（禁止可变）
　　 ├── 领域计算：纯函数，输入 → 输出
　　 ├── 数据转换：旧集合 → 新集合（map/filter/reduce）
　　 └── 业务规则：无副作用，易测试

Bob 大叔在书中提出了一种实践方案：把组件分为可变组件和不可变组件两类， 可变组件（处理状态的部分）越小越好，不可变组件（纯计算的部分）越大越好。 这正是"六边形架构"/"端口适配器架构"的底层逻辑。

Java 中的不可变实践

Record（Java 16+）：天然不可变的值对象

Java 16+ — Record 替代可变 DTO推荐写法
// 一行声明，编译器自动生成 final 字段、构造器、equals/hashCode/toString
public record OrderItem(String productId, int quantity, Money price) {}

// "修改"操作 → 返回新 record，原对象不变
OrderItem item    = new OrderItem("P001", 2, new Money(50));
OrderItem updated = new OrderItem(item.productId(), 3, item.price());

Stream API：不可变的集合处理管道

Java — Stream vs 命令式循环对比
List<Order> orders = getOrders();

// ——— 反例：命令式，累积可变状态 ———
List<String> result = new ArrayList<>();
for (Order o : orders) {
    if (o.getAmount() > 1000) {
        result.add(o.getId());   // 每次循环都在修改 result
    }
}

// ——— 正例：函数式管道，每步都产生新的数据流 ———
List<String> result = orders.stream()
    .filter(o -> o.getAmount() > 1000)   // 不修改 orders，产生新流
    .map(Order::getId)                    // 转换，不修改原对象
    .collect(Collectors.toList());       // 最终物化为新列表
// orders 原封不动；可并行：只需换 .parallelStream()

事件溯源：不可变性在架构层面的极致应用

Bob 大叔在书中举了一个深刻的例子：银行账户的余额，其实不是一个"值"，而是所有交易记录"相加"的结果。 如果我们不存储当前余额（可变状态），而是存储每一笔交易事件（不可变事件流）， 那么任何时刻的余额都可以通过重放事件推算出来，系统永远不需要"更新"任何记录，只需"追加"。

概念示意 — 事件溯源（Event Sourcing）架构思想
// 传统做法：存储可变状态
// UPDATE account SET balance = 150 WHERE id = 1;  ← 历史丢失

// 事件溯源：只追加不可变事件
List<AccountEvent> events = List.of(
    new Deposited(100),    // 永久保留，绝不修改
    new Deposited(50),
    new Withdrawn(30)
);

// 当前余额 = 重放所有事件的聚合结果（纯函数）
int balance = events.stream()
    .reduce(0, (acc, e) -> e.apply(acc), Integer::sum);
// balance = 120，且可以回溯任意时刻的历史状态
这就是 Bob 大叔的结论：如果存储空间和处理器速度足够快，应用程序完全可以没有可变状态—— 所有状态都通过重放不可变事件来推算。现实中的妥协是定期做"快照"（Snapshot）， 但核心思想不变：用追加替代更新，用转换替代赋值。

落地技巧与常见陷阱

场景 常见陷阱 推荐做法
Java 集合 返回 ArrayList，调用方直接修改 返回 List.copyOf() 或 Collections.unmodifiableList()
Builder 模式 build() 后继续修改 builder，影响已构建对象 build() 内部做防御性拷贝；或用 Record 替代
Lambda 捕获外部变量 在 lambda 内修改外部 int[] 数组绕过 effectively-final 检查 用 AtomicInteger 明确语义；或改用 reduce
Date / Calendar Date 是可变的，传递后被修改 全面改用 java.time（LocalDate 等均不可变）
缓存优化 担心不可变对象创建开销大 小对象 JVM 逃逸分析可栈上分配；必要时才用对象池
最容易忽略的细节：浅不可变（Shallow Immutability）
字段声明了 final 不代表对象真正不可变。如果字段是一个 List， 外部代码仍然可以调用 list.add() 修改其内容。 真正的不可变需要深层防御：在构造器中做防御性拷贝，并返回不可修改的视图。
Java — 浅不可变陷阱与修复细节
// ——— 陷阱：字段 final 但集合内容可变 ———
public final class Order {
    private final List<OrderItem> items;

    public Order(List<OrderItem> items) {
        this.items = items;   // 直接持有外部引用，外部可以修改！
    }
    public List<OrderItem> getItems() { return items; }  // 暴露可变引用
}

// ——— 修复：防御性拷贝 + 不可修改视图 ———
public final class Order {
    private final List<OrderItem> items;

    public Order(List<OrderItem> items) {
        this.items = List.copyOf(items);   // 拷贝入参，切断外部引用
    }
    public List<OrderItem> getItems() {
        return items;   // List.copyOf() 返回的本身就是不可修改的
    }
}

与其他范式的联系

三种范式共同构成了软件工程的基础约束，彼此互补而非竞争：

结构化编程　　限制 goto　　　→ 控制流可推理（顺序、分支、循环）
面向对象编程　限制 函数指针　→ 依赖关系可反转（多态 + 接口）
函数式编程　　限制 赋值语句　→ 状态变化可追踪（不可变 + 纯函数）

在实际架构中，这三种范式通常混合使用： 用结构化编程写流程，用面向对象管理模块依赖，用函数式处理领域内的数据转换逻辑。 Spring 的 @Service 层持有状态（可变边界），而其内部的计算方法尽量写成纯函数—— 这正是"可变组件最小化，不可变逻辑最大化"原则的体现。

面试高频考点

  • 函数式编程的本质约束是什么？——不是 Lambda，不是高阶函数，而是变量不可变。
  • 不可变对象为什么是线程安全的？——因为没有写操作，不存在写写/读写冲突，不需要加锁。
  • String 为什么设计成不可变？——字符串常量池复用、HashCode 缓存、线程安全——都依赖不可变性。
  • Java 的 final 字段能保证不可变吗？——不能，只能保证引用不变，还需防御性拷贝和不可修改视图才能达到深层不可变。
  • 函数式和面向对象能共存吗？——完全可以，现代 Java/Kotlin/Scala 都是多范式语言，关键是把副作用推到系统边界。
