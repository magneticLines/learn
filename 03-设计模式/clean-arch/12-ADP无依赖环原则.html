12. ADP — 无依赖环原则

Acyclic Dependencies Principle：组件依赖图中不能出现环。一旦有环，牵一发而动全身，构建与发布都会陷入泥潭。

核心主张：让依赖图保持有向无环（DAG）

Bob 大叔在《架构整洁之道》第 14 章提出 ADP（Acyclic Dependencies Principle）： 组件之间的依赖关系图中不允许存在环。

"组件"在 Java 中可以理解为一个 Maven/Gradle 模块、一个 JAR 包，或者一个独立部署单元。 ADP 要求这些组件之间形成一张有向无环图（DAG），每条边只能从"使用者"指向"被使用者"，绝对不允许出现 A → B → C → A 这样的环路。

为什么叫"原则"而不是"规范"？
原则意味着有其背后的工程动机。ADP 的动机是：独立开发、独立发布、独立构建。 有了它，团队才能拆分职责，组件才能单独升级，测试才能隔离运行。

没有环 vs 有环：两个世界的对比

没有环时——DAG 的世界

依赖关系清晰，构建顺序唯一确定，发布互不影响。

Main └──▶ Application ├──▶ Domain │ └──▶ Common └──▶ Infrastructure └──▶ Common

Common 不依赖任何其他模块，它可以独立发版。 Domain 只依赖 Common，升级 Common 后只需重新构建 Domain 及其上游，不会波及 Infrastructure。

有环时——牵一发而动全身

UserService ──▶ OrderService ──▶ UserService （循环！）

此时你无法单独构建 UserService，因为它需要 OrderService，而 OrderService 又需要 UserService。Maven 构建时会报"循环依赖"错误；即使绕过，任何一方改动都必须同时重新构建对方，独立发版彻底失去意义。

真实的痛点：有环的系统，一个程序员改了 OrderService 里的一个枚举， 另一个只在写 UserService 的程序员发现自己的代码突然编译不过了——他甚至不知道发生了什么。 Bob 把这种现象叫做"晨后综合征"（The Morning After Syndrome）。

反例 → 问题 → 正例

反例：两个模块互相依赖

假设电商系统中 user-service 需要查询用户的订单数，order-service 需要校验用户是否存在， 两位开发者各自"图方便"直接调用对方的接口类，于是出现了循环依赖。

UserService.java（user-service 模块）反例
// user-service 模块
public class UserService {
    // 直接依赖 order-service 模块里的类 ← 危险！
    private final OrderService orderService;

    public int getOrderCount(Long userId) {
        return orderService.countByUser(userId);
    }
}
OrderService.java（order-service 模块）反例
// order-service 模块
public class OrderService {
    // 又反过来依赖 user-service 模块里的类 ← 形成环！
    private final UserService userService;

    public void placeOrder(Long userId) {
        if (!userService.exists(userId)) throw new IllegalArgumentException("用户不存在");
        // ...
    }
}

问题清单：

  • Maven 构建报 cycle in dependency，CI 流水线直接挂掉。
  • 无法单独升级 user-service，每次必须同时发两个模块。
  • 单元测试需要同时 mock 对方，测试依赖链无限膨胀。
  • 两个团队每天互相等待对方接口稳定，开发节奏被绑死。

正例方案 A：抽取共享接口到第三个模块

将双方共同依赖的抽象提取到独立模块 user-api， 两个服务都只依赖这个纯接口模块，依赖环彻底消失。

user-service ──▶ user-api order-service ──▶ user-api （user-service 与 order-service 之间不再有任何依赖）
UserQueryPort.java（user-api 模块）正例 A
// user-api 模块：只放接口，不放实现
public interface UserQueryPort {
    boolean exists(Long userId);
    int     getOrderCount(Long userId);
}
OrderService.java（order-service 模块）正例 A
// order-service 只依赖 user-api，不依赖 user-service 实现
public class OrderService {
    private final UserQueryPort userQuery;  // 依赖倒置

    public void placeOrder(Long userId) {
        if (!userQuery.exists(userId))
            throw new IllegalArgumentException("用户不存在");
        // ...
    }
}
UserService.java（user-service 模块）正例 A
// user-service 实现接口，同时也依赖 user-api
@Service
public class UserServiceImpl implements UserQueryPort {

    @Override
    public boolean exists(Long userId) {
        // 查 DB ...
        return true;
    }

    @Override
    public int getOrderCount(Long userId) {
        // 注意：这里改用事件或 RPC 获取，不再直接调 OrderService
        return orderCountCache.get(userId);
    }
}
这正是 DIP（依赖倒置原则）与 ADP 协同发力的地方。 抽取接口模块既打破了环，又让高层策略（OrderService）不依赖低层细节（UserServiceImpl）。

正例方案 B：用事件解耦（异步场景）

当两个组件的通信天然是异步的（如"用户注册后统计订单"）， 可以改用领域事件代替直接调用，彻底消除编译期依赖。

UserRegisteredEvent.java（shared-events 模块）正例 B
// 事件对象放入独立的 shared-events 模块，任何人都可以订阅
public record UserRegisteredEvent(Long userId, String email) {}
OrderModule 订阅事件（order-service 模块）正例 B
@EventListener
public void onUserRegistered(UserRegisteredEvent event) {
    // 初始化该用户的订单统计，不需要调用 UserService
    orderStatsRepo.initForUser(event.userId());
}

打破依赖环的两种通用手法

手法 思路 适用场景 代价
抽取共享模块 把双方都需要的抽象（接口/DTO/事件定义）提到第三个模块 同步调用、强类型契约 多一个模块，需维护版本
依赖倒置（DIP） 让高层定义接口，低层实现，依赖方向与调用方向相反 想保留调用语义但消除编译期依赖 增加接口层，代码量微增

两种手法常常结合使用：DIP 倒置了方向，抽取模块解决了物理隔离问题。

如何检测依赖环

方式一：Maven Enforcer 插件（推荐，零成本）

pom.xml配置
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-enforcer-plugin</artifactId>
  <executions>
    <execution>
      <goals><goal>enforce</goal></goals>
      <configuration>
        <rules>
          <!-- 禁止依赖树中出现同一 artifact 的不同版本 -->
          <banDuplicatePomDependencyVersions/>
          <!-- 禁止循环依赖 -->
          <noCyclicDependencies/>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>

方式二：ArchUnit 在测试中断言（细粒度到包级别）

ArchitectureTest.java单元测试
@AnalyzeClasses(packages = "com.example")
class ArchitectureTest {

    @ArchTest
    static final ArchRule noCycles =
        slices().matching("com.example.(*)..")
               .should().beFreeOfCycles();
    // 任何包间循环都会让 CI 红掉，把规则写进代码比文档靠谱得多
}

实际开发中的落地技巧

  • 模块命名即分层：用 -api、-domain、-infra、-app 后缀约定层次， 一眼看出谁能依赖谁，反向依赖自然显眼。
  • 新模块加入前先画 DAG：在 Confluence/白板画出依赖图，确认没有新环再提 MR，比事后修复成本低 10 倍。
  • 把 ArchUnit 或 Enforcer 加进 CI 门禁：人工 review 会漏，工具不会。
  • 警惕"工具类模块"变成垃圾桶：common、utils 模块最容易悄悄反向依赖业务模块， 定期用 mvn dependency:tree 检查。
  • 同一模块内的类循环引用不违反 ADP：ADP 的粒度是组件（模块/包），不是单个类。 但类级别的循环往往是提取模块的信号。
容易踩的坑：Spring Boot 的 @Autowired 可以在运行时注入循环 bean（Spring 6 以前）， 让开发者以为"没有循环依赖"，但编译期的 Maven 模块依赖环依然存在。 不要用运行时注入掩盖模块结构问题。

与其他原则的联系

原则 与 ADP 的关系
DIP（依赖倒置） ADP 的打环手术刀。接口倒置依赖方向，是消除环最常用的外科手术
SDP（稳定依赖） ADP 解决了"有无环"，SDP 进一步规定"依赖方向应朝向更稳定的组件"
CCP（共同闭包） 把一起变化的类聚合进同一组件，减少跨组件依赖，从源头降低产生环的概率
ISP（接口隔离） 细粒度接口让抽取共享模块更轻量，减少共享模块膨胀
一句话记住 ADP：把组件依赖画成图，如果图里有环，就用"抽接口"或"倒置依赖"把环剪断。 让 CI 来守门，不要靠人工记忆。
