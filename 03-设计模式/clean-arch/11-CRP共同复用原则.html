11. CRP 共同复用原则

Common Reuse Principle — 不要强迫组件的用户依赖他们不需要的东西

原则定义

CRP（Common Reuse Principle，共同复用原则）： 一个组件中的所有类应当被一起复用。如果你复用了组件中的一个类，就等于复用了组件中的所有类。

换句话说：不要把不总是被一起复用的类放进同一个组件。 每当你对组件做出任何改动，所有依赖这个组件的系统都必须重新验证、重新编译、重新部署——即便那次改动根本和它们无关。

作者 Robert C. Martin 的核心主张： 组件应该是"复用的原子单元"。如果用户只需要组件里的一个类，他们也必须承担整个组件的升级、测试、部署成本。所以，只有真正被一起使用的类，才应该被放到同一个组件里。

为什么这样设计更好

在 Java/Maven 世界里，"组件"就是一个 JAR 包。你的项目引入了 utils-1.0.jar， 里面有 StringHelper、DateHelper、ExcelExporter 三个类。 你只用了 StringHelper，但 ExcelExporter 升级了（依赖 POI 从 4.x 升到 5.x）。 你必须跟着升级这个 JAR，必须重新测试，必须重新部署。 你的代码完全没变，你付出了毫无意义的代价。

CRP 要求我们在设计组件边界时问一个问题：这些类是因为"会被一起用"才放在一起的，还是仅仅因为"功能上有点像"才凑在一起的？ 只有前者才合理。

CRP 是组件层面的 ISP

CRP 是接口隔离原则（ISP）在组件层面的推广：

层面原则核心要求
类/接口ISP 接口隔离原则不要强迫类依赖它不使用的方法
组件/包CRP 共同复用原则不要强迫组件依赖它不使用的类

同一逻辑，不同粒度。ISP 控制接口的"宽度"，CRP 控制组件的"内聚度"。 违反任何一个，都会制造无谓的耦合和不必要的重编译。

反例 → 问题 → 正例

反例：把不相关的类塞进同一个组件

假设我们在构建一个内部工具库，初版图省事，把所有"工具类"丢进同一个 Maven 模块 common-utils：

common-utils/ ├── StringHelper.java <-- 字符串处理，几乎所有模块都用 ├── DateHelper.java <-- 日期格式化，几乎所有模块都用 ├── ExcelExporter.java <-- 生成 Excel，只有报表模块用 ├── PdfRenderer.java <-- 生成 PDF，只有报表模块用 └── SmsNotifier.java <-- 发短信，只有通知模块用（依赖阿里云 SDK）
反例 — common-utils/pom.xml（依赖臃肿）
<dependencies>
    <!-- StringHelper / DateHelper 根本用不到下面这些 -->
    <dependency>
        <groupId>org.apache.poi</groupId>
        <artifactId>poi-ooxml</artifactId>  <!-- 为 ExcelExporter -->
    </dependency>
    <dependency>
        <groupId>com.aliyun</groupId>
        <artifactId>aliyun-java-sdk-dysmsapi</artifactId>  <!-- 为 SmsNotifier -->
    </dependency>
</dependencies>
反例 — 用户模块只想用 StringHelper，却被迫引入整个 common-utils
// user-service/pom.xml 里引入了 common-utils
// → 传递依赖进来了 poi-ooxml（15 MB）和 aliyun-sdk
// → 阿里云 SDK 升级 API，common-utils 1.1.0 发版
// → user-service 啥都没改，也必须升级、重新测试、重新部署

public class UserService {
    public String normalize(String name) {
        return StringHelper.trim(name);  // 只用了这一个类
    }
}
问题所在： user-service 对 SmsNotifier、ExcelExporter 毫无兴趣， 但只要它们所依赖的第三方库有任何版本变动，user-service 就无法置身事外。 这是典型的"强迫用户依赖他们不需要的东西"。

正例：按真实复用关系拆分组件

问清楚：谁和谁总是被一起用？把那些类放在一起，其余的独立成包。

common-core/ <-- 被所有模块复用 ├── StringHelper.java └── DateHelper.java report-utils/ <-- 只被报表模块复用 ├── ExcelExporter.java └── PdfRenderer.java notify-utils/ <-- 只被通知模块复用 └── SmsNotifier.java
正例 — user-service/pom.xml（只依赖真正需要的）
<dependencies>
    <!-- 只引入 common-core，不再捆绑 POI 和阿里云 SDK -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>common-core</artifactId>
    </dependency>
</dependencies>

<!-- report-service/pom.xml -->
<dependencies>
    <dependency><artifactId>common-core</artifactId></dependency>
    <dependency><artifactId>report-utils</artifactId></dependency>
</dependencies>
效果： report-utils 升级 POI 版本，只有真正使用报表功能的模块需要跟进。 user-service、order-service 等无关模块完全不受影响， CI/CD 流水线不会被无辜触发。

CRP 在三大内聚原则中的位置

《架构整洁之道》提出了组件内聚的三条原则：REP（复用/发布等同原则）、CCP（共同闭包原则）、CRP（共同复用原则）。 它们之间存在内在张力，不可能同时最大化满足。

原则倾向代价
REP 把相关类打包在一起，方便整体发布复用 组件可能偏大
CCP 把因同一原因变化的类放一起，变更时改动面最小 组件数量少但更大
CRP 把不总是一起用的类拆开，减少无谓依赖 组件数量增多，管理成本上升

CRP 是一个"排斥力"：它告诉你哪些类不应该放在一起。 CCP 是一个"吸引力"：它告诉你哪些类应该放在一起。 架构师的工作就是在这两种力之间找到适合当前团队和项目阶段的平衡点。

实践中的经验法则： 项目初期，团队小、模块少，优先满足 CCP（减少跨组件变更的摩擦）； 随着系统成熟、外部使用者增多，逐步引入 CRP（减少无谓的传递性依赖和重编译）。

落地技巧与容易忽略的细节

1. 用"一起修改"和"一起复用"两把尺子审查组件

在 code review 中，每当有人向一个已有组件新增类时，问两个问题：

  • 这个新类和组件中的现有类，是否总会被同一个消费者引入？
  • 如果这个新类发生变更，组件的其他消费者是否会受到无辜牵连？

如果两个问题的答案都是"否"，这个类应该放到别的组件或新建一个组件。

2. 传递性依赖是 CRP 违反的放大器

Maven/Gradle 的传递性依赖会让 CRP 违反的影响成倍扩大。 一个"顺手加进来"的类引入了一个 20 MB 的第三方库， 所有间接依赖你组件的模块都会默默把这 20 MB 带进自己的 classpath。 定期执行 mvn dependency:analyze 检查"declared but unused"和"used but undeclared"依赖。

命令行 — 分析依赖使用情况
# 找出声明了但没用到的依赖（CRP 违反的直接证据）
mvn dependency:analyze

# 可视化依赖树，定位传递性依赖来源
mvn dependency:tree -Dincludes=org.apache.poi:poi-ooxml

3. 前端/Node 世界的对应实践

npm 包的设计同理。例如 lodash 早年是一个大包， 后来拆分成 lodash.get、lodash.merge 等单函数包， 正是 CRP 思想的体现——你只用 get，就不必为 merge、throttle 的变更买单。 现代 bundler 的 tree-shaking 是对"无法拆包"情况下的补救措施，但 tree-shaking 解决不了运行时依赖和版本锁定问题。

4. 不要把"技术层"当作组件边界

常见错误：把 util、helper、common 当成一个组件， 里面塞满了技术上类似（"都是工具类"）但业务上毫无关联的东西。 CRP 要求你以复用关系而非技术分类来划分边界。

高频陷阱： 一个叫 common 的模块是 CRP 违反的重灾区。 每次"懒得新建模块"就往里塞，最终它成为一个没有人能说清边界的"大垃圾桶"， 所有人都依赖它，任何改动都牵一发动全身。

5. 微服务和组件边界

在微服务架构里，一个服务本身就是一个部署单元（组件）。 CRP 的警示在这里体现为：不要把不相关的业务能力塞进同一个服务只是因为它们"都是后台服务"。 服务越臃肿，每次发版的影响面越大，独立部署的优势也就越小。

一句话记忆

CRP = 不用的东西，别强迫我依赖。 把真正总是被一起复用的类放到一个组件里，把其他的分开——这样每一次升级、每一次部署，都只影响真正需要关心的人。
