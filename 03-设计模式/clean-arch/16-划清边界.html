16. 划清边界（Boundaries）

架构师最核心的职责之一：把"需要频繁改变的东西"与"不该被它拖累的东西"隔离开来。

核心主张：边界是用来隔离变化的

Bob 大叔在书中给出了一个鲜明的论断： 软件架构的本质工作，就是在系统元素之间画线——这些线就叫"边界（Boundary）"。 边界存在的意义只有一个：让某类决策的变化无法穿透到另一侧。

他紧接着指出，最常见的架构错误是过早耦合细节——把数据库、框架、UI、第三方 SDK 直接焊进业务逻辑，导致"换个数据库"或"升级一个框架"就要动到核心业务代码。 这些细节本该处于边界的外侧，对内侧的业务规则保持不可见。

一句话总结：边界的两侧，依赖箭头只能从"易变"指向"稳定"，绝不能反过来。 业务规则是系统中最稳定的部分；数据库、UI、框架是最易变的部分。

为什么：细节污染业务规则的代价

想象一个在线订单系统，所有逻辑都揉在一起：

反例 — 业务逻辑直接耦合数据库与框架（Java）
// ❌ 反例：Service 层直接依赖 JPA Entity 和 Spring Web 细节
@Service
public class OrderService {

    @Autowired
    private OrderRepository repo;   // 直接依赖 Spring Data JPA

    @Autowired
    private HttpServletRequest request; // 直接依赖 Servlet API！

    public void placeOrder(Long userId, List<Long> itemIds) {
        // 从 HTTP Session 中取折扣——业务逻辑与 Web 层混在一起
        String coupon = request.getSession().getAttribute("coupon").toString();

        // JPA Entity 直接在业务方法里被组装并持久化
        OrderEntity order = new OrderEntity();
        order.setUserId(userId);
        order.setItems(itemIds);
        order.setCoupon(coupon);
        repo.save(order);  // 框架细节
    }
}

问题在哪里？

  • 换成消息队列触发（而不是 HTTP）？无法复用，因为依赖了 HttpServletRequest。
  • 换成 MongoDB？OrderEntity 是 JPA 注解类，需要大改。
  • 单元测试？必须启动 Spring 容器，测一个"计算折扣"的逻辑却要 mock Servlet。
  • 折扣规则变了？改动会顺手碰到数据库映射代码，引入意外 regression。
根本原因：没有边界，所有"变化频率不同"的代码紧密耦合在一起。 改任何一件事都可能牵动所有其他事——这正是 Bob 大叔说的"架构失败"。

正例：用边界把业务规则与细节分离

划清边界的手段很简单：引入接口（抽象），让依赖箭头反转。业务层只依赖抽象，细节实现依赖业务层定义的抽象。

src/ ├── domain/ ← 业务规则，无任何框架依赖 │ ├── Order.java ← 纯 POJO，核心实体 │ └── OrderRepository.java ← 接口（边界契约） ├── usecase/ │ └── PlaceOrderUseCase.java ← 业务流程，只依赖 domain 接口 ├── adapter/ │ ├── web/ │ │ └── OrderController.java ← 依赖 usecase，不反向 │ └── persistence/ │ └── JpaOrderRepository.java ← 实现 domain.OrderRepository └── config/ └── AppConfig.java ← 组装所有依赖（DI 配置）
domain/Order.java — 纯业务对象，零框架依赖
package domain;

// 纯 POJO，不带任何 @Entity / @Table 等框架注解
// 业务规则就在这里，换框架时这个类完全不动
public class Order {
    private final Long       userId;
    private final List<Long> itemIds;
    private final String     coupon;

    public Order(Long userId, List<Long> itemIds, String coupon) {
        // 边界内只做业务校验，不碰 IO
        if (itemIds.isEmpty())
            throw new IllegalArgumentException("订单至少含一件商品");
        this.userId  = userId;
        this.itemIds = List.copyOf(itemIds);
        this.coupon  = coupon;
    }
    // getters …
}
domain/OrderRepository.java — 边界契约（接口）
package domain;

// 这个接口由业务层定义，意思是：
// "我需要一个能存订单的东西，但我不关心它用什么技术实现"
public interface OrderRepository {
    void save(Order order);
}
usecase/PlaceOrderUseCase.java — 业务流程，只依赖 domain
package usecase;

// ✅ 构造函数注入：边界的依赖方向明确，可在单元测试中直接用 mock 替换
public class PlaceOrderUseCase {

    private final OrderRepository repo;  // 依赖的是接口，不是 JPA

    public PlaceOrderUseCase(OrderRepository repo) {
        this.repo = repo;
    }

    // coupon 由调用方（Controller/消息消费者）传入，use case 本身不知道来源
    public void execute(Long userId, List<Long> itemIds, String coupon) {
        Order order = new Order(userId, itemIds, coupon);
        repo.save(order);
    }
}
adapter/persistence/JpaOrderRepository.java — 细节实现，依赖业务接口
package adapter.persistence;

// JPA 注解在这里，污染范围仅限于 adapter 层
@Repository
public class JpaOrderRepository implements OrderRepository {

    @PersistenceContext
    private EntityManager em;

    @Override
    public void save(Order order) {
        // 把业务对象映射为 JPA Entity，变化只在这里
        OrderEntity entity = OrderMapper.toEntity(order);
        em.persist(entity);
    }
}
依赖箭头检查：
JpaOrderRepository → OrderRepository（接口） ✓
PlaceOrderUseCase → OrderRepository（接口） ✓
domain 层 → 任何框架 ✗（不存在）✓
所有箭头都从外（易变）指向内（稳定）。

边界的三种落地形态

Bob 大叔在书中把边界按"隔离强度"分为三档，成本从低到高：

形态 隔离机制 适用场景 成本
源码级边界 同一进程内，通过接口/包隔离（如上例） 单体应用内部模块解耦 低
进程级边界 独立进程，通过 IPC / 本地 Socket 通信 同机器上需隔离的服务 中
服务级边界 独立部署单元，通过网络（HTTP/gRPC/MQ）通信 微服务、分布式系统 高
常见误区：很多团队一上来就做微服务（最高成本的边界），却在单个服务内部毫无边界意识——Service 直接操作 Entity，Controller 里写 SQL。这是"宏观过度隔离，微观完全耦合"，两头都没做好。

边界的核心规则：依赖倒置（DIP）

划清边界最关键的机制是依赖倒置原则（Dependency Inversion Principle）： 高层策略（业务规则）不应依赖低层细节（数据库、框架），两者都应依赖抽象。 抽象由业务层定义，细节层去实现它。

（错误方向） 业务层 ──────────────────→ 数据库层 直接 import JPA （正确方向：边界 + DIP） ┌─────────────────────┐ 业务层 ──→ 接口（边界） ←── 数据库层实现 └─────────────────────┘ 业务层定义接口， 数据库层实现接口 依赖箭头两者都指向接口

这是为什么 Spring 的 @Autowired 按接口注入是正确用法， 而直接 new JpaXxxRepository() 是错误用法——后者绕过了边界， 把具体实现固化在了业务代码里。

实际开发：何时、在哪里画边界

不是每行代码都需要接口。Bob 大叔给出的判断标准是：

  • 变化频率不同：UI 每周变，数据库偶尔变，业务规则几乎不变——变化频率差异越大，越需要边界。
  • 部署独立性：如果两个组件需要独立部署或独立测试，就需要边界。
  • 团队独立性：如果两个团队分别维护两侧代码，边界是减少相互干扰的保险。
  • 第三方 SDK：任何第三方库（支付 SDK、邮件服务）都应被边界包裹，防止它的版本升级污染业务代码。

落地技巧：Humble Object 模式

边界两侧如果直接通信，测试就难写。Bob 大叔推荐使用 Humble Object 模式：把难以测试的部分（如数据库 IO、网络请求） 封装成一个极简（humble）的对象，业务逻辑不进入这个对象，只进入旁边可轻松测试的对象里。

Humble Object 示意 — 把 IO 细节隔离到最薄的壳里
// ✅ Humble Object：只做 IO，不含任何业务判断，薄得几乎不需要测试
public class JpaOrderRepository implements OrderRepository {
    public void save(Order o) {
        em.persist(OrderMapper.toEntity(o));  // 唯一职责：持久化
    }
}

// ✅ 业务逻辑在这里，完全不依赖 JPA，单元测试直接 new InMemoryOrderRepository()
public class PlaceOrderUseCase {
    public void execute(Long userId, List<Long> itemIds, String coupon) {
        // 业务规则在这里，可以充分测试
        repo.save(new Order(userId, itemIds, coupon));
    }
}

容易忽略的细节 & 面试高频考点

1. 边界两侧的数据结构要独立

很多人画了接口，却让业务层和数据库层共用同一个 OrderEntity 类—— 这等于边界形同虚设。跨边界传递的数据结构必须是简单的 DTO 或业务对象， 不能是带框架注解的 Entity。

2. 循环依赖是边界被打破的信号

如果 domain 包 import 了 adapter 包的类，就出现了循环依赖。 Maven/Gradle 的模块化 module 划分可以在编译期强制阻止这类错误—— 这是比代码 Review 更可靠的边界保证。

3. Spring 的 @Transactional 打在哪里

很多项目把 @Transactional 打在 Service 上，而 Service 里耦合了 JPA Entity—— 这已经是边界污染。正确做法：事务边界打在 adapter 层或 use case 层的入口， 业务对象 Order 本身不感知事务存在。

4. 不要为了边界而边界

Bob 大叔自己也说："过早划边界和过晚划边界一样有害"。 对于一个两人团队的小项目，强行引入六边形架构会带来大量冗余代码。 边界的成本是真实存在的，要等到变化信号出现（需求频繁改动某个方向）再补边界—— 因为好的边界是可以事后插入的（通过接口提取）。

面试常考："什么是依赖倒置？"标准答案不是背定义，而是能画出依赖箭头图， 说清楚"接口由谁定义、由谁实现、为什么要这样"。能结合 Spring 的 @Repository 接口化用法举例，加分。

与其他知识点的联系

  • 关联 15 节"架构是什么"：上一节讲"保留选项"是架构的目标，本节讲"边界"是实现这个目标的具体机制——边界是选项得以保留的物理保证。二者是目标与手段的关系，本节的依赖倒置示意图正是上一节抽象讨论的代码级落地。
  • 关联 DIP（依赖倒置原则）：本节反复强调"边界的核心规则就是 DIP"——业务层定义接口，细节层实现接口，依赖箭头从两侧都指向接口。这是全书最直接点名 DIP 与边界关系的一节，也解释了为什么 SOLID 里唯独 DIP 会被单独拎出来作为架构层面的核心手段。
  • 关联 SDP（稳定依赖原则）："依赖箭头只能从易变指向稳定"这句话，本节用在业务规则与框架/数据库之间，13 节 SDP 用在组件的 Fan-in/Fan-out 之间——两者是同一条规则在不同粒度上的重复出现：SDP 关心组件级的依赖方向，本节关心架构层（业务 vs 细节）的依赖方向。
  • 关联 GoF 适配器模式与仓储模式：本节 JpaOrderRepository implements OrderRepository 的写法，本质是适配器模式在持久化层的应用——把 JPA 的具体调用方式"适配"成业务层定义的接口形状。Repository 模式则是这种边界思想在数据访问层的专门命名，两者经常被同时使用。
  • 关联 Humble Object 模式与测试金字塔：本节提出的 Humble Object 模式，目的是把"难测试的 IO"和"该被充分测试的业务逻辑"分离，这与测试驱动开发中"单元测试聚焦纯逻辑、集成测试覆盖 IO 边界"的分层策略完全对应——边界画得好，单元测试才能不依赖 Spring 容器、不依赖真实数据库。
  • 关联微服务的服务级边界：本节"边界的三种形态"表格把源码级边界扩展到服务级边界，说明本节技术（接口隔离）和微服务拆分是同一套思想在不同部署成本下的应用——先在单体内用源码级边界验证清楚模块边界是否合理，再决定是否值得付出服务级边界的运维成本去拆分，这也是很多团队"先模块化单体、后微服务"的路径依据。

小结

  • 边界的本质：用抽象（接口）隔离变化，让依赖箭头从易变指向稳定。
  • 业务规则是系统最稳定的核心，必须受到边界保护，不能直接依赖框架、数据库、UI。
  • 边界有三种强度（源码级 / 进程级 / 服务级），按实际隔离需求和成本选择，不要过度设计。
  • Humble Object 模式帮助把"难以测试的 IO 细节"从"需要充分测试的业务逻辑"中剥离出来。
  • 实践中用 Maven 多模块或 Java 9 模块系统在编译期强制边界，比靠自律更可靠。
