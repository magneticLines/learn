04. SRP — 单一职责原则

一个模块应该只对一个行为者负责。不是"只做一件事"，而是"只服务一个主人"。

原则原文与核心主张

Robert C. Martin（Uncle Bob）在《架构整洁之道》里给出了 SRP 的精确表述：

A module should be responsible to one, and only one, actor.
一个模块应该只对一个行为者（Actor）负责。

注意关键词是 Actor（行为者），而不是"只做一件事"。很多人第一次接触 SRP 时，理解成"一个类只能有一个方法"或"一个函数只做一件事"，这是常见的误读。Uncle Bob 在书中专门纠正了这一点：

  • "只做一件事"是函数粒度的规则（第 2 章 Clean Code 中提到的），不是类/模块层面的 SRP。
  • SRP 关注的是变化的来源——谁会要求这段代码改变？如果有两个不同的业务部门都能驱动同一个类的变化，这个类就违反了 SRP。

用一句话概括作者的核心主张：把因相同原因而变化的东西聚合在一起，把因不同原因而变化的东西分离开来。

为什么这样设计代码更好

违反 SRP 的根本危害是意外耦合（Accidental Coupling）：两个完全不相干的业务需求，因为共享了同一个类，导致其中一方的改动意外破坏了另一方。

可以用一个生活类比来建立直觉：

想象一个"万能员工"，既负责财务报销又负责 HR 考勤，还兼顾 IT 运维。财务部要求他改变报销单格式时，他不得不修改同一张"表格模板"，但这张模板也被 HR 系统引用了——于是考勤报告莫名其妙出了问题。

这就是多个 Actor 共享同一模块的代价。

在代码层面，这会带来三类具体问题：

  • 合并冲突增多：不同团队修改同一个文件，Git 冲突频繁。
  • 测试范围扩大：改了财务逻辑，却要重新回归测试 HR 相关功能。
  • 理解成本上升：一个类承载多套业务规则，读代码时必须在脑子里切换上下文。

反例 → 问题 → 正例

场景：员工薪资系统

这是《架构整洁之道》书中的经典例子。Employee 类同时服务三个部门： 财务部（计算工资）、HR 部（统计工时）、DBA（持久化）。

反例 — 三个 Actor 共用一个类

Java · 反例Employee.java（违反 SRP）
// 问题：财务、HR、DBA 三个部门都能驱动这个类改变
public class Employee {

    private String name;
    private double baseSalary;
    private int    workedHours;

    // Actor 1：财务部 — 计算薪资
    public double calculatePay() {
        return baseSalary + regularHours() * 1.5;
    }

    // Actor 2：HR 部 — 上报工时（用于考勤统计）
    public void reportHours() {
        HRSystem.submit(name, regularHours());
    }

    // Actor 3：DBA — 保存到数据库
    public void save() {
        Database.save(this);
    }

    // 私有辅助方法 — 被上面两个 Actor 共享！这就是炸弹
    private int regularHours() {
        return Math.min(workedHours, 8);
    }
}

问题所在

表面上看，regularHours() 是一个无害的私有方法。但它同时被 calculatePay() 和 reportHours() 调用。

某天财务部要求：加班费计算改为"每天超过 8 小时的部分按 2 倍算"，于是开发者修改了 regularHours()。

结果 HR 的考勤报告也悄悄改变了——因为 reportHours() 复用了同一个方法。HR 部门直到月末对账时才发现数据异常，但那时已经无法追溯是哪次提交引入的 bug。

正例 — 按 Actor 拆分，各自独立

Java · 正例按 Actor 拆分后的结构
// 纯数据载体：只持有字段，不含业务逻辑
public class EmployeeData {
    public String name;
    public double baseSalary;
    public int    workedHours;
}

// Actor 1：财务部专属 — 薪资计算规则由财务部驱动
public class PayCalculator {
    public double calculatePay(EmployeeData e) {
        return e.baseSalary + regularHours(e) * 1.5;
    }

    // 财务部自己的"正常工时"定义，财务部改需求不影响 HR
    private int regularHours(EmployeeData e) {
        return Math.min(e.workedHours, 8);
    }
}

// Actor 2：HR 部专属 — 考勤逻辑由 HR 部驱动
public class HourReporter {
    public void reportHours(EmployeeData e) {
        // HR 自己的工时统计口径，与薪资计算完全隔离
        HRSystem.submit(e.name, regularHours(e));
    }

    private int regularHours(EmployeeData e) {
        // 即使 HR 对"正常工时"有不同定义，改这里不影响财务
        return Math.min(e.workedHours, 8);
    }
}

// Actor 3：DBA 专属 — 持久化策略由 DBA 驱动
public class EmployeeRepository {
    public void save(EmployeeData e) {
        Database.save(e);
    }
}

现在财务部修改薪资规则，只需要改 PayCalculator；HR 改考勤口径，只需要改 HourReporter；两者之间再也不会意外影响对方。

拆分后的目录结构

按 Actor 拆分后，包结构也随之清晰：

employee/
├── EmployeeData.java ← 纯数据，无 Actor 归属
├── payroll/
│ └── PayCalculator.java ← 财务部 Actor
├── hr/
│ └── HourReporter.java ← HR 部 Actor
└── persistence/
    └── EmployeeRepository.java ← DBA Actor

每个包对应一个业务部门，包内文件的变化只由该部门的需求驱动。当财务团队和 HR 团队同时提 PR 时，他们修改的是完全不同的文件，Git 冲突概率趋近于零。

用 Facade 保留"统一入口"

拆分后有时调用方会抱怨："以前用 employee.calculatePay() 一行搞定，现在要引入三个类太麻烦了。" Uncle Bob 建议用 Facade 模式提供一个薄薄的统一入口：

Java · FacadeEmployeeFacade.java
// Facade：不含任何业务逻辑，只负责转发调用
public class EmployeeFacade {

    private final PayCalculator      payCalc  = new PayCalculator();
    private final HourReporter        hrReport = new HourReporter();
    private final EmployeeRepository  repo     = new EmployeeRepository();

    public double calculatePay(EmployeeData e) {
        return payCalc.calculatePay(e);
    }

    public void reportHours(EmployeeData e) {
        hrReport.reportHours(e);
    }

    public void save(EmployeeData e) {
        repo.save(e);
    }
}

Facade 保持了对外接口的简洁，同时不破坏内部各 Actor 的隔离。调用方不需要知道内部有几个类，但各个业务类依然能独立演化。

落地技巧与容易踩的坑

技巧一：先识别 Actor，再设计类

开始设计之前，先问："谁（哪个部门/角色）会要求这段逻辑改变？"把答案列出来，答案有几个，职责就有几个，不要强行合并进同一个类。

技巧二：私有方法是 SRP 违规的隐藏地带

如书中例子所示，最危险的往往不是公共方法，而是被多个公共方法共享的私有方法。Code review 时重点检查：一个私有方法被哪些公共方法调用？这些公共方法是否服务于不同的 Actor？

技巧三：SRP 不等于"类越小越好"

过度拆分会引发"散弹枪（Shotgun Surgery）"问题：一个业务需求要同时改动十几个类。SRP 的目标是减少意外耦合，而不是追求类的数量最大化。拆分的依据永远是"变化的来源是否不同"，而非"方法数量是否超过 N 个"。

技巧四：在 Spring 项目中的映射

Spring 的分层架构天然支持 SRP，但层内仍然会违反：

  • @Service 类里混入了 DTO 转换逻辑（应提取到 Converter/Mapper）
  • @Controller 里写了业务判断（应下沉到 Service）
  • 一个 UserService 同时处理用户账户逻辑和消息通知逻辑（两个 Actor：账户管理组 vs 消息组）

技巧五：面试常见陷阱

面试官常问："SRP 说的是一个类只做一件事，对吗？"

标准答案：不完全对。SRP 是说一个模块只对一个 Actor 负责，Actor 是变化的来源（业务角色/部门）。"只做一件事"是函数粒度的建议，不是 SRP 的精确表述。Uncle Bob 在书中特别区分了这两个概念。

对比总结

维度 违反 SRP 遵守 SRP
变化来源 多个 Actor 驱动同一个类 每个类只被一个 Actor 驱动
共享代码 私有方法被不同业务复用，隐性耦合 各 Actor 各自持有私有辅助逻辑
合并冲突 高频，不同团队改同一文件 低频，各团队改各自的文件
测试范围 改一处需要回归多个业务场景 改一处只需回归该 Actor 的场景
可读性 类内混杂多套业务规则 类名即 Actor，意图清晰
