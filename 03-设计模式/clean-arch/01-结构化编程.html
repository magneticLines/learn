《架构整洁之道》 · 三种范式

01. 结构化编程

拿走 goto，约束控制流——第一种范式如何让程序从"不可证伪"变为"可推理"

背景：Dijkstra 的发现

1968 年，Edsger Dijkstra 在一封著名的公开信《Go To Statement Considered Harmful》里 正式向 goto 宣战。他发现：只要代码里存在任意跳转，程序员就无法 对"当前状态"做任何可靠的推断，因为执行路径可以从任何地方飞来飞去， 人脑和工具都难以追踪。

Bob Martin 在《架构整洁之道》里把这件事上升到范式高度： 结构化编程是人类历史上第一次主动对自己的能力加以约束—— 不是因为 goto 做不到某件事，而是因为它让程序无法被推理。

核心主张：任何程序都可以且应该只用三种控制结构来表达—— 顺序（Sequence）、分支（Selection）、 循环（Iteration）。Böhm 和 Jacopini 在 1966 年从数学上证明了这一点， Dijkstra 随后推动它成为编程规范。

goto 的真正问题：不可证伪性

Bob Martin 用了一个关键词：可证伪（falsifiable）。 科学方法要求命题必须能被反例推翻；同理，好的代码也必须能被推理—— 你能一眼断言某段代码"在进入时 x > 0，出来时 result 一定非空"。

goto 破坏了这种能力：一个标签可以被任意多处跳转， 控制流图从"有向无环树"退化成"任意有向图"， 模块的前置条件和后置条件都无从保证。

反例：充斥 goto 的伪 C 代码

C — 反例（goto 乱跳）
int processOrder(Order* order) {
    if (order == NULL) goto error;
    if (order->amount <= 0)  goto error;

    // 业务逻辑散落在多处 goto 之间，很难确定执行到这里时
    // order 的哪些字段已经被修改过
    applyDiscount(order);
    if (inventoryCheck(order) == 0) goto retry;
    goto done;

retry:
    // 问题：retry 也可能从第 3 行外部某处跳过来，
    // applyDiscount 会被执行第二次吗？没人说得清
    replenishStock(order);
    goto done;

error:
    return -1;
done:
    return 0;
}

问题所在：retry 标签可以被任意多处跳入， applyDiscount 是否已经执行过完全取决于是从哪里跳来的。 想写单元测试？你需要模拟每一条入口路径，组合爆炸。 想做静态分析？控制流图变成了一张蜘蛛网。

正例：用三种结构重写

C — 正例（顺序 + 分支 + 循环）
int processOrder(Order* order) {
    // 卫语句：前置条件检查，保证后续代码的上下文干净
    if (order == NULL || order->amount <= 0) return -1;

    // 顺序：每一步职责单一，执行到此处状态完全可预测
    applyDiscount(order);

    // 循环：重试逻辑用 while 明确表达，次数有上界，不会无限跳
    int retries = 3;
    while (!inventoryCheck(order) && retries-- > 0) {
        replenishStock(order);
    }

    // 分支：最终结果由清晰的条件决定，不依赖跳转历史
    return inventoryCheck(order) ? 0 : -2;
}

重写后，函数有清晰的入口前置条件（order 非空且金额合法） 和出口后置条件（返回 0 表示成功，否则为错误码）。 可以独立测试，可以静态分析，可以被人类大脑在几秒内推理完毕。

为什么三种结构就够了

Böhm-Jacopini 定理的结论很简单：任何可计算函数都可以用顺序、分支、循环来表达， 不需要 goto。这不是工程经验，是数学定理。

程序 = 顺序（Sequence）
      ├── 分支（Selection） ← if / switch / 三元运算符
      └── 循环（Iteration） ← while / for / do-while

goto 能做的事 = 这三者的组合
goto 带来的代价 = 控制流图不可推理
结论：goto 的存在只有害处，没有额外表达力

现代语言（Java、Go、Python、Rust…）都从语言设计层面移除了 goto， 这是结构化编程约束内化为语言的直接体现。 Java 仅保留了 break label / continue label 用于跳出嵌套循环， 但这是有界跳转，控制流仍然可追踪。

从代码范式到架构原则

Bob Martin 在书中反复强调：三种范式本质上都是在"告诉你不能做什么"， 而不是赋予你新能力。结构化编程拿走了 goto， 面向对象拿走了函数指针的直接操控，函数式编程拿走了赋值。

这背后是一个更大的软件工程哲学：约束是自由的基础。 正因为你承诺"函数只有一个入口一个出口（大体上）"， 架构工具才能对模块做依赖分析、做测试隔离、做并发安全推断。

架构连接：结构化编程是模块化的地基。 干净的控制流 → 函数可独立测试 → 函数可组合成模块 → 模块可替换 → 依赖可倒置。 Clean Architecture 的每一层抽象都建立在"控制流可推理"这个前提之上。

实际开发中的落地技巧

1. 卫语句（Guard Clause）优先于嵌套 if

深层 if-else 嵌套在视觉上仍然像乱跳的 goto—— 读者必须同时在脑中维护多个"当前条件"。 用卫语句提前 return，让主干逻辑保持在同一缩进层。

Java — 卫语句 vs 嵌套 if
// ❌ 嵌套 if：主干逻辑淹没在错误处理里
String getUsername(Long userId) {
    if (userId != null) {
        User user = userRepo.findById(userId);
        if (user != null) {
            if (user.isActive()) {
                return user.getName();
            }
        }
    }
    return "anonymous";
}

// ✅ 卫语句：每个异常情况独立处理，主干一目了然
String getUsername(Long userId) {
    if (userId == null)            return "anonymous";
    User user = userRepo.findById(userId);
    if (user == null || !user.isActive()) return "anonymous";
    return user.getName();   // 执行到此处，前置条件 100% 满足
}

2. 循环复杂度（Cyclomatic Complexity）是可量化的信号

每一个 if / else / for / while / case 都给控制流图增加一条边。McCabe 循环复杂度 = 边数 − 节点数 + 2。 业界经验：单函数复杂度超过 10，可读性和可测性就开始急剧下降。 SonarQube、Checkstyle、IDE 都能直接测量这个指标。

常见陷阱：有人认为"我没有用 goto，就已经是结构化编程了"。 错。结构化编程的精髓是控制流可推理， 而 200 行的 if-else if-else if 链或六层嵌套循环， 虽然语法合规，却与 goto 一样难以推理。范式是精神，不是语法检查。

3. 异常不是新的 goto

Java 异常机制可以让控制流"远程跳转"到任意 catch 块， 这和 goto 在本质上是一样的副作用。因此：

  • 异常只用于真正异常的情况，不用来做正常流程控制
  • catch 块应明确文档化"捕获后的状态保证"
  • 能用 Optional / 返回值表达的，不要抛异常

4. 提取函数是结构化的强化版

当一个函数内部有复杂分支时，最有效的重构手法是提取函数： 把每个分支的逻辑提取为具名函数，让顶层函数只剩"顺序调用"。 这正是结构化编程精神的延伸——让每一层的控制流都单纯到可以在 5 秒内理解。

横向对比：三种控制结构的特征

结构 Java 关键字 控制流特征 典型误用
顺序 — (默认) 线性，最易推理 过长函数，职责不单一
分支 if / switch 有限分叉，需枚举所有路径 深层嵌套、缺 default 分支
循环 for / while / do-while 有界重复，需证明终止性 无限循环、在循环内修改迭代变量

一句话总结

结构化编程的遗产不是"禁止 goto"，而是一个更深的工程直觉： 让每一块代码都能被独立推理，是可维护性和可测试性的根基， 也是一切更高层架构约束的出发点。

与其他知识点的联系

  • 00. 总览：总览④指出"控制流从外向内再返回外层，但源码依赖始终向内"——这个说法能立得住的前提，正是本章讲的"控制流可推理"。如果连单个函数内部的执行路径都靠 goto 式的跳转推断，更谈不上在模块、组件、系统层面分析依赖方向。
  • 02. 面向对象编程（下一节）：Bob Martin 把三种编程范式类比为"依次拿走一种能力"——结构化编程拿走 goto，OOP 拿走对函数指针的直接操控。两章的共同主题是"用约束换取可推理性"：本章约束控制流，下一章约束依赖方向，二者叠加才构成整洁架构的地基。
  • 《软件设计的哲学》的"代码应显而易见"：本章"陷阱：200 行 if-else 链条虽合规却不可推理"，与 Ousterhout 讲的"显而易见"是同一个问题的两种表述——前者关注控制流能否被形式化验证，后者关注读者能否在阅读时形成正确心智模型，二者殊途同归，都指向"降低认知负担"。
  • 红旗清单中的"复杂的 if 分支嵌套"：该红旗描述的"条件判断层层嵌套，说明模块边界没有把特殊情况吸收掉"，正是本章"卫语句优先于嵌套 if"落地技巧的反面案例——嵌套过深的本质，往往就是没有用结构化编程的三种基本结构把逻辑拆解清楚。
  • GoF 设计模式中的状态模式（State Pattern）：当一段代码里 if-else 链条是在根据"状态"做分支时，这正是状态模式的典型适用场景——把每个分支封装为独立的类，用多态取代分支判断，这也是下一节"用多态翻转依赖"思想的一个具体应用。
