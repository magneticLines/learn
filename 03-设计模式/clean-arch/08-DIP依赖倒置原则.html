08. DIP — 依赖倒置原则

Dependency Inversion Principle：高层策略不应依赖低层细节，二者都应依赖抽象；抽象不应依赖细节，细节应依赖抽象。

作者的核心主张

Robert C. Martin 认为，DIP 是 SOLID 中影响最深远的一条原则——它直接决定了系统的依赖结构， 是"插件式架构"和"框架与业务解耦"的底层基础。

原则的名字里有一个关键词：倒置（Inversion）。它倒置的不是依赖本身，而是依赖的方向。 在没有刻意设计的系统里，依赖方向总是从高层流向低层——业务逻辑直接调用数据库驱动、HTTP 库、文件系统…… 这种"顺流而下"的依赖让业务层变得脆弱：底层换一次实现，业务层就跟着改。

DIP 的做法是：在高层与低层之间插入一个接口（抽象），让高层依赖接口，低层也依赖（实现）接口。 接口的所有权归高层，低层只是"插进来的插件"。依赖方向从"高层 → 低层"变成了"高层 → 接口 ← 低层"， 控制权彻底倒转。

一句话记忆：依赖抽象，而不是依赖具体。高层定义规则，低层遵守规则。

为什么这样设计更好

先从现实类比入手：

  • 电源插座是一个标准接口（220V/两孔）。电视机、电风扇、手机充电器都依赖这个接口，互相之间完全不知道对方存在。
  • 如果电视机直接接了一条电线焊死到墙上，换墙就得换电视——这就是"高层依赖低层细节"的后果。

回到代码，DIP 带来三大好处：

  • 可替换性：只要实现了接口，任何低层模块都能被替换（真实 DB ↔ 内存 DB ↔ Mock）。
  • 可测试性：业务逻辑不再需要真实的数据库才能跑，单元测试变得纯粹。
  • 稳定性：接口是稳定的，细节是易变的。依赖稳定的东西，系统就不容易因为细节变化而雪崩。

反例 → 问题 → 正例

反例：业务层直接依赖数据库实现

Java — 违反 DIP
// ❌ 低层：MySQL 具体实现
public class MySQLUserRepository {
    public User findById(Long id) {
        // 直连 MySQL，执行 SQL ...
        return new User();
    }
}

// ❌ 高层：业务逻辑直接 new 了具体类
public class UserService {

    // 高层对低层的具体类产生了硬依赖
    private final MySQLUserRepository repo = new MySQLUserRepository();

    public User getUser(Long id) {
        return repo.findById(id);
    }
}

问题在哪里

  • 测试困难：单元测试 UserService 必须启动 MySQL，否则根本跑不起来。
  • 替换困难：要从 MySQL 切到 PostgreSQL，必须修改 UserService——高层受低层牵连。
  • 编译耦合：UserService 的编译依赖 MySQLUserRepository，它俩必须放在同一个模块里，模块划分无从谈起。

正例：引入接口，倒置依赖方向

Java — 遵守 DIP
// ✅ 接口：由"高层策略"这一侧定义，属于业务包
package com.example.domain;

public interface UserRepository {
    User findById(Long id);
}

// ✅ 高层：只依赖接口，通过构造器注入（不自己 new）
package com.example.domain;

public class UserService {

    private final UserRepository repo;

    // 依赖注入：具体是谁，由外部决定
    public UserService(UserRepository repo) {
        this.repo = repo;
    }

    public User getUser(Long id) {
        return repo.findById(id);
    }
}

// ✅ 低层：在基础设施包里实现接口，依赖方向反转
package com.example.infra;

public class MySQLUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        // 真实 MySQL 实现
        return new User();
    }
}

// ✅ 测试时换成内存实现，无需任何真实 DB
package com.example.test;

public class InMemoryUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();

    @Override
    public User findById(Long id) {
        return store.get(id);
    }
}

依赖方向可视化

用模块结构直观展示"倒置"发生在哪里：

【违反 DIP — 依赖顺流而下】
domain 包 └── UserService ──依赖──▶ infra 包 └── MySQLUserRepository
高层模块直接指向低层具体类，方向向下

【遵守 DIP — 依赖方向倒置】
domain 包 ├── UserRepository（接口，归高层所有） └── UserService ──依赖──▶ UserRepository（接口） ▲ 实现（implements） │ infra 包 └── MySQLUserRepository
低层模块依赖（实现）高层定义的接口，方向被"倒置"
关键细节：接口的归属权。接口必须放在高层（domain）包，而不是低层（infra）包。 如果接口放在 infra 里，高层依赖 infra 包才能使用这个接口，依赖方向依然是错的。 "接口归高层所有"才是 DIP 的精髓。

DIP、IoC 与 DI 的关系

这三个概念经常被混淆，先对齐定义：

概念层次说的是什么
DIP（依赖倒置原则） 设计原则 高层不应依赖低层细节，应依赖抽象——目标是什么
IoC（控制反转） 设计模式 / 思想 把"谁来创建依赖"的控制权从调用者手里转移出去——如何实现 DIP 的一种思路
DI（依赖注入） 具体手段 把依赖从外部注入进来（构造器注入 / Setter 注入 / 字段注入）——IoC 的一种落地方式

Spring 的 IoC 容器就是在框架层面强制推行 DIP：你写接口，容器负责在运行时把具体实现注入进来，你的业务代码始终只见接口。

DIP ≠ DI。DIP 是原则，DI 是工具。不用 Spring 一样可以遵守 DIP（手动构造器注入）； 用了 Spring 但接口设计错了，一样违反 DIP。

实际开发落地技巧

技巧 1：以"变化方向"识别哪里需要接口

不是每个地方都需要接口。问自己："这个依赖未来是否有可能被替换或在测试中被 Mock？" 如果是（数据库、消息队列、外部 HTTP 接口、文件系统），就引入接口； 如果几乎不变（String、Math、值对象），直接依赖具体类无妨。 过度抽象是另一种坏味道。

技巧 2：构造器注入优于字段注入

Java — 注入方式对比
// ❌ 字段注入：隐藏依赖，测试时必须用反射或启动容器
@Service
public class OrderService {
    @Autowired
    private OrderRepository repo;  // 依赖被隐藏
}

// ✅ 构造器注入：依赖显式声明，测试直接传 Mock
@Service
public class OrderService {
    private final OrderRepository repo;

    public OrderService(OrderRepository repo) {
        this.repo = repo;
    }
}

// 测试时，直接 new，不需要 Spring 容器
OrderService svc = new OrderService(new InMemoryOrderRepository());

技巧 3：包结构体现依赖方向

一个能直观展示 DIP 落地的目录结构：

src/ ├── domain/ # 高层：业务规则，零外部依赖 │ ├── model/ │ │ └── User.java │ ├── repository/ │ │ └── UserRepository.java # 接口，归高层所有 │ └── service/ │ └── UserService.java │ └── infra/ # 低层：实现细节，依赖 domain ├── persistence/ │ └── MySQLUserRepository.java # implements domain.UserRepository └── config/ └── AppConfig.java # 组装（依赖注入的"主程序"）

domain 包的 pom.xml（或 Gradle 模块）里没有任何 infra 依赖， 只有 infra 模块依赖 domain 模块。这从构建工具层面物理强制了依赖方向。

技巧 4：用"组装层"隔离具体实现的创建

既然高层不能 new 具体类，谁来 new？ 答案是组装层（Composition Root）——通常是 main 方法或 Spring 的 @Configuration 类。具体实现只在最外层被实例化一次，再注入给所有需要它的组件。

Java — Composition Root（无框架版）
public class Application {
    public static void main(String[] args) {
        // 唯一知道具体实现的地方
        UserRepository repo = new MySQLUserRepository();
        UserService    svc  = new UserService(repo);

        // 启动 Web 层，把 svc 传进去 ...
        new WebServer(svc).start();
    }
}

容易忽略的细节 & 常见陷阱

  • 接口放错包：把 UserRepository 接口放到 infra 包里，然后 domain 去引入 infra—— 依赖方向没有真正倒置，只是多绕了一圈。接口必须归高层所有。
  • 接口只有一个实现：不代表不需要接口。就算现在只有 MySQL 实现， 单元测试时你依然需要 InMemory 实现。"只有一个实现"不是跳过抽象的理由。
  • 过度抽象：把所有类都提取接口，包括 UserBuilder、PasswordEncoder（JDK 自带）…… 这只会增加噪音。只对"变化点"和"外部依赖"做抽象。
  • 静态方法绕过 DIP：DateUtil.now()、UUIDGenerator.generate() 这类静态工具 在业务逻辑里直接调用，使得单元测试无法控制时间或 ID——本质上是对具体实现的硬依赖。 应提取为接口（Clock、IdGenerator）注入进来。
  • DIP 不是为了"解耦而解耦"：它的目标是让高层策略不被低层细节拖累。 如果某段代码本身就是低层细节，它依赖另一个低层细节是完全合理的。
面试高频考点："DIP 和 DI 有什么区别？"—— DIP 是设计原则（目标），DI 是实现手段（工具）。Spring IoC 容器是在框架层面强制推行 DIP 的机制。 答出"接口归属权在高层"会让面试官眼前一亮。

与其他 SOLID 原则的联系

原则与 DIP 的关系
OCP（开闭原则） DIP 是 OCP 的底层保障——只有依赖抽象，才能通过新增实现来扩展，而无需修改高层
LSP（里氏替换原则） DIP 让替换成为可能，LSP 保证替换后行为不变；二者配合才能真正做到"可替换"
ISP（接口隔离原则） DIP 要求依赖接口，ISP 要求接口足够精小——ISP 决定接口的"形状"，DIP 决定接口的"方向"

小结

DIP 用一句话概括：让变化的细节依赖稳定的抽象，而不是反过来。

  • 高层策略定义接口，低层细节实现接口——接口归属权在高层。
  • 通过构造器注入把依赖从外部传入，让组装层成为唯一知道具体类的地方。
  • 包结构、模块划分要物理体现依赖方向，用构建工具强制约束。
  • 静态方法调用、字段注入、接口放错包是违反 DIP 的三大隐患。
