06. LSP 里氏替换原则

子类型必须能够替换其基类型——不只是语法上能替换，行为上也要能替换。

原则定义

1988 年 Barbara Liskov 在论文中给出了严格的数学表述：若对类型 S 的对象 o₁，存在类型 T 的对象 o₂，使得在所有以 T 定义的程序 P 中，用 o₁ 替换 o₂ 后程序 P 的行为不变，则 S 是 T 的子类型。

去掉数学外衣，核心主张只有一句话：调用方依赖的是抽象（接口/基类），任何一个实现类都必须完整兑现这份"契约"，而不能偷偷改变可见行为。

LSP 的出发点不是继承语法，而是行为兼容性。语法上能编译通过，不代表满足 LSP；行为上让调用方"无感知切换"，才算真正满足。

为什么这样设计更好

面向对象的核心价值之一是"面向接口编程"——调用方只依赖抽象，不依赖具体实现，从而获得可替换性。但这个价值能否兑现，完全取决于实现类是否真的满足 LSP。

如果子类静默地改变了基类约定的行为（抛出意外异常、收窄前置条件、放宽后置条件……），调用方就必须在运行时做类型检查（instanceof），或者充满 if-else 的特殊处理。这正是 Uncle Bob 最痛恨的"脆弱的设计"——每加一个新子类，每处调用点都要同步修改。

  • 可替换性保证了 OCP 得以成立：OCP 要求对扩展开放、对修改关闭。如果新加的子类不兼容基类行为，每次扩展都逼迫调用方修改——OCP 根本无从实现。
  • 降低心智负担：看到 Shape 就知道 draw() 一定可以调用；看到 Collection 就知道 add() 不会抛 UnsupportedOperationException。规则越稳定，阅读代码越轻松。
  • 架构层面同样适用：《架构整洁之道》明确指出，LSP 已经从类/接口层面上升到了架构层面——服务接口、REST API 契约、微服务之间的替换约定，都遵守同一套逻辑。

经典反例：正方形继承矩形

这是 LSP 文献中最著名的反例，Uncle Bob 在书中也专门用它说明问题。从数学上看，"正方形是矩形的特例"——但这个直觉在代码世界里会带来麻烦。

反例代码

Java · 反例
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int w)  { this.width  = w; }
    public void setHeight(int h) { this.height = h; }
    public int  area()            { return width * height; }
}

// 正方形"是"矩形——继承看起来天经地义
class Square extends Rectangle {
    @Override
    public void setWidth(int w) {
        // 保持正方形约束：宽高必须相等
        this.width  = w;
        this.height = w;  // 偷偷改了 height！
    }
    @Override
    public void setHeight(int h) {
        this.width  = h;  // 偷偷改了 width！
        this.height = h;
    }
}

问题所在

Java · 调用方视角暴露问题
void resize(Rectangle r) {
    r.setWidth(5);
    r.setHeight(4);
    // 调用方合理预期：面积 = 5 × 4 = 20
    assert r.area() == 20;  // 传入 Square 时：4×4=16，断言炸了！
}

// 调用方被迫写出防御性 instanceof 检查 ← LSP 被破坏的明显症状
void resizeSafe(Rectangle r) {
    if (r instanceof Square) {
        // 特殊处理……每加一个子类就要改这里
    }
}
诊断信号：调用方出现 instanceof + 分支处理，几乎总是 LSP 被违反的直接后果。每一个这样的判断，都在向你发出警报。

正例：用组合代替继承

正方形和矩形在数学上有"是"的关系，但在行为契约上并不兼容。解法是不让 Square 继承 Rectangle，而是让它们共享一个更抽象的接口，或者用组合。

Java · 正例（接口分离，各自实现）
interface Shape {
    int area();
}

class Rectangle implements Shape {
    private final int width, height;
    Rectangle(int w, int h) { this.width = w; this.height = h; }
    @Override public int area() { return width * height; }
}

class Square implements Shape {
    private final int side;
    Square(int s) { this.side = s; }
    @Override public int area() { return side * side; }
}

// 调用方只依赖 Shape.area()，两者都能无感知替换
void printArea(Shape s) {
    System.out.println(s.area());  // 不需要 instanceof，永远正确
}

契约三要素：前置条件、后置条件、不变量

LSP 的本质是维护"行为契约"。Bertrand Meyer 的契约式设计（Design by Contract）给出了三个维度，子类必须遵守：

契约维度基类定义的约定子类允许的做法违反举例
前置条件
调用方调用前必须满足		 参数 age >= 0 只能放宽（接受更多输入） 子类要求 age > 18，收窄了调用方的权利
后置条件
方法执行后必须保证		 返回值 > 0 只能收窄（提供更强保证） 子类返回负数，破坏了调用方的假设
不变量
对象始终成立的性质		 余额不能为负 必须保持 子类在某个操作后让余额变为负数
记忆口诀：前置只松不紧，后置只紧不松，不变量原封不动。子类可以给调用方更多自由（放宽前置），也可以给调用方更强承诺（收窄后置），但不能反过来。

Java 生态中的典型案例

案例 1：Collections.unmodifiableList 的 LSP 争议

Collections.unmodifiableList() 返回的对象实现了 List 接口，但调用 add() 会抛出 UnsupportedOperationException。List 的调用方合理期望 add() 能工作——这里存在对 LSP 的妥协。

这也是 Java 集合框架被诟病的历史遗留问题。Java 9+ 的 List.of() 选择了不同的路：直接返回一个独立的不可变类型，而不是伪装成可变 List。

案例 2：正确示范——InputStream 继承体系

InputStream ← 抽象基类，约定 read() 语义
├── FileInputStream ← 从文件读，完整履行契约
├── ByteArrayInputStream ← 从内存读，完整履行契约
└── BufferedInputStream ← 装饰器，增强性能，契约不变

任何接受 InputStream 的方法，传入上述任何子类都能正确工作。这是教科书级别的 LSP 成功案例——read() 的语义（读取字节，返回 -1 表示 EOF）在所有子类中被完整保留。

案例 3：Spring 中的典型违反场景

Java · 反例：子类抛出基类未声明的异常
interface UserRepository {
    User findById(Long id);  // 约定：找不到返回 null
}

class CachedUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        // 违反 LSP：找不到时抛异常而不是返回 null
        User user = cache.get(id);
        if (user == null) throw new CacheNotFoundException(id);
        return user;
    }
}
Java · 正例：遵守契约，找不到回退到 null
class CachedUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        User user = cache.get(id);
        if (user != null) return user;
        // 回退到数据库，与基础实现行为一致
        user = db.findById(id);
        if (user != null) cache.put(id, user);
        return user;  // 找不到仍然返回 null，契约不变
    }
}

架构层面的 LSP

Uncle Bob 在《架构整洁之道》中特别强调：LSP 已经超越了类和接口的范畴，升级为一条架构原则。

考虑一个 REST 聚合服务，它调用下游的"出租车服务"API。最初只有一家供应商，后来接入第二家。两家供应商的接口字段名不同：

JSON · 供应商 A 的响应
{ "driverId": "D001", "pickupTime": "14:30" }
JSON · 供应商 B 的响应（字段名不同）
{ "driver_id": "D002", "pickup_time": "14:45" }

供应商 B 违反了"接口契约"（响应格式即契约）。聚合服务要么为 B 单独写适配分支（if supplier == B），要么引入适配层统一格式。前者正是架构层面 LSP 被破坏的症状——调用方被迫感知具体实现的差异。

正确做法是引入适配器（Adapter）将供应商 B 的响应转换为统一格式，让聚合服务的核心逻辑完全不知道有两家供应商的存在。这正是 LSP + OCP + 适配器模式的联合应用。

实战落地技巧

技巧 1：用测试验证 LSP

针对基类/接口编写"契约测试套件"，然后对每一个实现类跑同一套测试。只要某个实现类让测试失败，就是 LSP 违反。

Java · 契约测试模板
// 抽象契约测试基类——所有实现类的测试都继承它
abstract class UserRepositoryContractTest {

    // 子类提供具体实现
    abstract UserRepository createRepository();

    @Test
    void findById_shouldReturnNullWhenNotFound() {
        UserRepository repo = createRepository();
        assertNull(repo.findById(-1L));  // 验证契约：找不到返回 null
    }
}

// 每个实现只需提供工厂方法，同一套测试自动验证 LSP
class CachedUserRepositoryTest extends UserRepositoryContractTest {
    @Override
    UserRepository createRepository() { return new CachedUserRepository(...); }
}

技巧 2：接口文档中显式写明契约

用 Javadoc 或注释明确写出前置条件、后置条件、异常约定。不要只写"获取用户"，要写"若不存在返回 null，不抛异常；参数 id 不能为 null"。

技巧 3：优先用不可变对象

正方形/矩形问题的根源是可变状态 + 继承的组合。使用不可变对象（构造时确定所有字段，无 setter）可以从根源上消除这类 LSP 陷阱。

技巧 4：警惕"空实现"和"仅抛异常"的方法

子类覆盖某个方法，方法体为空或直接 throw new UnsupportedOperationException()——这是 LSP 违反的高发区。如果真的不需要某个方法，应该用 ISP（接口隔离原则）把接口拆细，而不是靠空实现蒙混过关。

常见误区与面试高频考点

  • 误区：LSP = "子类可以重写父类方法"。不是能重写就行，重写后的行为必须与调用方对父类的预期相符。
  • 误区：LSP 只关心类，不关心接口。接口实现同样要遵守 LSP，REST API、消息格式等架构边界也适用。
  • 面试考点：如何判断某个继承关系违反了 LSP？ 核心判断标准：能否在不看运行时类型的情况下，用子类替换父类且程序行为不变？如果调用方需要 instanceof，几乎可以断定 LSP 被违反。
  • 面试考点：LSP 和 OCP 的关系？ LSP 是 OCP 的前提条件。只有子类能够无感知替换基类（LSP），"对扩展开放、对修改关闭"（OCP）才能真正落地。
  • 面试考点：Java 标准库中违反 LSP 的例子？ Properties extends Hashtable——Properties 应该只存字符串，但继承了 Hashtable 的 put(Object, Object)，允许放入非字符串，破坏了 Properties 自身的不变量。
一句话总结：LSP 的本质是行为契约的继承，而不仅仅是类型的继承。"是一个"关系在数学上成立，并不意味着在代码行为上也成立。判断标准只有一个：调用方能否在不知道具体类型的前提下，对任何子类都依赖相同的假设。

与其他知识点的联系

  • 关联 OCP（开闭原则）：OCP 允许通过新增子类来扩展行为而不修改既有代码，但这个承诺的前提是新子类必须能无感知替换基类——也就是满足 LSP。反过来说，一旦 LSP 被违反，调用方就得靠 instanceof 判断类型来"修改"逻辑，OCP 立刻失效。二者是因果关系，不是并列关系。
  • 关联 ISP（接口隔离原则）：本节"技巧四"提到的"空实现/仅抛异常"问题，根源往往是接口太宽——某个实现类只能履行接口的一部分契约。正确解法不是让子类偷偷违反契约，而是用 ISP 把接口拆细，让每个实现类只需要面对自己能完整履行的那部分契约。
  • 关联 DIP（依赖倒置原则）：DIP 让调用方依赖抽象类型（接口/基类），这个设计能带来好处的前提正是 LSP——如果实现类不能安全替换抽象，那么"依赖抽象"只是语法上的解耦，运行时仍然要靠类型判断绕开契约差异，DIP 的收益就名不副实。
  • 关联 GoF 模板方法模式 / 策略模式：这两个模式都依赖"多个实现可以互换"这一假设——模板方法里子类只重写特定步骤，策略模式里各策略可以任意替换。任何一个具体实现偷偷改变了基类/接口约定的行为，都会让这些模式在运行时出现难以定位的 bug，这也是本节"契约测试"技巧要解决的问题。
  • 关联架构层面的适配器模式：本节"出租车供应商"的例子展示了 LSP 从类设计升级为架构约束——REST API、微服务响应格式本身就是一种"契约"，供应商 B 破坏格式约定，本质与 Square 破坏 Rectangle 的契约是同一类问题，解法也同样是引入适配器统一契约，而不是让调用方感知差异。
