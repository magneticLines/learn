15. 架构是什么（保留选项、推迟决策）

架构不是框架选型，不是技术栈堆叠——它是一种推迟无关决策、保留未来选择余地的艺术。

作者的核心主张：架构师是什么人

Bob 大叔在书中给出了一个反直觉的定义：

软件架构师是这样一种程序员：他持续编写代码，同时引导团队将系统设计得易于开发、易于部署、易于运维，并保留最多的选项。

注意最后四个字——保留选项（keeping options open）。这才是架构的核心目标，而不是选一个"最好的框架"或"最流行的数据库"。

为什么？因为在项目早期，我们对以下问题几乎一无所知：

  • 用户规模究竟有多大，需要几台服务器？
  • 读写比例是多少，需要什么存储引擎？
  • 业务规则会往哪个方向演化？
  • 团队规模会扩大还是缩减？

早期做出的每一个"硬绑定"决策，都相当于用一把锁锁死了未来的选项。好的架构让这些决策尽可能晚做，并且做错了也能以低成本纠正。

两类决策：高层政策 vs 低层细节

理解"保留选项"的前提，是区分两类决策：

类型例子特征
高层政策（Policy） 业务规则、用例、核心实体 变化缓慢，是系统的价值所在
低层细节（Detail） 数据库、Web 框架、消息队列、UI 框架 变化频繁，是交付手段，不是目的

架构的职责是：让高层政策对低层细节一无所知，反过来让低层细节依赖高层政策。这样一来，当我们需要把 MySQL 换成 PostgreSQL，或把 Spring MVC 换成 Vert.x 时，高层业务规则毫发无损。

陷阱：把框架当架构。很多项目"架构"其实是 Spring Boot 项目结构——Controller / Service / Repository 三层。这只是框架的惯用分层，不是真正的架构。一旦框架本身成为核心依赖，业务逻辑就被框架绑架了。

反例 → 问题 → 正例

场景：用户注册业务逻辑

假设我们要实现"注册用户"的功能：校验邮箱唯一性、加密密码、发送欢迎邮件。

反例：业务逻辑直接耦合框架与基础设施

UserService.java（反例）
@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;   // Spring Data JPA — 直接依赖 ORM

    @Autowired
    private JavaMailSender mailSender;       // Spring Mail — 直接依赖邮件基础设施

    public void register(String email, String password) {
        // 业务规则 1：邮箱唯一性检查
        if (userRepository.existsByEmail(email)) {
            throw new RuntimeException("Email already exists");
        }
        // 业务规则 2：密码加密（BCrypt 硬编码）
        String hashed = BCrypt.hashpw(password, BCrypt.gensalt());
        // 数据存储：直接调用 JPA
        UserEntity entity = new UserEntity(email, hashed);
        userRepository.save(entity);
        // 发送欢迎邮件：直接用 JavaMailSender
        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setTo(email);
        msg.setSubject("Welcome!");
        mailSender.send(msg);
    }
}

问题在哪？

  • 想把 JPA 换成 MyBatis 或 MongoDB？UserService 必须修改。
  • 想把邮件换成消息队列异步发送？UserService 必须修改。
  • 想单元测试业务规则？必须启动 Spring 容器、连接数据库、连接邮件服务器。
  • 每一个"细节"的变化，都会穿透到业务逻辑层。选项被一一锁死。
本质问题：依赖方向反了。业务逻辑（高层）依赖了数据库和邮件（低层细节），而不是反过来。

正例：用接口隔离细节，让业务规则保持纯粹

先定义业务需要的端口（Port）——只关心"我需要什么能力"，不关心"谁来实现"：

端口接口（业务层定义）
// 业务层定义：我需要一个"用户存储"能力
public interface UserRepository {
    boolean existsByEmail(String email);
    void save(User user);
}

// 业务层定义：我需要一个"通知"能力
public interface NotificationGateway {
    void sendWelcome(String email);
}

// 业务层定义：密码加密策略
public interface PasswordEncoder {
    String encode(String raw);
}
RegisterUserUseCase.java（纯业务逻辑，零框架依赖）
// 只依赖接口，不依赖任何框架或基础设施
public class RegisterUserUseCase {

    private final UserRepository      users;
    private final PasswordEncoder     encoder;
    private final NotificationGateway notify;

    public RegisterUserUseCase(
            UserRepository users,
            PasswordEncoder encoder,
            NotificationGateway notify) {
        this.users   = users;
        this.encoder = encoder;
        this.notify  = notify;
    }

    public void execute(String email, String password) {
        // 业务规则 1：邮箱唯一性
        if (users.existsByEmail(email)) {
            throw new EmailAlreadyExistsException(email);
        }
        // 业务规则 2：密码加密
        String hashed = encoder.encode(password);
        // 保存实体
        users.save(new User(email, hashed));
        // 发送通知
        notify.sendWelcome(email);
    }
}
基础设施层：JPA 实现（可随时替换）
// 基础设施层实现接口——依赖方向：基础设施 → 业务，而不是反过来
@Repository
public class JpaUserRepository implements UserRepository {

    @Autowired
    private JpaUserJpaRepo jpa;   // Spring Data JPA 接口

    @Override
    public boolean existsByEmail(String email) {
        return jpa.existsByEmail(email);
    }

    @Override
    public void save(User user) {
        jpa.save(new UserEntity(user));
    }
}
单元测试：无需启动任何框架
class RegisterUserUseCaseTest {

    @Test
    void should_throw_when_email_exists() {
        // 用内存假对象代替真实数据库——这就是"保留选项"的红利
        UserRepository      fakeRepo   = email -> true, u -> {};
        PasswordEncoder     fakeEnc    = raw   -> "hashed";
        NotificationGateway fakeNotify = e     -> {};

        RegisterUserUseCase uc =
            new RegisterUserUseCase(fakeRepo, fakeEnc, fakeNotify);

        assertThrows(EmailAlreadyExistsException.class,
            () -> uc.execute("a@b.com", "pass"));
    }
}

依赖方向：谁依赖谁，一眼看清

┌──────────────────────────────────────────────┐
│ 业务层（Domain） │
│ RegisterUserUseCase User EmailAlreadyExistsException │
│ 接口：UserRepository PasswordEncoder NotificationGateway │
└──────────┬───────────────────────────────────┘
│ ← 基础设施层实现这些接口（依赖倒置）
┌──────────▼───────────────────────────────────┐
│ 基础设施层（Infrastructure） │
│ JpaUserRepository BcryptPasswordEncoder │
│ MailNotificationGateway / KafkaNotificationGateway │
└──────────────────────────────────────────────┘

业务层不知道基础设施层的存在。基础设施层依赖业务层定义的接口。这条箭头方向，就是"保留选项"的物理保证。

推迟决策：越晚做越好的具体例子

决策为什么要推迟如何推迟
用什么数据库 早期不知道读写比、数据规模、事务需求 先用 H2 内存库 + Repository 接口，上线前替换实现
同步还是异步发通知 早期不知道通知量，同步可能成为瓶颈 NotificationGateway 先同步实现，后期换 Kafka 实现无感迁移
REST 还是 gRPC 早期客户端可能只有一种，后期可能都需要 业务逻辑不绑定 HTTP，Controller 只是薄适配层
单体还是微服务 早期团队小，微服务开销大于收益 先做整洁单体（模块边界清晰），后期按边界拆服务
Bob 的原话："好的架构师让决策保持开放（open），并尽可能长时间地推迟做出决策，这样才能获得最多信息来做出正确判断。"

实际落地技巧

1. 以"用例"而不是"框架"组织代码

包结构按业务用例划分，而不是按技术层划分：

// ❌ 按技术层划分（框架驱动）
com.example
├── controller/ UserController OrderController
├── service/ UserService OrderService
└── repository/ UserRepo OrderRepo

// ✅ 按用例划分（业务驱动）
com.example
├── user/
│ ├── RegisterUserUseCase.java ← 业务逻辑
│ ├── UserRepository.java ← 端口接口（业务层定义）
│ └── infra/
│ └── JpaUserRepository.java ← 基础设施实现
└── order/
├── PlaceOrderUseCase.java
└── ...

2. 对框架保持"礼貌的距离"

框架是工具，不是架构。把 Spring 的 @Service、@Repository 注解视为"组装标签"而不是"业务身份"。业务类不应该 extend 框架类——一旦 extend，业务逻辑就和框架生命周期绑死了。

3. "假"实现驱动开发

在基础设施还没准备好时，先用内存 Map 实现接口，让业务逻辑和测试跑起来。这就是"推迟决策"在日常开发中的体现：

InMemoryUserRepository.java（临时实现，推迟数据库决策）
public class InMemoryUserRepository implements UserRepository {

    private final Map<String, User> store = new HashMap<>();

    @Override
    public boolean existsByEmail(String email) {
        return store.containsKey(email);
    }

    @Override
    public void save(User user) {
        store.put(user.getEmail(), user);
    }
}

// 当真实数据库准备好，只需换一个实现类，业务逻辑零改动。

4. 识别"假灵活性"陷阱

有一种常见误区：把接口当成灵活性的银弹。如果接口里的方法签名本身就暴露了数据库细节（如 findByEmailAndStatusOrderByCreatedAtDesc），那这个接口实质上已经把业务逻辑和数据库 schema 绑定了，换数据库一样困难。

检验标准：看接口方法名和参数，是否包含存储细节？是否包含框架特有类型？如果是，接口只是形式上的隔离，不是真正的边界。

面试高频考点

  • 架构和设计的区别是什么？ — 架构关注系统级别的结构和边界，设计关注模块内部实现。但 Bob 认为两者没有明确界线，都服务于同一目标：保留选项。
  • 依赖倒置原则（DIP）和"保留选项"是什么关系？ — DIP 是实现"保留选项"的技术手段：高层模块不依赖低层模块，都依赖抽象。抽象（接口）定义在高层，低层实现它，这样低层可替换。
  • 为什么说框架不是架构？ — 框架是低层细节，架构是高层结构。把框架强行置于中心，会让业务逻辑依赖框架生命周期，失去独立测试能力和迁移能力。
  • 什么样的代码算"架构好"？ — 能把数据库替换掉而不修改业务逻辑；能在不启动 Web 容器的情况下完整测试核心用例。
一句话总结：架构的核心价值不是"选了什么技术"，而是"保留了多少选项"。好的架构让你在获得更多信息之后，再做出更好的决策——而不是在信息最少的时候，就被迫锁死所有选择。

与其他知识点的联系

  • 关联 DIP（依赖倒置原则）：本节面试考点已指出，DIP 正是"保留选项"落地的技术手段——业务层定义接口，基础设施层实现它，依赖箭头因此反转。没有 DIP，"保留选项"只是一句口号；有了 DIP，换数据库、换消息中间件才真正只是"换一个实现类"这么简单。
  • 关联 SDP（稳定依赖原则）："高层政策 vs 低层细节"这组划分，本质就是 SDP 里"稳定 vs 不稳定"的另一种表述——业务规则变化缓慢（低 I 值），框架和数据库变化频繁（高 I 值）。SDP 给出的"依赖要指向稳定"规则，正是本节"细节依赖业务、业务不知道细节"这条依赖方向的理论依据。
  • 关联 SRP/CCP（单一职责 / 共同闭包）："以用例组织代码而非以技术层组织代码"这条落地技巧，背后的判断标准和 SRP、CCP 完全一致——用例（如 RegisterUserUseCase）是一个独立的变更单元，理应聚合在一起；而技术层（controller/service/repository）本身不是变更的驱动力，按技术层分包只是表面整齐，实际上把同一用例的代码拆散到了多个目录。
  • 关联下一节"划清边界"：本节讲的是"为什么要保留选项"（目标），下一节"划清边界"讲的是"用什么机制保留选项"（手段）——边界就是保留选项的物理实现，Humble Object、端口适配器等具体模式都是本节"依赖倒置图"的进一步展开。
  • 关联 GoF 策略模式与依赖注入容器：示例中 RegisterUserUseCase 通过构造函数接收 UserRepository、PasswordEncoder 等接口，这正是策略模式 + 依赖注入的组合应用——策略模式提供"可替换算法"的结构，Spring 的 IoC 容器负责在运行时把具体策略"装配"进来，二者合力实现了本节强调的"框架保持礼貌的距离"。
