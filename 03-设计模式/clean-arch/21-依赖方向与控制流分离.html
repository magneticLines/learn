21. 依赖方向与控制流分离

源码依赖可以与运行时控制流方向相反——这是架构边界得以成立的根本手段。

核心主张：依赖方向 ≠ 控制流方向

在没有任何设计介入的情况下，"调用者调用被调用者"意味着源码依赖方向与控制流方向完全一致： A 调用 B，A 的源文件就必须 import B。看起来天经地义，却暗藏风险—— 它使高层策略（业务规则）直接依赖低层细节（数据库、UI、第三方库）， 细节一变，策略就得跟着重新编译、测试、部署。

Robert C. Martin（Uncle Bob）在《架构整洁之道》中明确指出： 利用多态，我们可以让源码依赖方向与控制流方向相反。 这一能力被称为"依赖反转"（Dependency Inversion），是架构边界得以存在的物理基础。

一句话总结：控制流（谁在运行时调用谁）由业务逻辑决定， 源码依赖（谁在编译时 import 谁）则可以被我们人为扭转，从而保护业务规则不受细节污染。

用图建立直觉

未经设计时：依赖与控制流同向

控制流：UseCase ──调用──▶ MySQLOrderRepository
源码依赖：UseCase.java ──import──▶ MySQLOrderRepository.java

结果：UseCase 与 MySQL 紧耦合，换数据库需修改业务层。

引入接口后：依赖反转，控制流不变

控制流：UseCase ──调用──▶ «interface» OrderRepository ──(多态)──▶ MySQLOrderRepository

源码依赖：UseCase.java ──import──▶ OrderRepository.java（接口，属于业务层）
           MySQLOrderRepository.java ──implements──▶ OrderRepository.java

结果：MySQL 依赖业务层接口，依赖箭头反转，业务层零感知细节。

控制流的方向没有变（UseCase 依然在运行时驱动数据库操作）， 但源码依赖的箭头被我们通过接口翻转了。 这就是"分离"的含义：两者可以独立选择方向。

反例：业务规则直接依赖数据库实现

Java — 反例（依赖与控制流同向）
// ❌ 业务用例直接 import 数据库实现
import com.example.infrastructure.MySQLOrderRepository;

public class PlaceOrderUseCase {

    // 直接持有具体实现——编译期就与 MySQL 绑死
    private final MySQLOrderRepository repo;

    public PlaceOrderUseCase(MySQLOrderRepository repo) {
        this.repo = repo;
    }

    public void execute(Order order) {
        // 业务逻辑 …
        repo.save(order);
    }
}

这样写有什么问题？

  • 无法独立测试业务规则：单元测试必须启动数据库，或者用反射 hack，才能替换 MySQLOrderRepository。
  • 细节变动波及业务层：从 MySQL 换成 PostgreSQL，需要修改 PlaceOrderUseCase.java 并重新编译，哪怕业务逻辑一行未改。
  • 无法单独部署：业务规则与数据库实现必须一起打包，无法将它们部署到不同服务。
  • 违反稳定依赖原则：稳定的业务规则依赖了不稳定的数据库细节，依赖方向与变化频率相反，极易出现"改了细节、整个系统受影响"。

正例：用接口反转源码依赖

第一步：在业务层定义接口（接口归属业务层，不归数据库层）

Java — 业务层接口（属于 domain / usecase 包）
package com.example.domain.repository;

// ✅ 接口定义在业务层，由业务层"拥有"
public interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(OrderId id);
}

第二步：业务用例只依赖接口

Java — 业务用例（只 import 接口）
package com.example.usecase;

import com.example.domain.repository.OrderRepository; // 仍在业务层内部

public class PlaceOrderUseCase {

    // ✅ 只持有接口——编译期对 MySQL 零感知
    private final OrderRepository repo;

    public PlaceOrderUseCase(OrderRepository repo) {
        this.repo = repo;
    }

    public void execute(Order order) {
        // 纯业务逻辑，不涉及任何基础设施
        if (order.isValid()) {
            repo.save(order);
        }
    }
}

第三步：数据库实现依赖接口（依赖方向反转）

Java — 基础设施层实现（import 业务层接口）
package com.example.infrastructure.persistence;

import com.example.domain.repository.OrderRepository; // ← 朝业务层方向依赖

// ✅ MySQL 实现 import 业务层接口，依赖箭头指向业务层
public class MySQLOrderRepository implements OrderRepository {

    @Override
    public void save(Order order) {
        // JDBC / JPA 具体操作 …
    }

    @Override
    public Optional<Order> findById(OrderId id) {
        // …
        return Optional.empty();
    }
}

第四步：在组合根（Main / DI 容器）装配

Java — 组合根（Spring 示例）
@Configuration
public class AppConfig {

    // 组合根是唯一知道"接口 → 实现"映射的地方
    @Bean
    public PlaceOrderUseCase placeOrderUseCase(OrderRepository repo) {
        return new PlaceOrderUseCase(repo);
    }

    @Bean
    public OrderRepository orderRepository() {
        return new MySQLOrderRepository();
    }
}
测试时替换实现：单元测试只需传入 InMemoryOrderRepository（也实现 OrderRepository）， 业务用例代码不需要改一行，也不需要启动数据库。

为什么这样设计代码更好？

关注点 同向（无接口） 反转（有接口）
独立可测试性 必须启动数据库 内存实现即可测试业务逻辑
替换细节 修改业务层代码 只换实现类，业务层零改动
独立部署 业务 + 细节必须同包 可分 JAR / 微服务独立部署
编译依赖 业务层依赖基础设施 JAR 基础设施 JAR 依赖业务 JAR
稳定性 细节变化传染业务层 细节变化被接口隔离

深入原理：多态是唯一的魔法

接口（或抽象类）之所以能反转依赖，靠的是运行时多态： 调用者在源码层只知道抽象类型，真正的实现在运行时由 JVM 的虚方法表（vtable）决定。 编译器只检查"是否实现了接口"，而不关心具体是哪个实现。

这也是 Uncle Bob 说的"面向对象语言最重要的特性不是封装、继承，而是多态对依赖方向的控制能力"。 C 语言靠函数指针也能做到，只是没有语言级别的安全保障。Java/Kotlin/C# 的接口把这个能力标准化了。

常见误区：接口必须放在"接口包"或"基础设施包"里。 错！接口应该归属于它的调用方（业务层），而不是实现方（基础设施层）。 谁"拥有"接口，谁就能控制依赖方向。把接口放到 infrastructure 包里等于没有反转，只是多了个文件。

架构边界视角：跨越边界的规则

《架构整洁之道》的架构边界理论建立在一个规则上： 所有跨越边界的源码依赖，必须指向更高层次的策略（业务规则方向）。 控制流可以双向流动（请求进入业务层，结果返回 UI），但源码依赖必须单向——朝业务规则。

高层（稳定）
  Entities（企业业务规则）
    ▲ 依赖只能向上
  Use Cases（应用业务规则）
    ▲ 依赖只能向上
  Interface Adapters（控制器 / 网关 / Presenter）
    ▲ 依赖只能向上
  Frameworks & Drivers（数据库 / Web / UI）
低层（不稳定）

当控制流需要从高层流向低层时（如 Use Case 触发数据库写入）， 就在高层定义接口，让低层实现它——这就是依赖反转的全部动作。

实际开发中的落地技巧

1. 按"谁拥有接口"划分包结构

src/main/java/
  com.example.
    domain/ ← 业务实体 + 领域服务
    domain/repository/ ← 接口定义（属于业务层！）
    usecase/ ← 用例，只 import domain
    infrastructure/ ← 实现类，import domain 接口
    web/ ← Controller，import usecase
    config/ ← 组合根，唯一知道全局的地方

2. 用 ArchUnit 在 CI 中自动守护依赖规则

Java — ArchUnit 规则示例
@AnalyzeClasses(packages = "com.example")
public class ArchitectureTest {

    @ArchTest
    static final ArchRule domainMustNotDependOnInfrastructure =
        noClasses()
            .that().resideInAPackage("..domain..")
            .should().dependOnClassesThat()
            .resideInAPackage("..infrastructure..");
}

3. 控制流跨边界时，数据结构也要独立

跨越边界传递的数据应该是简单的数据结构（DTO / Value Object）， 而不是数据库 Entity 对象。否则边界两侧通过共享数据结构重新产生编译依赖， 接口虽然反转了，数据结构又把依赖拉回来了。

4. 组合根越小越好

Main 方法或 DI 配置类是唯一"知道一切"的地方， 它位于架构的最外层（最低层），允许依赖所有圈层。 但要保持它尽量薄——只做装配，不写业务逻辑。

容易忽略的细节与陷阱

  • 接口放错位置：把 OrderRepository 接口放到 infrastructure 包， 再让 usecase 去 import 它——依赖方向没有反转，只是多了一层空接口。 接口必须和它的调用方在同一圈层。
  • 返回值类型泄露细节：接口方法返回 JpaOrder（JPA Entity）而非 Order（领域对象）， 同样把底层类型引入了业务层，等于接口形同虚设。
  • 静态工厂方法绕过接口：OrderRepository.getInstance() 直接返回具体类， 或者用 new MySQLOrderRepository() 内嵌在业务代码里——依赖反转立刻失效。
  • 过度抽象：不是所有依赖都需要反转。只有跨越架构边界、 或者需要在测试时替换的依赖才值得引入接口。 同一圈层内部的类之间直接依赖反而更清晰。
  • 忘记 Output Port：业务用例调用 Presenter 输出结果时，同样需要接口（Output Port）， 否则用例依赖了 UI 层，又一次让控制流方向绑架了源码依赖方向。
面试高频："依赖倒置原则（DIP）"与"依赖注入（DI）"不是同一回事。 DIP 是设计原则（高层不依赖低层，二者都依赖抽象）；DI 是实现手段之一（通过构造器/setter 注入抽象）。 Spring IoC 容器让 DI 变得方便，但 DIP 靠的是接口，不靠容器。

与其他知识点的联系

  • 依赖倒置原则（DIP）：本文其实就是 DIP 的机制拆解——DIP 说"高层不应依赖低层，两者都应依赖抽象"，而本文回答了"这具体是怎么做到的"：靠多态，让实现类的源码 import 指向接口，而不是接口指向实现类，从而让控制流和依赖方向可以各走各的。
  • 整洁架构同心圆：同心圆里"依赖只能由外向内"这条铁律，正是本文讲的依赖反转在多层架构上的连续应用——每跨越一层边界，就要在内层定义一次接口，让外层实现它，四个圆层之间的边界都遵循同一套反转手法。
  • 谦卑对象模式（上一篇）：Humble Object 处于边界外侧、Logic Object 处于边界内侧，两者的解耦同样依赖本文的接口反转手法——如果 Logic Object 需要通知谦卑对象做某件事，也要通过接口（Output Port）而不是直接 import 具体类，否则控制流方向会把依赖方向拖着走。
  • 数据库是细节（下一篇）：本文是"数据库是细节"得以成立的技术基础——"数据库只是细节"这句话之所以不是空谈，正是因为可以用接口把 OrderRepository 的调用方（业务层）与实现方（JPA/MySQL）解耦，让依赖箭头从"业务→数据库"反转成"数据库→业务"。
  • Spring 框架实践：Spring 的 @Autowired/构造器注入只是帮你把接口和实现"接起来"的机械手段（对应本文"组合根"的角色），DIP 的关键决策——接口该定义在哪一层、由谁拥有——Spring 本身并不负责，需要开发者自己遵循依赖规则来放置。

小结

依赖方向与控制流分离，是整洁架构中所有"边界"得以维持的根本机制。 只需一个接口，就能让源码依赖箭头反转——高层业务规则不再被低层细节污染， 系统因此获得独立可测试、独立可部署、独立可替换的能力。 记住两个关键点：接口归属调用方，装配发生在组合根。
