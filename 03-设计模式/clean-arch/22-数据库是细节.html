22. 数据库是细节

架构视角下，关系型数据库只是一种 I/O 设备，业务规则不应感知它的存在。

核心主张：数据库不是架构，是实现细节

Robert C. Martin 在《架构整洁之道》中明确指出：数据库是一个细节（Detail）。 所谓"细节"，是指对业务规则没有本质影响、可以被替换的技术实现选择。 MySQL、PostgreSQL、MongoDB——换哪个，业务逻辑不应改变一行代码。

这个观点听起来颠覆常识。我们习惯把数据库放在架构图的正中央， 让 Entity、Service、Controller 全部依赖于它。 但 Clean Architecture 的核心原则恰恰相反： 依赖必须指向高层策略（业务规则），而非指向低层细节（数据库）。

类比：数据库之于业务规则，就像硬盘之于 CPU——CPU 执行的指令不应该写死"数据在哪块硬盘的哪个扇区"。 数据的持久化方式是 I/O 策略，CPU（业务逻辑）只关心"给我数据"，不关心"从哪里取"。

为什么我们总把数据库放在核心？

历史原因让这个错误显得"天经地义"：

  • 早期开发工具（如 Visual Basic、Delphi）以数据库为中心构建 UI，数据库表即模型。
  • ORM 框架（Hibernate、MyBatis）把数据库表映射为"实体类"，名字都叫 @Entity，强化了"实体 = 数据库表"的误解。
  • 初学者教程几乎都以"先建表、再写 DAO、再写 Service"为顺序，暗示数据库是出发点。

这种做法的代价是：业务规则与数据库实现紧耦合。 一旦要从 MySQL 迁移到 MongoDB，或者引入 Redis 缓存层， 业务代码就得大改——这不是数据库的错，是我们把细节渗透进了核心。

反例：业务逻辑直接依赖数据库框架

以下代码展示了一个常见的"坏味道"——Service 直接依赖 JPA Repository，业务逻辑与持久化实现混在一起。
UserService.java — 反例（依赖 JPA）
// 反例：Service 直接持有 Spring Data JPA 接口
@Service
public class UserService {

    // 直接依赖 JPA Repository —— 这是一个数据库框架接口
    private final UserJpaRepository userRepo;

    public UserService(UserJpaRepository userRepo) {
        this.userRepo = userRepo;
    }

    public void register(String email, String password) {
        // 业务规则：邮箱不能重复
        if (userRepo.existsByEmail(email)) {
            throw new IllegalArgumentException("Email already exists");
        }
        // 直接操作 JPA Entity，Entity 携带 @Column、@Table 等注解
        UserEntity entity = new UserEntity();
        entity.setEmail(email);
        entity.setPassword(hash(password));
        userRepo.save(entity);
    }
}

// UserEntity 身兼两职：业务模型 + 数据库映射
@Entity
@Table(name = "users")
public class UserEntity {
    @Id @GeneratedValue
    private Long id;
    @Column(unique = true)
    private String email;
    private String password;
    // getter/setter ...
}

这段代码的问题

  • 业务规则与持久化混在一起：UserEntity 既是业务对象，又是 JPA 表映射，改数据库 schema 就要改业务模型。
  • 测试困难：想单元测试 register() 的"邮箱不能重复"逻辑，必须启动 Spring 容器、准备 H2 内存库，测试速度慢、成本高。
  • 框架侵入：如果有一天换成 MongoDB，UserService 要全部重写，而它本质上只做了一件事——校验邮箱唯一性并保存。

正例：用依赖反转隔离数据库

解法是依赖反转原则（DIP）：业务层定义一个 Repository 接口（抽象）， 数据库实现类（JPA Adapter）在外层实现它。依赖箭头从"业务 → 数据库"变成了"数据库 → 业务"。

src/
├── domain/ ← 内层：业务规则，零框架依赖
│ ├── User.java ← 纯 POJO，无 @Entity
│ └── UserRepository.java ← 接口，业务层定义
├── application/
│ └── UserService.java ← 只依赖 domain 接口
└── infrastructure/
└── persistence/
├── UserJpaEntity.java ← JPA 映射对象
├── UserJpaRepository.java ← Spring Data 接口
└── UserRepositoryAdapter.java ← 实现 domain.UserRepository
domain/User.java — 纯业务对象
package com.example.domain;

// 没有任何 JPA 注解，不依赖任何框架
// 这才是真正的"领域实体"
public class User {
    private final String email;
    private final String hashedPassword;

    public User(String email, String hashedPassword) {
        this.email = email;
        this.hashedPassword = hashedPassword;
    }

    public String getEmail() { return email; }
    public String getHashedPassword() { return hashedPassword; }
}
domain/UserRepository.java — 业务层定义的接口（抽象）
package com.example.domain;

// 这是业务层的"愿望清单"：我需要这些数据操作能力
// 我不在乎你用 MySQL、MongoDB 还是内存 Map 实现
public interface UserRepository {
    boolean existsByEmail(String email);
    void save(User user);
}
application/UserService.java — 正例（只依赖接口）
package com.example.application;

@Service
public class UserService {

    // 依赖的是 domain 层的接口，不是任何框架类
    private final UserRepository userRepo;

    public UserService(UserRepository userRepo) {
        this.userRepo = userRepo;
    }

    public void register(String email, String rawPassword) {
        // 业务规则：邮箱唯一性校验
        if (userRepo.existsByEmail(email)) {
            throw new DuplicateEmailException(email);
        }
        User user = new User(email, hash(rawPassword));
        userRepo.save(user);
    }
}
infrastructure/persistence/UserRepositoryAdapter.java — 适配器（在外层）
package com.example.infrastructure.persistence;

// 外层（infrastructure）依赖内层（domain），方向正确
@Component
public class UserRepositoryAdapter implements UserRepository {

    private final UserJpaRepository jpa;

    public UserRepositoryAdapter(UserJpaRepository jpa) {
        this.jpa = jpa;
    }

    @Override
    public boolean existsByEmail(String email) {
        return jpa.existsByEmail(email);
    }

    @Override
    public void save(User user) {
        // domain.User → JPA Entity 转换，隔离在这里
        UserJpaEntity entity = new UserJpaEntity(
            user.getEmail(), user.getHashedPassword()
        );
        jpa.save(entity);
    }
}
现在测试 UserService 的成本极低：只需 Mock 掉 UserRepository 接口，传入内存实现，完全不需要数据库。 业务规则测试与持久化实现彻底解耦。

依赖方向：谁依赖谁？

──────────────────── 反例的依赖方向 ────────────────────

UserService ──────────────→ UserJpaRepository (JPA)
↓
MySQL

业务核心直接指向数据库框架，框架换了，业务也要改。

──────────────────── 正例的依赖方向 ────────────────────

UserService ─→ UserRepository（接口，在 domain 层）
↑
UserRepositoryAdapter ─→ UserJpaRepository ─→ MySQL

所有箭头都指向内层（domain），外层（JPA/MySQL）依赖内层接口。
替换 MySQL → MongoDB，只需换 Adapter，Service 零改动。

常见混淆：数据模型 ≠ 领域模型

ORM 框架起名叫 @Entity 是一种历史误导。 领域驱动设计（DDD）中的 Entity 是具有业务身份的领域对象， 而 JPA 的 @Entity 是数据库表的映射对象——两者责任完全不同。

维度 领域对象（Domain Model） 数据库映射对象（Persistence Model）
所在层 domain（最内层） infrastructure（最外层）
依赖 零框架依赖 JPA / MyBatis 等注解
职责 表达业务概念，封装业务规则 映射数据库表结构，处理 ORM 细节
变化原因 业务规则变了 数据库 schema 变了 / 换数据库
测试 纯单元测试，ms 级 需要数据库环境，集成测试
常见陷阱：把 JPA Entity 当 DTO 传给前端、或者直接在 Service 里调用 entityManager.flush()， 都是把数据库实现细节渗透进了业务层——未来每次数据库调整都会波及业务代码。

实际开发中的落地技巧

技巧一：先写接口，再写实现

开发新功能时，先在 domain 层定义 Repository 接口， 用内存实现（HashMap 存储）跑通业务逻辑和测试， 再去写 JPA Adapter。这个顺序能自然检验业务逻辑是否独立于数据库。

InMemoryUserRepository.java — 用于测试的内存实现
// 测试时注入这个实现，完全不需要 Spring 容器
public class InMemoryUserRepository implements UserRepository {
    private final Map<String, User> store = new HashMap<>();

    @Override
    public boolean existsByEmail(String email) {
        return store.containsKey(email);
    }

    @Override
    public void save(User user) {
        store.put(user.getEmail(), user);
    }
}

技巧二：Module / Package 边界即是架构边界

用 Java 的 module 系统或者 Maven 多模块来强制边界： domain 模块不得依赖 infrastructure 模块， 编译时就会报错，彻底杜绝"手滑"引入框架类。

pom.xml (parent)
├── domain/ ← 无任何外部依赖
├── application/ ← 仅依赖 domain
└── infrastructure/ ← 依赖 domain + Spring + JPA

技巧三：接口方法用领域语言，不用数据库语言

Repository 接口方法命名要体现业务语义，而不是数据库操作语义：

  • 好：findActiveUsersByRegion(String region)
  • 坏：selectFromUsersWhereStatusEqActiveAndRegionEq(String region)
  • 坏：queryWithJpql(String jpql)（把 JPQL 暴露给业务层）

技巧四：事务边界在应用层，不在领域层

@Transactional 属于持久化基础设施的概念， 理论上应放在 application service 层（协调层）， 而不应出现在 domain 对象上。 这样即使换掉 Spring 事务管理，domain 层无感知。

容易忽略的细节与面试高频考点

  • "数据库是细节"不代表数据库不重要。 架构上隔离数据库，并不是说可以随便选型——而是说数据库选型是一个可以被推迟、被替换的决策， 不应在项目初期就固化进业务代码。
  • Repository 接口不等于 DAO。 DAO（Data Access Object）是围绕表设计的，一张表一个 DAO； Repository 是围绕业务聚合根设计的，接口方法语义属于业务层。
  • CQRS 场景下可以适当松动。 查询侧（Query）往往性能要求高，允许直接用 SQL/JOOQ 穿透到数据层； 写侧（Command）才是需要严格隔离的业务核心。两侧规则不同。
  • Spring Data JPA 的 Repository 接口本质是框架接口。 继承 JpaRepository 的接口不应该直接注入进 Service， 而应包一层 Adapter——除非你确定这个项目永远不会换数据库且测试都用集成测试。
  • 面试常问："为什么要用 Repository 模式而不是直接用 JPA？" 标准答案就是本章主旨：隔离业务规则与持久化细节，使业务逻辑可独立测试、持久化可独立替换。

与其他知识点的联系

  • 依赖方向与控制流分离（上一篇）：本文是上一篇原理的一次完整实战——UserRepository 接口放在 domain 层、UserRepositoryAdapter 在 infrastructure 层实现它，正是"接口归属调用方、装配发生在组合根"这条原则的具体案例，验证了"数据库是细节"不是一句口号，而是可以用接口反转直接落地的工程手法。
  • 整洁架构同心圆：数据库处于同心圆最外层（Frameworks & Drivers），Entity/Use Case 处于最内层，本文的依赖方向图（UserService → UserRepository ← UserRepositoryAdapter）就是同心圆"依赖只能由外向内"规则在持久化场景下的具体投影。
  • 业务规则（Entity · Use Case）：文中反复强调 User（领域对象）不能等同于 UserEntity（JPA 映射对象），这与"业务规则"一文里 Entity 要"零框架依赖、能被直接 new 出来测试"的要求完全一致——JPA 的 @Entity 命名恰恰是最容易让人混淆两者的历史遗留问题。
  • DDD Repository 模式：本文明确区分了 Repository（面向业务聚合根设计）与 DAO（面向数据库表设计），这正是 DDD 中 Repository 模式的核心主张——接口方法要用领域语言命名，而不是暴露 SQL/JPQL 的技术细节。
  • Spring 框架实践：Spring Data JPA 的 JpaRepository 本身是一个框架接口，直接注入进 Service 会让业务层的可替换范围超出预期（继承来的 flush()、deleteInBatch() 等方法全部暴露）；本文技巧一提出先用 InMemoryUserRepository 跑通业务逻辑，是检验 Spring 项目里业务层是否真正独立于数据库的实用方法。
本章一句话总结：数据库只是把数据从内存搬到磁盘的 I/O 机制。 业务规则定义"数据是什么"，数据库决定"数据存在哪"。 前者是架构核心，后者是可替换的实现细节——用依赖反转让箭头指向正确的方向。
