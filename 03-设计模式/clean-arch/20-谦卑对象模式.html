20. 谦卑对象模式（Humble Object）

将"难以测试的行为"与"可测试的逻辑"强制分离，让边界天然成为可测试性的分水岭。

作者的核心主张

Bob 大叔在《架构整洁之道》中提出：架构边界天然就是测试边界。 Humble Object 模式是他用来解释这一主张的具体手法——这个模式本身并不新鲜， 最早由 Gerard Meszaros 在《xUnit Test Patterns》中归纳，Bob 大叔把它提升到了 架构层面来理解。

出发点是一个真实痛点：有些代码天生难以测试。 GUI 渲染、数据库驱动、消息队列消费者、定时器回调——它们依赖运行环境， 在单元测试里构造起来成本极高，甚至根本无法构造。

Humble Object 的做法很朴素：把这些对象"变谦卑"—— 剥掉里面一切业务判断、计算和决策，让它只剩下对外部世界的"机械调用"。 被剥走的那部分逻辑，搬进一个普通的、无副作用的对象里， 这个对象不依赖任何难以构造的外部设施，可以直接被单元测试覆盖。

一句话总结： 把"行为"从"机制"中分离出去。Humble Object 只负责机制（如何与外界交互）， 分离出去的那部分负责行为（做什么决策、计算什么结果）。

为什么这样设计代码更好

测试金字塔的现实困境

单元测试之所以快、稳定、便宜，是因为它不依赖 I/O。 一旦业务逻辑与 I/O 耦合在一起，你只剩下两个选择： 要么写昂贵的集成测试，要么放弃对这段逻辑的测试覆盖。 两者都是坏结果。

边界是天然的接缝

架构边界（UI 层 / 应用层 / 基础设施层之间的接缝）已经是代码的分割点。 Humble Object 的价值在于：它把可测试性的分割和架构边界的分割对齐在同一条线上。 这意味着你不需要为测试再额外引入抽象，边界本身就已经是接缝。

副作用被隔离到薄薄一层

Humble Object 越"谦卑"，副作用就越集中、越可控。 你不需要担心业务逻辑在某次数据库写入里偷偷修改了状态—— 因为业务逻辑根本不接触数据库。

反例 → 问题 → 正例

场景：订单列表页——把渲染逻辑塞进 Presenter

下面以一个 MVC 风格的 Web 应用为例。 前端 Presenter 需要把订单列表格式化后交给视图显示： 金额超过 1000 元显示红色警告标签，否则显示绿色正常标签。

反例 — OrderPresenter.java（业务判断与渲染机制混在一起）
// ❌ Presenter 直接依赖 HttpServletResponse，同时还做格式化判断
public class OrderPresenter {

    private final HttpServletResponse response;

    public OrderPresenter(HttpServletResponse response) {
        this.response = response;
    }

    public void present(List<Order> orders) throws IOException {
        StringBuilder sb = new StringBuilder();
        for (Order o : orders) {
            // 业务判断：金额阈值 1000
            String tag = o.getAmount().compareTo(new BigDecimal("1000")) > 0
                    ? "<span class='warn'>高额</span>"
                    : "<span class='ok'>正常</span>";
            sb.append("<li>").append(o.getId())
              .append(" - ").append(o.getAmount())
              .append(tag).append("</li>");
        }
        // 直接写入响应流——无法在单元测试中构造
        response.getWriter().write("<ul>" + sb + "</ul>");
    }
}
问题： 想测试"金额超 1000 是否显示高额标签"这条规则，必须先构造一个真实的 HttpServletResponse，或者用 Mock 框架模拟它，然后再从响应流里 解析 HTML。测试代码比被测代码还复杂，没有人愿意写。 结果：这段最核心的格式化判断，长期处于零覆盖状态。
正例 — 第一步：抽出 ViewModel（纯数据，无副作用）
// ✅ 只是一个普通的数据载体，零依赖
public class OrderViewModel {
    public final String id;
    public final String amountText;    // 已格式化为字符串
    public final String tagCssClass;  // "warn" 或 "ok"
    public final String tagLabel;     // "高额" 或 "正常"

    public OrderViewModel(String id, String amountText,
                           String tagCssClass, String tagLabel) {
        this.id          = id;
        this.amountText  = amountText;
        this.tagCssClass = tagCssClass;
        this.tagLabel    = tagLabel;
    }
}
正例 — 第二步：Formatter 承载全部判断逻辑（可纯单元测试）
// ✅ 这里放所有业务判断，不碰任何 I/O
public class OrderViewModelFormatter {

    private static final BigDecimal HIGH_AMOUNT_THRESHOLD = new BigDecimal("1000");

    public List<OrderViewModel> format(List<Order> orders) {
        return orders.stream()
            .map(this::toViewModel)
            .collect(Collectors.toList());
    }

    private OrderViewModel toViewModel(Order o) {
        boolean isHigh = o.getAmount().compareTo(HIGH_AMOUNT_THRESHOLD) > 0;
        return new OrderViewModel(
            o.getId(),
            o.getAmount().toPlainString(),
            isHigh ? "warn" : "ok",
            isHigh ? "高额" : "正常"
        );
    }
}
正例 — 第三步：Humble Presenter（谦卑对象，只做机械渲染）
// ✅ 这才是真正的"谦卑对象"——逻辑全被剥走，只剩机械的 HTML 拼接与写出
public class OrderPresenter {

    private final HttpServletResponse response;
    private final OrderViewModelFormatter formatter;

    public OrderPresenter(HttpServletResponse response,
                           OrderViewModelFormatter formatter) {
        this.response  = response;
        this.formatter = formatter;
    }

    public void present(List<Order> orders) throws IOException {
        // 1. 把判断委托给 Formatter（纯逻辑，可测）
        List<OrderViewModel> vms = formatter.format(orders);
        // 2. 自己只做没有判断的机械拼接
        StringBuilder sb = new StringBuilder();
        for (OrderViewModel vm : vms) {
            sb.append("<li>").append(vm.id)
              .append(" - ").append(vm.amountText)
              .append("<span class='").append(vm.tagCssClass)
              .append("'>").append(vm.tagLabel)
              .append("</span></li>");
        }
        response.getWriter().write("<ul>" + sb + "</ul>");
    }
}
正例 — 对 Formatter 的单元测试（极简，无任何 Mock）
@Test
void highAmountOrderShouldGetWarnTag() {
    OrderViewModelFormatter fmt = new OrderViewModelFormatter();
    Order order = new Order("ORD-001", new BigDecimal("1500"));

    List<OrderViewModel> result = fmt.format(List.of(order));

    assertEquals("warn", result.get(0).tagCssClass);
    assertEquals("高额",  result.get(0).tagLabel);
}

@Test
void normalAmountOrderShouldGetOkTag() {
    OrderViewModelFormatter fmt = new OrderViewModelFormatter();
    Order order = new Order("ORD-002", new BigDecimal("500"));

    List<OrderViewModel> result = fmt.format(List.of(order));

    assertEquals("ok",    result.get(0).tagCssClass);
    assertEquals("正常",  result.get(0).tagLabel);
}
分离后的效果： OrderPresenter 里已经没有任何 if，不需要测。 OrderViewModelFormatter 里全是纯函数，一行 Mock 都不用写， 测试极简、极快、极稳定。

模式结构：谁是"谦卑对象"，谁是"逻辑对象"

架构边界 │ ├── 难以测试的一侧（框架/外部设施） │ └── Humble Object（谦卑对象） │ ├── 只做机械调用：write response / send event / render frame │ └── 内部不含任何 if / 计算 / 格式化判断 │ └── 可以测试的一侧（核心逻辑） └── Logic Object（逻辑对象） ├── Formatter / UseCase / Interactor / Policy └── 只用纯粹的 Java 对象，无任何 I/O 依赖

常见落地场景

场景 Humble Object（谦卑侧） Logic Object（逻辑侧）
MVC Presenter 向 Response 写 HTML / JSON ViewModel 构建、格式化规则
GUI / Android Activity Activity / Fragment（调用 View API） Presenter / ViewModel（业务状态）
数据库 Repository JPA Repository 实现（SQL 执行） 领域对象、查询条件构造逻辑
消息队列 Consumer Listener（消费消息、提交 offset） 消息反序列化与业务处理逻辑
定时任务 @Scheduled 方法（触发入口） 实际业务执行逻辑（独立 Service）

实际开发落地技巧

1. 用"能否在没有 Spring Context 的情况下测试"作为判断标准

如果一个类必须依赖 @SpringBootTest 才能测， 说明它混入了不该混的东西。把业务判断剥离成一个无注解的普通 Java 类， 就可以用轻量的 JUnit 直接测。

2. ViewModel 是谦卑对象模式的最佳搭档

在 Spring MVC / React 架构下，Controller 只做参数绑定和结果写出， Formatter/Assembler 负责将领域对象映射成 ViewModel， ViewModel 是纯数据结构。这三层分工就是 Humble Object 模式的直接落地。

3. 谦卑对象的代码量应该极少

如果你的 Presenter / Listener / Controller 超过 30 行， 几乎可以断定里面还藏着业务逻辑没被剥离出去。 谦卑对象越小、越"无聊"越好——它就是一个纯粹的适配器。

4. 不要只在 Presenter 用，数据库层同样适用

Repository 实现类是另一个常见的 Humble Object。 SQL 的构造、查询条件的组合逻辑，应该提取到可测的 QueryBuilder 或 Specification 里，Repository 实现只做"执行 SQL、映射结果"。

5. 警惕"假谦卑"

有些团队会把业务逻辑放进私有方法，然后说"Presenter 只是调用私有方法"—— 这是假谦卑。谦卑的标准是：逻辑对象可以被独立实例化并直接测试， 而不是逻辑被藏在私有方法里只是看起来小。

与其他概念的关系

概念与 Humble Object 的关系
Clean Architecture 同心圆 Humble Object 通常位于最外圈（框架层），逻辑对象位于内圈
依赖倒置原则（DIP） 逻辑对象不直接依赖谦卑对象，通过接口解耦，进一步加强可测性
六边形架构（Ports & Adapters） Humble Object 对应 Adapter，逻辑对象对应 Port 背后的 Application Core
MVP 模式 View 是 Humble Object，Presenter 是 Logic Object
测试替身（Test Double） Humble Object 模式做好后，往往不再需要对核心逻辑使用 Mock

容易忽略的细节 / 面试高频考点

易错点 1： 认为"Humble Object 就是在 Controller 层套一层 Service"。 不对——重点不在于层数，而在于副作用是否被隔离。 Service 里如果直接调用 sendEmail() 同时又夹着业务条件判断， 它既不谦卑也不可测。
易错点 2： 把 ViewModel 设计成可变对象，然后在 Humble Object 里写入字段—— 这会把格式化逻辑悄悄带回 Humble Object 里。 ViewModel 应当在 Formatter 里一次性构建完毕并不可变。
面试角度： "谦卑对象模式解决了什么问题？" — 标准答案的核心是 可测试性和边界清晰两个词， 而不是"减少代码量"或"提高复用"。后两者是副产品，不是出发点。
