19. 尖叫架构（Screaming Architecture）

让目录结构大声喊出"这是什么系统"，而不是"这用了什么框架"。

核心主张：架构应该反映业务，而不是技术

Bob 大叔在书中问了一个很直白的问题：当你第一次打开一个陌生项目的代码目录，你能马上知道这是一个什么系统吗？

他把这个能力称为「尖叫」——优秀的架构应该对着你大喊：「这是一个医疗系统！」「这是一个电商平台！」 而不是悄悄嘀咕「这用了 Spring MVC」「这用了 Rails」。

原文引用：「架构应该在第一眼就告诉读者：这是一个医疗系统，不是一个 Rails 应用，也不是一个 Spring 应用。」
—— Robert C. Martin,《Clean Architecture》Chapter 21

这个主张看起来简单，背后藏着一个深刻的设计原则： 框架、数据库、HTTP、UI 都是工具，业务领域才是系统的本质。 目录结构是代码的第一张脸，它应该用业务概念命名，而不是用技术层次命名。

反例：按技术层次组织目录

下面是一个典型的"框架驱动"目录结构，Spring MVC 项目几乎人手一份：

src/main/java/com/example/
├── controller/
│ ├── UserController.java
│ ├── OrderController.java
│ └── ProductController.java
├── service/
│ ├── UserService.java
│ ├── OrderService.java
│ └── ProductService.java
├── repository/
│ ├── UserRepository.java
│ ├── OrderRepository.java
│ └── ProductRepository.java
└── model/
    ├── User.java
    ├── Order.java
    └── Product.java

这个结构有什么问题？

打开这个目录，你能知道这是什么业务吗？不能。你只知道它用了 MVC 分层。 controller/、service/、repository/ 这些名字描述的是技术角色，而不是业务。

  • 关注点分散： 想改"下单"功能，要同时跨越 controller、service、repository、model 四个目录
  • 隐藏了业务： 新同事看到目录，不知道系统做什么，要读代码才能猜出来
  • 框架绑架： 目录结构把 Spring MVC 的分层固化在了组织里，换框架就像拆房子
  • 高耦合低内聚： 同一个业务概念（Order）被切碎分散在四个包里，凝聚力弱
常见误区：很多团队把"按层分包"等同于"分层架构"，认为这就是遵守了架构规范。 实际上，分层架构描述的是依赖方向（业务层不依赖框架层），而不是目录怎么组织。两者可以完全正交。

正例：按业务领域组织目录

让架构尖叫的做法是以业务概念（领域/用例）为顶层划分：

src/main/java/com/example/
├── order/ ← 这里是"下单"的一切
│ ├── Order.java ← 实体
│ ├── OrderItem.java
│ ├── PlaceOrderUseCase.java ← 用例（核心业务逻辑）
│ ├── CancelOrderUseCase.java
│ ├── OrderRepository.java ← 接口（依赖反转）
│ └── infra/ ← 技术实现藏在最里层
│    ├── JpaOrderRepository.java
│    └── OrderController.java
├── product/
│ ├── Product.java
│ ├── SearchProductUseCase.java
│ └── infra/
│    └── ProductController.java
└── user/
    ├── User.java
    ├── RegisterUserUseCase.java
    └── infra/
        └── UserController.java

打开这个目录，你立刻知道：系统有订单、商品、用户三个核心业务。这就是尖叫架构—— 目录结构本身就是一份业务文档。

代码层面的对比

反例：业务逻辑渗入 Controller

Java — 反例：Controller 承担业务判断
@RestController
public class OrderController {

    @Autowired
    private OrderRepository orderRepo;

    @PostMapping("/orders")
    public ResponseEntity<Order> placeOrder(@RequestBody OrderRequest req) {
        // 业务规则混在 Controller 里——换个入口（定时任务、MQ）就要复制
        if (req.getItems().isEmpty())
            throw new BadRequestException("订单不能为空");
        if (req.getTotalAmount().compareTo(BigDecimal.ZERO) <= 0)
            throw new BadRequestException("金额非法");

        Order order = new Order(req);
        orderRepo.save(order);
        return ResponseEntity.ok(order);
    }
}

业务规则（订单不能为空、金额非法）住在 OrderController 里。 这段代码对框架的感知太强——它知道 HTTP 请求、ResponseEntity、Spring 注解。 一旦需要从消息队列触发下单，这段判断逻辑就得复制一份。

正例：业务核心与框架完全隔离

Java — 正例：用例类（纯 Java，无框架依赖）
// order/PlaceOrderUseCase.java — 不 import 任何 Spring 类
public class PlaceOrderUseCase {

    private final OrderRepository repo;   // 依赖接口，不依赖 JPA

    public PlaceOrderUseCase(OrderRepository repo) {
        this.repo = repo;
    }

    public Order execute(PlaceOrderCommand cmd) {
        // 业务规则住在这里，任何入口都能复用
        if (cmd.items().isEmpty())
            throw new DomainException("订单不能为空");
        if (cmd.totalAmount().compareTo(BigDecimal.ZERO) <= 0)
            throw new DomainException("金额非法");

        Order order = Order.create(cmd);
        repo.save(order);
        return order;
    }
}
Java — 正例：Controller 变成薄薄的适配器
// order/infra/OrderController.java — 只做协议转换
@RestController
public class OrderController {

    private final PlaceOrderUseCase placeOrder;

    public OrderController(PlaceOrderUseCase placeOrder) {
        this.placeOrder = placeOrder;
    }

    @PostMapping("/orders")
    public ResponseEntity<OrderResponse> place(@RequestBody OrderRequest req) {
        // Controller 不含业务判断，只负责：HTTP → Command → Response
        PlaceOrderCommand cmd = OrderRequest.toCommand(req);
        Order order = placeOrder.execute(cmd);
        return ResponseEntity.ok(OrderResponse.from(order));
    }
}

现在 PlaceOrderUseCase 是纯 Java 类，无任何框架注解。 要写单元测试？直接 new PlaceOrderUseCase(mockRepo)，不需要启动 Spring 容器。 要从 MQ 触发下单？写一个 MqOrderConsumer 调同一个用例，一行业务代码都不用改。

为什么这样设计更好

维度 按技术层分包 按业务领域分包（尖叫架构）
可读性 看目录不知道做什么 目录即业务文档
内聚性 同一业务散落四处 相关代码集中在一个包
可测试性 业务逻辑依赖框架，测试重 用例类纯 Java，单元测试极快
可替换性 换框架需要大面积重构 框架在最外层，业务核心不动
扩展性 新入口（MQ/CLI）需复制业务代码 新入口调同一用例，零重复

框架是工具，不是主人

Bob 大叔用一个类比来解释尖叫架构的精髓： 想象一个建筑师设计一栋住宅的蓝图。 蓝图应该尖叫着"这是一所住宅"——有卧室、厨房、客厅。 而不是尖叫"这用了钢结构"或"这用了混凝土浇灌"。 钢筋混凝土是建筑材料，是工具。住宅的用途才是设计的核心。

Spring、Rails、Django 都是框架，相当于建筑材料。它们是你用来构建业务的工具， 但不应该主导你的代码组织方式。当框架的约定侵入你的包结构、命名、乃至业务逻辑时， 你就被框架绑架了。

判断标准：删掉所有框架注解（@Controller、@Service…）， 核心业务代码还能编译运行吗？如果不能，说明业务已经被框架污染。

落地技巧与容易忽略的细节

技巧 1：顶层包用名词业务领域，不用技术层次

顶层包名应该是 order、inventory、payment， 而不是 controller、service、dao。 技术分层可以在领域包内部体现（如 order/infra/）， 但不要把它提升到顶层。

技巧 2：用例类用动词短语命名

用例代表"系统能做什么事"，用动词+名词命名最能表意： PlaceOrderUseCase、CancelOrderUseCase、RefundPaymentUseCase。 看到类名就知道是什么业务操作，不需要深入读代码。

技巧 3：接口定义在领域层，实现放 infra

OrderRepository（接口）属于业务领域，放在 order/ 包下。 JpaOrderRepository（实现）依赖 JPA，属于技术细节，放在 order/infra/。 这样业务层的依赖方向永远朝内，框架替换只需改 infra/。

技巧 4：不要让框架配置类污染业务包

SecurityConfig、WebMvcConfig、DataSourceConfig 这类纯框架配置类， 统一放在顶层 config/ 或 infra/config/ 包下，不要散落在各业务包里。 配置是横切关注点，不属于任何业务领域。

技巧 5：测试目录镜像领域结构

测试代码的包结构应与主代码保持一致： src/test/java/com/example/order/PlaceOrderUseCaseTest.java。 这让"找到某个功能的测试"变得直觉性，同时强化了"以业务为中心"的思维。

容易忽略的细节

  • 重构成本：存量项目从"按层分包"迁移到"按领域分包"需要大量移动文件。 建议先在新功能中实践，存量代码逐步迁移，不要一次性重构。
  • 模块内仍需分层：尖叫架构不是废除分层，只是把分层"沉入"业务模块内部。 每个业务包内的 entity、usecase、infra 子层仍要保持依赖方向。
  • 过度细分：小团队、小项目不必按每个实体都建一个顶层包。 可以先按业务域（catalog/、checkout/）聚合， 等规模增长再拆分。拆包的成本远低于后期重构。
  • 命名需要对齐领域语言：包名和类名最好与产品/业务方使用的术语一致（DDD 中的"统一语言"）。 技术人员说 UserOrderService，产品说"下单流程"——这种割裂会制造沟通成本。
高频误解："我们是 CRUD 系统，没有复杂业务，不需要尖叫架构。"
实际上越是简单的系统，越能从清晰的结构受益——新人上手更快，需求变更代价更小。 尖叫架构的核心是命名和组织原则，不是过度设计。

与整洁架构的关系

尖叫架构是整洁架构（Clean Architecture）的可见性体现。 整洁架构要求依赖方向从外向内（框架 → 应用层 → 领域层）， 尖叫架构要求这种设计意图在目录结构上直接可读。

两者的关系可以这样理解： 整洁架构是架构的骨骼（依赖方向、边界划分）， 尖叫架构是架构的皮肤（代码的第一印象、组织方式）。 有了整洁架构的骨骼，再让目录尖叫，这个系统才算真正对开发者友好。

与其他知识点的联系

  • 整洁架构同心圆（上一篇）：同心圆规定的是"依赖方向应该朝内"，尖叫架构规定的是"目录组织应该朝业务"——两者是正交的两条规则，同心圆做对了不代表目录就会尖叫，本文正例中 order/usecase/infra 的分包方式，正是把同心圆的分层"折叠"进了业务域包内部。
  • 依赖倒置原则（DIP）：尖叫架构能落地的前提是接口归属业务包（如 order/OrderRepository.java），实现放在 order/infra/——这依赖 DIP 提供的"内层定义接口、外层实现接口"手法，否则技术细节会反向侵入业务包，目录也就无法保持纯粹的业务视角。
  • DDD 统一语言：文中提到包名、类名要对齐产品/业务方的术语，这正是 DDD "统一语言（Ubiquitous Language）"的直接应用——尖叫架构可以看作统一语言在代码组织层面的强制落地，让目录结构本身成为业务文档。
  • Spring 框架实践：Spring Initializr 生成的默认脚手架（controller/service/repository）恰恰是本文反例的原型，是"框架驱动组织方式"最典型的例子；把顶层包从技术角色改为业务领域，是脱离框架默认约定、贯彻尖叫架构的第一步。
  • 谦卑对象模式（下一篇）：本文正例中 Controller 被压缩成"薄薄的适配器"、用例类保持纯 Java 可测试，这个"瘦入口 + 厚核心"的结构，正是下一篇谦卑对象模式要系统化阐述的可测试性分离手法。
