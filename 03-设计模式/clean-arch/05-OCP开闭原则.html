05. OCP — 开闭原则

Open-Closed Principle：对扩展开放，对修改关闭。Bob 大叔认为这是所有设计原则里最重要的一条。

原则定义

OCP 由 Bertrand Meyer 在 1988 年提出，Robert C. Martin 在《架构整洁之道》中将其推广到架构层面：

软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。
即：添加新行为时，应该通过新增代码实现，而不是修改已有代码。

这不是说代码永远不能改；而是说好的设计应当让"改需求"变成"加代码"，而不是"翻肠子"。 Bob 大叔给它贴的标签是："如果 OCP 被广泛使用，系统的架构就会变得非常灵活，维护成本也会大幅下降。"

为什么"不修改"反而更好？

直觉上觉得"改代码"天经地义，但修改已有代码有一个隐藏代价：已经通过测试的代码，一旦被修改就必须重新测试。改的越多，回归测试范围越大，风险越高。

更深层的原因在于依赖方向。如果高层模块依赖低层实现细节，每次细节变化都会向上传播，导致"改一处、动全身"。OCP 要求我们用抽象隔离变化：

  • 稳定的抽象（接口/抽象类）不变，对修改关闭
  • 易变的实现（具体类）可以无限扩展，对扩展开放
  • 高层策略只依赖抽象，永远不需要因低层细节变化而修改
一句话记忆：把变化点抽象出去，让调用方依赖抽象，让变化躲在抽象后面自己演化。

反例：用 if-else 承载变化

假设一个报表系统需要把财务数据导出为不同格式。最初只有 PDF，后来要加 Excel，再后来加 HTML……

Java · 反例 — ReportExporter.java
public class ReportExporter {

    // 每次新增格式都要来这里改 —— 违反 OCP
    public void export(ReportData data, String format) {
        if ("PDF".equals(format)) {
            // ... 生成 PDF 的 100 行代码
            generatePdf(data);
        } else if ("EXCEL".equals(format)) {
            // ... 生成 Excel 的 80 行代码
            generateExcel(data);
        } else if ("HTML".equals(format)) {
            generateHtml(data);
        }
        // 加 CSV？继续改这里，同时承担破坏 PDF/Excel 逻辑的风险
    }
}

问题清单：

  • 每次新增格式都必须修改 ReportExporter，测试覆盖范围随之扩大
  • 所有格式的逻辑挤在同一个类里，任何一处 bug 都可能影响其他格式
  • 如果 format 是用户输入的字符串，还存在"不识别格式时静默失败"的隐患
  • 随着格式增多，这个方法会变成一个无法维护的"上帝方法"

正例：用抽象承载变化

把"导出"这个行为抽象成接口，每种格式是一个独立实现。ReportExporter 只依赖接口，永远不需要改动。

Java · 正例 — 抽象接口
// 稳定的抽象 —— 对修改关闭
public interface ReportFormatter {
    byte[] format(ReportData data);
}
Java · 正例 — 具体实现（对扩展开放）
// 各格式独立演化，互不干扰
public class PdfFormatter implements ReportFormatter {
    @Override
    public byte[] format(ReportData data) {
        // 只关注 PDF 逻辑，不知道 Excel 存在
        return buildPdf(data);
    }
}

public class ExcelFormatter implements ReportFormatter {
    @Override
    public byte[] format(ReportData data) {
        return buildExcel(data);
    }
}

// 需要 CSV？新建一个类，其他代码零改动
public class CsvFormatter implements ReportFormatter {
    @Override
    public byte[] format(ReportData data) {
        return buildCsv(data);
    }
}
Java · 正例 — 高层调用方永远不变
// ReportExporter 只依赖抽象，从此不再需要修改
public class ReportExporter {

    private final ReportFormatter formatter;

    // 通过依赖注入传入具体实现（Spring / 手动注入均可）
    public ReportExporter(ReportFormatter formatter) {
        this.formatter = formatter;
    }

    public byte[] export(ReportData data) {
        return formatter.format(data);  // 不知道是哪种格式，也不需要知道
    }
}

现在新增任何格式，只需新建一个 XxxFormatter 类，注册到 IoC 容器或工厂即可。 ReportExporter、ReportFormatter 接口、已有的格式实现——全部零改动，全部无需重新测试。

架构层面的 OCP：保护高层策略

Bob 大叔在书中特别强调：OCP 不只是类级别的技巧，它是架构分层的核心动力。 关键洞察是：系统中的组件应该按"被修改的可能性"排列，形成单向依赖关系，越稳定的组件越靠近中心。

┌────────────────────────────────────────────────────┐
│ 业务策略层（最稳定） ← 永远不知道 UI / DB 怎么变 │
│ ↑ 依赖方向（箭头向内） │
│ 应用层（Use Cases） │
│ ↑ │
│ 接口适配层（Controllers / Presenters / Gateways） │
│ ↑ │
│ 框架与驱动层（最易变） Web框架 / ORM / 第三方 SDK │
└────────────────────────────────────────────────────┘

依赖永远从外层指向内层。外层（易变）依赖内层（稳定），而不是反过来。 这样外层无论怎么变化，内层的业务逻辑都对外层的修改关闭，不受波及。

常见误区：把框架放在核心位置。例如 Service 层直接依赖 HttpServletRequest 或 JPA Entity，一旦框架升级或替换，业务逻辑就必须跟着改。这正是违反 OCP 在架构层面的体现。

OCP 与其他原则的关联

原则 如何支撑 OCP
SRP 每个类只有一个变化原因，使得"新增格式"不会因为职责混杂而波及无关代码
LSP 子类必须可以替换父类，才能保证"扩展"时不破坏已有调用方
ISP 接口粒度合适，才能做到"只扩展我关心的那部分"，而不是被迫实现无关方法
DIP 依赖倒置是 OCP 的实现机制：高层依赖抽象，才能在不改高层的前提下替换低层实现

Bob 大叔的核心论断：DIP 是 OCP 的技术手段，OCP 是 DIP 的目标收益。两者缺一不可。

实际开发中的落地技巧

技巧 1：识别"变化轴"再动手

OCP 的前提是知道什么会变。不要一上来就过度抽象——等真的出现第二种实现时再提炼接口，这叫 "两次规则（Rule of Two）"：第一次出现变化就改代码；第二次出现同类变化时再抽象。 过早抽象会产生"为了 OCP 而 OCP"的虚假设计。

技巧 2：用策略模式承载行为变化

上面的报表例子本质就是策略模式。把算法族封装成接口，调用方持有接口引用，运行时注入具体策略。 Spring 的 @Autowired List<ReportFormatter> 可以自动收集所有实现，连注册代码都省了。

Java · Spring 自动收集所有实现
@Service
public class ReportService {

    // Spring 会把所有 ReportFormatter 实现注入进来
    private final Map<String, ReportFormatter> formatters;

    public ReportService(List<ReportFormatter> list) {
        // 假设每个实现都有 getName() 方法声明自己支持的格式
        this.formatters = list.stream()
            .collect(Collectors.toMap(ReportFormatter::getName, f -> f));
    }

    public byte[] export(ReportData data, String format) {
        ReportFormatter fmt = formatters.get(format);
        if (fmt == null) throw new IllegalArgumentException("不支持的格式: " + format);
        return fmt.format(data);
    }
}

// 新增格式：只需新建类 + @Component，零改动已有代码
@Component
public class CsvFormatter implements ReportFormatter {
    @Override public String getName() { return "CSV"; }
    @Override public byte[] format(ReportData d) { /* ... */ return null; }
}

技巧 3：用访问者模式处理数据结构扩展

OCP 有一个经典悖论：策略模式对"行为扩展"友好，但对"数据类型扩展"不友好（每加一种数据类型，所有策略都要改）。 反过来，访问者模式对"行为扩展"需要改代码，但对"数据类型扩展"封闭。 两者方向相反，选哪个取决于你预期哪个维度变化更频繁。

技巧 4：警惕"假扩展"

在接口里新加一个方法也是修改，会破坏所有已有实现。正确做法是：

  • 新建子接口继承旧接口，在子接口中声明新方法
  • 或者提供带默认实现的 default 方法（Java 8+），老实现无需改动
  • 在 Spring 等框架中，给接口加方法是影响范围极广的操作，务必谨慎

常见误区与面试考点

  • 误区：OCP = 禁止修改代码。 不是。修 bug、改错误的抽象，都是正当理由。OCP 说的是：对需求变化应优先通过扩展而非修改来应对。
  • 误区：每个类都要提炼接口。 过度抽象带来的类爆炸比 if-else 更难维护。OCP 要求的是在真实变化点上保持封闭，不是无脑接口化。
  • 面试高频：OCP 和 DIP 有什么关系？ DIP（依赖倒置）是实现 OCP 的手段——只有高层依赖抽象而非具体，才能在不改高层的前提下替换实现，OCP 才真正成立。
  • 面试高频：OCP 在微服务架构中如何体现？ 新增一个微服务（扩展），而不是修改已有服务的接口契约（关闭）。API 版本化（v1/v2 并存）也是 OCP 在接口设计上的体现。
核心记忆：OCP 的本质是用抽象管理变化方向。 找到真实的变化轴 → 在那里插入抽象 → 让高层策略永远只依赖那层抽象。 做到这一步，新需求就变成了"加文件"，而不是"改文件"。
