18. 整洁架构同心圆

依赖规则 · 四个圆层 · 边界穿越 · 真实代码落地

作者的核心主张：一条规则统治所有

Robert C. Martin（Uncle Bob）在《架构整洁之道》第 22 章提出了"整洁架构"（Clean Architecture）。 它并不是一套全新发明，而是对 Hexagonal Architecture、Onion Architecture、BCE 等思想的 综合与提炼。所有这些架构都指向同一个目标：

独立性四要素：框架无关 · 可测试 · UI 无关 · 数据库无关 · 任何外部代理无关。 业务规则不应该知道世界上存在数据库、Web 框架、消息队列这类东西。

实现这一目标的手段只有一条依赖规则（Dependency Rule）：

依赖规则：源代码的依赖关系只能由外向内指向，即外层依赖内层，内层对外层一无所知。 内层圆中的任何名字（类名、函数名、变量名）都不得出现在外层圆之外的代码里。

这一规则是整洁架构的灵魂。圆的形状只是隐喻，圈数也不是固定的；但"越靠内越稳定、越靠外越易变" 这个原则必须严格遵守。

四个同心圆各自负责什么

圆层 英文名 包含什么 变化频率
最内层 Entities 企业级业务规则：纯粹的 POJO / 领域对象，封装最通用的业务逻辑 极低（几年不变）
第二层 Use Cases 应用级业务规则：编排 Entity，描述系统"能做什么"，不关心 UI/DB 低（随需求变）
第三层 Interface Adapters Presenter、Controller、Gateway：把外部世界的格式转换为内层能理解的格式 中（随框架/协议变）
最外层 Frameworks & Drivers Spring MVC、JPA、React、消息队列等具体技术实现，细节在这里 高（技术迭代快）

场景：用户注册功能

我们有一个用户注册功能，需要校验邮箱唯一性并发送欢迎邮件。 看看两种写法的本质区别。

❌ 反例：Service 直接依赖框架/数据库细节

RegisterService.java（违反依赖规则）
// 问题：业务逻辑与 JPA、Spring、JavaMail 全部耦合在一起
@Service
public class RegisterService {

    @Autowired
    private UserJpaRepository userRepo;   // 直接依赖 JPA 仓库实现

    @Autowired
    private JavaMailSender mailSender;    // 直接依赖 Spring Mail

    public void register(String email, String password) {
        // 业务规则混在框架调用里，无法脱离 Spring 容器单独测试
        if (userRepo.existsByEmail(email)) {
            throw new RuntimeException("邮箱已注册");
        }
        UserEntity entity = new UserEntity();
        entity.setEmail(email);
        entity.setPassword(BCrypt.hashpw(password, BCrypt.gensalt()));
        userRepo.save(entity);

        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setTo(email);
        msg.setText("欢迎注册！");
        mailSender.send(msg);
    }
}

问题清单：

  • 测试必须启动 Spring 容器 + H2 数据库 + 邮件服务器，速度极慢
  • 换成 MongoDB 时，RegisterService 本身要改
  • 换成 SendGrid 发邮件时，业务层代码同样要改
  • 业务规则（邮箱唯一）被淹没在框架 API 调用里，难以阅读和审查

✅ 正例：用同心圆分层，依赖全部朝内

按整洁架构把代码分成四层，每层只知道它内侧圆的接口。

src/main/java/com/example/ ├── domain/ ← Entities（最内层） │ └── User.java ├── usecase/ ← Use Cases │ ├── RegisterUseCase.java │ ├── UserRepository.java ← 接口（端口） │ └── EmailNotifier.java ← 接口（端口） ├── adapter/ ← Interface Adapters │ ├── web/RegisterController.java │ ├── persistence/JpaUserRepository.java │ └── email/SmtpEmailNotifier.java └── framework/ ← Frameworks & Drivers └── SpringBootApp.java
domain/User.java — Entity（最内层，零依赖）
public class User {
    private final String email;
    private final String hashedPassword;

    // 业务规则：密码至少 8 位，在 Entity 内部强制
    public User(String email, String rawPassword) {
        if (rawPassword.length() < 8)
            throw new IllegalArgumentException("密码至少 8 位");
        this.email = email;
        this.hashedPassword = hash(rawPassword);
    }

    private String hash(String raw) {
        // 纯 Java，无框架依赖
        return Integer.toHexString(raw.hashCode());  // 演示简化
    }

    public String getEmail() { return email; }
    public String getHashedPassword() { return hashedPassword; }
}
usecase/UserRepository.java — 端口接口（Use Case 层定义，朝外打洞）
// Use Case 层定义接口，由外层实现 —— 这就是"依赖倒置"
public interface UserRepository {
    boolean existsByEmail(String email);
    void save(User user);
}
usecase/EmailNotifier.java — 端口接口
public interface EmailNotifier {
    void sendWelcome(String email);
}
usecase/RegisterUseCase.java — Use Case（只依赖 Entity + 自定义接口）
// 无任何 Spring 注解，纯 Java，可直接 new 来测试
public class RegisterUseCase {

    private final UserRepository users;
    private final EmailNotifier  notifier;

    public RegisterUseCase(UserRepository users, EmailNotifier notifier) {
        this.users    = users;
        this.notifier = notifier;
    }

    public void execute(String email, String password) {
        // 业务规则一眼看清：邮箱唯一校验
        if (users.existsByEmail(email))
            throw new IllegalStateException("邮箱已被注册");

        User user = new User(email, password);  // Entity 内部做密码校验 + 哈希
        users.save(user);
        notifier.sendWelcome(email);
    }
}
adapter/persistence/JpaUserRepository.java — 外层适配器（实现内层端口）
@Repository
public class JpaUserRepository implements UserRepository {

    @Autowired
    private SpringDataUserRepo jpa;  // Spring Data JPA 接口

    @Override
    public boolean existsByEmail(String email) {
        return jpa.existsByEmail(email);
    }

    @Override
    public void save(User user) {
        // 把 Domain User 转成 JPA UserEntity（防腐层职责）
        UserEntity entity = UserEntity.from(user);
        jpa.save(entity);
    }
}
RegisterUseCaseTest.java — 无框架的纯单元测试
class RegisterUseCaseTest {

    @Test
    void duplicateEmail_shouldThrow() {
        // 用 lambda 直接 mock，不需要 Mockito 或 Spring
        UserRepository fakeRepo    = new FakeUserRepository("existing@x.com");
        EmailNotifier  fakeNotifier = email -> {};  // 空实现

        RegisterUseCase uc = new RegisterUseCase(fakeRepo, fakeNotifier);

        assertThrows(
            IllegalStateException.class,
            () -> uc.execute("existing@x.com", "password123")
        );
    }
}
正例的收益：这个测试在毫秒级完成，不需要数据库、不需要 Spring 容器、不需要邮件服务器。 同时，换掉 JPA 改用 MongoDB 只需替换 JpaUserRepository 一个适配器类， RegisterUseCase 完全不动。

边界穿越：数据如何跨圆传递

依赖规则要求内层不能引用外层的类型。当控制流从外向内穿越边界时（Controller 调用 Use Case）， 数据该如何传递？答案是使用简单数据结构作为跨边界的 DTO（数据传输对象）。

Controller（外层） │ 调用 ▼ RegisterUseCase.execute(RegisterRequest) ← 简单 DTO，Use Case 层定义 │ 调用 ▼ UserRepository.save(User) ← Domain Entity，Use Case 层定义 │ 实现 ▼ JpaUserRepository（外层 Adapter 实现内层接口）

注意：RegisterRequest 由 Use Case 层定义，Controller 层使用它来传递数据。 这样 Controller 依赖的是 Use Case 层的类型，方向依然是由外向内——符合依赖规则。

常见错误：直接把 JPA 的 @Entity 类（外层）传给 Use Case（内层）处理。 这会让内层感知到数据库注解，违反依赖规则。应当在 Adapter 层做对象转换（mapping）。

DIP 是撑起同心圆的骨架

整洁架构的依赖规则本质上就是依赖倒置原则（DIP）的宏观应用。 当控制流需要从内层流向外层时（Use Case 调用数据库）， 通过在内层定义接口、外层实现接口，把依赖方向反转过来。

控制流方向：UseCase → JpaRepo（内 → 外，"违反"方向） ↓ 插入接口 依赖方向： UseCase → UserRepository（接口） ← JpaRepo 内层定义 外层实现

这正是"插件架构"的本质：框架和数据库只是可插拔的插件， 核心业务就是那个被插入的"底座"。删掉 Spring 换成 Quarkus， 底座（Entity + Use Case）一行不动。

实际落地技巧与容易踩的坑

技巧一：用包名强制圆层边界

在 Java 中没有语言级别的模块强制，但可以通过包结构+代码审查/ArchUnit 来约束。 推荐包命名规范如下：

  • com.example.domain — Entity 层，禁止引入 Spring / JPA
  • com.example.usecase — Use Case 层，只能引入 domain
  • com.example.adapter — Adapter 层，可引入框架，必须实现 usecase 接口
  • com.example.framework — 启动、配置，可引用所有层

技巧二：用 ArchUnit 自动化验证依赖方向

ArchitectureTest.java — 用 ArchUnit 阻止错误依赖
@AnalyzeClasses(packages = "com.example")
class ArchitectureTest {

    @ArchTest
    static final ArchRule domainMustNotDependOnAdapters =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat()
            .resideInAPackage("..adapter..");

    @ArchTest
    static final ArchRule usecaseMustNotDependOnFramework =
        noClasses().that().resideInAPackage("..usecase..")
            .should().dependOnClassesThat()
            .resideInAPackage("..framework..");
}

技巧三：圆层数量不必拘泥于四层

Uncle Bob 反复强调：四层只是示例，不是教条。 小项目两层（Domain + Infrastructure）就够；复杂系统可以再细分。 关键是"依赖方向始终朝内"这一规则绝不能妥协。

技巧四：Use Case 输入/输出要用独立 DTO，不用 Entity

Entity 是业务概念，不是 API 契约。把 Entity 直接暴露给 Controller 会导致两件事绑定： 业务对象变化 → API 响应格式随之变化，难以各自独立演进。 推荐为每个 Use Case 定义独立的 XxxRequest / XxxResponse 类。

容易忽略的细节

  • 异常类定义在哪层：业务异常（如 EmailAlreadyExistsException）应定义在 Use Case 层或 Domain 层，不要定义在 Adapter 层
  • 日志打在哪里：Domain 和 Use Case 不应直接调用 Logger 框架（会产生框架依赖），可通过事件/观察者模式或让 Adapter 负责日志
  • 事务注解位置：@Transactional 是框架关注点，应在 Adapter 层的实现类上标注，而非 Use Case 接口上
  • 配置属性：Use Case 所需的配置值（如最大重试次数）应通过构造函数注入，不要在 Use Case 内部读 @Value

与其他架构的关系

架构名称核心思想与整洁架构的关系
Hexagonal（端口与适配器） 应用核心通过"端口"与外界通信，适配器实现端口 整洁架构的 Use Case 接口 = 端口；Adapter = 适配器，思想完全相同
Onion Architecture 同心圆，Domain 在最内，Infrastructure 在最外 命名不同，依赖规则相同，整洁架构可视为其系统化表述
DDD 领域模型驱动，聚合根 / 实体 / 值对象 整洁架构的 Entity 层与 DDD Domain 层天然吻合，可组合使用
传统三层（Controller-Service-DAO） 按技术角色分层 依赖方向混乱（Service 依赖 DAO，但 DAO 是细节），违反依赖规则
一句话总结：整洁架构不是一套新框架，而是一套控制依赖方向的纪律。 同心圆是它的视觉化语言，依赖规则是它的唯一法则：依赖只能由外指内，细节依赖抽象，永远不反过来。

与其他知识点的联系

  • 业务规则（上一篇）：同心圆最内两层 Entities、Use Cases 就是"业务规则"一文的 Entity 和 Use Case，本文把它们放进一个统一的依赖模型里，说明它们和外层 Adapter、Framework 之间应该保持什么关系。
  • 依赖倒置原则（DIP）：依赖规则是 DIP 在架构尺度上的应用——当控制流需要从内层流向外层（Use Case 调用数据库）时，通过在内层定义接口、外层实现接口，把源码依赖方向反转回"由外向内"，这也是下一篇《依赖方向与控制流分离》要展开的机制。
  • 六边形架构 / DDD：本文表格已指出，Hexagonal 的"端口与适配器"、DDD 的领域模型分别与整洁架构的接口层、Entity 层一一对应——三者是同一套"隔离核心业务"思想在不同年代、不同术语下的变体，实践中完全可以互相借用词汇和落地手法。
  • Spring 框架实践：反例中 RegisterService 直接注入 UserJpaRepository、JavaMailSender，是 Spring 项目里最常见的架构腐化路径；正例用 @Repository/@Component 把框架注解限制在 Adapter 层，正是 Spring 场景下贯彻依赖规则的标准做法。
  • 尖叫架构（下一篇）：同心圆解决的是"依赖方向"问题，尖叫架构解决的是"目录组织"问题——二者正交但互补：即使依赖方向正确，如果顶层包名仍是 controller/service/repository，架构依然无法在目录上"尖叫"出业务领域。
