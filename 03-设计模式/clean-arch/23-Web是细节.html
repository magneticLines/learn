23. Web 是细节

GUI 是 I/O 设备；HTTP 是传输协议。这些都不是架构，只是交付机制。

作者的核心主张

Robert C. Martin 在本章开门见山地说：Web 是一个 I/O 设备。 浏览器、HTTP 协议、JSON 序列化、REST 风格——它们只是把数据从一台机器传输到另一台机器的管道。 这些东西在历史上反复翻新（CGI → Servlet → MVC → REST → GraphQL）， 而真正的业务逻辑一个字都不应该因此改变。

更深一层：Web 曾经不存在，将来也可能被某种新交互方式取代。 如果你的核心业务代码里充满了 HttpServletRequest、@RequestParam、 Response.json()，那么当交付方式改变时，你的整个系统就得推倒重来。

架构原则：把「做什么」（业务逻辑）和「怎么交付」（Web/CLI/gRPC/消息队列）严格隔离。 Web 层只做格式转换，不承载任何决策。

为什么这样设计更好

变更方向不同，不应耦合

业务规则因需求变化而改变；Web 层因协议/框架升级而改变。 两个变更原因不同的事物放在一起，就违反了单一职责原则（SRP）。 一次框架升级会迫使你重新测试所有业务逻辑，这是不必要的风险。

可测试性

如果业务用例不依赖 Web，就可以用普通单元测试在毫秒级验证所有业务分支， 无需启动容器、无需模拟 HTTP 请求、无需准备数据库。 测试金字塔的底座才是最稳固的。

可替换交付机制

同一套业务逻辑可以同时被 Web Controller、CLI 命令、消息消费者、gRPC 服务调用。 这不是假想场景——很多系统同时对外暴露 REST API 和内部 MQ 消费者，驱动的是同一批用例。

反例 → 问题 → 正例

反例：业务逻辑嵌入 Controller

Java · 反例：OrderController.java
@RestController
public class OrderController {

    @PostMapping("/orders")
    public ResponseEntity<String> placeOrder(@RequestBody OrderRequest req) {

        // ❌ 业务规则直接写在 Controller 里
        if (req.getQuantity() <= 0) {
            return ResponseEntity.badRequest().body("数量必须大于 0");
        }
        if (req.getPrice().compareTo(BigDecimal.ZERO) <= 0) {
            return ResponseEntity.badRequest().body("价格必须大于 0");
        }

        // ❌ 折扣计算混在 HTTP 层
        BigDecimal discount = req.getQuantity() > 100
            ? req.getPrice().multiply(new BigDecimal("0.9"))
            : req.getPrice();

        orderRepository.save(new Order(req.getProductId(), req.getQuantity(), discount));
        return ResponseEntity.ok("下单成功");
    }
}

这段代码的问题：

  • 业务规则（数量校验、折扣策略）被锁死在 HTTP 层，CLI 或消息队列触发时无法复用
  • 要测试折扣逻辑，必须构造 MockMvc、模拟 HTTP 请求，测试成本高
  • Spring MVC 升级或换成其他框架时，所有业务逻辑需要重新迁移
  • Controller 承担了多个职责：格式解析 + 业务校验 + 业务计算 + 持久化调用

正例：Web 层只做格式转换

按照整洁架构，系统应该分为四层：

┌─────────────────────────────────────┐ │ Web / Controller 层（框架细节） │ ← 只做 HTTP ↔ 用例输入的翻译 ├─────────────────────────────────────┤ │ Use Case / Application 层（业务流程） │ ← 核心，不依赖任何框架 ├─────────────────────────────────────┤ │ Domain / Entity 层（业务规则） │ ← 纯 Java，无任何依赖 ├─────────────────────────────────────┤ │ Infrastructure 层（DB / MQ / HTTP） │ ← 实现接口，可替换 └─────────────────────────────────────┘
Java · 正例：PlaceOrderUseCase.java（纯业务，不含任何 Web 概念）
// 用例输入：Plain Java Object，与 HTTP 无关
public record PlaceOrderInput(
    String productId,
    int quantity,
    BigDecimal unitPrice
) {}

public record PlaceOrderOutput(String orderId, BigDecimal finalPrice) {}

// 用例接口：由 Application 层定义，Controller 依赖接口而非实现
public interface PlaceOrderUseCase {
    PlaceOrderOutput execute(PlaceOrderInput input);
}

// 用例实现：所有业务规则在这里，没有 HttpServletRequest，没有注解
public class PlaceOrderUseCaseImpl implements PlaceOrderUseCase {

    private final OrderRepository orderRepository;

    public PlaceOrderUseCaseImpl(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @Override
    public PlaceOrderOutput execute(PlaceOrderInput input) {
        // ✅ 业务校验在用例层
        if (input.quantity() <= 0)
            throw new IllegalArgumentException("数量必须大于 0");
        if (input.unitPrice().compareTo(BigDecimal.ZERO) <= 0)
            throw new IllegalArgumentException("价格必须大于 0");

        // ✅ 折扣策略在用例层，可独立单元测试
        BigDecimal finalPrice = input.quantity() > 100
            ? input.unitPrice().multiply(new BigDecimal("0.9"))
            : input.unitPrice();

        Order order = new Order(input.productId(), input.quantity(), finalPrice);
        orderRepository.save(order);
        return new PlaceOrderOutput(order.getId(), finalPrice);
    }
}
Java · 正例：OrderController.java（只做格式转换）
@RestController
public class OrderController {

    private final PlaceOrderUseCase placeOrderUseCase;

    public OrderController(PlaceOrderUseCase placeOrderUseCase) {
        this.placeOrderUseCase = placeOrderUseCase;
    }

    @PostMapping("/orders")
    public ResponseEntity<OrderResponse> placeOrder(@RequestBody OrderRequest req) {
        // ✅ Controller 只负责：HTTP 请求 → 用例输入 的映射
        PlaceOrderInput input = new PlaceOrderInput(
            req.getProductId(),
            req.getQuantity(),
            req.getUnitPrice()
        );

        PlaceOrderOutput output = placeOrderUseCase.execute(input);

        // ✅ Controller 只负责：用例输出 → HTTP 响应 的映射
        return ResponseEntity.ok(new OrderResponse(output.orderId(), output.finalPrice()));
    }
}
Java · 单元测试：不需要 MockMvc，也不需要 Spring 容器
class PlaceOrderUseCaseTest {

    private final OrderRepository repo = mock(OrderRepository.class);
    private final PlaceOrderUseCase useCase = new PlaceOrderUseCaseImpl(repo);

    @Test
    void 批量超100件享九折() {
        PlaceOrderInput input = new PlaceOrderInput("P001", 150, new BigDecimal("100"));
        PlaceOrderOutput output = useCase.execute(input);
        // 纯 Java 断言，毫秒级，无需启动任何容器
        assertThat(output.finalPrice()).isEqualByComparingTo("90");
    }
}

同一用例被多种交付方式复用

隔离之后，同一个 PlaceOrderUseCase 可以被以下入口无缝调用：

HTTP POST /orders → OrderController → PlaceOrderUseCase CLI: java -jar place-order → OrderCommand → PlaceOrderUseCase Kafka consumer → OrderMessageHandler → PlaceOrderUseCase gRPC OrderService → OrderGrpcAdapter → PlaceOrderUseCase

每个入口只做格式转换，业务逻辑只写一次。 如果将来 REST 被废弃、换成 GraphQL，只需新增一个 Adapter，核心代码零改动。

异常处理：边界要双向隔离

隔离不只是"业务不依赖 Web"，反过来"Web 也不应暴露业务内部"。 常见的错误做法是在 Controller 里 catch IllegalArgumentException，然后直接返回 400。 正确方式是定义业务专用异常，由统一的 ExceptionHandler 负责翻译：

Java · 业务异常不携带任何 HTTP 概念
// 业务层：定义领域异常，不含 HTTP status code
public class InvalidOrderException extends RuntimeException {
    public InvalidOrderException(String message) { super(message); }
}

// Web 层：统一将业务异常翻译为 HTTP 响应，集中维护映射关系
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(InvalidOrderException.class)
    public ResponseEntity<ErrorResponse> handle(InvalidOrderException ex) {
        return ResponseEntity.badRequest().body(new ErrorResponse(ex.getMessage()));
    }
}
经验法则：业务层抛出的异常名字里不应出现 "Http"、"400"、"404" 这类词汇。 HTTP 状态码是 Web 层的概念，应在 ExceptionHandler 统一翻译。

前端同样是细节

Martin 在书中特别提到：Web GUI（浏览器里的 HTML/JS）也是细节。 后端 API 设计不应该被前端的展示需求所驱动。 如果后端接口因为某个页面的排版需求（例如"这个字段需要左对齐，所以帮我拆成两个字段"） 而反复改动，说明前端细节已经污染了后端设计。

BFF（Backend for Frontend）模式正是为此而生：在后端核心与前端之间加一层薄薄的适配层， 专门做字段裁剪和格式适配，让核心 API 保持稳定。

落地技巧与容易忽略的细节

场景易犯错误正确做法
校验逻辑 只用 @Valid / @NotNull 在 DTO 上做校验 格式校验放 DTO，业务规则校验（如库存是否充足）放用例层
分页参数 Pageable（Spring Data 对象）直接传入 Service 定义自己的 PageQuery(int page, int size)，Service 不依赖框架类型
响应 DTO 直接返回 Entity，Entity 含 @JsonIgnore 定义专用 Response DTO，Entity 不暴露给 Web 层
Session/Token 在 Service 中调用 SecurityContextHolder.getContext() Controller 解析出 userId 后作为参数传入用例，Service 无感 Spring Security
文件上传 Service 方法参数是 MultipartFile Controller 读取字节数组，传 byte[] 或 InputStream 给 Service
高频面试陷阱：「为什么 Spring 的 @Transactional 加在 private 方法上不生效？」 本质原因是事务代理基于 AOP，而 AOP 是 Web/框架层的基础设施机制。 你的业务方法（用例层）应当是 public 的，事务注解的行为才可预期。 这也是"Web/框架是细节"在底层的一个体现：当你把框架机制和业务方法混在一起时， 就会遇到这类让人困惑的框架副作用。

一句话总结

Web 是你的业务逻辑被交付到用户面前的方式之一，而不是业务逻辑本身。 Controller 只负责翻译，用例层承载决策，两者之间用 Plain Java Object 沟通， 依赖方向永远指向内部（业务），而非指向外部（框架）。

与上一章的联系：数据库是细节（第 22 章）强调持久化不该污染业务； 本章强调交付机制（Web）同样不该污染业务。两者共同构成整洁架构的核心边界： 业务核心对外界的任何技术细节保持无知。
