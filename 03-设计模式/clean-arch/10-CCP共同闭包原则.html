10. CCP — 共同闭包原则

Common Closure Principle：将因相同原因、在相同时间发生变化的类放在同一个组件中；将因不同原因或在不同时间变化的类放在不同组件中。

一、原则来源与核心主张

Robert C. Martin 在《架构整洁之道》第13章"组件内聚"中提出了三条原则：REP、CCP、CRP。 其中 CCP（Common Closure Principle，共同闭包原则） 是 SRP（单一职责原则）在组件层面的投影。

SRP 说：一个类不应该有多于一个被修改的理由。
CCP 说：一个组件不应该有多于一个被修改的理由——把同一类变更的代码集中在一起，把不同类变更的代码隔离开。

一句话记忆：把同时变化的东西放在一起，把不同时变化的东西分开。
就像办公室里的"项目文件"和"财务凭证"要分柜子放——混在一起，任何一方动了都要翻另一堆。

CCP 的终极目标是降低发布成本：当需求变更来临时，理想情况是只改动、测试、发布一个组件， 而不是同时牵动七八个模块、让整个团队都进入紧张状态。

二、为什么需要这个原则

软件系统的变更大多是局部的业务变更，而不是全面重写。CCP 帮我们把变更的"影响半径"控制在最小范围内。

违反 CCP 的后果 遵守 CCP 的收益
一个需求要改动 5 个组件，5 个组件都需要单独构建、测试、发布 一个需求只改动 1 个组件，构建 CI 只跑一条流水线
其他组件的使用方被强制升级依赖版本，即使与他们无关 不相关的使用方版本不变，零影响
变更在多个组件之间"跳跃"，排查问题像捉迷藏 变更有明确的"领地"，定位快、回滚范围小

三、反例 → 问题 → 正例

场景背景

你在开发一个电商系统，有如下功能：订单校验、优惠券折扣计算、发票生成。 某位工程师把这三部分放进了同一个 Maven 模块 order-core。

反例：把不同原因变化的类塞进同一组件

order-core 模块（违反 CCP）Java
// ❌ order-core 模块下混放了三类职责

package com.shop.ordercore;

// 职责一：订单合法性校验（随业务规则变化）
public class OrderValidator {
    public boolean validate(Order order) {
        return order.getItems().size() > 0
            && order.getTotalAmount().compareTo(BigDecimal.ZERO) > 0;
    }
}

// 职责二：优惠券折扣（随营销策略变化）
public class CouponDiscountCalculator {
    public BigDecimal apply(Order order, Coupon coupon) {
        return order.getTotalAmount().multiply(coupon.getRate());
    }
}

// 职责三：发票生成（随税务/财务规则变化）
public class InvoiceGenerator {
    public Invoice generate(Order order) {
        // 按国家税率生成发票，税务政策每年可能调整
        return new Invoice(order, TAX_RATE);
    }
}

问题在哪里

这三个类的变更驱动力来自完全不同的方向：

  • OrderValidator — 产品经理调整下单规则时修改（每季度可能改）
  • CouponDiscountCalculator — 运营搞活动/调折扣策略时修改（可能每周改）
  • InvoiceGenerator — 财务部门因税务政策调整时修改（一年改一次）

结果是：运营要改优惠券逻辑，却必须重新发布整个 order-core—— 哪怕 OrderValidator 和 InvoiceGenerator 完全没动过。 依赖 order-core 的其他微服务也要跟着升级依赖版本，进行回归测试。

常见后果："我只改了一行优惠券逻辑，怎么还要让财务模块重新上线？"——这就是 CCP 被违反时发生的事。

正例：按变更原因拆分为独立组件

order-validation/ ← 随下单业务规则变化 └── OrderValidator.java order-promotion/ ← 随营销/折扣策略变化 └── CouponDiscountCalculator.java order-invoice/ ← 随税务/财务规则变化 └── InvoiceGenerator.java
order-promotion 模块（遵守 CCP）Java
// ✅ order-promotion 只关心"折扣如何计算"这一个变化轴
// 它只依赖 Order 和 Coupon 的只读接口，不引入校验/发票的任何逻辑

package com.shop.promotion;

public class CouponDiscountCalculator {

    // 日后加"满减券"、"折扣上限"等营销规则，只改这一个组件
    public BigDecimal apply(OrderSnapshot order, Coupon coupon) {
        BigDecimal discounted = order.getTotalAmount().multiply(coupon.getRate());
        // 折扣不能超过上限（营销策略，只在本组件内控制）
        return discounted.min(coupon.getMaxDiscount());
    }
}

现在运营要改折扣上限逻辑，只需发布 order-promotion 这一个组件， order-validation 和 order-invoice 的使用方完全不受影响，无需任何回归测试。

四、CCP 与 OCP 的关系

CCP 的名字里有"闭包（Closure）"，这个词直接来自 OCP（开闭原则）： 对扩展开放、对修改关闭。

OCP 说：理想的类应该对某类变更"封闭"——变更来了不需要修改这个类，只需要扩展。 但完美的 OCP 只是理想，现实中总有些类需要被修改。 CCP 的思路是：把必须一起修改的类，收拢进同一个组件， 这样至少把修改的"副作用"（重新构建、重新发布）限制在最小范围内。

一个推论：如果两个类总是在同一次 PR 里被修改，那它们几乎一定应该放在同一个组件里。 代码审查中频繁出现"这两个文件又一起改了"，是提示你重新审视组件边界的信号。

五、CCP vs SRP：两个层次的同一思想

维度SRP（类层面）CCP（组件层面）
作用粒度 单个类 / 接口 JAR 包 / Maven 模块 / 微服务
核心问题 这个类有几个被修改的理由？ 这个组件有几个被修改的理由？
违反时的痛点 类膨胀，测试困难，职责耦合 组件频繁发布，级联影响其他团队
修复手段 拆分类，提取接口 拆分组件，重新划定模块边界

六、实际开发落地技巧

技巧 1：识别"变更轴"而不是"功能相关性"

初学者容易把"功能上相关"的类放在一起。比如 OrderValidator 和 CouponCalculator 都和订单有关，就放进了 order-core。 但 CCP 问的是谁来驱动这段代码改变：是产品、运营、财务、还是技术规范？ 驱动者不同 = 变更轴不同 = 应该放在不同组件。

技巧 2：用 Git 历史验证你的组件边界

运行一次 git log --name-only，观察哪些文件总是出现在同一次 commit 里。 如果 CouponCalculator.java 和 InvoiceGenerator.java 从来没有在同一次提交里出现过， 那它们不应该在同一个组件。反之，若两个文件几乎每次都一起改，则它们属于同一个"闭包"。

技巧 3：在微服务架构下，CCP 决定服务边界

微服务拆分不是按"技术层"（controller/service/dao 各一个服务），而是按"变更轴"。 把同一个业务能力中、同频率变化的逻辑放在同一个服务里， 不同变更频率的能力才考虑拆分。 否则一个业务需求要改三个服务，CCP 就被完全违反了。

技巧 4：警惕"工具类陷阱"

很多项目有一个 common 或 utils 模块，里面塞满了 StringUtil、DateUtil、JsonUtil…… 这些工具类几乎被所有模块依赖，但它们的变更原因各不相同。 一旦某个工具类被修改，所有依赖方都被迫升级——这是 CCP 的经典违反场景。 建议按领域拆分工具包，或者评估是否应该把工具逻辑内聚到对应的领域组件里。

判断口诀：改这个需求，我需要碰几个组件？如果答案大于 1，先问一句：这些组件是否应该合并？

七、常见误区与面试高频考点

误区 1：CCP 要求把所有相关代码放在一起

不对。CCP 说的是同一变更原因，不是功能相关性。 "登录"和"权限校验"功能上相关，但如果登录逻辑随用户体验迭代而变、权限逻辑随安全合规要求而变， 它们就应该在不同组件里。

误区 2：CCP 和 CRP 是一个意思

CCP 说"应该放在一起的放一起"，CRP 说"不该放在一起的不要放在一起"。 两者是张力的两端——CCP 倾向于聚合（减少组件数），CRP 倾向于分离（减少不必要的依赖）。 架构师的工作是在两者之间找到平衡点（这将在下一节 CRP 中展开）。

面试高频：CCP 和 SRP 有什么区别？

标准回答：SRP 是类级别的设计原则，CCP 是组件级别的设计原则，两者的核心思想相同—— 一个单元只应该有一个被修改的理由，区别在于"单元"的粒度。 CCP 在 SRP 的基础上回答了"类拆分后应该放进哪个包/模块"的问题。

小结

  • CCP 是 SRP 在组件层面的延伸：同一变更原因的类归同一组件，不同变更原因的类分属不同组件
  • 核心目标是将一次业务变更的影响半径控制在单个组件内，降低构建、测试、发布成本
  • 识别"变更轴"（谁驱动变化）比识别"功能相关性"更重要
  • Git 历史是验证组件边界是否合理的有效工具
  • CCP 与 CRP 构成张力：CCP 倾向聚合，CRP 倾向分离，两者共同约束组件内聚的最优边界
