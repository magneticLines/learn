17. 业务规则（Entity · Use Case）

软件最宝贵的资产不是 UI、不是数据库——是业务规则。把它们从框架和细节中解救出来，才是架构的核心使命。

一、为什么要单独谈业务规则

Robert C. Martin 的核心主张可以用一句话概括：业务规则是系统存在的理由，它们应该是最稳定、最纯粹的代码，不应依赖任何框架、数据库或 UI 细节。

想象你在给一家银行开发贷款系统。"贷款利息 = 本金 × 利率 × 时间"这条规则，无论你用 Spring Boot 还是 Django、无论存 MySQL 还是 MongoDB，无论界面是 Web 还是 App，它永远成立。它是关键业务规则（Critical Business Rule），是公司真正的护城河。

但在现实项目中，这些规则常常被淹没在 Controller、Service、SQL 甚至前端 JS 里。一旦需要换数据库或迁移框架，业务逻辑和技术细节纠缠在一起，改动牵一发而动全身。

架构的目标：让业务规则独立于技术细节存活。即使数据库挂掉，业务规则也应该可以被"在纸上跑通"。

二、两个核心概念：Entity 与 Use Case

Clean Architecture 将业务规则分成两层，分别用 Entity 和 Use Case 来承载。

概念中文职责变化驱动力
Entity 实体 封装企业级业务规则；不依赖任何用例，跨应用复用 企业核心业务策略变化（极少）
Use Case 用例 编排 Entity，实现应用级业务流程；只依赖 Entity，不依赖 UI/DB 具体应用需求变化（较少）

同心圆中的位置

┌──────────────────────────────────────────┐
│ 外层（Frameworks & Drivers / 易变） │
│ ┌────────────────────────────────────┐ │
│ │ Interface Adapters（Controller…） │ │
│ │ ┌──────────────────────────────┐ │ │
│ │ │ Use Cases（Application） │ │ │
│ │ │ ┌────────────────────────┐ │ │ │
│ │ │ │ Entities（Enterprise） │ │ │ │
│ │ │ └────────────────────────┘ │ │ │
│ │ └──────────────────────────────┘ │ │
│ └────────────────────────────────────┘ │
└──────────────────────────────────────────┘
依赖方向：只能由外向内，内层对外层一无所知

三、Entity：封装企业核心规则

Entity 不是 JPA 的 @Entity，不是数据库行映射对象！它是一个承载关键业务规则的普通对象（POJO），包含数据和操作这些数据的方法。

反例：规则泄漏到 Service 层

Java · 反例（业务规则散落在 Service）
// ❌ Loan 只是一个"哑巴"数据包，没有任何行为
public class Loan {
    public double principal;
    public double rate;
    public int    months;
}

// ❌ 业务规则被写在 Service 里，与 Spring、数据库注入混在一起
@Service
public class LoanService {
    @Autowired
    private LoanRepository repo;

    public double calcInterest(Long loanId) {
        Loan loan = repo.findById(loanId).orElseThrow();
        // 核心业务公式被埋在这里，换数据库或去掉 Spring 就找不到了
        return loan.principal * loan.rate * loan.months;
    }
}

问题所在：业务规则（利息计算公式）依附在 Spring 注解、Repository 注入的 Service 里。离开了 Spring 容器，这条规则就无法单独测试、无法复用到其他应用（比如批处理任务）。

正例：把规则还给 Entity

Java · 正例（Entity 拥有自己的行为）
// ✅ Entity：封装数据 + 关键业务规则，零依赖（无 @Entity、无框架注解）
public class Loan {
    private final double principal;  // 本金
    private final double rate;       // 月利率
    private final int    months;    // 期数

    public Loan(double principal, double rate, int months) {
        // 规则守卫：利率和期数必须合法，在对象创建时即刻验证
        if (rate <= 0 || months <= 0)
            throw new IllegalArgumentException("利率和期数必须为正数");
        this.principal = principal;
        this.rate = rate;
        this.months = months;
    }

    // ✅ 关键业务规则作为方法，随对象一起复用
    public double calcInterest() {
        return principal * rate * months;
    }

    // ✅ 另一条业务规则：等额还款月供
    public double monthlyPayment() {
        double r = rate;
        return principal * r * Math.pow(1 + r, months)
               / (Math.pow(1 + r, months) - 1);
    }
}

// ✅ 可以不依赖任何框架直接单元测试
class LoanTest {
    @Test
    void calcInterest_shouldReturnCorrectValue() {
        Loan loan = new Loan(10000, 0.005, 12);
        assertEquals(600.0, loan.calcInterest(), 0.001);
    }
}
Entity 的"纯粹"意味着：new 一个 Entity，不需要启动任何框架、连任何数据库。如果你的 Entity 需要 Spring 容器才能跑，它就被污染了。

四、Use Case：编排应用级业务流程

Use Case 描述的是"系统如何响应一个用户请求"。它调用一个或多个 Entity，并通过接口（端口）与外部打交道，但它自己不知道数据库的表结构，也不知道 HTTP 的存在。

Use Case 的标准结构

Java · Use Case 骨架
// 输入数据结构（Input Port / Request Model）——简单值对象，无行为
public record ApplyLoanRequest(
    String customerId,
    double principal,
    double monthlyRate,
    int    months
) {}

// 输出数据结构（Output Port / Response Model）
public record ApplyLoanResponse(
    String loanId,
    double totalInterest,
    double monthlyPayment
) {}

// 持久化接口（由外层 DB 适配器实现，Use Case 只依赖接口）
public interface LoanRepository {
    String save(Loan loan, String customerId);  // 返回生成的 ID
}

// ✅ Use Case 本体：编排流程，依赖 Entity + 接口，不依赖 Spring/JPA
public class ApplyLoanUseCase {

    private final LoanRepository loanRepo;   // 依赖倒置：依赖抽象

    public ApplyLoanUseCase(LoanRepository loanRepo) {
        this.loanRepo = loanRepo;
    }

    public ApplyLoanResponse execute(ApplyLoanRequest req) {
        // 1. 创建 Entity（Entity 自己验证规则）
        Loan loan = new Loan(req.principal(), req.monthlyRate(), req.months());

        // 2. 调用 Entity 的业务方法
        double interest = loan.calcInterest();
        double monthly  = loan.monthlyPayment();

        // 3. 通过接口持久化（具体是 MySQL/Redis 不关心）
        String loanId = loanRepo.save(loan, req.customerId());

        // 4. 返回结果给调用者（不关心调用者是 HTTP/消息队列/CLI）
        return new ApplyLoanResponse(loanId, interest, monthly);
    }
}
Use Case 的入参和出参用独立的 Request/Response Model 而不是 Entity 本身，原因是：
· Entity 对外暴露会导致外层可以随意修改内层数据
· 不同 Use Case 对同一 Entity 需要的字段可能不同
· 数据传输方向（内→外）与依赖方向（外→内）是反的，需要 Model 做缓冲

五、关键数据 vs 关键规则——一个容易混淆的细节

Martin 特别指出，Entity 中要区分两类东西：

  • 关键业务数据（Critical Business Data）：即使没有任何自动化系统，人工处理业务也必须记录的数据，如贷款本金、利率
  • 关键业务规则（Critical Business Rule）：与这些数据直接绑定的操作规则，如利息计算公式、还款约束

这两类东西天然地属于同一个 Entity。把数据和操作它的规则分开放，就是贫血模型的根源。

面试高频陷阱：面试官常问"贫血模型 vs 充血模型"。贫血模型（Anemic Domain Model）正是 Entity 退化成纯数据结构、业务规则流向 Service 的结果——Martin 认为这是反模式，会造成领域知识流失和可测试性下降。

六、Use Case 的边界：输入输出都不依赖外层

一个常见的错误是让 Use Case 直接接收 HTTP 的 HttpServletRequest 或返回 JPA 的 @Entity 对象。这让 Use Case 对外层产生了依赖，打破了边界。

Java · 正确接入方式（Controller 作适配器）
// Controller 是外层的"适配器"，负责把 HTTP 世界的数据翻译给 Use Case
@RestController
public class LoanController {

    private final ApplyLoanUseCase useCase;

    @PostMapping("/loans")
    public ResponseEntity<LoanVO> apply(@RequestBody LoanDTO dto) {
        // ① HTTP DTO → Use Case Request（适配）
        ApplyLoanRequest req = new ApplyLoanRequest(
            dto.customerId, dto.principal,
            dto.rate, dto.months
        );
        // ② 执行 Use Case（框架无关的纯业务）
        ApplyLoanResponse resp = useCase.execute(req);
        // ③ Use Case Response → HTTP VO（适配）
        return ResponseEntity.ok(new LoanVO(resp.loanId(), resp.totalInterest()));
    }
}

这三步"适配"看似啰嗦，实则是架构边界的物理体现——Controller 知道 HTTP，Use Case 不知道。

七、实际落地技巧

1. 包结构即架构

src/main/java/com/example/
├── domain/ ← Entity 层（零依赖）
│ ├── Loan.java
│ └── Customer.java
├── application/ ← Use Case 层（依赖 domain + 接口）
│ ├── ApplyLoanUseCase.java
│ └── port/
│ └── LoanRepository.java ← 接口（端口），由外层实现
├── infrastructure/ ← 外层（实现接口，依赖框架/DB）
│ └── JpaLoanRepository.java
└── interfaces/ ← 外层（HTTP / CLI / MQ）
└── LoanController.java

2. Entity 的测试不需要 Spring

Entity 和 Use Case 的测试应该是纯 JUnit，启动时间 < 1ms，无任何 @SpringBootTest。这是判断边界是否干净的最快检验方式。

3. Use Case 一个类只做一件事

不要写 LoanService 把申请贷款、还款、查询都塞在一起——那是传统三层架构的惯性。Clean Architecture 鼓励一个 Use Case 对应一个类，职责清晰，修改互不干扰。

4. 小心"假 Entity"

如果你的 Entity 只有 getter/setter、没有任何业务方法，它不是 Entity，它是 DTO。把业务方法还给它，或者老老实实承认它只是数据传输对象。

5. Request/Response Model 用不可变对象

Java 16+ 用 record，更低版本用 final 字段 + 全参构造器。不可变对象在多线程和流式处理中天然安全，也能防止 Use Case 被调用方偷偷修改入参。

一个实用的自检清单：
✓ Entity 能被 new 出来，不依赖任何注解
✓ Use Case 的构造函数只接受接口，不接受具体类
✓ Use Case 的测试文件里没有 import org.springframework.*
✓ Controller/Repository 不包含 if/else 业务判断，只做适配

八、与 DDD 的关系

Clean Architecture 的 Entity 与 DDD 的 Domain Entity / Value Object 理念高度重叠，Use Case 与 DDD 的 Application Service 对应。两者并不冲突，实践中常见的组合是：

Clean ArchitectureDDD 对应说明
EntityDomain Entity / Value Object携带业务规则的领域对象
Use CaseApplication Service编排领域对象，处理一个业务场景
LoanRepository（接口）Repository（接口）持久化抽象，属于 Domain 层
Controller / GatewayInfrastructure Adapter实现技术细节，依赖倒置指向内层

小结

  • Entity = 关键业务数据 + 关键业务规则，零框架依赖，是系统里最稳定的代码
  • Use Case = 应用级流程编排，只依赖 Entity 和接口，不知道 UI 和数据库的存在
  • 两者合力构成可独立测试、可独立演化的业务核心，是架构"内圆"的全部意义
  • 外层（框架、DB、HTTP）通过依赖倒置实现接口，内圆对外圆一无所知
