09. REP 复用 / 发布等同原则

Reuse/Release Equivalence Principle — 复用的粒度等于发布的粒度

原则核心主张

Robert C. Martin 在《架构整洁之道》第 13 章给出了这个原则的定义：

The granule of reuse is the granule of release.
复用的粒度就是发布的粒度。——只有经过正式发布流程管理的代码，才能被外部团队安全地复用。

翻译成人话：你打算让别人复用哪些代码，就把它们打成一个独立的发布单元（组件 / 包 / jar / npm 包）。 不要把"随手扔进同一个模块"的代码拿去复用，也不要把一个发布单元里的一半拿去用、另一半不用。

为什么这样设计更好

复用者的视角：我需要"稳定的承诺"

当你的团队依赖一段外部代码，你需要的不是"能跑起来的源码"，而是：

  • 清晰的版本号（我用的是 v2.3.1，不是"昨天下午的 main"）
  • 可预期的变更日志（升级到 v2.4 会破坏我什么接口？）
  • 明确的兼容承诺（同一 major 版本内向后兼容）

这些保障只有在代码经过正式发布流程管理时才存在。 没有发布流程的"直接复制源文件"，等于你在沙滩上盖楼——对方改一行代码你可能毫不知情。

提供者的视角：发布单元决定了维护边界

一旦你决定对外发布某个组件，你就对它做出了隐式承诺： 组件内的所有类/函数，都应该服务于同一个"可复用的主题"。 把不相关的代码混进同一个发布包，复用者就被迫引入他们根本不需要的东西—— 这制造了不必要的依赖，也给未来的升级带来摩擦。

REP 的反面教材：把日志工具、字符串工具、数据库连接池、业务 DTO 全部塞进一个 utils.jar 然后对外发布。复用者只需要字符串工具，却被迫依赖整个包，连同所有传递依赖。

代码示例：反例 → 问题 → 正例

反例：把不相关代码放进同一个"工具包"发布

假设团队把下面三样东西打进同一个 common-utils-1.0.jar：

Java — 反例：混杂的 common-utils 模块
// ① 字符串工具 —— 被 10 个微服务复用
public class StringUtils {
    public static String camelToSnake(String s) { ... }
    public static boolean isBlank(String s) { ... }
}

// ② 数据库连接池配置 —— 只有 order-service 使用
public class DbPoolConfig {
    private int maxPoolSize = 20;
    private int minIdle     = 5;
    // 依赖 HikariCP —— 传递给所有引用者
}

// ③ 业务 DTO —— 与具体业务强绑定，频繁变更
public class OrderCreateRequest {
    private Long   userId;
    private String productCode;
}

问题分析

当 order-service 需要修改 OrderCreateRequest 加一个字段，就必须发布新版本 common-utils-1.1.jar。

  • 所有 10 个微服务收到升级通知，即使它们只用 StringUtils，跟业务 DTO 毫无关系。
  • 各团队需要评估升级风险、跑回归测试、安排发版窗口——这就是"不相关的变更理由"带来的协作摩擦。
  • DbPoolConfig 把 HikariCP 作为传递依赖带给了所有引用者，哪怕他们用的是 Druid。
核心症结：发布粒度（整个 jar）远大于复用粒度（各团队各取所需），导致无关变更引发不必要的升级压力。

正例：按"复用主题"拆分成独立发布单元

common-string-utils ← 字符串工具，稳定，被 10 个服务复用
common-db-pool ← 连接池配置，只有需要它的服务引入
order-api-contract ← order-service 的 DTO / 接口定义，跟着业务变
Java — 正例：职责单一的发布单元（common-string-utils）
// 模块：common-string-utils
// 版本策略：语义化版本，patch 只修 bug，minor 向后兼容新增，major 才破坏性变更
// 依赖：零外部依赖，引入成本极低

public final class StringUtils {
    private StringUtils() {}

    public static String camelToSnake(String s) {
        // 只做字符串转换，不引入任何业务概念
        return s.replaceAll("([a-z])([A-Z])", "$1_$2").toLowerCase();
    }

    public static boolean isBlank(String s) {
        return s == null || s.isBlank();
    }
}
Maven pom.xml — 正例：各服务按需引入，版本独立管理
<!-- order-service 的 pom：只引入自己真正需要的包 -->
<dependencies>
    <!-- 字符串工具，零依赖，稳定 -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>common-string-utils</artifactId>
        <version>1.2.0</version>
    </dependency>

    <!-- order 接口契约，跟着业务演进独立发版 -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>order-api-contract</artifactId>
        <version>3.0.1</version>
    </dependency>
</dependencies>

现在 OrderCreateRequest 增加字段，只发布 order-api-contract-3.1.0。 其余 9 个只用字符串工具的服务完全不受影响，不需要评估、不需要测试、不需要发版。

发布流程的核心：语义化版本

REP 要求"正式发布"，而语义化版本（MAJOR.MINOR.PATCH）是实现这一承诺的行业标准：

版本段何时递增对复用者意味着
PATCH修 bug，无 API 变更可以安全升级，无需改代码
MINOR新增功能，向后兼容可选升级，享受新特性，旧代码不受影响
MAJOR破坏性变更需要主动适配，慎重评估升级时机
实践建议：在 CI 流水线中加入版本校验步骤——如果 API 签名发生破坏性变更，自动禁止合并，强制开发者提升 MAJOR 版本号。Java 生态可用 japicmp，npm 生态可用 publint 或 are-the-types-wrong。

REP 在"组件内聚三原则"中的位置

《架构整洁之道》第 13 章给出了三个组件内聚原则，REP 是第一个，也是其他两个的前提：

原则核心问题给组件施加的约束
REP（本节） 什么东西值得放在一起发布？ 组件内的类必须有同一个复用主题
CCP（下一节） 什么东西应该一起变更？ 同一原因变更的类放在一起，不同原因分开
CRP 什么东西不该放在一起？ 不总是被一起使用的类不应该放在同一组件

三个原则存在张力：REP 和 CCP 倾向于让组件变大（聚合更多相关内容），CRP 倾向于让组件变小（减少不必要的耦合）。 架构师的工作就是在这三者之间找到对当前团队最合适的平衡点。

实际开发落地技巧

1. 用"复用单位"反推组件边界

设计组件时先问自己：谁会用这个组件的哪些部分？ 如果两拨用户只各自用其中的一半，说明这个组件应该拆成两个。 一个好的检验方法：如果一个 issue 的标题自然地对应一个独立的组件名，组件划分就比较合理。

2. 发布前写 CHANGELOG

强迫自己在发布前写 CHANGELOG.md 是一个低成本的检验手段： 如果你发现一次发布里同时有"修了字符串 bug"和"更新了业务 DTO"两类无关条目， 这就是信号——这个组件的边界可能不够清晰。

3. 不要用 Git 子模块或直接复制源码替代正式发布

  • Git submodule：缺少版本承诺，子模块更新不可控，复用方随时可能"被更新"。
  • 直接复制源文件：产生代码分叉，bug 修复无法同步，完全违背 REP 的精神。
  • 正确做法：即使是内部库，也应发布到私有制品库（Nexus / Artifactory / GitHub Packages），用依赖管理工具引入，享受版本管理带来的一切保障。

4. 微服务 API 合约是天然的 REP 边界

在微服务架构中，服务间的接口契约（DTO、OpenAPI spec、Protobuf 定义）本身就是一个独立的"可复用发布单元"。 将它们从服务实现中分离出来单独发布（如 order-api-contract.jar）， 是 REP 原则在微服务场景下最直接的落地形式。

容易忽略的细节与陷阱

陷阱 1：把"功能相关"和"复用主题相同"混淆。
"都是 order 模块用的" 不等于 "应该在一个发布包里"。OrderValidator（稳定）和 OrderPromotionRule（频繁变更）功能相关，但它们的变更节奏不同， 应该由 CCP 而不是 REP 来决定是否放在一起。
陷阱 2：测试代码也是发布单元的一部分。
REP 不只约束生产代码。测试用例、测试 fixture 如果对外提供（例如 *-test-support.jar）， 同样需要遵循版本管理规范，否则复用方的测试会因为不可控的变更而莫名失败。
记忆口诀：你愿意为哪些代码写发版说明、管版本号、回答用户 issue，就把那些代码打成一个组件。 不愿意承担这个责任的代码，不要放进来；放进来的代码，都要承担这个责任。
