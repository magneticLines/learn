24. 框架是细节

框架很强大，但它不等于架构。把框架当架构，是软件腐烂最常见的根源之一。

作者的核心主张

Robert C. Martin 在《架构整洁之道》的「一切皆细节」章节中给出了一个让很多人不舒服的结论： 框架（Framework）只是实现细节，它不应该成为你系统的架构核心。

他的逻辑很清晰：架构的本质是对高层策略的决策——业务规则、用例、实体； 而框架提供的是 HTTP 路由、ORM 映射、依赖注入容器、任务调度……这些都是围绕核心业务的输入输出机制。 它们与"数据库是细节""Web 是细节"同属一类：它们服务于业务，业务不应该服务于它们。

一句话总结： 框架是工具，工具应该被架构使用，而不是定义架构的形状。

为什么框架会悄悄变成"架构"

框架作者和社区会鼓励你这样做：

  • 把业务对象直接继承框架的基类（如 Spring 的 JpaRepository、Rails 的 ActiveRecord::Base）
  • 在业务逻辑里直接调用框架的 API（如 @Autowired、HttpServletRequest）
  • 按框架规定的目录结构组织整个项目，而不是按业务领域
  • 用框架注解驱动业务对象的生命周期（如 @Entity、@Service）

这些做法让上手变得极快，Quick Start 五分钟就能跑起来。 但代价是：你的业务代码与框架深度耦合，业务规则开始依赖框架的生命周期和约束。

危险信号： 当你想对一个"纯业务逻辑"的类写单元测试，却发现必须启动 Spring 容器 / 连接数据库才能运行时， 说明业务层已经被框架污染了。

反例 → 问题 → 正例

场景：用户注册后发送欢迎邮件

反例 — 业务逻辑与 Spring 深度耦合Java
// 反例：业务类直接依赖框架注解和基础设施
@Service                          // 业务类变成了 Spring Bean
public class UserService {

    @Autowired                    // 直接依赖 Spring 容器注入
    private UserRepository repo;   // 继承自 JpaRepository

    @Autowired
    private JavaMailSender mailSender;  // Spring 邮件 API

    @Transactional                // 业务规则与事务注解混在一起
    public void register(String email, String password) {
        UserEntity user = new UserEntity();  // @Entity 标注的 JPA 实体
        user.setEmail(email);
        user.setPassword(BCrypt.hashpw(password, BCrypt.gensalt()));
        repo.save(user);

        SimpleMailMessage msg = new SimpleMailMessage();  // Spring Mail 对象
        msg.setTo(email);
        msg.setSubject("欢迎注册");
        mailSender.send(msg);
    }
}

问题出在哪里

  • 测试 register() 的业务规则（密码加密、邮件地址校验），必须启动 Spring 容器 + 数据库 + 邮件服务器，测试成本极高。
  • UserEntity 同时承担了"领域模型"和"JPA 映射表"两个职责，业务逻辑被 ORM 约束（必须有无参构造、字段不能是 final 等）。
  • 如果将来要从 Spring Mail 换成 AWS SES，需要修改业务类本身，违反开闭原则。
  • 框架升级时，@Transactional 语义的细微变化可能悄悄破坏业务逻辑。
正例 — 框架被推向边界，业务核心保持纯粹Java
// === 领域层（纯 POJO，零框架依赖）===
public class User {
    private final String email;
    private final String hashedPassword;

    // 工厂方法封装密码加密规则——这是业务规则，不是框架规则
    public static User register(String email, String rawPassword) {
        validateEmail(email);
        return new User(email, hash(rawPassword));
    }
    // ... getters, equals, hashCode — 纯 Java
}

// === 用例层（定义接口，不依赖具体框架）===
public interface UserRepository {          // 自己定义，不继承 JpaRepository
    void save(User user);
}

public interface EmailGateway {            // 自己定义，不依赖 JavaMailSender
    void sendWelcome(String email);
}

public class RegisterUseCase {            // 普通 Java 类，无任何注解
    private final UserRepository users;
    private final EmailGateway email;

    public RegisterUseCase(UserRepository users, EmailGateway email) {
        // 构造器注入——不需要 Spring，测试时直接 new
        this.users = users;
        this.email = email;
    }

    public void execute(String emailAddr, String password) {
        User user = User.register(emailAddr, password);
        users.save(user);
        email.sendWelcome(emailAddr);
    }
}

// === 框架适配层（框架细节被隔离在此）===
@Repository                               // Spring 注解只出现在适配层
public class JpaUserRepository implements UserRepository {
    @Autowired
    private UserJpaDao dao;              // Spring Data JPA 在这里

    @Override
    public void save(User user) {
        dao.save(UserJpaEntity.from(user));  // 领域对象 → JPA 实体的映射
    }
}

@Component
public class SpringEmailGateway implements EmailGateway {
    @Autowired
    private JavaMailSender sender;      // Spring Mail 细节封装在此

    @Override
    public void sendWelcome(String email) {
        // ... 构建 SimpleMailMessage 并发送
    }
}
正例的核心变化： RegisterUseCase 是一个可以用 new RegisterUseCase(mockRepo, mockEmail) 直接实例化的普通类， 无需任何框架基础设施即可测试全部业务规则。框架只出现在最外层的适配器里。

依赖方向：框架永远指向业务，而非反过来

┌─────────────────────────────────────────────┐ │ 框架 / 基础设施层 │ │ Spring Boot · JPA · Mail · Web · Scheduler │ │ │ │ JpaUserRepository SpringEmailGateway │ │ UserController @SpringBootApplication │ └──────────────┬──────────────────────────────┘ │ implements / calls（依赖箭头向内） ▼ ┌─────────────────────────────────────────────┐ │ 用例层（Use Cases） │ │ │ │ RegisterUseCase UserRepository(接口) │ │ ChangePasswordUseCase EmailGateway(接口) │ └──────────────┬──────────────────────────────┘ │ uses ▼ ┌─────────────────────────────────────────────┐ │ 领域层（Entities） │ │ │ │ User Order Product — 纯 POJO，零依赖 │ └─────────────────────────────────────────────┘

依赖箭头始终由外向内。框架层依赖接口（由用例层定义），用例层依赖领域层。 内层对框架一无所知——这才是"框架是细节"的结构化体现。

框架 vs 架构：一张对比表

维度 框架 架构
定义 解决通用技术问题的工具集 对系统高层结构和关键决策的描述
关注点 HTTP 路由、ORM、DI、任务调度 业务规则、用例、实体、边界
更换成本 可以被替换（Spring → Micronaut） 核心架构决策改变代价极高
业务知识 不包含任何业务知识 沉淀了领域核心知识
谁服务谁 框架服务于业务 架构约束框架的使用方式
测试依赖 测试框架代码需要框架环境 测试业务规则不需要任何框架

实际开发落地技巧

1. 用"框架自由测试"检验隔离程度

对每一个用例类写一个纯 JUnit 测试（不用 @SpringBootTest、不用 @DataJpaTest）。 如果能跑通，说明业务逻辑与框架成功隔离；如果报错，说明还有框架污染。 把这个测试当成架构守门员。

2. 接口定义权归属用例层，而非框架层

UserRepository 接口应该在 domain 或 usecase 包下，由业务需要定义方法签名， 而不是继承 JpaRepository<UserEntity, Long> 再暴露给业务层。 后者把 JPA 特有的方法（findAll、flush、deleteInBatch）全暴露给了业务，业务层看到的接口超出了它的需要。

3. 领域对象和持久化对象分开

User（领域对象）和 UserJpaEntity（JPA 映射对象）应该是两个类。 这虽然多了映射代码，但换来了领域对象的纯粹性： 字段可以是 final，可以没有无参构造，可以用值对象而不是 String 表达业务概念。

4. 框架注解只出现在适配层

一个简单的自检规则：在 domain/ 和 usecase/ 包下搜索 @Autowired、@Service、@Entity、@Transactional、HttpServletRequest。 如果有命中，就是需要重构的信号。

5. 框架配置本身也是细节

application.yml、@Configuration 类、@Bean 工厂方法——这些都是框架的组装胶水， 属于最外层的圆。不要在这些配置里写业务判断逻辑，那是职责错位。

常见误区与面试高频考点

  • 误区："用了整洁架构，Spring 注解就不能用了" —— 错。@Bean、@Configuration、@Controller 出现在外层是完全正常的。 问题在于注解渗透进了领域层和用例层。
  • 误区："框架生成的脚手架就是架构" —— spring-boot-starter 的默认包结构（controller/service/repository）是按技术分层， 不是按业务用例分层。它只是一个起点，不是架构蓝图。
  • 面试高频：如何在 Spring 项目里实践依赖倒置原则？ —— 答：在用例层定义接口，在框架/基础设施层实现，让 Spring 容器负责把实现注入进来， 而用例层本身无需 @Autowired，靠构造器参数接收依赖。
  • 面试高频：领域对象和 JPA 实体为什么要分开？ —— 答：JPA 对对象有侵入性约束（无参构造、非 final 字段、懒加载代理等）， 而领域对象应该遵循业务规则约束；两者目标不同，合并会导致互相妥协，业务模型被基础设施污染。
记住这条原则： 你可以在没有框架的情况下，仅凭核心业务类的代码，向非技术人员讲清楚系统做了什么—— 如果做不到，框架已经成为架构的一部分了，而不是细节。
