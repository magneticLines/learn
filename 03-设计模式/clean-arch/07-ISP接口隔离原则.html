07. ISP — 接口隔离原则

Interface Segregation Principle · 不要强迫使用者依赖他们不需要的东西

原则一句话

任何使用者都不应该被迫依赖它用不上的方法。 换言之：接口应该小而专，只包含特定使用者真正需要的那组操作。

Bob Martin 的原话："Clients should not be forced to depend upon interfaces that they do not use." 这里的"依赖"不只指调用，还包括编译依赖——只要你 import/实现了一个包含多余方法的接口，变动就会传染到你。

为什么这么设计：脂肪接口的代价

想象一个"万能遥控器"接口，上面有 40 个按钮，却把电视、空调、风扇的所有功能全塞在一起。 你只想控制电视音量，却不得不拿着那把覆盖所有功能的遥控器—— 改了空调的按钮逻辑，你的电视代码也得重新编译，甚至可能出现意料之外的副作用。

在代码层面，"脂肪接口"（Fat Interface）带来三个具体伤害：

  1. 不必要的重新编译：接口新增/修改一个方法，所有实现类和调用方都受牵连，哪怕他们根本不用那个方法。
  2. 测试变难：Mock 一个 20 个方法的接口，你要 stub 掉全部，或者用空实现到处放 throws UnsupportedOperationException，噪音极大。
  3. 强迫空实现：实现类被迫提供它没有能力实现的方法，通常以抛异常或返回 null 收场，违反了 LSP。

代码示例：反例 → 问题 → 正例

场景：多功能打印机系统

系统中有三种设备：复合打印机（打印 + 扫描 + 传真）、 普通打印机（只能打印）、 扫描仪（只能扫描）。

❌ 反例：脂肪接口

Java · 脂肪接口
// 一个接口把所有功能都塞进来
public interface MultiFunctionDevice {
    void print(Document doc);
    void scan(Document doc);
    void fax(Document doc);
}

// 普通打印机：被迫实现 scan / fax
public class SimplePrinter implements MultiFunctionDevice {

    @Override
    public void print(Document doc) {
        System.out.println("打印: " + doc.getContent());
    }

    @Override
    public void scan(Document doc) {
        // 普通打印机没有扫描功能，只能抛异常
        throw new UnsupportedOperationException("不支持扫描");
    }

    @Override
    public void fax(Document doc) {
        // 同上，被迫的空行为
        throw new UnsupportedOperationException("不支持传真");
    }
}
问题清单：
1. SimplePrinter 依赖了 scan/fax 的签名——接口一旦修改这两个方法，它也要重新编译。
2. UnsupportedOperationException 是 LSP 违规的经典信号：调用方无法安全地多态替换。
3. 为 MultiFunctionDevice 写单元测试时，必须 stub 3 个方法，哪怕测试只关心打印。

✅ 正例：拆分为小接口

Java · 隔离后的接口
// 按能力拆分：每个接口只描述一种职责
public interface Printable {
    void print(Document doc);
}

public interface Scannable {
    void scan(Document doc);
}

public interface Faxable {
    void fax(Document doc);
}

// 复合打印机：真正拥有全部能力，才实现全部接口
public class MultiFunctionPrinter implements Printable, Scannable, Faxable {
    @Override public void print(Document doc) { /* 打印逻辑 */ }
    @Override public void scan(Document doc)  { /* 扫描逻辑 */ }
    @Override public void fax(Document doc)   { /* 传真逻辑 */ }
}

// 普通打印机：只实现它实际具备的能力
public class SimplePrinter implements Printable {
    @Override
    public void print(Document doc) {
        System.out.println("打印: " + doc.getContent());
    }
    // 没有多余实现，没有 throw，干净
}

// 调用方只依赖自己真正需要的接口
public class PrintService {
    // 只注入 Printable，不关心扫描/传真是否存在
    private final Printable printer;

    public PrintService(Printable printer) {
        this.printer = printer;
    }

    public void printReport(Document doc) {
        printer.print(doc);
    }
}
现在 Faxable 接口改变，SimplePrinter 和 PrintService 的编译单元完全不受影响。 这正是 ISP 追求的：变更的涟漪被接口边界截断，只扩散到真正关心它的人。

架构视角：ISP 不只是"接口拆分"

Bob Martin 在书中强调，ISP 的本质是一条关于依赖方向的原则，而不仅仅是 Java 接口的写法技巧。 他举了一个更宏观的例子：如果系统 A 依赖框架 F，而框架 F 又依赖数据库 D， 哪怕 A 根本不用到 D 的功能，只要 D 的版本变动，A 也可能受影响——这是模块级别的 ISP 问题。

// 脂肪依赖：A 被迫依赖它不需要的 D System A ──依赖──▶ Framework F ──依赖──▶ Database D (包含大量 DB 相关代码) // 隔离后：F 对外只暴露 A 真正用到的接口 System A ──依赖──▶ <<interface>> FeaturePort ◀──实现── Framework F └─(内部依赖 D，对 A 不可见)

这个思路在微服务、插件系统、SDK 设计中都极其常见： 对外暴露最小接口，隐藏内部依赖，让使用者的编译依赖图尽可能窄。

与其他 SOLID 原则的关联

原则与 ISP 的关系
SRP SRP 说"一个模块只有一个变化原因"；ISP 说"使用者只依赖他需要的部分"。两者都在对抗耦合，方向互补。
LSP 脂肪接口经常导致 UnsupportedOperationException，违反 LSP。ISP 是 LSP 的前置保障。
DIP DIP 要求依赖抽象；ISP 要求抽象要小且聚焦。两者配合才能实现真正解耦：小接口 + 依赖注入。

实际开发落地技巧

1. 以"调用方视角"而非"实现方视角"设计接口

问自己："这个接口的每一个方法，同一个调用方真的会同时用到吗？" 如果答案是否，就该拆分。不要因为实现类恰好支持所有功能就把所有方法放在一个接口里。

2. 接口组合优于继承层次

Java 支持一个类实现多个接口。与其设计一个大接口，不如设计多个小接口， 让能力强的类通过"实现多接口"来组合能力，调用方按需依赖其中一个。

3. 用 default 方法谨慎扩展，不要滥用

Java 8 的 default 方法让接口可以有默认实现，有时被用来"向旧接口加方法而不破坏现有实现"。 但这本质上是在变相加宽接口。若大量使用，会逐渐退化回脂肪接口的问题。 新增功能优先考虑新建接口，而不是在旧接口加 default 方法。

4. 测试是接口合理性的最好检验

为某个类写单元测试时，如果需要 Mock 超过 3~4 个方法，接口很可能太胖了。 精简的接口让 Mock 变得简单：Mockito.mock(Printable.class) 只有一个方法要打桩。

5. Spring 中的实践

Spring 自身大量遵循 ISP。例如 ApplicationContext 虽然功能丰富， 但它实现了多个小接口（BeanFactory、ResourceLoader、ApplicationEventPublisher 等）。 业务代码中注入依赖时，应尽量注入最小接口，而不是直接注入 ApplicationContext—— 后者等于让业务类对整个容器产生了宽泛依赖。

Java · Spring 中正确示范
// ❌ 注入整个上下文：依赖了 N 个你用不到的方法
@Service
public class NotificationService {
    @Autowired
    private ApplicationContext ctx;  // 过宽依赖

    public void send(String msg) {
        ctx.publishEvent(new NotificationEvent(msg));
    }
}

// ✅ 只注入真正用到的最小接口
@Service
public class NotificationService {
    @Autowired
    private ApplicationEventPublisher publisher;  // 最小依赖

    public void send(String msg) {
        publisher.publishEvent(new NotificationEvent(msg));
    }
}

容易忽略的细节与误区

  • 误区：接口越小越好——走极端把每个方法拆成一个接口，反而造成接口爆炸，类要实现几十个接口，可读性和维护性变差。 粒度应以"同一调用方会同时用到"为准，而不是"越小越好"。
  • 误区：ISP 只适用于接口——抽象类、模块、包、甚至微服务的 API 设计都适用 ISP 思想。 一个提供了 50 个端点的 REST 服务，如果大多数消费者只用其中 3 个，就该考虑拆分或 BFF（Backend For Frontend）。
  • 细节：接口拆分后要注意命名语义——接口名应该描述"能力"而非"实现"， 例如 Printable、Exportable、Auditable，而不是 PrinterInterface。
  • 面试高频：ISP 与 SRP 的区别——SRP 针对模块/类（变化原因），ISP 针对接口/调用方（依赖范围）。 两者相辅相成，都在减小变更的影响范围。

小结

ISP 的核心洞见是：依赖本身就是耦合。 就算你只用了接口里的一个方法，只要你依赖了这个接口，接口的其他变化也会波及你。 因此，设计接口时要站在调用方的立场，按"使用者真正需要的能力集合"来划定边界， 让变更的涟漪只扩散到真正关心它的代码，其余部分保持稳定。

一句话记住：接口是给使用者用的，按使用者的需求切割，而不是按实现者的方便堆砌。
