HTTP 协议的演进

HTTP 从 1991 年一个只有一行请求的简陋协议，演进到今天承载整个互联网的复杂协议，每一代都是在解决上一代暴露出的真实痛点——这条时间线是理解现代 Web 性能优化的钥匙。

HTTP/0.9（1991）：够用就行的原型

最早的 HTTP 极度简陋：客户端只能发一行请求，比如 GET /index.html，服务器直接把 HTML 文本流回来，然后立刻关闭连接。没有请求头、没有状态码、没有其他方法（POST、PUT 都不存在），只能传 HTML，连图片都传不了。

这在只想"共享科研文档"的场景下够用，但互联网一旦要承载更丰富的内容（图片、脚本、错误提示），这套协议立刻捉襟见肘。

HTTP/1.0（1996）：补上元数据

HTTP/1.0 解决的核心问题是"只能传纯文本、无法描述内容"，引入了三个关键机制：

  • 请求头/响应头（Header）：可以携带 Content-Type（这是图片还是 HTML？）、User-Agent（谁在请求？）等元信息。
  • 状态码：用 200、404 这样的数字明确告诉客户端请求结果，而不是靠猜。
  • 更多方法：除了 GET，增加了 POST、HEAD。

但 HTTP/1.0 依然是每个请求一条 TCP 连接，用完就关（non-persistent connection）。一个网页里有 10 张图片，就要建立、关闭 10 次 TCP 连接——而 TCP 三次握手本身就有一轮网络往返的开销（这部分原理在传输层 Tab 会详细展开），这意味着大量时间浪费在"建连接"而不是"传数据"上。

HTTP/1.1（1997）：持久连接 + 管道化

HTTP/1.1 是用得最久、影响最深远的版本，核心解决"频繁建连接太浪费"的问题：

  • 持久连接（Persistent Connection）：默认开启 Connection: keep-alive，一条 TCP 连接可以连续发多个请求/响应，不必每次都重新握手。
  • 管道化（Pipelining）：允许不等上一个响应返回，就把下一个请求发出去，进一步减少等待时间。但管道化有一个致命缺陷——服务器必须按请求顺序依次返回响应，如果排在前面的请求处理得慢，后面的响应即使已经处理完也得排队等，这就是著名的队头阻塞（Head-of-Line Blocking）问题。正因如此，管道化在实践中很少被浏览器默认启用。

此外 HTTP/1.1 还引入了 Host 头（支持一台服务器承载多个域名的虚拟主机）、分块传输编码（Transfer-Encoding: chunked，内容大小提前未知时也能流式发送）等改进。

队头阻塞的另一层：即使不用管道化，浏览器为了并行加载资源，通常会对同一域名开 6 条左右的并行 TCP 连接来"绕开"单连接的队头阻塞——但这治标不治本，连接数一多，TCP 慢启动、服务器资源消耗都跟着上升。这正是 HTTP/2 要解决的问题。

HTTP/2（2015）：多路复用 + 二进制分帧

HTTP/2 的核心思路是：既然队头阻塞的根源是"一条连接里请求必须排队"，那就把请求和响应拆成更小的帧（Frame），多个请求的帧可以在同一条 TCP 连接上交错发送，接收端再按帧头里的流 ID 重新组装——这就是多路复用（Multiplexing）。配合的底层改动是把原来的纯文本报文改成二进制分帧，二进制格式更紧凑、更容易被机器高效解析。HTTP/2 还引入了头部压缩（HPACK）等特性。

这部分只做原理点到为止，安全 Tab 会有专篇《HTTP/2 的多路复用与头部压缩》详细展开帧结构和 HPACK 编码细节。

HTTP/3（2022 正式成为 RFC）：抛弃 TCP，拥抱 QUIC

HTTP/2 用一条 TCP 连接解决了"应用层"的队头阻塞，但暴露出一个更底层的问题：TCP 协议本身是有序字节流，如果这条连接上丢了一个 TCP 包，在收到重传数据之前，后面所有已经到达的数据都不能被应用层处理——多路复用又在传输层被"队头阻塞"了一次。

HTTP/3 的解法是釜底抽薪：不再基于 TCP，而是基于 Google 设计的 QUIC协议（构建在 UDP 之上）。QUIC 在用户态自己实现了可靠传输、拥塞控制，并且让多个流（stream）互相独立——一个流丢包只影响这一个流，不会拖累其他流。这部分原理会在传输层 Tab 的《为什么 HTTP/3 选择基于 UDP 的 QUIC 协议》专篇详细展开。

常见误区

误区一：以为 HTTP/2 多路复用彻底消灭了队头阻塞。实际上它只解决了应用层的队头阻塞，底层 TCP 的队头阻塞依然存在，这正是 HTTP/3 存在的意义。
误区二：以为版本越高就一定越快。HTTP/2、HTTP/3 的优势主要体现在资源多、并发请求多的场景（比如一个页面几十个资源），单个大文件传输或者弱网环境下，优势不一定明显，需要结合场景判断。
面试高频："HTTP/1.1 和 HTTP/2 的区别？"——抓手是"连接使用方式"：1.1 是持久连接但请求排队（或多连接并行），2.0 是单连接内多路复用、二进制分帧、头部压缩。"为什么还要 HTTP/3？"——抓手是"TCP 层面的队头阻塞是 TCP 协议本身决定的，应用层协议无法绕过，只能换成基于 UDP 的 QUIC"。

核心价值总结

  • 每一代都在解决上一代的性能瓶颈：0.9 解决"能不能用"，1.0/1.1 解决"够不够用"，2/3 解决"够不够快"。
  • 队头阻塞是贯穿整条演进线的主线索：先是连接级别（1.0）、再是应用层请求级别（1.1）、再是 TCP 字节流级别（2.0 之后暴露），每一次解决都换来下一层的新问题。
