分层与应用层

Cookie 与 Session

HTTP 天生"失忆"——服务器处理完一个请求就把它彻底忘掉。Cookie 和 Session 就是为了在这种"失忆协议"上，让服务器认出"你还是刚才那个你"而发明的补丁。

为什么 HTTP 是无状态协议

HTTP 被设计成无状态（Stateless）协议：服务器不会保留任何关于之前请求的记忆，每一个请求都是完全独立的，服务器处理完就"忘"。这不是设计缺陷，而是有意为之——早期的 HTTP 只是用来获取静态文档，服务器不需要记住"谁刚才访问过"，无状态让服务器实现更简单、也更容易水平扩展（不需要考虑"这个用户的请求必须固定发到同一台服务器"）。

但互联网很快就需要"登录后网站要记得我是谁"这样的能力——你在淘宝加入购物车、登录银行网站，都需要服务器在多次请求之间"认出你"。这就产生了一个矛盾：协议本身无状态，但业务需要有状态。Cookie 和 Session 就是解决这个矛盾的一对组合拳。

Cookie：让客户端替服务器"记笔记"

思路很朴素：服务器自己不记，那就让客户端（浏览器）帮忙记，每次请求的时候客户端主动把"小纸条"带上，服务器一看纸条上写的内容就知道"这是刚才那个用户"。这张小纸条就是 Cookie。

工作流程：

  1. 服务器在响应头里用 Set-Cookie 字段告诉浏览器"请保存这个值"。
  2. 浏览器把这个值存起来，跟这个域名绑定。
  3. 之后浏览器每次向同一域名发请求，都会自动在请求头的 Cookie 字段里把值带上。
Cookie 的下发与回传
// 服务器响应头（第一次登录后下发）
Set-Cookie: sessionId=abc123xyz; Path=/; HttpOnly; Max-Age=3600

// 浏览器之后每次请求都会自动带上
Cookie: sessionId=abc123xyz

如果直接把敏感数据（比如用户名、余额）明文存进 Cookie，任何人打开浏览器开发者工具都能看到甚至篡改——这是很不安全的。所以实践中 Cookie 里通常只放一个不可预测的随机 ID，真正的数据存在服务器端，这就引出了 Session。

Session：真正的数据留在服务器

Session 是服务器端维护的一块数据（通常是内存里的一个 Map，key 是 Session ID，value 是这个用户的登录信息、购物车等状态）。Cookie 里存的 sessionId 只是一把"钥匙"，服务器凭这把钥匙去自己的存储里查出对应的完整数据。

浏览器 ── 请求带 Cookie: sessionId=abc123 ──▶ 服务器
　　　　　　　　　　　　　　　　　　　　　　　　　│
　　　　　　　　　　　　　　　　　　服务器用 abc123 查 Session 存储
　　　　　　　　　　　　　　　　　　　　　　　　　│
　　　　　　　　　　　　　　　　查到 { userId: 42, role: "admin" }
　　　　　　　　　　　　　　　　　　　　　　　　　▼
　　　　　　　　　　　　　　　　　　　　知道"你是谁"，正常处理业务

这套配合机制解决的核心问题是：Cookie 负责"跨请求携带一个标识"，Session 负责"用这个标识关联真实数据，且数据不暴露给客户端"。两者缺一不可——只有 Cookie 没有 Session，数据要么不安全（明文存客户端）要么存不下（Cookie 大小有限制，通常单个 4KB）；只有 Session 没有 Cookie，服务器无法知道这次请求该去查哪个用户的数据。

分布式系统下的 Session 共享问题

Session 默认存在单台服务器的内存里，一旦系统水平扩展成多台服务器、请求经过负载均衡分发，问题就来了：用户第一次请求落在服务器 A（登录成功，Session 存在 A 的内存里），第二次请求被负载均衡分发到服务器 B，B 的内存里根本没有这个 Session，用户就被当成没登录。

常见解法有"粘性会话"（同一用户的请求固定转发到同一台服务器）、"Session 集中存储"（把 Session 数据放到 Redis 这类共享存储，所有服务器都去同一个地方查）等，这属于分布式系统设计范畴，本模块不展开，后续负载均衡相关内容会再点到。

Cookie vs Session 对比

维度CookieSession
存储位置客户端（浏览器）服务器端
安全性较低，可被查看/篡改，需配合 HttpOnly/Secure较高，数据不暴露给客户端
存储容量很小（单个约 4KB）不受浏览器限制，取决于服务器资源
扩展性天然无状态，不存在共享问题多机部署需要额外解决共享问题
典型用途存 Session ID、用户偏好设置、埋点标识存登录态、购物车等真实业务数据

常见误区

误区一：以为 Cookie 和 Session 是两种互斥的技术方案，只能选一个。实际上它们是配合关系——绝大多数基于 Session 的登录态方案，都依赖 Cookie 来传递 Session ID。
误区二：以为禁用 Cookie 就完全没法维持登录态。传递 Session ID 也可以通过 URL 参数（?sessionId=xxx，早期 JSP 的 jsessionid 就是这么做的）实现，只是安全性和体验都更差，容易泄露到日志、被浏览器历史记录留存。
面试高频："Cookie 和 Session 的区别？"——抓手是"存储位置不同"和"配合关系"：Cookie 存客户端、体积小、不够安全，Session 存服务端、更安全，二者通过 Session ID 这个 Cookie 值关联起来。追问"分布式下 Session 怎么共享？"——答"粘性会话或 Session 集中存储（如 Redis）"。

核心价值总结

  • 无状态是 HTTP 的设计选择，不是缺陷：简化了协议本身，但把"记住状态"的责任转移给了上层应用自己解决。
  • Cookie 负责携带标识，Session 负责存储真实数据：这是应用层最经典的"瘦客户端凭证 + 服务端状态存储"模式。
