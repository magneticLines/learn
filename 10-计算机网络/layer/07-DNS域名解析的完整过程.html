DNS 域名解析的完整过程

在浏览器发出那个真正的 HTTP 请求之前，还有一件事必须先完成：把人类好记的域名，翻译成机器真正认识的 IP 地址。这一篇是"分层与应用层"Tab 的收尾，也是通往传输层/网络层的入口。

为什么需要 DNS

网络世界里，真正定位一台主机靠的是 IP 地址（IP 地址本身的结构会在网络层 Tab 详细展开），但 IP 地址（比如 142.250.66.174）对人类极不友好——记不住，也没有含义。DNS（Domain Name System，域名系统）就是一套把人类可读的域名（www.google.com）翻译成 IP 地址的分布式查询系统。

类比：DNS 就像电话簿/通讯录——你想给"张三"打电话，不需要记住他的手机号，只要在通讯录里查"张三"这个名字，系统就帮你查出对应的号码再拨出去。DNS 做的是同样的事：应用只需要知道域名，DNS 负责把域名翻译成真正用来建立连接的 IP 地址。

DNS 之所以要设计成分布式、分层级的系统，而不是搞一台全球唯一的中心服务器，核心原因和 C/S 架构的单点瓶颈问题是一样的道理——全世界几十亿设备每天产生海量域名查询，单点服务器无论从算力、带宽还是容灾角度都扛不住，必须靠分层分区、层层缓存来分摊压力。

完整链路：从浏览器输入网址到拿到 IP

当你在浏览器输入 www.example.com 并回车，会依次经过下面这条查询链路：

1. 浏览器缓存 — 浏览器自己维护一份最近解析过的域名缓存，命中直接返回
　　　│ 未命中
2. 操作系统缓存 — 查系统的 hosts 文件和本地 DNS 缓存（Windows: ipconfig /displaydns）
　　　│ 未命中
3. 本地 DNS 服务器（也叫 递归域名服务器，通常是运营商或 8.8.8.8 这类公共 DNS）
　　　│ 自己没有缓存，代替你去公网上逐级查询
4. 根域名服务器 — 全球共 13 组，不知道具体 IP，但知道".com 归谁管"，返回顶级域名服务器地址
　　　│
5. 顶级域名服务器（TLD，如 .com/.cn 的服务器）— 不知道具体 IP，但知道"example.com 归哪个权威服务器管"
　　　│
6. 权威域名服务器 — 真正保存 example.com 完整 DNS 记录的服务器，返回最终 IP
　　　▼
本地 DNS 服务器把结果返回给你的电脑，并缓存起来（供后续查询和其他用户复用）

递归查询 vs 迭代查询

上面这条链路里，其实包含两种不同的"问法"，理解这个区别是这一篇的核心考点：

  • 递归查询（Recursive Query）：你的电脑向本地 DNS 服务器发起的查询是递归的——"我不管你怎么查，你必须给我一个最终答案（IP 地址），查不到也要告诉我错误"，本地 DNS 服务器把"一层层往下问"的全部脏活累活都包揽了。
  • 迭代查询（Iterative Query）：本地 DNS 服务器向根、顶级域、权威服务器发起的查询是迭代的——每一级服务器都只回答"我不知道最终答案，但我知道下一步该去问谁"，从不代劳往下查，把"继续问谁"这个责任丢回给发起查询的一方（本地 DNS 服务器自己）。
为什么这样设计：如果根域名服务器要对全世界所有客户端的查询都做递归（自己往下一层层查完再返回），它的负载会大到无法承受。让根/顶级域/权威服务器只做"指路"这种轻量级的迭代应答，把真正繁重的多轮查询工作分摊给数量众多、离用户更近的本地 DNS 服务器去做，是一种典型的"负载下沉"设计。

缓存：贯穿全程的性能优化

整条链路里几乎每一站都有缓存（浏览器缓存、系统缓存、本地 DNS 服务器缓存），这是因为域名到 IP 的映射相对稳定，短期内很少变化，重复查询同一个域名走一遍完整的递归+迭代流程纯属浪费。DNS 记录里的 TTL（Time To Live）字段就是权威服务器告诉大家"这条记录你可以缓存多久"，超过这个时间才需要重新查询，这也是为什么修改了域名解析记录之后，全网生效往往需要等待一段时间（旧记录的 TTL 还没过期）。

常见误区

误区一：以为浏览器发起的所有查询都是"迭代"的。实际上，客户端到本地 DNS 服务器这一段几乎总是递归查询，只有本地 DNS 服务器代替你向权威链路发起的查询才是迭代查询。混淆这两段是这道题最容易丢分的地方。
误区二：以为 DNS 只在建立连接前查一次就完事，之后就不用管了。实际上 DNS 解析发生在 TCP 连接建立之前——先解析出 IP，再用这个 IP 去和服务器完成三次握手（下一 Tab 会详细讲），DNS 解析的耗时会直接叠加在整个页面加载时间里，这也是为什么很多性能优化手段（DNS 预解析 <link rel="dns-prefetch">）专门针对这一步。
面试高频："说一下 DNS 解析的完整过程" ——按"浏览器缓存 → 系统缓存 → 本地 DNS 服务器 → 根 → 顶级域 → 权威"这条链路背下来，并补充"客户端到本地 DNS 是递归查询，本地 DNS 到根/顶级域/权威是迭代查询"这个关键区分点，基本能覆盖这道题的主要得分点。

核心价值总结

  • DNS 用分层 + 缓存解决了"域名翻译"的可扩展性问题：和根域名服务器只做"指路人"而不是"全能查询员"是同一个设计思路。
  • 递归查询和迭代查询的角色分工，是负载下沉思想的具体体现：越靠近核心的服务器职责越轻，越靠近用户的服务器承担越多的中转工作。
