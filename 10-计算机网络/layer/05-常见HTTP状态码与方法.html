常见 HTTP 状态码与方法

方法定义"要做什么"，状态码定义"结果如何"——这是 REST API 设计和面试里出现频率最高的一组知识点，也是最容易被想当然理解错的一组知识点。

HTTP 方法：语义、幂等性、安全性

每个 HTTP 方法都有约定俗成的语义，但方法本身只是个字符串，服务器实现完全可以不遵守约定（比如用 GET 做删除操作）。真正重要的是理解两个属性：

  • 安全性（Safe）：该方法是否会改变服务器上的资源状态。GET/HEAD 是安全的，只读不写。
  • 幂等性（Idempotent）：同一个请求重复执行 N 次，服务器最终状态和只执行 1 次是否一致。这决定了浏览器/网关能不能"放心地自动重试"——网络超时后能不能自动重发这个请求，取决于它是不是幂等的。
方法语义安全幂等典型场景
GET获取资源是是查询列表、查询详情
POST创建资源 / 提交数据否否新增订单、登录、提交表单
PUT整体替换资源否是用完整对象覆盖更新用户信息
PATCH部分修改资源否否（通常）只改用户的昵称字段
DELETE删除资源否是删除某条订单
HEAD只要响应头，不要 body是是探测资源是否存在/大小
为什么 POST 不幂等而 PUT 幂等：POST 语义是"新增一个"，重复提交两次表单会创建两条订单，结果不同，所以不幂等。PUT 语义是"把资源整体设置成某个状态"，重复执行"把用户名设置为张三"多少次，结果都是同一个状态，所以幂等。DELETE 也是幂等的——虽然第二次删除会返回"资源不存在"，但服务器最终状态（资源已被删除）是一致的。
陷阱：PATCH 是否幂等要看具体语义。如果 PATCH 传的是"整个字段的新值"（把 age 设为 25），是幂等的；如果传的是"增量操作"（把 age 加 1），重复执行结果就不同，不幂等。面试时如果被问 PATCH 幂等性，正确答法是"取决于具体实现"，而不是一刀切。

状态码分类：从第一位数字看大类

状态码是三位数字，第一位数字表示大类，这个设计让客户端即使不认识具体的状态码，也能靠首位数字做出合理的兜底处理：

分类含义
1xx信息性，请求已收到，继续处理（如 101 Switching Protocols，WebSocket 握手用到）
2xx成功，请求已被正确处理
3xx重定向，需要客户端进一步操作才能完成请求
4xx客户端错误，请求本身有问题
5xx服务器错误，请求没问题但服务器处理时出错

高频状态码逐个讲

状态码含义使用场景 / 辨析
200 OK请求成功最常见的成功响应
301 Moved Permanently永久重定向浏览器/搜索引擎会缓存新地址，下次直接跳转到新 URL
302 Found临时重定向不会被缓存，每次都要重新请求原地址确认；登录跳转常用
304 Not Modified资源未修改配合 If-Modified-Since/ETag 做协商缓存，告诉浏览器直接用本地缓存，省去传输 body
400 Bad Request请求参数错误客户端传的参数不合法，服务器无法理解
401 Unauthorized未认证你是谁都不知道，先登录
403 Forbidden已认证但无权限知道你是谁，但你没资格访问这个资源
404 Not Found资源不存在URL 对应的资源找不到
500 Internal Server Error服务器内部错误服务器代码抛异常了，笼统的"服务端炸了"
502 Bad Gateway网关错误Nginx/网关请求上游服务时，上游没有正常响应
503 Service Unavailable服务不可用服务器过载或维护中，暂时无法处理请求
301 vs 302 的实际影响：因为 301 会被浏览器和搜索引擎缓存，如果误把临时性的跳转配置成 301，用户下次访问会跳过服务器直接命中缓存的旧跳转规则，即使你已经把配置改回来了，用户浏览器可能还是老老实实跳到错误地址——这是运维上真实踩过的坑。

常见误区

误区一：把 401 和 403 搞反。记忆技巧：401（Unauthorized）是"没验明身份"，先登录；403（Forbidden）是"身份已验明，但你被禁止"，登录了也没用。
误区二：以为业务上所有错误都该返回 200，把真正的错误信息塞进 body 里的 code 字段（比如很多国内 API 的做法）。这不是"错"，但会让依赖标准 HTTP 语义的中间件（网关的重试策略、CDN 缓存、监控告警）失效——这是一个工程取舍，需要团队达成一致。
面试高频："GET 和 POST 的区别？" 除了教科书答案（GET 参数在 URL 里/POST 在 body 里，GET 有长度限制/POST 没有），更关键的答案是幂等性和安全性的差异，这决定了浏览器的缓存策略、回退行为（浏览器回退时对 POST 会提示"重新提交表单"，对 GET 不会）以及能否被代理自动重试。

核心价值总结

  • 方法回答"做什么"，靠语义约定而非强制校验：幂等性和安全性是设计 API 和判断能否安全重试的关键依据。
  • 状态码首位数字决定处理策略：不认识具体状态码时，靠首位数字也能做出合理兜底。
