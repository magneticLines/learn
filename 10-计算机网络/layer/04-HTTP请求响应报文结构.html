HTTP 请求响应报文结构

不管 HTTP 版本怎么演进，请求和响应报文的"语义结构"是相通的——搞懂这套结构，浏览器 F12 抓的每一行网络面板信息才不再是天书。

为什么报文要长这个样子

HTTP 是一个基于文本（HTTP/2、3 是二进制帧，但语义上仍等价于文本报文）的请求-响应协议：客户端发一个请求报文，服务器必须回一个响应报文。报文本质上就是"结构化的纯文本"，之所以设计成固定的四段式结构，是为了让接收方能用最简单的规则（按行、按空行分割）就把报文解析出来，而不需要复杂的语法分析器。

请求报文结构

请求行　　方法 空格 URL 空格 HTTP版本
请求头　　若干行 "字段名: 值"
　　　　　（空行，表示头部结束）
请求体　　可选，GET 通常没有，POST/PUT 常携带数据
真实抓包示例：POST 登录请求
POST /api/login HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: application/json
Content-Type: application/json
Content-Length: 38
Cookie: sessionId=abc123xyz

{"username": "shea", "password": "******"}

第一行是请求行：POST 是方法，/api/login 是请求的资源路径，HTTP/1.1 是协议版本。 接下来若干行是请求头，每行都是 字段名: 值 的键值对，携带各种元信息——Host 告诉服务器要访问哪个域名（一台服务器可能承载多个网站）、Content-Type 告诉服务器请求体是什么格式、Cookie 携带身份标识（下一篇会详细展开）。 然后是一个空行，它是报文里最容易被忽略但至关重要的部分——正是这个空行告诉解析器"头部结束了，后面是body"。 最后是请求体，携带真正要提交的数据。

响应报文结构

状态行　　HTTP版本 空格 状态码 空格 状态描述
响应头　　若干行 "字段名: 值"
　　　　　（空行，表示头部结束）
响应体　　实际返回的数据（HTML/JSON/图片二进制...）
真实抓包示例：登录成功响应
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 45
Set-Cookie: sessionId=abc123xyz; Path=/; HttpOnly
Cache-Control: no-store
Date: Fri, 03 Jul 2026 06:30:00 GMT

{"code": 0, "message": "login success"}

第一行是状态行：协议版本 + 状态码 200 + 状态描述 OK（状态码含义下一篇专门展开）。 响应头里 Content-Type 告诉浏览器该怎么解析响应体（这里是 JSON），Set-Cookie 是服务器让浏览器保存一个 Cookie（这是下一篇 Cookie/Session 机制的关键字段）。 空行之后是响应体——本例中是一段 JSON 文本，如果是访问网页则会是完整的 HTML。

常见头部字段速查

字段出现位置作用
Host请求头目标域名，支持虚拟主机
User-Agent请求头标识客户端（浏览器/系统/设备）
Content-Type请求/响应头body 的数据格式（如 text/html、application/json）
Content-Length请求/响应头body 的字节长度
Cookie请求头客户端携带的会话标识，回传给服务器
Set-Cookie响应头服务器让客户端保存的 Cookie
Connection请求/响应头是否保持持久连接（keep-alive）
Cache-Control响应头缓存策略

常见误区

误区一：以为空行只是"好看"，可有可无。空行是报文解析的关键分隔符，缺了它接收方无法判断头部在哪里结束——这也是为什么在 Postman 之类工具里手写原始报文时，必须严格保留这一空行。
误区二：以为 GET 请求一定没有请求体。HTTP 规范上并不禁止 GET 带 body，只是绝大多数服务器/框架实现约定俗成地忽略 GET 的 body，语义上 GET 也不应该有副作用地携带数据（这和下一篇的方法幂等性/安全性有关）。
面试高频："HTTP 报文由哪几部分组成？"——按顺序背出请求行/状态行、头部、空行、body 四段结构，并能说出空行的作用是分隔头部和 body。

核心价值总结

  • 四段式结构是解析报文的骨架：起始行描述"做什么/结果如何"，头部描述"元信息"，空行做分隔，body 装载真正的数据。
  • 浏览器 F12 网络面板本质就是在展示这套结构：Headers 面板对应请求头/响应头，Response 面板对应响应体。

与其他知识点的联系

  • 常见 HTTP 状态码与方法：请求行里的方法字段（GET/POST/...）和状态行里的状态码字段（200/404/...），正是本篇讲的报文结构里两个具体的"坑位"——报文结构规定了它们"写在哪一行、什么格式"，下一篇讲的是"这个坑位里能填什么值、每个值什么意思"。
  • HTTP 协议的演进：报文里的 Content-Type、Cookie、Connection: keep-alive 这些头部字段不是从一开始就有的——它们分别是 HTTP/1.0 补上元信息、后续版本补上会话保持机制的产物，报文结构本身也是协议演进的产物而不是一开始就设计完备的。
  • OSI 七层模型与 TCP/IP 四层模型：报文是应用层自己定义的数据格式，交给传输层的 TCP 时会被切成一个个 TCP 段，报文里的空行、请求头、请求体在 TCP 看来都只是无差别的字节流——分层模型保证了"应用层怎么组织报文格式"完全不需要传输层关心。
  • DNS 域名解析：报文里的 Host 头之所以存在，正是因为一次 DNS 解析只能把域名翻译成 IP，翻译不出"这个 IP 对应哪个域名"——同一个 IP 可能承载多个网站（虚拟主机），必须靠请求报文里的 Host 头告诉服务器到底要访问哪个域名对应的站点。
  • NAT 网络地址转换：报文里的 Host 头和 Content-Length 等字段是应用层数据，NAT 网关做地址转换时只改写 IP 包头和 TCP/UDP 端口，完全不会（也不需要）解析或修改报文里的这些应用层字段——这体现了 NAT 工作在网络层+传输层、报文结构是应用层的事，两者互不干扰。
