为什么 HTTP/3 选择基于 UDP 的 QUIC 协议

本 Tab 第一篇说过"可靠性是有代价的"，QUIC 的故事恰恰是这句话的最佳例证——它抛弃了内核里已经打磨了几十年的 TCP，自己在 UDP 之上重新造了一套可靠传输，只为解决 TCP 天生难以摆脱的一个结构性问题。

先回顾：HTTP/2 的多路复用带来了什么新问题

HTTP/1.1 时代，一个 TCP 连接同一时刻只能处理一个请求（或有限的并发靠开多条连接凑合），HTTP/2 引入了多路复用（Multiplexing）—— 一条 TCP 连接上可以同时并行传输多个请求/响应的数据帧，互相交织，大幅减少了连接数和延迟。

但这里有个致命的组合问题：HTTP/2 的多个请求跑在同一条 TCP 连接上，而 TCP 是严格保证字节流顺序的。

核心问题：TCP 层面的队头阻塞（Head-of-Line Blocking）

具体场景：HTTP/2 在一条 TCP 连接上并行发送请求 A、B、C 的数据帧。假设 TCP 报文段传输顺序是 A1, B1, C1, A2(丢失), B2, C2... 由于 TCP 必须保证字节流严格有序交付给上层应用，即使 B2、C2 已经先到达，TCP 也不会把它们交给应用层，而是全部阻塞等待 A2 重传成功， 因为在 TCP 眼里，这只是"一条流"，它不知道也不关心这里面混杂着三个逻辑上独立的 HTTP 请求。 结果就是：A 请求的一个丢包，拖累了 B、C 两个本来毫不相干的请求都卡住。

讽刺的是，HTTP/1.1 时代如果每个请求各自用一条独立的 TCP 连接，一个连接丢包只影响它自己，不会连累其他请求； HTTP/2 用多路复用换来了连接数减少和头部压缩的好处，却因为把多个独立的逻辑流强行塞进一条 TCP 连接，让底层的队头阻塞问题变得更严重了——这是 TCP 协议本身的顺序保证机制和 HTTP/2 多路复用理念之间根本性的不兼容。

QUIC 的解法：抛开 TCP，自己在 UDP 上重建一切

问题出在"TCP 只认识一条字节流，不认识里面的多个逻辑请求"，那解法就很直接：让传输层也能感知到多个独立的流（Stream），流与流之间互不阻塞。 但 TCP 协议栈通常固化在操作系统内核里，改动和升级周期以年计，几乎不可能推动全球范围的内核升级。 QUIC 的做法是：绕开内核里的 TCP，直接在应用层（用户态）基于 UDP 从头实现一套新的可靠传输协议，同时把 HTTP/2 想要的多路复用能力原生内建进这个新协议里：

  • 多路复用是协议原生的：一条 QUIC 连接内部天然支持多个独立的 Stream，每个 Stream 有自己独立的序号和确认，一个 Stream 丢包只会阻塞它自己，不会影响其他 Stream——彻底解决了 TCP 层面的队头阻塞。
  • 可靠传输、流量控制、拥塞控制全部自己实现：相当于把 TCP 的核心机制（序号确认、滑动窗口、拥塞控制算法）在用户态重写了一遍，跑在 UDP 报文之上，这样协议本身可以脱离操作系统内核独立快速迭代升级。
  • 内置 TLS 1.3：传统 HTTPS 是 "TCP 握手（1 RTT）+ TLS 握手（1~2 RTT）" 分两次串行完成，QUIC 把连接建立和加密握手合并在一起，理想情况下只需 1 RTT（甚至 0-RTT 恢复连接），显著减少建连延迟。

QUIC 的独特优势：连接迁移（Connection Migration）

TCP 连接是用一个四元组（源 IP、源端口、目的 IP、目的端口）唯一标识的。手机从 WiFi 切换到 4G 网络时，源 IP 会发生变化， TCP 连接会因为四元组变化而直接失效，必须重新三次握手建立新连接（这也是很多人手机切换网络时视频通话卡顿一下甚至断开的原因）。

QUIC 用一个独立于 IP 和端口的连接 ID（Connection ID）来标识连接，而不依赖四元组。 这意味着即使客户端的 IP 地址变了（WiFi 切 4G），只要连接 ID 不变，服务端依然能识别出这是同一个连接， 不需要重新握手就能无缝延续之前的传输状态（包括拥塞窗口等）——这就是连接迁移，是 QUIC 相比 TCP 的一个结构性优势，而不是简单的性能优化。

对比一览：TCP+TLS+HTTP2 vs QUIC(HTTP/3)

常见误区

误区："QUIC 基于 UDP，所以它和 UDP 一样不可靠。" 完全相反——QUIC 只是借用 UDP 作为最底层的"发送数据包"的通道（因为 UDP 本身几乎不做任何事，给了 QUIC 完全自由发挥的空间）， QUIC 自己在其上实现了完整的可靠传输、流量控制、拥塞控制机制，可靠性并不弱于 TCP，某些方面（如队头阻塞）甚至优于 TCP。 "基于 UDP" 说的是它复用了 UDP 的报文格式和无连接特性作为起点，不代表它放弃了可靠性。

核心价值总结

  • 问题根源：TCP 的字节流顺序保证和 HTTP/2 的多路复用理念存在结构性冲突，导致队头阻塞在应用层看不见摸不着却真实影响性能。
  • QUIC 的思路：与其在内核里改 TCP（几乎不可能推动），不如在用户态基于 UDP 从零重建一套支持原生多流的可靠传输协议。
  • 连接迁移是移动网络时代格外重要的优势，体现了"重新设计"相比"缝缝补补"能带来的结构性收益。
  • 最后一篇会继续深入 QUIC/现代网络中拥塞控制算法的演进——BBR。

与其他知识点的联系

  • TCP 三次握手与四次挥手：本篇讲的"QUIC 合并连接建立与加密握手"，前提是理解 TCP 本身建连要花 1-RTT、断连还要走四次挥手的开销。QUIC 不仅合并了握手，其"流"级别的关闭也不需要像 TCP 那样整条连接一起进入四次挥手流程，单个流用完即可结束，不牵连整条连接的生命周期。
  • TCP 拥塞控制（传统算法）：本篇聚焦的是"可靠性该怎么重新实现"这个大问题里的"顺序与队头阻塞"子问题；而拥塞控制——"发送速率该多快"——是另一个必须在 UDP 之上从零补齐的子问题。下一篇会讲到，QUIC 默认选用的正是抛弃"丢包即拥塞"假设的 BBR 算法，这和本篇"抛弃 TCP 内核束缚"是同一种设计哲学在不同维度上的体现。
  • TLS 1.3 握手优化：本篇提到 QUIC 把连接建立和加密握手合并，能做到这一点的前提是 TLS 1.3 本身已经把握手压缩到 1-RTT（甚至 0-RTT）。如果还是 TLS 1.2 两次往返的握手逻辑，即使合并到 QUIC 连接建立过程里，延迟优化也会大打折扣——两者是配合关系，QUIC 站在 TLS 1.3 的成果上才能实现如此低的建连延迟。
  • HTTP/2 多路复用：本篇开头分析的"HTTP/2 多路复用 + TCP 严格有序 = 传输层队头阻塞"，正是 QUIC 诞生的直接动因。没有 HTTP/2 先把多个请求塞进一条连接，队头阻塞问题不会被放大到必须重新设计传输层的程度——这是两篇内容之间因果链条上最紧密的一环。
