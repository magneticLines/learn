TCP 三次握手

上一篇说 TCP 是"面向连接"的，这个连接到底是怎么建立的？三次握手是面试必考题，但绝大多数人只背得出流程，答不出"为什么必须是三次"。

先明确：这里的"连接"是什么

IP 层是无连接的，每个包独立寻路，互不知晓对方状态。TCP 要在这样的网络上模拟出一条"可靠的虚拟管道"， 双方就必须先就几个关键信息达成一致：各自的初始序号（ISN，Initial Sequence Number）、 对方是否真的存在并愿意通信、双方的最大报文段长度（MSS）等参数。 三次握手本质上就是"双方交换初始序号并确认对方收发能力正常"的过程。

完整过程：SYN → SYN-ACK → ACK

客户端（Client） 服务端（Server） │ │ │ 状态: CLOSED 状态: LISTEN │ │ │──① SYN, seq=x─────────────────────▶│ 状态: SYN-RCVD │ 状态: SYN-SENT │ │ │ │◀─────②SYN, ACK, seq=y, ack=x+1──│ │ │ │──③ACK, seq=x+1, ack=y+1─────────▶│ 状态: ESTABLISHED │ 状态: ESTABLISHED │ │ │ │◀════════ 开始双向传输数据 ═══════════▶│
  • 第一次握手（SYN）：客户端发送一个 SYN=1、随机初始序号 seq=x 的报文，进入 SYN-SENT 状态，表示"我想和你建立连接"。
  • 第二次握手（SYN-ACK）：服务端收到后，回复 SYN=1, ACK=1，带上自己的初始序号 seq=y 和确认号 ack=x+1，进入 SYN-RCVD 状态，意思是"收到你的请求了，我也想建立连接，并确认收到了你的 SYN"。
  • 第三次握手（ACK）：客户端收到后回复 ACK=1, ack=y+1，双方都进入 ESTABLISHED 状态，连接正式建立，之后即可传输数据（第三次握手的 ACK 报文也可以顺带捎带业务数据）。

核心考点：为什么是三次，不是两次？

这是面试最爱追问的点。表面答案是"三次才能确认双方的收发能力都正常"，但更本质的原因是： 防止已经失效的连接请求报文突然又传送到了服务端，从而产生错误连接，浪费服务端资源。

具体场景：客户端发送了第一个 SYN 请求，因为网络拥塞在某个路由器里滞留了很久没有到达服务端。客户端等待超时后，以为这个包丢了，于是重新发送了第二个 SYN 并顺利完成两次握手、建立连接、传输数据、关闭连接。 这时，那个滞留很久的旧 SYN 报文终于姗姗来迟地到达了服务端。 如果只需要两次握手（服务端收到 SYN 就直接建立连接），服务端会误以为这是一个新的连接请求，立即分配资源建立连接并进入等待客户端发数据的状态——但客户端根本不知道有这个连接存在，永远不会发数据过来，服务端的资源就白白浪费在这个"僵尸连接"上了。 加上第三次握手后，服务端在收到 ACK 之前只是"半连接"状态，不会为一个孤立的旧 SYN 分配完整资源；即使服务端对这个旧 SYN 回复了 SYN-ACK，客户端因为并不认识这个连接，会回复 RST 拒绝，从而清除这个无效的半连接。
更本质的理解：握手次数的下限由"双方都要确认对方收到了自己发的确认"决定——A 发消息给 B，B 不知道 A 有没有收到自己的回复，所以 A 必须再发一次确认；这是分布式系统里经典的"两军问题（Two Generals' Problem）"，理论上永远无法 100% 确认，TCP 三次握手是工程上的实用折中：三次刚好能让双方都确信"我发的和收的都没问题"。

SYN Flood 攻击原理简介

利用的正是服务端在收到 SYN 后、收到最终 ACK 前的这段"半连接"等待期。 攻击者伪造大量不存在的源 IP，向服务端疯狂发送 SYN 报文。服务端对每一个 SYN 都回复 SYN-ACK， 并在半连接队列（SYN queue）里保留这个连接等待第三次握手，但由于源 IP 是伪造的， ACK 永远不会到达。大量半连接迅速占满服务端的半连接队列和内存资源，导致正常用户的 SYN 请求被丢弃，无法建立新连接——这就是拒绝服务攻击（DoS）。

常见防御手段包括：SYN Cookies（服务端不为半连接分配真实资源，而是把连接信息编码进返回的 SYN-ACK 序号里，等收到合法 ACK 后再从中还原出连接状态，从而不需要维护半连接队列）、缩短半连接超时时间、增大半连接队列容量、结合防火墙做速率限制等。

常见误区

误区："四次握手不是更安全吗，为什么不用？" 三次已经是能够可靠确认双方收发能力的最小次数，第四次握手不会带来额外的确认价值，纯属浪费一次 RTT，所以工程上没有采用的必要。这也是理解"四次挥手"时的重要参照——挥手多一次是因为业务语义不同（关闭连接双方可能不同步），不是握手不够可靠。

核心价值总结

  • 三次握手的目的：确认双方收发能力正常 + 同步初始序号 + 防止旧的失效连接请求突然到达造成资源浪费。
  • 半连接队列是把双刃剑：它是三次握手机制的必然产物，也正是 SYN Flood 攻击可以利用的弱点。
  • 下一篇讲连接的另一端——四次挥手，会发现"关闭连接"比"建立连接"还要多一分谨慎。

与其他知识点的联系

  • TCP 四次挥手：握手三次、挥手四次，表面看像是随意的数字差异，本质原因完全不同——握手时"确认对方 SYN"和"发出自己的 SYN"是同一时刻能决定的事，可以合并成一个 SYN-ACK；挥手时"确认对方 FIN"和"自己准备好关闭"往往不是同一时刻的事（还有数据没发完），所以无法合并，这也是理解下一篇的关键铺垫。
  • TCP 与 UDP 的本质区别：三次握手正是上一篇提到的"可靠性代价"里最先要付出的一笔——DNS 查询宁愿用 UDP 自己简单重试，也不愿意为一问一答的短小报文承受一次握手的 RTT 开销，两篇内容互为因果。
  • TCP 拥塞控制：三次握手过程中交换的 MSS（最大报文段长度）等参数，为后续的拥塞控制、滑动窗口调节提供了基础信息；同时初始 cwnd 的大小也是在连接建立阶段就确定好的起点，握手的信息直接影响了后面慢启动的第一步。
  • TLS 1.3 与 QUIC：SYN Flood 攻击利用的"半连接"弱点，本质是三次握手把连接状态维护放在了应用数据到达之前，服务端要先信任再验证；TLS 1.3 通过 0-RTT/1-RTT 握手优化、QUIC 把传输层和加密层握手合并，都是在尝试减少"先建立状态、后验证身份"这种模式带来的延迟和被滥用的风险。
  • 分布式系统里的两军问题：三次握手是"两军问题"在工程上的一个实用折中解，这个思路在分布式系统的一致性协议（如两阶段提交、Raft 的日志复制确认）中也反复出现——你永远无法用有限次数的消息交换做到 100% 确认，只能在可接受的代价下逼近足够可靠。
