TCP 与 UDP 的本质区别

传输层的核心使命是"把数据从一个进程送到另一个进程"，但送达的方式有两种哲学：不惜代价保证送到（TCP），还是尽力而为、快就完事（UDP）。理解这个取舍，是理解整个传输层的起点。

先搞清楚：传输层解决什么问题

网络层（IP）只负责把数据包从一台主机送到另一台主机，它不关心这台主机上到底是哪个程序在等这个数据。 传输层在此基础上加了一层进程到进程（process-to-process）的寻址——靠端口号（port）区分同一台主机上的不同应用。 这是 TCP 和 UDP 的共同点：它们都提供"端口寻址"和"差错检测（校验和）"这两项基本服务。

类比：IP 层好比"快递公司把包裹送到某栋写字楼"，传输层则负责"送到写字楼里具体哪个房间（端口号）"。TCP 和 UDP 的区别，在于要不要"签收确认、丢件包赔、按顺序派送"这一整套服务。

本质区别：可靠性是有代价的

TCP 和 UDP 最根本的区别不是"一个可靠一个不可靠"这么简单，而是TCP 用连接状态、确认应答、超时重传、拥塞控制换来了可靠性和有序性，而这一切都需要额外的时间开销和内存开销。 UDP 则把这些全部砍掉，只留最基本的"发出去"，换来的是极致的简单和低延迟。

维度TCPUDP
连接面向连接，通信前需三次握手建立连接无连接，发送前无需建立任何状态
可靠性保证送达，丢包会重传不保证送达，丢了就丢了
顺序保证接收方按发送顺序收到（靠序号重排）不保证顺序，谁先到算谁的
流量/拥塞控制有滑动窗口 + 拥塞控制算法没有，发送速率完全由应用层决定
报文开销首部至少 20 字节，含大量控制字段首部固定 8 字节（源端口/目的端口/长度/校验和）
传输方式字节流（Byte Stream），无消息边界数据报（Datagram），保留消息边界
典型延迟建连 + 重传机制引入额外 RTT发出即走，延迟最低

什么场景该用 UDP

UDP 并不是"落后于 TCP 的过时协议"，而是在特定场景下的更优解：

  • 实时音视频通话（VoIP、视频会议）：丢一帧画面或一小段声音，人眼耳朵几乎察觉不到，但如果用 TCP，一旦丢包触发重传，画面会卡顿甚至积压延迟——"迟到的数据不如没有"，实时性比完整性更重要。
  • DNS 查询：一次查询就是一问一答的短小报文，为此专门三次握手再挥手，开销远大于收益；应用层自己简单重试一次成本更低。
  • 直播、在线游戏：同理，玩家状态同步要的是"最新的位置"，而不是"每一个历史位置都不丢"。
  • 广播/多播场景：TCP 是点对点连接，无法天然支持一对多；UDP 可以配合 IP 多播把一份数据发给一组接收者。
记忆技巧：凡是"实时性 > 完整性"或者"一问一答、成本敏感"的场景，优先考虑 UDP；凡是"数据必须完整无误"的场景（文件传输、网页、支付），必须用 TCP。

常见误区

误区一："UDP 不可靠 = UDP 不能做可靠传输。" 错。可靠性是可以在应用层之上自己实现的——QUIC（HTTP/3 的基础）就是在 UDP 之上自己造了一套可靠传输、拥塞控制机制，绕开了 TCP 内核实现的僵化问题，这个话题在本 Tab 第 7 篇会详细展开。
误区二："TCP 是流，UDP 是包"这句话很多人只记了结论没理解含义。TCP 把数据看成一条没有边界的字节流，应用层调用几次 write 和对端调用几次 read 收到的次数完全可能对不上（可能粘包/拆包）；而 UDP 每次 sendto 对应一个独立数据报，接收方一次 recvfrom 正好收到一个完整报文，天然保留消息边界。这是很多"粘包问题"面试题的根源。

核心价值总结

  • 没有绝对更好的协议，只有更适合场景的协议：可靠性、有序性是用时间和资源换来的，不是免费的。
  • TCP 的复杂性正是本 Tab 接下来 5 篇的主线：三次握手建连、四次挥手断连、序号确认保证可靠、滑动窗口做流量控制、拥塞控制算法调节发送速率——每一项都是在为"可靠"这两个字买单。
  • 先建立这个全局印象，再逐个拆解细节，理解会顺畅很多。

与其他知识点的联系

  • TCP 三次握手/四次挥手：这两个动作正是本文所说"可靠性的代价"里最直观的部分——建连和断连本身就要多花 1~2 个 RTT，UDP 完全没有这个开销，这也是 DNS 查询、实时音视频优先选 UDP 的直接原因。
  • TCP 拥塞控制：本文表格里"流量/拥塞控制"一行只是点了一句，拥塞控制的本质是发送方对共享网络资源做出的"克制"，这是 UDP 完全没有的机制——UDP 应用如果想在网络拥塞时也讲"礼貌"，必须自己在应用层实现类似的退让逻辑，否则会成为网络拥堵的元凶。
  • QUIC / HTTP/3：QUIC 选择跑在 UDP 之上，恰恰是因为 UDP 没有 TCP 的内核状态机和队头阻塞（head-of-line blocking），QUIC 得以在用户态自己实现可靠传输、拥塞控制、多路复用，绕开了操作系统内核升级 TCP 实现的困难，这也是本文误区一提到的"UDP 不可靠≠不能做可靠传输"的最佳例证。
  • HTTP/1.1 与 HTTP/2：两者都基于 TCP，都要承受"字节流没有消息边界"带来的粘包问题（本文误区二），这也是 HTTP 报文里必须靠 Content-Length 或分帧机制自己标出边界的原因；而 HTTP/2 的多路复用共享一条 TCP 连接，反而会因为 TCP 层面的丢包重传而阻塞该连接上所有的流，这是 HTTP/3 改用 QUIC/UDP 的核心动机之一。
  • TLS 握手：基于 TCP 的 HTTPS 需要先完成 TCP 三次握手，再叠加 TLS 1.2 的两次往返（或 TLS 1.3 的一次往返）握手才能开始传数据，这是"可靠性代价"在真实网络请求里层层叠加的典型例子；QUIC 则把传输层握手和 TLS 1.3 握手合并在一起，进一步压缩建连延迟。
