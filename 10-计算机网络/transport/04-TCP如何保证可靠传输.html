TCP 如何保证可靠传输

IP 层是"尽力而为"的：包可能丢失、重复、乱序、损坏，什么都不保证。TCP 却要在这样的地基上盖出"数据一定能完整、有序地到达"的大楼——靠的就是序号、确认应答和重传这三板斧。

地基有多不可靠：IP 层的问题清单

  • 丢包：路由器缓冲区满了会直接丢弃报文。
  • 乱序：不同报文可能走不同路径，后发的可能先到。
  • 重复：网络重传机制可能导致同一个包被送达两次。
  • 损坏：物理链路噪声可能导致比特翻转。

TCP 必须在传输层单独解决这些问题，因为下面的 IP 层完全不管。

机制一：序号（Sequence Number）

TCP 把要发送的数据看作一条连续的字节流，每一个字节都有一个编号。 报文段首部里的 seq 字段，记录的是这个报文段第一个字节在整条流中的编号，而不是"这是第几个报文段"。

类比：就像寄一本厚书，不是按"第几页信封"编号，而是直接在每一页上标好在全书中的页码——即使信封顺序打乱了，收件人也能凭页码把整本书正确拼回去。

有了序号，接收方就能做两件事：把乱序到达的数据重新排好顺序，以及发现重复的数据并丢弃。

机制二：确认应答（ACK）

接收方每收到数据，会回复一个 ACK 报文，里面的确认号 ack 表示"我已经连续收到到编号 ack-1 为止的所有字节，请从 ack 开始发"。 这是一种累积确认（Cumulative ACK）——只确认"连续收到"的部分，中间如果有缺口（比如收到了 1-100 和 201-300，但 101-200 没到），ack 只能停在 101，不能跳过缺口去确认后面已经收到的字节。

陷阱：很多初学者以为 ACK 是"逐包确认"，其实 TCP 的 ACK 号代表的是"这个号之前的字节流全部连续收到"，这也是为什么后面"快速重传"要靠重复 ACK 来判断丢包位置。

机制三：超时重传（Timeout Retransmission）

发送方每发出一个报文段都会启动一个定时器，如果在超时时间（RTO, Retransmission TimeOut）内没有收到对应的 ACK，就认为这个报文段丢了，重新发送一次。

RTO 设多长是个技术活：太短会把正常延迟误判为丢包，造成不必要的重传，浪费带宽；太长又会在真正丢包时反应迟钝，拖累整体吞吐。 TCP 使用基于往返时间（RTT, Round-Trip Time）的动态估算算法（如经典的 Jacobson 算法）， 实时采样 RTT 及其抖动（方差），让 RTO 能够自适应网络状况的变化。

机制四：快速重传（Fast Retransmit）

超时重传有一个明显缺点：要等到定时器超时才会发现丢包，这个等待时间通常比 RTT 长得多，对性能是很大的拖累。 快速重传是一种更灵敏的丢包检测机制：

发送方发出: seq=1 seq=2(丢失) seq=3 seq=4 seq=5 接收方收到: 1 (无) 3 4 5 接收方回复: ACK=2 ACK=2 ACK=2 ACK=2 ← 连续 3 个重复 ACK 发送方逻辑: 一旦收到 3 个重复的 ACK，不再等超时定时器， 立即重传 seq=2，无需等待 RTO 到期

当接收方发现收到的数据出现空缺（比如收到 3、4、5 但缺 2），它会针对每个新到的乱序报文重复发送同一个 ACK（都要求"从 2 开始发"）。 发送方一旦连续收到 3 个重复 ACK，就判定 seq=2 大概率丢了，不必等待超时定时器，立刻重传，大幅缩短了丢包恢复时间。

与本 Tab 前面知识点的联系

序号机制正是三次握手里"交换初始序号（ISN）"要做的事——双方从一开始就要就"字节流从哪个编号开始计数"达成一致，后续所有的确认应答才有意义。 而超时重传的存在，也解释了为什么四次挥手中的 TIME_WAIT 要等待 2MSL：如果最后一个 ACK 丢了，被动关闭方会因为超时重传 FIN，这本质上正是同一套"超时重传"机制在断连阶段的体现。

常见误区

误区："TCP 可靠是因为它会一直重传直到成功，所以数据永远不会丢。" 重传次数是有上限的（由操作系统配置，如 Linux 的 tcp_retries2），超过上限后 TCP 会放弃并通知应用层"连接出错"。 TCP 保证的是"要么完整可靠地送达，要么明确地报告失败"，而不是"无论如何都能送达"——网络分区、对端宕机等情况下,可靠传输也无能为力，只能上报错误交给应用层处理。

核心价值总结

  • 序号解决"顺序和去重"问题，确认应答解决"对方收到没有"问题，超时重传兜底"确认丢了怎么办"，快速重传则是给超时重传的一个更灵敏的加速版。
  • 这四个机制组合在一起，才在不可靠的 IP 层之上撑起了"可靠传输"这四个字。
