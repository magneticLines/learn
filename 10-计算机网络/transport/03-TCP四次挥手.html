TCP 四次挥手

建立连接握三次手，断开连接却要挥四次手——这个"多出来的一次"恰恰是理解 TCP 全双工本质的关键，也是面试官最爱刨根问底的地方。

先明确：TCP 连接是全双工的

TCP 连接一旦建立，客户端和服务端都可以独立地向对方发送数据，这叫全双工（full-duplex）。 正因如此，"关闭连接"其实是两个方向各自独立关闭的过程：我说完了不代表你也说完了。 这正是四次挥手比三次握手多一次的根本原因。

完整过程：FIN → ACK → FIN → ACK

客户端（主动关闭方） 服务端（被动关闭方） │ 状态: ESTABLISHED 状态: ESTABLISHED │ │ │──①FIN, seq=u──────────────────────▶│ 状态: CLOSE-WAIT │ 状态: FIN-WAIT-1 │ （服务端可能还有数据要发） │ │ │◀────②ACK, ack=u+1───────────────────│ │ 状态: FIN-WAIT-2 │ │ │ │ (服务端把剩余数据发送完毕) │ │ │ │◀────③FIN, seq=v─────────────────────│ 状态: LAST-ACK │ 状态: TIME-WAIT │ │ │ │──④ACK, ack=v+1─────────────────────▶│ 状态: CLOSED │ │ │ 等待 2MSL 后... │ │ 状态: CLOSED │
  • 第一次挥手（FIN）：客户端数据发送完毕，发送 FIN=1, seq=u，进入 FIN-WAIT-1，表示"我这边没数据要发了，但我还能收"。
  • 第二次挥手（ACK）：服务端收到 FIN，立即回复 ACK, ack=u+1 确认，进入 CLOSE-WAIT；客户端收到后进入 FIN-WAIT-2，等待服务端也关闭。
  • 第三次挥手（FIN）：服务端把自己剩余的数据发送完毕后，才发送 FIN=1, seq=v，进入 LAST-ACK，表示"我这边现在也说完了"。
  • 第四次挥手（ACK）：客户端收到后回复 ACK, ack=v+1，进入 TIME-WAIT，等待 2MSL 后进入 CLOSED；服务端收到 ACK 后立即进入 CLOSED。

核心考点：为什么比三次握手多一次？

因为服务端收到客户端的 FIN 时，往往还有数据没发完，不能立刻关闭自己这个方向的连接。 握手时服务端可以把"确认收到 SYN"和"我也要建立连接"这两件事合并成一个 SYN-ACK 报文一起发出， 因为这两件事是同时决定的；但挥手时，"确认收到你的 FIN"和"我自己也准备好关闭了"往往不是同一时刻发生的事情—— 服务端需要先把 CLOSE-WAIT 期间积压的数据发完，才能发出自己的 FIN。因此 ACK 和 FIN 通常无法合并，必须拆成独立的两步，也就凑成了四次。

特殊情况：如果服务端在收到 FIN 时恰好也没有数据要发送了，某些实现会把第二、三次挥手合并发送（ACK+FIN 一起），这时看起来就像"三次挥手"，但这只是优化，不是协议规定的次数。

核心考点：TIME_WAIT 存在的意义

主动关闭方（先发 FIN 的一方）在发出最后的 ACK 后，不会立刻进入 CLOSED，而是停留在 TIME_WAIT 状态，等待 2 倍 MSL（Maximum Segment Lifetime，报文最大生存时间）。这不是多余的设计，而是有两个明确目的：

  1. 保证最后一个 ACK 能够到达对方。 如果这个 ACK 丢失，被动关闭方会因为超时没收到确认而重传 FIN。此时主动关闭方必须还处于能响应的状态（TIME_WAIT）才能再次回复 ACK；如果它已经直接 CLOSED，收到这个重传的 FIN 只会回复 RST，让对方以为出错。
  2. 让本次连接产生的所有报文段在网络中彻底消失。 经过 2MSL，本连接中滞留在网络里的任何延迟到达的旧报文段都会因为超过生存时间被路由器丢弃，从而避免它们被误认为是"下一个使用相同四元组（源IP、源端口、目的IP、目的端口）的新连接"的数据，产生数据错乱。

常见误区

误区一："TIME_WAIT 是被动关闭方的状态。" 恰恰相反，TIME_WAIT 出现在主动发起关闭的一方（谁先发 FIN，谁最后进入 TIME_WAIT）。被动关闭方在发出自己的 FIN 并收到对方的 ACK 后就直接进入 CLOSED。
误区二："TIME_WAIT 太多是个 bug。" 高并发短连接服务器（如某些 HTTP 服务）确实容易积累大量 TIME_WAIT 连接占用端口和内存，这是真实的运维问题，但解决方式是调整连接复用策略（长连接/连接池）或调整内核参数（如 tcp_tw_reuse），而不是消灭 TIME_WAIT 本身——它的存在是协议正确性的保证，强行缩短或跳过会引入数据错乱风险。
面试高频："为什么是 2MSL 而不是 1MSL 或 3MSL？" 1MSL 保证 FIN 最迟能到达对方，再加 1MSL 保证对方重传的 FIN（如果第一次的 ACK 丢了）最迟能返回到自己——两个方向各留一次报文的最大生存时间，2MSL 正好覆盖"最坏情况下一来一回都不丢"所需的时间。

核心价值总结

  • 四次挥手的根源：TCP 全双工特性决定了两个方向的关闭必须独立进行，FIN 和 ACK 通常无法合并。
  • TIME_WAIT 不是缺陷，是正确性保障：它用主动方等待的代价换来了"最后的 ACK 一定有效、旧连接的数据不会串到新连接里"。
  • 握手三次是为了防止旧连接请求造成资源浪费，挥手四次是为了应对全双工下双方各自独立关闭——两者的"多一次/少一次"都不是随意设计，而是各自场景下的必要代价。
