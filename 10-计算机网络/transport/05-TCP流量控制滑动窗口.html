TCP 流量控制（滑动窗口）

上一篇解决了"数据丢了怎么发现、怎么补"，但还有一个问题没解决：发送方能不能敞开了发？如果接收方处理不过来，疯狂发送只会把对方的缓冲区撑爆。滑动窗口就是解决这个问题的机制。

问题：发送方凭什么"悠着点发"

如果 TCP 只有序号和确认应答，一种最简单的做法是"发一个字节，等一个 ACK，再发下一个"——这叫停等协议（Stop-and-Wait）， 正确性没问题，但效率极低：每次都要等一个完整的 RTT 才能发下一批数据，链路大部分时间都在空闲。

另一个极端是"不管对方死活，数据只管往外发"——这样吞吐是上去了，但如果接收方处理速度跟不上（比如接收方 CPU 繁忙、应用层没来得及从缓冲区读走数据）， 接收缓冲区很快被填满，后续到达的数据只能被丢弃，反而造成大量无谓的重传。

滑动窗口机制就是在这两个极端之间找的平衡点：允许发送方一次性发送多个报文段而不必逐个等待确认，但发送的总量不能超过接收方当前还能承受的数据量。

滑动窗口如何工作

接收方在每个 ACK 报文里都会带上一个接收窗口（rwnd, Receive Window）字段， 告诉发送方"我这边的接收缓冲区还剩多少空间，你最多还能再发这么多字节过来，不需要逐个等确认"。

发送方缓冲区（字节流视角）： ┌──────────┬─────────────────────┬─────────────┐ │ 已发送已确认 │ 已发送未确认 + 可发送未发送 │ 不可发送 │ └──────────┴─────────────────────┴─────────────┘ ▲←──── 发送窗口 (由 rwnd 决定) ────→▲ 窗口左边界 窗口右边界 收到新 ACK → 窗口向右"滑动"（左边界右移，确认过的部分被移出窗口） 接收方处理能力下降 → rwnd 变小 → 窗口右边界收缩，发送方降速

随着 ACK 不断到达，窗口在字节流上像一把"可伸缩的取景框"一样向右滑动——这就是"滑动窗口"名字的由来。 发送方在窗口范围内的数据可以连续发出，不需要每发一个字节就停下来等确认，大大提升了吞吐率，同时又不会让接收方超载。

核心区分：流量控制 vs 拥塞控制

这是一个极高频的面试考点，很多人把两者混为一谈，其实它们保护的对象完全不同：

维度流量控制（Flow Control）拥塞控制（Congestion Control）
保护对象接收方（防止接收缓冲区溢出）整个网络（防止链路和路由器过载）
依据信息接收方主动通告的接收窗口 rwnd发送方根据丢包、延迟等信号自己估算的拥塞窗口 cwnd
是点对点还是全局的点对点：只关心这条连接对端的处理能力全局性：网络中间的路由器、链路带宽都可能是瓶颈，与本连接无关的其他流量也会造成影响
最终生效窗口实际允许发送的数据量 = min(rwnd, cwnd)，两者中更小的那个说了算
记忆技巧：流量控制是"一对一"的绅士协定（我告诉你我能吃多少，你就喂多少），拥塞控制是"多对一共享资源"的博弈（大家都在抢同一条网络管道，谁也不知道别人发了多少）。下一篇会详细展开拥塞控制的具体算法。

常见误区与边界情况

误区："窗口越大，性能一定越好。" 窗口大小和带宽时延积（BDP, Bandwidth-Delay Product = 带宽 × RTT）密切相关：如果窗口小于 BDP，链路的带宽没有被充分利用（发送方在等 ACK，但链路其实还有余量）；这也是为什么高带宽高延迟网络（长肥管道）需要更大的 TCP 窗口（涉及 TCP 窗口扩大选项 Window Scaling），否则吞吐会被严重限制。
陷阱：糊涂窗口综合征（Silly Window Syndrome）。 如果接收方每次只腾出很小一点缓冲区空间就立刻通告一个很小的非零窗口（比如 1 字节），会导致发送方发送大量微小的报文段，首部开销远大于实际数据，效率极低。 解决办法是接收方遵循 Clark 算法：只有当缓冲区空出的空间达到一定阈值（如 MSS 大小或缓冲区一半）才通告新窗口；发送方也配合 Nagle 算法：小数据量时先攒一攒，凑够一个较大的报文段或等到收到前一个 ACK 再发，避免发送过多小包。

与前面知识点的联系

滑动窗口的正确运作，建立在上一篇讲的序号机制之上——窗口的左右边界本质上就是字节流序号上的两个游标； 而窗口内报文丢失后的快速重传机制，也是与滑动窗口配合工作的：当某个报文段丢失，后续报文段仍在窗口内被发送方持续发出，接收方靠重复 ACK 提示发送方尽快重传空缺部分，而不必让整个窗口停摆。

核心价值总结

  • 滑动窗口解决"发送方一次能发多少"的效率问题，同时避免压垮接收方。
  • 流量控制保护接收方，拥塞控制保护网络，两者作用范围不同，缺一不可。
