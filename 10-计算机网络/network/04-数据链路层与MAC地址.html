网络层与链路层

数据链路层与 MAC 地址

前三篇讲的都是网络层——IP 地址、路由、NAT，负责跨网络的"全局寻址与转发"。但数据包真正要从一根网线、一段 Wi-Fi 信号里传出去，还要靠更底层的数据链路层来完成"最后一米"的搬运。

数据链路层的定位：负责相邻节点间的数据帧传输

网络层的 IP 协议解决的是"整个互联网范围内，这个包该经过哪些路由器转发到目的地"，是一种全局性的、跨网络的寻址。 但路由器和路由器之间、路由器和你的电脑之间，具体是怎么把比特流实实在在地送过一根网线或一段无线信号的？这就是数据链路层（Data Link Layer）的职责范围。

数据链路层只负责同一条链路上相邻两个节点之间的数据传输——注意这里的关键词是"相邻"：链路层完全不知道、也不关心"最终的目的地在哪"，它只管把数据以"帧（Frame）"的形式，从链路的一端送到另一端。 每次一个 IP 数据包要经过一台路由器，链路层的寻址信息都会被重新计算一遍（下一跳是谁，就重新查一次那一跳的物理地址）。

类比：如果说 IP 地址是快递单上写的"收件人完整地址"（全局唯一，指引包裹跨越城市和国家），那 MAC 地址就像"当前这一段运输路上，司机要把包裹亲手交给的下一个具体的人"——每一段运输可能换一个司机、换一个交接对象，但收件人地址（IP）自始至终不变。

MAC 地址：物理网卡的全球唯一标识

MAC 地址（Media Access Control Address），也叫物理地址或硬件地址，是烧录在网卡（无论是有线网卡还是 Wi-Fi 网卡）里的一串 48 位（6 字节）标识符， 通常写作 00:1A:2B:3C:4D:5E 这样的十六进制格式。理论上它在全球范围内是唯一的——因为地址的前 24 位由 IEEE 分配给各个网卡制造商（叫组织唯一标识符 OUI），后 24 位由厂商自行编号，两段组合起来保证不重复。

数据链路层封装的"帧"里包含源 MAC 地址和目的 MAC 地址：当一台设备要发送数据帧，它需要知道"这一跳该发给谁的网卡"，这就是下一篇要讲的 ARP 协议要解决的问题——如何用已知的 IP 地址查出对应的 MAC 地址。

MAC 地址 vs IP 地址：分工不同，缺一不可

这是初学者最容易混淆的一组概念——两者都叫"地址"，但工作在不同的层，解决的是完全不同的问题：

对比维度MAC 地址IP 地址
所属层次数据链路层网络层
作用范围局部有效，只在同一条物理链路（如同一个局域网）内有意义全局有效，用于跨网络路由寻址
是否随位置变化固定烧录在网卡硬件上，一般不随设备移动而改变随设备接入的网络变化而变化（换个 Wi-Fi 就换一个 IP）
寻址方式扁平结构，不能体现拓扑位置，不可用于跨网络路由层次化结构（网络号 + 主机号），可以据此做路由选择
类比身份证号（终身不变，但门卫不会靠身份证号找到你的门牌）门牌地址（描述你现在住在哪个小区哪栋楼，可以据此规划送货路线）

正因为 MAC 地址是"扁平的、不带拓扑信息的"，它没法像 IP 地址那样支持路由聚合和最长前缀匹配（回顾上上篇路由选择原理）——如果全球通信都靠 MAC 地址寻址， 每台路由器都要记住全世界几十亿设备的 MAC 地址才能转发，这是不可能实现的。这正是"网络层用 IP 做全局路由 + 链路层用 MAC 做局部传输"这套分层设计的价值所在： 用层次化的 IP 地址解决"怎么找到目标网络"，用扁平的 MAC 地址解决"这一跳具体交给谁"。

常见误区

误区一：以为 MAC 地址"永远不会变、能用来精确定位一台设备的物理位置"。MAC 地址虽然理论上全球唯一，但只在当前所处的局域网内有意义，出了这个网段就没有直接效力；而且现代操作系统（如 iOS、Android）出于隐私保护，在 Wi-Fi 扫描和连接时默认会使用"随机 MAC 地址"，同一台设备连接不同网络时表现的 MAC 地址可能并不相同。
误区二：把 MAC 地址和 IP 地址的分工搞反，以为路由器之间转发也是靠 MAC 地址一路不变地传到底。实际上数据包每经过一台路由器，链路层的源/目的 MAC 地址都会被重写一次（因为进入了新的一段链路），只有 IP 头里的源/目的 IP 地址在整个转发路径中保持不变。

核心价值总结

  • 数据链路层只管相邻节点间的帧传输，不关心全局路由，这是它和网络层最本质的边界。
  • MAC 地址局部有效、扁平寻址；IP 地址全局有效、层次化寻址——两者分工明确，缺一不可。
  • 理解了 MAC 与 IP 的分工，才能理解下一篇 ARP 协议"如何用 IP 查出 MAC"到底在解决什么问题。

与其他知识点的联系

  • 关联知识点：交换机的定向转发（下一篇）：交换机能不能做到"只把帧发给该收的那台设备"，前提正是链路层设备之间靠 MAC 地址互相识别——本篇讲清楚 MAC 地址是什么、为什么局部有效，下一篇讲的交换机 MAC 地址表机制才有依据可循。
  • 关联知识点：ARP 协议（下下篇）：网络层的应用程序、传输层协议全程只处理 IP 地址，从不知道对方网卡的 MAC 地址，链路层封装帧时却必须填上目的 MAC——这个"IP 已知、MAC 未知"的缺口正是 ARP 协议要填补的，本篇讲的 MAC 地址正是 ARP 查询要解析出来的目标。
  • 关联知识点：IP 地址与路由选择（前几篇）：本篇反复对比的"MAC 扁平寻址 vs IP 层次化寻址"，本质上呼应的是路由器为什么能用"网络号 + 最长前缀匹配"做全局路由，而 MAC 地址因为没有层次结构、无法聚合，天然只能用在局部链路——这也解释了为什么互联网骨干网不可能靠 MAC 地址转发。
  • 关联知识点：NAT（上一篇）：NAT 解决的是 IP 层"多台设备共享一个公网 IP"的问题，而 MAC 地址完全不参与这个转换——NAT 网关对内网设备而言仍然要靠 MAC 地址完成链路层的实际帧交付，两套地址体系各管一层，互不越界。
  • 关联知识点：ARP 欺骗与 HTTPS（安全 Tab）：MAC 地址的"扁平、无层次、可被随意声称归属"的特性，正是安全 Tab 里 ARP 欺骗攻击能够得手的物理层基础——理解了链路层设备如何被动信任对方报上来的 MAC 地址，才能理解为什么局域网内的身份伪造这么容易发生，也更能体会 HTTPS 证书体系"身份可验证"设计的必要性。
