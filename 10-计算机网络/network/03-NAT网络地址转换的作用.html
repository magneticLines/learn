NAT 网络地址转换的作用

第一篇提到 IPv4 公网地址早已枯竭，但你家里手机、电脑、平板明明都能同时上网——它们共享的是同一个公网 IP。这背后的功臣，就是 NAT。

问题的起点：IPv4 地址不够用了

还记得第一篇讲的吗：IPv4 只有 32 位地址空间，理论上限约 43 亿，早在 2011 年前后公网可分配地址就已见底。 但联网设备的数量早就远超这个数字——仅你家里可能就有手机、电脑、平板、智能音箱、摄像头等十几台设备同时需要联网。 如果每台设备都要占用一个独立公网 IP，这些地址根本不够分。NAT（Network Address Translation，网络地址转换）正是为了缓解这个矛盾而生的过渡方案。

NAT 的工作原理：多台内网设备共享一个公网 IP

核心思路：内网中的设备使用私网地址（上一篇讲的 192.168.x.x 等）通信，当流量要出内网访问公网时，由 NAT 网关（通常就是你家的路由器） 把数据包的源 IP + 源端口统一替换成路由器自己的公网 IP 加上一个临时分配的端口号，并在内部维护一张"转换表"记录这层映射关系；等公网响应回来时， 路由器再根据转换表反向替换，把数据包准确送回发起请求的那台内网设备。

NAT 转换表示例（简化版 PAT / NAPT）
// 内网设备 A 发起请求
源地址 192.168.1.10:53211  ->  目的地址 8.8.8.8:443

// 经过 NAT 网关转换后，公网上看到的是:
源地址 203.0.113.5:40001  ->  目的地址 8.8.8.8:443

// NAT 转换表记录了这条映射:
192.168.1.10:53211  <->  203.0.113.5:40001

// 响应包回来时，NAT 网关查表，把目的地址换回 192.168.1.10:53211

这种"用端口号区分多个内网设备"的方式严格来说叫 NAPT（Network Address Port Translation），也常被叫做 PAT，是现在家用路由器最常见的 NAT 实现方式—— 因为端口号有 16 位（0~65535），一个公网 IP 理论上可以同时支撑 6 万多条并发的内网连接映射，这就是为什么一个公网 IP 能"喂饱"一整个家庭甚至一整个小区的设备。

类比：NAT 就像公司的总机前台——公司内部每个员工有自己的分机号（私网地址），但对外只暴露一个总机号码（公网 IP）。外部客户打总机，前台根据"当前正在通话中的分机记录"把电话转接给对应的员工；员工也可以自己拨出去，前台记下"是哪个分机在跟外面通话"，方便对方回电时能转对人。

NAT 带来的问题：内网设备无法被公网直接访问

NAT 转换表的建立依赖于"内网设备主动发起连接"——正是这一点，带来了 NAT 最大的副作用：公网上的其他主机无法主动向内网设备发起连接， 因为在没有先发出请求之前，NAT 网关的转换表里根本没有这台内网设备的映射记录，网关不知道该把外部来的包转发给谁。

常见场景：你想在家里搭一台服务器，让公司同事直接访问，结果发现外部根本连不进来——这就是因为请求是外部主动发起的，NAT 网关没有对应的映射记录，只能丢弃或拒绝。解决办法是手动配置端口映射（Port Forwarding），让路由器把某个公网端口固定转发给内网某台设备的某个端口。

P2P 打洞技术简介

这个限制对于"两个都在 NAT 后面的用户想直接建立点对点连接"（比如视频通话、文件直传、游戏联机）是个大麻烦——双方互相都是对方眼里"连不进来的内网设备"。 为了解决这个问题，业界发展出了 NAT 穿透（NAT Traversal）技术，其中最常见的思路叫 打洞（Hole Punching）：

  1. 双方各自先向一个公网上的第三方服务器（STUN 服务器）发送请求，从而让自己的 NAT 网关生成一条"该设备曾经主动访问过外部"的映射记录（相当于在 NAT 上"开了一个洞"）；
  2. STUN 服务器把双方各自在公网上暴露的"IP:端口"信息互相转告对方；
  3. 双方几乎同时向对方刚刚暴露的"IP:端口"发送数据包——因为各自的 NAT 网关此前已经"预期"会有外部流量进来（洞已经打开），这次尝试就有较大概率被放行，从而绕开 NAT 直接建立点对点连接。

这项技术不能保证 100% 成功（取决于 NAT 类型是"锥形"还是"对称型"等实现差异），打洞失败时通常会退化为通过一台公网中继服务器转发流量（TURN），这也是 WebRTC 等实时通信技术底层依赖的核心机制之一。

与前两篇的呼应

NAT 的存在本质上是给 IPv4 地址不够用这个"病"开的一剂"缓释药"——它没有从根本上解决地址空间不足的问题（真正的根治方案是上一篇提到的 IPv6）， 但极大延缓了公网地址枯竭的速度，也是为什么 20 年过去了，互联网还能在 IPv4 为主的体系下正常运转的关键原因。理解 NAT，也是理解"为什么内网穿透""为什么打游戏容易连不上好友"这些日常问题的钥匙。

常见误区

误区一：把 NAT 当作安全机制。NAT 客观上让外部无法直接连入内网设备，看起来像"防火墙"，但它的设计初衷是解决地址不足问题，不是安全隔离——不能替代真正的防火墙和访问控制策略。
误区二：认为只要是私网 IP 就绝对无法从公网访问。只要通过端口映射、UPnP 自动映射或者打洞成功，私网设备依然可以被间接或直接访问到，"私网地址"和"不可访问"不是绝对的因果关系。

核心价值总结

  • NAT 让多台私网设备共享一个公网 IP，是缓解 IPv4 地址枯竭的关键过渡方案，NAPT/PAT 靠端口号区分不同设备的连接。
  • 代价是内网设备无法被公网主动访问，需要端口映射或打洞技术才能建立反向或点对点连接。
  • P2P 打洞是应对双方都在 NAT 后面这一常见场景的工程手段，是理解 WebRTC、游戏联机等实时通信技术的基础。

与其他知识点的联系

  • OSI 七层模型与 TCP/IP 四层模型：NAT 是少数会跨层改写数据的机制——NAPT 既要改网络层的 IP 头（源地址），也要改传输层的端口号，这也是为什么严格来说它叫"网络地址端口转换"而不只是"网络地址转换"，单纯站在某一层的视角都无法完整描述它做的事。
  • 路由选择原理：NAT 转换和路由选择往往发生在同一台设备（家用路由器）上，但职责边界很清晰——先由 NAT 决定要不要改写这个包的源地址和端口，再由路由模块查路由表决定从哪个接口转发出去，两者是先后接力而不是二选一的关系。
  • 客户端-服务器与 P2P 架构：P2P 架构追求"节点之间直接通信"，但 NAT 的存在恰恰破坏了这个前提——两个都在 NAT 后面的 peer 互相都连不进对方，必须靠 STUN 服务器辅助打洞，这也解释了为什么现实中很少有"纯粹"的 P2P 系统，多少都要依赖一点 C/S 组件（引导服务器）来完成初次连接。
  • DNS 域名解析：DNS 只负责把域名翻译成 IP 地址，但如果这个 IP 是某台 NAT 网关的公网 IP，光有 IP 还不够连到具体的内网服务——这也是为什么想让公网直接访问家里搭的服务器，除了要有稳定可解析的 IP/域名，还必须额外配置端口映射，DNS 解析和 NAT 穿透是两个独立但缺一不可的环节。
  • HTTP 请求响应报文结构：NAT 只改写 IP 包头和传输层端口，完全不touch应用层的 HTTP 报文内容——报文里的 Host 头、body 数据在经过 NAT 网关时是原样透传的，这体现了 NAT 工作在网络层+传输层、和应用层数据互不干扰的分层边界。
