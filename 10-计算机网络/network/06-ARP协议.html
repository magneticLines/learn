网络层与链路层

ARP 协议

前两篇分别讲了 MAC 地址（链路层，局部寻址）和交换机基于 MAC 地址表转发。但一个问题还没有答案：你的电脑只知道要访问的服务器/网关的 IP 地址，它是怎么知道该往哪个 MAC 地址发数据帧的？这正是 ARP 协议要解决的问题。

问题的起点：知道 IP，不知道 MAC

回顾第四篇讲的分工：网络层用 IP 地址做全局路由，链路层用 MAC 地址做同一链路内的定向传输。但发送数据帧时，链路层帧头里必须填入目的 MAC 地址—— 问题是，应用程序、传输层、网络层从头到尾都只在处理 IP 地址，从来不知道对方的 MAC 地址是什么。这中间缺了一环："已知 IP 地址，怎么查出对应的 MAC 地址？"

ARP（Address Resolution Protocol，地址解析协议）正是为了填上这一环而设计的：它工作在网络层和链路层之间，专门负责完成"IP 地址 → MAC 地址"的映射查询。

ARP 的工作原理：广播请求 + 单播应答

ARP 的机制非常直接，可以概括为四个字：广播问、单播答。假设主机 A（192.168.1.10）要给同一局域网内的主机 B（192.168.1.20）发送数据，但不知道 B 的 MAC 地址：

  1. 广播 ARP 请求：主机 A 在整个局域网内广播一个 ARP 请求包，大意是"谁的 IP 是 192.168.1.20？请把你的 MAC 地址告诉我"。这个请求会被发送到广播地址（FF:FF:FF:FF:FF:FF），局域网内的所有设备都会收到这个包。
  2. 其余设备忽略：局域网里除 B 之外的所有设备，检查请求中的目标 IP 与自己不符，直接丢弃这个包，不做任何响应。
  3. 目标单播应答：只有主机 B 发现"这个请求问的正是我的 IP"，于是直接以单播（点对点）方式回复一个 ARP 应答包，把自己的 MAC 地址告诉 A。
  4. A 缓存结果：主机 A 收到应答后，把这条"IP-MAC"映射记录缓存起来，之后就可以直接封装数据帧发送，不用每次都重新广播询问。
类比：ARP 就像在一间教室里大喊一声"谁的学号是 20号？请举手告诉我你的座位在哪"——全班都听到了这句话，但只有 20 号学生会回应"我在这里，第三排靠窗"。其他同学听到问题跟自己无关，直接无视。

ARP 缓存表：避免每次都重新广播

如果每发一个包都要重新广播一次 ARP 请求，会给局域网带来大量不必要的广播流量。因此每台设备都会维护一张ARP 缓存表（ARP Cache）， 把最近查询到的"IP-MAC"映射关系缓存一段时间（通常几分钟到几十分钟不等，取决于操作系统实现），后续访问同一个 IP 时直接查缓存，不再发起新的 ARP 广播。

查看本机 ARP 缓存表（Windows / Linux 通用思路）
$ arp -a

Internet Address      Physical Address       Type
192.168.1.1           aa-bb-cc-dd-ee-ff       dynamic
192.168.1.20          11-22-33-44-55-66       dynamic

// dynamic 表示这条记录是 ARP 自动学习得到的，有过期时间
// 也可以手动配置 static 静态记录，但局域网环境下很少这么做

缓存过期后，如果还需要访问该 IP，会重新触发一次 ARP 请求-应答流程。这种"缓存 + 过期重新查询"的模式，本质上和 DNS 缓存的思路是相通的（呼应"分层与应用层"Tab 里 DNS 解析那一篇）——都是用空间换时间，避免每次通信都重复一次昂贵的查询过程。

ARP 欺骗攻击的原理

ARP 协议设计之初完全没有考虑安全性——它假设网络里的每个设备都诚实作答，没有任何身份验证机制，任何设备都可以随意发送 ARP 应答包，声称"某个 IP 对应的 MAC 地址是我"， 而接收方几乎没有办法验证这个应答是否真实可信。这个设计缺陷催生了经典的攻击手法——ARP 欺骗（ARP Spoofing / ARP 中毒）：

  • 攻击者在局域网内主动发送伪造的 ARP 应答包，冒充网关（比如告诉大家"我才是 192.168.1.1"）；
  • 受害设备的 ARP 缓存被这条虚假记录"投毒"，之后所有原本要发给网关的流量，实际上都被发到了攻击者的机器上；
  • 攻击者可以选择直接丢弃这些流量（造成断网），也可以先截获、窃听甚至篡改流量内容，再转发给真正的网关（中间人攻击，MITM）。
常见后果：公司或学校局域网内偶尔出现的"莫名其妙全员断网"或"网速异常、账号密码疑似被盗"，很多时候就是有人在局域网内做了 ARP 欺骗。防护手段包括交换机启用动态 ARP 检测（DAI）、给关键设备配置静态 ARP 绑定、使用支持加密的协议（如 HTTPS）确保即使流量被中间人截获也无法被轻易篡改或窃听。

与本 Tab 前几篇的呼应

到这里，"网络层与链路层"这个 Tab 的知识点可以串成一条完整的链路：IP 协议负责跨网络的全局寻址和转发（第 1 篇），路由器靠路由表和最长前缀匹配选路（第 2 篇）， NAT 让多台内网设备共享一个公网 IP（第 3 篇），而在局域网内部，MAC 地址标识具体设备（第 4 篇），交换机靠 MAC 地址表做定向转发（第 5 篇）， ARP 协议则是连接"网络层的 IP"和"链路层的 MAC"之间的桥梁——没有 ARP，网络层和链路层这两套地址体系就是孤立的两张皮，数据包根本没法被真正封装成帧发送出去。

常见误区

误区一：以为 ARP 能跨网络工作，可以用来查询远程服务器的 MAC 地址。ARP 请求是以广播形式发送的，广播包不会被路由器转发到其他网络，因此 ARP 只在同一个局域网（同一个广播域）内有效——访问跨网络的目标时，主机实际查询的是"网关的 MAC 地址"，而不是最终目标服务器的 MAC 地址。
误区二：把 ARP 和 DNS 搞混。两者虽然都是"已知 A 求 B"的地址解析协议，但工作层次和范围完全不同：DNS 是应用层协议，解决的是"域名 → IP 地址"的全局解析；ARP 是链路层协议，解决的是"IP 地址 → MAC 地址"的局域网内解析。

核心价值总结

  • ARP 用"广播请求 + 单播应答"完成 IP 到 MAC 的地址解析，是打通网络层和链路层的关键胶水协议。
  • ARP 缓存表避免了重复广播，是"用空间换时间"思路在网络协议设计中的又一次体现。
  • ARP 协议缺乏身份验证机制，是 ARP 欺骗攻击得以存在的根本原因，也是局域网安全防护中的重要一环。
