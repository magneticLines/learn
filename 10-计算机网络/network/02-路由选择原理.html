网络层与链路层

路由选择原理

上一篇讲了 IP 协议只负责"尽力转发"，但转发到底怎么"选路"？一个数据包从你的电脑到大洋彼岸的服务器，中间要经过十几个路由器，每一跳都要做一次决策——这就是路由选择要解决的问题。

路由表：路由器做转发决策的依据

路由器本质上是一台专用于转发数据包的设备，它内部维护一张路由表（Routing Table），每次收到一个 IP 数据包，就查这张表决定"该从哪个网络接口把包发出去"。 路由表的每一行大致包含三个关键信息：

字段含义
目的网络（Destination）一个网络前缀，如 192.168.1.0/24
下一跳（Next Hop）要把包转发给哪个相邻路由器的 IP
出接口（Interface）从路由器的哪个物理/逻辑端口发出去
类比：路由表就像快递分拨中心的路线图——收到一个包裹，看收件地址属于哪个片区，就把它扔到对应片区的下一个转运车上，一站接一站传递，直到最后一个分拨点直接送到收件人手中。路由器不需要知道"完整路径"，只需要知道"下一步该交给谁"。

最长前缀匹配：路由表查找的核心原则

路由表里可能同时存在多条能匹配同一个目的 IP 的记录，比如同时有 0.0.0.0/0（默认路由，匹配一切）和 192.168.1.0/24（更精确的路由）。 这时路由器该选哪一条？答案是最长前缀匹配（Longest Prefix Match）：在所有能匹配目标地址的路由项里，选择网络前缀（掩码位数，即 /24 里的数字）最长、也就是匹配最精确的那一条。

示例：目标地址 192.168.1.100 命中哪条路由
// 路由表中同时存在:
0.0.0.0/0        -> 默认路由，走出口网关
192.168.0.0/16   -> 前缀 16 位匹配
192.168.1.0/24   -> 前缀 24 位匹配，最精确

// 三条路由对 192.168.1.100 都能匹配，
// 但 /24 的前缀最长，最终选择这一条转发

这样设计的意义在于：路由表不需要为每一个具体 IP 都存一条记录（那样表会大到无法维护），而是可以用"粗粒度的默认路由兜底 + 细粒度路由做例外"的方式组织， 既节省了路由表空间，又保证了转发的精确性——这也是 CIDR（无类别域间路由）能够工作的根本机制。

静态路由 vs 动态路由

路由表里的条目是怎么来的？有两种方式：

静态路由（Static Routing）

由网络管理员手动配置固定的路由条目。优点是简单、可控、不消耗额外带宽和计算资源；缺点是网络拓扑一旦变化（比如某条链路故障），静态路由不会自动更新，需要人工介入。 适合拓扑简单、很少变化的小型网络（比如公司内网出口只有一条线路）。

动态路由（Dynamic Routing）

路由器之间通过路由协议自动交换网络拓扑信息，动态计算并更新路由表，网络发生变化时能自动收敛到新的最优路径。互联网这种大规模、拓扑经常变化的网络，必须依赖动态路由算法。经典的两大类算法思路：

算法类型核心思路代表协议
距离矢量（Distance Vector） 每个路由器只知道"到每个目的地的距离（跳数）+ 该走哪个邻居"，定期把自己的路由表发给相邻路由器，靠"邻居的邻居"逐步扩散信息，类似"道听途说"式地拼凑全局认知 RIP
链路状态（Link State） 每个路由器主动向全网广播自己与直连邻居的链路信息，所有路由器因此都能拿到完整的网络拓扑图，再各自用最短路径算法（如 Dijkstra）独立计算出到每个目的地的最优路径 OSPF
面试高频：距离矢量算法存在经典的"无穷计数（count-to-infinity）"问题——当某条链路故障时，相邻路由器之间可能互相把对方过时的错误信息当作参考，导致路由收敛缓慢甚至震荡；链路状态算法因为每个节点都拥有全局拓扑视图，收敛更快也更不容易出现这种问题，但需要更多计算资源和带宽来广播拓扑信息。这是两类算法的核心取舍。

与传输层的呼应

回顾传输层 Tab 讲过的 TCP：三次握手建立的是"端到端的连接语义"，但这条"连接"背后走的物理路径，在网络层看来可能每个包走的路由都不完全一样（IP 是无连接的，不保证包按同一路径、同一顺序到达）。 路由选择只关心"下一跳去哪"，不维护"这个连接固定走哪条路"的状态——这也是网络层和传输层职责边界的一个直观体现。

常见误区

误区：以为路由器会为每个数据包计算出完整的端到端路径。实际上路由器只做"最长前缀匹配 + 决定下一跳"这一步决策，完整路径是所有路由器一跳一跳独立决策的结果，没有任何一个节点掌握全局路径信息（链路状态算法虽然掌握全局拓扑，但转发时依然是逐跳决策，不是提前规划好整条路径再发送）。

核心价值总结

  • 路由表 + 最长前缀匹配是路由器转发决策的核心机制，兼顾了表空间与精确性。
  • 静态路由简单可控，动态路由自动适应拓扑变化，大规模网络必须依赖动态路由协议。
  • 距离矢量 vs 链路状态是收敛速度、资源消耗、鲁棒性上的经典取舍，是很多路由协议设计的理论基础。

与其他知识点的联系

  • OSI 七层模型与 TCP/IP 四层模型：路由选择是网络层（第三层）的核心职责——路由器只解析 IP 头里的目的地址做转发决策，完全不关心传输层的端口号、也不关心应用层的 HTTP 报文内容，这是分层模型里"路由器这类中间设备只需要实现到网络层"的具体体现（对比主机需要实现完整的四层协议栈）。
  • DNS 域名解析：DNS 查询报文本身也是普通的 IP 数据包，要先经过路由器一跳一跳转发才能到达根/顶级域/权威服务器——路由选择是 DNS 整条递归+迭代查询链路能够物理送达的前提，没有路由能把包送到，再精妙的域名分层解析设计也无从谈起。
  • NAT 网络地址转换：NAT 转换通常就发生在路由器（家用网关）上，和路由选择是同一台设备做的两件不同的事——路由选择决定"这个包该往哪个接口转发"，NAT 决定"转发之前要不要先改写这个包的源地址和端口"，两者在处理顺序上通常是先做地址转换再查路由表决定出接口。
  • HTTP 协议的演进：HTTP 请求包无论是走 HTTP/1.1 的 TCP 还是 HTTP/3 的 QUIC/UDP，最终都要被拆成一个个 IP 数据包，靠路由选择逐跳送到目标服务器——路由层完全不区分传输层用的是 TCP 还是 UDP，它只认 IP 头。
  • 客户端-服务器与 P2P 架构：不管是 C/S 里客户端到服务器的单向请求，还是 P2P 节点之间的直连流量，数据包在网络层看到的都是无差别的转发任务——路由选择不理解、也不需要理解上层是什么架构范式，这正是分层解耦带来的好处。
