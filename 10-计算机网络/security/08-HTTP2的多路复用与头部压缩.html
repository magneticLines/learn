HTTP/2 的多路复用与头部压缩［HPBN 补充］

分层与应用层 Tab 讲过 HTTP 协议的演进历程，这里深入 HTTP/2 最核心的两个性能优化：二进制分帧带来的多路复用，以及 HPACK 头部压缩。

回顾问题：HTTP/1.1 的队头阻塞

HTTP/1.1 里，一条 TCP 连接上的请求是串行处理的——同一条连接上，前一个请求的响应没回来之前，下一个请求的响应无法先返回（这是应用层协议本身的限制，即使 TCP 层面数据已经乱序到达也没用）。 浏览器为了缓解这个问题，通常会对同一个域名开 6~8 条并发 TCP 连接来"并行"发请求，但这只是绕过问题而不是解决问题——连接数一多， 又会带来更多次 TCP 三次握手和 TLS 握手的开销（回顾传输层 Tab），而且服务器和中间网络设备维护连接本身也有成本。这种应用层的排队等待现象叫做队头阻塞（Head-of-Line Blocking）。

HTTP/2 的解法：二进制分帧层

HTTP/2 在应用层和传输层之间引入了一个新的二进制分帧层（Binary Framing Layer）。 它把每一个 HTTP 消息（请求或响应）拆分成一个个更小的帧（Frame），每个帧都带有一个流标识（Stream ID）， 标记这个帧属于哪一个逻辑上的请求-响应对。多个流的帧可以在同一条 TCP 连接上交织传输、乱序到达，再由接收端按 Stream ID 重新组装还原。

HTTP/1.1（同一连接上必须排队）：
[======请求A完整响应======][==请求B==][====请求C====]

HTTP/2（同一连接上交织并行）：
[A帧][B帧][C帧][A帧][C帧][B帧][A帧]... ──▶ 接收端按 Stream ID 重组
                                        ├─ 流A → 完整响应A
                                        ├─ 流B → 完整响应B
                                        └─ 流C → 完整响应C

这就是多路复用（Multiplexing）：一条 TCP 连接上可以同时并行处理多个请求/响应，某个请求的处理慢了不会挡住其他请求继续返回数据。 也因为一条连接就够用了，浏览器不再需要为同一域名开多条并发连接，减少了握手开销和服务器连接维护成本。

陷阱（面试高频）：HTTP/2 解决的是应用层的队头阻塞，但底层 TCP 是字节流、严格有序的协议——如果其中一个 TCP 报文在网络中丢失，TCP 层会要求重传，在收到重传数据之前，这条连接上所有流的后续数据都无法被应用层处理，哪怕它们已经先到达了。这就是"传输层的队头阻塞"，HTTP/2 治标不治本，真正解决要靠 HTTP/3 基于 QUIC 的每个流独立传输（下一篇详解）。

HPACK：头部压缩解决什么问题

HTTP 请求里有大量重复的头部字段——比如同一个浏览器发出的每个请求，User-Agent、Cookie、Accept-Language 等字段几乎一字不差地重复出现。 HTTP/1.1 里这些头部是纯文本，每次都要完整重传，随着 Cookie 越积越大，头部开销甚至可能超过请求体本身。

HTTP/2 引入 HPACK 压缩算法，核心思路有两个：

  • 静态表：预定义了一份包含常见头部字段（如 :method: GET、content-type 等）的表，双方都认识这份表，传输时直接用一个索引号代替完整字段名和值。
  • 动态表：连接建立后，双方会记住"之前发送过的头部字段"，如果后续请求的头部和之前重复，只需要传一个索引号，而不用重新传完整内容；只有第一次出现的新字段才需要完整传输(并加入动态表供后续复用)。
类比：HTTP/1.1 像是每次寄信都把发件人地址完整抄写一遍；HPACK 则像是双方提前约定"以后信里写个编号 3，就代表那串固定地址"，之后只需要写编号，大幅省下重复劳动。

常见误区

误区："HTTP/2 多路复用彻底解决了队头阻塞。" 只解决了应用层的队头阻塞。因为 HTTP/2 仍然跑在单条 TCP 连接上，TCP 的有序交付特性会导致"传输层队头阻塞"——这正是催生 HTTP/3 抛弃 TCP、改用基于 UDP 的 QUIC 协议的直接原因（呼应传输层 Tab 讲过的"为什么 HTTP/3 选择 QUIC"）。
面试高频："HTTP/2 相比 1.1 做了什么优化？" 答案要点：① 二进制分帧 + 多路复用，一条连接并行处理多个请求，缓解应用层队头阻塞；② HPACK 头部压缩，用静态表+动态表消除重复头部的传输开销；③ 支持服务器推送（Server Push，可以简单提及）。

核心价值总结

  • 二进制分帧层是 HTTP/2 的地基，把请求/响应拆成带 Stream ID 的帧，实现同连接内的多路复用。
  • HPACK 用静态表+动态表消除重复头部字段的传输冗余，尤其在 Cookie 较大时效果显著。
  • 多路复用解决了应用层队头阻塞，但 TCP 层面的队头阻塞依然存在，这是 HTTP/3 改用 QUIC 的根本动机。

与其他知识点的联系

  • TCP 严格有序交付：本篇提到的"传输层队头阻塞"，根源在于 TCP 把一条连接当作单一的字节流，只认序号不认逻辑请求——这是传输层 Tab 讲过的 TCP 可靠传输机制（序号确认、按序交付）决定的，不是 HTTP/2 能在应用层修复的问题，必须往下一层动刀才能根治。
  • QUIC 的每个流独立传输：HTTP/3 的 QUIC 协议把 HTTP/2 的"流"概念从应用层下沉到传输层原生实现，每个流有自己独立的序号空间和重传逻辑。这样一个流丢包只影响它自己，彻底消灭了本篇陷阱里说的"HTTP/2 治标不治本"的问题——理解这一篇的局限，才能理解下一篇 QUIC 为什么必须这样设计。
  • TLS 1.3 握手优化：HTTP/2 通常运行在 HTTPS 之上，一条连接建立前依然要走完 TCP 握手 + TLS 握手。多路复用能减少所需的连接数量（不再需要开 6~8 条连接），间接减少了握手开销的总次数，这和上一篇 TLS 1.3 压缩单次握手延迟是从两个不同维度共同优化连接建立成本。
  • 连接数与拥塞控制的关系：HTTP/1.1 靠开多条 TCP 连接绕过队头阻塞，但每条连接都要独立走一次慢启动（回顾传输层 Tab 拥塞控制），带宽利用率反而更低。HTTP/2 把请求收敛到一条连接，让这条连接更快地把拥塞窗口打满、达到链路真实带宽，这也是多路复用在吞吐量层面的隐性收益。
