HTTPS 的 TLS 握手过程

在 TCP 三次握手建立好可靠连接之上，TLS 还要再做一次"加密协商"的握手——这是面试中被问烂但很多人仍答不全的高频考点。以经典的 TLS 1.2 为例，逐步拆解。

为什么 TCP 握手之后还要再握手一次

回顾传输层学过的内容：TCP 三次握手（02-TCP三次握手）只解决了"双方确认彼此在线、约定好初始序号"， 建立的是一条明文可靠通道。HTTPS 要在这条通道之上再加一层，双方必须先协商好： 用什么加密算法、以及最关键的——用哪把对称密钥加密后续数据（回顾上一篇《对称加密与非对称加密》）。 这个协商过程就是 TLS 握手，握手成功后 HTTP 报文才开始加密传输，也就是我们说的 HTTPS。

TLS 1.2 握手全流程（2-RTT）

以下是经典 TLS 1.2 握手（RSA/ECDHE 密钥交换）的核心步骤：

客户端 服务端
1. ClientHello ────────────────────────▶
  (支持的 TLS 版本、加密套件列表、随机数 Client Random)

◀──────────────────────── 2. ServerHello
  (选定的 TLS 版本、加密套件、随机数 Server Random)
◀──────────────────────── 3. Certificate（服务器数字证书，内含公钥）
◀──────────────────────── 4. ServerHelloDone

  (客户端校验证书合法性 —— 见下一篇 CA 信任链)
  (客户端生成 Pre-Master Secret，用服务器公钥加密)
5. ClientKeyExchange ───────────────────▶
  (加密后的 Pre-Master Secret)
6. ChangeCipherSpec + Finished ──────────▶
  (通知：接下来我发的都用协商好的密钥加密了)

◀──────────────────────── 7. ChangeCipherSpec + Finished
  (服务器也确认切换到加密通信)

============ 双方开始用会话密钥加密传输 HTTP 数据 ============

每一步在做什么、为什么这么设计

  • ClientHello：客户端告诉服务器"我支持哪些 TLS 版本和加密套件（如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256）"，并附带一个随机数 Client Random。这个随机数之后会参与生成会话密钥——单靠一个固定密钥交换过程容易被预测或重放，加入双方各自的随机数能让每次会话的密钥都不同。
  • ServerHello + Certificate：服务器从客户端支持的套件里选一个（体现"双方协商"而非单方面决定），并返回自己的数字证书（内含公钥）和自己的随机数 Server Random。证书的作用是让客户端确信"这个公钥真的属于这个域名"，详见下一篇。
  • ClientKeyExchange：客户端校验证书合法后，生成第三个随机数 Pre-Master Secret，用服务器公钥（非对称加密）加密后发给服务器。这一步是非对称加密登场的地方——只有服务器的私钥能解出这个 Pre-Master Secret。
  • 密钥生成：双方各自用 Client Random + Server Random + Pre-Master Secret 通过同样的算法推导出同一把会话密钥（Master Secret）。三个随机数缺一不可地混合，是为了保证密钥的不可预测性和唯一性。
  • Finished：双方各自发一条用会话密钥加密的验证消息，对方能正确解密就说明密钥协商成功，可以正式开始加密通信了。

握手之后：切回对称加密

握手一结束，双方立刻抛开非对称加密（它已经完成使命——安全地协商出了会话密钥），改用对称加密加密所有后续 HTTP 报文。 这正是上一篇讲的"混合加密方案"在协议层面的具体落地：TLS 握手 = 用非对称加密走一遍密钥协商，握手结束后一路用对称加密。

握手的代价：额外的往返延迟

TLS 1.2 完整握手需要 2 个 RTT（往返时延）：先有 TCP 三次握手的 1 个 RTT，再加上 TLS 握手本身的 1 个 RTT（ClientHello 到 Finished 之间恰好来回一次）。 对首字节到达时间（TTFB）敏感的场景（比如首屏加载），这些额外的 RTT 累加起来很可观——这也是为什么 TLS 1.3 要专门优化握手轮数（见本 Tab 第 7 篇）。

常见误区与面试高频考点

误区一："HTTPS 握手全程都在用非对称加密。" 非对称加密只用在协商 Pre-Master Secret 这一步，真正传输 HTTP 数据用的是对称加密，这是性能和安全的平衡点。
误区二："服务器的私钥参与生成会话密钥的过程，所以私钥泄露会暴露历史流量。" 在 RSA 密钥交换模式下确实如此——这正是它逐渐被淘汰的原因；而 ECDHE（椭圆曲线 Diffie-Hellman）模式下，每次握手都用临时密钥对，即使私钥事后泄露也无法解密之前录下的流量，这个特性叫前向保密（Forward Secrecy），是现代 TLS 配置的标配。
面试高频："说一下 HTTPS 的握手过程" —— 按 ClientHello → ServerHello+证书 → 客户端验证证书并用公钥加密 Pre-Master Secret → 双方推导会话密钥 → Finished 确认 五步来说，再补一句"握手后切换成对称加密传输数据"，基本能覆盖考点。

核心价值总结

  • TLS 握手的本质是在不安全的信道上安全地协商出一把对称密钥。
  • 三个随机数（Client Random、Server Random、Pre-Master Secret）共同推导出会话密钥，保证唯一性和不可预测性。
  • 经典 TLS 1.2 握手需要额外 1 个 RTT，是后续协议优化（TLS 1.3、0-RTT）的直接动机。
