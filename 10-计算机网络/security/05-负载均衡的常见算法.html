负载均衡的常见算法

单台服务器扛不住流量时，负载均衡负责把请求"分派"给一群服务器。分派规则不同，适用场景也不同——这是后端架构面试的常客。

为什么需要负载均衡

一台服务器的处理能力和网络带宽是有上限的。当请求量超过单机承载能力，办法要么是"竖向扩展"（换更强的机器，总有天花板）， 要么是"横向扩展"（部署多台机器分摊流量）。横向扩展带来一个新问题：客户端的请求该发给哪一台？ 负载均衡器（Load Balancer）就是站在客户端和一组后端服务器之间，按某种策略把每个请求转发到某一台的组件。

常见负载均衡算法

1. 轮询（Round Robin）

请求按顺序依次分配给每台服务器，一圈轮完再从头开始。实现最简单，假设所有服务器性能均等时效果不错。

局限：如果某台机器配置更差或已经在处理慢请求，轮询依然一视同仁地分配同样多的请求，容易造成短板机器过载。

2. 加权轮询（Weighted Round Robin）

给每台服务器配置一个权重，权重越高分到的请求越多。比如三台服务器权重 5:3:2，每 10 个请求中就按这个比例分配。 解决了轮询"假设所有机器性能均等"的问题，适合异构集群（新旧机器混部）。

3. 最少连接数（Least Connections）

把新请求分配给当前处理连接数最少的服务器。相比轮询，它是"动态"的——实时感知每台机器的负载情况， 而不是按固定顺序机械分配。适合请求处理时长差异很大的场景（比如有的请求要处理几秒，有的几毫秒），轮询在这种场景下容易出现"忙的更忙"。

4. IP 哈希（IP Hash）

对客户端 IP 做哈希运算，结果映射到固定的一台服务器，同一个客户端的请求总是落到同一台机器上。 这解决了会话保持（Session Sticky）问题——如果业务逻辑依赖存在单机内存里的会话状态（比如未做分布式 Session 的登录态）， 同一用户的请求必须落在同一台机器，否则会出现"登录了但下一个请求又要求重新登录"的怪象。

陷阱：IP 哈希在真实场景中并不完美——大量用户可能共享同一出口 IP（比如公司/校园网做了 NAT），会导致负载分配不均；同时如果某台机器下线，哈希结果重新映射会导致大量原本固定在该机器的用户瞬间切换到别的机器，可能引发缓存失效等连锁问题。

算法对比一览

算法核心逻辑适用场景局限
轮询依次分配服务器性能相近、请求处理时长均匀不感知实际负载差异
加权轮询按权重比例分配异构集群（性能不同的机器混部）权重需要人工评估设置
最少连接数分给当前负载最低的机器请求处理时长差异大需要维护每台机器的实时连接状态
IP 哈希同一 IP 固定分到同一台依赖单机会话状态、需要会话保持分布不均、扩缩容时哈希漂移

四层负载均衡 vs 七层负载均衡

这是另一个正交的分类维度——负载均衡器在哪一层网络协议上做决策：

  • 四层负载均衡：工作在传输层，只看 IP + 端口（回顾传输层 Tab 学的 TCP/UDP），不解析应用层内容，直接转发数据包。优点是速度快、开销小；缺点是无法感知 HTTP 层面的信息（比如按 URL 路径路由）。典型实现：LVS。
  • 七层负载均衡：工作在应用层，会解析 HTTP 报文内容（URL、Header、Cookie 等），可以做更精细的路由决策，比如把 /api/* 转发给后端服务集群、把 /static/* 转发给静态资源服务器。代价是需要解析应用层协议，性能开销比四层大。典型实现：Nginx、HAProxy。
类比：四层负载均衡像快递分拣时只看"寄往哪个城市"就分流水线，不拆包裹；七层负载均衡则会拆开包裹看里面装的是什么类型的货物，再决定送到哪个具体仓库——更智能，但也更耗时。

常见误区

误区："负载均衡算法选一个'最好的'就行。" 没有放之四海皆准的最优算法，选择取决于机器是否同构、请求处理时长是否均匀、业务是否依赖会话状态。实际生产环境中常常是多种策略组合（比如七层按路径路由 + 每个路径内部用最少连接数）。
面试高频："负载均衡有哪些算法，怎么选？" 按"轮询→加权轮询→最少连接数→IP 哈希"讲清楚各自解决的问题即可；追问"四层和七层区别"，抓住"是否解析应用层内容"这个核心差异回答。

核心价值总结

  • 负载均衡的本质是横向扩展下"如何把请求合理分派给多台机器"的问题。
  • 算法选择要结合机器是否同构、请求耗时是否均匀、是否需要会话保持来综合判断。
  • 四层负载均衡快但"看不懂"应用层内容，七层负载均衡功能强但开销更大，二者常搭配使用。
