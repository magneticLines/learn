CDN 的工作原理

为什么访问大网站的静态资源(图片、JS、CSS)总是特别快，不管你人在北京还是广州？答案是 CDN——把内容缓存到离用户最近的地方。

问题：物理距离决定延迟下限

回顾传输层学过的内容，网络延迟很大一部分是光在光纤中传播的物理时间——这是硬性下限，再怎么优化协议也绕不开。 如果所有用户的请求都要打到部署在北京的源站服务器，广州用户访问，光是网络往返（RTT）就要比同城用户多出几十毫秒， 更别提如果源站服务器还有跨国用户。CDN（Content Delivery Network，内容分发网络）的核心思路就是：与其让用户舍近求远地跑到源站，不如把内容提前搬到用户"家门口"。

CDN 是怎么工作的：就近节点 + 缓存

CDN 服务商在全国乃至全球各地部署大量边缘节点（Edge Node），把源站的静态资源（图片、CSS、JS、视频分片等）提前复制并缓存到这些节点上。 用户请求资源时，并不是直接打到源站，而是被调度到离用户地理位置最近（或网络质量最好）的边缘节点，由这个节点直接返回缓存的内容。

类比：与其让全国顾客都跑到总仓库（源站）取货，不如在各个城市开分仓（边缘节点），提前把热门商品铺货过去。顾客就近取货，物流时间大幅缩短，总仓库也不会被所有订单挤爆。

关键问题：怎么把用户路由到"最近"的节点

常见的两种调度方式：

1. DNS 调度

回顾 07-DNS域名解析的完整过程——CDN 服务商接管了域名的解析权，当用户查询 cdn.example.com 时， CDN 的智能 DNS 服务器会根据用户的 IP 地址所属地理位置/运营商，返回离用户最近的边缘节点 IP，而不是固定返回同一个 IP。 不同地区的用户解析同一个域名，会得到完全不同的 IP 地址，这就是"就近调度"在 DNS 层面的实现。

2. Anycast（任播）

另一种更底层的方式是利用 IP 层的 Anycast 技术：多个地理位置不同的节点共享同一个 IP 地址， 通过 BGP 路由协议的特性，网络会自动把用户的请求路由到"网络路径上最近"的那个节点（本质是路由层面的最短路径选择，呼应网络层 Tab 学的路由选择原理）。 相比 DNS 调度需要额外一次域名解析，Anycast 在路由层面就完成了调度，更加透明。

缓存未命中怎么办：回源

边缘节点的存储空间有限，不可能缓存源站所有内容，也不是所有内容都适合缓存（比如高度个性化的动态数据）。 当用户请求的资源在边缘节点没有缓存（或缓存已过期）时，边缘节点会向源站发起请求把内容取回来，这个过程叫回源， 取回后既返回给用户，也在本地缓存一份供后续请求复用。

陷阱：CDN 主要适合缓存静态、变化不频繁的资源（图片、字体、JS/CSS、视频）。如果把频繁变化的动态接口数据也无脑塞进 CDN 缓存，容易导致用户看到过期数据；这类场景通常需要设置合理的缓存过期时间（TTL），或者用户主动发起缓存刷新（purge）来让边缘节点丢弃旧缓存。

CDN 带来的好处，不只是"快"

  • 降低延迟：就近访问，减少物理距离带来的传播时延。
  • 减轻源站压力：大部分请求在边缘节点就被缓存命中拦截了，真正打到源站的请求量大幅下降。
  • 提升可用性：即使源站短暂故障，边缘节点的缓存内容依然可以继续对外服务一段时间；同时 CDN 通常也具备一定的抗 DDoS 能力，因为攻击流量被分散到大量边缘节点上，而不是直接砸向源站。

常见误区

误区："CDN 就是给网站加速的，什么内容都能放。" 动态、个性化、强一致性要求的数据并不适合直接放 CDN；CDN 的能力边界在"内容可以被多个用户复用"，一旦内容是"千人千面"的，缓存命中率会很低，价值也随之下降。
面试高频："CDN 是怎么让用户访问最近节点的？" 抓住两个关键词回答：DNS 调度（按用户 IP 归属地返回不同节点 IP）和 Anycast（路由层面自动选路到最近节点）。

核心价值总结

  • CDN 通过在各地部署边缘节点缓存静态资源，用"就近访问"绕开物理距离带来的延迟下限。
  • DNS 调度和 Anycast 是把用户路由到最近节点的两种主流机制，分别工作在应用层和网络层。
  • CDN 不仅提速，还能显著减轻源站压力、提升整体可用性，是现代 Web 基础设施的标配一环。

与其他知识点的联系

  • 关联知识点：负载均衡的常见算法（上一篇）：CDN 可以理解为"全局地理尺度上的负载均衡"——上一篇讲的负载均衡在一个机房内部把请求分给多台机器，CDN 则是先把用户请求调度到离他最近的边缘节点集群，节点集群内部往往还要再叠加一层上一篇讲的负载均衡算法，两者是同一个"分流"问题在不同尺度的体现。
  • 关联知识点：数字证书与 CA 信任链（本 Tab 第三篇）：CDN 边缘节点要对外提供 HTTPS 服务，必须在每一个节点上都部署能通过信任链验证的证书——这也是大规模 CDN 场景下证书自动化管理（自动续期、多节点同步分发）成为独立技术议题的原因，回顾第三篇讲的信任链原理有助于理解这里的部署复杂度从哪来。
  • 关联知识点：DNS 域名解析（呼应网络层与链路层 Tab 里 ARP 的"缓存"思路）：本篇讲的 DNS 调度依赖的正是域名解析这套机制，而"缓存 + 过期重新查询"的设计思路，和网络层 Tab 里 ARP 缓存表"用空间换时间"的逻辑是同一种工程模式——不同的是 ARP 缓存的是"IP-MAC"映射，DNS/CDN 缓存的是"域名-节点 IP"映射和实际的资源内容本身。
  • 关联知识点：路由选择与 Anycast（网络层 Tab）：本篇提到的 Anycast 技术，本质上是把网络层 Tab 讲的"路由器基于路由表选择最短路径"这套机制，用在了"多个节点共享同一个 IP，自动路由到网络路径最近的那个"这个新场景上——理解了路由选择的原理，才能理解 Anycast 为什么能在路由层面就完成节点调度，而不需要像 DNS 调度那样多一次解析。
