TLS 1.3 握手优化［HPBN 补充］

《计算机网络：自顶向下方法》成书于 2016 年，TLS 1.3 是 2018 年才正式发布的标准，本篇引入《High Performance Browser Networking》的内容补齐这块空白：TLS 1.3 到底比 1.2 快在哪。

回顾问题：TLS 1.2 握手为什么慢

上一个 Tab 讲过，TLS 1.2 完整握手需要额外 1 个 RTT（ClientHello 到收到服务端 Finished 之间来回一次）， 加上 TCP 三次握手的 1 个 RTT，一次全新的 HTTPS 连接首字节到达前，至少要经过 2 个 RTT 的等待。 对首屏时间（TTFB, Time To First Byte）敏感的场景，这些 RTT 是实打实的延迟，尤其在移动网络下 RTT 动辄上百毫秒，累积起来非常明显。

TLS 1.3 的核心优化一：1-RTT 握手

TLS 1.2 慢在哪：客户端要先看服务器"选了哪个加密套件、给了什么证书"，然后才能生成 Pre-Master Secret 发回去——这是一个必须等服务器先回话的串行过程。

TLS 1.3 的做法是：客户端在 ClientHello 里就顺便"猜"服务器可能支持的密钥交换参数，直接把自己这一侧的密钥交换材料（key share）一并发过去， 不再等服务器回复之后才协商。服务端收到后，如果客户端猜的参数自己也支持，就能直接用这些材料算出会话密钥，并在返回 ServerHello 时就带上加密的证书和 Finished 消息。

TLS 1.2（2-RTT，握手+建连） TLS 1.3（1-RTT，握手）
ClientHello ──────────▶ ClientHello + Key Share ──────────▶
◀── ServerHello ◀── ServerHello + Key Share
◀── Certificate + Certificate + Finished
◀── ServerHelloDone
ClientKeyExchange ────▶
Finished ──────────────▶ Finished ──────────────▶
◀── Finished

（握手部分需要 1 个 RTT 才走完） （握手部分只需 1 个 RTT，且客户端发完
                                 第二条消息就能开始发业务数据）

本质是把"先商量用什么密钥交换算法，再交换密钥材料"这两步合并成一步乐观地并行发出——如果猜中了（服务器也支持这套参数，绝大多数情况下都能猜中，因为主流密钥交换算法就那么几种），就省下了一整个 RTT；如果没猜中，服务端会要求重新走一次（HelloRetryRequest），退化到接近 1.2 的开销，但这是极少数情况。

TLS 1.3 的核心优化二：0-RTT 恢复（Early Data）

如果客户端和服务端之前已经建立过一次 TLS 1.3 连接，服务端会给客户端发一个会话票据（session ticket）。 下次重新连接时，客户端可以直接用这个票据里派生出的密钥在第一条消息里就带上加密的业务数据一起发出去，不需要再等任何握手往返——这就是 0-RTT。

类比：1-RTT 像是你和熟人打电话，一接通就能直接说事，不用先寒暄"你是谁、我是谁"；0-RTT 更进一步，像是你直接把要说的话录成语音消息，对方一开门（哪怕人还没到）消息就已经送达了。

0-RTT 的代价：重放攻击风险

0-RTT 数据是在服务端还没来得及做任何"活性确认"的情况下就被接受和处理的——攻击者如果截获了这条 0-RTT 数据包，可以原封不动地重放给服务器多次。 如果这条数据是幂等的（比如查询商品信息），重放没什么危害；但如果是"转账""下单"这类非幂等操作，重放攻击可能导致重复扣款、重复下单等严重后果。

陷阱：0-RTT 不是免费的午餐。因为无法防御重放攻击，规范明确建议 0-RTT 数据只应该用于携带幂等请求（例如 GET 请求），业务上如果不做额外校验就贸然把非幂等操作也塞进 0-RTT，是一个真实的安全隐患。

为什么更快的握手对 TTFB 很重要

首屏时间（TTFB）直接决定用户感知的"网站快不快"，而 TTFB 的组成里，TCP 握手 + TLS 握手往往占了不小的比例，尤其是在延迟较高的移动网络下。 TLS 1.3 把握手从 2-RTT 降到 1-RTT（重连场景下甚至 0-RTT），相当于直接省掉几十到上百毫秒的等待——这对大流量网站来说，是能直接影响转化率的优化。

常见误区与面试高频考点

误区："TLS 1.3 的 0-RTT 是首次连接就能做到的。" 不对，0-RTT 依赖此前建立过连接留下的会话票据，首次连接依然要走完整的 1-RTT 握手；0-RTT 只在"恢复连接"场景下生效。
面试高频："TLS 1.3 相比 1.2 做了哪些优化？" 答案要点：① 把密钥交换参数提前和 ClientHello 一起发，握手从 2-RTT 降到 1-RTT；② 支持基于会话票据的 0-RTT 数据传输，但存在重放攻击风险，只适合幂等请求；③ 移除了一批不安全的旧加密套件（如 RSA 密钥交换，强制使用具备前向保密的 ECDHE）。

核心价值总结

  • TLS 1.3 通过让客户端在 ClientHello 阶段就"预测并携带"密钥交换材料，把握手从 2-RTT 压缩到 1-RTT。
  • 0-RTT 利用历史会话票据实现"无需握手直接发数据"，但因无法防重放，只适合幂等操作。
  • 握手轮数的减少直接转化为 TTFB 的降低，是现代高性能 Web 基础设施升级 TLS 版本的核心驱动力。
