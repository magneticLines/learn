对称加密与非对称加密

HTTPS 安全的地基。理解这两种加密方式各自的优劣，才能理解为什么 HTTPS 要把它们"混着用"，而不是二选一。

问题的起点：明文传输有什么风险

普通 HTTP 请求在网络上是明文传输的——你在公司交换机、小区路由器、运营商网关上抓个包，就能看到用户名密码。 传输层（03-TCP四次挥手 那一系列）只管"可靠地把字节从 A 搬到 B"，从不关心内容是否被第三方看到或篡改。 加密要解决的就是这个问题：即使数据被截获，攻击者也看不懂、改不了。

对称加密：一把钥匙开一把锁

对称加密指加密和解密用同一把密钥。常见算法有 AES、ChaCha20。 它的优点是速度快（只涉及位运算和查表，现代 CPU 甚至有 AES-NI 硬件指令加速），适合加密大量数据。

类比：你和朋友约定用同一把挂锁和钥匙锁快递箱。锁快递箱、开快递箱用的是同一把钥匙——简单高效，但前提是你们俩必须先在私下"面对面"把这把钥匙对上号，不能通过快递箱本身传钥匙（不然快递员半路截胡钥匙，箱子就白锁了）。

这就是对称加密的核心难题：密钥分发问题。A 和 B 要通信，必须先用某种方式安全地共享同一把密钥；但如果这条共享密钥的通道本身不安全，密钥在半路就被偷了，后面的加密全部形同虚设。 对于两个从未见过面、中间隔着整个互联网的浏览器和服务器，"线下碰头交换密钥"显然不现实。

非对称加密：公钥加密，私钥解密

非对称加密（也叫公钥加密）用一对数学上关联但不能互相推导的密钥：公钥（public key）可以随便公开给任何人， 私钥（private key）必须自己严格保密。规则是：用公钥加密的数据，只有对应的私钥能解开（RSA、ECDHE 是常见算法）。

类比：你在网上公开一把"只能锁、不能开"的挂锁（公钥），任何人都能拿这把锁把东西锁进箱子寄给你；但打开箱子的钥匙（私钥）只有你自己有。别人不需要先跟你偷偷碰面对钥匙——直接用你公开的锁就行，从根本上绕开了密钥分发问题。

非对称加密解决了密钥分发问题，但代价是速度慢得多——RSA 的运算涉及大数模幂运算，比 AES 慢两到三个数量级。 如果整个 HTTPS 会话期间所有数据都用非对称加密来加解密，性能会差到无法接受。

对比：该怎么选

维度对称加密非对称加密
密钥数量一把，双方共享一对，公钥公开 + 私钥保密
速度快，适合加密大量数据慢，只适合加密少量数据
核心难题密钥如何安全分发如何证明公钥确实属于目标（见下一篇数字证书）
常见算法AES、ChaCha20RSA、ECDHE（椭圆曲线）
典型用途加密正文数据加密/协商密钥、数字签名

HTTPS 的答案：混合加密方案

HTTPS（本质是 HTTP 跑在 TLS 之上）不是"二选一"，而是取两者之长：

  1. 用非对称加密解决"素未谋面的双方如何安全协商出一个只有彼此知道的密钥"这个冷启动难题；
  2. 协商出的这个密钥叫会话密钥（session key），之后双方改用它做对称加密来加密海量的实际业务数据。
一句话总结（面试高频）：HTTPS = 用非对称加密协商密钥，用对称加密传数据。前者解决密钥分发的安全问题，后者解决大流量下的性能问题，两者互补。下一篇《HTTPS 的 TLS 握手过程》会把这个协商过程的每一步拆开讲。

常见误区

误区一："非对称加密更安全，应该全程用非对称加密。" 安全性和加密方式的数学原理有关，和"对称/非对称"这个分类本身无关——AES-256 在合理使用下同样非常安全。选择对称加密做数据传输纯粹是性能考量。
误区二："公钥加密的数据，公钥自己能解开。" 恰恰相反，公钥只能加密不能解密，这是非对称加密安全性的根基。如果公钥自己能解密，那公开公钥就等于公开了解密能力，毫无意义。

核心价值总结

  • 对称加密快但密钥分发难，非对称加密解决分发问题但慢。
  • HTTPS 是混合方案：非对称加密协商会话密钥，对称加密加密业务数据。
  • 光有非对称加密还不够——公钥本身怎么证明"确实是目标网站的公钥"，需要数字证书体系，这是下下篇的内容。
