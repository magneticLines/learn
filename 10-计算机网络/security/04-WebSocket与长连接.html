WebSocket 与长连接

HTTP 天生是"一问一答"的请求-响应模型，服务端没法主动找客户端说话。WebSocket 解决的正是"服务端如何主动推送消息给客户端"这个问题。

问题：HTTP 模型下服务端无法主动推送

HTTP 协议（回顾 03-HTTP协议的演进）的通信模型是客户端发起请求，服务端返回响应，一来一回， 服务端永远是被动的一方。但很多场景需要反过来：聊天消息、股票行情、订单状态变化，都是服务端事件驱动的，客户端并不知道什么时候会有新消息，没法掐时间点去问。

在 WebSocket 出现之前：轮询和长轮询的权宜之计

方案做法问题
轮询（Polling） 客户端每隔固定时间（如 3 秒）发一次请求问"有新消息吗" 大部分请求都是"没有"的空跑，浪费带宽和服务器资源；消息实时性受轮询间隔限制
长轮询（Long Polling） 客户端发请求后，服务端不立刻返回，而是"挂起"这个连接，等有新消息了才返回，返回后客户端立刻发起下一次请求 实时性比轮询好，但本质仍是"伪装成长时间挂起的请求-响应"，每次数据到达都要重新建立请求，且服务端要为大量挂起连接占用资源

这两种方案都是在"HTTP 只能单向发起"这个约束下的妥协，没有从根本上解决双向通信的问题。

WebSocket：一次握手，之后全双工

WebSocket 的思路是：先借用一次 HTTP 请求完成握手，握手成功后把这条 TCP 连接"升级"成一条持久的全双工通道， 之后双方谁想发消息随时可以发，不再受"一问一答"约束。

类比：轮询像是每隔几分钟给朋友打个电话问"有事吗"，大部分时候白打；WebSocket 则像是直接建立一条专线电话，接通之后谁想说话随时可以说，不需要挂断重拨。

升级过程：Upgrade 握手

客户端 ────────▶ HTTP 请求，带特殊头：
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: (随机生成的 base64 值)

◀──────── 服务端返回 101 Switching Protocols
  Sec-WebSocket-Accept: (对 Key 做特定算法计算后的值)

======== 同一条 TCP 连接，协议从 HTTP 切换为 WebSocket ========
======== 之后双方可随时互发帧（frame），不再是请求-响应 ========

关键点在于：WebSocket 握手复用了 HTTP 的请求-响应结构（状态码 101 表示"切换协议"）， 这样设计的好处是能天然兼容现有的 HTTP 基础设施——同样的端口（80/443）、同样能被防火墙和代理正确处理，握手完成后再切换成 WebSocket 自己的二进制帧格式。

对比：三种方案的取舍

维度轮询长轮询WebSocket
连接方式短连接，反复建立挂起的长连接，收到消息后重建一次握手后持久全双工连接
实时性差，受轮询间隔限制较好好，服务端可即时推送
资源消耗大量无效请求大量挂起连接占用服务端资源连接建立后开销很低
协议基础标准 HTTP标准 HTTPHTTP 握手 + 独立的 WebSocket 协议(ws://, wss://)

常见误区

误区一："WebSocket 是 HTTP 的一种。" 不准确。WebSocket 是一个独立的协议（ws:// 或加密版 wss://），只是握手阶段借用了 HTTP 的请求格式，握手完成后传输的帧格式和 HTTP 报文完全不同。
误区二："用了 WebSocket 就不需要考虑连接维护了。" 长连接容易被中间的 NAT 网关、负载均衡器因为"长时间无数据"而判定为空闲并断开（回顾传输层的连接概念），实际使用中通常需要客户端定期发送心跳包（ping/pong）维持连接活跃。
面试高频："WebSocket 和 HTTP 长轮询的区别？" 核心答案：长轮询本质仍是重复的请求-响应，只是用挂起模拟"推送"；WebSocket 是真正的全双工协议，一次握手后连接始终打开，服务端能主动推送，且没有重复建连的开销。

核心价值总结

  • WebSocket 解决的核心问题是服务端主动推送，弥补 HTTP 请求-响应模型的天然缺陷。
  • 握手阶段复用 HTTP 的 Upgrade 机制，切换后使用独立的全双工帧协议。
  • 相比轮询/长轮询，WebSocket 大幅降低了实时通信场景下的延迟和资源开销。

与其他知识点的联系

  • 关联知识点：HTTPS 的 TLS 握手过程（本 Tab 第二篇）：加密版的 wss:// 需要先完成一次完整的 TLS 握手，再在这条已加密的 TCP 连接上做 HTTP Upgrade——也就是说一条加密的 WebSocket 连接建立起来，实际上叠加了两层握手（TCP 三次握手 + TLS 握手 + WebSocket Upgrade 握手），这也是为什么首次建立 wss:// 连接的延迟会比 ws:// 明显更高。
  • 关联知识点：负载均衡的常见算法（下一篇）：WebSocket 是长期占用的持久连接，负载均衡器如果用轮询之类的策略给每个新连接分配后端机器，同一条连接的生命周期内必须始终落在同一台服务器上（否则内存中的连接状态就丢了）——这正是下一篇要讲的"会话保持"在长连接场景下的直接体现，且比普通 HTTP 请求的会话保持要求更严格。
  • 关联知识点：CDN 与边缘节点（本 Tab 后几篇）：CDN 主要为静态、可缓存的内容做加速，而 WebSocket 传输的是动态、实时、双向的数据流，天然不适合被边缘节点缓存——理解了 WebSocket 连接必须直连到能维持状态的具体后端实例，才能理解为什么这类实时通信场景通常绕过 CDN 缓存层，直接面对负载均衡器。
  • 关联知识点：TCP 连接与 NAT 网关（网络层 Tab）：本篇提到的"长时间无数据会被中间设备判定为空闲并断开"，根源在于 NAT 网关和防火墙为了节省资源，会给每条转发中的 TCP 连接维护一个有超时时间的映射表项——回顾网络层 Tab 讲的 NAT 工作机制，就能理解为什么心跳包不是 WebSocket 自身要求，而是应对底层网络设备行为的工程妥协。
