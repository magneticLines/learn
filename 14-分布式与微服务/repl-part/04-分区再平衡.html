复制与分区

分区再平衡

集群规模一变（加机器扩容、机器故障下线），数据就得跟着挪窝——挪多挪少、挪得聪不聪明，直接决定了扩容是"丝滑加机器"还是"半夜挪数据挪到崩溃"。

什么是再平衡（Rebalancing）

再平衡指的是：当集群拓扑发生变化时（新增节点扩容、节点故障或主动下线），把数据/分区在节点之间重新分配的过程。 它是分区方案绕不开的问题——你不可能把分区策略设计成"只适用于固定数量的节点"，因为现实中集群规模一定会变。

再平衡的核心诉求只有一个：用最小的数据搬移代价，完成新的分配。 如果换一台机器就要搬走 90% 的数据， 那扩容这件事本身就会变成一场灾难——搬迁期间网络、磁盘 IO 被打满，集群对外服务能力反而下降，扩容"帮倒忙"。

反例：取模分区（hash(key) mod N）

最直觉的分区方式是：算出 key 的哈希值，对节点数 N 取模，余数就是这个 key 该去的节点编号——hash(key) mod N。 平时看起来毫无问题，但只要 N 发生变化（加一台机器，N 从 4 变成 5），几乎所有 key 的取模结果都会跟着变， 也就是说：新增一台节点，却导致几乎全部数据都要重新搬家。

示例：N=4 → N=5 时取模结果几乎全变
// 假设几个 key 的哈希值如下
key: "user_101"  hash = 17
key: "user_102"  hash = 18
key: "user_103"  hash = 19
key: "user_104"  hash = 20
key: "user_105"  hash = 21

// N = 4 时：hash mod 4
17 mod 4 = 1    18 mod 4 = 2    19 mod 4 = 3    20 mod 4 = 0    21 mod 4 = 1

// N = 5 时：hash mod 5（新增一个节点后）
17 mod 5 = 2    18 mod 5 = 3    19 mod 5 = 4    20 mod 5 = 0    21 mod 5 = 1

// 对比：5 个 key 里有 4 个的目标节点变了，只有 user_105 碰巧没变
// 节点数越大、增量越小，"碰巧不变"的比例反而更低，几乎必须全量重新分布

这就是取模分区的根本缺陷：N 一变，映射关系整体错位，而不是只有"新节点该接管的那部分"数据发生变化。 对于数据量巨大的系统，这意味着扩容一次就要触发一次几乎等同于"全量重建"的数据搬迁，代价高到无法在生产环境接受。

方案一：固定数量的分区（Fixed Number of Partitions）

思路很简单：一开始就创建远多于当前节点数的分区（比如无论未来集群多大，都固定切成 1000 个分区）， 然后把这些分区尽量均匀地分配给现有节点。加节点时，不需要重新计算每个 key 该落在哪个分区—— 分区和 key 的映射关系永远不变，变的只是"分区被分配给哪个节点"。

再平衡时只需要把若干个整块分区从旧节点搬到新节点，搬迁单位是分区而不是单条数据， 操作上简单很多（甚至可以直接搬整个数据文件），而且搬移量是可控的、和新增节点数成比例的，不会牵动全局。 这也是 Elasticsearch、Kafka 等系统采用的思路。

方案二：一致性哈希（Consistent Hashing）

一致性哈希的机制是：把节点和 key 哈希到同一个环形哈希空间（hash ring）上—— 节点的哈希值决定它在环上的位置，key 的哈希值也落在环上某处，规则是"顺时针找到的第一个节点就负责这个 key"。

这样设计的好处是：当某个节点被添加或移除时，只有环上紧挨着这个节点的那一小段区间内的 key 会受影响， 需要在相邻节点之间搬移；环上其他区间的 key 到其负责节点的映射完全不受影响。相比取模分区"牵一发动全身"， 一致性哈希把影响范围从"全部 key"收窄到了"局部相邻区间"。

虚拟节点（Virtual Nodes）：如果环上只放真实节点，节点数量少时，每个节点在环上占据的弧长可能很不均匀， 导致负载倾斜（有的节点分到的 key 特别多）。改进办法是让每个真实节点在环上映射出多个虚拟节点 （比如一台物理机对应 100 个虚拟节点，分散在环的不同位置），用数量换均匀度，从而让负载分布更平滑。

与 09-数据库「可扩展性」的呼应

如果你学过 09-数据库 模块「可扩展性」一节，会记得分库分表场景下也提到了一致性哈希—— 这不是巧合，而是同一种技术在分布式数据存储的再平衡问题上的直接应用： 无论是分布式缓存集群（类似 Redis Cluster 的哈希槽思路）、还是像 DynamoDB、ShardingSphere 这类支持动态扩缩容的分库分表中间件， 它们扩容/缩容时都要面对"怎么尽量少搬数据"这个同一个问题，答案也高度一致——固定分区数或一致性哈希，而不是简单取模。

三种策略对比

策略扩缩容时的数据搬移代价实现复杂度典型使用场景
取模分区（mod N） 极高，N 一变几乎全部 key 都要重新分配 最简单，一行 hash % N 节点数固定、几乎不扩缩容的小系统；教学示例
固定分区数 较低，只搬迁若干整块分区，与新增节点数成比例 中等，需要提前规划好分区总数并维护"分区→节点"映射 Elasticsearch、Kafka 等消息/搜索系统
一致性哈希 低，只影响环上相邻区间的 key 较高，需要维护哈希环，并引入虚拟节点解决负载不均 分布式缓存（Redis Cluster 类似思路）、DynamoDB、分库分表中间件
常见误区：很多人直觉认为"取模分区加一台节点，只需要搬 1/N 的数据"（比如 4 台变 5 台，只搬 1/5）。 这个直觉是错的——从上面的数值例子能看到，hash mod 4 和 hash mod 5 的结果几乎完全不同， 实际需要搬移的比例接近 100%，而不是 1/N。只有当新节点数恰好是旧节点数的整数倍等特殊情况下，重合比例才会碰巧高一些， 但这在真实场景中无法依赖，本质上取模分区就是"设计上不支持渐进式扩容"。
面试高频："一致性哈希解决了什么问题？为什么比取模分区好？" 标准答案：取模分区在节点数变化时会导致几乎全部 key 的归属重新计算，数据搬移代价极高； 一致性哈希把节点和 key 映射到同一个哈希环上，节点增减只影响环上相邻的局部区间，把数据搬移量从"全量"降到"局部"， 这正是"一致性"（consistent）这个名字的含义——大部分映射关系在拓扑变化前后保持一致。 如果追问负载均衡问题，要补充一点：虚拟节点是让哈希环在节点数较少时也能保持负载均匀的关键改进。

核心价值总结

  • 再平衡的目标是最小化数据搬移：集群拓扑一变化就要评估"这次变化到底需要挪多少数据"。
  • 取模分区是反面教材：N 一变几乎全量重新分布，看似简单实则完全不具备可扩展性。
  • 固定分区数和一致性哈希是两种主流解法：前者把搬移单位从"单条数据"变成"整块分区"，后者把影响范围从"全部 key"收窄到"局部区间"，两者都服务同一个目标——加/减节点时只动局部，不动全局。
