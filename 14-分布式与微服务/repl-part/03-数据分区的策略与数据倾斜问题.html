复制与分区

数据分区的策略与数据倾斜问题

复制解决的是"同一份数据放多处"，分区解决的是"数据太大放不进一台机器"——两者是正交的两个维度，合起来才是完整的分布式存储方案。

为什么需要分区（Partitioning / Sharding）

单台机器的容量和吞吐都是有上限的：磁盘装不下全部数据，CPU/网络也扛不住全部请求。 分区（也叫分片，Sharding）的思路很直接——把数据集切成若干个子集（分区），分别放到不同的节点上， 每个节点只需要负责自己那一份数据的存储和查询。

这样一来，容量和吞吐都能随节点数量线性扩展：加机器就能装更多数据、扛更多请求。 但分区绝不是"随便切一刀"就完事——怎么切直接决定了负载能不能真正被均匀分摊到各个节点上。 如果切分策略不当，即使加再多机器，请求也可能全部堆积到某一个节点，这就是本篇要讲的核心问题：数据倾斜。

哈希分区（Hash Partitioning）

对每条记录的 key 计算哈希值 hash(key)，再根据哈希值落在哪个区间，决定它归属哪个分区。 例如有 4 个分区，可以简单地用 hash(key) % 4 决定去向。

优点：只要哈希函数本身分布均匀，任意一批 key 经过哈希后几乎必然被打散到各个分区， 天然适合把负载均匀摊到所有节点上，不容易出现某个分区数据量特别大的情况。
代价：哈希会把原本连续、有序的 key 打得七零八落，牺牲了范围查询能力。 比如想查"时间戳在 T1 到 T2 之间的所有事件"，这些 key 哈希后分散在几乎所有分区里， 数据库只能向全部分区发起查询再合并结果（scatter-gather），完全无法像范围分区那样"定点打击"。

范围分区（Range Partitioning）

每个分区负责一段连续的 key 区间，例如按 key 的首字母分区（A-F 一个区，G-M 一个区……）， 或者按时间戳分区（每天/每小时一个分区）。分区边界通常根据实际数据分布动态调整，而不是死板地均分。

优点：范围查询效率高——查一段连续区间的数据，只需要命中覆盖这段区间的少数几个分区， 不用像哈希分区那样全网扫描。
代价：容易产生热点。最经典的例子：如果按时间戳给一张事件表/时序表分区（比如"今天"一个分区）， 那么所有新写入的数据都会落到同一个"最新"分区里——今天的分区被写爆， 而其他节点上存放历史数据的分区几乎无人问津、彻底闲置。分区数量再多，写入压力也全压在那一台机器上。

数据倾斜（Data Skew）：分区设计要规避的核心问题

数据倾斜指的是某些分区承载的数据量或访问负载远高于其他分区（后者也叫"热点分区"，Hot Partition）。 一旦出现倾斜，分区最初的目的——把负载摊开——在那个热点节点上就彻底失效了： 不管你总共部署了多少台机器，系统的瓶颈永远卡在那台最忙的节点上，其他节点的富余算力毫无意义。

常见缓解思路：

  • 给热点 key 加随机前缀/后缀（加盐，Salting）：比如把热点 key user:123 拆成 user:123:0 ~ user:123:9，人为分散到多个分区。 代价是读取时要反过来查询所有加盐后的变体再合并结果，读逻辑变复杂。
  • 应用层识别热点 key：提前监控访问模式，发现某些 key 明显比其他 key 热得多，针对性地做特殊处理（缓存、单独拆分等），而不是对所有 key 一视同仁。
与 09-数据库 呼应：在"可扩展性"一节讲分库分表时提到的路由策略，本质上就是这里的哈希分区和范围分区—— ShardingSphere 等分库分表中间件的路由算法，无非是"按 id % 分库数 取模路由"（哈希分区的工程实现） 和"按主键/时间范围路由到对应分库分表"（范围分区的工程实现）。两边的取舍完全一致： 取模路由分布均匀但不支持范围查询，范围路由对范围查询友好但容易在最新数据段产生热点。理解了这里的原理，再看分库分表中间件的配置项就会一目了然。

哈希分区 vs 范围分区

维度哈希分区范围分区
负载分布 均匀，key 打散到各分区 不均匀，依赖 key 的实际分布
范围查询支持 差，需 scatter-gather 全分区查询 好，只命中相关分区
热点风险 低（哈希函数均匀的前提下） 高，尤其是按时间戳分区时"最新分区"必热
典型场景 KV 存储、按用户 ID 分片 时序数据库、日志系统、需要按区间扫描的场景

常见误区

误区：新手常常直接拿自增 ID 或时间戳这类单调递增的 key 做范围分区， 以为"分区数量够多就均匀了"。实际上单调递增的 key 无论怎么切区间，新数据永远追加在最后一个区间里， 结果就是最新的那个分区持续被写爆，其余分区早早地"退休"。这正是范围分区热点问题的典型翻车现场。
面试高频："什么是数据倾斜，如何避免？" 回答要点：数据倾斜是指部分分区数据量或访问量远超其他分区，导致该分区成为整个集群的瓶颈； 避免思路包括——优先用哈希分区打散负载，若必须范围分区则对热点 key 做加盐（随机前缀/后缀）拆分， 并在应用层监控识别热点 key 做针对性处理。

核心价值总结

  • 分区解决容量和吞吐的扩展问题，与复制（解决可用性和读扩展）是两个正交的维度，实际系统通常两者结合使用。
  • 哈希分区换均匀、范围分区换范围查询能力，没有哪种分区策略是"更好"的，只有"更适合当前访问模式"的。
  • 数据倾斜是分区设计的头号敌人：好的分区策略必须结合实际的 key 分布和访问模式来设计，盲目分区反而可能制造出比不分区更糟的单点瓶颈。

与其他知识点的联系

  • 关联知识点：多主复制与无主复制：复制解决的是"同一份数据放多处以提升可用性和读扩展"，分区解决的是"数据太大放不进一台机器"，两者是正交的维度——真实系统里，每个分区通常还会再做多副本复制，所以理解分区之前先理解复制，再看分区，才能拼出完整的分布式存储图景。
  • 关联知识点：描述负载与性能的常用指标：数据倾斜本质上是"负载参数在多个分区间分布不均匀"的具体表现——某个分区承受的访问量远超其他分区，这和"平均响应时间掩盖尾部延迟"是同一类问题的不同变体：平均分布均匀不代表没有热点，必须看最忙的那个分区（类比 p99）才能发现真正的瓶颈。
  • 关联知识点：分区再平衡（下一篇）：本篇讲的是"怎么切"（哈希还是范围），下一篇讲的是"切完之后，集群拓扑变化时怎么重新分配"——如果分区策略本身选得不好（比如按单调递增 key 做范围分区），再平衡时热点问题只会被放大，两篇是分区设计的静态视角和动态视角。
  • 关联知识点：主从复制中的复制延迟问题：分区之后，每个分区的写入依然需要通过某种复制机制同步到副本，复制延迟问题依旧存在，而且分区数越多，需要监控和处理复制延迟的"点"也越多——分区把单机问题复制成了多份分布式问题，运维复杂度是叠加的。
