微服务架构实践

RPC 通信原理

服务拆开之后，进程内的方法调用变成了跨进程、跨网络的调用——RPC 的核心使命，就是让这种调用尽量还原成"像调本地方法一样简单"的体验。

RPC 解决的问题：屏蔽远程调用的复杂性

RPC（Remote Procedure Call，远程过程调用）的目标很直白：让调用远程服务的代码，写起来和调用本地方法一样。 如果没有 RPC 框架，你要自己处理一大堆繁琐细节：把方法参数序列化成字节流、通过 Socket 发送到目标机器、目标机器反序列化还原参数、执行对应方法、把返回值再序列化传回来、调用方反序列化拿到结果——还要处理超时、重试、连接管理。

类比：RPC 就像点外卖软件——你只需要在 App 里点"下单"这个按钮（调用方法），背后其实经过了"生成订单信息 → 传给商家系统 → 商家接单 → 骑手取餐 → 送达"一整套复杂流程，但这些细节完全被 App 封装屏蔽了，你感受到的只是"点了个按钮，东西就送到了"。RPC 框架做的就是这件事：把序列化、网络传输、反序列化这些"配送细节"都藏在框架内部，业务代码只管"调用接口、拿到结果"。

一次典型的 RPC 调用背后发生了什么：调用方通过动态代理生成一个和真实接口签名一致的"桩"（Stub），业务代码调用这个桩上的方法时，代理会把方法名、参数打包序列化成二进制数据，通过网络协议（TCP/HTTP2 等）发送给服务提供方；服务端收到数据后反序列化还原出方法名和参数，通过反射找到真正的实现类执行，再把结果序列化传回来，调用方反序列化后返回给业务代码——整个过程业务代码完全无感知，这就是"透明化远程调用"。

gRPC vs Dubbo：两种主流 RPC 框架

实际工程中最常见的两类 RPC 方案，分别代表了"跨语言通用"和"Java 生态深度集成"两种取舍：

gRPCDubbo
出身 Google 开源，基于 HTTP/2 + Protobuf 阿里巴巴开源，Java 生态的老牌 RPC 框架
序列化协议 Protobuf（二进制、体积小、跨语言强类型） Hessian2 / 支持多种序列化方式
跨语言能力 强，天生支持 Java/Go/Python/C++ 等多语言互通 主要面向 Java 生态，跨语言支持相对弱
传输协议 HTTP/2，原生支持双向流式调用（Streaming） 自定义私有协议（Dubbo 协议），也可扩展支持 REST
服务治理能力 相对基础，服务发现/负载均衡等需要额外集成（如配合 Envoy/Istio） 内置丰富，服务注册发现、负载均衡、路由规则、限流降级开箱即用
适用场景 多语言混合技术栈、需要流式调用（如实时推送、大文件传输） 纯 Java 技术栈、需要开箱即用的完整服务治理能力

值得一提的是 gRPC 的流式调用能力：普通 RPC 是"一次请求、一次响应"，而 gRPC 基于 HTTP/2 的多路复用支持客户端流、服务端流、双向流，适合需要持续推送数据的场景（比如实时行情推送），这是 Dubbo 传统模式不擅长的。

RPC vs REST：内部调用与对外 API 的取舍

另一个常见的困惑是：已经有 HTTP + JSON 的 REST 风格了，为什么还要用 RPC？两者并不是互相替代的关系，而是面向不同场景的工具：

REST（HTTP + JSON）RPC（如 gRPC / Dubbo）
通用性 极强，浏览器、移动端、第三方系统都能直接调用，人类可读 较弱，通常需要专门的客户端 SDK 或代理生成的桩代码
性能 JSON 文本序列化体积大、解析慢，HTTP/1.1 无多路复用 二进制序列化体积小，基于 HTTP/2 等支持多路复用，性能更优
调试便利性 用浏览器/curl/Postman 就能直接测试，排查问题方便 需要专门工具（如 grpcurl）或了解 IDL 定义才能调试
适用场景 对外开放的公共 API、前后端交互 内部服务之间的高频、高性能调用

实践中的常见组合是：系统对外暴露 REST API 给前端和第三方使用，内部微服务之间用 RPC 通信，兼顾了外部易用性和内部性能。这也是为什么 API 网关（后面会讲）常常需要做协议转换——外部进来的是 REST 请求，网关转成 RPC 调用内部服务。

面试高频："RPC 和 HTTP 调用有什么区别？"参考答案：RPC 是一种编程模型（让远程调用像本地调用一样），HTTP 只是它可以选用的传输协议之一（比如 gRPC 就是基于 HTTP/2）；RPC 框架通常还内置了序列化协议、服务发现、负载均衡等能力，而普通 REST 调用需要自己在业务代码里处理这些细节，或者依赖网关/Service Mesh 来补齐。

核心价值总结

  • RPC 的本质：用动态代理 + 序列化协议 + 网络传输，把远程调用伪装成本地方法调用，屏蔽底层复杂性。
  • gRPC vs Dubbo：前者靠 Protobuf + HTTP/2 换来跨语言和流式调用能力，后者靠深度绑定 Java 生态换来开箱即用的服务治理能力。
  • RPC vs REST 不是二选一：对外用 REST 保证通用性和易调试性，内部服务间用 RPC 换取性能，两者常常在同一个系统里协同工作。

与其他知识点的联系

  • 微服务拆分的原则与边界划分：上一篇说拆分之后进程内调用变成跨进程调用，本篇正是这句话的技术落地——RPC 框架就是专门用来"伪装"这次跨进程调用的工具，拆分粒度越细、服务数量越多，RPC 调用在系统里出现的频次就越高，这也是为什么"拆得太细"的代价里会包含调用链变长和序列化开销。
  • 服务发现与注册中心：本篇讲 RPC 调用需要知道"目标机器的地址"，但没深入说这个地址从哪来——下一篇服务发现要解决的正是这个问题：RPC 框架（如 Dubbo）在发起调用前，会先向注册中心查询目标服务当前可用的实例列表，再从中选一个发起真正的网络调用，服务发现是 RPC 能够"找到人"的前提。
  • API 网关的职责：本篇提到"系统对外暴露 REST API，内部用 RPC 通信"，这个内外协议差异正是 API 网关存在的原因之一——网关作为外部 REST 请求和内部 RPC 调用之间的转换层，需要把进来的 HTTP/JSON 请求转换成内部服务能理解的 RPC 协议，这是网关"协议转换"职责的具体体现。
  • 断路器模式 Circuit Breaker：本篇提到 RPC 框架要"处理超时、重试、连接管理"，而超时和重试如果没有节制，正是雪崩效应的导火索——断路器通常就包裹在 RPC 调用的这一层，统计的是每一次 RPC 调用的成功率，一旦某个下游的 RPC 调用持续超时/失败，断路器会介入让调用方快速失败，而不是傻等 RPC 超时。
  • 服务网格 Service Mesh 简介：本篇里 gRPC/Dubbo 这类 RPC 框架，服务治理能力（负载均衡、重试、超时）是通过客户端 SDK 内嵌在业务进程里实现的；服务网格则是把这套本该由 RPC 框架处理的治理逻辑整体挪到 Sidecar 里，业务代码发起的甚至可以是最朴素的调用，路由、负载均衡完全由 Sidecar 代劳——可以理解成服务网格是把 RPC 框架里"治理"这部分职责剥离出来，单独下沉成一层基础设施。
