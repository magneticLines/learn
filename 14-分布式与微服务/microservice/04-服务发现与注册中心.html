微服务架构实践

服务发现与注册中心

微服务实例会随着扩缩容、发布、故障重启不断变化 IP 和端口——服务发现机制解决的就是"调用方怎么知道该往哪个具体地址发请求"这个基础但关键的问题。

问题的根源：实例地址是动态的

在单体应用里，模块之间是进程内方法调用，压根不存在"地址"的概念。但拆成微服务之后，订单服务要调用库存服务，就必须知道库存服务实例的 IP 和端口。 如果把地址写死在配置文件里，会立刻遇到麻烦：库存服务因为扩容多开了 3 个实例怎么办？某个实例宕机重启换了 IP 怎么办？做灰度发布只想让部分流量打到新版本实例怎么办？ 手工维护地址列表在实例数量稍微多一点之后就完全不可行。

类比：服务发现就像打车软件的接单机制——你不需要提前知道"哪辆车、什么车牌"会来接你（提前写死地址），而是打车软件（注册中心）实时掌握着"当前哪些司机在线、位置在哪"，你发起叫车请求时，软件从这份实时列表里帮你匹配一个可用的司机。司机上线时会向平台"注册"自己的位置和状态，下线或熄火后这条记录也会被移除。

服务发现的基本流程

服务发现机制通常由三个角色构成：服务提供者、服务消费者、注册中心。整个协作过程是：

  1. 注册：服务提供者（比如库存服务的某个实例）启动时，主动向注册中心上报自己的地址信息（IP、端口、健康状态等），并通过心跳机制持续证明自己"还活着"。
  2. 订阅：服务消费者（比如订单服务）需要调用库存服务时，从注册中心查询/订阅当前所有健康的库存服务实例列表，而不是使用写死的地址。
  3. 剔除：如果某个实例的心跳超时（宕机、网络分区、被下线），注册中心会将其从可用列表中剔除，避免消费者继续把请求打到一个已经不存在的地址上。

在 Spring Cloud 生态里，这一套机制有现成的组件支持：Nacos（阿里开源，集服务注册发现和配置中心于一体）、Eureka（Netflix 开源，Spring Cloud Netflix 的经典组件，目前已进入维护模式）、Consul（HashiCorp 出品，支持多数据中心和更丰富的健康检查方式）都是常见选型，服务提供者和消费者只需要引入对应的 Starter 依赖，几乎不需要手写注册发现逻辑。

客户端负载均衡 vs 服务端负载均衡

拿到实例列表之后，还有一个问题：该选哪一个实例发请求？这就涉及负载均衡策略的选择，而"由谁来做负载均衡决策"衍生出两种架构：

客户端负载均衡服务端负载均衡
决策方 服务消费者自己（如集成 Ribbon / Spring Cloud LoadBalancer） 独立的负载均衡器/网关（如 Nginx、LVS，或云厂商的 SLB）
调用路径 消费者从注册中心拉取实例列表 → 本地按策略（轮询/随机/权重）选一个 → 直接发请求 消费者请求先打到统一的负载均衡器 → 负载均衡器转发到某个后端实例
优点 少一跳网络转发，延迟更低；负载均衡逻辑可以感知服务调用的上下文（如同机房优先） 对消费者透明，消费者不需要感知服务发现细节；语言无关，任何客户端都能用
缺点 负载均衡逻辑和具体语言/框架绑定，多语言团队要重复实现；消费者需要感知注册中心 负载均衡器本身可能成为单点或性能瓶颈；多了一跳网络转发，增加延迟

Spring Cloud 生态早期用 Ribbon 实现客户端负载均衡（消费者内嵌负载均衡逻辑），现在逐渐被 Spring Cloud LoadBalancer 取代；服务端负载均衡的思路则在 API 网关和 Service Mesh（本 Tab 最后一篇会讲）中被广泛使用——本质上是把"选实例"这个决策从业务代码里挪到了独立的基础设施层。

常见误区：认为"服务发现就是注册中心"。注册中心只是存储和维护实例列表的组件，服务发现是一整套包含注册、心跳保活、健康检查、订阅通知、负载均衡在内的机制。面试时如果只回答"用 Nacos 存地址"是不完整的，还要能说清楚心跳超时怎么处理、消费者怎么感知列表变化（推送 vs 轮询）。
面试高频："Eureka 和 Nacos 的区别？""服务发现的 CAP 取舍？"这是高频考点。Eureka 设计上倾向 AP（可用性优先，网络分区时宁可返回稍微过期的实例列表也不停服）；Nacos 支持 AP 和 CP 两种模式可切换，功能上还多了配置中心能力（呼应下一篇）。理解注册中心在 CP/AP 之间的取舍，本质上还是在呼应分布式系统的 CAP 定理。

与其他知识点的联系

  • RPC 通信原理：上一篇 RPC 调用的前提是"知道对方的地址"，本篇的注册中心正是给这个地址来源打底——Dubbo 这类 RPC 框架发起调用前，会先从注册中心拿到目标服务的实例列表，再由客户端负载均衡选一个发起真正的网络请求，服务发现是 RPC 调用链路里"寻址"这一环。
  • API 网关的职责：本篇讲客户端负载均衡是"消费者自己拉列表、自己选实例"，而网关本质上承担的是服务端负载均衡的角色——外部请求先统一打到网关，网关内部再去查询注册中心并转发到具体实例，这也是为什么网关往往要集成注册中心客户端（比如 Spring Cloud Gateway 配合 Nacos）才能完成路由转发这一职责。
  • 配置中心的设计：本篇提到 Nacos 是"集服务注册发现和配置中心于一体"的组件，这不是偶然——注册中心维护的实例列表和配置中心维护的配置项，本质上都是"需要被多个服务实时感知、动态变化的元数据"，两者都依赖类似的心跳/长轮询/推送机制来保证变更及时同步给订阅方，所以常常合并在同一个产品里实现。
  • 断路器模式 Circuit Breaker：本篇的"剔除"机制（心跳超时后从可用列表移除实例）和断路器的"打开"状态看起来像，但作用层次不同——服务发现剔除的是"确定已经宕机/下线"的实例，是注册中心主动的、最终一致的清理；断路器则是调用方对"还在列表里但响应变慢/报错"的实例做的更快速的止损，两者一个在服务发现层，一个在调用层，共同缩短故障影响的时间窗口。
  • 服务网格 Service Mesh 简介：本篇的客户端负载均衡需要消费者引入 Ribbon/LoadBalancer 这类 SDK，是典型的"治理逻辑内嵌进程"模式；服务网格把服务发现和负载均衡整体挪到 Sidecar 里完成，业务代码发起最朴素的调用即可，这正是本篇结尾提到的"把选实例的决策从业务代码挪到独立基础设施层"思路的进一步延伸。

核心价值总结

  • 服务发现解决的核心问题：实例地址动态变化时，让消费者始终能找到当前可用的服务实例。
  • 注册中心的职责：接收服务注册、维护心跳、剔除失效实例、向消费者提供实时的可用实例列表。
  • 客户端 vs 服务端负载均衡：前者性能更优但和语言/框架绑定，后者更通用但多一跳网络转发，两种架构在网关和 Service Mesh 时代常常结合使用。
