断路器模式 Circuit Breaker

[微服务设计补充] 一个服务的故障如果处理不当，会像多米诺骨牌一样沿着调用链传导，最终拖垮整个系统——断路器就是切断这条传导链的关键机制。

问题的根源：雪崩效应

设想这样一条调用链：网关 → 订单服务 → 库存服务。如果库存服务因为某种原因（数据库慢查询、依赖的第三方接口超时）变得响应缓慢，会发生什么？ 订单服务的每个请求线程在调用库存服务时都会阻塞等待，如果订单服务的线程池是有限的（几乎总是有限的），越来越多的线程被库存服务的慢响应占用，很快线程池就会被耗尽， 导致订单服务本身也变得无法响应——即便订单服务的其他逻辑完全健康。而订单服务的不可用又会拖累调用它的网关和上游服务，故障像雪崩一样一层层向上传导，最终整个系统瘫痪。这就是分布式系统里经典的雪崩效应。

类比：断路器这个名字直接借用了电路里的保险丝——家里的电路一旦某个电器短路导致电流异常，保险丝会自动跳闸切断电路，防止过载电流烧毁整个电路系统。它牺牲了"这一路暂时用不了电"的局部可用性，换来了"整个房子不会因为一个电器故障而起火"的整体安全。软件系统里的断路器也是同样的思路：牺牲对某个不健康下游的调用，换取整个调用链不被拖垮。

断路器的三种状态

断路器的核心是一个状态机，通过持续统计调用的成功率/失败率，在三种状态之间切换：

状态行为触发条件
关闭 Closed 正常状态，请求正常放行到下游服务，同时持续统计调用结果（成功/失败） 初始状态；半开状态下探测请求成功后也会回到这个状态
打开 Open 直接快速失败（Fail Fast），不再真正请求下游服务，而是立即返回错误或走降级逻辑，避免线程继续被下游拖住 Closed 状态下，一段时间窗口内的失败率超过预设阈值
半开 Half-Open Open 状态维持一段冷却时间后，放行少量试探性请求去真实调用下游服务，用结果判断下游是否已经恢复 Open 状态的冷却计时器到期

半开状态是设计中很巧妙的一环：如果一直卡在"打开"状态，即便下游服务早就恢复了，断路器也感知不到，会一直无谓地拒绝请求；但如果直接切回"关闭"状态全量放行，万一下游还没真正恢复，又会立刻把刚缓过来的服务重新打垮。 半开状态用"小流量试探"的方式在两者之间找到了平衡——探测成功则认为下游已恢复，切回关闭状态恢复正常流量；探测失败则说明下游还没好，退回打开状态继续冷却。

面试高频："断路器的三种状态分别是什么，触发条件是什么？"这是分布式系统面试的高频题，回答时要能讲清楚状态之间的转换条件（失败率阈值触发 Closed→Open，冷却时间触发 Open→Half-Open，探测结果决定 Half-Open 的去向），而不只是背出三个名词。

常见实现：Resilience4j / Hystrix

Hystrix 是 Netflix 早期开源的容错框架，曾是 Spring Cloud Netflix 生态的标配，但目前已经停止维护； Resilience4j 是目前更主流的替代方案，它是轻量级、模块化设计（断路器、限流器、隔离仓、重试等能力都是独立模块，按需引入），并且天然对 Java 8 的函数式编程风格更友好，Spring Cloud Circuit Breaker 也是以 Resilience4j 作为默认实现之一。 实际使用时通常是给某个远程调用方法包裹一层断路器逻辑，配置好失败率阈值、统计窗口大小、冷却时间等参数，剩下的状态切换完全由框架自动处理，业务代码只需要关心"调用失败/被熔断时的降级逻辑写什么"。

常见误区：把断路器和限流混为一谈。限流是"主动控制请求的准入速率"（防止流量洪峰打垮系统），断路器是"被动响应下游故障"（下游已经出问题了，快速失败止损）。两者经常配合使用，但解决的是不同阶段的问题——限流防患于未然，断路器亡羊补牢，实际系统里通常两者都需要，加上下一篇要讲的舱壁模式，共同构成了服务容错的组合拳。

核心价值总结

  • 断路器解决的问题：防止对不健康下游的持续调用耗尽调用方资源，进而引发雪崩效应。
  • 三种状态的设计逻辑：关闭状态正常放行并统计失败率，打开状态快速失败保护自身，半开状态小流量试探判断下游是否恢复。
  • 生态选型：Hystrix 已停止维护，Resilience4j 是当前 Spring Cloud 生态的主流选择，通常和限流、舱壁隔离配合构成完整的容错体系。
