微服务架构实践

API 网关的职责

当微服务数量变多，外部调用方不可能、也不应该直接面对几十个服务的地址——API 网关作为统一入口，把一堆和业务无关的"横切关注点"从每个服务里收拢到一处集中处理。

问题的根源：横切关注点不该散落在每个服务里

假设没有网关，外部客户端（App、前端、第三方系统）要直接调用订单服务、库存服务、用户服务……每个服务都要各自实现一遍：请求怎么鉴权、超过阈值怎么限流、下游异常怎么熔断、访问日志怎么记录。 这些能力和"订单""库存"本身的业务逻辑毫无关系，却要在每个服务里重复一遍——这正是横切关注点（Cross-Cutting Concerns）的典型特征：不属于任何一个业务模块，却贯穿所有模块。

类比：API 网关就像写字楼的前台和门禁系统——访客不需要、也不被允许直接跑到每一层楼自己核实身份，而是统一在一楼前台登记、刷卡验证，前台会根据访客要去的楼层把人引导过去。如果每层楼都自己搞一套门禁核验逻辑，不仅重复建设，标准还可能不一致（有的楼层验证严格，有的宽松），出了安全问题也很难统一排查。

网关承担的核心职责

职责具体说明
路由转发 根据请求路径、Header 等规则，把外部请求转发到对应的后端微服务，客户端不需要知道内部服务的具体地址和拆分方式。
统一鉴权 在请求真正到达业务服务之前，统一校验 Token/Session 是否合法，未登录或权限不足的请求直接在网关层拦截，业务服务不用重复写鉴权逻辑。
限流熔断 对请求量做统一的限流控制（比如某个接口每秒最多处理多少请求），防止突发流量打垮后端服务；也可以在网关层配合断路器策略，对已知异常的下游服务快速失败。
协议转换 外部通常是 REST/HTTP 请求，网关可以把它转换成内部服务使用的 RPC 协议（呼应上面 RPC 通信原理那篇提到的内外协议差异），实现"外部统一、内部灵活"。
日志监控 统一记录所有进出的请求日志、响应时间、状态码，作为链路追踪和问题排查的第一手数据来源，不需要每个服务各自维护一套访问日志格式。

为什么要在网关层统一收口，而不是让每个服务自己处理

核心原因是避免重复造轮子，同时保证标准统一。如果鉴权逻辑分散在几十个服务里各自实现：

  • 安全策略升级（比如 Token 过期时间调整、新增双因子校验）要改几十个服务，而不是改一个网关配置；
  • 不同服务团队实现的鉴权逻辑标准不一致，容易出现有的服务校验严格、有的存在漏洞的情况；
  • 限流阈值、熔断策略这类需要全局视角才能做好的能力（比如按用户维度做全局限流），单个服务局部视角很难做到位，而网关天然处于所有流量的汇聚点，适合做全局决策。

这本质上是一种关注点分离：让业务服务只关心业务逻辑本身，把和业务无关的通用能力下沉到网关这一层基础设施，这个思路和后面会讲的服务网格（Service Mesh）是一脉相承的，只是网关解决的是"南北向流量"（外部进入系统），服务网格解决的是"东西向流量"（服务之间）。

Spring Cloud Gateway

在 Spring Cloud 生态里，Spring Cloud Gateway 是官方主推的网关组件（早期还有 Zuul，目前 Gateway 是更现代化的选择），基于 Spring WebFlux 的响应式编程模型构建，具备高并发处理能力。 它通过声明式的路由规则（Route + Predicate + Filter）配置请求怎么匹配、怎么处理、转发到哪里，并且可以方便地和 Spring Cloud 生态里的注册中心（如 Nacos）、限流组件（如 Sentinel）集成，做到"服务发现 + 路由 + 限流"一体化。

常见误区：把业务逻辑也塞进网关（比如在网关层写订单状态判断），导致网关变成一个新的"单体瓶颈"。网关应该只处理通用的横切关注点，业务逻辑始终应该留在对应的微服务内部，否则网关会重新变成一个难以维护、耦合了各种业务分支判断的庞然大物，也失去了微服务拆分的意义。

与其他知识点的联系

  • 服务发现与注册中心：网关的"路由转发"职责离不开服务发现——网关拿到外部请求后，要转发到具体某个内部服务实例，这个实例地址同样来自注册中心，本质上网关内部也扮演了一次服务消费者的角色，只是它是所有外部流量共用的那一个消费者。
  • RPC 通信原理：本篇提到的"协议转换"职责，正是呼应 RPC 篇里"对外用 REST、内部用 RPC"这个组合——网关是这个组合真正落地的那道边界，外部进来的 HTTP/JSON 请求要在网关这一层被转换成内部服务能理解的 RPC 调用，网关因此天然要理解两套协议。
  • 断路器模式 Circuit Breaker：本篇表格里"限流熔断"这条职责，说的就是断路器在网关层的应用——因为网关是所有外部流量的汇聚点，天然具备全局视角，能够比单个业务服务更早发现某个下游持续异常，从而在网关这一层就快速失败，避免故障请求继续往后传导到已经不健康的服务。
  • 配置中心的设计：网关的路由规则、限流阈值本身也是一种需要动态调整的配置——比如大促期间临时调整某个接口的限流值，如果每次改动都要重启网关，响应速度和配置中心要解决的问题是同一个道理，实践中网关的路由和限流配置也常常托管在配置中心里实现热更新。
  • 服务网格 Service Mesh 简介：本篇结尾提到网关解决的是"南北向流量"（外部进入系统），服务网格解决的是"东西向流量"（服务之间）——两者的职责划分思路完全一致：都是把和业务无关的横切关注点（鉴权、限流、可观测性）从业务代码里剥离，收拢到独立的基础设施层，只是网关站在集群入口，服务网格的 Sidecar 站在每个服务实例旁边。

核心价值总结

  • API 网关是外部流量进入微服务集群的统一入口：客户端只需要认识一个地址，不需要感知内部服务的拆分细节。
  • 核心职责是收口横切关注点：路由转发、统一鉴权、限流熔断、协议转换、日志监控，这些能力放在网关一处维护，避免每个服务重复实现、标准不一。
  • 边界要清晰：网关只处理通用能力，业务逻辑仍然属于各个微服务，不能让网关重新变成一个新的单体。
