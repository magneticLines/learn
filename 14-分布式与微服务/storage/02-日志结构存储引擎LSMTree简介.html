日志结构存储引擎 LSM-Tree 简介

09-数据库模块已经讲透了 InnoDB 的 B+ 树是怎么组织磁盘数据的，这里不再重复 B+ 树细节，而是介绍另一条完全不同的技术路线——LSM-Tree，理解它为什么在写多读少的场景下能碾压 B+ 树。

一句话讲清楚

LSM-Tree（Log-Structured Merge-Tree，日志结构合并树）的核心思路是：写入操作永远只追加（append）到内存里一个有序的数据结构（称为 Memtable），从不在磁盘上原地修改数据；当 Memtable 攒到一定大小，就整体顺序刷写到磁盘，形成一个不可变的文件（称为 SSTable，Sorted String Table）；后台再定期把多个 SSTable 合并压缩（Compaction），清理掉过期和重复的数据。全程没有"找到某一行原地覆盖"这个动作，取而代之的是"不断追加新版本 + 后台异步整理"。

类比：B+ 树的更新像是"直接翻开账本某一页，用橡皮擦掉旧数字，写上新数字"——精确但每次都要先找到那一页（可能要翻好几层索引），而且擦写的位置在磁盘上是随机分散的。LSM-Tree 的更新像是"不管三七二十一，把新的一条流水记录直接写在账本最后一页"——写入永远是在末尾追加，飞快；但等你想查某个人的余额，就可能要往前翻好几本流水账本，把所有相关记录都翻出来核对，才能拼出最终结果。定期会有人把攒起来的旧账本重新誊抄、去重、合并成新的一本，这就是 compaction。

写入路径：为什么快

LSM-Tree 写入快的根本原因，是它把随机写变成了顺序写：

  1. 写请求先写入内存中的 Memtable（通常用跳表 SkipList 或红黑树实现，保持 key 有序），同时为了防止进程崩溃丢数据，会先顺序追加写一份预写日志（WAL）到磁盘。
  2. 当 Memtable 达到设定大小，把它整体、顺序地刷写到磁盘，生成一个新的、不可变的 SSTable 文件（文件内部 key 已经排好序）。
  3. 磁盘上会积累多个 SSTable 文件（从新到旧），后台线程定期把它们合并（Compaction），清理掉被覆盖或删除的旧版本数据，减少文件数量。

整个写入路径里，落盘操作永远是"顺序追加一大块连续数据"，而不是"随机定位到磁盘某个扇区改一个字节"。对机械硬盘而言，顺序写和随机写的性能差距可以达到几十上百倍；即便在 SSD 上，顺序写依然能减少写放大、更充分利用带宽。这正是 LSM-Tree 写入吞吐量远高于 B+ 树的根本原因。

读取路径：为什么相对弱

代价出现在读取上。因为一条 key 的最新值可能在 Memtable 里，也可能在磁盘上任意一个 SSTable 文件里（旧版本可能分散在多个尚未合并的文件中），一次读取在最坏情况下需要：

  • 先查内存里的 Memtable；
  • 如果没有，按从新到旧的顺序依次查磁盘上的多个 SSTable 文件，直到找到该 key 或者确认它不存在。

为了避免每次读都扫描全部 SSTable，实践中会加布隆过滤器（Bloom Filter）快速排除"这个 key 肯定不在这个文件里"的情况，并且 compaction 本身也是在减少需要检查的文件数量。但即便如此，LSM-Tree 的读取路径在理论上依然比 B+ 树"顺着索引一路下钻，一次磁盘定位就能读到数据"要复杂。

与 B+ 树的本质差异：原地更新 vs 追加合并

09-数据库模块讲过，InnoDB 用的 B+ 树是原地更新（in-place update）结构：数据行在磁盘上有固定位置，更新时直接定位到那个数据页并修改。这带来的是读写相对均衡的性能特征——不管是查询还是更新，都是"顺着树结构定位到目标页，读或者改"，路径长度接近，性能比较稳定可预测。而 LSM-Tree 用的是追加与合并（append + compaction）的思路，牺牲部分读性能和后台 compaction 的额外 IO 开销，换来远超原地更新结构的写入吞吐量。

B+ 树（如 InnoDB）LSM-Tree（如 LevelDB / RocksDB）
写入方式 原地更新，定位到具体数据页修改 只追加，写入 Memtable 再顺序刷盘
写入性能 受随机 IO 影响，相对较慢 顺序 IO，吞吐量高
读取性能 稳定，索引定位后一次读取 可能需要查多个 SSTable 文件，配合布隆过滤器优化
额外开销 无（更新即完成） 后台 compaction 消耗额外 CPU 和 IO
适合场景 读写均衡、需要稳定低延迟查询 写多读少，如日志、时序数据、埋点数据
常见误区："LSM-Tree 读比 B+ 树慢，所以整体性能更差"。这个结论忽略了负载场景——如果业务是写多读少（比如日志采集、监控指标、消息队列的持久化），LSM-Tree 用可控的读放大换来的写入吞吐量提升，往往是决定性的优势；反过来如果业务是频繁的点查、范围查询，且写入量不大，B+ 树的稳定读性能更合适。没有哪个结构"整体更好"，只有"更适合哪种负载"，这与前面讲的"扩展方案要基于负载设计"是同一个思路的延伸。
面试高频："LSM-Tree 和 B+ 树的区别，各自适合什么场景？" 标准答案框架：B+ 树原地更新，读写性能均衡，适合通用型 OLTP 数据库；LSM-Tree 只追加写、后台合并，写入性能极强但读取需要合并多个文件、依赖布隆过滤器优化，适合写多读少的场景。能进一步说出"顺序 IO vs 随机 IO"是两者性能差异的根本原因，以及 compaction 会带来额外的写放大（write amplification）开销，是加分项。

代表实现

LSM-Tree 思想被广泛应用在需要高写入吞吐的存储系统中：LevelDB（Google 开源的嵌入式 KV 存储，LSM-Tree 的经典实现）、RocksDB（Facebook 基于 LevelDB 改进，被大量数据库和消息队列用作底层存储引擎）、以及 Cassandra（分布式宽列数据库，其单机存储引擎就是 LSM-Tree），都是理解这一技术路线时值得进一步了解的具体系统。

核心价值总结

  • 核心思路是"只追加 + 后台合并"：写入永远顺序追加到 Memtable 再刷成不可变的 SSTable，用顺序 IO 换取远超原地更新的写入吞吐量。
  • 代价是读取路径变复杂：可能要查多个 SSTable 文件，需要布隆过滤器和 compaction 来控制这个代价。
  • 选型看负载而非"谁更先进"：写多读少（日志、时序数据）优先考虑 LSM-Tree，读写均衡、需要稳定低延迟查询优先考虑 B+ 树。下一部分进入"复制与分区"，看数据如何在多台机器之间同步与切分。

与其他知识点的联系

  • 关联知识点：应对负载增长的常见思路：那一篇提到"写多读少可以选用天生擅长高吞吐写入的存储引擎"，LSM-Tree 正是这句话的具体落地——在单机层面就能靠顺序 IO 把写入吞吐大幅拉高，是垂直扩展路线里应对写压力性价比很高的一种手段，不一定非要走到水平扩展/分区那一步。
  • 关联知识点：数据分区的策略与数据倾斜问题：LSM-Tree 解决的是单机存储引擎层面的写入吞吐问题，而分区解决的是数据量或并发量超过单机能力上限之后如何横向切分——两者不是互斥关系，实践中经常同时使用：每个分片内部用 LSM-Tree 作为存储引擎（Cassandra 就是典型例子），分片解决横向扩展，LSM-Tree 解决纵向的单机写入效率。
  • 关联知识点：主从复制中的复制延迟问题（下一篇）：LSM-Tree 里 Memtable 到 SSTable 的刷盘、WAL 日志的写入，都是"顺序追加写"这一思路的体现；主从复制里 Leader 向 Follower 发送的复制日志，同样是一份顺序追加的日志流——理解了 LSM-Tree 里日志和数据分离、顺序写优先的设计，再看复制日志的传输和重放会更容易建立直觉。
