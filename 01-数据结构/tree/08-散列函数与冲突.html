散列函数与冲突第5章

树把查找压到 O(log n) 已经很快,散列表更进一步——放弃"比较"这个动作本身,直接用计算定位到数据,换来接近 O(1) 的查找。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 5 章「散列」

一、核心思想:用计算代替比较

无论是 BST、AVL 还是红黑树,查找的本质都是不断比较,逐步缩小范围,代价是 O(log n)。散列表(Hash Table)换了一种完全不同的思路:不比较,直接算出数据该在哪。具体做法是用一个散列函数(hash function)把任意 key 映射成一个整数,再对数组长度取模,得到数组下标——存取数据时直接跳到这个下标,理想情况下不需要经过任何比较,平均时间复杂度是 O(1)。

类比:树查找像是"图书馆按字母顺序摆书,你一节节比对目录逐步靠近";散列表更像"每本书直接按照书名算出的编号摆在固定格子里,你知道书名就能算出格子号,一步到位"——前提是这个"编号算法"设计得足够好。

二、一个好的散列函数应该具备什么性质

  • 分布均匀:对于实际会用到的 key 集合,算出来的下标应该尽可能均匀撒在整个数组范围里,不能出现大量 key 挤在少数几个下标上。分布不均会让某些位置堆积大量数据,拖慢那部分的查找速度,整个哈希表的"平均 O(1)"承诺就名不副实。
  • 计算要快:散列函数本身是每次存取都要执行的开销,如果计算它比直接遍历比较还慢,就失去了意义。好的散列函数通常只涉及简单的位运算、乘法、取模,避免复杂逻辑。

三、冲突为什么不可避免:鸽笼原理

无论散列函数设计得多好,只要可能的 key 数量超过数组的槽位数量,就一定会有两个不同的 key 被算到同一个下标——这就是冲突(collision)。这不是"设计得不够好"导致的偶然现象,而是数学上的必然:鸽笼原理(Pigeonhole Principle) 说,如果有 n 个鸽子要塞进 m 个笼子,当 n > m 时,必然有至少一个笼子塞进了 2 只以上的鸽子。哈希表的槽位数量是有限的,而理论上可能的 key(比如所有字符串)是无限的,冲突在数学上必然发生,散列表的设计目标从来不是"消除冲突",而是"高效地处理冲突"——这正是后面两篇要讲的链地址法和开放寻址法要解决的问题。

四、联系 Java:hashCode() 与 equals() 的契约

散列表要正确工作,依赖一个基本前提:相等的对象必须产生相同的散列值,否则同一个逻辑上"相等"的 key 会被分散到不同的桶里,导致查找失败。Java 把这个前提固化成了 Object 的一条契约(contract):

极高频面试题:
  • 如果两个对象 equals() 返回 true,它们的 hashCode() 必须相等。(这是硬性要求,否则 HashMap/HashSet 会出现"存进去却再也找不到"的诡异 bug——因为查找时先用 hashCode 定位桶,再用 equals 逐一比对,如果 hashCode 都不一致,根本走不到 equals 那一步。)
  • 反过来不成立:hashCode() 相等,equals() 不一定相等。这正是"冲突"在 Java 里的具体体现——不同的对象完全可以拥有相同的 hashCode,散列表本来就要处理这种情况。

这也是为什么很多教材强调"重写 equals() 就必须同时重写 hashCode()"——如果只重写 equals(),两个字段完全相同的对象可能因为默认 hashCode()(通常基于内存地址)不同,被分到不同的桶,永远不会被判定为"相等",HashMap 里会出现重复 key、HashSet 里会出现重复元素,这是初学者最容易踩的坑之一。

Java · 违反契约的典型错误
class Point {
    int x, y;
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }
    // 忘记重写 hashCode():默认实现基于内存地址,两个"逻辑相等"的 Point
    // 会算出不同的哈希值,被分到不同的桶 —— HashSet 里会出现"重复"的 Point
}

小结

  • 散列表用散列函数把 key 映射到数组下标,靠"计算"代替"比较",平均查找/插入/删除是 O(1)。
  • 好的散列函数要分布均匀(减少堆积)、计算要快(否则失去意义)。
  • 鸽笼原理决定了冲突在数学上不可避免,散列表设计的重点是"高效处理冲突"而非"消除冲突"。
  • Java 的 hashCode/equals 契约:equals 相等则 hashCode 必须相等;反之不然——这是散列表正确工作的前提,也是最高频的面试考点之一。
