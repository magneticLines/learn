红黑树进阶

AVL 树追求"任何时刻都严格平衡",红黑树选择"允许一点点不平衡",用略慢的查找换来大幅减少的旋转——这是工程妥协的经典范例。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 4 章「树」

一、为什么要"近似平衡"而不是"严格平衡"

上一篇提到 AVL 树的代价:插入/删除后失衡可能沿路径向上传播,需要 O(log n) 次检查甚至多次旋转。在读多写少的场景这无所谓,但在写操作频繁的通用容器场景(比如标准库的 Map/Set),旋转的开销会显著拖累性能。

红黑树的思路是:放弃"绝对平衡"这个目标,只保证"最长路径不超过最短路径的 2 倍"。这个更宽松的约束依然能保证树高是 O(log n)(只是常数因子比 AVL 大一点),但换来的好处是:插入最多需要 2 次旋转,删除最多需要 3 次旋转,其余修复动作都只是改变节点颜色(变色的代价远低于旋转)。这就是"牺牲一点平衡性,换取更少的旋转次数"的核心权衡。

二、五条性质

红黑树给每个节点标记颜色(红或黑),靠以下五条规则间接约束树的形状:

  1. 每个节点是红色或黑色。
  2. 根节点是黑色。
  3. 所有叶子(NIL 空节点)视为黑色(哨兵)。
  4. 红色节点的子节点必须是黑色(即不允许两个红色节点相连)。
  5. 从任一节点到其每个叶子的路径上,黑色节点数目相同(黑高一致)。
理解捷径(2-3-4 树同构):把每个红色节点和它的黑色父节点"捏"成一个逻辑上的大节点,你会发现整棵树变成了一棵严格意义上所有叶子都在同一层的 2-3-4 树(4 阶 B 树)。规则 5 的"黑高一致"本质上就是 B 树"所有叶子同层"这一绝对平衡特性在二叉树形态下的投影;规则 4 则限制了一个"逻辑大节点"里最多只能塞 3 个键(1 黑 + 2 红),不会无限膨胀。理解了这层对应关系,五条规则就不再是死记硬背的教条。

三、插入的直觉:新节点先涂红,再修复

新插入的节点默认是红色。为什么?因为插入红色节点不会破坏规则 5(黑高不变),最多只会破坏规则 4(可能出现红父红子相连,即"双红冲突")——把需要修复的规则从两条减少到一条,修复逻辑自然更简单。修复主要靠两种手法交替进行:

  • 变色(Recolor):如果"叔叔"节点也是红色,说明这一层局部还有调整空间,直接把父亲和叔叔变黑、祖父变红,把问题递归地推给上一层处理——不需要任何指针操作。
  • 旋转(Rotate):如果叔叔是黑色,变色已经无法解决问题,才需要旋转(和 AVL 的左旋/右旋是同一套操作),旋转后局部结构立即满足规则,不会再向上传播。

这也是红黑树旋转次数少的根本原因:大部分失衡情况优先靠"改颜色"这种低成本操作解决,只有变色解决不了时才动用旋转这种高成本操作。

四、与 AVL 树的对比

维度AVL 树红黑树
平衡程度严格(高度差 ≤ 1)近似(最长路径 ≤ 2×最短路径)
树高更低,约 1.44 log n略高,约 2 log n
查找性能更优(树更矮)略逊于 AVL
插入/删除的旋转次数可能沿路径多次旋转插入 ≤ 2 次,删除 ≤ 3 次,其余是变色
适用场景读多写少(如很少变动的索引字典)读写都频繁的通用场景
典型应用早期数据库索引、Windows VAD 内存管理树TreeMap/TreeSet、Linux CFS 调度器、epoll

五、与 Java 标准库的联系

Java java.util.TreeMap 和 TreeSet 底层就是一棵标准红黑树(源自 Doug Lea 编写的经典 CLRS 风格实现)——这也解释了为什么它们的插入、删除、查找都稳定在 O(log n),同时又能保持 key 的有序性(靠中序遍历性质,回顾第一篇)。

面试高频联系点:JDK 8 之后,HashMap 的每个桶(bucket)如果链表长度超过 8,会把该桶从链表转换成一棵小型红黑树(链表长度 < 6 时会转换回来,避免频繁抖动)。这样做的原因是:链表在极端哈希碰撞下会退化成 O(n) 的查找,而少数几个桶转成红黑树后能把最坏情况兜底到 O(log n)。之所以选红黑树而不是 AVL 树,正是因为 HashMap 场景下同一个桶的插入删除也可能频繁发生,红黑树旋转更少的优势在此依然适用。

小结

  • 红黑树用五条性质保证"近似平衡"(最长路径 ≤ 2×最短路径),本质是 2-3-4 树的二叉树投影。
  • 插入的节点默认红色,修复时优先靠低成本的"变色"解决,只有变色无效才动用旋转。
  • 相比 AVL:查找略慢,但插入/删除的旋转次数显著更少,更适合读写都频繁的通用场景。
  • Java TreeMap/TreeSet 底层是红黑树;HashMap 链表长度超过 8 转红黑树兜底最坏情况。
