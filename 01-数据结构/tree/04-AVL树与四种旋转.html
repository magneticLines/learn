AVL 树与四种旋转

普通 BST 把树形的好坏完全交给运气(插入顺序),AVL 树则在每一步插入后主动检查、主动纠正——用局部的小调整,换来全局的高度可控。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 4 章「树」

一、AVL 树解决的问题:强制平衡

上一篇讲到 BST 的致命弱点——有序数据插入会让树退化成链表,查找从 O(log n) 劣化为 O(n)。AVL 树(Adelson-Velsky 和 Landis 提出)是第一种被广泛使用的自平衡二叉搜索树,它的核心思想只有一句话:

AVL 铁律:任意节点的左右子树高度差(平衡因子)的绝对值不超过 1。一旦某次插入/删除导致某个节点违反这条规则,立即通过"旋转"进行局部调整,把树重新掰回平衡状态。

这条铁律保证了树的高度始终是 O(log n) 级别(可以证明高度上界约为 1.44 log n),从而无论插入顺序如何,查找、插入、删除都稳定在 O(log n)——这就是 AVL 树对 BST 最根本的改进:把"看运气"变成了"有保证"。

二、平衡因子:失衡的度量标准

平衡因子(Balance Factor) = 左子树高度 − 右子树高度。每个节点额外维护一个 height 字段(而不是直接存平衡因子),因为高度更方便通过子节点递归计算得到。平衡因子的取值只有三种合法状态:-1、0、1;一旦插入/删除后变成 2 或 -2,就说明该节点"失衡"了,必须旋转修复。

三、四种旋转:对症下药

失衡有四种典型形态,分别对应四种修复手法。记忆的关键不是死记硬背,而是理解失衡发生在"插入路径"的哪个方向组合上:

失衡类型触发场景修复手法
LL(左左)在左孩子的左子树插入导致失衡对失衡节点做一次右旋
RR(右右)在右孩子的右子树插入导致失衡对失衡节点做一次左旋
LR(左右)在左孩子的右子树插入导致失衡先对左孩子左旋,再对失衡节点右旋
RL(右左)在右孩子的左子树插入导致失衡先对右孩子右旋,再对失衡节点左旋

直觉记法:LL/RR 是"一条直线"的失衡,一次旋转就能拉直;LR/RL 是"拐了个弯"的失衡,先把弯拉直变成直线情况,再套用上面的单次旋转。

Java · 右旋(修复 LL 失衡)
//      y                x
//     / \              /   \
//    x   T3   -->     x1    y
//   / \                    /  \
//  x1  x2                 x2   T3
// 抓住失衡节点 y,把它的左孩子 x 顶上来当新的子树根
Node rightRotate(Node y) {
    Node x = y.left;
    y.left = x.right;   // x 的右子树"过继"给 y 当左子树,不会破坏 BST 有序性
    x.right = y;
    updateHeight(y); updateHeight(x); // 顺序不能反:必须先算完子节点 y 的新高度,x 才能算对
    return x;              // x 成为这棵子树的新根,调用者要用它替换旧的引用
}

四、演示:插入序列 3, 2, 1 触发一次右旋(LL)

依次插入 3、2、1:插入 3、2 后树是 3 → 左孩子 2,平衡因子为 1,仍合法。再插入 1,它落在 2 的左子树,使节点 3 的平衡因子变成 2——典型的 LL 失衡(在左孩子的左子树插入)。对 3 做一次右旋,2 顶替成为新根,1 和 3 分别变成它的左右孩子,树恢复平衡且高度从 3 降到 2。

插入 3, 2, 1 的过程
插入3   插入2      插入1(失衡)      右旋后
 3       3            3                 2
        /            /                 / \
       2            2                 1   3
                    /
                   1   ← 平衡因子(3) = 2,触发 LL 右旋

如果继续插入 1.5(落在 1 的右子树),则会先形成 LR 形态,需要"先左旋后右旋"两步才能修复——这也是为什么 LR/RL 被称为"双旋转"。

五、代价:写入变贵了

陷阱提醒:AVL 树用"更贵的插入/删除"换来了"更稳的查找"。每次插入/删除后,失衡可能沿着从叶子到根的路径逐层传播,理论上最多需要检查/旋转 O(log n) 层。在读多写少的场景这个代价完全值得,但在插入删除极其频繁的场景(比如通用容器库),这个代价就成了负担——这正是下一篇要讲的红黑树要解决的问题:放宽一点平衡要求,换取更少的旋转次数。

小结

  • AVL 树用"平衡因子 ∈ {-1,0,1}"的铁律,强制树高稳定在 O(log n),不再依赖插入顺序的运气。
  • 四种旋转按失衡形态对症下药:LL→右旋,RR→左旋,LR→左旋再右旋,RL→右旋再左旋。
  • "一条直线"的失衡一次旋转解决,"拐弯"的失衡需要先拉直再旋转(双旋转)。
  • 代价是写操作变贵——插入/删除可能引发沿路径向上传播的多次旋转,这是红黑树要优化的方向。

与其他知识点的联系

  • 红黑树的旋转与再平衡策略对比:红黑树同样用左旋/右旋修复失衡,代码形式和 AVL 几乎一样,但触发条件更宽松——红黑树允许局部高度差在一定范围内(黑高相等即可,红节点不计入黑高),用"变色 + 更少的旋转次数"换取比 AVL 更便宜的插入删除,这正是 AVL 小结里提到的"下一篇要优化的方向"的具体做法。
  • Java TreeMap / TreeSet 底层实现:Java 集合框架没有直接用 AVL 树,而是选择了红黑树(java.util.TreeMap),原因正是 AVL 严格平衡带来的旋转开销在通用容器的写多场景下不划算;但理解 AVL 的旋转原理是读懂 TreeMap 源码里 rotateLeft/rotateRight 以及 fixAfterInsertion 的必经之路,两者的旋转手法在代码结构上高度相似。
  • B 树 / B+ 树与数据库索引:AVL 树解决的是"内存里如何让二叉树不退化",而数据库索引面对的是磁盘 IO 更贵的场景,B 树/B+ 树通过增大分支因子(一个节点存多个键)来压低树高、减少磁盘访问次数,是同一个"控制树高以保证操作复杂度"的思路在不同硬件约束下的延伸。
  • 旋转操作与树高度的复杂度证明:AVL 树高度上界约 1.44 log n 的证明,依赖于"平衡因子受限的树,节点数关于高度的增长至少是指数级"这一递推关系(类似斐波那契数列的增长速度),这是复杂度分析中"用递推关系证明渐进上界"的典型例子,也是很多算法教材讲平衡树时的经典证明题。
  • 面试高频"手写 AVL 树插入/旋转":面试中常要求现场写出四种旋转的代码,考察点不是死记代码,而是能否说清楚"为什么这么转不会破坏 BST 的有序性"(旋转前后中序遍历结果不变),以及能否准确判断失衡类型(LL/RR/LR/RL)属于哪一种。
  • Treap(树堆):Treap 是另一种自平衡策略——给每个节点额外随机赋一个优先级,靠随机化而不是严格的旋转规则来保证树高期望是 O(log n),是"用概率保证平衡"与"用确定性规则(如 AVL 的平衡因子)保证平衡"这两种设计哲学的直接对比。
