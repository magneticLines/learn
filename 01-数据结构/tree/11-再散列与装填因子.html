再散列 Rehashing 与装填因子

装填因子是哈希表性能的"仪表盘",再散列是它超标后的"扩容大手术"——两者共同决定了哈希表能不能一直保持接近 O(1)。

📖 来源:Weiss《数据结构与算法分析》 第 5 章「散列」

一、装填因子:为什么要死死盯住它

装填因子(load factor)λ = 元素个数 / 桶数组容量,是哈希表里最重要的一个数字,直接决定平均探测/比较次数:

  • λ 太高:分离链接法下链表变长,平均查找退化为 O(λ);开放定址法下探测序列变长,λ 越接近 1,探测次数增长越剧烈(理论上 λ→1 时期望探测次数趋于无穷)。
  • λ 太低:桶数组绝大部分是空的,内存被大量浪费,而且遍历空桶本身也有 cache miss 的隐性成本。

所以装填因子的作用是给"时间"和"空间"这两个互相矛盾的目标找一个平衡点——这也是后面理解 HashMap 为什么选 0.75 的关键。

二、再散列:扩容不是简单的数组拷贝

普通数组扩容(比如 ArrayList)只需要把旧元素按原顺序拷到新数组,下标不变。但哈希表扩容完全不同:下标是由哈希值对容量取模算出来的,容量一变,几乎所有元素的目标下标都会变。所以哈希表扩容必须做再散列(rehashing)——新建一个更大的桶数组,把旧表里的每一个元素重新计算哈希、重新定位、重新插入新表,旧表整个丢弃。

Java · 再散列的核心步骤
void rehash() {
    Node<K,V>[] old = table;
    table = new Node[old.length * 2];  // 容量翻倍(常见策略)
    for (Node<K,V> head : old) {
        for (Node<K,V> cur = head; cur != null; cur = cur.next)
            insert(table, cur.key, cur.value); // 按新容量重新算下标插入
    }
}

触发时机与均摊复杂度

触发条件通常是"插入后 λ 超过阈值"(如 size / capacity > 0.75)。单次再散列是 O(n) 的重操作(n 个元素都要重新插入一次),看起来很贵,但因为容量按倍数增长(不是加固定值),两次再散列之间能插入的新元素数量也按倍数增长,用均摊分析(amortized analysis)——这正是前面讲 ArrayList 扩容时用过的同一套思想——可以证明:从空表插入 n 个元素,总再散列开销是 O(n),均摊到每次插入只是 O(1)。

直觉验证: 容量翻倍扩容,前后两次再散列处理的元素数呈几何级数增长(…,n/8, n/4, n/2, n),等比数列求和收敛到 O(n),而不是简单地把每次 O(n) 乘以扩容次数 O(log n) 变成 O(n log n)——这是均摊分析里最常见的误算点。

三、联系 Java HashMap:0.75 这个数字怎么来的

HashMap 默认负载因子是 0.75,不是随便定的,而是时间与空间的权衡结果:

负载因子取值效果
过低(如 0.5)桶利用率低,一半空间浪费,但冲突少、查找更快
过高(如 0.9+)空间利用率高,但链表(或探测序列)变长,查找变慢
0.75(HashMap 默认)泊松分布下链表长度期望很小(绝大多数桶 0~2 个节点),在空间利用率和查找效率间取得工程上验证过的平衡点

HashMap 的扩容(resize())同样是容量翻倍(newCap = oldCap << 1),触发条件是 size > threshold(threshold = capacity * loadFactor)。翻倍策略保证了扩容的均摊复杂度是 O(1),这与本篇讲的再散列均摊分析完全对应——甚至 HashMap 还做了一个巧妙优化:因为容量永远是 2 的幂,扩容后一个旧桶里的元素只会拆分到新表的两个桶(原下标 / 原下标+oldCap),不需要重新计算完整哈希值,只需看哈希值多出来的那一个 bit,进一步降低了再散列的常数开销。

四、常见陷阱

陷阱 1 · 以为扩容按固定步长增长。 如果扩容是"容量 +1"这种线性增长策略,均摊分析会失效,总开销退化为 O(n²)——这是为什么几乎所有实现都选择"倍增"而不是"递增"。
陷阱 2 · 构造时不预估容量。 如果提前知道要存大量数据,却用默认小容量的 HashMap,会触发多次再散列,产生不必要的开销;应该在构造时用 new HashMap<>(expectedSize) 预留足够容量,一次到位。
陷阱 3 · 忽视迭代顺序被打乱。 再散列后元素在桶数组中的物理位置完全变了,遍历顺序也会随之改变——这是为什么 HashMap 从不保证遍历顺序,需要顺序保证时应该用 LinkedHashMap。

小结

  • 装填因子 λ = 元素数/容量,是哈希表时间与空间的核心权衡指标。
  • 再散列:容量变化后所有元素的桶下标都要重新计算,必须整体重建;倍增扩容配合均摊分析,把 O(n) 的重建开销摊到 O(1)/次。
  • Java HashMap 默认负载因子 0.75 + 容量倍增,是同一套均摊分析思想在真实源码里的落地,与 ArrayList 扩容的设计哲学一脉相承。
