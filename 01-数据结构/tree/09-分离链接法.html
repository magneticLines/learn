分离链接法 Separate Chaining

解决哈希冲突最直观的思路:同一个桶挂一个"次级容器"(链表或树),冲突的元素都塞进去,而不是抢桶位。

📖 来源:Weiss《数据结构与算法分析》 第 5 章「散列」

一、它解决的问题

哈希函数再精心设计,也逃不开抽屉原理:桶数有限,键的取值空间无限,必然存在不同的键被映射到同一个桶下标——这就是冲突(collision)。分离链接法的思路很朴素:不去争抢那个桶位,而是把桶从"存一个元素"变成"存一组元素"。每个桶数组元素不再直接存值,而是存一个链表(或更高级的结构)的头指针,所有哈希到同一下标的键都挂在这条链上。

这正是《数据结构与算法分析》给出的第一种、也是最基础的冲突解决方案——它把"哈希函数选址"和"同址元素管理"这两个问题彻底解耦,哈希函数只管定位到哪个桶,桶内部再用链表处理"撞车"。

二、插入 / 查找 / 删除的实现思路

Java · 链地址法核心结构
class Node<K, V> {
    K key; V value; Node<K,V> next; // 每个桶是一条单链表
}
Node<K,V>[] buckets;

// 查找:定位桶后,链表内最多遍历一遍,逐个比较 key
V get(K key) {
    int idx = hash(key) % buckets.length;
    for (Node<K,V> cur = buckets[idx]; cur != null; cur = cur.next)
        if (cur.key.equals(key)) return cur.value;
    return null;
}
  • 插入:定位桶下标后,先遍历链表看 key 是否已存在(存在则覆盖 value),不存在则头插新节点——头插是 O(1),不用像尾插那样遍历到链尾。
  • 查找:定位桶,顺着链表逐个比较 key,直到命中或到链尾。
  • 删除:定位桶,遍历时记录前驱节点,找到目标后把前驱的 next 跳过它即可,和普通单链表删除节点完全一样。

可以看到,分离链接法本质是"数组 + 链表"的组合:数组负责 O(1) 定位到大致区域,链表负责在小范围内精确查找——这也是为什么它常被称为"桶排序思想在查找问题上的应用"。

三、装填因子如何影响性能

设桶数为 M,元素数为 N,装填因子 λ = N / M 近似等于每条链的平均长度。查找一个键平均要比较约 λ 次(命中时约 λ/2,不命中时约 λ),也就是说分离链接法的平均时间复杂度是 O(1 + λ)——常数项之外还有一个和装填因子成正比的项。这就是为什么实现里必须在装填因子超过阈值时触发扩容+再散列,把 λ 重新拉回较小的值(下一篇会展开讲这个再散列的过程)。

四、联系 Java HashMap 源码

这是本篇的重点。java.util.HashMap 在 JDK 8 之前就是纯粹的分离链接法实现:Entry[] table 数组每个槽位是一条单链表,put 冲突时头插(JDK 7 之后改尾插以避免并发扩容时的链表环问题),get 就是"定位桶 + 遍历链表比较 key"。

JDK 8 的关键优化:链表转红黑树。 如果某个桶的链表长度超过阈值 TREEIFY_THRESHOLD = 8(且数组容量 ≥ 64),这条链会被转换成一棵红黑树。原因很直白:极端情况下(哈希函数被攻击、或者哈希分布极不均匀)所有元素可能挤进同一个桶,链表查找退化为 O(n);转成红黑树后,同一桶内的查找从 O(n) 降到 O(log n),兜住了最坏情况。反过来,如果树节点数缩减到 UNTREEIFY_THRESHOLD = 6 以下,又会退化回链表——因为节点数很少时,红黑树的维护开销(旋转、变色)反而比链表遍历更贵。

这也解释了 hashCode() / equals() 契约为什么重要:HashMap 靠 hashCode() 定位桶,靠 equals() 在链表/树内部精确匹配键——两者任一写错,分离链接法这套机制就会失效(查得到但比较不出来,或者插入了逻辑相同却被当成不同键的重复项)。

五、常见陷阱

陷阱 1 · 只关心平均复杂度,忽视最坏情况。 教材上说分离链接法平均 O(1+λ),但如果哈希函数设计得差(比如所有键的哈希值 mod 桶数后都落在同一格),会退化为一条长链表,整体变成 O(n) 的线性查找——这正是 JDK 8 引入红黑树优化的动机。
陷阱 2 · 忘记重写 hashCode/equals。 自定义对象作为 HashMap 的 key,若只重写 equals 不重写 hashCode(或反过来),会导致"逻辑相等的对象被分到不同桶"或"能定位到桶却比较不出相等",这是面试高频考点。
陷阱 3 · 桶数组不能无限扩,链表也不能无限长。 一味增大桶数组能降低装填因子,但会浪费内存;完全不控制装填因子,链表会越挂越长。二者需要配合再散列机制动态平衡,而不是靠单一手段解决。

小结

  • 分离链接法:冲突的键挂到同一个桶对应的链表(或树)上,而不是抢占别的桶位。
  • 平均时间复杂度 O(1 + λ),λ 是装填因子(平均链长)。
  • HashMap JDK 8 之前纯链表实现;JDK 8 起链长 ≥ 8 时转红黑树,把最坏情况从 O(n) 优化到 O(log n)。
  • 依赖 hashCode()/equals() 契约的正确实现,这是最容易踩坑的地方。
