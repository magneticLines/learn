左式堆 Leftist Heap 与合并进阶

普通二叉堆什么都好,唯独"合并两个堆"这件事做得很差。左式堆就是为了把合并操作也做到 O(log n) 而专门设计的。

📖 来源:Weiss《数据结构与算法分析》 第 6 章「优先队列(堆)」

一、普通二叉堆的软肋:合并很慢

二叉堆基于数组实现,插入、删除堆顶都是 O(log n),非常出色。但如果需求变成"把两个独立的堆合并成一个堆"(比如两个任务队列要汇合成一个),普通二叉堆没有比"暴力"更好的办法:把其中一个堆的全部元素逐个插入另一个堆,每次插入 O(log n),总共 O(n log n)(n 是两堆总元素数)。数组实现天生是"扁平"的连续内存块,两个数组没有办法优雅地"拼接"成一个新的合法堆结构。

这暴露了一个设计原则:用数组换来的"紧凑、cache 友好",牺牲的是结构上的灵活性——这正是左式堆要解决的问题,它换回了链式结构(用指针而非数组下标),用空间换取合并操作的效率。

二、零路径长(Null Path Length, NPL)

左式堆是一棵用节点+指针表示的二叉树(不再是数组),满足堆序性质,外加一条额外的结构性约束——左式性质(leftist property)。要理解它,先定义一个概念:

零路径长 npl(x):节点 x 到它子树中最近的一个"缺孩子"节点(即孩子数少于 2 的节点)的距离。叶子节点的 npl 定义为 0。可以证明 npl(x) = 1 + min(npl(left), npl(right))。

左式性质:对树中每一个节点,都要求左孩子的 npl ≥ 右孩子的 npl。直观理解就是"整棵树整体上往左边偏重"——所有"稀疏、快到头"的部分都被推挤到右侧。这个性质带来一个关键定理:右路径(从根一路往右孩子走到底)的长度是 O(log n)——即便左式堆整体上可能长得很不平衡(左子树可以很深很偏斜),右侧这条路径永远短。

三、合并的递归思路

有了"右路径必然短"这个保证,左式堆的合并操作 merge(h1, h2) 就可以设计成只沿着两棵树的右路径操作,而完全不用碰左子树:

  1. 比较两棵堆的根节点,取较小(小顶堆场景)的作为新的根——这一步保证结果依然满足堆序性质。
  2. 递归地把"较小根节点的右子树"和"另一棵堆"合并,结果挂到新根的右边。
  3. 合并完之后,检查新根的左右孩子的 npl,如果左边比右边小,就交换左右子树——用交换来重新维持左式性质(因为第 2 步只操作了右侧,可能已经把右子树"撑"得比左子树更重)。

因为每次递归只往右路径走一层,而两棵堆的右路径长度都是 O(log n),所以整个合并过程的递归深度是 O(log n),合并操作的总复杂度是 O(log n)——相比普通二叉堆合并的 O(n log n),这是本质的提升。

顺带的好处: 有了 O(log n) 的合并操作,插入操作可以重新定义为"把单个元素看成一个只有一个节点的堆,和原堆合并";删除堆顶可以定义为"去掉根节点后,把左右两棵子树合并成一棵"——左式堆用"合并"这一个基本操作,统一实现了插入、删除堆顶、堆合并三种功能,这是它在算法设计上的优雅之处。

四、与普通二叉堆的对比

维度普通二叉堆(数组)左式堆(链式)
存储结构数组,依赖完全二叉树的下标公式节点+指针,树可以不平衡
插入 / 删除堆顶O(log n)O(log n)(通过合并实现)
合并两个堆O(n log n)(逐个重新插入)O(log n)(沿右路径递归合并)
内存 / cache 友好度连续数组,cache 命中率高节点分散在堆内存,有指针开销
适用场景只需要插入/取最值,不需要合并需要频繁合并多个优先队列(如某些图算法、多路归并调度)

与其他知识点的联系

  • 可合并堆家族对比(左式堆/斜堆/二项堆/斐波那契堆):斜堆(Skew Heap)是左式堆的简化版——不维护 npl,合并时无条件交换左右子树,牺牲严格的 O(log n) 单次保证换来更简单的实现(摊还意义下仍是 O(log n));二项堆和斐波那契堆则进一步把"合并"做到 O(1)(斐波那契堆的 decrease-key 也是 O(1) 摊还),专门为 Dijkstra、Prim 这类需要频繁减小某个元素优先级的算法做了优化。把这几种堆放在一起看,能看清"合并效率"这一个维度上,数据结构设计是如何一步步从"够用"走向"极致"的。
  • 优先队列合并操作的应用场景:多路归并排序中,如果把每一路的"当前最小元素"用小顶堆维护,当某一路耗尽后需要把剩余的堆合并,左式堆的 O(log n) 合并能力就直接派上用场;类似地,任务调度系统里如果两个独立的任务队列(各自按优先级组织成堆)因为业务合并需要汇合成一个队列,左式堆能避免"重新插入所有任务"的 O(n log n) 代价。
  • 与普通二叉堆的合并复杂度对比:这是本文最核心的对比点——数组实现的普通堆合并是 O(n log n)(逐个插入),左式堆凭借"只在右路径上操作"的设计把合并降到 O(log n);这体现了一个反复出现的设计教训:数据结构选择数组还是指针,取决于你最看重"紧凑高效"还是"结构灵活性",没有免费的午餐。
  • 零路径长(NPL)与 AVL 平衡因子的类比:NPL 和 AVL 树的平衡因子都是"用一个局部的数值指标来约束树的全局形状"这一设计手法的具体应用,区别在于 AVL 约束的是"高度差不超过 1"(保证整棵树平衡),左式堆约束的是"左孩子 npl ≥ 右孩子 npl"(只保证右路径短,允许左边整体上很不平衡)——两者都不需要全局重新计算,插入/合并时只需要局部更新沿途节点的指标即可维持性质。
  • 分摊到具体图算法中的价值:Prim 算法求最小生成树、某些网络流算法在增广过程中需要合并多个优先队列结构,如果底层用普通二叉堆,合并会成为整体复杂度的瓶颈;这也是为什么算法教材在讲完基础堆之后,一定会引出左式堆/斐波那契堆这类"可合并堆",作为进阶图算法优化的必要铺垫。

「树与散列」全篇小结

回顾这个 Tab 走过的三条主线,可以看到一条清晰的脉络:它们都在解决"如何组织数据,让某种操作变快"这同一个母问题,只是针对的操作不同。

  • 树这条线(BST → AVL → 红黑树 → 伸展树 → B 树):解决"保持有序的同时支持快速查找/插入/删除"。BST 是起点但最坏退化成链表;AVL 用严格平衡换稳定的 O(log n);红黑树放宽平衡要求换更便宜的插入/删除;伸展树用"最近访问的元素挪到根"的启发式,针对局部性强的访问模式做优化;B 树把分支因子拉大,专门为磁盘 IO 场景把树高压到最低。
  • 散列这条线(冲突解决 → 装填因子):放弃"保持有序",换来平均 O(1) 的查找。分离链接法用链表/树容纳冲突;开放定址法在数组内部腾挪找空位;装填因子和再散列则是维持这套体系长期高效运转的"调节阀"。
  • 堆这条线(二叉堆 → 上/下滤 → d-堆 → 左式堆):放弃"全局有序",只保证"能快速拿到最值",换来 O(log n) 的插入和取最值。二叉堆用数组实现最紧凑的版本;d-堆调节分支因子在插入和删除之间做取舍;左式堆放弃数组紧凑性,换来数据结构里少见的 O(log n) 合并能力。

三条线看似独立,实则反复出现同一组权衡:要不要保持全局有序(树 vs 散列/堆)、用数组还是指针(紧凑 vs 灵活)、分支因子怎么选(树高 vs 节点内开销)。理解了这些权衡背后的原因,比记住每种结构的具体操作步骤更重要——这也是下一个 Tab「排序与图」要继续延伸的思路:排序算法本质上也是在时间、空间、稳定性之间做类似的取舍。
