二叉搜索树 BST(查找 / 插入 / 删除)

数组的查找快但写入慢,链表的写入快但查找慢——BST 试图把二分查找的思想"数据结构化",鱼与熊掌兼得。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 4 章「树」

一、BST 解决的问题:兼顾查找与更新

在 BST 出现之前有两个极端选择:有序数组查找可以二分,O(log n),但插入删除要挪动大量元素,O(n);链表插入删除只需改指针,O(1),但查找只能从头遍历,O(n)。BST 的思路是:把"二分查找"的比较逻辑,搬到一个可以自由生长的链式结构上。

BST 性质:对树中任意节点,其左子树所有节点的值都小于它,右子树所有节点的值都大于它。这条性质是递归的——每一棵子树自身也是一棵合法的 BST。

直觉类比:玩"猜数字"游戏,你猜 50,对方说"大了",你就直接排除 50~100 这一半;猜 25,对方说"小了",你排除 1~25。BST 里的每一个节点就是一次"猜测",每次比较排除一半的搜索空间——这正是二分查找的核心思想。

二、查找与插入:递归/迭代都很直白

查找沿着"比我小去左边,比我大去右边"的规则一路走到底;插入则是"查找失败的地方,就是新节点该放的位置"。

Java · 查找与插入
class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}

TreeNode search(TreeNode node, int key) {
    if (node == null || node.val == key) return node;
    // 每一步比较都排除掉另一半子树,无需真的去遍历它
    return key < node.val ? search(node.left, key) : search(node.right, key);
}

TreeNode insert(TreeNode node, int key) {
    if (node == null) return new TreeNode(key); // 查找失败的空位,就是新节点该待的地方
    if (key < node.val) node.left  = insert(node.left, key);
    else if (key > node.val) node.right = insert(node.right, key);
    return node; // 必须返回,让父节点重新挂接(否则新插入的子树会"断链")
}

三、删除:为什么它是三者中最难的一个

查找和插入只涉及"往下走",删除却要处理"节点没了,谁来接替它位置"的问题,同时还必须维持 BST 的有序性质不被破坏。要删除的节点分三种情况:

情况处理方式
叶子节点(无子节点)直接从父节点摘掉即可,最简单
只有一个子节点让父节点直接指向这个唯一的子节点,"孙子接管祖父的位置"
有两个子节点不能直接摘掉,必须先找一个"替身"顶上,再去删掉替身原来的位置

第三种情况的关键问题是:谁能顶替这个位置,还不破坏"左小右大"?答案是中序后继(右子树的最小值)或中序前驱(左子树的最大值)——它们是树中"刚好比被删节点大一点"或"刚好小一点"的值,放到被删节点的位置上,左右两边的大小关系依然成立。

Java · 删除(以找中序后继为例)
TreeNode delete(TreeNode node, int key) {
    if (node == null) return null;
    if (key < node.val) { node.left  = delete(node.left,  key); return node; }
    if (key > node.val) { node.right = delete(node.right, key); return node; }

    // 找到了要删除的节点:情况 1 & 2,直接用孩子(可能为 null)顶替
    if (node.left == null) return node.right;
    if (node.right == null) return node.left;

    // 情况 3:双子节点,用右子树最小值(中序后继)当替身
    TreeNode successor = node.right;
    while (successor.left != null) successor = successor.left;
    node.val = successor.val;            // 狸猫换太子:只搬值,不搬节点
    node.right = delete(node.right, successor.val); // 再去右子树里删掉真身(它必然是叶子或单子节点,递归到此已简化)
    return node;
}

四、最大的陷阱:数据有序时会退化成链表

面试高频考点:如果依次插入 1, 2, 3, 4, 5,每个新节点都比当前最大值还大,只会不断挂在最右边——整棵树变成一条只有右指针的"链表"。此时树高变成 O(n),查找、插入、删除全部退化为 O(n),BST 引以为傲的 O(log n) 优势荡然无存。

问题的根源在于:BST 的性能完全依赖于树的形状是否"矮胖",而形状又完全取决于插入顺序。普通 BST 没有任何机制去主动纠正一个逐渐"畸形"的结构,它只会老老实实按照插入顺序生长。这不是实现 bug,而是普通 BST 与生俱来的结构性缺陷——它无法应对最坏输入(有序或近似有序的数据)。

这正是引出下一篇AVL 树的动机:如果我们能在插入/删除的同时,让树"自己检测并纠正"失衡的趋势,就能把最坏情况也锁定在 O(log n)。这就是"自平衡二叉搜索树"要解决的问题。

五、和 Java/数据库的联系

Java 的 java.util.TreeMap/TreeSet 本质上都是有序的树结构,但它们底层用的不是普通 BST,而是自平衡的红黑树——原因正是上面提到的退化风险,标准库不能把性能寄托在"用户不会按顺序插入数据"这种假设上。数据库索引(如 MySQL InnoDB 的 B+ 树)也面临同样的问题,只是它们还要额外考虑磁盘 I/O,这些会在后面几篇分别展开。

小结

  • BST 性质:左子树 < 根 < 右子树,是把二分查找思想数据结构化的结果。
  • 查找/插入都是"比较后选择方向"的简单递归;删除麻烦在于双子节点情况要找中序后继/前驱当替身。
  • 最大陷阱:有序数据插入会让 BST 退化成链表,O(log n) 退化为 O(n)。
  • 这个缺陷是 AVL 树、红黑树等自平衡结构存在的根本原因。
