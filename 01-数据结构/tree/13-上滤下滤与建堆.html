上滤 / 下滤与建堆 buildHeap

堆的所有操作都建立在两个基本动作之上:新元素往上"冒泡"到正确位置,或者根节点往下"沉"到正确位置。

📖 来源:Weiss《数据结构与算法分析》 第 6 章「优先队列(堆)」

一、上滤(Percolate Up):插入时怎么维持堆序

插入新元素时,最简单的做法是先放到数组末尾(完全二叉树的下一个空位),这一步天然保证了"完全二叉树"的形状不被破坏。但新元素可能比它的父节点还小(小顶堆场景),堆序性质被打破了,需要修复。

修复方式是上滤:让新元素和父节点比较,如果比父节点小就交换位置,然后在新的位置继续和(新的)父节点比较,像"冒泡"一样一路往根的方向移动,直到找到不需要再交换的位置(或者到达根节点)为止。

Java · 上滤(以小顶堆为例)
void percolateUp(int i) {
    while (i > 0 && heap[i] < heap[parent(i)]) {
        swap(i, parent(i));
        i = parent(i);   // 位置上移,继续和新的父节点比较
    }
}

因为完全二叉树的高度是 O(log n),上滤最多沿着一条路径走到根,所以插入操作是 O(log n)。

二、下滤(Percolate Down):删除堆顶时怎么维持堆序

取出并删除最值(poll())时,根节点被拿走了,通常的处理是把数组最后一个元素挪到根节点位置,再把数组长度减一——这一步同样保证了完全二叉树形状不被破坏。但挪上去的这个元素很可能远大于它现在的两个子节点,需要修复。

修复方式是下滤:让这个元素和它的两个子节点中较小的那个比较(注意,不能只跟其中一个比,否则可能选错方向导致堆序仍然被破坏),如果比子节点大就交换,然后在新位置继续往下比较,像石头一样"下沉",直到没有子节点比它小或到达叶子为止。

Java · 下滤(以小顶堆为例)
void percolateDown(int i) {
    while (left(i) < size) {
        int smaller = smallerChild(i); // 必须挑较小的子节点比较,而非固定挑左或右
        if (heap[i] <= heap[smaller]) break;
        swap(i, smaller);
        i = smaller;
    }
}

同样因为树高 O(log n),下滤也是 O(log n)。

三、建堆 buildHeap:为什么是 O(n) 而不是 O(n log n)

把 n 个无序元素一次性组织成堆,直觉上的做法是"逐个插入 + 上滤",每次 O(log n),总共 O(n log n)。但 Weiss 书中给出了一个更快的经典算法:把 n 个元素按数组顺序放好,从最后一个非叶子节点开始,倒着往前对每个节点做一次下滤,总时间只需要 O(n)。这是很多人直觉上容易算错的地方,值得仔细拆开证明思路。

Java · buildHeap 核心:从下往上逐个下滤
void buildHeap() {
    // 叶子节点天然满足堆序(没有子节点可比),跳过;从最后一个非叶子开始
    for (int i = size / 2 - 1; i >= 0; i--)
        percolateDown(i);
}

为什么总时间是 O(n)

关键在于:大多数节点离叶子很近,下滤的路径很短——这一点"逐个插入"的方式完全没有利用到(逐个插入时,新元素每次都是从叶子层插入,上滤路径长度和它在最终堆里的深度相关,而多数元素最终都堆在下层,平均路径也不长,但每次插入还要动态维护"树形还没定型"的中间状态,分析起来不如 buildHeap 直观)。

完全二叉树中,高度为 h 的层大约有 n/2^(h+1) 个节点,而该层节点下滤的最大路径长度是 h。把每层的"节点数 × 该层下滤路径长度"加总:

总工作量 ≈ Σ (n / 2^(h+1)) · h,对 h 从 0 到 log n 求和。这是一个"每层节点数指数递减、路径长度线性递增"的级数,经典结论是它收敛到 O(n)——因为指数衰减的速度远快于线性增长的速度,底层(节点最多)的下滤路径反而最短,顶层(路径最长)的节点数又少得可以忽略。

直觉类比:一个金字塔状的组织,底层员工(人数最多)出问题只需要"下沉"一两级就能被上级发现处理(路径短);越往高层人数越少,即便处理链条变长,涉及的人数也呈指数减少——两者相乘,总的"工作量"是有限的常数倍 n,不会随着层数增加而爆炸到 n log n。

四、对比:插入法 vs buildHeap 法

建堆方式思路复杂度
逐个插入 + 上滤每插入一个元素就维护一次堆序O(n log n)
buildHeap(从下往上下滤)先把数据全放进数组,再统一从底层非叶子节点开始下滤修复O(n)
实践意义: 如果需要把一批已知的数据一次性变成堆(比如堆排序的第一步),用 buildHeap 而不是逐个插入,能省下一个 log n 因子——这也是堆排序整体复杂度稳定在 O(n log n) 而不是 O(n log²n) 的原因之一。

常见陷阱

陷阱 1 · 下滤时只比较左子节点(或固定比较某一边)。 必须比较两个子节点中较小的那个,否则可能出现"和左边比没问题,但右边其实更小"的错误堆序。
陷阱 2 · 直觉性地认为 buildHeap 是 O(n log n)。 这是本篇反复强调的点:"n 次操作,每次 O(log n),所以总共 O(n log n)"这个直觉在这里是错的,因为绝大多数节点的下滤路径远小于 log n,必须按层加权求和才能得到正确的 O(n) 结论。

小结

  • 上滤(插入用):新元素放到末尾,和父节点比较、交换,一路冒泡到根方向,O(log n)。
  • 下滤(删除堆顶用):末尾元素挪到根,和较小的子节点比较、交换,一路下沉到叶子方向,O(log n)。
  • buildHeap:从最后一个非叶子节点开始倒序下滤,利用"底层节点多但路径短"的特性,总复杂度 O(n),而不是直觉的 O(n log n)。
