开放定址法(线性 / 平方 / 双散列探测)

不额外借助链表,冲突时就在数组内部"挪个位置"——所有数据都放在桶数组本身,靠探测序列找空位。

📖 来源:Weiss《数据结构与算法分析》 第 5 章「散列」

一、核心思路:自己给自己找空位

分离链接法用链表"扩展"桶的容量,而开放定址法(open addressing)反其道而行:所有元素必须直接存在桶数组里,不允许借助额外结构。冲突发生时,不会新建链表节点,而是按某种规则依次探测 下一个候选位置,直到找到空桶为止。这组"依次探测的位置序列"由探测函数决定,不同探测函数就对应不同的开放定址策略。

线性探测(Linear Probing)

探测序列:h(x), h(x)+1, h(x)+2, ...
int findSlot(K key) {
    int idx = hash(key) % table.length;
    int i = 0;
    // 每次冲突就往后挪一格,兜圈子直到找到空位
    while (table[idx] != null && !table[idx].key.equals(key)) {
        idx = (idx + ++i) % table.length;
    }
    return idx;
}

最简单直接,但有一个经典缺陷:一次聚集(primary clustering)。一旦一段连续的桶被占满,任何哈希到这段区间任意位置的新键都会一路探测到这段区间的末尾才找到空位——聚集区越长,越容易吸引新的冲突落进来,雪球效应让聚集区越滚越大,平均探测次数急剧上升。

平方探测(Quadratic Probing)

探测序列改成 h(x), h(x)+1², h(x)+2², h(x)+3², ...,探测步长按平方增长,不再是相邻位置,这样就避免了一次聚集——不会有一整段连续区域互相"传染"。但它引入了新问题:二次聚集(secondary clustering),哈希到同一个初始位置的键,探测序列完全相同,只是没有前一种那么严重(不同初始位置之间不会互相干扰)。

平方探测的一个数学限制: 若表大小 M 是素数,且装填因子 λ ≤ 0.5,则平方探测一定能找到空位(可证明前 M/2 次探测互不相同)。这也是为什么很多开放定址实现要求桶数组大小取素数,并把装填因子上限压得比分离链接法更低。

双散列(Double Hashing)

用第二个独立的哈希函数 hash2(x) 来决定探测的步长,而不是固定用 1、平方等规律:探测序列变成 h(x), h(x)+hash2(x), h(x)+2·hash2(x), ...。因为步长本身由 key 决定,不同 key 的探测路径几乎不会重叠,二次聚集也被消除——这是三种方案里冲突分布最均匀的,但要多算一次哈希,常数开销更高。

二、删除操作的陷阱:为什么需要"墓碑"

这是开放定址法最容易被忽视的坑。 分离链接法删除一个节点,直接把它从链表摘掉即可,毫无副作用。但开放定址法不能这么做:如果直接把某个槽位设为 null,会打断后续元素的探测链——假设 A、B 两个键都哈希到槽位 3,A 落在 3,B 探测到 4;如果删除 A 导致槽位 3 变回 null,之后查找 B 时,从槽位 3 探测,一看是 null 就断定"B 不存在",直接漏查——而 B 其实还在槽位 4。

正确做法是给槽位加一个逻辑删除标记(墓碑,tombstone),状态从"空/占用"变成三态:空(EMPTY)/ 占用(ACTIVE)/ 已删除(DELETED)。查找时遇到 DELETED 要继续探测(不能当成空位停下),插入时遇到 DELETED 才可以复用这个槽位。代价是墓碑会越积越多,拖慢查找效率,所以开放定址表通常也需要定期整体重建(再散列时顺便清掉所有墓碑)。

三、与分离链接法的对比

维度分离链接法开放定址法
存储方式桶数组 + 链表/树(额外内存)全部挤在桶数组内(无额外指针开销)
内存局部性链表节点分散在堆上,cache 命中率较差数据紧凑排列在数组中,cache 友好
装填因子上限可以 > 1(链表能无限挂)必须 < 1(桶数组终归会被填满)
删除操作直接摘链表节点,简单直接必须用"墓碑"标记,不能真删,否则打断探测链
典型问题链表过长退化为线性查找聚集(一次/二次)导致探测变长

四、常见陷阱小结

陷阱 1 · 用 null 直接删除。 上面讲的墓碑问题,是开放定址法面试和实战中最常被问到的坑,一定要记住"删除是标记,不是清空"。
陷阱 2 · 装填因子控制不严。 开放定址法一旦装填因子接近 1,探测序列会越来越长,极端情况下退化为 O(n) 遍历整个数组;实践中通常要求 λ 不超过 0.5~0.7 就要触发扩容。

小结

  • 开放定址法:冲突时在数组内部按探测函数找下一个空位,不借助额外结构。
  • 线性探测简单但有一次聚集;平方探测缓解一次聚集但有二次聚集;双散列用第二个哈希函数决定步长,分布最均匀但开销更高。
  • 删除必须用"墓碑"标记,直接置空会打断探测链导致漏查——这是最容易踩的坑。
  • 相比分离链接法,开放定址法内存更紧凑、cache 友好,但装填因子必须严格小于 1。

与其他知识点的联系

  • Java HashMap 的分离链接法:JDK 的 HashMap 走的是分离链接路线(数组 + 链表,链表长度超过 8 时转红黑树),完全没有开放定址法的"墓碑"问题——删除一个键就是直接摘链表节点;把两者放在一起对比,能直观理解为什么 HashMap 的默认装填因子(负载因子)可以设到 0.75 而不用像开放定址表那样压得更低。
  • ThreadLocal 的哈希实现:ThreadLocal 内部的 ThreadLocalMap 恰恰是用开放定址法(线性探测的变种)实现的,而不是链表——因为每个线程的 ThreadLocalMap 通常很小,链表节点的指针开销相对更贵,而线性探测的数组紧凑存储在小规模场景下反而更划算;它同样存在"删除用 null 占位需要后续清理"的机制,和本文讲的墓碑思想是同一类问题的不同处理方式。
  • 负载因子与扩容:不管是分离链接法还是开放定址法,装填因子(load factor)都是控制冲突概率的核心参数,但两者对装填因子的容忍度完全不同——链地址法允许装填因子 > 1(链表可以无限长),开放定址法必须严格 < 1;这也是为什么设计一个哈希表时,选择哪种冲突解决策略要先想清楚预期的装填因子范围。
  • 哈希函数设计的质量要求:双散列对第二个哈希函数的要求比普通哈希函数更高(不能让 hash2(x) 结果为 0,否则会死循环),这引出了哈希函数设计中"分布均匀性"和"独立性"的话题——好的哈希函数应该让不同的 key 尽量映射到不同的探测序列,这与 HashMap 里对 hashCode() 做"扰动函数"(高低位混合)处理的目的是相通的:都是为了减少因哈希函数质量差而导致的聚集。
  • 再散列(rehashing)与装填因子的联动:本文提到墓碑积累多了需要"重建",这正是下一篇要讲的再散列机制的应用场景之一——不只是空间不够时才扩容,墓碑过多导致探测链变长同样是触发再散列的理由,这体现了哈希表设计里"多个信号共同决定何时重建"的思路。
  • 面试高频"设计一个哈希表":这类系统设计/数据结构设计题几乎必问"冲突怎么解决",回答时对比链地址法和开放定址法的取舍(内存局部性、删除复杂度、装填因子上限),并能说清楚"为什么删除操作在开放定址法里不能直接置空",是考察对哈希表原理理解深度的高频切入点。
