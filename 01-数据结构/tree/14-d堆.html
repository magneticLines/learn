d-堆 d-Heap

二叉堆是"每个节点 2 个孩子"的特例。把孩子数从 2 推广到 d,会发生什么?——一次关于"分支因子"的经典权衡。

📖 来源:Weiss《数据结构与算法分析》 第 6 章「优先队列(堆)」

一、推广的思路:为什么孩子数不一定要是 2

二叉堆规定每个节点最多 2 个孩子,这个"2"其实只是一个可以调整的参数。d-堆把它一般化:每个节点最多有 d 个孩子。下标公式也随之推广:

d-堆下标公式(d=2 时退化为普通二叉堆)
// 节点 i 的第 k 个孩子(k 从 1 到 d)
int child(int i, int k) { return d * i + k; }
// 节点 i 的父节点
int parent(int i) { return (i - 1) / d; }

依然是一棵用数组表示的完全 d 叉树,依然满足堆序性质(父 ≤/≥ 所有孩子),上滤/下滤的思路完全不变——唯一变化的是每个节点能"扇出"多少个孩子。

二、d 变大带来的权衡:上滤变慢,下滤变快

这正是本篇的核心:d 是一个纯粹的分支因子权衡参数,调大它会同时产生两个相反方向的效果。

操作d 变大的影响原因
上滤(插入)变快每上升一层只需跟唯一的父节点比较 1 次,和 d 无关;树变矮后层数变少,总比较次数随之下降
下滤(删除堆顶)变慢树更"矮"(树高 ≈ log_d n)使层数变少,但每层要在 d 个孩子中找出最小者,需要 d-1 次比较,d 越大单层开销越大,往往抵消甚至反超层数减少带来的收益

更精确地说:

  • 树高变矮:n 个节点的完全 d 叉树高度约为 log_d n,d 越大树越矮越"扁"。层数变少对两种操作都是潜在利好。
  • 上滤只吃到"层数变少"的好处:上滤每层只需要和唯一的父节点比较 1 次,这个开销和 d 无关,因此层数减少直接转化为总比较次数减少——d 越大,插入越快。
  • 下滤要多付"每层决策成本"的代价:下滤每到一层都要在 d 个孩子中挑出最小(或最大)的一个才能确定下沉方向,这一步是 d-1 次比较。d 越大,层数减少带来的好处会被单层比较次数增多所抵消——这正是教材里"d 变大插入更快、删除更慢"的权衡来源。
结论: d-堆的性价比取决于插入和删除堆顶操作的相对频率。如果场景是"插入远多于删除"(比如离线批量建图后只做少量取值),适当调大 d 能让占多数的插入操作更快;如果频繁交替插入删除,标准二叉堆(d=2)通常已经是较均衡的选择。

三、联系:分支因子影响树高的思想,在 B 树上也出现过

这其实是数据结构里反复出现的一个母题:分支因子(branching factor)越大,树越矮,但每个节点内部的"决策成本"越高。B 树把这个思想用到极致——每个节点存储几十到几百个键,分支因子拉到很大,换来树高极低(通常 3~4 层就能索引几十亿条记录),代价是每个节点内部需要做一次线性或二分查找来确定往哪个子树走。d-堆是同一个思想在堆这种数据结构上的迷你版本:d 越大,"层数"这个纵向开销降低,"每层内部决策"这个横向开销升高。

理解了这一层,就能看透很多数据结构设计背后的共同逻辑:纵向(树高)和横向(节点内部)的开销此消彼长,分支因子就是调节这个跷跷板的旋钮——无论是内存里的堆,还是磁盘上的 B 树索引,都在这同一个权衡框架下做选择,只是约束条件不同(堆看比较次数,B 树看磁盘 IO 次数)。

常见陷阱

陷阱 · 认为 d 越大总是越好。 d-堆不是"参数越大越强"的结构,它是纯粹的权衡:d 增大对插入有利、对删除堆顶不利。选择 d 之前要先分析实际场景里两种操作的比例,盲目调大 d 反而可能让高频操作变慢。

小结

  • d-堆把二叉堆的孩子数从固定的 2 推广为参数 d,依然是完全 d 叉树 + 堆序性质,数组下标公式随之推广。
  • d 变大:树变矮,层数变少,有利于上滤(插入);但下滤时每层要在 d 个孩子里选最值,单层开销变大,不利于下滤(删除堆顶)。
  • 这是"分支因子权衡树高与节点内决策成本"的经典案例,和 B 树用大分支因子换低树高、减少磁盘 IO 是同一个设计思想的两种应用场景。
