伸展树 Splay Tree进阶

AVL、红黑树都在追求"时刻保持平衡",伸展树反其道而行:不管眼下平不平衡,只关心"刚被访问过的数据,下次会不会很快又被访问"。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 4 章「树」

一、核心思想:把"最近访问"的节点伸展到根

前两篇的 AVL 树和红黑树都在解决同一个问题:无论访问模式如何,都要保证最坏情况下的性能。但现实中很多访问是有局部性(locality)的——刚查过的数据,大概率很快还会被再查一次(比如缓存热点 key、最近编辑的文档)。伸展树(Splay Tree)换了个赌注:它不追求任何时刻都平衡,而是每访问一个节点,就用一系列旋转把它一路"伸展(splay)"到树根。

直觉类比:想象一个不整理书架、只把"刚看完的书"随手插回最顺手位置(书架最前面)的人。虽然书架整体摆放可能很乱(不追求整体有序美观),但因为大部分阅读都有重复性(刚看过的书更容易再被翻),所以"常用的书永远在最前面"反而是效率更高的策略。

伸展操作本身就是复用 AVL/红黑树里那套旋转手法(单旋、双旋),只是触发条件完全不同:AVL/红黑树是"失衡了才旋转",伸展树是"只要访问了就旋转,不管平不平衡"。伸展的具体规则(zig / zig-zig / zig-zag)会根据被访问节点相对于父节点、祖父节点的位置组合选择不同的双旋转手法,但目的始终如一——把它推到根。

二、摊还分析的直觉:单次可能贵,均摊很便宜

既然不维持平衡,伸展树完全可能在某一时刻退化成一条链(比如刚好都在往一个方向插入)——此时单次查找/伸展操作最坏是 O(n)。这听起来很糟糕,但摊还分析(Amortized Analysis)告诉我们一个反直觉的结论:

关键结论:虽然单次操作最坏 O(n),但连续做 m 次操作的总代价是 O(m log n)——平均到每次操作是 O(log n)。直觉上,一次"很贵"的操作(把深埋的节点伸展到根)会顺带把访问路径上其他节点的深度都拉低,相当于"支付了一次高昂的账单,但顺手把周围环境也整理好了",后续访问会因此变得更便宜。

这和 ArrayList 扩容的摊还分析是同一种思维方式:偶尔一次扩容很贵(O(n)),但把这次代价分摊到之前所有的 O(1) 插入上,平均下来每次插入仍是 O(1)。伸展树是把这套"用长期均摊掩盖单次波动"的思想用在了树形结构的形状调整上。

Java · 伸展操作的框架(zig 情形示意)
// zig:被访问节点 x 是根的直接孩子,一次单旋就能把它顶到根
// 这一步复用的正是 AVL 那套左旋/右旋代码,只是触发时机不同
Node splay(Node root, int key) {
    if (root == null || root.val == key) return root;
    // 沿查找路径递归下探,回溯时不断把目标节点往上旋 —— 每访问一次都做,不判断是否失衡
    if (key < root.val) {
        root.left = splay(root.left, key);
        return rightRotate(root);  // 把刚访问的节点往根方向推一层
    } else {
        root.right = splay(root.right, key);
        return leftRotate(root);
    }
}

三、适用场景:访问模式有局部性的地方才划算

伸展树不是"处处都比 AVL/红黑树好",它的优势严格建立在访问模式确实存在局部性这个前提上:

  • LRU 类缓存的底层结构候选:最近使用的 key 天然被伸展到树根附近,和"最近使用"这个语义高度契合。
  • 局部性强的查询负载:比如一段时间内反复查询同一小批数据(热点数据),伸展树能让这批数据始终"浮"在树的上层,查询会越用越快。
  • 反例:均匀随机访问。如果每次访问的 key 完全随机、没有重复模式,伸展带来的"优化下次访问"的收益无法体现,此时伸展树只是白白多付出旋转成本,不如直接用红黑树。
易混淆点:伸展树不保证任意时刻的树高是 O(log n)(这点和 AVL/红黑树本质不同),它保证的是"一段时间内多次操作的平均代价"是 O(log n)。如果面试被问"伸展树最坏情况下单次操作复杂度是多少",正确答案是 O(n),但"m 次操作的总代价"才是 O(m log n)——一定要分清"单次最坏"和"摊还平均"是两个不同的指标。

小结

  • 伸展树不追求随时平衡,而是把每次被访问的节点通过旋转伸展到根,利用访问局部性。
  • 伸展动作复用了 AVL 那套旋转操作,区别只在于触发时机——每次访问都做,不看是否失衡。
  • 单次操作最坏 O(n),但连续 m 次操作的摊还总代价是 O(m log n),这与 ArrayList 扩容的摊还分析同源。
  • 只有访问模式存在局部性(如缓存热点)时才划算;均匀随机访问场景不如红黑树。
