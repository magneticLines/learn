二叉堆 Binary Heap 与堆性质第6章

不需要把所有元素排好序,只需要随时能拿到"当前最小/最大"的那一个——这正是堆存在的意义。

📖 来源:Weiss《数据结构与算法分析》 第 6 章「优先队列(堆)」

一、它解决的问题:不需要全排序,只需要"每次拿最值"

很多场景的真实需求不是"把数据排好序",而是反复问"当前最小/最大是谁",然后可能会插入新元素,再问一次——操作系统任务调度(谁优先级最高先跑)、Dijkstra 最短路(每次取距离最小的未访问节点)、Top-K 问题,都是这个模式。

如果用已排序数组:取最值 O(1),但插入要维持有序,是 O(n)。如果用普通数组/链表(不排序):插入 O(1),但取最值要扫一遍,是 O(n)。二叉堆的目标是让插入和取最值都做到 O(log n)——用一个"半有序"的结构,只保证局部的父子关系,不要求全局排序,换来两种操作的平衡。

二、堆序性质(Heap-Order Property)

二叉堆是一棵完全二叉树,并且满足:每个节点的值都 ≤(或 ≥)它的所有子节点——这叫堆序性质。满足"父 ≤ 子"的叫小顶堆(min-heap),根节点永远是全局最小值;反过来是大顶堆(max-heap)。

注意区分:堆序 ≠ 全排序。 堆只保证"父子"关系,不保证"兄弟"或"跨层"节点之间的大小关系——左子树的某个节点完全可能比右子树的某个节点大。这正是它比 BST 更"松"的地方,也是它能更快维护的原因:调整时只需要沿着一条从根到叶子的路径比较,不需要像排序那样两两比较。

三、为什么用数组就能表示完全二叉树

因为二叉堆要求是完全二叉树(除最后一层外每层都填满,最后一层从左往右填),这种规整性带来一个关键好处:父子关系可以用下标算术公式直接推导,完全不需要显式的左右指针。设根节点下标为 0:

数组下标 ↔ 树形关系
// 节点 i 的左孩子 / 右孩子 / 父节点,纯算术计算,O(1)
int left(int i)   { return 2 * i + 1; }
int right(int i)  { return 2 * i + 2; }
int parent(int i) { return (i - 1) / 2; }

相比链式二叉树(每个节点要存 left、right、可能还有 parent 三个指针),数组实现省掉了所有指针的内存开销,而且数组的连续内存布局对 CPU 缓存更友好(遍历时局部性好)。这是"完全二叉树的规整结构"和"数组的下标寻址能力"的完美契合——堆是数据结构里少数几个"用最简单的容器实现树"的经典案例。

陷阱: 这套下标公式的前提是树必须是完全二叉树。如果是普通二叉树(允许任意位置留空),数组表示会因为大量空洞而浪费内存,甚至在最坏情况下(比如一条链)退化到 O(2ⁿ) 的空间开销——这也是为什么"数组能表示树"这个技巧只对完全二叉树成立。

四、联系 Java PriorityQueue

java.util.PriorityQueue 的底层实现正是本篇讲的二叉堆:内部用一个 Object[] queue 数组存储元素,默认是小顶堆(poll() 每次返回最小元素),可以传入自定义 Comparator 改变排序规则(比如实现大顶堆)。它的 offer()/poll() 分别对应本组下一篇要讲的上滤(sift up)和下滤(sift down)操作,复杂度都是 O(log n)。

与 TreeMap/TreeSet(基于红黑树,全局有序)相比,PriorityQueue 只保证"取最值"这一个操作的高效,不支持按序遍历、不支持 O(log n) 的任意查找——这正是它"性质更弱但实现更简单、常数更小"的取舍。

五、堆能做与不能做的事

操作复杂度说明
查看最值 peek()O(1)永远在数组下标 0(根节点)
插入 offer()O(log n)插到数组末尾,再上滤到正确位置
取出并删除最值 poll()O(log n)根节点和末尾元素交换后删除末尾,再下滤
查找任意元素O(n)堆序性质不支持二分,只能线性扫描——这是堆做不到的事
按顺序遍历所有元素O(n log n)需要反复 poll,不是天然有序结构

小结

  • 堆解决"频繁取最值 + 动态插入"的场景,不追求全排序,换来插入/删除最值都是 O(log n)。
  • 堆序性质:父节点始终 ≤(或 ≥)子节点,只约束父子关系,比 BST 更"松"。
  • 完全二叉树的规整性让数组天然表示树形结构,下标公式 2i+1 / 2i+2 / (i-1)/2 替代显式指针,省内存又对 cache 友好。
  • Java PriorityQueue 就是二叉堆的直接实现,和 TreeMap 的"全序 vs 只取最值"是一组经典取舍对比。
