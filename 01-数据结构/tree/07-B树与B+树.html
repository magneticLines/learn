B 树 / B+ 树(磁盘与数据库索引)进阶

红黑树在内存里已经足够优秀,但一旦数据搬到磁盘上,决定性能的不再是比较次数,而是磁盘 I/O 次数——这彻底改变了"树该长成什么形状"的答案。

📖 来源:Weiss《数据结构与算法分析:Java语言描述》 第 4 章「树」

一、为什么内存里好用的树,搬到磁盘上就不好用了

红黑树、AVL 树的性能分析都建立在一个隐含假设上:访问任意一个节点的代价是均等的、很小的——这在内存里是对的,指针跳转几乎不消耗时间。但磁盘(或 SSD)完全是另一个世界:

  • 磁盘不是"按字节读",而是按块(page,通常 4KB/16KB)读——哪怕你只想要 1 个字节,也要把整个块读进内存。
  • 一次磁盘 I/O 的耗时(毫秒级)相比内存访问(纳秒级)慢了几个数量级。

红黑树是"瘦高"的二叉树,树高是 O(log₂ n)。如果每一层节点都存在磁盘的不同位置,查找一次就要做 O(log₂ n) 次磁盘 I/O——即使 n 不算特别大(比如百万级),log₂ n 也有 20 层,意味着一次查找要付出 20 次磁盘寻道的代价,这在真实系统里是不可接受的。

二、B 树:把树变"矮胖",用一次 I/O 顶多次比较

B 树的设计目标非常直接:减少树的高度,把"多路"塞进一个节点,让一个节点正好装满一个磁盘块。一个 B 树节点不再只存 1 个 key 和 2 个指针,而是存数十到数百个 key和对应数量的子节点指针——这样即使数据量是百万级,树高也能压缩到 3~4 层。

核心权衡:内存里比较的代价几乎为零,所以红黑树宁愿"多比较几次、少存点数据"来保持结构简单;磁盘上 I/O 的代价极高,所以 B 树宁愿"在一个节点里塞爆数据、多比较几次内存里的 key",也要换取"少读几次磁盘"。一次磁盘 I/O 的时间,足够在内存里做成百上千次比较——这就是"矮胖多路"划算的根本原因。

三、B+ 树:B 树的改进版,专为范围查询和顺序扫描优化

B 树已经解决了"减少 I/O 次数"的问题,但实际数据库场景还有一个高频需求:范围查询(比如"查询年龄在 18~25 之间的用户")。B+ 树针对这个需求做了两点改进:

维度B 树B+ 树
数据存放位置所有节点(包括内部节点)都存 key 和数据只有叶子节点存完整数据,非叶节点只存索引 key,不存数据
叶子节点之间无特殊连接叶子节点用链表相连
范围查询需要在树中反复上下跳转定位到起点后,沿叶子链表顺序扫描即可,不再需要多次回溯树
相同数据量下的树高较高(非叶节点也占空间,单节点能容纳的 key 少)更低(非叶节点不存数据,能塞更多 key,分叉更多)

非叶节点"只存索引不存数据"这一点看似简单,实际效果很显著:因为不用存完整数据,同样大小的一个磁盘块能塞进更多的索引 key,意味着每个节点的分支数(阶数)更大,树更矮,一次查找需要的 I/O 次数更少。而叶子节点之间的链表,则让"找到范围起点后一路扫到底"变成了简单的顺序遍历,不需要再回到树中反复查找——这对范围查询、排序、分页这类场景是决定性的优化。

四、联系:MySQL InnoDB 为什么用 B+ 树做索引

实际应用:MySQL 的 InnoDB 存储引擎,主键索引(聚簇索引)和二级索引底层都是 B+ 树。原因正好对应上面两点:(1) 数据库表往往是千万级甚至更大,B+ 树的"矮胖"特性把一次查询压缩到 3~4 次磁盘 I/O 以内;(2) SQL 里的 WHERE age BETWEEN 18 AND 25、ORDER BY、分页查询,都需要高效的范围扫描,B+ 树叶子链表让这类查询可以直接顺序读取,不再需要每条记录都单独做一次树查找。

这条路径完整回答了"为什么内存和磁盘要用不同的树":内存结构(红黑树)优化的是比较次数,磁盘结构(B+ 树)优化的是I/O 次数——评价标准变了,最优结构自然也变了,这也是数据结构选型里"脱离场景谈优劣毫无意义"的典型例子。

小结

  • 磁盘按块读取,一次 I/O 远比一次内存比较昂贵,内存里"瘦高"的红黑树搬到磁盘上会导致过多 I/O。
  • B 树用"矮胖多路"的设计,让一个节点装满一个磁盘块,大幅压缩树高,减少 I/O 次数。
  • B+ 树在 B 树基础上把数据全部下沉到叶子节点、非叶节点只存索引,并用链表连接叶子,进一步降低树高并优化范围查询。
  • MySQL InnoDB 用 B+ 树做索引,正是同时看重"树矮省 I/O"和"叶子链表利于范围扫描/排序/分页"这两点。
