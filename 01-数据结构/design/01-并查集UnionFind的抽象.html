并查集 Union-Find 的抽象

一种专门用来维护"分组关系"的数据结构——不关心组内元素的顺序，只关心"谁和谁是一伙的"。

要解决的问题

生活中有大量"动态分组"的场景：班级里同学结成朋友圈，公司里部门合并，社交网络里判断两个人是否间接认识。 这些问题的共同点是：一开始每个元素各自成组，随着关系不断建立，组会不断合并，我们只需要能随时回答两个问题：

  • Find（查找）：某个元素属于哪个组？（用来判断两个元素是否同组）
  • Union（合并）：把两个元素所在的组合并成一个组。

并查集（Union-Find，也叫 Disjoint Set，不相交集合）就是为这两个操作量身定做的数据结构。 它不是为了"存储"数据本身，而是为了高效维护元素之间的等价关系。

类比：把每个人想象成一个部落成员，部落有一个"酋长"（代表整个部落）。判断两个人是否同部落，只需要看他们的酋长是不是同一个人；两个部落联姻合并，就让其中一个酋长认另一个酋长做"上级"。

朴素实现：树森林

并查集最经典的实现方式是用树来表示集合，用森林（多棵树）表示多个集合：

  • 每个元素对应一个节点，节点里存一个 parent 指针（用数组下标模拟）指向它的父节点。
  • 根节点的 parent 指向自己，根节点就代表整个集合的"身份标识"。
  • 初始化时，每个元素自成一组，也就是每个节点的 parent 都指向自己（n 棵只有一个节点的树）。
UnionFind.java朴素实现（未优化）
public class UnionFind {
    private final int[] parent;

    public UnionFind(int n) {
        parent = new int[n];
        // 初始时每个元素是自己的根，即 n 个独立集合
        for (int i = 0; i < n; i++) parent[i] = i;
    }

    // 查找 x 所在集合的根节点（沿着 parent 指针一路往上走）
    public int find(int x) {
        while (parent[x] != x) {
            x = parent[x];
        }
        return x;
    }

    // 合并 x、y 所在的两个集合：把 x 的根挂到 y 的根下面
    public void union(int x, int y) {
        int rootX = find(x);
        int rootY = find(y);
        if (rootX != rootY) parent[rootX] = rootY;
    }

    // 判断两个元素是否属于同一集合
    public boolean connected(int x, int y) {
        return find(x) == find(y);
    }
}

这个朴素版本已经能正确工作，但有一个致命缺陷：合并时完全不考虑树的形状。 如果一直把新元素的根接到某个固定元素下面，树会退化成一条链，find 操作就会退化成 O(n)—— 这正是下一篇要解决的问题。

常见误区：不要把并查集和"图的连通分量"搞混。并查集本身不存储边的具体连接方式，它只回答"是否连通"这个粗粒度问题， 并且是动态的——可以随时插入新的合并操作。而如果只是静态地求一次连通分量，用 DFS/BFS 遍历图也能做到，不一定非要并查集。 并查集的价值在于"边一条一条到来、需要频繁判断连通性"的动态场景，比如 Kruskal 算法（后面 Ⅳ Tab 已经用过）判断加入一条边是否会成环。

核心价值总结

  • 专注于分组关系：并查集不存储元素的顺序或具体值，只维护"谁跟谁一伙"这一件事，接口极简（find + union）。
  • 树森林是天然的表示方式：根节点即集合标识，合并操作只需改一个指针，天然支持动态合并。
  • 朴素实现有隐患：树可能退化成链，下一篇的按秩合并 + 路径压缩会把它优化到接近 O(1)。
