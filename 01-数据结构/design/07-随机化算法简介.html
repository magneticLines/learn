随机化算法简介

进阶内容。故意引入随机性，反而能让算法在最坏情况下也表现稳定——这是一种"用不确定性对抗不确定性"的巧妙思路。

为什么要引入随机性

很多算法的最坏情况性能，取决于输入数据恰好"踩中"了某种不利的排列方式。随机化算法的核心思路是： 与其祈祷输入数据不要那么"倒霉"，不如主动在算法内部引入随机决策，把"最坏情况"被触发的概率分摊到所有可能的输入上。

类比：抓阄决定谁先发言，比按照座位顺序发言更公平——按座位顺序发言时，坐在最后一排的人可能永远抢不到好话题（这是一种"结构性劣势"， 对应算法里"某类输入总会触发最坏情况"）；抓阄让每个人被抽到"倒霉顺位"的概率是均等的，消除了这种结构性劣势。

经典案例：快速排序随机选取枢轴

回顾 Ⅳ Tab 讲过的快速排序：如果每次都固定选择数组的第一个（或最后一个）元素作为枢轴， 面对一个已经有序（或接近有序）的数组，每次划分都会把数组分成"0 个元素"和"n-1 个元素"两部分， 退化成 O(n²)——而"输入恰好有序"在实际业务数据中并不罕见（比如按时间戳排好序的日志、按 ID 递增插入的记录）。

QuickSort.java随机选取枢轴，规避最坏情况
private static int partition(int[] arr, int lo, int hi) {
    // 随机选一个位置的元素当枢轴，再和末尾交换——
    // 这样"输入本身有序"这种结构特征就不会被算法利用，最坏情况的触发变成了小概率事件
    int randomIndex = lo + new Random().nextInt(hi - lo + 1);
    swap(arr, randomIndex, hi);

    int pivot = arr[hi];
    int i = lo - 1;
    for (int j = lo; j < hi; j++) {
        if (arr[j] < pivot) {
            i++;
            swap(arr, i, j);
        }
    }
    swap(arr, i + 1, hi);
    return i + 1;
}

随机化之后，快速排序的期望时间复杂度是 O(n log n)，而且不再依赖输入数据的具体排列—— "最坏情况 O(n²)"依然理论上存在，但触发它需要极端不走运的随机数序列，概率随 n 增大迅速趋近于 0。

常见误区：随机化只是把"最坏情况的复杂度"变成了"期望复杂度"，并没有从理论上消除最坏情况的可能性。 面试中如果被问"随机化快排的最坏时间复杂度是多少"，正确答案仍然是 O(n²)，只是这种情况发生的概率极小，不代表它被根除了。

蒙特卡洛算法 vs 拉斯维加斯算法

随机化算法按照"随机性影响的是结果还是时间"，可以分成两大类：

类型特点例子
蒙特卡洛算法（Monte Carlo） 运行时间是确定的（或有明确上界），但结果可能是错的，正确率可以通过多次重复运行来提高 Miller-Rabin 素数测试：多次测试后判断"合数"几乎百分百正确，判断"素数"存在极小概率误判
拉斯维加斯算法（Las Vegas） 结果保证一定正确，但运行时间是随机变量，只能保证期望时间复杂度 随机化快速排序：排序结果永远正确，只是运行时间会因随机数不同而波动

简单的记忆方法：蒙特卡洛是"时间稳、结果可能不稳"，拉斯维加斯是"结果稳、时间可能不稳"—— 这正好对应"蒙特卡洛赌场靠概率赚钱（可能会输）"和"拉斯维加斯的表演一定会准时开场（但排队时间不确定）"这两个通俗记忆锚点。

核心价值总结

  • 随机化的目的：消除算法性能对特定输入排列的依赖，把"最坏情况"变成"小概率事件"。
  • 随机化快排是最直观的例子：用随机选取枢轴替代固定选取，规避了"输入已排序导致退化成 O(n²)"的结构性弱点。
  • 蒙特卡洛 vs 拉斯维加斯：前者拿结果的确定性换时间的确定性，后者反过来，理解这个对偶关系有助于分类记忆各种随机化算法。
