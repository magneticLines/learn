按秩合并与路径压缩

两个各自独立、组合起来威力倍增的优化手段，把并查集的操作效率从"可能退化成 O(n)"提升到"均摊接近 O(1)"。

问题回顾：树为什么会退化

上一篇的朴素实现里，union(x, y) 总是把 x 的根接到 y 的根下面，完全不管两棵树谁大谁小。 如果不断执行 union(1,2)、union(2,3)、union(3,4)...，树会变成一条长链，find 操作要一路爬到根，退化成 O(n)。

优化一：按秩合并（Union by Rank）

思路很直接：合并时，把矮的树接到高的树下面，而不是随意指定方向。这样树的高度增长会慢很多—— 只有当两棵"一样高"的树合并时，高度才会 +1，否则高度不变。

类比：两个小部落合并，让人数少（或者说"辈分浅"）的部落酋长认人数多的酋长做上级，而不是随便谁认谁——这样整个部落的"层级链条"不会无谓变长。

这里的"秩"（rank）通常记录的是树高的上界（严格算法分析中不是精确树高，因为路径压缩会改变高度但不更新 rank，所以叫"秩"而非"高度"）。

UnionFind.java按秩合并
int[] parent, rank;

public void union(int x, int y) {
    int rootX = find(x), rootY = find(y);
    if (rootX == rootY) return;

    // 秩小的树接到秩大的树下面，谁当"儿子"由树的规模决定
    if (rank[rootX] < rank[rootY]) {
        parent[rootX] = rootY;
    } else if (rank[rootX] > rank[rootY]) {
        parent[rootY] = rootX;
    } else {
        // 秩相同，谁当根都行，但树高会 +1
        parent[rootY] = rootX;
        rank[rootX]++;
    }
}

优化二：路径压缩（Path Compression）

按秩合并控制了树"长高"的速度，但没有解决已经存在的长路径问题。路径压缩的思路是： 既然 find 已经把从 x 到根的整条路径都走了一遍，不如顺手把路径上所有节点直接指向根，这样下次查询这些节点就是 O(1)。

UnionFind.java路径压缩（递归写法）
public int find(int x) {
    if (parent[x] != x) {
        // 递归找到根之后，把 x 直接挂到根上，压平路径
        parent[x] = find(parent[x]);
    }
    return parent[x];
}

效果非常直观：一条长链在做过一次 find 之后，链上所有节点都会变成直接挂在根下面的"扁平结构"， 后续对这些节点的查询全部变成 O(1)。

两者结合：均摊复杂度 O(α(n))

单独使用路径压缩，或单独使用按秩合并，都能把复杂度改善到 O(log n)。但两者结合使用之后， 经过严格的摊还分析（下一个 Tab 会讲摊还分析的通用方法），n 次操作的总时间复杂度是 O(n · α(n))， 其中 α(n) 是反阿克曼函数（inverse Ackermann function），增长极其缓慢—— 对于宇宙中原子数量级的 n，α(n) 也不超过 5。因此实践中普遍认为并查集的每次操作是均摊 O(1)。

面试高频："并查集的时间复杂度是多少？" 标准答案是 O(α(n))，接近但不严格等于 O(1)。 能说出"反阿克曼函数"、"按秩合并 + 路径压缩缺一不可才能达到这个复杂度"，是加分项。
常见误区：只写路径压缩、不写按秩合并（或者反过来）也能让代码正确运行，很多人因此偷懒只写一个。 虽然单独一个优化在大多数测试用例下也表现不错，但为了拿到理论上最优的均摊复杂度保证，两者都应该实现——面试和实际工程中都建议两个都写。

核心价值总结

  • 按秩合并：从"合并方向"上控制树高增长，避免树退化成链。
  • 路径压缩：从"查询过程"顺手优化树结构，越查树越扁平。
  • 两者互补：合并使用后均摊复杂度趋近 O(1)，是并查集能被广泛应用（如 Kruskal 算法判环）的关键。
