回溯法

本质是"试探 + 撤销"的深度优先搜索——走一步，发现不行就退回来换一条路，直到穷举完所有可能。

回溯法的本质

回溯法（Backtracking）是一种系统性地搜索问题所有（或部分）解的方法，核心动作只有两个：

  • 试探（choose）：做出一个选择，进入下一层决策。
  • 撤销（unchoose）：如果这条路走不通（或者已经穷举完这条路径下的可能性），撤销刚才的选择，回到上一层，尝试别的选择。

从算法结构上看，回溯法就是深度优先搜索（DFS）在"解空间树"上的应用——每个节点代表一个部分解， 从根节点出发不断往下试探，遇到死路就回溯到父节点换个分支继续。

类比：走迷宫时用"一直往右手摸墙走"的策略——走到死胡同就退回上一个岔路口，换一个没走过的方向继续走。 这种"往前走 + 碰壁就退"的过程，就是回溯法最直观的写照。

经典例子一：N 皇后问题

在 N×N 棋盘上放 N 个皇后，要求任意两个皇后都不能互相攻击（不同行、不同列、不同对角线）。

NQueens.java逐行试探 + 冲突就撤销
public class NQueens {
    int n;
    int[] queens; // queens[row] = col，表示第 row 行的皇后放在第几列
    int count = 0;

    public int solve(int n) {
        this.n = n;
        queens = new int[n];
        backtrack(0); // 从第 0 行开始试探
        return count;
    }

    private void backtrack(int row) {
        if (row == n) { count++; return; } // 所有行都放完，找到一个解

        for (int col = 0; col < n; col++) {
            if (isValid(row, col)) {
                queens[row] = col;       // 试探：把皇后放在 (row, col)
                backtrack(row + 1);   // 递归试探下一行
                // 不需要显式"撤销"，因为 queens[row] 在下一次循环会被覆盖
            }
        }
    }

    // 剪枝：检查 (row, col) 是否会和前面已放置的皇后冲突
    private boolean isValid(int row, int col) {
        for (int r = 0; r < row; r++) {
            int c = queens[r];
            if (c == col || Math.abs(c - col) == row - r) return false; // 同列或同对角线
        }
        return true;
    }
}

经典例子二：子集生成问题

给定一个集合，生成它的所有子集（幂集）。每个元素只有"选"或"不选"两种状态，天然适合用回溯枚举：

Subsets.java每个元素选或不选，两个分支
public void backtrack(int[] nums, int index, List<Integer> path, List<List<Integer>> result) {
    if (index == nums.length) {
        result.add(new ArrayList<>(path)); // 到达终点，记录当前路径（必须拷贝一份）
        return;
    }
    // 分支一：不选 nums[index]
    backtrack(nums, index + 1, path, result);

    // 分支二：选 nums[index]
    path.add(nums[index]);       // 试探
    backtrack(nums, index + 1, path, result);
    path.remove(path.size() - 1); // 撤销：把刚加进去的元素移除，恢复现场
}

这个例子里"撤销"动作非常明显——path.remove(...) 就是把共享的 path 列表恢复到试探之前的状态， 这样兄弟分支才能在一个干净的现场上继续试探。这是回溯法区别于普通 DFS 的标志性写法：用一份可变的共享状态贯穿整个递归，进入前修改，返回前复原。

剪枝：让回溯不再"傻找"

回溯法最朴素的版本是穷举所有可能（指数级复杂度），但大多数问题可以通过剪枝（Pruning）提前排除明显不可能产生解的分支， 大幅减少实际搜索的节点数。N 皇后例子中的 isValid 就是剪枝——一旦发现当前位置会冲突， 直接跳过这个分支，根本不进入递归，避免了后续行的无效试探。

面试高频："回溯法的时间复杂度怎么算？" 通常是指数级（如 N 皇后是 O(N!) 量级）， 但优秀的剪枝可以大幅减少实际运行时间。面试官关注的往往不是"能不能用回溯解决"（很容易看出来）， 而是"剪枝逻辑写得好不好"——能不能尽早排除不可能的分支，直接决定了代码在大输入下是否会超时。
常见误区：忘记"撤销"这一步，是回溯法代码里最常见的 bug。比如子集生成的例子里如果漏写 path.remove(...)，那么 path 会一直累积元素，后面所有分支得到的子集都是错的。 记住口诀：做选择、递归、撤销选择，三步必须成对出现。

核心价值总结

  • 回溯的本质：在解空间树上做深度优先搜索，靠"试探 + 撤销"这对操作探索所有可能的路径。
  • 共享状态是关键：用一份可变状态贯穿递归全程，进入分支前修改、退出分支前复原，这是回溯法有别于普通递归的标志。
  • 剪枝决定实际效率：朴素回溯是指数级复杂度，好的剪枝能在不改变正确性的前提下大幅减少实际搜索量。

与其他知识点的联系

  • 动态规划：DP 能用的前提是子问题存在重叠，一旦子问题各不相同（如 N 皇后每一步的棋盘状态几乎都不同），记忆化就没有意义，只能靠回溯穷举——这也是为什么"排列组合类"问题多用回溯、"最优值类"问题多用 DP 的经验规律。
  • 树/图的深度优先搜索（DFS）：回溯法就是 DFS 在隐式的"解空间树"上的应用——树的遍历里"进入子节点前处理、退出子节点后清理"的模式，和回溯法"做选择、递归、撤销选择"三步完全对应，只是解空间树的节点不是提前建好的，而是递归过程中动态生成的。
  • 分治法：分治和回溯都会把问题递归拆解，但分治的子问题互不影响、可以直接合并结果；回溯的多个分支共享同一份可变状态，必须靠"撤销"操作让分支之间互不干扰——这是两者在状态管理上的本质区别。
  • Java 集合框架源码：子集生成例子里 result.add(new ArrayList<>(path)) 必须拷贝一份，否则所有子集都会指向同一个 path 引用，后续的 remove 会把已保存的"子集"也污染掉——这是"浅拷贝 vs 深拷贝"在实际代码里最容易踩的坑之一。
  • 面试高频考点：LeetCode 里"全排列""组合总和""括号生成"等一大类题目都是回溯法的变形，面试考察的核心永远是"状态怎么设计""剪枝条件怎么写""该维护什么样的共享状态"，而不是死记某一道题的代码。
