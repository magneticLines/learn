证明方法(归纳法 / 反证法)

写代码时"我觉得这个算法是对的""我觉得它是 O(n log n)"这种直觉,能不能变成一个站得住脚的证明?数学归纳法和反证法就是把直觉变成证明的工具。

📖 来源:《数据结构与算法分析:Java语言描述》 第 1 章「引论」

一、为什么写代码也需要"证明"

很多人觉得归纳法、反证法是数学课的内容,和写代码没关系。但实际上,你在调试、写循环、设计递归算法时,一直在隐式地做证明——只是没有意识到,也没有做严谨。比如:

  • "这个循环执行完之后,数组一定是有序的" —— 这是一个需要证明的循环不变量命题。
  • "这个递归函数对任意输入 n 都能正确返回结果" —— 这本质上是一个用归纳法证明的命题。
  • "这个算法不可能是 O(n) 的" —— 这往往需要用反证法:假设存在 O(n) 算法,推出矛盾。

把这些"隐式的直觉"训练成"显式的证明能力",能帮你在设计算法时更早发现漏洞,而不是等写完代码跑出错误结果才回头排查。

二、数学归纳法:证明递归算法的利器

归纳法的结构和递归算法的结构几乎是镜像对应的,这不是巧合——递归本身就是"归纳定义"在代码层面的体现:

归纳法两步:
1. 基础步(Base Case):证明命题对最小的 n(如 n=0 或 n=1)成立。
2. 归纳步(Inductive Step):假设命题对所有小于 n 的情形成立(归纳假设),证明由此可以推出命题对 n 也成立。

对比递归的设计法则会发现:归纳法的"基础步"对应递归的"基准情形";归纳法的"假设对更小的情形成立"对应递归里"你必须相信"子问题的结果是对的。这就是为什么书里把两者放在一起讲——递归是归纳法在算法设计上的应用,归纳法是证明递归算法正确性的天然工具。

案例:用归纳法证明阶乘递归的复杂度

回顾上一篇的 factorial(n),设 T(n) 为它的运行时间(以基本操作次数计):

递推关系
T(0) = c1               // 基准情形,常数时间
T(n) = T(n-1) + c2       // 递归情形:一次乘法/比较 + 子问题

猜想:T(n) = c2·n + c1,即 O(n)。用归纳法证明:

  • 基础步:n=0 时,T(0) = c1 = c2·0 + c1,成立。
  • 归纳步:假设 T(n-1) = c2·(n-1) + c1 成立(归纳假设),代入递推式: T(n) = T(n-1) + c2 = [c2·(n-1) + c1] + c2 = c2·n + c1,与猜想一致。

两步都成立,由归纳法原理,对所有 n ≥ 0,T(n) = c2·n + c1 成立,即该算法是 O(n)。这个套路(猜结论 → 验证基础步 → 用归纳假设验证归纳步)是分析递归算法复杂度最标准的方法,后面分析归并排序、快速排序的递归式时会反复用到。

三、反证法:证明"不可能更好"

反证法的逻辑是:假设待证命题的反面成立,通过逻辑推导得出一个矛盾(与已知事实、假设条件冲突),从而说明反面不成立,原命题成立。它在算法分析里最常见的用途是证明下界——"任何算法解决这个问题都至少需要多少时间",这类结论正面很难直接构造,反证法往往是唯一入口。

直觉类比:反证法就像侦探破案——不是直接找到"凶手是谁"的正面证据,而是假设"某人是凶手",看这个假设是否会和已知的时间线、证据产生矛盾。如果矛盾,就排除这个假设。

四、和"循环不变量""算法终止性"的联系

归纳法在实际编码中最常见的落地场景是证明循环不变量(loop invariant)——即"某个条件在循环的每一次迭代前后都保持成立"。证明思路和归纳法一模一样:证明循环开始前不变量成立(基础步),证明每次迭代不会破坏这个不变量(归纳步),从而得出循环结束时不变量仍然成立,由此推出算法的正确性。这也是证明循环一定会终止的常用手段:找到一个随迭代严格递减且有下界的量(比如 剩余待处理元素个数),配合归纳法说明它最终会到达终止条件。

小结

  • 数学归纳法 = 基础步 + 归纳步,天然对应递归的"基准情形"和"相信子问题结果"。
  • 分析递归算法复杂度的标准套路:先猜出运行时间的表达式,再用归纳法验证。
  • 反证法用于证明"不存在更好的算法"这类下界结论:假设反面成立,推出矛盾。
  • 写代码时的"循环不变量"证明、"循环一定终止"的论证,本质都是归纳法的应用场景。
