分治递归的运行时间分析

为什么归并排序是 O(n log n),而不是 O(n) 或 O(n²)?答案藏在一个递推式里。

📖 来源:《数据结构与算法分析:Java语言描述》 第 1~2 章「引论/算法分析」

一、递推式从哪里来

一个分治算法通常长这样:把规模为 n 的问题拆成 a 个规模为 n/b 的子问题,分别递归求解,再花 f(n) 的代价把子问题的结果合并起来。这直接翻译成一个递推式:

T(n) = a·T(n/b) + f(n)
· a:拆出多少个子问题
· n/b:每个子问题的规模
· f(n):合并子问题结果所花的代价(不算递归本身)

例如归并排序:把数组从中间切成两半(a=2, b=2),分别递归排序,再用 O(n) 的代价合并两个有序数组,所以 T(n) = 2T(n/2) + O(n)。二分查找:每次只往一边递归(a=1, b=2),合并代价是 O(1)(只做一次比较),所以 T(n) = T(n/2) + O(1)。

二、主定理的直觉(不做严格证明)

主定理(Master Theorem)给出了这类递推式的通解,但比公式更重要的是它背后的直觉:把 a·T(n/b) 和 f(n) 看成两个"竞争对手",谁的增长更快,谁就主导最终的复杂度。关键是比较 n^(log_b a)(代表"递归树总共展开出多少个叶子节点,每个叶子做常数工作"的量级)和 f(n)(合并代价)谁更大:

情况直觉结论
子问题主导
(f(n) 增长比 n^(log_b a) 慢)		合并代价小到可以忽略,时间几乎全花在"递归到底、层层拆分"本身上——就像分蛋糕的动作远比每次分完后的整理耗时T(n) = O(n^(log_b a))
平衡
(f(n) 与 n^(log_b a) 同阶)		每一层花的合并代价和子问题展开的代价旗鼓相当,总代价 = 单层代价 × 层数T(n) = O(n^(log_b a) · log n)
合并主导
(f(n) 增长比 n^(log_b a) 快)		合并这一步本身就很重,子问题分得再细也无法抵消合并的开销,顶层的合并代价基本决定了总时间T(n) = O(f(n))

换一个更形象的类比:把递归想象成一棵树,每一层的总合并代价是 f(n) 按层缩放后的结果。三种情况对应「代价集中在叶子层」「代价在每层均匀分布」「代价集中在根部(顶层)」。

三、用归并排序验证

T(n) = 2T(n/2) + O(n),这里 a=2, b=2,所以 n^(log_b a) = n^(log2 2) = n^1 = n。而 f(n) = O(n),恰好和 n^(log_b a) 同阶——落入「平衡」情况,所以 T(n) = O(n log n)。

不用主定理,直接展开递归树也能验证这个结论:

归并排序递归树展开
// 第 0 层:1 个问题,规模 n,     合并代价 n
// 第 1 层:2 个问题,规模 n/2,   每个合并代价 n/2,总计 n
// 第 2 层:4 个问题,规模 n/4,   每个合并代价 n/4,总计 n
// ...
// 第 k 层:2^k 个问题,规模 n/2^k,每层合并代价总计仍是 n
// 共有 log2(n) 层(降到规模为 1 才停止递归)
// 总代价 = n(每层代价) × log2(n)(层数) = O(n log n)

四、用二分查找验证

T(n) = T(n/2) + O(1),这里 a=1, b=2,n^(log_b a) = n^(log2 1) = n^0 = 1(常数)。而 f(n) = O(1),同样是常数——两者同阶,落入「平衡」情况,所以 T(n) = O(1 · log n) = O(log n),正好对应本组第一篇讲的「对数级算法」的结论。

二分查找递归树展开
// 每一层只有 1 个子问题(不是 2 个!),规模减半,合并代价 O(1)
// 第 0 层:1 个问题,代价 1
// 第 1 层:1 个问题,代价 1
// ...
// 第 k 层:1 个问题,代价 1,直到规模缩小到 1
// 共有 log2(n) 层,每层代价都是常数
// 总代价 = O(1) × log2(n) = O(log n)

五、建立联系

这套分析方法把前面几篇的知识串了起来:「对数级算法为什么快」里二分查找的 T(n)=T(n/2)+O(1)、「最大子序列和」里分治解法的 T(n)=2T(n/2)+O(n),都可以用这里的递推式框架统一分析,而不需要每次都单独凑一个证明。以后遇到任何"递归拆分+合并"结构的算法(比如后面会学的快速排序、树的递归遍历),都可以先写出 T(n)=aT(n/b)+f(n) 的形式,再套用这里的直觉判断量级。

易错点(面试高频):把 a(子问题个数)和 b(规模缩小的倍数)搞混,或者忘记子问题个数 a 对结果的影响。例如三分归并(a=3, b=3)和二分归并(a=2, b=2)复杂度都是 O(n log n),因为 log_b a 恒为 1(a=b 时);但如果拆分方式变成 a=4, b=2(比如某些矩阵乘法算法),n^(log_2 4) = n²,会比 f(n)=O(n) 大得多,复杂度就退化成 O(n²) 而不再是 O(n log n)——"拆成几份"和"每份多小"必须一起看,不能只看其中一个。

小结

  • 分治算法的运行时间可以统一写成 T(n) = a·T(n/b) + f(n):a 个子问题、每个规模 n/b、合并代价 f(n)。
  • 主定理的直觉是比较 n^(log_b a)(子问题层面的总代价)与 f(n)(合并代价)谁主导:子问题主导、两者平衡、合并主导,分别对应三种结论。
  • 归并排序 T(n)=2T(n/2)+O(n) 属于"平衡"情况,结果是 O(n log n)。
  • 二分查找 T(n)=T(n/2)+O(1) 也属于"平衡"情况(因为 a=1 时 n^(log_b a) 恒为常数),结果是 O(log n)。
  • 这是"算法分析"这个 Tab 的收官篇:从常数级差异到对数级、从暴力到分治,最终都归结到"如何写出并求解递推式"这一套通用工具。
