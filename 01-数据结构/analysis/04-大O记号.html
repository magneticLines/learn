大 O / Ω / Θ / o 记号

四种记号描述的是增长趋势的四个不同角度:上界、下界、紧确界、严格上界。混用它们是复杂度分析里最常见的口误来源。

📖 来源:《数据结构与算法分析:Java语言描述》 第 2 章「算法分析」

一、为什么一个"大 O"不够用

日常聊算法时,大家习惯统一说"这个算法是 O(n log n)"。但严谨地说,大 O 只描述增长速度不超过某个函数——它是一个"上界"描述,并不保证这个界有多"紧"。比如说"这个算法最多花 O(n²) 时间"永远是对的(哪怕它实际上是 O(n)),因为 O(n²) 确实是一个成立的上界,只是不精确。为了能表达不同精度、不同方向的结论,需要四种记号分别刻画:

  • 上界:增长速度不会超过某个函数(最坏情况的保证)。
  • 下界:增长速度不会低于某个函数(最好情况/理论极限的保证)。
  • 紧确界:增长速度与某个函数同阶(既是上界又是下界,最精确的描述)。
  • 严格上界:增长速度严格慢于某个函数(比上界更强的一种"远小于"关系)。

二、四个记号的数学定义与对比

记号含义数学定义(存在正常数 c、n₀,当 n ≥ n₀ 时)使用场景
O(g(n))上界0 ≤ T(n) ≤ c·g(n)描述最坏情况下的运行时间,日常最常用
Ω(g(n))下界0 ≤ c·g(n) ≤ T(n)证明"任何算法都至少需要这么多时间"(理论下界),常配合反证法
Θ(g(n))紧确界T(n) = O(g(n)) 且 T(n) = Ω(g(n))最精确的描述,说明算法的真实增长阶
o(g(n))严格上界对任意常数 c > 0,存在 n₀ 使得 n ≥ n₀ 时 T(n) < c·g(n)表达"远小于",比如说明某算法比另一个"确实快了一个量级"而非"顶多一样快"
类比记忆:把 O、Ω、Θ 对应到数字比较上——O 像 ≤,Ω 像 ≥,Θ 像 =(在忽略常数因子的意义下),o 像严格的 <。这套对应关系能帮你快速判断该用哪个符号表达结论。

三、举例:同一个算法用四种记号描述

以顺序查找为例,在长度为 n 的数组里查找一个元素:

  • 最坏情况(元素不存在或在最后):运行时间是 Θ(n),同时也可以说是 O(n)、O(n²)(都是成立但后者不精确的上界)。
  • 最好情况(元素恰好在第一个位置):运行时间是 Θ(1)。
  • 说"顺序查找是 Ω(1)"是对的但没什么信息量——几乎任何算法都至少需要常数时间。
  • 如果一个新算法的运行时间是 o(n)(比如 O(log n)),就说明它严格快于顺序查找这个量级,不仅仅是"不比它差"。

四、常见误区

误区 1 · 把"大 O"当成"确切运行时间"。大 O 描述的是增长趋势的上界,不是具体跑了多少毫秒。两个都是 O(n) 的算法,实际运行时间可能相差几十倍(常数因子不同),大 O 记号刻意忽略了这些常数。
误区 2 · 混淆最坏情况与平均情况。"快速排序是 O(n log n)"这个说法不严谨——快速排序的最坏情况是 O(n²)(如输入已经有序且选取最后一个元素作为基准),平均情况才是 O(n log n)。分析时必须明确说的是最坏、最好还是平均情况,不能笼统地只说"是 O(...)"。
误区 3 · 认为 O(g(n)) 就是"紧的"。说一个 Θ(n) 的算法是 O(n³) 并不算错,但没有信息量。日常交流中大家用 O 时默认期望它是紧确界,但严谨表达("这个算法的复杂度恰好是 xxx 阶")应该用 Θ。

五、和后面知识点的联系

下一篇会讲运行时间的具体计算法则(顺序、循环、嵌套、分支、递归怎么算),那里推导出来的表达式,最终都需要用这四种记号中的合适一种去归纳陈述结论。后面学各种数据结构(数组、链表、树、堆)时,插入/删除/查找的复杂度对比,也都是用这套记号体系来精确表达"谁比谁快在哪个方面"。

小结

  • O 上界、Ω 下界、Θ 紧确界、o 严格上界,四者从不同角度刻画增长趋势。
  • 日常说"复杂度是 O(...)"其实隐含期望它是紧确界,严谨表达应优先用 Θ。
  • 大 O 不是确切运行时间,它忽略常数因子,只反映增长阶。
  • 分析复杂度必须说明是最坏、最好还是平均情况,三者可能完全不同(如快速排序)。
