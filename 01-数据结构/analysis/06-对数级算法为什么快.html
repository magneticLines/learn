对数级算法为什么快(二分查找 / 快速幂)

O(log n) 是介于「常数」和「线性」之间最容易被低估的复杂度——它快得不成比例。

📖 来源:《数据结构与算法分析:Java语言描述》 第 1~2 章「引论/算法分析」

一、log n 到底有多小

直觉上「查 10 亿条数据」听起来很吓人,但如果每一步能排除掉一半的候选范围,那么需要的步数是 log2(10^9) ≈ 30。也就是说,面对 10 亿规模的问题,对数级算法只需要 30 步左右就能给出答案。这就是本组前一篇讲的「运行时间计算法则」里,O(log n) 增长得极其缓慢的直观体现:

规律很直接:n 每翻倍,log2 n 只增加 1。这解释了为什么二分查找、平衡树、快速幂这类「每步砍掉一半」的算法,在海量数据下依然表现得像「瞬间完成」。

二、二分查找:每次排除一半

前提是数据已排序。核心思想:每次看中间元素,如果目标更小就丢掉右半边,更大就丢掉左半边——问题规模每次减半,所以是 T(n) = T(n/2) + O(1),对应 O(log n)(下一篇会讲这类递推式怎么解)。

Java · 二分查找
static int binarySearch(int[] a, int target) {
    int lo = 0, hi = a.length - 1;
    while (lo <= hi) { // 等号不能丢,否则漏掉最后一个候选
        int mid = lo + (hi - lo) / 2; // 不写 (lo+hi)/2,防止大数组时溢出
        if (a[mid] == target) return mid;
        else if (a[mid] < target) lo = mid + 1; // mid 已排除,下界要 +1
        else hi = mid - 1;              // mid 已排除,上界要 -1
    }
    return -1; // 没找到
}
二分查找最常见的三个坑(面试高频):
1. mid = (lo + hi) / 2 —— 当 lo、hi 都是接近 Integer.MAX_VALUE 的大数时会整型溢出,应写成 lo + (hi - lo) / 2。
2. 循环条件写成 lo < hi 而不是 lo <= hi —— 会漏掉「候选区间只剩一个元素」的情况。
3. 命中/未命中分支里忘记 mid ± 1,直接写 lo = mid 或 hi = mid —— mid 已经比较过,不排除它会导致死循环。

三、快速幂:分治 + 平方

朴素计算 x^n 需要循环乘 n 次,是 O(n)。但注意到一个性质:x^n = (x^(n/2))^2(n 为偶数),x^n = x · (x^((n-1)/2))^2(n 为奇数)。每递归一层,指数减半,所以是 O(log n)——这是「分治」思想的最简单体现:把大问题拆成一个规模减半的子问题,而不是拆成 n 个子问题。

Java · 快速幂
static long quickPow(long x, int n) {
    if (n == 0) return 1;
    long half = quickPow(x, n / 2); // 只递归一次,不是两次——这才是 O(log n) 而不是 O(n) 的关键
    long half2 = half * half;
    return (n % 2 == 0) ? half2 : x * half2;
}
易错点:如果写成 quickPow(x, n/2) * quickPow(x, n/2)(递归两次而不是复用结果),看起来一样,实际上递归树会变成 T(n) = 2T(n/2) + O(1),复杂度退化回 O(n)。「减半」和「复用同一个子结果」两者缺一不可。

四、建立联系

对数级复杂度会反复出现:平衡二叉搜索树(如红黑树)的查找/插入是 O(log n),因为树高是 O(log n);归并排序、快速排序的递归深度也是 O(log n);Java 里 Collections.binarySearch、TreeMap 的底层都在利用这个性质。理解「每步排除一半」的思想,是后面理解二叉搜索树、堆、分治算法的地基。

小结

  • log2 n 增长极慢,n 翻倍,它只 +1——这是对数级算法「以慢打快」的根本原因。
  • 二分查找:每次排除一半候选,写法上要注意 mid 计算防溢出、循环边界、排除 mid 后的 ±1。
  • 快速幂:递归时只求一次子问题再平方复用,这才是真正的 O(log n)。
  • 这类「减半」思想是后面二叉搜索树、堆、分治排序的共同基础。
