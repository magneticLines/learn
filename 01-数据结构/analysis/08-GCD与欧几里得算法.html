最大公约数 GCD 与欧几里得算法

两千多年前的算法,今天依然是「对数级算法」的经典范例。

📖 来源:《数据结构与算法分析:Java语言描述》 第 1~2 章「引论/算法分析」

一、问题与朴素解法

求两个正整数 a、b 的最大公约数(GCD)。最朴素的想法是从 min(a, b) 开始往下枚举,找第一个能同时整除 a 和 b 的数——但这在 a、b 差距悬殊(比如一个是质数、一个很大)时效率很低,最坏是 O(min(a, b))。

二、欧几里得算法的原理

核心公式只有一句话:

gcd(a, b) = gcd(b, a mod b),直到 b = 0 时,a 就是答案。

为什么这个公式成立?关键洞察是:a 和 b 的公约数,一定也是 b 和 (a mod b) 的公约数。因为 a = k·b + r(其中 r = a mod b),所以任何能同时整除 a 和 b 的数,必然也能整除 r = a - k·b;反过来,能同时整除 b 和 r 的数,也必然能整除 a = k·b + r。两个方向都成立,所以 (a, b) 和 (b, a mod b) 的公约数集合完全相同,最大公约数自然也相同。这一步把问题规模从「两个数都可能很大」变成「至少有一个数变小了」。

三、递归实现

Java · 欧几里得算法
static long gcd(long a, long b) {
    // b == 0 时,a 自身就是答案——这是递归的终止条件
    return (b == 0) ? a : gcd(b, a % b);
}

用一个具体例子看它的执行轨迹,gcd(1071, 462):

步骤调用计算
1gcd(1071, 462)1071 mod 462 = 147 → gcd(462, 147)
2gcd(462, 147)462 mod 147 = 21 → gcd(147, 21)
3gcd(147, 21)147 mod 21 = 0 → gcd(21, 0)
4gcd(21, 0)b == 0,返回 21

只用 3 步取模就从两个几百量级的数字算出了答案,直觉上比「从小到大枚举」快得多。

四、为什么它是对数级的

这正好和本组第一篇「对数级算法为什么快」的结论呼应:欧几里得算法每两步至少能让较大的数减少一半以上——可以证明 a mod b < a / 2(当 b ≤ a/2 时显然;当 b > a/2 时,a mod b = a - b < a/2)。也就是说,每递归两层,数值规模至少减半,因此总递归次数是 O(log(min(a, b)))——这比朴素枚举的 O(min(a, b)) 快了指数级的差距。这也是为什么即使 a、b 是几百位的大数(比如 RSA 加密中的应用场景),欧几里得算法依然能瞬间算出结果。

五、建立联系

  • 和「对数级算法为什么快」一样,欧几里得算法的高效也来自「每一步都让问题规模显著缩小」这一设计思想,只是这里不是精确减半,而是「至少减半」的数学证明保证。
  • GCD 是很多其他算法的基础模块:约分(化简分数)、求最小公倍数(lcm(a,b) = a*b/gcd(a,b))、密码学中的模逆元计算(扩展欧几里得算法)。
  • 递归写法简洁,但要留意:如果 a < b,第一次调用会自动交换(因为 a mod b 恒有 a mod b < b,下一次递归参数顺序会自我纠正),不需要额外写 if (a < b) swap。
易错点:不要把「a mod b」和「a 除以 b 取整」搞混——欧几里得算法用的是余数而不是商。另外初学者常犯的错误是把终止条件写成 a == b 而不是 b == 0,虽然多数情况凑巧也能跑出正确结果,但在某些输入下会提前退出导致答案错误。

小结

  • 核心公式:gcd(a, b) = gcd(b, a mod b),终止于 b == 0。
  • 原理:(a, b) 与 (b, a mod b) 的公约数集合完全相同。
  • 效率是 O(log(min(a,b)))——因为每两步至少让较大数减半,呼应了「对数级算法为什么快」。
  • 是约分、求 lcm、密码学模逆元等更复杂算法的基础构件。
