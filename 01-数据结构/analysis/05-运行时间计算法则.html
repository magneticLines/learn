运行时间的计算法则

把一段代码翻译成复杂度表达式,靠的不是灵感,而是几条可以机械套用的法则:顺序相加、循环相乘、分支取大、递归展开。

📖 来源:《数据结构与算法分析:Java语言描述》 第 2 章「算法分析」

一、五条基本法则

结构法则直觉
顺序执行各部分时间相加,取增长最快的一项先做完 A 再做 B,总耗时是两者之和,大 O 只保留量级最大的那部分
循环循环体的时间 × 循环次数循环体跑了多少次,就把单次开销乘以这个次数
嵌套循环由内向外逐层相乘内层循环相当于外层循环体的一部分,一层套一层地乘
if-else 分支取 两个分支中较大的那个,加上判断条件本身的开销最坏情况下会走到耗时更长的那条分支
递归调用列出递推关系式 T(n) = ...,再用归纳法/展开法求解递归的总时间由"子问题的时间"和"合并子问题结果的时间"共同决定
为什么"顺序相加"最后只留一项:大 O 记号本身只关心增长最快的项,比如 O(n) + O(n²) = O(n²)——当 n 足够大时,n² 那部分早就把 n 那部分甩得看不见了,这也是"顺序相加"法则背后更精确的表述:取所有部分里的最大量级。

二、逐步推导一个嵌套循环的例子

Java · 待分析代码
int countPairs(int[] a) {
    int n = a.length;
    int count = 0;
    for (int i = 0; i < n; i++) {            // 外层:n 次
        for (int j = i; j < n; j++) {        // 内层:随 i 递减的次数
            if (a[i] + a[j] == 0) count++; // O(1)
        }
    }
    return count;
}

推导步骤:

  • 第一步,看最内层:if 判断和 count++ 都是常数时间,记作 O(1)。
  • 第二步,看内层循环:当 i 固定时,j 从 i 到 n-1,循环体是 O(1),执行 n - i 次,所以内层循环耗时是 O(n - i)。
  • 第三步,看外层循环(累加):外层每次的耗时不是常数,而是 O(n-i),总耗时是对 i 从 0 到 n-1 求和: Σ(n - i), i=0..n-1 = n + (n-1) + ... + 1 = n(n+1)/2——这正是上一篇"数学工具"里的等差级数求和公式。
  • 第四步,套用大 O:n(n+1)/2 = O(n²)(忽略常数因子和低阶项)。
和"数学工具"篇的呼应:这个例子说明"循环相乘"法则在内层循环次数固定时才是简单的乘法;一旦内层循环次数随外层变量变化(如本例),就要退回到级数求和这个更基础的工具上——这正是为什么第一篇要先补数学基础。

三、if-else 分支:取较大分支

Java
if (condition) {
    fastPath();   // O(1)
} else {
    slowPath();   // O(n)
}

整体的最坏情况复杂度是 O(1) + O(n) 里取较大的,即 O(n)——不管 fastPath 多快,只要存在走到 slowPath 的可能,最坏情况分析就必须按它来算。这也是"误区:混淆最坏情况与平均情况"在实际推导中的具体体现。

四、递归调用:列递推式再求解

递归的运行时间无法直接用"相乘"或"相加"得到,必须先写出递推关系式,再用上一篇学的归纳法验证或"展开法"求解。以二分查找为例:

递推关系
T(n) = T(n/2) + O(1)   // 每次只递归处理一半,加上 O(1) 的比较
T(1) = O(1)             // 基准情形

展开:T(n) = T(n/2) + c = T(n/4) + 2c = ... = T(n/2^k) + kc。当 n/2^k = 1,即 k = log₂n 时到达基准情形,所以 T(n) = O(1) + c·log₂n = O(log n)——这正好呼应"数学工具"篇讲的对数律,以及"对数级算法为什么快"这个后续话题的核心结论。

小结

  • 顺序执行相加(取最大量级)、循环相乘、嵌套循环逐层相乘、分支取较大者、递归先列递推式再求解。
  • 当内层循环次数依赖外层变量时,不能简单相乘,要退回到级数求和。
  • 递归的复杂度分析 = 写出递推关系式 + 展开或用归纳法验证结果。
  • 这些法则最终推导出的表达式,都要用上一篇的 O / Ω / Θ / o 记号来精确陈述结论。
