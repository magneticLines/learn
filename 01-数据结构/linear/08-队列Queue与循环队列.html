队列 Queue(FIFO)与循环队列

一端进、另一端出的公平结构——为什么数组实现队列必须绕成一个圈,否则会出现"假溢出"。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、FIFO 是什么,为什么"公平"

队列(Queue)只允许在一端插入(队尾 rear)、另一端删除(队首 front)——先进先出,FIFO(First In First Out)。和栈相比,它的核心差异是"公平":谁先来谁先被处理,不存在后来的插队现象。这正是它天然适合排队、调度、缓冲类场景的原因。

生活类比:银行排队取号——先到的先办理,不会有人插队。栈则更像叠盘子——最后放上去的那个最先被拿走。理解这个类比就能秒判一个场景该用栈还是队列。

二、为什么数组实现队列必须做成"循环"

如果只用一个简单数组,front 和 rear 各自向后移动、从不回头,会出现一个典型问题:

反例 · 线性数组队列的"假溢出"
// 数组容量 5:[_,_,_,_,_]
// 依次入队 5 个元素,front=0, rear=4,数组已满(合理)
// 出队 3 个,front 移到 3,前面 0,1,2 空出来了,但没人回收
// 此时想再入队 2 个,rear 想走到 5、6 → 越界!
// 但其实数组前面明明空着 3 个位置——这就是"假溢出"

问题的根源是数组是有限的、连续的一段内存,而队列的读写位置一直在往同一方向漂移。解决办法就是让 front、rear 走到数组末尾后回到开头继续走——把数组在逻辑上首尾相接,变成一个环,这就是循环队列。

Java · 循环队列的核心:取模
class CircularQueue {
    private final int[] data;
    private int front = 0, size = 0; // size 记录元素个数,巧妙绕开 front==rear 的歧义

    void enqueue(int v) {
        int rear = (front + size) % data.length; // 取模 = 走到头就绕回开头
        data[rear] = v; size++;
    }
    int dequeue() {
        int v = data[front];
        front = (front + 1) % data.length; // 同样取模绕圈
        size--; return v;
    }
}
易错点:只用 front == rear 判断队列空/满会产生歧义——刚初始化(空)和恰好填满时,两个指针可能重合于同一位置。常见解法是:多留一个"哨兵"格子不用,或像上面那样额外维护一个 size 计数器来消除歧义。

三、联系 java.util.Queue 接口和 ArrayDeque

Queue 是接口,定义了入队 offer、出队 poll、查看队首 peek(这几个方法在空/满时返回特殊值而不是抛异常,是"安全版"api;add/remove/element 是对应的"抛异常版")。

实现底层结构特点
ArrayDeque可循环利用的数组(本质就是循环队列)官方推荐首选:无容量上限(自动扩容)、无锁、连续内存缓存友好
LinkedList双向链表也实现了 Queue,但每个节点独立对象,内存碎片化,缓存不友好
PriorityQueue二叉堆不是 FIFO,而是"优先级最高的先出",属于队列接口下的特例

ArrayDeque 的源码里能直接看到"取模绕圈"的思想——它内部维护一个数组和 head/tail 两个下标,通过 (index + 1) & (elements.length - 1) 这种位运算(数组容量总是 2 的幂,位运算比取模更快)来实现循环推进,和上面手写的 % data.length 是同一个原理的高性能版本。

四、生产者-消费者:FIFO 为什么适合这个场景

想象一个订单处理系统:多个下单请求(生产者)把订单丢进一个队列,后台任务(消费者)不断从队列头部取出订单处理。如果用栈(LIFO),最新下的订单反而最先被处理,旧订单可能长期"饿死"排在后面——这在业务上是不可接受的。FIFO 保证了处理顺序=到达顺序,这正是"排队等待服务"类场景公平性的来源。

联系 Java 并发:线程池 ThreadPoolExecutor 内部的任务队列(BlockingQueue)就是生产者-消费者模型的工业实现——提交任务的线程是生产者,工作线程是消费者,队列本身解决了"生产快、消费慢"时的缓冲问题。这是本篇 FIFO 思想和后续并发模块的重要联系点。

小结

  • 队列 FIFO,强调"公平":先到先服务。
  • 数组实现队列必须循环利用空间(取模/位运算绕圈),否则会出现前面有空位却报"满"的假溢出。
  • front==rear 判断空/满有歧义,常见解法是额外维护 size 或牺牲一个格子。
  • ArrayDeque 就是循环队列思想的工业实现,是 Java 里做队列的首选,优于 LinkedList。
  • FIFO 天然契合生产者-消费者模型,这一思路会在并发模块的阻塞队列中再次出现。
