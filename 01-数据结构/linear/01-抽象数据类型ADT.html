什么是抽象数据类型 ADT

先约定"能做什么"，再决定"怎么做"——这是所有数据结构学习的起点，也是面向接口编程的思想根源。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、它到底在解决什么问题

如果一开始就说"数组怎么实现表"、"链表怎么实现表"，你会陷入一堆实现细节，却搞不清楚"表"这个东西本质上是什么。抽象数据类型（Abstract Data Type, ADT）就是为了把这两层关切分开:

  • 逻辑层(做什么): 一组数据加上一组操作的契约——比如"表"支持插入、删除、查找、按位置访问，但不规定这些操作内部怎么跑。
  • 物理层(怎么做): 具体用数组还是链表来实现这些操作，各有不同的时间复杂度代价。

这正是 Weiss 书里反复强调的一句话:ADT 是数学意义上的抽象——它只关心"操作的行为"，不关心"操作的实现"。这一层抽象把"用什么数据结构"和"数据结构内部怎么写"解耦，使用者只需要认识契约,不需要关心背后是数组还是链表。

二、联系 Java:List 接口 vs 实现类

ADT 不是纯理论概念，Java 语言里到处都是它的落地。java.util.List 接口就是一个典型的 ADT 定义:

Java · List 是契约，不是实现
public interface List<E> extends Collection<E> {
    // 只声明"能做什么"，完全不管"怎么做"
    boolean add(E e);
    E get(int index);
    E remove(int index);
}

而 ArrayList（数组实现）和 LinkedList（链表实现）是同一个契约的两种物理实现——它们对外表现一致，但内部数据组织完全不同，导致同一个操作的时间复杂度天差地别（这正是接下来两篇要展开讲的）。

Java · 面向接口编程
// 声明为接口类型,调用者根本不知道、也不需要知道底层是数组还是链表
List<String> names = new ArrayList<>();
// 换成 LinkedList,上层调用代码一行都不用改——这就是 ADT 分层带来的可替换性
names = new LinkedList<>();

三、为什么要有这一层抽象

  • 可替换性: 底层实现可以随时替换（数组 ↔ 链表），只要行为契约不变，上层代码不用改一行——这是"开闭原则"的具体体现。
  • 降低认知负担: 使用一个"栈"时，你只需要知道 push/pop，不需要关心它内部是用数组还是链表实现的。分层之后，每一层只需要理解自己这一层的东西。
  • 先设计契约、再选实现: 工程实践中，往往先根据业务场景确定需要哪些操作（ADT），再根据这些操作的调用频率去选具体实现——这也是这一章接下来要讨论"数组实现 vs 链表实现"的意义所在。

四、易错点与面试高频考点

误区:把 ADT 等同于"接口(interface)"这个语法关键字。ADT 是一个概念，接口只是编程语言提供的、用来表达 ADT 的一种语法工具。C 语言没有 interface 关键字，但一样可以通过"只暴露函数指针 + 隐藏结构体细节"的方式实现 ADT。
面试高频:"接口和实现类分离的好处是什么?"——标准答案要点:解耦、可替换、可测试（可以用 mock 实现同一接口）、符合依赖倒置原则。回答时如果能举出 List/ArrayList/LinkedList 的例子，会显得对原理理解得更扎实。

小结

  • ADT = 一组数据 + 一组操作的行为契约，只关心"是什么"，不关心"怎么实现"。
  • Java 里 List 接口是 ADT，ArrayList/LinkedList 是同一个 ADT 的两种物理实现。
  • 这层抽象带来可替换性、降低认知负担，是"面向接口编程"思想的根源。
