表 List ADT 与基本操作

同一套操作契约，换一种底层存储，复杂度可能天差地别——这就是本章要解决的核心矛盾。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、它到底在解决什么问题

上一篇讲了 ADT 是"契约"，这一篇具体看"表(List)"这个 ADT 到底约定了哪些操作。表是一种线性表:除了第一个元素没有前驱、最后一个元素没有后继，其余每个元素恰好有一个前驱和一个后继。这是现实世界里最常见的组织方式——账单流水、待办事项、播放列表，本质上都是"表"。

Weiss 书里把表的基本操作归纳为一组标准动作，任何"表"的实现都必须支持:

操作语义Java List 对应方法
按下标访问取出第 i 个元素get(int index)
插入在指定位置插入一个元素add(int index, E e)
删除移除指定位置的元素remove(int index)
查找按值找元素所在位置indexOf(Object o)
求长度表中当前元素个数size()
遍历按顺序访问每个元素Iterator / for-each

二、核心矛盾:同样的操作，代价不同

如果只看这张表的方法签名，ArrayList 和 LinkedList 看起来完全一样——都实现了 List 接口，调用方式一模一样。但同一个 add(index, e) 操作，背后代价却完全不同:

Java · 同一个方法，两种代价
List<Integer> arr = new ArrayList<>();
List<Integer> link = new LinkedList<>();

// 语法上完全一样,但数组实现要挪动后面所有元素——O(n)
// 链表实现只需要改两个指针——O(1)(前提是已经拿到了插入点的引用)
arr.add(0, 1);
link.add(0, 1);

这就是本章接下来要展开的核心内容:List ADT 只定义了"做什么"，而"做得多快"完全取决于底层是数组还是链表。这也是为什么 Java 要同时提供 ArrayList 和 LinkedList 两种实现——没有一种底层结构能在所有操作上都占优，需要根据业务场景的操作分布来选择。

三、按下标访问 vs 按位置(引用)访问

这里有一个常被忽略的细节:List ADT 里的"位置"其实有两种含义，理解这个区别对后面理解链表操作的真实复杂度很关键。

  • 按下标(index)访问: 传入一个整数下标，比如 get(3)。这种方式对数组是 O(1)（直接算地址），对链表是 O(n)（必须从头遍历数下标）。
  • 按已有引用/迭代器位置访问: 如果你已经拿着一个指向某个节点的迭代器（如 ListIterator），在这个位置插入/删除是 O(1)——因为不需要重新定位。
这也是为什么 LinkedList 提供了 listIterator() 并鼓励用迭代器遍历修改，而不是 for (int i=0;...) list.get(i)——后者每次 get(i) 都要重新从头（或尾）遍历，把原本 O(n) 的整体遍历退化成 O(n²)。

四、易错点

常见误区:以为"链表插入是 O(1)"这句话在任何场景下都成立。真相是:只有拿到了目标位置的节点引用，插入本身才是 O(1)；如果是按下标插入（比如 add(index, e)），必须先花 O(n) 遍历找到该位置，插入操作整体仍是 O(n)。第 5 篇会专门辨析这个高频误答。

小结

  • List ADT 定义了一组标准操作:按下标访问、插入、删除、查找、求长度、遍历。
  • 同一组操作，在数组实现和链表实现下复杂度可能完全不同——这是本章要研究的核心问题。
  • "按下标访问"和"按已有引用访问"是两种不同的定位方式，复杂度差异很大，容易被面试题用来设陷阱。
  • 接下来两篇分别深入数组实现（ArrayList）和链表实现（LinkedList）的具体原理。

与其他知识点的联系

  • ADT 与接口/实现分离的设计思想：上一篇讲的"抽象数据类型"在 Java 里直接对应到 List 接口——接口只声明 get/add/remove/size 等方法签名，不涉及任何存储细节，这正是"契约"和"实现"分离的具体体现，也是面向对象设计里"依赖抽象而非具体实现"（针对接口编程）原则的教科书级案例。
  • Java Collection 框架的继承体系：List 继承自 Collection，Collection 继承自 Iterable——这套继承链把"可遍历"、"可增删查"、"有顺序"三层语义逐级叠加，理解这个层次结构有助于理解为什么 Set 没有 get(index) 方法（因为它不继承自 List，没有"顺序"这个契约），而 List 却有。
  • 迭代器模式（Iterator Pattern）：表的"遍历"操作在 ADT 层面只说"按顺序访问每个元素"，具体怎么实现顺序访问被抽象成了 Iterator 接口——这让遍历逻辑与容器的内部结构解耦，调用方不需要知道底层是数组还是链表，也是本 Tab 后面"迭代器 Iterator 设计"一篇要深入的内容。
  • 与 ArrayList/LinkedList 具体实现的承接关系：这一篇给出的操作契约和复杂度问题，直接是下两篇的研究对象——"按下标访问 O(1) vs O(n)"这个矛盾会在 ArrayList（数组连续寻址）和 LinkedList（指针遍历）两篇里给出具体的源码级解释。
  • 与 List ADT 平行的 Queue/Stack/Deque：Weiss 书把表、栈、队列放在同一章讲，是因为栈和队列本质上是"操作受限的表"——栈只能在一端进出，队列只能一端进一端出，理解了表的通用操作契约后，栈和队列可以看作是对表操作的子集约束，而不是全新的数据结构。
