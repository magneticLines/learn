双端队列 Deque

两端都能插删的结构——一个类型同时吃掉栈和队列的全部需求,也是本 Tab「线性结构」的收官篇。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、Deque 是什么:栈和队列的"超集"

Deque(Double-Ended Queue,双端队列)允许在两端都进行插入和删除——既能在头部操作,也能在尾部操作。想清楚一件事就够了:栈 = 只用 Deque 的一端;队列 = 用 Deque 的两端(一端进一端出)。Deque 不是"栈和队列之外的第三种东西",而是把两者的能力都装进了同一个容器。

想要的语义用 Deque 怎么做
当栈用(LIFO)只用头部:push=addFirst,pop=removeFirst,peek=peekFirst
当队列用(FIFO)头出尾进:offer=addLast,poll=removeFirst,peek=peekFirst

二、联系 Java Deque 接口:方法是"故意"重复的

打开 java.util.Deque 源码会发现同一个操作有好几个名字,这不是冗余,而是为了兼容不同的心智模型:

Java · Deque 方法家族
interface Deque<E> extends Queue<E> {
    // 双端队列本源方法:显式指定头/尾,语义最清晰
    void addFirst(E e);  void addLast(E e);
    E pollFirst();     E pollLast();

    // 栈的心智模型:push/pop 只操作头部
    void push(E e);       // = addFirst
    E pop();            // = removeFirst

    // 队列的心智模型(继承自 Queue):offer/poll
    boolean offer(E e);   // = addLast
    E poll();            // = pollFirst
}

这正是“一套底层能力,多套调用语义”的接口设计范式——写代码时用哪组方法名,直接表达了"我把它当栈用"还是"我把它当队列用",提升可读性。

三、ArrayDeque 为什么是 Stack 和 LinkedList 的现代替代方案

官方 Javadoc 明确写道:ArrayDeque 作为栈使用比 Stack 更快,作为队列使用比 LinkedList 更快。原因可以直接对应到前几篇学过的知识:

对比维度StackLinkedListArrayDeque
底层结构继承 Vector(数组)双向链表循环数组(本篇上一节的循环队列思想)
线程安全方法自带 synchronized无无
单线程下的代价白付同步开销每个元素是独立对象,内存碎片化,cache miss 多连续内存,顺序访问对 CPU 缓存友好
扩容倍增(继承自 Vector)不需要,但每次 new Node 有对象头开销倍增,均摊 O(1)

核心原因归纳为两点:无锁(单线程场景不需要为线程安全买单)和数组的空间局部性(CPU 从内存读数据是按缓存行批量读的,连续数组比散落各处的链表节点命中缓存的概率高得多,这也是"数组实现通常比链表实现快"的底层原因,不只是理论上的常数因子)。

结论:无论要栈还是要队列,只要不是必须要线程安全,ArrayDeque 都是默认首选。这不是"风格偏好",而是有真实的性能和设计依据。

四、本 Tab 小结:线性结构的完整脉络

回顾「线性结构」这个 Tab 走过的路,是一条完整的设计推理链:

  1. 抽象数据类型 ADT:先定义"表能做什么"(操作集合),再谈"怎么实现"——接口和实现分离,是一切后续讨论的起点。
  2. 数组实现:连续内存,随机访问 O(1),但中间插删要搬移,容量满了要扩容。
  3. 链表实现:不连续内存,中间插删 O(1)(有引用在手时),但随机访问要沿指针走,退化 O(n)。
  4. ArrayList vs LinkedList 取舍:没有免费的午餐,场景决定选择——查得多选数组,插删频繁(尤其两端)选链表/Deque。
  5. 迭代器 Iterator:把"怎么遍历"这件依赖具体实现的知识封装起来,让上层代码不必关心底层是数组还是链表,同时用 modCount 兜底检测并发结构修改。
  6. 栈 Stack:List 的裁剪特化——只留一端操作,LIFO,对应 JVM 调用栈、括号匹配、表达式求值。
  7. 队列 Queue:另一种裁剪特化——两端各管一件事,FIFO,数组实现必须"循环"才不会假溢出。
  8. 双端队列 Deque:把栈和队列的能力合并到同一个结构里,ArrayDeque 成为两者的现代最优实现。

这条脉络的共同主线是:先想清楚"需要什么操作"(接口),再根据操作的性能特征选"用什么内存布局实现"(数组/链表),最后用统一的抽象(迭代器)把差异封装掉——这套思考方式会在下一个 Tab「树」中继续出现:先定义树的遍历方式,再讨论 BST/AVL/红黑树等不同实现如何在增删查之间做取舍。
