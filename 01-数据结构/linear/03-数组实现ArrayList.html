数组实现 ArrayList(扩容 / 随机访问)

连续内存换来 O(1) 随机访问，但"连续"这个约束也决定了它扩容和中间插入的代价。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、连续内存带来的两面性

数组实现表的核心思路是:把所有元素挨着放在一段连续内存里，用下标做"寻址公式"——第 i 个元素的地址 = 首地址 + i × 元素大小。这是一个纯粹的数学计算，不需要任何遍历，所以随机访问是 O(1)。

但"连续"是一把双刃剑:

  • 好处: 随机访问 O(1)；元素紧密排列，缓存局部性好（CPU 预取一次能带上后面好几个元素，命中率高）。
  • 代价: 数组创建后长度固定，要"变长"就必须整块搬家；在中间插入/删除，必须把后面所有元素依次挪位，代价是 O(n)。

二、扩容机制:为什么是"倍数"而不是"+1"

Java 数组本身长度不可变，"动态数组"的错觉是 ArrayList 用扩容机制模拟出来的:数组满了就换一个更大的新数组，把旧数据搬过去。这里有一个关键设计决策:新容量为什么要按倍数增长，而不是每次只加固定的 1 个或 10 个？

如果每次只 +1:插入 n 个元素要扩容 n 次，每次搬运代价平均是 n/2，总代价是 O(n²)。如果按倍数扩容(比如 1.5 倍):扩容次数只有 O(log n) 次，且每次搬运的成本是几何级数——用摊还分析(amortized analysis)可以证明,平均下来每次 add 的代价仍是 O(1)。这是"空间换时间"的经典权衡:用偶尔的大代价搬运,换来绝大多数时候的常数时间插入。

联系 Java 源码:ArrayList.grow()

java.util.ArrayList 内部维护一个 Object[] elementData：

Java · ArrayList 扩容策略(简化还原)
private static final int DEFAULT_CAPACITY = 10; // 无参构造的默认容量
private Object[] elementData;

private Object[] grow(int minCapacity) {
    int oldCapacity = elementData.length;
    // 新容量 = 旧容量的 1.5 倍(oldCapacity >> 1 即除以2),不是简单 +1
    int newCapacity = oldCapacity + (oldCapacity >> 1);
    // 分配新数组并整块拷贝——这一步是 O(n),但发生频率随容量指数级降低
    return Arrays.copyOf(elementData, Math.max(newCapacity, minCapacity));
}

无参 new ArrayList<>() 并不会立即分配 10 个空间的数组，而是先指向一个空数组，第一次 add 时才真正分配容量 10 的数组——这是典型的延迟初始化(lazy initialization)，避免创建了很多空 ArrayList 却从没用过导致的内存浪费。

三、中间插入/删除为什么是 O(n)

因为数组要保持"连续"，在下标 i 处插入一个元素，i 后面的所有元素都要整体后移一位腾出空间；删除同理，后面的元素要整体前移填补空缺。

Java · System.arraycopy 搬运
public void add(int index, E element) {
    // 把 index 及之后的元素整体后移一位,为新元素腾位置——这是 O(n) 的根源
    System.arraycopy(elementData, index, elementData, index + 1, size - index);
    elementData[index] = element;
    size++;
}

但这个"搬运"是用 native 的 System.arraycopy 实现的，底层是一段内存块的批量复制指令，速度远快于逐个 Java 层面的赋值循环——这也是为什么即便复杂度是 O(n)，实际场景中 ArrayList 在中间插入删除也常常比"理论上 O(1)"的链表更快（下一篇会展开讲缓存局部性的影响）。

四、复杂度小结

操作复杂度原因
按下标访问 get(i)O(1)地址计算，无需遍历
尾部添加 add(e)均摊 O(1)大多数时候直接写入，偶尔触发一次 O(n) 扩容
中间插入/删除O(n)需要整体搬移后续元素
按值查找 indexOfO(n)无索引信息，只能逐个比较

五、易错点

坑 1: 频繁在循环里对 ArrayList 做"头部插入"(add(0, e))——每次都要整体后移，n 次头插的总代价是 O(n²)。如果业务场景确实需要频繁头部操作，应该考虑 LinkedList 或 Deque。
坑 2: 以为 ArrayList 的扩容倍数是 2 倍——JDK 实际用的是 1.5 倍（oldCapacity + oldCapacity/2），这是一个常被问错的面试细节。选 1.5 而不是 2，是为了在"扩容次数"和"内存浪费/无法复用旧内存空间"之间取得更好的折中。

小结

  • 连续内存让随机访问天生 O(1)，也让中间插入/删除天生 O(n)。
  • 扩容用倍数增长(JDK 是 1.5 倍)而不是固定增量，均摊复杂度可以做到 O(1)，本质是空间换时间。
  • ArrayList 的搬运操作依赖 native 的 System.arraycopy，实际性能远好于理论上的逐元素赋值。
