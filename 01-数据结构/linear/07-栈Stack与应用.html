栈 Stack(LIFO)与应用(括号匹配 / 表达式求值)

只在一端进出的线性表——最简单的结构,却撑起了函数调用、括号匹配、表达式求值等一大类问题。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、LIFO 到底是什么,为什么数组/链表都能实现

栈(Stack)是一种只允许在同一端做插入和删除的表,这一端叫栈顶(top)。后进的元素先出——LIFO(Last In First Out)。它是 List ADT 的一个特化子集:只暴露 push(压栈)、pop(弹栈)、peek(看栈顶) 三个操作,故意裁剪掉了随机访问、中间插入等能力。

这解释了为什么数组和链表都能高效实现栈:栈的所有操作都只发生在「一端」,而数组的尾部追加/删除、链表的头部插入/删除,恰好都是 O(1) 的操作——栈从不要求"随机访问"或"中间操作",所以两种底层实现都不会暴露自己的弱点。这也是为什么工业界实现栈几乎总选数组(如 ArrayDeque):没有短板,还多一个缓存友好的优点。

二、联系 JVM 调用栈:栈不是"数据结构里的抽象概念",而是每天都在跑

每次调用一个方法,JVM 就在虚拟机栈里压入一个栈帧(Stack Frame),保存局部变量、操作数栈、返回地址;方法返回时,栈帧被弹出。这正是最直观的 LIFO——最后调用的方法最先返回。

联系递归:递归函数不断调用自己,本质就是不断往调用栈里压栈帧。递归太深导致 StackOverflowError,本质就是这个"栈"(调用栈)满了——这也是为什么很多递归算法可以等价改写成显式用一个 Stack 对象 + 循环来实现,手动控制内存,避免调用栈溢出。

三、Java 里怎么选:java.util.Stack 是历史遗留

java.util.Stack 继承自 Vector,是 JDK 1.0 就有的老类。它能用,但有硬伤:

问题说明
继承 Vector所有方法都加了 synchronized,单线程场景白白承受锁开销
破坏封装继承下来的 get(i)、insertElementAt() 等方法能绕过栈的语义随意改中间元素
官方态度Javadoc 明确建议:更完整的 LIFO 操作应该用 Deque,ArrayDeque 是首选实现
Java · 用 Deque 代替 Stack
// 反例:能跑,但不推荐——线程安全用不上,还继承了一堆不该有的方法
Stack<Integer> old = new Stack<>();

// 正例:面试/代码规范都认这个——数组实现、无锁、接口语义干净
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);   // 压栈,等价于 addFirst
stack.pop();    // 弹栈,等价于 removeFirst,空栈抛异常
stack.peek();   // 看栈顶,不弹出,空栈返回 null
面试高频点:"Java 里栈用什么实现?"标准答案是 ArrayDeque,而不是 Stack。能说出为什么(继承 Vector 导致不必要的同步开销、破坏封装)才算答到点上。

四、应用一:括号匹配

思路很直接:遇到左括号就压栈"记住它在等一个右括号";遇到右括号就弹栈看栈顶是否是对应的左括号。这利用了栈"后进先出"恰好对应"最近打开的括号最先闭合"的直觉。

Java · 括号匹配
static boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    for (char c : s.toCharArray()) {
        if (c == '(' || c == '[' || c == '{') {
            stack.push(c); // 记住"欠"一个右括号
        } else {
            // 右括号出现,但没有左括号可配对 → 一定不合法
            if (stack.isEmpty()) return false;
            char top = stack.pop();
            if ((c == ')' && top != '(') ||
                (c == ']' && top != '[') ||
                (c == '}' && top != '{')) return false; // 类型不配对
        }
    }
    return stack.isEmpty(); // 走完还有没配对的左括号 → 不合法
}

五、应用二:表达式求值(中缀 → 后缀,栈是核心引擎)

人写的表达式 3 + 4 * 2 是中缀,必须先知道运算符优先级才能算对;而后缀(逆波兰)表达式 3 4 2 * + 不需要优先级和括号,从左到右扫描、遇到运算符就对栈顶两个数做运算——天然适合用栈实现,这也是编译器/计算器内部真实采用的做法。

后缀表达式求值:一个栈就够

Java · 后缀求值思路
// 扫描 tokens:数字就压栈;运算符就弹出栈顶两个数计算,把结果再压回去
for (String token : tokens) {
    if (isNumber(token)) {
        stack.push(Integer.parseInt(token));
    } else {
        int b = stack.pop(), a = stack.pop(); // 注意弹出顺序:先弹的是右操作数
        stack.push(apply(token, a, b)); // 结果重新入栈,继续参与后续计算
    }
} // 扫描结束,栈里剩的唯一元素就是最终结果

中缀转后缀(经典的调度场算法,Shunting-yard)也是靠一个运算符栈:数字直接输出;运算符入栈前,先把栈顶优先级 ≥ 自己的运算符弹出输出,保证优先级高的先算;遇到左括号直接压栈,遇到右括号则一路弹出到左括号为止。本质仍是利用栈"后进先出"来暂存"还没轮到计算"的运算符。

建立联系:括号匹配和表达式求值背后是同一个模式——用栈保存"还没处理完的上下文",等遇到能与之匹配/结合的东西时再弹出处理。DFS(深度优先搜索)的显式栈实现、浏览器的前进后退历史,本质也是这个模式的变体。

小结

  • 栈只在一端操作,LIFO;数组/链表实现栈都是 O(1),故工业界优先选数组(缓存友好)。
  • JVM 调用栈就是活生生的 LIFO 案例,递归深度受限于它的容量。
  • Java 里 java.util.Stack 是遗留类(继承 Vector,自带无用的同步开销,还破坏封装),应改用 Deque / ArrayDeque——面试高频点。
  • 括号匹配、表达式求值(中缀转后缀、后缀求值)都是"用栈暂存未完成上下文"这一模式的应用。
