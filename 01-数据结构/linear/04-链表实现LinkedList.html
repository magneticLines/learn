链表实现 LinkedList(单链 / 双链 / 循环)

放弃"连续内存"这个约束，用指针把散落各处的节点串起来，换来插入删除的常数时间。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、它到底在解决什么问题

上一篇看到数组实现的痛点:要保持内存连续，中间插入/删除就必须搬移一大片元素。链表的思路正好相反——不追求连续，每个数据元素(节点)只记录"下一个节点在哪"，用指针把逻辑上相邻的元素串起来。这样插入删除时，只需要修改前后节点的指针指向，完全不需要搬移其他元素。

代价是什么？失去了地址计算能力。数组能通过 首地址 + i × 元素大小 直接算出第 i 个元素在哪，链表做不到——它必须从头(或尾)一个个顺着指针走到第 i 个节点，按下标访问退化成 O(n)。这正是数组和链表的核心权衡:用随机访问能力换插入删除效率，或反过来。

二、三种结构的区别与适用场景

结构特点适用场景
单向链表每个节点仅有 next，只能从头向后单向遍历只需正向遍历、内存开销要求最小的场景
双向链表每个节点有 next 和 prev，可双向遍历需要"删除某节点时快速找前驱"、需要双向遍历(如 LinkedList)
循环链表尾节点的 next 指回头节点，形成环需要循环处理的场景，如时间片轮转调度、约瑟夫环问题

双向链表比单向链表多了一个 prev 指针，代价是每个节点多占 8 字节(64位引用)内存，但换来的是:删除一个节点时不需要额外遍历找它的前驱(单链表删除中间节点必须先找到前驱才能改指针)，且可以支持从尾部向前遍历。这也是为什么 Java 选择用双向链表实现 LinkedList。

三、联系 Java 源码:java.util.LinkedList

LinkedList 是双向链表，同时实现了 List 和 Deque(双端队列)两个接口——这不是偶然，正是因为双向链表天然支持从头尾两端 O(1) 存取，非常适合当队列/栈用。

Java · LinkedList 的节点结构(简化还原)
private static class Node<E> {
    E item;          // 数据域
    Node<E> next;   // 指向后继
    Node<E> prev;   // 指向前驱——单链表没有这个字段
}
// LinkedList 本身持有 first、last 引用,头尾操作天生 O(1)

get(index) 是 LinkedList 的性能痛点，但源码里做了一个小优化，值得注意:

Java · 从更近的一端开始遍历
Node<E> node(int index) {
    // 判断 index 离头近还是离尾近,减少一半的平均遍历步数
    if (index < (size >> 1)) {
        Node<E> x = first;
        for (int i = 0; i < index; i++) x = x.next;
        return x;
    } else {
        Node<E> x = last;
        for (int i = size - 1; i > index; i--) x = x.prev;
        return x;
    }
}

这个优化只是把最坏遍历步数减半，时间复杂度依然是 O(n)——量级没变，只是常数因子变小了，不要误以为它把 get(index) 优化成了 O(1)。

四、手写单向链表:头插与删除

用一个精简的单向链表来看核心操作为什么是 O(1)——关键在于"已经站在正确位置"，不需要搬移任何其他元素:

Java · 单链表头插/删头(节选自 SimpleLinkedList)
private static class Node {
    int data;
    Node next;
    Node(int data) { this.data = data; }
}
private Node head;

// 头插:只需要"接上旧头、更新头指针"两步,和链表长度无关——O(1)
public void addFirst(int data) {
    Node n = new Node(data);
    n.next = head;   // 新节点接住原来的头
    head = n;        // head 指向新节点,旧链表结构完全不动
}

// 删头:只是把 head 挪到下一个节点,原头节点自然被 GC 回收——O(1)
public void removeFirst() {
    if (head != null) head = head.next;
}
对比一下:这份 SimpleLinkedList 的 addLast 需要从头遍历到尾才能接上新节点，是 O(n)——因为它没有像 LinkedList 那样维护一个 last 引用。这提醒我们:链表的 O(1) 插入是有前提的，前提是你已经拿着目标位置的节点引用(比如维护了头尾指针，或用迭代器定位好了)，否则找到那个位置本身就要花 O(n)。

五、易错点

坑: 认为链表天生比数组"节省内存"。恰恰相反，每个节点除了数据本身，还要额外存 1~2 个指针(引用)，双向链表每个节点比数组多出的开销更大。数据量小、元素本身也小的时候，链表的指针开销占比反而更高。

小结

  • 链表放弃内存连续性，用指针换取插入删除时不需要搬移其他元素。
  • 单链表省内存但只能单向遍历；双向链表能双向遍历、删除时无需额外找前驱；循环链表适合轮询类场景。
  • java.util.LinkedList 是双向链表，同时实现 List 和 Deque，靠 first/last 引用让头尾操作恒为 O(1)。
  • 链表的"O(1) 插入"永远有前提:必须已经站在目标位置，否则定位过程本身就是 O(n)。
