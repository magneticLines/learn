迭代器 Iterator 设计

如何在不暴露底层是数组还是链表的前提下,给所有容器一套统一的遍历方式。

📖 来源:《数据结构与算法分析:Java语言描述》 第 3 章「表、栈和队列」

一、它到底在解决什么问题

上一篇讲到 ArrayList 底层是连续数组、LinkedList 底层是双向链表——两者遍历一遍元素的物理方式完全不同:数组是下标 i++ 直接跳,链表必须沿着 next 指针一步步走,否则退化成 O(n²)。

如果没有统一抽象,调用方就得为每种容器写一套遍历代码,还得知道内部结构才能写「快」的那种。迭代器模式解决的正是这个矛盾:把「怎么遍历」的知识封装进容器自己提供的一个游标对象里,调用方只认hasNext()/next() 两个方法,不关心、也看不到底层是数组还是链表。这是「针对接口编程,不针对实现编程」在容器场景下的直接体现。

二、Java 怎么把它做成语言层面的东西

Iterator 是接口,只有三个方法:

Java · java.util.Iterator
public interface Iterator<E> {
    boolean hasNext();
    E next();
    default void remove() { // 唯一安全的"遍历中删除"入口
        throw new UnsupportedOperationException();
    }
}

而容器本身实现的是 Iterable 接口,只有一个方法 iterator(),负责「生产」一个迭代器实例。for-each 语法糖就是编译器在背后把代码改写成迭代器调用:

Java · for-each 的真身
for (String s : list) { use(s); }

// 编译器实际生成的代码大致等价于:
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    use(s);
}

这也解释了一个常见疑问:为什么数组能用 for-each,却不需要实现 Iterable?——因为数组是语言内置类型,编译器对它做了特殊处理,直接生成下标循环,并不走迭代器协议。

三、和 ArrayList / LinkedList 源码的联系

两种容器的迭代器内部状态完全不同,但对外接口一致——这正是模式的价值所在:

容器迭代器内部持有什么next() 做的事
ArrayList一个下标 cursorelementData[cursor++],O(1)
LinkedList一个 Node 引用node = node.next,O(1)

注意两者的 next() 单步都是 O(1)——但如果你不用迭代器,而是对 LinkedList 用 get(i) 下标循环,每次 get 都要从头(或从近的一端)重新走链表,整体退化成 O(n²)。这是迭代器不只是「好看」,更是性能正确用法的直接证据。

四、ConcurrentModificationException 的原理

容器内部维护一个 modCount(结构性修改次数计数器),迭代器创建时把它记一份快照 expectedModCount。每次 next() 调用前都会检查两者是否相等:

Java · ArrayList$Itr 简化版
final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}

这是一种fail-fast(快速失败)策略:不保证检测到所有并发修改,只是「尽力而为」地尽早暴露问题,防止程序在数据被污染的情况下继续跑出更诡异的错误。

高频踩坑:遍历时直接调用集合的 remove()(而不是迭代器的 iterator.remove())。
Java · 反例
for (String s : list) {
    if (s.equals("x")) list.remove(s); // list.modCount++,但 expectedModCount 没跟着变
} // 下一次 next() 检查时两者不等 → 抛 CME
Java · 正例
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("x"))
        it.remove(); // 迭代器内部同步更新 expectedModCount,不会触发 CME
}

还有一个变体坑更隐蔽:巧合地删到倒数第二个元素时,由于 hasNext() 的判断条件恰好躲过了检查,程序不抛异常但结果错误(少处理了一个元素)——这比直接抛异常更危险,因为它不报错,却悄悄产生了错误结果。所以「用 for-each 遍历时不要调用集合的 remove」不能靠「有没有抛异常」来验证,必须靠规则本身来避免。

联系:CME 只在单线程「遍历中结构性修改」场景下抛出,它不是线程安全机制。多线程场景下即便检测到也可能来不及(检测本身有竞态),真正需要并发安全遍历要用 CopyOnWriteArrayList 或并发容器,那是另一套设计。

小结

  • 迭代器模式:把「怎么遍历」封装进容器自己的游标对象,调用方只认统一接口,不关心底层实现。
  • Iterable.iterator() 生产 Iterator;for-each 是编译器对这套协议的语法糖。
  • ArrayList 迭代器靠下标、LinkedList 迭代器靠 Node 引用,单步都是 O(1)——但绕开迭代器直接对链表用 get(i) 会退化成 O(n²)。
  • modCount 快照比对实现 fail-fast 检测,遍历中删除必须用 iterator.remove()。
