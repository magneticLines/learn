字典树 Trie 的结构与应用

专门为字符串集合设计的树形结构，靠"共享前缀"把存储和查询都做到极致——哈希表做不到的前缀查询，它轻松搞定。

要解决的问题

如果要存储一批字符串（比如一个词典），并支持"某个字符串是否存在"、"以某个前缀开头的字符串有哪些"这类查询， 最直接的想法是用哈希表——把每个字符串整体作为 key。但哈希表有一个天生的短板： 它把字符串当成一个不可分割的整体来处理，没法回答"前缀"相关的问题。

Trie（字典树，也叫前缀树）换了一种思路：把字符串拆解成一个个字符，用树的路径表示字符串， 公共前缀在树上自然地共享同一条路径。

类比：想象一本纸质字典，所有以"ap"开头的单词（apple、application、apply）在字典里是紧挨着排列的， 因为它们共享前缀"ap"。Trie 树就是把这种"共享前缀"的组织方式变成了一棵树——从根节点出发，每往下走一层就是确定一个字符， 走到某条路径的终点就代表一个完整单词。

Trie 树的结构

每个 Trie 节点通常包含两部分信息：

  • 指向子节点的引用数组（或哈希表）：比如只处理小写英文字母，可以用一个长度 26 的数组，下标对应 'a'~'z'。
  • 是否为单词结尾的标记：因为 Trie 上的路径是共享的，需要一个标记区分"这里正好是一个完整单词的结尾"还是"只是路过"。
Trie.java插入与查询
class TrieNode {
    TrieNode[] children = new TrieNode[26]; // 只处理小写字母 a~z
    boolean isEnd = false;             // 标记这里是否是某个单词的结尾
}

public class Trie {
    private final TrieNode root = new TrieNode();

    // 插入一个单词：沿着字符路径往下走，节点不存在就创建
    public void insert(String word) {
        TrieNode node = root;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (node.children[idx] == null) {
                node.children[idx] = new TrieNode();
            }
            node = node.children[idx];
        }
        node.isEnd = true; // 走到路径终点，标记为完整单词
    }

    // 查询完整单词是否存在：路径必须存在，且终点标记为 isEnd
    public boolean search(String word) {
        TrieNode node = findNode(word);
        return node != null && node.isEnd;
    }

    // 查询是否存在某个前缀：路径存在即可，不要求 isEnd
    public boolean startsWith(String prefix) {
        return findNode(prefix) != null;
    }

    private TrieNode findNode(String s) {
        TrieNode node = root;
        for (char c : s.toCharArray()) {
            int idx = c - 'a';
            if (node.children[idx] == null) return null;
            node = node.children[idx];
        }
        return node;
    }
}

插入和查询的时间复杂度都是 O(L)，L 是字符串长度，与 Trie 中存了多少个字符串完全无关—— 这是因为每次操作只沿着一条路径走到底，不需要遍历其他分支。

相比哈希表的优势：前缀匹配

哈希表判断字符串是否存在也是 O(L)（计算哈希值需要遍历整个字符串），看起来和 Trie 差不多， 但哈希表做不到"查询以某前缀开头的字符串有哪些"——因为哈希函数是把整个字符串映射成一个哈希值， 两个前缀相同、后缀不同的字符串，哈希值可能天差地别，完全没有"相邻"关系。

而 Trie 上，前缀相同的字符串必然共享同一段路径，只要找到前缀对应的节点，以这个节点为根往下的整棵子树， 就是所有以该前缀开头的字符串——这正是搜索引擎、输入法"自动补全"功能的核心数据结构。

能力哈希表Trie 树
判断字符串是否存在O(L)，支持O(L)，支持
前缀匹配 / 自动补全不支持（需要遍历所有 key）O(L) 定位前缀节点 + 遍历子树
空间开销较低，只存字符串本身较高，公共前缀虽然共享，但每个节点通常要预留字母表大小的子节点数组
按字典序遍历所有字符串不支持（哈希打乱了顺序）支持，DFS 遍历天然按字典序输出
面试高频："实现一个支持自动补全的输入法功能，用什么数据结构？" 标准答案就是 Trie—— 输入前缀后，定位到对应节点，再对该节点的子树做一次 DFS/BFS，收集若干个单词作为补全建议即可。
常见误区：Trie 树的空间开销容易被低估。如果字母表很大（比如要支持 Unicode 全字符集）， 每个节点都用固定大小数组存子节点会浪费大量空间，此时通常改用哈希表（Map<Character, TrieNode>）作为子节点容器， 牺牲一点常数时间换取空间效率。

核心价值总结

  • Trie 用树的路径表示字符串，公共前缀天然共享，插入和查询都是 O(L)，与字符串数量无关。
  • 它解决了哈希表做不到的问题：前缀匹配、自动补全、按字典序遍历，这些都要求"前缀相同的字符串在结构上相邻"，只有 Trie 具备这个特性。
  • 空间换时间的典型权衡：子节点数组预留了字母表大小的空间，字母表越大浪费越明显，需要根据场景选择数组或哈希表实现子节点。

与其他知识点的联系

  • HashMap：两者都能做"是否存在"查询，但组织数据的方式截然相反——HashMap 靠哈希函数把字符串打散成看似无关的桶，追求分布均匀以减少冲突；Trie 靠字符路径把有公共前缀的字符串聚拢到一起，牺牲了这种"打散"换来了前缀相关的查询能力。选型时的核心问题是：业务需不需要"前缀""排序"这类和字符串内部结构相关的操作，需要就选 Trie，只需要精确匹配就选 HashMap。
  • AC 自动机（Aho-Corasick）：Trie 只能回答"单个模式串是否是某字符串的前缀"，如果要在一段文本里同时查找成百上千个模式串是否出现（比如敏感词过滤），就需要在 Trie 的基础上叠加 KMP 的 fail 指针思想，构造出 AC 自动机——这是 Trie 和字符串匹配算法（下一篇 KMP）两个知识点的结合体，也是多模式串匹配问题的标准解法。
  • 字典序与树的遍历：Trie 按字符 'a'~'z' 顺序组织子节点，对整棵树做一次 DFS（先访问较小下标的子节点）天然按字典序输出所有字符串——这和二叉搜索树"中序遍历得到有序序列"是同一类性质：树的结构本身编码了顺序信息，遍历方式决定了能不能把这种顺序"读出来"。
  • 最长前缀匹配与路由查找：Trie 的思想不止用在字符串上——IP 路由表查找最长前缀匹配（路由器判断一个 IP 该走哪条线路）用的就是把 IP 地址按二进制位组织成一棵 Trie，每一位是 0 还是 1 决定往左还是往右子树走，和字符串 Trie 按字符分层的原理完全一致，只是"字母表"从 26 个字符变成了 {0,1} 两个二进制位。
  • 空间复杂度权衡与其他树结构：Trie 用"空间换查询速度"这一权衡，和红黑树/B+ 树用"额外的平衡开销换稳定的对数级操作"是同一类工程取舍——数据结构的设计很少有绝对的最优解，几乎都是在时间、空间、实现复杂度三者之间找一个适合具体场景的平衡点。
