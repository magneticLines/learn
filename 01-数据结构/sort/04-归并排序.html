归并排序 Merge Sort(分治)

分治思想的教科书式应用:把大问题拆成两个小问题递归解决,再合并结果——用 O(n) 的额外空间换来了稳定性和有保证的 O(n log n)。

📖 来源:《数据结构与算法分析:Java语言描述》 第 7 章「排序」

一、分治思想:分-治-合

归并排序解决的问题是:如何把"排序一个大数组"这个难题,转化为若干个"排序小数组"的简单问题。它遵循经典的分治三部曲:

  • 分(Divide):把数组从中点切成两半,递归处理左右两半。
  • 治(Conquer):递归的基准情形——数组长度为 1 时天然有序,直接返回。
  • 合(Merge):把两个已经各自排好序的子数组,合并成一个整体有序的数组。

整个算法的"智慧"全部集中在"合并"这一步:两个有序数组的合并可以在 O(m+n) 时间内完成,只需要用两个指针分别指向两个子数组的开头,每次比较取较小的放入结果数组,指针后移。这是归并排序效率的根本来源。

二、走一遍具体例子

数组 [6, 3, 8, 2, 9, 1]:

分:[6,3,8] 和 [2,9,1]
继续分:[6] [3,8]→[3][8];[2] [9,1]→[9][1]
合并回来:[3][8] 合并成 [3,8];[9][1] 合并成 [1,9]
再合并:[6] 与 [3,8] 合并成 [3,6,8];[2] 与 [1,9] 合并成 [1,2,9]
最终合并:[3,6,8] 与 [1,2,9] 合并 → 双指针比较 3 vs 1 取1,3 vs 2 取2,3 vs 9 取3,6 vs 9 取6,8 vs 9 取8,剩9 → [1,2,3,6,8,9]

三、代码实现

Java · 归并排序
public static void mergeSort(int[] a, int lo, int hi) {
    if (lo >= hi) return; // 治:长度<=1天然有序
    int mid = (lo + hi) / 2;
    mergeSort(a, lo, mid);       // 分:左半递归
    mergeSort(a, mid + 1, hi);  // 分:右半递归
    merge(a, lo, mid, hi);       // 合:两个有序区合并
}

private static void merge(int[] a, int lo, int mid, int hi) {
    int[] tmp = new int[hi - lo + 1]; // 必须借助额外数组,不能原地合并
    int i = lo, j = mid + 1, k = 0;
    while (i <= mid && j <= hi) {
        // <= 而不是 <,遇到相等时优先取左边——这正是稳定性的保证点
        tmp[k++] = (a[i] <= a[j]) ? a[i++] : a[j++];
    }
    while (i <= mid) tmp[k++] = a[i++];
    while (j <= hi)  tmp[k++] = a[j++];
    System.arraycopy(tmp, 0, a, lo, tmp.length);
}

四、为什么稳定,为什么需要 O(n) 额外空间

稳定性:关键就在合并那一行的比较条件 a[i] <= a[j]——当两个元素相等时,始终优先取左边子数组的元素,而左边子数组在原始数组中的位置本来就靠前。只要每一层合并都遵循这个规则,相同 key 的元素在最终结果里的相对顺序就和输入时完全一致。

为什么需要额外空间:合并两个有序区间时,如果直接在原数组上"腾位置",会覆盖掉还没被比较过的元素——所以必须借助一个临时数组存放合并结果,再拷贝回去。这是它区别于堆排序/快速排序"原地"特性的根本原因,空间复杂度为 O(n)(不是 O(log n) 的递归栈,是合并阶段真正分配的辅助数组)。

五、联系:Java Collections.sort / Arrays.sort(Object[]) 用的正是它的变种

这是本篇最重要的联系点。Java 对对象数组排序(Arrays.sort(Object[])、Collections.sort(List))底层用的是 TimSort——归并排序的高度优化变种(结合了插入排序处理小分区、识别已有序的"自然运行段"等优化),而不是快速排序。

为什么必须用归并而不是快排? 因为对象排序几乎总是搭配 Comparator,而业务上经常要求相等元素保持原始相对顺序(比如先按部门排序、再按入职时间排序的多字段排序,要求第二次排序不能打乱第一次排序已经确定的顺序)。快速排序天生不稳定,一旦用于对象排序就可能出现"看似排对了,但相等 key 的相对顺序被打乱"的诡异 bug。归并排序的稳定性是它被选中的决定性原因,牺牲一点空间换来正确性。

六、易错点

递归写反 <= 为 <。 合并时如果写成 a[i] < a[j] 才取左边,相等时会取右边,稳定性直接被破坏——这是实现归并排序最容易忽略的细节,也是笔试中考察"你是否真的理解稳定性怎么来的"的高频点。
以为归并排序也能原地做。 理论上存在原地归并算法,但代价是牺牲时间复杂度或引入极其复杂的实现;标准归并排序的 O(n) 额外空间是"用空间换稳定性和确定性能"的正常代价,不要强行追求省略这部分空间。

对比表

算法最坏时间空间稳定性特点
归并排序O(n log n)O(n)稳定分治+合并,需要额外空间,复杂度稳定不受输入影响

小结

  • 归并排序 = 分治:递归拆到最小,合并两个有序数组。
  • 合并阶段是效率核心,O(m+n) 双指针扫描,同时也是稳定性的来源(相等取左边)。
  • 需要 O(n) 额外空间,区别于堆排序/快排的原地特性。
  • Java Arrays.sort(Object[]) / Collections.sort 用的 TimSort 正是归并的优化变种,因为对象排序需要稳定性。
