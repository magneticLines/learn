快速排序 Quick Sort 与枢轴选取

实践中最快的通用排序算法,但"最快"是有条件的——枢轴选不好,平均 O(n log n) 会退化成最坏 O(n²)。

📖 来源:《数据结构与算法分析:Java语言描述》 第 7 章「排序」

一、思路:选枢轴、分区、递归

快速排序也是分治,但"分"的方式和归并排序完全不同:归并排序是"不管数据内容,机械地从中点切开";快速排序是先选一个基准值(枢轴,pivot),把数组重新排列成"小于枢轴的都在左边,大于枢轴的都在右边",枢轴自身就落在了它最终该在的位置上。之后只需要分别对左右两部分递归排序,不需要"合并"这一步——因为分区完成后整体已经是有序的骨架。

  1. 选枢轴:从数组中选一个元素作为基准。
  2. 分区(partition):遍历数组,把小于枢轴的元素移到左边,大于的移到右边,枢轴放到中间分界点。
  3. 递归:对左右两个子数组分别重复上述过程。

二、走一遍具体例子(以最后一个元素为枢轴)

数组 [5, 1, 8, 3, 9, 2],选最后一个元素 2 作为枢轴:

用双指针分区:遍历 5,1,8,3,9,与枢轴 2 比较,只有 1 比 2 小,放到左边区域
分区结束:[1, 5, 8, 3, 9, 2] → 把枢轴 2 换到分界位置 → [1, 2, 8, 3, 9, 5](2 已到最终位置,下标1)
递归左边:[1],长度1天然有序
递归右边 [8, 3, 9, 5]:选 5 为枢轴,分区后 [3, 5, 9, 8](5 到位,下标1,相对右区间)
继续递归 [3](有序)和 [9, 8]:选 8 为枢轴,分区后 [8, 9]
最终合起来:[1, 2, 3, 5, 8, 9]

三、代码实现(Lomuto 分区)

Java · 快速排序
public static void quickSort(int[] a, int lo, int hi) {
    if (lo >= hi) return;
    int p = partition(a, lo, hi); // 分区后枢轴已在最终位置 p
    quickSort(a, lo, p - 1);
    quickSort(a, p + 1, hi);
}

private static int partition(int[] a, int lo, int hi) {
    int pivot = a[hi]; // 简单起见选最后一个元素——但这正是最坏情况的隐患来源
    int i = lo;
    for (int j = lo; j < hi; j++) {
        if (a[j] < pivot) {
            int tmp = a[i]; a[i] = a[j]; a[j] = tmp;
            i++;
        }
    }
    int tmp = a[i]; a[i] = a[hi]; a[hi] = tmp; // 枢轴归位
    return i;
}

四、为什么平均最快,又为什么最坏会退化

平均 O(n log n):如果每次分区都能把数组大致分成两半,递归深度是 O(log n),每层分区总代价是 O(n),整体就是 O(n log n)——而且分区过程是顺序扫描 + 少量交换,对 CPU 缓存非常友好,这也是快排实测中常常比堆排序、归并排序更快的原因(常数因子小)。

最坏 O(n²):如果每次选的枢轴都是当前区间的最大值或最小值,分区后一边是空的、另一边有 n-1 个元素,递归深度退化为 O(n),总代价变成 n + (n-1) + ... + 1 ≈ O(n²)。最经典的触发场景就是:代码固定选"第一个元素"或"最后一个元素"作为枢轴,而输入数组恰好已经是正序或逆序排列的——这是面试高频考点,也是很多"手写快排被出题者用有序数组卡爆"的真实原因。

陷阱示例: 上面的代码固定选 a[hi] 作为枢轴。如果传入一个已经升序排好的数组,每次分区枢轴都是当前区间的最大值,分区结果永远是"左边 n-1 个、右边 0 个",递归树退化成一条链,退化到 O(n²) 且递归深度 O(n) 可能导致栈溢出。

五、规避最坏情况:枢轴选取策略

策略做法效果
随机化每次从当前区间随机选一个元素作为枢轴使最坏情况的触发依赖于"随机数运气差",期望复杂度仍是 O(n log n),几乎不可能被对手构造出的输入卡到最坏
三数取中取区间首、中、尾三个元素的中间值作为枢轴对"已经有序或接近有序"的输入特别有效,能大幅降低分区不均衡的概率
双轴/多轴同时选两个枢轴,一次分成三段JDK 双轴快排的思路,进一步减少递归层数和比较次数

六、联系:Java Arrays.sort(int[]) 用的正是它的变种

与上一篇"归并排序被用于对象数组"形成鲜明对比:Arrays.sort(int[])(以及其他基本类型数组)底层使用的是双轴快速排序(Dual-Pivot QuickSort)。原因很直接——基本类型没有"对象身份"的概念,两个值相等的 int 完全没有区别,不存在"相对顺序"需要保持,所以不需要稳定性,可以放心选用不稳定但平均更快、原地不需要额外空间的快速排序。

这组对比是理解 JDK 排序 API 设计的关键:基本类型数组用快排(不需要稳定性,追求速度),对象数组/List 用归并的变种 TimSort(需要稳定性)。同一个 Arrays.sort 方法名背后,因为参数类型不同,底层算法选择的依据完全不同。

对比表:快速排序 vs 归并排序

维度快速排序归并排序
分治方式先分区(枢轴归位)再递归,无需合并先机械对半分,递归后需要合并
平均时间O(n log n),常数因子更小O(n log n),常数因子略大

小结

  • 快速排序 = 选枢轴 + 分区(归位) + 递归,不需要合并步骤。
  • 平均 O(n log n) 且缓存友好,但枢轴选不好会退化到最坏 O(n²)(经典陷阱:有序数组 + 固定选首/尾元素)。
  • 随机化、三数取中是规避最坏情况的常见手段。
  • Java Arrays.sort(int[]) 用双轴快排(不需要稳定性);对象数组用归并变种 TimSort(需要稳定性)。
