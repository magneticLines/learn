排序与图

图的表示(邻接矩阵 / 邻接表)第9章

图是"顶点 + 边"的抽象,几乎所有关系型问题(依赖、路径、连通)都能建模成图。表示方式的选择,直接决定了后续算法的效率。

📖 来源:《数据结构与算法分析:Java语言描述》 第 9 章「图论算法」

一、图的基本概念

一张图 G = (V, E) 由顶点集合 V(vertex,代表实体,比如城市、任务、人)和边集合 E(edge,代表关系,比如道路、依赖、好友关系)组成。图比树更"自由":树里每个节点只有一个父节点,图里任意两个顶点都可能有边,甚至可能出现环。

  • 有向图 / 无向图:边有没有方向。"A 关注 B"是有向的(B 不一定关注 A);"A 和 B 是朋友"是无向的。
  • 带权图 / 无权图:边上有没有附加数值(权重)。权重可以表示距离、成本、时间。城市间的公路网就是带权无向图;课程先修关系就是无权有向图。

这些概念组合起来,就对应了现实中的各种系统:社交网络(无向、无权或带权)、路网导航(带权有向/无向)、任务依赖(有向、无权)。

二、两种表示方式

邻接矩阵(Adjacency Matrix)

用一个 V × V 的二维数组 matrix[i][j] 表示"顶点 i 到顶点 j 是否有边"(或者边的权重,没有边则记为 ∞)。这是最直观的表示——把"关系"直接摊平成一张表。

Java · 邻接矩阵
// 4 个顶点,matrix[i][j] 存权重,Integer.MAX_VALUE 表示无边
int[][] matrix = new int[4][4];
matrix[0][1] = 5; // 顶点0 -> 顶点1,权重5
// 查询"0到1是否有边":matrix[0][1] != 0,直接数组下标访问,O(1)

邻接表(Adjacency List)

每个顶点只维护"自己指向哪些顶点"的一张列表,而不是把所有顶点两两的关系都摊开。Java 里最常见的写法是 Map<Integer, List<Integer>> 或 List<List<Integer>>,带权图则用 Map<Integer, List<Edge>>(Edge 里存目标顶点和权重)。

Java · 邻接表
// key:起点; value:该顶点所有出边(目标顶点 + 权重)
Map<Integer, List<Edge>> graph = new HashMap<>();
graph.computeIfAbsent(0, k -> new ArrayList<>())
     .add(new Edge(1, 5)); // 0 -> 1,权重5
// 查询"0到1是否有边":遍历 graph.get(0) 这条链表,O(度数)

三、空间与查询效率的取舍

为什么要有两种表示?因为它们在空间占用和操作效率上做了完全相反的取舍——这本质上和"数组 vs 链表"是同一类权衡:矩阵用固定的、可预测的布局换查询速度,链表用紧凑的、按需分配的布局换空间。

维度邻接矩阵邻接表
空间复杂度O(V²),与边数无关O(V + E),与实际边数成正比
判断 i→j 是否有边O(1),直接下标O(度数),要遍历链表
遍历某顶点的所有邻居O(V),要扫一整行O(度数),链表本身就是邻居集合
适合场景稠密图(E 接近 V²,比如接近完全图)稀疏图(E 远小于 V²,现实中的图绝大多数是稀疏图)
经验法则:现实世界的图(社交网络、路网、依赖图)几乎都是稀疏图——一个人不可能和所有人都是好友,一个任务不可能依赖所有其他任务。所以邻接表是绝大多数场景的默认选择,只有在图很小、很稠密、且需要频繁判断任意两点是否直连时才考虑邻接矩阵。

四、建立联系

图不是孤立的概念——它是很多实际系统的地基:

  • 路由:网络中的路由器构成图,最短路径算法决定数据包怎么走。
  • 依赖解析:Maven/Gradle/npm 的包依赖关系是有向图,下一篇的拓扑排序就是解决"按什么顺序构建"的问题。
  • 社交网络:好友关系、关注关系天然就是图,推荐系统里的"共同好友"就是图上的邻居查询。

小结

  • 图 = 顶点 + 边,按方向分有向/无向,按是否带权分带权/无权。
  • 邻接矩阵:空间 O(V²),查边 O(1),适合稠密图。
  • 邻接表:空间 O(V+E),查边要遍历,适合稀疏图(现实中绝大多数场景)。
  • Java 里邻接表常用 Map<Integer, List<Edge>> 实现。
