堆排序 Heap Sort

把前面学过的二叉堆直接用来排序:建堆一次,取堆顶 n 次,原地完成,最坏情况也稳稳保持在 O(n log n)。

📖 来源:《数据结构与算法分析:Java语言描述》 第 7 章「排序」

一、思路:堆顶永远是极值,取出来放到末尾就行

你已经在二叉堆那一篇学过:大顶堆的堆顶永远是当前最大值。排序的需求恰好是"每次找出剩余元素里的最大值放到正确位置"——这不就是堆的拿手好戏吗?堆排序做的事情分两步:

  1. 建堆:把整个数组原地调整成一个大顶堆,O(n) 完成(自底向上的下滲,详见二叉堆一篇的建堆分析)。
  2. 反复取堆顶:每次把堆顶(当前最大值)与数组末尾未排序区域的最后一个元素交换,堆缩小 1,再对新堆顶做一次下滲(percolate down)恢复堆序。重复 n-1 次,数组从后往前逐渐被填满有序的最大值。

二、走一遍具体例子

数组 [3, 7, 1, 9, 5],建堆后(大顶堆数组表示)变为 [9, 7, 1, 3, 5]:

第1轮:堆顶 9 与末尾 5 交换 → [5, 7, 1, 3, 9],有效堆区缩小为前4个,对 5 下滲恢复堆序 → [7, 5, 1, 3 | 9]
第2轮:堆顶 7 与当前末尾 3 交换 → [3, 5, 1, 7 | 9],下滲恢复 → [5, 3, 1 | 7, 9]
第3轮:堆顶 5 与末尾 1 交换,下滲 → [3, 1 | 5, 7, 9]
第4轮:堆顶 3 与末尾 1 交换 → [1 | 3, 5, 7, 9],剩1个元素天然有序
最终:[1, 3, 5, 7, 9]

注意"|"右侧就是已经排好序的区域,且是从大到小依次确定——这跟归并、快排"从局部有序逐步扩展"的方式不同,堆排序是"每一轮精确确定一个全局最终位置"。

三、代码实现(复用二叉堆的下滲逻辑)

Java · 堆排序
public static void heapSort(int[] a) {
    int n = a.length;
    // 建堆:从最后一个非叶子节点开始,自底向上逐个下滲,O(n)
    for (int i = n / 2 - 1; i >= 0; i--) {
        percolateDown(a, i, n);
    }
    // 每次把堆顶(最大值)换到未排序区末尾,堆缩小1再修复堆序
    for (int end = n - 1; end > 0; end--) {
        int tmp = a[0]; a[0] = a[end]; a[end] = tmp;
        percolateDown(a, 0, end); // 堆区范围收缩到 [0, end)
    }
}

四、为什么是原地排序,而且最坏也是 O(n log n)

原地(in-place):整个过程只在原数组内部交换元素、维护堆结构,不需要像归并排序那样借助等长的辅助数组。空间复杂度只有 O(1)(不算递归下滲用的常数级栈空间)。这是它相对归并排序的最大优势。

最坏 O(n log n):建堆是 O(n),之后 n-1 轮每轮下滲最多 O(log n),总共 O(n log n)——不管输入数据是什么顺序,这个复杂度都不会变差。这正好对比出快速排序的软肋:快排平均 O(n log n) 但最坏会退化到 O(n²)(下一篇讲),而堆排序永远稳定在 O(n log n),不存在"最坏情况"这个隐患。

五、代价:不稳定,且缓存性能不如快排

不稳定。 下滲和交换的过程中,相等 key 的元素可能被交换到堆的不同分支,原始相对顺序无法保证。
缓存局部性差。 堆排序访问数组的模式是"下标 i 与下标 2i+1、2i+2"跳跃访问,这种访问模式在数组较大时会频繁跨越 CPU 缓存行,缓存命中率低。而快速排序的分区过程主要是相邻元素的顺序扫描,缓存局部性更好。这也是为什么实测中随机数据下快排通常比堆排序更快,尽管两者都是 O(n log n) 级别。

对比表

算法最坏时间空间稳定性特点
堆排序O(n log n)O(1)不稳定原地,最坏情况有保证,但缓存局部性差
快速排序O(n²)O(log n)不稳定平均最快,缓存友好,但最坏情况会退化
归并排序O(n log n)O(n)稳定需要额外空间,但稳定且最坏有保证

小结

  • 堆排序 = 建堆 O(n) + n 次"取堆顶、下滲" O(log n)。
  • 原地排序(O(1) 空间),区别于归并排序需要 O(n) 辅助空间。
  • 最坏情况也是 O(n log n),没有快排那种"退化到 O(n²)"的隐患。
  • 代价是不稳定,而且跳跃式数组访问导致缓存命中率低于快排。
