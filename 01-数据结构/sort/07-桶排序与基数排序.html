桶排序 / 基数排序

上一篇证明了比较排序逃不出 Ω(n log n),而这两种排序算法完全不做比较,靠"分配"信息,换来了真正的 O(n)。

📖 来源:《数据结构与算法分析:Java语言描述》 第 7 章「排序」

一、绕开下界的关键:不比较,靠分配

上一篇的结论是:只要排序算法通过"两两比较谁大谁小"来获取信息,就逃不出 Ω(n log n) 的下界。桶排序和基数排序换了一种完全不同的信息获取方式——它们不问"谁比谁大",而是直接利用元素的值本身,把元素"分配"到确定的位置或分组中。既然没有比较这一步,自然不受该下界约束,能做到线性时间 O(n)。

但天下没有免费的午餐:能做到 O(n) 是有代价的——必须提前知道数据的某种结构信息(比如取值范围、位数上限),这是它们相对比较排序算法最大的限制。

二、桶排序:已知范围,分桶再排序

思路:如果已知 n 个元素的值都落在一个不大的范围内(比如 0~99),可以直接开 100 个"桶",遍历一遍数组把每个元素放进对应编号的桶里,再按桶的编号顺序把元素取出来拼接,就得到了有序结果——整个过程只有"分配"和"收集"两次线性扫描,是 O(n)。

具体例子:数组 [3, 1, 4, 1, 5, 9, 2, 6],已知取值范围是 0~9,建 10 个桶(计数数组):
遍历一遍统计每个值出现的次数:count = [0,2,1,1,1,1,0,0,0,1](count[1]=2 表示 1 出现两次,count[9]=1 表示 9 出现一次)
按桶编号 0→9 依次把每个值重复 count[值] 次输出:[1,1,2,3,4,5,6,9]

这个例子里的桶排序退化为"计数排序"(每个桶只存计数,不存具体元素)。更一般的桶排序里,每个桶可以是一个链表/小数组,元素按值映射到桶后,桶内部再用插入排序处理(因为桶足够小,插入排序的 O(n²) 在小规模下反而更快——呼应第一篇讲的插入排序对小数据的优势)。

三、基数排序:按位分配,多轮收集

思路:如果数据是多位数字(或多字符字符串),可以不直接比较整个数字大小,而是按位(个位、十位、百位...)从低位到高位,依次用桶排序(计数排序)对该位分配和收集。因为每一轮都是稳定的桶排序,低位排好的相对顺序在高位排序时会被保留,多轮之后整体就有序了。

具体例子:数组 [170, 45, 75, 90, 802, 24, 2, 66],最大位数是3位(802),从个位开始:
按个位分配收集:170,90,802,2,24,45,75,66 → [170, 90, 802, 2, 24, 45, 75, 66]
按十位分配收集:802,2,24,45,66,170,75,90 → [802, 2, 24, 45, 66, 170, 75, 90]
按百位分配收集:2,24,45,66,75,90,170,802 → [2, 24, 45, 66, 75, 90, 170, 802](有序)

每一轮的分配收集(即对某一位做计数排序)是 O(n+k)(k 是这一位的取值范围,通常是10),总共做 d 轮(d 是最大位数),总复杂度是 O(d·(n+k))——当 d 和 k 都是常数或远小于 n 时,整体近似 O(n)。

四、代码实现:LSD 基数排序骨架

Java · 基数排序(按十进制位,从低位到高位)
public static void radixSort(int[] a) {
    int max = java.util.Arrays.stream(a).max().getAsInt();
    // exp 依次取 1, 10, 100...,决定当前按哪一位分配
    for (int exp = 1; max / exp > 0; exp *= 10) {
        countingSortByDigit(a, exp); // 必须是稳定的计数排序,否则前面轮次排好的顺序会被打乱
    }
}
关键约束:每一轮的子过程(计数排序)必须是稳定的。 基数排序的正确性完全建立在"低位排好的相对顺序,在高位排序时不能被打乱"这一前提上——如果某一轮用了不稳定的排序方式,前面几轮的成果会被破坏,最终结果就是错的。这是实现基数排序最容易忽略、也是面试常问的细节。

五、代价:对数据分布有限制

算法前提条件时间复杂度空间稳定性
桶排序/计数排序已知数据取值范围且范围不大O(n+k)O(n+k)稳定
基数排序已知最大位数,按位/字符分配O(d·(n+k))O(n+k)稳定
比较排序(归并/快排/堆)只需要能比较大小,无需知道分布Ω(n log n)视算法而定视算法而定

如果数据取值范围极大(比如 32 位整数全范围)、或位数极多,桶排序/基数排序反而会因为 k 或 d 过大而变得不划算,这时比较排序更通用。

六、实际应用场景

  • 字符串排序:字典序排序天然适合基数排序,按字符位置从左到右(MSD)或右到左(LSD)分配,常用于外部排序海量字符串数据。
  • 有限范围整数排序:比如按年龄(0~150)、按分数(0~100)排序一大批记录,桶排序/计数排序比通用排序快得多。
  • 数据库/大数据场景:很多列存储引擎对固定长度整数列排序会优先考虑基数排序,配合下一篇讲的外部排序处理超大数据集。

小结

  • 桶排序/基数排序不基于比较,所以能突破 Ω(n log n) 下界,做到 O(n) 或 O(d·n)。
  • 桶排序:已知取值范围,直接分桶计数;基数排序:按位/字符多轮分配收集。
  • 代价:对数据分布有限制(需要知道范围或最大位数),范围/位数过大时不划算。
  • 基数排序每一轮必须用稳定的子排序,否则前面轮次的成果会被破坏。

与其他知识点的联系

  • 哈希表的桶(bucket)概念:桶排序里"按值映射到固定编号的桶"和 HashMap 里"按哈希值对数组长度取模,落到某个桶(链表/红黑树)"是同一种思想——都是牺牲一部分空间,用直接定位取代比较查找。区别是哈希表的映射函数(哈希函数)是为了打散冲突,桶排序的映射函数是为了保持顺序,两者对映射函数的要求正好相反。
  • 插入排序:桶排序里"桶内部再排序"通常选插入排序,不是随便选的——这正好呼应前面讲插入排序时的结论:数据规模小、基本有序时,插入排序的 O(n²) 常数因子小,实际比 O(n log n) 的算法更快。桶排序把大问题拆成很多小问题,天然创造了插入排序发挥优势的场景。
  • 稳定性与多关键字排序:基数排序"先排低位再排高位、每轮必须稳定"的设计,本质上是多关键字排序(先按次要关键字排,再按主要关键字排)的一个特例——这也是 Collections.sort/Comparator.thenComparing 链式比较器背后的排序稳定性要求:只有前一轮排序结果稳定,后一轮按更高优先级排序时才不会打乱同优先级内部已经排好的顺序。
  • 排序下界 Ω(n log n):上一篇证明了这个下界只对"比较排序"成立,桶排序/基数排序是反例的具体实现,说明"下界"不是排序问题本身的下界,而是"只能通过比较获取信息"这一限制下的下界——这是算法分析里"模型假设决定结论边界"的典型案例,面试被问到"O(n log n) 是排序的理论下限吗"时容易踩坑。
  • 外部排序(下一篇):基数排序按位分配、多轮扫描的模式,天然适合处理内存放不下的海量数据——每一轮只需要顺序扫描一遍数据配合外存,不需要像快排那样做随机访问,这也是很多数据库/大数据系统对定长整数列排序优先选基数排序的原因。
