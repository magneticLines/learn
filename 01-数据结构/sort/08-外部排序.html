外部排序 External Sorting(多路归并)进阶

当数据大到内存装不下,前面所有排序算法的前提——"随机访问任意位置都很快"——瞬间失效,排序必须重新设计成"围着磁盘 IO 转"。

📖 来源:《数据结构与算法分析:Java语言描述》 第 7 章「排序」

一、为什么内部排序算法在这里全部失效

前面六篇讲的插入、希尔、堆、归并、快速排序,有一个共同的隐含假设:整个数组能放进内存,随机访问任意下标的代价是均匀的 O(1)。但如果要排序的数据是 100GB 的日志文件,而内存只有 8GB,这个假设直接破产——数据根本装不进内存,更不用说随机跳着访问了。

这时候真正的瓶颈不再是"比较次数"或"CPU 计算量",而是磁盘 IO 次数。磁盘的顺序读写很快,但随机寻道极慢(即使是 SSD,随机访问也远慢于顺序访问)。所以外部排序的设计目标变成:尽可能减少磁盘的随机访问,尽量让每次 IO 都是大块的顺序读写。

二、核心思路:分块内部排序 + 多路归并

外部排序(最常见的是多路归并外部排序)分两个阶段:

  1. 分块排序阶段:把大文件切成若干块,每块大小刚好能放进内存,读入内存后用前面学的任意内部排序算法(通常用快排或归并排序)排好序,再写回磁盘,形成若干个"有序的小文件"。
  2. 多路归并阶段:同时打开这些有序小文件,每个文件只在内存里保留一个"读取指针"指向的元素(不需要整块读入),从所有文件当前指向的元素里选出最小值写到结果文件,该文件指针前移。重复直到所有文件耗尽。

这个过程只对每个文件做顺序读、对结果做顺序写,完全避免了随机访问——这正是它能在磁盘时代高效工作的根本原因。

三、具体例子:8 块内存排 32 块数据

假设内存一次能装 2 个数据块,磁盘上有 8 个数据块要排序:
第一轮(分块排序):依次读入每 1 块到内存排好序,写回磁盘,得到 8 个各自有序的小文件
第二轮(2路归并):每次取 2 个有序小文件,同时顺序读取合并成 1 个更大的有序文件,8 个 → 4 个
第三轮:4 个 → 2 个
第四轮:2 个 → 1 个,排序完成

如果每轮只归并 2 路,总轮数是 log₂(块数);如果一次能同时归并更多路(k 路),轮数降为 log_k(块数),能显著减少磁盘的总读写遍数——这是外部排序里"归并路数越多越好"的直觉来源,但路数也受限于内存能同时容纳多少个文件的读取缓冲区。

四、如何用堆高效实现 k 路归并

多路归并阶段的核心操作是:在 k 个文件当前指向的元素中,反复选出最小值。如果 k 很大(比如 100 路归并),每次都线性扫描 k 个元素找最小值是 O(k),效率不高。这里正好用得上之前学过的优先队列/二叉堆:

维护一个大小为 k 的小顶堆,堆中每个元素记录"来自哪个文件、当前值是多少"。每次从堆顶取出最小值输出,再从该值所属的文件读取下一个元素放入堆中,重新调整堆序(下滲,O(log k))。这样每轮选最小值的代价从 O(k) 降到 O(log k),k 路归并的总代价变成 O(n log k) 而不是 O(n·k)——这就是"优先队列"那几篇学到的堆结构在系统级场景里的直接应用。
Java · k 路归并的堆选择骨架
// 堆中元素:当前值 + 该值来自第几个文件
record Entry(int value, int fileIndex) {}

PriorityQueue<Entry> heap = new PriorityQueue<>(
    Comparator.comparingInt(Entry::value));
// 初始化:每个有序小文件的第一个元素入堆
for (int i = 0; i < k; i++) heap.offer(readNext(i));
while (!heap.isEmpty()) {
    Entry min = heap.poll(); // O(log k) 取出全局最小
    writeOutput(min.value());
    Entry next = readNext(min.fileIndex()); // 从同一文件顺序读下一个,补入堆
    if (next != null) heap.offer(next);
}

五、联系实际场景

  • 数据库:ORDER BY 对超出 work_mem/排序缓冲区大小的结果集,会触发"外部排序"(常称为 disk-based sort 或 spill to disk),原理与本篇一致——分块排序后归并。
  • 大数据处理(MapReduce/Spark):Shuffle 阶段本质上就是外部排序问题的分布式版本——Map 端对每个分区数据排序后落盘,Reduce 端对多个已排序分片做多路归并,思路完全对应本篇的"分块排序+多路归并"。
  • 操作系统层面:任何"数据量远超内存"的排序需求(日志分析、大规模 ETL)都逃不开这个模型,核心永远是用顺序 IO 替代随机 IO,用堆加速多路选择。

小结

  • 数据超过内存容量后,随机访问假设失效,排序必须围绕"减少磁盘 IO、尽量顺序读写"重新设计。
  • 核心思路:分块读入内存用内部排序算法排好序写回磁盘,再对多个有序小文件做多路归并。
  • 多路归并用小顶堆维护每个文件当前元素,每次 O(log k) 选出全局最小值,而不是 O(k) 线性扫描。
  • 数据库排序溢出磁盘、MapReduce/Spark 的 Shuffle 阶段都是这个模型的实际应用。
