消息幂等性与重复消费的处理

上一篇说到"至少一次"语义下消息可能重复投递，这一篇讲怎么在消费端把"重复"这个隐患彻底消化掉。

回顾：消息为什么会重复

在"至少一次"语义下，只要生产者的确认没有及时收到，或者消费者提交位移前发生了重启，系统为了保证"不丢"，宁可选择"重发一次"。 这就好比快递员按门铃后你没及时应门，他就会隔一会儿再来敲一次——为了确保你收到货，代价是你可能会被敲两次门、甚至收到两份一样的包裹（如果他记录混乱的话）。

幂等性（Idempotence）说的就是：同一个操作无论执行一次还是重复执行 N 次，最终产生的效果都完全一样。让消费逻辑具备幂等性，就是解决"消息重复"问题最根本、最通用的办法——既然无法从物理层面完全杜绝重复投递，那就让重复投递"无害化"。

方案一：唯一 ID + 去重表

给每条消息（或者说每个业务事件）分配一个全局唯一的 ID，比如订单支付消息带上 paymentId。消费者处理消息前，先检查这个 ID 是否已经处理过：

伪代码
if (existsInDedupTable(message.id)) {
    // 已经处理过，直接忽略，保证幂等
    return;
}
// 处理业务逻辑 + 插入去重记录，必须在同一个事务里
doBusinessLogic(message);
insertDedupRecord(message.id);
与数据库知识的联系：这正好呼应"09-数据库"里讲过的唯一索引约束。去重表的消息 ID 字段建上唯一索引，即便高并发下出现两个线程同时判断"未处理过"，插入去重记录时数据库也会因为唯一约束冲突而拒绝其中一个，从根本上避免了"判断和写入之间的竞态窗口"导致的重复处理——这比单纯在业务代码里做 if 判断更可靠。

方案二：状态机判断

很多业务实体本身带有状态流转，重复消息落到状态机上时天然就是无效操作。比如订单有"待支付 → 已支付 → 已发货"的状态流转， 当消费者收到"支付成功"的消息时，先查一下订单当前状态：如果订单已经是"已支付"状态，说明这条支付成功的消息之前处理过了，直接忽略这次重复消息即可，不需要再次执行"扣减库存、发送通知"等副作用操作。

这种方式不需要额外的去重表，天然利用业务状态本身的语义去识别重复，但前提是业务状态的流转必须是单向且不可逆的，否则状态判断可能出现二义性。

更底层的保障：Kafka 幂等生产者

上面两种是消费端（业务层面）的幂等处理。而 Kafka 从生产者这一端也做了一层防护，叫幂等生产者（Idempotent Producer），专门解决"网络重试导致 Broker 端出现重复消息"的问题。

它的原理是：每个生产者被分配一个唯一的 Producer ID（PID），发送的每条消息都带上一个递增的序列号（Sequence Number）。 Broker 端为每个 <PID, Partition> 维护一个"最后写入的序列号"，如果收到的消息序列号小于等于已经记录的值，说明这是一次网络重试导致的重复发送，Broker 直接丢弃，不会重复写入日志。

类比：就像给每个快递包裹贴上"发件人 + 流水号"，仓库只要发现流水号已经收过，就知道这是同一个包裹的重复投递，直接退回，不会入库两次。
注意边界：幂等生产者只能保证"生产者到 Broker"这一段单个 Session 内不重复，它无法保证"消费者读到消息、处理完、写下游系统"这个更长链路上的幂等性——生产者重启后 PID 会变化，之前的去重状态也会失效。所以业务层面的幂等处理（方案一、二）依然是不可或缺的，两者是互补关系，不是替代关系。

三种幂等手段对比

方案作用层面解决的重复来源依赖
唯一 ID + 去重表 消费端 / 业务层 任意环节导致的重复消费 数据库唯一索引约束
状态机判断 消费端 / 业务层 任意环节导致的重复消费 业务状态单向流转
Kafka 幂等生产者 生产端 / Broker 网络重试导致的重复写入 PID + 序列号

常见误区

误区："开了 Kafka 的幂等生产者，我的消费逻辑就不用考虑重复消费了。" 幂等生产者解决的是生产者到 Broker 这一段的重复写入，跟消费者会不会重复拉取、重复处理是两回事（比如消费者处理完业务但提交位移失败导致重启后重新消费）。这是很多初学者容易混淆的边界，面试中经常被追问"幂等生产者解决的是哪一段的问题"。
面试高频："如何保证消费端的幂等性？" 标准答题结构：先说明消息重复的原因（至少一次语义下的重试），再给出业务层方案（唯一 ID 去重表、状态机判断），最后提一句 Kafka 自身也提供了幂等 Producer（PID + 序列号）来减少生产端重复，但两者作用层面不同，需要配合使用。

核心价值总结

  • 幂等性是"精确一次"语义落地的关键一环：MQ 只能保证不丢，重复交给消费端来消化。
  • 去重表方案的可靠性来自数据库唯一索引：把"判断 + 写入"的竞态问题交给数据库约束兜底，而不是纯业务代码判断。
  • Kafka 幂等生产者和消费端幂等处理是互补的两道防线：前者管生产端网络重试，后者管整条消费链路，不能互相替代。
