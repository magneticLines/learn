消息队列基础

点对点模式与发布订阅模式

一条消息该被"一个人拿走"还是"所有人都能看到"？这是消息队列的两种根本消费模型，Kafka 用一套机制同时把两者都实现了。

点对点模式（Point-to-Point）：Queue 模型

点对点模式对应传统的队列（Queue）模型：一条消息进入队列后，只会被一个消费者取走并消费，其他消费者拿不到这条消息了。 多个消费者同时监听同一个队列时，它们之间是竞争关系——谁先抢到消息，消息就归谁处理。

类比：就像银行的排号叫号系统。一张号码单只会被一个柜台窗口处理，哪怕同时开着 5 个窗口（5 个消费者），这 5 个窗口也只是在"抢客户"， 不会出现同一张号码单被 5 个窗口同时服务的情况。

这种模式的典型场景是任务分发：比如一批订单需要生成发票，把"生成发票"这个任务扔进队列，起 5 个 worker 进程去抢任务处理， 每个任务只需要被处理一次，多个 worker 的作用是并行分担工作量，而不是每个 worker 都重复处理一遍。

发布订阅模式（Publish-Subscribe）：Topic 模型

发布订阅模式对应主题（Topic）模型：生产者把消息发布到一个 Topic 上，所有订阅了这个 Topic 的消费者都能收到各自完整的一份消息副本。 消费者之间不是竞争关系，而是各自独立地接收广播。

类比：就像电视台直播。一场足球比赛的信号（消息）广播出去，所有打开电视调到这个频道（订阅了这个 Topic）的观众都能同时看到完整的比赛画面， 不会因为你家已经看了，隔壁老王家的电视就收不到信号了。

典型场景是广播通知：比如"订单已创建"这个事件，短信服务、积分服务、物流服务都需要各自独立地收到这条消息并处理自己的逻辑—— 这正好对应上一篇讲的下单场景，一条消息要被多个不同职责的消费者都消费到，而不是被其中一个"抢走"。

Kafka 的巧妙之处：一套机制同时支持两种模式

Kafka 没有分别实现"队列"和"主题"两套 API，而是靠 Topic + 消费者组（Consumer Group）这一套机制，让同一个 Topic 可以同时表现出点对点和发布订阅两种行为：

  • 同一个消费者组内的多个消费者——是点对点的竞争关系。一个 Partition 在同一时刻只会被组内的一个消费者消费，多个消费者是在瓜分这个 Topic 下的所有 Partition，分摊处理压力。
  • 不同消费者组之间——是发布订阅的广播关系。每个消费者组都会各自拿到这个 Topic 的完整消息流，互不影响，就像多个订阅了同一个频道的独立观众。
示意：订单 Topic 被两种消费者组处理
// Topic: order-created，假设有 3 个 Partition

// 消费者组 A：短信服务（3 个实例组成一个组，组内竞争消费 —— 点对点）
group.id = "sms-service-group"
  ├─ consumer-1  // 消费 partition-0
  ├─ consumer-2  // 消费 partition-1
  └─ consumer-3  // 消费 partition-2

// 消费者组 B：积分服务（另一个独立的组 —— 与组 A 是发布订阅关系）
group.id = "point-service-group"
  └─ consumer-1  // 独立消费全部 3 个 partition 的完整消息流

组 A 和组 B 都会收到完整的订单创建消息流（发布订阅），但组 A 内部的 3 个消费者实例是在瓜分这 3 个 Partition、各自只处理其中一部分（点对点）。

面试高频："Kafka 到底是点对点还是发布订阅？" 标准答案是"两者都是，取决于消费者组的划分方式"。 如果所有消费者都属于同一个组，表现为点对点（一条消息只被组内一个人处理）；如果每个消费者都属于不同的组，表现为发布订阅（每个消费者都能收到全部消息）。 这也是 Partition 与消费者组关系这个高频考点的基础，后面 kafka/04-消费者组与消费位移.html 会深入展开。

两种模式对比

维度点对点（Queue）发布订阅（Topic）
一条消息会被几个消费者消费 恰好 1 个 每个订阅者各 1 份，可以有多个订阅者
多个消费者之间的关系 竞争（抢消息） 独立（各自广播接收）
典型场景 任务分发、工作量分摊（如生成发票） 事件广播、多方通知（如订单创建通知多个下游）
Kafka 中的实现 同一消费者组内的多个消费者 多个不同的消费者组
常见误区：不要以为"Kafka 用了 Topic 这个词，所以它只能做发布订阅"。Topic 只是消息的逻辑分类容器， 真正决定消费行为是点对点还是发布订阅的，是消费者组的划分方式，而不是 Topic 本身。同理也不要以为消费者数量越多就消费得越快—— 如果消费者数量超过了 Partition 数量，同一个组内多出来的消费者会分不到任何 Partition，处于空闲状态（下一篇讲 Kafka 架构时会再次遇到这一点）。

核心价值总结

  • 点对点：Queue 模型，一条消息只能被一个消费者拿到，多个消费者是竞争关系，适合任务分发。
  • 发布订阅：Topic 模型，一条消息可以被多个订阅者各自完整地收到，适合广播通知。
  • Kafka 的设计：用"Topic + 消费者组"统一了两种模型——组内竞争（点对点），组间广播（发布订阅），不需要用两套系统。

与其他知识点的联系

  • 消费者组与消费位移（kafka/04）：本篇提到"同一消费者组内竞争、不同消费者组间广播"，这背后依赖的正是每个消费者组独立维护自己的消费位移（offset）——组 A 消费到哪里不影响组 B，位移是隔离两个模式互不干扰的关键机制，kafka/04 会展开讲位移具体怎么存储和提交。
  • Topic 与 Partition 的设计（kafka/01）：文中提到"消费者数量超过 Partition 数量会有消费者空闲"，这个现象的根源是 Kafka 规定一个 Partition 在同一时刻只能被组内一个消费者消费，Partition 数量直接决定了组内点对点竞争的并行度上限，下一篇会具体讲 Partition 数量如何规划。
  • 生产者的分区策略（kafka/03）：点对点模式下"消息分给谁处理"看似是消费者端决定的，但实际上消息先被生产者按 key 或轮询策略分配到某个 Partition，消费者只能从被分配到的 Partition 里取，所以点对点的竞争结果其实早在生产端就被分区策略预先规划了一部分。
  • 死信队列的设计（ops/03）：发布订阅模式下如果某个消费者组一直消费失败，broker 不会替它把消息转移走（Kafka 没有原生 DLQ），这和 RabbitMQ 原生支持的死信队列不同，ops/03 会讲 Kafka 场景下如何自己实现类似死信队列的失败消息兜底方案。
  • 顺序消息如何保证（ops/02）：点对点模式下"一个 Partition 只被一个消费者消费"这条规则，恰好是 Kafka 保证同一 Partition 内消息顺序的必要条件之一——如果同一 Partition 被组内多个消费者并发争抢，顺序就无法保证，ops/02 会详细展开这一点。
  • 消息队列解决的问题（basics/01）：本篇的点对点对应上一篇"任务分发/工作量分摊"的场景（如生成发票），发布订阅对应"广播通知"场景（如下单后通知短信、积分、物流），两种模式本质是在用不同的方式实现 basics/01 里说的"解耦"目标。
