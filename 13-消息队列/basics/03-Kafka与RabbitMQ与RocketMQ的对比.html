Kafka 与 RabbitMQ 与 RocketMQ 的对比

三款主流消息队列的设计哲学完全不同：Kafka 追求极致吞吐、RabbitMQ 追求路由灵活、RocketMQ 追求金融级可靠与高吞吐的平衡。选型前先搞清楚各自的取舍。

Kafka：基于日志的顺序写，为高吞吐而生

Kafka 的核心设计是把每个 Partition 实现成一份只追加写入的日志文件（append-only log）。生产者写消息永远是往文件末尾顺序追加， 消费者读消息也是按顺序从某个位置往后读——这跟机械磁盘的顺序 I/O 性能远高于随机 I/O 是同一个道理，也是 09-数据库 里讲 binlog 时同样用到的"顺序写日志"思想： 不管是 MySQL 的 binlog 还是 Kafka 的 Partition，本质都是"用顺序追加的日志文件换取极致的写入吞吐"。

因为消息落盘后不会被立刻删除，而是按照可配置的保留时间（比如 7 天）或者保留大小滚动清理，Kafka 天然适合日志采集、流处理、事件溯源这类需要海量吞吐、 并且可能需要"消息重放"（消费者从头再消费一遍）的场景。

RabbitMQ：基于 AMQP 协议，路由能力强大

RabbitMQ 实现了 AMQP（Advanced Message Queuing Protocol）协议，核心抽象是 Exchange（交换机）+ Binding（绑定规则）+ Queue（队列）： 生产者把消息发给 Exchange，Exchange 根据路由规则（direct / topic / fanout / headers 等类型）把消息分发到一个或多个 Queue。 这套模型让 RabbitMQ 的路由非常灵活，还原生支持很多开箱即用的高级特性：

  • 延迟队列：消息可以设置延迟多久才被消费者看到，常用于订单超时取消这类场景。
  • 死信队列（DLQ）：消费失败或过期的消息会自动转入专门的死信队列，便于后续排查和补偿处理（这也是 ops 目录里"死信队列设计"专门要讲的主题）。
  • 优先级队列：可以让紧急消息插队先被消费。

代价是 RabbitMQ 的吞吐量相对 Kafka 低一个量级（通常是万级 TPS，而 Kafka 可以做到十万甚至百万级），因为它的每条消息处理涉及更复杂的路由匹配和内存/磁盘管理， 不像 Kafka 那样靠简单的顺序写把吞吐推到极致。

RocketMQ：阿里出品，高吞吐与金融级事务并重

RocketMQ 最初为阿里"双十一"电商场景而生，设计目标是兼顾接近 Kafka 的高吞吐与严格的消息可靠性。 它的一个招牌特性是原生支持事务消息：可以保证"本地事务执行成功"和"消息成功发出"这两件事的最终一致性， 这在电商下单、支付这类要求"扣款和发消息必须同时成功或同时失败"的金融级场景里非常关键。

此外 RocketMQ 是 Java 编写、中文文档和社区都很活跃，国内公司自研或改造 RocketMQ 的成本相对较低，这也是它在国内电商、支付类系统中被广泛采用的原因之一。

面试高频："Kafka 和 RabbitMQ 该怎么选？" 简单记忆：要吞吐选 Kafka，要灵活路由/高级特性选 RabbitMQ，要国内金融级事务消息选 RocketMQ。 更深一层的答案是要说出"为什么"——Kafka 的高吞吐来自顺序写日志放弃了复杂路由能力，RabbitMQ 的路由灵活来自 Exchange 机制但增加了处理开销， 这是一个典型的"吞吐 vs 功能丰富度"的工程取舍，而不是谁绝对更好。
常见误区：不要把"Kafka 吞吐更高"简单理解为"Kafka 各方面都更强"。如果业务需要延迟队列、死信队列这类开箱即用的高级特性， 用 Kafka 自己实现的成本可能远高于直接用 RabbitMQ；同理如果需要严格的分布式事务消息保证，RocketMQ 的原生事务消息比在 Kafka 上自己拼凑事务方案要稳妥得多。 选型永远是"业务场景优先"，不是"参数越猛越好"。

核心价值总结

  • Kafka：顺序写日志换取极致吞吐，适合日志采集、流处理，消息可配置保留支持重放。
  • RabbitMQ：AMQP 协议带来灵活路由，原生支持延迟队列/死信队列，但吞吐相对较低。
  • RocketMQ：兼顾高吞吐与金融级事务消息，国内生态成熟，中文社区活跃。

与其他知识点的联系

  • Topic 与 Partition 的设计（kafka/01）：本篇说 Kafka 的高吞吐来自"顺序写日志"，而顺序写日志具体是怎么组织的——就是下一篇要讲的 Partition：每个 Partition 就是一份独立的 append-only 日志文件，Topic 只是多个 Partition 的逻辑分组。
  • 消息确认机制 ack（reliable/01）：表格里"事务消息"一行提到 RocketMQ/Kafka 都能支持不同程度的可靠性保证，这些保证的最底层机制都是 ack——生产者要等 broker 确认写入成功才算发送成功，消费者要等处理完再提交 ack 才算消费成功，reliable/01 会讲这个确认机制的具体实现。
  • 死信队列的设计（ops/03）：表中提到 RabbitMQ/RocketMQ 原生支持死信队列而 Kafka 需要自己实现，这个差异会直接影响后续 ops/03 讨论"Kafka 场景下如何设计死信队列"时的思路——因为没有现成机制，只能靠业务代码模拟。
  • 消息积压的排查思路（ops/01）：三者"吞吐量"差异（Kafka 十万级+ vs RabbitMQ 万级）意味着达到积压临界点的门槛不同，同样是每秒 5 万条的生产速率，对 Kafka 可能是正常范围，对 RabbitMQ 可能已经远超其处理能力导致持续积压，选型时的吞吐量数字要结合 ops/01 的积压判断标准一起看。
  • Broker 与副本机制（kafka/02）：Kafka 的高吞吐不仅靠顺序写，还依赖多 Broker 分布式承载与副本机制保证可靠性，这是 Kafka 区别于单机队列、能做到"十万级 TPS 且不丢数据"的关键，下一 Tab 会展开讲副本和 ISR 机制。
  • 消息队列解决的问题（basics/01）：不管选 Kafka、RabbitMQ 还是 RocketMQ，最终都是为了实现 basics/01 里的解耦、异步、削峰三大目标，选型只是在"用什么样的机制、付出什么样的代价"去实现这三个目标，目标本身不会因为选型变化。
