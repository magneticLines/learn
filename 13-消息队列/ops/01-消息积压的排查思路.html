消息积压的排查思路

线上最常见的 MQ 故障之一，也是面试高频的场景题：Topic 里堆了几百万条没被消费的消息，你该怎么一步步定位并解决？

什么是消息积压

消息积压（Consumer Lag）指的是生产者写入消息的速度长期大于消费者处理消息的速度，导致未消费的消息越堆越多。 可以类比传送带上的流水线：前面的工人往传送带上放零件的速度，比后面工人组装零件的速度快，零件就会在传送带上越积越多，最终传送带被堆满，甚至影响到上游放置零件的节奏。

衡量积压程度的核心指标是 Lag：某个 Partition 的最新位移（Log End Offset）减去消费者已提交的消费位移（Consumer Offset），Lag 越大说明积压越严重。

常见原因

1. 消费者处理逻辑太慢

消费逻辑里可能包含耗时的同步调用（比如同步调第三方接口、执行复杂计算、没有做批量优化的数据库操作），单条消息的处理耗时过长，自然跟不上生产速度。

2. 消费者数量少于 Partition 数量，并行度不够

Kafka 消费者组内，一个 Partition 同一时刻只能被组内一个消费者实例消费。如果 Topic 有 10 个 Partition，但只启动了 3 个消费者实例，就有 Partition 在"排队等某个消费者忙完再轮到自己"，没能把并行处理能力真正利用起来。

3. 下游依赖成为瓶颈

消费者处理消息经常需要写数据库、调缓存、发 HTTP 请求。如果下游数据库连接池打满、慢查询、外部接口限流，就会拖慢每条消息的处理时间，表现出来的现象仍然是"消费跟不上"，但根子并不在消费者自身的代码逻辑。

排查思路（面试重点）

  1. 先看 Lag 监控，定位是哪个 Topic / 哪些 Partition 在积压。 Kafka 自带 kafka-consumer-groups.sh --describe 命令可以查看每个消费者组、每个 Partition 的 Lag 情况，生产环境通常会接入 Kafka Manager、Burrow 或 Prometheus + Grafana 做持续监控和告警，而不是出问题了才手动查。
  2. 分析消费者的处理耗时，区分是"处理慢"还是"并行度不够"。 看单条消息的平均处理时间和消费者线程的 CPU/IO 使用情况：如果消费者本身很闲（CPU 低、等待多），大概率是并行度不足或者卡在下游调用上；如果消费者一直忙于计算，说明是代码逻辑本身效率的问题。
  3. 临时扩容消费者，或优化处理逻辑，快速止血。 如果是并行度不够，在不超过 Partition 数量的前提下增加消费者实例（超过 Partition 数的消费者会空闲，起不到作用）；如果是处理逻辑慢，可以考虑批量处理、异步化非核心步骤、增加缓存减少下游调用。
  4. 必要时创建新 Topic 做流量迁移，把积压和实时流量分开处理。 如果积压已经非常严重，直接原地加速消费可能会拖慢新消息的实时性。此时可以让消费者暂时切到一个新建的、专门"追赶积压"的临时 Topic 或临时消费者组，把新旧流量分离开，避免"旧消息没处理完、新消息又实时性要求高"两难。
面试高频："生产环境出现消息积压，你会怎么排查？" 按上面的顺序回答：先监控定位、再分析耗时区分瓶颈类型、再对症扩容或优化、必要时做流量迁移。面试官通常想听到的是"有章法的排查步骤"，而不是上来就说"加机器"。

常见误区

误区："消息积压了，直接加消费者实例数量就能解决。" 如果 Partition 数量本身就不够，加再多消费者实例也没用——多出来的消费者实例分不到 Partition，只能空闲等待。这也是为什么 Topic 的 Partition 数量要在设计初期就结合预期吞吐量、消费者扩展性一起规划，事后想临时加 Partition 还会打乱已有消息的 Key 分布规则（见下一篇顺序消息）。

核心价值总结

  • 消息积压的本质是生产、消费速度不匹配：排查的第一步永远是先弄清楚"卡在哪一段"，而不是盲目加资源。
  • Partition 数量决定了消费并行度的天花板：消费者数量超过 Partition 数没有意义。
  • 监控先行：Lag 指标要提前接入告警，而不是等业务方投诉"消息怎么这么久还没处理"才去查。
