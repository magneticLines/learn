死信队列的设计

消息队列模块的最后一站：当一条消息怎么处理都会失败时，总不能让它拖住后面所有正常消息，死信队列就是为此而生的"隔离区"。

问题场景：一条"毒消息"卡住整个 Partition

设想这样一个场景：某条消息的内容格式有问题（比如字段缺失导致反序列化失败），或者它触发了消费逻辑里的一个 Bug，导致每次消费都必然抛异常。 如果消费者的策略是"处理失败就不提交位移、下次重新拉取再处理一次"，这条消息会被无限重试——因为它的失败原因和网络抖动、临时超时完全不同，无论重试多少次结果都一样失败。

更麻烦的是，Kafka 单个 Partition 内消息是严格按顺序消费的，这条消息卡在这里无法被跳过，它后面排队的所有正常消息全部被阻塞，无法被处理。这就像高速公路上一辆抛锚的车堵住了整条车道，后面所有车都动弹不得。

死信队列：给"毒消息"一个隔离区

死信队列（Dead Letter Queue，DLQ）是一个专门用来存放"消费多次失败、确认无法在当前流程中处理"的消息的独立 Topic（或队列）。 它的核心思路是：与其让一条处理不了的消息无限阻塞正常流程，不如设定一个重试上限，超过上限就把它挪到旁边的"隔离区"，让主流程继续往前走。

类比：就像医院的分诊台，遇到当前设备处理不了的疑难病例，不会让它一直占用普通诊室影响其他病人排队，而是转到专门的专家会诊室，由更专业的人力或流程来处理，普通门诊照常运转不受影响。

典型设计：重试 N 次后转发

死信队列并不是 Kafka 内置的强制机制（不同于某些 MQ 如 RabbitMQ 有原生 DLX 支持），在 Kafka 生态里通常由消费端应用逻辑或者流处理框架（如 Spring Kafka）来实现，基本套路如下：

伪代码
int retryCount = getRetryCount(message);
try {
    processMessage(message);
    commitOffset();
} catch (Exception e) {
    if (retryCount < MAX_RETRY) {
        retryLater(message, retryCount + 1);
    } else {
        // 超过重试上限，转发到死信 Topic，不再阻塞主流程
        sendToDeadLetterTopic(message, e);
        commitOffset(); // 主流程位移照常推进
    }
}

转发到死信 Topic 之后，通常会带上失败原因、异常堆栈、重试次数等元信息，方便后续排查。死信 Topic 里的消息一般不会被自动重新处理，而是等待人工介入排查，或者由专门的补偿逻辑（修复数据后手动重放、走另一套兜底流程）来处理。

常见误区

误区："消费失败就应该立刻扔进死信队列，越快隔离越好。" 这个做法会把很多本可以通过重试自愈的临时性失败（比如短暂的网络抖动、下游服务瞬时不可用）也提前判了死刑，导致死信队列里堆积大量其实"重试一下就能成功"的消息，增加了不必要的人工排查负担。合理的做法是先允许有限次数、带退避策略的重试，确认多次重试仍然失败后，再转入死信队列，兼顾自愈能力和隔离效果。
面试高频："如何设计死信队列？" 回答要点：先说明死信队列要解决什么问题（避免毒消息阻塞正常消费流程），再讲典型实现（重试 N 次仍失败则转发到独立的死信 Topic，并保留失败上下文），最后强调死信队列消息通常需要人工或补偿逻辑介入，而不是自动无限重试。

核心价值总结

  • 死信队列本质是一种"故障隔离"手段：让少数处理不了的消息不再拖累大多数正常消息的处理进度。
  • 重试次数上限是关键参数：太小容易误判临时故障为永久故障，太大则起不到及时隔离的效果，需要结合失败类型合理设置。
  • 死信队列不是终点，而是排查的起点：它把"处理不了的消息"变成可观测、可追溯的数据，方便后续人工介入或做数据修复补偿。

与其他知识点的联系

  • 顺序消息如何保证（ops/02）：本篇"毒消息卡住整个 Partition"的问题场景，在顺序消息场景下更加致命——ops/02 里为了保序，同一批相关消息被限定只能落在同一个 Partition 且只能串行处理，一旦其中一条变成毒消息，会同时阻塞该批消息剩余的所有顺序处理，死信队列在这种场景下几乎是唯一的解药。
  • 三种消息传递语义（reliable/02）：死信队列的重试逻辑本质上是"至少一次"语义的一种延伸应用——先靠重试尝试自愈临时故障，重试次数耗尽后才判定为永久失败并转移，这个"重试 N 次"的阈值设计正是在至少一次语义（可能重复但要保证不丢）基础上叠加的一层容错策略。
  • 消息确认机制 ack（reliable/01）：本篇伪代码里"先处理消息，处理失败则重试，成功或转发死信后才提交位移"的逻辑，正是 reliable/01 讲的手动提交 ack 机制的具体应用——如果不理解 ack 提交时机如何影响消息是否会被重新拉取，就无法正确实现死信队列里"重试 N 次后再提交位移"这个关键设计。
  • 消息积压的排查思路（ops/01）：死信队列和消息积压是同一类"消费跟不上"问题的两种极端表现——积压是"慢慢跟不上"，毒消息导致的阻塞是"直接卡死跟不上"，实际排查积压时如果发现某个 Partition 的 Lag 只增不减、消费者一直报同一个异常，很可能就是遇到了本篇讨论的毒消息场景，该做的是隔离而不是继续等待重试。
  • Kafka 与 RabbitMQ 与 RocketMQ 的对比（basics/03）：本篇提到死信队列不是 Kafka 内置的强制机制，需要靠消费端应用逻辑或 Spring Kafka 实现，这与 basics/03 对比表里"RabbitMQ/RocketMQ 原生支持死信队列、Kafka 需要自己实现"的结论完全对应，是选型时需要纳入成本考量的一个具体差异点。
  • 消息队列解决的问题（basics/01）：死信队列的设计思路——"隔离处理不了的部分，让主流程继续往前走"——正是 basics/01 里"解耦"思想在故障处理场景下的再次应用：把异常处理和正常处理解耦开，避免一个环节的故障拖垮整条链路。
