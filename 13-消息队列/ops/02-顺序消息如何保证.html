顺序消息如何保证

Kafka 从来没有承诺过"全局有序"，它只承诺"单个 Partition 内有序"——搞懂这一点边界，顺序消息的设计就变得很直接。

Kafka 只保证单个 Partition 内有序

Kafka 的 Topic 由多个 Partition 组成，每个 Partition 内部是一个严格的、只能追加写入的日志（Log），消息按写入顺序被分配递增的位移（Offset）， 因此单个 Partition 内消息的顺序，天然与生产者写入的顺序一致，消费时读取也严格按位移递增读取。

但一个 Topic 通常有多个 Partition，多个 Partition 之间是完全并行、互不感知的，Kafka 不会跨 Partition 维护任何顺序关系。 如果两条相关的消息被分到了不同的 Partition，谁先被消费完全取决于两个 Partition 各自的处理速度，无法预测先后。

类比：把 Topic 想象成超市里的多条结账队伍（Partition）。同一条队伍里的顾客一定是按排队顺序被结账的（单队列有序），但队伍 A 的第 3 位顾客和队伍 B 的第 1 位顾客谁先结完账，完全取决于两条队伍各自的结账速度，没法保证"跨队伍"的先后关系。

如何让相关消息严格有序：用相同的 Key

Kafka 生产者发送消息时，如果指定了 Key，默认分区器会对 Key 做哈希，相同 Key 的消息永远会被发送到同一个 Partition。 利用这个特性，只要把"需要保证顺序"的这批消息都设置成相同的 Key，就能确保它们进入同一个 Partition，从而获得严格的顺序保证。

比如一个订单从创建、支付、发货到签收会产生多条状态变更消息，只要这些消息都用订单 ID 作为 Key，就能保证同一个订单的状态变更消息一定按发生顺序被消费，不会出现"签收消息先于支付消息被处理"的诡异情况。

伪代码
// 用订单 ID 作为 Key，保证同一订单的消息进入同一 Partition
producer.send(new ProducerRecord<>("order-events", orderId, event));

代价：只能被一个消费者串行处理

要维持顺序，消息必须落在同一个 Partition 里；而 Kafka 的规则是一个 Partition 在同一时刻只能被消费者组内一个消费者实例消费。 这意味着这批"需要保证顺序"的消息，只能被串行处理——无法通过增加消费者数量来并行处理同一个 Partition 内的消息，因为一旦并行，顺序保证就荡然无存了。

取舍本质：顺序性和并行吞吐是一对矛盾。同一个 Key 的消息越多，落到同一个 Partition 处理的压力越大，但又不能靠加消费者来分担，因为那样会破坏顺序。这是设计消息 Key 时必须提前想清楚的问题：不要为了图方便把所有消息都用同一个固定 Key（比如写死成常量），那样所有消息都会挤到同一个 Partition，其他 Partition 全部空闲，等于自己制造了消息积压和热点 Partition。

常见误区

误区："用了 Kafka 就等于消息全局有序。" 这是最常见的误解，Kafka 默认的顺序保证范围仅限于"同一个 Partition"，Topic 整体是不保证顺序的。真正需要顺序的业务，必须主动通过 Key 的设计把相关消息导向同一个 Partition，这是应用层需要主动做的事，不是 Kafka 自动帮你做的。
面试高频："Kafka 如何保证消息顺序？" 标准答案分两层：第一，Kafka 只保证单 Partition 内有序；第二，要让相关消息有序，需要用相同的 Key 把它们路由到同一个 Partition，代价是这些消息只能被一个消费者线程串行消费，牺牲了并行度。

核心价值总结

  • 顺序保证的边界是 Partition，不是 Topic：这是理解 Kafka 顺序性问题的第一前提。
  • Key 是控制消息路由、进而控制顺序的关键旋钮：相同 Key 进同一 Partition，是实现局部有序的唯一途径。
  • 顺序性和并行度是互斥的：设计 Key 时要按"顺序要求的粒度"来划分（比如按订单粒度而非全局粒度），既保证需要顺序的部分有序，又尽量保留整体的并行处理能力。

与其他知识点的联系

  • 生产者的分区策略（kafka/03）：本篇的核心手段"用相同 Key 让消息进同一 Partition"，直接依赖 kafka/03 讲的按 key 哈希取模的分区算法——顺序消息的实现，本质上是对生产者分区策略的一次业务化应用，而不是 Kafka 单独提供的新功能。
  • Topic 与 Partition 的设计（kafka/01）：本篇"顺序保证的边界是 Partition，不是 Topic"这个结论，正是 kafka/01 里"局部有序、全局不保证有序"的直接延伸，理解了 Partition 内部靠 offset 严格递增维持顺序这个机制，才能理解为什么顺序性只能局限在单 Partition 范围内。
  • 消费者组与消费位移（kafka/04）：本篇提到"顺序消息只能被一个消费者串行处理"，背后的约束正是 kafka/04 讲的"一个 Partition 同一时刻只能被组内一个消费者消费"这条硬规则——顺序性和消费并行度的取舍关系，本质上是这条分配规则的必然推论。
  • 消息积压的排查思路（ops/01）：本篇提到"把所有消息用同一个固定 Key 会导致热点 Partition"，这正是 ops/01 里"消费者数量少于并行度不够"这一积压原因的具体成因之一——热点 Partition 会让某一个消费者实例长期超负荷，而其他 Partition 却很空闲，形成局部积压。
  • 三种消息传递语义（reliable/02）：顺序保证解决的是"处理的先后关系"，但不解决"是否重复处理"的问题——如果消费者处理完消息但还没提交位移就宕机重启，即便顺序不变，同一条消息仍可能被重新消费一次，这仍然要靠 reliable/02 讨论的至少一次语义和幂等处理来兜底。
  • 死信队列的设计（ops/03）：如果顺序消息中某一条因为业务 Bug 反复处理失败，因为它卡在同一个 Partition 的队首，会阻塞该 Partition 后面所有需要保序的消息——这正是下一篇死信队列要解决的"毒消息卡住整条队伍"场景，顺序消息的场景下这个问题会更严重，因为无法跳过这条消息去消费后面不相关的消息。
