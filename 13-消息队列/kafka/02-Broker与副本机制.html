Kafka核心概念

Broker 与副本机制

分区解决了吞吐问题，但如果存 Partition 的那台机器宕机了怎么办？副本机制就是 Kafka 用"一份数据存多份"换取高可用的答案，而 ISR 是这套机制里最精妙的取舍设计。

Broker：Kafka 集群里的一个服务节点

Broker 就是运行着 Kafka 服务进程的一台机器（或一个进程实例）。一个 Kafka 集群通常由多个 Broker 组成， 上一篇讲到的 Partition 会被分散存储到不同的 Broker 上——这样既能利用多台机器的磁盘和网络带宽（提升吞吐）， 也为接下来要讲的副本容灾提供了物理基础（副本必须分布在不同 Broker 上才有容灾意义，都放在同一台机器上，机器一挂全没了）。

为什么需要副本：单份数据经不起宕机

如果一个 Partition 只有一份数据存在某个 Broker 上，一旦这台 Broker 宕机（硬件故障、断电、网络分区），这个 Partition 上的数据就直接不可用了， 甚至可能永久丢失。这在生产环境是不可接受的。Kafka 的解法是给每个 Partition 配置多个副本（Replica）， 让同一份数据在多个不同的 Broker 上都保留一份拷贝。

Leader 与 Follower：读写都走 Leader，Follower 负责同步

一个 Partition 的多个副本中，只有一个被选为 Leader，负责处理所有的读写请求； 其余副本都是 Follower，它们唯一的工作是不断从 Leader 拉取最新数据，让自己的数据尽量跟 Leader 保持一致。 生产者和消费者都只与 Leader 打交道，Follower 在正常情况下是"备胎"，不直接对外提供服务。

类比：这就像公司的主数据库和只读从库——所有写操作都打到主库（Leader），从库（Follower）通过复制日志追平主库的数据， 一旦主库挂了，就从追得最紧的那个从库里选一个顶上来当新主库。这跟 09-数据库 里 MySQL 主从复制、依赖 binlog 同步的思路是同一套逻辑， 只是 Kafka 把"复制的内容"换成了它自己的 Partition 日志。
示意：一个 Partition 的 3 副本分布
// Partition: order-created-0，副本因子(replication factor) = 3

Broker-0: order-created-0  [Leader]    // 处理所有读写
Broker-1: order-created-0  [Follower]  // 持续从 Leader 拉取数据同步
Broker-2: order-created-0  [Follower]  // 持续从 Leader 拉取数据同步

ISR：可用性与一致性之间的取舍

ISR（In-Sync Replicas，同步副本集合）是指那些数据进度跟 Leader 保持"足够接近"（没有落后太多）的副本集合， 包括 Leader 自己。Kafka 只会从 ISR 里选举新的 Leader，也只有当消息被 ISR 中的副本都写入成功后（配合生产者的 acks=all 配置）， 才会被认为是"已提交"的安全消息——这一点在 reliable/01-消息确认机制ack.html 会展开讲。

ISR 机制的巧妙之处在于它是一个动态集合：如果某个 Follower 因为网络慢、GC 卡顿等原因，同步进度长期落后于 Leader， Kafka 会把它移出 ISR，避免让选举新 Leader 或判断消息是否安全落盘时被这个"拖后腿"的副本影响； 等它追上进度后，又会被重新加回 ISR。

面试高频："ISR 解决了什么问题？" 核心是在可用性和一致性之间做平衡： 如果要求所有副本（包括慢的）都同步完成才算写入成功，一致性最强，但只要有一个副本卡住，整个 Partition 的写入都会被拖慢，可用性变差； 如果只要求 Leader 自己写完就算成功，可用性最好，但 Leader 一旦宕机，数据可能丢失，一致性没保障。 ISR 是中间方案：只要求"跟得上的那些副本"都写完，跟不上的直接踢出局，不阻塞整体写入，同时保证被选为新 Leader 的副本数据是新鲜的。
常见误区：不要以为副本数越多就一定越安全。副本数越多，需要同步的 Follower 也越多， 在 acks=all 的场景下写入延迟会相应增加（哪怕引入了 ISR 也是如此），而且更多副本意味着占用更多磁盘和网络资源。 副本因子（replication factor）通常设为 3 是工程实践中吞吐、成本、容灾能力三者的一个平衡点，不是越大越好。

核心价值总结

  • Broker：Kafka 集群中的一个服务节点，Partition 分布在不同 Broker 上。
  • 副本机制：每个 Partition 有一个 Leader 负责读写，多个 Follower 负责同步，用"一份数据存多份"换取容灾能力。
  • ISR：动态维护"跟得上 Leader"的副本集合，是 Kafka 在可用性和一致性之间做取舍的关键机制，也是新 Leader 选举和消息是否安全提交的判断依据。

与其他知识点的联系

  • 消息确认机制 ack（reliable/01）：本篇提到"消息被 ISR 中的副本都写入成功后才算已提交"，这个"已提交"的判断标准正是生产者 acks=all 配置要等待的对象——ack 机制是生产者感知"副本是否写成功"的手段，ISR 是判断"写成功"的范围标准，两者是同一个可靠性问题的两面。
  • 三种消息传递语义（reliable/02）：副本机制解决的是"数据不因单机故障丢失"，但这只是可靠性拼图的一块——即使数据在 Leader/Follower 间同步无误，生产者和消费者之间仍可能因为网络重试、ack 超时等原因出现重复或丢失，这正是 reliable/02 要讨论的至少一次/至多一次/精确一次语义要解决的另一层问题。
  • Topic 与 Partition 的设计（kafka/01）：本篇的副本是"依附"在上一篇的 Partition 之上的——每个 Partition 才是被复制的最小单元，副本因子决定了这个 Partition 有几份拷贝，理解 Partition 是物理存储单元，才能理解副本机制到底在复制什么。
  • 消息积压的排查思路（ops/01）：如果 Follower 因为同步慢被踢出 ISR，说明这台 Broker 本身处理能力已经跟不上，这类"个别副本拖后腿"的现象和消费者端积压是同一类"处理能力跟不上生产速率"的问题，只是发生的位置不同（副本同步 vs 消费拉取），ops/01 排查积压时也要考虑 Broker 端是否存在类似瓶颈。
  • 与 09-数据库主从复制的类比：本篇明确提到 Leader/Follower 复制思路和 MySQL 主从复制、binlog 同步是同一套逻辑，理解了关系型数据库如何靠主从复制保证高可用，再看 Kafka 的副本机制会更容易建立直觉，两者都面临"复制延迟""如何选新主"这类共性问题。
  • 消息队列解决的问题（basics/01）：副本机制保证的是"消息队列本身不丢数据"，这是 basics/01 里说的"系统解耦、异步、削峰"三大价值能够安全兑现的前提——如果队列自己都不可靠，削峰填谷攒下的消息随时可能因为单机故障而丢失，前面的价值就无从谈起。
