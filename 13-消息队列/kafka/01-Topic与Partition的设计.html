Kafka核心概念

Topic 与 Partition 的设计

Topic 是给人看的逻辑分类，Partition 才是 Kafka 真正干活的物理单元——理解这一层拆分，才能理解 Kafka 为什么能做到高吞吐、以及"局部有序、全局不保证有序"到底是什么意思。

Topic 是逻辑概念，Partition 才是物理存储单元

Topic（主题）只是一个逻辑上的消息分类名字，比如 order-created、user-login-log。 Kafka 在物理上并不会把一个 Topic 的所有消息塞进一个文件里，而是把它拆分成多个 Partition（分区）， 这些 Partition 可以分布在集群里不同的 Broker（服务节点）上。

类比：Topic 就像快递公司给"华东区包裹"起的一个名字，但真正处理包裹的是分布在上海、杭州、南京的多个分拣站（Partition）。 你把包裹寄给"华东区"这个统一入口，系统会把它分配到某一个具体的分拣站去处理——多个分拣站并行工作，处理能力自然比只有一个分拣站强得多。

为什么要分区：并行度是核心原因

如果一个 Topic 只有一个 Partition，那么无论你部署多少个消费者，同一时刻也只能有一个消费者在读这个 Partition （这一点会在"消费者组"那篇详细讲）。也就是说，消费吞吐量的天花板被"一个 Partition 只能有一个消费者读"这条规则死死限制住了。

把一个 Topic 拆成多个 Partition 后，多个消费者就可以并行地分别读取不同的 Partition，处理吞吐随 Partition 数量近似线性提升。 这正是 Kafka 高吞吐能力的核心来源之一：不是靠单个 Partition 写得有多快，而是靠"分区 + 并行"把总吞吐堆上去。

示意：Topic 拆分到多个 Broker 上的 Partition
// Topic: order-created，分成 3 个 Partition，分布在 3 台 Broker 上

Broker-0:  order-created-0   // partition 0，从 offset 0 开始顺序追加
Broker-1:  order-created-1   // partition 1，从 offset 0 开始顺序追加
Broker-2:  order-created-2   // partition 2，从 offset 0 开始顺序追加

Partition 内部严格有序，用 offset 标识位置

每个 Partition 内部是一个只能追加写入的有序日志，每条消息进来后会被分配一个单调递增的 offset（偏移量）， 表示这条消息在这个 Partition 里的位置。消费者按 offset 从小到大顺序读取，因此单个 Partition 内部的消息顺序是严格保证的—— 先写入的消息 offset 一定更小，一定会被先读到。

order-created-0（Partition 0 的日志文件）
  offset=0 → 消息A（先写入，offset 更小）
  offset=1 → 消息B
  offset=2 → 消息C（最后写入，offset 最大）

Topic 整体不保证全局有序

但如果一个 Topic 有多个 Partition，消息 A、B、C 完全可能被分散到不同的 Partition 上（分区策略下一篇详细讲）， 每个 Partition 各自独立追加、独立编号 offset。这意味着从整个 Topic 的视角看，消息之间的先后顺序是不保证的—— 你无法说"Partition 0 的第一条消息一定比 Partition 1 的第一条消息先发生"。

常见误区（面试高频）：很多初学者以为"Kafka 是消息队列，消息队列不就该是有序的吗"，这是把"单个 Partition 有序"和"整个 Topic 有序"搞混了。 正确理解是：Kafka 只保证单个 Partition 内部有序，不保证跨 Partition 的全局有序。 如果业务需要保证某一类消息的顺序（比如同一个用户的所有操作必须按顺序处理），必须想办法让这些消息落到同一个 Partition里—— 这就是下一篇"生产者的分区策略"里"需要保序的消息要用同一个 key"这条实践经验的由来。

分区数量不是越多越好

分区数直接决定了消费并行度的上限，但也不是无脑开越多越好：每个 Partition 在 Broker 上都会占用文件句柄、内存等资源， Partition 数量过多会增加 Broker 的管理开销，也会拉长 Leader 选举（下一篇讲副本机制时会展开）的耗时。 实践中通常根据"预期的最大消费者并发数"来倒推分区数，而不是一开始就设置成一个很夸张的数字。

核心价值总结

  • Topic 是逻辑分类，Partition 是物理存储和并行处理的基本单元。
  • 分区的意义是并行度：多个 Partition 可以被多个消费者并行消费，是 Kafka 高吞吐的核心来源。
  • 局部有序、全局不保证有序：单个 Partition 内部用 offset 保证严格顺序，但跨 Partition 不保证顺序——这是高频面试考点，也是设计分区 key 策略的出发点。

与其他知识点的联系

  • Broker 与副本机制（kafka/02）：本篇讲的 Partition 只是"数据切成了几份"，但每份数据具体存在哪台机器上、宕机了怎么办，是下一篇 Broker/副本机制要解决的问题——Partition 是切分的逻辑单元，副本是保证这个单元不丢数据的物理手段，两者是"分"与"保"的接力关系。
  • 生产者的分区策略（kafka/03）：本篇提出"单 Partition 有序、跨 Partition 不保证有序"的结论后，直接引出了一个问题——消息具体怎么落到某个 Partition？这就是 kafka/03 要讲的按 key 哈希取模的分区策略，也是本篇说的"需要保序要用同一个 key"这条经验的具体实现方式。
  • 消费者组与消费位移（kafka/04）：本篇提到"一个 Partition 只能被一个消费者读"限制了消费并行度上限，这条规则的完整表述其实是"一个 Partition 同一时刻只能被消费者组内一个消费者消费"，kafka/04 会展开讲消费者组如何瓜分 Partition、以及消费者数超过 Partition 数会导致空闲的现象。
  • 顺序消息如何保证（ops/02）：本篇"局部有序、全局不保证有序"的结论是理解顺序消息保证方案的前提——ops/02 会讲如何利用"同 key 落同 Partition"这个机制，在业务层面实现所需粒度的顺序保证，而不是妄想让 Kafka 全局有序。
  • 点对点模式与发布订阅模式（basics/02）：本篇"消费者数量超过 Partition 数量会有消费者空闲"这个细节，正是 basics/02 里点对点模式下"组内竞争关系"的直接体现——Partition 数量本质上就是点对点竞争的并行度天花板。
  • 消息队列解决的问题（basics/01）：本篇说的"分区是为了并行度、是 Kafka 高吞吐的核心来源"，正是在具体落实 basics/01 里"削峰填谷"这个目标——Partition 数量越多，理论上能并行消化的流量就越大，缓冲和消化突发流量的能力也就越强。
