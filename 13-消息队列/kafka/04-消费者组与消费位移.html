Kafka核心概念

消费者组与消费位移

这是 Kafka 消费端最核心也最容易在面试中被问倒的一篇：消费者组如何分配 Partition、offset 到底记录了什么、以及提交时机如何决定消息会不会丢或被重复消费。

消费者组：一组消费者共同消费一个 Topic

消费者组（Consumer Group）是若干个消费者实例的集合，它们共用同一个 group.id，共同承担消费某个 Topic 全部 Partition 的任务。 Kafka 的分配规则很硬核：一个 Partition 在同一时刻只能被组内的一个消费者消费，不会出现组内两个消费者同时读同一个 Partition 的情况。

示意：Partition 与消费者组内消费者的分配关系
// Topic: order-created，共 3 个 Partition；消费者组 group-A 有 3 个消费者实例

partition-0  ──→  consumer-1
partition-1  ──→  consumer-2
partition-2  ──→  consumer-3

// 如果 group-A 只有 2 个消费者，其中一个消费者要负责 2 个 partition：
partition-0  ──→  consumer-1
partition-1  ──→  consumer-1
partition-2  ──→  consumer-2

// 如果 group-A 有 4 个消费者，多出来的第 4 个消费者没有 partition 可分，处于空闲：
partition-0,1,2  ──→  consumer-1, consumer-2, consumer-3
consumer-4  ──→  空闲，一直分不到 partition
面试高频："消费者数量超过 Partition 数量会怎样？" 答案就是上面这个例子——多出来的消费者会分不到任何 Partition，白白闲置，不会提升任何吞吐。 这也反过来说明了为什么 Partition 数量的规划要考虑"预期的最大消费并发数"：消费的并行度上限就是 Partition 的数量， 这跟上一篇讲分区设计时的"并行度"是同一个结论的另一面。

offset：消费位移，记录"消费到哪里了"

offset（消费位移）记录的是"某个消费者组在某个 Partition 上，已经消费到了哪个位置"。它是三元组 (消费者组, Topic, Partition) 对应一个 offset 值， 而不是消息本身携带的属性——同一条消息，对消费者组 A 来说可能已经被消费过，对消费者组 B 来说可能还没被消费到，因为两个组的 offset 是各自独立维护的。

Kafka 把 offset 保存在一个内部特殊的 Topic __consumer_offsets 里（早期版本是存在 ZooKeeper 中）， 消费者重启或者发生 Partition 重新分配（rebalance）之后，就是靠读取这里记录的 offset，知道该从哪个位置继续往后消费，而不是从头重新消费一遍。

offset 提交时机：自动提交 vs 手动提交

这是决定"消息会不会丢失、会不会被重复消费"的关键。核心问题在于：offset 提交（更新"已消费到哪里"这个记录）和"业务逻辑真正处理完这条消息"这两件事，谁先谁后。

自动提交（enable.auto.commit=true）

Kafka 客户端会按固定间隔（比如每 5 秒）自动把当前读到的 offset 提交上去，提交动作和业务逻辑是否处理完全脱钩。 这就可能出现：消息已经被消费者拉取到本地、offset 已经自动提交了，但业务逻辑还没处理完（或者处理到一半程序崩了）—— 此时 Kafka 认为这条消息已经消费完毕，重启后会从提交过的 offset 之后继续消费，这条本该被处理但没处理完的消息就丢失了。

手动提交（enable.auto.commit=false）

由业务代码自己控制在什么时候调用 commitSync() / commitAsync() 提交 offset。 正确的做法是先完整处理完业务逻辑，再提交 offset：

手动提交的正确顺序
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, String> record : records) {
    processOrder(record.value());  // 1. 先把业务逻辑处理完（比如扣库存、发短信）
}
consumer.commitSync();          // 2. 处理成功后再提交 offset，标记这批消息"确实处理完了"

这样即使业务逻辑处理到一半程序崩溃，因为 offset 还没提交，重启后消费者会从上次提交的 offset 重新拉取， 这条消息会被重新消费一次——用"可能重复消费"换来了"不会丢消息"的保证。

常见误区（面试高频）："手动提交是不是就完全不会有问题了？" 不是。手动提交只是把"丢消息"的风险换成了"重复消费"的风险—— 如果业务逻辑本身不是幂等的（比如"扣库存"这种操作重复执行两次会导致库存被多扣一次），重复消费一样会造成业务错误。 这就是为什么 reliable/03-消息幂等性与重复消费的处理.html 要专门讨论"业务侧如何做幂等"， offset 提交策略只能保证"消息不丢"或"消息不重"其中一种取舍，不能同时保证两者绝对安全，最终还要靠业务逻辑的幂等设计兜底。

提交时机对比

策略提交时机崩溃后果适用场景
自动提交 固定时间间隔，与业务处理是否完成无关 可能丢消息（提交了但没处理完） 对丢消息容忍度较高、追求简单的场景
手动提交 业务逻辑处理成功之后 可能重复消费（处理完但没来得及提交） 对丢消息零容忍，业务能做幂等处理的场景

核心价值总结

  • 消费者组：组内多个消费者共同瓜分一个 Topic 的所有 Partition，一个 Partition 同一时刻只能被组内一个消费者消费——消费者数超过 Partition 数会导致多余消费者空闲。
  • offset：记录"某个消费者组在某个 Partition 上消费到哪里"，按 (组, Topic, Partition) 维度独立维护。
  • 提交时机决定丢失/重复：自动提交可能丢消息，手动提交（处理完再提交）可能重复消费，两者都需要业务逻辑的幂等设计来兜底最终的正确性。
