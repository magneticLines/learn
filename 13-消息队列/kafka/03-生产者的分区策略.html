Kafka核心概念

生产者的分区策略

一条消息发出去，Kafka 到底把它放进哪个 Partition？这个问题的答案直接决定了消息能不能保序、生产者的吞吐能不能打满——是这一篇要讲透的核心考点。

问题的起点：一条消息该进哪个 Partition

上一篇讲到 Topic 会被拆成多个 Partition，那生产者发送一条消息时，Kafka 客户端要决定这条消息具体落到哪一个 Partition 里。 这个决策逻辑就是分区策略（Partitioner），Kafka 默认按"是否指定了 key"分成两种情况处理。

指定 key：按 key 哈希，保证同 key 消息进同一分区

如果生产者发消息时指定了 key（比如用户 ID、订单 ID），Kafka 默认会对这个 key 做哈希运算， 然后对 Partition 数量取模，用结果决定消息进入哪个 Partition。同样的 key，只要 Partition 数量不变，哈希结果就是固定的， 也就是说同一个 key 的所有消息，一定会被发送到同一个 Partition。

示意：按 key 哈希分区
// 伪代码：默认分区器的核心逻辑
int partition = hash(key) % numPartitions;

// 同一个 userId 作为 key，消息永远落到同一个 partition
producer.send(new ProducerRecord<>("order-created", "user-1001", orderMsg));

结合上一篇讲的"单个 Partition 内部严格有序"，这就是需要保序的消息要用同一个 key这条实践经验的原理： 比如同一个用户的所有订单状态变更消息，只要都用 userId 作为 key，就能保证它们全部进入同一个 Partition， 从而在这个用户的维度上保持严格的先后顺序——这正是解决"Kafka 不保证全局有序"这个局限性的标准做法。

不指定 key：轮询或黏性分区

如果生产者发消息时没有指定 key（比如日志采集这类不关心顺序、只关心吞吐的场景），Kafka 早期版本采用轮询（Round-Robin）策略， 依次把消息分发到各个 Partition，尽量让每个 Partition 的负载均衡。

但轮询有个副作用：生产者客户端为了提升网络利用率，会把发往同一个 Partition 的多条消息攒成一个批次（batch）再一起发送。 如果每条消息都轮询到不同的 Partition，批次就很难攒起来，导致每个批次都很小、发送次数变多，反而降低了吞吐。 因此 Kafka 后来引入了黏性分区（Sticky Partitioning）：在一个批次还没被发送出去之前，尽量把无 key 的消息都黏在同一个 Partition 上， 凑够一个批次或者达到超时时间后才切换到下一个 Partition。

为什么叫"黏性"：可以理解成分区器有一点"惰性"——不是每条消息都重新随机选一个 Partition，而是像抹了胶水一样黏在当前 Partition 上， 直到当前批次凑满或超时才"挪窝"，这样能显著减少批次碎片化，提高吞吐。

两种策略对比

场景分区策略能否保序适用场景
指定 key 对 key 哈希后取模 同 key 消息在同一 Partition 内有序 需要按业务维度（用户、订单）保序的场景
不指定 key 轮询 / 黏性分区 不保证任何顺序 只关心吞吐、不关心顺序的场景（如日志采集）
常见误区（面试高频）：不要以为"指定了 key 就等于消息整体有序"。指定 key 只能保证同一个 key 的消息在同一个 Partition 内部有序， 不同 key 的消息完全可能落在不同 Partition，彼此之间不存在顺序关系。如果误以为随便指定个 key 就能让整个 Topic 有序， 在需要严格全局顺序的场景（比如金融流水）就会踩坑——这类场景要么把 Topic 只设一个 Partition（牺牲并行度换全局有序），要么在业务上重新设计有序性的粒度。
另一个陷阱：Partition 数量一旦扩容，同一个 key 的哈希取模结果会发生变化，导致新消息可能落到跟旧消息不同的 Partition 上， 历史上按同一个 key 保证的顺序性会被打破。这是生产环境扩容 Partition 时必须提前评估的风险点。

核心价值总结

  • 指定 key：按 key 哈希取模决定 Partition，同一个 key 的消息总是进入同一个 Partition，从而在该 key 维度上保序。
  • 不指定 key：轮询或黏性分区，黏性分区通过减少批次碎片提升吞吐，但完全不保证顺序。
  • 实践经验：需要保序的消息一定要用同一个 key，同时要意识到 Partition 扩容可能打破已有的 key 分布，进而影响顺序性。

与其他知识点的联系

  • 顺序消息如何保证（ops/02）：本篇是顺序消息保证方案的技术基础——ops/02 讲的"如何让同一类业务消息严格有序"，落地手段就是本篇的"按 key 哈希分区"策略：只要保证需要保序的消息用同一个 key，就能利用 Kafka 单 Partition 内部有序的特性，把顺序保证从"整个 Topic"缩小到"某个业务维度"这个可控的粒度。
  • Topic 与 Partition 的设计（kafka/01）：本篇直接承接上一篇"局部有序、全局不保证有序"的结论，把"怎么让消息落到指定 Partition"这件事讲清楚——分区策略是连接 Topic/Partition 静态设计和实际消息分布的桥梁。
  • 消费者组与消费位移（kafka/04）：生产者的分区策略决定了消息进入哪个 Partition，而消费者组的分配规则决定了这个 Partition 由哪个消费者实例处理——两者合在一起才能回答"这条消息最终被谁、按什么顺序处理"这个完整链路，本篇的黏性分区还会影响消费者端拉取到的批次大小和处理节奏。
  • 三种消息传递语义（reliable/02）：本篇提到 Partition 扩容会打破按 key 哈希的既有分布，导致历史顺序性被破坏，这类边界场景和 reliable/02 讨论的"精确一次语义在 rebalance/扩容时如何保证"是同一类问题——分布式系统的拓扑变化（扩容、rebalance）往往是打破各种"看似稳定"的保证的高发点。
  • Broker 与副本机制（kafka/02）：分区策略决定消息落到哪个 Partition 的 Leader，而这个 Leader 所在的 Broker 是否稳定、副本是否同步及时，直接影响生产者发送这条消息的延迟和成功率，分区策略和副本机制共同决定了单条消息从产生到落盘的完整路径。
  • 点对点模式与发布订阅模式（basics/02）：本篇分区策略决定的"消息去哪个 Partition"，其实早于 basics/02 里"消费者组内谁抢到这条消息"的竞争过程——点对点模式下的竞争结果，很大程度上是生产端分区策略预先划定好的范围，而不是消费者端随机抢到的。
