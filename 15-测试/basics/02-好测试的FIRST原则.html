单元测试基础

好测试的 FIRST 原则

能编译、能跑通不代表这是一个"好"的单元测试——FIRST 五个字母是判断测试质量的最基础标准，违反任何一条都会让测试逐渐失去价值，最后被团队放弃维护。

为什么需要一套标准

写测试和写产品代码一样，也有好坏之分。一个能跑、断言也没写错的测试，如果跑得很慢、依赖执行顺序、或者依赖外部网络，早晚会被团队嫌弃到弃用。 FIRST 是业界总结出的五条判断标准：Fast（快速）、Independent（独立）、Repeatable（可重复）、Self-Validating（自验证）、Timely（及时）。

Fast：快速

单元测试要能被频繁地运行——理想情况下每次改完代码就能跑一遍，甚至保存文件就自动触发。如果跑一次测试套件要几分钟甚至更久，开发者会本能地减少运行频率，测试失去了"及早发现问题"的意义。

反例：测试里做了真实的网络/IO 调用，跑起来很慢
@Test
void fetchUserInfo_validId_returnsUser() {
    // 真实调用远程API，一次几百毫秒，测试多了跑几分钟
    User user = new RealUserApiClient().fetch(123);
    assertNotNull(user);
}

正确做法是把真实的网络依赖替换成 Mock（见 mock/01、mock/02），让测试只验证业务逻辑本身。

Independent：独立

测试之间不应该互相依赖，也不应该依赖固定的执行顺序。一个测试的失败不应该连带影响其他测试的结果，否则排查问题时会异常痛苦——你不知道是这个测试本身的问题，还是被前一个测试"连累"的。

反例：后一个测试依赖前一个测试留下的状态
private static List<String> sharedList = new ArrayList<>();

@Test
void test1_addItem() {
    sharedList.add("a");
    assertEquals(1, sharedList.size());
}

@Test
void test2_removeItem() {
    // 假设 test1 先跑过，列表里已经有 "a"，这里才能删除成功
    sharedList.remove("a");
    assertTrue(sharedList.isEmpty());
}

如果测试框架换个顺序执行（JUnit5 默认就不保证声明顺序），test2 就会失败。每个测试都应该自己准备好所需的初始状态，互不依赖。

Repeatable：可重复

在任何环境、任何时间运行，测试结果都应该一致——不依赖当前系统时间、不依赖外部网络是否可用、不依赖随机数、不依赖某台机器特有的环境变量。

反例：依赖当前系统时间，结果不稳定
@Test
void isExpired_returnsFalse() {
    Coupon coupon = new Coupon(LocalDate.of(2026, 12, 31));
    // 今天测试通过，但半年后这个用例会突然开始失败
    assertFalse(coupon.isExpired());
}

正确做法是把"当前时间"作为可注入的依赖（比如传入 Clock 对象），测试时用固定时间点，结果才不会随时间推移悄悄变化。

Self-Validating：自验证

测试要能自己判断"通过"还是"失败"，不需要人工去看控制台日志、肉眼比对输出结果。断言就是自验证的核心手段——测试框架根据断言是否成立，直接给出明确的红/绿结论。

反例：只打印结果，靠人肉眼判断
@Test
void calculateTotal_test() {
    double result = calculator.calculateTotal(items);
    System.out.println("计算结果是: " + result); // 到底对不对，得自己算一遍再对比
}

没有断言的测试，本质上不是测试，只是一段"顺便跑一下"的代码——它无法被自动化流水线拿来当作质量门禁。

Timely：及时

测试要及时编写，理想情况是紧跟着产品代码、甚至先于产品代码编写（这正是 TDD 的思路，见 tdd/01）。如果拖到项目上线前才"统一补测试"，往往会出现两个问题：一是代码已经耦合度很高，测试很难写（呼应 basics/01 讲的设计反馈）；二是补测试的动力和优先级永远比不过新需求，最后不了了之。

常见反例："先快速把功能做完上线，测试后面再补。" 现实中"后面"往往永远不会到来——需求一个接一个排队，补测试的任务优先级总是最低。而且此时代码可能已经写成了难以测试的耦合结构，补测试的成本比一开始就写测试高出许多倍。

五条原则一览

原则要求违反的典型后果
Fast运行要快，能被频繁执行开发者减少运行频率，问题发现变晚
Independent测试之间互不依赖，不依赖执行顺序一个测试失败牵连一片，排查困难
Repeatable任何环境、任何时间结果一致出现"我这里能过、CI 上过不了"的诡异问题
Self-Validating测试自己判断通过/失败，不需人工看结果无法接入自动化 CI，失去质量门禁的意义
Timely及时编写，最好紧跟或先于产品代码代码耦合度已经很高，补测试成本暴涨
面试高频：FIRST 原则是考察"你是否真正理解好测试的标准"的常见问题，回答时不要只背首字母，最好能像本文一样各举一个违反原则的反例，说明后果。

核心价值总结

  • FIRST 不是死板的规则，而是"什么会让测试失去价值"的经验总结。
  • Fast + Repeatable 保证测试能被高频、稳定地运行，是测试能融入日常开发流程的前提。
  • Independent + Self-Validating 保证测试出问题时能快速定位，不需要额外的人工介入。
  • Timely 提醒我们测试和产品代码应该同步演进，而不是事后补作业。
