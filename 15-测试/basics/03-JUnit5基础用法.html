单元测试基础

JUnit5 基础用法

JUnit5 不只是给 JUnit4 加了几个新注解，而是把整个测试框架拆成了三层架构——理解这个架构变化，才能理解为什么 JUnit5 能同时兼容 JUnit3/4 的老测试，又能支持 TestNG 等第三方引擎。

JUnit5 的三层架构：Jupiter / Platform / Vintage

JUnit4 是一个单体框架：测试的编写方式、发现方式、执行方式全部耦合在一个 jar 包里。这带来一个问题——想扩展一种新的测试风格（比如参数化测试、嵌套测试），就得往这个单体框架里硬塞，久而久之框架越来越臃肿。

JUnit5 从架构上解决了这个问题，拆成三个独立模块：

JUnit Platform（运行时基座，定义 TestEngine 接口，负责发现、调度测试）
  ├─ JUnit Jupiter（新的编程模型 + 扩展模型，我们写的 @Test 都来自这里）
  └─ JUnit Vintage（兼容层，让 JUnit3/4 写的老测试也能在 Platform 上跑）

这种分层带来的好处是模块化：IDE、构建工具（Maven/Gradle）只需要对接 Platform 这一层，就能同时运行 Jupiter 写的新测试和 Vintage 兼容的老测试，甚至第三方测试框架（如 Spock）只要实现 TestEngine 接口也能接入同一套运行体系。这也是为什么老项目从 JUnit4 迁移到 JUnit5 时可以"新老共存、逐步迁移"，而不需要一次性推翻重写。

生命周期注解与执行顺序

JUnit5 用一组注解描述测试方法在什么时机运行：

生命周期注解演示
class CalculatorTest {

    @BeforeAll
    static void initOnce() {
        // 整个测试类只执行一次，常用于初始化重量级共享资源
        System.out.println("[BeforeAll] 类级别初始化");
    }

    @BeforeEach
    void setUp() {
        // 每个 @Test 方法执行前都会跑一次，通常用来准备"新鲜"的测试对象
        System.out.println("[BeforeEach] 准备测试数据");
    }

    @Test
    void add_twoPositiveNumbers_returnsSum() {
        assertEquals(5, 2 + 3);
    }

    @Test
    void divide_byZero_throwsException() {
        assertThrows(ArithmeticException.class, () -> 1 / 0);
    }

    @AfterEach
    void tearDown() {
        // 每个 @Test 方法执行后都会跑一次，通常用来清理资源
        System.out.println("[AfterEach] 清理");
    }

    @AfterAll
    static void tearDownOnce() {
        // 整个测试类只执行一次
        System.out.println("[AfterAll] 类级别清理");
    }
}

这个类跑起来时的实际顺序是：BeforeAll → (BeforeEach → 某个 @Test → AfterEach) → (BeforeEach → 另一个 @Test → AfterEach) → AfterAll。 也就是说 BeforeEach/AfterEach 会围绕每一个测试方法各跑一次，形成"新鲜环境"——这正是呼应上一篇 FIRST 原则里 Independent（独立）的实现手段：每个测试都从 @BeforeEach 里拿到一个全新初始化的对象，不会受到其他测试残留状态的影响。

常见误区：@BeforeAll/@AfterAll 修饰的方法默认必须是 static 的（因为它们在所有测试实例创建之前就要执行）。如果测试类使用了 @TestInstance(Lifecycle.PER_CLASS) 则可以去掉 static，但这是特例，不是默认行为。

@ParameterizedTest：参数化测试

很多时候我们想用不同的输入反复验证同一段逻辑，如果每组输入都写一个 @Test 方法，会有大量重复代码。JUnit5 的 @ParameterizedTest 解决了这个问题：

参数化测试示例
@ParameterizedTest
@ValueSource(ints = {2, 4, 6, 100})
void isEven_evenNumbers_returnsTrue(int number) {
    assertTrue(NumberUtils.isEven(number));
}

@ParameterizedTest
@CsvSource({
    "2, 3, 5",
    "10, -2, 8",
    "0, 0, 0"
})
void add_variousInputs_returnsExpectedSum(int a, int b, int expected) {
    assertEquals(expected, calculator.add(a, b));
}

一个 @ParameterizedTest 方法会针对每一组参数各自独立执行一次（在测试报告里也会分别显示），既减少了重复代码，又不会牺牲"每组输入独立验证"的清晰度。

JUnit4 vs JUnit5 对比

维度JUnit4JUnit5
架构单体框架，一个 jar 包搞定一切Platform / Jupiter / Vintage 三层，模块化
常用注解@Before / @After / @BeforeClass / @AfterClass@BeforeEach / @AfterEach / @BeforeAll / @AfterAll（命名更直观）
断言异常@Test(expected = XxxException.class)assertThrows(XxxException.class, () -> ...)，能拿到异常对象继续断言
参数化测试需要额外的 Runner（如 @RunWith(Parameterized.class)），用法繁琐内置 @ParameterizedTest，配合 @ValueSource/@CsvSource 等更灵活
Java 版本要求支持 Java 5+要求 Java 8+，充分利用 Lambda 和 Stream
扩展模型@RunWith，一个测试类只能指定一个 Runner，扩展能力受限@ExtendWith，支持多个扩展组合叠加，更灵活
面试高频："JUnit5 相比 JUnit4 有什么架构上的变化？" 不要只回答"注解名字变了"，重点要答出三层架构（Platform/Jupiter/Vintage）带来的模块化和可扩展性，这才是设计层面的核心变化。

核心价值总结

  • 三层架构让 JUnit5 既能兼容老测试，又能灵活扩展新的测试引擎，这是它相比 JUnit4 最大的设计升级。
  • BeforeEach/AfterEach 保证每个测试方法拿到独立、新鲜的环境，是实现 FIRST 原则中 Independent 的具体手段。
  • @ParameterizedTest 用一份测试逻辑覆盖多组输入，减少重复代码的同时保持每组用例独立可读。

与其他知识点的联系

  • 好测试的 FIRST 原则：@BeforeEach/@AfterEach 不只是两个生命周期注解，它们是 FIRST 原则中 Independent 这一条能被真正落地的机制保障——没有这套注解，"每个测试方法拿到全新环境"就只能靠开发者手动、易错地在每个方法开头重复初始化代码。
  • 断言与测试用例的组织方式：本文的 assertEquals/assertThrows 只是断言 API 的最小子集，下一篇会展开讲 AAA 结构、命名规范、AssertJ 流式断言——JUnit5 提供的是"骨架"（生命周期、扩展模型），断言的写法和组织方式才是决定测试可读性的"血肉"。
  • Mockito 实战： Mockito 的 @ExtendWith(MockitoExtension.class) 正是利用了 JUnit5 的 @ExtendWith 扩展模型——JUnit5 把 JUnit4 单一的 @RunWith 换成了可叠加组合的 @ExtendWith，这才让 Mockito 这类第三方框架能以"扩展"的形式无侵入地接入测试生命周期。
  • 红绿重构循环：TDD 循环里"红"阶段依赖的正是 JUnit5 能清晰报告"测试失败"这个信号——生命周期注解、断言机制共同构成了红绿重构能够快速反馈的基础设施，没有一个可靠的测试框架，"小步快跑"就无从验证每一步是否真的成功。
  • 测试覆盖率的意义与常见误区：参数化测试 @ParameterizedTest 能用一份逻辑跑多组输入，看起来能显著"拉高"覆盖率数字，但覆盖率高不代表每组输入的断言都严谨——这也提醒我们不要把"用参数化测试覆盖更多分支"和"测试质量更好"简单等同起来。
