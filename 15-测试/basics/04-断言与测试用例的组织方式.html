断言与测试用例的组织方式

会用 assertEquals 只是第一步，真正决定一个测试套件"半年后还有没有人看得懂"的，是断言的可读性、命名规范和代码结构——这些看似"风格问题"的细节，实际上决定了测试能不能长期发挥活文档的价值。

基础断言方法

JUnit5 的断言都在 org.junit.jupiter.api.Assertions 静态类里，最常用的几个：

常用断言
assertEquals(expected, actual);          // 值相等
assertTrue(condition);                    // 条件为真
assertFalse(condition);                   // 条件为假
assertNull(obj);                          // 对象为 null
assertNotNull(obj);                       // 对象不为 null
assertThrows(IllegalArgumentException.class,
    () -> validator.validate(null));    // 断言抛出指定异常，并返回该异常对象供继续断言
assertAll(                                // 组合多个断言，全部执行完再统一报告失败项，而不是遇到第一个失败就中断
    () -> assertEquals("Tom", user.getName()),
    () -> assertEquals(18, user.getAge())
);
常见误区：assertEquals(expected, actual) 参数顺序是"期望值在前，实际值在后"，写反了不影响测试通过与否，但失败时的报错信息会是"expected: <实际值> but was: <期望值>"，反而误导排查方向。

AssertJ：更可读的流式断言

原生断言方法名和参数堆在一起，读起来像"程序在说话"而不是"人在说话"。AssertJ 这类流式断言库（fluent assertions）通过链式调用，让断言更接近自然语言：

原生断言 vs AssertJ
// 原生断言：需要多次调用，可读性一般
assertTrue(list.size() == 3);
assertTrue(list.contains("apple"));
assertFalse(list.isEmpty());

// AssertJ：链式调用，读起来接近自然语言
assertThat(list)
    .hasSize(3)
    .contains("apple")
    .isNotEmpty();

AssertJ 的另一个优势是失败信息更详细：原生 assertTrue(list.contains("apple")) 失败时只会告诉你"期望 true，实际 false"，而 AssertJ 会直接打印出"期望列表包含 apple，但实际列表内容是 [banana, orange]"，排查问题时省去了额外加日志的功夫。

测试方法命名规范

一个好的测试方法名，应该在不看方法体的情况下就能读出"测试的是什么场景、期望什么结果"。业界常见的一种命名规范是：

方法名_场景_预期结果
例如：withdraw_amountExceedsBalance_throwsInsufficientFundsException
命名对比
// 反例：看名字完全不知道测的是什么
@Test
void test1() { ... }

@Test
void testWithdraw() { ... } // 只知道测的是 withdraw，不知道具体场景和预期

// 正例：方法名_场景_预期结果，一目了然
@Test
void withdraw_amountExceedsBalance_throwsInsufficientFundsException() { ... }

@Test
void withdraw_amountWithinBalance_decreasesBalanceByAmount() { ... }

这不仅方便阅读测试报告（CI 跑挂了，从方法名就能猜到大概是哪种场景出了问题），也间接呼应了 basics/01 讲的"活文档"价值——好的命名本身就是行为说明书。

AAA 模式：Arrange-Act-Assert

测试方法内部的代码结构也有讲究。AAA（Arrange-Act-Assert，准备-执行-断言）模式建议把测试方法分成三段，用空行隔开，各司其职：

AAA 模式示例
@Test
void withdraw_amountExceedsBalance_throwsInsufficientFundsException() {
    // Arrange：准备测试所需的对象和前置状态
    Account account = new Account(100);

    // Act：执行被测试的行为
    Executable action = () -> account.withdraw(200);

    // Assert：验证结果是否符合预期
    assertThrows(InsufficientFundsException.class, action);
}

这个结构的价值在于强制你的测试只做一件事：如果发现 Act 阶段要调用好几个方法、Assert 阶段要断言一大堆东西，往往说明这个测试试图验证太多场景，应该拆成多个更小的测试方法。三段式结构也让阅读者能快速定位"出问题的到底是准备阶段的数据不对，还是断言写错了"。

类比：AAA 就像做菜的三个步骤——备菜（Arrange）、下锅炒（Act）、尝味道打分（Assert）。如果一份"菜谱"里备菜、炒菜、尝味道的步骤全部混在一起写，别人照着做时会很难分清哪步是哪步；分开写，每一步职责清晰，也方便别人照着改。

核心价值总结

  • 断言方法的选择：原生 JUnit5 断言够用，但 AssertJ 这类流式断言库能显著提升可读性和失败信息的详细程度，值得在正式项目中引入。
  • 命名规范："方法名_场景_预期结果"让测试方法名本身就是一份行为说明，呼应"测试即活文档"的价值。
  • AAA 模式用固定的三段结构组织测试代码，强制每个测试聚焦单一场景，也让阅读和排查故障更高效。
  • 这三者结合起来，才能让单元测试基础 Tab 讲的"设计反馈、回归保护网、活文档"三大价值真正落地，而不只是停留在"能跑通"的层面。

与其他知识点的联系

  • 好测试的 FIRST 原则：Self-Validating 这一条要求测试必须能自己判断通过/失败——本文强调"没有断言的测试不是测试"，其实就是 Self-Validating 的具体展开：断言是让测试从"跑一遍看看"变成"自动判定红绿"的唯一手段。
  • Mock、Stub、Spy 的区别：本文讲的状态验证（assertEquals 断言返回值）和下一篇要讲的行为验证（verify() 断言调用情况）是一体两面——AAA 模式里的 Assert 阶段既可以是普通断言，也可以是对 Mock 的行为验证，两者都属于"验证结果是否符合预期"，只是验证的对象不同（返回值 vs 交互过程）。
  • Mockito 实战：Mockito 示例里的测试同样遵循 AAA 结构（Arrange 准备 Mock 和数据、Act 调用被测方法、Assert 结合 assertEquals 和 verify），说明 AAA 不是断言库特有的写法，而是所有测试通用的组织原则，Mock 场景下依然适用。
  • 红绿重构循环："方法名_场景_预期结果"的命名规范在 TDD 里价值更大——红阶段写测试时，方法名本身就是在描述"我期望实现出什么行为"，相当于用命名先把需求写成一句可执行的话，再去填充 Arrange/Act/Assert 的细节。
