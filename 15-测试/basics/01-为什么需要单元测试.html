单元测试基础

为什么需要单元测试

把单元测试仅仅理解为"找bug的工具"，是最常见也最可惜的误区——它真正的价值，在于改变你写代码时的设计决策和改需求时的心理状态。

误区：单元测试只是用来"找bug"的

很多人接触单元测试的第一反应是："我自己测过了，代码没问题，为什么还要写测试？" 这句话背后的假设是：测试的唯一作用是发现缺陷。 如果只从这个角度看，单元测试确实性价比不高——手动跑一遍、断点调试一下，好像也能达到同样效果，还更省事。

但这种理解遗漏了单元测试最重要的三个价值：设计反馈、回归保护网、活文档。找bug只是这三者的副产品。

价值一：设计反馈——难写的测试在报警

这是最容易被忽视、也是资深工程师最看重的一点：如果一段代码的单元测试很难写，往往说明这段代码的设计本身有问题，通常是耦合度过高。

反例：耦合度高，几乎没法单独测试
public class OrderService {
    public void placeOrder(Order order) {
        // 直接 new 出数据库连接、支付网关、短信服务
        Database db = new Database("jdbc:mysql://prod-server/db");
        db.save(order);
        PaymentGateway gateway = new PaymentGateway("real-api-key");
        gateway.charge(order.getAmount());
        new SmsSender().send(order.getPhone(), "下单成功");
    }
}

要给 placeOrder 写单元测试，你会发现测试会真实连接数据库、真实调用支付接口、真的发一条短信——这已经不是"单元"测试了，而是又慢又危险的集成测试。 这种"写不出好测试"的窘境，本质上是在提醒你：这段代码把业务逻辑和外部依赖硬编码在了一起。当你被迫为了可测试性把依赖改成构造器传入时（详见 mock/03 依赖注入一篇），代码的耦合度也随之下降了——测试在倒逼你写出更好的设计。

类比：这就像装修时想把某件家具单独搬出房间擦洗，结果发现它被钉死在墙上、又连着水管和电线，动一下牵连一片——这时"没法单独搬出来"这件事本身就是在告诉你，当初的安装方式有问题。

价值二：回归保护网——改需求时的心理负担

软件不是一次性写完就不动了，需求会一直变，代码会一直被重构。没有测试保护的代码和有测试覆盖的代码，在面对需求变更时的心理负担完全不同：

场景没有单元测试有单元测试覆盖
修改一个被多处调用的公共方法 提心吊胆，只能靠人工把所有调用路径重新点一遍 改完直接跑测试，几秒内知道有没有破坏其他功能
重构一段逻辑复杂的老代码 不敢动，"能跑就行"，技术债越滚越大 敢于大胆重构，测试红了立刻能定位到问题
半年后接手别人的代码改bug 改完不知道有没有引入新问题，只能祈祷 改完跑一遍测试套件，有绿灯兜底

这就是"回归保护网"（regression safety net）：测试像一张安全网，让你在做改动时即使摔一下也不会真的摔伤。它保护的不是现在的代码，而是未来所有的修改。

价值三：活文档——测试用例描述预期行为

好的单元测试方法名和内容本身就在描述"这个方法在什么情况下应该做什么"。相比注释和 Word 文档，测试代码有一个巨大优势：它会随代码一起被编译、被执行，一旦过时立刻报错，而普通文档过时了也没人会发现。

测试即文档
@Test
void calculateDiscount_orderOverThreshold_returnsTenPercentOff() {
    // 光看方法名，不用翻文档就知道：订单金额超过阈值时打9折
    Order order = new Order(1000);
    double result = discountCalculator.calculate(order);
    assertEquals(900.0, result);
}

一个新人接手项目时，读懂产品代码往往很难，但读一遍测试用例，能快速搞清楚"这个类到底被设计成支持哪些场景"。这也是为什么很多团队把测试代码也当作一等公民来维护，而不是"能跑就行"的辅助文件。

与已学内容的联系

在 02-并发 模块里我们提到过，并发代码之所以难测试，是因为线程调度的不确定性让"复现问题"变得极其困难——这也是为什么单元测试更适合覆盖确定性的业务逻辑，并发相关的正确性往往需要专门的工具（如压力测试、模型检查）来补充。 而在 04-工程实战 模块讲到的依赖注入（DI），正是解决"难测试代码"耦合问题的关键手段，我们会在 mock/03 详细展开。

常见误区："测试覆盖率100%就说明代码质量好。" 覆盖率只统计代码"被执行过"，不代表断言验证得够充分——一个只调用方法却不做任何断言的测试，覆盖率照样能拉满，但毫无保护价值。这个话题在 pyramid/02 会专门展开。

核心价值总结

  • 设计反馈：难写的测试是耦合度过高的信号，倒逼你写出更松耦合的代码。
  • 回归保护网：让重构和修改需求时有信心，不必靠人工全量回归。
  • 活文档：测试用例描述预期行为，且不会像普通文档一样过时了没人发现。
  • 找bug只是副产品：如果只把测试当作找bug的工具，你会低估它真正的价值，也更容易半途放弃写测试。
