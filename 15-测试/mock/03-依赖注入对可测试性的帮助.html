Mock与依赖隔离

依赖注入对可测试性的帮助

前两篇讲的 Mock/Stub 技巧有一个隐藏前提：被测代码必须允许你把真实依赖换成假的。这个前提能否成立，取决于代码是"内部 new 出依赖"还是"依赖从外部注入"——这正是依赖注入（DI）思想在可测试性上最直接的体现。

回顾：04-工程实战里讲过的 IoC / DI

在 04-工程实战 模块，我们讲过控制反转（IoC）和依赖注入（DI）——对象不再自己创建所依赖的对象，而是由外部容器（如 Spring）负责创建好之后"注入"进来。当时更多是从"解耦、降低模块间依赖"的角度讲的；这一篇要讲的是 DI 带来的另一个同样重要、却经常被忽视的好处：可测试性。

反例：内部 new 依赖，测试没法替换

如果一个类在方法内部或构造器里直接 new 出它依赖的对象，这个依赖就被"焊死"在类内部，外部无论如何都无法在测试时把它换成 Mock：

反例：内部 new，无法注入 Mock
public class OrderService {

    public void placeOrder(Order order) {
        // PaymentGateway 在方法内部被 new 出来，测试时无法替换成假对象
        PaymentGateway gateway = new PaymentGateway("real-api-key");
        boolean success = gateway.charge(order.getAmount());

        if (success) {
            order.markAsPaid();
        }
    }
}

想给 placeOrder 写单元测试，唯一的办法是真的调用支付网关的接口——这不仅慢（违反 Fast），还可能产生真实的资金扣款（违反 Repeatable，而且很危险）。问题的根源不在于"测试工具不够强大"，而在于 这段代码的设计从一开始就没有给"替换依赖"留出任何余地。

重构：把依赖改成构造器注入

解决方案是把 PaymentGateway 的创建权交给外部，类本身只声明"我需要一个 PaymentGateway"，不关心它具体是怎么创建出来的：

正例：面向接口编程 + 构造器注入
public interface PaymentGateway {
    boolean charge(double amount);
}

public class OrderService {
    private final PaymentGateway paymentGateway;

    // 依赖通过构造器从外部传入，OrderService 不关心它具体是真实网关还是测试替身
    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }

    public void placeOrder(Order order) {
        boolean success = paymentGateway.charge(order.getAmount());
        if (success) {
            order.markAsPaid();
        }
    }
}
现在可以自由替换成 Mock 测试
@Test
void placeOrder_paymentSucceeds_marksOrderAsPaid() {
    PaymentGateway mockGateway = mock(PaymentGateway.class);
    when(mockGateway.charge(anyDouble())).thenReturn(true); // 假装支付成功，不产生真实扣款

    OrderService orderService = new OrderService(mockGateway);
    Order order = new Order(100);
    orderService.placeOrder(order);

    assertTrue(order.isPaid());
}

这次重构只做了一件事：把"谁来创建 PaymentGateway"从 OrderService 内部移到了外部。生产环境下由 Spring 容器（或手写的组装代码）注入真实的 PaymentGateway 实现；测试环境下由测试代码注入 Mock 实现。OrderService 自己完全不知道、也不需要关心运行时到底拿到的是哪一个。

为什么"面向接口编程"同样重要

注意上面的例子里，PaymentGateway 被声明成了一个接口而不是具体类。这一点同样关键：Mockito 可以对接口和大多数具体类生成动态代理，但依赖接口而不是具体实现能让"真实实现"和"测试替身"处于完全平等的地位——生产代码里注入 AlipayGateway，测试代码里注入 Mock，对 OrderService 而言两者没有任何区别，因为它自始至终只认得 PaymentGateway 这个契约。

这也是为什么"面向接口编程 + 依赖注入"被称为可测试代码的地基：面向接口编程决定了依赖可以被替换成任何实现，依赖注入决定了替换的时机和方式（构造器/setter）交给了外部，两者结合，才让 Mock 有了用武之地。如果代码里到处是 new ConcreteClass()，再厉害的 Mock 框架也无能为力。

对比维度内部 new 依赖依赖注入
依赖的创建者类自己创建，硬编码在内部外部（调用方/容器）创建后传入
能否替换成 Mock不能，除非改代码能，构造参数直接传 Mock 对象
类与具体实现的耦合度高，写死了具体是哪个实现类低，只依赖接口/抽象契约
单元测试范围被迫牵连真实依赖，变成集成测试可以真正做到"只测这一个单元"
类比：内部 new 依赖就像把电器的电线直接焊死在插座上；依赖注入则是用标准插头——只要符合接口（国标插孔），换成任何品牌的插座都能用，也方便你在维修/测试时插上一个"假电源"（Mock）而不必真的接入市电。
常见误区：认为"用了 Spring 就自动等于可测试"。Spring 只是提供了依赖注入的容器机制，如果业务代码里仍然在方法内部 new 出某个协作对象（哪怕这个项目整体在用 Spring），这部分代码照样没法被 Mock 替换。可测试性取决于具体每个类的依赖是不是真正做到了"外部注入"，而不是"项目用没用 Spring"。

核心价值总结

  • 内部 new 依赖会把类和具体实现焊死，测试时无法替换，只能被迫牵连真实依赖，测试变慢变脆弱。
  • 构造器/setter 注入把依赖的创建权交给外部，测试代码可以自由地传入 Mock/Stub。
  • 面向接口编程让"真实实现"和"测试替身"对被测类而言完全等价，是 DI 发挥作用的前提。
  • "面向接口编程 + 依赖注入"这个设计习惯，正是可测试代码的地基——先把这一步做对，Mock/Stub/Spy 这些工具才有用武之地。

与其他知识点的联系

  • Mock、Stub、Spy 的区别：Test Double 技术本身是"武器"，但武器要有使用场景——本文讲的正是这个使用场景成立的前提：依赖必须能从外部替换。反例 OrderService 内部 new PaymentGateway 说明，脱离了依赖注入，Mock/Stub/Spy 再强大也无从下手。
  • Mockito 实战：@InjectMocks 能省去手写 new UserService(userRepository) 的样板代码，其底层逻辑正是本文讲的"构造器注入"——Mockito 会自动扫描目标类的构造器参数，把类型匹配的 @Mock 对象一一填进去，这也是为什么 @InjectMocks 对内部 new 依赖的类完全无效。
  • 好测试的 FIRST 原则：反例中直接调用真实支付网关不仅慢（违反 Fast），还可能产生真实扣款（违反 Repeatable，且很危险）——依赖注入解决的不只是"能不能替换成 Mock"的技术问题，更是让测试重新满足 FIRST 标准的设计前提。
  • TDD 如何反过来驱动设计："这段代码很难测"往往是设计问题的信号，这一点与本文的核心论点完全一致——如果写测试时发现某个依赖没法替换成 Mock，往回看几乎总能发现这个依赖是被内部 new 出来的，先写测试的习惯会倒逼你更早发现并修正这种耦合。
  • 红绿重构循环：红绿重构的"绿"阶段要求测试能快速独立运行，而这要求被测类的协作对象是可替换的——依赖注入不是 TDD 之外的额外要求，而是让红绿循环能够真正跑起来的隐性基础设施。
