Mock与依赖隔离

Mockito 实战

Mockito 是 Java 生态最主流的 Mock 框架，把上一篇讲的 Mock/Stub 概念落地成几个简单好记的 API——用一个 Service 依赖 Repository 的真实场景，演示如何隔离掉数据库依赖来单独测试业务逻辑。

场景：Service 依赖 Repository

假设有一个 UserService，依赖 UserRepository 从数据库查用户，业务逻辑是"给用户加积分，如果用户不存在则抛异常"：

被测代码
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) { // 通过构造器注入，而不是内部 new
        this.userRepository = userRepository;
    }

    public void addPoints(long userId, int points) {
        User user = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));
        user.setPoints(user.getPoints() + points);
        userRepository.save(user);
    }
}

我们想测试 addPoints 的业务逻辑，但不想真的连数据库——这正是 Mockito 要解决的问题。UserRepository 之所以能被替换，前提是它通过构造器传入而不是在方法内部 new 出来，这一点在下一篇 03-依赖注入对可测试性的帮助 会展开讲。

@Mock / @InjectMocks：声明式创建测试替身

UserServiceTest.java
@ExtendWith(MockitoExtension.class) // JUnit5 里启用 Mockito 注解的扩展
class UserServiceTest {

    @Mock
    private UserRepository userRepository; // 自动创建一个 Mock 对象

    @InjectMocks
    private UserService userService; // 自动把上面的 Mock 注入到 UserService 的构造器里

    @Test
    void addPoints_existingUser_increasesPointsAndSaves() {
        // Arrange
        User user = new User(1L, "Tom", 50);
        when(userRepository.findById(1L)).thenReturn(Optional.of(user));

        // Act
        userService.addPoints(1L, 30);

        // Assert：状态验证 + 行为验证 结合使用
        assertEquals(80, user.getPoints());       // 状态验证：积分确实加对了
        verify(userRepository).save(user);              // 行为验证：确实调用了 save 持久化
    }

    @Test
    void addPoints_userNotFound_throwsUserNotFoundException() {
        when(userRepository.findById(99L)).thenReturn(Optional.empty());

        assertThrows(UserNotFoundException.class,
            () -> userService.addPoints(99L, 30));

        // 用户不存在时，不应该调用 save —— 用 never() 验证某方法完全没被调用
        verify(userRepository, never()).save(any());
    }
}

@Mock 自动创建一个替身对象；@InjectMocks 自动识别 UserService 的构造器参数，把匹配类型的 @Mock 对象注入进去，省去手写 new UserService(userRepository) 的样板代码。

when().thenReturn()：打桩语法

when(mock.method(args)).thenReturn(value) 是 Mockito 最核心的打桩语法：告诉 Mock 对象"当你的某个方法被以特定参数调用时，返回指定的值"。还有几个常用变体：

打桩变体
when(repo.findById(1L)).thenReturn(Optional.of(user));
when(repo.findById(anyLong())).thenThrow(new RuntimeException("db error")); // 模拟异常
when(repo.count()).thenReturn(1L).thenReturn(2L); // 连续调用返回不同结果，第一次1，之后都是2

verify()：验证调用情况

verify 常见用法
verify(userRepository).save(user);                 // 验证被调用了恰好一次（默认）
verify(userRepository, times(2)).save(user);      // 验证被调用了恰好两次
verify(userRepository, never()).delete(any());       // 验证从未被调用
verify(userRepository, atLeastOnce()).findById(1L)); // 验证至少被调用一次

ArgumentCaptor：捕获调用时传入的参数

有时候我们不仅想验证"某方法被调用了"，还想验证"调用时传的参数具体长什么样"（比如验证保存前 user 对象的字段确实被正确修改了）。ArgumentCaptor 就是用来"截获"这个参数的：

ArgumentCaptor 示例
@Test
void addPoints_existingUser_savesUserWithUpdatedPoints() {
    User user = new User(1L, "Tom", 50);
    when(userRepository.findById(1L)).thenReturn(Optional.of(user));

    userService.addPoints(1L, 30);

    ArgumentCaptor<User> captor = ArgumentCaptor.forClass(User.class);
    verify(userRepository).save(captor.capture()); // 截获 save() 被调用时传入的真实参数

    assertEquals(80, captor.getValue().getPoints()); // 对捕获到的参数做进一步断言
}

ArgumentCaptor 常用于依赖内部构造了新对象再传给 Mock 的场景——如果不捕获参数，你只能验证"save 被调用了"，却验证不了"传进去的 user 对象积分到底对不对"。

常见误区：Mockito 的 mock() 方法名字带"mock"，但它创建出来的对象既可以当 Stub 用（只打桩不验证调用），也可以当 Mock 用（打桩 + verify）。是 Stub 还是 Mock，取决于你怎么用它，而不是 API 的名字——这也是初学者容易搞混的地方。

核心价值总结

  • @Mock + @InjectMocks 用声明式注解代替手写样板代码，快速搭建测试替身。
  • when().thenReturn() 负责打桩，verify() 负责验证交互，两者结合能同时覆盖状态验证和行为验证。
  • ArgumentCaptor 在需要断言"调用时传的参数细节"时非常有用，弥补了 verify() 只能验证"是否调用"的局限。
  • 所有这些技巧的前提是被测代码把依赖通过构造器/setter 注入进来，而不是内部 new 出来——下一篇详细展开这个设计原则。

与其他知识点的联系

  • Mock、Stub、Spy 的区别：本文示例里 when(userRepository.findById(1L)).thenReturn(...) 是纯打桩、只关心结果，属于 Stub 用法；而 verify(userRepository).save(user) 是在验证交互，属于 Mock 用法——同一个 @Mock 对象在一个测试里同时承担了 Stub 和 Mock 两种角色，印证了上一篇"是 Stub 还是 Mock 取决于用法，不取决于 API 名字"的结论。
  • JUnit5 基础用法：@ExtendWith(MockitoExtension.class) 依赖的正是 JUnit5 的扩展模型（@ExtendWith 取代了 JUnit4 的 @RunWith），这也是为什么 Mockito 能以"零侵入"的方式接管 @Mock 字段的初始化和校验，而不需要在每个测试方法里手写 MockitoAnnotations.openMocks(this)。
  • 依赖注入对可测试性的帮助：@InjectMocks 能自动工作的前提是 UserService 的依赖通过构造器传入——如果 UserRepository 是在 UserService 内部 new 出来的，@InjectMocks 找不到注入点，整套 Mockito 用法都会失效，这正是下一篇要展开讲的设计前提。
  • 断言与测试用例的组织方式：本文所有测试代码都遵循 Arrange（准备 Mock 和打桩）→ Act（调用被测方法）→ Assert（assertEquals + verify 组合断言）的三段式结构，说明引入 Mock 框架并不会打乱 AAA 模式，反而是 Arrange 阶段多了"打桩"这一具体动作。
