Mock与依赖隔离

Mock、Stub、Spy 的区别

这三个词在日常交流中经常被混用（很多人把所有"假对象"统称为 Mock），但在测试理论里它们有精确的定义和不同的使用场景——分不清楚，写出来的测试意图会含糊不清，也是面试中的高频考点。

为什么需要"假对象"

basics/01 提到过，被测代码如果直接依赖数据库、网络、文件系统这类外部资源，测试就会变慢、变得不稳定（违反 FIRST 的 Fast/Repeatable）。 解决方案是用一个"假对象"替代真实依赖，这类假对象在测试理论里统称为 Test Double（测试替身，类比电影里的替身演员）。Test Double 有好几种，Stub、Mock、Spy 是其中最常用的三种，本文重点讲清楚它们各自的定位。

Stub：提供预设返回值，不关心是否被调用

Stub（桩）的职责很单纯：当被测代码调用它时，返回一个预先设定好的值。它不关心自己有没有被调用、被调用了几次、调用时传了什么参数——它唯一的使命是让测试能跑起来，把不确定的外部依赖替换成确定的、可控的返回值。

Stub 用法示例
@Test
void getDiscountPrice_vipUser_appliesTenPercentOff() {
    // UserRepository 是 Stub：只负责在被调用时返回预设的 VIP 用户，不关心调用细节
    UserRepository stubRepo = mock(UserRepository.class);
    when(stubRepo.findById(1)).thenReturn(new User(1, "Tom", true));

    PriceCalculator calculator = new PriceCalculator(stubRepo);
    double price = calculator.getDiscountPrice(1, 100);

    // 关注点在"结果对不对"，而不是 stubRepo 有没有被调用
    assertEquals(90.0, price);
}

Mock：不仅提供预设行为，还能验证交互

Mock 在提供预设返回值的基础上，多了一层能力：验证被测代码是否按预期与依赖对象进行了交互——比如"某个方法确实被调用了一次"、"调用时传的参数是什么"。当你关心的不是返回值，而是"这个动作有没有发生"时，就该用 Mock。

Mock 用法示例
@Test
void placeOrder_validOrder_sendsConfirmationEmail() {
    EmailService mockEmailService = mock(EmailService.class);
    OrderService orderService = new OrderService(mockEmailService);

    orderService.placeOrder(new Order("a@b.com", 100));

    // 关注点在"确认邮件的动作有没有发生"，而不是某个返回值
    verify(mockEmailService, times(1)).sendConfirmation("a@b.com");
}

Spy：包装真实对象，默认走真实逻辑，可选择性打桩

Spy（间谍）包装的是一个真实对象，默认情况下调用它的方法会执行真实逻辑，但你可以选择性地对其中某些方法打桩，替换掉不想真实执行的部分。因此 Spy 常被称为"部分 Mock"（partial mock）。

Spy 用法示例
@Test
void processList_usesRealLogicExceptOneMethod() {
    List<String> realList = new ArrayList<>(List.of("a", "b"));
    List<String> spyList = spy(realList);

    // size() 走真实逻辑，返回真实的 2
    assertEquals(2, spyList.size());

    // 但 get(0) 被选择性打桩，不再走真实逻辑
    doReturn("mocked").when(spyList).get(0);
    assertEquals("mocked", spyList.get(0));
}

Spy 常用于"老代码难以彻底重构成可测试结构，但又想局部隔离某个依赖"的过渡场景，或者需要验证真实对象某个方法调用次数、同时又需要其余方法保持真实行为的情况。

State Verification vs Behavior Verification

Stub 和 Mock 的差异背后，对应着两种不同的测试风格：

测试风格关注点典型配合断言方式
State Verification（状态验证） 执行完动作后，对象的状态/返回值是否符合预期 Stub assertEquals(expected, actual)
Behavior Verification（行为验证） 执行动作的过程中，是否与依赖对象发生了预期的交互 Mock verify(mock).someMethod(args)

State Verification 更贴近"黑盒"思维——不关心内部怎么实现的，只看输入输出对不对，因此测试对实现细节的改动更宽容，重构时不容易误报。 Behavior Verification 则会绑定具体的交互细节（调用了哪个方法、调用几次、传了什么参数），当内部实现调整（哪怕功能没变）时，测试可能会跟着报错——这也是 Mock 容易被滥用的地方：过度使用行为验证会让测试变得脆弱，稍微重构一下实现就要跟着改一堆测试。

常见误区：把"能不能验证调用"当作 Stub 和 Mock 唯一的区别，然后不管什么场景都无脑用 Mock（包括 Mockito 的 mock() 方法既能造 Stub 也能造 Mock，这也加剧了混淆）。实际上应该按"我到底关心的是结果还是交互过程"来决定用状态验证还是行为验证，而不是工具用得顺手就固定一种风格。
面试高频："说说 Mock、Stub、Spy 的区别？" 标准答案框架：Stub 提供预设返回值、不关心调用与否；Mock 除了预设行为还能验证交互（是否被调用、调用几次、传参是什么）；Spy 包装真实对象，默认真实执行、可选择性打桩，是"部分 Mock"。再补一句"Stub 配合状态验证，Mock 配合行为验证"会显得理解更深入。

核心价值总结

  • Stub：只提供数据，让测试跑起来，配合状态验证（断言返回值/结果状态）。
  • Mock：提供数据的同时还能验证交互，配合行为验证（断言方法调用情况）。
  • Spy：包装真实对象，默认真实执行，可选择性打桩，是"部分 Mock"，适合过渡场景。
  • 选用哪种，取决于测试到底关心"结果对不对"还是"过程有没有按预期交互"——滥用行为验证会让测试和实现细节强耦合，牺牲了重构的自由度。

与其他知识点的联系

  • 好测试的 FIRST 原则：本文开篇提到的"被测代码直接依赖数据库/网络会拖慢、拖不稳测试"正是 Fast 和 Repeatable 两条原则的具体表现——Test Double（Stub/Mock/Spy）就是为了消除这种外部依赖，让测试重新满足 FIRST 标准而存在的技术手段。
  • 断言与测试用例的组织方式：State Verification 用的正是上一篇讲的 assertEquals 等普通断言，Behavior Verification 用的是 Mockito 的 verify()——AAA 模式中的 Assert 阶段可以是这两种断言方式中的任意一种，取决于当前测试关心的是结果还是交互过程。
  • Mockito 实战：本文讲的是 Stub/Mock/Spy 的概念和取舍原则，下一篇 Mockito 实战则是把这些概念落地成 @Mock、when().thenReturn()、verify() 等具体 API——不理解本文的区分，写 Mockito 代码时容易分不清自己此刻到底是在做状态验证还是行为验证。
  • 依赖注入对可测试性的帮助：Test Double 有一个隐藏前提：被测代码必须允许外部把真实依赖替换成假对象，而这个前提能否成立完全取决于依赖是不是通过构造器/setter 从外部注入的——Test Double 是"武器"，依赖注入是能让这把武器用得上的"战场"。
  • TDD 如何反过来驱动设计："过度使用行为验证会让测试变脆弱"这一点，恰好是 TDD 强调"面向调用者视角设计接口"的一个反面教材——如果一个方法必须靠大量 verify() 才能证明它做对了事，往往说明它承担了太多副作用，接口设计本身就有改进空间。
