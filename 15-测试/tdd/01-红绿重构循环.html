TDD实践

红绿重构循环

TDD 的全部纪律都浓缩在这三个词里：红（写一个失败的测试）→ 绿（用最少的代码让它通过）→ 重构（在测试保护下改善结构）。看起来简单，但真正的难点在于"小步快跑"这四个字。

为什么要有这个循环

传统写法是"先想清楚整体设计，写完全部实现，最后补测试"。这条路看似高效，实际有个致命问题：你没有任何一个时刻能确认代码是对的。写了两百行代码才第一次运行，出了 bug 你很难定位是哪一步错了——因为改动面太大了。

Kent Beck 在《测试驱动开发》里提出的解法是：把"验证正确性"这件事切成无数个微小的、可以立刻确认的步骤。每次只往前迈一小步，迈完立刻验证，错了也只可能错在刚迈的这一小步里。这就是红绿重构循环存在的根本原因——它不是为了"多写测试"，而是为了把大风险拆成一串小风险，每个小风险都可控。

类比：走钢丝的人不会一步跨到对岸，而是小碎步挪，每一步都先感受重心稳不稳再迈下一步。红绿重构循环就是给写代码这件事装上"小碎步"——每次只解决一个最小的失败点，绝不在不稳的地基上继续叠代码。

三个阶段分别做什么

红（Red）：写一个新的测试，运行它，看着它失败
  ├─ 失败是必须的——它证明这个测试真的在测东西，而不是一段永远为真的摆设
  └─ 此时代码还没实现功能，甚至方法都可以还不存在（编译不过也算"失败"）

绿（Green）：写最少、最"丑"的代码，让测试通过
  ├─ 不追求优雅，甚至可以硬编码返回值——只求测试变绿
  └─ 目的是尽快回到"有安全网保护"的状态，别在这一步纠结设计

重构（Refactor）：在测试全绿的前提下，改善代码结构
  ├─ 消除重复、拆分方法、去掉硬编码，但绝不改变外部行为
  └─ 每改一小步就重新跑测试，任何一次变红都意味着刚才的重构引入了问题

这三步循环往复，每一轮只解决一个最小的需求点。关键心法是"小步快跑"：步子迈得越小，出错时需要排查的范围就越小，"绿"的状态就是你随时可以退回去的存档点。

完整案例：购物车计价

需求：实现一个购物车，可以添加商品（单价 × 数量），并计算总价。我们用 TDD 的方式一小步一小步实现它，而不是先设计好整个 Cart 类。

第一轮：空购物车总价为 0

CartTest.java · 红
@Test
void emptyCartTotalIsZero() {
    Cart cart = new Cart();
    assertEquals(0, cart.total());
}
// 此时 Cart 类还不存在，编译都过不了——这也是一种"红"
Cart.java · 绿
public class Cart {
    public int total() {
        return 0; // 最少的代码：硬编码返回 0，先让测试变绿
    }
}

此时没有需要重构的地方，直接进入下一轮。

第二轮：加入一件商品，总价等于单价 × 数量

CartTest.java · 红
@Test
void totalReflectsAddedItem() {
    Cart cart = new Cart();
    cart.add(new Item("apple", 10, 3)); // 单价10，数量3
    assertEquals(30, cart.total());
}
Cart.java · 绿
public class Cart {
    private final List<Item> items = new ArrayList<>();

    public void add(Item item) {
        items.add(item);
    }

    public int total() {
        int sum = 0;
        for (Item item : items) {
            sum += item.unitPrice() * item.quantity();
        }
        return sum; // 老测试(emptyCartTotalIsZero)依然通过：空列表求和自然是 0
    }
}

两个测试都是绿的，进入重构阶段：这里代码已经比较清晰，暂时不需要额外重构。

第三轮：加入两件不同商品，总价累加

CartTest.java · 红
@Test
void totalSumsMultipleItems() {
    Cart cart = new Cart();
    cart.add(new Item("apple", 10, 3));
    cart.add(new Item("pen", 5, 2));
    assertEquals(40, cart.total()); // 30 + 10
}

运行发现——这个测试直接就是绿的，因为第二轮写的 total() 用循环累加，天然支持多件商品。这正说明了"最少代码"并不等于"偷懒的代码"：for 循环并不比硬编码复杂多少，但它是真正解决问题的最小实现，而不是只应付眼前这一个用例的临时补丁。

面试高频："绿阶段的代码为什么允许写得丑？"因为丑不丑是重构阶段解决的问题，绿阶段唯一的目标是"用最短路径拿到一个通过的测试"，把"让代码能跑"和"让代码好看"两个目标分开处理，你的注意力才不会被同时分散到两件事上。

常见误区

误区一：步子迈得太大，一次性把好几个断言、好几个分支都写进一个测试里。这样即使测试红了，你也不知道该实现哪一部分——等于把"大风险拆小"的初衷又合并回了大风险。正确做法是每次只加一个最小的、能失败的断言。
误区二：跳过重构阶段，测试一变绿就急着写下一个测试。代码里的重复和坏味道会不断累积，到后期即使有测试保护也很难改——因为结构已经僵化。重构不是可选项，是循环里和红、绿同等重要的一环。

核心价值总结

  • 红绿重构的本质是把"验证正确性"切成可控的小步，每一步都有明确的、可运行的反馈信号。
  • 红确认测试真的在测东西；绿用最短路径拿到安全网；重构在安全网保护下偿还设计债。
  • 小步快跑不是慢，是把犯错的成本降到最低——出错时只需要审查刚迈的那一小步，而不是审查过去两百行代码。
