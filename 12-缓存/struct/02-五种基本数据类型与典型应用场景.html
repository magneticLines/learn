数据结构与内核

五种基本数据类型与典型应用场景

Redis 不是简单的 key-value 缓存，value 部分支持五种结构，每种结构对应一类真实业务问题——选对类型，往往能省掉一层应用层逻辑。

一句话讲清楚

String（字符串）、Hash（哈希表）、List（双向链表）、Set（无序集合）、ZSet（有序集合） 是 Redis 对外暴露的五种基本数据类型。它们不是随便设计的，每一种都精确对应一类"数据形状"和"操作模式"，选型的关键是问自己： 我需要的是简单值、字段集合、有序队列、去重集合，还是带权重排序？

String：最简单的类型，用途最广

String 可以存文本、数字、甚至二进制数据（图片、序列化对象）。因为支持原子的 INCR/DECR， 是做计数器的天然选择。

redis-cli
# 场景：文章阅读量计数器，天然支持并发安全自增
SET article:1001:views 0
INCR article:1001:views
# 场景：缓存序列化后的用户对象，设置 10 分钟过期
SET user:2001:profile '{"name":"Tom","age":20}' EX 600

Hash：适合存"一个对象的多个字段"

如果把整个对象序列化成一个 String，改一个字段就要整体读出、修改、写回。Hash 允许对单个字段独立读写， 适合存储用户信息、商品信息这类"有多个属性的对象"。

redis-cli
# 场景：购物车商品，只更新数量字段，不用整体覆盖
HSET cart:user2001 sku1001 2
HINCRBY cart:user2001 sku1001 1
HGETALL cart:user2001

List：有序、可两端操作，适合队列和最新列表

List 底层是双向链表语义，支持从两端 push/pop，天然适合做简单消息队列（生产者 LPUSH，消费者 BRPOP）； 也适合做"最新 N 条"这种截断展示，比如朋友圈最新动态、最新登录记录。

redis-cli
# 场景：简易任务队列，生产者入队，消费者阻塞式取出
LPUSH task:queue "send_email:1001"
BRPOP task:queue 0
# 场景：只保留最近 100 条浏览记录
LPUSH user:2001:history "item:9527"
LTRIM user:2001:history 0 99

Set：无序不重复，天然适合去重和集合运算

Set 的核心价值是唯一性和集合代数运算（交集、并集、差集）， 适合做标签系统、共同好友、抽奖去重这类场景。

redis-cli
# 场景：给用户打标签，判断是否已存在自动去重
SADD user:2001:tags "vip" "male"
# 场景：求两个用户的共同关注（交集运算）
SINTER user:2001:following user:2002:following

ZSet：带分数的有序集合，排行榜的标准解

ZSet 给每个元素关联一个 score，底层用跳跃表维护有序性（下一篇会展开讲跳表实现）， 天然适合排行榜（分数即排名依据）和延迟队列（分数存"到期时间戳"，定时轮询取出已到期的任务）。

redis-cli
# 场景：游戏积分排行榜，按分数自动排序
ZADD leaderboard 9500 "player:Tom"
ZREVRANGE leaderboard 0 9 WITHSCORES
# 场景：延迟队列，score 存到期的时间戳，定时取出已到期任务
ZADD delay:queue 1735689600 "order:1001:timeout"
ZRANGEBYSCORE delay:queue 0 <now_timestamp>

五种类型速查

类型核心特性典型场景
String简单值，支持原子自增计数器、缓存对象、分布式锁
Hash字段级读写，避免整体覆盖用户信息、购物车
List有序、两端操作、阻塞弹出消息队列、最新列表
Set无序不重复、集合运算标签、共同好友、抽奖去重
ZSet按分数有序、范围查询排行榜、延迟队列
面试高频："用 Redis 实现排行榜和延迟队列都用 ZSet，两者区别是什么？" 排行榜的 score 是业务分值，关心的是排名顺序； 延迟队列的 score 是时间戳，关心的是是否已经小于等于当前时间，本质上是把"定时触发"问题转化成了"范围查询"问题。

核心价值总结

  • 五种类型对应五类数据形状，选型前先想清楚"我到底需要有序、去重、字段独立读写，还是纯粹存值"。
  • 很多"分布式锁""延迟队列""排行榜"这些中间件功能，本质上都是对基础类型特性的巧妙复用，而不是 Redis 单独实现的新功能。
