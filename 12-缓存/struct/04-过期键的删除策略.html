过期键的删除策略

设置了 EXPIRE 的键到期后不会立刻消失，Redis 要在"及时释放内存"和"不浪费 CPU 去主动扫描"之间找平衡——这正是面试爱问的考点。

一句话讲清楚

Redis 用惰性删除（lazy expire）+ 定期删除（active expire）两种策略结合的方式处理过期键， 而不是理论上最"及时"但代价最大的定时删除（为每个设置了过期时间的键单独创建一个定时器，到点立刻删除）。

类比：定时删除像是给冰箱里每样快过期的食物都单独设一个闹钟，食物一多闹钟就多到吵得没法专心干别的事； 惰性删除像是"只有你打开冰箱拿某样东西时才检查它是不是过期了"，平时不主动看；定期删除则像是"每隔一段时间随手检查一遍冰箱，扔掉明显过期的东西"， 两者结合，既不会占用太多额外精力，又不至于让过期食物长期占着位置不管。

为什么不用定时删除

定时删除对内存最友好——键一过期立刻释放——但需要维护大量定时器，在键数量庞大、过期时间密集的场景下， 会消耗大量 CPU 去管理和触发这些定时器，这与 Redis 单线程执行命令、需要把 CPU 时间优先让给正常请求的设计目标相冲突。 所以 Redis 放弃了定时删除，选择了更"折中"的两种策略。

惰性删除：访问时才检查

每次访问一个键时，Redis 先检查它是否已经过期，过期就删除并返回"键不存在"，不过期才正常处理。 这种策略的优点是几乎不占用额外 CPU——没有后台扫描的开销；缺点是如果一个过期键始终没有被访问，它会一直占用内存不被回收， 这在过期键数量多、访问频率低的场景下可能造成明显的内存浪费。

定期删除：周期性抽查

Redis 内部有一个定时任务，周期性地从设置了过期时间的键里随机抽取一部分进行检查，删除其中已过期的键。 这弥补了惰性删除"无人访问就永不回收"的缺陷，同时通过限制每次抽查的数量和执行时间，避免像定时删除那样占用过多 CPU。

面试高频："Redis 如何删除过期键？" 标准答案是惰性删除 + 定期删除相结合，而不是单独一种。 可以进一步追问延伸——"如果一个过期键既没被访问、又没被定期删除抽到，会怎样？" 答案是：它会一直占用内存， 直到被访问触发惰性删除、或被下一轮定期删除抽中，这也是为什么 Redis 还需要"内存淘汰策略"作为兜底（下一篇讲）。

常见误区

误区："设置了 EXPIRE 之后，这个键到点就会自动从内存里消失。" 实际上 Redis 不保证这一点——过期只是给键打上了"逻辑上已失效"的标记，真正从内存里删除，依赖的是惰性删除或定期删除被触发。 在两者都没触发之前，这个键的数据依然占用着内存空间，只是正常的 GET 等命令会表现得像它不存在一样。

核心价值总结

  • 定时删除内存最友好但 CPU 开销最大，Redis 因此放弃了它。
  • 惰性删除几乎零开销，但可能让冷门的过期键长期占用内存。
  • 定期删除是折中方案，周期性随机抽查，弥补惰性删除的盲区。
  • 两者结合仍不能保证"内存绝对不会因为过期键而浪费"，这正是为什么还需要内存淘汰策略作为最后一道防线。
