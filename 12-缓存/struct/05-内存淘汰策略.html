内存淘汰策略

当内存用满了怎么办？惰性删除和定期删除只处理"已过期"的键，真正的内存兜底方案是淘汰策略——面试高频，也是理解 LRU/LFU 区别的关键。

一句话讲清楚

当 Redis 使用的内存达到 maxmemory 配置的上限时，会根据配置的淘汰策略（eviction policy）决定如何腾出空间： 要么直接拒绝写入报错（noeviction），要么按某种规则挑选一部分键删除，为新数据腾地方。 挑选规则主要有三大类：LRU（淘汰最近最少使用的）、LFU（淘汰访问频率最低的）、random（随机淘汰）。

noeviction：默认策略，不淘汰直接报错

这是 Redis 的默认策略：内存满了之后，所有会让内存继续增长的写命令都会返回错误，但读命令仍可正常执行。 适合把 Redis 当作存储而非单纯缓存使用、绝不容忍数据被静默删除的场景，代价是应用层必须处理写入失败的情况。

LRU 系列：淘汰最近最少使用的键

LRU（Least Recently Used）关注的维度是时间——一个键多久没被访问过。 越久没被访问的键，越优先被淘汰。Redis 提供 allkeys-lru（在所有键中淘汰）和 volatile-lru（只在设置了过期时间的键中淘汰）两个变体。

LRU 的误判场景：假设一个键在过去一小时内被访问了 1000 次，但最近 1 分钟恰好没人访问； 另一个键一小时内只被访问了 1 次，但恰好是刚刚访问的。纯 LRU 会认为后者"更新鲜"从而保留它、淘汰前者—— 但从整体访问热度看，前者其实更应该被保留。这正是 LFU 被引入的原因。

LFU 系列：淘汰访问频率最低的键（Redis 4.0+）

LFU（Least Frequently Used）关注的维度是频率——一个键历史上被访问了多少次， 而不只看"最近一次访问的时间"。Redis 4.0 引入 allkeys-lfu 和 volatile-lfu， 更适合区分"真正的热点数据"和"偶尔被碰一下的冷数据"，避免 LRU 那种"只看最近一次时间就误判成活跃"的问题。

redis.conf
# 达到 maxmemory 后，只在设置了过期时间的键中按访问频率淘汰
maxmemory 4gb
maxmemory-policy volatile-lfu

random：随机淘汰

allkeys-random / volatile-random 顾名思义，随机挑选键淘汰，不做任何"热度"判断。 开销最低，但完全不考虑访问模式，一般只在对淘汰公平性没有要求、追求极致淘汰性能的场景下使用。

LRU vs LFU：核心区别一览

策略判断维度典型误判场景引入版本
LRU时间：最近一次访问的时间历史高频但最近恰好没访问的键被误淘汰早期版本即有
LFU频率：历史访问次数（带衰减）需要额外维护计数器，内存略有开销Redis 4.0+
random无，纯随机可能淘汰掉真正的热点数据早期版本即有
面试高频："LRU 和 LFU 的区别？" 一句话答案：LRU 看"最后一次访问是什么时候"，LFU 看"总共被访问了多少次"。 进一步追问"什么场景 LFU 更好？" 答：短时间内偶然被访问一次的冷数据，在 LRU 下会被误判为"最近使用"从而侥幸存活， LFU 因为看重历史累计频率，能更准确地识别出真正的热点数据。

核心价值总结

  • maxmemory + 淘汰策略是 Redis 内存管理的最后一道防线，弥补惰性删除和定期删除"无法保证内存绝对可控"的缺口。
  • noeviction 保数据完整性但会拒绝写入，LRU/LFU/random 保写入可用性但可能丢数据，本质是"一致性 vs 可用性"的取舍。
  • LFU 是对 LRU 局限性的补充，二者的核心区别是"时间维度"还是"频率维度"，这是本篇乃至整个 Redis 面试里的高频细节题。

与其他知识点的联系

  • 过期键的删除策略：上一篇讲的惰性删除和定期删除只处理"已经过期"的键，而内存淘汰策略处理的是"内存满了但键还没过期"的情况——两者是内存管理体系里互补的两道防线，volatile 系列的淘汰策略（如 volatile-lru）甚至只在设置了过期时间的键里挑选，说明淘汰策略和过期机制在实现上也有交集。
  • RDB 快照持久化：下一个 Tab 开篇讲的 RDB 持久化，本质是把内存淘汰策略"保下来"的这些数据落盘保存——如果淘汰策略配置不当导致关键数据被误删，即便持久化机制再完善，快照里也不会有那份已经被淘汰掉的数据，说明内存管理策略的选择会直接影响持久化能保住什么。
  • 缓存穿透/击穿/雪崩：如果淘汰策略选择不当（比如用 allkeys-random 随机淘汰掉了正在被大量并发访问的热点 key），相当于人为制造了一次缓存击穿——原本还没过期的热点数据被淘汰策略提前清理掉，下一波请求会集中穿透到数据库，这是淘汰策略选型不当引发的一种"自己造成的击穿"。
  • 用 Redis 实现分布式锁：如果 Redis 实例内存紧张并配置了 allkeys-lru 或 allkeys-random 之类"在所有键中淘汰"的策略，一把正在被持有的分布式锁理论上也可能被当作普通键淘汰掉，导致锁提前失效——这提示分布式锁场景下应当谨慎选择淘汰策略，或者用独立的 Redis 实例承载锁，避免和普通缓存数据混用同一套淘汰规则。
  • Redis 为什么快：LRU/LFU 都需要额外维护"最近访问时间"或"访问频率计数器"这类元数据，这是在"淘汰决策的准确性"和"额外内存 / 计算开销"之间的取舍，和 struct 模块第一篇讲的"单线程、内存存储"的极简设计哲学一脉相承——Redis 的很多设计选择都在朴素和精确之间选择了一个能接受的折中点，而不是追求理论最优。
