AOF 追加日志与重写机制

RDB 会在两次快照之间丢数据，AOF 换了个思路：记录每一条写命令，通过"重放日志"来恢复数据——代价是文件会越滚越大，需要重写机制压缩体积。

一句话讲清楚

AOF（Append Only File）不存数据快照，而是把每一条写命令以协议文本的形式追加写入日志文件。 宕机重启后，Redis 依次重新执行 AOF 文件里的命令，就能把数据恢复到宕机前的状态。 这个思路和 09-数据库模块讲过的redo log本质相同：都是"先把操作记下来，出问题了靠重放日志恢复"， 用日志的持久化换取比"只在某个时间点拍快照"更强的数据完整性保证。

三种刷盘策略：性能与安全性的取舍

写命令先追加到内存缓冲区，再由操作系统决定何时真正落盘（fsync）。Redis 提供三种刷盘时机配置，本质是在 "数据安全性"和"写入性能"之间做取舍：

策略刷盘时机性能最坏情况丢失数据
always每条写命令都立刻同步刷盘最慢几乎不丢（除非磁盘本身损坏）
everysec每秒批量刷盘一次（默认）接近内存速度，兼顾安全最多丢失 1 秒内的写入
no由操作系统决定何时刷盘最快可能丢失较多数据（依赖 OS 策略，通常数十秒）
为什么 everysec 是默认值：always 每条命令都要等一次磁盘 IO 完成才返回，吞吐量会明显下降； no 完全交给操作系统，虽然快但丢失窗口不可控。everysec 把丢失窗口限制在"最多 1 秒"，性能损失又很小， 是绝大多数场景下"够用又不太慢"的折中选择。

AOF 重写（rewrite）：不是去重命令，而是重新生成

AOF 记录的是命令流，如果一个键被反复 INCR 一万次，文件里就会有一万条 INCR 记录，文件体积会无限增长。 AOF 重写要解决这个问题，但它不是简单地把重复或可合并的命令去重、压缩，而是读取当前内存里的实际数据状态， 为每个键重新生成能够达到当前状态的最少命令集合。

比如那个被 INCR 一万次的键，当前值是多少，重写后就直接生成一条 SET key value， 彻底跳过中间过程，文件体积因此大幅压缩。这个过程同样通过 fork 子进程完成（子进程遍历内存数据生成新 AOF， 主进程仍正常处理请求并把新写入的命令同时追加到重写缓冲区，重写完成后再合并），避免阻塞主线程。

redis-cli / redis.conf
# 手动触发一次 AOF 重写
BGREWRITEAOF
# 自动触发条件：文件比上次重写后增长 100%，且体积超过 64MB
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
面试高频："AOF 重写是怎么压缩文件体积的？" 很多人会误答"去重相同的命令"，正确答案是 基于当前内存中的实际数据状态，重新生成一份能达到相同结果的最简命令集合， 这和"删除重复行"是完全不同的思路——重写后的文件里可能根本没有出现过原始命令序列中的任何一条命令。

核心价值总结

  • AOF 记录写命令而非数据快照，与 redo log 思路一致，数据完整性比 RDB 更强。
  • 三种刷盘策略是"安全性 vs 性能"的经典取舍，everysec 是大多数场景的默认选择。
  • AOF 重写不是去重，而是基于当前数据状态重新生成最小命令集合，从根本上压缩文件体积。
  • 下一篇讲混合持久化，看 Redis 如何把 RDB 的"恢复快"和 AOF 的"数据全"结合到一份文件里。

与其他知识点的联系

  • RDB 快照持久化：AOF 和 RDB 是同一个问题（如何持久化内存数据）的两种不同答案——RDB 记录"某一时刻的结果状态"，AOF 记录"从头到尾的变更过程"，两者在"恢复速度 vs 数据完整性"这条轴上正好互补，理解了 RDB 的 fork + 写时复制，才能对比出 AOF 重写为什么也要用同样的 fork 机制来避免阻塞主线程。
  • 混合持久化：下一篇讲的混合持久化本质是把 AOF 重写这个动作产出的"历史部分"从命令文本格式换成了 RDB 二进制格式，重写之后的增量部分依然是本文讲的 AOF 追加写模式——不理解 AOF 重写的触发时机和过程，就看不懂混合持久化到底"混合"了什么。
  • 主从复制的增量同步：AOF 记录写命令、重放命令来恢复数据的思路，和主从复制里"命令传播"（增量同步）几乎是同一套逻辑——一个是"把命令写给磁盘上的自己"，一个是"把命令发给网络另一端的从库"，本质都是"日志流 + 重放"，这也是为什么 Redis 6.0 之后的多部分 AOF 机制在设计上会参考复制积压缓冲区的思路。
  • 与 MySQL redo log 的类比：文中提到 AOF 和 09-数据库模块讲的 redo log 思路一致，都是"先落盘操作记录，出问题靠重放恢复"，这是一种可以迁移到其他系统的通用心智模型：只要看到"某个系统需要在崩溃后恢复状态"，就可以想到"要不要维护一份操作日志"这个设计选项。
  • 缓存与数据库一致性问题：everysec 刷盘策略"最多丢失 1 秒数据"的取舍，和缓存实战模块里"接受最终一致而不是追求强一致"是同一种工程哲学——都是在性能和数据安全之间找一个务实的平衡点，而不是走极端。
