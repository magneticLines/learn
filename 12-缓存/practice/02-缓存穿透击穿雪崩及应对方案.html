缓存实战

缓存穿透/击穿/雪崩及应对方案

三个名字很像、经常被混着问的面试高频概念——但它们的触发条件、影响范围和解法完全不同，一定要能准确区分。

先建立直觉：三者都会导致"本该被缓存挡住的请求，最终打到了数据库上"，但原因各不相同——穿透是查询根本不存在的数据，击穿是某一个热点 key 恰好过期，雪崩是大量 key 同时失效或缓存整体宕机。记住"不存在 / 单点过期 / 大面积失效"这三个关键词，就不会混淆。

缓存穿透：查询根本不存在的数据

现象：请求查询一个缓存和数据库里都没有的 key（比如用户传入一个不存在的 id=-1）。因为缓存里没有，请求必然穿透到数据库；因为数据库里也没有，这次查询无法被缓存住，下次同样的请求还是会再打到数据库。 如果有人恶意构造大量不存在的 key 发起请求，就能绕开缓存的保护，直接把压力全部压给数据库，甚至打垮数据库。

解决方案：

  • 缓存空值：查询数据库发现确实不存在时，也在缓存里写入一个特殊的空值标记（设置一个较短的过期时间），下次同样的请求直接命中这个空值缓存，不用再查数据库；
  • 布隆过滤器（Bloom Filter）：提前用一个布隆过滤器记录所有"确实存在"的 key，请求进来先过滤器判断，如果过滤器说"一定不存在"，直接拒绝，连缓存都不用查，从最前端就挡住无效请求。

缓存击穿：单个热点 key 过期瞬间被并发打爆

现象：某一个访问量极高的热点 key（比如某个爆款商品详情）在缓存中过期的那一瞬间，恰好有大量并发请求同时到来。由于缓存已过期，这些请求会同时判断"未命中"，全部去查数据库，相当于短时间内大量重复查询同一份数据直接砸向数据库。

解决方案：

  • 互斥锁（Mutex）：缓存未命中时，只让第一个请求去查数据库并回填缓存，其他并发请求等待或重试，等缓存回填完毕后直接从缓存读取，避免重复查库；
  • 热点数据不设置过期时间（或者用逻辑过期）：对于明确知道是热点的数据，干脆不让它自然过期，而是通过后台任务主动刷新，从根源上避免"过期瞬间被击穿"这个场景发生。

缓存雪崩：大量 key 同时失效，或缓存整体宕机

现象：缓存雪崩有两种常见诱因：一是大量 key 设置了相同的过期时间（比如批量预热数据时统一设置了 1 小时过期），到点后集中失效，瞬间有大批请求同时穿透到数据库；二是Redis 实例本身宕机或不可用，导致所有请求都无法命中缓存，全部直接打到数据库。无论哪种诱因，最终结果都是数据库在短时间内被瞬时高并发压垮，甚至引发连锁的服务雪崩（数据库扛不住，上游服务也跟着超时）。

解决方案：

  • 过期时间加随机值：批量设置缓存时，在基础过期时间上叠加一个小的随机偏移量（比如 1 小时 ± 5 分钟），把原本集中在同一时刻的失效点打散，避免同时失效；
  • 搭建高可用集群：用前面讲过的主从 + Sentinel 或 Cluster，避免 Redis 单点宕机导致缓存整体不可用；配合服务端限流、熔断、数据库层面的降级策略，兜底最坏情况。

三者对比一览

问题触发条件影响范围核心解法
缓存穿透 查询缓存和数据库中都不存在的 key 可被恶意放大，任意重复查询同一无效 key 缓存空值 / 布隆过滤器
缓存击穿 单个热点 key 恰好在高并发时过期 集中在一个 key 上的瞬时高并发查库 互斥锁 / 热点数据不过期
缓存雪崩 大量 key 同时过期，或 Redis 整体宕机 大面积、大批 key 同时失效，冲击整个数据库 过期时间加随机值 / 搭建高可用集群
面试高频：被问到"讲讲缓存穿透、击穿、雪崩的区别"时，按"不存在的数据 / 单点过期 / 大面积失效"这个思路组织回答，再各自补充一到两个对应解法，是最容易讲清楚、面试官也最容易听懂的顺序。
常见误区：很多初学者会把"击穿"和"雪崩"搞混，关键区分点是波及的 key 数量——击穿永远是"某一个热点 key"的问题，雪崩则是"一大批 key 或者整个缓存层"的问题。穿透和另外两者的本质区别则是"数据是否真的存在"。

核心价值总结

  • 穿透：查不存在的数据 → 缓存空值 / 布隆过滤器。
  • 击穿：单个热点 key 并发过期 → 互斥锁 / 不设过期。
  • 雪崩：大批 key 集中失效或缓存宕机 → 过期时间加随机 / 高可用集群。
  • 三者本质都是"缓存没能挡住请求，数据库直接暴露在高并发下"，理解了触发条件的差异，解法也就顺理成章。

与其他知识点的联系

  • 缓存更新策略：上一篇讲的 Cache Aside"读未命中查库回填"正是穿透、击穿发生的具体环节——穿透是这个回填流程被无效 key 反复触发，击穿是这个回填流程在热点 key 过期瞬间被并发触发，理解了 Cache Aside 的读流程，才能看清这两个问题到底发生在哪一步。
  • 主从复制、Sentinel、Cluster（高可用主线）：缓存雪崩的第二个诱因是"Redis 整体宕机"，这正是高可用主线三篇要解决的问题——主从复制提供数据冗余，Sentinel 提供自动故障转移，Cluster 提供水平扩展和分片级容错，搭建这套高可用架构是应对"实例宕机型雪崩"的根本手段，而不是事后限流降级的补救。
  • 过期键的删除策略与内存淘汰策略：缓存雪崩里"大量 key 集中失效"的诱因，和 struct 模块讲的过期机制直接相关——如果批量写入时都设置了相同的固定过期时间，到点后这些 key 几乎同时被判定为过期，配合"过期时间加随机值"的解法，本质是主动错开 key 的失效时间点，避免集中触发惰性删除或定期删除后的大量回源查询。
  • 缓存与数据库一致性问题：缓存空值这个解法本身也会带来新的一致性问题——如果数据库里后续真的插入了这条数据，之前缓存的空值标记必须及时失效或更新，否则会出现"数据库有数据但缓存说没有"的新脏数据，这和下一篇讨论的一致性窗口是同一类问题在不同场景下的体现。
  • 用 Redis 实现分布式锁：缓存击穿的互斥锁解法（只让第一个请求查库回填，其他请求等待）本质上就是一次简化版的分布式锁应用——如果部署了多个应用实例，这把"只让一个请求去查库"的锁必须是跨实例生效的分布式锁，而不能是单机内的本地锁，这就直接呼应了本模块最后一篇的主题。
