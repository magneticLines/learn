用 Redis 实现分布式锁

单机场景下 synchronized、Lock 就能解决并发问题，但多个进程/多台机器之间要互斥怎么办？Redis 凭借"单线程执行命令"的特性，天然适合做分布式锁的载体。

为什么需要分布式锁

Java 里的 synchronized、ReentrantLock 只能保证同一个 JVM 进程内的线程互斥——如果你的服务部署了多个实例（这在现代分布式架构里是常态）， 这些锁完全管不到"实例 A 的线程"和"实例 B 的线程"之间的互斥。这时候需要一个所有实例都能访问到的公共仲裁者，Redis 常被用来扮演这个角色，因为它的命令执行是单线程的，天然具备"同一时刻只有一个客户端能抢到某个操作结果"的原子性保证。

最基本的实现：SET key value NX PX

Redis 分布式锁的核心是一条原子命令：

redis-cli
SET lock:order:1001 "client-uuid-abc123" NX PX 30000
  • NX：只有 key 不存在时才设置成功（Not eXists），保证"抢锁"这个动作是互斥的——多个客户端同时执行，只有一个能设置成功；
  • PX 30000：设置 30 秒后自动过期，防止持锁的客户端崩溃后锁永远得不到释放（死锁）；
  • 关键点："判断 key 不存在" + "设置值和过期时间" 是一条命令、一次原子操作完成的，不会出现"两个客户端都判断 key 不存在，然后都设置成功"的竞态。

释放锁就是删除这个 key：DEL lock:order:1001。看起来简单，但这里藏着几个容易踩的坑。

陷阱一：锁的 value 必须是唯一标识，不能随便设个固定值

设想 value 固定写成 "1" 这种没有区分度的值，会发生什么：客户端 A 抢到锁，处理业务耗时较长，超过了锁的过期时间 30 秒，锁自动过期释放；这时客户端 B 抢到了同一把锁； A 处理完业务后，不知道锁已经不是自己的了，直接执行 DEL，把 B 正持有的锁误删了——B 还在执行业务，锁却提前没了，可能导致第三个客户端 C 又抢到锁，出现多方同时操作临界区的严重问题。

解法：value 必须设置成能唯一标识"这把锁是谁加的"的值（比如 UUID，或者线程 ID + 机器标识），释放锁时先判断 GET 出来的 value 是不是自己当初设置的那个值，只有匹配才执行删除——这样即便锁已经过期被别人抢走，自己也不会误删别人的锁。

陷阱二："判断 value + 删除"必须用 Lua 脚本保证原子性

知道了要"先判断 value 匹配，再删除"，很多人会写成两条命令：先 GET 判断，再 DEL。但这两条命令之间不是原子的——如果判断通过后，恰好这把锁自然过期， 然后被另一个客户端抢到，紧接着自己执行的 DEL 依然会把新客户端的锁误删，陷阱一想解决的问题又绕回来了。

释放锁的 Lua 脚本（保证判断+删除是一次原子操作）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end

Redis 执行 Lua 脚本时，整段脚本作为一个整体原子执行，中间不会被其他客户端的命令插入——这正是利用了 Redis 单线程模型的特性，用脚本把"读 + 判断 + 删"打包成一次不可分割的操作。

陷阱三：单机 Redis 锁在主从切换场景下可能不安全

即便解决了前两个陷阱，还有一个更深层的问题：Redis 主从复制是异步的（第一篇讲过）。设想客户端 A 在主库上 SET NX 抢锁成功， 但这个写命令还没来得及同步给从库，此时主库突然宕机，Sentinel 触发故障转移，某个从库被提升为新主库——但新主库上根本没有这把锁的数据。 如果客户端 B 这时候去新主库抢锁，会发现锁不存在，直接抢锁成功，于是 A、B 同时认为自己持有锁，互斥被击穿。

本质原因：主从复制的异步性决定了"写入主库成功"不等于"所有副本都已经拿到这份数据"，在主从切换的时间窗口内，单机部署的 Redis 锁无法保证绝对的互斥安全——这是分布式锁在 Redis 这种最终一致存储上实现时的固有局限，而不是代码写得好坏的问题。

了解即可：Redlock 算法的思路

针对这个问题，Redis 作者提出过 Redlock 算法，核心思路是：不只在一个 Redis 实例上加锁，而是在多个互相独立的 Redis 主节点（比如 5 个）上分别尝试加锁， 只有当客户端在超过半数的节点上都加锁成功，才认为真正拿到了锁，以此降低单点主从切换带来的风险。这个方案本身在业界存在一些争议和讨论，这里作为思路了解即可，不需要深入展开实现细节。

面试高频："用 Redis 实现分布式锁要注意什么？" 按层次回答：基础实现是 SET key uniqueValue NX PX；释放锁要先判断 value 是不是自己的，且这个判断+删除必须用 Lua 脚本保证原子性，防止误删别人的锁；再进一步可以提到单机 Redis 因为异步复制，在主从切换时锁的安全性无法绝对保证，Redlock 是针对这个问题的一种改进思路。层层递进的回答比只说"用 SETNX"更能体现理解深度。

核心价值总结

  • 基础：SET key value NX PX 一条命令原子地完成"抢锁 + 设置过期时间"。
  • 唯一标识 + Lua 脚本：解决"释放锁"阶段的两个原子性陷阱，防止误删别人持有的锁。
  • 主从复制的异步性是单机 Redis 锁的天花板：理解这一局限比死记 Redlock 的实现细节更重要，也呼应了本模块第一篇讲过的复制机制。
