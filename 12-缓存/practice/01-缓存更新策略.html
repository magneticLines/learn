缓存更新策略（Cache Aside 等模式）

缓存和数据库是两份独立的数据，写操作要同时维护两边——先写哪个、更新缓存还是删除缓存，选择不同直接决定了系统会不会出现脏数据。

Cache Aside（旁路缓存）：最常用的模式

Cache Aside 的意思是"缓存旁路"——应用代码自己负责读写缓存和数据库，缓存不直接和数据库打交道，两者的同步逻辑全部写在业务代码里。这是实际项目中最常见的模式，规则很简单：

  • 读：先查缓存，命中直接返回；未命中则查数据库，把结果写回缓存，再返回给调用方；
  • 写：先更新数据库，再删除缓存中对应的 key（而不是更新缓存）。
类比：数据库是仓库总账，缓存是柜台上贴的"价目表"。改价格时，仓库账本改完之后，最省心的做法不是费劲去柜台把价目表现改成新数字，而是直接把这张旧价目表撕掉——下一个顾客来问价格时，柜员自然会重新去查一次账本，抄一张新的贴上去。这样不用担心"账本改了但价目表没跟上改对"的问题。

为什么写操作是"删除缓存"，而不是"更新缓存"

初学者很容易觉得：既然要更新数据库，顺手把缓存也更新成新值不是更高效吗？为什么 Cache Aside 偏偏选择删除？原因是并发场景下"更新缓存"更容易留下脏数据。设想两个并发的写请求 A、B，都要把某个 key 从旧值改成新值：

并发写更新缓存的问题：如果两个线程都是"写库 + 更新缓存"，由于写库和写缓存不是原子操作，可能出现 A 先写库、B 后写库、但 B 先写完缓存、A 后写完缓存 的乱序情况——最终缓存里留下的是 A 的（旧的）值，和数据库里 B 的（新的）值不一致，而且这种脏数据会一直留在缓存里，直到下次有人再写这个 key 才会被覆盖，影响面很大。

改成"删除缓存"之后，就算删除操作出现乱序，最坏结果也只是缓存暂时是空的，下一次读请求会触发"查库回填"，用当时数据库里的最新值重新写入缓存——不会出现"缓存里长期停留着一个错误的旧值"的情况。删除是"清空重来"，比"猜测新值该是什么"更安全、更简单。

面试高频："Cache Aside 更新数据为什么是删缓存而不是更新缓存？" 答：并发写场景下更新缓存容易因为写入顺序错乱导致缓存里长期留着脏数据；删除缓存则天然幂等——删了之后无论谁先谁后，最坏情况只是缓存短暂为空，下次读会重新从数据库回填最新值，不会有持久的脏数据。

对比：Read/Write Through 与 Write Behind

Cache Aside 之外，还有两种把"缓存与数据库同步"这件事交给缓存层（而不是业务代码）来做的模式：

模式读流程写流程特点
Cache Aside
旁路缓存		 应用先查缓存，未命中查库并回填 应用先写库，再删缓存 业务代码直接控制缓存和数据库，最灵活、最常用，Redis 场景下的事实标准
Read/Write Through
穿透读写		 应用只和缓存打交道，缓存内部未命中时自己去加载数据库 应用只写缓存，缓存内部同步写数据库 业务代码更简单（不用感知数据库），但需要缓存组件本身支持这种"自动穿透"能力
Write Behind
异步回写		 同 Read Through 应用只写缓存，缓存异步、批量地把变更刷回数据库 写入延迟最低（不用等数据库落盘），但存在数据库落后于缓存的窗口，宕机可能丢数据，一致性要求高的场景要谨慎使用

Redis 本身并不内置"缓存未命中自动查库"的能力，所以 Read/Write Through、Write Behind 在 Redis 场景下通常需要额外的中间层去实现；而 Cache Aside 由业务代码直接掌控读写细节，简单直接，也是本系列后续讨论一致性问题的默认前提。

核心价值总结

  • Cache Aside 是工程实践中的默认选择：读时旁路查询回填，写时先更新库再删缓存。
  • 删除优于更新的核心原因是并发安全：删除是幂等操作，出错的代价是"缓存暂时为空"，而不是"缓存里躺着一个错误值"。
  • 但即便是 Cache Aside，"先更新库再删缓存"的顺序在极端并发下仍可能出现短暂不一致，这个问题会在下一篇详细展开。
