缓存与数据库的一致性问题

缓存和数据库是两份物理上独立的数据副本，只要写操作分两步走，中间就必然存在一个"两者不一致"的时间窗口——问题不是能不能完全消除它，而是怎么把窗口缩到足够短、足够安全。

问题的本质：两次写操作不是原子的

上一篇讲到 Cache Aside 模式下，写操作是"更新数据库 + 删除缓存"两个步骤。这两步无法在一个原子操作里完成（数据库和 Redis 是两个独立的系统，没有跨系统的分布式事务保护）， 这意味着在两步之间，必然存在一个短暂的窗口，此时数据库和缓存看到的数据是不一样的。09-数据库模块讲事务和锁时反复强调"原子性"的重要性，这里正是同样的问题在跨系统场景下的体现——只是这次没有数据库事务机制来兜底。

时序陷阱一：先删缓存，再更新数据库

如果写操作的顺序是"先删缓存，再更新数据库"，并发场景下可能出现下面的时序：

时间写请求 A（更新数据）读请求 B
t1删除缓存中的 key
t2查缓存未命中，转去查数据库，读到旧值（A 还没写完库）
t3把这个旧值写回缓存
t4把数据库更新为新值

结果是：数据库里是新值，但缓存里被 B 塞回去的是旧值，而且这个旧值会一直留在缓存里，直到下次这个 key 再被写、或者自然过期，才会被纠正——这是一个持续时间可能较长的不一致窗口。

时序陷阱二：先更新数据库，再删缓存（更常见，但也不是绝对安全）

Cache Aside 推荐"先更新数据库，再删除缓存"，正是因为它比上面的顺序更不容易出问题——但"更不容易"不等于"绝对不会"。极端并发下依然可能出现类似问题：

时间读请求 B（缓存恰好过期）写请求 A
t1查缓存未命中，转去查数据库，读到旧值
t2更新数据库为新值
t3删除缓存（此时缓存本来就是空的，删了也没用）
t4把 t1 读到的旧值写回缓存

这个时序需要"B 的读请求和回填动作，恰好跨越了 A 的整个写操作"，触发条件比陷阱一苛刻得多（要求数据库写入比缓存回填还快，现实中数据库写通常比 Redis 写慢，所以概率更低）， 但理论上依然存在，这也是为什么"先更新库再删缓存"只能说是更优，而不是完全没有不一致风险。

进阶方案一：延迟双删

针对陷阱二这种小概率场景，一种常见的加固手段是延迟双删：写请求先删一次缓存、更新数据库，然后等待一小段时间（比如几百毫秒，覆盖典型的并发读写窗口）后再删一次缓存。 这样即便中间有读请求把旧值写回了缓存，延迟的第二次删除也能把这个脏数据清理掉。代价是写请求的响应链路变长（或者要额外起一个异步任务去做第二次删除），是"用一点延迟换取一致性窗口更短"的取舍。

进阶方案二：订阅数据库 binlog 异步更新缓存（如 Canal）

另一种思路是不让业务代码直接负责删缓存，而是订阅数据库的 binlog（09-数据库模块讲过，binlog 是 MySQL 记录所有数据变更的日志）， 用类似阿里开源的 Canal 这样的工具伪装成一个 MySQL 从库，实时接收 binlog 变更事件，解析出"哪个表哪一行改了"，再据此去清理或更新对应的缓存 key。

价值：这种方式把"更新数据库"和"维护缓存"彻底解耦——业务代码只管写数据库，不用记得手动删缓存，减少了因为漏写、写错导致的不一致，而且天然是基于数据库已经真正落盘的变更去更新缓存，比业务代码里"猜测性地"删缓存更可靠。代价是引入了额外的中间件和一定的处理延迟。

工程取舍：缓存场景通常追求"最终一致"，而不是"强一致"

看到这里应该能体会到：只要缓存和数据库是两份独立存储，要做到任意时刻两者都完全一致（强一致），要么需要昂贵的分布式事务（严重拖慢性能，违背了用缓存提升性能的初衷），要么根本无法做到。 实际工程中更现实的目标是最终一致性：允许存在一个很短的不一致窗口，但保证这个窗口会在有限时间内（通常是毫秒到秒级）被纠正过来，不会无限期地留着脏数据。

面试高频："如何保证缓存和数据库的一致性？" 优秀的回答不是宣称"能做到强一致"，而是分层次讲：基础方案是 Cache Aside 的"先更新库再删缓存"（比先删缓存更不容易出问题）；针对极端并发窗口可以用延迟双删加固；更彻底的方案是订阅 binlog 异步更新缓存；最后点明"缓存场景的目标通常是最终一致，而不是强一致"，这是对工程权衡的准确认识。

核心价值总结

  • 不一致的根源：写数据库和写缓存是两个独立、非原子的操作，中间必然存在时间窗口。
  • "先更新库再删缓存"优于"先删缓存再更新库"，但不能消除所有不一致场景，只是把发生概率降到很低。
  • 延迟双删和订阅 binlog（如 Canal）是两种不同思路的加固方案：前者是业务代码层面的补救，后者是把一致性维护下沉到基础设施层。
  • 接受"最终一致"是缓存场景下更现实、更成熟的工程判断，而不是一味追求理论上的强一致。

与其他知识点的联系

  • 缓存更新策略：本文的两个时序陷阱都建立在上一篇讲的 Cache Aside 模式之上——"先更新库再删缓存"这个顺序正是上一篇给出的推荐做法，本文进一步说明这个推荐做法为什么"更优"而不是"绝对安全"，是对上一篇结论的深入验证和补充。
  • 主从复制的异步性：如果数据库本身也有主从架构（读写分离），"更新数据库"这个动作在主库完成后，读请求如果恰好落在还没同步到最新数据的从库上，也会读到旧值——这和缓存不一致的成因是同一类"异步复制延迟"问题，只是发生在数据库层而不是缓存层，说明这种时序陷阱在分布式系统里是普遍存在的，不是 Redis 独有。
  • 用 Redis 实现分布式锁：延迟双删方案里"等待一小段时间再执行第二次删除"，如果多个实例都在处理同一个 key 的更新，这个延迟删除的调度本身也可能需要用分布式锁或者去重机制协调，避免同一个 key 被重复地延迟删除，这是本文方案在多实例部署下的一个隐藏细节。
  • 持久化模块（RDB/AOF）：持久化解决的是"Redis 自己的数据在重启后还在不在"，而本文讨论的是"Redis 里的数据和数据库里的数据是否一致"，两者是完全不同维度的问题——即使 Redis 用了最强的持久化方案，也不能保证缓存内容和数据库内容一致，这是初学者容易混淆的两个概念边界。
  • 缓存穿透/击穿/雪崩：缓存空值方案（应对穿透）和延迟双删、订阅 binlog（应对一致性）都在处理"缓存内容什么时候该失效或更新"这个共同主题，只是穿透关注的是"从来没有过的数据"，本文关注的是"曾经有正确值、现在数据变了"，两者结合起来才是缓存内容生命周期管理的完整图景。
