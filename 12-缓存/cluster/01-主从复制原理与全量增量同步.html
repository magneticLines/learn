主从复制原理与全量/增量同步

单个 Redis 实例挂了，数据就没了——主从复制是 Redis 高可用的地基：先复制一份完整数据，再让副本持续跟上主库的每一次写入。

为什么需要主从复制

单机 Redis 有两个致命问题：一是单点故障，进程一挂，内存里的数据和对外服务同时消失；二是读性能瓶颈，所有读写都压在一个实例上。 主从复制（Replication）让一个主库（Master）挂载多个从库（Replica/Slave），从库实时复制主库的数据，天然带来两个好处：读写分离（写主库、读从库，分摊压力）和数据冗余（主库挂了，从库上还有一份完整数据，是后面讲的 Sentinel 自动故障转移的前提）。

类比：主库是"总部账本"，从库是"分部账本副本"。分部第一次开张时，总部把整本账册完整复印一份寄过去（全量同步）；之后总部每记一笔新账，就顺手抄送一份给所有分部（增量同步），分部账本就能和总部保持同步。

全量同步：第一次连接时的完整快照

从库第一次连接主库（或者断线重连但发现数据差异太大）时，会触发全量同步（Full Resync），核心流程：

  1. 从库发送 PSYNC 命令，携带自己的复制历史信息（runid + offset）；
  2. 主库发现是新从库（或历史对不上），返回 +FULLRESYNC，随即执行 BGSAVE 在后台生成一份 RDB 快照；
  3. 关键点：生成快照期间，主库仍在正常处理客户端的写请求，这些写命令不会丢，而是被写入一块复制积压缓冲区（replication backlog，一个环形缓冲区）；
  4. RDB 快照生成完毕后，主库把整个 RDB 文件发送给从库，从库清空自己的旧数据、加载这份 RDB；
  5. 加载完成后，主库再把刚才缓存在积压缓冲区里、快照生成期间产生的写命令，一并发给从库回放，补齐这段时间的差异。

整个过程结束后，从库的数据状态就和主库对齐了，之后就进入增量同步阶段。

增量同步：持续的命令流异步复制

全量同步之后，主库不会再发送 RDB，而是把之后每一条写命令持续、异步地追加发送给所有从库，从库收到命令后在本地重新执行一遍（命令传播）。 这就是增量同步（Command Propagation）——本质上是"主库把自己的写操作日志，实时广播给从库去重放"。

异步的代价：主库执行完写命令会立即返回给客户端，不会等从库确认收到，这意味着主从复制是异步的、存在复制延迟。如果主库刚写完就宕机，还没来得及同步给从库的那部分数据就会丢失。这也是后面讲一致性、分布式锁陷阱时会反复提到的一个坑。

断线重连不用总是全量：部分重同步

如果只是网络抖动导致从库短暂断线，重连后没必要把整个 RDB 再传一遍——只要断开时间不长、积压缓冲区还没被新写命令覆盖掉丢失的那段数据， 主库就可以直接把从库断线期间错过的命令，从积压缓冲区里补发过去，这叫部分重同步（Partial Resync），比全量同步轻量得多。这也是复制积压缓冲区存在的第二个价值。

对比：和 MySQL 主从复制（binlog）的异同

09-数据库模块讲过，MySQL 主从复制的核心思想是：主库把所有变更写入 binlog（二进制日志），从库拉取 binlog 并重放，从而保持数据一致。 Redis 的主从复制在设计思想上高度相似——都是"主库产生一份变更日志，从库拉取并重放"，但落地机制有几处关键差异：

维度MySQL 主从复制Redis 主从复制
复制内容 binlog（可以是行级变更，也可以是 SQL 语句） 写命令本身（命令传播），全量阶段传的是 RDB 快照
初次同步 从库先用 mysqldump 等工具拿到一份基准数据，再从某个 binlog 位点开始追 主库自动 BGSAVE 生成 RDB 快照发给从库，机制内置、无需额外工具
断点续传依据 binlog 文件名 + 偏移量（position），或 GTID replication backlog + offset（复制积压缓冲区里的偏移量）
持久化载体 binlog 落盘持久保存，可长期回溯 积压缓冲区是内存中的环形缓冲区，容量有限，太久断线还是要全量

一句话总结：两者都是"日志流复制"思想的体现，但 MySQL 的 binlog 是持久化的、可无限回溯的日志文件，Redis 的积压缓冲区只是一个内存中的"短期缓冲垫"，超出容量就只能退回全量同步——这也是为什么 Redis 复制更适合"从库紧跟主库"的场景，而不擅长做大范围的时间点回溯。

面试高频："Redis 主从复制第一次连接做什么？" 答：全量同步——主库 BGSAVE 生成 RDB 发给从库，期间的写命令缓存在复制积压缓冲区，之后再补发；后续都是增量同步，即持续异步传播写命令。断线重连优先尝试部分重同步，减少全量开销。

核心价值总结

  • 全量同步解决"从0到1"：RDB 快照 + 积压缓冲区补差，保证从库拿到一份完整且不遗漏的数据。
  • 增量同步解决"持续跟进"：命令传播是异步的，带来延迟和潜在的数据丢失风险，理解这一点是后面理解 Sentinel、Cluster、缓存一致性问题的基础。
  • 和 MySQL 的相似之处是"日志流复制"的设计思想，差异在于持久化程度和复制粒度，这种跨技术栈的类比能帮你更快建立直觉。
