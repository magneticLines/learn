Cluster 分片集群的原理

主从 + Sentinel 解决了"挂了怎么办"，但没解决"数据太多一台机器放不下"——Cluster 用哈希槽把数据横向分散到多个主节点，是 Redis 原生的分片方案。

为什么需要 Cluster

主从复制 + Sentinel 架构下，无论有多少个从库，写入始终只能打到唯一的主库，主库的内存容量和写入吞吐都是整个集群的天花板。 当数据量大到单机内存放不下，或者写并发大到单机扛不住时，就需要把数据水平拆分（分片，Sharding）到多个主节点上，每个主节点只负责一部分数据——这就是 Redis Cluster 要解决的问题。

为什么用哈希槽，而不是直接对节点数取模

最直观的分片思路是：node_index = hash(key) % 节点数量，哪个 key 落在哪个节点，直接算出来。这个思路简单，但有一个致命问题：

取模分片的问题：一旦节点数量发生变化（扩容加一台机器，或缩容减一台机器），% 节点数量 的结果几乎对所有 key 都会变化，意味着绝大多数 key 都要重新计算归属、跨节点搬迁，扩缩容代价极其昂贵，生产环境几乎不可接受。

Redis Cluster 的解法是引入一层间接的映射：把整个 key 空间划分成固定的 16384 个哈希槽（hash slot），每个 key 通过 CRC16(key) % 16384 计算出唯一归属的槽位，然后槽位再分配给具体的节点（比如 3 个主节点，可以分别负责 0~5460、5461~10922、10923~16383 号槽）。

类比：取模分片就像"按学号除以班级数直接分班"——转来一个新班，几乎所有学生的班级都要重排。哈希槽则像"先把学生按学号分到 16384 个固定的储物格，储物格再打包分给各个班"——转来新班，只需要把部分储物格挪给新班，不用打乱所有学生的储物格编号，绝大多数学生的归属完全不受影响。

这样扩缩容时，只需要把一部分槽（连同槽里的 key）从旧节点迁移到新节点，迁移的是槽，不是重新计算所有 key 的归属，代价可控，且可以在业务不停机的情况下逐步完成迁移。

客户端怎么知道该连哪个节点

Redis Cluster 里每个节点都知道完整的"槽位 → 节点"映射表。客户端可以连接任意一个节点发送命令，如果这个 key 恰好不归当前节点管，节点会返回一个 MOVED 重定向响应，告诉客户端"这个槽在另一个节点，请重新请求"。成熟的 Redis 客户端会缓存这份槽位映射，减少重定向带来的额外往返。

对比：Cluster（去中心化分片） vs 主从 + Sentinel（单主写入）

维度主从 + SentinelRedis Cluster
写入吞吐 / 容量上限 受限于单个主库的内存和处理能力 数据分散到多个主节点，容量和吞吐可随节点数横向扩展
架构复杂度 相对简单，一套主从 + 若干 Sentinel 进程即可 更复杂，涉及槽位分配、迁移、多主节点协调
多 key 操作 没有限制，所有 key 都在同一个主库 受限——跨槽的多 key 操作（如某些场景下的事务、MGET 涉及不同槽）不被直接支持，需要用 hash tag 把相关 key 强制分到同一槽
适用场景 数据量单机放得下、追求架构简单、运维成本低的中小型场景 数据量大、并发高、需要水平扩展的大型场景

值得一提的是，Redis Cluster 本身也内置了主从思想：每个负责一部分槽的主节点，通常还会挂自己的从节点，主节点挂了由其从节点自动顶上（类似 Sentinel 的故障转移，但内置在 Cluster 协议里，不需要额外部署 Sentinel 进程）——可以理解为 Cluster = 分片 + 每个分片自带迷你版的高可用。

面试高频："Redis Cluster 为什么用 16384 个哈希槽，而不是直接对节点数取模？" 答：取模分片在节点数变化时会导致几乎所有 key 重新分布，代价太大；哈希槽把 key 空间划分成固定数量的槽，扩缩容只需迁移部分槽，代价可控，且槽的数量（16384）是提前定好的固定值，不随节点增减而变化。

核心价值总结

  • 哈希槽是"取模分片"和"一致性哈希"之外的第三种思路：用固定数量的槽做一层间接映射，扩缩容只搬槽不搬全部 key。
  • Cluster 解决的是容量和吞吐的水平扩展问题，Sentinel 解决的是单点故障的自动恢复问题，两者不是互斥的，Cluster 内部的每个分片本身也具备主从自动切换能力。
  • 选型要看数据规模：不是所有场景都需要 Cluster 的复杂度，单机内存够用、追求简单可维护，主从 + Sentinel 往往是更务实的选择。

与其他知识点的联系

  • 主从复制与 Sentinel：Cluster 里每个负责一部分槽的主节点仍然挂从节点、仍然靠类似 Sentinel 的机制做故障转移，可以把 Cluster 理解成"多组主从复制 + 内置故障转移"再叠加一层分片路由，前两篇讲的复制原理和故障转移思路在这里被直接复用，而不是重新发明。
  • 持久化模块（RDB/AOF）：每个分片节点内部照样要做持久化，Cluster 只解决了"数据分散到哪个节点"的问题，并没有改变单个节点自身如何落盘、如何恢复数据——RDB、AOF、混合持久化的原理在 Cluster 的每一个主节点上原样适用。
  • 用 Redis 实现分布式锁：分布式锁一篇提到的 Redlock 算法思路——"在多个独立节点上加锁，超过半数成功才算拿到锁"——和 Cluster 用哈希槽把 key 分散到多个独立主节点，虽然目的不同（一个为互斥安全，一个为容量扩展），但都体现了"不信任单个节点、用多节点分摊风险"的分布式设计思路。
  • 缓存穿透/击穿/雪崩：Cluster 的多主节点架构天然把压力分散到多台机器，这意味着即便发生缓存击穿或局部雪崩，受影响的往往只是持有对应槽位的那一个节点，而不是整个集群——这是 Cluster 在缓存实战问题上带来的间接收益，但也要注意跨槽的多 key 操作在 Cluster 下会受限，设计缓存 key 结构时需要考虑 hash tag。
  • 哈希槽与一致性哈希的对比：哈希槽是"固定数量槽 + 槽分配给节点"这种间接映射思路的具体实现，本质上和分布式系统里常见的一致性哈希是同一类问题（如何在节点增减时最小化数据迁移）的不同解法，理解了哈希槽，再学一致性哈希会更容易抓住共性和差异。
