哨兵 Sentinel 的自动故障转移

主从复制解决了"有没有备份"，但主库挂了之后，谁来发现、谁来决策、谁来把从库扶正？这就是 Sentinel 要解决的问题——面试高频考点。

为什么需要 Sentinel

有了主从复制，主库挂了从库上还有一份数据，但这只是"数据不丢"，并不等于"服务不断"——客户端原本连的是主库，主库一挂，写请求全部失败， 如果没有人主动把某个从库提升为新主库并通知客户端切换，业务就要一直等运维人工介入。 Sentinel（哨兵）就是 Redis 官方提供的一套自动化监控 + 决策 + 故障转移系统，让主从架构具备自我修复能力。

类比：Sentinel 好比小区里的多个保安轮流巡逻，每人定期给"业主大群"（主库）打电话确认对方是否还在。如果只有一个保安打不通电话就断定业主失联未免草率（可能只是他自己手机没信号），但如果大多数保安都打不通，那就有理由相信业主真的出事了，需要启动应急预案（选一个新负责人顶上）。

主观下线 vs 客观下线：避免误判的核心设计

Sentinel 会周期性地对主库、从库以及其他 Sentinel 发送 PING 命令，判断节点是否存活。这里的核心设计难点是：网络抖动很常见，单个 Sentinel 的一次判断并不可靠，如果直接根据一个 Sentinel 的观察就发起故障转移，很容易发生误判（主库其实还活着，只是这个 Sentinel 自己网络不好）。 为此 Redis 把下线判断拆成了两个阶段：

阶段触发条件含义
主观下线
（Subjectively Down, SDOWN）		 某一个 Sentinel 在指定超时时间内，PING 不通某个节点 只是"我觉得它挂了"，纯粹是这一个 Sentinel 的单方面判断，不代表事实
客观下线
（Objectively Down, ODOWN）		 足够多（达到配置的 quorum 法定人数）的 Sentinel 都认为该节点主观下线 "大家都这么觉得"，才真正采信，触发故障转移流程

这是一个非常经典的用多数派共识来消除单点误判的设计思路，在分布式系统里反复出现（类似的思想在很多共识协议中都能看到）。只有主库达到客观下线，Sentinel 集群才会真正启动故障转移；从库主观下线通常只是记录状态，不会触发转移。

多个 Sentinel 之间如何达成一致：Raft 选举 Leader

判断出主库客观下线后，还有一个问题：谁来牵头执行故障转移？ 如果每个 Sentinel 都自己动手选新主库，可能会选出不同结果，造成混乱。 Redis Sentinel 借助 Raft 算法思想在多个 Sentinel 之间选出一个 Leader，由这个 Leader 独自负责执行故障转移的具体步骤，其他 Sentinel 只负责监控和投票，不参与实际操作。

要点：Raft 选举保证了"故障转移由谁来做"这件事本身也是有共识、不冲突的——这是 Sentinel 集群本身高可用的关键（Sentinel 集群自己也要能容忍部分节点失联）。

故障转移的完整流程

选出 Leader Sentinel 之后，故障转移大致分四步：

  1. 挑选新主库：在存活的从库中，按优先级（slave-priority）、复制进度（谁的数据和旧主库最接近）、runid 大小等规则，挑出一个最合适的从库；
  2. 提升为主库：对选中的从库执行 REPLICAOF NO ONE，让它脱离从库身份，变成独立的主库；
  3. 其余从库改投新主：让其他从库执行 REPLICAOF 指向新主库，重新建立复制关系（这里同样会经历一次全量或部分重同步）；
  4. 通知客户端：Sentinel 更新自己维护的"当前主库是谁"的配置，客户端（配置了 Sentinel 地址而非直连主库地址）通过订阅 Sentinel 的事件通知，或者下次查询时得到新主库地址，从而自动切换写入目标。
常见误区：客户端不应该在配置文件里写死主库 IP，而应该配置 Sentinel 的地址列表，通过 Sentinel 查询"当前谁是主库"。如果写死主库 IP，Sentinel 就算完成了故障转移，客户端也不知道主库已经换人，业务照样中断——这是很多人上线 Sentinel 之后仍然踩坑的地方。
面试高频："Sentinel 怎么判断主库真的挂了？" 答：先由单个 Sentinel 判定主观下线，再由多个 Sentinel 达成共识判定客观下线，避免网络抖动导致的误判；随后通过类 Raft 选举出 Leader Sentinel 执行故障转移：选新主、提升、其余从库改投、通知客户端。

核心价值总结

  • 主观下线 vs 客观下线：用"多数派共识"取代"单点判断"，是分布式系统里防误判的经典设计。
  • Raft 选举 Leader：保证故障转移这个动作本身有唯一的执行者，Sentinel 集群自身也具备高可用性。
  • Sentinel 只解决"主库挂了怎么办"，不解决"数据量太大放不下"——这是下一篇 Cluster 分片集群要解决的问题。
