日志体系

ELK 技术栈简介

结构化日志解决了"日志能不能被机器读懂"的问题，ELK 解决的是下一个问题："几十上百台机器里的日志，怎么统一采集、存储、检索、可视化"。

为什么需要集中式日志系统

在单体应用时代，排查问题登录一台服务器、tail -f 一个日志文件就够了。但在微服务架构下（呼应分布式与微服务模块）， 一个系统可能拆成几十个服务，每个服务又部署了多个实例，日志文件分散在几十上百台机器的本地磁盘上。

这时候"人工登录每台机器查日志"这条路已经彻底走不通了：一次跨服务的故障排查，可能需要依次登录 10 台机器、执行 grep、再手动按时间对齐——效率极低，还容易漏看。 必须有一个集中式的系统，把所有实例的日志实时采集起来，统一存储、统一索引、统一检索，这就是 ELK 技术栈要解决的问题。

类比：集中式日志系统就像机场的行李追踪中心——每一件行李（每条日志）从各个航站楼（各个实例）产生后，都会被扫描上传到中心系统，工作人员不需要跑遍每个航站楼去人工翻找，在中心系统里输入行李条码（traceId）就能查到它经过的每一个环节。

ELK 三大组件各自的职责

ELK 是 Elasticsearch、Logstash、Kibana 三个开源项目首字母的组合，各自负责日志生命周期里的一段：

应用实例（几十~上百个）
  └─ 输出结构化日志到本地文件
      ↓
Filebeat / Logstash（采集 + 传输）
  ├─ Filebeat：轻量级采集器，常驻每台机器，只做"读文件 + 转发"
  └─ Logstash：更重的处理管道，可以做字段解析、过滤、格式转换
      ↓
Elasticsearch（存储 + 索引 + 检索）
  └─ 分布式搜索引擎，为每个结构化字段建立倒排索引，支持海量数据的近实时查询
      ↓
Kibana（可视化 + 查询界面）
  └─ Web 界面，提供检索、过滤、图表、仪表盘，供人查询和排查问题

Elasticsearch：分布式搜索引擎

ES 是整个技术栈的存储核心，本质是一个基于 Lucene 的分布式搜索引擎。它把每条日志当作一个文档，为 level、service、traceId 等结构化字段建立倒排索引， 这样"查询过去一小时 order-service 里 traceId=8f3a2b1c 的所有日志"这类查询能在海量数据（动辄几十亿条）里做到秒级返回。分布式体现在数据会按分片（shard）打散存储在多个节点上，既解决了单机存不下的容量问题，也支撑了并行检索。

Logstash / Filebeat：采集与传输

日志文件躺在每台机器本地磁盘上，需要有一个组件把它们"搬运"到 Elasticsearch。Filebeat 是一个用 Go 写的轻量级采集代理，占用资源极小，适合大规模部署在每个实例上，职责单一——监控日志文件变化、读取新增内容、转发出去。 Logstash 功能更强大也更重，除了传输还能做复杂的字段解析、过滤、脱敏、格式转换，常见架构是 Filebeat 采集 → Logstash 加工 → Elasticsearch 存储，把"轻量常驻"和"重量级处理"分开部署。

Kibana：可视化查询界面

Kibana 是面向人的 Web 界面，提供关键字搜索、字段过滤、时间范围筛选，还能把日志数据做成折线图、饼图、仪表盘（Dashboard）。 排查问题时，工程师直接在 Kibana 里按 traceId 过滤，就能看到这次请求跨越的所有服务实例打印的日志，按时间顺序排列，而不需要知道这次请求具体落在了哪台物理机器上。

常见误区与面试高频考点

误区：认为上了 ELK 就不需要考虑日志采样和保留策略。日志集中存储之后，存储和索引成本会变得非常显著——ES 的存储成本和查询性能都和数据总量强相关，实践中通常需要配合"日志分级保留"（ERROR 保留更久、DEBUG 只保留几天）和"索引生命周期管理（ILM）"，而不是无脑全量永久保存。
面试高频："ELK 各组件的作用是什么？为什么要拆成三个组件？" 回答要点：Filebeat/Logstash 负责采集传输、Elasticsearch 负责存储和检索、Kibana 负责可视化——拆分是典型的关注点分离，采集端要轻量常驻、存储端要能水平扩展、展示端要灵活易用，合在一起做不到三头兼顾。
延伸：结构化日志（上一篇）是 ELK 能发挥威力的前提——如果日志是非结构化文本，Elasticsearch 也能存，但只能做全文检索，没法按字段精确过滤和聚合，等于浪费了 ES 真正的索引能力。

核心价值总结

  • 集中式日志系统是分布式系统的刚需：规模一大，人工登录机器查日志的方式必然失效。
  • ELK 三个组件各司其职：采集传输、存储检索、可视化查询，组合起来构成完整的日志生命周期管理。
  • 日志体系只是可观测性的第一根支柱：它擅长回答"具体发生了什么"，但要回答"系统整体健康状况如何"，还需要指标监控——这是下一个 Tab 要讲的内容。

与其他知识点的联系

  • 日志分级：ELK 能不能发挥威力，很大程度取决于喂给它的日志质量——如果生产环境不做分级控制、DEBUG 也全量采集，ES 的存储和索引成本会随着日志量线性甚至更快地增长，分级过滤是控制 ELK 成本的第一道闸门。
  • 结构化日志：ELK 的核心能力是按字段精确检索和聚合，这必须建立在日志已经是结构化（JSON/键值对）的前提下——非结构化文本日志存进 ES 也只能退化成全文检索，等于浪费了 Elasticsearch 真正擅长的倒排索引能力，这也是本文提到"结构化日志是 ELK 发挥威力的前提"的原因。
  • 指标监控（Metrics）：ELK 擅长回答"具体发生了什么"（靠 traceId 精确定位单次请求的完整上下文），但天然不适合做长周期的趋势观察和阈值告警——这正是指标监控要接管的领域，两者在可观测性里分工明确、互不替代。
  • 链路追踪：Kibana 里按 traceId 过滤查看一次请求跨服务的完整日志，这个能力本身依赖链路追踪体系提前把 TraceID 注入到每一条日志里——没有链路追踪打好的"关联字段"基础，ELK 再强大也只能按时间和关键字模糊拼凑，做不到精确串联。
  • 高基数问题（指标设计）：ELK 里可以任意按 userId、orderId 这类高基数字段做检索，而这恰恰是指标系统的软肋（会引发基数爆炸）——这也是"该用日志还是该用指标"的天然分界线：需要精确检索单个事件用日志/ELK，需要低基数聚合统计用指标。
