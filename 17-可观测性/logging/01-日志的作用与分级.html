日志体系

日志的作用与分级

日志是排查线上问题最基础、成本最低的手段——但"基础"不代表"简单"，级别用错、日志量控制不好，反而会让排查问题变得更难。

日志解决了什么问题

程序运行起来之后，它内部到底发生了什么，外部是看不到的——这就像一辆行驶中的汽车，仪表盘之外的发动机运转细节，司机全靠听声音、看仪表来猜。 日志就是程序主动留下的"行车记录仪"：某个时间点、某个方法、发生了什么事、带着什么数据。 没有日志，出了问题只能靠猜、靠复现、靠加断点本地调试——但线上环境很多问题根本无法在本地复现（依赖特定的数据、并发时序、网络状况），日志往往是唯一的线索来源。

类比：日志之于程序，相当于飞机的黑匣子之于飞行事故调查——事故（故障）发生后，调查员（开发/运维）靠回放黑匣子里记录的一系列事件，还原事发经过。日志记得越准确、越完整，故障排查就越快。

日志级别：从 DEBUG 到 FATAL

日志级别（Log Level）的本质是给每条日志标注"重要程度"和"目标读者"，方便按需过滤。主流日志框架（Logback、Log4j2、java.util.logging）基本都遵循下面这套从低到高的级别体系：

级别使用场景典型例子
DEBUG 开发调试用的细节信息，帮助追踪代码执行路径和中间变量 DEBUG: 查询参数 userId=1001, pageSize=20
INFO 记录系统正常运行中的关键节点，证明"系统按预期在工作" INFO: 订单 #20260703001 创建成功
WARN 出现了不正常但暂不影响系统继续运行的情况，是"苗头" WARN: 第三方接口响应耗时 4.2s，超过预期阈值
ERROR 某个操作/请求失败了，但系统整体还能继续对外提供服务 ERROR: 支付回调处理失败，订单号 #20260703001
FATAL 致命错误，系统即将或已经无法继续运行（部分框架用 ERROR 顶格代替） FATAL: 数据库连接池初始化失败，服务无法启动
日志级别使用示例（Slf4j + Logback）
public class OrderService {
    private static final Logger log = LoggerFactory.getLogger(OrderService.class);

    public void createOrder(OrderRequest req) {
        // DEBUG：只在排查问题时打开，记录详细入参，生产环境默认关闭
        log.debug("创建订单入参: {}", req);

        // INFO：记录关键业务节点，生产环境常开，量要克制
        log.info("订单创建成功, orderId={}", order.getId());

        try {
            paymentClient.notify(order);
        } catch (TimeoutException e) {
            // WARN：不影响主流程（订单已创建），但要留意苗头
            log.warn("支付通知超时, orderId={}", order.getId(), e);
        } catch (Exception e) {
            // ERROR：这次操作确实失败了，需要人工介入或告警
            log.error("支付通知失败, orderId={}", order.getId(), e);
        }
    }
}

为什么生产环境要合理控制日志级别

这是一个典型的两难：日志开太多和开太少，都会让排查问题变得困难，只是困难的方式不同。

日志开太多（生产环境开 DEBUG）的问题： 每个请求打印几十条 DEBUG 日志，看起来"信息很全"，实际上会带来两个致命问题： 一是性能开销——高并发下频繁的字符串拼接、磁盘 I/O、日志采集网络传输会实实在在拖慢系统，甚至日志本身就能把磁盘写满； 二是"信噪比"太低——真正出问题时，有用的那一条 ERROR 淹没在成千上万条 DEBUG 里，人工翻找如同大海捞针，日志系统的检索和存储成本也会飙升。
日志开太少（关键路径完全没日志）的问题： 只有几条稀疏的 INFO，出问题时想还原现场却发现关键的中间状态、异常堆栈全都没有记录，只能靠猜测和临时加日志重新发布、等待问题复现——排查周期可能从几分钟拖成几小时甚至几天。
实践中的折中方案：生产环境默认开 INFO 及以上（正常运行留痕 + 异常告警），关键异常路径打印完整堆栈和上下文；同时保留"动态调级"的能力——通过配置中心或 JMX，在需要排查问题时临时把某个包/某个类的日志级别调到 DEBUG，排查完再调回去，而不需要重启或重新发布应用。

常见误区与面试高频考点

误区一：把 WARN 和 ERROR 混用，比如把所有 catch 到的异常都打成 ERROR。这会导致告警系统被"狼来了"式的噪音淹没——真正需要人介入的 ERROR 被大量无关紧要的 WARN 级别问题掩盖。区分标准很简单：这次失败是否影响了当前请求/操作的最终结果？影响了才是 ERROR，只是有点异常但结果依然正确/有兜底方案就是 WARN。
误区二：在循环体或高频调用路径里直接用字符串拼接打日志，比如 log.debug("id=" + id)。即使 DEBUG 级别没开，这行代码依然会执行字符串拼接的开销。正确写法是用占位符 log.debug("id={}", id)，框架内部会先判断级别是否启用，未启用则完全跳过参数求值和拼接。
面试高频："生产环境该开什么日志级别？为什么不直接全开 DEBUG 图方便？" 回答要点：默认 INFO，性能与信息量的折中；DEBUG 全开会拖累性能且信息过载难以定位问题；应该借助动态调级能力按需临时开启。

核心价值总结

  • 日志是排查问题的第一道防线：没有其他任何手段能像日志一样精确还原代码执行时刻的上下文。
  • 级别是过滤噪音的关键旋钮：合理分级让"平时安静、出事详细"成为可能。
  • 但日志本身是非结构化文本，规模一大就难以检索分析——这正是下一篇"结构化日志"要解决的问题。
