高基数问题与指标设计

给指标加标签很方便，能按维度细分查询——但如果标签的取值空间太大，看似无害的一个字段就可能把时序数据库的存储和查询拖垮。

标签（Label）：让指标可以按维度切片

Prometheus 这类时序数据库允许给指标打上标签，比如 http_requests_total{method="GET", status="200", endpoint="/api/order"}。 标签的价值在于可以按任意维度组合过滤和聚合——"只看 /api/order 这个接口的错误率"，或者"按 status 分组统计请求数"，这些查询都要靠标签实现。

但标签不是免费的：标签的每一种不同取值组合，都会在时序数据库里独立开辟一条新的时间序列。这就引出了高基数问题的根源。

什么是高基数（High Cardinality）问题

"基数"指的是一个标签可能取值的数量。status 标签只有 200、404、500 等寥寥几种取值，基数很低； 但如果有人把 userId 当成标签打到指标上，比如 http_requests_total{userId="1001", endpoint="/api/order"}， 问题就来了：系统里可能有几十万甚至上千万个不同的 userId，每一个 userId 都会和其他标签组合，生成一条独立的时间序列。

高基数标签导致的时间序列爆炸
// 假设：10 个 endpoint × 3 个 status × 50 万个 userId
// 时间序列总数 = 10 × 3 × 500000 = 1500 万条独立时间序列
// 而不加 userId 时，只需要 10 × 3 = 30 条时间序列

http_requests_total{endpoint="/api/order", status="200", userId="1001"}
http_requests_total{endpoint="/api/order", status="200", userId="1002"}
http_requests_total{endpoint="/api/order", status="200", userId="1003"}
// ... 数十万条类似的序列

时序数据库需要为每一条独立的时间序列单独维护索引和存储块，序列数量从 30 条暴涨到 1500 万条，会直接导致内存占用暴涨（时序数据库通常把活跃序列的索引放在内存里）、写入吞吐下降、查询变慢甚至查询超时，严重时可以把整个 Prometheus 实例拖垮。这在 SRE 圈子里有个专门的名字，叫作"基数爆炸"（Cardinality Explosion）。

类比：这就像给图书馆按"书名首字母"分类，一共 26 个分类，管理员轻松应对；但如果改成按"每一本书的独立 ISBN 编号"分类，相当于给每一本书单独开一个分类抽屉——分类系统本身的管理开销远远超过了它带来的检索价值。userId 这种高基数字段就是"每本书一个抽屉"式的滥用。

指标标签设计原则：避免使用高基数字段

设计指标标签时，一个基本准则是：标签的取值空间应该是有限且可枚举的，比如 HTTP 方法（GET/POST/...）、状态码分类（2xx/4xx/5xx）、服务名、接口路径模板（注意是模板 /api/order/:id 而不是把具体 id 拼进路径）。 像 userId、订单号、IP 地址、traceId 这类取值几乎无穷多的字段，天然不适合做指标标签。

场景需求该用哪种可观测性手段原因
"这个接口整体错误率是多少" 指标（Metrics） 低基数聚合查询，指标系统的强项
"用户 1001 的这次请求为什么失败了" 日志（Logging）+ 链路追踪（Tracing） 需要定位到具体一次请求的完整上下文，高基数、需要精确查询单个事件

换句话说，需要按高基数字段做精确查询的场景，本身就不该用指标系统解决——日志系统（如 Elasticsearch）天生就是为高基数、可任意字段检索的场景设计的，链路追踪则专门解决"追踪某一次具体请求"的问题，指标系统的价值在于"快速聚合统计"，而不是"精确检索单个个体"。

常见误区与面试高频考点

误区：为了方便排查问题，把尽可能多的信息都塞进指标标签，觉得"标签越细，以后查起来越方便"。这是从日志思维直接照搬到指标系统上的典型误区——日志一条一条独立存储，加字段的边际成本很低；但指标是按标签组合聚合存储的时间序列，加一个高基数标签的边际成本是指数级的。
常见踩坑点：把 URL 全路径（包含具体参数）直接当标签，比如 /api/order/20260703001 而不是归一化成 /api/order/:id 模板。很多 HTTP 监控中间件默认这么做，如果不做路径归一化处理，订单号的数量就是这个标签的基数，规模上来之后同样会引发基数爆炸。
面试高频："什么是高基数问题？怎么规避？" 回答要点：标签的不同取值组合数决定了时间序列数量，高基数标签（如 userId）会导致存储爆炸和查询变慢；规避方法是只给低基数、可枚举的字段打标签，需要按用户/请求精确查询的场景改用日志或链路追踪。

核心价值总结

  • 标签让指标可以按维度切片查询，但每个标签组合都是一条独立的存储代价：基数越高，代价越大。
  • 指标系统适合低基数的聚合统计，不适合高基数的精确检索：这是它和日志系统的本质分工差异。
  • 日志和指标各自解决了可观测性的一部分问题，但都无法回答"一次请求跨越了哪些服务、耗时分布在哪一段"——这正是下一个 Tab 链路追踪要解决的问题。
