RED 方法与 USE 方法论

有了 Prometheus 这样的采集工具，接下来的问题是："到底该监控哪些指标？" 面对成百上千个可能的指标，RED 和 USE 提供了两套不同角度的取舍框架。

为什么需要一套"该监控什么"的方法论

刚接触监控的人容易走进两个极端：要么把能采集到的指标全部打上仪表盘，导致信息过载、真正重要的信号被淹没；要么只监控几个自己想到的指标，遗漏了关键的健康信号。 RED 方法和 USE 方法论都来自一线运维经验的总结（RED 由 Tom Wilkie 提出，USE 由 Brendan Gregg 提出），核心价值是把"我该看什么"从主观拍脑袋变成一套可复制的检查清单。

RED 方法：面向"请求驱动型"服务

RED 针对的是对外提供请求响应的服务，比如一个 HTTP API、一个 RPC 服务。它回答的核心问题是："这个服务好不好用？"

维度含义典型指标
Rate 请求速率：单位时间内收到多少请求 http_requests_total 的增长速率（QPS）
Errors 错误率：这些请求里有多少失败了 http_requests_total{status="5xx"} 占比
Duration 响应耗时：处理一个请求花了多长时间（通常看分位数而非平均值） http_request_duration_seconds 的 P50/P95/P99
为什么看分位数而不是平均值：平均耗时很容易被"大多数请求很快、少数请求极慢"的分布掩盖——1000 个请求里 990 个 10ms、10 个 5000ms，平均值大约是 60ms，看起来很健康，但已经有 1% 的用户在遭受几乎 5 秒的延迟。P95/P99 这类分位数能暴露"长尾"问题，这也是 SRE 实践里衡量服务质量的常见做法。

USE 方法：面向"资源"

USE 针对的是基础设施资源，比如 CPU、内存、磁盘、网络带宽、数据库连接池。它回答的核心问题是："这些资源够不够用？"

维度含义典型指标
Utilization 使用率：这个资源有多少比例正在被使用 CPU 使用率、内存占用率
Saturation 饱和度：资源排队等待的程度，超出其处理能力的部分 CPU 运行队列长度、连接池等待队列长度
Errors 错误数：资源层面发生的错误事件 磁盘 I/O 错误数、网卡丢包数
陷阱：只看 Utilization 容易产生误判——CPU 使用率 70% 看起来"还有余量"，但如果此时运行队列已经排起了长队（Saturation 很高），说明系统其实已经处理不过来了，只是恰好没有 100% 打满。饱和度往往比使用率更早暴露"快撑不住了"的信号。

RED vs USE：回答不同的问题，用于不同的对象

类比：RED 方法像餐厅的"顾客体验"报表——今天来了多少客人（Rate）、有多少人投诉（Errors）、上菜等了多久（Duration）；USE 方法像餐厅的"后厨资源"报表——炉灶用了多满（Utilization）、有多少订单在排队等着做（Saturation）、有没有食材损坏或设备故障（Errors）。两张报表回答的是完全不同层面的问题，但结合起来才能完整判断这家餐厅是否运转正常。
维度RED 方法USE 方法
监控对象 请求驱动型服务（API、微服务） 基础设施资源（CPU/内存/磁盘/网络）
回答的问题 服务好不好用（对外视角） 资源够不够用（对内视角）
典型使用者 业务方、SRE 关注 SLO 达成情况 基础设施运维、容量规划

实践中两者是互补关系而非二选一：当 RED 指标显示某个服务的 Duration 突然升高，下一步排查往往就要切到这个服务所在实例的 USE 指标，看是不是 CPU 或内存资源先出现了瓶颈——RED 告诉你"哪里出了问题"，USE 帮你回答"为什么会出问题"。

常见误区与面试高频考点

误区：把 RED 方法套用在基础设施资源上，比如问"CPU 的 Rate 是多少"——这没有意义，CPU 不是一个被请求调用的服务。选用哪套方法论，关键看监控对象的性质：对外提供请求处理能力的用 RED，纯资源性质的用 USE。
面试高频："RED 和 USE 方法论的区别是什么？" 回答要点：RED 面向请求驱动的服务，关注速率/错误率/耗时；USE 面向资源，关注使用率/饱和度/错误数；两者分别回答"服务好不好用"和"资源够不够用"，实践中常配合使用做根因定位。

核心价值总结

  • 方法论的价值是把"该监控什么"标准化：避免遗漏关键信号，也避免仪表盘信息过载。
  • 选择哪套方法论取决于监控对象：服务用 RED，资源用 USE，两者常常配合使用做故障定位。
  • 但方法论没有解决"指标标签怎么设计"的问题——标签设计不当会引发下一篇要讲的高基数问题。

与其他知识点的联系

  • Prometheus / Grafana 的监控体系：RED/USE 只是"该监控什么"的方法论清单，真正把 Rate、Errors、Duration、Utilization 这些维度采集、存储、查询出来，依赖的正是上一篇讲的 Prometheus 拉模式采集体系——方法论指导指标设计，Prometheus 提供落地能力，两者缺一不可。
  • 高基数问题与指标设计：RED 方法里的 Rate、Errors、Duration 天然是低基数指标（按 endpoint、status 分类就够了），但如果照搬业务思维，试图对每个 Duration 指标按 userId 细分统计，就会撞上下一篇要讲的高基数问题——RED/USE 划定了"该监控什么维度"，高基数问题划定了"标签能细到什么程度"，两者共同约束一个健康的指标设计。
  • SLI / SLO / SLA：RED 方法里的 Duration（尤其是 P99 分位数）和 Errors（错误率）几乎是 SLI 最常见的原始来源——一个服务的可用性 SLO 往往就是直接用 RED 方法里的 Rate 和 Errors 换算出来的，RED 方法给 SRE 体系提供了现成的指标候选集。
  • 链路追踪：本文提到"RED 告诉你哪里出了问题，USE 帮你回答为什么会出问题"，但如果问题出现在服务内部某个具体调用环节（比如某次数据库查询变慢），USE 层面的资源指标也未必能精确定位——这时需要链路追踪把一次慢请求拆解成一个个 Span，看耗时具体花在了哪一跳。
  • 日志体系：RED/USE 指标能告诉你"错误率升高了"这个量化事实，但无法说明具体是哪次请求、因为什么原因失败——要还原失败的具体上下文，还是要回到日志系统里按时间和相关标识去检索，指标定位异常区间，日志还原异常细节。
