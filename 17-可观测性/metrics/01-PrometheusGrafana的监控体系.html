Prometheus 与 Grafana 的监控体系

日志回答"发生了什么"，指标回答"系统现在怎么样"——Prometheus + Grafana 是目前云原生生态里最主流的指标监控组合，核心设计是一个反直觉的选择：主动去"拉"数据，而不是等别人"推"过来。

指标：可聚合的数值型时间序列

和一条条独立的日志不同，指标（Metric）是随时间变化的数值，比如"当前 QPS"、"CPU 使用率"、"HTTP 500 错误数"。 每一条指标数据本质上是 (指标名, 标签集合, 时间戳, 数值) 这样一个四元组，成千上万个时间点连起来就是一条时间序列（Time Series）。 相比日志，指标的优势是体积小、能长期保存、天然适合做趋势图和阈值告警——没人会把几十亿条日志画成一张实时曲线图，但指标可以。

Prometheus 的核心设计：拉模式（Pull）

大多数人直觉上会觉得监控数据应该是"被监控的服务主动推送出去"（Push 模式），但 Prometheus 反其道而行之： 每个被监控的服务只需要在本地暴露一个 /metrics 的 HTTP 端点，把当前所有指标以文本格式列出来；Prometheus Server 周期性地（比如每 15 秒）主动去请求这个端点，把数据抓回来存进自己的时序数据库。

/metrics 端点输出示例（文本格式）
# HELP http_requests_total 处理的 HTTP 请求总数
# TYPE http_requests_total counter
http_requests_total{method="GET",status="200"} 128934
http_requests_total{method="GET",status="500"} 23

# HELP jvm_memory_used_bytes 当前 JVM 堆内存使用量
# TYPE jvm_memory_used_bytes gauge
jvm_memory_used_bytes 536870912

为什么选择拉模式

这个设计选择解决了推模式下两个不容易察觉的问题：

1. 天然能感知目标是否存活。拉模式下，如果 Prometheus 连续几次抓取某个目标失败（连不上、超时），本身就是一个强信号——目标大概率已经宕机或者网络不通，Prometheus 会自动把它标记为 DOWN 并可以据此触发告警。而推模式下，服务挂了就直接不再推数据，监控系统很难区分"服务正常但确实没有新指标要上报"和"服务已经挂了"这两种情况。
2. 配置集中在监控端，运维成本更低。要监控哪些目标、抓取频率是多少，这些配置全部集中写在 Prometheus 一端；新增一个被监控的服务，只需要把它加进 Prometheus 的抓取列表，被监控方完全不需要知道监控系统的地址、也不需要引入推送客户端去反过来连接监控系统。这在大规模集群里显著降低了配置分散带来的维护成本。
拉模式的局限：不适合"短生命周期任务"（比如一个跑几秒钟就退出的批处理 Job）——任务可能在 Prometheus 下一次抓取之前就已经结束了，指标根本来不及被抓到。这种场景 Prometheus 生态里用 Pushgateway 作为折中方案：短任务主动把指标推给 Pushgateway 暂存，再由 Prometheus 按正常拉模式去抓 Pushgateway。

拉模式 vs 推模式

维度拉模式（Prometheus）推模式（如 StatsD/Graphite）
存活感知 抓不到即视为异常，天然支持 较弱，需要额外的心跳机制
配置管理 集中在监控端，新增目标改一处配置 分散在每个被监控方，需各自配置推送地址
短生命周期任务 不友好，需借助 Pushgateway 中转 天然友好，任务退出前推一次即可
网络方向 监控系统需要能访问到每个目标（内网友好） 被监控方需要能访问到监控系统（适合出网环境）

整体监控体系：Prometheus + Grafana

被监控服务（暴露 /metrics 端点）
    ↑ 周期性拉取
Prometheus Server
  ├─ 抓取调度：按配置的间隔主动请求各目标的 /metrics
  ├─ 时序数据库（TSDB）：按时间序列存储抓到的数据
  ├─ PromQL：类 SQL 的查询语言，支持聚合、过滤、计算
  └─ Alertmanager：根据规则触发告警（邮件/钉钉/PagerDuty）
      ↓ PromQL 查询
Grafana
  └─ 可视化仪表盘：把查询结果画成曲线图、仪表盘、热力图

Prometheus 负责"采集 + 存储 + 查询 + 告警规则"，而可视化和告警配置的用户界面通常交给 Grafana——它不仅能接 Prometheus，也能同时接 Elasticsearch、MySQL 等多种数据源，在同一个仪表盘里混合展示不同来源的数据，这也是它比 Prometheus 自带的简陋 Web UI 更常被使用的原因。

常见误区与面试高频考点

误区：把指标和日志混为一谈，试图用 Prometheus 存储详细的业务日志。指标系统为"数值型时间序列"做了极致优化，但完全不适合存储非数值的详细上下文（比如某次请求的完整堆栈）——这类信息应该留给日志系统，指标系统只回答"多少"、"多快"、"是否超阈值"这类量化问题。
面试高频："Prometheus 为什么用拉模式而不是推模式？" 回答要点：拉模式能天然感知目标存活状态（抓不到=可能挂了），配置集中在监控端便于管理；代价是不适合短生命周期任务，需要 Pushgateway 兜底。

核心价值总结

  • 指标是量化系统健康状况的基本单位：体积小、可长期保存、适合画趋势图和设置告警阈值。
  • 拉模式是 Prometheus 的核心设计决策：用"抓不到即异常"的简单逻辑换来了存活感知和集中配置的双重好处。
  • 光有采集和存储还不够，"该监控哪些指标"是另一个问题——下一篇的 RED / USE 方法论会给出系统性的答案。

与其他知识点的联系

  • ELK / 日志体系：日志擅长回答"具体发生了什么"，指标擅长回答"系统整体健康状况如何"——两者不是替代关系，日志适合精确检索单个事件的详细上下文，指标适合海量数据下的趋势观察和阈值告警，这是可观测性里日志支柱和指标支柱的核心分工。
  • RED 方法与 USE 方法论：本文解决的是"采集和存储指标"的基础设施问题，但"具体该采集哪些指标"是另一个问题——下一篇的 RED/USE 方法论正是在 Prometheus 这套采集能力之上，给出一套系统性的指标选型清单，两者是"先有工具，再有方法论"的先后关系。
  • 高基数问题与指标设计：Prometheus 的时序数据库对标签基数高度敏感，本文提到的 method、status 这类低基数标签能安全使用，但如果不了解高基数问题就随意加标签（比如加 userId），会直接把 Prometheus 实例拖垮——这是使用 Prometheus 时绕不开的设计约束。
  • 链路追踪：当 Grafana 上的某个指标曲线（比如 P99 延迟）出现异常尖峰时，指标本身只能告诉你"变慢了"，无法回答"慢在哪一个下游调用"——这时需要切换到链路追踪去看具体是哪个 Span 拖慢了整体耗时，指标负责"发现问题"，链路追踪负责"定位问题"。
  • SLI / SLO / SLA：SRE 体系里的 SLI（比如请求成功率、P99 延迟）绝大多数就是直接从 Prometheus 里用 PromQL 查询计算出来的——没有本文讲的这套指标采集体系，SLI/SLO 就没有可靠的数据来源，错误预算的计算更是无从谈起。
