常见链路追踪工具简介（SkyWalking / Zipkin / Jaeger）

理解了 TraceID / SpanID 的原理之后，实际落地不需要自己造轮子。SkyWalking、Zipkin、Jaeger 是目前最主流的三款开源链路追踪系统，选型思路大同小异：接入成本、性能开销、生态集成。

三款主流工具速览

这三款工具解决的核心问题是同一个——采集各服务上报的 Span 数据，还原调用链路树，提供可视化查询界面。区别主要在数据采集方式、社区生态、UI 体验上。

Apache SkyWalking

国产开源项目（现为 Apache 顶级项目），最大的特点是探针无侵入式接入——通过 Java Agent 字节码增强技术，在应用启动时通过 -javaagent 参数挂载探针，不需要修改任何业务代码就能自动采集主流框架（Spring、Dubbo、MyBatis、Kafka 等）的调用链路。UI 界面友好，支持拓扑图、告警、指标看板，中文文档和社区都很活跃，国内公司接入较多。

Zipkin

Twitter 开源的老牌链路追踪系统，架构和实现都非常轻量简单：一个采集器（Collector）+ 一个存储（内存/MySQL/Elasticsearch）+ 一个查询 UI。接入方式通常需要在代码里引入客户端库（如 Brave）做少量埋点配置，不像 SkyWalking 那样"零代码"。胜在简单可控、学习曲线低，适合中小规模系统或者想理解链路追踪基本原理时上手实践。

Jaeger

Uber 开源，目前是 CNCF（云原生计算基金会）的毕业项目，与 Kubernetes、Prometheus、OpenTelemetry 等云原生生态集成度很高。如果技术栈已经是 Kubernetes + Service Mesh（如 Istio），Jaeger 往往是默认选项，因为 Service Mesh 的 Sidecar 可以直接把追踪数据吐给 Jaeger，业务代码几乎不用改动。

选型对比一览

工具接入成本性能开销生态集成适用场景
SkyWalking 低（探针无侵入，挂 Agent 即可） 中等（字节码增强有一定开销） 国内框架生态好，中文社区活跃 Java 技术栈为主、想快速接入且不改代码
Zipkin 中（需引入客户端库，手动/半自动埋点） 低（架构简单、组件少） 生态较老，多语言 SDK 成熟 系统规模不大、想要简单可控的方案
Jaeger 低（配合 Service Mesh 可零代码） 低-中（依赖采样策略） 云原生生态最佳，CNCF 毕业项目 Kubernetes / Service Mesh 技术栈
OpenTelemetry 的出现：近几年 OpenTelemetry（OTel）逐渐成为埋点标准的事实规范——它统一了数据采集的 API 和协议，后端可以自由选择 Zipkin、Jaeger 或其他兼容 OTel 协议的存储/展示系统。也就是说，埋点这一层正在"标准化"，选型的重点逐渐从"用哪个 SDK 埋点"转向"用哪个后端做存储和展示"。

选型思路：三个维度

  • 接入成本：团队愿不愿意为了链路追踪去修改业务代码？如果希望"零侵入"，优先看 SkyWalking（Java Agent）或 Jaeger + Service Mesh 的组合。
  • 性能开销：探针本身会消耗一定 CPU/内存，字节码增强类的方案（SkyWalking）比纯 SDK 埋点（Zipkin）开销略高，但差距在合理配置下通常可以接受，需要结合采样策略（下一篇会讲）一起权衡。
  • 生态集成：技术栈是纯 Java 微服务，还是已经上了 Kubernetes + Service Mesh？前者 SkyWalking 更顺手，后者 Jaeger 更贴合云原生工具链。

常见误区

误区：以为选了某个链路追踪工具，所有调用链路就会"自动"完整可见。实际上，无论哪种工具，跨语言调用、自定义线程池、消息队列等场景都需要额外适配，探针能自动覆盖的通常只是主流框架的标准调用路径。工具解决的是"采集和展示"问题，链路是否完整仍取决于埋点是否到位。
面试高频："SkyWalking、Zipkin、Jaeger 怎么选？" 抓手是三个维度——接入成本（要不要改代码）、性能开销（探针方式）、生态集成（是否用了 Service Mesh）。能说出这三个维度并结合具体场景判断，比背工具参数更能体现理解深度。

核心价值总结

  • 三款工具解决的是同一个问题：采集 Span、还原调用树、提供可视化，区别在接入方式和生态适配。
  • 选型不是比"谁更好"，而是比"跟当前技术栈契合度"：Java 重技术栈选 SkyWalking，追求简单可控选 Zipkin，云原生技术栈选 Jaeger。
  • OpenTelemetry 正在统一埋点标准，理解这一趋势有助于判断未来的技术选型方向。
