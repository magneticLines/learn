链路追踪的采样策略 [补充]

如果每一次请求都完整记录追踪数据，存储和性能开销会随流量线性膨胀——高并发系统一天产生的 Span 数据可能是天文数字。采样策略决定了"到底要不要记录这次请求的追踪数据"。

为什么不能全量追踪

假设一个系统每秒处理 10000 次请求，每次请求平均产生 20 个 Span，全量记录意味着每秒要写入 20 万条 Span 数据，一天下来就是超过 170 亿条。这些数据不仅要存储，还要在采集、传输、写入的每一个环节消耗 CPU 和网络带宽——链路追踪本身也是一种"观测负载"，它自己也会拖慢被观测的系统，这是一个容易被忽视的悖论。

采样（Sampling） 就是在"追踪数据的完整性"和"系统开销"之间做取舍：只记录一部分请求的完整链路数据，而不是全部。关键问题是：用什么规则决定哪些请求被记录、哪些被丢弃？ 这里引出两种主流思路：头部采样和尾部采样。

头部采样（Head-based Sampling）

在请求刚进入系统的入口处（比如网关生成 TraceID 的那一刻），就立即决定这次请求要不要被追踪。最常见的做法是按固定比例随机采样，比如"每 100 次请求随机记录 1 次"（采样率 1%），这个决定一旦做出，会通过 Header 透传给下游所有服务——下游服务看到"不采样"的标记，直接跳过 Span 的记录和上报。

头部采样：入口处一次性决定
// 网关生成 TraceID 时，同时掷骰子决定采样与否
boolean sampled = Math.random() < 0.01; // 1% 采样率
request.header("trace-id", traceId)
       .header("sampled", sampled ? "1" : "0");
// 下游所有服务读取 "sampled" 标记，为 0 就不记录 Span、不上报

优点：实现简单，决策在最早的时间点一次性做完，下游服务不需要额外判断逻辑，对系统开销的控制最直接、最可预测（采样率固定，数据量就固定）。

缺点：决策发生在请求还没执行的时候，根本不知道这次请求后面会不会变慢、会不会报错。如果恰好这次请求命中了慢查询或者触发了异常，但没被抽中采样，这条最有价值的追踪数据就永久丢失了——好比在赛跑开始之前就随机选好"要拍谁的比赛录像"，万一没被选中的那个人恰好摔倒了，这个关键画面就没拍到。

尾部采样（Tail-based Sampling）

尾部采样反过来：先把这次请求的完整链路数据都记录下来（不管采不采样都先收集），等这次请求彻底结束之后，再根据结果决定要不要保留这份数据。判断条件通常是"是否耗时超过阈值""是否返回了错误状态码""是否包含异常堆栈"等。

这样一来，那些真正有排查价值的请求——慢请求、错误请求——几乎不会被漏掉，因为筛选发生在结果已知之后。同样是刚才赛跑的例子，尾部采样相当于"全程录像，比赛结束后只留下有人摔倒或破纪录的那几段"，价值密度高得多。

代价：尾部采样必须在请求执行期间，把所有 Span 数据先临时缓存起来（通常在专门的采集层做聚合缓冲），等链路结束、判断条件成立后再决定落盘或丢弃。这意味着需要额外的临时存储和更复杂的收集架构（要等一次请求涉及的所有 Span 都到齐才能做判断，跨服务的 Span 到达时间还可能不一致），实现和运维成本比头部采样高不少。

两种策略对比

维度头部采样尾部采样
决策时机 请求刚进入系统时 请求完整结束之后
决策依据 随机比例，与请求结果无关 是否慢、是否出错等结果指标
实现复杂度 简单，下游只需读标记 复杂，需要临时缓存全部 Span
数据价值密度 低，可能错过关键的慢/错请求 高，精准保留有排查价值的数据
资源开销 可预测，按比例线性增长 需要额外的缓冲存储和聚合逻辑
实践中的折中：很多团队会把两者结合——头部采样保证正常请求也有一个基础的抽样比例（用于统计整体延迟分布等），同时对错误请求、超时请求做"强制 100% 采样"（这本质上是一种简化版的尾部判断，只是判断条件更前置，比如网关直接对非 2xx 响应强制标记采样）。

常见误区

误区：认为采样率越低越好，"反正大部分请求都是正常的，不用记"。采样率设置需要结合排查需求：如果采样率过低，即使用了尾部采样，统计类的分析（比如某个接口 P99 延迟的整体趋势）样本量也会不足，产生偏差。采样策略的目标是"该留的留下、该省的省下"，而不是单纯追求更低的数据量。
面试高频："头部采样和尾部采样的区别？" 一句话：头部采样在请求开始前就随机决定采不采样，简单但可能漏掉有价值的慢请求/错误请求；尾部采样等请求结束后根据实际结果（是否慢、是否错）决定要不要保留，精准但需要额外的临时存储和更复杂的架构。

核心价值总结

  • 采样是在"数据完整性"与"系统开销"之间的权衡，全量追踪在高并发系统下不现实。
  • 头部采样简单但可能"抽不中"关键请求，尾部采样能精准捕获慢请求/错误请求，但需要更复杂的缓冲架构。
  • 实践中常常两者结合：基础比例的头部采样 + 对错误/超时请求的强制采样，兼顾统计意义和排查价值。
