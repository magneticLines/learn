# 4登录 - 前后端分离式模块登录 (Token/JWT Based)

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"当客户端不再只是浏览器，当服务器变成了无状态的 API 接口。"*

*   **痛点分析:** 
    *   **移动端兼容性:** 原生 App (iOS/Android) 不像浏览器那样天生对 Cookie/Session 支持良好。
    *   **跨域问题 (CORS):** 前后端分离（前端 Vue/React 在 localhost:8080，后端在 localhost:8000），Cookie 的跨域传输配置非常麻烦。
    *   **服务器无状态 (Stateless):** 为了实现微服务的弹性伸缩，我们希望应用服务器本身不存储状态，这样可以随意加减机器而不用担心 Session 丢失。
*   **核心定义:** 
    *   **Token-Based Auth (JWT):** 服务端不保存会话数据。认证成功后，签发一个包含用户信息的加密令牌（Token）给客户端。客户端每次请求都把令牌带上，服务端解密验证。

## 2. 🧩 抽象与心智模型 (The Mental Model)

> **设计视角:** *"服务器不再记账，钞票本身就是凭证。"*

*   **生活类比 (现金 vs 记账):**
    *   **Session 模式** 就像**赊账**。老板（Server）有个账本（Memory），记录“张三欠 100 元”。张三刷脸（Cookie）就行。
    *   **Token 模式** 就像**现金**。老板（Server）不记账。张三把 100 元钞票（Token）给老板，老板验证钞票的防伪水印（Signature）。只要钞票是真的，就提供服务。老板不需要记得张三是谁，只需要认钞票。
    *   **关键点:** 所有的信息（金额、发行方）都写在钞票上了。

*   **抽象设计 (JWT 结构):**
    *   **JSON Web Token (JWT):** `Header.Payload.Signature`
    *   **Header:** 算法类型 (HS256)。
    *   **Payload:** 数据 (UserId: 101, Role: Admin, Expire: 2h)。**注意：这里仅仅是 Base64 编码，不是加密，谁都能看到内容，所以不能放密码！**
    *   **Signature:** 签名。`Hash(Header + Payload + SecretKey)`。防止数据被篡改。

## 3. 🏗️ 系统架构与设计 (System & Architecture)

> **架构视角:** *"用 CPU 计算时间换取内存空间和水平扩展能力。"*

*   **在系统中的位置:** 
    *   **API 网关 (Gateway) 或 拦截器层。**
    *   通常在 HTTP Header 中传递：`Authorization: Bearer <token>`。

*   **关键流程:**
    1.  **Login:** Client 发送账号密码 -> Server 验证 -> Server 使用私钥生成 JWT -> 返回 JWT。
    2.  **Storage:** Client (Vue/React) 将 JWT 存入 `localStorage` 或 `sessionStorage`。
    3.  **Request:** Client 拦截器在每个请求头加入 `Authorization: Bearer ...`。
    4.  **Verify:** Server 解析 Token -> 验签 -> 检查是否过期 -> 提取 User Info -> 放行。

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"工具类封装是关键。"*

```java
// 使用 jjwt 库的工具类示例

import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.SignatureAlgorithm;

public class JwtUtils {

    private static final String SECRET_KEY = "MySuperSecretKey_Don't_Tell_Anyone"; // 密钥
    private static final long EXPIRATION_TIME = 3600_000; // 1小时

    // 1. 生成 Token
    public static String generateToken(String username, String role) {
        return Jwts.builder()
                .setSubject(username)
                .claim("role", role) // 自定义载荷
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION_TIME))
                .signWith(SignatureAlgorithm.HS256, SECRET_KEY)
                .compact();
    }

    // 2. 验证并解析 Token
    public static Claims parseToken(String token) {
        return Jwts.parser()
                .setSigningKey(SECRET_KEY)
                .parseClaimsJws(token)
                .getBody();
    }
    
    // 3. 拦截器中使用
    public boolean verify(String token) {
        try {
            Claims claims = parseToken(token);
            String username = claims.getSubject();
            // 可以进一步检查 Token 是否在黑名单中 (Logout 逻辑)
            return username != null;
        } catch (Exception e) {
            // Token 过期或签名错误
            return false;
        }
    }
}
```

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"无状态的代价是失去了控制权。"*

*   **优点 (Pros):**
    *   **完全无状态:** 服务器可以在任何时候重启，不需要同步 Session。
    *   **跨域友好:** HTTP Header 传输，天然支持 CORS。
    *   **多端适配:** App、小程序、Web 通用。

*   **代价 (Cons) - 续签与注销难题:**
    *   **注销失效:** JWT 一旦签发，在过期之前一直有效。用户点击“注销”，服务器无法物理销毁客户端手里的 Token。
        *   *解决:* 必须引入 Redis 做“黑名单”机制（注销时把 Token ID 放入 Redis，验证时查一下），但这又变回了有状态！
    *   **续签 (Refresh Token):** Token 过期了怎么办？让用户重登体验不好。
        *   *双 Token 机制:* Access Token (短效, 15min) + Refresh Token (长效, 7天)。Access 过期了用 Refresh 换新的。

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"安全与便利的博弈。"*

*   **安全性警示:** 
    *   **XSS 攻击:** 如果 Token 存在 `localStorage`，JS 可以读取，XSS 攻击者可以偷走 Token。
    *   **CSRF 攻击:** 如果 Token 存在 `Cookie` (httponly)，可以防 XSS，但要注意 CSRF。
    *   **最佳实践:** 结合使用。或者使用 BFF (Backend For Frontend) 层处理 Token，前端只保留 Session。

*   **思考题:**
    *   JWT 的 Payload 越来越大会发生什么？(提示：HTTP Header 大小限制，带宽消耗)。
    *   如果密钥 (Secret Key) 泄露了，会发生什么灾难性的后果？如何进行密钥轮换 (Key Rotation)？
