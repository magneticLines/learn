# 3登录 - 后端模块式登录 (Session-Based)

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"HTTP 是无状态协议，但我们需要知道‘谁’发起了这个请求。"*

*   **痛点分析:** 
    *   HTTP 协议设计之初是无状态的（Stateless），请求 A 和请求 B 之间没有关联。
    *   但在 Web 应用中，用户登录后，后续的“加入购物车”、“下单”等操作必须知道当前用户已登录且身份是谁。
    *   我们需要一种机制在客户端和服务器之间维持一种“会话状态”。
*   **核心定义:** 
    *   **Session-Based Login:** 利用服务器端的内存（Session）存储用户状态，并利用浏览器端的 Cookie 存储 Session ID 的一种传统但经典的登录模式。

## 2. 🧩 抽象与心智模型 (The Mental Model)

> **设计视角:** *"服务器是那个记性很好的店员，Cookie 是你的排队号码牌。"*

*   **生活类比 (寄存处):**
    *   你去商场存包。
    *   管理员（**Server**）把你的包放到柜子里，记录下位置（**Session Data**）。
    *   管理员给你一个手牌，上面写着 `1024` 号（**Session ID** / **Cookie**）。
    *   你离开去逛街（**HTTP 无状态**）。
    *   你回来取包时，出示手牌 `1024`。
    *   管理员看一眼手牌，找到对应的柜子，把包还给你。
    *   **关键点:** 包（数据）在管理员那里，你手里只有个号码（ID）。如果你自己伪造个 `1025` 号牌，可能取走别人的包（Session 劫持风险）。

*   **抽象设计:**
    *   **Stateful (有状态):** 服务器必须保存所有在线用户的信息。
    *   **流程:** 
        1.  Login -> Server 创建 Session -> 返回 Set-Cookie: JSESSIONID=xyz。
        2.  Next Request -> Browser 自动携带 Cookie: JSESSIONID=xyz -> Server 查找内存。

## 3. 🏗️ 系统架构与设计 (System & Architecture)

> **架构视角:** *"单体应用的默认选择，分布式系统的挑战点。"*

*   **在系统中的位置:** 
    *   **Web 容器层 (Container Layer):** Tomcat/Jetty 内置了 Session 管理机制。
    *   **应用层:** 开发者通过 `request.getSession()` 操作。

*   **关键组件:**
    *   **Cookie:** 浏览器端的小文本文件，用于携带 Session ID。
    *   **Session Map:** 服务器端的哈希表，Key 是 Session ID，Value 是用户对象。
    *   **JSESSIONID:** Java Web 中默认的 Cookie 键名。

*   **设计模式:**
    *   **代理模式:** 浏览器自动代理了 Cookie 的发送，前端 JS 代码通常不需要手动操作。

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"Servlet API 封装了所有细节。"*

```java
// Spring MVC Controller 示例

@RestController
@RequestMapping("/auth")
public class LoginController {

    // 1. 登录逻辑
    @PostMapping("/login")
    public String login(@RequestBody UserLoginDto loginDto, HttpServletRequest request) {
        // ... 校验用户名密码 ...
        User user = userService.check(loginDto);
        
        if (user != null) {
            // 核心：创建或获取 Session
            // Tomcat 会自动生成 JSESSIONID 并通过 Set-Cookie 头写回浏览器
            HttpSession session = request.getSession(true);
            
            // 将用户信息存入服务器内存
            session.setAttribute("CURRENT_USER", user);
            
            return "Login Success";
        }
        return "Login Failed";
    }

    // 2. 业务逻辑 (需要登录才能访问)
    @GetMapping("/profile")
    public User profile(HttpServletRequest request) {
        HttpSession session = request.getSession(false); // 不自动创建
        
        if (session == null || session.getAttribute("CURRENT_USER") == null) {
            throw new RuntimeException("未登录");
        }
        
        // 从内存中获取用户信息
        return (User) session.getAttribute("CURRENT_USER");
    }
    
    // 3. 登出
    @PostMapping("/logout")
    public String logout(HttpServletRequest request) {
        HttpSession session = request.getSession(false);
        if (session != null) {
            session.invalidate(); // 销毁服务器端的 Session 数据
        }
        return "Logout Success";
    }
}
```

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"水平扩展是 Session 模式的死穴。"*

*   **优点 (Pros):**
    *   **开发极其简单:** 框架和容器帮我们做好了所有工作。
    *   **安全性尚可:** 敏感数据（用户信息）在服务器端，客户端只有 ID。
    *   **控制力强:** 服务器可以随时踢掉某个用户（`session.invalidate()`），立即生效。

*   **代价 (Cons) - 分布式集群问题:**
    *   **Session 不共享:** 如果你有 Server A 和 Server B。用户在 A 登录了，Session 在 A 的内存里。下次请求负载均衡到了 B，B 的内存里没有这个 Session ID，用户被迫重新登录。
    *   **解决方案:** 
        1.  **Session 复制 (Sticky Session):** 效率低，浪费内存。
        2.  **Spring Session + Redis:** 这是一个工业级方案。将 Session 数据从 Tomcat 内存移到 Redis 中集中存储。所有 Server 都去 Redis 查。

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"从内存到缓存的演进。"*

*   **底层原理:** 
    *   **Cookie 属性:** `HttpOnly` (防止 JS 读取，防 XSS), `Secure` (只允许 HTTPS), `SameSite` (防 CSRF)。理解这些属性对于安全至关重要。
    *   **序列化:** 存入 Redis 的对象必须实现 `Serializable` 接口，因为需要网络传输。

*   **思考题:**
    *   如果用户禁用了浏览器的 Cookie，Session 还能工作吗？(提示：URL 重写，`?jsessionid=xxx`，但这有安全风险)。
    *   相比于 JWT，为什么银行等高安全要求的系统往往更倾向于使用 Session 模式？(提示：撤销功能的即时性)。
