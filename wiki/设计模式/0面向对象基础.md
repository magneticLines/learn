# 面向对象编程基础

> 在学习设计模式之前，必须深入理解面向对象的三大特性和思维方式

## 一、什么是面向对象

### 1.1 编程范式对比

| 范式 | 核心思想 | 代码组织方式 |
|------|----------|--------------|
| 面向过程 | 按步骤解决问题 | 函数 + 数据结构 |
| 面向对象 | 按对象职责划分 | 类 = 数据 + 行为 |

**面向过程思维**：把大象装进冰箱分几步？
1. 打开冰箱门
2. 把大象放进去
3. 关上冰箱门

**面向对象思维**：有哪些对象？各自负责什么？
- `冰箱` 对象：负责开门、关门
- `大象` 对象：负责走进去
- `人` 对象：负责协调整个过程

### 1.2 面向对象的核心概念

```
类 (Class)     → 对象的模板/蓝图
对象 (Object)  → 类的实例
属性 (Field)   → 对象的状态/数据
方法 (Method)  → 对象的行为/能力
```

## 二、封装 (Encapsulation)

### 2.1 什么是封装

> **封装 = 隐藏实现细节 + 暴露必要接口**

封装的本质是**信息隐藏**，就像：
- 电视机：你只需要遥控器，不需要了解电路原理
- ATM机：你只需要插卡取钱，不关心内部如何处理

### 2.2 封装的层次

```
┌─────────────────────────────────────┐
│  对外接口 (public)                   │  ← 用户可见
├─────────────────────────────────────┤
│  内部实现细节 (private)              │  ← 用户不可见
│  - 数据如何存储                      │
│  - 算法如何实现                      │
│  - 依赖哪些组件                      │
└─────────────────────────────────────┘
```

### 2.3 代码示例

**❌ 没有封装（暴露内部实现）**

```java
public class BankAccount {
    public double balance;  // 余额直接暴露
}

// 使用时
account.balance = -1000;  // 可以随意设置非法值！
```

**✅ 良好封装（隐藏实现，提供接口）**

```java
public class BankAccount {
    private double balance;  // 私有属性
    
    // 通过方法控制访问
    public void deposit(double amount) {
        if (amount > 0) {
            this.balance += amount;
        }
    }
    
    public boolean withdraw(double amount) {
        if (amount > 0 && balance >= amount) {
            this.balance -= amount;
            return true;
        }
        return false;  // 余额不足
    }
    
    public double getBalance() {
        return this.balance;  // 只读访问
    }
}
```

### 2.4 封装的好处

| 好处 | 说明 |
|------|------|
| **数据保护** | 防止外部代码直接修改内部状态 |
| **降低耦合** | 内部实现变化不影响外部调用者 |
| **易于维护** | 修改内部逻辑只需改一处 |
| **增加灵活性** | 可以在 getter/setter 中加验证、日志等 |

### 2.5 访问修饰符

```java
public    → 所有类可访问
protected → 同包 + 子类可访问
default   → 同包可访问（不写修饰符）
private   → 仅本类可访问
```

**经验法则**：
- 属性尽量用 `private`
- 方法根据需要选择最小可见范围
- 优先用 `private`，需要时再放开

## 三、继承 (Inheritance)

### 3.1 什么是继承

> **继承 = 子类复用父类的属性和方法**

继承表达的是 **"is-a"（是一种）** 关系：
- 狗 is-a 动物
- 轿车 is-a 汽车
- 正方形 is-a 矩形（⚠️ 这个例子其实有争议）

### 3.2 继承的结构

```
        ┌───────────┐
        │   Animal  │  ← 父类/基类/超类
        │  - name   │
        │  + eat()  │
        └─────┬─────┘
              │ extends
    ┌─────────┴─────────┐
    │                   │
┌───┴───┐         ┌─────┴───┐
│  Dog  │         │   Cat   │  ← 子类/派生类
│+ bark()│        │+ meow() │
└───────┘         └─────────┘
```

### 3.3 代码示例

```java
// 父类
public class Animal {
    protected String name;
    
    public Animal(String name) {
        this.name = name;
    }
    
    public void eat() {
        System.out.println(name + " is eating");
    }
    
    public void sleep() {
        System.out.println(name + " is sleeping");
    }
}

// 子类
public class Dog extends Animal {
    
    public Dog(String name) {
        super(name);  // 调用父类构造器
    }
    
    // 子类特有方法
    public void bark() {
        System.out.println(name + " is barking: Woof!");
    }
    
    // 重写父类方法
    @Override
    public void eat() {
        System.out.println(name + " is eating bones");
    }
}
```

### 3.4 继承的注意事项

**✅ 适合继承的场景**
- 真正的 "is-a" 关系
- 子类是父类的特殊化
- 需要复用大量代码

**❌ 不适合继承的场景**
- 仅仅为了复用代码（应该用组合）
- "has-a" 关系（汽车有引擎，不是汽车继承引擎）
- 父类不稳定，经常修改

### 3.5 继承 vs 组合

> **优先使用组合，而非继承**

```java
// ❌ 用继承实现（不推荐）
class Car extends Engine { }  // 汽车不是发动机！

// ✅ 用组合实现（推荐）
class Car {
    private Engine engine;  // 汽车有一个发动机
    
    public void start() {
        engine.ignite();
    }
}
```

**组合的优势**：
- 更灵活，可以运行时切换
- 降低耦合
- 避免继承层次过深

## 四、多态 (Polymorphism)

### 4.1 什么是多态

> **多态 = 同一接口，不同实现**

多态让我们可以用**统一的方式处理不同类型的对象**。

生活中的例子：
- "动物叫" → 狗汪汪叫，猫喵喵叫，不同动物不同叫声
- "交通工具移动" → 汽车开，飞机飞，船航行

### 4.2 多态的三个条件

```
1. 继承关系（或实现接口）
2. 方法重写 (@Override)
3. 父类引用指向子类对象
```

### 4.3 代码示例

```java
// 父类
abstract class Shape {
    abstract double area();  // 抽象方法
}

// 子类1
class Rectangle extends Shape {
    private double width, height;
    
    public Rectangle(double w, double h) {
        this.width = w;
        this.height = h;
    }
    
    @Override
    double area() {
        return width * height;
    }
}

// 子类2
class Circle extends Shape {
    private double radius;
    
    public Circle(double r) {
        this.radius = r;
    }
    
    @Override
    double area() {
        return Math.PI * radius * radius;
    }
}

// 使用多态
public class Main {
    public static void main(String[] args) {
        // 父类引用指向子类对象
        Shape shape1 = new Rectangle(3, 4);
        Shape shape2 = new Circle(5);
        
        // 同样的方法调用，不同的执行结果
        printArea(shape1);  // 输出: 12.0
        printArea(shape2);  // 输出: 78.54...
    }
    
    // 统一处理所有形状
    static void printArea(Shape shape) {
        System.out.println("面积: " + shape.area());
    }
}
```

### 4.4 多态的好处

| 好处 | 说明 |
|------|------|
| **可扩展性** | 新增子类无需修改现有代码 |
| **解耦合** | 调用者不依赖具体实现类 |
| **灵活性** | 运行时动态决定执行哪个方法 |

### 4.5 接口与多态

接口是实现多态的另一种（更推荐的）方式：

```java
// 定义接口
interface Flyable {
    void fly();
}

// 不同类实现同一接口
class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("Bird is flying with wings");
    }
}

class Airplane implements Flyable {
    @Override
    public void fly() {
        System.out.println("Airplane is flying with engines");
    }
}

class Superman implements Flyable {
    @Override
    public void fly() {
        System.out.println("Superman is flying... somehow");
    }
}

// 统一处理
void makeFly(Flyable f) {
    f.fly();  // 多态调用
}
```

## 五、面向对象的思维方式

### 5.1 如何用 OOP 思维分析问题

**四步法**：

```
1. 找名词 → 识别对象（类）
2. 找动词 → 识别行为（方法）
3. 找形容词 → 识别属性（字段）
4. 找关系 → 确定类之间的关联
```

**例子：设计一个简单的图书馆系统**

```
需求：图书馆有多本书，读者可以借书和还书

1. 名词（类）：图书馆、书、读者
2. 动词（方法）：借书、还书
3. 形容词（属性）：书名、作者、是否借出、读者姓名
4. 关系：
   - 图书馆 "有" 多本书 (has-many)
   - 读者 "借" 书 (关联)
```

### 5.2 设计类的原则

```
┌─────────────────────────────────────────────────┐
│  高内聚：类的职责要单一、聚焦                      │
│  低耦合：类之间的依赖要尽可能少                    │
└─────────────────────────────────────────────────┘
```

**高内聚示例**：

```java
// ❌ 低内聚（一个类做太多事）
class User {
    void login() { }
    void sendEmail() { }      // 不应该在 User 类
    void generateReport() { } // 不应该在 User 类
}

// ✅ 高内聚（职责单一）
class User {
    void login() { }
    void logout() { }
}

class EmailService {
    void sendEmail() { }
}

class ReportGenerator {
    void generateReport() { }
}
```

### 5.3 抽象的层次

```
更抽象（接口/抽象类）
    ↓
  Flyable(接口)
    ↓
  Bird(抽象类)
    ↓
  Sparrow(具体类)
    ↓
更具体（实例对象）
```

> **依赖抽象，而非具体** — 这是设计模式的核心思想

### 5.4 常见的类关系

| 关系 | 描述 | UML 符号 | 例子 |
|------|------|----------|------|
| **继承** | is-a | 空心三角箭头 | Dog extends Animal |
| **实现** | can-do | 虚线空心三角 | Bird implements Flyable |
| **关联** | 长期持有 | 实线箭头 | Order → Customer |
| **依赖** | 临时使用 | 虚线箭头 | Controller → Service |
| **组合** | 强拥有 | 实心菱形 | House ◆→ Room |
| **聚合** | 弱拥有 | 空心菱形 | Team ◇→ Player |

## 七、总结

```
┌─────────────────────────────────────────────────────────────┐
│                     面向对象三大特性                          │
├─────────────┬─────────────────┬─────────────────────────────┤
│   封装      │      继承       │          多态                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ 隐藏细节    │ 代码复用        │ 统一接口，不同实现            │
│ 保护数据    │ is-a 关系       │ 运行时动态绑定               │
│ 降低耦合    │ 优先用组合      │ 可扩展性强                   │
└─────────────┴─────────────────┴─────────────────────────────┘
```

**核心心法**：
1. **封装变化** — 把可能变化的部分隔离
2. **面向接口编程** — 依赖抽象，而非具体
3. **组合优于继承** — 更灵活，耦合更低

> 掌握了这些基础，你就可以开始学习设计模式了！设计模式本质上就是对这三大特性的灵活运用。
