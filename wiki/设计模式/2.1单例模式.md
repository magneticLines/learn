# 单例模式 (Singleton Pattern)

> 创建型模式 | 难度：⭐ | 使用频率：⭐⭐⭐⭐⭐

---

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"任何技术都是为了解决特定痛点而诞生的。"*

### 痛点分析

想象你在开发一个系统，有一些对象**在整个程序中只应该存在一个**：

- **数据库连接池** —— 如果创建多个连接池，每个都占用大量资源，数据库会被打爆
- **配置管理器** —— 系统配置只有一份，多个实例可能导致配置不一致
- **日志记录器** —— 多个日志实例可能导致日志写入混乱、文件锁冲突
- **线程池** —— 多个线程池会无节制地创建线程，耗尽系统资源

如果没有单例模式，任何地方都可以随意 `new` 这些对象：

```java
// 灾难场景：每个类都自己创建连接池
class OrderService {
    private ConnectionPool pool = new ConnectionPool(); // 第1个连接池
}
class UserService {
    private ConnectionPool pool = new ConnectionPool(); // 第2个连接池
}
class ProductService {
    private ConnectionPool pool = new ConnectionPool(); // 第3个连接池
}
// 💥 3个连接池，每个都占用资源，数据库扛不住！
```

### 核心定义

**单例模式确保一个类在整个应用程序中只有一个实例，并提供一个全局访问点来获取它。**

两个关键要素：
1. **唯一性** —— 全局只有一个实例
2. **全局访问** —— 任何地方都能拿到这个实例

> **设计视角:** *"不仅要看代码，更要看懂设计。"*

### 生活类比

**地球上只有一个太阳 ☀️**

- 不管你在北京还是纽约，抬头看到的太阳都是**同一个**

另一个更贴切的类比：**公司的 CEO**
- 一个公司只有一个 CEO（唯一性）
- 任何部门需要找 CEO，都是找同一个人（全局访问点）
- 你不能自己"任命"一个新 CEO，只能通过公司的机制获取（控制实例化）

### 抽象设计（黑盒模型）

```
┌─────────────────────────────────────┐
│         Singleton（单例类）           │
│                                     │
│  ┌──────────────────────────────┐   │
│  │  static instance (唯一实例)    │   │
│  └──────────────────────────────┘   │
│                                     │
│  构造函数：private（外部不能 new）    │

└─────────────────────────────────────┘

调用方A ──getInstance()──→ 返回 instance ←──getInstance()── 调用方B

```

**输入：** 无（或者说"获取实例的请求"）
**输出：** 始终返回同一个对象引用
**本质：** 一种**对象创建的控制机制**——把对象的"生产权"从调用方收回到类自身

## 3. 🏗️ 系统架构与设计 (System & Architecture)

> **架构视角:** *"站得高，看得远。"*

### 在系统中的位置

单例模式通常出现在**基础设施层**和**服务层**：

```
┌─────────────────────────────────┐
│         表现层 (Controller)       │
├─────────────────────────────────┤
│         业务层 (Service)          │   ← 业务服务有时用单例
├─────────────────────────────────┤
│         数据层 (Repository)       │
├─────────────────────────────────┤
│    基础设施层 (Infrastructure)    │   ← 单例模式的主战场
│  ┌─────────┐ ┌───────┐ ┌─────┐ │
│  │连接池    │ │日志器  │ │配置  │ │
│  │Singleton │ │Singleton│ │Singleton│
│  └─────────┘ └───────┘ └─────┘ │
└─────────────────────────────────┘
```

### 关键组件

实现单例模式需要三个核心要素：

1. **私有构造函数** —— 禁止外部通过 `new` 创建实例
2. **静态私有实例变量** —— 类自身持有唯一的实例
3. **静态公有获取方法** —— 提供全局访问点 `getInstance()`

### 涉及的设计原则

- **单一职责原则**：单例类自己管理自己的实例创建（有争议，因为它承担了两个职责）
- **开闭原则**：单例模式对扩展不太友好（子类化困难）

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"优雅的代码是清晰思维的体现。"*

### 4.1 饿汉式（Eager Initialization）

> **思想：类加载时就创建实例，"我先吃了再说"**

```java
/**
 * 饿汉式单例
 * 
 * 为什么叫"饿汉"？因为这个类很"饥饿"，
 * 不管你用不用它，它在类加载时就迫不及待地创建好实例了。
 */
public class EagerSingleton {

    // 1. 类加载时就创建实例（JVM 保证类加载过程是线程安全的）
    //    为什么用 static final？
    //    - static: 属于类本身，不属于某个对象
    //    - final: 引用一旦赋值就不能再改变，防止被意外修改
    private static final EagerSingleton INSTANCE = new EagerSingleton();

    // 2. 私有构造函数：封死 new 的入口
    //    如果不写 private，Java 默认是 public，任何人都能 new
    private EagerSingleton() {
        // 防止通过反射创建第二个实例
        if (INSTANCE != null) {
            throw new RuntimeException("不允许通过反射创建单例！");
        }
    }

    // 3. 全局访问点：任何地方都通过这个方法获取实例
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
}
```

**验证：**
```java
EagerSingleton a = EagerSingleton.getInstance();
EagerSingleton b = EagerSingleton.getInstance();
System.out.println(a == b); // true ← 同一个对象
```

**优点：** 实现简单，天生线程安全（JVM 类加载机制保证）
**缺点：** 即使程序从头到尾没用过这个类，实例也会被创建，浪费内存

### 4.2 懒汉式（Lazy Initialization）

> **思想：你需要的时候我再创建，"不急，等你叫我"**

```java
/**
 * 懒汉式单例（非线程安全版本）
 * 
 * 为什么叫"懒汉"？因为它很"懒"，
 * 只有在第一次被调用时才会创建实例。
 */
public class LazySingleton {

    // 注意：这里没有 final，也没有立即赋值
    private static LazySingleton instance;

    private LazySingleton() {}

    // ⚠️ 这个版本在多线程环境下有 Bug！
    public static LazySingleton getInstance() {
        if (instance == null) {           // 线程A 判断为 null
            // 假设此时线程B 也进来了，也判断为 null
            instance = new LazySingleton(); // 两个线程各创建一个，单例被破坏！
        }
        return instance;
    }
}
```

**问题演示：为什么多线程不安全？**

```
时间线：
  线程A                          线程B
  ────────────────────           ────────────────────
  if (instance == null) ✓ true
                                 if (instance == null) ✓ true
  instance = new ...  → 实例1
                                 instance = new ...  → 实例2 💥
  return 实例1                   return 实例2
  
  → 产生了两个不同的实例，单例被破坏了！
```

### 4.3 双重检查锁（Double-Checked Locking, DCL）⭐ 推荐

> **思想：先快速检查，再精确加锁，兼顾性能和安全**

```java
/**
 * 双重检查锁单例 —— 面试高频考点
 * 
 * 设计意图：
 * - 第一次检查：避免每次调用都进入同步块（性能优化）
 * - synchronized 锁：保证只有一个线程创建实例
 * - 第二次检查：防止多个线程同时通过第一次检查后重复创建
 */
public class DCLSingleton {

    // 为什么用 volatile？
    // 防止指令重排序。new 操作分三步：
    // ① 分配内存空间
    // ② 初始化对象
    // ③ 将引用指向内存地址
    // 如果不加 volatile，JVM 可能优化为 ①③②
    // 线程B 可能拿到一个"已分配地址但未初始化"的对象 → 空指针
    private static volatile DCLSingleton instance;

    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        // 第一次检查（无锁，快速路径）
        // 如果已经创建了，直接返回，根本不需要获取锁
        if (instance == null) {

            // 进入同步块（只有第一次调用时才会走到这里）
            synchronized (DCLSingleton.class) {

                // 第二次检查（有锁，安全路径）
                // 为什么要再检查一次？
                // 因为可能有多个线程同时通过了第一次检查，
                // 在排队等锁，第一个进来的创建了实例，
                // 后面的进来后必须再次确认
                if (instance == null) {
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

### 4.4 静态内部类（Holder Pattern）⭐⭐ 最推荐

> **思想：利用 JVM 的类加载机制，既懒加载又线程安全，优雅！**

```java
/**
 * 静态内部类单例 —— 最优雅的实现方式
 * 
 * 设计意图：
 * 利用 Java 类加载机制的特性：
 * - 内部类不会随外部类加载而加载（实现懒加载）
 * - 类加载过程由 JVM 保证线程安全（不需要 synchronized）
 * 
 * 既有饿汉式的线程安全，又有懒汉式的延迟加载，两全其美！
 */
public class HolderSingleton {

    private HolderSingleton() {}

    // 静态内部类：只有在被引用时才会加载
    // JVM 保证类加载过程是线程安全的 → 不需要加锁
    private static class SingletonHolder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }

    public static HolderSingleton getInstance() {
        // 第一次调用 getInstance() 时，才会触发 SingletonHolder 的类加载
        // 后续调用直接返回已创建的实例
        return SingletonHolder.INSTANCE;
    }
}
```

**为什么说它最优雅？**

| 特性 | 饿汉式 | DCL | 静态内部类 |
|------|--------|-----|-----------|
| 懒加载 | ❌ | ✅ | ✅ |
| 线程安全 | ✅ | ✅ | ✅ |
| 代码简洁 | ✅ | ❌（复杂） | ✅ |
| 性能 | ✅ | ✅ | ✅ |

### 4.5 枚举实现（Enum Singleton）⭐⭐⭐ 最安全

> **思想：Java 枚举天然就是单例，连反射和序列化都无法破坏**

```java
/**
 * 枚举单例 —— Effective Java 作者 Joshua Bloch 推荐的方式
 * 
 * 为什么枚举是最安全的？
 * 1. JVM 保证枚举实例只会被实例化一次 → 天生单例
 * 2. 枚举的构造函数是 JVM 内部调用的，反射无法创建新实例
 * 3. 枚举默认实现了 Serializable，但序列化机制保证不会创建新实例
 */
public enum EnumSingleton {

    INSTANCE;  // 这就是唯一的实例，简单到让人怀疑人生

    // 可以添加属性和方法
    private int count = 0;

    public void doSomething() {
        count++;
        System.out.println("执行操作，第 " + count + " 次调用");
    }
}

// 使用方式
// EnumSingleton.INSTANCE.doSomething();
```

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"一切都是权衡 (It depends)。"*

### 优点 (Pros)
- ✅ **全局唯一实例**：保证共享资源的一致性（如配置、连接池）
- ✅ **节约资源**：避免重复创建重量级对象
- ✅ **全局访问点**：方便获取共享对象

### 代价 (Cons)
- ❌ **违反单一职责**：类既负责业务逻辑，又负责管理自己的实例化
- ❌ **难以测试**：单例引入了全局状态，单元测试之间可能互相影响
- ❌ **隐藏依赖**：代码中直接调用 `Xxx.getInstance()` 看不出类的依赖关系
- ❌ **扩展困难**：单例类很难被子类化（构造函数是 private 的）

### 现代开发中的替代方案

在 **Spring** 等框架中，我们通常不手写单例，而是使用框架的 **IoC 容器**：

```java
@Component  // Spring 默认就是单例作用域（singleton scope）
public class UserService {
    // Spring 帮你管理生命周期，你不需要自己写 getInstance()
}
```

**手写单例 vs Spring 单例：**

| | 手写单例 | Spring 单例 |
|---|---|---|
| 管理方式 | 类自己管理 | 容器管理 |
| 测试友好 | 差（全局状态） | 好（可注入 Mock） |
| 灵活性 | 差 | 好（可改为 prototype 等） |
| 适用场景 | 工具类、基础设施 | 业务 Bean |

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"计算机科学是互通的。"*

### 底层原理

**为什么 `volatile` 能防止指令重排序？**

这涉及 **Java 内存模型 (JMM)** 和 **CPU 内存屏障**：

- CPU 为了性能会对指令进行**乱序执行 (Out-of-Order Execution)**
- `volatile` 会在写操作后插入一个 **StoreLoad 屏障**
- 这个屏障告诉 CPU：在这个变量的写操作完成之前，不准执行后面的读操作
- 本质上，这是**软件层面的规约**与**硬件层面的性能优化**之间的博弈

**为什么 JVM 类加载是线程安全的？**

JVM 规范规定：当类被加载时，会获取一个**初始化锁 (initialization lock)**：
- 如果线程 A 正在初始化某个类，线程 B 也要用这个类时，B 会被阻塞
- 直到线程 A 完成初始化，B 才能继续
- 这就是静态内部类单例线程安全的根本原因

### 单例在 JDK 源码中的身影

### 🤔 思考题与解答

#### 思考题 1：集群环境中 JVM 单例还能保证"全局唯一"吗？

**答案：不能。**

每个 JVM 都是独立的进程，有自己独立的内存空间。`Singleton.getInstance()` 在每台服务器上各自返回各自的实例，互不相干。

```
┌─────────────────────────────────────────────────────┐
│                    集群环境                           │
│                                                     │
│  ┌──────────────┐    ┌──────────────┐              │
│  │  服务器 A      │    │  服务器 B      │              │
│  │  JVM 进程 1   │    │  JVM 进程 2   │              │
│  │  Singleton   │    │  Singleton   │              │
│  │  实例 ①      │    │  实例 ②      │  ← 两个不同的实例！
│  └──────────────┘    └──────────────┘              │
└─────────────────────────────────────────────────────┘
```

**分布式环境的解决方案：把"唯一性"从 JVM 内存 → 提升到外部共享存储**

**方案一：基于 Redis（最常用）⭐**

```java
// Redis 天然单线程，操作具有原子性
@Component
public class RedisCounter {
    @Autowired
    private StringRedisTemplate redisTemplate;

    // Redis 的 INCR 命令是原子操作，天生线程安全
    public Long increment() {
        return redisTemplate.opsForValue().increment("global:visit:count");
    }
}
```

```
服务器A ──increment()──→ ┌────────┐
                         │ Redis  │  count = 800 ✅ 全局唯一！
服务器B ──increment()──→ └────────┘
```

**方案二：基于分布式锁（Redisson）**

```java
// 多个节点竞争，只让一个拿到"钥匙"
public void executeUniqueTask() {
    RLock lock = redisson.getLock("global:unique-task-lock");
    try {
        if (lock.tryLock(0, 30, TimeUnit.SECONDS)) {
            // 只有一个节点能执行
            System.out.println("我拿到锁了，执行唯一任务！");
        }
    } finally {
        if (lock.isHeldByCurrentThread()) lock.unlock();
    }
}
```

**方案三：基于 ZooKeeper（Leader 选举）**

适合"集群中只有一个节点执行定时任务"等场景，通过临时节点 + 选举机制实现。

**方案选择指南：**

| 场景 | 推荐方案 |
|------|----------|
| 共享数据/计数 | Redis ⭐ |
| 分布式锁（只一个节点执行任务） | Redisson / ZooKeeper |
| Leader 选举（一主多备） | ZooKeeper / etcd |
| 并发量不高，追求简单 | 数据库（唯一约束 / SELECT FOR UPDATE） |

**本质认知：** 单例的本质不是"一个 JVM 一个实例"，而是"在某个作用域内保证唯一"。作用域从单机扩展到集群时，保证唯一性的手段也必须从 JVM 内部机制升级到分布式协调机制。

#### 思考题 2：单例 vs 静态工具类的区别

**一句话概括：**
- **静态工具类 = 一堆工具（没有身份，没有状态）**
- **单例 = 一个活着的对象（有身份，有状态，可以被替换）**

**六维对比：**

**① 状态（最关键的区别）**

```java
// 静态工具类：无状态，纯计算
public class MathUtils {
    public static int add(int a, int b) {
        return a + b;  // 输入相同 → 输出永远相同
    }
}

// 单例：有状态，维护数据
public class ConnectionPool {
    private List<Connection> pool = new ArrayList<>();  // ← 状态！
    private int activeCount = 0;                        // ← 状态！
    
    public Connection getConnection() {
        activeCount++;
        return pool.remove(0);  // 每次调用结果不同，取决于当前池的状态
    }
}
```

**② 多态与接口（能否"替换"）**

```java
// 静态方法：不能实现接口，不能多态，写死了就是这个类
// 单例：可以实现接口，运行时替换实现
public interface Logger { void log(String msg); }

Logger logger = config.isDebug() 
    ? ConsoleLogger.getInstance()   // 开发环境：输出到控制台
    : FileLogger.getInstance();     // 生产环境：写入文件
logger.log("Hello");  // 多态调用 ✅
```

**③ 单元测试（能否 Mock）**

```java
// 静态方法：难以 Mock（写死了，测试时没法替换）
TaxUtils.calculate(amount);  // ❌ 无法在测试中模拟

// 单例（注入方式）：容易 Mock
OrderService service = new OrderService(amount -> 0.0);  // ✅ 注入 Mock
```

**④ 生命周期管理**

- 静态方法：没有生命周期，没有"初始化"和"销毁"
- 单例：有完整的生命周期（构造时初始化连接，销毁时释放资源）

**⑤ 继承**

- 静态方法：不能被继承/重写（只能"隐藏"）
- 单例：是真正的对象，支持继承

**⑥ 序列化与传输**

- 静态工具类：不是对象，不能序列化，不能作为参数传递
- 单例：是对象，可以传递、序列化

**什么时候用什么？速查表：**

```
你的需求是什么？
    │
    ├── 纯粹的无状态计算？（字符串处理、数学运算、格式转换）
    │       → 静态工具类 ✅（StringUtils, MathUtils, Collections）
    │
    ├── 需要维护内部状态？（连接数、缓存、配置）
    │       → 单例 ✅（ConnectionPool, ConfigManager）
    │
    ├── 需要实现接口或支持多态替换？
    │       → 单例 ✅（Logger, PaymentProcessor）
    │
    ├── 需要被 Mock 测试？
    │       → 单例（最好通过接口注入） ✅
    │
    ├── 需要管理资源生命周期？（初始化/销毁）
    │       → 单例 ✅（线程池、数据库连接池）
    │
    └── 以上都不需要，只是通用工具方法？
            → 静态工具类 ✅（Objects.equals(), Arrays.sort()）
```

**JDK 中的真实例子：**

| 类 | 类型 | 为什么 |
|---|---|---|
| `Math.random()` | 静态方法 | 纯计算，无需维护状态 |
| `Collections.sort()` | 静态方法 | 纯工具操作 |
| `Runtime.getRuntime()` | 单例 | 需要管理 JVM 运行时状态 |
| `Desktop.getDesktop()` | 单例 | 需要操作系统桌面资源 |

**终极判断法：** 问自己「我有没有可能在未来需要替换这个实现？」
- 可能需要 → 用单例 + 接口
- 不可能需要（如 `1+1=2`） → 用静态方法
