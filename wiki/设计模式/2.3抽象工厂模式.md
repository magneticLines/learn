# 抽象工厂模式 (Abstract Factory Pattern)

> 创建型模式 | 难度：⭐⭐⭐ | 使用频率：⭐⭐⭐⭐

---

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"任何技术都是为了解决特定痛点而诞生的。"*

### 痛点分析

上一节我们用**工厂方法模式**解决了"创建单一产品"的问题。但现实世界往往更复杂——

假设你在做一个**跨平台 UI 框架**，需要支持 Windows 和 macOS 两个平台：

```java
// 😰 第一版：直接 new
public class Application {
    public void createUI() {
        // Windows 风格的按钮
        WindowsButton button = new WindowsButton();
        // Windows 风格的文本框
        WindowsTextField textField = new WindowsTextField();
        // Windows 风格的复选框
        WindowsCheckbox checkbox = new WindowsCheckbox();
    }
}
```

要支持 macOS 了，你可能会这样改：

```java
// 💥 灾难代码：平台判断散落在各处
public class Application {
    public void createUI(String os) {
        Button button;
        TextField textField;
        Checkbox checkbox;

        if ("windows".equals(os)) {
            button = new WindowsButton();
            textField = new WindowsTextField();
            checkbox = new WindowsCheckbox();
        } else if ("mac".equals(os)) {
            button = new MacButton();
            textField = new MacTextField();
            checkbox = new MacCheckbox();
        }
        // 💥 如果再加 Linux 呢？每个 if 块都要改！
        // 💥 而且万一写错了，搞出 WindowsButton + MacTextField 的混搭怎么办？
    }
}
```

**问题总结：**

| 痛点 | 描述 |
|------|------|
| **产品不配套** | 手动创建容易"混搭"——Windows 按钮 + Mac 文本框，系统会崩溃 |
| **违反开闭原则** | 新增平台要改所有 if-else 块 |
| **创建逻辑分散** | 相关产品的创建代码没有集中管理 |
| **工厂方法不够用** | 工厂方法每个工厂只创建**一种**产品，但这里需要创建**一组配套**的产品 |

### 核心问题

> 🔑 **工厂方法解决的是"创建一种产品"的问题，但当你需要创建"一组相关的配套产品"时，工厂方法就力不从心了。**

### 核心定义

**抽象工厂模式提供一个接口，用于创建一系列相关或相互依赖的对象，而无需指定它们的具体类。**

核心思想：**保证同一个工厂生产出来的产品是"配套的"，不会出现混搭。**

## 2. 🧩 抽象与心智模型 (The Mental Model)

> **设计视角:** *"不仅要看代码，更要看懂设计。"*

### 生活类比

**宜家家具套装 🛋️**

想象你去宜家买家具：

- **工厂方法的思路：** 你分别去"椅子区"买椅子、"桌子区"买桌子、"沙发区"买沙发 → 每个区域是一个"工厂"，只负责一种产品。但问题是——你可能买了一把北欧风的椅子、一张中式的桌子、一个美式的沙发……**风格完全不搭配！**

- **抽象工厂的思路：** 宜家按"风格系列"来销售 → 你直接选"北欧简约系列"，整套家具（椅子+桌子+沙发）都是该系列的，**风格保证统一**。或者选"现代工业系列"，也是整套配好的。

```
【工厂方法】每个工厂只生产一种产品：
  椅子工厂 → 椅子
  桌子工厂 → 桌子
  沙发工厂 → 沙发
  → 风格混搭？没人管！

【抽象工厂】每个工厂生产一整套产品族：
  北欧系列工厂 → 北欧椅子 + 北欧桌子 + 北欧沙发  ✅ 配套！
  工业系列工厂 → 工业椅子 + 工业桌子 + 工业沙发  ✅ 配套！
```

### 两个关键概念

在学抽象工厂之前，必须理解两个核心概念：

```
                    产品等级结构（Product Hierarchy）
                    ——同一类产品的不同实现
                    
                      按钮        文本框       复选框
                    (Button)   (TextField)  (Checkbox)
                       │           │            │
          ┌────────────┼───────────┼────────────┼────────────┐
          │            │           │            │            │
产品族    Windows    Windows    Windows      Windows      Windows ← 一整套
(Product  按钮       文本框      复选框       ...          风格统一
 Family)  │           │            │
          │            │           │
          Mac        Mac         Mac                     Mac ← 一整套
          按钮       文本框      复选框                    风格统一
```

| 概念 | 含义 | 举例 |
|------|------|------|
| **产品等级结构** | 同一类产品的继承层次（纵向） | Button → WindowsButton / MacButton |
| **产品族** | 不同类但相互配套的产品（横向） | WindowsButton + WindowsTextField + WindowsCheckbox |

> **抽象工厂 = 按"产品族"来组织创建逻辑。一个工厂负责一整套配套产品。**

### 抽象设计（黑盒模型）

```
                        ┌───────────────────────┐
                        │    抽象工厂 (接口)       │
                        │  createButton()        │
                        │  createTextField()     │
                        │  createCheckbox()      │
                        └───────────┬───────────┘
                     ┌──────────────┼──────────────┐
                     ↓              ↓              ↓
              ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
              │ Windows 工厂  │ │  Mac 工厂     │ │ Linux 工厂   │
              │→Win按钮       │ │→Mac按钮       │ │→Linux按钮    │
              │→Win文本框     │ │→Mac文本框     │ │→Linux文本框  │
              │→Win复选框     │ │→Mac复选框     │ │→Linux复选框  │
              └──────────────┘ └──────────────┘ └──────────────┘
                  ↑                                    ↑
              一个工厂                             新增平台？
              生产一整套                           只需新增一个工厂！
```

> **架构视角:** *"站得高，看得远。"*

### 在系统中的位置

抽象工厂模式通常位于**基础设施层**或**业务层底部**，为上层提供一组配套的对象：

```
┌─────────────────────────────────────────┐

├─────────────────────────────────────────┤
│      业务层 (Service)                     │
│      Application                         │
│      → 只依赖抽象工厂接口                   │
│      → factory.createButton()            │
│      → factory.createTextField()         │
├─────────────────────────────────────────┤
│      工厂层 (Abstract Factory) ← 在这里    │
│      ┌───────────────┐ ┌──────────────┐ │
│      │ WindowsFactory │ │  MacFactory  │ │
│      │→WinButton      │ │→MacButton    │ │
│      │→WinTextField   │ │→MacTextField │ │
│      │→WinCheckbox    │ │→MacCheckbox  │ │
│      └───────────────┘ └──────────────┘ │
├─────────────────────────────────────────┤
│      基础设施层                            │
└─────────────────────────────────────────┘
```

### 涉及的设计原则

- **开闭原则 (OCP)**：新增产品族时，只需新增一个工厂类和对应产品类 ✅
- **依赖倒置原则 (DIP)**：客户端依赖抽象工厂和抽象产品接口，不依赖具体实现 ✅
- **单一职责原则 (SRP)**：每个工厂只负责创建一个产品族 ✅
- **接口隔离原则 (ISP)**：抽象工厂接口按产品类型拆分为多个创建方法 ✅

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"优雅的代码是清晰思维的体现。"*

### 4.1 场景设定：跨平台 UI 组件

我们要开发一个跨平台 GUI 框架，支持 **Windows** 和 **Mac** 两个平台。每个平台需要提供配套的 **按钮 (Button)**、**文本框 (TextField)** 和 **复选框 (Checkbox)**。

### 4.2 第一步：定义抽象产品（产品等级结构）

```java
/**
 * 抽象产品A：按钮
 * 
 * 为什么用接口？因为不同平台的按钮实现方式完全不同，
 * 它们只共享 "行为契约"，没有共享的实现逻辑
 */
public interface Button {
    void render();   // 渲染按钮
    void onClick();  // 点击事件
}

/**
 * 抽象产品B：文本框
 */
public interface TextField {
    void render();       // 渲染文本框
    String getValue();   // 获取输入值
}

/**
 * 抽象产品C：复选框
 */
public interface Checkbox {
    void render();         // 渲染复选框
    boolean isChecked();   // 是否选中
}
```

### 4.3 第二步：实现具体产品

### 4.4 第三步：定义抽象工厂

```java
/**
 * 抽象工厂：定义 "创建一整套 UI 组件" 的接口
 * 
 * 关键设计意图：
 * 1. 每个方法对应一种产品（createButton、createTextField、createCheckbox）
 * 2. 一个工厂实例会创建一整套风格统一的产品
 * 3. 客户端只依赖这个接口，不依赖任何具体产品类
 * 
 * 这和工厂方法的区别是什么？
 * → 工厂方法：一个接口只有一个 createXxx() 方法，创建一种产品
 * → 抽象工厂：一个接口有多个 createXxx() 方法，创建一组配套产品
 */
public interface GUIFactory {
    Button createButton();
    TextField createTextField();
    Checkbox createCheckbox();
}
```

### 4.5 第四步：实现具体工厂

```java
/**
 * Windows 工厂：生产一整套 Windows 风格的 UI 组件
 * 
 * 设计意图：
 * 所有 Windows 组件的创建集中在这一个工厂里，
 * 保证产出的 Button、TextField、Checkbox 风格一致
 */
public class WindowsFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new WindowsButton();     // 只创建 Windows 按钮
    }
    @Override
    public TextField createTextField() {
        return new WindowsTextField();   // 只创建 Windows 文本框
    }
    @Override
    public Checkbox createCheckbox() {
        return new WindowsCheckbox();    // 只创建 Windows 复选框
    }
}

/**
 * Mac 工厂：生产一整套 Mac 风格的 UI 组件
 */
public class MacFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new MacButton();
    }
    @Override
    public TextField createTextField() {
        return new MacTextField();
    }
    @Override
    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}
```

### 4.6 第五步：客户端使用

```java
/**
 * 客户端：Application
 * 
 * 注意观察：整个类中没有出现任何 Windows 或 Mac 的具体类名！
 * 它只依赖 GUIFactory、Button、TextField、Checkbox 这些抽象接口。
 * 
 * 这意味着：
 * - 传入 WindowsFactory → 所有 UI 都是 Windows 风格
 * - 传入 MacFactory     → 所有 UI 都是 Mac 风格
 * - 不可能出现混搭！（因为一个工厂只生产一套风格的产品）
 */
public class Application {

    private Button button;
    private TextField textField;
    private Checkbox checkbox;

    /**
     * 通过构造函数注入抽象工厂
     * 具体用哪个工厂，由外部（调用方/配置文件/Spring容器）决定
     */
    public Application(GUIFactory factory) {
        // 用同一个工厂创建所有组件 → 保证风格统一！
        this.button = factory.createButton();
        this.textField = factory.createTextField();
        this.checkbox = factory.createCheckbox();
    }

    public void renderUI() {
        System.out.println("====== 渲染 UI 界面 ======");
        button.render();
        textField.render();
        checkbox.render();
        System.out.println("========================");
    }

    public void simulateClick() {
        button.onClick();
    }
}
```

```java
/**
 * 启动入口：根据操作系统选择工厂
 * 
 * "选择哪个工厂" 的逻辑只在这里出现一次！
 * 整个应用的其他地方都不需要关心平台差异
 */
public class Main {
    public static void main(String[] args) {
        // 根据系统属性自动选择工厂
        String osName = System.getProperty("os.name").toLowerCase();
        GUIFactory factory;

        if (osName.contains("windows")) {
            factory = new WindowsFactory();
        } else if (osName.contains("mac")) {
            factory = new MacFactory();
        } else {
            throw new UnsupportedOperationException("不支持的操作系统: " + osName);
        }

        // 把工厂注入应用 —— 应用内部不再关心平台
        Application app = new Application(factory);
        app.renderUI();
        app.simulateClick();
    }
}
```

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"一切都是权衡 (It depends)。"*

### 优点 (Pros)

- ✅ **保证产品配套**：同一个工厂创建出来的产品一定是风格统一的，不会出现混搭
- ✅ **符合开闭原则**：新增产品族（如新平台）只需新增工厂和产品类
- ✅ **解耦创建与使用**：客户端只依赖抽象接口
- ✅ **集中管理**：相关产品的创建逻辑集中在一个工厂中

### 代价 (Cons)

- ❌ **类爆炸（更严重）**：每新增一个产品族 → 需要新增 1 个工厂类 + N 个产品类
- ❌ **新增产品等级困难**：如果要新增一种产品（比如 `Scrollbar`），需要修改**所有工厂**接口和实现类 → **违反开闭原则！**
- ❌ **复杂度高**：比工厂方法复杂得多，需要设计更多的接口和类

### 🚨 最大的坑：两个方向的扩展性不对称

```
                      新增产品族（横向扩展）     新增产品等级（纵向扩展）
                      如：新增 Linux 平台       如：新增 Scrollbar 组件
  
  抽象工厂模式          ✅ 非常容易               ❌ 非常痛苦
                       只需新增工厂和产品类        需要改抽象工厂接口
                                                 + 改所有工厂实现类
```

```java
// ❌ 痛苦的扩展：新增一种产品 Scrollbar
// 所有工厂接口和实现都要改！

public interface GUIFactory {
    Button createButton();
    TextField createTextField();
    Checkbox createCheckbox();
    Scrollbar createScrollbar();    // ← 💥 新增！接口必须改
}

public class WindowsFactory implements GUIFactory {
    // ... 之前的方法 ...
    @Override
    public Scrollbar createScrollbar() {      // ← 💥 必须实现
        return new WindowsScrollbar();
    }
}

public class MacFactory implements GUIFactory {
    // ... 之前的方法 ...
    @Override
    public Scrollbar createScrollbar() {      // ← 💥 必须实现
        return new MacScrollbar();
    }
}

// 如果有 10 个工厂实现类，10 个都要改……
```

### 什么时候该用抽象工厂？

| ✅ 适合使用 | ❌ 不适合使用 |
|------------|-------------|
| 产品有多个系列/族 | 只有一种产品 → 用工厂方法 |
| 产品之间需要配套使用 | 产品之间相互独立、无关联 |
| 产品族可能扩展（新平台、新主题） | 产品等级经常扩展（频繁加新组件） |
| 系统需要保证一致性（同一风格/同一平台） | 简单场景，就2个类 → 别过度设计 |

### 三种工厂模式对比

| 维度 | 简单工厂 | 工厂方法 | 抽象工厂 |
|------|---------|---------|---------|
| **工厂数量** | 1 个 | N 个（一种产品一个） | N 个（一个产品族一个） |
| **产品数量** | 一种产品的多个类型 | 一种产品的多个类型 | 多种产品的多个类型 |
| **创建什么** | 一个产品 | 一个产品 | **一组配套产品** |
| **新增产品类型** | 改 switch ❌ | 加工厂类 ✅ | 加工厂类 + 一组产品 ✅ |
| **新增产品种类** | - | - | 改接口 + 改所有工厂 ❌ |
| **复杂度** | 低 | 中 | **高** |
| **开闭原则** | 违反 | 遵循 | 部分遵循（产品族 ✅ / 产品等级 ❌） |

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"计算机科学是互通的。"*

### 实际应用场景

#### 场景一：数据库访问层（最经典）

不同数据库的 Connection、Statement、ResultSet 都不同，但必须配套：

```java
// JDBC 的设计就是抽象工厂的精髓
// DriverManager 根据 URL 选择具体的 Driver（工厂）
// 每个 Driver 创建一整套配套的数据库访问对象

// MySQL 的工厂
public class MySQLDriver implements Driver {
    public Connection connect(String url) {
        return new MySQLConnection();      // MySQL 连接
        // MySQLConnection.createStatement() → MySQLStatement
        // MySQLStatement.executeQuery()     → MySQLResultSet
        // 整套都是 MySQL 的！
    }
}

// PostgreSQL 的工厂
public class PostgreSQLDriver implements Driver {
    public Connection connect(String url) {
        return new PostgreSQLConnection();  // PostgreSQL 连接
        // 整套都是 PostgreSQL 的！
    }
}

// 你的代码只用接口：Connection、Statement、ResultSet
// 完全不关心底层是 MySQL 还是 PostgreSQL
```

#### 场景二：Spring 的 BeanFactory 体系

```java
// Spring 中的 BeanFactory 就是一种抽象工厂
// 不同的 ApplicationContext 实现就是不同的具体工厂

// XML 配置的工厂
ApplicationContext ctx1 = new ClassPathXmlApplicationContext("beans.xml");

// 注解配置的工厂
ApplicationContext ctx2 = new AnnotationConfigApplicationContext(AppConfig.class);

// 不管用哪个工厂，产出的 Bean 都是配套的
// （同一个配置体系下的 Bean 相互可以注入和依赖）
```

#### 场景三：主题切换（暗色/亮色模式）

```java
// 手机 App 的暗色/亮色模式就是典型的抽象工厂场景

public interface ThemeFactory {
    Color getBackgroundColor();
    Color getTextColor();
    Color getAccentColor();
    Icon getIcon(String name);
}

public class DarkThemeFactory implements ThemeFactory {
    public Color getBackgroundColor() { return Color.BLACK; }
    public Color getTextColor() { return Color.WHITE; }
    public Color getAccentColor() { return Color.BLUE; }
    // 所有颜色和图标都是暗色系搭配
}

public class LightThemeFactory implements ThemeFactory {
    public Color getBackgroundColor() { return Color.WHITE; }
    public Color getTextColor() { return Color.BLACK; }
    public Color getAccentColor() { return Color.ORANGE; }
    // 所有颜色和图标都是亮色系搭配
}
```

### 与其他模式的关系

```
抽象工厂
    │
    ├── 与工厂方法：抽象工厂中的每个 createXxx() 方法
    │    本质上就是一个工厂方法。
    │    → 抽象工厂 = 多个工厂方法的组合
    │
    ├── 与单例模式：具体工厂通常是单例的
    │    （全局只需要一个 WindowsFactory 实例）
    │
    ├── 与策略模式：切换工厂 ≈ 切换"创建策略"
    │    （换一个工厂实现，整套产品都变了）
    │
    └── 与建造者模式：建造者关注"复杂对象的构建步骤"
         抽象工厂关注"一组配套对象的创建"
         → 两者可以结合：工厂创建的产品可以用建造者来构建
```

### 底层视角：运行时多态

抽象工厂的核心依赖 Java 的**运行时多态**（动态绑定）：

```java
GUIFactory factory = new WindowsFactory();  // 编译时类型：GUIFactory
                                            // 运行时类型：WindowsFactory
Button button = factory.createButton();     // 编译时不知道是 WindowsButton
                                            // 运行时 JVM 通过虚方法表找到正确实现
```

在 JVM 层面：
1. `factory.createButton()` → JVM 查虚方法表（vtable）
2. 发现运行时类型是 `WindowsFactory` → 调用 `WindowsFactory.createButton()`
3. 返回 `WindowsButton`，但调用方只看到 `Button` 接口

**这就是"面向接口编程"的底层原理——所有的灵活性，都建立在 JVM 的动态分派机制之上。**

### 🤔 思考题

> **如果要在抽象工厂中新增一种产品（如 Scrollbar），有没有办法不修改已有的工厂接口？**

**提示方向：**

1. **Java 8 的 default 方法**：可以在接口中提供默认实现，旧工厂不用改
```java
public interface GUIFactory {
    Button createButton();
    TextField createTextField();
    Checkbox createCheckbox();
    
    // Java 8 default 方法：提供默认实现，旧工厂不用改
    default Scrollbar createScrollbar() {
        return new DefaultScrollbar();  // 给一个默认实现
    }
}
```

2. **用 Map 代替固定方法**：让工厂变得更灵活（但失去类型安全）
```java
public interface FlexibleFactory {
    <T> T create(Class<T> type);  // 泛型工厂方法
}
```

3. **拆分接口**：把新增的产品放到新接口中（接口隔离原则）
```java
public interface ScrollbarFactory {
    Scrollbar createScrollbar();
}
// 新工厂同时实现 GUIFactory 和 ScrollbarFactory
public class WindowsFactoryV2 implements GUIFactory, ScrollbarFactory { ... }
```

每种方案都有利弊，没有完美解。这再次印证了——**软件工程的本质就是权衡**。
