# 工厂方法模式 (Factory Method Pattern)

> 创建型模式 | 难度：⭐⭐ | 使用频率：⭐⭐⭐⭐⭐

---

## 1. 🎯 问题与背景 (The Problem & Context)

> **工程视角:** *"任何技术都是为了解决特定痛点而诞生的。"*

### 痛点分析

假设你在开发一个**物流系统**，最初只支持**卡车运输**：

```java
public class LogisticsApp {
    public void planDelivery() {
        // 直接 new 一个卡车
        Truck truck = new Truck();
        truck.deliver();
    }
}
```

一切正常，直到：

- 📦 老板说：「我们要支持**海运**了，加个船」
- 📦 过了一个月：「再加个**空运**，用飞机」
- 📦 又过了一个月：「某些地区用**快递小哥骑电动车**」

你的代码变成了这样：

```java
// 💥 灾难代码：每加一种运输方式，这里就要改一次
public class LogisticsApp {
    public void planDelivery(String type) {
        if ("truck".equals(type)) {
            Truck truck = new Truck();
            truck.deliver();
        } else if ("ship".equals(type)) {
            Ship ship = new Ship();
            ship.deliver();
        } else if ("airplane".equals(type)) {
            Airplane airplane = new Airplane();
            airplane.deliver();
        } else if ("bike".equals(type)) {
            Bike bike = new Bike();
            bike.deliver();
        }
        // 每次新增运输方式，都要修改这个类 → 违反开闭原则！
    }
}
```

**问题总结：**

| 痛点 | 描述 |
|------|------|
| **违反开闭原则** | 每新增一种产品，就要修改已有的业务代码 |
| **违反单一职责** | 业务逻辑和对象创建逻辑混在一起 |
| **高耦合** | 调用方直接依赖具体类（`new Truck()`），而不是抽象 |
| **难以扩展** | 一堆 `if-else`，代码膨胀，难以维护 |

### 核心定义

**工厂方法模式定义一个创建对象的接口，但将具体创建哪个类的决定延迟到子类中。让子类决定实例化哪个类。**

核心思想：**把 `new` 操作从调用方手中夺走，交给专门的"工厂"来做。**

## 2. 🧩 抽象与心智模型 (The Mental Model)

> **设计视角:** *"不仅要看代码，更要看懂设计。"*

### 生活类比

**披萨店加盟 🍕**

- **没有工厂模式：** 你自己在家做披萨 → 你需要知道每种披萨怎么做（面团配方、烤箱温度、配料比例……）
- **简单工厂：** 你去一家披萨店 → 你只需要说"要一个夏威夷披萨"，店里帮你做
- **工厂方法：** 你去**加盟品牌** → 纽约分店做的是纽约风味披萨，芝加哥分店做的是芝加哥深盘披萨。每个分店（子类工厂）自己决定怎么做，但流程标准（父类模板）是一样的

### 抽象设计（黑盒模型）

```
【简单工厂】
                          ┌─────────────────┐
  客户端 ──"要XX产品"──→  │   工厂 (Factory)  │ ──→ 产品A / 产品B / 产品C
                          │  (一个if-else)    │
                          └─────────────────┘

【工厂方法】
                          ┌─────────────────┐
                          │  抽象工厂 (接口)   │  ← 定义"创建产品"的规范
                          │  createProduct() │
                          └───────┬─────────┘
                       ┌──────────┼──────────┐
                       ↓          ↓          ↓
                  ┌─────────┐ ┌─────────┐ ┌─────────┐
                  │ 工厂A   │ │ 工厂B   │ │ 工厂C   │  ← 每个工厂决定创建什么
                  │→产品A   │ │→产品B   │ │→产品C   │
                  └─────────┘ └─────────┘ └─────────┘
```

**本质区别：**
- **简单工厂** = 一个工厂类，用 `if-else` 决定创建什么（违反开闭原则）
- **工厂方法** = 多个工厂类，每个类只负责创建一种产品（符合开闭原则）

## 3. 🏗️ 系统架构与设计 (System & Architecture)

> **架构视角:** *"站得高，看得远。"*

### 在系统中的位置

工厂方法模式通常位于**业务层**和**数据层**之间，负责对象的创建和组装：

```
┌─────────────────────────────────────┐
│      表现层 (Controller)              │
│      "我需要一个支付处理器"             │
├─────────────────────────────────────┤
│      业务层 (Service)                 │
│      PaymentService                  │
│      → 调用 factory.createProcessor() │
├─────────────────────────────────────┤
│      工厂层 (Factory) ← 工厂在这里     │
│      ┌──────────┐ ┌──────────┐      │
│      │支付宝工厂  │ │微信工厂   │      │
│      │→支付宝处理 │ │→微信处理  │      │
│      └──────────┘ └──────────┘      │
├─────────────────────────────────────┤
│      数据层 (Repository)              │
└─────────────────────────────────────┘
```

### 角色说明（四个核心角色）

```
┌──────────────────────────────────────────────────────────┐
│                    工厂方法模式的四个角色                    │
│                                                          │
│  ┌──────────────────┐      ┌──────────────────┐         │
│  │  抽象产品          │      │  抽象工厂          │         │
│  │  (Product)        │      │  (Creator)        │         │
│  │  定义产品的接口     │      │  定义创建方法的接口  │         │
│  └────────┬─────────┘      └────────┬─────────┘         │
│           │ 实现                     │ 实现               │
│  ┌────────┴─────────┐      ┌────────┴─────────┐         │
│  │  具体产品          │      │  具体工厂          │         │
│  │  (ConcreteProduct)│      │  (ConcreteCreator)│         │
│  │  产品的真正实现     │      │  创建具体的产品     │         │
│  └──────────────────┘      └──────────────────┘         │
└──────────────────────────────────────────────────────────┘
```

### 涉及的设计原则

- **开闭原则 (OCP)**：新增产品时，只需新增工厂子类，不修改已有代码 ✅
- **依赖倒置原则 (DIP)**：调用方依赖抽象（接口），不依赖具体产品类 ✅
- **单一职责原则 (SRP)**：对象创建和使用分离 ✅

## 4. 💻 核心实现 (Implementation)

> **落地视角:** *"优雅的代码是清晰思维的体现。"*

### 4.1 简单工厂（Simple Factory）—— 入门理解用

> ⚠️ 严格来说，简单工厂不是 GoF 23 种设计模式之一，而是一种编程习惯。
> 但它是理解工厂方法的基础，必须先掌握。

**场景：** 咖啡店，根据客户需求创建不同咖啡。

```java
/**
 * 第一步：定义抽象产品（所有咖啡的共性）
 */
public interface Coffee {
    String getName();    // 咖啡名称
    double getPrice();   // 价格
    void prepare();      // 制作过程
}

/**
 * 第二步：定义具体产品
 */
public class Americano implements Coffee {
    @Override
    public String getName() { return "美式咖啡"; }
    @Override
    public double getPrice() { return 25.0; }
    @Override
    public void prepare() {
        System.out.println("研磨咖啡豆 → 萃取浓缩 → 加热水");
    }
}

public class Latte implements Coffee {
    @Override
    public String getName() { return "拿铁"; }
    @Override
    public double getPrice() { return 32.0; }
    @Override
    public void prepare() {
        System.out.println("研磨咖啡豆 → 萃取浓缩 → 打奶泡 → 拉花");
    }
}

public class Cappuccino implements Coffee {
    @Override
    public String getName() { return "卡布奇诺"; }
    @Override
    public double getPrice() { return 30.0; }
    @Override
    public void prepare() {
        System.out.println("研磨咖啡豆 → 萃取浓缩 → 厚奶泡 → 撒可可粉");
    }
}
```

```java
/**
 * 第三步：简单工厂（唯一的一个工厂类）
 * 
 * 设计意图：把 new 操作集中到一个地方
 * 调用方不再关心 "怎么创建"，只需要告诉工厂 "我要什么"
 */
public class SimpleCoffeeFactory {

    // 通过 type 参数决定创建哪种咖啡
    public static Coffee createCoffee(String type) {
        switch (type) {
            case "americano":   return new Americano();
            case "latte":       return new Latte();
            case "cappuccino":  return new Cappuccino();
            default:
                throw new IllegalArgumentException("未知咖啡类型: " + type);
        }
    }
}

// 使用方式
Coffee coffee = SimpleCoffeeFactory.createCoffee("latte");
coffee.prepare();  // 研磨咖啡豆 → 萃取浓缩 → 打奶泡 → 拉花
```

**简单工厂的问题：**

```java
// 如果要新增 "摩卡咖啡"……
// 你必须打开 SimpleCoffeeFactory，在 switch 里加一个 case
// → 修改了已有代码 → 违反开闭原则！

case "mocha": return new Mocha();  // 💥 必须改工厂代码
```

### 4.2 工厂方法模式（Factory Method）—— 正式版 ⭐

> **核心思想：每种产品都有自己专属的工厂，新增产品只需新增工厂，不改已有代码**

```java
/**
 * 第一步：抽象产品（不变）
 */
public interface Coffee {
    String getName();
    double getPrice();
    void prepare();
}

/**
 * 第二步：具体产品（不变）
 */
public class Americano implements Coffee {
    @Override
    public String getName() { return "美式咖啡"; }
    @Override
    public double getPrice() { return 25.0; }
    @Override
    public void prepare() {
        System.out.println("☕ 研磨咖啡豆 → 萃取浓缩 → 加热水");
    }
}

public class Latte implements Coffee {
    @Override
    public String getName() { return "拿铁"; }
    @Override
    public double getPrice() { return 32.0; }
    @Override
    public void prepare() {
        System.out.println("☕ 研磨咖啡豆 → 萃取浓缩 → 打奶泡 → 拉花");
    }
}
```

```java
/**
 * 第三步：抽象工厂（定义"创建产品"的规范）
 * 
 * 为什么用接口而不是具体类？
 * → 依赖倒置原则：调用方依赖接口，不依赖具体工厂
 */
public interface CoffeeFactory {
    Coffee createCoffee();  // 工厂方法：子类决定创建什么
}
```

```java
/**
 * 第四步：具体工厂（每个工厂负责创建一种产品）
 * 
 * 设计意图：
 * 把 "创建哪种咖啡" 的决定从 if-else 中解放出来，
 * 分散到各个专属的工厂类中
 */
public class AmericanoFactory implements CoffeeFactory {
    @Override
    public Coffee createCoffee() {
        // 美式工厂只知道怎么做美式
        return new Americano();
    }
}

public class LatteFactory implements CoffeeFactory {
    @Override
    public Coffee createCoffee() {
        // 拿铁工厂只知道怎么做拿铁
        return new Latte();
    }
}
```

```java
/**
 * 第五步：客户端使用
 * 
 * 注意：客户端只依赖 CoffeeFactory 接口和 Coffee 接口
 * 完全不知道 Americano、Latte 这些具体类的存在
 */
public class CoffeeShop {
    
    private CoffeeFactory factory;  // 依赖抽象工厂接口

    // 通过构造函数注入工厂（依赖注入）
    public CoffeeShop(CoffeeFactory factory) {
        this.factory = factory;
    }

    public Coffee orderCoffee() {
        // 调用工厂方法创建咖啡 —— 不关心具体是什么咖啡
        Coffee coffee = factory.createCoffee();
        
        System.out.println("顾客点了: " + coffee.getName());
        System.out.println("价格: ¥" + coffee.getPrice());
        coffee.prepare();
        System.out.println("制作完成！请取餐 ✅");

        return coffee;
    }
}

// 使用
CoffeeShop shop1 = new CoffeeShop(new AmericanoFactory());
shop1.orderCoffee();
// 输出：
// 顾客点了: 美式咖啡
// 价格: ¥25.0
// ☕ 研磨咖啡豆 → 萃取浓缩 → 加热水
// 制作完成！请取餐 ✅

CoffeeShop shop2 = new CoffeeShop(new LatteFactory());
shop2.orderCoffee();
// 输出：
// 顾客点了: 拿铁
// 价格: ¥32.0
// ☕ 研磨咖啡豆 → 萃取浓缩 → 打奶泡 → 拉花
// 制作完成！请取餐 ✅
```

### 🔑 关键时刻：新增产品

现在老板说：「加一个**摩卡咖啡**！」

```java
// 第一步：新增产品类（新代码）
public class Mocha implements Coffee {
    @Override
    public String getName() { return "摩卡"; }
    @Override
    public double getPrice() { return 35.0; }
    @Override
    public void prepare() {
        System.out.println("☕ 研磨咖啡豆 → 萃取浓缩 → 加巧克力酱 → 打奶泡");
    }
}

// 第二步：新增工厂类（新代码）
public class MochaFactory implements CoffeeFactory {
    @Override
    public Coffee createCoffee() {
        return new Mocha();
    }
}

// 第三步：使用（已有代码一个字都不用改！）
CoffeeShop shop = new CoffeeShop(new MochaFactory());
shop.orderCoffee();  // 完美运行 ✅
```

```
对比：
  简单工厂：新增产品 → 修改工厂类的 switch → 改旧代码 ❌
  工厂方法：新增产品 → 新增一个工厂类   → 只写新代码 ✅
```

### 4.3 简单工厂 vs 工厂方法 全面对比

| 维度 | 简单工厂 | 工厂方法 |
|------|---------|---------|
| **工厂数量** | 1 个（包含所有 if-else） | N 个（每种产品一个） |
| **新增产品** | 修改工厂类 ❌ | 新增工厂类 ✅ |
| **开闭原则** | 违反 | 遵循 |
| **代码量** | 少 | 多（每个产品多一个工厂类） |
| **复杂度** | 简单 | 稍复杂 |
| **适用场景** | 产品种类少且不常变化 | 产品种类多或频繁扩展 |

**选择建议：**
- 产品就2-3种，未来不太变 → 简单工厂就够了，别过度设计
- 产品经常扩展、需要灵活替换 → 工厂方法

## 5. ⚖️ 工程权衡与局限 (Trade-offs & Constraints)

> **工程视角:** *"一切都是权衡 (It depends)。"*

### 优点 (Pros)
- ✅ **符合开闭原则**：新增产品不修改已有代码
- ✅ **解耦创建与使用**：调用方不依赖具体产品类
- ✅ **易于测试**：可以注入 Mock 工厂
- ✅ **符合单一职责**：每个工厂只负责创建一种产品

### 代价 (Cons)
- ❌ **类爆炸**：每新增一种产品，就多一个产品类 + 一个工厂类（2个类）
- ❌ **增加复杂度**：简单场景下使用工厂方法属于**过度设计**
- ❌ **客户端需要知道具体工厂**：虽然不依赖具体产品，但得知道用哪个工厂

### 何时不该使用工厂方法？

```java
// ❌ 过度设计的例子：产品只有一种，还搞工厂方法
public interface StringUtilFactory {
    StringUtil create();
}
public class DefaultStringUtilFactory implements StringUtilFactory {
    @Override
    public StringUtil create() { return new StringUtil(); }
}
// 大可不必！直接 new StringUtil() 就行了
```

**判断标准：** 如果你的 `new` 操作只出现在一个地方，而且不太会变，就别用工厂模式。

## 6. 🌐 深度联想与扩展 (Deep Dive & Connections)

> **通识视角:** *"计算机科学是互通的。"*

### 实际应用场景

#### 场景一：Spring 中的 BeanFactory

Spring 框架的核心就是一个巨大的工厂：

```java
// Spring 的 BeanFactory 就是工厂方法的体现
// ApplicationContext 是 BeanFactory 的子接口
ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");

// getBean() 就是工厂方法 —— 你说要什么 Bean，它帮你创建
UserService userService = context.getBean(UserService.class);
```

#### 场景二：JDBC 的 Connection 创建

```java
// DriverManager.getConnection() 是简单工厂
// 不同的数据库驱动（MySQL、Oracle、PostgreSQL）会返回不同的 Connection 实现
Connection conn = DriverManager.getConnection("jdbc:mysql://localhost:3306/db");
// 你不需要知道返回的是 com.mysql.cj.jdbc.ConnectionImpl
// 你只需要用 Connection 接口操作
```

#### 场景三：日志框架 SLF4J

```java
// LoggerFactory.getLogger() 是工厂方法
// 底层可能是 Logback、Log4j2 或 JDK Logging，你完全不需要知道
Logger logger = LoggerFactory.getLogger(MyClass.class);
logger.info("Hello");  // 用哪个日志框架取决于类路径中的绑定
```

#### 场景四：消除 if-else 的实战技巧

在实际项目中，工厂方法常与 **Map + 策略模式** 结合使用：

```java
/**
 * 实战中常用的注册表工厂
 * 既有简单工厂的便利，又避免了 if-else
 */
public class CoffeeFactoryRegistry {
    
    // 用 Map 代替 if-else
    private static final Map<String, CoffeeFactory> REGISTRY = new HashMap<>();

    // 静态注册（也可以用 Spring 的 @Autowired List<CoffeeFactory> 自动注册）
    static {
        REGISTRY.put("americano", new AmericanoFactory());
        REGISTRY.put("latte", new LatteFactory());
        REGISTRY.put("mocha", new MochaFactory());
    }

    // 新增产品？只需要在 REGISTRY 中注册，不用修改任何逻辑代码
    public static Coffee createCoffee(String type) {
        CoffeeFactory factory = REGISTRY.get(type);
        if (factory == null) {
            throw new IllegalArgumentException("未知咖啡类型: " + type);
        }
        return factory.createCoffee();
    }
}
```

### 工厂方法 vs 抽象工厂（预告）

| | 工厂方法 | 抽象工厂 |
|---|---|---|
| **创建什么** | 一种产品 | 一族相关产品 |
| **例子** | 咖啡工厂创建咖啡 | 家具工厂创建桌子+椅子+沙发（一套） |
| **扩展方向** | 新增产品种类 | 新增产品族 |

这将在下一节**抽象工厂模式**中详细讲解。

### 与其他模式的关系

```
工厂方法
    │
    ├── 与模板方法：工厂方法常常是模板方法的一个步骤
    │    （父类定义业务流程，其中某一步"创建产品"由子类实现）
    │
    ├── 与策略模式：工厂可以看作是"创建策略"
    │    （不同的工厂 = 不同的创建策略）
    │
    ├── 与抽象工厂：工厂方法是抽象工厂的基础
    │    （抽象工厂 = 多个工厂方法的组合）
    │
    └── 与单例模式：工厂本身往往是单例的
         （全局只需要一个工厂实例）
```

### 🤔 思考题与解答

#### 思考题：工厂方法能彻底消除 `new` 吗？

**答案：不能，也不应该。**

工厂方法只是把 `new` 从"散落在各处"集中到了"工厂类中"。`new` 操作只是被移动了，并没有消失。

```java
// new 还是存在的，只是在工厂内部
public class LatteFactory implements CoffeeFactory {
    @Override
    public Coffee createCoffee() {
        return new Latte();  // ← new 还在这里
    }
}
```

那这样做有什么意义？意义在于**隔离变化**：
- 调用方（`CoffeeShop`）完全不知道 `Latte` 的存在
- 如果 `Latte` 的构造方式变了（比如需要传参数），只需要改 `LatteFactory`
- 如果要换成 `SuperLatte`，只需要改工厂，调用方一个字不用动

**`new` 不是罪恶，关键是 `new` 不应该出现在"不该关心创建细节"的地方。**

在 Spring 中，连工厂里的 `new` 都被消除了 —— 因为 Spring 的 IoC 容器用**反射**来创建对象：

```java
// Spring 通过反射创建对象，你连 new 都不用写
@Component
public class LatteFactory implements CoffeeFactory {
    @Override
    public Coffee createCoffee() {
        // Spring 甚至可以帮你注入依赖
        return applicationContext.getBean(Latte.class);
    }
}
```
