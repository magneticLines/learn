重构 · 常用手法

搬移特性（在对象间移动）

把函数、字段、语句放到它们"真正属于"的地方——这是重构中使用频率最高、收益最稳定的一类手法。

1. 为什么要搬移特性

一个设计良好的类有明确的职责边界：它只关心自己应该关心的数据，只做自己应该做的事。 现实中代码却常常发生"特性漂移"——某个方法其实在操作另一个对象的字段；某个字段只被一个外部类使用； 某几行语句放在调用方比放在方法体里更清晰。

搬移特性（Moving Features）包含四种子手法：

搬移特性
├── 搬移函数（Move Function）
├── 搬移字段（Move Field）
├── 搬移语句到函数（Move Statements into Function）
└── 搬移语句到调用者（Move Statements to Callers）

判断是否需要搬移的核心信号：一个方法反复引用另一个对象的数据，远多于引用自身数据。 这说明它"住错了地方"。

2. 搬移函数（Move Function）

场景：订单折扣计算散落在服务层

电商项目里，OrderService 里堆满了各种折扣规则判断，而这些判断全部依赖 Order 对象自身的字段。 这是最典型的"特性漂移"——折扣逻辑是订单自身的业务规则，不是服务层协调多个对象的编排逻辑。

反例 — 问题代码

OrderService.java（重构前）
public class OrderService {

    // 问题：这个方法只用到了 order 自己的字段，根本没用到 Service 的任何状态
    public BigDecimal calculateDiscount(Order order) {
        BigDecimal discount = BigDecimal.ZERO;

        // 会员等级折扣
        if (order.getMemberLevel() >= 3) {
            discount = discount.add(order.getTotalAmount().multiply(new BigDecimal("0.05")));
        }

        // 满减折扣
        if (order.getTotalAmount().compareTo(new BigDecimal("200")) >= 0) {
            discount = discount.add(new BigDecimal("20"));
        }

        return discount;
    }

    public void submitOrder(Order order) {
        // 调用方：Service 调自己算折扣，再减掉
        BigDecimal discount = calculateDiscount(order);
        order.setPayableAmount(order.getTotalAmount().subtract(discount));
        // ... 持久化等操作
    }
}

问题所在：

  • calculateDiscount 只依赖 order 的字段，与 OrderService 的状态毫无关系；
  • 写单测时必须构造整个 OrderService 才能测折扣逻辑，测试成本高；
  • 其他地方（如退款模块）复用这段逻辑时，必须注入 OrderService，造成不必要的依赖；
  • 折扣规则变更后，需要改一个"服务"而不是"订单"，与直觉相悖。

正例 — 搬移后

Order.java（重构后）
public class Order {

    private int        memberLevel;
    private BigDecimal totalAmount;

    // 折扣规则是订单自身的业务规则，搬到这里后：
    // 1. 单测直接 new Order() 就能测，不依赖 Spring 容器
    // 2. 任何地方用到 Order 就能算折扣，无需注入 Service
    public BigDecimal calculateDiscount() {
        BigDecimal discount = BigDecimal.ZERO;

        if (memberLevel >= 3) {
            discount = discount.add(totalAmount.multiply(new BigDecimal("0.05")));
        }
        if (totalAmount.compareTo(new BigDecimal("200")) >= 0) {
            discount = discount.add(new BigDecimal("20"));
        }
        return discount;
    }

    // getter / setter 省略
}
OrderService.java（重构后）
public class OrderService {

    public void submitOrder(Order order) {
        // Service 只做编排：调用 Order 自己计算折扣
        order.setPayableAmount(
            order.getTotalAmount().subtract(order.calculateDiscount())
        );
        // ... 持久化等操作
    }
}
判断准则：一个方法的入参如果只有一个对象，且方法体只访问这个对象的字段， 那它大概率应该是那个对象的方法——除非这个对象是 DTO / POJO，业务逻辑不应放进去（见下方陷阱）。

3. 搬移字段（Move Field）

场景：折扣率字段放错了宿主

系统迭代后，每个会员等级对应一个固定折扣率。折扣率最初被存在 Order 里（因为最早只支持一种折扣）， 后来引入了 MemberLevel 枚举，但折扣率字段没有随之迁移。

反例

Order.java（反例）
public class Order {
    private MemberLevel memberLevel;
    // 问题：discountRate 描述的是"等级"的属性，不是"订单"的属性
    // 每次创建订单都要手动传这个值，极易与 memberLevel 不一致
    private BigDecimal  discountRate;
}

public enum MemberLevel {
    NORMAL, SILVER, GOLD
}

问题：不同订单可能因为构造时传值不同，导致同一会员等级出现不同折扣率——数据一致性难以保证。

正例 — 字段归位

MemberLevel.java（重构后）
public enum MemberLevel {
    NORMAL (BigDecimal.ZERO),
    SILVER (new BigDecimal("0.03")),
    GOLD   (new BigDecimal("0.05"));

    // discountRate 是等级的固有属性，枚举是它的天然宿主
    private final BigDecimal discountRate;

    MemberLevel(BigDecimal rate) {
        this.discountRate = rate;
    }

    public BigDecimal getDiscountRate() {
        return discountRate;
    }
}
Order.java（重构后）
public class Order {
    private MemberLevel memberLevel;
    private BigDecimal  totalAmount;

    public BigDecimal calculateDiscount() {
        // 折扣率从等级本身取，永远不会不一致
        BigDecimal rateDiscount = totalAmount.multiply(memberLevel.getDiscountRate());
        BigDecimal fullReduction = totalAmount.compareTo(new BigDecimal("200")) >= 0
                ? new BigDecimal("20") : BigDecimal.ZERO;
        return rateDiscount.add(fullReduction);
    }
}

4. 搬移语句：进函数 vs 出函数

这两个手法是互逆操作，用来调整代码的"归属粒度"。 判断依据是：这段语句属于函数的核心职责，还是只属于某个特定调用场景？

4.1 搬移语句到函数

当多个调用方都在调用某个函数前/后写了相同的前置/后置代码，说明这段代码应该被纳入函数本身。

反例 — 日志打印重复散落在调用方
// 调用方 A
log.info("开始渲染报表, reportId={}", reportId);
renderReport(reportId);

// 调用方 B
log.info("开始渲染报表, reportId={}", reportId);
renderReport(reportId);
正例 — 日志纳入函数
private void renderReport(Long reportId) {
    // 日志是渲染动作不可分割的一部分，搬进来后调用方保持整洁
    log.info("开始渲染报表, reportId={}", reportId);
    // ... 渲染逻辑
}

4.2 搬移语句到调用者

当同一个函数在不同场景下行为微妙不同，部分逻辑只对某些调用方有意义时，把差异化语句移回调用方。

反例 — 函数内部用 flag 区分场景
public void exportData(List<Record> records, boolean needHeader) {
    // 问题：needHeader 是调用方的关注点，不是导出逻辑的通用关注点
    // 函数越来越多这种 flag，职责逐渐混乱
    if (needHeader) {
        writeHeader();
    }
    writeRows(records);
}
正例 — 语句搬回调用者
public void exportData(List<Record> records) {
    // 函数只负责写数据行，职责单一
    writeRows(records);
}

// 调用方 A（需要 header）
writeHeader();
exportData(records);

// 调用方 B（不需要 header）
exportData(records);

5. 综合案例：告警模块重构

以下是生产系统中一个真实场景的简化版：告警触发逻辑最初全部写在 AlertService 中， 随着规则增多，代码开始崩溃。

重构前：服务类知道太多

AlertService.java（重构前）
public class AlertService {

    public void checkAndAlert(MetricSnapshot snapshot) {

        // ① 阈值判断 —— 完全依赖 snapshot 字段，跟 Service 无关
        if (snapshot.getCpuUsage() > 0.9 || snapshot.getMemUsage() > 0.85) {

            // ② 构造告警消息 —— 同样只依赖 snapshot
            String msg = "[告警] 主机:" + snapshot.getHostname()
                    + " CPU:" + snapshot.getCpuUsage()
                    + " MEM:" + snapshot.getMemUsage();

            sendAlert(msg);
        }
    }

    private void sendAlert(String msg) { /* 发送钉钉/邮件 */ }
}

重构后：职责各归其位

MetricSnapshot.java（重构后）
public class MetricSnapshot {
    private String hostname;
    private double cpuUsage;
    private double memUsage;

    // 搬移函数：判断是否触发告警，属于 snapshot 自身的业务规则
    public boolean shouldAlert() {
        return cpuUsage > 0.9 || memUsage > 0.85;
    }

    // 搬移函数：构建告警描述，同属 snapshot 的职责
    public String buildAlertMessage() {
        return String.format("[告警] 主机:%s CPU:%.0f%% MEM:%.0f%%",
                hostname, cpuUsage * 100, memUsage * 100);
    }

    // getter 省略
}
AlertService.java（重构后）
public class AlertService {

    // Service 只剩编排职责：判断 → 发送，职责一目了然
    public void checkAndAlert(MetricSnapshot snapshot) {
        if (snapshot.shouldAlert()) {
            sendAlert(snapshot.buildAlertMessage());
        }
    }

    private void sendAlert(String msg) { /* 发送钉钉/邮件 */ }
}
重构前后 外部行为完全不变，但可测试性大幅提升： 现在可以直接对 MetricSnapshot 写单元测试，覆盖各种阈值边界，不需要构造 AlertService。

6. 手法速查对照表

手法 触发信号 常见场景
搬移函数 方法大量使用另一个对象的字段/方法 Service 里的领域规则搬到实体/值对象
搬移字段 字段只被一个外部类使用，或字段语义归属另一个类 配置参数放错宿主、枚举属性外置
语句入函数 多处调用方有相同的前/后置代码 日志、参数校验、状态更新重复出现
语句出函数 函数内部靠 flag 区分不同调用场景 导出/渲染函数里的条件分支

7. 生产环境常见坑

坑 1：把业务逻辑搬进 DTO / DO
DO（数据库实体）和 DTO 通常是纯数据载体，框架（MyBatis / JPA）会反射操作它们。 搬进去的方法如果依赖 Spring Bean 或有副作用，会与框架的生命周期冲突， 还会导致序列化问题（Fastjson / Jackson 可能调用看起来像 getter 的方法）。
原则：只把"纯计算"搬进 DO/DTO，有 IO、事务的逻辑留在 Service。
坑 2：搬移后忘记删旧方法，导致双写
在分布式团队中，旧方法未删除而新方法已上线，其他人继续调旧方法。 建议：搬移的同时立刻用 @Deprecated 标记旧方法并写明迁移路径， 设定一个 Sprint 内完成删除。
坑 3：过度搬移造成实体膨胀
DDD 的充血模型虽然提倡逻辑归实体，但如果实体变成了"上帝对象"—— 承载了几十个方法——就走向了反面。搬移的依据始终是"职责归属"，而不是"数量均衡"。
坑 4：忽略循环依赖风险
将方法从类 A 搬到类 B 时，若 B 已经依赖 A，就会产生循环依赖。 搬移前先画一张简单的依赖图，确认方向单向。

8. 安全操作步骤（以搬移函数为例）

  1. 确认目标宿主：找到方法最频繁访问的那个对象，它就是目标类。
  2. 在目标类中新建方法：先复制，不要删原方法。
  3. 修改原方法委托调用：让旧方法直接 return target.newMethod()，保证编译通过且行为不变。
  4. 全量替换调用方：IDE 全局搜索所有调用处，逐一改为调用新方法。
  5. 跑测试：确认没有回归。
  6. 删除旧方法：没有调用方后，安全删除。
IntelliJ IDEA 的 Refactor → Move Instance Method 可以自动完成上述步骤 3-5， 建议优先使用 IDE 重构工具以避免手工操作失误。
