重构 · 认知:何时重构

何时重构 · 重构与测试的关系

重构不是某个项目阶段的"大扫除"，而是开发节奏中随时触发的小动作。本文讲清楚：什么信号意味着该重构了，以及为什么没有测试的重构只是在赌运气。

一、先对齐一个认知：重构不等于重写

很多团队说"我们要重构这个模块"，实际上在做的是重写（Rewrite）——把整个模块删掉从头来过。这是两件截然不同的事。

维度重构（Refactoring）重写（Rewrite）
外部行为保持不变可能改变
节奏小步频繁，随时可停大块时间，中途难停
风险低，每步可验证高，集成时才暴露问题
测试依赖强依赖（必须有测试护网）需要，但重写后补测试也行

Martin Fowler 的定义极为精准：重构是在不改变软件可观察行为的前提下，改善其内部结构的过程。每一个小步骤都必须让测试继续绿灯。

二、何时重构：三次法则与四个触发点

三次法则（Rule of Three）

Don Roberts 提出的经验法则，Fowler 在《重构》中引用并推广：

  • 第一次：直接做，无需担心重复
  • 第二次：做类似的事，留意重复，但先做完
  • 第三次：再遇到类似的事，立刻重构

这个法则避免了过早抽象（YAGNI）和放任腐烂两个极端。两次才意味着"凑巧"，三次才说明这是稳定的模式。

触发点一：添加功能时（预备性重构）

你要往一个方法里加新逻辑，但它现在写得很乱——先花 10 分钟整理结构，再加新代码。这叫预备性重构（Preparatory Refactoring）。类比：在刷墙之前先打磨墙面，不然漆根本刷不均匀。

工程直觉：如果不先重构就直接加功能，新代码也会和旧代码一样乱，而且你刚才的"直觉"告诉你应该重构，说明当前结构已经在阻碍你了。

触发点二：修 Bug 时（理解性重构）

读懂一段代码之后，把你理解到的东西通过重命名、提炼函数等方式表达出来，让下一个读代码的人（很可能是三个月后的你）更快看懂。这叫理解性重构（Comprehension Refactoring）。

触发点三：Code Review 时

Review 是发现坏味道最自然的时机。看别人的代码，换个视角，更容易看出结构问题。一旦发现，现场讨论并立即处理，而不是开 ticket 留着"以后再说"——以后往往等于永远不做。

触发点四：专项重构冲刺（计划性重构）

前三种是机会主义式的，嵌入日常开发节奏。但当技术债已经大到阻碍迭代速度时，需要和业务方协商专门的重构时间窗口。这种情况下，重构必须有量化的收益指标（如单测覆盖率从 20% 提升到 60%，构建速度从 8 分钟降到 3 分钟），否则无法获得业务侧的支持。

三、不该重构的时机

  • 代码本身不需要修改时：如果这段代码永远不需要动，它再丑也无所谓，重构它只是在浪费时间
  • 临近发版窗口时：距离发版还有 2 小时，不要手痒去动"顺手能改的地方"，这是生产事故的高发场景
  • 完全没有测试覆盖且改动范围很大时：先补测试，再重构，否则你不知道自己是在"改善结构"还是在"引入 Bug"
生产坑警告：最危险的重构是"我只是改了个变量名"。IDE 的全局重命名看起来很安全，但如果项目里有反射（Class.forName、Spring 的 @Value("${field.name}")、MyBatis 的 XML 列映射），字符串引用不会被 IDE 同步修改，上线后才会发现运行时异常。

四、重构与测试的关系：安全网不是可选项

Fowler 把测试称为重构的安全网（Safety Net）。这个比喻很准确：高空走钢丝，有没有安全网是两种完全不同的心理状态和风险等级。

为什么没有测试的重构危险？

重构的核心承诺是"外部行为不变"，而测试是验证这个承诺的唯一客观手段。没有测试，你只能靠肉眼 review 和手工测试，这两者都无法覆盖所有边界场景，尤其是并发、异常路径和边界条件。

五、反例 → 问题 → 正例

场景：订单金额计算逻辑，需要把 if-else 链重构为策略模式

反例：没有测试直接动手

OrderService.java（重构前）反例
// 重构前：一坨 if-else，需要拆成策略
public BigDecimal calcDiscount(Order order) {
    if (order.getMemberLevel() == MemberLevel.VIP) {
        return order.getAmount().multiply(new BigDecimal("0.9"));
    } else if (order.getMemberLevel() == MemberLevel.SVIP) {
        return order.getAmount().multiply(new BigDecimal("0.8"));
    } else {
        return order.getAmount();
    }
}

// 开发者直接开始重构，没有任何单测
// 重构成策略模式后，手工点了几次前端页面，"看起来没问题"
// 上线后发现：SVIP 叠加满减券时，折扣被计算了两次
// 原因：策略类的 apply() 方法语义和旧逻辑有细微差别，肉眼没看出来

问题所在：没有测试，重构前后的行为差异无法被自动检测。手工测试只覆盖了主流程，漏掉了"叠加优惠券"这个边界场景。这类 Bug 在生产上往往涉及资金，后果严重。

正例：先写测试，再重构，每步验绿

OrderDiscountTest.java（先补测试）Step 1 — 补测试，锁定现有行为
@ExtendWith(MockitoExtension.class)
class OrderDiscountTest {

    private OrderService service = new OrderService();

    @Test
    void normalMember_noDiscount() {
        Order o = orderOf(MemberLevel.NORMAL, "100.00");
        assertThat(service.calcDiscount(o))
            .isEqualByComparingTo("100.00");
    }

    @Test
    void vipMember_tenPercentOff() {
        Order o = orderOf(MemberLevel.VIP, "100.00");
        assertThat(service.calcDiscount(o))
            .isEqualByComparingTo("90.00");
    }

    @Test
    void svipMember_twentyPercentOff() {
        Order o = orderOf(MemberLevel.SVIP, "100.00");
        assertThat(service.calcDiscount(o))
            .isEqualByComparingTo("80.00");
    }

    @Test
    void svipMember_withCoupon_discountNotDoubled() {
        // 关键边界：叠加优惠券时，会员折扣只作用于原价，不叠加
        Order o = orderOf(MemberLevel.SVIP, "100.00");
        o.setCouponDiscount(new BigDecimal("10.00"));
        // 期望：先算会员折扣 100*0.8=80，再减券 80-10=70
        assertThat(service.calcFinalPrice(o))
            .isEqualByComparingTo("70.00");
    }
}
OrderService.java（重构后）Step 2 — 重构，每步跑测试
// 策略接口：语义明确，只算"折后金额"，不叠加其他逻辑
public interface DiscountStrategy {
    BigDecimal apply(BigDecimal originalAmount);
}

// 具体策略：每次改完一个，立刻跑测试，绿了再改下一个
public enum MemberDiscountStrategy implements DiscountStrategy {
    NORMAL {
        public BigDecimal apply(BigDecimal amount) { return amount; }
    },
    VIP {
        public BigDecimal apply(BigDecimal amount) {
            return amount.multiply(new BigDecimal("0.9"));
        }
    },
    SVIP {
        public BigDecimal apply(BigDecimal amount) {
            return amount.multiply(new BigDecimal("0.8"));
        }
    };
}

public class OrderService {

    public BigDecimal calcDiscount(Order order) {
        // 通过 MemberLevel 直接映射策略，消灭 if-else
        DiscountStrategy strategy = MemberDiscountStrategy
            .valueOf(order.getMemberLevel().name());
        return strategy.apply(order.getAmount());
    }

    public BigDecimal calcFinalPrice(Order order) {
        // 流程清晰：先折扣，再减券。两步独立，不会交叉叠加
        BigDecimal afterDiscount = calcDiscount(order);
        return afterDiscount.subtract(order.getCouponDiscount());
    }
}

// Step 3：所有测试重新跑，全绿，重构完成
关键节奏：每完成一个最小步骤（如提炼一个方法、重命名一个变量）就跑一次测试。这样一旦测试变红，立刻知道是刚才那一步引入了问题，回退成本极低。而不是攒了 200 行改动再跑，那时候定位问题要花几倍的时间。

六、重构节奏与 CI 流水线

重构要真正落地，离不开 CI 的配合。一个好的 CI 配置应该让每次 push 都自动跑单测，把"测试绿灯"变成一个客观可见的信号，而不是靠人工记住"我改完要跑测试"。

重构工作流
├── 发现触发点（坏味道 / 添加功能 / 修 Bug）
├── 确认测试覆盖（没有 → 先补测试）
├── 最小步骤重构（每步跑测试，保持绿灯）
├── push → CI 流水线（自动运行全量测试）
└── 合并主干（CI 绿灯是硬门槛）

测试覆盖率的正确姿势

覆盖率是工具，不是目标。不要为了提高覆盖率数字而写没有断言的测试——那种测试只会执行代码，不会验证行为，在重构时毫无保护价值。真正有价值的是对业务语义的断言，即"这段代码在这种输入下应该产生这种输出/副作用"。

测试类型重构时的作用适合保护的场景
单元测试最快反馈，毫秒级纯计算逻辑、策略类、工具方法
集成测试验证模块间协作Service + Repository 的交互
契约测试防止 API 签名悄悄变化微服务接口、SDK 对外接口
端到端测试最慢，覆盖核心流程关键业务链路（下单、支付）

七、生产环境中的常见坑

  1. 重构与功能开发混在同一个 PR
    Review 时无法分辨哪些改动是"保持行为"的重构、哪些是"改变行为"的功能。一旦出问题，回滚代价极高。养成习惯：重构单独提 PR，功能开发单独提 PR。
  2. 在没有测试的遗留代码上直接大范围重构
    正确做法是先用"特性测试（Characterization Test）"锁定现有行为——不管代码逻辑对不对，先把它现在的输出记录成测试用例，再开始改。
  3. 用 IDE 自动重构功能时忽略非 Java 资源
    MyBatis XML 的 resultMap、Spring XML 配置中的 Bean 名、application.yml 里的属性名，IDE 重命名都无法同步。改完要全文搜索字符串引用。
  4. 重构产生了大量无意义的 git diff，污染历史
    格式化整个文件（如用 IDE 的"Reformat Code"一次性格式化 500 行）会让真正的逻辑改动淹没在格式噪音里，git blame 也会失效。只格式化你实际改动的部分。
  5. 把"性能优化"和"结构重构"混为一谈
    性能优化可能需要改变数据结构或算法，这往往确实改变了"内部结构"，但其驱动力和验证方式（性能基准测试）完全不同。不要把它和重构放在同一个提交里。

八、一句话总结

重构的时机：添加功能前、修 Bug 时、Code Review 时，都可以顺手做小步重构；大范围技术债需要单独排期并量化收益。

重构与测试的关系：测试是重构的安全网，没有测试的重构是赌博。先补测试，再重构，每步验绿，是唯一可靠的姿势。

  • 提炼函数：案例中把 calcDiscount 从 if-else 链重构为策略模式，第一步就是先识别出可以独立测试的最小单元——这正是 refactor/03-提炼函数.html 里"提炼函数是几乎所有其他重构手法的前置动作"这句话的具体印证。
  • 单元测试与 TDD：本文反复强调"测试是重构的安全网"，这与 clean/07-单元测试与TDD.html（若已覆盖）中测试驱动开发的理念一致——重构和 TDD 都依赖"小步快跑、每步验证"的工作节奏，只是触发顺序相反（TDD 先写测试再写代码，重构先有代码再补测试）。
  • 策略模式与开闭原则（OCP）：把 if-else 折扣计算重构为 MemberDiscountStrategy 枚举策略，正是 clean/08-SOLID原则.html 中 OCP 与 mechanism/04-设计套路落地.html 中策略模式的落地案例——重构的目标方向往往就是让代码更贴近某个 SOLID 原则。
  • 搬移特性与重新组织数据：本文的"三次法则"和"触发点"决定了*要不要*重构，而具体*怎么*重构则要靠 refactor/03-提炼函数.html、refactor/04-搬移特性.html、refactor/05-重新组织数据.html 中的手法库——这几篇文章共同构成了"识别 → 决策 → 执行"的完整重构工作流。
