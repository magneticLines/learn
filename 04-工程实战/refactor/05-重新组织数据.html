重构 · 常用手法

重新组织数据

让数据结构说真话——封装魔法值、用对象替代裸字段、消灭重复含义，是重构中最能提升可维护性的一类手法。

为什么要重新组织数据

代码里的数据结构往往是最早腐化的部分。一个系统诞生时只有几个字段，随着需求迭代，字段越加越多，魔法常量到处散落，一个"订单状态"可能同时存在 int status、String statusCode、boolean isPaid 三种表达方式，相互之间还可能不一致。

重新组织数据是 Fowler《重构》中一类手法的统称，核心目标是：

  • 让数据有明确的类型边界，不依赖"约定"来理解含义
  • 消灭散落各处的魔法值（magic number / magic string）
  • 用值对象（Value Object）封装具有业务含义的组合数据
  • 打破对外暴露可变内部集合的陷阱

手法一：以对象替代基本类型

当一个原始类型字段开始附带业务规则（范围校验、格式约束、派生计算），就应该把它提升为一个值对象。

反例 — 裸 String 承载电话号码

Java · 反例
public class Customer {
    private String phoneNumber;  // 格式？+86？分机号？谁都不知道

    public void setPhoneNumber(String phoneNumber) {
        // 每个调用方各自校验，格式验证散落在 Service / Controller / DTO
        this.phoneNumber = phoneNumber;
    }
}

问题：同一份校验逻辑散落在至少三个地方——Controller 接收时、Service 业务逻辑里、数据库存储前；一旦格式规则变化，需要改多处，极易遗漏。

正例 — PhoneNumber 值对象

Java · 正例
// 值对象：不可变，校验内聚，equals/hashCode 按值语义
public final class PhoneNumber {
    private static final Pattern PATTERN =
        Pattern.compile("^1[3-9]\\d{9}$");

    private final String value;

    public PhoneNumber(String raw) {
        // 创建即合法，外部永远拿不到一个非法的 PhoneNumber
        if (!PATTERN.matcher(raw).matches())
            throw new IllegalArgumentException("无效手机号: " + raw);
        this.value = raw;
    }

    public String desensitized() {
        // 派生行为内聚在对象里，不再散落到 Utils 类
        return value.substring(0, 3) + "****" + value.substring(7);
    }

    @Override public String toString() { return value; }
    // equals / hashCode 省略（按 value 比较）
}

public class Customer {
    private PhoneNumber phoneNumber;  // 类型即文档
}
实战经验：值对象在 JPA/MyBatis 中持久化时，可用 @Converter（JPA）或 TypeHandler（MyBatis）在框架层做透明转换，业务代码无感知。

手法二：以枚举替代魔法值

订单状态用 0/1/2/3，支付方式用 "alipay"/"wechat"/"card"……这类魔法值是代码腐化的温床：没有编译器约束、重构时无法安全重命名、新人读代码必须翻文档。

反例 — 魔法数字状态机

Java · 反例
// 看到 2 你知道这是什么状态吗？
if (order.getStatus() == 2) {
    sendShippingNotice(order);
}

// 更糟糕的：数据库里存 "PAID"，代码里写 "paid"，
// 某个工程师大小写不一致，线上排查两小时
if ("paid".equals(order.getStatusCode())) { ... }

正例 — 富枚举承载状态与行为

Java · 正例
public enum OrderStatus {
    PENDING  (0, "待支付"),
    PAID     (1, "已支付"),
    SHIPPED  (2, "已发货"),
    DONE     (3, "已完成"),
    CANCELLED(4, "已取消");

    private final int    code;
    private final String label;

    OrderStatus(int code, String label) {
        this.code  = code;
        this.label = label;
    }

    // 从数据库 code 反查枚举，避免 ordinal() 陷阱
    public static OrderStatus fromCode(int code) {
        for (OrderStatus s : values())
            if (s.code == code) return s;
        throw new IllegalArgumentException("未知状态码: " + code);
    }

    // 状态流转规则内聚在枚举里，不再散落到 if/else 链
    public boolean canTransitTo(OrderStatus next) {
        return switch (this) {
            case PENDING  -> next == PAID || next == CANCELLED;
            case PAID     -> next == SHIPPED || next == CANCELLED;
            case SHIPPED  -> next == DONE;
            default       -> false;
        };
    }
}

// 调用侧：可读、类型安全、IDE 自动补全
if (order.getStatus() == OrderStatus.SHIPPED) {
    sendShippingNotice(order);
}
生产坑：永远不要用 ordinal() 存数据库。枚举 ordinal 随声明顺序变化，中间插入一个枚举值就会导致历史数据全部错位。务必用显式 code 字段映射。

手法三：封装集合

把 List<Item> 直接作为字段对外暴露，调用方就可以绕过业务规则直接往里塞数据——这是最常见的"数据封装形同虚设"的场景。

反例 — 直接暴露可变集合

Java · 反例
public class ShoppingCart {
    private List<CartItem> items = new ArrayList<>();

    // 返回内部引用！外部可以随意 add/remove，绕过所有业务校验
    public List<CartItem> getItems() { return items; }
}

// 调用方直接写，完全不经过业务逻辑
cart.getItems().add(new CartItem("商品A", -1));  // 数量 -1！

正例 — 封装修改入口，只读对外暴露

Java · 正例
public class ShoppingCart {
    private static final int MAX_ITEMS = 99;
    private final List<CartItem> items = new ArrayList<>();

    // 唯一的写入入口，校验在这里统一做
    public void addItem(CartItem item) {
        Objects.requireNonNull(item);
        if (item.getQuantity() <= 0)
            throw new IllegalArgumentException("数量必须 > 0");
        if (items.size() >= MAX_ITEMS)
            throw new IllegalStateException("购物车已满");
        items.add(item);
    }

    // 外部只读视图，无法通过返回值修改内部状态
    public List<CartItem> getItems() {
        return Collections.unmodifiableList(items);
    }

    public Money totalPrice() {
        // 聚合计算内聚在对象里
        return items.stream()
            .map(CartItem::subtotal)
            .reduce(Money.ZERO, Money::add);
    }
}

手法四：拆分变量

一个变量被赋值两次以上，却承担了两个完全不同的职责——这是极其常见的可读性杀手。

反例 — 一个变量身兼两职

Java · 反例
double temp = 2 * (height + width);
System.out.println("周长: " + temp);

// temp 被复用，含义完全变了，读者必须重新在大脑里追踪它
temp = height * width;
System.out.println("面积: " + temp);

正例 — 每个概念一个变量

Java · 正例
final double perimeter = 2 * (height + width);  // final 表达不可变意图
System.out.println("周长: " + perimeter);

final double area = height * width;
System.out.println("面积: " + area);
延伸：在循环变量、累加器等场合，i/sum 天然就是"可重复赋值"的，不需要拆分。拆分变量针对的是语义发生了根本变化的情况。

手法五：将引用对象改为值对象

当一个对象只是一组数据的容器，没有独立的生命周期，不需要被共享修改，就应该用不可变值对象而非可变引用对象来表达。典型例子：金额、坐标、地址、日期范围。

反例 — 可变 Money 对象引发的数据错位

Java · 反例
public class Money {
    private BigDecimal amount;
    private String     currency;

    // setter 存在 → 任何拿到引用的代码都能悄悄改金额
    public void setAmount(BigDecimal a) { this.amount = a; }
}

// 订单和发票共享同一个 Money 对象，修改一处影响另一处
Money price = new Money("CNY", new BigDecimal("100"));
order.setPrice(price);
invoice.setAmount(price);                           // 同一引用！
price.setAmount(new BigDecimal("50"));  // 灾难：发票金额也变了

正例 — 不可变值对象

Java · 正例
public final class Money {
    private final BigDecimal amount;
    private final String     currency;

    public Money(String currency, BigDecimal amount) {
        this.currency = Objects.requireNonNull(currency);
        this.amount   = Objects.requireNonNull(amount);
    }

    // 返回新对象，原对象不受影响，天然线程安全
    public Money add(Money other) {
        if (!currency.equals(other.currency))
            throw new IllegalArgumentException("货币类型不一致");
        return new Money(currency, amount.add(other.amount));
    }

    public BigDecimal getAmount() { return amount; }
    // equals/hashCode 按 amount + currency 比较
}

手法六：以查询替代派生字段

当一个字段的值可以完全由其他字段计算得出，存储它就会引入"数据不一致"的风险——两处修改必须保持同步，但代码里往往只更新了一处。

反例 — 冗余的 totalPrice 字段

Java · 反例
public class OrderLine {
    private int        quantity;
    private BigDecimal unitPrice;
    private BigDecimal totalPrice;  // 派生字段，随时可能与前两者不一致

    public void setQuantity(int q) {
        this.quantity = q;
        // 开发者忘记同步 totalPrice → 数据错位
        // this.totalPrice = unitPrice.multiply(BigDecimal.valueOf(q));
    }
}

正例 — 用方法即时计算

Java · 正例
public class OrderLine {
    private int        quantity;
    private BigDecimal unitPrice;

    // 永远准确，不存在"忘记同步"的问题
    public BigDecimal totalPrice() {
        return unitPrice.multiply(BigDecimal.valueOf(quantity));
    }
}
性能考量：如果该字段在热路径上被频繁读取（如缓存 key、数据库排序列），可以保留存储但采用"写入时更新"模式，并用单元测试保证一致性。不要把"可能有性能问题"作为不重构的默认借口——先测量，再优化。

常用手法速查

手法 触发信号 核心收益
以对象替代基本类型 原始类型开始有校验、格式化、派生逻辑 校验内聚、类型文档化
以枚举替代魔法值 多处出现 == 0、.equals("paid") 编译器保护、消灭散落校验
封装集合 getList().add() 绕过业务规则 不变式有保障
拆分变量 变量被二次赋值且含义不同 可读性、可 final 化
改为值对象 共享引用 + 可变 → 神秘数据变化 线程安全、无共享副作用
以查询替代派生字段 字段值 = 其他字段的计算结果 消灭数据不一致根源

生产环境中的高频坑

  • 枚举用 ordinal() 存数据库，插入新枚举项后历史数据全部错位——必须用显式 code 字段
  • 值对象忘记重写 equals/hashCode，放进 Set/Map 后行为异常
  • 返回内部集合的引用，调用方通过返回值删除元素，触发 ConcurrentModificationException
  • BigDecimal 做金额时用 == 比较（引用比较），应使用 compareTo 或 equals
  • 值对象序列化时缺少无参构造器，Jackson/MyBatis 反序列化失败
  • 派生字段存入数据库后，应用层和 SQL 层各自计算出不同结果，形成"两个真相"
重构节奏建议：不要一次性将所有裸字段提升为值对象。优先处理被多处读取且有业务含义的字段（如状态、金额、手机号），每次只动一个概念，保持测试绿灯。
