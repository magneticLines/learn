重构 · 认知:何时重构

代码的坏味道（Code Smells）

Martin Fowler 在《重构》中提出：坏味道不是 Bug，但它是 Bug 孕育的温床。识别坏味道，是决定"何时重构"的第一步。

什么是坏味道，为什么它重要

"代码坏味道"（Code Smell）是 Kent Beck 和 Martin Fowler 在合著《重构》时引入的术语。它不是编译错误，也不是运行时异常，而是代码在结构上发出的警告信号——提示你这里的设计可能有问题，未来改动时会付出额外代价。

生产环境中，坏味道最直接的危害不在于"代码丑"，而是变更成本指数级上升。一个需求改动本来 1 小时能完成，但因为到处是重复代码、超长方法、神类，你要花 4 小时理解上下文，再花 2 小时修复因为改了一处而引发的 3 个其他地方的 Bug。

核心认知：坏味道是"重构信号"，而非"必须立刻修"的命令。识别它是为了在合适的时机（如需求变更、添加新功能时）做针对性重构，而不是为了重构而重构。

22 种坏味道速览（按影响程度分组）

Fowler 在《重构（第 2 版）》中整理了 22 种坏味道。按照生产环境中的危害频率，分三个层级：

层级 坏味道名称 一句话描述
高频 · 高危 Duplicated Code（重复代码） 同样的逻辑出现在多处，改一处忘改另一处
Long Method（过长函数） 方法太长，难以理解、测试和复用
Large Class（神类） 一个类承担了太多职责，违反 SRP
Long Parameter List（过长参数列表） 参数超过 3~4 个，调用时极易传错顺序
Divergent Change（发散式变化） 一个类因为多种不同原因被修改
Shotgun Surgery（散弹式修改） 一个变化需要修改多个类
中频 · 中危 Feature Envy（依恋情结） 方法更多地使用其他类的数据，而非本类
Data Clumps（数据泥团） 多处出现同一组字段，应封装为对象
Switch Statements（Switch 惊悚现身） 大量 if-else/switch 分支，扩展时需同步修改
Primitive Obsession（基本类型偏执） 用 String/int 表达业务概念，缺少领域类型
Parallel Inheritance Hierarchies（平行继承体系） 每增加一个子类，另一个体系也必须跟着加
Lazy Class（冗余类） 一个类做的事情太少，不值得存在
Speculative Generality（过度设计） 为了"将来可能用到"引入不必要的抽象
Temporary Field（临时字段） 字段只在某些情况下有值，其余时候为 null
Message Chains（消息链） a.getB().getC().getD() 式的调用链
Middle Man（中间人） 类的大部分方法只是委托给另一个类
低频 · 局部 Inappropriate Intimacy（过度耦合） 两个类互相深入访问对方私有数据
Alternative Classes with Different Interfaces 功能相似但接口不同的类，无法互换
Incomplete Library Class（不完美的库类） 库类不满足需求，且无法修改
Data Class（纯数据类） 只有字段 + getter/setter，行为全在外部
Refused Bequest（被拒绝的遗赠） 子类不想要父类的部分方法或数据
Comments（注释） 注释本身没问题，但过多注释往往掩盖了坏设计

下面重点剖析生产环境中最常出现、危害最大的几种。

重点坏味道详解

1. 重复代码（Duplicated Code）— 最高发坏味道

重复代码是所有坏味道中发生频率最高的。它的危害不在于写的时候，而在于维护的时候：改 A 处忘了改 B 处，线上出现不一致的行为，这类 Bug 极难排查。

生产坑：订单服务和退款服务都有一份"计算运费折扣"的逻辑，运营改了折扣规则，开发只改了订单服务，退款服务漏掉了。上线后退款金额计算错误，资损。
❌ 反例 — 两处重复的折扣计算Java
// OrderService.java
public BigDecimal calcShippingFee(Order order) {
    BigDecimal fee = order.getBaseShippingFee();
    if (order.isMember()) fee = fee.multiply(new BigDecimal("0.8"));
    if (order.getAmount().compareTo(new BigDecimal("200")) >= 0) fee = BigDecimal.ZERO;
    return fee;
}

// RefundService.java — 完全相同的逻辑，复制粘贴而来
public BigDecimal calcRefundShippingFee(Order order) {
    BigDecimal fee = order.getBaseShippingFee();
    if (order.isMember()) fee = fee.multiply(new BigDecimal("0.8"));
    if (order.getAmount().compareTo(new BigDecimal("200")) >= 0) fee = BigDecimal.ZERO;
    return fee;
}
✅ 正例 — 提取到领域服务，单一来源Java
// ShippingFeeCalculator.java — 唯一真相来源
@Component
public class ShippingFeeCalculator {

    // 所有折扣规则集中在此：改一处，全局生效
    public BigDecimal calculate(Order order) {
        BigDecimal fee = order.getBaseShippingFee();
        if (order.isMember())             fee = fee.multiply(new BigDecimal("0.8"));
        if (isFullFreeShipping(order))    fee = BigDecimal.ZERO;
        return fee;
    }

    private boolean isFullFreeShipping(Order order) {
        return order.getAmount().compareTo(new BigDecimal("200")) >= 0;
    }
}

2. 过长函数（Long Method）— 可读性杀手

Fowler 的经验法则：如果你需要注释来解释一段代码在做什么，就应该把这段代码提取成一个方法，用方法名来表达意图。

真实项目中，一个处理"下单"的方法往往包含：参数校验 → 库存检查 → 优惠券验证 → 价格计算 → 创建订单 → 发消息通知，全塞在一个 200 行方法里。问题在于：

  • 新人接手时，无法快速定位某个子逻辑在哪里
  • 单元测试只能整体测，无法单独测"优惠券验证"
  • 多人并行开发时，大概率产生 Git 冲突
❌ 反例 — 一个方法干所有事Java
public OrderResult placeOrder(PlaceOrderRequest req) {
    // === 1. 参数校验 ===
    if (req.getUserId() == null) throw new IllegalArgumentException("userId 不能为空");
    if (req.getItems().isEmpty()) throw new IllegalArgumentException("商品不能为空");
    // ... 更多校验

    // === 2. 库存检查 ===
    for (OrderItem item : req.getItems()) {
        Stock stock = stockRepo.findById(item.getSkuId());
        if (stock.getAvailable() < item.getQty())
            throw new BizException("库存不足");
    }

    // === 3. 价格计算（含优惠券、折扣，省略几十行）===
    BigDecimal totalAmount = calcTotal(req);
    // ... 省略更多逻辑
}
✅ 正例 — 每个方法名即文档Java
public OrderResult placeOrder(PlaceOrderRequest req) {
    // 读这 5 行，就知道下单流程的全貌
    validateRequest(req);
    checkStock(req.getItems());
    BigDecimal amount = calcAmount(req);
    Order order       = createOrder(req, amount);
    publishOrderCreatedEvent(order);
    return OrderResult.success(order.getId());
}

// 每个子方法职责单一，可独立测试
private void checkStock(List<OrderItem> items) {
    items.forEach(item -> {
        Stock stock = stockRepo.findById(item.getSkuId());
        if (stock.getAvailable() < item.getQty())
            throw new StockInsufficientException(item.getSkuId());
    });
}

3. 散弹式修改 vs 发散式变化 — 最易混淆的一对

这两种坏味道互为镜像，理解它们需要分清"变化方向"：

坏味道表现根因重构方向
散弹式修改 改一个需求，要动 N 个类 一个职责被拆散在多个类里 内联/合并：把相关逻辑聚合
发散式变化 一个类因为 N 种原因被修改 一个类承担了多个职责 提取类：把不同职责分离

典型生产场景：每次新增支付渠道，需要同时修改 PaymentService、RefundService、ReconciliationService、NotificationService——这是典型的散弹式修改，说明"支付渠道适配"这个职责被散落在各处，应该用策略模式 + 渠道抽象层集中。

4. 基本类型偏执（Primitive Obsession）— 领域模型贫血的根源

用 String 表示手机号、用 int 表示金额（单位分）、用 String 表示订单状态……这不仅让代码丧失表达力，还会导致：

  • 手机号格式校验散落在各处，不一致
  • 金额单位（分/元）混用，引发计算错误
  • 字符串状态可以随意赋非法值，没有编译期保护
❌ 反例 — 基本类型堆砌Java
// 调用方不看文档根本不知道 amount 是"分"还是"元"
public void createOrder(String userId, String phone,
                         long amount, String status) {
    // 各处都要重复校验 phone 格式
    if (!phone.matches("^1[3-9]\\d{9}$"))
        throw new IllegalArgumentException("手机号格式错误");
    // status 可以是任意字符串，无法在编译期防止非法值
}
✅ 正例 — 引入值对象和枚举Java
// 手机号值对象：校验逻辑封装在构造时，一次定义处处有效
public final class PhoneNumber {
    private final String value;

    public PhoneNumber(String value) {
        if (!value.matches("^1[3-9]\\d{9}$"))
            throw new IllegalArgumentException("手机号格式错误: " + value);
        this.value = value;
    }
}

// Money 值对象：单位明确，运算安全
public final class Money {
    private final long amountInFen;  // 字段名本身就是文档
    private final Currency currency;

    public Money add(Money other) {
        if (!this.currency.equals(other.currency))
            throw new CurrencyMismatchException();
        return new Money(this.amountInFen + other.amountInFen, this.currency);
    }
}

// 枚举替代字符串状态：编译期保证合法值
public enum OrderStatus {
    PENDING, PAID, SHIPPED, COMPLETED, CANCELLED
}

5. 过长参数列表（Long Parameter List）

方法参数超过 3~4 个，是典型的坏味道。顺序依赖是根本问题：调用方必须记住参数顺序，类型相近时极易传错（如两个 String 参数互换）。

❌ 反例 — 6 个参数，你能记住顺序吗Java
public void sendNotification(
        String userId, String title, String content,
        String channel, boolean urgent, LocalDateTime sendAt) {
    // ...
}

// 调用处：title 和 content 传反了，编译不报错，静默出错
sendNotification("u123", "您的订单已发货，请注意查收...", "发货通知",
                 "SMS", false, null);
✅ 正例 — 引入参数对象Java
// 用 Builder 封装：字段名自描述，顺序无关，扩展无需改签名
NotificationRequest req = NotificationRequest.builder()
    .userId("u123")
    .title("发货通知")
    .content("您的订单已发货，请注意查收...")
    .channel(Channel.SMS)
    .urgent(false)
    .build();
notificationService.send(req);

6. Switch / 大量 if-else（Switch Statements）

switch 本身没有问题，问题在于同一个 switch(type) 的判断逻辑散落在代码各处。每新增一种类型，需要找到所有相关的 switch 逐个修改，极易遗漏。

❌ 反例 — 类型判断到处复制Java
// 计算运费时
switch (order.getType()) {
    case "NORMAL":  fee = normalFee;  break;
    case "EXPRESS": fee = expressFee; break;
}
// 计算预计送达时间时
switch (order.getType()) {
    case "NORMAL":  days = 3; break;
    case "EXPRESS": days = 1; break;
}
// 新增"同城配送"时，需要找到所有 switch 挨个加 case
✅ 正例 — 策略模式消灭 switchJava
// 每种配送类型是一个独立策略，新增类型只需新增实现类
public interface DeliveryStrategy {
    BigDecimal calcFee(Order order);
    int        estimateDays(Order order);
}

@Component
public class DeliveryStrategyFactory {
    // Spring 自动注入所有实现，新增实现类无需改工厂
    private final Map<String, DeliveryStrategy> strategies;

    public DeliveryStrategyFactory(List<DeliveryStrategy> list) {
        this.strategies = list.stream()
            .collect(Collectors.toMap(s -> s.getClass().getSimpleName(), s -> s));
    }

    public DeliveryStrategy get(String type) {
        return Optional.ofNullable(strategies.get(type))
            .orElseThrow(() -> new UnsupportedDeliveryTypeException(type));
    }
}

7. 数据泥团（Data Clumps）

如果在多个地方看到同一组字段总是一起出现（如 province、city、district、detailAddress），它们就应该被封装成一个对象 Address。

判断方法：删掉其中一个字段，其余字段还有意义吗？如果没有，它们就是数据泥团。

✅ 正例 — 封装为值对象Java
// 封装前：address 相关字段到处传递
public void createShipment(String province, String city,
                            String district, String detail) {}

// 封装后：地址概念内聚，Address 可复用于订单、退款、仓库等多个场景
public void createShipment(Address address) {}

public final class Address {
    private final String province;
    private final String city;
    private final String district;
    private final String detail;

    // 格式化地址的行为也归属于此，不会再散落在 Service 各处
    public String toFullAddress() {
        return province + city + district + detail;
    }
}

坏味道快速识别速查

以下是生产 Code Review 中最实用的快速识别问题清单：

重复代码：Ctrl+F 搜一下，同样的表达式在 3 处以上
过长函数：函数超过 30 行；或者你在方法里写了分隔注释 "=== 第2步 ==="
神类：类的字段超过 10 个；类的方法超过 20 个；import 超过 30 行
参数过长：方法签名超过 4 个参数，或有 2 个以上同类型相邻参数
基本类型偏执：用 String 表达电话、邮箱、状态、金额
散弹式修改：需求卡片需要改动 5 个以上文件，且这些文件跨多个包
消息链：代码中出现 a.getB().getC().getD()
数据泥团：同一组字段在多个方法签名中一起出现 3 次以上

生产环境中的常见坑

坑 1：把坏味道当成 Bug 立刻修
坏味道不是 Bug，不需要立刻全部消灭。在不需要修改的代码上强行重构，反而会引入新的问题，浪费 sprint 工时。正确做法：在修改相关代码时顺带重构（童子军原则：离开时比来时更整洁）。
坑 2：重构没有测试保护
重构的前提是有单元测试兜底。如果没有测试就改重复代码、提取方法，很容易在"微小改动"中引入行为变化。重构三步走：先补测试 → 再重构 → 跑测试验证行为不变。
坑 3：过度设计也是坏味道
Speculative Generality（过度设计）同样是坏味道。为了"将来某天可能会有第二种实现"引入 Strategy 接口，但实际上永远只有一种实现——这种抽象增加了阅读负担，是不必要的复杂度。YAGNI（You Aren't Gonna Need It）原则同样适用于重构。
工具提示：IntelliJ IDEA 内置了大量坏味道检测，Analyze → Inspect Code 可以扫描出过长方法（默认 >30 行）、过多参数（默认 >5 个）等。SonarQube 在 CI 流水线中集成后可以持续跟踪代码质量趋势。

坏味道与"何时重构"的关系

坏味道是"重构信号"，但触发重构动作的往往是外部事件。Fowler 总结了三个最常见的时机：

  • 添加功能时：新需求让你深入某段代码，发现坏味道，顺手重构再加功能——比事后专门重构成本更低
  • 修复 Bug 时：如果一个 Bug 因为重复代码造成，修完 Bug 就顺手提取消除重复
  • Code Review 时：Review 阶段发现坏味道，在合并前重构，成本最小

与其他知识点的联系

  • 单一职责原则（SRP）："神类""发散式变化"这两种坏味道本质上都是 SRP 被违反的表现——一个类承担了多个变化的理由，参见 clean/08-SOLID原则.html 中 Report 类被拆分为三个单一职责类的例子，思路完全一致。
  • 开闭原则（OCP）与策略模式："Switch Statements"坏味道的修正方案（用策略模式消灭 switch）与 clean/08-SOLID原则.html 中 OCP 违反 → 修正的案例、以及 mechanism/04-设计套路落地.html 中策略模式一节，讲的是同一件事的三个角度：坏味道诊断、原则定义、模式落地。
  • 提炼函数："过长函数"坏味道的修正方式正是下一篇 refactor/03-提炼函数.html 的核心手法——按注释分隔的"第几步"代码块，天然对应可以提炼成独立方法的边界。
  • 搬移特性："依恋情结（Feature Envy）"坏味道（虽未在本文详细展开，但属于 Fowler 22 种之一）与 refactor/04-搬移特性.html 中"方法更多使用另一个对象的数据"的判断标准是同一个信号，重构方向就是把方法搬到它真正依恋的那个类里。
  • 基本类型偏执与重新组织数据：本文"基本类型偏执"一节引入的 PhoneNumber、Money 值对象和 OrderStatus 枚举，会在 refactor/05-重新组织数据.html 中作为独立手法（以对象替代基本类型、以枚举替代魔法值）被系统化讲解，坏味道是信号，重新组织数据是具体处方。
  • 何时重构：本文结尾"坏味道与重构时机的关系"直接承接下一篇 refactor/02-何时重构.html 的三次法则和四个触发点——坏味道回答"哪里该改"，何时重构回答"什么时候动手改"，两者缺一不可。
