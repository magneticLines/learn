提炼函数 / 内联函数

重构频率最高的两个互逆手法——什么时候拆、什么时候合，以及生产环境里真正会踩的坑。

为什么这两个手法排在"最常用"第一位

Martin Fowler 在《重构》第二版中把提炼函数（Extract Function）列为最常施用的手法。 几乎所有其他重构——搬移特性、以多态取代条件表达式——最终都依赖先把代码切成足够小的函数。 而内联函数（Inline Function）是它的逆操作：当一个函数的函数体和其名字一样清晰、 或者一层层薄薄的委托反而遮蔽了真实意图时，就该把它合回去。

两者并不是对立的。它们更像剪刀和胶水：代码先随业务野蛮生长，提炼函数把混乱切开， 内联函数把无意义的间接层粘回去。循环往复，代码才会越来越干净。

提炼函数（Extract Function）

核心判断标准：意图与实现的距离

Kent Beck 给出的经验规则是：如果你需要花超过几秒才能理解一段代码在做什么， 就把它提炼出来，用一个能说明意图的名字命名。 长度不是触发条件，意图不透明才是。哪怕只有一行，只要名字能让阅读者"不用看实现就明白发生了什么"，提炼就值得。

函数名是给阅读者看的，不是给编译器看的。 calculateDiscountedPrice() 比注释 // 计算折扣价 更可靠——注释会撒谎，函数名随代码一起演进。

反例：一块 80 行的订单处理方法

这是真实业务中很常见的形态——一个 Service 方法把校验、计算、持久化全塞在一起：

OrderService.java — 重构前
public OrderResult submitOrder(OrderRequest req) {

    // ---- 校验库存 ----
    List<String> outOfStock = new ArrayList<>();
    for (OrderItem item : req.getItems()) {
        int stock = inventoryRepo.findStock(item.getSkuId());
        if (stock < item.getQuantity()) {
            outOfStock.add(item.getSkuId());
        }
    }
    if (!outOfStock.isEmpty()) {
        return OrderResult.fail("库存不足: " + outOfStock);
    }

    // ---- 计算总价 ----
    BigDecimal total = BigDecimal.ZERO;
    for (OrderItem item : req.getItems()) {
        BigDecimal price = productRepo.findPrice(item.getSkuId());
        BigDecimal discount = promotionService.getDiscount(item.getSkuId(), req.getUserId());
        total = total.add(price.multiply(discount).multiply(
            BigDecimal.valueOf(item.getQuantity())));
    }

    // ---- 创建订单并持久化 ----
    Order order = new Order();
    order.setUserId(req.getUserId());
    order.setItems(req.getItems());
    order.setTotalAmount(total);
    order.setStatus(OrderStatus.PENDING);
    order.setCreatedAt(LocalDateTime.now());
    orderRepo.save(order);

    return OrderResult.success(order.getId());
}

问题在哪里？

  • 阅读者必须从头到尾扫完才能理解整体流程，认知负担极高
  • 三块逻辑完全独立，却被迫共享同一个方法的局部作用域
  • 测试时无法单独验证"计算总价"逻辑，只能走完整 E2E 路径
  • 后续需求（如增加满减规则）必须在 80 行里找到正确插入点，出错概率高

正例：提炼后的结构

OrderService.java — 重构后
public OrderResult submitOrder(OrderRequest req) {
    // 主流程一眼可读，不需要看任何实现细节
    List<String> outOfStock = findOutOfStockSkus(req.getItems());
    if (!outOfStock.isEmpty()) {
        return OrderResult.fail("库存不足: " + outOfStock);
    }
    BigDecimal total = calculateTotal(req);
    Order order  = buildAndSaveOrder(req, total);
    return OrderResult.success(order.getId());
}

// 每个私有方法只做一件事，命名即文档
private List<String> findOutOfStockSkus(List<OrderItem> items) {
    return items.stream()
        .filter(item ->
            inventoryRepo.findStock(item.getSkuId()) < item.getQuantity())
        .map(OrderItem::getSkuId)
        .collect(Collectors.toList());
}

private BigDecimal calculateTotal(OrderRequest req) {
    return req.getItems().stream()
        .map(item -> {
            BigDecimal price    = productRepo.findPrice(item.getSkuId());
            BigDecimal discount = promotionService.getDiscount(
                item.getSkuId(), req.getUserId());
            return price.multiply(discount)
                        .multiply(BigDecimal.valueOf(item.getQuantity()));
        })
        .reduce(BigDecimal.ZERO, BigDecimal::add);
}

private Order buildAndSaveOrder(OrderRequest req, BigDecimal total) {
    Order order = Order.builder()
        .userId(req.getUserId())
        .items(req.getItems())
        .totalAmount(total)
        .status(OrderStatus.PENDING)
        .createdAt(LocalDateTime.now())
        .build();
    orderRepo.save(order);
    return order;
}

重构后主方法只剩 5 行，读起来像伪代码：检查库存 → 算价格 → 保存订单。 每个子方法都可以独立单测，以后改折扣逻辑只动 calculateTotal，不影响其他部分。

提炼函数的操作步骤（不依赖 IDE 手工重构时）

  1. 圈出要提炼的代码段，分析它使用了哪些局部变量
  2. 判断这些变量在提炼后是充当入参还是返回值
  3. 若有多个变量需要返回，先考虑是否可以提炼成独立对象，或拆成更小的函数
  4. 给新函数起一个动词短语命名（说明它做了什么，而不是怎么做）
  5. 编译 + 运行已有测试，确认行为未变
IntelliJ IDEA 快捷键：选中代码段 → Ctrl + Alt + M（Windows/Linux）/ Cmd + Alt + M（Mac）可自动完成提炼函数，IDE 会帮你分析入参和返回值。强烈推荐先手工推演一次，再用 IDE 执行，确保理解而不是盲目操作。

内联函数（Inline Function）

什么时候该"合回去"

内联函数是提炼函数的逆操作，但它不是"撤销"，而是另一种判断。 当满足以下任意一条时，就应该果断内联：

  • 函数体和函数名说的是同一件事，名字没有带来额外信息
  • 函数只有一个调用点，且该调用点的上下文已经足够清晰
  • 函数是一堆过度分解留下的碎片，层层委托让调试时的调用栈毫无意义
  • 准备做更大规模的重组（如搬移特性），先内联以便重新切割

反例：过度提炼导致的"导航迷宫"

UserValidator.java — 过度提炼
public boolean isEligible(User user) {
    return isActive(user) && isAdult(user);
}

private boolean isActive(User user) {
    return user.getStatus() == UserStatus.ACTIVE;
}

private boolean isAdult(User user) {
    return user.getAge() >= 18;
}

isActive 和 isAdult 的函数体就是一个表达式，名字并没有提供超越代码本身的信息。 阅读者反而要在三个方法之间跳转，增加了认知摩擦。

正例：内联后更清晰

UserValidator.java — 内联后
public boolean isEligible(User user) {
    // 两个条件都一目了然，不需要额外命名
    return user.getStatus() == UserStatus.ACTIVE
        && user.getAge() >= 18;
}
内联函数最容易踩的坑：被内联的函数有多个调用点。 内联时必须逐一替换，漏掉任何一处都会产生行为差异。 IDE 的"Find Usages"（Alt+F7）是必须要用的，手工查找极易遗漏。

两者的对称关系与决策树

这段代码让我停顿超过几秒？ ├── 是 → 考虑 提炼函数 │ ├── 能用一个动词短语概括意图？→ 提炼，并用该短语命名 │ └── 不能概括 → 先理解业务，再决定是否提炼 └── 否 → 这个函数只有一行 / 名字等于实现？ ├── 是 → 考虑 内联函数 └── 否 → 保持现状
维度 提炼函数 内联函数
触发时机 意图不透明、代码重复、函数过长 名字无增量信息、过度间接、准备重组
核心收益 提升可读性、可测试性、可复用性 降低导航成本、减少无意义层级
主要风险 入参/返回值分析错误导致行为改变 遗漏多调用点；内联后方法变长
IDE 快捷键（IDEA） Ctrl+Alt+M Ctrl+Alt+N

生产环境中的常见坑

坑一：提炼函数后"悄悄"改变了执行次数

将循环体中的代码提炼成函数时，如果原始代码依赖循环变量或有副作用， 需要格外注意提炼后是否意外多执行或少执行了某些操作。 例如将带有 i++ 副作用的表达式提炼成方法入参，求值顺序可能改变。

坑二：内联函数时漏掉继承体系中的 override

如果被内联的方法在子类中被 override，直接删掉父类方法会破坏多态。 操作前必须检查是否有子类重写。IDE 的"Find Usages + 层次结构视图"可帮你发现。

坑三：提炼函数命名用"技术动词"而非"业务动词"

命名对比
// 坏：技术动词，描述"怎么做"
private boolean checkListAndCompareWithThreshold(List<Order> orders) { ... }

// 好：业务动词，描述"做什么"
private boolean hasExceededDailyOrderLimit(List<Order> orders) { ... }

坑四：在事务边界内随意提炼导致事务失效

Spring 的 @Transactional 依赖 AOP 代理，同一个类内部的方法调用不走代理。 如果把一段带有数据库写操作的代码提炼成同类的私有方法，事务传播行为不变，没有问题。 但若误将其提炼到另一个 Bean，且新 Bean 没有加 @Transactional，事务就会断裂。 提炼跨 Bean 的函数时，务必评估事务、安全上下文、ThreadLocal 传播是否受影响。

高频面试点：@Transactional 的自调用失效问题，本质就是"提炼函数提炼到了同一个类内部"的场景。 理解了这一点，面试官问"Spring 事务什么时候失效"就能从重构视角给出有深度的答案。

坑五：过度提炼让函数数量爆炸，类变成"方法目录"

把每一个表达式都提炼成函数是另一个极端。当一个类有 50 个私有方法， 每个方法只有 1~2 行，阅读者同样无法建立整体认知。 提炼的粒度应服务于可读性，而不是行数最小化。 经验法则：提炼后的函数，如果你能在 30 秒内向同事口头描述它在做什么，粒度就是合适的。

提炼函数与测试覆盖的关系

提炼函数最直接的工程收益之一是提升可测试性。 原本埋在 80 行方法里的"计算折扣"逻辑，提炼成 calculateTotal() 之后， 可以独立编写单元测试，覆盖各种折扣组合，而不必 Mock 整个订单提交链路。

反过来，没有测试时进行大规模提炼函数风险极高——你无法验证重构前后行为是否等价。 正确的顺序是：先补充集成测试（或至少主流程的 Happy Path 测试）， 再施用提炼函数，每提炼一个函数就跑一次测试，保持绿灯。

这也是为什么《重构》第二版专门有一章讲"重构与测试的关系"——两者是共生关系，不是先后关系。 没有测试的重构叫"碰运气"，有测试的重构才叫"有纪律的演进"。

小结

  • 提炼函数的触发标准是意图不透明，而不是代码太长
  • 命名要用业务动词短语，名字本身就是文档
  • 内联函数是合理的反向操作，不是失败的撤销——过度分解同样是坏味道
  • 操作前要有测试保护；跨 Bean 提炼时检查事务、安全上下文等横切关注点
  • 两个手法互为工具，交替使用才能让代码持续向更清晰的方向演进

与其他知识点的联系

  • 代码坏味道——过长函数：本文的反例（80 行的 submitOrder）正是 refactor/01-代码坏味道.html 中"Long Method"坏味道的具体案例，提炼函数是这种坏味道的标准处方。
  • 何时重构：refactor/02-何时重构.html 中"先补测试、再重构、每步验绿"的节奏，在本文"提炼函数与测试覆盖的关系"一节里被具体化为"提炼一个函数就跑一次测试"，两篇讲的是同一套工程纪律在不同粒度上的应用。
  • 搬移特性：提炼函数是搬移特性的前置步骤——本文"两者的对称关系"一节提到"准备做更大规模的重组时，先内联以便重新切割"，而重新切割之后往往就是把新提炼出的函数搬到它真正归属的类，这正是 refactor/04-搬移特性.html 的主题。
  • AOP 与事务失效："坑四：在事务边界内随意提炼导致事务失效"直接对应 mechanism/02-AOP.html 中"同类内部调用切面不生效"的坑——两者根因相同：Spring AOP 依赖代理对象拦截调用，而同类内部的方法调用（包括提炼后新方法）走的是 this 引用，绕开了代理。
  • 单一职责原则（SRP）：提炼函数的核心判断标准"意图与实现的距离"，本质上是把一个函数里混杂的多个职责拆分成命名清晰的子函数，这是 SRP（clean/08-SOLID原则.html）在方法粒度上的应用，而不仅仅是类粒度。
