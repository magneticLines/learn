OAuth2 概览（四种授权模式）

OAuth2 不是认证协议，是授权委托协议。理解它要从"第三方如何安全获取权限"出发，而不是从"用户如何登录"出发。

1. OAuth2 解决了什么问题

设想这个场景：你在用一款打印服务，它需要读取你 Google Drive 里的文件。在 OAuth2 出现之前，唯一的做法是把 Google 账号密码告诉这个打印服务，让它替你登录。

这会带来什么问题？

  • 打印服务拿到密码后，权限无边界——它可以删你的文件、改你的邮箱
  • 你无法单独撤销打印服务的权限，只能改密码（同时踢掉所有其他应用）
  • 打印服务一旦泄漏数据库，你的 Google 账号直接裸奔

OAuth2 的解法是引入一个授权服务器，让用户授予第三方"有限的、可撤销的权限"，整个过程用户密码从不离开授权服务器。

OAuth2 vs OIDC：OAuth2 只解决"授权"（你能做什么），不解决"认证"（你是谁）。OpenID Connect（OIDC）是在 OAuth2 之上增加了身份层（id_token），"用 GitHub 登录"这类功能走的是 OIDC，不是纯 OAuth2。

2. 四个核心角色

角色英文例子
资源所有者Resource Owner用户（你）
客户端Client打印服务（第三方应用）
授权服务器Authorization ServerGoogle 的 OAuth 服务
资源服务器Resource ServerGoogle Drive API

授权服务器和资源服务器可以是同一个服务（自建 OAuth），也可以分离（大厂通常分离部署）。

3. 四种授权模式

① 授权码模式（Authorization Code）最安全，推荐

用于有后端的 Web 应用。核心安全点：access token 只在后端流转，永远不经过浏览器。

用户 ── 点击"用 GitHub 登录" ──► 客户端（你的应用） 客户端 ── 重定向 ────────────────► 授权服务器 携带：client_id, redirect_uri, scope, state（防 CSRF） 用户 ── 在授权服务器输入密码 ──► 授权服务器 授权服务器 ── 重定向回 redirect_uri ──► 客户端（浏览器） URL 参数携带：code（短期授权码，用一次即失效） 客户端后端 ── POST /token ───────────► 授权服务器 携带：code + client_secret（只在后端，用户不可见） 授权服务器 ── 返回 ──────────────────► 客户端后端 返回：access_token + refresh_token（不过浏览器）

关键设计：code 暴露在 URL 中（可能被日志记录），但它只能用一次且有效期极短（通常 10 分钟）；真正的 access_token 通过后端 server-to-server 调用获取，不经过浏览器地址栏。

② 隐式模式（Implicit）已废弃，不要使用

为没有后端的纯前端应用设计——浏览器直接从授权服务器拿 access_token，跳过 code 换 token 这步。

为什么废弃？access_token 直接出现在 URL fragment（#access_token=xxx），容易被浏览器历史、Referer 头、第三方脚本读取。OAuth 2.1 草案已将其移除。

现有代码中如果还在用 Implicit Flow，必须迁移到授权码 + PKCE 模式。这是安全要求，不是建议。

③ 密码模式（Resource Owner Password Credentials）仅内部信任应用

用户直接把用户名密码给客户端，客户端拿着去换 token。

HTTP 请求示例
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=password
&username=alice
&password=secret
&client_id=my-app
&scope=read write

为什么受限？它要求用户信任客户端到直接给出密码的程度——这完全绕过了 OAuth2"密码不离开授权服务器"的初衷。只在第一方应用（比如公司内部自己的 App 对接自己的认证服务器）才考虑使用，且 OAuth 2.1 已将其废弃。

④ 客户端凭证模式（Client Credentials）机器间通信

没有用户参与，客户端（服务）用自己的 client_id + client_secret 直接换 token。用于微服务间调用、定时任务、后台 Job 等场景。

HTTP 请求示例
POST /oauth/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&scope=service:read
四种模式选择口诀：有后端 Web → 授权码；纯前端/移动端 → 授权码 + PKCE；机器间 → 客户端凭证；密码模式和隐式模式 → 别用了。

4. 四种模式对比

模式有用户参与密码暴露给客户端推荐使用适用场景
授权码是否是有后端的 Web 应用
授权码 + PKCE是否是移动端 / SPA 前端
隐式是否已废弃—
密码模式是是仅内部信任第一方应用过渡期
客户端凭证否—是服务间调用

5. PKCE：移动端和 SPA 的必选扩展

问题背景：移动 App 无法安全保存 client_secret（反编译即可获取），所以不能像 Web 后端那样用 client_secret 来换 token。PKCE（Proof Key for Code Exchange，读作"pixie"）解决这个问题。

PKCE 的工作原理

客户端 随机生成 code_verifier（高熵随机字符串，43~128字符） 客户端 计算 code_challenge = BASE64URL(SHA256(code_verifier)) ── 授权请求 ──► 授权服务器 携带：code_challenge + code_challenge_method=S256 （不需要 client_secret） ◄── 返回 code ── 授权服务器 授权服务器记住这次请求对应的 code_challenge ── 换 token 请求 ──► 授权服务器 携带：code + code_verifier（原始随机字符串，明文） 授权服务器 验证：SHA256(code_verifier) == 存储的 code_challenge ? ◄── 返回 access_token ──

安全性来源：攻击者即使截获了授权码 code，由于不知道 code_verifier，也无法用这个 code 换到 token。整个过程不需要 client_secret，也不会有安全漏洞。

OAuth 2.1 要求所有公开客户端（移动端、SPA）必须使用 PKCE。即使是有后端的 Web 应用，也推荐加上 PKCE 作为额外防护层。

6. Spring Security OAuth2 集成要点

Spring 生态下，OAuth2 相关依赖分两个方向：

角色依赖说明
OAuth2 客户端spring-boot-starter-oauth2-client你的应用要对接 GitHub / Google 等第三方登录
OAuth2 资源服务器spring-boot-starter-oauth2-resource-server你的 API 需要验证 access_token（JWT 或 opaque token）
授权服务器spring-authorization-server你自己要搭建 OAuth2 / OIDC 授权服务器

配置资源服务器（验证 JWT）

application.yml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          # 授权服务器会在这个地址暴露公钥（JWKS 端点）
          jwk-set-uri: https://auth.example.com/.well-known/jwks.json
SecurityConfig.java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                // 使用 scope 做权限控制，scope 来自 JWT 的 scp/scope claim
                .requestMatchers("/api/admin/**").hasAuthority("SCOPE_admin")
                .anyRequest().authenticated())
            // 声明这是资源服务器，使用 JWT 验证
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(Customizer.withDefaults()));
        return http.build();
    }
}

配置 OAuth2 客户端（对接 GitHub 登录）

application.yml
spring:
  security:
    oauth2:
      client:
        registration:
          github:  # Spring 内置了 GitHub 的 provider 配置
            client-id: ${GITHUB_CLIENT_ID}
            client-secret: ${GITHUB_CLIENT_SECRET}
            scope: read:user,user:email

只需这段配置，Spring Security 会自动处理授权码流程，包括重定向、code 换 token、获取用户信息等。访问 /oauth2/authorization/github 即可触发 GitHub 登录。

7. 工程实践中的坑

  • state 参数不验证：授权码请求必须携带随机 state，回调时验证它是否与发出时一致，否则容易受到 CSRF 攻击
  • redirect_uri 校验宽松：授权服务器必须严格匹配 redirect_uri（包括协议、域名、路径），不能允许通配符，否则攻击者可构造恶意重定向劫持 code
  • token 存在 URL 里：不要在 URL 参数中传递 access_token，应用 Authorization 请求头。URL 会被记录在服务器日志和浏览器历史中
  • scope 粒度太粗：申请权限时遵循最小权限原则，只申请本次操作需要的 scope，不要申请 * 或 admin
  • client_secret 写在前端：SPA 和移动端无法安全保存 client_secret，必须用 PKCE 替代，或将 token 换取逻辑放在后端 BFF（Backend for Frontend）层
  • 混淆 OAuth2 和 OIDC：OAuth2 给你 access_token（权限），OIDC 额外给你 id_token（身份）。"第三方登录"是 OIDC，别只实现 OAuth2 然后用 access_token 去查用户信息当登录凭证

8. OAuth2 与 JWT 是什么关系？

这是最常见的混淆点。

OAuth2JWT
本质授权框架（流程协议）token 格式（数据结构）
解决如何安全颁发和传递权限如何用自包含方式表达声明
关系OAuth2 的 access_token 可以用 JWT 格式，也可以用不透明字符串（opaque token）。两者正交，互相不依赖。
面试高频考点：OAuth2 四种模式哪种最安全，为什么？→ 授权码模式，因为 access_token 只在后端流转，不经过浏览器；code 一次性且短期；结合 PKCE 后即使 code 被截获也无法换 token。

与其他知识点的联系

  • JWT：OAuth2 只规定了授权服务器如何颁发、客户端如何交换 token，但没规定 token 长什么样——access_token 可以是一段不透明字符串（opaque token，需要资源服务器回调授权服务器查询），也可以直接用 JWT 编码，把用户身份、scope 等信息自包含在 token 里。资源服务器用 JWKS 端点拿到的公钥验签，正是上一篇 JWT 讲的 RS256 非对称签名的典型应用场景。
  • 单点登录 SSO：OIDC 在 OAuth2 之上加的 id_token，本质上就是实现 SSO 的核心机制——多个子系统都信任同一个授权服务器颁发的身份令牌，用户在一处登录后，其它系统凭 id_token 就能确认身份，不需要重复输入密码。这也是"用 GitHub 登录""企业内部统一门户"这类场景背后的协议基础。
  • API Gateway 鉴权：在微服务架构里，客户端凭证模式（Client Credentials）常用于服务间调用鉴权，API Gateway 可以在入口统一校验 access_token（无论是 JWT 还是 opaque token），验证通过后再把请求转发给具体业务服务，业务服务本身就不用重复实现鉴权逻辑——这是"关注点分离"思想在网关层的具体落地。
  • Spring Security 的 Filter 链与 AOP：oauth2ResourceServer().jwt() 和自定义 JwtAuthFilter（见上一篇）解决的是同一个问题，只是实现方式不同：前者是 Spring Security 提供的标准化组件，通过配置声明式接入，本质上仍然是在 SecurityFilterChain 上插入一个 Filter，校验通过后把认证信息写入 SecurityContext，后续 Controller 通过 @PreAuthorize 等注解做的其实是一种 AOP 式的权限拦截。
  • CSRF 与 Web 安全基础：授权码模式中的 state 参数本质上是 CSRF Token 的一种应用——防止攻击者构造一个指向自己账号的授权回调，诱导用户在不知情的情况下把自己的账号绑定到攻击者的 Session 上（OAuth2 CSRF 攻击）。这和表单提交场景下防 CSRF 的思路完全一致，都是靠"一次性、不可预测的随机值"来确认请求确实来自预期的发起方。
  • 最小权限原则与设计模式中的接口隔离：OAuth2 的 scope 机制是"最小权限原则"（Principle of Least Privilege）在授权协议层面的体现，和面向对象设计中的接口隔离原则（ISP）思路相通——都是通过拆分细粒度的能力单元，让调用方只能拿到、只需要申请自己真正要用到的那一部分权限，而不是笼统地"全有或全无"。
