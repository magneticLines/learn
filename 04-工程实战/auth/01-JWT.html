JWT（无状态令牌）

理解 JWT 的结构与工作机制，掌握它在分布式系统中取代 Session 的原因，以及生产环境中不得不防的那些坑。

1. 问题从 Session 说起

传统 Session 认证的流程是：浏览器登录 → 服务器在内存中创建 Session，生成 sessionId 写入 Cookie → 每次请求携带 Cookie，服务器查 Session 表验证身份。

单体应用里这毫无问题。麻烦出在水平扩展时：用户第一次请求打到 A 节点，Session 在 A 的内存里；下次负载均衡打到 B，B 不认识这个 Session，用户被踢出。

传统解法是"Session 共享"——把 Session 集中存到 Redis。这能用，但带来了强依赖：Redis 挂了，整个认证瘫痪；微服务之间跨服务调用也要携带 Cookie 或手动传 sessionId，很别扭。

结论：单体 + 少量流量，Session 完全够用，不要为了用 JWT 而用 JWT。分布式 / 微服务 / 需要跨域 / 移动端 API 调用，JWT 才是更自然的选择。

2. JWT 结构：三段 Base64 拼出来的字符串

JWT（JSON Web Token）本质上是一个自包含的令牌——它把用户信息直接编码进 token，服务器无需查库就能验证身份。格式是：

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 ← Header（算法 + 类型） . eyJ1c2VySWQiOiIxMjMiLCJyb2xlIjoiQURNSU4iLCJleHAiOjE3MDAwMDAwMDB9 ← Payload（声明） . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c ← Signature（防篡改）

Header

JSON 结构，指定签名算法（如 HS256、RS256）和 token 类型，Base64Url 编码后作为第一段。

Payload（声明 Claims）

存放你想携带的业务数据。分三类：

  • Registered Claims（注册声明）：JWT 规范预定义，如 iss（签发方）、sub（主题）、exp（过期时间）、iat（签发时间）
  • Public Claims：自定义但需要在 IANA 注册，避免冲突
  • Private Claims：双方约定的私有字段，如 userId、role
Payload 没有加密！Base64Url 只是编码，任何人都能解码看到内容。不要把密码、银行卡号等敏感数据放进去。如需加密，使用 JWE（JSON Web Encryption）。

Signature（签名）

签名保证 token 不被篡改。生成方式：

伪代码
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secretKey   // 服务器保管，绝不对外暴露
)

服务器收到 token 后，用同一密钥重新计算签名并比对。攻击者改了 Payload，签名就对不上，验证失败。

3. 对称 vs 非对称签名算法

算法类型适用场景注意
HS256 / HS512对称（共享密钥）单一服务，只有自己签发和验证密钥泄漏则全线崩溃
RS256 / RS512非对称（RSA）多服务：认证服务私钥签发，各微服务公钥验证公钥可以安全分发
ES256非对称（ECDSA）同 RS256，但密钥更短、性能更好密钥生成稍复杂
微服务推荐 RS256：认证中心持有私钥，其他服务只需公钥即可验签，公钥可以放到配置中心或通过 JWKS 端点动态获取，私钥不扩散。

4. 用 JJWT 生成与验证（代码实战）

JJWT 是 Java 生态最主流的 JWT 库。引入依赖：

pom.xml
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.12.6</version>
</dependency>
<!-- 运行时还需要 jjwt-impl 和 jjwt-jackson -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>

生成 Token

JwtUtil.java
public class JwtUtil {

    // 密钥要足够长（≥ 256 bit）且从配置中读取，绝不硬编码在代码里
    private static final SecretKey SECRET = Keys.hmacShaKeyFor(
        Decoders.BASE64.decode(System.getenv("JWT_SECRET"))
    );

    public static String generateToken(Long userId, String role) {
        return Jwts.builder()
            // subject 一般放用户唯一标识，不放用户名（怕泄露）
            .subject(String.valueOf(userId))
            .claim("role", role)
            .issuedAt(new Date())
            // access token 通常 15~30 分钟；refresh token 7~30 天
            .expiration(Date.from(Instant.now().plusSeconds(1800)))
            .signWith(SECRET)
            .compact();
    }
}

验证并解析 Token

JwtUtil.java（续）
public static Claims parseToken(String token) {
    try {
        return Jwts.parser()
            .verifyWith(SECRET)
            .build()
            // 一步到位：验签 + 检查 exp + 返回 Claims
            .parseSignedClaims(token)
            .getPayload();
    } catch (ExpiredJwtException e) {
        // token 已过期，前端应用 refresh token 换新
        throw new TokenExpiredException();
    } catch (JwtException e) {
        // 签名不匹配 or 格式错误 → 可能是伪造
        throw new InvalidTokenException();
    }
}

在 Spring Filter 中拦截请求

JwtAuthFilter.java（关键逻辑）
@Override
protected void doFilterInternal(HttpServletRequest req, ...) {
    String header = req.getHeader("Authorization");
    // 标准做法：Authorization: Bearer <token>
    if (header == null || !header.startsWith("Bearer ")) {
        filterChain.doFilter(req, resp);
        return;
    }
    String token = header.substring(7);
    Claims claims = JwtUtil.parseToken(token);
    // 把解析出的用户信息注入 SecurityContext，后续 Controller 可直接用
    UsernamePasswordAuthenticationToken auth = new UsernamePasswordAuthenticationToken(
        claims.getSubject(), null, buildAuthorities(claims)
    );
    SecurityContextHolder.getContext().setAuthentication(auth);
    filterChain.doFilter(req, resp);
}

5. 双 Token 机制：Access Token + Refresh Token

JWT 的核心矛盾是：过期时间越长越方便，但泄漏的危害窗口也越大。行业解法是双 Token：

Access TokenRefresh Token
作用携带在每个请求里，证明身份用来换新的 Access Token
有效期短：15~30 分钟长：7~30 天
存储位置内存（前端变量）HttpOnly Cookie（防 XSS 读取）
发送方式Authorization: Bearer 头通过专用 /auth/refresh 接口
服务端存储无需存储（无状态）建议存 Redis，便于主动吊销

刷新流程：

前端发现 401（Access Token 过期） → 静默发起 POST /auth/refresh（携带 Refresh Token Cookie） → 服务端验证 Refresh Token（查 Redis 确认未被吊销） → 返回新 Access Token（+可选滚动刷新 Refresh Token） → 前端更新内存中的 Access Token，重放原请求

6. JWT 的核心缺陷：无法即时吊销

这是最多人踩的坑。JWT 是无状态的，服务器不记录已发出的 token。一旦发出，在 exp 到期前永远有效——即使：

  • 用户已经退出登录
  • 用户被管理员封号
  • token 已经泄漏，你想立即让它失效

常见的应对方案（按代价排序）：

方案原理代价
缩短过期时间Access Token 15 分钟，泄漏窗口小用户体验差（频繁刷新）
Token 黑名单注销时把 token 的 jti（唯一 ID）存入 Redis，验证时查一下每次请求多一次 Redis 查询，引入了"有状态"
版本号机制用户表加 tokenVersion 字段，每次登出 +1；token 里带版本号，验证时对比需要查库，但只存一行数据，开销可控
Refresh Token 吊销Access Token 短期，Refresh Token 存 Redis 可主动删除Access Token 窗口期内仍有效
没有银弹。真正做到"立即吊销"就必须引入服务端状态（Redis 查询），JWT 的无状态优势就部分打折。根据业务安全等级选合适方案，而不是追求纯粹的无状态。

7. 生产环境常见坑清单

  • 密钥硬编码：不要把 secretKey = "my-secret" 写死在代码里，应通过环境变量或 Vault 注入
  • 密钥长度不够：HS256 要求密钥 ≥ 256 bit（32字节），太短 JJWT 会抛异常
  • 时钟不同步：多节点部署时各机器时钟偏差可能导致 exp 校验异常，建议用 NTP 同步 + 允许 60 秒容差（clockSkewSeconds）
  • token 放 localStorage：容易被 XSS 脚本读取。Access Token 放内存，Refresh Token 放 HttpOnly Cookie 是更安全的做法
  • 敏感信息放 Payload：Payload 只是 Base64 编码，不是加密，任何人解码可见
  • alg: none 攻击：验证时不要信任 token 自己声明的算法，应在服务端强制指定算法
  • Refresh Token 不轮换：如果 Refresh Token 被盗，攻击者可以一直刷新。建议每次刷新都下发新的 Refresh Token 并让旧的失效（Refresh Token Rotation）

8. Session vs JWT：怎么选

维度SessionJWT
服务端状态有状态（需要存储）无状态（token 自包含）
水平扩展需要 Session 共享（Redis）天然支持，节点无感
即时吊销直接删 Session，立即生效需要额外机制（黑名单等）
跨域 / 跨端依赖 Cookie，跨域麻烦Authorization 头，无跨域限制
服务间调用需要透传 sessionId直接透传 token，天然适合
适用场景单体 Web 应用分布式、微服务、移动端 API
面试高频考点：JWT 如何实现登出？→ 这正是考察你是否理解 JWT 无状态的本质。回答：Access Token 短期（15min）自然失效，Refresh Token 存 Redis 可以主动删除，配合 Token 黑名单或版本号机制可实现即时吊销。

与其他知识点的联系

  • OAuth2：JWT 只是一种 token 的数据格式，OAuth2 是颁发和传递 token 的流程协议。OAuth2 授权码模式最终换到的 access_token 完全可以用 JWT 格式编码，Spring Security 的 oauth2ResourceServer().jwt() 本质上就是"用 JWT 校验逻辑去验证 OAuth2 颁发的 token"，二者是组合关系而非替代关系，学完本文再看下一篇 OAuth2 会更容易分清"协议"和"格式"这两个层面。
  • Spring Security 的 Filter 链：本文的 JwtAuthFilter 只是插入到 SecurityFilterChain 中的一环，它和 UsernamePasswordAuthenticationFilter、ExceptionTranslationFilter 等按顺序组成职责链模式（Chain of Responsibility）。理解这一点后再看 Spring Security 的其他自定义 Filter（如限流、IP 黑名单）会发现套路完全一致：都是在链上找准插入位置，做完自己的事再 doFilter 放行。
  • 密码学基础（哈希 vs 加密 vs 编码）：JWT 的三段结构正好对应三种不同的密码学操作——Header/Payload 只是 Base64编码（可逆、无密钥、任何人可读），Signature 是哈希 + 密钥的 HMAC 运算（不可逆、用于验证完整性），如果要对 Payload 保密则要用 JWE 做真正的加密。混淆"编码"和"加密"正是很多人以为 Payload 是安全的根源。
  • 分布式系统 / CAP 理论：JWT 用"无状态"换来水平扩展能力，本质上是在 CAP 权衡里放弃了服务端的强一致状态（谁的 token 有效、谁已登出），用密码学验证代替了数据库查询。第 6 节讲的"无法即时吊销"正是这个权衡的代价——想要拿回强一致性（即时吊销），就必须引入 Redis 查询，退回到部分有状态。
  • 缓存与 Redis 的典型应用：Token 黑名单、Refresh Token 存储、版本号机制，都是 Redis「设置 TTL 自动过期 + 快速查键」特性的典型应用场景，和缓存穿透/缓存雪崩等 Redis 相关知识点是同一套基础设施能力的不同用法。
  • 微服务鉴权与 API Gateway：RS256 的"私钥签发、公钥验签"模式天然适合微服务架构——认证中心统一签发 token，各业务服务甚至 API Gateway 层只需公钥即可完成鉴权，不用每个服务都对接认证中心做同步调用，这也是"关注点分离"在鉴权场景下的具体体现。
