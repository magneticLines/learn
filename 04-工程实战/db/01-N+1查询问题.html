N+1 查询问题

ORM 让你像访问对象属性一样拿关联数据——但这份"廉价假象"会泄漏：一个无害的 for 循环，可能在背后制造上百次数据库往返。N+1 是线上事故最高发的 ORM 陷阱，也是面试高频题。它正是「抽象泄漏」最典型的一个案例。

一、现象：1 条 SQL 怎么变成了 11 条

假设有「订单 Order」和「订单明细 OrderItem」，一对多关系。你要查 10 个订单，顺便显示每个订单的明细：

反例 · 看起来人畜无害
List<Order> orders = orderRepository.findAll();      // ① 查出 10 个订单
for (Order order : orders) {
    order.getItems().size();                  // ② 访问明细 —— 陷阱在这
}

你以为只发了 1 条 SQL。打开日志一看：

SELECT * FROM orders; -- ① 这是那个 "1" SELECT * FROM order_items WHERE order_id = 1; -- ② 这是 "N" SELECT * FROM order_items WHERE order_id = 2; SELECT * FROM order_items WHERE order_id = 3; ... -- 一共 10 条

总共 1 + N 条 SQL（这里 1 + 10 = 11），这就是名字的由来。

真正的杀手不是 SQL 慢，是"条数"。每条 SQL 都是一次数据库网络往返(round-trip)。首页列 100 个订单 = 101 次往返；若再嵌套（订单→明细→商品），会膨胀到 N×M 级别，足以拖垮整个数据库。

二、为什么会发生：一次教科书级的「抽象泄漏」

order.getItems() 在业务代码层(L7)看起来就是一次普通的对象属性访问——廉价、内存级、瞬间完成，和 order.getId() 没区别。

但 ORM(L6) 在背后撒了个谎：items 字段是懒加载(Lazy)的。你第一次访问它的瞬间，ORM 才偷偷去 数据库(L4) 发一条 SQL + 一次网络往返。

泄漏点就在这："对象导航"的廉价假象，漏掉了"每次访问 = 一次远程查询"的真相。你在 L7 写了个无害的 for 循环，实际在 L4 制造了 N 次网络风暴。这正是 N+1 隐蔽的原因——代码读起来完全正常。

三、怎么定位（先会抓，再会治）

光看代码看不出 N+1，必须看真实 SQL：

  • 打开 SQL 日志：JPA 用 spring.jpa.show-sql=true（粗糙），或上 p6spy 看真实条数和耗时；MyBatis 把 Mapper 日志设为 DEBUG。
  • 看条数是否随数据量线性增长：一个接口的 SQL 条数若随记录数变多而变多，基本就是 N+1。
  • Hibernate 可用 Statistics 或 hibernate.session.events.log 统计查询次数。

四、正例：四种解法

解法 A · JOIN FETCH（JPA 最常用）

一条 SQL 用 JOIN 把关联一次性捞回来。

JPQL
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();

✅ 把 1+N 压成 1。⚠️ 但有三个坑（见第五节），尤其不能用于分页。

解法 B · @EntityGraph（JPA 更声明式）

不想写 JPQL，用注解声明"这次查询要带哪些关联"，适合复用 Repository 方法：

注解
@EntityGraph(attributePaths = {"items"})
List<Order> findAll();

解法 C · @BatchSize（JPA 优雅折中，推荐做默认）

不消灭懒加载，而是把"N 条单查"合并成 IN 查询：

实体配置
@OneToMany(mappedBy = "order")
@BatchSize(size = 100)                  // 关键
private List<OrderItem> items;
// 或全局：hibernate.default_batch_fetch_size=100
SELECT * FROM orders; SELECT * FROM order_items WHERE order_id IN (1,2,...,100); -- N 条变 1~2 条

✅ 从 1+N 降到 1 + ⌈N/size⌉，且天然兼容分页、不产生笛卡尔积。👉 强烈建议设为全局默认，兜底掉大量你没注意到的 N+1。

解法 D · 手动 IN 查询 + 内存组装（MyBatis 首选，最可控）

显式控制
// 1. 查订单
List<Order> orders = orderMapper.findAll();
List<Long> ids = orders.stream().map(Order::getId).toList();

// 2. 一条 IN 查询捞出所有明细
List<OrderItem> items = itemMapper.findByOrderIds(ids);

// 3. 内存里 group by 组装（O(n)，纯内存，极快）
Map<Long, List<OrderItem>> map = items.stream()
        .collect(groupingBy(OrderItem::getOrderId));
orders.forEach(o -> o.setItems(map.getOrDefault(o.getId(), List.of())));

✅ 永远只有 2 条 SQL，可控、可分页、无笛卡尔积。

MyBatis 最容易踩 N+1 的写法：resultMap 里用 <association> / <collection> 的 select= 嵌套查询——它本质就是 N+1。要改成嵌套结果（单条 JOIN 映射一个 resultMap），或用上面的手动 IN。

五、JOIN FETCH 的三个坑（治好 N+1 又踩这里）

  1. 笛卡尔积重复：一对多 JOIN 后，一个订单有 3 个明细，订单字段就重复 3 行。JPQL 要加 DISTINCT（它在内存去重对象，还你正确的对象列表）。
  2. 分页直接失效（最危险）：JOIN FETCH + 分页时，Hibernate 没法在 SQL 层分页，会偷偷把全表拉进内存再分页，并打印一条易被忽略的警告：
    HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!
    数据量一大直接 OOM。要分页又要带集合，请改用 @BatchSize。
  3. 不能同时 fetch 两个集合：JOIN FETCH o.items JOIN FETCH o.payments 会抛 MultipleBagFetchException，或产生 items×payments 的笛卡尔积爆炸。多个集合 → 用 @BatchSize。
小结（回到抽象塔）：① N+1 的本质是 L6(ORM) 用"对象导航"伪装了 L4(数据库网络往返)；② 所有解法的核心思想都一样——把"N 次往返"合并成"1~2 次往返"，要么 JOIN 一次拉全，要么 IN 一批拉回内存组装；③ JPA 项目全局开 default_batch_fetch_size 兜底；④ MyBatis 警惕 <collection select> 嵌套查询，优先手动 IN；⑤ 上线前看 SQL 日志条数是发现 N+1 唯一可靠的手段。
