框架机制 · Spring 核心思想

IoC / DI 思想（控制反转）

从"对象自己造依赖"到"容器送货上门"——理解 Spring 最根本的设计哲学，以及它在真实项目中解决的工程问题。

一句话讲清楚

IoC（Inversion of Control） 是一种设计原则：对象不再自己创建依赖，而是把"创建和组装依赖"的控制权交给外部容器。 DI（Dependency Injection，依赖注入）是 IoC 最主流的实现方式——容器在运行时把依赖"注入"进来，对象只管声明自己需要什么。

类比：餐厅点菜（IoC）vs 自己下厨（传统方式）。你只需告诉服务员"我要一碗牛肉面"，厨师、食材、锅具全部由餐厅统一调配——你不再关心它们从哪来、怎么组合的。

反例：对象自己 new 依赖

下面是一个典型的订单服务，强耦合写法在小项目里看起来没问题，但一旦规模变大就会爆炸。

OrderService.java（反例 — 强耦合）
public class OrderService {

    // 直接 new 出来，OrderService 和具体实现死死绑定
    private final OrderRepository orderRepo  = new MysqlOrderRepository();
    private final PaymentService  payService = new AlipayPaymentService();
    private final NotifyService   notify     = new SmsNotifyService();

    public void createOrder(OrderDTO dto) {
        orderRepo.save(dto);
        payService.pay(dto);
        notify.send(dto);
    }
}

这段代码有哪些工程问题？

  • 无法单元测试：想 Mock MysqlOrderRepository？不可能，它被 new 死了。
  • 切换实现需要改源码：从支付宝换微信支付，要找到所有 new AlipayPaymentService() 一个个改。
  • 对象图管理混乱：如果 AlipayPaymentService 内部还 new 了 HttpClient，线程池配置散落四处，无法全局复用/调优。
  • 循环依赖风险：A new B，B new A，直接栈溢出。

正例：Spring IoC 容器接管

用 Spring 重写后，OrderService 只声明"我需要这些依赖"，容器负责创建和注入。

OrderService.java（正例 — 构造器注入）
@Service
public class OrderService {

    private final OrderRepository orderRepo;
    private final PaymentService  payService;
    private final NotifyService   notify;

    // Spring 推荐：构造器注入，依赖不可变，天然线程安全
    public OrderService(OrderRepository orderRepo,
                         PaymentService  payService,
                         NotifyService   notify) {
        this.orderRepo  = orderRepo;
        this.payService = payService;
        this.notify     = notify;
    }

    public void createOrder(OrderDTO dto) {
        orderRepo.save(dto);
        payService.pay(dto);
        notify.send(dto);
    }
}
PaymentService.java（接口抽象 — 对换实现零修改）
// 依赖接口，不依赖具体实现
public interface PaymentService {
    void pay(OrderDTO dto);
}

@Service
@ConditionalOnProperty(name = "payment.channel", havingValue = "wechat")
public class WechatPaymentService implements PaymentService {
    @Override
    public void pay(OrderDTO dto) { /* 微信支付逻辑 */ }
}

@Service
@ConditionalOnProperty(name = "payment.channel", havingValue = "alipay")
public class AlipayPaymentService implements PaymentService {
    @Override
    public void pay(OrderDTO dto) { /* 支付宝逻辑 */ }
}

改配置文件中的 payment.channel 就能切换支付渠道，OrderService 代码一行不改。这就是 IoC 带来的"开闭原则"落地。

容器做了什么？（原理走一遍）

Spring 的 ApplicationContext 启动时大致经历三个阶段：

① 扫描（Scan）
  扫描 @Component / @Service / @Repository 等注解
  解析 @Configuration + @Bean 工厂方法
  → 生成 BeanDefinition（蓝图，描述如何创建这个 Bean）

② 实例化 + 依赖注入（Instantiate & Inject）
  按拓扑顺序创建 Bean 实例
  解析构造器参数 / @Autowired 字段 → 从容器取出对应 Bean 注入
  → 生成完整的 单例 Bean 对象图

③ 初始化回调（Initialize）
  调用 @PostConstruct、InitializingBean.afterPropertiesSet()
  执行 BeanPostProcessor（AOP 代理在这里插入）
  → Bean 进入就绪状态，放入单例池
单例池（singletonObjects） 是一个 ConcurrentHashMap<String, Object>，key 是 Bean 名称，value 是实例。 容器启动后这张 Map 就定了，后续每次 getBean() 都直接从这里取，不再创建新对象。

三种注入方式对比

方式 写法 优点 缺点 / 坑
构造器注入
（官方推荐）		 @Service + 构造函数 依赖不可变；可写 final；循环依赖早发现；易测试 依赖多时构造器参数列表长（可用 Lombok @RequiredArgsConstructor）
Setter 注入 @Autowired 在 setter 上 依赖可选；可在运行时替换（测试场景） 对象在 setter 调用前处于不完整状态；依赖可变，线程安全需自己保证
字段注入 @Autowired 直接标在字段 写起来最省事 无法写 final；单元测试必须启动 Spring 容器或反射注入；隐藏依赖数量膨胀的问题
构造器注入 + Lombok（生产首选）
@Service
@RequiredArgsConstructor  // Lombok 自动生成含所有 final 字段的构造器
public class OrderService {

    private final OrderRepository orderRepo;
    private final PaymentService  payService;
    private final NotifyService   notify;

    // 业务代码干净，不再有 @Autowired 污染
    public void createOrder(OrderDTO dto) { ... }
}

IoC 如何让测试变得简单

依赖通过构造器注入后，测试时直接用 Mock 对象替换，完全不需要启动 Spring 容器：

OrderServiceTest.java（纯单元测试，毫秒级）
class OrderServiceTest {

    @Test
    void createOrder_shouldSaveAndPay() {
        // 直接 new Mock，不启动 Spring 容器
        OrderRepository mockRepo   = mock(OrderRepository.class);
        PaymentService  mockPay    = mock(PaymentService.class);
        NotifyService   mockNotify = mock(NotifyService.class);

        OrderService sut = new OrderService(mockRepo, mockPay, mockNotify);
        OrderDTO dto = new OrderDTO("item-001", 99);

        sut.createOrder(dto);

        verify(mockRepo).save(dto);
        verify(mockPay).pay(dto);
    }
}

如果用字段注入（@Autowired 直接标字段），这段测试就必须改用 @SpringBootTest 或 ReflectionTestUtils.setField()，启动时间从毫秒变成秒级甚至分钟级。

生产环境常见坑

坑 1：循环依赖（Spring Boot 2.6+ 默认禁止）

循环依赖示意
@Service class A {
    @Autowired B b;   // A 依赖 B
}
@Service class B {
    @Autowired A a;   // B 又依赖 A —— 死锁！
}

旧版 Spring 靠"三级缓存"勉强解决字段注入的循环依赖，但这本质上是设计问题。正确做法：重新拆分职责，或用 @Lazy 延迟加载（临时方案）。不要靠框架掩盖设计缺陷。

坑 2：在非 Spring 管理的对象里注入

手动 new MyService() 创建的对象不在容器里，其内部的 @Autowired 字段永远是 null。这是新手最常见的 NPE 来源。常见场景：工具类静态方法调用、手动 new 的 Listener、反序列化回调。

坑 3：原型 Bean 注入到单例 Bean

Scope 陷阱
@Service                        // 单例
public class ReportService {

    @Autowired
    private ReportContext ctx;   // @Scope("prototype") — 本意是每次用新对象

    // 但容器只在 ReportService 初始化时注入一次 ctx！
    // 之后每次调用 generate() 拿到的都是同一个 ctx 实例 → 状态污染
    public void generate() { ctx.doWork(); }
}

解决方案：注入 ObjectProvider<ReportContext> 或 ApplicationContext，每次调用 getBean() 手动取新实例。

坑 4：@Autowired 注入集合

当容器里有多个同类型 Bean 时，可以直接注入 List<PaymentService>——Spring 会把所有实现全部收集进来。这是策略模式的优雅实现方式，但要注意顺序（用 @Order 控制）。

注入多实现列表（策略模式）
@Service
@RequiredArgsConstructor
public class PaymentRouter {

    // Spring 自动收集所有 PaymentService 实现注入
    private final List<PaymentService> handlers;

    public void route(OrderDTO dto) {
        handlers.stream()
                .filter(h -> h.supports(dto))
                .findFirst()
                .orElseThrow()
                .pay(dto);
    }
}

概念辨析：IoC / DI / 容器

概念层次一句话
IoC 设计原则 把"谁来控制依赖的创建"这件事反转——从对象自己控制，变成外部控制
DI 实现模式 IoC 的具体落地方式：由容器把依赖"注入"进对象（而不是对象去"拉取"）
IoC 容器 框架实现 Spring 的 ApplicationContext，负责管理 Bean 的生命周期、依赖关系、作用域
面试高频："IoC 和 DI 的区别？"——IoC 是思想，DI 是 IoC 的一种实现手段。除了 DI，Service Locator（服务定位器）也是 IoC 的实现，但 Spring 明确推荐 DI 而非 Service Locator，因为后者让依赖关系隐式化，更难测试。

核心价值总结

  • 解耦：对象只依赖接口，不依赖具体实现，切换实现只改配置。
  • 可测试：依赖从外部注入，测试时可随意替换为 Mock，单元测试不需要启动容器。
  • 集中管理：对象的生命周期（创建、初始化、销毁）、作用域、配置全部由容器统一管理，不再散落各处。
  • 扩展点丰富：BeanPostProcessor、BeanFactoryPostProcessor 等扩展点让 AOP、事务、监控等横切能力可以无侵入地插入，这是下一节 AOP 的基础。
