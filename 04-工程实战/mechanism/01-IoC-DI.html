框架机制 · 01

IoC / DI 思想（控制反转）

理解 Spring 最核心的设计哲学：让框架掌控对象生命周期，而不是你亲手 new。

好莱坞原则：Don't call us, we'll call you

好莱坞原则来自演艺圈：演员不主动打电话给导演，而是等导演在合适的时机主动联系你。放到软件世界里，就是：高层框架不让你主动去拿依赖，而是在合适的时机把依赖推送给你。

这句话本质上就是 IoC（Inversion of Control，控制反转）的全部含义。控制权从调用方（你的业务代码）转交给了框架（Spring 容器）。

类比：传统方式好比你自己开车去超市买菜（你控制全程）；IoC 好比叫了外卖——你只需要告诉平台"我要一份宫保鸡丁"，平台帮你安排骑手、路线、时间，你等着就行。

IoC 是思想，DI 是实现

IoC 只是一个设计原则，本身并没有规定如何实现。最主流的实现方式叫 DI（Dependency Injection，依赖注入）：框架在创建对象时，自动把它所需要的依赖塞进去。

除了 DI，IoC 还有另一种实现叫 Service Locator（服务定位器），但它把查找依赖的逻辑暴露到了业务代码里，耦合度更高，现代框架已基本弃用。

没有 IoC 的世界（手动 new）

Java — 强耦合的反例
public class OrderService {
    // 直接 new：OrderService 知道 UserRepository 的具体实现
    // 想换实现？必须改这里的源码——这就是耦合
    private UserRepository userRepo = new MysqlUserRepository();
    private PaymentGateway payment  = new AlipayGateway();

    public void placeOrder(Order order) {
        // 业务逻辑...
    }
}

引入手动 DI

Java — 手动依赖注入（构造器）
public class OrderService {
    private final UserRepository userRepo;
    private final PaymentGateway  payment;

    // 依赖从外部传入，OrderService 不关心具体是哪个实现
    public OrderService(UserRepository userRepo, PaymentGateway payment) {
        this.userRepo = userRepo;
        this.payment  = payment;
    }
}

// 组装工作由"外部"完成，这个"外部"在 Spring 里就是容器
OrderService svc = new OrderService(
    new MysqlUserRepository(),
    new AlipayGateway()
);

手动 DI 虽然解耦了，但随着依赖链变深（A 依赖 B，B 依赖 C，C 依赖 D……），手动组装会变得极其繁琐。Spring 容器就是为了解决这个问题而生的。

三种注入方式的比较

Spring 支持三种方式把依赖注入进来，各有取舍：

1. 构造器注入（推荐）

Java — 构造器注入
@Service
public class OrderService {
    private final UserRepository userRepo;

    // Spring 4.3+ 单构造器可省略 @Autowired
    public OrderService(UserRepository userRepo) {
        this.userRepo = userRepo;
    }
}

2. Setter 注入

Java — Setter 注入
@Service
public class OrderService {
    private UserRepository userRepo;

    @Autowired
    public void setUserRepo(UserRepository userRepo) {
        this.userRepo = userRepo;
    }
}

3. Field 注入（不推荐）

Java — Field 注入（反例）
@Service
public class OrderService {
    // 看起来简洁，但隐藏了大量问题（见下表）
    @Autowired
    private UserRepository userRepo;
}
注入方式 不变性 循环依赖 可测试性 推荐度
构造器注入 final，不可变 启动时报错，问题暴露早 直接 new 即可测试 ★★★ 最推荐
Setter 注入 可变，可选依赖用 能绕过，但不安全 需手动调 setter ★★ 可选依赖场景
Field 注入 不能 final 掩盖循环依赖 必须启动容器才能测 ★ 不推荐

Spring @Autowired 背后发生了什么

Spring 容器启动时经历三个关键阶段：

  1. 扫描阶段：扫描 classpath，找到所有 @Component/@Service/@Repository 等注解的类，生成 BeanDefinition（可以理解为"造 Bean 的配方"）。
  2. 实例化阶段：按 BeanDefinition 反射创建对象实例（但还没注入）。
  3. 依赖注入阶段：通过 AutowiredAnnotationBeanPostProcessor 扫描字段/方法上的 @Autowired，从容器中按类型（或名称）查找匹配的 Bean，通过反射设置进去。
构造器注入在第 2 步就完成了注入——创建对象的同时就传入依赖，所以可以加 final，也不存在"对象已创建但依赖还没注入"的空窗期。这是构造器注入最本质的优势。

为什么构造器注入更好

除了上面的表格，还有几个深层原因值得记住：

  • 依赖关系清晰可见：构造器参数就是依赖清单，一眼就能看出这个类依赖了多少东西。如果构造器参数超过 4-5 个，这本身就是"该拆分类了"的警告信号。
  • 不可变性：final 字段确保注入后不会被意外替换，在并发场景下更安全。
  • 单元测试友好：不依赖 Spring 容器，直接 new OrderService(mockRepo) 就能跑测试。Field 注入必须用 @SpringBootTest 或 Mockito 的 @InjectMocks 才能跑起来。
  • 防止 NPE：Field 注入的 Bean 在构造函数里访问注入字段会得到 null（容器还没来得及注入），这是常见的坑。
面试高频陷阱：Spring 为什么能解决循环依赖？答：通过"三级缓存"在实例化完成后、注入完成前暴露一个早期引用。但这个机制只对 Setter/Field 注入有效，构造器注入因为在实例化时就需要依赖，所以无法通过三级缓存解决，Spring 会直接抛出 BeanCurrentlyInCreationException——这反而是好事，循环依赖本身就是设计问题。
