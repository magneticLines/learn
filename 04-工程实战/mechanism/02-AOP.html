AOP 面向切面（代理 / 织入）

把日志、事务、权限等横切逻辑从业务代码中剥离出去，让每个类只做一件事。

一、横切关注点：AOP 要解决的本质问题

想象一个电商系统，OrderService、PaymentService、UserService 里都需要： 记日志、开事务、做权限校验、统计耗时。这些逻辑和业务本身没有关系，却散落在几十个类里—— 这就是横切关注点（Cross-cutting Concern）。

OOP 擅长纵向抽象（继承、多态），但面对这种"横向穿透多个类"的逻辑无能为力。 AOP 补齐了这块短板：把横切逻辑集中写一次，运行时自动织入目标方法。

一句话定义：AOP（Aspect-Oriented Programming）通过动态代理在不修改原有类的前提下， 把额外的行为（切面）"织入"到指定方法的执行前后。

二、反例 → 问题 → 正例

反例：手动在每个方法里写日志

OrderService.java — 反例
public class OrderService {

    public Order createOrder(CreateOrderReq req) {
        // 业务代码前后全是与业务无关的横切逻辑
        log.info("createOrder start, req={}", req);
        long start = System.currentTimeMillis();
        if (!hasPermission("ORDER_CREATE")) {
            throw new AccessDeniedException("no permission");
        }

        // 真正的业务逻辑被埋在一堆横切代码里
        Order order = orderRepo.save(new Order(req));

        log.info("createOrder end, cost={}ms",
            System.currentTimeMillis() - start);
        return order;
    }
}

问题清单：

  • 业务逻辑被淹没，可读性差；
  • 日志格式改一次，要改几十个文件；
  • 测试 createOrder 时，必须 Mock 日志、权限等无关依赖；
  • 新增方法很容易忘记加这些模板代码，导致行为不一致。

正例：用 Spring AOP 切面

OrderService.java — 正例（只剩业务）
@Service
public class OrderService {

    // 干净！只有业务逻辑，没有任何横切代码
    public Order createOrder(CreateOrderReq req) {
        return orderRepo.save(new Order(req));
    }
}
LogAspect.java — 切面集中管理
@Aspect
@Component
public class LogAspect {

    // Pointcut：匹配 service 包下所有 public 方法
    @Around("execution(public * com.example.service..*(..))")
    public Object logAround(ProceedingJoinPoint pjp) throws Throwable {
        String method = pjp.getSignature().getName();
        log.info("[START] {}", method);
        long t0 = System.currentTimeMillis();
        try {
            Object result = pjp.proceed(); // 执行真实方法
            log.info("[END]   {} cost={}ms", method,
                System.currentTimeMillis() - t0);
            return result;
        } catch (Throwable e) {
            log.error("[ERROR] {} threw: {}", method, e.getMessage());
            throw e;
        }
    }
}

三、AOP 五大核心概念速查

术语直白解释例子
Aspect 切面类，横切逻辑的载体 LogAspect、TransactionAspect
Pointcut 切入点表达式，描述"在哪些方法上生效" execution(* com.example.service..*(..))
Advice 通知，切面在切入点执行的动作 @Before / @After / @Around
JoinPoint 连接点，程序执行中可被拦截的位置（Spring 仅支持方法级别） 方法调用、方法执行
Weaving 织入，把切面逻辑插入目标对象的过程 Spring 在运行时动态织入（非编译时）

四、底层原理：Spring AOP 是怎么做到的

Spring AOP 的核心是动态代理。容器启动时，对于需要织入切面的 Bean， Spring 不会直接返回原始对象，而是返回一个代理对象，代理对象在调用原方法前后执行切面逻辑。

Spring 使用两种动态代理策略：

策略适用条件原理
JDK 动态代理 目标类实现了接口 运行时生成接口的实现类，通过 InvocationHandler 拦截方法
CGLIB 代理 目标类没有接口（或强制配置） 运行时生成目标类的子类，覆盖父类方法注入切面逻辑
Spring Boot 2.x 起默认对所有 Bean 优先使用 CGLIB（spring.aop.proxy-target-class=true）， 避免"实现了接口才能被代理"的限制，但这也意味着目标类不能是 final。

代理链路示意：

调用方
  │
  ▼
代理对象（LogAspect.before → ...）
  │
  ▼
代理对象（TransactionAspect.before → ...）
  │
  ▼
真实对象（OrderService.createOrder）
  │
  ▼（返回值沿链路反向传播）
TransactionAspect.after → commit/rollback
  ▼
LogAspect.after → 记录耗时

五、五种 Advice 类型与执行顺序

Advice 类型示例
@Aspect @Component
public class DemoAspect {

    // 方法执行前（不能阻止目标方法执行，除非抛异常）
    @Before("execution(* com.example.service..*(..))")
    public void before(JoinPoint jp) { }

    // 方法正常返回后（可拿到返回值，但不能修改它）
    @AfterReturning(pointcut = "...", returning = "result")
    public void afterReturning(Object result) { }

    // 方法抛异常后
    @AfterThrowing(pointcut = "...", throwing = "ex")
    public void afterThrowing(Exception ex) { }

    // 无论正常还是异常，方法结束后都执行（类似 finally）
    @After("...")
    public void after() { }

    // 环绕（最强大，可以控制是否调用目标方法、修改返回值）
    @Around("...")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        // pjp.proceed() 执行真实方法，可以不调用（相当于拦截）
        return pjp.proceed();
    }
}

执行顺序（正常流）：@Around(前半) → @Before → 目标方法 → @Around(后半) → @AfterReturning → @After

执行顺序（异常流）：@Around(前半) → @Before → 目标方法抛异常 → @AfterThrowing → @After

六、生产中最常用的三个切面场景

场景 1：自定义注解 + 切面实现权限校验

先定义注解，再写切面识别注解——这是生产中最优雅的用法， 比在切点表达式里写包名更灵活，颗粒度可以精确到方法。

@RequirePermission 注解 + PermissionAspect
// 1. 定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePermission {
    String value(); // 权限码，如 "ORDER_CREATE"
}

// 2. 切面识别注解并校验
@Aspect @Component
public class PermissionAspect {

    @Before("@annotation(requirePermission)")
    public void checkPermission(RequirePermission requirePermission) {
        String code = requirePermission.value();
        if (!SecurityContext.hasPermission(code)) {
            throw new AccessDeniedException("缺少权限: " + code);
        }
    }
}

// 3. 业务方法只加一行注解
@RequirePermission("ORDER_CREATE")
public Order createOrder(CreateOrderReq req) { /* 只有业务 */ }

场景 2：接口幂等性校验（防重复提交）

IdempotentAspect — 基于 Redis 防重提交
@Around("@annotation(idempotent)")
public Object idempotentCheck(ProceedingJoinPoint pjp,
                                Idempotent idempotent) throws Throwable {
    // 从请求头提取幂等 Key（如 X-Request-Id）
    String key = "idempotent:" + extractRequestId();
    boolean absent = redis.setIfAbsent(key, "1", 60, SECONDS);
    if (!absent) {
        // Key 已存在，说明是重复请求
        throw new RepeatSubmitException("请勿重复提交");
    }
    return pjp.proceed();
}

场景 3：@Transactional 就是 AOP

Spring 的 @Transactional 本身就是一个切面—— Spring 用 TransactionInterceptor 拦截带注解的方法， 在方法前 begin，正常返回后 commit，异常后 rollback。 理解了 AOP 原理，事务失效的坑就迎刃而解了（见下文）。

七、生产环境常见坑

坑 1：同类内部调用，切面不生效

AOP 依赖代理对象，但同一个类的方法互相调用时，走的是 this 而不是代理对象，切面被绕过。

事务失效经典场景
@Service
public class OrderService {

    // 错误：methodA 内部直接调用 methodB，methodB 上的事务不生效
    public void methodA() {
        methodB(); // this.methodB()，走真实对象，绕过代理
    }

    @Transactional
    public void methodB() { /* 事务切面不会执行 */ }

    // 解法1：拆成两个 Service（推荐，职责更清晰）
    // 解法2：注入自身代理 @Autowired OrderService self; self.methodB();
    // 解法3：AopContext.currentProxy()（需开启 exposeProxy=true）
}
高频面试题：@Transactional 为何失效？根本原因就是"内部调用绕过代理"。 凡是 AOP 实现的功能（事务、缓存、权限），都有这个通病。

坑 2：切面作用于 final 类 / final 方法

CGLIB 通过继承生成子类，final 类无法被继承，final 方法无法被覆盖。 在这两种情况下，CGLIB 代理会静默失败（不报错，但切面不执行），极难排查。

  • Kotlin 的类默认是 final，与 Spring AOP 集成时必须加 open 或使用 allopen 插件。
  • 凡是不想被代理的类，加 final 是一种主动防御手段（但要清楚切面会失效）。

坑 3：切点表达式过于宽泛，性能拖垮

execution(* *.*(..)) 这种表达式会拦截容器内所有方法（包括 Spring 内部 Bean）， 造成代理链路过深、启动变慢、运行时每次调用都有额外开销。 生产中应尽量精确到包路径，或配合自定义注解做精准拦截。

坑 4：多切面顺序不确定

多个切面作用在同一方法时，默认执行顺序是不确定的。 用 @Order(n) 显式指定优先级，数值越小越先执行（越靠外层包裹）。 事务切面要包在日志切面外面，否则日志切面里的异常可能导致事务切面已经提交，再回滚就晚了。

指定切面顺序
@Order(1) // 最外层：安全切面先拦截
public class SecurityAspect {}

@Order(2) // 中间层：事务管理
public class TransactionAspect {}

@Order(3) // 最内层：日志记录真实执行情况
public class LogAspect {}

坑 5：@Around 忘记调用 proceed()

@Around 中如果不调用 pjp.proceed()，真实方法永远不会执行。 这是初学者最常犯的错，且没有任何编译期或启动期报错，只有运行时发现方法"没反应"。

八、织入时机：Spring AOP vs AspectJ

框架织入时机能力边界适用场景
Spring AOP 运行时（动态代理） 仅 Spring Bean 的方法调用 绝大多数业务场景
AspectJ（编译时） 编译期（ajc 编译器） 任意类、构造函数、字段访问 需要拦截非 Spring 管理对象
AspectJ（LTW） 类加载时（Java Agent） 同上，不需要特殊编译器 遗留系统改造
实战建议：99% 的业务需求用 Spring AOP 就够了。AspectJ 只在需要拦截 new 出来的对象（非 Bean）或字段级别的切面时才有必要引入。

九、一句话总结

AOP 的本质是在不改动业务代码的前提下，通过动态代理将横切逻辑织入目标方法。 Spring 中的事务、缓存、安全全都构建在这一机制之上。 理解了"代理对象 → 切面链 → 真实对象"的调用链路， 80% 的 Spring 疑难杂症（事务失效、切面不生效）都可以从这里找到根因。

面试高频：Spring AOP 和 AspectJ 有什么区别？ —— Spring AOP 是运行时动态代理，只能拦截 Spring Bean 的方法调用； AspectJ 是编译期或加载期静态织入，功能更强但引入成本更高。 实际项目优先选 Spring AOP。

与其他知识点的联系

  • IoC 容器：AOP 代理对象的生成时机就在 mechanism/01-IoC-DI.html 讲的容器启动第三阶段——BeanPostProcessor 在 Bean 初始化后介入，把原始 Bean 替换成代理对象，理解了 IoC 容器的生命周期，才能理解"为什么代理是运行时生成的，而不是编译期就存在"。
  • 开闭原则（OCP）：横切逻辑（日志、事务、权限）通过切面"织入"而不是侵入业务代码，正是 clean/08-SOLID原则.html 中 OCP"对扩展开放、对修改关闭"的另一种体现——新增一个横切能力只需新增切面类，不用改动任何业务方法。
  • 责任链模式：多个 Advice 通过 @Order 排序形成调用链、每层可以决定是否继续往下传播（是否调用 proceed()），这与 mechanism/04-设计套路落地.html 中责任链模式的"可中断"特性是同一种设计思想在不同层面的实现。
  • 坏味道中的重复代码：反例里在每个方法手动写日志、权限校验，正是 refactor/01-代码坏味道.html 中"重复代码（Duplicated Code）"的典型场景——AOP 提供了一种框架级的"提炼函数"手段，把重复的横切逻辑一次性抽到切面里。
  • 提炼函数与搬移特性："坑 1：同类内部调用切面不生效"本质上和 refactor/03-提炼函数.html 中"坑四：在事务边界内随意提炼导致事务失效"是同一个根因——AOP 依赖代理对象拦截调用，而 this 调用绕开了代理，理解这一点后，重构时要不要把方法搬到另一个 Bean（refactor/04-搬移特性.html）就多了一层横切关注点的判断依据。
