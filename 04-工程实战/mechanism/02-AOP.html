AOP 面向切面（代理 / 织入）

把日志、事务、安全这类"横切逻辑"从业务代码中抽走——这就是 AOP 要解决的问题。

横切关注点：散落在各处的重复逻辑

设想你有几十个 Service 方法，每个方法都需要：记录日志、开启事务、鉴权。如果把这些代码直接写进每个方法，会发生什么？

  • 代码重复：同样的日志模板抄了几十份
  • 关注点混杂：业务逻辑被非业务代码淹没，难以阅读
  • 修改代价高：改一处日志格式要改几十个地方

这类"横向穿过多个模块"的逻辑，叫做横切关注点（Cross-Cutting Concern）。AOP（Aspect-Oriented Programming）的核心思路是：把横切逻辑从业务代码里抽离成独立的切面（Aspect），再通过某种机制在合适的时机自动织入。

类比：AOP 就像高速公路的收费站。车（方法调用）在各自的车道（业务逻辑）上行驶，不需要每辆车自己带收费功能，而是经过一个统一的收费口（切面）自动扣费，之后继续前行。

织入方式：静态 vs 动态

把切面逻辑"嵌入"到目标代码的过程叫做织入（Weaving）。根据织入时机分为两类：

方式时机代表框架特点
静态织入（编译期） 编译时修改字节码 AspectJ（ajc 编译器） 性能最好，但需要特殊编译器，侵入构建流程
静态织入（类加载期） 类加载时通过 Java Agent 修改字节码 AspectJ LTW 不改源码，但配置复杂
动态代理（运行期） 运行时生成代理对象 Spring AOP 配置简单，但只能代理方法调用，性能有少量开销

Spring AOP 走的是动态代理路线——它不修改任何字节码，而是在运行时生成一个代理对象包裹你的 Bean，所有调用先经过代理，代理再决定前后插入什么逻辑。

JDK 动态代理 vs CGLIB

Spring AOP 的动态代理有两种底层实现，选哪种取决于目标 Bean 是否实现了接口：

JDK 动态代理

基于 java.lang.reflect.Proxy，要求目标类必须实现至少一个接口。代理对象实现相同的接口，通过 InvocationHandler 拦截方法调用。

Java — JDK 动态代理原理（简化版）
// 目标接口
public interface OrderService {
    void placeOrder(Order order);
}

// 生成代理：Proxy.newProxyInstance 在运行时生成实现了 OrderService 的新类
OrderService proxy = (OrderService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    new Class[]{ OrderService.class },
    (proxy2, method, args) -> {
        System.out.println("[LOG] before: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("[LOG] after: " + method.getName());
        return result;
    }
);

CGLIB 代理

当目标类没有实现接口时，Spring 退而使用 CGLIB。CGLIB 通过继承目标类（ASM 字节码库在运行时生成子类）来拦截方法调用，因此 final 类和 final 方法无法被 CGLIB 代理。

Spring Boot 2.x 起默认对所有 Bean 使用 CGLIB（即使有接口），因为 CGLIB 性能更好且避免了接口代理的一些坑。通过 spring.aop.proxy-target-class=false 可切换回 JDK 代理。

Spring AOP 核心概念

理解 AOP 必须掌握这五个概念，它们的关系是：切面（Aspect）= 切入点（Pointcut）+ 通知（Advice）。

概念含义类比
Aspect（切面） 横切逻辑的封装单元，包含 Pointcut 和 Advice 收费站整体
Pointcut（切入点） 用表达式描述"拦截哪些方法" 哪些车道需要收费
Advice（通知） 在切入点前/后/环绕执行的逻辑 收费动作本身
JoinPoint（连接点） 程序执行的某个具体点（方法调用、异常抛出等） 某辆具体的车经过
Weaving（织入） 把切面应用到目标对象的过程 建好收费站并投入使用
Java — 完整切面示例
@Aspect
@Component
public class LogAspect {

    // Pointcut：匹配 service 包下所有 public 方法
    @Pointcut("execution(public * com.example.service.*.*(..))")
    public void serviceLayer() {}

    // Around Advice：环绕通知，最强大，可控制是否执行原方法
    @Around("serviceLayer()")
    public Object logAround(ProceedingJoinPoint pjp) throws Throwable {
        String method = pjp.getSignature().getName();
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed(); // 执行原方法
            System.out.printf("[OK] %s 耗时 %dms%n", method,
                System.currentTimeMillis() - start);
            return result;
        } catch (Exception e) {
            System.out.printf("[ERR] %s 异常: %s%n", method, e.getMessage());
            throw e;
        }
    }
}

@Transactional 失效的三种场景

@Transactional 本质上就是一个 AOP 切面。理解它失效的原因，就是理解动态代理的本质。

场景 1：同类内部自调用（最常见的坑）

Java — 自调用导致事务失效（反例）
@Service
public class OrderService {

    // 外部调用 createOrder 时，调用的是代理对象，事务正常
    @Transactional
    public void createOrder(Order order) {
        saveOrder(order); // ❌ 直接调用 this.saveOrder，绕过了代理！
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveOrder(Order order) { /* ... */ }
}

// 解决方案：注入自身代理，或拆分到另一个 Service
@Autowired
private OrderService self; // 注入的是代理对象
self.saveOrder(order); // ✅ 经过代理，事务生效

场景 2：方法非 public

Spring AOP 基于动态代理，代理只能拦截 public 方法。@Transactional 加在 private/protected/package-private 方法上，注解会被忽略，不会开启事务。

场景 3：异常被吞掉或类型不对

Java — 异常类型导致回滚失效
// @Transactional 默认只对 RuntimeException 和 Error 回滚
// 受检异常（IOException 等）不会触发回滚
@Transactional
public void transferMoney() throws IOException {
    dao.deduct();
    throw new IOException("网络错误"); // ❌ 不会回滚！
}

// 修复：指定 rollbackFor
@Transactional(rollbackFor = Exception.class)
public void transferMoney() throws IOException {
    dao.deduct();
    throw new IOException("网络错误"); // ✅ 指定后会回滚
}
面试必备：除了上面三种，还有一种场景：Bean 没有被 Spring 管理（比如 new 出来的对象），AOP 根本没机会介入。另外，如果数据库引擎是 MyISAM，本身不支持事务，加不加 @Transactional 都没用。
