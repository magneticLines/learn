解耦设计（面向接口 / 依赖倒置）

好的系统可以独立替换每个零件——这需要刻意的解耦设计，而不是偶然。

耦合的危害：牵一发动全身

耦合度高的代码，改动一处会触发连锁反应。以下是一个典型场景：

Java — 高耦合反例
public class OrderService {
    // 直接依赖具体实现类
    private MysqlOrderRepository repo = new MysqlOrderRepository();
    private AlipayGateway         pay  = new AlipayGateway();
    private EmailNotifier          mail = new EmailNotifier();
}

// 问题：
// 1. 想把 MySQL 换成 PostgreSQL？ → 改 OrderService 源码
// 2. 想测试 OrderService？ → 必须启动数据库 + 支付网关 + 邮件服务
// 3. 想换支付商？ → 搜索所有 new AlipayGateway() 逐一替换

高耦合有三种衡量维度：修改的代价（改 A 需要改多少地方）、测试的难度（测 A 需要准备多少环境）、复用的可能（A 能不能在另一个项目里直接用）。

面向接口编程 vs 面向实现编程

解耦最基础的手段：依赖抽象（接口），而不是依赖具体实现。

Java — 面向接口改造
// 定义抽象：接口描述"能做什么"，不关心"怎么做"
public interface PaymentGateway {
    void pay(BigDecimal amount, String orderId);
}

// 多种实现：对接口负责，互相独立
public class AlipayGateway  implements PaymentGateway { ... }
public class WechatGateway  implements PaymentGateway { ... }
public class MockGateway    implements PaymentGateway { ... } // 测试用

// 调用方只认接口：切换实现只需换注入的 Bean，不改业务代码
@Service
public class OrderService {
    private final PaymentGateway payment;
    public OrderService(PaymentGateway payment) {
        this.payment = payment;
    }
}

依赖倒置原则（DIP）在 Spring 中的体现

SOLID 中的 D——依赖倒置原则（Dependency Inversion Principle）说的是：

  • 高层模块不应该依赖低层模块，二者都应该依赖抽象。
  • 抽象不应该依赖细节，细节应该依赖抽象。

传统方式的依赖方向：OrderService → AlipayGateway（高层依赖低层具体实现）。

DIP 之后：OrderService → PaymentGateway（接口）← AlipayGateway。依赖关系"倒置"了——低层实现去依赖（实现）高层定义的接口。

Spring 的整套 IoC/DI 机制本质上就是在工程层面强制落地 DIP。你注入的永远是接口类型，Spring 容器在运行时决定注入哪个实现——这个决策权从调用方移交给了配置（或注解）。

策略模式：运行时解耦

策略模式是 DIP 的一个经典落地：把"算法"封装成可替换的策略接口，运行时按需切换，调用方对切换过程无感知。

Java — 策略模式实现促销计算
// 策略接口
public interface DiscountStrategy {
    BigDecimal calculate(BigDecimal original);
}

// 具体策略（各自独立，互不影响）
@Component("noDiscount")
class NoDiscountStrategy     implements DiscountStrategy {
    public BigDecimal calculate(BigDecimal p) { return p; }
}
@Component("vip")
class VipDiscountStrategy     implements DiscountStrategy {
    public BigDecimal calculate(BigDecimal p) { return p.multiply(new BigDecimal("0.9")); }
}

// 调用方：注入所有策略，运行时按 key 选择
@Service
public class OrderPriceService {
    // Spring 会把所有 DiscountStrategy 的 Bean 注入到 Map 中
    // key = Bean 名称，value = Bean 实例
    private final Map<String, DiscountStrategy> strategies;

    public OrderPriceService(Map<String, DiscountStrategy> strategies) {
        this.strategies = strategies;
    }

    public BigDecimal price(String type, BigDecimal original) {
        // 新增促销类型只需新增 Strategy Bean，这里不改
        return strategies.getOrDefault(type, strategies.get("noDiscount"))
                          .calculate(original);
    }
}
这个技巧（注入 Map<String, Interface>）是 Spring 中实现策略模式最优雅的写法，避免了大量 if-else，也满足开闭原则：新增策略只加类，不改现有代码。

SPI 机制：框架级的插件化解耦

SPI（Service Provider Interface）是 Java 提供的一种框架与实现分离的机制。框架只定义接口，实现由外部 JAR 提供，运行时动态发现。

Java 原生 SPI

文件结构 — META-INF/services/
# 文件路径：META-INF/services/com.example.PaymentGateway
# 内容：每行一个实现类的全限定名
com.alipay.AlipayGateway
com.wechat.WechatGateway
Java — ServiceLoader 加载实现
// JDK 的 ServiceLoader 扫描 classpath 下所有 META-INF/services 文件
ServiceLoader<PaymentGateway> loader =
    ServiceLoader.load(PaymentGateway.class);
loader.forEach(gw -> System.out.println(gw.getClass()));

Spring SPI（spring.factories）

Spring Boot 的自动配置（AutoConfiguration）就是 SPI 的深度应用。每个 Starter JAR 在 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 中声明自己的配置类，Spring Boot 启动时统一加载。这就是为什么你只加一个 spring-boot-starter-redis 依赖，Redis 就自动配好了——它通过 SPI 告诉框架"我在这里，请加载我"。

事件驱动：发布者不认识订阅者

策略模式和 SPI 都要求调用方至少知道接口。事件驱动连接口都不需要知道——发布方只管发事件，订阅方只管监听，二者完全解耦。

Java — Spring 事件机制
// 1. 定义事件（值对象，携带数据）
public class OrderCreatedEvent extends ApplicationEvent {
    private final String orderId;
    public OrderCreatedEvent(Object source, String orderId) {
        super(source);
        this.orderId = orderId;
    }
    public String getOrderId() { return orderId; }
}

// 2. 发布方：只需要注入 ApplicationEventPublisher，不知道谁在监听
@Service
public class OrderService {
    private final ApplicationEventPublisher eventPublisher;

    public void createOrder(Order order) {
        // 保存订单...
        eventPublisher.publishEvent(new OrderCreatedEvent(this, order.getId()));
    }
}

// 3. 订阅方（可以有多个，互不影响，随时新增）
@Component
public class EmailNotificationListener {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        // 发邮件通知——OrderService 完全不知道这里的存在
    }
}

@Component
public class InventoryListener {
    @EventListener
    @Async // 异步处理，不阻塞主流程
    public void onOrderCreated(OrderCreatedEvent event) {
        // 扣减库存
    }
}
注意：Spring 事件默认是同步的（同一个线程顺序执行所有监听器）。加 @Async 可以变成异步，但要注意事务边界——异步监听器在新线程里，不共享调用方的事务上下文。
