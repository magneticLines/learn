框架机制 · 解耦与扩展机制

常见设计套路在工程中的落地

策略、模板方法、责任链、工厂——从 GoF 定义到生产代码，理解每个套路真正解决了什么工程问题。

1. 为什么设计模式在工程中常被用错

GoF 的 23 个模式描述的是意图，而非模板。工程师最常见的失误是先选模式、再套业务——结果代码复杂度翻倍，却没有解决任何真实问题。

正确姿势：先识别痛点，再看哪个套路天然契合。下面列出生产代码中频次最高的四个套路，每个都从"反例 → 问题 → 正例 → 坑"结构展开。

本文所有示例均来自电商/支付/消息推送等真实业务场景，变量名刻意贴近生产命名风格。

2. 策略模式（Strategy）——消灭 if-else 分支爆炸

痛点场景：支付渠道选择

订单服务需要根据用户选择调用不同支付渠道（微信、支付宝、银行卡、Apple Pay……），最直觉的写法是一棵 if-else 树。

反例 — PaymentService.java（if-else 地狱）
public String pay(Order order, String channel) {
    if ("wechat".equals(channel)) {
        // 微信 SDK 初始化、签名、调用……30 行
        return wechatSdk.pay(order);
    } else if ("alipay".equals(channel)) {
        // 支付宝逻辑……40 行
        return alipaySdk.pay(order);
    } else if ("card".equals(channel)) {
        // 银行卡逻辑……
    }
    // 每新增渠道 = 改这个文件，违反开闭原则
    throw new UnsupportedOperationException("unknown channel");
}

问题：每次新增/删除渠道都要动核心类，单元测试难以隔离，多人协作必然产生 git 冲突。

正例：Strategy + Spring 自动注册

PaymentStrategy.java — 策略接口
public interface PaymentStrategy {
    /** 渠道标识，与前端传参保持一致 */
    String channel();
    String pay(Order order);
}
WechatPaymentStrategy.java — 具体策略
@Component  // Spring 托管，自动加入容器
public class WechatPaymentStrategy implements PaymentStrategy {

    @Override
    public String channel() { return "wechat"; }

    @Override
    public String pay(Order order) {
        // 只关注微信逻辑，不干扰其他渠道
        return wechatSdk.pay(order);
    }
}
PaymentService.java — 上下文（Context）
@Service
public class PaymentService {

    // Spring 自动将所有 PaymentStrategy bean 注入 List
    // 再由构造器转成 channel -> strategy 的 Map
    private final Map<String, PaymentStrategy> strategyMap;

    public PaymentService(List<PaymentStrategy> strategies) {
        // 启动时一次性构建 channel → strategy 映射
        strategyMap = strategies.stream()
            .collect(Collectors.toMap(
                PaymentStrategy::channel,
                Function::identity
            ));
    }

    public String pay(Order order, String channel) {
        PaymentStrategy strategy = Optional
            .ofNullable(strategyMap.get(channel))
            .orElseThrow(() ->
                new BusinessException("不支持的支付渠道: " + channel));
        return strategy.pay(order);
    }
}

新增渠道只需新建一个 @Component 类，不动任何已有代码，完全符合开闭原则。

生产坑：channel() 值与前端/数据库枚举不统一。建议用常量或枚举统一维护渠道标识，而不是散落的字符串字面量。另一个坑是同一个 channel 注册了两个 Strategy Bean，toMap 会抛 IllegalStateException，上线时才爆。可用 toMap(..., ..., (a, b) -> { throw ... }) 在启动阶段就发现冲突。

3. 模板方法（Template Method）——固化流程，开放细节

痛点场景：多渠道消息推送

短信、邮件、站内信、企业微信的推送流程高度相似：校验参数 → 拼装内容 → 调用渠道 SDK → 记录发送日志。不同的只有"拼装内容"和"调用 SDK"这两步。

反例 — SmsSender.java 与 EmailSender.java（代码大量重复）
// SmsSender
public void send(Notification n) {
    validate(n);              // 复制粘贴
    String content = buildSmsContent(n);
    smsSdk.send(n.getPhone(), content);
    logResult(n, "SMS");       // 复制粘贴
}

// EmailSender — 几乎一模一样，只有中间两行不同
public void send(Notification n) {
    validate(n);
    String content = buildEmailHtml(n);
    emailSdk.send(n.getEmail(), content);
    logResult(n, "EMAIL");
}

问题：校验逻辑或日志格式一旦变更，需要同步修改所有渠道实现，极易遗漏。

正例：抽象类固化骨架

AbstractNotificationSender.java — 模板
public abstract class AbstractNotificationSender {

    /**
     * 模板方法：final 保证子类不能改变流程顺序
     * 这是模式的核心——"骨架不可变，细节可扩展"
     */
    public final void send(Notification notification) {
        validate(notification);
        String content = buildContent(notification);  // 抽象步骤
        doSend(notification, content);               // 抽象步骤
        logResult(notification, channelName());
    }

    // private：公共实现，子类不得覆盖
    private void validate(Notification n) {
        Objects.requireNonNull(n.getUserId(), "userId must not be null");
    }

    private void logResult(Notification n, String channel) {
        // 统一写入 send_log 表，所有渠道共享同一套日志逻辑
        sendLogRepository.save(SendLog.of(n, channel));
    }

    // 子类必须实现的差异点
    protected abstract String buildContent(Notification n);
    protected abstract void   doSend(Notification n, String content);
    protected abstract String channelName();
}
SmsSender.java — 子类只填差异
@Component
public class SmsSender extends AbstractNotificationSender {

    @Override
    protected String buildContent(Notification n) {
        return "【XX平台】" + n.getMessage();
    }

    @Override
    protected void doSend(Notification n, String content) {
        smsSdk.send(n.getPhone(), content);
    }

    @Override
    protected String channelName() { return "SMS"; }
}
何时用模板方法而非策略？当各实现之间有公共骨架且骨架顺序不能乱（如事务提交必须在业务逻辑之后），用模板方法。当实现彼此完全独立、可以互换时，用策略。两者常组合使用：骨架固定（模板方法），骨架中的某步可插拔（策略）。
生产坑：滥用 protected 导致子类偷偷覆盖公共步骤。模板方法中的公共步骤（如 validate、logResult）必须是 private，绝不能是 protected。一旦子类能覆盖，"骨架不可变"的保证就形同虚设。

4. 责任链（Chain of Responsibility）——请求流水线

痛点场景：订单创建前置校验

订单创建前需要依次校验：库存、风控、优惠券有效性、用户信誉分……校验项随着业务增长越来越多，且每项校验都可能提前终止流程。

反例 — OrderService.java（校验逻辑堆砌）
public Order createOrder(CreateOrderRequest req) {
    // 库存校验
    if (!inventoryService.isAvailable(req)) {
        throw new OrderException("库存不足");
    }
    // 风控校验
    if (riskService.isBlocked(req.getUserId())) {
        throw new OrderException("账号风控拦截");
    }
    // 优惠券校验 …… 越来越长
    // OrderService 职责越来越重，单测越来越难写
}

问题：校验项顺序调整、新增/删除一项，都必须改 OrderService。该类会无限膨胀，最终成为"上帝类"。

正例：责任链 + Spring 有序注入

OrderCheckHandler.java — 处理器接口
public interface OrderCheckHandler extends Ordered {
    /**
     * @param ctx 贯穿整条链的上下文，既传数据也回写结果
     * @return true=继续往下传，false=终止链
     */
    boolean handle(OrderCheckContext ctx);
}
InventoryCheckHandler.java — 一个节点
@Component
@Order(10)  // 数字越小越先执行；库存校验最优先
public class InventoryCheckHandler implements OrderCheckHandler {

    @Override
    public boolean handle(OrderCheckContext ctx) {
        if (!inventoryService.isAvailable(ctx.getRequest())) {
            ctx.reject("库存不足");
            return false;  // 短路，后续节点不再执行
        }
        return true;
    }

    @Override
    public int getOrder() { return 10; }
}
OrderCheckChain.java — 链驱动器
@Component
public class OrderCheckChain {

    // Spring 按 @Order 排好序后注入，无需手动排列
    private final List<OrderCheckHandler> handlers;

    public OrderCheckChain(List<OrderCheckHandler> handlers) {
        this.handlers = handlers;  // 已由 Spring 自动排序
    }

    public void check(OrderCheckContext ctx) {
        for (OrderCheckHandler handler : handlers) {
            if (!handler.handle(ctx)) {
                return;  // 某节点返回 false，立即中断
            }
        }
    }
}

增加"黑名单校验"节点只需新建一个 @Component @Order(5) 的类，无需修改 OrderService 或 OrderCheckChain。

生产坑1：节点执行顺序依赖未显式声明。@Order 值没有统一规划时，两个开发者各自用了 @Order(1)，顺序就变成不确定的（取决于 Spring 扫描顺序）。建议在项目里用常量类统一管理 Order 值，例如 OrderHandlerOrders.INVENTORY = 100。

生产坑2：链上下文（Context）线程安全问题。如果链中有异步节点，Context 对象必须是线程安全的，或者每次请求都 new 一个新实例（别把 Context 设计成单例 Bean）。

5. 工厂模式（Factory）——隔离对象创建与使用

痛点场景：多环境数据源切换

测试环境用 H2、预发环境用 MySQL、生产环境用 RDS。如果在业务代码里直接 new 数据源，环境切换就意味着改代码。

反例 — 直接在业务层 new 对象
// 业务代码里出现了环境判断——绝对不应该
DataSource ds;
if ("prod".equals(env)) {
    ds = new RdsDataSource(rdsUrl);
} else {
    ds = new H2DataSource();
}

正例：工厂方法 + Spring Conditional

DataSourceFactory.java — 工厂接口
public interface DataSourceFactory {
    DataSource create();
}
RdsDataSourceFactory.java — 生产实现
@Component
@ConditionalOnProperty(name = "app.env", havingValue = "prod")
public class RdsDataSourceFactory implements DataSourceFactory {

    @Override
    public DataSource create() {
        // 生产 RDS 连接配置，包含 SSL、连接池调优参数
        return DataSourceBuilder.create().url(rdsUrl).build();
    }
}

业务代码依赖 DataSourceFactory 接口，环境切换只需改配置文件，与业务逻辑零耦合。

抽象工厂：同族对象一起创建

当一组相关对象需要保持一致（例如：同一云厂商的 OSS 客户端、消息队列客户端、KV 存储客户端必须配套）时，用抽象工厂保证"同族一致性"。

CloudInfraFactory.java — 抽象工厂
public interface CloudInfraFactory {
    OssClient   createOssClient();
    MqClient    createMqClient();
    CacheClient createCacheClient();
}

// 阿里云实现：三个客户端都来自阿里云 SDK，不会混用
@Component
@ConditionalOnProperty(name = "cloud.provider", havingValue = "aliyun")
public class AliyunInfraFactory implements CloudInfraFactory {
    @Override public OssClient   createOssClient() { return new AliyunOssClient(); }
    @Override public MqClient    createMqClient() {  return new RocketMQClient();  }
    @Override public CacheClient createCacheClient() {return new TairClient();      }
}
生产坑：工厂被滥用成"ServiceLocator"。如果工厂内部持有 ApplicationContext 并调用 getBean()，就退化成了服务定位器——依赖关系隐蔽，单测极难 Mock。正确做法是工厂只负责 new / 组装对象，不从容器里查 Bean。

6. 组合运用：Spring 的 Filter / Interceptor 链

Spring MVC 的 HandlerInterceptor 和 Servlet Filter 就是责任链的标准工程实现，理解它们的设计能让你更快掌握上面的套路。

请求进入
  ↓
Filter 链（Servlet 规范，框架无关）
  ↓
HandlerInterceptor 链（Spring MVC）
  ↓
Controller 方法
  ↑
HandlerInterceptor.postHandle（反向执行）
  ↑
HandlerInterceptor.afterCompletion（反向执行）

每一层都可以短路（返回 false 或不调用 chain.doFilter），这正是责任链"可中断"特性在框架层的体现。鉴权 Filter 就利用了这点：Token 非法直接返回 401，后续所有处理器不执行。

面试高频：Filter 和 Interceptor 的区别？
Filter 是 Servlet 规范，在 DispatcherServlet 之前，能拦截所有请求（包括静态资源）；Interceptor 是 Spring MVC 层面，只能拦截 Controller 请求，但可以访问 HandlerMethod，方便拿到方法上的注解（如 @RequiresLogin）做鉴权。两者都是责任链，但切入点不同。

7. 套路对比速查

套路 核心意图 典型工程场景 常见坑
策略 同一行为，多种算法可互换 支付渠道、促销计算、排序规则 Key 冲突、枚举与 Key 不统一
模板方法 固化流程骨架，开放细节步骤 消息推送、数据导入导出、批处理 公共步骤误设 protected 被覆盖
责任链 请求依次经过多个处理器，可中断 前置校验、审批流、日志/鉴权 Filter Order 冲突、Context 线程安全
工厂方法 隔离对象创建，调用方不依赖具体类 多环境数据源、插件化组件 退化成 ServiceLocator
抽象工厂 一族相关对象保持一致创建 多云基础设施、多主题 UI 组件 族内对象膨胀后接口难以维护

8. 生产环境通用建议

  • 先识别痛点，再选套路。问自己：如果再来一个新的"XX"（渠道/校验项/渲染器），我需要改几个文件？如果答案是"只新建一个类"，说明套路用对了。
  • Spring 自动注入天然支持这些套路。List<T> 注入 + @Order 是责任链和策略注册的标准实践，无需手写链表指针。
  • Context 对象是链/流水线的生命线。设计好贯穿整个流程的 Context 类，比节点间互相传参更清晰，也更易扩展。
  • 单测验证套路正确性。策略和责任链的最大价值之一就是可独立单测每个节点，上生产前一定覆盖到各节点的短路/通过逻辑。
  • 警惕"模式膨胀"。如果只有两三种变体且变化频率极低，直接 if-else 比引入策略模式更易读。模式是工具，不是目标。

与其他知识点的联系

  • IoC 容器与策略注册：本文策略模式、责任链示例里 List<PaymentStrategy>、List<OrderCheckHandler> 的自动收集机制，正是 mechanism/01-IoC-DI.html 中"IoC 容器如何让多实现注入变得优雅"的具体应用——理解了容器如何按类型收集 Bean，才能理解这里为什么新增一个 @Component 就能自动加入策略/责任链，不需要手动注册。
  • 开闭原则（OCP）与单一职责原则（SRP）：本文反复强调"新增渠道/校验项只需新建一个类"，这正是 clean/08-SOLID原则.html 中 OCP 的核心检验标准；而策略类、处理器类各自只负责一件事，则对应 SRP。设计模式在这里是 SOLID 原则的具体实现载体，而不是孤立的技巧。
  • Switch 惊悚现身与基本类型偏执：策略模式一节的反例（if-else 判断支付渠道字符串）正是 refactor/01-代码坏味道.html 中"Switch Statements"和"Primitive Obsession"两种坏味道的组合体现，策略模式则是这两种坏味道共同的重构目标。
  • 搬移函数：模板方法一节里把校验、日志等公共步骤收敛到抽象类的 final 方法中，与 refactor/04-搬移特性.html 中"搬移语句到函数"是同一种手法——多处重复的前后置代码应该归入方法本身，而不是让每个子类各自复制。
  • AOP 与责任链/Filter：第 6 节提到的 Spring Filter / Interceptor 链，本质上和 mechanism/02-AOP.html 中切面链的"调用前 → proceed → 调用后"结构高度相似，两者都是"可短路的处理链"，只是切入的框架层次不同（Servlet 规范 vs Spring AOP 动态代理）。
