框架机制 · 04

常见设计套路在工程中的落地

教科书里的设计模式真正在 Spring 源码和日常开发中长什么样？本篇一一对照。

工厂模式 → BeanFactory / FactoryBean

核心思路：把"如何创建对象"封装起来，调用方只需要告诉工厂"我要什么"，不关心怎么造。

Spring 源码 org.springframework.beans.factory.BeanFactory 就是顶层工厂接口，DefaultListableBeanFactory 是最核心的实现，负责管理所有 Bean 的创建与缓存。

Java — 简化版 BeanFactory（手写理解原理）
// 工厂接口：只暴露"拿 Bean"的方法，隐藏创建细节
public interface SimpleBeanFactory {
    Object getBean(String name);
}

// 工厂实现：持有 BeanDefinition 注册表 + 单例缓存
public class DefaultBeanFactory implements SimpleBeanFactory {
    private final Map<String, Class<?>> registry   = new HashMap<>();
    private final Map<String, Object>   singletons = new HashMap<>();

    public void register(String name, Class<?> clazz) {
        registry.put(name, clazz);
    }

    @Override
    public Object getBean(String name) {
        // 单例缓存命中则直接返回，否则反射创建
        return singletons.computeIfAbsent(name, k -> {
            try {
                return registry.get(k).getDeclaredConstructor().newInstance();
            } catch (Exception e) { throw new RuntimeException(e); }
        });
    }
}
FactoryBean vs BeanFactory：名字相似，概念不同。BeanFactory 是整个容器；FactoryBean 是一个特殊的 Bean，它自己充当工厂来生产另一个 Bean（常见于 MyBatis 的 SqlSessionFactoryBean、Feign 的代理生成等）。

模板方法 → JdbcTemplate / AbstractList

核心思路：父类定义算法骨架（流程），把变化的步骤留给子类填充。"流程不变，细节各异"。

Spring 源码 JdbcTemplate 的 query() 方法封装了"获取连接 → 创建 Statement → 执行 SQL → 处理结果集 → 释放资源"的全部流程，你只需要提供"SQL 是什么"和"怎么映射结果"。

Java — 手写模板方法（数据库查询骨架）
// 抽象模板：固定查询流程，子类实现变化部分
public abstract class QueryTemplate<T> {

    // 模板方法：final 防止子类破坏流程
    public final List<T> query(String sql) {
        Connection conn = getConnection();       // 步骤1：获取连接（固定）
        ResultSet  rs   = execute(conn, sql);  // 步骤2：执行（固定）
        List<T>   list = mapRows(rs);          // 步骤3：映射结果（抽象，子类实现）
        close(conn, rs);                       // 步骤4：释放资源（固定）
        return list;
    }

    // 子类只需实现这一个变化点
    protected abstract List<T> mapRows(ResultSet rs);
    // getConnection / execute / close 是固定实现，省略
}

// Spring 的做法更现代：用回调接口（RowMapper）代替继承
// 效果等同于模板方法，但更灵活（Lambda 友好）
List<User> users = jdbcTemplate.query(
    "SELECT * FROM users WHERE age > ?",
    (rs, row) -> new User(rs.getLong("id"), rs.getString("name")),
    18
);

观察者模式 → ApplicationEvent / Listener

核心思路：被观察者（发布者）状态改变时，自动通知所有观察者（订阅者），双方通过事件接口松耦合。

Spring 源码 ApplicationContext 实现了 ApplicationEventPublisher 接口，内部通过 SimpleApplicationEventMulticaster 维护监听器列表并广播事件。

Java — 手写最简观察者（不依赖 Spring）
// 观察者接口
public interface EventListener<E> {
    void onEvent(E event);
}

// 事件总线（发布者）
public class EventBus<E> {
    private final List<EventListener<E>> listeners = new CopyOnWriteArrayList<>();

    public void subscribe(EventListener<E> listener) {
        listeners.add(listener);
    }

    public void publish(E event) {
        // 发布方不知道有哪些监听者，列表动态维护
        listeners.forEach(l -> l.onEvent(event));
    }
}

// 使用
EventBus<String> bus = new EventBus<>();
bus.subscribe(e -> System.out.println("邮件: " + e));
bus.subscribe(e -> System.out.println("短信: " + e));
bus.publish("订单已创建");

责任链模式 → Filter / HandlerInterceptor

核心思路：把多个处理者串成链，请求沿链传递，每个处理者决定是"处理后继续传递"还是"拦截终止"。

Spring 源码 Servlet 的 FilterChain、Spring MVC 的 HandlerInterceptor、Spring Security 的过滤器链都是责任链的工程实现。

Java — 手写责任链（请求校验场景）
// 处理者接口
public abstract class RequestHandler {
    protected RequestHandler next;

    public RequestHandler setNext(RequestHandler next) {
        this.next = next;
        return next; // 返回 next 支持链式调用
    }

    // 子类实现：处理后调用 next.handle() 继续传递
    public abstract void handle(Request req);
}

// 具体处理者：鉴权
public class AuthHandler extends RequestHandler {
    public void handle(Request req) {
        if (!req.hasToken()) {
            throw new UnauthorizedException(); // 拦截，不往下传
        }
        if (next != null) next.handle(req); // 通过则继续传递
    }
}

// 具体处理者：限流
public class RateLimitHandler extends RequestHandler {
    public void handle(Request req) {
        if (isOverLimit(req)) throw new TooManyRequestsException();
        if (next != null) next.handle(req);
    }
}

// 组装责任链
RequestHandler chain = new AuthHandler();
chain.setNext(new RateLimitHandler())
     .setNext(new LoggingHandler());
chain.handle(request);
Spring Security 的过滤器链（SecurityFilterChain）就是标准责任链。它有 15+ 个内置 Filter 按顺序排列，每个处理一种安全逻辑（CSRF、Session 管理、JWT 校验……），你可以在链中任意位置插入自定义 Filter。

装饰器模式 → InputStream / BufferedReader

核心思路：在不改变原始对象的前提下，动态地给它叠加新功能。与继承的区别在于：装饰器是"包裹"而非"扩展"，可以在运行时动态组合。

JDK 源码 java.io 包是装饰器模式的教科书级实现。InputStream 是组件接口，FileInputStream 是基础实现，BufferedInputStream / GZIPInputStream / DataInputStream 都是装饰器，可以任意嵌套叠加。

Java — IO 流的装饰器嵌套
// 每一层包裹都在原有能力上叠加新功能
// FileInputStream：读文件字节
// BufferedInputStream：在其上加缓冲，减少系统调用
// GZIPInputStream：在其上加解压
// InputStreamReader：字节流转字符流
// BufferedReader：在字符流上加缓冲 + readLine()
BufferedReader reader = new BufferedReader(
    new InputStreamReader(
        new GZIPInputStream(
            new BufferedInputStream(
                new FileInputStream("data.gz")
            )
        ),
        StandardCharsets.UTF_8
    )
);
Java — 手写装饰器（给日志记录加时间戳）
// 组件接口
public interface Logger {
    void log(String msg);
}

// 基础实现
public class ConsoleLogger implements Logger {
    public void log(String msg) {
        System.out.println(msg);
    }
}

// 装饰器：持有被装饰的 Logger，在原功能上叠加时间戳
public class TimestampLogger implements Logger {
    private final Logger delegate;                // 被包裹的对象
    public TimestampLogger(Logger delegate) {
        this.delegate = delegate;
    }
    public void log(String msg) {
        // 增强：在前面加时间戳，然后委托给原始 Logger
        delegate.log(LocalDateTime.now() + " | " + msg);
    }
}

// 使用：嵌套组合
Logger logger = new TimestampLogger(new ConsoleLogger());
logger.log("服务启动"); // 输出: 2026-06-29T... | 服务启动

五种模式对比速查

模式解决的问题Spring/JDK 落地关键特征
工厂模式 隐藏对象创建复杂性 BeanFactory, FactoryBean 调用方不 new，只向工厂要
模板方法 复用算法骨架，隔离变化点 JdbcTemplate, AbstractList 父类定流程，子类填细节
观察者模式 事件广播，发布者/订阅者解耦 ApplicationEvent/Listener 一对多通知，双方不直接依赖
责任链模式 多步骤处理，可插拔可排序 FilterChain, Interceptor 处理者链式传递，可中断
装饰器模式 动态叠加功能，不改原始类 InputStream 家族 包裹同接口，委托后增强
面试区分点：装饰器 vs 代理。两者结构相似（都是包裹），但意图不同——装饰器是增强功能（调用方知道装饰的存在）；代理是控制访问（调用方通常不感知代理）。Spring AOP 用的是代理，BufferedReader 用的是装饰器。
