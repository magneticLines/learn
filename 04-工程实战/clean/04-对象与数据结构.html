对象与数据结构——得墨忒耳律

对象不是数据的容器，它是行为的封装。混淆这两者，是面向对象设计中最常见的误解之一。

对象 vs 数据结构的本质区别

这是一对常被混淆的概念，但它们在设计上是截然对立的：

维度对象（Object）数据结构（Data Structure）
暴露什么行为（方法），隐藏数据数据（字段），没有行为
扩展新类型容易（新增子类，不改现有代码）困难（所有处理函数都要改）
扩展新操作困难（所有子类都要加方法）容易（新增函数，不影响结构）
典型代表Domain Entity、ServiceDTO、Record、JSON 映射类

理解这个对立后，就能明白为什么 OOP 不是银弹：对象适合频繁扩展类型，数据结构适合频繁扩展操作。选错了，维护成本会成倍增长。

得墨忒耳律（Law of Demeter）

得墨忒耳律，也叫「最少知识原则」，用一句话描述：只和你的直接朋友说话，不要和朋友的朋友说话。

更准确地说，一个方法 m 只能调用以下对象的方法：

  • 它所在类的其他方法（this）
  • 方法 m 的参数
  • 方法 m 内部创建的对象
  • 类的直接字段（成员变量）

火车失事代码（Train Wreck）

违反得墨忒耳律最典型的表现，是一串链式调用——代码像火车出轨：

Java — 火车失事反例
// 调用者需要了解 Order → User → Address 三层结构
// 一旦中间任何一层结构变化，这行代码就要改
String city = order.getUser().getAddress().getCity();

重构：把行为推回对象内部

Java — 重构后
// Order 自己知道怎么获取配送城市，调用者不需要了解内部结构
String city = order.getDeliveryCity();

// Order 类内部封装细节
public class Order {
    private final User user;

    public String getDeliveryCity() {
        // 细节被封装在这里，外部调用者不感知
        return user.getAddress().getCity();
    }
}
注意：得墨忒耳律针对的是对象（有行为的东西），不是数据结构。对纯数据结构（DTO、Record）做链式访问是完全正常的——dto.getAddress().getCity() 没有违反任何原则，因为这里没有隐藏的行为，只是在读数据。

DTO / VO / Entity 的区分

在实际 Java 项目（尤其是 Spring 项目）中，这三种对象经常混用，但它们有明确的职责边界：

类型全称职责特点
Entity领域实体映射数据库表，承载核心业务规则有 ID，有生命周期，可以包含业务方法
DTOData Transfer Object跨层传输数据（Controller ↔ Service）纯数据，无业务逻辑，随接口设计变化
VOValue Object表示没有标识的值（钱、地址、颜色）不可变，用值相等（equals）而非引用相等
Java — VO 示例：Money（值对象）
// 值对象：不可变，用 equals 比较，封装了货币运算规则
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money add(Money other) {
        if (!currency.equals(other.currency))
            throw new CurrencyMismatchException();
        // 返回新对象而不是修改 this——不可变性保证
        return new Money(amount.add(other.amount), currency);
    }
}
面试考点：Entity 和 VO 的区别在于有没有「身份标识」。同样 100 元，两张不同的钞票是不同的 Entity（有序列号），但在业务逻辑里它们是相同的 VO（只关心金额和货币）。

与其他知识点的联系

  • 错误处理（下一篇）：对象和数据结构在错误处理上的取舍也不同——领域对象（如 Order）适合在内部方法中直接抛出携带业务上下文的异常，因为它了解自己的业务规则；而 DTO 这类纯数据结构通常不承担校验职责，校验逻辑应该放在专门的 Validator 或对象的构造过程中，不要指望数据结构自己"知道"什么时候该报错。
  • 封装与多态（OOP 三大特性）：得墨忒耳律本质上是"封装"原则的延伸——真正的封装不只是把字段设为 private，还包括不让调用者需要了解对象内部的协作关系。order.getDeliveryCity() 把"Order 依赖 User 依赖 Address"这个内部协作链隐藏起来，这正是封装要解决的核心问题：调用者只需要知道"做什么"，不需要知道"怎么做"。
  • 设计模式中的访问者模式与策略模式："对象适合扩展类型、数据结构适合扩展操作"这个对立，正是访问者模式（Visitor Pattern）要解决的经典矛盾——访问者模式本质上是在对象身上模拟数据结构的"易扩展操作"特性，理解了这个对立关系，再学访问者模式会更容易明白它为什么长这个样子。
  • Spring 生态中的 DTO/Entity/VO 分层：Controller 用 DTO 接收请求、Service 内部用 Entity 处理业务、返回给前端再转成 VO——这套分层实践的合理性正来自本文的核心结论：跨层传输的数据宜用"数据结构"（结构稳定、无行为），承载业务规则的核心领域对象宜用"对象"（封装行为、隐藏细节），混用两者（比如让 DTO 承担校验逻辑）是很多项目架构混乱的起点。
  • 命名（回顾）：getDeliveryCity() 这样的方法命名要能准确表达"业务意图"而不是"实现细节"，这需要命名一文里强调的"见名知意"能力——调用者看到方法名就该知道能拿到什么，而不用关心背后是从 User 还是 Address 取的。
