边界——隔离第三方代码

我们的代码总要与外部世界打交道——第三方库、框架、外部服务。边界管理的核心问题是：当外部世界变化时，如何让我们的代码保持稳定？

为什么要隔离第三方库

第三方库有自己的演化节奏，与我们的业务需求无关。直接在业务代码中大量使用第三方 API，会造成以下问题：

  • 升级成本高：库升级/换库时，需要修改散落在各处的调用代码
  • 测试困难：第三方库往往难以 mock，导致单元测试难以隔离
  • API 泄漏：库的类型、枚举、异常泄露到业务层，增加耦合
  • 接口不符合业务语义：库的 API 设计面向通用场景，不符合你的业务术语
核心思路：在你的代码和第三方代码之间建立一个「边界层」（Boundary Layer）——一个由你控制、符合你业务语义的接口。外部库在边界层背后实现细节，业务代码只依赖边界层。

Adapter/Wrapper 模式实现边界

边界层最常用的实现方式是 Wrapper（包装器）或 Adapter（适配器）——对外暴露符合业务的接口，在内部调用第三方 API。

反例：直接使用 Jackson，API 散落各处

Java — 反例：Jackson 直接调用
// UserService.java — Jackson API 直接出现在业务层，第三方细节泄漏
public String exportUser(User user) {
    try {
        ObjectMapper mapper = new ObjectMapper();
        mapper.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);
        return mapper.writeValueAsString(user);
    } catch (JsonProcessingException e) {  // Jackson 异常也泄漏到方法签名
        throw new RuntimeException(e);
    }
}

正例：用 Wrapper 封装 Jackson，定义业务边界接口

Java — 正例：JsonSerializer 边界层
// 1. 边界接口：业务代码只依赖这个接口，不知道 Jackson 的存在
public interface JsonSerializer {
    String serialize(Object obj);
    <T> T deserialize(String json, Class<T> type);
}

// 2. Jackson 实现：第三方细节被隔离在这里
@Component
public class JacksonJsonSerializer implements JsonSerializer {
    private final ObjectMapper mapper;

    public JacksonJsonSerializer() {
        this.mapper = new ObjectMapper();
        mapper.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);
    }

    public String serialize(Object obj) {
        try {
            return mapper.writeValueAsString(obj);
        } catch (JsonProcessingException e) {
            // 把 Jackson 异常转换为业务异常，不暴露第三方类型
            throw new SerializationException("Failed to serialize", e);
        }
    }
}

// 3. 业务代码：只知道 JsonSerializer 接口
public class UserService {
    private final JsonSerializer serializer;  // 依赖抽象，非具体

    public String exportUser(User user) {
        return serializer.serialize(user);
    }
}

现在如果把 Jackson 换成 Gson，只需新写一个 GsonJsonSerializer 实现，业务代码零修改。测试时也可以直接 mock JsonSerializer 接口，无需真实的 Jackson 依赖。

学习性测试（Learning Tests）

当引入一个新的第三方库时，Martin 推荐先写「学习性测试」——不是为了测试库本身，而是为了验证你对这个库的理解是否正确，并在库升级时检测 API 变化。

Java — Redis 学习性测试示例
@Test
void redisSetAndGet_shouldReturnStoredValue() {
    // 验证我们理解的 Redis 行为：set 后 get 返回同一值
    redisTemplate.opsForValue().set("test-key", "hello");
    String value = redisTemplate.opsForValue().get("test-key");
    assertThat(value).isEqualTo("hello");
}

@Test
void redisExpiry_shouldExpireAfterTtl() {
    // 验证 TTL 过期行为——库升级时此行为可能悄悄改变
    redisTemplate.opsForValue().set("expire-key", "bye", 1, TimeUnit.SECONDS);
    Thread.sleep(1500);
    assertThat(redisTemplate.opsForValue().get("expire-key")).isNull();
}
好处双重：学习性测试既帮你探索新 API，也在库升级时提供自动保护网——一旦库的行为发生变化，测试失败会立即提醒你。

边界管理的实践清单

场景推荐做法
引入新第三方库先写学习性测试，再写 Wrapper 接口
第三方库异常在 Wrapper 内部转换为业务异常，不暴露第三方类型
第三方数据类型在边界层做类型转换，内部使用自定义类型
库升级只修改 Wrapper 实现，业务代码不感知
单元测试mock 边界接口，不依赖真实库实例
