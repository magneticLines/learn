命名：见名知意 / 一致 / 可搜索

命名是代码最廉价的文档，也是最容易被忽视的工程问题。一个好名字能让接手者瞬间理解意图；一个坏名字会让人反复跳来跳去猜它是什么。

一、为什么命名是工程问题，而不是风格偏好

很多团队把命名归为"风格"，觉得只要代码能跑就行。但在真实项目中，命名直接影响三件事：

  • 阅读成本：据 Google 内部统计，工程师阅读代码的时间是编写代码的 10 倍以上。名字越差，阅读越贵。
  • 搜索效率：全局搜索 getUserById 能立刻定位；搜索 get 会命中几百个无关方法。
  • 重构安全性：含义模糊的名字在重构时很难判断能否安全修改，导致"不敢动"的遗留代码越堆越多。
本质：命名是一次强制的需求澄清。当你发现某个变量很难命名，通常是因为它承担了多个职责，或者需求本身还不清晰。

二、见名知意：名字应该回答"它是什么"

见名知意的核心是：读者不需要打开实现，就能从名字猜出它的职责、类型和用途。

反例 1：无意义的缩写和单字母

Java — 反例
public List<User> getU(int t, String s) {
    List<User> res = new ArrayList<>();
    for (User u : db.all()) {
        if (u.getType() == t && u.getStatus().equals(s))
            res.add(u);
    }
    return res;
}

问题：t 是什么类型？s 是什么状态？调用方必须跳进来才能理解。getU 完全无法搜索，IDE 重命名时也容易误伤。

Java — 正例
public List<User> findActiveUsersByType(UserType userType, UserStatus status) {
    return userRepository.findAll().stream()
        .filter(user -> user.getType() == userType
                     && user.getStatus() == status)
        .collect(Collectors.toList());
}

调用处 findActiveUsersByType(UserType.PREMIUM, UserStatus.ACTIVE) 本身就是文档，无需注释。

反例 2：布尔变量名不表达真/假的含义

Java — 反例 vs 正例
// 反例：flag 代表什么？true 意味着什么？
boolean flag = user.checkPermission();
if (flag) { ... }

// 正例：用 isXxx / hasXxx / canXxx 前缀，语义自明
boolean hasEditPermission = user.hasPermission(Permission.EDIT);
if (hasEditPermission) { ... }

反例 3：数字魔法值

Java — 反例 vs 正例
// 反例：3 是什么？为什么是 3？
if (order.getStatus() == 3) {
    sendShippingNotice();
}

// 正例：枚举让状态机清晰可读
if (order.getStatus() == OrderStatus.SHIPPED) {
    sendShippingNotice();
}
生产坑：数据库里存的是 status=3，但 3 代表什么，六个月后没人记得。枚举 + 数据库映射是标准解法，不要用裸整数表示状态。

三、一致性：同一概念用同一个词

一致性命名的核心规则：同一个业务概念，在整个项目（Controller / Service / Repository / 数据库）中只用一个词，不混用同义词。

同义词混用是最常见的一致性破坏

业务概念混用反例（同一项目内）统一正例
获取单个对象 getUser / fetchOrder / retrieveProduct 统一用 getXxx
获取列表 listUsers / queryOrders / findProducts 统一用 listXxx 或 findXxx
删除 deleteUser / removeOrder / destroyProduct 统一用 deleteXxx
用户租户ID tenantId / orgId / companyId 统一用 tenantId

反例：同一项目中的三种"获取用户"

Java — 反例（Controller / Service / Repository 各说各话）
// UserController.java
public UserVO fetchUser(Long userId) { ... }

// UserService.java
public User   retrieveUserById(Long id) { ... }

// UserRepository.java
Optional<User> getOne(Long uid) { ... }
Java — 正例（垂直贯通，一词到底）
// UserController.java
public UserVO       getUserById(Long userId) { ... }

// UserService.java
public User         getUserById(Long userId) { ... }

// UserRepository.java
Optional<User>     findById(Long userId) { ... }
// Repository 层用 Spring Data 的 findById 惯例，Service 向上统一为 getUserById

一致的好处是：在任意一层全局搜索 getUserById，就能看到完整调用链，不需要靠 IDE 的调用图。

一致性还要延伸到参数命名

Java — 反例 vs 正例
// 反例：同一个"用户ID"，三种写法
public Order createOrder(Long uid,  Long productId) { ... }
public void  cancelOrder(Long user, Long orderId) { ... }
public List<Order> listOrders(Long userId) { ... }

// 正例：统一用 userId
public Order createOrder(Long userId, Long productId) { ... }
public void  cancelOrder(Long userId, Long orderId) { ... }
public List<Order> listOrders(Long userId) { ... }

四、可搜索：名字要能被全局 grep 到

可搜索性解决的是问题定位：线上报警了，你要能在几十万行代码里快速找到出事的地方。

反例：无法搜索的通用词

Java — 反例
// 方法名太泛，搜 "process" 会命中数百个方法
public void process(Object data) { ... }
public void handle(Event  e)    { ... }
public void execute(Task  t)   { ... }

// 常量也不行，搜 "MAX" 会命中所有系统常量
static final int MAX = 100;
Java — 正例
// 加上业务前缀，全局搜索唯一命中
public void processPaymentRefund(RefundRequest refundRequest) { ... }
public void handleOrderCreatedEvent(OrderCreatedEvent event) { ... }
public void executeInventoryDeduction(DeductionTask task) { ... }

// 常量加上领域前缀
static final int ORDER_ITEM_MAX_QUANTITY = 100;

可搜索性与日志的关系

可搜索性不只是代码层面的，还关系到日志。在关键流程里打印包含方法名或业务 ID 的日志，才能在海量日志中快速定位。

Java — 日志可搜索正例
// 反例：日志没有上下文，无法搜索
log.info("开始处理");
log.info("处理完成");

// 正例：包含业务 ID + 操作名，grep orderId=xxx 立刻找到链路
log.info("[processPaymentRefund] start, orderId={}, refundAmount={}",
         refundRequest.getOrderId(), refundRequest.getAmount());
log.info("[processPaymentRefund] done,  orderId={}, refundId={}",
         refundRequest.getOrderId(), result.getRefundId());

五、完整业务场景：支付退款流程的命名示范

以一个真实的"发起退款"接口为例，展示从 Controller 到 Repository 全链路的一致 + 可搜索命名。

PaymentController
  └── initiateRefund(RefundRequest) ← 动词+名词，表达意图
      └── PaymentService.initiateRefund(RefundRequest)
          ├── validateRefundEligibility(order) ← 校验：validate前缀
          ├── calculateRefundAmount(order) ← 计算：calculate前缀
          ├── RefundRepository.saveRefundRecord() ← 持久化：save前缀
          └── PaymentGateway.submitRefundToGateway() ← 外调：submit前缀
Java — 退款流程（命名示范）
@Service
public class PaymentService {

    /**
     * 发起退款。校验资格 → 计算金额 → 落库 → 提交网关。
     * 方法名清晰表达"做什么"，调用方无需读实现。
     */
    public RefundResult initiateRefund(RefundRequest refundRequest) {
        // 1. 校验：方法名用 validate 前缀，说明这是守卫型逻辑
        validateRefundEligibility(refundRequest);

        // 2. 计算：calculate 前缀表明有业务规则运算
        BigDecimal refundAmount = calculateRefundAmount(refundRequest);

        // 3. 构建领域对象，字段名与 RefundRequest 保持一致
        RefundRecord refundRecord = RefundRecord.builder()
            .orderId(refundRequest.getOrderId())
            .refundAmount(refundAmount)
            .status(RefundStatus.PENDING)   // 枚举，不用 0/1/2
            .build();

        // 4. 落库：save 前缀是 Spring Data 惯例，全项目统一
        refundRepository.saveRefundRecord(refundRecord);

        // 5. 外调：submit 表明是向外部系统提交，而不是内部调用
        return paymentGateway.submitRefundToGateway(refundRecord);
    }
}

七、生产环境常见坑

坑 1：DTO / Entity / VO 字段命名不一致导致映射 bug。 比如 Entity.userId 在 DTO 里叫 uid，MapStruct / BeanUtils.copyProperties 静默忽略，数据丢失却无报错。全链路统一字段名，或在 MapStruct 中显式声明 @Mapping。
坑 2：数据库列名与 Java 字段名不匹配。 数据库 create_time，Java 字段 createDate，MyBatis 不报错但映射为 null。约定：数据库 snake_case，Java camelCase，通过 MyBatis 的 mapUnderscoreToCamelCase=true 统一转换。
坑 3：接口返回字段随意缩写，前后端理解不一致。 后端返回 amt，前端以为是 amount，另一个前端以为是 amountType。API 字段应完整拼写，配合 Swagger 文档注释，不省略元音。
坑 4：方法名动词时态错误引起误解。 orderCreated() 看起来像事件回调（已创建），实际是"创建订单"的动作方法。动作方法用动词原形：createOrder()；事件回调用过去式或 on 前缀：onOrderCreated()。
实践建议：在团队 Wiki 维护一份《领域词汇表》，把核心业务概念（用户、订单、商品、退款……）和对应的英文单词固定下来，所有人从这张表取词，杜绝同义词混用。

小结

  • 见名知意：名字应回答"它是什么、能做什么"，避免单字母、无意义缩写和魔法数字。
  • 一致性：同一概念在项目全链路（Controller → Service → Repository → DB）只用一个词，建立领域词汇表统一取词。
  • 可搜索：加业务前缀让名字全局唯一，日志也要带业务 ID，线上排查才能快速定位。
  • 命名困难本身是信号：说明这段代码职责不清晰，先理清职责再命名。

与其他知识点的联系

  • 函数（下一篇）：命名困难往往是函数设计出了问题的信号——如果一个函数很难起一个"见名知意"的名字，很可能是因为它同时做了好几件事，职责不单一。反过来说，把函数拆到"只做一件事"之后，命名反而会变得容易，这两个知识点其实是同一个"职责是否清晰"问题的两个侧面。
  • 注释：好的命名是消灭大部分注释的第一道防线。当你想给一段代码加注释解释"这是干什么的"时，先问自己能不能换一个更好的变量名或方法名来表达，而不是依赖注释——命名做得好，注释章节里说的"废话注释""解释是什么的注释"这类坏味道会自然减少。
  • DTO / Entity 分层实践：本文提到的字段命名一致性问题（userId vs uid），在 Spring 项目的 Controller/Service/Repository 分层和 DTO/Entity 转换中最容易踩坑——这和后面"对象与数据结构"一文讨论的 DTO/VO/Entity 职责划分是同一套工程实践的不同角度：分层越多，命名一致性的价值就越大。
  • 设计模式中的"意图导向命名"：像 validateXxx、calculateXxx、submitXxx 这类前缀约定，本质上是把设计模式（策略、模板方法等）里"角色分工"用命名规则固化下来，让读者不需要看类图就能猜到某个方法在整体流程中扮演的角色。
  • Code Review 与团队协作：领域词汇表（Ubiquitous Language）的概念来自 DDD（领域驱动设计），命名一致性差的项目往往是因为团队没有统一业务语言，Code Review 时可以把"命名是否符合词汇表"作为一项具体、可执行的检查点，而不是含糊地说"命名不好"。
