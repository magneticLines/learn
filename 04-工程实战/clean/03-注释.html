注释——好注释 vs 坏注释

注释不是越多越好。每一行注释都是一次代码表达能力不足的承认——先想办法让代码自述，注释是最后的手段。

注释是代码失败的标志

这句话听起来极端，但背后逻辑成立：注释无法被编译器验证，会随代码演进而腐烂。代码变了，注释常常没人更新，最终留下误导性的谎言。

Martin 的核心观点：与其花时间写注释解释烂代码，不如花时间把代码写好。好的命名、短小的函数、清晰的结构，可以替代 90% 的注释。

现实中最危险的注释：一段写了详细说明的注释，但对应的代码逻辑三个月前已经改了。读者相信了注释，产生了错误的心智模型，引发了难以排查的 bug。

好注释：值得保留的几类

1. 法律注释

版权声明、许可协议——不得不写，有工具自动生成，放在文件顶部。

Java — 法律注释
// Copyright (c) 2024 Example Corp. All rights reserved.
// Licensed under the Apache License, Version 2.0

2. 意图注释——解释"为什么"，而非"是什么"

当代码做了一个反直觉的决策，注释解释背后的原因是有价值的。

Java — 意图注释
// 使用 LinkedList 而非 ArrayList：此队列有频繁的头部删除操作，
// O(1) 的头部移除比 ArrayList 的 O(n) 移动要高效得多
Queue<Task> taskQueue = new LinkedList<>();

// 故意不加锁：此方法只在单线程初始化阶段调用，
// 加锁只会增加开销，不提供任何安全保证
private void initCache() { ... }

3. 警示注释

警告后来的维护者某些重要的约束或陷阱。

Java — 警示注释
// 警告：SimpleDateFormat 不是线程安全的，
// 不要将此实例共享给多个线程
private final SimpleDateFormat formatter = new SimpleDateFormat("yyyy-MM-dd");

4. TODO 注释

标记已知的遗留问题或未来要做的改进，需要配合 IDE 的 TODO 追踪功能使用。但 TODO 不是烂代码的借口——记得定期清理。

Java — TODO 注释
// TODO: 当前用线性扫描，数据量大时需改为索引查询 — @sheayin 2024-01
return orders.stream().filter(o -> o.getId().equals(id)).findFirst();

坏注释：应当删除的几类

1. 废话注释——重复代码已经说清楚的内容

Java — 废话注释（反例）
// 获取用户名               // 函数名已经说明了
public String getUsername() {
    return this.username;  // 返回用户名  // 与代码完全重复
}

// i 加 1
i++;

2. 误导性注释

注释描述的行为与代码实际行为不一致——这比没有注释更糟糕。

Java — 误导性注释（反例）
// 如果用户已登录则返回 true
public boolean checkUser(String token) {
    // 实际上还会刷新 token 过期时间——副作用没有在注释中提及！
    tokenStore.refresh(token);
    return tokenStore.isValid(token);
}

3. 注释掉的代码——最危险的坏味道

注释掉的代码让所有人都不敢删除——万一还有用呢？于是它像幽灵一样永远留在代码库里，腐烂，积累，污染视野。

解决方案：直接删除。我们有版本控制系统（Git），任何删除的代码都可以通过 git log 找回。注释掉代码是对 Git 的不信任。

4. 位置标记注释

Java — 位置标记（反例）
// ////////////////// Actions //////////////////
// ================== Utils ==================

这类注释说明类本身太大了——应该拆分类，而不是用注释标记分区。

好注释 vs 坏注释速查

类型保留还是删除原因
法律/版权声明保留合规需要
解释"为什么"的意图注释保留代码无法表达的决策原因
警示重要陷阱保留防止后人踩坑
TODO（有跟踪）保留明确标记技术债
重复代码的废话删除噪音，不增加信息
注释掉的旧代码删除Git 已经记录历史
解释"是什么"的注释重构代码用更好的命名替代
误导性注释立即删除比没有注释更危险
写注释前先问自己：能不能通过更好的命名、更小的函数、更清晰的结构来消除这个注释的必要性？如果可以，先重构代码。

与其他知识点的联系

  • 命名与函数：本文开篇就说"好注释是代码表达能力不足的补偿"，这直接呼应前两篇——大部分"解释是什么"的坏注释，根源都是命名不够清楚或函数职责不单一。与其花时间写注释说明 process(Object data) 到底处理了什么，不如把方法名改成 processPaymentRefund(RefundRequest)，这才是治本的做法。
  • 对象与数据结构（下一篇）：意图注释的价值在于解释"为什么这么设计"，而这类决策往往发生在对象建模阶段——比如为什么某个字段用 Optional 包装、为什么某个类要设计成不可变的值对象。好的对象设计本身能减少很多需要注释解释的"为什么"，因为设计意图已经体现在类型结构里。
  • 错误处理：警示注释常用于提醒并发安全、资源释放等陷阱，这和错误处理里"用异常上下文说明意图"是同一个思路的两种手段——注释是给人看的静态提示，异常消息是运行时给排查者的动态提示，两者都在解决"如何把隐藏的风险显式表达出来"这个问题。
  • 版本控制与 Git 工作流："注释掉的代码"这类坏味道之所以能被安全删除，根本原因是 Git 已经完整记录了历史——这提醒我们，代码整洁的很多实践都建立在"版本控制系统足够可靠"这个前提上，不需要用注释当备份手段。
  • TODO 与技术债管理：TODO 注释如果不配合 IDE 的追踪功能或工单系统定期清理，最终会和"位置标记注释"一样变成噪音——这也是为什么很多团队要求 TODO 必须关联工单号，把注释里的技术债转化为可跟踪、可排期的工程任务。
