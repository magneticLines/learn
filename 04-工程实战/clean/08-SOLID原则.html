SOLID 原则

SOLID 是五个面向对象设计原则的首字母缩写，由 Robert C. Martin 整理提出。它们不是规则，而是权衡指南——帮助你在变化面前写出更容易维护的代码。

字母原则名称一句话
SSingle Responsibility一个类只有一个变化的理由
OOpen/Closed对扩展开放，对修改关闭
LLiskov Substitution子类可以替换父类而不破坏程序
IInterface Segregation不强迫客户依赖它不使用的方法
DDependency Inversion高层模块不依赖低层模块，都依赖抽象

S 单一职责原则（SRP）

一个类应该只有一个变化的理由。如果一个类会因为两个不同的业务原因而变化，说明它承担了两个职责，应当拆分。

违反 → 问题 → 修正

Java — SRP 违反
// 违反：Report 同时负责业务数据计算 和 输出格式，两个职责
public class Report {
    public String calculateSummary() { ... }  // 业务逻辑
    public void printToPdf() { ... }           // 输出格式
    public void sendByEmail() { ... }          // 发送渠道
}
Java — SRP 修正
// 修正：每个类只有一个职责
public class Report {
    public String calculateSummary() { ... }  // 只负责业务逻辑
}
public class PdfReportRenderer {
    public void render(Report report) { ... }   // 只负责 PDF 输出
}
public class ReportEmailSender {
    public void send(Report report) { ... }     // 只负责发送
}

O 开闭原则（OCP）

软件实体应该对扩展开放，对修改关闭。新增功能不应该改动已有的稳定代码，而是通过扩展来实现。

Java — OCP 违反 → 修正
// 违反：每增加一种支付方式，都要修改 processPayment
public void processPayment(String type, double amount) {
    if (type.equals("alipay"))  processAlipay(amount);
    else if (type.equals("wechat")) processWechat(amount);
    // 新增银联？要改这里 ← 违反 OCP
}

// 修正：用多态替代条件分支——扩展只需新增实现类
public interface PaymentProcessor {
    void process(double amount);
}
public class AlipayProcessor  implements PaymentProcessor { ... }
public class WechatProcessor implements PaymentProcessor { ... }
// 新增银联：只要加 UnionPayProcessor，不动任何已有代码

L 里氏替换原则（LSP）

子类必须能够替换父类，且不改变程序的正确性。如果某处使用了父类，把它换成子类后程序行为应该一致。

LSP 的经典违反案例是"正方形继承矩形"：数学上正方形是矩形，但在代码中，Square.setWidth() 需要同时改高度，违反了 Rectangle 的约定，导致使用 Rectangle 的地方无法安全地用 Square 替换。

Java — LSP 违反示例
public class Rectangle {
    protected int width, height;
    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(int w) {
        this.width = w;
        this.height = w;  // 违反父类约定：setWidth 不应该改 height
    }
}

// 使用父类的代码会被 Square 破坏：
Rectangle r = new Square();
r.setWidth(5);
r.setHeight(3);
// 期望面积 = 15，实际 = 9（height 被 setWidth 覆盖了）
LSP 提示："IS-A" 关系在数学上成立，不代表在代码继承上成立。优先用组合而非继承，能避免大多数 LSP 问题。

I 接口隔离原则（ISP）

不应强迫客户依赖它不使用的方法。一个大而全的接口，应拆分成多个小而专注的接口。

Java — ISP 违反 → 修正
// 违反：Bird 接口包含飞行方法，导致企鹅被迫实现无法完成的方法
interface Bird {
    void fly();
    void eat();
    void swim();
}
class Penguin implements Bird {
    public void fly() { throw new UnsupportedOperationException(); }  // 被迫
}

// 修正：接口按能力拆分，企鹅只实现它拥有的能力
interface Flyable  { void fly(); }
interface Swimmable { void swim(); }
interface Eatable  { void eat(); }

class Eagle   implements Flyable, Eatable   { ... }
class Penguin implements Swimmable, Eatable { ... }

D 依赖倒置原则（DIP）

高层模块不应依赖低层模块，两者都应依赖抽象。抽象不应依赖细节，细节应依赖抽象。

这正是 Spring 依赖注入的设计哲学：Service 依赖 Repository 接口（抽象），而不是 JpaUserRepository（具体实现）。

Java — DIP 违反 → 修正
// 违反：OrderService（高层）直接依赖 MySqlOrderRepository（低层具体实现）
public class OrderService {
    // new 了具体实现，无法换实现，也无法 mock 测试
    private final MySqlOrderRepository repo = new MySqlOrderRepository();
}

// 修正：OrderService 依赖抽象接口，具体实现由外部注入
public class OrderService {
    // 依赖抽象——高层/低层都依赖这个接口
    private final OrderRepository repo;

    // 构造注入（Spring 推荐方式）：具体实现由 IoC 容器决定
    public OrderService(OrderRepository repo) {
        this.repo = repo;
    }
}

// 可以有多个实现，Service 不感知
public class MySqlOrderRepository  implements OrderRepository { ... }
public class InMemoryOrderRepository implements OrderRepository { ... } // 测试用
SOLID 与 Spring 的关系：Spring 框架的核心机制——IoC 容器（DIP）、AOP（OCP）、接口优先（ISP）——本质上都是 SOLID 原则的工程化实践。理解 SOLID，就理解了 Spring 为什么这么设计。
