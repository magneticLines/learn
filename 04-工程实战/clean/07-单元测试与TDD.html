单元测试与 TDD——FIRST 原则

测试不仅是验证工具，更是设计工具。先写测试，会迫使你从使用者角度思考接口设计，往往比先写实现更能产出简洁的 API。

TDD 的红绿重构循环

TDD（Test-Driven Development，测试驱动开发）的核心是一个三步循环，周期通常以分钟为单位：

RED
写一个失败的测试
→
GREEN
写最少代码让测试通过
→
REFACTOR
消除重复，优化设计
  • RED：先写测试，此时测试必须失败（如果测试一开始就通过，说明没有测试到任何东西）
  • GREEN：只写让测试通过所需的最少代码，不要超前实现
  • REFACTOR：在测试保护下安全地改善代码结构，不改变行为
TDD 的深层价值不是测试覆盖率，而是设计反馈。如果一个类难以测试（需要大量 mock、setUp 复杂），说明它的设计有问题——职责太多、耦合太高。测试是最直接的设计批评者。

FIRST 原则

好的单元测试必须满足 FIRST 原则，否则测试本身会成为负担：

字母含义做法
Fast快速单元测试不访问数据库、文件系统、网络；毫秒级完成
Independent独立测试之间无依赖，任意顺序运行结果一致
Repeatable可重复在任何环境（本地/CI/生产）运行结果一致，无随机性
Self-validating自验证用断言表达预期，通过/失败明确，不需要人工看日志
Timely及时和生产代码同步写，不是上线前补测

JUnit 5 实战示例

用一个完整的 TDD 示例展示红绿重构的完整循环：场景是实现一个密码强度校验器。

第一步：RED — 先写测试

Java — JUnit 5 + AssertJ 测试
class PasswordValidatorTest {

    private PasswordValidator validator;

    @BeforeEach
    void setUp() {
        validator = new PasswordValidator();
    }

    @Test
    void validate_shouldFail_whenPasswordTooShort() {
        // 测试名即文档：场景_预期结果_触发条件
        assertThatThrownBy(() -> validator.validate("abc"))
            .isInstanceOf(WeakPasswordException.class)
            .hasMessageContaining("too short");
    }

    @Test
    void validate_shouldPass_whenPasswordMeetsAllCriteria() {
        // 不抛异常即为通过
        assertThatCode(() -> validator.validate("Str0ng!Pass"))
            .doesNotThrowAnyException();
    }

    @ParameterizedTest
    @ValueSource(strings = {"nodigit", "12345678", "nouppercase1"})
    void validate_shouldFail_forVariousWeakPasswords(String weakPwd) {
        // 参数化测试：用一个测试方法覆盖多种输入
        assertThatThrownBy(() -> validator.validate(weakPwd))
            .isInstanceOf(WeakPasswordException.class);
    }
}

第二步：GREEN — 最少代码让测试通过

Java — 最小实现
public class PasswordValidator {
    public void validate(String password) {
        if (password.length() < 8)
            throw new WeakPasswordException("Password too short");
        if (!password.matches(".*\\d.*"))
            throw new WeakPasswordException("Must contain a digit");
        if (!password.matches(".*[A-Z].*"))
            throw new WeakPasswordException("Must contain uppercase");
    }
}

测试替身：Mock / Stub / Fake

单元测试要求隔离被测对象，不能依赖真实的数据库、网络等。测试替身（Test Double）是替代真实依赖的对象：

类型定义使用场景
Stub返回固定答案的假对象控制依赖的返回值，不关心调用次数
Mock可以验证交互的假对象验证某个方法是否被调用了、被调用了几次
Fake有简单实现的替代品内存数据库、内存消息队列，行为接近真实
Spy真实对象 + 可以监听调用在真实实现上验证方法调用情况
Java — Mockito Mock 示例
@Test
void registerUser_shouldSendWelcomeEmail() {
    // Mock：用 Mockito 创建假的 EmailService
    EmailService mockEmail = mock(EmailService.class);
    UserRepository stubRepo = mock(UserRepository.class);
    // Stub：让 save 返回我们指定的 user 对象
    when(stubRepo.save(any())).thenReturn(new User(1L, "test@example.com"));

    UserService service = new UserService(stubRepo, mockEmail);
    service.registerUser(new UserDto("test@example.com", "Str0ng!Pass"));

    // 验证 send 方法被调用了一次（Mock 的核心价值）
    verify(mockEmail, times(1)).send(eq("test@example.com"), any(), any());
}
过度 Mock 是反模式：如果一个测试需要 mock 5 个以上的依赖，说明被测类职责太多——先拆分类，再写测试。测试难写往往是设计问题的诊断，而不是要写更聪明的测试。

与其他知识点的联系

  • 边界（上一篇）：学习性测试和边界层的 Wrapper 接口是配套实践——先写学习性测试验证你对第三方库行为的理解，再把这些理解沉淀到 Wrapper 接口的设计里；反过来，边界层接口一旦定义好，业务代码的单元测试就可以直接 mock 这个接口，不需要真实的第三方依赖，这正是 FIRST 原则里"Fast"和"Independent"的实现基础。
  • 函数设计："只做一件事"的函数天然更好测——本文的 PasswordValidator.validate 就是纯函数式的校验逻辑，不涉及外部依赖，写测试几乎不需要任何 mock。如果你发现一个方法的测试需要准备大量数据、mock 很多依赖，往回看几乎总能发现函数一文里提到的"职责过多"问题，这也是"过度 Mock 是反模式"这条提示的根源。
  • 错误处理：本文测试用例大量使用 assertThatThrownBy(...).isInstanceOf(WeakPasswordException.class)，这依赖错误处理一文里"用自定义异常类型 + 有意义的消息表达失败场景"的设计——如果 validate 用错误码或抛泛化的 RuntimeException，测试断言会变得模糊，无法精确验证是哪种失败原因触发的。
  • 设计模式与依赖注入：Mock/Stub/Fake 之所以能替换真实依赖，前提是被测类通过构造函数或 Setter 注入依赖（依赖注入），而不是在内部用 new 直接创建——这也是 Spring 依赖注入的核心价值之一：不仅是为了框架管理生命周期，更是为了让类天然具备可测试性。
  • Code Review 与持续集成：FIRST 原则里的 Repeatable（可重复）和 CI 流水线直接相关——如果测试依赖真实时间、随机数或外部网络，在 CI 环境里跑起来会时不时失败（Flaky Test），拖慢整个团队的交付节奏，这也是为什么强调单元测试要 mock 掉一切不确定性来源。
