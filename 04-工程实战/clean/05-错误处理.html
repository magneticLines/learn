错误处理——异常优于错误码

错误处理是业务逻辑的一部分，但它不应该淹没业务逻辑本身。异常机制的设计，正是为了把"正常流程"和"异常流程"在代码层面分离开来。

异常 vs 错误码的本质区别

错误码是 C 语言时代的产物——当时没有异常机制，只能用返回值表示成功/失败。在有异常机制的语言（Java、C#、Python）里，继续用错误码会带来一系列问题：

Java — 错误码反例
// 错误码风格：业务逻辑被 if 检查淹没，主流程难以看清
int result = userService.login(username, password);
if (result == -1) {
    handleUserNotFound();
} else if (result == -2) {
    handlePasswordWrong();
} else if (result == -3) {
    handleAccountLocked();
} else {
    proceedWithSession();
}
Java — 异常风格正例
// 异常风格：主流程清晰，错误处理被隔离到 catch 块
try {
    User user = userService.login(username, password);
    proceedWithSession(user);
} catch (UserNotFoundException e) {
    handleUserNotFound(e);
} catch (InvalidPasswordException e) {
    handlePasswordWrong(e);
} catch (AccountLockedException e) {
    handleAccountLocked(e);
}

异常的核心优势：强制调用者处理错误（编译器/运行时保证）、错误类型携带完整上下文信息、主流程与错误处理逻辑在视觉上分离。

受检异常（Checked Exception）的争议

Java 特有的受检异常（必须在方法签名 throws 或用 try-catch 处理的异常）在 Clean Code 中饱受批评。Martin 认为它违反了开闭原则：

  • 底层方法新增一个受检异常，调用链上的每一层方法签名都要修改
  • 大量的 throws Exception 透传让签名毫无意义
  • C#、Python、Kotlin 等后续语言都放弃了受检异常
实践建议：在应用层代码（Service/Controller）优先使用 RuntimeException 的子类。受检异常仅在明确要求调用者必须处理的底层 API（如 IO、JDBC）中使用。Spring 框架本身也将大多数 JDBC 异常转换为 DataAccessException（非受检）正是此原因。

别返回 null，别传 null

返回 null 是将错误处理的责任扔给调用者——每一个调用者都要写 null 检查，忘一个就 NPE（NullPointerException）。

Java — 返回 null 的危险与替代方案
// 差：调用者必须记得检查 null，忘了就 NPE
public User findUser(Long id) {
    return userRepository.findById(id).orElse(null);  // 不要这样做
}

// 好选项 1：抛出业务异常，让调用者知道发生了什么
public User getUserOrThrow(Long id) {
    return userRepository.findById(id)
        .orElseThrow(() -> new UserNotFoundException(id));
}

// 好选项 2：返回 Optional，强迫调用者思考"找不到"的情况
public Optional<User> findUser(Long id) {
    return userRepository.findById(id);
}

同理，不要把 null 传给方法。如果一个参数可能是 null，意味着函数本身需要针对 null 写特殊分支——这是隐藏的 flag 参数。用方法重载或 Optional 参数替代。

用异常上下文说明意图

抛出异常时，消息要携带足够的上下文，让日志中的一行错误记录就能定位问题，不必再去 debug。

Java — 好异常 vs 坏异常
// 差：没有任何上下文，日志里只能看到 "User not found"
throw new RuntimeException("User not found");

// 好：自定义异常类 + 详细上下文，可以根据 userId 直接定位
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(Long userId) {
        super(String.format("User not found: id=%d", userId));
    }
}

// 调用处：异常名本身就是意图说明
throw new UserNotFoundException(userId);
面试高频：受检异常和非受检异常的区别？继承 Exception 是受检（必须处理），继承 RuntimeException 是非受检。Spring 为什么大量使用 RuntimeException？因为受检异常破坏调用链的封装性，迫使每一层都暴露底层细节。

与其他知识点的联系

  • 函数设计：本文"用异常隔离正常流程和错误流程"的思路，和函数一文里"抽象层级一致"的原则是一致的——try 块里应该只保留高层业务步骤，具体的错误分支逻辑下沉到各个 catch 块或专门的异常处理方法里，不要让一堆 if (result == -1) 式的错误码判断打断主流程的阅读节奏。
  • 对象与数据结构：自定义异常类本身就是一种小型的领域对象——UserNotFoundException(userId) 把"谁没找到""为什么没找到"这些上下文封装在异常内部，调用者不需要自己拼接错误信息，这和"对象封装行为、隐藏细节"的思路完全一致，只是这里封装的是"失败场景"而不是"成功场景"的业务逻辑。
  • 单元测试与 TDD（下一篇）：用异常代替错误码后，测试也更好写——assertThatThrownBy(...).isInstanceOf(UserNotFoundException.class) 直接断言异常类型即可，不需要判断魔法数字返回值；而错误码风格的测试往往要记住"-1 代表什么"，测试代码本身就不清晰。设计良好的异常体系是可测试性的前提之一。
  • Spring 的 @ControllerAdvice 统一异常处理：本文提到 Service 层应该优先抛出非受检异常，这正是为了配合 Spring MVC 里 @RestControllerAdvice + @ExceptionHandler 的统一异常处理机制——业务代码只管抛异常表达意图，具体转换成什么 HTTP 状态码、什么响应体，集中在一处处理，Controller 和 Service 都不需要写重复的 try-catch。
  • 设计模式中的异常层次设计：Java 的异常继承体系本身就是"is-a"关系的应用——把业务异常都继承自统一的 BusinessException 基类，再按场景细分子类（UserNotFoundException、InvalidPasswordException），这样 @ExceptionHandler(BusinessException.class) 可以一次性兜底处理所有业务异常，是继承与多态在异常设计上的典型应用。
