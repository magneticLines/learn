函数：短小 / 只做一件事 / 抽象层级一致

函数是代码可读性的最小单元。写好函数，是写出可维护代码的第一道门槛。

为什么函数质量如此重要

读代码的时间远多于写代码的时间。Robert C. Martin 在《Clean Code》中统计，开发者花在读代码上的时间与写代码之比约为 10:1。 函数是阅读代码时最常打开的"抽屉"——如果每个抽屉里都塞满杂物，整个项目就会让人望而生畏。

真实项目中，函数质量差带来的代价非常具体：新人上手慢、改一处出一堆 Bug、单元测试写不下去、Code Review 变成扯皮。 这三条原则——短小、只做一件事、抽象层级一致——不是美学偏好，而是可维护性的工程基础。

原则一：短小

"短小"不是让你死抠行数，而是让函数只包含完成一件事所必需的逻辑。 Clean Code 建议函数体控制在 5–20 行，超过 30 行就值得重新审视。

反例：一个拥有 60 行的"上帝方法"

OrderService.java — 反例❌ 过长
public OrderResult submitOrder(Long userId, List<CartItem> items, String couponCode) {

    // 1. 校验用户
    User user = userRepository.findById(userId)
        .orElseThrow(() -> new UserNotFoundException(userId));
    if (!user.isActive()) {
        throw new BusinessException("用户已被禁用");
    }

    // 2. 校验库存
    for (CartItem item : items) {
        Product product = productRepository.findById(item.getProductId())
            .orElseThrow(() -> new ProductNotFoundException(item.getProductId()));
        if (product.getStock() < item.getQuantity()) {
            throw new InsufficientStockException(product.getName());
        }
    }

    // 3. 计算金额
    BigDecimal total = BigDecimal.ZERO;
    for (CartItem item : items) {
        total = total.add(item.getPrice().multiply(new BigDecimal(item.getQuantity())));
    }
    if (couponCode != null && !couponCode.isBlank()) {
        Coupon coupon = couponRepository.findByCode(couponCode)
            .orElseThrow(() -> new InvalidCouponException(couponCode));
        total = total.subtract(coupon.getDiscountAmount());
    }

    // 4. 创建订单 + 扣库存 + 发通知（还有更多...）
    Order order = new Order(userId, items, total);
    orderRepository.save(order);
    items.forEach(item -> productRepository.decreaseStock(item.getProductId(), item.getQuantity()));
    notificationService.sendOrderConfirmation(user.getEmail(), order);

    return new OrderResult(order);
}
问题：这个函数同时处理了用户校验、库存校验、金额计算、优惠券应用、订单持久化、库存扣减、通知发送——至少 7 件事。 当促销逻辑变化时，你不得不在这 60 行里找到对应位置，同时冒着改坏其他逻辑的风险。

正例：拆分后，每个函数只做一件事

OrderService.java — 正例✅ 短小清晰
public OrderResult submitOrder(Long userId, List<CartItem> items, String couponCode) {
    User       user  = validateUser(userId);
                        validateStock(items);
    BigDecimal total = calculateTotal(items, couponCode);
    Order      order = createOrder(user, items, total);
                        notifyUser(user, order);
    return new OrderResult(order);
}

// 每个私有方法只做一件事，可独立测试、独立修改
private User validateUser(Long userId) {
    User user = userRepository.findById(userId)
        .orElseThrow(() -> new UserNotFoundException(userId));
    if (!user.isActive()) throw new BusinessException("用户已被禁用");
    return user;
}

private BigDecimal calculateTotal(List<CartItem> items, String couponCode) {
    BigDecimal subtotal = sumItemPrices(items);
    return applyCoupon(subtotal, couponCode);
}
阅读体验的本质改变：拆分后，submitOrder 读起来像一段业务流程描述，而不是实现细节的堆砌。 读者可以先理解整体流程，再按需深入某个步骤——这就是好的抽象。

原则二：只做一件事（单一职责）

判断一个函数是否"只做一件事"，有一个实用标准：你能否用"并且"来描述它做的事？ 如果可以——"它校验参数并且发送邮件并且写日志"——那它就做了不止一件事。

函数名是最直接的信号：如果你需要在函数名里加 And、OrElse、AndThen，几乎可以确定要拆分了。

反例：函数名暴露了多职责

UserService.java — 反例❌ 多职责
// 函数名里已经出现了"并且"的意思
public void validateAndSendWelcomeEmail(User user) {
    if (user.getEmail() == null) {
        throw new IllegalArgumentException("邮箱不能为空");
    }
    if (!EMAIL_PATTERN.matcher(user.getEmail()).matches()) {
        throw new IllegalArgumentException("邮箱格式不合法");
    }
    // 校验完了，顺便发邮件
    emailService.send(user.getEmail(), "欢迎注册", buildWelcomeContent(user));
}
问题：测试邮箱格式校验时，必须 mock emailService，否则单元测试会真的发邮件。 更糟的是：如果将来某个场景需要"只校验不发邮件"，你根本没法复用这个函数。

正例：职责分离

UserService.java — 正例✅ 单一职责
// 校验：纯逻辑，无副作用，测试极简
private void validateEmail(String email) {
    if (email == null || !EMAIL_PATTERN.matcher(email).matches()) {
        throw new IllegalArgumentException("邮箱格式不合法: " + email);
    }
}

// 发送：独立的副作用，可单独 mock
private void sendWelcomeEmail(User user) {
    emailService.send(user.getEmail(), "欢迎注册", buildWelcomeContent(user));
}

// 调用者决定何时组合
public void registerUser(User user) {
    validateEmail(user.getEmail());
    userRepository.save(user);
    sendWelcomeEmail(user);
}

原则三：抽象层级一致（Step Down 规则）

这是三条原则里最难察觉、也最容易被忽视的一条。 所谓"抽象层级"，指的是代码距离业务意图的远近： 高层级描述"做什么"（业务意图），低层级描述"怎么做"（实现细节）。

Clean Code 提出的 Step Down 规则：一个函数内的所有语句，应处于同一抽象层级。 读者从上到下读代码时，就像沿着楼梯一步一步走下去——每一层的粒度应当一致。

抽象层级对照

层级示例描述风格
高层（业务）submitOrder()校验 → 计算 → 创建 → 通知
中层（领域）calculateTotal()求和商品价格 → 应用优惠券
低层（技术）sumItemPrices()遍历列表 → multiply → add → 返回

反例：混合了高层意图和低层实现

ReportService.java — 反例❌ 层级混乱
public void generateMonthlyReport(YearMonth month) {
    // 高层意图：获取数据
    List<Order> orders = orderRepository.findByMonth(month);

    // 突然插入低层细节：手动拼接 CSV
    StringBuilder csv = new StringBuilder("orderId,amount,status\n");
    for (Order o : orders) {
        csv.append(o.getId()).append(",")
           .append(o.getAmount()).append(",")
           .append(o.getStatus()).append("\n");
    }

    // 高层意图：上传文件
    String filename = "report-" + month + ".csv";
    storageService.upload(filename, csv.toString().getBytes());
}
问题："拼 CSV"是低层细节，它夹在两段高层意图之间，打断了阅读节奏。 以后如果要改成 Excel 格式，你必须在这个"报表生成"函数里修改拼接逻辑——关注点没有分离。

正例：每一层只关注同一粒度

ReportService.java — 正例✅ 层级一致
// 高层：只有业务步骤，读起来像伪代码
public void generateMonthlyReport(YearMonth month) {
    List<Order> orders  = fetchOrders(month);
    byte[]      content = renderAsCsv(orders);
    String      path    = buildReportPath(month);
    storageService.upload(path, content);
}

// 低层：专注 CSV 拼接细节，可独立替换为 Excel 实现
private byte[] renderAsCsv(List<Order> orders) {
    StringBuilder sb = new StringBuilder("orderId,amount,status\n");
    orders.forEach(o -> sb.append(o.getId()).append(",")
        .append(o.getAmount()).append(",")
        .append(o.getStatus()).append("\n"));
    return sb.toString().getBytes(StandardCharsets.UTF_8);
}

函数参数：越少越好

参数数量直接影响函数的认知负担。Clean Code 建议： 零参数最理想，一个次之，两个可以接受，三个就要审慎，超过三个几乎总是需要重构。

参数过多时的两种解法

重构方向参数对象 / Builder
// ❌ 5 个参数，调用时完全看不懂各参数含义
createUser("alice", "alice@example.com", "18888888888", 1, true);

// ✅ 引入参数对象，语义清晰
CreateUserRequest req = CreateUserRequest.builder()
    .username("alice")
    .email("alice@example.com")
    .phone("18888888888")
    .roleId(1)
    .active(true)
    .build();
createUser(req);
特别注意布尔参数：函数接受一个 boolean 参数，几乎意味着函数内部有 if (flag) 分支——这通常意味着该拆成两个函数。 render(true) 毫无语义，而 renderForPrint() 和 renderForScreen() 一目了然。

副作用：让函数行为可预测

副作用指函数除返回值之外，还修改了外部状态（数据库、缓存、文件、成员变量等）。 副作用本身不是坏事，但隐藏的副作用是灾难：读者以为这是一个查询函数，实际上它悄悄改了数据库。

反例：查询函数暗含写操作

SessionService.java — 反例❌ 隐藏副作用
// 函数名叫 "check"，但它悄悄修改了 lastLoginTime
public boolean checkPassword(String username, String password) {
    User user = userRepository.findByUsername(username);
    if (passwordEncoder.matches(password, user.getPasswordHash())) {
        // 调用者完全不知道这里会更新数据库！
        user.setLastLoginTime(LocalDateTime.now());
        userRepository.save(user);
        return true;
    }
    return false;
}

正例：查询与命令分离（CQS 原则）

SessionService.java — 正例✅ 查询与命令分离
// 查询：纯判断，无副作用
public boolean isPasswordCorrect(String username, String password) {
    User user = userRepository.findByUsername(username);
    return passwordEncoder.matches(password, user.getPasswordHash());
}

// 命令：明确表达"我会修改状态"
public void recordLogin(String username) {
    User user = userRepository.findByUsername(username);
    user.setLastLoginTime(LocalDateTime.now());
    userRepository.save(user);
}

// 调用方：流程清晰，副作用在哪里一目了然
public void login(String username, String password) {
    if (!isPasswordCorrect(username, password)) {
        throw new AuthenticationException("用户名或密码错误");
    }
    recordLogin(username);
}

生产环境常见坑

坑一：用注释分段代替拆函数

常见反模式❌ 注释段替代函数
public void process() {
    // ===== Step 1: 初始化 =====
    // ... 30 行代码

    // ===== Step 2: 处理数据 =====
    // ... 40 行代码

    // ===== Step 3: 持久化 =====
    // ... 20 行代码
}

当你发现自己在用注释给函数分"段落"，这是一个强烈信号：每个段落都应该是一个独立函数。 注释段不能被单独测试、不能被复用，而函数可以。

坑二：过度拆分导致追踪困难

拆分是有成本的。函数调用链过深（超过 5 层）时，调试时需要频繁跳转，同样令人痛苦。 原则是：每个函数要有"存在的意义"——如果一个私有函数只被调用一次，且提取后并没有提升可读性，就没有必要提取。

坑三：事务边界被函数拆分破坏

在 Spring 中拆分函数时，要特别注意 @Transactional 的作用范围。 将带有数据库操作的私有方法拆成另一个 public 方法并放在同一个 Bean 中， 如果通过 this.xxx() 调用，事务不会生效（Spring AOP 的自调用问题）。 拆分时要确保事务边界清晰，必要时将子方法放入另一个 Service Bean。

坑四：Stream 链无限拉长

StreamService.java❌ 过长的流水线
// 10 个操作写在一条链上，出错时无法定位
return orders.stream()
    .filter(o -> o.getStatus() == PAID)
    .filter(o -> o.getCreatedAt().isAfter(startDate))
    .map(o -> o.getItems())
    .flatMap(Collection::stream)
    .filter(item -> item.getCategoryId() == targetCategory)
    .mapToLong(OrderItem::getQuantity)
    .sum();

超过 4–5 个操作的 Stream 链可以考虑拆成具名中间变量，或抽取为独立方法，既能提高可读性，也能在 debug 时打断点。

三原则速查

函数设计三原则
├── 短小
│ ├── 目标 5–20 行，超 30 行警觉
│ └── 注释分段 → 信号：可以拆函数
├── 只做一件事
│ ├── 判断标准：函数名能否用"并且"描述
│ ├── 查询与命令分离（CQS）
│ └── 布尔参数 → 通常应拆为两个函数
└── 抽象层级一致（Step Down）
├── 同一函数内的语句粒度保持一致
├── 高层：业务步骤；低层：技术实现
└── 混合层级 → 将细节下沉为子函数
实践建议：每次 Code Review 时，对超过 20 行的函数多问一句——"它做了几件事？每行代码的抽象层级一致吗？" 形成这个习惯，比记住任何规则都有效。
