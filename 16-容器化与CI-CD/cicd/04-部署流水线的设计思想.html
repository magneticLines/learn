部署流水线的设计思想

[补充：《持续交付》Jez Humble & David Farley] 前面两篇已经讲了流水线"有哪些阶段"，这一篇回到更本质的问题：为什么要把发布过程设计成一条"流水线"，而不是一份人工检查清单？

问题的本质：如何回答"这个版本能不能发布"

《持续交付》这本书提出了一个很朴素但很关键的问题：在任意一个时间点，团队能不能立刻、有把握地回答"我们手上这个版本，现在能不能发布到生产环境"？ 在很多团队里，这个问题的答案是"不知道，得等 QA 团队手工测完才知道"——这意味着发布决策依赖的是人的记忆和临时的手工验证，而不是可复现、可追溯的证据。

常见误区：把"部署流水线"简单理解成"CI 工具里配置的那几个 job"。真正的部署流水线是一种设计思想：它要求团队把"从代码提交到生产发布"的整个过程，显式建模成一系列有先后依赖的验证阶段，而不仅仅是"能跑起来的自动化脚本"。

漏斗式设计：每通过一关，信心增加一分

部署流水线把发布过程建模成一个"漏斗"：代码提交时，候选版本（Release Candidate）进入漏斗的最宽处，每经过一个阶段的验证，只有通过的版本才能继续往下走，漏斗逐渐收窄，最终能走到漏斗尽头（生产环境）的，是已经被层层验证过的、有充分信心的版本。

提交阶段（Commit Stage）— 编译 + 单元测试，几分钟内完成，覆盖大部分候选版本
  ↓ 通过才能进入下一阶段
自动化验收测试（Acceptance Stage）— 验证业务功能是否符合预期，耗时更长
  ↓
性能 / 容量测试 — 验证非功能性需求（响应时间、吞吐量）
  ↓
手工探索性测试 / UAT（如需要）— 人工介入，验证自动化难以覆盖的场景
  ↓
生产发布 — 走到这里的版本，每一步都留有验证记录

关键在于：每个阶段既是一道过滤网，也是一份证据。如果某个版本走到了"性能测试"阶段，就说明它已经确定性地通过了编译、单元测试和验收测试——这不是"我觉得应该没问题"，而是有据可查的事实。

为什么"漏斗"比"人工检查清单"更可靠

  • 确定性：自动化阶段的判定结果是可重复的——同一个版本、同一套测试，今天跑和明天跑结果一致。人工检查清单容易受人的状态、经验、疲劳程度影响，结果因人而异。
  • 可见性：团队里任何人在任何时刻都能查看流水线状态，知道当前哪个版本在漏斗的哪个位置、卡在了哪一步——不需要开会问"现在能不能发布"。
  • 快速反馈：漏斗越靠前的阶段执行越快、成本越低（呼应上一篇"快失败"原则），一旦某个阶段失败，流水线立刻停止，不会把有问题的版本继续往后推。
  • 强制单一事实来源：所有环境部署的都是同一个经过流水线验证的制品（Build Once, Deploy Many），而不是每个环境各自重新编译一次——避免"测试环境测过的代码"和"生产环境跑的代码"其实不是同一份二进制的经典陷阱。
面试高频："什么是部署流水线？它解决了什么问题？" 参考答案：部署流水线把从代码提交到生产发布的过程建模成一系列自动化验证阶段，前一阶段通过才能进入下一阶段。它解决的核心问题是让"这个版本能不能发布"这件事有清晰、可追溯、可重复验证的依据，而不是靠人拍脑袋判断。

与发布策略的关系

部署流水线关注的是"发布之前如何验证"，蓝绿部署/金丝雀发布（上一篇）关注的是"发布本身如何执行、如何控制风险"——两者是互补关系。一条设计良好的流水线，走到最后一步时，通常正是触发蓝绿切换或金丝雀放量的时机：流水线负责"确认这个版本值得发"，发布策略负责"把这个版本安全地送到用户面前"。

呼应 15-测试模块：漏斗模型的每一层能不能生效，完全取决于对应的自动化测试写得好不好——单元测试覆盖率不够，"提交阶段"这道关卡就形同虚设；验收测试写得不稳定（Flaky），漏斗中间就会经常"假性堵塞"。部署流水线是把测试的价值放大到整个发布流程的关键机制。

与其他知识点的联系

  • GitHub Actions / Jenkins 流水线设计思路（前篇）：前篇讲的八个阶段是这套"漏斗模型"的具体实现载体——GitHub Actions 里的多个 job、Jenkinsfile 里的多个 stage，本质上都是在用工具语法实现本篇讲的"逐级收窄、层层验证"的设计思想。
  • 特性开关 Feature Toggle（下一篇）：部署流水线解决的是"这份代码值不值得发布"，特性开关解决的是"发布了之后要不要立刻对用户生效"——两者合起来才能既满足持续集成要求的高频合并，又不必因为功能没做完而卡住整条流水线。
  • 蓝绿部署与金丝雀发布（前篇）：文中"与发布策略的关系"一节已经点出：流水线负责漏斗尽头之前的"验证"，发布策略负责漏斗尽头之后的"如何安全地把流量切过去"，两者合起来才是一次完整、低风险的发布。
  • 基础设施即代码 IaC：IaC 一节明确提到，基础设施变更同样要走"检出 → 格式校验 → plan 预览 → 人工审批 → apply"这套漏斗式流水线——本篇讲的"用一系列自动化验证建立信心"的思想，不只适用于应用代码，也适用于服务器和云资源的变更。
  • 与 15-测试模块的呼应（文中已提及）：漏斗每一层能不能真正拦住问题，完全取决于对应的自动化测试写得扎不扎实，这是本篇反复强调的"确定性"和"可追溯性"的根基。
