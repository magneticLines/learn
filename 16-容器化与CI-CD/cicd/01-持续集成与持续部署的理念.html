CI/CD

持续集成与持续部署的理念

CI、CD（交付）、CD（部署）三个概念经常被混着用，但它们代表着"自动化程度"逐级递增的三个台阶。理解这个递进关系，才能看懂团队为什么要投入精力搭流水线。

先看没有自动化时是什么样子

想象一个五人小团队，每个人在自己的分支上开发一两周，到了发布日大家把代码合到一起——这时候往往会爆发一堆"集成冲突"：你改的接口我这边没同步，他新增的字段我这边测试挂了。 这种"合并地狱"不是因为团队菜，而是因为集成的间隔太长，问题堆积到无法一次性消化。

发布环节同样痛苦：手工登录服务器、手工执行部署脚本、手工重启服务，任何一步手滑都可能导致生产事故。久而久之，"发布"变成一件让人如临大敌的大事，团队本能地想少发布、晚发布——这与快速迭代的初衷完全背道而驰。

核心矛盾：软件的价值只有交付给用户才能体现，但人工操作的发布流程天然是慢、且容易出错的。CI/CD 要解决的正是"如何又快又稳地交付"这个矛盾。

持续集成 CI：让集成冲突无处遁形

持续集成（Continuous Integration）的核心实践是：团队成员频繁地（一天至少一次，理想情况下更频繁）把自己的代码合并到主干分支，每次合并都会自动触发一次构建和一轮自动化测试。

关键在"频繁"和"自动化"这两个词。频繁合并意味着每次的改动量都很小，即使产生冲突，也是"两行代码的冲突"而不是"两周工作量的冲突"，修起来轻松得多。自动化构建 + 测试意味着问题在几分钟内就会被发现，而不是等到发布前几天才暴露。

类比：CI 就像每天收拾房间，而不是等到年底大扫除。垃圾天天倒，永远不会堆积成灾难；反过来一年不打扫，最后清理成本高到让人想放弃。

持续交付 CD：随时可发布，但由人决定何时发布

持续交付（Continuous Delivery）是在 CI 的基础上更进一步：代码通过了所有自动化测试之后，流水线会继续把它自动打包成一个随时可以发布到生产环境的制品（比如一个 Docker 镜像、一个 jar 包），并且这个制品已经在类生产环境里验证过。

"随时可发布"是持续交付的精髓——它意味着团队在任意时刻都对"这个版本能不能上线"有明确的答案，而不用临时加班去验证。但是否真的把这个制品推到生产环境，仍然是人工按下一个按钮来决定，通常出于业务节奏、审批流程等考量。

持续部署：连"按按钮"这一步也省了

持续部署（Continuous Deployment）是最彻底的自动化形态：只要代码通过了流水线中的所有自动化验证，就自动直接发布到生产环境，全程无需人工干预。

这要求团队对自动化测试的覆盖率和质量有极高的信心——因为一旦测试没抓住的 bug，会在几分钟内直接影响真实用户。像 Etsy、Netflix 这类工程文化成熟的公司，一天可以做到几十甚至上百次持续部署。

三者的递进关系

阶段自动化范围发布到生产环境典型触发方式
持续集成 CI 合并代码 → 自动构建 → 自动测试 不涉及 每次 push / PR
持续交付 CD CI 的全部 + 自动打包成可发布制品 人工点一下"发布"按钮 手动触发上线
持续部署 持续交付的全部 + 自动发布 全自动，无需人工干预 测试通过即自动上线
面试高频："持续交付和持续部署的区别？" 一句话答案：两者都要求代码随时处于可发布状态，区别只在于最后"发不发到生产环境"这一步是人工点按钮还是全自动。 很多团队说自己在做"持续部署"，实际上做的是"持续交付"——因为发布前还留了一个人工审批环节。

核心动机：为什么非要自动化

  • 降低人工操作的错误率：人手动执行重复步骤，早晚会漏一步、敲错一个参数；机器执行同一套脚本，结果是确定且可重复的。
  • 缩短反馈周期：问题发现得越晚，修复成本越高。CI 让"代码写错了"在几分钟内暴露，而不是等到几周后的集成测试甚至生产环境。
  • 让发布从"大事"变成"小事"：如果每次发布都要提心吊胆，团队会本能地推迟发布、攒大版本一起发，而大版本发布恰恰是风险最高的做法。自动化让小步快跑、频繁发布成为可能，每次发布的改动量小，出问题也容易定位和回滚。
  • 与自动化测试相辅相成：CI/CD 流水线的信心完全建立在自动化测试之上（呼应 15-测试模块）——没有靠谱的测试覆盖，流水线再快也只是"更快地把 bug 发布到生产环境"。
