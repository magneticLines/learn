蓝绿部署与金丝雀发布

两种最常被拿来对比的"零停机、低风险"发布策略。名字听起来花哨，本质其实是在回答同一个问题：新版本出问题了，怎么把影响范围和回滚成本降到最低？

为什么不能直接"覆盖上线"

最朴素的发布方式是：停掉旧版本，用新版本覆盖，再启动。这样做有两个致命问题——发布过程中服务不可用（哪怕只有几秒钟），以及一旦新版本有问题，只能手忙脚乱地重新部署回旧版本，回滚同样有停机时间。蓝绿部署和金丝雀发布都是为了解决这两个问题而生，但走了两条不同的路。

蓝绿部署：整体切换，回滚是"秒级"的

蓝绿部署（Blue-Green Deployment）同时维护两套完全独立、配置对等的生产环境：一套叫"蓝"，一套叫"绿"。假设当前"蓝"环境正在承接所有线上流量，发布新版本时，把新版本完整部署到闲置的"绿"环境，在"绿"环境上做完充分验证（冒烟测试、内部走查）后，再把负载均衡 / 网关的流量入口瞬间切到"绿"环境。

发布前：用户流量 → 负载均衡 → 蓝环境（v1，在线）　　绿环境（空闲）
部署新版本：用户流量 → 负载均衡 → 蓝环境（v1，在线）　　绿环境（部署 v2，验证中）
切换流量：用户流量 → 负载均衡 → 绿环境（v2，在线）　　蓝环境（v1，保留，随时可回滚）

如果切换后发现 v2 有问题，只需要把流量入口切回蓝环境即可，这个操作通常在秒级完成，因为蓝环境的 v1 从始至终都是热的、随时能接流量的。

类比：就像商场里的两部电梯，一部在检修（部署新版本），检修完毕、确认没问题后把所有人引导到新电梯；如果新电梯出故障，立刻把大家引回旧电梯——旧电梯全程没有拆，随时能用。

代价也很直观：你需要同时维护两套一模一样规模的生产环境，也就是双倍的资源成本（至少在切换的窗口期是双倍）。此外，蓝绿部署是"整体切换"，所有用户在同一时刻都切到新版本，一旦新版本真的有问题，是全量用户都会受到影响。

金丝雀发布：先小范围观察，再逐步放量

金丝雀发布（Canary Release）这个名字来自矿工带金丝雀下矿井探测瓦斯的传统——金丝雀对毒气更敏感，一旦异常就是提前预警。放到发布场景里：先把新版本只暴露给一小部分流量或用户（比如 5%），密切观察错误率、延迟等关键指标，确认没问题后再逐步扩大比例（5% → 25% → 50% → 100%），直到全量切换完成。

第一步：95% 流量 → v1（旧版本）　　5% 流量 → v2（新版本，金丝雀）
观察指标：错误率、P99 延迟、业务指标是否异常
逐步放量：50% → v1　　50% → v2　　……直到 0% → v1　　100% → v2

金丝雀发布的最大优势是把影响范围控制在很小的比例内——即使新版本真的有 bug，也只有一小部分用户受影响，而且能在扩大范围之前就发现问题并叫停。代价是整个发布过程比较慢（要留出观察窗口），并且在新旧版本共存期间，需要处理好兼容性问题：比如新版本改了数据库字段格式，旧版本的代码还在跑，必须保证两个版本都能正确读写数据（也就是常说的"向前兼容、向后兼容"）。

核心区别（面试高频）

维度蓝绿部署金丝雀发布
切换方式 整体瞬间切换（All-or-Nothing） 按比例逐步放量
问题影响范围 一旦切换出问题，影响全量用户 先影响一小部分用户，可及时止损
回滚速度 秒级（切回旧环境的流量入口） 较快，但需要把已放量的流量收回
资源成本 需要两套对等的生产环境（约双倍资源） 不需要双倍环境，新旧版本共存即可
兼容性负担 切换瞬间完成，共存时间极短 新旧版本较长时间共存，需处理兼容性
发布耗时 快（验证完直接切） 慢（需要多轮观察和放量）
面试高频："蓝绿部署和金丝雀发布的区别？" 一句话答案：蓝绿部署是"整体瞬间切换、双倍资源换取秒级回滚"，金丝雀发布是"小流量试探、逐步放量以缩小问题的影响半径"。 两者不是互斥的，实践中常常组合使用：先用金丝雀验证正确性，确认无误后再用蓝绿式的整体切换完成全量发布。

再补一种：滚动发布

滚动发布（Rolling Update，Kubernetes 的默认发布方式）逐个替换旧版本实例：先关闭一个旧实例、启动一个新实例，确认健康后再处理下一个，直到全部替换完成。它不需要额外的双倍资源（只是短暂多出一两个实例），但发布过程中新旧版本必然同时在线，且如果发现问题，已经替换的部分需要逐个回滚，速度介于蓝绿部署和金丝雀发布之间。

常见误区：不要把"滚动发布"和"金丝雀发布"混为一谈。滚动发布的目标是"平滑地、不中断服务地替换全部实例"，最终一定会全量替换；金丝雀发布的目标是"先小范围验证，觉得有问题可以随时叫停，不必继续扩大"，两者出发点不同。

与其他知识点的联系

  • 部署流水线的设计思想（下一篇）：本篇讲的是"发布本身怎么执行、风险怎么控制"，下一篇讲的是"发布之前怎么验证这个版本值得发"——两者是前后接力的关系：流水线跑完所有验证阶段之后，走到最后一步，正是触发蓝绿切换或金丝雀放量的时机。
  • 特性开关 Feature Toggle：金丝雀发布"先给一小部分流量"这个能力，在代码层面往往就是靠特性开关实现的——开关可以按用户比例、按地理位置精细控制某个功能是否生效，这正是金丝雀发布在应用层的落地手段，而蓝绿/金丝雀更多是在基础设施/网关层面控制流量入口。
  • K8s 声明式 API 与控制循环：金丝雀发布和滚动发布在 Kubernetes 里都是通过 Deployment 声明期望状态（比如副本比例）来实现的，K8s 的控制循环会持续调整实际的 Pod 数量去逼近这个期望状态——发布策略的"渐进式"特性，恰好是声明式系统天然擅长的场景。
  • Docker 网络模式（容器化主线）：无论蓝绿部署的整体流量切换，还是金丝雀发布的按比例分流，落地时都要靠负载均衡器/网关重新配置流量路由——这和容器网络模式里"外部流量如何接入容器"是同一层面的问题，只是这里多了一个"按策略动态调整路由目标"的维度。
