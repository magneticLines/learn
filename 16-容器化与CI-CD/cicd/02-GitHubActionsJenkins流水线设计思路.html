CI/CD

GitHub Actions / Jenkins 流水线设计思路

不管用什么工具，一条像样的 CI/CD 流水线都会经历相似的阶段。先搞懂"标准流程长什么样"，再看两大主流工具各自的设计取舍。

流水线的典型阶段

一条完整的流水线本质上是把"代码提交"到"用户用上新功能"之间的所有步骤串起来，每一步都是对上一步产出的进一步加工或验证：

1. 代码检出（Checkout）— 从代码仓库拉取本次提交的源码
  ↓
2. 编译构建（Build）— 把源码编译成可执行产物（如 Maven/Gradle 打包）
  ↓
3. 单元测试（Unit Test）— 跑最快、最底层的自动化测试，快速拦截逻辑错误
  ↓
4. 代码质量扫描（Lint / SonarQube）— 静态检查代码规范、坏味道、安全漏洞
  ↓
5. 打包镜像（Docker Build & Push）— 产出带版本号的可发布制品
  ↓
6. 部署到测试环境（Deploy to Staging）— 在接近生产的环境里先跑一遍
  ↓
7. 集成测试（Integration Test）— 验证服务间协作、真实依赖是否正常
  ↓
8. 部署到生产环境（Deploy to Production）— 持续交付需人工确认，持续部署则自动执行
设计思想：每个阶段都是一道"关卡"，前一关不过，后一关就不会执行——这正是下一篇要讲的"部署流水线"思想的雏形：用一系列自动化验证逐步建立起对发布候选版本的信心。同时，越靠前的阶段执行越快、成本越低（单元测试几秒钟，集成测试可能要几分钟），所以要"快失败"，把便宜的检查放在前面，尽早拦截问题。
呼应 15-测试模块：流水线里的第 3、7 步直接依赖你写的自动化测试。没有单元测试和集成测试打底，流水线的"质量关卡"就是摆设——CI/CD 工具只是负责"自动跑"，跑得准不准全看测试本身的质量。

GitHub Actions：YAML 声明式，与生态深度融合

GitHub Actions 把流水线定义为仓库里的一个 YAML 文件（.github/workflows/ci.yml），触发条件、运行环境、每一步要执行的命令都以声明式的方式写清楚：

.github/workflows/ci.yml
name: CI Pipeline
on: [push, pull_request]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4       # 1. 代码检出
      - uses: actions/setup-java@v4
        with: { java-version: '17' }
      - run: mvn -B compile            # 2. 编译构建
      - run: mvn -B test               # 3. 单元测试

这种"托管在代码仓库里、随代码一起版本化"的方式，加上 GitHub 官方维护和社区贡献的海量现成 Action（如 setup-java、docker/build-push-action），让 GitHub Actions 特别适合开源项目和中小团队：几乎零运维成本，配置即生效。

Jenkins：功能强大，但要自己运维

Jenkins 是更"重"的老牌方案，核心是 Master/Agent 架构：Master 负责调度和管理流水线定义，实际的构建任务分发给一个或多个 Agent（可以是不同操作系统、不同硬件配置的机器）去执行。流水线通常用 Jenkinsfile（Groovy DSL）描述：

Jenkinsfile
pipeline {
    agent any
    stages {
        stage('Build') { steps { sh 'mvn compile' } }
        stage('Test')  { steps { sh 'mvn test'    } }
    }
}

Jenkins 的优势在于插件生态极其丰富（几乎所有你能想到的工具都有对应插件）、可以灵活自建 Agent 集群应对复杂的企业内部环境（多机房、多种构建环境、严格的权限隔离）。代价是你需要自己部署、升级、维护 Jenkins Master 本身，插件版本冲突、Master 单点故障等运维问题也需要团队投入精力应对。

两者的取舍

维度GitHub ActionsJenkins
配置方式 YAML，声明式，随仓库版本化 Jenkinsfile（Groovy），也可视化配置
运维成本 托管服务，零运维 需自建 Master/Agent，自行运维升级
生态集成 与 GitHub 深度绑定，社区 Action 丰富 插件生态极其丰富，几乎不挑代码托管平台
适用场景 开源项目、中小团队、GitHub 生态用户 企业内部复杂流水线、私有化部署、强定制需求
常见误区：并不是"Jenkins 比 GitHub Actions 更强大所以更好"。工具选型要看团队规模和场景——中小团队为了省下运维一套 Jenkins 的人力成本，用 GitHub Actions 往往性价比更高；而大型企业内部有大量遗留系统、复杂的网络隔离要求时，自建 Jenkins 集群反而更可控。

与其他知识点的联系

  • 持续集成与持续部署的理念（上一篇）：上一篇讲的 CI/持续交付/持续部署三级台阶是"目标"，本篇的八个阶段是"怎么用具体的 job/stage 去实现这个目标"——流水线里第 1～5 步对应 CI，第 6～8 步对应持续交付/部署，两篇是理念和落地的关系。
  • 部署流水线的设计思想（下一篇）：本篇列的八个阶段只是"流水线长什么样"，下一篇要回答的是"为什么必须设计成前一关不过、后一关不执行的漏斗结构"——本篇是现象，下一篇是背后的设计原理。
  • Dockerfile 与镜像分层缓存：流水线第 5 步"打包镜像"直接依赖 Dockerfile 的分层缓存设计，缓存命中率越高，流水线里这一步跑得越快，团队感知到的"CI 反馈速度"就越快。
  • 蓝绿部署与金丝雀发布：流水线的第 8 步"部署到生产环境"具体怎么执行，正是下一篇要展开的内容——流水线负责把版本一步步验证到"可以发布"的状态，发布策略负责把这个版本安全地推给用户。
  • 基础设施即代码 IaC：本篇的流水线操作对象是"应用代码和镜像"，IaC 的流水线操作对象是"服务器和云资源"，但触发方式（提交变更 → 自动执行一系列验证阶段）是完全一致的设计模式，GitHub Actions / Jenkins 同样可以用来跑 terraform plan/apply。
