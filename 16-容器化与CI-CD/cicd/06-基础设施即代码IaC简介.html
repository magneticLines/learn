基础设施即代码 IaC 简介

[补充：《持续交付》Jez Humble & David Farley] 进阶内容。前面几篇都在讲"代码怎么发布"，这一篇往底层再走一步：承载代码运行的服务器、网络、云资源本身，要怎么管理才不会失控？

手工配置服务器的老问题：环境漂移

在没有 IaC 之前，配置一台服务器往往是运维人员手动 SSH 登录上去，一条条敲命令安装软件、改配置文件、调整防火墙规则。这种方式短期内能用，但时间一长会暴露出一个典型问题——环境漂移（Configuration Drift）：

  • 测试环境和生产环境配置本应一致，但因为是不同时间、不同人手动搭建的，实际上"看起来差不多但细节不一样"，导致"测试环境跑得好好的，一上生产就出问题"。
  • 某台服务器上有一条奇怪的配置，没人知道是谁、什么时候、为什么加上去的——原来负责的人已经离职，配置的来龙去脉全靠猜。
  • 想复制一套一模一样的环境（比如扩容、灾备），只能凭记忆或者旧文档重新手动操作一遍，很难保证百分之百复现。
核心痛点：手工配置的服务器，其真实状态只存在于服务器本身，而不存在于任何可审查、可追溯的地方。这就是"宠物 vs 牲畜"比喻里"宠物式服务器"的问题——每台服务器都是独一无二、精心照料、一旦出问题就很难重建的"宠物"。

核心思想：把基础设施的定义写成代码

基础设施即代码（Infrastructure as Code）的做法是：用 Terraform、Ansible 这类工具，把服务器配置、网络规则、云资源（虚拟机、数据库实例、负载均衡器等）都用声明式的配置文件描述出来，并且像管理应用代码一样，把这些配置文件纳入 Git 版本控制。

main.tf（Terraform 示例）
resource "aws_instance" "web_server" {
  ami           = "ami-0abcd1234"
  instance_type = "t3.medium"
  tags = {
    Name = "ccc-backend-prod"
  }
}

这段代码描述的是"我想要一台什么规格的服务器"，而不是"具体要执行哪些命令"——这就是声明式的核心：你只声明期望的最终状态，工具负责计算出从当前状态到目标状态需要做什么操作，并且下次执行时能自动识别"哪些已经存在、哪些需要新建、哪些需要修改"。

IaC 解决的问题

  • 可 Review：基础设施的变更以 Pull Request 的形式提交，团队成员可以像 review 代码一样 review "要不要开放这个端口""要不要把实例规格调大"，而不是变更完了才发现有问题。
  • 可追溯：每一次基础设施变更都对应一次 Git 提交，谁在什么时候改了什么、为什么改，全部有据可查，彻底消灭"没人知道这台服务器为什么这么配"的窘境。
  • 可重复执行：同一份配置文件可以在测试环境、预发环境、生产环境反复执行，天然保证多环境配置一致，从根源上消除环境漂移。
  • 可快速重建：如果一台服务器出了故障，不需要凭记忆手动重建，直接重新跑一遍 IaC 配置就能得到一台配置完全相同的新服务器——服务器从"宠物"变成了"牲畜"（坏了直接换一个，而不是精心抢救）。
与本模块前面内容的关系：IaC 让"部署环境"本身也可以被纳入 CI/CD 流水线——变更基础设施配置同样要经过代码检出、格式校验、计划预览（terraform plan）、人工审批、执行（terraform apply）这样的流水线式流程，思路与部署应用代码的流水线（第 2、4 篇）完全一致，只是操作对象从"应用制品"变成了"云资源"。
面试高频："什么是基础设施即代码，解决了什么问题？" 参考答案：IaC 是用代码描述和管理服务器、网络等基础设施资源，并纳入版本控制。它解决的核心问题是消除手工配置导致的环境漂移，让基础设施变更像应用代码变更一样可 review、可追溯、可重复执行。

模块小结

至此，"容器化与 CI/CD"这个模块的三个 Tab 全部完成：容器化解决了"环境一致性"和"打包部署"的问题，Kubernetes 解决了"大规模容器编排与调度"的问题，本 Tab 的 CI/CD 则解决了"如何又快又稳地把代码从提交到上线"的问题——三者共同构成了现代软件工程从开发到生产的完整链路。
