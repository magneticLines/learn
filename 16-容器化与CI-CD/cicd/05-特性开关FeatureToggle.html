特性开关 Feature Toggle

[补充：《持续交付》Jez Humble & David Farley] 持续集成要求"频繁合并到主干"，但一个大功能往往要开发好几周——这两者听起来是矛盾的。特性开关就是化解这个矛盾的关键武器。

矛盾：频繁合并 vs 功能还没做完

第 1 篇讲过，持续集成要求代码频繁地合并到主干，间隔越短，集成冲突越小。但现实中一个完整的新功能常常需要几周甚至几个月才能做完，难道要一直憋着不合并，等做完了再一次性合入主干吗？

如果选择"憋着不合并"，会退化回文章开头提到的老问题：长期存在的功能分支和主干差异越来越大，合并时爆发大规模冲突，正是持续集成想要消灭的东西。这就是特性开关要解决的问题。

核心思想：把"合并到主干"和"对用户可见"解耦

特性开关（Feature Toggle，也叫 Feature Flag）的做法是：把未完成或还需要灰度验证的功能代码，正常提交、正常合并到主干，但用一个配置开关包裹这段代码，控制它是否真正对用户生效。

FeatureToggleExample.java
if (featureToggleService.isEnabled("new-checkout-flow")) {
    // 新的结算流程：代码已经合并到主干，但默认关闭
    newCheckoutFlow.process(order);
} else {
    // 老的结算流程：在新功能验证完成前，用户始终走这条路径
    legacyCheckoutFlow.process(order);
}

这样一来，开发者可以把"新结算流程"拆成很多个小提交，每完成一小块就合并到主干一次——主干代码库里始终存在这段还没完工的代码，但因为开关默认关闭，它对生产环境的真实用户完全不可见，不会有任何风险。等功能开发完毕、内部验证通过，只需要把配置中心里的开关值改成"开启"，功能立刻生效，不需要重新走一次代码发布流程。

类比：特性开关就像装修房子时先把水电线路都预埋好，但阀门先关着——线路施工（代码合并）和真正通水通电（功能上线）是两件可以分开安排时间的事，不用等所有房间都装修完才统一通电。

特性开关解决的问题

  • 支持持续集成：不用等一个大功能完全做完才合并，可以频繁小步提交，从根源上避免长期分支和大规模合并冲突。
  • 发布节奏与代码提交节奏解耦：代码什么时候合并到主干、和功能什么时候对用户开放，是两条独立的时间线。产品经理可以选择在业务合适的时间点（比如某个营销活动开始时）"开灯"，而不受代码开发进度的约束。
  • 为金丝雀发布和 A/B 测试提供基础设施：特性开关往往不是简单的"开/关"两态，而是可以按用户比例、按用户群体、按地理位置精细控制——这正是上一篇金丝雀发布"先给一小部分用户开放"在代码层面的实现手段。
  • 出问题时能快速止损：如果新功能上线后发现问题，直接把开关关掉即可让功能立刻下线，不需要走一次完整的代码回滚和重新部署流程，止损速度是分钟级甚至秒级的。
常见误区/陷阱：特性开关不是免费的午餐。开关一多，代码里会堆积大量 if/else 分支，测试组合数量随开关数量指数级增长，长期不清理的"僵尸开关"会让代码库变得难以理解。最佳实践是给开关设定生命周期：功能全量稳定后，要及时把开关判断逻辑和废弃分支从代码中清理掉，而不是让开关永久留在代码里。
面试高频："特性开关解决了什么问题？" 参考答案：它把"代码合并到主干"和"功能对用户生效"这两件事解耦，使得团队既能保持持续集成要求的高频合并节奏，又能灵活控制功能的发布时机和灰度范围。
