Docker 镜像与容器的关系

镜像和容器是 Docker 里最基础也最容易混淆的一对概念——一个是"类"，一个是"实例"，用面向对象的思维去理解，一下子就通了。

镜像是模板，容器是实例

镜像（Image）是一个只读的静态模板，它打包了运行一个应用所需要的一切：操作系统的基础文件系统、依赖库、运行时环境、应用代码、启动配置。镜像本身不运行，它只是"待命"在那里的一份完整快照。

容器（Container）是镜像的一个运行实例。当你执行 docker run，Docker 会在镜像的只读层之上，叠加一层可写层（Container Layer），然后在这个基础上启动一个进程。容器运行过程中产生的所有变化——新建文件、修改配置、写日志——都发生在这层可写层里，不会污染底下的只读镜像。

类比 Java：如果把镜像看作一个 class，容器就是 new 出来的对象实例。同一个类可以 new 出很多个相互独立的对象，同一个镜像也可以 docker run 启动出很多个相互独立的容器——它们共享同一份"类定义"（镜像的只读层），但各自的运行时状态（可写层）互不干扰。
终端验证"一个镜像可以启动多个容器"
# 同一个镜像，启动 3 个相互独立的容器实例
docker run -d --name web1 nginx
docker run -d --name web2 nginx
docker run -d --name web3 nginx

# 三个容器各自独立运行、独立可写层，互不影响
docker ps

镜像的分层存储结构

镜像不是一整块打包文件，而是由多个只读层（Layer）叠加而成，每一层记录的是相对上一层的文件系统变更（新增、修改、删除了哪些文件）。这套机制底层依赖 UnionFS（联合文件系统），把多层内容"合并挂载"成一个统一的文件系统视图给容器使用。

镜像分层示意（自上而下叠加）
容器可写层（Container Layer，仅本容器可见，可读写）
─────────────────────────
Layer 4：COPY app.jar /app（业务代码）
Layer 3：RUN apt-get install -y curl（安装依赖）
Layer 2：RUN apt-get update（更新源）
Layer 1：FROM ubuntu:22.04（基础操作系统层）

分层的好处不止是"结构清晰"，更重要的是共享和复用：如果多个镜像都基于同一个 ubuntu:22.04 基础层构建，这一层在本地磁盘上只会存一份，多个镜像共享引用它，而不是各自复制一份完整的操作系统文件。这也是为什么同一台机器上跑几十个基于相同基础镜像的容器，磁盘占用远比想象中小。

常见误区

误区："容器里改的文件，是不是也会改到镜像？" 不会。容器的可写层和镜像的只读层是彻底分开的，容器内的任何修改都只写入这层临时的可写层。容器删除后，可写层跟着一起被销毁，改动全部丢失；镜像本身永远不会被容器的运行过程修改。如果想把改动固化下来，需要显式执行 docker commit（不推荐，生产环境应该改 Dockerfile 重新构建），或者用数据卷把需要持久化的数据挂到容器外部。
面试高频："镜像和容器的关系？" 标准答案：镜像是静态的只读模板，容器是镜像的运行实例，二者关系类似类和对象。追问"容器里的改动去哪了？" 答：写入容器专属的可写层，容器删除即丢失，不会影响镜像本身。

核心价值总结

  • 镜像负责"定义"，容器负责"运行"：职责分离让同一份环境定义可以被反复、一致地实例化。
  • 分层存储实现了空间和时间的双重节省：相同的基础层可以被多个镜像共享，减少磁盘占用；未变化的层可以直接复用，为下一篇要讲的构建缓存打下基础。
  • 只读 + 可写分离保证了镜像的不可变性：不管容器内部怎么折腾，镜像本身始终是干净、可复现的，这是"环境一致性"承诺的根基。

与其他知识点的联系

  • 容器与虚拟机的本质区别（上一篇）：上一篇讲容器运行时靠 namespace + cgroup 实现隔离，本篇讲的镜像分层结构是"容器运行之前，这些隔离出来的文件系统视图从哪来"——镜像分层本质上是 UnionFS 把多层只读文件系统合并挂载成容器看到的统一视图。
  • Dockerfile 的编写与镜像分层缓存（下一篇）：本篇讲清楚了"镜像是分层的、层可以共享复用"这个静态结构，下一篇要讲的构建缓存机制正是建立在这个结构之上——理解了"层"是什么，才能理解为什么调整 Dockerfile 指令顺序能加速构建。
  • Docker 网络模式与数据卷：本篇提到"容器可写层随容器删除而销毁"，这正是数据卷篇章要解决的核心矛盾的源头——数据卷本质上是把需要持久化的数据从这层易失的可写层里"搬出去"，托管到独立于容器生命周期的存储空间。
  • K8s 声明式 API 与控制循环：K8s 的 Deployment 里指定的 image 字段，本质上就是在声明"我要用这份不可变的模板去实例化 Pod"——镜像的不可变性和可复现性，正是 K8s 能够放心地反复创建、销毁、重建 Pod 而不担心状态不一致的前提。
