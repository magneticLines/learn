Docker 网络模式与数据卷

容器化基础的最后一块拼图：容器之间怎么通信，容器的数据怎么在容器销毁后还能活下来。这两个问题分别由网络模式和数据卷解决。

网络模式：容器怎么和外界通信

容器依靠 namespace 拥有独立的网络视图（独立的网卡、IP、路由表），但独立不等于"与世隔绝"，Docker 提供几种网络模式来决定容器如何接入网络：

  • bridge（桥接模式，默认）：Docker 会创建一个虚拟网桥（如 docker0），每个容器接入这个网桥，分配一个私有 IP。同一网桥下的容器之间可以互相访问，容器要访问宿主机外部网络需要经过 NAT 转换，外部要访问容器则需要显式做端口映射（-p 参数）。
  • host（主机模式）：容器不再拥有独立的网络 namespace，而是直接共享宿主机的网络栈。容器里监听的端口就是宿主机上的端口，少了 NAT 转发这一层，性能更好，但代价是失去了网络层面的隔离性——容器和宿主机、以及宿主机上其他 host 模式容器共用同一套网络资源，端口冲突需要自己协调。
  • none（无网络模式）：容器拥有独立的网络 namespace，但不做任何网络配置，只有一个回环接口（lo），无法与外界通信。适合完全不需要网络、只做本地计算的场景。
终端三种网络模式的使用示例
# bridge：默认模式，做端口映射，宿主机 8080 转发到容器 80
docker run -d -p 8080:80 nginx

# host：容器直接用宿主机网络栈，容器里监听 80 就是宿主机的 80
docker run -d --network host nginx

# none：无网络，容器完全隔离
docker run -d --network none alpine sleep 1000
类比：bridge 像小区里每户单独装一个门牌号和信箱，快递员（外部请求）要通过小区大门保安（NAT）转发才能送到具体某户；host 模式像直接把这户人家的门牌号焊死成小区大门的地址，效率高但少了保安这层过滤和隔离。

数据卷：解决容器"用完即焚"的矛盾

容器的可写层是和容器生命周期绑定的——容器一旦被删除，可写层里的所有数据都会一并消失。这对无状态应用（比如无状态的 Web 服务）没问题，但对数据库、日志这类需要持久化的数据就是灾难：难道每次升级重启容器，数据库里的数据就要清空重来？

数据卷（Volume）就是为了解决这个矛盾而生的：它是一块独立于容器生命周期的存储空间，由 Docker 在宿主机上专门管理（默认位于 /var/lib/docker/volumes/）。容器把某个目录挂载到数据卷上，读写这个目录时实际读写的是卷里的数据；容器被删除，卷不会跟着被删除，下次新容器挂载同一个卷，还能拿到之前的数据。

终端用数据卷持久化 MySQL 数据
# 创建一个具名数据卷
docker volume create mysql-data

# 启动容器时把 /var/lib/mysql 挂载到这个数据卷上
docker run -d --name mysql -v mysql-data:/var/lib/mysql mysql:8

# 即使删除并重新创建容器，只要挂载同一个卷，数据依然还在
docker rm -f mysql
docker run -d --name mysql-new -v mysql-data:/var/lib/mysql mysql:8
补充：除了具名数据卷，Docker 还支持 Bind Mount（绑定挂载），直接把宿主机上的某个具体路径挂载进容器（如 -v /host/path:/container/path），常用于开发阶段把本地代码实时映射进容器；数据卷则更适合生产环境，因为它由 Docker 统一管理、不依赖宿主机具体目录结构，也更方便备份和迁移。

常见误区

误区："容器里的数据默认就是持久化的，删了容器数据也不会丢。" 恰恰相反，这是初学者最容易踩的坑。不挂载数据卷的容器，数据只存在于容器的可写层，删除容器数据就彻底丢失。凡是数据库、消息队列这类有状态服务，必须显式挂载数据卷，否则一次误删容器就是一次数据灾难。
面试高频："Docker 数据卷解决了什么问题？" 答案要点：容器默认是无状态、易失的（可写层随容器销毁而消失），数据卷把数据的生命周期从容器的生命周期中解耦出来，从而实现数据持久化，同时也方便多个容器共享同一份数据。

核心价值总结

  • 网络模式的选择是"隔离性"与"性能/便利性"的取舍：bridge 默认最安全，host 适合对网络性能敏感且信任宿主机环境的场景。
  • 数据卷把"数据的生命周期"从"容器的生命周期"中剥离：这是有状态应用能够容器化的关键前提。
