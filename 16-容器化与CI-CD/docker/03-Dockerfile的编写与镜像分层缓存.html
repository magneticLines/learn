Dockerfile 的编写与镜像分层缓存

Dockerfile 是"如何构建镜像"的说明书，每一条指令都会生成一层。指令的顺序不只是风格问题，直接决定了你每次改代码要等多久。

常用指令一览

指令作用
FROM指定基础镜像，是 Dockerfile 的第一条指令，决定了后续所有层的起点
RUN在构建阶段执行命令（如安装依赖），执行结果会被提交为新的一层
COPY把宿主机文件复制进镜像，最常用来复制业务代码
ADD类似 COPY，但额外支持自动解压 tar 包和拉取 URL，语义更"重"，官方建议优先用 COPY
WORKDIR设置后续指令的工作目录，等价于 cd
ENV设置环境变量，构建阶段和容器运行阶段都生效
EXPOSE声明容器运行时监听的端口，仅作文档说明，不会真的做端口映射
CMD容器启动时的默认命令，可以被 docker run 后面的参数整体覆盖
ENTRYPOINT容器启动时固定执行的命令，docker run 后面的参数会作为参数追加，而不是覆盖整个命令
Dockerfile一个典型的 Java 应用 Dockerfile
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "target/app.jar"]

分层缓存机制：为什么构建有时候秒过

Dockerfile 里每一条指令都会生成镜像的一层。Docker 构建镜像时，会按顺序逐层检查：如果某一层的指令内容和它依赖的上下文（比如 COPY 涉及的文件内容）跟上一次构建完全一致，就直接复用缓存中的那一层，跳过真正执行，直接进入下一条指令。

但这套缓存机制有个关键规则：一旦某一层缓存失效（miss），它之后的所有层都会跟着失效，即使那些层的指令内容本身没变化。这就是为什么指令的先后顺序如此重要。

反例：把 COPY . .（复制全部代码）放在 RUN mvn dependency:go-offline（下载依赖）之前。这样只要业务代码改了一行，COPY 这一层缓存就失效，导致后面"重新下载全部依赖"这一层也被迫重新执行——每次改一行代码都要等几分钟重新拉依赖，非常低效。
正例（最佳实践）：把变化频率低的指令放前面，变化频率高的放后面。上面的示例 Dockerfile 就是这个思路：先单独 COPY pom.xml 并下载依赖（pom.xml 极少变化，这一层缓存能长期复用），再 COPY src 复制业务代码并编译（代码天天改，但只影响它之后的层）。这样日常开发时，改代码只需要重新执行"复制代码 + 编译"两层，依赖下载那层永远命中缓存。

缓存命中与失效示意

第一次构建（全部执行）
FROM ... [执行]
COPY pom.xml . [执行]
RUN go:offline [执行，耗时下载依赖]
COPY src ./src [执行]
RUN mvn package [执行]

只改了 src 下代码，第二次构建
FROM ... [缓存命中，跳过]
COPY pom.xml . [缓存命中，跳过]
RUN go:offline [缓存命中，跳过 —— 依赖不用重新下载]
COPY src ./src [缓存失效，重新执行]
RUN mvn package [失效层之后全部重新执行]

常见误区

误区："CMD 和 ENTRYPOINT 有什么区别，随便用哪个不都一样？" 区别在于和 docker run 命令行参数的交互方式：CMD 指定的默认命令会被 docker run image_name some_cmd 中的 some_cmd 整体覆盖；而 ENTRYPOINT 指定的命令是固定的，命令行传入的参数只会作为追加参数拼接在后面。生产 Dockerfile 里常见的写法是 ENTRYPOINT 定死启动程序，CMD 提供默认参数（可被覆盖）。
面试高频："Docker 镜像构建缓存的原理和最佳实践？" 答案要点：每条指令一层，前面某层缓存失效会导致后面所有层都重新构建，所以要把不常变的指令（装依赖）放前，常变的指令（复制业务代码）放后，最大化利用缓存、加速 CI 构建。

核心价值总结

  • Dockerfile 是声明式的构建脚本：每条指令对应一层，指令集合起来就是完整的镜像构建过程，天然可追溯、可复现。
  • 分层缓存的关键是"顺序"：缓存失效会像多米诺骨牌一样向后传导，把稳定的部分前置、易变的部分后置是加速构建的核心技巧。
  • 这不只是性能优化，也是团队协作和 CI/CD 效率的基础：一个写得好的 Dockerfile 能让团队每次提交后的镜像构建从几分钟缩短到几秒钟。

与其他知识点的联系

  • Docker 镜像与容器的关系（上一篇）：上一篇讲镜像是由多个只读层叠加而成的静态结构，本篇的构建缓存机制正是利用了这个结构——"某一层缓存命中就跳过执行"能生效的前提，正是每条指令严格对应一层这个约定。
  • GitHub Actions / Jenkins 流水线设计思路：流水线里"打包镜像"这一步的耗时高度依赖本篇讲的缓存命中率——一个 Dockerfile 指令顺序写得好，CI 流水线里镜像构建这一步可能从几分钟缩短到几秒钟，这直接影响团队感知到的整体反馈速度。
  • 基础设施即代码 IaC：Dockerfile 本质上也是一种"基础设施即代码"——它用声明式的指令描述"这个运行环境该长什么样"并纳入版本控制，IaC 篇里 Terraform 描述云资源、Dockerfile 描述应用运行环境，是同一思想在不同抽象层级的体现。
  • 持续集成与持续部署的理念：本篇强调的"分层缓存是团队协作和 CI/CD 效率的基础"直接呼应第 1 篇的核心动机——缩短反馈周期。镜像构建慢，意味着每次提交后要更久才能知道结果，这正是持续集成想要消灭的"反馈延迟"。
