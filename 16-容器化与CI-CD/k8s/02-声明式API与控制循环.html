Kubernetes 核心概念

声明式 API 与控制循环

K8s 最深刻的设计哲学不是"某个具体功能"，而是一种编程范式的转变——从"告诉它怎么做"变成"告诉它要什么"，剩下的交给系统自己去趋近。

声明式 API：只说"要什么"，不说"怎么做"

传统运维脚本大多是命令式（Imperative）的：一步步写"先启动 3 台机器，再逐个部署应用，再配置负载均衡"。这类脚本的问题是，它只描述了"从当前状态到目标状态"的一条固定路径——一旦中途出错、或者运行环境和预期不符，脚本很可能就跑不下去或者产生错误结果。

K8s 采用的是声明式（Declarative）方式：你只需要在 YAML 里描述"最终我想要的状态"——比如"我要 3 个副本，用这个镜像"——然后把这份声明提交给 K8s，不关心具体怎么一步步达成，也不关心当前是什么状态。

对比命令式思维 vs 声明式思维
// 命令式：一步步告诉系统"怎么做"
if (currentReplicas < 3) {
    createPod(); createPod();
}
if (currentReplicas > 3) {
    deletePod();
}

# 声明式：只描述"要什么"，交给系统自己去趋近
spec:
  replicas: 3   # 我不管现在几个，我只要最终变成 3 个

控制循环：Reconcile Loop 是怎么工作的

声明式的"我只说要什么"能落地，靠的是 K8s 内部运行着大量控制器（Controller），每个控制器都在不停地执行同一个逻辑循环，叫做控制循环（Control Loop）或调谐循环（Reconcile Loop）：

Reconcile Loop 伪代码
for (;;) {
  desired = 读取用户声明的期望状态（如 replicas: 3）
  actual = 观察集群里的实际状态（如实际运行了几个 Pod）
  if (desired != actual) {
    执行操作让 actual 趋近 desired（创建/删除 Pod）
  }
  sleep(一段时间) // 然后无限循环，持续观察和纠偏
}

这个循环永不停止，也不管这次"意外"是怎么发生的——不管是节点宕机、Pod 被误删、还是网络抖动导致 Pod 假死，控制器只关心一件事：现在的实际状态和期望状态是否一致，不一致就想办法拉齐。

相比命令式脚本的核心优势：自愈能力

举例：假设某个 Pod 被运维同学手滑误删了。命令式脚本运行完就结束了，不会有人盯着这个"事故"去补救；但 K8s 的 Deployment 控制器因为持续在跑 Reconcile Loop，下一次循环观察到"期望 3 个副本，实际只有 2 个"，立刻就会创建一个新 Pod 补上——整个自愈过程不需要任何人工介入，这就是"声明式 + 控制循环"这套设计带来的核心价值：系统具备了对抗各种意外情况的韧性。

这种设计还有一个好处：幂等性。同一份 YAML 声明，提交一次和提交一百次，最终收敛到的状态是完全一样的——不会因为"重复执行了脚本"而产生副作用，这对自动化流水线（CI/CD 反复部署同一份配置）尤其重要。

常见误区

误区："我执行了 kubectl apply，K8s 应该立刻把状态改好了吧？" 声明式系统的状态收敛不是瞬时的，而是最终一致（Eventually Consistent）的——控制循环需要若干次迭代才能把实际状态拉到期望状态，中间会有短暂的过渡态（比如滚动更新过程中新旧 Pod 短暂并存）。理解"最终一致"而不是"立刻一致"，是排查 K8s 问题时的重要心智模型。
面试高频："K8s 为什么要用声明式 API？" 答案要点：声明式把"如何达成目标"的复杂性交给系统本身，用户只需要维护期望状态；配合永不停止的控制循环，系统天然具备自愈能力和幂等性，这是大规模集群自动化运维的基础。

核心价值总结

  • 声明式 API 降低了心智负担：用户只描述目标状态，不用操心中间步骤和异常处理逻辑。
  • 控制循环是自愈能力的来源：持续观察 + 持续纠偏，让集群能够自动从各种意外中恢复。
  • 这是 K8s 几乎所有资源对象（Deployment、Service、甚至用户自定义 CRD）背后共同的设计范式，理解了这一篇，再看任何 K8s 控制器的行为都会豁然开朗。
