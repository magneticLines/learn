Kubernetes 核心概念

Pod / Deployment / Service 三大核心对象

K8s 里成百上千种资源对象，但真正打地基的只有这三个：Pod 定义"跑什么"，Deployment 定义"跑多少、怎么更新"，Service 定义"怎么访问"。

Pod：K8s 最小的调度单位

Pod 是 Kubernetes 中最小的可调度、可管理单位——注意，不是"容器"。一个 Pod 内部可以运行一个或多个容器，这些容器共享同一个网络 namespace（同一个 IP、可以用 localhost 互相访问）和可选的存储卷。

为什么不直接调度容器，要多包一层 Pod？因为现实中很多场景需要"一组紧密协作、必须同生共死"的容器打包在一起调度，最经典的是边车模式（Sidecar Pattern）：主容器跑业务逻辑，边车容器负责收集日志、做服务网格代理（如 Istio 的 Envoy）、同步配置文件等辅助工作。这些容器必须被调度到同一台节点、共享网络和生命周期，Pod 正是为了表达"这些容器是一个不可拆分的部署单元"而存在的抽象层。
pod.yaml一个包含主容器和边车容器的 Pod
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  containers:
    - name: main-app
      image: my-app:1.0
    - name: log-collector      # 边车容器，收集主容器日志
      image: log-agent:1.0

Deployment：声明式管理 Pod 的副本与更新

直接创建 Pod 有个致命问题：Pod 一旦所在节点故障或被删除，不会自动重建。生产环境显然不能靠人肉盯着重建。Deployment 就是用来解决"如何维护一组 Pod 的稳定副本数量，以及如何安全地滚动更新它们"的控制器。

你只需要在 Deployment 里声明"我要 3 个副本，用这个镜像"，Deployment 控制器就会持续确保集群里始终有 3 个符合条件的 Pod 在运行——某个 Pod 挂了，立刻补一个新的；升级镜像版本时，按策略（默认滚动更新）逐个替换旧 Pod，保证升级过程中服务不中断。

deployment.yaml声明 3 个副本的 Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels: { app: my-app }
  template:
    metadata:
      labels: { app: my-app }
    spec:
      containers:
        - name: my-app
          image: my-app:1.0

Service：给动态变化的 Pod 一个固定入口

Pod 是"易失"的——被 Deployment 重建后，Pod 的 IP 地址会变化。如果上游服务直接记着某个 Pod 的 IP 去调用，Pod 一重建，这个地址立马失效。Service 就是解决这个问题的抽象：它对外暴露一个固定不变的虚拟 IP（ClusterIP）和 DNS 名称，请求打到 Service 上后，由 Service 按标签选择器动态转发给背后一组健康的 Pod，并做负载均衡。

类比：Pod 好比呼叫中心里随时可能换座位的客服员工，Service 就是那个对外统一的客服热线号码——打这个号码的人不需要知道具体是哪个工位的哪个员工接的电话，号码本身永远不变，内部换人、加人、减人都对外部呼叫者透明。

三者如何协作

Deployment（声明"要 3 个副本"）
  └─ 创建并维护 → Pod × 3（每个 Pod 内跑 1+ 个容器，IP 易变）

Service（提供固定虚拟 IP + DNS 名称）
  └─ 通过 Label Selector 动态关联 → 上面这一组 Pod
  └─ 外部请求打到 Service → 负载均衡转发到某个健康 Pod

常见误区

误区："Pod 和容器是一回事，就是换了个名字。" 不是。Pod 是"一组共享网络和存储的容器"的封装，一个 Pod 里可以有多个容器；而且直接管理 Pod 是不推荐的，生产环境几乎不会手写裸 Pod，而是通过 Deployment（或 StatefulSet、DaemonSet 等）来间接管理 Pod 的生命周期。
面试高频："Pod、Deployment、Service 分别是做什么的？" 一句话答案：Pod 是最小调度单位（跑容器），Deployment 管理 Pod 的副本数量和滚动更新（保证跑多少、怎么升级），Service 提供稳定的访问入口和负载均衡（解决 Pod IP 易变的问题）。三者分别解决"运行、伸缩更新、访问"三个独立的问题。

核心价值总结

  • Pod 解决"协作容器的打包问题"：共享网络和存储的容器组，是调度的原子单位。
  • Deployment 解决"可用性和可更新性问题"：自动补齐副本、安全地滚动升级，把"运维人肉操作"变成"声明式配置"。
  • Service 解决"服务发现和稳定访问问题"：屏蔽了 Pod IP 的动态变化，是 K8s 集群内部通信的基石。
  • 这三者的组合，构成了 K8s 里最常见的应用部署单元，后面两篇会分别展开"为什么 K8s 要设计成声明式"和"Service 的服务发现细节"。
