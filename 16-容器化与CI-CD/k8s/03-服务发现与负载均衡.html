Kubernetes 核心概念

服务发现与负载均衡

上一篇提到 Service 给动态变化的 Pod 提供固定入口，这一篇深入它的实现细节——标签选择器怎么匹配 Pod，服务名怎么变成可访问的地址。

Label Selector：Service 如何找到它的 Pod

Service 并不知道具体有哪些 Pod、它们的 IP 是什么，它靠的是标签选择器（Label Selector）做动态匹配：Pod 在创建时会被打上一组 Label（如 app: my-app），Service 声明自己要选中"所有带有 app: my-app 标签的 Pod"。

这是一种动态、松耦合的关联方式——不需要手动维护一份 IP 列表。Deployment 每次创建新 Pod、销毁旧 Pod，只要新 Pod 依然带着同样的 Label，Service 就能自动把它纳入自己的转发范围，整个过程不需要人工介入去更新 Service 的配置。

service.yamlService 通过 selector 关联 Pod
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app        # 匹配所有带这个 label 的 Pod
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP

K8s 集群内部有一个持续运行的组件叫 Endpoints Controller，它会持续监听匹配这个 selector 的 Pod 列表变化，实时更新 Service 背后实际的转发目标（Endpoints）。这本质上也是上一篇讲的控制循环模式的一个具体应用。

基于 DNS 的服务发现

K8s 集群内置了一个 DNS 组件（通常是 CoreDNS），每创建一个 Service，就会自动为它注册一条 DNS 记录：Service 名称可以直接被解析为它的虚拟 IP（ClusterIP）。这意味着集群里的应用要调用另一个服务，不需要知道对方任何一个 Pod 的真实 IP，只需要知道对方 Service 的名字，剩下的交给 DNS 解析 + Service 负载均衡。

示例应用代码里直接用 Service 名调用
// 应用代码里不写死任何 IP，直接用 Service 名 + 端口
String url = "http://my-app-svc:80/api/hello";

// K8s 内部 DNS 会把 my-app-svc 解析成对应的 ClusterIP
// 再由 Service 做负载均衡，转发到某个健康的 Pod

呼应：K8s Service 本质上是一种内建的服务发现方案

呼应 14-分布式与微服务模块：在没有 K8s 的传统微服务架构里，服务发现通常需要额外引入 Eureka、Nacos、Consul 这类注册中心组件——服务启动时向注册中心"注册"自己的地址，调用方向注册中心"查询"目标服务的地址列表，再自己做负载均衡（比如 Ribbon）。而在 K8s 里，这套服务注册、发现、负载均衡的能力被下沉到了平台层：Pod 的创建/销毁自动触发 Endpoints 更新（相当于自动注册/注销），DNS + Service 完成了查询和负载均衡。业务代码不再需要接入专门的注册中心 SDK，这是 K8s 相比传统微服务框架的一大简化。

常见误区

误区："Service 会帮我把请求转发到具体某一个固定的 Pod 上。" 不对，Service 做的是动态、无状态的负载均衡——每次请求（或者说每次建立连接）具体转发给背后哪个 Pod，是由转发规则（默认基于 iptables 或 IPVS 的随机/轮询策略）决定的，不会绑定某个固定 Pod。如果误以为 Service 会话粘滞到固定 Pod，可能会在需要会话保持的场景踩坑——这种情况需要显式配置 sessionAffinity。
面试高频："K8s 里服务之间是怎么互相发现和调用的？" 答案要点：Pod 打标签，Service 通过 Label Selector 动态关联一组 Pod；K8s 内置 DNS 把 Service 名解析为 ClusterIP；调用方只需要用 Service 名访问，请求会被自动负载均衡到健康的 Pod 上——整个过程无需接入外部注册中心。

核心价值总结

  • Label Selector 是 K8s 里"动态分组"的通用机制：不仅 Service 用它关联 Pod，很多其他资源对象也依赖同样的模式。
  • DNS + ClusterIP 让服务调用变得"位置透明"：调用方不关心目标服务背后有多少个 Pod、部署在哪，只需要记住一个稳定的服务名。
  • K8s 把服务发现和负载均衡能力内建到了平台层：这是 K8s 相较传统需要额外接入注册中心的微服务架构的一大简化，也是"容器化基础"到"K8s 核心概念"这两个 Tab 学习内容的自然落点——下一步就该进入 CI/CD，学习如何把代码自动构建、打包、部署到 K8s 集群里。
