Git · 分支

HEAD:指向"当前分支的指针"的指针

如果分支是指向 commit 的指针,那 Git 怎么知道你现在"站在"哪个分支上?答案是 HEAD——一个特殊的符号引用。理解 HEAD,就理解了"切换分支""提交往哪走""detached HEAD"这些看似神秘的现象。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、HEAD 是什么:它多套了一层

上一篇说:分支是指向 commit 的指针。而 HEAD 是指向"当前分支"的指针——它比分支又多套了一层间接。所以常把它叫作"指针的指针",或者更准确地说是一个符号引用(symbolic reference)。

它同样是一个纯文本文件,就在 .git/HEAD。正常情况下,它的内容不是哈希,而是一句"我现在跟着哪个分支":

bash · HEAD 里存的是分支引用,不是哈希
# 正常状态:HEAD 指向一个分支(符号引用)
$ cat .git/HEAD
ref: refs/heads/main

# 顺着这条引用,才找到 main 分支真正指向的 commit
$ cat .git/refs/heads/main
f30ab2a4e8c6b1d9f5c3a7e2b8d4c6f1a9e3b5d7

于是这条查找链是:HEAD → main(分支) → C3(commit) → 项目快照。三层间接,各司其职。

二、为什么要多这一层?一次提交看懂它的作用

当你执行 git commit 时,Git 需要知道两件事:(1) 新 commit 的父是谁 (2) 提交后哪个分支要前移。这两个答案都来自 HEAD:

图示 · 提交后 HEAD 带着当前分支一起前移
# 提交前:HEAD → main → C3
    C1 <── C2 <── C3      <── main <── HEAD

# git commit:新建 C4(父=C3),然后 main 前移到 C4,HEAD 仍指向 main
    C1 <── C2 <── C3 <── C4  <── main <── HEAD

正是因为有 HEAD 这一层,Git 才能"自动"知道该让哪个分支跟着前进。切换分支的本质,就是改写 .git/HEAD 这个文件的内容,让它指向另一个分支,然后把工作区内容更新成那个分支所指 commit 的快照。

三、查看 HEAD 现在指向哪里

bash · 观察 HEAD
# git log 里 (HEAD -> main) 就是在告诉你:HEAD 跟着 main
$ git log --oneline
f30ab2a (HEAD -> main, dev) add feature
9c8b7a1 init

# 用 symbolic-ref 直接问 HEAD 跟的是哪个分支
$ git symbolic-ref HEAD
refs/heads/main

四、detached HEAD:HEAD 直接指向 commit,脱离了分支

如果你执行 git checkout <某个commit哈希>(而不是分支名),Git 会把 .git/HEAD 里的内容从"ref: refs/heads/xxx"改成一个具体的哈希。这时 HEAD 不再跟着任何分支,而是"游离"地直接指向某个 commit——这就是 detached HEAD(分离头指针)。

图示 · detached HEAD
# 正常:HEAD 通过分支间接指向 commit
    HEAD ── main ──> C3

# git checkout <C2的哈希> 后:HEAD 直接钉在 C2 上,没有分支跟着它
    HEAD ─────────> C2        main ──> C3

# .git/HEAD 此时内容变成一个哈希,而不是 ref:
$ cat .git/HEAD
a1b2c3d4...

它危险在哪:新提交会"无家可归"

在 detached HEAD 状态下你照样可以提交,新 commit 会正常产生,HEAD 也会前移。但问题是:没有任何分支指向这些新 commit。一旦你 checkout 回其他分支,这些提交就失去了引用——从 git log 里再也看不到,最终会被垃圾回收清掉。

面试/踩坑高频点:初学者常在"看历史某个版本长啥样"时 checkout 了一个旧哈希,然后随手改了东西提交,切回 main 后发现提交"消失"了。Git 其实会在切走时警告你 You are in 'detached HEAD' state...。补救办法:切走前先 git branch 新分支名 把这些提交"接住"。
什么时候会主动进入 detached HEAD?查看历史版本、临时试验某个旧提交、基于某个 tag/commit 做一次性构建。只要不打算在此长期开发,detached HEAD 完全正常且有用;要长期开发就立刻建分支。

五、怎么救回 detached HEAD 里的提交

bash · 用 git branch 接住游离的提交
# 假设你在 detached HEAD 下提交了几次,现在想保住它们
# 方法一:当场建分支,让分支指向当前 HEAD,提交就有了归属
$ git switch -c rescue-branch

# 方法二:已经切走了才想起来?用 reflog 找回那个哈希
$ git reflog                 # reflog 记录了 HEAD 的每一次移动
$ git branch rescue <找到的哈希>

小结

  • HEAD 是"指向当前分支的指针",本质是 .git/HEAD 里一个符号引用(ref: refs/heads/xxx)。
  • 查找链:HEAD → 分支 → commit → 快照;多这一层间接,commit 才知道父是谁、提交后哪个分支前移。
  • 切换分支 = 改写 .git/HEAD 的内容 + 更新工作区。
  • detached HEAD:HEAD 直接指向某个 commit 哈希、脱离分支;在此提交的内容无分支归属,切走后易丢失、会被 GC。
  • 补救:切走前 git switch -c 分支名 接住;已切走用 git reflog 找回哈希再建分支。
