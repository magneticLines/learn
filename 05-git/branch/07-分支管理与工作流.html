分支管理与工作流

命令会用只是基础,团队真正需要的是一套约定:哪些分支长期存在、功能怎么隔离、什么时候合并。这一篇讲清长期分支 vs 特性分支,以及 Git Flow / GitHub Flow 两套主流工作流。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、长期分支:项目的"稳定线"

长期分支(long-running branch)是贯穿整个项目生命周期、不会被删掉的分支。它们代表项目不同的稳定程度:

分支角色稳定性
main(旧称 master)随时可发布的生产代码,只放经过验证的东西最稳定
develop(或 dev)集成分支,汇集下一个版本的所有功能较稳定
心智模型:把提交想象成"从不稳定往稳定推进的流水线"。功能先在特性分支里搅动,测试稳定后合入 develop,充分验证后再合入 main 发布。越靠近 main 的分支,越干净、越可信。

二、特性分支:短命的"隔离气泡"

特性分支(topic/feature branch)是为某个具体功能、实验或 bug 修复临时开的分支,用完即删。正因为第一篇讲过分支只是 41 字节的指针、成本极低,Git 才鼓励你为每件小事都开一个。

bash · 特性分支的一生
$ git switch -c feature/login   # 从主线切出一个隔离环境
# ...在这里自由折腾、多次提交,不影响任何人...
$ git switch develop
$ git merge --no-ff feature/login   # 完成后合回集成分支
$ git branch -d feature/login       # 用完删除,标签回收
好处:①主线始终可用,半成品被隔离;②多个功能可并行推进互不干扰;③代码评审(PR/MR)天然以特性分支为单位;④出问题只需丢弃这个分支,主线毫发无伤。

三、两套主流工作流

Git Flow:分支多、规矩严,适合有版本发布节奏的项目

Git Flow 定义了严格的分支模型:两条长期分支 main + develop,外加三类临时分支——feature/*(功能)、release/*(发布准备)、hotfix/*(线上紧急修复)。

图示 · Git Flow 分支结构
main     ──●───────────────●────────●──   # 每个●是一次发布(打 tag)
             \             / \      /
hotfix       \           /   ●────/       # 紧急修复,同时并回 main+develop
               \         /
develop  ──●──●──●──●──●──●──────────●──   # 集成线
            \  /    \  /
feature    ●●      ●●●                       # 各功能分支
取舍:Git Flow 表达力强,但分支和步骤繁多,对持续部署的现代 Web 项目往往显得笨重。它更适合有明确版本号、需要同时维护多个发布的产品(如客户端 App、需长期支持的软件)。

GitHub Flow:极简,适合持续部署

GitHub Flow 只有一条长期分支 main(始终可部署)。流程极简:从 main 切特性分支 → 开发提交 → 开 Pull Request 评审 → 合回 main → 立即部署。没有 develop、没有 release 分支。

怎么选?发布节奏快、一天多次上线、SaaS/网站 → GitHub Flow(或其变体 Trunk-Based)。有严格版本管理、多版本并行维护 → Git Flow。团队没有强诉求时,优先选简单的,别为了"看起来专业"上一套用不上的复杂流程。

四、删除分支:-d 与 -D 的区别

bash · 删除分支
$ git branch -d feature/login   # 安全删除:若未被合并进当前分支,Git 会拒绝并警告
$ git branch -D experiment      # 强制删除:不管有没有合并,直接删(慎用!)
为什么要有 -d 的保护?删分支只是删指针,但如果这个分支上有还没被任何其他分支引用的提交,删完这些提交就"无家可归"、终将被 GC。-d 会检查"是否已合并",没合并就拦你一下,防止丢工作;确认要丢弃实验代码时才用 -D。

五、查看分支状态:合并了没?

bash · 分支盘点常用命令
$ git branch --merged           # 已并入当前分支的分支 → 通常可安全 -d 删除
$ git branch --no-merged        # 尚未合并的分支 → 删除前要三思
$ git branch -a                 # 连远程跟踪分支一起列出
$ git branch -vv                # 显示每个分支的上游及领先/落后情况
$ git log --oneline --graph --all  # 全局俯瞰所有分支的分叉汇合
定期清理:用 git branch --merged 找出已合并的旧特性分支批量删除,保持分支列表清爽。远程侧则用 git remote prune origin / git fetch --prune 清理已在远端删除的跟踪分支。

小结

  • 长期分支(main/develop)贯穿项目、代表不同稳定级别;特性分支短命、用完即删。
  • 廉价分支催生"每个功能一个隔离分支"的工作流:主线稳定、并行开发、天然对应代码评审。
  • Git Flow 分支多、规矩严,适合有版本发布节奏的产品;GitHub Flow 极简,适合持续部署。没有强诉求就选简单的。
  • git branch -d 安全删除(未合并会拦截),-D 强制删除(慎用)。
  • --merged/--no-merged/-vv/--graph 用来盘点分支状态并定期清理。

与其他知识点的联系

  • 关联创建与切换分支:本篇讲的"特性分支用完即删"这套工作流,能够成立的前提正是上一篇提到的分支极低成本——switch -c 建分支只是写一个指针文件,所以才敢鼓励"每个功能都开一个分支"这种高频次操作。
  • 关联快进合并 vs 三方合并:特性分支合入 develop/main 时用 --no-ff,正是为了在采用本篇工作流时保留"这是一次功能合入"的边界,方便后续按合并提交整体回滚,这也是 Git Flow 里合并动作的标准姿势。
  • 关联变基与 rebase vs merge 取舍:GitHub Flow、Trunk-Based 这类追求简洁历史的工作流,往往会搭配"特性分支先 rebase 主线、合入时再 merge --no-ff"的组合;而 Git Flow 因为分支结构复杂、协作者多,更依赖 merge 的"真实记录"特性,较少大量 rebase。
  • 关联远程协作(fetch/pull/push):长期分支和特性分支的划分,直接决定了远程仓库里哪些分支需要长期保护(如 main 禁止直接 push)、哪些分支推送后可以随意删除——这是团队配置分支保护规则、PR 评审流程的依据。
  • 关联 CI/CD:GitHub Flow"合入 main 即部署"的假设,要求 CI 在合并到 main 前跑完整测试、合并后自动触发部署;Git Flow 的 release 分支则常对应"预发布环境"的 CI 流水线——工作流的选择本质上也是在为 CI/CD 的触发策略做设计。
