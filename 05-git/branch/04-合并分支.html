合并分支:git merge 与"有两个父"的提交

开分支是为了并行开发,最终还要把成果汇回主线。merge 到底做了什么?为什么会冒出一个"有两个父指针"的特殊提交?这一篇把合并的机制讲清楚。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、合并的方向:把"别人"并入"当前分支"

关键先记住:git merge 是把指定分支合入你当前所在的分支。所以合并前,一定要先切到"接收方"分支。典型场景:功能开发完,要把 feature 合进 main。

bash · 标准合并流程
$ git switch main            # ① 先站到接收方 main 上(合并会写在这里)
$ git merge feature          # ② 把 feature 合进来

# 合完确认没问题,删掉用完的特性分支
$ git branch -d feature
方向搞反是新手最常见的错:站在 feature 上执行 git merge main,是把 main 合进 feature——结果 main 没动,feature 反而更新了。要"让 main 拿到 feature 的成果",必须先切到 main。

二、merge 到底做了什么:找共同祖先做三方合并

当两个分支各自都有了新提交(历史分叉了),Git 无法简单地把一个指针挪过去。它会做一次三方合并(three-way merge):

图示 · 分叉的历史
          C4 <── main        # main 在 C2 之后自己走了一步 C4
         /
    C2 <┤
         \
          C5 <── feature     # feature 在 C2 之后走了 C5

# C2 是二者的"共同祖先"(merge base)

Git 取三个快照:共同祖先 C2、当前分支末端 C4、被合并分支末端 C5,逐文件对比:某文件只有一边改了,就采用改动方;两边都改了同一处,才需要人工解决(下一篇讲冲突)。合成的结果生成一个新的快照。

三、合并提交:一个"有两个父"的 commit

三方合并成功后,Git 会自动创建一个特殊的 合并提交(merge commit)。它和普通 commit 最大的区别是:有两个父指针——分别指向被合并的两条线的末端。

图示 · 合并提交 C6 有两个父
          C4 ──────┐
         /          \
    C2 <┤            C6 <── main <── HEAD
         \          /
          C5 ──────┘         (C6 的两个父 = C4 和 C5)
                    ↑
              feature 仍指向 C5,没动

这就是为什么 git log 里能看到"分叉又汇合"的网状历史:每一个汇合点,就是一个有两个父的合并提交。它忠实记录了"这里曾经有两条并行的开发线,在此汇合"这一事实。

联系对象模型:普通 commit 结构里 parent 字段只有一个;合并提交则有两个 parent 字段。可以 git cat-file -p <合并提交哈希> 亲眼看到两行 parent ...。commit 是"快照 + 父指针"的设计,天然就支持多父,合并只是用上了这个能力。

四、特例:快进合并没有合并提交

如果当前分支没有任何自己的新提交(它只是"落后"于目标分支,历史没分叉),Git 就不需要三方合并,也不产生合并提交,而是直接把当前分支指针快进(fast-forward)到目标位置。这种情况下 git log 是一条直线。

图示 · 快进(fast-forward)
# main 停在 C2,feature 一路走到 C4,main 没有自己的提交
    C2 <── C3 <── C4 <── feature
     ↑
    main

# git merge feature:main 无分叉,直接前移到 C4,无合并提交
    C2 <── C3 <── C4 <── feature <── main
预告:快进合并 vs 三方合并、以及 --no-ff 强制产生合并提交的取舍,是第 6 篇专门要讲的重点。这里先知道:历史有没有分叉,决定了 merge 走"快进"还是"三方合并"两条路。

五、常用选项与查看

bash · merge 相关命令
$ git merge --no-ff feature     # 即使能快进,也强制生成合并提交(保留分支痕迹)
$ git merge --squash feature    # 把 feature 所有改动压成一次普通提交,不建立父关系
$ git log --graph --oneline     # 用 ASCII 图看分叉/汇合结构
$ git branch --merged           # 查看哪些分支已经并入当前分支(可安全删)
误区:--squash 不是普通合并——它把对方一堆提交"压扁"成一个新提交、且不记录父关系,所以之后 Git 不认为两分支合并过,再次 merge 可能重复带入。它适合"我只要结果、不要对方的杂乱提交历史"的场景。

小结

  • git merge X 是把 X 合进当前分支;合并前先切到接收方,方向别搞反。
  • 历史分叉时做三方合并:取共同祖先 + 两分支末端三个快照对比,合出新快照。
  • 三方合并会生成一个有两个父指针的合并提交,忠实记录"两线汇合"的历史。
  • 若当前分支无自己的提交(未分叉),则走快进(fast-forward),只挪指针、无合并提交。
  • --no-ff 强制合并提交、--squash 压成单次提交(不建父关系),各有适用场景。
