变基 rebase:把提交"择出来重放到新基底"

merge 是把两条线"汇合"，rebase 则是把你的提交"搬家"到另一条线的末端。理解 rebase = 挑出提交 → 逐个重放,就能明白为什么变基后 commit 哈希会全变。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、rebase 的本质:摘下补丁，换个基底重新贴上

"rebase" = re-base，重新设定基底(base)。你的特性分支是从主线某个 commit 分叉出来的，那个分叉点就是它当前的"基底"。变基做的事，是把你分支上的每个提交算成一个补丁(diff)，逐个重放到目标分支的最新末端，相当于"假装你是从最新的主线才开始开发的"。

图示 · rebase 前(历史分叉)
          C4 <── main          # main 在分叉后前进到 C4
         /
    C2 <┤
         \
          C3 <── feature       # feature 从 C2 分叉，走了 C3
bash + 图示 · git rebase 后（线性）
$ git switch feature
$ git rebase main            # 把 feature 的提交重放到 main(C4)之后

# C3 被"择出→重放"成 C3'，接在 C4 后面，历史拉直了：
    C2 <── C4 <── C3'  <── feature
            ↑
           main                # main 还指向 C4；下一步通常再快进 main

$ git switch main
$ git merge feature          # 此时是快进合并，main 直接到 C3'

二、为什么变基后 commit 哈希会变？

这是理解 rebase 的关键。回顾对象模型：commit 的 SHA-1 哈希是由它的全部内容算出来的，而内容里包含了父提交的哈希。变基后，C3 的父从 C2 变成了 C4——父变了，内容就变了，算出的哈希自然全变。所以图里写的是 C3'（C3 prime）而不是 C3。

本质结论：rebase不是移动了原来的提交，而是照着原提交造了一批全新的提交（新哈希），原来的 C3 变成了无人引用的"孤儿"，终将被 GC。这就是为什么说"rebase 会重写历史"。

三、rebase 和 merge 达成的结果一样，历史却不同

mergerebase
做法找共同祖先做三方合并把提交摘出，重放到新基底
产物一个有两个父的合并提交一串哈希全新的普通提交
历史形状保留分叉与汇合（网状）拉成一条直线
原提交原样保留被替换为新哈希提交

两者最终代码内容完全一样，差别纯粹在于历史记录长什么样。取舍留到下一篇专门讲。

四、交互式 rebase：整理提交历史的利器

git rebase -i（interactive）让你在重放前，用一个编辑器逐条编辑将要重放的提交：改顺序、合并、拆分、改提交信息、丢弃。它是提交前"打扫历史"的标准工具。

bash · 交互式 rebase 编辑清单
$ git rebase -i HEAD~3        # 编辑最近 3 个提交

# 编辑器里出现（自上而下 = 由旧到新）：
pick   a1b2c3 add login form
squash d4e5f6 fix typo          # squash：并入上一条，合成一个提交
reword 7g8h9i add validation     # reword：只改这条的提交信息

# 常用指令：
#   pick   = 保留          reword = 改信息
#   squash = 合并入上一条   fixup  = 合并且丢弃本条信息
#   edit   = 停下让你修改    drop   = 删除该提交
典型用途：把开发过程中"wip""再改一下""修个错别字"这类杂乱提交，在提 PR 前 squash 成几个语义清晰的提交，让评审者和未来的自己看到干净的历史。

五、变基中的冲突与放弃

重放某个补丁时若与新基底冲突，rebase 会像 merge 一样停下，让你解决。注意：rebase 是逐个提交重放，可能连续遇到多次冲突。

bash · rebase 冲突处理
# 解决冲突后，用 --continue 继续重放下一个提交（不是 commit！）
$ git add <冲突文件>
$ git rebase --continue

$ git rebase --skip          # 跳过当前这个提交
$ git rebase --abort         # 彻底放弃，回到 rebase 之前的状态
易错点：①rebase 里 ours/theirs 会反过来（因为是把你的提交往目标上重放，"当前"变成了目标基底），解冲突时别按 merge 的直觉。②解决完是 rebase --continue，不是 git commit。③连续冲突时容易烦躁乱选，务必逐个看清。

小结

  • rebase 本质 = 把分支上的提交当作补丁逐个摘出、重放到新基底，把历史拉直。
  • 重放后父提交变了，故 commit 内容变、哈希全变（C3→C3'）——这就是"重写历史"。
  • rebase 与 merge 最终代码一致，区别只在历史形状：直线 vs 保留分叉汇合。
  • rebase -i 可 squash/reword/reorder/drop，是提交前整理历史的利器。
  • 冲突逐个重放，解决后用 --continue；--abort 可安全放弃；ours/theirs 相对 merge 是反的。

与其他知识点的联系

  • 关联快进合并 vs 三方合并：rebase 后再 merge 之所以能触发快进，正是因为 rebase 已经提前把历史"拉成一条线"、消除了分叉——这两篇讲的是同一个目标(线性历史)的两种不同实现路径。
  • 关联解决合并冲突：rebase 的冲突处理表面像 merge，但 ours/theirs 语义相反、可能连续遇到多次冲突、解决后用 rebase --continue 而不是 commit——这些差异都是因为 rebase 是"逐个提交重放"而非"一次性三方比较"。
  • 关联 Git 对象模型(commit 哈希)：本篇解释的"父提交变了导致哈希全变"，直接依赖于 commit 对象"哈希由内容计算、内容包含父指针"这一底层设计——理解了对象模型，才能明白 rebase 为什么必然是"重写历史"而不是"移动历史"。
  • 关联 rebase vs merge 取舍：本篇建立的"重放=重写"这个事实，是下一篇黄金法则("不要 rebase 已推送的提交")成立的根本原因；不理解哈希为什么会变，就无法真正理解为什么这条法则不能违反。
  • 关联远程协作(fetch/pull/push)： git pull --rebase 就是把本篇的 rebase 用在"同步远程更新"这个场景——把本地未推送的提交重放到远端最新提交之后，避免产生多余的合并提交，前提同样是这些提交还没被别人拉取过。
