分支的本质:一个 41 字节的指针文件

为什么 Git 创建分支近乎零成本、瞬间完成?因为分支根本不是"复制代码",它只是一个指向某个 commit 的轻量指针。理解了这一点,整个分支模型就通透了。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、先回顾 commit:分支之所以能这么轻,是被 commit 撑起来的

要讲透分支,必须先记住上一章的对象模型。在 Git 里,一次 commit 不是"记录本次改了哪几行",而是一个快照(snapshot):它保存了当时整个项目文件树的一份完整指向(通过 tree 对象),外加作者、时间、提交信息,以及最关键的——指向父 commit 的指针。

于是所有 commit 通过"父指针"串成一条链(合并时会有两个父,后面会讲)。这条链就是项目的历史。而分支,就是给这条链上某个 commit 贴的一个可移动的名字标签。

建立联系:commit 用 SHA-1 哈希(40 个十六进制字符)标识,例如 f30ab...。哈希由内容算出,内容变一点哈希就全变——这保证了历史不可篡改。分支要"指向"一个 commit,存的就是这 40 个字符。

二、分支到底是什么:refs/heads 下的一个纯文本文件

打开 .git/refs/heads/ 目录,你会看到每个分支对应一个同名文件。用 cat 看它的内容,你会大吃一惊——里面只有一行:那个 commit 的 40 位哈希 + 一个换行符,总共 41 字节。

bash · 亲手看看分支是什么
# 分支文件就躺在这里,每个分支一个文件
$ ls .git/refs/heads/
main   dev

# 直接读 main 分支文件——它只存了一个 commit 哈希
$ cat .git/refs/heads/main
f30ab2a4e8c6b1d9f5c3a7e2b8d4c6f1a9e3b5d7

# 数一下字节:40 个哈希字符 + 1 个换行 = 41 字节
$ wc -c .git/refs/heads/main
41 .git/refs/heads/main

所以"创建一个分支",本质就是往 refs/heads 里写一个 41 字节的小文件。无论你的项目有 10 个文件还是 10 万个文件,这个操作都是瞬间完成、几乎不占空间的。

三、图示:创建分支只是多贴一个标签

假设已有三次提交,main 指向最新的 C3。现在执行 git branch testing:

提交历史图示 · git branch testing
# 创建前:只有 main 指向 C3
    C1 <── C2 <── C3   <── main   (HEAD)
                (每个箭头是"指向父 commit"的指针)

# 执行 git branch testing 后:凭空多了一个指向同一个 C3 的标签
    C1 <── C2 <── C3   <── main   (HEAD)
                     \___ testing

# Git 做的全部事情,就是新建 .git/refs/heads/testing,内容 = C3 的哈希

注意此时 没有任何文件被复制,没有切换到新分支(HEAD 还在 main,下一篇细讲)。仅仅是多了一个名字而已。

四、与 SVN 对比:为什么老程序员对"随便建分支"感到不可思议

在 SVN(以及 CVS 等集中式系统)里,"分支"是通过把整个目录复制一份来实现的(svn copy trunk branches/feature)。项目越大,复制越慢、越占空间;而且分支之间的合并常常痛苦不堪。结果就是团队不敢轻易开分支,一个分支往往要慎重规划、长期存在。

维度SVN 的分支Git 的分支
本质复制整个目录树写一个 41 字节的指针文件
耗时随项目大小线性增长恒定,近乎瞬间
占用空间可能很大41 字节
使用心态慎重、少建随手就建,用完就删
这就是 Git 分支模型的精髓:正因为分支如此廉价,Git 才鼓励你为每一个小功能、每一次实验、每一个 bug 修复都开一个分支。这种"轻量分支 + 频繁合并"的工作流,是 Git 相比集中式系统最大的生产力飞跃。

五、常见误区提醒

  • 误区一:"建分支会把代码复制一份"。不会。分支只是指针,所有 commit 数据是共享的,直到你在某分支上产生新提交才会有新对象。
  • 误区二:"删分支会删掉代码"。删分支只是删掉那个 41 字节的标签文件。只要 commit 还被其他分支/引用可达,数据就还在(否则会被 GC 回收)。
  • 误区三:把分支想成一条"线"。分支不是那一整条 commit 链,而是链末端的那个可移动指针。链是 commit 之间的父子关系构成的,分支只是给某个点命名。

小结

  • commit 是"快照 + 父指针";所有 commit 靠父指针串成历史链。
  • 分支 = .git/refs/heads/xxx 里一个仅 41 字节(40 位哈希 + 换行)的指针文件,指向某个 commit。
  • 创建分支只是写一个小文件,不复制任何代码,所以近乎零成本、瞬间完成。
  • 对比 SVN 复制整个目录:Git 的廉价分支催生了"为每个功能开分支、用完即删"的现代工作流。
  • 牢记:分支是"链末端的可移动指针",不是那条链本身。
