解决合并冲突:冲突标记与处理流程

冲突不是 Git 出错,而是它诚实地告诉你:"这一处两个人改得不一样,我不敢替你做决定。"看懂冲突标记、走对处理流程,冲突就一点都不可怕。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、冲突为何产生:同一处被两边改成了不同样子

回顾上一篇的三方合并:Git 对比共同祖先、当前分支、被合并分支三个快照。只要一处只有一边改动,Git 就能自动决定。冲突只在一种情况发生:两个分支都修改了同一个文件的同一块区域(或一边改一边删)。此时 Git 无法判断该保留谁,于是停下来把决定权交还给你。

建立直觉:合并不是按文件、而是按"文本块(hunk)"去比对的。所以就算两人改的是同一个文件,只要改的是不同的行段落,Git 也能自动合并,不产生冲突。冲突真正的触发条件是"同一区域的分歧"。

二、冲突长什么样:三段式的冲突标记

发生冲突时,合并会中断,Git 把冲突文件里有分歧的地方用冲突标记包起来,让你直接在文件里看到两边的版本:

冲突文件 · index.html 中的冲突块
<div id="footer">
<<<<<<< HEAD                       # 从这里开始:当前分支(HEAD)的版本
  contact : email.support@github.com
=======                            # 分隔线:上面是我方,下面是对方
  please contact us at support@github.com
>>>>>>> feature                     # 到这里结束:被合并分支(feature)的版本
</div>
标记含义
<<<<<<< HEAD冲突块开始;它下面到 ======= 之间是你当前分支的内容
=======分隔线,上半是我方(ours),下半是对方(theirs)
>>>>>>> feature冲突块结束;它上面到 ======= 之间是被合并分支的内容

三、解决流程:改文件 → add → commit

解决冲突就是把冲突标记连同分歧内容,手工编辑成你最终想要的样子——可以留我方、留对方、都留、或者重写。关键是三行标记必须全部删掉。

编辑后 · 手工定稿(标记已全部删除)
<div id="footer">
  please contact us at email.support@github.com   # 我方地址 + 对方措辞,融合而成
</div>
bash · 冲突处理完整流程
$ git merge feature
# CONFLICT (content): Merge conflict in index.html
# Automatic merge failed; fix conflicts and then commit the result.

$ git status          # 看哪些文件还处于 "Unmerged" 状态,一个都不能漏
$ vim index.html      # 逐个编辑,删掉 <<< === >>> 标记,定稿内容

$ git add index.html  # add = 告诉 Git "这个文件我解决好了"(标记为已解决)
$ git commit          # 所有冲突解决后提交,生成那个合并提交(消息已预填)
为什么解决完要 git add?因为对 Git 来说,冲突文件处于特殊的"未合并(unmerged)"状态。git add 是在说"我已经拍板,以工作区当前内容为准",把它标记为已解决。只有所有冲突文件都 add 过,git commit 才允许提交。

四、不想合了:git merge --abort

如果冲突太多、或者你发现方向错了,想彻底放弃这次合并,回到 merge 之前的干净状态:

bash · 中止合并
$ git merge --abort   # 一键回退到 merge 前的状态,冲突标记全部撤销

# 想借助图形工具解决,而非手动删标记
$ git mergetool       # 调起配置好的可视化三方合并工具(如 vscode / meld)
为什么 abort 是安全的?merge 冲突发生时,合并提交还没生成,当前分支指针也没动。--abort 只是把工作区/暂存区还原到 merge 命令执行前——因为那个"before"状态一直被完整保留着。

五、常见误区与实用技巧

  • 忘删冲突标记就提交:把 <<<<<<< 之类的字符一起提交进去,代码直接编译报错。提交前用 git diff --check 检查是否残留冲突标记。
  • 以为"选一边就行":很多冲突正确答案是两边都要(比如各自加了一个新函数),盲目"保留我方/对方"会丢功能。要读懂语义再定稿。
  • ours/theirs 记反:在 merge 里,HEAD/ours 是你当前所在分支,theirs 是被合并进来的分支。(注意:在 rebase 里这俩会反过来,后面变基篇会提醒。)
  • 批量取舍:确定整个文件都要某一方时,git checkout --ours 文件 或 --theirs 文件 可整体选定,再 git add。
面试高频:"冲突是谁造成的、Git 能不能自动解决?"标准回答:冲突源于两分支对同一区域的不同修改,Git 无法擅自决定,故交由人工;非同一区域的修改 Git 会自动三方合并。"解决完为什么要 add?"——把 unmerged 状态标记为已解决。

小结

  • 冲突只在"两分支改了同一文件的同一区域"时发生;不同区域 Git 能自动合并。
  • 冲突标记三段式:<<<<<<< HEAD(我方)/ ======= / >>>>>>> 分支(对方)。
  • 流程:编辑定稿并删净标记 → git add(标记已解决)→ git commit 生成合并提交。
  • git merge --abort 可安全回到合并前状态;git mergetool 可视化解决。
  • 易错:忘删标记、盲目二选一丢功能、ours/theirs 记反;提交前 git diff --check。

与其他知识点的联系

  • 关联快进合并 vs 三方合并:冲突只可能发生在三方合并里——快进合并没有"两边都改过"的场景,自然也不存在冲突。理解上一篇的判定规则,才能预判什么时候该做好解决冲突的心理准备。
  • 关联变基 rebase:rebase 逐个重放提交时同样会遇到冲突,处理手法看起来相似(编辑标记、add、continue),但 ours/theirs 的含义是反过来的,而且可能连续遇到多次冲突——这是从 merge 冲突迁移到 rebase 冲突时最容易踩的坑。
  • 关联 Git 对象模型:冲突标记里的 HEAD 和分支名,本质上都是指向某个 commit 的引用;三方合并靠比较"共同祖先、HEAD、被合并分支"三个树对象的差异来决定哪些地方能自动处理、哪些要交给人工,这背后就是 blob/tree/commit 的对比逻辑。
  • 关联协作工作流(fetch/pull/push): git pull 遇到分叉历史时会自动触发 merge,如果双方恰好改了同一处,就会在 pull 的过程中突然弹出冲突——这也是"pull 隐藏合并"这个陷阱最常见的表现形式之一。
  • 关联 CI/CD:很多团队在 PR/MR 流程里要求"必须先在本地或通过 rebase/merge 解决冲突才能合并",CI 系统本身检测到目标分支有冲突时会直接标红并阻止合并,这就是本篇冲突处理流程在自动化流水线里的映射。
