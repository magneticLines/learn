快进合并 vs 三方合并,以及 --no-ff

同样一句 git merge,有时悄无声息地挪个指针就完事(快进),有时却生成一个合并提交(三方合并)。差别的根源是"历史有没有分叉"。搞懂它,你才能有意识地选择 --no-ff。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、判定规则:合并走哪条路,取决于共同祖先在哪

merge 时 Git 先找两分支的共同祖先(merge base),然后看一个关键问题:共同祖先是不是恰好就是当前分支的位置?

  • 是 → 说明当前分支从分岔点起没有任何自己的新提交,只是"落后"→ 快进(fast-forward)。
  • 否 → 两边都各自前进过,历史真的分叉了 → 三方合并(three-way merge),产生合并提交。

二、快进合并:只是把指针往前挪

当前分支的所有历史都被目标分支完整包含,合并根本不需要"融合"任何东西,Git 只是把当前分支指针直接快进到目标末端。

图示 · fast-forward
# 合并前:main 停在 C2,hotfix 从 C2 出发走到 C4
    C2 <── C3 <── C4 <── hotfix
     ↑
    main        (C2 是共同祖先,恰好=main 的位置)

# git switch main; git merge hotfix —— 直接把 main 快进到 C4
    C2 <── C3 <── C4 <── hotfix <── main
# 没有新提交、没有合并提交,历史是一条直线
为什么叫"快进"?类比看视频:进度条直接拖到后面,没有插入新内容。指针 main 只是沿着已有的 commit 链往前"进"了几格而已。

三、三方合并:分叉后必须生成合并提交

如果 main 在分岔后自己也提交过(比如别人往 main 推了东西),共同祖先就不再等于 main 当前位置,无法快进。Git 会做三方合并、生成一个有两个父的合并提交(见第 4 篇)。

图示 · three-way merge
          C4 ──────┐        # main 自己走了 C4
         /          \
    C2 <┤            C6 <── main   # C6 = 合并提交,两个父
         \          /
          C5 ──────┘        # feature 走了 C5
# 历史呈现"分叉→汇合"的形状

四、--no-ff:即使能快进,也强制留一个合并提交

快进很干净,但有个"缺点":合并后历史是直线,完全看不出这些提交曾经属于一个独立的特性分支。对于想保留"功能边界"的团队,这不利于回溯和回滚。--no-ff(no fast-forward)就是为此而生:

bash + 图示 · --no-ff 强制合并提交
$ git merge --no-ff feature

# 本来能快进(直线),--no-ff 偏要造一个合并提交 C5:
    C2 <── C3 <── C4 ──┐
     \                 \
      \_______________ C5 <── main   # C5 两个父:C2 和 C4
                       feature ──> C4

# 好处:一眼看出 C3~C4 是同一个特性分支的工作,回滚可整体 revert C5
维度快进(默认,可快进时)--no-ff
是否产生合并提交否是(强制)
历史形状直线,干净有"鼓包",保留分支痕迹
能否看出功能边界看不出能,合并提交即边界
整体回滚难(要逐个 revert)易(revert -m 一个合并提交)
适合个人小改、线性历史偏好团队协作、功能可追溯
实践取舍:很多团队在合并特性分支到 develop/main 时约定统一用 --no-ff(GitLab/GitHub 的 "Create a merge commit" 就是这个),让主干历史清晰地记录"每个功能一次合入"。而 git pull 拉取远端更新这种场景,快进反而更自然、不想平白多出合并提交。

五、常见误区

  • "我 merge 了怎么没有合并提交?"——因为触发了快进,历史本就没分叉。这不是 bug。
  • 把 --no-ff 当成"解决冲突的开关"——它只影响是否生成合并提交,和冲突有没有毫无关系。
  • 误以为快进"更危险"——快进不会丢任何提交,它只是不记录分支边界。信息量少,不代表不安全。
  • 禁用快进的另一种方式:在配置里 git config merge.ff false 可让默认就走 --no-ff;需要个别快进时用 --ff-only 反向强制(不能快进就直接失败,避免意外的合并提交)。

小结

  • 合并走快进还是三方合并,取决于共同祖先是否等于当前分支位置(即当前分支有没有自己的新提交)。
  • 快进:仅把指针前移,无合并提交,历史是直线。
  • 三方合并:历史分叉,生成有两个父的合并提交。
  • --no-ff 强制生成合并提交,保留特性分支边界、便于整体回滚,团队协作常用。
  • --ff-only 反向强制只允许快进;merge.ff false 可全局改默认行为。
