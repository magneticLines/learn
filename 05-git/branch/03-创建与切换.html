Git · 分支

创建与切换:branch / checkout / switch

建分支、切分支这些日常操作,命令却有新旧两套。为什么 Git 2.23 要新造 switch 和 restore 来分掉 checkout 的活?搞懂动机,你就不会再被 checkout 的"一词多义"绕晕。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、git branch:只创建,不切换

git branch <名字> 做的事,正如第一篇所讲——在 refs/heads 下写一个指针文件,指向当前 HEAD 所在的 commit。它不会把你切过去,HEAD 还留在原地。

bash · git branch 的几种用法
$ git branch testing        # 基于当前 commit 建分支,但人不过去
$ git branch                # 列出所有本地分支,* 标出当前所在
$ git branch -v             # 顺便看每个分支最新 commit
$ git branch dev abc1234    # 基于指定 commit 建分支,而不是当前位置
为什么"建了但没切过去"?因为 git branch 只写 refs/heads/testing,没动 .git/HEAD。HEAD 还指着原分支,所以你其实还站在原地。这正好印证了上一篇 HEAD 的作用。

二、git checkout:老命令,一个词干了两件事

git checkout 是老牌命令,它身兼数职,这正是它容易让人困惑的根源:

bash · checkout 的多重身份
$ git checkout testing        # 职责A:切换分支(改写 HEAD + 更新工作区)
$ git checkout -b feature     # 职责A':建分支并立刻切过去(= branch + checkout)
$ git checkout -- file.txt    # 职责B:丢弃工作区对 file.txt 的修改(还原文件!)
危险的一词多义:同一个 checkout,加个分支名是"切分支",加个文件名是"丢弃改动"。新手一不小心把没保存的修改用 checkout -- file 冲掉了,而且这种还原不进入 reflog、难以找回。这正是 Git 官方要拆分它的直接原因。

三、git switch / git restore:Git 2.23 把职责拆开了

2019 年 Git 2.23 引入两个新命令,把 checkout 的两份工作各自独立,让命令名直接表达意图:

意图老命令(checkout)新命令
切换分支git checkout devgit switch dev
建分支并切过去git checkout -b featgit switch -c feat
丢弃工作区改动git checkout -- filegit restore file
取消暂存(unstage)git reset HEAD filegit restore --staged file
bash · 推荐用新命令,意图一目了然
$ git switch dev            # 切到 dev,只做"换分支"这一件事
$ git switch -c hotfix      # -c = create,建 hotfix 并切过去
$ git switch -             # 切回上一个分支(类似 cd -)

$ git restore config.yml    # 只做"还原文件",不会误切分支
拆分的动机 = 安全 + 可读:一个只会切分支的命令,绝不可能因为你打错参数就把文件改动冲掉;一个只会还原文件的命令,也绝不会莫名切走分支。命令的能力边界越窄,误操作的代价就越小——这是良好 CLI 设计的通用原则。checkout 仍保留是为了向后兼容,新代码建议一律用 switch/restore。

四、切换时到底发生了什么

执行 git switch testing,Git 做三件事:

图示 · 切换分支的三步
# 切换前:HEAD → main → C3,工作区是 C3 的快照
    C1 <── C2 <── C3   <── main <── HEAD
                   \___ testing

# git switch testing 后:
#  ① 改写 .git/HEAD = "ref: refs/heads/testing"
#  ② 用 testing 所指 commit 的快照刷新暂存区和工作区
    C1 <── C2 <── C3   <── main
                   \___ testing <── HEAD

五、切换的前提:工作区要"干净"

切换会改动工作区文件。如果你当前有未提交的修改,而目标分支这些文件不同,Git 会拒绝切换以免覆盖你的工作:error: Your local changes ... would be overwritten by checkout。

bash · 切换被拦住时的三条出路
# 出路1:先提交,把改动固化到当前分支
$ git commit -am "wip"

# 出路2:临时收起来,切完再回来恢复(推荐做"半成品"时用)
$ git stash          # 把改动压入栈,工作区变干净
$ git switch dev
$ git switch -
$ git stash pop      # 取回刚才收起的改动

# 出路3:确实不要了,直接丢弃
$ git restore .
常见误区:以为"分支之间是隔离的,改了这个分支不影响那个"。其实未提交的改动不属于任何分支,它们浮在工作区上。只有 commit 之后,改动才真正"落到"某个分支上。这也是为什么切换前要么提交、要么 stash。

小结

  • git branch <名> 只创建指针、不切换(不动 HEAD)。
  • git checkout 是老命令,一词多义:切分支、建分支切换(-b)、还原文件(-- file),易误伤。
  • Git 2.23 拆分为 switch(专管分支:-c 建并切)与 restore(专管还原文件/取消暂存),意图清晰、更安全。
  • 切换 = 改写 .git/HEAD + 用目标 commit 快照刷新工作区。
  • 未提交改动浮在工作区、不属于任何分支;切换前用 commit 或 git stash 处理干净。
