rebase vs merge：该怎么选

两者代码结果一致，差别在历史。到底追求"整洁的线性历史"还是"如实记录的真实历史"？这是一场没有唯一正确答案的取舍——但有一条绝不能违背的黄金法则。

📖 来源:《Pro Git》 第 3 章「Git 分支」

一、核心分歧：历史应该"如实"还是"整洁"

这背后其实是两种哲学：

  • merge 派（记录派）：历史是一份事实档案，应如实反映"事情本来怎么发生"——你确实是从旧基底分叉、并行开发、最后汇合的，那就该留下分叉与合并提交。
  • rebase 派（整洁派）：历史是给人读的故事，应编辑成清晰易懂的线性叙事——那些"何时分叉、何时同步"的过程细节是噪音，拉直更好读、git bisect 也更好用。
图示 · 同一份工作，两种历史
# merge：保留分叉与汇合，能看出并行过程
    C2 ── C4 ──── M <── main
      \        /
       C3 ── C5

# rebase：拉成直线，仿佛按顺序一气呵成
    C2 ── C4 ── C3' ── C5' <── main

二、对照表：优缺点一览

可追溯性合并提交标记功能边界，好回溯丢失"何时同步主线"的信息
历史可读性多分支时图形复杂清爽，好读
冲突处理一次性集中解决可能逐个提交多次解决
安全性安全对已共享的提交有风险（见下）

三、黄金法则：绝不 rebase 已经推送到公共分支的提交

这是 rebase 唯一的、也是最重要的红线。不要对已经推送出去、别人可能已经基于它工作的提交做 rebase。

原因回到上一篇：rebase 会把提交替换成新哈希的提交。如果这些旧提交已经被别人拉走，你一 rebase 再强推，公共历史里就出现了"两份同样内容但哈希不同"的提交。队友再来同步时，Git 会把你重写的和他手里的旧版本都算进去，制造出重复提交、诡异冲突，历史一团乱麻。

图示 · 违反黄金法则的灾难
# 你推送了 C3、C4，队友已基于它们开发
    ...── C2 ── C3 ── C4 <── origin/main（大家共享的历史）

# 你却对已推送的 C3/C4 做 rebase，得到 C3'、C4'，force push
    ...── C2 ── C3' ── C4'      # 远端被你强推覆盖
# 队友本地还有 C3/C4，再 pull → C3 与 C3' 内容同但哈希异 → 重复+冲突地狱
一句话记牢：rebase 只用来整理还没分享给别人的本地提交。一旦提交进入公共分支、可能被他人依赖，就只能用 merge 前进，绝不重写。这是团队协作里最容易踩、后果最严重的坑。

四、务实的团队约定

成熟团队往往不是二选一，而是组合使用：

  • 本地私有分支：随便 rebase。推送前用 git rebase -i 把杂乱提交整理干净，是好习惯。
  • 同步主线更新：用 rebase 拉直。特性分支落后了，git pull --rebase 或 git rebase main 把自己的提交重放到最新主线之上，避免一堆无意义的合并提交。前提是你的提交还没被别人依赖。
  • 合入主干：用 merge（常带 --no-ff）。特性分支合进 main/develop 时用合并提交，保留"一次功能合入"的清晰边界，便于整体回滚。
bash · 常见组合工作流
# 让 pull 默认用 rebase，保持特性分支线性、少产生合并提交
$ git config --global pull.rebase true

# 特性分支上：先整理本地提交，再重放到最新主线
$ git switch feature
$ git rebase -i main

# 合入主干：用合并提交保留功能边界
$ git switch main
$ git merge --no-ff feature
没有银弹：Linux 内核、许多大型开源项目偏爱 rebase 保持线性、便于审阅；也有团队坚持 merge 强调真实性。关键不是"哪个更好"，而是团队达成一致并遵守黄金法则——一致性本身比选哪派更重要。

五、面试高频问答

  • Q：merge 和 rebase 区别？A：结果代码一致；merge 保留分叉、生成有两个父的合并提交，rebase 把提交重放到新基底、拉成直线并重写哈希。
  • Q：为什么 rebase 后哈希变了？A：commit 哈希由内容算出、内容含父指针；rebase 改了父，内容变，哈希全变。
  • Q：rebase 的黄金法则？A：不要 rebase 已推送到公共分支、可能被他人依赖的提交，否则强推会制造重复提交与冲突。
  • Q：git pull --rebase 好在哪？A：拉取远端更新时把本地未推送的提交重放到远端末端，避免产生无意义的合并提交，历史更线性。

小结

  • 核心分歧：merge 如实记录（真实但网状）vs rebase 整洁叙事（线性但重写历史）。
  • 黄金法则：绝不 rebase 已推送到公共分支、可能被他人依赖的提交，否则强推引发重复提交与冲突。
  • 务实组合：本地私有分支随便 rebase 整理；同步主线用 rebase 拉直；合入主干用 merge --no-ff 保留边界。
  • git config pull.rebase true 让拉取默认线性化。
  • 没有唯一正确答案，团队一致 + 遵守黄金法则，比"选哪派"更重要。
