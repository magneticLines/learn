撤销操作:amend / restore / reset / revert

"撤销"在 Git 里不是一个动作,而是一组针对不同区域、危险程度天差地别的命令。用错了会真丢代码。

📖 来源:《Pro Git》 第 2.4 节「撤销操作」

一、先问自己:要撤销哪一层?

撤销之所以让人头大,是因为改动可能停在三个不同的地方。先定位改动在哪一层,再选对应命令,一切就清晰了:

我想撤销的是……用的命令
工作区里"改了还没 add"的改动git restore <file>
已经 add 进暂存区、想取消暂存(退回工作区)git restore --staged <file>
刚提交完发现信息写错 / 漏加文件git commit --amend
想撤销"已经提交(甚至已推送)"的某次提交git revert / git reset

二、git commit --amend:修补最后一次提交

刚 commit 完发现提交信息写错、或漏 add 了一个文件,不必再来一个"修正提交"污染历史。先把漏的文件 add 上,再 --amend:

修补上一次提交
git add forgotten.java          # 补上漏掉的文件
git commit --amend             # 用暂存区当前内容"替换"上一次提交,可顺便改信息
git commit --amend -m "新的提交信息"   # 只改信息
本质与陷阱:amend 不是修改原提交,而是生成一个全新的提交去替换它(哈希会变)。所以——只对还没 push 的提交用 amend。如果原提交已经 push 且别人拉过,你 amend 后再强推会改写公共历史,导致队友的仓库和你对不上,是团队协作的大忌。

三、git restore:新时代的撤销工作区/暂存区

Git 2.23 引入 restore 和 switch,专门把老 checkout 那"既能切分支又能撤文件"的过载职责拆开,语义更清晰。

restore 两个方向
# 丢弃工作区里未暂存的改动,让文件恢复成暂存区(或最新提交)的样子
git restore App.java

# 取消暂存:把已 add 的文件退回"未暂存"状态(改动仍在工作区,不丢)
git restore --staged App.java
危险区别:git restore --staged 只是把改动从暂存区退回工作区,内容不丢,很安全。但不带 --staged 的 git restore App.java 会直接丢弃工作区里未提交的改动,且无法用 reflog 找回(因为它从没进过 Git 的对象库)。这是初学者最容易误删代码的地方——执行前务必确认这些改动真的不要了。

老写法 git checkout(仍常见)

你会在大量老教程、老项目里看到等价的旧命令:git checkout -- App.java(丢弃工作区改动)、git reset HEAD App.java(取消暂存)。它们和 restore 效果一样,只是 checkout 一词既管切分支又管撤文件,含义太重,才被拆分。新代码建议用 restore/switch,但要能读懂 checkout 写法。

四、reset vs restore vs revert:三者到底怎么分

命令操作对象会否改写历史危险度
git restore工作区 / 暂存区的文件不动历史低(带 --staged)/ 中(丢工作区)
git reset移动分支指针(可连带暂存区/工作区)会(丢弃它后面的提交)高(尤其 --hard)
git revert生成一个反向的新提交不改写,历史向前追加低(安全)

reset 三种模式(高频面试点)

  • --soft:只把分支指针往回移,暂存区和工作区都不动(被撤的改动还留在暂存区,可重新组织提交)。
  • --mixed(默认):指针回移 + 重置暂存区,工作区保留(改动退回"未暂存")。
  • --hard:指针回移 + 重置暂存区 + 重置工作区——工作区未提交的改动会被彻底丢弃。
核心区别(必背):reset 是"把历史指针挪回去,假装那些提交没发生",会改写历史,只适合还没 push 的本地提交;revert 是"提交一个把某次改动反做回来的新提交",历史照实向前记录,适合已经 push、共享出去的提交——因为它不改动已有历史,不会坑到队友。reset --hard 是最容易丢代码的命令,动手前先想清楚。

五、后悔药:git reflog

就算不小心 reset 甚至 reset --hard 丢了提交,只要那些提交曾经被提交过(进过对象库),通常都能救回来。因为 Git 记录了 HEAD 每一次移动的日志:

找回丢失的提交
git reflog                     # 列出 HEAD 走过的每一步,找到丢失提交的哈希
git reset --hard a1b2c3d       # 回到那个提交,救回来
呼应快照模型:还记得"Git 一般只添加数据、已提交的很难真丢"吗?reflog 就是这一特性的兜底体现——被 reset 甩掉的提交对象并没删,只是没有分支指向它,reflog 帮你找回那个哈希。但reflog 救不了从没提交过的工作区改动(比如被 git restore 丢弃的),所以真正要小心的永远是"未提交的东西"。

小结

  • 撤销前先定位改动在哪层(工作区 / 暂存区 / 已提交),再选命令。
  • commit --amend 用新提交替换上一次(哈希变),只对未 push 的提交用。
  • restore --staged 取消暂存(安全);restore <file> 丢弃工作区改动(不可 reflog 找回,危险)。
  • 三者定位:restore 管文件、reset 挪分支指针会改写历史(--soft/--mixed/--hard 逐级危险)、revert 追加反向提交最安全。
  • 本地未推送用 reset,已共享用 revert;git reflog 是找回被 reset 丢弃提交的后悔药,但救不了未提交的改动。
