暂存与提交:git add 与 git commit

add 到底往哪儿"加"了什么?commit 又生成了个什么对象?搞清这两个问题,你才真正理解了 Git 的日常。

📖 来源:《Pro Git》 第 2.2 节「记录更新到仓库」

一、git add 的实质:往暂存区写入文件快照

字面上 git add 像"添加文件",但更准确的理解是:把指定文件当前的内容,以快照形式写入暂存区(index)。它的作用有两层:

  • 对未跟踪(untracked)的新文件——把它纳入跟踪并放入暂存区。
  • 对已跟踪但被修改(modified)的文件——把它这一刻的新内容更新进暂存区。
底层发生了什么:Git 会把文件内容算出 SHA-1、存成一个 blob 对象,并在 index 里登记"这个路径 → 这个 blob"。所以"add 记录的是当下的快照"这句话是字面真实——它把内容真的存进了对象库,index 只是记了个指针。
add 的几种用法
git add App.java          # 暂存单个文件
git add src/              # 暂存整个目录(递归)
git add .                 # 暂存当前目录下所有改动(含新增/修改)
git add -p                # patch 模式:同一文件内逐块挑选要暂存的改动

二、"add 后又改了"——那个经典陷阱再复习

因为 add 记录的是 add 那一刻的快照,如果你 add 之后又改了同一个文件,那么新改动并不会自动进入暂存区。此时 git status 会把同一文件同时列在"Changes to be committed"(已暂存的旧版本)和"Changes not staged"(未暂存的新改动)两栏。想把新改动也带上,必须再 add 一次。

三、git commit:把暂存区快照永久记进仓库

commit 只做一件事:把当前暂存区的内容,打包成一个提交对象,永久写入仓库。注意——它提交的是暂存区,不是工作区。没 add 进暂存区的改动,commit 不会带上。

提交
git commit -m "修复登录空指针异常"    # -m 直接给出提交信息,不弹编辑器
git commit                             # 不带 -m 会打开编辑器让你写多行信息

四、commit 对象到底是什么

一次 commit 在 Git 对象库里生成一个提交对象(commit object),它记录了:

  • 一个指向 tree 对象的指针——tree 就是本次提交时整个项目目录结构的快照(哪些文件、各自指向哪个 blob)。
  • 父提交(parent)——指向上一个 commit,正是这一串 parent 指针把提交连成了历史链条(合并提交有两个 parent)。
  • 作者/提交者信息 + 时间(来自你配的 user.name/user.email)。
  • 你写的提交信息。
联系快照模型:还记得"Git 存快照不存差异"吗?commit → tree → blob 这条链就是那张快照的真身。每个 commit 有唯一的 SHA-1(那串 40 位哈希),它就是这个提交的"身份证号",你可以用它(或前几位)引用这次提交。父指针串起来的链,就是 git log 看到的历史。

五、-m 与 -a:两个最常用的参数

参数作用注意
-m "msg"直接指定提交信息,不打开编辑器信息要写清"做了什么、为什么",别写 "update"
-a提交前自动把所有已跟踪文件的改动 add 进来,跳过手动 add只对已跟踪文件有效!新建(untracked)文件不会被带上
-a 的便捷与边界
git commit -a -m "批量提交所有已跟踪文件的改动"
# 等价于:先 git add(仅已跟踪文件) 再 commit
# 但!新建的 new.txt 是 untracked,-a 抓不到它,仍需先 git add new.txt
陷阱:-a 省事,但它绕过了暂存区的精挑细选——会把所有已跟踪文件的改动一股脑提交,容易把不该一起提交的东西混进来。它也不覆盖新文件,新手常以为 commit -am 能提交一切,结果漏掉了新建文件。养成先 git status 确认、再 add 想要的、最后 commit 的习惯更稳。

六、写好提交信息

提交信息是给未来的自己和队友看的。约定俗成:第一行 50 字以内的简短摘要(用祈使句,如 "Fix login NPE"),空一行,再写详细说明"为什么这么改"。清晰的历史让 git log、code review、事故排查都事半功倍。

小结

  • git add = 把文件当前内容以快照写入暂存区(底层:存 blob + 在 index 登记指针);对新文件还兼有"纳入跟踪"之责。
  • add 后再改同一文件,新改动不自动入暂存区,需再 add——同一文件可同时 staged + modified。
  • git commit 提交的是暂存区不是工作区;生成的 commit 对象含 tree(快照)、parent(历史链)、作者、信息,有唯一 SHA-1。
  • -m 直接给信息;-a 自动 add 已跟踪文件的改动但抓不到新建文件,且绕过精挑细选,慎用。
  • 提交信息写清"做了什么、为什么",第一行简短摘要——受益的是未来的自己。
