Git 与其他 VCS 的区别:快照,而非差异

几乎所有 Git 的"奇怪行为"和"惊人速度",都能从这一句话推导出来:Git 存的是快照,不是差异。

📖 来源:《Pro Git》 第 1.3 节「Git 是什么」

一、传统 VCS 怎么存:基于差异(delta)

CVS、SVN、Perforce 这类系统,把数据看成一组文件以及每个文件随时间累积的差异。它们记录的是"文件 A 在版本 1 到版本 2 之间,第 3 行从 X 改成了 Y"这样的变更(delta)。想得到某个版本的完整文件,需要拿基础版本 + 一路叠加所有 delta 算出来。

这叫基于差异的版本控制(delta-based)。它省空间,但代价是:每次要还原一个历史版本,都可能需要重放一长串 patch,版本越久、计算越慢。

二、Git 怎么存:每次提交都是一张完整快照

Git 把数据看成一系列文件系统的快照(snapshot)。每次你 commit,Git 会给当前项目里所有文件的状态拍一张全景照片,然后保存这个快照的一个引用。

关键的省空间技巧:如果某个文件这次没变,Git 不会重复存它,而只存一个指向上一次那个完全相同文件的指针。所以"快照"不等于"每次都全量复制一遍",没变的文件靠指针复用——既有快照的速度,又不浪费空间。

用一句话概括这个差异:SVN 想的是"这个文件改了什么",Git 想的是"这次提交时整个项目长什么样"。

基于差异(SVN)基于快照(Git)
存储单位每个文件的一串变更每次提交时整棵文件树的快照
未改动的文件不涉及(该文件无 delta)用指针指向上一版的相同文件,不重存
还原某版本叠加一长串 patch 计算得出直接取那张快照

三、为什么这个设计带来"几乎所有操作都是本地的、快到离谱"

因为整个仓库(所有快照、所有历史)都完整躺在你本地磁盘上,而且历史版本就是现成的快照——查看历史、对比版本、切换分支,几乎不需要任何网络,也不需要重放 delta 计算,大多是查表和移动指针的操作。这就是为什么用惯 SVN 的人第一次用 Git 会觉得"怎么这么快"。

联系底层:Git 内部把每个文件内容、每棵目录树、每次提交都作为一个对象存进对象数据库,用内容的 SHA-1 哈希作为地址(后面「Git 内部原理」会展开)。"没变的文件复用指针"本质就是:内容没变 → 哈希相同 → 指向同一个对象。这也是 Git 能做到完整性校验的原因——任何内容被悄悄篡改,哈希都会对不上。

四、一切皆有校验和:Git 的完整性保证

Git 在存储任何东西之前,都会先对其内容算一个 SHA-1 校验和(40 位十六进制,如 24b9da6552252987aa493b52f8696cd6d3b00373),并用这个校验和来引用它。这意味着:你不可能在 Git 不知情的情况下改动任何文件内容或损坏任何数据——因为内容一变,哈希就变了,Git 立刻能察觉。

五、Git 一般只添加数据

你在 Git 里做的几乎所有操作,本质都是往数据库里添加新数据,而很难真正"删掉"什么。一旦你提交了快照(尤其是 push 过之后),它就很难丢失。哪怕你 reset 把分支指针挪走了,那个提交对象通常还静静躺在数据库里,可以靠 git reflog 找回来。

心态转变:这条特性让 Git "用起来很安全"——可以大胆试验分支、大胆改。真正会导致数据丢失的,主要是那些还没提交的改动(工作区里没 add 也没 commit 的东西)。所以口诀是:常 commit,想丢东西没那么容易。

小结

  • 传统 VCS 存"每个文件随时间的差异";Git 存"每次提交时整棵文件树的快照"。
  • 没变的文件不重复存,只存一个指向旧对象的指针——快照既快又不浪费空间。
  • 整仓库带完整历史在本地 + 历史是现成快照 ⇒ 操作几乎全在本地、极快、可离线。
  • 一切内容用 SHA-1 校验和寻址 ⇒ 天然完整性校验,内容一改哈希就变。
  • Git 大多只"添加"数据,已提交的东西很难真丢——所以要养成勤提交的习惯。
