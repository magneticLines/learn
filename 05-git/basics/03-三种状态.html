三个区域与三种状态:工作区 / 暂存区 / 仓库

这是整个 Git 使用心智模型的地基。理解了这三个区域,后面所有命令(add / commit / diff / reset / restore)才不会靠死记硬背。

📖 来源:《Pro Git》 第 1.3 节「三种状态」

一、三个区域:文件在 Git 里经过的三站

一个 Git 项目里,你的文件在三个地方之间流动:

区域物理位置是什么
工作区
Working Directory		你能直接看到、用编辑器打开的那些文件从 Git 仓库中检出的某个版本的实际文件,你在这里改代码
暂存区
Staging Area / Index		.git/index 这个文件一份"下次提交要包含哪些改动"的清单/快照
仓库
Repository		.git 目录Git 存放所有对象和历史元数据的地方,commit 后的快照都在这
暂存区的正式名字是 index(索引)。你在很多命令输出和文档里会看到 "index" 这个词,它和"暂存区(staging area)"是同一个东西,别被两个叫法搞混。

二、三种状态:文件相对这三个区域处于什么状态

一个已被 Git 跟踪的文件,在任意时刻处于以下三种状态之一:

  • 已修改(modified):你在工作区改了它,但这个改动还没被放进暂存区。
  • 已暂存(staged):你对它当前版本做了标记,下次提交时会带上这个版本(已经 git add 过了)。
  • 已提交(committed):数据已经安全地保存进本地仓库(.git),成为一个历史快照。

三、标准工作流:改 → add → commit

基本三步走
# 1. 在工作区修改文件(用编辑器改 App.java)……此时它是 modified
# 2. 把改动放进暂存区,标记"这个版本要进下次提交"
git add App.java          # App.java 变成 staged

# 3. 把暂存区里那份快照永久记录进仓库
git commit -m "修复登录空指针"   # 生成一个 commit,文件变成 committed
最经典的新手陷阱:git add 后又改了同一个文件,直接 commit 会漏掉"add 之后的那次改动"。因为 add 记录的是你 add 那一刻的文件快照,后续再改的内容并没有自动进入暂存区。所以你会看到同一个文件同时出现在"已暂存"和"未暂存"两栏——需要再 add 一次才会把新改动也纳入。

四、暂存区为什么存在?——它才是 Git 真正的巧思

很多人第一反应是:"为什么不能改完直接提交,非要多一步 add?" 因为暂存区提供了一个可以精心构造"下一次提交长什么样"的缓冲层,好处非常实际:

  • 分批提交(拆分逻辑):你可能一口气改了三个不相关的地方(修了个 bug、顺手改了个文案、动了个配置)。可以只 add 修 bug 的那部分单独提交,让每个 commit 只做一件事,历史清晰、方便回退和 review。
  • 构造"干净的提交":甚至可以用 git add -p(patch 模式)在同一个文件内部只挑其中几行加入本次提交,把无关的调试代码留在工作区。
  • 提交前最后检查:暂存区就是"待提交清单",commit 前可以用 git diff --staged 确认这次到底要提交什么,避免误提交。
建立直觉:把暂存区想成购物车——工作区是货架(所有商品/改动),你把想一起结账的东西放进购物车(add),结账(commit)时只买购物车里的,货架上其他东西原地不动。这个类比能解释几乎所有 add/commit 的行为。

五、还有一种文件:未跟踪(untracked)

上面三种状态针对的是"已被 Git 跟踪"的文件。新建的、Git 还没见过的文件属于未跟踪(untracked)——它既不在暂存区也不在仓库历史里,git status 会单独列出来。第一次 git add 会同时完成两件事:把它纳入跟踪并放入暂存区。

小结

  • 三个区域:工作区(实际文件)、暂存区/index(.git/index,下次提交清单)、仓库(.git,历史快照)。
  • 三种状态:modified(改了没 add)、staged(add 了待提交)、committed(已进仓库)。
  • add 记录的是 add 那一刻的快照,add 后再改要重新 add——同一文件可同时处于 staged 和 modified。
  • 暂存区的意义:像购物车,支持分批提交、按行挑选、提交前审查,让每个 commit 干净、单一。
  • Git 没见过的新文件是 untracked,首次 add 会同时"纳入跟踪 + 放进暂存区"。

与其他知识点的联系

  • 撤销操作(restore/reset/revert):撤销命令本质上都是在这三个区域之间"搬动"或"回退"数据——restore 是在工作区/暂存区内操作,reset 是移动分支指针同时联动重置暂存区和(可选)工作区。不先搞清楚三个区域各自是什么,就无法理解为什么同样叫"撤销"的命令危险程度天差地别。
  • .gitignore:.gitignore 的生效边界正好卡在"未跟踪"和"已跟踪"这条线上——它只对还没进过暂存区/仓库的文件有效。这说明 .gitignore 不是一个孤立机制,而是依附在三种状态模型之上的一层过滤规则。
  • Git 对象模型(blob/tree/commit):git add 时 Git 会把文件内容算成一个 blob 对象存进对象库,暂存区(index)记录的其实是"文件路径 → blob 哈希"的映射;commit 则是把当前暂存区的映射打包成 tree 对象,再包一层 commit 对象。三种状态的底层,就是对象在这几层结构间的移动。
  • 远程仓库与 push/pull:三种状态只描述本地单个仓库内部的流转,还没涉及"仓库之间"的同步。commit 完成只是让改动进了本地仓库,要让远程也看到,还需要 push 这一步——这也是很多新手以为"commit 了就等于交上去了"的常见误区来源。
  • CI/CD 流水线:CI 系统触发构建通常依据的是"某个分支收到了新的 commit",也就是数据完成了从工作区到暂存区再到仓库的完整流转,并被推送到远程。理解这条流转链路,才能理解 CI 到底在监听什么事件。
