查看状态与差异:status 与 diff / diff --staged

"我明明改了文件,git diff 怎么什么都不显示?"——这是新手最高频的困惑,答案全在这一篇。

📖 来源:《Pro Git》 第 2.2 节「检查当前文件状态 · 查看已暂存和未暂存的修改」

一、git status:全局体检

git status 回答"现在仓库处于什么状态":哪些文件未跟踪、哪些已修改未暂存、哪些已暂存待提交、当前在哪个分支、和远程差了几个提交。它是你每次操作前后都该随手一敲的命令。

git status 典型输出
On branch main
Changes to be committed:          # ← 已暂存(staged),下次 commit 会带上
        modified:   config.yml
Changes not staged for commit:     # ← 已修改但未暂存(modified)
        modified:   App.java
Untracked files:                   # ← Git 还没跟踪的新文件
        notes.txt
更省屏幕的写法:git status -s(或 --short)一行一个文件,左列表示暂存区状态、右列表示工作区状态。M = 已暂存的修改, M= 未暂存的修改,MM= 两者都有(add 后又改了),??= 未跟踪。

二、核心困惑:git diff 默认比的是什么

关键要记住 Git 有三个区域(工作区 / 暂存区 / 仓库)。"差异"必然是两个区域之间的比较,所以 diff 有两种最常用的方向:

命令比较的两端回答的问题
git diff工作区 ↔ 暂存区"我改了、但还没 add 的内容是什么"
git diff --staged
(= --cached)		暂存区 ↔ 最新提交(仓库)"我已经 add、下次 commit 会提交的内容是什么"
git diff HEAD工作区 ↔ 最新提交"相对上次提交,我一共改了多少(不管 add 没 add)"

三、为什么"改了却 diff 不出来"——真相

破案:你改完文件,git add 了,然后敲 git diff——空的!因为 git diff 比的是工作区 vs 暂存区,而你 add 之后这两者内容一致,自然没差异。你要看的其实是暂存区 vs 仓库,应该用 git diff --staged。
反过来:如果你改了但没 add,那么改动在"工作区 vs 暂存区"这一层,git diff 能看到,而 git diff --staged 是空的。

把这个规律记牢:没 add 的改动 → 用 git diff;已 add 的改动 → 用 git diff --staged;想一次看全部(不区分是否 add) → 用 git diff HEAD。

四、读懂 diff 输出

git diff 输出片段
diff --git a/App.java b/App.java       # a/ 是旧版本,b/ 是新版本
@@ -12,6 +12,7 @@                        # 旧文件从第12行起6行,新文件从第12行起7行
 public void login() {              # 前面有空格 = 未改动的上下文行
+    if (user == null) return;          # + 开头 = 新增的行
-    doLogin(user);                     # - 开头 = 删除的行

@@ ... @@ 叫 hunk 头,标明改动发生在文件哪一段。+ 是新增,- 是删除,前导空格是保留不变的上下文——这套格式源自 Unix 的 diff 工具,是行级 diff 算法(最长公共子序列 LCS 的变体)的标准表示。

五、几个实用变体

常用 diff 变体
git diff --stat                # 只看每个文件加了几行、删了几行的统计,不看具体内容
git diff HEAD~1 HEAD            # 比较上一个提交与当前提交
git diff main feature          # 比较两个分支的差异
git diff -- App.java           # 只看某个文件的 diff
建立联系:还记得"Git 存快照不存差异"吗?那 diff 从哪来?——Git 存的是快照,但需要时会即时算出两张快照之间的差异展示给你。存快照保证了速度和完整性,展示差异则靠现算,两者并不矛盾:存储方式和展示方式是两回事。

小结

  • git status 是全局体检,报告未跟踪/未暂存/已暂存/分支状态;-s 给出紧凑双列视图。
  • git diff(无参)= 工作区 vs 暂存区,看"还没 add 的改动"。
  • git diff --staged/--cached = 暂存区 vs 最新提交,看"已 add、下次会提交的改动"。
  • "改了却 diff 不出来"多半是已经 add 了,该用 --staged;想不分区一次看全用 git diff HEAD。
  • Git 底层存快照,diff 是需要时现算出来的——存储与展示是两回事。
