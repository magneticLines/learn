二分排错 bisect:把二分查找搬进提交历史

"上周还好好的,今天就崩了,中间几百个提交,是哪个引入的?"——bisect 用 O(log n) 次测试帮你揪出真凶。

📖 来源:《Pro Git》 第 7.10 章「使用 Git 调试」

一、原理:这就是二分查找算法

假设你有一段连续的提交历史,某个"好"的旧提交功能正常,某个"坏"的新提交出了 bug。bug 一定是这中间某一个提交引入的。如果线性一个个试,最坏要测 n 次。但历史是有序的(旧→新),而且"bug 一旦引入就一直存在"具有单调性——这正是二分查找的前提条件。

于是 git bisect 每次取区间中点的提交让你测试:如果中点是坏的,真凶在前半段;如果中点是好的,真凶在后半段。每测一次区间砍半,n 个提交只需约 log₂(n) 次测试——1000 个提交只要测约 10 次。

建立联系:这和数组里的二分查找是同一个算法,只不过"数组下标"换成了"提交历史的位置","查找目标值"换成了"从好变坏的那个分界点"。前提也一样:必须单调有序。

二、手动 bisect 流程

bash · start / bad / good / reset
git bisect start          # 开始二分会话
git bisect bad             # 标记当前(最新)提交是坏的
git bisect good v1.0       # 标记一个已知正常的旧提交/标签是好的

# Git 自动 checkout 到区间中点的提交,你去编译/运行/测试……
git bisect good            # 测出来这个中点是好的 → 真凶在更新的一半
# 或者
git bisect bad             # 这个中点是坏的 → 真凶在更旧的一半

# 反复几次后,Git 报告:"xxxxxx is the first bad commit"
git bisect reset           # 结束,回到 bisect 开始前的分支/提交(务必执行,否则停留在游离 HEAD)
陷阱:bisect 过程中你处于游离 HEAD(detached HEAD)状态,Git 会不断把你 checkout 到各个历史提交上。测完一定要 git bisect reset 回到原来的分支,否则你以为自己在 main,其实停在某个中间提交上,后续提交会"无家可归"。

三、自动化:bisect run

如果你能写一个脚本自动判断"好/坏"(退出码 0=好,1~127 且非 125=坏),就能让 Git 全自动跑完整个二分过程,你去喝杯咖啡即可。

bash · 全自动二分
git bisect start HEAD v1.0    # 一行同时给出 bad(HEAD)和 good(v1.0)
git bisect run ./test.sh     # 每个中点自动跑脚本判定好坏,直到定位

# 也可以直接跑测试命令,比如:
git bisect run mvn -q test -Dtest=BugRepoTest
约定 125 退出码:如果某个中点提交根本无法判定(比如编译不过、无法测试),让脚本返回退出码 125,Git 会跳过这个提交换一个附近的来测,而不会误判为好或坏。这是写 bisect run 脚本时的关键约定。

四、什么样的 bug 适合 bisect

  • 可稳定复现的 bug——每次测试结果确定,才能保证二分的单调前提。偶发/竞态 bug 不适合。
  • 有一个明确"好"的历史锚点——比如上个 release 标签,能界定二分区间的下界。
  • 提交历史大致可编译可运行——否则要靠 125 跳过大量无法测试的中间提交。

小结

  • bisect 就是把二分查找应用到提交历史,利用"bug 引入后持续存在"的单调性,log₂(n) 次测试定位首个坏提交。
  • 手动流程:start → bad 标坏 → good 标好 → 反复测中点标 good/bad → reset 收尾。
  • git bisect run 脚本 全自动化;退出码 0=好、1~127=坏、125=跳过无法判定的提交。
  • 结束务必 reset 脱离游离 HEAD;bug 需可稳定复现、有明确好锚点。
