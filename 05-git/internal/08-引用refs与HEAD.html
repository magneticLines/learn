引用 refs 与 HEAD:分支只是指向哈希的指针文件

上一篇建了对象模型的地基,这一篇揭穿一个"魔法":所谓分支、标签、HEAD,不过是 .git 里几个存着哈希的小文本文件。

📖 来源:《Pro Git》 第 10.3 章「Git 引用」

一、40 个字符哈希太难记,于是有了 refs

对象模型告诉我们:每个提交由一个 SHA-1 哈希唯一标识。但没人愿意记 a0b34bb... 这种字符串。引用(reference,ref)就是给哈希起的一个"人类可读的别名"——本质是一个文本文件,里面存着一个提交对象的哈希。它们都放在 .git/refs/ 目录下。

bash · 亲眼看看分支就是个文件
cat .git/refs/heads/main
# a0b34bb1f2...    ← 就这一行,40 位哈希,别无他物

# 所以"创建分支"= 写一个存着当前提交哈希的文件,几乎零成本
git branch feature       # 等价于 echo <当前哈希> > .git/refs/heads/feature
为什么 Git 建分支这么快:SVN 建分支要复制整个目录树,Git 只是写一个 41 字节的文件(40 字符 + 换行)。这就是"Git 鼓励频繁开分支"的底气——分支廉价到几乎免费。

二、refs 的三大命名空间

目录存什么访问名
refs/heads/本地分支,每个文件指向该分支最新提交main、feature
refs/tags/标签,通常指向某个固定提交(发版打点)v1.0
refs/remotes/远程跟踪分支,记录你上次同步时远程分支的位置origin/main
建立联系:origin/main 不是远程实时状态,而是你本地缓存的"上次 fetch 时远程 main 在哪"。这解释了为什么 git status 说"落后 origin/main 2 个提交"却不自动更新——它比的是本地缓存的 ref,只有 git fetch 才会刷新 refs/remotes/。

三、轻量标签 vs 附注标签

标签有两种,底层不同:

  • 轻量标签(lightweight):和分支一样,就是 refs/tags/ 下一个直接指向 commit 哈希的文件,只是它不会移动。
  • 附注标签(annotated):git tag -a v1.0 -m "发版" 会创建一个独立的 tag 对象(Git 的第四种对象),里面存打标签者、日期、说明、GPG 签名,再由它指向 commit。正式发版应用附注标签。

四、HEAD:指向"当前在哪个分支"的符号引用

前面的 ref 直接存哈希,而 HEAD 特殊:它通常存的不是哈希,而是"指向另一个引用"的符号引用(symbolic ref)——它记录"我当前在哪个分支上"。

bash · HEAD 里存的是什么
cat .git/HEAD
# ref: refs/heads/main    ← 不是哈希!是"我现在在 main 分支上"

# 所以 git commit 时,Git 顺着 HEAD → main → 更新 main 指向新提交
# git checkout feature 就是把 HEAD 改写成 ref: refs/heads/feature

这套两级间接(HEAD → 分支ref → commit哈希)解释了日常行为:你 commit 时 Git 顺着 HEAD 找到当前分支,把该分支的 ref 更新到新提交;你 checkout 另一分支时,只是改写 HEAD 里那行文本。

游离 HEAD(detached HEAD):如果你 git checkout <某个提交哈希> 而不是分支,HEAD 里会直接存那个哈希,而不是 ref: refs/heads/xxx。此时的新提交不属于任何分支,一旦切走就成悬空对象易丢失——这正是 bisect、submodule 里游离 HEAD 警告的底层原因。

五、packed-refs:引用太多时的优化

一个大项目可能有成千上万个标签和分支。若每个 ref 都是一个独立小文件,目录会塞满碎文件,遍历也慢。于是 Git 会在 git gc 时把这些松散的 ref 文件打包进一个 .git/packed-refs 文件(一行一个 ref),删除原来的单个文件。

bash · packed-refs 长这样
cat .git/packed-refs
# a0b34bb... refs/heads/main
# c1d2e3f... refs/tags/v1.0
# ...
查找顺序:Git 解析一个 ref 时,先看 refs/ 下有没有对应的松散文件,没有再去 packed-refs 里找。所以打包后你更新了某个分支,它会以松散文件形式重新出现,优先级更高——两种形式共存并不矛盾。别手动编辑 packed-refs,用 git update-ref 等命令操作。

小结

  • ref 是给 SHA-1 哈希起的可读别名,本质是 .git/refs/ 下存着哈希的文本文件;建分支=写个小文件,所以极快。
  • 三大命名空间:refs/heads/(本地分支)、refs/tags/(标签)、refs/remotes/(远程跟踪,是本地缓存非实时)。
  • HEAD 是符号引用,通常存 ref: refs/heads/xxx 表示当前分支;checkout 提交哈希会进入游离 HEAD,新提交不属于任何分支易丢。
  • ref 过多时 git gc 把它们打包进 packed-refs;解析时松散文件优先于打包文件。
