储藏 stash:把没做完的活先塞进抽屉

改到一半突然要切分支救火,又不想凑合提交一个半成品——stash 让你把工作现场原样收起来,回头再原样铺开。

📖 来源:《Pro Git》 第 7.3 章「储藏与清理」

一、它解决什么问题

Git 有个约束:切换分支时,如果当前改动会与目标分支冲突,Git 会拒绝切换。而你手头改了一半,还没到能提交的程度,直接提交又会留下一个"WIP"垃圾提交污染历史。git stash 就是为这个尴尬场景设计的:它把工作区 + 暂存区的所有未提交改动打包收起来,让工作区瞬间变干净(回到 HEAD 状态),你就能自由切分支了;活干完回来再取出来接着改。

二、核心命令

bash · stash 常用操作
git stash                        # 收起改动,工作区变干净(等价 git stash push)
git stash push -m "改到一半的登录页"  # 带说明,list 时更好认

git stash list                   # 列出所有储藏:stash@{0}、stash@{1}...(栈,越新编号越小)

git stash apply                   # 取出改动但【保留】储藏(可重复应用到多个分支)
git stash pop                     # 取出改动并【删除】储藏(=apply + drop)
git stash pop stash@{1}          # 指定取哪一个,不指定默认最新的 {0}

git stash drop stash@{0}         # 删除某个储藏
git stash clear                   # 清空所有储藏(不可轻易恢复)
apply vs pop:核心区别是 pop 成功后会自动删掉储藏,apply 不删。习惯用 apply 更保险——万一取出来发现应用到了错误的分支,储藏还在,可以再来一次;确认无误后再手动 drop。

三、揭底:stash 其实是几个"隐藏的提交"

stash 不是什么魔法。它的实现基于 Git 的对象模型:git stash 会悄悄创建提交对象来保存现场——通常是两到三个:一个记录暂存区的状态,一个记录工作区的状态,再用一个 commit 把它们打包,挂到特殊引用 refs/stash 上。

bash · 印证 stash 的本质是提交
git stash
git log --oneline stash@{0}   # 能看到 stash 生成的提交,还带着父提交
cat .git/refs/stash          # 里面就是一个提交对象的 SHA-1
建立联系:正因为 stash 底层就是提交对象,即使你误 drop 了一个储藏,它对应的提交对象短期内还在对象库里(悬空状态),可以用 git fsck --unreachable 或 git stash 相关 reflog 找回——和上一篇 reflog 找回提交是同一个原理。

四、常见选项

选项作用
git stash -u连未跟踪(untracked)的新文件一起储藏(默认不含新文件,是常见坑)
git stash -a连被 .gitignore 忽略的文件也储藏
git stash --keep-index储藏后,已暂存的改动仍留在工作区(方便只测试暂存部分)
git stash branch <name>基于储藏创建时的提交新建分支并弹出储藏,适合应用时有冲突的场景
陷阱:默认 git stash 不会储藏未跟踪的新文件。如果你新建了文件还没 git add,stash 后它们会原地不动地留在工作区——切分支时反而可能带来意外。要连新文件一起收,记得加 -u。

五、典型场景

  1. 临时切分支救火:正在 feature 分支开发,线上告急要去 main 修 bug。git stash → 切 main → 修复提交 → 切回 feature → git stash pop 继续。
  2. 拉取远程更新前清场:git pull 遇到本地改动阻塞,先 stash 再 pull,pull 完 pop 回来(不过更推荐 git pull --rebase --autostash 自动完成)。
  3. 换个基准试验:把当前改动 stash,checkout 到别的提交上,apply 同一个储藏做对比测试。

小结

  • stash 把工作区 + 暂存区的未提交改动打包收起,让工作区变干净,解决"改到一半要切分支"的两难。
  • apply 取出保留储藏、pop 取出并删除;list 看栈、drop/clear 删除。推荐 apply 更保险。
  • 底层就是几个悬空的提交对象挂在 refs/stash 上,所以误删也可能找回。
  • 默认不含未跟踪文件,需 -u;拉取场景更推荐 git pull --rebase --autostash。
