reset 三种模式:soft / mixed / hard

同一个命令,一个参数之差,轻则挪一下指针,重则抹掉你的工作区改动——搞懂"三棵树"才能真正驾驭 reset。

📖 来源:《Pro Git》 第 7.7 章「重置揭密」、第 2 章

一、先立地基:Git 的"三棵树"

要讲清楚 reset,必须先建立一个模型:Git 在你本地管理着三棵树(three trees)——不是数据结构里的二叉树,而是"三份文件快照"。理解 reset 的一切,都是理解它对这三棵树各自做了什么。

树是什么你怎么看它
HEAD你当前所在分支最后一次提交的快照——"上次提交长啥样"git cat-file -p HEAD^{tree}
Index(暂存区)你下一次准备提交的快照——"预演的提交"git ls-files -s
工作区你磁盘上真实的文件——"你正在编辑的东西"直接看文件

正常工作流其实就是在三棵树之间搬运数据:git add 把工作区的改动搬进暂存区,git commit 把暂存区快照固化成新的 HEAD。而 reset 做的是反方向的事,它按三个层次逐级往下"倒":先移 HEAD,再重置暂存区,再覆盖工作区。三种模式就是控制它"倒"到哪一层为止。

二、soft:只移动 HEAD 指针

--soft 只做一件事:把当前分支指针指向目标提交。暂存区和工作区一动不动。这意味着被"撤销"的那些提交里的改动,现在原封不动地躺在暂存区里,随时可以重新提交。

bash · soft:合并最近 3 个提交为 1 个
# 把 HEAD 退回 3 个提交前,但这 3 个提交的所有改动都还在暂存区里
git reset --soft HEAD~3
# 此时 git status 显示一堆已暂存的改动,直接重新提交即可
git commit -m "合并成一个干净的提交"

典型用途:压缩(squash)提交——把凌乱的多个 WIP 提交揉成一个。因为工作区和暂存区都没动,所以 soft 是三者中最安全的。

三、mixed:移 HEAD + 重置暂存区(默认)

git reset 不带参数时,默认就是 --mixed。它在 soft 的基础上多走一步:把暂存区也重置成目标提交的样子,但不碰工作区。

bash · mixed:撤销 add,或撤销提交但保留改动
# 场景1:取消暂存(取消一次 add),改动退回工作区
git reset HEAD file.txt   # 不指定提交=HEAD,只重置该文件的暂存状态

# 场景2:撤销上一次提交,但代码改动全部保留在工作区(未暂存)
git reset HEAD~1       # 等价于 --mixed,提交没了,改动还在,需要重新 add
为什么默认是 mixed:它是最"中庸"的——既撤销了提交(移 HEAD),又把改动退回工作区让你重新决定怎么组织,还不会丢失你磁盘上的劳动成果。日常"提交提早了想重来"用它最顺手。

四、hard:连工作区一起重置(危险)

--hard 三棵树全部对齐目标提交:移 HEAD、重置暂存区、并且强制覆盖工作区。你磁盘上还没提交、还没暂存的改动会被直接丢弃,不进回收站。

bash · hard:彻底放弃,回到某个提交的干净状态
# 三棵树全部重置——工作区改动会被无声抹掉!
git reset --hard HEAD~1
# 常见变体:放弃所有本地改动,回到与提交完全一致
git reset --hard HEAD
陷阱:--hard 是新手事故重灾区。它只对已被 Git 追踪的改动负责——未提交的改动被覆盖后无法用常规手段找回(还没进对象库的东西根本没被 Git 存过)。已经 commit 过的提交则不同:即使被 hard reset "丢掉",提交对象还在对象库里,可以用 reflog 找回(见下一篇)。

五、三种模式对照表

模式移 HEAD重置暂存区覆盖工作区危险度
--soft✅❌❌低
--mixed(默认)✅✅❌中
--hard✅✅✅高

记忆口诀:soft 动一层,mixed 动两层,hard 动三层——层数越多越危险,因为越往下越接近你真实的、可能还没保存进 Git 的劳动成果。

六、对比 revert:改写历史 vs 追加历史

面试常问:"撤销一个提交,用 reset 还是 revert?"关键区别在于是否改写历史。

git resetgit revert
做法移动分支指针,让旧提交"消失"新建一个"反向提交"抵消目标改动
历史改写历史(旧提交从分支线上消失)追加历史(旧提交仍在,多一个撤销提交)
适用本地、未推送的分支已推送、多人共享的分支
黄金准则:已经 push 到远程、别人可能已经拉取的提交,不要用 reset 改写——否则你 force push 后,同事的本地历史会与远程分叉,一片混乱。共享分支上的撤销一律用 revert,它只追加不改写,对所有人都安全。

小结

  • Git 三棵树:HEAD(上次提交)、Index(暂存区/下次提交)、工作区(磁盘真实文件)。reset 就是逐级把它们往目标提交对齐。
  • --soft 只移 HEAD(改动留暂存区,用于压缩提交);--mixed(默认)移 HEAD + 重置暂存区(改动退回工作区);--hard 三棵树全重置,会丢工作区未提交改动。
  • 层数越多越危险;hard 丢掉的已提交内容可用 reflog 找回,但未提交的改动一般找不回。
  • reset 改写历史,只用于本地未推送分支;共享分支上撤销请用 revert(只追加不改写)。
