reflog 引用日志:误操作后的后悔药

hard reset 把提交弄丢了?分支删错了?rebase 崩了?先别慌——只要 HEAD 曾经指向过它,reflog 就记着账。

📖 来源:《Pro Git》 第 7.7 章、第 10.3 章「维护与数据恢复」

一、reflog 是什么:HEAD 的移动流水账

reflog(reference log,引用日志)记录的是你本地引用(HEAD、各分支)的指针每一次移动的历史。每当你 commit、checkout、reset、merge、rebase——凡是让 HEAD 或某个分支指向了新提交,Git 就在 reflog 里追加一行:"某时某刻,HEAD 从 A 移到了 B,因为你执行了 X 操作"。

bash · 查看 HEAD 的移动历史
git reflog
# 输出形如:
# a0b34bb HEAD@{0}: commit: 补齐数据结构文件
# 056bd2b HEAD@{1}: reset: moving to HEAD~1   ← 就是这次 reset 弄丢了 a0b34bb
# fe210c6 HEAD@{2}: commit: 添加文档

其中 HEAD@{n} 表示"HEAD 在 n 次移动之前指向的位置"。这是理解 reflog 的关键语法。

二、它凭什么能当"后悔药"

要理解 reflog 为何有效,得回到 Git 的对象模型:一个提交只要被创建,它的提交对象就永久地躺在 .git/objects 里,不会因为"没有分支指向它"就立刻消失。当你 hard reset 掉一个提交,只是让分支指针不再指向它——提交对象本身还在,只是变成了"悬空(dangling)"、没人引用的孤儿。

联系上一篇:hard reset 丢掉的已提交内容能找回,正是因为提交对象还在对象库、且 reflog 里还留着它的哈希。而未提交的工作区改动从没进过对象库,所以 reflog 也救不了它——这条界线很重要。

三、实战:hard reset 后找回提交

bash · 从误删中恢复
# 惨案:一时手抖,丢了最近 3 个提交
git reset --hard HEAD~3

# 抢救第一步:看 reflog,找到丢失前 HEAD 指向的那个哈希
git reflog
# 找到:a0b34bb HEAD@{1}: commit: ...(reset 之前的位置)

# 抢救第二步:让当前分支重新指回那个提交
git reset --hard a0b34bb
# 或者更稳妥——先建个分支保住它,确认无误再合并
git branch recover a0b34bb

四、实战:找回被删除的分支

删分支(git branch -D feature)只是删掉了那个指针文件,分支尖端的提交对象还在。

bash · 恢复被删分支
# reflog 默认只记 HEAD;分支自己的日志用 git reflog show
git reflog # 在里面找到该分支最后一次 commit/checkout 的哈希
# 用那个哈希重新创建分支
git branch feature <哈希>
rebase 救援:rebase 出错想回到 rebase 之前的状态,不用记哈希——Git 提供了别名 ORIG_HEAD,它保存了危险操作(reset/rebase/merge)前 HEAD 的位置。直接 git reset --hard ORIG_HEAD 即可一键还原。

五、reflog 的边界与过期清理

reflog 不是万能的,有三条硬边界:

  • 本地专属:reflog 存在 .git/logs/ 下,不会随 push/clone 传播。你的后悔药救不了别人的仓库,别人也看不到你的 reflog。
  • 会过期:reflog 条目默认保留期是——可达提交 90 天、不可达(悬空)提交 30 天。由 gc.reflogExpire / gc.reflogExpireUnreachable 控制。
  • 会被 gc 清理:reflog 过期后,那些只被 reflog 引用的悬空提交就真正失去引用,会被 git gc 当垃圾回收掉——到那时才是真的找不回了。
bash · 主动清理(谨慎)
git reflog expire --expire=now --all   # 立即过期所有 reflog
git gc --prune=now                     # 随后 gc 会真正删除悬空对象
陷阱:不要在慌乱时先跑 git gc——如果你刚丢了提交,gc + prune 可能把你还想找回的悬空对象直接清掉。找回顺序永远是:先 reflog 定位、建分支保住,再谈清理。

小结

  • reflog 记录本地 HEAD/分支指针的每一次移动,格式 HEAD@{n}=n 次移动前的位置。
  • 它能当后悔药,是因为提交对象一旦创建就留在对象库,hard reset/删分支只是移除了引用,对象还在。
  • 找回套路:git reflog 定位哈希 → git branch/reset 重新引用;rebase 前状态可用 ORIG_HEAD 一键还原。
  • 边界:reflog 本地专属不随 push 传播;条目默认 30/90 天过期;过期后 git gc 会真正回收——慌乱时先找回再清理。
