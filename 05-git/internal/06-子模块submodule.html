子模块 submodule:在仓库里嵌一个仓库

想把某个第三方库或共享代码当独立仓库维护,又要嵌进主项目里锁定版本?submodule 就是干这个的——但它的坑也是出了名的多。

📖 来源:《Pro Git》 第 7.11 章「子模块」

一、它解决什么问题

你的主项目依赖一个内部共享库,这个库有自己独立的仓库、独立的版本节奏。你既不想把库的代码复制粘贴进主项目(无法同步更新),也不想把两个仓库硬合成一个(丧失独立维护)。submodule 的方案是:在主项目里放一个"指针",指向子仓库的某一个具体提交。主仓库只记录"我依赖子库的哪个 commit",子库代码仍在它自己的仓库里管理。

二、核心机制:主仓库只存一个提交指针

这是理解 submodule 一切行为的关键:主仓库不存子模块的文件内容,它在自己的 tree 里记录的是子模块目录 = 一个gitlink(特殊类型条目,值是子仓库的一个 commit SHA)。加上一个 .gitmodules 文件记录子模块的 URL 和路径。

bash · 添加子模块后生成了什么
git submodule add https://example.com/libfoo.git libs/foo

# 生成 .gitmodules(受版本控制,记录 URL 与路径):
# [submodule "libs/foo"]
#   path = libs/foo
#   url  = https://example.com/libfoo.git

# 主仓库暂存区里,libs/foo 是一个 commit 指针,不是普通目录:
git diff --cached
# +Subproject commit a1b2c3d...   ← 就是子仓库的一个提交哈希

三、克隆与初始化(最大的坑)

普通 git clone 主仓库,子模块目录会是空的——因为主仓库只存了指针,没存内容。

bash · 正确拉取带子模块的仓库
# 一步到位:克隆时递归拉取所有子模块
git clone --recursive https://example.com/main.git

# 如果已经普通 clone 了才发现子模块是空的,补救:
git submodule update --init --recursive
# init:根据 .gitmodules 注册子模块;update:拉取并 checkout 到主仓库记录的那个提交
头号坑——忘记初始化:协作者 clone 后不加 --recursive,子模块目录空空如也,一编译就报"找不到文件",还以为是你没提交。团队里凡是用了 submodule,一定要在 README 显眼处写明"克隆要加 --recursive"。

四、游离 HEAD 与更新

当你进入子模块目录,会发现它处于游离 HEAD状态——因为主仓库记录的是"某个提交",子模块被 checkout 到了那个具体 commit 上,而不是某个分支。

bash · 更新子模块到最新
# 想改子模块代码,先进去切到真正的分支,否则改动挂在游离 HEAD 上易丢
cd libs/foo && git checkout main

# 让子模块跟进它远程的最新提交
git submodule update --remote libs/foo
# 之后回主仓库,会看到子模块指针变了,需要在【主仓库】提交这个指针变更
cd ../.. && git add libs/foo && git commit -m "升级 foo 到新版本"
陷阱——游离 HEAD 丢改动:直接在子模块的游离 HEAD 状态下 commit,这个提交不属于任何分支,回主仓库切换后极易变成悬空对象丢失。改子模块前务必先 checkout 到一个真实分支。

五、对比 subtree

submodulesubtree
子代码存哪只存指针,内容在子仓库子代码直接合并进主仓库历史
克隆需 --recursive,否则空普通 clone 即可,协作者无感
学习成本高(游离 HEAD、初始化、指针提交)低(用起来像普通目录)
版本追踪精确锁定子库某提交,清晰合入后混在主历史里,回溯较模糊
怎么选:需要严格独立维护、精确锁版本、且团队愿意接受它的复杂度时选 submodule;想让协作者零感知、克隆即用、不想教大家 --recursive 时选 subtree。近年很多团队干脆改用包管理器(Maven/npm 等)来管依赖,从根本上绕开这两者。

小结

  • submodule 让主仓库以"指向子仓库某个 commit 的指针(gitlink)"方式嵌入子仓库,配 .gitmodules 记录 URL/路径。
  • 主仓库只存指针不存内容,所以 clone 要 --recursive,或事后 submodule update --init --recursive。
  • 子模块默认处于游离 HEAD;改动前先 checkout 分支;升级子模块后要在主仓库提交指针变更。
  • 对比 subtree:submodule 独立/锁版本但复杂,subtree 简单无感但历史混合;或直接用包管理器替代。

与其他知识点的联系

  • 关联 Git 对象模型(下一篇):子模块在主仓库 tree 里对应的 gitlink 条目,本质是对象模型里除 blob/tree/commit 之外的第四种特殊引用——它不像 tree 指向内容,而是直接指向另一个仓库里的一个 commit 哈希,理解 blob/tree/commit 的组织方式后再看 gitlink 会豁然开朗。
  • 关联「跟踪分支」:子模块目录一旦 submodule update --init,内部就变成一个独立的 Git 仓库,同样可以为它的分支设置跟踪关系(如跟踪子仓库的 origin/main);"先 checkout 到真实分支再改动"这条建议本质就是给子模块建立正常的跟踪分支工作流,避免游离 HEAD。
  • 关联「Fork 派生仓库」:子模块和 fork 都是"引用外部仓库又保持独立"的手段,但粒度不同——fork 是对整仓库的派生与后续同步,submodule 是把一个仓库当依赖精确锁定在某个 commit 上嵌入另一个项目;两者都需要"手动同步上游"这一相似心智模型。
  • 关联「reset 三种模式」:对主仓库执行 reset(尤其 --hard)到较早的提交时,子模块的 gitlink 指针也会随之回退到当时记录的那个 commit,如果子模块目录本身还有未提交的改动,就会与「游离 HEAD 丢改动」的陷阱叠加,风险更高。
  • 关联「打包文件 packfile」:主仓库的对象库和子模块的对象库是完全独立的两套 .git/objects,各自独立打包、独立 gc;clone 主仓库时若不加 --recursive,子模块的 packfile 根本不会被传输,这也是"子模块目录为空"这一现象在存储层面的体现。
