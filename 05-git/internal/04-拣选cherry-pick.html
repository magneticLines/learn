拣选 cherry-pick:只摘那一颗提交

不想合并整条分支,只想把其中某一个 bugfix 提交单独搬过来?cherry-pick 就是这把"精准镊子"。

📖 来源:《Pro Git》 第 5.3 章「维护项目」

一、它是什么:复制一个提交的改动到当前分支

git cherry-pick <commit> 把指定提交所引入的改动,像打补丁一样应用到你当前所在的分支,并生成一个全新的提交。注意:它复制的是"这个提交做了什么改动(diff)",而不是把原提交对象搬过来——所以新提交有不同的 SHA-1,只是内容改动相同。

bash · 基本用法
git checkout release       # 先切到要接收补丁的分支
git cherry-pick a1b2c3d     # 把 a1b2c3d 的改动摘到 release,生成新提交

git cherry-pick a1b2c3d f4e5d6a   # 一次摘多个
git cherry-pick a1b2c3d..f4e5d6a  # 摘一个区间(不含左端点)
git cherry-pick -x a1b2c3d  # -x:在提交信息里追加"cherry picked from ..."便于追溯
git cherry-pick -n a1b2c3d  # -n:只应用改动不自动提交,方便先改再交

二、典型场景:把 bugfix 单独挑到 release 分支

最经典的用途:main 分支上修了一个 bug(提交 X),这个修复也需要进已经拉出去的 release/1.0 分支,但你绝不想把 main 上其他还没准备好的新功能一起带过去。这时不能 merge 整条 main,只能精准地把提交 X 摘到 release。

为什么不 merge:merge 会把两条分支的全部差异合到一起。你只想要一个 bugfix,merge 却会把 main 上尚未测试的十几个新功能提交一并带入 release——这正是 cherry-pick 存在的意义:粒度精确到单个提交。

三、与 merge / rebase 对比

mergerebasecherry-pick
搬运粒度整条分支的全部差异一段连续提交(换基)任意挑选的单个/几个提交
是否新建提交生成合并提交为每个提交生成新提交为每个被摘提交生成新提交
原提交 SHA保留改变改变(内容同、哈希不同)
适用把整个功能分支并入让分支基于最新主线、线性化历史只要某几个特定提交
陷阱——重复提交:如果一个提交被 cherry-pick 到 release,之后 release 又整体 merge 回 main,由于 cherry-pick 产生的新提交哈希不同,Git 可能认不出这是同一改动,导致同一处修改出现两个提交,甚至引发合并冲突。所以 cherry-pick 适合"单向补丁下发",要谨慎用在会双向合并的分支之间。

四、冲突处理

被摘的改动应用到当前分支时,如果上下文对不上就会冲突,流程和 merge 冲突一样:

bash · cherry-pick 冲突三部曲
git cherry-pick a1b2c3d
# CONFLICT!编辑冲突文件,解决冲突标记 <<<<<<< ======= >>>>>>>
git add <解决后的文件>
git cherry-pick --continue   # 解决完继续,完成这次拣选

git cherry-pick --abort      # 或者放弃,回到 cherry-pick 之前的状态
git cherry-pick --skip       # 跳过当前这个(多提交批量时用)

小结

  • cherry-pick 把指定提交的改动复制到当前分支,生成哈希不同、内容相同的新提交。
  • 核心场景:把某个 bugfix 单独挑到 release 分支,而不带入其他未就绪的提交。
  • 对比:merge 搬整条分支差异、rebase 换基一段提交、cherry-pick 精确到单个提交。
  • 陷阱:哈希改变可能导致后续双向合并时出现重复提交;冲突处理用 --continue / --abort / --skip。

与其他知识点的联系

  • 关联 Git 对象模型:cherry-pick 生成"内容相同、哈希不同"的新提交,根源在于 commit 对象的哈希由其 tree、parent、作者时间等元数据共同决定——parent 变了(挂到当前分支而非原分支),哈希必然不同,这正是对象模型"内容寻址、环环相扣"的直接体现。
  • 关联「reset 三种模式」:reset --soft 常用于把多个提交压缩成一个干净的提交,cherry-pick 则是把单个提交的改动搬到另一分支——两者都不依赖完整的 merge/rebase 流程,而是对历史做外科手术式的精细操作,只是作用对象和粒度不同。
  • 关联「子模块 submodule」:子模块升级后,主仓库需要单独提交"指针变更"这一个 commit;如果这个升级提交需要单独同步到另一个分支(而不带上其他改动),同样可以用 cherry-pick 摘取,而不是 merge 整条分支。
  • 关联「跟踪分支」与协作场景:cherry-pick 常见于"给 release 分支打补丁"这类场景,而 release 分支往往也需要 push 到远程并被其他人跟踪——用 cherry-pick 摘取的新提交推送后,其他人 pull 到的是这个新哈希的提交,而非原始提交,这也是"重复提交"陷阱容易在协作中被放大的原因。
  • 关联「打包文件 packfile」:cherry-pick 产生的新提交与原提交内容几乎相同,这类高度相似的对象正是 delta 压缩最擅长处理的情况——即使历史里同时存在"原提交"和"摘取后的提交",packfile 也能把它们压缩得很小,不会显著增大仓库体积。
