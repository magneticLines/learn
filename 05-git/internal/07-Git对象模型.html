Git 对象模型:blob / tree / commit 三块地基

这是理解 Git 的地基。搞懂三种对象和内容寻址,前面所有命令的行为——分支、reset、merge、stash——都会从"死记硬背"变成"一眼看穿"。

📖 来源:《Pro Git》 第 10 章「Git 内部原理」(10.2 Git 对象)

一、Git 本质是一个"内容寻址文件系统"

抛开分支、提交这些高层概念,Git 最核心的其实是一个简单的键值数据库:你给它一段内容,它算出一个SHA-1 哈希(40 位十六进制)作为 key,把内容存进 .git/objects;之后凭这个哈希就能取回内容。这叫内容寻址(content-addressable)——地址由内容本身决定,而不是由文件名或位置决定。

bash · 用底层命令直接存/取内容
# hash-object:把内容存进对象库,返回它的 SHA-1(-w 表示真的写入)
echo "hello git" | git hash-object -w --stdin
# → 8d0e41234f24b6da002d962a26c2495ea16a425f

# cat-file:凭哈希取回内容 / 查看类型
git cat-file -p 8d0e41234f   # -p:按类型美化打印内容 → hello git
git cat-file -t 8d0e41234f   # -t:看类型 → blob
为什么用 SHA-1 而不是自增 ID:内容寻址带来两个巨大好处——① 天然去重:两个完全相同的文件内容算出同一个哈希,只存一份;② 天然防篡改:哈希由内容决定,任何一个字节被改,哈希就变,整条引用链随之失效。这正是 Git 数据完整性的根基。

二、三种核心对象

Git 用这套键值库存储三种(严格说四种,含 tag)对象类型,层层组合成完整的项目快照:

对象存什么类比
blob纯文件内容(不含文件名、不含权限)文件的"肉"
tree一个目录:列出该目录下每个条目的名字 + 权限 + 指向的 blob/tree 哈希目录(文件夹)
commit指向一个顶层 tree + 父提交 + 作者/提交者 + 提交信息一次快照 + 元数据

blob:只有内容,没有身份

关键认知:blob 里不存文件名。它只存内容本身。所以两个内容相同、名字不同的文件,共用同一个 blob。"这个 blob 叫什么名字"这个信息,是记在上一层的 tree 里的。

tree:把名字和内容缝合起来

bash · 查看一个 tree 对象
git cat-file -p HEAD^{tree}
# 输出形如(每行 = 权限 类型 哈希 名字):
# 100644 blob a1b2c3...   README.md    ← 指向一个 blob,起名 README.md
# 100644 blob d4e5f6...   pom.xml
# 040000 tree 778899...   src          ← 指向另一个 tree(子目录),形成递归

tree 通过引用子 tree 实现目录嵌套——整个项目的目录结构就是一棵 tree 组成的树。tree 记录名字与权限,blob 记录内容,两者分工,共同表达"某个目录此刻长什么样"。

commit:给快照盖上时间戳和签名

bash · 查看一个 commit 对象
git cat-file -p HEAD
# tree   9fceb0...              ← 指向本次快照的顶层目录 tree
# parent 056bd2b...            ← 指向上一个 commit(首个提交无此行)
# author  shea <...> 1719...   ← 作者 + 时间戳
# committer shea <...> 1719... ← 提交者
#
# 添加文档                       ← 提交信息

三、它们如何组成一次快照

把三者串起来:一次 git commit 会——① 把每个变化的文件存成 blob;② 为每层目录生成 tree,tree 里记录文件名指向 blob、子目录指向子 tree;③ 生成一个 commit,指向顶层 tree、指向父提交。于是"提交"= 一个指向完整目录树的指针。

对象引用关系(自上而下)
commit ──parent──> 上一个 commit
   │
   └─tree─> 顶层 tree ──> blob(README.md 内容)
                 ├──> blob(pom.xml 内容)
                 └──> tree(src/) ──> blob(App.java 内容)
为什么说 Git 存的是"快照"而非"差异":每次提交,Git 记录的是那一刻整个项目的完整树(一组 tree + blob),而不是"相对上次改了哪几行"。这与 SVN 等"存差异"的系统本质不同——正因如此,checkout 任意提交都是 O(1) 直接取树,不必层层叠加差异。(存储层面的 delta 压缩是另一回事,见「打包文件」篇,它不改变这个快照语义。)
去重的威力:连续两次提交,只改了一个文件——那么没变的文件仍复用原来的 blob,没变的子目录仍复用原来的 tree(哈希没变)。Git 只需为变化的文件/目录创建新对象。所以"每次存完整快照"并不会浪费海量空间。

四、这套模型如何解释高层命令

  • 分支只是一个指针:一个分支就是一个 41 字节的文件,里面是它最新 commit 的哈希。创建分支 = 写个文件,所以 Git 建分支才那么快(下一篇细讲)。
  • reset 就是改指针指向哪个 commit——因为 commit 就是快照入口。
  • SHA-1 让完整性可验证:commit 哈希由其内容(含 tree 哈希、parent 哈希)决定,tree 哈希由其内容(含 blob 哈希)决定……环环相扣。改动历史中任何一个字节,从那里往后所有哈希全变——这就是 Git 无法悄悄篡改历史的原因。
常见误区:初学者以为"提交存的是 diff"。不是。提交存的是完整快照(整棵 tree),diff 是 Git 在你 git diff / git log -p 时临时对比两个快照算出来的,并非存储形式。这个误区一破,merge/rebase/cherry-pick 的很多行为就顺理成章了。

小结

  • Git 本质是内容寻址键值库:内容 → SHA-1 → 存 .git/objects。地址由内容决定,天然去重、天然防篡改。
  • 三种对象:blob(纯内容,无文件名)、tree(目录:名字+权限+指向子对象)、commit(指向顶层 tree + 父提交 + 元数据)。
  • 一次提交 = 一整棵 tree/blob 快照 + 一个 commit 入口;没变的对象跨提交复用,所以快照不浪费空间。
  • Git 存快照不存差异(diff 是临时算的);哈希环环相扣使历史不可悄悄篡改。分支/reset 等高层命令本质都是"操作指向 commit 的指针"。
