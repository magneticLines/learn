Pull Request:协作的"提案—评审—合并"

Pull Request(PR,GitLab 里叫 Merge Request)是现代团队协作的中枢。但初学者常有个误解:PR 不是 Git 的命令,而是 GitHub/GitLab 等平台在 Git 之上加的一层协作功能。

📖 来源:《Pro Git》 第 6 章「GitHub」·「对项目做出贡献」

一、PR 是平台功能,不是 git 命令

你在终端里敲不出 git pull-request(gh CLI 那种是另说)。PR 是 GitHub / GitLab / Gitee 这些托管平台提供的东西:它本质是一个"请求",内容是——"请把我这个分支的改动,合并到你那个分支去",并围绕这个请求提供讨论、代码评审、CI 检查、合并按钮等一整套界面。

建立联系:回忆 01-分布式工作流模型 里的"集成管理者工作流"——贡献者没有权威仓库的写权限,只能"请求"维护者合并。PR 就是把这个"请求—评审—合并"的社交过程,变成了平台上的一个可视化、可追溯的对象。

二、为什么需要 PR 而不是直接 push

直接 push 到主分支走 PR
改动直接进主线,没人把关合并前必须经过评审(review)
无讨论记录评论、逐行 review、修改建议都留痕
CI 挂了才发现合并前自动跑 CI,红了不让合
需要主分支写权限外部贡献者无写权限也能提代码

一句话:PR 把"合并"从一个孤立的动作,变成一个有质量门禁和协作讨论的过程。这是团队保证主线质量的核心手段。

三、完整流程:fork → 分支 → 提交 → 发起 PR → review → 合并

以给一个开源项目贡献代码为例(内部团队若有仓库写权限,可跳过 fork 直接开分支):

贡献者侧:本地命令部分
# 1. 在平台上 fork 目标仓库到自己名下(点按钮,详见 05 篇)
# 2. clone 自己的 fork
git clone git@github.com:you/project.git
cd project

# 3. 开一个主题分支 —— 永远不要直接在 main 上改!便于隔离、评审、返工
git checkout -b fix-login-bug

# 4. 改代码,提交(提交信息写清"为什么改")
git commit -am "fix: 修复空密码可绕过登录校验的问题"

# 5. 推送分支到自己的 fork,并建立上游
git push -u origin fix-login-bug

推送成功后,GitHub 页面上会出现一个 "Compare & pull request" 的提示按钮。点它,填写标题和描述(说清改了什么、为什么、怎么测的),就发起了一个 PR——目标是把你 fork 里的 fix-login-bug 合并到原仓库的 main。

维护者侧:review 与合并

  • Review(评审):维护者逐行看 diff、留评论、提修改建议,或直接 Approve。
  • 迭代:你根据意见改代码,再 git push 到同一个分支——PR 会自动更新,无需重新发起。这是 PR 一个很妙的点:分支就是 PR 的载体。
  • CI:每次 push 自动触发测试/构建,全绿才允许合并。
  • 合并:评审通过后,维护者点 "Merge" 按钮完成合并。

四、PR 合并的三种方式(与 merge 的关系)

点下 "Merge" 时,平台通常给三个选项,它们底层对应不同的 git 操作:

方式底层行为历史效果
Merge commit普通 git merge --no-ff保留分支全部提交 + 一个合并提交,历史真实但较杂
Squash and merge把分支所有提交压成一个主线只留一个干净提交,最常用
Rebase and merge把提交逐个 rebase 到 main 上线性历史,无合并提交
取舍:团队若追求主线整洁,多用 Squash——一个 PR 对应一个提交,回滚、二分查错(bisect)都清爽。若要保留每个提交的细节则用 Merge commit。所以 "PR 合并"归根结底还是 git merge / rebase,只是平台替你按了那颗按钮。

五、常见陷阱与好习惯

陷阱一:在 main 上直接开发再发 PR。一旦要同时做第二件事就乱套了,而且 fork 的 main 应保持和上游同步。永远为每个 PR 开独立主题分支。
陷阱二:一个 PR 塞进十个不相关的改动。评审者看不动,也没法只回滚其中一部分。一个 PR 只做一件事(single responsibility)。
陷阱三:PR 期间上游 main 前进了,你的分支落后甚至冲突。定期把上游改动同步进你的分支(git pull --rebase 上游 main),保持可合并状态。

小结

  • PR 是 GitHub/GitLab 等平台的功能,不是 git 命令;本质是"请把我的分支合进你的分支"的请求。
  • PR 在合并前加了评审、讨论、CI 门禁,让外部贡献者无写权限也能提代码——是团队保质量的核心。
  • 典型流程:fork → clone → 开主题分支 → 提交 → push → 平台发起 PR → review 迭代 → 合并。
  • 向同一分支 push 会自动更新 PR,分支就是 PR 的载体。
  • 合并方式 Merge / Squash / Rebase 分别对应不同 git 操作;Squash 最能保持主线整洁。
  • 好习惯:一分支一 PR、一 PR 一件事、定期与上游同步避免冲突。
