获取与推送:fetch 与 pull 到底差在哪

fetch 和 pull 是面试高频、也是新手最容易踩坑的地方。一句话:pull = fetch + merge。理解这个等式,以及它"隐藏的自动合并"陷阱,你就掌握了远程同步的一大半。

📖 来源:《Pro Git》 第 2 章「Git 基础」、第 5 章「分布式 Git」

一、先分清三种"分支":本地 / 远程 / 远程跟踪

要讲清 fetch/pull,必须先有三个概念在脑子里:

名称例子含义
本地分支main你自己在本地开发、能提交的分支
远程跟踪分支origin/main本地缓存的"远端 main 上次看到的样子",只读,由 fetch 更新
远程分支服务器上的 main真正在远端服务器上的分支
关键直觉:origin/main 不是"远端的分支",而是你本地对远端的一份快照缓存。你昨天 fetch 过,今天 origin/main 显示的还是昨天的状态——它不会自动更新,只有你再次 fetch 才会刷新。很多人把 origin/main 当成实时的远端状态,这是误区之源。

二、git fetch:只下载,不动你的工作

git fetch 做且仅做一件事:把远端的新提交下载下来,更新远程跟踪分支(如 origin/main),但绝不碰你的本地分支和工作区。

fetch:安全地看看远端有什么新东西
git fetch origin           # 下载 origin 的所有新提交,更新 origin/* 跟踪分支
# 此刻你的本地 main 一点没变,工作区也没变 —— 完全安全

git log main..origin/main   # 看看远端比我多了哪些提交(先评估再决定合不合)
git diff main origin/main    # 看看具体差异内容

# 觉得没问题了,再手动整合
git merge origin/main        # 把远程跟踪分支合进当前本地分支

为什么推荐 fetch:它把"下载"和"合并"两步拆开了。你可以先下载、检查差异、评估影响,再决定要不要合、怎么合。这是一种"先看后动"的安全习惯。

三、git pull:fetch + merge 的自动二合一

git pull 本质就是一条组合命令:

pull 的等价展开
git pull origin main
# 完全等价于下面两步:
git fetch origin              # 第一步:下载,更新 origin/main
git merge origin/main         # 第二步:立即自动合并进当前分支

方便是方便,但危险也在这个"自动"上——它把 merge 这一步替你做了主,你还没看差异,合并就已经发生了。

四、pull 的隐藏陷阱:意外的合并提交

设想这个场景:你和同事都基于同一个 main 各自提交了。你想推送,但被拒绝(远端已被同事更新)。这时你顺手 git pull,结果:

  • 因为你的本地和远端分叉了(各有各的新提交),pull 里的 merge 无法快进,于是 Git 自动生成一个合并提交(merge commit)。
  • 历史里凭空多出一堆 Merge branch 'main' of ... 的提交,把线性历史搅成"意大利面"。
  • 更糟的是,如果有冲突,你会在毫无准备时被丢进冲突解决状态。
面试高频 & 实战陷阱:"你和同事同时改了,pull 之后 log 里多了一堆莫名其妙的 merge 提交"——根源就是 pull 默认走 merge,遇到分叉就自动建合并提交。解决办法是用 --rebase。

用 --rebase 保持线性历史

pull --rebase:把自己的提交"接"到远端之后
git pull --rebase origin main
# 等价于:fetch,然后 rebase(而不是 merge)
# 效果:把你本地的提交暂存,先应用远端的新提交,再把你的提交重放到最上面
# 结果是一条干净的直线,没有多余的 merge commit

# 嫌每次都敲 --rebase 麻烦?配成默认行为:
git config --global pull.rebase true
取舍对比:merge 保留"分叉又汇合"的真实历史、绝不改写已有提交,更安全;rebase 得到线性、易读的历史,但会改写提交(生成新的 commit id)。经验法则:本地还没推送出去的提交用 rebase 整理没问题;已经推送、别人可能基于它开发的提交绝不要 rebase。

五、git push:把本地提交上传到远端

push 的常见用法与拒绝
git push origin main          # 把本地 main 推到 origin 的 main

# 若被拒绝(! [rejected] ... non-fast-forward):
# 说明远端有你本地没有的提交 —— 别急着 -f 强推!
git pull --rebase origin main  # 先把远端提交整合进来
git push origin main          # 再推就成功了
切忌盲目 git push -f(强推):它会用你本地的历史覆盖远端,同事在这期间推的提交会被抹掉。若非必要(如整理自己独占的分支),永远不要对共享分支强推;真要强推请用更安全的 --force-with-lease,它在远端被别人动过时会拒绝。

小结

  • origin/main 是远程跟踪分支,是本地对远端的只读快照,只有 fetch 才会刷新。
  • git fetch 只下载、更新跟踪分支,不动本地分支与工作区——先看后动,最安全。
  • git pull = fetch + merge,方便但会替你自动合并。
  • pull 遇到分叉会自动生成 merge commit,搅乱历史;用 pull --rebase 保持线性。
  • rebase 只对本地未推送的提交用;已共享的提交别 rebase、别强推。
  • push 被拒是因为远端更新了,先 pull --rebase 再推,不要盲目 -f。
