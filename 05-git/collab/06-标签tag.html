标签 tag:给某个提交贴上"版本"标记

分支会移动,标签不会。标签是钉在某个提交上的不动的名字,专门用来标记发布版本(如 v1.0.0)。理解轻量标签与附注标签的区别,是做规范发布的基础。

📖 来源:《Pro Git》 第 2 章「Git 基础」·「打标签」

一、标签 vs 分支:一个固定,一个移动

分支(如 main)是一个会随新提交移动的指针——你每提交一次,main 就往前挪一格。而标签是一个钉死在特定提交上的指针,永远不动。

为什么需要标签:当你发布 v1.0.0 时,你想要一个永久、稳定的名字指向"发布 1.0.0 时的那个确切代码状态"。用分支不行——分支会继续前进。标签正好:打上 v1.0.0 后,无论 main 怎么往前走,v1.0.0 永远指向那一刻的提交。这就是"版本快照"。

二、两种标签:轻量标签 vs 附注标签

轻量标签(lightweight)附注标签(annotated)
本质就是一个指向提交的名字,别无他物Git 里一个完整对象
存储内容只有提交指针打标签者、日期、说明、可 GPG 签名
命令git tag v1.0git tag -a v1.0 -m "..."
适用临时、私人书签正式发布(推荐)
打标签的两种方式
# 轻量标签:像给提交起个别名,不带任何元信息
git tag v1.0-beta

# 附注标签:-a 表示 annotated,-m 写标签说明 —— 正式发布用这个
git tag -a v1.0.0 -m "正式发布 1.0.0:完成核心功能"

git show v1.0.0        # 附注标签能看到打标签者、日期、说明,轻量标签则只显示提交本身
选择建议:正式发布一律用附注标签(-a)——它记录了"谁、什么时候、为什么"打的标签,还能签名验证真伪。轻量标签仅适合自己临时标记。面试若问区别,答"附注标签是独立对象、含元信息、可签名"即可。

三、给历史提交补打标签

标签不一定只能打在当前 HEAD 上——你可以给过去任意一个提交补打标签,只要在命令末尾给出它的 commit id:

给某个历史提交补标签
git log --oneline
# a1b2c3d 发布准备
# 9f8e7d6 修复导出 bug   ← 想给这个提交补个 v0.9

git tag -a v0.9 9f8e7d6 -m "补标记 0.9 版本"   # 末尾指定 commit 即可

这在"忘了在发布时打标签、事后补救"的场景很常见。

四、推送标签:push 默认不带标签

一个高频踩坑:git push 默认不会推送标签。你在本地打的标签,同事和远端根本看不到,必须显式推:

推送标签到远端
git push origin v1.0.0     # 推送单个标签
git push origin --tags     # 一次性推送本地所有还没推的标签

# 删除标签:本地和远端要分别删
git tag -d v1.0.0                    # 删本地标签
git push origin --delete v1.0.0     # 删远端标签
常见误区:"我明明打了 tag,GitHub 上却没有 Release / 也看不到 tag"——十有八九是忘了 git push --tags。标签是需要单独推送的,别以为它会跟着 commit 一起走。

五、标签与发布版本(Release)的关系

标签本身只是"钉在提交上的名字",但它是各种发布机制的锚点:

  • GitHub Release:你推一个 tag(如 v1.0.0),GitHub 就能基于它创建一个 Release——附上更新说明、编译好的二进制文件、自动打的源码压缩包。Release 是围绕 tag 展开的平台功能。
  • 语义化版本(SemVer):标签名常遵循 vMAJOR.MINOR.PATCH(如 v2.3.1)——主版本破坏兼容、次版本加功能、补丁修 bug。这让使用者一眼知道升级风险。
  • CI/CD 触发:很多流水线配置成"push 一个 v* 标签就自动构建并发布",标签成了发布动作的触发器。
  • 依赖管理:Maven/npm 等包管理器发布的版本号,通常就对应仓库里的一个 tag,方便追溯某个版本对应哪份源码。
建立联系:你用 Maven 引 spring-core:6.1.0,这个 6.1.0 在 Spring 的 Git 仓库里就对应一个 tag。标签把"发布出去的版本号"和"仓库里的确切源码状态"牢牢绑定,是可追溯性的基石。

小结

  • 分支会随提交移动,标签固定钉在某个提交上,专门用来标记版本快照。
  • 轻量标签只是个名字;附注标签(-a)是独立对象,含打标签者/日期/说明、可签名,正式发布用它。
  • 可用 git tag -a 名字 commitId 给历史提交补打标签。
  • git push 默认不推标签,要 git push origin 标签 或 --tags;删标签本地远端分开删。
  • 标签是 Release、SemVer、CI/CD 触发、依赖版本追溯的共同锚点。
