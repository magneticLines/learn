Fork:在没有写权限时也能参与

Fork 是开源协作的入场券。它和 clone 常被搞混,但两者层级完全不同。理解 fork,以及随之而来的 origin / upstream 双远程配置,你才能顺畅地为任何开源项目贡献代码。

📖 来源:《Pro Git》 第 6 章「GitHub」·「对项目做出贡献」

一、fork 与 clone 的本质区别

forkclone
发生在哪平台服务器上(GitHub 等)本地机器上
是谁的功能平台功能(点按钮)git 命令
结果在你账号下多一个服务端副本,你对它有写权限把某个仓库下载到本地
写权限fork 出来的是你的,随便推能不能推回原仓库取决于你有没有权限

关键区别:fork 是"服务器到服务器"的复制(在 GitHub 内部把别人的仓库复制一份到你名下);clone 是"服务器到本地"的下载。

为什么开源要先 fork:你对陌生开源项目没有写权限,不能直接 push。fork 出一份属于你的服务端副本后,你就对自己的副本有了完全的写权限,可以随意推送分支;再通过 PR 请求原作者合并。fork 解决的正是"无写权限如何贡献"这个问题。

二、典型工作流:fork → clone → 双远程

fork 之后,你本地 clone 的是自己的 fork,于是需要配置两个远程仓库:

远程名指向用途
origin你自己的 fork你 push 分支的地方(你有写权限)
upstream原始项目仓库你 fetch 上游更新的地方(通常只读)
配置 origin 与 upstream 双远程
# 1. clone 自己的 fork —— 这一步 origin 自动指向你的 fork
git clone git@github.com:you/project.git
cd project

# 2. 手动加一个 upstream 指向原始仓库(注意是原作者的地址)
git remote add upstream git@github.com:original/project.git

# 3. 确认两个远程都在
git remote -v
# origin    git@github.com:you/project.git       (fetch/push)  ← 你的 fork
# upstream  git@github.com:original/project.git  (fetch/push)  ← 原始项目
命名约定:origin = 我的、upstream = 上游源头,是社区通用惯例。看到别人这么写,一眼就知道谁是谁。惯例本身没有强制力,但遵守它能让协作沟通零成本。

三、同步上游更新:fork 不会自动跟随原仓库

一个大坑:你 fork 出去之后,原仓库继续更新,但你的 fork 不会自动跟着变。fork 只是某一时刻的快照。时间一长,你的 fork 就落后了,基于旧代码开发容易冲突。所以要定期把上游的更新同步进来:

同步上游最新代码到本地并推回 fork
# 1. 拉取上游的最新提交(只更新 upstream/* 跟踪分支,不动本地)
git fetch upstream

# 2. 切到本地 main,把上游 main 合并进来
git checkout main
git merge upstream/main       # 你的 main 通常没有独立提交,这里多为快进合并

# 3.(可选)把同步后的 main 推回自己的 fork,让 fork 也保持最新
git push origin main
好习惯:保持你 fork 的 main 永远和 upstream/main 一致,不要往自己 fork 的 main 上直接提交。所有开发都在从最新 main 切出的主题分支上进行。这样每次开新分支都基于最新代码,冲突最少,PR 也最干净。

用 rebase 让分支基于最新上游

把开发中的分支重新基于最新 upstream/main
git fetch upstream
git checkout my-feature
git rebase upstream/main       # 把我的提交重放到上游最新代码之上,历史线性、易 review
# rebase 改写了提交,推自己的 fork 分支时可能要 --force-with-lease
git push --force-with-lease origin my-feature

四、fork 与镜像 / 直接协作的对比

  • fork 模式:无写权限的开源贡献场景。人人可参与,维护者当守门人。
  • 共享仓库直接开分支:公司内部、你有仓库写权限时,不必 fork,直接在原仓库开分支发 PR 即可,少一层 fork 更省事。
  • 判断依据:你对目标仓库有没有写权限——有就直接开分支,没有才 fork。

小结

  • fork 是平台的"服务器到服务器"复制,给你一份可写的服务端副本;clone 是 git 的"服务器到本地"下载。
  • 无写权限的开源贡献必须先 fork,再通过 PR 请求合并。
  • 双远程配置:origin 指自己的 fork(可推),upstream 指原始仓库(拉更新)。
  • fork 不会自动跟随原仓库,要定期 git fetch upstream + merge upstream/main 同步。
  • 保持 fork 的 main 与上游一致,开发都在主题分支上做,冲突最少。
  • 有写权限时直接在原仓库开分支即可,无需 fork。
