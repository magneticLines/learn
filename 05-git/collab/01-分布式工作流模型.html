分布式工作流模型(集中式 / 集成管理者 / 主管副主管)

Git 是分布式版本控制系统——它不规定"团队该怎么协作",而是给了你搭建各种协作模式的原材料。理解这一点,才能理解为什么同样是 Git,有的团队直接推同一个仓库,有的却要 fork。

📖 来源:《Pro Git》 第 5 章「分布式 Git」

一、分布式的核心:每个 clone 都是完整仓库

先建立一个最关键的直觉:git clone 下来的不是"工作副本",而是一份完整的仓库——包含全部历史、全部分支、全部对象。这和 SVN/CVS 这类集中式(CVCS)系统有本质区别:在 SVN 里,只有中央服务器有完整历史,你本地只是某个版本的快照;服务器一挂,历史就危险了。

在 Git 里,没有"中央仓库"这个技术概念——所谓的"中央仓库"(如 GitHub 上那个)只是团队约定拿来当协作枢纽的一个普通仓库而已。任意一个开发者的本地 clone 都能反过来重建它。这就是"分布式"的字面含义:仓库是可以互相拷贝、对等的。

为什么这么设计:因为完整历史都在本地,所以 git log、git diff、git commit、切分支等绝大多数操作都不需要联网,速度极快。集中式系统里这些操作往往要走网络请求服务器。这也是 Git 比 SVN 体验好的根本原因之一。

正因为仓库对等,Git 本身对"谁能改、改了怎么合"不做规定。于是就演化出下面几种典型的协作工作流——它们不是 Git 的功能,而是团队用 Git 搭出来的约定。

二、集中式工作流(Centralized Workflow)

最简单、也最接近 SVN 习惯的模式:团队约定一个共享仓库当"中央",所有人都对它有推送权限,大家都往同一个仓库推。

集中式:所有人推同一个 origin
# 开发者 A、B、C 都克隆同一个仓库,都有写权限
git clone git@server:project.git   # 每个人拿到完整仓库
# A 改完直接推
git push origin main
# B 此时也想推,但 A 已经推过了,B 会被拒绝(non-fast-forward)
# B 必须先把 A 的改动拉下来整合,再推 —— 这是集中式最常见的"卡点"
git pull                        # 先整合远端最新的提交
git push origin main            # 整合后才能推成功

适用场景:小团队(2~5 人)、成员互相信任、大家能直接约谈。缺点:随人数增多,"推之前必须先 pull 整合"的冲突会越来越频繁,而且所有人都有主分支写权限,风险较高。

三、集成管理者工作流(Integration-Manager Workflow)

这是 GitHub / GitLab 上开源项目最主流的模式,核心角色是一个集成管理者(维护者)。每个贡献者都有两个仓库:

仓库角色
公开的"权威"仓库只有集成管理者能推,是项目的正式版本
贡献者自己的公开副本(fork)贡献者有写权限,先推到这里

流程:贡献者 fork 权威仓库 → 在自己 fork 上开发并推送 → 通知管理者"请把我的改动合进去"(在 GitHub 上就是发起 Pull Request)→ 管理者拉取、审查、合并到权威仓库。

建立联系:你在 GitHub 上给任何一个不属于自己的项目提代码,走的就是这套流程。"fork + Pull Request"正是集成管理者工作流的具体落地——后面的 04-PullRequest流程 和 05-Fork派生仓库 会展开讲。

优点:贡献者不需要权威仓库的写权限,任何人都能参与;管理者是唯一的"守门人",保证主线质量。这就是开源协作能规模化的关键。

四、主管与副主管工作流(Dictator and Lieutenants)

这是集成管理者模式的层级化扩展,用于超大规模项目——最经典的例子就是 Linux 内核。

  • 普通开发者:在各自的主题分支上开发,针对某个子系统。
  • 副主管(Lieutenants):每人负责一个子系统(如网络、文件系统),负责合并本领域开发者的改动。
  • 仁慈的独裁者(Dictator,如 Linus):只从副主管们那里拉取,合并成最终的权威版本。

普通开发者的改动 → 汇聚到对应副主管 → 再汇聚到主管。这本质是一棵合并的树,把"审查合并"这个工作分层摊派下去,让一个人不至于成为几千个贡献者的唯一瓶颈。

五、三种工作流对比与选择

工作流写权限规模典型场景
集中式人人可推主仓库小团队公司内部小项目、私有仓库
集成管理者只有维护者推权威仓库中到大GitHub 开源项目(fork + PR)
主管副主管分层合并超大规模Linux 内核等
常见误区:不要以为"分布式"就意味着"没有中央仓库、大家乱推"。恰恰相反,几乎所有团队都会约定一个仓库当协作枢纽。"分布式"指的是技术上每个仓库对等、可离线操作,而不是流程上没有中心。分清"技术能力"和"团队约定"是理解 Git 协作的关键。

小结

  • Git 是分布式的:每个 clone 都是完整仓库,含全部历史,绝大多数操作可离线,速度快。
  • "中央仓库"是团队约定,不是 Git 的技术概念;仓库之间技术上是对等的。
  • 集中式:人人推同一仓库,适合小团队,推前常需先 pull 整合。
  • 集成管理者:fork + Pull Request,维护者当守门人,是 GitHub 开源主流模式。
  • 主管副主管:把合并审查分层摊派,用于 Linux 内核这类超大项目。
  • 选工作流看团队规模和信任度,别把"分布式技术能力"误当成"流程无中心"。
