# Java 集合框架 (Collections Framework)

> **一句话核心概念总结:** 集合框架是 JDK 提供的一套「把你在数据结构篇手写过的东西」的生产级实现 —— 链表、哈希表、红黑树都在里面，只不过 Java 帮你处理好了泛型、扩容、线程不安全警告。

---

## 1. 整体结构

集合框架的核心接口只有两棵树：

```
Iterable
  └── Collection
        ├── List        有序、可重复
        │     ├── ArrayList
        │     └── LinkedList
        ├── Set         无序（TreeSet 有序）、不可重复
        │     ├── HashSet
        │     ├── LinkedHashSet
        │     └── TreeSet
        └── Queue       队列
              ├── LinkedList
              └── PriorityQueue

Map（独立体系，不继承 Collection）
  ├── HashMap
  ├── LinkedHashMap
  └── TreeMap
```

**记住这个分类**：
- 需要「通过 key 找 value」→ 用 `Map`
- 需要「存一组元素、不重复」→ 用 `Set`
- 需要「存一组元素、按顺序」→ 用 `List`

## 2. List

### 2.1 ArrayList

**底层**：Object[] 数组。

**你在数据结构篇已经知道数组的特性**：随机访问 O(1)，中间插入/删除 O(n)（要搬移元素）。ArrayList 就是把数组包了一层，加上自动扩容。

**扩容机制**（这是 ArrayList 最重要的内部细节）：

```java
// 初始容量是 10
new ArrayList<>();       // 实际上是一个空数组，第一次 add 时才分配 10 个槽
new ArrayList<>(100);    // 预设 100，避免频繁扩容

// 扩容时机：add 发现容量不够
// 扩容倍数：newCapacity = oldCapacity + (oldCapacity >> 1) = 1.5 倍
// 底层调用：Arrays.copyOf() —— 申请新数组 + 拷贝旧数据
```

**为什么是 1.5 倍而不是 2 倍？**
权衡空间浪费和扩容频率。Python list 用 1.125 倍增长，Java 用 1.5 倍，Go 的 slice 用 2 倍（元素少时）——不同语言选择了不同的权衡点，没有绝对正确。

**常见陷阱**：

```java
// 陷阱1：for 循环删除元素 —— 会漏删
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "b", "c"));
for (int i = 0; i < list.size(); i++) {
    if ("b".equals(list.get(i))) {
        list.remove(i);   // 删除后后面的元素左移，下标 i 对应的元素变了！
    }
}
// 结果：只删了第一个 "b"，第二个 "b" 被跳过

// 正确做法1：倒序遍历
for (int i = list.size() - 1; i >= 0; i--) {
    if ("b".equals(list.get(i))) list.remove(i);
}

// 正确做法2：使用迭代器
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if ("b".equals(it.next())) it.remove();  // 迭代器的 remove，安全
}

// 正确做法3（Java 8+，最简洁）
list.removeIf("b"::equals);
```

```java
// 陷阱2：Arrays.asList() 返回的是固定长度的 List，不能 add/remove
List<String> fixed = Arrays.asList("a", "b", "c");
fixed.add("d");   // 抛 UnsupportedOperationException！

// 如果需要可变 List：
List<String> mutable = new ArrayList<>(Arrays.asList("a", "b", "c"));
```

### 2.2 LinkedList

**底层**：双向链表（每个节点有 prev 和 next 两个指针），你在数据结构篇已经实现过。

```java
// LinkedList 同时实现了 List 和 Deque（双端队列）接口
// 可以当栈用：
LinkedList<Integer> stack = new LinkedList<>();
stack.push(1);   // 等价于 addFirst
stack.pop();     // 等价于 removeFirst

// 可以当队列用：
LinkedList<Integer> queue = new LinkedList<>();
queue.offer(1);  // 等价于 addLast
queue.poll();    // 等价于 removeFirst
```

### 2.3 ArrayList vs LinkedList —— 怎么选？

| 场景 | 选谁 | 原因 |
|---|---|---|
| 随机访问（`get(i)`）| ArrayList | O(1) vs LinkedList O(n) |
| 尾部添加 | ArrayList | O(1) 均摊；LinkedList 也是 O(1) 但对象开销更大 |
| 中间插入/删除（已知节点）| LinkedList | O(1) 改指针；ArrayList O(n) 搬元素 |
| 中间插入/删除（按索引）| 都很慢 | LinkedList 找到位置也要 O(n) 遍历 |
| 内存敏感 | ArrayList | LinkedList 每个节点多两个指针引用开销 |

**结论**：**90% 的情况用 ArrayList**。LinkedList 只在「频繁从头部插入删除」且不需要随机访问时才有优势，比如实现 LRU 缓存的底层链表部分。

## 3. HashMap（重点）

你在数据结构篇已经学过哈希表原理（哈希函数、链地址法、负载因子、扩容），HashMap 就是 JDK 的实现，但有几个关键细节值得单独讲。

### 3.1 底层结构：数组 + 链表 + 红黑树

```
JDK 1.7：数组 + 链表（链地址法）
JDK 1.8：数组 + 链表 + 红黑树
```

**为什么 1.8 加了红黑树？**
当哈希冲突严重时，同一个桶里的链表会很长，查找退化到 O(n)。你在数据结构篇学过红黑树的查找是 O(log n)，所以 1.8 做了优化：**链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转成红黑树**。

```
为什么是 8？
根据泊松分布，理想负载因子 0.75 下，一个桶里出现 8 个冲突的概率是 0.00000006（约千万分之六），
属于极端情况。低于 8 时链表的常数更小，不值得引入红黑树的复杂度。
```

### 3.2 关键参数

```java
// 初始容量（必须是 2 的幂次方，为什么？见下文）
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4;  // 16

// 负载因子（存储密度超过这个值就扩容）
static final float DEFAULT_LOAD_FACTOR = 0.75f;

// 扩容时机：size > capacity * loadFactor = 16 * 0.75 = 12 时触发扩容
// 扩容倍数：2 倍（capacity <<= 1）
```

**为什么容量必须是 2 的幂？**

确定桶下标的操作是：`index = hash & (capacity - 1)`

当 capacity 是 2 的幂时，`capacity - 1` 的二进制全是 1（如 16-1=15=01111），
`hash & 01111` 等价于取 hash 的低 4 位，等价于 `hash % 16`，
但位运算比取模快得多，这是一个刻意的性能优化。

### 3.3 hash() 扰动函数

```java
// HashMap 对 key 的 hashCode 做了二次处理：
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

**为什么不直接用 hashCode？**

如果只取低位（`hash & (n-1)`），高位信息就浪费了，高位不同的两个 key 可能落到同一个桶。
把高 16 位与低 16 位做异或（`h ^ (h >>> 16)`），把高位的信息"揉进"低位，让分布更均匀。

### 3.4 扩容 rehash

扩容是 HashMap 性能最敏感的操作：新建一个 2 倍大的数组，把所有 key-value 重新计算下标并搬过去。

**JDK 1.8 的优化**：扩容后 key 的新下标只有两种可能：
- 原下标（低位 bit 是 0）
- 原下标 + 旧容量（低位 bit 是 1）

可以用 `hash & oldCapacity` 直接判断，不用重新计算 hash，扩容效率更高。

### 3.5 常见陷阱

```java
// 陷阱1：用可变对象做 key 然后改它
Map<List<Integer>, String> map = new HashMap<>();
List<Integer> key = new ArrayList<>(Arrays.asList(1, 2, 3));
map.put(key, "value");

key.add(4);  // key 的 hashCode 变了！
map.get(key); // 返回 null —— 找不到了

// 结论：HashMap 的 key 应该是不可变对象（String、Integer 等）

// 陷阱2：HashMap 不是线程安全的
// 多线程并发 put 可能导致死循环（JDK 1.7）或数据丢失（JDK 1.8）
// 线程安全场景用 ConcurrentHashMap，不要用 Collections.synchronizedMap（性能差）

// 陷阱3：遍历时不能修改（除了 Iterator.remove）
Map<String, Integer> m = new HashMap<>();
for (String k : m.keySet()) {
    m.remove(k);  // 抛 ConcurrentModificationException
}
// 正确：
m.entrySet().removeIf(e -> someCondition(e.getKey()));
```

## 4. LinkedHashMap

HashMap 不保证顺序，LinkedHashMap 在其基础上加了一条「双向链表」把所有 entry 串起来，维护**插入顺序**（或访问顺序）。

```java
// 默认：插入顺序
LinkedHashMap<String, Integer> map = new LinkedHashMap<>();
map.put("c", 3); map.put("a", 1); map.put("b", 2);
System.out.println(map.keySet()); // [c, a, b] — 按插入顺序

// accessOrder = true：访问顺序（LRU 缓存的关键）
LinkedHashMap<String, Integer> lru = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Integer> eldest) {
        return size() > 3;  // 超过容量就删最久未访问的
    }
};
lru.put("a", 1); lru.put("b", 2); lru.put("c", 3);
lru.get("a");          // 访问 a，a 移到链表尾部
lru.put("d", 4);       // 触发 removeEldestEntry，删 b（最久未访问）
System.out.println(lru.keySet()); // [c, a, d]
```

## 5. TreeMap

**底层**：红黑树（你在数据结构篇学过）。所有 key 按自然顺序排列，或按传入的 `Comparator` 排列。

```java
// 自然顺序（String 按字典序）
TreeMap<String, Integer> map = new TreeMap<>();
map.put("banana", 2); map.put("apple", 1); map.put("cherry", 3);
System.out.println(map.keySet()); // [apple, banana, cherry]

// TreeMap 独有的范围查询 API（HashMap 没有）
map.firstKey();                    // apple
map.lastKey();                     // cherry
map.headMap("cherry");             // {apple=1, banana=2} —— 小于 cherry 的
map.tailMap("banana");             // {banana=2, cherry=3} —— 大于等于 banana 的
map.subMap("apple", "cherry");     // {apple=1, banana=2}
map.floorKey("blueberry");         // banana —— 小于等于 blueberry 的最大 key
map.ceilingKey("blueberry");       // cherry —— 大于等于 blueberry 的最小 key
```

## 6. Set

Set 的内部实现直接复用了 Map：
- `HashSet` = HashMap，value 统一是一个 `PRESENT` 占位对象
- `LinkedHashSet` = LinkedHashMap
- `TreeSet` = TreeMap

知道这一点就够了，Set 的所有特性和陷阱与对应 Map 完全一致。

```java
// 去重的本质：调用 hashCode() + equals() 判断是否重复
// 陷阱：自定义对象要同时重写这两个方法，否则 Set 去重失效
class Point {
    int x, y;
    // 如果只重写 equals 不重写 hashCode：
    // 两个内容相同的 Point，hashCode 不同 → 落在不同桶 → Set 认为不重复
}
```

## 7. Map 三兄弟对比

| | HashMap | LinkedHashMap | TreeMap |
|---|---|---|---|
| 底层 | 数组+链表+红黑树 | HashMap + 双向链表 | 红黑树 |
| 顺序 | 无序 | 插入顺序 / 访问顺序 | key 排序 |
| null key | 允许（1 个）| 允许 | 不允许 |
| 查找 | O(1) 均摊 | O(1) 均摊 | O(log n) |
| 适用场景 | 通用 | LRU 缓存、需要保留插入顺序 | 范围查询、排行榜 |

## 8. 选哪个？—— 决策树

```
需要 key-value 映射？
  是 → 用 Map
        需要排序 / 范围查询？ → TreeMap
        需要保留插入/访问顺序？ → LinkedHashMap
        其他（绝大多数情况）→ HashMap

只需要存一组不重复的元素？
  是 → 用 Set（对应 Map 的 key-only 版本）

需要有序序列、允许重复？
  是 → 用 List
        频繁随机访问？ → ArrayList
        频繁头部插入删除？ → LinkedList（少见）
```

## 9. 面试高频考点

1. **HashMap 的扩容机制**：容量 2 倍扩容，rehash 时利用高位 bit 判断新位置（不用重新计算 hash）
2. **HashMap 1.7 的死循环**：多线程并发 rehash 时，头插法会形成环形链表，导致 get 死循环；1.8 改为尾插法解决了死循环但仍然线程不安全
3. **为什么负载因子是 0.75**：时间和空间的折中，泊松分布下冲突概率最小
4. **HashMap key 为什么要 `hashCode()` 和 `equals()` 同时重写**：先用 hashCode 定位桶，再用 equals 在桶内比较；只重写其中一个必出 bug
5. **ConcurrentHashMap vs Hashtable**：Hashtable 是方法级 synchronized，全表锁；ConcurrentHashMap 1.8 用 CAS + synchronized 锁单个桶，并发粒度细得多
