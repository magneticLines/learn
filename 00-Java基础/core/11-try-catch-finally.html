核心类库 · 异常处理

try-catch-finally 与 try-with-resources

从资源泄漏到自动关闭——理解 Java 异常处理的两个核心语法，以及 JVM 在背后做了什么。

1. 为什么需要 finally？

Java 的异常机制让我们可以将"正常逻辑"与"错误处理"分开写。但有一类代码天生必须执行——比如关闭文件、释放数据库连接、解锁互斥量——无论 try 块是否抛出异常。finally 就是为了解决这个问题而存在的。

设想没有 finally 的世界：你要在每一个可能的出口（正常返回、每个 catch 分支、嵌套 catch……）都写一遍清理逻辑。这不只是冗余，更是灾难——任何一处遗漏都会导致资源泄漏。finally 把"出口守卫"这一职责统一到了一个地方。

基础结构

Javatry-catch-finally 骨架
try {
    // 可能抛出异常的危险代码
    InputStream in = new FileInputStream("data.txt");
    in.read();
} catch (FileNotFoundException e) {
    // 处理"文件不存在"这一具体情况
    System.err.println("文件未找到: " + e.getMessage());
} catch (IOException e) {
    // 处理其他 IO 错误（父类放后面）
    e.printStackTrace();
} finally {
    // 无论如何都会执行——即使 catch 里又抛了异常
    System.out.println("清理资源");
}
catch 顺序很重要：子类异常必须写在父类前面。如果把 IOException 写在 FileNotFoundException 前，编译器会报"已捕获异常"错误，因为后者是前者的子类，永远到不了第二个 catch。

2. finally 的执行时机与底层机制

finally 几乎在所有情况下都会执行。JVM 实现上，Java 编译器并不是插入一条"跳转到 finally"的指令，而是将 finally 块的字节码复制到每个可能的出口路径之后（称为 inline copying）。这意味着 finally 块实际上被内联了多份。

finally 一定会执行吗？三个例外

  • System.exit() 被调用——JVM 进程直接终止，finally 不会执行
  • JVM 崩溃（如 SIGSEGV、OOM 导致的 kill）
  • 线程被强制 kill（Thread.stop()，已废弃，但理论上存在）

陷阱一：finally 中的 return 会吞掉异常

Java危险：finally 里 return
public static int danger() {
    try {
        throw new RuntimeException("出错了");
    } finally {
        // ⚠️ 这个 return 会"压制"上面的异常，调用方根本看不到异常！
        return 42;
    }
}

// 调用方：
int result = danger();  // 返回 42，异常被静默吞掉！
规则：finally 块中永远不要写 return、throw，也不要修改方法返回值。这是 Java 界公认的坏味道，会导致异常被静默吞掉，极难调试。

陷阱二：finally 中的异常会覆盖原始异常

Javafinally 抛出新异常时原始异常消失
try {
    throw new RuntimeException("原始异常");
} finally {
    // 如果这里也抛出异常，"原始异常"就永远丢失了
    throw new RuntimeException("finally 里的异常");
}
// 调用方只能看到 "finally 里的异常"，原始异常的堆栈已消失

3. 传统 finally 关闭资源的痛点

在 Java 7 之前，正确关闭多个资源需要非常繁琐的嵌套结构。来看一个"教科书级正确但极丑陋"的写法：

Java（Java 6 写法）正确但丑陋的双资源关闭
InputStream in = null;
OutputStream out = null;
try {
    in  = new FileInputStream("src.txt");
    out = new FileOutputStream("dst.txt");
    // ... 复制逻辑 ...
} finally {
    // 必须分别 try，否则 in.close() 抛异常会导致 out 永远不关
    if (in != null) {
        try { in.close(); } catch (IOException ignored) {}
    }
    if (out != null) {
        try { out.close(); } catch (IOException ignored) {}
    }
}
// 两个资源、代码已经膨胀成这样；三个资源呢？

这段代码有两个核心问题：

  1. 冗长易错：每加一个资源都要多写一段 null 检查 + 嵌套 try-catch
  2. 异常信息丢失：如果 close() 抛出了异常，被 ignored 静默掉了，调试时找不到根因

4. try-with-resources（Java 7+）

Java 7 引入 try-with-resources，用语法糖彻底解决资源关闭问题。凡是实现了 AutoCloseable 接口（只有一个方法：void close() throws Exception）的对象，都可以放入 try() 括号中，JVM 保证在 try 块结束后自动调用 close()，顺序与声明顺序相反。

Java 7+try-with-resources 等价改写
// 简洁版：编译器自动插入 close() 调用
try (
    InputStream  in  = new FileInputStream("src.txt");
    OutputStream out = new FileOutputStream("dst.txt")
) {
    // 复制逻辑
    in.transferTo(out);
}
// out.close() 先被调用，然后 in.close()（逆序）
// 无论是否抛出异常，都会自动关闭，无需手写 finally

编译器生成的等价字节码逻辑（伪代码）

伪代码编译器展开后的真实逻辑
Throwable primaryEx = null;
InputStream in = new FileInputStream("src.txt");
try {
    in.read();
} catch (Throwable t) {
    primaryEx = t;
    throw t;
} finally {
    if (in != null) {
        if (primaryEx != null) {
            try {
                in.close();
            } catch (Throwable closeEx) {
                // 关键：close() 抛出的异常被"抑制"，附加到原始异常上
                primaryEx.addSuppressed(closeEx);
            }
        } else {
            in.close();  // 没有原始异常，直接 close
        }
    }
}
Suppressed Exception（抑制异常）：这是 try-with-resources 最精妙的设计。close() 抛出的异常不会覆盖原始异常，而是通过 Throwable.addSuppressed() 附加在原始异常上。调试时用 e.getSuppressed() 或直接看堆栈中的 Suppressed: 行，信息完整保留，一条都不会丢失。

5. 自定义资源类实现 AutoCloseable

任何需要"用完即关"的资源都应实现 AutoCloseable。常见场景：数据库连接、Redis 连接、文件句柄、线程池、网络 Socket。

Java自定义可自动关闭的资源
public class DatabaseConnection implements AutoCloseable {

    private final String url;

    public DatabaseConnection(String url) {
        this.url = url;
        System.out.println("连接已建立: " + url);
    }

    public void query(String sql) {
        System.out.println("执行: " + sql);
    }

    @Override
    public void close() {
        // close() 可以抛 Exception（AutoCloseable），也可以只抛 IOException（Closeable）
        System.out.println("连接已关闭: " + url);
    }
}

// 使用方：
try (DatabaseConnection conn = new DatabaseConnection("jdbc:mysql://localhost/db")) {
    conn.query("SELECT * FROM users");
}
// 输出：连接已建立 → 执行 → 连接已关闭

AutoCloseable vs Closeable

接口 引入版本 close() 签名 适用场景
AutoCloseable Java 7 void close() throws Exception 通用资源（数据库、网络）
Closeable Java 5 void close() throws IOException IO 流（是 AutoCloseable 的子接口）

Closeable 继承了 AutoCloseable，所以所有 IO 流（InputStream、OutputStream、Reader、Writer）天然支持 try-with-resources。

6. Java 9 增强：引用已有变量

Java 9 之前，try() 括号内只能声明新变量。Java 9 开始，已经声明为 effectively final 的变量也可以直接放入括号，不必重新赋值。

Java 9+引用已有的 effectively final 变量
InputStream in  = openInputStream();   // 外部已赋值
OutputStream out = openOutputStream();

// Java 9+：直接引用，不需要在括号内重新赋值
try (in; out) {
    in.transferTo(out);
}

7. multi-catch（Java 7+）

同一个 catch 块处理多种不相关的异常类型，避免重复代码：

Java 7+multi-catch 语法
try {
    // 可能抛出多种异常的操作
    riskyOperation();
} catch (SQLException | IOException e) {
    // e 的编译期类型是两者的公共父类（这里是 Exception）
    // e 在这里是 effectively final，不能重新赋值
    logger.error("操作失败", e);
}
注意：multi-catch 中的异常类型不能有继承关系。如果写 catch (Exception | IOException e)，编译器会报错，因为 IOException 已经被 Exception 覆盖了。

8. 执行顺序全景图

try 块正常执行 ├── 无异常 → [finally] → 方法正常返回 └── 抛出异常 ├── 有匹配的 catch → [catch 块] → [finally] → 继续执行（或重抛） └── 无匹配的 catch → [finally] → 异常继续向上传播 try-with-resources 额外步骤： finally 执行前 → 按声明逆序自动调用 close() close() 异常 → addSuppressed() 到原始异常，不丢失

高频面试题

问题 答案要点
finally 一定执行吗？ 绝大多数情况是。System.exit()、JVM 崩溃、线程强制 kill 除外
finally 中 return，方法返回什么？ finally 的 return 值。原始异常会被静默吞掉（生产禁忌）
try-with-resources 关闭顺序？ 与声明顺序相反（栈式关闭，后开先关）
close() 抛异常会怎样？ 通过 addSuppressed() 附加到原始异常，不覆盖原始异常
Closeable 和 AutoCloseable 的区别？ Closeable 是子接口，close() 只抛 IOException；AutoCloseable 抛 Exception
try 可以没有 catch 只有 finally 吗？ 可以。常见于只需确保清理、不关心具体异常类型的场景

9. 与其他知识的联系

  • JDBC：Connection、Statement、ResultSet 都实现了 AutoCloseable，Spring 的 JdbcTemplate 内部也大量使用 try-with-resources 确保连接归还连接池
  • 并发：Lock.unlock() 应在 finally 中调用（不实现 AutoCloseable），这是 Java 并发编程的惯用法：lock.lock(); try { ... } finally { lock.unlock(); }
  • NIO / Files：Java 7 的 Files.newInputStream() 返回的流同样实现了 AutoCloseable，与 try-with-resources 完美配合
  • Lombok：@Cleanup 注解本质上是在字节码层面插入类似 try-with-resources 的关闭逻辑
  • 设计模式：try-with-resources 是"模板方法模式"的语言级实现——框架（JVM）控制流程骨架，用户只填充 try 块内容
最佳实践总结：凡是涉及外部资源（文件、网络、数据库），优先使用 try-with-resources；finally 块只用于不实现 AutoCloseable 的清理逻辑（如释放锁）；永远不要在 finally 中写 return 或抛出新异常。
