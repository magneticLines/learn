核心类库 · 集合框架

线程安全集合

ConcurrentHashMap、CopyOnWriteArrayList —— 从锁分段到写时复制，彻底搞懂并发集合的设计哲学与使用边界。

1. 为什么普通集合在多线程下会崩

HashMap 和 ArrayList 的所有操作都不带任何同步机制。两个线程同时写入时，内部数组扩容、链表挂接、size 自增都可能交错执行，造成数据丢失甚至死循环（JDK 7 的 HashMap 在并发 resize 时会形成环形链表，导致 CPU 100%）。

经典事故：JDK 7 HashMap 并发死循环
JDK 7 的 resize 采用头插法，并发时两个线程都完成扩容后链表节点顺序相互引用，形成环，之后任何 get 都会死循环。JDK 8 改为尾插法修复了死循环，但数据丢失问题依然存在。

解决思路有三种：

  • 用 Collections.synchronizedMap() 包装 —— 每个方法加 synchronized，粒度粗，并发性差
  • 用 Hashtable —— 同上，全方法加锁，基本被淘汰
  • 用 java.util.concurrent 包的专用并发集合 —— 细粒度锁或无锁，性能好

2. ConcurrentHashMap — 分段锁 → CAS + synchronized

2.1 JDK 7：分段锁（Segment）

JDK 7 将整个 HashMap 切成 16 个 Segment，每个 Segment 是一把独立的 ReentrantLock。两个线程操作不同 Segment 互不影响，并发度最高为 16。

ConcurrentHashMap (JDK 7) ├── Segment[0] → ReentrantLock + HashEntry[] ├── Segment[1] → ReentrantLock + HashEntry[] ├── ... └── Segment[15] → ReentrantLock + HashEntry[]

2.2 JDK 8：Node 数组 + CAS + synchronized（现行方案）

JDK 8 彻底抛弃了 Segment，改用与 HashMap 相同的 数组 + 链表 / 红黑树结构，但在并发控制上做了两层优化：

  • 写入时只锁桶头节点（synchronized(f)，f 为链表或树的第一个节点），不同桶完全并行
  • 空桶用 CAS 插入，完全无锁
  • 扩容时多线程协作：每个线程领取一段桶范围，并行迁移数据
ConcurrentHashMap 核心 put 逻辑（JDK 8 简化版）
final V putVal(K key, V val, boolean onlyIfAbsent) {
    for (;;) {
        Node<K,V>[] tab; Node<K,V> f; int n, i, fh;
        if (tab == null)
            tab = initTable();          // 懒初始化，用 CAS 保证只有一个线程完成
        else if ((f = tabAt(tab, i = (n-1) & hash)) == null) {
            // 空桶：用 CAS 原子插入，失败则重试循环 —— 完全无锁
            if (casTabAt(tab, i, null, new Node<>(hash, key, val)))
                break;
        } else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);   // 正在扩容：帮忙迁移数据
        else {
            // 有冲突：只锁住该桶的头节点 f
            synchronized (f) {
                // 遍历链表 / 红黑树，找到则更新，否则追加
                ...
            }
        }
    }
}
为什么用 synchronized 而非 ReentrantLock？
JDK 8 的 JVM 对 synchronized 做了大量优化（偏向锁、轻量级锁、锁粗化），在竞争不激烈时开销极低。而 ReentrantLock 需要额外的 AbstractQueuedSynchronizer 对象，反而更重。

2.3 常用 API 与注意事项

ConcurrentHashMap 典型用法
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// 1. 原子"不存在才放" —— putIfAbsent
map.putIfAbsent("key", 1);

// 2. 原子累加计数器（推荐写法）
map.merge("word", 1, Integer::sum);

// 3. 原子"不存在则计算"—— computeIfAbsent（常用于多值 Map）
ConcurrentHashMap<String, List<String>> multiMap = new ConcurrentHashMap<>();
multiMap.computeIfAbsent("group", k -> new ArrayList<>()).add("item");

// ❌ 错误：复合操作不是原子的！
if (!map.containsKey("key")) {       // check
    map.put("key", 1);              // act — 两步之间可能被其他线程插入
}
// ✅ 正确：用原子方法替代 check-then-act
map.putIfAbsent("key", 1);
陷阱：size() 不精确
size() 返回的是估算值（通过汇总各 Cell 的 baseCount），在遍历期间其他线程仍可修改 Map。如果需要精确统计，应在外部使用 LongAdder 或 AtomicLong 自己维护。
陷阱：key/value 不允许为 null
ConcurrentHashMap 禁止 null key 和 null value。原因：在并发场景下，get(key) == null 无法区分"key 不存在"还是"value 本身是 null"，会造成二义性。而 HashMap 允许 null 是因为它本来就不是线程安全的。

3. CopyOnWriteArrayList — 写时复制

3.1 核心思想

写时复制（Copy-On-Write，COW）：每次写操作（add/set/remove）都会先把底层数组完整地复制一份，修改完副本后再用 volatile 引用原子地指向新数组。读操作始终操作"快照"，完全无锁。

读线程 → 旧数组 [A, B, C] （无锁，直接读） 写线程 → 加锁 复制 → [A, B, C, D] （新数组） 修改完毕 volatile 引用 → 新数组 释放锁

3.2 源码核心：add 方法

CopyOnWriteArrayList#add（JDK 11 简化）
public boolean add(E e) {
    synchronized (lock) {              // 写操作互斥
        Object[] es = getArray();
        int len = es.length;
        // 复制出一个长度 +1 的新数组 —— 关键！
        es = Arrays.copyOf(es, len + 1);
        es[len] = e;
        setArray(es);                    // volatile 写，对所有线程立即可见
        return true;
    }
}

// get 不加任何锁 —— 读的是某一时刻的快照数组
public E get(int index) {
    return elementAt(getArray(), index);
}

3.3 迭代器的快照语义

iterator() 在创建时固定住了当前数组的引用。此后无论写线程怎么修改，迭代器看到的始终是创建时的快照，不会抛出 ConcurrentModificationException，也不会看到新写入的数据。

迭代器快照演示
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>(List.of("A", "B"));
Iterator<String> it = list.iterator();  // 此时快照 = ["A", "B"]

list.add("C");                            // 写线程：list 变为 ["A","B","C"]

while (it.hasNext())
    System.out.println(it.next());      // 输出 A B，看不到 C
适用场景：CopyOnWriteArrayList 适合读多写极少的场景，例如：监听器列表（事件总线）、白名单/黑名单缓存、配置项列表。每次写都复制全量数组，写频繁时内存和 GC 压力极大。

4. 并发集合横向对比

类 锁机制 读性能 写性能 null 适用场景
HashMap 无 最高 最高 允许 单线程
Hashtable 全方法 synchronized 低 低 不允许 已过时
synchronizedMap 包装锁（this） 低 低 允许 简单需求
ConcurrentHashMap CAS + 桶头 synchronized 高 高 不允许 读写均衡
ArrayList 无 最高 最高 允许 单线程

ConcurrentLinkedQueue CAS 无锁 高 高 不允许 并发队列

5. 实战：多线程词频统计

用 ConcurrentHashMap 统计多个线程并发处理文本的词频，体会原子 API 的价值：

WordCounter.java
import java.util.concurrent.*;
import java.util.concurrent.atomic.LongAdder;

public class WordCounter {

    // LongAdder 比 AtomicLong 在高并发写时更高效（分段累加，读时合并）
    private final ConcurrentHashMap<String, LongAdder> counts
            = new ConcurrentHashMap<>();

    public void count(String word) {
        // computeIfAbsent 保证原子性地创建 LongAdder，再 increment
        counts.computeIfAbsent(word, k -> new LongAdder()).increment();
    }

    public long getCount(String word) {
        LongAdder adder = counts.get(word);
        return adder == null ? 0 : adder.sum();
    }

    public static void main(String[] args) throws Exception {
        WordCounter counter = new WordCounter();
        ExecutorService pool = Executors.newFixedThreadPool(4);
        CountDownLatch latch = new CountDownLatch(4);

        for (int t = 0; t < 4; t++) {
            pool.submit(() -> {
                for (int i = 0; i < 10_000; i++)
                    counter.count("hello");
                latch.countDown();
            });
        }
        latch.await();
        // 4 个线程各 10000 次，期望结果 40000
        System.out.println(counter.getCount("hello"));  // 40000
        pool.shutdown();
    }
}

6. 知识联系

  • volatile：CopyOnWriteArrayList 用 volatile Object[] array 保证数组引用的可见性；读线程无需加锁即可感知到写线程的替换。
  • CAS（Unsafe）：ConcurrentHashMap 的空桶插入用 Unsafe.compareAndSetObject 完成，是整个无锁路径的基础。
  • AQS / ReentrantLock：JDK 7 的 Segment 继承自 ReentrantLock；JDK 8 虽然换成 synchronized，但理解 AQS 有助于理解锁升级过程。
  • Java 内存模型（happens-before）：volatile 写 happens-before 后续 volatile 读，这是 COW 读无锁仍能看到正确数据的理论依据。
  • Stream / forEach：ConcurrentHashMap 的 forEach、reduce 等并行流方法天然利用了 ForkJoinPool，可以指定并行度阈值。

7. 面试高频考点

1. ConcurrentHashMap JDK 7 vs JDK 8 区别？
JDK 7 用分段锁（Segment + ReentrantLock），并发度固定为 16；JDK 8 改为 Node 数组 + CAS + synchronized（锁粒度降至桶级），并发度最大为桶数（默认 16，扩容后更多），且数据结构与 HashMap 对齐（链表 + 红黑树）。
2. 为什么 ConcurrentHashMap 不允许 null？
在并发场景下，get(key) == null 存在二义性：无法区分 key 不存在还是 value 是 null。而在单线程 HashMap 中可以用 containsKey 再判断，但并发场景下这两步之间可能被修改，没有意义。
3. CopyOnWriteArrayList 的 iterator 为什么不抛 CME？
因为迭代器持有的是创建时的数组快照引用，写操作替换的是 volatile 引用指向的新数组，迭代器的引用不会变，所以永远不会有结构修改，也就不会触发 modCount 检查，不抛 ConcurrentModificationException。
4. ConcurrentHashMap 的 size() 为什么不精确？
JDK 8 用 baseCount + CounterCell[]（类似 LongAdder）维护元素数量，size() 汇总各 cell 时不加锁，其他线程仍可并发修改，所以返回的是近似值。若需精确数量，在外部自行维护计数器。
5. 什么时候用 CopyOnWriteArrayList？
读写比例悬殊（读远多于写）且集合规模不大时，如 Spring ApplicationContext 中的事件监听器列表。若写频繁，每次全量复制数组会造成大量 GC，此时应换用 ConcurrentLinkedQueue 或加锁的 ArrayList。
