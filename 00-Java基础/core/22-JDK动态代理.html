22. JDK 动态代理

运行时生成代理类，是 AOP、RPC 框架的底层基石；理解它就理解了 Spring AOP 的一半。

① 为什么需要代理？从一个真实问题出发

假设你写了一个 UserService，某天领导说"每个方法调用前后都要打日志、统计耗时"。 最笨的方式是逐个方法手写 System.currentTimeMillis()，但这会把业务代码和监控代码搅在一起， 改一次全身痛。

代理模式的核心思想：在调用者和真实对象之间插入一个中间人， 由中间人决定"调用前做什么、调用后做什么、要不要真正调用"。 静态代理需要手写代理类（每个接口都要写一遍），而 JDK 动态代理 在运行时自动生成代理类，一套逻辑复用所有接口。

代理模式本质是关注点分离（Separation of Concerns）：业务逻辑归业务， 横切逻辑（日志、事务、鉴权）归代理。这正是 AOP（面向切面编程）的哲学。

② 核心 API：只需记住三样东西

java.lang.reflect
├── Proxy ← 工厂类，newProxyInstance() 生成代理对象
├── InvocationHandler ← 接口，你只需实现 invoke() 方法
└── $Proxy0（运行时生成） ← 真正的代理类，extends Proxy implements 目标接口
元素角色你需要做什么
InvocationHandler 拦截逻辑的容器 实现 invoke(proxy, method, args)
Proxy.newProxyInstance() 代理对象工厂 传入 ClassLoader、接口列表、Handler
$ProxyN 运行时生成的代理类 JDK 自动完成，你不需要写

③ 最小可运行示例：给任意接口加日志

UserService.java — 被代理的接口
public interface UserService {
    String findById(long id);
    void   save(String name);
}
UserServiceImpl.java — 真实实现
public class UserServiceImpl implements UserService {
    @Override
    public String findById(long id) {
        return "User-" + id;
    }
    @Override
    public void save(String name) {
        System.out.println("saving: " + name);
    }
}
TimingHandler.java — 你真正要写的唯一逻辑
import java.lang.reflect.*;

public class TimingHandler implements InvocationHandler {

    private final Object target;          // 被包裹的真实对象

    public TimingHandler(Object target) {
        this.target = target;
    }

    // 每次调用代理的任意方法，都会来到这里
    @Override
    public Object invoke(Object proxy, Method method, Object[] args)
            throws Throwable {

        long start = System.nanoTime();
        try {
            // 真正调用目标方法（反射）
            return method.invoke(target, args);
        } finally {
            long ms = (System.nanoTime() - start) / 1_000_000;
            System.out.printf("[TIMING] %s() cost %d ms%n",
                    method.getName(), ms);
        }
    }
}
Main.java — 组装并使用
UserService real  = new UserServiceImpl();
UserService proxy = (UserService) Proxy.newProxyInstance(
        real.getClass().getClassLoader(),  // ① 用谁的 ClassLoader 加载代理类
        new Class[]{ UserService.class },     // ② 代理要实现哪些接口
        new TimingHandler(real)              // ③ 拦截逻辑
);

proxy.findById(42);   // 输出: [TIMING] findById() cost X ms
proxy.save("Alice"); // 输出: saving: Alice  +  [TIMING] save() cost X ms

④ 运行时发生了什么：$Proxy0 的秘密

调用 Proxy.newProxyInstance() 后，JDK 在内存里生成了一个名叫 $Proxy0（或 $Proxy1、$Proxy2…）的类， 它的结构大致如下：

$Proxy0（JDK 自动生成，反编译后示意）
final class $Proxy0 extends Proxy implements UserService {

    // 每个接口方法对应一个静态 Method 字段，类加载时反射获取
    private static Method m_findById;
    private static Method m_save;

    static {
        m_findById = UserService.class.getMethod("findById", long.class);
        m_save     = UserService.class.getMethod("save", String.class);
    }

    // 每个接口方法的实现：统统转发给 InvocationHandler.invoke()
    public String findById(long id) {
        return (String) h.invoke(this, m_findById, new Object[]{ id });
    }
    public void save(String name) {
        h.invoke(this, m_save, new Object[]{ name });
    }
    // h 是父类 Proxy 里的 InvocationHandler 字段
}

调用链整理如下：

调用方
  ↓ proxy.findById(42)
$Proxy0.findById() ← JDK 生成的代理类
  ↓ h.invoke(this, m_findById, [42])
TimingHandler.invoke() ← 你写的拦截逻辑
  ↓ method.invoke(target, args)
UserServiceImpl.findById() ← 真实业务代码
关键约束：$Proxy0 已经 extends Proxy， Java 是单继承，所以 JDK 动态代理只能代理接口，无法代理纯类。 这是 JDK 代理最重要的限制，也是 CGLIB 存在的原因。

⑤ newProxyInstance 三个参数深挖

参数 1：ClassLoader

代理类是在运行时动态生成的字节码，需要用某个 ClassLoader 来加载它。 通常传 target.getClass().getClassLoader()，确保代理类和目标类 在同一个类加载器命名空间里，才能互相访问。

在 OSGi、Tomcat 这类多 ClassLoader 环境中，ClassLoader 传错会导致 ClassCastException 或 NoClassDefFoundError，这是常见的坑。

参数 2：interfaces 数组

代理类会实现你传入的所有接口。可以传多个接口，代理对象就同时拥有多个类型。 但接口数量越多，方法越多，生成的代理类就越大。

参数 3：InvocationHandler

这是你真正要实现的唯一接口。invoke 的三个参数含义：

  • proxy：代理对象本身（不是目标对象！）；一般不用，但在链式调用时可能需要
  • method：被调用的方法的 Method 对象，可获取方法名、注解、参数类型
  • args：调用时传入的参数数组；无参时为 null（不是空数组！）

⑥ 进阶用法：基于注解的条件拦截

实际项目中，不是所有方法都需要拦截。通过在 invoke 里检查方法注解， 可以实现"只拦截带 @Transactional 的方法"这种精细控制——这正是 Spring 事务的底层思路。

注解驱动的拦截示例
@Override
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {

    // 只对带 @Transactional 注解的方法开启事务
    if (method.isAnnotationPresent(Transactional.class)) {
        beginTransaction();
        try {
            Object result = method.invoke(target, args);
            commit();
            return result;
        } catch (InvocationTargetException e) {
            rollback();
            throw e.getCause();  // 解包真实异常再抛出
        }
    }
    // 不带注解的方法直接透传
    return method.invoke(target, args);
}

⑦ 与其他知识的联系

使用场景框架/类用到动态代理的地方
Spring AOP JdkDynamicAopProxy Bean 实现了接口时，用 JDK 代理；否则用 CGLIB
MyBatis Mapper MapperProxyFactory Mapper 接口没有实现类，全靠动态代理在运行时生成
RPC 框架（Dubbo/gRPC） 客户端 Stub 调用远程接口方法 → invoke() 里序列化参数并发起网络请求
Spring @Transactional TransactionInterceptor invoke() 里判断注解、开启/提交/回滚事务
JUnit Mockito mock(UserService.class) 生成 Mock 对象，invoke() 里返回预设值
与反射的关系：动态代理依赖反射（Method.invoke()）来调用目标方法。 所以动态代理天生带有反射的性能开销。JDK 17+ 对反射做了内联优化，但在 极高频路径上仍应考虑 CGLIB 或 MethodHandle。

⑧ 常见陷阱与误区

陷阱 1：args 为 null 而非空数组

无参方法调用时，args 是 null，不是 new Object[0]。 如果你写 args.length 会抛 NullPointerException。

反例 → 正例
// 反例：直接用 args.length，无参时 NPE
System.out.println("args count: " + args.length);

// 正例：防 null
int count = args == null ? 0 : args.length;

陷阱 2：异常被包装成 InvocationTargetException

目标方法抛出的异常会被 Method.invoke() 包装成 InvocationTargetException。如果你直接 throw e， 调用方看到的是包装后的异常，而不是业务异常，会让 Spring 的事务回滚机制失效。

正确的异常处理模式
try {
    return method.invoke(target, args);
} catch (InvocationTargetException e) {
    throw e.getCause();  // 解包，抛出真实业务异常
} catch (IllegalAccessException e) {
    throw new RuntimeException(e);
}

陷阱 3：代理对象不能强转为实现类

$Proxy0 只实现了接口，并不 extends UserServiceImpl。 把代理对象强转为实现类会抛 ClassCastException，只能转为接口类型。

反例
// 反例：ClassCastException！
UserServiceImpl impl = (UserServiceImpl) proxy;

// 正例：只能转接口
UserService svc = (UserService) proxy;

陷阱 4：Spring Bean 自调用绕过代理

如果一个 Spring Bean 的方法 A 内部直接调用同类方法 B（this.methodB()）， this 指向真实对象而非代理，所以 B 上的 AOP 增强（事务、日志）不会生效。

Spring 自调用问题是使用 AOP 时最高频的生产 Bug。解决方案：注入自身代理 (@Autowired ApplicationContext 获取代理)，或改用 AspectJ 编译时织入。

⑨ 面试高频考点

问题核心答案
JDK 动态代理为什么只能代理接口？ 生成的 $ProxyN 已 extends Proxy，Java 单继承，无法再继承目标类
JDK 代理 vs CGLIB 代理，Spring 怎么选？ 有接口 → JDK 代理（默认）；无接口或 proxyTargetClass=true → CGLIB
invoke 的 proxy 参数有什么用？ 代理对象本身；可用于在 Handler 内部调用代理方法（而非真实对象方法），实现链式/装饰
动态代理和静态代理的本质区别？ 静态代理编译时确定，每个接口写一个类；动态代理运行时字节码生成，一套 Handler 复用所有接口
MyBatis 的 Mapper 为什么不需要实现类？ MapperProxy 实现了 InvocationHandler，在 invoke() 里解析 SQL 并执行
如何获取代理对象背后的 InvocationHandler？ Proxy.getInvocationHandler(proxyObj)

⑩ JDK 版本演进：有什么变化？

版本变化
JDK 8 经典实现：ASM 生成字节码，存入 sun.misc.ProxyGenerator
JDK 9 代理类移入模块系统，sun.misc.ProxyGenerator 改为 java.lang.reflect.ProxyGenerator
JDK 15（JEP 371） 正式引入 Hidden Classes；新版本 JDK 的动态代理实现在类生成机制上有过底层优化（涉及 Hidden Classes 等新特性），具体实现细节可能随版本变化，这里不深入展开底层类生成机制
JDK 17+ 反射内联优化，Method.invoke() 在 JIT 热点后可内联，性能大幅接近直接调用
可以用 -Djdk.proxy.ProxyGenerator.saveGeneratedFiles=true（JDK 8： sun.misc.ProxyGenerator.saveGeneratedFiles）把生成的 $Proxy0.class 保存到磁盘，再用 javap -c 反编译，亲眼看到 JDK 生成了什么代码。
