02 · 迭代器与 for-each

Iterator 模式的设计动机 · for-each 编译本质 · CME 陷阱 · 六种遍历对比

1为什么要有迭代器

Java 集合框架里有 ArrayList（数组实现）、LinkedList（链表）、 HashSet（哈希表）、TreeSet（红黑树）……内部结构完全不同。 如果让调用方自己操心"如何遍历"，就要分别写 for(int i=0;i<n;i++)、 指针跳跳、桶+链表双层循环……这违反了面向接口编程的原则。

生活类比 — 图书馆书单

图书馆有平装书、有线装古籍、有期刊合订本，你不需要知道每种书怎么翻页。 管理员给你一个"下一本"按钮——这就是迭代器。 不管内部是怎么存的，对外永远只暴露 hasNext / next。

这正是 GoF 迭代器模式的核心：把"遍历逻辑"从集合中剥离出来， 封装成独立对象，使得调用方代码对所有集合统一。 

// 调用方不关心 col 是 ArrayList 还是 LinkedList 还是 TreeSet
Collection<String> col = getFromSomewhere();
Iterator<String> it = col.iterator();
while (it.hasNext()) {
    System.out.println(it.next());
}

2Iterator 三个核心方法

public interface Iterator<E> {
    boolean hasNext();  // 还有下一个元素吗？
    E       next();     // 返回下一个元素，并推进指针
    default void remove() {  // 删除 next() 刚返回的元素（可选操作）
        throw new UnsupportedOperationException();
    }
}
设计细节

hasNext() 不消耗元素，可以反复调用； next() 每调用一次才向前移动一步。 remove() 必须在 next() 之后调用， 且每次 next() 只能调用一次 remove()——底层靠 lastRet 记录位置。

完整手写遍历示例

List<String> names = new ArrayList<>(List.of("Alice", "Bob", "Charlie", "Dave"));

Iterator<String> it = names.iterator();
while (it.hasNext()) {
    String name = it.next();
    if (name.startsWith("C")) {
        it.remove();   // 安全删除，不会 CME
    }
}
// 结果: [Alice, Bob, Dave]
System.out.println(names);

3for-each 的本质

for (T x : col) 是语法糖，javac 在编译时把它展开， 展开方式取决于操作对象的类型。

编译前（你写的）
// 集合
for (String s : list) {
    System.out.println(s);
}

// 数组
for (int x : arr) {
    System.out.println(x);
}
编译后（javac 生成）
// 集合 → Iterator while 循环
Iterator<String> $i = list.iterator();
while ($i.hasNext()) {
    String s = $i.next();
    System.out.println(s);
}

// 数组 → 下标 for 循环（注意：不是 Iterator！）
for (int $i = 0; $i < arr.length; $i++) {
    int x = arr[$i];
    System.out.println(x);
}
关键区别

数组的 for-each 不走 Iterator，直接翻译成下标循环，性能与手写 for(int i) 完全相同。 集合的 for-each 走 Iterator，会有对象分配（现代 JIT 通常会消除）。

集合能支持 for-each，前提是实现了 java.lang.Iterable<T> 接口： 

public interface Iterable<T> {
    Iterator<T> iterator();
    // Java 8 新增了 forEach / spliterator 默认方法
}

4自定义 Iterable — 让任意类支持 for-each

实现 Iterable 即可。下面用一个整数范围类演示，不分配任何数组，按需生成：

/**
 * 表示 [start, end) 的整数范围，支持 for-each 遍历。
 * 零内存分配：不预先存储元素，迭代器内部只维护一个 int 计数器。
 */
public class Range implements Iterable<Integer> {

    private final int start;
    private final int end;

    public Range(int start, int end) {
        this.start = start;
        this.end   = end;
    }

    @Override
    public Iterator<Integer> iterator() {
        // 匿名内部类，也可用 Lambda + 工厂方法
        return new Iterator<>() {
            int cur = start;

            @Override public boolean hasNext() { return cur < end; }
            @Override public Integer  next()   {
                if (!hasNext()) throw new NoSuchElementException();
                return cur++;
            }
        };
    }

    public static void main(String[] args) {
        for (int i : new Range(1, 6)) {
            System.out.print(i + " ");  // 1 2 3 4 5
        }
    }
}
实战价值

数据库游标、文件逐行读取、分页 API 懒加载……只要实现 Iterable， 调用方就能用 for-each，无需知道底层是 IO 还是 HTTP。

5ConcurrentModificationException — 最大陷阱

原理：fail-fast 机制

ArrayList 内部有一个 int modCount 字段， 每次结构性修改（add / remove / clear）都让它 +1。 创建迭代器时，迭代器把当时的 modCount 复制到 expectedModCount。 每次调用 next() 都校验： 

// ArrayList$Itr.next() 源码（简化）
final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}

只要迭代期间通过集合本身（而非迭代器）修改了结构， modCount 就变了，下次 next() 立刻抛异常。 这叫 fail-fast——宁可报错，也不静默给出错误结果。

注意

fail-fast 是尽力而为的行为，Java 文档明确说不保证一定抛出。 多线程场景下不能依赖它来保证线程安全，要用 CopyOnWriteArrayList 等并发集合。

反例 — 触发 CME

List<String> list = new ArrayList<>(List.of("a", "b", "c"));

// 反例 1：在 for-each 中用 list.remove()
for (String s : list) {
    if (s.equals("b")) {
        list.remove(s);  // 抛 ConcurrentModificationException！
    }
}

// 反例 2：普通 for-i 删除时漏跳元素（不抛异常，但结果错误）
for (int i = 0; i < list.size(); i++) {
    if (list.get(i).equals("b")) {
        list.remove(i);  // 删除后 i 不回退，下一个元素被跳过
    }
}

三种正确解法

// 解法 1：Iterator.remove()（经典，适合顺序删除）
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove();
}

// 解法 2：Java 8 removeIf（最简洁，底层也是迭代器）
list.removeIf(s -> s.equals("b"));

// 解法 3：倒序下标删除（适合需要索引的场景）
for (int i = list.size() - 1; i >= 0; i--) {
    if (list.get(i).equals("b")) list.remove(i);
}

6ListIterator — List 专属的双向迭代器

ListIterator 扩展了 Iterator，专为 List 设计，支持双向遍历和就地修改： 

List<String> list = new ArrayList<>(List.of("A", "B", "C"));
ListIterator<String> lit = list.listIterator(list.size()); // 从尾部开始

// 反向遍历
while (lit.hasPrevious()) {
    System.out.print(lit.previous() + " "); // C B A
}

// 从头正向：set 替换 / add 插入
lit = list.listIterator();
while (lit.hasNext()) {
    String s = lit.next();
    lit.set(s.toLowerCase());  // 原地替换，不触发 CME
}
System.out.println(list); // [a, b, c]

lit.add("d"); // 在当前位置插入，不触发 CME
ListIterator 方法速查
  • hasNext() / next() / nextIndex() — 向前
  • hasPrevious() / previous() / previousIndex() — 向后
  • set(E e) — 替换 next() 或 previous() 返回的元素
  • add(E e) — 在当前游标位置插入
  • remove() — 删除 next() 或 previous() 返回的元素

7六种遍历方式对比

遍历方式 适用对象 能否删除 能否取索引 性能 代码简洁度
for (int i) 数组 / List 可（需小心跳元素） ✅ 最快（无对象分配） 一般
for-each 数组 / Iterable ❌（集合）/ ❌（数组） ❌ 集合稍慢（迭代器对象） 最简洁
Iterator while 所有 Collection ✅ it.remove() ❌ 同 for-each 一般
ListIterator List 专属 ✅ + set/add ✅ nextIndex 同 Iterator 较啰嗦
forEach(lambda) 所有 Iterable ❌（需配合 removeIf） ❌ 接近 Iterator 极简洁
Stream 所有 Collection ❌（流是只读视图） ❌（需 IntStream.range） 小集合略慢；大集合可并行 链式极佳

8for-each 的局限性与替代方案

  • 无法取索引 → 用 for(int i=0;i<list.size();i++) 或 IntStream.range(0, list.size()).forEach(i -> ...)
  • 无法同时遍历两个集合 → 用下标循环，或先转成 Iterator 手动推进
  • 无法在遍历中删除 → 用 Iterator.remove() 或 removeIf
  • 无法提前退出并拿到迭代器当前位置 → 用 Iterator 手写，break 后迭代器对象还在，可继续使用
  • 无法在遍历中修改元素本身（替换） → 用 ListIterator.set() 或 replaceAll

9Java 8+ 新 API 的关系

Java 8 在 Iterable、Iterator、Collection 上增加了一批默认方法， 都是对 Iterator 的封装，不是替代：

List<String> list = new ArrayList<>(List.of("a", "b", "c", "d"));

// Iterable.forEach — 等价于 for-each，但用 Lambda
list.forEach(System.out::println);

// Collection.removeIf — 安全删除，底层用 Iterator.remove()
list.removeIf(s -> s.equals("b"));

// List.replaceAll — 就地替换每个元素，底层用 ListIterator.set()
list.replaceAll(String::toUpperCase);  // [A, C, D]

// Iterator.forEachRemaining — 消费迭代器剩余元素
Iterator<String> it = list.iterator();
it.next();                             // 消耗第一个
it.forEachRemaining(System.out::println); // 打印剩余

// Spliterator — 可分割迭代器，Stream 并行的底层基础
Spliterator<String> sp = list.spliterator();
sp.forEachRemaining(System.out::println);
脉络总结

Iterator（Java 1.2）→ Iterable（Java 5，for-each 基础）→ ListIterator（双向）→ Spliterator（Java 8，并行 Stream 基础）→ forEach / removeIf / replaceAll（Java 8，语法糖）

越往后越方便，但底层都是迭代器思想。

10面试高频考点

for-each 和 Iterator 有什么关系？能互换吗？
for-each 是 Iterator 的语法糖，编译器将 for(T x : col) 翻译成 Iterator while 循环。二者完全等价。 区别：for-each 代码更简洁，但无法调用 it.remove()， 需要删除时必须显式使用 Iterator。
ConcurrentModificationException 的根本原因是什么？如何避免？
迭代器创建时记录 expectedModCount = modCount， 若迭代期间通过集合本身（非迭代器）修改了结构，modCount 变化， next() 检测到不一致就抛异常（fail-fast）。 正确做法：用 Iterator.remove()、removeIf()， 或多线程场景换 CopyOnWriteArrayList。
数组能用 for-each 吗？原理和集合一样吗？
能用，但原理不同。数组的 for-each 被翻译成下标 for 循环， 不走 Iterator，因为数组没有实现 Iterable。 集合的 for-each 才走 Iterator while 循环。
Iterator.remove() 和 List.remove() 有什么区别？
Iterator.remove() 删除后会同步更新 expectedModCount = modCount， 所以后续 next() 不会抛 CME。 List.remove() 只更新 modCount， 迭代器的 expectedModCount 未同步，下次 next() 即抛 CME。
如何让自定义类支持 for-each？
实现 java.lang.Iterable<T> 接口，重写 iterator() 方法返回一个 Iterator<T> 实现即可。不需要继承任何集合类。 适合数据库游标、分页懒加载、数学序列等场景。
