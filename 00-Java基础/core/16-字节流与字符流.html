字节流与字符流

InputStream / OutputStream vs Reader / Writer — 两套体系的设计哲学、核心 API、编码原理与使用陷阱

为什么要有两套流体系？

计算机存储数据的最小单位是 字节（byte），文本只是人类赋予字节序列的一种"解读方式"—— 同一组字节用 UTF-8 读是汉字，用 Latin-1 读就是乱码。 Java 在 JDK 1.0 时只有字节流，很快开发者就发现：每次处理文本都要手动 new String(bytes, charset)，既繁琐又容易出错。 于是 JDK 1.1 引入了 字符流（Reader / Writer），统一处理 char（Unicode 字符）。

设计哲学：字节流面向"原始数据"（图片、音频、网络协议等），字符流面向"文本"。 字符流内部仍然依赖字节流，只不过它在两者之间完成了编解码的桥接。

类族全景

java.io ├── InputStream ← 字节输入流抽象基类 │ ├── FileInputStream ← 从文件读字节 │ ├── ByteArrayInputStream ← 从内存 byte[] 读 │ ├── FilterInputStream ← 装饰器基类 │ │ ├── BufferedInputStream │ │ └── DataInputStream │ └── ObjectInputStream ← 对象反序列化 │ ├── OutputStream ← 字节输出流抽象基类 │ ├── FileOutputStream │ ├── ByteArrayOutputStream │ ├── FilterOutputStream │ │ ├── BufferedOutputStream │ │ ├── DataOutputStream │ │ └── PrintStream ← System.out 就是它 │ └── ObjectOutputStream │ ├── Reader ← 字符输入流抽象基类 │ ├── InputStreamReader ← 字节→字符的桥接器（解码） │ │ └── FileReader ← 方便类，固定用平台编码 │ ├── StringReader │ └── BufferedReader ← 下一节详讲 │ └── Writer ← 字符输出流抽象基类 ├── OutputStreamWriter ← 字符→字节的桥接器（编码） │ └── FileWriter ├── StringWriter ├── PrintWriter └── BufferedWriter

两棵树都遵循 装饰器模式（Decorator Pattern）： 核心"节点流"（如 FileInputStream）负责真正的 I/O； "处理流"（如 BufferedInputStream）套在外面叠加能力，可以无限嵌套。 这正是 GoF 装饰器模式的教科书案例。

字节流核心 API

InputStream 关键方法

方法说明
int read()读一个字节，返回 0–255；到达流末尾返回 -1
int read(byte[] b)批量读，返回实际读到的字节数（非 b.length！）
int read(byte[] b, int off, int len)带偏移量的批量读
byte[] readAllBytes()JDK 9+，读取全部字节（注意大文件 OOM）
long transferTo(OutputStream out)JDK 9+，流拷贝的简洁写法
void close()释放底层资源；未关闭会导致文件句柄泄漏

示例：逐字节 vs 批量读取的性能差距

Java字节流读文件
// ❌ 反例：逐字节读，每次都要系统调用，100KB 文件要调用 10 万次
try (InputStream in = new FileInputStream("data.bin")) {
    int b;
    while ((b = in.read()) != -1) {
        process((byte) b);
    }
}

// ✅ 正例：使用 byte[] 缓冲区，大幅减少系统调用次数
try (InputStream in = new FileInputStream("data.bin")) {
    byte[] buf = new byte[8192];  // 8KB，典型的 OS page-aware 大小
    int n;
    while ((n = in.read(buf)) != -1) {
        process(buf, 0, n);  // 注意：只处理实际读到的 n 个字节
    }
}
陷阱：批量读时 read(buf) 的返回值 n 可能小于 buf.length！ 网络流、管道流中这种情况极为常见。永远不要假设"读满了 buf 才返回"，要用 n 来划定有效范围。

字节流实战：文件拷贝

Java文件拷贝 — 三种写法对比
import java.io.*;
import java.nio.file.*;

public class FileCopyDemo {

    // 写法 1：手动 byte[] 缓冲（最通用，兼容 JDK 6+）
    static void copyV1(String src, String dst) throws IOException {
        try (InputStream  in  = new FileInputStream(src);
             OutputStream out = new FileOutputStream(dst)) {
            byte[] buf = new byte[8192];
            int n;
            while ((n = in.read(buf)) != -1)
                out.write(buf, 0, n);
        }
    }

    // 写法 2：JDK 9+ transferTo（内部仍是 byte[] 循环）
    static void copyV2(String src, String dst) throws IOException {
        try (InputStream  in  = new FileInputStream(src);
             OutputStream out = new FileOutputStream(dst)) {
            in.transferTo(out);
        }
    }

    // 写法 3：NIO Files.copy（推荐生产使用，底层可走 OS zero-copy）
    static void copyV3(String src, String dst) throws IOException {
        Files.copy(Path.of(src), Path.of(dst),
                   StandardCopyOption.REPLACE_EXISTING);
    }
}

编解码桥接器：InputStreamReader / OutputStreamWriter

这是两套体系的"胶水层"。理解它的关键在于：字节流不知道字符集，字符流不知道如何做 I/O， 桥接器把两件事撮合在一起。

Java显式指定编码（强烈推荐）
import java.io.*;
import java.nio.charset.StandardCharsets;

// 读：字节流 → 指定 UTF-8 解码 → 字符流
try (Reader r = new InputStreamReader(
        new FileInputStream("readme.txt"),
        StandardCharsets.UTF_8)) {
    char[] cbuf = new char[1024];
    int n;
    while ((n = r.read(cbuf)) != -1)
        System.out.print(new String(cbuf, 0, n));
}

// 写：字符流 → 指定 UTF-8 编码 → 字节流
try (Writer w = new OutputStreamWriter(
        new FileOutputStream("output.txt"),
        StandardCharsets.UTF_8)) {
    w.write("你好，世界\n");
    w.flush();  // 字符流有内部缓冲，必须 flush 才能落盘
}
FileReader / FileWriter 的坑：这两个"方便类"在 JDK 11 之前强制使用平台默认编码（Windows 上通常是 GBK）。 把代码从 Windows 移到 Linux 时中文就会乱码。JDK 11 才增加了接受 Charset 参数的构造器。 结论：生产代码始终用 InputStreamReader(stream, charset) 而不是 FileReader。

资源管理：try-with-resources

InputStream、OutputStream、Reader、Writer 都实现了 Closeable（进而是 AutoCloseable），因此可以用 JDK 7 引入的 try-with-resources 语法自动关闭。

Java多个流的正确嵌套关闭顺序
// try-with-resources 会按"声明的逆序"关闭：先关 reader，再关 fis
try (FileInputStream   fis    = new FileInputStream("file.txt");
     InputStreamReader reader = new InputStreamReader(fis, "UTF-8")) {
    // 使用 reader...
}
// 等价于：finally { try { reader.close(); } finally { fis.close(); } }

// ❌ 反例：只把最外层流放进 try-with-resources，内层流若初始化失败会泄漏
FileInputStream fis = new FileInputStream("file.txt");
try (InputStreamReader r = new InputStreamReader(fis)) {  // 如果这里抛异常...
    // fis 已经打开但没有被 try 管理，fis 将泄漏
}

装饰器模式在 IO 中的体现

Java IO 的设计是教科书级别的装饰器模式案例。理解这一点，你就能看懂为什么 new DataInputStream(new BufferedInputStream(new FileInputStream(...))) 这样的"俄罗斯套娃"写法在 Java IO 中无处不在。

Java装饰器层叠 — 读写带缓冲的基本数据类型
// 层 1：FileInputStream   → 真正从磁盘读字节（系统调用层）
// 层 2：BufferedInputStream → 加 8KB 内存缓冲区（减少系统调用）
// 层 3：DataInputStream   → 把字节解读为 int/double/boolean 等类型
try (DataInputStream dis = new DataInputStream(
        new BufferedInputStream(
            new FileInputStream("data.bin")))) {

    int    version = dis.readInt();
    double value   = dis.readDouble();
    boolean flag   = dis.readBoolean();
}

// 对应的写端：每层职责相同，顺序相反
try (DataOutputStream dos = new DataOutputStream(
        new BufferedOutputStream(
            new FileOutputStream("data.bin")))) {

    dos.writeInt(1);
    dos.writeDouble(3.14);
    dos.writeBoolean(true);
}

两套体系横向对比

维度字节流（InputStream / OutputStream）字符流（Reader / Writer）
数据单位 byte（8 bit） char（16 bit，Unicode）
适用场景 图片、音频、视频、二进制协议、对象序列化 文本文件、XML、JSON、日志
编码意识 无，透传原始字节 有，构造时指定 Charset
性能 略高（少一层编解码） 略低（多一层 charset 转换）

桥接类 — InputStreamReader / OutputStreamWriter
引入版本 JDK 1.0 JDK 1.1

与 NIO（java.nio）的关系

JDK 1.4 引入的 NIO 并不是"替换"IO，而是在它之上提供了更高效的 Channel + Buffer 模型：

  • java.io（BIO）：面向流，每次 read() 都阻塞当前线程等待数据，适合连接数少的场景
  • java.nio（NIO）：面向缓冲区，配合 Selector 实现非阻塞多路复用，一个线程可管理数千连接
  • java.nio.file.Files：工具类，内部实现用 NIO，但 API 依然提供 InputStream 接口兼容老代码
实践建议：文件操作优先用 java.nio.file.Files（如 Files.readString、Files.writeString、Files.copy）， 网络 I/O 需要高并发时用 NIO 或 Netty，传统 IO 流仍是最易理解、最适合学习底层原理的起点。

常见陷阱汇总

陷阱 1：忘记 flush()

JavaWriter 的缓冲陷阱
Writer w = new FileWriter("log.txt");
w.write("重要日志");
// ❌ 程序崩溃，"重要日志" 还在内存缓冲区，文件为空！
w.flush();  // ✅ 正确：写完立刻刷
w.close();  // close() 内部会调用 flush()，但崩溃时不会走到这里

陷阱 2：read() 返回 int，不是 byte

InputStream.read() 返回 int（范围 0–255），不是 byte（范围 -128–127）。 这是故意的：-1 表示 EOF，如果返回 byte 就无法区分"读到了 0xFF 字节"和"流结束"。

Javaread() 类型陷阱
int  b = in.read();        // ✅ 正确，用 int 接
if (b == -1) break;          // EOF 检测

byte b2 = (byte) in.read();  // ❌ 若读到 0xFF，强转为 -1，与 EOF 值混淆

陷阱 3：在循环外只判断一次 -1

Java
// ❌ 常见错误写法：读取网络流时会丢失数据
int n = in.read(buf);
if (n != -1) process(buf, 0, n);  // 只读了第一批！

// ✅ 正确：循环读取直到 -1
int n;
while ((n = in.read(buf)) != -1)
    process(buf, 0, n);

陷阱 4：FileOutputStream 默认截断文件

Java
// ❌ 每次运行都会清空文件，再写新内容
new FileOutputStream("log.txt");

// ✅ 追加模式：第二个参数 true = append
new FileOutputStream("log.txt", true);

面试高频考点

Q1：字节流和字符流的区别？
字节流以 byte 为单位，无编码概念；字符流以 char 为单位，内部通过 Charset 完成编解码。 字符流底层仍然依赖字节流，InputStreamReader/OutputStreamWriter 是桥接层。
Q2：Java IO 用了什么设计模式？
装饰器模式（Decorator）。InputStream 是抽象组件，FileInputStream 是具体组件， FilterInputStream 是抽象装饰器，BufferedInputStream / DataInputStream 是具体装饰器。 可以无限嵌套叠加功能，符合"对扩展开放、对修改关闭"原则。
Q3：FileReader 为什么不推荐？
JDK 11 之前 FileReader 硬编码使用平台默认字符集（Charset.defaultCharset()），跨平台时中文必定乱码。 应改用 new InputStreamReader(new FileInputStream(path), StandardCharsets.UTF_8)。
Q4：read() 为什么返回 int 而不是 byte？
需要用 -1 表示流末尾（EOF）。如果返回 byte（范围 -128~127），则字节值 0xFF 会被解读为 -1， 与 EOF 混淆，无法正确区分。
Q5：BIO 和 NIO 的本质区别？
BIO（java.io）每次 read 都阻塞线程，一个线程对应一个连接； NIO（java.nio）面向缓冲区，结合 Selector 实现非阻塞多路复用，一个线程可监听多个 Channel 的事件， 适合高并发场景（如 Netty 就基于 NIO 构建）。
