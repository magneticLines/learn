09 异常体系

Java 核心基础 · 第 09 节 — 为什么 Java 设计了两类异常？如何正确用？

① 为什么需要异常机制

在 C 语言时代，错误处理靠返回值约定：成功返回 0，失败返回 -1 或 NULL。这带来三个痛点：

C 风格痛点具体表现
忘记检查返回值fopen() 返回 NULL，不检查直接用 → 段错误
正常流程被污染业务代码里到处夹杂 if (ret < 0) goto err;
错误信息稀薄-1 无法携带"哪行代码出错、为什么出错"

Java 的异常机制把错误对象（携带消息+栈轨迹）与正常流程分离：正常代码在 try 块里写，错误处理在 catch 块里写，层层向上传播，中途任何一层都可以拦截，不拦就继续抛。

② 完整继承树

java.lang.Object
└── java.lang.Throwable
    ├── java.lang.Error　　　　　　　　　　— JVM/系统级别，不应捕获
    │   ├── OutOfMemoryError
    │   ├── StackOverflowError
    │   └── AssertionError …
    └── java.lang.Exception
        ├── IOException　　　　　　　　　　　— checked，编译器强制处理
        ├── SQLException
        ├── ClassNotFoundException
        └── RuntimeException　　　　　　　　— unchecked，编译器不强制
            ├── NullPointerException
            ├── ClassCastException
            ├── ArrayIndexOutOfBoundsException
            ├── IllegalArgumentException
            └── IllegalStateException …

③ Throwable 核心字段（源码级）

字段类型作用
detailMessageStringgetMessage() 返回的文字，描述出错原因
causeThrowable异常链：记录"导致我被抛出的根因"，用 initCause() 或构造器传入
stackTraceStackTraceElement[]栈轨迹快照，首次调用 fillInStackTrace() 时捕获（昂贵！）
suppressedExceptionsList<Throwable>try-with-resources 中被"压制"的异常，通过 getSuppressed() 可取出

异常链示例：底层 SQLException 被包成业务异常时，不应丢弃原因：

// 反例：吞掉根因
throw new ServiceException("DB error");   // 排查时一脸懵

// 正例：保留 cause
throw new ServiceException("DB error", e);  // e 是原始 SQLException
// 打印时 getCause() 可追溯到根源

④ Error — JVM 级别的灾难

Error 类型典型触发场景能否恢复
OutOfMemoryError堆/元空间耗尽基本不能
StackOverflowError无限递归（栈帧耗尽）当次请求失败，JVM 仍可继续
AssertionError-ea 断言失败修代码
LinkageError类版本冲突（NoClassDefFoundError）修依赖
⚠ 不要 catch(Error)
Error 代表 JVM 无法继续正常运行的状态。捕获后假装没事，后续代码在脆弱环境下运行， 结果更难预测。顶层框架（如 Tomcat）会统一处理线程级别的 Throwable，无需业务代码介入。

⑤ checked vs unchecked 对比

checked（受检异常）unchecked（非受检异常）
继承自Exception（但不含 RuntimeException 分支）RuntimeException 或 Error
编译器要求必须 try-catch 或 throws 声明无要求
设计意图调用方可以合理预期并处理的外部故障（文件不存在、网络断开）调用方编程错误（传了 null、数组越界），修代码比 catch 更正确
典型例子IOException、SQLException、ParseExceptionNullPointerException、IllegalArgumentException
Spring/现代框架态度往往包装为 unchecked 再抛（DataAccessException）直接使用

争议点：Java 是业界少数坚持 checked 异常的主流语言。Kotlin、Scala 全放弃了 checked，理由是它导致 API 层层 throws 污染，且开发者倾向于空 catch 了事。实际项目中常见做法：底层抛 checked，框架边界统一转为 unchecked。

⑥ RuntimeException 典型触发场景

NullPointerException

String s = null;
s.length();   // 抛 NPE — Java 14+ 有 Helpful NPE 信息："Cannot invoke ... because s is null"

ClassCastException

Object o = "hello";
Integer n = (Integer) o;   // 抛 CCE — 类型转换失败
// 正确做法：先 instanceof 判断（Java 16+ 可用 pattern matching）
if (o instanceof Integer i) { ... }

ArrayIndexOutOfBoundsException

int[] arr = {1, 2, 3};
int x = arr[5];   // 合法索引 0~2，访问 5 → 抛 AIOOBE

IllegalArgumentException

// 主动抛出，告知调用方参数不合法
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("age out of range: " + age);
        // 为什么用 IAE 而不是自定义异常？
        // 参数校验是编程错误，IAE 是约定俗成的信号；JDK 内部也大量使用
    }
    this.age = age;
}

IllegalStateException

Iterator<String> it = list.iterator();
it.remove();   // 没有先调用 next() → 抛 ISE，对象处于不合法状态

⑦ 自定义异常最佳实践

/**
 * 业务异常：继承 RuntimeException（unchecked），避免层层 throws 污染。
 * 提供带 cause 的构造器是关键——排查时能追溯到根源。
 */
public class OrderNotFoundException extends RuntimeException {

    private final long orderId;

    public OrderNotFoundException(long orderId) {
        super("Order not found: " + orderId);
        this.orderId = orderId;
    }

    // 带 cause 的构造器，包装底层异常时使用
    public OrderNotFoundException(long orderId, Throwable cause) {
        super("Order not found: " + orderId, cause);
        this.orderId = orderId;
    }

    public long getOrderId() { return orderId; }
}

// ---- 高性能技巧：禁用栈轨迹填充 ----
// 场景：高频业务异常（如参数校验失败），fillInStackTrace() 是性能瓶颈
public class FastValidationException extends RuntimeException {

    public FastValidationException(String msg) {
        // writableStackTrace=false → 跳过昂贵的栈快照，性能可提升 10~100x
        super(msg, null, true, false);
    }
}

⑧ 4 个经典陷阱

陷阱 1：吞异常（空 catch）

// ❌ 最危险的写法——异常消失，排查时毫无线索
try {
    loadConfig();
} catch (IOException e) {
    // 什么都不写，或只写 e.printStackTrace()（生产日志里找不到）
}

// ✅ 至少记录日志，或重新抛出
} catch (IOException e) {
    log.error("Failed to load config", e);   // 保留 e，日志框架会打印完整栈
    throw new AppStartupException("Config load failed", e);
}

陷阱 2：finally 里 return / throw（吞掉 try 中的异常）

// ❌ finally 的 return 会静默丢弃 try 块里的异常
public int badMethod() {
    try {
        throw new RuntimeException("important error");
    } finally {
        return 42;   // 异常被 JVM 静默丢弃，调用方拿到 42，毫不知情
    }
}

// 原则：finally 块里不要 return，不要抛新异常（除非有意替换）

陷阱 3：catch 顺序写反（编译报错）

// ❌ 父类在前，子类永远不可达 → 编译错误
try { ... }
catch (Exception e) { ... }
catch (IOException e) { ... }   // 编译器报错：已被上面捕获

// ✅ 子类在前，父类在后（或用多异常 catch）
catch (FileNotFoundException e) { ... }
catch (IOException e) { ... }
catch (Exception e) { ... }

陷阱 4：new Exception() 性能问题

// 每次 new Exception() 都会调用 fillInStackTrace()，
// 遍历当前线程全部栈帧并生成 StackTraceElement 数组——非常昂贵。
// 在高频路径（循环、热点方法）里不要用异常做流程控制。

// 反例：用异常做 break
for (...) {
    try {
        if (done) throw new BreakException();   // ❌ 性能灾难
    } catch (BreakException e) { break; }
}
// 正例：直接 break 或 return，异常只用于"真正的异常情况"

⑨ 与其他知识的联系

try-with-resources（Java 7）

// 自动调用 close()，即使 read() 抛异常；
// close() 自身的异常被"压制"（添加到 suppressed 列表，不丢弃）
try (InputStream in = new FileInputStream("data.txt")) {
    in.read();
}
// 等价于 try-finally，但更安全：不会出现 finally return 吞异常的问题

多异常 catch（Java 7）

catch (IOException | SQLException e) {
    // e 的类型是两者的公共父类（此处为 Exception）
    log.error("Data access error", e);
}

Spring @ExceptionHandler

Spring MVC 在 DispatcherServlet 统一捕获 Controller 抛出的异常，通过 @ExceptionHandler / @ControllerAdvice 转为 HTTP 响应，这是"顶层统一处理"模式的体现——业务代码只管抛，框架负责兜底。

Optional 与异常的关系

Optional 用于表达"值可能不存在"，避免返回 null 导致的 NPE；而异常表达"出现了意外故障"。两者不可互换：不要用 Optional.empty() 来表示业务错误，也不要用 NPE 来表示"未找到"。

并发中的异常：Future

Future<String> f = executor.submit(() -> riskyTask());
try {
    String result = f.get();   // 阻塞
} catch (ExecutionException e) {
    // 线程内的异常被包装为 ExecutionException
    Throwable cause = e.getCause();   // 取出真实原因
}

⑩ 面试高频考点

问题核心答案
Error 和 Exception 的区别？ Error 是 JVM/系统级别无法恢复的问题（OOM、SOE），不应捕获；Exception 是可处理的程序异常，分 checked/unchecked 两类。
checked 和 unchecked 的设计意图？ checked 面向"可预期的外部故障"（IO、网络），强制调用方处理；unchecked 面向"编程错误"（null、越界），应修复代码而非 catch。
finally 一定执行吗？ 几乎一定，除非：System.exit()、JVM 崩溃、执行 finally 的线程被强制 kill。注意：finally 里的 return 会覆盖 try/catch 的返回值。
try-with-resources 的原理？ 编译器将其展开为 try-finally，并自动调用 AutoCloseable.close()；若 try 体和 close 都抛异常，close 的异常被"压制"（addSuppressed），不丢弃。
如何保证异常链不丢失？ 包装异常时必须传入 cause：new MyException("msg", e)。不传则 getCause() 返回 null，根因永久丢失。
new Exception() 为何慢？ 构造时调用 fillInStackTrace() 遍历当前线程所有栈帧，生成 StackTraceElement[] 数组，是 native 调用，非常昂贵。可用 super(msg, null, true, false) 禁用栈填充。
