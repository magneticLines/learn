BufferedReader / BufferedWriter

缓冲流的设计哲学：用一次大 I/O 换掉千万次小 I/O，彻底消除系统调用开销。

为什么需要缓冲流？

上一节讲到 FileReader 每次调用 read() 都会触发一次操作系统的 系统调用（syscall）， 从用户态切换到内核态再切回来，开销极大。假设读取一个 1 MB 的文件，每次只读 1 个字符（2 字节）， 需要约 50 万次系统调用。

缓冲流的思路很朴素：在 JVM 堆内存里开一块缓冲区（默认 8192 字节）， 一次从操作系统拿 8192 个字符填满缓冲区，后续的 read() 调用都从内存缓冲区取， 直到缓冲区耗尽才再发起一次 syscall。这样 50 万次 syscall 就压缩成了约 128 次。

类比：你要从仓库搬 1000 件货到车间。每次拿一件来回跑 1000 趟 vs. 推一辆小车每次搬 50 件只跑 20 趟——缓冲区就是那辆小车。

类层次结构

BufferedReader 和 BufferedWriter 都是装饰器（Decorator）模式的典型实现，它们本身不做任何真实 I/O，只包装已有的 Reader / Writer。

java.io.Reader
├── java.io.BufferedReader ← 本节主角
│ └── java.io.LineNumberReader
└── java.io.InputStreamReader
    └── java.io.FileReader

java.io.Writer
├── java.io.BufferedWriter ← 本节主角
└── java.io.OutputStreamWriter
    └── java.io.FileWriter
装饰器模式要点：BufferedReader 构造函数接受任意 Reader，这意味着它能包装 FileReader、InputStreamReader、StringReader……一律适用，无需为每种流单独写缓冲逻辑。

核心 API 速览

方法说明返回值
read()读单个字符，从缓冲区取int（-1 表示 EOF）
read(char[], off, len)批量读入字符数组实际读取字符数

write(String)写字符串到缓冲区void

基本用法：逐行读文件

BasicRead.java经典写法（Java 7+ try-with-resources）
import java.io.*;

public class BasicRead {
    public static void main(String[] args) throws IOException {
        // 装饰链：FileReader → BufferedReader
        // try-with-resources 保证 close() 被调用，从外到内依次关闭
        try (BufferedReader br = new BufferedReader(new FileReader("data.txt"))) {
            String line;
            // readLine() 返回 null 表示到达文件末尾
            while ((line = br.readLine()) != null) {
                System.out.println(line);
            }
        }
    }
}
陷阱：readLine() 丢失行终止符。返回的字符串不包含 \n 或 \r\n。 如果你要原样还原文件内容，需要自己拼回换行符，否则写出的文件会丢失最后一行的换行。

Java 8+ 写法：lines() Stream

lines() 返回的是惰性求值（lazy）的 Stream，不会把整个文件一次性读入内存，非常适合处理大文件。

StreamRead.javaJava 8 函数式风格
import java.io.*;
import java.nio.file.*;

public class StreamRead {
    public static void main(String[] args) throws IOException {
        // Files.newBufferedReader 会自动使用 UTF-8（Java 11+），更推荐
        try (BufferedReader br = Files.newBufferedReader(Path.of("data.txt"))) {
            br.lines()
              .filter(l -> l.startsWith("ERROR"))    // 筛选错误行
              .limit(100)                         // 只取前 100 条，不会读完整个文件
              .forEach(System.out::println);
        } // close() 自动关闭底层流，Stream 中途不再消费时也安全
    }
}

BufferedWriter：写文件

写操作的瓶颈同样在 syscall。BufferedWriter 把小块写操作先攒在缓冲区， 等缓冲区满或显式调用 flush() / close() 时才真正写入磁盘。

BasicWrite.java写文件标准用法
import java.io.*;
import java.nio.file.*;
import java.nio.charset.*;

public class BasicWrite {
    public static void main(String[] args) throws IOException {
        // Files.newBufferedWriter 支持 OpenOption，这里 APPEND 模式追加写入
        try (BufferedWriter bw = Files.newBufferedWriter(
                Path.of("output.txt"),
                StandardCharsets.UTF_8,
                StandardOpenOption.CREATE,
                StandardOpenOption.APPEND)) {

            bw.write("第一行内容");
            bw.newLine();  // 平台无关换行：Windows=\r\n，Unix=\n
            bw.write("第二行内容");
            bw.newLine();
            // close() 会自动 flush，但长时间不关闭的场景需要手动 flush()
        }
    }
}
最常见的坑：忘记 flush / close，文件内容为空！
如果程序崩溃或流未关闭，缓冲区里的数据就永远停在内存里，文件是空的或不完整的。 必须用 try-with-resources 确保 close() 一定执行。

自定义缓冲区大小

构造函数的第二个参数可指定缓冲区字节数，默认 8192（8 KB）。

CustomBuffer.java
// 对于读取超大日志文件，可以把缓冲区放大，减少 syscall 次数
BufferedReader br = new BufferedReader(new FileReader("huge.log"), 65536);
// 64 KB 缓冲区：一次 syscall 能取 64K 字符，适合顺序读取大文件

// 对于网络流（每次数据包 ~1.5KB），太大的缓冲区反而增加延迟
BufferedReader netBr = new BufferedReader(socket.getReader(), 4096);

mark / reset：可回退读取

BufferedReader 支持 mark() + reset()， 因为数据已在内存缓冲区里，回退只是移动指针，不需要重新 I/O。 而 FileReader 本身不支持 mark（底层 OS 文件流不可随机回退）。

MarkReset.javapeek 下一行而不消耗它
try (BufferedReader br = new BufferedReader(new FileReader("data.txt"))) {
    // markSupported() 返回 true，才能安全使用 mark/reset
    if (br.markSupported()) {
        br.mark(512);         // 记录当前位置，允许回退最多 512 字符
        String peek = br.readLine();  // 偷看一行
        if (peek.startsWith("#")) {
            br.reset();       // 是注释行，回退重读
        }
    }
}

字符编码：最容易踩坑的地方

new FileReader("file.txt") 使用的是 JVM 默认字符集（由 file.encoding 系统属性决定）， 在不同操作系统上可能是 UTF-8、GBK、ISO-8859-1 等。 这是乱码问题的根源之一。

Encoding.java正确指定编码的三种写法
import java.nio.charset.*;
import java.nio.file.*;

// ✗ 反例：依赖平台默认编码，跨平台乱码
BufferedReader bad = new BufferedReader(new FileReader("file.txt"));

// ✓ 方式一：InputStreamReader 明确指定编码（老代码兼容性好）
BufferedReader br1 = new BufferedReader(
    new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8));

// ✓ 方式二：Java 11 新增 FileReader(File, Charset)，更简洁
BufferedReader br2 = new BufferedReader(
    new FileReader("file.txt", StandardCharsets.UTF_8));

// ✓ 方式三：NIO Files 工厂方法，最推荐（Java 7+）
BufferedReader br3 = Files.newBufferedReader(Path.of("file.txt"), StandardCharsets.UTF_8);

BufferedWriter vs PrintWriter

两者都能写文本，但设计目标不同，选错会踩坑：

特性BufferedWriterPrintWriter
主要设计目标提升 I/O 性能（缓冲）格式化输出（printf 风格）
异常处理抛出 IOException，强迫处理吞掉异常，通过 checkError() 查询
println()没有此方法有，自带换行
自动 flush不支持构造时可开启 auto-flush
适合场景性能敏感的文件写入控制台输出、日志格式化
PrintWriter 的隐患：它会默默吞掉 IOException（仅设置内部 error 标志位）， 程序看起来正常运行，但文件实际没写成功。生产代码处理文件时请优先用 BufferedWriter。

综合实战：文件内容替换

用 BufferedReader 读原文件，用 BufferedWriter 写临时文件，最后 rename——这是编辑文件的标准模式。

ReplaceInFile.java原子性文件替换
import java.io.*;
import java.nio.file.*;
import java.nio.charset.*;

public class ReplaceInFile {

    public static void replaceAll(Path src, String from, String to)
            throws IOException {

        Path tmp = src.resolveSibling(src.getFileName() + ".tmp");

        try (BufferedReader reader = Files.newBufferedReader(src, StandardCharsets.UTF_8);
             BufferedWriter writer = Files.newBufferedWriter(tmp,  StandardCharsets.UTF_8)) {

            String line;
            while ((line = reader.readLine()) != null) {
                writer.write(line.replace(from, to));
                writer.newLine();
            }
        }

        // 原子性替换：rename 在同一文件系统上是原子操作
        // REPLACE_EXISTING 保证覆盖原文件，不留残留临时文件
        Files.move(tmp, src, StandardCopyOption.REPLACE_EXISTING,
                   StandardCopyOption.ATOMIC_MOVE);
    }
}

底层原理：缓冲区的内部结构

查看 JDK 源码（java.io.BufferedReader）：内部维护一个 char[] cb 数组， 以及 nChars（缓冲区有效字符数）和 nextChar（下一次读取的位置）两个指针。

缓冲区状态示意（cb.length = 8192）

┌──────────────────────────────────────────────────────────────┐
│ H e l l o \n W o r l d \n │ <未填充区> │
└──────────────────────────────────────────────────────────────┘
  ↑ ↑ ↑
  0 nextChar=6 nChars=12 8191

每次 readLine() 从 nextChar 开始扫描到 \n，
不需要 syscall，直接返回内存中的切片。

当 nextChar >= nChars 时，才调用 fill()， 此方法通过底层 Reader.read(cb, ...) 触发一次真实 I/O，重新填满缓冲区。

常见陷阱汇总

  • 未关闭流：没有 try-with-resources，BufferedWriter 缓冲区数据永久丢失，文件为空或残缺。
  • 编码未指定：new FileReader(path) 依赖平台默认编码，跨操作系统必现乱码。
  • readLine() 丢换行符：用它做文件复制时，最后一行如果原本无换行，复制后也没有，但中间行会多一个 newLine()，内容可能偏移。
  • 多线程共享一个 BufferedReader：不是线程安全的，两个线程同时读会导致 nextChar 指针竞争，数据错乱。
  • mark 边界溢出：mark(n) 表示"允许回退最多 n 个字符"，若两次 mark/reset 之间读取超过 n 个字符，reset() 会抛 IOException: Mark invalid。
  • 过大文件用 readLine 拼接：lines().collect(Collectors.joining()) 会把整个文件读进内存，大文件直接 OOM。应流式处理，不 collect。

面试高频考点

Q1：BufferedReader 的缓冲区大小默认是多少？为什么是这个值？
默认 8192 字节（8 KB），与大多数操作系统的页大小（4 KB）或磁盘扇区（512 B）对齐， 一次 I/O 能读满一到两个内存页，性价比最高。
Q2：BufferedReader 用了什么设计模式？
装饰器模式（Decorator）。它与 FilterReader 同属装饰器体系， 通过组合而非继承来扩展 Reader 的功能，符合"对扩展开放、对修改封闭"的开闭原则。
Q3：为什么关闭外层 BufferedWriter 就够了，不需要单独关闭 FileWriter？
BufferedWriter.close() 内部会先调 flush()，再调用被包装的 Writer.close()， 形成链式关闭。try-with-resources 会调用最外层的 close()，底层资源自动释放。
Q4：BufferedReader.readLine() 如何判断行尾？
识别三种终止符：\n（Unix）、\r（旧 Mac）、\r\n（Windows）。 三种都能识别，返回的字符串均不含这些字符。
Q5：lines() Stream 和 readLine() while 循环有什么本质区别？
两者底层都调用 readLine()，性能相同。区别在于 lines() 是惰性的， 可以和 filter/limit/map 等操作组合，不会一次性把所有行加载进内存； while 循环每次都必须执行循环体，无法提前短路退出（除非 break）。

与其他知识的联系

  • 装饰器模式：整个 java.io 包是装饰器模式的教科书级实现，理解了 Buffered 系列，其他如 DataInputStream、GZIPOutputStream 都是同一套思路。
  • NIO.2（下一节）：Files.newBufferedReader/Writer 是推荐的工厂方法，背后依然是 BufferedReader/Writer，但 API 更简洁、编码更安全。
  • 操作系统 I/O：缓冲的本质是减少 read(2)/write(2) 系统调用，与操作系统的 page cache、mmap 都是同一层次的优化思路。
  • Java Stream（java.util.stream）：lines() 返回的 Stream 是惰性求值的，与集合 Stream 的行为完全一致，但数据源是 I/O 而非内存集合。
