集合框架（Collections Framework）

JDK 提供的一套生产级数据结构实现——你在数据结构篇手写过的链表、哈希表、红黑树都在里面，Java 帮你处理好了泛型、扩容与线程不安全警告。

整体结构

集合框架的核心接口只有两棵独立的树：以 Collection 为根的单元素集合，和独立存在的 Map 键值对集合。

Iterable └── Collection ├── List 有序、可重复 │ ├── ArrayList │ └── LinkedList ├── Set 无序（TreeSet 有序）、不可重复 │ ├── HashSet │ ├── LinkedHashSet │ └── TreeSet └── Queue 队列 ├── LinkedList └── PriorityQueue Map（独立体系，不继承 Collection） ├── HashMap ├── LinkedHashMap └── TreeMap
选集合的三个问题：需要「key → value」？ → Map；需要「不重复的元素集」？ → Set；需要「有序、可重复的序列」？ → List。

List

ArrayList

底层：Object[] 数组。随机访问 O(1)，中间插入/删除 O(n)（需搬移元素）。核心知识点是自动扩容。

扩容机制
// 初始容量 10（懒分配：构造时是空数组，第一次 add 才申请 10 个槽）
new ArrayList<>();
new ArrayList<>(100);  // 预设容量，避免频繁扩容

// 扩容公式：newCapacity = oldCapacity + (oldCapacity >> 1)  ≈ 1.5 倍
// 底层：Arrays.copyOf()  申请新数组 + 拷贝旧数据

为什么是 1.5 倍而不是 2 倍？权衡空间浪费与扩容频率。Python list 用 ~1.125 倍，Go slice 用 2 倍（少量元素时）——不同语言选了不同权衡点，没有绝对正确。

陷阱一：for 循环删除会漏删
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "b", "c"));
for (int i = 0; i < list.size(); i++)
    if ("b".equals(list.get(i)))
        list.remove(i);  // 删后后面元素左移，下标 i 对应的元素变了！漏删第二个 "b"

// ✅ 正例1：倒序遍历
for (int i = list.size() - 1; i >= 0; i--)
    if ("b".equals(list.get(i))) list.remove(i);

// ✅ 正例2：Iterator（for-each 底层也是 Iterator，但不能调 list.remove）
Iterator<String> it = list.iterator();
while (it.hasNext())
    if ("b".equals(it.next())) it.remove();  // 迭代器自己的 remove，安全

// ✅ 正例3（Java 8+ 最简洁）
list.removeIf("b"::equals);
陷阱二：Arrays.asList() 返回固定长度 List
List<String> fixed = Arrays.asList("a", "b");
fixed.add("c");   // ❌ UnsupportedOperationException

// 需要可变 List：
List<String> mutable = new ArrayList<>(Arrays.asList("a", "b"));

LinkedList

底层：双向链表。同时实现了 List 和 Deque 接口，可以当栈或双端队列用。

LinkedList 作为栈 / 队列
LinkedList<Integer> stack = new LinkedList<>();
stack.push(1);   // addFirst
stack.pop();    // removeFirst

LinkedList<Integer> queue = new LinkedList<>();
queue.offer(1);  // addLast
queue.poll();   // removeFirst

ArrayList vs LinkedList — 怎么选？

场景选谁原因
随机访问 get(i)ArrayListO(1) vs LinkedList O(n)
尾部添加ArrayList均摊 O(1)；LinkedList 也 O(1) 但每个节点有额外对象开销
中间插入/删除（已有节点引用）LinkedListO(1) 改指针；ArrayList O(n) 搬元素
中间插入/删除（按索引）都很慢LinkedList 找位置也要 O(n) 遍历
内存敏感ArrayListLinkedList 每节点多两个引用指针开销
结论：90% 情况用 ArrayList。LinkedList 只在「频繁从头部插入删除且不需要随机访问」时才有优势，比如实现 LRU 缓存的底层链表部分。

HashMap（重点）

哈希表原理（哈希函数、链地址法、负载因子、扩容）在数据结构篇已经讲过，HashMap 是 JDK 实现，但有几个关键细节值得单独记。

底层结构：数组 + 链表 + 红黑树

JDK 1.7：数组 + 链表（链地址法） JDK 1.8：数组 + 链表 + 红黑树

为什么 1.8 加了红黑树？冲突严重时链表查找退化到 O(n)。优化规则：链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转红黑树（O(log n) 查找）。

为什么阈值是 8？泊松分布下，负载因子 0.75 时一个桶出现 8 个冲突的概率约千万分之六，属于极端情况。低于 8 时链表常数更小，引入红黑树得不偿失。

关键参数

初始容量与负载因子
static final int   DEFAULT_INITIAL_CAPACITY = 1 << 4;  // 16，必须是 2 的幂次方
static final float DEFAULT_LOAD_FACTOR      = 0.75f;

// 扩容时机：size > capacity × loadFactor = 16 × 0.75 = 12
// 扩容倍数：2 倍（capacity <<= 1）

为什么容量必须是 2 的幂？计算桶下标用 index = hash & (capacity - 1)。2 的幂减 1 的二进制全是 1（如 15 = 01111），位运算等价于取模但快得多，这是刻意的性能优化。

hash() 扰动函数

HashMap 对 hashCode 的二次处理
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
// 高 16 位与低 16 位异或 —— 把高位信息"揉进"低位，让分布更均匀
// 原因：桶下标只取低位（n 通常很小），不扰动则高位不同的 key 会大量碰撞

陷阱

三个高频陷阱
// 陷阱1：用可变对象做 key 然后改它
Map<List<Integer>, String> map = new HashMap<>();
List<Integer> key = new ArrayList<>(Arrays.asList(1, 2));
map.put(key, "value");
key.add(3);           // hashCode 变了！
map.get(key);          // 返回 null，找不到了
// 结论：HashMap 的 key 应是不可变对象（String、Integer 等）

// 陷阱2：HashMap 不是线程安全的
// 多线程并发 put 可能导致死循环（JDK 1.7 头插法）或数据丢失（JDK 1.8）
// 线程安全场景用 ConcurrentHashMap，不要用 Collections.synchronizedMap（性能差）

// 陷阱3：遍历时不能直接 map.remove
for (String k : map.keySet())
    map.remove(k);  // ❌ ConcurrentModificationException

// ✅ 正确：
map.entrySet().removeIf(e -> someCondition(e.getKey()));

LinkedHashMap

在 HashMap 基础上用一条双向链表串起所有 entry，维护插入顺序（或访问顺序）。LRU 缓存的经典实现就靠它。

用 LinkedHashMap 实现 LRU 缓存
// accessOrder = true：最近访问的 entry 移到链表尾部
LinkedHashMap<String, Integer> lru = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Integer> eldest) {
        return size() > 3;  // 超过容量就删最久未访问的
    }
};
lru.put("a", 1); lru.put("b", 2); lru.put("c", 3);
lru.get("a");   // a 移到链表尾部
lru.put("d", 4);   // 触发 removeEldestEntry，删 b（最久未访问）
// 结果：{c=3, a=1, d=4}

TreeMap

底层：红黑树。所有 key 按自然顺序或传入的 Comparator 排列，支持范围查询 API。

TreeMap 范围查询 API
TreeMap<String, Integer> map = new TreeMap<>();
map.put("banana", 2); map.put("apple", 1); map.put("cherry", 3);

map.firstKey();                 // apple
map.headMap("cherry");          // {apple=1, banana=2}  小于 cherry
map.tailMap("banana");          // {banana=2, cherry=3}  大于等于 banana
map.floorKey("blueberry");      // banana  —— 小于等于 blueberry 的最大 key
map.ceilingKey("blueberry");    // cherry  —— 大于等于 blueberry 的最小 key

Set

Set 的内部实现直接复用了 Map：HashSet = HashMap（value 统一是占位对象 PRESENT），LinkedHashSet = LinkedHashMap，TreeSet = TreeMap。Set 的所有特性和陷阱与对应 Map 完全一致。

Set 去重的本质：hashCode + equals
// 自定义对象必须同时重写 hashCode 和 equals，否则去重失效
class Point {
    int x, y;
    // 只重写 equals 不重写 hashCode：
    // 两个内容相同的 Point 的 hashCode 不同 → 落在不同桶 → Set 认为不重复
}

怎么选？决策树

需要 key-value 映射？ 是 → Map 需要排序 / 范围查询？ → TreeMap 需要保留插入 / 访问顺序？ → LinkedHashMap 其他（绝大多数情况）→ HashMap 只需要存一组不重复的元素？ 是 → Set（对应 Map 的 key-only 版本） 需要有序序列、允许重复？ 是 → List 频繁随机访问？ → ArrayList 频繁头部插入删除？ → LinkedList（少见）

面试高频考点

  1. HashMap 扩容机制：2 倍扩容，rehash 时用 hash & oldCapacity 判断新位置（高位 bit 0 → 原下标，1 → 原下标 + 旧容量），不重新计算 hash。
  2. HashMap 1.7 的死循环：多线程并发 rehash 时，头插法会形成环形链表，导致 get 死循环；1.8 改为尾插法解决死循环，但仍线程不安全。
  3. 负载因子为什么是 0.75：时间（冲突概率）和空间（浪费）的折中，泊松分布下此点综合开销最小。
  4. 为什么 key 要同时重写 hashCode 和 equals：先用 hashCode 定位桶，再用 equals 在桶内比较；只重写其一必出 bug。
  5. ConcurrentHashMap vs Hashtable：Hashtable 是方法级 synchronized（全表锁）；ConcurrentHashMap 1.8 用 CAS + synchronized 锁单个桶，并发粒度细得多。

与其他知识点的联系

  • 迭代器与 for-each：集合框架的所有实现类都要实现 Iterable 接口才能被 for-each 遍历，ArrayList 用下标游标、HashMap 用桶+链表双层游标——不同底层结构对应不同的迭代器实现，这正是迭代器模式"统一遍历接口，隐藏内部差异"的价值所在。
  • Optional：Java 8 之后 Map.getOrDefault()、Optional.ofNullable(map.get(key)) 都是在解决集合取值可能返回 null 的问题，理解集合的"查不到返回 null"约定，才能理解 Optional 存在的必要性。
  • Stream API：集合是 Stream 最主要的数据来源，list.stream()、Collectors.toList() 把集合和 Stream 双向打通，集合负责"存储"，Stream 负责"处理"，二者是互补关系而非替代关系。
  • 泛型：List<Integer>、Map<String, User> 这类写法依赖泛型在编译期做类型检查，避免了 Java 5 之前 Object 集合到处强转的混乱写法，集合框架是学习泛型最直观的场景。
  • 并发编程：HashMap/ArrayList 均非线程安全，对应的并发版本 ConcurrentHashMap、CopyOnWriteArrayList 分别用分段锁/CAS、写时复制解决并发问题，理解单线程集合的实现细节是理解并发集合优化点的前提。
  • equals/hashCode 契约：HashSet 去重、HashMap 定位桶都强依赖对象正确实现 hashCode() 和 equals()，这是 Object 类方法契约在集合框架中最重要的落地场景。
