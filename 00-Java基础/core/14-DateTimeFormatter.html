核心类库 · 日期与时间

14. DateTimeFormatter

日期时间的"翻译官"——在人类可读字符串与机器精确时间对象之间双向转换，线程安全、模式丰富、支持本地化。

为什么需要 DateTimeFormatter？

在上一节学习的 LocalDate、LocalDateTime、ZonedDateTime 都是机器视角的时间——精确、无歧义，但不直接面向人类展示。 现实中我们要处理两类需求：

  • 格式化（Format）：把时间对象变成字符串，展示给用户或写入日志
  • 解析（Parse）：把用户输入或外部系统返回的字符串变成时间对象

Java 8 之前用的是 SimpleDateFormat（SDF），它有一个致命缺陷：线程不安全。 SDF 内部用可变字段存储中间状态，多线程共享一个实例会导致格式化/解析结果错乱，是高并发场景的经典 Bug 来源。

SimpleDateFormat 的线程安全问题
SDF 内部的 Calendar 字段在 format() 和 parse() 期间被反复读写。两个线程同时使用同一个 SDF 实例时，A 线程刚写入的日期可能被 B 线程的操作覆盖，导致解析出错误的时间甚至抛出异常。

DateTimeFormatter 从设计上就是不可变（immutable）且线程安全的：所有状态在构建时一次性固定，之后只读，可以安全地作为 static final 常量在全应用共享。

三种获取方式

1. 使用内置预定义格式

DateTimeFormatter 类上直接提供了一批 static final 常量，覆盖了 ISO 8601 和 RFC 系列标准格式，无需任何配置即可使用。

Java
LocalDate date = LocalDate.of(2025, 6, 29);

// ISO_LOCAL_DATE → "2025-06-29"
String s1 = date.format(DateTimeFormatter.ISO_LOCAL_DATE);

LocalDateTime dt = LocalDateTime.now();
// ISO_LOCAL_DATE_TIME → "2025-06-29T14:30:00"
String s2 = dt.format(DateTimeFormatter.ISO_LOCAL_DATE_TIME);

ZonedDateTime zdt = ZonedDateTime.now(ZoneId.of("Asia/Shanghai"));
// ISO_ZONED_DATE_TIME → "2025-06-29T14:30:00+08:00[Asia/Shanghai]"
String s3 = zdt.format(DateTimeFormatter.ISO_ZONED_DATE_TIME);

2. 使用 ofPattern 自定义模式

这是最常用的方式，模式字母与 SimpleDateFormat 大体相似，但有一些重要差异（见后文表格）。

Java
// 推荐：声明为 static final，一次构建，全局复用
private static final DateTimeFormatter FMT_CN =
    DateTimeFormatter.ofPattern("yyyy年MM月dd日 HH:mm:ss");

private static final DateTimeFormatter FMT_LOG =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss.SSS");

LocalDateTime now = LocalDateTime.now();
System.out.println(now.format(FMT_CN));   // 2025年06月29日 14:30:00
System.out.println(now.format(FMT_LOG));  // 2025-06-29 14:30:00.123

3. 使用 DateTimeFormatterBuilder 精细构建

当需要可选部分、填充、条件分支时，ofPattern 已无法满足，需要用构建器 API。

Java
// 解析既支持 "2025-06-29" 也支持 "2025-06-29 14:30:00" 的混合输入
DateTimeFormatter flexible = new DateTimeFormatterBuilder()
    .appendPattern("yyyy-MM-dd")
    .optionalStart()                      // [ 开始可选部分
        .appendLiteral(' ')
        .appendPattern("HH:mm:ss")
    .optionalEnd()                        // ] 结束可选部分
    .parseDefaulting(ChronoField.HOUR_OF_DAY, 0)
    .parseDefaulting(ChronoField.MINUTE_OF_HOUR, 0)
    .parseDefaulting(ChronoField.SECOND_OF_MINUTE, 0)
    .toFormatter();

LocalDateTime a = LocalDateTime.parse("2025-06-29", flexible);
LocalDateTime b = LocalDateTime.parse("2025-06-29 14:30:00", flexible);

格式化模式字母速查

字母含义示例与 SDF 的差异
y 年（纪元年） yyyy → 2025 — 基本一致
Y 周基准年（ISO 周历） YYYY → 2025 跨年周时与 y 结果不同，容易踩坑
M / L 月份（上下文 / 独立） MM → 06，MMM → Jun SDF 只有 M
d 月中第几天 dd → 29 —
H 小时（0–23） HH → 14 —
h 小时（1–12，AM/PM） hh → 02 —
m 分钟 mm → 30 —
s 秒 ss → 00 —
S 纳秒的分数部分 SSS → 123 SDF 中 S 是毫秒；DTF 中 S 是纳秒分数，语义不同
n 纳秒（0–999999999） n → 123000000 SDF 无此字段
z 时区名称 z → CST —
Z 时区偏移（RFC 822） Z → +0800 —
x / X ISO 时区偏移 xxx → +08:00 SDF 无 x；X 在零偏移时输出 Z
e 周中第几天（数字） e → 7（周日） SDF 用 u；DTF 两者均支持
经典陷阱：YYYY 和 yyyy 的区别
YYYY 是 ISO 周历的"周基准年"。2019 年 12 月 30 日（星期一）属于 2020 年的第 1 周，所以 YYYY 会输出 2020，而 yyyy 输出 2019。日期格式化几乎永远应该用小写 yyyy。

格式化（format）

格式化有两种调用风格，结果完全相同，选用哪种看个人习惯和语义表达：

Java
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd");
LocalDate date = LocalDate.of(2025, 6, 29);

// 风格 1：时间对象.format(formatter)  ← 更常用，阅读顺畅
String s1 = date.format(fmt);

// 风格 2：formatter.format(时间对象)  ← 强调 formatter 是主体
String s2 = fmt.format(date);

// s1.equals(s2) → true，结果相同："2025-06-29"

解析（parse）

解析到具体类型（推荐）

Java
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");

// 直接解析为 LocalDateTime（最常用）
LocalDateTime dt = LocalDateTime.parse("2025-06-29 14:30:00", fmt);

// 解析为 LocalDate（字符串中不含时间部分）
DateTimeFormatter dateFmt = DateTimeFormatter.ofPattern("yyyy/MM/dd");
LocalDate d = LocalDate.parse("2025/06/29", dateFmt);

// 解析带时区的字符串
DateTimeFormatter zonedFmt = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss z");
ZonedDateTime zdt = ZonedDateTime.parse("2025-06-29 14:30:00 CST", zonedFmt);

使用 parseBest 应对不确定类型

有时外部输入可能是日期、也可能是日期时间，parseBest 按优先级尝试解析，返回第一个成功的类型。

Java
DateTimeFormatter flexible = DateTimeFormatter.ofPattern(
    "[yyyy-MM-dd HH:mm:ss][yyyy-MM-dd]"  // [] 表示可选块
);

TemporalAccessor ta = flexible.parseBest(
    "2025-06-29",
    LocalDateTime::from,   // 优先尝试 LocalDateTime
    LocalDate::from          // 降级尝试 LocalDate
);

if (ta instanceof LocalDateTime) {
    // 包含时间
} else if (ta instanceof LocalDate) {
    // 仅日期
}

本地化支持

ofLocalizedDate、ofLocalizedDateTime 等方法结合 Locale 可以输出符合当地习惯的格式，不需要手写本地化模式字符串。

Java
LocalDate date = LocalDate.of(2025, 6, 29);

// 中文短格式 → "2025/6/29"
String cn = date.format(
    DateTimeFormatter.ofLocalizedDate(FormatStyle.SHORT).withLocale(Locale.CHINA)
);

// 英文长格式 → "June 29, 2025"
String en = date.format(
    DateTimeFormatter.ofLocalizedDate(FormatStyle.LONG).withLocale(Locale.US)
);

// 日文完整日期时间 → "2025年6月29日 14時30分00秒 JST"
String ja = LocalDateTime.now().format(
    DateTimeFormatter.ofLocalizedDateTime(FormatStyle.FULL, FormatStyle.MEDIUM)
                      .withLocale(Locale.JAPAN)
);
FormatStyle 有四个级别：FULL（完整描述）、LONG（长格式）、MEDIUM（中等）、SHORT（最简短）。ofLocalizedDateTime 可以分别为日期和时间部分指定不同的样式级别。

与旧 API 的互转

实际项目中难免遇到遗留代码使用 java.util.Date 或 java.sql.Timestamp，需要在新旧 API 之间转换。

Java
// ──── java.util.Date → LocalDateTime ────
Date legacyDate = new Date();
LocalDateTime ldt = legacyDate.toInstant()
    .atZone(ZoneId.systemDefault())
    .toLocalDateTime();

// ──── LocalDateTime → java.util.Date ────
Date back = Date.from(
    ldt.atZone(ZoneId.systemDefault()).toInstant()
);

// ──── java.sql.Timestamp（数据库场景）────
Timestamp ts = Timestamp.valueOf(ldt);          // LocalDateTime → Timestamp
LocalDateTime fromTs = ts.toLocalDateTime();   // Timestamp → LocalDateTime

内部原理：TemporalAccessor 与字段体系

理解 DateTimeFormatter 为何能同时处理 LocalDate、LocalDateTime、ZonedDateTime，需要看其底层设计。

TemporalAccessor（只读接口，所有时间类型的统一视图） ├── LocalDate — 有 YEAR / MONTH_OF_YEAR / DAY_OF_MONTH ├── LocalTime — 有 HOUR_OF_DAY / MINUTE_OF_HOUR / SECOND_OF_MINUTE ├── LocalDateTime — 以上全有 ├── ZonedDateTime — 以上全有 + OFFSET_SECONDS + ZONE_ID └── Instant — 仅有 INSTANT_SECONDS + NANO_OF_SECOND

DateTimeFormatter.format(TemporalAccessor) 的逻辑是：对每个模式字母，调用 temporal.get(ChronoField.XXX) 取出对应字段值，拼接成字符串。 如果时间对象缺少某个字段（比如用 LocalDate 格式化含 HH 的模式），就会抛出 UnsupportedTemporalTypeException。

常见异常：UnsupportedTemporalTypeException
用 LocalDate 调用含 HH:mm:ss 的格式器，或用 LocalDateTime 格式化含 z（时区）的模式，都会抛此异常。根本原因是对象不携带那个字段，格式器无法取到值。解决：换对象类型，或用 atZone() 补充时区信息。

常见陷阱汇总

陷阱 1：每次调用都 new DateTimeFormatter

Java — 反例（每次 new，浪费资源）
// 反例：每次格式化都创建新的 formatter，性能差
public String format(LocalDateTime dt) {
    return dt.format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
}
Java — 正例（static final 共享实例）
// 正例：线程安全，一次创建，全局复用
private static final DateTimeFormatter FMT =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");

public String format(LocalDateTime dt) {
    return dt.format(FMT);
}

陷阱 2：解析异常不做处理

Java
// parse 失败时抛 DateTimeParseException（RuntimeException 子类）
// 对外部输入应做防御性处理
public Optional<LocalDate> safeParse(String input) {
    try {
        return Optional.of(LocalDate.parse(input, FMT));
    } catch (DateTimeParseException e) {
        return Optional.empty();
    }
}

陷阱 3：月份字母大小写混淆

MM 是月份数字（01–12），mm 是分钟（00–59）。写成 "yyyy-mm-dd" 会把月份位置格式化成分钟值——编译不报错，运行结果很诡异，是最难排查的低级错误之一。

陷阱 4：forPattern 与特定地区的月份名

Java
// "Jun" 是英文月份缩写，解析时必须指定 Locale.ENGLISH
// 否则在非英文系统上可能抛 DateTimeParseException
DateTimeFormatter fmt = DateTimeFormatter
    .ofPattern("dd-MMM-yyyy", Locale.ENGLISH);  // ← 指定语言环境
LocalDate d = LocalDate.parse("29-Jun-2025", fmt);

与 Spring / Jackson 的集成

在 Spring Boot 项目中，LocalDateTime 的序列化/反序列化是高频话题。Jackson 默认会把 LocalDateTime 序列化为数组 [2025,6,29,14,30,0]，需要额外配置：

application.yml（Spring Boot 推荐配置）
spring:
  jackson:
    date-format: yyyy-MM-dd HH:mm:ss
    time-zone: Asia/Shanghai
    serialization:
      write-dates-as-timestamps: false
    deserialization:
      fail-on-unknown-properties: false
Java — 字段级精细控制
public class OrderDTO {
    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss", timezone = "Asia/Shanghai")
    private LocalDateTime createTime;

    @JsonFormat(pattern = "yyyy-MM-dd")
    private LocalDate deliveryDate;
}
MyBatis Plus 从 3.x 起内置了 JSR-310 类型处理器，LocalDateTime 字段与数据库 DATETIME 列可以直接映射，无需手写 TypeHandler。但要确保驱动版本支持（MySQL Connector/J 8.x 起完整支持）。

面试高频考点

  1. DateTimeFormatter 与 SimpleDateFormat 的核心区别？
    DTF 不可变、线程安全，可作为 static final 常量共享；SDF 有可变状态，多线程共享会出现数据错乱，必须每次 new 或用 ThreadLocal 包装。
  2. YYYY 和 yyyy 有什么区别？
    YYYY 是 ISO 周历年，跨年周（如 12 月底的某些天属于下一年的第 1 周）会输出下一年年号；yyyy 是日历年，日常使用应始终用 yyyy。
  3. 如何处理不确定格式的日期字符串输入？
    方案一：DateTimeFormatterBuilder + optionalStart/End；方案二：parseBest 指定候选类型列表；方案三：用多个 formatter 依次 try-catch。
  4. LocalDate 格式化含时间字段的模式会发生什么？
    抛出 UnsupportedTemporalTypeException，因为 LocalDate 不携带时间字段，formatter 无法从 TemporalAccessor 中读取对应值。
  5. 如何在旧代码（java.util.Date）和新 API 之间互转？
    通过 Instant 作为桥梁：Date → Instant → ZonedDateTime/LocalDateTime；反向通过 .atZone().toInstant() 再 Date.from()。

与其他知识点的联系

  • 并发：SimpleDateFormat 内部靠可变的 Calendar 字段保存中间状态，多线程共享一个实例会相互踩踏；DateTimeFormatter 不可变、无副作用，天然线程安全，是"用不可变对象规避锁"这一设计思路的典型案例，与 String、LocalDate 本身的不可变设计一脉相承。
  • 不可变对象设计：DTF 一旦构建完成，所有状态只读，可以放心作为 static final 全局共享，这与 Integer/String 的不可变哲学相同——用"创建后不可变"换取"无需同步"的自由。
  • Locale / 国际化：ofLocalizedDate、withLocale 依赖 Locale 体系决定月份名称、日期顺序等本地化展示规则，这是 Java 国际化（i18n）机制在日期领域的具体应用。
  • 字符串处理与正则：模式字符串本质是一套自定义的"迷你 DSL"，解析逻辑与正则表达式的思想相似——都是用声明式模式去匹配/生成文本，区别在于 DTF 的模式字母有特定语义而非通配符。
  • Spring / Jackson 集成：Web 项目里 LocalDateTime 字段的 JSON 序列化/反序列化必须依赖 DateTimeFormatter 的模式串（通过 @JsonFormat 或全局配置），这是本文最后一节展示的实际落地场景，直接决定接口返回的时间格式是否符合前端预期。
  • 历史 API 的包袱：SimpleDateFormat/Date/Calendar 是 JDK 1.0 时代的设计遗留，可变、非线程安全、月份从 0 开始等反直觉设计，是 Java 8 推出 java.time 整套新体系的直接动因，理解这段历史有助于看懂遗留代码中大量的兼容性转换代码。
