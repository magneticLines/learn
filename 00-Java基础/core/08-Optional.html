08 · Optional

Optional<T> 是 Java 8 引入的容器类，核心目标只有一个： 用类型系统表达"值可能不存在"这一语义，从而消灭 NullPointerException。 它不是银弹，但用对了会让代码的意图更清晰、流式管道更顺畅。

1. 为什么需要 Optional

Tony Hoare 在 1965 年设计 ALGOL W 时引入了 null 引用。 他后来在 2009 年公开道歉，称其为"十亿美元错误"——因为 null 让任何引用类型都隐式可空， 而编译器对此无能为力。

Java 的类型系统中，String name 既可以指向一个真实的字符串， 也可以是 null，编译器看不出区别。 调用者必须靠文档或"经验"判断某个返回值是否可能为 null， 一旦遗漏就是运行时 NullPointerException。

反例 — 防御性 null 检查泛滥

// 每一层都要手动判空，极易漏掉
User user = userRepo.findById(id);
if (user != null) {
    Address addr = user.getAddress();
    if (addr != null) {
        return addr.getCity();
    }
}
return "Unknown";

正例 — Optional 链式管道

// 语义清晰：findById 明确可能返回空
return userRepo.findById(id)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
本质理解

Optional 是一个最多包含一个元素的容器。 它的价值不在于避免你写判空代码，而在于把"可能为空"这件事提升到类型层面， 强迫调用者在编译期就思考空值处理。

2. 类结构与三种工厂方法

Optional<T> 是 final 类，不可继承，内部只有一个 private final T value 字段。 创建实例必须通过静态工厂方法，不提供 new。

工厂方法传入 null 时典型使用场景
Optional.of(T value) 立即抛 NullPointerException 已知值非 null，用 of 能在数据入口快速暴露问题
Optional.ofNullable(T value) 返回 Optional.empty() 对接遗留 API / 外部数据，值是否为 null 不确定时
Optional.empty() — 明确表示"无值"，如方法返回空 Optional 
// of：value 必须非 null，否则 NPE（在这里比在调用链深处更好定位）
Optional<String> a = Optional.of("hello");

// ofNullable：安全包装，null 变 empty
Optional<String> b = Optional.ofNullable(getFromCache());

// empty：语义明确的"无值"
public Optional<User> findByEmail(String email) {
    if (!isValidEmail(email)) return Optional.empty();
    return db.query(email);
}
陷阱：of vs ofNullable

用 ofNullable 包装"永远非 null"的值， 会掩盖上游的 bug——空值会被悄悄吞掉。 如果你能确保值非 null，优先用 of 让程序在错误源头就崩溃。

3. 取值方法全景

方法有值时空时推荐度
get() 返回值 抛 NoSuchElementException 谨慎
orElse(T other) 返回值 返回 other 推荐
orElseGet(Supplier) 返回值 执行 Supplier 并返回结果 推荐
orElseThrow(Supplier) 返回值 抛出指定异常 推荐
ifPresent(Consumer) 执行 Consumer 什么都不做 推荐
isPresent() 返回 true 返回 false 有更好替代
isEmpty() Java 11 返回 false 返回 true 推荐

orElse vs orElseGet — 求值时机不同

高频陷阱：orElse 的默认值总是被提前求值

orElse(defaultValue) 无论 Optional 是否有值， defaultValue 表达式始终会被执行。 如果默认值的构建代价高（如数据库查询），或有副作用，必须换用 orElseGet。 

// orElse：createDefault() 无论有无值都会执行！
User u1 = opt.orElse(createDefault());   // createDefault 总被调用

// orElseGet：Lambda 只在 opt 为空时才执行（惰性求值）
User u2 = opt.orElseGet(() -> createDefault()); // 按需调用 ✓

// orElseThrow：空时抛业务异常，适合服务层
User u3 = opt.orElseThrow(() ->
    new UserNotFoundException("用户不存在"));

4. 链式操作：map / flatMap / filter

这三个方法是 Optional 的灵魂，让空值在整个管道中"透明传播"， 无需在每一步手动判断。

map — 值变换

有值时对值应用函数并包装结果；空时直接返回 empty()。 返回类型是 Optional<R>，所以可以链式调用。 

// map：String -> Integer，空值自动穿透
Optional<Integer> len = Optional.ofNullable(name)
    .map(String::trim)
    .map(String::length);

flatMap — 函数本身返回 Optional

如果映射函数已经返回 Optional，直接用 map 会产生 Optional<Optional<T>> 嵌套。 flatMap 会自动"压平"一层。 

// getAddress() 返回 Optional<Address>
// map 会得到 Optional<Optional<Address>> —— 错误
// flatMap 自动展开，得到 Optional<Address> —— 正确
Optional<String> city = userOpt
    .flatMap(User::getAddress)   // getAddress 返回 Optional<Address>
    .map(Address::getCity);       // getCity 返回 String

filter — 条件过滤

// 值存在且满足条件才保留，否则变 empty
Optional<String> result = Optional.ofNullable(email)
    .filter(s -> s.contains("@"))   // 不含 @ 则变 empty
    .map(String::toLowerCase);

反例 — 嵌套 null 检查

if (user != null) {
  Address a = user.getAddress();
  if (a != null) {
    String c = a.getCity();
    if (c != null && !c.isEmpty()) {
      return c.toUpperCase();
    }
  }
}
return "N/A";

正例 — Optional 管道

return Optional.ofNullable(user)
  .map(User::getAddress)
  .map(Address::getCity)
  .filter(c -> !c.isEmpty())
  .map(String::toUpperCase)
  .orElse("N/A");

5. Java 9+ 新增方法 Java 9/11

方法版本作用
or(Supplier<Optional<T>>) Java 9 有值返回自身；空时用 Supplier 提供备用 Optional（flatMap 的"or"版）
ifPresentOrElse(Consumer, Runnable) Java 9 有值时执行 Consumer，空时执行 Runnable，消除 if/else 配对
stream() Java 9 有值返回单元素 Stream；空返回空 Stream，可接入 Stream 管道
isEmpty() Java 11 !isPresent() 的语义糖，可读性更强 
// or：优先级备用数据源
Optional<Config> cfg = fromDatabase()
    .or(() -> fromFile())
    .or(() -> fromDefault());

// ifPresentOrElse：有值打印，无值警告
opt.ifPresentOrElse(
    v -> log.info("Found: {}", v),
    () -> log.warn("Not found")
);

// stream()：将 Optional 流式化，与 flatMap(Optional::stream) 等价
List<String> cities = users.stream()
    .map(User::getOptionalCity)   // Stream<Optional<String>>
    .flatMap(Optional::stream)     // Stream<String>，自动过滤空
    .collect(Collectors.toList());

6. 最佳实践

适合用 Optional 的场景

  • 方法返回值：当方法可能没有结果时，返回 Optional<T> 而不是 null。 这是 Optional 设计的主要目的。
  • Stream 中间过滤：配合 flatMap(Optional::stream) 过滤掉空值。
  • 链式数据提取：多级属性访问，用 map/flatMap 管道。

不适合用 Optional 的场景

不要把 Optional 用于以下场景
  • 方法参数：应让调用者决定传什么，不要用 Optional 参数（API 设计丑陋，应重载方法）
  • 类字段：Optional 不可序列化，放在实体 / DTO 字段上会破坏 JSON、JPA 等框架
  • 集合元素：集合本身已能表达"无元素"，List<Optional<T>> 是过度包装
  • 基本类型包装：用 OptionalInt / OptionalLong / OptionalDouble 避免自动拆装箱开销

7. 常见陷阱

陷阱 1：isPresent + get 反模式

反例 — 回退到 null 风格

if (opt.isPresent()) {
    return opt.get();
}
return "default";

正例 — 用 orElse

return opt.orElse("default");

陷阱 2：orElse 提前求值（已在第 3 节说明）

默认值构建有副作用或成本高时，必须换用 orElseGet(Supplier)。

陷阱 3：Optional 不可序列化

Optional 没有实现 Serializable，不要放在需要序列化的对象（JPA 实体、 Kryo/Java 序列化对象、默认 Jackson POJO）上会报错或产生不预期结果。 此外，Optional 作为字段会带来 16 字节额外对象开销，批量处理时影响 GC 压力。

陷阱 4：嵌套 Optional

Optional<Optional<T>> 几乎总是设计错误的信号。 用 flatMap 展平，或重新设计 API。

陷阱 5：直接调用 get() 而不检查

// 危险：空时抛 NoSuchElementException
String val = opt.get(); // ← 与 NPE 换汤不换药

// 正确：总是提供空值处理策略
String val = opt.orElseThrow(() -> new BusinessException("X"));

8. 与 Stream 的关系

Optional 和 Stream 共享同一个容器模型（Monad 的工程实现）： 都是对值的包装，都提供 map / flatMap / filter，都支持函数式管道。

维度Optional<T>Stream<T>
元素数量0 或 10 到 N
核心用途表达可能缺失的单个结果处理元素集合
终结操作orElse / get / ifPresentcollect / forEach / reduce
互转opt.stream() → Streamstream.findFirst() → Optional

打通点：Optional.stream()（Java 9） 是两个世界的桥梁： 

// 场景：从 ID 列表批量查用户，过滤掉查不到的
List<User> users = ids.stream()
    .map(userRepo::findById)     // Stream<Optional<User>>
    .flatMap(Optional::stream)   // Stream<User>，空 Optional 被过滤
    .collect(Collectors.toList());

// Java 8 等价写法（更冗长）
List<User> users = ids.stream()
    .map(userRepo::findById)
    .filter(Optional::isPresent)
    .map(Optional::get)
    .collect(Collectors.toList());

9. 实战综合示例

场景：根据用户 ID 查询用户 → 获取地址 → 获取城市 → 获取区号。 各级数据均可能不存在。 

// 领域对象（简化）
class AreaCode {
    private final String code;
    public AreaCode(String code) { this.code = code; }
    public String getCode() { return code; }
}
class City {
    public Optional<AreaCode> getAreaCode() { ... }
}
class Address {
    public Optional<City> getCity() { ... }
}
class User {
    public Optional<Address> getAddress() { ... }
}

// 服务层：链式 flatMap 处理多级 Optional
public String getAreaCode(long userId) {
    return userRepo.findById(userId)     // Optional<User>
        .flatMap(User::getAddress)        // Optional<Address>
        .flatMap(Address::getCity)        // Optional<City>
        .flatMap(City::getAreaCode)       // Optional<AreaCode>
        .map(AreaCode::getCode)           // Optional<String>
        .orElse("000");                   // 任意一层为空都返回默认值
}

// 对比：如果各级返回 null 而非 Optional，需要这么多判空 ↓
public String getAreaCodeLegacy(long userId) {
    User user = userRepo.findById(userId);
    if (user == null) return "000";
    Address addr = user.getAddress();
    if (addr == null) return "000";
    City city = addr.getCity();
    if (city == null) return "000";
    AreaCode ac = city.getAreaCode();
    if (ac == null) return "000";
    return ac.getCode();
}

10. 面试考点速览

  • Q1. Optional 是为了完全消灭 null 吗？
    不是。Optional 的定位是方法返回值的语义增强——让 API 的可空性在类型层面可见。 Java 语言本身仍然支持 null，Optional 无法阻止你在代码中使用 null， 但它提供了一种"显式声明可能无值"的惯用法。
  • Q2. orElse 和 orElseGet 的区别？
    orElse(T) 的参数是一个已求值的表达式，无论 Optional 是否有值都会执行。 orElseGet(Supplier) 是惰性求值，只在 Optional 为空时才调用 Supplier。 如果默认值构建有开销或副作用，必须用 orElseGet。
  • Q3. map 和 flatMap 的区别？
    map 的函数返回普通值 T，Optional 自动包装为 Optional<T>。 flatMap 的函数本身返回 Optional<T>，不再做第二次包装（展平）。 如果映射函数已经返回 Optional，用 map 会产生 Optional<Optional<T>>， 必须用 flatMap。
  • Q4. 为什么不推荐把 Optional 用作方法参数？
    Optional 参数会污染调用方——调用者要额外包一层 Optional.of(x)， 且语义不清晰（是"允许不传"还是"我不知道有没有值"？）。 正确做法是方法重载或使用 @Nullable 注解（配合静态分析）。
  • Q5. Optional 为什么不能放在字段上？
    Optional 未实现 Serializable，放在需要序列化的对象（JPA 实体、 Kryo/Java 序列化对象、默认 Jackson POJO）上会报错或产生不预期结果。 此外，Optional 作为字段会带来 16 字节额外对象开销，批量处理时影响 GC 压力。
  • Q6. Optional.of(null) 和 Optional.ofNullable(null) 分别发生什么？
    Optional.of(null) 立即抛 NullPointerException（故意的，在入口快速失败）。 Optional.ofNullable(null) 返回 Optional.empty()，不抛异常。 两者都不存储 null 值——Optional 要么有非 null 值，要么为 empty。

与其他知识点的联系

  • Stream API：Optional 与 Stream 共享同一套容器模型（最多 0/1 个元素 vs 0 到 N 个元素），都提供 map/flatMap/filter，本质是同一种函数式管道思想在"单值"和"多值"两个场景下的具体实现；Optional.stream() 是两者之间的官方桥梁。
  • 集合框架：Map.get(key) 返回 null 是集合 API 遗留的历史设计，而 Map.getOrDefault()、Optional.ofNullable(map.get(key)) 都是在弥补这个设计缺陷；理解集合"查不到就返回 null"的老约定，才能理解 Optional 解决的到底是什么问题。
  • Lambda 与函数式接口：Optional 的 map(Function)、filter(Predicate)、orElseGet(Supplier) 全部依赖函数式接口，是学习 Lambda 表达式之后最容易上手的实战应用场景之一。
  • 泛型：Optional<T> 本身就是一个泛型容器类，它的类型安全（编译期就能看出"这个方法可能没有结果"）依赖泛型系统，这与包装类、集合框架依赖泛型的原理完全一致。
  • 序列化与 JPA/Jackson：Optional 未实现 Serializable，不能直接放在需要序列化的实体字段上，这是学习 JPA 实体设计、Jackson JSON 序列化时必须提前了解的坑，也解释了为什么框架文档普遍建议"只在方法返回值用 Optional"。
  • 异常处理：orElseThrow(Supplier) 把"空值"转换成受检的业务异常，是 Optional 与异常体系结合最紧密的地方，也是 Service 层"查不到就抛业务异常"这一常见模式的标准写法。
