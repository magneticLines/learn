12. 自定义异常

从"描述错误"到"传递语义"——理解为何要自定义异常、如何正确设计异常层次、以及工程中踩过的那些坑。

为什么需要自定义异常？

JDK 内置了数十个异常类，为什么还要自己定义？根本原因在于：异常的核心价值不是"终止程序"，而是"传递语义"。

考虑一个用户注册场景。如果用户名已存在，你可以抛出：

  • RuntimeException("username already exists") — 调用方只能 catch 宽泛的 RuntimeException，无法精准响应
  • IllegalArgumentException("username already exists") — 稍好，但语义模糊，是参数格式错？还是业务冲突？
  • UsernameAlreadyExistsException — 精准，调用方可以针对这一业务场景单独处理（例如提示"换个用户名"而非"系统错误"）

自定义异常的三大价值：

  1. 语义精准：异常类名本身就是文档，不需要读 message 字符串才知道发生了什么
  2. 可区分处理：多个 catch 块可以针对不同异常类型执行不同逻辑
  3. 携带上下文：可在异常类中添加业务字段（如错误码、资源ID），而不仅是一条字符串

异常体系回顾：先搞清楚继承的起点

自定义异常的第一步是选择正确的父类。Java 异常体系：

Throwable ├── Error ← JVM 级别错误（OOM、StackOverflow），通常不捕获 └── Exception ← 程序级异常，需要处理 ├── RuntimeException ← 非受检异常（unchecked） │ ├── NullPointerException │ ├── IllegalArgumentException │ └── ... 你的自定义 RuntimeException 子类 └── IOException / SQLException / ... ← 受检异常（checked） └── ... 你的自定义 Exception 子类

选择继承 Exception 还是 RuntimeException，是设计自定义异常时最重要的决策。

Checked vs Unchecked：这是最关键的设计决策

维度 受检异常（Checked）
继承 Exception		 非受检异常（Unchecked）
继承 RuntimeException
编译器强制 必须 try-catch 或 throws 声明 可以不处理，编译器不报错
适用场景 调用方有能力、有必要处理的错误（如文件不存在、网络超时） 程序 bug 或不可恢复的状态（如非法参数、业务规则违反）
API 污染风险 每一层都要 throws，接口污染严重 无需层层声明
Spring/现代框架 框架层面大量转换为 Unchecked 业界主流倾向（Spring DataAccessException 就是 Unchecked）
经验法则：现代业务开发中，自定义业务异常优先继承 RuntimeException。只有当你能明确告知调用方"这个错误你必须在编译期处理"时，才考虑 Checked。

最小可用的自定义异常：4 个构造方法

继承 Exception 或 RuntimeException 后，强烈建议提供这 4 个构造方法，原因是框架（如 Spring、序列化机制）需要无参构造，而异常链需要接收 Throwable cause 的构造。

BusinessException.java最小模板
public class BusinessException extends RuntimeException {

    // ① 无参构造（反序列化、框架创建实例时需要）
    public BusinessException() {
        super();
    }

    // ② 只带 message（最常用）
    public BusinessException(String message) {
        super(message);
    }

    // ③ message + cause（异常链，保留原始异常，勿丢失！）
    public BusinessException(String message, Throwable cause) {
        super(message, cause);
    }

    // ④ 只带 cause（包装底层异常时使用）
    public BusinessException(Throwable cause) {
        super(cause);
    }
}
陷阱：丢失异常链（Exception Swallowing）
只接受 message 的构造方法，在包装底层异常时如果忘传 cause，堆栈信息就永久丢失了。日志里只有一条业务 message，完全不知道底层为何失败——这是线上排查的噩梦。

带业务字段的自定义异常：携带上下文

纯 message 字符串有时不够用。实战中往往需要：错误码（给前端/客户端用）、资源标识（哪条记录出了问题）。把这些信息作为字段放进异常类，比塞进 message 字符串要规范得多。

BusinessException.java带错误码 + 资源ID
public class BusinessException extends RuntimeException {

    private final String  errorCode;   // 给前端的错误码，如 "USER_NOT_FOUND"
    private final Object  resourceId;  // 出问题的资源标识

    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode  = errorCode;
        this.resourceId = null;
    }

    public BusinessException(String errorCode, String message, Object resourceId) {
        super(message);
        this.errorCode  = errorCode;
        this.resourceId = resourceId;
    }

    public BusinessException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode  = errorCode;
        this.resourceId = null;
    }

    public String  getErrorCode()  { return errorCode; }
    public Object  getResourceId() { return resourceId; }
}

构建异常层次：基类 + 子类

实战中不要只有一个异常类。合理的做法是：定义一个项目级基类，再按业务域细分子类。这样既能在全局异常处理器统一 catch 基类，又能在局部 catch 具体子类做差异化处理。

异常层次设计示例
// ── 第一层：项目基类（所有业务异常的父类）──
public class AppException extends RuntimeException {
    private final int httpStatus;
    private final String errorCode;

    public AppException(int httpStatus, String errorCode, String message) {
        super(message);
        this.httpStatus = httpStatus;
        this.errorCode  = errorCode;
    }
    // getters 省略
}

// ── 第二层：按业务域细分 ──
public class NotFoundException extends AppException {
    public NotFoundException(String resource, Object id) {
        super(404, "NOT_FOUND",
            resource + " not found: id=" + id);
    }
}

public class ForbiddenException extends AppException {
    public ForbiddenException(String action) {
        super(403, "FORBIDDEN",
            "No permission to: " + action);
    }
}

public class ValidationException extends AppException {
    public ValidationException(String field, String reason) {
        super(400, "VALIDATION_FAILED",
            "Field '" + field + "': " + reason);
    }
}

在 Spring Boot 的全局异常处理器中统一拦截：

GlobalExceptionHandler.javaSpring Boot 示例
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AppException.class)
    public ResponseEntity<ErrorResponse> handleAppException(AppException ex) {
        // catch 基类，统一转 HTTP 响应
        return ResponseEntity
            .status(ex.getHttpStatus())
            .body(new ErrorResponse(ex.getErrorCode(), ex.getMessage()));
    }
}

反例 → 问题 → 正例：异常链不要断

这是实战中最常见的错误之一。底层抛出了 SQLException，Service 层把它包装成业务异常，但忘了传 cause，导致原始堆栈永久丢失。

反例 ❌异常链断裂
try {
    userDao.save(user);
} catch (SQLException e) {
    // 丢失了 e！日志里只有 "保存用户失败"，完全不知道 SQL 为何报错
    throw new BusinessException("保存用户失败");
}
正例 ✅保留异常链
try {
    userDao.save(user);
} catch (SQLException e) {
    // 传入 cause，完整堆栈链路保留
    throw new BusinessException("SAVE_FAILED", "保存用户失败", e);
}

打印堆栈时，Caused by: 段就来自 cause。没有它，你在生产环境只能靠猜。

用枚举统一管理错误码

随着项目增大，错误码散落各处很难维护。一个常见做法是用枚举集中定义所有错误码，再由枚举直接生成异常实例。

ErrorCode.java + 用法
public enum ErrorCode {

    USER_NOT_FOUND    (404, "用户不存在"),
    ORDER_CANCELLED   (400, "订单已取消，不可操作"),
    STOCK_INSUFFICIENT(400, "库存不足");

    private final int    httpStatus;
    private final String defaultMessage;

    ErrorCode(int httpStatus, String defaultMessage) {
        this.httpStatus      = httpStatus;
        this.defaultMessage  = defaultMessage;
    }

    // 工厂方法：由枚举直接生成异常，调用方超简洁
    public AppException toException() {
        return new AppException(httpStatus, this.name(), defaultMessage);
    }

    public AppException toException(String extraInfo) {
        return new AppException(httpStatus, this.name(),
            defaultMessage + "（" + extraInfo + "）");
    }
}

// 使用时极其简洁，错误码集中可查
throw ErrorCode.USER_NOT_FOUND.toException("id=" + userId);

性能注意：suppressed 与 fillInStackTrace

异常对象创建时最昂贵的操作是 fillInStackTrace()——它要遍历当前线程的整个调用栈并保存快照。如果你的异常用于流程控制（比如用异常代替返回值，高频路径），这会成为性能瓶颈。

高频路径优化：跳过堆栈填充
public class FastBusinessException extends RuntimeException {

    public FastBusinessException(String message) {
        // 最后两个参数：enableSuppression=false, writableStackTrace=false
        // 跳过 fillInStackTrace()，创建成本接近普通对象
        super(message, null, false, false);
    }
}
副作用：关闭 writableStackTrace 后，printStackTrace() 不会打印调用栈，排查问题时信息缺失。只在确实有性能压力的高频路径使用，普通业务异常不要这样做。

与 JDK / Spring 生态的联系

体系做法可借鉴的设计
JDK IOException 体系，Checked 异常 子类细化：FileNotFoundException、SocketTimeoutException，都是 IOException 的子类
Spring DataAccessException，Unchecked 将各种数据库（JDBC/Hibernate/MyBatis）的 Checked 异常统一包装为一个 Unchecked 层次，屏蔽实现细节
Spring Security AuthenticationException / AccessDeniedException 框架 catch 基类，自动映射 HTTP 401/403，业务代码只管抛子类
MyBatis PersistenceException（Unchecked） 包装原始 SQLException，保留 cause 链

常见陷阱汇总

  • 吞掉 cause：catch 后 new 异常不传 cause，原始堆栈永久丢失（最高频陷阱）
  • 用异常做流程控制：if/else 能解决的问题不要用 throw/catch，异常是"异常情况"，不是正常分支
  • Checked 异常层层 throws：每一层接口都要加 throws，接口污染严重，难以变更
  • catch (Exception e) 大网捞鱼：把所有异常都包成业务异常，掩盖了 NullPointerException、OutOfMemoryError 等真正的 bug
  • 异常 message 硬编码中文：国际化场景应使用 errorCode，由前端根据 code 取本地化文案
  • 在 finally / @PreDestroy 中 throw：会掩盖原始异常，破坏异常链

面试高频考点

Q1：Checked 和 Unchecked 异常的区别是什么？什么时候用哪种？
Checked 继承 Exception，编译器强制处理；Unchecked 继承 RuntimeException，无需声明。业务异常通常用 Unchecked，因为调用方往往无法在调用点恢复，且 Checked 会污染接口签名。
Q2：自定义异常为什么要提供 4 个构造方法？
无参构造供反序列化和框架使用；带 cause 的构造供包装底层异常用，防止异常链断裂。缺少任何一个都可能在某些框架场景下出问题。
Q3：异常链（Exception Chaining）是什么？有什么用？
通过 super(message, cause) 将原始异常作为 cause 保存在新异常中。Throwable.getCause() 可以取出。日志打印时显示为 Caused by:，保留完整根因，是排查线上问题的关键。
Q4：为什么不建议 catch (Exception e) 后 throw new MyException(e.getMessage())？
e.getMessage() 只取了 message 字符串，丢失了原始 cause 的类型信息和堆栈，等同于吞掉了异常链。正确做法是 throw new MyException(msg, e)。
