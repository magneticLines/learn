03. Collections 工具类

标准库里最容易被低估的一个工具箱——一行代码背后藏着排序、二分查找、不可变封装、线程安全包装等经典算法与设计思想。

为什么需要 Collections 工具类？

Java 集合框架（JCF）的设计哲学是"接口与实现分离"：List、Set、Map 定义行为， ArrayList、HashMap 提供实现。但有一类操作天然"跨实现"—— 排序、搜索、洗牌、填充……它们的逻辑与底层数据结构无关，强行塞进接口会污染 API， 塞进每个实现类又会造成代码重复。

解决方案：静态工具类。 java.util.Collections（注意有 s） 扮演"算法仓库"的角色，所有方法都是 static，接收集合接口类型， 对任意实现透明可用。这与 Arrays（处理数组）、Objects（处理对象） 是同一设计风格，也是 JDK 中"工具类模式"的典型范例。

注意区分：Collection（无 s）是集合根接口； Collections（有 s）是工具类。两者经常被初学者混淆。

排序与二分查找

sort() — 底层用 TimSort

Collections.sort(list) 要求元素实现 Comparable（自然顺序）； sort(list, comparator) 接受外部比较器，不要求元素可比较。 底层委托给 list.sort(comparator)，JDK 8 后实现为 TimSort—— 归并排序 + 插入排序的混合，对现实中"几乎有序"的数据极友好，时间复杂度 O(n log n)，稳定排序。

Javasort 示例
List<String> names = new ArrayList<>(Arrays.asList("Charlie", "Alice", "Bob"));

// 自然顺序（字典序）
Collections.sort(names);
// → [Alice, Bob, Charlie]

// 逆序：用现成的 Comparator.reverseOrder()
names.sort(Comparator.reverseOrder());
// → [Charlie, Bob, Alice]

// 按字符串长度排序（相同长度保持原相对顺序：稳定排序的意义）
names.sort(Comparator.comparingInt(String::length));
JDK 8 的变化：推荐直接调用 list.sort(comparator)， 内部同样是 TimSort，但省去了一次方法调用。Collections.sort 在 JDK 8 后也已改为委托给 list.sort。

binarySearch() — 前提是已排序

二分查找要求 List 已按相同顺序排好序，否则结果未定义（不会报错，只会返回错误结果，是个隐蔽的 bug）。 返回值：找到时返回索引（≥0）；找不到时返回 -(插入点) - 1，负数。

JavabinarySearch 示例
List<Integer> nums = Arrays.asList(1, 3, 5, 7, 9);

int idx = Collections.binarySearch(nums, 5);  // → 2
int miss = Collections.binarySearch(nums, 4);  // → -3（插入点=2，返回 -(2)-1=-3）

// 利用返回的负数计算"应插入的位置"：
int insertionPoint = -(miss) - 1;  // → 2，即 4 应插在 index 2
陷阱：对 LinkedList 使用 binarySearch 性能退化为 O(n)， 因为二分需要随机访问，LinkedList 不支持 RandomAccess， JDK 会自动降级为顺序扫描。

洗牌、反转、填充

Javashuffle / reverse / fill
List<Integer> cards = new ArrayList<>(Arrays.asList(1,2,3,4,5));

// Fisher-Yates 洗牌算法，O(n)，均匀随机
Collections.shuffle(cards);

// 带种子，测试时可复现随机结果
Collections.shuffle(cards, new Random(42));

// 就地反转，不创建新集合
Collections.reverse(cards);

// 将 List 所有元素替换为同一个值
Collections.fill(cards, 0);  // → [0, 0, 0, 0, 0]

// 创建 n 个相同元素的不可变 List（常用于初始化占位）
List<String> placeholder = Collections.nCopies(3, "N/A");
// → [N/A, N/A, N/A]（不可修改！）
陷阱：nCopies 存储的是同一引用。 如果元素是可变对象（如自定义类），修改其中一个会影响所有"副本"， 因为它们指向同一块内存。

min / max

从任意 Collection（不限于 List）中找极值，使用自然顺序或自定义比较器。 底层是线性扫描 O(n)，比先排序再取首尾元素高效。

Javamin / max
List<Integer> scores = Arrays.asList(88, 72, 95, 61);

int lowest  = Collections.min(scores);  // → 61
int highest = Collections.max(scores);  // → 95

// 字符串按长度取最短
String shortest = Collections.min(
    Arrays.asList("apple", "hi", "banana"),
    Comparator.comparingInt(String::length)
);  // → "hi"

不可变包装（Unmodifiable 视图）

Collections.unmodifiableList(list) 返回原 List 的视图， 所有写操作（add、remove、set） 都会抛出 UnsupportedOperationException，但读操作直接穿透到原集合。 这是"防御式编程"的核心手段：对外暴露只读视图，阻止调用方意外修改内部状态。

Javaunmodifiable 包装
public class UserService {
    private final List<String> admins = new ArrayList<>();

    // 对外只暴露只读视图，调用方无法篡改 admins
    public List<String> getAdmins() {
        return Collections.unmodifiableList(admins);
    }
}

// 调用方尝试修改 → 运行时抛 UnsupportedOperationException
List<String> view = service.getAdmins();
view.add("hacker");  // 抛异常！
关键陷阱：视图，不是副本。 原始集合修改后，unmodifiableList 视图能"看到"变化—— 它只是禁止通过视图修改，没有做任何数据复制。如需真正的不可变快照， 应使用 List.copyOf(list)（JDK 10+）或 new ArrayList<>(list)。

singleton 系列 — 单元素不可变集合

当方法签名要求 Collection 但你只有一个元素时， Collections.singletonList(e) 比 new ArrayList<>() 加 add 更简洁高效， 且不可修改，占用内存极小（只包装一个引用）。

Javasingleton 系列
// 单元素 List / Set / Map
List<String>          one  = Collections.singletonList("admin");
Set<Integer>          oneS = Collections.singleton(42);
Map<String,Integer>  oneM = Collections.singletonMap("key", 1);

// emptyList / emptySet / emptyMap：零元素的不可变集合（全局单例，节省内存）
List<String> empty = Collections.emptyList();
// 等价于 JDK 9+ 的 List.of()

线程安全包装（synchronized 系列）

Collections.synchronizedList(list) 返回一个每个方法都加了 synchronized 锁的包装集合，锁对象是包装类自身（this）。 这让原本线程不安全的 ArrayList、HashMap 变得基本线程安全，代价是所有操作串行化，并发性能差。

JavasynchronizedList 与复合操作陷阱
List<String> safeList =
    Collections.synchronizedList(new ArrayList<>());

// 单个操作是原子的，安全
safeList.add("item");

// ❌ 危险：迭代时必须手动加锁！
// 两个线程同时迭代+修改 → ConcurrentModificationException
for (String s : safeList) { // 不安全！
    System.out.println(s);
}

// ✅ 正确：在 safeList 上显式同步再迭代
synchronized (safeList) {
    for (String s : safeList) {
        System.out.println(s);
    }
}
现代替代方案：生产环境中应优先使用 java.util.concurrent 包： CopyOnWriteArrayList（读多写少）、ConcurrentHashMap（高并发 Map）。 synchronized 包装集合因为粗粒度锁，在高并发下性能远不如 JUC 集合。
包装方式 适用场景 迭代安全？ 并发性能
synchronizedList 低并发、简单场景 手动加锁 差（全局锁）
CopyOnWriteArrayList 读多写极少 天然安全（迭代快照） 读极好，写差
ConcurrentHashMap 通用高并发 Map 弱一致性迭代器 优秀（分段锁）

frequency / disjoint / copy

Javafrequency / disjoint / copy
List<String> words = Arrays.asList("a","b","a","c","a");

// 统计 "a" 出现次数，O(n)
int count = Collections.frequency(words, "a");  // → 3

// 判断两个集合是否没有公共元素
boolean noOverlap = Collections.disjoint(
    Arrays.asList(1,2,3),
    Arrays.asList(4,5,6)
);  // → true

// copy：dest 必须预先有足够空间（size >= src.size()），否则抛 IndexOutOfBoundsException
List<String> src  = Arrays.asList("x","y");
List<String> dest = new ArrayList<>(Arrays.asList("a","b","c"));
Collections.copy(dest, src);  // dest → [x, y, c]（原位覆写，不改变 dest.size()）

与现代 API 的对比

JDK 9+ 引入了更简洁的工厂方法，很多场景可以替代 Collections：

需求 传统写法（Collections） 现代写法（JDK 9+）
不可变空 List Collections.emptyList() List.of()
单元素不可变 List Collections.singletonList(e) List.of(e)
多元素不可变 List unmodifiableList(Arrays.asList(...)) List.of(a, b, c)
不可变 Map Collections.singletonMap(k, v) Map.of(k, v)
关键区别：List.of() 不允许元素为 null（会抛 NullPointerException）， 而 Collections.singletonList(null) 允许。如果集合中可能存 null，请用旧写法。

面试高频考点

  • Collections.sort 的底层算法是什么？ TimSort（归并 + 插入排序的混合），稳定排序，对近似有序数据接近 O(n)。
  • Collections.sort 和 Arrays.sort 的区别？ Arrays.sort 对基本类型用 DualPivot QuickSort（不稳定），对对象数组用 TimSort（稳定）； Collections.sort 内部委托给 list.sort，始终是 TimSort。
  • unmodifiableList 是深拷贝吗？ 不是，是视图。原 List 修改会透过视图可见；调用方只是无法通过视图写。
  • synchronizedList 的迭代为什么不安全？ 每个单独方法是同步的，但 for-each 迭代是多次方法调用的组合，不在同一锁范围内， 需要手动在集合对象上 synchronized 包裹整个迭代块。
  • binarySearch 返回负数时如何解读？ 返回 -(插入点) - 1。加 1 取反可得到"若插入应在哪个位置"， 用于查找最接近的值。
  • 为什么 Collections 的构造器是 private？ 工具类不需要实例，private 构造器防止实例化，这是"工具类模式"的标准做法， JDK 中 Math、Arrays、Objects 同理。
