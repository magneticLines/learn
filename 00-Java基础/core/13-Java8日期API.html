13 — Java 8 日期 API

java.time：不可变、线程安全、时区清晰的现代日期体系

1. 为什么要重新设计？旧 API 的五大历史包袱

Java 1.0 就提供了 java.util.Date，1.1 补充了 Calendar。但这两个类沿用了 30 多年，积累了很多问题：

问题具体表现影响
可变性（Mutable） Date.setTime() 可随时修改内部状态 多线程共享时需要加锁；作为 Map key 不安全
月份从 0 开始 Calendar.MONTH 中 1月=0，12月=11 off-by-one bug 极其常见，新人必踩
职责混乱 Date 既代表"时刻"又格式化日期；SimpleDateFormat 不安全 类之间边界模糊，容易误用
API 繁琐 计算两日期相差天数需要自己做毫秒除法 简单业务需要写大量样板代码
时区含义模糊 Date 内部存 UTC 毫秒，但 toString() 却用本地时区显示 日志/序列化时区混乱，排查问题困难
设计来源：Joda-Time → JSR-310 Java 8 的 java.time 由 Stephen Colebourne（Joda-Time 作者）主导，通过 JSR-310 进入 JDK。 核心设计哲学：不可变对象（Immutable）、职责单一（每个类只表达一种概念）、流式 API（方法链）、时区显式化。

2. java.time 类族全景

java.time
├── LocalDate ── 无时区的日期 (2024-03-15)
├── LocalTime ── 无时区的时间 (10:30:00)
├── LocalDateTime ── 无时区的日期+时间
├── ZonedDateTime ── 带时区的完整时刻
├── OffsetDateTime ── 带偏移量(+08:00)的时刻，无DST规则
├── Instant ── 机器视角时刻 (Unix epoch 纳秒)
├── Period ── 日期间隔 (年/月/日)
├── Duration ── 时间间隔 (秒/纳秒)
├── ZoneId ── 时区 ID ("Asia/Shanghai")
├── ZoneOffset ── 固定偏移 (+08:00)
└── format/
    ├── DateTimeFormatter ── 线程安全的格式化器
    └── DateTimeFormatterBuilder

选哪个类？判断准则

场景推荐类型
只需要日期（生日、节假日）LocalDate
只需要时间（营业时间）LocalTime
本地日期+时间，不关心时区LocalDateTime
跨时区的业务（会议、日志）ZonedDateTime
存入数据库 / REST API 传输OffsetDateTime 或 Instant
计算时间间隔（秒级精度）Duration
计算日期间隔（年月日）Period

3. LocalDate — 只有日期，没有时区

创建

// 当前日期（使用系统默认时区确定"今天"）
LocalDate today = LocalDate.now();

// 指定日期 — 月份直接用数字，1 = 1月，不再从 0 开始！
LocalDate birthday = LocalDate.of(1990, 8, 20);

// 从字符串解析（默认 ISO 格式 yyyy-MM-dd）
LocalDate date = LocalDate.parse("2024-03-15");

// 从字符串 + 自定义格式
LocalDate date2 = LocalDate.parse("20240315",
    DateTimeFormatter.ofPattern("yyyyMMdd"));

读取字段

LocalDate d = LocalDate.of(2024, 3, 15);
d.getYear();        // 2024
d.getMonth();       // Month.MARCH（枚举）
d.getMonthValue();  // 3（int，更常用）
d.getDayOfMonth();  // 15
d.getDayOfWeek();   // DayOfWeek.FRIDAY
d.getDayOfYear();   // 75
d.isLeapYear();     // false

加减与比较

LocalDate d = LocalDate.of(2024, 3, 15);

// ⚠️ 不可变！plus/minus 返回新对象，原对象不变
LocalDate nextWeek  = d.plusWeeks(1);         // 2024-03-22
LocalDate lastMonth = d.minusMonths(1);       // 2024-02-15
LocalDate nextYear  = d.plus(1, ChronoUnit.YEARS);

// 比较
d.isBefore(nextWeek);   // true
d.isAfter(lastMonth);   // true
d.isEqual(LocalDate.of(2024,3,15));  // true

// ChronoUnit 计算任意单位差值
long days = ChronoUnit.DAYS.between(d, nextWeek);  // 7
long months = ChronoUnit.MONTHS.between(
    LocalDate.of(2024,1,1), LocalDate.of(2024,3,15)); // 2
不可变设计的好处 LocalDate 所有方法都返回新对象，自身永不改变。这意味着：线程安全（无需 synchronized）、可作为 Map key（hashCode 稳定）、可作为方法参数传递而不担心被修改。代价是轻微的对象分配开销，在现代 JVM 上可忽略不计。

4. LocalDateTime — 日期 + 时间，仍无时区

创建方式

// 当前时刻（无时区概念）
LocalDateTime now = LocalDateTime.now();

// 直接指定
LocalDateTime dt = LocalDateTime.of(2024, 3, 15, 10, 30, 0);

// 拼接：LocalDate + LocalTime → LocalDateTime
LocalDate date = LocalDate.of(2024, 3, 15);
LocalTime time = LocalTime.of(10, 30);
LocalDateTime dt2 = date.atTime(time);
// 等价于：date.atTime(10, 30)

// 解析
LocalDateTime dt3 = LocalDateTime.parse("2024-03-15T10:30:00");

拆分与修改

LocalDateTime dt = LocalDateTime.of(2024, 3, 15, 10, 30, 45);

// 拆分
LocalDate datePart = dt.toLocalDate();  // 2024-03-15
LocalTime timePart = dt.toLocalTime();  // 10:30:45

// with 系列 — 修改某个字段（返回新对象）
dt.withYear(2025);         // 2025-03-15T10:30:45
dt.withMonth(6);           // 2024-06-15T10:30:45
dt.withDayOfMonth(1);      // 2024-03-01T10:30:45
dt.withHour(0).withMinute(0).withSecond(0); // 当天零点

升级为 ZonedDateTime

ZoneId shanghai = ZoneId.of("Asia/Shanghai");
ZonedDateTime zdt = dt.atZone(shanghai);
// 2024-03-15T10:30:45+08:00[Asia/Shanghai]

// 注意：atZone 假设"这个 LocalDateTime 就是在 Shanghai 时区发生的"
// 它不做任何时间转换，只是附加时区信息

5. ZonedDateTime — 带时区的完整时刻

创建

ZoneId tokyo   = ZoneId.of("Asia/Tokyo");
ZoneId london  = ZoneId.of("Europe/London");

ZonedDateTime now = ZonedDateTime.now(tokyo);

// 从 LocalDateTime + ZoneId
ZonedDateTime zdt = ZonedDateTime.of(
    LocalDateTime.of(2024, 3, 15, 10, 30, 0),
    ZoneId.of("Asia/Shanghai"));

withZoneSameInstant vs withZoneSameLocal

这是最容易混淆的两个方法，必须搞清楚：

withZoneSameLocal — 保持"表盘时间"不变
// 上海 10:30 → 东京 10:30
// 背后的时刻变了！
zdt.withZoneSameLocal(tokyo)
// 2024-03-15T10:30:00+09:00[Asia/Tokyo]
// 上海 10:30 = UTC 02:30
// 东京 10:30 = UTC 01:30
// 两者差 1 小时
withZoneSameInstant — 保持"绝对时刻"不变
// 上海 10:30 → 东京 11:30
// 背后的时刻不变！
zdt.withZoneSameInstant(tokyo)
// 2024-03-15T11:30:00+09:00[Asia/Tokyo]
// 两者都等于 UTC 02:30
// 这才是"时区转换"
记忆口诀 SameInstant = 同一时刻，表盘不同（跨时区开会用这个）；SameLocal = 表盘相同，时刻不同（迁移数据时区标注用这个）。
绝大多数业务场景要用 withZoneSameInstant，因为你需要的是"真实转换"。

获取所有可用时区

// 打印全部 ZoneId（约 600 个）
ZoneId.getAvailableZoneIds()
      .stream()
      .filter(id -> id.startsWith("Asia"))
      .sorted()
      .forEach(System.out::println);

6. Instant — 机器视角的时刻

Instant 是"距 Unix Epoch (1970-01-01T00:00:00Z) 的秒数 + 纳秒调整量"。它不关心时区，是绝对时刻的最纯粹表示。

// 当前时刻（UTC）
Instant now = Instant.now();
System.out.println(now);  // 2024-03-15T02:30:00.123456789Z

// 转换为毫秒时间戳
long epochMilli = now.toEpochMilli();

// 从毫秒构造
Instant i = Instant.ofEpochMilli(System.currentTimeMillis());

// Instant → ZonedDateTime（需要指定时区才能显示为人类可读）
ZonedDateTime zdt = now.atZone(ZoneId.of("Asia/Shanghai"));

// ZonedDateTime → Instant（丢弃时区信息，取绝对时刻）
Instant back = zdt.toInstant();

// 与旧 Date 互转
Date legacyDate = Date.from(now);
Instant fromDate = legacyDate.toInstant();
什么时候用 Instant？ 记录事件发生时刻（日志时间戳、审计记录）、测量耗时（Instant.now() 做 stopwatch）、存入数据库的 UTC 时间戳。
不适合：需要"年/月/日"等日历概念的场景，这时应先转为 ZonedDateTime。

7. Period vs Duration — 两种间隔

PeriodDuration
适用对象LocalDateLocalTime / LocalDateTime / Instant
单位年、月、日秒、纳秒（也有 hours/minutes 辅助）
构造Period.of(1,2,3) / between(d1,d2)Duration.ofHours(2) / between(t1,t2)
含义日历间隔（考虑月份长短）精确时间长度（固定秒数）
// Period
LocalDate start = LocalDate.of(2023, 1, 1);
LocalDate end   = LocalDate.of(2024, 3, 15);

Period p = Period.between(start, end);
System.out.println(p.getYears());   // 1
System.out.println(p.getMonths());  // 2
System.out.println(p.getDays());    // 14

// Duration
LocalDateTime t1 = LocalDateTime.of(2024,3,15, 8,0,0);
LocalDateTime t2 = LocalDateTime.of(2024,3,15,10,30,0);

Duration d = Duration.between(t1, t2);
System.out.println(d.toHours());    // 2
System.out.println(d.toMinutes());  // 150
System.out.println(d.getSeconds()); // 9000
陷阱：Period.getDays() 不等于总天数！ Period.between(start, end).getDays() 只返回"零散天数"（14），不是两个日期相差的总天数（440）。
如果你要总天数，请用：ChronoUnit.DAYS.between(start, end)（返回 440）。

8. 旧代码互操作 — Date / Timestamp 与新类互转

源类型目标类型转换方式
Date Instant date.toInstant()
Date LocalDateTime date.toInstant().atZone(ZoneId.systemDefault()).toLocalDateTime()
LocalDateTime Date Date.from(ldt.atZone(ZoneId.systemDefault()).toInstant())
Timestamp LocalDateTime timestamp.toLocalDateTime()（直接支持）
LocalDateTime Timestamp Timestamp.valueOf(ldt)
long（毫秒） Instant Instant.ofEpochMilli(millis)
Instant long（毫秒） instant.toEpochMilli()
夏令时陷阱（DST） 当用 ZoneId.systemDefault() 在欧美时区做 Date ↔ LocalDateTime 互转时，夏令时切换那天某些本地时间不存在（时钟拨快）或存在两次（时钟拨回）。 建议：数据库和接口层统一使用 UTC（Instant/OffsetDateTime），只在展示层转为用户时区的 ZonedDateTime。

9. 实战场景

计算年龄

LocalDate birthday = LocalDate.of(1990, 8, 20);
LocalDate today    = LocalDate.now();
int age = Period.between(birthday, today).getYears();
// 直接得到周岁，自动处理是否已过生日

判断 30 天内到期

LocalDate expireDate = LocalDate.of(2024, 4, 14);
LocalDate today      = LocalDate.now();
boolean expiringSoon = !expireDate.isBefore(today)
    && ChronoUnit.DAYS.between(today, expireDate) <= 30;

获取当月首末日

LocalDate today = LocalDate.now();
LocalDate firstDay = today.withDayOfMonth(1);
LocalDate lastDay  = today.withDayOfMonth(today.lengthOfMonth());

// 或用 TemporalAdjusters（语义更清晰）
LocalDate first2 = today.with(TemporalAdjusters.firstDayOfMonth());
LocalDate last2  = today.with(TemporalAdjusters.lastDayOfMonth());

TemporalAdjusters 常用调整器

// 下一个周一
today.with(TemporalAdjusters.next(DayOfWeek.MONDAY));

// 本月第一个周五
today.with(TemporalAdjusters.firstInMonth(DayOfWeek.FRIDAY));

// 下一年的第一天
today.with(TemporalAdjusters.firstDayOfNextYear());

JPA / Hibernate 注解（Spring Boot 2.2+ 自动支持）

@Entity
public class Order {
    // MySQL DATE 列 ↔ LocalDate（无需 @Temporal）
    private LocalDate orderDate;

    // MySQL DATETIME / TIMESTAMP 列 ↔ LocalDateTime
    private LocalDateTime createdAt;

    // 存为 UTC Instant（推荐跨时区系统）
    @Column(columnDefinition = "TIMESTAMP")
    private Instant updatedAt;
}

10. 四个常见陷阱

陷阱 1：忘记接收返回值

错误
LocalDate date = LocalDate.now();
date.plusDays(7);  // 返回值被丢弃！
System.out.println(date); // 还是今天
正确
LocalDate date = LocalDate.now();
LocalDate future = date.plusDays(7); // 接住返回值
System.out.println(future);

陷阱 2：now() 依赖默认时区

// ❌ 在服务器时区 ≠ 用户时区时产生错误结果
LocalDate today = LocalDate.now();

// ✅ 显式传入时区
LocalDate today = LocalDate.now(ZoneId.of("Asia/Shanghai"));

陷阱 3：Period.getDays() 只是零散天

Period p = Period.between(
    LocalDate.of(2023,1,1),
    LocalDate.of(2024,3,15));

p.getDays();   // 14  ← 只是零散天！
// ✅ 总天数
ChronoUnit.DAYS.between(
    LocalDate.of(2023,1,1),
    LocalDate.of(2024,3,15));  // 440

陷阱 4：夏令时切换时加减天数与加减小时不等价

ZonedDateTime zdt = ZonedDateTime.of(
    LocalDateTime.of(2024, 3, 10, 1, 30),  // 美国夏令时切换前
    ZoneId.of("America/New_York"));

// plusDays(1) 保持时钟时间（1:30 AM），自动调整偏移
ZonedDateTime nextDay = zdt.plusDays(1);   // 2024-03-11T01:30-04:00

// plusHours(24) 固定增加 86400 秒，时钟时间变为 2:30
ZonedDateTime plus24h = zdt.plusHours(24); // 2024-03-11T02:30-04:00

11. 知识联系

关联知识点说明
DateTimeFormatter 线程安全（与 SimpleDateFormat 相反），可预定义常量（ISO_DATE）或自定义模式；支持本地化（ofLocalizedDate()）
JPA / Hibernate Spring Boot 2.2+ 通过 hibernate-types 自动映射 LocalDate/LocalDateTime；旧版需 @Temporal(TemporalType.DATE)
Jackson 序列化 需引入 jackson-datatype-jsr310，注册 JavaTimeModule，否则 LocalDate 默认序列化为数组
Stream + 日期分组 Collectors.groupingBy(order -> order.getDate().getMonth()) 按月统计，结合 TreeMap 保持顺序
不可变设计模式 java.time 是 Effective Java Item 17（最小化可变性）的最佳实践范本；同类有 String、BigDecimal

12. 面试高频考点

问题关键答题点
Java 8 为什么重新设计日期 API？ Date 可变、月份 0 开始、SimpleDateFormat 线程不安全、时区混乱；JSR-310 = Joda-Time 进入 JDK
LocalDateTime 和 ZonedDateTime 的区别？ LocalDateTime 无时区概念，不能表示绝对时刻；ZonedDateTime 含 ZoneId，能做时区转换和 DST 处理
Instant 是什么？和 Date 有什么关系？ Unix epoch 纳秒精度时刻；Date.from(instant) / date.toInstant() 互转；两者都表示绝对时刻
Period.getDays() 能获取总天数吗？ 不能，只是零散天部分；总天数用 ChronoUnit.DAYS.between()
withZoneSameInstant 和 withZoneSameLocal 的区别？ SameInstant 保持绝对时刻不变（表盘变）= 真正的时区转换；SameLocal 保持表盘时间不变（时刻变）
为什么 java.time 类是线程安全的？ 所有类都是不可变的（Immutable），所有修改方法返回新对象，不存在共享可变状态
