06. 方法引用

Lambda 的进化形式——当 Lambda 体只是"转发调用"时，方法引用让代码更简洁、意图更清晰。

1. 为什么需要方法引用？

上一节我们学了 Lambda 表达式。但很多时候，Lambda 只是在"包一层皮"——把参数原封不动地转给某个已有方法：

重复包装的 Lambda
list.forEach(s -> System.out.println(s));   // Lambda 体仅一行，且只是转发
list.sort((a, b) -> String.CASE_INSENSITIVE_ORDER.compare(a, b));
list.stream().map(s -> s.toUpperCase()).collect(...);

方法引用（Method Reference）是 Java 8 提供的语法糖，让你直接把方法"作为值"传递，省掉不必要的参数变量名：

改写为方法引用
list.forEach(System.out::println);
list.sort(String.CASE_INSENSITIVE_ORDER::compare);
list.stream().map(String::toUpperCase).collect(...);
设计动机：函数式编程强调"把行为当数据传递"。方法引用让已有方法无需包装即可充当函数式接口的实现，减少噪音、提升可读性。

2. 四种形式一览

形式 语法 等价 Lambda 结构 典型例子
静态方法引用 类名::静态方法 (args) -> 类名.静态方法(args) Integer::parseInt
特定实例方法引用 实例::实例方法 (args) -> 实例.实例方法(args) System.out::println
任意实例方法引用 类名::实例方法 (obj, args) -> obj.实例方法(args) String::toUpperCase
构造器引用 类名::new (args) -> new 类名(args) ArrayList::new
核心规则：方法引用的签名必须与目标函数式接口的抽象方法签名兼容——参数类型、数量和返回值类型要能对上。编译器会在赋值上下文中自动做类型推断。

3. 静态方法引用

最简单的一种。把一个静态方法直接当作 Function、Predicate 等接口的实现来传递。

静态方法引用示例
List<String> strs = List.of("1", "2", "3");

// Function<String, Integer> 的抽象方法签名：Integer apply(String s)
// Integer.parseInt(String s) 签名完全匹配 → 可以引用
List<Integer> nums = strs.stream()
        .map(Integer::parseInt)   // 等价于 s -> Integer.parseInt(s)
        .collect(Collectors.toList());

// Predicate<String>：boolean test(String s)
// 自定义静态谓词
class Validator {
    static boolean isNumeric(String s) {
        return s.matches("\\d+");
    }
}
strs.stream().filter(Validator::isNumeric).forEach(System.out::println);

4. 特定实例方法引用

绑定到一个已知的具体对象上。调用时，那个对象是固定的（被"捕获"的），额外参数才是函数式接口传进来的。

特定实例方法引用示例
// System.out 是一个具体的 PrintStream 实例
// Consumer<String>：void accept(String s)
// System.out.println(String s) 签名匹配
List<String> names = List.of("Alice", "Bob");
names.forEach(System.out::println);

// 更清晰的自定义例子：用已有的 Comparator 实例引用其 compare 方法
Comparator<String> cmp = String.CASE_INSENSITIVE_ORDER;
names.stream().sorted(cmp::compare).forEach(System.out::println);

// 实例来自局部变量，同样可以引用
String prefix = "Hello_";
Function<String, String> concat = prefix::concat;  // s -> "Hello_".concat(s)
System.out.println(concat.apply("World"));  // Hello_World

5. 任意实例方法引用（未绑定）

这是最容易迷惑初学者的一种。语法形式与静态方法引用一样是 类名::方法名，但引用的是实例方法。

关键理解：第一个参数会自动变成调用者（接收者），其余参数才是原始参数。

任意实例方法引用示例
// String::toUpperCase
// Function<String, String> 的签名：String apply(String s)
// 相当于 Lambda：(String s) -> s.toUpperCase()
// 第一个参数 s 变成了 toUpperCase() 的调用者
List<String> words = List.of("hello", "world");
words.stream().map(String::toUpperCase).forEach(System.out::println);

// 有额外参数的情况
// String::startsWith
// BiFunction<String, String, Boolean>：Boolean apply(String s, String prefix)
// 等价 Lambda：(s, prefix) -> s.startsWith(prefix)
BiFunction<String, String, Boolean> fn = String::startsWith;
System.out.println(fn.apply("Java", "Ja"));  // true
常见混淆点：String::valueOf 是静态方法引用，因为 valueOf 是 static 的；String::toUpperCase 是任意实例方法引用，因为 toUpperCase 是实例方法。两者语法相同，区分靠方法本身是否是 static。

6. 构造器引用

用 类名::new 把构造器当函数传递，常见于工厂方法模式和 Stream 收集器场景。

构造器引用示例
// Supplier<T>：T get()  → 无参构造器
Supplier<ArrayList<String>> factory = ArrayList::new;
ArrayList<String> list = factory.get();

// Function<String, Integer>：有参构造器，参数就是构造器参数
// new Integer(String s)  →  等价 Lambda：s -> new Integer(s)
Function<String, Integer> boxed = Integer::new;   // 已过时，仅演示

// Stream.toCollection 使用构造器引用作为集合工厂
Set<String> result = Stream.of("a", "b")
        .collect(Collectors.toCollection(TreeSet::new));

// 数组构造器引用：IntFunction<int[]>
IntFunction<int[]> arrFactory = int[]::new;
int[] arr = arrFactory.apply(5);  // new int[5]

7. 方法引用与 Lambda 的本质关系

方法引用不是什么新的运行时机制，它只是特定 Lambda 的简写形式。编译器在字节码层面对两者的处理是一致的：生成 invokedynamic 指令，运行时由 LambdaMetafactory 动态生成函数式接口的匿名实现类。

方法引用 ──编译器等价变换──> Lambda 表达式
                                              ↓
                                invokedynamic 指令
                                              ↓
                     LambdaMetafactory 动态生成匿名实现类

因此，方法引用没有额外的性能开销，它只是更清晰地表达了"我只是转发调用这个方法"的意图。

8. 与常用函数式接口的对应

理解每种引用对应哪类接口，有助于在实战中快速判断能否使用方法引用：

典型搭配速查
// Predicate<T>：T -> boolean
Predicate<String> empty   = String::isEmpty;       // 任意实例方法引用
Predicate<String> numeric = Validator::isNumeric;  // 静态方法引用

// Function<T, R>：T -> R
Function<String, Integer> parse = Integer::parseInt;    // 静态方法引用
Function<String, String>  upper = String::toUpperCase; // 任意实例方法引用

// Consumer<T>：T -> void
Consumer<String> print = System.out::println;       // 特定实例方法引用

// Supplier<T>：() -> T
Supplier<List<String>> listSupply = ArrayList::new;  // 构造器引用

// BiFunction<T, U, R>：(T, U) -> R
BiFunction<String, String, Boolean> starts = String::startsWith; // 任意实例方法引用

9. 综合实战：Stream 管道中的方法引用

在真实代码中，方法引用几乎总是出现在 Stream 操作链中。下面展示一个完整示例，把多种形式串联起来：

完整 Stream 管道示例
record Employee(String name, int salary) {
    static boolean isHighEarner(Employee e) {
        return e.salary() > 20000;
    }
}

List<Employee> staff = List.of(
    new Employee("Alice", 25000),
    new Employee("bob",   18000),
    new Employee("CAROL", 30000)
);

List<String> result = staff.stream()
    .filter(Employee::isHighEarner)    // 静态方法引用：Predicate
    .map(Employee::name)               // 任意实例方法引用（record accessor）
    .map(String::toLowerCase)          // 任意实例方法引用：统一转小写
    .sorted(String::compareTo)         // 任意实例方法引用：Comparator
    .collect(Collectors.toList());

result.forEach(System.out::println);  // 特定实例方法引用：Consumer
// 输出：alice、carol（高薪 + 小写 + 排序）

10. 常见陷阱与误区

陷阱 1：静态方法 vs 任意实例方法的语法相同

Integer::parseInt 和 String::toUpperCase 语法一模一样，但前者是静态引用，后者是实例引用。区分依据是方法本身的修饰符，不是语法。

陷阱 2：重载方法导致编译歧义

重载歧义示例
// println 有多个重载版本：println(String)、println(int)、println(Object)…
// 当目标类型明确时没问题
Consumer<String>  c1 = System.out::println;  // OK，编译器选 println(String)
Consumer<Integer> c2 = System.out::println;  // OK，编译器选 println(Integer/Object)

// 但如果目标类型不够具体，就会报"ambiguous method reference"
// var 在这里无法推断：
// var c3 = System.out::println;  // 编译错误！

陷阱 3：null 接收者问题

特定实例方法引用中，如果那个实例是 null，引用本身可以创建成功，但调用时会抛 NullPointerException。

null 接收者陷阱
String s = null;
Supplier<String> sup = s::toUpperCase;  // 编译通过，不报错
sup.get();  // 运行时 NullPointerException ！

陷阱 4：不能引用带 checked exception 的方法（直接用）

如果目标函数式接口的抽象方法没有声明 checked exception，而被引用的方法抛出 checked exception，编译直接报错。需要包一层 try-catch 或自定义函数式接口。

Checked Exception 陷阱
// Files.readString(Path) 抛出 IOException（checked exception）
// Function<Path, String> 的 apply 没有声明 throws IOException
// 以下代码无法编译：
// Function<Path, String> reader = Files::readString;  // 编译错误！

// 解决方案一：包 Lambda
Function<Path, String> reader = path -> {
    try { return Files.readString(path); }
    catch (IOException e) { throw new UncheckedIOException(e); }
};

// 解决方案二：自定义声明 throws 的函数式接口
@FunctionalInterface
interface ThrowingFunction<T, R> {
    R apply(T t) throws Exception;
}
ThrowingFunction<Path, String> safeReader = Files::readString;  // 现在可以了

11. 面试高频考点

  • 方法引用与 Lambda 的关系？方法引用是 Lambda 的语法糖，当 Lambda 体只是转发到已有方法时可以替换。两者在字节码层面通过 invokedynamic 实现，无性能差异。
  • 四种形式分别是什么，举例？静态方法引用（Integer::parseInt）、特定实例方法引用（System.out::println）、任意实例方法引用（String::toUpperCase）、构造器引用（ArrayList::new）。
  • 任意实例方法引用的参数如何对应？第一个参数自动成为调用者（接收者），其余参数依次对应方法的形参。
  • String::valueOf 是哪种？String::toUpperCase 呢？valueOf 是静态方法引用；toUpperCase 是任意实例方法引用。语法相同，靠方法修饰符区分。
  • 为什么方法引用不能直接引用抛 checked exception 的方法？因为目标函数式接口的抽象方法签名不包含 throws，类型不匹配，需要包装或自定义函数式接口。

12. 与其他知识点的联系

向前联系：方法引用依赖函数式接口（@FunctionalInterface）和 Lambda 表达式（上一节），三者构成 Java 函数式编程的基础三角。
向后联系：方法引用是 Stream API（下一节）管道操作中最常见的参数形式。map、filter、sorted、forEach 几乎全都可以用方法引用替代 Lambda。
JDK 源码联系：Comparator.comparing(Person::getName) 内部就是对方法引用的典型使用——把 key extractor 包装成 Comparator，这是 Java 8 之后 API 设计的惯用手法。
Spring 中的方法引用：Spring Security 的 http.authorizeHttpRequests(auth -> auth.requestMatchers(...).permitAll())、Spring Data 的 Specification 模式中大量出现方法引用，理解四种形式是阅读框架源码的前提。
