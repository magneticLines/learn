07. Stream API
中间操作 / 终端操作

Java 8 引入的声明式数据处理管道——让集合操作从"告诉 JVM 怎么做"变成"告诉 JVM 要什么"。

1. 为什么需要 Stream？

Java 8 之前，处理集合数据只能写命令式循环：外部迭代，逻辑与遍历耦合在一起，代码冗长且难以并行。 Stream 带来了内部迭代：你描述"做什么"，JVM 决定"怎么遍历"，甚至可以透明地并行。

一个典型对比——找出所有价格 > 100 的商品名称并排序：

命令式 vs 声明式Java
// ❌ 命令式：细节淹没意图
List<String> result = new ArrayList<>();
for (Product p : products) {
    if (p.getPrice() > 100) {
        result.add(p.getName());
    }
}
Collections.sort(result);

// ✅ Stream：意图一目了然
List<String> result = products.stream()
    .filter(p -> p.getPrice() > 100)
    .map(Product::getName)
    .sorted()
    .collect(Collectors.toList());

2. Stream 的本质：延迟求值的管道

Stream 不是数据结构，它是一个计算描述。整条管道由三段组成：

数据源（Source）
  └── 中间操作（Intermediate Operation）—— 返回新 Stream，惰性执行
      └── 终端操作（Terminal Operation）—— 触发整条管道，产生结果
惰性（Lazy）求值：调用 filter()、map() 时什么都不会执行，只是在管道上"登记"了一个操作。只有当终端操作被调用时，JVM 才会把元素一个接一个地推过整条管道。这意味着 findFirst() 遇到第一个匹配元素就会停止，不会遍历整个集合。

3. 创建 Stream 的四种方式

Stream 的创建Java
// 1. 从集合创建（最常用）
Stream<String> s1 = list.stream();
Stream<String> s2 = list.parallelStream();  // 并行流

// 2. 从数组创建
Stream<String> s3 = Arrays.stream(new String[]{"a", "b"});

// 3. Stream.of() 直接构造
Stream<Integer> s4 = Stream.of(1, 2, 3, 4);

// 4. 无限流（配合 limit 截断）
Stream<Integer> s5 = Stream.iterate(0, n -> n + 2).limit(10);  // 0,2,4,6,8...
Stream<Double>  s6 = Stream.generate(Math::random).limit(5);

// 基本类型特化：避免 Integer 装箱拆箱开销
IntStream    is = IntStream.range(0, 100);     // [0, 100)
LongStream   ls = LongStream.rangeClosed(1, 100);  // [1, 100]

4. 中间操作（Intermediate Operations）

中间操作返回一个新的 Stream，可以链式调用，不会立即执行。

操作签名（简化）说明
filterfilter(Predicate)保留满足条件的元素
mapmap(Function)元素一对一转换（改变类型/值）
flatMapflatMap(Function)元素一对多展开，然后合并成一个流
distinctdistinct()去重，依赖 equals/hashCode
sortedsorted() / sorted(Comparator)排序（有状态，需读完全部元素）
limitlimit(long n)截取前 n 个元素（短路）
skipskip(long n)跳过前 n 个元素
peekpeek(Consumer)调试用途：不改变元素，执行副作用
mapToIntmapToInt(ToIntFunction)转换为 IntStream，避免装箱

filter + map：最常见的组合

filter + mapJava
List<String> names = List.of("Alice", "Bob", "Charlie", "Dave");

List<String> result = names.stream()
    .filter(n -> n.length() > 3)       // 保留长度 > 3 的名字
    .map(String::toUpperCase)          // 全部转大写
    .collect(Collectors.toList());
// ["ALICE", "CHARLIE", "DAVE"]

flatMap：拍平嵌套结构

map 是一对一，flatMap 是一对多再合并——想象把多个小盒子里的东西全倒进一个大盒子。

flatMap 展开单词字母Java
List<String> words = List.of("hello", "world");

// map 的结果：Stream<String[]> —— 流里还是数组，不是字母
words.stream().map(String::chars);   // ❌ 得到 Stream<IntStream>

// flatMap 把每个单词拆成字符流后合并
List<String> letters = words.stream()
    .flatMap(w -> Arrays.stream(w.split("")))
    .distinct()
    .collect(Collectors.toList());
// [h, e, l, o, w, r, d]

peek：调试管道的利器

peek 追踪管道执行Java
List<Integer> result = Stream.of(1,2,3,4,5)
    .peek(n -> System.out.println("before filter: " + n))
    .filter(n -> n % 2 == 0)
    .peek(n -> System.out.println("after filter: "  + n))
    .collect(Collectors.toList());
// 输出证明：元素逐个穿过管道，而非先 filter 全部再 peek

5. 终端操作（Terminal Operations）

终端操作触发整条管道的执行，产生一个非 Stream 的结果（或副作用）。执行后 Stream 即关闭，不可复用。

操作返回类型说明
collectR最灵活的归约，配合 Collectors 工厂
forEachvoid遍历，纯副作用，不返回值
countlong元素总数
reduceOptional<T> / T折叠：把流"压缩"成一个值
findFirstOptional<T>返回第一个元素（短路）
findAnyOptional<T>任意一个元素（并行流更快）
anyMatchboolean任意一个满足条件（短路）
allMatchboolean全部满足条件（短路）
noneMatchboolean全部不满足条件（短路）
min / maxOptional<T>按 Comparator 取最小/最大值
toArrayObject[]转为数组

reduce：理解归约的本质

reduce 是所有"聚合操作"的底层抽象，sum/count/max 本质上都是特化的 reduce。

reduce 手动实现求和与字符串拼接Java
// 带初始值：reduce(identity, BinaryOperator)
int sum = IntStream.rangeClosed(1, 10)
    .reduce(0, Integer::sum);  // = 55

// 不带初始值：结果是 Optional（流可能为空）
Optional<String> joined = Stream.of("a", "b", "c")
    .reduce((s1, s2) -> s1 + "-" + s2);
joined.ifPresent(System.out::println);  // a-b-c

collect：最强大的终端操作

Collectors 提供了大量内置收集器，覆盖绝大多数实际需求：

Collectors 常用收集器Java
List<Employee> employees = /* ... */;

// 分组：按部门分类
Map<String, List<Employee>> byDept = employees.stream()
    .collect(Collectors.groupingBy(Employee::getDept));

// 分区：满足条件 vs 不满足
Map<Boolean, List<Employee>> partitioned = employees.stream()
    .collect(Collectors.partitioningBy(e -> e.getSalary() > 10000));

// 统计：各部门薪资均值
Map<String, Double> avgSalary = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.averagingDouble(Employee::getSalary)
    ));

// 拼接字符串
String csv = employees.stream()
    .map(Employee::getName)
    .collect(Collectors.joining(", ", "[", "]"));
// [Alice, Bob, Charlie]

6. 有状态操作 vs 无状态操作

这个区别决定了操作在并行流下的安全性和性能：

分类操作说明
无状态filter、map、flatMap、peek每个元素独立处理，天然并行安全
有状态（有界）distinct、sorted需要看到所有元素才能处理，parallel 下有同步开销
短路limit、skip、findFirst、anyMatch不需要处理全部元素即可提前结束

7. 并行流（Parallel Stream）

调用 .parallelStream() 或 .parallel() 即可开启并行处理，底层使用 ForkJoinPool.commonPool()。

并行流不是银弹，三大陷阱：
1. 数据量小时反而更慢：线程切换、合并开销远大于计算本身。
2. forEach 顺序不保证：需要有序输出时用 forEachOrdered。
3. 共享可变状态 = 竞态条件：在 lambda 里修改外部变量会导致结果错误，必须使用无副作用的操作。
并行流的正确与错误用法Java
// ❌ 错误：在 forEach 里修改共享 List
List<Integer> bad = new ArrayList<>();
IntStream.range(0, 1000).parallel().forEach(bad::add);  // 结果不确定！

// ✅ 正确：用 collect 线程安全地归约
List<Integer> good = IntStream.range(0, 1000)
    .parallel()
    .boxed()
    .collect(Collectors.toList());  // 安全，结果确定

// ✅ 并行流真正适合的场景：CPU密集型大数据量
long count = LongStream.rangeClosed(1, 10_000_000L)
    .parallel()
    .filter(n -> n % 2 == 0)
    .count();

8. 综合案例：电商订单分析

结合中间操作与终端操作，统计各城市的订单总金额并降序排列：

综合案例Java
record Order(String city, double amount, String status) {}

List<Order> orders = /* 数据源 */;

// 需求：已完成订单 → 按城市汇总金额 → 取 Top 3
orders.stream()
    .filter(o -> "PAID".equals(o.status()))               // 只看已支付
    .collect(Collectors.groupingBy(
        Order::city,
        Collectors.summingDouble(Order::amount)           // 城市 → 总金额
    )).entrySet().stream()                               // Map.Entry 流
    .sorted(Map.Entry.comparingByValue(Comparator.reverseOrder()))
    .limit(3)                                             // Top 3
    .forEach(e -> System.out.printf("%s: %.2f%n", e.getKey(), e.getValue()));

9. 与其他知识的联系

依赖函数式接口（Lambda 的基础）

Stream API 的每个操作都接受一个函数式接口：filter 接受 Predicate<T>，map 接受 Function<T,R>，forEach 接受 Consumer<T>。没有 Lambda（第 04 节）和函数式接口，Stream 的写法会极其臃肿。

依赖方法引用（代码更简洁）

.map(String::toUpperCase) 比 .map(s -> s.toUpperCase()) 更直观。方法引用（第 06 节）是 Stream 的最佳拍档。

与 Optional 的协作

findFirst()、min()、max() 等操作返回 Optional<T>（第 08 节），提醒调用者"结果可能为空"，避免 NPE。

背后的 Spliterator 与 ForkJoin

并行流的数据分割由 Spliterator 完成，执行引擎是 ForkJoinPool.commonPool()，这与并发编程体系（第 10 节）深度关联。

10. 常见陷阱与注意事项

陷阱 1：Stream 不可复用
Stream 被终端操作消费后即关闭，再次调用会抛出 IllegalStateException: stream has already been operated upon or closed。每次使用前都要重新获取流。
陷阱 2：forEach 不适合做数据转换
forEach 是纯副作用操作，用它往外部 List 里 add 元素既不安全（并行）也不优雅（串行）。数据转换用 map + collect，不要用 forEach。
陷阱 3：基本类型装箱开销
Stream<Integer> 里的每个元素都是装箱对象，大数量级下内存和 GC 开销显著。数值计算优先用 IntStream、LongStream、DoubleStream。
陷阱 4：sorted 在并行流下的性能坑
sorted 是有状态操作，并行流下需要合并各子任务结果后才能排序，反而可能比串行更慢。如无必要，不要对并行流排序。
最佳实践：管道的"过滤前置"原则
把 filter（尤其是高过滤率的）放在管道最前面，让后续操作处理的元素越少越好，减少不必要的 map 等操作开销。

11. 面试高频考点

  • Stream 和 Collection 的区别是什么？——Collection 是数据结构（存储），Stream 是计算描述（处理），不存储数据，不修改数据源。
  • 解释 Stream 的惰性求值。——中间操作不执行，只是构建管道；终端操作触发管道，元素逐个穿过所有中间步骤。
  • map 和 flatMap 的区别？——map 一对一，flatMap 一对多后展开合并，用于处理嵌套结构（如 List<List>）。
  • 并行流底层是什么？有什么风险？——ForkJoinPool.commonPool()；风险：共享可变状态导致竞态，有状态操作性能下降，数据量小时开销反超。
  • reduce 和 collect 的区别？——reduce 适合不可变的值归约；collect 用于构建可变容器（List/Map/String），是 reduce 的可变版本（mutable reduction）。
  • Stream 操作链中，limit(5).sorted() 和 sorted().limit(5) 效率一样吗？——完全不同。前者只排序 5 个元素；后者需要先对整个流排序，再取前 5 个，差距可达数量级。
