10. checked vs unchecked 异常

Java 独树一帜地把异常分成两类，编译器对它们态度截然不同——一类必须显式处理，另一类可以沉默忽略。理解这个设计决策，是写出健壮 Java 代码的关键。

为什么要区分两类异常？

Java 在 1995 年设计时，James Gosling 团队面对一个核心问题：如何迫使程序员处理可预见的错误？

网络断开、文件不存在、数据库连接失败——这些都是"业务上可能发生、调用方应当预料到"的情况。如果不强制处理，程序员可能会忘记，运行时悄无声息地崩溃，用户一脸茫然。

于是 Java 引入了 checked 异常：由编译器在编译期强制检查，你要么 try-catch 处理它，要么 throws 声明向上抛。不处理就编译不过。

而另一类错误，如数组越界、空指针、除以零，往往是程序逻辑 bug，不是正常业务流程的一部分——强制 catch 反而会污染代码。这类就是 unchecked 异常，编译器对其睁一只眼闭一只眼。

本质区分：checked = 外部世界可能出错（I/O、网络、数据库）；unchecked = 程序自身逻辑错误。前者调用方可以合理应对，后者应该修 bug 而不是 catch。

类继承结构一览

Throwable ├── Error　　　　　　　　　　　　　　← unchecked，不应捕获 │ ├── OutOfMemoryError │ ├── StackOverflowError │ └── ... └── Exception ├── RuntimeException　　　　　　← unchecked（及其所有子类） │ ├── NullPointerException │ ├── IllegalArgumentException │ ├── IndexOutOfBoundsException │ └── ... └── （其他 Exception 子类）　　← checked ├── IOException ├── SQLException ├── ClassNotFoundException └── ...

判断口诀：继承自 RuntimeException 或 Error 的 → unchecked；其他继承自 Exception 的 → checked。

核心对比

维度 checked 异常 unchecked 异常
父类 Exception（非 RuntimeException） RuntimeException / Error
编译器检查 必须处理，否则编译失败 不强制，可以忽略
适用场景 外部资源故障、可预见的业务错误 编程逻辑 bug、不可恢复错误
调用方如何处理 有意义的恢复（重试、降级、提示用户） 修复代码；生产环境统一捕获记录日志
典型例子 IOException、SQLException、ParseException NullPointerException、IllegalArgumentException、ArrayIndexOutOfBoundsException
方法签名 必须 throws XxxException 可以不声明（声明也合法）

代码示例：亲眼看编译器的"强制"

4.1 checked 异常——不处理，编译报错

Javachecked 异常必须声明或捕获
// 错误示范：编译器会报 "Unhandled exception: java.io.IOException"
public class BadExample {
    public void readFile() {
        // IOException 是 checked，不处理 → 编译失败
        Files.readAllBytes(Path.of("data.txt"));
    }
}

// 正确做法 1：try-catch 就地处理
public void readFile() {
    try {
        byte[] data = Files.readAllBytes(Path.of("data.txt"));
        // ... 处理数据
    } catch (IOException e) {
        // 有意义的恢复逻辑，而不是空 catch！
        System.err.println("文件读取失败: " + e.getMessage());
    }
}

// 正确做法 2：throws 向上传递，让调用方决定
public void readFile() throws IOException {
    byte[] data = Files.readAllBytes(Path.of("data.txt"));
}

4.2 unchecked 异常——编译通过，运行时爆炸

Javaunchecked 异常：编译器沉默，运行时崩溃
public class UncheckedDemo {

    // 编译完全正常，但 index=-1 时运行时抛 ArrayIndexOutOfBoundsException
    public int get(int[] arr, int index) {
        return arr[index];  // 正确做法：先做边界检查，而不是 try-catch
    }

    // IllegalArgumentException 是 unchecked，用于"调用者传了非法参数"
    public void setAge(int age) {
        if (age < 0 || age > 150) {
            // 不需要 throws 声明，调用方也不需要 try-catch
            throw new IllegalArgumentException("年龄必须在 0-150 之间，实际：" + age);
        }
        this.age = age;
    }
}

4.3 自定义异常：如何选择继承哪个父类

Java自定义 checked vs unchecked 异常
// 场景：用户不存在 → 业务异常，调用方需要感知并处理（如返回 404）
// 继承 Exception → checked，强制调用方处理
public class UserNotFoundException extends Exception {
    public UserNotFoundException(String userId) {
        super("用户不存在: " + userId);
    }
}

// 场景：系统配置文件损坏 → 程序无法继续，属于不可恢复错误
// 继承 RuntimeException → unchecked，不污染方法签名
public class ConfigCorruptedException extends RuntimeException {
    public ConfigCorruptedException(String msg, Throwable cause) {
        super(msg, cause);  // 保留原始 cause，便于排查
    }
}

// 使用示例
public User findUser(String id) throws UserNotFoundException {
    User user = db.find(id);
    if (user == null) throw new UserNotFoundException(id);
    return user;
}

checked 异常的设计争议

checked 异常是 Java 最具争议的设计之一。支持者认为它让 API 契约更清晰，调用方被迫思考错误处理。反对者（包括 C#、Kotlin、Scala 的设计者）认为它导致了大量"吃掉异常"的反模式。

反模式警告：空 catch 块是比不处理更危险的行为——它让错误悄无声息地消失，排查时毫无线索。
Javachecked 异常的三种反模式
// ❌ 反模式 1：空 catch——异常被吞，程序状态未知
try {
    connect();
} catch (IOException e) {
    // TODO 先这样  ← 最终忘记填，上线就是定时炸弹
}

// ❌ 反模式 2：catch Exception 一把抓——掩盖了真正的问题
try {
    doSomething();
} catch (Exception e) {
    System.out.println("出错了");  // NullPointerException 也被吃掉了
}

// ❌ 反模式 3：把 checked 包成 RuntimeException 只为通过编译（不记录 cause）
try {
    connect();
} catch (IOException e) {
    throw new RuntimeException("error");  // 丢失了原始 cause，堆栈断链
}

// ✅ 正确：包装时保留 cause
try {
    connect();
} catch (IOException e) {
    throw new RuntimeException("数据库连接失败", e);  // e 作为 cause 传入
}

Spring 的选择：全面转向 unchecked

Spring 框架几乎所有自定义异常都继承自 RuntimeException。例如：

  • DataAccessException（unchecked）— 封装所有数据库操作异常，让 Service 层无需 throws SQLException
  • HttpClientErrorException（unchecked）— RestTemplate 调用失败时抛出
  • BeansException（unchecked）— Bean 容器相关异常

这体现了现代 Java 框架的主流哲学：在框架边界统一捕获并转换，而不是让业务代码到处 catch。

Kotlin 完全抛弃了 checked 异常机制（所有异常都是 unchecked）；C# 从未引入 checked 异常。Java 生态也逐渐向 unchecked 倾斜，但理解 checked 异常的设计意图仍然是 Java 程序员的必备素养。

与其他知识点的联系

与方法重写（Override）的关系

子类重写方法时，不能抛出比父类更宽泛的 checked 异常，但可以抛更窄的、或不抛。这是里氏替换原则（LSP）在异常上的体现——子类对象必须可以替换父类对象使用，调用方依赖父类的 throws 声明，子类不能违背。

Java重写时 checked 异常的限制
class Base {
    public void read() throws IOException { }
}

class Child extends Base {
    @Override
    // ✅ 可以：FileNotFoundException 是 IOException 的子类（更窄）
    public void read() throws FileNotFoundException { }

    // ❌ 不可以：Exception 比 IOException 更宽泛，编译失败
    // public void read() throws Exception { }
}

与函数式接口（Lambda）的关系

Java 内置的函数式接口（如 Runnable、Function、Consumer）的函数方法都不声明 throws，因此 Lambda 表达式内部不能直接抛 checked 异常，需要包装成 unchecked。这也是为什么 Stream API 中处理 IO 特别繁琐的根本原因。

JavaLambda 与 checked 异常的摩擦
// ❌ 编译失败：Consumer 不声明 throws IOException
List<String> files = List.of("a.txt", "b.txt");
files.forEach(f -> Files.delete(Path.of(f)));  // IOException 无处声明

// ✅ 包装成 unchecked 来绕过
files.forEach(f -> {
    try {
        Files.delete(Path.of(f));
    } catch (IOException e) {
        throw new RuntimeException(e);
    }
});

常见陷阱

  • 陷阱 1：Error 也是 unchecked — OutOfMemoryError、StackOverflowError 不属于 Exception，不应该 catch（即使能 catch，程序状态已不可信）。
  • 陷阱 2：catch 顺序错误 — 子类异常必须写在父类前面，否则子类 catch 永远不会被执行，编译器会报"已捕获的异常"错误。
  • 陷阱 3：InterruptedException 被静默吃掉 — 这是 checked 异常，但很多人直接空 catch，导致线程中断标志丢失，并发代码出现诡异行为。正确做法是恢复中断标志：Thread.currentThread().interrupt()。
  • 陷阱 4：多 catch 时同一异常被 "覆盖" — 使用多 catch（catch (A | B e)）时，A 和 B 不能有继承关系，否则编译报错。

面试高频考点

问题要点
checked 和 unchecked 异常的区别？ 是否继承 RuntimeException；编译器是否强制处理；适用场景不同
为什么 Spring 全用 unchecked？ 避免污染方法签名；在框架边界统一处理；降低业务代码耦合
重写方法能否抛出更宽泛的 checked 异常？ 不能。只能抛同类、子类，或不抛。unchecked 不受此限制
