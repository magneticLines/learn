核心类库 · IO

序列化与反序列化

将对象的状态转换为字节流（持久化/传输），再从字节流还原对象——理解 Java 对象生命周期跨越 JVM 边界的核心机制。

为什么需要序列化

Java 对象活在 JVM 堆内存里，JVM 一旦退出，对象就消失了。 序列化（Serialization）的本质是把对象的字段值编码成一段字节流，让它能：

  • 写入磁盘 —— 持久化（保存游戏存档、缓存）
  • 通过网络发送 —— RPC / 消息队列（RMI、Dubbo 等早期方案）
  • 在 JVM 之间传递 —— 分布式计算（Spark、早期 Hadoop）
  • 深拷贝一个对象图 —— 序列化再反序列化等于完整克隆

反序列化（Deserialization）则是逆过程：把字节流重新解码成堆上的对象， 绕过构造器直接恢复字段值，这也是它最危险的地方（见陷阱一节）。

Serializable 接口：标记即授权

实现 java.io.Serializable 是序列化的最低门槛。它是一个标记接口（无任何方法）， 只是告诉 JVM："我允许你对我做序列化"。 没有这个标记，ObjectOutputStream 会直接抛出 NotSerializableException。

User.java — 最简示例
import java.io.Serializable;

public class User implements Serializable {

    // 版本号：控制向后兼容性（见下文详解）
    private static final long serialVersionUID = 1L;

    private String name;
    private int    age;
    private transient String password;  // transient：不参与序列化

    public User(String name, int age, String password) {
        this.name     = name;
        this.age      = age;
        this.password = password;
    }

    @Override
    public String toString() {
        return "User{name='" + name + "', age=" + age +
               ", password='" + password + "'}";
    }
}

ObjectOutputStream / ObjectInputStream 基本用法

SerializeDemo.java — 写出再读回
import java.io.*;
import java.nio.file.*;

public class SerializeDemo {

    static final Path FILE = Path.of("user.ser");

    // ── 序列化：对象 → 字节流 → 文件 ──────────────────────────
    static void save(User u) throws IOException {
        try (var oos = new ObjectOutputStream(
                          Files.newOutputStream(FILE))) {
            oos.writeObject(u);
            // writeObject 会先写一个魔数 0xACED，再写类描述符，最后写字段值
        }
    }

    // ── 反序列化：文件 → 字节流 → 对象 ──────────────────────────
    static User load() throws IOException, ClassNotFoundException {
        try (var ois = new ObjectInputStream(
                          Files.newInputStream(FILE))) {
            return (User) ois.readObject();
            // 注意：绕过构造器直接分配内存并填充字段
        }
    }

    public static void main(String[] args) throws Exception {
        User original = new User("Alice", 30, "s3cr3t");
        save(original);

        User restored = load();
        System.out.println(restored);
        // 输出：User{name='Alice', age=30, password='null'}
        // ↑ transient 字段反序列化后为默认值 null
    }
}
为什么用装饰器模式？
ObjectOutputStream 包裹任意 OutputStream（文件、网络 socket、内存字节数组……）， 这是 Java IO 装饰器设计的典型体现——序列化逻辑与传输介质完全解耦。

serialVersionUID：版本契约

serialVersionUID 是序列化字节流和本地类之间的版本合同。 反序列化时，JVM 会比对字节流里的 UID 和当前类的 UID，不一致就抛 InvalidClassException。

不声明 serialVersionUID 时 JVM 怎么算？
├── 取类名、字段名、方法签名等信息
├── SHA-1 哈希 → 取前 8 字节
└── 问题：改一个方法签名就会导致 UID 变化 → 旧数据无法读取！
演示 UID 不匹配的后果
// 版本 1：序列化时的类
public class Config implements Serializable {
    // 没声明 serialVersionUID，JVM 自动算出某个值，比如 -8901234567L
    private String host;
}

// 版本 2：新增了一个方法（看似无害），JVM 重新算出不同的 UID
public class Config implements Serializable {
    private String host;
    public String getHost() { return host; }  // ← 新增方法
    // 反序列化旧文件 → InvalidClassException: local class incompatible
}

// 正确做法：显式声明，并按实际兼容性手动递增
public class Config implements Serializable {
    private static final long serialVersionUID = 1L;  // 手动控制
    private String host;
    public String getHost() { return host; }
}
面试高频：为什么建议显式声明 serialVersionUID？ 因为 JVM 自动生成的 UID 对任何结构变化都极其敏感，稍微重构就会导致线上存储的旧数据全部不可读，造成生产事故。

transient：排除敏感字段

用 transient 修饰的字段不参与序列化，反序列化后得到该类型的零值（引用类型为 null，基本类型为 0/false）。 典型使用场景：

  • 密码、Token 等安全敏感字段
  • 可根据其他字段重新计算的缓存（如 HashMap 中的 hash 字段）
  • 不可序列化的资源句柄（数据库连接、线程、文件流）
  • static 字段本身就不属于对象状态，默认也不参与序列化
transient 与 static 的区别
public class Counter implements Serializable {
    private static final long serialVersionUID = 1L;

    private int count;               // ✓ 参与序列化（对象状态）
    private transient long lastAccess; // ✗ 不序列化（可重建的缓存）
    private static int instanceCount; // ✗ 不序列化（类级别，不属于某个对象）
}

自定义序列化：writeObject / readObject

有时默认的字段逐一复制并不够用——比如要加密、压缩、做数据迁移（旧格式→新格式）。 Java 允许在类内部声明两个私有方法来完全接管序列化逻辑：

自定义序列化：对密码做 Base64 混淆
import java.io.*;
import java.util.Base64;

public class Credential implements Serializable {
    private static final long serialVersionUID = 1L;

    private String username;
    private transient String password;  // transient，防止默认序列化泄漏

    /**
     * JVM 序列化时会反射调用这个私有方法（特殊协议，不需要接口）
     * defaultWriteObject() 先写普通字段，再手动处理 transient
     */
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();  // 写 username
        String encoded = Base64.getEncoder().encodeToString(
            password.getBytes());
        oos.writeObject(encoded);   // 写混淆后的密码
    }

    private void readObject(ObjectInputStream ois)
            throws IOException, ClassNotFoundException {
        ois.defaultReadObject();    // 读 username
        String encoded = (String) ois.readObject();
        password = new String(Base64.getDecoder().decode(encoded));
    }
}
这两个方法为什么声明成 private？
JVM 内部通过反射调用它们，用 private 是为了防止子类意外覆盖或外部直接调用，维护封装性。 这是 Java 序列化协议的一个"魔法约定"，不在任何接口里声明，靠方法名和签名精确匹配触发。

Externalizable：完全自主的序列化

Externalizable 继承自 Serializable，但声明了两个必须实现的方法， 要求你从零手写全部序列化逻辑，没有任何默认行为。优点是性能更好、格式更可控；缺点是写起来繁琐。

Externalizable 示例
import java.io.*;

public class Point implements Externalizable {
    private double x, y;

    // ⚠️ 必须有无参构造器！反序列化时先调用它再 readExternal
    public Point() {}
    public Point(double x, double y) { this.x=x; this.y=y; }

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeDouble(x);
        out.writeDouble(y);
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException {
        x = in.readDouble();
        y = in.readDouble();
    }
}
对比维度SerializableExternalizable
实现成本低（标记即可）高（手写全部逻辑）
性能一般更好（可省略不必要字段）
无参构造器要求不要求必须有 public 无参构造器
反序列化入口绕过构造器先调无参构造器
适用场景通用持久化对性能/格式极度敏感

对象图与引用共享

Java 序列化会追踪整个对象图（通过引用可达的所有对象）， 并自动处理循环引用和共享引用——同一个对象只写一次，用引用 ID 记录重复出现的地方。

共享引用被正确保留
List<String> shared = new ArrayList<>();
shared.add("hello");

Object[] arr = {shared, shared};  // arr[0] 和 arr[1] 指向同一个列表

// 序列化后再反序列化：
Object[] restored = ...;
System.out.println(restored[0] == restored[1]);  // true，引用共享被保留
副作用：如果对象图很大、引用层次很深（如递归数据结构）， ObjectOutputStream 内部会持有一个引用表防止循环引用， 导致内存暴涨甚至 StackOverflowError。 对于深度递归结构（如自定义链表），需要自行实现 writeObject 改写成迭代。

readResolve 与 writeReplace：单例与代理

这两个钩子方法让你在对象被写出/读入的最后一步替换对象，是保护单例模式的唯一正确手段。

单例序列化：没有 readResolve 会破坏单例
public class AppConfig implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final AppConfig INSTANCE = new AppConfig();

    private AppConfig() {}
    public static AppConfig getInstance() { return INSTANCE; }

    /**
     * 反序列化完毕后，JVM 调用此方法，用返回值替换刚创建的新对象。
     * 不加这个方法，readObject 会 new 出第二个实例，单例被破坏！
     */
    private Object readResolve() {
        return INSTANCE;  // 把新对象扔掉，返回唯一实例
    }
}
更好的做法：用 enum 实现单例。枚举序列化由 JVM 特殊处理（只序列化枚举名称，反序列化时通过 Enum.valueOf 查找），天然防止重复实例化，也防止反射攻击，是《Effective Java》第 3 版推荐的方式。

安全陷阱：反序列化漏洞

反序列化绕过构造器这一特性，成了历史上最严重的 Java 安全漏洞之一。 攻击者构造一段恶意字节流，触发目标类在 readObject/finalize 中执行任意代码（RCE）。 典型案例：Apache Commons Collections 反序列化 RCE（CVE-2015-4852）。

核心原则：永远不要反序列化来自不可信来源的数据。
如果必须接受外部数据，优先使用 JSON（Jackson/Gson）、Protocol Buffers、Avro 等格式， 它们在设计上没有"任意代码执行"的语义。

Java 9+ 提供了 序列化过滤器（JEP 290） 作为防御手段：

ObjectInputFilter — 白名单过滤（Java 9+）
ObjectInputStream ois = new ObjectInputStream(stream);

// 只允许反序列化 com.example 包下的类，拒绝其他一切
ois.setObjectInputFilter(
    ObjectInputFilter.Config.createFilter("com.example.*;!*"));

Object obj = ois.readObject();  // 其他类会抛 InvalidClassException

现代替代方案对比

Java 原生序列化（ObjectOutputStream）生成的字节流体积大、性能差、不跨语言、有安全风险， 在实际项目中已基本被替代：

方案格式优点缺点
Java 原生序列化 二进制（专有） 无需依赖，自动处理引用图 体积大、慢、不跨语言、安全风险
Jackson / Gson JSON（文本） 可读、跨语言、生态成熟 体积比二进制大，不保留引用共享
Protocol Buffers 二进制（IDL） 极小体积、极快、强类型、跨语言 需要 .proto 文件，不可读
Kryo 二进制 比原生快 10x，体积小 不保证跨版本兼容，需注册类
Avro 二进制（Schema） 动态 Schema，适合大数据管道 上手成本高

序列化实现深拷贝

利用"序列化 → 写入内存 → 反序列化"可以对整个对象图做深拷贝， 是一种不需要手写 clone() 的快捷方式（但性能较差，仅适合小对象/低频场景）：

工具方法：基于序列化的深拷贝
@SuppressWarnings("unchecked")
public static <T extends Serializable> T deepCopy(T obj) {
    try (var baos = new ByteArrayOutputStream();
         var oos  = new ObjectOutputStream(baos)) {

        oos.writeObject(obj);  // 序列化到内存

        try (var ois = new ObjectInputStream(
                new ByteArrayInputStream(baos.toByteArray()))) {
            return (T) ois.readObject();  // 反序列化为新对象
        }
    } catch (Exception e) {
        throw new RuntimeException("深拷贝失败", e);
    }
}

常见陷阱汇总

  • 父类未实现 Serializable：子类序列化时，父类字段不会被保存；反序列化时父类无参构造器会被调用，可能导致父类字段为默认值，产生静默的数据错误。
  • 集合中含不可序列化对象：如 ArrayList 本身可序列化，但元素若不是 Serializable，运行时才会抛出 NotSerializableException，编译期不报错。
  • 内部类的隐藏引用：非静态内部类持有外部类的引用，序列化内部类会把外部类一起拖进来，如果外部类不可序列化则失败。用 static 嵌套类可避免此问题。
  • Lambda 不能序列化：普通 Lambda 不实现 Serializable，若要序列化 Lambda，需要显式转型：(Serializable & Runnable) () -> {}，但这种用法非常少见且脆弱。
  • 版本升级后字段改名：即使 UID 不变，字段名改了也会导致旧字段值丢失（静默，不抛异常），这是迁移数据时最难察觉的 bug。

面试高频考点

Q1：serialVersionUID 不声明会怎样？
JVM 自动根据类结构计算，任何结构变化（包括新增方法）都可能导致 UID 变化，使旧数据无法反序列化。生产中必须显式声明。
Q2：反序列化会调用构造器吗？
Serializable 的反序列化不调用目标类的构造器，直接通过 sun.misc.Unsafe.allocateInstance 分配内存并填充字段。 Externalizable 则会先调用无参构造器。
Q3：如何防止序列化破坏单例？
实现 readResolve() 返回已有的单例实例；或者改用 enum 实现单例（JVM 对枚举序列化有特殊保护）。
Q4：static 字段和 transient 字段序列化有什么区别？
两者都不参与序列化。static 字段属于类不属于对象，天然不在序列化范围内；transient 是显式告诉序列化机制"跳过这个实例字段"。
Q5：为什么说 Java 原生序列化不安全？
readObject 在反序列化时可以触发任意类的初始化逻辑，攻击者可构造特殊字节流，利用 classpath 上的"gadget chain"（如 Commons Collections）实现远程代码执行（RCE）。应使用白名单过滤器或换用 JSON/Protobuf。

与其他知识点的联系

  • 字节流：序列化的本质就是把对象转换成一串字节，ObjectOutputStream/ObjectInputStream 属于字节流家族，可以包裹任意 OutputStream/InputStream（文件、网络 Socket、内存数组），这正是上一节讲的装饰器模式的延伸应用。
  • 深拷贝与浅拷贝：本文演示的"序列化再反序列化"深拷贝技巧，本质上遍历了整个对象图并重新分配内存，与 clone() 的默认浅拷贝行为形成鲜明对比——浅拷贝只复制引用，序列化深拷贝连引用指向的对象也一并复制。
  • 设计模式（原型模式）：原型模式要求"克隆已有对象而非重新构造"，序列化提供了一种通用、无需为每个类手写 clone() 的实现方式，代价是性能较差，仅适合低频场景。
  • 安全问题：反序列化绕过构造器、可以触发任意 readObject 逻辑，这一特性被 gadget chain 攻击（如 Commons Collections RCE）利用，是 Java 生态历史上影响最广的安全漏洞类型之一，直接推动了 JEP 290 序列化过滤器的诞生。
  • 分布式与 RPC：RMI（Java 远程方法调用）、早期的 Dubbo、Redis 存储对象缓存等场景都依赖序列化把对象跨越 JVM 边界传输，理解序列化机制是理解分布式系统对象传递的基础。
  • JSON 序列化对比：Jackson/Gson 生成的是可读的文本格式，天然跨语言、不保留引用共享；Java 原生序列化生成专有二进制格式，体积大且不安全。这也是文中"现代替代方案对比"一节的核心结论——生产环境中原生序列化基本只用于内部、可信场景。
