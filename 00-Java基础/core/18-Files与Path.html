18. Files 与 Path（NIO.2）

Java 7 引入的现代文件系统 API，彻底取代了残缺不全的 java.io.File，用声明式、链式的方式处理路径与文件操作。

为什么要重新设计？File 的七宗罪

在 Java 7 之前，java.io.File 是唯一的文件操作入口，但它有几个致命缺陷：

  • 方法失败无提示：file.delete() / file.mkdir() 只返回 boolean，失败原因完全不可知。
  • 不支持符号链接：无法区分真实文件与软链接。
  • 不可变路径难以操作：拼接路径要手写字符串，跨平台分隔符问题一团糟。
  • 缺少元数据能力：无法读取文件权限、所有者、创建时间等属性。
  • 遍历目录很麻烦：需要递归手写，没有流式 API。
  • 原子操作缺失：移动文件、创建临时文件等不保证原子性。

NIO.2（JSR-203）的解法是将「路径」与「文件操作」彻底分离：Path 只负责描述位置，Files 工具类负责所有 I/O 动作，并在失败时抛出携带原因的 IOException 子类。

Path：描述位置，而非文件本身

Path 是一个接口（java.nio.file.Path），代表文件系统中的一条路径。它不要求路径实际存在，更像一个"地址字符串的智能封装"。

java.nio.file
├── Path ← 接口，描述路径
├── Files ← 工具类，执行操作
├── Paths ← 工厂类，创建 Path（Java 11+ 可用 Path.of）
├── FileSystem ← 抽象文件系统（本地 / ZIP 等）
└── FileSystems ← FileSystem 工厂

创建 Path

创建 Path 的三种方式Java
// 方式 1：Paths.of（推荐，Java 11+）
Path p1 = Path.of("/home/user/docs/readme.txt");

// 方式 2：Paths.get（Java 7+，功能等价）
Path p2 = Paths.get("/home/user", "docs", "readme.txt");

// 方式 3：从旧 File 对象转换（兼容遗留代码）
Path p3 = new File("/home/user/docs/readme.txt").toPath();

// 反向转换：Path → File（与旧 API 互操作）
File f = p1.toFile();

路径的常用操作

Path 的解析与拼接Java
Path base = Path.of("/home/user/docs/readme.txt");

// 获取各部分
base.getFileName();   // readme.txt
base.getParent();     // /home/user/docs
base.getRoot();       // /（Windows 上是 C:\）
base.getNameCount();  // 4（home, user, docs, readme.txt）
base.getName(0);      // home

// 路径拼接：resolve = 在当前路径后追加
Path dir = Path.of("/home/user");
Path full = dir.resolve("docs/readme.txt");  // /home/user/docs/readme.txt

// 相对路径：relativize = 从 this 出发怎么走到 other
Path from = Path.of("/home/user");
Path to   = Path.of("/home/user/docs/readme.txt");
from.relativize(to);  // docs/readme.txt

// 规范化：消除 . 和 ..
Path.of("/home/user/../user/./docs").normalize();  // /home/user/docs

// 转为绝对路径（相对路径 + 当前工作目录）
Path.of("docs/readme.txt").toAbsolutePath();

// 真实路径：跟踪符号链接，文件必须存在
Path real = p1.toRealPath();  // 抛 IOException 如果不存在
设计直觉：resolve() 类似字符串拼接，但如果参数是绝对路径，则直接返回参数本身——这和 URL 解析行为一致，是"合并路径"的标准语义。

Files：文件操作的瑞士军刀

java.nio.file.Files 是一个全静态工具类，覆盖了文件的读写、复制、移动、删除、遍历等几乎所有操作，并在失败时抛出具体异常（如 NoSuchFileException、FileAlreadyExistsException）。

读写文件：小文件一行搞定

Files 读写小文件Java
Path file = Path.of("data.txt");

// 写入（覆盖）
Files.writeString(file, "Hello, NIO.2!", StandardCharsets.UTF_8);

// 追加写入
Files.writeString(file, "\nAppended line",
    StandardOpenOption.APPEND);

// 写入字节数组
Files.write(file, "bytes".getBytes());

// 写入多行
List<String> lines = List.of("line1", "line2", "line3");
Files.write(file, lines);

// ---- 读取 ----

// 读取全部内容为字符串
String content = Files.readString(file);

// 读取全部行（适合小文件）
List<String> allLines = Files.readAllLines(file, StandardCharsets.UTF_8);

// 读取为字节数组
byte[] bytes = Files.readAllBytes(file);

// 流式读取大文件（惰性加载，记得关闭）
try (Stream<String> stream = Files.lines(file)) {
    stream.filter(l -> l.startsWith("ERROR"))
          .forEach(System.out::println);
}
大文件禁用 readAllLines / readString！它们会把整个文件加载进堆内存。处理日志、CSV 等大文件时务必用 Files.lines()，它返回惰性 Stream，逐行读取。

获取 BufferedReader / InputStream（与旧 API 桥接）

与流式 API 互操作Java
// 获取 BufferedReader（可传给需要 Reader 的 API）
try (BufferedReader br = Files.newBufferedReader(file)) {
    br.lines().forEach(System.out::println);
}

// 获取 BufferedWriter
try (BufferedWriter bw = Files.newBufferedWriter(file)) {
    bw.write("content");
}

// 获取 InputStream（处理二进制）
try (InputStream is = Files.newInputStream(file)) {
    // 传给 ImageIO.read(is) / ObjectInputStream 等
}

目录与文件管理

创建、复制、移动、删除Java
Path src  = Path.of("/tmp/a.txt");
Path dest = Path.of("/tmp/backup/a.txt");

// 创建单层目录
Files.createDirectory(Path.of("/tmp/newdir"));

// 创建多层目录（类似 mkdir -p）
Files.createDirectories(Path.of("/tmp/a/b/c"));

// 复制文件（目标存在时默认抛异常）
Files.copy(src, dest);

// 复制并覆盖目标文件
Files.copy(src, dest, StandardCopyOption.REPLACE_EXISTING);

// 移动文件（原子性取决于文件系统）
Files.move(src, dest, StandardCopyOption.ATOMIC_MOVE);

// 删除（文件不存在时抛 NoSuchFileException）
Files.delete(src);

// 安静删除（不存在也不报错，返回 boolean）
Files.deleteIfExists(src);

// 创建临时文件（系统临时目录）
Path tmp = Files.createTempFile("prefix-", ".tmp");
ATOMIC_MOVE 的本质：在同一文件系统内，移动操作只是修改目录项指针，内核保证原子性。跨文件系统时会退化为复制+删除，AtomicMoveNotSupportedException 会被抛出。这是安全地替换配置文件（先写临时文件，再原子移动）的标准范式。

状态检测与文件属性

Files 状态检测Java
Path p = Path.of("/tmp/data.csv");

Files.exists(p);                  // 存在？
Files.notExists(p);               // 注意：!exists ≠ notExists（文件系统不可达时两者都 false）
Files.isDirectory(p);
Files.isRegularFile(p);
Files.isSymbolicLink(p);
Files.isReadable(p);
Files.isHidden(p);

// 文件大小（字节）
long size = Files.size(p);

// 最后修改时间
FileTime modified = Files.getLastModifiedTime(p);

// 一次性读取所有 POSIX 属性（Linux/macOS）
PosixFileAttributes attrs = Files.readAttributes(
    p, PosixFileAttributes.class);
System.out.println(attrs.owner().getName());
System.out.println(PosixFilePermissions.toString(attrs.permissions()));  // rwxr-xr-x

目录遍历：list / walk / find

NIO.2 提供了三个层次的目录遍历，都返回 Stream<Path>，用完必须关闭（try-with-resources）。

方法深度典型用途
Files.list(dir)仅当前层列出目录条目（不递归）
Files.walk(dir, depth)指定深度递归遍历所有文件
Files.find(dir, depth, BiPredicate)指定深度带条件的递归搜索
三种遍历方式示例Java
Path root = Path.of("/project/src");

// 1. list：只列当前目录，类似 ls
try (Stream<Path> entries = Files.list(root)) {
    entries.filter(Files::isDirectory)
           .forEach(System.out::println);
}

// 2. walk：深度优先递归，找出所有 .java 文件
try (Stream<Path> all = Files.walk(root)) {
    long count = all
        .filter(p -> p.toString().endsWith(".java"))
        .count();
    System.out.println("共 " + count + " 个 Java 文件");
}

// 3. find：带属性条件搜索（比 walk + filter 更高效，属性已预取）
try (Stream<Path> found = Files.find(root, 10,
        (p, attr) -> attr.isRegularFile()
                  && attr.size() > 1024 * 1024)) {  // 大于 1MB
    found.forEach(System.out::println);
}
Stream 必须关闭！Files.walk() 等方法在底层持有目录流句柄（DirectoryStream），若不关闭会导致文件描述符泄漏。务必用 try-with-resources。

FileVisitor：递归删除 / 复制目录树

对于"删除非空目录"这类需要在遍历的同时执行操作的场景，FileVisitor 是最合适的工具。SimpleFileVisitor 是它的默认空实现，按需覆盖即可。

递归删除目录（FileVisitor）Java
public static void deleteDirectory(Path dir) throws IOException {
    Files.walkFileTree(dir, new SimpleFileVisitor<Path>() {

        @Override
        public FileVisitResult visitFile(Path file, BasicFileAttributes attrs)
                throws IOException {
            Files.delete(file);
            return FileVisitResult.CONTINUE;
        }

        @Override
        public FileVisitResult postVisitDirectory(Path dir, IOException exc)
                throws IOException {
            // 子文件都删完了，再删目录本身
            Files.delete(dir);
            return FileVisitResult.CONTINUE;
        }
    });
}

WatchService：监听目录变更事件

NIO.2 提供了操作系统级的目录监听机制（Linux 的 inotify、macOS 的 kqueue、Windows 的 ReadDirectoryChangesW），比定时轮询高效得多。

监听目录文件变化Java
Path watchDir = Path.of("/tmp/watch-me");

try (WatchService watcher = FileSystems.getDefault().newWatchService()) {

    watchDir.register(watcher,
        StandardWatchEventKinds.ENTRY_CREATE,
        StandardWatchEventKinds.ENTRY_DELETE,
        StandardWatchEventKinds.ENTRY_MODIFY);

    while (true) {
        WatchKey key = watcher.take();  // 阻塞等待事件
        for (WatchEvent<?> event : key.pollEvents()) {
            System.out.println(event.kind() + ": " + event.context());
        }
        boolean valid = key.reset();  // 必须 reset，否则不再收到事件
        if (!valid) break;           // 目录被删除，退出
    }
}

NIO.2 vs 旧 java.io.File：全面对比

能力java.io.FileNIO.2（Files + Path）
失败处理返回 boolean，原因不明抛具体 IOException 子类
符号链接不支持完整支持（toRealPath 等）
文件元数据仅 lastModified / lengthPOSIX 权限、所有者、创建时间等
路径操作字符串拼接resolve / relativize / normalize
目录遍历listFiles()，需手写递归walk / find / list，返回 Stream
原子移动不支持ATOMIC_MOVE
目录监听无WatchService（OS 级事件）
跨文件系统仅本地可插拔 FileSystem（ZIP、内存等）

常见陷阱与面试考点

陷阱 1：exists 与 notExists 不是互反

当文件系统不可达（权限拒绝、网络挂载断开）时，Files.exists(p) 返回 false，Files.notExists(p) 也返回 false。两个方法都无法确定，应捕获异常而非依赖返回值。

陷阱 2：Stream 未关闭导致文件描述符泄漏

Files.walk()、Files.list()、Files.lines() 返回的 Stream 底层持有 OS 句柄，必须用 try-with-resources 关闭，或显式调用 stream.close()。

陷阱 3：resolve 传入绝对路径会覆盖前缀

resolve 绝对路径陷阱Java
Path base = Path.of("/home/user");

// 预期：/home/user/docs —— 实际也是这个（相对路径正常）
base.resolve("docs");  // /home/user/docs ✓

// 危险：参数是绝对路径，base 被完全忽略！
base.resolve("/etc/passwd");  // /etc/passwd ← 不是你想要的

陷阱 4：copy 不递归，delete 不能删非空目录

Files.copy() 只复制单个文件或空目录（不递归），Files.delete() 只能删除空目录。需要递归操作时使用 walkFileTree。

面试高频考点

  • Path 是接口还是类？接口，具体实现由各平台的 FileSystem 提供（如 UnixPath）。
  • Files.lines() 为什么比 readAllLines() 好？前者是惰性 Stream，后者全量加载到内存。大文件用前者。
  • 如何原子地更新配置文件？先写临时文件，再用 Files.move(tmp, target, ATOMIC_MOVE)。
  • normalize() 和 toRealPath() 的区别？normalize() 纯字符串操作，不访问文件系统；toRealPath() 访问文件系统、跟踪符号链接，文件不存在时抛异常。
  • WatchService 的事件丢失问题？事件队列满时会产生 OVERFLOW 事件，高频变更场景需自行节流。

综合实战：安全的配置文件更新

先写临时文件，再原子移动——避免在写到一半时程序崩溃导致配置损坏。

原子写入配置文件Java
public static void saveConfig(Path configFile, String content)
        throws IOException {

    // 1. 在同一目录下创建临时文件（保证同一文件系统，支持 ATOMIC_MOVE）
    Path dir = configFile.getParent();
    Path tmp = Files.createTempFile(dir, ".tmp-", ".cfg");

    try {
        // 2. 写入临时文件
        Files.writeString(tmp, content, StandardCharsets.UTF_8);

        // 3. 原子替换目标文件（目标已存在也没问题）
        Files.move(tmp, configFile,
            StandardCopyOption.REPLACE_EXISTING,
            StandardCopyOption.ATOMIC_MOVE);

    } catch (IOException e) {
        // 写入失败：清理临时文件，不影响原配置
        Files.deleteIfExists(tmp);
        throw e;
    }
}
知识连接：NIO.2 的 FileSystem 是可插拔的抽象——FileSystems.newFileSystem(zipPath, ...) 可以把 ZIP 文件当目录操作，Path、Files 的所有 API 都能无缝用于 ZIP 内部文件，这正是 Java 模块系统（JPMS）内部读取 JAR 资源的底层机制。
