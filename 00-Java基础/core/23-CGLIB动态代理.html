23. CGLIB 动态代理（vs JDK 动态代理）

JDK 代理要求目标类实现接口；CGLIB 通过继承生成子类绕过这一限制——这正是 Spring AOP 能增强任意 Bean 的底层秘密。

1. JDK 代理的局限 → CGLIB 的诞生动机

上一节我们看到，JDK 动态代理的核心是 java.lang.reflect.Proxy，它生成的代理类必须实现至少一个接口。原因很简单：Proxy 生成的字节码继承自 Proxy 类，Java 不支持多重继承，所以只能通过接口建立"契约"。

但现实中大量的类根本没有接口——尤其是 Spring 管理的 Service、Controller。为了给这些类织入切面逻辑，Spring 引入了 CGLIB（Code Generation Library），它的思路是：不依赖接口，而是在运行时生成目标类的子类，在子类中拦截父类方法。

一句话概括区别：JDK 代理是"基于接口的委托"，CGLIB 是"基于继承的子类化"。两种方式都不修改原始字节码，都在运行时生成新类。

2. CGLIB 的核心原理：ASM + 字节码继承

CGLIB 底层直接调用 ASM（一个极低层的字节码操作库）生成 .class 字节码并加载到 JVM。整个调用链如下：

调用者
  └─ ProxyBean$$EnhancerByCGLIB$$xxxxxx ← 运行期生成的子类
       └─ MethodInterceptor.intercept() ← 你写的拦截逻辑
            └─ invokeSuper() ← 调用父类（真实目标）的方法

生成的子类会对每个非 final、非 private 的方法生成两份实现：一份走 MethodInterceptor 拦截链，一份通过 FastClass 直接调用父类原始逻辑（避免反射开销）。

FastClass 是什么？

CGLIB 自己发明了 FastClass 机制：为目标类的每个方法预先生成一个"索引 → 方法体"的分发器，调用时用 int index 查表而非 Method.invoke()，省去了反射的安全检查开销，这是 CGLIB 在某些场景下比 JDK 代理更快的原因。

3. 最小可运行示例

先引入依赖（Spring 已内嵌 CGLIB，独立使用时单独引）：

pom.xmlMaven
<dependency>
    <groupId>cglib</groupId>
    <artifactId>cglib</artifactId>
    <version>3.3.0</version>
</dependency>

目标类（无接口）

OrderService.javaJava
public class OrderService {

    public String createOrder(String item) {
        System.out.println("[真实方法] 创建订单: " + item);
        return "ORDER-" + item;
    }

    // final 方法：CGLIB 无法拦截，子类覆盖不了父类的 final 方法
    public final void cancelOrder() {
        System.out.println("[真实方法] 取消订单");
    }
}

MethodInterceptor 实现

LogInterceptor.javaJava
import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;
import java.lang.reflect.Method;

public class LogInterceptor implements MethodInterceptor {

    /**
     * @param obj    CGLIB 生成的子类实例（即代理对象本身）
     * @param method 被拦截的方法（反射 Method 对象）
     * @param args   方法参数
     * @param proxy  CGLIB 的 MethodProxy，可绕过反射直接调父类方法
     */
    @Override
    public Object intercept(Object obj, Method method,
                             Object[] args, MethodProxy proxy) throws Throwable {

        System.out.println("[前置增强] 方法: " + method.getName());
        long start = System.currentTimeMillis();

        // 关键：用 invokeSuper 而非 method.invoke(obj, args)
        // invokeSuper 走 FastClass 路径，不会触发二次代理，也无反射开销
        Object result = proxy.invokeSuper(obj, args);

        System.out.println("[后置增强] 耗时: " + (System.currentTimeMillis() - start) + "ms");
        return result;
    }
}

创建代理并使用

CglibDemo.javaJava
import net.sf.cglib.proxy.Enhancer;

public class CglibDemo {
    public static void main(String[] args) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(OrderService.class);  // 设置父类（目标类）
        enhancer.setCallback(new LogInterceptor()); // 设置拦截器

        // 生成子类并实例化；CGLIB 调用父类无参构造器
        OrderService proxy = (OrderService) enhancer.create();

        proxy.createOrder("iPhone");  // 会被拦截
        proxy.cancelOrder();            // final 方法，不会被拦截！
    }
}

运行输出：

输出
[前置增强] 方法: createOrder
[真实方法] 创建订单: iPhone
[后置增强] 耗时: 1ms
[真实方法] 取消订单

4. invokeSuper vs invoke：一个致命的易错点

在 MethodInterceptor.intercept() 中，你有两种方式调用目标方法：

对比Java
// ✅ 正确：invokeSuper 直接调父类字节码，不再经过代理拦截
proxy.invokeSuper(obj, args);

// ❌ 危险：method.invoke(obj, args) 中 obj 是代理对象本身
// 代理方法再调代理方法 → StackOverflowError（无限递归）
method.invoke(obj, args);
陷阱：在 CGLIB 的 intercept 方法里，obj 是代理子类实例，不是原始目标对象。如果用 method.invoke(obj, args)，等于在代理上再次触发拦截，造成无限递归栈溢出。务必使用 proxy.invokeSuper(obj, args)。

5. CGLIB vs JDK 动态代理：全维度对比

维度 JDK 动态代理 CGLIB 动态代理
依赖 JDK 内置，无需额外依赖 需要 cglib.jar（Spring 已内嵌）
目标类要求 必须实现接口 无需接口，任意类均可
代理机制 实现同一接口（委托） 继承目标类（子类化）
final 类/方法 不受影响（接口方法不能是 final） 无法代理 final 类或 final 方法
调用目标方法 method.invoke(target, args)（反射） proxy.invokeSuper()（FastClass，更快）
代理对象类型 接口类型，不是原始类的实例 原始类的子类实例（instanceof 为 true）
构造器要求 无要求 目标类需有可访问的无参构造器（默认）
生成类数量 1 个代理类 代理子类 + 2 个 FastClass（目标类一个、代理类一个）
Spring 默认策略 目标类有接口时使用（Spring Boot 2.x 前） Spring Boot 2.x 起默认使用 CGLIB
Spring Boot 2.x 的重要变化：从 Spring Boot 2.0 开始，spring.aop.proxy-target-class 默认为 true，即即使目标类有接口也优先用 CGLIB。原因是团队踩了太多"同一个 Bean 在有接口和无接口时行为不一致"的坑。

6. 生成的子类长什么样

可以用 -Dcglib.debugLocation=/tmp/cglib JVM 参数把 CGLIB 生成的 .class 文件落盘，用 javap 反编译后大致结构如下：

反编译伪代码（示意）Java
public class OrderService$$EnhancerByCGLIB$$a1b2c3
        extends OrderService            // 继承目标类
        implements Factory {           // CGLIB 内部接口

    private MethodInterceptor CGLIB$CALLBACK_0;

    @Override
    public String createOrder(String item) {
        MethodInterceptor cb = this.CGLIB$CALLBACK_0;
        if (cb != null) {
            // 走拦截链
            return (String) cb.intercept(this, CGLIB$createOrder$0$Method,
                                         new Object[] {item}, CGLIB$createOrder$0$Proxy);
        }
        // 无拦截器时直接调父类
        return super.createOrder(item);
    }

    // final cancelOrder() 不会被 override，直接继承父类实现
}

7. 与 Spring AOP 的关系

Spring AOP 并不直接暴露 CGLIB API，而是在 ProxyFactory 内部做了封装：

@Transactional / @Aspect
  └─ Spring 解析切点，构建 Advisor 链
      └─ ProxyFactory.getProxy()
          ├─ 有接口 且 proxy-target-class=false → JdkDynamicAopProxy
          └─ 无接口 或 proxy-target-class=true → ObjenesisCglibAopProxy

ObjenesisCglibAopProxy 是 Spring 对 CGLIB 的包装，额外集成了 Objenesis——一个可以不调用构造器直接实例化对象的库，解决了"目标类没有无参构造器时 CGLIB 崩溃"的问题。

知识联系：@Transactional 失效的经典场景——在同一个类内部方法 A 调方法 B（B 上有 @Transactional），事务不生效。本质原因是内部调用走的是 this.B()，绕过了 CGLIB 生成的代理子类，直接执行父类原始方法，拦截器从未介入。

8. 常见陷阱汇总

  • final 类/方法无法被代理：CGLIB 子类无法 override final 方法，也无法继承 final 类。若目标类或方法被声明 final，AOP 切面静默失效——没有报错，只是增强不生效，极难排查。
  • 内部调用穿透代理：同类中 this.method() 绕过代理，导致 @Transactional、@Cacheable 等注解失效。解决方案：注入自身代理（@Autowired ApplicationContext 取自身），或改用 AspectJ 编译期织入。
  • 构造器被调用两次：早期版本 CGLIB 创建代理时会调用目标类构造器，若构造器有副作用（如初始化数据库连接）则出问题。Spring 引入 Objenesis 后此问题已消除。
  • 代理对象不能强转为 final 字段类型：若目标类实现了接口，而你把代理对象赋给接口类型变量没有问题，但若声明为具体类类型且用 JDK 代理，运行时 ClassCastException。
  • 序列化问题：CGLIB 生成的子类类名含 $$，若把代理对象序列化传输，反序列化端没有该类则失败。不要序列化 Spring Bean。

9. 性能：CGLIB 一定比 JDK 代理快吗？

这是个常见误解。现实情况更复杂：

  • 代理对象创建：CGLIB 生成字节码 + 加载类，首次创建显著慢于 JDK 代理（JDK 代理只需一次 Proxy.newProxyInstance）。CGLIB 会缓存生成的类，后续创建同类代理会快很多。
  • 方法调用：JDK 21 之前，CGLIB 的 FastClass 确实比反射 Method.invoke() 快；但 JDK 17+ 的 JIT 对反射的优化已极为激进，实际差距在微基准测试中几乎可以忽略。
  • 内存占用：CGLIB 生成 3 个类（代理类 + 2 个 FastClass），JDK 代理只生成 1 个，CGLIB 在类数量多时 Metaspace 压力更大。
结论：不要因为"CGLIB 性能好"而刻意选择它。在 Spring 体系下跟随框架默认策略即可；自行选择时，有接口优先 JDK 代理（更符合 OOP 设计），无接口才用 CGLIB。

10. 面试高频考点

Q1：Spring 什么时候用 JDK 代理，什么时候用 CGLIB？

Spring Boot 2.x 之后，默认 proxy-target-class=true，始终使用 CGLIB。若设置为 false，则：有接口用 JDK，无接口用 CGLIB。可通过 @EnableAspectJAutoProxy(proxyTargetClass=false) 切换。

Q2：CGLIB 代理为什么不能代理 final 方法？

CGLIB 通过继承生成子类并 override 方法来插入拦截逻辑。Java 中 final 方法禁止被子类覆盖，编译器和 JVM 都会强制执行这一约束，所以 CGLIB 根本无法在子类中生成 override 版本，该方法就会直接继承父类实现，拦截器永远不会被调用。

Q3：@Transactional 为什么在同类内部调用时失效？

因为 Spring 的事务是通过 AOP 代理实现的，代理对象包了一层拦截逻辑。类内部 this.method() 中的 this 指向原始对象，不是代理对象，调用链直接进入目标方法，拦截器从未介入，事务边界自然不存在。

Q4：MethodProxy.invokeSuper 和 Method.invoke 的区别？

invokeSuper 通过 FastClass（预生成的索引分发器）直接调父类字节码，不走反射，不会触发二次代理。method.invoke(obj, args) 若 obj 是代理对象，则触发再次拦截，导致无限递归栈溢出。

Q5：Objenesis 解决了什么问题？

早期 CGLIB 需要调用目标类的无参构造器来实例化代理对象。一旦目标类没有无参构造器，代理创建失败。Objenesis 通过 JVM 底层 API（sun.misc.Unsafe 或 ReflectionFactory）绕过构造器直接分配内存，让 Spring AOP 可以代理任意类。
