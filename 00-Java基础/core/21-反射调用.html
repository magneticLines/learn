21. 反射调用（Field / Method / Constructor）

在运行时拆解一个类，读写任意字段、调用任意方法、创建任意对象——反射是框架的基石，也是理解 Java 元编程的第一道门。

1. 为什么需要反射？它解决了什么问题

Java 是静态语言：编译期就必须确定调用哪个类、哪个方法。这对普通业务代码足够了，但写框架时却遇到困境—— 框架不知道用户会写什么类，却要在运行时帮他们创建对象、注入依赖、扫描注解。

反射（Reflection）打破了这一限制：只要有一个 Class 对象，就能在运行时检视结构、动态调用， 把"你要操作哪个类"从编译期推迟到运行期决定。

类比：静态调用就像查纸质通讯录——打印时已经定好每个号码；反射就像实时搜索联系人——运行时才知道拨给谁。

Spring 的 @Autowired 注入、MyBatis 的结果集映射、Jackson 的 JSON 序列化，背后全是反射在驱动。

2. 核心 API 全景

反射的入口是上一节学过的 Class 对象，三类操作对应三个核心类型：

java.lang.Class
├── getDeclaredField(name) → java.lang.reflect.Field
├── getDeclaredMethod(name, ...) → java.lang.reflect.Method
└── getDeclaredConstructor(...) → java.lang.reflect.Constructor<T>
操作目标 获取方法（本类+私有） 获取方法（含继承，仅public）
字段 getDeclaredField(s) getField(s)
方法 getDeclaredMethod(s) getMethod(s)
构造器 getDeclaredConstructor(s) getConstructor(s)
记住这条规则：getDeclared* = 本类声明的所有成员（含 private），但不含父类； get*（不带 Declared）= 含父类继承，但仅 public。 访问 private 成员前必须先调 setAccessible(true)，否则抛 IllegalAccessException。

3. Field —— 读写字段

3.1 基本用法

JavaField 读写示例
public class Person {
    private String name = "Alice";
    private int age  = 30;
}

// --- 调用方 ---
Person p = new Person();

// 1. 获取 Field 对象（getDeclaredField 可取 private 字段）
Field nameField = Person.class.getDeclaredField("name");

// 2. 解除私有保护（不调这行会抛 IllegalAccessException）
nameField.setAccessible(true);

// 3. 读取值
String val = (String) nameField.get(p);   // "Alice"

// 4. 修改值
nameField.set(p, "Bob");

// 5. 基本类型有专属方法，避免装箱
Field ageField = Person.class.getDeclaredField("age");
ageField.setAccessible(true);
int age = ageField.getInt(p);    // 比 (int) get(p) 少一次拆箱
ageField.setInt(p, 25);

3.2 遍历所有字段（含父类）

getDeclaredFields() 只返回本类声明的字段，若要遍历整条继承链，需要沿 getSuperclass() 向上爬。 这正是 Jackson、MyBatis 等框架在序列化/映射时的做法。

Java遍历继承链上的所有字段
static List<Field> getAllFields(Class<?> clazz) {
    List<Field> result = new ArrayList<>();
    while (clazz != null) {
        Collections.addAll(result, clazz.getDeclaredFields());
        clazz = clazz.getSuperclass();  // 到 Object 再往上就是 null，循环结束
    }
    return result;
}

4. Method —— 动态调用方法

4.1 基本用法

JavaMethod 调用示例
public class Calculator {
    public int add(int a, int b) { return a + b; }
    private static String secret() { return "42"; }
}

Calculator calc = new Calculator();

// getMethod：含继承链，只能拿 public 方法
// 第二参数是参数类型的 Class 数组（用于重载区分）
Method addMethod = Calculator.class.getMethod("add", int.class, int.class);
int sum = (int) addMethod.invoke(calc, 3, 5);  // 结果 8

// getDeclaredMethod：取 private static 方法
Method secretMethod = Calculator.class.getDeclaredMethod("secret");
secretMethod.setAccessible(true);
// 静态方法 invoke 第一个参数传 null（无实例）
String s = (String) secretMethod.invoke(null);  // "42"

4.2 invoke 的异常包装机制

invoke 声明抛 InvocationTargetException，这是一个包装异常—— 目标方法本身抛出的异常会被装入 cause。 这是框架调用用户代码时的重要细节：不能直接 catch Exception 了事，要先 unwrap。

Java正确处理 InvocationTargetException
try {
    method.invoke(target, args);
} catch (InvocationTargetException e) {
    // 取出真正的业务异常
    Throwable cause = e.getCause();
    if (cause instanceof RuntimeException) throw (RuntimeException) cause;
    throw new RuntimeException(cause);
}

5. Constructor —— 动态创建对象

5.1 newInstance 对比

有两种方式动态创建对象，它们的区别很关键：

方式API能调用私有构造器？能传参数？
Class 上的旧方法 clazz.newInstance() 否（已废弃 @Deprecated） 否（只能无参）
Constructor 方式 constructor.newInstance(args) 是（setAccessible 后） 是（任意参数）
JavaConstructor 示例
public class Config {
    private final String host;
    private final int    port;

    // 只有一个带参的私有构造器（单例或工厂模式常见）
    private Config(String host, int port) {
        this.host = host; this.port = port;
    }
}

// 1. 拿到私有构造器（参数类型要精确匹配）
Constructor<Config> ctor = Config.class
        .getDeclaredConstructor(String.class, int.class);

// 2. 解除访问限制
ctor.setAccessible(true);

// 3. 创建实例（等价于 new Config("localhost", 8080)）
Config cfg = ctor.newInstance("localhost", 8080);
为什么 Class.newInstance() 被废弃？ 它会把构造器内部的 checked exception 悄悄变成 unchecked 抛出，绕过编译器检查，非常危险。Constructor.newInstance() 会正常包装进 InvocationTargetException，行为可预期。

6. 综合实战：实现一个 mini 依赖注入容器

把 Field + Method + Constructor 串起来，用 80 行代码实现 Spring @Autowired 的核心逻辑：

JavaMini DI 容器（展示反射三件套协同工作）
// 自定义注解，标记需要注入的字段
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Inject {}

// 容器
public class SimpleContainer {
    private final Map<Class<?>, Object> registry = new HashMap<>();

    // 注册 Bean：用 Constructor 反射创建实例
    public <T> T register(Class<T> clazz) throws Exception {
        Constructor<T> ctor = clazz.getDeclaredConstructor();
        ctor.setAccessible(true);
        T instance = ctor.newInstance();
        registry.put(clazz, instance);
        return instance;
    }

    // 依赖注入：用 Field 反射找到 @Inject 字段并填值
    public void inject(Object target) throws Exception {
        for (Field f : target.getClass().getDeclaredFields()) {
            if (!f.isAnnotationPresent(Inject.class)) continue;
            Object dep = registry.get(f.getType());  // 按类型查依赖
            if (dep == null) throw new RuntimeException("No bean for " + f.getType());
            f.setAccessible(true);
            f.set(target, dep);
        }
    }
}

// 使用示例
class UserService {
    @Inject UserRepository repo;  // 字段不用 public
    public String find(long id) { return repo.findById(id).getName(); }
}

SimpleContainer c = new SimpleContainer();
c.register(UserRepository.class);
UserService svc = c.register(UserService.class);
c.inject(svc);   // repo 字段被自动填充

7. 性能：反射慢在哪里，怎么优化

反射调用比直接调用慢，但慢的主要原因不是"反射本身"，而是安全检查和没有缓存。

慢的原因优化手段
每次 getMethod 触发字符串搜索 缓存 Method/Field/Constructor 对象，只查一次
未调 setAccessible(true) 时每次 invoke 都做权限检查 提前调用 setAccessible(true)，后续 invoke 跳过检查
基本类型参数装箱 尽量用 getInt / setInt 等专属方法
高频调用场景（如序列化） 用 MethodHandle（JDK 7+）或代码生成（ASM/ByteBuddy）彻底避开反射
数量级参考：直接调用 ≈ 1 ns；缓存了 Method 且 setAccessible 后的反射 ≈ 10~50 ns； 每次重新 getMethod 的反射 ≈ 几百 ns 到 μs 级。 框架启动时初始化一次，运行时命中缓存，性能完全可接受。

8. 与 MethodHandle 的关系（JDK 7+）

java.lang.invoke.MethodHandle 是反射的现代替代品。它们的核心差异：

反射 Method.invokeMethodHandle.invoke
类型安全 运行时检查，返回 Object 编译期签名，JVM 可内联优化
性能 有装箱开销，较慢 JIT 可将其优化为直接调用
适用场景 框架扫描、通用工具 高频调用、Lambda 实现底层
访问控制 setAccessible 强制绕过 Lookup 对象控制，更安全
Java 的 Lambda 底层就是用 invokedynamic + MethodHandle 实现的，比反射快得多。 如果你在框架中高频调用某个方法，可以考虑用 MethodHandles.lookup().unreflect(method) 把 Method 转成 MethodHandle 后缓存使用。

9. 常见陷阱与面试考点

陷阱一：忘记区分 int.class 和 Integer.class

getMethod("add", Integer.class) 找不到参数类型是 int 的方法，会抛 NoSuchMethodException。基本类型和包装类型在反射中是不同的 Class 对象。

陷阱二：修改 final 字段

setAccessible(true) 后对 final 字段调用 set() 在 JDK 8 及之前可能成功， 但从 JDK 9 开始模块系统加强了限制，且即便成功，JIT 可能已经将 final 字段的值内联了—— 你改了字段，读到的还是旧值，极难排查。生产代码中严禁这样做。

陷阱三：getMethod 时传错参数类型顺序

Java反例
// 目标方法：void copy(String src, String dest)
// 错误：顺序写反了，getMethod 成功但 invoke 语义完全错误
Method m = clazz.getMethod("copy", String.class, String.class);
// invoke 时参数顺序同样可能写反，编译器无法帮你检查
m.invoke(target, dest, src);  // 逻辑 bug，不会报错

陷阱四：反射绕过泛型擦除后的类型检查

List<String> 经过泛型擦除变成 List，通过反射可以向其中插入 Integer——编译器不报错，运行时取出元素时才会抛 ClassCastException， 堆栈信息非常难以定位。反射操作泛型集合时要格外小心。

面试高频考点总结

  • getDeclared* vs get* 的区别：本类所有成员 vs 含继承链的 public 成员
  • setAccessible(true) 的作用：跳过 JVM 的访问权限检查（非破解字节码，只影响当前 JVM 调用栈）
  • invoke 的返回值和异常：返回 Object（void 方法返回 null），目标异常包装在 InvocationTargetException.getCause()
  • 反射为什么慢：安全检查 + 方法查找 + 装箱，三种优化手段：缓存、setAccessible、MethodHandle
  • Spring 用反射做什么：Bean 实例化（Constructor）、字段注入（Field）、初始化方法调用（Method）
  • 反射能破坏单例吗：能，通过 getDeclaredConstructor + setAccessible 调私有构造器。防御手段：构造器中判断实例是否已存在并抛异常，或用枚举单例（反射无法实例化枚举）
