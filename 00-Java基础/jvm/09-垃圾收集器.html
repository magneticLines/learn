JVM · 垃圾回收

09. 垃圾收集器

Serial · Parallel · CMS · G1 · ZGC —— 从单线程到亚毫秒停顿的演进史

为什么需要多种垃圾收集器？

上一篇讲了 GC 算法（标记清除 / 复制 / 标记整理），它们是理论蓝图； 垃圾收集器是算法的工程实现，需要在三个相互冲突的目标之间取舍：

  • 吞吐量（Throughput）：单位时间内 GC 之外的有效计算占比，批处理任务追求这个
  • 停顿时间（Latency / Pause）：单次 STW（Stop-The-World）的最长时间，在线服务追求这个
  • 内存占用（Footprint）：GC 本身的额外内存消耗，嵌入式 / 资源受限场景追求这个

这三者构成 GC 的"不可能三角"——不同收集器选择了不同的优先级，这就是为什么 JDK 要提供一个"工具箱"而不是唯一答案。

STW 是什么？
Stop-The-World：JVM 暂停所有用户线程，只跑 GC 线程。就像超市关门盘点库存，盘点期间顾客不能进出。 几乎所有收集器都有 STW，差别在于哪个阶段 STW、STW 多久。

收集器演进全景

JDK 1.3 Serial 单线程，Client VM 默认
JDK 1.4 Parallel Scavenge 多线程新生代，吞吐量优先
ParNew 多线程新生代，配合 CMS
JDK 1.4 CMS 并发标记清除，低停顿先驱
JDK 7u4 G1 Region 化堆，平衡吞吐+停顿
JDK 11 ZGC 着色指针，亚毫秒停顿
JDK 12 Shenandoah Red Hat 贡献，同类低停顿

JDK 9 起 CMS 被标记为 deprecated，JDK 14 正式移除。JDK 9 起 G1 成为默认收集器。

Serial 收集器

设计思路

单线程 GC，回收时全程 STW。听起来很落后，但它有存在价值： 单核 CPU（容器 / 树莓派 / 嵌入式）上没有线程切换开销，实际表现并不差； 内存占比极低（几 MB），适合堆不超过 100 MB 的场景。

  • 新生代：复制算法（Serial 本体）
  • 老年代：标记整理算法（Serial Old，配套使用）
启用参数：-XX:+UseSerialGC

Parallel（吞吐量优先）收集器

设计思路

多线程版 Serial。GC 线程数默认等于 CPU 核数（可用 -XX:ParallelGCThreads 调整）。 目标是最大化吞吐量，允许较长的单次停顿。适合批处理、数据分析等"不在意瞬时抖动"的场景。

  • Parallel Scavenge：新生代，复制算法，暴露 -XX:MaxGCPauseMillis 和 -XX:GCTimeRatio
  • Parallel Old：老年代，标记整理，JDK 6 引入，取代 Serial Old 成为 Parallel Scavenge 的最佳搭档
JVM 参数shell
# 吞吐量优先，允许最大停顿 200ms，GC 时间不超过总时间的 1/(1+19)=5%
-XX:+UseParallelGC
-XX:MaxGCPauseMillis=200
-XX:GCTimeRatio=19
陷阱：MaxGCPauseMillis 不是硬上限
设置越小，JVM 会把堆调小来"凑"停顿时间，但堆小了反而 GC 更频繁，吞吐量下降。 不要随手设成 -XX:MaxGCPauseMillis=1，那只会适得其反。

CMS（Concurrent Mark Sweep）收集器

设计思路：第一次真正并发

CMS 的核心思想是：把大部分标记工作移到与用户线程并发执行的阶段， STW 只保留必要的短暂时刻。这是 Java 低延迟 GC 的开山之作。

四个阶段

阶段是否 STW做什么
初始标记（Initial Mark）STW（极短）只标记 GC Roots 直接关联的对象
并发标记（Concurrent Mark）并发从 GC Roots 出发遍历整个对象图，与用户线程同时跑
重新标记（Remark）STW（稍长）修正并发标记期间用户线程引起的引用变化（增量更新算法）
并发清除（Concurrent Sweep）并发清除死亡对象，与用户线程同时跑

三大致命缺陷

  • 浮动垃圾（Floating Garbage）：并发清除阶段用户线程还在产生垃圾，这批垃圾只能等下次 GC，因此 CMS 必须在老年代未满时提前触发（默认 68%），否则触发 Concurrent Mode Failure，退化为 Serial Old 全停顿
  • 内存碎片：标记清除算法不整理内存，长期运行后大对象无处安置，触发 Full GC
  • CPU 敏感：并发阶段与用户线程抢 CPU，低核数机器吞吐量下降明显（默认占用 (CPU数+3)/4 个线程）
CMS 常用参数shell
-XX:+UseConcMarkSweepGC
# 老年代使用率达到 70% 时提前触发（默认 68%）
-XX:CMSInitiatingOccupancyFraction=70
# Full GC 后压缩内存，缓解碎片（但 Full GC 会更慢）
-XX:+UseCMSCompactAtFullCollection

G1（Garbage First）收集器

设计思路：打破"物理分代"

CMS 的问题本质是：老年代是一块连续大区域，碎片无法避免。 G1 的解法是把堆切成大小相等的 Region（默认 1~32 MB，共约 2048 个）， 每个 Region 在逻辑上可以扮演 Eden / Survivor / Old / Humongous（大对象） 任意角色， 随时可以换角色。

"Garbage First"这个名字来自策略：优先回收垃圾最多的 Region， 用最少的停顿时间换取最大的释放量——这也是它能提供"可预期停顿"的原因。

堆内存（例如 4GB）
┌──────────────────────────────────────────┐
│ E │ E │ S │ O │ O │ H │ ... │
│ │ │ │ │ │(大对象)│ │
└──────────────────────────────────────────┘
E=Eden S=Survivor O=Old H=Humongous
Region 之间不必物理相邻

四个阶段（混合 GC 模式）

阶段是否 STW说明
初始标记STW（短）标记 GC Roots，附着在 Minor GC 上执行
并发标记并发遍历全堆对象图，使用 SATB（原始快照）维护一致性
最终标记STW（短）处理 SATB 缓冲区，完成并发阶段遗留的标记
筛选回收STW计算各 Region 的回收价值，选出"性价比最高"的一批，复制存活对象，释放整个 Region

关键数据结构：RSet（Remembered Set）

因为 Region 之间存在跨 Region 引用（老年代 Region 指向年轻代 Region）， G1 为每个 Region 维护一个 RSet，记录"谁引用了我"。 回收某个 Region 时只需扫描它的 RSet，而不必扫描整个堆——这是 G1 能做到局部回收的基石。

陷阱：RSet 的内存开销
RSet 本身要占堆内存的 5%~20%。如果堆很小（<4 GB），G1 的额外开销可能比 Parallel GC 还重， JDK 9 起 G1 虽然是默认，但堆小于 1 GB 的容器场景反而要考虑换回 Parallel 或 Serial。
G1 常用参数shell
-XX:+UseG1GC
# 期望停顿目标（软目标，G1 会尽力但不保证）
-XX:MaxGCPauseMillis=200
# Region 大小，建议让 JVM 自动计算（2 的幂次，1-32MB）
-XX:G1HeapRegionSize=4m
# 触发并发标记的堆使用比例（默认 45%）
-XX:InitiatingHeapOccupancyPercent=45

ZGC（Z Garbage Collector）

设计思路：停顿时间与堆大小无关

G1 的停顿时间在大堆（几十 GB）下仍会达到数百毫秒。ZGC 的目标更激进： 无论堆多大，GC 停顿不超过 10ms（JDK 16 后通常 <1ms）。

ZGC 实现这个目标的核心武器是着色指针（Colored Pointer） + 读屏障（Load Barrier），把几乎所有工作都移到并发阶段， STW 只剩三次极短暂停（均为毫秒级甚至微秒级）。

着色指针：把元数据藏进指针里

64 位系统的指针实际只用 42~44 位表示地址，高位剩余比特被 ZGC 用来存储对象的 GC 元信息：

64 位指针：
[ 18位未用 | Finalizable(1) | Remapped(1) | Marked1(1) | Marked0(1) | 42位对象地址 ]

这 4 个颜色位 = 对象当前处于哪个 GC 阶段

好处：无需额外的标记位数组，颜色随指针走，并发阶段通过读屏障在访问对象时顺便检查和修正颜色， 完全不需要 STW 来统一更新所有引用。

ZGC 并发阶段

阶段STW？说明
Pause Mark StartSTW（<1ms）标记 GC Roots
Concurrent Mark并发遍历对象图，更新颜色位
Pause Mark EndSTW（<1ms）处理弱引用等收尾工作
Concurrent Relocate并发把存活对象复制到新 Region，读屏障负责转发旧地址访问
Pause Relocate StartSTW（<1ms）扫描 Roots 中指向被迁移对象的引用
ZGC vs Shenandoah
两者目标一致（亚毫秒停顿），但实现路径不同：ZGC 用着色指针，Shenandoah 用 Brooks Forwarding Pointer（每个对象头多一个转发指针）。 ZGC 是 Oracle 官方维护的，进入 JDK 主线；Shenandoah 是 Red Hat 贡献的，同样在 OpenJDK 主线。
ZGC 参数shell
# JDK 15 起 ZGC 正式生产可用（不再需要 -XX:+UnlockExperimentalVMOptions）
-XX:+UseZGC
# JDK 21 引入 Generational ZGC，分代后吞吐量进一步提升
-XX:+UseZGC -XX:+ZGenerational
# 同样支持 MaxGCPauseMillis（但 ZGC 本身就极短，通常无需设置）
-XX:MaxGCPauseMillis=5

五大收集器横向对比

收集器线程模型老年代算法STW 停顿适用场景
Serial单线程标记整理全程 STW单核 / 小堆 / 嵌入式
Parallel多线程标记整理全程 STW（短）批处理 / 高吞吐
CMS并发标记清除两次短 STW低延迟 Web（已废弃）
G1并发+并行复制（Region）可预期停顿大堆（4G+）通用
ZGC全并发复制<10ms（通常 <1ms）超大堆 / 极低延迟

实战选择指南

  • 堆 < 1 GB + 单核容器：-XX:+UseSerialGC，开销最小
  • 批处理 / 计算型任务：-XX:+UseParallelGC，最大化 CPU 有效使用率
  • 普通 Web 服务（JDK 9+）：G1 默认即可，酌情调 MaxGCPauseMillis=100~200
  • 超低延迟（P99 < 50ms）+ 大堆：ZGC（JDK 15+），尤其是 JDK 21 Generational ZGC
  • 老系统 JDK 8：G1 稳定可用；CMS 可用但别新起项目
监控先于调优
在改收集器之前，先用 -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m 采集 GC 日志，用 GCEasy / GCViewer 分析真实瓶颈，否则盲目换收集器可能适得其反。

代码示例：观察收集器行为

GcObserveDemo.javaJava
// 运行参数：-Xms64m -Xmx64m -XX:+UseG1GC -Xlog:gc*
public class GcObserveDemo {

    public static void main(String[] args) throws InterruptedException {
        List<byte[]> list = new ArrayList<>();

        for (int i = 0; i < 100; i++) {
            // 每次分配 1MB，模拟对象创建
            list.add(new byte[1024 * 1024]);

            if (i % 10 == 9) {
                // 每 10 次清除一半，模拟短命对象
                list.subList(0, list.size() / 2).clear();
            }
            Thread.sleep(50);
        }
    }
}

运行时切换 -XX:+UseSerialGC / -XX:+UseG1GC / -XX:+UseZGC， 观察 GC 日志里的 Pause 时长和 GC 频率，直觉上感受三者的差距。

常见陷阱 & 误区

陷阱 1：堆设得越大越好

堆越大，单次 Full GC 时 STW 越长（尤其 Serial / Parallel / CMS）。 比如 32 GB 堆用 CMS，一次 Remark 可能停顿 2~3 秒。G1 / ZGC 才真正做到了"大堆不怕"。

陷阱 2：CMS 的 Concurrent Mode Failure

并发清除阶段老年代满了，JVM 应急退化为 Serial Old 做一次完整的 STW Full GC， 停顿时间从几十毫秒突变为几秒。根本原因：CMSInitiatingOccupancyFraction 设太高， 或晋升速度超过并发清除速度。

陷阱 3：G1 的 Humongous 对象

大于 Region 一半的对象直接进 Humongous Region，由老年代管理， 不经历年轻代的短命淘汰。如果频繁分配大对象（如大 byte[]）， 会导致频繁触发并发标记周期。解法：增大 G1HeapRegionSize，或减少大对象分配。

陷阱 4：ZGC 在低版本 JDK 上是实验特性

JDK 11~14 的 ZGC 需要 -XX:+UnlockExperimentalVMOptions； JDK 15 起才正式 GA。用 JDK 11 的同学注意加实验标志，否则 JVM 拒绝启动。

面试高频考点

  1. CMS 为什么会产生浮动垃圾？如何解决？
    并发清除阶段用户线程仍在运行，新产生的垃圾无法在本次 GC 中清除。 解法：调低 CMSInitiatingOccupancyFraction，提前触发 GC；或直接换 G1。
  2. G1 和 CMS 的本质区别是什么？
    CMS 是老年代收集器（需配合 ParNew），不整理内存（标记清除），会碎片化； G1 是全堆收集器，基于 Region + 复制算法，无碎片，且能提供可预期停顿目标。
  3. ZGC 怎么做到停顿不超过 10ms 的？
    着色指针 + 读屏障，把对象图遍历、对象迁移全部移到并发阶段， STW 只剩三个极短的 Root 扫描点，且停顿时间与堆大小无关（O(GC Root) 而非 O(堆大小)）。
  4. 默认收集器是什么？各 JDK 版本有何变化？
    JDK 8：Parallel（服务端）/ Serial（客户端）；JDK 9+：G1；JDK 21 起推荐 ZGC（Generational）。
  5. RSet 是什么？为什么 G1 需要它？
    Remembered Set，记录"哪些 Region 的对象引用了我"。 G1 每次只回收部分 Region，如果没有 RSet 就必须扫描全堆才能判断一个对象是否存活， RSet 把这个代价降到 O(引用者 Region 数量)。
