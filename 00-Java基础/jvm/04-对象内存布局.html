JVM · 内存模型

对象内存布局
Header / Instance Data / Padding

深入堆内存，看清 JVM 在 new 一个对象时，到底在内存里放了什么，以及为什么要这样放。

为什么要研究对象内存布局？

对很多初学者而言，new Object() 只是"在堆上分配了一块内存"。 但当你需要优化内存占用、分析 GC 停顿、理解 synchronized 原理、或者看 JVM 源码时， 就必须清楚这块内存里面长什么样。

HotSpot JVM（OpenJDK 主流实现）规定：堆上每一个 Java 对象都由三部分组成：

┌──────────────────────────────────┐
│ Object Header（对象头） │ ← JVM 自己用，Java 代码不可见
│ Mark Word （4 或 8 字节） │
│ Klass Word （4 或 8 字节） │
│ [Array Length（仅数组，4 字节）]│
├──────────────────────────────────┤
│ Instance Data（实例数据） │ ← 你定义的所有字段
│ int age （4 字节） │
│ long salary （8 字节） │
│ Object ref （4 字节） │
│ … │
├──────────────────────────────────┤
│ Padding（对齐填充） │ ← 凑够 8 字节倍数（可能为 0）
└──────────────────────────────────┘

① Mark Word —— 对象的"身份证"

Mark Word 是对象头中最核心的部分，64 位 JVM 占 8 字节。 它是一个多路复用字段——同一块内存在不同状态下存放的内容完全不同，JVM 通过最低几位的标志位来区分当前状态：

对象状态 存储内容（64 bit） 标志位（低 2-3 bit）
无锁 hashCode（31 bit）+ 分代年龄（4 bit）+ 偏向模式 0 01
偏向锁 线程 ID（54 bit）+ Epoch + 分代年龄 01（偏向位=1）
轻量级锁 指向线程栈 Lock Record 的指针 00
重量级锁 指向 Monitor 对象的指针 10
GC 标记 空（转移期 forwarding pointer） 11
设计精妙之处：同一块 8 字节被"时分复用"。 对象未被锁定时，这里存 hashCode；加了 synchronized 之后，hashCode 被挤走， 等锁释放再存回来——这就是为什么在加锁之后才第一次调用 hashCode() 有特殊逻辑的根本原因。

为什么 hashCode 只有 31 bit？

HotSpot 默认用 Marsaglia xor-shift 算法生成 identity hashCode， 结果存入 Mark Word 的 31 位字段。一旦写入，这个值就永远不变（即使对象被 GC 移动地址也不变）， 这是 System.identityHashCode() 的语义保证。

② Klass Word —— 类型指针

Klass Word 指向方法区（Metaspace）中该类的 InstanceKlass 结构， JVM 靠它知道这个对象是哪个类的实例，从而找到虚方法表、字段描述等。

64 位 JVM 默认开启指针压缩（CompressedOops），Klass Word 从 8 字节压缩到 4 字节。 堆地址不超过 32 GB 时均可使用，节省约 30% 的对象头开销。

JVM 参数对照：
-XX:+UseCompressedOops（默认开启）：普通对象引用 4 字节
-XX:+UseCompressedClassPointers（默认开启）：Klass Word 4 字节
当堆 > 32 GB 时两者自动关闭，对象头膨胀，需重新评估内存规划。

③ Instance Data —— 字段的真实存储

Instance Data 存放你在类中声明的所有实例字段（静态字段在 Metaspace 中，不在这里）。 排列规则不是按源码顺序，HotSpot 会按字段类型做重新排序以减少 padding 浪费：

字段类型大小排列优先级
long / double8 字节最先
int / float4 字节其次
short / char2 字节再次
byte / boolean1 字节再次
引用（压缩后）4 字节最后

父类字段在子类字段之前排列，但为节省空间，子类的小字段可能插入父类的 padding 空洞中 （取决于 JVM 实现，HotSpot 有此优化，参数 -XX:+CompactFields 默认开启）。

④ Padding —— 对齐填充

HotSpot 要求每个对象的总大小必须是 8 字节的倍数（与 CPU Cache Line 和 GC 的 oop 压缩算法有关）。 如果 Header + Instance Data 之和不是 8 的倍数，JVM 在末尾补零填充。

这些 padding 字节没有任何语义，只是"凑数"，但它们真实占用内存。 一个只有 boolean 字段的类，Instance Data 只有 1 字节，却要补 7 字节 padding—— 这是很多人没想到的内存浪费源头。

完整示例：用 JOL 看实际布局

JOL（Java Object Layout）是 OpenJDK 官方工具库，可以在运行时打印对象的真实内存布局。

pom.xml 依赖
<dependency>
    <groupId>org.openjdk.jol</groupId>
    <artifactId>jol-core</artifactId>
    <version>0.17</version>
</dependency>
示例类Java
class Order {
    boolean paid;      // 1 字节，但会被提升或补 padding
    int     quantity;  // 4 字节
    long    amount;    // 8 字节，对齐要求最高
    String  note;      // 引用，压缩后 4 字节
}
JOL 打印布局Java
import org.openjdk.jol.info.ClassLayout;

public class JolDemo {
    public static void main(String[] args) {
        System.out.println(
            ClassLayout.parseInstance(new Order()).toPrintable());
    }
}
典型输出（64 位 JVM + 指针压缩）
OFFSET  SIZE    TYPE DESCRIPTION
     0     4         (object header: mark)    ← Mark Word 低 4 字节
     4     4         (object header: mark)    ← Mark Word 高 4 字节
     8     4         (object header: class)   ← Klass Word（压缩后 4 字节）
    12     4   int  Order.quantity             ← 先排 4 字节字段
    16     8  long  Order.amount               ← 再排 8 字节字段
    24     1  boolean Order.paid               ← 1 字节字段
    25     3         (alignment/padding gap)   ← 补 3 字节，凑到 4 字节边界
    28     4   ref  Order.note                 ← 引用 4 字节
Instance size: 32 bytes
注意字段排序变了：源码顺序是 paid → quantity → amount → note， 但实际内存顺序是 quantity → amount → paid → note。 JVM 重排是为了让每个字段都自然对齐（long 必须在 8 的倍数偏移处），减少 padding 浪费。

数组对象的特殊结构

数组是特殊的对象，它在 Klass Word 之后多一个 4 字节的 Array Length 字段， 记录数组元素个数。因此一个 int[] 的对象头占 16 字节（Mark 8 + Klass 4 + Length 4）， 而普通对象头只有 12 字节（压缩指针开启时）。

类型对象头大小备注
普通对象（压缩指针）12 字节Mark 8 + Klass 4
普通对象（未压缩）16 字节Mark 8 + Klass 8
数组对象（压缩指针）16 字节Mark 8 + Klass 4 + Length 4
数组对象（未压缩）24 字节Mark 8 + Klass 8 + Length 4 + pad 4

最小对象是多大？

new Object()——没有任何实例字段： Mark Word 8 字节 + Klass Word 4 字节 = 12 字节， 不满 8 的倍数，padding 补 4 字节，最终 16 字节。

这告诉我们：哪怕一个空对象也要占 16 字节。 用大量细粒度对象（如 new Boolean(true)）会产生巨大内存开销， 这正是 JVM 用 Integer 缓存池（-128~127）和 Boolean.TRUE/FALSE 单例的原因。

与 synchronized 的关系

理解了 Mark Word，你就能真正搞清楚 synchronized 的锁升级过程：

  1. 无锁：Mark Word 存 hashCode + 分代年龄，标志位 01。
  2. 偏向锁：将当前线程 ID 写入 Mark Word，后续同一线程无 CAS 直接进入，极低开销。
  3. 轻量级锁：竞争出现，Mark Word 被替换成指向栈帧 Lock Record 的指针，CAS 抢占，自旋等待。
  4. 重量级锁：自旋超限，膨胀为操作系统 Mutex，Mark Word 指向堆上的 Monitor 对象。
陷阱：调用 hashCode() 会禁用偏向锁
偏向锁的 Mark Word 没有位置存 hashCode。 如果一个对象先调用了 System.identityHashCode()，再对它加锁， JVM 会直接跳过偏向锁，从轻量级锁开始。 反之，如果已经偏向，调用 hashCode 会撤销偏向锁，代价较大。

实战：减少内存占用的技巧

1. 字段声明顺序不影响布局，但字段类型组合影响 padding

差的写法（多 4 字节 padding）Java
class Bad {
    byte  a;   // offset 12, 1 byte
    // JVM 会在此插入 3 字节 padding，让 int 对齐到 4
    long  b;   // offset 16, 8 bytes
    byte  c;   // offset 24, 1 byte
    // 末尾 7 字节 padding
}  // 总大小: 32 字节

class Good {
    long  b;   // offset 12 → JVM 重排后 offset 12
    byte  a;
    byte  c;   // a 和 c 紧凑排在一起
    // 末尾 6 字节 padding
}  // 总大小: 24 字节（JVM 会自动重排，Good 和 Bad 实际相同）
HotSpot 会自动重排字段，所以手写顺序对最终布局影响不大。 真正影响内存的是字段数量和类型的组合，以及父子类继承层次。

2. 用基本类型替代包装类型

Integer 对象 = 16 字节对象头 + 4 字节 int 字段 = 16 字节（含 padding）。 而 int 基本类型只占 4 字节。 在数组或大量集合中，两者内存差距高达 4 倍。

3. 关注继承层次带来的 padding 叠加

父子类 padding 叠加示例Java
class Parent {
    int x;    // 12~15（4 字节）
              // 16 字节结束，恰好 8 的倍数，无需 padding
}

class Child extends Parent {
    long y;   // 必须对齐到 8 的倍数 → offset 16，OK
    byte z;   // offset 24，1 字节
              // 末尾补 7 字节 padding
}  // 总 32 字节

面试高频考点

  • new Object() 占多少字节？ 开启压缩指针时 16 字节（12 字节头 + 4 字节 padding），关闭时 16 字节（头本身就是 16 字节）。
  • Mark Word 里存了什么？ 要按锁状态分情况回答：无锁存 hashCode + 年龄，偏向锁存线程 ID，轻量级锁存 Lock Record 指针，重量级锁存 Monitor 指针。
  • 为什么对象大小必须是 8 的倍数？ 两个原因：CPU 缓存行对齐提升访问性能；压缩指针（CompressedOops）将 32 位 oop 左移 3 位来寻址，要求对象首地址按 8 字节对齐。
  • 调用 hashCode() 后还能用偏向锁吗？ 不能。Mark Word 中偏向锁没有存放 hashCode 的空间，计算过 identity hash 的对象会直接跳过偏向锁。
  • CompressedOops 最大支持多大堆？ 理论上 2³² × 8 = 32 GB，超过后自动关闭，对象引用回到 8 字节。
  • 静态字段在哪里？ 静态字段属于类本身，存放在 Metaspace 的 InstanceKlass 对应的 Java mirror（即 Class 对象）中，不在堆上的实例对象里。

小结

Java 对象在堆内存中由三部分构成：

  • Object Header：Mark Word（多路复用，存锁态/hashCode/GC 信息）+ Klass Word（类型指针）+ 可选的 Array Length。
  • Instance Data：所有实例字段，JVM 按类型重排以减少 padding，父类字段在前。
  • Padding：末尾凑 8 字节倍数的填充，无语义但真实占内存。

理解这个布局，是读懂 synchronized 锁升级、GC 移动对象、JVM 优化内存的基础。 推荐在自己机器上跑一跑 JOL，数字比文字更有说服力。

与其他知识点的联系

  • synchronized 与锁升级：Mark Word 是理解偏向锁/轻量级锁/重量级锁状态切换的物理基础，脱离对象头就只能死记锁升级流程而不懂"为什么"。
  • 可达性分析与 GC：对象在被移动（复制算法、标记整理）时，Klass Word 和字段中的引用地址都要同步更新，对象头的存在让 GC 能识别对象类型和大小，这是移动式 GC 能工作的前提。
  • 包装类与集合框架：每个 Integer/Long 对象都要背负至少 12~16 字节的对象头，这正是 Integer 缓存池、基本类型数组优于包装类数组的根本原因——在 ArrayList<Integer> 里，装箱带来的对象头开销比数据本身还大。
  • hashCode() 与 HashMap：无锁状态下 identity hashCode 计算后写入 Mark Word 并永久固定，这解释了为什么重写 hashCode() 的类不受此限制，而未重写的类第一次调用 hashCode() 之后对象状态会发生细微变化（禁用偏向锁）。
  • JMM 与 volatile：对象头是对象私有的元信息，和 JMM 讨论的字段可见性是两个维度的问题，但两者都依赖"堆上对象"这一共同物理载体，理解了对象布局能更好区分"对象结构"与"内存可见性"这两个常被混淆的概念。
