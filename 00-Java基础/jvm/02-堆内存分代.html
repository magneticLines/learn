JVM · 内存模型

02. 堆内存分代

Eden / Survivor / Old Gen 的设计原理、对象晋升流程与 GC 调优基础

为什么要分代？从一个根本矛盾说起

如果把堆看成一整块连续内存，GC 每次都要扫描所有对象，代价极高。 但工程实践告诉我们一件事：大多数对象"朝生夕死"—— 一次 HTTP 请求里创建的临时 DTO、循环里的中间变量，活不过几毫秒。 真正长命的对象（缓存、连接池、Spring Bean）只占少数。

这就是 弱代假说（Weak Generational Hypothesis）： 绝大多数对象只存活很短时间；少数对象则几乎永远存活。 分代 GC 正是为了利用这个规律——只频繁扫描"可能含有大量垃圾"的年轻区域， 减少全堆扫描频率，从而大幅降低平均 GC 停顿。

弱代假说并非 Java 首创，它源自 1980 年代对 Smalltalk 程序的统计研究， 后来被 C++、Python、Ruby 等几乎所有托管运行时采纳。

堆的分代结构（HotSpot 视角）

Heap（-Xms / -Xmx） ├── Young Generation（年轻代） ← -Xmn / -XX:NewRatio │ ├── Eden 约 8/10 │ ├── Survivor 0 约 1/10 (from) │ └── Survivor 1 约 1/10 (to) └── Old Generation（老年代） ← 剩余空间

JDK 8 及之前还有 Permanent Generation（永久代）存放类元数据， JDK 8 起改为堆外的 Metaspace，不再受 -Xmx 约束。 本文聚焦 Eden / Survivor / Old Gen 三个核心区域。

区域默认比例GC 类型GC 频率
Eden + Survivor~1/3 堆Minor GC（Young GC）高（秒级）
Old Gen~2/3 堆Major GC / Full GC低（分钟级）

对象从出生到"养老"的完整流程

① 新对象优先分配到 Eden

JVM 在 Eden 区使用 指针碰撞（Bump-the-pointer）分配， 速度几乎等同于 C 的 malloc（有 TLAB 缓冲，甚至可做到无锁）。 当 Eden 满时触发一次 Minor GC。

② Minor GC：复制算法清理年轻代

年轻代使用 标记-复制（Mark-Copy）算法。 过程如下：

  1. 扫描 Eden + 当前 Survivor（from 区），标记存活对象。
  2. 将存活对象 复制到另一块 Survivor（to 区），并给每个对象的年龄计数器 +1。
  3. 清空 Eden 和旧 from 区（直接抹掉，极快）。
  4. 交换 from / to 角色，等待下次 Minor GC。
复制算法的代价是空间浪费：始终有一块 Survivor 闲置。 但年轻代对象存活率极低（<10%），需要复制的量很小，所以速度非常快，停顿通常在 10ms 以内。

③ 对象年龄与晋升老年代

每次 Minor GC 存活后，对象年龄 +1。 默认年龄达到 15（参数 -XX:MaxTenuringThreshold=15）时晋升老年代。

还有两种提前晋升的情况：

  • 大对象直接进老年代：超过 -XX:PretenureSizeThreshold 的对象直接绕过 Eden，避免在年轻代来回复制。
  • 动态年龄判断：若 Survivor 中年龄 N 的对象总大小 > Survivor 容量的 50%，则年龄 ≥ N 的对象全部晋升，无需等到 15。

④ 老年代：标记-整理算法

老年代对象存活率高，复制算法不划算（需要额外 50% 空间）。 Serial Old / Parallel Old 使用 标记-整理（Mark-Compact）： 标记存活对象，向一端移动，释放边界外的空间，消除碎片，但会 STW（Stop-The-World）较长。 CMS 使用标记-清除，G1 / ZGC 有各自的混合策略（后续章节详述）。

代码示例：观察分代 GC 行为

用 -verbose:gc -XX:+PrintGCDetails（JDK 8）或 -Xlog:gc*（JDK 9+）观察。

AllocationDemo.javaJVM · 分代观察
/**
 * 启动参数（JDK 8）：
 *   -Xms64m -Xmx64m -Xmn20m -XX:+PrintGCDetails -XX:+PrintGCDateStamps
 *
 * 目的：让 Eden 快速填满，触发 Minor GC，再让部分对象晋升 Old Gen
 */
public class AllocationDemo {

    private static final int _1MB = 1024 * 1024;

    public static void main(String[] args) {
        // 分配 4 个 2MB 对象；前三个 Minor GC 后会进 Old Gen，
        // 因为 Survivor 放不下（动态年龄判断提前晋升）
        byte[] a1 = new byte[2 * _1MB];
        byte[] a2 = new byte[2 * _1MB];
        byte[] a3 = new byte[2 * _1MB];

        // 触发 Minor GC：Eden 不够放下 a4，先回收前三个
        byte[] a4 = new byte[4 * _1MB];

        // 此时 a1/a2/a3 已无引用，GC 后 Eden 只剩 a4
        // 若 a1/a2/a3 仍有引用，则会晋升 Old Gen
        System.out.println("分配完成，查看 GC 日志观察分代变化");
    }
}

TLAB：让每个线程都有自己的"私人 Eden"

Eden 是所有线程共享的，若每次分配都加锁，并发性能会很差。 JVM 的解法是 TLAB（Thread-Local Allocation Buffer）： 在 Eden 内为每个线程预先划一块私有缓冲区，线程在自己的 TLAB 内分配对象无需加锁。 TLAB 用完才回到 Eden 竞争申请新缓冲，争锁概率大幅降低。

可通过 -XX:+PrintTLAB 查看 TLAB 的使用情况； 大对象（超过 TLAB 剩余空间）会直接在 Eden 做 CAS 分配或直接进老年代。

关键 JVM 参数一览

参数含义建议
-Xms / -Xmx堆初始 / 最大大小生产环境设为相等，避免动态扩容触发 Full GC
-Xmn年轻代大小（绝对值）与 -XX:NewRatio 二选一
-XX:NewRatio=2老年代 / 年轻代 = 2:1默认值；对象存活率高时可调大老年代
-XX:SurvivorRatio=8Eden : S0 : S1 = 8:1:1默认；Eden 越大，Minor GC 越少但每次更慢
-XX:MaxTenuringThreshold晋升年龄阈值默认 15；短命对象多时可调小，减少复制次数
-XX:PretenureSizeThreshold大对象直接进老年代阈值Serial/ParNew 有效；G1 用 -XX:G1HeapRegionSize

常见陷阱

陷阱 1：Survivor 区太小导致对象提前晋升
如果 Survivor 只有 1MB，一次 Minor GC 后存活对象超过 500KB，动态年龄判断就会把年龄较大的对象全部晋升老年代，最终频繁触发 Full GC。 可通过 -XX:SurvivorRatio 调小（让 Survivor 更大）来缓解。
陷阱 2：以为 new 对象一定在堆上
JIT 的 逃逸分析（Escape Analysis）可以把不逃逸出方法的对象栈上分配或做标量替换， 完全绕开堆分配，GC 压力为零。不要手动做过早优化去复用对象池，先让 JVM 自己优化。
陷阱 3：-Xmn 设太大
年轻代过大会导致 Minor GC 扫描时间变长，反而增加停顿； 老年代空间被挤压，促使 Full GC 更频繁。 经验值：年轻代一般为堆的 1/4 ~ 1/3，具体看对象存活率。
陷阱 4：G1 / ZGC 下"分代"概念已演化
G1 将堆拆分为等大的 Region，Eden/Survivor/Old 是逻辑标签而非连续物理区域。 JDK 21 引入 Generational ZGC，ZGC 也正式支持分代回收。 用 G1/ZGC 时，-Xmn、-XX:NewRatio 等参数不再生效或意义改变，需重新学习调优方法。

面试高频考点

Q1：Minor GC 和 Full GC 的触发条件？

Minor GC：Eden 区空间不足，无法分配新对象时触发。
Full GC（会 STW 更长）触发条件包括：①老年代空间不足；②System.gc()（不推荐）； ③CMS 的 Concurrent Mode Failure；④方法区（Metaspace）空间不足；⑤空间分配担保失败（晋升对象大于老年代可用空间）。

Q2：为什么 Survivor 要设计成两块而不是一块？

复制算法需要一个"干净的目标区"来接收存活对象。 若只有一块 Survivor，GC 后存活对象和新分配对象混在一起，年龄无法准确追踪，也无法整理碎片。 两块 Survivor 交替使用，保证每次复制目标区始终是空的。

Q3：对象一定先在 Eden 分配吗？

不一定。大对象直接进老年代（Serial/ParNew 下超过 PretenureSizeThreshold）； 逃逸分析可能让对象在栈上分配； TLAB 溢出的对象会直接在 Eden 做 CAS 分配。

Q4：空间分配担保（Handle Promotion）是什么？

Minor GC 前，JVM 检查老年代最大可用连续空间是否大于年轻代所有对象总大小。 如果是，Minor GC 是安全的；如果不是，JVM 根据历史晋升量做概率预测， 若预测失败则直接触发 Full GC，这就是担保失败。
