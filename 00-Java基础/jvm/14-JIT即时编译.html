14. JIT 即时编译（C1 / C2 / 分层编译）

Java 为何能从"慢语言"蜕变为接近 C++ 性能？秘密就在 JIT——它在运行时把热点字节码翻译成本地机器码，并结合真实运行数据做出比静态编译更激进的优化。

1. 为什么需要 JIT？

Java 的执行路径分三层：源码 → 字节码 → 机器码。字节码由 JVM 解释执行，每条指令都要经历"取指→解释→执行"，天然比直接执行机器码慢 10～50 倍。

解决思路有两个极端：

  • 纯解释执行（Python 早期）：启动快，但运行慢；
  • AOT（Ahead-of-Time）提前编译（Go、GraalVM Native Image）：运行快，但启动慢、丢失运行时信息。

JIT 走中间路线：先解释运行，收集真实的调用频率与类型信息，再对热点代码进行编译优化。这样既保留了快速启动的优势，又能利用运行时 profile 数据做出比 AOT 更激进的优化（例如内联虚方法、消除空检查）。

直觉类比：就像一本厚厚的外语书——刚开始你一字一字查词典（解释执行），等某段文字反复出现，你把它整段记住（JIT 编译），之后看到直接念出来，无需再查。

2. HotSpot 的编译器：C1 与 C2

HotSpot JVM 内置两个 JIT 编译器，定位截然不同：

特性 C1（Client Compiler） C2（Server Compiler）
别名 -client，快速编译器 -server，优化编译器
编译速度 快（毫秒级） 慢（数十毫秒）
生成代码质量 中等 极高（接近 C++ -O2）
主要优化手段 基础内联、常量折叠、寄存器分配 激进内联、逃逸分析、标量替换、循环展开、向量化
适合场景 GUI / 短命进程，需要快速响应 长期运行的服务端应用
注意：C2 使用 C++ 实现，其内部用的是一种称为"Sea of Nodes"的 IR（中间表示）。由于过于复杂，JDK 团队曾多次尝试重写 C2（Project Metropolis、Graal），至今 C2 仍是生产主力。

3. 分层编译（Tiered Compilation）

Java 7 引入、Java 8 默认开启。分层编译把解释器 + C1 + C2 串成一条流水线，共 5 个层次：

Level 0 解释执行（Interpreter） │ 收集方法调用次数（invocation counter）与回边次数（back-edge counter） ▼ Level 1 C1 编译 — 无 profiling │ 最简单的编译，适合只调用少量次数的方法 ▼ Level 2 C1 编译 — 有有限 profiling（方法调用计数） ▼ Level 3 C1 编译 — 完整 profiling（类型 + 分支概率） │ 大量收集运行时数据，为 C2 做准备 ▼ Level 4 C2 编译 — 激进优化，利用 Level 3 收集的 profile

方法通常经历 0 → 3 → 4 的升级路径。如果 C2 编译队列繁忙，则走 0 → 2 → 3 → 4，先用 C1 顶上。如果方法很简单（叶方法、getter/setter），可能只停在 Level 1，无需升到 C2。

为什么这样设计？纯 C2 需要等大量 profile 数据才能编译，启动初期吞吐会很低。分层编译让 C1 先快速"接管"热点方法提升性能，C2 在后台慢慢优化，两者各司其职，兼顾了启动速度与峰值吞吐。

4. 热点探测：什么时候触发编译？

JVM 用两个计数器决定何时把方法提交给 JIT：

  • 方法调用计数器（invocation counter）：每次方法被调用 +1；
  • 回边计数器（back-edge counter）：每次循环回跳 +1，用于检测循环热点（OSR 触发点）。

两者之和超过阈值（默认 -XX:CompileThreshold=10000，分层编译下由内部动态调整）时，方法被放入编译队列异步编译。

OSR（On-Stack Replacement，栈上替换）

如果一个方法里有超大循环，方法调用次数不多但循环执行了数万次，回边计数器会先触发编译。编译完成后，JVM 能在循环正在执行时把栈帧替换成编译后版本，这就是 OSR。

Java — OSR 触发场景示意
public static long sum() {
    long result = 0;
    // 这个循环只执行一次（方法调用计数=1），但回边计数会飙升
    // 回边计数超阈值 → OSR 触发 → 循环体被替换为编译后代码
    for (long i = 0; i < 1_000_000_000L; i++) {
        result += i;
    }
    return result;
}

5. JIT 的核心优化技术

5.1 方法内联（Inlining）

把被调用方法的代码直接嵌入调用处，消除栈帧创建与参数传递的开销。这是所有后续优化的基础——内联后代码展平，编译器才能看到更大的分析窗口。

Java — 内联前 vs 内联后（逻辑等价）
// 内联前：每次调用 add() 都要压栈/弹栈
int add(int a, int b) { return a + b; }
int r = add(3, 4);

// 内联后：编译器直接计算常量，甚至折叠成 7
int r = 3 + 4;  // → 常量折叠 → r = 7

控制内联的关键参数：

  • -XX:MaxInlineSize=35：字节码小于 35 字节的方法直接内联；
  • -XX:FreqInlineSize=325：热点方法允许内联到 325 字节；
  • -XX:MaxInlineLevel=9：最大内联深度 9 层。

5.2 逃逸分析（Escape Analysis）

分析对象的引用是否"逃逸"出当前方法/线程。若未逃逸，JIT 可以做三件事：

  • 栈上分配：对象分配在栈帧而非堆，随方法返回自动释放，减轻 GC 压力；
  • 标量替换：把对象各字段拆散为局部变量，彻底消除对象头开销；
  • 锁消除：局部对象上的 synchronized 块无需真正加锁。
Java — 逃逸分析示例
public int compute() {
    // Point 未逃逸出 compute()，JIT 可将其标量替换为 x=1, y=2 两个 int
    Point p = new Point(1, 2);
    return p.x + p.y;  // 优化后等价于 return 3;
}

public String getStr() {
    StringBuilder sb = new StringBuilder();
    // sb 未逃逸 → synchronized 内部锁被消除（锁消除）
    sb.append("hello");
    return sb.toString();
}

5.3 去虚化（Devirtualization）与条件内联

Java 虚方法调用（invokevirtual）需要运行时查找 vtable，无法直接内联。JIT 利用 profile 数据发现"这个虚调用 99% 都是 ArrayList 的实现"，便生成：

伪代码 — 双态内联（bimorphic inline）
if (receiver.getClass() == ArrayList.class) {
    // 直接内联 ArrayList.size() 实现
    return this.size;
} else {
    // 罕见路径：慢路径，走正常 vtable 查找
    return receiver.size();
}

这是 JIT 比 AOT 更强的关键所在——AOT 不知道运行时的实际类型分布，只能保守处理。

5.4 反优化（Deoptimization）

JIT 的激进优化基于"假设"（如单态内联假设类型不会变）。当假设被打破（例如加载了新的子类），JVM 会触发反优化：丢弃编译后代码，退回解释执行，重新收集 profile，再重新编译。

陷阱：频繁触发反优化（Deopt）会导致性能抖动。可通过 -XX:+PrintCompilation 观察带 made not entrant / made zombie 的日志行，结合 JITWatch 工具定位根因。

6. 查看 JIT 编译日志

Shell — 常用诊断参数
# 打印每次 JIT 编译事件

# 输出示例（每列含义：时间戳 编号 标志 层级 方法 字节码大小）

Java — 手动触发 JIT 预热（基准测试必做）
/**
 * 为什么要预热：JMH 等基准框架会在正式计时前多次调用目标方法，
 * 确保 JIT 已将其编译到 Level 4，测量的才是稳定的峰值性能。
 * 不预热的结果既包含解释执行时间，毫无参考价值。
 */
@Benchmark
@Warmup(iterations = 5, time = 1)   // 预热 5 轮，每轮 1 秒
@Measurement(iterations = 5, time = 1)
public int benchmarkMethod() {
    return target.compute();
}

7. 常见陷阱与踩坑指南

陷阱 1：微基准不预热，测出的是解释执行性能

直接在 main 里计时 10 次循环的平均值，前几轮跑的是解释器。必须使用 JMH 并配置足够的 warmup。

陷阱 2：方法太大，无法内联

一个 600 字节的"上帝方法"永远无法被内联，也限制了调用它的方法的优化深度。保持方法短小是让 JIT 充分发挥的前提。

实战规则：核心热路径上的方法字节码尽量控制在 35 字节（trivial inline threshold）以内；若超过 325 字节则几乎不可能被内联。可用 javap -c 查看字节码大小。

陷阱 3：过度多态阻止去虚化

如果一个接口方法在运行时有超过 2 种实现（megamorphic），C2 会放弃去虚化，退化为间接调用，无法内联。

Java — 多态调用导致去虚化失败
List<Processor> processors = List.of(
    new TypeA(), new TypeB(), new TypeC()  // 3 种实现 → megamorphic
);
for (Processor p : processors) {
    // process() 无法内联，因为 JIT 不知道运行时类型
    p.process();
}
// 优化：若热路径只有 1 种类型，用具体类型变量持有引用

陷阱 4：逃逸分析被 JNI / 反射打破

通过反射访问或 JNI 传递对象，JIT 无法证明对象未逃逸，标量替换和栈上分配均失效。

陷阱 5：Warmup 阶段 GC 与 JIT 共同抢占 CPU

服务启动初期同时面临解释执行（慢）+ JIT 编译线程（占 CPU）+ GC（分配压力大），三重叠加会导致"启动风暴"。可通过 -XX:CICompilerCount=2 控制编译线程数，或用 AppCDS/AOT 缓存已编译代码缩短预热时间。

8. 面试高频考点

问题答题要点
JIT 和 AOT 的区别？ JIT 运行时编译、能利用 profile 数据做激进优化（去虚化/内联）、启动慢；AOT 提前编译、启动快、无运行时信息、优化保守
分层编译有几层？默认哪个版本开启？ 5 层（0-4），Java 8 默认开启，-XX:+TieredCompilation
C1 和 C2 的主要区别？ C1 编译快、优化少，适合短命进程；C2 编译慢、优化激进（逃逸分析、循环展开），适合服务端长期运行
什么是 OSR？ On-Stack Replacement，循环执行过程中将栈帧替换为编译后版本，解决"单次调用的大循环"无法被方法调用计数触发的问题
逃逸分析能带来哪些优化？ 栈上分配、标量替换、锁消除
什么是反优化？何时触发？ Deoptimization，当 JIT 基于的假设被打破（如新子类加载、类型 profile 改变）时，丢弃编译代码退回解释执行
如何排查 JIT 内联失败？ -XX:+PrintInlining 查看内联决策；方法字节码超过阈值、虚方法 megamorphic、方法含异常处理都可能阻止内联
为什么 JMH 测试要预热？ 预热确保目标方法被 JIT 编译到 Level 4，测量的是稳定峰值性能，而非夹杂解释执行的平均值

9. 小结

核心思路一句话：JIT 用"先解释收集数据，再编译利用数据"的策略，在运行时做出比静态编译更好的优化决策。分层编译（C1 快速上线 + C2 深度优化）是这一策略的工程实现；逃逸分析、内联、去虚化是 C2 最有价值的三大优化手段。

作为开发者，最重要的两点：① 核心热路径保持方法短小（助力内联）；② 避免在热路径引入过多多态（助力去虚化）。其余的，放心交给 JIT。

10. 与其他知识点的联系

  • 集合框架：ArrayList、HashMap 等热点方法（get()/size()）体积小、调用频繁，是 C1/C2 内联和去虚化的典型受益对象；反过来说，业务代码把简单的 getter/setter 拆得过碎反而有利于 JIT 分析。
  • 并发编程：逃逸分析里的锁消除直接作用于 synchronized 块——如果 JIT 判断锁对象未逃逸出线程，会在编译期去掉这把"看起来存在但不需要"的锁，这也是很多人误以为"局部 synchronized 没有性能损耗"的底层原因。
  • JVM 内存模型：栈上分配和标量替换是逃逸分析的产物，它们让本该进堆的对象改为存在栈帧里，直接减少了 Eden 区的分配压力，是 JIT 与 GC 协同降低停顿的关键路径。
  • 反射与动态代理：反射调用、CGLib/JDK 动态代理生成的字节码，会打破 JIT 对"类型稳定"的假设，导致逃逸分析失效、内联受阻，这也是 Spring AOP 代理方法性能不如直接调用的部分原因。
  • JVM 参数调优：本文的分层编译、内联阈值等行为都可以用上一篇的 -XX:CompileThreshold、-XX:MaxInlineSize 等参数调整，JVM 参数是控制 JIT 行为的旋钮，两者是配置与机制的关系。
