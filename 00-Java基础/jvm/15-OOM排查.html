15. 常见 OOM 场景与排查

OutOfMemoryError 不是一个错误，而是 JVM 的求救信号——它告诉你内存已经被耗尽，并且 GC 已经竭尽全力仍无能为力。读懂这个信号，才能找到真正的病根。

1. 为什么会发生 OOM

JVM 内存分为若干区域，每个区域都有自己的上限。OOM 本质上是某个区域的分配请求无法被满足，且 GC 已无法回收足够空间。

JVM 内存区域 ├── 堆（Heap） ← 对象主战场，-Xmx 控制上限 │ ├── 新生代（Eden / S0 / S1） │ └── 老年代（Old Gen） ├── 方法区 / Metaspace ← 类元数据，-XX:MaxMetaspaceSize ├── 虚拟机栈（每线程一个） ← 每帧一个栈帧，-Xss 控制单线程栈大小 ├── 本地方法栈 └── 直接内存（Direct） ← NIO ByteBuffer.allocateDirect()，-XX:MaxDirectMemorySize

不同区域耗尽，抛出的 OutOfMemoryError 消息不同，这是排查的第一条线索。

OOM 消息发生区域最常见根因
Java heap space堆内存泄漏 / 堆太小 / 大对象
GC overhead limit exceeded堆GC 占用 >98% 时间但回收 <2% 空间
Metaspace方法区类加载失控（动态代理/CGLib/JSP）
unable to create new native threadOS 线程线程数超系统限制
Direct buffer memory直接内存NIO 未释放 / Netty 内存泄漏
StackOverflowError虚拟机栈无限递归（技术上不属于 OOM，但同类问题）

2. 堆 OOM：Java heap space

2.1 内存泄漏（最常见）

内存泄漏不是 C/C++ 专利。Java 里的泄漏是：对象不再被业务使用，但 GC Root 仍然可达，GC 无法回收。最典型的三种：

  • 静态集合持有对象：static Map 只增不删，随请求累积
  • 监听器/回调未注销：向 EventBus / 消息队列注册，业务结束后忘记移除
  • ThreadLocal 未 remove：线程池线程复用，ThreadLocal 里的 value 跟着线程永久存活
Java — ThreadLocal 泄漏反例 → 正例
// ❌ 反例：线程池场景下，每次请求写入 ThreadLocal 但不清理
private static final ThreadLocal<LargeContext> ctx = new ThreadLocal<>();

void handle(Request req) {
    ctx.set(new LargeContext(req));
    doWork();
    // 方法结束，线程回到线程池，LargeContext 无法被回收！
}

// ✅ 正例：try-finally 保证清理
void handle(Request req) {
    try {
        ctx.set(new LargeContext(req));
        doWork();
    } finally {
        ctx.remove(); // 归还线程前必须清理
    }
}

2.2 大对象一次性分配

加载百 MB 的 Excel、一次性 SELECT * 返回百万行数据，直接打爆堆。解法：流式处理（ResultSet 逐行读取 / EasyExcel SAX 模式）。

陷阱：String.intern() 在 JDK 6 及之前会把字符串放入 PermGen（永久代），大量 intern 会导致 PermGen OOM。JDK 7+ intern 改为放在堆上，但大量不同字符串 intern 仍会撑大堆。

3. GC overhead limit exceeded

这是 JVM 内置的熔断保护：当连续多次 GC、每次 GC 时间占比超过 98%，却只回收了不到 2% 的堆空间，JVM 主动抛出此 OOM，而不是让应用陷入"GC 地狱"（CPU 100%、吞吐量接近 0）。

本质原因依然是堆内存不足/泄漏，只是 JVM 更早介入。排查方式与 Java heap space 相同：dump 堆分析存活对象。

可以用 -XX:-UseGCOverheadLimit 关闭这个保护，但不推荐——关掉只会让应用挣扎更久再崩，问题本身没有解决。

4. Metaspace OOM

4.1 为什么 Metaspace 会耗尽

JDK 8 把方法区从固定大小的 PermGen（永久代）改为 Metaspace（本地内存），默认无上限（受物理内存约束）。类元数据占用的 Metaspace 只有在类加载器被 GC 回收后才能释放。

常见触发场景：

  • CGLib 动态代理失控：每次调用没有缓存代理类，导致类爆炸式增长
  • JSP 热部署：Tomcat 每次修改 JSP 都生成新类，旧类加载器无法卸载
  • Groovy / 脚本引擎：每次 eval 都创建新类加载器
  • Spring AOP 误用：在循环中对同一 bean 重复创建代理
Java — CGLib 代理泄漏反例
// ❌ 每次调用都生成一个新的代理类，类加载器持有引用无法卸载
for (Order order : orders) {
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(OrderService.class);
    enhancer.setCallback(myInterceptor);
    OrderService proxy = (OrderService) enhancer.create();
    proxy.process(order);
}

// ✅ 正例：代理类应单例复用，在应用启动时创建一次
// Spring 容器帮你管理，不要手动在业务代码里 new Enhancer

4.2 排查 Metaspace 增长

Shell — 监控已加载类数量
# jstat 每秒打印一次 Class 加载数
jstat -class <pid> 1000

# 如果 Loaded 数一直单调递增，说明类在持续加载
Loaded  Bytes  Unloaded  Bytes     Time
 12345  24.5       0    0.0       3.21
 12567  25.1       0    0.0       3.25  ← Unloaded 始终 0，泄漏！

5. unable to create new native thread

这个 OOM 与堆无关，是 OS 线程资源耗尽。每个 Java 线程对应一个 OS 线程，Linux 默认单进程最多 ~1024 个线程（ulimit -u）。

触发原因：

  • 无界线程池（Executors.newCachedThreadPool() 无限制创建线程）
  • 每次请求都 new Thread() 而不使用线程池
  • 堆外内存不足（每个线程栈默认 512KB~1MB，堆大了反而挤占线程可用地址空间）
反直觉陷阱：有时候堆设得越大，反而越容易触发此 OOM。原因是 32 位 JVM 进程地址空间只有 4GB，堆占 3GB 后剩余地址空间不足以分配线程栈。JDK 8+ 64 位 JVM 基本不受此限制，但 OS 线程数上限依然存在。
Java — 无界线程池反例 → 正例
// ❌ 高并发下会创建无数线程
ExecutorService pool = Executors.newCachedThreadPool();

// ✅ 明确上限，超出用队列缓冲，拒绝策略兜底
ExecutorService pool = new ThreadPoolExecutor(
    10,                           // corePoolSize
    50,                           // maximumPoolSize
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()
);

6. Direct buffer memory

ByteBuffer.allocateDirect() 在堆外分配内存，不受 -Xmx 控制，受 -XX:MaxDirectMemorySize 限制（默认等于 -Xmx）。Netty 大量使用直接内存以减少数据拷贝。

泄漏原因：DirectByteBuffer 依赖 GC 触发 Cleaner 来释放直接内存。堆对象小、GC 不频繁，直接内存却迅速耗尽，形成"堆内存很空，直接内存爆了"的诡异现象。

Netty 提供 -Dio.netty.leakDetection.level=paranoid 开启内存泄漏检测，会在日志中报告哪些 ByteBuf 未被释放——上线前务必在 staging 环境跑一遍。

7. 通用排查流程

Step 1：读异常消息，确定区域

如上面的对照表，先锁定是堆、Metaspace 还是线程。

Step 2：开启 Heap Dump 自动转储

JVM 启动参数
# OOM 时自动 dump，不需要手动抓
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/logs/oom-<pid>.hprof

Step 3：用 MAT 或 JProfiler 分析 hprof

打开 Eclipse MAT，执行 Leak Suspects Report：工具会自动找出占用内存最多的对象链，90% 的堆泄漏问题靠这一步就能定位。

MAT 排查流程 ├── 打开 .hprof ├── Leak Suspects → 自动定位最大嫌疑对象 ├── Dominator Tree → 按对象保留大小排序 ├── Histogram → 按类型统计实例数和大小 └── Path to GC Roots → 找到是谁持有引用（根因）

Step 4：线上实时诊断（不重启）

Shell — 手动触发 Heap Dump
# jmap（会短暂 STW，谨慎在生产使用）
jmap -dump:format=b,file=/tmp/heap.hprof <pid>

# 查看当前堆使用摘要（轻量，几乎无影响）
jmap -heap <pid>

# 实时 GC 监控（每 2 秒打印一次 Full GC 情况）
jstat -gcutil <pid> 2000

8. 真实案例：缓存 + 弱引用的误用

某微服务启动后内存缓慢上涨，24 小时后 OOM。Heap Dump 显示老年代被数十万个 byte[] 占满，追溯来源是一个"自以为安全"的本地缓存：

Java — WeakReference 缓存失效反例
// 开发者以为用了 WeakReference，GC 压力大时会自动清理
private final Map<String, WeakReference<byte[]>> cache = new ConcurrentHashMap<>();

void put(String key, byte[] data) {
    cache.put(key, new WeakReference<>(data));
}

// 问题：WeakReference 的 value（byte[]）会被 GC，但
// Map 的 Key（String）和 WeakReference 壳子本身不会！
// 随着 key 不断增加，Map 本身无限膨胀 → OOM

正确做法是使用 Guava Cache 或 Caffeine，它们提供基于容量/时间的自动淘汰，并且内部正确处理了 Entry 的回收。

9. 面试高频考点

问题关键答点
Java 有哪几种 OOM？ 堆（heap space / GC overhead）、Metaspace、线程（native thread）、直接内存、StackOverflow（非 OOM 但同类）
内存泄漏 vs 内存溢出 泄漏（leak）是根因：对象可达但无用；溢出（overflow/OOM）是结果：空间耗尽。泄漏持续累积最终导致溢出
ThreadLocal 为什么会泄漏？ Entry 的 key 是 WeakReference（可被 GC），但 value 是强引用。key 被回收后 value 成为孤儿，线程不销毁则永久存活。线程池场景必须 remove()
如何在不重启的情况下排查 OOM？ jmap dump → MAT 分析；或提前加 -XX:+HeapDumpOnOutOfMemoryError；Arthas 的 memory 命令可实时查看各区占用
Metaspace 与 PermGen 区别？ PermGen 固定大小在 JVM 进程内存；Metaspace 使用本地内存，默认无上限，避免了 PermGen 频繁调优的痛点，但仍需设 -XX:MaxMetaspaceSize 防止无限增长
堆设越大越好吗？ 不是。堆越大：① GC 停顿时间越长；② 在 32 位 JVM 中挤占线程栈地址空间；③ 掩盖了真正的内存泄漏，让问题更难排查

10. 快速预防清单

  • 启动参数加 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/，让 OOM 现场可复现
  • 线程池必须有界：禁用 newCachedThreadPool()，明确 maximumPoolSize 和有界队列
  • ThreadLocal 在 finally 块中 remove()
  • 本地缓存用 Caffeine，设置 maximumSize 和 expireAfterWrite
  • Metaspace 设上限：-XX:MaxMetaspaceSize=256m，防止动态代理失控吃光内存
  • 大数据读写使用流式 API（EasyExcel、JdbcTemplate queryForStream），杜绝一次性加载全量数据
  • Netty 项目开启 -Dio.netty.leakDetection.level=simple 至少在测试环境验证

11. 与其他知识点的联系

  • JVM 常用参数：本文所有排查手段的前提，是提前配好 -XX:+HeapDumpOnOutOfMemoryError、-XX:MaxMetaspaceSize 等启动参数；没有这些参数，OOM 发生后现场即销毁，排查退化为"猜"。
  • jstack / jmap / jstat：OOM 排查依赖的 jmap -histo、jstat -gcutil 正是下一篇要讲的工具，本文是"什么场景会 OOM"，下一篇是"用什么工具确认"，二者是问题与手段的对应关系。
  • 并发编程：ThreadLocal 内存泄漏的根因是 ThreadLocalMap 的 Entry key 为弱引用而 value 为强引用，这是并发编程里线程池复用线程时最容易被忽视的陷阱，必须结合线程生命周期理解。
  • 集合框架：静态 Map/List 持有对象导致的泄漏，本质是集合的默认语义"只增不减"，与 WeakHashMap、Guava Cache/Caffeine 的自动淘汰机制形成对比，选错数据结构是很多内存泄漏的第一张倒下的多米诺骨牌。
  • Netty / NIO：直接内存 OOM 与堆内存 OOM 是两套独立的记账体系，理解 DirectByteBuffer 依赖 GC 触发 Cleaner 回收堆外内存，才能解释"堆很空但直接内存爆了"这种反直觉现象。
