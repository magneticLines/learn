12. 破坏双亲委派的场景（SPI / Tomcat）

双亲委派是 JVM 类加载的默认秩序，但它并非不可打破。理解"何时以及为何要打破它"，是深入理解 Java 模块化、框架隔离的关键。

为什么要破坏双亲委派？

上一节我们学到，双亲委派的核心价值是安全性与唯一性：核心类只由 Bootstrap ClassLoader 加载，不会被第三方替换；同一个类在 JVM 里只有一份。

但这个模型有一个根本性矛盾：委派方向是"自下而上"，而有时加载需求是"自上而下"。

经典悖论：java.sql.Driver 接口在 rt.jar（Bootstrap 加载），但具体实现（如 com.mysql.cj.jdbc.Driver）在应用 classpath（App ClassLoader 加载）。
Bootstrap ClassLoader 无法"向下"看到应用 classpath，所以按双亲委派它根本找不到实现类。

JDK、Tomcat、OSGi 三个场景分别用了三种不同的方式解开这个死结，各有巧妙之处。

场景一：SPI 与线程上下文类加载器

SPI 是什么

SPI（Service Provider Interface） 是 JDK 提供的插件化机制：在 META-INF/services/ 目录下放一个以接口全限定名命名的文件，文件内容是实现类名。ServiceLoader 在运行时扫描并实例化这些实现。

JDBC、JNDI、JAX-WS、日志门面（SLF4J）都用了这套机制。问题在于：ServiceLoader 自身在 rt.jar 里，由 Bootstrap ClassLoader 加载，但它要加载的"实现类"却在用户 classpath 里——Bootstrap ClassLoader 根本看不到。

解法：线程上下文类加载器（Thread Context ClassLoader）

JDK 1.2 引入了 Thread.currentThread().getContextClassLoader()。主线程启动时，JVM 把它默认设置为 App ClassLoader。 ServiceLoader 拿到这个 ClassLoader 来加载实现类，绕过了 Bootstrap 向下看不到的问题。

这本质上是一种"父 ClassLoader 委托子 ClassLoader 去加载"的逆向委派，打破了双亲委派的单向性。 它是一个妥协方案，JDK 开发者自己也承认这是一种"不得不为"的设计。
ServiceLoader 核心源码（简化）Java
// java.util.ServiceLoader#load(Class<S>)
public static <S> ServiceLoader<S> load(Class<S> service) {
    // 关键：不用自己（Bootstrap），而是拿线程上下文类加载器
    ClassLoader cl = Thread.currentThread().getContextClassLoader();
    return load(service, cl);
}

JDBC 加载流程（完整链路）

1. Class.forName("com.mysql.cj.jdbc.Driver") ← 旧写法，现在不需要了
2. DriverManager.getConnection(url) ← 触发 DriverManager 静态初始化
3. DriverManager 由 Bootstrap 加载，调用 ServiceLoader.load(Driver.class)
4. ServiceLoader 拿到线程上下文类加载器（App ClassLoader）
5. 扫描 classpath 中 META-INF/services/java.sql.Driver
6. App ClassLoader 加载 com.mysql.cj.jdbc.Driver
7. Driver 自动注册到 DriverManager ✓

场景二：Tomcat 的类加载隔离

Tomcat 面临的问题

一个 Tomcat 实例可以同时跑多个 Web 应用（WAR 包）。这带来三个相互冲突的需求：

  • 隔离性：App-A 和 App-B 各自依赖 jackson-2.12 和 jackson-2.15，不能互相干扰
  • 共享性：Tomcat 自身的 Servlet API、公共工具类要被所有应用共享，不能每个 WAR 都加载一份
  • 安全性：JDK 核心类始终由 Bootstrap 加载，不可被覆盖

标准双亲委派只有一条线性委派链，无法同时满足"隔离"和"共享"，所以 Tomcat 设计了一套多层次并行的类加载器树。

Tomcat 类加载器层次

Bootstrap ClassLoader（JDK 核心） └── Extension ClassLoader（JDK 扩展） └── App ClassLoader（Tomcat 启动 classpath） └── CommonClassLoader ← $CATALINA_HOME/lib，所有 Web 应用可见 ├── CatalinaClassLoader ← Tomcat 容器私有，Web 应用不可见 ├── SharedClassLoader ← 所有 Web 应用共享，Tomcat 不可见 │ ├── WebappClassLoader-App1 ← /WEB-INF/classes + /WEB-INF/lib │ └── WebappClassLoader-App2 ← /WEB-INF/classes + /WEB-INF/lib

WebappClassLoader 如何打破双亲委派

标准双亲委派：先委派父加载器，父加载器找不到再自己加载。
WebappClassLoader 策略：先自己加载，自己找不到再委派父加载器（对非 JDK 核心类）。

WebappClassLoader#loadClass 逻辑（伪代码，基于 Tomcat 源码简化）Java
@Override
public Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {

    // 1. 先查本地缓存（已加载过的类）
    Class<?> clazz = findLoadedClass(name);
    if (clazz != null) return clazz;

    // 2. java.* 必须委派给 Bootstrap，安全红线不可破
    if (name.startsWith("java.")) {
        return parent.loadClass(name);
    }

    // 3. 【关键】先用自己加载 WEB-INF 下的类（打破双亲委派！）
    try {
        clazz = findClass(name);  // 在 /WEB-INF/classes 和 /WEB-INF/lib 中找
        if (clazz != null) return clazz;
    } catch (ClassNotFoundException e) {
        // 自己找不到，继续往下走
    }

    // 4. 自己找不到，再委派父加载器（SharedClassLoader → CommonClassLoader）
    return parent.loadClass(name);
}

这种"先自己、再父级"的策略让每个 WebApp 都优先使用自己 WEB-INF 下的 JAR，从而实现版本隔离。

具体隔离效果

类的来源由谁加载App1 / App2 是否共享
java.*、javax.*（JDK）Bootstrap ClassLoader是（强制共享）
Servlet API（javax.servlet.*）CommonClassLoader是
Tomcat 内部实现CatalinaClassLoader否（Web 应用看不到）
各 App 的 WEB-INF/libWebappClassLoader（各自独立）否（互相隔离）

场景三：OSGi 的网状类加载（扩展了解）

OSGi（如 Eclipse 插件系统）彻底抛弃了树形委派链，每个 Bundle（插件）都有自己的 ClassLoader，Bundle 之间通过声明式依赖（Import-Package/Export-Package）形成一张网状图。

类加载时不再沿树形向上委派，而是根据包名在图中寻找"谁导出了这个包"再去那个 Bundle 的 ClassLoader 加载。这是最彻底的双亲委派破坏，也带来了最复杂的依赖管理难度（bundle 地狱）。

Java 9 的 JPMS（模块系统）吸收了 OSGi 的思想，在 JDK 层面原生支持模块边界和封装，是 OSGi 问题的官方解法。

三种破坏方式对比

场景破坏方式解决的核心问题代价
SPI（JDBC 等） 线程上下文类加载器，父加载器委托子加载器 父无法看到子 classpath 中的实现类 类加载方向混乱，隐式依赖线程状态
Tomcat WebApp 重写 loadClass，先自己加载再委派父 多应用版本隔离 不同 WebApp 间同名类互不可见
OSGi 彻底放弃树形委派，网状图协商 细粒度插件化、热插拔 依赖管理极复杂，调试困难

手写一个"先自己加载"的 ClassLoader

理解原理最好的方式是自己写一遍。下面模拟 Tomcat WebappClassLoader 的核心逻辑：

MyWebappClassLoader.javaJava
public class MyWebappClassLoader extends ClassLoader {

    private final String webInfClassesPath;

    public MyWebappClassLoader(String path, ClassLoader parent) {
        super(parent);
        this.webInfClassesPath = path;
    }

    @Override
    protected Class<?> loadClass(String name, boolean resolve)
            throws ClassNotFoundException {

        // 1. 查缓存
        Class<?> loaded = findLoadedClass(name);
        if (loaded != null) return loaded;

        // 2. java.* 强制委派（安全红线）
        if (name.startsWith("java.")) {
            return super.loadClass(name, resolve);
        }

        // 3. 先尝试从 WEB-INF/classes 加载（自己先来，打破委派顺序）
        try {
            Class<?> clazz = findClass(name);
            if (resolve) resolveClass(clazz);
            return clazz;
        } catch (ClassNotFoundException ignored) {
            // 自己没有，交给父加载器（SharedClassLoader）
        }

        // 4. 委派父加载器
        return super.loadClass(name, resolve);
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 把类名转为文件路径，读取字节码，调用 defineClass
        String classFile = webInfClassesPath +
            name.replace(".", "/") + ".class";
        try (InputStream in = new FileInputStream(classFile)) {
            byte[] bytes = in.readAllBytes();
            return defineClass(name, bytes, 0, bytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name);
        }
    }
}

常见陷阱与坑

陷阱 1：ClassCastException 明明是同一个类

症状：(MyService) obj 抛出 ClassCastException: MyService cannot be cast to MyService，两边明明是"同一个类"。

根本原因：JVM 的类唯一性由 ClassLoader + 全限定名 共同决定。同一个 .class 文件被两个不同的 ClassLoader 加载，在 JVM 眼中就是两个不同的类，互相强转必然抛 ClassCastException。

解法：确保接口/父类由共同的父 ClassLoader（如 SharedClassLoader）加载，实现类才用各自的 WebappClassLoader。这也是 Tomcat 把 Servlet API 放到 Common 层的原因。

陷阱 2：线程上下文类加载器被"污染"

某些框架（如旧版 Spring）在创建线程池时没有正确传递线程上下文类加载器，导致线程池中的 SPI 加载失败，报 ServiceConfigurationError。

规避：在提交任务前保存并恢复：Thread.currentThread().setContextClassLoader(appClassLoader)。

陷阱 3：热加载/热部署导致 Metaspace 泄漏

Tomcat 热部署时会创建新的 WebappClassLoader，旧的需要被 GC 回收。但如果有静态变量、ThreadLocal、JDBC Driver 注册等持有对旧 ClassLoader 的引用，旧 ClassLoader 就无法被回收，导致 Metaspace 持续增长直至 OOM。

排查命令：jmap -dump:live,format=b,file=heap.hprof <pid>，然后用 MAT 搜索 WebappClassLoader 实例数量。正常热部署后只应存在一个。

面试高频考点

Q1：为什么要打破双亲委派？举两个例子。

标准答法：双亲委派保证了安全性和唯一性，但无法解决"父 ClassLoader 需要使用子 ClassLoader 才能看到的类"的问题。 例一：SPI（JDBC），DriverManager 由 Bootstrap 加载，但 MySQL 驱动在用户 classpath，通过线程上下文类加载器解决。 例二：Tomcat，多应用版本隔离，WebappClassLoader 先加载自身 WEB-INF 目录，不先委派父加载器。

Q2：Tomcat 如何实现应用隔离？

每个 Web 应用对应一个独立的 WebappClassLoader 实例。它重写了 loadClass 方法，对非 JDK 核心类优先从自己的 WEB-INF 目录加载，找不到才委派父加载器（SharedClassLoader）。由于两个 WebApp 的 ClassLoader 实例不同，即使是同名类在 JVM 内也是两个不同的类对象，实现了彻底隔离。

Q3：SPI 用的是什么机制破坏双亲委派？有什么缺点？

线程上下文类加载器（Thread Context ClassLoader）。缺点：类加载方向违反了直觉（父委托子），且隐式依赖线程状态，在线程切换或线程池场景下需要手动维护，容易出错。Java 9 的模块系统在一定程度上改善了这个问题。

Q4：同一个类被两个 ClassLoader 加载会怎样？

JVM 会把它们视为两个完全不同的类型，相互之间不能强转（抛 ClassCastException），也不满足 instanceof。这是多 ClassLoader 隔离设计中最容易踩的坑，解法是将共享的接口提升到公共的父 ClassLoader 中加载。

与其他知识点的联系

  • 双亲委派模型：本节的三种破坏方式（SPI、Tomcat、OSGi）都是在上一节"先问父再自己"的默认约定基础上做的定制化调整，理解标准委派流程是理解"如何、为何打破它"的必要前提。
  • 类加载流程：无论委派方向如何变化，最终都要落到 findClass() + defineClass() 完成加载阶段，再走验证/准备/解析/初始化——破坏双亲委派改变的只是"谁来加载"，不改变类加载五阶段本身的流程。
  • 可达性分析与内存泄漏：Tomcat 热部署场景下，旧的 WebappClassLoader 若因静态变量/ThreadLocal/JDBC Driver 注册等被意外持有引用，就无法被 GC 判定为不可达，导致 Metaspace 持续增长——这是可达性分析理论在类加载器这个特殊对象上的直接应用。
  • SPI 与 Spring/微服务生态：SLF4J、Spring Boot 的自动配置（spring.factories/AutoConfiguration.imports）都借鉴了 JDK SPI 的思想，本质上都是靠线程上下文类加载器或约定文件实现"接口与实现解耦、运行时动态发现"，理解 JDK SPI 有助于理解这些框架级机制。
  • 线程池与 ThreadLocal：线程上下文类加载器依附在 Thread 对象上，线程池复用线程时如果不正确传递/重置它，会导致 SPI 加载使用了错误的类加载器——这是线程池编程中一个容易被忽视的隐藏状态，和 ThreadLocal 未清理导致的问题属于同一类"线程复用引发的状态污染"陷阱。
