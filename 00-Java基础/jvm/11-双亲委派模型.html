11. 双亲委派模型

类加载器的信任体系——为什么 java.lang.String 永远只有一个？

什么是双亲委派模型

双亲委派（Parent Delegation Model）是 JVM 类加载器之间的一条协作约定： 当一个类加载器收到加载请求时，它不会自己先去尝试加载，而是先把请求委托给父加载器， 父加载器再向上委托，直到最顶层的启动类加载器（Bootstrap ClassLoader）。 只有当父加载器无法完成加载（在自己的搜索范围内找不到该类）时，子加载器才会尝试自己加载。

这条规则来自 JDK 1.2，写在 java.lang.ClassLoader#loadClass() 里，不是 JVM 规范强制要求，而是一个被广泛遵循的设计惯例。

三层类加载器结构

Bootstrap ClassLoader（C++ 实现，无 Java 对象） 负责加载 $JAVA_HOME/lib（rt.jar、core libraries） │ └── Extension ClassLoader（JDK 8）/ Platform ClassLoader（JDK 9+） 负责加载 $JAVA_HOME/lib/ext 或 java.ext.dirs │ └── Application ClassLoader（也叫 System ClassLoader） 负责加载 classpath 上的应用类 │ └── 用户自定义 ClassLoader（可继续向下扩展）
JDK 9 模块化后的变化： Extension ClassLoader 被重命名为 Platform ClassLoader，负责加载平台模块（java.se 等）。 Bootstrap ClassLoader 加载 java.base 等核心模块。整体双亲委派逻辑不变，但边界从 jar 文件变为模块。

为什么这么设计——解决了什么问题

问题 1：安全性 / 防篡改

假设没有双亲委派，任何人都可以写一个自己的 java.lang.String 并让应用加载器加载它。 恶意代码可以借此替换核心类、窃取密码、绕过安全检查。 有了委派机制，java.lang.String 的请求永远会被委托到 Bootstrap ClassLoader， 它只会从 JDK 的 rt.jar 里加载，用户写的同名类根本没机会被加载。

问题 2：类的唯一性 / 避免重复加载

JVM 里"同一个类"的判断依据是：类全限定名 + 加载它的 ClassLoader。 如果 java.lang.Object 被 Bootstrap 和 AppClassLoader 各加载一次，它们就是两个不同的类—— 你的对象无法赋给 Object 类型变量，instanceof 也会失效。 双亲委派保证同一个类名在整个 JVM 生命周期里只被加载一次（由最顶层能处理的加载器负责）。

问题 3：稳定基础库

JDK 核心类（java.lang.*、java.util.*）由 Bootstrap 加载， 任何第三方库都无法意外覆盖它们，整个平台的行为可预期。

源码：loadClass() 如何实现委派

来看 JDK 8 java.lang.ClassLoader 的核心逻辑（已精简注释）：

ClassLoader.java（JDK 8，精简版）
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {

    // 1. 先查缓存：同一类只加载一次
    Class<?> c = findLoadedClass(name);

    if (c == null) {
        try {
            if (parent != null) {
                // 2. 有父加载器 → 先委托父加载器
                c = parent.loadClass(name, false);
            } else {
                // 3. 父为 null → 委托给 Bootstrap ClassLoader（C++ 层）
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // 父加载器找不到 → 捕获异常，不向上抛，让子加载器自己尝试
        }

        if (c == null) {
            // 4. 父加载器也加载不了 → 自己尝试（findClass 是子类覆写的扩展点）
            c = findClass(name);
        }
    }

    if (resolve) {
        resolveClass(c);  // 链接阶段（可选）
    }
    return c;
}
扩展点设计： JDK 把 loadClass()（委派逻辑）和 findClass()（实际加载逻辑）分开。 自定义类加载器只需覆写 findClass()，委派逻辑由父类的 loadClass() 保证， 这是模板方法模式的经典应用。

自定义类加载器：只覆写 findClass

假设我们要从磁盘某个目录加载加密过的 .class 文件（简单 XOR 解密）：

MyClassLoader.java
public class MyClassLoader extends ClassLoader {

    private final String baseDir;
    private static final byte XOR_KEY = 0x42;

    public MyClassLoader(String baseDir) {
        // 显式将 AppClassLoader 设为父加载器（默认行为，这里明写更清晰）
        super(ClassLoader.getSystemClassLoader());
        this.baseDir = baseDir;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 只有父加载器（AppCL → ExtCL → Bootstrap）都找不到时，才会进到这里
        String path = baseDir + "/" + name.replace('.', '/') + ".xclass";
        try {
            byte[] raw  = Files.readAllBytes(Path.of(path));
            byte[] data = decrypt(raw);
            return defineClass(name, data, 0, data.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }

    private byte[] decrypt(byte[] data) {
        byte[] out = new byte[data.length];
        for (int i = 0; i < data.length; i++)
            out[i] = (byte) (data[i] ^ XOR_KEY);
        return out;
    }
}

类的相等性：加载器不同 → 类不同

这是最容易被忽视的隐患：

ClassIdentityDemo.java
MyClassLoader cl1 = new MyClassLoader("/tmp/classes");
MyClassLoader cl2 = new MyClassLoader("/tmp/classes");

Class<?> c1 = cl1.loadClass("com.example.Foo");
Class<?> c2 = cl2.loadClass("com.example.Foo");

// 两个类加载器各自加载了同名类 → 是两个不同的 Class 对象
System.out.println(c1 == c2);              // false
System.out.println(c1.equals(c2));         // false

Object obj1 = c1.newInstance();
// obj1 instanceof c2 → 编译不过；强转 c2.cast(obj1) → ClassCastException!
经典陷阱： Tomcat 等容器里每个 Web 应用有自己的类加载器，如果 Foo.class 被两个 WebApp 各自加载， 它们的 Class 对象就不相等。在共享的 Filter / Listener 里做类型转换会抛 ClassCastException， 即使全限定名完全一样。

委派过程完整示例

加载 com.example.MyService 时，发生了什么：

AppClassLoader.loadClass("com.example.MyService") │ 缓存未命中 ↓ 委托 ExtClassLoader.loadClass("com.example.MyService") │ 缓存未命中 ↓ 委托 Bootstrap findBootstrapClassOrNull("com.example.MyService") → 找不到（不在 rt.jar 里）→ 返回 null ↑ ExtClassLoader 自己 findClass → lib/ext 里也没有 → 抛 CNFE ↑ AppClassLoader 捕获 CNFE，自己 findClass → 在 classpath 里找到 → 加载成功 ✓

常见陷阱

陷阱 1：误覆写 loadClass() 而不是 findClass()

如果你覆写了 loadClass() 但忘记调用 super.loadClass()， 就彻底绕过了双亲委派，可能导致核心类被重复加载或篡改。 除非你明确想打破委派（如 OSGi、热部署），否则只覆写 findClass()。

陷阱 2：获取 Bootstrap ClassLoader 得到 null

String.class.getClassLoader() 返回 null， 因为 Bootstrap ClassLoader 是 C++ 实现，Java 层没有对应对象。 不要以为拿到 null 就是出错了——这是预期行为。

陷阱 3：线程上下文类加载器（TCCL）打破委派

SPI（如 JDBC、JNDI）的接口由 Bootstrap 加载，实现类在 classpath 由 AppClassLoader 加载。 Bootstrap 无法向下委托，所以 JDK 通过 Thread.currentThread().getContextClassLoader() 绕过委派，让父加载器能够"反向"用子加载器加载实现类。这是双亲委派的第一次官方破坏，下一节详讲。

陷阱 4：热部署 / OSGI 里的类泄漏

自定义加载器加载的类，只有在加载器本身被 GC 时才会卸载。 如果有静态变量或线程持有对旧加载器的引用，就会发生类加载器泄漏（Metaspace OOM）， 这是频繁热部署的 Tomcat 应用最常见的内存问题之一。

面试高频考点

问题关键答点
双亲委派的好处？ 安全性（核心类不可被替换）、唯一性（同名类只加载一次）、稳定性
能不能自己写 java.lang.String？ 写了也没用，会被委托到 Bootstrap，加载的是 JDK 里的那个。
若强行用 defineClass 绕过，JVM 会抛 SecurityException（禁止自定义 java.* 包）
如何自定义类加载器？ 继承 ClassLoader，覆写 findClass()，调用 defineClass()
两个类什么情况下相等？ 全限定名相同 且 由同一个 ClassLoader 实例加载
如何获取各层 ClassLoader？ getSystemClassLoader() / getPlatformClassLoader() / null（Bootstrap）
双亲委派可以被打破吗？ 可以：①覆写 loadClass；②线程上下文类加载器（SPI）；③OSGi 网状模型；④JDK 9 模块层。下节详讲。

小结

  • 双亲委派是"先问父，父不行再自己干"的加载策略，不是 JVM 规范而是惯例约定。
  • 核心实现在 ClassLoader.loadClass()；扩展点是 findClass()。
  • 它保障了核心类的安全性与唯一性，防止 java.lang.Object 被用户替换。
  • 类的身份 = 全限定名 + ClassLoader，两者缺一不可，跨加载器的强转是常见陷阱。
  • SPI、热部署、OSGi 都需要打破委派，这是下一节的主题。
