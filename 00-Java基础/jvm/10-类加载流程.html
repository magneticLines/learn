JVM · 类加载

10. 类加载流程

加载 → 验证 → 准备 → 解析 → 初始化，五个阶段把 .class 字节码变成 JVM 内存中可用的类型

为什么需要类加载？

C/C++ 在编译时就把所有代码链接为一个可执行文件，运行时直接寻址调用。
Java 选择了另一条路：运行时动态加载。你写的 .java 文件 被 javac 编译成 .class 字节码，字节码本身并不包含任何内存地址， 只有在真正被用到时，JVM 才把它"加工"成可运行的类型信息，并完成符号引用到直接地址的绑定。

这个设计带来了三大好处：

  • 按需加载，节省内存：只有真正被用到的类才会进入内存
  • 热替换/热部署：OSGi、Tomcat、Spring DevTools 底层均依赖自定义类加载器实现不重启替换 class
  • 安全隔离：不同 ClassLoader 加载的类彼此隔离，Tomcat 用此隔离多个 WebApp

五阶段总览

加载（Loading） ── 读取字节流，在堆中创建 Class 对象 验证（Verification） ── 校验字节码合法性 准备（Preparation） ── 为静态变量分配内存并赋零值 解析（Resolution） ── 符号引用 → 直接引用 初始化（Initialization） ── 执行 <clinit>，赋程序员写的值
术语：连接（Linking） = 验证 + 准备 + 解析，三个阶段合称连接阶段。 JVM 规范要求加载在连接之前，但验证/准备/解析三者之间允许交叉进行。

各阶段详解

① 加载（Loading）

JVM 需要完成三件事：

  1. 通过类的全限定名获取其字节流（可以来自 jar、网络、数据库、动态生成……）
  2. 将字节流所代表的静态存储结构转化为方法区的运行时数据结构
  3. 在 Java 堆中生成一个代表这个类的 java.lang.Class 对象，作为方法区数据的访问入口

关键理解：字节流的来源 JVM 并不限制，这正是自定义 ClassLoader 的切入口—— 你可以从加密 jar、远程 URL、甚至内存里的字节数组里读取字节流。

自定义 ClassLoader 示例（加载阶段入口）Java
public class ByteArrayClassLoader extends ClassLoader {

    private final Map<String, byte[]> classBytes;

    public ByteArrayClassLoader(Map<String, byte[]> classBytes) {
        this.classBytes = classBytes;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = classBytes.get(name);
        if (bytes == null) throw new ClassNotFoundException(name);
        // defineClass 才是真正触发"加载"阶段的 JVM 原生方法
        return defineClass(name, bytes, 0, bytes.length);
    }
}

② 验证（Verification）

字节码来源不受限制，JVM 必须保证"任何人提交的字节码都不能破坏 JVM 自身的安全"。 验证阶段就是这道安全门，分四道检查：

子阶段 检查内容 举例
文件格式验证 魔数、版本号、常量池类型 0xCAFEBABE 开头；版本号不超过当前 JDK
元数据验证 类继承语义合法性 final 类不能被继承；父类必须是 Object 或合法类
字节码验证 指令序列的语义合法性 操作数栈溢出；类型转换不合法（int 赋给引用类型）
符号引用验证 引用的类/方法/字段是否存在且可访问 调用了 private 方法；字段类型不匹配
可以关掉吗？ 可以。生产环境信任 jar 时加 -Xverify:none（JDK 13+ 改为 -XX:-BytecodeVerificationLocal）可跳过验证，加快启动速度，但有安全风险。

③ 准备（Preparation）

为类变量（static 字段）在方法区分配内存，并赋零值—— 注意：不是你代码里写的初始值，而是数据类型的默认值。

数据类型零值
int / short / byte / char0
long0L
float / double0.0
booleanfalse
引用类型null
准备 vs 初始化的分工Java
public class Config {
    // 准备阶段：value = 0（零值）
    // 初始化阶段：value = 100（执行赋值指令）
    static int value = 100;

    // 例外：static final 常量在准备阶段直接赋 42，不等到初始化
    // 因为 ConstantValue 属性在编译期就确定了
    static final int MAX = 42;
}
常见陷阱：static final 的基本类型或 String 字面量， 准备阶段就会赋实际值；但 static final 引用类型（如 static final Object obj = new Object()） 仍要等到初始化阶段，因为 new 是运行时行为。

④ 解析（Resolution）

字节码里的所有调用（类引用、方法引用、字段引用）在编译期都只是符号引用—— 即一个字符串 + 描述符，例如 "com/example/Foo.bar:(I)V"。 运行时内存布局才确定，解析阶段把符号引用替换为直接引用（内存地址/偏移量）。

解析的目标分为七类：

  • 类或接口引用
  • 字段引用
  • 类方法引用
  • 接口方法引用
  • 方法类型（MethodType）
  • 方法句柄（MethodHandle）
  • 调用点限定符（invokedynamic，lambda 底层）
解析可以延迟：JVM 规范允许"在字节码指令执行前再解析"， HotSpot 正是如此——首次执行某条指令时才解析对应的符号引用，这叫延迟解析（Lazy Resolution）。

⑤ 初始化（Initialization）

初始化阶段才真正执行 Java 代码，本质是执行编译器合成的 <clinit> 方法。 它由所有 static 变量赋值语句和静态代码块按源码顺序合并而来。

clinit 顺序演示Java
public class Order {
    static int a = 1;         // clinit 第1步

    static {
        a = 2;                // clinit 第2步
        System.out.println("static block, a=" + a);
    }

    static int b = a * 10;     // clinit 第3步，此时 a=2，所以 b=20

    // 最终：a=2, b=20
}

触发初始化的六种"主动引用"（JVM 规范强制要求）：

  1. 遇到 new、getstatic、putstatic、invokestatic 指令
  2. 使用 java.lang.reflect 对类进行反射调用
  3. 初始化子类时，父类尚未初始化
  4. 虚拟机启动时指定的主类（含 main() 的类）
  5. JDK 7+：java.lang.invoke.MethodHandle 解析结果为 REF_getStatic 等句柄时
  6. JDK 8+：接口定义了 default 方法，其实现类初始化时该接口也会初始化
被动引用不触发初始化（高频考点）
  • 子类访问父类的静态字段：只初始化父类，不初始化子类
  • 通过数组定义引用类：MyClass[] arr = new MyClass[10] 不触发 MyClass 初始化
  • 访问 static final 常量（编译期已内联到调用方字节码）
被动引用反例验证Java
class Parent {
    static int x = 10;
    static { System.out.println("Parent init"); }
}

class Child extends Parent {
    static { System.out.println("Child init"); }
}

// 输出只有 "Parent init"，Child 不会被初始化
System.out.println(Child.x);

<clinit> vs <init>

对比项 <clinit> <init>
对应 Java 结构 static 变量 + static 代码块 实例变量 + 构造器 + 实例初始化块
执行时机 类初始化阶段，每个类只执行一次 每次 new 时执行
线程安全 JVM 保证加锁互斥，多线程安全 需要开发者自己保证
可以没有 是（无 static 变量/块时编译器不生成） 否（至少有默认无参构造器）
父类关系 父类 <clinit> 先执行 父类 <init> 先执行（super()）

初始化的线程安全与死锁陷阱

JVM 保证 <clinit> 在多线程下只执行一次，内部通过类对象上的锁实现。 但如果静态初始化块里有耗时操作或阻塞调用，其他线程会卡住—— 这是 JVM 层面的隐式锁，极难排查。

静态初始化死锁（真实案例）Java
// 反例：两个类互相在 static 块里触发对方初始化
class A {
    static B b = new B();   // 初始化 A 时，触发 B 的初始化
}
class B {
    static A a = new A();   // 初始化 B 时，触发 A 的初始化
}
// 结果：循环依赖 → 两个线程分别持有 A/B 的类锁等待对方，死锁！
最佳实践：static 块中避免调用其他类的方法，尤其是可能触发另一个类初始化的方法。 如确实需要，改用懒加载（Holder 模式或 lazy val）。

完整流程图（文字版）

① 加载：ClassLoader.loadClass() → defineClass() → 堆中 Class 对象 ↓ ② 验证：文件格式 → 元数据 → 字节码 → 符号引用 （抛 VerifyError） ↓ ③ 准备：static 字段 → 零值 （static final 常量直接赋实际值） ↓ ④ 解析：符号引用 → 直接引用 （可延迟到首次使用） ↓ ⑤ 初始化：执行 <clinit>，按源码顺序执行 static 赋值 + static 块 ↓ 类可用（可被 new、被反射调用……）

高频面试考点

Q1：准备阶段赋零值，初始化阶段赋实际值，这两步为什么要分开？

准备阶段在连接期，此时 JVM 还没有执行任何 Java 代码，方法区结构刚建好。 先赋零值可以保证类字段绝不会有"未初始化"的垃圾值，即使初始化阶段因异常中断， 程序也不会读到不可预期的内存内容，这是 JVM 的一种安全保障。

Q2：接口有 <clinit> 吗？

有，但只有在接口中定义了 static final 且赋值不是编译期常量（如 = new Foo()） 或有 default 方法时才会生成 <clinit>。 接口的 <clinit> 执行不要求父接口先初始化（与类不同）。

Q3：Class.forName() 和 ClassLoader.loadClass() 有什么区别？

方法 是否触发初始化 典型使用场景
Class.forName("xxx") 是（默认） JDBC 驱动注册（靠静态块注册到 DriverManager）
Class.forName("xxx", false, cl) 否 只想拿到 Class 对象，不执行初始化
ClassLoader.loadClass("xxx") 否 框架懒加载，Spring Bean 延迟实例化

Q4：static 字段的初始化顺序是怎样的？

父类 <clinit> 先于子类执行；同一类内部， static 变量赋值和 static 代码块按源码从上到下的顺序执行。

Q5：用 JVM 参数观察类加载过程

常用 JVM 参数Shell
# 打印所有被加载的类
java -verbose:class MyApp

# 打印类加载/卸载信息（HotSpot）
java -XX:+TraceClassLoading -XX:+TraceClassUnloading MyApp

# 跳过字节码验证（慎用）
java -Xverify:none MyApp

小结

  • 加载：ClassLoader 读字节流，堆中创建 Class 对象——来源不限
  • 验证：四道门禁，确保字节码安全合法
  • 准备：static 字段赋零值；static final 编译期常量直接赋实际值
  • 解析：符号引用 → 直接引用，HotSpot 采用延迟解析
  • 初始化：执行 <clinit>，JVM 保证线程安全；六种主动引用触发，三种被动引用不触发

与其他知识点的联系

  • 双亲委派模型：加载阶段"获取字节流"这一步由具体的 ClassLoader 完成，而多个 ClassLoader 之间该由谁先尝试加载，正是下一节双亲委派要解决的协作问题——加载流程回答"怎么把字节码变成类"，双亲委派回答"该找谁要字节码"。
  • 可达性分析与 GC Roots：方法区中类的静态属性是 GC Roots 的一种，而静态字段的初始化零值/实际值正是本节"准备"与"初始化"两阶段的产物——理解类加载才能理解为什么 static 字段引用的对象永远可达。
  • 对象内存布局：初始化阶段执行 <clinit> 时才会真正 new 对象，此时对象头、实例数据的内存布局规则才会生效；而"准备"阶段的零值只是在方法区/Metaspace 里占位，还没有涉及堆上对象的创建。
  • 反射与 Spring：反射调用（Class.forName() 默认触发初始化）与依赖注入框架大量依赖类加载的这套生命周期——Spring 懒加载 Bean 本质上就是延迟触发目标类的"主动引用"，避免启动时就执行所有 <clinit>。
  • 单例模式：基于类初始化的单例（如静态内部类 Holder 模式）能保证线程安全，正是因为 JVM 保证 <clinit> 在多线程下只执行一次且互斥——这是比手写 synchronized 双重检查锁更简洁的写法，根源就在本节的初始化线程安全保证。
