JVM · 垃圾回收

07. 可达性分析与 GC Roots

理解 JVM 判断"对象是否存活"的核心算法，以及哪些对象构成 GC 扫描的起点。

为什么不用引用计数？

判断一个对象是否可以被回收，最直觉的方案是引用计数法：每个对象维护一个计数器，有引用 +1，引用失效 -1，计数为 0 即可回收。CPython 就用这种方式。

但它有一个无法回避的致命缺陷——循环引用：

循环引用导致内存泄漏Java
Object a = new Object();
Object b = new Object();

// a 持有 b，b 持有 a
a.ref = b;
b.ref = a;

// 栈变量置 null，外部已无路径能访问到 a / b
a = null;
b = null;

// 引用计数法：a.count=1, b.count=1，永远不会为 0 → 永远不被回收

Java 选择了另一条路：可达性分析（Reachability Analysis）。不问"有没有人引用你"，而问"从根出发，能不能走到你"。走不到的对象，即便彼此引用，也一律判为垃圾。

可达性分析原理

可达性分析本质上是一次图的遍历（DFS / BFS）。把堆内存中所有对象想象成一张有向图，节点是对象，边是引用关系。JVM 选定一组绝对可信的起点——称为 GC Roots——从这些节点出发遍历引用链。遍历结束后：

  • 被访问到的对象 → 存活，不回收
  • 未被访问到的对象 → 不可达，判定为垃圾，可以回收
GC Roots ├── Thread Stack (局部变量 obj1) │ └── Object A ← 可达 │ └── Object B ← 可达 │ └── Static Field (MyClass.cache) └── Object C ← 可达 Object D ──→ Object E （互相引用，但从 GC Roots 无路径到达） Object E ──→ Object D ← 不可达，均可回收
这个设计的本质是：从"程序当前运行时真正能用到的对象"反向定义存活。只要程序的任何执行路径都无法再访问到某对象，它就是死对象，无论引用计数是多少。

GC Roots 有哪些？

GC Roots 是那些不存储在堆上，或生命周期由 JVM 直接管理的引用集合。HotSpot JVM 主要包括以下几类：

GC Root 类型 说明 典型场景
虚拟机栈中的引用 每个线程的栈帧里，局部变量表中的对象引用 方法参数、局部变量 Object obj = new ...
本地方法栈中的引用 JNI（Native 方法）中引用的 Java 对象 native 方法持有的 Java 对象
方法区中类静态属性 类的 static 字段所引用的对象 static List<User> cache = new ArrayList<>()
方法区中常量引用 运行时常量池中的引用（如字符串常量） static final String NAME = "hello"
JVM 内部引用 基本数据类型对应的 Class 对象、常驻异常类、系统类加载器等 NullPointerException、Integer.class
被同步锁持有的对象 synchronized 正在加锁的对象 synchronized(obj) { ... } 期间 obj 不可回收
跨代/跨区引用 老年代引用新生代对象（记录在 Remembered Set） Minor GC 时需扫描老年代对新生代的引用
注意"临时性"：GC Roots 不是固定的，而是 GC 发生那一瞬间的快照。方法返回后，其栈帧里的局部变量自动从 GC Roots 中消失；静态字段赋 null 后，其引用的对象也会离开 GC Roots 体系。

Stop-The-World 与 OopMap

可达性分析有一个隐藏难题：在遍历引用链期间，如果应用线程还在跑，对象引用关系随时在变化——就像你在数一堆移动的小球，数着数着球跑了或者又变多了，根本数不准。

HotSpot 的解法是在安全点（Safepoint）让所有线程暂停（Stop-The-World，STW），确保引用关系静止后再扫描。

为了快速找到所有引用（而不是扫描整个栈/寄存器），JVM 在 JIT 编译时会记录一张叫 OopMap 的表，精确标记"哪个位置存放着对象引用"：

OopMap 的作用示意（概念性）伪代码
// JIT 编译某个方法时，在安全点记录：
// 栈偏移 +8  是一个引用 (Object)
// 栈偏移 +16 是一个引用 (String)
// 寄存器 rbx  是一个引用 (List)

// GC 时直接查表，精确找到所有根引用
// 无需逐字节扫描栈，大幅降低 STW 时间
OopMap 是 HotSpot 能做到精确式 GC（Exact GC）的关键。相比之下，保守式 GC 需要把每个内存位置都当成"可能是引用"来对待，既慢又不能移动对象。

两次标记与 finalize() 的"缓刑"机制

不可达 ≠ 立刻死亡。JVM 给了对象一次"救赎"的机会，整个流程分两次标记：

  1. 第一次标记：可达性分析后，不可达的对象被标记，同时检查它是否有必要执行 finalize()（即是否重写了 finalize 且还没执行过）。
  2. 放入 F-Queue：有必要执行 finalize 的对象放进一个队列，由低优先级的 Finalizer 线程异步执行（JVM 不保证等待其完成）。
  3. 第二次标记：如果对象在 finalize() 中把自己重新挂到了某个 GC Root 可达的引用链上，它就逃过了这次回收。否则，真正死亡。
finalize 自救演示（反例，勿在生产用）Java
public class FinalizeEscape {
    static FinalizeEscape saved;  // 静态字段 = GC Root

    @Override
    protected void finalize() throws Throwable {
        // 把自己挂到 GC Root 可达的链上 → 逃过本轮回收
        FinalizeEscape.saved = this;
    }

    public static void main(String[] args) throws Exception {
        FinalizeEscape obj = new FinalizeEscape();
        obj = null;
        System.gc();
        Thread.sleep(500);
        // 第一次 GC 后 saved != null，对象被救了
        System.out.println(saved != null ? "活了" : "死了");

        saved = null;
        System.gc();
        Thread.sleep(500);
        // 第二次 GC：finalize 已执行过，不再调用 → 对象真正死亡
        System.out.println(saved != null ? "活了" : "死了");
    }
}
finalize() 是过时机制，Java 9 已废弃（@Deprecated），Java 18 标记为 removal。 它执行时机不确定、异常会被忽略、每个对象只会执行一次，极易导致资源泄漏和诡异 bug。资源清理请用 try-with-resources + AutoCloseable，或 Cleaner（Java 9+）。

实际扫描的优化：记忆集与卡表

理论上每次 GC 都要从所有 GC Roots 出发扫描整个堆，代价极高。实际上 JVM 分代收集时，Minor GC 只扫描新生代，但老年代对象也可能引用新生代对象——如果忽略这些引用，会误判新生代对象不可达。

解法是记忆集（Remembered Set）：记录从非收集区域指向收集区域的引用。HotSpot 用卡表（Card Table）实现：

  • 将整个堆按 512 字节分成一张张"卡页"
  • 某老年代卡页中存在跨代引用时，该卡页标记为"脏"（dirty）
  • Minor GC 时只扫描脏卡页，而非全部老年代，大幅降低 GC 成本
G1 GC 将堆拆成等大的 Region，每个 Region 都维护自己的 Remembered Set，记录其他 Region 对自己的引用。这样 G1 可以选择性地只回收部分 Region，进一步控制 STW 时间。

常见陷阱与误区

陷阱 1：以为局部变量超出作用域就会被 GC

代码块退出 ≠ 立刻 GCJava
void method() {
    {
        byte[] chunk = new byte[10 * 1024 * 1024];
        // 代码块结束，chunk 不再可用
    }
    // 但 JIT 优化可能仍保留 slot，不一定马上释放引用
    // 主动 chunk = null 在某些场景下可以帮助 GC 更早回收
    System.gc();  // 也只是"建议"，不保证立刻执行
}

陷阱 2：静态集合持有对象导致内存泄漏

静态字段是 GC Root，其引用链上的所有对象永远可达。把大量临时对象放进静态集合却不清理，是最常见的 Java 内存泄漏模式。

静态集合内存泄漏（反例 → 正例）Java
// 反例：cache 是 GC Root，放进去的对象永远不会被回收
static List<byte[]> cache = new ArrayList<>();
void onRequest() {
    cache.add(new byte[1024]);  // OOM 只是时间问题
}

// 正例：用 WeakReference 或带容量限制的 LRU Cache
static Map<String, WeakReference<byte[]>> safeCache = new HashMap<>();
// GC 时 WeakReference 的 referent 可以被回收，不阻止 GC

陷阱 3：ThreadLocal 未 remove 导致内存泄漏

ThreadLocalMap 的 Key 是 WeakReference（ThreadLocal 对象），但 Value 是强引用。ThreadLocal 被 GC 后，Key 为 null，但 Value 仍无法被回收——在线程池场景下（线程长期存活），这些 Value 永远是 GC Root 可达的，造成泄漏。用完必须调 ThreadLocal.remove()。

面试高频考点

问题关键答案
引用计数法有什么缺点？Java 为什么不用？ 无法解决循环引用。Java 用可达性分析，从 GC Roots 出发遍历引用链，循环引用的孤岛仍会被回收。
哪些对象可以作为 GC Roots？ 虚拟机栈局部变量、本地方法栈 JNI 引用、静态字段、运行时常量池引用、同步锁持有对象、JVM 内部引用、跨代引用（Remembered Set）。
可达性分析为什么需要 STW？ 确保分析期间对象引用关系不变。否则"对象消失"（已访问的对象引用被断开）或"浮动垃圾"（新增不可达对象）会影响正确性。
对象一定要经历两次标记才会被回收吗？ 如果对象没有重写 finalize()（或已执行过），第一次标记后直接回收，不进 F-Queue。只有重写且首次触发才有"两次标记"流程。
Minor GC 时会扫描老年代吗？ 不全扫，只扫描卡表中被标记为脏的卡页（存在跨代引用的老年代区域），以此处理老年代→新生代的引用。
System.gc() 能保证立刻触发 GC 吗？ 不能，只是向 JVM 发出建议。加 -XX:+DisableExplicitGC 后甚至完全无效。生产代码不应依赖 System.gc()。

总结

  • 可达性分析解决了引用计数无法处理循环引用的根本缺陷，是 JVM 判断对象存活的核心算法。
  • GC Roots 是程序"当前真正在用"的对象入口，包括栈帧局部变量、静态字段、JNI 引用等。
  • OopMap 让 JVM 在安全点能精确、高效地找到所有根引用，支撑了精确式 GC 和对象移动。
  • finalize() 提供了一次自救机会，但机制脆弱，已被废弃，不要在生产代码中依赖它。
  • 静态集合、ThreadLocal 未清理是最常见的"对象意外可达"内存泄漏根源。
