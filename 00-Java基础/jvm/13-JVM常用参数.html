13. JVM 常用参数
-Xmx / -Xms / -XX:+UseG1GC

从零理解 JVM 启动参数体系，掌握堆内存配置、GC 策略选择与生产调优套路。

为什么需要手动配置 JVM 参数？

JVM 的设计目标是"一次编写，到处运行"，但它无法预知你的程序会处理多大的数据、用什么 GC 算法最合适。 JVM 参数本质上是你与运行时之间的约定——你告诉 JVM 可以使用多少内存、用哪种垃圾回收器、出问题时怎么记录现场。

没有合理的参数，JVM 只能靠默认值猜测。默认值在开发阶段够用，但在生产环境（几十 GB 内存、几千并发）往往导致：

  • 堆太小 → 频繁 Full GC → 应用停顿，吞吐下降
  • 堆太大但 GC 策略不匹配 → 单次 STW 暂停几秒，接口超时
  • 没有 OOM 转储 → 出问题无法复盘

参数的三类语法

HotSpot 的参数有三种前缀形式，含义不同：

前缀含义示例
-X 非标准参数，HotSpot 专有，各版本稳定 -Xmx4g, -Xms4g, -Xss512k
-XX:+/- 不稳定的高级参数；+ 开启，- 关闭 -XX:+UseG1GC, -XX:-UseBiasedLocking
-XX:key=value 不稳定参数，赋值形式 -XX:MaxGCPauseMillis=200
-X 与 -XX 的区别：-X 参数相对稳定，跨版本不会随便删；-XX 参数属于"实现细节"，官方保留随时修改的权利，但实践中核心参数（G1、GC 日志等）多年未变。

堆内存两兄弟：-Xmx 与 -Xms

它们控制什么？

-Xms（initial heap size）：JVM 启动时向 OS 申请的初始堆大小。
-Xmx（maximum heap size）：堆可以扩展到的最大上限。

JVM 启动后堆的实际大小在 Xms ~ Xmx 之间动态伸缩。当对象分配速度超过当前堆容量时，JVM 会触发 GC；GC 后仍不够用，则扩堆直到 Xmx；再无法分配则抛 OutOfMemoryError: Java heap space。

启动命令示例Shell
# 最常见的生产写法：初始 = 最大，避免运行时扩堆带来的 GC 压力
java -Xms4g -Xmx4g -jar app.jar

# 单位：k / m / g（不区分大小写）
java -Xms512m -Xmx2g -jar app.jar

为什么生产环境要让 Xms = Xmx？

堆扩容本身要触发一次 GC（需要重新整理内存布局），还会导致堆大小不可预测，增加调优难度。 固定堆大小让系统行为更稳定，也更容易做容量规划。

陷阱：设置 Xmx 超过物理内存
JVM 的堆之外还有元空间（Metaspace）、线程栈、直接内存（NIO）、JVM 本身代码等开销。 通常建议 -Xmx 设为可用物理内存的 60%~75%，为非堆区域留出余量。 容器化场景要格外注意：JDK 8u191+ / JDK 10+ 才正确感知 cgroup 内存限制，否则 JVM 会读到宿主机物理内存，导致 OOM Killed。

线程栈大小：-Xss

每个线程都有独立的栈帧空间，-Xss 控制单个线程的栈大小。默认值随平台不同（Linux x64 通常 512k，Windows 默认 256k）。

  • 栈太小 → 深递归时抛 StackOverflowError
  • 栈太大 × 线程数多 → 整体内存占用暴增（1000 线程 × 1MB = 1GB）
  • 微服务 / 线程池场景通常设 -Xss256k 或 -Xss512k 即可

G1 的核心思想：Region 化

传统 CMS/Parallel 把堆划分为固定的新生代（Young）和老年代（Old）。 G1 把堆切成若干大小相等的 Region（默认 1~32MB），每个 Region 动态扮演 Eden / Survivor / Old / Humongous 角色。 GC 时优先回收"垃圾最多、回收收益最高"的 Region，这就是名字 Garbage-First 的由来。

堆（Heap） ├── Region 0 [Eden] ├── Region 1 [Eden] ├── Region 2 [Survivor] ├── Region 3 [Old] ├── Region 4 [Old] ├── Region 5 [Humongous] ← 大对象（> Region 的 50%） └── ...（可动态重新角色分配）

G1 通过 Remembered Set（RSet）记录跨 Region 引用，回收某个 Region 时无需扫描全堆，从而控制单次 GC 的暂停时间。

G1 的配套参数

G1 典型配置JVM Args
-XX:+UseG1GC                    # 启用 G1 收集器（JDK 9+ 已是默认）
-XX:MaxGCPauseMillis=200         # 期望最大 STW 暂停（软目标，G1 尽力达到）
-XX:G1HeapRegionSize=16m         # Region 大小，必须是 2 的幂（1~32m）
-XX:G1NewSizePercent=20          # 新生代最小占堆比例（默认 5%）
-XX:G1MaxNewSizePercent=40       # 新生代最大占堆比例（默认 60%）
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用达到多少比例触发并发标记（默认 45）
MaxGCPauseMillis 不是硬限制
这是给 G1 的"目标"，G1 会据此动态调整 Young 区大小和混合 GC 的 Region 数量。 设得太小（如 50ms）反而可能让 G1 频繁触发 GC，降低吞吐。200ms 是大多数 Web 服务的合理起点。

GC 日志：排查问题的眼睛

光配了参数还不够，必须把 GC 日志打开，否则出问题无从分析。

JDK 9+ 统一日志（Unified Logging）JVM Args

# gc*        → 输出所有 GC 相关子系统日志
# file=...   → 输出到文件
# time       → 打印时间戳
# filecount  → 最多保留 5 个滚动文件
# filesize   → 每个文件 20MB

OOM 诊断三件套

出现 OOM 时，堆转储（Heap Dump）是最重要的案发现场证据，务必提前配好。

OOM 诊断参数JVM Args
-XX:+HeapDumpOnOutOfMemoryError          # OOM 时自动 dump 堆
-XX:HeapDumpPath=/var/log/heap-dump.hprof # dump 文件路径
-XX:+ExitOnOutOfMemoryError              # OOM 后直接退出（配合容器健康检查重启）
陷阱：没有配 HeapDumpOnOutOfMemoryError
线上出现 OOM 后进程崩溃，重启后现场销毁，只剩一行"java.lang.OutOfMemoryError"，无法定位内存泄漏点。 这是最常见的生产事故复盘困难原因之一。

完整生产启动模板（Spring Boot 示例）

生产级 JVM 参数（4c8g 容器）Shell
java \
  # ── 堆内存（固定，避免扩缩堆触发 GC）──
  -Xms4g -Xmx4g \
  # ── 线程栈（Web 服务足够用）──
  -Xss256k \
  # ── GC 策略 ──
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:InitiatingHeapOccupancyPercent=40 \
  # ── 元空间（防止 Metaspace 无限增长）──
  -XX:MetaspaceSize=256m \
  -XX:MaxMetaspaceSize=512m \
  # ── GC 日志（JDK 11+）──
  -Xlog:gc*:file=/app/logs/gc.log:time,uptime:filecount=5,filesize=20m \
  # ── OOM 诊断 ──
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/app/logs/heap.hprof \
  -XX:+ExitOnOutOfMemoryError \
  # ── 容器感知（JDK 8u191+ 必须显式加）──
  -XX:+UseContainerSupport \
  -jar app.jar

元空间（Metaspace）与 PermGen 的区别

JDK 7 及以前有永久代（PermGen），存储类的元数据。PermGen 是堆的一部分，大小固定，容易出现 OutOfMemoryError: PermGen space。

JDK 8 彻底移除 PermGen，改用元空间（Metaspace），存在于本地内存（Native Memory）而非堆内。 默认无上限，理论上只要 OS 有内存就能增长——这反而是风险：类加载泄漏会悄悄耗尽系统内存。 因此生产环境必须设 -XX:MaxMetaspaceSize。

对比项PermGen（JDK ≤7）Metaspace（JDK 8+）
位置堆内（受 -Xmx 约束）本地内存（独立于堆）
默认上限64~85MB，固定无上限（受 OS 限制）
GC 管理Full GC 时回收GC 触发时回收
配置参数-XX:MaxPermSize-XX:MaxMetaspaceSize

常见陷阱汇总

  • Xmx 设得接近物理内存上限：非堆区域（直接内存、Metaspace、栈）没有空间，导致进程被 OOM Killer 杀死，而不是 JVM 内部的 OOM。
  • 容器里忘加 -XX:+UseContainerSupport（JDK 8u191 以下）：JVM 读到宿主机内存，以为自己有 128GB，堆设到 96GB，实际 cgroup 限制 4GB，直接被杀。
  • Xms 远小于 Xmx 用于"节省内存"：启动后流量一来堆频繁扩容，每次扩容都触发 GC，早高峰响应时间飙升。
  • JDK 8 环境加了 JDK 9+ 的 -Xlog 参数：JVM 无法识别直接报错退出，而不是忽略。要用 -XX:+PrintGCDetails 等旧参数。
  • MaxGCPauseMillis 设成 50ms 后吞吐骤降：G1 为了满足低暂停目标，把 Young 区压缩得很小，GC 频率大幅提高，CPU 大量消耗在 GC 上。

面试高频考点

Q：-Xmx 和 -Xms 建议设成一样吗？为什么？

生产环境建议相同。原因：堆扩容时需要触发一次 GC 来保证内存连续性，会引入额外停顿；固定堆大小让 GC 行为更可预测，便于调优基线对比。 开发/测试环境不必相同，让 JVM 按需分配可以节省资源。

Q：G1GC 和 CMS 的主要区别？

CMS 只处理老年代，与 ParNew（新生代）配合使用，两块区域分别管理；G1 统一管理整个堆，用 Region 化替代固定代划分，可以同时控制新生代和老年代的回收粒度，并且提供可预期的暂停时间目标，而 CMS 无法直接设置暂停目标。 CMS 在 JDK 9 被标记废弃，JDK 14 完全移除。

Q：什么情况下考虑换用 ZGC？

当 G1 的 GC 暂停（即使调优后）仍然超过 100ms，且业务对延迟极度敏感（如实时交易、游戏服务器）时考虑 ZGC。 ZGC 的并发 GC 设计可以将暂停控制在 1~10ms，但吞吐略低于 G1，且需要 JDK 11+（生产可用建议 JDK 15+）。

Q：Metaspace OOM 怎么排查？

常见原因是动态类加载泄漏（如 JSP 热重载、反射生成代理类没有使用自定义 ClassLoader 并及时卸载）。 排查方法：jcmd <pid> VM.native_memory 查看 Metaspace 使用量，配合 MAT / JProfiler 分析 ClassLoader 数量，定位哪个 ClassLoader 加载了大量类而未被回收。
