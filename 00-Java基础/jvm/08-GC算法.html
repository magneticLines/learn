JVM · 垃圾回收

08. 常用 GC 算法

标记-清除、复制、标记-整理——三种基础算法各自解决什么问题，又带来哪些新问题？

为什么需要专门设计 GC 算法？

上一节的可达性分析告诉 JVM「谁该死」，但还没回答「怎么死、怎么收尸」。回收内存看似简单：把对象删掉就好。但真实堆内存有几 GB，对象几百万个，删掉之后还要保证：

  • 空间能被复用——不能出现越回收越碎的情况
  • 分配足够快——新对象分配是高频操作，慢一点点都影响吞吐
  • STW 尽量短——Stop-The-World 期间所有业务线程暂停，用户感知到卡顿

三种基础算法分别在不同维度上做出了取舍，现代 GC（G1、ZGC）都是在这三种算法之上组合演进的。

算法一：标记-清除（Mark-Sweep）

原理

分两个阶段：

  1. 标记阶段：从 GC Roots 出发，遍历对象图，给所有可达对象打上「存活」标记。
  2. 清除阶段：线性扫描整个堆，把没有标记的对象所占内存归还到空闲列表（Free List）。

优点

  • 实现简单，不需要移动对象，引用地址不变
  • 存活对象多时（老年代场景）不需要大量复制

缺点：内存碎片

清除后形成大量不连续的碎片。下次分配一个大对象时，即使总空闲量足够，也可能找不到连续的内存块，被迫提前触发 GC 甚至 OOM。

陷阱：内存碎片不是「浪费」，而是「不可用」。空闲列表里有 100 个 16 字节的碎片 = 1600 字节，但分配 2000 字节的数组会直接失败。这就是为什么 CMS（使用标记-清除）在运行一段时间后需要做一次 Full GC 来整理碎片。

算法二：复制（Copying）

原理

把堆内存一分为二：From 区（当前使用）和 To 区（保持空白）。GC 时将 From 中所有存活对象依次复制到 To 区，复制完成后直接清空整个 From 区，然后交换 From/To 角色。

优点

  • 无碎片：存活对象紧凑排列，新对象分配只需移动一个指针（bump-the-pointer），极快
  • GC 开销与存活对象成正比：死亡对象越多，GC 越快（新生代正好符合：大多数对象朝生夕死）

缺点：空间浪费 50%

任何时刻只有一半内存可用。这在老年代是不可接受的——存活对象占比高，复制代价巨大且空间浪费严重。

HotSpot 的改进：Eden + Survivor

HotSpot 针对新生代的「大多数对象短命」特性做了优化，不是五五开，而是按 8:1:1 划分 Eden、Survivor From、Survivor To 三个区。每次只有 10% 的空间闲置，空间利用率提升到 90%。

新生代内存布局（默认比例）
// -XX:SurvivorRatio=8  （默认值）
// 新生代总大小 = Eden(80%) + S0(10%) + S1(10%)
//
// 分配流程：
// 1. 新对象优先进 Eden
// 2. Eden 满 → Minor GC：存活对象复制到 S1，Eden+S0 清空
// 3. 年龄 >= 15（默认）→ 晋升老年代
// 4. S1 放不下的对象直接晋升老年代（动态年龄判断）

new Object()  // → Eden
// Minor GC 后存活 → Survivor（年龄+1）
// 若干次 Minor GC → 老年代
为什么新生代特别适合复制算法？
IBM 研究表明，98% 的 Java 对象在第一次 Minor GC 前就死亡。存活率极低，意味着每次 GC 只需复制很少的对象，速度极快。这是复制算法在新生代大放异彩的根本原因。

算法三：标记-整理（Mark-Compact）

原理

前半段与标记-清除相同（标记存活对象），但清理阶段不是「就地清除」，而是将所有存活对象向一端滑动，然后清理掉边界另一侧的所有内存。

优点

  • 无碎片：整理后内存连续，新对象分配同样可用 bump-the-pointer
  • 不需要额外空间：在原区域内移动，不像复制算法需要 50% 备用区

缺点：移动对象代价高

移动存活对象后，所有指向这些对象的引用（栈帧里的局部变量、其他对象的字段等）都必须同步更新为新地址。这需要再次扫描整个堆来修正引用，STW 时间比标记-清除更长，适合老年代的 Full GC（低频、彻底）。

为什么老年代适合标记-整理？
老年代存活率高（大量长生命周期对象），用复制算法要复制大量对象且浪费空间；用标记-清除会产生碎片导致大对象分配失败。标记-整理是唯一既不浪费空间、又不产生碎片的选择，代价是 STW 时间稍长，但 Full GC 本就是低频操作，可以接受。

分代收集：算法的组合使用

没有任何一种算法能在所有场景下最优。HotSpot 的解法是分代收集（Generational Collection）：按对象生命周期将堆划分为新生代和老年代，对不同区域使用不同算法。

堆 ├── 新生代（Young Generation） → 使用 复制算法 │ ├── Eden（80%） │ ├── Survivor From（10%） │ └── Survivor To（10%） └── 老年代（Old Generation） → 使用 标记-整理 或 标记-清除

分代假设（Generational Hypothesis）是分代收集的理论基石：大多数对象都是朝生夕死的，少数对象会存活很长时间。这个假设在 Java 应用中高度成立，使得复制算法在新生代能以极低代价完成 GC。

跨代引用：写屏障与 Card Table

分代收集引入了新问题：老年代对象引用新生代对象时，Minor GC 必须扫描老年代才能找到所有 GC Roots，这完全抵消了分代带来的好处。

HotSpot 的解法是 Card Table + 写屏障（Write Barrier）：

  • 把老年代堆内存按 512 字节分为一个个「卡片（Card）」
  • 任何写操作修改老年代字段时，写屏障把对应 Card 标记为「脏」（dirty）
  • Minor GC 只需扫描脏 Card，而非整个老年代
写屏障伪代码（JVM 内部逻辑，非 Java 代码）
// 每次执行 obj.field = ref 时，JIT 插入如下逻辑：
void writeBarrier(Object obj, Object ref) {
    obj.field = ref;          // 真正的写操作

    // 写屏障：如果 obj 在老年代，标记对应 Card 为脏
    if (isOldGen(obj)) {
        cardTable[addressOf(obj) / 512] = DIRTY;
    }
}

陷阱与面试高频考点

陷阱 1：复制算法「浪费一半内存」是误解

HotSpot 新生代用 8:1:1 的 Eden+S0+S1 方案，实际浪费仅 10%（S0 或 S1 中始终有一个闲置）。只有在极端情况下（Survivor 放不下），才会借用老年代空间（Handle Promotion）。

陷阱 2：标记-整理比标记-清除慢不是绝对的

标记-整理修正引用确实需要额外时间，但它消除了碎片，使后续分配更快，还减少了未来 GC 次数（不会因碎片导致提前触发 GC）。单次 GC 慢，但整体吞吐可能更高。

陷阱 3：GC 算法 ≠ 垃圾收集器

算法是思想，收集器是实现。一个收集器可以组合多种算法：G1 对年轻代用复制，对老年代用标记-整理；CMS 的并发清除阶段用标记-清除，但 Full GC 退化时用标记-整理。

面试高频题汇总
  • 标记-清除的缺点是什么？为什么会产生碎片？
  • 复制算法为什么适合新生代？新生代为什么是 8:1:1 而不是 1:1？
  • 标记-整理为什么比标记-清除 STW 更长？
  • 分代收集的理论基础是什么？新生代和老年代分别用什么算法？
  • 什么是 Card Table？它解决了什么问题？
  • Minor GC 和 Full GC 的区别？什么时候触发 Full GC？

Minor GC vs Full GC 触发条件

GC 类型 触发条件 算法 STW 时长
Minor GC Eden 区满 复制算法 短（毫秒级）
Major GC 老年代空间不足 标记-整理 / 标记-清除 长（通常是 Minor 的 10x）
Full GC 老年代满、显式 System.gc()、元空间满、空间担保失败 整堆收集 最长，影响最大
生产建议：避免在代码中调用 System.gc()——它仅是建议，但在大多数 JVM 实现中会触发 Full GC，造成长时间 STW。可以用 -XX:+DisableExplicitGC 参数禁止显式 GC。
