06. 引用类型（强 / 软 / 弱 / 虚）

JDK 1.2 引入四种引用强度，让开发者能精细控制对象的生命周期，是实现缓存、监听器、资源管理的核心机制。

为什么需要四种引用？

在 JDK 1.2 之前，Java 世界只有一种引用：要么引用着对象，要么为 null。 这带来了一个两难问题：

  • 缓存场景：对象放进缓存后，GC 永远不会回收它，内存越用越多。
  • 监听器 / 回调场景：注册后忘记注销，持有大量业务对象，造成内存泄漏。

解决办法是让引用本身具有"强度"，当内存紧张时，GC 可以按照引用强度从弱到强依次回收， 开发者只需选择合适的引用类型，剩下的交给 JVM。

类比：四种引用就像四种租约强度——强引用是"永久产权"，软引用是"到期前不驱逐"， 弱引用是"下次扫房就清走"，虚引用是"已经空壳，只剩通知"。

四种引用速览

类型 类 GC 回收时机 典型用途 get() 返回
强引用 直接赋值，无包装类 永不回收（只要可达） 普通对象 —
软引用 SoftReference<T> 内存不足时回收 内存敏感缓存 对象 or null
弱引用 WeakReference<T> 下次 GC 必回收 WeakHashMap、ThreadLocal 对象 or null
虚引用 PhantomReference<T> 对象已终结后通知 堆外内存释放、资源清理 始终 null

强引用（Strong Reference）

最普通的赋值就是强引用。只要 GC Roots 能沿引用链到达该对象，GC 绝不会回收它， 哪怕抛出 OutOfMemoryError。

StrongRef.java
Object obj = new Object();  // 强引用，GC 不会碰它
obj = null;               // 手动断开，才有资格被回收
陷阱：方法内的局部强引用随栈帧出栈自动失效；但静态字段、集合中的元素不会自动清除， 是内存泄漏的高发地。

软引用（SoftReference）

软引用的语义是："内存够就留着，快 OOM 了再回收"。 JVM 会在抛出 OutOfMemoryError 之前清除所有软引用对象，因此不会导致 OOM。

典型用法：图片 / 数据缓存

SoftCache.java
Map<String, SoftReference<byte[]>> cache = new HashMap<>();

// 放入缓存
byte[] data = loadFromDisk("logo.png");
cache.put("logo", new SoftReference<>(data));

// 读取缓存（必须判 null，因为 GC 可能已回收）
SoftReference<byte[]> ref = cache.get("logo");
byte[] img = ref != null ? ref.get() : null;
if (img == null) {
    // 缓存失效，重新加载
    img = loadFromDisk("logo.png");
    cache.put("logo", new SoftReference<>(img));
}
配合 ReferenceQueue 可以监听软引用被回收的时机，及时从 Map 中移除"空壳" Key， 避免 Map 本身无限增长（Key 还在，Value 已变 null）。

弱引用（WeakReference）

弱引用比软引用更"脆"：只要 GC 运行，不管内存是否充足，都会回收只被弱引用持有的对象。 它的核心价值不是缓存，而是"不阻止对象被回收"。

WeakHashMap 原理

WeakHashMap 的 Key 就是弱引用。当外部没有强引用持有某个 Key 时， GC 会回收该 Key，对应的 Entry 也在下次操作时被清除，从而自动避免内存泄漏。

ThreadLocal 内存泄漏剖析

ThreadLocal 内部结构（简化）
// ThreadLocalMap 的 Entry 继承 WeakReference
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;          // value 是强引用！
    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}

Key（ThreadLocal 实例）是弱引用，GC 后会变 null；但 value 是强引用， 只要线程存活（线程池中线程不会死），value 就一直被持有。 这就是ThreadLocal 内存泄漏的根本原因。

使用 ThreadLocal 后务必调用 remove()，否则在线程池环境下会造成内存泄漏， 甚至出现"数据串行"（上一个请求的数据泄漏给下一个请求）。

弱引用完整示例

WeakRefDemo.java
Object strong = new Object();
WeakReference<Object> weak = new WeakReference<>(strong);

System.out.println(weak.get());  // 非 null，强引用还在

strong = null;                   // 断开强引用
System.gc();                    // 触发 GC

System.out.println(weak.get());  // null，已被回收

虚引用（PhantomReference）

虚引用是最"弱"的引用，get() 永远返回 null。 它的唯一作用是：当对象被回收后，往 ReferenceQueue 里放一条通知， 让你有机会做清理工作。

这正是 JDK 9 之后 Cleaner 的实现原理，也是 DirectByteBuffer 释放堆外内存的机制。

虚引用 + ReferenceQueue 示例

PhantomRefDemo.java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
Object obj = new Object();

// 创建虚引用时必须关联 ReferenceQueue，否则毫无意义
PhantomReference<Object> phantom = new PhantomReference<>(obj, queue);

System.out.println(phantom.get());  // 始终 null

obj = null;
System.gc();

// 对象回收后，虚引用入队
Reference<?> ref = queue.poll();
if (ref != null) {
    // 在此处释放堆外内存、关闭文件句柄等资源
    System.out.println("对象已回收，执行清理");
}
为什么不用 finalize()？ finalize() 执行时机不确定、会让对象"复活"一次（延迟回收）、也不能保证一定执行。 虚引用 + ReferenceQueue 是更可靠的资源清理替代方案，JDK 9 的 java.lang.ref.Cleaner 正是基于此实现的。

ReferenceQueue：引用回收通知机制

软引用、弱引用、虚引用都可以在构造时传入一个 ReferenceQueue。 当对象被回收后，GC 会把对应的 Reference 对象本身加入该队列，供业务代码轮询。

引用对象被回收的时序 ┌──────────────────────────────────────────────────┐ │ 1. GC 判定对象仅剩弱/软/虚引用可达 │ │ 2. 清除引用（referent 置 null） │ │ 3. 将 Reference 对象本身入队 ReferenceQueue │ │ 4. 业务代码 queue.poll() / queue.remove() 感知 │ └──────────────────────────────────────────────────┘

对比与选型

场景 推荐引用类型 理由
普通对象持有 强引用 默认，无需特殊处理
内存敏感缓存（图片、字节数组） SoftReference OOM 前自动清除，比手动 LRU 更安全
防止内存泄漏的 Map（Key 生命周期由外部控制） WeakReference / WeakHashMap Key 失效后自动清除 Entry
线程本地变量 ThreadLocal（内部用弱引用） 记得 remove()
堆外资源清理（NIO、Native 内存） PhantomReference / Cleaner finalize 的替代品，时机更可控

常见陷阱

  • 软引用缓存 + Map 不清理：软引用被回收后，Map 的 Key 还在，Value 变 null， Map 本身不断增长。解决：配合 ReferenceQueue 定期清理。
  • 弱引用 get() 结果不稳定：两次调用 get() 之间可能发生 GC， 第一次非 null，第二次就是 null。应将 get() 结果赋给局部强引用变量再使用。
  • 虚引用不传 ReferenceQueue：这样的虚引用毫无意义，get() 永远 null， 也不会有任何通知。
  • SoftReference 回收时机因 JVM 而异：HotSpot 的策略是： 软引用上次访问时间距当前时间超过某个阈值（与 -Xmx 剩余量相关），才会被回收。 不同 GC（G1 / ZGC）策略有差异，不能硬依赖"一定会保留"。

面试高频考点

  1. 四种引用强度顺序及回收时机？ 强 > 软 > 弱 > 虚；强引用不回收，软引用 OOM 前回收，弱引用 GC 时回收，虚引用已回收后通知。
  2. WeakHashMap 的应用场景和原理？ Key 是弱引用，GC 后 Key 被清除，配合 expungeStaleEntries() 清理 null Key 的 Entry， 适合"Key 生命周期由外部决定"的缓存。
  3. ThreadLocal 为什么会内存泄漏，如何避免？ Key 弱引用 + Value 强引用，线程不死则 Value 不回收。使用完后必须 remove()。
  4. 虚引用 get() 为何永远返回 null？ 设计如此——虚引用不是为了访问对象，只是为了接收回收通知，强迫你通过 ReferenceQueue 操作， 避免意外"复活"对象（类似 finalize 问题）。
  5. finalize() vs PhantomReference，哪个更好？ PhantomReference 更好：时机更可控，不会让对象复活，不影响 GC 停顿时间。 finalize() 已在 JDK 9 被标注为 Deprecated，JDK 18 起强烈不推荐。
