jstack / jmap / jstat 使用

JDK 自带的三把"手术刀"——不依赖任何第三方工具，在生产环境直接诊断线程死锁、内存泄漏和 GC 压力。

为什么需要这三个工具

线上出故障时，大多数情况下你既没有 IDE，也不能随便重启。JDK 从很早期就内置了一套诊断命令，它们通过 JVM 暴露的 Attach API 与目标进程通信，做到"不停机、不侵入、零依赖"。

三个工具各司其职：

  • jstack：拍下所有线程的快照（Thread Dump），定位死锁、CPU 飙高的线程
  • jmap：查看堆内存分布或导出堆转储文件（Heap Dump），定位内存泄漏
  • jstat：持续采样 GC、类加载等统计数据，感知 GC 压力趋势
三个工具都需要 PID（进程号）。先用 jps -l 找到目标 Java 进程，再传给对应命令。

jstack —— 线程快照

设计动机

Java 线程出问题时，问题的本质是"某个线程在做什么"——是在等锁？在死循环？在 I/O 阻塞？ jstack 把 JVM 内部所有线程的调用栈 + 锁状态一次性打印出来，就像给程序拍了一张 X 光片。

基础用法

Shell查找 PID 并打印线程快照
# 找到目标进程 PID
jps -l
# 输出示例：
# 12345 com.example.MyApp

# 打印线程快照（标准输出）
jstack 12345

# 强制模式：即使 JVM 处于挂起状态也尽量输出（谨慎使用）
jstack -F 12345

# 保存到文件，方便后续分析
jstack 12345 > thread-dump-$(date +%H%M%S).txt

死锁场景复现与分析

下面是一个经典死锁：线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1。

JavaDeadlockDemo.java
public class DeadlockDemo {
    static final Object lock1 = new Object();
    static final Object lock2 = new Object();

    public static void main(String[] args) {
        // 线程 A：先拿 lock1，再拿 lock2
        new Thread(() -> {
            synchronized (lock1) {
                sleep(100);
                synchronized (lock2) { }
            }
        }, "Thread-A").start();

        // 线程 B：先拿 lock2，再拿 lock1 —— 顺序相反，必然死锁
        new Thread(() -> {
            synchronized (lock2) {
                sleep(100);
                synchronized (lock1) { }
            }
        }, "Thread-B").start();
    }
}

运行后执行 jstack <pid>，输出末尾会看到：

Thread Dump 片段jstack 自动识别死锁
Found one Java-level deadlock:
=============================
"Thread-A":
  waiting to lock monitor 0x...（lock2）
  which is held by "Thread-B"

"Thread-B":
  waiting to lock monitor 0x...（lock1）
  which is held by "Thread-A"

Java stack information for the threads listed above:
"Thread-A":
  at DeadlockDemo.lambda$main$0(DeadlockDemo.java:9)
  - waiting to lock <0x...> (lock2)
  - locked <0x...> (lock1)

CPU 飙高排查套路（高频面试题）

步骤如下：

  1. 用 top 找到 CPU 最高的 Java 进程 PID
  2. top -H -p <pid> 找到 CPU 最高的线程 TID（十进制）
  3. 把 TID 转为十六进制：printf "%x\n" <tid>
  4. jstack <pid> | grep -A 30 "<十六进制>" 找到该线程的调用栈
jstack 输出里每个线程头部的 nid=0x... 就是该线程在操作系统层面的 TID（十六进制）。通过它可以精确对应 top 里看到的线程。

jmap —— 堆内存分析

设计动机

内存泄漏的本质是"对象不断增长但永远不被回收"。jmap 让你看清楚堆里到底装了多少个什么对象， 或者把整个堆序列化成文件，用 MAT / VisualVM 等工具离线分析引用链，找到泄漏根源。

常用命令

命令作用说明
jmap -heap <pid> 打印堆内存配置与使用情况 各区域容量、已用、GC 算法
jmap -histo <pid> 对象实例数量统计（轻量） 不 STW，适合快速扫一眼
jmap -histo:live <pid> 只统计存活对象（先触发 Full GC） 会 STW，生产慎用
jmap -dump:format=b,file=heap.hprof <pid> 导出堆转储文件 文件大，会 STW，配合 MAT 分析
Shell典型 jmap 使用流程
# 快速看一下堆的使用率和 GC 算法
jmap -heap 12345

# 看实例数量排行（按字节数降序），快速锁定嫌疑类
jmap -histo 12345 | head -30

# 导出 Heap Dump（生产环境建议加 live，减少文件体积）
jmap -dump:live,format=b,file=/tmp/heap.hprof 12345

# 用 MAT（Eclipse Memory Analyzer）打开 heap.hprof 做引用链分析

-histo 输出解读

输出示例jmap -histo
 num     #instances          #bytes  class name
----------------------------------------------
   1:        580312        46424960  [B          ← byte[]，常见于字符串底层
   2:        521000        41680000  java.lang.String
   3:         98765        15802400  com.example.UserSession  ← 业务类排前面需警惕

关注点：业务类（你自己写的类）出现在前几名，往往就是泄漏源。 [B 代表 byte 数组，[C 代表 char 数组，[Ljava.lang.Object; 代表 Object 数组。

生产环境陷阱：jmap -dump 会导致 STW（Stop The World），堆越大停顿越长。8 GB 堆可能停顿 10–30 秒。 建议通过 JVM 参数 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/ 让 JVM 在 OOM 时自动 dump，避免手动触发。

jstat —— GC 统计监控

设计动机

GC 问题往往是趋势性的——不是某一刻的瞬间值，而是"老年代以每秒 10 MB 的速度增长，1 小时后必然 Full GC"。 jstat 以固定间隔采样，帮你看清楚这条趋势线。

常用选项

选项含义
-gc各代容量、已用空间、GC 次数和耗时
-gcutil各代使用率（百分比），更直观
-gccausegcutil + 上次 GC 触发原因
-gcnew只看新生代（Eden/S0/S1）
-gcold只看老年代
-class类加载数量统计
Shell每 1 秒打印一次 GC 概览，共 20 次
# 语法：jstat -<option> <pid> [间隔ms] [次数]
jstat -gcutil 12345 1000 20

gcutil 输出解读

输出示例jstat -gcutil（列含义注释）
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
  0.00  42.31  78.54  63.20  95.81  90.12    128     2.341     3    1.023     3.364

  S0/S1  = Survivor 0/1 使用率（%）
  E      = Eden 使用率（%）
  O      = Old（老年代）使用率（%）  ← 持续涨 = 内存泄漏警报
  M      = Metaspace 使用率（%）
  YGC    = Young GC 总次数
  YGCT   = Young GC 总耗时（秒）
  FGC    = Full GC 总次数           ← 频繁 = 严重问题
  FGCT   = Full GC 总耗时（秒）
  GCT    = 所有 GC 总耗时（秒）

关键判断规则

  • O 列持续上升，不回落：老年代内存泄漏，用 jmap 进一步分析
  • YGC 频率极高（几秒一次）：新生代太小或对象分配速率过快
  • FGC 次数增加：严重信号，每次 Full GC 都会 STW，影响响应时间
  • M 列接近 100%：Metaspace 不足，可能类加载器泄漏（动态代理/反射过多）
实际排查时，连续采样 5–10 分钟，把输出重定向到文件，再绘图观察趋势，比看瞬间值有效得多。

三工具协作：实战排查流程

下面是一个完整的"内存持续增长 → OOM"排查路径：

Step 1 jps -l 找到目标进程 PID Step 2 jstat -gcutil <pid> 2000 30 观察老年代 O 列是否持续上涨 Step 3 jmap -histo <pid> | head -20 快速看对象数量排行，锁定嫌疑类 Step 4 jmap -dump:live,format=b,file=/tmp/heap.hprof <pid> 导出存活对象堆转储 Step 5 MAT / VisualVM 打开 heap.hprof → Dominator Tree 找到大对象 → Path to GC Roots 分析为何无法回收 Step 6 jstack <pid> 同时检查有无线程积压/死锁导致对象持有

常见陷阱

陷阱 1：在容器/Docker 里找不到进程

容器内运行的 Java 进程，宿主机用 jps 可能看不到，或看到但无法 attach。 原因：Attach API 依赖 /tmp/.java_pidXXX 文件，容器有独立 PID namespace。 解法：在容器内执行 jstack/jmap，或使用 nsenter 进入容器的 namespace。

陷阱 2：jmap -dump 卡死生产

对大堆执行 dump 会触发 STW，期间所有请求超时。强烈建议提前配置 -XX:+HeapDumpOnOutOfMemoryError，让 JVM 在 OOM 时自动保存， 而不是手动在高峰期执行。

陷阱 3：jstack 输出的线程状态误读

线程状态 WAITING 不代表有问题，线程池的 worker 线程在没有任务时就是 WAITING。 真正需要关注的是 BLOCKED（等锁）和 RUNNABLE 但 CPU 飙高的线程。

陷阱 4：jstat 数据 "看起来正常" 但 GC 日志说有问题

jstat 是采样的，两次采样之间可能发生了多次 GC。GC 次数的增量才有意义， 瞬时快照的 O 列值只是一个参考。建议同时开启 GC 日志（-Xlog:gc*）作为更精确的数据源。

面试高频考点

问题核心答案
线上 CPU 100%，如何排查？ top → top -H -p 找线程 TID → printf 转十六进制 → jstack grep nid 找调用栈
如何判断内存泄漏？ jstat -gcutil 观察 O 列持续上涨不回落；Full GC 后 O 列仍高
jmap -histo 与 -dump 的区别？ histo 只统计计数，轻量；dump 导出全量对象图，重量级、会 STW
如何不重启获取 OOM 时的 Heap Dump？ 启动参数加 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/
jstack 能检测到 ReentrantLock 死锁吗？ 能，但需要线程调用了 lock() 并在 await，jstack 会显示 LockSupport.park 调用栈；synchronized 死锁会额外打印 "Found one Java-level deadlock"
jstat -gcutil 中 YGC 很高但 FGC 为 0，说明什么？ 对象晋升老年代速度慢，新生代回收及时，整体健康；但若 YGC 过于频繁（每秒多次），需考虑扩大新生代

与其他工具对比

工具优势劣势适合场景
jstack / jmap / jstat JDK 自带，无依赖，生产直接用 命令行，分析能力有限 线上快速诊断
VisualVM 图形界面，可远程连接，集成分析 需要 JMX 端口开放，有安全风险 开发/测试环境
Arthas 功能更强（热更新、方法追踪），交互式 需要额外部署 jar 线上深度诊断
MAT (Eclipse) Heap Dump 分析极强，能找泄漏根因 只能离线分析 hprof 文件 内存泄漏根因分析
经验法则：快速判断用 jstack/jstat，根因挖掘用 jmap + MAT，日常监控用 Prometheus + JVM Exporter，在线动态调试用 Arthas。

与其他知识点的联系

  • OOM 排查：本文三个工具是上一篇"常见 OOM 场景"的实操落地——jstat 发现老年代持续增长，jmap 定位到具体嫌疑类，二者串联起从"感知问题"到"定位根因"的完整链路。
  • JVM 常用参数：jmap -dump 会触发 STW，等价于手动执行一次本该由 -XX:+HeapDumpOnOutOfMemoryError 自动完成的动作；提前配好该参数可以避免生产环境手动执行 jmap 带来的额外风险。
  • 并发编程：jstack 输出中 BLOCKED/WAITING 状态和锁信息，是理解 synchronized、ReentrantLock 底层监视器（monitor）行为的直观入口，死锁案例正是 AQS/synchronized 竞争的真实现场。
  • GC 算法：jstat -gcutil 的 S0/S1/E/O 列直接对应新生代-老年代分代模型，读懂这份输出需要先理解 Minor GC/Full GC 触发条件与对象晋升规则，工具是原理的观测窗口而非替代品。
  • 类加载机制：jstat -class 与 jmap 结合可以定位 Metaspace 增长是否由类加载器泄漏引起（如 CGLib 代理、JSP 热部署），呼应了 OOM 篇里 Metaspace OOM 的排查思路。
