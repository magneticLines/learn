JVM · 内存模型

03. 栈帧结构

局部变量表、操作数栈、动态链接、返回地址——方法调用的最小执行单元。

为什么要设计栈帧？

JVM 是一台基于栈的虚拟机。每次调用一个方法，JVM 就在当前线程的虚拟机栈上压入一块专属的内存区域——这就是栈帧（Stack Frame）。 方法执行完毕（正常返回或抛出异常）后，栈帧被弹出销毁。

这种设计解决了两个核心问题：

  • 隔离性：每次方法调用拥有独立的局部变量空间，递归调用不会相互污染数据。
  • 确定性：栈帧大小在编译期（.class 文件）就已确定，JVM 无需运行时动态分配，执行效率高且内存布局可预测。
类比：可以把虚拟机栈想象成一摞托盘。每次调用方法就新拿一个托盘（栈帧）放在最上面，托盘上摆着该方法专属的"食材"（变量）和"操作台"（操作数栈）。方法结束就撤走最上面的托盘，回到下面那个托盘继续工作。

栈帧的四大组成部分

Stack Frame（栈帧）
├── 局部变量表（Local Variable Table）
├── 操作数栈（Operand Stack）
├── 动态链接（Dynamic Linking）
└── 方法返回地址（Return Address）
组成部分 作用 大小确定时机
局部变量表 存放方法参数 + 方法内局部变量 编译期（max_locals）
操作数栈 字节码指令的运算"草稿纸" 编译期（max_stack）
动态链接 指向运行时常量池中该方法的符号引用，用于虚方法分派 运行期动态解析
返回地址 记录调用者 PC，方法返回后恢复执行 进入方法时记录

局部变量表（Local Variable Table）

局部变量表是一个以 Slot 为单位的数组。每个 Slot 占 32 位（4 字节）。 long 和 double 占 2 个连续 Slot，其余类型（包括引用类型）占 1 个 Slot。

Slot 的分配遵循固定规则：

  • 实例方法：Slot 0 固定是 this，从 Slot 1 开始按声明顺序排列参数，然后是局部变量。
  • 静态方法：没有 this，从 Slot 0 开始排参数。
  • 局部变量的 Slot 可以复用：当一个变量的作用域结束后，它的 Slot 可被后续变量占用，节省空间。
Java 源码示例方法
public class FrameDemo {

    // 实例方法：Slot0=this, Slot1=a, Slot2=b（long 占 Slot2+3）, Slot4=result
    public int add(int a, long b) {
        int result = a + (int) b;
        return result;
    }

    // Slot 复用：i 和 j 的作用域不重叠，JVM 可让它们共享同一个 Slot
    public void slotReuse() {
        {
            int i = 10;   // Slot1，出了花括号作用域即可释放
        }
        int j = 20;     // 复用 Slot1，max_locals 不会因此变成 2
    }
}
陷阱：局部变量没有默认值！
类的成员变量在对象创建时会被 JVM 零值初始化（int→0，引用→null），但局部变量没有这个待遇。 使用未赋值的局部变量会直接导致编译错误，而不是像 C/C++ 那样产生随机值，这是 Java 语言规范的显式保护。

操作数栈（Operand Stack）

操作数栈是字节码指令的运算工作区，是一个后进先出（LIFO）的栈结构。 字节码指令不像 x86 汇编那样直接操作寄存器，而是统一通过压栈（push）和弹栈（pop）完成计算。

为什么选择栈而不是寄存器？因为 JVM 是跨平台的。寄存器数量因 CPU 架构不同而差异巨大（x86 只有 8 个通用寄存器，ARM 有 16 个），基于栈的设计使得字节码与底层硬件完全解耦，一套字节码可以在任何平台上执行。

字节码执行追踪：a + b

Java 源码int add(int a, int b)
public int add(int a, int b) {
    return a + b;
}
字节码（javap -c 输出）逐步拆解
// Slot0=this, Slot1=a, Slot2=b
iload_1   // 把局部变量表 Slot1（a）压入操作数栈   → 栈: [a]
iload_2   // 把局部变量表 Slot2（b）压入操作数栈   → 栈: [a, b]
iadd      // 弹出栈顶两个 int，相加，结果压栈      → 栈: [a+b]
ireturn   // 弹出栈顶值作为返回值，栈帧销毁

操作数栈的深度（max_stack）在编译期就由 javac 精确计算好，写入 .class 文件的 Code 属性中。JVM 在分配栈帧时直接按此值开辟空间，没有任何运行时开销。

方法调用时栈帧的完整生命周期

Java 源码两层调用栈
public class CallDemo {
    public static void main(String[] args) {
        int x = square(5);   // 调用 square，压入新栈帧
        // square 返回后，x=25，继续执行
    }

    static int square(int n) {
        return n * n;         // 执行完毕，弹出栈帧，返回值压入调用方操作数栈
    }
}
调用 square(5) 时虚拟机栈快照（栈顶在上）：

┌────────────────────────────────┐ ← 当前帧（square）
│ 局部变量表: Slot0=n(5) │
│ 操作数栈: [5, 5] → iadd → [25]│
│ 返回地址: main PC#4 │
└────────────────────────────────┘
┌────────────────────────────────┐ ← 调用方帧（main）
│ 局部变量表: Slot0=args, Slot1=x │
│ 操作数栈: 等待 square 返回值 │
└────────────────────────────────┘

动态链接 & 方法返回地址

动态链接（Dynamic Linking）

每个栈帧都持有一个指向运行时常量池（Runtime Constant Pool）中该方法符号引用的指针。 这个指针主要服务于虚方法分派：当调用 invokeVirtual 指令时，JVM 通过动态链接找到对象实际类型的方法表，实现多态。

方法返回地址（Return Address）

方法执行完毕有两种出口：

  • 正常返回：执行 return 指令，将返回值（如果有）压入调用方的操作数栈，恢复调用方 PC 继续执行。
  • 异常退出：抛出未被捕获的异常，此时没有返回值，PC 由异常表（Exception Table）决定跳转位置。

StackOverflowError 的本质

虚拟机栈的深度是有限的（默认通常 512 KB～1 MB，可用 -Xss 调整）。 无限递归或超深调用链会不断压入新栈帧，最终超过栈的最大深度，JVM 抛出 StackOverflowError。

Java 源码复现 StackOverflowError
// 反例：无终止条件的递归
public void infinite() {
    infinite();  // 每次调用都压入新栈帧，最终 StackOverflowError
}

// 正例：尾递归改迭代（JVM 不优化尾调用，必须手动改）
public int factorial(int n) {
    int result = 1;
    for (int i = 1; i <= n; i++) result *= i;
    return result;  // 只有一个栈帧，O(1) 栈空间
}
注意：JVM 不做尾调用优化（TCO）
Scala 的 @tailrec 是编译器层面的语法糖，把尾递归转成字节码层的循环。纯 Java 语言层面的递归在字节码上仍会逐帧压栈，写深递归时务必手动改迭代。

面试高频考点

问题核心回答
局部变量表的 Slot 是什么？long/double 为何占 2 个？ Slot 是 32 位存储单元。long/double 是 64 位类型，需要 2 个连续 Slot 存储，且不保证原子性（Java 内存模型对 32 位操作才保证原子性）。
实例方法的 Slot 0 是什么？ 固定是 this 引用。静态方法没有隐式 this，所以 Slot 0 从参数开始。
操作数栈和 CPU 寄存器有什么关系？ JIT 编译器（C1/C2）会将操作数栈的操作映射为真实的寄存器指令，解释执行时才实际用内存模拟栈操作。
栈帧大小何时确定？ 编译期由 javac 确定并写入 .class 的 Code 属性（max_stack + max_locals），JVM 在方法入口直接按此值分配，零运行时开销。
StackOverflowError vs OutOfMemoryError（对栈而言） 线程请求的栈深度超过 JVM 允许的最大值 → StackOverflowError；JVM 无法为新线程申请足够的栈内存（整体内存不足）→ OutOfMemoryError。
如何调整栈大小？ -Xss256k（每线程虚拟机栈大小）。增大可容纳更深递归，但线程数多时总内存占用也线性增长。

小结

  • 栈帧是方法调用的最小执行单元，随方法入栈/出栈，天然隔离各次调用。
  • 局部变量表以 Slot 数组存储参数和局部变量，编译期固定大小，实例方法 Slot 0 恒为 this，局部变量没有默认值。
  • 操作数栈是字节码指令的运算"草稿纸"，基于栈的设计保证了字节码跨平台，JIT 编译后会映射到真实寄存器。
  • 动态链接支持多态分派，返回地址保证方法结束后能正确回到调用方。
  • 栈深度有限，无限递归导致 StackOverflowError，生产中应改用迭代或限制递归深度。
