JVM · 内存模型

运行时数据区

堆 / 栈 / 方法区 / PC 寄存器 — JVM 如何在内存中安排你的程序

为什么要有运行时数据区？

C/C++ 直接操作操作系统内存，程序员手动 malloc/free。这带来了极高的自由度，也带来了 野指针、内存泄漏、缓冲区溢出等一系列灾难。Java 的设计目标之一就是 「Write Once, Run Anywhere」——屏蔽底层差异，提供一个统一、安全的执行环境。

JVM 规范（JVMS）定义了运行时数据区（Runtime Data Areas），把内存按用途切分为若干逻辑区域， 每块区域有明确的生命周期和访问规则。这样做的好处：

  • GC 只需关心堆，不会误回收栈帧
  • 线程私有区与共享区隔离，天然避免部分并发问题
  • OutOfMemoryError / StackOverflowError 可精准定位到具体区域

总览：五大区域

JVM 运行时数据区 ├── 线程共享（随 JVM 启动而生，随 JVM 停止而灭） │ ├── 堆（Heap） ← 对象实例 / 数组 │ └── 方法区（Method Area） ← 类元数据 / 常量池 / 静态变量 │ └── 线程私有（随线程创建而生，随线程销毁而灭） ├── 虚拟机栈（JVM Stack） ← 方法调用栈帧 ├── 本地方法栈（Native Stack） ← native 方法 └── 程序计数器（PC Register）← 当前字节码行号
规范 vs 实现：上面是 JVMS 定义的逻辑模型。HotSpot 等具体实现可以把多个区域合并或拆分， 比如 HotSpot 将"本地方法栈"与"虚拟机栈"合并为同一块内存。

① 程序计数器（PC Register）

可以把 PC 寄存器理解为书签：它记录当前线程正在执行的字节码指令的地址（行号）。 CPU 在线程间切换时，靠 PC 记录的位置恢复执行。

  • 线程私有：每条线程独享一个，互不干扰
  • 执行 Java 方法时，保存字节码指令的地址
  • 执行 native 方法时，PC 值为 undefined（规范定义）
  • 唯一不会 OOM 的区域——它只存一个整数地址，大小固定
生活类比：多人阅读同一本书（共享字节码），每人用自己的书签（私有 PC）记录读到哪一页。 书签不占多少空间，永远不会"放不下"。

② 虚拟机栈（JVM Stack）

每次调用一个 Java 方法，JVM 就往当前线程的栈里压入一个栈帧（Stack Frame）； 方法返回时弹出。栈帧是方法执行的完整上下文，包含：

栈帧（Stack Frame） ├── 局部变量表（Local Variable Table） ← 方法参数 + 局部变量 ├── 操作数栈（Operand Stack） ← 字节码执行时的计算暂存区 ├── 动态链接（Dynamic Linking） ← 指向方法区中该方法的符号引用 └── 返回地址（Return Address） ← 方法正常/异常返回后跳到哪里

用代码感受栈帧的压入与弹出

StackDemo.javaJava
public class StackDemo {

    public static void main(String[] args) {
        // 调用 a() → 压入 a 的栈帧
        a();
    }

    static void a() {
        int x = 10;         // x 存放在 a 的局部变量表
        b(x);               // 调用 b() → 再压入 b 的栈帧
    }                      // a() 返回 → 弹出 a 的栈帧，x 随之消失

    static void b(int n) {
        int y = n * 2;      // y 存放在 b 的局部变量表
    }                      // b() 返回 → 弹出 b 的栈帧，y 随之消失
}

执行到 b() 内部时，栈的状态是：

栈顶 → [ b 的栈帧：n=10, y=20 ] [ a 的栈帧：x=10 ] 栈底 → [ main 的栈帧 ]

两种错误及触发场景

错误触发原因典型场景
StackOverflowError栈深度超过 JVM 允许的上限无终止条件的递归
OutOfMemoryError动态扩展栈时系统内存不足大量线程并发创建
陷阱：-Xss 参数设置每个线程的栈大小（默认 512KB～1MB）。 线程数 × Xss ≈ 栈总内存占用。创建大量线程时，调小 Xss 是扩大线程数的有效手段， 而不是加大堆。

③ 堆（Heap）

堆是 JVM 中最大的一块内存，也是 GC 的主战场。 所有线程共享，用于存放：

  • 通过 new 创建的对象实例
  • 数组（数组也是对象）

堆被进一步划分为新生代（Young Gen）和老年代（Old Gen）， 这是 GC 分代假设（大多数对象朝生夕死）驱动的设计——下一节「堆内存分代」会详细展开。

对象在堆上的生命周期

HeapDemo.javaJava
public class HeapDemo {
    public static void main(String[] args) {

        // new 触发在堆上分配内存，obj 是栈上的引用（仅 4/8 字节）
        Object obj = new Object();

        // obj 指向堆地址；obj 本身存在栈的局部变量表里
        System.out.println(obj);

        // 方法结束后 obj 出栈，堆上的 Object 实例失去引用
        // → GC 在下次回收时释放堆内存
    }
}
关键区分：引用（reference）在栈，对象实例在堆。 Object obj 这个变量只有 4 或 8 字节（压缩指针）， 真正的 Object 数据在堆上。这是理解 GC、内存泄漏的基础。

控制堆大小的 JVM 参数

参数含义示例
-Xms堆初始大小-Xms256m
-Xmx堆最大大小-Xmx1g
生产建议：将 -Xms 与 -Xmx 设为相同值，避免堆动态扩展时触发 Full GC 和内存抖动。

④ 方法区（Method Area）

方法区存放类级别的数据，是类加载器加载 .class 文件后写入的目的地：

  • 类的结构信息：字段、方法描述符、访问标志
  • 运行时常量池（Runtime Constant Pool）：字面量 + 符号引用解析后的直接引用
  • 静态变量（static 字段）
  • 即时编译后的代码缓存（JIT 代码）

实现演变：从 PermGen 到 Metaspace

JDK 版本方法区实现位置大小限制
≤ JDK 7永久代（PermGen）堆内（逻辑上）-XX:MaxPermSize，默认 64～256MB，容易 OOM
≥ JDK 8元空间（Metaspace）本地内存（堆外）默认无上限，受物理内存约束，可用 -XX:MaxMetaspaceSize 限制

为什么要移除 PermGen？ PermGen 大小静态配置，难以预估合适的值；类动态生成（反射、CGLib、JSP）会把 PermGen 撑爆， 出现臭名昭著的 java.lang.OutOfMemoryError: PermGen space。 Metaspace 使用本地内存，弹性更好，JVM 本身也能更好地与操作系统协作管理。

运行时常量池

StringPoolDemo.javaJava
public class StringPoolDemo {
    public static void main(String[] args) {

        // 字面量 "hello" 在编译期写入常量池；运行时驻留字符串池（String Pool）
        String s1 = "hello";
        String s2 = "hello";
        System.out.println(s1 == s2);  // true：同一个常量池对象

        // new String() 在堆上新建对象，不走常量池
        String s3 = new String("hello");
        System.out.println(s1 == s3);  // false：s3 指向堆中新对象

        // intern() 将堆对象引用"注册"进字符串池，返回池中的引用
        System.out.println(s1 == s3.intern());  // true
    }
}
JDK 7 之后，字符串常量池（String Pool）从方法区移入了堆， 受 GC 管理。这是理解 intern() 行为的关键背景。

⑤ 本地方法栈（Native Method Stack）

功能与虚拟机栈完全对称，区别是服务于 native 方法——即用 C/C++ 实现、通过 JNI 调用的方法。HotSpot 把它与虚拟机栈合并实现，实际上是同一块内存区域。 同样可能抛出 StackOverflowError 和 OutOfMemoryError。

五区协作：一次方法调用的内存全景

Memory全景示例Java
public class Order {
    // ① 类加载时，Order 的元数据写入方法区
    private static final String TYPE = "retail";  // 静态变量 → 方法区
    private int amount;                          // 实例字段 → 随对象存在堆上

    public Order(int amount) {
        this.amount = amount;
    }

    public int getAmount() {
        // ② 调用 getAmount()：JVM 压入栈帧
        //    PC 寄存器记录下一条字节码地址
        //    this 引用存于局部变量表 slot 0
        return this.amount;  // 从堆上的对象读取字段
    }                          // ③ 方法返回：弹出栈帧，PC 跳回调用处
}

class Main {
    public static void main(String[] args) {
        // order 引用 → 栈的局部变量表（4/8 字节）
        // new Order(100) → 堆上分配对象
        Order order = new Order(100);
        int amt = order.getAmount();  // amt → 栈的局部变量表
    }
}

常见陷阱 & 误区

陷阱 1：「局部变量是线程安全的」——不完全对

基本类型局部变量存在栈帧里，确实线程私有、天然安全。 但若局部变量是对象引用，且该对象被传递给其他线程，对象本身在堆上被共享， 依然存在并发风险。

陷阱 2：方法区不受 GC 管理

错误！方法区会被 GC，但条件苛刻：类需要满足「类的实例全部回收 + ClassLoader 已被回收 + Class 对象没有被引用」 三个条件才能卸载。动态代理/CGLib 大量生成类时若 ClassLoader 管理不当，会导致 Metaspace OOM。

陷阱 3：new 的对象一定在堆上

JDK 的 JIT 编译器支持逃逸分析（Escape Analysis）。 若对象不会"逃逸"出方法（不被外部引用），JIT 可能将其栈上分配， GC 完全不参与，性能更优。这是 -XX:+DoEscapeAnalysis（JDK 8 默认开启）的效果。

面试高频考点

  1. JVM 运行时数据区有哪些？哪些是线程私有的？
    五大区：堆、方法区（共享）；虚拟机栈、本地方法栈、PC 寄存器（私有）。
  2. StackOverflowError 和 OutOfMemoryError 分别出现在哪？
    SOE：栈深度超限（无限递归）；OOM：可发生在堆、方法区、栈（动态扩展失败）。
  3. JDK 8 为什么移除 PermGen 改为 Metaspace？
    PermGen 大小固定，动态类生成容易 OOM；Metaspace 使用本地内存，弹性伸缩， 避免 OutOfMemoryError: PermGen space。
  4. 字符串常量池在哪里？
    JDK 6 在方法区；JDK 7+ 移入堆，可被 GC 回收。
  5. 对象一定分配在堆上吗？
    不一定。逃逸分析后 JIT 可将不逃逸的对象栈上分配；标量替换可进一步将对象字段拆散到寄存器/栈。
