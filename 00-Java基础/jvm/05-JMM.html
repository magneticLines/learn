JVM · 内存模型

05. Java 内存模型 JMM
happens-before / 指令重排

JMM 是 Java 并发的基石——它定义了多线程程序中变量读写的可见性规则，以及编译器 / CPU 被允许做哪些重排优化。

1. 为什么需要 JMM？

现代计算机为了榨干性能，做了三件让并发变复杂的事：

  • 多级缓存（L1/L2/L3）：每个 CPU 核心有自己的私有缓存，写入主内存是延迟的。
  • 编译器重排序：JIT / javac 在不改变单线程语义的前提下，可以调整指令顺序提升流水线效率。
  • CPU 乱序执行（Out-of-Order Execution）：CPU 会自动打乱指令执行顺序，只保证最终结果在本核上"看起来"正确。

这三件事在单线程下完全透明，但一旦多线程共享数据，线程 A 写入的值，线程 B 未必能即时看到，甚至看到的顺序可能与 A 的书写顺序不同。

JMM 的本质：不是描述硬件如何运作，而是给 Java 程序员一份契约—— "只要你遵守这些规则，JVM 保证无论底层怎么优化，你的程序跨平台都有确定的语义。"

直观类比

把主内存想象成公司的共享文档，每个线程是一名员工，手边有一份本地草稿（工作内存 / CPU 缓存）。 员工可以先在草稿上改，何时同步到共享文档、何时从共享文档刷新，JMM 给出了明确的时机规定。

2. 主内存与工作内存

JMM 将内存划分为两个抽象层次：

主内存（Main Memory） ├── 所有线程共享的变量（堆上的实例/静态字段） └── 线程间通信的唯一合法渠道 工作内存（Working Memory）（每线程私有） ├── 主内存变量的副本（即 CPU 寄存器 + 缓存抽象） ├── 本地变量、操作数栈 └── 不能被其他线程直接访问

线程对变量的所有操作（读取、赋值）都必须在工作内存中完成，不能直接读写主内存。 线程间的值传递必须经过主内存这个"中介"。

陷阱：工作内存 ≠ JVM 运行时数据区的某一块。它是一个抽象概念，对应的物理实现可能是 L1 缓存、寄存器或写缓冲区。

3. 指令重排序（Reordering）

重排序分三类，层层叠加：

类型发生位置典型手段
编译器重排序javac / JIT调整字节码指令顺序
指令级并行重排序CPU 流水线乱序执行（OoO）
内存系统重排序写缓冲区 / 缓存写操作延迟刷入主内存

经典问题：双重检查锁（DCL）

在没有 volatile 的情况下，下面的单例代码在多核环境下是危险的：

Java — 有问题的 DCL（反例）
// ❌ 错误：instance = new Singleton() 并非原子操作
class Singleton {
    private static Singleton instance;   // 没有 volatile！

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();  // ← 危险点
                }
            }
        }
        return instance;
    }
}

new Singleton() 在字节码层面分三步：

  1. 分配内存空间（new）
  2. 调用构造方法初始化字段（invokespecial）
  3. 将引用赋值给变量（putstatic）

步骤 2 和 3 可以被重排序为 先赋引用，再初始化。若线程 A 执行了第 3 步但还未执行第 2 步，线程 B 第一个 null 检查会看到非 null 的 instance，但该对象尚未初始化完成——拿到一个半成品对象。

Java — 正确的 DCL（加 volatile）
// ✅ 正确：volatile 禁止步骤 2/3 重排序
class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

4. happens-before 规则

happens-before 是 JMM 的核心概念，由 JSR-133 规范定义：

定义：如果操作 A happens-before 操作 B，则 A 的结果对 B 可见，且 A 的执行顺序排在 B 之前。 happens-before 不等于"时间上先发生"，它是一个关于可见性的承诺。

JMM 给出了 8 条天然的 happens-before 规则（程序员不需要额外同步即可依赖）：

规则名称含义
程序顺序规则同一线程内，前面的操作 hb 后面的操作（单线程表现有序）
监视器锁规则unlock hb 后续对同一锁的 lock
volatile 变量规则对 volatile 变量的写 hb 后续对该变量的读
线程启动规则Thread.start() hb 新线程内的任意操作
线程终止规则线程内所有操作 hb Thread.join() 返回
线程中断规则interrupt() hb 被中断线程检测到中断
对象终结规则构造方法结束 hb finalize() 开始
传递性规则A hb B，B hb C，则 A hb C

用 happens-before 分析 volatile

Java — volatile 可见性验证
class VisibilityDemo {
    private volatile boolean flag = false;
    private int data = 0;

    // 线程 A 执行
    void writer() {
        data = 42;          // ① 普通写
        flag = true;         // ② volatile 写
    }

    // 线程 B 执行
    void reader() {
        if (flag) {           // ③ volatile 读
            // ④ 此处一定能看到 data == 42
            System.out.println(data);
        }
    }
}

分析链路：① hb ②（程序顺序规则）→ ② hb ③（volatile 变量规则）→ ③ hb ④（程序顺序规则）→ 由传递性得 ① hb ④。所以 B 读到 flag==true 时，data 必然已可见为 42。

5. volatile 的两层语义

语义一：可见性

写 volatile 变量时，JMM 要求将工作内存中的值立即刷新到主内存；读 volatile 变量时，要求先从主内存加载最新值，丢弃本地缓存副本。

语义二：禁止特定重排序

JMM 通过 内存屏障（Memory Barrier） 来实现重排序禁止：

屏障类型作用插入时机
StoreStore屏障前的写不能重排到屏障后volatile 写之前
StoreLoad屏障前的写必须对屏障后的读可见volatile 写之后（最重量级）
LoadLoad屏障前的读不能重排到屏障后volatile 读之后
LoadStore屏障前的读不能重排到屏障后的写volatile 读之后
volatile 不能保证原子性！
volatile int i; i++; 依然线程不安全——i++ 是「读→加→写」三步，volatile 只保证每步的可见性，但三步之间可以插入其他线程的操作。 需要原子性时用 AtomicInteger 或 synchronized。

6. synchronized 与 JMM

synchronized 同时保证三大特性：

  • 原子性：同一时刻只有一个线程能进入同步块。
  • 可见性：unlock 前会把工作内存的变量刷回主内存；lock 后会清空工作内存副本，从主内存重新读。
  • 有序性：synchronized 建立了 happens-before 关系（监视器锁规则）。
Java — synchronized 保证可见性
class Counter {
    private int count = 0;

    // lock 时从主内存读最新值，unlock 时写回主内存
    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}

7. final 字段的特殊保证

JMM 为 final 字段提供了"构造安全发布"保证：

  • 构造方法中对 final 字段的写，在构造方法完成之前不会被重排序到构造方法之外。
  • 其他线程通过正常途径获得该对象引用后，一定能看到 final 字段的初始化值，无需任何额外同步。
Java — final 的安全发布
class ImmutablePoint {
    final int x;
    final int y;

    ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
        // 构造结束前，JMM 保证 x/y 写入对所有线程可见
    }
}

// 只要不在构造方法中逸出 this，其他线程读到的 x/y 一定是完整初始化的
ImmutablePoint p = new ImmutablePoint(3, 4);
陷阱：this 逸出
如果在构造方法中将 this 传递给其他线程（如注册监听器），final 的安全保证失效——其他线程可能在构造完成之前就访问该对象。

8. as-if-serial 语义

无论编译器、CPU 如何重排，单线程程序的执行结果不变——这是 JMM 对单线程的最低保证，称为 as-if-serial。

有数据依赖的操作不会被重排：

Java — 数据依赖禁止重排
int a = 1;     // ①
int b = 2;     // ②  ①② 无依赖，可以重排
int c = a + b; // ③  ③ 依赖 ①②，必须在 ①② 之后

注意：as-if-serial 只在单线程内有效。多线程场景下，两个线程各自遵守 as-if-serial，但相互之间的操作顺序没有保证。

9. 常见陷阱 & 面试考点

陷阱一：用布尔标志停止线程——必须加 volatile

Java — 线程停止标志
class Worker implements Runnable {
    // ❌ 没有 volatile：JIT 可能将 while(!stopped) 优化成死循环
    private boolean stopped = false;

    // ✅ 正确：加 volatile 保证主线程的写对 Worker 线程立即可见
    private volatile boolean stopped = false;

    @Override
    public void run() {
        while (!stopped) {
            doWork();
        }
    }

    public void stop() { stopped = true; }
}

陷阱二：long / double 的非原子性

在 32 位 JVM 上，对 long 和 double 的读写不是原子的（被拆成两次 32 位操作）。在 64 位 JVM 上通常已优化为原子操作，但 JMM 规范并不强制保证。 多线程共享 long/double 时，加 volatile 或 synchronized 是安全写法。

面试高频：volatile 能否替代 synchronized？

特性volatilesynchronized
可见性✅✅
原子性❌（复合操作不安全）✅
有序性✅（禁止特定重排）✅
阻塞❌（非阻塞）✅（互斥阻塞）
性能较轻量相对重（但 JDK6+ 有偏向锁优化）

面试高频：happens-before 与时间先后的区别

时间先发生 ≠ happens-before：A 在时间上先于 B 执行，不代表 A 的写对 B 可见。 happens-before 是一个逻辑上的偏序关系，由 JMM 的 8 条规则建立，与物理时钟无关。 只有满足 happens-before，JMM 才保证 B 能看到 A 的结果。

面试高频：JMM 三大特性

  • 原子性：操作不可分割。基本类型读写（除 long/double）是原子的；复合操作需要同步。
  • 可见性：一个线程修改的值，其他线程能立即看到。volatile、synchronized、final 均可保证。
  • 有序性：程序执行的顺序符合代码书写顺序。JMM 允许编译器和CPU重排，但对外暴露的结果有序性由 happens-before 保证。

小结

JMM 知识体系 ├── 为什么：多核 CPU 缓存 + 编译器/CPU 重排 → 并发可见性问题 ├── 核心抽象：主内存 ↔ 工作内存，线程通过主内存通信 ├── happens-before：8 条规则建立操作间的可见性偏序关系 ├── volatile：可见性 + 禁止特定重排（内存屏障）；不保证原子性 ├── synchronized：原子性 + 可见性 + 有序性三合一 └── final：构造安全发布，无需额外同步
