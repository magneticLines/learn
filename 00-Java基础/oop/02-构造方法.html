OOP · 第 02 节

构造方法

对象从"出生"那一刻就应处于合法状态——构造方法正是为此而设计

1. 为什么需要构造方法

想象一条汽车生产线：零件焊接完毕却没有装发动机，车就被推出工厂。这台"半成品"进入道路后任何操作都可能崩溃。软件里的对象也一样——如果允许字段在使用前保持未初始化状态，就会留下随时爆炸的"定时炸弹"。

反例：半初始化对象的危险 
class BankAccount {
    String owner;   // 忘记赋值 → null
    double balance; // 默认 0，但未经任何验证
}

// 调用方可以这样构造：
BankAccount acc = new BankAccount();
acc.balance = -99999;   // 非法余额被直接写入！
System.out.println(acc.owner.toUpperCase()); // NullPointerException

构造方法解决两个核心问题：

  1. 强制初始化：通过参数列表声明"创建对象必须提供什么"，让编译器帮你检查。
  2. 集中校验：把不变量（invariant）的检查写在一处，而不是散落在每个 setter 里。
本质：构造方法是"对象的入场许可证" 通过编译期约束 + 运行时校验，保证任何到达业务逻辑的对象都处于有效状态。 这是"防御性编程"（Defensive Programming）的基础实践。

2. 默认构造方法

当你没有显式声明任何构造方法时，编译器会自动生成一个无参、无方法体的构造方法：

// 你写的
class Point {
    int x, y;
}

// 编译器等价于补了这个（你看不见，但它存在）
class Point {
    int x, y;
    public Point() {
        super(); // 隐式调用 Object 的无参构造
    }
}
⚠️ 高频陷阱：手写有参构造后，默认构造方法自动消失 
class Point {
    int x, y;
    public Point(int x, int y) { /* ... */ }
    // 此时默认无参构造已消失！
}

Point p = new Point(); // 编译错误：找不到 Point()
这是初学者最常踩的坑，也是 Spring / Jackson 反序列化必须保留 无参构造 的原因（见面试题第 5 条）。

字段的默认值规则（由 JVM 在分配堆内存时完成，早于构造方法执行）：

类型默认值
int / long / short / byte0
double / float0.0
booleanfalse

3. 有参构造方法

有参构造方法让"非法状态的对象根本无法被创建"，是实现不可变对象（Immutable Object）的关键手段。

public final class Money {
    private final double amount;   // final：赋值后不可更改
    private final String currency;

    public Money(double amount, String currency) {
        // 在"入场"时就校验——非法对象根本无法存活
        if (amount < 0) {
            throw new IllegalArgumentException("金额不能为负: " + amount);
        }
        if (currency == null || currency.isBlank()) {
            throw new IllegalArgumentException("币种不能为空");
        }
        // this.xxx = xxx 消除参数与字段同名的歧义
        this.amount   = amount;
        this.currency = currency;
    }

    public double  getAmount()   { return amount; }
    public String  getCurrency() { return currency; }
}

// 使用：无论何处拿到 Money 对象，一定合法
Money m = new Money(9.9, "CNY");
Money bad = new Money(-1, "USD"); // 直接抛出，非法对象不存在
为什么 final 字段要在构造方法里赋值？ final 字段只能赋值一次，且必须在构造方法（或字段声明处）完成。 这由编译器在编译期强制检查，与运行时无关——比任何文档注释都可靠。

4. 构造方法重载与 this() 委托

同一个类可以有多个构造方法，只要参数列表不同（类型、数量、顺序）。用 this(...) 把多个重载委托给"最完整"的那个，避免重复校验逻辑。

public class Connection {
    private final String host;
    private final int    port;
    private final int    timeout; // 毫秒

    // ① 最完整的构造方法，所有校验集中在这里
    public Connection(String host, int port, int timeout) {
        if (host == null || host.isBlank())
            throw new IllegalArgumentException("host 不能为空");
        if (port <= 0 || port > 65535)
            throw new IllegalArgumentException("端口非法: " + port);
        if (timeout <= 0)
            throw new IllegalArgumentException("超时须 > 0");
        this.host    = host;
        this.port    = port;
        this.timeout = timeout;
    }

    // ② 省略 timeout，使用默认值；委托给 ①
    public Connection(String host, int port) {
        this(host, port, 3000); // this() 必须是构造方法第一行
    }

    // ③ 只给 host，其余全用默认值；委托给 ②
    public Connection(String host) {
        this(host, 3306); // 链式：③ → ② → ①，校验只写一次
    }

    public String info() {
        return host + ":" + port + " (timeout=" + timeout + "ms)";
    }
}

// 三种用法都能用
new Connection("db.local").info();             // db.local:3306 (timeout=3000ms)
new Connection("db.local", 5432).info();        // db.local:5432 (timeout=3000ms)
new Connection("db.local", 5432, 500).info();   // db.local:5432 (timeout=500ms)

重载合法性对照表

情形合法？说明
参数数量不同✅最常见
参数类型不同✅(int) vs (double)
参数顺序不同✅ 但不推荐容易造成歧义，可读性差
只有参数名不同❌参数名不参与方法签名
只有返回类型不同❌（构造无返回类型）构造方法本身无返回值概念
⚠️ this() 的两个硬性约束
  • 必须是构造方法体内的第一条语句，否则编译报错。
  • 不能形成循环委托（A → B → A），编译器会检测并报错。

5. 构造方法与继承

super() 调用规则

子类构造方法必须调用父类构造方法（显式或隐式）。若父类没有无参构造，子类必须显式调用 super(...)。

class Animal {
    final String name;
    Animal(String name) {   // 父类只有有参构造
        this.name = name;
    }
}

class Dog extends Animal {
    final String breed;
    Dog(String name, String breed) {
        super(name);       // ① 必须是第一行；完成父类初始化
        this.breed = breed; // ② 再初始化子类自己的字段
    }
}

// 若不写 super(name)，编译器尝试隐式调用 Animal()，但它不存在 → 编译错误

⚠️ 经典陷阱：构造方法里调用可覆盖方法

这是 Java 中最隐蔽的 bug 来源之一，堪称"面试必考"。

class Base {
    Base() {
        init(); // ← 危险！调用了可被子类覆盖的方法
    }
    void init() {
        System.out.println("Base.init");
    }
}

class Child extends Base {
    private final String msg;

    Child() {
        // super() 隐式调用，在 msg 赋值之前就调用了 init()
        msg = "hello";
    }

    @Override
    void init() {
        // 此时 msg 还没被赋值，依然是 null！
        System.out.println(msg.toUpperCase()); // NullPointerException！
    }
}

new Child(); // 抛 NPE，而不是打印 HELLO
执行顺序（对象创建全流程）：
  1. JVM 为对象分配堆内存，所有字段置零/null（默认值阶段）
  2. 执行父类构造方法（从顶层祖先开始，层层向下）
  3. 执行子类字段的显式初始化表达式和实例初始化块
  4. 执行子类构造方法体
所以步骤 2 里调用子类覆盖的方法时，子类字段还在"步骤 1"的零值状态。
规则：构造方法里只调用 private 或 final 方法 private 和 final 方法不能被覆盖，因此不存在"子类字段未初始化"的问题。

6. 三种构造方法横向对比

类型 何时使用 核心优势 注意事项
默认无参构造 字段有合理默认值；需要反射创建（框架） 最简便；框架友好 字段未初始化风险；一旦写有参构造需显式补回
有参构造 字段必须在创建时指定；不可变对象 强制合法化；配合 final 实现线程安全 参数多时可读性差，考虑 Builder
重载构造（+ this()） 多种合理默认值组合 兼顾灵活与校验集中 this() 必须第一行；层次不宜超过 3 层

7. 进阶：构造方法 vs 静态工厂方法

当构造方法的局限性开始阻碍你时，考虑静态工厂方法（Static Factory Method）——这是《Effective Java》第 3 版条款 1 的核心建议。

构造方法 new Foo(...)静态工厂方法 Foo.of(...)
名称必须与类名相同可自由命名，表达语义（valueOf、getInstance、create）
缓存每次 new 都创建新对象可返回缓存对象（Integer.valueOf(127)）
返回类型只能返回本类可返回子类型（多态工厂）
可读性参数意义靠参数名/文档方法名即文档
可被继承不能可以被子类覆盖
// 构造方法方式：不知道 true 代表什么
Boolean b1 = new Boolean(true);

// 静态工厂：语义清晰；且会复用缓存的 Boolean.TRUE 实例
Boolean b2 = Boolean.valueOf(true);
静态工厂方法是 Builder 模式、单例模式的基础。建议在字段超过 4 个或需要缓存时优先考虑。

8. 面试高频考点

Q1. 构造方法能被继承或重写吗？
不能。构造方法不是成员方法，不参与继承体系。子类无法继承父类构造方法，也无法用 @Override 覆盖它。子类只能通过 super() 显式调用父类构造方法。
Q2. 构造方法可以是 private 吗？有什么用？
可以。private 构造方法外部无法直接 new，常用于：① 单例模式（限制实例数量）；② 工具类（如 Math，防止实例化）；③ 配合静态工厂方法，控制对象创建逻辑。
Q3. this() 和 super() 能同时出现在一个构造方法里吗？
不能。两者都必须是构造方法的第一条语句，因此只能选其一。this() 最终会调用到某个不再委托的构造方法，那个构造方法里会有（显式或隐式的）super()。
Q4. 对象创建的执行顺序是什么？
① 父类静态字段 / 静态块（类首次加载时，只执行一次）
② 子类静态字段 / 静态块
③ 父类实例字段显式初始化 + 实例初始化块（按源码顺序）
④ 父类构造方法体
⑤ 子类实例字段显式初始化 + 实例初始化块
⑥ 子类构造方法体
Q5. 为什么 Spring / Jackson 要求 Bean 必须有无参构造方法？
Spring IoC 和 Jackson 默认通过反射创建对象：clazz.getDeclaredConstructor().newInstance()。这个 API 只能调用无参构造方法。如果没有无参构造，运行时会抛 NoSuchMethodException。解决方案：显式保留无参构造，或使用 Jackson 的 @JsonCreator 指定有参构造 / 静态工厂方法。

9. 知识联系树

  • 构造方法
    • → this 关键字：this.field 消歧义；this() 委托构造
    • → 继承 / super()：子类构造链路；执行顺序；可覆盖方法陷阱
    • → 不可变对象（Immutable）：final 字段 + 有参构造 = 线程安全对象
    • → 单例模式：private 构造 + 静态字段/方法
    • → Builder 模式：参数过多时替代大量重载构造的方案（见《Effective Java》条款 2）
    • → Spring IoC / 反射：框架依赖无参构造；@Autowired 构造注入是推荐做法
    • → Java 内存模型：final 字段的初始化安全性保证（JMM §17.5）
