20. equals() 与 hashCode() 契约

Java 中最容易踩坑的一对方法。一旦违反它们之间的契约，HashMap、HashSet 会静默出错，且极难排查。

1. 先搞清楚：== 解决不了什么问题

Java 中 == 比较的是引用（内存地址），而不是对象的业务含义。两个内容完全相同的 String、User 对象，用 == 比较永远是 false。

反例：用 == 比较对象Java
String a = new String("hello");
String b = new String("hello");

System.out.println(a == b);       // false：两个不同堆对象
System.out.println(a.equals(b));  // true：String 已覆写 equals，比较内容

Object.equals() 的默认实现就是 ==，所以自定义类必须覆写 equals 才能表达"业务上相等"的语义。

2. equals() 的五条契约

《Effective Java》条目 11 明确规定，覆写 equals 必须满足：

性质含义常见违反场景
自反性 x.equals(x) 必须为 true 几乎不会违反，除非故意写 bug
对称性 x.equals(y) 与 y.equals(x) 结果相同 子类与父类混用 instanceof，导致单向相等
传递性 x=y, y=z 则 x=z 继承中加入新字段，导致三角不等式破坏
一致性 多次调用结果相同（对象未变化时） equals 里依赖可变外部状态（如时间戳）
非空性 x.equals(null) 必须为 false 没有 null 检查，直接调用参数方法导致 NPE

3. 如何正确覆写 equals()

以一个 Point 类为例，演示标准写法：

正例：标准 equals 写法Java
public class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public boolean equals(Object o) {
        // 步骤 1：自引用优化，同一对象直接返回 true
        if (this == o) return true;

        // 步骤 2：null 检查 + 类型检查（同时满足非空性和对称性）
        // 用 getClass() 而非 instanceof，防止子类破坏对称性
        if (o == null || getClass() != o.getClass()) return false;

        // 步骤 3：强转后逐字段比较
        Point that = (Point) o;
        return this.x == that.x
            && this.y == that.y;
    }
}
getClass() vs instanceof
用 instanceof 时，子类实例可以等于父类实例（单向），但父类实例不能等于子类实例，导致对称性破坏。用 getClass() 则严格要求运行时类型完全一致。除非你的类本身就是为继承设计的值类型（如 AbstractSet），否则优先用 getClass()。

4. hashCode() 契约：与 equals 的绑定关系

Java 规范要求 equals 和 hashCode 满足：

规则 1：同一对象多次调用 hashCode()，返回值必须一致（运行期内）
规则 2：equals() 为 true 的两个对象，hashCode() 必须相等
规则 3：equals() 为 false 的两个对象，hashCode() 可以相等（允许哈希碰撞）

逆否命题（常考）：hashCode() 不等 → equals() 一定为 false

规则 2 是契约的核心。只覆写 equals 不覆写 hashCode，是 Java 开发中最常见的 bug 之一。

为什么违反规则 2 会导致 HashMap 出错？

HashMap 的查找逻辑分两步：先用 hashCode() 定位桶，再用 equals() 在桶内找元素。如果 equals 相等但 hashCode 不等，两个"相同"的键会落到不同的桶，永远找不到对方：

反例：只覆写 equals，不覆写 hashCodeJava
public class BadKey {
    private final String id;

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof BadKey)) return false;
        return id.equals((BadKey)o.id);
    }
    // 没有覆写 hashCode()！继承自 Object，基于内存地址
}

Map<BadKey, String> map = new HashMap<>();
BadKey k1 = new BadKey("001");
BadKey k2 = new BadKey("001");

map.put(k1, "Alice");
System.out.println(map.get(k2));  // 输出 null！k1.equals(k2) 为 true，但 hashCode 不同
// 原因：k2 的 hashCode 指向另一个桶，那个桶里什么都没有

5. 如何正确覆写 hashCode()

方法一：Objects.hash()（推荐，简洁）

Point 类完整实现（加上 hashCode）Java
import java.util.Objects;

public class Point {
    private final int x;
    private final int y;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Point that = (Point) o;
        return x == that.x && y == that.y;
    }

    @Override
    public int hashCode() {
        // 将参与 equals 的所有字段都传入 Objects.hash()
        // 内部用 31 为乘子的多项式滚动哈希，平衡性能与分布
        return Objects.hash(x, y);
    }
}

方法二：手写（理解原理）

手写 hashCode — 31 乘子法Java
@Override
public int hashCode() {
    int result = 17;               // 任意非零奇数作为初始值
    result = 31 * result + x;      // 31 是奇质数，JIT 可优化为位运算：(result << 5) - result
    result = 31 * result + y;
    return result;
}
为什么选 31？
31 是奇质数，乘以奇数不会导致信息因溢出而丢失（偶数乘可能使低位全为 0）。 同时 31 * n = (n << 5) - n，现代 JIT 会自动优化为位移运算，性能极好。 这也是 JDK 中 String.hashCode() 采用 31 的原因。

6. 不同字段类型的处理技巧

字段类型equals 比较方式hashCode 贡献方式
基本类型（int/long/...） == 直接使用（long 需 (int)(f ^ f>>>32)）
float Float.compare(a, b) == 0 Float.floatToIntBits(f)
double Double.compare(a, b) == 0 (int)(Double.doubleToLongBits(f) ^ ...>>>32)
引用类型（非数组） Objects.equals(a, b)（处理 null） Objects.hashCode(f)（null 返回 0）
数组 Arrays.equals(a, b) Arrays.hashCode(a)
float/double 不能用 == 比较！
浮点数有精度问题，0.1 + 0.2 == 0.3 为 false。 应用 Float.compare() 或 Double.compare()，它们内部转换为 IEEE 754 位模式再比较，语义更正确。 同理 NaN != NaN，也需要这两个方法处理。

7. 继承中的对称性陷阱

这是契约中最难处理的场景。假设 ColorPoint extends Point：

反例：子类用 instanceof 破坏对称性Java
public class ColorPoint extends Point {
    private final Color color;

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint)) return false;  // 问题在这里
        return super.equals(o) && color.equals((ColorPoint)o.color);
    }
}

Point p  = new Point(1, 2);
ColorPoint cp = new ColorPoint(1, 2, Color.RED);

// Point.equals 用 getClass()，所以 p.equals(cp) = false（类不同）
// ColorPoint.equals 用 instanceof，cp.equals(p) 也是 false（p 不是 ColorPoint）
// 结果：虽然看起来"对称"，但语义不自洽

《Effective Java》的推荐方案：继承值类型时，优先用组合代替继承，再提供视图方法：

正例：组合代替继承，绕开对称性难题Java
public class ColorPoint {
    private final Point point;   // 组合，不继承
    private final Color color;

    public Point asPoint() { return point; }  // 提供视图方法

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof ColorPoint)) return false;
        ColorPoint that = (ColorPoint) o;
        return point.equals(that.point) && color.equals(that.color);
    }

    @Override
    public int hashCode() {
        return Objects.hash(point, color);
    }
}

8. 与集合框架的联系

理解这个契约，能解释集合框架的很多行为：

集合类依赖违反契约的后果
HashMap / HashSet hashCode + equals put 进去用相等 key 找不到；HashSet 存入重复元素
TreeMap / TreeSet Comparable / Comparator 不依赖 equals，但 compareTo 与 equals 不一致时行为怪异
ArrayList equals（用于 contains/remove） contains 找不到"相等"的元素
HashSet 去重依赖 hashCode + equals 双重保障
向 HashSet 添加元素时，先算 hashCode 定位桶，桶中已有元素再用 equals 判断是否重复。 两者缺一不可：只覆写 equals 不覆写 hashCode，相同元素会进入不同桶，Set 无法去重。

9. 不要用可变字段作为 key

如果一个对象被放入 HashMap/HashSet 后，其参与 hashCode 计算的字段发生了变化，这个对象就会"消失"在集合里——因为它现在的哈希值与存入时不同，查找时去错了桶：

反例：可变对象作 HashMap key 导致内存泄漏Java
List<Integer> key = new ArrayList<>(List.of(1, 2));
Map<List<Integer>, String> map = new HashMap<>();
map.put(key, "value");

key.add(3);  // 修改 key！hashCode 变了

System.out.println(map.get(key));    // null：去了新的桶，什么都没有
System.out.println(map.size());      // 1：entry 还在，但永远找不到了

这是一种隐性的内存泄漏：对象存入集合后无法删除，GC 也无法回收。HashMap 的 key 应当是不可变对象（如 String、基本类型包装类、自定义 final 类）。

10. 三种实现方式对比

方式优点缺点
IDE 自动生成 快速，覆盖所有字段，不易遗漏 字段变更后忘记重新生成，导致逻辑过时
Objects.hash() / Objects.equals() 简洁，空安全，推荐日常使用 有轻微装箱开销（基本类型会被 varargs 装箱）
Lombok @EqualsAndHashCode 注解驱动，零样板 继承场景需配置 callSuper=true，否则忘记父类字段
Java 14+ record 编译器自动生成，语义最清晰 不可变类，不适合需要 setter 的场景
record 是最佳实践
JDK 16 正式发布的 record 会自动生成基于所有组件字段的 equals()、hashCode()、toString()，且字段天然不可变，彻底规避可变对象作 key 的风险。新项目中值对象优先考虑 record。

11. 面试高频考点

  • 为什么覆写 equals 必须覆写 hashCode？ — 因为 HashMap/HashSet 先用 hashCode 定位桶，再用 equals 比较；只覆写 equals 会让逻辑相等的对象散落在不同桶中。
  • 两个对象 hashCode 相同，equals 一定为 true 吗？ — 不一定。哈希碰撞允许 hashCode 相同但 equals 为 false（规则 3）。反过来，equals 为 true 则 hashCode 必须相同（规则 2）。
  • String 的 hashCode 是怎么计算的？ — s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]，31 为乘子，首次调用后缓存在 hash 字段（懒计算）。
  • 为什么 HashMap 的 key 要用不可变对象？ — 可变对象作 key，修改字段后 hashCode 变化，对象在 HashMap 中"消失"，无法 get/remove，形成内存泄漏。
  • null 作为 HashMap key 时发生什么？ — HashMap 允许一个 null key，固定放在 0 号桶（hash(null) = 0）。
核心口诀：equals 相等 → hashCode 必须相等；hashCode 相等 → equals 不一定相等。
两个方法必须同时覆写，参与 equals 的字段必须全部参与 hashCode 的计算。
