16. 内部类（静态 / 成员 / 匿名 / 局部）

Java 允许在一个类的内部再定义类。四种内部类各有适用场景，理解它们的访问规则、生命周期与字节码本质，是读懂 JDK 源码（迭代器、回调、Builder）的前提。

为什么要有内部类？

Java 是单继承语言，一个类只能继承一个父类。当你需要让同一个类"扮演多种角色"，或者需要把一段逻辑封装得更紧（只有外部类才能看到），内部类就是答案。

内部类解决的核心问题：

  • 逻辑分组：把只对外部类有用的类隐藏在内部，减少包级别的污染
  • 紧密封装：内部类可以访问外部类的私有成员，外部无法直接 new
  • 回调 / 事件处理：匿名内部类在 Lambda 出现前是唯一简洁写法
  • 模拟多继承：通过多个成员内部类各自继承不同父类
内部类（Inner Class） ├── 静态嵌套类 static nested class — 不持有外部引用，最常用 ├── 成员内部类 member inner class — 持有外部实例引用 ├── 局部内部类 local inner class — 定义在方法体内 └── 匿名内部类 anonymous inner class — 定义同时实例化，无类名

① 静态嵌套类（Static Nested Class）

用 static 修饰的内部类。它和外部类的关系更像是"命名空间上的归属"，而不是"实例上的从属"。不持有外部类实例的引用，因此不会造成外部类内存泄漏。

典型场景

  • Builder 模式（Person.Builder）
  • JDK 中 Map.Entry<K,V> 就是 Map 的静态嵌套接口
  • 测试辅助类、数据容器（不需要访问外部实例状态）
Person.java — Builder 模式
public class Person {
    private final String name;
    private final int    age;

    // 私有构造，只能通过 Builder 创建
    private Person(Builder b) {
        this.name = b.name;
        this.age  = b.age;
    }

    // ★ 静态嵌套类：不需要 Person 实例就能 new Builder()
    public static class Builder {
        private String name;
        private int    age;

        public Builder name(String name) { this.name = name; return this; }
        public Builder age(int age)    { this.age  = age;  return this; }
        public Person  build()          { return new Person(this); }
    }

    @Override
    public String toString() {
        return name + " (" + age + ")";
    }
}

// 调用方
Person p = new Person.Builder()
        .name("Alice")
        .age(28)
        .build();
关键规则：静态嵌套类可以访问外部类的 所有静态成员（包括私有），但不能直接访问外部类的实例成员（因为根本没有外部实例引用）。

② 成员内部类（Member Inner Class）

没有 static 修饰、直接定义在类体中的内部类。每个内部类实例都隐式持有一个外部类实例的引用（编译器自动插入 Outer.this）。

访问外部类私有成员

Outer.java — 成员内部类访问外部私有字段
public class Outer {
    private String secret = "外部私有字段";

    class Inner {
        void reveal() {
            // 直接访问 Outer 的 private 成员，编译器翻译为 Outer.this.secret
            System.out.println(secret);
        }
    }

    void demo() {
        // 必须先有 Outer 实例，才能 new Inner()
        Inner i = new Inner();
        i.reveal();
    }
}

// 外部 new Inner 必须先有 Outer 实例：
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();  // 注意：outer.new 语法
inner.reveal();

字节码真相：隐藏的构造参数

编译后，Inner 会被生成为 Outer$Inner.class，其构造器第一个参数是 Outer 实例：

反编译伪代码（理解原理用）
// 编译器生成的 Outer$Inner 构造器
Outer$Inner(Outer this$0) {
    this.this$0 = this$0;   // 存下外部引用！
}
内存泄漏陷阱：成员内部类持有外部类引用。如果将内部类实例传给长生命周期的对象（如静态集合、Handler），外部类实例将无法被 GC，造成内存泄漏。Android 中最常见的内存泄漏原因之一就是非静态内部类 Handler 持有 Activity 引用。修复方式：改用静态嵌套类 + WeakReference。

③ 局部内部类（Local Inner Class）

定义在方法体或代码块内部的类，作用域仅限该方法。它能访问外部类的实例成员，以及所在方法中的 final / effectively final 局部变量。

局部内部类示例
public class Validator {

    public boolean validate(String input, String prefix) {
        // prefix 是 effectively final（方法内未被重新赋值）
        class PrefixChecker {
            boolean check(String s) {
                return s.startsWith(prefix);  // 捕获了 prefix
            }
        }
        return new PrefixChecker().check(input);
    }
}
为什么只能捕获 effectively final 变量？
局部变量存在栈帧上，方法结束后栈帧销毁。局部内部类实例可能活得比方法更长（比如被返回出去），所以编译器会把被捕获的变量复制一份存到内部类实例的字段里。若变量可变，副本和原始值就会不一致，Java 选择直接禁止以避免语义混乱。Lambda 捕获变量同理。

局部内部类在实践中很少使用，主要出现在需要在方法内部临时实现接口、又需要复用（多次调用）的场景。如果只用一次，匿名内部类更简洁。

④ 匿名内部类（Anonymous Inner Class）

定义与实例化同时发生，没有类名。它是局部内部类的简化形式，只能被使用一次（无法再 new 第二个同类型实例）。Lambda 出现前，它是 Java 实现回调、策略、监听器的标准手段。

语法形式

匿名内部类的三种写法
// 1. 实现接口
Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("running");
    }
};

// 2. 继承抽象类（或具体类）
Thread t = new Thread() {
    @Override
    public void run() {
        System.out.println("thread running");
    }
};

// 3. Java 8+ 等价 Lambda（函数式接口才能用）
Runnable r2 = () -> System.out.println("running");  // 更简洁

经典使用场景：双括号初始化（反例！）

双括号初始化 — 看起来优雅，实则危险
// 反例：双括号初始化
List<String> list = new ArrayList<>() {{
    add("a");
    add("b");
}};
// 问题：每次都创建一个 ArrayList 的匿名子类，
//       持有外部类引用，serialVersionUID 不一致，
//       equals() 比较可能失败，增加类加载负担

// 正例：Java 9+ 用工厂方法
List<String> list2 = List.of("a", "b");
匿名内部类 vs Lambda：匿名内部类的 this 指向匿名类自身实例；Lambda 的 this 指向外层类实例。这是二者最容易踩坑的区别。另外，匿名内部类可以实现有多个方法的接口，Lambda 只能用于函数式接口（单抽象方法）。

四种内部类横向对比

类型 持有外部引用 能访问外部实例成员 能定义静态成员 有类名 典型用途
static 嵌套类 否 否（仅静态） 是 是 Builder、Entry、工具类
成员内部类 是 是 仅常量 是 迭代器、视图类

JDK 源码中的内部类

理解内部类后，再读 JDK 源码会豁然开朗：

  • HashMap.Node<K,V>：静态嵌套类，存储键值对节点，不需要持有 HashMap 引用
  • ArrayList.Itr：成员内部类，实现 Iterator，持有外部 ArrayList 的 cursor 和 modCount
  • AbstractList.SubList：成员内部类，subList() 返回的视图对象，修改视图会影响原列表
  • ReentrantLock 中的 Sync / FairSync / NonfairSync：静态嵌套类层次结构
  • 各种 Swing/AWT 的事件监听器：大量使用匿名内部类（Java 8 前）
ArrayList.Itr 简化版（理解成员内部类的真实应用）
public class ArrayList<E> extends AbstractList<E> {
    private Object[] elementData;
    private int size;

    // 成员内部类，可直接访问 elementData 和 size
    private class Itr implements Iterator<E> {
        int cursor;       // 下一个元素的下标
        int lastRet = -1;

        public boolean hasNext() {
            return cursor != size;  // 直接用外部类的 size
        }

        @SuppressWarnings("unchecked")
        public E next() {
            int i = cursor++;
            return (E) elementData[i];  // 直接用外部类的 elementData
        }
    }
}

常见陷阱与注意事项

陷阱 1：成员内部类不能有 static 字段（Java 16 之前）

Java 16 之前，成员内部类不能声明 static 字段（除了编译期常量 static final）。Java 16 放开了这个限制，允许内部类有静态成员。

陷阱 2：序列化问题

成员内部类和匿名内部类持有外部类引用，序列化时会连外部类一起序列化，可能导致意外的序列化行为或 NotSerializableException。

陷阱 3：this 指针混淆

区分 this 的两层含义
public class Outer {
    int x = 10;

    class Inner {
        int x = 20;

        void print() {
            System.out.println(this.x);        // 20：Inner.this.x
            System.out.println(Outer.this.x);  // 10：Outer.this.x
        }
    }
}

陷阱 4：局部变量捕获后被修改

effectively final 违规示例
void demo() {
    int count = 0;
    Runnable r = new Runnable() {
        public void run() {
            // 编译错误：count 在下面被重新赋值，不是 effectively final
            System.out.println(count);
        }
    };
    count = 1;  // ← 这行导致上面编译失败
}

// 绕过方式（如需可变状态）：用数组或 AtomicInteger 包装
int[] counter = {0};  // 数组引用本身不变，内容可变
Runnable r2 = new Runnable() {
    public void run() { counter[0]++; }
};

面试高频考点

  • 静态嵌套类 vs 成员内部类的区别？ — 是否持有外部类引用、是否需要外部类实例才能实例化、内存泄漏风险
  • 匿名内部类为什么只能访问 effectively final 变量？ — 栈帧生命周期 vs 对象生命周期，编译器拷贝副本的语义一致性问题
  • 匿名内部类和 Lambda 的区别？ — this 指向不同；Lambda 只能用于函数式接口；Lambda 不会生成独立 class 文件（invokedynamic）
  • 内部类如何访问外部类的 private 成员？ — 编译器为外部类生成 package-private 的 access 方法（Java 11 之前），Java 11 引入 Nest 机制（NestHost/NestMembers）直接允许
  • 内部类的 class 文件命名规则？ — Outer$Inner.class；匿名内部类为 Outer$1.class、Outer$2.class……
  • 成员内部类能否有静态方法？ — Java 16 之前不能（除 static final 常量），Java 16 起允许
实际开发建议：优先选择静态嵌套类（无内存泄漏风险）。只有当内部类确实需要频繁访问外部类实例状态时才使用成员内部类。Lambda 能替代的匿名内部类就用 Lambda，代码更简洁、性能更优（invokedynamic 懒加载）。

与其他知识点的联系

  • 设计模式：静态嵌套类 → Builder 模式；成员内部类 → Iterator 模式；匿名内部类 → Strategy / Observer 模式
  • Lambda 与函数式编程：Lambda 是匿名内部类（函数式接口）的语法糖，但底层用 invokedynamic 实现，无额外 class 文件
  • 泛型：内部类常与泛型结合（如 Map.Entry<K,V>），下一节将深入讲解
  • 并发：ReentrantLock 的 Sync 层次结构是静态嵌套类的经典应用；Android 的 Handler 内存泄漏是成员内部类危害的典型案例
  • 反射：通过 Class.getEnclosingClass()、Class.getDeclaredClasses() 可以在运行时遍历内部类关系
