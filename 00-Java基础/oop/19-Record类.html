19. Record 类（JDK 14+）

用一行声明替代几十行样板代码——Record 是 Java 对"数据载体类"的原生抽象。

1. 问题根源：Java 的"数据载体"困境

在 Java 里，如果你想表示一个不可变的二维坐标点，传统写法需要：

传统 POJO 写法（约 40 行样板代码）
public final class Point {
    private final int x;
    private final int y;

    // 构造器
    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    // Getter（没有 setter，因为不可变）
    public int x() { return x; }
    public int y() { return y; }

    // equals / hashCode
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;
        return x == p.x && y == p.y;
    }
    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    // toString
    @Override
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}

这些代码没有任何业务逻辑，只是在描述"我有两个字段"。Record 的设计目标就是：让语义意图直接成为代码，消除这类纯机械样板。

Record 写法（1 行）
public record Point(int x, int y) {}

编译器自动生成：全参构造器、accessor 方法、equals、hashCode、toString，全部基于你声明的"组件"（components）。

版本历程：JDK 14 / 15 为预览特性（需 --enable-preview），JDK 16 正式转正（JEP 395）。

2. 核心语法全览

Record 的完整语法结构
// 基础形式：record 关键字 + 名称 + 组件列表（header）
public record Person(String name, int age) {

    // ① 紧凑构造器（Compact Constructor）—— 最常用的定制入口
    // 不写参数列表，编译器自动把赋值语句追加到末尾
    public Person {
        if (age < 0)
            throw new IllegalArgumentException("年龄不能为负: " + age);
        // 无需写 this.name = name; — 编译器自动补
    }

    // ② 自定义实例方法
    public String greeting() {
        return "Hi, I'm " + name + " and I'm " + age + " years old.";
    }

    // ③ 静态工厂方法（常见惯例）
    public static Person of(String name, int age) {
        return new Person(name, age);
    }

    // ④ 覆盖自动生成的 accessor（可以做包装，但一般不需要）
    @Override
    public String name() {
        return name.trim();  // 返回前去掉首尾空格
    }
}
Accessor 命名规则：Record 生成的访问方法名与字段同名（name() 而非 getName()），这是有意与传统 JavaBean 风格区分，强调 Record 是"数据型"而非"对象型"。

3. 编译器在背后做了什么

理解 Record 的本质，最好看 javap 反编译结果。下面是 record Point(int x, int y) {} 展开后的等价 class：

编译器为 Record 自动生成的成员
├── private final 字段（每个组件一个）
├── 全参构造器（canonical constructor）
├── accessor 方法：x()、y()（与字段同名）
├── equals()：比较所有组件，基于 invokedynamic
├── hashCode()：基于所有组件，invokedynamic
└── toString()：格式 "Point[x=1, y=2]"，invokedynamic

为什么用 invokedynamic？JVM 在运行时生成 equals/hashCode/toString 的实现，避免在每个 Record 的字节码里重复写循环逻辑，这是与 Kotlin data class 的底层实现差异之一。

验证自动生成行为
Point p1 = new Point(3, 4);
Point p2 = new Point(3, 4);

System.out.println(p1.x());          // 3  — accessor
System.out.println(p1);             // Point[x=3, y=4]  — toString
System.out.println(p1.equals(p2));  // true  — 值相等即相等
System.out.println(p1 == p2);       // false — 引用不同

4. 三种构造器写法对比

写法 示例 适用场景
紧凑构造器（推荐） public Point { validate(); } 参数校验、规范化处理
规范构造器 public Point(int x, int y) { ... } 需要完全控制赋值逻辑
自定义重载构造器 public Point(int xy) { this(xy, xy); } 提供便捷创建方式，必须委托规范构造器
紧凑构造器：规范化 + 防御性拷贝
public record Range(int min, int max) {
    // 紧凑构造器：自动把 min/max 赋值给字段，我们只写校验逻辑
    public Range {
        if (min > max)
            // 也可以选择自动修正，而不是抛异常
            throw new IllegalArgumentException(
                "min (%d) > max (%d)".formatted(min, max));
    }
}

// 组件含数组时，紧凑构造器做防御性拷贝
public record Matrix(int[] data) {
    public Matrix {
        // 必须在紧凑构造器里重新赋值，才能替换掉外部传入的引用
        data = data.clone();
    }
    @Override
    public int[] data() {
        return data.clone();  // 读取时也拷贝，防止外部修改
    }
}

5. Record 的限制（设计边界）

这些限制不是"缺陷"，而是 Record 语义约束的体现——Record 代表的是纯数据，不是通用对象：

  • 不能继承类：Record 隐式 extends java.lang.Record（final class），无法再继承其他类
  • 不能被继承：Record 类本身是 final 的
  • 不能添加实例字段：只能在 header 里声明组件，不能在 body 里加 private int extra;
  • 组件字段天然不可变：private final，没有 setter
  • 可以实现接口：Record 实现接口完全合法，常用于与 sealed interface 配合
陷阱：浅层不可变
Record 的 final 只保证字段引用不变，如果组件是可变对象（如 List、数组），内容仍可被外部修改。必须手动做防御性拷贝（见上面 Matrix 示例）。

6. 与 Sealed Interface 协作：代数数据类型

Record + Sealed Interface 是 Java 对函数式编程中代数数据类型（ADT）的表达，非常适合建模有限状态的领域对象：

建模"形状"——sealed + record 组合
// sealed interface 限定了所有子类型
public sealed interface Shape
    permits Circle, Rectangle, Triangle {
    double area();
}

// 每种形状用 record 表达——不可变、有意义的组件
public record Circle(double radius) implements Shape {
    @Override
    public double area() { return Math.PI * radius * radius; }
}
public record Rectangle(double w, double h) implements Shape {
    @Override
    public double area() { return w * h; }
}
public record Triangle(double base, double height) implements Shape {
    @Override
    public double area() { return 0.5 * base * height; }
}

// JDK 21 模式匹配 switch 可穷举所有分支
static String describe(Shape s) {
    return switch (s) {
        case Circle    c -> "圆，半径 " + c.radius();
        case Rectangle r -> "矩形 " + r.w() + "×" + r.h();
        case Triangle  t -> "三角形，底 " + t.base();
        // 编译器知道 sealed 类型已穷举，无需 default
    };
}

7. Record 的典型使用场景

① DTO / Value Object

API 响应 DTO
// 以前需要 Lombok @Data，现在 Record 原生支持
public record UserDto(Long id, String username, String email) {}

// Spring MVC 可直接绑定 Record 作为请求体
@PostMapping("/users")
public ResponseEntity<UserDto> create(@RequestBody UserDto dto) {
    return ResponseEntity.ok(dto);
}

② Map 的复合 Key

Record 作为 HashMap 的 Key
// 以前需要手写 equals+hashCode 才能当 Map key
record CacheKey(String userId, String resourceId) {}

Map<CacheKey, Permission> cache = new HashMap<>();
cache.put(new CacheKey("u1", "doc-42"), Permission.READ);
// Record 自动实现了正确的 equals/hashCode，完全符合 Map key 契约

③ 多返回值

方法返回多个值
// 局部 record（JDK 16+ 可在方法内部声明）
public Stats analyze(List<Integer> nums) {
    record Stats(double avg, int max, int min) {}
    return new Stats(
        nums.stream().mapToInt(Integer::intValue).average().orElse(0),
        Collections.max(nums),
        Collections.min(nums));
}

8. Record vs Lombok vs 传统 POJO

维度 传统 POJO Lombok @Data Record
代码量 多（手写） 少（注解生成） 最少（语言内置）
不可变性 需手动 @Value 才不可变 天然不可变
调试友好度 好 差（源码与字节码不对应） 好（编译器原生支持）
继承 可继承/被继承 可继承/被继承 只能实现接口，final
可变字段 可以 可以 不可以
额外依赖 无 需引入 Lombok 无（JDK 16+）
序列化（Jackson） 需 getter 自动 需 @JsonProperty 或 Jackson 2.12+
选择建议：新项目 JDK 17+ 优先用 Record 表达不可变数据；需要可变对象或 JPA Entity 时仍用传统 POJO（JPA 要求无参构造器，Record 不适合）。

9. 常见陷阱与误区

陷阱 1：把 Record 用于 JPA Entity

反例
// ❌ 错误：JPA 要求无参构造器 + 可变字段，Record 两者都没有
@Entity
public record UserEntity(Long id, String name) {}
// 运行时报 "No default constructor for entity" 异常

陷阱 2：数组组件的浅拷贝问题

反例 → 正例
// ❌ 没有防御性拷贝，外部可以修改 Record 内部的数组
record BadMatrix(int[] data) {}
int[] arr = {1, 2, 3};
BadMatrix m = new BadMatrix(arr);
arr[0] = 999;  // m.data()[0] 也变成 999，"不可变"被破坏！

// ✅ 正确：紧凑构造器 + accessor 双重防御拷贝（见第 4 节 Matrix 示例）

陷阱 3：Jackson 反序列化失败

Jackson 2.12 以前不支持直接反序列化 Record。升级到 Jackson 2.12+ 或添加 jackson-module-parameter-names 模块即可解决。Spring Boot 2.5+ 已内置支持。

陷阱 4：混淆 accessor 与 getter 命名

Record accessor 命名与 JavaBean getter 不同
record Item(String name) {}
Item item = new Item("book");
item.name();     // ✅ accessor 方法，与字段同名
item.getName();  // ❌ 不存在，Record 不生成 getXxx 风格方法
// 某些依赖 JavaBean 规范的框架（如 JSP EL）可能无法直接读取 Record 字段

10. 面试高频考点

考点汇总（建议逐条掌握）
  1. Record 与普通 class 的本质区别是什么？
    Record 是一种受限的类，隐式 final、不可变字段、编译器自动生成标准成员。语义上代表"纯数据载体"，而非通用对象。
  2. 紧凑构造器与规范构造器有什么区别？
    紧凑构造器不写参数列表和赋值语句，编译器自动在末尾追加 this.x = x；规范构造器需要手动写赋值。
  3. 为什么 Record 不能继承其他类？
    Record 隐式继承 java.lang.Record，Java 单继承限制下无法再继承其他类；同时 final 保证了 equals/hashCode 语义的稳定性。
  4. Record 的 equals 是如何实现的？
    基于 invokedynamic，比较所有组件。与普通类的区别：不能只比较部分字段，要么覆盖全部，要么不覆盖。
  5. Record 适合做 JPA Entity 吗？为什么？
    不适合。JPA 要求无参构造器和可变字段，Record 两者都不满足。
  6. Record 与 Kotlin data class 的主要区别？
    data class 有 copy()、componentN()（解构），可以继承、可以有可变字段（var）；Record 更严格、更纯粹。

11. 建立联系：Record 在 Java 生态中的位置

Java 现代特性演进路线
├── Record（JDK 16） ←── 本文
├── Sealed Interface（JDK 17）── 与 Record 组合表达 ADT
├── Pattern Matching for switch（JDK 21）── 解构 Record 组件
├── Text Blocks（JDK 15）── 字符串字面量增强
└── instanceof Pattern Matching（JDK 16）── 类型模式匹配

Record 是 Project Amber（Java 语言演进项目）的核心成果之一，与 sealed interface、模式匹配共同构成 Java 对函数式 + 数据导向编程的支撑。在 JDK 21 中，你可以这样解构 Record：

JDK 21 Record 模式解构（Pattern Matching）
record Point(int x, int y) {}

static String format(Object obj) {
    return switch (obj) {
        // 直接解构 Record 组件，无需调用 accessor
        case Point(int x, int y) when x == 0 -> "Y 轴上的点，y=" + y;
        case Point(int x, int y)           -> "普通点 (%d, %d)".formatted(x, y);
        default                              -> "不是点";
    };
}
