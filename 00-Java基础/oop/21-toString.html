21. toString()

把对象"说清楚"——为什么重写它、怎么写对它，以及它背后藏着哪些你不知道的细节。

① 为什么 Object 要提供 toString()

Java 的根类 Object 定义了一个协议：任何对象都可以被转换成一段人类可读的字符串。 这个设计的动机非常实用——日志、调试、异常消息、SQL 拼接……到处都需要把对象"打印"出来。

如果没有这个约定，每个框架都得发明自己的 describe()、dump()……生态会一片混乱。 有了统一的 toString()，日志库、IDE 调试器、String.valueOf()、字符串拼接运算符 + 才能无缝对接任何对象。

Object 默认实现长什么样

Object.java（JDK 源码节选）
// 返回 "类名@十六进制哈希值"，例如 com.example.User@1b6d3586
public String toString() {
    return getClass().getName()
         + "@"
         + Integer.toHexString(hashCode());
}

这个默认值对调试毫无意义——你看到 User@1b6d3586 根本不知道这个用户是谁。 这就是为什么 Effective Java（第 12 条）明确说：始终覆盖 toString()。

② toString() 被隐式调用的五个场景

很多人不知道 toString() 在什么时候会被"悄悄"调用，踩坑后才后悔没重写。

场景触发方式示例
字符串拼接 + 编译器生成 StringBuilder.append(obj)，内部调用 toString() "user=" + user
System.out.println(obj) PrintStream.println(Object) 内部调用 String.valueOf(obj) System.out.println(user)
String.valueOf(obj) 先判空，非 null 则调用 obj.toString() String.valueOf(user)
日志框架占位符 SLF4J {} 占位符在格式化时调用 toString() log.info("user={}", user)
断言 / 测试框架 AssertJ、JUnit 断言失败时用 toString() 打印期望值与实际值 assertThat(user).isEqualTo(other)
陷阱：String 拼接时若对象为 null，编译器生成的代码会输出字面量 "null"（不抛 NPE）。 但如果你显式调用 null.toString()，则直接 NullPointerException。两种写法行为不同，别搞混。

③ 如何正确重写 toString()

好的 toString() 应该包含所有对调试有意义的字段，格式简洁，必要时标注字段名。 不需要与构造器严格对称，但要让人一眼看出对象状态。

手写版（最清晰）

User.java
public class User {
    private final long   id;
    private final String name;
    private final String email;

    // 构造器、getter 省略...

    @Override
    public String toString() {
        // 格式：类名{字段=值, ...}，字段名让阅读者不用猜位置含义
        return "User{"
             + "id="    + id
             + ", name='" + name  + '\''
             + ", email='" + email + '\''
             + '}';
    }
}

// 输出：User{id=42, name='Alice', email='alice@example.com'}

使用 String.format（可读性更高，但略慢）

使用 String.format
@Override
public String toString() {
    // 适合字段较多、格式复杂的场景；热路径上避免使用（每次创建格式化对象）
    return String.format("User{id=%d, name='%s', email='%s'}",
                          id, name, email);
}

IDE / Lombok 生成（日常工程首选）

Lombok @ToString
import lombok.ToString;

@ToString                        // 自动生成，包含所有非 static 字段
@ToString(exclude = "password") // 排除敏感字段
@ToString(of = {"id", "name"})   // 只包含指定字段
public class User { ... }

④ 继承场景中的 toString()

子类重写 toString() 时，往往需要包含父类的字段。 直接调用 super.toString() 是常见做法，但要注意格式是否统一。

继承中调用 super.toString()
public class Employee extends User {
    private String department;

    @Override
    public String toString() {
        // super.toString() 拿到父类的描述，再追加子类字段
        return "Employee{"
             + "user="       + super.toString()
             + ", department='" + department + '\''
             + '}';
    }
}

// 输出：Employee{user=User{id=42, name='Alice', email='...'}, department='Engineering'}

⑤ 与 equals() / hashCode() 的关系

toString()、equals()、hashCode() 是 Object 的"三件套"，但它们的职责完全不同：

方法职责必须一起重写？
equals() 定义"逻辑相等" 是，equals 与 hashCode 必须同步
hashCode() 为散列集合提供桶索引 是，equals 相等则 hashCode 必须相等
toString() 提供可读的调试/日志字符串 否，独立契约，与前两者无约束关系

toString() 展示的字段不必与 equals() 所比较的字段完全一致。 例如一个实体类可能只用 id 做相等判断，但 toString() 输出 id + name + email 便于日志排查。

⑥ 常见陷阱

陷阱 1：循环引用导致 StackOverflowError

反例：双向关联
public class Order {
    private Customer customer;   // Order → Customer
    @Override public String toString() {
        return "Order{customer=" + customer + "}"; // 触发 customer.toString()
    }
}

public class Customer {
    private List<Order> orders;  // Customer → Order（双向！）
    @Override public String toString() {
        return "Customer{orders=" + orders + "}"; // 再触发 order.toString() → 死循环！
    }
}
修复：在 toString() 中只输出关联对象的 ID（如 customerId），而不是整个对象。 Lombok 的 @ToString(exclude = "orders") 也能规避这个问题。

陷阱 2：暴露敏感字段

密码、Token、身份证号这类敏感字段绝对不能出现在 toString() 里——日志文件会落盘、传输，安全审计会直接挂你。

正例：排除敏感字段
@Override
public String toString() {
    // password 字段故意不包含
    return "User{id=" + id + ", name='" + name + '\'' + '}';
}

陷阱 3：toString() 里做了有副作用的操作

toString() 应该是纯观察，不能修改对象状态、不能发起网络/DB 请求、不能抛出受检异常。 调试器在鼠标悬停时会隐式调用 toString()，如果有副作用，你的程序状态会在调试过程中被改变——这是噩梦级 bug。

陷阱 4：数组的 toString() 输出无意义

数组的坑
int[] arr = {1, 2, 3};
System.out.println(arr);               // 输出 [I@1b6d3586，毫无意义！
System.out.println(Arrays.toString(arr)); // 正确：[1, 2, 3]

String[][] matrix = ...;
System.out.println(Arrays.deepToString(matrix)); // 多维数组用 deepToString

数组没有重写 toString()，继承的是 Object 的默认实现。 [I 是 JVM 内部类型描述符，[I 代表 int[]，[Ljava.lang.String; 代表 String[]。

⑦ 字符串拼接底层：+ 运算符与 toString() 的关系

"prefix" + obj 这行代码，Java 编译器（javac）会把它翻译成：

编译器生成的字节码等价代码（Java 8）
// 源码：String result = "user=" + user;
// 编译后等价于：
String result = new StringBuilder()
        .append("user=")
        .append(user)   // StringBuilder.append(Object) → String.valueOf(obj) → obj.toString()
        .toString();

Java 9+ 引入了 invokedynamic 字符串拼接优化（StringConcatFactory），底层实现换了， 但对 toString() 的调用语义不变——对象的 toString() 仍然会被触发。

⑧ 与 Java 生态的联系

toString() 在生态中的位置
├─ Object.toString() ← 根协议
├─ String.valueOf(obj) ← 空安全包装，内部调用 toString()
├─ StringBuilder.append(obj) ← 字符串拼接底层
├─ PrintStream.println(obj) ← System.out.println
├─ SLF4J / Logback {}占位符 ← 日志框架延迟调用
├─ Jackson ObjectMapper ← 序列化时不调用 toString()（走反射/注解）
└─ Lombok @ToString ← 编译时代码生成，无运行时开销
Jackson 的误区：很多人以为 @JsonSerialize 会走 toString()，实际上 Jackson 用反射读字段/getter， 与 toString() 完全无关。只有在配置了 WRITE_ENUMS_USING_TO_STRING 等特殊选项时，枚举才会走 toString()。

⑨ 面试高频考点

  • Q：为什么说"始终覆盖 toString()"？ 默认实现只输出类名和哈希值，调试价值为零。重写后，日志、异常信息、调试器都能直接读懂对象状态，降低排查成本。
  • Q：toString() 与 equals()/hashCode() 有什么约束关系？ 没有强制约束，三者职责独立。但良好实践是：equals() 使用的字段在 toString() 中也应可见，便于调试相等性断言失败时定位问题。
  • Q：null + "" 会报 NPE 吗？ 不会，输出字面量 "null"。因为 StringBuilder.append(Object) 内部先判断 null，null 时直接追加 "null" 字符串，不调用 toString()。
  • Q：打印数组为什么输出 [I@xxxxx？ 数组是 Object 子类但没有重写 toString()，走的是默认实现。应用 Arrays.toString() 或 Arrays.deepToString()。
  • Q：toString() 里可以抛异常吗？ 理论上可以，但强烈不建议。调试器、日志框架都会悄悄调用它，一旦抛异常会掩盖真正的 bug，让排查极其困难。
最佳实践总结：
  • 每个领域对象都重写 toString()，用 IDE 生成或 Lombok @ToString
  • 排除敏感字段（密码、Token、证件号）
  • 有双向关联时只输出对方的 ID，避免循环引用
  • toString() 保持纯查询，无副作用、无 IO、无异常
  • 打印数组用 Arrays.toString()，不要直接拼接数组
