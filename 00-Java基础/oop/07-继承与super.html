07. 继承与 super 关键字

extends 建立 IS-A 关系；super 是子类与父类沟通的桥梁。理解继承，才能真正理解多态。

1. 为什么需要继承

继承最常被解释为"代码复用"——但这是误导。继承的本质是建立 IS-A 类型关系： Dog IS-A Animal，这让 Dog 可以被当作 Animal 使用，这才是继承真正的威力（多态的基础）。

如果只想复用代码，组合（Composition）往往更合适。继承带来的是强耦合： 子类依赖父类的实现细节，父类的任何修改都可能悄悄破坏子类——这就是著名的 脆弱基类问题（Fragile Base Class Problem）。

经典失误（JDK 亲历）：Stack extends Vector。 Stack 本应是一个只支持 push/pop 的栈，但因为继承了 Vector，外部代码可以调用 stack.add(0, element) 从底部插入，完全违反栈的语义。 这是"为了复用而继承"的教科书级反例，后来 Deque 接口才是正确方案。
使用继承的判断标准：问自己"子类 IS-A 父类吗？在任何上下文下都成立吗？" 如果只是"子类 HAS-A 父类的功能"，请用组合。

2. extends 基本语法

Java 用 extends 关键字声明继承，且只支持单继承 （一个类只能有一个直接父类）。所有类隐式继承自 Object。

Animal.java + Dog.java — 完整继承示例
public class Animal {

    // private 字段：子类不能直接访问，必须通过方法
    private String name;
    private int    age;

    // 父类构造器（不可被继承，但会被 super() 调用）
    public Animal(String name, int age) {
        this.name = name;
        this.age  = age;
    }

    public String getName() { return name; }
    public int    getAge()  { return age; }

    public void speak() {
        System.out.println(name + " 发出声音");
    }
}

public class Dog extends Animal {

    // 子类专有字段
    private String breed;

    public Dog(String name, int age, String breed) {
        // 必须先调用父类构造器，且必须是第一行
        super(name, age);
        this.breed = breed;
    }

    // 覆盖父类方法，扩展行为
    @Override
    public void speak() {
        System.out.println(getName() + " 汪汪叫！品种：" + breed);
    }
}

注意 Dog 中无法直接访问 name 和 age，因为它们是 Animal 的 private 字段。 子类必须通过父类提供的 getName()、getAge() 等方法访问。 这是封装的体现：父类控制自己的数据，子类通过接口而非直接操作字段来交互。

3. 对象内存模型：堆上的"套娃"结构

创建 new Dog("旺财", 3, "拉布拉多") 时，JVM 在堆上分配的空间同时包含 Animal 部分和 Dog 部分，是一个整体对象，不是两个对象：

堆内存（一块连续空间）
┌─────────────────────────────────┐
│ [ Animal 部分 ] │
│ name = "旺财" │
│ age = 3 │
│ │
│ [ Dog 部分 ] │
│ breed = "拉布拉多" │
└─────────────────────────────────┘
↑
dog 引用（栈上）

super 关键字本质上就是"访问同一个对象中属于父类的那一部分"。 它不是父类对象的引用，而是在同一个对象内部选择调用哪个版本的方法或字段。

4. super 的三种用法

用法一：构造器链 — super()

在子类构造器中调用父类构造器，必须是构造器的第一条语句。 这保证父类部分先被正确初始化。

super() — 调用父类构造器
public class Vehicle {
    private String brand;
    private int    year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year  = year;
    }
}

public class Car extends Vehicle {
    private int doors;

    public Car(String brand, int year, int doors) {
        // ① 第一行必须是 super()，否则编译报错
        super(brand, year);
        // ② 之后才能初始化子类自身字段
        this.doors = doors;
    }
}

用法二：方法扩展 — super.method()

子类覆盖父类方法后，若想在子类逻辑之外复用父类的实现，用 super.method()。 这实现了"扩展"而非"替换"父类行为。

super.method() — 扩展父类方法
public class Manager extends Employee {

    private String department;

    @Override
    public String toString() {
        // 复用父类的 toString，再追加经理专属信息
        return super.toString() + " [部门: " + department + "]";
    }

    @Override
    public double computeBonus() {
        // 在父类奖金基础上再加部门奖，而不是重写整个逻辑
        return super.computeBonus() + 5000;
    }
}

用法三：字段消歧 — super.field

当子类定义了与父类同名的字段时（字段遮蔽），用 super.field 明确访问父类的那个字段。 但实际开发中应避免字段遮蔽，这只是语言允许的一个边角特性。

super.field — 字段消歧（了解即可，实践中避免）
class Base {
    String name = "Base";
}

class Child extends Base {
    String name = "Child";  // 遮蔽父类的 name

    void print() {
        System.out.println(name);        // → "Child"（子类字段）
        System.out.println(super.name);  // → "Base"（父类字段）
    }
}

5. 隐式 super() — 编译器自动插入

如果子类构造器没有显式调用 super() 或 this()， 编译器会在第一行自动插入 super()（调用父类无参构造器）。 这个隐式行为是很多编译错误的根源：

隐式 super() 引发的编译错误
class Animal {
    // 定义了有参构造器后，编译器不再自动生成无参构造器
    public Animal(String name) { ... }
}

class Dog extends Animal {
    public Dog() {
        // 编译器在此插入 super()，但 Animal 没有无参构造器
        // ❌ 编译错误：There is no default constructor available in 'Animal'
    }
}

// 修复方案：显式调用有参构造器
class Dog extends Animal {
    public Dog() {
        super("未命名");  // ✅ 明确告诉编译器用哪个构造器
    }
}
规律：一旦你在父类中定义了任何构造器，编译器就不再自动生成无参构造器。 如果父类有有参构造器但没有无参构造器，所有子类都必须显式调用 super(有参版本)。

6. 字段遮蔽陷阱：静态绑定 vs 动态绑定

这是 Java 继承中最容易让人困惑的点：字段访问是静态绑定（编译时确定），方法调用是动态绑定（运行时确定）。

字段静态绑定 vs 方法动态绑定 — 反例验证
class Parent {
    String type = "Parent";

    String getType() { return "Parent"; }
}

class Child extends Parent {
    String type = "Child";  // 遮蔽父类字段

    @Override
    String getType() { return "Child"; }
}

Parent obj = new Child();  // 父类引用，指向子类对象

System.out.println(obj.type);      // → "Parent"（字段静态绑定，看引用类型 Parent）
System.out.println(obj.getType());  // → "Child"（方法动态绑定，看对象类型 Child）
陷阱总结：
  • 字段不参与多态，访问哪个字段由引用的声明类型决定
  • 方法参与多态，调用哪个方法由对象的实际类型决定
  • 所以：永远不要定义与父类同名的字段，这只会造成混乱

7. 继承 vs 组合

对比维度 继承（extends） 组合（has-a 字段）
关系类型 IS-A（是一种） HAS-A（拥有一个）
耦合程度 强耦合，父类实现影响子类 弱耦合，可替换不同实现
灵活性 运行时无法改变父类 可在运行时替换组合对象
封装性 父类实现对子类部分可见 内部完全封装
适用场景 真正的分类关系 + 需要多态 功能复用，没有类型关系
Stack 正确设计：组合 Deque 而非继承 Vector
// 错误：继承 Vector 导致 Stack 暴露了 add(index, element) 等方法
// class Stack<E> extends Vector<E> { ... }  ← JDK 的历史错误

// 正确：用组合，把 Deque 作为内部实现
public class Stack<E> {

    // 内部持有 Deque，不暴露 Deque 的任何接口给外部
    private final Deque<E> deque = new ArrayDeque<>();

    public void push(E item) { deque.push(item); }
    public E    pop()        { return deque.pop(); }
    public E    peek()       { return deque.peek(); }
    public boolean isEmpty() { return deque.isEmpty(); }
    // 对外完全看不到 Deque 的其他方法，语义纯粹
}

8. final 与继承

final 与继承密切相关，有三种用法：

用法 含义 典型例子
final class 类不可被继承 String、Integer、所有包装类
final method 方法不可被覆盖 Object.getClass()
final field 字段只能赋值一次（常量或不可变性） Math.PI、各种 ID 字段

String 为什么是 final？

String 被设计为不可变（Immutable）对象，不可变性带来：

  • 线程安全：不可变对象可以自由共享，无需同步
  • 字符串常量池：JVM 能放心地让多个引用指向同一个字符串实例
  • 安全性：作为 HashMap 的 key、文件路径、类名时，不被意外修改
  • 缓存 hashCode：不可变则 hash 值不会变，可以安全缓存

如果 String 可被继承，子类可以覆盖 charAt()、length() 等方法， JVM 和标准库对 String 不可变性的所有假设都会被打破。 final class 从语言层面封死了这条路。

9. 完整初始化顺序

当 new Dog(...) 时，JVM 按以下严格顺序执行初始化， 父类总是先于子类：

1. 父类静态字段 初始化 & 父类静态初始化块（仅类首次加载时执行一次）
2. 子类静态字段 初始化 & 子类静态初始化块（仅类首次加载时执行一次）
── 以上是类加载阶段，后续每次 new 都会重复以下步骤 ──
3. 父类实例字段 初始化 & 父类实例初始化块
4. 父类构造器 方法体执行
5. 子类实例字段 初始化 & 子类实例初始化块
6. 子类构造器 方法体执行（super() 之后的部分）
可运行的初始化顺序验证代码
class Parent {
    static { System.out.println("1. 父类静态块"); }
    {       System.out.println("3. 父类实例块"); }
    public Parent() { System.out.println("4. 父类构造器"); }
}

class Child extends Parent {
    static { System.out.println("2. 子类静态块"); }
    {       System.out.println("5. 子类实例块"); }
    public Child() {
        // super() 隐式调用，对应第 3、4 步
        System.out.println("6. 子类构造器"); }
}

// 运行 new Child() 的输出：
// 1. 父类静态块
// 2. 子类静态块
// 3. 父类实例块
// 4. 父类构造器
// 5. 子类实例块
// 6. 子类构造器
构造器中调用可被覆盖方法的危险： 如果父类构造器中调用了一个被子类覆盖的方法，此时子类字段尚未初始化（第 5 步还没到）， 子类方法看到的是字段的默认值（0、null、false），导致诡异 Bug。 规则：构造器中只调用 private 或 final 方法。

10. 面试高频考点

Q1：Java 为什么不支持多继承（类）？
多继承会引发菱形问题（Diamond Problem）：若 A 和 B 都有方法 m()， C 同时继承 A 和 B，C 不知道该用哪个 m()，且构造器链会导致公共祖先被初始化多次。 Java 用接口（默认方法冲突时强制子类覆盖）解决了多类型的需求，同时避免了多继承的复杂性。
Q2：super() 和 this() 可以同时出现在一个构造器中吗？
不能。两者都必须是构造器的第一条语句，因此互斥——一个构造器中只能有其中一个。 this() 会转发到本类另一个构造器，最终链上的某个构造器会调用 super()。
Q3：构造器可以被继承吗？
不能。构造器不是普通方法，它与类名强绑定，无法被子类继承。 子类通过 super() 显式或隐式地调用父类构造器，但这不是继承。
Q4：所有类的最终父类是什么？
java.lang.Object。它提供了 equals()、hashCode()、 toString()、clone()、getClass()、 wait()/notify()/notifyAll() 等基础方法。 即使你写 class Foo { }，它隐式等价于 class Foo extends Object { }。
Q5：private 方法能被继承和覆盖吗？
不能。private 方法对子类不可见，子类定义同名方法是全新的方法（方法隐藏）， 不是覆盖（override）。加 @Override 注解编译器会报错，正因为如此推荐总是加该注解。
Q6：继承关系中，字段和方法的绑定方式有何不同？
字段是静态绑定：编译时根据引用的声明类型确定，访问父类引用则访问父类字段。
方法是动态绑定：运行时根据对象的实际类型确定，这正是多态的底层机制（虚方法表 vtable）。
