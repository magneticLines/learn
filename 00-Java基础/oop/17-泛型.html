17. 泛型（类 / 方法 / 通配符 ? extends / ? super）

让类型成为参数——编写一次，安全地用于任何类型，同时在编译期把类型错误挡在门外。

1. 为什么需要泛型？

泛型（Generics）在 Java 5 引入，核心动机是类型安全 + 代码复用。先看没有泛型时的痛点：

Java（泛型之前）
// 没有泛型时，集合只能存 Object，取出时必须强转
List list = new ArrayList();
list.add("hello");
list.add(42);            // 编译通过，运行时却埋着炸弹
String s = (String) list.get(1);  // 运行时 ClassCastException！

泛型把类型作为参数传入，让编译器帮你检查每一次存取是否类型匹配，错误在编译期就暴露，而不是运行时崩溃。

设计哲学：把"这个容器里装什么"这个信息从注释/文档提升为编译器可验证的约束。这是从动态安全向静态安全迈进的一大步。

2. 泛型类

在类名后加 <T> 声明类型参数，T 是占位符，代表"使用时才决定的类型"。

GenericBox.java
// T 是 type parameter，命名惯例：T=Type, E=Element, K=Key, V=Value, N=Number
public class Box<T> {

    private T value;

    public Box(T value) {
        this.value = value;
    }

    public T get() { return value; }

    public void set(T value) { this.value = value; }
}

// 使用：类型参数在实例化时绑定
Box<String> strBox = new Box<>("hello");  // 钻石运算符 <> 让编译器推断
Box<Integer> intBox = new Box<>(42);

String s = strBox.get();   // 无需强转，编译器已确保类型安全
// strBox.set(100);       // 编译错误！类型不匹配

多类型参数

Pair.java
// 多个类型参数，Map.Entry<K,V> 就是这种结构
public class Pair<K, V> {
    private final K key;
    private final V value;

    public Pair(K key, V value) {
        this.key = key; this.value = value;
    }

    public K getKey()   { return key; }
    public V getValue() { return value; }
}

Pair<String, Integer> p = new Pair<>("age", 18);

3. 泛型方法

类型参数也可以只作用于单个方法，在返回值前声明 <T>。这样即使是非泛型类，也可以有泛型方法。

ArrayUtils.java
public class ArrayUtils {

    // <T> 在返回类型前声明，T 由调用时传入的实参推断
    public static <T> T[] fill(T[] arr, T val) {
        for (int i = 0; i < arr.length; i++)
            arr[i] = val;
        return arr;
    }

    // 带上界的泛型方法：T 必须实现 Comparable，才能比较大小
    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) >= 0 ? a : b;
    }
}

// 调用时 T 由编译器自动推断，无需显式传入
String[] words = ArrayUtils.fill(new String[3], "ok");
String bigger = ArrayUtils.max("apple", "banana");  // "banana"
泛型类 vs 泛型方法：如果类型参数只在一两个方法中出现，优先用泛型方法而不是泛型类。泛型类的 T 贯穿整个实例的生命周期，泛型方法的 T 只在一次调用中存活。

4. 类型擦除：理解泛型的本质

Java 泛型是编译期特性，运行时 JVM 并不知道泛型参数。编译器将所有泛型类型替换为上界（默认 Object），这叫做类型擦除（Type Erasure）。

源代码（有泛型） → 字节码（类型擦除后） ───────────────────────────────────────────── Box<String> → Box（T 变成 Object） Box<T extends Num> → Box（T 变成 Num） List<Integer> → List（E 变成 Object）
类型擦除的影响
// 运行时无法获取泛型的实际类型
List<String> strList = new ArrayList<>();
List<Integer> intList = new ArrayList<>();

// true！运行时两者都是 ArrayList，T 已被擦除
System.out.println(strList.getClass() == intList.getClass());

// 以下操作因类型擦除而被禁止：
// new T()           ← 不知道 T 是谁，无法实例化
// new T[]           ← 同理，不能创建泛型数组
// T instanceof Foo  ← 运行时 T 已消失
// static T field    ← static 共享，T 不属于类层面
陷阱：泛型与数组天生不合。new ArrayList<String>[10] 编译报错，根本原因是数组在运行时保留元素类型（协变 + 具化），泛型擦除后无法保证类型安全。应使用 List<List<String>> 代替。

5. 通配符 ?

通配符 ? 表示"未知类型"，解决泛型不变性（invariance）的限制。

Java 中 List<String> 不是 List<Object> 的子类型（尽管 String 是 Object 的子类），这叫做泛型不变。原因很简单：

为什么泛型不变——假设它可以会怎样
// 假设 List<String> 是 List<Object> 的子类（实际上不是）
List<String> strings = new ArrayList<>();
List<Object> objects = strings;   // 若被允许…
objects.add(42);                   // 向 strings 塞了一个 Integer！
String s = strings.get(0);         // 运行时 ClassCastException
// 所以 Java 禁止了这种赋值，泛型是不变的

但这导致编写通用方法时很受限。通配符 ? 正是打破这个限制的工具。

6. 上界通配符 ? extends T（协变，只读）

? extends T 表示"T 或 T 的某个子类"。可以安全地读取（结果至少是 T），但不能写入（不知道具体是哪个子类）。

? extends —— 生产者场景
// 计算所有数字之和：入参只需"能读出 Number"即可
public static double sumList(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list)          // 读出来的都是 Number，安全
        sum += n.doubleValue();
    return sum;
}

// 现在三种 List 都能传入：
sumList(List.of(1, 2, 3));               // List<Integer>
sumList(List.of(1.5, 2.5));             // List<Double>
sumList(List.of(1L, 2L));              // List<Long>

// 但不能写入！编译器不知道具体是哪种 Number
void bad(List<? extends Number> list) {
    // list.add(1.0);  // 编译错误：可能是 List<Integer>，不能塞 Double
    list.add(null);      // null 是唯一例外（所有引用类型都可以是 null）
}

7. 下界通配符 ? super T（逆变，只写）

? super T 表示"T 或 T 的某个父类"。可以安全地写入 T 及其子类，但读出来只能当 Object（因为不知道确切的父类是哪个）。

? super —— 消费者场景
// 把一批 Integer 填入目标列表：目标只需能"接收 Integer"即可
public static void addNumbers(List<? super Integer> dest) {
    for (int i = 1; i <= 5; i++)
        dest.add(i);   // Integer 写进去一定合法
}

List<Integer> ints    = new ArrayList<>();
List<Number>  nums    = new ArrayList<>();
List<Object>  objects = new ArrayList<>();

addNumbers(ints);     // ok，Integer super Integer
addNumbers(nums);     // ok，Number  super Integer
addNumbers(objects);  // ok，Object  super Integer

// 读出来只能是 Object，因为不确定父类链有多深
void readOnly(List<? super Integer> list) {
    Object o = list.get(0);   // 只能拿 Object
    // Integer i = list.get(0);  // 编译错误
}

8. PECS 原则——记住通配符用法的口诀

Effective Java 作者 Josh Bloch 总结了一句口诀：Producer Extends, Consumer Super（生产者用 extends，消费者用 super）。

角色 通配符 能做什么 不能做什么 典型场景
生产者（向外提供数据） ? extends T 读取（得到 T） 写入 Collections.copy src
消费者（向内接收数据） ? super T 写入 T 及子类 有意义地读取 Collections.copy dest
既读又写 具体类型 T 读写都可 — 内部数据结构
Collections.copy 源码（JDK 中 PECS 的经典范例）
// src 是生产者（提供元素）→ extends
// dest 是消费者（接收元素）→ super
public static <T> void copy(List<? super    T> dest,
                              List<? extends T> src) {
    for (T t : src)   // 从 src 读，src 是生产者
        dest.add(t);   // 往 dest 写，dest 是消费者
}

// 因此可以把 List<Integer> 复制到 List<Number> 里
List<Integer> src  = List.of(1, 2, 3);
List<Number>  dest = new ArrayList<>(Collections.nCopies(3, null));
Collections.copy(dest, src);

9. 有界类型参数（Bounded Type Parameter）

在定义泛型类/方法时，也可以给类型参数加约束，用 extends 指定上界（支持多个，接口用 & 连接）：

多界约束
// T 必须同时是 Comparable 且实现了 Serializable
// 多个边界时，类放最前，接口用 & 分隔
public <T extends Comparable<T> & Serializable>
T clampAndSerialize(T val, T min, T max) {
    if (val.compareTo(min) < 0) return min;
    if (val.compareTo(max) > 0) return max;
    return val;
}

// 注意：这里的 extends 和通配符的 extends 语义相同
// 但作用位置不同：类型参数定义时 vs 通配符使用时

10. 泛型与继承

泛型类的继承方式
// 方式1：子类保留类型参数
class NumberBox<T extends Number> extends Box<T> {
    public NumberBox(T v) { super(v); }
    public double toDouble() { return get().doubleValue(); }
}

// 方式2：子类固定类型参数（特化）
class StringBox extends Box<String> {
    public StringBox(String s) { super(s); }
    public String upper() { return get().toUpperCase(); }
}

11. 常见陷阱 & 面试高频考点

陷阱 1：泛型不是协变的

陷阱：协变误解
// 数组是协变的（运行时会检查，可能 ArrayStoreException）
Object[] objs = new String[3];   // 编译通过
objs[0] = 42;                    // 运行时 ArrayStoreException

// 泛型是不变的（编译期就拦截）
List<Object> list = new ArrayList<String>(); // 编译错误！更安全

陷阱 2：原生类型（Raw Type）污染

陷阱：Raw Type
// Raw Type 是为了兼容 Java 5 之前的旧代码，现代代码禁止使用
List raw = new ArrayList();      // ← 警告：unchecked
raw.add("hello");
List<Integer> ints = raw;       // unchecked warning，但编译通过
Integer i = ints.get(0);        // 运行时 ClassCastException

// Raw Type 绕过了编译器的类型检查，埋下运行时炸弹
// 始终使用参数化类型，或至少用 List<?>

陷阱 3：堆污染（Heap Pollution）

陷阱：可变参数 + 泛型
// 泛型可变参数会产生 unchecked warning，因为内部创建了泛型数组
@SafeVarargs    // 告诉编译器：我保证这个方法不会污染堆
public static <T> List<T> listOf(T... elems) {
    return Arrays.asList(elems);  // 只读，不写，安全
}

面试高频：? extends 和 ? super 的区别

维度 ? extends T ? super T
含义 T 或 T 的子类 T 或 T 的父类
读取 安全（得到 T） 只能得到 Object
写入 不能写（null 除外） 可以写入 T 及子类
变型 协变（covariant） 逆变（contravariant）
角色 生产者 Producer 消费者 Consumer
口诀 PECS → Extends PECS → Super

12. 与其他知识的联系

泛型 ├── 类型擦除 → 与 JVM 字节码、反射有关（运行时需 TypeToken 绕过擦除） ├── 集合框架 → List<E>、Map<K,V> 都是泛型类，通配符是集合 API 设计的基础 ├── Comparable / Comparator → <T extends Comparable<T>> 是排序算法的标准约束 ├── 函数式接口 → Function<T,R>、Predicate<T>、Consumer<T> 大量使用泛型 ├── Stream API → Stream<T>，flatMap、map 等操作的类型安全来自泛型 ├── 注解处理器 → APT 在编译期处理泛型信息，生成类型安全代码 └── Kotlin / Scala → 用 out/in 关键字代替 ? extends/? super，更直观
反射与泛型：类型擦除后，运行时无法通过 obj.getClass() 得到泛型参数。Gson、Jackson 等框架使用 TypeToken（匿名子类技巧）来在运行时保留泛型信息——因为父类泛型参数不会被擦除，只有局部变量/字段的泛型参数会被擦除。
TypeToken 技巧（Gson 运行时保留泛型）
// 直接反射 List<String> 得到的是 List（擦除了）
// Gson 的 TypeToken：创建匿名子类，父类泛型参数保留在字节码中
Type type = new TypeToken<List<String>>() {}.getType();
List<String> list = new Gson().fromJson(json, type);
// 原理：TypeToken<T> 的子类（匿名类）的父类签名包含 T 的实际类型，
//       通过 getGenericSuperclass() 可以读到，不受擦除影响

小结

  • 泛型类：在类名后声明 <T>，实例化时绑定具体类型。
  • 泛型方法：在返回值前声明 <T>，T 在每次调用时独立推断。
  • 类型擦除：泛型仅在编译期存在，运行时被替换为上界（Object 或指定类型）。
  • ? extends T：只读，接收 T 及子类的集合（生产者），PECS 中的 E。
  • ? super T：可写入 T，接收 T 及父类的集合（消费者），PECS 中的 S。
  • 有界类型参数：定义时用 T extends Upper 加约束，支持多界 &。
  • 禁止使用 Raw Type，始终使用参数化类型或通配符。
