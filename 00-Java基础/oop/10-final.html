10. final 类 / 方法 / 字段

用一个关键字，向编译器和同事宣告：这里不可改变。理解 final 的三种用法，以及它如何与不可变性、JMM、内联优化产生深度联系。

为什么需要 final？

Java 的继承和多态给了我们极大的灵活性，但灵活性本身有代价：不确定性。当一个类可以被任意继承、方法可以被任意覆盖、字段可以被任意重新赋值时，代码的行为变得难以推理。

final 是 Java 里最直接的"约束声明"——它不增加功能，只减少可能性。减少可能性的好处是：编译器可以做更多优化，读者可以做更多假设，并发代码可以省掉显式同步。

设计哲学：越少的可能性 = 越强的保证。final 的每一种用法都是在对"变化"设置边界。

三种用法总览

final ├── 字段（field）：只能赋值一次，之后不可重新赋值 │ ├── 实例字段：必须在构造器或声明处完成赋值 │ └── 静态字段：必须在静态初始化块或声明处完成赋值 ├── 方法（method）：不可被子类覆盖（override） └── 类（class）：不可被继承
修饰对象含义典型用途
final 字段引用/值只能被赋值一次不可变对象、常量
final 方法禁止子类覆盖模板方法骨架、安全关键逻辑
final 类禁止继承工具类、值类（String、Integer）

final 字段

基本规则

final 字段一旦赋值就不能重新赋值。对于实例字段，编译器要求每条构造路径都必须完成赋值——否则编译报错，这是编译期安全保证，不是运行时检查。

Java — final 字段基础
public class Point {
    // 声明即赋值：最常见写法
    public final double x;
    public final double y;

    public Point(double x, double y) {
        // 在构造器中完成赋值——每条分支都必须赋值
        this.x = x;
        this.y = y;
    }

    public void move() {
        // ❌ 编译错误：cannot assign a value to final variable x
        // this.x = 1.0;
    }
}

static final：编译期常量

当 static final 字段的右侧是编译期常量表达式（字面量或常量表达式），编译器会在编译时直接把用到这个常量的地方替换成值，称为常量内联（constant folding）。

Java — 编译期常量内联
public class Config {
    // 编译期常量：基本类型或 String + 字面量
    public static final int    MAX_SIZE = 100;
    public static final String VERSION  = "1.0";

    // 非编译期常量：运行时才知道值
    public static final String UUID = UUID.randomUUID().toString();
}

class Demo {
    void test() {
        // 编译后等价于：if (100 > 50)，Config.class 不必被加载
        if (Config.MAX_SIZE > 50) { ... }
    }
}
陷阱：常量内联导致的"版本脱节"
如果你把 Config.MAX_SIZE 从 100 改成 200，只重新编译 Config.class， 而没有重新编译引用它的 Demo.class——Demo 里仍然是硬编码的 100！ 这是 Java 常量内联的副作用，部署时必须全量重编译。

final 字段与 Java 内存模型（JMM）

这是面试高频考点，也是 final 最深刻的价值之一。

JMM 为 final 字段提供了特殊的初始化安全保证（initialization safety guarantee）：只要对象构造器正常完成（没有 this 逃逸），其他线程看到这个对象的引用时，一定能看到 final 字段的完整初始化值，无需任何额外同步。

Java — final 字段的 JMM 保证
public class ImmutablePoint {
    public final int x;
    public final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
        // ✅ 构造器正常完成，JMM 保证其他线程看到的 x、y 一定是初始化后的值
    }
}

// 如果字段不是 final：
class MutablePoint {
    public int x;  // 没有 final
    public int y;
    public MutablePoint(int x, int y) { this.x = x; this.y = y; }
    // ⚠️ 在没有同步的多线程环境下，其他线程可能看到 x=0, y=0（默认值）
}
结论：不可变对象（所有字段都是 final）天然是线程安全的，不需要 synchronized 或 volatile。这正是 String、Integer 等类的设计基础。

final ≠ 深度不可变

final 只保证引用不变，不保证引用指向的对象本身不变。这是最常见的误解。

Java — final 引用 vs 对象内容
public class Trap {
    // final 只锁住"引用"，不锁住列表内容
    private final List<String> names = new ArrayList<>();

    public void demo() {
        names.add("Alice");   // ✅ 合法：修改对象内容
        names.add("Bob");     // ✅ 合法

        // ❌ 编译错误：cannot assign a value to final variable names
        // names = new ArrayList<>();
    }
}

// 真正的不可变集合需要额外包装：
private final List<String> immutableNames =
    Collections.unmodifiableList(List.of("Alice", "Bob"));

final 方法

给方法加 final，意味着任何子类都不能用 @Override 覆盖它。

为什么要禁止覆盖？

  • 安全性：防止子类篡改安全关键逻辑（如权限校验、加密算法）
  • 模板方法模式：骨架方法定义流程，步骤方法允许扩展，但骨架本身用 final 保护
  • 性能：JVM 可以对 final 方法做静态分派，甚至内联，省去虚方法查找的开销
Java — final 方法 + 模板方法模式
public abstract class DataProcessor {

    // 骨架：流程固定，子类不能改变顺序
    public final void process() {
        validate();       // 子类可扩展
        transform();     // 子类可扩展
        save();           // 子类可扩展
        audit();          // final：审计逻辑不允许子类篡改
    }

    protected abstract void validate();
    protected abstract void transform();
    protected abstract void save();

    // 安全关键：记录操作日志，任何子类都不能跳过
    private final void audit() {
        System.out.println("操作已记录到审计日志");
    }
}

class CsvProcessor extends DataProcessor {
    @Override protected void validate() { /* CSV 格式校验 */ }
    @Override protected void transform() { /* 解析 CSV */ }
    @Override protected void save() { /* 写入数据库 */ }

    // ❌ 编译错误：不能覆盖 final 方法 process()
    // @Override public void process() { ... }
}
private 方法隐式 final：private 方法天然无法被覆盖（子类根本看不到它），所以 JVM 对 private 方法的处理与 final 方法相同。显式加 final private 是多余的，但语法上合法。

final 类

加了 final 的类不能被继承。这是最强的封闭声明，JDK 中大量使用。

JDK 中的 final 类

  • String：如果能被继承，子类可以覆盖 charAt()、hashCode() 等，哈希表行为将不可预测
  • Integer、Long 等包装类：防止子类破坏自动装箱/拆箱的语义
  • Math：纯工具类，继承毫无意义
  • System：系统资源类，不允许扩展
Java — final 类
// JDK 源码（简化）：String 是 final 类
public final class String
    implements Serializable, Comparable<String>, CharSequence {
    // ... 内部实现
}

// ❌ 编译错误：cannot inherit from final 'java.lang.String'
class MyString extends String { }

// ✅ 正确做法：组合而非继承（Effective Java: Item 18）
class RichString {
    private final String value;  // 持有引用，不继承

    public RichString(String value) {
        this.value = value;
    }

    public boolean isPalindrome() {
        String reversed = new StringBuilder(value).reverse().toString();
        return value.equals(reversed);
    }
}

构造 final 局部变量

final 也可以用于局部变量（方法体内的变量），表示该变量只能赋值一次。Java 编译器不强制要求，但这是一种有益的编码习惯。

Java — final 局部变量
void calculate(int input) {
    // 声明为 final，告诉读者：这个中间值不会变化
    final int doubled = input * 2;
    final int result  = doubled + 10;

    // ❌ 编译错误：不能重新赋值
    // doubled = 0;

    System.out.println(result);
}

// lambda / 匿名类捕获：必须是 effectively final
void lambdaDemo() {
    int count = 5;  // 没写 final，但 effectively final（从未重新赋值）
    Runnable r = () -> System.out.println(count);  // ✅ 合法

    int mutable = 0;
    mutable++;  // 重新赋值 → 不再是 effectively final
    // ❌ 编译错误：Variable used in lambda should be final or effectively final
    // Runnable r2 = () -> System.out.println(mutable);
}
effectively final（Java 8+）：即使没有写 final，只要局部变量在初始化后从未被重新赋值，编译器就认为它是 "effectively final"，可以被 lambda 和匿名类捕获。

完整对比：反例 → 问题 → 正例

场景：设计一个不可变的货币金额类

❌ 反例 — 可变金额类，多线程下危险
class Money {
    private BigDecimal amount;  // 没有 final，随时可以改
    private String currency;

    public void setAmount(BigDecimal amount) {
        this.amount = amount;  // ⚠️ 多线程下不安全
    }
}

问题：在多线程环境，线程 A 正在读 amount，线程 B 同时调用 setAmount，没有同步，可能读到脏数据。

✅ 正例 — final 字段 + final 类 = 天然线程安全
public final class Money {
    // 所有字段 final：构造后不可变，JMM 提供安全发布保证
    private final BigDecimal amount;
    private final String     currency;

    public Money(BigDecimal amount, String currency) {
        this.amount   = Objects.requireNonNull(amount);
        this.currency = Objects.requireNonNull(currency);
    }

    public BigDecimal getAmount() { return amount; }
    public String     getCurrency() { return currency; }

    // "修改"返回新对象，原对象不变（函数式风格）
    public Money add(Money other) {
        if (!currency.equals(other.currency))
            throw new IllegalArgumentException("货币不一致");
        return new Money(amount.add(other.amount), currency);
    }
}

与其他知识点的联系

关联知识联系说明
多态 / 虚方法 final 方法走静态分派，非 final 方法走动态分派（虚方法表查找）。JVM 对 final 方法可以安全地做内联优化。
Java 内存模型（JMM） final 字段有特殊的 happens-before 规则，构造器完成即对其他线程可见，是实现安全发布（safe publication）的基础。
不可变对象 / 函数式编程 final 字段是构建不可变类的核心手段；Java 16+ 的 record 本质上就是所有字段均为 final 的 final 类。
String 的设计 String 是 final 类 + final 字段，确保字符串常量池、哈希缓存等机制的正确性，同时保证线程安全。
Lambda / effectively final Java 8 引入 effectively final，让 lambda 捕获局部变量时无需显式写 final，降低了样板代码。

常见陷阱汇总

  • final 引用 ≠ 内容不可变：final List 的元素仍然可以增删，只是引用本身不能换。
  • 常量内联的"部署陷阱"：修改 static final 常量后，调用方必须重新编译，否则用的是旧值。
  • this 逃逸破坏 final 语义：在构造器中把 this 传给另一个线程，final 的 JMM 保证失效——对方可能看到未初始化的 final 字段。
  • 反射可以绕过 final：通过反射强制修改 final 字段（setAccessible(true)）在技术上可行，但会导致未定义行为，生产代码绝对不要这样做。
  • static final 不一定是编译期常量：只有基本类型 / String + 字面量才是；其他如 new Object()、UUID.randomUUID() 是运行时常量，不会内联。

面试高频考点

  1. final 字段和 volatile 字段有什么区别？
    final 保证构造后不变，volatile 保证每次读写都直接操作主存（可见性）。两者都解决可见性，但 final 只在构造阶段生效，volatile 在整个生命周期有效。
  2. String 为什么设计成 final 类？
    保证哈希一致性（哈希值可缓存）、字符串常量池的正确性、线程安全，以及防止子类破坏 equals/hashCode 契约。
  3. final 字段为什么在多线程下安全发布不需要 synchronized？
    JMM 为 final 字段的初始化插入了写屏障（store barrier），确保构造器退出前所有 final 字段的写操作对其他线程可见。
  4. 什么是 effectively final？它的引入动机是什么？
    Java 8 起，未被重新赋值的局部变量即使不写 final 也被视为 effectively final，可被 lambda/匿名类捕获。动机是减少样板代码，同时保留"捕获值不可变"的语义。
  5. 可以用反射修改 final 字段吗？
    可以调用，但结果是未定义行为，JVM 的常量内联优化可能让其他地方仍然使用旧值。实践中不允许。
