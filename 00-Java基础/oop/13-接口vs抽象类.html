面向对象 · 抽象与接口

13. 接口 vs 抽象类的取舍

两种抽象机制各有边界——选错了会让代码越改越难受，选对了则结构天然稳固。

1. 它们分别解决什么问题？

Java 只允许单继承，这是刻意设计的——多继承会引发"菱形继承"的二义性（C++ 深受其苦）。 但现实世界里，一个对象往往需要扮演多个角色。Java 的解法是：

  • 抽象类：表达"是什么（IS-A）"——描述同一家族的共同骨架，允许有状态和实现。
  • 接口：表达"能做什么（CAN-DO）"——描述能力契约，天然支持多实现。

一句话记忆：抽象类是模板，接口是协议。 模板强调"你是我的子类"，协议强调"你同意遵守这套规范"，两者不是竞争关系，而是互补的。

2. 核心差异一览

维度 抽象类 abstract class 接口 interface
继承数量 只能继承一个 可以实现多个
实例字段 可以有任意实例字段 只能有 public static final 常量
构造器 可以定义构造器（子类 super 调用） 不能有构造器
方法实现 可以有具体方法（普通方法） Java 8+ 可有 default / static 方法
访问修饰符 方法可以是任意可见性 方法默认 public，不可降低
状态管理 可以管理共享状态（字段 + 方法） 无状态（不能持有实例数据）
语义 IS-A 关系，强调共同"身份" CAN-DO 关系，强调"能力"
典型场景 AbstractList、HttpServlet Comparable、Runnable、Serializable

3. 选抽象类的时机

当下列任一条件成立时，优先考虑抽象类：

  • 多个子类共享代码逻辑（字段 + 方法），不想每个子类都重复实现。
  • 需要管理共享状态，例如所有子类都要维护一个 id 或 createdAt。
  • 想强制子类走模板方法（Template Method）流程——固定骨架，让子类填空。
  • 子类之间存在明显的IS-A 关系，例如 Cat 和 Dog 都 IS-A Animal。

经典示例：模板方法模式

AbstractGame.java — 抽象类做骨架
// 抽象类：固定"游戏流程"的骨架，子类只需填充各步细节
public abstract class AbstractGame {

    // 共享状态：所有游戏都有玩家数量
    protected int playerCount;

    public AbstractGame(int playerCount) {
        this.playerCount = playerCount;
    }

    // 模板方法：骨架固定，不允许子类覆盖（final）
    public final void play() {
        initialize();
        while (!isDone()) {
            makeMove();
        }
        printResult();
    }

    // 抽象方法：强制子类实现，各自填入不同逻辑
    protected abstract void initialize();
    protected abstract void makeMove();
    protected abstract boolean isDone();
    protected abstract void printResult();
}
Chess.java — 子类填充细节
public class Chess extends AbstractGame {

    public Chess() {
        super(2); // 调用父类构造器，设置玩家数
    }

    @Override protected void initialize() {
        System.out.println("摆好棋盘，共 " + playerCount + " 位玩家");
    }
    @Override protected void makeMove() { System.out.println("下一步棋"); }
    @Override protected boolean isDone() { return true; }
    @Override protected void printResult() { System.out.println("将军！"); }
}
抽象类里的构造器很关键：子类必须通过 super(...) 显式初始化父类字段，这保证了共享状态在对象创建时就处于合法状态。接口做不到这一点。

4. 选接口的时机

当下列任一条件成立时，优先考虑接口：

  • 需要跨继承树的能力描述——String、Integer 都能 Comparable，但它们毫无血缘关系。
  • 希望一个类同时具备多种角色，例如一个类既是 Runnable 又是 Closeable。
  • 描述的是能力（行为契约）而非"身份"，例如"能序列化"、"能比较"。
  • 你是库作者，想让调用方自由实现，不强迫其进入某个继承树。

示例：同一个类实现多个接口

Employee.java — 同时实现多个能力接口
// Employee 既能比较（排序）、又能序列化、又能自我描述
// 如果用抽象类，同时继承三个是不可能的
public class Employee
        implements Comparable<Employee>, Serializable, Printable {

    private String name;
    private int salary;

    @Override
    public int compareTo(Employee other) {
        // 按薪资升序比较
        return Integer.compare(this.salary, other.salary);
    }

    @Override
    public void print() {
        System.out.println(name + " - " + salary);
    }
}

5. 决策黄金法则

我需要一种抽象机制 ├── 多个类有共同的状态（字段）或共享实现？ │ └── → 抽象类（IS-A，有状态的骨架） ├── 只是描述行为契约，不关心是谁？ │ └── → 接口（CAN-DO，能力标签） ├── 一个类需要多种角色（多重能力）？ │ └── → 接口（Java 不支持多继承，接口可以多实现） └── 两者都想要？ └── → 抽象类 + 接口组合（骨架 + 能力标签，如 AbstractList + List）

6. JDK 的最佳实践：骨架实现（Skeletal Implementation）

这是 《Effective Java》第 20 条的核心思想：接口定义契约，抽象类提供骨架实现，两者配合使用，获得两全其美的效果。

JDK 的 AbstractList 就是这样：List 接口定义了完整的 API 契约，AbstractList 抽象类实现了其中绝大多数方法（如 iterator()、indexOf()），子类只需实现 get(int) 和 size() 两个抽象方法，即可得到一个完整的 List。

MyImmutableList.java — 骨架实现的威力
// 只需实现 2 个方法，就能获得完整的 List 行为（含 iterator、subList、indexOf 等）
public class MyImmutableList<E> extends AbstractList<E> {

    private final Object[] data;

    public MyImmutableList(Object[] data) {
        this.data = data.clone();
    }

    @Override
    @SuppressWarnings("unchecked")
    public E get(int index) {
        return (E) data[index];
    }

    @Override
    public int size() {
        return data.length;
    }

    // AbstractList 的 add/set/remove 默认抛 UnsupportedOperationException
    // 这正是"不可变 List"期望的行为——无需额外覆盖
}
这种模式的命名惯例：接口叫 Foo，骨架实现叫 AbstractFoo。调用方可以直接面向 Foo 接口编程，内部用 AbstractFoo 省去重复代码，两全其美。

7. 常见陷阱

陷阱一：用抽象类描述能力——锁死继承槽

反例：用抽象类描述能力
// 错误示范：Flyable 只是一种"能力"，却用了抽象类
public abstract class Flyable {
    public abstract void fly();
}

// 后果：Bird 继承了 Animal，就无法再继承 Flyable
// public class Bird extends Animal, Flyable { ... }  // 编译报错！
public class Bird extends Animal implements Flyable { // 正解：Flyable 应是接口
    @Override public void fly() { System.out.println("鸟儿飞翔"); }
}

陷阱二：接口里的 default 方法与抽象类产生冲突

Java 8 引入 default 方法后，接口和抽象类的边界变模糊了，容易出现"菱形冲突"：

冲突示例与解决
interface A {
    default void hello() { System.out.println("A"); }
}
interface B extends A {
    default void hello() { System.out.println("B"); }
}

// C 同时实现 A 和 B，编译器无法决定用哪个 default ——必须手动覆盖
class C implements A, B {
    @Override
    public void hello() {
        B.super.hello(); // 显式选择 B 的实现，消除歧义
    }
}
default 方法的冲突解决规则：
1. 类的具体方法 > 接口 default 方法（类总是赢）
2. 子接口的 default 方法 > 父接口的 default 方法
3. 两个无关接口都有同名 default，必须在类里手动覆盖，否则编译报错。

陷阱三：接口里添加新方法破坏旧实现

在 Java 8 之前，向接口添加新方法会让所有实现类立即编译失败。这正是 Java 8 引入 default 方法的原因——允许向接口追加有默认实现的方法，不破坏既有代码。 但 default 方法也不是万能的：若默认实现语义上不适合某些实现类，就需要那些类手动覆盖，否则会悄悄引入 bug。

8. 与其他知识的联系

关联知识联系点
Collections.sort / Arrays.sort 依赖 Comparable 接口契约，而非某个抽象类，体现了"面向接口编程"
设计模式：模板方法 天然与抽象类绑定——骨架在父类，变化点通过抽象方法暴露给子类
设计模式：策略模式 天然与接口绑定——把算法封装为接口，运行时可以随意替换实现
Spring 依赖注入 Spring 优先按接口类型注入（@Autowired），接口是解耦的关键
函数式接口（下一节） @FunctionalInterface 限定接口只有一个抽象方法，可用 Lambda 替代匿名类
Java 内存模型 接口无实例字段，不存在字段可见性问题；抽象类有字段需注意多线程共享状态

9. 面试高频考点

Q：接口和抽象类的区别？
标准答应从四个维度答：①语法差异（字段、构造器、多继承）②语义差异（IS-A vs CAN-DO）③设计意图（骨架复用 vs 行为契约）④Java 8 后的变化（default/static 方法拉近了差距，但接口仍无状态）。
Q：Java 8 的 default 方法让接口和抽象类几乎一样了，还有区别吗？
接口仍然没有实例字段和构造器，无法管理状态；接口方法仍默认 public；接口多实现 vs 抽象类单继承的本质差异也未变。default 方法只是为了接口演化（向后兼容地加方法）而生，不是为了让接口变成抽象类。
Q：《Effective Java》第 20 条说什么？
"接口优于抽象类"——接口更灵活，可多实现；而当你需要减少实现者的工作量时，再配合一个"骨架实现"抽象类（AbstractFoo），两全其美。
Q：什么时候必须用抽象类？
需要共享实例字段，或需要构造器强制初始化状态，或需要非 public 方法（接口方法只能是 public）时，只能用抽象类。

一句话总结：接口是"我能做什么"的承诺（能力，无状态，可多实现），抽象类是"我是什么"的骨架（身份，有状态，可共享实现）。优先接口，有状态共享时才引入抽象类，两者配合形成"接口 + 骨架抽象类"是 JDK 中最成熟的模式。
