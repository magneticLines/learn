06. 访问修饰符

public / protected / default / private — 封装的四道门，决定谁能看见什么。

1. 为什么需要访问控制

面向对象的三大特性之一是封装：把数据和行为打包在一起，并 隐藏内部实现细节，只暴露必要的接口。 访问修饰符就是实现这一目标的语言机制。

设想一个银行账户类，如果 balance 字段对外可随意修改， 任何代码都可以写 account.balance = -999，业务规则形同虚设。 访问控制让编译器强制保证"只有 deposit() / withdraw() 才能改余额"。

设计原则：能收紧就收紧。字段几乎永远 private； 方法按最小可见范围选择，public 是最后的选择，而不是默认。

2. 四个修饰符一览

Java 提供四个级别，可见范围由小到大排列：

修饰符 同类内 同包 子类（不同包） 任意代码
private ✅ ❌ ❌ ❌
default（无关键字） ✅ ✅ ❌ ❌
protected ✅ ✅ ✅ ❌
public ✅ ✅ ✅ ✅
陷阱：default 并不是关键字，写 default int x; 会报错（那是 switch 的 default）。 "包级访问"就是什么都不写。

3. 可见范围示意图

private ──── 类内部
  ↓
default ──── 类内部 + 同包所有类
  ↓
protected ─── 类内部 + 同包 + 不同包的子类
  ↓
public ──── 全局可见

4. private — 最严格的封装

private 成员只在声明它的类内部可见，连同一包的其他类都无法直接访问。 这是封装字段的标准做法，配合 getter / setter 暴露受控接口。

BankAccount.java — private 字段 + getter/setter
public class BankAccount {

    // 余额对外完全不可见，必须通过方法操作
    private double balance;

    public void deposit(double amount) {
        if (amount <= 0)
            throw new IllegalArgumentException("存款金额必须为正");
        balance += amount;
    }

    public double getBalance() {
        return balance;  // 只读，不提供 setBalance
    }
}

注意：同一个类的不同实例可以互访对方的 private 成员 （因为访问控制是类级别，不是实例级别）。这在链表、克隆等场景常见：

同类不同实例互访 private
public class Node {
    private int value;

    public boolean equals(Node other) {
        // other.value 是 private，但在同类内可以直接访问
        return this.value == other.value;
    }
}

5. default（包访问）— 包是最小模块单元

不写任何修饰符时，成员对同一包内所有类可见。 Java 早期把包视为模块边界，包内部组件可以紧密协作，但不对外暴露。

典型使用场景：工具类、测试助手、包内协作的实现细节。

com/bank/Validator.java — 包级工具类
package com.bank;

// 没有 public，整个类都是 default 可见
class Validator {
    // 同包内的 BankAccount 可以直接 new Validator()
    static boolean isPositive(double v) {
        return v > 0;
    }
}
实践建议：用 default 替代 public 来隐藏实现类，是 Java 9 模块系统（JPMS）出现前常见的封装手段。 即使现在，大量框架的内部类仍是包级可见（如 Spring 内部的 BeanUtils 包级辅助类）。

6. protected — 为继承而生

protected 比 default 多了一条规则： 不同包的子类也可以访问。它专为继承体系设计， 让父类把"实现细节"开放给子类，而不暴露给无关代码。

Shape.java + Circle.java — protected 供子类覆盖
// 包 com.shape
package com.shape;
public class Shape {
    // 子类可以覆盖，外部调用者不能直接调用
    protected double computeArea() {
        return 0;
    }

    public String describe() {
        return "面积 = " + computeArea();
    }
}

// 包 com.geometry（不同包）
package com.geometry;
import com.shape.Shape;
public class Circle extends Shape {
    private double radius;

    @Override
    protected double computeArea() {
        return Math.PI * radius * radius;
    }
}
protected 的一个微妙陷阱：
子类可以访问自身继承来的 protected 成员，但不能通过父类引用访问同包之外其他对象的 protected 成员。

Circle c2 = new Circle();
c2.computeArea(); — ❌ 在 com.geometry 包外的 Circle 内，通过另一个实例访问从 Shape 继承的 protected 方法是不允许的（JLS §6.6.2）。

这条规则非常隐蔽，面试和实际编码都容易出错。记住：protected 的跨包访问只适用于"通过 this 或本类对象"的方式。

7. public — 全局接口

public 对任意包、任意类完全可见，是 API 的入口。 一旦公开，修改签名就是破坏性变更（breaking change）， 所有调用者都受影响。

顶层类若声明为 public，其所在文件名必须与类名完全一致（包括大小写），否则编译报错。

一个文件只能有一个 public 顶层类。 同文件的其他顶层类只能是 default 可见，且通常拆分到独立文件更易维护。

8. 综合示例：四种修饰符同台

Employee.java — 四种修饰符协作
package com.company.hr;

public class Employee {

    // ① private：只有本类能直接读写
    private String name;
    private double salary;

    // ② default：同包的 HR 工具类可以调用，外部无法访问
    String getDepartmentCode() {
        return "HR-001";
    }

    // ③ protected：子类（Manager 等）可覆盖计算逻辑
    protected double computeBonus() {
        return salary * 0.1;
    }

    // ④ public：对外暴露的唯一 API
    public double getTotalPay() {
        return salary + computeBonus();
    }

    public String getName() { return name; }
}

// 子类在不同包中
package com.company.executive;
import com.company.hr.Employee;

public class Manager extends Employee {

    @Override
    protected double computeBonus() {
        // 可以调用：Manager 继承了 computeBonus，通过 super 访问
        return super.computeBonus() * 2;
    }
}

9. 修饰符能用在哪些地方

目标 可用修饰符 备注
顶层类 public / default 顶层类不能是 protected / private
内部类（嵌套类） 全部四种 内部类是类的成员，规则与字段相同
字段 全部四种 推荐始终 private
方法 全部四种 接口方法默认 public，JDK 9+ 接口支持 private 方法
构造器 全部四种 private 构造器用于单例、工厂模式、工具类

10. 常见陷阱与误区

陷阱 1：子类"缩小"可见性

覆盖父类方法时，可见性只能扩大，不能缩小。 父类方法是 public，子类覆盖时不能改为 protected，否则编译报错。 原因：里氏替换原则——任何用父类引用的地方都应该能用子类替换。

反例 — 缩小可见性导致编译错误
class Animal {
    public void speak() { }
}
class Dog extends Animal {
    // ❌ 编译错误：cannot reduce the visibility of the inherited method
    @Override
    protected void speak() { }
}

陷阱 2：getter/setter 不是万能的

为每个字段都生成 getter/setter 并不等于封装。 若 setter 没有任何验证逻辑，只是赋值，那不如直接用 public 字段，至少代码更简洁。 真正的封装是：通过方法定义业务操作，字段的变化是操作的副作用，而不是被外部任意赋值。

陷阱 3：protected 不等于"跨包子类都能随意访问"

如第 6 节所述，跨包的子类只能通过自身继承来的引用访问 protected 成员， 不能通过父类类型的引用访问另一个对象的 protected 成员（JLS §6.6.2）。

陷阱 4：接口成员的默认可见性

接口中声明的方法和常量默认是 public（即使不写）。 JDK 9 开始接口支持 private 方法（供 default 方法内部复用）， 但不支持 protected。

11. 与其他知识点的联系

  • 封装原则：访问修饰符是封装的直接实现工具，而封装是 OOP 三大特性之一。
  • 继承与覆盖：protected 专为继承设计，override 必须遵守可见性只增不减的规则。
  • 设计模式：单例模式的 private 构造器、工厂方法中的包级类、模板方法中的 protected 钩子，都深度依赖访问控制。
  • Java 9 模块系统（JPMS）：模块层面的封装是更高层的访问控制，即使是 public 类，在未 exports 的包中也对外不可见。
  • 反射：Field.setAccessible(true) 可以绕过访问控制，但在 Java 9+ 的强封装模块下会抛出异常（InaccessibleObjectException）。

12. 面试高频考点

Q1：四种访问修饰符的作用范围？
按可见范围从小到大：private（类内）→ default（包内）→ protected（包内 + 子类）→ public（全局）。
Q2：protected 和 default 的区别？
protected 比 default 多了"不同包的子类可访问"。但跨包时只能通过本类型的引用访问（JLS §6.6.2）。
Q3：顶层类为什么不能是 private 或 protected？
private 类对外完全不可见，意味着无法被任何外部代码使用，没有实际意义。 protected 是为继承设计的，而顶层类的继承关系与包结构无法配合。 内部类则没有这个限制，因为内部类的上下文就是外部类。
Q4：覆盖方法时可以缩小可见性吗？
不可以。子类方法的可见性必须 >= 父类方法的可见性，否则违反里氏替换原则，编译器报错。
Q5：private 方法可以被继承和覆盖吗？
不能。private 方法对子类不可见，不存在继承。子类定义同名方法是新方法，不是覆盖（override）。 用 @Override 标注时编译器会报错，正因为如此 @Override 是好习惯。
