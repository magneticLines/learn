22. Comparable vs Comparator

Java 排序的两条路：内置自然顺序 vs 外挂灵活策略——搞清楚两者的设计哲学，才能在正确的场景选对工具。

为什么需要比较接口？

Java 集合框架（Collections.sort、TreeSet、PriorityQueue……）需要知道两个对象谁大谁小， 但 JDK 不可能提前知道你的业务对象（比如 Student、Order）应该按什么排序。 因此设计了两个"合同接口"，把比较逻辑交给调用方来实现。

这是经典的策略模式思想：把"如何比较"这个可变行为抽取出来，外置为接口，让算法（排序）与数据（对象）解耦。

Comparable：内置自然顺序

java.lang.Comparable<T> 只有一个方法：compareTo(T o)。 实现它意味着：这个类的对象天生就有一种"标准"排序方式。 JDK 内置类型如 Integer、String、LocalDate 都实现了它。

约定： compareTo 返回负数（this < o）、0（this == o）、正数（this > o）。 强烈建议与 equals 保持一致：a.compareTo(b) == 0 应当与 a.equals(b) 相同，否则在 TreeSet 等有序集合里会出现奇怪行为。
Student.java — 实现 Comparable
// 学生按学号升序排列 —— 这是"自然顺序"，写死在类里
public class Student implements Comparable<Student> {
    private final int    id;
    private final String name;
    private final double gpa;

    public Student(int id, String name, double gpa) {
        this.id = id; this.name = name; this.gpa = gpa;
    }

    @Override
    public int compareTo(Student other) {
        // 直接用 Integer.compare 避免 id1 - id2 的整数溢出陷阱
        return Integer.compare(this.id, other.id);
    }

    @Override
    public String toString() {
        return "Student{" + id + ", " + name + ", GPA=" + gpa + '}';
    }
}
使用自然顺序排序
List<Student> list = Arrays.asList(
    new Student(3, "Charlie", 3.5),
    new Student(1, "Alice",   3.9),
    new Student(2, "Bob",     3.7)
);

Collections.sort(list);  // 自动调用 compareTo，按 id 升序
// 输出：[Student{1, Alice, GPA=3.9}, Student{2, Bob, GPA=3.7}, Student{3, Charlie, GPA=3.5}]

// TreeSet 也依赖自然顺序去重和排序
TreeSet<Student> set = new TreeSet<>(list);

Comparator：外置灵活策略

java.util.Comparator<T> 是一个独立的比较器对象。它不修改被比较类，只是在外部提供一种比较规则。 这解决了 Comparable 的根本局限：一个类只能有一种自然顺序，但业务上可能需要多种排序方式。

同样是 Student，有时需要按 GPA 降序，有时按姓名字母序——这些都不应该写死到类里。

匿名类写法（Java 7 及以前常见）
Comparator<Student> byGpaDesc = new Comparator<Student>() {
    @Override
    public int compare(Student a, Student b) {
        // 注意：降序要反过来，用 b 和 a 的顺序
        return Double.compare(b.getGpa(), a.getGpa());
    }
};

list.sort(byGpaDesc);  // 按 GPA 从高到低
Lambda + Comparator 链式 API（Java 8+，推荐）
// 单字段：按姓名字母升序
Comparator<Student> byName =
    Comparator.comparing(Student::getName);

// 多字段：先按 GPA 降序，GPA 相同再按姓名升序
Comparator<Student> complex =
    Comparator.comparingDouble(Student::getGpa)
              .reversed()
              .thenComparing(Student::getName);

list.sort(complex);

// null 安全：把 null 排到最后
Comparator<Student> nullSafe =
    Comparator.nullsLast(Comparator.comparing(Student::getName));

设计哲学：为什么要分两个接口？

初学者常问：为什么不合并成一个？答案藏在职责分离原则里：

  • Comparable 表达的是"这个类本身的排序语义"——就像 equals 和 hashCode 一样，是类的内在特性。
  • Comparator 表达的是"某一特定场景下的排序策略"——它是外部视角，与类的定义无关。
  • 第三方库的类（如 java.sql.Timestamp）你无法修改，只能用 Comparator。
类比理解：人的身份证按出生日期排（自然顺序，Comparable）；但图书馆可以按书名、按出版年、按借阅量排（多种外部策略，Comparator）。

常见陷阱

陷阱 1：用减法代替 compare

新手常写 return this.id - other.id，在绝大多数情况下能工作，但存在整数溢出风险。 当 this.id = Integer.MAX_VALUE、other.id = -1 时，减法结果溢出为负数，导致排序错误。

反例 vs 正例
// ❌ 危险：整数溢出
return this.id - other.id;

// ✅ 安全：使用静态工具方法
return Integer.compare(this.id, other.id);

陷阱 2：compareTo 与 equals 不一致

BigDecimal("1.0") 和 BigDecimal("1.00") 的 equals 返回 false（精度不同），但 compareTo 返回 0。 这导致 TreeSet<BigDecimal> 认为它们是同一个元素（只保留一个），而 HashSet 认为是两个。

如果你的类进入 TreeSet / TreeMap，请确保 compareTo == 0 当且仅当 equals == true，否则行为违反 Set/Map 语义。

陷阱 3：Comparator 违反传递性

比较器必须满足三个数学性质：反对称性（cmp(a,b) == -cmp(b,a)）、传递性、一致性。 违反这些性质会导致 Arrays.sort 在 Java 7+ 中抛出 IllegalArgumentException: Comparison method violates its general contract!。

陷阱 4：reversed() 之后 thenComparing 顺序问题

链式调用的执行顺序
// 先按 GPA 升序，reversed 变为降序，再按 name 升序
Comparator.comparingDouble(Student::getGpa)
          .reversed()               // 只翻转 GPA 这一层
          .thenComparing(Student::getName);

// ❌ 常见误解：下面写法会把整个比较器翻转，name 也变成降序
Comparator.comparingDouble(Student::getGpa)
          .thenComparing(Student::getName)
          .reversed();               // 两个字段全部翻转

与集合框架的联系

排序相关 API ├── Collections.sort(list) → 要求元素实现 Comparable ├── list.sort(comparator) → 传入 Comparator，更灵活 ├── Arrays.sort(arr) → 基本类型用双轴快排；对象类型用 TimSort + Comparable ├── new TreeSet<>() → 依赖元素的 Comparable（无参构造） ├── new TreeSet<>(comparator) → 传入 Comparator 覆盖自然顺序 ├── new PriorityQueue<>(comparator) → 堆排序，需要 Comparator └── stream.sorted(comparator) → Stream 管道排序

值得注意：TreeMap / TreeSet 用比较器判断 key 相等（compareTo == 0），而不是 equals。 这与 HashMap / HashSet 用 hashCode + equals 判断相等形成鲜明对比。

实战：对第三方类排序

假设你拿到一个无法修改的第三方类 Product，需要在不同页面用不同排序——这正是 Comparator 最典型的使用场景。

对第三方 Product 类多维度排序
// 第三方类，无法修改，没有实现 Comparable
class Product {
    String name; double price; int sales;
}

// 按价格升序（商品详情页）
Comparator<Product> cheapFirst =
    Comparator.comparingDouble(p -> p.price);

// 按销量降序（热销榜）
Comparator<Product> hotFirst =
    Comparator.comparingInt((Product p) -> p.sales).reversed();

// 综合排序：先按销量降序，再按价格升序
Comparator<Product> comprehensive =
    Comparator.comparingInt((Product p) -> p.sales).reversed()
              .thenComparingDouble(p -> p.price);

products.sort(comprehensive);

面试高频考点

  • 两者区别？何时用哪个？ —— Comparable 是类的内置自然顺序，一个类只有一种；Comparator 是外置策略，可以有多种，适合不能修改源码或需要多种排序的场景。
  • compareTo 返回值的约定？ —— 负数/0/正数，分别对应小于/等于/大于；不要用减法，有整数溢出风险。
  • TreeSet 是用 equals 还是 compareTo 判断重复？ —— 用 compareTo（或 Comparator.compare），返回 0 即视为同一个元素。
  • BigDecimal 的 equals 和 compareTo 不一致，在集合里有何表现？ —— HashSet 认为 1.0 和 1.00 是两个，TreeSet 认为是同一个。
  • 如何实现多字段排序？ —— Java 8 Comparator.comparing().thenComparing() 链式调用。
  • Comparator 违反约定会怎样？ —— Java 7+ TimSort 检测到违反传递性会抛 IllegalArgumentException。

小结

选型口诀：
类有"天然"排序（如 id、日期）且你能改源码 → Comparable；
需要多种排序、或无法改源码 → Comparator；
两者可以共存：类实现 Comparable 定义默认顺序，同时在需要时传入 Comparator 覆盖。
