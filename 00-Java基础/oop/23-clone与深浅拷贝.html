23. clone() 与深浅拷贝

一次看透：为什么 Java 的复制这么难？浅拷贝、深拷贝、防御性拷贝各自适合什么场景？

1. 为什么需要「拷贝」这个概念？

在 Java 里，对象变量保存的是引用（地址），而不是对象本身。直接赋值只是复制了引用，两个变量指向堆上同一块内存：

直接赋值 ≠ 拷贝
Point a = new Point(1, 2);
Point b = a;           // b 和 a 指向同一对象！
b.x = 99;
System.out.println(a.x); // 打印 99，a 被意外修改了

这在防御性编程（getter 不暴露内部可变对象）、原型模式（基于已有对象快速创建副本）等场景中会引发难以追踪的 bug。拷贝机制正是为了隔离「原件」与「副本」的引用链而存在的。

2. Object.clone() 的机制与协议

Object.clone() 是 JVM 原生支持的拷贝操作，定义在 Object 中：

Object.java（JDK 源码摘录）
protected native Object clone() throws CloneNotSupportedException;

有三个关键点：

  • native：由 JVM 在内存层面逐字节复制对象，速度快
  • protected：外部不能直接调用，必须在子类中 override 并升级为 public
  • Cloneable 标记接口：类必须 implements Cloneable，否则调用时抛 CloneNotSupportedException——这是 Java 设计中少见的「以异常作权限控制」的反面教材
设计缺陷：Cloneable 接口本身没有任何方法，却控制着 clone() 的行为——这违反了接口即契约的原则。Josh Bloch 在《Effective Java》中直言：Cloneable 接口是一个失败的设计。

3. 浅拷贝（Shallow Copy）

调用 super.clone() 得到的默认行为就是浅拷贝：基本类型字段按值复制，引用类型字段只复制引用（地址），两个对象共享同一个子对象。

原对象 user ──▶ [ name="Alice" | scores ──▶ [90, 80, 70] ] ↗ 副本 copy ──▶ [ name="Alice" | scores ─┘ (同一个数组！)
浅拷贝示例
public class User implements Cloneable {
    private String   name;
    private int[]    scores;       // 引用类型

    @Override
    public User clone() {
        try {
            return (User) super.clone(); // 只复制引用，不复制数组本身
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(); // 永远不会发生，因为已实现 Cloneable
        }
    }
}

// 验证「共享」的危险
User original = new User("Alice", new int[]{90, 80});
User copy     = original.clone();

copy.scores[0] = 0;
System.out.println(original.scores[0]); // 0！原件被污染

String 字段看起来安全，是因为 String 是不可变类——复制引用后即便"修改"也只是让副本指向新字符串，不会影响原件。但数组、普通可变对象就不一样了。

4. 深拷贝（Deep Copy）

深拷贝要求递归复制整个对象图，副本与原件在内存上完全独立。实现方式有多种：

方式一：手动递归克隆（推荐）

手动深拷贝
@Override
public User clone() {
    try {
        User copy = (User) super.clone();
        // 对每个引用字段单独深复制
        copy.scores = scores.clone();   // 数组的 clone() 是浅拷贝，但 int[] 元素是原始类型，所以此处相当于深拷贝
        copy.address = address.clone(); // Address 也需要实现 Cloneable
        return copy;
    } catch (CloneNotSupportedException e) {
        throw new AssertionError();
    }
}

方式二：序列化（最省力，性能较差）

序列化深拷贝（所有字段须 Serializable）
public static <T extends Serializable> T deepCopy(T obj) {
    try (ByteArrayOutputStream bos = new ByteArrayOutputStream();
         ObjectOutputStream    oos = new ObjectOutputStream(bos)) {
        oos.writeObject(obj);
        try (ObjectInputStream ois = new ObjectInputStream(
                new ByteArrayInputStream(bos.toByteArray()))) {
            return (T) ois.readObject(); // 反序列化得到全新对象图
        }
    } catch (Exception e) {
        throw new RuntimeException(e);
    }
}
序列化深拷贝会走完整的 I/O 流程，比手动克隆慢 10~100 倍，不适合性能敏感路径。优点是零代码量，并能处理循环引用。

方式三：拷贝构造器（Effective Java 推荐）

拷贝构造器 —— 最清晰、最可控
public class User {
    private String  name;
    private int[]   scores;

    // 拷贝构造器：明确每个字段如何复制，一目了然
    public User(User other) {
        this.name   = other.name;
        this.scores = Arrays.copyOf(other.scores, other.scores.length);
    }
}

// 使用方式清晰语义明确
User copy = new User(original);

6. 浅拷贝的合法使用场景

浅拷贝并非一无是处。只要满足以下任一条件，浅拷贝就是安全的：

  • 所有引用字段都是不可变对象（如 String、Integer、LocalDate），共享引用不会被外部修改
  • 字段本身是原始类型数组，且元素是原始类型：此时数组的 .clone() 等效于深拷贝
  • 对象是值对象（Value Object），本身设计为不可变，复制只是复用

例如 java.util.Date 在做防御性拷贝时通常用浅拷贝就够了（因为它只需隔离引用本身）：

防御性拷贝（Defensive Copy）
public class Period {
    private final Date start;
    private final Date end;

    public Period(Date start, Date end) {
        // 构造时立刻拷贝，防止调用方持有引用后修改
        this.start = new Date(start.getTime());
        this.end   = new Date(end.getTime());
    }

    public Date getStart() {
        return new Date(start.getTime()); // getter 也要防御，否则仍可泄漏
    }
}

7. 与其他知识点的联系

与原型模式（Prototype Pattern）

GoF 23 种设计模式中的原型模式就是以 clone() 为基础的——当创建对象成本高（如复杂初始化、DB 查询）时，直接克隆已有对象比 new 更高效。Spring 的 prototype 作用域 Bean 在内部也借助了类似思路。

与不可变类（Immutable Class）

如果类是不可变的（所有字段 final 且引用字段指向不可变对象），根本不需要深拷贝——共享引用是安全的。String、BigDecimal 都是此类。设计时优先考虑不可变，再考虑拷贝。

与 Java 内存模型

Object.clone() 在底层通过 System.arraycopy 级别的内存块复制完成，逐字节复制对象头和字段区域，因此不触发任何构造器——这意味着构造器里的校验逻辑会被绕过，是一个安全隐患。

与 ArrayList 等集合类

ArrayList.clone() 返回的是浅拷贝：新列表容器独立，但元素引用共享。修改新列表的元素仍会影响原列表中的对象，这是初学者非常容易踩的坑。

ArrayList.clone() 的浅拷贝陷阱
List<int[]> origin = new ArrayList<>();
origin.add(new int[]{1, 2});

List<int[]> copy = new ArrayList<>(origin); // 浅拷贝
copy.get(0)[0] = 999;
System.out.println(origin.get(0)[0]); // 999，原列表元素被改了！

8. 常见陷阱汇总

陷阱 1：以为浅拷贝等于独立副本
只要有引用类型字段，浅拷贝的副本就会共享子对象。修改副本的子对象会影响原件。
陷阱 2：继承体系中漏掉父类字段
子类重写 clone() 时必须调用 super.clone()，否则父类字段不会被复制，且返回类型不对。
陷阱 3：clone() 绕过构造器
clone() 不走构造器，构造器中的参数校验、初始化逻辑会被跳过，可能产生非法状态对象。
陷阱 4：循环引用导致 StackOverflow
手动递归克隆时，若对象图存在循环引用（A 引用 B，B 引用 A），会无限递归。序列化方式能自动处理此情形。
最佳实践：优先设计不可变类 → 其次用拷贝构造器 → 必要时才用 clone()，并在注释中说明是深拷贝还是浅拷贝。

9. 面试高频考点

问题核心答案
浅拷贝与深拷贝的区别？ 浅拷贝只复制引用，深拷贝递归复制对象图，副本与原件完全独立
为什么要实现 Cloneable？ 标记接口，JVM 以此判断是否允许 clone()，否则抛 CloneNotSupportedException
String 字段需要深拷贝吗？ 不需要，String 不可变，共享引用安全
ArrayList.clone() 是深拷贝吗？ 不是，容器独立但元素共享（浅拷贝）
《Effective Java》推荐用什么替代 clone()？ 拷贝构造器（Copy Constructor）或拷贝工厂方法（Copy Factory）
clone() 会调用构造器吗？ 不会，直接内存复制，构造器逻辑被绕过
