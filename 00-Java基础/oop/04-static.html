04. static 字段与方法

理解"属于类"而非"属于对象"的设计哲学，掌握静态成员的内存模型、使用场景与常见陷阱。

1. 为什么需要 static？

在面向对象世界里，每个对象都拥有自己独立的一份实例字段。这在大多数情况下是正确的——每只猫都有自己的名字和年龄。但有些数据天然就属于"这个类全体"，而不是某一个具体对象，例如：

  • 全局计数器：已创建了多少个 User 对象？
  • 共享常量：圆周率 π 不应该每个 Circle 对象各存一份。
  • 工具方法：Math.abs() 不依赖任何 Math 实例，调用前没必要先 new Math()。

Java 用 static 关键字来表达这层语义：该成员归属于类本身，而非类的某个实例。

2. 内存模型：方法区 vs 堆

理解 static 的关键是理解它在内存里住在哪里。JVM 把内存分为几个区域：

JVM 内存结构（简化）
├── 方法区（Method Area / Metaspace）
│ ├── 类的元信息（字段描述、方法字节码…）
│ └── static 字段的值 ← 就住在这里
├── 堆（Heap）
│ └── 每次 new 出来的对象实例（含实例字段）
└── 栈（Stack）
└── 方法调用帧、局部变量

静态字段随着类加载（Class Loading）时一起初始化，存放在方法区（Java 8+ 叫 Metaspace），整个 JVM 生命周期内只有一份，所有实例共享。

Java 8 之前静态字段存放在 PermGen（永久代），Java 8 起迁移到 Metaspace（本地内存），但"唯一一份、类加载时存在"的语义没有变化。

3. static 字段（静态变量）

3.1 基本用法

Counter.javastatic field 示例
public class Counter {

    // 静态字段：属于类，所有实例共享同一个 count
    private static int count = 0;

    // 实例字段：每个对象各有自己的 id
    private final int id;

    public Counter() {
        // 每次创建对象，共享计数器 +1
        count++;
        this.id = count;
    }

    public static int getCount() {
        return count;
    }

    public int getId() {
        return id;
    }
}
Main.java调用验证
Counter a = new Counter();
Counter b = new Counter();
Counter c = new Counter();

System.out.println(Counter.getCount());  // 3 — 类级别的计数
System.out.println(a.getId());           // 1 — 对象自己的 id
System.out.println(c.getId());           // 3

3.2 静态常量（static final）

最常见的 static 使用场景是定义类级别的常量，配合 final 防止被修改：

MathConst.java静态常量
public class MathConst {
    // 惯例：静态常量全大写+下划线
    public static final double PI    = 3.141592653589793;
    public static final double E     = 2.718281828459045;
    public static final int    MAX_RETRY = 3;
}

// 使用时直接用类名访问，清晰且不需要创建对象
double area = MathConst.PI * r * r;
JDK 源码中大量使用静态常量：Integer.MAX_VALUE、Math.PI、HttpURLConnection.HTTP_OK。编译器会对 static final 基本类型/String 做常量折叠（constant folding），在调用处直接内联数值，几乎零开销。

4. static 方法（静态方法）

4.1 工厂方法模式

静态方法不依赖对象状态，天然适合作为工厂方法——提供比构造器更具语义的创建入口：

LocalDate 风格的工厂方法类比 JDK
public class Color {
    private final int r, g, b;

    // 构造器设为 private，强制外部使用工厂方法
    private Color(int r, int g, int b) {
        this.r = r; this.g = g; this.b = b;
    }

    // 工厂方法语义清晰：from RGB、from 十六进制
    public static Color ofRgb(int r, int g, int b) {
        return new Color(r, g, b);
    }

    public static Color ofHex(String hex) {
        int val = Integer.parseInt(hex.replace("#", ""), 16);
        return new Color((val >> 16) & 0xFF, (val >> 8) & 0xFF, val & 0xFF);
    }
}

// 调用侧——意图一眼明了
Color orange = Color.ofHex("#fb923c");

4.2 工具类（Utility Class）

工具类的所有方法都是静态的，不维护任何状态。标准做法是把构造器私有化防止误用：

StringUtils.java工具类标准写法
public final class StringUtils {

    // 私有构造器 —— 防止 new StringUtils()，明确"此类不可实例化"
    private StringUtils() {
        throw new UnsupportedOperationException("utility class");
    }

    public static boolean isBlank(String s) {
        return s == null || s.isBlank();
    }

    public static String capitalize(String s) {
        if (isBlank(s)) return s;
        return Character.toUpperCase(s.charAt(0)) + s.substring(1);
    }
}

5. 核心限制：静态上下文没有 this

静态方法在没有任何对象存在的情况下就可以被调用，因此它根本不知道 this 是谁。这带来一条铁律：

  • 静态方法不能直接访问实例字段（非 static 字段）
  • 静态方法不能直接调用实例方法（非 static 方法）
  • 静态方法不能使用 this 或 super
反例 → 问题 → 正例静态方法访问实例成员
public class Bad {
    private int value = 42;

    // ❌ 编译错误：静态方法里访问了实例字段
    public static void printValue() {
        System.out.println(value);  // Cannot make a static reference to the non-static field
    }
}

public class Good {
    private int value = 42;

    // ✅ 通过参数传入对象，明确"我要访问哪个实例"
    public static void printValue(Good obj) {
        System.out.println(obj.value);
    }
}
反过来，实例方法可以访问静态成员，因为静态成员永远存在、不依赖对象。但要注意：通过对象引用访问静态成员（obj.staticField）在 Java 中合法，但会被 IDE 和编译器警告——它会产生误导，应始终用类名访问 ClassName.staticField。

6. static 与继承：隐藏，不是覆盖

实例方法支持多态覆盖（Override），而静态方法不行。子类可以定义同名静态方法，但这叫做方法隐藏（Method Hiding），调用哪个版本取决于引用的声明类型，而非运行时类型：

StaticHiding.javastatic 方法隐藏 vs 实例方法覆盖
class Animal {
    public static String type() { return "Animal"; }
    public        String sound() { return "..."; }
}

class Dog extends Animal {
    public static String type() { return "Dog"; }    // 隐藏，非覆盖
    @Override
    public        String sound() { return "Woof"; }   // 覆盖，有多态
}

Animal ref = new Dog();

// 静态方法：编译期绑定，看声明类型 Animal
System.out.println(Animal.type());   // Animal
System.out.println(Dog.type());      // Dog

// 实例方法：运行时绑定，看对象的真实类型 Dog
System.out.println(ref.sound());    // Woof  ← 多态生效
这是面试高频考点！静态方法没有运行时多态，不能在接口/抽象类里用 @Override 覆盖静态方法。Java 8 起接口可以有 static 方法，但同样不能被实现类"覆盖"（只是隐藏）。

7. 常见陷阱

陷阱 1：通过对象引用（甚至 null）调用静态方法

NullStatic.java令人困惑的合法代码
Counter obj = null;
// ⚠️ 不会 NullPointerException！静态方法与对象无关
System.out.println(obj.getCount());  // 合法，但强烈不建议

// ✅ 应该这样写，语义清晰
System.out.println(Counter.getCount());

陷阱 2：静态字段的多线程安全

静态字段是所有线程共享的，在多线程环境中如果多个线程同时修改同一个静态字段，会产生竞态条件（Race Condition）：

ThreadUnsafe.java反例与修正
// ❌ 反例：多线程下 count++ 不是原子操作（读-改-写），会出现计数丢失
public class UnsafeCounter {
    private static int count = 0;
    public static void increment() { count++; }  // 线程不安全
}

// ✅ 正例：使用 AtomicInteger 保证原子性
import java.util.concurrent.atomic.AtomicInteger;

public class SafeCounter {
    private static final AtomicInteger count = new AtomicInteger(0);
    public static void increment() { count.incrementAndGet(); }
    public static int getCount() { return count.get(); }
}

陷阱 3：静态字段会被子类"共享"

StaticInherit.java静态字段不属于子类
class Base {
    static int x = 10;
}
class Child extends Base {}

Child.x = 99;  // 实际上修改的是 Base.x，不是 Child 独有的副本
System.out.println(Base.x);   // 99 ← 被改了！

8. 与 Spring 的联系：为什么不用 static 注入？

Spring 的依赖注入（DI）是基于对象实例的——Spring 管理 Bean 的生命周期，通过构造器或 setter 注入依赖。静态字段在类加载时就存在，此时 Spring 容器可能尚未初始化完毕：

SpringWrong.java反例：尝试用 @Autowired 注入 static 字段
@Component
public class MyService {

    // ❌ @Autowired 加在 static 字段上不会生效（Spring 警告）
    @Autowired
    private static UserRepository repo;

    // ✅ 如果工具类必须用 Spring Bean，用 setter 注入后赋给 static 字段
    private static UserRepository repoStatic;

    @Autowired
    public void setRepo(UserRepository r) {
        MyService.repoStatic = r;
    }
}

9. 对比总结

特性 实例成员 静态成员（static）
归属 每个对象各一份 类本身，所有实例共享
内存位置 堆（Heap），随对象创建 方法区，随类加载存在
生命周期 对象被 GC 回收后消失 类被卸载前一直存在
访问方式 必须通过对象引用 推荐通过类名，也可对象引用
能否用 this 可以 不可以
多态 支持（运行时绑定） 不支持（编译期绑定，只有隐藏）
典型场景 对象的状态与行为 工具方法、工厂方法、全局常量、计数器

10. 面试高频考点

  • static 方法能不能被重写（Override）？——不能，只能被隐藏（Hiding）。@Override 注解加上去会编译报错。
  • static 方法中为什么不能用 this？——静态方法可以在没有任何对象时调用，this 指向当前对象实例，逻辑上不存在。
  • 静态变量和实例变量的区别？——归属、内存位置、生命周期三个维度，见上表。
  • static 修饰内部类有什么含义？——静态内部类（static nested class）不持有外部类实例的引用，可独立实例化；非静态内部类必须先有外部类实例。
  • 单例模式为什么用 static？——单例的唯一实例通常保存在静态字段中，确保全局唯一。双重检查锁定（DCL）中还需配合 volatile 防止指令重排序。
  • 接口里的 static 方法（Java 8+）有什么特点？——接口静态方法必须通过接口名调用，不能通过实现类名或对象调用，且不被继承。
