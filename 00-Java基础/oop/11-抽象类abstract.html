面向对象 · 抽象与接口

11. 抽象类 abstract

用"不完整的类"强制规范子类行为——理解抽象类的设计动机、语法约束、与接口的本质区别，以及它在模板方法模式中的核心地位。

为什么需要抽象类

先看一个问题：我们已经有了继承，父类可以定义通用行为，子类可以覆盖——这已经够用了，为什么还需要 abstract？

答案在于约束力。普通继承是"建议"，抽象类是"合同"。

考虑一个图形库，需要计算不同形状的面积：

反例：普通父类，无法强制子类实现
class Shape {
    public double area() {
        return 0;  // 返回 0？这根本没有意义
    }
}

class Circle extends Shape {
    // 开发者忘记重写 area()，编译器不报错，运行时静默返回 0
    // 这是一个隐藏的 bug！
}
正例：抽象类，编译期强制子类实现
abstract class Shape {
    public abstract double area();  // 无方法体，子类必须实现
}

class Circle extends Shape {
    // 不实现 area() → 编译报错
    // 错误: Circle is not abstract and does not override abstract method area()
}
设计动机总结：抽象类用来表达"这个概念存在，但它本身没有具体实例"。现实中 Shape（形状）是个概念，你画出来的永远是圆、矩形，不存在一个叫"形状"的具体图形。抽象类正是把这种"概念层"映射到代码里。

语法与核心规则

Shape.java — 抽象类完整结构
public abstract class Shape {

    // ① 普通字段：子类可以继承
    private String color;

    // ② 构造器：抽象类有构造器，但只能被子类 super() 调用
    public Shape(String color) {
        this.color = color;
    }

    // ③ 抽象方法：没有方法体，子类必须实现
    public abstract double area();
    public abstract double perimeter();

    // ④ 普通方法：有方法体，子类可直接使用或覆盖
    public String getColor() {
        return color;
    }

    // ⑤ 模板方法：固定算法骨架，细节留给子类
    public final void describe() {
        System.out.println("颜色: " + color
            + ", 面积: " + area()       // 调用子类实现
            + ", 周长: " + perimeter());
    }
}
Circle.java — 具体子类
public class Circle extends Shape {

    private final double radius;

    public Circle(String color, double radius) {
        super(color);  // 必须调用父类构造器
        this.radius = radius;
    }

    @Override
    public double area() {
        return Math.PI * radius * radius;
    }

    @Override
    public double perimeter() {
        return 2 * Math.PI * radius;
    }
}

六条硬性规则

规则 说明
抽象类不能被实例化 new Shape() → 编译报错，即使它没有任何抽象方法
抽象方法没有方法体 只有签名，以分号结尾；有方法体则编译报错
子类必须实现所有抽象方法 否则子类自身也必须声明为 abstract
抽象类可以有构造器 供子类 super() 调用，不能直接 new
abstract 与 final 不能共存 final 禁止继承，abstract 要求继承，矛盾
abstract 与 static （方法上）不能共存 静态方法属于类，无多态，抽象没意义

模板方法模式 — 抽象类最经典的用法

抽象类天然适合实现模板方法模式（Template Method Pattern）：父类定义算法的骨架（步骤顺序固定），将可变的步骤声明为抽象方法，由子类填充细节。

数据导出框架 — 模板方法
public abstract class DataExporter {

    // 模板方法：final 防止子类打乱流程
    public final void export(String query) {
        List<Object> data = fetchData(query);   // 步骤1: 查数据（可变）
        String content = format(data);          // 步骤2: 格式化（可变）
        writeToFile(content);                   // 步骤3: 写文件（固定）
    }

    // 抽象步骤：子类决定如何查数据
    protected abstract List<Object> fetchData(String query);

    // 抽象步骤：子类决定导出格式（CSV / JSON / Excel）
    protected abstract String format(List<Object> data);

    // 固定步骤：所有子类共用
    private void writeToFile(String content) {
        System.out.println("[写入文件] " + content.length() + " bytes");
    }
}

// CSV 子类只需关注自己的职责
public class CsvExporter extends DataExporter {
    @Override
    protected List<Object> fetchData(String query) { /* 查 DB */ return List.of(); }
    @Override
    protected String format(List<Object> data) { return "a,b,c\n1,2,3"; }
}
JDK 中的真实例子：java.io.InputStream 就是这样设计的。它把 read(byte[], int, int) 实现为模板（循环调用单字节读取），只把 read() 声明为抽象方法。FileInputStream、ByteArrayInputStream 各自实现单字节读取，框架层面的逻辑完全复用。

抽象类 vs 接口 — 最重要的对比

这是面试必考题，也是很多初学者混淆的地方。记住一句话：抽象类是"is-a"（是一种），接口是"can-do"（能做什么）。

维度 抽象类 接口
关键字 abstract class interface
继承数量 只能继承一个 可以实现多个
字段 可以有实例字段、状态 只有 public static final 常量
构造器 有构造器 无构造器
方法访问修饰符 任意（private/protected/public） 默认 public（Java 9+ 可有 private）
代码复用 强——可共享字段和方法实现 弱——主要靠 default 方法（Java 8+）
适用场景 有共同状态和行为的同类事物 跨类型的能力约定（Serializable、Runnable）
选择示例 — 何时用抽象类，何时用接口
// 抽象类：鸟类共享"有翅膀、会呼吸"的状态和行为
abstract class Bird {
    protected String name;
    protected int wingSpan;
    public abstract void makeSound();
    public void breathe() { System.out.println("呼吸..."); }
}

// 接口：会飞是一种"能力"，鸟会飞，飞机也会飞，没有共同父类
interface Flyable {
    void fly();
}

// Eagle 继承鸟的本质，同时具备飞行能力
class Eagle extends Bird implements Flyable {
    @Override public void makeSound() { System.out.println("嘎——"); }
    @Override public void fly() { System.out.println("展翅翱翔"); }
}

// Penguin 是鸟但不会飞：不实现 Flyable，编译期就能保证正确性
class Penguin extends Bird {
    @Override public void makeSound() { System.out.println("嗯嗯"); }
}

抽象类与多态

抽象类引用变量可以指向任意子类对象，这正是多态的核心：用父类类型统一操作不同子类对象，调用时自动分派到子类的具体实现。

多态调用
Shape[] shapes = {
    new Circle("红色", 5.0),
    new Rectangle("蓝色", 4.0, 6.0),
};

// 统一遍历，运行时动态分派 area() 到对应子类实现
for (Shape s : shapes) {
    s.describe();
}
// 输出：
// 颜色: 红色, 面积: 78.53..., 周长: 31.41...
// 颜色: 蓝色, 面积: 24.0, 周长: 20.0
抽象类引用不能 new，但可以作为变量类型：
Shape s = new Shape() → 报错
Shape s = new Circle(...) → 合法，这是多态赋值

抽象类的继承链

抽象类可以继承另一个抽象类，且不必实现父类的所有抽象方法——未实现的抽象方法会继续传递给下一层具体类。

AbstractAnimal（抽象） ← breathe() 抽象
    └── AbstractMammal（抽象）← breathe() 实现：肺呼吸；move() 抽象
        └── Dog（具体）← 必须实现 move()
多层抽象继承
abstract class AbstractAnimal {
    public abstract void breathe();
}

// 中间层抽象类：实现了部分方法，新增抽象方法
abstract class AbstractMammal extends AbstractAnimal {
    @Override
    public void breathe() {
        System.out.println("用肺呼吸");  // 实现了父类的抽象方法
    }
    public abstract void move();   // 新增抽象方法，继续往下传递
}

// 具体类：必须实现所有未被实现的抽象方法
class Dog extends AbstractMammal {
    @Override
    public void move() {
        System.out.println("四足奔跑");
    }
    // breathe() 已被 AbstractMammal 实现，无需再次覆盖
}

JDK 中的抽象类 — 学设计思路

抽象类 关键抽象方法 设计意图
InputStream read() 只要会读一个字节，缓冲读取框架全部复用
AbstractList get(int)、size() iterator()、contains() 等几十个方法全部基于这两个抽象方法实现
AbstractQueuedSynchronizer (AQS) tryAcquire()、tryRelease() ReentrantLock、Semaphore、CountDownLatch 的底层框架全来自这里
HttpServlet doGet()、doPost() Servlet 请求分发的模板方法，我们只需重写对应 HTTP 方法

常见陷阱与易错点

陷阱 1：抽象类中调用抽象方法，this 指向子类

构造器中调用抽象方法 — 危险！
abstract class Base {
    public Base() {
        init();  // 危险！此时子类字段还未初始化
    }
    protected abstract void init();
}

class Child extends Base {
    private String name = "hello";
    @Override
    protected void init() {
        // name 此时是 null！父类构造器先于子类字段初始化执行
        System.out.println(name.length());  // NullPointerException！
    }
}
规则：永远不要在构造器中调用可被子类覆盖的方法（抽象方法或非 final 方法）。这是 Effective Java 第 19 条明确禁止的做法。

陷阱 2：忘记在子类标注 @Override

如果方法签名打错了（如把 area() 写成 Area()），没有 @Override 时编译器认为你定义了一个新方法，子类依然是抽象类（如果还有未实现的抽象方法）或者悄悄跑了错误版本。始终加 @Override。

陷阱 3：把抽象类当接口用（滥用继承）

如果一个"抽象类"没有任何字段、没有任何非抽象方法，只有抽象方法——它应该是接口，不是抽象类。用抽象类会白白占用那唯一的继承名额。

陷阱 4：匿名子类实例化抽象类

合法的匿名子类写法
// 抽象类不能 new，但可以通过匿名类语法创建一次性子类
Shape s = new Shape("绿色") {  // 匿名子类，不是直接 new Shape
    @Override public double area() { return 0; }
    @Override public double perimeter() { return 0; }
};
// 常见于测试代码或一次性简单实现，生产代码中谨慎使用

面试高频考点

  • 抽象类和接口的区别？ — 从字段、构造器、多继承、设计语义（is-a vs can-do）几个维度回答，再结合 Java 8 default 方法讲接口的演进。
  • 抽象类可以没有抽象方法吗？ — 可以。只要加了 abstract 关键字就无法被实例化。常见用途：防止某个类被直接实例化，但不强制子类实现特定方法。
  • 抽象类可以实现接口吗？ — 可以，且不必实现接口的所有方法，未实现的方法自动成为抽象方法留给具体子类实现。
  • AQS 为什么用抽象类而不是接口？ — 因为 AQS 需要维护大量状态（state 字段、CLH 队列）和提供大量已实现的公共方法（acquire、release 等），接口无法持有状态，只有抽象类能胜任。
  • 构造器中调用抽象方法为什么危险？ — 见上方陷阱 1，子类字段初始化晚于父类构造器执行，会导致 NPE 或状态错误。
记忆口诀：抽象类 = 半成品模板；接口 = 能力合同。有"家族共同状态"选抽象类；要"跨族能力组合"选接口；两者不互斥，一个类可以同时继承抽象类并实现多个接口。
