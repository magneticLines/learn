面向对象 · 抽象与接口

12. 接口 interface

从"纯规范"到"带默认实现"——理解接口的设计哲学，掌握普通方法、default 方法、static 方法三种成员的用法与边界。

1. 为什么需要接口？

Java 只允许单继承，但现实中一个类往往要扮演多种角色。比如一只鸭子既会游泳又会飞——如果把"游泳"和"飞行"都塞进父类，就会污染不会飞的子类（企鹅继承了 fly()，岂不荒谬？）。

接口解决的核心问题：将"能力声明"与"具体实现"彻底分离。接口只说"你必须能做 X"，至于怎么做，由实现类自己决定。这样一个类可以同时实现多个接口，拥有多种"能力"。

面向对象有一条黄金原则：面向接口编程，而非面向实现编程（Program to an interface, not an implementation）。接口让调用方与具体实现解耦，这是 Spring 依赖注入、策略模式、工厂模式的根基。

2. 接口的本质：一份"合同"

可以把接口理解成一份法律合同：合同条款（抽象方法）规定了"必须履行的义务"，签了合同（implements）的类必须全部兑现，否则只能声明为抽象类。

接口有几个关键特征：

  • 接口本身不能被实例化（不能 new 一个合同）
  • 接口中的字段默认是 public static final（常量）
  • 接口中的普通方法默认是 public abstract（Java 8 前唯一选项）
  • Java 8 起支持 default 方法和 static 方法
  • Java 9 起还支持 private 方法（供 default/static 复用逻辑）
接口的完整结构速览Java
public interface Animal {

    // ① 常量（public static final 可省略）
    String KINGDOM = "Animalia";

    // ② 抽象方法（public abstract 可省略）
    void breathe();
    String sound();

    // ③ default 方法（Java 8+）：有默认实现，子类可选择覆盖
    default void introduce() {
        System.out.println("I am an animal. I say: " + sound());
    }

    // ④ static 方法（Java 8+）：属于接口本身，不继承到实现类
    static Animal create(String type) {
        if ("dog".equals(type)) return new Dog();
        throw new IllegalArgumentException("Unknown: " + type);
    }
}

3. 普通抽象方法：最严格的"合同条款"

接口里最经典的成员——抽象方法，没有方法体，实现类必须全部实现。

这是接口存在的根本意义：强制约定行为规范。

定义接口 + 实现类Java
// 接口：只管"能做什么"，不管"怎么做"
public interface Flyable {
    void fly();           // 编译器会自动加 public abstract
    int maxAltitude();   // 每种飞行物的极限高度不同
}

public interface Swimmable {
    void swim();
}

// 鸭子同时实现两个接口 —— 多继承能力
public class Duck implements Flyable, Swimmable {

    @Override
    public void fly() {
        System.out.println("Duck flaps its wings");
    }

    @Override
    public int maxAltitude() {
        return 500;  // 单位：米
    }

    @Override
    public void swim() {
        System.out.println("Duck paddles in water");
    }
}

// 调用方只依赖接口，不关心是 Duck 还是 Eagle
public class Main {
    static void letItFly(Flyable f) {
        f.fly();
        System.out.println("Max altitude: " + f.maxAltitude());
    }

    public static void main(String[] args) {
        letItFly(new Duck());  // 多态：传什么类型就用什么实现
    }
}
多接口 vs 多继承：Java 禁止多继承类（避免菱形问题），但允许实现多个接口。因为接口的抽象方法没有实现体，不存在"两个父类都有实现、子类不知道选哪个"的冲突。

4. default 方法：接口的"向后兼容"神器

Java 8 之前，接口一旦发布就无法新增方法——否则所有实现类都会编译报错。为了解决这个接口演进难题，Java 8 引入了 default 方法。

4.1 设计动机：List.sort() 的故事

Java 8 要给 List 接口加 sort() 方法，但全球有无数 List 实现类。若直接加抽象方法，所有人的代码都炸了。于是用 default：

JDK 源码简化版：List.sort()Java
// java.util.List（JDK 8 简化）
public interface List<E> {

    // 原有方法不变，新增 default 方法
    default void sort(Comparator<? super E> c) {
        Object[] a = this.toArray();
        Arrays.sort(a, (Comparator) c);
        // ... 回写排序结果
    }
}

// 老代码（已存在的 MyList）不用改，直接继承了默认实现
public class MyList<E> implements List<E> {
    // 无需实现 sort()，使用接口的默认实现即可
}

4.2 实战示例：给接口加日志能力

default 方法复用逻辑Java
public interface DataProcessor {

    // 抽象方法：子类必须实现核心逻辑
    String process(String data);

    // default 方法：封装"处理前后打日志"的通用流程
    // 子类只需关心 process()，日志逻辑无需重复写
    default String processWithLog(String data) {
        System.out.println("[START] input=" + data);
        String result = process(data);  // 调用抽象方法（多态）
        System.out.println("[END]   output=" + result);
        return result;
    }
}

public class UpperCaseProcessor implements DataProcessor {
    @Override
    public String process(String data) {
        return data.toUpperCase();
    }
    // processWithLog 无需重写，直接继承
}

public class TrimProcessor implements DataProcessor {
    @Override
    public String process(String data) {
        return data.trim();
    }
    // 若想覆盖，直接 @Override processWithLog 即可
}

4.3 菱形冲突：两个接口都有同名 default 方法

这是 default 方法引入的新问题。当一个类实现两个接口，且它们有相同签名的 default 方法时，编译器会报错，强制要求实现类手动解决冲突。

菱形冲突解决方案Java
interface A {
    default void hello() { System.out.println("Hello from A"); }
}
interface B {
    default void hello() { System.out.println("Hello from B"); }
}

// ❌ 编译报错：class C inherits unrelated defaults for hello()
// ✅ 必须手动 override，明确选择哪个
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // 通过"接口名.super.方法名"调用指定接口的默认实现
    }
}
陷阱：default 方法 vs 类继承的优先级。若实现类的父类有同名方法，父类方法优先于接口 default 方法。规则：类 > 接口 default > 接口 default 继承链。这是 Java 设计的"类永远赢"原则，避免引入 default 后破坏已有类继承体系。

5. static 方法：属于接口本身的工具方法

Java 8 之前，和接口配套的工具方法只能放在单独的工具类里（如 Collections 配合 Collection，Arrays 配合 Array）。Java 8 起可以直接放在接口内部，让代码更内聚。

5.1 接口 static 方法 vs 工具类

旧写法 vs 新写法对比Java
// ===== 旧写法：工具类（Java 8 之前的惯例）=====
public class Validators {
    private Validators() {}  // 防止实例化
    public static boolean isEmail(String s) { ... }
}

// ===== 新写法：放进接口（Java 8+）=====
public interface Validator<T> {

    boolean validate(T value);

    // 工厂方法：创建常用验证器实例
    static Validator<String> nonEmpty() {
        return s -> s != null && !s.isEmpty();
    }

    static Validator<String> email() {
        return s -> s != null && s.contains("@");
    }

    // default 方法：组合两个验证器（and 逻辑）
    default Validator<T> and(Validator<T> other) {
        return value -> this.validate(value) && other.validate(value);
    }
}

// 使用：Validator.email() 而不是 Validators.isEmail()
Validator<String> check = Validator.nonEmpty().and(Validator.email());
check.validate("hello@world.com");  // true
关键区别：接口的 static 方法不会被实现类继承，只能通过 接口名.方法名() 调用，不能通过实现类或实例调用。这和抽象类的 static 方法不同——抽象类的 static 方法可以被子类继承（隐藏）。

7. 函数式接口：接口与 Lambda 的结合

只含一个抽象方法的接口称为函数式接口（Functional Interface）。Java 8 引入 @FunctionalInterface 注解标记它，从而可以用 Lambda 表达式实例化。

函数式接口 + LambdaJava
@FunctionalInterface   // 编译器会验证：只能有 1 个抽象方法
public interface Transformer<T> {
    T transform(T input);

    // default 和 static 方法不影响函数式接口资格
    default Transformer<T> andThen(Transformer<T> next) {
        return input -> next.transform(this.transform(input));
    }
}

// 用 Lambda 直接实例化接口（本质是匿名类的语法糖）
Transformer<String> trim      = String::trim;
Transformer<String> upper     = String::toUpperCase;
Transformer<String> pipeline  = trim.andThen(upper);

pipeline.transform("  hello  ");  // "HELLO"

JDK 内置了大量函数式接口，全在 java.util.function 包：

java.util.function ├── Function<T,R> — T → R （映射转换） ├── Predicate<T> — T → boolean （条件判断） ├── Consumer<T> — T → void （消费/副作用） ├── Supplier<T> — () → T （生产/工厂） ├── BiFunction<T,U,R> — (T,U) → R （双参数映射） └── UnaryOperator<T> — T → T （同类型转换）

8. 接口继承接口（extends）

接口可以用 extends 继承一个或多个接口，形成接口继承体系。这在设计分层 API 时非常常见。

接口继承接口Java
interface Readable {
    String read();
}
interface Writable {
    void write(String data);
}

// ReadWrite 同时继承两个接口 —— 接口可以多继承
interface ReadWrite extends Readable, Writable {
    void flush();  // 新增方法
}

// 实现类需实现 read() + write() + flush() 三个方法
class Buffer implements ReadWrite {
    private StringBuilder buf = new StringBuilder();

    @Override public String read() { return buf.toString(); }
    @Override public void write(String d) { buf.append(d); }
    @Override public void flush() { buf.setLength(0); }
}

9. 常见陷阱

陷阱一：接口字段是常量，不是变量

接口字段的隐式修饰符Java
interface Config {
    int MAX_RETRY = 3;  // 编译器自动加 public static final
}

class Service implements Config {
    void test() {
        // MAX_RETRY = 5;  ❌ 编译错误：final 常量不可修改
        System.out.println(MAX_RETRY);  // ✅ 可读取
    }
}

陷阱二：static 方法不能通过实现类调用

static 方法调用方式Java
interface MyInterface {
    static void hello() { System.out.println("hello"); }
}
class Impl implements MyInterface {}

// ✅ 正确：通过接口名调用
MyInterface.hello();

// ❌ 编译错误（不同于抽象类 static 方法）
// Impl.hello();       — 接口 static 方法不继承到实现类
// new Impl().hello(); — 也不行

陷阱三：误用接口作常量集合（反模式）

常量接口反模式Java
// ❌ 反模式：用接口存常量（"Constant Interface Anti-Pattern"）
// 《Effective Java》Item 22 明确反对此做法
interface HttpStatus {
    int OK        = 200;
    int NOT_FOUND  = 404;
}

// ✅ 正确做法：用 enum 或 final class
public enum HttpStatus {
    OK(200), NOT_FOUND(404);
    private final int code;
    HttpStatus(int code) { this.code = code; }
    public int getCode() { return code; }
}

11. 面试高频考点

  • 接口和抽象类的区别？——多继承、字段性质、方法默认修饰符、构造方法、设计意图（"能做什么" vs "是什么"）
  • default 方法为何引入？解决了什么问题？——接口演进的向后兼容；顺带说菱形冲突解决规则
  • 接口的 static 方法能否被继承？——不能，只能通过接口名调用；与抽象类 static 方法的区别
  • 什么是函数式接口？@FunctionalInterface 有什么用？——只有一个抽象方法；注解让编译器校验，不是运行时必须
  • 接口字段是什么性质？——默认 public static final，本质是常量
  • 两个接口都有同名 default 方法时怎么处理？——实现类必须 @Override，可用 InterfaceName.super.method() 指定调用哪个
  • JDK 动态代理为什么要求目标类实现接口？——代理类本身继承了 Proxy，Java 单继承限制，只能通过接口实现多态
