18. 注解 @Annotation

给代码贴"标签"，让编译器、框架和工具读懂你的意图——注解是 Java 元编程的基石。

为什么需要注解？

在注解出现之前（Java 1.4 及以前），框架想描述"这个类是个 Servlet""这个方法需要事务"， 只能靠 XML 配置文件。配置与代码分离，修改一处要改两个地方，极易出错。

注解（Annotation，JDK 5 引入）让开发者直接在源代码里打"标签"， 由编译器、字节码处理器或运行时反射来读取并执行对应逻辑。 这就是"约定优于配置"（Convention Over Configuration）的基础。

注解本身不会改变代码执行逻辑——它只是元数据（metadata）。 真正产生行为的是读取注解的处理器，例如 Spring 容器、编译器插件或 APT（Annotation Processing Tool）。

注解的本质

注解在底层是一个继承自 java.lang.annotation.Annotation 的接口。 用 @interface 关键字声明，编译器会把它编译成一个接口类。

MyAnnotation.java — 注解的本质
// 你写的：
public @interface MyAnnotation {
    String value();
}

// 编译器看到的（伪代码）：
public interface MyAnnotation extends java.lang.annotation.Annotation {
    String value();
}

注解的"属性"写法像方法声明，但没有方法体，可以有 default 默认值。 使用时 value() 这个单一属性可以省略属性名直接写值。

元注解：注解的注解

JDK 提供了 5 个元注解（标注在注解定义上的注解），控制注解自身的行为。

元注解 作用 常用值
@Retention 注解保留到哪个阶段 SOURCE / CLASS / RUNTIME
@Target 注解可以打在哪些元素上 TYPE / METHOD / FIELD / PARAMETER 等
@Documented 是否出现在 Javadoc 中 无属性，标记性
@Inherited 子类是否继承父类的该注解（仅对类注解有效） 无属性，标记性
@Repeatable（JDK 8） 同一位置可重复使用该注解 指定容器注解类型
陷阱：@Retention 选错导致运行时读不到注解。 默认是 CLASS（保存到 .class 文件，但 JVM 加载后丢弃）， 如果你打算用反射在运行时读取，必须显式写 @Retention(RetentionPolicy.RUNTIME)。

自定义注解：从声明到读取

第一步：声明注解

RateLimit.java
import java.lang.annotation.*;

@Retention(RetentionPolicy.RUNTIME)  // 运行时可读
@Target(ElementType.METHOD)           // 只能打在方法上
@Documented
public @interface RateLimit {
    /** 每秒最大请求数，默认 100 */
    int qps() default 100;

    /** 超限提示信息 */
    String message() default "请求过于频繁，请稍后重试";
}

第二步：使用注解

OrderService.java
public class OrderService {

    @RateLimit(qps = 10, message = "下单太频繁啦")
    public void createOrder() {
        // 业务逻辑
    }

    @RateLimit  // 使用默认值 qps=100
    public void queryOrder() {
        // 业务逻辑
    }
}

第三步：反射读取注解（运行时处理器）

RateLimitProcessor.java — 反射读取
import java.lang.reflect.*;

public class RateLimitProcessor {

    public static void check(Object target, String methodName)
            throws NoSuchMethodException {

        Method method = target.getClass().getMethod(methodName);

        // isAnnotationPresent 先判空，再取值——好习惯
        if (method.isAnnotationPresent(RateLimit.class)) {
            RateLimit limit = method.getAnnotation(RateLimit.class);
            System.out.printf("方法 %s 限流 qps=%d，提示：%s%n",
                methodName, limit.qps(), limit.message());
            // 真实场景：在这里接入令牌桶 / 滑动窗口算法
        }
    }
}

JDK 内置注解速览

注解位置作用
@Override方法让编译器检查是否真的重写了父类/接口方法，拼错方法名时直接报错
@Deprecated任意标记已过时，调用方会出现删除线警告
@SuppressWarnings任意压制指定类型的编译器警告（如 "unchecked"）
@FunctionalInterface接口声明函数式接口，接口有多个抽象方法时编译报错
@SafeVarargs方法/构造器抑制泛型可变参数的堆污染警告

三种 RetentionPolicy 的适用场景

RetentionPolicy ├── SOURCE → 编译后丢弃，只供编译器/APT 使用 │ 例：@Override @SuppressWarnings Lombok 的 @Data │ ├── CLASS → 保存到 .class，但 JVM 不加载到内存（默认值） │ 例：字节码增强工具（ASM、ByteBuddy）读取 │ └── RUNTIME → JVM 加载后仍可通过反射读取 例：Spring @Autowired JUnit @Test 你自己的业务注解

注解属性支持的类型

注解属性类型有严格限制，只支持以下几种：

  • 8 种基本类型（int、long、boolean 等）
  • String
  • Class（及泛型形式 Class<?>）
  • 枚举类型（enum）
  • 其他注解类型
  • 以上任意类型的一维数组
陷阱：不能用 Integer（包装类）、List、Map 等作为注解属性类型， 编译器会直接报错。需要多个值时，用数组 String[] 代替。

@Repeatable：可重复注解（JDK 8+）

同一位置需要打多个相同注解时，需要配合一个"容器注解"使用。

Schedule.java — 可重复注解
// 1. 容器注解
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Schedules {
    Schedule[] value();
}

// 2. 可重复注解本体，指向容器
@Repeatable(Schedules.class)
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Schedule {
    String cron();
}

// 3. 使用——同一方法打两次
public class ReportJob {
    @Schedule(cron = "0 0 8 * * ?")   // 每天 08:00
    @Schedule(cron = "0 0 20 * * ?")  // 每天 20:00
    public void generateReport() { }
}

注解在 Spring 中的核心应用

Spring 框架几乎全部基于注解驱动，理解注解原理能帮你看穿 Spring 的魔法：

Spring 注解背后机制
@Component / @Service类扫描时反射读取，注册为 Bean
@Autowired反射注入依赖，Field.set() 绕过 private
@TransactionalAOP 动态代理，运行时读取注解，织入事务逻辑
@RequestMappingDispatcherServlet 启动时扫描，建立 URL → 方法映射表
@Validated + JSR-380APT 或 AOP 在方法调用前触发参数校验
建立联系：注解 + 反射 + 动态代理 = Spring AOP 的完整链路。 学完注解后再看 AOP 源码，会豁然开朗。

APT：编译期注解处理器

并非所有注解都依赖运行时反射。APT（Annotation Processing Tool） 在javac编译期间运行，可以生成新的源文件，零运行时开销。

  • Lombok：@Data → 编译期自动生成 getter/setter/toString/equals
  • MapStruct：@Mapper → 生成类型安全的对象映射代码
  • AutoValue（Google）：生成不可变值类
  • Dagger2：编译期依赖注入，Android 中替代 Spring 反射注入
APT 的入口是实现 javax.annotation.processing.AbstractProcessor， 覆写 process() 方法，通过 ProcessingEnvironment 读取注解元素并写出 Java 文件。

常见陷阱 & 面试考点

陷阱 1：@Inherited 只对类注解有效

@Inherited 让子类继承父类类级别的注解， 但接口上的注解不会被实现类继承，方法注解也不继承。 Spring 的 @Transactional 打在接口方法上， 在某些代理模式下可能失效，原因就在这里。

陷阱 2：注解属性值必须是编译期常量

编译报错示例
public class Foo {
    static final String OK = "hello";        // ✅ 编译期常量
    static String BAD = "world";             // ❌ 非 final，不是常量

    @MyAnnotation(value = OK)               // ✅
    public void a() {}

    @MyAnnotation(value = BAD)              // ❌ 编译报错
    public void b() {}
}

陷阱 3：注解不能继承注解

@interface 不支持 extends 另一个 @interface。 Spring 用注解组合（meta-annotation）来模拟继承： @Service 本身被 @Component 标注，Spring 扫描时向上遍历元注解。

面试高频：注解与反射的性能

Method.getAnnotation() 底层有缓存，首次调用较慢，后续走缓存很快。 但频繁调用 getDeclaredMethods() 会产生数组拷贝， 高性能场景应该在启动时扫描一次，结果存入 Map，避免每次请求都反射。

运行时注解 vs APT 注解：怎么选？

维度运行时注解（RUNTIME）编译期注解（APT）
执行时机JVM 运行时，反射读取javac 编译阶段
性能有反射开销零运行时开销
能力可动态决策（AOP 等）只能生成代码，不能修改已有代码（Lombok 用 hack 方式修改 AST）
调试难度栈帧可见，易调试生成代码可能难以追踪
典型场景Spring DI / AOP / 权限校验Lombok / MapStruct / Dagger
