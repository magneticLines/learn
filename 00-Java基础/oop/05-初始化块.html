05. 初始化块（static / 实例）

初始化块是构造器之外的第三种初始化手段，理解它的执行时机与顺序， 是彻底搞清 Java 对象生命周期的关键一环。

① 为什么需要初始化块？

Java 提供了三种给字段赋初值的方式：字段声明处直接赋值、 构造器、以及本节要讲的初始化块。 前两种大家熟悉，为什么还需要第三种？

设想你有 4 个构造器重载，但它们都需要做同一段「校验 + 填充默认值」的逻辑—— 写 4 遍既重复又容易遗漏，提取成私有方法是一种解法，但实例初始化块提供了更直接的语言级支持： 把公共初始化逻辑放在块里，编译器会自动把它复制到每个构造器的开头。

而静态初始化块则解决了另一个问题：某些 `static` 字段的值 无法用一个表达式算出来（比如需要 try-catch 的 JDBC 驱动加载、从配置文件读取常量）， 这时就必须依赖静态块。

② 两种初始化块的语法

Java — 语法骨架
public class Demo {

    // 静态初始化块 —— 类加载时执行一次
    static {
        System.out.println("静态块执行");
    }

    // 实例初始化块 —— 每次 new 时，在构造器之前执行
    {
        System.out.println("实例块执行");
    }

    public Demo() {
        System.out.println("构造器执行");
    }
}

两种块的语法区别只有一个关键字：有 static 就是静态块，没有就是实例块。

③ 执行顺序——最核心的知识点

Java 类的初始化分两个阶段：类加载阶段（只发生一次）和 对象创建阶段（每次 new 都发生）。

【类加载阶段（ClassLoader 触发，仅一次）】 ├─ ① 静态字段按声明顺序赋默认值（0 / null / false） ├─ ② 静态字段初始化表达式 & 静态块（按文本顺序交替执行） └─ 类加载完成，驻留方法区 【对象创建阶段（每次 new）】 ├─ ③ 实例字段按声明顺序赋默认值 ├─ ④ 实例字段初始化表达式 & 实例块（按文本顺序交替执行） └─ ⑤ 构造器体执行
关键口诀：静态 > 实例 > 构造器；父类 > 子类；声明顺序即执行顺序。 静态块和静态字段初始化是"并列"的——谁在文本上靠前谁先执行。

完整顺序验证代码

Java — 执行顺序验证
public class OrderDemo {

    // 静态字段初始化表达式（位置①）
    static int A = initA();

    // 静态块（位置②，在 A 之后）
    static {
        System.out.println("static block, A=" + A);  // 打印 A=1
    }

    // 实例字段初始化表达式（位置③）
    int b = initB();

    // 实例块（位置④，在 b 之后）
    {
        System.out.println("instance block, b=" + b); // 打印 b=10
    }

    // 构造器（位置⑤，最后执行）
    public OrderDemo() {
        System.out.println("constructor");
    }

    static int initA() {
        System.out.println("init A");
        return 1;
    }
    int initB() {
        System.out.println("init b");
        return 10;
    }

    public static void main(String[] args) {
        System.out.println("--- new 第一个对象 ---");
        new OrderDemo();
        System.out.println("--- new 第二个对象 ---");
        new OrderDemo();
    }
}

实际输出：

输出结果
// 类加载时（仅一次）
init A
static block, A=1

--- new 第一个对象 ---
// 每次 new 都重复执行实例块和构造器
init b
instance block, b=10
constructor

--- new 第二个对象 ---
init b
instance block, b=10
constructor
注意：静态块只打印了一次——因为类只加载一次。第二个 new 不会再触发静态块。

④ 静态初始化块的典型用法

静态块的核心价值是：处理无法用单个表达式完成的静态字段初始化， 尤其是需要 try-catch、循环、多步骤计算的场景。

场景一：加载 JDBC 驱动（经典用法）

Java — JDBC 驱动加载
public class DbConfig {

    static {
        try {
            // Class.forName 需要处理受检异常，必须用 try-catch
            // 不能写成 static String driver = Class.forName("...");
            Class.forName("com.mysql.cj.jdbc.Driver");
        } catch (ClassNotFoundException e) {
            throw new RuntimeException("数据库驱动加载失败", e);
        }
    }
}

场景二：初始化不可变的静态集合

Java — 静态集合初始化
public class CountryCode {

    private static final Map<String, String> CODE_MAP;

    static {
        // Map.of() 只支持有限条目，复杂初始化需要静态块
        Map<String, String> map = new HashMap<>();
        map.put("CN", "China");
        map.put("US", "United States");
        map.put("JP", "Japan");
        // 最后包装成不可变视图，防止外部修改
        CODE_MAP = Collections.unmodifiableMap(map);
    }
}

⑤ 实例初始化块的典型用法

实例块在现代 Java（有了构造器链 this(...)）中用得相对少， 但仍有两个经典场景。

场景一：多构造器共享公共逻辑

Java — 反例 vs 正例
// ❌ 反例：3 个构造器重复了同一段校验逻辑
public class Order {
    private final String id;
    private List<String> items;

    public Order(String id) {
        if (id == null) throw new NullPointerException();
        items = new ArrayList<>();  // 重复
        this.id = id;
    }
    public Order(String id, String firstItem) {
        if (id == null) throw new NullPointerException();
        items = new ArrayList<>();  // 重复
        this.id = id;
        items.add(firstItem);
    }
}

// ✅ 正例：公共逻辑放入实例块，构造器只写差异部分
public class Order {
    private final String id;
    private List<String> items;

    // 实例块：每次 new 都会执行
    {
        items = new ArrayList<>();
    }

    public Order(String id) {
        if (id == null) throw new NullPointerException();
        this.id = id;
    }
    public Order(String id, String firstItem) {
        if (id == null) throw new NullPointerException();
        this.id = id;
        items.add(firstItem);
    }
}

场景二：匿名类中的初始化

匿名类没有名字，无法定义命名构造器，实例块是匿名类唯一的初始化入口。 常见于"双括号初始化"（Double Brace Initialization）反模式：

Java — 匿名类实例块（了解即可，慎用）
// 外层 {} 创建匿名子类，内层 {} 是实例块
List<String> list = new ArrayList<>() {{
    add("a");
    add("b");
}};
双括号初始化会创建匿名子类，持有外部类引用可能导致内存泄漏，也无法序列化。 实际项目中请改用 List.of("a", "b")（Java 9+）或 Arrays.asList(...)。

⑥ 编译器如何处理初始化块（字节码视角）

初始化块不是运行时的独立结构，它在编译阶段就被"内联"了：

  • 实例块被编译器复制到每个构造器的开头（紧接在 super() 之后）。
  • 静态块被合并进类的 <clinit>（class initializer）方法。
伪字节码 — 编译器做了什么
// 你写的源码
class Foo {
    int x = 1;
    { x = 2; }          // 实例块
    public Foo() { x = 3; }
}

// 编译器生成的等价代码（概念上）
class Foo {
    int x;
    public Foo() {
        super();          // 隐式 super()
        x = 1;            // 字段声明处赋值
        x = 2;            // 实例块被内联进来
        x = 3;            // 原来的构造器体
    }                    // 最终 x == 3
}

用 javap -c Foo.class 反编译可以直接看到 <init> 方法里三条赋值指令依次出现，印证了这一点。

⑦ 继承场景下的执行顺序

有继承时，JVM 保证父类先于子类初始化。完整顺序如下：

① 父类 静态字段 + 静态块（文本顺序） ② 子类 静态字段 + 静态块（文本顺序） ③ 父类 实例字段 + 实例块（文本顺序） ④ 父类 构造器体 ⑤ 子类 实例字段 + 实例块（文本顺序） ⑥ 子类 构造器体
Java — 继承顺序验证
class Parent {
    static { System.out.println("Parent static"); }
    { System.out.println("Parent instance"); }
    Parent() { System.out.println("Parent constructor"); }
}

class Child extends Parent {
    static { System.out.println("Child static"); }
    { System.out.println("Child instance"); }
    Child() { System.out.println("Child constructor"); }

    public static void main(String[] args) {
        new Child();
    }
}
/*
输出：
Parent static
Child static
Parent instance
Parent constructor
Child instance
Child constructor
*/

⑧ 常见陷阱

陷阱一：静态块中引用"还未赋值"的静态字段

Java — 前向引用问题
class Trap {
    static {
        // MAX 此时只有默认值 0，因为声明在块之后
        System.out.println("MAX = " + MAX);  // 打印 MAX = 0 !!!
    }
    static int MAX = 100;  // 声明在块之后
}

Java 允许静态块"读"到后面声明的字段（不会编译报错），但此时该字段只有默认值（int 是 0，对象是 null）。 规则：字段初始化和静态块按文本从上到下顺序执行。

陷阱二：静态块中抛异常导致类加载失败

Java — ExceptionInInitializerError
class BadInit {
    static {
        if (true) throw new RuntimeException("初始化失败");
    }
}

// 第一次使用 BadInit 时，JVM 抛出 ExceptionInInitializerError
// 此后再次使用同一类，会直接抛出 NoClassDefFoundError
// 即：类进入"损坏"状态，无法恢复！
静态块里的未捕获异常是不可逆的。JVM 会将该类标记为「初始化失败」， 后续所有引用该类的代码都会得到 NoClassDefFoundError，即使重试也无济于事。 生产代码的静态块必须做好异常处理。

陷阱三：在静态块里调用实例方法

Java — 编译错误
class Trap2 {
    int x = 1;

    static {
        // 编译报错：non-static variable x cannot be referenced from a static context
        System.out.println(x);
    }
}

静态块属于类级别，此时对象还不存在，自然无法访问实例成员。 这和 static 方法不能直接调用实例方法是同样的道理。

⑨ 与框架的联系

Spring 中的 @PostConstruct

Spring Bean 的生命周期引入了 @PostConstruct 注解，作用类似实例块—— 在构造器和依赖注入都完成之后再执行初始化逻辑。 这是静态块 / 实例块无法替代的，因为依赖注入在对象构造期间还未就绪：

Java — Spring Bean 初始化对比
@Component
public class UserService {

    @Autowired
    private UserRepository repo;

    // ❌ 实例块在注入前执行，此时 repo == null
    {
        // repo.count(); // NullPointerException!
    }

    // ✅ @PostConstruct 在注入完成后执行，repo 已就绪
    @PostConstruct
    public void init() {
        System.out.println("用户总数: " + repo.count());
    }
}

枚举类的静态块

枚举常量本身就是 static final 字段，由编译器在 <clinit> 方法（即静态块）中按声明顺序创建。因此枚举的执行顺序与普通类完全一致—— 理解静态块是理解枚举内部机制的基础。

⑩ 横向对比

面试高频考点

  • 执行顺序题（最高频）：给一段含静态块、实例块、构造器的代码， 要求写出输出。关键记住：静态 → 实例 → 构造器；父类 → 子类；文本顺序。
  • 静态块 vs 静态方法：静态块是自动执行的，无法手动调用； 静态方法需要显式调用。两者都不能访问实例成员。
  • <clinit> 线程安全吗？ JVM 保证 <clinit> 只执行一次且线程安全（加锁）， 这正是「饿汉式单例」用静态字段实现线程安全的底层原因。
  • 静态块抛异常会怎样？ 触发 ExceptionInInitializerError，类进入不可恢复的损坏状态， 后续访问报 NoClassDefFoundError。
  • 为什么 final 字段也可以在实例块里赋值？ 因为编译器把实例块内联到构造器，本质上就是构造器赋值，语法合法。
