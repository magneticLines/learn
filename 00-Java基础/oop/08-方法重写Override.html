方法重写 @Override

子类重新定义父类方法的行为——多态的根基，运行时绑定的核心机制。

1. 为什么需要方法重写？

继承让子类复用父类代码，但"复用"并不等于"原封不动地沿用"。 现实世界里，同一个动作在不同对象上表现不同： 动物都会"叫"，但狗叫"汪汪"、猫叫"喵喵"。 如果没有重写机制，子类只能继承父类的同一套叫法，多态便无从谈起。

方法重写（Method Overriding）允许子类用相同的方法签名提供新的实现， 而调用者无需知道对象的具体类型——这就是"对扩展开放、对修改关闭"的 OCP 原则的底层支撑。

核心公式：继承 + 重写 = 多态。没有重写，继承只是代码复用；有了重写，继承才能实现行为替换。

2. 重写的五条硬性规则

Java 编译器对重写有严格的规定，违反任意一条都会编译报错或产生不符合预期的行为。

规则 说明 为什么这样设计？
方法名相同 与父类完全一致 调用者通过父类引用调用，JVM 靠方法名定位
参数列表相同 类型、顺序、个数完全一致 参数不同就变成了重载，而非重写
返回类型相同或协变 可以是父类返回类型的子类（Java 5+） 协变返回类型让 API 更精确，无需强转
访问权限不能更严 public → public，不能改为 private 里氏替换原则：子类对外承诺的能力只能更多，不能更少
异常声明不能更宽 checked 异常只能一样或更小，unchecked 不限 调用者按父类声明捕获异常，子类抛更宽会破坏调用方

3. @Override 注解的作用

@Override 是一个编译期注解，本身不影响运行时行为。 它的唯一作用是：告诉编译器"我认为这是重写，请帮我校验"。 一旦父类中找不到匹配的方法，编译立刻报错。

不加 @Override 的陷阱：你以为在重写，其实在定义一个新方法。 比如父类方法叫 toString()，你手误写成 tostring()——不加注解编译通过，运行时行为却完全错误，调试极其困难。
错误示范 vs 正确写法
class Animal {
    public String toString() {
        return "Animal";
    }
}

class Dog extends Animal {

    // ❌ 没有 @Override，手误写错方法名，编译不报错！
    public String tostring() {
        return "Dog";
    }

    // ✅ 加了 @Override，拼错立刻编译报错，提早发现问题
    @Override
    public String toString() {
        return "Dog";
    }
}
最佳实践：凡是你认为在重写的方法，永远加 @Override。这是 Java 编码规范的基本要求，也是所有主流代码检查工具（Checkstyle、SonarQube）的默认强制规则。

4. 完整示例：动物叫声

用一个经典的继承体系演示重写的全貌，包括调用时的动态分派过程。

Animal.java / Dog.java / Cat.java
// 父类：定义"通用契约"
class Animal {
    public void sound() {
        System.out.println("某种声音...");
    }

    public String describe() {
        return "我是一只动物";
    }
}

// 子类：重写父类方法，提供具体行为
class Dog extends Animal {

    @Override
    public void sound() {
        // 完全替换父类实现
        System.out.println("汪汪！");
    }

    @Override
    public String describe() {
        // 用 super 保留父类逻辑，再追加子类特有内容
        return super.describe() + "，具体来说是狗";
    }
}

class Cat extends Animal {

    @Override
    public void sound() {
        System.out.println("喵喵～");
    }
}
Main.java — 多态调用
public class Main {
    public static void main(String[] args) {

        // 父类引用指向子类对象——多态的标志写法
        Animal a1 = new Dog();
        Animal a2 = new Cat();

        a1.sound();   // 运行时查 Dog 的方法表 → 输出"汪汪！"
        a2.sound();   // 运行时查 Cat 的方法表 → 输出"喵喵～"

        System.out.println(a1.describe());
        // 输出：我是一只动物，具体来说是狗
    }
}

5. 动态分派：重写为什么能"运行时选方法"

编译期，编译器只知道引用类型是 Animal，所以它在字节码里写的是 invokevirtual Animal.sound。 真正决定调用哪个实现的，是 JVM 在运行时根据对象的实际类型查询方法表（vtable）。

编译期字节码：invokevirtual Animal.sound     ↓ 运行时：查对象头 → 找到实际类型 Dog     ↓ 查 Dog 的 vtable → 找到 Dog.sound()     ↓ 执行 Dog.sound() → 输出"汪汪！"

这个过程叫动态单分派（Dynamic Single Dispatch）。 "单"指的是只根据接收者对象的类型来决定方法，而不看参数类型 （参数类型决定的是重载，属于静态分派）。

vtable 简介：每个类在 JVM 方法区（元空间）中都维护一张虚方法表。 子类的 vtable 会把从父类继承的条目用自己的重写版本覆盖，其余继承不变。 这就是重写在底层的实现机制。

6. 重写 vs 重载 — 最高频考点

这两个概念极易混淆，面试必考，务必掌握本质区别。

维度 重写 Override 重载 Overload
发生在 父子类之间 同一个类内
方法名 相同 相同
参数列表 必须完全相同 必须不同（类型/顺序/个数）
返回类型 相同或协变（子类） 可以不同，但不能以此区分重载
绑定时机 运行时（动态绑定） 编译期（静态绑定）
多态关系 是多态的核心实现 与多态无关，是便利性语法糖
@Override 应该加，用于校验 不能加（会编译报错）

7. 协变返回类型（Java 5+）

子类重写方法时，返回类型可以是父类返回类型的子类，称为协变返回。 这让子类 API 更精确，调用者无需强转。

协变返回类型示例
class Animal {
    // 父类返回 Animal
    public Animal create() {
        return new Animal();
    }
}

class Dog extends Animal {
    @Override
    // ✅ 返回类型改为子类 Dog，合法！这就是协变返回
    public Dog create() {
        return new Dog();
    }
}

// 使用时无需强转
Dog dog = new Dog().create();  // 直接得到 Dog 类型，干净

8. 哪些方法不能被重写？

有四类方法天然排斥重写，理解原因比背结论更重要。

  • private 方法：子类根本看不见父类的 private 方法，写同名方法是新定义而非重写，@Override 会编译报错。
  • static 方法：静态方法属于类，不属于对象，没有动态分派。子类写同名静态方法叫"隐藏（hide）"，不叫重写。
  • final 方法：显式禁止重写。常用于模板方法模式中固定骨架流程，防止子类破坏算法结构。
  • 构造方法：构造方法名必须与类名相同，子类不可能有相同名字，根本无法重写。
static 方法"隐藏"陷阱：父类引用调用被隐藏的静态方法时，执行的是父类版本，而不是子类版本。这和重写的动态分派完全相反，面试常考。
static 隐藏 vs Override 对比
class Parent {
    public static void staticMethod() {
        System.out.println("Parent.static");
    }
    public void instanceMethod() {
        System.out.println("Parent.instance");
    }
}

class Child extends Parent {
    public static void staticMethod() {   // 隐藏，不是重写
        System.out.println("Child.static");
    }
    @Override
    public void instanceMethod() {      // 真正的重写
        System.out.println("Child.instance");
    }
}

Parent p = new Child();
p.staticMethod();    // 输出 "Parent.static"  ← 编译期绑定，看引用类型
p.instanceMethod();  // 输出 "Child.instance" ← 运行时绑定，看实际类型

9. 重写中的 super 调用

重写时，子类并不是非要完全替换父类逻辑。 通过 super.方法名() 可以先执行父类逻辑，再追加子类逻辑， 类似于"在原有基础上扩展"，这在实际开发中非常常见。

super 保留父类逻辑示例
class HttpFilter {
    public void doFilter(String request) {
        System.out.println("记录请求日志: " + request);
    }
}

class AuthFilter extends HttpFilter {
    @Override
    public void doFilter(String request) {
        // 先执行父类的日志记录
        super.doFilter(request);
        // 再追加鉴权逻辑
        System.out.println("验证 Token...");
    }
}
Spring 中的体现：Spring MVC 的 HandlerInterceptor、 Spring Security 的 OncePerRequestFilter 都大量使用"重写 + super 调用" 来构建可扩展的过滤链，掌握重写原理是读懂框架源码的前提。

10. 接口默认方法与重写

Java 8 引入的 default 方法也可以被实现类重写。 当类同时实现多个接口且存在同名 default 方法时，必须重写来消除歧义，否则编译报错。

消除接口 default 方法歧义
interface Flyable {
    default void move() { System.out.println("飞行移动"); }
}
interface Swimmable {
    default void move() { System.out.println("游泳移动"); }
}

// 同时实现两个接口 → 必须重写 move()，否则编译报错
class Duck implements Flyable, Swimmable {
    @Override
    public void move() {
        // 可以选择其中一个，或者自己完全重新实现
        Flyable.super.move();
    }
}

11. 面试高频考点

Q1：重写时访问权限为什么不能更严格？

里氏替换原则（LSP）：任何使用父类的地方都应该能无感地换成子类。 如果子类把 public 改成 private，调用方拿到父类引用调用时就会失败，违反替换原则。

Q2：构造方法可以被重写吗？

不能。构造方法名必须与类名相同，子类构造方法名也必须是子类名，不可能同名，根本不满足重写的前提条件。

Q3：重写与隐藏（static）的核心区别？

重写是运行时绑定，看对象的实际类型；隐藏是编译时绑定，看引用的声明类型。 这是两者最根本的区别。

Q4：final 类里的方法能重写吗？

不能。final 类（如 String、Integer）不能被继承， 自然也就无从谈起重写。

Q5：@Override 是运行时注解吗？

不是。@Override 的 retention 是 SOURCE，编译后字节码里不保留。 它纯粹服务于编译期校验和 IDE 提示。
