14. 函数式接口与 @FunctionalInterface

Java 8 最重要的语言扩展之一：让接口可以作为"行为"直接传递，Lambda 和方法引用的类型基础。

1. 问题背景：行为无法直接传递

Java 是面向对象语言，一切皆对象。在 Java 8 之前，如果想把一段"行为"（逻辑）传给另一个方法，唯一的办法是把它包进匿名内部类。

Java 7：匿名内部类传行为
// 对列表按字符串长度排序
List<String> names = Arrays.asList("Alice", "Bob", "Charlie");

Collections.sort(names, new Comparator<String>() {
    @Override
    public int compare(String a, String b) {
        return Integer.compare(a.length(), b.length());
    }
});
// 大量样板代码，核心逻辑却只有一行

这段代码的真正意图是"按长度比较"，却被 new Comparator<>() { @Override ... } 这些样板淹没。Java 8 引入 Lambda，让行为可以直接写成 (a, b) -> Integer.compare(a.length(), b.length())。

但 Lambda 不是新类型，它需要一个类型载体——这个载体就是函数式接口。

核心设计思路：函数式接口 = 只有一个抽象方法的接口。Lambda 表达式的类型就是对应的函数式接口，编译器通过上下文推断出具体是哪个接口。

2. 函数式接口的定义规则

函数式接口（Functional Interface）：有且仅有一个抽象方法的接口，也叫 SAM 接口（Single Abstract Method）。

最简函数式接口
@FunctionalInterface
public interface Transformer<T, R> {
    R transform(T input);  // 唯一的抽象方法
}

哪些方法不计入"抽象方法数"？

  • default 方法：有方法体，不算抽象
  • static 方法：同上
  • 来自 java.lang.Object 的公共方法（如 equals、toString、hashCode）：所有类都会实现，不算额外约束
合法的函数式接口——带 default 和 static
@FunctionalInterface
public interface Validator<T> {
    boolean validate(T value);      // 唯一抽象方法

    default Validator<T> and(Validator<T> other) {
        return v -> this.validate(v) && other.validate(v);
    }

    static Validator<String> notEmpty() {
        return s -> s != null && !s.isEmpty();
    }

    // Object 的方法不算抽象方法，声明这个不影响函数式接口身份
    boolean equals(Object obj);
}

3. @FunctionalInterface 注解的作用

@FunctionalInterface 是一个编译器保护注解，本身不改变接口的语义，但它让编译器帮你做检查：

  • 若接口抽象方法数 ≠ 1，编译报错，防止误加方法破坏下游 Lambda 代码
  • 向读者明确表达"这个接口设计用于 Lambda"，提升可读性
  • 不加注解也能作为函数式接口使用，注解只是显式声明意图
编译器报错示例
@FunctionalInterface
interface Bad {
    void methodA();
    void methodB();
    // 编译错误：Multiple non-overriding abstract methods found in interface Bad
}
类比：就像 @Override 不会改变方法覆盖的规则，但能让编译器在你写错方法签名时报错。@FunctionalInterface 是同样的"意图声明 + 保护"模式。

4. JDK 内置的核心函数式接口

Java 8 在 java.util.function 包中内置了 43 个函数式接口，掌握 4 个核心类型就能覆盖 90% 的场景：

接口 方法签名 语义 典型使用
Function<T,R> R apply(T t) 输入 T，返回 R（转换） Stream.map()
Predicate<T> boolean test(T t) 输入 T，返回 boolean（判断） Stream.filter()
Consumer<T> void accept(T t) 消费 T，无返回（副作用） Stream.forEach()
Supplier<T> T get() 不接受参数，提供 T（工厂） Optional.orElseGet()

记忆口诀：Function 转换、Predicate 判断、Consumer 消费、Supplier 生产。

四大接口实战——用 Lambda 表达式
import java.util.function.*;

// Function：字符串 -> 长度
Function<String, Integer> len = s -> s.length();
System.out.println(len.apply("hello"));  // 5

// Predicate：判断字符串是否非空
Predicate<String> notEmpty = s -> !s.isEmpty();
System.out.println(notEmpty.test(""));      // false

// Consumer：打印每个元素
Consumer<String> printer = System.out::println;
printer.accept("world");                  // world

// Supplier：延迟创建对象（只在需要时才构造）
Supplier<List<String>> listFactory = ArrayList::new;
List<String> list = listFactory.get();   // 每次调用 get() 才 new

扩展接口一览（快速记忆）

java.util.function ├── Function<T,R> T → R │ ├── UnaryOperator<T> T → T（特化：输入输出同类型） │ └── BiFunction<T,U,R> (T,U) → R │ └── BinaryOperator<T> (T,T) → T ├── Predicate<T> T → boolean │ └── BiPredicate<T,U> (T,U) → boolean ├── Consumer<T> T → void │ └── BiConsumer<T,U> (T,U) → void └── Supplier<T> () → T

5. Lambda 是如何对应到函数式接口的

Lambda 表达式本身没有类型，编译器根据目标类型（Target Type）推断它应该适配哪个函数式接口。

同一个 Lambda，目标类型不同，语义不同
// 同一段 Lambda 代码
Runnable    r = () -> System.out.println("hi");
Callable<Void> c = () -> { System.out.println("hi"); return null; };

// 编译器看到赋值目标是 Runnable，就把 Lambda 编译为 Runnable 的实现
// 看到 Callable<Void>，就编译为 Callable 的实现

// 如果目标类型不明确，编译器报错
var bad = () -> System.out.println("hi");
// 编译错误：Cannot infer type: lambda expression requires an explicit target-type

方法引用也是函数式接口的实现

四种方法引用形式
// 1. 静态方法引用：ClassName::staticMethod
Function<String, Integer> parse = Integer::parseInt;

// 2. 实例方法引用（特定实例）：instance::method
String prefix = "Hello, ";
Function<String, String> greet = prefix::concat;

// 3. 实例方法引用（任意实例）：ClassName::instanceMethod
Function<String, String> upper = String::toUpperCase;
// 等价于 s -> s.toUpperCase()

// 4. 构造方法引用：ClassName::new
Supplier<StringBuilder> sbFactory = StringBuilder::new;

6. 何时需要自定义函数式接口

JDK 内置的 43 个接口已经覆盖了大部分场景，但以下情况需要自定义：

  • 需要抛出受检异常（JDK 内置接口的方法不声明 throws）
  • 三个或以上参数
  • 需要语义化的接口名提升可读性（如 PasswordEncoder 比 Function<String, String> 更清晰）
自定义：支持受检异常的 Function
@FunctionalInterface
public interface CheckedFunction<T, R> {
    R apply(T t) throws Exception;

    // 工具方法：把受检异常包装成运行时异常，方便在 Stream 里使用
    static <T, R> Function<T, R> wrap(CheckedFunction<T, R> f) {
        return t -> {
            try {
                return f.apply(t);
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        };
    }
}

// 使用：Stream.map 不允许 throws，通过 wrap 解决
paths.stream()
     .map(CheckedFunction.wrap(Files::readString))
     .forEach(System.out::println);

7. 函数组合：default 方法的妙用

JDK 内置的函数式接口都提供了 default 方法用于组合，让多个函数链式拼接，体现了高阶函数思想。

Function 的组合：andThen / compose
Function<String, String> trim  = String::trim;
Function<String, String> upper = String::toUpperCase;
Function<String, Integer> len  = String::length;

// andThen：先执行 this，再执行参数（从左到右）
Function<String, Integer> pipeline = trim.andThen(upper).andThen(len);
pipeline.apply("  hello  ");  // trim -> "hello" -> "HELLO" -> 5

// Predicate 的组合：and / or / negate
Predicate<String> notEmpty  = s -> !s.isEmpty();
Predicate<String> notTooLong = s -> s.length() <= 20;
Predicate<String> valid = notEmpty.and(notTooLong);
valid.test("hello");  // true

8. 与 Stream API 的联系

Stream 的每个操作都以函数式接口为参数，函数式接口是 Stream API 的类型基础：

Stream 方法接受的函数式接口语义
map(Function)Function<T,R>转换每个元素
filter(Predicate)Predicate<T>过滤元素
forEach(Consumer)Consumer<T>消费每个元素
reduce(BinaryOperator)BinaryOperator<T>归约（合并）
sorted(Comparator)Comparator<T>（特殊）排序
collect(Collector)内含多个函数式接口聚合到容器
注意：Comparator<T> 有两个抽象方法（compare 和来自 Object 的 equals），但 equals 不计，所以它也是函数式接口，可以用 Lambda 表达。

9. 底层实现：Lambda 不是匿名内部类

很多人以为 Lambda 只是匿名内部类的语法糖，实际上底层完全不同。

反编译对比
// Lambda 写法
Runnable r = () -> System.out.println("hi");

// 编译后字节码使用 invokedynamic 指令：
//   INVOKEDYNAMIC run()Ljava/lang/Runnable; [LambdaMetafactory.metafactory]
// 不生成 $1.class 文件，不加载多余类

// 匿名内部类写法
Runnable r2 = new Runnable() {
    public void run() { System.out.println("hi"); }
};
// 编译后生成 OuterClass$1.class 文件，每次 new 都创建对象

Lambda 通过 invokedynamic 指令 + LambdaMetafactory 在运行时动态生成实现类，避免了大量匿名内部类文件的开销。对于不捕获变量的 Lambda，JVM 会缓存同一个实例（相当于单例），性能更优。

10. 常见陷阱与面试考点

陷阱 1：捕获变量必须是 effectively final

编译错误：捕获了被修改的变量
int count = 0;
Runnable r = () -> System.out.println(count);
count++;  // 修改了 count，编译错误
// Variable used in lambda expression should be final or effectively final

// 解决方案：用 AtomicInteger 或数组包装
int[] countRef = {0};
Runnable r2 = () -> System.out.println(countRef[0]);
countRef[0]++;  // 数组引用本身不变，内容可变，编译通过（但并发不安全）

陷阱 2：Lambda 中的 this 是外层类

Lambda vs 匿名内部类中 this 的区别
class Demo {
    void test() {
        Runnable lambda = () ->
            System.out.println(this);   // this = Demo 实例

        Runnable anon = new Runnable() {
            public void run() {
                System.out.println(this);  // this = 匿名内部类实例
            }
        };
    }
}
// Lambda 没有自己的 this，复用外层作用域

陷阱 3：不要把函数式接口与普通接口混用

没有 @FunctionalInterface 但意外变成函数式接口
// 危险写法：没有加注解，后来某人在接口里加了第二个方法
interface Handler {
    void handle(String event);
    // 某天新增了这行：
    void cleanup();
    // 所有使用 Lambda 赋值 Handler 的地方全部编译报错
}

// 安全写法：加上注解，第一时间发现问题
@FunctionalInterface
interface Handler {
    void handle(String event);
    // 再加 cleanup() → 编译阶段立刻报错，不会悄悄破坏下游
}

陷阱 4：基本类型自动装拆箱的性能问题

在高频调用场景下，Function<Integer, Integer> 会触发装拆箱。JDK 提供了基本类型特化版本避免此开销：

通用版特化版（无装拆箱）
Function<Integer, Integer>IntUnaryOperator
Function<Integer, Long>IntToLongFunction
Predicate<Integer>IntPredicate
Consumer<Integer>IntConsumer
Supplier<Integer>IntSupplier

11. 面试高频考点

高频考点 1：什么是函数式接口？是否必须加 @FunctionalInterface？
答：只有一个抽象方法的接口。注解非必须，但强烈推荐——它是编译器保护，防止接口被不小心破坏。
高频考点 2：Lambda 是匿名内部类的语法糖吗？
答：不是。匿名内部类编译生成独立 .class，每次 new 创建对象。Lambda 用 invokedynamic，运行时动态生成，无捕获时共享实例，性能更好，也不会产生 $n.class 文件。
高频考点 3：Function.andThen 和 Function.compose 的区别？
答：f.andThen(g) = 先 f 再 g（f → g）；f.compose(g) = 先 g 再 f（g → f）。数学记法：andThen 是 g(f(x))，compose 是 f(g(x))。
高频考点 4：Comparator 有两个抽象方法（compare 和 equals），为什么也是函数式接口？
答：来自 Object 的公共方法（如 equals）不计入抽象方法数，因为所有对象都已从 Object 继承了实现。
高频考点 5：Lambda 捕获变量为何必须 effectively final？
答：Lambda 可能在不同线程或延迟执行，若变量可被修改，会导致竞态条件和语义不一致。Java 选择在编译期禁止，而不是允许修改后面临并发 bug。
