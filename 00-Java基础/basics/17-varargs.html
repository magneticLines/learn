17 · 可变参数 varargs

语法糖背后是普通数组——掌握它，才能驾驭它的陷阱与性能成本。

1. 为什么需要 varargs

痛点一：重载爆炸

在 JDK 1.5 之前，如果你想让一个方法接受"任意数量"的整数，只能手写重载：

int sum(int a) { ... }
int sum(int a, int b) { ... }
int sum(int a, int b, int c) { ... }
// 传 10 个参数？再写 7 个重载？

痛点二：数组繁琐

另一种方案是传 int[]，但调用端必须手动构造数组：

sum(new int[]{1, 2, 3});  // 啰嗦

JDK 1.5 引入 varargs，让编译器帮我们完成"把零散参数打包成数组"这件事：

int sum(int... nums) {
    int total = 0;
    for (int n : nums) total += n;
    return total;
}

sum(1, 2, 3);      // 编译器 → sum(new int[]{1,2,3})
sum();             // 编译器 → sum(new int[]{})
设计哲学
varargs 本质是一个"调用端语法糖"——方法签名里的 T... 会被编译为 T[]，JVM 字节码层面没有任何新指令，完全是编译器的工作。

2. 基本语法

// 规则：varargs 只能出现在参数列表最后一位
static void log(String level, String... messages) {
    for (String msg : messages) {
        System.out.println("[" + level + "] " + msg);
    }
}

// 三种合法调用形式
log("INFO");                          // 零参数 → messages = new String[]{}
log("WARN", "disk low");             // 逐一传参
log("ERROR", "a", "b", "c");        // 多个值
log("DEBUG", new String[]{"x","y"}); // 直接传数组也合法
限制
  • 每个方法最多只能有一个 varargs 参数。
  • varargs 必须是最后一个参数，void f(int... a, int b) 编译报错。

3. 字节码真相：每次调用都分配新数组

用 javap -c 观察 sum(1, 2, 3) 的调用侧字节码：

iconst_3 // 数组长度 3
newarray int // new int[3]
dup
iconst_0
iconst_1
iastore // arr[0] = 1
dup
iconst_1
iconst_2
iastore // arr[1] = 2
dup
iconst_2
iconst_3
iastore // arr[2] = 3
invokestatic sum([I)I // 方法签名里是 int[]

关键结论：

  • 调用 varargs 方法时，编译器在调用侧插入 newarray + iastore 序列。
  • 方法签名在 .class 文件中是 int[]，仅用 ACC_VARARGS 标志位区分。
  • 每次调用都会分配一个新数组对象（哪怕是空参数 sum() 也会分配 new int[0]）。
空参数的数组分配
JVM 规范保证 new int[0] 是合法对象，不是 null。 所以方法体内对 varargs 参数直接 for-each 是安全的，但如果外部显式传 null 则会得到 null（见陷阱一）。

4. 与方法重载的关系

Java 重载解析分三轮，varargs 在最后一轮（优先级最低）：

轮次规则示例
第一轮精确匹配（不允许装拆箱、不允许 varargs）f(1) → f(int)
第二轮允许装拆箱，不允许 varargsf(1) → f(Integer)
第三轮允许 varargsf(1) → f(int...)
 
static void f(int x)      { System.out.println("精确"); }
static void f(int... xs) { System.out.println("varargs"); }

f(1);     // 输出 "精确"——第一轮就命中，varargs 被跳过
f(1,2);   // 输出 "varargs"——只有 varargs 能接受两个 int
常见面试陷阱
同时定义 f(int a, int b) 和 f(int... xs)，调用 f(1,2) 时 会优先匹配第一个（精确，第一轮），不会产生歧义编译错误——但容易让读者误以为走了 varargs。

5. 泛型 varargs 与堆污染

根因

Java 泛型在运行时会被类型擦除；而数组在运行时保留元素类型（协变数组）。 两者叠加，就会产生"堆污染"（heap pollution）——变量的实际运行时类型与声明类型不符。

危险写法
@SuppressWarnings("unchecked")
static <T> T[] toArray(T... items) {
    // 编译器警告：Possible heap pollution
    return items;
    // items 的实际类型是 Object[],
    // 强转给 String[] 时会在外部 ClassCastException
}

String[] arr = toArray("a", "b");
// ↑ 运行时 ClassCastException！
安全写法
@SafeVarargs
static <T> List<T> listOf(T... items) {
    // 只读，不暴露 items 引用
    return Arrays.asList(items);
}

// 调用侧无 unchecked 警告
List<String> s = listOf("a", "b");

堆污染的真正危险不在方法内部，而在于把 varargs 数组引用泄漏到外部：

  • 直接 return items → 调用方拿到类型错误的数组引用。
  • 把 items 存入 Object[] 字段后在别处强转 → 延迟爆炸。

6. @SafeVarargs 使用规则

可用位置 要求 JDK 案例
static 方法 方法不能被子类覆盖，所以类型安全可以静态保证 Arrays.asList(T...)
final 实例方法 同上，不可覆盖 Collections.unmodifiableList 等内部用法
private 实例方法 JDK 9+ 支持（在模块化后 private 已足够限制） List.of 内部私有方法
不可用：普通可覆盖实例方法 子类可改变行为，编译器无法保证安全 —

前提条件（你负责保证，编译器不验证）：

  1. 方法内部不向 varargs 数组写入任何泛型类型值。
  2. 不把 varargs 数组引用暴露（return / 赋值给外部变量）给外部代码。
经验法则
"只读不写、不泄漏引用"——满足这两条就可以加 @SafeVarargs， 这也是 JDK 源码中所有使用该注解的地方都遵守的约定。

7. 实战：printf / log 参数化模式

SLF4J 的参数化日志就是 varargs 的典型用法，避免在 DEBUG 级别关闭时做无意义的字符串拼接：

/**
 * 仿 SLF4J 参数化日志
 * 关键设计：只有真正需要打印时才调用 String.format()
 */
public class SimpleLogger {

    private static final boolean DEBUG_ENABLED = false;

    // varargs 接收占位符参数，延迟格式化
    public static void debug(String template, Object... args) {
        if (!DEBUG_ENABLED) return;  // 关闭时直接返回，数组虽已分配但格式化被跳过
        System.out.println(String.format(template, args));
    }

    public static void main(String[] args) {
        String user = "Alice";
        int age = 30;
        // 传参写法：清晰，无字符串拼接开销（格式化被延迟）
        debug("user=%s age=%d", user, age);
    }
}
注意：数组仍会被分配
即使 DEBUG_ENABLED=false，调用 debug(...) 时数组对象依然被创建。 这正是 SLF4J 推荐两参数快捷方法（debug(msg, arg1, arg2)）以及 JDK 中 List.of 提供固定重载的原因——见下一节。

8. 性能优化：固定重载覆盖常见情况

JDK 的 List.of() 源码（JDK 9+）展示了经典的优化技巧：

// 源码节选（java.util.List）
static <E> List<E> of()                          { return ImmutableCollections.emptyList(); }
static <E> List<E> of(E e1)                      { return new ImmutableCollections.List12<>(e1); }
static <E> List<E> of(E e1, E e2)               { return new ImmutableCollections.List12<>(e1, e2); }
// ... 一直到 of(e1..e10) 共 11 个固定重载

@SafeVarargs
static <E> List<E> of(E... elements)            { /* 兜底：11+ 个元素 */ }

好处：

  • 调用 List.of(1, 2) 命中固定重载，零数组分配，GC 压力更低。
  • 固定重载走重载解析第一轮，比 varargs 的第三轮解析更快。
  • 热路径（如框架初始化、高频日志）中的性能差异可被 JMH 测量出来。
何时值得做这个优化
只有当该方法处于热路径（高频调用、低参数量占多数）时才值得复制出固定重载。 普通业务代码不必如此，过度优化反而增加维护成本。

9. 三大陷阱

陷阱一：显式传 null 的歧义

static void f(String... args) {
    System.out.println(args == null ? "null" : args.length);
}

f((String[]) null); // args == null → 打印 "null"
f((String)   null); // 等价于 f(new String[]{null}) → 打印 1
f(null);             // 歧义警告：编译器选择把 null 当作 String[] 传入 → 打印 "null"

裸 null 会被编译器优先解释为 String[] 类型的 null，而不是单个 String 元素。 在方法体内必须先判断 args != null。

陷阱二：基本类型数组不自动装箱为对象数组

static void print(Object... objs) {
    for (Object o : objs) System.out.print(o + " ");
}

int[] arr = {1, 2, 3};
print(arr);        // 输出 "[I@3764951d"（整个 int[] 作为一个 Object）
print(1, 2, 3);    // 输出 "1 2 3"（各自装箱为 Integer）

int[] 不是 Object[] 的子类，它整体被当作一个 Object 传入，而不是三个元素。

陷阱三：热路径重复调用的 GC 压力

// 每次调用 validate(rules, value) 都分配 new Object[]
for (int i = 0; i < 1_000_000; i++) {
    validate(value, rule1, rule2, rule3);  // 百万次 × new Object[3]
}

// 优化：提前构造数组，复用
Object[] rules = {rule1, rule2, rule3};
for (int i = 0; i < 1_000_000; i++) {
    validate(value, rules);   // 直接传数组，不再重复分配
}

10. 知识联系

数组（Array）
varargs 编译后就是数组；理解数组内存布局才能准确评估 GC 成本。
泛型擦除
堆污染根源；擦除使运行时无法还原 T 的真实类型，数组却保留，导致不匹配。
方法重载解析
varargs 在第三轮才被考虑；与装箱规则交叉时需要仔细推导优先级。
Stream / Collections
Stream.of、List.of、Set.of 都大量使用 varargs + 固定重载优化模式。
反射
Method.invoke(obj, args) 的 Object... varargs 在反射调用中也适用堆污染警告。
printf / MessageFormat
String.format、printf 均以 Object... 实现；了解 varargs 才能理解其性能特性。

11. 面试高频考点

varargs 与数组参数有何区别？可以互相重载吗？

字节码层面两者相同（都是数组），不能共存为重载——f(int[]) 与 f(int...) 会产生"方法已定义"编译错误。区别仅在调用侧语法：varargs 允许逐一传参，普通数组参数不行。

调用 varargs 方法时传入 null 会发生什么？

裸 null 优先被解释为数组类型的 null，方法内部的 for-each 会抛出 NullPointerException。建议方法体内加 if (args == null) return; 防御，或使用 Objects.requireNonNull。

为什么泛型 varargs 会有编译警告？@SafeVarargs 做了什么？

泛型擦除后数组实际类型为 Object[]，但声明类型携带泛型信息，两者不一致即"堆污染"。@SafeVarargs 是程序员向编译器的承诺：方法不写入 / 不泄漏该数组引用，请压制警告。编译器不验证这个承诺，安全性由开发者自己保证。

varargs 和方法重载同时存在时，哪个优先？

精确匹配（第一轮）> 装拆箱（第二轮）> varargs（第三轮）。存在精确匹配时 varargs 永远不会被选中；只有在前两轮均无法匹配时，才进入 varargs 匹配。

List.of 为什么要写 11 个固定重载，而不是只用 varargs？

每次调用 varargs 方法都会在堆上分配一个新数组对象。List.of 是高频调用的工厂方法，固定重载（0~10 个元素）可消除数组分配，降低 GC 压力，对 JVM JIT 编译也更友好（可内联）。11 个元素以上才走 varargs 兜底。

int[] 传给 Object... 参数时发生了什么？

int[] 是引用类型，它本身是 Object 的子类，但不是 Object[] 的子类。因此整个 int[] 对象作为单个元素传入，方法接收到的是长度为 1 的 Object[]，其唯一元素是那个 int[] 引用。若需要逐元素传入，需先转换为 Integer[]。
