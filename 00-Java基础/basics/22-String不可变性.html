22. String 不可变性与常量池

String 是 Java 中使用最频繁的类，却藏着最多被误解的细节——不可变性的设计哲学、常量池的复用机制、intern() 的边界行为，每一条都是面试的高频地带。

1. 为什么 String 要设计成不可变？

Java 的 String 是 不可变对象（Immutable Object）。一旦创建，字符序列就永远不变。这不是偶然，而是有意为之的权衡。

原因一：安全性

网络连接、文件路径、数据库 URL——这些都是字符串。如果 String 可变，攻击者可以在权限检查之后、实际使用之前偷偷改掉字符串内容，构成安全漏洞（TOCTOU 攻击）。

原因二：线程安全零成本

不可变对象天然是线程安全的。多个线程共享同一个 String 实例，无需任何同步开销，直接复用。

原因三：支撑常量池（String Pool）

正因为内容不可变，JVM 才敢让多个引用指向同一个对象。如果 String 可变，复用就会造成"改了 A 也影响了 B"的灾难。

原因四：hashCode 可以缓存

String 内部缓存了 hash 字段，第一次计算后就不再重算。这让 String 作为 HashMap 的 key 极其高效。

设计哲学：不可变性是"一次付出（构造时复制），永久受益（安全、缓存、共享）"的典型范式。Josh Bloch 在《Effective Java》中将其列为第一推荐的设计原则。

2. String 的内部结构

看 JDK 源码（以 JDK 9+ 的 Compact Strings 实现为例）：

String.java（JDK 9+，精简）
public final class String implements Serializable, Comparable<String>, CharSequence {

    // 核心：字节数组，不是 char[]（JDK 9 引入 Compact Strings 优化）
    private final byte[] value;

    // 编码标志：LATIN1(0) 或 UTF16(1)
    private final byte coder;

    // 缓存的 hashCode，默认 0 表示未计算
    private int hash;

    // final 字段 + private 数组 = 外部无法修改内容
}

三重防护确保不可变：类是 final（不能被继承覆写）、value 数组是 private final（引用不可变）、没有任何方法暴露数组本身（内容不可修改）。

JDK 9 优化：JDK 8 及以前用 char[]（每个字符 2 字节），JDK 9 改为 byte[]。纯 ASCII 字符串只占 1 字节/字符，内存减半。这是在不改变 API 的前提下做的透明优化，也只有不可变性才能保证这种底层替换的安全性。

3. 字符串常量池（String Pool）

常量池是 JVM 维护的一块字符串缓存区，存放字符串字面量（literal）和通过 intern() 手动放入的字符串。它类似一张由 JVM 维护的、我们在 Java 代码里看不到内部结构的全局字符串表（StringTable），思路上接近 HashSet 去重，但并不是 Java 层面的 HashSet 对象，而是 JVM 内部的原生实现，保证内容相同的字符串只存一份。

内存位置的演变

JDK 6：常量池在 PermGen（永久代），大小固定，OOM 风险高 JDK 7：常量池迁移到 堆（Heap），受 GC 管理，可回收 JDK 8：PermGen 彻底移除，改用 Metaspace（本地内存），常量池仍在堆

字面量 vs new 的区别

StringPool.java
String a = "hello";          // 指向常量池中的对象
String b = "hello";          // 复用同一个池对象
String c = new String("hello"); // 强制在堆上新建对象，绕过常量池

System.out.println(a == b);   // true  — 同一个对象
System.out.println(a == c);   // false — 堆对象 vs 池对象
System.out.println(a.equals(c)); // true  — 内容相同

用内存图理解：

栈 堆 ──── ────────────────────────────── a ──────────────────────► [ String Pool: "hello" ] ◄── b c ──► [ Heap: "hello" ] (c 的内容也是 "hello"，但是独立的对象)

4. intern() 的作用与风险

intern() 是手动将字符串纳入常量池的方法：如果池中已有相同内容的字符串，返回池中的引用；否则将当前对象放入池中并返回。

intern() 演示
String c = new String("hello");  // 堆上新对象
String d = c.intern();           // 返回池中的 "hello"
String a = "hello";

System.out.println(a == d);    // true — d 现在指向池对象
System.out.println(a == c);    // false — c 仍是堆对象
风险：大量调用 intern() 会将字符串塞入常量池，JDK 7+ 虽迁移到堆上，但 StringTable 在 Minor GC 时也会被扫描，符合条件（不再被引用）的驻留字符串在 Minor GC 阶段就可能被回收，并不需要等到 Full GC；不过大量长期存活的驻留字符串仍会占用内存，日志、用户输入等动态字符串绝对不要 intern()。

5. 编译期常量折叠

编译器对纯字面量拼接会在编译期直接计算结果，因此结果也进常量池。但凡涉及变量，就会退化为运行期拼接（生成新堆对象）。

编译期折叠 vs 运行期拼接
String a = "hello";
String b = "hel" + "lo";    // 编译期折叠为 "hello"，进常量池
String prefix = "hel";
String c = prefix + "lo";  // 运行期拼接（变量参与），堆上新对象

System.out.println(a == b);  // true
System.out.println(a == c);  // false

// 特例：final 变量视为编译期常量
final String fp = "hel";
String d = fp + "lo";       // 编译期折叠，等价于 "hello"
System.out.println(a == d);  // true
原理：final 修饰的局部变量（且初始化为字面量）在字节码层面是常量，编译器可以内联它的值，因此参与的拼接也能被折叠。

6. == 与 equals 的区别

这是最高频的 String 面试陷阱，必须牢记本质区别：

比较方式 比较的是什么 适用场景
== 引用地址（是否同一个对象） 只在明确要比较引用时使用
.equals() 字符序列内容 几乎所有业务场景
.equalsIgnoreCase() 内容，忽略大小写 用户输入、命令解析
经典 NullPointerException 陷阱：str.equals("hello") 当 str 为 null 时抛 NPE。正确写法是把常量写左边："hello".equals(str)，或用 Objects.equals(str, "hello")。

7. 不可变性带来的性能问题与解决方案

正因为 String 不可变，每次"修改"都会创建新对象。循环拼接时会产生大量临时对象，造成 GC 压力：

反例：循环拼接（O(n²) 时间复杂度）
// 每次 += 都创建新的 String 对象，旧对象立刻变垃圾
String result = "";
for (int i = 0; i < 10000; i++) {
    result += i;  // 等价于 result = new String(result + i)，极慢！
}
正例：StringBuilder（O(n) 时间复杂度）
// StringBuilder 内部是可变 char[]，追加操作不产生临时对象
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append(i);
}
String result = sb.toString();  // 最后一次性转为 String

三者对比：

类 可变 线程安全 适用场景
String 否 是（天然） 常量、配置、少量拼接
StringBuilder 是 否 单线程频繁拼接（首选）
StringBuffer 是 是（synchronized） 多线程共享拼接（少见）
编译器的聪明之处：简单的 a + b + c（非循环）编译器会自动优化为 StringBuilder.append()，所以不必手动改写。只有循环内的拼接才需要显式使用 StringBuilder。

8. 关联知识：与其他核心机制的连接

与 HashMap 的关联

String 是最常用的 HashMap key，不可变性确保了 hashCode() 返回值永远稳定，且缓存机制让多次 hash 查找极快。如果用可变对象做 key 并在放入后修改其内容，hash 值变化会导致永远找不到这个 key——这是一个典型的 Map 使用陷阱。

与类加载的关联

JVM 类加载器也依赖 String 来标识类名路径，不可变性防止类名在加载过程中被篡改，保证类加载的安全性。

与 switch 的关联

Java 7 开始 switch 支持 String，底层实现先比较 hashCode()，再用 equals() 二次确认，正是利用了 String 不可变 hashCode 的特性。

9. 面试高频考点总结

Q1：String s = new String("abc") 创建了几个对象？
最多 2 个：如果常量池中已有 "abc"，则只创建堆上 1 个；如果常量池中没有，则同时在常量池创建 1 个 + 堆上 1 个，共 2 个。
Q2：为什么 String 不能被继承（final class）？
防止子类覆写方法来绕过不可变性约束（比如覆写 charAt() 返回修改后的值），确保任何 String 引用指向的对象行为一致。
Q3：s1 == s2 结果为 true/false 的临界条件？
true 的条件：都来自字面量或编译期常量折叠，或都经过 intern() 后指向池中同一对象。false 的条件：任意一个是通过 new 创建的、或是运行期拼接出的字符串。
Q4：能用反射修改 String 的内容吗？
技术上可以——通过反射访问 value 字节数组并修改，但这会破坏常量池共享机制（修改一个引用会影响所有共享这个常量的地方），属于严重的反模式，仅作面试知识点了解，生产代码绝对禁止。

10. 常见陷阱速查

  • 用 == 比较字符串内容（应用 .equals()）
  • 对可能为 null 的字符串调用 .equals()（应把常量写左边，或用 Objects.equals()）
  • 在循环中用 += 拼接字符串（应用 StringBuilder）
  • 对大量动态字符串滥用 intern()（会导致常量池膨胀）
  • 认为 new String("abc") 和 "abc" 等价（前者绕过常量池，浪费内存）
  • 修改 final String 变量后期望 == 结果改变（final 只保证引用不变，不影响常量折叠已发生的比较）
