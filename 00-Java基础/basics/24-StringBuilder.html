24. StringBuilder vs StringBuffer

可变字符序列的两兄弟——搞清线程安全的代价，才能做出正确的性能选择。

为什么 String 不够用？

上一篇讲过，String 是不可变对象。每次拼接都会在堆上创建新对象，旧对象等待 GC 回收。当拼接发生在循环里，问题就被放大了。

反例：循环里用 + 拼接Java
// 10 万次循环会创建 10 万个临时 String 对象！
String result = "";
for (int i = 0; i < 100_000; i++) {
    result += i;  // 每次都：new String(result + i)，旧对象抛弃
}

Java 设计者的解决方案：提供一个内部持有可变字符数组的类，把所有追加操作都在同一块内存上完成。这就是 StringBuilder 和 StringBuffer 共同的底层思路。

内部结构：一个可伸缩的 char 数组

两者都继承自 AbstractStringBuilder，核心字段只有两个：

AbstractStringBuilder ├── char[] value ← 真正存字符的数组（Java 9 后改为 byte[]） └── int count ← 当前已使用的字符数（不等于数组长度）

初始容量默认为 16。当 count + 新增字符数 > value.length 时，触发扩容：

AbstractStringBuilder#ensureCapacityInternal（JDK 源码逻辑）Java
void ensureCapacityInternal(int minimumCapacity) {
    if (minimumCapacity - value.length > 0)
        expandCapacity(minimumCapacity);
}

void expandCapacity(int minimumCapacity) {
    // 新容量 = 旧容量 * 2 + 2
    int newCapacity = value.length * 2 + 2;
    if (newCapacity - minimumCapacity < 0)
        newCapacity = minimumCapacity;
    value = Arrays.copyOf(value, newCapacity);
}
类比： 就像 ArrayList 的动态扩容——预先多申请空间，用完再翻倍，避免每次追加都复制整个数组。

核心区别：synchronized 的有无

两者的 API 完全相同，唯一区别在于 StringBuffer 的每个公开方法都加了 synchronized：

StringBuffer（线程安全）Java
public synchronized StringBuffer append(String str) {
    // 锁住 this，同一时刻只有一个线程能进来
    super.append(str);
    return this;
}
StringBuilder（非线程安全）Java
public StringBuilder append(String str) {
    // 没有 synchronized，直接委托父类，速度更快
    super.append(str);
    return this;
}

synchronized 会带来锁竞争开销（monitor enter/exit 指令 + 内存可见性屏障）。即使在没有竞争的单线程场景，JVM 也需要额外工作来保证语义正确性（尽管 JIT 会做锁消除优化，但代码本身的语义成本依然存在）。

String / StringBuilder / StringBuffer 三者对比

维度 String StringBuilder StringBuffer
可变性 不可变 可变 可变
线程安全 安全（天然） 不安全 安全（synchronized）
性能 拼接慢（频繁 new） 最快 比 StringBuilder 慢约 10~20%
适用场景 少量拼接、常量 单线程字符串构建 多线程共享可变字符串
引入版本 JDK 1.0 JDK 1.5 JDK 1.0
实际工程中的选择： 99% 的场景用 StringBuilder。StringBuffer 出现在需要多线程共享同一个构建器对象的极少数情况。

常用 API 演示

以下示例以 StringBuilder 展示，StringBuffer API 完全相同。

StringBuilder 常用操作Java
StringBuilder sb = new StringBuilder(64);  // 预估容量，避免扩容

// 追加各种类型，返回 this，支持链式调用
sb.append("Hello")
  .append(', ')
  .append(2024)
  .append(true);
System.out.println(sb);  // Hello, 2024true

// 在指定位置插入
sb.insert(7, "World ");
System.out.println(sb);  // Hello, World 2024true

// 删除区间 [start, end)
sb.delete(13, sb.length());
System.out.println(sb);  // Hello, World

// 反转内容
sb.reverse();
System.out.println(sb);  // dlroW ,olleH

// 替换区间内容
sb.replace(0, 5, "WORLD");
System.out.println(sb);  // WORLD ,olleH

// 获取子字符串（返回新 String）
String sub = sb.substring(7);  // "olleH"

// 最终转为不可变 String
String result = sb.toString();

编译器的秘密：+ 号背后的 StringBuilder

你可能听说过"Java 编译器会把字符串 + 优化成 StringBuilder"，这是真的，但有重要前提。

源码Java
String a = "Hello";
String b = "World";
String c = a + " " + b;  // 这一行编译后等价于下面
编译后等价代码（javap 反编译）Java
String c = new StringBuilder()
               .append(a)
               .append(" ")
               .append(b)
               .toString();
陷阱：循环内的 + 号，编译器无法合并！
每次循环迭代都会 new StringBuilder()，再 toString()，产生大量临时对象。循环里请手动用 StringBuilder。
循环里 + 的编译结果（反例揭示）Java
// 源码
String result = "";
for (String item : list) {
    result += item;
}

// 编译器实际生成（每次循环都 new！）
for (String item : list) {
    result = new StringBuilder()
                 .append(result)
                 .append(item)
                 .toString();
}

// 正确写法：把 StringBuilder 提到循环外
StringBuilder sb = new StringBuilder();
for (String item : list) {
    sb.append(item);
}
String result = sb.toString();

性能调优：预设容量

如果能预估最终字符串长度，构造时传入容量可以完全避免扩容，这是一个低成本高收益的优化习惯。

预设容量 vs 默认容量Java
// 默认容量 16，拼接长字符串时会多次扩容（每次都 Arrays.copyOf）
StringBuilder sb1 = new StringBuilder();

// 预估 512 字符，一次分配到位，避免扩容
StringBuilder sb2 = new StringBuilder(512);

// 查看当前容量（capacity）和已用长度（length）
sb2.append("abc");
System.out.println(sb2.length());    // 3（实际字符数）
System.out.println(sb2.capacity());  // 512（底层数组大小）

多线程陷阱：StringBuffer 也不是万能的

StringBuffer 保证单个方法的原子性，但复合操作仍然不是线程安全的。

复合操作的竞态条件Java
StringBuffer sb = new StringBuffer("Hello");

// 线程 A 和线程 B 同时执行以下操作：
if (sb.length() < 10) {       // ← check（原子）
    sb.append(" World");    // ← act（原子）
}
// check 和 act 之间没有锁保护，仍可能被打断！
// 真正的线程安全需要在更高层级加锁
结论： 如果真的需要多线程操作字符串，与其用 StringBuffer，不如在外层加 synchronized 块或使用 Lock，语义更清晰，控制粒度更准确。

现代替代方案

JDK 8+ 提供了更高级别的 API，底层仍然基于 StringBuilder，但代码更简洁：

String.join / StringJoiner / StreamJava
import java.util.StringJoiner;
import java.util.stream.Collectors;
import java.util.List;

List<String> words = List.of("a", "b", "c");

// 方式 1：String.join（最简单）
String r1 = String.join("-", words);  // "a-b-c"

// 方式 2：StringJoiner（可指定前缀/后缀）
StringJoiner sj = new StringJoiner(", ", "[", "]");
words.forEach(sj::add);
System.out.println(sj);  // [a, b, c]

// 方式 3：Stream.collect（最灵活，可先过滤/转换）
String r3 = words.stream()
               .filter(w -> !w.equals("b"))
               .collect(Collectors.joining("+"));  // "a+c"

面试高频考点

  • Q：String、StringBuilder、StringBuffer 的区别？
    String 不可变；StringBuilder 可变但非线程安全；StringBuffer 可变且线程安全（方法加 synchronized）。性能：StringBuilder > StringBuffer > 循环拼接 String。
  • Q：为什么 Java 5 才引入 StringBuilder？
    JDK 1.0 就有 StringBuffer，但 synchronized 在单线程下有不必要开销。JDK 5 引入 StringBuilder 作为单线程高性能替代。
  • Q：编译器会把 + 优化成 StringBuilder，还需要手动用吗？
    单行表达式的 + 会被优化，但循环体内的 + 每次迭代都会 new StringBuilder，必须手动提到循环外。
  • Q：StringBuffer 是线程安全的，多线程下可以放心使用吗？
    单方法调用原子，但多步复合操作（check-then-act）仍有竞态条件，需要在更高层级加锁。
  • Q：如何避免 StringBuilder 频繁扩容？
    构造时传入预估容量，如 new StringBuilder(256)。
  • Q：StringBuilder 的 reverse() 能正确处理 Unicode 代理对吗？
    JDK 源码中 reverse() 会检测并保护代理对，不会拆散 emoji 等增补字符。
记忆口诀： 单线程用 Builder，多线程用 Buffer，现代代码优先 String.join / Stream。

知识联系

  • String 不可变 → 是使用 StringBuilder 的根本动机，详见上一篇
  • synchronized 关键字 → StringBuffer 的线程安全机制，将在并发章节深讲
  • JIT 锁消除优化 → 局部变量 StringBuffer 不会逃逸，JIT 可消除锁，但这是运行时行为，代码层面仍应选 StringBuilder
  • AbstractStringBuilder → 两者的共同父类，是模板方法模式的典型应用
  • CharSequence 接口 → String、StringBuilder、StringBuffer 都实现了它，是多态接受字符序列的通用类型
