18. 递归

方法调用自身——一把拆解复杂问题的利刃，也是调用栈溢出的常见凶手。

1. 什么是递归，为什么需要它

递归（Recursion）是指一个方法在执行过程中直接或间接地调用自身。 它的本质是把"大问题"拆成"形状相同但规模更小的子问题"，直到子问题小到可以直接回答为止。

生活类比：俄罗斯套娃。要打开最外层娃娃，你就得打开下一层； 要打开下一层，还得打开再下一层……直到摸到最里面那个实心娃娃（不需要再打开）才停下来。 "摸到实心娃娃"就是递归的终止条件（Base Case）。

为什么要有递归？
某些问题（树遍历、分治、回溯、数学归纳）天然具有自相似结构，用循环描述时需要手动维护栈， 逻辑繁琐；递归让代码结构与问题结构一一对应，可读性大幅提升。

2. 递归的两大要素

任何正确的递归都必须同时满足：

  • Base Case（终止条件）：问题规模足够小时直接返回结果，不再递归。
  • Recursive Case（递归体）：把当前问题分解为规模更小的子问题，并调用自身。

缺少终止条件，或终止条件永远无法到达，都会导致无限递归直至 StackOverflowError。

3. 入门示例：阶乘

数学定义：n! = n × (n-1)!，且 0! = 1。它天然就是递归结构。

Factorial.java
public class Factorial {

    // 递归版本：代码与数学定义完全吻合
    public static long factorial(int n) {
        // ① Base Case：最小子问题，直接给答案
        if (n == 0) return 1;
        // ② Recursive Case：大问题 → 小问题
        return n * factorial(n - 1);
    }

    public static void main(String[] args) {
        System.out.println(factorial(5)); // 120
    }
}

调用 factorial(4) 时，调用栈的展开与收缩过程如下：

factorial(4) ├── 4 × factorial(3) │ ├── 3 × factorial(2) │ │ ├── 2 × factorial(1) │ │ │ └── 1 × factorial(0) → return 1 ← Base Case │ │ └── return 2 │ └── return 6 └── return 24

关键洞察：调用阶段不断把问题压入栈，返回阶段才真正计算结果。 每一帧都在内存（Java 虚拟机栈）中占据一个栈帧，直到 Base Case 触发后才逐层弹出。

4. 递归与 JVM 调用栈

每次方法调用，JVM 都会在线程栈上分配一个栈帧（Stack Frame）， 存储局部变量、操作数栈、返回地址等信息。递归就是不断往栈顶追加栈帧。

JVM 默认线程栈大小约为 512 KB ～ 1 MB（不同平台不同）。 一旦栈帧总数超出限制，就抛出 StackOverflowError。 可以用 -Xss2m 启动参数调大，但这只是治标不治本。

陷阱：递归深度与输入规模成正比时要格外小心。 计算 factorial(20000) 会有约 20000 个栈帧同时存活，几乎必然 StackOverflowError。

5. 经典陷阱：朴素 Fibonacci 的指数爆炸

斐波那契数列定义：fib(n) = fib(n-1) + fib(n-2)，fib(0)=0, fib(1)=1。

朴素递归（反例）
// 看起来优雅，实际上是 O(2^n) 时间复杂度
public static long fib(int n) {
    if (n <= 1) return n;
    // 问题：fib(n-2) 和 fib(n-1) 大量重复计算同一子问题
    return fib(n - 1) + fib(n - 2);
}

计算 fib(5) 时，fib(3) 被计算了 2 次，fib(2) 被计算了 3 次……重复工作呈指数增长。

解法 1：记忆化递归（Memoization）

记忆化递归（正例）
// 用 Map 缓存已算过的结果，时间降为 O(n)
private static Map<Integer, Long> memo = new HashMap<>();

public static long fibMemo(int n) {
    if (n <= 1) return n;
    if (memo.containsKey(n)) return memo.get(n);
    long result = fibMemo(n - 1) + fibMemo(n - 2);
    memo.put(n, result);
    return result;
}

解法 2：迭代（彻底消除递归）

迭代版本（O(n) 时间，O(1) 空间）
public static long fibIter(int n) {
    if (n <= 1) return n;
    long a = 0, b = 1;
    for (int i = 2; i <= n; i++) {
        long tmp = a + b;
        a = b;
        b = tmp;
    }
    return b;
}

6. 尾递归与 Java 的尴尬

尾递归指递归调用是方法体中最后一个操作，返回值直接来自子调用， 不需要"回来"再做任何计算。理论上编译器可以把尾递归优化成循环，从而不增长调用栈。

尾递归形式的阶乘
// acc 是累积器，代替"返回后再乘"
public static long factTail(int n, long acc) {
    if (n == 0) return acc;
    // 最后一步只是调用自身，不再需要当前栈帧的任何数据
    return factTail(n - 1, n * acc);
}
// 调用：factTail(5, 1)
Java 不做尾递归优化（TCO）！
JVM 规范没有要求 TCO，Hotspot JVM 也没实现它（这与方便 debug 时能看到完整调用栈有关）。 因此在 Java 里写尾递归与普通递归在栈消耗上没有任何区别——深度一样，依然会 StackOverflow。 Scala/Kotlin 的 @tailrec 注解由编译器在字节码层面转换为循环，Java 没有等价物。

7. 递归的舞台：二叉树遍历

树结构天然是递归定义的：一棵树 = 根节点 + 左子树 + 右子树， 而左/右子树本身也是树。用递归遍历比用手动栈清晰十倍。

BinaryTree.java — 前序/中序/后序遍历
class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}

public class BinaryTree {

    // 前序：根 → 左 → 右
    public static void preOrder(TreeNode node) {
        if (node == null) return; // Base Case
        System.out.print(node.val + " ");
        preOrder(node.left);
        preOrder(node.right);
    }

    // 中序：左 → 根 → 右（BST 中序 = 升序排列）
    public static void inOrder(TreeNode node) {
        if (node == null) return;
        inOrder(node.left);
        System.out.print(node.val + " ");
        inOrder(node.right);
    }

    // 后序：左 → 右 → 根（计算目录大小时常用）
    public static void postOrder(TreeNode node) {
        if (node == null) return;
        postOrder(node.left);
        postOrder(node.right);
        System.out.print(node.val + " ");
    }
}

8. 分治递归：归并排序骨架

分治（Divide & Conquer）是递归最重要的应用模式： 把数组一分为二分别排序，再合并两个有序数组。

MergeSort.java（骨架）
public static void mergeSort(int[] arr, int lo, int hi) {
    // Base Case：区间长度 <= 1，已有序，无需处理
    if (lo >= hi) return;

    int mid = lo + (hi - lo) / 2;
    mergeSort(arr, lo, mid);      // 递归左半
    mergeSort(arr, mid + 1, hi);  // 递归右半
    merge(arr, lo, mid, hi);        // 合并两段有序区间
}

归并排序时间复杂度 O(n log n)，递归深度为 O(log n)， 对 100 万数据也只有约 20 层递归，栈消耗极低。

9. 递归 vs 迭代：如何选择

维度 递归 迭代
可读性 问题本身具有递归结构时极高 线性逻辑时更直观
栈消耗 O(深度) 的额外栈帧 通常 O(1)（显式栈除外）
性能 函数调用开销，可能有重复计算 通常更快
适用场景 树/图遍历、分治、回溯、DFS 简单线性遍历、大深度场景
Java 中的风险 深度大时 StackOverflowError 无此风险
实践原则：如果问题结构天然是递归的（树、图、分治），优先用递归写出清晰版本； 若递归深度可能超过几千层，或性能是瓶颈，再考虑改写为迭代 + 显式栈。

10. 常见陷阱汇总

陷阱 1：忘写 Base Case 或条件写错

反例 — 死循环递归
// n 永远不会等于 0，因为这里写的是 n != 1
public static long bad(int n) {
    if (n == 1) return 1;  // 传入 0 时不命中！
    return n * bad(n - 1);  // bad(0) → bad(-1) → ... → StackOverflow
}

陷阱 2：对同一子问题重复计算（见 Fibonacci 章节）

解决方案：记忆化（Memoization）或改为动态规划（DP）。

陷阱 3：传引用对象时的副作用

回溯时忘记撤销操作
List<Integer> path = new ArrayList<>();

void backtrack(int[] nums, int start) {
    result.add(new ArrayList<>(path)); // 快照，不能直接 add(path)
    for (int i = start; i < nums.length; i++) {
        path.add(nums[i]);
        backtrack(nums, i + 1);
        path.remove(path.size() - 1); // 关键：撤销，恢复现场
    }
}

陷阱 4：整数溢出

递归计算阶乘、幂次时，结果很快超出 int（约 2.1×10⁹）范围。 要及时改用 long 或 BigInteger。

11. 与 Java 生态的联系

  • 集合框架：TreeMap、TreeSet 底层是红黑树，树的很多操作（旋转、查找前驱/后继）都用递归实现。
  • JSON 解析 / XML 解析：嵌套结构天然递归，Jackson、Gson 在处理嵌套对象时内部有递归调用。
  • Spring AOP / 代理链：拦截器链本质上是递归调用 proceed()，每一层拦截器决定是否继续往下。
  • JDK Files.walkFileTree：用访问者模式 + 递归遍历目录树，是生产代码里最典型的递归应用之一。
  • 动态规划：DP 可以看作"记忆化递归的自底向上改写"，理解了递归才能真正理解 DP 的状态转移方程。

12. 面试高频考点

面试官最爱问的几个点：
  • 递归的两个必要条件是什么？没有 Base Case 会发生什么？
  • 说说 StackOverflowError 的原因，如何避免？
  • Java 支持尾递归优化吗？为什么不支持？
  • 朴素递归 Fibonacci 的时间复杂度是多少？如何优化到 O(n)？
  • 递归和迭代分别适合什么场景？能把递归转为迭代吗？（手动用栈模拟）
  • 写出二叉树的中序遍历（递归版 + 迭代版）。

手写：递归转迭代（以前序遍历为例）

迭代版前序遍历（用显式 Deque 模拟系统栈）
public static List<Integer> preOrderIter(TreeNode root) {
    List<Integer> res = new ArrayList<>();
    if (root == null) return res;
    Deque<TreeNode> stack = new ArrayDeque<>();
    stack.push(root);
    while (!stack.isEmpty()) {
        TreeNode node = stack.pop();
        res.add(node.val);
        // 先压右子，再压左子，弹出时就是左先右后
        if (node.right != null) stack.push(node.right);
        if (node.left  != null) stack.push(node.left);
    }
    return res;
}

规律：系统栈 → 显式栈。把递归调用压栈的隐式行为， 改写为手动管理 Deque，就完成了递归到迭代的转换。 这个思路可以推广到任何递归问题。
