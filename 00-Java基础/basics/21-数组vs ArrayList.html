数组 vs ArrayList 的取舍

两者都能存储一组数据，但底层设计目标截然不同——理解这一点，你才能在性能敏感的场景做出正确选择。

1. 先建立直觉：两种不同的"货架"

想象一排固定格子的货架（数组）：格子数量建好后不能增减，但你知道第 N 格在哪里，一步就能拿到。再想象一排能自动伸缩的手风琴货架（ArrayList）：随时可以插入或删除格子，但每次"拉伸"都要整体挪动。

这不是比喻，而是内存里真实发生的事情。

数组（连续内存，固定长度） ┌─────┬─────┬─────┬─────┬─────┐ │ 10 │ 20 │ 30 │ 40 │ 50 │ └─────┴─────┴─────┴─────┴─────┘ [0] [1] [2] [3] [4] ← O(1) 随机访问，地址 = base + index × elementSize ArrayList（内部维护一个 Object[] elementData） ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │ 10 │ 20 │ 30 │ 40 │ 50 │ null│ null│ null│ └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘ size=5，capacity=8 ← 预留空间避免频繁扩容

2. 底层原理：为什么这么设计？

2.1 数组：CPU 最爱的数据结构

数组在内存中连续存储。连续意味着：

  • 随机访问 O(1)：第 i 个元素的地址 = base + i × sizeof(element)，一次乘加即可，无需遍历。
  • Cache 友好：CPU 预取机制会把相邻内存一起载入缓存行（Cache Line，通常 64 字节），顺序遍历数组时命中率极高，比链表快 5~10 倍不罕见。
  • 零装箱开销：int[] 直接存 4 字节整数，ArrayList<Integer> 存的是堆上 Integer 对象的引用（8 字节），对象本身还要额外 16 字节对象头，内存放大约 6 倍。

代价是：长度在创建时必须确定，无法动态扩缩。

2.2 ArrayList：用空间换灵活性

ArrayList 内部持有一个 Object[] elementData。当元素数量超过容量时，它会申请一块更大的数组（JDK 8 扩容因子约 1.5 倍），把旧数组内容复制过去，再释放旧数组。

ArrayList 扩容核心逻辑（JDK 17 简化版）Java
// java.util.ArrayList#grow
private Object[] grow(int minCapacity) {
    int oldCapacity = elementData.length;
    // 新容量 = 旧容量 + 旧容量 >> 1，即约 1.5 倍
    int newCapacity = oldCapacity + (oldCapacity >> 1);
    if (newCapacity - minCapacity < 0)
        newCapacity = minCapacity;
    // Arrays.copyOf 底层调用 System.arraycopy，native 方法，但仍是 O(n)
    return elementData = Arrays.copyOf(elementData, newCapacity);
}

扩容是 O(n) 操作，但均摊到每次 add 上是 O(1)（摊还分析）。不过在最坏情况下（恰好触发扩容的那次 add）会有明显的延迟峰值，对实时系统是个风险点。

3. 代码对比：同一需求的两种写法

反例：用 ArrayList 做固定大小的矩阵运算

反例：装箱 + 二维 ArrayList，性能差Java
// 100×100 矩阵，每个元素都是 Integer 对象，内存爆炸
List<List<Integer>> matrix = new ArrayList<>();
for (int i = 0; i < 100; i++) {
    List<Integer> row = new ArrayList<>();
    for (int j = 0; j < 100; j++) {
        row.add(0);  // 0 被自动装箱为 Integer.valueOf(0)
    }
    matrix.add(row);
}
问题：10000 个 Integer 对象散落堆上，CPU 访问时频繁 Cache Miss；加上对象头开销，内存是原始 int[][] 的 6 倍以上。

正例：用原生数组，连续内存 + 零装箱

正例：int[][]，内存紧凑，Cache 友好Java
// 尺寸固定，直接用二维数组——简单、快速、节省内存
int[][] matrix = new int[100][100];
// 行优先遍历：连续读内存，L1 Cache 命中率接近 100%
for (int i = 0; i < 100; i++)
    for (int j = 0; j < 100; j++)
        matrix[i][j] = i * 100 + j;

正例：动态增删时用 ArrayList，并预估容量避免扩容

正例：知道大概数量时提前 initialCapacityJava
// 已知结果集大约 500 条，避免从默认容量 10 开始多次扩容
List<String> results = new ArrayList<>(512);

// 读文件时行数未知，边读边 add——ArrayList 最典型的使用场景
List<String> lines = new ArrayList<>();
try (BufferedReader br = new BufferedReader(new FileReader("data.txt"))) {
    String line;
    while ((line = br.readLine()) != null)
        lines.add(line);
}
// 收集完毕后如需传给只接受 List 的 API，直接用；
// 如需转数组（如排序后给 native 方法），toArray() 一步完成
String[] arr = lines.toArray(new String[0]);

5. 深入陷阱：为什么不能 new T[]？

Java 泛型是编译期擦除的——运行时 JVM 不知道 T 是什么类型，因此无法在堆上创建正确类型的数组对象。

泛型数组：编译报错 vs 正确绕过方式Java
// 错误：编译报 "Cannot create a generic array of T"
public class Stack<T> {
    private T[] data = new T[16];  // 编译报错！
}

// 正确方式一：强转（会产生 unchecked 警告，但运行时安全）
@SuppressWarnings("unchecked")
private T[] data = (T[]) new Object[16];

// 正确方式二：传入 Class<T> 令牌，反射创建
private T[] data;
public Stack(Class<T> clazz) {
    @SuppressWarnings("unchecked")
    T[] arr = (T[]) Array.newInstance(clazz, 16);
    this.data = arr;
}

// 最简方式：直接用 ArrayList<T>，省心
private List<T> data = new ArrayList<>();
数组协变（Object[] a = new String[3]）在运行时是安全检查的——如果你往里写入非 String 对象，会抛 ArrayStoreException。而 List 的不变性（List<Object> 不能接收 List<String>）则是编译期就拒绝，更早暴露问题。这是 Java 类型系统设计取舍的经典案例。

7. 决策树：该用哪个？

需要存一组数据 │ ├─ 基本类型（int/long/double）且不需要动态增删？ │ └─ 用原生数组（int[] / long[] / double[]） │ ├─ 元素数量编译期已知、运行期不变（如固定协议字段、查找表）？ │ └─ 用原生数组，配合 Arrays 工具类操作 │ ├─ 需要动态增删 / 不知道数量 / 要用 Collection API？ │ └─ 用 ArrayList（知道大概数量就传 initialCapacity） │ ├─ 需要频繁头部插入/删除（如队列/栈）？ │ └─ 考虑 LinkedList 或 ArrayDeque（另一节讲） │ └─ 作为方法返回值 / API 边界？ └─ 优先返回 List<T>（接口），便于调用方替换实现

8. 与其他知识的联系

  • Arrays 工具类（上一节）：Arrays.sort()、Arrays.binarySearch() 直接操作数组，底层用 DualPivotQuicksort，比 Collections.sort(List) 少一层装箱。
  • Stream API：IntStream / LongStream 可以在不装箱的情况下对基本类型数组做函数式操作，兼顾可读性与性能。
  • System.arraycopy：ArrayList 扩容、add(index, e)、remove(index) 底层都调用这个 native 方法——它比手写循环快，但仍是 O(n)。
  • var 与泛型：Java 10 的 var 可以推断 ArrayList<String>，但声明 API 参数时仍应写 List<String>，面向接口编程。
  • 并发：两者都不是线程安全的。并发场景用 CopyOnWriteArrayList（读多写少）或手动加锁，不要用已废弃的 Vector。

9. 常见陷阱

陷阱 1：Arrays.asList() 返回的 List 不能增删
List<String> list = Arrays.asList("a", "b"); 返回的是固定大小的视图，调用 add() 会抛 UnsupportedOperationException。需要可变 List 要 new ArrayList<>(Arrays.asList(...))。
陷阱 2：for-each 中 remove 元素
对 ArrayList 用 for-each 遍历时调用 list.remove()，会抛 ConcurrentModificationException。应用 Iterator.remove() 或改用 removeIf()（Java 8+）。
陷阱 3：忽视 initialCapacity
默认容量 10，如果你要 add 10000 个元素，会触发约 14 次扩容（每次 O(n) 复制）。已知数量级时一定要传 new ArrayList<>(expectedSize)。
陷阱 4：数组协变导致运行时异常
Object[] objs = new String[2]; objs[0] = 42; 编译通过，运行时抛 ArrayStoreException。这是数组协变的历史遗留问题，使用时要小心。

10. 面试高频考点

  1. ArrayList 扩容机制：初始容量 10，每次扩容为原来的 1.5 倍（oldCapacity + (oldCapacity >> 1)），底层调用 Arrays.copyOf（即 System.arraycopy）。
  2. 为什么 ArrayList 不能存基本类型：泛型擦除后只能存 Object，基本类型不是 Object，必须装箱。
  3. 数组和 ArrayList 的时间复杂度对比：随机访问都是 O(1)；末尾增删 ArrayList 均摊 O(1)；中间插入删除都是 O(n)。
  4. Arrays.asList 的坑：返回的是固定大小 List，不支持 add/remove，面试中非常高频。
  5. 如何将 List 转 Array：list.toArray(new T[0])，传 new T[0] 比 new T[list.size()] 在 JVM 上更快（JVM 会优化 size 为 0 的 toArray 调用）。
  6. 什么场景必须用数组而不是 ArrayList：存基本类型且量大（避免装箱）、多维数值计算（矩阵、图像处理）、实现底层数据结构（手写 HashMap/Stack）。
核心结论：默认用 ArrayList，它 API 丰富、更安全；当你明确知道"大小固定"或"存基本类型且性能敏感"时，切回原生数组。不要因为"数组更底层"就盲目用数组，也不要因为"ArrayList 更方便"就在热点路径上忍受装箱开销。
