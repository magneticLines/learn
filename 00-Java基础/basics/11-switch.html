11. switch（传统 / switch 表达式 JDK14+）

从 C 时代的"跳转表"到 JDK 14 的 switch 表达式，理解设计演进背后的每一个取舍。

一、为什么需要 switch——它解决了什么问题？

当你需要根据同一个变量的多个离散值走不同分支时，连串的 if-else if 虽然能工作，但有两个痛点：

  • 可读性差：每次都要把变量名写出来，阅读时大脑负担重。
  • 性能次优：多路 if-else if 是线性扫描，每条分支都要比较一次。

switch 的底层由 JVM 指令 tableswitch（值连续时，O(1) 跳转表）或 lookupswitch（值稀疏时，O(log n) 二分）实现， 理论上比等价的 if-else if 快——但现代 JIT 会做优化，实际差距通常可忽略，可读性才是第一选择理由。

二、传统 switch 语法（JDK 1.0 起）

传统 switch 基本结构Java
int day = 3;
switch (day) {
    case 1:
        System.out.println("周一");
        break;   // ← 不写 break 会 fall-through！
    case 2:
        System.out.println("周二");
        break;
    case 3:
        System.out.println("周三");
        break;
    default:
        System.out.println("其他");
}

支持的类型

传统 switch 的括号里可以放什么，是面试常问点：

类型从哪个版本支持备注
byte short char intJDK 1.0本质都是整数
Byte Short Character IntegerJDK 5自动拆箱为对应基本类型
enumJDK 5编译器把枚举值转成序号整数
StringJDK 7编译器用 hashCode() + equals() 两步匹配
long float double❌ 不支持JVM 跳转表只支持 int 范围
为什么不支持 long？
JVM 的 tableswitch/lookupswitch 指令的 case 标签是 32-bit int。long 有 64 位，无法直接用跳转表索引，因此语言规范直接禁止。

三、fall-through：最经典的陷阱

传统 switch 每个 case 执行完后，不会自动跳出，而是继续执行下一个 case 的代码——这叫 fall-through（贯穿）。 这是从 C 语言继承来的设计：最初目的是允许多个 case 共用同一段代码。

fall-through 反例Java
// 错误示范：忘写 break，case 2 的代码也会执行！
int x = 1;
switch (x) {
    case 1:
        System.out.println("one");   // 打印 "one"
        // 没有 break！
    case 2:
        System.out.println("two");   // 也会打印 "two"！
        break;
}
fall-through 正当用法：多值共享分支Java
// 合法利用 fall-through：工作日 vs 周末
switch (day) {
    case 1:
    case 2:
    case 3:
    case 4:
    case 5:
        System.out.println("工作日");
        break;
    case 6:
    case 7:
        System.out.println("周末");
        break;
}
最佳实践：除了"多个空 case 共用一个 break"这种形式，其他情况的 fall-through 都应该加上 // fall through 注释（IntelliJ IDEA、Checkstyle 都会检查）。

四、switch 表达式（JDK 14 正式，JDK 12/13 预览）

传统 switch 有三大痛点：fall-through 陷阱、不能作为表达式赋值、冗余的 break。 JDK 14 引入 switch 表达式，用两种新语法彻底解决了这些问题。

新语法一：箭头 case（→）

箭头 case——每个分支自动隔离，不 fall-throughJava 14+
int day = 3;
String type = switch (day) {
    case 1, 2, 3, 4, 5 -> "工作日";   // 逗号分隔多值，无需 break
    case 6, 7             -> "周末";
    default               -> "未知";
};
System.out.println(type);   // 工作日

新语法二：yield 关键字（多行 case 块返回值）

yield——在块中返回值Java 14+
int score = 85;
String grade = switch (score / 10) {
    case 10, 9 -> "A";
    case 8     -> "B";
    case 7     -> "C";
    case 6     -> "D";
    default    -> {
        // 多行逻辑时用块，必须用 yield 返回值
        System.out.println("不及格：分数 = " + score);
        yield "F";   // ← yield 是新增关键字，专门为此而生
    }
};
为什么用 yield 而不是 return？
return 语义是"从方法返回"，而 yield 语义是"从 switch 块产出一个值"。 两者作用域不同。若复用 return，在 switch 表达式嵌套于方法中时会造成歧义。

五、传统 switch 语句 vs switch 表达式对比

维度传统 switch（语句）switch 表达式（JDK14+）
可作为表达式赋值❌ 不行✅ 可直接赋值给变量
fall-through⚠️ 默认贯穿，需手动 break✅ 箭头 case 不贯穿
多值匹配一个分支多个空 case 堆叠case 1, 2, 3 -> 逗号分隔

代码量多（break / 重复变量名）少，更简洁

六、与 enum 结合——穷举性检查的威力

switch 表达式与 enum 搭配时，编译器会检查是否覆盖了所有枚举值，新增枚举后漏写分支会直接报编译错误，这是传统 if-else 做不到的。

enum + switch 表达式：漏写 default 但穷举了所有值，编译通过Java 14+
enum Season { SPRING, SUMMER, AUTUMN, WINTER }

Season s = Season.SUMMER;
String desc = switch (s) {
    case SPRING -> "春暖花开";
    case SUMMER -> "骄阳似火";
    case AUTUMN -> "金风送爽";
    case WINTER -> "白雪皑皑";
    // 四个值全覆盖，不需要 default，编译器满意
    // 若后来新增 Season.MONSOON，此处立刻编译报错！
};

七、String switch 的底层原理

JDK 7 开始支持对 String 做 switch，但 JVM 字节码仍然只认整数跳转表。 编译器会将其编译成两次 switch：

第一次 switch：对 hashCode() 做 tableswitch/lookupswitch └─ 找到候选 case 后，用 equals() 做精确匹配（防哈希碰撞） 第二次 switch：对 equals() 的结果（一个内部 int 索引）做真正的分支跳转

这意味着 String switch 有一次哈希计算 + 一次字符串比较的开销，性能略低于 int switch，但通常可以忽略。

陷阱：null 会 NullPointerException
无论传统还是新式 switch，若传入 null 都会抛 NPE。 JDK 21 的模式匹配 switch（Preview）才原生支持 case null。

八、展望：模式匹配 switch（JDK 21 正式）

JDK 21 将 switch 的能力再次扩展，支持类型模式匹配，可以替代大量 instanceof + 强转的代码：

模式匹配 switch（JDK 21）Java 21
// 旧写法：丑陋的 instanceof 链
if (obj instanceof String s) {
    System.out.println(s.length());
} else if (obj instanceof Integer i) {
    System.out.println(i * 2);
}

// 新写法：switch 模式匹配，清晰且安全
switch (obj) {
    case String s  -> System.out.println(s.length());
    case Integer i -> System.out.println(i * 2);
    case null      -> System.out.println("null");
    default        -> System.out.println("其他");
}

九、switch 版本演进时间线

JDK 1.0 ── 传统 switch 语句（int / char 等整型） JDK 5 ── 支持 enum（编译为序号整数） JDK 7 ── 支持 String（hashCode + equals 两步匹配） JDK 12 ── switch 表达式（Preview）：→ 箭头 case JDK 13 ── switch 表达式（Preview）：yield 关键字引入 JDK 14 ── switch 表达式 正式发布（JEP 361） JDK 21 ── 模式匹配 switch 正式发布（JEP 441）：case 类型模式 / case null

十、常见陷阱与面试考点

陷阱 1：忘写 break 导致 fall-through

传统 switch 最高频 Bug。新代码优先使用箭头 case 彻底规避。

陷阱 2：switch 传入 null 导致 NPE

null 导致 NPEJava
String s = null;
switch (s) {   // ← 直接 NullPointerException
    case "a" -> ...
}
// 解决：switch 之前先判空，或升级到 JDK 21 使用 case null

陷阱 3：case 值必须是编译期常量

case 后面只能是字面量、枚举常量、或 final 基本类型/String 变量，不能是运行时变量或方法调用。

编译期常量要求Java
final int A = 1;       // ✅ 编译期常量，可用于 case
int       B = 1;       // ❌ 运行时变量，case B → 编译错误
int       C = foo();   // ❌ 方法返回值，同上

陷阱 4：switch 表达式必须穷举或有 default

当 switch 表达式用于赋值时，编译器要求所有可能值必须有对应分支（否则变量可能未被赋值）。 未穷举会报编译错误，这是比传统 switch 更严格的地方，也是优点。

面试高频考点

  • switch 支持哪些类型？为什么不支持 long？（JVM tableswitch 指令只有 32 位）
  • 什么是 fall-through？如何避免？（break / 箭头 case）
  • yield 和 return 的区别？（作用域不同：yield 从 switch 块产出值，return 从方法返回）
  • switch 表达式与 switch 语句的核心区别？（表达式有返回值、无 fall-through、编译器穷举检查）
  • String switch 的底层实现？（两次 switch：hashCode 定位 + equals 精确匹配）
  • JDK 版本演进：String 支持（JDK 7）、switch 表达式正式（JDK 14）、模式匹配（JDK 21）
