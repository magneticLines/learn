15 · 值传递机制

核心结论（先记住这一句话）：
Java 中方法参数传递只有一种方式——值传递（pass by value）。
传基本类型，传的是数据值的副本；传引用类型，传的是地址值的副本。
无论哪种，方法内部都操作的是副本，永远无法让原变量指向另一个对象。

1. 内存模型：变量里究竟存的是什么？

要彻底搞懂值传递，先要明白变量在内存里保存的内容：

基本类型变量 引用类型变量 ┌──────────┐ ┌──────────┐ 堆（Heap） │ int x │ │ Person p │ ┌─────────────────┐ │ value:5 │ │ value:0x7f│ ───► │ name="Alice" │ └──────────┘ └──────────┘ │ age=30 │ 存的是数据本身 存的是对象的地址 └─────────────────┘

方法调用时，JVM 会把变量的内容复制一份压入新的栈帧。这就是"值传递"：传递的是那份内容的副本。

2. 场景一：传基本类型

public class Demo {
    // 尝试在方法里修改 x，看调用方是否感知
    static void doubleIt(int x) {
        x = x * 2;          // 修改的是副本，不影响原变量
        System.out.println("方法内 x = " + x); // 10
    }

    public static void main(String[] args) {
        int a = 5;
        doubleIt(a);
        System.out.println("方法后 a = " + a); // 仍然是 5
    }
}

调用 doubleIt(a) 时，JVM 把 a 的值 5 复制给参数 x，两者从此互不干扰。

3. 场景二：传引用类型（重点，有两个子场景）

3-A：修改字段值 — 调用方可以感知

static class Person {
    String name;
    Person(String name) { this.name = name; }
}

static void rename(Person p) {
    p.name = "Bob";    // 通过地址副本找到同一个对象，修改其字段
}

public static void main(String[] args) {
    Person person = new Person("Alice");
    rename(person);
    System.out.println(person.name); // "Bob" ← 调用方看到了变化
}
参数 p 和变量 person 持有相同的地址值，都指向堆上同一个对象。 通过 p 修改对象的字段，当然对 person 可见。

3-B：重新赋值 — 调用方感知不到

static void replacePerson(Person p) {
    p = new Person("Charlie"); // p 指向了新对象，但 person 变量没变
}

public static void main(String[] args) {
    Person person = new Person("Alice");
    replacePerson(person);
    System.out.println(person.name); // 仍然是 "Alice"
}
p = new Person(...) 只是让参数 p（地址副本）指向了另一个对象， 原变量 person 存的地址完全没变。这就是"无法传引用本身"的含义。

4. 内存图解：两种场景的栈/堆变化

【场景 3-A：修改字段】 调用前 调用后（rename 执行完） 栈 堆 栈 堆 ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ person │→ │name=Alice│ │ person │→ │name=Bob │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ ↑ 同一对象，字段被修改了 进入 rename 时： ┌──────────┐ │ p(副本) │→ 同一个堆对象 p.name="Bob" → 对象被改 └──────────┘ 【场景 3-B：重新赋值】 进入 replacePerson 时： ┌──────────┐ ┌──────────┐ │ person │→ │name=Alice│ ← 原变量，始终指这里 └──────────┘ └──────────┘ ┌──────────┐ │ p(副本) │→ 同上（初始） └──────────┘ ↓ p = new Person("Charlie") ┌──────────┐ ┌────────────┐ │ p(副本) │→ │name=Charlie│ ← p 指向新对象 └──────────┘ └────────────┘ person 仍然指向 Alice 那个对象，毫无感知

5. String 的特殊性

String 是引用类型，但由于它是不可变类（immutable），表现上和基本类型非常像：

static void changeStr(String s) {
    s = s + "!";   // 实际上创建了新的 String 对象，原对象不变
}

public static void main(String[] args) {
    String str = "hello";
    changeStr(str);
    System.out.println(str); // "hello"，没有变
}
s + "!" 在堆上创建了新的 String 对象，参数 s 指向新对象， 但 str 的地址副本没有变，所以 str 仍然是 "hello"。
本质还是场景 3-B，只是 String 的不可变性让它"看起来"和基本类型行为一致。

6. 数组示例：fillZero vs replaceArray

// 可见：通过地址副本修改数组元素
static void fillZero(int[] arr) {
    for (int i = 0; i < arr.length; i++) arr[i] = 0;
}

// 不可见：让参数指向新数组
static void replaceArray(int[] arr) {
    arr = new int[]{9, 9, 9}; // 只改了副本，调用方感知不到
}

public static void main(String[] args) {
    int[] a = {1, 2, 3};

    fillZero(a);
    System.out.println(Arrays.toString(a)); // [0, 0, 0] ← 可见

    replaceArray(a);
    System.out.println(Arrays.toString(a)); // [0, 0, 0] ← 仍然是 fillZero 的结果，replaceArray 无效
}

7. 与其他语言的对比

语言 基本类型 引用/对象类型 能否在方法内改变调用方的变量指向？
Java 值传递 地址值的值传递 ❌ 不能
C++ 值传递（默认） 可选引用传递 & ✅ 用引用参数可以
C# 值传递（默认） 可用 ref / out ✅ 加 ref 关键字可以
Python 对象引用传递 对象引用传递 ❌ 不能（和 Java 类似）
Go 值传递（默认） 可传指针 *T ✅ 传指针可以
Java 刻意不支持引用传递（不提供 ref 类似语法），是语言设计的主动选择，见下节。

8. 设计思考：为什么 Java 只有值传递？

Java 选择只有值传递，背后有明确的工程考量：

  • 简单性与可预测性：调用方不用担心传入的变量被方法"偷偷改掉"指向。代码行为更容易推理。
  • 安全性：如果允许引用传递，外部代码可以通过参数修改调用方的本地变量，这在多线程和安全沙箱场景下非常危险。
  • OO 风格下够用：Java 鼓励通过对象字段（或返回值）来传递数据，而不是依赖"出参"。需要多返回值时，封装成对象返回即可。
常见误区：很多人看到"传引用类型时对象字段能被修改"就以为是引用传递。 这是混淆了"传的是什么（地址值）"和"传递方式（值传递）"。 判断标准只有一个：能不能在方法内让调用方的变量指向另一个对象。Java 的答案是不能，所以只有值传递。

9. 知识联系

  • JVM 栈帧：每次方法调用 JVM 压入一个新栈帧，参数是局部变量表的一部分。值传递本质上就是"把值复制到新栈帧的局部变量表里"。
  • final 参数：final int x 只是禁止在方法体内给 x 重新赋值，和值传递是正交的概念，不影响外部变量。
  • 不可变对象（Immutable）：String、Integer 等不可变类与值传递配合，使得"传参不会改变原对象"成为双重保证——地址副本指向的对象本身也不可变。
  • 深拷贝 vs 浅拷贝：值传递传的是地址副本（浅），如果不想让方法修改对象字段，需要在传入前做深拷贝，或使用不可变对象。
  • 多线程可见性：两个线程传同一个对象引用，都能通过地址找到同一个堆对象，字段修改对彼此可见（但需要同步保证有序性）。

10. 面试高频考点

Q：Java 是值传递还是引用传递？
标准答案：Java 只有值传递。传基本类型时传数据值的副本；传引用类型时传地址值的副本。 两种情况都是"把变量里存的内容复制一份"，因此永远无法在方法内让调用方的变量指向另一个对象。
Q：下面的 swap 方法能交换 a、b 吗？ 
static void swap(int a, int b) {
    int tmp = a;
    a = b;
    b = tmp;
}

public static void main(String[] args) {
    int x = 1, y = 2;
    swap(x, y);
    System.out.println(x + " " + y); // 1 2，没有交换
}
答：不能。a、b 是 x、y 的副本，方法内交换的是副本，原变量不受影响。
如果要真正交换，可以：① 返回数组/Pair；② 用单元素数组包装（int[] xRef = {x}）；③ 使用 AtomicInteger 等可变包装类。
Q：下面两段代码有什么区别？ 
// 代码 A
static void modify(StringBuilder sb) {
    sb.append("!");        // 修改对象内容，调用方可见
}

// 代码 B
static void reassign(StringBuilder sb) {
    sb = new StringBuilder("new"); // 重新赋值，调用方不可见
}
答：代码 A 通过地址副本操作同一个 StringBuilder 对象，调用方能看到变化（append 修改了堆上的对象）。 代码 B 让参数 sb 指向新对象，但调用方的变量地址没变，看不到变化。 这正是"值传递传的是地址副本"的典型体现。
