08. 运算符优先级

优先级决定了表达式的求值顺序。读懂优先级表，才能看懂别人的代码，写出不坑自己的代码。

为什么要有"优先级"这个概念？

数学里我们从小就学"先乘除后加减"。计算机语言同理：当一条表达式里出现多个运算符时，编译器必须有一套明确的规则来决定先算哪个、后算哪个，否则同一段代码在不同编译器下会得出不同结果，程序就不可靠了。

Java 的运算符优先级直接继承自 C/C++，规则定死在 JLS（Java Language Specification）里。理解优先级的本质是理解语法树的构建顺序——编译器在解析 a + b * c 时，会先把 b * c 作为一个子树，再把结果与 a 相加。

完整优先级速查表（从高到低）

优先级数字越小，越先求值。同一优先级看结合性（左结合 / 右结合）。

优先级 运算符 说明 结合性
1[] . () 方法调用 x++ x--后缀/访问左→右
2++x --x +x -x ~ !前缀 / 一元右→左
3new (type)对象创建 / 强转右→左
4* / %乘除取模左→右
5+ -加减 / 字符串拼接左→右
6<< >> >>>位移左→右
7< <= > >= instanceof关系左→右
8== !=相等判断左→右

10^按位 XOR左→右

记忆口诀：后缀 → 一元 → 乘除 → 加减 → 移位 → 关系 → 相等 → 位与 → 位异 → 位或 → 逻辑与 → 逻辑或 → 三元 → 赋值。重点记住"关系比相等高"和"位运算比逻辑运算高"这两个常被混淆的点。

结合性（Associativity）——优先级相同时谁先算？

当相邻两个运算符优先级相同时，结合性决定方向。Java 绝大多数运算符是左结合（从左到右求值），只有一元运算符、三元运算符、赋值运算符是右结合。

结合性示例.java
// 左结合：减法从左到右
int a = 10 - 3 - 2;   // 等价于 (10 - 3) - 2 = 5，不是 10 - (3 - 2) = 9

// 右结合：赋值从右到左
int x, y, z;
x = y = z = 5;          // 等价于 x = (y = (z = 5))，z先赋5，再y，再x

// 右结合：三元运算符
int v = true ? 1 : false ? 2 : 3;
// 等价于 true ? 1 : (false ? 2 : 3)，结果是 1

高频陷阱：这些表达式你能算对吗？

陷阱一：位运算 & 的优先级低于 ==

位运算陷阱.java
int n = 4;

// 错误写法（极其常见的 bug！）
if (n & 1 == 0) {          // 实际等价于 n & (1 == 0) → n & false → 编译错误！
    // ...
}

// 正确写法：加括号确保先算位与
if ((n & 1) == 0) {        // n是偶数
    System.out.println("偶数");
}

// 原因：优先级表中 == (8) 比 & (9) 更高（数字越小越高），
// 所以 & 后于 == 执行。这是从 C 继承来的历史设计，被公认为糟糕的决策。
Java（和 C）把 &、^、| 的优先级设计得低于比较运算符，这是历史遗留问题，Java 之父 James Gosling 也承认这是从 C 继承来的错误。凡是用位运算做条件判断，必须加括号！

陷阱二：++ 前缀 vs 后缀

自增陷阱.java
int i = 5;

// 后缀 ++ 优先级最高，但"返回旧值"再自增
int a = i++ + 2;     // a = 5 + 2 = 7，之后 i = 6

// 前缀 ++ 优先级也高，先自增"返回新值"
int b = ++i + 2;     // 先 i 变 7，b = 7 + 2 = 9

// 多个副作用混在同一表达式：未定义的求值顺序（不要这样写！）
int j = 0;
int c = j++ + j++;   // Java 定义为 0 + 1 = 1（左到右求操作数），但可读性极差，避免！

陷阱三：加法 vs 字符串拼接的优先级与求值顺序

字符串拼接陷阱.java
int x = 1, y = 2;

// 从左到右：先 "sum=" + 1 → "sum=1"，再 + 2 → "sum=12"（不是3！）
System.out.println("sum=" + x + y);   // 输出：sum=12

// 加括号让数字先算
System.out.println("sum=" + (x + y)); // 输出：sum=3

// 反方向：数字在前，遇到字符串才切换为拼接
System.out.println(x + y + "=sum"); // 输出：3=sum

陷阱四：赋值运算符优先级最低

赋值优先级陷阱.java
boolean found;

// 常见误写：想检测 list.contains() 并赋值给 found
if (found = list.contains("x")) {  // 合法！赋值结果作为 boolean 条件
    // found 已被赋值，这里是"赋值+判断"的惯用法
}

// 危险：== 和 = 一字之差，赋值优先级低到不会影响条件，但逻辑全变了
int val = 0;
if (val == 1 || val = 2) {  // 编译错误：= 返回 int，不能直接当 boolean
    // Java 比 C 严格：不允许把 int 赋值结果用作条件，这是一种保护
}

最佳实践：括号是最便宜的代码文档

优先级规则是给编译器看的，括号是给人看的。当你对优先级有丁点不确定时，加括号是最正确的选择——它不影响性能，却极大提升可读性。

可读性对比.java
// 难读版：需要在脑子里查优先级表
if (a < 10 && b > 0 || c != 5 && d == 0) {}

// 易读版：括号明确分组，意图一目了然
if ((a < 10 && b > 0) || (c != 5 && d == 0)) {}

// 位掩码操作：必须加括号，无一例外
final int MASK = 0xFF;
int low8  = (value &  MASK);         // 取低8位
int high8 = ((value >> 8) & MASK);   // 取高8位，两层括号清晰

与编译器的联系：优先级如何影响字节码？

优先级是编译期概念，编译器在生成字节码时已经按优先级构建好了 AST（抽象语法树），字节码里完全看不到运算符优先级，只有明确的操作数入栈顺序和指令序列。这意味着：

  • 括号不产生额外字节码，不影响运行时性能
  • 反编译工具（如 javap -c）输出的字节码序列就是最终求值顺序
  • IDE 的"不必要的括号"警告是风格建议，不是性能问题
字节码对比（javap -c 输出片段）

面试高频考点

考点 1：System.out.println(1 + 2 + "3") 输出什么？
答案：33。加法左结合，先算 1+2=3（整数加），再 3+"3"="33"（字符串拼接）。
考点 2：System.out.println("1" + 2 + 3) 输出什么？
答案：123。先 "1"+2="12"，再 "12"+3="123"。
考点 3：int a = 5; System.out.println(a++ + ++a); 输出什么？
答案：12。Java 按从左到右求操作数：a++ 返回 5（a变6），++a 先加变7再返回7，5+7=12。
考点 4：if (flag = true) 与 if (flag == true) 有何区别？
答案：前者是赋值（把 true 赋给 flag，条件恒为 true，是 bug）；后者是相等判断。Java 此处不像 C，int 赋值不能直接用作 boolean 条件，但 boolean 赋值可以——这是一个隐蔽的陷阱。
考点 5：位运算 & 和逻辑 && 优先级谁高？
答案：&（按位与，优先级 9）比 &&（逻辑与，优先级 12）优先级更高（数字越小越高）。但注意：& 依然低于 ==（优先级 8），所以 n & 1 == 0 实际上是 n & (1 == 0)。

与其他知识点的联系

运算符优先级 ├── 三元运算符（上一节）：右结合，嵌套三元要靠括号理清 ├── 位运算：& ^ | 优先级低于比较运算符，必须加括号 ├── 逻辑运算短路：&& || 优先级低，但短路特性影响副作用执行 ├── 字符串拼接：+ 左结合导致 "a" + 1 + 2 ≠ "a" + (1+2) ├── 赋值表达式：= 优先级最低，可在条件语句中合法使用 └── 编译期优化：常量折叠在编译时按优先级计算，字节码不再体现
