01. package 与 import

Java 的命名空间机制——从「为什么需要它」到「编译器如何解析它」，一次讲透。

① 为什么需要 package？

想象一家大公司：财务部有个员工叫「张伟」，技术部也有个员工叫「张伟」。 公司内部通讯时，两个人同名完全不冲突，因为大家都知道前面要加部门名——这正是 package 的本质：给类名加上「所属命名空间」，彻底消除同名冲突。

Java 生态有数以万计的第三方库。如果没有包机制，不同厂商都写了一个叫 StringUtils 的工具类，你的项目引用两个库，编译器根本不知道用哪个。 package 用「反向域名」约定（如 com.google.common.collect）在全球范围内保证唯一性。

② package 声明语法

package 语句必须是源文件的第一行有效代码（注释和空行不算），且一个文件只能有一条。

UserService.java
// 必须是第一行有效代码，否则编译报错
package com.example.service;

public class UserService {
    // 这个类的全限定名（FQN）= com.example.service.UserService
}

包名与目录结构必须完全一致。编译器和 JVM 都依赖这个约定定位 .class 文件。 包名通常全小写，多段之间用 . 分隔，每一段对应磁盘上的一级目录。

src/
└── com/
    └── example/
        └── service/
            UserService.java ← package com.example.service;
        └── model/
            User.java ← package com.example.model;
默认包（default package）：如果源文件不写 package，类属于"无名包"。这种类无法被有包名的类 import，也无法被反射工具按全限定名找到。生产代码中永远不要用默认包。

③ import 语法与解析原理

有了包之后，引用其他包的类需要写全限定名，代码会变得极其冗长。 import 的作用是告诉编译器「在当前文件里，遇到这个短名字，去哪个包里找」，本质是编译期的名字替换，不影响运行时性能，也不会把字节码嵌入当前类。

三种 import 形式

三种 import 用法对比
// ① 精确导入（最推荐，IDE 默认行为）
import java.util.ArrayList;
import java.util.List;

// ② 通配符导入（导入包下所有 public 类，但不包括子包）
import java.util.*;

// ③ 静态导入（JDK 5+，导入静态成员，避免写类名前缀）
import static java.lang.Math.PI;
import static java.lang.Math.*;

public class Demo {
    void calc() {
        // 有了静态导入，不必写 Math.sqrt(Math.PI)
        double r = sqrt(PI);
    }
}

java.lang 自动导入

java.lang 包（String、Integer、System、Object……）由编译器隐式导入，不需要手写 import java.lang.*;。 这是 Java 设计者给最核心 API 的"特权"，其他包没有这个待遇。

④ 同包的类互相不需要 import

同一个包下的类天然可见，就像同一个部门的同事互相打招呼不需要报工号。 只有跨包引用时才需要 import（或者写全限定名）。

com/example/service/OrderService.java
package com.example.service;

// UserService 和 OrderService 同包，无需 import
public class OrderService {
    private UserService userService;  // 直接用短名，OK
}

⑤ 全限定名（FQN）与 import 的关系

import 只是一个"编译期别名"，任何时候你都可以不写 import，直接使用全限定名。 这在同名冲突时非常有用：

同名类冲突时的解决方案
import java.util.Date;   // import 一个，另一个用全限定名

public class Report {
    Date utilDate;                    // java.util.Date（短名）
    java.sql.Date sqlDate;           // java.sql.Date（全限定名）

    // 两个同名类共存，互不干扰
}
陷阱：通配符不能解决冲突——如果同时 import java.util.* 和 import java.sql.*，使用 Date 会编译报错（歧义），此时必须至少有一个写全限定名。

⑥ 包是访问控制的边界

package 不只是命名工具，它还是 Java 四种访问级别之一的作用域边界。 不写访问修饰符（package-private / 包私有）的成员只对同包类可见，是一种天然的模块封装手段。

修饰符 同类 同包 子类（跨包） 任意类
private ✓ ✗ ✗ ✗
（无修饰符） ✓ ✓ ✗ ✗
protected ✓ ✓ ✓ ✗
public ✓ ✓ ✓ ✓

Spring、JDK 源码大量使用包私有来隐藏实现细节。例如 java.util.ArrayList 的内部迭代器 Itr 就是包私有类，外部无法直接访问。

⑦ 编译器与 JVM 如何解析包路径

理解这个机制能帮你快速排查 ClassNotFoundException 和 NoClassDefFoundError。

  1. 编译期：javac 根据 -sourcepath 和 -classpath，把包名中的 . 换成目录分隔符，在对应目录下寻找 .java 或 .class 文件。
  2. 运行期：JVM 类加载器（ClassLoader）同样把全限定名的 . 换成 /，在 Classpath 里的每个根目录（或 JAR 包）中寻找对应的 .class 文件。
  3. JAR 包：本质是按包目录结构打包的 ZIP，JVM 可以直接从中读取 .class 文件，逻辑与文件夹完全相同。
手动编译与运行示例（理解路径规则）
# 编译（-d 指定 class 输出根目录，javac 会自动创建包目录）
javac -d out src/com/example/service/UserService.java

# 运行（从 out 目录出发，用全限定名）
java -cp out com.example.service.UserService

⑧ 常见陷阱 & 面试高频考点

陷阱 1：import 不导入子包

import java.util.* 只导入 java.util 包下的类，不包括 java.util.concurrent.*。 子包与父包在 Java 中是完全独立的命名空间，没有继承关系。

陷阱 2：package 声明与文件名/目录不匹配

包声明是 com.example.service，但文件放在 com/example/ 目录下，编译器会报错： class UserService is public, should be declared in a file named UserService.java（或目录不匹配错误）。

陷阱 3：静态导入滥用

import static 方便了书写，但过度使用会让读者看不出方法来自哪个类，降低可读性。 建议只在测试断言（import static org.junit.jupiter.api.Assertions.*）和常量集合（如 Math.*）场景使用。

面试高频：import 会不会增加内存开销？

不会。 import 是纯编译期指令，编译后字节码里只有全限定名，没有任何 import 信息。 类是否被加载进内存取决于是否真正使用了该类，与 import 无关。

面试高频：同名类冲突怎么处理？

最多 import 一个，另一个必须在代码中用全限定名引用。两个都不 import、全部用全限定名也可以。

面试高频：package-private 的实际用途

典型场景：在同一个包里写一个"包内公开、外部隐藏"的辅助类或工厂方法，外部包必须通过公开 API 间接使用，无法绕过封装。这是 Java 模块化设计的基石之一（JPMS 出现前的主要手段）。

⑨ 延伸：Java 9 模块系统（JPMS）与 package

Java 9 引入了 module-info.java，在包之上再加了一层"模块"边界。 模块可以控制哪些包对外 exports，即使是 public 类，未被导出的包中的类也无法被外部模块访问。 这解决了包机制的一个长期痛点：public 类太容易被外部依赖。

module-info.java（概念预览）
module com.example.service {
    // 只把 API 包导出，实现包对外不可见
    exports com.example.service.api;
    requires com.example.model;
}

日常开发中不一定会用 JPMS，但理解"包是模块的子集"有助于看懂 JDK 源码的组织方式（如 java.base 模块）。
