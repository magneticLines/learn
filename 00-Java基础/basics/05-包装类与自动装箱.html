05 · 包装类与自动装箱

Java 是"几乎纯面向对象"的语言——集合、泛型、反射全部只接受对象，但 int、long 这些基本类型不是对象。包装类就是架在两者之间的桥梁，而自动装箱/拆箱则是编译器为我们自动搭桥的语法糖。

1. 为什么需要包装类

Java 泛型、集合、反射等 API 的类型参数要求 Object 或其子类。基本类型不继承 Object，无法直接放入 List<Integer>，也无法作为泛型参数。

// 不加包装类，以下代码根本无法编译
List<int> ids = new ArrayList<>(); // 编译错误：基本类型不能做泛型参数

// 正确做法：使用包装类
List<Integer> ids = new ArrayList<>();
ids.add(42);

完整对照表

基本类型包装类字节数缓存范围（IntegerCache 等）

包装类除了充当"对象容器"，还携带了大量工具方法，比如 Integer.parseInt()、Integer.toBinaryString()、Integer.bitCount() 等，这是它的第二大价值。

2. 自动装箱 / 拆箱

自动装箱（autoboxing）和拆箱（unboxing）是 Java 5 引入的编译器语法糖，编译后的字节码里会自动插入 valueOf() 和 xxxValue() 调用。

// 源代码（你写的）
Integer a = 42;        // 装箱
int     b = a;         // 拆箱
List<Integer> list = new ArrayList<>();
list.add(100);         // 隐式装箱

// 编译器翻译后（等价于）
Integer a = Integer.valueOf(42);   // 装箱
int     b = a.intValue();          // 拆箱
list.add(Integer.valueOf(100));    // 隐式装箱
记忆要点：装箱 = valueOf()，拆箱 = xxxValue()（如 intValue()、longValue()）。这个细节在面试中常被追问。

3. Integer 缓存池（IntegerCache）

Integer.valueOf() 并不总是 new Integer()。JDK 源码（java.lang.Integer）中有一段静态内部类：

// JDK 源码节选（OpenJDK 17，略有简化）
private static class IntegerCache {
    static final int low  = -128;
    static final int high;          // 默认 127，可通过 JVM 参数调大
    static final Integer[] cache;

    static {
        int h = 127;
        // 读取 JVM 参数 java.lang.Integer.IntegerCacheHighVal
        String integerCacheHighPropValue =
            VM.getSavedProperty("java.lang.Integer.IntegerCacheHighVal");
        if (integerCacheHighPropValue != null) {
            int i = parseInt(integerCacheHighPropValue);
            i = Math.max(i, 127);
            h = Math.min(i, Integer.MAX_VALUE - (-low) - 1);
        }
        high = h;
        cache = new Integer[(high - low) + 1];
        int j = low;
        for (int k = 0; k < cache.length; k++)
            cache[k] = new Integer(j++);   // 提前创建好，直接复用
    }
}

public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];  // 返回缓存对象
    return new Integer(i);   // 超出范围才 new
}

== 比较陷阱

Integer x = 127;
Integer y = 127;
System.out.println(x == y);   // true  ← 同一个缓存对象

Integer p = 128;
Integer q = 128;
System.out.println(p == q);   // false ← 各自 new，不同对象
System.out.println(p.equals(q)); // true ← equals() 比较值，始终正确
陷阱：用 == 比较 Integer 只在 -128~127 范围内"碰巧正确"，超出范围必然返回 false。
正确做法：始终用 equals() 比较包装类的值。

4. 拆箱时的 NullPointerException

拆箱要求对象不为 null，否则 null.intValue() 直接 NPE。以下三种场景最为隐蔽：

场景一：Map.get 返回 null 直接赋给基本类型

Map<String, Integer> map = new HashMap<>();
// 反例：key 不存在时 get 返回 null，拆箱触发 NPE
int count = map.get("missing");  // NullPointerException!

// 正例：先判 null，或使用 getOrDefault
int count = map.getOrDefault("missing", 0);

场景二：三元运算符的隐式拆箱

Integer a = null;
// 反例：三元运算符结果类型推断为 int，自动拆箱 a → NPE
int result = (a != null) ? a : -1;   // 看起来安全，实则不然
// 原因：编译器将整个表达式推断为 int，所以 a 和 -1 都会被拆箱
// 等价于：int result = (a != null) ? a.intValue() : -1;
// 当 a != null 为 false 时执行 -1 分支，但 a 已经被"准备好拆箱"，某些编译器实现会先拆

// 正例：明确返回 Integer
Integer result = (a != null) ? a : -1;   // 返回 Integer 对象，不拆箱
三元运算符的类型推断规则：若一侧是基本类型，另一侧会被自动拆箱以匹配类型。当那一侧是 null 时就 NPE。这是 Java 语言规范 §15.25 的行为，很多人不知道。

场景三：直接赋值给基本类型

// 反例：从数据库/JSON 返回的包装类可能为 null
Integer dbValue = getUserAge();   // 可能返回 null
int age = dbValue;                // 拆箱 → NPE

// 正例
int age = dbValue != null ? dbValue : 0;
// 或使用 Optional
int age = Optional.ofNullable(dbValue).orElse(0);

5. 性能影响

每次装箱（超出缓存范围时）都会在堆上创建一个新对象，大量装箱会增加 GC 压力。

// 反例：1000 万次装箱，产生大量短命 Integer 对象
public static long sumBoxed(int n) {
    Long sum = 0L;               // 注意：Long，不是 long！
    for (int i = 0; i < n; i++) {
        sum += i;                // 每次 += 都会：拆箱 sum → 计算 → 装箱结果
    }
    return sum;
}

// 正例：全程使用基本类型，零装箱
public static long sumPrimitive(int n) {
    long sum = 0L;
    for (int i = 0; i < n; i++) {
        sum += i;
    }
    return sum;
}

// 性能对比（1000 万次，参考值）
// sumBoxed:     ~200ms，产生约 1000 万个 Long 对象
// sumPrimitive: ~5ms，零堆分配
原则：在热点循环和高频调用中坚持使用基本类型；只有在需要放入集合、传入泛型 API、或需要 null 值时才使用包装类。

6. 常用工具方法

// 字符串 → 基本类型
int    i = Integer.parseInt("42");
long   l = Long.parseLong("9876543210");
double d = Double.parseDouble("3.14");
boolean b = Boolean.parseBoolean("true");   // 大小写不敏感

// 进制转换（返回 String）
String bin = Integer.toBinaryString(255);   // "11111111"
String oct = Integer.toOctalString(255);    // "377"
String hex = Integer.toHexString(255);      // "ff"
int    dec = Integer.parseInt("ff", 16);    // 255，指定进制解析

// 比较（避免溢出，比 a - b 安全）
int cmp = Integer.compare(a, b);   // 相当于 a < b ? -1 : (a == b ? 0 : 1)
int cmp2 = Long.compare(x, y);

// 位运算工具
int ones  = Integer.bitCount(255);          // 8，统计二进制中 1 的个数
int high  = Integer.highestOneBit(100);     // 64（最高位置 1 的值）
int rev   = Integer.reverse(1);             // 位翻转
int ctz   = Integer.numberOfTrailingZeros(8); // 3

// 最大/最小/加法（函数式编程友好）
int max = Integer.max(a, b);    // 等价于 Math.max，可直接传入方法引用
int sum = Integer.sum(a, b);    // 可用于 Stream.reduce(0, Integer::sum)

// 无符号运算（Java 8+）
int udiv  = Integer.divideUnsigned(-1, 2);  // 把 int 当作无符号处理
String us = Integer.toUnsignedString(-1);   // "4294967295"

7. 与其他知识的联系

泛型擦除

Java 泛型在编译后会被擦除为 Object，这正是为什么泛型类型参数必须是对象（包装类）而不能是基本类型。List<int> 编译错误，而 List<Integer> 擦除后变成 List<Object>，可以正常运行。

集合框架

ArrayList、HashMap、HashSet 等全部基于 Object，存取基本类型时都会触发装箱/拆箱。若大量存储数值，可考虑 Eclipse Collections 或 Trove 等提供原生类型集合的第三方库（如 IntArrayList）。

Stream API

为了避免 Stream 中大量装箱，Java 8 专门提供了 IntStream、LongStream、DoubleStream：

// 反例：Stream<Integer> 会装箱
int sum = list.stream()
              .mapToInt(Integer::intValue)  // 显式转换回基本类型流
              .sum();

// 正例：直接用 IntStream
int sum = IntStream.rangeClosed(1, 100).sum();   // 零装箱

享元模式（Flyweight）与字符串常量池类比

IntegerCache 的设计思想就是享元模式：对高频使用的小整数对象预先创建并复用，避免反复分配。这与 JVM 字符串常量池（String Pool）的思想完全一致——字符串字面量也会被池化复用。两者都是"用空间换时间"的经典实现。

8. 面试高频考点

Q1：Integer a = 127; Integer b = 127; a == b 是 true 还是 false？
true。因为 127 在 IntegerCache 范围内，两个变量指向同一个缓存对象。但如果换成 128，结果就是 false。

Q2：自动装箱的底层实现是什么？
编译器会将 Integer a = 1 翻译为 Integer a = Integer.valueOf(1)，拆箱则翻译为 a.intValue()。这是纯编译期行为，运行时 JVM 无感知。

Q3：以下代码会 NPE 吗？
Integer a = null; int b = a; → 会 NPE。拆箱等价于 a.intValue()，对 null 调用方法必然 NPE。

Q4：IntegerCache 的范围是多少？能修改吗？
默认 -128 ~ 127，下限固定不可改，上限可通过 JVM 参数 -XX:AutoBoxCacheMax=<size>（或 -Djava.lang.Integer.IntegerCacheHighVal=<n>）调大。Long/Short/Byte 的缓存范围固定为 -128~127，不可调。

Q5：new Integer(42) 和 Integer.valueOf(42) 有什么区别？
new Integer(42) 强制在堆上创建新对象（Java 9 已标记为 @Deprecated）；Integer.valueOf(42) 会优先返回缓存对象，效率更高，也是自动装箱使用的方式。日常代码应始终用 valueOf() 或直接赋值触发自动装箱。
