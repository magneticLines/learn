04. 类型转换（自动 / 强制）

Java 是强静态类型语言，不同类型之间的赋值与运算必须遵守严格的转换规则。理解自动拓宽与强制收窄背后的精度-范围权衡，能让你彻底告别 ClassCastException 和截断 bug。

一、为什么 Java 需要类型转换？

Java 的 8 种基本类型在内存中占用的字节数各不相同。把小盒子里的数放进大盒子，一定装得下；但把大盒子里的数硬塞进小盒子，可能会溢出或截断。

byte (1B) → short (2B) → int (4B) → long (8B) → float (4B, 精度 ≈7位) → double (8B, 精度 ≈15位)

箭头方向 = 拓宽方向，沿箭头向右走是自动转换（Widening），逆方向需要强制转换（Narrowing）。char 是无符号 16 位整型，可以自动拓宽到 int、long、float、double。

为什么 long → float 是"拓宽"却可能丢精度？
float 只有 23 位尾数（约 7 位十进制精度），而 long 有 63 位有效值，因此超过 float 精度范围的 long 会被静默舍入。Java 允许这种自动转换，但会在精度上付出代价——这是语言设计的一个历史妥协。

二、自动类型转换（Widening / 隐式转换）

编译器在不丢失类型能力的前提下，自动把较小类型提升为较大类型。无需任何语法标记，赋值或运算时自动完成。

2.1 赋值时的拓宽

Widening.java
byte   b = 42;
short  s = b;          // byte → short：安全，范围变大
int    i = s;          // short → int：安全
long   l = i;          // int → long：安全
float  f = l;          // long → float：范围够，但精度可能丢！
double d = f;          // float → double：安全

// 演示 long → float 精度损失
long  big = 123456789012345L;
float fv  = big;        // 实际存储约 1.2345679E14，末尾被舍入
System.out.println(fv == big); // false！

2.2 运算时的拓宽（数值提升）

Java 规定：任何比 int 小的整型（byte、short、char）参与算术运算时，先被提升为 int，再计算。这是为了避免溢出和实现简单性（JVM 字节码主要基于 32 位操作）。

NumericPromotion.java
byte a = 100;
byte b = 100;
// byte c = a + b;  // ❌ 编译错误！a+b 的结果是 int，不能直接赋给 byte
int c = a + b;      // ✅ 正确：结果是 int(200)

// 混合类型运算：较小类型向较大类型提升
int    x = 10;
double y = 3.14;
double z = x + y;   // x 被提升为 double，结果 13.14
高频陷阱：byte/short 赋值与运算不对称
byte b = 1 + 2; 可以编译——因为右边是编译期常量表达式，编译器直接算出结果 3 并检查是否在范围内。
但 byte b1 = 1; byte b2 = 2; byte b3 = b1 + b2; 就报错——因为变量参与运算时会做数值提升，结果是 int。

三、强制类型转换（Narrowing / 显式转换）

把大类型转换为小类型时，需要程序员用 强制转换运算符 (目标类型) 明确表达"我知道可能丢失数据，我接受这个风险"。

3.1 基本语法

Narrowing.java
double d = 9.99;
int    i = (int) d;   // i = 9，小数部分直接截断（不是四舍五入！）

int  big  = 300;
byte tiny = (byte) big; // byte 范围 -128~127，300 溢出
                            // 300 = 0x0000_012C，只保留低8位 0x2C = 44
System.out.println(tiny);  // 输出 44

int  neg  = 200;
byte bNeg = (byte) neg;  // 200 = 0xC8，符号位为1，解释为 -56
System.out.println(bNeg); // 输出 -56

3.2 截断规则（底层原理）

强制转换整型时，直接截取低位二进制，丢弃高位。以 (byte) 300 为例：

300 的二进制（32位）： 00000000 00000000 00000001 00101100 ↑ 只保留低 8 位 byte 结果： 00101100 = 44

浮点转整型时，直接截断小数部分（向零取整，不是 floor）：

FloatToInt.java
System.out.println((int)  3.9);   // 3  （向零截断）
System.out.println((int) -3.9);  // -3 （向零截断，不是 -4）
System.out.println((int)  3.0);   // 3  （整数部分相同）

// 如需四舍五入，应使用 Math.round()
System.out.println(Math.round(3.9));  // 4

四、char 与整型的互转

char 是 16 位无符号整型，存储 Unicode 码点（0 ~ 65535）。它可以自动拓宽到 int，也能与整型强制互转，这一特性在字符运算中经常用到。

CharConvert.java
char c = 'A';
int  code = c;          // 自动拓宽：65（Unicode 码点）
System.out.println(code); // 65

// 利用 char 运算实现大小写转换
char lower = (char) (c + 32);  // 'A' + 32 = 'a'（ASCII 规律）
System.out.println(lower);      // a

// int 转 char（反查字符）
char ch = (char) 9786;           // ☺  (U+263A)
System.out.println(ch);

// char 参与字符串拼接：自动转成字符，不变成数字
System.out.println("char: " + c);    // char: A
System.out.println(c + " :char"); // 65 :char ← 坑！c 先被提升为 int
字符串拼接中的 char 陷阱
"prefix" + c 中，Java 先对 c 调用 String.valueOf(char)，所以输出字符 "A"。
但 c + " suffix" 中，+ 的左操作数是 char，Java 先做数值提升得到 int 65，再拼接字符串，结果是 "65 suffix"。顺序不同，行为不同！

五、引用类型的转换

基本类型转换改变的是值的表示；引用类型转换改变的是编译器对对象的"视角"，对象本身不变。

5.1 向上转型（Upcasting）— 自动，安全

Upcasting.java
class Animal { void speak() {} }
class Dog extends Animal {
    @Override void speak() { System.out.println("Woof"); }
    void fetch() { System.out.println("Fetching!"); }
}

Dog dog = new Dog();
Animal a = dog;    // 向上转型：自动，无需强制。a 的编译类型是 Animal
a.speak();         // 运行时调用 Dog.speak() ← 多态！实际类型决定方法
// a.fetch();       // ❌ 编译错误：Animal 没有 fetch()，编译器只看编译类型

5.2 向下转型（Downcasting）— 强制，可能抛异常

Downcasting.java
Animal a1 = new Dog();   // 实际是 Dog
Animal a2 = new Animal(); // 实际是 Animal

// 安全的向下转型：用 instanceof 先检查
if (a1 instanceof Dog) {
    Dog d = (Dog) a1;      // ✅ 安全
    d.fetch();
}

// 危险的向下转型：不检查直接强转
Dog d2 = (Dog) a2;       // ❌ 运行时 ClassCastException！
                             // a2 实际是 Animal，不是 Dog

// Java 16+ 的 Pattern Matching instanceof（推荐写法）
if (a1 instanceof Dog dog) {  // 检查 + 转型 + 绑定变量，一步完成
    dog.fetch();
}

六、String 与基本类型的互转

String 不是基本类型，与基本类型之间不存在 cast 运算符，必须通过标准库方法完成互转。

方向常用方式说明
基本类型 → StringString.valueOf(x)推荐，null 安全
基本类型 → String"" + x简便但创建多余对象，不推荐在循环中使用
String → intInteger.parseInt(s)非法输入抛 NumberFormatException
String → doubleDouble.parseDouble(s)同上
String → IntegerInteger.valueOf(s)返回包装类，有缓存
StringConvert.java
// 基本类型 → String
int n = 42;
String s1 = String.valueOf(n);   // "42"（推荐）
String s2 = Integer.toString(n); // "42"（等价）

// String → 基本类型
int    parsed = Integer.parseInt("123");
double pi     = Double.parseDouble("3.14");
boolean flag  = Boolean.parseBoolean("true");

// 陷阱：parseBoolean 对非 "true"（不区分大小写）一律返回 false
System.out.println(Boolean.parseBoolean("yes"));  // false，不是 true！

七、常见陷阱汇总

陷阱 1：整数除法悄悄丢精度

IntDivision.java
int a = 7, b = 2;
double result = a / b;         // 3.0，不是 3.5！先整除得 3，再拓宽为 double
double correct = (double) a / b; // 3.5，先把 a 转 double 再除

陷阱 2：long 字面量忘加 L

LongLiteral.java
// int 最大值约 21亿，超出后溢出
long wrong   = 3000000000;    // ❌ 编译错误：字面量先被当作 int，已溢出
long correct = 3000000000L;   // ✅ L 后缀告诉编译器这是 long 字面量

陷阱 3：float 精度比较

FloatCompare.java
float f = 0.1f;
double d = 0.1;
System.out.println(f == d);  // false！f 拓宽为 double 时尾数填充 0，
                               // 与原始 0.1 的 double 表示不同
// 正确比较浮点：用误差范围
System.out.println(Math.abs(f - d) < 1e-6);  // true

陷阱 4：强转 + 复合赋值运算符

+=、-=、*=、/= 等复合赋值运算符隐含了强制转换，这是许多人不知道的语法糖：

CompoundAssign.java
byte b = 10;
// b = b + 1;  // ❌ 编译错误：b+1 是 int
b += 1;       // ✅ 等价于 b = (byte)(b + 1)，编译器自动插入强转
              // 好处：简洁；坏处：溢出时静默截断，不报错

byte max = 127;
max += 1;
System.out.println(max);  // -128，悄悄溢出了！

九、与其他知识点的联系

  • 包装类与自动装箱（下一节）：int 和 Integer 之间的自动装箱/拆箱，本质也是一种"隐式转换"，但走的是方法调用路径，不是字节码 cast 指令。
  • 多态与 OOP：向上/向下转型是多态的基础机制，instanceof 检查运行时类型，与编译期类型完全独立。
  • 泛型擦除：Java 泛型在字节码层面会插入 checkcast 指令，本质是强制类型检查，ClassCastException 经常出现在泛型使用不当时。
  • JVM 字节码：i2b、i2s、i2l、l2f、f2d 等是 JVM 专门的类型转换指令，编译器生成的强转语句最终都映射到这些指令。

面试高频考点

1. byte b = 1 + 2 能编译吗？byte b1=1; byte b2=2; byte b3=b1+b2 呢？
前者可以：右边是编译期常量表达式，编译器计算出 3 并通过范围检查。后者不行：变量参与运算触发数值提升，结果为 int，不能隐式赋给 byte。
2. (int)3.9 的结果是多少？
结果是 3，不是 4。浮点转整型是向零截断（truncation），不是四舍五入。
3. long 转 float 是向上转型（自动），为什么可能丢精度？
因为"类型容量"和"精度"是两个维度。float 的范围（1.4E-45 ~ 3.4E38）包含了 long 的全部范围，但 float 只有约 7 位十进制精度，超过精度范围的 long 值会被舍入。
4. ClassCastException 什么时候抛出？
向下转型时，若引用实际指向的对象不是目标类型或其子类，JVM 在运行时抛出 ClassCastException。编译期不报错（因为编译器只检查静态类型是否有继承关系），错误在运行时才暴露。解决方案：转型前用 instanceof 检查，或 Java 16+ 使用 Pattern Matching instanceof。
