10. if / else if / else

条件分支是程序「做决策」的基础——理解它的求值规则、短路行为和常见陷阱，能避开 90% 的逻辑 bug。

① 为什么需要条件分支

程序的本质是处理不同情况并给出不同响应。没有分支，代码只能从头到尾执行同一条路径，无法区分「用户已登录」和「未登录」、「余额充足」和「余额不足」这类截然不同的场景。

从 CPU 的角度看，条件分支对应条件跳转指令（如 x86 的 jne/je）。Java 编译器把 if 语句翻译为字节码中的 ifeq/ifne 等指令，JVM 再映射到具体平台的机器码。理解这一层有助于后续理解分支预测对性能的影响。

② 语法结构全貌

Java 的条件分支有三种形态，从简到繁依次叠加：

if (条件) { ← 只处理"真"的情况
  ...
}

if (条件) { ← 二选一
  ...
} else {
  ...
}

if (条件1) { ← 多选一（链式）
  ...
} else if (条件2) {
  ...
} else {
  ...
}
设计原则：每条 else if 只在前面所有条件都为 false 时才被求值——这叫做顺序短路，一旦命中就不再向下检查。

③ 基础代码示例

成绩分级（多分支经典场景）

ScoreGrade.java
public class ScoreGrade {
    public static String grade(int score) {
        // 条件从"最严格"到"最宽松"排列，命中即返回
        if (score < 0 || score > 100) {
            throw new IllegalArgumentException("分数超出范围: " + score);
        } else if (score >= 90) {
            return "A";
        } else if (score >= 80) {
            // 能到这里，score 必然 < 90，无需再写 score < 90
            return "B";
        } else if (score >= 70) {
            return "C";
        } else if (score >= 60) {
            return "D";
        } else {
            return "F";
        }
    }
}

注意第二个分支里的注释——由于顺序短路，每个 else if 分支已经隐含了「前面所有条件都不成立」这一前提，不需要重复写 score < 90。这是新手最容易冗余的地方。

④ 条件表达式求值规则

if 的括号内必须是 boolean 类型（或能自动拆箱为 boolean 的 Boolean）。Java 不像 C/C++ 那样把整数 0/非 0 视为假/真，写 if (count) 直接编译报错。

条件表达式类型示例
int count = 5;

// ✗ 编译错误：int 不是 boolean
// if (count) { ... }

// ✓ 显式比较
if (count != 0) { ... }

// ✓ boolean 变量直接用
boolean isReady = true;
if (isReady) { ... }

// ⚠ Boolean 自动拆箱——null 时抛 NullPointerException！
Boolean flag = null;
if (flag) { ... }  // 运行时 NullPointerException

⑤ 短路求值（Short-Circuit Evaluation）

条件中常用的 && 和 || 都有短路行为：

  • A && B：若 A 为 false，B 不会被求值
  • A || B：若 A 为 true，B 不会被求值

这既是性能优化（避免无谓计算），也是空指针防护的常用技巧：

短路求值防护 null
String name = getName();  // 可能返回 null

// ✓ 先判 null，短路后 name.isEmpty() 不会执行
if (name != null && !name.isEmpty()) {
    System.out.println("欢迎，" + name);
}

// ✗ 顺序颠倒——name 为 null 时直接 NPE
if (!name.isEmpty() && name != null) {  // 危险！
    ...
}
技巧：利用短路特性，把成本低、排除力强的条件放在最前面。例如先判 null，再调用方法；先判简单整数比较，再调用 DB 查询。

⑥ 省略大括号的危险

Java 语法允许单行 if 省略大括号，但这是最经典的维护陷阱之一——2014 年 Apple 的 goto fail 漏洞就源于此。

Apple goto fail 风格错误复现
// 看似只有一行受 if 控制
if (condition)
    doA();
    doB();  // ⚠ 缩进骗了你！这行永远执行，与 condition 无关

// 正确写法：永远加大括号
if (condition) {
    doA();
    doB();
}
Google / Alibaba Java 规范均强制要求：即使 if 块只有一行语句，也必须加大括号。代码审查中发现无括号 if 应立即要求修改。

⑦ 悬挂 else（Dangling Else）

当 if 和 else 数量不匹配时，编译器按就近匹配原则：else 属于离它最近的那个 if。

悬挂 else 示例
// 缩进暗示 else 属于外层 if，但实际属于内层 if
if (x > 0)
    if (y > 0)
        System.out.println("第一象限");
else
    System.out.println("x <= 0");  // 实际是 y <= 0 时执行！

// 用大括号消除歧义
if (x > 0) {
    if (y > 0) {
        System.out.println("第一象限");
    }
} else {
    System.out.println("x <= 0");
}

⑧ 经典陷阱：赋值 vs 比较

C/C++ 程序员转 Java 时常把 =（赋值）和 ==（比较）混淆。Java 编译器会拒绝 if (x = 5)，因为赋值表达式的类型是 int，不是 boolean——这是 Java 比 C 更安全的地方之一。

Java 编译器帮你挡住这个错误
int x = 3;

// ✗ 编译错误（Java 不允许，C 里是合法的 bug）
// if (x = 5) { ... }

// ✓ 正确比较
if (x == 5) { ... }

// ⚠ 但 boolean 变量赋值不会报错！
boolean flag = false;
if (flag = true) {   // 合法但逻辑错误！永远为 true
    ...
}
陷阱：boolean 变量的赋值表达式本身就是 boolean，所以 if (flag = true) 能编译通过，却是 bug。习惯写 if (flag) 而不是 if (flag == true) 可避免此类问题。

⑨ 对象比较：== 与 equals

在条件判断中比较字符串（或其他对象）时，== 比较的是引用地址，而不是内容。这是 Java 初学者最高频的 bug 之一。

字符串比较陷阱
String a = new String("hello");
String b = new String("hello");

// ✗ 比较引用：两个不同对象，结果为 false
if (a == b) { // false，即使内容相同！
    ...
}

// ✓ 比较内容
if (a.equals(b)) {  // true
    ...
}

// ✓ 常量放前面，防止 a 为 null 时 NPE
if ("hello".equals(a)) {  // 推荐写法
    ...
}

// ✓ JDK 7+ Objects.equals 同时处理 null
if (Objects.equals(a, b)) {  // 最安全
    ...
}

⑩ 守卫子句（Guard Clause）：让代码更扁平

深层嵌套的 if-else 会形成「箭头形代码」，可读性极差。守卫子句（Guard Clause / Early Return） 的思路是：先处理边界/错误情况并提前返回，让主干逻辑不再嵌套。

反例：箭头代码
public String process(User user) {
    if (user != null) {
        if (user.isActive()) {
            if (user.hasPermission()) {
                return "OK";
            }
        }
    }
    return "FAIL";
}
正例：守卫子句（扁平化）
public String process(User user) {
    // 先把不符合条件的情况排除，提前 return
    if (user == null)          return "FAIL";
    if (!user.isActive())      return "FAIL";
    if (!user.hasPermission()) return "FAIL";

    // 主干逻辑零嵌套，一目了然
    return "OK";
}

⑪ 与三元运算符的关系

condition ? expr1 : expr2 是 if-else 的表达式版本，可以内嵌在赋值、方法参数等位置。两者的本质相同，选择哪个取决于可读性：

场景 推荐形式 原因
简单二选一赋值 int max = a > b ? a : b; 简洁，一行表达意图
条件含副作用/多语句 if-else 块 三元嵌套三元可读性极差
方法参数中的条件 三元（谨慎） 避免为了三元而三元，可读性第一
JDK 14+ 补充：对于多值分支，新增的 switch 表达式（见下一节）往往比一串 else if 更清晰，尤其是对枚举或常量的多路匹配。

⑫ 与其他知识的联系

与 switch 的对比

if-else if 适合范围判断（score >= 90）和复杂条件组合；switch 适合对单一变量的离散值（枚举、整数常量、字符串）做多路匹配，编译器可优化为跳转表（O(1)）。

与 instanceof + 模式匹配（JDK 16+）

上一节学到的 instanceof 模式匹配本质上是一个带类型检查的 if 分支，JDK 21 的 switch 模式匹配则进一步把多个类型分支整合进 switch。

与异常处理的关系

守卫子句的另一种写法是在前置条件不满足时直接抛异常（fail-fast），而不是返回 null 或错误码。这正是 Objects.requireNonNull、Preconditions.checkArgument 等工具方法的设计思路。

字节码层面

if 在字节码中对应 ifeq/ifne/iflt 等指令。JIT 编译器会做分支预测优化：如果某个分支被反复执行，JIT 会把它编译为内联的热路径，少走的分支走慢路径。这意味着在性能敏感代码中，把最可能发生的条件放在最前面不仅是可读性问题，也是性能问题。

⑬ 面试高频考点

  • == vs equals 的区别：引用比较 vs 内容比较，String 字面量在常量池可共享（"a" == "a" 为 true，new String("a") == new String("a") 为 false）
  • 短路求值的顺序：考察能否用 null != obj && obj.method() 防 NPE
  • Boolean 拆箱 NPE：if (Boolean obj) 在 obj 为 null 时抛 NPE
  • if-else vs switch 的性能差异：switch 可被编译为跳转表（tableswitch）或查找表（lookupswitch），大量离散值时优于线性 if-else
  • 省略大括号的悬挂 else 问题：给出含歧义的代码片段，考察实际执行逻辑
  • 条件分支对分支预测的影响（高级）：预测失败导致流水线冲刷，影响纳秒级热点代码性能
