03. 变量声明与作用域

理解 Java 变量的四种分类、生命周期管理，以及编译器如何通过作用域规则防止命名冲突与内存泄漏。

为什么需要变量和作用域这两个概念？

变量是程序用来给内存中某块区域贴标签的机制——没有标签就只能用裸地址操作内存， 既危险又难读。而作用域（Scope）则是编译器给每个标签划定的可见范围： 超出范围后标签自动失效，避免两处同名变量互相干扰，也让 JVM 得以及时回收栈帧内存。

你可以把作用域想象成一栋办公楼里的员工工牌：同一层楼同名的工牌会冲突， 但不同楼层的同名工牌互不影响——离开那层楼，工牌就自动注销。

Java 变量的四种分类

分类 声明位置 存储区域 生命周期 默认值
局部变量 方法 / 代码块内 栈帧（Stack Frame） 方法调用 → 方法返回 无（必须显式初始化）
实例变量 类内、方法外（无 static） 堆（Heap） 对象创建 → 对象被 GC 有（0 / false / null）
类变量（静态变量） 类内、方法外（有 static） 方法区（Metaspace） 类加载 → 类卸载 有（0 / false / null）
参数变量 方法签名括号内 栈帧 方法调用 → 方法返回 无（由调用者传入）
底层联系：局部变量和参数变量住在栈帧里，方法一结束栈帧就弹出， 无需 GC 介入——这也是它们速度快的根本原因。 实例变量住在堆里，需要 GC 追踪引用才能回收。

声明语法全景

VariableDeclaration.javaJava
public class VariableDeclaration {

    // ① 类变量（静态变量）：类加载时初始化，所有实例共享
    static int instanceCount = 0;

    // ② 实例变量：每个对象独立拥有一份
    String name;          // 未赋值时默认 null
    int    age;           // 未赋值时默认 0
    boolean active;      // 未赋值时默认 false

    public VariableDeclaration(String name, int age) {
        // ③ 参数变量：name / age 在整个方法体内可见
        this.name = name;
        this.age  = age;
        instanceCount++;
    }

    public void demo() {
        // ④ 局部变量：必须先赋值才能使用，否则编译报错
        int x = 10;
        String msg = "hello";

        if (x > 5) {
            // 内层块可以读取外层变量 x
            int y = x * 2;  // y 的作用域仅限此 if 块
            System.out.println(y);
        }
        // System.out.println(y);  // ❌ 编译错误：y 已超出作用域
    }
}

作用域（Scope）规则详解

作用域的嵌套结构

Java 的作用域以花括号 {} 为边界，内层块可以访问外层变量，外层块不能访问内层变量。

类作用域（整个类体）
└── 方法作用域（方法体 {}）
    ├── 局部变量 x（方法级）
    └── if 块作用域
        └── 局部变量 y（块级，离开 if 即失效）

变量遮蔽（Shadowing）

局部变量可以和实例变量同名，此时局部变量会遮蔽实例变量。 想访问实例变量需加 this. 前缀——这也是构造函数里常见的写法。

Shadowing.javaJava
public class Counter {
    int count = 100;  // 实例变量

    public void reset() {
        int count = 0;   // 局部变量遮蔽了实例变量
        System.out.println(count);       // 打印 0（局部）
        System.out.println(this.count);  // 打印 100（实例）
    }
}
陷阱：Java 不允许在同一方法的嵌套块中重复声明同名局部变量。 下面的代码在 C/C++ 里合法，在 Java 里会编译报错：

int x = 1;
if (true) { int x = 2; } // ❌ 编译错误：Variable 'x' is already defined

final 变量：不可变的绑定

用 final 修饰的变量只能赋值一次。对基本类型来说，值不可变； 对引用类型来说，引用（地址）不可变，但引用指向的对象内容仍可修改。

FinalDemo.javaJava
final int MAX = 100;
// MAX = 200;  // ❌ 编译错误：cannot assign a value to final variable

final List<String> list = new ArrayList<>();
list.add("ok");  // ✅ 合法：list 引用未变，只是修改了对象内容
// list = new ArrayList<>();  // ❌ 编译错误：list 引用本身不能改变
最佳实践：凡是不需要重新赋值的局部变量，建议加 final。 这能让读者立即知道"这个变量不会变"，减少大脑追踪负担，也帮助 JIT 编译器做常量折叠优化。

var —— Java 10+ 的局部变量类型推断

Java 10 引入 var，让编译器从右侧初始化表达式中推断类型， 避免冗长的类型名重复书写。var 不是动态类型，编译后等价于写死了具体类型。

VarDemo.javaJava 10+
// 传统写法
Map<String, List<Integer>> map1 = new HashMap<>();

// var 写法：编译器推断为 HashMap<String, List<Integer>>
var map2 = new HashMap<String, List<Integer>>();

// var 只能用于局部变量，且必须有初始化表达式
// var x;          // ❌ 无法推断类型
// var x = null;   // ❌ null 类型不明确

// 在 for-each 中非常实用
for (var entry : map2.entrySet()) {
    System.out.println(entry.getKey());
}
可以用 var 的场合不能用 var 的场合
局部变量（有初始化表达式）实例变量 / 类变量
for / for-each 循环变量方法参数
try-with-resources 资源变量返回类型声明
lambda 参数（Java 11，加注解用）var x = null;（歧义）

初始化时机与默认值陷阱

实例变量和类变量在内存分配时由 JVM 自动清零（赋默认值）， 但局部变量没有这个待遇——JVM 规范规定必须由程序员显式初始化，否则编译器拒绝通过。 这样设计的原因是：局部变量存在栈帧里，栈帧复用时旧数据可能残留，自动清零会带来额外开销。

InitTrap.javaJava — 常见陷阱
public void trap() {
    int result;
    if (true) {
        result = 42;
    }
    // 编译器无法确定 if 分支一定执行（即使条件是 true）
    // ❌ 下面这行编译报错：variable result might not have been initialized
    System.out.println(result);
}

public void fix() {
    // 给一个安全的初始值，编译器就放行了
    int result = -1;
    if (true) {
        result = 42;
    }
    System.out.println(result);  // ✅ 输出 42
}

命名规范（不是风格，是工程共识）

变量类型命名规范示例
局部变量 / 实例变量 / 参数小驼峰（camelCase）userName, maxRetry
类变量（静态）小驼峰（同上）instanceCount
常量（static final）全大写 + 下划线MAX_SIZE, DEFAULT_TIMEOUT
类型参数（泛型）单个大写字母T, E, K, V

与其他知识点的联系

  • 栈与堆（JVM 内存模型）：局部变量在栈帧，实例变量在堆，这直接决定了两者的生命周期和 GC 行为。
  • 线程安全：局部变量是线程私有的（每个线程有自己的栈），天然线程安全；实例变量和类变量是共享的，多线程访问需要同步。
  • Lambda 与闭包：Lambda 表达式捕获的局部变量必须是 final 或「effectively final」（事实上未被修改），因为 Lambda 可能在不同栈帧中执行，Java 用值复制而非引用来捕获局部变量。
  • Spring Bean 作用域：Spring 的 @Scope（singleton/prototype/request/session）是在框架层面模拟了变量作用域的思想，控制对象的生命周期。
  • 下一篇（类型转换）：变量的声明类型决定了编译器允许哪些隐式转换，理解类型声明是学习自动拓宽和强制转换的前提。

面试高频考点

Q1：局部变量为什么没有默认值？
JVM 规范有意为之：局部变量存在栈帧里，JVM 不对栈帧内存清零以节省开销； 同时强制编译期检查"definitively assigned"可消灭整类 bug。
Q2：static 变量和实例变量的区别？
static 变量属于类，存在方法区（Metaspace），所有实例共享同一份； 实例变量属于对象，每个实例独立拥有，存在堆上。
Q3：final 修饰引用类型，对象内容能改变吗？
能。final 只锁住变量与对象的绑定关系（引用地址），不阻止对象自身状态的修改。 要做到"深度不可变"需要类本身也设计为 immutable（如 String、Integer）。
Q4：Lambda 捕获局部变量为何必须 effectively final？
Lambda 对象可能在局部变量所在栈帧已销毁后才执行，Java 通过复制值的方式捕获。 如果允许修改，复制值就会和原变量脱节，造成语义混乱，因此编译器直接禁止。
