常见加密算法误用场景

密码学最大的风险往往不是算法本身被破解，而是被用错了地方——用哈希当加密、用错模式、甚至自己"发明"算法，这些坑比算法本身的数学缺陷更常见。

误用一：用 MD5/SHA1 直接存储密码

MD5、SHA1 是哈希算法，不是加密算法——这是最容易混淆的概念。加密算法（如 AES）是可逆的：有密钥就能解密还原明文；哈希算法是单向的：只能从明文算出摘要，理论上不能从摘要反推明文。 但"理论上不能反推"不等于"实际很安全"——对于密码这种值域很小、用户习惯高度重复（"123456"、"password"排在使用频率最前列）的数据，攻击者可以：

  • 彩虹表攻击：提前把海量常见密码的 MD5/SHA1 值算好存成表，拿到泄露的哈希值直接查表反查明文，因为 MD5/SHA1 计算速度极快，攻击者能在短时间内枚举海量密码。
  • 撞库/字典攻击：不加盐（salt）的情况下，两个用户设置了相同密码，哈希值也完全一样，一旦一个库被破解，同密码的所有账号一起沦陷。
正确做法：密码存储应该用专为密码设计的慢哈希算法——bcrypt、scrypt、Argon2。它们的共同特点是刻意设计得很慢（可配置迭代次数/内存消耗），并且内置加盐（salt），让"批量彩虹表攻击"在计算成本上变得不划算。 MD5/SHA1 的设计目标是"快速校验数据完整性"（比如文件校验和），这个"快"的特性放到密码存储场景恰恰是灾难——攻击者破解也一样快。
Java 中的正确用法（Spring Security 示例）
// 反例：自己用 MD5 加个盐就当加密方案 —— 仍然不安全，MD5 太快，暴力破解成本低
String hash = DigestUtils.md5Hex(password + salt);

// 正例：使用 BCrypt，内置加盐 + 可配置计算强度（cost factor）
PasswordEncoder encoder = new BCryptPasswordEncoder();
String hash = encoder.encode(rawPassword); // 存这个结果，而不是原始密码
boolean matched = encoder.matches(rawPassword, hash); // 登录时校验

误用二：块加密使用 ECB 模式

AES 这类分组加密算法需要指定一种"工作模式"来处理多个数据块，ECB（Electronic Codebook，电子密码本）模式是最简单也最容易被误用的一种：它把明文切成固定大小的块，每一块独立加密，相同的明文块永远产生相同的密文块。

问题："相同输入产生相同输出"这个特性会把明文的结构规律直接泄露到密文里。经典例子是对一张图片用 ECB 模式加密，加密后的图片依然能隐约看出原图轮廓——因为大片相同颜色的像素块（对应相同明文）被加密成了相同的密文块，图案规律被完整保留了下来。

正确做法是使用 CBC（Cipher Block Chaining）或 GCM（Galois/Counter Mode）等模式：CBC 让每一块的加密结果依赖前一块的密文（引入链式依赖，相同明文块在不同位置会产生不同密文），GCM 则更进一步，在加密的同时提供完整性校验（AEAD，认证加密），是目前 TLS 1.3 等现代协议的首选模式。

模式特点是否推荐
ECB各块独立加密，相同明文块产生相同密文块，泄露数据规律不推荐，绝大多数密码学库和安全规范都明确禁止
CBC块间链式依赖，需要 IV（初始向量），能消除 ECB 的规律泄露问题可用，但需要正确处理 IV 和填充（padding oracle 攻击需留意）
GCM加密 + 完整性校验一体化（AEAD），性能好，是现代协议首选推荐，TLS 1.3 等现代协议的默认选择

误用三：自己发明或魔改加密算法

另一个常见但更隐蔽的误用是：工程师觉得标准算法"太重"或"看不懂"，于是自己设计一套简单的加解密逻辑（比如自定义的位移+异或组合），或者对标准算法做"优化"（比如减少 AES 的加密轮数以提升性能）。

为什么这是大忌：密码学是一个高度专业化的领域，标准算法（AES、RSA、SHA-256 等）都经过了全世界密码学家几十年的公开分析、攻击尝试和理论证明，能存活下来的都是扛住了大量已知攻击手段的"幸存者"。 自己设计的算法没有经过这种"千锤百炼"，看起来"够复杂就够安全"是一种典型的幻觉——历史上无数自研加密方案在专业分析下几分钟到几小时内就被破解，因为设计者的直觉往往抓不住密码学分析中真正的薄弱点（比如统计规律、旁路信息泄露）。
正确原则：永远优先使用语言/框架自带的、经过审计的标准密码学库实现（Java 的 javax.crypto、Spring Security 的 PasswordEncoder 体系等），不要"重新发明轮子"。这也是 Kerckhoffs 原则的现代体现：系统的安全性应该只依赖密钥的保密，而不应该依赖算法本身的保密——公开、标准、久经考验的算法反而比"自己保密的独门算法"更值得信赖。
