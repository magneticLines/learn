传输安全

HTTPS 原理回顾

TLS 握手的具体步骤在 10-计算机网络模块已经讲透，这里换一个视角——不再关心"怎么握手"，而是关心"HTTPS 到底替 Web 安全挡住了什么"。

与 10-计算机网络模块的关系：非对称加密协商密钥、证书链验证、TLS 握手的四次/一次往返细节，10-计算机网络的安全 Tab 已经详细讲过，这里不再重复展开。这一篇聚焦在：从 Web 安全的角度看，HTTPS 到底解决了哪三类问题，以及为什么"全站 HTTPS"在今天是底线要求而不是加分项。

安全视角：HTTPS 提供的三重保障

把 TLS 握手的技术细节抽象掉之后，HTTPS 相对于明文 HTTP，本质上是在传输层之上叠加了三种安全属性，分别对应三类不同的攻击：

保障解决的问题底层机制（简述）
防窃听
机密性		 中间人（同一 WiFi 下的攻击者、运营商）能直接看到明文内容，包括账号密码、Cookie、隐私数据 握手协商出的对称密钥对所有应用层数据加密
防篡改
完整性		 中间人在传输途中修改响应内容，比如注入广告脚本、篡改下载文件、插入恶意 JS 每条记录附带 MAC（消息认证码）或 AEAD 加密模式自带完整性校验
防冒充
身份认证		 用户以为在访问银行官网，实际连接的是攻击者伪装的钓鱼站点（DNS 劫持、ARP 欺骗等场景） CA 证书链验证服务器公钥确实属于该域名的合法持有者
具体到 Web 安全的例子：如果登录页面用 HTTP 明文传输，即便密码本身做了前端加密，Session Cookie 通常也是明文暴露在响应头 Set-Cookie 里的——攻击者在同一局域网用抓包工具（如 Wireshark、Fiddler）就能直接截获 Cookie，拿去伪造请求实现会话劫持，完全不需要破解密码。这也是为什么"只给登录页加 HTTPS，其他页面用 HTTP"这种做法是不够的——只要 Cookie 在任何一次请求中以明文传输过，就有被窃听的窗口。

为什么"全站 HTTPS"是今天的底线，而不是可选项

早期互联网的常见做法是"只在登录、支付等敏感页面用 HTTPS，其余静态页面用 HTTP"，理由是 HTTPS 有额外的计算开销。但这个思路在今天已经被证明是不成立的：

  • Cookie 没有页面边界：浏览器发请求时，只要域名匹配就会带上 Cookie，不区分这个页面是不是"敏感页面"——只要有一个页面用了 HTTP，登录态 Cookie 就有暴露机会。
  • 混合内容风险：HTTPS 页面里如果引用了 HTTP 的资源（图片、脚本），会被浏览器拦截或降级警告，且这类资源本身仍可被篡改，比如中间人在 HTTP 加载的 JS 文件里注入恶意代码。
  • 硬件和算法的进步早已抵消了性能顾虑：现代 CPU 普遍有 AES-NI 硬件加速，TLS 1.3 又把握手往返砍到了 1-RTT（甚至 0-RTT），性能开销已经不是拒绝全站 HTTPS 的理由。
  • 免费证书降低了成本门槛：Let's Encrypt 等机构提供免费自动续期证书，"证书太贵"也不再是借口。
结论：现代 Web 安全的基本要求是全站强制 HTTPS（配合 HSTS 响应头 Strict-Transport-Security，让浏览器记住"以后一律用 HTTPS 访问本站，不要再尝试 HTTP"），而不是挑几个"看起来敏感"的页面单独加密——安全防线只要有一个缺口，整体就形同虚设。
