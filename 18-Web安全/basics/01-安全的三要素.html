安全的三要素：机密性 / 完整性 / 可用性

学 Web 安全最容易一上来就背一堆漏洞名词——XSS、CSRF、SQL 注入……但如果不知道这些攻击到底"破坏了什么"，学完就是一堆孤立的知识点。CIA 三要素就是给所有安全问题分类的坐标系。

一句话讲清楚

信息安全领域有一个经典模型叫 CIA 三要素（和美国中情局没关系，纯粹缩写撞了）： 机密性（Confidentiality）、完整性（Integrity）、可用性（Availability）。 这不是某本书发明的口号，而是安全行业几十年总结出来的共识——几乎所有安全事件，最终都可以归结为这三者中的一个或多个被破坏了。

为什么需要这个模型：如果没有一个统一的分类框架，你面对"数据库被拖库了""网站被刷崩了""订单金额被改了"这些五花八门的事故时，很难判断"这算不算安全问题、算什么级别的安全问题"。CIA 三要素给了你一把统一的尺子——不管什么攻击手法，先问一句"它破坏的是机密性、完整性还是可用性"，思路立刻就清晰了。

机密性（Confidentiality）

信息不能被未授权的人获取。换句话说，只有被允许看到某份数据的人才能看到，其他人即使拿到了这份数据的载体（比如数据库文件、网络传输的数据包），也不该能读懂或读到内容。

破坏机密性的例子：数据库被 SQL 注入拖库，用户的手机号、密码泄露；服务器没做权限校验，任何人改一下 URL 里的用户 ID 就能看到别人的订单（水平越权）；传输过程没加密，路由器上的人用抓包工具就能看到你的登录密码（中间人攻击）。

生活类比：你把信寄给朋友，机密性对应的是"这封信只有收信人能拆开看，路上不会被人偷拆偷看"。加密、访问控制、脱敏展示，都是为了保护机密性。

完整性（Integrity）

信息在传输和存储过程中不能被未授权地篡改。即使攻击者看不到内容（机密性没被破坏），如果他能悄悄改掉内容而不被发现，同样是严重的安全问题。

破坏完整性的例子：攻击者篡改支付请求里的订单金额，把 1000 元改成 1 元；CSRF 攻击伪造你的身份发起转账请求，篡改了"谁转钱给谁"这件事的真实性；文件在传输途中被中间人替换成带病毒的版本。

生活类比：还是寄信，完整性对应的是"信件内容在路上没有被人偷偷改过一个字"。数字签名、消息摘要（MD5/SHA 校验）、CSRF Token，都是为了保护完整性——让接收方能验证"这份数据确实没被改过"。

可用性（Availability）

系统在被合法用户需要的时候，能够正常提供服务。信息没泄露、没被改，但如果用户想用的时候用不了，同样是安全事故——只是攻击者破坏的不是数据本身，而是"服务持续可用"这件事。

破坏可用性的例子：DDoS 攻击用海量请求把服务器打瘫痪，正常用户完全无法访问；勒索软件把硬盘数据加密锁死，业务系统无法启动；一个恶意接口被无限调用耗尽数据库连接池，导致整个网站响应超时。

生活类比：邮局被人堵门，导致正常人没法寄信收信——信件本身没被看也没被改，但"寄信"这个服务瘫痪了。限流、容灾、负载均衡、DDoS 防护，都是为了保护可用性。

三要素与常见攻击的对应关系

要素被破坏时的表现典型攻击/事故常见防护手段
机密性 不该看到的人看到了数据 SQL 注入拖库、越权查看他人数据、明文传输被窃听 加密、访问控制、脱敏、HTTPS
完整性 数据被悄悄篡改而不自知 CSRF 伪造请求、篡改支付金额、中间人替换数据 数字签名、消息摘要、CSRF Token、参数化查询
可用性 需要时服务不可用 DDoS、资源耗尽、勒索软件锁死数据 限流、容灾备份、负载均衡、WAF

值得注意的是，很多攻击其实同时破坏多个要素。比如一次成功的 SQL 注入，攻击者既可能读走数据（破坏机密性），也可能顺手执行 UPDATE/DROP TABLE（破坏完整性和可用性）。学后面具体的攻击手法时，建议养成习惯：先问自己"这个漏洞主要威胁的是 CIA 里的哪一项（或哪几项）"，这样知识点才不会零散。

常见误区

误区："安全就是防止数据泄露"——很多新手把安全等同于机密性，只关心"数据会不会被偷看"，却忽略了完整性和可用性同样重要。事实上金融行业最怕的往往是完整性问题（金额被篡改），而互联网公司最怕的常常是可用性问题（服务瘫痪导致的业务损失）。
面试高频："什么是信息安全的 CIA 三要素？"——机密性(Confidentiality)、完整性(Integrity)、可用性(Availability)。追问"举例说明"时，记得从"信息有没有被未授权访问 / 有没有被篡改 / 服务是否持续可用"三个角度分别举一个具体攻击场景，而不是只会背名词。

核心价值总结

  • CIA 三要素是分类框架，不是三条孤立的规则：后面学的每一种攻击手法，本质上都是在破坏这三者之一或多个，用这个框架去归类，知识才不会散。
  • 安全设计的目标就是围绕 CIA 做取舍：不同系统对三者的优先级不同——银行系统可能更看重完整性，社交媒体可能更看重可用性，医疗数据可能更看重机密性。
  • 下一篇讲"怎么系统性地想清楚威胁在哪"——光知道三要素还不够，还得有方法论去主动识别自己的系统会在哪些环节被攻破。

与其他知识点的联系

  • 威胁建模（下一篇）：CIA 三要素是"目标"，威胁建模的 STRIDE 模型是"手段"——STRIDE 六类威胁最终都能映射回 CIA 中的一项或多项，先建立 CIA 这个坐标系，再学 STRIDE 才知道每一类威胁具体威胁的是什么。
  • XSS/CSRF/SQL 注入/越权访问：后面要学的每一种具体攻击手法，都可以先问一句"它主要破坏的是机密性、完整性还是可用性"——XSS/SQL 注入/越权主要威胁机密性，CSRF 主要威胁完整性，反序列化和文件上传因为能拿到代码执行权限，往往三者一起威胁，用 CIA 归类能帮你把零散的攻击手法组织成体系。
  • HTTPS 与加密算法：传输安全模块讲的 HTTPS 恰好对应 CIA 三要素中的"防窃听（机密性）"和"防篡改（完整性）"，而加密算法误用场景里讲的密码存储、AES 模式选择，本质上都是"用具体的密码学工具去实现机密性/完整性目标"的落地实践。
  • Session 固定攻击/越权访问：这两类攻击表面上看起来是"认证/授权"问题，但归根到底也是在破坏机密性（拿到别人的数据）或完整性（冒充别人发起操作）——CIA 三要素不只用来分类"攻击手法"，也可以用来分类"认证授权类漏洞"的最终危害。
