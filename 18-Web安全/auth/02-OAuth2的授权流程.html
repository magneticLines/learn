OAuth2 的授权流程

"用微信登录""授权访问你的 GitHub 仓库"背后都是 OAuth2 在工作——它解决的不是"你是谁"，而是"你能不能让别人代你办事"。

OAuth2 解决的问题

设想一个需求：第三方应用 A 想读取你在平台 B（比如微信、GitHub）上的部分数据（头像、昵称、联系人列表）。最原始的做法是——你把 B 平台的账号密码直接告诉 A，让 A 登录 B 帮你取数据。 这样做的风险显而易见：A 拿到的是你的完整账号密码，它想干什么都行，你也没法单独收回给 A 的权限，除非把密码整个改掉连累所有已知晓密码的第三方。

类比：OAuth2 就像是"给房产中介一把只能开客厅的临时钥匙卡，还能设定几点后自动失效"，而不是把家门的总钥匙直接复制一份给中介。 你（资源所有者）不需要把总钥匙（账号密码）交给中介（第三方应用 A），而是通过物业（平台 B 的授权服务器）发一张范围受限、可撤销、可过期的临时卡（access token）。

OAuth2 的核心目标就是：让用户能够授权第三方应用访问自己在另一个平台上的部分数据，而不需要把账号密码直接暴露给第三方应用，并且这个授权是可以限定范围（scope）、可以随时撤销的。

四个角色

角色对应例子职责
Resource Owner（资源所有者）你本人拥有数据，决定是否授权
Client（客户端）第三方应用 A想要访问你数据的应用
Authorization Server（授权服务器）微信/GitHub 的登录授权页验证身份、颁发 token
Resource Server（资源服务器）微信/GitHub 的开放 API凭 token 提供数据

授权码模式（Authorization Code）：最常用、最安全

OAuth2 定义了几种"授权模式"（grant type），其中授权码模式是 Web 应用场景下最常用、安全性最高的一种，因为它比其他模式多了一步"先换授权码、再换 token"，避免了 access token 直接暴露在浏览器地址栏或前端 JS 里。

1. 用户点击"用 XX 登录" → 第三方应用把用户重定向到授权服务器的授权页
    ↳ 带上 client_id、redirect_uri、scope（申请的权限范围）、state（防 CSRF 随机值）

2. 用户在授权页登录并同意授权 → 这一步用户是在平台 B 的域名下操作，账号密码从未经过第三方应用

3. 授权服务器重定向回第三方，URL 上带一个"授权码"（code）
    ↳ 授权码是一次性的、短时效的（通常几分钟），暴露在浏览器地址栏也相对安全

4. 第三方应用的后端服务器用这个授权码 + 自己的 client_secret，向授权服务器换取 access token
    ↳ 这一步是服务端到服务端的请求，client_secret 不会暴露给浏览器

5. 第三方应用拿 access token 调用资源服务器 API，获取用户的头像、昵称等授权范围内的数据
为什么要多一道"换 code"的步骤？如果授权服务器在第 3 步直接把 access token 放在重定向 URL 上（这是"隐式模式 Implicit Grant"的做法），token 就会经过浏览器地址栏、浏览器历史记录、Referer 头，泄露风险高得多。 授权码本身是一次性、短期有效、且必须配合只有服务端知道的 client_secret 才能换到 token，即使授权码在传输中被截获，攻击者没有 client_secret 也换不到真正的 access token。

其他几种授权模式速览（面试高频）

模式适用场景安全性/取舍
授权码模式
Authorization Code		 有后端的 Web 应用（第三方登录、开放平台对接） 最安全，多一次换 token 的步骤，client_secret 不暴露给浏览器
隐式模式
Implicit Grant		 早期纯前端单页应用（无后端） token 直接暴露在重定向 URL 里，风险较高，已被官方标准建议弃用，推荐改用授权码模式 + PKCE
密码模式
Password Credentials		 用户对客户端高度信任的自家应用（如自家 App 调自家 API） 用户要把账号密码直接交给 Client，违背 OAuth2 本意，只适合完全可信的第一方场景
客户端模式
Client Credentials		 服务间调用，没有用户参与（比如后台任务访问自己的资源） 不涉及用户授权，Client 用自己的身份直接换 token
面试高频："OAuth2 几种授权模式分别用在什么场景？" 记忆口诀：有后端用授权码，纯前端用授权码+PKCE（而不是过时的隐式模式），自家 App 信得过才用密码模式，机器对机器没用户用客户端模式。

与 Spring Security 的联系

07-Spring 模块中 Spring Security 生态下的 spring-security-oauth2-client 和 spring-authorization-server 分别对应 OAuth2 的 Client 端和 Authorization Server 端实现：前者帮你处理"用 GitHub/Google 登录"这类第三方登录集成时的重定向、code 换 token 等繁琐流程，后者可以让你的系统自己扮演授权服务器角色，对外提供标准 OAuth2 授权能力。理解上面这套角色和流程，再看 Spring Security 的配置类会清晰很多——那些配置项本质上就是在填 client_id、redirect_uri、scope 这些 OAuth2 规范里定义的参数。
