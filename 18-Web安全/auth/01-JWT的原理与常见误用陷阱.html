JWT 的原理与常见误用陷阱

JWT 是目前分布式系统里最主流的令牌格式，但它的"自包含"特性既是优点也是陷阱——很多线上事故都出在对这一点的误解上。

JWT 是什么：自包含的令牌

JWT（JSON Web Token）由三段用 . 拼接的 Base64URL 字符串组成：Header.Payload.Signature。 它要解决的核心问题是：在不查数据库、不查共享存储的情况下，服务端就能验证"这个请求方是谁、有没有权限"。

类比：传统 Session 像是"会员卡 + 后台会员系统"——卡本身只有一个编号，你的姓名、等级、权限都存在商场后台数据库里，前台一刷卡就要查后台。 JWT 像是"火车票"——票面上直接印着姓名、车次、座位号，检票员只需要验证票是不是伪造的（对着防伪水印/签名看一眼），不需要联网查中央数据库就能放行。 这就是"自包含"：信息就编码在令牌本身里。

三段结构长什么样

JWT 示例（换行仅为可读性，实际是单行用 . 连接）
// Header：声明令牌类型和签名算法
{ "alg": "HS256", "typ": "JWT" }
// → Base64URL 编码后成为第一段

// Payload：携带的业务数据（claims）
{ "sub": "user-1001", "role": "admin", "exp": 1735689600 }
// → Base64URL 编码后成为第二段，注意：这里只是编码，不是加密！

// Signature：对前两段的签名，防止内容被篡改
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secretKey  // 只有服务端知道的密钥
)

服务端验证一个 JWT 时，只需要用同样的密钥对 Header+Payload 重新算一遍签名，跟第三段比对是否一致——全程不需要访问数据库或 Redis，这正是它天然适合分布式、无状态场景的原因：任意一台无状态的服务器拿到密钥，都能独立完成验证。

三个高频误用陷阱

陷阱一：Payload 只是编码，不是加密

误区："Payload 看起来是一堆乱码，应该很安全吧？" 错。Base64URL 是编码，不是加密，任何人拿到 token 都能用浏览器控制台一行 atob() 还原出明文 JSON。 绝对不能把密码、身份证号、手机号这类敏感信息直接塞进 Payload，签名只能保证内容没被篡改，不能保证内容不被看到。

陷阱二：alg=none 算法混淆攻击

JWT 的 Header 里声明了签名算法，但这个声明是攻击者可控的——如果服务端验证逻辑天真地"读 Header 里写的算法，就用这个算法验证"，攻击者可以把 alg 改成 "none"，然后拿掉 Signature 段，一些实现松散的库会直接放行这种"未签名"的 token。 更进一步的变种是把 alg 从非对称的 RS256（服务端私钥签，公钥验）改成对称的 HS256，如果服务端傻乎乎地拿"公钥"当 HMAC 密钥去验证，攻击者用公开的公钥就能伪造签名。

防御：服务端必须硬编码/白名单强制指定验证时使用的算法，而不是信任 token 头里声明的 alg 字段。主流 JWT 库（如 jjwt、jose）新版本都已经默认要求显式指定允许的算法，这也是为什么升级库版本很重要。

陷阱三：过期时间设置不合理

JWT 一旦签发，在到期之前是"自我证明有效"的——服务端不查状态表，就意味着没法像 Session 那样一键让某个 token 失效。 如果 exp（过期时间）设置得过长甚至不设置，一旦 token 泄露（比如被 XSS 偷走、日志里打印出来），攻击者能长期冒充用户身份，且服务端束手无策。

常见应对方案：access token 设置很短的有效期（如 15 分钟），配合一个较长有效期、且能在服务端主动吊销的 refresh token 来换取新的 access token；对于必须支持"强制下线"的场景，通常还要维护一个黑名单（存 Redis，key 为 token 的 jti，过期时间设为剩余有效期）。

JWT vs 传统 Session：怎么选

维度JWT传统 Session
状态存储 无状态，信息编码在 token 里 有状态，服务端保存 session 数据
分布式扩展 天然友好，任意节点独立验证 需要共享存储（Redis）或粘性会话支持多机
主动失效 困难，需要额外维护黑名单机制 容易，服务端删掉 session 记录即可
体积 较大（携带完整 payload），每次请求都要传输 很小，客户端只带一个 session id
典型场景 微服务间调用、移动端 API、第三方开放平台 传统单体 Web 应用、对"随时踢人下线"要求高的系统
面试高频："JWT 怎么实现登出/强制下线？" 标准答案是：JWT 本身做不到真正的主动失效，工程上常见做法是维护一个短小的黑名单（记录已注销但还未过期的 token），或者干脆把 access token 有效期设得很短，配合 refresh token 机制把"失效"这件事转移到 refresh 环节处理。

与 Spring Security 的联系

在 07-Spring 模块讲到的 Spring Security 过滤器链中，JWT 通常通过一个自定义 OncePerRequestFilter 实现：从请求头 Authorization: Bearer <token> 中取出 token，验证签名和过期时间后，构造一个 Authentication 对象塞进 SecurityContextHolder，后续的权限校验逻辑（@PreAuthorize 等）就能正常工作——这也是为什么基于 JWT 的系统里 SecurityContext 通常配置成 STATELESS（不创建 HttpSession）。
