Session 固定攻击

不用偷 Session ID，而是"提前塞给你一个我知道的 Session ID"——这种反其道而行之的思路，是 Session 会话管理里最容易被忽视的漏洞。

攻击原理：不偷，而是"下发"

大多数会话相关的攻击（比如 XSS 窃取 Cookie）思路是"想办法偷到受害者已经登录的 Session ID"。但 Session 固定攻击（Session Fixation）换了个方向： 攻击者提前准备好一个自己知道的、合法的 Session ID，然后想办法让受害者用这个 Session ID 去登录。

类比：好比你去存包柜寄存行李，正常流程是柜子随机分配一个号码给你，只有你自己知道。 Session 固定攻击相当于攻击者提前偷偷把某个柜子的号码条硬塞到你手里，说"你就用 88 号柜子吧"——你把行李（登录凭证）放进了 88 号柜子，但攻击者本来就抄下了 88 号柜子的密码，你存完之后攻击者直接打开就能拿走你的行李。

完整攻击链

1. 攻击者访问目标网站，获得一个未登录状态的合法 Session ID（比如 SESSIONID=ABC123）

2. 攻击者诱导受害者使用这个 Session ID
    ↳ 常见手法：发一个带 Session ID 参数的链接，如 http://target.com/login?jsessionid=ABC123
    ↳ 如果站点支持通过 URL 传递 Session ID（早期 Java Web 常见的 URL 重写机制），这一步很容易得手

3. 受害者点击链接，用自己的账号密码登录
    ↳ 关键问题：如果服务端在登录成功后，沿用了登录前的这个 Session ID，没有生成新的

4. 这个 Session（ID=ABC123）现在被标记为"受害者已认证"
    ↳ 但攻击者从第 1 步开始就知道这个 ID，于是攻击者带着 SESSIONID=ABC123 直接访问网站，服务端一看"这个 Session 已认证"，直接把攻击者当成受害者放行
本质：问题根源不在于 Session ID 被"偷走"，而在于服务端把"登录前的匿名会话"和"登录后的已认证会话"当成了同一个东西，没有在权限发生变化（从匿名到已认证）的这个关键节点上做隔离。

防御手段：登录成功后必须换新 Session ID

核心防御原则只有一句话：用户登录成功的那一刻，服务端必须销毁旧 Session、生成一个全新的 Session ID，而不是在原来的 Session 基础上仅仅修改"已登录"状态位。 这样即使攻击者提前"种"下了一个 Session ID 并诱导受害者登录，登录成功后系统换发的新 ID 攻击者并不知道，攻击链在这一步就被切断。

Java Web 场景下的典型写法（Servlet API）
// 反例：登录成功后直接复用旧 session，只改认证状态 —— 存在固定攻击风险
HttpSession session = request.getSession();
session.setAttribute("user", user); // 危险：旧 session id 没有变化

// 正例：登录成功后先让旧 session 失效，再创建新的
HttpSession oldSession = request.getSession(false);
if (oldSession != null) {
    oldSession.invalidate(); // 关键一步：使旧 session id 失效
}
HttpSession newSession = request.getSession(true); // 生成全新的 session id
newSession.setAttribute("user", user);
Spring Security 里怎么处理：07-Spring 模块讲到的 Spring Security 默认就内置了这个防御——SessionManagementConfigurer 的 sessionFixation() 策略默认是 changeSessionId()（Servlet 3.1+ 环境下），登录成功后自动更换 Session ID，不需要开发者手写上面这段逻辑；只有在自己手撸认证逻辑、绕开框架默认行为时，才需要特别留意这一点。
面试高频："Session 固定攻击和 Session 劫持有什么区别？" Session 劫持（Session Hijacking）是攻击者窃取受害者已登录的合法 Session ID（比如通过 XSS 读 Cookie、网络嗅探）； Session 固定攻击是攻击者提前准备好一个 Session ID 并诱导受害者用它登录，攻击者从头到尾都"合法地"知道这个 ID，根本不需要"窃取"的动作。两者防御思路也不同：劫持防的是"ID 不能被看到/传输过程要加密"，固定防的是"登录前后 ID 必须不同"。
