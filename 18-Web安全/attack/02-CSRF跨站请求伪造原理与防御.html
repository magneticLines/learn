CSRF 跨站请求伪造原理与防御

浏览器有一个"忠诚"到近乎愚蠢的习惯：不管这个请求是从哪个页面发起的，只要目标域名匹配，就把这个域名的 Cookie 老老实实带上——攻击者要的就是这份"忠诚"。

核心机制：浏览器自动携带 Cookie

CSRF（Cross-Site Request Forgery，跨站请求伪造）指攻击者诱导已经登录目标网站的受害者， 在受害者毫不知情的情况下，让其浏览器向目标网站发出一个请求，而这个请求会被服务器当作受害者本人的合法操作来处理。

这一切之所以可能，是因为浏览器的一条基础规则：只要请求的目标域名匹配，浏览器就会自动把该域名下的 Cookie 附加到请求上， 完全不管这个请求是从哪个页面、由谁发起的。也就是说，Cookie 的携带只跟"请求发往哪里"有关，跟"请求从哪里发起"无关—— 这正是让 CSRF 得以生效的那个特性。

类比：好比你随身带着一枚印章（Cookie），只要有人拿着写好的文件让你"顺手盖个章"，你的手（浏览器）看到文件抬头是自己单位（目标域名）就会条件反射地盖上去——它不会去核实"这份文件到底是不是你自己想盖的"，只认抬头对不对。

攻击示例

假设受害者刚刚登录了 bank.com，浏览器里保存着有效的 session cookie。攻击者诱导受害者访问一个恶意页面 evil.com，页面里藏着这样的内容：

恶意页面 evil.com 的内容
<!-- 最简单的形式：img 标签的 src 会自动发起 GET 请求 -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" style="display:none">

<!-- 或者用自动提交的表单发起 POST 请求，绕过"只能 GET"的限制 -->
<form id="f" action="https://bank.com/transfer" method="POST">
  <input name="to" value="attacker">
  <input name="amount" value="10000">
</form>
<script>document.getElementById('f').submit()</script>

受害者的浏览器一打开这个页面，就会自动向 bank.com/transfer 发出请求，并且自动附带受害者在 bank.com 的合法 session cookie。服务器收到请求后，看到 cookie 有效，就认为"这是登录用户本人发起的转账操作"，于是照常处理—— 整个过程受害者可能连页面加载了什么都没注意到。

与 XSS 的本质区别（面试关键点）

核心区分：代码究竟在谁的浏览器里跑了？

XSS：攻击者的恶意代码真正在受害者浏览器里执行了——相当于攻击者"借用"了受害者的浏览器来跑自己的脚本， 脚本能读 Cookie、能操作 DOM、能发任意请求，能力几乎不受限。

CSRF：攻击者的代码从未在受害者浏览器里执行过一行——攻击者只是精心构造了一个请求， 让受害者的浏览器"顺手"把这个请求发出去，浏览器只是按照自己一贯的规则老老实实带上了本该带的 Cookie。 攻击者拿到的不是"执行能力"，而是"以受害者身份发出一次请求"的能力。

正因为 XSS 能让攻击者的代码真正跑起来，XSS 可以被用来绕过 CSRF 的防御：如果攻击者已经能在你的页面上执行 JS， 那么无论页面里藏了多隐蔽的 CSRF Token，攻击者的脚本都可以直接读取页面 DOM 或发起同源请求把这个 Token 拿到手， 再用它伪造出一个"合法"的请求——CSRF 防御在这种情况下形同虚设。这也是为什么很多人认为 XSS 是更根本、更危险的漏洞： XSS 一旦存在，往往可以连带击穿其他防线。

防御机制

1. CSRF Token

服务器为每个会话（或每个表单）生成一个不可预测的唯一 token，嵌入到页面表单的隐藏字段中； 提交状态变更类请求（转账、改密码等）时必须带上这个 token，服务器校验通过才处理。

服务端渲染的表单示例
<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="a1b2c3d4-随机不可预测的值">
  <input name="amount">
  <button>转账</button>
</form>

// attacker.com 上的伪造表单没有这个页面，读不到这个动态生成的 token
// 因为同源策略禁止 evil.com 的 JS 跨域读取 bank.com 页面的内容
// 所以伪造请求要么不带 token，要么带错——服务端校验直接拒绝

攻击者之所以拿不到这个 token，根源在于同源策略（Same-Origin Policy）禁止跨域读取另一个站点页面的内容——这也是本模块前面讲过的浏览器安全基石在这里的具体应用。

2. SameSite Cookie 属性

这是浏览器层面的防御，从根源上限制"跨站请求要不要带 Cookie"，无需服务端额外生成 token：

取值行为
Strict 完全禁止跨站请求携带该 Cookie，哪怕是从其他网站点击链接跳转过来也不带——最安全，但可能影响"从外部链接跳转后仍保持登录"的体验
Lax（多数浏览器默认值） 跨站的"顶层导航 + GET 请求"（如点击链接跳转）允许携带，但跨站的 POST 表单提交、图片/iframe 等子请求不携带——在安全与体验之间取折中
None 不限制，任何跨站请求都携带该 Cookie，必须同时搭配 Secure（仅 HTTPS 下发送）——用于确实需要跨站共享 Cookie 的场景（如第三方嵌入）

3. Referer / Origin 头校验

服务器检查请求头中的 Referer 或 Origin，确认请求确实是从自己的域名发起的，而不是从 evil.com 发起的。这种方式实现简单，但只能作为补充手段，不能作为主要防御—— 因为部分浏览器隐私设置、代理、企业网络策略可能会剥离或改写 Referer 头，导致合法请求被误判；反过来说， 它对没有精心处理请求头的简单攻击仍有一定拦截效果，属于"聊胜于无"的辅助层。

常见误区

误区：把 CSRF Token 存在 Cookie 里就万事大吉了。 如果服务端校验逻辑只是简单地"读取 Cookie 里的 token，看看跟请求里的 token 是否一致"，这个防御毫无意义—— 因为 Cookie 本身就是浏览器自动携带的东西，攻击者伪造的跨站请求同样会自动带上这个 Cookie 里的 token， 两边一对比自然"一致"，等于白防。正确做法是 token 必须存放在浏览器不会自动携带的位置—— 比如页面表单里的隐藏字段（需要能读到页面 DOM 才能拿到，跨站页面读不到），或是要求前端手动放进自定义请求头 （如 X-CSRF-Token），浏览器同样不会替跨站请求自动加上自定义头。校验的关键是"这个值是不是攻击者伪造请求时也能自动拿到"。
面试高频："CSRF 的防御为什么不能只依赖 Cookie？" 答案就是上面这条误区——凡是浏览器会自动附带的东西 （不管是 Cookie 本身，还是简单存在 Cookie 里的 token），都不能用来证明"这是用户主动发起的请求"， 因为伪造请求同样能自动带上它。真正有效的凭证必须依赖"跨站页面拿不到"这条同源策略保障的边界。

核心价值总结

  • CSRF 利用的是浏览器"无条件携带同域 Cookie"这一基础特性，攻击者的代码从未在受害者浏览器里真正执行过。
  • XSS 是"代码在受害者浏览器里跑"，CSRF 是"请求以受害者身份被发出"——这是理解两者关系、区分二者危害的关键，也是 XSS 能绕过 CSRF 防御的原因。
  • CSRF Token 必须放在浏览器不会自动携带的位置（表单隐藏字段/自定义请求头），SameSite Cookie 提供浏览器层兜底，Referer/Origin 校验只能作为补充。

与其他知识点的联系

  • XSS：前面讲过 XSS 一旦得手，攻击者的脚本可以直接读取页面 DOM 拿到 CSRF Token，让 CSRF 防御形同虚设——这也说明本模块把 XSS 排在 CSRF 前面讲是有意为之：先理解"代码真的在浏览器里跑了"这种更强的攻击能力，再理解"只是借用了浏览器的自动行为"这种较弱但更常见的攻击。
  • SQL 注入：CSRF 伪造的请求如果打到一个存在 SQL 注入的接口上，两个漏洞会叠加放大危害——攻击者不仅能以受害者身份发起操作，还能借这个操作篡改或窃取数据库内容，理解漏洞组合利用是安全评估里的常见思路。
  • 威胁建模（STRIDE）：CSRF 对应 STRIDE 里的 Spoofing（冒充受害者的身份意图）和 Tampering（篡改本该由用户主动确认的操作），做威胁建模时凡是"会改变系统状态的接口"（转账、改密码、下单）都应该默认假设存在 CSRF 风险去检查。
  • 安全三要素：CSRF 破坏的核心是完整性——它不窃取任何数据，只是让一次"未经用户真实同意的操作"被当成合法操作执行，这也是为什么 CSRF Token 常被归类为"消息完整性/来源可信性校验手段"，与数字签名、消息摘要的设计目的是同一类问题。
  • Session 固定攻击：两者都利用了"浏览器/服务端对会话凭证的信任链"里的薄弱环节——CSRF 利用的是浏览器自动携带 Cookie 的机制，Session 固定攻击利用的是服务端在登录前后复用了同一个 Session ID，都是"身份凭证的边界没有被正确划清"导致的问题。
