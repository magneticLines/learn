文件上传漏洞 [核心]

把一个文件上传功能想象成一扇"只收快递、不开门"的传达室——真正的危险不是收了一个伪装成水果的包裹，而是传达室自己把包裹拆开、按包裹里的指令去执行。本篇内容参考《黑客攻防技术宝典：Web 实战篇》第 2 版。

核心问题：上传 ≠ 危险，上传 + 能被执行 = 危险

很多人以为"文件上传漏洞"就是"服务器允许上传恶意文件"，其实这只说对了一半。如果一个上传功能不严格校验文件的真实类型/内容， 攻击者确实可以上传一个伪装成图片的文件（比如文件名是 shell.jpg，但内容其实是一段可执行的脚本代码）——但光是这一步并不足以造成危害， 因为服务器如果只是把这个文件当成静态图片存起来、原样返回给浏览器，它就只是一张"打不开的图片"而已，没有任何代码会被执行。

真正致命的是两个条件同时满足：

  • 条件一：攻击者能控制上传文件的内容（这一步几乎总是成立，因为上传接口本来就是给用户提交内容用的）；
  • 条件二：上传目录允许"执行"落地的脚本文件（例如把一个 .jsp/.php 文件放进一个 Web 可访问、且服务器会将其当作脚本解析执行的目录）。

两者叠加，攻击者就获得了一个 Webshell——一个可以通过 URL 直接访问、并在服务器上执行任意代码的"后门脚本"，这本质上就是远程代码执行（RCE）。

类比：传达室（上传目录）只收快递不代表危险；危险的是传达室里有个人，只要包裹里写着"打开我并按里面的说明做事"，他就真的照做了。文件校验解决的是"能不能把危险包裹送进来"，而目录执行权限解决的是"送进来之后有没有人会照着执行"——两道防线缺一不可。

攻击是如何发生的

假设一个头像上传功能，前后端都只检查文件扩展名是否为 .jpg/.png。攻击者可能通过以下方式绕过：

  • 扩展名伪造：把 Webshell 命名为 shell.jsp 后，尝试改成 shell.jsp.jpg 或利用服务器的解析怪癖（比如老版本 Apache 对"未知后缀"的处理规则，会把 evil.jsp;.jpg 这种多后缀文件仍按 .jsp 解析执行）；
  • Content-Type 伪造：上传时把 HTTP 请求里的 Content-Type 字段改成 image/jpeg，而后端如果只信任这个客户端可自由填写的字段，就会被骗过；
  • 目录穿越配合：如果上传路径处理不当，甚至可以尝试把文件写到 Web 根目录下的可执行路径中。

下面是一个"概念性"的 Webshell 示意（仅用于理解原理，不是可直接使用的攻击代码）——它的核心逻辑极其简单：接收一个请求参数，把参数内容当作系统命令去执行：

概念示意（JSP 伪代码，说明原理用）
// 危险之处在于：请求参数完全由攻击者控制，却被直接传给了命令执行函数
<%
  String cmd = request.getParameter("cmd");  // 攻击者通过 URL 参数传入命令
  Process p = Runtime.getRuntime().exec(cmd);      // 服务器代为执行，等于交出了主机控制权
  // 之后把 p 的输出流写回响应，攻击者就能像开了一个远程终端一样操作服务器
%>

只要这个文件能被放进一个"会执行脚本"的目录，并且能通过 URL 访问到，攻击者此后就无需再依赖上传接口——直接访问 http://target/uploads/shell.jsp?cmd=whoami 就能在目标服务器上为所欲为。

防御措施：为什么这么做能生效

1. 校验文件真实类型而不只是扩展名

文件名后缀和浏览器提交的 Content-Type 都是攻击者完全可控的字符串，仅凭这两者做判断形同虚设。 正确做法是读取文件内容的魔数（Magic Bytes / 文件签名）来判断真实类型——例如 PNG 文件必然以 \x89PNG 开头， JPEG 以 \xFF\xD8\xFF 开头。伪装文件名可以随便改，但真实的文件头很难在保持"能被当作图片使用"的同时被随意篡改。

2. 上传目录禁止执行权限

这是最根本的一道防线：在 Web 服务器或操作系统层面配置，让上传目录无论存放什么文件都不会被当作脚本执行。 比如 Nginx 可以针对上传路径单独配置，拒绝将其中的文件交给 PHP-FPM 等脚本解析器处理。这样一来，即便有恶意文件通过了内容校验混了进来， 它也只是一个"打不开的脚本文件"，没有任何执行入口——相当于把"能不能执行"这个必要条件直接掐断。

3. 文件重命名

保存文件时用随机生成的文件名（如 UUID）替换用户上传时的原始文件名，彻底剥夺攻击者对文件名的控制权。 这能防住依赖"精心构造的文件名/多重后缀"的绕过手法，也让攻击者难以直接猜测或访问到已上传的文件路径。

4. 隔离存储域名

把用户上传的文件放到一个独立的域名或子域名下提供访问（例如专门的静态资源域名/CDN），该域名不携带主站 Cookie、 也不具备任何脚本执行能力。这样即便前面的防线都被突破，恶意文件也无法在主应用的域名/会话上下文中执行，从而无法窃取主站用户的登录态或触发同源攻击。

四道防线对比

防御措施防住了什么单独使用是否足够
校验真实文件类型 伪装成图片的恶意文件被识别并拒绝 否——总可能存在解析器层面的绕过技巧
上传目录禁止执行 即使文件混入，也无法被当作脚本运行 否——配置疏漏、目录迁移都可能失效
文件重命名 攻击者无法通过构造文件名触发解析怪癖，也难以定位文件 否——不解决"内容"层面的风险
隔离存储域名 即使文件被执行，也不影响主站会话与用户数据 否——治标不治本，仍应堵住上游漏洞
纵深防御（Defense in Depth）：这四道防线要同时使用，而不是挑一个自认为"最强"的来用。 任何单一检查都可能被足够有创意的攻击者找到绕过方式（历史上各种 Web 服务器、框架的解析怪癖层出不穷）， 只有层层叠加，才能保证"某一层被突破"不等于"全盘沦陷"。

常见误区

误区："前端 JS 已经判断了文件扩展名，后端就不用再校验了。" 前端校验只是为了提升用户体验（少一次无效的网络请求），对安全毫无意义——攻击者只需打开浏览器开发者工具禁用校验， 或者干脆用 curl / Postman 直接向上传接口发请求，完全绕开前端页面。所有安全校验必须在服务器端完成，前端校验永远只是锦上添花。
面试高频："如何设计一个安全的文件上传功能？" 标准答案应覆盖：校验真实文件内容（而非扩展名/Content-Type）、 上传目录禁止脚本执行、随机化文件名、按需做病毒扫描、必要时使用独立域名隔离——并强调这是"纵深防御"而非单点校验。

与其他知识点的联系

  • 反序列化漏洞：两者都以拿到服务器的代码执行能力（RCE）为终点，区别在于攻击载体——反序列化靠字节流触发已有代码里的方法调用链，文件上传靠把一个新的脚本文件"种"进一个会执行脚本的目录，本质都是"让攻击者能控制的内容变成了会被执行的代码"。
  • SSRF：下一篇的 SSRF 和文件上传都体现了"信任了不该信任的输入来源"——文件上传信任了用户提供的文件内容和类型标记，SSRF 信任了用户提供的 URL，两者的防御思路也相似：都需要在服务端对"真实内容/真实目标"做严格校验，而不是相信客户端上报的元信息（Content-Type、文件名后缀，或是 URL 表面上看起来的目标）。
  • 威胁建模："上传目录禁止执行权限"这条防线正体现了纵深防御和威胁建模里"评估风险优先级"的思路——即便内容校验这一层被绕过，只要执行权限这一层没被突破，攻击链就断在半路，这提醒我们做威胁建模时要为同一个资产设计多层独立防线，而不是依赖单点检查。
  • 安全三要素：文件上传漏洞一旦形成 Webshell，攻击者获得的是服务器的执行能力，进而可以任意读写文件、执行系统命令——这意味着机密性、完整性、可用性三者都可能被一次性突破，危害等级和反序列化漏洞相当。
