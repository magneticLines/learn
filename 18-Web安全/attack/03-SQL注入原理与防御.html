SQL 注入原理与防御

把用户输入原样拼进 SQL 字符串，等于把"改写你程序逻辑的权力"交给了攻击者——这是 Web 安全里最经典、也是面试最高频的一道题。

SQL 注入的本质

SQL 注入（SQL Injection）的根本原因只有一句话：用户输入被直接拼接进了 SQL 语句字符串，导致输入内容不再只是"数据"，而是被数据库当成了 SQL 语法的一部分去解析。 正常情况下，一段 SQL 语句的结构（要查哪张表、用什么条件）应该完全由开发者决定，用户输入只应该填进"数据值"的位置。 一旦拼接方式不当，攻击者就能通过输入里的引号、注释符（--）、逻辑运算符（OR）等特殊字符，改变 SQL 语句本身的结构，而不仅仅是改变查询的数据值。

类比：拼接 SQL 就像把用户填的表单内容直接打印进一份"公文模板"里再交给审批系统执行。如果用户在姓名栏填的不是"张三"，而是"张三；同时批准我的所有请求"，而公文系统又不区分"这是姓名数据"还是"这是公文指令"，审批系统就会把整句话当成指令执行——这正是注入攻击的核心：让数据被误当成指令。

经典案例：登录绕过

假设登录逻辑是这样拼接 SQL 的（这是一段有漏洞的写法）：

VulnerableLogin.java
String sql = "SELECT * FROM users WHERE username='" + username +
             "' AND password='" + password + "'";
// 直接执行拼接出来的字符串，把用户输入当成了 SQL 语法的一部分
stmt.executeQuery(sql);

如果用户在密码框输入 ' OR '1'='1，用户名随便填一个 admin，拼接后的完整 SQL 字符串会变成：

拼接结果
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1'

注意这里的 WHERE 子句被彻底改变了语义：password='' 这个条件是否成立已经不重要，因为后面用 OR 接了一个恒真条件 '1'='1'。 根据 SQL 运算符优先级，整个 WHERE 表达式等价于 username='admin' AND (password='' OR 恒真)，结果永远为真——攻击者不需要知道密码，就能查出这一行用户记录，从而绕过登录校验。

另一种常见手法是利用注释符 --（后面通常再加一个空格）直接把密码校验部分注释掉。用户名填 admin' -- ，密码随便填，拼接结果是：

拼接结果（注释绕过）
SELECT * FROM users WHERE username='admin' -- ' AND password='任意值'

-- 之后的所有内容在 SQL 里都变成了注释，等于 AND password=... 这部分校验根本不存在，语句退化成"只要用户名是 admin 就能登录"。

危害不止于登录绕过：如果注入点出现在一个会把结果显示给用户的查询里（比如商品搜索、订单查询），攻击者还可以用 UNION SELECT 拼接出额外的一列结果，把 users 表里的用户名密码、其他业务表里的敏感数据一起"伪装成"正常查询结果显示出来，实现整表甚至整库的数据窃取。 即使页面不直接回显数据，攻击者也可以用盲注（Blind Injection）——通过构造条件语句观察页面返回的细微差异（有无数据、响应时间长短）逐个字符地猜出数据库里的内容，虽然慢，但理论上能把整个数据库"抠"出来。 也就是说 SQL 注入的破坏面从"绕过一个登录框"一直到"拖库"，级别相差极大。

根本防御：参数化查询 / 预编译语句

修复方式是使用 参数化查询（Parameterized Query），在 Java 里对应 PreparedStatement：

SafeLogin.java
String sql = "SELECT * FROM users WHERE username=? AND password=?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, username);
ps.setString(2, password);
ResultSet rs = ps.executeQuery();

这不是"看起来更安全"的写法，而是从机制上彻底堵死了注入的可能性。原理是：调用 prepareStatement(sql) 时，数据库驱动会先把带 ? 占位符的 SQL 模板单独发送给数据库并完成编译/解析， 这一步就已经确定了整条语句的结构——要查哪张表、有几个条件、条件之间是 AND 还是 OR，全都在这一步定型。之后 setString 传入的用户输入是作为纯数据值单独发送的， 数据库不会把它当成 SQL 文本重新解析、重新拼接进语句结构里。换句话说，无论用户输入里包含多少个引号、OR、--，它们都只会被当作字符串的字面内容去比较，不可能被解析成 SQL 语法。 结构编译和数据传值是两个独立的阶段，攻击者能控制的只有后一个阶段，而后一个阶段根本不参与语法解析——这就是参数化查询能"治本"的原因。

呼应已学内容：这里正好对应 09-数据库 模块中讲过的预编译语句（PreparedStatement）——如果你已经学过 JDBC 的预编译机制，会发现它不仅是性能优化手段（SQL 执行计划可以被数据库缓存、复用，避免每次都重新解析和生成执行计划）， 更是 SQL 注入防御的根本方案。一个机制同时解决了性能和安全两个问题，这也是为什么它是 JDBC 里几乎必须使用的 API。

为什么"转义特殊字符"不是好办法

很多初学者的第一反应是："那我把用户输入里的单引号 ' 转义一下（比如换成 \'）或者把危险字符过滤掉不就行了？" 这种思路本质上是黑名单过滤，存在结构性缺陷。

方案思路问题
转义 / 过滤特殊字符（黑名单） 识别并处理引号、分号、注释符等"危险字符" 需要开发者每一处拼接 SQL 的地方都记得正确处理，只要漏掉一处就前功尽弃； 不同数据库、不同编码（如宽字节注入）下转义规则不同，容易被绕过； 本质是"堵漏洞"而不是"消除漏洞产生的原因"
参数化查询 / PreparedStatement 从驱动层面把"语句结构"和"数据值"物理隔离 无论输入是什么字符组合都不会影响语句结构，防御是结构性、框架级的， 个别开发者忘记处理也不会破功（只要用对了 API）

简单说：转义是"希望每个人都不犯错"的人工防线，参数化查询是"让犯错这件事在机制上不可能发生"的结构防线。安全领域里，能用机制解决的问题，尽量不要依赖人的自觉。

ORM 框架下的注意事项

使用 MyBatis、JPA/Hibernate 等 ORM 框架时，框架底层通常也是通过 PreparedStatement 实现参数绑定的，所以按框架推荐方式写通常是安全的。以 MyBatis 为例：

  • #{} —— 会被编译成 ? 占位符，走参数化查询，安全。
  • ${} —— 是原样的字符串替换，直接把值拼进 SQL 文本里，等同于手写字符串拼接，不具备任何防注入能力。
UserMapper.xml
<!-- 安全：#{} 会生成 PreparedStatement 占位符 -->
<select id="findByName">
  SELECT * FROM users WHERE username = #{username}
</select>

<!-- 危险：${} 是纯字符串拼接，等价于手写 SQL 拼接 -->
<select id="findByNameUnsafe">
  SELECT * FROM users WHERE username = '${username}'
</select>
常见误区："我用了 ORM 框架，所以自动就防住了 SQL 注入。" 这是错的——ORM 只是把"是否使用参数化查询"这件事包装得更方便了，但像 MyBatis 的 ${} 这种 API 依然是原始字符串拼接，一旦某个字段名、排序字段、表名等需要动态拼接的地方图省事用了 ${}，注入漏洞照样存在。看到 ${} 时要额外警惕，能用 #{} 的地方绝不用 ${}。
面试高频："SQL 注入的原因和修复方式？" 标准答案的核心一句话是： "参数化查询让用户输入永远只作为数据值，不会被解析为 SQL 语法的一部分"——记住这句话，再补充"SQL 语句结构在编译阶段就已经确定，用户输入是之后单独绑定的数据"，基本能拿满分。

核心价值总结

  • 根因：用户输入被拼接进 SQL 字符串，导致输入能改变语句的语法结构而不只是数据值。
  • 根本修复：使用 PreparedStatement 参数化查询，把"语句结构编译"和"数据值绑定"拆成两个独立阶段。
  • 不要依赖黑名单转义——它是人工防线，参数化查询才是结构性、框架级的防线。
  • 用 ORM 不等于自动安全，警惕 MyBatis 的 ${} 等本质是字符串拼接的 API。

与其他知识点的联系

  • XSS：SQL 注入和 XSS 是同一个根因"数据被当成代码解析"在两个不同解析器上的体现——SQL 注入是数据库把用户输入解析成了 SQL 语法，XSS 是浏览器把用户输入解析成了 HTML/JS 语法，防御思路也高度对称：参数化查询把"数据"和"语句结构"隔离，输出编码把"数据"和"标签结构"隔离。
  • 反序列化漏洞：下一篇的反序列化漏洞是这个根因的第三个变体——字节流里的数据能触发方法调用链，本质仍是"数据决定了要执行的逻辑"，只是这次连接的是 Java 对象的构造/恢复过程而不是 SQL 语法解析器。
  • 越权访问：SQL 注入是"攻破查询条件本身"，越权访问是"查询条件写对了但没人检查这个条件该不该让当前用户查"——两者都可能导致读到不该读的数据，但前者是语法层面被绕过，后者是业务权限判断压根没做，面试时容易被拿来对比考察是否理解清楚。
  • 安全三要素：SQL 注入几乎能同时打穿三要素——读数据破坏机密性，改/删数据破坏完整性和可用性，这也是它长期占据漏洞排行榜前列、面试高频的原因：破坏面全、危害上限高、且历史悠久有大量真实案例。
  • 威胁建模：任何"用户输入会参与拼接查询逻辑"的输入点都是 SQL 注入的潜在攻击面，威胁建模第二步"识别攻击面"时，凡是搜索框、筛选条件、排序字段这类看似"只是查询参数"的输入，都需要专门确认走的是参数化查询而不是字符串拼接。
