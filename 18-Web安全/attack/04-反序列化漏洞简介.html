反序列化漏洞简介

反序列化本质上是"让外部数据决定接下来执行哪些代码"——一旦数据来源不可信，你就相当于把一部分程序控制权拱手让给了攻击者。这是比 SQL 注入更"底层"、危害也可能更大的一类漏洞（可导致远程代码执行）。

什么是反序列化漏洞

序列化（Serialization）是把内存中的对象转换成字节流，方便存储或网络传输；反序列化（Deserialization）则是反过来，把字节流还原成内存中的对象。 问题出在 Java 的原生序列化机制里：某些类实现了 Serializable 接口后，可以定义一个特殊方法 readObject()，这个方法会在反序列化过程中被自动调用，用来做一些自定义的对象恢复逻辑。

危险之处在于：如果反序列化的字节数据来自不可信来源（比如用户上传的文件、网络请求体），攻击者可以精心构造一段特殊的序列化字节数据。当这段数据被 ObjectInputStream.readObject() 处理时， 会自动触发目标类的 readObject() 方法，而如果这个类内部又调用了另一个类的某个方法、那个方法又调用了别的类……只要能找到一条"方法调用链"，最终执行到某个危险操作（比如执行系统命令、读写文件）， 就能实现远程代码执行（RCE，Remote Code Execution）。这条精心挑选、串联起来的调用链，安全领域称为 "gadget chain"（利用链）。

核心直觉：反序列化本质上是"让传入的数据决定接下来运行哪些代码"。正常情况下，程序执行什么逻辑是由开发者写死的代码决定的；但反序列化机制里，字节流的内容本身能触发一连串方法调用—— 相当于攻击者往你的程序里"塞了一小段脚本"，脚本内容就是他们精心挑选的对象结构。数据来源一旦不可信，你等于允许攻击者部分地编写你 JVM 里将要执行的行为。
真实案例：Java 生态里最著名的例子是 Apache Commons Collections 的反序列化利用链（2015 年被公开），因为该库里几个常用类（如 InvokerTransformer）组合起来正好能构成一条从 readObject() 通往任意代码执行的完整链路，而 Commons Collections 又被大量项目间接依赖，导致当时包括 WebLogic、WebSphere、JBoss 等在内的一大批 Java 中间件被曝出严重的 RCE 漏洞（俗称 "Java 反序列化漏洞潮"）。 这不是理论风险，而是造成过大规模真实入侵事件的漏洞类型。

为什么 readObject() 会自动执行是危险的

下面是一个极度简化的示意，用来说明问题所在——重点不是记住利用链的具体写法，而是理解"这个钩子函数在你的校验逻辑生效之前就已经跑起来了"：

DangerousExample.java（概念示意，非真实利用链）
class Payload implements Serializable {
    // readObject 会在反序列化时被 JVM 自动调用，早于你能做任何校验
    private void readObject(ObjectInputStream in) throws Exception {
        in.defaultReadObject();
        // 攻击者精心构造的字节数据，会让这里链式触发一系列方法调用
        // 最终可能落到 Runtime.exec() 这类危险操作上
        someChainedCall();
    }
}

// 你的业务代码只是老老实实地反序列化一份"数据"
ObjectInputStream ois = new ObjectInputStream(untrustedInputStream);
Object obj = ois.readObject(); // 这一行执行完，恶意逻辑可能已经跑完了

关键点在于：调用 readObject() 这一行代码本身，就已经把控制权交给了字节流里描述的对象类型及其内部逻辑，你的业务代码此时还完全没有机会对"这个对象到底是不是我期望的类型"做任何判断。 等你的代码拿到反序列化结果准备校验时，触发链可能早就执行完毕了。

防御原则

防御手段思路说明
最佳方案：不反序列化不可信数据 跨信任边界传输数据时，优先使用 JSON 等更安全的数据格式 把外部传入的字节反序列化成 DTO 时使用 JSON（如 Jackson/Gson），JSON 反序列化只是"按字段填充普通对象"，不存在 Java 原生序列化那种"任意触发方法调用链"的钩子问题
白名单限制可反序列化的类 如果确实无法避免对外部数据做 Java 原生反序列化，限制允许哪些类被还原 现代 Java 提供 ObjectInputFilter，可以指定一份允许的类名单，拒绝还原名单之外的任何类型；也可以使用某些安全库提供的类级别白名单机制
及时升级依赖 大量真实利用都依赖某个已知库里现成的 gadget chain 保持 Commons Collections、Fastjson 等历史上出过反序列化漏洞的库处于最新且已修复的版本，是性价比很高的防御措施
三条防线记忆法：能不用 Java 原生序列化就不用（换 JSON）→ 必须用就白名单限制类型（ObjectInputFilter）→ 无论如何都要保证依赖库是最新版本。三条防线是递进关系，优先级从上到下。

常见误区

误区："我的代码里从来没有主动调用过 readObject()，所以反序列化漏洞跟我没关系。" 这个想法很危险，因为很多框架和中间件会隐式地做反序列化，开发者往往意识不到： 比如 Java RMI（远程方法调用）在网络层就依赖对象序列化传输参数；一些缓存框架、Session 持久化机制（把 Session 对象序列化后存到 Redis 或磁盘）也在背后悄悄调用反序列化。 只要数据链路上有一处会对不可信来源的数据做 Java 原生反序列化，风险就存在——即使你自己的业务代码从没写过 ObjectInputStream。

核心价值总结

  • 本质：反序列化让外部字节数据能够触发预先存在的方法调用链，风险来自"数据能控制执行流程"这一设计。
  • 危害上限高：不同于登录绕过类攻击，反序列化漏洞的典型后果是远程代码执行（RCE），Apache Commons Collections 利用链是 Java 生态的经典案例。
  • 最佳实践：信任边界之外的数据优先用 JSON 等更安全的格式，避免使用 Java 原生序列化；无法避免时用类白名单兜底，并持续更新依赖。
  • 隐蔽性强：风险可能藏在 RMI、缓存、Session 持久化等框架背后，不能只检查自己显式写的代码。
