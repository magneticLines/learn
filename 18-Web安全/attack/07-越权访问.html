越权访问：水平越权与垂直越权

SQL 注入、XSS、CSRF、SSRF 这些漏洞都需要攻击者"构造点什么"才能触发；而越权访问往往连构造都不需要——只是把 URL 里的一个数字改了改，或者直接调用了一个后端本不该让它调的接口。它是最朴素、也最容易被忽视的一类漏洞。

什么是越权访问（Broken Access Control）

越权访问的本质是：系统对"当前登录用户是谁"做了校验（认证，Authentication），但没有对"这个用户能不能操作这个具体资源/这个具体功能"做校验（授权，Authorization）。 换句话说，系统知道"你是谁"，却没有再往下问一句"你有没有权限做这件事"。业界通常把它分成两类，攻击路径完全不同，必须分开理解。

水平越权：同级用户之间越界

水平越权（Horizontal Privilege Escalation，也常被称为 IDOR——Insecure Direct Object Reference）指的是： A 用户和 B 用户拥有完全相同的权限等级（比如都只是"普通客户"角色），A 却能够访问或修改本该只属于 B 的数据。

最典型的场景：一个普通用户在查看自己订单详情时，接口是 GET /api/orders/1001。他把 URL 里的 1001 改成 1002， 发现返回的是另一个陌生人的订单——收件地址、手机号、支付信息全部泄露。这个 bug 跟角色/权限层级毫无关系——两个人都是"注册用户"， 问题出在服务端压根没有检查"这个订单 ID 是不是属于当前发请求的这个人"。

类比：水平越权就像酒店房卡——你手里拿着自己 305 房间的钥匙卡，本不该能刷开 306 的门。如果酒店的门禁系统只检查"这张卡是不是有效卡"而不检查"这张卡对应的是哪个房间"，任何一张合法房卡就能开所有房间的门。

垂直越权：低权限跨越到高权限

垂直越权（Vertical Privilege Escalation）指的是：一个权限等级更低的用户，成功执行了本该只有更高权限等级的角色才能执行的操作。 最典型的场景：前端管理后台只给 role = ADMIN 的用户展示"删除用户"按钮，普通用户登录后界面上根本看不到这个按钮、也无法通过页面导航跳转过去。 但如果攻击者用浏览器开发者工具、Postman 或 curl 直接对 POST /api/admin/deleteUser 发起请求，而后端在处理这个请求时没有重新校验"当前会话用户的角色是不是 ADMIN"，请求依然会被正常执行。

常见误区（务必牢记）："前端隐藏了按钮/菜单项，就等于做了权限控制"——这是初学者最常见、也最危险的误解。 前端代码（HTML、JS、路由配置）是完全下发到浏览器、对客户端完全可见可控的，攻击者可以查看源码、篡改请求、绕过路由跳转，直接拼出被隐藏的接口地址发送请求。 前端从来不是安全边界，它只是用户体验层面的"引导"；真正能拦住攻击者的，只有服务端在每次请求时做的权限校验。

根本原因：只信任前端，后端没有对每个请求都校验权限

无论是水平越权还是垂直越权，追到底都是同一句话：只在前端做了权限判断，后端接口没有对每一次请求都校验"当前用户是否有权限操作目标资源"。 开发者容易产生一种错觉——"用户登录过了，说明他是合法用户，应该可以放行"；但"合法用户"和"有权限操作这个具体资源/执行这个具体操作"完全是两回事。 认证只回答"你是谁"，授权才回答"你能干什么"，这两步必须每次请求都独立完成，缺一不可。

水平越权：代码对比

OrderController.java · 有漏洞
// 有漏洞：只按 orderId 查询，没有校验这个订单是否属于当前登录用户
@GetMapping("/api/orders/{orderId}")
public Order getOrder(@PathVariable Long orderId) {
    return orderService.findById(orderId);
}
OrderController.java · 正确写法
// 正确：查出资源后，必须再校验资源归属，两步缺一不可
@GetMapping("/api/orders/{orderId}")
public Order getOrder(@PathVariable Long orderId,
                        @AuthenticationPrincipal User currentUser) {
    Order order = orderService.findById(orderId);
    if (!order.getUserId().equals(currentUser.getId())) {
        throw new AccessDeniedException("无权访问该订单");
    }
    return order;
}

垂直越权：代码对比

AdminController.java · 有漏洞
// 有漏洞：接口本身没有任何角色校验，谁登录了都能调
@PostMapping("/api/admin/deleteUser")
public void deleteUser(@RequestParam Long userId) {
    userService.delete(userId);
}
AdminController.java · 正确写法
// 正确：用 Spring Security 的方法级注解，在服务端强制校验角色
// 具体机制在 Spring Security 专题里详细展开，这里先记住这个模式
@PreAuthorize("hasRole('ADMIN')")
@PostMapping("/api/admin/deleteUser")
public void deleteUser(@RequestParam Long userId) {
    userService.delete(userId);
}

水平越权 vs 垂直越权

维度水平越权垂直越权
定义 同一权限等级的用户之间互相越界，访问/修改本不属于自己的数据 低权限用户越界执行了只有高权限角色才能做的操作
举例 把 /api/orders/1001 改成 /api/orders/1002 查看他人订单 普通用户直接调用 /api/admin/deleteUser 删除账号
越界方向 横向：同一层级，跨"资源归属" 纵向：跨"权限层级"
根本原因是否相同 是同一个 bug 模式：后端没有在每次请求时校验"当前用户是否有权限操作目标资源"。 只是水平越权体现在"资源归属"维度缺失校验，垂直越权体现在"权限层级"维度缺失校验——本质都是把权限判断的责任错误地留给了前端。

防御原则

服务端必须在每一次请求时都校验"当前用户是否有权限操作目标资源"，这个校验不能因为用户已经登录过一次、或者在别的接口已经校验过身份就被省略。 HTTP 是无状态协议，每一个请求都可能是被伪造、被重放、被绕过前端流程直接发出的——不能假设"用户能走到这一步，说明前面的调用链已经做过检查了"。 每个涉及敏感数据或敏感操作的接口，都要独立、显式地做归属校验或角色校验，绝不依赖调用方（尤其是前端）的"自觉"。

面试高频："讲讲水平越权和垂直越权的区别，以及越权漏洞的根本原因？" 标准答法：水平越权是同级用户跨资源归属越界，垂直越权是低权限跨越到高权限操作；两者的根本原因其实是同一句话——后端没有在每次请求时对"当前用户是否有权限操作目标资源"做独立校验，而错误地把这个判断责任交给了前端。 能一句话说清根本原因，比分别记住两个定义更能体现对这个漏洞的理解深度。

补充阅读：《黑客攻防技术宝典：Web 实战篇》（第 2 版）中对访问控制缺陷有更系统的分类和实战案例，感兴趣可以延伸阅读，不影响本篇核心内容的理解。

核心价值总结

  • 越权的本质是"认证"和"授权"被混为一谈：系统验证了"你是谁"，却没有验证"你能不能做这件事"。
  • 前端从不是安全边界：隐藏按钮、隐藏菜单、前端路由拦截，都只是体验层面的引导，攻击者可以完全绕过前端直接打接口。
  • 唯一可靠的防线在服务端，且必须逐接口、逐请求地做：不能因为"上游已经登录过"就省略当前接口的权限校验。
