垃圾收集

G1 收集器(Garbage First)

CMS 的问题根源在于它仍然困在"新生代/老年代物理连续内存"这个传统框架里。G1 的做法是：干脆把这个框架本身推翻重来。

一句话讲清楚

G1（Garbage First）最大的创新，是抛弃了"新生代、老年代各自占据一块连续物理内存"的传统设计，把整个堆划分成多个大小相等的独立区域（Region）， 每个 Region 都可以根据需要灵活扮演 Eden、Survivor 或 Old 的角色。G1 依然遵循分代收集理论，只是分代不再体现为物理隔离，而是逻辑上的角色标记。

传统收集器的堆布局（物理连续）
[ 　　新生代（Eden+2 Survivor）　　 ][ 　　　　　老年代　　　　　 ]

G1 的堆布局（逻辑分区，Region 化）
[E][O][S][E][O][O][H][E][S][O][E][O][H][E][O][S] ...
  E=Eden　S=Survivor　O=Old　H=Humongous（专门存放超大对象的特殊 Region）
  → 每个 Region 独立管理，角色可以随每次 GC 动态调整

"Garbage First"名字的由来

因为 Region 众多且相互独立，G1 可以对每个 Region 的"回收价值"（回收后能腾出多少空间 ÷ 回收这个 Region 需要花费的时间）做单独统计， 并维护一份优先级列表。每次收集时，G1 优先挑选那些垃圾最多、回收收益最高的 Region 下手，用最少的收集代价换取最多的空间——这正是"Garbage First"（垃圾优先）这个名字的来源。

这也是为什么 G1 不需要像 CMS 那样对整个老年代做一次完整收集——它可以只挑几个垃圾最多的 Region 来收集，回收效果好、耗时又可控，这是它能实现"停顿时间可预测"的关键机制之一。

停顿时间可预测模型

G1 支持设置一个期望的停顿时间目标（-XX:MaxGCPauseMillis，默认 200ms）。它会根据历史收集数据，预测每个 Region 的平均回收耗时， 在停顿时间预算内，尽量多地挑选高价值 Region 进行回收，而不是像早期收集器那样"要收就必须收完整个新生代/老年代"。

跨 Region 引用问题：记忆集 + 卡表

Region 化之后，一个新问题出现了：某个 Region 里的对象可能被其他任意 Region 中的对象引用（不再局限于"老年代引用新生代"这一种情况）。 如果每次收集一个 Region 都要扫描整个堆找引用，Region 化带来的优势就荡然无存了。

G1 给每个 Region 维护了一个记忆集（Remembered Set），记录"哪些其他 Region 中的对象引用了我"; 具体实现上依赖卡表（Card Table）——把内存划分成一个个固定大小的"卡片（Card）"，只要某张卡片内有跨 Region 的引用变化，就把这张卡标记为"脏"。 这样收集某个 Region 时，只需要扫描它的记忆集里记录的少量脏卡，而不必扫描整个堆，大幅提升了扫描效率。

G1 与 CMS 对比

维度CMSG1
内存布局 传统物理分代（新生代/老年代连续内存） Region 化，逻辑分代
回收算法 标记-清除（会产生碎片） 整体是标记-整理，局部（Region 间）是标记-复制，不产生碎片
停顿目标 无法主动设定，只能被动接受 可设置期望停顿时间目标，停顿时间可预测
回收范围 必须处理整个老年代 按 Region 价值排序，挑性价比最高的部分回收
最坏情况 碎片过多退化为 Serial Old，停顿暴涨 没有类似的整体退化风险（但 Region 数量、堆大小设置不当仍可能影响效果）
面试高频："G1 相比 CMS 的核心改进是什么？" 记住一句话：用 Region 化的内存布局，把"必须整体处理老年代"变成"按需挑选高价值区域处理"， 从而同时解决了 CMS 的碎片问题（局部用标记-整理/复制）和停顿时间不可控问题（可预测停顿模型）。

核心价值总结

  • G1 抛弃物理分代，改用 Region 化 + 逻辑角色标记，让内存管理更灵活。
  • "Garbage First"体现在：维护 Region 回收价值的优先级列表，优先回收性价比最高的 Region。
  • 记忆集 + 卡表解决了 Region 化之后的跨区域引用扫描效率问题，是支撑"只回收部分 Region"这一策略的基础设施。
  • G1 是"停顿时间可预测"模型的代表，可以设置期望停顿时间目标，这是它能取代 CMS 成为 JDK 9+ 默认收集器的核心原因。
