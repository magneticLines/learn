垃圾收集

Serial / ParNew 收集器

理论讲完了，从这一篇开始看 HotSpot 真实提供的收集器实现。Serial 是最古老的一个——它简单到近乎"原始"，却也因此成为理解一切后续收集器的最佳起点。

一句话讲清楚

Serial 是最古老、最简单的新生代收集器：单线程工作，进行垃圾收集时必须暂停所有其他用户线程（Stop The World）， 直到它收集结束。ParNew 是 Serial 的多线程并行版本——底层算法（标记-复制）完全一样，唯一区别是用多条 GC 线程同时进行收集工作，从而缩短停顿时间。

Serial：单线程为什么还能存在

"单线程还要 STW"听起来像是缺点，但 Serial 恰恰是 Client 模式下（内存较小、CPU 核心数少的场景）表现最好的收集器之一，原因是：

  • 没有线程交互开销：多线程收集需要线程创建、切换、同步等额外开销，在单核或双核环境下这些开销反而可能超过并行带来的收益。
  • 简单高效：没有线程管理的复杂度，专心做一件事，在小内存场景下停顿时间本身也很短。
Serial 收集器工作流程
用户线程运行中 ...
  → Eden 区满，触发 Minor GC
  → Stop The World：所有用户线程暂停
  → 单条 GC 线程执行标记-复制算法
  → 收集完成，恢复用户线程
（老年代对应 Serial Old，采用标记-整理算法，同样单线程 + STW）

ParNew：把"单线程"换成"多线程"

ParNew 除了收集时使用多条线程之外，其余行为（收集算法、Stop The World、对象分配规则等）与 Serial 完全一致，某种意义上可以把它理解成"Serial 的多线程翻版"。 在多核 CPU 环境下，ParNew 能利用多核并行的优势，比 Serial 更快地完成一次收集，从而缩短停顿时间。

ParNew 收集器工作流程
用户线程运行中 ...
  → Eden 区满，触发 Minor GC
  → Stop The World：所有用户线程暂停
  → 多条 GC 线程并行执行标记-复制算法
  → 收集完成，恢复用户线程
→ 停顿窗口仍然存在（依然是 STW），但停顿时长通常更短
关键概念辨析（面试高频）：并行（Parallel）≠ 并发（Concurrent）
  • 并行（Parallel）：特指多条垃圾收集线程之间同时工作，但用户线程仍然处于暂停状态。ParNew、Parallel Scavenge、Parallel Old 都是"并行"收集器。
  • 并发（Concurrent）：特指垃圾收集线程和用户线程同时工作（用户程序在继续运行，收集程序在另一个 CPU 上同步执行）。CMS、G1 部分阶段属于"并发"。
这个区分是理解下一篇 CMS 收集器"为什么特殊"的前提——CMS 是第一款真正做到收集线程与用户线程基本同时工作的收集器。

面试高频："Serial 收集器为什么现在还没被淘汰？" 因为在单核 CPU或者内存很小的场景下（比如某些嵌入式/客户端应用），多线程带来的线程切换开销可能比省下的收集时间还多，此时简单的单线程 Serial 反而是最优解。这提醒我们：没有绝对过时的算法/收集器，只有不匹配的应用场景。

核心价值总结

  • Serial 是最简单的新生代收集器，单线程 + STW，适合核数少、内存小的场景，是理解其他收集器的基准线。
  • ParNew 是 Serial 的并行版本，算法不变，只是用多条 GC 线程同时工作以缩短停顿时间。
  • 必须分清"并行"（多条 GC 线程同时干活，用户线程仍暂停）和"并发"（GC 线程和用户线程同时干活）这两个概念，这是后面理解 CMS/G1 的前提。
  • ParNew 长期是 CMS 收集器唯一广泛使用的新生代搭档，理解它是理解 CMS 组合工作模式的基础。

与其他知识点的联系

  • 标记-复制算法（05 篇）：Serial 和 ParNew 底层都采用标记-复制算法处理新生代，二者的差异只在于用单线程还是多条线程并行完成"标记+复制"这套动作——理解 05 篇的 Eden+Survivor 工作流程，才能理解本篇 Serial/ParNew 收集流程图里那一步"执行标记-复制算法"具体在做什么。
  • 分代收集理论（07 篇）：Serial（新生代）与 Serial Old（老年代）成对出现，正是分代收集理论最直接的落地——新生代对象朝生夕灭，交给标记-复制的 Serial 处理；老年代对象存活率高，交给标记-整理的 Serial Old 处理，两者的算法选择完全遵循 07 篇讲的弱/强分代假说。
  • STW 与锁竞争阻塞（并发模块）：Serial/ParNew 收集时的 Stop The World，本质和并发模块里线程因竞争锁被阻塞挂起是同一类"暂停换正确性"的手段——前者暂停所有用户线程保证标记和复制过程的数据一致性，后者暂停部分线程避免竞态条件，两者面对的都是"并发修改同一份数据"的根本矛盾。
  • 并行与并发的区分：本篇特别强调 ParNew 是"并行"而非"并发"收集器（GC 线程内部并行，但用户线程仍全部暂停）——这个区分是后续理解 CMS"收集线程与用户线程真正并发工作"为何是一次质的突破的必要前提，没有这一篇的铺垫，CMS 的"特殊"就无从谈起。
  • CMS 收集器的搭档关系（后续章节）：ParNew 长期是 CMS 唯一广泛使用的新生代搭档，这不是偶然的组合，而是因为 CMS 只解决老年代的并发收集问题，新生代仍然需要一个高效的标记-复制实现来配合——ParNew 恰好满足这个角色，这也是收集器之间"组合使用"而非"孤立选择"的典型例子。
