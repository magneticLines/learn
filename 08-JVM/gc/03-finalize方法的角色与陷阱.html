finalize() 方法的角色与陷阱

一个对象在真正死亡前，Java 曾经给过它"最后一次自救机会"——听起来很贴心，但这恰恰是 Java 历史上公认失败的设计之一。

一句话讲清楚

Object.finalize() 是 JVM 提供的一个"临终关怀"钩子：对象在被真正回收之前，如果重写了这个方法，收集器会调用它， 给对象一次做收尾工作（比如释放非 Java 内存资源）的机会。听起来合理，但执行时机不可控、可能永远不执行、性能代价高， JDK 9 起被标记为 @Deprecated，官方建议彻底不要使用。

可达性分析中对象的真实死亡过程

一个对象被判定为不可达后，并不会立刻消失，而是要经历一个"两次标记"的过程：

① 第一次标记：可达性分析后发现对象不可达
  → 检查该对象是否重写了 finalize() 且尚未执行过
  → 如果没重写 / 已执行过：直接判死刑
  → 如果重写且未执行：放入 F-Queue 队列，稍后由低优先级的 Finalizer 线程执行

② 执行 finalize()（不保证等待执行完成，也不保证按时执行）
  → 对象可以在这里把 this 重新赋给某个 GC Roots 可达的引用，实现“自我拯救”

③ 第二次标记：再次检查该对象是否可达
  → 仍不可达：真正被回收（finalize() 只有这一次机会，第二次不会再触发）
  → 变为可达：逃过一劫，存活下来

可以看到，一个重写了 finalize() 的对象，至少要经历两次标记、可能横跨两次垃圾收集才能真正被清理掉，这本身就是效率上的负担。

反例：自我拯救带来的混乱

SelfRescue.java（自救演示，生产代码严禁模仿）
public class SelfRescue {

    public static SelfRescue HOOK = null;

    @Override
    protected void finalize() throws Throwable {
        // 把 this 重新挂到一个 GC Roots 可达的静态字段上 —— 复活成功
        SelfRescue.HOOK = this;
        System.out.println("我又活过来了");
    }
}

但这个把戏只能玩一次——同一个对象的 finalize() 方法只会被调用一次，第二次不可达时不会再给自救机会，直接被回收。

为什么这是失败设计：
  • 执行时机不确定：由低优先级的 Finalizer 线程异步执行，无法保证何时运行，甚至无法保证运行结束。
  • 可能永远不执行：如果 JVM 提前退出，或者 F-Queue 队列积压严重，finalize() 可能永远没机会跑。
  • 性能开销大：每次都要额外经历一次标记+入队+异步执行，拖慢回收速度，且 Finalizer 线程本身可能成为瓶颈（历史上多个 OOM 案例都是因为 finalize 队列堆积）。
  • 可能"复活"对象：让对象生命周期变得不可预测，破坏了"不可达即死亡"的简单心智模型。

现代 Java 的替代方案

方案思路优点
finalize()（已弃用） 依赖 GC 时机异步调用 无——时机不可控、有性能开销、可能不执行
try-with-resources 实现 AutoCloseable，编译器在 try 块结束时自动调用 close() 确定性释放，作用域结束立刻执行，不依赖 GC
java.lang.ref.Cleaner 基于虚引用 + ReferenceQueue，对象被回收时触发注册好的清理动作 不会让对象复活，也不阻塞 GC 主流程，作为兜底清理手段
正例：try-with-resources 确定性释放资源
try (FileInputStream in = new FileInputStream("data.txt")) {
    // 使用 in 读取文件
}
// 无论正常结束还是抛异常，close() 都会在这里被立即调用，不依赖 GC 时机
面试高频："为什么不推荐用 finalize() 释放资源？" 一句话答案：它把"资源释放"这件本该确定性发生的事情，绑定到了本质上不确定的 GC 时机上。正确姿势永远是显式管理资源生命周期（try-with-resources），把"清理"当作兜底而非主力手段。

核心价值总结

  • finalize() 曾试图给对象一次"临终关怀"，但因执行时机不可控、可能被复活、性能开销大，被官方判定为失败设计。
  • 对象死亡要经历两次标记，重写了 finalize() 的对象只有一次自救机会，之后无论如何都会被回收。
  • 现代 Java 应该用 try-with-resources 做确定性的资源释放，用 Cleaner（基于虚引用）做不影响对象生命周期的兜底清理。
  • 记住这条规则：资源释放不能依赖 GC 时机，这是贯穿 Java 资源管理的一条重要原则。

与其他知识点的联系

  • 可达性分析（01 篇）：finalize() 的"两次标记"流程直接建立在可达性分析之上——第一次标记判定不可达后才检查 finalize()，自救成功的本质是对象在 finalize() 执行期间重新建立了一条到 GC Roots 的引用链，让第二次可达性分析的结果发生反转。
  • 四种引用类型与 Cleaner（02 篇）：本篇提到的替代方案 java.lang.ref.Cleaner 正是基于虚引用（Phantom Reference）+ ReferenceQueue 实现的——虚引用本身唯一的作用就是在对象被回收时收到一次通知，恰好绕开了 finalize() "可能复活对象"的问题，这是两篇内容前后呼应的直接证据。
  • 标记-清除等回收算法（04-06 篇）：finalize() 机制会让对象的死亡横跨两次收集，直接拖慢标记阶段的整体效率——无论后续用标记-清除、标记-复制还是标记-整理算法执行清理，凡是重写了 finalize() 且未执行过的对象，都必须先入队等待，这是所有算法共同要承担的额外成本。
  • STW 与收集器停顿：finalize() 由独立的、低优先级的 Finalizer 线程异步执行，不占用 STW 窗口，但如果 F-Queue 队列积压，会间接导致更多对象滞留在堆上无法及时清理，反而增加下一次 GC 的工作量——这是"看似不阻塞 GC，实际拖慢 GC"的隐蔽陷阱。
