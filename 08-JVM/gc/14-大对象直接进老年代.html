大对象直接进老年代

如果一个又长又大的字符串或数组，先在 Eden 出生，再被反复复制到 Survivor、又复制回来，你会立刻感觉到这个流程有多不划算。虚拟机也这么想，所以给大对象开了"绿色通道"。

一句话讲清楚

需要大量连续内存空间的对象（典型的比如很长的字符串、元素很多的大数组）会直接分配到老年代， 绕过新生代的 Eden 和 Survivor 区。这样做的目的是避免在 Eden 和两个 Survivor 区之间发生大量的内存复制。

为什么大对象不适合走标记-复制这条路

回忆 05 篇讲过的标记-复制算法：每次 Minor GC，存活对象都要被整体复制到另一块 Survivor 区。如果一个大对象恰好在这次 GC 中存活下来， 就要经历一次"大块内存搬家"，这个复制操作本身的开销和对象大小成正比。更糟的是，如果这个大对象连续扛过多次 Minor GC 还没死， 就要被反复复制很多次——每一次都是一笔不小的性能开销。

如果让大对象走默认流程（反例思路）
大对象在 Eden 创建 → Minor GC → 复制到 Survivor A（一次大块复制）
　→ 又一次 Minor GC → 复制到 Survivor B（又一次大块复制）
　→ ... 每次 GC 都要重复这个昂贵的复制过程

大对象直接进老年代（实际策略）
大对象创建时直接判定为"大对象" → 跳过 Eden/Survivor，直接分配到老年代
　→ 老年代用标记-清除/整理算法，本来就不做这种频繁的整体复制，避免了无谓的搬家开销

控制阈值：-XX:PretenureSizeThreshold

可以通过参数 -XX:PretenureSizeThreshold 设置一个字节数阈值，超过这个大小的对象会被直接判定为"大对象"，绕过新生代直接进入老年代分配 （注意：该参数只对 Serial 和 ParNew 两款收集器有效，Parallel Scavenge 等收集器一般不认这个参数，而是有自己的判断逻辑）。

常见误区/坑点：大对象直接进老年代虽然避免了复制开销，但也带来了新的风险——如果程序频繁创建大对象（比如反复解析大 XML、拼接超长字符串）， 老年代空间会被迅速消耗，提前触发 Major GC/Full GC，而老年代的收集通常比新生代慢得多、停顿更长。这也是《深入理解 Java 虚拟机》里提醒的一条经典调优建议：尽量避免在代码中创建过多"朝生夕灭"的大对象，比如避免在循环体内反复拼接长字符串（应使用 StringBuilder）。

核心价值总结

  • 大对象（如长字符串、大数组）直接分配到老年代，是为了避免标记-复制算法对大对象反复搬家造成的高昂复制开销。
  • 这条规则和上一篇"对象优先在 Eden 分配"是互补关系——绝大多数对象走默认规则，只有超过阈值的大对象才走这条"绿色通道"。
  • -XX:PretenureSizeThreshold 是控制这个阈值的参数，但只对部分收集器（Serial/ParNew）生效。
  • 反复创建大对象会加速老年代空间消耗、提前触发代价更高的 Full GC，这是编码阶段就应该规避的性能隐患。

与其他知识点的联系

  • 关联标记-复制算法（05 篇）：大对象绕开新生代的根本原因，就是标记-复制算法的复制成本和对象大小成正比——这条分配规则本质上是在"提前规避"一种已知算法的弱点，而不是凭空设计出来的特殊规则，理解算法本身的开销模型才能理解这条规则为什么存在。
  • 关联对象优先在 Eden 分配（13 篇）：13 篇讲的是默认规则，本篇是它的第一个例外——绝大多数对象走 Eden，只有超过 PretenureSizeThreshold 的大对象走"绿色通道"。两条规则合起来才说明了 JVM 分配决策不是"一刀切"，而是按对象特征分类处理。
  • 关联空间分配担保机制（16 篇）：大对象直接进老年代和担保机制导致的晋升是两种不同的"提前进入老年代"路径——前者是基于对象大小的主动判断，后者是基于 Survivor 空间不足的被动兜底，16 篇的面试高频提醒里也把这两条并列为对象晋升老年代的完整路径之一。
  • 关联收集器的选择（09 篇）：-XX:PretenureSizeThreshold 只对 Serial/ParNew 生效，Parallel Scavenge 等收集器有自己的判断逻辑——这提醒我们分配规则的具体实现细节是和收集器绑定的，脱离具体收集器讨论参数生效范围容易踩坑，这也是面试里容易被追问的细节。
