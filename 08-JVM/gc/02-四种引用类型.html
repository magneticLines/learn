垃圾收集

四种引用类型(强 / 软 / 弱 / 虚)

可达性分析告诉我们"能不能找到这个对象"，但真实世界还需要更细腻的控制——有些对象想留久一点（做缓存），有些对象希望"能用就用，不能用就丢"。这就是引用类型要解决的问题。

一句话讲清楚

JDK 1.2 之后，Java 把引用分为四种强度依次减弱的类型：强引用永不回收、软引用内存不够才回收、 弱引用见到 GC 就回收、虚引用形同虚设，只用来收一个"对象要没了"的通知。 这套分级设计的本质是：把"对象该不该被回收"这个决策权，从虚拟机独占，部分开放给应用程序自己控制。

逐个拆解

强引用（Strong Reference）—— 默认状态

平时写的 Object obj = new Object(); 就是强引用。只要强引用还在，垃圾收集器永远不会回收这个对象， 哪怕内存已经不足、即将抛出 OutOfMemoryError，也不会为了腾内存而牺牲强引用对象。

软引用（SoftReference）—— 内存告急才牺牲

用来描述一些还有用但非必须的对象。系统内存足够时不会回收，只有在即将发生内存溢出前，才会把这些对象列入回收范围。

SoftReference 用法（做缓存的经典场景）
SoftReference<byte[]> cache = new SoftReference<>(new byte[10 * 1024 * 1024]);

// 内存充足时，get() 一直能拿到对象
byte[] data = cache.get();
if (data == null) {
    // 内存不足时对象已被回收，get() 返回 null，需要重新加载
    data = reloadFromDisk();
}

弱引用（WeakReference）—— 逢 GC 必死

比软引用更弱：只能存活到下一次垃圾收集发生之前。不管当前内存是否充足，只要 GC 发生，弱引用关联的对象就会被回收。

呼应 ThreadLocal：Java 里 ThreadLocalMap 的 Entry 内部对 ThreadLocal 本身用的就是弱引用作为 key。 这样设计是为了防止内存泄漏：一旦外部不再持有 ThreadLocal 的强引用，下次 GC 就能把它回收，key 变成 null。 但要注意——这只解决了 key 的泄漏，value 依然是强引用，如果不手动调用 remove()，value 会随线程存活而一直挂在那里，尤其在线程池场景下极易造成内存泄漏，这是高频面试坑点。

虚引用（PhantomReference）—— 只为收一个通知

虚引用是最弱的一种关系，一个对象是否有虚引用的存在，完全不影响它的生存时间，也无法通过虚引用获取一个对象实例 （get() 永远返回 null）。它唯一的作用是：在这个对象被收集器回收时，收到一个系统通知（通过关联的 ReferenceQueue）。

四种引用对比

引用类型回收时机get() 能否取到对象典型应用场景
强引用 永不回收（除非置为 null 且不可达） 能 日常业务对象，默认引用方式
软引用 内存不足时才回收 回收前能 内存敏感的缓存（如图片缓存、二级缓存）
弱引用 下次 GC 时无条件回收 回收前能 ThreadLocalMap 的 key、WeakHashMap、监听器注册防泄漏
虚引用 不影响存活时间，回收时才收到通知 永远不能（get() 恒为 null） 跟踪对象被回收的活动，如 Cleaner 替代 finalize() 做资源清理
常见误区：很多人以为"软引用 = 缓存的银弹"，直接用 SoftReference 包一层就万事大吉。 但软引用的回收策略完全由 JVM 内部堆内存状况决定，不受你控制，缓存命中率会很不稳定。生产环境做本地缓存更常用 Caffeine、Guava Cache 这类有明确淘汰策略（LRU/LFU/超时）的方案，软引用更多是一种"最后兜底防 OOM"的手段，而不是主力缓存策略。

核心价值总结

  • 四种引用类型本质是给"对象的生存意愿"分级，把回收决策权部分让渡给应用程序。
  • 强引用永生，软引用扛到内存告急，弱引用扛不过下一次 GC，虚引用毫无生存意愿只求一个回收通知。
  • 弱引用是理解 ThreadLocal 内存泄漏问题的关键——key 会被回收，但 value 需要手动 remove()。
  • 虚引用配合 ReferenceQueue，是现代 Java（java.lang.ref.Cleaner）替代 finalize() 做资源清理的基础设施。
