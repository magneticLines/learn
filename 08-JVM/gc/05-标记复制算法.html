垃圾收集

标记-复制算法(Mark-Copy)

标记-清除留下的碎片问题该怎么解决？答案简单粗暴：干脆不清理碎片，把存活对象整体搬到一块全新的干净空间去。

一句话讲清楚

把可用内存按容量划分为大小相等的两块，每次只使用其中一块。当这一块用完了，就把还存活的对象一次性复制到另一块上， 然后把已使用的这块内存整个清理掉。因为每次都是对整块内存进行回收，不用考虑碎片问题，实现简单、运行高效，代价是可用内存直接缩小为原来的一半。

回收前（From 区在用）：
From: [A: 存活] [B: 垃圾] [C: 存活] [D: 垃圾]
To : [　　　　　　空闲　　　　　　]

① 复制：把存活对象 A、C 整体搬到 To 区，紧凑排列
② 清理：From 区整体清空

回收后（角色互换，To 变成新的使用区）：
From: [　　　　　　空闲　　　　　　]
To : [A: 存活][C: 存活][　　空闲　　]
  → 存活对象紧凑排列在一起，完全没有碎片，分配新对象只需要移动指针（指针碰撞）

为什么值得牺牲一半空间

表面看，1:1 划分浪费了 50% 的内存，这个代价似乎很大。但这个算法的效率还取决于另一个变量：存活对象的数量。 如果一块区域里大多数对象都是垃圾、只有少量存活，那么需要复制的对象很少，效率反而非常高——这正是它被选中用在新生代的原因（下一篇分代收集理论会详细展开）。

现代收集器如何优化"浪费一半空间"

HotSpot 虚拟机没有采用 1:1 的划分方式，而是采用了 Appel 式回收：把新生代划分为一块较大的 Eden 区和两块较小的 Survivor 区（默认比例 8:1:1）。 每次分配对象优先在 Eden 区，发生垃圾收集时，把 Eden 和其中一块 Survivor（记作 From）中仍存活的对象一次性复制到另一块 Survivor（记作 To）中，然后清理掉 Eden 和 From。

Eden（80%）+ Survivor From（10%）+ Survivor To（10%）

Minor GC 发生：
  Eden 中存活对象 + From 中存活对象 → 一起复制到 To
  Eden、From 整体清空
  From / To 角色互换，等待下一次 Minor GC

→ 每次新生代实际"浪费"的空间只有一个 Survivor（10%），而不是 50%
陷阱：Appel 式回收的前提是"新生代对象存活率很低"（弱分代假说），Survivor 区只需要装下少数存活对象即可。 但这只是一个统计假设，不是保证——当 Survivor 空间不足以容纳一次 Minor GC 后的存活对象时，需要依赖老年代的空间分配担保机制兜底（本组最后一篇会讲），多余的存活对象会直接晋升到老年代。

三种基础算法初步对比

算法是否移动对象内存利用率适用场景
标记-清除 不移动 高，但有碎片 存活率高、不适合频繁复制的场景（如老年代）
标记-复制 移动（复制到另一块空间） 低（Appel 式优化后接近 90%） 存活率低的场景（如新生代）——复制成本和存活对象数量成正比，越少越划算
标记-整理 移动（原地压缩） 高，无碎片 存活率高、不想复制却又要解决碎片的场景（如老年代）

核心价值总结

  • 标记-复制用"空间换效率"，把存活对象整体搬迁到另一块干净空间，彻底消灭内存碎片。
  • 代价是可用内存减半，但如果对象存活率低，需要复制的对象很少，反而是高效方案——这个前提正好契合新生代的对象特征。
  • HotSpot 用 Eden + 两个 Survivor（8:1:1）取代 1:1 划分，把浪费的空间压缩到 10%，这是标记-复制算法在工程上的经典优化。
  • 标记-复制不适合存活率高的场景（比如老年代），因为对象越多、复制代价越高，这也解释了为什么老年代不用这个算法。

与其他知识点的联系

  • 标记-清除算法（04 篇）：标记-复制是直接针对标记-清除"内存碎片化"缺点给出的解法——不再原地清除留下碎片，而是把存活对象整体搬到另一块干净空间，代价是要多预留一块内存，这是理解本篇"为什么值得牺牲一半空间"的直接背景。
  • 标记-整理算法（06 篇）：标记-复制和标记-整理都通过"移动对象"消灭碎片，区别在于搬到哪里——标记-复制搬到另一块预留空间（需要浪费空间），标记-整理原地向一端压缩（不浪费空间但同样要 STW）。两者是解决同一个问题的两种代价分配方式。
  • 分代收集理论（07 篇）：标记-复制"复制成本和存活对象数量成正比"的特性，恰好和弱分代假说（新生代对象朝生夕灭、存活率低）严丝合缝地契合——这正是分代收集理论选择"新生代用标记-复制"的根本原因，而不是一个任意的工程约定。
  • Serial/ParNew 收集器（08 篇）：Serial 和 ParNew 都是新生代收集器，底层采用的正是本篇讲的标记-复制算法（Appel 式 Eden+两个 Survivor），它们的差异只在于用单线程还是多线程执行这套复制流程，算法本身完全一致。
  • 老年代空间分配担保（分代收集理论提及）：本篇"陷阱"提到 Survivor 空间不足时依赖老年代空间分配担保兜底——这说明标记-复制并非孤立工作，新生代和老年代之间存在紧密的协作机制，这也是后续理解 Minor GC 触发 Full GC 场景的关键前提。
