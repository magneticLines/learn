空间分配担保机制

本组的最后一块拼图：新生代和老年代不是各自为战的，它们之间有一套"互保"机制，专门用来兜住"Minor GC 后 Survivor 区装不下存活对象"这种意外情况。

一句话讲清楚

在发生 Minor GC 之前，虚拟机会先检查老年代最大可用连续空间是否大于新生代所有对象的总空间。 如果这个条件成立，说明就算这次 Minor GC 后所有新生代对象都存活、全部要挪到老年代，老年代也装得下，那么这次 Minor GC 是安全的，可以放心执行。 如果不成立，则要进一步判断是否允许担保失败，来决定改成直接发起一次代价更高的 Full GC，还是冒险继续尝试 Minor GC。

这套机制到底在担保什么

回忆 05 篇讲的 Appel 式回收：新生代按 8:1:1 划分为 Eden、Survivor From、Survivor To，Minor GC 时把存活对象从 Eden 和 From 复制到 To。 但 Survivor 区默认只有新生代总容量的 10%，理论上完全可能出现某次 Minor GC 后存活对象的总大小超过了 Survivor（To）区容量的情况 （比如程序瞬间产生了大量长期存活的对象）。这时候多出来的存活对象没地方安放，唯一的出路是让它们直接晋升到老年代。 空间分配担保机制要保证的就是：当出现这种"Survivor 装不下"的意外情况时，老年代必须有足够空间接得住，不能出现"新生代腾出空间了，但存活对象却无处安放"的尴尬局面。

Minor GC 发生前的检查流程
检查：老年代最大可用连续空间 > 新生代所有对象总空间？
  → 是：绝对安全，正常执行 Minor GC
  → 否：查看是否允许担保失败（HandlePromotionFailure，JDK 6 Update 24 后此参数已失效，默认视为允许）
    → 允许：继续检查老年代最大可用连续空间是否大于历次晋升到老年代对象的平均大小
      → 大于：冒险执行 Minor GC（有风险，但历史数据显示大概率能扛住）
      → 小于/不允许：改为直接执行一次 Full GC，先腾出老年代空间再说
担保失败的后果：如果"冒险执行"的 Minor GC 过程中，存活对象总量真的超出了老年代剩余空间，会出现 Promotion Failed（晋升失败），这种情况下 JVM 不得不立刻发起一次 Full GC 来收拾残局，代价是一次 Minor GC 最终演变成了一次停顿时间更长的 Full GC， 这是生产环境排查"GC 停顿突然变长"问题时的一个经典线索。
面试高频："什么情况下对象会直接进入老年代？" 除了前两篇讲的"大对象"和"年龄达到阈值"，还应该补上第三种：Minor GC 时 Survivor 区空间不足以容纳全部存活对象，多出来的部分会直接分配到老年代——这正是空间分配担保机制存在的意义：它不是决定"谁该晋升"，而是提前检查"如果真的都要晋升，老年代兜不兜得住"。

核心价值总结

  • 空间分配担保机制是新生代和老年代之间的一处协作保障：在 Minor GC 前预先检查老年代是否有能力兜住"最坏情况"下的晋升需求。
  • 它要解决的核心问题是：Survivor 区默认只有新生代容量的 10%，一旦某次 GC 存活对象超出这个容量，多出来的对象必须能顺利晋升到老年代，而不是无处安放。
  • 检查失败且不允许冒险时，虚拟机会直接改为执行一次 Full GC，这也是"Minor GC 变身 Full GC"这一现象在日志中出现的原因之一。
  • 至此，本组把"判断死活 → 四种引用 → 三种基础算法 → 分代收集理论 → 具体收集器 → 对象分配与晋升规则"完整串联起来，构成了 JVM 垃圾收集的整体知识框架。
