垃圾收集

Parallel Scavenge / Parallel Old

ParNew 追求的是"停顿更短"，但如果你的场景根本不关心某一次停顿有多长，只关心程序跑一整天能干多少活呢？这就是 Parallel 系列收集器要解决的问题。

一句话讲清楚

Parallel Scavenge（新生代，标记-复制）和 Parallel Old（老年代，标记-整理）都是并行收集器， 但设计目标和 ParNew 有本质区别：ParNew 的目标是配合老年代收集器缩短停顿时间，而 Parallel Scavenge 的目标是达到一个可控制的吞吐量—— 也就是说，它关心的不是"这一次停顿多久"，而是"长时间运行下来，用户代码到底能拿到多少 CPU 时间"。

吞吐量（Throughput） = 用户代码运行时间 ÷ （用户代码运行时间 + 垃圾收集时间）。 比如虚拟机总共运行了 100 分钟，垃圾收集耗时 1 分钟，吞吐量就是 99/100 = 99%。吞吐量优先，意味着愿意接受单次停顿更长，以换取总体上 GC 占用的时间比例更低。

两个关键参数：把"控制权"交给使用者

Parallel Scavenge 最大的特色是提供了两个非常直接的调节参数，不需要理解收集器内部细节，只需要按业务需求"许愿"：

参数作用取舍
-XX:MaxGCPauseMillis 设置垃圾收集时最大的停顿时间（毫秒） 调小它 → 收集器会尽量缩短单次停顿（比如缩小新生代），但会导致收集更频繁，牺牲吞吐量
-XX:GCTimeRatio 设置垃圾收集时间占总时间的比例（值越大，容许的 GC 时间占比越小） 调大它 → 追求更高吞吐量，但可能导致单次停顿变长

这两个参数本质上是同一个跷跷板的两端——停顿时间和吞吐量不可能同时最优，Parallel Scavenge 允许你根据业务场景明确表态"我更看重哪一头"。

Parallel Old：Parallel Scavenge 的老年代搭档

在 Parallel Old 出现之前（JDK 5 之前），如果新生代用 Parallel Scavenge，老年代却只能选择 Serial Old（单线程）， 这会拖累整体吞吐量——新生代并行收集得很快，但老年代收集时全程单线程 STW，反而成为瓶颈。 JDK 6 引入的 Parallel Old 使用多线程 + 标记-整理算法，与 Parallel Scavenge 搭配后，才真正实现了"吞吐量优先"从新生代到老年代的完整闭环。

吞吐量优先收集组合
新生代：Parallel Scavenge（多线程，标记-复制）
老年代：Parallel Old（多线程，标记-整理）
→ 两者都追求最大化 CPU 利用率，适合后台批处理、报表运算、无需太多用户交互的服务

ParNew vs Parallel Scavenge 对比

维度ParNewParallel Scavenge
设计目标 配合老年代收集器，缩短停顿时间 达到可控的吞吐量目标
核心参数 无自适应调节机制 MaxGCPauseMillis / GCTimeRatio，支持自适应调节策略（-XX:+UseAdaptiveSizePolicy）
常见搭档 CMS（老年代） Parallel Old（老年代）
适用场景 需要和用户交互、追求响应速度的应用 后台运算类任务，追求单位时间内完成更多工作
面试高频："停顿时间和吞吐量为什么不能兼得？" 因为缩短停顿的常见手段（比如减小新生代空间、更频繁地收集）会增加 GC 触发次数， 从总时间占比看反而拉低了吞吐量；反过来追求高吞吐量往往意味着让内存空间更大、收集更少发生，但每次收集处理的对象更多，单次停顿必然更长。这正是"没有免费午餐"在 GC 领域的体现。

核心价值总结

  • Parallel Scavenge / Parallel Old 是"吞吐量优先"收集器的代表，目标是最大化 CPU 利用率，而不是最小化单次停顿。
  • MaxGCPauseMillis 和 GCTimeRatio 让使用者可以直接声明自己更看重停顿时间还是吞吐量，收集器会据此自适应调节新生代大小等参数。
  • 停顿时间和吞吐量是一个此消彼长的跷跷板，理解这一点比记住具体参数名更重要。
  • 这类收集器适合后台批处理、离线运算等不需要频繁用户交互的服务，而不适合对响应延迟敏感的在线服务（那是 CMS/G1 的战场）。

与其他知识点的联系

  • 关联 ParNew（08 篇）：两者都是新生代并行收集器、算法都是标记-复制，代码实现上甚至高度相似，但设计取向完全不同——ParNew 为"配合 CMS 缩短停顿"而生，Parallel Scavenge 为"最大化吞吐量"而生。理解这一对"孪生兄弟"的分歧点，是分辨后续所有收集器设计目标的基础参照系。
  • 关联 Serial Old：Parallel Old 出现之前，Parallel Scavenge 只能搭配单线程的 Serial Old，新生代再快也会被老年代的单线程 STW 拖累，这正是"吞吐量优先"这个目标必须在新老两代都落地才算完整的直接证据。
  • 关联对象分配规则（13～16 篇）：-XX:MaxGCPauseMillis 影响的不只是收集频率，还会反过来影响虚拟机的自适应调节策略（-XX:+UseAdaptiveSizePolicy）去动态调整 Eden、Survivor 的大小比例，这些调整直接决定了对象在 13～16 篇里讲的分配、晋升规则的实际触发时机。
  • 关联 G1（11 篇）：G1 同样支持设置停顿时间目标，但 Parallel Scavenge 只能通过牺牲吞吐量、缩小新生代来"被动"逼近这个目标，而 G1 靠 Region 化 + 优先回收高价值区域实现了更精细的停顿控制——对比二者能看出"停顿可控"这件事在收集器演进史上是如何一步步被做得更精确的。
