对象优先在 Eden 分配

讲完了各种收集器"怎么收"，接下来几篇转向另一半问题：对象一开始"分配在哪"、"什么时候会被搬家"。这是分代收集理论在对象分配层面的具体落地规则。

一句话讲清楚

绝大多数情况下，新创建的对象会优先在新生代的 Eden 区分配。当 Eden 区没有足够空间完成本次分配时，虚拟机会发起一次 Minor GC（新生代垃圾收集），先清理掉 Eden 和 Survivor 区中的死亡对象、腾出空间，再继续完成分配。

呼应 memory 组：为什么分配要走 TLAB

memory 组讲过 TLAB（Thread Local Allocation Buffer）机制——每个线程在 Eden 区预先分到一小块私有内存，对象优先在这块私有区域里分配， 不需要每次分配都进行加锁同步。这正是"对象优先在 Eden 分配"这条规则在并发场景下的工程优化：把 Eden 区的竞争问题，转化为线程内部无竞争的指针碰撞。

线程 A 请求分配对象
  → 先看自己的 TLAB 是否有空闲空间
  → 有：直接在 TLAB 内移动指针完成分配（无锁，极快）
  → 没有：TLAB 用完，重新向 Eden 区申请一块新的 TLAB
    → Eden 区也没有足够空间了 → 触发 Minor GC

Minor GC 的触发条件与特点

  • 触发时机：Eden 区空间不足，无法满足新对象的分配请求时。
  • 回收范围：只回收新生代（Eden + 两个 Survivor），不涉及老年代，所以速度通常很快。
  • 使用算法：标记-复制算法（呼应 05 篇），因为新生代对象存活率低，复制少量存活对象的代价很小。
  • 频率：由于新生代空间相对较小、对象"朝生夕灭"，Minor GC 发生频率远高于老年代的 Major/Full GC。
常见误区："Minor GC 会不会导致长时间停顿？" 一般不会——因为新生代空间小、存活对象少，一次 Minor GC 通常在几毫秒到几十毫秒内完成。 真正容易造成长停顿的是老年代的 Major GC / Full GC（对象多、存活率高，标记和整理的成本都更高）。区分这两者的触发条件和影响范围是排查 GC 停顿问题的基本功。
面试高频："新对象一定会分配在 Eden 区吗？" 不一定——这只是默认规则。下两篇要讲的"大对象直接进老年代"和"长期存活对象进入老年代"都是这条默认规则的例外情况，虚拟机会根据对象的大小和存活周期动态调整分配策略。

核心价值总结

  • 绝大多数新对象默认优先在 Eden 区分配，这是分代收集理论（弱分代假说）在对象分配阶段的具体体现。
  • TLAB 机制让每个线程在 Eden 区拥有私有分配缓冲区，把"多线程抢 Eden 区空间"的同步问题转化为线程内部无锁的指针碰撞，是这条默认规则能高效落地的关键工程手段。
  • Eden 区空间不足会触发 Minor GC，只回收新生代，通常速度很快、停顿很短。
  • "优先在 Eden 分配"是默认规则而非绝对规则，接下来两篇要讲的大对象直接进老年代、长期存活对象晋升老年代，都是对这条默认规则的重要补充。
