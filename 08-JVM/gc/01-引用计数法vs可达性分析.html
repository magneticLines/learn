垃圾收集

引用计数法 vs 可达性分析

垃圾收集要做的第一件事，不是"怎么回收"，而是"哪些对象已经死了"——这是整个 GC 体系的地基，地基选错了，后面的算法全部立不住。

一句话讲清楚

判断对象是否存活有两种主流思路：引用计数法（给每个对象挂一个计数器，谁引用我就 +1，引用失效就 -1，计数为 0 说明没人再用我了）和 可达性分析（从一批公认"肯定存活"的根对象出发，顺着引用关系向下搜索，搜不到的对象就是垃圾）。 Java 虚拟机选择了后者，原因很简单：引用计数法有一个无法自行解决的硬伤——循环引用。

反例：引用计数法搞不定循环引用

设想两个对象互相持有对方的引用，而外部已经没有任何变量指向它们：

循环引用示意（伪代码）
ObjectA a = new ObjectA();
ObjectB b = new ObjectB();
a.instance = b;  // a 引用 b，b 的计数器 = 1
b.instance = a;  // b 引用 a，a 的计数器 = 1

a = null;  // 外部不再持有 a，但 b.instance 还指着它 → a 计数器仍是 1
b = null;  // 外部不再持有 b，但 a.instance 还指着它 → b 计数器仍是 1
// 结果：a、b 互相“续命”，计数器永远不会归零，内存永远回收不了
这不是引用计数法“实现得不好”，而是这套思路本身的结构性缺陷：它只能感知局部的引用关系（我被谁引用了）， 没法感知"这一整团对象是否已经和外部世界失联"。要解决循环引用，必须跳出局部计数，站在全局视角做一次可达性判断——这正是可达性分析的思路。

正例：可达性分析（Reachability Analysis）

可达性分析的核心思想是：把一批称为 GC Roots 的对象作为起始点，从这些节点开始向下搜索， 搜索走过的路径称为引用链（Reference Chain）。当一个对象到 GC Roots 之间没有任何引用链相连时， 就证明这个对象是不可用的，可以被回收。

GC Roots（虚拟机栈中的局部变量、静态变量……）
  ├── 对象 A ── 对象 C （可达，存活）
  └── 对象 B （可达，存活）

对象 D ── 对象 E
  （D、E 互相引用，但都没有任何路径连到 GC Roots）
  → 不可达，即使互相引用也照样判死刑

可以看到，即使 D 和 E 互相引用（类似上面的循环引用场景），只要它们整体上够不着任何一个 GC Root，照样会被判定为垃圾——这正是可达性分析天然免疫循环引用问题的原因。

哪些对象可以作为 GC Roots？

  • 虚拟机栈（栈帧中的本地变量表）引用的对象——比如方法里正在使用的局部变量。
  • 方法区中类静态属性引用的对象——比如 static 字段引用的实例。
  • 方法区中常量引用的对象——比如字符串常量池里的引用。
  • 本地方法栈（Native 方法）中 JNI 引用的对象。
  • Java 虚拟机内部的引用，如基本数据类型对应的 Class 对象、常驻的异常对象、系统类加载器。
  • 所有被同步锁（synchronized）持有的对象。

两种方案对比

维度引用计数法可达性分析
实现复杂度 简单，判定效率高（计数为 0 立刻可回收） 较复杂，需要一次全局搜索
循环引用 无法解决（结构性缺陷） 天然免疫，不依赖局部计数
额外开销 每次引用赋值都要维护计数器，高频操作有性能损耗 分析时需要 STW（Stop The World）保证一致性快照，但赋值本身没有额外开销
代表实现 Python（配合分代回收 + 弱引用打破循环引用辅助解决）、早期 COM HotSpot 等主流 JVM 实现
面试高频："Java 为什么不用引用计数法？"——标准答案要点两个：① 无法解决循环引用；② 维护计数器本身有性能开销（每次引用赋值都要加减操作，多线程下还要保证原子性）。 补充一句加分项：Python 其实用的就是引用计数法为主，但额外加了一个分代回收器专门去扫描、打破循环引用，说明这个问题不是无解，只是可达性分析对 Java 场景是更省心的方案。

核心价值总结

  • 垃圾收集的前提是"判断对象存活"，这一步选错方案，后面所有算法都无从谈起。
  • 引用计数法简单直观，但循环引用是结构性硬伤，无法在局部信息下自我解决。
  • 可达性分析站在全局视角，从 GC Roots 出发做可达性搜索，天然规避了循环引用问题，是 HotSpot 等主流 JVM 的选择。
  • GC Roots 的范围（栈、静态属性、常量、JNI 引用等）要记住，是后面理解各种收集器工作过程的基础。

与其他知识点的联系

  • 标记-清除/复制/整理三种算法（04-06 篇）：可达性分析只回答"谁是垃圾"，不回答"怎么清理"——它是这三种算法共同的前置步骤：三者的标记阶段都是同一套可达性分析逻辑，区别只在标记完成之后如何处理垃圾和存活对象。
  • finalize() 的两次标记（03 篇）：finalize() 机制之所以需要"两次标记"，正是建立在可达性分析基础上的——第一次标记发现不可达后先检查是否重写了 finalize()，给对象一次在 finalize() 里重新挂上 GC Roots 引用的自救机会，第二次标记才是终审。
  • 四种引用类型（02 篇）：可达性分析判断"引用链是否连通"，但强/软/弱/虚四种引用类型定义了"连通"的强度差异——软引用、弱引用是可达性分析里"半可达"的特殊情况，专门用来在内存紧张时提前判定某些对象可回收。
  • 并发模块的锁与 GC Roots：GC Roots 的范围里包含"所有被同步锁持有的对象"，这意味着可达性分析和线程同步机制是有交集的——一个对象只要还在被某把锁持有，就一定被判定为存活，这是理解并发安全和内存回收如何互相影响的关键连接点。
  • 分代收集理论（07 篇）：分代收集把堆拆成新生代、老年代分别处理，但每次 Minor GC/Full GC 判断对象是否存活，用的仍然是这里讲的可达性分析——分代只是决定"用什么算法清理"，不改变"怎么判断谁是垃圾"这一步的逻辑。
