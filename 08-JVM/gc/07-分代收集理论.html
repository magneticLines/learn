分代收集理论(Generational Collection)

前面三篇讲了三种算法，各有取舍，谁都不是万能药。分代收集理论要解决的问题是：能不能针对不同"性格"的对象，各自用最适合它的算法？答案是可以，而且这正是现代 JVM 几乎所有收集器的设计基石。

一句话讲清楚

分代收集理论建立在两个经过大量实证观察得出的经验假说之上：

  • 弱分代假说（Weak Generational Hypothesis）：绝大多数对象都是朝生夕灭的——刚创建不久就变得不可达。
  • 强分代假说（Strong Generational Hypothesis）：熬过越多次垃圾收集的对象，就越难以消亡。一个对象活得越久，它继续存活下去的概率就越高。

这两个假说不是理论推导出来的，而是对真实 Java 程序做大量统计后得出的规律。既然大多数对象"朝生夕灭"、少数对象"历久弥坚"， 那么把内存按对象"年龄"划分成不同的区域，让新对象和老对象分开处理，各自用最匹配它们存活率特征的算法，收益就会远大于"一刀切"用同一种算法处理所有对象。

把 memory 组学过的堆分区 和 04/05/06 篇的三种算法串起来

Java 堆
├── 新生代（Young Generation）　占比小，对象朝生夕灭（弱分代假说）
│　　├── Eden　　　　　新对象优先在这里分配
│　　├── Survivor From
│　　└── Survivor To
│　　→ 存活率低，复制少量存活对象代价很小
│　　→ 采用 标记-复制算法（05 篇），高效且不产生碎片
│
└── 老年代（Old Generation）　占比大，装的是扛过多次 GC 的老对象（强分代假说）
　　　→ 存活率高，如果用标记-复制，绝大多数对象都要复制，代价过高
　　　→ 采用 标记-清除 或 标记-整理算法（04 / 06 篇），不做无谓的复制

新生代对象活不过一次 Minor GC 是常态，所以标记-复制算法"复制少数存活对象、清空整块内存"的特点被最大化利用； 老年代对象已经证明了自己的"生命力"，标记-复制的复制代价太不划算，所以选用标记-清除（不移动，牺牲空间换效率，如 CMS）或标记-整理（移动，换取无碎片，如 Parallel Old）。

面试高频："为什么要分代？" 标准答案不是背"新生代用复制、老年代用整理"这个结论，而是要讲清楚因果链： 弱/强分代假说 → 不同代对象存活率差异巨大 → 存活率低的适合标记-复制（代价和存活对象数成正比）、存活率高的适合标记-清除/整理（避免大量复制）→ 因此产生了分代这一综合方案。记住"为什么"，比记住"是什么"更容易在面试中讲清楚。

分代之后新出现的问题：跨代引用

分代收集把堆拆成了多个区域，但对象之间的引用关系并不会因为分区而消失——老年代对象完全可能引用新生代对象（比如一个长期存活的缓存对象引用了刚创建的临时数据）。 这带来一个新问题：只对新生代做 Minor GC 时，要不要把整个老年代也扫一遍，才能确认新生代对象是否可达？ 如果每次 Minor GC 都要扫描整个老年代找 GC Roots，分代带来的效率优势就会被这个开销抵消。

这个问题由记忆集（Remembered Set）和卡表（Card Table）技术解决——本质是在新生代之外维护一份"哪些老年代区域可能存在跨代引用"的记录， Minor GC 时只需要扫描这份记录涉及的少量区域，而不用扫描整个老年代。这里先点到为止，后面讲 G1 收集器时会具体展开卡表的实现细节。

核心价值总结

  • 分代收集理论建立在弱分代假说（大多数对象朝生夕灭）和强分代假说（活得越久越难死）这两个经验规律之上。
  • 正是这两个假说，驱动了"新生代用标记-复制、老年代用标记-清除/整理"这个组合方案——本质是让算法的取舍去匹配对象的真实存活率特征。
  • 分代不是免费的：拆分内存区域后引入了跨代引用问题，靠记忆集+卡表技术把扫描范围从"整个老年代"收窄到"少量可能存在跨代引用的区域"。
  • 理解了这一篇，后面 Serial/ParNew（新生代）、CMS/Parallel Old（老年代）、G1（打破物理分代边界）等具体收集器的设计动机就都有了统一的解释框架。
