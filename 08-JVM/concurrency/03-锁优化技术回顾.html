锁优化技术回顾(承接 02-并发)

不重讲锁分类和锁膨胀的完整过程，只补一个"锁状态存在哪儿"的连接点——把锁优化和对象内存布局串起来。作为 08-JVM 模块的收尾篇。

1. 先接上：02-并发模块已经讲过什么

02-并发/advanced/01-锁的分类总览 已经把锁按四个维度（悲观/乐观、公平/非公平、可重入/不可重入、独占/共享）梳理清楚， 并专门提到了 synchronized 的锁升级路径。这里不重复完整原理，只用一张表钉住核心结论：

锁状态触发条件核心手段
偏向锁 长期只有一个线程访问该锁对象 不做真正的同步，仅在对象头记录持有线程 ID
轻量级锁 有竞争，但线程交替执行、不长时间持有 CAS 自旋，避免陷入内核态互斥量
重量级锁 竞争激烈、自旋失败次数过多 操作系统互斥量（mutex），线程阻塞挂起

锁膨胀方向是单向的：无锁 → 偏向锁 → 轻量级锁 → 重量级锁，一旦升级不会再降级（偏向锁可以被"撤销"重置，但整体升级趋势不可逆）。

版本变化：偏向锁自 JDK 15（JEP 374）起默认关闭，不再是锁升级路径上默认经过的一环——如果不显式加 -XX:+UseBiasedLocking，会直接从无锁进入轻量级锁；JDK 18 更是移除了偏向锁的实现代码，只保留兼容参数。上面这张表描述的是 JDK 15 之前的默认行为。

2. JVM 视角的新连接点：锁状态存在对象头的 Mark Word 里

02-并发模块讲锁升级时，关注点在"竞争激烈程度"这条业务/调度线索上；而 08-JVM 模块的 memory 组第 8 篇「对象头与 Mark Word」讲的是另一件事—— Java 对象在堆内存里的布局，其中对象头（Object Header）包含一部分叫 Mark Word 的区域。

把这两条线索接起来就会发现：偏向锁、轻量级锁、重量级锁这几个"锁状态"，本质上就是 Mark Word 里几个比特位在不同阶段被改写成的不同编码。 锁膨胀不是发生在某个抽象的"锁对象"里，而是实实在在地发生在被加锁对象的对象头这几十个比特上—— 偏向锁模式下 Mark Word 存的是线程 ID，轻量级锁模式下存的是指向线程栈中锁记录（Lock Record）的指针，重量级锁模式下存的是指向操作系统互斥量（Monitor）的指针。

融会贯通的意义：如果只学过 02-并发模块，"锁升级"停留在一个抽象的状态机故事；只学过 08-JVM 的 memory 组，Mark Word 只是内存布局里的一段比特。 把两者接起来，才会真正理解——为什么"给一个对象加锁"这件事，是在直接修改这个对象自己的内存数据，而不是外部某个独立的锁资源。这正是学完整个 JVM 模块之后应该具备的能力。

3. 08-JVM 模块回顾：一段代码到虚拟机里跑起来的完整链路

到这里，08-JVM 模块的全部内容已经走完一遍。回头看这条学习路径，其实就是在完整回答"一段 Java 代码从写下到跑在虚拟机里"这一件事： 从 JVM 全景建立整体地图 → 内存区域与溢出搞清楚代码运行时数据放在哪里、会在哪里爆掉 → 垃圾收集解决堆上的对象什么时候该被回收 → 性能监控与调优学会用工具观察和优化前面这套内存/GC 系统的真实表现 → 类加载机制搞清楚 .class 文件是怎么被读进虚拟机、变成可用的类的 → 字节码执行引擎搞清楚字节码指令具体是怎么被解释/编译执行的 → 编译与优化补上 JIT 如何把热点代码编译成机器码、又做了哪些优化 → 最后这一组并发基础回顾，把 02-并发模块的知识接到 JVM 视角上，串成一张完整的网。

小结

  • 锁分类与锁升级的完整原理见 02-并发/advanced 目录，本篇不重复。
  • 偏向锁/轻量级锁/重量级锁的状态编码，实际存储在对象头的 Mark Word 里——锁优化技术和对象内存布局是同一件事的两个视角。
  • 08-JVM 模块的学习路径是一条完整链路：JVM 全景 → 内存区域与溢出 → 垃圾收集 → 性能监控与调优 → 类加载机制 → 字节码执行引擎 → 编译与优化 → 并发基础回顾。

与其他知识点的联系

  • 对象头与 Mark Word（memory 组）：锁状态的三次膨胀（无锁 → 偏向锁 → 轻量级锁 → 重量级锁）不是发生在抽象的"锁"上，而是逐字节改写被加锁对象自己的 Mark Word——这也解释了为什么"给对象加锁"天然是一个和该对象内存布局绑死的操作，而不是外部资源分配。
  • 垃圾收集的 GC Roots（gc 组第 1 篇）：可达性分析明确把"所有被同步锁持有的对象"列为 GC Roots 之一——一个对象只要还处于 synchronized 锁定状态，就一定存活，这是并发和 GC 两个模块唯一直接共享的判定规则。
  • STW 与收集器停顿（gc 组标记-整理、Serial/ParNew 等篇）：锁竞争失败时线程会被挂起阻塞，垂直方向看是"线程调度层的停顿"；GC 收集器做标记-整理时全程 STW，是"内存管理层的停顿"。两者都是"暂停某些线程换取另一件事顺利完成"的同一种设计手法，只是暂停的对象和目的不同。
  • CAS 与轻量级锁自旋：轻量级锁用 CAS 自旋而非直接进入内核态互斥量，这与 GC 里"标记-整理需要移动对象、必须保证不出现读旧写新的数据竞争"是同一类问题的两种解法——前者靠硬件原子指令避免竞争开销，后者靠 STW 彻底消除并发窗口。
  • 分代收集理论的"权衡"思维（gc 组第 7 篇）：锁优化按竞争程度从轻到重逐级升级（偏向锁→轻量级锁→重量级锁），本质和分代收集"用不同算法匹配不同存活率对象"是同一种工程哲学——不存在一劳永逸的万能方案，只有根据实际竞争/存活特征动态匹配最合适手段的自适应策略。
