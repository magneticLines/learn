线程实现与协程简介

接上"进程 vs 线程"的结论，从虚拟机实现的角度补一句"Java 线程到底是怎么落地的"，再前瞻性认识一下协程与虚拟线程。

1. 先接上：02-并发模块已经讲过什么

02-并发/basics/05-进程vs线程 已经从操作系统资源模型的角度讲清楚了进程与线程的本质区别， 也提到了 HotSpot 采用 1:1 模型（Java 线程直接映射到 OS 内核线程）。本篇不重复这部分内容，只做两件事： 补一点"虚拟机层面为什么是 1:1"的实现视角，然后引出协程这个更前沿的话题。

2. 线程实现模型：1:1 / 1:N / M:N

《深入理解 Java 虚拟机》第 12 章从虚拟机实现的角度，把线程实现方式归纳为三种模型，Java 线程在主流虚拟机（HotSpot）上采用的正是最简单直接的一对一模型（1:1）： 每一个 Java 线程都直接映射为一条操作系统原生线程，线程的创建、销毁、调度全部委托给操作系统内核完成。

模型说明代表
1:1（内核线程模型） 一个用户线程对应一个内核线程，由 OS 直接调度 HotSpot 的 java.lang.Thread（平台线程）
1:N（用户线程模型） 多个用户线程映射到一个内核线程，调度、切换全部在用户态完成，内核完全不知道线程的存在 早期部分语言运行时（历史方案，现代主流虚拟机已很少使用）
M:N（混合模型） 用户线程与内核线程数量不对等，通过少量内核线程复用调度多个用户线程 Go 的 goroutine、Java 19+ 虚拟线程
为什么 HotSpot 选 1:1 而不是 1:N？ 1:N 模型下，一旦某个用户线程发起阻塞式系统调用，整个进程内所有映射在同一内核线程上的用户线程都会被一起阻塞，无法真正利用多核。 1:1 模型把这个问题交给操作系统内核解决，实现简单、可靠，代价是线程创建和上下文切换的开销都由内核承担（这也是后面协程要解决的问题）。

3. 协程：比线程更轻量的用户态调度单元

协程（Coroutine）本质上是 1:N / M:N 模型的现代复兴：它是运行在用户态、由程序自身（而非操作系统内核）调度的执行单元。 因为协程的创建和切换不需要陷入内核态、不需要保存/恢复完整的寄存器组和内核栈，所以成本比操作系统线程低一到两个数量级—— 创建几十万个协程是常见操作，而创建几十万个 OS 线程几乎必然把系统压垮。

4. Java 19+ 虚拟线程：M:N 模型的现代实现

JDK 21 正式发布（19/20 预览）的虚拟线程（Virtual Thread，JEP 444），正是 M:N 模型在 Java 生态里的落地： 大量虚拟线程复用少量操作系统"载体线程"（Carrier Thread），当虚拟线程执行阻塞式 I/O 时，JVM 会自动把它从载体线程上"卸下"， 让载体线程去执行其他虚拟线程，等 I/O 就绪后再把它"挂回"某个载体线程继续执行。

虚拟线程解决的问题，和响应式编程是同一个问题： 07-Spring 模块的 reactive 组讲过，响应式编程（如 WebFlux）用少量线程通过事件循环 + 回调链的方式处理海量并发请求，换来了高吞吐， 但代价是编程模型变得复杂（链式操作符、上下文丢失、调试困难）。虚拟线程提供了另一条更平滑的路径： 程序员依然按同步阻塞的直观方式写代码（该 while 就 while，该 xxx.get() 就 xxx.get()）， 却能在 I/O 密集场景下获得接近异步非阻塞的吞吐量——复杂度被下沉到了 JVM 内部，而不是转嫁给业务代码。

这是一个前瞻性了解的内容，不需要在这里深入实践；只需要建立一个认知坐标：线程池调优、响应式编程、虚拟线程， 本质上都是在同一个问题（如何用有限的系统资源处理海量并发 I/O）上给出的不同答案，各自代价不同。

小结

  • Java 线程在主流虚拟机上是 1:1 模型，直接映射到 OS 原生线程；历史上还存在 1:N 和 M:N 模型。
  • 协程是比 OS 线程更轻量的用户态调度单元，创建和切换成本远低于线程。
  • Java 19+ 虚拟线程是 M:N 模型的现代实现，用同步阻塞的编程方式获得接近异步非阻塞的吞吐量，是响应式编程之外的另一条路径。

与其他知识点的联系

  • Java 内存模型回顾（上一节）：线程实现模型回答的是"执行单元怎么被调度"，JMM 回答的是"共享变量何时可见"——两者是正交的两个维度：无论 Java 线程是 1:1 映射到内核线程，还是虚拟线程这种 M:N 调度，一旦涉及共享变量的读写，依然要受 happens-before 规则约束，调度模型的变化不会改变可见性规则。
  • 锁消除/逃逸分析（compile 组）：逃逸分析判断"对象是否只被单线程访问"这件事，其判断依据本质上是线程模型层面的——不管底层是 OS 线程还是虚拟线程，只要编译器能确定该对象的访问范围不跨越执行单元的边界，锁消除依然成立，这说明编译期优化和线程调度模型是相互独立、又彼此配合的两层机制。
  • 解释执行 vs 编译执行（compile 组）：1:1 线程模型下每个 Java 线程都对应一个内核线程独立执行字节码/机器码，JIT 编译产出的机器码在多个线程间是共享和复用的（同一个类的同一个方法只编译一次），这也是为什么 JIT 编译天然要考虑多线程环境下代码替换的原子性和可见性问题。
  • 响应式编程（07-Spring 模块）：虚拟线程和响应式编程（如 WebFlux）解决的是同一个"如何用有限系统资源应对海量并发 I/O"的问题，只是取舍不同——响应式编程把复杂度暴露给业务代码（回调链、操作符），虚拟线程把复杂度下沉到 JVM 内部，让程序员继续用同步阻塞的直观写法。
  • 进程 vs 线程（02-并发模块）：本节讲的 1:1/1:N/M:N 模型是对"进程 vs 线程"结论的进一步细化——上一模块讲清楚了线程是操作系统调度的基本单位，本节回答的是"Java 的线程具体是怎么落地到操作系统线程上的"这一实现层面的问题。
