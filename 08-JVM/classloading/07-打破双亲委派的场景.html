打破双亲委派的场景(SPI / 热部署)(进阶)

双亲委派不是绝对不能碰的铁律——当核心类库需要反向调用第三方实现、或者容器需要做类隔离时，就必须有意打破它。

一句话讲清楚

双亲委派模型是一种约定俗成的最佳实践，而不是虚拟机规范强制要求的铁律。 当遇到"父加载器需要使用子加载器才能加载到的类"、或者"需要严格隔离不同模块的类"这类场景时，Java 通过一些巧妙的设计（线程上下文类加载器）或直接自定义类加载器的方式，主动打破了委派的方向或层级关系。

场景一：SPI 机制——最经典的"反向委派"

这一段直接呼应之前学过的 SPI 机制（04-工程实战/notes/5解耦机制之Java的SPI机制.md）。回忆一下 JDBC 的场景：

  • java.sql.Driver 接口定义在 JDK 核心类库里，由启动类加载器加载。
  • 具体的驱动实现类（比如 MySQL 的 Driver 实现）在第三方 jar 包里，只能由应用类加载器加载。

问题来了：java.util.ServiceLoader（乃至更早的 DriverManager）这些负责"发现并加载实现类"的代码，本身是 JDK 核心类库的一部分，由启动类加载器加载。 按照双亲委派的方向，父加载器（启动类加载器）根本不能主动去问子加载器（应用类加载器）"你有没有加载到什么类"——委派方向天生就是单向向上的，父加载器没有"向下"这个动作。 但 ServiceLoader 恰恰需要加载到 classpath 里第三方 jar 包提供的实现类，这些类只有应用类加载器的搜索范围能找到。

矛盾点：如果严格遵守双亲委派，启动类加载器"看不到"任何 classpath 上的第三方实现类，SPI 机制根本无法工作。

解决方案：线程上下文类加载器（Thread Context ClassLoader）

Java 给每个线程都绑定了一个"线程上下文类加载器"，默认情况下就是应用类加载器。核心类库里的代码（比如 ServiceLoader.load()）在需要加载具体实现类时，不使用加载自己的那个启动类加载器，而是主动获取"当前线程的上下文类加载器"来加载：

ServiceLoader 加载实现类时的典型做法（简化示意）
// ServiceLoader 这段代码本身运行在由 Bootstrap 加载的核心类库里
// 但它没有用自己所在类的类加载器，而是拿"当前线程"绑定的类加载器
ClassLoader cl = Thread.currentThread().getContextClassLoader();

// 默认情况下这就是应用类加载器，能在 classpath 里找到第三方驱动实现
ServiceLoader<Driver> loader = ServiceLoader.load(Driver.class, cl);

这本质上是让"由 Bootstrap 加载的代码"反向借用了应用类加载器的加载能力，实现了"父加载器使用子加载器加载类"这个双亲委派原本不允许的方向——所以说 SPI 是打破双亲委派方向的典型场景。

建立联系：结合之前学过的反射知识来看，ServiceLoader 内部读取 META-INF/services/ 下的配置文件拿到实现类的全限定名字符串后，正是通过反射（Class.forName(name, true, classLoader) + 反射构造实例）把它变成一个真正可用的对象——这也解释了为什么 SPI、动态代理、依赖注入框架背后往往都离不开"类加载器 + 反射"这套组合拳。

场景二：模块化热部署——类隔离

另一类打破双亲委派的场景是需要类隔离而非共享：

  • Tomcat：每个部署的 Web 应用都有自己独立的 WebAppClassLoader，不同应用即便用了同名但不同版本的类库（比如两个应用分别依赖 Jackson 的不同版本），也能互不干扰——这就要求每个 WebAppClassLoader 优先在自己的 WEB-INF/lib 里找类，而不是老老实实先向上委派给共享的应用类加载器。
  • OSGi：模块化框架要支持"模块热插拔"（运行时动态安装/卸载/升级某个模块，不重启整个应用），每个模块（Bundle）都有自己的类加载器，模块之间的类加载关系是网状的（可以互相依赖），而不是双亲委派这种严格的树状父子关系。
了解即可：Tomcat、OSGi 这类容器级别的类加载器设计属于进阶内容，不需要记住实现细节，只需要理解"当业务场景需要类隔离/热插拔时，严格的双亲委派（一切共享、只能向上）反而成了限制，因此需要自定义加载器打破默认的委派关系"这个设计动机。

对比：遵守 vs 打破双亲委派

场景是否遵守双亲委派原因
加载 java.lang.* 等核心类库 严格遵守 需要安全性和一致性，核心类型全局唯一
SPI（JDBC 驱动、日志门面等） 打破（反向借用子加载器） 父加载器里的代码需要使用只有子加载器能找到的第三方实现类
Tomcat 多应用类隔离 打破（优先自己找，而非先向上委派） 不同应用可能依赖同一个库的不同版本，必须互不干扰
OSGi 模块化热部署 打破（网状而非树状委派关系） 模块需要支持独立安装/卸载/升级，不能被单一委派链绑死

核心价值总结

  • 双亲委派是约定，不是虚拟机强制的规则：只要自定义 ClassLoader 时重写 loadClass()（而不只是 findClass()），就可以改变委派逻辑。
  • SPI 是"反向委派"的典型场景：通过线程上下文类加载器，让核心类库的代码能反过来借用应用类加载器的能力，这是理解 JDBC、SLF4J、Spring Boot 自动配置等机制底层原理的关键一环。
  • 热部署/类隔离是"改变层级关系"的典型场景：Tomcat、OSGi 通过自定义类加载器实现兄弟或网状关系，而非严格的父子委派，代价是需要自己处理好"共享 vs 隔离"的边界。
