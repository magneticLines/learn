为什么要有双亲委派(安全与避免重复加载)

上一篇讲了双亲委派"怎么委派"，这一篇讲清楚它到底解决了什么真实的工程问题——不是为了好看，而是为了安全和一致性。

一句话讲清楚

双亲委派解决两个核心问题：一是安全性——防止核心类库被用户自定义的同名类冒充或篡改；二是一致性——避免同一个类被不同加载器重复加载多份，导致类型判断出问题。 这两个问题都不是"理论上的担忧"，而是没有双亲委派时会实打实发生的工程事故。

问题一：安全性——防止核心类库被冒充

设想一下，如果没有双亲委派，任何加载器都可以"抢先"加载自己写的类：

恶意示例：自己写一个 java.lang.String
// 把这个文件放在 java.lang 包下，编译进自己的 classpath
package java.lang;

public class String {
    // 伪装成 java.lang.String，但内部逻辑被篡改
    // 比如在这里偷偷把用户密码上传到远程服务器
}

如果没有委派机制，应用类加载器在 classpath 里找到了这个"假的" java.lang.String，就会直接加载它，用户程序里所有 new String(...) 都会变成这个被篡改的实现——这是极其危险的攻击面。

有了双亲委派，应用类加载器收到"加载 java.lang.String"的请求后，永远先向上委派，一路问到启动类加载器；启动类加载器在 <JAVA_HOME>/lib 里已经能找到官方的 java.lang.String，直接返回这个官方版本。应用类加载器自己写的那个假类根本没有机会被加载——因为请求在启动类加载器那一层就已经被满足了，不会再下沉到应用类加载器。

这被称为"沙箱安全机制"：核心类库（java.lang、java.util 等）永远只能由最顶层的启动类加载器加载，任何用户代码试图伪造同名类都会被这套委派机制自动挡在外面，用户完全不需要手写任何校验逻辑。

问题二：一致性——避免同一个类被重复加载

JVM 判断"两个类是否是同一个类型"，不只看类的全限定名，还要看加载它的类加载器是不是同一个。如果同一个类被两个不同的类加载器分别加载了一份，JVM 会认为这是两个完全不同的类型，即便类名、包名、字节码内容完全一样。

反例：同一个类被两个加载器加载，类型比较炸了
// 假设 MyClassLoader 是一个自定义加载器，且没有遵守双亲委派
// 它自己重新实现了 findClass()，去 classpath 上重新加载了 ArrayList

ClassLoader loader1 = new MyClassLoader();
ClassLoader loader2 = new MyClassLoader();

Class<?> clazz1 = loader1.loadClass("com.example.Foo");
Class<?> clazz2 = loader2.loadClass("com.example.Foo");

System.out.println(clazz1 == clazz2);              // false！明明是同一份字节码

Object obj = clazz2.getDeclaredConstructor().newInstance();
boolean isInstance = clazz1.isInstance(obj);  // false！instanceof 判断也会失败
// 甚至把这个对象强转成 com.example.Foo 类型都会抛 ClassCastException
// "两个看起来一模一样的类，在 JVM 眼里是完全不同的两个类型"
面试高频坑："两个 equals() 看起来一样的类为什么 instanceof 判断失败？"——十有八九是类加载器不一致导致的。这类问题在 Web 容器（Tomcat 给每个应用用独立的类加载器）、模块化框架、热部署场景里经常出现，也是排查"明明依赖版本对得上却报 ClassCastException / NoSuchMethodError"这类诡异问题的重要方向。

双亲委派通过"核心类库永远只由固定的顶层加载器加载一次"，从根源上保证了同一个类在整个虚拟机里只会被加载一份，任何加载器请求加载它时，最终都会拿到同一份由启动类加载器缓存好的 Class 对象，天然避免了"同名不同类"的问题。

核心价值总结

  • 安全性：核心类库固定由顶层加载器加载，用户自定义的同名类没有机会替换/污染基础类库，构成"沙箱安全机制"。
  • 一致性：类型相等性判断依赖"类全限定名 + 加载它的类加载器"这个二元组，委派机制保证核心类型只有唯一一份，避免"同名不同类"的诡异 bug。
