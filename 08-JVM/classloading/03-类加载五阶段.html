类加载机制

加载-验证-准备-解析-初始化五阶段

一个 .class 文件从磁盘上的二进制流，变成内存里可以 new 实例、可以调用方法的"活的类型"，中间到底发生了什么。

一句话讲清楚

类加载过程分为加载、验证、准备、解析、初始化五个阶段（还有使用、卸载，但那是加载"之后"的事）。 这五个阶段并非彼此独立、严格按顺序"做完一个再做下一个"——它们通常是交叉进行的（比如解析阶段在某些情况下可以在初始化之后才发生，这叫"动态绑定"/晚期绑定），但开始的顺序是固定的：加载必须最先开始，初始化必须最后开始。

原理走一遍：五个阶段各自做了什么

① 加载（Loading）
  通过类的全限定名获取这个类的二进制字节流（可以来自 jar 包、网络、动态生成）
  将字节流代表的静态存储结构转化为方法区的运行时数据结构
  在内存中生成一个代表这个类的 java.lang.Class 对象，作为方法区数据的访问入口

② 验证（Verification）
  文件格式校验：魔数、版本号是否合法
  元数据校验：这个类是否有父类、是否继承了 final 类等语义合法性
  字节码校验：方法体的字节码指令语义是否合法（比如跳转指令是否指向方法体外）
  符号引用校验：常量池里描述的类/方法/字段是否真实存在、访问权限是否够
  → 目的只有一个：确保这份字节流不会危害虚拟机自身的安全

③ 准备（Preparation）
  为类变量（即 static 修饰的字段）在方法区分配内存
  将这些类变量设置为该数据类型的零值（int 是 0，boolean 是 false，引用类型是 null）
  ⚠️ 注意：这里赋的是零值，不是代码里写的初始值——真实值要等到初始化阶段才会被赋上

④ 解析（Resolution）
  把常量池内的符号引用替换为直接引用
  符号引用：用一组符号描述目标（比如全限定名字符串），还没有和具体内存地址关联
  直接引用：可以直接定位到目标在内存中的位置（指针/偏移量/句柄）

⑤ 初始化（Initialization）
  真正执行类构造器 <clinit>() 方法
  这个方法由编译器自动收集类中所有类变量的赋值动作和静态语句块（static {}）合并而成
  → 类变量在这里才被赋上程序员指定的真实值，静态代码块也在这里执行

反直觉的核心：准备阶段的"零值" vs 初始化阶段的"真实值"

这是新手最容易搞混的一点，看一个具体例子：

Config.java
public class Config {
    public static int maxRetry = 3;  // 类变量，程序员指定的初始值是 3
}

JVM 处理 maxRetry 分两步：准备阶段先在方法区给它分配内存并置为零值 0（此时如果有别的代码提前读到它，看到的是 0，而不是 3）； 初始化阶段执行 <clinit> 方法时，才真正把 3 这个字面量赋给它。 这个"先归零、后赋值"的两步走设计，是为了让类加载的每个阶段职责单一：准备阶段只管"开辟空间、给个确定的默认状态"，不牵涉任何用户代码的执行；真正有业务含义的赋值和副作用（静态代码块里可能有 I/O、可能抛异常）全部收敛到初始化阶段，边界清晰、也便于虚拟机控制"什么时候才允许触发初始化"（下一篇讲触发时机）。

常见误区：很多人以为"static 变量在类加载时就被赋上了代码里写的值"，其实这中间隐藏了准备和初始化两个独立阶段。static final 修饰的编译期常量是个例外——它在准备阶段就会被直接赋上真实值（比如 public static final int MAX = 100;），因为编译器在编译期就把它的值内联进了常量池，不需要等到 <clinit> 执行。这个例外也是下一篇"类初始化触发时机"里"被动引用"现象的根源之一。

五阶段速查表

阶段核心动作一句话记忆点
加载字节流 → 方法区数据结构 + Class 对象"把文件读进内存，生成入口"
验证格式/元数据/字节码/符号引用四类校验"防止非法字节码危害虚拟机"
准备static 变量分配内存 + 置零值"零值，不是代码里写的值"
解析符号引用 → 直接引用"从名字定位到内存地址"
初始化执行 <clinit>，赋真实值、跑静态块"真正的赋值和副作用发生在这里"

核心价值总结

  • 五阶段职责单一：每个阶段只做一件事，虚拟机才能在各阶段之间精确控制"什么时候可以往下走"。
  • 零值 vs 真实值是关键分界线：准备阶段给的是"安全的默认状态"，初始化阶段才执行真正有业务含义、可能有副作用的代码。
  • <clinit> 是理解静态初始化顺序、类初始化线程安全（单例的静态内部类写法）的钥匙：虚拟机会保证同一个类的 <clinit> 在多线程环境下只被正确加锁执行一次。
