类初始化的触发时机

虚拟机规范严格列出了"必须初始化"的几种场景——除此之外的引用方式，即便看起来在"用"这个类，也不会触发它的初始化。

一句话讲清楚

Java 虚拟机规范严格规定了有且只有 6 种"主动引用"场景会触发类的初始化，除此之外的引用方式都是"被动引用"，不会触发初始化。 这个规定很重要，因为初始化阶段要执行 <clinit>（可能有静态代码块、可能有副作用），虚拟机必须精确控制"什么时候才允许触发"，不能一"碰到"这个类就无脑初始化。

6 种主动引用（会触发初始化）

① 遇到 new / getstatic / putstatic / invokestatic 这 4 条字节码指令时
  对应 Java 代码：new 一个实例 / 读写类的静态字段（final 常量除外）/ 调用类的静态方法

② 使用 java.lang.reflect 包对类进行反射调用时
  如果类还没初始化，先触发它的初始化

③ 初始化一个类时，如果发现它的父类还没初始化，先触发父类初始化

④ 虚拟机启动时，用户指定的主类（包含 main 方法的类）会先被初始化

⑤ 使用 JDK 7 动态语言支持时，MethodHandle 解析结果为 REF_getStatic / REF_putStatic / REF_invokeStatic 的方法句柄，其对应的类没有初始化则触发初始化

⑥ 接口中定义了默认方法（JDK 8+），当这个接口的实现类初始化时，接口要在此之前被初始化
建立联系：反射（②）之所以会触发初始化，是因为 Class.forName(name) 默认参数就是"加载并初始化"；而 Class.forName(name, false, loader) 把 initialize 参数设为 false 就只加载不初始化——这正是理解反射底层行为、以及后面 SPI 机制里 ServiceLoader 为何要处理"类加载了但可能还没初始化"这类细节的基础。

反直觉的重点：被动引用（不会触发初始化）

下面几种代码写法"看起来"在使用某个类，但虚拟机规范明确规定它们不属于主动引用，不会触发该类的初始化：

反例 1：通过子类引用父类的静态字段，不会触发子类初始化

被动引用示例 1
class Parent {
    static {
        System.out.println("Parent 初始化");
    }
    public static int value = 100;
}

class Child extends Parent {
    static {
        System.out.println("Child 初始化");  // 不会执行
    }
}

public class Test {
    public static void main(String[] args) {
        System.out.println(Child.value);
        // 输出只有 "Parent 初始化" 和 100，Child 的静态块不会执行
        // 因为 value 字段实际定义在 Parent 里，对静态字段的直接引用只会初始化定义它的那个类
    }
}

反例 2：通过数组定义来引用类，不会触发该类初始化

被动引用示例 2
Parent[] arr = new Parent[10];
// 这行代码只会触发一个由虚拟机自动生成的、继承自 Object 的数组类 [LParent; 的加载
// 不会触发 Parent 类本身的初始化——因为还没有真正 new 出任何一个 Parent 实例

反例 3：引用 static final 编译期常量，不会触发定义它的类初始化

被动引用示例 3
class Constants {
    static {
        System.out.println("Constants 初始化");  // 不会执行
    }
    public static final String VERSION = "1.0";  // 编译期常量
}

public class Test {
    public static void main(String[] args) {
        System.out.println(Constants.VERSION);
        // 只输出 "1.0"，Constants 的静态块永远不会执行
        // 因为 "1.0" 在编译期就已经存入了 Test 类自己的常量池，Test 和 Constants 类在运行期没有任何关联
    }
}
面试高频坑：反例 3 是上一篇提到的"static final 编译期常量在准备阶段就直接赋真实值"这个规则的延伸——正因为常量在编译期已经被内联进调用方的常量池，调用方运行时压根不需要再去找 Constants 这个类要值，自然也就不会触发它的初始化。但如果把 VERSION 的值换成一个运行时才能确定的表达式（比如调用某个方法返回的结果），它就不再是"编译期常量"，会退化成一次真正的字段访问，从而触发初始化。

核心价值总结

  • 只有 6 种场景触发初始化：其余所有"看起来在用类"的写法都是被动引用，规范这样设计是为了让"什么时候执行用户的静态代码"完全可预测、可控制。
  • 被动引用三连：子类引用父类静态字段、数组定义、引用编译期常量——这三个反例是面试和代码审查中最常考的"类初始化陷阱"。
