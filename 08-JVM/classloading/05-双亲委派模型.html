类加载机制

双亲委派模型

一个类加载请求，为什么总是先往上交给"爹"，而不是自己先动手——理解 JVM 类加载器之间的层级协作关系。

一句话讲清楚

双亲委派模型规定：一个类加载器收到加载请求时，不会自己先尝试加载，而是把请求委托给父加载器去做，逐级向上，直到最顶层；只有当父加载器反馈"自己无法完成这个加载"（在它的搜索范围内没有找到这个类）时，子加载器才会尝试自己去加载。 注意名字里的"双亲"其实是历史翻译习惯，本质是单一父加载器的层级关系（parent，不是 biological 意义的"双亲"），并不是有两个平级的父加载器。

类加载器的层级关系

启动类加载器（Bootstrap ClassLoader）
  C++ 实现，是虚拟机自身的一部分，没有对应的 Java 对象
  负责加载 <JAVA_HOME>/lib 下的核心类库（java.lang.* 等）
  ↑
扩展 / 平台类加载器（Extension / Platform ClassLoader）
  JDK 9 之前叫 Extension ClassLoader，负责 <JAVA_HOME>/lib/ext
  JDK 9 引入模块化后改名为 Platform ClassLoader
  ↑
应用类加载器（Application ClassLoader）
  负责加载 classpath 上用户自己写的类，是 getSystemClassLoader() 返回的加载器
  绝大多数 Java 程序默认都由它加载
  ↑
用户自定义类加载器（可选）
  继承 ClassLoader 自己实现，比如 Tomcat 的 WebAppClassLoader

原理走一遍：委派的工作过程

假设应用类加载器收到一个"请加载 com.example.Foo"的请求：

① 应用类加载器收到请求
  不自己找，先委托给上一级：平台类加载器

② 平台类加载器收到请求
  不自己找，继续委托给上一级：启动类加载器

③ 启动类加载器收到请求（已到顶层，无法再向上）
  在自己的搜索范围（<JAVA_HOME>/lib）里查找 com.example.Foo
  找不到 → 向下反馈"我加载不了"

④ 平台类加载器收到"启动类加载器加载不了"的反馈
  在自己的搜索范围（<JAVA_HOME>/lib/ext）里查找
  找不到 → 继续向下反馈

⑤ 应用类加载器收到反馈，轮到自己尝试
  在 classpath 里找到了 com.example.Foo.class
  → 由应用类加载器亲自完成加载
关键点：只有当所有祖先加载器都明确表示"我范围内没有"，请求才会真正下沉回最初发起请求的那个加载器自己动手。这是一个"向上问一圈，问不到才轮到自己"的过程，而不是"谁离得近谁先试"。

核心价值总结

  • 委派方向永远向上：子加载器先问父加载器，父加载器优先级最高，只有父加载器明确"找不到"才轮到子加载器自己找。
  • 层级是 Bootstrap → Extension/Platform → Application → 自定义：越往上加载的类越"基础"，越往下加载的类越贴近具体业务代码。
  • 这不是一个"性能优化"设计，而是一个"安全和一致性"设计：具体解决了什么问题，下一篇详细展开。
