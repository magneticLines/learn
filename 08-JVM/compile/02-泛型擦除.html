编译与优化

泛型擦除

List<String> 和 List<Integer> 在字节码里其实是同一个类型——Java 的泛型只是编译器变的一场"戏法"。

一句话讲清楚

Java 泛型是一种编译期的语法糖：<T> 这样的参数化类型信息，只存在于源码和编译阶段。 javac 编译时会把所有泛型类型参数擦除（Type Erasure）成它的限定类型（没写 extends 就擦成 Object）， 字节码层面根本不存在"参数化类型"这回事，List<String> 和 List<Integer> 编译后是完全相同的 List 类型。

类比：泛型就像点餐单上写的"儿童套餐（限12岁以下）"——这行字只是给收银员（编译器）看的提示，帮它在下单时把关；厨房（JVM）拿到的永远是同一张"套餐"制作单，根本不知道、也不关心谁点的。

验证：反射看 Class 对象是否相同

如果泛型信息真的保留到了运行时，List<String> 和 List<Integer> 的 Class 对象应该不同。实际运行一下：

GenericErasureDemo.java
List<String>  strList = new ArrayList<>();
List<Integer> intList = new ArrayList<>();

System.out.println(strList.getClass() == intList.getClass());
// 输出: true —— 运行时它们是同一个 java.util.ArrayList 类型
// 泛型参数 / 编译后早已被擦除，运行时根本不存在

再看编译器如何靠"偷偷插入强转"来保证类型安全：

源码（有糖）vs 擦除后（编译器视角）
// 源码写法
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);   // 看起来不需要强转

// 擦除后字节码层面等价于
List list = new ArrayList();  // 泛型参数没了
list.add("hello");
String s = (String) list.get(0);  // 编译器悄悄插入了强制类型转换

这个"偷偷插入的强转"就是泛型类型安全的真相：编译期做类型检查 + 插入强转，运行期该干嘛干嘛，类型信息早已不在。

为什么泛型有这些"反直觉"的限制？

一旦理解了擦除，下面几条 Java 泛型的"怪规则"就都能讲通——它们的根源都是运行时压根没有 T 这个类型。

三个编译不通过的例子
class Box<T> {

    // ① 不能 new T[]，因为擦除后 T 变成 Object，
    //    new Object[] 无法安全地转型回 T[]（数组是协变且带运行时类型检查的）
    T[] arr = new T[10];  // 编译错误

    // ② 不能对泛型类型做 instanceof，因为运行时根本没有 List 这个类型
    //    只有 List，判断 "是不是 List" 这个问题在运行时是无意义的
    void check(Object o) {
        if (o instanceof List<String>) { }  // 编译错误
    }

    // ③ 不能重载两个擦除后签名相同的泛型方法
    //    List 和 List 擦除后都是 List，方法签名完全一样 → 冲突
    void process(List<String> list) { }
    void process(List<Integer> list) { }  // 编译错误：erasure 后签名冲突
}

擦除规则走一遍

无界类型参数 <T>
  擦除为 Object
  例：class Box<T> → class Box { Object value; }

有界类型参数 <T extends Comparable>
  擦除为限定的上界类型 Comparable
  例：class Box<T extends Number> → class Box { Number value; }

泛型方法调用处
  编译器插入 强制类型转换，保证取值时类型安全
  这也是为什么泛型被称为"编译器的类型检查语法糖"

对比：Java 擦除式泛型 vs C# 具体化泛型

Java（类型擦除）C#（Reified Generics，真泛型）
运行时类型信息 擦除，List<String> 和 List<Integer> 运行时是同一类型 保留，List<string> 和 List<int> 运行时是不同的具体类型
数组 / instanceof 不能 new T[]，不能对参数化类型 instanceof 可以，因为类型信息在运行时依然存在
值类型（如 int）泛型 只能用包装类型 Integer，触发自动装箱（下一节详解） 可为值类型生成专用的特化版本，避免装箱开销
兼容性代价 擦除是为了兼容 JDK 1.5 之前编译的字节码（向后兼容优先） CLR 从设计上就支持泛型具体化，无历史包袱
为什么 Java 选择擦除式泛型？历史原因：泛型是 JDK 5 才引入的，此前已经有大量 ArrayList、HashMap 之类的非泛型代码和字节码。用类型擦除可以让泛型代码编译后与老代码保持二进制兼容（“伪泛型”），代价就是牺牲了运行时的类型信息。

面试高频 / 常见坑

坑：用反射拿到的 List 可以被强行塞入类型不符的元素——((List) list).add(123) 能绕过编译器检查，往 List<String> 里塞进一个 Integer，等到后面 get() 取出时才会在插入的强转处抛 ClassCastException。这说明泛型的类型安全只是编译期承诺，运行时没有真正的边界保护。

核心价值总结

  • 泛型是编译期语法糖：字节码里没有参数化类型，List<String> 和 List<Integer> 运行时完全相同。
  • 擦除规则：无界类型参数擦成 Object，有界类型参数擦成其限定的上界类型；调用处编译器插入强转保证类型安全。
  • 反直觉限制的根源：new T[]、泛型 instanceof、擦除后签名冲突的重载，全部源自"运行时没有 T"这一事实。
  • 对比 C#：Java 为了兼容老字节码选择了擦除式泛型，C# 从设计之初就支持真正的具体化泛型，各有取舍。
