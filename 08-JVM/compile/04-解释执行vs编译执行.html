编译与优化

解释执行 vs 编译执行

Java 常被说成"半编译半解释"的语言——为什么 HotSpot 不干脆全部编译成机器码一次搞定？答案是：启动速度和峰值性能不可兼得，所以它两个都要。

一句话讲清楚

javac 编译出的字节码（.class）并不能被 CPU 直接执行，它还要经过一层转换才能变成真正的机器指令，这一步有两种做法： 解释执行（Interpreter 逐条读取字节码指令，边读边翻译成机器码并立即执行，翻译过程不会被缓存） vs 编译执行（即时编译器 JIT，Just-In-Time Compiler 把整个方法一次性编译成机器码并缓存起来，下次调用直接跑缓存的机器码，不用重新翻译）。

类比：解释执行像"同声传译"——说一句翻一句，现场就能开始，但每次都要重新翻译，遇到重复的话依然要重翻一遍；编译执行像"先把整篇发言稿翻译成文档打印出来"——准备阶段（翻译+印刷）比较慢，但发言稿会被反复宣读时，后面每一次宣读都是直接照着译文念，速度飞快。

两种方式各自的取舍

解释执行（Interpreter）编译执行（JIT Compiler）
启动速度 快，字节码加载后立即可以逐条解释运行，不需要等待 慢，编译整个方法成机器码需要时间，方法越复杂编译越久
执行速度 慢，每次执行到同一条指令都要重新"翻译"一次，没有累积优化 快，编译后的机器码直接被 CPU 执行，且 JIT 会做大量深度优化（下两节详解）
内存 / CPU 占用 较低，不需要额外的编译过程和机器码缓存空间 较高，编译本身消耗 CPU，编译结果需要额外内存缓存（Code Cache）
适合场景 只执行一两次的代码（如启动阶段的初始化代码），编译代价不值得付出 被反复执行的"热点代码"（如循环体、高频调用方法），一次编译长期受益

HotSpot 的选择：解释器 + 编译器共存

如果只用解释执行，长期运行的服务端程序（比如跑几天的后端服务）峰值性能会一直很差；如果只用编译执行， 每次启动一个 Java 程序都要先花大量时间把所有方法编译成机器码才能开始运行，简单脚本、命令行小工具的启动体验会非常糟糕。 HotSpot 采用的是"解释器和编译器搭配工作"的混合模式——这也正是 overview 组讲过的 "HotSpot" 这个名字的由来：它会持续监控程序运行过程中的"热点（Hot Spot）"，发现某段代码被频繁执行，就用编译器把它编译成机器码替换掉解释执行的部分。

① 程序启动
  字节码加载 → 解释器立即逐条解释执行 → 程序快速跑起来，无需等待编译

② 后台监控（Profiling）
  虚拟机在解释执行的同时，用计数器统计每个方法/循环体的执行频率
  → 发现某段代码是热点代码（下一节详细讲阈值判定规则）

③ JIT 编译介入
  后台线程异步把热点方法编译成本地机器码，编译期间解释器继续照常工作（不阻塞）
  编译完成后，后续调用直接切换到编译后的机器码执行

结果：兼顾启动速度与峰值性能
  冷代码：解释执行，付出的编译成本为 0
  热代码：编译执行，用一次编译换来长期的高速运行
这也是为什么 Java 服务端程序常常有"预热（Warm-up）"现象——刚启动时响应慢，跑一段时间后性能才逐渐提升到稳定水平，就是因为 JIT 还没来得及把热点代码编译完成，程序还在用解释执行"顶着"。生产环境做压测/发布前预热，就是刻意提前触发这个编译过程。

对比：静态编译型语言 vs Java 混合模式

C/C++（静态编译，AOT）Java（解释 + JIT 混合）
编译时机 部署前一次性编译成目标平台机器码 先编译成与平台无关的字节码，运行时按需再编译成本地机器码
跨平台性 编译产物和平台强绑定，换平台要重新编译 字节码"一次编译，到处运行"（呼应 overview 组讲过的跨平台特性），JIT 编译在目标机器本地完成
运行时优化空间 编译时对运行时的实际数据分布一无所知，优化基于静态假设 JIT 可以利用运行时的真实 profiling 数据做更激进的优化（如内联、逃逸分析），见后续两节

常见误区

误区："Java 是解释型语言，所以比 C++ 慢"——这是一个过时的刻板印象。现代 HotSpot 虚拟机对热点代码采用 JIT 编译，长期运行的服务端场景下热点代码的执行效率可以接近甚至持平静态编译语言。真正的性能差距，主要体现在启动阶段的预热成本和某些无法被 JIT 覆盖到的冷路径代码上，而不是"Java 天生就慢"。

核心价值总结

  • 解释执行：逐条翻译字节码并立即执行，启动快、无编译成本，但重复执行时没有累积收益。
  • 编译执行（JIT）：把方法整体编译成机器码缓存复用，编译本身有代价，但长期高频调用时收益巨大。
  • HotSpot 的核心设计：解释器负责快速启动 + 兜底执行冷代码，JIT 编译器负责把探测到的热点代码编译成高效机器码，两者协同、互补短板。
