javac 做了什么(语法糖解糖)

javac 不只是把 Java 代码"翻译"成字节码，它还要先把各种为程序员而生的"甜味剂"还原成最朴素的写法——这个过程叫解语法糖。

一句话讲清楚

语法糖（Syntactic Sugar）是编程语言提供的一种"更甜"的写法——它不增加语言的表达能力，只是让某些常见模式写起来更简洁、更符合直觉。 javac 编译期会把所有语法糖"解糖（Desugar）"还原成最基础的语言结构，字节码里根本不存在语法糖这回事。

类比：点单时说"老三样"，服务员心里立刻翻译成"一个煎饼、一碗豆浆、一根油条"分别下单。你说的是"糖"，厨房收到的永远是最原始的菜单项——JVM 就是那个只认"原始菜单"的厨房，它从没见过 for-each、没见过字符串 switch，这些都在 javac 这层就被拆解掉了。

例子一：增强 for 循环 → 迭代器 while 循环

我们写 for-each 遍历一个 List，看起来是全新的语法：

源码（有糖）
List<String> names = getNames();
for (String name : names) {
    System.out.println(name);
}

但反编译（javap）或看 javac 生成的语法树就会发现，它被还原成了迭代器写法：

解糖后（javac 编译期还原）
List<String> names = getNames();
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    String name = it.next();
    System.out.println(name);
}
陷阱：如果遍历的是数组而不是 Iterable，for-each 会被解糖成下标循环而不是迭代器（数组没有 iterator() 方法）。所以 for-each 对数组和集合的解糖结果并不一样，但字节码层面都是最朴素的循环结构。

例子二：字符串 switch → hashCode + equals 判断链

Java 7 引入的字符串 switch，字节码指令集里根本没有"按字符串跳转"这种指令（tableswitch/lookupswitch 只支持 int），所以 javac 必须把它转换成先比 hashCode() 再比 equals() 的判断链。

源码（有糖）
switch (status) {
    case "OK":      handleOk();      break;
    case "FAILED":  handleFailed();  break;
    default:          handleUnknown();
}
解糖后思路（先按 hashCode 分支，再用 equals 精确匹配防哈希碰撞）
int hash = status.hashCode();
int branch = -1;
switch (hash) {                     // 这里才是真正的 int switch（合法字节码指令）
    case 2543:   // "OK".hashCode()
        if (status.equals("OK")) branch = 0; break;
    case -1268779017: // "FAILED".hashCode()
        if (status.equals("FAILED")) branch = 1; break;
}
switch (branch) {                 // 再按分支号执行真正的业务逻辑
    case 0: handleOk();      break;
    case 1: handleFailed();  break;
    default: handleUnknown();
}

为什么要多套一层 equals 判断？因为不同字符串可能哈希碰撞——只比 hashCode 不够安全，必须用 equals 做二次精确校验。这也是面试里常问的"字符串 switch 底层原理"的标准答案。

例子三：try-with-resources → try-finally + close()

Java 7 的 try-with-resources 让资源关闭变得优雅，但字节码层面它只是被展开成了 try-finally：

源码（有糖）
try (InputStream in = new FileInputStream("a.txt")) {
    in.read();
}
解糖后（简化示意，真实字节码还会处理 close() 抛异常时的 suppressed 逻辑）
InputStream in = new FileInputStream("a.txt");
try {
    in.read();
} finally {
    if (in != null) {
        in.close();   // 无论 try 块是否抛异常，都保证被调用
    }
}

其他常见的语法糖还包括：泛型（下一节详细讲）、自动装拆箱（下下节详细讲）、变长参数（解糖成数组）、内部类（解糖成独立的 Outer$Inner.class 并生成合成的桥接字段/方法）、Lombok 的 @Data（这是注解处理器 + 字节码生成，原理不同但效果类似）。

解糖发生在编译流程的哪个阶段？

① 词法分析 + 语法分析
  源码字符流 → Token 流 → 抽象语法树 AST

② 语义分析
  标注检查(变量是否声明)、类型检查、解语法糖(Desugar) 在此阶段完成
  for-each → 迭代器循环；字符串 switch → hashCode 判断链；泛型 → 类型擦除

③ 字节码生成
  遍历"解糖后"的语法树，生成 .class 文件里的字节码指令
  → 生成的字节码里已经完全没有语法糖的痕迹
这就是为什么"语法糖只存在于源码层面"——javap 反编译字节码时你永远看不到 for-each、字符串 switch 这些写法，只能看到最基础的循环、跳转、方法调用指令。字节码是 JVM 唯一认识的语言，它不需要"甜"，只需要"准确"。

常见坑 / 面试高频

坑：在 for-each 循环里对集合做增删（list.remove()），会抛 ConcurrentModificationException。原因就在解糖结果里——迭代器的 modCount 校验机制发现集合结构被外部修改了，而不是通过迭代器自身的 remove() 修改的。理解了 for-each 的解糖过程，这个"经典坑"就不再神秘。

核心价值总结

  • 语法糖只存在于源码和编译期：它不增加 JVM 的能力，只是让程序员写得更爽，编译后一律还原成最基础的形式。
  • 理解解糖有助于排查"诡异"问题：比如 for-each 的 ConcurrentModificationException、字符串 switch 的哈希碰撞风险，本质都源自解糖后的真实代码结构。
  • javac 的编译流程：词法/语法分析 → 语义分析(含解语法糖) → 字节码生成，解糖发生在语义分析阶段，早于字节码生成。
