热点探测与 JIT 编译

上一节说 JIT 只编译"热点代码"——虚拟机怎么知道一段代码算不算热点？答案是给每个方法、每个循环装一个计数器。

一句话讲清楚

热点探测（Hot Spot Detection）就是虚拟机判断"这段代码值不值得花代价编译"的过程。 主流方案是基于计数器（Counter-Based）：给每个方法、每个循环体维护一个计数器，统计它被调用/循环的次数， 一旦超过预设阈值，就认为它是"热点代码"，触发 JIT 编译。这是 HotSpot 虚拟机名字里 "Hot Spot" 的真正来源——持续追踪代码中的"热点"。

两种计数器：方法调用计数器 vs 回边计数器

HotSpot 实际上维护了两类计数器，服务于两种不同的热点场景：

方法调用计数器
(Invocation Counter)		回边计数器
(Back Edge Counter)
统计对象 某个方法被调用的次数 方法体内循环体每次跳转回循环起始位置的次数（"回边"指字节码中控制流跳回前面的指令）
触发场景 一个方法被反复调用（比如被外层高频循环调用的小方法） 一个方法内部本身有大循环，方法可能只被调用一次，但循环体执行了几百万次
触发后的编译方式 标准编译（Standard Compilation）：等下次方法调用时，直接用编译好的机器码 栈上替换（OSR, On-Stack Replacement）：方法还没执行完就切换，让正在跑的这次循环立即切到编译后的代码继续执行
为什么需要 OSR（栈上替换）？典型场景是 main 方法里有一个死循环或超大循环——这个方法本身只被调用了一次（方法调用计数器永远不会超阈值），但循环体内部执行了几百万次。如果没有回边计数器 + OSR，这种代码就永远只能解释执行，白白错过 JIT 优化的巨大收益。
典型触发 OSR 的场景
public static void main(String[] args) {
    long sum = 0;
    // main 方法只调用了一次，但这个循环的"回边"会被执行一亿次
    // 回边计数器很快超过阈值，触发 OSR：循环还在跑的时候就切换到编译后的机器码
    for (long i = 0; i < 100_000_000L; i++) {
        sum += i;
    }
}

分层编译（Tiered Compilation）：C1 + C2

只有一种编译器时会陷入两难：编译得快（优化少，机器码质量一般）还是编译得好（优化多，但编译本身耗时更长）？ HotSpot 的答案是"两个编译器分工"——现代 HotSpot 默认开启分层编译，让两种编译器接力工作：

Level 0：解释执行
  程序刚启动，全部代码走解释器，收集基础的 profiling 数据

Level 1-3：C1 编译器（Client Compiler）
  做轻量级、编译速度快的优化（如简单方法内联、去除部分冗余）
  目标：尽快把热点代码从"解释执行"提升到"能跑机器码"，抢时间

Level 4：C2 编译器（Server Compiler）
  基于更长时间的 profiling 数据，做重量级、深度的优化
  逃逸分析、激进方法内联、循环展开等(下两节详细讲)
  目标：榨干真正的超热点代码的峰值性能，编译耗时更久也值得

这种"C1 先顶上、C2 后续精修"的接力模式，让 HotSpot 既能快速响应刚变热的代码（不必等 C2 那么久的编译时间），又能对真正的核心热点代码做最彻底的优化，是"编译速度"与"优化质量"之间的又一次典型权衡（呼应上一节"解释执行 vs 编译执行"同样的启动速度/峰值性能权衡逻辑，只是发生在编译器内部的更细粒度上）。

JDK 8 之后分层编译默认开启（-XX:+TieredCompilation）。可以用 -XX:CompileThreshold 查看/调整方法调用计数器的阈值（Client 模式默认 1500，Server 模式默认 10000），面试中常问"JIT 编译阈值"就是指这个参数。

常见误区

误区："方法加了 final/写得越短就一定会被 JIT 编译"——热点探测只看执行频率，和方法长短、是否 final 本身无关（那是下面"方法内联"要考虑的另一个维度）。一个只被调用几次的巨大方法不会被编译，一个被调用几十万次的短小方法一定会被编译，两者是独立的判断维度，不要混为一谈。

核心价值总结

  • 基于计数器的热点探测：方法调用计数器统计方法被调次数，回边计数器统计循环执行次数，两者独立触发编译。
  • 栈上替换（OSR）：专门解决"方法调用次数少但内部循环巨大"的场景，让正在执行的循环直接切换到编译后的机器码。
  • 分层编译：C1 负责快速响应、轻量优化，C2 负责深度优化真正的超热点代码，两者接力兼顾编译速度与运行时性能。
