自动装拆箱陷阱

Integer a = 100 看起来是赋值，其实是编译器悄悄帮你调用了 Integer.valueOf(100) —— 这颗"糖"吃多了会咬人。

一句话讲清楚

自动装箱（Autoboxing）/自动拆箱（Unboxing）本质是编译器提供的语法糖：在需要把基本类型和包装类型互相转换的地方， javac 会自动插入 Integer.valueOf(int)（装箱）或 Integer.intValue()（拆箱）的调用。 这呼应 00-Java基础模块讲过的包装类——包装类的存在是为了让基本类型也能像对象一样被泛型、集合框架使用，而自动装拆箱就是让这层转换变得"无感"的语法糖。

类比：你往自动贩卖机（List<Integer>）投硬币（int），机器内部其实先把硬币装进一个塑料袋（Integer 对象）才能放进货道——因为货道（泛型容器）设计上只认"打包好的商品"，不认散装硬币。这个"装袋"动作就是自动装箱，你感觉不到，但它确实发生了，而且每次都要新造一个袋子（大多数情况下）。

反例：循环里对 Integer 做累加

下面这段代码"看起来"和用 int 累加没什么区别，但性能相差巨大：

BoxingLoop.java（反例 — 隐藏的性能陷阱）
Integer sum = 0;          // 包装类型
for (int i = 0; i < 1_000_000; i++) {
    sum += i;       // 编译器实际展开为下面这行
}

// 编译器真正生成的等价代码：
// sum = Integer.valueOf(sum.intValue() + i);
// 每次循环：拆箱(intValue) → int 相加 → 装箱(valueOf) 生成一个新的 Integer 对象
// 100 万次循环 ≈ 产生近 100 万个临时 Integer 对象，全部要靠 GC 回收
PrimitiveLoop.java（正例 — 用基本类型）
int sum = 0;             // 基本类型，栈上存储，没有对象、没有 GC 压力
for (int i = 0; i < 1_000_000; i++) {
    sum += i;       // 纯粹的整数加法指令，没有任何装拆箱
}

这呼应 memory 组讲过的"堆是对象分配的主战场"——反例每轮循环都在堆上新建一个 Integer 对象（大多数值超出缓存范围），大循环下会产生大量短命对象，显著增加 GC 压力；而基本类型 int 直接是栈帧里的一个局部变量槽，没有对象头开销，也不需要 GC 介入。

面试高频："为什么不要在大循环里用包装类型做累加？" —— 标准答案就是这里：每次 += 都会触发一次拆箱 + 装箱，产生大量临时对象，性能远低于基本类型。这是 Java 面试和阿里代码规范都明确提到的坑。

反例二：Integer 缓存池与 == 比较

为了减少小整数频繁装箱造成的对象浪费，Integer.valueOf() 内部维护了一个 -128 ~ 127 的缓存池，这个范围内的值会复用缓存对象而不是新建。这带来一个经典的"部分相等"诡异现象：

IntegerCachePitfall.java
Integer a = 100;
Integer b = 100;
System.out.println(a == b);   // true！100 在 -128~127 缓存范围内，a、b 指向同一个缓存对象

Integer c = 200;
Integer d = 200;
System.out.println(c == d);   // false！200 超出缓存范围，每次 valueOf 都 new 了一个新对象

// Integer.valueOf 源码简化示意：
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];  // 命中缓存，返回同一对象
    return new Integer(i);  // 未命中，new 一个新对象
}
陷阱：字面量赋值 Integer a = 100 走的是 valueOf()（享受缓存），而 new Integer(100) 永远新建对象，即便 JDK 9+ 已标记该构造器为过时，老代码里依然常见。比较包装类型的值永远应该用 .equals() 而不是 ==，这是 Java 面试最高频的坑之一，没有例外。

装拆箱触发的场景走一遍

① 赋值 / 传参场景
  基本类型 → 包装类型：编译器插入 XXX.valueOf(x) → 装箱
  包装类型 → 基本类型：编译器插入 x.xxxValue() → 拆箱

② 运算场景（+ - * / 比较等）
  包装类型参与运算前会先拆箱成基本类型，算完如需赋回包装类型再装箱
  → Integer a += 1 展开为 a = Integer.valueOf(a.intValue() + 1)

③ 集合 / 泛型场景
  泛型擦除后只能存 Object，基本类型无法直接放入 → 必须先装箱
  List<Integer> list; list.add(1) → list.add(Integer.valueOf(1))

基本类型 vs 包装类型 对比

基本类型（int）包装类型（Integer）
存储位置 栈帧局部变量表/操作数栈，无对象头开销 堆上的对象（缓存范围内复用缓存对象）
默认值 0（不能为 null） null（拆箱 null 会抛 NullPointerException）
能否用于泛型 不能，泛型擦除后只认引用类型 可以，这是包装类型存在的核心原因之一
比较方式 == 直接比较数值 应用 .equals()；== 比较的是引用，受缓存池影响
适用场景 高频计算、大数组、大循环等追求性能的场景 集合元素、泛型参数、需要表达"空值"语义的字段

常见坑 / 面试高频

坑：拆箱 null 引发 NPE。 Integer count = map.get(key); int result = count + 1; —— 如果 map.get(key) 返回 null（key 不存在），count + 1 会触发拆箱 count.intValue()，对 null 调用方法直接抛 NullPointerException。这是"业务代码里最隐蔽的 NPE 来源"之一，因为源码里根本看不到 .intValue() 这个调用。

核心价值总结

  • 装拆箱是编译器插入调用的语法糖：装箱对应 valueOf()，拆箱对应 xxxValue()，字节码里看不到"自动"二字。
  • 大循环慎用包装类型：频繁装拆箱会产生大量临时对象，增加 GC 压力，追求性能时应使用基本类型。
  • 缓存池是把双刃剑：-128~127 的 Integer 缓存减少了小整数的对象创建，但也让 == 比较的结果变得"看数值大小"，比较包装类型永远用 .equals()。
  • 拆箱 null 会 NPE：这是隐藏最深的坑，源码里看不到任何异常线索，排查时要留意包装类型参与运算的地方。
