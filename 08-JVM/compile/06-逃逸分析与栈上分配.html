逃逸分析与栈上分配(进阶)

memory 组讲过"堆是对象分配的主战场"——但如果编译器能证明一个对象"走不出"这个方法，它完全可以不劳烦堆和 GC。

一句话讲清楚

逃逸分析（Escape Analysis）是一种编译期优化技术：JIT 编译器分析一个对象的作用域， 判断它是否会"逃逸"出当前方法——比如被 return 返回、被赋值给成员变量/静态变量、被传给其他线程访问。 如果编译器能确定这个对象绝对不会逃逸，就可以大胆做几种"堆分配"之外的激进优化，而不用担心破坏程序语义。

类比：你在自己家厨房（方法内部）用完即扔的一次性筷子（局部对象），只要这双筷子全程没有被带出家门（没有逃逸），根本不需要交给市政垃圾车（GC）统一处理——你完全可以厨房内部自行处理掉，甚至压根不用真的"造"出一双筷子实体，直接用手抓（标量替换）。

例子：一个不逃逸的局部对象

EscapeAnalysisDemo.java
public int calcDistance(int x1, int y1, int x2, int y2) {
    // Point 对象只在方法内部被创建、使用，用完就没人再引用它了
    // 它既没有被 return 出去，也没有赋值给字段或静态变量，也没有传给其他线程
    // → 编译器可以确定它"不逃逸"
    Point p = new Point(x1, y1);
    return Math.abs(p.x - x2) + Math.abs(p.y - y2);
}

// 反例：如果把 p 返回出去，或赋给一个字段，它就"逃逸"了，无法再做下面的优化
private Point cachedPoint;
public Point calcAndCache(int x, int y) {
    Point p = new Point(x, y);
    this.cachedPoint = p;  // 赋值给成员变量 → 逃逸了，必须老老实实在堆上分配
    return p;
}

不逃逸之后能做哪三件事

一旦编译器确认对象不逃逸，就打开了三扇优化的门：

① 栈上分配（Stack Allocation）
  把原本要在堆上分配的对象，直接分配在当前方法的栈帧里
  方法执行完、栈帧弹出，对象随之自动销毁，完全不需要 GC 介入
  → 减轻堆的分配压力和 GC 的回收压力，这是对 memory 组"堆是分配主战场"结论的一个重要例外

② 标量替换（Scalar Replacement）
  把对象"打散"成它内部的基本类型字段（标量），直接以局部变量的形式存在
  → 连对象本身都不创建了，字段直接存在栈帧的局部变量表/寄存器里
  例：Point p 拆成两个独立的 int 变量 p_x、p_y，对象头开销都省了

③ 锁消除（Lock Elimination）
  如果确定某个对象只会被当前一个线程访问，那么它身上的 synchronized 同步操作是多余的
  → 编译器直接把这把锁的加锁/解锁指令消除掉，节省同步开销
  呼应 02-并发模块讲过的锁优化：这是锁优化家族里"发生在编译期"的一种手段
锁消除的经典场景：StringBuffer 局部变量
public String concat(String a, String b) {
    // StringBuffer 的 append 方法内部有 synchronized，
    // 但 sb 只是一个局部变量，不可能被其他线程访问到
    // 逃逸分析确定 sb 不逃逸 → JIT 直接消除掉 append() 内部的加锁操作
    StringBuffer sb = new StringBuffer();
    sb.append(a).append(b);
    return sb.toString();
}

对比：常规堆分配 vs 逃逸分析优化后

常规情况（对象逃逸）逃逸分析优化后（对象不逃逸）
分配位置 堆（Heap），由垃圾收集器统一管理 栈帧内部，甚至直接拆解为局部变量/寄存器
回收方式 依赖 GC 扫描回收，增加 GC 停顿和吞吐压力 随栈帧出栈自动销毁，零 GC 参与
同步锁 synchronized 正常加锁/解锁 如果只被单线程访问，锁可被直接消除
触发条件 - 需要 JIT 编译介入(方法必须先被判定为热点，见上一节)，解释执行阶段不会做这个分析

常见误区 / 面试高频

误区："逃逸分析已经很成熟，所以 Java 对象创建不用担心性能"——这是过度乐观的说法。逃逸分析只在对象被 JIT 编译到的热点方法里、且能静态确定不逃逸时才生效；一旦对象被存进集合、被多线程共享、或者方法本身太复杂导致编译器无法准确分析（分析本身也有成本，过于复杂的方法可能直接放弃分析），这些优化统统不会发生，对象依然老老实实地进堆。不能把逃逸分析当成"不用关心对象创建性能"的免死金牌，只是它在合适场景下能提供意外的加成。

核心价值总结

  • 逃逸分析：判断对象作用域是否会跑出方法，是栈上分配、标量替换、锁消除这三种优化共同的前提条件。
  • 栈上分配：不逃逸的对象可以分配在栈帧内，随方法返回自动销毁，减轻堆和 GC 的压力，是"堆是分配主战场"这一结论的重要例外。
  • 标量替换：更进一步——对象都不用创建了，直接把字段打散成局部变量存放，连栈上分配的开销都省了。
  • 锁消除：确定只被单线程访问的对象，其 synchronized 同步操作会被直接消除，呼应 02-并发模块讲过的锁优化技术。
  • 这几种优化都发生在 JIT 编译阶段，而且只对已经被判定为"热点"的代码生效——下一节讲的方法内联，则是这些优化能够生效的重要前提之一(内联把调用方和被调用方的代码合并，逃逸分析才能看到更完整的对象生命周期)。
