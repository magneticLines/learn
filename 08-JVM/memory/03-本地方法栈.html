本地方法栈

Java 虚拟机栈服务 Java 方法，本地方法栈服务的是那些"跳出 Java 世界"直接调用 C/C++ 代码的方法。

一句话讲清楚

本地方法栈（Native Method Stack）的作用与虚拟机栈非常相似，区别在于：虚拟机栈为虚拟机执行 Java 方法（字节码）服务， 而本地方法栈为虚拟机使用到的 native 方法服务——这些方法不是用 Java 写的，而是通过 JNI（Java Native Interface）调用的 C/C++ 等本地代码。

什么时候会用到 native 方法？比如 Object.hashCode()、System.arraycopy()、文件 IO、线程的底层调度等，很多 JDK 核心能力最终都要下沉到操作系统层面，这一层交接就是靠 native 方法完成的。

为什么 HotSpot 把它和虚拟机栈合并了？

《Java 虚拟机规范》对本地方法栈中方法使用的语言、使用方式与数据结构并没有强制规定，虚拟机可以自由实现。 HotSpot 虚拟机直接选择把本地方法栈和 Java 虚拟机栈合二为一——也就是说在 HotSpot 里，你不需要把它当成一个完全独立的内存区域来理解，它们在实现层面共用一套栈机制。

这也是为什么《深入理解 Java 虚拟机》里对本地方法栈只是简要带过：它更多是规范上的一个"占位概念"，具体实现取决于虚拟机厂商。理解到"native 方法调用也需要一个栈来管理调用帧"这个层面即可，不需要深入展开。

核心价值总结

  • 服务对象不同：虚拟机栈服务 Java 方法，本地方法栈服务 native 方法。
  • 规范不强制实现方式：虚拟机厂商可以自由决定如何实现，HotSpot 选择与虚拟机栈合并。
  • 和虚拟机栈一样，本地方法栈也会因为深度过大抛 StackOverflowError，因为申请不到足够内存也可能抛 OutOfMemoryError——这一点在 HotSpot 里体现为和虚拟机栈共用的限制。
