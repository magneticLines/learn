对象访问定位(句柄 vs 直接指针)

写 User user = new User(); 时，user 这个引用变量里到底存的是什么？答案决定了 GC 移动对象的代价。

一句话讲清楚

Java 程序通过引用（reference）操作堆上的具体对象，但《Java 虚拟机规范》只规定引用是"一个指向对象的引用"， 没有规定这个引用具体应该通过什么方式去定位、访问堆中对象的具体位置——这给了不同虚拟机实现自由发挥的空间，主流的实现方式有两种：句柄访问和直接指针访问。

方式一：句柄访问（Handle）

如果用句柄访问，Java 堆中会划分出一块内存作为句柄池，reference 中存储的是对象的句柄地址， 句柄中包含了对象实例数据的指针和对象类型数据的指针两部分信息。

reference（栈帧局部变量表中）
  └─→ 句柄池中的句柄
      ├─→ 指向实例数据（堆中）
      └─→ 指向类型数据（方法区/元空间中）

优点：reference 中存储的是稳定的句柄地址，当对象被移动时（比如垃圾收集时的整理/复制过程），只需要改变句柄中的实例数据指针，reference 本身不需要修改。

方式二：直接指针访问（Direct Pointer）

如果用直接指针访问，reference 中存储的直接就是对象在堆中的地址。对象内存布局中（呼应上一篇讲的对象头）必须考虑如何放置访问类型数据的相关信息，通常是在对象头里存一个指向类型数据的指针。

reference（栈帧局部变量表中）
  └─→ 堆中的对象实例数据　（对象头里的类型指针再指向方法区的类型数据）

优点：速度更快，因为只需要一次指针解引用就能访问到对象，节省了句柄访问中"先找句柄、再找实例数据"的一次转向开销。HotSpot 虚拟机主要使用这种方式。

两种方式对比

维度句柄访问直接指针访问（HotSpot 采用）
访问速度 较慢，需要经过句柄这层间接层 更快，reference 直接指向实例数据，少一次跳转
对象被移动时的开销 只需修改句柄中的指针，reference 本身不变——对 GC 移动对象更友好 需要更新所有指向该对象的 reference，因为它们直接存的就是旧地址
额外空间开销 需要维护一块独立的句柄池 不需要额外的句柄池
使用频率 访问相对少（reference 赋值/传递较频繁，但访问对象本身次数远高于此，故直接指针更划算），概念上更常见于教学讨论 Java 中访问对象是最频繁的操作之一，节省的这一次间接开销，累积起来是非常可观的性能收益
面试高频："HotSpot 用哪种方式？为什么？"——直接指针访问，因为对象访问在 Java 程序中发生的频率极高，直接指针省去了句柄这一层间接寻址，这类"高频路径优化"正是虚拟机设计里常见的权衡取舍：牺牲一定的 GC 移动对象时的便利性，换取日常访问路径上的速度提升。
不要死记"HotSpot 只能用直接指针"：规范并未强制规定访问方式，这是 HotSpot 的实现选择，不代表所有 JVM 实现都这样。理解取舍逻辑比死记结论更重要——这也是本书反复强调的"透过现象看虚拟机设计哲学"。

核心价值总结

  • reference 只是一个访问入口：规范不规定具体怎么定位对象，留给虚拟机自由实现。
  • 句柄访问：多一层间接层，换来对象移动（GC 整理/复制）时 reference 不用改的好处。
  • 直接指针访问：少一层间接层，访问更快，但对象被移动时所有引用都要跟着更新——HotSpot 选择了这种方式。
  • 这个知识点会在后面"垃圾收集"Tab 讲到复制算法/标记整理算法时再次被提起：对象被移动后指针要如何更新，是垃圾收集器实现中的一个核心工程问题。

与其他知识点的联系

  • 08-对象头与MarkWord：直接指针访问方式下，对象头里的类型指针承担了"指向类型数据"的职责，这正是对象访问路径能够只用一次解引用就找全信息的关键，两篇合起来才是完整的"reference 如何找到对象"的故事。
  • 07-对象的内存布局：直接指针访问要求对象头本身就要留出类型指针这块空间，这也解释了为什么对象头不能只放 Mark Word，必须同时兼顾"访问定位"这个职责。
  • 05-标记复制算法（gc）：句柄访问和直接指针访问的核心分歧就在"对象被移动时谁的代价更小"，标记复制算法要把存活对象整体搬到另一块空间，HotSpot 选择直接指针后，意味着复制算法执行时必须逐一更新所有指向被移动对象的引用，这是访问定位方式对垃圾收集算法实现复杂度的直接影响。
  • 11-G1收集器（gc）：G1 等现代收集器在移动对象后需要修正引用，这类"指针更新"的工程量正是本篇末尾提到的"直接指针访问下对象移动开销更大"的具体体现。
