堆(Heap)——对象分配的主战场

Java 里几乎所有 new 出来的东西都住在这里，也是垃圾收集器工作最频繁的战场。

一句话讲清楚

Java 堆（Heap）是虚拟机管理的内存中最大的一块，被所有线程共享，在虚拟机启动时创建。 它唯一的目的就是存放对象实例——《Java 虚拟机规范》里的描述是"所有的对象实例以及数组都要在堆上分配"。

这里用了"几乎"，是因为随着即时编译技术的发展，逃逸分析可以判断某些对象不会被方法外部引用，从而做栈上分配优化——这是编译优化范畴的内容，先埋个伏笔，后面"编译优化"Tab 会展开讲。这里先建立"堆是对象分配的主战场"这个主干认知。

和虚拟机栈的根本区别：虚拟机栈是线程私有的，栈帧随方法调用产生、随方法返回销毁；堆是线程共享的，对象的生命周期不与某个方法调用绑定，可能被多个线程同时持有引用——这正是为什么堆上的对象需要垃圾收集器来管理生命周期，而栈帧不需要（栈帧的销毁是确定性的，方法一返回就自动弹出）。

举例：一段代码里对象分配到了哪

Demo.java
public void createUser() {
    User user = new User("Tom");  // user 引用在栈帧的局部变量表里
                                    // new User(...) 这个对象实例在堆上分配
    user.setAge(18);
}
虚拟机栈（线程私有）
  createUser() 栈帧
    局部变量表：user → [堆上某地址] ← 只存了一个引用（地址）

堆（线程共享）
  ┌─────────────────────┐
  │ User 对象实例 │ ← name="Tom", age=18 真正的数据都在这
  └─────────────────────┘

局部变量 user 本身占用的是栈帧里一个变量槽，存的只是"指向堆上对象的引用"；new User("Tom") 真正申请的内存，在堆上。 方法执行完 user 这个引用随栈帧一起销毁，但 User 对象是否能被回收，要看堆里还有没有其他引用指向它——这是垃圾收集要解决的问题。

堆的内部结构：分代收集理论下的划分

现代商用虚拟机大多基于分代收集理论设计垃圾收集器：绝大多数对象都是"朝生夕死"的（比如方法内的临时对象），少数对象会存活很久。 基于这个经验规律，HotSpot 把堆进一步划分为新生代（Young Generation）和老年代（Old Generation），新生代里又细分出 Eden 区和两个 Survivor 区（通常称为 From 和 To）。

Java 堆
├── 新生代（Young Generation）　— 对象刚创建时优先分配在这
│　　├── Eden 区　　　　　— 绝大多数新对象在这里诞生
│　　├── Survivor From　　— 存放至少经历过一次 GC 还存活的对象
│　　└── Survivor To　　　— GC 时 From 存活对象拷贝到这，角色互换
│　　（默认比例 Eden : From : To = 8 : 1 : 1）
│
└── 老年代（Old Generation）　　— 存放长期存活或占用内存较大的对象

新生代里的对象每熬过一次垃圾收集，年龄就加一，达到一定阈值（默认 15）后会被晋升到老年代。这一整套"新对象在 Eden 分配 → 存活对象在两个 Survivor 间倒腾 → 长期存活对象晋升老年代"的机制，正是为后面"垃圾收集"Tab（各种收集器如何回收新生代、老年代）打下的地基，这里先建立结构认知即可。

堆 vs 栈：一张表看清本质区别

维度虚拟机栈堆
共享性 线程私有，随线程创建销毁 线程共享，随虚拟机启动创建
存放内容 栈帧：局部变量表、操作数栈等 对象实例、数组
分配/回收方式 方法调用时压栈，方法返回时自动出栈，确定性极强 由垃圾收集器根据可达性动态回收，时机不确定
速度 快（指针移动即可） 相对慢（需要考虑并发分配、垃圾回收开销）
常见异常 StackOverflowError（深度超限） OutOfMemoryError: Java heap space（对象过多回收不及）
常见误区："堆是所有对象存放的唯一地方"——这个说法在教学上够用，但不完全准确。逃逸分析支持下，未逃逸出方法的对象可能被优化为栈上分配（甚至标量替换后直接消失，字段变成局部变量）。面试时如果被问到这一点，能提一句"逃逸分析下不完全成立"会显得理解更深入。

核心价值总结

  • 堆是对象分配的主战场：几乎所有对象实例和数组都在这里分配，是虚拟机管理的最大一块内存。
  • 线程共享：和线程私有的虚拟机栈形成鲜明对比，也正因为共享，才需要垃圾收集器介入管理生命周期。
  • 分代结构：新生代（Eden + 两个 Survivor）+ 老年代，基于"对象朝生夕死"的经验规律设计，是垃圾收集算法和收集器实现的基础。
  • 堆是可以按需扩容/收缩的（-Xms/-Xmx），如果扩容到最大值仍无法容纳新对象，会抛 OutOfMemoryError: Java heap space，后面会专门讲排查思路。

与其他知识点的联系

  • 02-Java虚拟机栈与栈帧：堆和虚拟机栈是"共享 vs 私有"这一对核心对比关系，堆需要垃圾收集器介入正是因为对象生命周期不与某次方法调用绑定，而栈帧的销毁是确定性的，理解这一点才能理解为什么 GC 只盯着堆（以及方法区）而不用管栈帧回收。
  • 06-对象的创建过程：堆的分代结构（Eden + 两个 Survivor + 老年代）正是对象创建过程中"分配内存"这一步的落地场所，TLAB 机制也是在 Eden 区里划出的私有分配区，两者是同一件事的不同视角。
  • 05-方法区与元空间的演变：堆和方法区都是线程共享区域，但方法区在 JDK 8 后已经搬出堆用本地内存实现，这也是为什么"Java 堆 OOM"和"元空间 OOM"是两种完全不同的报错和排查路径。
  • 13-对象优先在Eden分配（gc）：这一篇讲的分代结构是"对象优先在 Eden 分配"这条 GC 规则的前提知识，Eden 满了触发 Minor GC、存活对象在 Survivor 间倒腾、晋升老年代的完整流程，都是建立在这里划分出的堆内部结构之上。
  • 05-标记复制算法（gc）：新生代之所以用复制算法而不是标记整理，正是因为堆里新生代的"朝生夕死"特性（这里讲的分代收集理论前提），复制算法牺牲一半空间换来的高效率在新生代这个场景下性价比最高。
