对象的创建过程

new 一个对象看起来只是一行代码，虚拟机在背后其实要走完一整套严谨流程。

一句话讲清楚

执行 new User() 这样一条字节码指令时，虚拟机要依次完成：类加载检查 → 分配内存 → 处理并发安全 → 初始化零值 → 设置对象头 → 执行构造方法， 只有全部走完，一个"完整可用"的对象才真正诞生。

完整流程走一遍

① 类加载检查
  虚拟机遇到 new 指令时，先检查这个类是否已被加载、解析、初始化过
  没有 → 先走类加载过程（呼应"类加载机制"Tab：加载→验证→准备→解析→初始化）

② 分配内存
  类加载完成后，对象所需内存大小已经确定
  从堆中划分出一块对应大小的内存，两种分配方式：
    · 指针碰撞（Bump the Pointer）：堆内存规整（用过的在一边，空闲的在另一边，中间一个指针），分配就是指针往空闲方向挪一段距离
    · 空闲列表（Free List）：堆内存不规整（用过和空闲的交错），虚拟机需要维护一张列表记录哪些块可用
  采用哪种方式取决于所用垃圾收集器是否带压缩整理功能

③ 处理并发分配安全问题
  多线程同时 new 对象，如果都直接挪同一个指针，会有竞态条件
  方案一：对分配内存空间的动作做同步（CAS + 失败重试）
  方案二（HotSpot 默认）：TLAB（Thread Local Allocation Buffer，本地线程分配缓冲）
    每个线程在 Eden 区预先分配一小块专属内存，分配对象时优先在自己的 TLAB 里分，不用每次都和别的线程抢锁

④ 初始化零值
  虚拟机将分配到的内存空间（除对象头外）全部初始化为零值（int 为 0，引用为 null...）
  保证对象的实例字段在 Java 代码中不赋初值也能直接使用，程序能访问到这些字段的类型对应的零值

⑤ 设置对象头
  设置对象是哪个类的实例、如何找到类的元数据信息、对象的哈希码、GC 分代年龄等信息
  这些信息存放在对象头（Object Header）中（下一篇会详细展开）

⑥ 执行 <init>（构造方法）
  到这一步，从虚拟机视角看，一个新对象已经产生
  但从 Java 程序视角看，"对象创建"才刚刚开始——接下来执行构造方法，按照程序员的意图对字段赋值
  这一步完成后，才是一个真正可用的对象
面试高频坑："字段初始化为 0 和构造方法里赋值，是同一步吗？"——不是。第 ④ 步的"零值初始化"是虚拟机层面的安全兜底，保证内存不会有脏数据；第 ⑥ 步执行构造方法才是按 Java 语义真正赋予业务含义的值。这也是为什么 int i; 不赋值默认是 0 而不是随机值——这是虚拟机保证的，不是"侥幸"。

为什么需要 TLAB？——呼应线程封闭思想

这一点直接联系到 02-并发模块 讲过的"用线程封闭规避竞争"思路：如果每次分配内存都要给多个线程共享的分配指针加锁，堆内存分配会成为高并发场景下的严重瓶颈——毕竟对象创建是 Java 程序里发生频率最高的操作之一。

TLAB 的思路是：与其每次都加锁争抢，不如干脆提前给每个线程划一块只属于自己的"私有仓库"（在 Eden 区里划出一小块），线程在自己的仓库里分配对象完全不需要同步。只有当 TLAB 用完需要重新申请一块新的时，才需要短暂地做一次同步。

注意边界：TLAB 解决的是"多线程并发分配对象"的效率问题，不是让对象本身线程安全。对象一旦创建完毕，如果被多个线程共享持有，依然需要靠锁、不可变、原子类等手段保证并发安全——这是两个不同层面的问题，不要混为一谈。

核心价值总结

  • 六个步骤：类加载检查 → 分配内存 → 并发安全处理（TLAB）→ 零值初始化 → 设置对象头 → 执行构造方法。
  • 指针碰撞 vs 空闲列表：取决于堆是否规整，而堆是否规整又取决于所用垃圾收集器有没有压缩整理能力，这是内存分配和垃圾收集算法之间的一处重要联动。
  • TLAB 是"线程封闭"思想在内存分配上的落地：用空间换时间，避免高频对象分配场景下的锁竞争。
  • 理解这个流程后，下一篇讲对象内存布局（对象头/实例数据/对齐填充）就会顺理成章——第 ⑤ 步"设置对象头"正是衔接点。
