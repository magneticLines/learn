内存区域与溢出

常见 OOM 场景排查(堆 / 栈 / 元空间 / 直接内存)

把前面十篇讲过的内存区域串起来，看看它们各自"爆掉"时长什么样，以及排查该往哪个方向想。

一句话讲清楚

不同内存区域溢出时，异常信息里会带上明确的区域标识（这是 JDK 特意设计的，方便定位问题）。记住这几个关键字，看到异常信息第一时间就能判断问题出在哪个区域，而不用瞎猜。

场景一：堆溢出 — Java heap space

最常见的一种。典型成因是内存泄漏：一部分对象本该被回收，却因为被某个长生命周期的引用（比如 static 集合、未关闭的连接、缓存没有淘汰策略）一直持有，导致垃圾收集器无法回收，堆越占越多。

典型内存泄漏模式
// 反例：静态集合只增不减，本质是把对象的生命周期"意外"延长到了和应用一样长
public class UserCache {
    private static final Map<String, User> CACHE = new HashMap<>();

    public static void put(String id, User user) {
        CACHE.put(id, user);  // 只 put，从不 remove/过期淘汰
    }
}

排查方向：用 jmap -dump 或添加 -XX:+HeapDumpOnOutOfMemoryError 让 OOM 发生时自动生成堆转储文件，再用 MAT（Memory Analyzer Tool）等工具分析，找到支配树（Dominator Tree）里占用内存最大、且不该继续存活的对象，顺藤摸瓜找到持有它的引用链。

场景二：元空间溢出 — Metaspace

呼应第 5 篇讲过的方法区演变：元空间存的是类的元信息，典型成因是动态生成类过多——比如大量使用 CGLIB 动态代理但没有复用（每次都生成新的代理类）、频繁的类加载器泄漏（自定义 ClassLoader 加载完没释放，导致它加载的类也无法被卸载）。

常见坑：如果应用支持"热部署""插件热加载"，每次重新加载都创建一个新的 ClassLoader 却没有正确销毁旧的，旧 ClassLoader 及其加载的所有类会一直驻留在元空间里，多次重复热部署后必然 OOM。

场景三：直接内存溢出 — Direct buffer memory

直接内存不属于堆内存，但也受物理机总内存限制。典型成因是 NIO 中通过 ByteBuffer.allocateDirect() 分配的堆外内存使用不当——比如没有及时释放，或者分配的直接内存总量超过了 -XX:MaxDirectMemorySize 的限制。

直接内存溢出示意
for (;;) {
    ByteBuffer.allocateDirect(1024 * 1024);  // 不断分配堆外内存，没有释放
}
// 抛出：java.lang.OutOfMemoryError: Direct buffer memory

排查方向：这类 OOM 发生时，用常规的堆转储分析工具往往看不到明显异常（因为对象本身在堆里很小，只是持有一个指向堆外内存的引用），需要专门关注 NIO 相关代码是否正确调用释放逻辑，或检查是否有第三方库内部使用了直接内存但没有妥善管理。

三种场景一览

OOM 类型关键字典型成因排查工具/方向
堆溢出 Java heap space 对象一直被引用无法回收（内存泄漏），常见于集合类无限增长 jmap + MAT 分析堆转储，找支配树里的"大对象"和引用链
元空间溢出 Metaspace 动态生成类过多，或类加载器泄漏导致类无法卸载 检查是否有热部署/动态代理场景未正确复用或释放 ClassLoader

通用排查思路总结

  1. 看异常信息的关键字，先确定是哪个区域溢出（heap / Metaspace / Direct buffer memory / unable to create new native thread 等）。
  2. 复现并加参数留证据：生产环境提前加上 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=xxx，OOM 发生时自动落盘堆转储文件，避免"重启一次问题消失，什么证据都没留下"。
  3. 用工具分析而不是猜：MAT、JProfiler、Arthas 等工具能直接告诉你占用最多内存的对象类型和引用链，比凭经验瞎猜效率高得多——这类工具的具体使用，留给后面"性能监控与调优"Tab 详细展开。
  4. 区分"参数不够"和"代码有问题"：简单粗暴地调大 -Xmx/-XX:MaxMetaspaceSize 有时能临时缓解，但如果根因是内存泄漏，调大参数只是让 OOM 发生得晚一点，问题迟早还会复现。
贯穿全篇的核心心法：OOM 排查的本质，就是把"某个区域内存不够用了"这个笼统的报错，通过工具和方法论，一步步缩小范围到"具体是哪些对象、被谁持有、为什么没被回收"。前面十篇讲的每个内存区域的作用和边界，就是让你在看到 OOM 时能第一时间做出正确判断的知识基础。

核心价值总结

  • 三类高频 OOM：堆溢出（对象泄漏）、元空间溢出（类泄漏）、直接内存溢出（堆外内存泄漏），各自有清晰的异常关键字标识。
  • 排查靠工具不靠猜：堆转储 + MAT 分析是定位内存泄漏最可靠的手段。
  • 治本而非治标：调大内存参数只能延缓问题，找到并修复引用链上的泄漏点才是根本解法。
  • 至此"内存区域与溢出"这一组的十一篇内容已经形成闭环：从五大内存区域的划分，到对象创建/布局/访问的底层机制，再到两类典型错误的辨析与排查——这些都是后面"垃圾收集"Tab 展开各类收集算法和收集器的地基。
