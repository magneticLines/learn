对象头与 Mark Word(进阶)

只有 8 字节的 Mark Word，是怎么在无锁、偏向锁、轻量级锁、重量级锁之间"身兼数职"的？

一句话讲清楚

Mark Word 是对象头中占 8 字节（64 位 JVM）的一个字段，它被设计成与对象自身定义的数据无关的额外存储空间， 而且是高度复用的——在代码运行过程中，Mark Word 存储的内容会随着对象锁状态的变化而变化，用同一块内存在不同时刻表达不同含义，以尽量节省对象头占用的空间。

为什么要这么"抠门"地复用？因为 Java 程序里对象数量极其庞大，哪怕对象头只多占几个字节，乘以海量对象后也是可观的内存开销。与其给每种状态都单独留一块固定空间，不如设计成"哪种状态用哪种解读方式"，这是典型的空间换算法复杂度换来的极致内存优化。

Mark Word 在不同锁状态下存的是什么

这里直接呼应 02-并发模块讲过的锁升级过程——偏向锁 → 轻量级锁 → 重量级锁的"锁膨胀"，其存储载体正是 Mark Word。

对象状态Mark Word（64位，简化示意）存储内容
无锁（Normal） 对象的哈希码（HashCode）+ 分代年龄（Age）+ 锁标志位
偏向锁（Biased） 偏向的线程 ID + 偏向时间戳 + 分代年龄 + 锁标志位——记录"这把锁偏向哪个线程"，同一线程再次进入直接免检查
轻量级锁（Lightweight） 指向栈帧中锁记录（Lock Record）的指针——不再存哈希码，而是把 Mark Word 原内容拷贝到线程栈的锁记录里，自己变成一个指针
重量级锁（Heavyweight） 指向互斥量（Monitor，操作系统 mutex）的指针——真正依赖操作系统级别的同步原语，涉及线程阻塞和内核态切换，代价最高
GC 标记（Marked for GC） 在某些垃圾收集算法的标记阶段，Mark Word 还可能被复用来存标记信息

这就是为什么理解"锁"不能只停留在 synchronized 关键字的语法层面：从内存角度看，锁状态的变化本质上是同一个 8 字节字段的解读方式在切换——无锁时它存哈希码，加锁后它存的是"到哪里能找到锁的详细信息"的指针。

面试高频："对象的哈希码存在哪？调用了 hashCode() 之后再加锁会怎样？"——一旦对象计算过哈希码（调用过 Object.hashCode() 或 System.identityHashCode()），Mark Word 就必须保留这个哈希码，此时该对象不能再进入偏向锁状态（因为偏向锁模式下 Mark Word 没地方放哈希码了），只能直接走轻量级锁/重量级锁路径。这是一个非常经典的"两个特性互相挤占同一块存储空间"的例子。

为什么这块内容属于"进阶"

Mark Word 的具体位分布（哪几位存哈希码、哪几位存锁标志）依赖虚拟机具体实现，并且在 32 位和 64 位虚拟机、是否开启压缩指针等条件下都会有差异，业务开发日常并不需要记忆具体的位布局。 真正有价值的是建立这个认知模型：锁状态的膨胀，本质上是对象头这块存储空间里内容的重新解读和搬迁，这能帮你更深刻理解为什么"锁升级不可逆"（轻量级锁升级为重量级锁后不会再降级回来）——因为一旦 Mark Word 变成指向 Monitor 的指针，原有的哈希码、偏向信息等就已经被搬走或作废，退回去的代价和收益不成正比，JVM 设计者索性不支持降级。

核心价值总结

  • Mark Word 是多用途字段：同一块 8 字节内存，根据对象当前锁状态复用存储不同信息。
  • 锁膨胀的存储基础：偏向锁存线程 ID，轻量级锁存栈帧锁记录指针，重量级锁存 Monitor 指针——这是理解"锁升级"绕不开的底层视角。
  • 哈希码与偏向锁互斥：调用过 hashCode() 的对象无法进入偏向锁状态，这是一个常见的面试陷阱和性能细节。
  • 这一篇是对 02-并发模块锁升级知识的"补课"——从内存布局视角，把"锁到底存在哪里"这个问题彻底讲透。
