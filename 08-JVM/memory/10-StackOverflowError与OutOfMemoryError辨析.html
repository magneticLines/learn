内存区域与溢出

StackOverflowError 与 OutOfMemoryError 辨析

两个新手最容易混淆的错误，本质区别只有一句话：一个是"深度"问题，一个是"总量"问题。

一句话讲清楚

StackOverflowError（SOE）：线程请求的栈深度超过了虚拟机所允许的最大深度，典型触发场景是没有终止条件的递归调用。
OutOfMemoryError（OOM）：某个内存区域（堆、方法区/元空间、直接内存等）已经没有足够空间容纳新的数据，而且这个区域还能继续扩展（否则会先抛别的错误）。

本质区别一句话记住：SOE 是"想要的深度超过了允许的深度"（哪怕总内存很充裕，栈也可能溢出）；OOM 是"想要的空间超过了能给的总量"。两者都属于 java.lang.Error 而不是 Exception，含义是：这是虚拟机层面的严重问题，通常不建议、也很难通过 try-catch 优雅恢复。

反例：没有终止条件的递归

这是呼应 00-Java基础模块讲过的递归概念——递归本质上就是不断地方法自己调用自己，每调用一层就压入一个新栈帧。如果忘记写终止条件（或者终止条件永远不满足），栈帧会无限压入，直到耗尽虚拟机分配给这个线程的栈空间。

StackOverflowDemo.java（反例）
public class StackOverflowDemo {

    private int depth = 0;

    public void recurse() {
        depth++;
        recurse();  // 没有任何终止条件，无限递归下去
    }

    public static void main(String[] args) {
        new StackOverflowDemo().recurse();
        // 抛出：java.lang.StackOverflowError
    }
}

每调用一次 recurse()，就有一个新栈帧压入虚拟机栈，栈帧越压越多，最终超过 -Xss 参数设置的单线程栈容量上限，抛出 StackOverflowError。

常见误区："栈溢出是因为内存不够"——不完全准确。栈溢出是深度超限，即使机器物理内存非常充裕，只要单个线程分配的栈空间用尽（受 -Xss 限制），一样会抛 SOE。反而是加大 -Xss 能一定程度缓解（允许更深的递归），但这只是"续命"，真正的问题永远是代码逻辑（缺少终止条件、非尾递归转循环等）。

OutOfMemoryError：想要的空间要不到了

与 SOE 不同，OOM 是某个内存区域总量不够用了。比如堆里一直有对象被引用、无法被回收，持续增长最终撑满堆空间：

HeapOOMDemo.java（反例 — 集合无限增长导致内存泄漏）
public class HeapOOMDemo {

    static List<byte[]> cache = new ArrayList<>();  // static，永远不会被回收

    public static void main(String[] args) {
        while (true) {
            cache.add(new byte[1024 * 1024]);  // 不断往静态集合塞数据，永不释放
        }
        // 抛出：java.lang.OutOfMemoryError: Java heap space
    }
}

两者对比一览

维度StackOverflowErrorOutOfMemoryError
本质 请求的栈"深度"超过限制 请求的内存"总量"超过限制
发生区域 虚拟机栈 / 本地方法栈 堆、方法区/元空间、直接内存等
典型触发场景 无终止条件（或条件写错）的递归调用 内存泄漏（对象一直被引用回收不掉）、动态生成类过多、直接内存分配未释放
能否靠调参数根治 加大 -Xss 只能延缓，根治要靠改代码逻辑 加大 -Xmx 等参数同样只能延缓，根治要找到内存泄漏点或不合理占用
排查方向 看异常堆栈里是否有大量重复的方法调用链（一眼能看出递归没收敛） 需要用 jmap/MAT 等工具分析堆转储（heap dump），找到占用最多、无法被回收的对象
面试高频："栈溢出会不会引发 OOM，反过来呢？"——理论上如果虚拟机实现允许栈动态扩展且没有明确上限，扩展栈时可能因为申请不到内存而抛出 OOM 而不是 SOE（规范里也提到了这种可能性）；但在 HotSpot 默认配置下，两者边界通常是清晰分开的：栈深度超限就是 SOE，其他区域总量不够就是 OOM。能提到这一层"规范留了口子，但 HotSpot 实践中边界分明"，说明理解足够细致。

核心价值总结

  • SOE 是深度问题：典型元凶是失控的递归，排查看异常堆栈的重复模式。
  • OOM 是总量问题：典型元凶是内存泄漏、动态类过多、本地内存未释放，排查需要堆转储分析工具。
  • 两者都是 Error 级别，代表虚拟机层面的严重问题，不应该用 try-catch 当作业务逻辑的一部分去"兜底掩盖"，而应该找到并修复根因。
