Java 虚拟机栈与栈帧

方法调用为什么能"层层深入又层层返回"？答案就是一叠不断压入弹出的栈帧。

一句话讲清楚

Java 虚拟机栈（Java Virtual Machine Stack）是线程私有的，生命周期与线程相同。 每个方法被调用时，虚拟机都会同步创建一个栈帧（Stack Frame）压入这个栈中，方法执行完毕（正常返回或抛出异常）后，对应栈帧就出栈。 整个调用链的过程，本质上就是栈帧不断压栈（push）和出栈（pop）的过程。

类比：一摞盘子。你炒菜时用一个盘子（方法 A 执行），中途需要装配菜（调用方法 B），就在 A 的盘子上面再摞一个盘子（B 入栈）；配菜做完撤走这个盘子（B 出栈），继续用下面的盘子（回到 A 继续执行）。永远只能动最上面那个盘子——这正是"栈"后进先出的特性。

栈帧里有什么？

每个栈帧至少包含以下几部分信息（这里先简要认识，第 8 章"字节码执行引擎"会展开讲解操作数栈如何配合字节码指令完成计算）：

  • 局部变量表（Local Variable Table）：存放方法参数和方法内定义的局部变量，以"变量槽（Slot）"为最小单位。
  • 操作数栈（Operand Stack）：方法执行过程中做运算、传参、接收返回值使用的临时工作区。
  • 动态链接（Dynamic Linking）：指向运行时常量池中该方法所属类的引用，方便运行期动态解析方法调用。
  • 方法返回地址：方法退出后要回到调用者的哪个位置继续执行。

举例：A 调 B 调 C，栈是怎么变化的

Demo.java
public class Demo {

    public void methodA() {
        methodB();   // A 调用 B
    }

    public void methodB() {
        methodC();   // B 调用 C
    }

    public void methodC() {
        int result = 1 + 1;  // C 执行自己的逻辑后返回
    }
}
调用 methodA()
  ┌───────────────┐
  │ methodA 栈帧 │ ← 栈底，最先入栈
  └───────────────┘

methodA 内部调用 methodB()
  ┌───────────────┐
  │ methodB 栈帧 │ ← 新栈帧压入栈顶
  ├───────────────┤
  │ methodA 栈帧 │
  └───────────────┘

methodB 内部调用 methodC()
  ┌───────────────┐
  │ methodC 栈帧 │ ← 当前正在执行的方法，永远在栈顶
  ├───────────────┤
  │ methodB 栈帧 │
  ├───────────────┤
  │ methodA 栈帧 │
  └───────────────┘

methodC 执行完毕，出栈 → 回到 methodB 继续执行 → methodB 执行完出栈 → 回到 methodA ... 直至栈空

我们平时看到的异常堆栈信息（Stack Trace），本质上就是打印这一叠栈帧从栈顶到栈底的方法调用链——这也是为什么异常堆栈能精确告诉你"报错发生在哪个方法、被谁调用"。

为什么虚拟机栈也是线程私有的？

和程序计数器一样，虚拟机栈同样是线程私有的：每个线程在创建时都会拥有一个独立的虚拟机栈，其生命周期与线程相同。 这意味着局部变量表里的数据天然只属于当前线程，不会被其他线程读写——这正是 02-并发模块讲过的"线程封闭"在方法调用层面的体现：局部变量不需要加锁就是线程安全的，而共享的堆内存则不然。

面试高频："为什么方法内的局部变量是线程安全的？"——因为局部变量存在虚拟机栈的栈帧里，栈是线程私有的，不存在多线程竞争。但如果局部变量是一个指向堆中共享对象的引用，那个对象本身依然可能被多线程共享访问，需要额外考虑同步问题。

核心价值总结

  • 栈帧对应方法调用：每次方法调用创建一个栈帧压栈，方法返回时栈帧出栈，栈的深度就是当前方法调用链的深度。
  • 局部变量表 + 操作数栈：是方法执行时存参数、存中间结果的地方，细节留给"字节码执行引擎"章节展开。
  • 线程私有：虚拟机栈随线程创建而创建，是局部变量天然线程安全的根本原因。
  • 栈的深度是有限制的，方法调用链过深（典型场景：无终止条件的递归）会导致 StackOverflowError，后面会专门辨析这个错误。

与其他知识点的联系

  • 关联程序计数器（01 篇）：程序计数器记录的是"当前线程执行到哪一行字节码"，虚拟机栈记录的是"当前线程调用链的完整现场"——每次方法调用切换时，程序计数器指向新方法的入口，同时一个新栈帧压入虚拟机栈，两者配合才能保证方法返回后能准确恢复到调用者当时的执行位置和局部变量状态。
  • 关联对象分代年龄记录在 Mark Word（gc 组 15 篇）：栈帧里的局部变量表存的是变量的值或对堆中对象的引用，而对象自身的元数据（如分代年龄、锁状态）存在堆中对象头的 Mark Word 里——这提醒我们"栈上的引用"和"堆上的对象"是两块完全不同的内存区域，垃圾收集器判断对象是否存活，正是要从栈帧的局部变量表这类 GC Roots 出发去追踪堆中对象的引用链。
  • 关联可达性分析算法（gc 组关于判断对象存活的篇章）：虚拟机栈中每个栈帧的局部变量表，正是可达性分析里 GC Roots 的重要来源之一——只要某个对象能从某个存活线程的栈帧局部变量表出发被引用链追踪到，它就不会被判定为垃圾，这是"线程私有的栈"和"线程共享的堆"之间唯一的合法连接点。
  • 关联大对象直接进老年代（gc 组 14 篇）：栈帧本身大小相对固定且较小，但方法内创建的对象（尤其是大对象）是分配在堆上而非栈上——理解"栈存引用、堆存对象"这条基本分工，才能理解为什么栈溢出（StackOverflowError）和堆内存分配策略是两套完全独立的问题，不能混为一谈。
