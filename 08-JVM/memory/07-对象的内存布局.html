对象的内存布局

一个对象在堆里到底占多少字节？拆开来看，它由三部分严丝合缝地拼接而成。

一句话讲清楚

在 HotSpot 虚拟机中，一个对象在堆内存中的存储布局可以分为三个部分：对象头（Header）、实例数据（Instance Data）、对齐填充（Padding）。

┌─────────────────────────┐
│ 对象头 Header │ Mark Word + 类型指针（+ 数组长度，若是数组）
├─────────────────────────┤
│ 实例数据 Instance Data │ 各字段真正的值（父类字段 + 自身字段）
├─────────────────────────┤
│ 对齐填充 Padding │ 补齐到 8 字节整数倍（可有可无）
└─────────────────────────┘

逐部分拆解

1. 对象头（Header）

对象头本身又分为两块（如果是数组对象，还会多一块记录数组长度）：

  • Mark Word：存储对象自身的运行时数据，比如哈希码、GC 分代年龄、锁状态标志等。在 64 位虚拟机中占 8 字节。这是一个高度复用的字段，下一篇会专门展开。
  • 类型指针（Klass Pointer）：指向该对象对应的类的元数据（存在方法区/元空间），虚拟机通过它确定这个对象是哪个类的实例。默认开启指针压缩时占 4 字节，未开启则是 8 字节。

2. 实例数据（Instance Data）

这是对象真正"有意义"的部分——程序代码中定义的各种类型的字段内容，包括从父类继承下来的和自身定义的。存储顺序会受到虚拟机分配策略（比如相同宽度的字段分配到一起）以及字段在 Java 源码中定义顺序的影响。

3. 对齐填充（Padding）

这部分不是必然存在的，也没有特别含义，仅仅起占位符的作用。因为 HotSpot 虚拟机的自动内存管理系统要求对象起始地址必须是 8 字节的整数倍，换句话说对象大小必须是 8 字节的整数倍。当对象头 + 实例数据算下来不满足这个条件时，就用对齐填充来补齐。

为什么要 8 字节对齐？这是为了方便虚拟机高效寻址和内存管理——很多 CPU 架构按字长对齐访问内存效率更高，8 字节对齐也是指针压缩（Compressed Oops）能正常工作的前提之一。

举例：算一下一个简单对象大概占多少字节

User.java（64 位 JVM，默认开启指针压缩）
public class User {
    private int age;       // int：4 字节
    private boolean vip;   // boolean：1 字节
}
对象头：Mark Word 8 字节 + 压缩类型指针 4 字节　　　　　= 12 字节
实例数据：int age 4 字节 + boolean vip 1 字节　　　　　= 5 字节
小计：12 + 5 = 17 字节
对齐填充：补到 8 的整数倍（下一个是 24）　　　　　　　= 7 字节
────────────────────────────
对象总大小 ≈ 24 字节
注意：这只是示意性估算，实际大小会受 JVM 版本、是否开启指针压缩（-XX:+UseCompressedOops，64 位 JVM 默认开启）、字段重排列策略等因素影响。想要精确测量，生产环境常用 JOL（Java Object Layout）工具直接打印真实内存布局，不要凭手算当作绝对结论去做容量规划。

核心价值总结

  • 三段式布局：对象头（Mark Word + 类型指针）+ 实例数据 + 对齐填充。
  • 对象头是"元信息仓库"：不存业务数据，存的是哈希码、锁状态、GC 年龄、类型指向等虚拟机需要的运行时信息。
  • 8 字节对齐是硬性要求：对齐填充的唯一使命就是让对象总大小凑成 8 的整数倍，方便寻址。
  • 理解了这个布局，才能理解下一篇要讲的 Mark Word 为什么能"一块内存存多种状态"——本质上是虚拟机在有限的对象头空间里做的精打细算。
