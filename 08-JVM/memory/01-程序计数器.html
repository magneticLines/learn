内存区域与溢出

程序计数器

JVM 里最小、最不起眼，却是唯一一个"永远不会 OOM"的内存区域——它只做一件事：记住当前线程执行到哪一行字节码。

一句话讲清楚

程序计数器（Program Counter Register）可以看作是当前线程所执行的字节码的行号指示器。 字节码解释器工作时，就是通过改变这个计数器的值来选取下一条需要执行的字节码指令，分支、循环、跳转、异常处理、线程恢复等都依赖它。

类比：看书时夹在书页里的书签。你随时可能被打断（比如接电话），但书签帮你记住"我看到第几页第几行了"，回来继续看时不用从头找。CPU 在多个线程间切换执行，也需要这样一个"书签"。

为什么必须是"线程私有"的？

这要联系到 02-并发模块 讲过的线程调度：现代操作系统通过时间片轮转让多个线程"并发"共享 CPU，某个线程执行到一半会被挂起，CPU 切去执行另一个线程，之后再切回来继续执行。

如果所有线程共用一个程序计数器，线程 A 执行到字节码第 10 行时被挂起，线程 B 上来执行把计数器改成了第 3 行，等线程 A 恢复时就完全不知道自己刚才执行到哪了。 所以 JVM 规定每个线程都拥有一个独立的、互不干扰的程序计数器——这是它"天生线程安全"的原因，不需要加锁，因为根本不存在共享。

线程 A PC = 10 // A 执行到第 10 条字节码时被挂起
线程 B PC = 3 // CPU 切到 B，B 有自己独立的 PC，互不影响
线程 A PC = 10 // CPU 切回 A，A 从自己 PC 记录的位置 10 继续执行

这也正是线程封闭（Thread Confinement）思想最原始的体现：数据天然只属于一个线程，从设计上就杜绝了竞争，而不是靠锁去"事后补救"。

为什么它是唯一没有规定 OutOfMemoryError 的区域？

《Java 虚拟机规范》中，程序计数器是唯一一个没有规定任何 OutOfMemoryError 情况的内存区域。原因很简单：

  • 它占用的内存空间极小——本质上就是一个指针大小的寄存器（在概念模型里，物理机上可能直接用 CPU 寄存器实现）。
  • 它的大小是固定不变的，不会随着程序运行动态增长，所以不存在"内存不够用"的情况。

一个容易忽略的细节：执行 Native 方法时 PC 是什么值？

如果线程正在执行的是一个 Java 方法，PC 记录的是正在执行的字节码指令地址；如果执行的是 native 方法，PC 值为空（Undefined）——因为 native 方法交由本地方法栈执行，不涉及字节码。

面试高频坑：很多人会把"程序计数器不会 OOM"错误地泛化成"这个区域没有内存限制"。准确说法是：它不会因为内存不足抛 OOM，但这不代表它没有内存占用——只是占用小到规范认为不必单独规定异常情况。

核心价值总结

  • 作用：记录当前线程执行到哪一条字节码指令，是实现分支、循环、异常处理、线程恢复的基础。
  • 线程私有：每个线程独立拥有一份，天然线程安全，不需要同步机制，是"线程封闭"思想的典型体现。
  • 唯一不会 OOM 的区域：因为占用内存极小且固定，规范中不为它定义 OutOfMemoryError 场景。
  • 它虽然不起眼，但理解它是理解后面"虚拟机栈线程私有""方法调用如何恢复执行现场"的基础。
