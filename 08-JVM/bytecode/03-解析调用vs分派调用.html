解析调用 vs 分派调用

不是所有方法调用都需要"猜"——有些方法从编译那一刻起就已经注定了唯一的实现，有些方法则必须等到运行时才能知道到底调用的是谁。

一句话讲清楚

方法调用在字节码层面分两大类：解析调用（Resolution）——如果方法在编译期就能确定唯一的调用版本，且这个版本在运行期不可变， 虚拟机可以在类加载阶段就把符号引用直接解析为明确的调用入口；分派调用（Dispatch）——调用的目标方法要在运行期根据对象的实际类型才能确定， 这正是多态得以实现的基础，将在下一篇展开。

非虚方法：解析调用的资格证

能被"解析调用"的方法有一个共同点：它们不具备多态性，不可能在运行期被子类重写出另一个版本，因此虚拟机在类加载时就能一次性把符号引用锁定为直接引用。 这类方法统称为非虚方法（Non-Virtual Method）。

方法类别关键特征对应字节码指令为什么不可能被重写
静态方法 static invokestatic 属于类，不属于对象，没有"实际类型"的概念
私有方法 private invokespecial 对子类不可见，子类无法声明同名方法来"覆盖"它
实例构造器 <init> invokespecial 构造器本身不存在"重写"这个语义
final 方法 final invokevirtual（但目标唯一） 语法层面禁止子类重写，编译器会直接拒绝重写代码
父类方法调用 super.xxx() invokespecial 显式指定了调用父类版本，不参与多态查找
注意 final 方法的特殊之处：final 方法在字节码层面仍然使用 invokevirtual 指令（和普通虚方法一样）， 但因为语法上禁止被重写，虚拟机和 JIT 编译器知道它的目标版本是唯一的，因此可以把它当作非虚方法一样提前绑定甚至内联优化——这也是"非虚方法"这个概念比"invokestatic/invokespecial 调用的方法"范围略宽的原因。

原理走一遍：四条调用指令

invokestatic   调用静态方法 → 非虚方法 → 类加载阶段解析
invokespecial 调用私有方法 / 构造器 / super 方法 → 非虚方法 → 类加载阶段解析
invokevirtual 调用普通实例方法（含 final）→ 若 final 则等同非虚方法；否则是虚方法，需要动态分派
invokeinterface 调用接口方法 → 典型虚方法，运行期确定实现类
invokedynamic 调用动态生成的调用点（如 Lambda）→ 由调用点的引导方法在运行期决定目标

可以看到，invokestatic 和 invokespecial 是"解析调用"的专属指令——只要看到这两条指令，就能确定这次调用在类加载阶段已经被解析成了直接引用； invokevirtual 和 invokeinterface 则是"分派调用"的主战场，具体调用哪个方法版本，要交给下一篇的静态分派 / 动态分派机制来决定。

常见坑 & 面试高频考点

坑：把"解析"理解成"运行前会做的所有优化"。 这里的"解析"特指"符号引用 → 直接引用"这一步转换，只针对方法调用/字段访问等少数几类指令，不要和 JIT 编译、内联优化等其他运行期优化手段混为一谈。
面试高频："invokespecial 和 invokevirtual 的区别？" —— invokespecial 用于调用编译期就能确定唯一版本的方法（私有方法、构造器、super 调用），不经过运行时的多态查找； invokevirtual 用于调用可能被重写的普通实例方法，运行时要根据接收者的实际类型去查找方法。这也解释了"为什么私有方法和构造器不能被重写、也不能构成多态"——它们从字节码指令层面就已经被排除在动态分派之外。

核心价值总结

  • 方法调用先分两大类：能在类加载期锁定版本的走"解析调用"，需要运行期才能确定版本的走"分派调用"。
  • 静态方法、私有方法、构造器、final 方法这几类"非虚方法"是解析调用的全部对象——它们的共同点是不可能被重写。
  • 四条主要调用指令（invokestatic / invokespecial / invokevirtual / invokeinterface）天然映射着"是否需要分派"这条分界线。
  • 分派调用正是 Java 多态的字节码实现基础，下一篇正式展开静态分派（重载）与动态分派（重写）。
