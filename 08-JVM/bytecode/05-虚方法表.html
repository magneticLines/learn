虚方法表(进阶)

上一篇讲动态分派要"沿继承链逐层向上找方法"——如果每次调用都真的这么查一遍，性能会很难看。HotSpot 用一张提前算好的表，把这件事的开销降到了极致。

一句话讲清楚

为了避免动态分派时每次方法调用都要在类的方法元数据中重新搜索一遍（效率低），HotSpot 虚拟机为每个类维护了一张虚方法表（Virtual Method Table，vtable）， 表中按固定索引存放各个方法的实际入口地址。调用虚方法时，只需要根据对象的实际类型找到对应的方法表， 再按固定的索引位置直接取出方法地址即可，不需要再做一次沿继承链的名字查找。这是一种典型的用空间换时间的优化思路。

vtable 是怎么构建的：继承与覆盖

子类的 vtable 并不是从零重新生成的，而是在父类 vtable 的基础上扩展 / 覆盖得到的：

  • 如果子类没有重写父类的某个方法，子类 vtable 中对应位置的方法入口地址和父类完全一致——都指向父类的实现。
  • 如果子类重写了父类的某个方法，子类 vtable 中对应位置的地址会被替换成子类自己的实现入口。
  • 子类新增的方法，会在 vtable 末尾追加新的索引位。
Base.java / Derived.java
class Base {
    void foo() { /* Base 的实现 */ }
    void bar() { /* Base 的实现 */ }
}

class Derived extends Base {
    @Override
    void bar() { /* 重写了 bar，foo 没有重写 */ }
}
Base 的 vtable
  index 0: Base.equals()
  index 1: Base.hashCode()
  index 2: Base.foo()  ← Base 自己的实现
  index 3: Base.bar()

Derived 的 vtable（未重写 foo，重写了 bar）
  index 0: Base.equals()  （继承，未变）
  index 1: Base.hashCode()（继承，未变）
  index 2: Base.foo()  ← 没重写，仍指向 Base 的实现
  index 3: Derived.bar() ← 重写了，替换成子类自己的实现
关键规则：父类和子类中"同一个方法"在 vtable 里的索引位置永远相同。这是整套机制成立的前提——正因为索引位置在继承链上是稳定的， 虚拟机才能做到"不管对象的实际类型是 Base 还是 Derived，只要按同一个索引去查表，就能拿到该类型下 foo 正确的实现"。

原理走一遍：invokevirtual 结合 vtable 的调用过程

① 定位实际类型
  通过对象头找到对象的实际类型（方法区中的类元数据指针）

② 定位 vtable
  找到该实际类型对应的虚方法表

③ 按固定索引取方法地址
  编译期已经确定了方法名在 vtable 中的索引位置，直接按索引取出入口地址

④ 跳转执行
  无需再按名字/描述符做任何字符串匹配式的查找

对比朴素的"沿继承链逐层查找方法"，vtable 把原本 O(继承深度) 的查找开销压缩成了 O(1) 的数组索引访问—— 这正是"提前算好、重复利用"的空间换时间套路：把本该每次调用都要做的搜索工作，提前挪到类加载阶段一次性算好。

有 vtable vs 无 vtable（朴素动态分派）

维度朴素动态分派（无 vtable）HotSpot 实际做法（有 vtable）
每次调用的开销 沿继承链逐层比对方法名 + 描述符，最坏情况 O(继承深度) 按固定索引直接取地址，O(1)
额外空间成本 无 每个类多维护一张方法入口表
表的生成时机 不需要提前生成 类加载（准备/初始化前后）阶段生成一次

常见坑 & 面试高频考点

坑：把 vtable 当成"动态分派的定义"。 vtable 只是 HotSpot 对动态分派这件事做的实现优化，动态分派的语义（运行期按实际类型确定调用哪个方法）即使没有 vtable，用朴素的逐层查找也一样成立，只是效率差很多。面试时如果只回答"动态分派就是虚方法表"，是把手段和目的搞反了。
进阶延伸：接口方法调用的 itable。接口方法调用（invokeinterface）不能像 vtable 一样用固定单一索引，因为一个类可能同时实现多个互不相关的接口，方法在不同接口里的排列顺序无法统一编号。 HotSpot 为此设计了接口方法表（itable）：先按接口类型定位到对应的 itable，再在这张表内按方法索引取地址，原理与 vtable 类似但多了一层"先找接口"的过程，因此接口方法调用理论上比普通虚方法调用略慢，这也是早期 Java 性能调优文章常提到的一个细节点。

核心价值总结

  • vtable 是"用空间换时间"的经典范例：把本该每次调用都要做的方法查找，提前搬到类加载阶段做一次。
  • 子类 vtable 在父类基础上扩展 / 覆盖得到，同一方法在继承链上索引位置保持不变，是按索引查表能够生效的前提。
  • vtable 只是动态分派的一种高效实现手段，不是动态分派本身的定义。
  • 接口方法调用因为多继承的复杂性，需要额外的 itable 机制来配合，是 vtable 思路的进一步延伸。
  • 至此，从局部变量表/操作数栈、动态链接，到解析调用与分派调用、静态/动态分派、再到 vtable 优化，字节码执行引擎"方法怎么被调用"的完整链路就串起来了——下一篇转向 javac 编译期做了哪些"语法糖"，看看源码到字节码之间还发生了什么。

与其他知识点的联系

  • 静态分派与动态分派（上一篇）：vtable 不是一个孤立的新概念，而是上一篇动态分派"沿继承链逐层查找"这个朴素过程的性能优化版——没有 vtable，动态分派的语义依然成立，只是效率差很多，这一点在坑里也特意强调过。
  • 类加载的准备阶段：vtable 的构建时机是类加载过程中的一环，和"类加载五阶段"里的准备/初始化阶段前后衔接——理解 vtable 何时生成，能帮助建立"类加载不只是把字节码读进内存，还要为运行期性能做一些提前计算"这个整体认知。
  • 双亲委派与类型唯一性：vtable 按"实际类型"去查，前提是同一个类在虚拟机里只有唯一一份、唯一一张 vtable；如果双亲委派被打破导致同一个类被不同加载器加载出两份"不同类型"，会连带出现两份独立的 vtable，这也是类加载一致性问题会连锁影响到方法调用效率机制的一个例子。
  • Java 多态与面向对象设计：vtable 里"子类未重写的方法直接复用父类地址，重写的方法替换成子类地址"这条构建规则，正是 Java 重写语义在内存布局层面的具体实现——这也解释了为什么重写方法必须保持相同的方法签名，因为签名对应的是 vtable 里固定不变的索引位置。
  • JIT 内联优化：虚方法调用因为要走 vtable 查表，天然比非虚方法调用多一层间接跳转，不利于内联；HotSpot 的内联缓存（Inline Cache）、类型继承分析等 JIT 优化技术，本质上都是在尝试进一步消除这层 vtable 查找开销，是本篇内容在 JIT 层面的延伸。
