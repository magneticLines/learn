静态分派与动态分派(重载 vs 重写)

"重载和重写到底哪里不一样"是每个 Java 初学者都学过的规则，但从虚拟机层面看，两者其实分别对应两套完全不同的方法确定机制——一个在编译期拍板，一个在运行期才揭晓。

一句话讲清楚

重载（Overload）的实现基础是静态分派——依据的是变量的静态类型（编译期就确定的类型，即声明时写的类型）， 在编译阶段就把调用哪个重载版本写死进字节码。 重写（Override）的实现基础是动态分派——依据的是变量的实际类型（运行期才知道，比如父类引用指向的子类实例）， 必须等到程序真正跑起来才能确定调用的是谁。

静态类型 vs 实际类型
Human man = new Man();
// Human 是 man 的"静态类型"——写在代码里，编译期就固定，永远不变
// Man   是 man 的"实际类型"——运行期才知道，同一个变量可以在不同时刻指向不同实际类型

静态分派：编译器怎么给重载"消歧"

当一个方法名存在多个重载版本时，编译器要在编译期根据实参的静态类型，从候选版本中选出唯一一个写进字节码的调用指令里。 这个选择过程遵循一套优先级规则：

  1. 精确类型匹配——参数类型与实参类型完全一致，最优先。
  2. 基本类型自动转型（宽化转换）——如 char → int → long → float → double。
  3. 自动装箱 / 拆箱——如 int → Integer。
  4. 可变长参数（varargs）——优先级最低，只有前面都匹配不上才会考虑。

经典陷阱：重载消歧（面试高频难题）

下面这段代码是《深入理解 Java 虚拟机》里的经典例子的变体，故意制造"传 null / 传字面量"时的歧义，考察对上面优先级规则的理解。

OverloadTrap.java（反例 — 传字面量的消歧陷阱）
public class OverloadTrap {

    public static void f(Integer i) { System.out.println("Integer"); }
    public static void f(long l)    { System.out.println("long");    }

    public static void main(String[] args) {
        f(1);
        // 输出 "long"，不是 "Integer"！
        // 原因：int -> long 是"自动转型"，优先级高于 int -> Integer 的"自动装箱"
    }
}
OverloadTrap2.java（反例 — 传 null 的消歧陷阱）
public class OverloadTrap2 {

    public static void foo(String s)  { System.out.println("String");  }
    public static void foo(Object o)  { System.out.println("Object");  }

    public static void main(String[] args) {
        foo(null);
        // 输出 "String"：编译器在能匹配 null 的候选版本中，选参数类型"更精确"的那个
        // 如果同时存在 foo(String) 和 foo(StringBuilder)，两者之间没有父子关系可比较
        // 编译器无法判断谁更精确，直接报编译错误：reference to foo is ambiguous
    }
}
规则小结：编译器选择重载版本时，永远优先选择不需要发生类型转换或转换幅度最小、层级最精确的版本。 自动转型（宽化）比自动装箱优先级高，装箱又比变长参数优先级高。传 null 时，编译器会在能接受 null 的候选中选类型继承关系上最精确（最具体的子类型）的那个，如果两个候选类型之间不存在父子关系，就会产生编译期歧义错误。

动态分派：重写在虚拟机层面怎么做

重写方法调用使用 invokevirtual 指令，它在运行期执行时会按照固定的查找过程来确定真正调用哪个方法：

① 找接收者的实际类型
  取操作数栈顶的对象引用，找到它所指向对象的实际类型 C

② 在 C 中查找匹配的方法
  如果在 C 中找到与常量池描述符、简单名都相符的方法，且通过访问权限校验
  → 返回这个方法的直接引用，查找结束

③ 沿继承链向上找
  否则按继承关系从下往上，对 C 的各个父类依次重复第②步

④ 都没找到
  抛出 AbstractMethodError

关键在于第①步——接收者的实际类型只有运行到这行代码时才知道，所以同一条 invokevirtual 指令， 在程序不同的执行路径上（同一个父类引用指向不同子类实例）可能调用到完全不同的方法实现，这就是运行时多态的字节码本质。

静态分派 vs 动态分派

维度静态分派（重载）动态分派（重写）
依据 变量的静态类型 变量的实际类型
确定时机 编译期 运行期
字节码体现 编译器在编译期就选定具体重载版本，写入常量池方法符号引用（方法描述符不同） 同一条 invokevirtual 指令，运行时才沿继承链查找实际类型对应的方法
是否具有运行时多态性 没有——很多资料称其为"编译时多态"，但本质上只是编译期的一次选择 有——这是狭义上真正的"运行时多态"
面试高频："重载算不算多态？" —— 这是个有争议但有标准答案的问题：Java 语言规范意义上，重载依赖的是编译期静态类型，属于编译期行为，很多教材称之为"编译时多态"作为一种说法上的兼容，但更严谨的表述是——只有依赖动态分派的重写才是真正意义上的运行时多态；重载只是"同名方法在编译期被区分成了不同方法"。

核心价值总结

  • 重载看的是"你声明的类型"（静态类型），重写看的是"对象实际长什么样"（实际类型）。
  • 静态分派在编译期就把重载版本写死进字节码；动态分派把决定权留到运行期的 invokevirtual 查找过程。
  • 传 null、传字面量、int/long/Integer 混合重载是重载消歧的三大面试高频坑，核心规则是"优先选不需要类型转换、转换幅度最小的版本"。
  • 动态分派的查找过程（沿继承链逐层向上找）如果不做任何优化，每次调用都要重新走一遍——这正是下一篇"虚方法表"要解决的效率问题。

与其他知识点的联系

  • 解析调用 vs 分派调用（上一篇）：本篇的静态分派、动态分派都属于上一篇"分派调用"这个大类下的具体机制——静态分派对应可以直接写死目标的场景，动态分派才是真正需要运行期查找的那部分，两篇是"分类"和"分类下细节"的关系。
  • 虚方法表 vtable（下一篇）：本篇描述的动态分派查找过程（沿继承链逐层向上找同名同描述符的方法）如果每次调用都真的这么做，开销是 O(继承深度) 的；下一篇的 vtable 正是把这个过程提前算好、压缩成 O(1) 查表的具体实现优化。
  • Java 泛型与方法重载：静态分派依赖的是参数的静态类型和方法描述符，而泛型在字节码层面会被类型擦除——这也是为什么"两个仅泛型参数不同的重载方法"编译不通过：擦除后方法描述符完全一样，编译器在静态分派阶段就没法区分它们。
  • 字段访问不参与动态分派：容易被忽视的一点是，Java 里字段的访问永远只看静态类型，不存在"字段重写"的概念——只有方法调用才会走动态分派，这也是"隐藏（Hiding）"和"重写（Override）"这两个容易混淆的概念的根本区别。
  • 面向对象设计原则：动态分派是里氏替换原则（子类可替换父类）在字节码层面得以成立的技术基础——如果没有"运行期按实际类型查找方法"这套机制，父类引用调用子类重写方法的多态用法就无法实现。
