局部变量表与操作数栈

栈帧不是一个黑盒——它内部由两块具体的存储结构支撑方法执行：一块放变量，一块做计算。理解这两块结构，才算真正理解"字节码是怎么跑起来的"。

一句话讲清楚

局部变量表（Local Variable Table）是方法运行时私有的一块存储空间，存放方法参数和方法体内定义的局部变量。 操作数栈（Operand Stack）是一个后入先出（LIFO）的栈，字节码指令通过对它做入栈 / 出栈操作来完成实际的计算。 这两块结构都位于方法对应的栈帧（Stack Frame）之中，随着方法调用而创建，随方法结束而销毁。

类比：局部变量表像是工作台上摆放整齐、按编号取用的原料格子；操作数栈则是台面中央那块"正在加工"的空地——原料从格子里被取出放到台面上（入栈），加工完的半成品再放回格子（出栈存储）。所有计算都必须先把数据搬到这块空地上才能进行。

局部变量表：以 Slot 为单位

局部变量表以变量槽（Variable Slot）为最小单位。32 位以内的数据类型（int、float、reference、returnAddress 等）占用 1 个 slot； 64 位的 long 和 double 占用 2 个连续 slot。

实例方法的 slot 0 固定存放 this（对当前实例的隐式引用），之后依次是方法参数，再之后才是方法体内声明的局部变量；static 方法没有 this，参数直接从 slot 0 开始排。

原理走一遍：一个加法方法的字节码

Calc.java
public class Calc {
    public int calc(int a, int b) {
        int c = a + b;
        return c;
    }
}

用 javap -c 反编译后，局部变量表分配是 slot0=this，slot1=a，slot2=b，slot3=c；对应的字节码大致如下：

javap -c 反编译结果（简化）
0: iload_1        // 把 slot1(a) 的值压入操作数栈
1: iload_2        // 把 slot2(b) 的值压入操作数栈
2: iadd           // 弹出栈顶两个数，相加，结果压回栈顶
3: istore_3       // 弹出栈顶结果，存入 slot3(c)
4: iload_3        // 把 slot3(c) 的值压入操作数栈
5: ireturn        // 弹出栈顶值作为方法返回值

操作数栈的变化过程（栈底在左，栈顶在右）：

iload_1  → 操作数栈: [a]
iload_2  → 操作数栈: [a, b]
iadd     → 弹出 a、b，压入 a+b → 操作数栈: [a+b]
istore_3 → 弹出栈顶存入 slot3(c) → 操作数栈: []
iload_3  → 操作数栈: [c]
ireturn  → 弹出 c 作为返回值 → 操作数栈: []

这就是"字节码本质上是一套以栈为核心的指令集"最直观的体现——几乎所有计算指令都遵循"从操作数栈取数据、算完再压回去"的模式。

局部变量表 vs 操作数栈

维度局部变量表操作数栈
存取方式 按索引（slot 编号）随机存取，如 iload_1 只能操作栈顶，先入后出
用途 保存方法参数、局部变量的"当前值" 作为指令执行过程中的临时工作区
大小确定时机 编译期确定，写入 Code 属性的 max_locals 编译期确定，写入 Code 属性的 max_stack
是否包含 this 实例方法的 slot0 是 this 无此概念

常见坑 & 面试高频考点

坑 1：栈帧所需内存在编译期就已经固定。 max_locals 和 max_stack 是 javac 编译时静态计算好、写死在 Class 文件 Code 属性里的，虚拟机据此一次性分配栈帧空间，运行期不会因为"这次调用变量多一点"就动态扩容。这一点和 Java 对象在堆上运行时动态分配形成鲜明对比。

坑 2：long / double 的 slot 索引会"跳号"。 一旦方法里出现了 long 或 double 类型的参数/变量，后续变量的 slot 编号会跳过一位，手工分析字节码时很容易数错索引。

坑 3：局部变量没有默认值。 成员变量（字段）在对象创建时会被赋予默认值（如 int 默认 0），但局部变量表中的 slot 不会自动清零或赋默认值——如果代码路径上存在"声明了但没赋值就使用"的情况，会在编译期就被 javac 拒绝，这是字节码校验（Verifier）阶段保证运行安全的体现之一。

核心价值总结

  • 局部变量表负责"存"：按 slot 索引存放方法参数和局部变量，编译期确定大小。
  • 操作数栈负责"算"：一切运算都是"弹出操作数 → 计算 → 压回结果"的循环。
  • 字节码指令的本质就是围绕这两块结构做"取数、算数、存数"，理解这个模型就理解了字节码执行的基本节奏。
  • 栈帧里其实还有第三块重要结构——指向运行时常量池的方法引用，它是下一篇"动态链接"的主角。

与其他知识点的联系

  • 动态链接（下一篇）：栈帧里除了局部变量表和操作数栈，还有第三块结构——指向运行时常量池的方法引用，方法调用时正是从操作数栈弹出参数、再通过这个引用找到目标方法的字节码入口，两块结构在"调用一个方法"这件事上是配合工作的。
  • 解析调用 vs 分派调用：不管是解析调用还是分派调用，最终落地执行时都要依赖当前栈帧的局部变量表传参、操作数栈传结果——分派机制决定"调哪个方法"，局部变量表和操作数栈决定"这个方法怎么跑起来"，是上下游关系。
  • 字节码指令集：本篇里 iload_1/iadd/istore_3 这些指令正是"字节码指令集简介"一篇里说的"基于操作数栈架构"的具体落地——理解了操作数栈的入栈出栈规则，才能看懂几乎所有计算类指令的真实行为。
  • 字节码校验（Verifier）：局部变量表"没有默认值"这个特性依赖类加载阶段的验证过程来兜底——javac 会在编译期就拒绝"声明未赋值就使用"的代码，这也是字节码校验保证运行时安全的体现之一，和"验证阶段"这个类加载子阶段是同一套安全机制在不同层面的表现。
  • JIT 编译与内联优化：解释执行时严格按栈操作模式执行，但 JIT 编译热点方法时会把局部变量表和操作数栈上的数据尽量映射到物理寄存器，消除大量不必要的入栈出栈操作——这是从"字节码怎么写"到"机器码怎么跑得快"的关键一跳。
