字节码执行引擎

动态链接

字节码里的方法调用写的其实不是"地址"，而是一个"待翻译的名字"——栈帧里悄悄藏着一个引用，负责把这个名字翻译成真正能跳过去执行的入口地址。

一句话讲清楚

每个栈帧内部，除了上一篇讲的局部变量表和操作数栈，还持有一个指向运行时常量池中该栈帧所属方法的引用。 持有这个引用是为了支持方法调用过程中的动态链接（Dynamic Linking）——字节码里的方法调用指令一开始使用的是常量池中的符号引用， 需要在类加载的解析阶段或者运行期，把它转换为该方法在内存中的直接引用（实际入口地址）。

类比：符号引用就像通讯录里存的"张三"这个名字，直接引用则是"张三手机号 138xxxx"。字节码指令拿到的只是"名字"，真正打电话之前必须先查一次通讯录，把名字换成号码——这个"查号码"的动作就是链接。

承接前置概念：符号引用 vs 直接引用

这一对概念在类加载体系里第一次出现在"解析"阶段：Class 文件是纯文本描述式的，编译器不知道类被加载后各个方法、字段实际会落在内存的哪个位置， 所以只能先用符号引用（类的全限定名 + 方法名 + 方法描述符）占位，等到类加载或者第一次执行到该指令时， 虚拟机才把符号引用替换为可以直接定位到目标的直接引用（指针、偏移量或者句柄）。

A.java
class A {
    void foo() {
        bar();   // 编译期只知道要调用"bar"，还不知道它的内存地址
    }
    void bar() {}
}

编译 foo() 时，javac 生成的字节码调用指令（如 invokevirtual）携带的操作数并不是地址，而是一个常量池索引， 指向一条 CONSTANT_Methodref_info 记录——里面写的是"类名 + 方法名 + 描述符"这样的符号信息。

原理走一遍：从符号引用到直接引用

① 调用指令 invokevirtual #2
  操作数 #2 是运行时常量池的一个索引

② 符号引用（CONSTANT_Methodref_info）
  记录着 "A" + "bar" + "()V"（类名 + 方法名 + 方法描述符）
  此时还不知道 bar() 实际在内存哪里

③ 解析 / 链接
  虚拟机在类加载解析阶段或运行期第一次执行到这条指令时
  去查找 A（或其实际类型）的方法元数据，定位到 bar() 的真实入口

④ 直接引用
  得到方法在内存中的入口地址 / 相对偏移量
  → 后续调用可以直接跳转执行，不必重新查找

这里"什么时候转换"是关键区分点：如果目标方法在编译期就唯一确定（比如静态方法、私有方法、构造器、final 方法）， 虚拟机可以在类加载阶段就把符号引用永久替换成直接引用，这称为静态解析； 如果目标方法要在运行期根据实际类型才能确定（比如被重写的虚方法），转换就必须推迟到每次调用时才做，这才是狭义上的"动态链接"。 下一篇会专门展开这两条路径的区别。

常见坑 & 面试高频考点

坑：把"动态链接"等同于"每次调用都要重新查找"。 实际上 HotSpot 对方法调用做了缓存优化——常量池的 CONSTANT_Methodref_info 一旦解析成功，会把结果缓存起来（例如非虚方法直接内联缓存直接引用）， 并不会在每一次执行到调用指令时都重新走一遍完整的查找流程。"动态"强调的是"确定时机在运行期"，而不是"每次都要重新算一遍"。
面试高频："符号引用和直接引用的区别？什么时候发生转换？" —— 记住关键判据："这个方法在编译期是否唯一确定、运行期是否可能变化"。能唯一确定的（非虚方法）类加载期就能转直接引用；不能唯一确定的（虚方法）必须留到运行期，这也正是下一篇"解析调用 vs 分派调用"要划分的两条路线。

核心价值总结

  • 栈帧持有指向运行时常量池的方法引用，是方法调用能够"落地执行"的桥梁。
  • 字节码里的调用指令天生使用符号引用，必须经过解析才能变成可执行的直接引用。
  • 转换时机是区分"解析调用"和"分派调用"的核心分水岭——这正是下一篇要讲的内容。
