内存泄漏排查思路

Java 有垃圾回收器，为什么还会"内存泄漏"？因为泄漏的从来不是内存本身，而是本该被回收的对象，被一条你没注意到的强引用链拽住了。

一句话讲清楚

C++ 里的内存泄漏是"忘了 free"；Java 里的内存泄漏是"该被回收的对象，因为存在一条从 GC Roots 出发、意外没有断开的强引用链，导致 GC 判断它'可达'，永远不会被回收"。 这就要回到前面 GC 模块讲过的可达性分析：只要一个对象能从 GC Roots 出发被追踪到，它就是"存活"的，哪怕业务逻辑上早就用不到它了。

回顾四种引用类型：强引用（Object o = new Object()）——只要引用还在，GC 绝不回收，这也是内存泄漏的根源；软引用——内存不足时才回收，适合做缓存；弱引用——下一次 GC 必被回收，WeakHashMap/ThreadLocal 内部的 Entry 就用它；虚引用——只用来在对象被回收时收到通知。内存泄漏的本质，几乎都是"该用弱引用或该主动清理的地方，用了强引用又没清理"。

常见的四类"意外强引用"

场景 1：静态集合无限增长

反例 — static 缓存永不清理
public class UserCache {
    // static 字段生命周期和类一样长，几乎等同于 GC Root 直接持有
    private static final Map<Long, User> cache = new HashMap<>();

    public static void put(Long id, User u) {
        cache.put(id, u);  // 只 put，从不 remove，越攒越多
    }
}

正例应改用带淘汰策略的缓存（如 Caffeine 的 maximumSize + LRU），或者用 WeakHashMap/软引用让 GC 有机会介入回收。

场景 2：监听器 / 回调忘记注销

观察者模式里，被观察者持有一份监听器列表；如果监听器注册后从不注销，被观察者的生命周期比监听器所属对象更长时，监听器（连同它引用的外部对象）就会被永久拽住，无法回收。

场景 3：ThreadLocal 未清理

反例 — 线程池场景下 ThreadLocal 未 remove
private static final ThreadLocal<BigContext> CTX = new ThreadLocal<>();

void handleRequest() {
    CTX.set(new BigContext(...));
    // ... 业务逻辑
    // 忘了 CTX.remove()！线程池里线程会被复用，Entry 永远挂在 Thread 上
}

ThreadLocalMap 里的 Entry 的 key 是弱引用（指向 ThreadLocal 本身），但value 是强引用——线程池里线程是被复用的，如果不主动 remove()，key 被回收后 value 依然强引用挂在 Thread 对象上，永远不会释放，这是 ThreadLocal 内存泄漏最经典的成因。

场景 4：资源未关闭 / 内部类隐式持有外部引用

非静态内部类、匿名内部类默认持有外层对象的引用（编译器自动生成的 this$0 字段）——如果这个内部类实例的生命周期比外层对象更长（比如作为 Handler 被扔进消息队列），外层对象就被间接强引用住，无法回收。

典型排查流程

① 观察现象：监控显示老年代 持续增长，且 Full GC 后回收不下来
  （用 jstat -gc 观察 OU 字段，或 GC 日志里 Full GC 前后堆大小几乎不变）

② 导出堆转储：jmap -dump:format=b,file=heap.hprof pid
  （尽量在问题复现、内存已经明显偏高时导出，样本才有意义）

③ 用 MAT 打开 hprof 文件
  运行 Leak Suspects 报告，MAT 会自动分析并给出"可疑泄漏点"
  查看 Dominator Tree（支配树）：按 Retained Size 排序，找到占用内存最大的对象

④ 追溯 GC Roots 引用链：右键目标对象 → Merge Shortest Paths to GC Roots
  （排除 weak/soft 引用路径，只看强引用链）
  → 找到那条"意外的强引用"，比如某个 static Map、某个未清理的 ThreadLocal

⑤ 定位代码，修复：加淘汰策略 / 注销监听器 / remove ThreadLocal / 用弱引用

MAT 里的几个关键概念

概念含义
Shallow Size 对象自身占用的内存大小，不包含它引用的其他对象（比如一个空的 ArrayList 对象本身很小）
Retained Size 如果这个对象被回收，能连带释放的总内存（自身 + 只能通过它才能访问到的所有对象）——排查泄漏时更应该看这个字段
Dominator Tree（支配树） 把对象引用图重新组织成"谁支配谁"的树形结构：如果到达对象 B 的所有路径都必须经过 A，那么 A "支配" B——按 Retained Size 排序这棵树，最容易找到"大对象"
GC Roots 可达性分析的起点：虚拟机栈的本地变量、方法区的静态引用、JNI 引用等——引用链能追溯到 GC Roots 说明对象存活，追不到说明该被回收
Leak Suspects Report MAT 内置的自动化分析报告，基于经验规则（比如某个对象的 Retained Size 占堆总大小很大比例）自动给出可疑泄漏点，通常是排查的第一步
坑：不要看到内存高就立刻断定是"泄漏"。如果只是堆参数（-Xmx）设置偏小、业务量本来就需要更多内存，属于容量不足而非泄漏；区分标准是——多次 Full GC 之后老年代占用是否能回落到一个稳定的基线。如果每次都回落，只是正常的内存使用高峰；如果一次比一次高、永远回不去，才是真正的泄漏。

核心价值总结

  • Java 的"内存泄漏"本质是强引用链意外未断开，不是忘了释放内存，这和 C++ 的内存泄漏是两码事。
  • 四种引用类型是理解和预防内存泄漏的理论基础：该用弱引用/软引用的地方用了强引用，就是泄漏的温床。
  • 排查主线是：观察趋势（jstat）→ 拍快照（jmap）→ 分析支配树找大对象（MAT）→ 追 GC Roots 引用链 → 定位代码。
  • 静态集合、监听器、ThreadLocal、内部类隐式引用是四个最高频的"背锅"场景，写代码时看到这几种模式要多留意生命周期问题。

与其他知识点的联系

  • 垃圾回收算法与可达性分析（GC 模块）：内存泄漏这个概念本身就建立在可达性分析之上——一个对象"泄漏"的定义就是"从 GC Roots 出发仍然可达，但业务上早已不再需要"，如果不理解可达性分析怎么判定对象存活，就无法理解为什么 Java 里"有 GC 也会泄漏"这个看似矛盾的现象。
  • JVM 运行时内存结构：本文里几乎所有反例都对应着具体的内存区域——static 缓存和 ThreadLocal 的 Entry 挂在堆上的老年代，动态代理滥用堆积的类元数据挂在元空间，理解堆/方法区/元空间各自存什么、由谁触发回收，才能看懂 jstat 里 OU、MC 这些字段为什么能提示泄漏的位置。
  • jps / jstat / jmap / jstack 速查：排查流程里的"观察趋势"和"拍快照"两步直接复用这组命令行工具——用 jstat -gc 的 OU 字段观察老年代是否持续走高、Full GC 后能否回落，一旦确认有泄漏嫌疑，就用 jmap -dump 拍下堆快照交给 MAT 分析，这两个工具是本文排查思路能落地的具体抓手。
  • JConsole 与 VisualVM：如果不想单独装 MAT，VisualVM 自带的堆转储分析器同样能做类似的事——按类统计实例数量和大小、查看对象的 GC Roots 引用链，对轻量场景够用，复杂的大堆分析仍推荐专业的 MAT。
  • Arthas 实战：Arthas 的 heapdump/memory 命令可以在生产环境上以更轻量、更少干扰的方式辅助定位内存问题，作为 jmap 之外的另一种手段。
  • ThreadLocal 与线程状态（并发模块）：场景 3 的 ThreadLocal 未清理是内存泄漏和并发知识交叉的经典案例——线程池复用线程导致 Entry 常驻不释放，这既要理解 ThreadLocalMap 里 key 是弱引用、value 是强引用的设计，也要理解线程池"线程不会随任务结束而销毁"这个并发模块讲过的特性，两者叠加才会踩中这个坑。
