jps / jstat / jmap / jstack 速查

排查问题前，先要"看得见"——这四个 JDK 自带命令行工具是最基础的观测手段，不装任何软件、随处可用。

一句话讲清楚

这四个工具分别回答四个问题：机器上跑着哪些 Java 进程（jps）、某个进程的运行时数据长什么样（jstat）、 某个进程的堆里都装了什么（jmap）、某个进程的线程都卡在哪（jstack）。 它们都在 JDK 的 bin 目录下，是 JDK Development Tools（JDT）的一部分，不需要额外安装，任何一台装了 JDK 的机器都能用——这也是为什么它们是生产排障的"第一梯队"工具。

它们是怎么拿到这些信息的？JVM 启动时会在本地创建一块共享内存映射文件（Linux 下在 /tmp/hsperfdata_<user>/<pid>，Windows 下是匿名共享内存），里面持续写入进程基本信息和运行时计数器（PerfData）。jps/jstat 本质上就是在读这块共享内存，完全不需要 attach 进 JVM 内部，所以速度快、开销极低；而 jmap/jstack 需要通过 Attach API 真正连接到目标 JVM 内部，让它执行一次"自省"动作（导出堆或线程栈），开销相对更大。

jps：先找到 pid

jps（JVM Process Status）是这一组工具的入口——后面三个工具都需要指定 pid，而 jps 就是用来找 pid 的，效果类似 ps -ef | grep java，但它专门认 Java 进程，还能直接显示主类名。

终端 — jps -l（显示完整主类名/jar 路径）
$ jps -l
12580 com.supernovacompanies.ccc.CccApplication
8842 org.apache.catalina.startup.Bootstrap
15021 sun.tools.jps.Jps

常用参数：-l 输出主类全限定名或 jar 包路径；-v 输出启动时的 JVM 参数（比如 -Xmx 配了多少，排查内存问题第一步就该看这个）；-m 输出传递给 main 方法的参数。

jstat：实时看运行时数据

jstat（JVM Statistics Monitoring）用来周期性采样 JVM 内部的运行时数据，最常用的场景是观察 GC 情况——这直接对应前面 GC 模块讲过的分代结构：新生代 Eden/Survivor、老年代、方法区（元空间）。

终端 — jstat -gc pid 1000（每 1 秒采样一次）
$ jstat -gc 12580 1000
 S0C    S1C    S0U    S1U      EC       EU        OC         OU       MC     MU    YGC   YGCT   FGC   FGCT    GCT
10240.0 10240.0  0.0   9216.0  81920.0  40960.0  204800.0   52480.5  21504.0 20812.3   42    0.312    2     0.198    0.510

关键字段解读（单位 KB）：

  • S0C / S1C / S0U / S1U：两个 Survivor 区的容量（Capacity）和已用（Used）——正常情况下同一时刻只有一个 Survivor 在用，另一个应接近 0。
  • EC / EU：Eden 区容量/已用——涨得最快的区域，涨满就触发 Minor GC。
  • OC / OU：老年代容量/已用——持续走高且 Full GC 后降不下来，就是内存泄漏的信号（详见本组第 4 篇）。
  • MC / MU：元空间（Metaspace）容量/已用——对应类元数据，动态生成大量类（如 CGLIB 代理滥用）会把它撑爆。
  • YGC / YGCT：Young GC 次数 / 累计耗时（秒）；FGC / FGCT：Full GC 次数 / 累计耗时；GCT：GC 总耗时。
如果发现 FGC 数字在短时间内飞涨，或者 FGCT 单次耗时很长（说明用的还是 CMS/Parallel 这类会长时间 STW 的收集器），这就是需要立刻介入排查的信号——先用 jstat 定位"是不是 GC 的锅"，再决定用 jmap 还是 jstack 深入。

jmap：拍一张堆快照

jmap（JVM Memory Map）用来导出堆转储快照（Heap Dump），生成的 .hprof 文件可以用 MAT、VisualVM 等工具离线分析——jstat 只能看到"各区用了多少"，而 jmap 能看到"堆里具体都装了哪些对象、有多少个、多大"。

终端 — 导出堆转储 / 查看对象直方图
# 导出完整堆快照到文件，供 MAT/VisualVM 离线分析
$ jmap -dump:format=b,file=heap.hprof 12580
Dumping heap to heap.hprof ...
Heap dump file created

# 不落盘，直接在命令行看存活对象直方图（按占用空间倒序）
$ jmap -histo:live 12580 | head -5
 num     #instances         #bytes  class name
----------------------------------------------
   1:       128340       20534400  [B                  # byte[]，常是缓存/序列化数据堆积
   2:        45210        7233600  com.alibaba.fastjson.JSONObject
   3:        98123        3924920  java.lang.String
坑：jmap -dump（不带 :live）会导出堆上所有对象，包括本该被回收但还没被 GC 清理的垃圾；加上 :live（或 -histo:live）会先触发一次 Full GC 再统计存活对象——这意味着在生产环境上执行会造成一次明显的 STW 停顿，堆越大停顿越久（几十 GB 的堆可能停顿数十秒），务必避开业务高峰期，能用 Arthas 的非侵入式方式排查就优先用它（见本组第 3 篇）。

jstack：拍一张线程快照

jstack 导出目标进程当前所有线程的调用栈（Thread Dump），是排查死锁、线程阻塞、CPU 飙高的核心工具——它能告诉你"每个线程这一刻正卡在代码的哪一行"。

终端 — jstack 输出片段
$ jstack 12580

"http-nio-8080-exec-3" #45 daemon prio=5 os_prio=0 tid=0x... nid=0x1a2b waiting on condition [0x...]
   java.lang.Thread.State: WAITING (parking)
	at sun.misc.Unsafe.park(Native Method)
	at java.util.concurrent.locks.LockSupport.park(LockSupport.java:175)
	at ...OrderService.lambda$createOrder$0(OrderService.java:42)

Found one Java-level deadlock:
=============================
"Thread-1":
  waiting to lock monitor 0x... (object 0x...), which is held by "Thread-2"
"Thread-2":
  waiting to lock monitor 0x... (object 0x...), which is held by "Thread-1"

jstack 会自动检测经典的 Java 级死锁并直接打印 Found one Java-level deadlock，连等待链都列出来了，不需要人工比对——这是它相比人工翻代码效率高得多的地方。

遇到什么问题，用什么工具

现象 / 问题该用的工具为什么
不知道目标 Java 进程的 pid jps -l 所有其他工具的前置步骤
怀疑 GC 太频繁 / 停顿太长 jstat -gc pid interval 轻量、可持续采样，先定性是不是 GC 问题
内存持续增长，想看是哪些对象在占用 jmap -histo:live / jmap -dump 拿到对象级别的快照，配合 MAT 分析引用链
接口卡死 / 怀疑死锁 / CPU 飙高定位代码行 jstack 看到每个线程当前的调用栈和状态
面试高频："jmap/jstack 是怎么在不停止 JVM 的情况下拿到内部信息的？"——答案是 Attach 机制：这些工具通过 JDK 提供的 Attach API 向目标 JVM 发送一个信号（Linux 下是 SIGQUIT 类似机制），目标 JVM 收到后由内部的 Attach Listener 线程响应，执行相应操作（如导出线程栈），再把结果写到指定位置。这也是为什么工具版本和目标 JVM 版本差异过大时可能 attach 失败。

核心价值总结

  • jps 是入口：先拿到 pid，才能用后面三个工具。
  • jstat 走的是"读共享内存"的轻量路径，适合高频、持续观察 GC 趋势，几乎零开销。
  • jmap 走的是"attach + 自省"的重路径，能拿到对象级别的堆快照，但会有 STW 代价，生产环境要谨慎。
  • jstack 同样走 attach 路径，是排查死锁/阻塞/CPU 飙高最直接的第一手资料。
  • 它们是"单次快照"型工具，想要持续可视化观察，就需要下一篇的 JConsole / VisualVM。
