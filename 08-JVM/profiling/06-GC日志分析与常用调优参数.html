GC 日志分析与常用调优参数

调优不是背参数表——是先从日志里看清楚瓶颈在哪，再有针对性地调整，前面几篇讲过的内存区域和收集器知识，都是为这一步做准备。

一句话讲清楚

GC 日志是 JVM 留下的"体检报告"，记录了每一次垃圾回收发生的时间、类型、回收前后各内存区域的大小、耗时。调优的正确顺序永远是"先看日志定位瓶颈，再针对性调参数"，而不是照抄一份参数表就上生产——同一套参数在不同的堆大小、不同的对象分配速率下，效果可能完全相反。

开启 GC 日志

JDK 9 之前用 -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log 这类零散参数；JDK 9 引入了 统一日志框架（JEP 158 / JEP 271），所有日志（不只是 GC）都通过统一的 -Xlog 语法开启，格式更规范、可控性更强。

终端 — 开启 GC 日志（JDK 9+ 统一日志格式）
$ java -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=100M \
     -jar app.jar

# 语法拆解：
#   gc*           —— 输出 gc 相关的所有子标签（gc、gc+heap、gc+age 等），排查问题时信息越全越好
#   file=gc.log   —— 输出到文件而不是控制台
#   time,uptime,level,tags —— 每条日志附带的修饰信息：绝对时间、JVM 运行时长、日志级别、标签
#   filecount=5,filesize=100M —— 滚动日志，最多 5 个文件，每个 100M，避免日志把磁盘写满

一段典型日志逐字段解读

gc.log 片段（G1 收集器，JDK 9+ 格式）
[12.345s][info][gc,start] GC(42) Pause Young (Normal) (G1 Evacuation Pause)
[12.345s][info][gc,heap] GC(42) Eden regions: 40->0(42)
[12.345s][info][gc,heap] GC(42) Survivor regions: 4->5(6)
[12.345s][info][gc,heap] GC(42) Old regions: 18->19(18)
[12.358s][info][gc      ] GC(42) Pause Young (Normal) (G1 Evacuation Pause) 168M->142M(256M) 13.421ms

关键字段：

  • [12.345s]：距 JVM 启动的运行时长（uptime），用来判断问题是启动初期（可能是预热/编译问题）还是运行一段时间后才出现（更像是泄漏或负载问题）。
  • GC(42)：第 42 次 GC 事件的编号，同一次 GC 的多条日志行共享这个编号，方便把 Eden/Survivor/Old 的变化关联起来看。
  • Pause Young (Normal) (G1 Evacuation Pause)：本次 GC 的类型——一次年轻代的疏散暂停，对应前面 GC 模块讲过的 Minor GC；G1 特有的说法是 "Evacuation Pause"（把存活对象从一个 Region 疏散到另一个 Region）。
  • Eden regions: 40->0(42)：Eden 区从占用 40 个 Region 回收到 0 个（说明年轻代对象大部分是垃圾，符合"朝生夕死"的假设），括号里的 42 是 Eden 区当前的总容量。
  • 168M->142M(256M)：整个堆从回收前 168M 降到回收后 142M，堆总容量 256M——这一行是判断"回收效果好不好"最直接的数据：如果回收后几乎没降，说明大部分对象都存活下来了，可能意味着晋升速度过快或者存在泄漏。
  • 13.421ms：本次 GC 的停顿耗时——这是用户能感知到的"卡顿时长"，也是调优最终要优化的核心指标。
这些字段和前面内存模块讲的堆结构、GC 模块讲的分代收集理论是完全对应的：Eden/Survivor/Old regions 就是内存区域，Pause Young 就是 Minor GC，日志分析本质上是在"用数据验证内存模型和收集器理论在这台机器上具体是怎么运作的"。

常用调优参数速查表

参数作用调优要点
-Xms / -Xmx 堆的初始大小 / 最大大小 生产环境建议两者设为相同值，避免堆动态扩容/收缩带来的额外停顿和抖动
-Xmn 新生代大小（Eden + 2×Survivor） 新生代太小会导致对象过早晋升到老年代，加重 Full GC 压力；太大则 Minor GC 单次耗时变长
-XX:SurvivorRatio Eden 区和单个 Survivor 区的比例 默认 8（即 Eden:S0:S1 = 8:1:1），对象存活率高的场景可适当调小，避免 Survivor 装不下触发担保
-XX:MaxTenuringThreshold 对象在 Survivor 区熬过多少次 Minor GC 才晋升老年代 配合 -XX:+PrintTenuringDistribution 观察实际的年龄分布再调整，不要拍脑袋设值
-XX:+UseG1GC 指定使用 G1 收集器 JDK 9+ 默认收集器；大堆、追求可控停顿时间的场景首选
-XX:MaxGCPauseMillis G1 的目标最大停顿时间（软目标，不是硬保证） 设得越小，G1 会越倾向于每次少回收一点、更频繁地回收，需要在停顿时长和吞吐之间权衡
-XX:+HeapDumpOnOutOfMemoryError 发生 OOM 时自动生成堆转储文件 生产环境强烈建议默认开启，配合 -XX:HeapDumpPath 指定路径，OOM 现场往往稍纵即逝，没有这个参数出问题后就什么都拿不到了
坑（也是面试高频）：不要迷信"网上抄来的一套参数"。比如把 -Xmn 设置得过大，看似减少了对象晋升、降低 Full GC 频率，但会导致每次 Minor GC 要扫描的存活对象变多，单次 Minor GC 停顿反而变长——调优永远是多个指标之间的权衡（吞吐量 vs 停顿时间 vs 内存占用），没有放之四海而皆准的参数组合，必须结合自己应用的对象分配速率、存活周期，通过日志反复验证。

调优方法论：先分析，再调整

① 明确目标：是要降低单次停顿时间（延迟敏感型服务），还是提升吞吐量（离线批处理）？
  目标不同，调优方向可能完全相反

② 开启 GC 日志，观察一段时间：Minor GC 频率、单次耗时；Full GC 是否频繁、是否耗时过长

③ 定位瓶颈是哪个区域：新生代太小导致早晋升？老年代碎片导致 Full GC 变慢？
  元空间不够导致频繁触发类卸载扫描？

④ 针对性调整一个参数：只改一个变量，改完重新观察日志对比效果
  （一次改多个参数，出了问题根本不知道是哪个参数导致的）

⑤ 压测验证，确认没有引入新问题后再上生产

这也是为什么本模块把内存区域、垃圾收集器原理放在前面讲：不理解 Eden/Survivor/老年代怎么协作，不理解 G1 的 Region 和停顿模型，看到 GC 日志也只是一堆数字，无从下手调整。日志分析和调优参数，是把前面的理论知识落地成实际生产力的最后一步。

核心价值总结

  • GC 日志是判断调优是否有效的唯一客观依据，任何调优动作前后都要有日志数据支撑，不能凭感觉。
  • JDK 9+ 的统一日志框架用 -Xlog:gc* 一个入口就能控制所有 GC 相关日志的输出内容和格式，比老版本零散的 -XX:+PrintGCDetails 系列参数更规范。
  • 调优参数速查表只是工具箱，真正的功夫在于"先分析瓶颈在哪，再决定用哪个参数、往哪个方向调"这套方法论。
