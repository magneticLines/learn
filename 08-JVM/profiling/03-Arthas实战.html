性能监控与调优

Arthas 实战

生产环境不能停机、不能重启，但线上问题必须马上定位——Arthas 是国内 Java 团队排查线上问题的事实标准工具。

一句话讲清楚

Arthas 是阿里巴巴开源的 Java 诊断工具，核心能力是在不重启进程、不修改一行源码的情况下，实时观测一个正在运行的方法的入参、返回值、异常，甚至动态替换它的字节码。 它解决的核心痛点是：传统排查方式需要"加日志 → 重新打包 → 重新部署 → 等待复现"，一套流程走下来可能几十分钟甚至几小时，而且很多线上问题（比如依赖特定的并发时序、特定用户的脏数据）根本无法在测试环境稳定复现。

反例（传统方式）：线上某个接口偶尔返回错误的订单金额，你的第一反应可能是在 calcAmount() 里加一行 log.info(...)，然后走完整套 CI/CD 流程重新发布——等日志真正打出来，可能已经过了 20 分钟，而且如果问题是间歇性的，你还不一定能等到它再次复现。

它是怎么做到"不重启不改代码"的？

Arthas 依赖 JDK 提供的 Instrumentation API（java.lang.instrument 包，Attach 机制的延伸）：它以 Agent 的形式动态 attach 到目标 JVM 进程，然后通过 retransformClasses() 对指定的类做字节码增强——在目标方法的入口和出口"织入"一段监控逻辑（比如记录参数、返回值），执行完再把结果输出到你的 Arthas 控制台，全程目标类的原始逻辑一行没变。

① Attach：Arthas 通过 Attach API 挂载到目标 JVM，加载自己的 Agent
  → 与 jmap/jstack 用的是同一套 Attach 机制，只是 Arthas 常驻在里面持续交互

② 字节码增强（retransform）：对目标类做 ASM 字节码改写
  在方法入口插入"记录参数"逻辑，出口插入"记录返回值/异常"逻辑
  → 原方法的业务逻辑本身完全不动，只是被"包了一层"

③ 数据回传：增强后的代码执行时把捕获的数据通过 Arthas 内建的 Web/Telnet 通道
  实时推送到你的终端

④ 命令结束后自动"卸妆"：命令退出时 Arthas 会把类还原回原始字节码
  → 不会给正常运行留下永久性的性能负担

实战：用 watch 排查一个返回异常值的方法

假设线上偶尔出现订单金额计算错误，先启动 Arthas 并 attach 到目标进程：

终端 — 启动并 attach
# 下载并启动 arthas-boot.jar，会自动列出本机 Java 进程供选择（原理同 jps）
$ java -jar arthas-boot.jar
* [1]: 12580  com.supernovacompanies.ccc.CccApplication
请输入序号选择要监控的进程: 1
Attach 成功
Arthas 控制台 — watch 观测方法入参/返回值
$ watch com.supernovacompanies.ccc.service.OrderService calcAmount "{params, returnObj, throwExp}" -x 2

method=com.supernovacompanies.ccc.service.OrderService.calcAmount location=AtExit
ts=2026-07-02 14:30:12; [cost=1.2ms] result=@ArrayList[
    @Object[][
        @OrderDTO[itemId='item-001', discountRate=-0.2],   // 发现！折扣率是负数
    ],
    @BigDecimal[-19.80],                             // 返回了负金额，这就是 bug 根源
    null,
]

watch 命令的三个核心参数：类名 + 方法名 定位目标方法；表达式（params/returnObj/throwExp）决定观测入参、返回值还是异常；-x 控制对象展开的深度。 整个过程没有重启进程，没有改一行代码——两分钟就定位到问题是上游传入了负的折扣率，比"加日志重新发布"快了一个数量级。

常用命令速查

命令作用典型场景
dashboard 类似 top，实时展示线程/内存/GC 概览 刚 attach 上去，先看一眼整体健康状况
watch 观测方法的入参、返回值、异常 方法返回值不符合预期，想知道具体是什么数据触发的
trace 追踪方法内部调用链路，展示每一步耗时 接口响应慢，想知道具体卡在调用链的哪一步（哪次 RPC/SQL）
stack 打印方法被调用时的调用栈 方法被意外调用，想知道是谁在什么路径下调的它
sc / sm 查看已加载的类信息 / 类中的方法信息 确认线上运行的到底是哪个版本的类（多版本 jar 冲突排查）
classloader 查看类加载器树、某个类是被哪个 ClassLoader 加载的 排查 ClassNotFoundException / 类加载器隔离导致的诡异问题
jad 反编译已加载的类，还原成 Java 源码 线上代码和本地代码疑似不一致，直接反编译对比
redefine 用新的字节码热替换某个类，不重启生效 紧急修复一个已定位的小 bug，来不及走完整发布流程（谨慎使用）
坑：trace/watch 本质上是在目标方法上插桩，如果作用范围太大（比如对一个高频调用的方法加 trace，且没有加表达式过滤条件），会显著增加目标方法的开销，甚至拖慢整个服务——生产环境务必加过滤条件（如 watch ... 'params[0].id==123'），排查完立刻用 stop 结束命令，避免观测逻辑常驻。
redefine 只能替换方法体，不能新增/删除字段和方法签名（JVM 字节码热替换的固有限制），也不能跨越已经在执行栈上的旧代码——它更适合"临时打个补丁应急"，而不是常规发布手段，正式修复仍要走代码库和正常发布流程。

核心价值总结

  • Arthas 用字节码增强解决了"生产环境不能停机改代码，但必须实时诊断"这个根本矛盾。
  • 它和 jmap/jstack 一样基于 Attach 机制，但走得更远——不只是拍一次快照，而是持续动态地插入观测逻辑。
  • watch/trace 是日常排查用得最多的两个命令，分别对应"看数据对不对"和"看时间花在哪"。
  • 正因为效率碾压传统"加日志重新部署"的方式，Arthas 已经成为国内 Java 团队线上问题排查的事实标准工具，几乎是必备技能。
