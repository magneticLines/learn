CPU 飙高排查思路

"top 找进程 → top -Hp 找线程 → 转十六进制 → jstack 对号入座"，四步串联，是 Java 圈里排查 CPU 飙高最通用的套路。

一句话讲清楚

CPU 飙高的本质是某个（或某几个）线程在疯狂执行代码而不是在等待（等锁、等 IO 都不占 CPU）。排查思路就是层层缩小范围：先从全部进程里找到 CPU 高的那个 Java 进程，再从进程内所有线程里找到 CPU 高的那几个线程，最后用 jstack 把这个线程 ID 对应到具体在执行哪一行代码。

这条排查链之所以能走通，关键在于 Linux 下操作系统的线程和 JVM 的线程是一一对应的（Java 线程模型基于原生线程实现，一个 Java 线程对应一个内核级 LWP（轻量级进程）——所以 top -H 看到的线程 tid，和 jstack 输出里的 nid（native id）指向的是同一个操作系统线程，两者才能对上号。

完整排查流程

终端 — 完整命令串联
# ① top 找到 CPU 占用最高的 Java 进程 pid
$ top
  PID  USER  %CPU  %MEM  COMMAND
12580 app   385.0  12.3  java
#                    ↑ 注意：多核机器上超过 100% 是正常的（385% 约等于占满近 4 个核）

# ② top -Hp pid：在这个进程内部，按线程维度看 CPU 占用
$ top -Hp 12580
  PID  USER  %CPU  %MEM  COMMAND
12699 app   98.5  12.3  java   # 这个线程 tid=12699，几乎跑满一个核，重点嫌疑对象
12700 app   2.1   12.3  java

# ③ 把线程 id 转成十六进制（jstack 输出的 nid 是十六进制）
$ printf "%x\n" 12699
319b

# ④ jstack 导出全部线程栈，用十六进制 nid 定位到具体线程
$ jstack 12580 | grep "nid=0x319b" -A 20
"pool-3-thread-7" #52 prio=5 os_prio=0 tid=0x... nid=0x319b runnable [0x...]
   java.lang.Thread.State: RUNNABLE
	at com.supernovacompanies.ccc.util.RetryHelper.retryLoop(RetryHelper.java:88)
	at com.supernovacompanies.ccc.service.SyncService.pull(SyncService.java:34)
	# 定位到：RetryHelper.retryLoop 在第 88 行死循环空转

看到 nid=0x319b 且线程状态是 RUNNABLE，堆栈稳定停在同一段业务代码（多采几次样确认不是正常的短暂计算），就可以断定：问题代码就是 RetryHelper.retryLoop 第 88 行附近，接下来直接去看这段逻辑是不是缺少退出条件、或者重试间隔为 0 导致死循环空转。

实战中更快的方式是用 Arthas 的 thread -n 3 命令——它会自动帮你完成"找 CPU 最高的几个线程 + 打印堆栈"这一整套动作，不用再手动 top -Hp + printf 转换，一条命令搞定。

排查时要排除的干扰项

现象容易误判为如何区分
大量线程状态是 BLOCKED CPU 飙高 BLOCKED（等锁）不占 CPU，反而是吞吐下降的表现；CPU 飙高对应的是 RUNNABLE 且真的在跑代码的线程
频繁 Full GC 导致 CPU 高 业务代码死循环 用 jstack 或 top -Hp 定位到高 CPU 线程后，如果线程名类似 GC Thread/G1 Young RemSet Sampling，说明是垃圾收集器本身在忙，需要回头看 GC 日志（见本组第 6 篇）而不是业务代码
短暂的 CPU 毛刺（几秒后自愈） 持续性问题 正常的批处理任务、JIT 编译预热都会造成短暂 CPU 高，只有持续不降才需要认真排查
坑：容器化环境（Docker/K8s）里，容器内看到的 pid 和宿主机看到的 pid 不是一回事——如果在宿主机上 top 看到的 pid 是宿主机命名空间下的编号，需要用 docker top <container> 或者直接进入容器网络/PID 命名空间执行 jstack，否则会出现"pid 对不上"或者"attach 失败"的问题。

核心价值总结

  • 整套排查依赖一个前提：Java 线程和操作系统线程一一对应，才能把 top -Hp 的 tid 和 jstack 的 nid 对上号。
  • 四步口诀：top 找进程 → top -Hp 找线程 → printf 转十六进制 → jstack 对号入座，是最基础也最通用的排查手段，脱离任何图形化工具也能在纯命令行的服务器上完成。
  • 排查前先排除干扰项：区分是业务代码死循环，还是 GC 线程本身在忙，还是短暂的正常毛刺。
  • Arthas 的 thread -n N 命令把这一整套流程自动化了，日常排查优先用它，纯命令行流程作为兜底和原理理解的基础。

与其他知识点的联系

  • jps / jstat / jmap / jstack 速查：本文四步排查法里的第四步就是直接用 jstack——先靠 top -Hp 定位到 CPU 占用最高的线程 tid，再靠 jstack 输出里的 nid 字段对上号，看到这个线程当前具体停在哪一行代码，这是命令行工具组合出的"CPU 飙高排查三件套：top + printf 转换 + jstack"，任何图形化工具背后都是同一套原理。
  • Arthas 实战：Arthas 的 thread -n 3 把"找高 CPU 线程 + 打印堆栈"这一整套手动流程自动化了，一条命令替代了 top -Hp、十六进制转换、jstack 搜索三个步骤；如果高 CPU 线程卡在某个具体方法内部，还可以进一步用 trace 命令看清楚这个方法内部每一步调用花了多长时间，比单纯看一次堆栈快照更细致。
  • 垃圾回收算法（GC 模块）：表格里提到的"频繁 Full GC 导致 CPU 高"是把本文和 GC 知识点串起来的关键线索——如果 jstack 定位到的高 CPU 线程名是 GC Thread 类似的名字，说明问题根源不在业务代码，而要回头看 GC 日志、检查是不是堆参数配置不合理或者存在内存泄漏（见本组第 4 篇）导致老年代频繁触发 Full GC，进而占满 CPU。
  • 字节码执行引擎与 JIT 编译（JVM 理论）：业务代码为什么会导致 CPU 飙高，本质要回到代码是怎么被执行的——死循环之所以能把一个核跑满，是因为解释执行或 JIT 编译后的机器码在不停地被 CPU 取指执行而没有让出；理解 JIT 编译预热阶段本身也会有一小段 CPU 使用率偏高（表格里"短暂毛刺"的一种成因），能帮助排除这类误判。
  • 锁竞争与线程状态（并发模块）：本文强调 BLOCKED 状态的线程不占 CPU 容易被误判——但如果是自旋锁竞争（比如某些无锁结构在竞争失败时会短暂自旋重试而不是直接挂起），线程状态会显示为 RUNNABLE 却并不是在正常执行业务逻辑，这时候需要结合并发模块里对锁实现方式的理解，才能正确判断这究竟是业务代码问题还是锁竞争问题。
