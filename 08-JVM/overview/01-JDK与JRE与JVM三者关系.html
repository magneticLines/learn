JVM 全景

JDK / JRE / JVM 三者关系

三个最容易被新手混为一谈的名词，其实是一层套一层的"俄罗斯套娃"——搞清楚各自的职责边界，后面学 JVM 内部结构才不会张冠李戴。

一句话讲清楚

JVM（Java Virtual Machine） 是一份规范加一份实现，它只负责一件事：装载并执行字节码（.class 文件）。 JRE（Java Runtime Environment） = JVM + 运行程序所需的核心类库（java.lang、java.util 等），是"能跑起来 Java 程序的最小环境"。 JDK（Java Development Kit） = JRE + 编译器 javac、调试器 jdb、打包工具 jar 等开发工具，是"能写、能编、能跑"的完整工具箱。

类比：JVM 好比一台"能读懂特定语言指令的发动机"；JRE 是把这台发动机装进一辆能上路开的车（加上油箱、方向盘这些运行必需品）；JDK 则是连维修工具箱、诊断电脑都配好的整套设备——你不仅能开车，还能造车、修车。

包含关系：JDK ⊃ JRE ⊃ JVM

把三者按"你只用来运行程序" vs "你还要开发程序"拆开看，关系是严格的层层包含：

JDK（Java Development Kit，开发工具包）
  ├─ javac（编译器：.java → .class）
  ├─ jdb / jconsole / jstack / jmap（调试、监控、诊断工具）
  ├─ jar / javadoc（打包、文档工具）
  └─ JRE（Java Runtime Environment，运行环境）
      ├─ 核心类库（java.lang、java.util、java.io ...）
      └─ JVM（Java Virtual Machine，虚拟机本体）
          ├─ 类加载子系统（把 .class 装进内存）
          ├─ 运行时数据区（堆、栈、方法区……）
          └─ 执行引擎（解释器 + JIT 编译器）

也就是说：只有 JVM，字节码没有类库可用，几乎什么都跑不起来（连 System.out.println 都要靠 java.io 类库）； 只有 JRE，能跑程序但没法把 .java 编译成 .class，因为 javac 属于 JDK。

三者对比一览

名词包含内容面向人群能不能编译源码
JVM 类加载子系统 + 运行时数据区 + 执行引擎 虚拟机实现者 / 底层原理学习者 不能，它只认字节码
JRE JVM + 核心类库 + 支撑文件 只需要"运行" Java 程序的普通用户 不能，没有 javac
JDK JRE + 编译器/调试器/打包工具等开发套件 Java 开发者 能，javac 就在其中

常见误区

误区："JDK 9 之后不是不单独发布 JRE 了吗，是不是 JRE 这个概念就没用了？" 从 JDK 11 开始，Oracle 官方确实不再单独打包发布 JRE 安装包（可以用 jlink 按需定制精简运行时），但 JRE 作为"JVM + 类库"这个概念依然成立——它只是不再以独立安装包的形式存在，而是始终作为 JDK 的一部分内置着。理解三者的职责划分，比纠结"有没有单独安装包"更重要。
面试高频："JDK 和 JRE 的区别？" 一句话答案：JRE 用来运行 Java 程序，JDK 用来开发 Java 程序，JDK 比 JRE 多了编译和调试工具。 追问"JVM 呢？" 答：JVM 是两者共同依赖的执行引擎，是真正跑字节码的那个东西。

核心价值总结

  • 层次划分清晰：JVM 管执行，类库管功能，开发工具管编译调试，各司其职。
  • JVM 才是本模块的主角：接下来讲的类加载、内存分区、垃圾回收、JIT 编译，全部发生在 JVM 这一层，与 JDK/JRE 的其他组件无关。
  • 理解这一层划分，才能理解为什么"一次编写、到处运行"是可能的——真正跨平台的是 JVM 这层抽象，下一篇展开讲。
