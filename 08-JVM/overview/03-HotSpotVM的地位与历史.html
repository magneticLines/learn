HotSpot VM 的地位与历史

我们平常说"研究 JVM 原理"，绝大多数情况下研究的都是 HotSpot 这一份具体实现——它是目前 Oracle JDK / OpenJDK 的默认虚拟机，也是事实上的行业标准。

一句话讲清楚

JVM 是规范，HotSpot 是实现。 就像"HTTP"是协议规范，Nginx/Apache 是具体实现一样，Java 虚拟机规范只定义了字节码指令集、类文件格式、运行时数据区的行为要求， 并没有规定"必须怎么实现"。HotSpot VM 就是目前最主流、最成熟的一份具体实现，从 JDK 6 开始成为 Sun/Oracle JDK 的默认虚拟机，也是《深入理解 Java 虚拟机》这本书讨论的主要对象。

"HotSpot" 这个名字从哪来

名字直译就是"热点"——HotSpot 的核心突破在于：它会在运行时持续监控代码的执行频率，找出被反复调用的方法或循环体（"热点代码"），只对这部分代码做即时编译（JIT）优化，而不是一上来就编译全部代码。

类比：一本 800 页的教材，你不会把每一页都背得滚瓜烂熟——真正该花时间精读的是"高频考点"那几十页。HotSpot 对代码做的就是这件事：先用解释器把全书"通读一遍"（保证能跑起来），同时统计哪些"页"被反复翻阅，把这些热点页单独抽出来精加工成机器码，考试（真正执行）时直接用精加工版本，效率飙升。

这套"热点探测 + 分层编译"机制在当年是相对早期 JVM 实现的核心突破：早期方案要么"完全不编译"（纯解释，慢），要么"启动时全量编译"（启动慢，还可能编译了很多只跑一次的冷代码，白费功夫）。HotSpot 用"识别热点、按需编译"取得了启动速度和峰值性能之间的平衡。

简要历史：从 Sun Classic VM 到 HotSpot

Sun Classic VM（JDK 1.0 时代）
  纯解释执行，没有 JIT，逐条解释字节码
  → 性能很差，热点代码和普通代码一视同仁地被反复解释

Exact VM（JDK 1.2 试验性引入，昙花一现）
  尝试引入准确式内存管理等改进，很快被 HotSpot 取代

HotSpot VM（JDK 1.3 起提供，JDK 6 起成为默认虚拟机）
  引入热点探测 + JIT 编译，解释器与编译器混合执行
  → 此后持续演进（分层编译、G1/ZGC 等垃圾回收器迭代），至今仍是主流

HotSpot 最初并非 Sun 公司原生研发，而是收购 Longview Technologies 公司获得的技术，经过 Sun 的持续打磨才成为默认虚拟机。这段历史说明一件事："热点探测 + JIT"这个思路的价值，是被市场和工程实践反复验证出来的，不是一开始就理所当然的设计。

纯解释 vs HotSpot 混合执行

维度纯解释执行（如 Sun Classic VM）HotSpot 混合执行
启动速度 快（不需要编译等待） 快（默认先解释执行）
峰值性能 差，所有代码永远被逐条解释 好，热点代码被编译成本地机器码后接近原生速度
资源开销 低（无编译器开销） 需要额外线程和内存做运行时监控与编译
适用场景 已基本被淘汰 绝大多数通用场景的默认选择
面试高频："JVM 和 HotSpot 是什么关系？" 标准答案：JVM 是 Java 虚拟机规范（The Java Virtual Machine Specification）定义的一套抽象机器行为，HotSpot 是 Oracle 官方提供的一份具体实现，两者是"规范"与"实现"的关系，就像接口和实现类。
误区："看到 JVM 调优参数（如 -XX:+UseG1GC），以为这是 Java 语言规范的一部分。" 这类 -XX 参数绝大多数是 HotSpot 这个具体实现私有的调优开关，换成 GraalVM、OpenJ9 这些其他实现，很多参数并不通用。区分"语言/规范层面的东西"和"某个 JVM 实现私有的东西"，是后面学 JVM 参数调优时避免踩坑的关键。

核心价值总结

  • HotSpot 是实现，不是规范：本模块后续讲的类加载机制、垃圾回收器、JIT 编译细节，绝大多数都是 HotSpot 的具体行为，不完全等同于"所有 JVM 都这样"。
  • "热点探测 + JIT"是 HotSpot 的立身之本：只优化真正值得优化的代码，兼顾启动速度与峰值性能，这个思路贯穿整个执行引擎的设计。
  • 了解历史演进有助于理解"为什么现在这样设计"：纯解释执行被淘汰的教训，正是 JIT 编译存在的原因。
