链接与异常控制流

静态链接与动态链接的区别

你写的 .java 文件从来不是孤立运行的——它依赖 JDK 类库、依赖你引入的第三方 jar。"链接"就是把这些散落的代码片段拼装成一个能跑起来的整体的过程，而拼装的时机和方式，决定了程序的体积、启动速度和可维护性。

为什么需要"链接"这一步

编译器（如 gcc 或 Java 里的 javac）只能按文件为单位处理代码：它编译你写的这一个 .c 或 .java 文件，把用到但没有在本文件定义的符号（比如调用了 printf、调用了别的类的方法）先标记为"未解析的引用"，生成一份不完整的目标文件。

链接器（Linker） 的工作就是把多个目标文件、以及它们各自依赖的库，按符号名字对应起来，拼接成一个完整、可以真正被加载执行的程序。这一步"把编译单元粘合成整体"的方式，主要有静态和动态两种，两者的核心分歧在于：粘合发生在什么时间点。

静态链接：编译期就把代码焊死

静态链接（Static Linking）在生成可执行文件的阶段，就把程序用到的库函数代码原样拷贝进最终的可执行文件里。链接完成之后，这个可执行文件就是一个自给自足的整体，不再依赖外部的库文件——把它拷贝到另一台机器上，只要 CPU 架构和操作系统兼容，直接就能跑。

静态链接示意
// main.o 里调用了 printf，但 printf 的实现在 libc.a 里
gcc main.o -static -o main_static

// 链接完成后，libc 里 printf 相关的机器码
// 被原样拷贝进了 main_static 这个文件
// 该文件体积变大，但运行时不再需要 libc.so
类比 Java：静态链接就像打一个 fat jar / uber jar（用 Maven Shade 插件或 Spring Boot 的可执行 jar）——把所有依赖的 class 文件全部塞进同一个 jar 包，运行时用 java -jar app.jar 就能跑，不需要额外指定 classpath 去找依赖库。体积大，但"打包即完整"。

动态链接：运行时才去加载共享库

动态链接（Dynamic Linking）在编译链接阶段，只在可执行文件里记录"我依赖哪些共享库、需要哪些符号"，并不把库代码拷贝进来。真正的代码拷贝和符号解析，推迟到程序加载运行时才由动态链接器（如 Linux 的 ld.so，Windows 的 DLL 加载机制）完成——把 .so（Linux）或 .dll（Windows）文件加载进程的地址空间，再把符号引用绑定到具体地址。

动态链接示意
gcc main.o -o main_dynamic  // 默认就是动态链接

// main_dynamic 文件很小，里面只记录：
// "运行时请帮我加载 libc.so，并解析 printf 符号"
// 真正执行时，操作系统才把 libc.so 映射进进程地址空间

动态链接最大的好处是共享：如果机器上同时运行 100 个使用了 libc.so 的进程，物理内存里只需要保留一份 libc.so 的代码段，所有进程通过虚拟内存映射共享它。这既节省磁盘空间（可执行文件本身小），也节省物理内存。

类比 Java：动态链接就像传统的"瘦 jar + classpath"模式——你的应用 jar 里只有自己的业务代码，运行时通过 -cp 或 lib/ 目录去加载 Spring、Jackson 等依赖 jar。多个应用如果依赖同一个版本的 jar，理论上也可以共享同一份 jar 文件（虽然 JVM 层面不像 OS 共享库那样做进程间物理内存共享，但磁盘层面是复用的）。

取舍对比

维度静态链接动态链接
链接时机 编译期（生成可执行文件时） 加载期 / 运行期
可执行文件体积 大（依赖代码全部拷贝进来） 小（只记录依赖关系）
运行时依赖 无需外部库，独立运行 必须能找到对应版本的共享库
内存占用（多进程） 每个进程各自占一份库代码 多进程共享同一份库的物理内存
库更新 库改了必须重新编译整个程序 只需替换 .so/.dll 文件，程序无需重编译
典型风险 体积膨胀，多个程序重复携带相同代码 "DLL 地狱"——版本不兼容、库缺失导致运行时报错

常见误区

误区："动态链接一定比静态链接好，因为它更省空间。" 这是只看到了共享的好处，忽略了部署环境的不确定性。动态链接要求目标机器上必须存在版本兼容的共享库，这正是经典的 "DLL 地狱" / "依赖地狱" 问题的根源——同一个符号在不同版本的库里可能行为不同，甚至签名都变了。这也是为什么 Docker 容器化流行起来后，很多场景反而更倾向于把依赖打包进镜像（本质上是把"动态依赖"问题转化成"构建期确定、运行期不变"的静态思路）。
面试高频："为什么现代很多语言（Go、Rust）默认倾向静态链接？" 答案核心是部署简单性优先于磁盘/内存节省——一个自带全部依赖的二进制文件可以直接扔到任意机器上运行，不用担心目标环境缺库或库版本不对，这在容器化、云原生场景下是非常大的运维便利。

核心价值总结

  • 链接的本质是符号解析 + 地址重定位，静态和动态的区别只是"在什么时间点做"。
  • 静态链接拿空间换独立性：体积大，但部署简单、不依赖外部环境。
  • 动态链接拿依赖换共享：体积小、内存可共享，但引入了运行时才会暴露的版本兼容风险。
  • 理解这组权衡后再看 Java 的 fat jar vs 瘦 jar + classpath，会发现这其实是同一个底层问题在不同抽象层的重现。
