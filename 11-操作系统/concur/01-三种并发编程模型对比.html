并发编程回顾

基于进程 / IO多路复用 / 线程的三种并发编程模型对比

02-并发模块讲的是"Java 里怎么写多线程代码、怎么保证线程安全"，这一篇往下退一层：操作系统层面，让一个服务器同时服务多个客户端，其实有三条完全不同的技术路线，线程只是其中一种。

共同的问题：一个服务器如何同时服务多个客户端

设想写一个网络服务器，要同时处理成百上千个客户端连接。最朴素的想法是"来一个客户端就处理一个，处理完再接待下一个"——但这样任何一个客户端卡住（比如网络慢、发送数据很慢），后面所有客户端都要排队等待，完全没法用。操作系统和应用程序设计者对此提出了三种截然不同的解法，对应三种并发编程模型。

模型一：基于进程的并发

每接受一个客户端连接，就用 fork() 创建一个全新的子进程专门服务这个客户端，父进程继续监听新连接。每个子进程都有独立的地址空间，彼此完全隔离。

优点是隔离性极好：一个子进程崩溃不会波及其他进程，也不用担心多个客户端的处理逻辑互相踩内存。缺点也很明显：fork() 创建进程的开销大（需要复制页表等一整套进程上下文，虽然现代系统用写时复制 Copy-on-Write 优化过，但仍然比创建线程重得多），进程间如果需要共享数据，还必须通过管道、共享内存、Socket 等笨重的进程间通信（IPC）机制，无法像线程那样直接共享同一份内存。

模型二：基于 IO 多路复用的并发

整个服务器只用一个进程、一个线程，但通过 select/poll/epoll 这类系统调用，同时"盯着"成百上千个客户端的 Socket 连接，哪个连接上有数据到达就处理哪个，处理完立刻转向下一个就绪的连接，绝不在某一个客户端上阻塞等待。这正是 Nginx、Redis 等高性能服务器的核心并发模型（Redis 早期版本单线程处理所有请求，靠的就是这套机制）。

优点是没有多进程/多线程的切换开销，也不存在多线程共享数据带来的竞态问题（因为始终只有一个执行流在跑业务逻辑）。缺点是编程模型更复杂——所有逻辑都要写成事件驱动、非阻塞的风格（不能有一步"卡住等结果"的代码），业务逻辑一旦有耗时的 CPU 密集计算，会直接卡住整个事件循环，拖慢所有其他连接的响应。

模型三：基于线程的并发

每个客户端连接分配一个线程而不是进程，所有线程共享同一个进程的地址空间。这是 Java 传统 Web 服务器（如 Tomcat 的经典 BIO 模式）最常用的模型，也是 02-并发模块里"如何保证线程安全"讨论的直接应用场景。

优点是创建和切换开销远小于进程（线程共享地址空间，不需要复制页表），同时因为共享内存，线程间通信天然方便（直接读写共享变量即可）。但这个"方便"正是双刃剑的另一面：正因为共享内存，才产生了竞态条件、需要用锁保护临界区——02-并发模块讲的 synchronized、volatile、各种并发工具类，本质上都是在为这个模型的"共享带来的风险"打补丁。

三种模型取舍对比

维度基于进程基于 IO 多路复用基于线程
隔离性 最好，独立地址空间 无隔离概念（单执行流） 差，共享地址空间
创建/切换开销 最大 无切换开销（单线程） 中等
数据共享方式 需要 IPC（管道/共享内存） 天然共享（同一进程内变量） 天然共享（同一进程内变量）
需要处理竞态条件 基本不需要 不需要（单执行流不并发访问） 需要，是主要复杂度来源
编程模型复杂度 简单（每个进程按顺序写逻辑） 复杂（必须事件驱动/非阻塞） 中等（顺序写逻辑，但要考虑同步）
典型代表 传统 Apache prefork 模式 Nginx、Redis、Node.js Java 传统 BIO 服务器、大部分 Java 应用

常见误区

误区："IO 多路复用是单线程，所以性能一定比多线程差。" 恰恰相反，对于以 IO 等待为主、CPU 计算量小的场景（如大部分网络请求转发、KV 存储读写），IO 多路复用往往性能更好——因为它省去了线程切换的上下文开销和竞态处理的锁开销。但如果业务里混入了大量 CPU 密集计算，单线程模型就会成为瓶颈，这也是为什么 Redis 6.0 之后引入了多线程处理网络 IO（但核心数据操作仍是单线程，以此保证操作的原子性）。三种模型没有绝对的优劣，只有"适合什么场景"的取舍。
面试高频："为什么 Redis 是单线程还能这么快？" 标准答案要点：数据全部在内存里操作，避免了慢速磁盘 IO；采用 IO 多路复用模型处理网络连接，避免了线程切换和锁竞争的开销；单线程的操作天然是原子的，不需要额外加锁。这道题本质上考的就是这三种并发模型的取舍理解。

核心价值总结

  • 三种模型是操作系统层面"如何组织并发执行流"的三条不同路线，02-并发模块讨论的线程安全问题，只是"基于线程"这一种模型下才会遇到的问题。
  • 隔离性和共享效率是核心权衡：进程隔离最好但开销最大，线程共享高效但引入竞态风险，IO 多路复用用单线程规避了竞态但要求业务逻辑必须非阻塞。
  • 理解了这三种模型，才能理解 Java NIO、Netty 的 Reactor 模型本质上是把"IO 多路复用"这个操作系统能力包装成了 Java 层面好用的编程接口。

与其他知识点的联系

  • 用户级线程与内核级线程：本篇"基于线程"的模型只说了"用线程"，没深挖线程本身是内核级还是用户级——下一篇继续下潜，讲清楚 Java 传统线程为什么是重量级的 1:1 内核线程，以及虚拟线程如何用 M:N 模型让"基于线程的并发"这条路线变得更轻量、能撑起百万级并发连接。
  • 竞态条件的底层硬件原因：本篇提到"基于线程的模型需要处理竞态问题，是主要复杂度来源"，再下一篇会把这句话拆到 CPU 指令级别——解释为什么共享内存必然带来"读-改-写"被打断的风险，这是"基于进程"和"IO 多路复用"两种模型完全不需要操心、而"基于线程"模型必须正面解决的根本原因。
  • 进程是什么：本篇"基于进程"模型里"每个子进程独立地址空间、需要 IPC 通信"的描述，直接呼应 process 篇对"进程 = 私有地址空间 + 执行线索"的定义；理解了进程的地址空间隔离性，才能理解为什么基于进程的并发天生没有竞态问题，但也失去了直接共享内存的便利。
  • 常见调度算法：基于线程和基于进程的模型都依赖操作系统调度器在多个执行流之间做时间片分配，本篇讨论的"切换开销"正是调度算法里"上下文切换成本"的具体体现；而 IO 多路复用模型只有一个执行流，调度器几乎没有介入的必要，这也是它能省掉切换开销的根本原因。
