用户级线程与内核级线程

02-并发模块里创建一个 Thread、调用 start()，背后到底是谁在负责"调度"这个线程去 CPU 上跑？答案取决于线程是"用户级"还是"内核级"的——这决定了线程调度是操作系统的事，还是应用程序自己的事。

谁来调度线程：内核，还是用户态的线程库

"线程"这个概念可以在两个完全不同的层面实现，区别在于谁负责决定"下一刻该运行哪个线程"。

内核级线程（Kernel-Level Thread, KLT）

线程的创建、调度、切换全部由操作系统内核完成。内核维护着每个线程的调度信息，能感知到系统里究竟有多少个线程在跑。这类线程能被内核直接分配到不同的 CPU 核心上并行执行，也能在一个线程发生阻塞式系统调用（比如等待磁盘 IO）时，内核直接调度同进程的另一个线程继续运行，不会因为一个线程卡住而拖累整个进程。代价是：线程的创建和上下文切换都需要陷入内核态，开销比纯用户态操作要大。

用户级线程（User-Level Thread, ULT）

线程的创建、调度、切换完全在用户态由一个线程库自己管理，内核对这些线程一无所知——在内核眼里，整个进程始终只是"一个"内核调度单位。线程库自己维护一张线程表，用协作式或时间片轮转的方式在用户态"手动"切换执行上下文，切换速度极快（不需要陷入内核态）。但致命缺点是：一旦某个用户级线程发起了阻塞式系统调用（如同步读文件），整个进程（包括其他所有用户级线程）都会被内核一起阻塞——因为内核根本不知道进程内部还有其他"线程"在等着运行，也无法做到只调度别的线程去替补。同时，多个用户级线程也无法真正利用多核 CPU 并行——因为内核只把整个进程当成一个调度单位分配给一个核。

Java 线程本质上是内核级线程（1:1 模型）

传统 JVM（HotSpot）里，一个 Java Thread 对象在创建时会直接映射为一个操作系统的内核级线程，这被称为 1:1 线程模型：一个用户可见的线程实体，对应一个内核调度实体。这也是为什么 Java 线程能利用多核并行、也能在某个线程阻塞 IO 时不影响其他线程调度——因为调度权完全交给了操作系统内核，内核有能力单独暂停/恢复某一个线程。这个映射关系也解释了为什么创建大量 Java 线程会显著消耗系统资源：每个 Thread 背后都真实占用一个内核线程的调度开销和默认栈空间（通常 512KB～1MB）。

现代复兴：协程与虚拟线程

用户级线程"切换快"这个优点，在现代高并发场景下重新被重视起来——只不过换了个名字：协程（Coroutine）、Java 21 引入的虚拟线程（Virtual Thread，JEP 444）。这些方案的共同思路是：在用户态（或者说 JVM 内部）维护大量轻量级的执行单元，由用户态的调度器（而不是操作系统内核）负责把它们映射到少量的内核线程上运行，这被称为 M:N 模型——M 个用户态线程复用 N 个内核线程。

虚拟线程巧妙地解决了传统用户级线程"一阻塞就全阻塞"的老问题：当虚拟线程执行到阻塞式 IO 操作时，JVM 会把它从底层的内核线程上"卸下来"挂起，让内核线程去执行其他就绪的虚拟线程，等 IO 完成后再把它重新调度回某个内核线程上继续执行——这一整套"卸下再挂载"的机制完全在 JVM 用户态内部完成，不需要每个虚拟线程各自占用一个昂贵的内核线程，因此可以轻松创建成百万个虚拟线程。

用户级线程 vs 内核级线程

维度用户级线程内核级线程
调度者 用户态线程库 操作系统内核
创建/切换开销 极小（不陷入内核态） 较大（需要系统调用、上下文切换）
多核并行能力 无法真正并行（内核只见一个调度单位） 可以并行到不同 CPU 核心
阻塞影响范围 一个阻塞，整个进程都阻塞 只阻塞发起调用的那个线程
典型代表 早期协程库、Java 虚拟线程（M:N） Java 传统 Thread（1:1）、pthread

常见误区

误区："虚拟线程就是用户级线程，所以是老技术的简单重命名。" 虚拟线程比早期纯粹的用户级线程更聪明的地方在于：它并不是完全脱离内核独立运作，而是依赖 JVM 与操作系统协作，在虚拟线程执行阻塞 IO 时精确识别"这是一次可以被挂起的阻塞点"，主动让出底层内核线程。这解决了传统用户级线程"一阻塞就全阻塞"的根本缺陷，是一种更精细的 M:N 调度实现，而不是简单地把线程搬到用户态就完事。
面试高频："为什么大量创建 Java 线程会导致系统性能急剧下降？" 标准答案：每个 Java 线程都是一个真实的内核级线程，创建开销大、默认占用较大的线程栈内存，操作系统调度成百上千个线程时上下文切换的开销也会显著增加。这正是线程池要"复用线程"、以及虚拟线程要解决的核心问题。

核心价值总结

  • 线程调度权在内核还是用户态，决定了阻塞行为和多核利用能力的天壤之别：内核级线程调度精细但切换贵，用户级线程切换快但传统实现有"一阻塞全阻塞"的硬伤。
  • 传统 Java 线程是 1:1 内核级线程，创建成本高是所有线程池化实践的根本原因。
  • 虚拟线程是 M:N 模型在现代 JVM 里的落地，本质上是用户级线程思想的复兴，但通过与内核协作解决了老问题，是理解 Java 高并发新方向的关键背景知识。

与其他知识点的联系

  • 三种并发编程模型对比：上一篇说"基于线程的并发"是一整类模型，本篇进一步区分了这类模型内部"线程本身怎么实现"的差异——传统 Java 线程是内核级线程，创建一个就要付出真实的内核调度开销，这也是为什么高并发场景下"基于线程"的开销会显著高于"IO 多路复用"，直到虚拟线程把这个开销降下来才重新拉近了两者的差距。
  • 常见调度算法：内核级线程的调度完全由本 Tab 讲的 FCFS/RR/MLFQ 等算法负责；而用户级线程/虚拟线程的调度是在用户态（JVM 内部）自己实现的一套简化调度逻辑，本质上是把"调度"这件事从内核搬到了应用层，但要解决的问题（谁下一个跑、怎么避免饥饿）是相通的。
  • 竞态条件的底层硬件原因：不管是内核级线程还是虚拟线程，只要它们共享同一份堆内存，就无法逃开竞态条件的硬件根源——线程调度层面的差异只影响"谁负责切换、切换开销多大"，不影响"共享内存读改写会不会被打断"这个更底层的问题，两者是完全独立的维度。
  • 虚拟内存与 JVM 堆内存的关系：本篇提到 Java 传统线程默认占用 512KB～1MB 的栈空间，这部分栈内存和堆内存一样，都是 JVM 进程虚拟地址空间里的一段区域，只是分配策略和生命周期不同（栈线程私有、随线程创建销毁）；大量创建内核级线程之所以会显著消耗系统资源，一部分原因正是每个线程栈都要占用一段虚拟地址（并可能真正提交物理内存）。
