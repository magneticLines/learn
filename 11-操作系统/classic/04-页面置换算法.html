进程调度与经典理论

页面置换算法

process 篇讲过虚拟内存靠页表把虚拟地址映射到物理地址——但物理内存是有限的，装不下所有进程的所有页面时，谁该被换出去，就是页面置换算法要回答的问题。

问题的本质：物理内存不够用怎么办

虚拟内存给了每个进程"独占一整片地址空间"的假象，但物理内存（RAM）是实打实有限的。当进程访问的页面不在物理内存里（缺页中断，Page Fault），操作系统要从磁盘把它调进来。如果物理内存已经满了，就必须先淘汰一个已经在内存里的页面腾出空间——选哪个页面淘汰，就是页面置换算法要解决的问题。

本质是一个预测问题：页面置换算法要做的事情，说到底是回答"未来最长一段时间内不会被用到的是哪个页面"——但没人能真正预知未来，所以所有算法都只是在用不同的方式近似这个理想答案。选得越准，缺页中断越少，程序跑得越快；选错了，可能刚换出去的页面马上又要用，产生"抖动"（Thrashing）。

理论最优：最佳置换算法（OPT / Belady's Algorithm）

如果能预知未来，最优策略当然是：淘汰未来最长时间不会被访问的页面。这个算法叫 OPT，是所有页面置换算法效果的理论上限，但它需要"上帝视角"知道程序完整的未来访存序列，现实中不可能实现，只用来作为评价其他算法优劣的基准线。

先进先出（FIFO）

最朴素的思路：把页面按调入内存的先后顺序排成一个队列，需要淘汰时，永远淘汰最早调入的那个页面——不管它是不是刚被用过。

实现简单，但完全不考虑页面的实际使用情况，效果往往很差。更麻烦的是它有一个反直觉的现象：

Belady 异常（Belady's Anomaly）：正常来说，给进程分配更多的物理页框（内存更大）应该让缺页率下降或不变，但 FIFO 算法在某些访问序列下，物理页框数增加，缺页次数反而增多了。这是一个经典的反例，说明"直觉上更好的资源分配"在 FIFO 下不一定成立，也是 FIFO 被认为设计上有缺陷的重要原因。LRU 等基于栈结构实现的算法则不会出现这种异常。

最近最少使用（LRU，Least Recently Used）

LRU 的思路更贴近现实：淘汰最近一段时间里最久没有被访问过的页面。它的理论依据是局部性原理（Locality of Reference）——程序在某一时间段内，往往集中访问某一小部分数据和代码，"最近用过的，很可能马上还会再用"，反过来"最近没用过的，短期内大概率也不会用"。

LRU 比 FIFO 更接近 OPT 的效果，但代价是实现开销大：要精确知道"最近最少使用"，理论上需要在每次访存时都更新一个全局的访问时间戳或者维护一个双向链表把最近访问的页面移到头部，这在高频访存场景下的硬件和软件开销都无法接受。这也是为什么真实操作系统很少用严格意义上的 LRU，而是用近似算法。

Clock 算法（时钟算法，LRU 的近似实现）

Clock 算法用一个很巧妙的低成本方式去近似 LRU：

  • 所有页框在逻辑上排成一个环形队列（像钟表一样），有一个指针依次扫描。
  • 每个页面维护一个引用位（Reference Bit），页面被访问时硬件自动把这一位置 1。
  • 需要淘汰页面时，指针沿着环扫描：如果当前页面引用位是 1，说明它最近被用过，就把引用位清 0（给它一次"免死机会"），指针继续往前走；如果引用位是 0，说明它一轮扫描下来都没被访问过，就淘汰它。

这样只需要一个 bit 位加一个指针，就能获得"淘汰最近没怎么用的页面"的近似效果，硬件开销远小于维护精确的访问时间戳或链表。这是现实操作系统（包括 Linux 的 CLOCK-Pro 等变种）里被广泛采用的思路。

面试高频："LRU 和 Clock 算法的关系？" 一句话答案：Clock 是 LRU 的近似实现，用一个引用位 + 循环指针的低成本方式，逼近 LRU"淘汰最久未使用页面"的效果，同时避免了 LRU 精确实现的高昂开销。

三种算法对比

算法核心思路优点缺点
FIFO 淘汰最早调入的页面 实现极简单 不考虑使用频率，可能有 Belady 异常
LRU 淘汰最久未被访问的页面 符合局部性原理，效果接近最优 精确实现开销大（需要记录访问时序）
Clock 引用位 + 环形指针近似 LRU 开销小，效果接近 LRU 只是近似，精度不如严格 LRU

和缓存淘汰算法是同一个思路

页面置换算法要解决的问题——有限空间里，选谁淘汰以最大化命中率——本质上和应用层的缓存淘汰算法是完全一样的问题。比如 Redis 的 maxmemory-policy 里的 allkeys-lru/volatile-lru，用的就是 LRU 近似算法（Redis 实际用的是一种基于随机采样的近似 LRU，思路上和 Clock 算法"用低成本手段近似精确 LRU"是相通的，采样出几个 key 比较访问时间，淘汰其中最久未用的）。这一块在 12-缓存模块会详细展开，这里先建立起"操作系统页面置换"和"应用层缓存淘汰"是同一类问题在不同层次上的体现这个连接。

核心价值总结

  • 页面置换本质是预测问题的近似解：OPT 是理论上限但不可实现，其余算法都是在效果和开销之间找平衡点。
  • FIFO 简单但可能有 Belady 异常，LRU 效果好但开销大，Clock 是现实中的折中方案。
  • 思路可以迁移：操作系统的页面置换和应用层的缓存淘汰（如 Redis LRU）解决的是同一类问题，理解了一层，另一层就是触类旁通。
