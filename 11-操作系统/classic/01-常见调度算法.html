进程调度与经典理论

常见调度算法

CPU 只有一个（或几个核），但"就绪"的进程往往有一堆——调度算法就是回答"下一个该轮到谁"这个问题的策略。不同策略背后是完全不同的价值取舍。

为什么需要调度算法

多道程序设计的核心矛盾是：CPU 是稀缺的独占资源，而就绪进程往往不止一个。操作系统必须有一套规则来决定谁先用、用多久、什么时候被换下去。 这套规则设计得好不好，直接决定了系统"用起来"的体验——是像超市排队一样公平，还是像插队严重的窗口一样让人抓狂。

调度算法要在三个经常互相冲突的目标之间做权衡：

  • 公平性（Fairness）：每个进程都能拿到应得的 CPU 时间，不会有人永远排不上号。
  • 响应时间（Response Time）：尤其是交互式场景，用户点一下鼠标，程序要"感觉上"立刻有反应，哪怕后台还有一堆计算任务在跑。
  • 吞吐量（Throughput）：单位时间内系统整体处理完的任务数量，太频繁地切换进程本身也要花时间（上下文切换开销），切多了吞吐量反而下降。
类比：调度算法就像餐厅安排厨师做菜的策略——"先来后到"简单但可能让点了小菜的顾客等一桌满汉全席做完；"轮流切菜、每人做一点"响应快但切换菜品本身也要洗刀具（开销）；"VIP 优先"体验好但普通顾客可能永远排不上。

先来先服务（FCFS，First-Come First-Served）

最朴素的思路：进程按到达就绪队列的顺序排队，谁先来谁先用 CPU，用完（或阻塞）才轮到下一个，非抢占式。

问题很直观：如果一个耗时很长的任务排在前面，后面一堆短任务都得干等，整体的平均等待时间会被拉得很长。这在操作系统理论里叫 护航效应（Convoy Effect）——想象高速公路上一辆超载卡车挡在前面，后面一长串小车只能龟速跟着挪。

陷阱：FCFS 本身不复杂，但很多人会以为它"公平"，其实它只是"按到达顺序公平"，对任务本身的长短完全不敏感，短任务的用户体验会很差。

时间片轮转（RR，Round Robin）

给每个进程分配一个固定的时间片（比如 20ms），用完就强制切换给下一个进程，抢占式调度的代表。就绪队列是一个环形队列，轮流处理，保证谁都不会被无限期地晾在一边。

时间片长度是关键的调优参数：

  • 时间片太长 → 退化成接近 FCFS，交互响应变差。
  • 时间片太短 → 上下文切换过于频繁，CPU 大量时间花在"保存/恢复现场"上，有效吞吐量下降。
经验法则：《现代操作系统》里提到一个直觉判断——时间片要设置得比典型的上下文切换开销大得多（比如切换开销 1ms，时间片设 20-50ms 比较合理），否则大部分 CPU 时间都在"切换"本身上空转。

优先级调度（Priority Scheduling）

给每个进程赋一个优先级，调度器永远选优先级最高的就绪进程运行。优先级可以是静态的（创建时定死），也可以是动态的（运行过程中动态调整，比如等待越久优先级越高）。

最大的问题是 饥饿（Starvation）：如果高优先级进程源源不断地到来，低优先级进程可能永远轮不到 CPU。经典的解决方案是 老化（Aging）——进程等待的时间越长，就逐渐提升它的优先级，保证再低优先级的任务最终也能被调度到。

面试高频："优先级调度会有什么问题？怎么解决？" 标准答案就是"饥饿 + 老化机制"，一定要能说出这两个关键词。

多级反馈队列（MLFQ，Multilevel Feedback Queue）

这是现代操作系统（Linux、Windows 的调度器都是它的变种）实际采用的思路，可以理解成前几种算法的"综合与进化"：

  • 系统维护多个优先级队列，每个队列对应不同的时间片长度（优先级越高，时间片通常越短）。
  • 新进程从最高优先级队列开始跑。如果一个时间片内没跑完（说明它是计算密集型的长任务），就把它降级到下一级队列，时间片也变长。
  • 如果进程在时间片用完之前就主动让出 CPU（比如等待 IO，说明它是交互式的短任务），就保持在原队列甚至适当提升优先级。
  • 同样引入老化机制，防止低优先级队列的任务被饿死。

这套机制的巧妙之处在于：它不需要事先知道任务是"长任务"还是"短任务"，而是通过运行时的行为反馈动态学习出来——短任务（交互式）会一直呆在高优先级、短时间片的队列里，响应快；长任务（计算密集）会逐渐下沉到低优先级、长时间片的队列里，减少切换开销、保证吞吐量。这正好同时兼顾了响应时间和吞吐量。

算法对比一览

算法是否抢占优点缺点
FCFS 否 实现简单，无饥饿 护航效应，短任务体验差
时间片轮转 RR 是 公平，响应及时 时间片选取敏感，切换开销
优先级调度 可抢占可不抢占 能体现任务重要性差异 低优先级可能饥饿（需老化缓解）
多级反馈队列 MLFQ 是 无需预知任务类型，自适应响应与吞吐的平衡 实现复杂，参数（队列数/时间片/降级规则）需调优

核心价值总结

  • 没有万能算法，只有权衡：公平性、响应时间、吞吐量三者很难同时最优，选算法本质是选取舍。
  • MLFQ 是现实答案：它不是发明了新的原理，而是把 FCFS、时间片、优先级的思路组合起来，用运行时反馈自适应地逼近最优。
  • 面试要记住的三个词：护航效应（FCFS）、时间片长度权衡（RR）、饥饿与老化（优先级）——这是最容易被追问细节的地方。

与其他知识点的联系

  • 死锁的四个必要条件：调度算法决定"谁能拿到 CPU"，死锁讨论的是"谁能拿到资源"，两者都是操作系统对有限资源做仲裁的问题，只是调度器抢占的是 CPU 这一种特殊资源——可以随时暂停恢复，而死锁涉及的资源往往不可剥夺，这也是为什么调度不会死锁但资源分配会。
  • 用户级线程与内核级线程：本篇讨论的 FCFS、RR、优先级、MLFQ 都是内核调度器对线程/进程的调度策略；如果线程是用户级的，内核完全看不到它，调度决策就要下沉到用户态的线程库自己实现一套类似但更简陋的策略，这也是虚拟线程需要 JVM 自己做"调度"的原因。
  • 三种并发编程模型对比：基于线程的并发模型之所以能同时服务多个客户端，靠的正是本篇的调度算法在后台把 CPU 时间切成时间片分给各个线程；而 IO 多路复用模型只有一个线程，根本不需要调度器介入多个执行流之间的切换。
  • 页面置换算法：调度算法和页面置换算法是同一类问题在不同资源维度上的重现——前者回答"有限的 CPU 时间该分给谁"，后者回答"有限的物理页框该留给谁"，两者都要在"性能开销"与"决策准确性"之间做权衡（MLFQ 用运行时反馈自适应，LRU/Clock 同样依赖运行时的访问历史来做决策）。
