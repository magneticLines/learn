进程调度与经典理论

死锁的四个必要条件

02-并发模块里排查过 Java 死锁的现象，这里补上理论根基——死锁不是玄学，它有严格的、可以逐条验证的四个必要条件，缺一个都不会发生。

什么是死锁

死锁（Deadlock）：一组进程（或线程）中的每一个都在等待这组进程中另一个成员占有的资源，导致所有成员都永远阻塞、谁也无法继续往下走。

类比：两个人在一座只能容一人通过的独木桥上迎面相遇，谁都不肯退让——A 说"你先让开我才能走"，B 说"你先让开我才能走"，双方就这样永远僵在桥中间。这就是死锁最直观的画面：每个人都拿着对方需要的资源（桥上的位置），同时又在等待对方释放。

四个必要条件

1971 年 Coffman 等人提出，死锁发生必须同时满足以下四个条件，这也是这个知识点最容易被考的地方——四个条件要求逐字背下来，因为面试常常要求"破坏哪一个条件来避免死锁"。

1. 互斥条件（Mutual Exclusion）

资源一次只能被一个进程占用，不能共享。比如一台打印机同一时刻只能有一个进程在用，别的进程想用就必须等待。

这个条件往往是资源本身的物理属性决定的（打印机、独占锁），很难被打破——你不能强行让互斥资源变成共享资源，比如两个人同时抢一把椅子坐，本质上物理空间就不允许共享。

2. 请求与保持条件（Hold and Wait）

进程已经至少持有一个资源，同时又在请求新的资源，而这个新资源正被别的进程占用。也就是说，进程不会一次性申请完所有需要的资源，而是拿到一部分后再去申请剩下的，这就给"卡在中间"留下了空间。

3. 不可剥夺条件（No Preemption）

进程已经获得的资源，在未使用完之前，不能被其他进程或操作系统强行剥夺，只能由持有者自己主动释放。如果资源可以被强制抢走，死锁就能通过"抢占"来打破。

4. 循环等待条件（Circular Wait）

存在一个进程等待资源的环形链：P1 等待 P2 占有的资源，P2 等待 P3 占有的资源，……，Pn 等待 P1 占有的资源，形成一个闭环。

回到独木桥类比：循环等待条件就是"A 在等 B 让路，B 在等 A 让路"——这正是一个最小的、长度为 2 的循环等待环。如果是三个人在三岔路口互不相让，环就变成 A→B→C→A，本质完全一样。

为什么"四个都满足"才会死锁

这四个条件是必要条件的合取：只要破坏其中任意一个，死锁就不可能发生。这也是死锁预防策略的理论基础（下一篇会展开）：

  • 如果资源可以共享（破坏互斥）→ 大家都能用，不存在互相等待。
  • 如果进程必须一次性申请完所有资源才能开始运行（破坏请求与保持）→ 不会出现"拿着一半、等另一半"的中间状态。
  • 如果资源可以被强制剥夺（破坏不可剥夺）→ 系统可以把某个进程手里的资源抢过来给别人，打破僵局。
  • 如果所有进程申请资源必须按照统一的全局顺序（破坏循环等待）→ 资源请求图不可能形成环。
常见误区：很多人以为"只要有循环等待就一定是死锁"，其实循环等待只是必要条件之一——如果资源是可以共享的（不满足互斥），即使表面上看起来在"互相等"，实际上大家都能同时拿到资源，并不会真正卡死。四个条件必须同时成立才叫死锁。

呼应 Java 中的死锁

02-并发模块里排查过的经典 Java 死锁场景——两个线程分别持有 lockA、lockB，又分别尝试获取对方持有的锁——正好是这四个条件的完美实例：

DeadlockDemo.java
// 线程1: synchronized(lockA) { synchronized(lockB) { ... } }
// 线程2: synchronized(lockB) { synchronized(lockA) { ... } }
// 互斥: synchronized 锁本身就是互斥资源
// 请求与保持: 线程1拿着lockA又去请求lockB，反之亦然
// 不可剥夺: JVM 的内置锁不会被强制剥夺，只能线程自己释放
// 循环等待: 线程1等线程2的锁，线程2等线程1的锁，形成环

用 jstack 抓线程栈时能看到 Found one Java-level deadlock 的提示，本质上就是 JVM 检测到了资源分配图里的一个环。这也解释了为什么"固定加锁顺序"是 Java 里最常见的死锁规避手段——它直接破坏的就是循环等待条件。

核心价值总结

  • 四个条件缺一不可：互斥、请求与保持、不可剥夺、循环等待，必须同时成立死锁才会发生。
  • 破坏任意一个即可避免死锁：这是所有死锁预防策略的理论出发点，下一篇展开讲具体怎么破坏。
  • 面试必背：能脱口而出四个条件的名字，并且能举一个具体场景说明"这里满足了哪一个"，是这个知识点的基本要求。
