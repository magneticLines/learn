进程与虚拟内存

进程控制：fork / exec / wait

上一篇讲了进程"是什么"，这一篇讲进程的"生老病死"——Unix 用三个看似简单的系统调用，拼出了一整套进程创建、替换、回收的机制，理解它们能让你彻底看懂 Runtime.exec()、Shell 管道、乃至容器技术底层的原理。

三个系统调用各管一段生命周期

Unix 系统把"启动一个新程序"这件事拆成了两个独立的步骤，分别由两个系统调用负责，这是 Unix 设计哲学里非常经典的"单一职责"体现：

系统调用职责一句话理解
fork()创建一个新进程复制"我自己"，得到一个几乎一模一样的副本
exec()替换进程的代码和数据让当前进程"改头换面"，变成执行另一个程序
wait()父进程等待并回收子进程父进程阻塞，直到子进程结束并交出退出状态

为什么不直接搞一个"创建并运行新程序"的函数了事？因为拆开之后更灵活：fork 之后、exec 之前，子进程有机会做一些定制化的准备工作（比如重定向标准输出到文件、关闭某些文件描述符），这正是 Shell 实现 command > file.txt 重定向、cmd1 | cmd2 管道的底层手段。

fork()：一次调用，两次返回

fork() 是这三者里最反直觉的一个：它被调用一次，却会返回两次——一次在父进程里返回，一次在新创建的子进程里返回。父子进程从 fork() 返回后的那一行代码开始，各自独立地并发执行下去。

Cfork_demo.c
#include <stdio.h>
#include <unistd.h>

int main() {
    // fork 之前只有一个进程在跑这行之前的代码
    pid_t pid = fork();

    if (pid == 0) {
        // 返回值为 0：说明这是"子进程"在执行这个分支
        printf("我是子进程\n");
    } else {
        // 返回值为子进程的 PID：说明这是"父进程"在执行这个分支
        printf("我是父进程，子进程 PID = %d\n", pid);
    }
    // 注意：这一行父子进程都会各自执行一次，输出两次
    printf("这行话会被打印两次\n");
    return 0;
}
关键点：调用之后系统里凭空多了一个进程，两个进程从此刻起各自独立执行、独立调度，谁先运行完全由调度器决定，代码里不能假设执行顺序。子进程拿到的是父进程地址空间、文件描述符表等状态的一份"逻辑拷贝"（不是真的一模一样的两块物理内存，下面细讲），但拥有自己独立的 PID。

写时复制（Copy-on-Write）：不是真的复制整个地址空间

如果 fork() 每次调用都真的把父进程的整个虚拟地址空间（可能几百 MB 甚至几 GB）原样复制一份给子进程，开销会大到不可接受——尤其考虑到很多场景下 fork() 后立刻就调用 exec() 把这份拷贝整个扔掉重新加载新程序，复制工作完全是浪费。

所以现代操作系统采用写时复制（Copy-on-Write，COW）优化：fork() 时父子进程的页表被设置为指向同一批物理页面，且都标记为只读。只有当父进程或子进程中的某一方尝试写入某个页面时，才会触发一次缺页异常，内核这时才真正为那一个页面复制出一份独立的物理拷贝，让两边分道扬镳。没被写过的页面（通常是大部分只读代码段）会一直共享下去，从来不会被复制。

类比：COW 就像图书馆的"共享笔记"——两个学生 fork 出来的笔记本封面上都写着同一份内容的引用，谁都还没动笔前，两人翻的其实是同一本书。等到有人真的要在某一页上写字涂改时，管理员才悄悄给他复印一份那一页，让他改自己的复印件，不影响另一个人看到的原件。这是"惰性求值"思想在操作系统里的典型应用，后面讲虚拟内存的缺页异常时还会遇到同一套机制。

exec()：让进程"脱胎换骨"

fork() 复制出来的子进程运行的还是父进程的那份代码。如果想让子进程去执行一个完全不同的程序（比如 Shell 里输入 ls，Shell 进程 fork 出子进程后要让它变成 ls 程序），就要用 execve()（及其一系列变体，如 execvp）。

Cfork_exec_demo.c
pid_t pid = fork();
if (pid == 0) {
    // 子进程：把自己的代码段、数据段、堆栈全部替换成 ls 程序的内容
    char *args[] = {"ls", "-l", NULL};
    execvp("ls", args);
    // 正常情况下这一行永远不会被执行到，因为 exec 成功后当前进程的代码已经被整体替换了
    printf("execvp 失败才会走到这里\n");
}

exec 系列调用的关键特性：它不创建新进程，PID 保持不变，只是把当前进程的虚拟地址空间内容（代码、数据、堆、栈）整体替换成新程序的内容，然后从新程序的 main 开始执行。一旦 exec 成功，原来那份代码就彻底不存在了，正常流程下永远不会"返回"到调用它的那行代码之后。

这就是为什么 Unix 世界"创建新进程"和"运行新程序"要拆成 fork + exec 两步——先 fork 出一个身份独立的进程壳子，再用 exec 往壳子里装入想要运行的程序内容，两个正交的关注点被彻底解耦。

wait()：父进程回收子进程，避免"僵尸"

子进程结束运行后，它的退出状态（正常退出还是被信号杀死、退出码是多少）需要被内核保留下来，直到父进程来"收尸"。父进程调用 wait() 或 waitpid() 会阻塞，直到某个子进程终止，然后拿到它的退出状态，内核才彻底释放这个子进程占用的所有资源。

Cfork_wait_demo.c
pid_t pid = fork();
if (pid == 0) {
    exit(42); // 子进程退出，退出码 42
} else {
    int status;
    waitpid(pid, &status, 0); // 父进程阻塞等待，直到子进程退出
    if (WIFEXITED(status)) {
        printf("子进程退出码: %d\n", WEXITSTATUS(status)); // 打印 42
    }
}
面试高频：僵尸进程（Zombie） —— 如果子进程已经终止，但父进程还没调用 wait() 回收它，这个子进程就变成"僵尸"：它已经释放了大部分资源，但进程表里的一个条目（保存着退出状态）还占着，处于"死了但没埋"的状态。如果父进程一直不调用 wait（比如父进程逻辑有 bug，或者父进程自己先退出了），僵尸进程会一直占用进程表项，大量堆积可能耗尽系统的进程号资源。

孤儿进程则是另一种情况：父进程比子进程先退出，子进程会被过继给 1 号进程（init/systemd）领养，由它负责后续的 wait 回收，不会变成僵尸。

三者协作的全景：以 Shell 执行一条命令为例

1. Shell 主进程：读取用户输入的命令行，如 "ls -l"
2. fork()：Shell 复制出一个子进程（写时复制，几乎零开销）
3. 子进程调用 exec()：把自己变身为 ls 程序，开始真正执行 ls 的逻辑
4. Shell（父进程）调用 wait()：阻塞等待，直到子进程（ls）执行完毕
5. ls 执行完毕，退出：内核保留退出状态
6. wait() 返回：Shell 拿到退出状态，回收子进程资源，打印下一个提示符

与 Java、02-并发 模块的联系

Java 里对应什么？Java 的 ProcessBuilder / Runtime.exec() 底层就是对 fork + exec（Windows 上是 CreateProcess，机制略有不同但目的一致）的封装，用来启动一个外部进程（比如从 Java 里调用一个 Shell 脚本）。而 Java 里 new Thread() 创建的是线程，走的是另一条路径（Linux 上是 clone() 系统调用，与父线程共享地址空间），跟 fork() 完全不是一回事——fork 复制出的子进程有独立地址空间，而线程共享地址空间，这也是为什么"进程间通信需要 IPC，线程间通信直接共享变量"的根本原因，02-并发 模块里讨论的可见性、竞态条件问题，本质上都建立在"线程共享地址空间"这个前提之上。
面试高频："fork 和多线程创建线程的区别？" 回答要点：fork 创建的是独立地址空间的新进程，开销相对更大（即使有 COW 优化，也仍需要复制页表等元数据），进程间通信需要 IPC；而创建线程共享同一地址空间，开销更小，线程间通信直接读写共享内存即可，但需要额外的同步机制保证正确性。选择哪种取决于是否需要隔离性：需要强隔离用多进程（如浏览器的多进程架构、Nginx 的 worker 进程），需要高频共享数据且性能敏感用多线程。

核心价值总结

  • fork() 创建新进程（一次调用两次返回），exec() 替换进程的代码和数据，wait() 让父进程回收子进程、避免僵尸进程。
  • 写时复制（COW）是 fork() 高效的关键：并不真正复制物理内存，而是共享物理页面直到某一方写入时才复制，这是"惰性求值"思想的典型应用。
  • fork + exec 分离让 Unix 拥有了极大的灵活性（重定向、管道都靠这个间隙实现），这是 Unix 设计哲学"小工具组合"的直接体现。
