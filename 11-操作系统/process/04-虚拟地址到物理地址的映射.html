进程与虚拟内存

虚拟地址到物理地址的映射

上一篇讲了"为什么"要有虚拟内存，这一篇讲"怎么实现"——页表如何完成地址转换、为什么要按"页"这个固定大小的单位划分、TLB 和多级页表分别解决了转换过程中的哪两个新问题。

核心机制：页（Page）与页表（Page Table）

虚拟内存不是以字节为单位、随意地把虚拟地址映射到物理地址（那样映射表会大到无法维护），而是把整个地址空间切成大小固定的块，每一块叫一页（Page）（典型大小 4KB），物理内存也按同样大小切成一块块物理页帧（Page Frame）。地址转换的粒度就是"页"，而不是"字节"。

操作系统为每个进程维护一张页表，本质上是一个数组（或更复杂的树形结构），每一项（Page Table Entry，PTE）记录"虚拟页号 → 物理页帧号"的映射关系，外加一些标志位（是否有效、是否可写、是否在物理内存里等）。

虚拟地址（VA）= 虚拟页号（VPN） + 页内偏移量（Offset）
物理地址（PA）= 物理页帧号（PFN） + 页内偏移量（Offset，不变）

地址转换过程：
1. CPU 拿到虚拟地址，拆成 VPN 和 Offset 两部分
2. 用 VPN 去页表里查，找到对应的 PFN
3. 用 PFN 替换 VPN，Offset 部分原样保留，拼出物理地址 PA
4. 用 PA 去物理内存条上真正取数据

关键点：页内偏移量部分完全不需要转换，因为"页"是转换的最小单位，同一页内部的相对位置在虚拟和物理地址里是一致的——这也是为什么页大小必须是 2 的幂次（比如 4096 = 2^12），这样可以直接用地址的低 12 位作为 Offset，高位作为 VPN，用位运算而不是除法来拆分地址，效率极高。

为什么要用固定大小的"页"作为管理单位

如果不用固定大小的页、而是让每个进程按需申请任意大小的连续虚拟内存块（类似早期的"分段"方案），会重新引入物理内存分配里最头疼的问题：内存碎片。今天分配一块 300 字节的、明天分配一块 700 字节的，用久了物理内存会变得七零八落，出现大量小得无法再利用的空隙，即使总空闲空间够，也可能找不到一段连续的大空间来满足新的分配请求。

用固定大小的页作为单位后，物理页帧可以像标准化的集装箱一样管理——任何一个空闲页帧都可以用来存放任何一个虚拟页的内容，操作系统只需要维护一份简单的"空闲页帧列表"，分配和回收都是 O(1) 级别的简单操作，彻底避免了外部碎片问题（只会有页内部没用满的"内部碎片"，代价小得多）。

类比：页表机制就像给公寓楼里所有房间都装修成统一规格的"标准间"（页），物业不需要记住"张三家占了从 3.2 米到 8.7 米这段不规则空间"这种连续区间描述，只需要一张表格记录"张三的第 1 号标准间对应大楼的第 205 号物理房间、第 2 号标准间对应第 88 号物理房间……"。房间可以东一间西一间地零散分布在大楼各处，通过这张表拼接成张三眼里"连续"的一套房。

缺页异常（Page Fault）：页表也能表示"还没加载"

页表项里有一个"有效位（valid bit）"，标记这个虚拟页当前是否已经有对应的物理页帧。如果一个虚拟页对应的内容还在磁盘上、尚未被装入物理内存，有效位是 0。当 CPU 试图访问这样的虚拟地址时，会触发一个缺页异常（Page Fault），把控制权交给操作系统内核的缺页处理程序：内核找一个空闲物理页帧（或者先换出一个不常用的旧页），把需要的数据从磁盘读进来，更新页表项，再让原来的指令重新执行一次。

这正是上一篇提到的"按需装载"的具体实现机制，也是上上篇 fork() 写时复制的实现基础——写时复制页面被标记为只读，写入时触发的其实是一种特殊的保护异常（利用了类似缺页异常的硬件机制），内核借此机会介入，完成真正的页面复制。

TLB：给页表查询加一层缓存

页表本身也存放在内存里。如果每次访存都要先去内存里查一次页表拿到物理地址，再去内存里真正读写数据，等于每次内存访问都变成了两次，性能直接腰斩。

解决办法是给地址转换本身加一层高速缓存：TLB（Translation Lookaside Buffer，转址旁路缓存/快表），它是 CPU 内部一块很小但极快的硬件缓存，专门缓存最近用过的"虚拟页号 → 物理页帧号"映射。地址转换时先查 TLB，命中就直接拿到物理地址，只有 TLB 未命中时才需要真正去内存里查页表（这个过程叫 Page Table Walk），然后把结果填回 TLB 供下次使用。

由于程序访存具有局部性原理（短时间内反复访问相近的地址，这一点在 08-JVM 讲对象分配、GC 分代假设时也出现过同样的思想），TLB 的命中率通常非常高（可达 99% 以上），这层缓存才使得虚拟内存机制的实际开销可以被压到几乎可以忽略。

多级页表：解决页表本身占用内存太多的问题

另一个现实问题：64 位地址空间如果用单级、扁平的页表来存储全部映射关系，页表本身会大到离谱（哪怕大部分虚拟地址空间进程根本用不到，也要为这些"空洞"预留页表项）。解决办法是把页表也做成多级树形结构：一级页表的每一项不直接指向物理页帧，而是指向一个二级页表，二级页表才指向真正的物理页帧（层数因体系结构而异，x86-64 常见是 4 级）。

单级页表（问题：即使大片虚拟地址空间没用到，页表项也要全部预先分配）：
VPN → PTE[0], PTE[1], PTE[2] ... PTE[N]（N 巨大，绝大多数项其实用不到）

多级页表（优化：某个一级页表项对应的整个区域都没用到，就不用创建它下面的二级页表）：
一级页表 → [指向二级页表A, 空(未分配), 指向二级页表B, 空(未分配) ...]
    二级页表A → 真正的物理页帧映射

好处是：只有实际被使用到的地址区间才需要真正分配对应层级的页表结构，大量从未被触碰的虚拟地址区域（比如一个进程的虚拟地址空间里，代码段和堆之间空出来的一大片区域）根本不需要占用任何页表内存，用空间换空间的方式反而节省了空间。

常见误区

误区："页表查询和 TLB 查询是软件做的事，跟 CPU 硬件没关系" —— 恰恰相反，地址转换的高频路径（TLB 查询、根据页表项拼出物理地址）是由 CPU 里专门的硬件单元 MMU（Memory Management Unit）完成的，操作系统只负责在缺页异常发生时介入、维护页表内容、管理 TLB 的失效（比如进程切换时可能需要刷新 TLB）。硬件和内核软件是分工协作的关系，纯软件模拟地址转换的性能是不可接受的。
面试高频："什么是 TLB，为什么需要它？" 答案要点：TLB 是缓存虚拟地址到物理地址映射关系的硬件缓存，因为页表本身存在内存里，若每次访存都要先访问内存查页表，会让访存开销翻倍；利用程序访存的局部性原理，用一块小而快的缓存记住近期的地址转换结果，可以把这个额外开销降到接近于零。

核心价值总结

  • 虚拟内存的地址转换以"页"为固定单位，用页表记录虚拟页到物理页帧的映射，避免了内存碎片问题。
  • 缺页异常让"页表项暂时无效"成为可能，从而支撑起按需装载。
  • TLB 用硬件缓存加速高频的地址转换查询，多级页表用树形结构避免页表本身占用过多内存——这两个优化分别针对"时间开销"和"空间开销"。
  • 这一整套机制正是下一篇的基础：当我们说"JVM 堆内存"，指的其实就是进程虚拟地址空间里的一段，它的物理内存占用完全取决于操作系统的页表和按需装载机制。
