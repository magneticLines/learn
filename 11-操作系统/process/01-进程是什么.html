进程与虚拟内存

进程是什么

进程不是"一个正在运行的程序"这么简单——它是操作系统精心构造出来的一个幻觉：让每个程序都以为自己独占了整台电脑。理解这个幻觉是怎么造出来的，是理解虚拟内存、并发、系统调用的总开关。

先从一个矛盾说起

你的电脑上同时开着 IDE、浏览器几十个标签页、聊天软件、音乐播放器……但 CPU 核心数可能只有 8 个，甚至更少。 每个程序在运行时都会用到 CPU、内存、磁盘、网卡这些硬件资源，而硬件资源是有限且需要互斥访问的——两个程序不可能真的在同一时刻使用同一个 CPU 核心。

矛盾就在这里：硬件资源是有限的、独占的，但用户希望程序之间互不干扰、各自"独立运行"。 操作系统解决这个矛盾的答案，就是进程（Process）这个抽象。

类比：把 CPU 想象成一位只能同时接待一位客人的理发师，把内存想象成一整栋公寓楼。如果没有"预约系统"和"门牌号"，客人们会互相打断、乱闯别人的房间。操作系统就是这套预约系统 + 门牌号管理员：它让每位客人都感觉理发师专属于自己、都感觉自己独占了一整栋楼——但实际上大家在分时共享同一批硬件。

进程的本质定义

CSAPP 给出的定义非常精炼：进程是操作系统对一个正在运行的程序的一种抽象，它提供给应用程序两个关键的幻觉——

  1. 独占处理器的幻觉：好像只有你这个程序在使用 CPU，尽管实际上系统里可能同时有几百个进程在"抢" CPU。
  2. 独占内存的幻觉：好像只有你这个程序在使用内存，你的地址空间从 0 开始、连续、只属于你，尽管物理内存条上同时塞着几十个进程的数据。

这两个幻觉分别由两套机制支撑：

逻辑控制流：进程的运行轨迹

如果你用调试器单步跟踪一个进程，会看到一系列 PC（程序计数器）值的序列——这个序列就叫控制流。 在单核 CPU 上，如果只运行一个进程，它的控制流是连续不间断的，这叫物理控制流。

但现实是，CPU 会在多个进程之间来回切换：运行进程 A 一小段时间，暂停，切到进程 B，再切回 A…… 从进程 A 自己的视角看，它的指令序列（去掉被切走的那些间隙）依然是连续的一条执行轨迹，这条"只看自己"的轨迹就叫逻辑控制流（Logical Control Flow）。

时间线（物理视角，单核 CPU 真实发生的顺序）：
── A1 A2 A3 [切换] B1 B2 [切换] A4 A5 [切换] B3 B4 ──

进程 A 的逻辑控制流（A 自己视角，感觉不到被打断过）：
── A1 A2 A3 A4 A5 ──

进程 B 的逻辑控制流（B 自己视角）：
── B1 B2 B3 B4 ──

这就是"独占 CPU 幻觉"的技术本质：每个进程只能看到自己的逻辑控制流，看不到别的进程插进来的间隙。 多个进程的执行在时间上交错进行，这种现象叫并发（Concurrency）——注意，哪怕只有一个 CPU 核心，只要通过上下文切换来回交替执行多个进程，就已经构成并发了，这跟"并行（Parallelism，多核同时执行）"是两回事。这一点会在 02-并发 模块反复用到。

进程 = 私有地址空间 + 一条（或多条）执行线索

再精确一点，一个进程在操作系统眼里由两部分核心状态构成：

  • 私有的虚拟地址空间：这个进程能看到、能访问的全部内存"地图"，包括代码、数据、堆、栈。这张地图是每个进程独享的一份拷贝（幻觉），别的进程看不到、也碰不到。
  • 一条执行这段代码的控制流：也就是线程。早期操作系统里一个进程只有一条控制流，现代操作系统里一个进程可以包含多条线程，共享同一份地址空间，但各自拥有独立的程序计数器和栈。这也是"进程 vs 线程"最本质的区别——线程间共享地址空间，进程间地址空间相互隔离。
连接 02-并发：Java 里 Thread、线程池背后的线程，都是运行在同一个 JVM 进程内部的多条逻辑控制流，它们共享同一份堆内存（所以才需要 synchronized/volatile 处理可见性和竞态问题）。而如果你 Runtime.exec() 启动一个外部进程，那是彻底隔离的地址空间，两边只能通过管道、Socket 这类进程间通信（IPC）机制交换数据，不能直接共享变量。这个区别贯穿整个并发模块的设计动机。

为什么一定要由操作系统内核来提供这层抽象

也许你会想：能不能让应用程序自己协商谁先用 CPU、谁用哪块内存？历史上真的这么干过（早期批处理系统），但问题很明显：

  • 程序之间互相不信任、互相不知道对方的存在，没法自觉协调。
  • 一个程序死循环或崩溃，会直接拖垮整台机器——没有人能把它强制切走。
  • 一个程序可以随意读写另一个程序的内存，恶意或者写崩了都无法被阻止。

所以必须有一个拥有最高权限、独立于所有应用程序之外的仲裁者——这就是操作系统内核。 内核运行在 CPU 的特权模式（内核态）下，普通进程运行在非特权模式（用户态）下，进程想请求内核帮忙（读文件、分配内存、创建新进程……）必须通过系统调用这个受控入口，这也是链接与异常控制流 Tab 里"系统调用的本质"要展开的内容。

常见误区

误区一："进程就是程序" —— 不对。程序是磁盘上一堆静态的指令和数据（比如一个 .exe 或 .class 文件），进程是这个程序被装载、运行起来之后，操作系统为它分配的一整套运行时状态（地址空间、寄存器、打开的文件描述符……）。同一个程序可以同时启动多个进程（比如打开两个记事本窗口），它们是两个完全独立的进程，互不干扰。
误区二："并发就是多核并行" —— 并发描述的是"多个逻辑控制流在时间上交错"，并行描述的是"真的有多个硬件在同一时刻执行"。单核 CPU 通过快速切换也能实现并发（只是没有并行）；多核 CPU 既能并发也能并行。面试里混用这两个词会被认为概念不清晰。

核心价值总结

  • 进程是操作系统提供的抽象，用来解决"硬件资源有限 vs 程序希望互不干扰独立运行"这个根本矛盾。
  • 它同时制造了独占 CPU（靠上下文切换 + 逻辑控制流）和独占内存（靠虚拟内存，下一 Tab 主题）两个幻觉。
  • 进程的隔离性由内核在硬件特权级别上强制保证，普通程序无法绕过——这正是系统安全和稳定性的基石。
