IO 多路复用：select / poll / epoll

一个线程如何同时"盯"着上万个网络连接，只在真正有数据到达时才去处理？这是高并发服务器的核心问题，也是操作系统面试里出现频率最高的考点之一。

要解决的问题：一个线程，成千上万个连接

上一篇提到，阻塞 IO 逼着我们"一个连接一个线程"，而裸非阻塞 IO 的忙轮询又白白浪费 CPU。理想的方案是：用一个（或少数几个）线程， 把成千上万个 fd 交给内核去"看管"，只有当其中某些 fd 真的有数据可读/可写时，内核才通知应用程序去处理。这就是 IO 多路复用（IO Multiplexing）要做的事。

类比：你在餐厅当服务员，如果每桌客人一喊你就得贴身站着等（阻塞 IO 的思路），一个服务员顶多伺候一桌。 IO 多路复用相当于：所有桌子上装了呼叫器，你站在中控台，呼叫器一响你就知道具体哪几桌需要服务，一个人轻松应付上百桌——多路复用的本质就是这个"呼叫器中控台"。

第一代方案：select

select() 把一批 fd 打包成一个位图（fd_set）传给内核，内核遍历这些 fd，检查哪些已经就绪，然后把结果写回这个位图， select() 返回后，应用程序还要再遍历一遍全部 fd，才能知道到底是哪几个就绪了。

c
fd_set readfds;
FD_ZERO(&readfds);
for (int i = 0; i < n; i++)
    FD_SET(fds[i], &readfds);  // 每次调用都要重新构造整个集合

select(maxfd + 1, &readfds, NULL, NULL, NULL); // 陷入内核，遍历检查

for (int i = 0; i < n; i++)          // 用户态再遍历一遍，才知道谁就绪了
    if (FD_ISSET(fds[i], &readfds))
        /* 处理 fds[i] */;

select 有三个明显的硬伤：

  • fd 数量有上限：fd_set 底层是定长位图，Linux 下通常限制在 1024 个（FD_SETSIZE），扛不住高并发。
  • 每次调用都要把整个 fd 集合从用户态拷贝到内核态，fd 越多，拷贝开销越大。
  • 时间复杂度 O(n)：内核要遍历全部 fd 判断就绪状态，返回后用户态还要再遍历一遍，两次线性扫描。

第二代方案：poll

poll() 用一个 pollfd 结构体数组代替位图，理论上没有 1024 个的数量限制，接口也更清晰（把关注的事件和返回的事件分开存放）。

c
struct pollfd fds[N];
for (int i = 0; i < N; i++) {
    fds[i].fd = sockfds[i];
    fds[i].events = POLLIN;  // 关心"可读"事件
}
poll(fds, N, -1);        // 依然是把整个数组拷进内核，内核依然线性遍历
陷阱：很多人以为 poll 解决了 select 的所有问题，其实 poll 只解决了"数量上限"这一个问题。 每次调用仍然要拷贝整个 fd 数组到内核态，内核仍然要 O(n) 线性遍历所有 fd 才能找出就绪的那几个——当 fd 数量达到几万时， 多数 fd 其实都是空闲的，但每次调用都要把它们全部检查一遍，这是巨大的浪费。select/poll 本质上是同一类方案：都是"轮询式"的多路复用。

第三代方案：epoll（Linux 高并发标配）

epoll 的设计思路完全不同：不再每次调用都全量传入 fd 集合，而是让内核长期持有一份 fd 列表，并且用事件回调机制主动维护一个"就绪列表"。 具体分成三个系统调用：

  • epoll_create()：在内核中创建一个 epoll 实例，返回一个 epoll fd。
  • epoll_ctl()：向这个实例注册/修改/删除要监控的 fd（只需要做一次，不用每次轮询都重新传入）。
  • epoll_wait()：阻塞等待，只返回真正就绪的 fd 列表，不需要应用层再遍历全部 fd 去判断。
c
int epfd = epoll_create1(0);

struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = sockfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev); // 只注册一次

struct epoll_event events[1024];
int n = epoll_wait(epfd, events, 1024, -1);
// events[0..n) 里全是已就绪的 fd，不用再遍历判断，直接处理即可
for (int i = 0; i < n; i++)
    /* 处理 events[i].data.fd */;

epoll 底层用的是事件驱动 + 回调：当某个被监控的 fd 状态变化（比如网卡收到数据触发中断），内核会通过回调把这个 fd 直接放进一个"就绪列表"， epoll_wait() 只需要返回这个列表，而不需要遍历所有被监控的 fd。活跃连接数与总连接数无关——就算你监控了 10 万个 fd， 但只有 10 个是活跃的，epoll_wait() 也只需要处理这 10 个，复杂度趋近 O(1)（准确说是 O(就绪 fd 数)，而不是 O(总 fd 数)）。

select / poll / epoll 对比一览（高频考点）

面试高频："为什么 Redis、Nginx 这类高并发服务都用 epoll？" 核心原因是它们要维护成千上万个长连接，但同一时刻真正活跃（有数据可读写）的连接往往只是很小一部分。 select/poll 每次都要把全部连接扫一遍才能找到这一小撮活跃连接，浪费大量 CPU 在"确认那些没事发生的连接确实没事"上；epoll 通过事件回调机制， 直接精确地告诉你"就是这几个"，天然适合"大量连接、少量活跃"这种典型的互联网服务器场景。

与 Java NIO 的联系

Java 的 java.nio.channels.Selector 就是对多路复用机制的跨平台封装：在 Linux 上，Selector 的底层实现（EPollSelectorProvider） 直接基于 epoll；在早期或其他平台上可能退化为 poll/select 的实现。写 Netty、Tomcat NIO 连接器这类高并发网络框架时，本质上都是在用户层面复用同一套 "一个（或几个）线程 + Selector 轮询 + 事件回调处理"的模型，这正是本篇 epoll 思想在 Java 生态里的直接体现。

核心价值总结

  • select → poll → epoll 的演进主线：从"有数量限制的全量轮询"到"无限制的全量轮询"再到"事件驱动、只返回活跃 fd"。
  • epoll 高效的关键：fd 注册与事件收集分离，配合内核态事件回调，避免了每次调用都要线性扫描全部 fd 的开销。
  • 这套多路复用机制正是下一篇 Socket 编程里 accept 之后能同时处理海量连接的技术基础。
