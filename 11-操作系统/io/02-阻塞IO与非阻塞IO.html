阻塞 IO 与非阻塞 IO

调用 read() 的时候，数据还没到怎么办？是老老实实等着，还是立刻回来告诉你"没数据，你自己再来问"？这是所有 IO 模型讨论的起点。

为什么会有"等待"这件事

上一篇讲到，读写一个 fd 用统一的 read/write。但读数据这件事本身有个天然的时间差：网络数据要经过网卡、内核协议栈才能到达； 磁盘数据要等机械寻道或者 SSD 完成读取；管道另一端的数据要等对方写进来。这个"数据还没准备好"的窗口期该怎么处理，就是阻塞与非阻塞的分歧点。

阻塞 IO：老老实实排队等

阻塞 IO（Blocking IO）是默认行为：调用 read() 时，如果数据还没就绪，这个线程就会被操作系统挂起（放入等待队列，让出 CPU）， 直到数据到达才被唤醒继续执行。整个等待过程中，这个线程什么都干不了。

c
// 阻塞模式下，如果对端还没发数据，这一行代码会一直"卡住"
ssize_t n = read(sockfd, buf, 1024);
// 直到数据到达，read 才返回，程序才往下走
printf("收到 %zd 字节\n", n);
类比：阻塞 IO 就像你去银行柜台排队办业务，前面没人处理完你就干站着，什么也不能干，直到轮到你。CPU 层面对应的是：线程被移出运行队列，完全不占用 CPU 时间片，直到数据就绪被重新唤醒调度。

阻塞 IO 的优点是编程模型简单——代码逻辑跟着数据流走，读到哪算哪，不用操心"数据没来怎么办"。 代价是：如果要同时处理多个连接，就必须每个连接开一个线程，否则一个连接卡住会拖死整个进程。而线程数量一多，上下文切换开销和内存开销（每个线程默认几百 KB～几 MB 的栈）会迅速成为瓶颈。

非阻塞 IO：不断回来问一遍

非阻塞 IO（Non-blocking IO）通过 fcntl(fd, F_SETFL, O_NONBLOCK) 把 fd 设置为非阻塞模式后， read() 不管数据是否就绪都会立刻返回：有数据就返回数据，没数据就返回一个错误码（EWOULDBLOCK / EAGAIN），把选择权交还给调用者。

c
fcntl(sockfd, F_SETFL, O_NONBLOCK);

while (1) {
    ssize_t n = read(sockfd, buf, 1024);
    if (n > 0) {
        // 拿到数据，正常处理
        break;
    } else if (errno == EWOULDBLOCK) {
        // 数据还没到，read 立刻返回而不是挂起线程
        // 但这里啥也没做就直接进入下一轮循环——这就是"轮询"
        continue;
    }
}
陷阱：非阻塞 IO 本身并不能提升性能，如果像上面这样写一个死循环不停调用 read()，这叫忙轮询（busy polling）， 会把一个 CPU 核心的使用率直接打满，只是在反复问"好了没好了没"，而不是在真正做事。这是初学者最容易踩的坑：以为非阻塞就等于高性能，实际上裸用非阻塞反而更浪费 CPU。

阻塞 vs 非阻塞：核心取舍

阻塞 IO非阻塞 IO（裸轮询）
数据未就绪时线程挂起，让出 CPU立即返回错误码，CPU 继续跑
CPU 开销低（等待期间不占 CPU）高（忙轮询会打满 CPU）
并发多连接方式一个连接一个线程一个线程可以轮询多个 fd
编程复杂度简单，顺序编程较复杂，需要处理"未就绪"分支
适用场景连接数少、追求代码简洁连接数多，但需要配合更好的通知机制

非阻塞 IO 真正的价值：为多路复用铺路

既然裸轮询会浪费 CPU，那非阻塞 IO 存在的意义是什么？答案是：它是"IO 多路复用"技术的基础。 如果一个线程要同时盯着 1000 个 fd，用阻塞模式根本无从下手——阻塞在其中任何一个 fd 上，其他 999 个就没法处理了。

非阻塞模式让"检查一个 fd 是否就绪"变成一个不耗时的操作，于是操作系统可以提供一个专门的系统调用，一次性帮你检查一大批 fd 中哪些已经就绪， 而不需要应用层自己写 for 循环挨个轮询。这正是下一篇 select/poll/epoll 要解决的问题： 把"用户态忙轮询"升级为"内核态高效通知"。

面试高频："阻塞 IO 和非阻塞 IO 的区别？" 关键点在于调用 read/write 时，数据未就绪的处理方式不同——阻塞是挂起等待，非阻塞是立即返回错误码。 很多人会把非阻塞 IO 和 IO 多路复用混为一谈，其实非阻塞只是"不等待"，多路复用是"用一次系统调用同时监控多个 fd"，两者不是一回事，但多路复用的实现依赖非阻塞模式。

与其他知识点的联系

  • 呼应 02-并发模块：线程池的设计初衷之一就是避免"一个连接一个线程"在阻塞 IO 模型下带来的线程数爆炸问题。
  • Java 里 java.net.Socket 默认是阻塞模式，而 java.nio.channels.SocketChannel 可以 configureBlocking(false) 切到非阻塞模式，这是 NIO 名字里 "New IO / Non-blocking IO" 的由来。

核心价值总结

  • 阻塞 IO 简单但不适合高并发：连接数一多，线程开销就是瓶颈。
  • 裸非阻塞 IO 不是银弹：忙轮询同样会浪费 CPU，它的真正价值在于为多路复用打基础。
