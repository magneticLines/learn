系统级IO与网络编程

Socket 编程的基本模型

Socket 是应用程序打开"网络通信"这扇门的钥匙——它把复杂的 TCP/IP 协议栈包装成了几个简单的系统调用，是从"单机 IO"迈向"网络 IO"的关键一步。

Socket 到底是什么

回顾第一篇"一切皆文件"的视角：Socket 本质上就是一种特殊的文件描述符，只不过它背后不是磁盘上的文件，而是一段"网络通信端点"—— 一旦创建出来，就可以用统一的 read/write（或者专门的 send/recv）去读写它，数据会经过内核的 TCP/IP 协议栈被送到网络上，或者从网络上收进来。

类比：如果把"打电话"类比网络通信，Socket 就是那部电话机。你不需要懂电话线怎么布线、信号怎么调制解调（这些是传输层/物理层的事，对应 10-计算机网络 模块）， 只需要"拨号（connect）"、"接听（accept）"、"说话（write）"、"听（read）"、"挂机（close）"——Socket API 就是这几个动作的编程接口。

服务端的经典流程

服务端要先"占好位置"等别人连进来，流程是固定的五步：

c
// 1. socket()：创建一个 socket，拿到一个 fd（此时还没和任何地址绑定）
int listenfd = socket(AF_INET, SOCK_STREAM, 0);

// 2. bind()：把这个 fd 绑定到具体的 IP + 端口上，相当于"占号"
struct sockaddr_in addr = { .sin_family = AF_INET, .sin_port = htons(8080) };
bind(listenfd, (struct sockaddr*)&addr, sizeof(addr));

// 3. listen()：把 socket 切换为"监听模式"，开始接受连接请求，排队等待处理
listen(listenfd, 128);  // 128 是全连接队列的最大长度

// 4. accept()：从连接队列里取出一个已完成三次握手的连接，返回一个新 fd
int connfd = accept(listenfd, NULL, NULL);

// 5. read/write：用这个新 fd 与客户端通信，跟操作普通文件毫无区别
char buf[1024];
ssize_t n = read(connfd, buf, sizeof(buf));
write(connfd, "HTTP/1.1 200 OK\r\n\r\n", 19);

close(connfd);   // 关闭这次连接
易混淆点：listenfd 和 connfd 是两个不同的 fd！listenfd 只负责"接待新连接"，永远不会用来读写业务数据； 每来一个新连接，accept() 都会返回一个全新的 connfd 专门用于跟这个客户端通信。一个常见的服务端可以同时持有一个 listenfd 和成千上万个 connfd——这正是上一篇 epoll 要同时监控的那一大批 fd 的来源。

客户端的经典流程

客户端不需要 bind/listen/accept，只需要主动发起连接：

c
// 1. socket()：同样先创建一个 socket
int sockfd = socket(AF_INET, SOCK_STREAM, 0);

// 2. connect()：主动向服务端发起 TCP 三次握手
struct sockaddr_in server_addr = { .sin_family = AF_INET, .sin_port = htons(8080) };
inet_pton(AF_INET, "127.0.0.1", &server_addr.sin_addr);
connect(sockfd, (struct sockaddr*)&server_addr, sizeof(server_addr));

// 3. read/write：连接建立后，同样用统一的读写接口收发数据
write(sockfd, "GET / HTTP/1.1\r\n\r\n", 18);
char buf[1024];
read(sockfd, buf, sizeof(buf));

close(sockfd);

服务端 vs 客户端流程对照

步骤服务端客户端
1socket() 创建 fdsocket() 创建 fd
2bind() 绑定 IP+端口（无需，由内核自动分配临时端口）
3listen() 进入监听状态connect() 主动发起连接
4accept() 取出已建立的连接-
5read/write 通信read/write 通信
6close() 关闭close() 关闭

Socket 是应用层调用传输层服务的接口

在 OSI/TCP 四层模型里，Socket 本身不是一个"协议层"，而是操作系统提供给应用层的一套编程接口（API），让应用程序能够使用下面传输层 （TCP 或 UDP）提供的能力。SOCK_STREAM 对应 TCP（面向连接、可靠传输、字节流），SOCK_DGRAM 对应 UDP（无连接、不保证送达、数据报）。

呼应 10-计算机网络模块：这里的 connect() 触发的正是 TCP 三次握手，close()（如果是主动关闭方）触发的是四次挥手—— Socket 编程只是给这些协议细节包了一层 API，理解了 TCP 状态机，再回头看 Socket 调用会豁然开朗：每一个系统调用背后都对应着协议层面的具体动作。

与前两篇 IO 模型的联系

默认创建出来的 socket 是阻塞模式：accept() 在没有新连接时会挂起线程，read() 在没有数据时也会挂起线程。 如果要支撑高并发连接，通常会把 listenfd 和所有 connfd 设置为非阻塞模式，再配合 epoll 统一监控—— 这正是把本篇讲的 Socket API 和前两篇讲的 IO 模型拼接起来的地方：Socket 提供"是什么"（通信端点），IO 模型决定"怎么等"（阻塞/非阻塞/多路复用）。

常见误区：初学者常把 Socket 和 TCP 划等号，其实 Socket 只是接口，TCP/UDP 才是背后真正干活的协议。同一套 socket()/read()/write() 调用形式，通过参数 SOCK_STREAM 还是 SOCK_DGRAM 就能选择完全不同的传输层协议，这也是"统一接口、不同实现"思想的又一体现。

核心价值总结

  • 服务端五步：socket → bind → listen → accept → read/write，客户端三步：socket → connect → read/write。
  • Socket 是"网络版的文件描述符"，把网络通信统一到了和文件、管道一样的读写接口下。
  • Socket 决定"和谁通信"，IO 模型决定"如何高效地等待和处理"——两者结合才是完整的高并发网络服务器。
