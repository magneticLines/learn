Unix IO 的统一视角：一切皆文件

普通文件、目录、终端、磁盘设备、管道、socket——底层千差万别的资源，为什么在 Unix 里都能用同一套 open/read/write/close 来操作？这是操作系统设计里最优雅的抽象之一。

先看一个反直觉的事实

如果你写过 C 语言的文件操作，会发现一件很神奇的事：读一个磁盘文件和读一个网络连接，用的居然是同一个函数 read()； 往终端打印字符和往磁盘写数据，用的也是同一个函数 write()。

c
int fd1 = open("/tmp/data.txt", O_RDONLY);   // 打开一个磁盘文件
int fd2 = socket(AF_INET, SOCK_STREAM, 0); // 建立一个 TCP socket
int fd3 = open("/dev/tty", O_WRONLY);     // 打开终端设备

// 三种完全不同的底层资源，读写方式却一模一样
read(fd1, buf, 128);
read(fd2, buf, 128);
write(fd3, "hello\n", 6);

这不是巧合，而是 Unix 从设计之初就贯彻的一条哲学：Everything is a file（一切皆文件）。

为什么要设计成"一切皆文件"

操作系统要管理的 IO 设备五花八门：硬盘、网卡、键盘、鼠标、打印机……如果每一种设备都设计一套专属的 API（比如 disk_read()、net_read()、keyboard_read()）， 会带来两个致命问题：

  • 应用层代码没法复用：写一段处理数据的逻辑，换个数据来源（从文件读改成从网络读）就要重写整套调用代码。
  • 操作系统的复杂度会指数级上升：每新增一种设备类型，就要给所有上层工具（shell、编辑器、编译器……)重新适配一遍。
类比：这就好比现实世界里的"插座标准化"。不管你家电器是电视、冰箱还是台灯，只要插头是国标两孔/三孔，就能插进同一个插座里获取电力——你不需要为每种电器发明新的供电接口。 Unix 把这套思路用在了 IO 上：不管底层是磁盘、网卡还是管道，只要它愿意实现"读"和"写"这两个基本动作，就能套进同一套接口里。

这个抽象的核心载体就是文件描述符（file descriptor，简称 fd）——一个小整数，是进程与某个"可读写资源"之间的句柄。 内核在背后维护一张表，把这个整数映射到真正的资源（可能是磁盘上的一个 inode，也可能是一个 socket 的通信缓冲区，也可能是一个管道）。 应用层完全不关心 fd 背后具体是什么，只管调用统一的四个动作：open（获取 fd）、read（读）、write（写）、close（释放 fd）。

被这套抽象统一起来的资源

资源类型典型例子获取 fd 的方式
普通文件磁盘上的 .txt、.logopen("/path/file")
目录文件系统目录本身也是一种特殊文件open("/path/dir")
字符/块设备终端 /dev/tty、磁盘 /dev/sdaopen("/dev/...")
管道Shell 里的 ls | grep xxxpipe()
SocketTCP/UDP 网络连接socket()

正因为这套统一，Shell 里那句耳熟能详的 command > output.txt 才能成立——重定向的本质，就是把标准输出这个 fd（默认指向终端） 换成指向一个磁盘文件的 fd，而 command 本身的代码完全不用感知这个变化，它只管往 write(1, ...) 里写数据。

这套抽象也不是万能的

常见误区："一切皆文件"不等于"所有资源的行为完全一样"。比如磁盘文件支持随机访问（lseek 跳到任意偏移量）， 而 socket、管道是面向流的，不支持 lseek；socket 还需要额外的 connect/bind/listen/accept 来建立连接语义，这是纯文件操作没有的部分。 统一的只是"最基本的读写接口"，不同资源在此之上仍然保留了各自的特化操作。

与其他知识点的联系

  • Java NIO 的 Channel 抽象正是对这套统一视角的直接映射：FileChannel、SocketChannel 底层都对应一个 fd，Java 只是把它包装成了面向对象的接口。
  • 后面讲阻塞 IO / 非阻塞 IO、IO 多路复用时，讨论的"fd 是否就绪"，正是建立在本篇这套 fd 抽象之上——没有统一的 fd，就没法用同一个 select/epoll 同时监控文件和 socket。
  • 呼应 10-计算机网络 模块：Socket 本质上就是"传输层协议的 fd 化包装"，下一篇 Socket 编程会展开讲。

核心价值总结

  • 统一接口带来代码复用：同一段读写逻辑可以无缝切换数据源，这是 Unix 管道、重定向等强大特性的根基。
  • fd 是这套抽象的核心载体：一个小整数，屏蔽了背后千差万别的资源实现细节。
