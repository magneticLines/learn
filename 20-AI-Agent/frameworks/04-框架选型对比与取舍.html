框架选型对比与取舍

看完 LangChain/LangGraph、AutoGen/CrewAI、smolagents 三种不同流派之后,真正要回答的问题是:什么时候该用框架,什么时候该自己手写?

汇总:各方案的定位

方案学习曲线灵活度多 Agent 支持社区生态适合场景
手写最小 ReAct 循环 低(原理简单,代码量小) 极高(完全掌控每一行逻辑) 需要自己实现 无(没有现成组件可复用) 学习阶段理解原理;需求简单、追求极致可控和轻量的生产场景
LangChain / LangGraph 中~高(概念多,LangGraph 需要理解状态图) 中高(状态图可以表达复杂分支) 支持(可搭配多节点/多图协作) 非常成熟(生态最大,集成最多) 需要复杂分支/循环逻辑、希望复用大量现成工具集成的团队
AutoGen 较高(需自行设计对话拓扑) 高(消息驱动,拓扑自由) 核心卖点 成熟(微软背书) 协作模式复杂多变、需要 Agent 间自由协商的场景
CrewAI 低(角色+任务模板化) 中(约定了角色分工的固定模式) 核心卖点 成长中 分工明确、流程相对固定的团队式任务,追求快速上手
smolagents 低~中(需理解 Code Agent 心智模型) 高(直接写代码表达任意逻辑) 非核心场景 成长中(HuggingFace 背书) 单 Agent 需要处理较复杂的多步逻辑,且能接受沙箱执行成本

核心判断原则:框架的价值和代价

所有框架本质上都在做同一件事:把某种重复出现的工程模式提前封装好—— Prompt 拼接、工具调用循环、多 Agent 消息路由、状态管理……这些代码在足够多的项目里重复出现, 才有了被抽象成框架的价值。所以框架能带来的收益是明确的:省去重复造轮子的工程量。

但代价同样明确:多一层抽象,就多一份学习成本,出问题时也更难排查。 当 Agent 的行为不符合预期时,你需要先搞清楚"是我的 Prompt 写错了,还是框架内部某个默认参数、 某层封装的行为跟我预期的不一样",这个排查过程往往比读自己写的代码耗时得多。

更扎实的路径:对于学习阶段,或者需求足够简单的场景,建议先按照"实战演练"Tab 里的思路 (参见 ../practice/01-Java手写最小Agent.html)手写一遍最小 ReAct 循环—— 自己实现一次"推理 → 决定调用工具 → 观察结果 → 继续推理"的完整循环,才能真正理解 Agent 的运行机制, 而不是停留在"调用框架的 API,黑盒地跑起来了"这个层面。理解原理之后,再回头评估要不要引入框架, 判断会更准确,也更清楚自己在用框架换取什么、放弃什么。

陷阱:不要为了用框架而用框架

"重框架综合征"是真实存在的工程陷阱。不少团队看到 LangChain、AutoGen 这类热门项目, 出于"业界都在用"、"显得更专业"的心理直接引入到生产项目里,结果做完之后发现: 项目实际用到的能力,可能只是框架提供的 90% 功能里的一小部分,剩下的复杂抽象反而成了团队所有人都要学习、 都要绕过的负担——升级框架版本可能引发不兼容,排查一个奇怪的 Bug 可能要先读懂框架源码。 选型前先问自己:这个需求,不用框架、手写 100~200 行代码能不能搞定? 如果答案是"能",那么框架带来的"省事"可能远不如它带来的"负担"。

小结

  • 本模块梳理了 LangChain/LangGraph(状态图编排)、AutoGen/CrewAI(多 Agent 协作)、smolagents(Code Agent)三条主流路线,以及手写最小实现这个基准线。
  • 选型没有唯一正确答案,取决于任务复杂度、团队规模、对可控性和排查效率的要求。
  • 核心原则:框架换来的是"省去重复造轮子",代价是"多一层抽象、多一份学习和排查成本"——先理解原理,再决定要不要引入框架,是更扎实的路径。
