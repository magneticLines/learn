框架生态

AutoGen / CrewAI

单个 Agent 能力有上限,把多个各有分工的 Agent 组织起来协作完成任务,是近两年最热的方向之一。AutoGen 和 CrewAI 是这条赛道上两种不同的心智模型。

为什么需要多 Agent

一个通用型 Agent 想同时擅长"写代码"、"查资料"、"审核质量"、"总结报告",往往每个能力都做得不够精。 更常见的做法是拆分角色——让一个 Agent 专注写代码、一个专注做 Code Review、一个专注整合结果, 每个 Agent 的 Prompt 和工具集都更聚焦、更容易调优,这正是"Agent 开发核心"里"多 Agent 协作模式"讨论的场景。 AutoGen 和 CrewAI 就是把这套协作方式框架化的两个代表。

AutoGen:用"对话"驱动协作

AutoGen(微软出品)的核心抽象是Conversation(对话)——每个 Agent 都是对话里的一方, Agent 之间通过互相发消息来协作,谁该在什么时候发言、协作的拓扑结构(一对一、群聊、层级式)都可以灵活自定义。 这种设计的好处是足够通用:无论是"两个 Agent 互相辩论找出更优方案",还是"多个 Agent 群聊分工完成任务", 都能用同一套对话机制表达。

CrewAI:用"团队角色分工"驱动协作

CrewAI 走的是另一条路:它把协作建模成组建一个团队(Crew)——给每个 Agent 明确设定 Role(角色)、Goal(目标)、Backstory(背景设定), 就像招聘时写清楚这个岗位是做什么的、期望达成什么;然后定义一系列 Task(任务), 按顺序或依赖关系分派给团队里合适的 Agent 去执行。这种心智模型更贴近"管理一个人类团队"的直觉,上手门槛相对更低。

AutoGen vs CrewAI

维度AutoGenCrewAI
核心抽象 Conversation(消息驱动的对话) Role + Task(角色分工 + 任务分派)
适合场景 协作拓扑复杂多变、需要 Agent 之间自由辩论/协商的场景 分工明确、流程相对固定的团队式任务(比如"调研 → 写作 → 审校"流水线)
学习曲线 较陡——需要自己设计对话拓扑和终止条件 较平缓——照着"角色 + 任务"模板填空即可跑起来
类比:AutoGen 更像一个开放式会议室,谁都能随时发言,协作规则要自己定; CrewAI 更像一家职责分明的小公司,每个岗位说明书写得清清楚楚,任务按流程往下传。

陷阱:演示效果 ≠ 生产可用性

多 Agent 框架有一个共同的风险:Demo 效果往往比生产可用性更亮眼。 在演示视频里,几个 Agent 一唱一和配合得天衣无缝;但协作流程一旦跑偏——比如某个 Agent 理解错了上一个 Agent 的输出、 在某一轮对话里"说错了话"导致后续全部跑歪——排查起来比单 Agent 系统困难得多: 到底是哪个 Agent、在哪一步、基于什么样的上下文做出了错误判断,往往需要把整条消息链路完整回放才能定位。 引入多 Agent 协作前,要先确认单 Agent + 良好的工具设计是否已经能满足需求。

小结

  • 多 Agent 协作的核心动机是"分工聚焦",而不是"越多 Agent 越强"。
  • AutoGen 用消息对话驱动协作,灵活但需要自己设计拓扑;CrewAI 用角色分工驱动协作,心智模型更直观但灵活度较低。
  • 多 Agent 系统的调试成本远高于单 Agent,生产落地前要评估协作链路的可观测性。
