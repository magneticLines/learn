smolagents(HuggingFace)

如果让模型直接写一段 Python 代码来"行动",而不是拼一份 JSON 参数,会怎样?smolagents 把这个朴素的想法做成了一个轻量框架。

核心理念:Code Agent

主流的 Function Calling 方式,是让模型输出一份结构化的 JSON,描述"要调用哪个函数、传什么参数", 再由外部程序解析这份 JSON、真正执行函数调用。这种方式规整、易于校验,但也有局限: 遇到"先查询结果,再根据结果做一次循环处理,最后汇总"这类稍微复杂一点的逻辑, 纯粹靠一次次 JSON 参数拼接会显得笨拙,经常需要好几轮工具调用来回折腾。

smolagents 的设计理念是反过来想:LLM 的训练数据里包含海量的代码(GitHub、Stack Overflow、技术文档), 模型对"写代码"这件事的熟练程度,天然比"按照某个特定 JSON Schema 拼参数"更高。 于是 smolagents 让模型直接输出一段可执行的 Python 代码作为它的"行动"—— 要调用工具就在代码里调用对应的函数,要做循环、条件判断、字符串处理,直接写 Python 逻辑即可, 这种模式被称为 Code Agent。

传统 Function Calling(JSON)json
{
  "tool_calls": [
    {
      "name": "search_web",
      "arguments": { "query": "北京今天天气" }
    }
  ]
}
// 需要再等一轮模型输出,才能对搜索结果做进一步处理
// 比如"从结果里提取温度,再判断是否需要提醒带伞"
// 往往要再触发一次 JSON 格式的工具调用
smolagents 风格(直接写代码)python
# 模型一次性生成一段完整逻辑,查询 + 处理 + 判断都在这段代码里完成
result = search_web("北京今天天气")
temperature = extract_temperature(result)

if "雨" in result or temperature < 10:
    final_answer("建议带伞,今天有雨或气温较低")
else:
    final_answer("天气不错,不用带伞")

可以看到,同样是"查天气 + 判断是否带伞",Code Agent 一段代码就能表达完整逻辑,不需要在"模型输出 → 外部解析 → 再问模型"之间来回折腾好几轮, 调用次数更少,出错率也往往更低——因为模型是在"写它最擅长的东西"。

陷阱:执行模型生成的代码,本质是在执行不受信任的代码

安全风险是这套方案绕不开的代价。模型生成的代码有可能因为幻觉、被恶意 Prompt 注入, 或者单纯"理解错了任务"而包含危险操作——比如删除文件、访问不该访问的网络地址、执行系统命令。 如果直接在生产环境的主进程里 exec() 这段代码,后果可能是灾难性的。

smolagents 的做法是依赖沙箱执行环境把这种风险隔离开——把代码放到一个资源和权限都受限的隔离环境里执行 (比如受限的 Python 解释器、独立的容器),即使生成的代码想干坏事,影响范围也被限制在沙箱内部,不会波及宿主系统。

预告:"代码执行 + 沙箱隔离"这套组合拳不是 smolagents 独有的,后面拆解开源项目 OpenHands 时会看到, 它同样需要面对"AI 生成的代码不能直接在宿主机上跑"这个问题,给出的方案是用 Docker 沙箱做隔离,思路是相通的。

小结

  • smolagents 用"让模型写代码"替代"让模型拼 JSON 参数",利用了 LLM 对代码天然更熟练这一特点。
  • Code Agent 能把多步逻辑压缩进一段代码里执行,减少了工具调用的轮次。
  • 执行模型生成的代码存在安全风险,必须依赖沙箱环境隔离,这是"代码即行动"模式无法回避的工程代价。

与其他知识点的联系

  • Java 手写最小 Agent(实战演练 Tab):手写实现里用正则匹配 Action: 工具名(参数) 这种脆弱的文本协议, 恰好是 smolagents 想绕开的问题——Code Agent 直接让模型输出可执行代码,不需要再设计一套"怎么从自然语言里可靠解析出结构化指令"的协议, 对比两种方式能直观体会"文本协议解析"和"直接执行代码"在可靠性上的差异。
  • Python 框架实战(本模块实战演练 Tab 02):那一篇用真实的 CodeAgent + @tool 跑了一遍穿衣建议的例子, 是这一篇"核心理念"的实操验证——读完这一篇的原理再看那份可运行代码,能看清 docstring 里的 Args 段是怎么被框架解析成 喂给模型的参数说明的。
  • Docker 沙箱与 Runtime 隔离设计(OpenHands 源码拆解 Tab):这一篇说的"执行模型生成的代码本质是执行不受信任的代码", 和 OpenHands 面临的"AI 生成的命令/代码不能直接在宿主机上跑"是同一个问题,只是 smolagents 给出的是语言层沙箱方案, OpenHands 给出的是容器级隔离方案——隔离粒度不同,但解决的都是"不能信任模型生成内容"这一核心风险。
  • 框架选型对比与取舍(本 Tab 04):04 篇的选型表格把 smolagents 定位为"单 Agent 需要处理较复杂多步逻辑、且能接受沙箱执行成本" 的场景专用方案——这是因为它的核心卖点(代码表达力)和核心代价(必须付出沙箱隔离的工程成本)是一体两面的,选型时要一起权衡,不能只看收益。
  • 责任链模式与解释器模式:传统 Function Calling 是"模型产出指令 → 外部代码解释执行"的两段式流程,类似解释器模式; Code Agent 则让模型直接产出可执行逻辑本身,减少了"外部代码逐条翻译指令"这一层——这也是为什么它能一次表达"查询+循环+条件判断" 这种多步逻辑,而不需要靠多轮"指令-解析-执行"往返来拼出同样的效果。
