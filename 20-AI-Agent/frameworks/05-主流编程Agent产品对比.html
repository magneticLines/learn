框架生态

主流编程 Agent 产品对比:Claude Code / Codex / OpenCode / Hermes

前面几篇讲的 LangChain、AutoGen、smolagents 是"给开发者拼代码用的库";这一篇讲的四个是"打开就能用的终端 Agent 产品"——理解它们的设计取舍,能直接指导你选工具,也能给自己写 Agent 提供参考。

为什么框架和产品要分开看

框架(LangChain/LangGraph/smolagents)解决的问题是"我要自己开发一个 Agent 应用,用什么积木"; 而 Claude Code、Codex CLI、OpenCode、Hermes Agent 这类工具解决的是另一个问题——直接把一个已经做好的编程 Agent 接到你的终端里,当天就能用来写代码、跑测试、改 bug。 两者面向的用户不同,但架构设计上的取舍高度相关:这四个产品各自在"安全边界怎么划""自动化程度怎么控""怎么和真实开发环境结合"这几个问题上, 给出了不同但都经过真实生产验证的答案,值得对照着看。

Claude Code:分层安全 + 可委派的子代理

Claude Code 的核心是一个很朴素的循环——不断调用模型、执行工具、再把结果喂回去,和"Agent 开发核心"Tab 讲的 ReAct 循环本质相同。 它真正的复杂度都在这个循环外围的护栏系统上:

  • 分层安全,而不是单一开关。权限规则、PreToolUse 钩子、可选的自动模式分类器、shell 沙箱这几层是并行生效的独立机制, 任何一层都可以拦下一次危险操作——这是"深度防御(defense in depth)"思路在产品设计里的直接体现,和 OpenHands 的多层防护是同一个道理。
  • 子代理(Subagents)拥有独立的上下文和工具权限。可以给一个子代理只开放 Read/Grep/Glob 这类只读工具, 让它专门做"代码审查"而无法修改任何文件——这是"多 Agent 协作"Tab 里"角色分工"思路在真实产品里的具体实现:用工具权限而不是靠"口头约定"来约束一个子代理的能力边界。
  • 可扩展性靠四层机制组合:MCP(连接外部系统)、Plugins(打包分发可复用组件)、Skills(注入领域知识)、Hooks(在生命周期节点插入自定义逻辑)—— 各自职责单一,组合起来覆盖了"连接什么、复用什么、教它什么、什么时候拦截"四个不同维度的扩展需求。

Codex CLI:把"能做什么"和"要不要问"分成两套机制

OpenAI 的 Codex CLI(开源,Apache 2.0)最值得学的设计,是把沙箱(sandbox)和审批策略(approval policy) 做成两个独立的控制维度,而不是混成一个"允许/不允许"的开关:

下面两组枚举取自 openai/codex 仓库真实的 Rust 源码(codex-rs/protocol/src/config_types.rs),而不是转述:

config_types.rsrust · 节选自真实源码
// 沙箱模式:定义"能力上限"——Agent 技术上能碰到什么
pub enum SandboxMode {
    ReadOnly,          // "read-only",默认值:只能读,不能写、不能联网
    WorkspaceWrite,    // "workspace-write":可以在工作目录内读写、执行常规命令
    DangerFullAccess,  // "danger-full-access":不设限制,仅在明确需要时使用
}

// 审批策略:定义"流程闸门"——技术上允许,但要不要先停下来问一声
type AskForApproval = "untrusted" | "on-request" | "never"
    | { granular: { sandbox_approval, rules, skill_approval, request_permissions, mcp_elicitations } };

真实的默认沙箱模式是 read-only(比很多资料里说的"默认 workspace-write"更保守), 命令执行默认禁用网络访问、文件操作被限定在当前目录树内——这和"OpenHands 拆解"Tab 讲的 风险分级确认机制(SecurityRisk + ConfirmationPolicy)是同一类设计思路的不同实现:不是让 Agent 自证清白, 而是无论 Agent 怎么决策,外层都有一道它绕不过去的技术闸门。

一个很值得记住的细节:审批请求默认路由给谁审批,也是可配置的——ApprovalsReviewer 除了默认的 User(人工审批),还支持 AutoReview:官方注释原文是"uses a carefully prompted subagent to gather relevant context and apply a risk-based decision framework before approving or denying the request"(用一个精心设计 Prompt 的子代理, 收集上下文并套用风险决策框架,来决定批准还是拒绝)。也就是说连"要不要批准这个危险操作"这件事,都可以交给另一个 Agent 去做风险评估—— 这是"用 AI 审查 AI"这个模式在真实生产系统里的具体落地,和"如何使用 Agent"Tab 里"让另一个视角复查"的建议是同一个思路。
架构演进也值得一看:Codex CLI 的实现语言已经从 TypeScript 迁移成了 Rust(仓库里的真实目录就叫 codex-rs), 原因是 Rust 版本没有运行时 GC 停顿、性能更可控,同时定义了一套"线协议(wire protocol)"允许用不同语言扩展 Agent 行为—— 这和 OpenHands 用标准化 Action/Observation 协议解耦 Agent 与 Runtime 是同一个道理:协议标准化之后,底层实现语言反而成了可以自由替换的细节。

OpenCode:客户端/服务端分离 + 编译器真实反馈闭环

OpenCode 在架构上最有特点的选择,是采用客户端/服务端(client/server)模式——Agent 的执行进程可以跑在一台远程开发服务器上, 终端 UI、桌面客户端等多种前端连上去操作同一个后端会话。这解决了一个很实际的问题:笔记本电脑关机、网络断开,不应该杀死一个正在长时间运行的编程任务。

另一个值得借鉴的设计是 Build / Plan 双模式:build 模式拥有完整读写权限,plan 模式只读、 执行命令前必须先申请权限,两者可以用一个快捷键随时切换——探索一个陌生代码库或者规划一次重构时用 plan 模式零风险, 确认方案后再切到 build 模式真正动手。这和 Claude Code 的分层权限、Codex 的沙箱模式解决的是同一类问题, 但 OpenCode 选择了"给用户一个显式的模式开关"这种更直观的呈现方式。

OpenCode 还把 LSP(Language Server Protocol) 接入了 Agent 的反馈闭环——每次编辑代码后, 真实的编译器诊断信息(类型错误、未使用的导入、函数签名不匹配)会被喂回给模型,让它在"报告任务完成"之前就有机会自我纠正, 而不是等用户运行代码才发现问题。这是一个很值得记住的模式:让 Agent 的 Observation 来自真实、结构化的工具输出(编译器/类型检查器), 比单纯让模型"自己检查一遍"可靠得多。

Hermes Agent:把"记忆"当作一等设计目标

Nous Research 的 Hermes Agent 采用三层架构(用户界面层 / 核心 Agent 层 / 工具与执行层),这个划分本身和 OpenHands 的 Agent/Workspace 解耦异曲同工。它真正的差异化特色在于把长期记忆和技能沉淀当作核心卖点,而不是事后补丁:

  • 持久学习循环:每完成一次任务,系统会提炼出一个可复用的"技能"(skill),下次遇到类似任务直接复用并继续打磨—— 这比"Agent 开发核心"Tab 讲的通用长期记忆(向量库检索历史)更进一步,是把"记住做过什么"升级成了"记住怎么做得更好"。
  • 隔离子代理做委派:可以把某个子任务派发给一个拥有独立对话、独立终端的子代理去跑,主线程不需要承担这部分上下文成本—— 是"多 Agent 协作"里"分工不分心"思路的具体落地。
  • 五种执行后端:本地、Docker、SSH、Singularity(常见于科研计算集群)、Modal(云端函数计算),覆盖了从"本机随手跑"到"云端大规模并行"的完整光谱, 印证了"Agent 与执行环境解耦"这个架构决策的价值——同一套 Agent 逻辑,换个后端配置就能从笔记本切到集群。
一个独特的定位:Hermes Agent 同时也是一个用来生成训练数据、做强化学习实验的平台——它能并行生成成千上万条工具调用轨迹用于模型微调。 这提醒我们:一个设计良好的 Agent 执行框架(标准化的 Action/Observation、可复现的沙箱环境),不仅能用来"跑任务", 还天然适合用作"给下一代模型生产训练数据"的基础设施,这是 Agent 系统架构价值的一个容易被忽视的延伸。

开发者可以借鉴的设计模式

抛开具体产品,这四者共同验证了几个在自己动手做 Agent 时值得直接复用的模式:

  • 安全控制要分层,不要指望一道闸门解决所有问题——技术沙箱(能力上限)和流程审批(要不要问人)是两件独立的事,分开设计更清晰也更难被绕过。
  • 用"工具权限"而不是"提示词约定"来约束子代理——给子代理限定只读工具集,比在 Prompt 里写"请不要修改文件"可靠得多。
  • 让反馈来自真实、结构化的工具输出——OpenCode 的 LSP 集成证明,编译器诊断这类"权威数据源"喂给模型,比让模型自我检查更能减少幻觉和返工。
  • 执行环境要能替换,Agent 逻辑不用跟着变——Hermes 的多后端、Codex 的沙箱分层、OpenCode 的客户端/服务端分离,都是同一条原则的不同实现:决策逻辑和执行环境之间隔一层可替换的抽象。

这些模式和"OpenHands 源码拆解"Tab 里总结的架构取舍高度一致——说明它们不是某一个项目的孤立选择, 而是"工业级 Agent 系统"这个问题空间里反复被验证有效的通用答案。
