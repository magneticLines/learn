框架生态

LangChain / LangGraph

LangChain 是最早流行的 LLM 应用编排框架,LangGraph 是它面向复杂 Agent 场景的后续演进——把"循环该怎么转"从隐式约定,变成显式的一张图。

LangChain:把 LLM 调用串成流水线

在 LangChain 出现之前,开发者调用 LLM 大多是"拼字符串 + 发 HTTP 请求 + 手动解析返回文本"这种糙活。LangChain 把这套流程抽象成几个可复用的组件: Prompt 模板(把变量填进预先设计好的提示词结构)、模型调用(统一不同厂商 LLM 的调用接口)、 输出解析器(把模型返回的自由文本解析成结构化数据)。把这三者用管道符串起来,就是 LangChain 最核心的抽象——Chain。

在 Chain 的基础上,LangChain 又加入了 Agent 概念:不再是一条写死的流水线,而是让 LLM 自己决定"这一步该调用哪个工具", 再把工具返回结果喂回模型,形成一个调用循环。这本质上就是"什么是 Agent"一节里讲的 ReAct 循环的一种具体实现。

LangGraph:把循环显式画成一张图

纯 Chain/Agent 模式有一个天然的局限:它的执行流程是"隐式"的——循环什么时候停、遇到分支该走哪条路, 都被塞进框架内部一段不太透明的调度逻辑里。一旦任务变复杂(比如"先做数据校验,校验失败要重试,重试三次还失败就转人工"), 这种隐式循环就很难表达清楚,调试也无从下手。

LangGraph 的解法是把 Agent 的执行流程显式建模成一张状态图(State Graph):

图的要素对应的含义
节点(Node)一个具体的处理步骤,比如"调用 LLM 推理"、"执行某个工具"、"人工审核"
边(Edge)节点之间的转移路径,可以是固定的,也可以是"条件边"——根据上一步的输出动态决定走向哪个节点
状态(State)在图中流转的一份共享数据,每个节点读取状态、处理后再更新状态,类似"传送带上的一个工单"

这样一来,"循环"、"分支"、"重试"都变成了图上可以画出来、可以打日志追踪的显式结构,而不是埋在框架内部的黑盒调度。

下面是 LangGraph 官方文档中的真实 API 用法(不是伪代码,可以直接安装 langgraph 运行):

state.pypython · 状态定义
from langchain.messages import AnyMessage
from typing_extensions import TypedDict, Annotated
import operator

# State 是流转在图上的那份共享数据;Annotated + operator.add 告诉 LangGraph:
# 每个节点返回新消息时是"追加"到 messages 列表,而不是整体替换掉
class MessagesState(TypedDict):
    messages: Annotated[list[AnyMessage], operator.add]
    llm_calls: int
graph.pypython · 组装图
from langgraph.graph import StateGraph, START, END

# add_node 注册处理函数(llm_call/tool_node 的具体实现略,分别对应"调用模型"和"执行工具")
agent_builder = StateGraph(MessagesState)
agent_builder.add_node("llm_call", llm_call)
agent_builder.add_node("tool_node", tool_node)

# START/END 是 LangGraph 内置的两个特殊节点,标记图的入口和出口
agent_builder.add_edge(START, "llm_call")

# add_conditional_edges: should_continue 返回值决定走向 tool_node 还是直接结束(END)
# 这就是 ReAct 循环里"要不要继续调用工具"这个判断的显式图结构
agent_builder.add_conditional_edges(
    "llm_call",
    should_continue,
    ["tool_node", END]
)
agent_builder.add_edge("tool_node", "llm_call")  # 工具执行完,回到 llm_call 继续推理——循环由这条边构成

agent = agent_builder.compile()

from langchain.messages import HumanMessage
messages = [HumanMessage(content="帮我查一下北京天气")]
result = agent.invoke({"messages": messages})
for m in result["messages"]:
    m.pretty_print()
来源说明:以上代码取自 LangGraph 官方 Quickstart 文档的真实示例(截至撰写时的 API), llm_call、tool_node、should_continue 三个函数的具体实现在官方教程里有完整版本, 这里只保留图结构的搭建部分——重点是让你看清"节点/边/条件边"这三个要素在真实代码里长什么样,而不是抄一份能跑的完整脚本。 实际使用前建议对照 官方文档 确认最新 API。
和前面内容的联系:可以看到 llm_call → tool_node → 回到 llm_call 这条边, 就是"Agent 开发核心"Tab 里 ReAct 循环的显式结构化版本——LangGraph 并没有发明新的原理,而是把原本靠代码隐式控制的循环, 变成了一份可以画出来、可以单独测试每个节点的图结构。

生产落地时会用到的能力:持久化与流式输出

上面的最小示例每次 invoke() 都是一次性跑完,状态不会保留。真实产品里的对话通常要支持"用户中途断开、下次回来接着聊", LangGraph 对此提供了 Checkpointer(检查点)机制——给 compile(checkpointer=...) 传入一个存储后端 (内存、SQLite、Postgres 均有官方适配),图在每个节点执行完都会自动把当前 State 存一份快照,下次用同一个 thread_id 调用时会自动从上次的状态继续,而不需要自己维护"这个用户聊到哪一步了"这份状态。

另外,示例里的 agent.invoke(...) 是等整个图跑完才拿到结果,真实产品体验通常需要"模型一边生成一边显示", 对应官方提供的 agent.stream(...) / astream(...) 接口,可以流式拿到每个节点产生的中间消息—— 这也是为什么"手写最小 Agent"一节的收尾会提到框架内置了"流式输出支持"这件事,在 LangGraph 里就是这一组 API。

陷阱:不要为了"看起来专业"而引入

过度封装的历史包袱:LangChain 早期版本因为抽象层级过多(一个简单的 Prompt 拼接都要套好几层类)、 API 在大版本之间频繁大改,被社区诟病"学习成本高、debug 困难、升级即断裂"。 引入这类框架前值得先问自己:团队真的需要它提供的编排能力(比如复杂的多分支状态图、丰富的官方工具集成), 还是手写一个几十行的 ReAct 循环(见"实战演练"Tab)就完全够用、还更可控、更容易排查问题? 框架省下的是"重复造轮子"的工程量,代价是多一层抽象、多一份学习成本。

小结

  • LangChain 的核心抽象是 Chain(流水线)和 Agent(带工具调用循环的 Chain)。
  • LangGraph 用"状态图"显式建模 Agent 的执行流程,解决了复杂分支/循环在纯 Chain 模式下难以表达和调试的问题。
  • 状态图的节点、边、状态,本质上是把 ReAct 循环这一套抽象概念工程化落地。
  • 引入框架前先评估必要性,避免为了"看起来专业"而承担不必要的抽象成本。
