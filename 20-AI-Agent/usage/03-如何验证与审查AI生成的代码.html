如何验证与审查 AI 生成的代码

"能跑起来"不等于"写对了"。AI 生成代码有它特有的错误模式,审查时如果沿用审查人类代码的老习惯,很容易漏掉真正的问题。

AI 生成代码的错误模式,和人写的不一样

人写错代码,常见原因是"没想到某个边界情况"或者"手滑打错了字符";AI 生成代码出错,更常见的模式是另外几种, 审查时需要专门留意:

  • 一本正经地编造不存在的 API——调用一个听起来很合理、但项目里根本不存在的方法或库函数,如果不实际编译/运行,光看代码逻辑很容易被"骗过去"。
  • 看起来完整,实际上遗漏了边界情况——比如实现一个函数只覆盖了"正常输入"路径,空值、超大输入、并发调用这些边界情况完全没考虑,但代码风格工整、注释齐全,视觉上"很像对的"。
  • 过度设计——引入了任务根本用不到的抽象层、配置项、设计模式,原因是训练数据里"写得规范"的代码往往带着这些模式,模型学会了"看起来专业"但不一定"恰到好处"(这也是"软件设计的哲学"模块反复强调的复杂性下沉问题)。
  • 复制粘贴式的重复逻辑——在多个地方写相似但不完全一致的代码,而不是提炼成一个共享函数,因为模型倾向于"就着当前上下文续写",而不是主动做全局重构。
最容易踩的心理陷阱:因为代码格式规范、注释齐全,很多人会不自觉地降低审查标准,潜意识觉得"AI 写的应该没问题"。 实际上恰恰相反——AI 生成代码的"专业外观"和"实际正确性"之间的相关性,比人写的代码更弱,审查时需要刻意提醒自己不要被表面的工整所迷惑。

三层验证关卡

层级关注什么
①自动化检查 能跑测试就跑测试,能做类型检查就做类型检查——这是最便宜、最客观的一层关卡,呼应"任务描述"一篇讲的"验收标准要客观可执行"。 没有测试覆盖的改动,至少手动跑一遍验证核心路径。
②Diff 审查 像审查同事的 PR 一样逐行看 diff,重点关注上面列的四种 AI 特有错误模式,而不是只看"逻辑通不通"。 小步提交(每次只做一件事)能让这一步的审查负担小很多——这也是为什么"Agentic Loop"一篇建议把大任务拆小的另一个理由。
③让另一个视角复查 让另一个 Agent(甚至换一个不同的模型)专门做一次"只读审查",或者用不同的 Prompt 角度重新审视同一份代码—— 这是"多 Agent 协作"Tab 里"辩论式"协作模式的实战应用:同一个模型很难发现自己犯的错误,换一个独立视角复查往往更有效。

审查时值得追问的几个问题

  • 这段代码处理了哪些边界情况?哪些没处理——是有意为之,还是被遗漏了?
  • 这个改动是不是引入了任务本身不需要的抽象或配置项?
  • 项目里有没有类似逻辑已经存在,这次是不是应该复用而不是重新写一遍?
  • 调用的这些方法/库函数,是不是真实存在于这个项目依赖里(尤其是不熟悉的 API,顺手确认一下比事后排查省时间)?

小结:自动化不能替代审查,只能降低审查成本

回顾这个 Tab 三篇文章的完整链路:第一篇讲怎么把任务描述清楚(减少 Agent 猜错方向的概率), 第二篇讲什么时候可以放手用 Agentic Loop 自动跑(把自动化范围限定在代价可控的场景), 这一篇讲拿到结果后怎么验证(自动化检查 + 人工 diff 审查 + 独立视角复查三层关卡)。 三者共同的原则是一致的:Agent 能大幅降低"把事情做出来"的成本,但"确认它做对了"这件事,责任始终在使用者身上, 这也是整个 AI Agent 模块反复出现的主题——效率提升的另一面,永远是新的验证成本,工程判断力不会因为有了 Agent 就变得不重要,反而更重要。
