如何使用Agent

任务描述与 Prompt 实战技巧

Claude Code、Codex 这类工具已经把 ReAct 循环、工具调用、上下文管理都封装好了,用户唯一还要亲自做好的一件事,就是把任务描述清楚。这一篇讲怎么描述任务,才能让 Agent 少走弯路。

一份"够好"的任务描述包含什么

"帮我优化这个函数"和"帮我修复这个 bug"是新手最常见的任务描述,但对 Agent 来说信息量严重不足—— 这和"Agent 开发核心"Tab 讲的"规划与任务分解"是同一个道理:Agent 的第一步是理解目标,目标描述得越模糊,它拆解出来的执行路径就越可能跑偏。 一份够好的任务描述至少要包含四个要素:

要素作用反例 → 正例
目标 要达成什么效果,而不是"要做什么动作" "改一下这个函数" → "让这个函数在输入为空数组时不要抛异常,而是返回空结果"
范围约束 哪些文件/模块可以动,哪些绝对不能动 "帮我重构一下" → "只重构 UserService 这个类的内部实现,不要改它的公开方法签名"
验收标准 怎么算"完成了"——最好是客观可执行的检验方式 "确保没问题" → "跑一下 npm test,现有测试要全部通过,并新增一个覆盖空数组场景的测试"
已知约束/背景 项目里那些"不写在代码里,但大家都知道"的规矩 "按项目规范来" → "这个项目的错误处理统一走 Result 类型,不要用抛异常的方式"

项目级上下文文件:把"背景"写一次,处处生效

如果每次任务描述都要重新解释一遍"这个项目用什么框架、代码风格是什么、测试怎么跑",效率会很低。 Claude Code(CLAUDE.md)、Codex/OpenCode 等工具(AGENTS.md)都支持在项目根目录放一份约定文件, Agent 每次执行任务前会自动读取,相当于把"背景知识"一次性交代清楚,不用在每条任务描述里重复。

这份文件该写什么:项目用的技术栈和版本、代码风格约定(比如"不写无意义注释")、测试怎么跑、 有没有必须遵守的架构规则(比如"不允许直接操作数据库,必须走 Repository 层")。 这些是每次任务都会用到的"通用背景",不属于某一次具体任务描述该管的内容。

陷阱一:任务太模糊,Agent 只能"猜"

模糊的任务描述会逼着 Agent 自己脑补细节,而它脑补的方向不一定符合你的预期——尤其是在"改动范围"和"验收标准"这两点上。 比如"优化性能"这种描述,Agent 可能选择一个和你预期完全不同的优化方向(比如改了缓存策略而不是你想要的算法复杂度), 等你发现方向不对,已经浪费了一整轮执行时间。越是想让 Agent 自动跑得久一点,任务描述就越需要具体—— 这个原则会在下一篇"Agentic Loop"里进一步展开。

陷阱二:任务太大,一次说不清楚

另一个极端是把好几件事塞进一次任务描述里("重构这个模块,顺便把测试也补全,再优化一下接口性能")。 这类复合任务最好拆成多个独立的小任务顺序执行,原因和"规划与任务分解"一篇讲的一样: 步骤越多,中间出错的概率累积得越高,而且一旦某一步出问题,你很难判断是哪个子目标出的错。 更实用的做法是先让 Agent(或者你自己)把大任务拆成几个明确的小任务,一个一个来,每完成一个就检查一次。

和"概念与AI基础"Tab 的联系

这里讲的技巧本质上就是"Prompt Engineering 基础"一篇里角色设定、结构化输出这些原则,应用在"给 Agent 下任务"这个具体场景里—— 只是这里更强调"验收标准"这个维度,因为 Agent 不像单轮问答那样"回答完就结束",它会自主执行多步操作, 没有清晰的验收标准,你没法知道它什么时候真的做对了、什么时候只是"看起来做完了"(这个问题会在"如何验证与审查 AI 生成的代码"一篇继续深入)。
