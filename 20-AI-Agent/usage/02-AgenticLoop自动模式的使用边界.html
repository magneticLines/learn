Agentic Loop:自动模式该怎么用、什么时候不该用

"设定一个目标,让 Agent 自己循环执行到完成,中途不用人盯着"——这种用法现在很流行,但不是所有任务都适合完全放手。这一篇讲怎么判断,以及怎么分阶段放权。

什么是 Agentic Loop

Claude Code、Codex、OpenCode 都提供不止一种运行模式:一种是每一步关键操作都停下来等你确认;另一种是给一个目标后, Agent 自主执行"推理 → 行动 → 观察 → 再推理"的完整循环,直到任务完成或达到步数上限才停下来,中途不需要你逐步点头—— 这后一种用法就是所谓的 Agentic Loop(有的产品也叫 auto 模式、YOLO 模式、build 模式)。 它本质上就是"Agent 开发核心"Tab 里 ReAct 循环的原始形态,只是这里讨论的是"要不要让它无人值守地跑"这个使用层面的问题。

适合放手自动跑的场景

条件为什么重要
任务边界清晰 目标、范围、验收标准都写明白了(呼应上一篇),Agent 不需要"猜"你的意图
有客观的验收手段 有现成的测试套件、类型检查、CI 流水线——Agent 能自己判断"是不是真的做对了",而不是全靠自我感觉
操作可回滚 工作在独立的 git 分支或 worktree 里,跑砸了直接丢弃重来,没有不可逆的代价
没有破坏性副作用 不涉及删除生产数据、发送真实邮件、执行真实转账这类无法撤销的操作
一个好用的组合:给每个自动执行的任务开一个独立的 git 分支或 worktree,让它在隔离环境里跑完整个 Agentic Loop, 跑完你只需要审查一份 diff,不满意直接丢弃分支——相当于把"回滚成本"降到了几乎为零,这也是很多团队敢放心用自动模式的关键前提。

不适合完全自动跑的场景

自动化程度应该和"出错代价"成反比,而不是和"看起来能不能自动化"挂钩。下面这些场景即使技术上可以让 Agent 自动跑完, 也建议至少保留人工确认关键步骤:
  • 任务目标本身还模糊——比如"帮我看看这个系统架构有什么问题并改进",这种探索性任务连"验收标准是什么"都说不清楚, Agentic Loop 很容易在没有明确终点的情况下"自己觉得做完了",但结果未必是你想要的。这类任务更适合先用只读的 Plan/探索模式摸清情况,再手动拆解成具体任务。
  • 操作有破坏性且难以撤销——数据库迁移、生产环境部署、删除文件这类操作,即使 Agent 判断"这样做没问题", 也应该保留人工确认这一步,对照"OpenHands 拆解"Tab 讲的风险分级确认机制,这类操作理应落在"必须人工点头"的那一档。
  • 缺少客观验证手段——没有测试、没有类型检查、任务成果只能靠"看起来对不对"来判断时,自动跑完你也没法快速确认它是不是真的做对了, 不如让 Agent 走得慢一点,分阶段检查。

更现实的用法:分阶段放权

实际工作中很少是"要么全自动、要么全程盯着"这种二选一。更常见、也更稳妥的做法是分阶段放权,和 OpenCode 的 Plan/Build 双模式思路一致:

第一阶段:探索(只读) 让 Agent 先读代码、列出它打算怎么改,不实际执行任何写操作
  ↓ 你审查这份计划,确认方向没问题
第二阶段:执行(可写,隔离环境) 切到有写权限的模式,在独立分支/worktree 里跑完整个 Agentic Loop
  ↓ 跑完后自动/手动执行验收标准里定义的检查(跑测试、跑 lint)
第三阶段:审查 人工审查最终 diff(见下一篇),确认无误后再合并

这个流程的核心思想是:把"信息不确定、代价可能很高"的探索阶段和"目标明确、代价可控"的执行阶段分开, 只在代价可控的阶段放手用 Agentic Loop 自动跑,而不是把整个不确定的过程都交给自动模式。
