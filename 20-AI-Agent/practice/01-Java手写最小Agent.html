Java 手写最小 Agent(ReAct 循环 + 工具调用,不依赖框架)

不引入任何 Agent 框架,只用纯 Java 手写一个最小可用的 ReAct 循环,亲手实现一遍"推理 → 调用工具 → 观察结果 → 再推理"的过程,才能看清框架到底替你封装了什么。

目标:一个会查天气的最小 Agent

还是"什么是 Agent"里那个例子:用户问"北京今天天气怎么样,要不要带伞"。这句话本身模型答不了——它不知道"今天"的实时天气, 必须先调用一个天气工具拿到数据,再结合数据给出建议。这一篇要做的,就是用纯 Java 代码把这个"先查再答"的过程跑通,不依赖 LangChain、Spring AI 等任何框架—— 包括真实调用外部天气 API、真实调用 LLM 接口,全部用 JDK 自带的 HttpClient 手写,不依赖第三方 SDK。

跑之前需要准备:一个 OpenAI 兼容的 API Key,通过环境变量 LLM_API_KEY 传入(下面 LlmClient 会读取)。 天气工具用的 Open-Meteo 免费无需注册,不需要额外准备。

整体思路:Agent 的核心不是"调用了 LLM",而是一个循环——每一轮把"目标 + 历史记录 + 可用工具清单"拼成 Prompt 丢给 LLM, 让 LLM 自己决定"现在该调用哪个工具"还是"已经有把握给出最终答案"。这个决策权完全交给模型,代码只负责"解析模型说了什么"和"执行模型要求的动作"。

第一步:定义最小工具接口

工具对 Agent 来说就是"手"——一个统一的调用协议。这里只留三个方法:name() 给模型看的工具名(要和 Prompt 里的约定一致)、 description() 告诉模型这个工具是干什么用的、能力边界(模型是靠读这段描述来决定"要不要用它"的,不是靠读源码), execute(String args) 真正执行动作并返回结果。

Tool.java接口定义
public interface Tool {
    // 工具名必须和 Prompt 约定里 "Action: 工具名(参数)" 的写法一一对应
    // 否则模型吐出的 Action 文本会匹配不到任何工具,这是最容易踩的坑
    String name();

    // 这段文字会被塞进 Prompt 喂给模型,模型完全是"读文档"来决定用不用这个工具
    // 描述写得越清楚(参数格式、返回什么),模型选错工具、传错参数的概率越低
    String description();

    // 参数统一用 String,而不是设计成强类型方法签名
    // 因为模型只会生成文本,拿到手的天然就是一段字符串,交给工具自己去解析更简单可靠
    String execute(String args);
}

第二步:实现一个真实调用外部 API 的天气工具

很多教程到这一步就用一个写死的 Map mock 数据糊弄过去,但这样看不出"工具真正对接外部世界"时要处理的问题——网络请求会失败、 会超时、返回的数据要解析。这里改用 Open-Meteo(免费、无需 API Key 的天气服务)做一次真实的两段式调用: 先用城市名查经纬度(地理编码),再用经纬度查当前气温。

Json.java极简 JSON 取值工具
import java.util.regex.Matcher;
import java.util.regex.Pattern;

// 生产项目请直接用 Jackson/Gson——这里手写是为了不引入任何第三方依赖
// 只支持从"扁平"JSON 里按 key 取字符串/数值,足够应付本文的两个 API,不是通用解析器
final class Json {
    static String extractString(String json, String key) {
        Matcher m = Pattern.compile("\"" + key + "\"\\s*:\\s*\"([^\"]*)\"").matcher(json);
        if (!m.find()) throw new IllegalArgumentException("missing key: " + key);
        return m.group(1);
    }

    static double extractDouble(String json, String key) {
        Matcher m = Pattern.compile("\"" + key + "\"\\s*:\\s*(-?[0-9.]+)").matcher(json);
        if (!m.find()) throw new IllegalArgumentException("missing key: " + key);
        return Double.parseDouble(m.group(1));
    }
}
WeatherTool.java真实 HTTP 调用
import java.net.URI;
import java.net.URLEncoder;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.nio.charset.StandardCharsets;
import java.time.Duration;

public class WeatherTool implements Tool {

    // HttpClient 建议全局复用一个实例(内部维护连接池),不要每次请求都 new 一个
    private final HttpClient http = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(5))
        .build();

    @Override
    public String name() { return "get_weather"; }

    @Override
    public String description() {
        return "查询城市当前气温。参数为城市名(建议英文拼写,如 Beijing),返回该城市当前气温";
    }

    @Override
    public String execute(String args) {
        String city = args.trim();
        try {
            double[] coord = geocode(city);
            return forecast(coord[0], coord[1], city);
        } catch (Exception e) {
            // 网络请求随时可能失败(超时/限流/城市名查不到),绝不能让异常直接抛出炸掉整个 Agent 循环
            // 把错误信息当作正常的 Observation 返回,交给模型自己判断下一步(换个拼写重试,还是如实告知用户)
            return "查询失败: " + e.getMessage();
        }
    }

    private double[] geocode(String city) throws Exception {
        String url = "https://geocoding-api.open-meteo.com/v1/search?count=1&name="
            + URLEncoder.encode(city, StandardCharsets.UTF_8);
        String body = get(url);
        return new double[] { Json.extractDouble(body, "latitude"), Json.extractDouble(body, "longitude") };
    }

    private String forecast(double lat, double lon, String city) throws Exception {
        String url = String.format(
            "https://api.open-meteo.com/v1/forecast?latitude=%.4f&longitude=%.4f&current=temperature_2m", lat, lon);
        String body = get(url);
        double temp = Json.extractDouble(body, "temperature_2m");
        return city + " 当前气温 " + temp + "℃";
    }

    private String get(String url) throws Exception {
        HttpRequest req = HttpRequest.newBuilder(URI.create(url))
            .timeout(Duration.ofSeconds(5)).GET().build();
        HttpResponse<String> resp = http.send(req, HttpResponse.BodyHandlers.ofString());
        // 每次请求都显式检查状态码——不判断直接读 body,一旦对方返回错误页会在解析阶段才炸,定位更麻烦
        if (resp.statusCode() != 200) throw new RuntimeException("HTTP " + resp.statusCode());
        return resp.body();
    }
}
为什么选 Open-Meteo 而不是随便找个天气 API 举例:它免注册、免费、无需 API Key,这份代码复制下来就能直接跑, 不需要读者先去申请密钥才能验证效果——这是选教学示例 API 时值得考虑的一个实际因素。

第三步:约定模型的输出格式

Agent 和 LLM 之间需要一份"协议",让代码能从模型返回的自然语言文本里,可靠地解析出"它想干什么"。最简单的做法是在系统 Prompt 里明确要求模型 只能用下面两种格式之一结尾:

格式含义示例
Action: 工具名(参数) 模型认为需要调用某个工具,还不能直接回答 Action: get_weather(北京)
Final Answer: ... 模型认为信息已经够了,给出最终回复 Final Answer: 北京今天有雨 18℃,建议带伞
这是教学最小实现,不是生产做法。用字符串前缀 + 正则去匹配 Action: xxx(yyy) 非常脆弱——模型偶尔会多输出一句解释、 参数里带括号就直接解析失败。生产级 Agent 几乎都会改用 Function Calling / 结构化 JSON 输出 (让模型直接吐出 {"tool": "get_weather", "args": "北京"} 这样的结构化数据),配合异常重试、超时控制、 对模型输出做 schema 校验。这些工程化处理的更多细节,可以对照"开发难点与业界解法"一节继续深入。

第四步:Agent 主循环(ReAct)

Agent 类只做三件事:维护一份"历史记录"字符串、每轮拼 Prompt 调用 LLM、解析返回文本并决定"调用工具"还是"结束循环"。 加一个最大步数限制,防止模型陷入死循环、无限消耗 Token。

Agent.javaReAct 主循环
import java.util.List;
import java.util.Map;
import java.util.regex.Matcher;
import java.util.regex.Pattern;
import java.util.stream.Collectors;

public class Agent {

    // Action 的解析规则只认这一种格式,先把正则集中定义,方便以后统一升级成 JSON 解析
    private static final Pattern ACTION_PATTERN =
        Pattern.compile("Action:\\s*(\\w+)\\((.*)\\)");

    // 步数上限是唯一的安全护栏:哪怕模型陷入"一直调用工具却给不出结论"的死循环
    // 也能保证程序最终会退出,而不是无限消耗 Token 和调用次数
    private static final int MAX_STEPS = 5;

    // 用工具名做 key,查找 O(1),避免每次遍历 List 做字符串比较
    private final Map<String, Tool> tools;

    // LLM 调用被抽成独立的 LlmClient(见第五步),Agent 只关心"给 Prompt、拿文本",不关心 HTTP 细节
    private final LlmClient llmClient;

    public Agent(List<Tool> toolList, LlmClient llmClient) {
        this.llmClient = llmClient;
        tools = toolList.stream()
            .collect(Collectors.toMap(Tool::name, t -> t));
    }

    public String run(String userGoal) {
        // history 是整个循环共享的"记忆",每一轮的模型输出和工具观察都会追加进来
        // 循环之所以能"越问越准",本质就是靠这份不断增长的上下文喂给模型
        StringBuilder history = new StringBuilder();
        history.append("用户目标: ").append(userGoal).append("\n");

        for (int step = 1; step <= MAX_STEPS; step++) {
            String prompt = buildPrompt(history.toString());

            // callLLM 是伪代码,实际项目里替换成你使用的 LLM SDK 调用
            // 比如 OpenAI 的 chat.completions.create(...),这里只关心"输入 Prompt、拿到文本"这一层抽象
            String modelOutput = callLLM(prompt);

            if (modelOutput.contains("Final Answer:")) {
                // 出现终止信号,直接把最终答案截出来返回,循环结束
                return modelOutput.substring(
                    modelOutput.indexOf("Final Answer:") + "Final Answer:".length()
                ).trim();
            }

            Matcher matcher = ACTION_PATTERN.matcher(modelOutput);
            if (matcher.find()) {
                String toolName = matcher.group(1);
                String toolArgs = matcher.group(2);
                Tool tool = tools.get(toolName);

                // 模型完全可能"幻觉"出一个不存在的工具名,这里必须兜底
                // 而不是直接 NPE 崩掉整个 Agent —— 把错误也当成一次"观察"喂回给模型,让它自己纠正
                String observation = (tool != null)
                    ? tool.execute(toolArgs)
                    : "错误: 不存在名为 " + toolName + " 的工具";

                // 把"模型说了什么"和"工具观察到什么"都写回历史,下一轮 Prompt 会带着这些上下文
                history.append(modelOutput).append("\n");
                history.append("Observation: ").append(observation).append("\n");
            } else {
                // 模型输出既不是 Action 也不是 Final Answer,格式跑偏了
                // 最小实现里直接把原文当作观察记下来,让模型在下一轮里自己意识到格式不对并纠正
                history.append(modelOutput).append("\n");
            }
        }

        // 超过最大步数仍未收敛,不能让调用方无限等待,明确返回一个失败信号
        return "未能在限定步数内得出结论";
    }

    private String buildPrompt(String history) {
        // 每轮都把完整的工具清单和历史重新拼进 Prompt —— LLM 是无状态的
        // 它不会"记得"上一轮说过什么,所有上下文都必须在这一次调用里显式给全
        StringBuilder sb = new StringBuilder();
        sb.append("你是一个可以调用工具的助手,可用工具如下:\n");
        for (Tool tool : tools.values()) {
            sb.append("- ").append(tool.name()).append(": ")
              .append(tool.description()).append("\n");
        }
        sb.append("需要用工具时输出 Action: 工具名(参数);有把握回答时输出 Final Answer: 内容\n\n");
        sb.append(history);
        return sb.toString();
    }

    private String callLLM(String prompt) {
        return llmClient.complete(prompt); // 见下一节 LlmClient 的真实实现
    }
}

第五步:接入真实 LLM API(而不是留一个占位方法)

上面的 callLLM 不能只是一句"这里换成真实调用"就带过——真实调用要处理三件事:怎么发请求(HTTP + JSON body)、 怎么应对失败(网络抖动、被限流)、密钥怎么管理(绝不能硬编码进代码)。下面是一个对接 OpenAI 兼容 /v1/chat/completions 接口的最小可用实现,同样只用 JDK 自带的 HttpClient,不引入第三方 SDK。

LlmClient.java真实 HTTP 调用 + 重试
import java.io.IOException;
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.time.Duration;

public class LlmClient {

    private final HttpClient http = HttpClient.newHttpClient();

    // 密钥一律从环境变量读,绝不写进源码或提交进版本库——这是最容易被忽视也最容易泄露的一步
    private final String apiKey = System.getenv("LLM_API_KEY");
    private final String endpoint = "https://api.openai.com/v1/chat/completions";

    public String complete(String prompt) {
        // 手写 JSON 转义,只处理反斜杠/双引号/换行三种最常见的破坏字符——够用但不是通用实现
        String escaped = prompt.replace("\\", "\\\\")
            .replace("\"", "\\\"").replace("\n", "\\n");
        String requestBody = "{\"model\":\"gpt-4o-mini\",\"temperature\":0,\"messages\":[{\"role\":\"user\",\"content\":\""
            + escaped + "\"}]}";

        HttpRequest req = HttpRequest.newBuilder(URI.create(endpoint))
            .timeout(Duration.ofSeconds(30)) // 推理请求比普通 API 慢得多,超时要给够,但也不能无限等
            .header("Content-Type", "application/json")
            .header("Authorization", "Bearer " + apiKey)
            .POST(HttpRequest.BodyPublishers.ofString(requestBody))
            .build();

        return sendWithRetry(req, 3);
    }

    private String sendWithRetry(HttpRequest req, int maxRetries) {
        int attempt = 0;
        while (true) {
            try {
                HttpResponse<String> resp = http.send(req, HttpResponse.BodyHandlers.ofString());
                if (resp.statusCode() == 200) {
                    return Json.extractString(resp.body(), "content");
                }
                // 429(被限流)和 5xx(对方服务端临时故障)值得重试;4xx 参数错误重试没有意义,直接暴露问题
                if (resp.statusCode() == 429 || resp.statusCode() >= 500) {
                    if (++attempt > maxRetries) {
                        throw new RuntimeException("LLM 调用重试耗尽: HTTP " + resp.statusCode());
                    }
                    // 指数退避:第 1 次等 1s,第 2 次 2s,第 3 次 4s —— 避免在对方限流窗口内继续加压,让情况更糟
                    Thread.sleep(1000L << (attempt - 1));
                    continue;
                }
                throw new RuntimeException("LLM 调用失败,不可重试: HTTP " + resp.statusCode());
            } catch (IOException | InterruptedException e) {
                throw new RuntimeException("LLM 调用异常", e);
            }
        }
    }
}
这里还有意简化了一处:Json.extractString(body, "content") 假设响应体里 content 字段是唯一的、 格式扁平的。真实的 Chat Completions 响应是嵌套结构(choices[0].message.content),用正则硬取字段名在这种场景下能凑合工作, 但严格来说不安全——如果换一个把 content 作为其他字段值的接口就会取错。这也是"极简 JSON 工具"的边界所在, 真实项目请换成 Jackson/Gson 这类做过完整测试的解析库。

跑一遍完整流程

把前面几步组装起来,一个真实可编译的 main 方法长这样(设置好 LLM_API_KEY 环境变量后可以直接运行):

Main.java组装并运行
import java.util.List;

public class Main {
    public static void main(String[] args) {
        Agent agent = new Agent(List.of(new WeatherTool()), new LlmClient());
        System.out.println(agent.run("北京今天天气怎么样,要不要带伞"));
    }
}

一次典型的运行过程会是这样:

Step 1 模型看到目标"北京今天天气怎么样,要不要带伞",输出
  Action: get_weather(Beijing)
  → Agent 调用 WeatherTool,真实请求 Open-Meteo,得到 "Beijing 当前气温 16.2℃"
  → 写入 history: Observation: Beijing 当前气温 16.2℃

Step 2 模型看到 history 里已经有天气数据,输出
  Final Answer: 北京当前气温 16.2℃,建议穿薄外套,暂时不需要带伞
  → 循环检测到 Final Answer,直接返回结果,结束

整个过程只用了两轮,但如果模型第一次解析参数出错、或者天气工具返回的内容让模型觉得"还不够",它完全可以再多循环几轮—— 这正是 ReAct 循环"动态决定下一步"的体现,而不是写死"先查天气再回答"这样的固定两步流程。

手写之后回头看框架

写完这个最小实现,再回头看 LangChain / LangGraph 这类框架,会发现它们的核心骨架其实就是这里的 run() 循环—— 只是做得更完善:用 Function Calling(结构化 JSON)替代这里脆弱的正则字符串匹配、内置了几十种现成工具(搜索、代码执行、文件读写)、 重试策略比这里的固定指数退避更精细(区分不同错误类型、支持熔断)、还有流式输出支持(边生成边展示,而不是等一整轮推理完才拿到结果)。 理解了这个最小循环再学框架,就不是"记 API",而是"认出它在哪一步做了什么增强",这一点在对比框架选型取舍时会更有体感。
