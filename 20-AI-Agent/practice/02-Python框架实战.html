实战演练

Python + 框架(LangChain/smolagents)实战一个小任务型 Agent

上一篇用纯 Java 手写了 ReAct 循环,看清了"引擎盖底下"的样子;这一篇换到 Python 生态,用真实框架的开发方式实现同样的任务,体会工业级开发和手写实现的差距。

为什么实战要换到 Python

AI Agent 生态目前是 Python 主场——LangChain、LangGraph、smolagents、AutoGPT 等主流框架,以及绝大多数官方教程、开源示例、社区讨论, 都是 Python 优先甚至 Python 独占。这不是说 Java 不能做 Agent(上一篇已经证明完全可以),而是"如果要接触现成框架、抄现成方案、 看懂业界主流实践",Python 生态的资源密度高得多。所以实战的第二步,主动切换到 Python,用真实框架体验一次"工业级"的开发方式。

任务场景:穿衣建议 Agent

选一个和上一篇结构类似、但换了个角度的小任务:给定一个城市名,查询天气并给出穿衣建议。 这里用 smolagents 风格来演示——它的核心思路是 Code Agent:不像传统 ReAct 那样让模型输出 "Action: 工具名(参数)" 这种约定文本,而是直接让模型生成一段可执行的 Python 代码去调用工具, 执行环境跑完这段代码拿到结果,再喂回给模型。这种方式天然更贴近 Python 的表达能力,复杂逻辑(比如"先查两个城市再比较") 模型可以直接写 if/for 而不必受限于简单的文本协议。

真实可跑的示例代码

下面这份代码基于 smolagents 的公开 API(@tool 装饰器 + CodeAgent),没有用 mock 数据—— get_weather 真的会去请求 Open-Meteo(免费、无需 API Key 的天气服务),先按城市名做地理编码拿到经纬度, 再查当前气温。只要装好依赖、配置好模型的 API Key,这份代码可以直接跑起来。

weather_agent.pysmolagents(真实 API)
# 依赖: pip install smolagents requests
import os
import requests
from smolagents import CodeAgent, tool, OpenAIModel

@tool
def get_weather(city: str) -> str:
    """查询指定城市当前气温(数据源: Open-Meteo,免费无需 Key)。

    Args:
        city: 城市名,建议用英文拼写以提高地理编码命中率,例如 "Beijing"
    """
    # smolagents 对 docstring 的 Args 段有硬性要求——它会被原样解析成
    # 喂给模型的参数说明,写得含糊,模型传参出错的概率就越高
    geo = requests.get(
        "https://geocoding-api.open-meteo.com/v1/search",
        params={"name": city, "count": 1},
        timeout=5,
    ).json()
    if not geo.get("results"):
        # 工具执行失败不能让整个 Agent 崩掉,把原因当作正常返回值交回给模型
        # 模型看到这句话,大概率会自己决定换个拼写或者告知用户查询失败
        return f"找不到城市: {city},请确认拼写"

    lat = geo["results"][0]["latitude"]
    lon = geo["results"][0]["longitude"]

    weather = requests.get(
        "https://api.open-meteo.com/v1/forecast",
        params={"latitude": lat, "longitude": lon, "current": "temperature_2m"},
        timeout=5,
    ).json()
    temp = weather["current"]["temperature_2m"]
    return f"{city} 当前气温 {temp}摄氏度"


# OpenAIModel 是 smolagents 内置的模型封装之一,对接任意 OpenAI 兼容接口
# API Key 从环境变量读取,绝不写死在代码里——这一点无论 Java 还是 Python 都一样
model = OpenAIModel(model_id="gpt-4o-mini", api_key=os.environ["LLM_API_KEY"])

# max_steps 就是"开发核心"一节反复强调的死循环护栏,框架已经内置,不用自己写
agent = CodeAgent(tools=[get_weather], model=model, max_steps=5)

result = agent.run("北京今天穿什么合适")
print(result)
# 输出类似: "北京当前气温 16.2摄氏度,建议穿薄外套或长袖"
CodeAgent 运行时实际发生了什么:模型不会直接输出"调用 get_weather"这样的约定文本,而是被要求直接生成一段 Python 代码, 大致相当于 print(get_weather("Beijing"));smolagents 在一个受限的 Python 解释器里执行这段代码,把 print 输出的内容当作 Observation 喂回给模型,模型再决定是否要生成新一轮代码或者直接给出最终答案。这就是"框架生态"一节提到的 Code Agent 范式,比传统 Function Calling 更自然地支持"先查天气,再判断温度区间,再拼穿衣建议"这种带条件逻辑的多步操作。
安全提醒:执行模型生成的代码存在风险(模型可能写出访问文件系统、发起意外网络请求的代码), smolagents 默认会限制可用的内置函数和模块;生产环境如果开放给不受信任的用户输入,建议参考"OpenHands 源码拆解"一节讲的 Docker 沙箱隔离思路,把执行环境完全隔离出去,而不是依赖语言层面的沙箱。

和上一篇 Java 版本对比

Java 手写版Python + smolagents
Prompt 拼接 手动把工具列表、历史记录拼成字符串 框架自动生成,包含工具的函数签名和 docstring
解析模型输出 手写正则匹配 Action: xxx(yyy),脆弱且要自己兜底异常格式 框架直接执行模型生成的 Python 代码,天然支持复杂逻辑(条件、循环、多工具组合)
循环控制 手写 while/for 循环 + 最大步数限制 封装在 agent.run() 内部,同样有步数上限等护栏,但不需要自己写
错误重试 需要自己加 try/catch、超时控制 框架内置重试机制和更完善的异常处理
你需要写的代码 整个循环 + 工具接口 + 解析逻辑 只需要:定义工具(@tool 函数)+ 给出任务目标(agent.run(...))

对比之后能看得很清楚:框架把"怎么让模型和工具配合起来循环工作"这件苦活全部吃掉了,开发者的注意力被压缩到两件事上—— 工具怎么定义(签名、docstring 要写清楚,模型是靠读这些来决策的)和任务目标怎么描述。 这也是为什么工业界做 Agent 开发大多直接上框架,而不是每个项目都从零手写一遍 ReAct 循环。

模块总结:从概念到动手的完整路径

回顾一下整个 AI Agent 模块是怎么串起来的:

  • 概念与 AI 基础——先搞清楚"Agent 是什么"、"和 Chatbot 有什么本质区别",再补上 LLM 的基础概念(参数量、Tokenization、Embedding、注意力机制),打好理解上限的地基。
  • Agent 开发核心——深入 ReAct 循环的机制本身,以及"上下文爆炸""工具调用可靠性""死循环护栏"这些真实开发中会撞上的难点。
  • 框架生态——了解 LangChain/LangGraph、smolagents 等现成工具解决了哪些问题,以及不同框架之间的选型取舍。
  • OpenHands 拆解——挑一个工业级开源 Agent 项目,看真实生产系统是怎么把"大脑-手-记忆"三层做工程化落地的。
  • 实战演练(本节)——先用纯 Java 手写一遍最小循环建立底层直觉,再用 Python 框架体验一次工业级开发方式,两相对照,理解框架到底帮你封装了什么。
AI Agent 是一个还在快速演进的领域,新框架、新范式(比如更强的多 Agent 协作、更成熟的记忆机制)会持续出现。 这个模块不是一次性写完就结束,后续会随着生态发展持续追加新内容——如果你在实际项目中踩到了新的坑或发现了更好的实践,也值得回来补充。
