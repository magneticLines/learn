OpenHands源码拆解

Event-Stream 机制(Agent → Action → Observation)

OpenHands 把 Agent 和 Runtime 之间的所有交互,都建模成了一条只能追加(append-only)的事件流。这一篇讲清楚:为什么不用直接的方法调用,事件流到底换来了什么。

为什么不是"Agent 直接调用 Runtime 的方法"

最直观的实现方式是:Agent 决定"运行命令 ls -la",直接调 runtime.execute("ls -la"),拿到返回值继续处理。 这种写法在玩具项目里完全够用,但放到一个要长期运行、支持多用户、需要可观测性的系统里,会暴露出三个问题:

  • 没有完整的可回放历史——函数调用完就完了,除非你额外写日志,否则"Agent 到底经历了哪些步骤"没有一份天然、结构化的记录,调试和向用户展示执行过程都很困难。
  • 无法让多个组件同时消费同一份交互——前端要实时展示"Agent 正在做什么",日志系统要记录审计信息,Agent 自己要读历史做下一步决策。如果是直接函数调用,这些需求就得各自埋点、各写一套。
  • 难以中断和恢复——如果任务执行到一半用户断线了,或者需要暂停排查,直接函数调用模型里"当前进行到哪一步"这个状态很难被完整保存下来。
核心思路:把"Agent 说要做什么"和"Runtime 说结果如何"都变成一条不断追加的事件流(EventStream), 上面三个问题就都变成了这份数据结构自带的性质——历史就是这条流本身,谁想订阅就去监听这条流,想恢复任务就把这条流重新加载进来。

事件的基本结构

可以把 EventStream 里的事件粗略理解成两大类:Action(Agent 想做什么)和 Observation(Runtime 执行后看到了什么)。 一个简化的示意结构大致是这样(伪代码,非真实源码):

伪代码示意:Action / Observation 事件结构
// Agent 产生的 Action 事件——"我想做什么"
Event {
  id: 42,
  type: "CmdRunAction",      // 动作类型:运行 shell 命令
  source: "agent",
  args: { command: "pytest tests/" }
}

// Runtime 执行后产生的 Observation 事件——"结果是什么"
Event {
  id: 43,
  type: "CmdOutputObservation",
  source: "runtime",
  causeId: 42,          // 指回触发它的那个 Action
  content: "3 passed, 1 failed ...",
  exitCode: 1
}

注意 Observation 里带着一个 causeId 指回触发它的 Action——这让整条事件流不是一堆散落的记录,而是一条能追溯"哪个动作导致了哪个结果"的因果链。

一次完整交互的事件序列

把这些事件按时间顺序串起来,一次"帮我跑一下测试并修复失败用例"的任务,事件流大致长这样:

UserMessage "帮我跑一下测试,修复失败的用例"
  ↓ Agent 推理:应该先跑一下测试看看现状
CmdRunAction command: "pytest tests/"
  ↓ Runtime 在沙箱容器里执行
CmdOutputObservation "3 passed, 1 failed: test_login"
  ↓ Agent 读取这条 Observation,推理下一步:该去看看失败用例的代码
FileEditAction 修改 login.py 里的一处逻辑
  ↓ Runtime 执行文件编辑
FileEditObservation "修改已应用"
  ↓ Agent 再次验证
CmdRunAction command: "pytest tests/"
  ↓ Runtime 再次执行
CmdOutputObservation "4 passed"
  ↓ Agent 判断任务已完成
FinishAction "已修复失败用例,全部测试通过"

可以看到,Agent 每一步的决策都建立在"读取 EventStream 里已经发生的历史"之上——它不需要额外的状态变量来记住"我刚才跑过测试、修了哪个文件", 这些信息都天然地存在于事件流里。这正是上一篇提到的"EventStream 即短期记忆"的具体体现。

陷阱:事件流会不断增长

上下文爆炸的具体化。EventStream 是只追加的,任务跑得越久、步骤越多,累积的事件就越多。 但 Agent 每一步做决策时,并不能把无限增长的历史原样喂给 LLM——这正是"概念与AI基础"一篇提到的 Context Window 有限且昂贵这个问题,在真实系统里的具体体现。 OpenHands 需要做历史截断(只保留最近若干步的完整细节)或历史摘要(把久远的事件压缩成一段摘要文本) 等机制,才能让 Agent 在长任务里既不"失忆"又不会把上下文塞爆、成本飙升。

这也是事件流架构的一个有意思的张力:它的优点是"完整记录一切",但完整记录本身又会成为下一步要解决的工程问题—— 如何在"保留足够信息做出正确决策"和"控制喂给 LLM 的上下文体积"之间做取舍,是任何基于事件流/历史回放做记忆机制的 Agent 系统都绕不开的课题。
