Event-Stream 机制(Agent → Action → Observation)

OpenHands 把 Agent 和 Tool 之间的所有交互都建模成事件。这一篇讲清楚:为什么不用直接的方法调用,以及新旧两代架构在"事件"这件事上真实的区别。

关于标题里的"Event-Stream":这是 OpenHands 早期单体架构里的真实叫法——一条只能追加(append-only)的事件日志。 在 2025 年拆分出来的 openhands-sdk(见上一篇)里,这套机制演进成了一棵事件树而不是一条线性的流, 下面会讲清楚这个变化,标题沿用旧名是因为"事件驱动、解耦决策与执行"这个核心思路并没有变。

为什么不是"Agent 直接调用 Runtime 的方法"

最直观的实现方式是:Agent 决定"运行命令 ls -la",直接调 runtime.execute("ls -la"),拿到返回值继续处理。 这种写法在玩具项目里完全够用,但放到一个要长期运行、支持多用户、需要可观测性的系统里,会暴露出三个问题:

  • 没有完整的可回放历史——函数调用完就完了,除非你额外写日志,否则"Agent 到底经历了哪些步骤"没有一份天然、结构化的记录,调试和向用户展示执行过程都很困难。
  • 无法让多个组件同时消费同一份交互——前端要实时展示"Agent 正在做什么",日志系统要记录审计信息,Agent 自己要读历史做下一步决策。如果是直接函数调用,这些需求就得各自埋点、各写一套。
  • 难以中断和恢复——如果任务执行到一半用户断线了,或者需要暂停排查,直接函数调用模型里"当前进行到哪一步"这个状态很难被完整保存下来。
核心思路:把"Agent 说要做什么"和"Runtime 说结果如何"都变成一条不断追加的事件流(EventStream), 上面三个问题就都变成了这份数据结构自带的性质——历史就是这条流本身,谁想订阅就去监听这条流,想恢复任务就把这条流重新加载进来。

事件的真实结构:一棵树,不是一条线

openhands-sdk 里所有事件的基类是 Event(定义在 openhands/sdk/event/base.py), 真实字段比想象的更有意思——除了 id、timestamp、source,还有一个 parent_id:

event/base.pypython · 节选自真实源码
class Event(DiscriminatedUnionMixin, ABC):
    """Base class for all events."""

    id: EventID = Field(default_factory=lambda: str(uuid.uuid4()))
    timestamp: str = Field(default_factory=lambda: datetime.now().isoformat())
    source: SourceType = Field(...)

    # parent_id 让事件构成一棵树而不是一条链:同一个 parent_id 下的多个事件是"兄弟分支"
    # 官方注释原文:Events sharing a parent_id are sibling branches
    parent_id: EventID | None = Field(default=None)

这是比"一条只能追加的日志"更进一步的设计——树形结构原生支持会话分叉(比如用户回到某个历史节点重新提问, 产生一条新的分支,而不是线性覆盖掉原来的路径),这是纯线性 EventStream 天然做不到的能力。

在这棵树上,真正驱动 ReAct 循环的是几个具体的事件子类型(定义在 event/llm_convertible/ 目录下):

事件类型含义
ActionEventAgent 决定要调用的工具——真实字段包括 tool_name、tool_call_id、 thought(推理过程)以及 LLM 原始返回的 tool_call
ObservationEvent工具执行后的结果,通过 ActionEvent 的 tool_call_id 关联回是哪次调用产生的
MessageEvent用户或 Agent 的自然语言消息
AgentErrorEventAgent 执行过程中出现的错误,单独建模成一种事件类型,而不是让错误信息淹没在普通 Observation 里

值得注意的是 ActionEvent 直接复用了 LLM 厂商原生的 Function Calling 格式 (字段里保留了一份 MessageToolCall),而不是自造一套私有协议——这和"框架生态"Tab 里讨论的"结构化 JSON 输出"是同一套思路。

一次完整交互的事件序列

把这些真实事件类型按时间顺序串起来,一次"帮我跑一下测试并修复失败用例"的任务,事件树(在没有分叉的情况下退化成一条线)大致长这样:

MessageEvent (source=user) "帮我跑一下测试,修复失败的用例"
  ↓ Agent 推理:应该先跑一下测试看看现状
ActionEvent tool_name=execute_bash, 参数: "pytest tests/"
  ↓ 对应 Tool 在 Workspace 里执行
ObservationEvent "3 passed, 1 failed: test_login"
  ↓ Agent 读取这条 Observation,推理下一步:该去看看失败用例的代码
ActionEvent tool_name=str_replace_editor, 修改 login.py 里的一处逻辑
  ↓ 文件编辑工具执行
ObservationEvent "修改已应用"
  ↓ Agent 再次验证
ActionEvent tool_name=execute_bash, 参数: "pytest tests/"
  ↓ 再次执行
ObservationEvent "4 passed"
  ↓ Agent 判断任务已完成
MessageEvent (source=agent) "已修复失败用例,全部测试通过"

可以看到,Agent 每一步的决策都建立在"读取事件历史"之上——它不需要额外的状态变量来记住"我刚才跑过测试、修了哪个文件", 这些信息都天然地存在于事件树里。这正是上一篇提到的"事件历史即短期记忆"的具体体现。

历史会不断增长:真实的解法是 Condensation 事件

上下文爆炸的具体化。事件树是只追加的,任务跑得越久、步骤越多,累积的事件就越多。 Agent 每一步做决策时,并不能把无限增长的历史原样喂给 LLM——这正是"概念与AI基础"一篇提到的 Context Window 有限且昂贵这个问题,在真实系统里的具体体现。

SDK 里对应的真实机制叫 Condensation(定义在 event/condenser.py),它本身也是一种事件, 记录了"哪些历史事件 ID 被遗忘了"(forgotten_event_ids)以及一段可选的summary摘要文本:

event/condenser.pypython · 节选自真实源码
class Condensation(Event):
    """This action indicates a condensation of the conversation history is happening."""

    # 被"忘掉"(不再喂给 LLM)的历史事件 ID 集合——注意事件本身不会被物理删除,只是从视图里排除
    forgotten_event_ids: set[EventID] = Field(default_factory=set)
    summary: str | None = Field(default=None)

这个设计有个细节值得注意:"忘记"不是物理删除事件,而是生成一条 Condensation 事件, 记录"接下来构造喂给 LLM 的上下文时,跳过这些事件、换成这段摘要"。原始事件依然完整保留在事件树里(用于展示完整历史、调试、审计), 只是"喂给模型看的那个视图"和"完整保存的历史"是两回事——这比简单粗暴地截断或物理删除更稳妥。

这也是事件驱动架构的一个有意思的张力:它的优点是"完整记录一切",但完整记录本身又会成为下一步要解决的工程问题—— 如何在"保留足够信息做出正确决策"和"控制喂给 LLM 的上下文体积"之间做取舍,是任何基于事件历史做记忆机制的 Agent 系统都绕不开的课题。
