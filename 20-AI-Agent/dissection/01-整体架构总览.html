整体架构总览

与其空谈"Agent 架构应该怎么设计",不如挑一个真实跑在生产环境里的开源系统拆开看。这个 Tab 选择 OpenHands 作为解剖对象,理解一套工业级 Agent 系统长什么样。

为什么选 OpenHands 作为拆解对象

Agent 相关的开源项目很多,但适合拿来"学architecture"的不多——很多项目要么是薄薄一层 Prompt 拼接,没什么架构可言; 要么工程细节堆得太深,没有文档帮你建立全局图景。OpenHands(前身叫 OpenDevin,GitHub 地址 https://github.com/All-Hands-AI/OpenHands,star 数 7 万+)是一个例外:它是一个开源的自主编程 Agent, 目标是让 Agent 像人类工程师一样,能写代码、跑命令、浏览网页、调试报错。选它作为拆解样本,主要基于三点:

  • 架构清晰——它把"Agent 决策"和"代码执行"这两件事,通过一个统一的事件模型彻底解耦,这个设计思路本身就值得学习,而不只是"能跑起来"。
  • 文档与教程完整——项目的公开架构文档把 Agent/Runtime/EventStream 的职责边界讲得很清楚,适合作为"看得懂的样本"而不是"猜源码"。
  • 社区活跃、迭代真实——它不是一个玩具项目或论文配套代码,而是长期演进、被大量用户实际使用的系统,架构决策背后有真实的工程压力做验证。
说明:本 Tab 后续 4 篇内容,都是基于 OpenHands 公开的架构设计思路做的原理性拆解讲解,重点是讲清楚"为什么这样设计、解决了什么问题", 不会引用具体源码行号,也不需要你去读源码才能看懂。

整体组件关系

先建立一个全局地图。一个用户提交的任务,在 OpenHands 里大致会经过这样的链路:

用户界面 / CLI (提交任务目标,展示执行过程)
  ↓ 用户目标
Session / Controller (维护一次任务的完整生命周期)
  ↓ 驱动循环
Agent (决策:调用 LLM 决定下一步 Action)
  ↕ 读写事件
EventStream (事件总线:记录 Action / Observation 全部历史)
  ↕ 消费 Action / 产生 Observation
Runtime (执行:Docker 沙箱,跑 Shell / Jupyter / 浏览器)

这条链路里,Session/Controller 是"项目经理"角色:接收用户目标,启动并持续驱动 Agent 的执行循环, 同时判断什么时候该收工——任务完成了、达到最大步数了、还是用户主动中断了。它不参与具体的推理和执行,只管"循环该不该继续转下去"。

核心思想:用统一的事件模型解耦决策和执行

这套架构最值得学习的地方,不是"用了 Docker"或者"接了 LLM",而是它把 Agent 和 Runtime 之间的所有交互,都收敛成了一种统一的数据结构——事件(Event)。 Agent 不会直接调用 Runtime 的某个方法,Runtime 也不会直接调用 Agent 的某个回调,双方只通过往 EventStream 里读写 Action 事件和 Observation 事件来打交道。

这正好呼应"概念与AI基础"Tab 里"什么是 Agent"一篇提到的大脑-手-记忆三层划分:在 OpenHands 里, Agent 是大脑——负责推理和决策;Runtime 是手——负责真正在沙箱里执行命令、改文件、跑代码; EventStream 里积累的历史就是短期记忆——Agent 每一步做决策时,都要参考这段历史来理解"当前任务进行到哪一步了、之前做过什么、结果如何"。

类比:可以把这套架构想象成一个远程指挥中心——指挥官(Agent)不会亲自跑到现场拧螺丝, 他通过一份共享的行动记录(EventStream)下达指令(Action),前线队员(Runtime)执行指令后把结果(Observation)写回同一份记录, 指挥官看着记录决定下一步指令。指挥官和队员之间唯一的接口就是这份记录,谁也不需要知道对方内部是怎么工作的。

为什么不直接用函数调用

一个自然的问题是:既然 Agent 决定"运行一个 shell 命令",为什么不直接在代码里调用 runtime.runCommand(cmd), 非要绕一圈经过事件总线？这个问题的答案,会在下一篇《Event-Stream 机制》里详细展开——简单说,直接函数调用会让"记录历史""多方订阅""中断恢复"这些工业级系统必须具备的能力, 变得难以实现;而事件流架构把这些能力,变成了这个数据结构自带的性质。

后面三篇的路线图

  • 02 Event-Stream 机制——深入 Action/Observation 事件的结构,以及为什么"一切皆事件"能解决可回放、多订阅、可中断这几个问题。
  • 03 Docker 沙箱与 Runtime 隔离设计——深入 Runtime 内部,讲清楚为什么执行环境必须隔离,以及隔离到什么程度才算够。
  • 04 Agent 与 Runtime 解耦的设计取舍——跳出具体机制,站在架构设计的高度,讨论这种解耦模式的收益与代价。
