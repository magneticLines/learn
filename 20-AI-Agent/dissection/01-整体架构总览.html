OpenHands源码拆解

整体架构总览

与其空谈"Agent 架构应该怎么设计",不如挑一个真实跑在生产环境里的开源系统拆开看。这个 Tab 选择 OpenHands 作为解剖对象,理解一套工业级 Agent 系统长什么样。

为什么选 OpenHands 作为拆解对象

Agent 相关的开源项目很多,但适合拿来"学架构"的不多——很多项目要么是薄薄一层 Prompt 拼接,没什么架构可言; 要么工程细节堆得太深,没有文档帮你建立全局图景。OpenHands(前身叫 OpenDevin,主仓库 All-Hands-AI/OpenHands,近 8 万 star)是一个例外:它是一个开源的自主编程 Agent, 目标是让 Agent 像人类工程师一样,能写代码、跑命令、浏览网页、调试报错。选它作为拆解样本,主要基于三点:

  • 架构清晰且真的在演进——它把"Agent 决策"和"代码执行"彻底解耦,而且这套架构本身经历过一次公开的大重构(下面会讲),能看到真实系统架构是怎么迭代的,而不是一份一成不变的教科书设计。
  • 核心引擎已经拆成独立、可读的开源库——不需要在一个巨型单体仓库里"猜"架构边界在哪。
  • 社区活跃、迭代真实——被大量用户实际使用,架构决策背后有真实的工程压力做验证,不是论文配套代码。
说明:本 Tab 后续 4 篇基于 OpenHands 公开仓库的真实源码结构整理(截至撰写时的版本), 引用的类名、模块路径、代码片段均来自公开 GitHub 仓库,具体实现细节以你实际查看的版本为准——这类活跃项目的内部结构会持续变化。

一次值得知道的架构演进:从单体到独立 SDK

早期的 OpenHands(包括很多流传的技术博客描述的版本)是一个单体应用:Agent 的决策逻辑、事件记录、Runtime 执行都耦合在 All-Hands-AI/OpenHands 这一个仓库里,核心抽象叫 EventStream——一条只能追加的事件日志。 2025 年该团队把 Agent 的核心引擎抽成了一个独立仓库 OpenHands/software-agent-sdk (配套技术报告), 定位是"可组合、可扩展的生产级 Agent 基础设施",OpenHands 的 Web 产品、CLI、GitHub 集成都是这个 SDK 的具体应用之一。 本 Tab 后面几篇拆解的就是这个更新、更聚焦的 SDK 仓库。

整体组件关系

SDK 的架构文档把核心组件划分成六个,职责边界很清楚:

组件职责
Agent中枢决策者,实现"推理-行动循环"——查询 LLM 决定下一步该做什么
Conversation管理一次交互的状态与生命周期,接收用户消息喂给 Agent,再根据结果更新状态
LLM厂商无关的语言模型接口,内置重试和可观测性(telemetry)
Tool实现 Action / Observation / Executor 模式,真正执行文件操作、命令执行等具体工作
Event贯穿全系统的类型化事件框架,记录动作、观察、用户消息、状态更新
Workspace执行环境的抽象——决定 Agent 和 Tool 到底跑在本地、容器里,还是远程服务器上

官方文档给出的执行流程是一条很直观的链路:

User → Conversation → Agent → LLM(决策)→ Tool(执行)→ Observation → LLM(判断是否继续)→ Agent → Conversation → User 收到回复

官方文档特别强调一句话:"同一份 Agent 代码,只需要替换 Workspace 配置,就能跑在不同的执行环境里" (LocalWorkspace 本地直接跑、DockerWorkspace 起一个容器、RemoteAPIWorkspace 连接远程服务)。 这正是"Agent 与 Runtime 解耦"这个设计取舍换来的直接收益,后面 03、04 两篇会深入展开。

真实的最小可运行示例

下面是 SDK 官方仓库 README 里的真实代码(不是伪代码),可以直接体会 Agent / Conversation / Tool 三者怎么协作:

quickstart.pypython · 来自官方 README
import os
from openhands.sdk import LLM, Agent, Conversation, Tool
from openhands.tools.file_editor import FileEditorTool
from openhands.tools.task_tracker import TaskTrackerTool
from openhands.tools.terminal import TerminalTool

llm = LLM(model="gpt-5.5", api_key=os.getenv("LLM_API_KEY"))

# Agent 只声明"有哪些工具",具体怎么隔离执行由 Workspace 决定,这里没有出现在 Agent 定义里
agent = Agent(
    llm=llm,
    tools=[
        Tool(name=TerminalTool.name),
        Tool(name=FileEditorTool.name),
        Tool(name=TaskTrackerTool.name),
    ],
)

# workspace=cwd 这里用的是本地当前目录,换成 DockerWorkspace(...) 就能切到容器隔离执行,Agent 代码完全不用改
conversation = Conversation(agent=agent, workspace=os.getcwd())
conversation.send_message("把当前项目的 3 个关键事实写进 FACTS.txt")
conversation.run()
和"什么是 Agent"一篇的对应关系:Agent 是大脑(决策),Tool 是手(执行), Conversation 维护的事件历史是短期记忆——概念没变,只是这里能看到它们在真实代码里对应哪个具体的类。

与其他知识点的联系

  • Agent 与 Runtime 解耦的设计取舍(本 Tab 04):这一篇提到的"同一份 Agent 代码只需替换 Workspace 配置就能换执行环境" 只是先给结论,04 篇会回过头来分析这个设计到底换来了什么好处、付出了什么代价——先看到"是什么"再看"为什么值得",是这个 Tab 刻意安排的顺序。
  • 开发难点与业界解法(开发核心 Tab):那一篇讲的"工具调用可靠性""上下文爆炸"是纯原理层面的难点, 而这里的 Tool、Event 组件,正是 OpenHands 在真实工程里用来应对这些难点的具体载体——比如 Event 记录的完整历史, 就是"上下文管理"问题在生产系统里落地后的样子。
  • Java 手写最小 Agent(实战演练 Tab):手写实现里的 Agent.run() 循环、Tool 接口、 history 字符串,分别对应这里 SDK 里的 Agent、Tool、Event 三个组件——手写版本是"教学简化版", 这一篇拆解的是"工业级完整版",两者对照着看能清楚看到哪些地方被简化、哪些地方被补全。
  • 策略模式:Workspace 抽象出 LocalWorkspace/DockerWorkspace/RemoteAPIWorkspace 几种实现, Agent 侧代码只依赖 Workspace 这个接口而不关心具体实现,这正是策略模式的典型应用——把"在哪执行"这个可变的策略 从"怎么决策"这个稳定的主流程中抽出来,替换策略不需要改动主流程代码。
  • 框架选型对比与取舍(框架生态 Tab):OpenHands 的六个组件划分(Agent/Conversation/LLM/Tool/Event/Workspace) 和 LangGraph 的节点、AutoGen 的 Conversation 概念虽然叫法不同,但都是在解决同一组问题——如何清晰划分决策、执行、状态管理的边界, 对比不同框架的组件划分方式,能帮助理解"Agent 系统的核心组件"存在哪些共性。
