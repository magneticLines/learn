Docker 沙箱与 Runtime 隔离设计

Agent 生成的命令和代码是 LLM"猜"出来的,不可能保证 100% 正确安全。Runtime 把执行环境关进 Docker 容器,本质上是在给"猜错了怎么办"这件事上保险。

为什么执行必须隔离

Agent 的"手"(Runtime)要做的事情包括跑 shell 命令、执行代码、改文件。这些操作都来自 LLM 的生成结果——而 LLM 会犯错。 可能生成的命令语法对但语义错(比如把 rm -rf ./build 误写成 rm -rf / 的变体), 也可能在处理不受信任的网页内容、第三方代码时,被注入进恶意指令(prompt injection 导致的间接攻击)。 如果这些命令直接跑在宿主机上,一次"猜错"就可能删掉重要文件、影响其他正在运行的服务,甚至波及同一台机器上其他用户的会话。

核心问题:你没办法要求 LLM 生成的每一条命令都绝对正确安全,那能做的就是限制"猜错了"的爆炸半径。 把执行环境放进一个隔离的 Docker 容器,就是把"最坏情况"框定在容器内部——容器可以被销毁重建,宿主机和其他会话不受影响。

Runtime 容器里通常有什么

一个编程 Agent 要完成"写代码、跑代码、调试报错、查资料"这类任务,Runtime 容器里通常会准备好这几类能力:

能力用途
Shell 执行环境(如 SSH) 让 Agent 能运行任意命令行工具——装依赖、跑测试、查看文件、执行脚本
Jupyter Kernel 支持逐步执行 Python 代码并保持状态(变量、已导入的库不会每次执行都清空),适合数据探索、调试这类需要"接着上次结果继续跑"的场景
浏览器自动化 让 Agent 能像人一样打开网页、点击、读取页面内容,用于查资料、验证 Web 应用的实际效果
文件系统 提供一份隔离的工作目录,Agent 在这里读写、编辑代码文件,不会碰到宿主机的真实文件系统

这些能力共同的特点是:它们都跑在同一个隔离容器内,Agent(在容器外,作为决策进程)通过 Action/Observation 这套接口跟容器内的能力打交道,自己完全不直接接触宿主机资源。

陷阱:沙箱不是银弹

容器隔离降低风险,但不等于零风险。生产级系统至少还要考虑以下几层加固,单纯"套一层 Docker"并不够:
  • 容器逃逸风险——容器隔离依赖内核层面的命名空间和 cgroup,历史上出现过容器逃逸漏洞,不能假设容器天生绝对安全,需要跟进安全补丁、避免用特权模式启动容器。
  • 资源限制——需要给容器设置 CPU、内存配额,否则一个失控的死循环脚本可能耗尽宿主机资源,影响同机的其他会话。
  • 网络访问控制——容器如果能不受限地访问外网,沙箱内的代码理论上可以对外发起攻击(比如被诱导执行了一段恶意生成的爬虫/攻击脚本),需要按需限制出站网络访问。
  • 执行超时与命令黑名单——给单次命令执行设置超时,拦截明显危险的命令模式(比如清空磁盘、修改系统关键配置),作为在沙箱之外的又一层防护。

换句话说,沙箱解决的是"把爆炸半径限制在一个可控范围内",但这个范围本身还需要额外的资源限额、网络策略、超时熔断来收窄, 是一套多层防护(defense in depth),而不是加一层容器就能一劳永逸的问题。

联系:这是"直接执行模型生成代码"这个老问题的工业级解法

"AI 框架"Tab 讨论 smolagents 时提到过一个尖锐的问题:让 LLM 直接生成并执行代码,天然存在安全风险——毕竟你没法完全信任模型的输出。 OpenHands 面对的是同一个问题,只是场景更重(要跑任意 shell 命令、装依赖、操作浏览器,而不只是执行一段 Python 代码片段)。 它给出的答案就是这一篇讲的:不追求模型输出绝对正确,而是用运行时的隔离机制兜底——这也是"安全性"在 Agent 系统设计里的通用思路:与其试图让决策层完美无缺,不如在执行层建立可靠的护栏。
