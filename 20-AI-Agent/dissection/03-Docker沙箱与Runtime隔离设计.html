Docker 沙箱与 Runtime 隔离设计

Agent 生成的命令和代码是 LLM"猜"出来的,不可能保证 100% 正确安全。Runtime 把执行环境关进 Docker 容器,本质上是在给"猜错了怎么办"这件事上保险。

为什么执行必须隔离

Agent 的"手"(Runtime)要做的事情包括跑 shell 命令、执行代码、改文件。这些操作都来自 LLM 的生成结果——而 LLM 会犯错。 可能生成的命令语法对但语义错(比如把 rm -rf ./build 误写成 rm -rf / 的变体), 也可能在处理不受信任的网页内容、第三方代码时,被注入进恶意指令(prompt injection 导致的间接攻击)。 如果这些命令直接跑在宿主机上,一次"猜错"就可能删掉重要文件、影响其他正在运行的服务,甚至波及同一台机器上其他用户的会话。

核心问题:你没办法要求 LLM 生成的每一条命令都绝对正确安全,那能做的就是限制"猜错了"的爆炸半径。 把执行环境放进一个隔离的 Docker 容器,就是把"最坏情况"框定在容器内部——容器可以被销毁重建,宿主机和其他会话不受影响。

真实实现:DockerWorkspace 怎么把容器接进来

在 openhands-workspace 包里,Docker 隔离由 DockerWorkspace 类实现(继承自 RemoteWorkspace)。 它的工作方式不是"把 Agent 进程本身塞进容器",而是:启动一个运行着"Agent Server"镜像的容器,等它健康检查通过, 再通过 HTTP API 跟容器内部通信。

workspace/docker/workspace.pypython · 节选自真实源码
class DockerWorkspace(RemoteWorkspace):
    """Remote workspace that sets up and manages a Docker container.

    This workspace creates a Docker container running a pre-built OpenHands agent
    server image, waits for it to become healthy, and then provides remote workspace
    operations through the container's HTTP API.
    """

    # 容器内的工作目录,和宿主机上 Agent 进程所在的目录是两回事
    working_dir: str = Field(default="/workspace")

这个细节很重要:Agent 的决策进程可以运行在你的笔记本上,而真正执行命令的地方是远端容器里的一个 Agent Server 进程, 两者之间用一份轻量的 REST/WebSocket 协议通信。这也解释了为什么"整体架构总览"一篇里 Workspace 能做到 "换个配置就切换执行环境"——本质上就是换了一个 HTTP 服务端点而已,Agent 侧的调用方式完全不变。 同一个抽象基类下还有 ApptainerWorkspace(HPC 场景常用的容器方案)和 CloudWorkspace,思路一致。

比容器隔离更细粒度的一层:按风险等级要求人工确认

容器只解决"最坏情况的爆炸半径",不能解决"这个操作到底该不该做"——比如"删除一个文件"在沙箱里执行是安全的, 但如果这个文件是用户唯一没有备份的重要数据,容器隔离并不能阻止这种"合法但危险"的操作。 OpenHands SDK 为此单独设计了一层风险分级 + 人工确认机制(openhands/sdk/security/):

security/risk.pypython · 节选自真实源码
class SecurityRisk(str, Enum):
    """Security risk levels for actions."""
    UNKNOWN = "UNKNOWN"
    LOW = "LOW"
    MEDIUM = "MEDIUM"
    HIGH = "HIGH"
security/confirmation_policy.pypython · 节选自真实源码
class ConfirmRisky(ConfirmationPolicyBase):
    # 默认阈值是 HIGH:风险等级达到 HIGH 才需要人工点头,LOW/MEDIUM 自动放行
    threshold: SecurityRisk = SecurityRisk.HIGH
    confirm_unknown: bool = True  # 风险等级判断不出来时,保守起见也要求确认

每个 ActionEvent 在执行前都可以先经过一次风险评估(由 LLM 自身预测,或专门的分析器判断), 再对照 ConfirmationPolicyBase 的具体策略(AlwaysConfirm 全部确认 / NeverConfirm 全自动 / ConfirmRisky 只在超过阈值时确认)决定要不要暂停下来等人类点头。 这和容器隔离是两条互补的防线:容器管的是"万一执行出问题,影响范围有多大";风险确认管的是"这个操作即使成功执行,后果是否可接受"。

陷阱:沙箱不是银弹

容器隔离降低风险,但不等于零风险。生产级系统至少还要考虑以下几层加固,单纯"套一层 Docker"并不够—— 这也是为什么 SDK 里专门有一个名为 defense_in_depth(纵深防御)的子模块,而不是只靠一层容器:
  • 容器逃逸风险——容器隔离依赖内核层面的命名空间和 cgroup,历史上出现过容器逃逸漏洞,不能假设容器天生绝对安全,需要跟进安全补丁、避免用特权模式启动容器。
  • 资源限制——需要给容器设置 CPU、内存配额,否则一个失控的死循环脚本可能耗尽宿主机资源,影响同机的其他会话。
  • 网络访问控制——容器如果能不受限地访问外网,沙箱内的代码理论上可以对外发起攻击(比如被诱导执行了一段恶意生成的爬虫/攻击脚本),需要按需限制出站网络访问。
  • 执行超时与风险确认——给单次命令执行设置超时,配合上面讲的风险分级确认机制,在容器之外再加一层"人类兜底"。

换句话说,沙箱解决的是"把爆炸半径限制在一个可控范围内",风险确认解决的是"操作本身该不该被允许", 是一套多层防护,而不是加一层容器就能一劳永逸的问题。

联系:这是"直接执行模型生成代码"这个老问题的工业级解法

"AI 框架"Tab 讨论 smolagents 时提到过一个尖锐的问题:让 LLM 直接生成并执行代码,天然存在安全风险——毕竟你没法完全信任模型的输出。 OpenHands 面对的是同一个问题,只是场景更重(要跑任意 shell 命令、装依赖、操作浏览器,而不只是执行一段 Python 代码片段)。 它给出的答案就是这一篇讲的:不追求模型输出绝对正确,而是用运行时的隔离机制兜底——这也是"安全性"在 Agent 系统设计里的通用思路:与其试图让决策层完美无缺,不如在执行层建立可靠的护栏。
