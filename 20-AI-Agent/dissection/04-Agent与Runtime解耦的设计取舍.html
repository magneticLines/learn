Agent 与 Runtime 解耦的设计取舍进阶

前三篇分别看了整体架构、事件流机制、沙箱隔离。这一篇跳出具体机制,站在架构设计的高度问一个问题:把决策和执行拆成两个独立组件,到底值不值。

解耦带来了什么好处

OpenHands 把"决策逻辑"(Agent)和"执行环境"(Runtime)设计成两个通过 Action/Observation 标准协议通信的独立组件, 而不是把两者写在同一个进程、同一套调用栈里。这个决定带来三个明显的好处:

  • Runtime 可以水平扩展。多用户场景下,每个用户的任务会话对应一个独立的 Runtime 实例(独立的容器), 互相之间没有共享状态,天然支持并发扩容——需要支持更多并发用户,加更多 Runtime 实例就行,不需要改 Agent 的任何代码。
  • Runtime 的具体实现可以替换。只要遵守同一套 Action/Observation 协议,Runtime 背后到底是本地 Docker、 远程沙箱服务、还是云端 VM,对 Agent 完全透明。今天用本地容器,明天想换成托管的远程执行服务,Agent 的决策逻辑不需要动一行。
  • 安全加固可以独立进行。资源配额、网络策略、执行超时这些"上一篇"提到的多层防护,都是 Runtime 内部的事情, 可以单独迭代、单独测试,不会牵扯到 Agent 的推理逻辑,职责边界很干净。

解耦的代价

没有免费的架构。解耦换来了灵活性和可扩展性,但也实实在在地引入了三类成本:
  • 通信开销。Agent 和 Runtime 不再是一次函数调用那么轻——Action 和 Observation 需要被序列化(比如变成 JSON)、 通过某种传输机制(进程间通信、网络请求)在两个组件之间传递,再反序列化回结构化数据。相比直接的内存内函数调用,这多出来的序列化和网络往返是实打实的延迟成本。
  • 调试复杂度上升。出问题时,你不能只看一份调用栈——需要同时关注 Agent 侧"为什么决定发这个 Action" 和 Runtime 侧"这个 Action 执行时到底发生了什么"两边的日志,再靠 EventStream 里的因果关系把两边对上,排查链路变长了。
  • 职责边界容易模糊。比如"命令执行超时该谁来判断"——是 Agent 在决策层设置一个等待上限,还是 Runtime 在执行层强制熔断？边界设计得不好,容易出现两边都做了一部分、逻辑重复,或者两边都没做、留下漏洞的情况。这种"到底该谁负责"的判断, 在解耦架构里是需要持续打磨的设计难点,而不是拆开组件就自动解决的。

类比:前后端分离

这套取舍并不陌生。Agent/Runtime 的解耦,本质上跟 Web 开发里"前后端分离"是同一种架构思路—— 前端(Agent)只管展示和决策("用户想要什么、下一步该做什么"),后端(Runtime)只管真正执行操作("落库、跑逻辑、返回结果"), 两者通过一套约定好的接口(HTTP API ↔ Action/Observation)通信。好处一样:前后端可以独立部署、独立扩容、各自替换技术栈; 代价也一样:多了一层网络往返,调试要同时看两端日志,接口契约设计不好就会出现"这个校验到底前端做还是后端做"的扯皮。 理解了前后端分离的取舍,就能直接类比理解 Agent/Runtime 解耦的取舍。
