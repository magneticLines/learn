如何评价一个大模型的好坏

"这个模型好不好"从来不是一句话能回答的问题。理解主流评测基准衡量的是什么、它们的局限在哪,才能明白为什么真正的生产系统往往需要自建评测集,而不是只看榜单排名。

常见的公开评测基准

模型发布时的"跑分"通常来自一批标准化的公开测试集,各自聚焦不同能力维度:

  • MMLU(Massive Multitask Language Understanding)——覆盖数学、历史、法律、医学等 57 个学科的选择题测验, 衡量模型的多学科知识广度。
  • HumanEval——一组编程题,要求模型生成函数实现,再用单元测试自动判断代码是否正确运行, 衡量代码生成的正确率。
  • GSM8K——小学难度的数学应用题集合,看似简单,却能有效检验模型是否具备多步骤推理能力 (很多模型知识渊博但推理链条一长就会算错)。
  • Chatbot Arena——和前面几个"有标准答案"的基准不同,它是让真人用户同时和两个匿名模型对话, 再投票选出哪个回答更好,通过大量投票统计出一个 Elo 分数,衡量的是人类主观偏好而非客观正确率。

评测基准的局限

局限一:榜单刷分与真实场景脱节。公开评测集的题目和答案往往也在互联网上流传,如果训练语料不小心"背过"了测试题 (业内称为数据污染 / benchmark contamination),模型在榜单上的高分可能只是"记住了答案", 而不是真的具备对应能力,一到略微换个问法的真实场景就露馅。
局限二:单一分数掩盖具体短板。MMLU 综合分 80 分,可能是模型在人文学科上考了 95 分、但在需要精确计算的学科上只有 60 分平均出来的结果。 如果你的业务场景恰好命中它的薄弱环节,总分再高也没用——评测需要看细分维度,而不是一个笼统的总分。
局限三:榜单更新快,时效性有限。大模型迭代速度极快,今天登顶的模型,三个月后大概率已经被新模型超越。 把某次评测结果当成"永久结论"去做技术选型,是很常见的错误。

业务场景下更靠谱的做法:自定义评测

公开榜单衡量的是模型的通用能力,但你的产品往往只用到模型能力的一个切面——比如"能不能正确调用某个内部工具"、 "生成的 SQL 语句是否可以直接执行且结果正确"、"输出的 JSON 是否总能被程序稳定解析"。这些问题,MMLU 分数完全回答不了。

更靠谱的做法是:针对自己的实际任务,构造一个小规模但贴近真实分布的测试集(哪怕只有几十到几百条, 只要覆盖了典型场景和边界情况),然后用两种方式打分:

  • LLM-as-judge——用另一个能力更强(或者经过校准)的模型,对照评分标准给每条回答打分, 优点是能规模化、快速跑完整个测试集,缺点是裁判模型本身也可能有偏好偏差,需要定期用人工抽检校准裁判的判断是否可信。
  • 人工抽检——对关键场景、复杂 case 保留人工复核,尤其是涉及安全、合规、金额计算等高风险输出。
关键心法:公开榜单回答的是"这个模型整体聪明吗",自定义评测回答的是"这个模型在我的场景下好用吗"—— 两者不是替代关系,而是"先用公开榜单做初筛,缩小候选范围,再用自定义评测集做最终决策"的组合拳。

和 Agent 评估的关系

这里的方法论会在后面"Agent 开发核心"一节被直接复用——评估一个 Agent 好不好,同样不能只看"它能不能正常回答问题", 而要构造针对性的测试场景(比如"给定这个用户请求,Agent 是否选对了工具、参数是否正确、是否在合理步数内完成任务"), 并且同样会用到 LLM-as-judge 去自动化评估 Agent 的多步执行轨迹是否合理。评价 LLM 和评价 Agent,本质上是同一套"业务对齐的评测思路"在不同粒度上的应用。
