什么是 Agent

Agent 不是一个新模型,而是一种"用法"——把 LLM 从一个只会一问一答的对话框,改造成一个能自己拆解任务、调用工具、观察结果、决定下一步的执行体。

先从一个具体场景说起

假设你让 ChatGPT"帮我查一下今天北京的天气,然后告诉我要不要带伞"。普通 Chatbot 的回答是: "抱歉,我无法访问实时数据,建议你自己查天气预报。" 模型的知识被训练数据的截止日期锁死,它没有"手"去联网查东西,也没有"脑子"去规划"先查天气、再判断是否下雨、再给建议"这三步动作。

而一个 Agent 系统会这样处理:①拆解任务——识别出这个请求需要先获取实时天气数据; ②调用工具——真的去请求一个天气 API;③观察结果——拿到"北京今天有雨,25℃"; ④再做一次推理——结合"有雨"这个事实生成"建议带伞"的最终回答。这四步循环,就是 Agent 和普通 Chatbot 最本质的区别。

三个层面的定义

层面说明
大脑(LLM) 负责推理、拆解任务、决定"下一步该做什么",是整个 Agent 的决策核心
手(工具/Tools) 让 LLM 具备操作真实世界的能力——查天气、搜网页、读写文件、调用数据库、执行代码
记忆(Memory) 记住之前做过什么、观察到什么,避免每一步都是"失忆重来",支撑多轮、长任务的连贯性

这三者组合成一个循环而不是一次性调用:LLM 推理 → 决定调用某个工具 → 拿到工具返回结果(观察) → 把观察结果喂回给 LLM 继续推理 → 直到任务完成或者达到某个终止条件。这个循环在业界有个专门的名字叫 ReAct(Reason + Act), 会在下一个 Tab(Agent 开发核心)里详细展开。

Agent vs 传统程序 vs Chatbot

传统程序Chatbot(纯对话)Agent
控制流 开发者预先写死 if/else,分支路径是确定的 模型直接生成一段文本回复,没有"下一步动作"的概念 由 LLM 在运行时动态决定下一步调用哪个工具、要不要继续循环
与外部世界交互 通过预先写好的接口调用,路径固定 通常没有,只能基于训练数据里"记住"的知识回答 能主动调用工具获取实时信息、执行副作用(发邮件、下单、写代码)
任务复杂度 只能处理开发者事先想到的场景 单轮或多轮问答,不具备"分步执行长任务"的能力 能把一个模糊的大目标,自己拆成多个可执行的小步骤
类比:传统程序像一份提前写死的菜谱,按步骤走绝不会错但也绝不会应变; Chatbot 像一个博学但"四肢瘫痪"的顾问,只能动嘴不能动手; Agent 则像一个带着工具箱的实习生——会自己判断"这个问题需要先查资料还是先跑个脚本", 执行完一步会看看结果对不对,再决定下一步怎么走。

常见误区

误区一:"套了个工具调用就是 Agent"。 单次工具调用(比如 ChatGPT 插件查一次天气就直接回答)只是"增强型 Chatbot", 真正的 Agent 强调的是循环——能根据上一步的观察结果,自主决定要不要再调用一次工具、调用哪个工具, 而不是"调用一次工具、拼接结果、结束"。
误区二:"Agent 越自主越好"。 完全不受约束的自主循环,在生产环境里非常危险——可能无限循环消耗 token、 可能在拆解任务时跑偏、可能在调用有副作用的工具(比如删除文件、转账)时闯祸。 实际工程中几乎都会加上最大步数限制、人工确认关键操作等护栏, 这些"如何管住 Agent"的难点会在"Agent 开发核心"Tab 里专门讨论。

和后面内容的联系

  • Agent 的"大脑"依赖 LLM 的推理能力,所以先要过一遍 LLM 基础(Prompt/Token/Context Window)和训练方法,才能理解 Agent 能力的上限从哪来。
  • ReAct 循环、工具调用、记忆机制是"Agent 开发核心"Tab 的重点,是把这里的概念落到代码层面的具体机制。
  • 后面拆解 OpenHands 时会看到,一个工业级 Agent 系统本质上就是把"大脑-手-记忆"这三层,用工程手段做了隔离和可靠性加固。
