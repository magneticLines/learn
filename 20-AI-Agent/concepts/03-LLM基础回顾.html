概念与AI基础

LLM 基础回顾:Prompt / Token / Context Window

在写 Agent 代码之前,先把和"输入输出"最直接相关的三个词钉死:Prompt 到底长什么样、Token 怎么计费、Context Window 的边界在哪。这三者是后面所有工程约束的源头。

Prompt 不是"一句话",而是一份结构化输入

初学者常把 Prompt 理解成"你打在输入框里的那句话",但对模型 API 来说,Prompt 实际上是一个带角色的消息列表—— 每条消息都有一个 role(角色)和 content(内容)。常见的三种角色:

  • system——设定模型的身份、行为约束、输出格式要求,通常只出现一次,贯穿整个对话
  • user——用户实际输入的问题或指令
  • assistant——模型之前的回复,多轮对话中会被重新传回去,让模型"记得"自己说过什么
多轮对话的 message 结构(JSON 片段)
[
  { "role": "system", "content": "你是一个只回答天气相关问题的助手" },
  { "role": "user", "content": "北京今天天气怎么样" },
  { "role": "assistant", "content": "北京今天多云,25℃" },
  { "role": "user", "content": "那要不要带伞" }
]

关键点在于:模型本身没有"记忆",它每次调用都是无状态的——第二轮问"那要不要带伞"时,模型能听懂"那"指代什么, 完全是因为整个历史消息数组被原样重新传了一遍。这也是为什么多轮对话越长,每一次调用要传的内容就越多—— Agent 里的"记忆机制"要解决的核心问题之一,正是"历史太长该怎么裁剪、摘要、检索",而不是无脑全部塞进去。

Token:计费和截断的最小单位

上一篇提到过,模型看到的不是"字"而是 Token。这里从工程视角再强调两个直接影响开发的事实:

  • 计费按 Token 算——输入(prompt token)和输出(completion token)通常分别计价,而且输出往往比输入更贵。 Agent 每多循环一轮,就要把之前的工具调用结果也当作输入重新传一遍,Token 消耗是滚雪球式增长的。
  • 截断按 Token 算,不是按字符数——当你的输入超过模型上限时,系统会按 Token 边界截断,可能截断在一个词/字的中间语义单元上, 导致模型"看到"的是不完整甚至错乱的内容。
常见误区:不要用字符数或字数去估算成本和长度限制,一定要用 Token 数。中文一段 1000 字的文本, Token 数可能在 1300~2000 之间浮动(取决于分词器),用"字数"去逼近上下文上限,很容易实际调用时才发现已经超限报错。

Context Window:模型一次能"看到"多少

Context Window(上下文窗口)的本质,是模型单次推理时能够接纳的 Token 总量上限——包括输入的所有历史消息、 system prompt、工具定义、以及即将生成的输出,全部加起来不能超过这个数字。比如一个 128K 上下文窗口的模型, 输入 + 输出的 Token 总数不能超过 128000。

这个限制的根源在上一篇提到的注意力机制——计算量随序列长度呈平方级增长,窗口越大,推理成本和延迟也越高, 这不是一个可以无限放大的免费资源。

陷阱:上下文窗口越大越好?很多人看到"这个模型支持 100 万 Token 上下文"就认为可以放心地把所有资料一股脑塞进去, 但这忽略了两个现实问题:
  1. 成本——上下文越长,每次调用的输入 Token 费用越高,哪怕只问一个简单问题,也要为无关的历史内容付费。
  2. Lost in the middle——研究发现,当相关信息被放在很长上下文的中间部分时, 模型对它的利用率明显下降,反而是开头和结尾的信息更容易被"注意到"。也就是说,"塞得下"不等于"记得住、用得好"。
工程启示:面对上下文限制,常见的应对策略包括——只保留最近几轮对话 + 摘要更早的历史、 用 Embedding 检索出真正相关的片段再拼接进 Prompt(即 RAG)、把关键指令放在开头或结尾而不是中间。 这些策略会在后面"开发难点"一节的"上下文爆炸"话题里具体展开。

三者的关系

Prompt(结构化消息列表:system/user/assistant)
  ↓ 全部拼接后按 Tokenizer 切分
Token 序列 —— 计费和长度限制的最小单位
  ↓ 总量不能超过
Context Window —— 单次推理能接纳的 Token 上限(输入+输出)

和前后内容的联系

  • 上一篇讲的 Tokenization 是这里"Token 计费"的底层机制,Embedding 则是解决"上下文塞不下"问题的关键技术(RAG)。
  • Context Window 的有限性,是"Agent 开发核心"里"上下文爆炸"这个开发难点的根本原因——多轮工具调用会不断消耗窗口空间。
