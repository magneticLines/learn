Prompt Engineering 基础

同一个意图,换一种表达方式,模型的输出质量可能天差地别。Prompt Engineering 就是研究"怎么把话说清楚",让模型稳定输出你真正想要的结果,这是后面 Agent 循环、工具调用能可靠工作的地基。

为什么写 Prompt 是一门技术活

模型对措辞、示例、结构的敏感程度,远超大多数人的直觉。同样是让模型"帮我总结这段文字", "总结一下"、"用三句话概括核心观点"、"按重要性排序列出三个要点"这三种问法,得到的结果结构和质量可能完全不同—— 模型没有"读心术",它只能根据你给出的文字线索去猜测你真正想要的格式和详细程度。这也是为什么"调 Prompt"会成为一项专门的技能: 它本质上是在用自然语言,尽可能消除意图的歧义。

核心技巧一:角色设定(System Prompt)

通过 system 角色的消息,给模型设定一个身份、行为边界、输出风格约束,让它在整个对话过程中都"扮演"这个角色。 比如"你是一个只回答 Java 技术问题的助手,遇到无关问题礼貌拒绝,回答时优先给出代码示例"。

角色设定的价值不只是"营造氛围",更实际的作用是圈定任务边界——减少模型自由发挥的空间, 降低它跑题、啰嗦、输出不相关内容的概率。

核心技巧二:Few-shot(小样本示例)

比起用大段文字描述"我要什么格式",直接给模型看几个"输入 → 输出"的示例(即 Few-shot),往往比抽象描述更有效—— 模型擅长"模式匹配",给它看范例,它能更准确地照猫画虎。

Few-shot 示例:引导模型按固定格式抽取信息
// 输入:"张三,28岁,来自北京" → 输出:{"name":"张三","age":28,"city":"北京"}
// 输入:"李四,35岁,来自上海" → 输出:{"name":"李四","age":35,"city":"上海"}
// 输入:"王五,42岁,来自深圳" → 输出:

给出前两组"示范",模型看到第三个输入时,几乎不需要额外说明就能照着同样的 JSON 结构输出——这比写一大段"请按照 JSON 格式输出, 字段包括 name、age、city"的文字描述更直接、更不容易出偏差。

核心技巧三:思维链(Chain-of-Thought)

对于需要多步推理的复杂问题,直接要求模型"给出答案"往往容易出错,因为模型是逐词生成的,一旦跳过中间推理步骤直接蹦出结论, 相当于省略了本该展开的计算过程。思维链(CoT)技巧就是显式引导模型"先一步步想清楚,再给出最终答案"—— 比如加一句"请一步步推理,再给出最终结论"。

原理并不玄乎:因为模型生成是自回归的(每个词的生成都依赖前面已经生成的内容),让它先把中间推理过程"写出来", 相当于把这些中间结果也纳入了后续生成的上下文,后面的答案就能"站在"这些中间步骤之上,而不是凭空猜一个结论。

核心技巧四:结构化输出约束

当模型的输出要被程序解析(而不是给人读)时,必须明确要求它按固定格式输出——比如"只输出一个 JSON 对象,不要包含任何额外解释文字, 字段为 {"action": string, "params": object}"。很多模型 API 还提供了专门的 结构化输出 模式(如约束模型必须生成符合某个 JSON Schema 的内容), 比单纯靠文字描述格式要求更可靠。

为什么这对 Agent 至关重要:Agent 的"决定下一步调用哪个工具"这个动作,本质上就是要求模型输出一段能被代码稳定解析的结构化内容 (工具名 + 参数)。如果输出格式不稳定,哪怕模型的"判断"是对的,程序也可能因为解析失败而整个执行链路崩溃—— 结构化输出约束是后面 ReAct 循环、工具调用能顺利运转的前提条件。

常见误区

Prompt 不是"咒语"。网上流传的各种"神奇话术"(比如"如果你回答得好我给你小费")效果并不稳定, 过度依赖玄学式的措辞技巧,而不去理解任务本身的结构和模型的局限,长期看是不可持续的。 真正靠谱的 Prompt 工程应该更像写单元测试——先明确"这个 Prompt 要处理哪些典型输入和边界 case", 针对每一种情况实际跑一遍验证输出是否符合预期,而不是写完一次"感觉还不错"就上线。

四个技巧的组合示例

system:角色设定,圈定身份和输出边界
  + Few-shot:给出 2~3 个示例,示范期望的输入输出模式
  + 思维链引导:要求复杂问题先展开推理步骤
  + 结构化输出约束:明确最终答案必须是固定格式,便于程序解析
  ↓
稳定、可复现、可被下游代码消费的模型输出

和后面内容的联系

  • 结构化输出约束是"Agent 开发核心"里工具调用机制的直接前提——模型要能稳定输出"调用哪个工具、传什么参数"。
  • 思维链是 ReAct(Reason + Act)循环里"Reason"这一步的具体实现手段,让 Agent 在每次行动前先做显式推理。
  • Few-shot 和角色设定的思路,后面在给 Agent 设计 system prompt、编写工具描述时会被反复用到。
