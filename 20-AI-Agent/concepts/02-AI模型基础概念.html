概念与AI基础

AI 模型基础概念

在动手写 Agent 之前,至少要理解 LLM "黑盒"里几个绕不开的词——参数量、Tokenization、Embedding、注意力机制。不需要会推公式,但要建立"为什么它是这样工作的"的直觉。

参数量:模型的"容量"

大语言模型本质上是一个巨大的函数,输入一段文字,输出下一个词的概率分布。这个函数由数以十亿计的参数(权重)组成—— 你可以把参数量粗略理解为"这个函数里有多少个可调节的旋钮"。GPT-3 有 1750 亿参数,现在主流模型普遍在几十亿到万亿级别。

类比:参数量有点像一个人大脑里"能存多少细微规律"的容量。参数越多,模型能编码的语言规律、世界知识、推理模式就越丰富, 但也意味着训练和推理需要更多算力——这也是为什么"小模型跑得快但笨,大模型聪明但贵"的取舍会贯穿整个 Agent 系统的选型。

Tokenization:模型眼里没有"字",只有 Token

模型不会直接读懂"你好世界"这四个汉字,它先要把文本切成一个个 Token(可以粗略理解为"子词"或"字词片段"), 再把每个 Token 映射成一个数字 ID,模型实际计算的是这些数字。英文里 "tokenization" 常被切成 "token" + "ization" 两个 Token, 中文常见按字或常见词切分。

常见误区:很多人以为"1 个 Token ≈ 1 个字",但英文里 1 个 Token 平均约等于 0.75 个单词,中文里 1 个汉字往往会被切成 1~2 个 Token。 这直接影响成本——**API 通常按 Token 计费**,同样一段话,中文消耗的 Token 数往往比看起来的字数更多,这在做 Agent 成本控制时是个绕不开的细节。

Embedding:把语义变成一串数字

Token 的数字 ID 只是一个索引,真正参与计算的是 Embedding——每个 Token 会被映射成一个高维向量(比如 4096 维的一串浮点数)。 这个向量的巧妙之处在于:语义相近的词,向量在空间中的距离也相近。"国王"减去"男人"加上"女人",在向量空间里的结果会非常接近"女王"。

这个特性不只是训练阶段的内部机制,在 Agent 开发中也会直接用到——"长期记忆"模块常见的做法就是把历史对话、文档切片都转成 Embedding 向量存进向量数据库, 检索时把问题也转成向量,找"距离最近"的记忆片段,这就是 RAG(检索增强生成) 的基本原理,后面会在"记忆机制"一节展开。

注意力机制:模型怎么知道"该看哪里"

一句话里的每个词,含义往往依赖上下文。"苹果"在"我买了个苹果手机"和"我吃了个苹果"里指代完全不同的东西。 注意力机制(Attention) 让模型在处理某个词时,能够"回头看"整个输入序列里的其他词,并给每个词分配一个权重, 决定"理解当前这个词时,应该更多参考哪些上下文"。这也是 Transformer 架构(几乎所有主流 LLM 的底层架构)的核心创新。

为什么这对 Agent 很重要:注意力机制的计算量随输入长度呈平方级增长,这直接决定了 Context Window(上下文窗口)是有限且昂贵的资源。 Agent 在多轮循环中会不断把历史对话、工具调用结果塞进上下文,上下文爆炸导致的成本飙升和性能下降,是"开发难点"一节要重点解决的问题。

四个概念怎么串起来

输入文本 "帮我查天气"
  ↓ Tokenization(切成 Token)
Token 序列 [帮, 我, 查, 天气]
  ↓ Embedding(映射成向量)
向量序列 每个 Token → 一个高维向量
  ↓ 注意力机制(在参数构成的网络中逐层计算)
输出 预测下一个最可能的 Token,循环生成整句回复

和后面内容的联系

  • Token 计费机制直接影响"开发难点"里的成本控制话题——Agent 循环越多轮,消耗的 Token 越多。
  • Embedding 是长期记忆(向量库检索)和 RAG 的基础,会在"Agent 开发核心"的记忆机制一节复用。
  • 注意力机制带来的 Context Window 限制,是理解"上下文爆炸"这个开发难点的根本原因。
