多 Agent 协作模式进阶

一个 Agent 包打天下,职责一多 Prompt 就会臃肿、注意力就会分散——让多个专职 Agent 分工协作,是应对复杂任务的另一条路。

为什么不用一个大 Agent 搞定一切

随着任务复杂度上升,给单一 Agent 塞进的职责会越来越多:既要会规划任务,又要会调用十几种工具,还要负责检查自己的输出对不对。 这会导致它的 System Prompt 越写越长、越写越啰嗦——里面混杂着"如何规划""何时调用哪个工具""如何自我检查"等一堆规则, 模型在推理时的注意力被分散到太多互相无关的指令上,反而更容易在某个环节掉链子。

类比:这就像一个团队让一个人身兼产品经理、程序员、测试三个角色——不是不可能,但这个人脑子里要同时装下三套完全不同的思维方式, 效果通常不如让三个人各自专注做好一件事,再通过明确的交接流程配合起来。多 Agent 协作就是把这种"团队分工"的思路搬到 Agent 系统里: 每个 Agent 只负责一个明确的角色,Prompt 更短、更聚焦,推理时不会被无关指令干扰。

三种常见协作模式

① 流水线式(Pipeline)

多个 Agent 按固定顺序排成一条线,前一个 Agent 的输出直接作为后一个的输入。比如"资料收集 Agent → 摘要 Agent → 报告撰写 Agent", 每一环节职责单一、边界清晰,类似工厂流水线的分工模式。

② 主管-worker 模式(Orchestrator-Worker)

一个 Orchestrator Agent 负责理解总目标、把任务拆分派发给多个专职 Worker Agent(比如"搜索 Agent""代码执行 Agent""数据分析 Agent"), 每个 Worker 各自完成子任务后把结果汇报回 Orchestrator,由它负责汇总、判断是否需要进一步调整分工。

③ 辩论式(Debate)

让多个 Agent 针对同一个问题独立作答,再互相质疑、指出对方答案里的漏洞,经过几轮交锋后再给出最终结论, 常用于对准确性要求较高的场景(比如代码审查、事实核查),原理类似"多人评审比一个人自查更容易发现问题"。

流水线式: Agent A → Agent B → Agent C(单向传递)
主管-worker: Orchestrator ⇄ Worker1 / Worker2 / Worker3(分派 + 汇总)
辩论式: Agent A ⇄ Agent B ⇄ Agent C(互相质疑,收敛到共识)

陷阱:多 Agent 不是免费的午餐

引入多 Agent 协作会带来实打实的额外成本:①协调开销——需要额外的逻辑决定"任务该分给谁、什么时候该交给下一个 Agent"; ②消息传递成本——Agent 之间传递的上下文本身也要消耗 Token,多一层传递就多一份开销; ③上下文不一致——如果各个 Agent 拿到的背景信息不完全同步,很容易出现"各说各话"的问题,比如 Worker Agent 基于过时的假设做出了和 Orchestrator 预期不符的判断。 不是所有场景都值得引入多 Agent 的复杂度——如果单一 Agent 配合清晰的 Prompt 和有限的工具集就能稳定解决问题, 往往比拆成多个 Agent 更简单、更容易调试、成本也更低。

和前面内容的联系

多 Agent 协作本质上是把"规划与任务分解"里的思路进一步升级——不只是把大目标拆成小步骤,而是把小步骤的执行主体也拆分给不同的专职 Agent。 每个子 Agent 内部,依然遵循前面几篇讲过的 ReAct 循环、工具调用、记忆机制这套基本运作方式。
