记忆机制:短期上下文 vs 长期记忆(向量库)

Agent 的"记忆"分两层——受 Token 上限约束、随对话增长会被挤掉的短期上下文,和把知识转成向量存起来、按需检索的长期记忆。

短期记忆:其实就是上下文窗口本身

很多人以为 Agent 需要专门的"记忆模块"才能记住之前说过的话,但最基础的短期记忆机制其实非常朴素—— 把完整的对话历史(包括之前的 Thought、Action、Observation)原样拼接进每一次发给模型的请求里。 模型本身是无状态的,它并不会"记得"上一轮对话,每次调用都是一次全新的、独立的推理,记忆感完全来自于"把历史重新喂给它看一遍"。

陷阱:短期记忆会"忘记"早期内容。上下文窗口(Context Window)的大小是有硬性上限的(比如 128K token), 随着 ReAct 循环轮数增多,历史记录会不断膨胀,一旦超出窗口上限,要么请求直接报错,要么早期的内容被截断丢弃。 工程上常见做法是对早期历史做摘要压缩(把第 1~5 轮的对话浓缩成一句话摘要),或者只保留"最近 N 轮 + 关键结论"。

长期记忆:把知识"外部化"存进向量库

如果信息需要跨越多次会话依然被记住(比如用户的长期偏好、过去处理过的历史事实、大量文档知识), 塞进每次都要重新传输的上下文窗口既不现实也不经济。长期记忆的做法是:把这些信息提前转成 Embedding 向量, 存入专门的向量数据库;需要用到的时候,把当前的问题也转成向量,在库里检索"语义距离最近"的若干条记忆, 只把这几条相关的内容拼回本次的上下文,而不是把所有历史一股脑塞进去。

类比:短期记忆像是你脑子里"正在想的这件事"——容量有限,想的东西一多,前面想的就被挤出去了; 长期记忆像是你的笔记本或者搜索引擎——内容存在别处,需要的时候才翻出来查,而不是随时都摆在眼前。

两者对比

短期记忆长期记忆
存储位置 本次请求的上下文窗口(内存中,随请求发送) 外部向量数据库(持久化存储,跨会话保留)
容量 受模型 Context Window 硬上限约束,通常几万到几十万 Token 理论上不受限,可存海量文档、历史记录
检索方式 无需检索,全部原样拼接进请求 按语义相似度检索(Embedding 距离),只取最相关的若干条
典型问题 早期内容被挤出/截断,导致"失忆" 召回内容可能"语义相关但答非所问"

陷阱:向量检索不是精确匹配

向量检索找到的是"语义上接近"的内容,而不是"逻辑上正确回答问题"的内容。比如问"如何取消订单", 可能召回一篇讲"如何修改订单"的文档——两者向量距离很近,但答案完全对不上。 工程上常见的补救方式是混合检索(向量检索 + 关键字检索一起做,互相补充), 或者在检索结果之上再加一层 Rerank(用一个专门的模型对召回结果重新打分排序,把真正相关的排到前面)。 单纯依赖"向量距离最近"作为唯一判断标准,在真实业务场景里往往不够可靠。

向量检索的底层原理,直接复用了"概念与AI基础"Tab 里讲过的 Embedding——语义相近的内容,向量空间里的距离也相近。 如果对 Embedding 的原理还不熟悉,建议先回顾 《AI 模型基础概念》 里对 Embedding 的讲解,再来看长期记忆的检索机制会更容易建立直觉。
