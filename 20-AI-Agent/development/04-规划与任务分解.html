Agent开发核心

规划与任务分解

面对一个模糊的大目标,Agent 要先学会把它拆成一串可执行的小步骤——但拆多细、什么时候拆,本身就是一门需要权衡的学问。

为什么大目标不能直接执行

像"帮我做一份竞品分析报告"这样的请求,并不对应任何一个单一的工具调用。它其实隐含了一连串子任务: 确定要分析哪几个竞品、逐个搜集资料、提炼关键信息、按统一维度对比、最后组织成一份结构化报告。 Agent 要先把这个模糊目标分解成一串具体、可执行的步骤,才能一步步用工具去完成。

两种规划策略

策略做法优点缺点
一次性规划
(Plan-and-Execute)		 先让模型输出完整的步骤列表,再按顺序逐步执行每一步 思路清晰、执行路径可预先审查,方便人工确认关键步骤 计划是在没有任何真实反馈时定的,某一步执行结果和预期不符时,后续步骤容易连锁失效
边执行边规划
(类 ReAct 式)		 每执行完一步,根据这一步的真实观察结果,再决定下一步具体做什么 更灵活,能及时根据实际情况调整方向 缺乏全局视角,容易在探索中"发散"、绕远路,甚至偏离最初目标

实际工程中很多系统会把两者结合:先用一次性规划列出一个粗粒度的大纲(比如"1. 确定竞品名单 2. 逐个调研 3. 汇总对比 4. 输出报告"), 再对每个大纲条目用"边执行边规划"的方式细化执行,兼顾全局方向感和局部灵活性。

陷阱:分解粒度的取舍

粒度太粗:比如把"调研竞品 A"当作单独一步直接丢给模型执行,模型可能要在一次工具调用里同时做"搜索资料 + 提炼要点 + 结构化输出"多件事, 单步任务过重,出错概率高,而且出错后不好定位到底是哪个环节的问题。

粒度太细:把每个动作都拆到"搜索关键词1 → 搜索关键词2 → 提取第一段 → 提取第二段 → ..."这种程度, 步骤数量会急剧膨胀。这里有个容易被忽视的数学直觉:如果每一步执行成功的概率是 95%,拆成 5 步的整体成功率约为 0.95^5 ≈ 77%, 拆成 20 步则骤降到 0.95^20 ≈ 36%——步骤越多,误差累积效应越明显,整体成功率反而越低,同时每一步都是一次模型调用,成本和延迟也线性叠加。
关键心法:任务分解没有一个万能的"最优粒度",实践中通常以"每一步的失败大概率不会波及全局、失败了容易重试或修正"作为粒度参考, 而不是一味追求"拆得越细越可控"。

和前面内容的联系

规划与任务分解,本质上是把"01-ReAct 循环"里讲的 Thought→Action→Observation 循环,应用在多步骤、复杂任务这个更大的场景下—— ReAct 关注的是单步"思考-行动"如何交替,而规划关注的是这一整串行动该如何组织、拆分、排序。两者不是互斥的两套机制,而是同一套"边做边想"思路在不同粒度上的体现。
