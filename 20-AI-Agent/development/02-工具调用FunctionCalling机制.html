工具调用(Function Calling)机制

模型本身不能"调用"任何东西——Function Calling 的真相是模型只负责"说出"要调用什么、参数是什么,真正的执行永远是外部代码在做。

一个常见的误解

很多人听到"模型调用了一个工具",会以为模型内部真的执行了这段代码。但 LLM 本质上只是一个"输入文本 → 输出文本"的函数, 它没有网络权限、没有文件系统访问权限,也不能真的运行任何程序。Function Calling 的真实机制是:

  1. 开发者预先把每个工具的名称、功能描述、参数 schema(通常是 JSON Schema)一起打包塞进发给模型的请求里;
  2. 模型根据当前上下文,判断"这一步该调用哪个工具、参数该填什么",并输出一段结构化的 JSON(而不是自然语言);
  3. 外部代码解析这段 JSON,真正去执行对应的函数(调 API、查数据库、跑脚本);
  4. 把执行的真实结果重新拼进对话历史,作为下一轮输入喂回模型。
类比:模型更像一个只能"打电话下指令"的甲方——它说"帮我查一下北京的天气,参数是城市=北京", 但电话那头真正"跑腿去查"的是外部代码这个乙方。模型自己永远不会亲自去接触真实世界。

工具定义与模型输出示例

① 开发者提供的工具 Schema
{
  "name": "get_weather",
  "description": "查询指定城市当天的天气情况,返回天气状况和气温",
  "parameters": {
    "type": "object",
    "properties": {
      "city": { "type": "string", "description": "城市名称,如 北京" }
    },
    "required": ["city"]
  }
}
② 模型返回的"调用意图" JSON
{
  "tool_call": {
    "name": "get_weather",
    "arguments": { "city": "北京" }
  }
}
// 外部代码收到这段 JSON 后,真正去调用天气 API,
// 再把结果(如"北京今天有雨")作为 Observation 拼回下一轮请求

整个过程和上一篇"ReAct 循环"里的 Action → Observation 是同一件事的两种描述方式:Function Calling 是 Action 在工程上的具体实现形式, 很多现代 LLM API(如 OpenAI、Claude)都把这套机制原生集成进了接口,不再需要开发者自己解析模型输出的自然语言来猜测"它想调用什么"。

陷阱一:模型会"幻觉"出不存在的调用

模型可能编造根本不存在的函数名,或者给已有函数编造不合法的参数(比如城市名瞎写、传入了 schema 里没有的字段、数值类型传成了字符串)。 这是 LLM 生成式本质决定的——它只是在"模仿"一个看起来合理的调用,并不保证这个调用一定合法。 因此外部代码在真正执行之前,必须做严格校验:函数名是否在白名单里、参数是否符合 JSON Schema、必填字段是否缺失, 校验失败要能给模型一个明确的错误 Observation,而不是让程序直接崩溃或者悄悄地把错误参数传给下游系统。

陷阱二:工具描述本身就是"给模型看的文档"

如果 description 写得含糊(比如两个工具的描述都是"查询数据"),模型很容易选错工具,或者对参数的含义理解错误。 工具描述不是写给同事看的代码注释,而是模型在推理时唯一能依赖的"文档"——需要像写 API 文档一样精心打磨: 说清楚这个工具是做什么的、什么场景该用它、参数的格式和取值范围。工具描述质量的高低,直接决定了 Function Calling 的可靠性上限。

和前后内容的联系

  • 呼应"什么是 Agent"里说的"手"——工具调用就是 Agent 具备操作真实世界能力的具体实现机制。
  • 模型幻觉调用、工具描述不清晰这两个问题,是"开发难点与业界解法"一篇里"工具调用可靠性"话题的直接源头。
