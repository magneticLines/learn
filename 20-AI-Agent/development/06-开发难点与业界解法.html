Agent开发核心

开发难点与业界解法

幻觉、上下文爆炸、工具调用可靠性、成本与延迟——这四个问题贯穿前面五篇内容的始终,是每一个工业级 Agent 系统都绕不开的坎。

四大难点全景

前面五篇分别讲了 ReAct 循环、工具调用、记忆机制、任务分解、多 Agent 协作,但没有一套机制是"零成本"的—— 循环越多轮,出错的机会就越多;工具越丰富,模型选错的概率就越高。这一篇把散落在前面几篇里的问题串起来, 用"问题现象 → 根本原因 → 业界解法"的结构做个收尾总结。

难点一:幻觉(Hallucination)

维度内容
问题现象模型编造不存在的事实,或者调用根本不存在的工具、编造参数
根本原因LLM 本质是基于概率的生成模型,它没有内建"我不知道"的机制——不管知不知道,它都会生成一段"看起来合理"的文本
业界解法用 RAG 引入外部真实信息源,让回答有据可查;要求模型输出时标注信息来源;关键操作强制走工具调用获取真实数据,而不是让模型凭"记忆"编造答案

具体怎么落地

  • RAG 的检索参数不是随便定的:文档切片(chunk)大小一般在 300~500 token 之间,切片间保留 10%~20% 的重叠(overlap), 避免关键信息刚好被切断在两片之间;检索时不要只取 Top-1,通常取 Top 3~5 片喂给模型,让它有交叉验证的余地。
  • 混合检索比纯向量检索更稳:纯语义(向量)检索容易出现"意思相关但答非所问",生产系统常见做法是 向量检索 + 关键字检索(如 BM25)混合召回,再用一个 rerank 模型(如 bge-reranker、Cohere Rerank)对召回结果重新排序取前几名, 牺牲一点延迟换取准确率。
  • 要求模型标注信息来源:在 Prompt 里明确要求"回答中引用的每个事实都要标注来源片段编号(如 [来源:2])", 这样即使模型偶尔编造,也能通过"引用编号对不上检索结果"这个信号被程序自动识别出来,而不是纯靠人工抽查。
  • 关键数据不能"背",必须"查":订单号、金额、库存这类高风险字段,即使模型在对话历史里"见过", 也应该在生成最终答案前再走一次工具调用做实时校验,而不是信任模型从上下文里"记住"的数字——这类数字被模型悄悄改动一位数是幻觉里最难被发现的一种。

难点二:上下文爆炸

维度内容
问题现象多轮 ReAct 循环下来,历史记录越滚越大,逼近甚至超出 Context Window 上限
根本原因每一步的 Thought、Action、Observation 都会被原样追加进上下文,循环轮数和上下文长度成正比增长
业界解法对早期历史做摘要压缩;只保留"最近 N 轮 + 关键结论";把不需要每次都看的信息外部化存进长期记忆(向量库),按需检索而非全量携带

具体怎么落地

  • 先要能算出"现在占用了多少":Python 生态用 tiktoken 库按模型的分词规则精确计数,Java 生态没有官方对应库, 常见做法是调用同一家厂商提供的 tokenizer 服务,或者用"字符数 ÷ 2 估算 Token 数"这种粗略经验值做预警(仅用于触发压缩的阈值判断,不追求精确)。
  • 设置触发阈值,而不是等溢出了才处理:常见做法是当历史内容达到模型 Context Window 的 70%~80% 时就主动触发压缩,而不是等到真的超限报错才补救——报错时机太晚,补救成本更高(可能已经丢了关键信息)。
  • 摘要压缩用更便宜的模型做:没必要用主模型做摘要,通常另起一个更小、更便宜的模型(如 gpt-4o-mini) 把"较早的历史"压缩成一段几句话的摘要,和"最近 N 轮完整对话"拼在一起,构成下一轮真正喂给主模型的上下文——摘要本身产生的调用成本远低于让主模型消化全部历史。
  • 滑动窗口 vs 摘要怎么选:滑动窗口(只保留最近 N 轮,丢弃更早内容)实现简单但会彻底丢失早期信息; 摘要压缩保留更多信息但多一次模型调用的成本和延迟。经验做法是两者结合——摘要负责"讲过什么大方向",滑动窗口负责"最近的确切细节"。

难点三:工具调用可靠性

维度内容
问题现象模型编造参数、参数格式不对、选错了工具
根本原因工具描述写得不够清晰,模型无法准确判断该用哪个工具、参数该怎么填;也可能单纯是模型能力不足导致理解偏差
业界解法用严格的 JSON Schema 做参数校验,不合法就拒绝执行并返回明确错误;失败后允许有限次数重试;把工具集控制在"少而精",避免功能重叠、描述含糊的工具互相干扰模型判断

具体怎么落地

  • 校验库不用自己写:Python 生态直接用 pydantic(定义参数模型,校验失败自动抛出结构化错误)或 jsonschema 库; Java 生态用 networknt/json-schema-validator 这类现成库校验模型返回的参数 JSON 是否符合工具声明的 Schema,不要手写正则去校验结构化数据。
  • 重试要区分"值得重试"和"重试也没用":参数类型错误(比如该传数字传了字符串)重试意义不大,更好的做法是把校验错误信息原样作为 Observation 返回给模型,让模型自己看着错误改正参数重新生成,通常 1~2 次内能收敛;而工具执行本身的网络超时/限流,才是"直接重试"该处理的场景 (对照上一篇 Java 手写最小 Agent 里 LlmClient 的指数退避实现)。
  • 工具描述要给"反例":光说"查询天气,参数是城市名"不够,更可靠的描述会补充参数格式要求和边界情况,比如 "城市名需要用英文拼写,不支持省份或国家名,查询不到会返回空结果"——模型是靠读这段文字做决策的,信息缺口越大,出错概率越高。
  • 工具数量有一个隐性上限:经验上单次暴露给模型的工具数超过 10~20 个后,模型选错工具的概率明显上升。 工具多的系统通常会先用一层轻量的"路由"(可以是规则,也可以是一次单独的分类模型调用)判断任务大类, 再只把该场景相关的一小撮工具喂给主 Agent,而不是把全部工具一次性塞进去。

难点四:成本与延迟控制

维度内容
问题现象多步循环意味着多次模型调用,费用和响应延迟随循环轮数线性叠加
根本原因ReAct 循环的每一轮 Thought 都是一次完整的模型推理请求,不是免费的本地计算
业界解法简单任务路由到更便宜的小模型处理,复杂任务才用大模型;把互相独立的工具调用并行执行而不是串行等待;设置最大循环步数上限,避免 Agent 无限循环消耗资源

具体怎么落地

  • 路由判断可以很轻量:不需要为"判断任务复杂度"专门训练模型,常见做法是用规则(问题长度、是否命中关键词库) 或者用一次极便宜的小模型分类调用(几十个 Token 的输入输出,成本可以忽略不计)判断"这个请求走小模型够不够", 再决定要不要升级到更贵的大模型处理。
  • 独立工具调用要并行,不要串行等:如果一轮里模型同时要求查天气和查日历这两个互不依赖的工具, Java 用 CompletableFuture.allOf(...)、Python 用 asyncio.gather(...) 并发发起,总耗时等于最慢的那一个, 而不是两个耗时相加——这是最容易被忽视、但收益很直接的一个优化点。
  • 步数上限和超时给出经验区间:MAX_STEPS 通常设在 5~10 之间(简单任务 5 步以内应该收敛, 超过 10 步大概率是陷入了某种循环而不是在正常推进);单次 LLM 调用的超时通常设在 30~60 秒,太短容易在模型正常但较慢的响应上误杀, 太长会让用户在一次死等中失去耐心。
  • 把 Token 用量当作一等监控指标:LLM API 的响应体里通常带有 usage 字段(prompt tokens / completion tokens), 生产系统应该把这个数字记录进日志或指标系统,按用户、按任务类型统计,而不是等账单出来才发现某类任务悄悄跑出了几十轮循环—— 这也是"上下文爆炸"和"成本控制"这两个难点在监控层面天然应该合并来看的原因。
关键心法:这四个难点不是孤立的——比如"工具集少而精"既能提升调用可靠性,也顺带降低了成本; "长期记忆外部化"既缓解了上下文爆炸,也让 Agent 具备了跨会话的能力。在设计 Agent 系统时, 与其单独优化某一个指标,不如把这四个问题当作一个整体去权衡取舍。

回顾整个 Agent 开发核心 Tab

  • ReAct 循环是 Agent 运作的基本骨架——边想边做。
  • 工具调用是 Action 落地的具体机制——模型只负责"说",执行永远在外部代码。
  • 记忆机制解决了"记得住多少、记多久"的问题——短期上下文 + 长期向量库。
  • 规划与任务分解解决了"大目标怎么拆"的问题。
  • 多 Agent 协作是把上述机制在更大规模、更复杂分工场景下的延伸。
  • 而这一篇的四个难点,则是把这些机制真正落地到生产环境时,必须正视的工程代价。

与其他知识点的联系

  • OpenHands 源码拆解:这四个难点不是纯理论——OpenHands 的 StuckDetector(见"Agent 与 Runtime 解耦的设计取舍"一篇) 就是"死循环护栏"这个思路在真实生产系统里的落地,它比这里说的"步数上限"更精细,专门识别"重复 Action-Observation"等多种卡死模式, 说明业界解法往往比教学示例更细致。
  • Java 手写最小 Agent:"实战演练"一篇里 LlmClient 的指数退避重试、MAX_STEPS 步数上限、 把工具执行异常兜底成 Observation 而不是直接抛出,都是这一篇"工具调用可靠性"和"成本与延迟控制"两个难点的具体代码实现—— 读完这一篇再回看那份代码,能看出每一行防御性代码背后对应的是哪个具体问题。
  • 框架选型对比与取舍:选框架本质上是在"自己实现这四个难点的解法"和"用框架内置的现成解法"之间做取舍—— LangGraph 内置的状态持久化、AutoGen/CrewAI 的消息路由,某种程度上都是在替开发者解决"上下文爆炸"和"工具调用可靠性"问题, 理解了这四个难点,才能判断框架的某个功能到底在帮你解决什么。
  • 责任链模式:"工具集少而精 + 先路由再执行"的做法,和责任链模式的思路是一致的——不是把所有工具一次性摊开让模型挑, 而是先用一层轻量判断缩小范围,再把请求交给真正该处理它的那个子集,减少每一层需要做判断的候选项数量。
  • 分布式系统里的限流与降级:"简单任务路由到小模型、复杂任务才用大模型"和"429/5xx 才重试、4xx 不重试"这两条经验规则, 分别对应分布式系统里的分级服务和可重试性判断——LLM 调用本质上就是一次不可靠的远程调用, 工程上要面对的问题(超时、限流、重试风暴)和调用任何外部服务 API 是同一类问题,只是多了"结果本身也可能不准确"这一层幻觉风险。
