Agent开发核心

ReAct 循环(Reason + Act)

ReAct 是让 LLM 从"一次性吐出答案"变成"边想边做"的核心机制——让模型显式输出思考,再把行动的真实观察结果喂回去,一步步逼近答案。

为什么需要"边想边做"

如果只是让模型一次性写出"第一步做什么、第二步做什么、第三步做什么"的完整计划,再按计划机械执行,现实中会经常翻车—— 因为计划是在没有任何真实反馈的情况下拍脑袋写出来的。比如计划第二步是"调用天气 API",但如果 API 报错或者返回的城市名对不上, 后面所有依赖这个结果的步骤都会跟着错,而模型对此一无所知,因为它压根没看到中间发生了什么。

ReAct(Reason + Act)论文提出的思路很直接:让模型每一步都输出两件事——Thought(这一步该做什么、为什么这么做的思考过程) 和 Action(具体要执行的动作,比如调用哪个工具)。动作执行后,把真实的执行结果作为 Observation(观察)喂回给模型, 模型基于这个真实观察继续下一轮 Thought。如此循环,直到模型判断任务已经完成,输出 Final Answer。

循环示意

Thought(思考下一步该做什么)
  ↓
Action(执行一个具体动作,如调用工具)
  ↓
Observation(拿到这个动作的真实结果)
  ↓ 喂回给模型,回到
Thought(基于新观察继续思考)
  ↓ ... 重复若干轮 ...
Final Answer(模型判断信息已足够,给出最终答案)

一个简化示例

ReAct 多轮文本示例(简化)
Thought: 我需要先查一下北京今天的天气,才能判断要不要提醒带伞
Action: get_weather(city="北京")
Observation: 今天北京有雨,气温 18~25℃

Thought: 已经知道今天有雨了,需要提醒用户带伞,不需要再调用其他工具
Final Answer: 今天北京有雨,建议出门带伞,气温 18~25℃,注意保暖。

注意这里的关键:第二轮 Thought 是基于第一轮真实拿到的 Observation("有雨")做出的判断, 而不是模型在开始时就已经"预判"好了要不要带伞。如果 Observation 变成"今天晴天",第二轮 Thought 和 Final Answer 会完全不同—— 这正是 ReAct 相比"一次性规划"更灵活、更贴近真实世界的地方。

常见陷阱:先想后做 ≠ ReAct

误区:"模型先输出一段完整计划,再按计划顺序执行"就是 ReAct。 这其实是"一次性规划"(Plan-and-Execute),和 ReAct 有本质区别。一次性规划的所有步骤是在看不到任何真实执行结果的情况下写死的, 一旦中间某一步的真实结果和预期不符(API 报错、数据格式变了、查到的信息和假设的不一样),后面的步骤大概率会跟着失效, 而模型此时并不会重新调整计划。ReAct 的核心价值恰恰在于每一步的思考都锚定在上一步的真实观察结果上, 做到"边做边想",能及时根据实际情况调整下一步动作。

三种模式对比

模式是否显式思考是否根据真实反馈调整典型问题
纯 Chain-of-Thought 是,输出详细推理过程 否,不调用外部工具,全靠模型自身知识推理到底 没有外部信息校验,容易在需要实时数据或精确计算的任务上出错
纯 Act 否,直接输出动作,不显式暴露思考过程 是,但决策过程是"黑盒",难以调试和追溯为什么选了这个动作 出错时难以定位是哪一步的判断出了问题,可解释性差
ReAct 是 是,每步思考基于上一步真实 Observation 需要更多轮模型调用,上下文会随循环增长(见"开发难点"一节)
关键心法:ReAct 把模型的"思考过程"显式写出来,不仅仅是为了让模型自己想得更清楚(有点像人做数学题写演算过程), 更重要的是这段 Thought 会被持久化在日志里,让开发者能追溯 Agent 每一步"为什么这么做",这对调试和信任建立至关重要。
