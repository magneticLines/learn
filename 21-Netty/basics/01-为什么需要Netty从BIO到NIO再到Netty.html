为什么需要 Netty(BIO → NIO → Netty)

不先搞清楚 Netty 解决了什么问题,直接背 ChannelPipeline、EventLoop 这些概念只会是空中楼阁。

📖 来源:《Netty in Action》 第 1 章「Netty——异步与事件驱动」

一、BIO 模型:一个连接一个线程,撑不住高并发

Java 最早提供的网络编程模型是 BIO(Blocking IO,同步阻塞 IO)。它的编程模型非常直观:服务端 accept() 到一个连接,就为它分配一个线程,这个线程专门负责这条连接上的 read()/write()——读不到数据就一直阻塞在那儿,不会返回。

BIO 典型写法
while (true) {
    Socket socket = serverSocket.accept(); // 阻塞,直到有新连接
    new Thread(() -> {
        // 这个线程专属于这条连接,read() 会一直阻塞到有数据可读
        InputStream in = socket.getInputStream();
        in.read(buffer); // 阻塞点
    }).start();
}

这个模型的痛点不在于"写起来复杂"——恰恰相反,它是所有模型里最好懂的。痛点在于线程本身的成本:一条连接占一个线程,而线程在 Linux 上通常对应一个轻量级进程,默认栈大小 512KB~1MB,操作系统调度它需要上下文切换开销。当并发连接数从几百涨到几万(即所谓 C10K 问题),线程数量跟着涨到几万,光是线程本身占用的内存和调度开销就能把机器拖垮——而绝大多数连接在绝大多数时间里其实什么数据都没有,线程却在死等,白白占着资源。

本质矛盾:BIO 把"连接数"和"线程数"死死绑在一起。连接是可以轻松开到几万的(内存占用很小),但线程不行——这是 BIO 在高并发场景下的天花板。

二、NIO 带来的转机:一个线程管理成千上万个连接

JDK 1.4 引入的 NIO(New IO / Non-blocking IO) 换了个思路:不再让线程阻塞等某一条连接的数据,而是用一个 Selector 同时"盯着"一大批 Channel,问操作系统"这些连接里哪些已经就绪、可以读写了",然后只处理真正就绪的那些。这在操作系统层面依赖的是 select/poll/epoll 这类 IO 多路复用机制(详见 11-操作系统/IO多路复用)。

这样一来,一个线程就能同时服务成千上万个连接——因为线程不再为某一条连接死等,而是循环处理"当前就绪"的那一批,时间都花在真正有数据可处理的连接上。这正是 Redis、Nginx、Netty 这类高并发中间件的共同底座。

类比:BIO 像餐厅给每桌配一个服务员,服务员就站在那桌边上等顾客点单,哪怕这桌客人还在看菜单发呆;NIO 像餐厅只有几个服务员来回巡视,谁的手举起来了(数据就绪了)就去服务谁——同样的服务员数量,能覆盖多得多的桌数。

三、原生 NIO 的三个真实痛点

既然 JDK 已经提供了 NIO,为什么还要 Netty?因为原生 NIO API 好用的部分只是表面,深挖下去全是坑:

痛点具体表现
API 复杂、心智负担重Selector/SelectionKey/Channel/ByteBuffer 一套概念要正确组合才能跑通,稍不注意就漏读、漏写或者死循环。
臭名昭著的 Epoll 空轮询 BugJDK 某些版本的 Selector 在 Linux 上会出现 select() 没有事件也立刻返回的 bug,导致 CPU 被空转到 100%,需要业务代码自己检测并重建 Selector 来规避。
没有现成的拆包/粘包处理与协议编解码框架TCP 是流式协议,"一次 read 读到多少字节"和业务上"一条消息的边界"是两回事,原生 NIO 完全不管这个,得自己从零手写。
陷阱提醒:不少初学者以为"用了 NIO 就自动获得高性能",直接拿 Selector 手撸一个服务端。真实项目里几乎没人这么干——不是因为 NIO 不够快,而是因为把它做稳、做对(处理好半包/粘包、异常断连、内存零拷贝、Bug 规避)所需的工程量,远超大多数团队愿意投入的成本。这正是 Netty 存在的意义。

四、Netty 的定位:把"正确"和"好用"都封装好的 NIO 框架

Netty 不是发明了新的 IO 模型,它的价值在于:在 JDK NIO 之上,把所有那些容易做错、做繁琐的部分封装成了统一、易用、经过大规模生产环境验证的 API。具体体现在:

  • 统一的 Channel 抽象——屏蔽 NIO/OIO(阻塞 IO)的差异,业务代码基本不用关心底层用的是哪种 IO 模型。
  • ChannelPipeline + ChannelHandler 职责链——把"编解码"“业务处理”“异常处理”拆成一个个可插拔的 Handler,而不是全部堆在一个巨大的 if-else 里。
  • 健壮性修复——自动检测并规避上面提到的 Epoll 空轮询 Bug 等一系列 JDK 层面的已知问题。
  • 零拷贝的 ByteBuf——比 JDK 的 ByteBuffer 更好用(支持自动扩容、读写指针分离),内存分配上还提供了池化机制减少 GC 压力。
  • 开箱即用的编解码器——常见的拆包策略(定长、分隔符、长度字段)、HTTP、WebSocket 等协议都有现成实现。

所以准确地说:Netty 是一个基于 NIO 事件驱动模型、把工程实践中的坑都填平了的高性能网络编程框架,而不是一种全新的 IO 技术。业界大量中间件——Dubbo、RocketMQ、Elasticsearch(部分模块)、gRPC-java 的底层网络层——都是直接构建在 Netty 之上,而不是自己重新造轮子对接原生 NIO。

小结

  • BIO:一条连接绑一个线程,并发连接数受限于操作系统能承受的线程数,C10K 问题的根源。
  • NIO:通过 Selector + IO 多路复用(select/poll/epoll),用少量线程管理海量连接,解决了并发瓶颈。
  • 原生 NIO 的坑:API 繁琐、有已知 Bug(空轮询)、没有编解码/拆包框架,直接用于生产的工程成本很高。
  • Netty 的定位:在 NIO 之上封装一套统一、健壮、开箱即用的网络编程框架,而非另起炉灶的新技术。

与其他知识点的联系

  • 操作系统 IO 多路复用(select/poll/epoll):NIO 能用一个线程管理海量连接,底层就是靠这套系统调用——理解了 select/poll/epoll 的原理,才能明白 Netty 的 NioEventLoop 内部到底在"等"什么、"通知"了什么。
  • Java 并发编程:BIO 的"一连接一线程"本质上是线程池/多线程编程的一种应用场景,而 Netty 的 EventLoop 也是一种特殊的单线程执行器——理解 Java 并发的线程模型,能帮你更快理解 Netty 为什么要设计出 EventLoop 这个概念。
  • Kafka / RocketMQ 等消息中间件:这些系统的网络通信层几乎都是基于 Netty(或类 Netty 的自研 NIO 框架)构建的——学完 Netty 回头看 消息队列 的网络层设计,会有一种"原来如此"的感觉。
  • 系统设计中的高并发架构:C10K 问题及其解法(多路复用、事件驱动)是系统设计里"如何支撑海量连接"的经典考点,和 系统设计 里聊天系统、长连接推送系统的架构选型直接相关。
