Reactor 线程模型与 Netty 的实现

为什么 bossGroup 和 workerGroup 要分开?为什么 workerGroup 默认是 CPU 核数 × 2 而不是 1?这一篇把线程分工的全貌讲清楚。

📖 来源:《Netty in Action》 第 3~4 章「EventLoop 与线程模型」延伸

一、Reactor 模式:事件驱动架构的通用范式

Reactor 模式是一种应对高并发 IO 的经典架构范式,核心思想是:用一个(或多个)Reactor 线程通过 Selector 统一监听一批连接上的事件,一旦某个事件就绪,就把它分发(dispatch)给对应的 Handler 去处理——而不是像 BIO 那样让每个连接各自占用一个线程死等。这不是 Netty 独有的发明,Redis、Nginx、Node.js 的事件循环都是这一模式的不同变体。

Reactor 模式有三种经典演进形态,理解这条演进线,就能理解 Netty 为什么最终选择了它现在的实现:

1. 单 Reactor 单线程

一个线程既做事件监听(accept、read、write 事件的分发),又亲自执行业务逻辑。结构最简单,但致命问题是无法利用多核——CPU 有 8 核,却只有 1 个线程在干活,而且一旦某个业务逻辑处理慢了(比如做了一次耗时的数据库查询),会卡住这唯一的线程,导致所有其他连接的事件都得不到及时处理。

2. 单 Reactor 多线程

监听和分发还是由 1 个 Reactor 线程做,但真正的业务逻辑丢给一个独立的线程池去执行,处理完再由 Reactor 线程把结果写回去。这样业务逻辑能利用多核了,但Reactor 线程本身仍然只有一个——它既要处理新连接的 accept,又要处理所有已建立连接的事件分发,当连接数极大时,这一个线程的分发能力会成为瓶颈。

3. 主从 Reactor 多线程(Netty 的选择)

把 Reactor 本身也拆成多个:主 Reactor(对应 bossGroup)只负责 accept 新连接,接到后立刻"甩给"从 Reactor 组(对应 workerGroup,内部是多个独立的 EventLoop 线程)中的某一个,由它全权负责这条连接后续所有的事件监听、分发乃至业务处理。这样 accept 和已连接的 IO 处理各自都能并行扩展,不再有单一环节的瓶颈。

回顾上一篇的绑定规则:"一个 Channel 终身绑定一个 EventLoop"正是主从 Reactor 模型的具体实现方式——新连接一旦被某个从 Reactor(EventLoop)接管,这个绑定关系就固定下来,不会来回切换,这也是为什么同一连接上的 Handler 调用天然线程安全。
模型典型代表核心限制
单 Reactor 单线程Redis(单机核心命令处理)无法利用多核;一件慢事卡住全部连接
单 Reactor 多线程较少见的过渡形态Reactor 分发本身仍是单线程瓶颈
主从 Reactor 多线程Netty、Nginx(多进程各自单 Reactor 变体)实现复杂度更高,但扩展性最好
为什么 Redis 反而选择单线程?Redis 的核心命令处理是纯内存操作,CPU 消耗极低,瓶颈根本不在"计算",而在网络 IO 和内存带宽——用单线程换取"不用加锁、没有上下文切换开销"的简单性价比更高。而 Netty 面对的是通用网络框架,业务逻辑复杂度不可控(可能是几十毫秒的调用),所以必须支持多线程并行处理,不能像 Redis 那样赌"业务足够轻"。

二、Netty 里主从 Reactor 的具体落地

对照回第二篇的代码:bossGroup 就是主 Reactor(通常只放 1 个线程,因为 accept 本身极快,1 个线程完全够用),workerGroup 就是从 Reactor 组(默认 CPU 核数 × 2 个线程)。一条新连接进来后:

  1. bossGroup 里的 EventLoop 监听到 OP_ACCEPT 事件,调用 accept() 拿到新的 SocketChannel。
  2. bossGroup 按一定策略(默认轮询)从 workerGroup 里挑一个 EventLoop,把这个新 Channel 注册到它上面。
  3. 此后这条连接上所有的读写事件、Handler 调用,都固定由这一个被选中的 EventLoop 线程处理,直到连接关闭。

三、最容易踩的坑:在 EventLoop 线程里做耗时操作

既然一个 EventLoop 要服务多个 Channel,那么任何一个 Channel 上的 Handler 如果执行了耗时操作(比如同步调用数据库、远程 RPC、复杂计算),都会阻塞这同一个 EventLoop 线程——而这个线程上挂着的其他所有连接,在这段时间里全都得不到处理,哪怕它们的数据早就已经就绪。

反例 → 问题:在 channelRead 里直接同步调用一个耗时 200ms 的数据库查询。假设这个 EventLoop 上还绑定了 5000 个空闲连接,这 200ms 里,这 5000 个连接的任何事件(哪怕只是要发一个心跳包)都会被硬生生延迟,整个服务的 P99 延迟会急剧劣化,而且是那种"偶发、难复现"的劣化,排查起来很痛苦。
正例:把耗时的业务逻辑提交到一个独立的业务线程池去执行,EventLoop 线程只负责快速的编解码和分发,拿到业务结果后再切回 EventLoop 线程把响应写回去。
正确写法:业务逻辑不占用 EventLoop 线程
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    businessExecutor.execute(() -> {
        Object result = doSlowBusinessLogic(msg); // 耗时逻辑丢给独立线程池
        ctx.writeAndFlush(result); // 写回操作会自动切回该 Channel 绑定的 EventLoop 执行,线程安全
    });
}

小结

  • Reactor 模式演进:单 Reactor 单线程(无法利用多核)→ 单 Reactor 多线程(分发仍是瓶颈)→ 主从 Reactor 多线程(accept 与 IO 处理都能并行扩展)。
  • Netty 用 bossGroup(主 Reactor,处理 accept)+ workerGroup(从 Reactor组,处理已连接的 IO 与业务)实现主从 Reactor 多线程模型。
  • 一条连接一旦分配给某个 EventLoop 就终身绑定,这是"同一 Channel 上 Handler 无需加锁"的根源。
  • 最大陷阱:在 EventLoop 线程里做耗时操作会阻塞该线程上挂着的所有其他连接,必须把耗时业务丢到独立线程池。

与其他知识点的联系

  • 操作系统 IO 多路复用:Reactor 模式里"用 Selector 统一监听一批 Channel"这件事,底层就是 select/poll/epoll 的应用——Reactor 模式是这套系统调用之上的一层编程范式抽象。
  • Redis 单线程模型:Redis 选择单 Reactor 单线程,与 Netty 选择主从 Reactor 多线程,是同一套 Reactor 思想在不同场景(纯内存计算 vs 通用网络框架)下的不同权衡,可以对照 缓存 模块里 Redis 单线程为什么快的讨论一起理解。
  • 系统设计中的长连接架构:聊天系统、消息推送系统这类需要维护海量长连接的场景,选型时常会直接对比"要不要用 Netty、要撑多少并发连接",而这背后就是本篇讲的主从 Reactor 能扩展到多核的能力,呼应 系统设计 模块。
  • Java 线程池的任务提交:"耗时业务丢给独立线程池"这个套路,和 线程池 里"IO 线程与计算线程分离"的设计理念完全一致——本质都是不要让宝贵的、数量有限的调度线程被业务逻辑的执行时间拖住。
