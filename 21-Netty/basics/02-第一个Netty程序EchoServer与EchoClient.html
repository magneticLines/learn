第一个 Netty 程序:EchoServer 与 EchoClient

跑通一个最简单的回显服务,把上一篇讲的抽象概念(事件驱动、Channel)对应到具体代码上。

📖 来源:《Netty in Action》 第 2 章「你的第一个 Netty 应用」 · 🔧 配套:netty-learning-example/netty-helloworld

一、服务端:ServerBootstrap 组装启动流程

Netty 的服务端搭建遵循一套固定套路:准备两组线程(EventLoopGroup)、用 ServerBootstrap 把各个组件"装配"起来、最后绑定端口启动。先看完整代码,再逐行拆解每个组件在干什么。

EchoServer.java
EventLoopGroup bossGroup = new NioEventLoopGroup(1);   // 只负责 accept 新连接
EventLoopGroup workerGroup = new NioEventLoopGroup();  // 负责已建立连接的读写与业务处理
try {
    ServerBootstrap b = new ServerBootstrap();
    b.group(bossGroup, workerGroup)
     .channel(NioServerSocketChannel.class)      // 服务端用哪种 Channel 实现
     .childHandler(new ChannelInitializer<SocketChannel>() {
         protected void initChannel(SocketChannel ch) {
             ch.pipeline().addLast(new EchoServerHandler());
         }
     });
    ChannelFuture f = b.bind(8080).sync();  // 绑定端口,同步等待启动完成
    f.channel().closeFuture().sync();     // 阻塞主线程,直到服务端关闭
} finally {
    bossGroup.shutdownGracefully();
    workerGroup.shutdownGracefully();
}
组件职责
bossGroup专职处理 accept 事件——有新连接进来时,负责接收并把它"甩给" workerGroup。通常只需要 1 个线程,因为 accept 本身很轻量。
workerGroup负责已建立连接的实际 IO 读写和业务处理,默认线程数是 CPU 核数 × 2。
ServerBootstrap一个"建造者",把 EventLoopGroup、Channel 类型、业务 Handler 这些零件组装起来,最后调用 bind() 才真正启动。
ChannelInitializer每当有新连接建立,就会被调用一次,用来给这条新连接的 pipeline 装上业务 Handler——它本身执行完就会自动从 pipeline 移除,不会重复触发。
类比:bossGroup 就像餐厅门口的迎宾,只负责把客人领进来分配到某张桌子,不负责点单上菜;workerGroup 就像各桌的服务员,真正处理这一桌接下来的所有交互。这种"接待"与"服务"分离的模型,就是后面要讲的主从 Reactor 线程模型的雏形。

二、业务处理:ChannelHandler 里写什么

真正的业务逻辑写在 ChannelHandler 里。Netty 的 IO 事件(连接建立、收到数据、连接断开、异常)都会经过 ChannelPipeline 依次传给注册在里面的 Handler。回显服务的逻辑很简单:收到什么就原样发回去。

EchoServerHandler.java
public class EchoServerHandler extends ChannelInboundHandlerAdapter {
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        ctx.write(msg);   // 写入出站缓冲区,但还没真正发到网络上
        ctx.flush();      // 把缓冲区里的数据真正冲刷到 Socket
    }
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();       // 出异常时主动关闭连接,避免连接泄漏
    }
}
常见坑 1:只 write 不 flush。write() 只是把数据放进了 Netty 内部的出站缓冲区,并不会立刻发送——真正触发系统调用把数据写到 Socket 的是 flush()。忘记调用 flush(),客户端就永远收不到回包,而且现象很隐蔽,不会报错,只是"卡住"。日常写法建议直接用 ctx.writeAndFlush(msg) 一步到位。
常见坑 2:忘记处理 exceptionCaught 导致连接泄漏。如果 Handler 抛出未捕获异常又没有覆写 exceptionCaught,Netty 默认只是打一条日志,不会自动关闭连接——这条连接会一直占着资源却再也不能正常工作。生产代码里 exceptionCaught 几乎是必写项。

三、客户端:Bootstrap 与服务端的镜像结构

客户端的搭建代码和服务端高度对称,区别只是不需要"accept"这个环节,因此只用一个 EventLoopGroup,用 Bootstrap(不带 Server 前缀)而不是 ServerBootstrap,并调用 connect() 而不是 bind()。

EchoClient.java(核心片段)
EventLoopGroup group = new NioEventLoopGroup();
try {
    Bootstrap b = new Bootstrap();
    b.group(group)
     .channel(NioSocketChannel.class)
     .handler(new ChannelInitializer<SocketChannel>() {
         protected void initChannel(SocketChannel ch) {
             ch.pipeline().addLast(new EchoClientHandler());
         }
     });
    ChannelFuture f = b.connect("localhost", 8080).sync();
    f.channel().closeFuture().sync();
} finally {
    group.shutdownGracefully();
}

四、跟着开源项目动手跑一遍

光看代码片段建立不起真实的手感。配套仓库 netty-learning-example 的 netty-helloworld 模块就是这个例子的完整可运行版本,建议动手做这几件事:

  • clone 下来直接跑通,用 telnet localhost 8080 手动发几行字,观察是否原样回显。
  • 故意注释掉 flush(),重新观察现象——验证"常见坑 1"里说的"卡住但不报错"。
  • 在 channelRead 里手动抛一个 RuntimeException,不写 exceptionCaught,观察连接是否真的没有被关闭(可以用 netstat 查连接状态)。
为什么要这么折腾:这几个"故意搞坏"的实验,正是为了把上面两条陷阱从"我知道有这么回事"变成"我亲眼见过这个坑"——后者才是真正能在排查线上问题时派上用场的记忆。

小结

  • 服务端套路:bossGroup(accept) + workerGroup(IO/业务) + ServerBootstrap 组装 + bind()。
  • 客户端套路:单个 EventLoopGroup + Bootstrap + connect(),结构与服务端对称。
  • 业务逻辑写在 ChannelHandler 里,通过 ChannelPipeline 按顺序被调用。
  • 两个高频坑:忘记 flush() 导致数据发不出去;不写 exceptionCaught 导致异常连接不会被关闭。

与其他知识点的联系

  • 为什么需要 Netty:本篇的 bossGroup/workerGroup 拆分,正是上一篇讲的"用少量线程处理海量连接"思想的直接落地——accept 和 IO 处理各自用专门的 EventLoopGroup,而不是像 BIO 那样一个连接开一个线程。
  • Channel / EventLoop / ChannelFuture(下一篇):这里出现的 ChannelFuture.sync()、SocketChannel 都是下一篇要拆解的核心接口,本篇先建立"它们组合起来长什么样"的直觉,下一篇再深入"每个接口单独负责什么"。
  • Java 并发编程里的 try-finally 资源释放:shutdownGracefully() 放在 finally 里是一种资源释放的通用范式,和 线程池优雅关闭 的思路是一致的——都要考虑"正在执行中的任务如何收尾"而不是粗暴 kill。
