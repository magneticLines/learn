Netty 的单元测试:EmbeddedChannel

编解码器写完了,总不能每次改动都真的起一个 Server + Client 手动连接测试——EmbeddedChannel 就是为此而生。

📖 来源:《Netty in Action》 第 9 章「单元测试」

一、为什么 Handler/编解码器的测试特别麻烦

正常情况下,要验证一个 ChannelHandler(尤其是编解码器)的逻辑是否正确,似乎得启动一个真实的 ServerBootstrap,再用一个真实的客户端连接上去发数据、收数据,断言结果——这意味着每条测试用例都要真的占用一个端口、走一遍完整的网络协议栈,又慢又不稳定(端口冲突、时序问题),完全不适合放进 CI 里跑成百上千次。

二、EmbeddedChannel:没有真实网络的 Channel

Netty 提供了 EmbeddedChannel——一个不需要真实 Socket、不需要 EventLoop 线程池、完全在内存里模拟 Pipeline 行为的测试专用 Channel。它把"进网络"和"出网络"这两个动作,替换成了直接往 Pipeline 里塞数据、从 Pipeline 里取数据的方法调用,速度极快且完全确定,是测试 Handler/编解码器逻辑的标准工具。

测试一个解码器:把 4 字节长度头 + 内容解码成完整帧
EmbeddedChannel channel = new EmbeddedChannel(new LengthFieldBasedFrameDecoder(1024, 0, 4));

ByteBuf input = Unpooled.buffer();
input.writeInt(5).writeBytes("hello".getBytes());

// writeInbound 模拟"从网络收到了这段字节",触发入站 Handler 处理
channel.writeInbound(input);

ByteBuf output = channel.readInbound();
assertEquals(5, output.readableBytes()); // 断言:解码后只剩下 "hello" 这 5 个字节的内容
API作用
writeInbound(msg)模拟"网络收到数据",触发这条数据依次经过所有入站 Handler
readInbound()取出经过入站处理链之后、最终产出的结果对象
writeOutbound(msg)模拟"业务代码要发送数据",触发它经过所有出站 Handler(如编码器)
readOutbound()取出编码完成、最终会被写到网络上的字节数据
finish()关闭这个测试 Channel,并返回是否还有未被读取的数据(常用来断言"没有遗留数据")

三、测试编码器同理:反过来用 writeOutbound

测试一个编码器:把业务对象编码成带长度头的字节
EmbeddedChannel channel = new EmbeddedChannel(new LengthFieldPrepender(4));

channel.writeOutbound(Unpooled.wrappedBuffer("hello".getBytes()));

ByteBuf encoded = channel.readOutbound();
assertEquals(5, encoded.readInt()); // 断言:开头 4 字节是内容长度 5

四、为什么这比手写 Mock 更靠谱

有人可能会想:直接 Mock 一个 ChannelHandlerContext 传给 Handler,不也能测?但 ChannelHandlerContext 和 Pipeline 之间的联动关系(fireChannelRead 传递给下一个 Handler、多个 Handler 的组合效果)非常复杂,手写 Mock 很容易漏掉边界情况,而且没法测试"多个 Handler 组合在一起会不会互相干扰"这种真正的集成场景。EmbeddedChannel 允许把整条 Pipeline(多个 Handler)一起塞进去测试,比孤立 Mock 单个 Handler 更接近真实运行环境,同时又不需要真实网络。

工程实践建议:写编解码器/协议相关 Handler 时,建议养成"每写一个,配一段 EmbeddedChannel 测试"的习惯——尤其是拆包/粘包这类边界条件特别多的逻辑(下一篇的主题),用 EmbeddedChannel 把"半包"“粘包"“刚好整包"这几种输入分别构造出来验证,比上线后被半包问题坑一次的成本低得多。

小结

  • EmbeddedChannel 是 Netty 官方提供的、不依赖真实网络和 EventLoop 的测试专用 Channel,用于单元测试 Handler/编解码器。
  • 入站方向用 writeInbound() + readInbound(),出站方向用 writeOutbound() + readOutbound(),分别对应"模拟收到数据"和"模拟发送数据"。
  • 相比手写 Mock,EmbeddedChannel 能测试多个 Handler 组合的真实联动效果,更接近生产环境的行为。
  • 协议编解码这类边界条件密集的逻辑,强烈建议用 EmbeddedChannel 补齐单元测试,而不是只靠手动联调。

与其他知识点的联系

  • 单元测试与 Mock 的取舍:本篇"用真实 Pipeline 组合测试而非孤立 Mock"的思路,和 测试 模块里"该 Mock 到什么颗粒度"的讨论直接相关——过度 Mock 会让测试脱离真实协作关系,失去发现集成问题的能力。
  • 测试驱动开发(TDD):EmbeddedChannel 让"先写编解码逻辑的测试用例,再实现 Handler"变得可行且轻量,可以直接套用 TDD 模块讲的红绿重构循环。
