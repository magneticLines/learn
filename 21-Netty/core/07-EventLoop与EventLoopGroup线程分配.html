EventLoop 与 EventLoopGroup 线程分配

一个 EventLoopGroup 里到底装了几个 EventLoop?新连接进来时具体是怎么被分配到某一个上面的?

📖 来源:《Netty in Action》 第 7 章「EventLoop 和线程模型」

一、EventLoopGroup 是 EventLoop 的容器与调度者

回顾第 3 篇讲过的:EventLoop 是单线程的事件循环,EventLoopGroup 则是若干个 EventLoop 的集合,负责对外提供"分配一个 EventLoop"的能力。以最常用的 NioEventLoopGroup 为例,不传参数时默认创建的线程数是 CPU 核数 × 2——这是一个经过 Netty 官方权衡后的默认值:既能利用多核并行处理不同连接,又不会因为线程数过多导致上下文切换开销反噬性能。

显式指定线程数
EventLoopGroup group = new NioEventLoopGroup(4); // 固定 4 个 EventLoop 线程

二、新 Channel 如何被分配到某个 EventLoop

当 bossGroup accept 到一条新连接后,需要把这条连接注册到 workerGroup 里的某一个 EventLoop 上。Netty 默认采用轮询(Round-Robin)策略:第 1 条连接分给 EventLoop-0,第 2 条给 EventLoop-1,以此类推,循环往复。这样能保证长期运行下,每个 EventLoop 上挂的连接数大致均衡。

为什么不是"选负载最轻的那个"?看起来"挑当前连接数最少的 EventLoop"更聪明,但这需要额外维护和比较每个 EventLoop 的实时负载,本身就有开销,而且"连接数少"不等于"当前 CPU 负载低"(可能那几个连接反而是重度活跃的)。轮询策略简单、无锁、开销几乎为零,在绝大多数场景下已经足够均衡——这也是很多分布式系统里"负载均衡算法选型"时的常见取舍:简单策略在绝大多数场景下往往比复杂策略的性价比更高。

三、bossGroup 只需要 1 个线程,为什么还要传 EventLoopGroup

不少人会疑惑:既然 accept 操作很轻量,bossGroup 通常只放 1 个 EventLoop,为什么 API 上还是要求传一个 EventLoopGroup 而不是单个 EventLoop?原因是:①API 设计上 boss 和 worker 保持同一种类型(EventLoopGroup),使用更一致;②某些特殊场景下(比如需要同时监听多个端口),给 bossGroup 配置多个线程依然是有意义的,每个线程负责一个端口的 accept。

四、EventLoop 既跑 IO,也跑普通任务

容易被忽视的一点:EventLoop 不仅处理网络 IO 事件,它本身也是一个通用的任务执行器,实现了 ScheduledExecutorService 接口。可以直接把普通任务甚至定时任务提交给它执行:

向 Channel 绑定的 EventLoop 提交任务
ctx.channel().eventLoop().execute(() -> {
    // 这个任务会排队进入该 Channel 绑定的 EventLoop 的任务队列,
    // 和这条连接上的 IO 事件处理共享同一条时间线、串行执行
});

ctx.channel().eventLoop().schedule(() -> {
    // 定时任务,同样运行在这个 EventLoop 线程上
}, 5, TimeUnit.SECONDS);
陷阱:把这个能力当成通用线程池滥用。提交给 EventLoop 的任务和这条连接的 IO 事件处理共享同一个线程——如果提交了一个耗时任务,效果和"在 Handler 里直接写耗时逻辑"是完全一样的,会阻塞这个 EventLoop 上其他所有连接的处理(呼应上一篇讲的核心陷阱)。这个 API 应该只用来提交与该连接相关、执行很快的小任务(比如安排一次心跳检查),真正的重活儿依然要丢给独立的业务线程池。

小结

  • EventLoopGroup 默认线程数为 CPU 核数 × 2,是并行能力与调度开销之间的权衡结果。
  • 新连接默认按轮询策略分配给 EventLoopGroup 里的某个 EventLoop,简单、无锁、足够均衡。
  • bossGroup 通常只需 1 个线程,但 API 上依然使用 EventLoopGroup 类型,保持一致性并支持多端口监听场景。
  • EventLoop 本身也是通用任务执行器,但提交给它的任务会占用该连接所在的那唯一一个线程,只适合放轻量任务。

与其他知识点的联系

  • 负载均衡算法(承接 10-计算机网络):EventLoopGroup 内部的轮询分配策略,和 负载均衡的常见算法 里"轮询 vs 加权 vs 最小连接数"是同一类问题在不同尺度上的重现——一个是分配 Channel 给 EventLoop,一个是分配请求给服务器实例,权衡逻辑高度相似。
  • Java 的 ScheduledExecutorService:EventLoop 实现这个接口意味着它可以直接套用 JUC 线程池 的定时任务用法,理解了 JDK 的 ScheduledThreadPoolExecutor,这里的 API 就是现成的知识迁移。
  • 心跳检测与 IdleStateHandler(后续章节):EventLoop 的 schedule() 能力正是 Netty 内置的 IdleStateHandler 检测空闲连接的实现基础——理解了本篇,再学心跳检测会更容易看懂它的内部原理而不只是会用 API。
