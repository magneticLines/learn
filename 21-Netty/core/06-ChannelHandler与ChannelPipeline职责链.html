ChannelHandler 与 ChannelPipeline 职责链

为什么 Netty 里的编解码、业务逻辑、异常处理要拆成一个个独立的 Handler,而不是写在一个大方法里?

📖 来源:《Netty in Action》 第 6 章「ChannelHandler 和 ChannelPipeline」

一、ChannelPipeline:每条连接自带的一条流水线

每个 Channel 创建时都会自动配备一个属于自己的 ChannelPipeline——本质上是一个由 ChannelHandler 组成的双向链表。一个数据包从网络进来,要依次经过 pipeline 里注册的一系列 入站(Inbound)Handler 处理(解码、业务逻辑);反过来,业务要发送数据出去,则依次经过一系列 出站(Outbound)Handler 处理(编码、加密)。这正是经典的职责链设计模式——每个 Handler 只关心自己那一小段职责,处理完决定"是否继续往下传"。

装配一条 Pipeline
ch.pipeline()
  .addLast(new FrameDecoder())      // 入站:先解决拆包问题
  .addLast(new ProtocolDecoder())   // 入站:再把字节解析成业务对象
  .addLast(new BusinessHandler())    // 入站:业务逻辑
  .addLast(new ProtocolEncoder());  // 出站:把响应对象编码回字节
类比:Pipeline 就像工厂里的流水线——原材料(字节流)从一端进来,依次经过拆箱工位(拆包)、质检工位(解码校验)、组装工位(业务处理),最后成品(响应)从另一端出去。每个工位只做一件事,想加一道新工序(比如加个限流 Handler),只要往流水线上插一个工位,不需要改其他工位的代码。

二、入站与出站:方向决定了 Pipeline 里的遍历顺序

这是初学者最容易搞混的地方。入站事件按 addLast 的顺序从前往后遍历(模拟数据"流入"),出站事件按 addLast 的顺序从后往前遍历(模拟数据"流出")。用上面的例子:一条收到的消息会按 FrameDecoder → ProtocolDecoder → BusinessHandler 的顺序处理;而当 BusinessHandler 调用 ctx.writeAndFlush() 发出响应时,只会经过在它之后注册的出站 Handler(也就是 ProtocolEncoder),而不会重新回头经过前面的入站 Handler。

Handler 类型典型基类遍历方向职责举例
入站(Inbound)ChannelInboundHandlerAdapter从头到尾解码、鉴权、业务处理
出站(Outbound)ChannelOutboundHandlerAdapter从尾到头编码、加密、限流
高频陷阱:Handler 添加顺序放错导致逻辑失效。如果把 ProtocolDecoder 加在 BusinessHandler 之后,那么 BusinessHandler 收到的还是原始字节而不是业务对象,大概率会直接抛出类型转换异常。Pipeline 的顺序不是随意的,必须和数据实际的处理先后顺序一致。

三、ChannelHandlerContext:每个 Handler 与 Pipeline 之间的桥梁

每个 ChannelHandler 被添加进 Pipeline 时,都会关联一个专属的 ChannelHandlerContext,它持有指向前一个、后一个 Handler 的引用。调用 ctx.fireChannelRead(msg) 会把事件传给下一个入站 Handler,而不是重新从头开始——这也是为什么想中断处理链、不再往下传,只要不调用 fireChannelRead() 就行。

另一个易错点:ctx.write() vs channel.write()。channel.write() 会让事件从 Pipeline 的尾部开始重新走一遍所有出站 Handler;而 ctx.write() 只会从当前 Handler 的下一个出站 Handler 开始处理。如果你在中间某个 Handler 里想"跳过前面已经处理过的编码逻辑",用错 API 会导致数据被重复编码或者跳过了本该经过的处理环节。

四、@Sharable:一个 Handler 实例能否被多个 Channel 共用

默认情况下,建议为每个新连接创建一个新的 Handler 实例(通常在 ChannelInitializer 里 new 出来),因为很多 Handler 会保存与某条连接相关的状态(比如已经读到的半包数据)。但如果一个 Handler 确实无状态(比如单纯做日志打印),可以用 @Sharable 注解标记后,让多个 Channel 的 Pipeline 共用同一个实例,省去重复创建的开销。

陷阱:给有状态的 Handler 加 @Sharable。如果一个 Handler 内部有实例字段用来记录某条连接的中间状态,却被错误地标记为 @Sharable 并在多个连接间共享,不同连接的数据会互相污染彼此的状态——这是一类隐蔽且难以复现的并发 bug,因为触发条件依赖多条连接的时序交织。

小结

  • ChannelPipeline 是职责链模式的应用:每条连接自带一条由 Handler 组成的双向链表,入站从头到尾、出站从尾到头。
  • Handler 的添加顺序必须和数据实际处理的先后顺序一致,顺序放错是最常见的入门坑。
  • ctx.write() 只从当前节点往后走,channel.write() 会从尾部重新走一遍——两者语义不同,不能混用。
  • 有状态的 Handler 每连接一个实例;确认无状态才能用 @Sharable 共享,否则会导致连接间状态污染。

与其他知识点的联系

  • 设计模式·职责链模式:ChannelPipeline 是职责链模式在网络框架里的教科书级应用,可以对照 设计模式 模块里职责链模式的通用定义,理解 Netty 是如何把这个模式和"入站/出站双向遍历"结合起来的。
  • Servlet 的 Filter 链:Java Web 开发里 Filter/拦截器链 和 ChannelPipeline 是同一种思路——把横切关注点(鉴权、日志、编解码)拆成独立、可插拔的处理单元,而不是耦合在业务代码里。
  • 线程封闭(承接 03):本篇的 Handler 之所以经常"每连接一个实例",本质上是在利用前面讲过的"一个 Channel 终身绑定一个 EventLoop"这条规则——既然同一 Channel 的处理是单线程串行的,Handler 内部的状态字段自然不需要加锁,但前提是这个实例真的只服务这一条连接。
