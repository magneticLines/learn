Bootstrap 与 ServerBootstrap 启动流程

option() 和 childOption() 有什么区别?handler() 和 childHandler() 又管的是谁?这一篇把"装配阶段"的细节讲清楚。

📖 来源:《Netty in Action》 第 8 章「引导」

一、Bootstrap 是建造者模式的应用

Bootstrap(客户端)和 ServerBootstrap(服务端)都是建造者模式的实现——通过一连串的链式调用把各个配置项、组件收集起来,最后调用 connect() 或 bind() 才真正触发启动动作。这样设计的好处是:装配阶段(设置线程组、Channel 类型、Handler)和启动动作是分离的,配置错了在编译期或启动瞬间就能发现,不会等到运行中间才暴露问题。

二、ServerBootstrap 特有的"两组配置" 服务端连接 vs 已接受连接

服务端场景比客户端复杂一层:ServerBootstrap 需要同时配置服务端自身这个 Channel(负责 accept)和每条被接受的子连接 Channel,这正是很多 API 都带 child 前缀的原因。

API作用对象典型用途
group(boss, worker)第一个参数是服务端 Channel 用的线程组,第二个是子连接用的线程组指定 accept 与 IO 处理分别用哪组 EventLoop
option(...)服务端 ServerSocketChannel 本身的参数如 SO_BACKLOG(等待 accept 的连接队列长度)
childOption(...)每条被接受的子连接的参数如 TCP_NODELAY(禁用 Nagle 算法)、SO_KEEPALIVE
handler(...)服务端 Channel 自己的 Pipeline很少用到,通常只做启动日志之类的事
childHandler(...)每条子连接的 Pipeline真正装配业务逻辑 Handler 的地方,几乎所有业务代码都写在这里
高频陷阱:把业务 Handler 加到了 handler() 而不是 childHandler()。初学者常常把两者搞混——handler() 配置的是服务端 ServerSocketChannel 自己的 Pipeline,它只在 accept 阶段起作用,和实际客户端连接的数据处理完全无关;真正处理客户端读写的业务逻辑,必须写进 childHandler()。写错了不会报错,只是业务逻辑永远不会被触发,现象是"服务端能启动、能 accept,但收不到任何数据处理效果",排查起来容易摸不着头脑。

三、ChannelOption 常见配置项

典型的 ServerBootstrap 配置
ServerBootstrap b = new ServerBootstrap();
b.group(bossGroup, workerGroup)
 .channel(NioServerSocketChannel.class)
 .option(ChannelOption.SO_BACKLOG, 1024)          // 服务端:accept 队列长度
 .childOption(ChannelOption.TCP_NODELAY, true)  // 子连接:关闭 Nagle 算法,降低小包延迟
 .childOption(ChannelOption.SO_KEEPALIVE, true) // 子连接:开启 TCP 层保活探测
 .childHandler(new ChannelInitializer<SocketChannel>() { ... });
TCP_NODELAY 为什么值得关注:TCP 默认开启 Nagle 算法,会把多个小包攒起来一起发送以提高网络利用率,但这会带来额外延迟。对于要求低延迟的实时通信场景(游戏、IM、RPC),几乎都会显式设置 TCP_NODELAY=true 关闭这个"攒包"行为,用带宽利用率换取延迟——这和 计算机网络 里 TCP 拥塞控制、流量控制的取舍思路是一致的:没有免费的性能提升,都是在某个维度上的权衡。

四、group() 的两种调用方式

ServerBootstrap 的 group() 既可以传两个参数(bossGroup、workerGroup),也可以只传一个——此时这一个 group 会同时承担 accept 和 IO 处理的职责,退化成"单 Reactor 多线程"模型。多数生产场景仍推荐分开配置两组,让 accept 和已有连接的处理互不干扰。

五、bind() 之后连接是否已经真正建立

b.bind(port) 返回的是一个 ChannelFuture,绑定端口本身也是异步操作。回顾第 3 篇的教训:不要在业务代码里到处滥用 sync(),但启动流程是个例外——在 main 方法(非 EventLoop 线程)里对 bind().sync() 阻塞等待,是安全且常见的写法,因为这里的目的就是"确认端口真的绑定成功了再继续往下走"。

小结

  • Bootstrap/ServerBootstrap 是建造者模式:先装配、后启动,configure 与 start 分离。
  • ServerBootstrap 特有一套 child 前缀 API:option/handler 管服务端 Channel 自己,childOption/childHandler 管每条被接受的子连接——业务逻辑一定写在 childHandler 里。
  • TCP_NODELAY/SO_KEEPALIVE/SO_BACKLOG 是最常用的几个 ChannelOption,分别对应延迟、连接保活、accept 队列容量的调优。
  • 启动阶段的 bind().sync() 是少数几个可以安全阻塞等待的场景之一。

与其他知识点的联系

  • 设计模式·建造者模式:Bootstrap 的链式配置写法是 建造者模式 的经典实践,和 Java 里常见的 StringBuilder、各种 xxxConfig.builder()...build() 写法是同一套思路。
  • TCP 三次握手与 SO_BACKLOG:SO_BACKLOG 控制的是完成三次握手、等待应用调用 accept 的连接队列长度,这个参数如果设得太小,在瞬时高并发下会导致连接被内核直接丢弃——建议对照 计算机网络 模块里 TCP 连接建立与关闭的细节一起理解。
  • ChannelHandler 与 Pipeline(上一篇):childHandler 里的 ChannelInitializer 正是上一篇讲的"给每条新连接的 Pipeline 装配 Handler"的具体入口,本篇是把它放进了完整的启动流程里看它的位置。
