一、RPC 框架要解决的核心问题:让远程调用看起来像本地调用

RPC(Remote Procedure Call)的目标是让调用方写 userService.getUser(id) 时,感觉不到这背后其实发生了一次网络通信。要实现这种"透明性",框架需要在背后完成:①把方法名、参数序列化成字节;②通过网络发送到远端;③远端反序列化还原出方法名和参数,反射调用真正的实现;④把结果序列化后传回来;⑤调用方阻塞等待(或者用 Future/回调异步等待)结果返回,再反序列化还原成 Java 对象返回给业务代码。而第②③④步的高性能网络通信层,正是 Netty 的用武之地。

这张表想说明的事:一个"能用"的 RPC 框架,几乎不需要任何 Netty 没有覆盖到的新概念——它只是把前面 16 篇讲的每一个组件,按照"通信 + 协议 + 线程模型"这条主线正确地组合了起来。这也是为什么理解 Netty 原理比死记 API 更重要:任何一个新的网络中间件需求,拆解开来看,大概率都是这些积木的不同排列组合。

二、请求-响应模式的关联:requestId

TCP 是一条连接上可能有多个并发请求在"飞行中"的(尤其是异步 RPC 允许一个连接上同时发出多个请求而不必等前一个返回)。要把某个响应正确匹配回它对应的那次调用,协议里必须带一个唯一的 requestId,客户端发送请求前生成、维护一个 Map<Long, CompletableFuture<Response>>,响应到达时按 requestId 找到对应的 CompletableFuture 并完成它。

客户端:请求发送与响应回填的核心骨架
private final Map<Long, CompletableFuture<Response>> pending = new ConcurrentHashMap<>();

CompletableFuture<Response> send(Request req) {
    CompletableFuture<Response> future = new CompletableFuture<>();
    pending.put(req.getRequestId(), future); // 先登记,等响应回来再"认领"
    channel.writeAndFlush(req);
    return future; // 调用方可以 .get() 阻塞等,也可以 .thenAccept() 异步处理
}

// 收到响应的 Handler 里:
CompletableFuture<Response> future = pending.remove(response.getRequestId());
if (future != null) future.complete(response);
容易被忽视的陷阱:超时清理。如果服务端异常、网络丢包导致某个请求永远收不到响应,pending 这张 Map 里对应的 CompletableFuture 会永远留在那里得不到清理,长期运行会造成内存泄漏。生产级实现必须为每个请求配一个超时任务(可以用第 7 篇讲的 EventLoop 定时能力),超时后主动完成这个 Future 为异常状态并从 Map 中移除。

三、去配套仓库里印证这套设计

建议现在回到 netty-learning-example,把整个仓库过一遍,重点关注这几件事:

  • 对照仓库里涉及 RPC/自定义协议的模块,看它的协议头设计和第 13 篇讲的有什么异同——真实项目往往会因为具体需求增删字段,理解"为什么这么改"比死记某一种标准格式更重要。
  • 找一下仓库里是否有单元测试目录,对照第 9 篇看它是不是也用了 EmbeddedChannel,以及测试覆盖了哪些边界场景。
  • 观察它的 ChannelInitializer 里 Pipeline 的组装顺序,验证是否符合第 6 篇讲的"入站从头到尾、出站从尾到头"的规则。
关于后续学习:把这本书的核心内容配合这个仓库跑通之后,再去读 Dubbo 或 RocketMQ 的网络层源码会容易得多——它们在协议设计、线程模型上的思路,和这里学到的东西是高度一致的,只是工程细节更复杂、健壮性处理更完善。

小结

  • RPC 框架的本质是"让远程调用看起来像本地调用",网络通信层的高性能实现正是 Netty 的核心价值所在。
  • 一次完整的 RPC 调用,几乎串联了本模块讲过的全部核心组件——理解原理比记忆 API 更能应对新场景。
  • requestId + Map<Long, CompletableFuture> 是异步 RPC 请求-响应匹配的标准做法,必须配合超时清理机制防止内存泄漏。
  • 学完原理后回到配套开源仓库验证,是把"知道"转化为"会用"的关键一步。

与其他知识点的联系

  • CompletableFuture 与异步编程:本篇 requestId 映射表的设计,是 CompletableFuture 在网络编程场景下的典型应用——理解了 CompletableFuture 的 complete()/thenAccept(),这里的代码不需要额外学习新概念。
  • 分布式与微服务里的 RPC 选型:可以对照 分布式与微服务 模块,理解 Dubbo(基于 Netty)、gRPC(基于 HTTP/2)等主流 RPC 框架在协议、序列化、连接管理上的选型差异,本篇讲的是"最小可行版本",生产级框架还要考虑负载均衡、服务发现、熔断降级等更多问题。
  • 系统设计中的超时与重试:请求超时清理机制,呼应 系统设计 里"任何跨网络调用都必须设置超时"这条通用原则——没有超时保护的远程调用,在网络分区发生时会拖垮整个调用链。
