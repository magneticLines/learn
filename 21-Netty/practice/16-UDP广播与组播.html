UDP 广播与组播

前面十几篇几乎都在讲 TCP,但 Netty 同样是一个通用的网络框架——UDP 场景下它一样好用。

📖 来源:《Netty in Action》 第 12 章「用 UDP 广播事件」

一、UDP 与 TCP 的关键差异回顾

UDP 是无连接、不保证可靠交付的协议:没有握手,发送方直接把数据报(Datagram)丢出去,不管对方收没收到,也不管顺序对不对。这些"缺点"恰恰在特定场景下变成了优点:没有连接建立/维护的开销,没有丢包重传导致的延迟抖动,天然支持一对多的广播(Broadcast)和组播(Multicast)——这是基于连接的 TCP 完全做不到的。

为什么面向连接的 TCP 没法做广播:TCP 连接是"一对一"的点对点通道,给 100 个客户端广播消息就得维护 100 条独立连接、发送 100 次。UDP 不需要事先建立连接,一个数据报可以直接发往一个广播地址或组播地址,底层网络设备负责把它复制分发给多个接收方,发送方只需要发一次。

二、Netty 里 UDP 的 Channel 类型:NioDatagramChannel

UDP 场景下不再需要区分"服务端/客户端"或"bossGroup/workerGroup"——因为 UDP 本身没有连接的概念,自然也没有 accept 这回事。只需要一个 Bootstrap(不是 ServerBootstrap)配合 NioDatagramChannel:

广播消息的最小示例
EventLoopGroup group = new NioEventLoopGroup();
try {
    Bootstrap b = new Bootstrap();
    b.group(group)
     .channel(NioDatagramChannel.class)
     .option(ChannelOption.SO_BROADCAST, true) // 允许发送广播包
     .handler(new SimpleChannelInboundHandler<DatagramPacket>() {
         protected void channelRead0(ChannelHandlerContext ctx, DatagramPacket packet) {
             // 处理收到的数据报,packet.sender() 能拿到发送方地址
         }
     });
    Channel ch = b.bind(0).sync().channel(); // 绑定到一个随机可用端口
    ByteBuf data = Unpooled.copiedBuffer("discovery".getBytes());
    ch.writeAndFlush(new DatagramPacket(
        data, new InetSocketAddress("255.255.255.255", 8888) // 广播地址
    )).sync();
} finally {
    group.shutdownGracefully();
}

可以看到 UDP 场景下收发的消息类型是 DatagramPacket,它把"数据内容"和"目标/来源地址"打包在了一起——因为 UDP 没有像 TCP 那样"连上之后地址就固定"的概念,每个数据报都要独立指明地址。

三、组播:比广播更精确的"订阅式"分发

广播会发给同一网段内的所有设备,不管它是否关心这条消息,浪费带宽也存在安全隐患(任何设备都能收到)。组播(Multicast)则是让接收方先"加入"一个特定的组播地址(相当于订阅一个频道),发送方发往这个地址的数据,只有加入了这个组的接收方才会收到。

加入组播组接收消息
NioDatagramChannel ch = (NioDatagramChannel) b.bind(8888).sync().channel();
InetSocketAddress groupAddress = new InetSocketAddress("230.0.0.1", 8888); // 组播地址范围 224.0.0.0~239.255.255.255
ch.joinGroup(groupAddress).sync(); // 加入这个组播组,之后能收到发往这个地址的消息
实际应用场景:服务发现(新启动的服务节点通过组播向局域网内的其他节点广播自己的存在)、集群内部状态同步、局域网内的设备发现协议(如 UPnP)等,是 UDP 广播/组播的典型用武之地——这些场景的共同特点是:参与方数量不固定、不要求可靠交付(丢一次可以靠下一次心跳补上)、追求低延迟和简单性。

四、为什么这些场景能容忍 UDP 的不可靠性

关键判断:UDP 是否合适,取决于"丢一条消息的后果是否可以接受"。服务发现场景里,即便某一次广播丢失了,新节点通常会定期重复发送("我还在这里"),下一次广播大概率就能被收到——单次丢失的代价很低。但如果是转账这类每一条消息都必须送达、且不能重复的业务,UDP 的不可靠性是不可接受的,必须用 TCP 或者在 UDP 之上自己实现确认重传机制(类似 QUIC 协议的做法)。选择 UDP 还是 TCP,本质上是在"性能/简单性"与"可靠性保证"之间做业务层面的权衡,不存在绝对更优的一方。

小结

  • UDP 无连接、不保证可靠交付,但天然支持广播和组播这种一对多分发,这是 TCP 做不到的。
  • Netty 用 NioDatagramChannel + Bootstrap(无需 bossGroup/workerGroup 区分)支持 UDP,消息类型是携带地址信息的 DatagramPacket。
  • 组播比广播更精确,只有主动 joinGroup() 的接收方才会收到消息,节省带宽也减少无关方接收到消息的风险。
  • 是否用 UDP,取决于业务能否容忍消息丢失——服务发现、状态同步这类"丢了会自愈"的场景是理想用武之地。

与其他知识点的联系

  • TCP 与 UDP 的本质区别:本篇是 计算机网络 模块里 TCP/UDP 对比理论的实际编程落地——协议层面学到的取舍,在这里变成了具体要不要调用 joinGroup()、要不要处理丢包重传的工程决策。
  • 分布式系统的服务发现:UDP 组播是服务发现的一种实现方式,可以对照 分布式与微服务 模块里基于注册中心(如 ZooKeeper、Nacos)的服务发现方案,理解"轻量组播 vs 中心化注册中心"两种思路的取舍——前者简单但仅适合局域网内的小规模场景,后者更适合大规模、跨网段的生产环境。
  • QUIC 协议:"在 UDP 之上自己实现可靠传输"正是 HTTP/3 底层 QUIC 协议的设计思路,值得在学完 计算机网络 现代基础设施部分后回头对照理解。
