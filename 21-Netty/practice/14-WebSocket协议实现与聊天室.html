WebSocket 协议实现与聊天室

前面十几篇讲的都是"造轮子"的原理,这一篇用 Netty 内置的编解码器直接组装一个可运行的多人聊天室。

📖 来源:《Netty in Action》 第 11 章「WebSocket」 · 🔧 配套:netty-learning-example/netty-websocket

一、为什么需要 WebSocket:HTTP 短连接的天然局限

HTTP 是"请求-响应"模式:客户端问一句,服务端答一句,服务端没法主动推送消息给客户端。传统做法是客户端不断轮询("每隔几秒问一次有没有新消息"),既浪费资源又有明显延迟。WebSocket 在一次 HTTP 请求的基础上"升级"成一条全双工的长连接,之后服务端和客户端都能随时主动向对方发送数据——这正是聊天室、实时通知、股票行情推送等场景的标准选择。

二、握手阶段:先是 HTTP,升级后才是 WebSocket

WebSocket 连接建立的第一步依然是一个标准 HTTP 请求,只是带上几个特殊头部表明"我想升级成 WebSocket":

WebSocket 握手请求(简化)
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

服务端如果同意升级,会返回 101 Switching Protocols,之后这条 TCP 连接上的通信就完全切换成 WebSocket 自己的帧格式,不再是 HTTP 报文。Netty 把这套复杂的握手协商流程封装进了 WebSocketServerProtocolHandler,业务代码基本不需要关心握手细节。

三、组装一个最小可用的聊天室 Pipeline

WebSocket 服务端 Pipeline
ch.pipeline()
  .addLast(new HttpServerCodec())               // 先按 HTTP 协议解析(握手阶段需要）
  .addLast(new ChunkedWriteHandler())         // 支持大数据分块写(可选,常见配置）
  .addLast(new HttpObjectAggregator(8192)) // 把 HTTP 请求的多个部分聚合成完整的 FullHttpRequest
  .addLast(new WebSocketServerProtocolHandler("/chat")) // 处理握手协商,握手完成后自动把后续帧转换成 WebSocketFrame
  .addLast(new ChatHandler());               // 业务逻辑:处理聊天消息
ChatHandler:群发消息给所有在线连接
public class ChatHandler extends SimpleChannelInboundHandler<TextWebSocketFrame> {
    // ChannelGroup 是 Netty 提供的线程安全容器,专门用来管理一批 Channel 并支持批量操作
    private static final ChannelGroup ONLINE_USERS =
        new DefaultChannelGroup(GlobalEventExecutor.INSTANCE);

    public void handlerAdded(ChannelHandlerContext ctx) {
        ONLINE_USERS.add(ctx.channel()); // 新连接握手完成后加入群组
    }
    protected void channelRead0(ChannelHandlerContext ctx, TextWebSocketFrame msg) {
        // 收到一条消息,原样广播给所有在线用户(含自己)
        ONLINE_USERS.writeAndFlush(new TextWebSocketFrame(msg.text()));
    }
    public void handlerRemoved(ChannelHandlerContext ctx) {
        ONLINE_USERS.remove(ctx.channel()); // 连接断开自动从群组移除,ChannelGroup 内部也会做失效检测
    }
}
ChannelGroup 为什么重要:聊天室、广播推送这类场景,天然要维护一批"当前在线的连接",而这些连接可能分布在 workerGroup 里的不同 EventLoop 线程上——直接用普通的 List<Channel> 会有多线程安全问题(不同连接的 add/remove 可能来自不同 EventLoop 线程)。ChannelGroup 是 Netty 提供的线程安全实现,并且它的 writeAndFlush() 已经处理好了"把消息分别路由到每个 Channel 各自绑定的 EventLoop 上执行"这件事,业务代码不需要操心线程安全问题。

四、SimpleChannelInboundHandler 与自动 release 的便利与陷阱

上面用的是 SimpleChannelInboundHandler<T> 而不是 ChannelInboundHandlerAdapter——它有两个特点:①泛型参数 T 会自动过滤消息类型,不匹配的消息自动传给下一个 Handler;②处理完 channelRead0() 之后,会自动帮你调用 ReferenceCountUtil.release(msg),不需要像第 5 篇讲的那样手动 release()。

陷阱:如果需要把消息转发给下一个 Handler 或者异步处理后再释放,不能用 SimpleChannelInboundHandler 的自动释放。如果业务逻辑是把消息丢给一个异步线程池处理(呼应第 4 篇的教训),而 channelRead0() 本身已经返回,SimpleChannelInboundHandler 会在异步任务还没跑完时就已经把消息 release 掉了,等异步任务真正读取 msg 时会遇到已经被释放的 ByteBuf,抛出 IllegalReferenceCountException。这种场景应该用普通的 ChannelInboundHandlerAdapter 自己控制 release 的时机,或者在丢给异步线程前调用 retain() 增加一次引用计数。

小结

  • WebSocket 握手复用 HTTP,通过 Upgrade 头协商升级成全双工长连接,解决了 HTTP 短连接无法服务端主动推送的问题。
  • Netty 用 WebSocketServerProtocolHandler 封装了整个握手协商流程,业务代码只需处理升级完成后的 WebSocketFrame。
  • ChannelGroup 是管理一批在线连接、支持线程安全广播的标准容器,聊天室/推送场景的必备组件。
  • SimpleChannelInboundHandler 会自动释放消息,如果要异步处理消息,必须手动 retain() 或改用非自动释放的基类。

与其他知识点的联系

  • HTTP 协议基础:WebSocket 握手复用 HTTP 请求/响应格式,理解 HTTP 状态码与头部机制 能帮助更快看懂 101 Switching Protocols 这个不太常见的状态码背后的含义。
  • ByteBuf 引用计数(承接第 5 篇):本篇 SimpleChannelInboundHandler 的自动 release 陷阱,正是对第 5 篇引用计数机制的一次实战应用——原理没吃透的话,遇到 IllegalReferenceCountException 会一头雾水。
  • 系统设计里的实时推送架构:ChannelGroup 广播的思路,和 系统设计 模块里聊天系统/Feed 流的"扇出(fan-out)"策略是同一个问题在不同规模下的体现——小规模可以直接遍历所有连接广播,超大规模则需要引入消息队列做异步扇出。
