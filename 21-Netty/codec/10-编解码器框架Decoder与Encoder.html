编解码器框架:Decoder 与 Encoder

把字节流变成 Java 对象是解码,反过来是编码——Netty 为这套转换过程专门定义了一套模板类。

📖 来源:《Netty in Action》 第 10 章「编解码器框架」

一、为什么不直接在业务 Handler 里手写字节解析

理论上完全可以在一个 ChannelInboundHandlerAdapter 里直接对 ByteBuf 做各种 readInt()/readBytes() 手工解析,但这样做会把"从字节到业务对象"和"拿到业务对象后做什么"这两件完全不同的关注点耦合在一起——协议一旦有变化,业务逻辑代码也要跟着改,测试也没法把两者分开验证。Netty 把"解码"和"业务处理"拆成了两个独立的 Handler:Decoder 只管把字节转换成对象,转换完调用 fireChannelRead() 交给下一个 Handler,业务 Handler 只需要面对已经解析好的对象。

二、ByteToMessageDecoder:处理"半包"问题的核心基类

解码器最需要解决的问题是:TCP 是流式协议,一次 channelRead() 回调收到的字节,不一定刚好是一条完整的消息——可能只收到半条(半包),也可能一次收到好几条粘在一起(粘包,下一篇详细展开)。ByteToMessageDecoder 内部维护了一个累积缓冲区(cumulation buffer),自动处理"数据不够就先攒着,等下次数据到了再继续尝试解析"这件麻烦事,业务代码只需要专注于"给定足够的字节,如何解析出一条消息"这个相对简单的逻辑。

一个简单的定长解码器:每 4 字节解析成一个 int
public class IntDecoder extends ByteToMessageDecoder {
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < 4) {
            return; // 数据不够 4 字节,直接返回,框架会自动帮你等下一批数据到来后再调用
        }
        out.add(in.readInt()); // 够了就解析出来,放进 out;可以在一次调用里解析出多条
    }
}
这个模板方法的关键约定:如果当前累积的数据还不足以解析出一条完整消息,decode() 方法什么都不做直接返回即可——不需要自己写"数据不够怎么办"的分支逻辑,框架会记得这批不完整的数据,等下一次有新数据到达时,自动把新旧数据拼在一起再调用一次 decode()。这正是 ByteToMessageDecoder 存在的核心价值:把"攒够数据再处理"这件所有协议解析都要面对的通用逻辑,从具体的业务解析逻辑里抽离出来了。

三、MessageToByteEncoder:反方向的模板

编码方向的模板类是 MessageToByteEncoder<T>,泛型参数 T 指定这个编码器只处理哪种类型的出站消息(其他类型会被自动跳过,交给 pipeline 里下一个出站 Handler),避免每个编码器都要写一遍 instanceof 判断。

把 Integer 编码成 4 字节
public class IntEncoder extends MessageToByteEncoder<Integer> {
    protected void encode(ChannelHandlerContext ctx, Integer msg, ByteBuf out) {
        out.writeInt(msg);
    }
}

四、一步到位:同时处理编解码的 Codec 类

如果一个协议的编码和解码逻辑天然是配对、共享状态的(比如某些需要维护会话上下文的协议),Netty 还提供了 ByteToMessageCodec<T>,把 decode() 和 encode() 合并到同一个类里,避免拆成两个 Handler 后还要额外传递共享状态的麻烦。但多数场景下,把编码和解码分成两个独立、职责单一的 Handler 依然是更推荐的做法——这符合 Pipeline 职责链"一个 Handler 只做一件事"的整体设计哲学。

小结

  • Netty 把"解码"从业务 Handler 里独立出来,专门用 Decoder/Encoder 处理字节与业务对象的相互转换。
  • ByteToMessageDecoder 内置累积缓冲区,自动处理"数据不够就先攒着"的半包问题,业务代码只需要写"数据足够时如何解析"的逻辑。
  • MessageToByteEncoder<T> 通过泛型参数自动过滤消息类型,避免手写 instanceof 判断。
  • 编解码逻辑独立、可复用、可单独用 EmbeddedChannel(上一篇)测试,是 Netty 编解码器设计的核心价值。
