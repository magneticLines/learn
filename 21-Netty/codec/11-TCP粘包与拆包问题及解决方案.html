TCP 粘包与拆包问题及解决方案

Netty 面试里出现频率最高的问题之一,也是所有基于 TCP 做自定义协议时绕不开的第一道坎。

📖 来源:《Netty in Action》 编解码框架章节延伸(TCP 协议特性导致的通用问题)

一、根源:TCP 是"流"协议,没有消息边界的概念

UDP 是面向消息的协议——发送方调用一次 send(),接收方对应地能通过一次 recv() 收到完整的一条消息,消息边界天然保留。但 TCP 是面向字节流的协议,它只保证字节按发送顺序、不丢不重地到达,完全不关心"发送方调用了几次 write()"这件事——操作系统内核会根据 Nagle 算法、拥塞窗口、缓冲区状态等因素,自行决定把这些字节如何打包成 TCP 报文段发送,接收方每次 read() 拿到的字节数量,和发送方每次 write() 写入的字节数量完全没有对应关系。

类比:UDP 像寄快递,一件一件包裹独立配送,拆开一个就是一件完整商品;TCP 像往一根水管里倒水,发送方倒了三杯水(三次 write),接收方在另一头接水时,可能一次接到一杯半,也可能一次接到三杯半——水管只保证"倒进去多少水,最终就流出来多少水,顺序不变",但完全不保留"分了几次倒"这个边界信息。

二、粘包与半包(拆包):同一个问题的两种表现

现象具体表现
粘包发送方连续快速发送了消息 A 和消息 B,接收方一次 read() 却把 A 和 B 的字节粘在一起收到了。
半包(拆包)发送方发送的一条较大的消息 C,被拆成了两次(甚至多次)read() 才收全,接收方第一次读到的只是 C 的前半段。
为什么这在开发和测试阶段容易被忽略:在本机测试、消息很小、发送频率不高的情况下,巧合之下往往"一次 write 对应一次 read",看起来工作正常,这会让开发者误以为消息边界是自动保留的。但只要换到真实网络环境、消息变大、并发变高,粘包/半包必然会出现——这是 TCP 协议的本质决定的,不是"运气不好才会遇到"的边缘情况,而是必须在协议设计阶段就正面解决的问题。

三、三种通用解决方案

既然 TCP 本身不提供消息边界,就只能在应用层协议里自己定义边界规则。业界总结出的通用方案主要是这三类:

方案思路Netty 对应实现取舍
固定长度每条消息都是固定的 N 字节,不够就补齐(padding)FixedLengthFrameDecoder实现最简单,但消息长度不固定的场景会浪费带宽
分隔符消息之间用一个特殊字符(如 \n)分隔,读到分隔符就认为一条消息结束LineBasedFrameDecoder、DelimiterBasedFrameDecoder直观(如 HTTP 头、Redis 协议部分场景),但消息内容里如果恰好出现分隔符本身需要转义处理
长度字段消息头部先写一个固定长度的字段表示"接下来的消息体有多长",解码器先读长度、再据此读够对应字节数LengthFieldBasedFrameDecoder最通用、最健壮,是绝大多数自定义二进制协议(RPC 框架、私有协议)的标准做法,下一篇会专门展开
为什么长度字段方案是业界主流:固定长度浪费空间,分隔符方案需要处理转义、且需要扫描每个字节找分隔符效率较低;长度字段方案只需要先读一个固定大小的数字,就能一次性知道要读多少字节,既不浪费带宽也不需要逐字节扫描,这正是 Dubbo、RocketMQ、gRPC 等主流 RPC 框架私有协议的通用设计。

四、用 EmbeddedChannel 验证一个自定义 Decoder 是否正确处理了粘包

结合上一篇学到的测试工具,验证解码器是否正确处理粘包最有效的方式,就是故意构造一个"两条消息粘在一起"的输入:

用 EmbeddedChannel 模拟粘包场景
ByteBuf sticky = Unpooled.buffer();
sticky.writeInt(5).writeBytes("hello".getBytes()); // 第一条消息
sticky.writeInt(5).writeBytes("world".getBytes()); // 第二条消息,和第一条粘在了同一个 ByteBuf 里

channel.writeInbound(sticky);
assertEquals("hello", channel.<ByteBuf>readInbound().toString(CharsetUtil.UTF_8));
assertEquals("world", channel.<ByteBuf>readInbound().toString(CharsetUtil.UTF_8)); // 应该被正确拆成两条

小结

  • 粘包/半包的根源:TCP 是面向字节流的协议,不保留发送方每次 write 的消息边界。
  • 这不是偶发问题,而是 TCP 的协议本质决定的,必须在应用层协议设计阶段就解决,不能靠"测试时看起来没问题"来判断。
  • 三种通用解法:固定长度(简单但浪费)、分隔符(直观但需转义)、长度字段(高效、通用,是业界主流)。
  • 用 EmbeddedChannel 构造粘包/半包输入进行单元测试,是验证自定义协议解码器正确性的标准做法。

与其他知识点的联系

  • Nagle 算法与 TCP_NODELAY(承接 08):粘包现象经常是因为 Nagle 算法把多次小数据量的 write 合并成一次发送导致的——这也是为什么第 8 篇提到低延迟场景要关闭 TCP_NODELAY,但即便关闭了,粘包问题依然可能因为接收方的读取时机而出现,应用层协议的边界处理始终是必须的,不能指望关掉某个 TCP 参数就能规避。
  • HTTP 协议的消息边界设计:HTTP/1.1 用 Content-Length 头(长度字段方案)或 Transfer-Encoding: chunked(类似分隔符+长度混合方案)解决同样的问题,可以对照 计算机网络 模块里 HTTP 报文格式的设计一起理解——这是同一个 TCP 特性在不同协议里催生出的相似解法。
  • 内置解码器与自定义私有协议(下两篇):本篇讲的是"为什么需要处理"和"有哪几类通用思路",下一篇会具体展开 Netty 内置的 LengthFieldBasedFrameDecoder 这个最常用的实现,再下一篇则是把这些积木组合成一份完整的私有协议设计。
