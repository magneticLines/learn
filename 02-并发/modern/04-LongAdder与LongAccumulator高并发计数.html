LongAdder 与 LongAccumulator 高并发计数

AtomicLong 用 CAS 做无锁计数已经很快了，但当成百上千个线程同时抢着自增同一个变量时，CAS 也会失守。LongAdder 给出的答案是：别抢了，各写各的。

回顾：AtomicLong 的瓶颈在哪

前面在 CAS 一章讲过，AtomicLong.incrementAndGet() 底层是"读取当前值 → 计算新值 → CAS 写回，失败就自旋重试"。这在竞争不激烈时确实比加锁快得多。但注意 CAS 的本质：所有线程都在竞争同一个内存地址。

当线程数远超 CPU 核数、并且都在高频修改同一个 AtomicLong 时，会出现大量 CAS 失败——一个线程 CAS 成功，其他所有正在自旋的线程读到的"旧值"瞬间作废，只能继续重试。这就是所谓的热点数据竞争：本质原因不是"锁"，而是"太多线程盯着同一个值"。这跟"锁竞争与可伸缩性"里讲的锁竞争问题是同一类瓶颈，只是这里连锁都没有，纯粹是 CAS 自旋的浪费。

核心思路：化整为零（分段累加）

LongAdder（JDK 8 引入）的解法非常直接：既然大家抢一个变量会冲突，那就拆成多个变量，每个线程尽量写不同的那份。

  • 内部维护一个 base 值和一个 Cell[] 数组，每个 Cell 内部包着一个独立的、经过缓存行填充（避免伪共享）的 long。
  • 低竞争时直接 CAS 更新 base，跟 AtomicLong 一样快。
  • 一旦检测到 CAS 冲突，就会给当前线程"哈希"到某个 Cell 上，之后这个线程优先更新自己命中的 Cell，而不是所有线程挤到同一个 base 上竞争。
  • 调用 sum() 汇总结果时，把 base 和所有 Cell 的值加起来返回。
AtomicLong：所有线程 ──CAS──→ [ 同一个 value ] （竞争激烈时大量自旋失败）

LongAdder： 线程A ──→ [Cell 0]
         线程B ──→ [Cell 1]
         线程C ──→ [Cell 2]
         sum() = base + Cell0 + Cell1 + Cell2 + ...
生活类比：就像超市结账，一个收银台排长队（AtomicLong 单点竞争），不如多开几个收银台分流（LongAdder 多 Cell 分散），最后对账时把每个收银台的流水加起来就是总数。

取舍：牺牲了什么换来了性能

陷阱：sum() 的过程中，如果其他线程仍在并发写入不同的 Cell，sum() 读到的是"遍历过程中的一个快照拼凑值"，不保证绝对精确——它没有把所有 Cell 都锁住再求和。对于统计 QPS、埋点计数这类"最终一致、允许短暂误差"的场景完全没问题；但如果你需要每一次 sum() 都绝对精确（比如库存扣减这种强一致场景），LongAdder 并不适合。

LongAdder vs AtomicLong：怎么选

维度AtomicLongLongAdder
低并发场景性能好，实现简单性能相当，但多了 Cell 数组的空间开销
高并发写场景大量 CAS 自旋失败，性能骤降分散到多个 Cell，性能远超 AtomicLong
读取实时性get() 绝对精确sum() 是弱一致的近似值
典型用途需要精确值的计数器（如库存、余额）高并发下的统计计数（QPS、访问量、埋点）

LongAccumulator 是 LongAdder 的泛化版：LongAdder 本质上是 LongAccumulator 在"求和"这个特定运算上的特化。LongAccumulator 允许你传入任意一个 LongBinaryOperator（比如求最大值 Long::max），思路完全一样——分散到多个 Cell 各自应用这个运算，最终再把所有 Cell 结果按同一运算聚合。

常见误区 & 面试高频

误区：认为 LongAdder 可以完全取代 AtomicLong。不是取代关系，而是场景互补——LongAdder 用弱一致性换取高并发写性能，只适合"最终求和、允许统计过程中的瞬时误差"的计数场景。
面试高频："LongAdder 为什么比 AtomicLong 快？" 答题要点：① AtomicLong 高并发下所有线程竞争同一个值，CAS 自旋失败率高；② LongAdder 用多个 Cell 分散热点，每个线程尽量写不同的 Cell，减少了 CAS 冲突；③ 代价是 sum() 不保证强一致、有额外内存占用。

核心价值总结

  • 问题根源：高并发下所有线程 CAS 同一个变量，本质是"热点竞争"，而不是加锁开销。
  • 解法：化整为零，用 Cell 数组分散写入压力，读取时再汇总。
  • 取舍：空间换时间 + 最终一致性换并发性能，不适合要求绝对精确的强一致场景。

与其他知识点的联系

  • 关联知识点：伪共享与缓存行填充。LongAdder 内部的 Cell 会做缓存行填充（padding），避免多个 Cell 落在同一条 CPU 缓存行里造成"伪共享"——这跟 JMM/CPU 缓存一致性（MESI 协议）里讲的伪共享问题是同一个坑，理解了伪共享才能明白 Cell 为什么不是简单地用一个 long[] 数组实现。
  • 关联知识点：CAS 自旋与锁竞争的本质。AtomicLong 的瓶颈和 synchronized 的锁竞争其实是同一类问题的两种表现形式——大量线程盯着同一份共享状态；LongAdder"分散热点"的思路也能类比到 ConcurrentHashMap 的分段锁/CAS+链表设计，都是"把一个大热点拆成多个小热点"这一并发优化母题的具体应用。
  • 关联知识点：ThreadLocalRandom 与线程哈希分桶。LongAdder 决定某个线程该写哪个 Cell，用的是类似 ThreadLocalRandom 内部 probe 值的哈希分桶思路（发生冲突时重新计算哈希换桶），这跟 ThreadLocal 存储、ConcurrentHashMap 的哈希桶扩容都能连起来看，都是"给每个线程/每次访问分配一个尽量不冲突的槽位"的同一套手法。
  • 关联知识点：JDK 8 并发包的整体演进。LongAdder、StampedLock 都是 JDK 8 同一批引入的"性能特化型"并发工具，配合当时 ConcurrentHashMap 从分段锁重构为 CAS+synchronized 细粒度锁，反映出 JDK 并发包整体从"通用但较重的锁"向"针对具体访问模式做定制优化"的演进方向，面试聊"JDK 8 并发新特性"时可以整体串讲。
  • 关联知识点：最终一致性，与分布式系统的类比。LongAdder 的 sum() 是"弱一致性"读取，这跟分布式系统里"最终一致性 vs 强一致性"的取舍是同一种设计哲学在不同尺度（单机多线程 vs 多机集群）上的重现，理解这里的取舍有助于理解更大范围系统设计中的一致性权衡。
