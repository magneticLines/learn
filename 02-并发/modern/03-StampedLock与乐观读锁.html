StampedLock 与乐观读锁

ReadWriteLock 已经让"读读不互斥"，但读线程之间获取共享锁时仍然要维护一套状态、存在轻量竞争。StampedLock 更进一步：读的时候干脆不加锁，读完再校验有没有被写过。

回顾：ReadWriteLock 还差在哪

前面讲过 ReentrantReadWriteLock：读锁之间共享，写锁独占，读写互斥。这在"读多写少"场景下已经比 synchronized 好很多，但读锁本质上还是"锁"——多个读线程去获取共享读锁时，仍然需要维护读者计数这类共享状态，在读线程非常多、竞争很激烈时，这个计数的维护本身也会成为瓶颈。

StampedLock（JDK 8 引入）在读写锁的悲观读、独占写之上，新增了第三种模式——乐观读（Optimistic Read）：读线程完全不加锁，就像什么都没发生一样直接读数据。

核心原理：戳（Stamp）与乐观读校验

StampedLock 的每次加锁操作都会返回一个 long 类型的"戳"（stamp），可以理解成一个版本号：

  • 乐观读 tryOptimisticRead()：不加任何锁，仅仅记录下当前的戳，然后直接去读共享变量。
  • 读完之后 validate(stamp)：检查这期间戳有没有变化。如果没有写操作发生过，戳不变，说明刚才读到的数据是有效、一致的；如果戳变了（说明中途有写线程修改了数据），刚才读到的值可能是脏的，就要升级为悲观读锁重新读一遍。
乐观读流程：
1. stamp = lock.tryOptimisticRead() // 不加锁，只记录版本号
2. 读取共享变量到局部变量 // 这期间完全没有加锁开销
3. if (!lock.validate(stamp)) { // 校验：期间是否有写操作介入
    // 校验失败，说明数据可能被改过，升级悲观读锁重新读
  }
生活类比：乐观读像是路过公告栏"扫一眼"记下内容，出来后再回头看一眼公告栏有没有被换过——没换过就放心用刚才看到的内容，换过了就重新走近去仔细看（悲观读）。而悲观读锁像是直接把公告栏锁起来看，别人这段时间读不了。

这跟我们在 CAS 一章讲过的"乐观锁"思想是同一套哲学：假设冲突很少发生，先不加锁去做，事后校验，冲突了再退化成加锁重试。区别是 CAS 校验的是一个变量的值，StampedLock 校验的是一个代表"是否发生过写"的版本戳。

为什么比 ReadWriteLock 更快

ReentrantReadWriteLock 的读锁虽然读读共享，但多个读线程仍然要竞争同一把锁对象内部的状态（AQS 的 state 字段，用 CAS 维护读者计数），高并发读时这个计数本身会成为热点。而 StampedLock 的乐观读完全不修改任何共享状态，多个读线程之间连"读读竞争"都不存在了，真正做到了"读不加锁"。

方案读读之间读写之间是否可重入
synchronized互斥互斥可重入
ReentrantReadWriteLock共享（仍需维护计数）互斥可重入
StampedLock（乐观读）无锁，互不影响不阻塞写，事后校验不可重入

陷阱：不可重入，容易死锁

陷阱：StampedLock 不支持重入！如果一个线程已经持有写锁，在同一个线程里再次调用 writeLock() 获取锁，会因为等待自己释放而永久阻塞——直接死锁。这一点跟 ReentrantReadWriteLock（顾名思义，可重入）形成鲜明对比，从别的锁迁移过来的开发者极易忽视。此外，StampedLock 也不支持 Condition，功能上是"读写锁的性能特化版"，不是"读写锁的超集"。
面试高频："StampedLock 比 ReadWriteLock 好在哪？" 答：多了乐观读模式，读线程完全不加锁，读完校验戳是否变化，适合读多写少且读操作耗时短的场景；代价是不可重入、API 更复杂、用错了容易死锁。

核心价值总结

  • 乐观读：不加锁，先读后验，验证失败再升级悲观读，进一步消除了"读读"之间的隐性竞争。
  • 适用场景：读远多于写、读操作耗时很短（校验失败重读代价小）的高并发场景。
  • 代价：不可重入、不支持 Condition，用错容易死锁，须谨慎评估是否真的需要这份极致性能。

与其他知识点的联系

  • 关联知识点：CAS 与乐观锁思想。StampedLock 的乐观读本质上是把 CAS 一章讲的"读旧值 → 算新值 → 校验没被改过 → 否则重试"的乐观并发思路，从"单个变量的原子更新"搬到了"一段读代码的一致性保证"上——校验对象从一个 int/long 值变成了一个代表"是否发生过写"的 stamp，思路是同源的。
  • 关联知识点：ReentrantReadWriteLock 与 AQS。ReentrantReadWriteLock 的读锁共享是靠 AQS 的 state 字段用 CAS 维护读者计数实现的，读线程之间仍存在对这个计数的竞争；而 StampedLock 没有基于 AQS 实现，乐观读完全不触碰任何共享状态，这也是它能在高并发读场景下比 ReadWriteLock 更快的根本原因，理解 AQS 的 state 语义才能看清两者的性能差异从哪来。
  • 关联知识点：可见性与内存屏障。乐观读读到的数据要靠 validate() 校验，而不是靠 volatile 或锁来保证可见性——这提醒我们，StampedLock 保证的"最终读到正确值"和 volatile 保证的"每次读都可见最新值"是两种不同的机制，不能把 StampedLock 修饰的字段当成不需要额外同步的普通字段来用。
  • 关联知识点：不可重入引发的死锁，与 ReentrantLock 对比。前面学 ReentrantLock/ReentrantReadWriteLock 时强调过重入计数机制能避免"自己等自己"，StampedLock 反而故意放弃了重入支持来换取更简单、更快的实现——这是一个典型的"性能与易用性/安全性此消彼长"的设计案例，面试里常拿来考察对锁重入本质的理解。
  • 关联知识点：读多写少场景的选型链条。synchronized → ReentrantReadWriteLock → StampedLock 乐观读，是同一类"读多写少"问题在不同并发强度下的递进解法，串起来看能理解"为什么 JDK 要不断引入新的锁"这条演进主线，而不是孤立记忆每个 API。
