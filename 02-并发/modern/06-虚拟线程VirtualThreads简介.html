虚拟线程 Virtual Threads 简介

本模块的收官内容：Project Loom 带来的虚拟线程，Java 21 正式发布。它没有发明新的并发原语，而是重新定义了"线程"本身的成本，让"一个请求一个线程"这种最朴素的写法重新变得可行。

回顾：传统平台线程的成本问题

我们在本模块最开始学 ThreadPoolExecutor 时就强调过：线程是昂贵的资源。每个 Java 线程（即"平台线程"，Platform Thread）背后对应一个操作系统线程，创建时要分配几百 KB 到几 MB 的栈内存，操作系统内核态的线程调度、上下文切换也有实打实的开销。这就是为什么我们要用线程池复用线程，而不是来一个请求就 new Thread()。

但线程池治标不治本：如果任务本身是阻塞 IO（等数据库、等下游 HTTP 响应），那么占用的这个平台线程在等待期间什么也干不了，纯粹是"占着茅坑"。要提升吞吐量，要么加大线程池（受限于内存和调度开销，不可能无限加），要么改写成异步回调风格（比如 CompletableFuture 链式调用）——但异步代码的可读性、调试难度都远不如同步阻塞写法直观。

虚拟线程：JVM 用户态调度的轻量级线程

虚拟线程（Virtual Thread）是 JDK 21（Project Loom 成果）引入的一种全新线程实现：它不再一对一绑定操作系统线程，而是由 JVM 自己在用户态调度，多个（成千上万个）虚拟线程共享、复用少量几个真正的操作系统线程——这些被复用的操作系统线程称为载体线程（Carrier Thread）。

传统模型：请求1 → 平台线程1（阻塞等IO时，线程整个被占用，白白闲置）
         请求2 → 平台线程2
         ...（线程数量受操作系统资源限制，加不上去）

虚拟线程模型：请求1 → 虚拟线程1 ─┐
         请求2 → 虚拟线程2 ─┼─→ 少量载体线程（如 CPU 核数个）
         请求N → 虚拟线程N ─┘ （阻塞时自动让出，去跑别的虚拟线程）

核心机制：当虚拟线程执行到阻塞 IO 操作（比如 Socket 读写、Thread.sleep()）时，JVM 会自动把这个虚拟线程从载体线程上"卸下来"（挂起并保存其调用栈状态到堆内存），让载体线程立刻去执行另一个就绪的虚拟线程；等 IO 就绪后，这个虚拟线程再被重新挂载到某个空闲的载体线程上继续执行。这个"卸下—挂载"的调度全程发生在 JVM 内部，完全不需要陷入操作系统内核态，开销极小。

生活类比：载体线程像餐厅里的服务员，虚拟线程像顾客的点单。传统模型是"一个服务员死守一桌，等顾客点完菜、吃完饭全程不能去伺候别桌"；虚拟线程模型是"顾客点完菜进入等待（阻塞 IO）时，服务员立刻转身去伺候下一桌，等这桌菜做好了再随便找个空闲服务员端过去"——服务员（载体线程）的数量远小于顾客（虚拟线程）数量，但整体翻台率（吞吐量）反而更高。

最大的意义：用同步写法获得异步的吞吐量

虚拟线程最打动人心的地方，不是"更快"，而是写法不用变：你依然写最朴素的同步阻塞代码——InputStream.read()、ReentrantLock.lock()、普通的 for 循环顺序调用——不需要改成 CompletableFuture 的链式回调或响应式编程的 Mono/Flux，JVM 在底层帮你把"阻塞等待"自动转换成了"高效调度"。这大幅降低了高并发服务端编程的心智负担和调试难度（同步代码的堆栈、断点调试远比异步回调直观）。

创建虚拟线程：API 和平台线程几乎一样
// 每个任务开一个虚拟线程，哪怕有 10 万个任务也毫无压力
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            callBlockingHttpApi(); // 同步阻塞写法，底层自动让出载体线程
        });
    }
} // try-with-resources 会等待所有任务完成

虚拟线程 vs 平台线程

维度平台线程（Platform Thread）虚拟线程（Virtual Thread）
与 OS 线程关系一对一绑定多对少，动态复用载体线程
创建/调度成本高（内核态调度、MB 级栈）极低（用户态调度，栈按需增长，占堆内存）
可创建数量几千个就吃紧可轻松创建成千上万甚至百万个
适用任务CPU 密集型计算（虚拟线程反而没有优势）IO 密集、大量阻塞等待的任务（网络调用、数据库访问）
线程池化需要，创建代价高必须复用不建议池化，即用即抛，一任务一虚拟线程
陷阱：虚拟线程不是万能加速器。对CPU 密集型任务（比如前面讲的 ForkJoinPool 数组求和），瓶颈是计算本身而不是阻塞等待，虚拟线程并不能提升吞吐量，该用 ForkJoinPool/并行流的场景仍然应该用。另外，如果代码里用了 synchronized 块做同步，虚拟线程在早期版本中执行到 synchronized 内部的阻塞操作时会被"钉住"（pinned）在载体线程上，无法让出，需要注意用 ReentrantLock 替代以获得虚拟线程的完整收益。

与操作系统"用户级线程"思想的呼应

如果你学过操作系统模块里"并发"那部分讲的用户级线程（User-Level Thread）vs 内核级线程（Kernel-Level Thread），会发现虚拟线程的设计思路正是"用户级线程"理念在 JVM 里的现代复兴：调度逻辑完全在用户态（这里是 JVM 内部）完成，不需要每次都陷入内核态，从而把线程的创建和切换成本降到极低。只是当年操作系统层面的用户级线程因为"一个线程阻塞、内核不知道整个进程都卡住"的老问题（M:1 模型的缺陷）没有大规模流行起来；而虚拟线程站在 JVM 这一层，天然掌控了所有阻塞点（IO、锁等待都会经过 JDK 自己的类库），得以精确地做"阻塞时自动让出"，绕开了这个历史难题。

常见误区 & 面试高频

误区：以为虚拟线程能让 CPU 密集型代码变快，或者认为虚拟线程数量越多性能越好。虚拟线程解决的是"阻塞 IO 浪费线程资源"的问题，不改变 CPU 总算力这个硬约束。
面试高频："虚拟线程和平台线程的本质区别？" 答题要点：① 虚拟线程由 JVM 在用户态调度，多个虚拟线程复用少量载体线程；② 阻塞 IO 时自动让出载体线程，不浪费操作系统线程资源；③ 让开发者能用同步阻塞的简单写法写出异步级别的高吞吐服务；④ 只对 IO 密集型任务有意义，不适合 CPU 密集型计算。

核心价值总结

  • 问题：传统"一个请求一个平台线程"模型下，阻塞 IO 会白白占用宝贵的操作系统线程资源，线程数量存在硬上限。
  • 方案：虚拟线程由 JVM 用户态调度，阻塞时自动让出载体线程，实现少量物理线程支撑海量并发任务。
  • 价值：用最朴素的同步阻塞写法，获得接近异步编程的吞吐量，大幅降低高并发服务端编程的复杂度。
