ForkJoinTask 的分治式任务拆分

上一篇讲了 ForkJoinPool 底层怎么调度，这一篇讲你实际要写的代码：怎么把一个大任务表达成"拆到足够小就直接算，否则 fork 出子任务再 join 合并"。

两个抽象基类：RecursiveTask vs RecursiveAction

ForkJoinTask 是提交给 ForkJoinPool 的任务基类，日常使用它的两个子类就够了：

  • RecursiveTask<V>：需要返回结果的任务，重写 compute() 返回 V，比如"求数组和"要返回一个 Long。
  • RecursiveAction：不需要返回结果的任务，重写 compute() 无返回值，比如"把数组每个元素原地加一"。

二者的区别仅仅是有没有返回值，分治的套路完全一样，可以类比之前学过的 Callable（有返回值）和 Runnable（无返回值）的关系。

经典写法：判断阈值 → 直接算 or fork/join

每个 compute() 方法都遵循同一个模板：

分治任务的通用骨架
protected V compute() {
    if (任务规模 <= 阈值) {
        // 足够小，直接顺序计算，不再往下拆
        return 直接计算结果;
    }
    // 还不够小，拆成左右两半
    SubTask left  = new SubTask(左半部分);
    SubTask right = new SubTask(右半部分);
    left.fork();              // 异步提交左任务（压入自己队列队头）
    V rightResult = right.compute(); // 右任务直接在当前线程算，省一次调度
    V leftResult = left.join();   // 等左任务结果（可能被别的线程偷去执行了）
    return merge(leftResult, rightResult);
}
小技巧：右边直接调用 compute() 而不是 fork()，只对左边 fork()，这是官方推荐的写法——能省下一次任务提交和窃取的开销，因为反正当前线程也要执行一个任务，不如让它直接算右边，别浪费。

代码示例：并行求数组和

SumTask.java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 1000; // 阈值：低于这个规模就不再拆
    private final int[] arr;
    private final int start, end;

    SumTask(int[] arr, int start, int end) {
        this.arr = arr; this.start = start; this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) sum += arr[i];
            return sum; // 足够小，直接顺序求和
        }
        int mid = (start + end) >>> 1;
        SumTask left  = new SumTask(arr, start, mid);
        SumTask right = new SumTask(arr, mid, end);
        left.fork();                     // 左边扔给其他线程（或被偷走）
        return right.compute() + left.join(); // 右边自己算，再等左边结果合并
    }
}

// 提交任务
ForkJoinPool pool = new ForkJoinPool();
long total = pool.invoke(new SumTask(arr, 0, arr.length));

阈值怎么定：拆得太细 vs 太粗

阈值设置后果
太小（拆得太细）任务对象创建、fork/join 调度开销远超实际计算量，反而比单线程慢
太大（拆得太粗）并行度不够，没能充分利用多核，退化成接近串行

实践中阈值需要结合数据规模和 CPU 核数做压测调优，没有放之四海而皆准的固定值。

常见误区 & 面试高频

误区：在 compute() 里对左右两个子任务都调用 fork() 再分别 join()。这样写不算错，但比"左 fork、右直接 compute"多了一次不必要的任务提交/窃取开销，属于能省则省的细节，但不理解原理容易在面试中说不清楚为什么官方样例这么写。
面试高频："RecursiveTask 和 RecursiveAction 的区别？" 一句话：有没有返回值，套路完全一样。再追问"分治的终止条件怎么定？" 答：设定一个阈值，任务规模小于阈值就直接算，否则继续 fork 拆分。

核心价值总结

  • RecursiveTask/RecursiveAction 只是"有无返回值"的区别，分治套路一致。
  • 核心模板：够小就直接算，否则拆成两半，一半 fork 异步执行，另一半当前线程直接算，最后 join 合并。
  • 阈值的选择直接决定并行收益，是这类算法调优的关键手段。
