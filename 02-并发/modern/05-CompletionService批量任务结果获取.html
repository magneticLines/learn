CompletionService 批量任务结果获取

提交一批异步任务后，"谁先做完就先给我谁的结果"听起来是理所应当的需求，但用 Future.get() 挨个等，反而会被排在前面、还没做完的任务拖慢整体进度。

问题：Future.get() 的"排队等待"陷阱

假设你用线程池提交了 5 个耗时不同的任务，拿到 5 个 Future，最直接的写法是：

朴素写法：按提交顺序 get()
List<Future<Result>> futures = new ArrayList<>();
for (Task task : tasks) {
    futures.add(executor.submit(task));
}
for (Future<Result> f : futures) {
    Result r = f.get(); // 阻塞等待：哪怕任务5早就做完了，也要先等任务1
    process(r);
}

问题很明显：futures 列表里的任务 1 如果恰好最慢，即使任务 2、3、4、5 早就跑完了，主线程依然卡在 futures.get(0).get() 上干等，已经完成的结果没能被第一时间处理，整体的"结果消费"被最慢的那个任务拖累。这跟前面 CompletableFuture 那一节里讲的"避免不必要的阻塞等待"是同一个思路的延伸。

CompletionService 的设计：BlockingQueue + Executor

CompletionService 不是一种新的线程池，而是把任务执行和结果获取解耦的一层封装。它的标准实现 ExecutorCompletionService 内部很简单：

  • 持有一个真正的 Executor（比如你已有的线程池）来执行任务；
  • 额外维护一个 BlockingQueue<Future<V>>；
  • 每个任务提交时，包一层：任务一旦执行完成，就自动把对应的 Future 放进这个队列；
  • 你调用 take() 或 poll() 时，直接从队列里拿——谁先完成，谁的 Future 就先出现在队列里，跟提交顺序无关。
提交：task1(慢) task2(快) task3(中等) → 线程池并发执行
完成顺序：task2 完成 → task3 完成 → task1 完成
take() 依次返回：task2 的 Future → task3 的 Future → task1 的 Future
（完全按"谁先做完"排列，而不是"谁先提交"）

代码示例

用 CompletionService 按完成顺序处理结果
ExecutorService pool = Executors.newFixedThreadPool(4);
CompletionService<Result> cs = new ExecutorCompletionService<>(pool);

for (Task task : tasks) {
    cs.submit(task); // 提交，内部自动包装完成回调
}

for (int i = 0; i < tasks.size(); i++) {
    Result r = cs.take().get(); // 谁先完成先拿到谁，不再空等最慢的任务
    process(r); // 立即处理，缩短整体的"结果消费"延迟
}
典型场景：同时向多个数据源发起查询，谁先返回就先展示谁的结果（比如比价系统同时查多个供应商报价）；或者一批子任务里只要有一个失败就要尽快感知、终止其他任务——都不适合"按提交顺序死等"。

和 CompletableFuture 的关系：不是竞争，是不同粒度的工具

CompletableFuture 更适合表达"任务之间有依赖关系的编排"（先做 A，再基于 A 的结果做 B，或者等多个都完成后再合并），也提供了 anyOf() 应对"谁先完成就用谁"的单点场景。而 CompletionService 更适合"一批同构、独立、数量不确定的任务，按完成顺序批量消费结果"这种场景，写法更直接，不需要写一堆 thenApply/thenCombine 的链式回调。

常见误区 & 面试高频

误区：以为 CompletionService 是一种新的线程池实现。它本质上只是"Executor + BlockingQueue"的组合封装，真正执行任务的还是你传进去的那个 Executor。
面试高频："CompletionService 解决了什么问题？" 一句话：让结果按完成顺序而不是提交顺序被取走，避免快任务的结果被慢任务阻塞。 底层实现关键词：内部维护 BlockingQueue，任务完成后自动把 Future 放入队列。

核心价值总结

  • 问题：按提交顺序 Future.get() 会被排在前面但还没完成的慢任务阻塞。
  • 方案：CompletionService 内部用 BlockingQueue 收集"已完成"的 Future，take() 按完成顺序返回。
  • 适用场景：一批独立同构任务，需要"谁先完成谁先处理"的批量消费模式。

与其他知识点的联系

  • 关联知识点：BlockingQueue 的生产者-消费者模型。CompletionService 内部就是把线程池包装成"生产者"（任务跑完自动把 Future 塞进队列）、把调用方包装成"消费者"（take() 阻塞拿结果），这正是前面学 BlockingQueue 时讲的生产者-消费者模式的直接应用，理解了 ArrayBlockingQueue/LinkedBlockingQueue 的阻塞语义，就能一眼看透 CompletionService 的实现原理，而不用把它当成黑箱 API。
  • 关联知识点：Future/FutureTask 的完成回调机制。"任务完成后自动放入队列"依赖的是 FutureTask 内部在任务结束时触发的收尾动作（类似 done() 钩子），这跟后面 CompletableFuture 的回调链本质上是同一个"任务完成时通知"机制的不同实现层级，前者是线程池层面的简单封装，后者是更完整的函数式编排框架。
  • 关联知识点：Executor 与线程池的解耦设计。CompletionService 不自己管理线程，而是包一层在已有 Executor 之上，这是"关注点分离"的典型体现——任务提交、任务执行、结果消费是三个独立职责，跟 ThreadPoolExecutor 把任务队列、线程管理、拒绝策略拆成可插拔组件是同一种设计思路。
  • 关联知识点：CompletableFuture 的 anyOf/allOf。本节提到两者是"不同粒度的工具"，具体来说 CompletableFuture.anyOf() 解决的是"少数几个任务里谁先完成就用谁"的单点场景，而 CompletionService 解决的是"一批数量不定的同构任务按完成顺序批量消费"，做设计选型时要按任务数量、是否需要编排依赖关系来区分该用哪个。
  • 关联知识点：面试高频对比题。"CompletionService 和 Future.get() 的区别""如何实现谁先完成谁先处理"是常见面试题，回答时如果能同时提到 BlockingQueue 的阻塞取值语义和 FutureTask 的完成通知机制，会比只背 API 用法显得更扎实。
