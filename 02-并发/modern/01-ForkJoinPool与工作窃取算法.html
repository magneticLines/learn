ForkJoinPool 与工作窃取算法

普通线程池擅长处理一堆互不相关的任务，但如果一个任务本身可以"拆成子任务，子任务再拆成孙任务"呢？ForkJoinPool 就是为这种分治式计算量身定做的执行引擎。

为什么普通线程池不够用

回忆一下前面讲过的 ThreadPoolExecutor：它维护一个共享的任务队列，所有线程从同一个队列里 take() 任务。这个模型处理"提交一批独立任务"非常合适——比如处理 100 个互不相关的 HTTP 请求。

但如果任务是递归可拆分的呢？比如"计算一个 1000 万长度数组的和"，直觉的并行做法是：拆成两半，各自求和，再把两个结果相加；每一半又可以继续往下拆，直到拆成足够小的一段直接顺序计算。这类任务如果丢给 ThreadPoolExecutor，会出现一个大麻烦：

陷阱：用普通线程池做递归拆分时，子任务在 run() 内部又通过 executor.submit() 提交新任务并 get() 等待结果——如果线程池核心线程数有限，父任务占着线程等子任务，子任务却排在队列里等不到空闲线程执行，就会造成线程饥饿式死锁。

ForkJoinPool 正是为了解决"递归拆分 + 结果合并"这种计算密集型场景而生的专用线程池，JDK 7 引入，Java 8 的 Stream.parallelStream()、CompletableFuture 默认的公共线程池（ForkJoinPool.commonPool()）背后都是它。

核心原理：工作窃取（Work-Stealing）算法

如果 ForkJoinPool 也用一个共享队列，那么每次线程 fork 出子任务、或者去 join 子任务结果时都要抢这一把公共锁，任务拆分得越细，锁竞争越激烈——这正是我们在"锁竞争与可伸缩性"里讲过的热点竞争问题。ForkJoinPool 的解法是彻底放弃共享队列：

  • 每个工作线程都有自己的双端队列（deque），线程 fork 出子任务时，把子任务压入自己队列的队头，自己也优先从队头取任务执行（后进先出，局部性好，符合分治"先拆的后合并"的天然顺序）。
  • 当某个线程把自己的队列处理完、变得空闲时，不会傻等，而是随机挑一个其他线程，从它队列的队尾"偷"一个任务过来执行（先进先出，偷最老、通常也是最大的任务，减少下次还要偷的次数）。
  • 自己操作自己的队头、别人偷队尾，两端操作互不冲突，绝大多数时候完全不需要加锁，只有队列里剩最后一个元素时才需要 CAS 竞争。
线程 A 的双端队列：[任务4] [任务3] [任务2] [任务1] ← A 从队头 pop 执行
线程 B（队列已空，太闲了）：────────偷──────→ 从 A 的队尾 steal 任务4

A 和 B 分别操作队列两端，天然减少了竞争
生活类比：就像几个人分头去搬一堆箱子，每人面前有自己的一摞箱子，先搬完的人不会闲着，而是走到最慢那个人的箱子堆最底下抽几个来搬——不会跟对方抢他手头正在搬的那几个（队头），减少了争抢。

ForkJoinPool vs ThreadPoolExecutor：怎么选

维度ThreadPoolExecutorForkJoinPool
任务特征互相独立、不可再拆分可以递归拆分为子任务的分治任务
典型场景IO 密集型（数据库查询、HTTP 请求）CPU 密集型（数值计算、大数组/大集合处理）
队列模型1 个共享阻塞队列每线程 1 个双端队列 + 工作窃取
任务提交方外部线程统一提交任务内部可以继续 fork 出子任务
线程数设置常配合 IO 等待，可以多于 CPU 核数默认等于 CPU 核数（Runtime.availableProcessors()）

一句话记忆：ThreadPoolExecutor 管"一堆互不相关的任务"，ForkJoinPool 管"一个任务拆出来的一棵任务树"。如果拿 ForkJoinPool 去跑一堆阻塞 IO 任务，少数几个核心线程数会成为瓶颈；反过来拿 ThreadPoolExecutor 做递归分治，则容易踩到前面说的线程饥饿死锁。

常见误区 & 面试高频

误区：以为工作窃取能保证任务"公平"分配。恰恰相反，工作窃取的设计目标是整体吞吐量最大化，而不是公平——空闲线程主动找活干，忙的线程完全不受打扰，这与 AQS 公平锁"先来后到"的设计哲学是相反的取向。
面试高频："ForkJoinPool 和普通线程池的区别？" 答题要点：① 每线程独立双端队列 + 工作窃取，减少竞争；② 面向可递归拆分的计算密集型任务；③ Java 8 的并行流、CompletableFuture 默认都用 ForkJoinPool.commonPool()。

核心价值总结

  • 问题：递归分治任务用普通线程池容易死锁，共享队列在任务拆分很细时锁竞争严重。
  • 方案：每线程一个双端队列，自己走队头，被偷走队尾，把锁竞争降到最低。
  • 适用边界：CPU 密集、可递归拆分的计算任务，不要用来跑阻塞 IO。

与其他知识点的联系

  • ThreadPoolExecutor 的共享队列模型：ForkJoinPool 的每线程双端队列设计，正是为了规避 ThreadPoolExecutor 那种"所有线程抢同一把队列锁"的竞争模式——理解了 ThreadPoolExecutor 里 workQueue 是单一共享阻塞队列，才能体会到工作窃取"每线程私有队列 + 只在队尾窃取时才竞争"的设计有多精妙，这是两种线程池在队列结构上的根本分歧点。
  • 锁竞争与可伸缩性（上一篇）：工作窃取算法本质上是"减少共享状态、把竞争从全局收缩到局部"这一伸缩性原则的具体实现——队头无锁访问、队尾窃取才需要 CAS，是"尽量让每个线程操作自己的数据，只在必要时才同步"这一通用原则在调度器设计上的落地。
  • Stream.parallelStream()：Java 8 的并行流底层默认复用 ForkJoinPool.commonPool()，这意味着你写的每一处 parallelStream() 实际上都在和其他并行流、CompletableFuture.supplyAsync()（不指定线程池时）共享同一个公共池——如果不了解这一点，很容易在多处滥用并行流后遇到"任务互相挤占导致的性能不升反降"的问题。
  • 操作系统调度与 CPU 亲和性：ForkJoinPool 默认把并行度设为 Runtime.availableProcessors()，这是因为计算密集型任务的最优并发数应该贴近物理核数，超过核数只会增加上下文切换开销而不提升吞吐——这与 IO 密集型线程池"线程数可以远超核数"的经验法则形成直接对比，根源在于 CPU 密集任务不存在"等待期"，线程闲不下来去做别的事。
  • AQS 公平锁：文中提到工作窃取以吞吐量优先、不保证公平，这与 ReentrantLock 公平模式下严格按 FIFO 顺序分配锁的设计目标正好相反——两者代表了并发系统设计里"公平性"与"吞吐量"这对经典取舍的两个不同选择，理解其中一个的取舍，有助于举一反三理解另一个。
