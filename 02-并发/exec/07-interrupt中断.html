07. 中断机制 interrupt 的正确姿势

理解 Java 中断协议的本质，掌握"请求—响应"协作模型，彻底告别吞异常与死循环两大陷阱。

① 线程停止为什么是个难题

Java 1.0 提供了 Thread.stop()，它会在目标线程执行到任意字节码处强行抛出 ThreadDeath。听起来很彻底，但问题在于：

  • 对象可能只写了一半字段，锁已经被自动释放——共享状态不一致。
  • 调用方根本不知道对象处于什么状态，无法做清理。

这就像写文件写到一半直接断电，文件系统没有 fsync 就崩了。

设计哲学：Java 放弃了"强制杀死"，转而采用协作式中断（Cooperative Interruption）——调用方只是发出一个"请停下来"的信号，被中断线程自己决定何时、如何停止。这保证了对象不变式始终成立。

中断机制的核心是 Thread 里一个 boolean 标志位（中断状态），配合三个 API：

方法含义是否清除标志位
t.interrupt()设置目标线程的中断标志为 true；若目标正阻塞在 sleep/wait/join 则立刻抛 InterruptedException抛异常时清除
Thread.interrupted()查询当前线程的中断状态（静态方法）是（副作用！）
t.isInterrupted()查询指定线程的中断状态（实例方法）否

② 阻塞操作与 InterruptedException 的关系

JDK 的阻塞方法（Thread.sleep、Object.wait、BlockingQueue.take、Lock.lockInterruptibly 等）在等待期间会持续检查中断标志。一旦发现标志为 true，它们会：

  1. 清除中断标志（重置为 false）
  2. 抛出 InterruptedException

这就是为什么 InterruptedException 是受检异常——JDK 强迫你在编译时处理它，否则代码根本编译不过。它不是"出错了"，而是"有人请求我停止"。

陷阱：很多初学者写 catch (InterruptedException e) { } 直接吞掉异常，中断标志也被清除了，上层调用者永远无法感知中断请求——这是并发 Bug 的温床。

③ 两种正确的处理策略

策略一：向上传播（推荐，最简洁）

如果你的方法本身就是"可中断"的，直接在方法签名上声明 throws InterruptedException，让调用链自己决定如何处理。

策略一：直接传播Java
// 不捕获，直接声明抛出——调用方得到完整的中断信号
public BlockingTask take() throws InterruptedException {
    return queue.take();  // 直接传播 InterruptedException
}

策略二：捕获后恢复中断标志（当不能抛出时）

某些情况下你实现的是 Runnable.run() 或者框架接口，签名不允许抛出受检异常。此时正确做法是：catch 后立即调用 Thread.currentThread().interrupt() 重新设置标志，保证上层仍能感知到中断请求。

策略二：捕获后恢复标志Java
@Override
public void run() {
    while (!Thread.currentThread().isInterrupted()) {
        try {
            Task task = queue.take();
            process(task);
        } catch (InterruptedException e) {
            // 无法向上传播（run() 不允许抛出受检异常）
            // 必须恢复标志，否则循环条件下一次检查会看到 false，无法退出
            Thread.currentThread().interrupt();
            break;  // 或 return，视业务决定清理逻辑
        }
    }
}
记忆口诀：要么 throws，要么 re-interrupt，永远不要只 catch 不处理。

④ 反例 → 问题 → 正例完整对比

反例 A：吞掉异常（最常见）

反例 A — 吞异常Java
public void doWork() {
    try {
        Thread.sleep(1000);
    } catch (InterruptedException e) {
        // ❌ 什么都不做——中断信号消失，线程无法被外部取消
    }
}

问题：调用 shutdownNow() 或 future.cancel(true) 之后，线程仍然继续跑，取消操作完全失效。

反例 B：用 stop() / destroy()（已废弃）

反例 B — 已废弃 APIJava
thread.stop();    // ❌ 已废弃，可能导致共享状态不一致
thread.destroy(); // ❌ 从未实现，直接抛 UnsupportedOperationException

反例 C：用 volatile 标志但忽略阻塞

反例 C — volatile 标志无法唤醒阻塞线程Java
volatile boolean cancelled = false;

public void run() {
    while (!cancelled) {
        // ❌ 如果 queue.take() 永久阻塞，cancelled=true 根本没机会被检查
        Task t = queue.take();
        process(t);
    }
}

问题：线程阻塞在 take() 时，即使设置 cancelled = true 也不会被唤醒。interrupt() 才能打断阻塞。

正例：interrupt + isInterrupted 双重检查

正例 — 完整可取消的工作线程Java
public class CancellableWorker implements Runnable {

    private final BlockingQueue<Task> queue;

    public CancellableWorker(BlockingQueue<Task> queue) {
        this.queue = queue;
    }

    @Override
    public void run() {
        // 循环条件：检查中断标志（非阻塞路径用 isInterrupted）
        while (!Thread.currentThread().isInterrupted()) {
            try {
                // take() 是阻塞点：中断会让它立刻抛 InterruptedException
                Task task = queue.take();
                process(task);
            } catch (InterruptedException e) {
                // 恢复标志，让循环条件在下一次迭代感知到中断并退出
                Thread.currentThread().interrupt();
            }
        }
        // 到达此处：做清理工作（关闭资源、写日志等）
        cleanup();
    }

    private void process(Task task) { /* ... */ }
    private void cleanup()          { /* 关闭 DB 连接、flush 缓冲区 */ }
}

⑤ 中断在 ExecutorService 中的表现

当你调用 future.cancel(true) 或 executorService.shutdownNow() 时，背后就是对执行任务的线程调用 interrupt()。

future.cancel(true) 的底层行为Java
ExecutorService exec = Executors.newSingleThreadExecutor();
Future<?> future = exec.submit(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        // 可响应中断的工作循环
        doSomeWork();
    }
});

// 参数 true 表示：若任务正在运行，对其线程调用 interrupt()
future.cancel(true);

// shutdownNow() 同理：返回未执行的任务列表，并中断正在执行的线程
List<Runnable> pending = exec.shutdownNow();
重要推论：如果你的任务吞掉了 InterruptedException，shutdownNow() 发出的中断信号就石沉大海，线程池永远关不掉，JVM 进程无法优雅退出。

⑥ interrupted() vs isInterrupted() 易混点

静态方法有副作用——这是高频面试陷阱Java
Thread.currentThread().interrupt();  // 设置标志

// Thread.interrupted() 是静态方法，检查并清除当前线程的标志
boolean a = Thread.interrupted();  // true，同时清除标志
boolean b = Thread.interrupted();  // false！标志已被第一次调用清除

// isInterrupted() 是实例方法，只查询不清除
Thread.currentThread().interrupt();
boolean c = Thread.currentThread().isInterrupted();  // true
boolean d = Thread.currentThread().isInterrupted();  // 仍然 true
方法作用域清除标志推荐使用场景
Thread.interrupted()当前线程（静态）是在循环内消费中断并决定是否继续
t.isInterrupted()任意线程（实例）否查询中断状态但不想清除（最常用）

⑦ 不可中断的阻塞——额外手段

不是所有阻塞都能被 interrupt() 打断。遇到这些情况需要额外处理：

阻塞类型interrupt 是否有效替代方案
等待 synchronized 锁无效（继续阻塞）改用 Lock.lockInterruptibly()
Socket I/O（InputStream.read）无效关闭底层 Socket：socket.close()
NIO Selector 阻塞无效调用 selector.wakeup()
Thread.sleep / wait / take有效，抛 IE——
对 synchronized 无效 → 改用 lockInterruptiblyJava
ReentrantLock lock = new ReentrantLock();

try {
    // 若等待锁期间收到中断，立刻抛 InterruptedException
    lock.lockInterruptibly();
    try {
        doLockedWork();
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复标志
}

⑧ 整体协作流程图

调用方
  └─ thread.interrupt()  ← 发出请求
       │
       ▼
目标线程
  ├─ 正在阻塞（sleep/wait/take）
  │   └─ 立刻抛 InterruptedException（标志被清除）
  │       └─ catch → 策略1: throws IE / 策略2: re-interrupt
  │
  └─ 正在运行（非阻塞）
       └─ 下次循环检查 isInterrupted() → true → 退出循环
               └─ cleanup() → 线程正常结束

⑨ 面试高频考点速查

  • interrupt() 是请求，不是命令。目标线程可以忽略它（尽管这是错误做法）。
  • sleep() 抛出 IE 后，中断标志被清除。这是 catch 里必须 re-interrupt 的根本原因。
  • Thread.interrupted() 有副作用，调用后标志变 false——这是超高频考点，很多人混淆静态/实例方法。
  • synchronized 锁等待对 interrupt 不响应，只有 lockInterruptibly() 才能被中断。
  • shutdownNow() 的本质就是对每个正在执行任务的线程调用 interrupt()，任务本身必须正确处理中断才能真正停止。
一句话总结：中断是 Java 协作式取消的信号协议——发出方负责设置标志，接收方负责检查并响应，绝不能悄悄吞掉信号。

⑩ 与其他知识点的联系

  • 线程池的优雅关闭（下一篇）：shutdownNow() 的底层实现就是对每个工作线程调用 interrupt()——本文讲的"中断只是请求，任务必须主动检查才会真正停下"这条规则，直接决定了下一篇里"任务写了 while(true) 不检查中断标志会导致线程池永久无法终止"这个陷阱的根本原因，两篇内容是同一个机制在不同层面的体现。
  • AQS（AbstractQueuedSynchronizer）：ReentrantLock.lockInterruptibly() 之所以能在等待锁期间响应中断，是因为 AQS 的 acquireInterruptibly 方法在每次被唤醒时都会检查中断标志并抛出 InterruptedException，而 synchronized 对中断完全无感，是因为 synchronized 的锁等待逻辑写在 JVM 层（monitor），根本没有暴露中断检查点——这是 Lock 相比 synchronized 的一个关键优势。
  • Future.cancel(true) 与 CompletableFuture：取消一个正在执行的异步任务，本质上也是靠 interrupt 信号驱动，如果任务内部执行的是不响应中断的阻塞操作（如 Socket I/O），cancel(true) 会显得"没反应"，这与本文⑦节讲的"不可中断阻塞"是同一个坑，排查异步任务取消不掉的问题时要先检查任务内部用了什么阻塞 API。
  • JUC 阻塞队列（BlockingQueue）：take()/put() 之所以要声明 throws InterruptedException，正是因为它们内部通过 Condition.await()（基于 AQS）实现阻塞等待，而 Condition.await() 的语义就包含"响应中断并清除标志后抛异常"，这解释了为什么几乎所有 JUC 的阻塞方法都是受检异常而不是普通阻塞。
  • 面试高频考点："如何优雅地停止一个线程" 这类问题的标准答案就是本文的核心内容——不要用废弃的 stop()，而是用 interrupt() 配合 volatile 标志或可中断的阻塞 API，这也是几乎每一轮 Java 并发面试都会问到的必考题。
