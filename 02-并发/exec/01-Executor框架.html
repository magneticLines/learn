Part III 结构化应用 · 任务执行

01. Executor 框架与解耦思想

从"手工管理线程"到"提交任务即可"——理解 Executor 的设计哲学与并发解耦模型。

为什么需要 Executor 框架？

在没有任何框架的年代，Java 程序员写并发代码只有一条路：手工创建和管理 Thread。 表面上看很直接，但暗藏大量并发陷阱。

反例：每个请求 new 一个线程

ThreadPerTask.java（反例）
// ❌ 反例：无上限地为每个任务创建线程
public class ThreadPerTaskServer {
    public static void main(String[] args) throws Exception {
        ServerSocket ss = new ServerSocket(8080);
        while (true) {
            Socket conn = ss.accept();
            // 每来一个连接就 new 一个线程——并发量一大就 OOM
            new Thread(() -> handle(conn)).start();
        }
    }
}

这段代码至少有三个并发问题：

  1. 线程创建开销不可控：每次 new Thread() 都要在 OS 内核分配栈空间（默认 512KB–1MB），10000 个并发连接 = 10000 个线程 = 直接 OOM。
  2. 任务提交与任务执行强耦合：谁提交任务，谁就决定执行策略（几个线程、排队还是丢弃）。换一种策略就得改业务代码。
  3. 没有生命周期管理：线程跑完自生自灭，出错无处感知，也无法优雅关闭。
核心矛盾：线程是昂贵的系统资源，任务是轻量的业务逻辑。把它们 1:1 绑定，就像每次外卖都买一辆摩托车，用完扔掉。

Executor 接口：一句话抽象

Java 5 引入 java.util.concurrent.Executor，它只有一个方法：

Executor.java（JDK 源码精简）
public interface Executor {
    // 提交一个 Runnable，具体怎么跑——我不管
    void execute(Runnable command);
}

这一个方法完成了最关键的设计决策：把"提交任务"和"执行任务"解耦。 调用方只需关心"我有一个任务"，执行策略（线程池大小、排队方式、拒绝策略）完全由 Executor 的实现决定。

设计模式视角：这是标准的命令模式（Command Pattern）——Runnable 是命令对象，Executor 是调用者，真正的执行者（线程池、当前线程、新线程）对客户端透明。

接口层次：三层递进

Executor ← 只有 execute(Runnable)
└── ExecutorService ← 增加生命周期管理 + submit/invokeAll
└── ScheduledExecutorService ← 再增加定时 / 周期调度
接口新增能力核心方法
Executor 任务提交与执行解耦 execute(Runnable)
ExecutorService 生命周期管理、返回值支持 submit() shutdown() awaitTermination() invokeAll()
ScheduledExecutorService 延迟 / 周期执行 schedule() scheduleAtFixedRate()

ThreadPoolExecutor：真正的实现核心

所有常用线程池（newFixedThreadPool、newCachedThreadPool 等）底层都是 ThreadPoolExecutor，它由 5 个核心参数共同决定线程池行为：

ThreadPoolExecutor 构造参数
new ThreadPoolExecutor(
    4,                             // corePoolSize：常驻线程数
    16,                            // maximumPoolSize：最大线程数
    60, TimeUnit.SECONDS,         // keepAliveTime：空闲线程存活时长
    new LinkedBlockingQueue<>(1000), // workQueue：任务排队缓冲区
    new ThreadPoolExecutor.CallerRunsPolicy() // rejectedHandler：队列满时的策略
);

执行流程的决策树（这是面试必考点，要能背诵）：

提交任务
├── 当前线程数 < corePoolSize → 新建线程立即执行
├── 当前线程数 ≥ corePoolSize → 尝试加入 workQueue
│ ├── 队列未满 → 入队等待
│ └── 队列已满
│ ├── 线程数 < maximumPoolSize → 新建非核心线程执行
│ └── 线程数 = maximumPoolSize → 触发 RejectedExecutionHandler
└── 核心线程空闲超 keepAliveTime → 回收（core 默认不回收）

Executors 工厂方法与其陷阱

JDK 提供 Executors 工具类快速创建线程池，但每种工厂方法都有其适用场景和潜在陷阱：

工厂方法底层配置适用场景陷阱
newFixedThreadPool(n) core=max=n，无界队列 CPU 密集型，线程数可控 队列无上限，任务堆积可 OOM
newCachedThreadPool() core=0，max=Integer.MAX_VALUE 短生命周期大量异步任务 线程数无上限，瞬时并发高时 OOM
newSingleThreadExecutor() core=max=1，无界队列 串行化任务，保证顺序 同 Fixed，队列无上限
newScheduledThreadPool(n) DelayedWorkQueue 定时/周期任务 任务抛异常会静默终止调度
阿里巴巴 Java 开发手册明令禁止直接使用 Executors 工厂方法，原因正是上表中的 OOM 风险。 生产环境应显式构造 ThreadPoolExecutor，让每个参数都有意图可查。

正例：用 ExecutorService 重写 Web 服务器

ExecutorServer.java（正例）
public class ExecutorServer {

    // 显式构造：所有参数意图清晰，有边界保护
    private static final ExecutorService POOL = new ThreadPoolExecutor(
        8, 32,
        60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(500),       // 有界队列，防止任务堆积
        new ThreadFactoryBuilder().setNameFormat("server-worker-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy()   // 背压：让调用方自己跑，起到限速效果
    );

    public static void main(String[] args) throws Exception {
        ServerSocket ss = new ServerSocket(8080);
        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            // 优雅关闭：不再接受新任务，等已提交任务跑完
            POOL.shutdown();
            try {
                if (!POOL.awaitTermination(30, TimeUnit.SECONDS))
                    POOL.shutdownNow();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }));

        while (!POOL.isShutdown()) {
            Socket conn = ss.accept();
            // 只提交任务，不关心谁去跑——解耦完成
            POOL.execute(() -> handle(conn));
        }
    }
}

解耦的深层意义：策略可替换

当任务提交与执行策略解耦后，你可以在不修改业务代码的前提下切换执行策略：

ExecutionPolicy.java — 同一任务，三种执行策略
Executor executor;

// 策略 1：单线程串行（测试/调试用，方便复现问题）
executor = Executors.newSingleThreadExecutor();

// 策略 2：固定大小线程池（生产常用）
executor = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());

// 策略 3：同步执行（在调用线程跑，单元测试追求确定性时有用）
executor = Runnable::run;  // Executor 是函数式接口！

// 业务代码完全不变：
executor.execute(() -> processOrder(orderId));
单元测试技巧：将 Executor 作为依赖注入，测试时传入 Runnable::run（同步执行）， 生产时传入真实线程池。这样测试完全确定性，不需要 sleep/CountDownLatch 来等待异步结果。

并发陷阱汇总

陷阱 1：忘记关闭 ExecutorService

ExecutorService 内部持有线程，JVM 不会因为 main 方法结束就自动关闭它（非守护线程会阻止 JVM 退出）。 必须显式调用 shutdown()，或将线程设置为守护线程。

陷阱 2：用 execute() 丢失异常

异常丢失问题
// ❌ execute() 中抛出的 RuntimeException 会被 UncaughtExceptionHandler 捕获
//    但如果没有设置 handler，异常会被静默打印到 stderr 然后消失
pool.execute(() -> {
    throw new RuntimeException("我消失了");  // 调用方永远不知道
});

// ✅ 用 submit() 返回 Future，异常会被包装进 ExecutionException
Future<?> f = pool.submit(() -> {
    throw new RuntimeException("我被捕获了");
});
try {
    f.get();  // 这里会抛出 ExecutionException，cause 就是原始异常
} catch (ExecutionException e) {
    log.error("任务失败", e.getCause());
}

陷阱 3：ScheduledExecutorService 任务静默消亡

用 scheduleAtFixedRate 提交的周期任务，一旦任务体内抛出未被捕获的异常， 调度会永久停止，且没有任何警告——这是生产环境中最难排查的 bug 之一。

ScheduledTask 保护写法
scheduler.scheduleAtFixedRate(() -> {
    try {
        doWork();  // 真正的业务逻辑
    } catch (Exception e) {
        // 必须在这里捕获所有异常，否则调度链断裂
        log.error("定时任务异常，本次跳过", e);
    }
}, 0, 1, TimeUnit.MINUTES);

陷阱 4：拒绝策略选择不当

拒绝策略行为适用场景
AbortPolicy（默认）抛出 RejectedExecutionException快速失败，调用方需处理异常
CallerRunsPolicy由提交线程自己执行背压限速，保证任务不丢失
DiscardPolicy静默丢弃允许丢失、容忍数据损失
DiscardOldestPolicy丢弃队列头部最旧任务新任务优先，旧任务可牺牲

与 Spring / 框架的联系

Spring 的 @Async、Spring Boot 的 TaskExecutor、Spring MVC 的 AsyncTaskExecutor 底层全部基于 ExecutorService。 Spring 只是在框架层面帮你管理线程池的生命周期，核心抽象还是 JDK 的 Executor。

Spring Boot 配置线程池的正确姿势
@Configuration
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
        exec.setCorePoolSize(8);
        exec.setMaxPoolSize(32);
        exec.setQueueCapacity(500);
        exec.setThreadNamePrefix("async-");
        exec.initialize();
        return exec;
    }
}

小结

  • Executor 的核心价值是解耦"任务提交"与"执行策略"，让业务代码不感知并发细节。
  • ThreadPoolExecutor 是绝大多数场景的实现，5 个核心参数决定线程池行为，生产代码请显式构造。
  • 执行流程优先级：核心线程 → 队列 → 非核心线程 → 拒绝策略，牢记这条决策链。
  • execute() 会静默丢失异常；需要感知结果和异常时，用 submit() 配合 Future.get()。
  • 定时任务必须在任务体内 try-catch，否则一次异常会永久终止调度。
  • 下一节将深入 Callable、Future、FutureTask——有了 Executor 解耦基础，这三个类负责把"异步结果"带回调用方。
