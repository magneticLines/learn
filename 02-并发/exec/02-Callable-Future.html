Part III 结构化应用 · 任务执行

02. Callable / Future / FutureTask

让任务能够"带着答案回来"——Java 异步计算结果的标准抽象，以及它们背后的设计取舍。

为什么 Runnable 不够用？

Runnable 是 Java 最早的任务抽象，接口非常简单：

Runnable.java（JDK 源码）
public interface Runnable {
    void run();  // 无返回值，无法抛出受检异常
}

它有两个硬伤：

  • 无返回值：任务算完结果存哪？只能写到共享变量，引发可见性/竞态问题。
  • 不能抛受检异常：网络 IO、数据库操作必须在内部 try-catch，错误信息容易被吞掉。

JDK 5 用 Callable<V> 填补了这两个空缺，同时引入 Future<V> 作为"异步结果的占位符"，两者配合就构成了 Java 标准的异步任务模型。

三个核心接口 / 类的关系

<<interface>> Callable<V> — 描述「有结果的任务」 └── V call() throws Exception <<interface>> Future<V> — 描述「异步结果的句柄」 ├── boolean isDone() ├── boolean cancel(boolean mayInterrupt) ├── V get() throws ... └── V get(long timeout, TimeUnit unit) throws ... FutureTask<V> — 同时实现 Runnable + Future ├── 持有一个 Callable（或包装后的 Runnable） ├── 状态机：NEW → COMPLETING → NORMAL / EXCEPTIONAL / CANCELLED └── 可直接提交给 Thread / Executor，也可手动 run()

可以把 Future 理解为一张快递单：你下单（提交 Callable），快递员（线程池）在后台打包发货，你拿着这张单随时可以查快递状态（isDone()）、取消订单（cancel()），或者坐在那等包裹到手（get()）。

Callable 接口详解

Callable.java（JDK 源码）
@FunctionalInterface
public interface Callable<V> {
    V call() throws Exception;
}

与 Runnable 的对比：

特性RunnableCallable<V>
返回值voidV（泛型）
受检异常不允许允许 throws Exception
配套工具Thread / ExecutorExecutorService.submit()
适用场景fire-and-forget需要结果 / 需要异常传播

Future 接口详解

Future<V> 核心方法
public interface Future<V> {

    // 阻塞直到有结果，或抛出异常
    V       get() throws InterruptedException, ExecutionException;

    // 带超时的 get，超时抛 TimeoutException
    V       get(long timeout, TimeUnit unit)
                throws InterruptedException, ExecutionException, TimeoutException;

    // 任务是否已结束（正常完成 / 异常 / 取消 均算 done）
    boolean isDone();

    // 尝试取消；mayInterruptIfRunning=true 会给执行线程发中断信号
    boolean cancel(boolean mayInterruptIfRunning);

    // 是否被取消（cancel 成功 → true）
    boolean isCancelled();

}
ExecutionException 的设计意图
Callable.call() 中抛出的任何异常都会被包裹进 ExecutionException，通过 getCause() 取出真实异常。这样做是为了跨线程传播异常——子线程的受检异常被安全地"搬运"到调用 get() 的线程。

FutureTask：唯一内置实现

FutureTask 同时实现了 RunnableFuture（即 Runnable + Future），是 JDK 里 Future 的唯一标准实现。它的核心是一个状态机：

NEW │ run() 开始执行 ▼ COMPLETING ← 过渡态（outcome 写入中），极短暂 │ ├─► NORMAL ← call() 正常返回，outcome = 返回值 ├─► EXCEPTIONAL ← call() 抛异常，outcome = Throwable └─► CANCELLED ← cancel(false) 在运行前取消 │ └─► INTERRUPTING → INTERRUPTED ← cancel(true) 中断执行中的线程

状态一旦离开 NEW 就不可逆，这保证了 get() 的结果只会被写入一次（happens-before 语义由 volatile + CAS 保证，无需额外同步）。

FutureTask 的两种构造方式

FutureTask 构造
// 方式 1：包装 Callable（推荐）
FutureTask<String> task1 = new FutureTask<>(
    () -> "hello from callable"
);

// 方式 2：包装 Runnable，并指定完成后的返回值
FutureTask<String> task2 = new FutureTask<>(
    () -> System.out.println("running"),
    "done"  // Runnable 完成后 get() 返回这个值
);

标准用法：配合 ExecutorService

BasicFutureDemo.java
ExecutorService pool = Executors.newFixedThreadPool(4);

// submit(Callable) 内部会把 Callable 包成 FutureTask 再提交
Future<Integer> future = pool.submit(() -> {
    Thread.sleep(1000);          // 模拟耗时计算
    return expensiveCompute();
});

// 提交后主线程可以做其他事
doOtherWork();

try {
    // 阻塞等结果，建议始终指定超时
    Integer result = future.get(5, TimeUnit.SECONDS);
    System.out.println("结果: " + result);
} catch (TimeoutException e) {
    future.cancel(true);         // 超时取消，释放线程
    throw new RuntimeException("计算超时", e);
} catch (ExecutionException e) {
    // 子线程内的异常被包裹在此，getCause() 取真实异常
    Throwable cause = e.getCause();
    logger.error("任务执行异常", cause);
} finally {
    pool.shutdown();
}

进阶：手动使用 FutureTask

有时你需要手动控制 FutureTask，例如在 缓存预热 场景中，保证相同的昂贵计算只执行一次：

Memoizer.java — 用 FutureTask 实现线程安全的缓存
public class Memoizer<K, V> {

    // ConcurrentHashMap 保证 map 操作的线程安全
    private final ConcurrentHashMap<K, FutureTask<V>> cache
        = new ConcurrentHashMap<>();

    private final Function<K, V> loader;

    public V get(K key) throws Exception {
        while (true) {
            FutureTask<V> f = cache.get(key);
            if (f == null) {
                FutureTask<V> ft = new FutureTask<>(() -> loader.apply(key));
                // putIfAbsent：只有当 key 不存在时才插入，原子操作
                f = cache.putIfAbsent(key, ft);
                if (f == null) {
                    f = ft;
                    ft.run();    // 由当前线程同步执行（或提交给线程池）
                }
            }
            try {
                return f.get();
            } catch (CancellationException e) {
                // 被取消时清除缓存，下次重试
                cache.remove(key, f);
            }
        }
    }
}
为什么用 FutureTask 而不是直接存结果 V？
如果存的是 V，两个线程同时判断 cache 为空就会各自计算一次（check-then-act 竞态）。存 FutureTask 后，第二个线程拿到已有的 FutureTask 直接 get() 阻塞等待，第一个线程计算完 FutureTask 就完成，所有等待者自动唤醒——这是 Future 作为"一次性屏障" 的经典用法。

并发陷阱：反例 vs 正例

陷阱 1：get() 忘写超时 → 线程池雪崩

反例：无限阻塞
// ❌ 如果远程服务挂起，这个调用线程永远阻塞
String result = future.get();
正例：有界等待
try {
    // ✅ 始终传超时参数，结合业务 SLA 设置合理值
    String result = future.get(3, TimeUnit.SECONDS);
} catch (TimeoutException e) {
    future.cancel(true);   // 超时后一定要 cancel，避免线程池中任务继续跑
    throw new ServiceUnavailableException("下游超时");
}

陷阱 2：循环提交后顺序 get() → 串行化陷阱

反例：并行提交但串行等待，慢任务拖累所有结果
List<Future<Result>> futures = new ArrayList<>();
for (Task t : tasks) futures.add(pool.submit(t));

// ❌ 按提交顺序逐一 get()：第一个任务最慢则后面全阻塞
for (Future<Result> f : futures) process(f.get());
正例：用 ExecutorCompletionService 按完成顺序消费
ExecutorCompletionService<Result> ecs
    = new ExecutorCompletionService<>(pool);

for (Task t : tasks) ecs.submit(t);

// ✅ take() 返回最先完成的那个 Future，不被慢任务阻塞
for (int i = 0; i < tasks.size(); i++) {
    Future<Result> f = ecs.take();  // 按完成顺序出队
    process(f.get());
}

陷阱 3：cancel() 不等于真正停止任务

cancel(true) 的局限性
// cancel(true) 只是发中断信号，任务代码必须主动响应
Future<Void> f = pool.submit(() -> {
    // ❌ 死循环不检查中断，cancel(true) 完全无效
    while (true) doWork();
    return null;
});

f.cancel(true);  // Future 状态变 CANCELLED，但线程仍在跑！

// ✅ 正确做法：任务内部检查中断状态
Future<Void> f2 = pool.submit(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doWork();
    }
    return null;
});
cancel(false) vs cancel(true)
cancel(false)：只阻止任务被启动，如果已在运行则不干预。
cancel(true)：给执行线程发中断信号（Thread.interrupt()），但任务代码必须主动检查 isInterrupted() 或调用会抛 InterruptedException 的方法（如 sleep/wait）才能真正停下来。

陷阱 4：ExecutionException 被吞掉

反例 vs 正例
// ❌ 反例：吞掉异常，生产问题无从排查
try {
    future.get();
} catch (Exception e) {
    e.printStackTrace();  // 打的是 ExecutionException，真实原因在 cause 里
}

// ✅ 正例：分类处理，透传真实原因
try {
    future.get(5, TimeUnit.SECONDS);
} catch (ExecutionException e) {
    Throwable cause = e.getCause();
    if (cause instanceof IOException)  throw (IOException) cause;
    if (cause instanceof RuntimeException) throw (RuntimeException) cause;
    throw new IllegalStateException("未预期异常", cause);
} catch (InterruptedException e) {
    // 当前线程被中断，务必恢复中断状态
    Thread.currentThread().interrupt();
    throw new RuntimeException("等待结果时被中断", e);
}

invokeAll / invokeAny：批量任务的两种策略

ExecutorService 提供了两个批量提交方法，分别对应两种常见的聚合需求：

方法语义典型场景
invokeAll(tasks)等所有任务完成，返回 Future 列表需要全部结果才能继续（如汇总报表）
invokeAll(tasks, timeout, unit)超时后未完成的 Future 被自动取消有 SLA 限制的批量调用
invokeAny(tasks)第一个成功完成即返回，取消其余任务最快响应即可（如多地区冗余查询）
invokeAny 示例 — 冗余查询取最快结果
List<Callable<String>> mirrors = List.of(
    () -> fetchFrom("us-east"),
    () -> fetchFrom("eu-west"),
    () -> fetchFrom("ap-east")
);

// 哪个机房最先响应就用哪个，其余请求自动取消
String fastest = pool.invokeAny(mirrors, 2, TimeUnit.SECONDS);

面试高频考点

  • Runnable vs Callable 区别？ 返回值 + 受检异常两点，延伸到 Future 如何传播异常。
  • FutureTask 是怎么保证线程安全的？ 状态机 + volatile + CAS，outcome 写入 happens-before get() 读取。
  • get() 抛哪些异常？分别代表什么？ InterruptedException（等待时被中断）、ExecutionException（任务抛异常）、TimeoutException（超时）、CancellationException（任务被取消）。
  • cancel(true) 一定能停止任务吗？ 不能，只是发中断信号，任务必须主动响应中断。
  • Future 的局限性是什么？ 无法链式组合、无法在完成时触发回调（这正是 CompletableFuture 诞生的原因）。
Future 与 CompletableFuture 的分水岭
Future 是拉模型（你主动 get() 等结果），阻塞调用线程；CompletableFuture 是推模型（任务完成后主动触发回调），支持流式链式编排，彻底消除阻塞等待。下一节将详细展开。

与其他知识点的联系

  • 与 Executor 框架的分层关系：Callable/Future 是"任务与结果"这一层抽象，ExecutorService.submit() 负责把 Callable 包装成 FutureTask 并交给底层线程池调度——理解本文后回看上一节 Executor 框架，会发现 submit() 只是"Runnable 提交"的语义升级版，二者共享同一套线程池调度逻辑，只是返回值处理方式不同。
  • FutureTask 状态机与 volatile/CAS：本文状态机图里"状态一旦离开 NEW 就不可逆"，其安全性依赖 volatile 字段 + CAS 操作保证的 happens-before 语义，这与 CountDownLatch 内部 AQS 用 CAS 修改 state 是同一套底层机制的不同应用——都是"用 CAS 实现无锁的一次性状态翻转"这一模式的具体案例。
  • Memoizer 与单例懒加载的血缘关系：本文用 FutureTask + ConcurrentHashMap.putIfAbsent 实现的 Memoizer，本质和"双重检查锁定实现单例"要解决的是同一类问题——"保证昂贵操作只执行一次，且并发场景下后来者能安全拿到结果"，只是 Memoizer 是通用的按 key 缓存，DCL 单例是固定单个实例的特化版本。
  • 与 CompletableFuture 的能力落差：本文陷阱 2（顺序 get 导致串行化）和陷阱 3（cancel 不能真正停止任务）恰好是 CompletableFuture 要解决的两大痛点——前者被 allOf/thenCombine 的并行编排取代，后者本质上没有被根本解决（CompletableFuture 的 cancel 同样只是标记状态），但链式回调模型从设计上减少了"手动管理多个 Future"的复杂度，这是下一节要讲的核心动机。
  • ExecutionException 包装与异常传播的通用模式：本文强调"子线程异常被包裹进 ExecutionException，需 getCause() 还原"，这与 CompletableFuture 用 CompletionException 包装异常、反射调用用 InvocationTargetException 包装异常是同一种"跨越执行边界（线程/反射调用）时必须显式包装真实异常"的设计范式，面试中常作为对比考点。
  • 面试高频对比："Future.cancel(true) 为什么不能保证任务立即停止"经常被追问底层原理——本质是 Java 的协作式中断模型（cooperative interruption）：JVM 不会强行杀死线程，只是设置中断标志或对阻塞方法抛异常，任务代码必须主动检查响应，这与后面 07-interrupt中断.html 要讲的中断机制完全一致。
