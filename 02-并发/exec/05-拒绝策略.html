Part III 结构化应用 · 线程池

05. 四种拒绝策略

当线程池的洪流遇上堤坝——拒绝不是失败，而是系统自保的最后一道闸门。

为什么需要拒绝策略？

回顾上一节 ThreadPoolExecutor 的七大参数：核心线程数、最大线程数、工作队列构成了三道缓冲。 当任务提交速率持续超过处理速率，这三道缓冲全部打满时，线程池进入饱和状态（saturated）， 此时再提交新任务该怎么办？

提交新任务 │ ├─ 核心线程未满 ──────────────────→ 直接创建核心线程执行 │ ├─ 核心线程已满，队列未满 ─────────→ 任务入队等待 │ ├─ 队列已满，线程数 < maximumPoolSize ─→ 创建非核心线程执行 │ └─ 队列已满，线程数 = maximumPoolSize ─→ 触发拒绝策略

设计者在这里留了一个策略接口而不是写死行为，原因在于： 不同业务场景对"拒绝"的容忍度完全不同—— 支付服务宁可抛异常让调用方感知，日志系统宁可丢弃， 而批量导入任务则希望由提交线程自己跑完以产生背压。 策略模式让这四种截然不同的处置方式可以即插即用。

RejectedExecutionHandler 接口

整个拒绝机制只有一个接口，一个方法：

javaRejectedExecutionHandler.java（JDK 源码）
public interface RejectedExecutionHandler {
    // r  = 被拒绝的任务
    // e  = 触发拒绝的线程池实例
    void rejectedExecution(Runnable r, ThreadPoolExecutor e);
}

JDK 在 ThreadPoolExecutor 内部以静态内部类的形式提供了四种标准实现， 全部位于 java.util.concurrent 包。

四种内置策略详解

① AbortPolicy — 中止（默认）

直接抛出 RejectedExecutionException（非受检异常），让调用方立即感知压力。 这是 Fail-Fast 思想：系统宁可暴露问题，也不要静默地吞掉任务。

javaAbortPolicy 源码（精简）
public static class AbortPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        throw new RejectedExecutionException(
            "Task " + r.toString() + " rejected from " + e.toString()
        );
    }
}
陷阱：AbortPolicy 是默认策略，但很多人忘记在调用处捕获 RejectedExecutionException，导致提交线程（往往是业务主线程）直接崩溃。 正确做法是在 executor.submit(task) 外包一层 try-catch。
java正确捕获拒绝异常
// 反例：提交不捕获，主线程会被异常打断
executor.execute(task);  // 危险！

// 正例：显式捕获，记录告警或降级处理
try {
    executor.execute(task);
} catch (RejectedExecutionException ex) {
    log.warn("线程池饱和，任务被拒绝: {}", task);
    metrics.increment("pool.rejected");  // 上报监控
    // 视业务决定：入 MQ 重试 / 返回降级结果 / 直接丢弃
}

② CallerRunsPolicy — 调用者运行

不抛异常，也不丢弃，而是让提交任务的线程自己执行这个任务。 这天然产生了背压（back-pressure）机制： 提交线程被阻塞在任务执行上，无暇继续提交新任务， 给线程池争取到喘息时间，整个系统自动降速而不是崩溃。

javaCallerRunsPolicy 源码（精简）
public static class CallerRunsPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 线程池已关闭则直接丢弃，否则在调用方线程同步执行
        if (!e.isShutdown()) {
            r.run();  // 注意：是 run() 不是 start()，直接在当前线程执行
        }
    }
}
背压原理：假设生产者线程以 10000 QPS 提交任务，线程池只能处理 8000 QPS。 CallerRunsPolicy 让生产者线程每隔一段时间就被迫执行一个任务，提交速率自动从 10000 降到接近 8000， 系统平衡而非崩溃。这是响应式系统设计中"流量整形"的最简形式。
陷阱：如果提交线程是 Netty 的 I/O 线程或 Tomcat 的请求线程， CallerRunsPolicy 会让网络 I/O 被任务长期占用，导致请求超时甚至连接堆积。 此时应选 AbortPolicy 配合调用方降级，而不是 CallerRunsPolicy。

③ DiscardPolicy — 静默丢弃

最"沉默"的策略——直接丢弃被拒绝的任务，不抛异常，不通知任何人。 源码是一个空方法体，什么都不做。

javaDiscardPolicy 源码
public static class DiscardPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 什么都不做，任务就此消失
    }
}

适用于任务本身具有幂等性且允许丢失的场景， 例如心跳上报、实时监控采样点、埋点日志—— 偶尔丢一两个无关紧要，强行处理反而拖垮系统。

高频陷阱：DiscardPolicy 用于重要业务（如订单、支付）是灾难级错误。 任务被静默丢弃后没有任何日志，排查问题时毫无线索。 即便业务允许丢弃，也强烈建议自定义策略，在丢弃前打印 warn 日志并上报监控指标。

④ DiscardOldestPolicy — 丢弃最老任务

从工作队列的队头取出等待最久的那个任务并丢弃， 然后尝试重新提交当前被拒绝的新任务。 逻辑上是"新任务比老任务更重要"的优先级假设。

javaDiscardOldestPolicy 源码（精简）
public static class DiscardOldestPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        if (!e.isShutdown()) {
            e.getQueue().poll();  // 丢弃队头（最老）任务
            e.execute(r);        // 重新提交当前任务
        }
    }
}
重大陷阱：配合 PriorityBlockingQueue 时会出现反直觉行为。 poll() 从队头取出的是优先级最高的任务，恰好与"丢旧留新"的初衷相反—— 你把最重要的任务丢了，却保留了低优先级任务。 因此 DiscardOldestPolicy 不得与 PriorityBlockingQueue 组合使用。

四种策略横向对比

策略 行为 任务是否丢失 是否通知调用方 典型场景
AbortPolicy 抛出 RejectedExecutionException 是 是（异常） 支付、订单等核心链路
CallerRunsPolicy 由提交线程同步执行 否 间接（阻塞提交） 批量处理、允许降速的场景
DiscardPolicy 静默丢弃新任务 是 否 心跳、采样、埋点等可丢场景
DiscardOldestPolicy 丢弃队头最老任务，重提新任务 是（老任务） 否 实时性要求高、新数据优先的场景

自定义拒绝策略：生产实践

内置四种策略往往不够用——生产环境至少应该做到： 拒绝时记录日志 + 上报监控指标 + 根据任务类型差异化处置。

java生产级自定义拒绝策略示例
public class MonitoredAbortPolicy implements RejectedExecutionHandler {

    private final String  poolName;
    private final Counter rejectedCounter;  // Micrometer / Prometheus 计数器

    public MonitoredAbortPolicy(String poolName, MeterRegistry registry) {
        this.poolName = poolName;
        this.rejectedCounter = Counter.builder("threadpool.rejected")
                .tag("pool", poolName)
                .register(registry);
    }

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 1. 关键指标上报，触发告警
        rejectedCounter.increment();

        // 2. 结构化日志，方便排查
        log.warn("[{}] 任务被拒绝 | activeThreads={} | queueSize={} | task={}",
                poolName,
                e.getActiveCount(),
                e.getQueue().size(),
                r
        );

        // 3. 根据任务类型决定处置方式
        if (r instanceof PersistableTask) {
            // 重要任务：持久化到 DB/MQ 稍后重试
            ((PersistableTask) r).persistForRetry();
        } else {
            // 非关键任务：抛异常让调用方感知
            throw new RejectedExecutionException(
                    "Pool [" + poolName + "] saturated");
        }
    }
}

触发时机的一个微妙细节

很多人以为"队列满 + 线程满"才触发拒绝，但有一个常见的误解： 使用 SynchronousQueue（Executors.newCachedThreadPool 的默认队列）时， 队列容量为 0，只要没有空闲线程可接收任务，立刻触发拒绝， 不必等到 maximumPoolSize 线程跑满。

javaSynchronousQueue 触发拒绝的场景
// SynchronousQueue 不存储任务，每次 offer() 必须立即有线程接收
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2, 4,                          // core=2, max=4
    60, TimeUnit.SECONDS,
    new SynchronousQueue<>(),     // 零容量队列
    new ThreadPoolExecutor.AbortPolicy()
);
// 同时提交 5 个任务：前 4 个由线程执行，第 5 个立即被拒绝
// 因为 SynchronousQueue 无法缓存，没有第 5 条线程可接收它

如何选择合适的拒绝策略？

  • 核心业务链路（支付、订单）：AbortPolicy + 调用方捕获异常，配合熔断降级。
  • CPU 密集批处理、允许降速：CallerRunsPolicy，利用背压自动平衡速率。
  • 可丢弃的实时数据（心跳、采样）：自定义策略（在 DiscardPolicy 基础上加日志和监控）。
  • 需要最新数据的实时推送：DiscardOldestPolicy，但必须用 LinkedBlockingQueue，禁止用 PriorityBlockingQueue。
  • 任何生产环境：都应自定义策略，最低要求是拒绝时打印日志 + 上报监控指标。
面试高频考点： "线程池默认拒绝策略是什么？" → AbortPolicy。 "CallerRunsPolicy 为什么能产生背压？" → 提交线程被占用执行任务，提交速率自动降低，系统趋于平衡。 "DiscardOldestPolicy + PriorityBlockingQueue 有什么问题？" → poll() 取出的是优先级最高（最重要）的任务，与预期相反。
