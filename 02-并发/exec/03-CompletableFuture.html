03. CompletableFuture 异步编排

从阻塞式 Future 到响应式流水线——理解 Java 8 最重要的并发工具之一

1. Future 的痛点：阻塞是原罪

上一节我们学了 Future<T>。它确实能拿到异步任务的结果，但有一个根本缺陷：获取结果必须阻塞当前线程。

场景还原：电商首页需要同时查询「商品信息」「库存」「用户推荐」三个服务，最后聚合返回。如果用 Future.get()，三次 get 都会顺序阻塞，实际退化成串行——白白浪费了多线程。
Java — Future 串行阻塞反例
Future<Product> pf   = executor.submit(() -> fetchProduct(id));
Future<Stock>   sf   = executor.submit(() -> fetchStock(id));
Future<Rec>     rf   = executor.submit(() -> fetchRec(userId));

// 问题：三次 get() 顺序阻塞，总耗时 = T1 + T2 + T3，并非 max(T1,T2,T3)
Product p = pf.get();
Stock   s = sf.get();
Rec     r = rf.get();

CompletableFuture 的核心思想是：把"拿到结果后做什么"注册成回调，让线程池驱动，当前线程完全不阻塞。这是 Promise/A+（JavaScript）、Scala Future、Reactor 等所有异步模型的共同设计哲学。

2. 设计哲学：可组合的异步状态机

CompletableFuture<T> 本质上是一个单次写入的共享状态容器，内部维护：

  • 结果槽：初始为 null，完成后写入值或异常，只写一次
  • 等待链：所有通过 thenApply、thenCompose 等方法注册的后续任务，形成一棵树
  • 触发机制：状态从"未完成"→"完成"时，通过 CAS 原子翻转，然后顺序唤醒等待链上的所有节点
CompletableFuture 内部状态机 PENDING │ 某线程调用 complete(v) / completeExceptionally(ex) ▼ COMPLETED ──── CAS 写入结果，唤醒所有 Dependent │ ├── thenApply 回调1 → 新 CF ├── thenAccept 回调2 → 新 CF(Void) └── whenComplete 回调3（无论成功/失败）

这与 FutureTask 最大的区别是：CompletableFuture 支持链式注册 N 个后继，而 FutureTask 只能被动等 get()。这也是它能做"编排"的根本原因。

3. 核心 API 全景

分类方法说明
创建supplyAsync(Supplier)有返回值，异步执行
runAsync(Runnable)无返回值，异步执行
completedFuture(v)直接创建已完成的 CF（常用于测试/默认值）
串行变换thenApply(fn)结果 → 新结果（同步函数，在上游线程执行）
thenApplyAsync(fn)结果 → 新结果（异步，提交到线程池）
thenCompose(fn)结果 → 新 CF（flatMap，避免 CF<CF<T>>）
串行消费thenAccept(consumer)消费结果，返回 CF<Void>
thenRun(runnable)不关心结果，直接跑后续
并行聚合thenCombine(other, fn)等两个 CF 都完成，合并结果
allOf(cfs...)等所有完成（返回 CF<Void>）
anyOf(cfs...)任意一个完成即触发
异常处理exceptionally(fn)捕获异常，提供备用值
handle(fn)无论成功/失败都执行，类似 try-finally
whenComplete(action)副作用（不改变结果），常用于日志

4. 创建 & 串行链式调用

Java — 基础链式：查用户 → 查订单 → 计算总额
CompletableFuture<Double> totalFuture =
    CompletableFuture.supplyAsync(() -> findUser(42), executor)
        // thenCompose：返回值本身是 CF，用它可以"展平"，避免 CF<CF<T>>
        .thenCompose(user ->
            CompletableFuture.supplyAsync(() -> findOrders(user.getId()), executor))
        // thenApply：同步变换，运行在上游完成的那个线程
        .thenApply(orders -> orders.stream()
            .mapToDouble(Order::getAmount)
            .sum());

// 链尾消费（仍然不阻塞主线程）
totalFuture.thenAccept(total ->
    System.out.println("总金额: " + total));
thenApply vs thenCompose：
把 CF 类比 Stream：thenApply 像 map（T → R），thenCompose 像 flatMap（T → Stream<R>）。当下游操作本身返回 CF 时，用 thenApply 会得到 CF<CF<T>>——嵌套地狱；用 thenCompose 则自动展平为 CF<T>。

5. 并行聚合：allOf / thenCombine

这是 CompletableFuture 最能体现价值的场景——把多个独立的异步任务真正并发执行，然后等全部完成再聚合。

Java — allOf 并发查三个服务，总耗时 ≈ max(T1,T2,T3)
CompletableFuture<Product> cfP =
    CompletableFuture.supplyAsync(() -> fetchProduct(id), executor);
CompletableFuture<Stock>   cfS =
    CompletableFuture.supplyAsync(() -> fetchStock(id), executor);
CompletableFuture<Rec>     cfR =
    CompletableFuture.supplyAsync(() -> fetchRec(userId), executor);

// allOf 本身只返回 CF<Void>，需要自己再 join() 各子 CF
CompletableFuture.allOf(cfP, cfS, cfR)
    .thenRun(() -> {
        // 走到这里，三个 CF 均已完成，join() 不会阻塞
        PageVO vo = new PageVO(
            cfP.join(),
            cfS.join(),
            cfR.join());
        render(vo);
    });
Java — thenCombine：等两个 CF 都完成，合并结果
CompletableFuture<Double> price    =
    CompletableFuture.supplyAsync(() -> getPrice(sku), executor);
CompletableFuture<Double> discount =
    CompletableFuture.supplyAsync(() -> getDiscount(userId), executor);

// 两个并发跑，都完成时回调合并函数
CompletableFuture<Double> finalPrice =
    price.thenCombine(discount, (p, d) -> p * (1 - d));

6. 异常处理：exceptionally / handle

异步链中一旦某步抛出异常，异常会沿着 CF 链向下传播，后续所有 thenApply / thenAccept 都会被跳过，直到遇到 exceptionally 或 handle。

Java — 异常传播与恢复
CompletableFuture.supplyAsync(() -> {
    if (id < 0) throw new IllegalArgumentException("id 不合法");
    return fetchUser(id);
})
// 正常路径：user → orders
.thenApply(user -> fetchOrders(user.getId()))
// 异常路径：提供兜底空列表，链路可以继续
.exceptionally(ex -> {
    log.warn("查询失败: {}", ex.getMessage());
    return Collections.emptyList();
})
// handle：无论成功/失败都执行，ex 为 null 则代表成功
.handle((orders, ex) -> {
    if (ex != null) return Result.fail(ex.getMessage());
    return Result.ok(orders);
});
异常包装：CF 链内部抛出的所有异常都会被包装成 CompletionException，调用 join() 时抛出的也是它（而 get() 抛出 ExecutionException）。在 exceptionally 的回调里，参数 ex 是 CompletionException，真正原因在 ex.getCause()。

7. Async 后缀 & 线程池选择

几乎每个操作符都有三种重载：

thenApply(fn) — 在上游任务完成的线程执行回调（可能是 ForkJoinPool 线程，也可能是调用线程） thenApplyAsync(fn) — 提交到默认 ForkJoinPool.commonPool() thenApplyAsync(fn, e) — 提交到指定 Executor（推荐！）
陷阱：默认 commonPool 的危险
supplyAsync(fn)（无 Executor 参数）使用 ForkJoinPool.commonPool()。这个池与所有使用默认参数的 CF、并行 Stream 共享。如果 fn 里有 IO 操作（HTTP、DB）且并发量大，会把 commonPool 的线程耗尽，影响整个 JVM 的并行 Stream 性能。生产代码请始终传入自己的 Executor。
Java — 正例：IO 密集型任务用专属线程池
// IO 密集型：核心线程数 = CPU核 * 2，允许充分等待
ExecutorService ioPool = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors() * 2);

// CPU 密集型：用 ForkJoinPool，线程数 = CPU核
ForkJoinPool cpuPool = new ForkJoinPool(
    Runtime.getRuntime().availableProcessors());

CompletableFuture.supplyAsync(() -> httpGet(url), ioPool)
    // 拿到原始数据后，CPU 密集的解析放到 cpuPool
    .thenApplyAsync(JsonParser::parse, cpuPool)
    .thenAcceptAsync(result -> save(result), ioPool);

8. 并发陷阱汇总

陷阱 1：get() vs join() 的异常类型

Java — 反例 vs 正例
// ❌ 反例：get() 抛 checked Exception，代码冗余
try {
    String r = cf.get();
} catch (InterruptedException | ExecutionException e) {
    throw new RuntimeException(e);
}

// ✅ 正例：join() 抛 unchecked CompletionException，配合 exceptionally 更清晰
String r = cf.exceptionally(ex -> "default").join();

陷阱 2：在回调里抛出受检异常

Lambda 内部如果要抛受检异常，必须手动包装。如果直接 throw new IOException() 会编译报错。推荐统一包装成 CompletionException：

Java
cf.thenApply(s -> {
    try {
        return parseJson(s);  // throws IOException
    } catch (IOException e) {
        // 包装成 unchecked，让 exceptionally 能统一捕获
        throw new CompletionException(e);
    }
});

陷阱 3：completeOnTimeout 与 orTimeout（Java 9+）

Java 9+ — 超时控制
// orTimeout：超时则以 TimeoutException 完成（异常路径）
cf.orTimeout(3, TimeUnit.SECONDS)
  .exceptionally(ex -> "timeout fallback");

// completeOnTimeout：超时则以默认值完成（正常路径，更优雅）
cf.completeOnTimeout("default", 3, TimeUnit.SECONDS);

陷阱 4：泄漏——忘记处理异常导致 CF 永久悬挂

典型案例：某个 supplyAsync 回调里抛了异常，但整条链没有 exceptionally，也没人调 join()，这个 CF 对象就静静地悬浮在内存里，它引用的所有闭包变量也无法 GC。

规范：所有 CF 链的末尾都应挂一个 .exceptionally(ex -> { log.error(...); return null; })，即使只是打日志。

陷阱 5：thenApply 与 thenApplyAsync 的执行线程差异

Java — 执行线程演示
CompletableFuture.supplyAsync(() -> {
    System.out.println("supply: " + Thread.currentThread().getName());
    return 1;
})
// 若 supply 极快完成，这里可能在调用 thenApply 的「主线程」执行！
.thenApply(n -> {
    System.out.println("apply: " + Thread.currentThread().getName());
    return n + 1;
});

// 使用 thenApplyAsync 则保证一定在线程池线程执行，行为可预测

9. 综合示例：电商首页聚合接口

Java — 生产级写法：超时 + 异常兜底 + 专属线程池
@Service
public class HomePageService {

    private final ExecutorService ioPool =
        Executors.newFixedThreadPool(20);

    public HomePageVO assemble(Long userId, Long skuId) {

        CompletableFuture<Product> cfProduct =
            CompletableFuture.supplyAsync(() -> productSvc.get(skuId), ioPool)
                .completeOnTimeout(Product.empty(), 2, TimeUnit.SECONDS)
                .exceptionally(ex -> {
                    log.error("product svc error", ex);
                    return Product.empty();
                });

        CompletableFuture<List<Rec>> cfRec =
            CompletableFuture.supplyAsync(() -> recSvc.list(userId), ioPool)
                .completeOnTimeout(Collections.emptyList(), 2, TimeUnit.SECONDS)
                .exceptionally(ex -> {
                    log.error("rec svc error", ex);
                    return Collections.emptyList();
                });

        // 等两个都完成，合并返回
        return cfProduct.thenCombine(cfRec, (p, r) ->
            new HomePageVO(p, r)
        ).join();  // Controller 层统一等待，业务线程只在此处短暂阻塞
    }
}
实践建议：在 Spring MVC / Spring WebFlux 的 Controller 里，如果已经有 Servlet 线程池，不必一定把最后的 join() 改成非阻塞——把耗时操作并发化本身就是主要收益。真正需要全链路非阻塞的场景，请考虑 Project Reactor（Mono/Flux）或 Kotlin 协程。

10. CompletableFuture vs Future 对比

能力Future / FutureTaskCompletableFuture
获取结果get()，必须阻塞join() 或链式回调，可不阻塞
链式编排不支持thenApply / thenCompose / …
并行聚合手动循环 get()allOf / anyOf / thenCombine
异常处理catch ExecutionExceptionexceptionally / handle，链式统一
超时get(timeout) 阻塞orTimeout / completeOnTimeout（非阻塞）
手动完成不支持complete(v) / completeExceptionally(e)
适用场景简单单任务异步多任务编排、微服务聚合、异步流水线

11. 面试高频考点

  • thenApply vs thenCompose：前者同步变换结果，后者用于下游也是异步操作（避免 CF 嵌套）
  • thenApply vs thenApplyAsync：前者在上游线程执行（行为不确定），后者强制提交到线程池（推荐）
  • allOf 的返回类型为何是 CF<Void>：因为各子 CF 泛型不同，无法统一类型；需要自己在回调里逐个 join() 子 CF
  • 默认线程池的危险：无 Executor 参数时使用 ForkJoinPool.commonPool()，IO 操作会造成全局线程饥饿
  • join() vs get()：join() 抛 unchecked CompletionException，适合链式；get() 抛 checked，适合需要区分中断的场景
  • CompletableFuture 是否取消：cancel() 只设置 CancellationException，不会真正中断正在执行的任务（与 Future.cancel(true) 语义不同）

12. 与其他知识点的联系

  • Stream API 的设计哲学延续：本文第 4 节的 thenApply/thenCompose 与 Stream 的 map/flatMap 是同一套函数式组合思想在"同步集合"与"异步单值"两个维度上的对应实现——都是 Java 8 一起引入的函数式编程范式落地，理解一个能直接迁移到另一个，这也是为什么本文用 map/flatMap 来类比 thenApply/thenCompose。
  • ForkJoinPool 与 Fork/Join 框架的共享底座：本文第 7 节提到的默认线程池 ForkJoinPool.commonPool()，正是 Fork/Join 框架（如 parallelStream()）使用的同一个池——这解释了为什么"CompletableFuture 里塞 IO 操作"会拖慢看似无关的并行 Stream 代码：它们背后抢的是同一批工作线程，这是生产事故里一个真实存在过的"隐藏耦合"陷阱。
  • 与 Future/FutureTask 的继承与超越：本文第 10 节的对比表系统列出了 CompletableFuture 对 Future 痛点的解决方案，但更深层的联系是：CompletableFuture 实现了 Future 接口（可以调用 get()），却在其之上叠加了一整套"推模型"回调机制——是"新旧接口共存、新接口向后兼容并扩展能力"的典型 API 演进案例，类似 List 接口新增 default 方法而不破坏已有实现。
  • 响应式编程的思想源头：本文①节提到的 Promise/A+、Scala Future 与 CompletableFuture 同源，而这套"注册回调、不阻塞、异常沿链传播"的模式正是 Project Reactor（Mono/Flux）、RxJava 等响应式框架的设计起点——CompletableFuture 可以看作"单个异步值"的响应式编程，Flux/Mono 则把它扩展到了"异步流"，这是从并发工具走向响应式架构的必经台阶。
  • 异常处理与 CyclicBarrier 的 broken 机制呼应：本文 exceptionally/handle 的链式异常处理，与 CyclicBarrier 的 BrokenBarrierException 传染机制、以及 ThreadPoolExecutor 拒绝策略的设计出发点一致——都体现了"并发协作中局部失败必须被显式感知和处理，而不能被静默丢弃"这一贯穿全书的设计原则。
  • 面试高频对比："CompletableFuture 和 Future 的本质区别是什么"最容易被追问到"为什么需要链式编排"——本质原因是分布式/微服务场景下一次请求常常依赖多个下游服务，Future.get() 的阻塞式等待会让"并发调用"退化为"串行调用"，而 CompletableFuture 的回调驱动模型让 CPU 在等待 IO 时可以调度其他任务，这也是 Reactor/Netty 等高性能网络框架普遍采用异步回调模型的根本原因。
