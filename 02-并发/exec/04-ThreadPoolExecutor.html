Part III 结构化应用 · 线程池

04. ThreadPoolExecutor 七大参数

彻底理解线程池的内部调度模型，掌握每个参数背后的设计动机，从此不再"黑盒"调参。

为什么需要线程池？

每次 new Thread(task).start() 都需要在操作系统层面分配内核栈、用户栈（默认 512KB~1MB）、 并做上下文注册——在高并发场景下，这笔开销会迅速耗尽内存，还带来大量上下文切换。 线程池解决三个核心问题：

  • 复用：线程执行完任务后不销毁，等待新任务，省去反复创建/销毁的 OS 开销
  • 限流：通过最大线程数约束并发上限，防止系统被压垮
  • 缓冲：任务队列作为"削峰填谷"的蓄水池，平滑突发流量

七大参数一览

ThreadPoolExecutor 最完整的构造函数签名如下：

JavaThreadPoolExecutor.java（JDK 源码）
public ThreadPoolExecutor(
    int             corePoolSize,          // ① 核心线程数
    int             maximumPoolSize,       // ② 最大线程数
    long            keepAliveTime,         // ③ 空闲线程存活时间
    TimeUnit        unit,                  // ④ 时间单位
    BlockingQueue<Runnable> workQueue,   // ⑤ 任务队列
    ThreadFactory   threadFactory,         // ⑥ 线程工厂
    RejectedExecutionHandler handler       // ⑦ 拒绝策略
)

Executors.newFixedThreadPool、newCachedThreadPool 等工厂方法都是 对这个构造函数的封装，只是给了不同的默认值——理解七大参数，就能彻底掌控所有变体。

调度模型：三级缓冲

任务提交到线程池时，JDK 按固定优先级依次判断：

提交任务 execute(task)
│
├─ 当前线程数 < corePoolSize ──→ 直接创建核心线程执行
│
├─ 队列未满 ──────────────────────→ 入队 workQueue，等待核心线程消费
│
├─ 当前线程数 < maximumPoolSize ─→ 创建非核心线程执行（救急线程）
│
└─ 以上均不满足 ──────────────────→ 触发 RejectedExecutionHandler
关键洞察：队列优先于扩容。只有队列满了，才会尝试创建救急线程。 这与很多人直觉相反——"先排队，再扩线程"是 JDK 的刻意设计，目的是优先复用已有线程， 减少不必要的线程创建开销。

① corePoolSize — 核心线程数

线程池"常驻"的线程数量。即使这些线程处于空闲状态，默认也不会被回收。 它们随时待命，响应新任务的延迟极低。

类比餐厅：核心线程就是编制内的正式员工，没客人时也照样领工资（占内存）。

经验值：
• CPU 密集型任务：corePoolSize = CPU 核数 + 1（+1 防止偶发 GC 停顿让 CPU 空转）
• IO 密集型任务：corePoolSize = CPU 核数 × (1 + 等待时间/计算时间)， 实践中常设 2 × CPU 核数 作为起点

可通过 allowCoreThreadTimeOut(true) 让核心线程也参与超时回收， 适合任务量极不均匀的场景（如定时批处理）。

② maximumPoolSize — 最大线程数

线程池能同时存在的线程总上限，包含核心线程。 当队列已满、还有新任务进来时，线程池才会在 corePoolSize 之上继续创建线程， 直到达到 maximumPoolSize。

多出来的线程（maximumPoolSize - corePoolSize 部分）称为救急线程， 它们在空闲 keepAliveTime 后会被自动销毁。

陷阱：使用无界队列（如默认的 LinkedBlockingQueue）时， 队列永远不会"满"，maximumPoolSize 形同虚设——救急线程永远不会被创建。 这正是 Executors.newFixedThreadPool 的隐患所在。

③④ keepAliveTime + unit — 救急线程存活时间

救急线程执行完任务后，若在 keepAliveTime 时间内没有新任务可执行， 则被线程池回收销毁。单位由 TimeUnit 参数指定（秒、毫秒、纳秒均可）。

设置过短：流量波动时频繁创建/销毁救急线程，白白浪费 OS 资源。
设置过长：流量低谷时救急线程长时间空占内存。

⑤ workQueue — 任务队列

JDK 提供了三种常用队列，各有取舍：

队列类型 容量 特点 适用场景
LinkedBlockingQueue 默认无界（Integer.MAX_VALUE） 永不拒绝，但可能 OOM 任务量可预期、不会暴增
ArrayBlockingQueue 固定有界 队列满时触发扩线程/拒绝 需要严格限流的生产系统
SynchronousQueue 容量为 0（直接交付） 每个任务必须立刻被线程接手，否则触发扩线程 newCachedThreadPool，短时高并发
生产高频踩坑：Executors.newFixedThreadPool(n) 内部使用 new LinkedBlockingQueue()（无界），任务堆积时内存会被耗尽，引发 OOM。 阿里 Java 开发手册明确禁止直接使用 Executors 工厂方法创建线程池， 原因正在于此。

⑥ threadFactory — 线程工厂

负责创建新线程。默认实现 Executors.defaultThreadFactory() 创建的线程 名字形如 pool-1-thread-1，在日志和监控里几乎无法定位来源。

生产环境必须自定义 ThreadFactory，至少要：

  • 给线程命名（如 order-pool-），方便排查死锁、CPU 打满等问题
  • 设置 UncaughtExceptionHandler，防止异常被线程池默默吞掉
  • 按需设置 daemon 属性（守护线程会随 JVM 退出而终止）
Java自定义 ThreadFactory 示例
ThreadFactory factory = new ThreadFactory() {
    private final AtomicInteger counter = new AtomicInteger(1);

    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, "order-pool-" + counter.getAndIncrement());
        // 非守护线程：JVM 不会强制中断正在处理的订单任务
        t.setDaemon(false);
        // 统一捕获未处理异常，接入监控告警
        t.setUncaughtExceptionHandler((thread, ex) ->
            log.error("[{}] 未捕获异常", thread.getName(), ex)
        );
        return t;
    }
};

⑦ handler — 拒绝策略

当队列已满且线程数已达 maximumPoolSize，继续提交的任务触发拒绝策略。 JDK 内置四种，详见下一篇。这里先了解调用时机：

队列已满 AND 线程数 == maximumPoolSize
│
└─ 调用 handler.rejectedExecution(task, executor)

正例：生产级线程池配置

综合七大参数，写出一个可用于生产的订单处理线程池：

JavaOrderExecutorConfig.java
int cpuCores = Runtime.getRuntime().availableProcessors();

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    cpuCores,                          // ① corePoolSize：CPU 核数，轻量 IO 可乘以 2
    cpuCores * 2,                      // ② maximumPoolSize：流量洪峰时最多扩到 2 倍核数
    60, TimeUnit.SECONDS,             // ③④ keepAliveTime：救急线程空闲 60s 后销毁
    new ArrayBlockingQueue<>(500),    // ⑤ 有界队列：队满才触发扩线程，防止 OOM
    factory,                           // ⑥ 自定义 ThreadFactory（见上方代码）
    new ThreadPoolExecutor.CallerRunsPolicy() // ⑦ 调用者自己跑：天然背压，不丢任务
);

// 优雅关闭：不再接收新任务，等待已提交任务跑完
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    executor.shutdown();
    try {
        if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
            executor.shutdownNow(); // 超时强制中断
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}));

反例：Executors 工厂方法的隐患

Java — 反例（禁止在生产使用）
// 反例 1：newFixedThreadPool —— 无界队列，任务堆积 → OOM
ExecutorService bad1 = Executors.newFixedThreadPool(10);
// 内部等价于：new ThreadPoolExecutor(10, 10, 0, MILLISECONDS,
//   new LinkedBlockingQueue())  ← 无界！

// 反例 2：newCachedThreadPool —— 无上限线程数，高并发 → 线程爆炸 → OOM
ExecutorService bad2 = Executors.newCachedThreadPool();
// 内部等价于：new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60, SECONDS,
//   new SynchronousQueue())  ← 最大线程数无限！

并发陷阱与面试高频考点

陷阱一：任务里抛异常被线程池吞掉

execute(task) 提交的任务抛出未检查异常时，线程池会静默回收该线程并创建新线程， 没有任何日志输出。必须在 ThreadFactory 里设置 UncaughtExceptionHandler， 或改用 submit(task)（异常被封装进 Future，调用 get() 时才抛出）。

陷阱二：线程池共享导致任务互相阻塞（线程饥饿死锁）

父任务和子任务共用同一个线程池，且父任务调用 future.get() 等待子任务完成。 若线程都被父任务占满，子任务无线程可用，形成死锁。 解法：子任务使用独立线程池，或改用 CompletableFuture 异步编排。

陷阱三：shutdown 后 submit 新任务

调用 shutdown() 后继续提交任务会直接触发拒绝策略（默认抛 RejectedExecutionException），常见于应用关闭流程写得不严谨时。

面试高频：核心线程数为 0 时的行为

当 corePoolSize = 0（如 newCachedThreadPool）， 提交任务时线程池会先尝试入队（SynchronousQueue.offer() 立即返回 false）， 失败后直接创建救急线程执行。因此即使没有核心线程，任务也能立即被执行而不是被放入队列等待。

七大参数速查表

# 参数名 作用 常见问题
① corePoolSize 常驻线程数，默认不回收 IO 密集型设置过小，CPU 打不满
② maximumPoolSize 线程总上限（含核心线程） 无界队列时此值无效
③④ keepAliveTime / unit 救急线程空闲超时时间 过短导致频繁创建/销毁
⑤ workQueue 任务缓冲队列 无界队列导致 OOM
⑥ threadFactory 线程创建策略（命名、异常处理） 默认命名无法定位问题
⑦ handler 队列满且线程满时的兜底策略 默认抛异常，任务会丢失
