06. 为什么不建议用 Executors 快捷工厂

从 OOM 到饥饿——拆解三大快捷方法背后埋下的并发定时炸弹，再给出生产级正例。

为什么会有 Executors？

Executors 是 JDK 1.5 随 java.util.concurrent 一起引入的工厂类，目的是让开发者 一行代码就能拿到一个线程池，降低入门门槛。它提供三个最常用的快捷方法：

  • Executors.newFixedThreadPool(n) — 固定线程数
  • Executors.newCachedThreadPool() — 弹性缓存线程
  • Executors.newSingleThreadExecutor() — 单线程串行

表面上很方便，但阿里 Java 开发手册明确禁止在生产代码中直接使用它们， 原因全在这三个方法底层参数的默认值里。

快捷工厂的真实参数

所有快捷方法本质上都是 ThreadPoolExecutor 的包装，只是传入了不同的默认参数：

JDK 源码节选（简化）
// ① newFixedThreadPool —— 无界队列！
public static ExecutorService newFixedThreadPool(int nThreads) {
    return new ThreadPoolExecutor(
        nThreads, nThreads,          // corePoolSize = maximumPoolSize
        0L, TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<>()  // ← 无界队列，容量 = Integer.MAX_VALUE
    );
}

// ② newCachedThreadPool —— 无限线程！
public static ExecutorService newCachedThreadPool() {
    return new ThreadPoolExecutor(
        0, Integer.MAX_VALUE,          // ← 最大线程数 = 21 亿
        60L, TimeUnit.SECONDS,
        new SynchronousQueue<>()     // 无缓冲，任务直接交给新线程
    );
}

// ③ newSingleThreadExecutor —— 同样无界队列
public static ExecutorService newSingleThreadExecutor() {
    return new FinalizableDelegatedExecutorService(
        new ThreadPoolExecutor(
            1, 1,
            0L, TimeUnit.MILLISECONDS,
            new LinkedBlockingQueue<>()  // ← 无界队列
        )
    );
}
关键结论：快捷工厂要么用无界队列（Fixed / Single），要么允许无限创建线程（Cached）。 两种极端在高并发下都是资源炸弹。

三大并发陷阱逐一拆解

陷阱一：newFixedThreadPool / newSingleThreadExecutor — 无界队列导致 OOM

LinkedBlockingQueue<>() 无参构造的容量是 Integer.MAX_VALUE（约 21 亿）。 当生产速度 > 消费速度时，任务会无限堆积在队列里， 最终吃光堆内存，抛出 OutOfMemoryError: Java heap space。

更隐蔽的是：由于队列永远不满，maximumPoolSize 设置的值永远不会被触发， 拒绝策略也永远不会执行，系统在 OOM 之前没有任何预警。

反例：无界队列打穿内存
// 反例：表面上只有 4 个线程，实则任务无限堆积
ExecutorService pool = Executors.newFixedThreadPool(4);

for (int i = 0; i < 1_000_000; i++) {
    pool.submit(() -> {
        // 每个任务持有一个 1MB 的字节数组模拟业务数据
        byte[] data = new byte[1024 * 1024];
        Thread.sleep(1000); // 模拟耗时 IO
    });
}
// → 队列堆积 ~100 万个 Runnable + data，迅速 OOM

陷阱二：newCachedThreadPool — 无限线程导致 OOM / 上下文切换风暴

SynchronousQueue 没有任何缓冲：提交一个任务， 若没有空闲线程立刻接手，就会立即创建新线程。 高并发下线程数会爆炸式增长——每个线程默认栈 512KB~1MB， 几万个线程直接撑爆堆外内存，或引发严重的 CPU 上下文切换。

反例：请求洪峰秒杀线程上限
// 反例：秒杀活动瞬间涌入 5 万请求
ExecutorService pool = Executors.newCachedThreadPool();

for (int i = 0; i < 50_000; i++) {
    pool.submit(() -> {
        processOrder(); // 每次操作耗时 200ms
    });
}
// → 瞬间创建 5 万线程，每线程 512KB 栈 = 约 25GB 内存
// → OutOfMemoryError: unable to create new native thread

陷阱三：newSingleThreadExecutor — 包装后无法强制转型，隐藏拒绝策略

newSingleThreadExecutor() 返回的是 FinalizableDelegatedExecutorService，它对外隐藏了 ThreadPoolExecutor 的类型——你无法通过强制转型来修改核心参数， 也无法在运行时收紧队列大小，陷入"买了个黑盒子"的困境。 同时它继承了无界队列的 OOM 风险。

三大工厂方法风险对比

工厂方法 核心问题 触发条件 后果
newFixedThreadPool 无界队列 LinkedBlockingQueue() 生产 > 消费 堆内存 OOM，拒绝策略失效
newCachedThreadPool 最大线程数 = Integer.MAX_VALUE 请求洪峰 线程数暴增，OOM / CPU 风暴
newSingleThreadExecutor 无界队列 + 黑盒包装 任务积压 OOM，且无法运行时修参

并发根因：资源耗尽与背压缺失

这三个陷阱本质上都是同一个并发问题：系统没有背压（Back-pressure）机制。

健壮的并发系统必须在任意一层设置明确的上界：

请求 → 队列（有界）→ 线程（有界）→ 处理
        ↑ 满了就触发拒绝策略  ↑ 超了就不再创建

快捷工厂的问题：
  Fixed/Single → 队列无界，背压缺失
  Cached       → 线程无界，背压缺失

线程池的四个参数 corePoolSize、maximumPoolSize、 workQueue、RejectedExecutionHandler 共同构成背压链条，任何一环失控都会造成资源耗尽。 快捷工厂让其中关键的"队列容量"或"线程上限"变为无界，等同于拆掉了安全阀。

正例：手动构造 ThreadPoolExecutor

生产代码应该显式地写出每一个参数，让意图一目了然， 同时配合有界队列和合理的拒绝策略构成完整的背压链。

正例：显式参数 + 有界队列 + 拒绝策略 + 命名线程
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class ThreadPoolFactory {

    public static ThreadPoolExecutor buildOrderPool() {

        // 1. 自定义线程工厂——线程命名，便于排查
        AtomicInteger counter = new AtomicInteger(1);
        ThreadFactory factory = r -> {
            Thread t = new Thread(r);
            t.setName("order-pool-" + counter.getAndIncrement());
            t.setDaemon(false);   // 业务线程不要设为守护线程
            return t;
        };

        return new ThreadPoolExecutor(
            8,                              // corePoolSize：CPU 密集取 N+1，IO 密集可翻倍
            16,                             // maximumPoolSize：弹性上限，不能无限
            60L, TimeUnit.SECONDS,        // keepAliveTime：超额线程空闲多久后回收
            new LinkedBlockingQueue<>(1024), // 有界队列！超出才触发扩线程/拒绝策略
            factory,                        // 命名线程工厂
            new ThreadPoolExecutor.CallerRunsPolicy() // 调用者降速，天然背压
        );
    }
}
CallerRunsPolicy 是最温柔的背压：当线程池和队列都满时， 任务会在提交者线程里直接执行，从而让生产者自动减速， 不丢任务、不抛异常，适合大多数业务场景。其他策略见上一篇。

在 Spring Boot 中的正确姿势

Spring 提供了 ThreadPoolTaskExecutor，它是对 ThreadPoolExecutor 的薄封装， 支持 Bean 化配置，并提供优雅关闭（setWaitForTasksToCompleteOnShutdown）。

Spring Boot 配置示例
@Configuration
public class AsyncConfig {

    @Bean("orderExecutor")
    public ThreadPoolTaskExecutor orderExecutor() {
        ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
        exec.setCorePoolSize(8);
        exec.setMaxPoolSize(16);
        exec.setQueueCapacity(1024);       // 有界！
        exec.setThreadNamePrefix("order-");
        exec.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy());
        exec.setWaitForTasksToCompleteOnShutdown(true);  // 优雅关闭
        exec.setAwaitTerminationSeconds(30);
        exec.initialize();
        return exec;
    }
}

参数选型速查

参数 CPU 密集型任务 IO 密集型任务 混合型任务
corePoolSize CPU 核数 + 1 CPU 核数 × 2 压测后调整
maximumPoolSize = corePoolSize 2 × corePoolSize 压测后调整
workQueue ArrayBlockingQueue(n) LinkedBlockingQueue(n) 有界即可
RejectedPolicy AbortPolicy + 告警 CallerRunsPolicy 视业务定
面试高频陷阱：面试官最爱问"maximumPoolSize 什么时候生效？" 答案是：仅当有界队列满了之后，线程池才会尝试扩充到 maximumPoolSize。 如果用的是无界队列，maximumPoolSize 永远不会被触发——这正是 Executors 快捷工厂最大的误导性。

总结

  • newFixedThreadPool / newSingleThreadExecutor：无界队列，任务积压会 OOM。
  • newCachedThreadPool：无限线程，请求洪峰会打爆内存或 CPU。
  • 快捷工厂的本质缺陷是背压链条的关键环节被设为无界，系统没有自保能力。
  • 生产代码应显式使用 ThreadPoolExecutor 或 ThreadPoolTaskExecutor， 明确每一个参数，搭配有界队列与合适的拒绝策略。
  • 线程命名是可观测性的基础，出了问题能在 thread dump 里一眼找到归属。
