Part III 结构化应用 · 取消与关闭

08. 优雅关闭线程池（shutdown vs shutdownNow）

理解线程池的生命周期状态机，掌握两种关闭语义的本质差异，写出在 JVM 退出时不丢任务、不泄漏线程的生产级代码。

1. 为什么关闭线程池是个难题

线程池内部维护着工作线程 + 阻塞队列两个资源。 关闭时至少要回答三个问题：

  • 队列里还没来得及执行的任务，要不要继续跑？
  • 正在执行的任务，要不要打断？
  • 调用方如何知道"已经彻底关干净了"？

这三个问题对应了两种截然不同的语义：shutdown() 和 shutdownNow()。 它们不是强弱之分，而是适用场景不同的两种策略。

2. 线程池的五态状态机

ThreadPoolExecutor 用一个 AtomicInteger ctl 将运行状态（高 3 位）和工作线程数（低 29 位） 打包进同一个整型——节省一次 CAS。 理解状态机是理解关闭语义的前提。

RUNNING → shutdown() → SHUTDOWN (不再接新任务，队列继续消费)
RUNNING → shutdownNow() → STOP (不接新任务，清空队列，中断工作线程)
SHUTDOWN → 队列为空 → TIDYING (所有任务执行完毕，工作线程数归零)
STOP → 工作线程归零 → TIDYING
TIDYING → terminated() → TERMINATED (钩子执行完，线程池彻底结束)
关键设计：状态只能单向前进（数值递增），永远不会回退。 这保证了多线程下的可见性和 CAS 的正确性。

3. shutdown() — 温柔的等待

语义

  • 拒绝新提交（调用 execute() 会触发拒绝策略）
  • 队列中的任务继续执行到完成
  • 正在跑的任务不中断
  • 返回后不代表关闭完成，需配合 awaitTermination()

源码关键路径（JDK 17）

ThreadPoolExecutor.java（简化）
public void shutdown() {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        // ① CAS 将状态从 RUNNING → SHUTDOWN
        advanceRunState(SHUTDOWN);
        // ② 只中断"空闲"的工作线程（正在 take() 阻塞的那些）
        interruptIdleWorkers();
        // ③ 给子类留的钩子（ScheduledThreadPoolExecutor 用来清理延迟队列）
        onShutdown();
    } finally {
        mainLock.unlock();
    }
    tryTerminate();  // 尝试推进到 TERMINATED
}

注意 interruptIdleWorkers()：它只中断当前在 getTask() 里阻塞等待的线程，不中断正在执行任务的线程。 阻塞线程被唤醒后，发现状态是 SHUTDOWN 且队列为空，就自然退出。

4. shutdownNow() — 强硬的驱逐

语义

  • 状态推进到 STOP
  • 清空并返回队列中尚未开始执行的任务列表
  • 对所有工作线程调用 interrupt()
  • 正在执行的任务是否真的停下来，取决于任务自身是否响应中断

源码关键路径（JDK 17）

ThreadPoolExecutor.java（简化）
public List<Runnable> shutdownNow() {
    List<Runnable> tasks;
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        // ① 状态推进到 STOP
        advanceRunState(STOP);
        // ② 中断所有工作线程（包括正在跑任务的）
        interruptWorkers();
        // ③ 把队列里没跑的任务全部取出来，返回给调用方
        tasks = drainQueue();
    } finally {
        mainLock.unlock();
    }
    tryTerminate();
    return tasks;
}
陷阱：shutdownNow() 发出的 interrupt() 只是请求，不是强制杀线程。如果你的任务里写了 while(true) 而不检查中断标志，它会永远跑下去，线程池永远不能 TERMINATED。

5. shutdown vs shutdownNow 核心对比

维度 shutdown() shutdownNow()
目标状态 SHUTDOWN STOP
队列中的任务 继续执行 清空，以列表形式返回
正在执行的任务 不中断，跑完为止 发送 interrupt()，能否停取决于任务
返回值 void List<Runnable>（未执行的任务）
适用场景 批处理结束、应用正常退出 超时强制退出、紧急取消
数据安全 高（任务都跑完） 低（正在执行的任务可能处于中间状态）

6. awaitTermination() — 等待关闭完成的正确方式

shutdown() 和 shutdownNow() 都是异步的，调用后立即返回，线程池后台还在收尾。 必须用 awaitTermination(timeout, unit) 来阻塞等待。

反例：只调用 shutdown，不等待

反例 — 可能在任务未完成时 JVM 已退出
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> {
    // 耗时 5 秒的任务
    Thread.sleep(5000);
});
pool.shutdown();
// main 线程直接退出，但 JVM 里还有非守护线程在跑
// 如果之后还有 System.exit()，任务就被强制中断了

正例：标准两阶段关闭模式

正例 — 《Java 并发编程实战》推荐的两阶段关闭
/**
 * 优雅关闭线程池：先礼后兵。
 * 先给 timeout 秒让任务自行完成，超时后强制中断，
 * 再等 timeout 秒确认清理完毕。
 */
public static void shutdownAndAwaitTermination(
        ExecutorService pool, long timeout, TimeUnit unit) {

    // ① 第一阶段：温柔关闭，等待正在执行的任务跑完
    pool.shutdown();
    try {
        if (!pool.awaitTermination(timeout, unit)) {
            // ② 第二阶段：超时后强制中断，收回未开始的任务
            List<Runnable> dropped = pool.shutdownNow();
            log.warn("线程池强制关闭，丢弃了 {} 个排队任务", dropped.size());

            // ③ 再等一轮，让被中断的任务有时间做清理
            if (!pool.awaitTermination(timeout, unit)) {
                log.error("线程池未能在限定时间内终止！");
            }
        }
    } catch (InterruptedException ex) {
        // 当前线程被中断：补发中断信号，并再次强制关闭
        pool.shutdownNow();
        Thread.currentThread().interrupt();  // 恢复中断状态，不要吞掉
    }
}

// 使用方式：10 秒内完成则正常，否则强制
shutdownAndAwaitTermination(myPool, 10, TimeUnit.SECONDS);

7. 让任务真正响应中断

shutdownNow() 的中断只有在任务主动检查中断标志时才有效。 写任务时需要遵守中断协议。

反例：不响应中断的无限循环

反例 — shutdownNow() 对此任务无效
pool.submit(() -> {
    while (true) {
        // 忽略中断，永远不退出
        doWork();
    }
});

正例：两种响应中断的写法

正例 A — 手动检查 isInterrupted()
pool.submit(() -> {
    // 每次循环检查中断标志；shutdownNow() 设置了标志后，下次循环即可退出
    while (!Thread.currentThread().isInterrupted()) {
        doWork();
    }
    log.info("任务收到中断，清理资源后退出");
});
正例 B — 依赖阻塞调用抛出 InterruptedException
pool.submit(() -> {
    try {
        while (true) {
            // BlockingQueue.take() 在被中断时会抛 InterruptedException
            WorkItem item = queue.take();
            process(item);
        }
    } catch (InterruptedException e) {
        // 正常退出路径；不要再 swallow 这个异常
        Thread.currentThread().interrupt();
    }
});

8. 与 JVM Shutdown Hook 集成

生产应用往往需要在 JVM 退出时自动清理线程池。 Runtime.addShutdownHook() 是标准做法。

Spring Boot 风格的线程池 Bean + 关闭钩子
@Configuration
public class AsyncConfig {

    @Bean(destroyMethod = "")  // 关闭 Spring 默认的 destroy，由我们自己控制
    public ExecutorService taskExecutor() {
        ExecutorService pool = new ThreadPoolExecutor(
            4, 16, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadFactoryBuilder().setNameFormat("task-%d").build());

        // 注册 JVM 关闭钩子：进程收到 SIGTERM 时触发
        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            shutdownAndAwaitTermination(pool, 30, TimeUnit.SECONDS);
        }, "pool-shutdown-hook"));

        return pool;
    }
}
Spring Boot 用户：ThreadPoolTaskExecutor 已内置 setWaitForTasksToCompleteOnShutdown(true) 和 setAwaitTerminationSeconds(30)， 配置这两个属性即可，无需手写钩子。

9. 常见陷阱汇总

陷阱 后果 正确做法
只调用 shutdown()，不调用 awaitTermination() main 线程退出，任务被 JVM 强杀 两阶段关闭模式（见第 6 节）
捕获 InterruptedException 后不恢复中断状态 上层感知不到中断，shutdownNow() 失效 Thread.currentThread().interrupt()
任务里 while(true) 不检查中断 shutdownNow() 后线程池永久悬挂 检查 isInterrupted() 或依赖阻塞 API
忽略 shutdownNow() 的返回值 队列里的任务静默丢失，无从追踪 记录返回的 List<Runnable>，必要时重投
对已关闭的线程池调用 execute() 触发拒绝策略（默认抛异常），业务中断 用 isShutdown() 提前检查，或设合理拒绝策略
Shutdown Hook 里做耗时操作，无超时保护 进程超时被 SIGKILL 杀死，与不关闭一样 超时设置不超过容器/K8s terminationGracePeriodSeconds

10. 状态查询 API 速查

实际开发中经常需要根据线程池状态做分支判断：

状态查询 API
ExecutorService pool = ...;

pool.isShutdown();        // true = 已调用过 shutdown/shutdownNow（含 SHUTDOWN/STOP/TIDYING/TERMINATED）
pool.isTerminated();     // true = 已完全终止（TERMINATED），所有任务+线程都结束了
pool.awaitTermination(
    10, TimeUnit.SECONDS); // 阻塞最多 10 秒，返回 true 表示在超时前已 TERMINATED

// ThreadPoolExecutor 专有（需要向下转型）
((ThreadPoolExecutor) pool).getActiveCount();   // 正在执行任务的线程数
((ThreadPoolExecutor) pool).getQueue().size(); // 队列中等待的任务数
((ThreadPoolExecutor) pool).getCompletedTaskCount(); // 历史已完成任务总数

11. 面试高频考点

  • shutdown 和 shutdownNow 的区别是什么？ — 前者等待队列任务执行完再关，后者清空队列并中断所有工作线程，但中断不保证任务立即停止。
  • 线程池有哪几种状态，如何流转？ — 五态：RUNNING → SHUTDOWN / STOP → TIDYING → TERMINATED，状态单调递增，原子操作保证线程安全。
  • 如何等待线程池彻底关闭？ — awaitTermination(timeout, unit)，返回 true 代表在超时内达到 TERMINATED。
  • shutdownNow 的中断为什么有时没有效果？ — 中断是协作式机制，任务必须主动检查 isInterrupted() 或使用可中断的阻塞调用，否则 interrupt 标志被设置后无任何行为变化。
  • Spring 中如何优雅关闭线程池？ — ThreadPoolTaskExecutor 设置 waitForTasksToCompleteOnShutdown=true 和 awaitTerminationSeconds； 或在 @PreDestroy 中调用两阶段关闭工具方法。

12. 与其他知识点的联系

  • 中断机制 interrupt（上一篇）：本文反复强调"shutdownNow 的中断只是请求，任务必须主动检查才会停下"，这句话不是本文的新知识，而是上一篇 interrupt 协议的直接应用——线程池的优雅关闭本质上就是"如何在框架层大规模、正确地使用中断协议"的一个具体案例，两篇内容需要放在一起理解才完整。
  • JVM 关闭钩子与进程生命周期：Runtime.addShutdownHook() 的触发时机（收到 SIGTERM 或所有非守护线程结束）与 JVM 的退出流程紧密相关——如果线程池里的线程是非守护线程且没有正确关闭，JVM 会一直挂起无法退出，这也是很多"进程退出不了"的生产事故的根本原因，和线程的 daemon 属性设置直接相关。
  • Spring 容器生命周期：本文提到的 ThreadPoolTaskExecutor 的 setWaitForTasksToCompleteOnShutdown，其触发时机挂载在 Spring Bean 的销毁回调（DisposableBean/@PreDestroy）上，理解 Spring 容器关闭时 Bean 销毁的顺序，才能正确判断"业务线程池会不会在其他依赖资源（如数据源）关闭之前先被关掉"。
  • K8s / 容器编排的优雅终止：本文提到的 terminationGracePeriodSeconds 并不是 Java 概念，而是容器编排层给进程发送 SIGTERM 后、发送 SIGKILL 强杀前的宽限期——线程池 awaitTermination 的超时时间必须小于这个宽限期，否则应用还没关完就被容器强杀，业务任务会在中间状态被直接终止，这是云原生环境下部署 Java 服务必须协调好的两层超时配置。
  • ThreadPoolExecutor 的 ctl 状态机与 CAS：本文第 2 节提到用一个 AtomicInteger 同时编码运行状态和线程数，这是 JDK 里典型的"位打包 + CAS"优化手段，与 AbstractQueuedSynchronizer 用一个 int 表示锁状态的思路一致，理解这种手法有助于阅读更多 JUC 源码。
一句话记忆：shutdown() 是"挂牌停止营业，等客人用完餐再关门"； shutdownNow() 是"立刻清场，把还没上桌的菜原样退回厨房，但正在吃饭的客人能不能马上离席，取决于他们自己"。 两者都需要配合 awaitTermination() 才算真正等到门关上。
