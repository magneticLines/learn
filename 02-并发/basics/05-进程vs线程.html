Part I 基础知识 · 线程基础

05. 进程 vs 线程 · 为什么需要并发

从操作系统资源模型出发，理解进程与线程的本质区别，再回答"为什么程序需要并发"这个根本问题。

1. 进程：操作系统的资源容器

进程（Process）是操作系统进行资源分配的最小单位。 当你双击一个 .jar 文件，OS 就为它分配一套独立的"行李箱"：

进程 A（JVM 实例）
├── 虚拟地址空间（堆 · 栈 · 代码段 · 方法区）
├── 文件描述符表（打开的 Socket / 文件）
├── 信号处理表
└── 至少一个线程（main 线程）

两个进程之间的内存天然隔离——进程 A 崩溃不会踩到进程 B 的数据。 这是 OS 通过 MMU（内存管理单元）的页表映射实现的，每个进程看到的都是"假地址"（虚拟地址）， 真正的物理内存由内核统一调度。

为什么进程隔离如此重要？
Chrome 把每个 Tab 设计成独立进程，正是利用这一点：一个 Tab 崩溃不会拖死整个浏览器。 代价是：进程间通信（IPC）需要走内核——管道、Socket、共享内存——成本远高于同进程内的函数调用。

2. 线程：进程内的轻量执行流

线程（Thread）是操作系统进行调度的最小单位。 同一进程内的所有线程共享进程的地址空间（堆、方法区、全局变量）， 但每个线程拥有自己私有的：

  • 程序计数器（PC）：记录当前执行到哪条字节码指令
  • 虚拟机栈（Stack）：局部变量、方法调用帧，线程私有不共享
  • 本地方法栈：执行 native 方法时使用
进程（JVM）
├── 堆（所有线程共享）
├── 方法区 / Metaspace（所有线程共享）
├── Thread-1：PC + Stack（私有）
├── Thread-2：PC + Stack（私有）
└── Thread-N：PC + Stack（私有）

共享堆内存是线程的最大优势，也是并发 bug 的根源—— 两个线程同时读写同一个对象，就可能产生竞争条件（Race Condition）。

3. 进程 vs 线程：一张表看清本质

维度 进程 线程
资源 独立地址空间、文件描述符 共享进程地址空间，私有栈 + PC
创建开销 重（需 fork/clone，复制页表） 轻（Linux 约 10 KB 栈，JVM 默认 512 KB~1 MB）
切换开销 重（TLB 刷新，页表切换） 轻（只需切换寄存器组、栈指针）

典型用途 微服务、多进程模型（Nginx worker） 同一服务内的并发请求处理、异步任务

4. 为什么需要并发？——三个根本动机

动机一：利用多核 CPU，提升吞吐量

现代 CPU 普遍是多核（8 核、16 核甚至更多）。 单线程程序无论代码多优秀，同一时刻只能跑在一个核上，剩余的核全部闲置。 并发让多个线程同时运行在不同核上，理论上 N 核可将 CPU 密集型任务提速 N 倍。

Amdahl 定律的警示： 假设程序 5% 的代码必须串行执行，即使用无限多核，加速比上限也只有 20 倍（1 / 0.05）。 识别并缩短"串行瓶颈"，比无脑堆线程更重要。

动机二：I/O 等待期间不阻塞 CPU，提升响应性

磁盘 I/O 约 10 ms，网络 RTT 约 1–100 ms，而 CPU 执行一条指令约 0.3 ns。 单线程在等待 I/O 时 CPU 完全空转；多线程（或异步 I/O）允许 CPU 在等待期间 切换去处理其他请求，这是 Web 服务器能同时服务数千连接的基础。

动机三：建模现实世界的并发性

真实业务天然是并发的：多个用户同时下单、多个传感器同时上报数据。 用多线程建模这类场景比单线程轮询更直观，代码结构更清晰。

5. 并发的代价——必须面对的三大挑战

并发不是免费的午餐。引入线程会带来三类根本性挑战：

挑战根因典型表现
安全性问题 多线程共享可变状态，操作非原子 计数器丢失更新、单例对象被部分构造
活跃性问题 线程互相等待资源，调度策略不公平 死锁、活锁、饥饿
性能问题 上下文切换、锁竞争、内存可见性同步 加了锁反而比单线程更慢

6. 反例 → 问题 → 正例：计数器的并发陷阱

反例：非线程安全的计数器

count++ 看起来是一行代码，但在字节码层面分解为三步： 读取 → 加一 → 写回。 两个线程可能同时读到相同旧值，导致最终结果比预期少。

UnsafeCounter.java反例 ❌
public class UnsafeCounter {
    private int count = 0;

    // 问题：read-modify-write 三步不是原子操作
    // Thread-1 读到 count=5，Thread-2 也读到 count=5
    // 两者都写回 6，结果丢了一次 +1
    public void increment() {
        count++;
    }

    public int get() {
        return count;
    }
}
实验验证：启动 1000 个线程各调用 increment() 一次，预期结果 1000， 实际运行往往得到 970~999 的随机值——每次运行结果都不同，这就是数据竞争（Data Race）的典型症状。

正例一：synchronized 关键字

SynchronizedCounter.java正例 ✅
public class SynchronizedCounter {
    private int count = 0;

    // synchronized 保证同一时刻只有一个线程执行此方法
    // 内置锁（monitor）附着在 this 对象上
    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}

正例二：AtomicInteger（更高效）

AtomicCounter.java正例 ✅（推荐）
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    // AtomicInteger 内部用 CAS（Compare-And-Swap）指令
    // 无锁实现原子操作，在低竞争场景比 synchronized 性能更好
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();
    }

    public int get() {
        return count.get();
    }
}

7. Java 线程与操作系统线程的关系

Java 线程（java.lang.Thread）在 HotSpot JVM 中是一对一映射到 OS 内核线程的（称为 1:1 模型）。 这意味着线程的创建、销毁、上下文切换都由操作系统内核负责， 调度粒度是操作系统的时间片（Linux 默认约 4 ms）。

Java 21 虚拟线程（Virtual Thread）： JDK 21 引入的 Project Loom 提供了 M:N 模型——大量虚拟线程复用少量 OS 线程。 虚拟线程的上下文切换由 JVM 完成，成本极低（约 200 ns）， 适合 I/O 密集型场景下开启百万级并发。但本节聚焦平台线程（Platform Thread）， 理解其 1:1 模型是进一步学习的基础。

8. 并发 vs 并行：两个容易混淆的概念

并发（Concurrency）是问题域的概念：多个任务在逻辑上同时进行， 不要求物理上同时执行（单核 CPU 通过时间片切换也能实现并发）。

并行（Parallelism）是解决域的概念：多个任务在物理上真正同时执行， 依赖多核 CPU 或多机。并行是并发的一种实现方式。

并发（Concurrency）
└── 强调"结构"：如何将程序分解为可独立推进的任务

并行（Parallelism）
└── 强调"执行"：多个计算单元同时物理执行

单核 CPU：可以并发，不能并行
多核 CPU：可以并发，也可以并行

9. 面试高频考点

  • 进程和线程的区别？——资源分配 vs 调度单位，隔离性 vs 共享内存，创建开销，通信方式。 答完记得补充：线程共享堆 + 方法区，私有 PC + 栈，这是 Java 内存模型的基础。
  • 为什么引入线程而不是直接用多进程？——线程更轻量，同进程内通信零拷贝， 但代价是需要程序员自己保证线程安全。
  • 并发和并行的区别？——并发是结构问题，并行是执行问题；单核也能并发。
  • count++ 是线程安全的吗？——不是。JVM 会编译成 getstatic / iadd / putstatic 三条字节码，非原子操作，需要 synchronized 或 AtomicInteger。
  • Java 线程和操作系统线程是什么关系？——HotSpot 采用 1:1 模型，Java 线程对应一个 OS 内核线程； JDK 21 虚拟线程采用 M:N 模型。
学习路径提示： 理解"进程 vs 线程"是并发编程的地基。后续章节的 synchronized、volatile、 JMM（Java 内存模型）、锁、线程池，都建立在"多线程共享堆内存"这一根本事实之上。 遇到并发 bug 时，第一步永远是问："哪些线程在访问哪些共享状态？"
