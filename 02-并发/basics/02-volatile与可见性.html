02. volatile 与可见性

理解 CPU 缓存、指令重排序与 Java 内存模型，掌握 volatile 的语义边界与正确使用场景。

1. 为什么会有可见性问题？

在单线程世界里，代码顺序执行，写入的值立刻能读回来，天经地义。 但在多核 CPU + JVM 的现实世界里，这个假设全面失效。根源在于三层"加速机制"：

CPU 核心 0 CPU 核心 1 ┌─────────────────┐ ┌─────────────────┐ │ 寄存器 (ns级) │ │ 寄存器 (ns级) │ │ L1 Cache │ │ L1 Cache │ │ L2 Cache │ │ L2 Cache │ └────────┬────────┘ └────────┬────────┘ │ │ └──────────┬─────────────┘ │ L3 Cache (共享) │ 主内存 (RAM) ← 真正的"共享状态"在这里

线程运行在某个 CPU 核心上，写操作只更新了该核心的 L1/L2 缓存， 还没有刷回主内存；另一个核心的线程读同一变量，拿到的是它自己缓存里的旧副本。 这就是可见性问题。

除了 CPU 缓存，还有两个"帮凶"：

  • 编译器重排序：JIT 编译器为了提升性能，可能调整指令顺序（不影响单线程结果，但会破坏多线程语义）
  • CPU 指令重排序：现代 CPU 的乱序执行引擎（Out-of-Order Execution）在不违反数据依赖的前提下自由排序
Java 内存模型（JMM） 就是为了屏蔽不同硬件平台的差异，定义了一套统一的规则：什么情况下线程间的写操作对其他线程"保证可见"。volatile 就是 JMM 提供的最轻量可见性保证手段。

2. 反例：没有 volatile 时的经典陷阱

这是一个看起来"应该能正常停止"的循环，实际上可能永远不会停下来：

StopFlag.java — 反例❌ 不安全
public class StopFlag {

    // 没有 volatile：JIT 可能把 running 缓存在寄存器里
    private static boolean running = true;

    public static void main(String[] args) throws InterruptedException {

        Thread worker = new Thread(() -> {
            // JIT 热身后，可能将循环编译为：
            // if (running) { while (true) {} }  ← 死循环！
            while (running) {
                // 忙等待
            }
            System.out.println("worker 已停止");
        });

        worker.start();
        Thread.sleep(1000);

        running = false;  // 主线程的写操作对 worker 不可见
        System.out.println("已发出停止信号");
    }
}
为什么 JIT 会这样优化？
worker 线程的循环体内没有任何同步点，JIT 推断"这个变量不会在本线程内改变"， 于是把 running 提升到寄存器，只读一次。这是合法的单线程优化，但在多线程下是灾难。

3. volatile 的两大语义

3.1 可见性保证

对一个 volatile 变量的写操作，JVM 保证立刻刷新到主内存； 对它的读操作，JVM 保证从主内存重新加载，而不是读缓存副本。

StopFlag.java — 正例✅ 安全
public class StopFlag {

    // volatile：每次读都从主内存加载，每次写都立即刷回主内存
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) { }  // 每次迭代都从主内存读 running
            System.out.println("worker 已停止");
        });
        worker.start();
        Thread.sleep(1000);
        running = false;  // 写入主内存，worker 下次循环必然看到
    }
}

3.2 禁止指令重排序（有序性保证）

volatile 通过插入内存屏障（Memory Barrier）来禁止特定方向的重排序：

写操作前插入 StoreStore 屏障 → 前面的写不会排到 volatile 写之后 写操作后插入 StoreLoad 屏障 → volatile 写不会排到后面的读之前 ← 最重量级，防止写缓冲区绕过缓存 读操作前插入 LoadLoad 屏障 → volatile 读不会排到后面的读之前 读操作后插入 LoadStore 屏障 → volatile 读不会排到后面的写之前

最经典的受益场景是双重检查锁（DCL）单例：

Singleton.java — DCL 单例Java 5+
public class Singleton {

    // 必须加 volatile，否则 instance = new Singleton() 的三步操作
    // 可能被重排为：分配内存 → 赋值引用 → 初始化对象
    // 另一线程看到非 null 的引用，但对象尚未初始化 → NPE 或状态错乱
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {            // 第一次检查：避免每次都加锁
            synchronized (Singleton.class) {
                if (instance == null) {    // 第二次检查：防止重复创建
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
new Singleton() 发生了什么？
字节码层面分三步：① ALLOCATE 分配内存；② INVOKESPECIAL 调用构造器（初始化字段）；③ ASTORE 将引用赋给 instance。
没有 volatile 时，JIT 可能将 ② 和 ③ 对调。另一个线程进入第一次 null 检查，看到 instance != null，但拿到的是未完成初始化的对象。 volatile 的写屏障保证 ② 一定在 ③ 之前对所有线程可见。

4. 核心陷阱：volatile 不保证原子性

这是最常见的误用。volatile 只保证单次读/写的可见性， 对于复合操作（读-改-写），它什么也保证不了。

Counter.java — 反例❌ 线程不安全
public class Counter {
    // volatile 保证每次读到最新值，但 ++ 是三步操作：
    // 1. 读取 count 到本地  2. +1  3. 写回 count
    // 步骤 1~3 不是原子的，两个线程可能同时读到相同的值，各自加 1，最终只增加了 1
    private volatile int count = 0;

    public void increment() {
        count++;  // ❌ 看起来一行，实际是非原子的读-改-写
    }
}
Counter.java — 正例（三种修复方式）✅ 安全
import java.util.concurrent.atomic.AtomicInteger;

// 方式 1：AtomicInteger（推荐，无锁 CAS）
private final AtomicInteger count = new AtomicInteger(0);
public void increment() { count.incrementAndGet();}

// 方式 2：synchronized（简单粗暴）
private int count = 0;
public synchronized void increment() { count++; }

// 方式 3：LongAdder（高并发累加，Java 8+，吞吐量最高）
import java.util.concurrent.atomic.LongAdder;
private final LongAdder count = new LongAdder();
public void increment() { count.increment();}

5. volatile vs synchronized：如何选择？

维度 volatile synchronized
可见性 ✅ 保证 ✅ 保证（解锁时刷主内存）
有序性 ✅ 禁止重排序 ✅ 保证（临界区内有序）
原子性 ❌ 仅单次读/写原子 ✅ 临界区整体原子
阻塞 非阻塞，无锁竞争 阻塞，有上下文切换开销
适用场景 状态标志位、单写多读、DCL 复合操作、临界区、条件等待
选择原则：如果写操作是独立赋值（不依赖当前值），用 volatile； 如果写操作依赖当前值（如 count++、map.put），用 synchronized 或 Atomic 类。

6. 底层实现：内存屏障与 LOCK 前缀

在 x86 架构上，JVM 通常将 volatile 写编译为带 LOCK 前缀的指令（如 LOCK XCHG 或 LOCK ADD）。 LOCK 前缀有两个效果：

  1. 将当前处理器缓存行写回主内存（刷缓存）
  2. 使其他核心缓存了该地址的缓存行失效（MESI 协议中的 Invalid 状态），强迫它们下次读时重新从主内存加载
主线程写 volatile running = false │ ├─ CPU 0 L1 Cache 更新 ├─ LOCK 指令 → 缓存行写回主内存 └─ 通过 MESI 协议广播 → CPU 1 的缓存行标记为 Invalid │ worker 线程读 running └─ 缓存 miss → 从主内存加载 → 得到 false ✓ 可见

在 ARM/PowerPC 等弱内存模型架构上，JVM 会插入专用的 dmb（Data Memory Barrier）指令， 效果相同但开销更高。这就是为什么 volatile 在 ARM 上的性能损耗通常大于 x86。

7. volatile 的正确使用场景

场景 1：状态标志位（最经典）

状态标志位
private volatile boolean shutdown = false;

public void shutdown() { shutdown = true; }  // 单次赋值，volatile 够用

public void run() {
    while (!shutdown) {
        doWork();
    }
}

场景 2：发布不可变对象（安全发布）

安全发布配置对象
public class ConfigHolder {
    // Config 内部字段全部 final，对象构造完成后不再改变
    // volatile 保证：看到非 null 的 config 引用时，Config 对象也已完全初始化
    private volatile Config config;

    public void reload() {
        config = loadFromFile();  // 原子替换引用
    }

    public Config get() { return config; }
}

8. 面试高频考点

  • volatile 能保证线程安全吗？ — 不能。线程安全需要原子性 + 可见性 + 有序性，volatile 只提供后两者。
  • long/double 为什么建议加 volatile？ — JVM 规范允许 64 位读写分两个 32 位操作完成（非原子写），在 32 位 JVM 上可能读到"半个值"。加 volatile 后强制原子读写。
  • volatile 和 happens-before 的关系？ — volatile 写 happens-before 后续对同一变量的 volatile 读，这是 JMM 的规则之一。下一节详细讲。
  • 为什么 DCL 中第一次 null 检查不加锁却安全？ — volatile 写屏障保证 instance 赋值前构造器已完成，读到非 null 就意味着对象完全可用。第一次检查只是快速路径，不需要互斥。
陷阱：volatile 数组
volatile int[] arr 只保证数组引用的可见性，不保证数组元素的可见性。 修改 arr[0] = 1 对其他线程不可见。需要用 AtomicIntegerArray 或加锁。

小结

volatile 是 Java 并发工具箱里最轻量的可见性武器： 无锁、无上下文切换、开销极低，但职责边界清晰—— 它只解决可见性和有序性，不解决原子性。 正确使用 volatile 的关键是：写操作必须是独立赋值（不依赖当前值）。 下一节我们进入 happens-before 规则，建立更完整的 JMM 直觉。

与其他知识点的联系

  • JVM 内存模型（JMM）：本文开篇讲的 CPU 缓存分层、编译器/CPU 重排序，正是 JMM 要屏蔽的硬件差异；volatile 只是 JMM 对外暴露的一个"轻量级同步原语"，理解 JMM 的抽象模型（工作内存 vs 主内存）才能理解为什么 volatile 写会触发 LOCK 前缀指令、为什么读要从主内存重新加载。
  • happens-before：下一节要讲的核心规则之一就是"volatile 写 happens-before 后续对同一变量的 volatile 读"——本文所有关于可见性的直觉（比如 DCL 单例为什么安全），最终都可以用这一条 happens-before 规则严格表述，而不用每次都从内存屏障、MESI 协议这些底层细节推导。
  • synchronized：本文第 5 节的对比表格已经指出，synchronized 解锁时同样会把工作内存刷回主内存，因此可见性保证不弱于 volatile，区别在于 synchronized 额外提供原子性和互斥，但代价是阻塞和上下文切换；这是"选 volatile 还是 synchronized"这个高频决策的理论依据。
  • CAS/Atomic 类：本文用 count++ 的反例说明 volatile 不保证原子性，AtomicInteger 正是用 CAS 循环解决"读-改-写"复合操作的原子性问题——CAS 底层同样依赖 volatile 语义读取当前值，可以说 Atomic 类是"volatile 的可见性 + CAS 的原子性"组合而成。
  • LongAdder vs AtomicLong：本文正例里提到 LongAdder 作为高并发累加场景的第三种修复方式，它内部的 Cell 数组同样依赖 volatile 保证跨线程可见，但通过分散热点把 CAS 竞争降到最低，是 volatile+CAS 思路在高竞争场景下的进一阶设计。
  • 面试高频考点："volatile 能不能替代锁""long/double 为什么建议加 volatile"是本文已经列出的经典题，往往还会连带追问"实现一个不用 synchronized 的线程安全单例"，答案正是本文的 DCL + volatile 方案，是检验候选人是否真正理解可见性与重排序关系的标准考题。
