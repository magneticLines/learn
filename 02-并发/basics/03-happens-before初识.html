03. happens-before 初识

JMM 对"谁先于谁可见"的正式承诺——读懂它，才能真正理解并发安全的边界。

为什么需要这个概念？

在上一节我们知道，volatile 能解决可见性问题。但"可见性"本身究竟是什么？ Java 内存模型（JMM） 用一个精确的偏序关系来回答这个问题，这个关系叫做 happens-before（先行发生）。

CPU 和编译器为了性能会做两件事：缓存写操作（写入寄存器/L1 而非主内存）和 指令重排序（改变执行顺序）。这两件事在单线程里完全安全，但在多线程里会造成 一个线程的写操作对另一个线程不可见，或者看到的顺序与源码顺序不符。

happens-before 是 JMM 给程序员的一份契约： 只要操作 A happens-before 操作 B，JMM 就保证 A 的写结果对 B 可见，且 A 在执行顺序上不会被重排到 B 之后。 它不是说"A 一定在时间上先执行"，而是说"就结果而言，B 能看见 A 的一切效果"。

一句话定义：如果操作 A happens-before 操作 B，则 A 的执行结果对 B 可见，且 A 的执行顺序排在 B 之前。这是 JMM 对可见性与有序性的统一保证。

生活类比：白板与备忘录

想象两位程序员 Alice 和 Bob 共用一块白板（主内存），但各自还有一个随身便签（CPU 缓存）。

  • Alice 在便签上写了 done = true，还没贴到白板上。
  • Bob 看了眼白板，发现 done 仍是 false，继续等待。
  • 这就是可见性问题：Alice 的写操作没有 happens-before Bob 的读操作。

而 happens-before 就像一个规则："把便签贴到白板（刷新到主内存）这件事， 必须在对方拿起笔看白板（从主内存读取）之前完成"。满足这条规则，Bob 才能保证看到 Alice 写的内容。

JMM 内置的八条 happens-before 规则

JMM 规范（JSR-133）定义了以下规则，不需要任何额外同步即可自动成立：

规则名称 含义
程序顺序规则 同一线程内，按代码顺序，前面的操作 happens-before 后面的操作（仅保证结果，不保证真实执行顺序）
监视器锁规则 unlock happens-before 后续对同一锁的 lock
volatile 变量规则 对 volatile 变量的写 happens-before 后续对它的读
线程启动规则 Thread.start() happens-before 新线程的任何操作
线程终止规则 线程内所有操作 happens-before Thread.join() 返回
线程中断规则 对线程 interrupt() happens-before 被中断线程检测到中断
对象终结规则 对象构造函数执行完毕 happens-before finalize() 开始
传递性规则 若 A hb B，B hb C，则 A hb C
重点记住前四条：程序顺序、监视器锁、volatile、线程启动/终止。它们是日常并发编程的核心保证，传递性则让这些规则可以"串联"使用。

传递性：happens-before 的"乘法器"

单条规则的威力有限，真正强大的是传递性。通过一个中间操作把两条规则串联起来， 可以得出原本无关的两个操作之间也存在 happens-before 关系。

传递性示例.java
class TransitivityDemo {
    private volatile boolean ready = false;
    private int value = 0;  // 普通变量，本身没有可见性保证

    void writer() {
        value = 42;          // ① 写普通变量
        ready = true;        // ② 写 volatile（刷新屏障）
    }

    void reader() {
        if (ready) {         // ③ 读 volatile
            System.out.println(value);  // ④ 读普通变量 —— 一定看到 42
        }
    }
}

传递链推导：

① 写 value=42
  → 程序顺序规则：① hb ②
② 写 volatile ready=true
  → volatile 变量规则：② hb ③
③ 读 volatile ready（值为 true）
  → 程序顺序规则：③ hb ④
④ 读 value

由传递性：① hb ④ ✓ → value 的写对 ④ 可见，结果必为 42
反例陷阱：如果 ready 不是 volatile，传递链在第②步断裂， ② hb ③ 不成立，④ 读到的 value 可能是 0（初始值）。 这是"标志位忘加 volatile"这一经典 Bug 的根因。

反例 → 问题 → 正例

反例：没有 happens-before 保证的标志位

BadFlag.java — 危险！
class BadFlag {
    private boolean stop = false;  // 普通变量，无可见性保证

    void run() {
        new Thread(() -> {
            while (!stop) {  // 可能永远读到缓存中的 false，死循环
                doWork();
            }
        }).start();

        Thread.sleep(1000);
        stop = true;  // 主线程写，但子线程不一定能看到
    }
}

问题根因：主线程对 stop 的写与子线程的读之间，没有任何 happens-before 关系。 JIT 编译器甚至可能把循环优化成 if (!stop) while(true)，让子线程完全看不到写操作。

正例：用 volatile 建立 happens-before 链

GoodFlag.java — 正确
class GoodFlag {
    // volatile 写 hb volatile 读，跨线程可见性得到保证
    private volatile boolean stop = false;

    void run() {
        new Thread(() -> {
            while (!stop) {  // 每次都从主内存读，不会被优化掉
                doWork();
            }
        }).start();

        Thread.sleep(1000);
        stop = true;  // 写 volatile → happens-before 子线程的下一次读
    }
}

synchronized 的 happens-before 保证

监视器锁规则是另一个核心：解锁（unlock）happens-before 后续的加锁（lock）。 这意味着在临界区内的所有写操作，在锁释放时都会刷新到主内存，下一个获得锁的线程能看到全部写操作。

SyncHB.java
class SyncHB {
    private int x = 0;
    private int y = 0;

    synchronized void writer() {
        x = 1;
        y = 2;
    }  // unlock → happens-before → 下一个 lock

    synchronized void reader() {
        // 此处一定能看到 x=1, y=2（监视器锁规则 + 程序顺序规则 + 传递性）
        System.out.println(x + ", " + y);
    }
}

常见误区与面试陷阱

误区 1：happens-before = 时间上先发生

错！ happens-before 是可见性与顺序性的逻辑保证， 与物理时间无关。操作 A 可能在时间上比 B 晚执行（CPU 乱序），但只要 A hb B， 就等价于"B 能看到 A 的结果"。

误区 2：同一线程内不需要考虑 happens-before

程序顺序规则保证了同一线程内代码在结果等价上是顺序的， 但真实执行顺序可能被重排。这在单线程内没有影响（as-if-serial 语义）， 但一旦涉及多线程共享，就需要显式建立跨线程的 happens-before 关系。

误区 3：只要加了 synchronized 就线程安全

synchronized 保证的是同一把锁下的互斥与 happens-before。 如果两个线程用了不同的锁对象（或者一个有锁一个没锁），监视器锁规则根本不成立，可见性仍然无法保证。

WrongLock.java — 常见错误
class WrongLock {
    private int count = 0;
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    void inc() {
        synchronized (lockA) { count++; }
    }

    void get() {
        // 锁 lockB ≠ lockA，unlock(lockA) hb lock(lockB) 不成立！
        synchronized (lockB) { System.out.println(count); }
    }
}

happens-before 在 JDK 中的体现

JDK 并发工具的正确性几乎全部建立在 happens-before 之上，了解这些可以帮助你 "不背记 API，而是理解为什么它是安全的"：

  • CountDownLatch.countDown() happens-before await() 返回（底层 volatile state）
  • Future.get() 的返回 happens-before 你读取结果（任务写入 result 后设 volatile 状态）
  • ConcurrentHashMap 的 put happens-before 后续 get 到相同 key（内部 volatile 写读链）
  • ThreadPoolExecutor 提交任务 happens-before 任务开始执行（队列的 volatile 操作）
结论：happens-before 不是 Java 独有的概念，C++11 内存模型、Go 语言规范都有类似的偏序关系。理解它，意味着你理解了现代多核处理器上并发正确性的本质。

小结

  • happens-before 是 JMM 对可见性与有序性的统一形式化表达，不等于时间上先发生。
  • JMM 内置八条规则，最常用的是：程序顺序、监视器锁、volatile 变量、线程启动/终止。
  • 传递性让这些规则可以"串联"，是推导跨线程可见性的核心工具。
  • 没有 happens-before 关系的两个操作，JMM 不对可见性做任何承诺——这是并发 Bug 的根本来源。
  • 下一节将进入"发布与逸出"：当对象在构造期间就被共享，happens-before 链可能在你意想不到的地方断裂。

与其他知识点的联系

  • JVM 内存模型（JMM）：happens-before 是 JMM 用来定义"可见性"的正式工具，本文八条规则全部来自 JMM 规范（JSR-133）——JMM 是理论母体，happens-before 是它对外暴露的可操作契约，二者不是两个独立概念，而是"模型"与"模型给出的判定规则"的关系。
  • volatile：上一节讲的 volatile 可见性保证，本质就是happens-before 八条规则中的"volatile 变量规则"的具体体现；本文第 4 节的传递性例子（`value` 借 `ready` 的 volatile 语义传递可见性）说明了 volatile 不只保护自己，还能把它前面的普通写操作"带出去"，这是理解安全发布模式的关键。
  • synchronized：本文第 6 节讲的"监视器锁规则"（unlock happens-before 后续 lock）是 synchronized 提供可见性保证的理论依据；配合"误区 3"中的反例（用不同锁对象导致规则不成立），说明 happens-before 分析必须精确到"同一个锁对象"这个粒度，不能只看代码表面有没有 synchronized 关键字。
  • AQS 体系（CountDownLatch/Future 等）：本文第 8 节列出的 CountDownLatch.countDown()、Future.get()、ConcurrentHashMap 的可见性保证，底层全部依赖 AQS 内部的 volatile state 字段——这正是"线程终止规则"和"volatile 变量规则"在实际并发工具类中的具象化，理解这一点后可以不用死记 API 语义，而是从 happens-before 直接推导出它们为什么安全。
  • Thread.join：本文提到的"线程终止规则"（线程内所有操作 happens-before join() 返回）正是 Thread.join() 常被用来做"等待子线程完成后安全读取其计算结果"的理论基础，不需要额外加锁或 volatile，join 本身就建立了跨线程的 happens-before 边界。
  • 面试高频考点："说出你知道的 happens-before 规则并举例"是本文内容的直接考点，常见追问还包括"为什么两个线程用不同锁对象就不安全"（对应误区 3）和"happens-before 是不是等于时间上先发生"（对应误区 1），考察的是能否把抽象规则还原到具体代码场景。
