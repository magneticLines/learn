synchronized 关键字

Java 内置的同步机制。一个关键字,同时解决了原子性、可见性、有序性三个问题。

📖 来源:《Java 并发编程实战》 第 2 章「线程安全性」

一、它到底在解决什么问题

并发 Bug 的根源只有一句话:多个线程同时读写同一份共享状态,而这些读写不是原子的。synchronized 的使命就是把「一段操作」变成对其他线程而言不可分割、且结果立即可见的整体。它一次性兜住了并发的三大特性:

  • 原子性:同一时刻只有一个线程能进入同步块,中间状态不会被别人看到。
  • 可见性:释放锁时把工作内存的修改刷回主内存,获取锁时从主内存重新读——别人能看到你的修改。
  • 有序性:进入/退出同步块构成内存屏障,块内对外表现为「不会被重排到块外」。

这正好对应了《并发编程实战》反复强调的概念:线程安全 = 对共享的、可变的状态进行正确的访问协调。synchronized 是协调的最基本工具。

二、反例 → 问题 → 正例

反例:看似简单的 i++

Java · 竞态条件
class Counter {
    private int count = 0;
    public void inc() { count++; // 不是一步! }
}

问题:count++ 在字节码层面是三步:读取 count → 加 1 → 写回。两个线程可能同时读到 10,各自加 1 写回 11,本该是 12——这就是竞态条件(race condition),经典的「读-改-写」复合操作丢失更新。

正例:用 synchronized 把三步焊成一步

Java
public synchronized void inc() { count++; }
// 现在「读-改-写」整体被锁保护,任意时刻只有一个线程能执行

三、锁的到底是谁(三种用法)

初学者最大的误区是「以为 synchronized 锁的是代码」。它锁的永远是某个对象(对象的内置锁 / 监视器锁)。三种写法的区别只是「锁哪个对象」:

写法锁的对象影响范围
修饰实例方法当前实例 this同一实例的所有同步实例方法互斥
修饰静态方法当前类的 Class 对象全类所有实例共享一把锁
同步代码块括号里指定的对象锁粒度最小、最灵活
Java · 同步代码块最灵活
private final Object lock = new Object();
void m() {
    // 只把真正需要互斥的几行锁起来,缩小临界区
    synchronized (lock) { /* 临界区 */ }
}

四、它的三个隐藏特性

  • 可重入:同一线程能重复获取自己已持有的锁。否则一个同步方法调用本对象的另一个同步方法就会自己锁死自己。
  • 非公平:锁释放时,所有等待线程平等竞争,不保证先来先得——性能更好,但可能「饿死」某些线程。
  • 不可中断:等锁期间无法被 interrupt() 打断,只能死等。这正是 ReentrantLock 想改进的痛点之一。

五、底层:锁升级(联系 JVM 对象头)

JDK 6 以后,synchronized 不再是「一上来就找操作系统要互斥量」的重操作。JVM 在对象头的 Mark Word 里记录锁状态,按竞争激烈程度单向升级:

无锁 → 偏向锁 → 轻量级锁 → 重量级锁
· 偏向锁:没竞争,只有一个线程反复进出 → 在对象头记下线程 ID,下次直接放行,几乎零开销。
· 轻量级锁:线程交替进入但不激烈 → 用 CAS + 自旋(忙等)抢锁,避免线程挂起/唤醒的上下文切换。
· 重量级锁:竞争激烈 → 自旋无果,膨胀为依赖 OS 互斥量的重锁,抢不到的线程被阻塞挂起。

这条链路解释了一个常见疑问:「为什么现在都说 synchronized 不慢了?」——因为大多数场景根本走不到重量级锁那一步。

六、与 ReentrantLock 的取舍

维度synchronizedReentrantLock
本质JVM 关键字,自动加解锁JUC 的 API 类,手动 lock()/unlock()
释放出块/异常自动释放,不会忘必须在 finally 里 unlock(),否则锁泄漏
公平性只能非公平可选公平/非公平
可中断 / 超时不支持lockInterruptibly() / tryLock(timeout)
条件队列单一 wait/notify多个 Condition,可分组唤醒
选择原则:需求简单就用 synchronized(代码短、由 JVM 兜底不易错);需要公平锁 / 可中断 / 超时 / 多条件队列时才上 ReentrantLock。JDK 6 之后性能已不是选择的主要依据。

七、最容易踩的坑

坑 1 · 锁错对象。 在两个不同实例上分别调用同步实例方法,锁的是两个不同的 this,根本互斥不了。想全局互斥要锁 Class 或同一个静态对象。
坑 2 · 拿可变/可被复用的对象当锁。 用 String 字面量或 Integer 缓存对象当锁,可能和无关代码「撞锁」;锁字段中途被重新赋值,则前后线程锁的不是同一个对象。锁对象要用 private final。
坑 3 · 临界区过大。 把耗时的 IO / 网络请求放进同步块,会让锁长期被占,并发度骤降。原则:临界区只保护共享状态,能短则短。

小结

  • synchronized 一次性解决原子性、可见性、有序性。
  • 它锁的是对象:实例方法锁 this、静态方法锁 Class、代码块锁指定对象。
  • 特性:可重入、非公平、不可中断。
  • 底层按竞争升级:偏向 → 轻量(自旋)→ 重量(OS 互斥量)。
  • 需要高级能力(公平/中断/超时/多条件)才换 ReentrantLock。
