06. 线程的创建与生命周期

从 OS 调度单元到 JVM 线程模型——搞清楚线程是怎么生出来的、怎么活着、怎么死掉，是一切并发编程的起点。

为什么需要"线程"这个抽象

上一节讲过，进程的切换代价很高：虚拟内存映射、文件描述符表都要跟着换。 线程的设计目标就是：在同一进程内，提供多条独立的执行流，同时共享堆内存与文件资源。

这个设计决定了线程的两个核心特征，也是之后所有并发问题的根源：

  • 共享堆（heap）：数据交换成本低，但竞态条件从这里开始
  • 独占栈（stack）：局部变量天然隔离，无需同步
JVM 线程 vs OS 线程：现代 JVM（HotSpot）采用 1:1 映射——每个 Java 线程对应一个内核线程（KLT）。调度完全由操作系统负责，Java 层面无法控制具体在哪个 CPU 上跑。这也是 Java 19+ 引入虚拟线程（Project Loom）的动机：M:N 映射，用少量 KLT 承载海量 VThread。

线程的三种创建方式

方式一：继承 Thread（不推荐用于生产）

Java继承 Thread
// 反例：把"任务"和"线程载体"耦合在一起
class PrintTask extends Thread {
    @Override
    public void run() {
        System.out.println("running in: " + Thread.currentThread().getName());
    }
}

// 问题：Java 是单继承，PrintTask 再也无法继承其他类
new PrintTask().start();

方式二：实现 Runnable（推荐，任务与线程解耦）

Java实现 Runnable
// 正例：Runnable 描述"要做什么"，Thread 描述"谁来跑"
Runnable task = () ->
    System.out.println("hello from " + Thread.currentThread().getName());

Thread t = new Thread(task, "worker-1");
t.start();  // 通知 JVM 创建 OS 线程并调度，并非立即执行
陷阱：run() vs start()
直接调 t.run() 不会创建新线程，任务在当前线程同步执行——这是初学者最常踩的坑，也是面试必考点。

方式三：实现 Callable + FutureTask（可获取返回值）

JavaCallable + FutureTask
Callable<Integer> calc = () -> {
    // 模拟耗时计算
    Thread.sleep(500);
    return 42;
};

FutureTask<Integer> future = new FutureTask<>(calc);
new Thread(future, "calc-thread").start();

// 主线程可以做别的事，再来取结果
Integer result = future.get();  // 阻塞直到任务完成
System.out.println("result = " + result);  // 42

Callable 相比 Runnable 多了两点：有返回值、可抛受检异常。 FutureTask 同时实现了 Runnable 和 Future，是它们之间的桥梁。

三种方式对比

方式返回值异常处理可复用任务对象推荐场景
extends Thread无只能 try-catch否（线程即任务）学习演示
Runnable无只能 try-catch是提交给线程池
Callable有可抛受检异常是需要结果 / 异常传播
实际工程中几乎不直接 new Thread：应当用 ExecutorService（线程池）统一管理线程的生命周期，避免线程数失控。

线程生命周期的六个状态

Java 用枚举 Thread.State 定义了六个状态，这与操作系统层面的"就绪/运行"合并成了同一个 RUNNABLE——因为 JVM 无法感知 OS 调度细节。

NEW  ── start() ──────────────────> RUNNABLE                                │  ↑ 获得 CPU                                │ 失去锁/等通知 ↓                         BLOCKED / WAITING / TIMED_WAITING                                │ 条件满足                                ↓ run() 返回 / 抛出未捕获异常                               TERMINATED
状态含义如何进入如何离开
NEW 线程对象已创建，OS 线程尚未启动 new Thread() 调用 start()
RUNNABLE 可运行（含在 CPU 上执行 + 等待 CPU 调度） start() / 锁/IO 恢复 主动阻塞 / 任务完成
BLOCKED 等待进入 synchronized 块/方法 争锁失败 获得 monitor
WAITING 无限期等待另一线程通知 Object.wait() / LockSupport.park() / Thread.join()（无超时） notify() / unpark() / 被 join 的线程结束
TIMED_WAITING 有超时的等待 sleep(ms) / wait(ms) / join(ms) 超时或被唤醒
TERMINATED 线程已结束，不可重启 run() 正常返回或异常退出 —（终态）

BLOCKED vs WAITING：两者的本质区别

这是面试高频考点，很多人混淆。一句话区分：

  • BLOCKED：被动等锁，由 JVM/OS 控制，不需要显式唤醒——只要锁释放，竞争者自动醒来
  • WAITING：主动挂起，必须有对应的 notify() / unpark() 才能恢复，否则永远等下去
Java观察状态
Object lock = new Object();

Thread blocker = new Thread(() -> {
    synchronized (lock) {
        try { Thread.sleep(5000); }
        catch (InterruptedException e) {}
    }
}, "holder");

Thread contender = new Thread(() -> {
    synchronized (lock) {}  // 争不到锁 → BLOCKED
}, "contender");

blocker.start();
Thread.sleep(100);  // 确保 holder 先拿到锁
contender.start();
Thread.sleep(100);

System.out.println(contender.getState());  // BLOCKED
System.out.println(blocker.getState());    // TIMED_WAITING (sleep)

线程属性：守护线程、优先级、中断标志

守护线程（Daemon Thread）

普通线程（用户线程）存活期间，JVM 不会退出。守护线程则相反：所有用户线程结束后，JVM 会强制终止所有守护线程并退出。 典型守护线程是 GC 线程、finalizer 线程。

Java守护线程陷阱
Thread daemon = new Thread(() -> {
    try {
        Thread.sleep(10_000);
        // ⚠️ 这行代码可能永远不会执行
        System.out.println("done");
    } catch (InterruptedException e) {}
});
// 必须在 start() 之前设置，否则抛 IllegalThreadStateException
daemon.setDaemon(true);
daemon.start();

// 主线程结束 → JVM 退出 → daemon 被强杀，"done" 不会打印

线程优先级

Java 提供 1–10 的优先级（默认 5），但优先级只是对 OS 调度器的"建议"，不保证执行顺序。 不同 OS 对优先级的映射差异巨大（Linux 甚至会忽略），生产代码不应依赖优先级来保证正确性。

中断机制（Interruption）

Java 没有提供强制停止线程的 API（Thread.stop() 已废弃，因为它会破坏锁的语义）。 正确的停止方式是协作式中断：通过中断标志通知线程"请你自己停下来"。

Java反例 vs 正例：如何停止线程
// ❌ 反例：用 volatile 标志位（无法中断阻塞状态的线程）
volatile boolean stopped = false;
Thread bad = new Thread(() -> {
    while (!stopped) {
        try {
            // 如果阻塞在这里，stopped=true 后线程依然不会退出
            Thread.sleep(1000);
        } catch (InterruptedException e) {}
    }
});

// ✅ 正例：使用 interrupt() 协作停止
Thread good = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            // sleep/wait 抛出 IE 后会清除中断标志，需要重新设置
            Thread.currentThread().interrupt();
            break;
        }
    }
});
good.start();
good.interrupt();  // 通知线程停止
中断标志被"吃掉"是常见陷阱
当 sleep()/wait()/join() 因中断而抛出 InterruptedException 时，中断标志会被自动清除。 如果 catch 块什么都不做，调用者就无法感知到这次中断。要么重新设置标志（Thread.currentThread().interrupt()），要么直接向上抛出异常。

start() 背后发生了什么

调用 t.start() 是一条 JNI 调用链，最终到达 OS 创建内核线程：

Thread.start() // Java 层   └─ start0() // native 方法 (JNI)      └─ JVM_StartThread // JVM 内部 (thread.cpp)          └─ os::create_thread() // OS 抽象层              └─ pthread_create() // Linux 内核线程                  └─ 内核分配 task_struct，加入调度队列

整个过程会在 JVM 堆外为新线程分配线程私有栈（默认 512KB–1MB，可用 -Xss 调整）。 这也是为什么线程不能无限创建——每个线程都要消耗一块连续的原生内存。

线程数量的代价：从 Thread 到 ThreadPool

每个 Java 线程背后的成本：

  • 内存：栈空间（默认约 512KB），4000 个线程 = ~2GB 栈内存
  • 创建/销毁：pthread_create 涉及系统调用，耗时约 50–100μs
  • 上下文切换：寄存器保存/恢复，L1/L2 Cache 失效，吞吐量下降
Java正确做法：使用线程池
// 用 ExecutorService 管理线程，复用已有线程，控制并发上限
ExecutorService pool = Executors.newFixedThreadPool(4);

for (int i = 0; i < 100; i++) {
    final int id = i;
    pool.submit(() ->
        System.out.println("task " + id + " on "
            + Thread.currentThread().getName())
    );
}
pool.shutdown();  // 等待已提交任务完成后关闭
Java 21 虚拟线程（Virtual Thread）
Thread.ofVirtual().start(runnable) 创建的虚拟线程由 JVM 调度，挂载在少量平台线程上。 阻塞时自动卸载（unmount），不占用 OS 线程。适合高并发 IO 密集型场景，每个请求一个虚拟线程成为可能。

面试高频考点总结

  • start() vs run()：start() 创建新线程，run() 在当前线程同步调用
  • 线程能否重启：不能。TERMINATED 是终态，再次调 start() 会抛 IllegalThreadStateException
  • BLOCKED vs WAITING：前者等锁（JVM/OS 自动管理），后者等通知（需显式唤醒）
  • 守护线程不保证 finally：JVM 强杀时 finally 块不一定执行，不要在守护线程里做资源清理
  • 中断标志被清除：catch InterruptedException 后必须处理（重设标志或向上抛），不能吞掉
  • 线程优先级不可靠：不要用优先级保证执行顺序，这是平台相关行为

与其他知识点的联系

  • 线程池 ThreadPoolExecutor：本文"线程数量的代价"一节列出的内存、创建/销毁、上下文切换三项成本，正是线程池存在的根本原因——线程池通过复用固定数量的线程避免了 pthread_create 的重复系统调用开销，把"每个任务一个线程"的模式改造成"任务排队、线程复用"，这是本文 Executors.newFixedThreadPool 示例背后的设计动机。
  • Runnable vs Callable：本文"线程的三种创建方式"已经对比了两者的核心差异（有无返回值、能否抛受检异常），这个选择在提交任务给线程池时依然存在——ExecutorService.submit(Runnable) 返回 Future<?>，而 submit(Callable) 才能拿到真正的计算结果，是本文内容在线程池场景下的直接延伸。
  • 中断机制 interrupt：本文用"反例 vs 正例"详细讲了协作式中断如何替代已废弃的 Thread.stop()，这套机制在线程池中同样适用——ExecutorService.shutdownNow() 内部就是对池中线程逐一调用 interrupt()，任务代码如果没有正确响应中断标志（如本文强调的"重设标志"），线程池的优雅关闭就会失效。
  • 守护线程 daemon：本文指出守护线程被 JVM 强杀时 finally 不保证执行，这与线程池里的工作线程形成对比——线程池默认创建的是非守护线程（用户线程），这也是为什么很多服务如果忘记调用 shutdown()，JVM 会因为线程池线程存活而无法退出，是本文守护线程语义的一个重要反向应用场景。
  • 虚拟线程 Virtual Threads（Java 21）：本文多处提到虚拟线程作为 1:1 平台线程模型的补充——理解本文"start() 背后发生了什么"里 pthread_create 的完整链路和每个线程独占的私有栈开销，才能真正体会虚拟线程"阻塞时自动卸载、不占用 OS 线程"的价值所在，两者是同一问题（线程太贵）的两代不同解法。
  • JVM 线程状态与调度：本文详细展开的六个 Thread.State，是排查并发问题（死锁、活锁、饥饿）时用 jstack 读懂线程转储的前提——比如判断某线程是 BLOCKED（等锁）还是 WAITING（等通知），直接决定了死锁诊断和普通阻塞诊断的思路分叉点。
