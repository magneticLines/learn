11. 对象的组合与同步策略

把零散的线程安全技巧"封装"进对象边界，让并发正确性变成可组合、可推理的设计属性。

1. 零散同步 vs 有边界的封装

前面几节讲了 volatile、synchronized、原子类、安全发布…… 但如果每处共享状态都需要调用者自己记住"先获取锁再操作"，系统就会充满隐患： 调用者一旦忘记，或者换了一个人维护代码，原有的不变量就悄悄被破坏。

对象的组合（Object Composition）解决的正是这个问题： 把同步策略内聚进类的实现，调用方无需关心锁在哪里，只要知道这个对象"是线程安全的"即可。 这是"缩小共享 = 减少 Bug"理念在设计层面的落地。

核心思想：线程安全不是一堆 synchronized 的堆砌， 而是一组不变量（invariant）在所有可能并发执行路径下始终成立的承诺。 封装让这个承诺的维护范围缩小到类内部。

2. 状态所有权：先想清楚"谁管这块状态"

设计线程安全类的第一步，不是选择什么锁，而是搞清楚对象状态的所有权。 Java 语言里对象状态通过域（field）体现，域的引用关系决定了哪些对象共同构成"状态"。

状态所有权模型
├── 独占所有权 对象持有域的唯一引用，外部无法访问 → 最易推理
├── 共享所有权 多个对象共享同一引用（如缓存）→ 需要协调
├── 发布后转移 创建时独占，发布后交出控制权 → 见安全发布
└── 借用 临时传入，用完归还，不保留引用 → 方法参数常见

独占所有权让你可以只在类内部维护不变量，无需担心外部线程突然改写状态。 一旦把可变状态的引用"泄露"出去，所有权就变成共享的，同步策略也随之复杂化。

3. 实例封闭（Instance Confinement）

如果一个对象本身不是线程安全的（如 ArrayList、HashMap）， 但它被完全封闭在一个已做好同步的对象内部，外部线程无法直接访问它—— 这就是实例封闭，也是最常用的组合策略。

反例：封闭被破坏，内部状态泄露

Java反例 — 返回内部集合引用
public class UnsafeTracker {
    // 用 HashMap 保存位置，HashMap 非线程安全
    private final Map<String, Point> locations = new HashMap<>();

    public synchronized void addVehicle(String id, Point p) {
        locations.put(id, p);
    }

    // ❌ 直接返回内部 Map 引用！外部拿到后可绕过同步直接修改
    public Map<String, Point> getLocations() {
        return locations;   // 封闭被打破
    }
}

外部代码拿到 locations 引用后，可以在没有锁的情况下直接 put， 完全绕过 synchronized，导致不变量被破坏。

正例：返回防御性副本，保持封闭

Java正例 — 返回不可变快照
@ThreadSafe
public class SafeTracker {
    @GuardedBy("this")
    private final Map<String, Point> locations = new HashMap<>();

    public synchronized void addVehicle(String id, Point p) {
        locations.put(id, p);
    }

    // ✅ 返回一份不可变快照，外部修改不影响内部状态
    public synchronized Map<String, Point> getLocations() {
        return Collections.unmodifiableMap(new HashMap<>(locations));
    }
}
Java 平台库的最佳实践：Collections.unmodifiableMap、 List.copyOf、Map.copyOf 都是这种思路的体现—— 把可变状态的快照转成不可变视图再发布出去。

4. 文档化同步策略：@GuardedBy

同步策略是类设计的核心契约，必须显式声明，而不是藏在代码里让人猜。 @GuardedBy 注解（来自 net.jcip.annotations 或手动约定） 明确标注"这个域由哪把锁保护"。

Java@GuardedBy 文档化示例
@ThreadSafe
public class ServerStatus {

    @GuardedBy("this")
    private int activeConnections;

    @GuardedBy("this")
    private final Set<String> connectedUsers = new HashSet<>();

    // 注意：两个域必须在同一把锁下一起更新，保持 size == count 的不变量
    public synchronized void userConnected(String user) {
        connectedUsers.add(user);
        activeConnections++;
    }

    public synchronized void userDisconnected(String user) {
        connectedUsers.remove(user);
        activeConnections--;
    }
}

connectedUsers.size() 与 activeConnections 必须始终相等—— 这就是这个类的复合不变量。 只有把两个域都用同一把锁保护，并在同一个 synchronized 方法内同时更新， 这个不变量才能在并发下成立。

5. 委托给线程安全组件

如果一个类的状态可以完全委托给一个（或多个独立的）线程安全组件来管理， 那么这个类本身就可以不需要额外的同步。这是比实例封闭更轻量的路线。

单个组件委托 — 直接委托即可

Java委托给 AtomicInteger
@ThreadSafe
public class Counter {
    // 把并发控制完全委托给 AtomicInteger，自己不需要任何同步
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() { count.incrementAndGet(); }
    public int  get()       { return count.get();             }
}

多个独立组件 — 各自委托，不存在跨组件不变量

Java两个独立的 AtomicLong，无复合不变量
@ThreadSafe
public class NumberRange {
    private final AtomicLong lower = new AtomicLong(0);
    private final AtomicLong upper = new AtomicLong(100);

    // 各自独立更新没问题，因为 lower 和 upper 之间没有联动约束
    public void setLower(long v) { lower.set(v); }
    public void setUpper(long v) { upper.set(v); }
}

反例：多个组件存在复合不变量时，委托失败

Java反例 — lower ≤ upper 的不变量无法单靠委托保证
@NotThreadSafe
public class BrokenRange {
    private final AtomicInteger lower = new AtomicInteger(0);
    private final AtomicInteger upper = new AtomicInteger(10);

    // ❌ 先检查再设置：检查通过后，另一个线程可能已经改了 upper
    public void setLower(int v) {
        if (v > upper.get())
            throw new IllegalArgumentException();
        lower.set(v);  // check-then-act，不原子！
    }

    public void setUpper(int v) {
        if (v < lower.get())
            throw new IllegalArgumentException();
        upper.set(v);  // 同理
    }
}
委托失败条件：当多个状态变量之间存在复合不变量（如 lower ≤ upper）， 单靠把每个变量委托给独立的原子类是不够的。 两个原子操作合在一起不是原子的，必须在一把锁的保护下统一更新。

6. 扩展已有的线程安全类

有时需要给现有线程安全类添加新的复合操作（如"若不存在则插入"）。 常见的三种方式各有利弊：

方式做法风险
直接修改源码 在原类里加方法 无法修改第三方库
继承并重写 extends Vector 依赖父类同步机制，父类改实现则失效（脆弱）
客户端加锁 在调用方对同一把锁 synchronized 锁耦合，需要知道被委托类用的是哪把锁
组合（推荐） 包装类持有内部对象，用自己的锁保护一切 性能略有开销，但最安全、最解耦

正例：组合方式添加"若不存在则添加"

Java组合扩展 — 安全的 putIfAbsent
@ThreadSafe
public class ImprovedList<T> {
    private final List<T> list;

    public ImprovedList(List<T> list) { this.list = list; }

    // 用 ImprovedList 自己的内置锁保护整个 check-then-act
    public synchronized boolean putIfAbsent(T x) {
        boolean absent = !list.contains(x);
        if (absent) list.add(x);
        return absent;
    }

    // 所有代理方法也需要同步，确保锁不被绕过
    public synchronized boolean add(T x)        { return list.add(x);        }
    public synchronized boolean remove(Object o) { return list.remove(o);     }
}

关键在于：ImprovedList 用的是自己的内置锁（this）， 而不依赖被包装的 list 内部用什么锁。 无论传入的是 ArrayList（非线程安全）还是 Collections.synchronizedList， 组合层面的同步都由包装类统一负责，不会出现锁耦合问题。

7. 客户端加锁：常见陷阱

有时为了复用已有的线程安全集合，开发者会在客户端加锁来实现复合操作。 这种方式可行，但极易出错。

Java反例 — 客户端加锁用了错误的锁对象
List<String> list = Collections.synchronizedList(new ArrayList<>());

// ❌ 用的是 BuggyHelper.class 的锁，而 synchronizedList 内部用的是 list 本身
public synchronized boolean putIfAbsent(String x) {
    boolean absent = !list.contains(x);
    if (absent) list.add(x);
    return absent;
}

// ✅ 必须锁在 list 对象本身上，才和 synchronizedList 内部的锁一致
public boolean putIfAbsentCorrect(String x) {
    synchronized (list) {
        boolean absent = !list.contains(x);
        if (absent) list.add(x);
        return absent;
    }
}
并发陷阱：客户端加锁必须锁在被委托对象使用的同一把锁上。 Collections.synchronizedList 内部用包装后的 list 对象本身作为锁， 所以客户端也必须 synchronized(list)，而不是其他对象。 锁不一致 = 形同虚设。

8. 设计清单：如何记录同步策略

一个设计良好的线程安全类，应在类级别和域级别都清晰记录同步策略。 以下是最小化的文档模板：

Java同步策略文档模板
/**
 * 线程安全的订单队列。
 *
 * <p>同步策略：所有可变状态由内置锁（this）保护。
 * 调用方无需额外同步；不要将内部集合的引用暴露给外部。
 *
 * <p>不变量：
 *   - pendingOrders.size() == pendingCount（两者必须一起更新）
 *   - 已完成的订单不会再出现在 pendingOrders 中
 */
@ThreadSafe
public class OrderQueue {
    @GuardedBy("this") private final Queue<Order> pendingOrders = new ArrayDeque<>();
    @GuardedBy("this") private int pendingCount;
    // ...方法省略...
}

9. 总结：组合策略的选择树

选择同步策略时的决策路径
├── 状态能完全委托给一个线程安全组件？
│ └── 是 → 直接委托，类本身无需同步（最简单）
├── 状态由多个独立变量组成，无复合不变量？
│ └── 是 → 各自委托独立的原子类
├── 状态间存在复合不变量？
│ └── 是 → 用同一把锁保护所有相关域（实例封闭）
├── 需要给现有类添加复合操作？
│ ├── 能修改源码 → 直接在类里加同步方法
│ ├── 不能修改源码 → 组合包装（优先），或客户端加锁（需知道正确的锁）
└── 所有情况都要：记录同步策略（@ThreadSafe / @GuardedBy / JavaDoc）
记住：同步策略是类的 API 的一部分。 调用者有权知道这个类是否线程安全、需要哪些外部协作。 不记录 = 对未来的维护者埋雷。

与其他知识点的联系

  • 关联知识点：安全发布（上一节）：对象组合关心的是"多线程看到对象后如何持续安全地操作它"，而安全发布关心的是"对象引用如何第一次被正确看到"——本节 @GuardedBy 标注的锁保护，正是建立在字段已被安全发布的前提之上，两节合起来才是设计线程安全类的完整流程。
  • 关联知识点：线程安全的定义（原子性/可见性/有序性）：本节的"复合不变量"概念（如 lower ≤ upper、size == count）本质上是把原子性的粒度从单个变量扩大到多个变量的联合状态——单个 AtomicInteger 保证不了跨变量的一致性，这正是为什么委托模式在有复合不变量时会失效。
  • 关联知识点：同步容器（Vector / Collections.synchronizedXxx）：客户端加锁必须使用与被委托对象相同的锁这一原则，直接对应下一部分"同步容器"里 Collections.synchronizedList 的 mutex 陷阱——用错锁对象导致复合操作实际未被保护，是同一类错误在不同章节的重复出现。
  • 关联知识点：ConcurrentHashMap 的原子复合操作："委托给线程安全组件"策略的局限性（存在复合不变量时失效），恰好说明了为什么 ConcurrentHashMap 要专门提供 computeIfAbsent、compute 等原子复合操作 API——它们把"检查+更新"这一复合逻辑下沉到容器内部的锁保护范围内，避免了客户端再次面对本节的陷阱。
  • 关联知识点：Spring 依赖注入与线程安全设计：Spring 单例 Bean 中若持有可变状态，本质上就是在做本节讨论的"状态所有权"设计——是该把状态委托给线程安全的成员（如 AtomicLong），还是该用 synchronized/Lock 保护复合不变量，这正是 Bean 设计阶段必须回答的问题。
  • 关联知识点：分布式锁与分布式一致性：本节"用同一把锁保护所有相关不变量"的思路是单机场景下的解法；当状态跨多个服务/实例共享时，本地锁失效，需要用分布式锁（Redis/ZooKeeper）或分布式事务来保证跨进程的复合不变量，这是并发设计思想从单机到分布式的自然延伸。
