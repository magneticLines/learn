07. 线程安全的定义
（原子性 / 可见性 / 有序性）

并发 Bug 的根源几乎全部来自这三个性质被破坏——理解它们，就读懂了 synchronized、volatile、Lock 背后所有设计决策的逻辑起点。

1. 什么叫"线程安全"

《Java 并发编程实战》给出的经典定义：

当多个线程访问某个对象时，不管运行时环境采用何种调度方式或者这些线程将如何交替执行，并且在调用代码中不需要任何额外的同步或协同，这个对象都能表现出正确的行为，那么就称这个对象是线程安全的。

关键词是正确的行为——无论线程怎样调度，结果都和单线程执行完全一致。要做到这一点，必须同时保证三个性质：原子性、可见性、有序性。破坏任意一个，程序就可能产生并发 Bug。

线程安全的三根支柱 ├── 原子性（Atomicity） ← 操作不可被中途打断 ├── 可见性（Visibility） ← 修改对其他线程立即可见 └── 有序性（Ordering） ← 代码执行顺序符合预期

这三个性质并非 Java 独有，它们是现代多核体系结构（多级 CPU 缓存 + 乱序执行）与编译器优化共同带来的天然挑战，所有并发语言都必须面对。

2. 原子性——操作不可被"对半劈"

2.1 为什么会有原子性问题

一条看似简单的 Java 语句 count++ 在字节码层面被拆解成三步：

① getstatic 读取 count 的当前值到操作数栈 ② iconst_1 / iadd 执行 +1 ③ putstatic 把结果写回 count

如果线程 A 执行完第①步、还没执行第③步时，线程 B 也读取了 count，就会拿到旧值，两个线程各自 +1 后写回，实际只加了一次——这就是竞态条件（Race Condition）。

2.2 反例 → 问题 → 正例

反例：非原子的计数器（unsafe）Java
// ❌ 多线程下 count 最终值几乎必然小于预期
public class UnsafeCounter {
    private int count = 0;

    public void increment() {
        count++;  // 读-改-写，三步非原子，随时可被打断
    }

    public int get() {
        return count;
    }
}
正例 A：synchronized 保证原子性Java
public class SynchronizedCounter {
    private int count = 0;

    // synchronized 让整个方法成为一个临界区：
    // 同一时刻只有一个线程能执行，三步操作不会被插入其他线程
    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}
正例 B：AtomicInteger — CAS 硬件级原子指令Java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    // 底层调用 Unsafe.compareAndSetInt，一条 CPU 指令完成读-比较-写
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();
    }

    public int get() {
        return count.get();
    }
}
陷阱：long / double 的非原子读写
在 32 位 JVM 上，对 long 和 double 的读写不是原子的（它们占 64 位，JVM 可能分两次 32 位操作完成）。在多线程环境中读写 long/double 字段，必须加 volatile 或用 synchronized，否则可能读到"半截"值。64 位 JVM 上虽然实现通常是原子的，但 JLS 并不保证，保险起见始终加 volatile。

3. 可见性——你写的我能看到吗

3.1 为什么会有可见性问题

现代 CPU 有多级缓存（L1/L2/L3）。每个核心有自己的 L1/L2 缓存，写操作先落到本地缓存，不会立刻刷回主内存。其他核心读取同一变量时，可能读到的是自己缓存里的旧值，即使写线程早已修改。

Thread A (Core 0) Thread B (Core 1) L1 Cache: flag = true L1 Cache: flag = false ← 旧值！ ↓ 未刷到主内存 ↑ 未从主内存更新 Main Memory: flag = ?

Java 内存模型（JMM）抽象地把这一现象描述为：每个线程都有自己的"工作内存"（对应 CPU 缓存），线程对共享变量的操作先在工作内存中进行，何时同步到主内存由 JMM 规则决定——而不是立刻。

3.2 反例：死循环的停止标志

反例：volatile 缺失导致线程永远不停止Java
public class StopThread {
    // ❌ 没有 volatile：主线程的修改可能永远对子线程不可见
    private static boolean stopRequested = false;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            // JIT 可能将此循环优化为：if (!stopRequested) while(true) {}
            while (!stopRequested) {
                // 工作...
            }
        });
        t.start();

        Thread.sleep(1000);
        stopRequested = true;  // 主线程修改，子线程可能永远看不到
    }
}
正例：volatile 保证可见性Java
public class StopThread {
    // ✅ volatile 告诉 JMM：每次读都从主内存取，每次写都立刻刷到主内存
    private static volatile boolean stopRequested = false;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (!stopRequested) { }
        });
        t.start();

        Thread.sleep(1000);
        stopRequested = true;  // 子线程在下一次循环必然看到 true
    }
}
synchronized 也保证可见性：synchronized 块退出时会把工作内存中的变量刷回主内存，进入时会从主内存重新加载。所以 synchronized = 原子性 + 可见性，而 volatile 只保证可见性（不保证原子性）。
陷阱：volatile 不能替代锁
volatile 保证单次读/写的可见性，但 count++ 是复合操作（读-改-写），即使加了 volatile 也无法保证原子性。需要原子性时必须用 synchronized 或 Atomic* 类。

4. 有序性——代码执行的顺序真的如你所写吗

4.1 重排序的来源

为了提升性能，有三个层次会对指令进行重排序：

层次谁在做目的
编译器重排序javac / JIT减少寄存器读写，提高指令并行度
指令级并行重排序CPU 流水线无数据依赖的指令乱序执行
内存系统重排序CPU 缓存 / 写缓冲区Store Buffer 延迟写，Load 提前读

在单线程中，JMM 保证重排序不改变最终结果（as-if-serial 语义）。但在多线程中，一个线程看到的另一个线程的操作顺序可能是被重排后的，导致错误。

4.2 经典案例：双重检查锁（DCL）的有序性陷阱

反例：DCL 单例——指令重排导致返回未初始化对象Java
public class Singleton {
    // ❌ 缺少 volatile：instance 的赋值可能被重排序
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {          // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {   // 第二次检查（有锁）
                    // new 操作分三步：① 分配内存 ② 初始化对象 ③ 引用指向内存
                    // JIT 可能将顺序重排为：① ③ ②
                    // 线程 B 在第一次检查时看到 instance != null，
                    // 但拿到的是还未完成初始化的对象！
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
正例：volatile 禁止重排序，DCL 正确实现Java
public class Singleton {
    // ✅ volatile 在写操作前后插入内存屏障，禁止 new 的三步被重排序
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
更优雅：静态内部类（Initialization-on-demand Holder）
利用类加载机制天然的线程安全性，完全不需要 volatile 和 synchronized：

private static class Holder { static final Singleton INSTANCE = new Singleton(); }
public static Singleton getInstance() { return Holder.INSTANCE; }

4.3 happens-before：JMM 的有序性保证

JMM 通过 happens-before（先行发生） 规则向程序员承诺哪些操作之间存在确定的顺序关系，不需要额外同步：

规则含义
程序顺序规则同一线程内，前面的操作 happens-before 后面的操作
监视器锁规则unlock happens-before 后续对同一锁的 lock
volatile 变量规则对 volatile 的写 happens-before 后续对同一变量的读
线程启动规则Thread.start() happens-before 新线程的任意操作
线程终止规则线程的所有操作 happens-before 其他线程检测到该线程终止
传递性若 A hb B，B hb C，则 A hb C

两个操作之间如果没有 happens-before 关系，JMM 不保证它们的顺序，编译器和 CPU 可以任意重排。

5. 三个性质的横向对比

性质 根本原因 典型症状 解决手段
原子性 复合操作被线程切换打断 计数器丢失更新、转账金额错误 synchronized、Lock、Atomic*
可见性 CPU 缓存未及时同步到主内存 标志位修改后线程不退出、读到旧值 volatile、synchronized、final
有序性 编译器 / CPU 指令重排 DCL 返回半初始化对象、初始化顺序错乱 volatile（内存屏障）、synchronized
synchronized 一招解三忧：synchronized 同时保证原子性（互斥）、可见性（进出临界区刷内存）、有序性（happens-before 规则）。代价是有竞争时的线程阻塞开销。volatile 只保证可见性和有序性，不保证原子性，但无锁，开销极低。

6. 综合示例：初始化发布（安全发布）

一个对象被创建后，如何确保其他线程看到完整初始化的状态？这是三个性质综合作用的典型场景。

反例：不安全发布——其他线程可能看到部分初始化的对象Java
public class Holder {
    private int n;

    public Holder(int n) {
        this.n = n;
    }

    public void assertSanity() {
        // ❌ 可能抛出 AssertionError：第一次和第二次读取 n 的值不同
        // 因为构造函数写入和引用发布之间没有 happens-before 保证
        if (n != n)
            throw new AssertionError("This statement is false");
    }
}

// 发布代码（全局共享，无同步）
public static Holder holder;  // ❌ 不安全发布
holder = new Holder(42);
正例：四种安全发布方式Java
// 方式 1：静态初始化（类加载时由 JVM 保证线程安全）
public static Holder holder = new Holder(42);

// 方式 2：volatile 引用（写 happens-before 后续读）
public static volatile Holder holder = new Holder(42);

// 方式 3：加锁保护（synchronized 保证可见性）
public synchronized void publish(Holder h) {
    this.holder = h;
}

// 方式 4：放入线程安全容器（如 ConcurrentHashMap、BlockingQueue）
BlockingQueue<Holder> queue = new LinkedBlockingQueue<>();
queue.put(new Holder(42));

7. 面试高频考点速查

  • volatile 能保证原子性吗？ 不能。只保证单次读/写的可见性和禁止指令重排序。
  • synchronized 能保证有序性吗？ 能，通过 happens-before 的监视器锁规则，并在进出临界区时插入内存屏障。
  • i++ 是原子操作吗？ 不是，三步操作。需要 AtomicInteger 或 synchronized。
  • DCL 为什么需要 volatile？ 禁止 new 操作中"分配内存→指向引用→初始化对象"的重排序，防止其他线程拿到未初始化的对象。
  • final 字段有可见性保证吗？ 有。构造函数结束后，final 字段对所有线程可见，不需要额外同步（前提是 this 引用没有在构造函数中"逸出"）。
  • happens-before 是什么？ JMM 对程序员的可见性承诺：若 A happens-before B，则 A 的所有修改对 B 可见。它不等于"时间上 A 先于 B 执行"。
