10. 安全发布的几种方式

对象构造完之后如何让其他线程"看到"它——这件看似平凡的事，暗藏 Java 内存模型最经典的并发陷阱。

什么是"发布"与"安全发布"

发布（publish）是指让对象在其作用域之外可见——比如把它存入静态字段、传给其他对象的方法、或放进共享容器。 安全发布（safe publication）则要求：当其他线程看到这个引用时，它也能看到对象完整初始化后的状态。

听起来理所当然，但 Java 内存模型（JMM）并不保证"写操作对另一个线程立刻可见"—— 除非两个操作之间存在 happens-before 关系。没有这条保证，编译器和 CPU 都可以对指令重排， 读线程可能看到一个"半初始化"的对象。

核心问题：对象引用的可见性 ≠ 对象内部字段的可见性。 线程 A 把引用写入共享变量，线程 B 读到了这个引用，却不保证能看到 A 在构造器里写的那些字段值。

反例：不安全发布导致的半初始化对象

JavaUnsafePublication.java
public class UnsafePublication {

    // 普通字段，无任何同步保护
    public static Holder holder;

    public static void initialize() {
        holder = new Holder(42);   // 线程 A 发布
    }
}

class Holder {
    private int n;

    public Holder(int n) {
        this.n = n;
    }

    public void assertSanity() {
        // 线程 B 可能两次读到不同的 n——第一次 0（默认值），第二次 42
        if (n != n)
            throw new AssertionError("This statement is false.");
    }
}

乍看 n != n 永远为 false，但 JMM 允许编译器将字段读取展开成两次独立加载。 如果第一次加载时 CPU 缓存还没刷新（看到 0），第二次刷新后（看到 42）， 这个断言就会抛出 —— 现实中真实发生过。

陷阱：即使对象引用已经可见，字段值仍可能是"旧值"或"默认值"。 这不是偶发性的竞态条件，而是 JMM 规范允许的合法行为。

安全发布的本质：happens-before

JMM 用 happens-before（HB） 定义"谁能看到谁的写入"： 若操作 A happens-before 操作 B，则 A 的所有写入对 B 可见。 安全发布就是在"写引用"和"读引用"之间建立一条 HB 链。

写线程（构造 + 发布） ├── 构造器内字段赋值 ──HB──▶ 构造完成 └── 构造完成 ──HB──▶ 安全发布动作 │ ▼ 读线程（看到引用） └── 读到引用 ──HB──▶ 读到完整初始化字段

Java 规范给出了几种可以在写/读之间建立 HB 的机制，下面逐一讲解。

安全发布的四种方式

方式 1：静态初始化器（最简单）

在静态初始化器中初始化对象引用，JVM 保证类加载时的 HB，所有后续读取均安全。 这是最安全也最简洁的方式，适合单例。

JavaStaticInit.java
public class StaticInit {
    // JVM 类加载时执行，天然线程安全
    private static final Holder HOLDER = new Holder(42);

    public static Holder getInstance() {
        return HOLDER;
    }
}
JLS §12.4.2 规定：类初始化期间，JVM 持有类初始化锁（CL lock），所有线程对同一类的初始化操作是串行的， 初始化完成后的写入对所有后续读取线程可见。

方式 2：volatile 字段

将共享引用声明为 volatile，JMM 对 volatile 写入和读取建立 HB： 写入 volatile 字段 HB 于后续所有读取该字段的操作。

JavaVolatilePublication.java
public class VolatilePublication {
    // volatile 写 happens-before volatile 读
    public volatile Holder holder;

    public void publish() {
        // 先完整构造，再赋给 volatile —— 赋值那一刻建立 HB
        Holder tmp = new Holder(42);
        holder = tmp;  // volatile write
    }

    public void consume() {
        Holder h = holder;  // volatile read，HB 已建立
        if (h != null) h.assertSanity();  // 安全：可见到完整的 n=42
    }
}
陷阱：volatile 只保证引用本身的可见性与有序性，不保证对象内部的原子复合操作。 如果 Holder 发布后还会被修改（多字段更新），仍需额外同步。

方式 3：synchronized 保护的字段

对同一把锁上的 unlock HB 于后续的 lock（监视器规则）。 用 synchronized 方法或块写入、读取共享引用，即可保证安全发布。

JavaSynchronizedPublication.java
public class SynchronizedPublication {
    private Holder holder;

    public synchronized void publish(Holder h) {
        holder = h;  // unlock 时刷新到主内存
    }

    public synchronized Holder get() {
        return holder;  // lock 时从主内存重新加载
    }
}

Vector、Hashtable、Collections.synchronizedXxx() 内部就是这种方式——所有读写都在同一把锁上，安全但吞吐量受限。

方式 4：并发容器（ConcurrentHashMap / BlockingQueue 等）

JDK 并发容器内部已实现了安全发布语义。将对象放入 ConcurrentHashMap、BlockingQueue、CopyOnWriteArrayList 等， 其他线程从中取出时即可安全访问。

JavaConcurrentPublication.java
import java.util.concurrent.*;

public class ConcurrentPublication {
    private final BlockingQueue<Holder> queue = new LinkedBlockingQueue<>();

    // 生产者线程：构造完毕后 put 进队列
    public void producer() {
        queue.put(new Holder(42));  // BlockingQueue 内部有 volatile/lock，建立 HB
    }

    // 消费者线程：take 出来即安全可见
    public void consumer() {
        Holder h = queue.take();
        h.assertSanity();  // 安全
    }
}

特殊情况：final 字段的特殊保证

JMM 为 final 字段提供了初始化安全性（initialization safety）保证： 只要对象在构造期间没有"逸出"（this 没有提前泄露），任何线程通过任意方式看到该引用时， 都能看到所有 final 字段的正确值——即使没有 volatile/synchronized。

JavaFinalSafety.java
public class ImmutablePoint {
    public final int x;
    public final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }
}

// 即便用普通变量发布，其他线程也能看到 x=3, y=4
public static ImmutablePoint sharedPoint = new ImmutablePoint(3, 4);
前提：构造器不能逸出 this！若在构造器里把 this 注册到监听器、启动线程、 或赋值给静态字段，final 保证立即失效。这是上一节"不变性与 final"的核心陷阱。

经典陷阱：双重检查锁定（DCL）

双重检查锁定（Double-Checked Locking）是 Java 并发史上最著名的错误模式之一， 正确理解它需要同时掌握安全发布与 volatile 语义。

错误版本（Java 5 之前）

Java — 有缺陷BrokenDCL.java
public class BrokenSingleton {
    private static BrokenSingleton instance;  // ❌ 缺少 volatile

    public static BrokenSingleton getInstance() {
        if (instance == null) {                // 第一次检查（无锁）
            synchronized (BrokenSingleton.class) {
                if (instance == null) {        // 第二次检查（持锁）
                    instance = new BrokenSingleton();
                    // 问题：new 分三步（分配内存 → 初始化字段 → 赋值给 instance）
                    // JIT 可能将步骤 2、3 重排：先赋值，再初始化
                    // 其他线程在第一次检查时看到非 null，却拿到未初始化的对象
                }
            }
        }
        return instance;
    }
}

正确版本：volatile DCL（Java 5+）

Java — 正确VolatileDCL.java
public class SafeSingleton {
    // volatile 禁止"赋值"与"初始化"之间的重排序
    private volatile static SafeSingleton instance;

    public static SafeSingleton getInstance() {
        if (instance == null) {
            synchronized (SafeSingleton.class) {
                if (instance == null) {
                    instance = new SafeSingleton();  // ✅ volatile write，禁止重排
                }
            }
        }
        return instance;
    }
}

更优雅：Initialization-on-demand Holder（推荐）

Java — 最简洁LazyHolder.java
public class LazyHolder {
    // 静态内部类，只在第一次调用 getInstance() 时被加载
    private static class Holder {
        static final LazyHolder INSTANCE = new LazyHolder();
    }

    public static LazyHolder getInstance() {
        return Holder.INSTANCE;  // 利用类加载的线程安全性，零额外开销
    }
}
Initialization-on-demand Holder 利用了 JVM 类加载天然的线程安全性，无需 volatile 或 synchronized， 是实现懒加载单例的最优雅方式。Enum 单例（《Effective Java》推荐）本质上也是静态初始化器，同理。

四种方式对比总览

方式 HB 来源 适用场景 性能 注意点
静态初始化器 类加载锁 饿汉单例、常量 最优（无运行时开销） 无法懒加载
volatile 字段 volatile write → read 可变共享引用，DCL 良好（仅内存屏障） 不保护对象内部复合操作
synchronized unlock → lock 任意可变状态 有锁竞争开销 读写都需同步同一把锁
并发容器 容器内部机制 生产者-消费者、缓存 良好（无锁或分段锁） 对象放入后不应再修改

逸出：无意间的不安全发布

逸出（escape）是指对象在构造完成前就被其他线程访问到，常见场景：

  • 构造器中把 this 注册到事件监听器
  • 构造器中启动一个新线程，并传入 this
  • 构造器中调用可覆盖的方法（子类可能泄露 this）
Java — 逸出反例ThisEscape.java
public class ThisEscape {
    private final int value;

    public ThisEscape(EventSource source) {
        // ❌ 在构造器尚未完成时，this 就逸出到监听器
        source.registerListener(new EventListener() {
            public void onEvent(Event e) {
                doSomething(e);  // 隐式引用了外部类的 this
            }
        });
        value = 42;  // 若事件在此之前触发，value 可能看到 0
    }
}
Java — 工厂方法正例SafePublish.java
public class SafeListener {
    private final int value;

    // 私有构造器：确保构造完全完成
    private SafeListener() {
        value = 42;
    }

    // ✅ 工厂方法：构造完毕后再注册，避免 this 逸出
    public static SafeListener newInstance(EventSource source) {
        SafeListener listener = new SafeListener();
        source.registerListener(listener);  // 构造完成后再注册
        return listener;
    }
}

小结

安全发布的本质是在"写入引用"和"读取引用"之间建立 happens-before 关系， 使读线程不仅能看到引用本身，还能看到对象所有字段的正确值。

安全发布的四条路 ├── 静态初始化器 ── 最简单，类加载天然保证 ├── volatile 字段 ── 轻量，适合"写一次读多次"的引用 ├── synchronized ── 万能，写读都需持同一把锁 └── 并发容器 ── 优雅，生产者-消费者首选 记住：对象在构造器中不能逸出 this，否则一切保证都失效
面试高频考点： DCL 为什么需要 volatile？—— 因为 new 操作在字节码层面是三步，JIT 可以将"赋值"提前到"初始化"之前。 volatile 写禁止了这个重排序，使其他线程要么看到 null，要么看到完整初始化的对象，没有中间状态。
