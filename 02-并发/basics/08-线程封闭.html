08. 线程封闭（ThreadLocal / 栈封闭）

彻底消除共享——不共享的数据天然线程安全，根本不需要加锁。

为什么"不共享"是最好的并发策略？

上一节我们定义了线程安全的三要素：原子性、可见性、有序性。 这三者都是在多线程争抢同一份数据时才会出问题。 既然如此，最彻底的解决方案就是：让每个线程只看到属于自己的数据副本，根本不存在争抢。

这就是线程封闭（Thread Confinement）的核心思想： 把对象的使用限制在单一线程内，无需同步就能保证线程安全。

生活类比：厕所里的马桶不需要"加锁"协调多人同时使用——因为每次只有一个人进去，门关上就天然隔离了。线程封闭就是给每个线程都装一扇"单人间的门"。
线程封闭的三种实现方式 ├── Ad-hoc 封闭 ── 完全靠程序员纪律维护，脆弱，不推荐 ├── 栈封闭 ── 局部变量天然封闭，JVM 栈帧隔离保证 └── ThreadLocal ── JDK 提供的线程局部存储，最常用

栈封闭：最简单、最安全的封闭形式

Java 中每个线程都有独立的 JVM 栈。方法调用时创建栈帧，栈帧中的局部变量只在当前线程的栈上存活。 只要对象引用不逃逸出方法（不被赋给字段、不传入其他线程），它就永远只属于当前线程。

正例：局部变量天然线程安全

Java栈封闭示例
public int sumList(List<Integer> data) {
    // sum 是局部变量，在当前线程的栈帧里，其他线程根本看不到
    int sum = 0;
    for (int v : data) {
        sum += v;
    }
    return sum;  // 返回的是 int 值，不是引用，安全
}

反例：局部变量"逃逸"后封闭被破坏

Java逃逸分析——封闭失效
private List<String> sharedList;  // 字段 = 共享！

public void dangerousMethod() {
    List<String> local = new ArrayList<>();
    local.add("data");

    // 把 local 引用赋给实例字段 → 对象逃逸出了当前线程
    // 此后任何其他线程都能通过 sharedList 访问同一个 ArrayList
    sharedList = local;  // ❌ 封闭被破坏
}
逃逸场景速查：① 赋给实例/静态字段；② 作为参数传给另一个线程；③ 在构造方法中发布 this；④ 存入共享容器（Map、Queue 等）。任意一种都会破坏栈封闭。

ThreadLocal：JDK 内置的线程局部存储

栈封闭只能管住"方法调用期间"。如果需要在同一线程的多个方法间共享数据，又不想暴露给其他线程， 就需要 ThreadLocal。

内部数据结构

ThreadLocal 本身只是一个 Key。每个 Thread 对象内部有一个 ThreadLocalMap（类似 HashMap），Key 是 ThreadLocal 实例，Value 是该线程存的值。

Thread A Thread B └── threadLocals (Map) └── threadLocals (Map) ├── userContext → UserA ├── userContext → UserB └── dbConn → ConnA └── dbConn → ConnB ↑ 同一个 ThreadLocal 对象作为 Key，但 Value 完全独立

这就是为什么多线程访问同一个 ThreadLocal 实例，却拿到的是各自的值—— 存取操作实际上在自己线程的 Map 上进行，根本不存在竞争。

ThreadLocal 基本用法

JavaThreadLocal 基本用法
// 通常声明为 static final，整个应用只需一个 Key 对象
private static final ThreadLocal<SimpleDateFormat> DATE_FORMAT =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

public String format(Date date) {
    // 每个线程第一次调用 get() 时触发 withInitial 的 Supplier
    // 之后同一线程复用同一个 SimpleDateFormat 实例，无需加锁
    return DATE_FORMAT.get().format(date);
}
为什么 SimpleDateFormat 需要 ThreadLocal？它内部有 Calendar 字段，format() 会修改这个字段，是有状态且非线程安全的。用 ThreadLocal 给每个线程一个独立实例，比每次 new 节省对象分配，比 synchronized 减少竞争。

经典场景：Web 请求上下文传递

Web 应用中，一个 HTTP 请求从 Filter → Controller → Service → DAO 经过多层调用， 但它们都运行在同一个线程上（Servlet 线程模型）。 把当前用户信息存入 ThreadLocal，任何层都能直接取用，无需层层传参。

Java请求上下文 ThreadLocal 模式
// 上下文持有者（工具类）
public class UserContext {
    private static final ThreadLocal<User> CURRENT_USER = new ThreadLocal<>();

    public static void set(User user) {  CURRENT_USER.set(user); }
    public static User  get() {             return CURRENT_USER.get(); }

    // ⚠ 必须在请求结束时清除，否则线程复用时会读到上一次的脏数据
    public static void clear() {          CURRENT_USER.remove(); }
}

// Filter 中：请求进来时设置，响应后清理
@Override
public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) throws Exception {
    try {
        User user = parseToken((HttpServletRequest) req);
        UserContext.set(user);
        chain.doFilter(req, res);
    } finally {
        UserContext.clear();  // finally 保证必执行，哪怕出异常
    }
}

// Service 中：直接取，不需要方法参数传递
public void createOrder(OrderDTO dto) {
    User user = UserContext.get();  // 无需从 Controller 一路传参
    // ... 业务逻辑
}

并发陷阱：ThreadLocal 内存泄漏

这是 ThreadLocal 最著名的坑，特别在线程池场景下必须理解。

为什么会泄漏？

ThreadLocalMap 的 Entry 使用弱引用指向 ThreadLocal 键， 但 Value 是强引用。当外部不再持有 ThreadLocal 对象时，Key 会被 GC 回收变为 null， 但 Value 仍被 Thread → ThreadLocalMap → Entry → Value 这条强引用链持有，无法回收。

Thread（长生命周期，线程池中） └── ThreadLocalMap └── Entry ├── key → WeakRef → ThreadLocal（已被GC，key=null） └── value → 大对象（强引用！永远无法被GC） ← 泄漏点

反例：线程池中忘记 remove

Java内存泄漏场景
private static final ThreadLocal<byte[]> BUFFER =
    ThreadLocal.withInitial(() -> new byte[1024 * 1024]);  // 1MB

ExecutorService pool = Executors.newFixedThreadPool(10);
pool.submit(() -> {
    byte[] buf = BUFFER.get();
    // 处理完后没有 remove()
    // 线程池中的线程不会销毁，这 1MB 将永远驻留在该线程
});  // ❌ 10 个线程 × 1MB = 10MB 永久泄漏

正例：始终在 finally 中 remove

Java正确使用姿势
pool.submit(() -> {
    try {
        byte[] buf = BUFFER.get();
        process(buf);
    } finally {
        BUFFER.remove();  // ✅ 任务结束立即清理，防止泄漏和脏数据
    }
});
线程池场景两大必知原则：
1. 用完必须 remove()，否则内存泄漏且下一个任务可能读到脏数据。
2. 不要在 ThreadLocal 中存大对象（比如完整的 HTTP 请求体），即便 remove 了也会增大 GC 压力。

父子线程传递：InheritableThreadLocal

普通 ThreadLocal 只属于当前线程，子线程无法继承。 如果需要在创建子线程时把父线程的值复制过去，可以用 InheritableThreadLocal。

JavaInheritableThreadLocal 示例
private static final InheritableThreadLocal<String> TRACE_ID =
    new InheritableThreadLocal<>();

TRACE_ID.set("req-abc-123");

Thread child = new Thread(() -> {
    // 子线程继承了父线程创建时的快照值
    System.out.println(TRACE_ID.get());  // 输出: req-abc-123
});
child.start();
陷阱：线程池中 InheritableThreadLocal 不可靠。线程池里的线程是复用的，不是每次任务都新建。只有在线程创建时才会复制父线程的值，后续任务提交时不会再触发复制。如需在线程池中传递上下文，推荐使用阿里开源的 TransmittableThreadLocal（TTL）。

三种线程封闭方式对比

方式 原理 适用场景 主要风险
栈封闭 局部变量在线程私有栈帧中，不共享 方法内部使用的临时对象 引用逃逸后失效，编译器不报错
ThreadLocal Thread 内部 Map，同一 Key 不同线程有独立 Value 跨方法传递线程上下文（用户信息、DB连接等） 线程池中内存泄漏、脏数据
InheritableThreadLocal 子线程创建时复制父线程快照 链路追踪 TraceId 等需父子继承的场景 线程池中不可靠，需 TTL 替代

面试高频考点

  • ThreadLocal 为什么不会有线程安全问题？——每个线程操作的是自己 Thread 对象内部的 Map，完全独立，无争抢。
  • ThreadLocalMap 的 Key 为什么用弱引用？——防止外部不再需要 ThreadLocal 时，因 Map 持有强引用而无法 GC。但弱 Key + 强 Value 的组合导致了 Value 泄漏问题。
  • 如何防止 ThreadLocal 内存泄漏？——使用完毕后调用 remove()，尤其在线程池任务的 finally 块中。
  • Spring 的事务是怎么传播的？——通过 TransactionSynchronizationManager 内部的 ThreadLocal，把数据库连接绑定到当前线程，保证同一事务内的多次 DAO 调用使用同一连接。
  • ThreadLocal vs synchronized 怎么选？——如果数据本就不需要跨线程共享，用 ThreadLocal（空间换时间）；如果必须共享且需要协调，用 synchronized / Lock（时间换正确性）。

小结

线程封闭的本质是从根源消除共享，是并发编程中最干净的解法。 优先考虑能否让数据只属于一个线程，做不到时再引入同步机制。

  • 局部变量 = 天然栈封闭，但要防止引用逃逸
  • ThreadLocal 是跨方法传递线程私有状态的标准工具
  • 线程池场景下，ThreadLocal 用完必须 remove()
  • 父子线程传递推荐用 TransmittableThreadLocal（TTL）
  • Spring 事务、MDC 日志追踪等框架大量使用 ThreadLocal，理解原理有助于排查诡异 Bug

与其他知识点的联系

  • 关联知识点：线程安全的定义（原子性/可见性/有序性）：上一节讲的三个性质问题都建立在"多个线程访问同一份可变状态"的前提上，线程封闭直接消灭了这个前提——没有共享，原子性、可见性、有序性问题根本不会发生，这是比加锁更彻底的解法。
  • 关联知识点：不变性与 final：线程封闭和不变性是解决并发问题的两条不同路径：前者是"不共享"，后者是"共享但不可变"。两者常常配合使用，例如 ThreadLocal 里存放的对象往往也设计成不可变的，双重保险。
  • 关联知识点：JMM 与 happens-before：栈封闭的安全性本质上是 JVM 栈帧天然隔离带来的，不需要 happens-before 关系；而 ThreadLocal 的读写发生在同一线程内，天然满足程序顺序规则（happens-before 的第一条），所以不需要 volatile 或锁。
  • 关联知识点：线程池（Executor 框架）：ThreadLocal 内存泄漏和脏数据问题几乎只在线程池场景下才会暴露——因为线程池复用线程，若不在任务结束时 remove，ThreadLocalMap 会像"没清空的储物柜"一样越攒越多，这是线程池与 ThreadLocal 结合使用时必须理解的组合陷阱。
  • 关联知识点：Spring 事务与请求上下文：Spring 的 TransactionSynchronizationManager、Web MVC 的 RequestContextHolder、日志框架的 MDC，全部基于 ThreadLocal 实现"同一线程内跨层共享、线程间隔离"的效果，理解本节原理有助于排查"异步调用后拿不到用户上下文/事务失效"等经典 Bug。
  • 关联知识点：安全发布（下一节）：线程封闭是不发布（对象根本不出线程），而安全发布要解决的是对象确实需要被多个线程共享时，如何让引用和内部状态都正确可见——两者是同一个问题（共享状态的可见性）在不同场景下的两种应对策略。
