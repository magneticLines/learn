09. 不变性与 final

彻底消灭可变状态——不可变对象天生线程安全，无需任何同步。

① 并发 Bug 的根本原因

并发程序出错，几乎都指向同一个病灶：多个线程同时访问同一个可变状态， 而访问没有被充分同步。消灭其中任一要素，Bug 就消失了：

  • 消灭"多个线程"：线程封闭（上一节）
  • 消灭"可变状态"：不变性（本节）
  • 保证"充分同步"：锁、volatile、原子类（后续节）

不变性（Immutability）是三条路里最干净的一条——根本就没有状态变化， 就根本不存在竞态条件（Race Condition）、数据竞争（Data Race）。 你不需要加锁，不需要用 volatile，也不需要担心"看到了过期值"。

JMM 视角：Java 内存模型（JMM）要求对共享变量的读写必须通过同步动作才能建立 happens-before 关系。 不可变对象一旦构造完毕，其字段就再也不会改变，自然不存在"写"，也就不需要 happens-before 了。

② 什么是"真正的"不可变对象

很多人以为"加了 final 就不可变了"，这是最常见的误区。不可变对象需要满足三个条件：

  1. 对象创建后其状态不能被修改（没有 setter，也没有任何修改内部状态的方法）
  2. 所有字段都是 final 的
  3. 对象被正确构造（构造期间 this 引用没有逸出）
陷阱：final 字段 ≠ 字段引用的对象不可变
final List<String> list 只保证 list 引用本身不变， 但 list.add("x") 仍然合法。这种对象不是不可变对象， 而只是一个"引用不可变"的对象。

③ 反例 → 问题 → 正例

反例：以为加了 final 就安全

MutablePoint.java — 错误示范
public class MutablePoint {
    // final 只是防止引用被重新赋值，x/y 本身仍可改变
    public final int[] coords;

    public MutablePoint(int x, int y) {
        this.coords = new int[] {x, y};
    }
}

// 线程 A 和线程 B 同时操作同一个 point：
point.coords[0] = 10;  // 线程 A 写
int x = point.coords[0];  // 线程 B 读 —— 数据竞争！

数组是引用类型，final coords 只保证 coords 变量不会指向别的数组， 但数组元素本身完全可以被任意修改，多线程访问时依然存在竞态条件。

正例：真正的不可变点

ImmutablePoint.java — 正确示范
// 满足三个条件：状态不可变 + 全 final + 正确构造
public final class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
        // 构造结束前绝不把 this 传出去（不逸出）
    }

    public int getX() { return x; }
    public int getY() { return y; }

    // "修改"操作返回新对象，原对象不变
    public ImmutablePoint translate(int dx, int dy) {
        return new ImmutablePoint(x + dx, y + dy);
    }
}

任意多个线程同时读取同一个 ImmutablePoint，永远安全， 不需要任何同步。

④ final 的内存语义（JMM 保证）

final 在 JMM 里有特殊地位，远比你想象的重要。JMM 规定：

  • 写 final 字段的重排序规则：构造函数对 final 字段的写入， 不能被重排序到构造函数返回之后。也就是说，当其他线程看到这个对象引用时， final 字段已经完成初始化。
  • 读 final 字段的重排序规则：首次读对象引用和随后读该对象的 final 字段， 不能被重排序。
这意味着什么？
不可变对象的 final 字段，即使在没有同步的情况下发布（例如把引用直接存入普通变量）， 其他线程也能安全地读取到正确初始化后的值。 这是 final 独有的特权，普通字段没有这个保证。
FinalFieldVisibility.java — final 的可见性保证
public class FinalFieldVisibility {
    final int x;
    int y;  // 普通字段，无保证

    FinalFieldVisibility() {
        x = 1;
        y = 2;
    }
}

// 发布（无同步）
static FinalFieldVisibility obj;
// 线程 A：
obj = new FinalFieldVisibility();

// 线程 B：
if (obj != null) {
    int a = obj.x;  // JMM 保证一定看到 1（final 保证）
    int b = obj.y;  // 可能看到 0（普通字段，无保证）
}

⑤ 含可变引用的不可变类：防御性拷贝

如果不可变类的字段中包含可变对象的引用（如数组、集合）， 必须在构造时做防御性拷贝（Defensive Copy）， 同时 getter 也不能直接返回内部引用。

ImmutablePeriod.java — 防御性拷贝
import java.util.Date;

public final class ImmutablePeriod {
    private final Date start;
    private final Date end;

    public ImmutablePeriod(Date start, Date end) {
        // 防御性拷贝：不信任外部传入的 Date，立刻复制一份
        this.start = new Date(start.getTime());
        this.end   = new Date(end.getTime());
        if (this.start.after(this.end))
            throw new IllegalArgumentException("start after end");
    }

    public Date getStart() {
        // 返回副本，不暴露内部引用
        return new Date(start.getTime());
    }

    public Date getEnd() {
        return new Date(end.getTime());
    }
}
常见漏洞：只在构造时拷贝，但 getter 直接返回 this.start。 调用者拿到引用后执行 period.getStart().setTime(0)， 就悄悄修改了"不可变"对象内部。这是《Effective Java》第 50 条重点警告的问题。

⑥ 不可变对象 + volatile 实现原子更新

不可变对象有一个经典用法：将多个相关字段打包成一个不可变对象， 再用 volatile 引用持有它，就能在无锁的情况下保证"读到的一组数据是一致的"。

CachedFactorizer.java — 不可变对象 + volatile 的惯用法
// 把"上次输入"和"上次结果"打包，保证两者的一致性
@Immutable
class OneValueCache {
    private final BigInteger lastNumber;
    private final BigInteger[] lastFactors;

    public OneValueCache(BigInteger i, BigInteger[] factors) {
        lastNumber  = i;
        // 防御性拷贝
        lastFactors = Arrays.copyOf(factors, factors.length);
    }

    public BigInteger[] getFactors(BigInteger i) {
        if (lastNumber == null || !lastNumber.equals(i))
            return null;
        return Arrays.copyOf(lastFactors, lastFactors.length);
    }
}

@ThreadSafe
public class CachedFactorizer {
    // volatile 保证引用的可见性；不可变对象保证状态的一致性
    private volatile OneValueCache cache = new OneValueCache(null, null);

    public BigInteger[] service(BigInteger i) {
        BigInteger[] cached = cache.getFactors(i);
        if (cached != null) {
            return cached;
        }
        BigInteger[] factors = factor(i);
        // 整体替换引用，无锁，无竞态
        cache = new OneValueCache(i, factors);
        return factors;
    }
}

关键思路：读-修改-写被拆成"构造新对象"和"替换引用"两步， 任何线程读到的 cache 引用，要么是旧的完整状态，要么是新的完整状态， 绝不会看到"lastNumber 更新了但 lastFactors 还没更新"这种半截子状态。

⑦ 标准库中的不可变类

类备注
String最常用的不可变类，substring/concat 均返回新对象
Integer/Long/… 等基本类型包装类运算结果返回新实例
BigInteger/BigDecimal所有算术操作均返回新对象
LocalDate/LocalDateTime（Java 8+）新时间 API 设计为不可变，彻底取代线程不安全的 Date
List.of()/Map.of()（Java 9+）返回不可修改（unmodifiable）视图，注意≠深度不可变
注意：Collections.unmodifiableList() 和 List.of() 只是"不可修改视图"，不是真正的不可变——若底层集合被修改，视图也会变。 只有字段全为 final 且字段指向的对象也不可变，才算深度不可变。

⑧ 设计不可变类的最佳实践

✅ 声明类为 final（防止子类破坏不变性） ✅ 所有字段声明为 private final ✅ 构造时对可变引用做 防御性拷贝 ✅ getter 返回可变对象时也做 防御性拷贝 ✅ 不提供任何修改状态的方法（no setter） ✅ "修改"操作返回 新对象（函数式风格） ✅ 构造期间 不让 this 逸出 ⚠️ 字段 final ≠ 字段引用的对象不可变 ⚠️ unmodifiable 视图 ≠ 真正不可变

⑨ 面试高频考点

  • String 为什么设计成不可变？ 安全性（字符串常量池、类加载）、线程安全、hashCode 缓存三个原因。
  • final 能保证可见性吗？ 能，但仅限于 final 字段在构造函数中的初始化写入—— 这是 JMM 对 final 的特殊内存语义，普通字段没有这个保证。
  • 不可变对象一定线程安全吗？ 对象本身是线程安全的，但持有它的引用变量不一定—— 若引用变量是普通变量（非 volatile、非 final）， 其他线程读引用本身可能拿到 null 或过期值。
  • String 的 + 拼接为什么慢？ 每次 + 都创建新的 String 对象（不可变的代价）， 循环中应用 StringBuilder。

⑩ 小结

不可变性是并发安全的终极武器：没有状态变化，就没有竞态。 final 的内存语义由 JMM 保障，让不可变对象的字段在构造完成后对所有线程安全可见。 设计不可变类的关键在于"全 final 字段 + 防御性拷贝 + 不逸出 this"， 三者缺一不可。当需要"修改"时，返回新对象而不是改旧对象，这也是函数式编程的核心思想。
