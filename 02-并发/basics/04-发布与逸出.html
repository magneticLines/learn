04. 发布与逸出（this 逸出）

对象在构造完成之前被其他线程看到，是并发 Bug 中最隐蔽的一类——JVM 指令重排加上构造器竞态，足以让"看起来正确"的代码在生产环境崩溃。

① 什么是"发布"（Publishing）

发布指的是让一个对象的引用对当前作用域之外的代码可见。常见形式包括：

  • 把引用存入一个 public static 字段
  • 把引用传给外部方法（包括回调、监听器）
  • 把引用存入某个可被其他线程访问的集合
  • 在构造器里把 this 传出去（最危险的一种）

发布本身不是问题；问题在于在对象完全构造好之前就发布了——这就是"逸出（Escape）"。

为什么单独拿出来讲？
在上一节 happens-before 中我们学到：JVM 保证构造函数内的写操作对"正常获得引用的线程"可见。 但一旦 this 在构造器执行期间就被发布，这个保证就被打破了—— 其他线程可能拿到一个字段尚未初始化完毕的"半成品对象"。

② 直觉模型：半生不熟的对象

想象烤蛋糕的过程：面糊刚放进烤箱（构造器开始执行），你就把"蛋糕已做好"的消息广播出去（发布 this）。其他人跑来吃，却发现还是面糊。这就是 this 逸出的本质。

在 JVM 视角里，构造器执行期间，对象字段的写入对其他线程来说不保证按顺序可见。 更糟的是，指令重排可能让字段赋值顺序与源码顺序不同，因此拿到逸出引用的线程可能看到字段的默认值（0/null/false），或部分初始化的状态。

③ 经典反例：构造器里注册监听器

EventSource.java — 危险：this 在构造器里逸出
public class EventSource {
    private final String name;
    private final int[] data;

    public EventSource(EventBus bus) {
        // 第一步：注册到 bus — 此时 this 已被发布！
        bus.register(this);  // ← this 逸出

        // 第二步：字段初始化尚未完成
        this.name = "source-1";
        this.data = new int[1024];
    }

    public String getName() { return name; }
}

如果 bus.register(this) 内部立刻把 this 放到共享队列里，另一个线程就可能在 name / data 赋值之前访问它，读到 null 和 null。 即使 name / data 用了 final 修饰，逸出仍然打破了 final 字段的安全发布保证。

面试高频考点： "final 字段不是线程安全的吗？" — final 字段的安全性依赖于"构造器正常完成后引用才被发布"这一前提。 一旦 this 在构造器中逸出，final 保证失效。

④ 另一种逸出：匿名内部类隐式持有 this

匿名内部类（包括 Lambda 捕获外部类实例）会隐式持有外部类的 this。 如果在构造器里把匿名内部类的实例传出去，效果等同于直接传 this。

Server.java — 匿名内部类导致隐式 this 逸出
public class Server {
    private final int port;

    public Server(int port, ThreadPool pool) {
        this.port = port;
        // 匿名 Runnable 隐式捕获了外部的 this（Server 实例）
        pool.submit(new Runnable() {
            @Override
            public void run() {
                // 这里访问的 Server.this 可能还没构造完
                System.out.println("listening on " + port);
            }
        });
        // ... 后续字段初始化
    }
}

即使 Runnable 只访问了 port，只要匿名类实例被发布，外部类 this 就随之逸出。 如果 port 之后还有其他字段初始化，那些字段在任务执行时仍可能不可见。

⑤ 底层原因：JMM 与指令重排

从 JMM（Java 内存模型）角度理解这个问题：

构造器执行（单线程视角） 其他线程（可能的读取时序） ───────────────────────────── ───────────────────────────── bus.register(this) ← 发布 读取 name → null ← 看到默认值 this.name = "src" 读取 data → null ← 看到默认值 this.data = new int[1024]

JMM 不保证构造器内的写操作对"通过非安全路径获得引用的线程"有序可见。 更重要的是，JIT 编译器和 CPU 都可能对构造器内指令重排序， 使得字段赋值在 register(this) 调用之后才真正落地到主内存。

final 字段通过在构造器末尾插入 StoreStore 屏障 来保证可见性， 但这个屏障前提是：引用在构造器正常退出后才被发布。逸出让屏障提前失效。

⑥ 正例：工厂方法模式修复逸出

最通用的解法是工厂方法：把注册/发布动作移出构造器，等对象完全构造后再发布。

EventSource.java — 安全：工厂方法延迟发布
public class EventSource {
    private final String name;
    private final int[] data;

    // 构造器：只做字段初始化，绝不把 this 传出去
    private EventSource() {
        this.name = "source-1";
        this.data = new int[1024];
    }

    // 工厂方法：构造完成后再注册，引用此时已完全初始化
    public static EventSource create(EventBus bus) {
        EventSource source = new EventSource();
        bus.register(source);  // ← 发布在构造完成之后
        return source;
    }

    public String getName() { return name; }
}

工厂方法保证了：当 bus.register(source) 执行时，构造器已完整执行完毕， JMM 的 构造器 happens-before 后续引用读取 保证生效，所有字段对其他线程安全可见。

⑦ 线程启动的修复：延迟 submit

Server.java — 安全：start() 方法延迟提交任务
public class Server {
    private final int port;
    private final ThreadPool pool;

    // 构造器只保存依赖，不启动任何后台任务
    public Server(int port, ThreadPool pool) {
        this.port = port;
        this.pool = pool;
    }

    // 由外部在确认构造完成后显式调用
    public void start() {
        pool.submit(() ->
            System.out.println("listening on " + port)
        );
    }
}

// 调用端
Server srv = new Server(8080, pool);
srv.start();  // 构造完成后再启动后台任务，引用已安全发布

⑧ 反例 vs 正例对比

场景 危险做法 安全做法
注册监听器 构造器内 bus.register(this) 工厂方法：构造完后再 register
启动后台线程 构造器内 new Thread(this).start() 提供 start() 方法，外部显式调用
匿名内部类 构造器内把匿名类实例发布给共享结构 用 Lambda/方法引用替代，或移出构造器
静态字段发布 构造器执行中途赋值给 public static 字段 静态初始化块或工厂方法完成后赋值

⑨ 安全发布的四种通用手段

《Java 并发编程实战》总结了让对象被安全发布的四种方式，满足任意一种即可：

  1. 在静态初始化函数中初始化引用（JVM 初始化阶段的内置锁保证）
  2. 将引用保存到 volatile 字段或 AtomicReference 中
  3. 将引用保存到某个被正确加锁保护的字段中
  4. 将引用保存到 final 字段中（前提：构造器不逸出）
记忆口诀：静态 / volatile / 锁 / final。 只要发布动作走其中一条路，接收方就能看到一个完整初始化的对象。

⑩ 并发陷阱速查

  • 构造器里调用可覆盖方法：子类覆盖方法可能读取尚未初始化的字段，this 同样处于逸出状态
  • Spring Bean 的 @PostConstruct：Spring 保证在所有依赖注入完成后才调用 @PostConstruct，比构造器内注册安全
  • ThreadLocal 不能防逸出：ThreadLocal 只隔离"存储"，若构造器把 this 传到了别的线程，ThreadLocal 无效
  • 双重检查锁（DCL）：如果单例构造器里逸出了 this，DCL 的 volatile 保证也救不了你
  • this 传入方法不一定逸出：只要该方法不把引用存储到可被其他线程访问的位置，就是局部使用，不构成逸出
