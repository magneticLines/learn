06. Semaphore 信号量

用"令牌桶"控制并发度——许可证计数器背后的操作系统哲学与 Java 实践。

① 为什么需要信号量

synchronized 和 ReentrantLock 解决的是互斥问题：同一时刻只允许一个线程进入临界区。 但现实中更常见的约束是：同时最多允许 N 个线程并发访问某资源。

典型场景：

  • 数据库连接池最多持有 10 条连接，超出请求必须等待
  • 限流：下载服务同时最多 5 个并发传输，防止带宽耗尽
  • 将无界并发压缩成有界并发（把任意多线程映射到固定大小的资源池）

互斥锁是 N=1 的特殊信号量。信号量把"许可证数量"参数化，是互斥锁的广义版本。

操作系统溯源：Semaphore 由 Dijkstra 在 1965 年提出（THE 操作系统论文）， 定义了两个原语 P（proberen，尝试减一，对应 acquire） 和 V（verhogen，增一，对应 release）。 Java 的 java.util.concurrent.Semaphore 是同一思想的高层封装。

② 核心模型：令牌桶

把 Semaphore 想象成一个装有 N 枚令牌的桶：

┌─────────────────────────────────────┐ │ Semaphore(permits = 3) │ │ │ │ bucket: [ 🪙 🪙 🪙 ] │ │ │ │ acquire() → 取走一枚，桶空则阻塞 │ │ release() → 放回一枚，唤醒等待者 │ └─────────────────────────────────────┘ Thread-A ──acquire()──▶ [ 🪙 🪙 ] ✓ 进入 Thread-B ──acquire()──▶ [ 🪙 ] ✓ 进入 Thread-C ──acquire()──▶ [ ] ✓ 进入 Thread-D ──acquire()──▶ BLOCKED（等待） Thread-A ──release()──▶ [ 🪙 ] Thread-D 被唤醒

③ API 速查

方法语义阻塞？
acquire()申请 1 个许可，不足则阻塞（可响应中断）是
acquire(n)申请 n 个许可，全部满足才返回是
acquireUninterruptibly()申请 1 个许可，忽略中断是
tryAcquire()尝试获取，立即返回 true/false否
tryAcquire(timeout, unit)限时等待超时前
release()归还 1 个许可否
release(n)归还 n 个许可否
availablePermits()当前剩余许可数（仅参考，非精确）否
drainPermits()取走所有剩余许可，返回数量否

④ 构造参数：公平 vs 非公平

Java
// 非公平（默认）：新来线程可能插队，吞吐更高
Semaphore unfair = new Semaphore(3);

// 公平：严格 FIFO，避免饥饿，但吞吐略低
Semaphore fair   = new Semaphore(3, true);

背后是 AQS：Semaphore 内部继承 AbstractQueuedSynchronizer， permits 存储在 AQS 的 state 字段中， 公平/非公平对应 AQS 两种 tryAcquireShared 实现。

⑤ 实战示例：数据库连接池

用 Semaphore 为一个简化版连接池添加并发控制：

Java — BoundedConnectionPool.java
public class BoundedConnectionPool {

    private final Semaphore         semaphore;
    private final Queue<Connection> pool;

    public BoundedConnectionPool(int size) {
        // 许可数 == 池容量，超出部分阻塞等待
        semaphore = new Semaphore(size, true);
        pool      = new ArrayDeque<>();
        for (int i = 0; i < size; i++)
            pool.add(createConnection());
    }

    public Connection acquire() throws InterruptedException {
        // 先拿令牌（不足则阻塞），再取连接对象
        semaphore.acquire();
        return pollConnection();  // 此时一定有空闲连接
    }

    public void release(Connection conn) {
        returnConnection(conn);  // 先归还对象
        semaphore.release();       // 再释放令牌，唤醒等待者
    }
}
设计要点：先 acquire() 再取对象、先归还对象再 release()， 保证"有令牌 → 一定有对象"的不变式，避免令牌与对象数量失配。

⑥ 并发陷阱

陷阱 1：release() 在 finally 外——许可泄漏

Java — 反例（许可永久泄漏）
semaphore.acquire();
doWork();          // 若此处抛异常 ↓
semaphore.release(); // 永远不会执行！许可数永久 -1
Java — 正例（finally 保证释放）
semaphore.acquire();
try {
    doWork();
} finally {
    semaphore.release();  // 无论正常/异常，必定执行
}

陷阱 2：release() 比 acquire() 多——许可膨胀

Semaphore 不绑定线程：任何线程都可以调用 release()， 即使它从未调用过 acquire()。这与 ReentrantLock 只有持有者才能解锁完全不同。

Java — 反例（许可越界膨胀）
Semaphore sem = new Semaphore(2);
// 某线程不小心多调了一次 release
sem.release();  // permits 变为 3！超出初始容量
// 后续可能同时有 3 个线程进入，超出资源限制

正例：用封装隐藏 Semaphore（如上一节的连接池）， 外部只暴露 acquire/release 业务方法，内部保证对称调用。

陷阱 3：tryAcquire() 返回 false 后仍执行受保护逻辑

Java — 反例
semaphore.tryAcquire();  // 返回值被忽略！
accessResource();        // 可能在没有许可的情况下进入
Java — 正例
if (semaphore.tryAcquire(100, TimeUnit.MILLISECONDS)) {
    try {
        accessResource();
    } finally {
        semaphore.release();
    }
} else {
    handleTimeout();  // 降级：返回错误 / 重试 / 排队
}

陷阱 4：availablePermits() 用于并发决策

Java — 反例（TOCTOU 竞态）
if (semaphore.availablePermits() > 0) {  // 检查时有许可
    semaphore.acquire();                   // 获取时可能已被抢走 → 仍会阻塞
}

availablePermits() 仅供监控/日志，不能用来做"先判断再获取"的并发决策， 正确做法是直接调用 tryAcquire()（原子地检查并获取）。

⑦ 进阶：用 Semaphore 实现有界阻塞集合

《Java 并发编程实战》第 5 章经典示例——将任意无界集合包装成有界阻塞集合：

Java — BoundedHashSet.java（JCIP 清单 5-14 简化版）
public class BoundedHashSet<T> {

    private final Set<T>      set;
    private final Semaphore   sem;

    public BoundedHashSet(int bound) {
        // 用 synchronizedSet 保护集合自身线程安全
        set = Collections.synchronizedSet(new HashSet<>());
        sem = new Semaphore(bound);
    }

    public boolean add(T o) throws InterruptedException {
        sem.acquire();          // 先占位（桶满则阻塞）
        boolean wasAdded = false;
        try {
            wasAdded = set.add(o);
            return wasAdded;
        } finally {
            // 若元素已存在（add 返回 false），需归还许可
            if (!wasAdded)
                sem.release();
        }
    }

    public boolean remove(Object o) {
        boolean wasRemoved = set.remove(o);
        if (wasRemoved)
            sem.release();  // 元素真正被移除，才归还许可
        return wasRemoved;
    }
}
精妙之处：Semaphore 的许可数代表集合"剩余容量"， 而非"已用容量"——这与连接池恰好相反，但逻辑一致： 许可代表"可用资源槽"，acquire 消耗槽，release 归还槽。

⑧ 与其他同步器横向对比

同步器核心语义计数方向可重置？典型场景
Semaphore 控制并发度（N 个许可） 双向（可增可减） 是（release 可超初始值） 连接池、限流、有界容器
CountDownLatch 一次性等待 N 个事件 单向减（到 0 开门） 否 启动门、结束门
CyclicBarrier N 个线程互相等待 计到 N 后重置 是（自动循环） 分阶段并行计算
ReentrantLock 互斥（N=1 信号量） 0/1 是（可重入） 临界区保护

⑨ 面试高频考点

  • Semaphore 与 ReentrantLock 的关系：互斥锁是 permits=1 的非公平信号量，但 Semaphore 不可重入、不绑定线程。
  • 底层实现：基于 AQS，state 存储许可数，tryAcquireShared / tryReleaseShared 以 CAS 原子修改。
  • 为什么 release() 不绑定线程：信号量用于资源计数，而非互斥所有权；可以用于生产者-消费者（生产者 release，消费者 acquire）。
  • 公平模式代价：需维护 CLH 队列 FIFO 顺序，每次 acquire 先检查队列头，吞吐低于非公平模式约 10~20%（视竞争强度）。
  • permits 为 0 的信号量：初始 permits=0 可实现"先 acquire 后等 release"的一次性通知，类似 CountDownLatch(1)，但可重复使用。
常见误区：认为 Semaphore 能替代锁保证原子性。 Semaphore 只控制并发数量，不提供互斥语义—— 如果 N=1 且需要原子性操作，仍应使用 ReentrantLock 而非 Semaphore， 因为 Semaphore 无重入保护，递归调用 acquire 会死锁。

⑩ 小结

Semaphore 核心思想 ├── 令牌桶模型：N 枚许可 = N 个并发配额 ├── acquire() → 消耗许可（不足则阻塞/超时/返回 false） ├── release() → 归还许可（任何线程均可调用） ├── 底层：AQS 共享模式，state = 剩余许可数 ├── 公平/非公平：构造参数决定，影响吞吐 vs 饥饿 └── 必须在 finally 中 release()，否则许可永久泄漏

与其他知识点的联系

  • 与 ReentrantLock 的关系：本文①、⑨节都提到互斥锁是 permits=1 的特殊信号量，但深层区别在于所有权语义——ReentrantLock 有"持有者"概念，支持重入和 Condition，unlock 必须由 lock 的同一线程调用；Semaphore 不认线程，任何线程都能 release，这正是陷阱二的根源，也是它能天然支撑"生产者 release、消费者 acquire"这种跨线程协作模式的原因，而 Lock 做不到。
  • AQS 共享模式与 CountDownLatch 的对比：Semaphore 和 CountDownLatch 底层都继承 AQS 并使用共享模式（tryAcquireShared/tryReleaseShared），区别是 state 的变化方向——Semaphore 的 state 可增可减（对应本文⑧节"计数方向：双向"），CountDownLatch 只能单向递减到 0 后不可逆。这也解释了为什么 Semaphore 能被 release 到超过初始值（陷阱二），而 CountDownLatch 的 countDown 在 0 之后调用是安全的无操作。
  • 令牌桶与限流算法：本文②节的"令牌桶"类比不只是教学比喻——生产环境的限流组件（如 Guava RateLimiter、Sentinel）本质上就是"令牌桶算法"的时间维度扩展版：Semaphore 控制的是"同时并发数"，令牌桶限流器控制的是"单位时间内的速率"，两者可以组合使用（先限速率，再限并发）。
  • 与 BlockingQueue 实现有界容器的殊途同归：本文⑦节用 Semaphore 包装 HashSet 实现有界集合，而上一节 BlockingQueue 是"原生"就有有界语义的容器——二者是同一个问题（限制并发访问的资源数量）在"内部维护容量控制逻辑"与"外部叠加一层信号量控制"两种不同设计层次上的解法，理解这点有助于判断什么时候该选现成的有界容器、什么时候该用 Semaphore 自己包装。
  • Dijkstra 的操作系统理论到 Java 并发包：本文①节提到的 P/V 原语不仅是 Semaphore 的来源，也是操作系统课程里进程同步的经典模型（哲学家就餐问题等经典并发问题都用信号量求解）；Java 把这套 50 多年前的理论直接落地成一个可用的类，是"并发理论 → 工程实现"的典范案例，面试官有时会追问信号量的历史来源来考察知识面广度。
  • 面试高频对比："Semaphore 能不能实现互斥锁的功能"是常见追问——permits=1 时表面上能做到互斥，但因为不可重入（同一线程递归 acquire 会自锁死锁）且无所有权检查（任何线程可 release 导致临界区被破坏），生产代码永远不能用 Semaphore 替代 ReentrantLock 做真正的互斥保护。
