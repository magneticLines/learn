04. 阻塞队列 BlockingQueue 与生产者-消费者

并发程序中最优雅的解耦手段——用队列把"生产速度"和"消费速度"彻底分离。

1. 问题起点：无协调的生产者-消费者

想象一条流水线：上游工人（生产者）焊接零件，下游工人（消费者）喷漆。两者速度不一致， 不加协调会产生两类经典并发问题：

  • 消费者空转（busy-wait）：队列为空时消费者不断轮询，白白消耗 CPU。
  • 生产者撑爆内存：队列无界且消费慢，堆内存被积压任务耗尽，触发 OOM。

手动用 wait/notifyAll 实现需要正确处理锁、条件判断、虚假唤醒…… BlockingQueue 把这一切封装进接口契约，让你只需关注"放"与"取"的业务逻辑。

设计哲学：BlockingQueue 是"条件等待"的高阶封装。内部用 ReentrantLock + 两个 Condition（notEmpty / notFull）实现，比 synchronized+wait 更精确地唤醒对应的等待方。

2. BlockingQueue 接口：四组操作的对称设计

接口定义了四种行为模式，同一逻辑（入队/出队）对"队满/队空"时的处理方式不同：

操作 抛异常 返回特殊值 阻塞 超时阻塞
入队 add(e) offer(e) put(e) offer(e, t, unit)
出队 remove() poll() take() poll(t, unit)

生产者-消费者模型中最常用的是 put 和 take：队满时 put 自动挂起生产者线程，队空时 take 自动挂起消费者线程， 无需任何额外同步代码。

3. 五大实现类：选哪个？

实现类底层结构容量公平性适用场景
ArrayBlockingQueue 循环数组 有界（必填） 可选公平锁 背压控制、内存敏感
LinkedBlockingQueue 单向链表 可选（默认 Integer.MAX_VALUE） 否 高吞吐，入队/出队用独立锁
SynchronousQueue 无存储 0（直接交付） 可选 直接握手传递，CachedThreadPool 用它
PriorityBlockingQueue 堆 无界 否 按优先级消费，注意饥饿
DelayQueue 堆 + Delayed 无界 否 定时任务、缓存过期
陷阱：LinkedBlockingQueue 的"默认无界"
不传容量参数时，容量为 Integer.MAX_VALUE（约 21 亿）。 如果消费者比生产者慢，任务会无限堆积，最终 OOM。 线上环境务必显式指定容量，例如 new LinkedBlockingQueue<>(1000)。

4. 原理深挖：ArrayBlockingQueue 如何实现阻塞

以 ArrayBlockingQueue 为例看源码骨架。它用一把 ReentrantLock + 两个 Condition 实现读写互斥与条件等待：

ArrayBlockingQueue
├── final ReentrantLock lock
├── final Condition notEmpty ← take() 在此等待
├── final Condition notFull ← put() 在此等待
├── Object[] items ← 循环数组
├── int putIndex ← 下次写入位置
└── int takeIndex ← 下次读取位置
ArrayBlockingQueue — put / take 核心逻辑（简化）Java
// -------- put：队满则等待 --------
public void put(E e) throws InterruptedException {
    checkNotNull(e);
    final ReentrantLock lock = this.lock;
    lock.lockInterruptibly();          // 可响应中断，优于 lock()
    try {
        while (count == items.length)   // 用 while 而非 if，防虚假唤醒
            notFull.await();           // 释放锁并挂起
        enqueue(e);                    // 写入数组，putIndex 推进
    } finally {
        lock.unlock();
    }
}

private void enqueue(E x) {
    items[putIndex] = x;
    if (++putIndex == items.length) putIndex = 0;  // 循环
    count++;
    notEmpty.signal();               // 唤醒一个等待的消费者
}

// -------- take：队空则等待 --------
public E take() throws InterruptedException {
    final ReentrantLock lock = this.lock;
    lock.lockInterruptibly();
    try {
        while (count == 0)              // 同样用 while 防虚假唤醒
            notEmpty.await();
        return dequeue();
    } finally {
        lock.unlock();
    }
}

private E dequeue() {
    E x = (E) items[takeIndex];
    items[takeIndex] = null;          // help GC
    if (++takeIndex == items.length) takeIndex = 0;
    count--;
    notFull.signal();                // 唤醒一个等待的生产者
    return x;
}
为什么条件判断用 while 而非 if？
虚假唤醒（Spurious Wakeup）：JVM 规范允许线程在没有 notify 的情况下从 await 返回。用 while 重新检查条件可以正确应对虚假唤醒， 这是所有"条件等待"的标准写法。

5. LinkedBlockingQueue：双锁拆分提升吞吐

ArrayBlockingQueue 入队和出队共用同一把锁，高并发下互相阻塞。 LinkedBlockingQueue 用两把独立的锁（putLock 和 takeLock）分别保护队尾和队头，入队和出队可以真正并发执行，吞吐更高。

LinkedBlockingQueue
├── ReentrantLock putLock + Condition notFull
├── ReentrantLock takeLock + Condition notEmpty
├── AtomicInteger count ← 原子计数，跨两把锁共享
├── Node<E> head ← 虚头节点
└── Node<E> last

代价是链表节点频繁分配，GC 压力比数组高。高吞吐选 LinkedBlockingQueue， 内存敏感或需要公平性选 ArrayBlockingQueue。

6. 完整示例：文件扫描 + 内容搜索流水线

经典的"多生产者-多消费者"场景：一个线程扫描目录产出文件路径，多个消费者线程 并发读取文件内容并搜索关键词。队列把两阶段完全解耦。

FileCrawler（生产者）+ FileSearcher（消费者）Java
public class IndexingService {

    // 有界队列：控制背压，防止扫描速度远超搜索速度时 OOM
    private static final int CAPACITY = 100;
    private static final Path POISON = Path.of("__POISON__");  // 毒丸

    public void start(Path root, String keyword, int consumers)
            throws InterruptedException {

        BlockingQueue<Path> queue = new LinkedBlockingQueue<>(CAPACITY);

        // 生产者：扫描目录，遇到文件就 put
        Thread crawler = new Thread(() -> {
            try {
                crawl(root, queue);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                // 每个消费者放一颗毒丸，保证所有消费者都能退出
                for (int i = 0; i < consumers; i++)
                    try { queue.put(POISON); } catch (InterruptedException ignored) {}
            }
        }, "crawler");

        // 消费者：取出文件路径，读取内容搜索关键词
        ExecutorService pool = Executors.newFixedThreadPool(consumers);
        for (int i = 0; i < consumers; i++) {
            pool.submit(() -> {
                try {
                    while (true) {
                        Path file = queue.take();     // 队空则阻塞
                        if (file == POISON) break;  // 毒丸 = 退出信号
                        search(file, keyword);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }

        crawler.start();
        pool.shutdown();
        pool.awaitTermination(1, TimeUnit.HOURS);
    }

    private void crawl(Path dir, BlockingQueue<Path> q) throws InterruptedException {
        // Files.walk 遍历目录，只 put 普通文件
        try (Stream<Path> s = Files.walk(dir)) {
            s.filter(Files::isRegularFile)
             .forEach(p -> {
                 try { q.put(p); } // 队满则阻塞，实现背压
                 catch (InterruptedException e) {
                     Thread.currentThread().interrupt();
                 }
             });
        } catch (IOException e) { /* 日志 */ }
    }

    private void search(Path file, String keyword) {
        // 实际搜索逻辑省略
    }
}

7. 停止信号：毒丸（Poison Pill）模式

如何优雅地通知消费者"没有更多任务了"？常见有两种方案：

  • 毒丸对象：放入一个哨兵元素，消费者取到后退出。简单，适合消费者数量固定时。
  • 标志位 + interrupt：设置 volatile boolean shutdown， 消费者检查标志后退出。但需要额外打断 take() 的阻塞。
毒丸数量必须等于消费者线程数！
只有每个消费者都能"喝到"一颗毒丸才能全部退出。如果有 N 个消费者， 生产者必须放 N 颗毒丸。否则某些消费者将永远阻塞在 take()。

8. 并发陷阱全解

陷阱一：用 if 而非 while 检查条件

反例 — 虚假唤醒导致数据错乱Java (bad)
// 手写版本，if 判断条件
synchronized void put(E e) throws InterruptedException {
    if (count == capacity)   // ❌ 虚假唤醒后不重新检查
        wait();
    items[putIndex++] = e;     // 可能在队满时写入，越界或覆盖
}
正例 — while 循环重检条件Java (good)
synchronized void put(E e) throws InterruptedException {
    while (count == capacity)  // ✅ 每次唤醒后重新检查
        wait();
    items[putIndex++] = e;
    notifyAll();
}

陷阱二：put/take 与 interrupt 的交互

put 和 take 都声明了 throws InterruptedException， 意味着它们响应中断。线程池关闭时会调用 shutdownNow 中断所有线程，此时阻塞在 take 的消费者线程会被唤醒并抛出异常。 捕获到 InterruptedException 后必须重置中断标志：

InterruptedException 的正确处理Java
try {
    E item = queue.take();
    process(item);
} catch (InterruptedException e) {
    // ❌ 错误：直接 return 或吞掉异常，中断标志丢失
    // return;

    // ✅ 正确：恢复中断标志，让外层代码有机会响应
    Thread.currentThread().interrupt();
}

陷阱三：offer 的非阻塞误用

offer 失败时任务静默丢失Java (bad)
// ❌ offer 在队满时返回 false，任务直接丢弃，没有任何日志或重试
boolean ok = queue.offer(task);
// 没检查 ok！
offer 加超时或主动拒绝策略Java (good)
// ✅ 方案一：带超时的 offer，超时后执行降级逻辑
boolean enqueued = queue.offer(task, 500, TimeUnit.MILLISECONDS);
if (!enqueued) {
    log.warn("队列满，任务被丢弃: {}", task);
    metrics.increment("task.dropped");
}

// ✅ 方案二：用 ThreadPoolExecutor 的 RejectedExecutionHandler
new ThreadPoolExecutor(4, 8, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(200),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 队满时调用方线程直接执行，天然背压
);

9. SynchronousQueue：零缓冲直连

SynchronousQueue 没有内部存储。每次 put 都必须有一个线程 同时在 take，否则生产者阻塞（反之亦然）。 它的语义是"线程间直接握手传递"，而不是"缓冲"。

Executors.newCachedThreadPool() 内部就用了它：任务提交后若没有空闲线程， 立即创建新线程；若有空闲线程则直接交付，不做任何排队。

CachedThreadPool 的隐患：SynchronousQueue 不限流， 在任务爆发时线程数无上限增长，可能创建数万个线程导致 OOM 或系统崩溃。 线上服务用 ThreadPoolExecutor + 有界 LinkedBlockingQueue 更安全。

10. BlockingQueue 与 ThreadPoolExecutor 的关系

ThreadPoolExecutor 的第五个参数就是一个 BlockingQueue<Runnable>， 队列的选型直接决定线程池的行为策略：

队列类型效果风险
SynchronousQueue 提交直接给线程，核心满了就扩容 线程无上限，OOM
LinkedBlockingQueue（无界） 任务永远排队，maxPoolSize 无效 内存 OOM，任务积压
LinkedBlockingQueue（有界） 排队满后扩线程，扩到 max 再触发拒绝策略 最推荐
ArrayBlockingQueue（有界） 同上，内存更稳定 推荐

11. 面试高频考点

  • ArrayBlockingQueue vs LinkedBlockingQueue 的区别？
    数组 vs 链表；单锁 vs 双锁；必须指定容量 vs 默认无界。
  • BlockingQueue 为什么不允许 null？
    poll() 返回 null 表示"队列为空"，允许 null 元素会产生歧义。
  • put 和 offer 的区别？
    put 无限阻塞；offer 立即返回 false 或带超时阻塞。生产者速度可控时优先 put + 有界队列实现背压。
  • 如何优雅停止消费者？
    毒丸模式（N 个消费者放 N 颗毒丸）或 interrupt + volatile 标志位。
  • SynchronousQueue 有什么实际用途？
    CachedThreadPool、Exchanger 的底层思路、两个线程需要"同步握手"时。

12. 与其他知识点的联系

  • wait/notify 与 Condition：本文源码里的 notEmpty.await() / notFull.signal() 本质是 Lock + Condition 对 synchronized + wait/notify 的替代方案——区别在于 Condition 能拆分出多个等待队列（notEmpty、notFull 各自独立），而内置锁的 wait/notifyAll 只有一个等待集合，只能广播唤醒，精度不够。这也是"为什么 while 判断防虚假唤醒"这一条规则在两种写法里都成立的原因。
  • 与 CountDownLatch / Semaphore 的关系：三者底层都是"条件不满足就阻塞，条件变化后唤醒"的模式，但语义不同：BlockingQueue 关心的是"容器空/满"，CountDownLatch 关心"计数是否归零"，Semaphore 关心"许可证是否充足"。理解 BlockingQueue 内部这套 ReentrantLock+Condition 的写法，后续学 Semaphore/CountDownLatch 的 AQS 实现会更容易类比。
  • ThreadPoolExecutor 的任务队列：本文第 10 节已经点出 BlockingQueue 是线程池的第五个构造参数，队列类型直接决定线程池的"排队 vs 扩容 vs 拒绝"策略——这是承接到 exec/04-ThreadPoolExecutor.html 七大参数、以及"为什么不推荐 Executors 静态工厂"的关键桥梁，因为 Executors.newFixedThreadPool 默认用的就是无界 LinkedBlockingQueue，正是本文陷阱一直在强调的隐患。
  • 生产者-消费者模式的分布式延伸：本文的毒丸模式、有界队列实现背压，本质上和消息队列（Kafka、RabbitMQ）里的"消费者拉取+确认"机制是同一套思想的单机版——单机内存队列解决线程间协调，分布式消息队列解决进程/机器间协调，背压（backpressure）的概念是共通的。
  • 中断机制的复用：put/take 声明 throws InterruptedException，和后面 exec/07-interrupt中断.html 要讲的"响应中断的标准写法"完全一致——捕获后必须 Thread.currentThread().interrupt() 恢复标志，这是贯穿整个并发包的统一约定，不是 BlockingQueue 独有的规则。
  • 面试高频对比："手写 wait/notify 版生产者消费者和用 BlockingQueue 的区别"常被追问到"为什么标准库要拆两个 Condition"，答案正是"精确唤醒 vs 广播唤醒"的性能差异——避免了 notifyAll 把不该被唤醒的线程也叫醒再重新检查条件的浪费。
一句话总结：BlockingQueue 是并发编程中"解耦速度差异"的标准工具—— 用有界队列实现背压，用 put/take 屏蔽繁琐的 wait/notify，用毒丸或 interrupt 优雅停止。 选型上：内存敏感用 ArrayBlockingQueue，高吞吐用有界 LinkedBlockingQueue， 零缓冲直传用 SynchronousQueue。
