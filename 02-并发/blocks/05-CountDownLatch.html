05. CountDownLatch 倒计时门闩

让一个或多个线程等待，直到其他线程完成一组操作——一次性的同步屏障，不可重置。

为什么需要 CountDownLatch？

在并发程序里，等待若干事件全部发生之后再继续是极其常见的需求：

  • 主线程启动 N 个工作线程，等它们全部就绪后再一起开跑（赛跑模型）
  • 主线程把任务拆成 N 份并行处理，等全部子任务完成后汇总结果
  • 集成测试中，等待多个依赖服务启动完毕再开始测试

在 JDK 1.5 之前，实现上述逻辑需要手写 wait/notifyAll + 计数器，容易出错。 CountDownLatch 把这个模式封装成一个语义明确的工具类： 内部维护一个整数计数器，每次 countDown() 减 1，减到 0 时所有阻塞在 await() 的线程同时放行。

设计哲学：CountDownLatch 是"一次性"的——计数归零后不能重置。 这个设计来自 AQS（AbstractQueuedSynchronizer）的共享锁机制：归零是不可逆状态， 简化了实现，也明确了语义。如果需要可重置的屏障，应该用 CyclicBarrier。

底层原理：AQS 共享锁

CountDownLatch 内部有一个私有静态类 Sync，继承自 AbstractQueuedSynchronizer（AQS）。

CountDownLatch
└── Sync extends AbstractQueuedSynchronizer
    ├── state = count（构造时传入）
    ├── tryAcquireShared: state==0 ? 放行 : 阻塞
    └── tryReleaseShared: CAS(state, s, s-1)，归零时返回 true 唤醒所有等待线程

await() 调用 acquireSharedInterruptibly(1)， 如果 state != 0 则进入 AQS 的 CLH 等待队列挂起。 countDown() 调用 releaseShared(1)，通过 CAS 将 state 减 1； 当 state 变为 0 时，tryReleaseShared 返回 true， AQS 以广播方式唤醒队列中所有等待线程（共享锁语义）。

对比独占锁：独占锁 release 后只唤醒队头一个等待者； CountDownLatch 用共享锁，归零后会传播唤醒，直到遇到第一个独占节点为止—— 这就是 N 个 await() 线程能同时放行的根本原因。

核心 API 一览

方法语义线程安全
new CountDownLatch(int count) 初始化计数器，count 必须 ≥ 0 —
await() 阻塞直到 count == 0 或线程被中断 可在多线程调用

两种经典使用模式

模式一：主线程等待所有子任务完成（任务完成门）

主线程把工作拆给 N 个子线程，每个子线程完成后 countDown()， 主线程 await() 汇总。

TaskCompletionDemo.java
import java.util.concurrent.CountDownLatch;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class TaskCompletionDemo {

    public static void main(String[] args) throws InterruptedException {
        int N = 5;
        // 计数器初始化为子任务数量
        CountDownLatch latch = new CountDownLatch(N);
        ExecutorService pool = Executors.newFixedThreadPool(N);

        for (int i = 0; i < N; i++) {
            final int id = i;
            pool.submit(() -> {
                try {
                    doWork(id);
                } finally {
                    // 必须放 finally，确保异常时也能 countDown，否则主线程永远阻塞
                    latch.countDown();
                }
            });
        }

        // 主线程在此阻塞，直到 5 个子任务全部调用 countDown()
        latch.await();
        System.out.println("所有子任务完成，开始汇总结果");
        pool.shutdown();
    }

    static void doWork(int id) throws InterruptedException {
        Thread.sleep(100 * id);
        System.out.println("Task " + id + " done");
    }
}

模式二：多线程同时起跑（发令枪门）

反过来，用一个 count=1 的 Latch 作"发令枪"：所有工作线程启动后先 await()， 等主线程做好准备后 countDown()，所有线程同时冲出。 常用于并发测试：尽可能让多线程在同一时刻竞争同一资源，暴露竞态条件。

StartingGunDemo.java
public class StartingGunDemo {

    public static void main(String[] args) throws InterruptedException {
        int N = 8;
        // 发令枪：count=1，主线程控制
        CountDownLatch startGun = new CountDownLatch(1);
        // 完成门：count=N，等所有选手跑完
        CountDownLatch finishLine = new CountDownLatch(N);

        for (int i = 0; i < N; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    System.out.println("选手 " + id + " 就位");
                    // 等发令枪
                    startGun.await();
                    race(id);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    finishLine.countDown();
                }
            }).start();
        }

        Thread.sleep(500); // 确认所有选手就位
        System.out.println("预备——跑！");
        startGun.countDown();  // 一声令下，所有线程同时放行

        finishLine.await();
        System.out.println("比赛结束");
    }

    static void race(int id) throws InterruptedException {
        Thread.sleep((long)(Math.random() * 200));
        System.out.println("选手 " + id + " 到达终点");
    }
}

并发陷阱与反例分析

陷阱 1：countDown() 未放 finally，子任务异常导致主线程死锁

死锁风险：如果 countDown() 写在业务代码之后而不是 finally 块里，一旦业务抛出未检查异常， countDown() 不会执行，计数器永远不归零，主线程永远阻塞。
反例 — countDown 不在 finally
pool.submit(() -> {
    doWork();         // 若此处抛出 RuntimeException
    latch.countDown(); // ← 永远不会执行！主线程死等
});
正例 — countDown 放 finally
pool.submit(() -> {
    try {
        doWork();
    } finally {
        latch.countDown(); // 无论正常还是异常，都保证执行
    }
});

陷阱 2：await() 无超时，线程永久阻塞

生产代码应使用带超时的 await(long, TimeUnit)，超时后检查返回值决定是否继续。

带超时的安全写法
boolean ok = latch.await(30, TimeUnit.SECONDS);
if (!ok) {
    // 超时：记录告警、取消任务、降级处理……
    handleTimeout();
}

陷阱 3：把 CountDownLatch 当 CyclicBarrier 用

CountDownLatch 是一次性的，计数归零后不可重置，再次 await() 会立即返回。 如果需要"跑完一轮再跑下一轮"的周期性屏障，必须换成 CyclicBarrier。

反例 — 错误地复用 CountDownLatch
CountDownLatch latch = new CountDownLatch(3);
// 第一轮：正常工作
latch.countDown(); latch.countDown(); latch.countDown();
latch.await();  // OK，放行

// 第二轮：再次调用 countDown 无效，await 立即返回
latch.countDown();  // state 已为 0，无操作
latch.await();      // 立即返回！根本没等待

陷阱 4：count 传入 0

new CountDownLatch(0) 是合法的，await() 会立即返回，不阻塞。 这在条件初始化（如无子任务时）可以利用，但要确保语义符合预期，不要因笔误把初始值算错。

内存可见性保证

JDK 规范明确：countDown() happens-before await() 返回。 也就是说，子线程在 countDown() 之前写入的所有数据，在主线程 await() 返回之后可以安全读取， 无需额外的 volatile 或 synchronized。

可见性示例
List<String> results = new CopyOnWriteArrayList<>();
CountDownLatch latch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    final int id = i;
    new Thread(() -> {
        results.add("result-" + id);  // 写入结果
        latch.countDown();               // happens-before 下面的 await() 返回
    }).start();
}

latch.await();
// 此处读取 results，三个子线程的写入均已可见，不会有脏读
results.forEach(System.out::println);

CountDownLatch vs CyclicBarrier vs Phaser

特性CountDownLatchCyclicBarrierPhaser
可重置 否（一次性） 是（自动重置） 是（多阶段）
等待方向 N 个事件 → 1+ 个等待者 N 个线程互相等 任意注册/注销
屏障动作 无 有（barrierAction） 有（onAdvance）
参与方数量 固定（构造时确定） 固定 动态注册/注销
典型场景 主线程等子任务、并发压测起跑 多线程分阶段协作 Fork/Join 风格分阶段任务

面试高频考点

  • CountDownLatch 和 join() 有什么区别？ join() 只能等待线程本身结束，且只能在线程对象上调用；CountDownLatch 更灵活，可以等待任意事件（不止线程结束），且支持线程池场景。
  • 为什么 CountDownLatch 不可重置？ 基于 AQS state 的 CAS 操作是单向递减到 0，归零即广播唤醒，没有"复位"逻辑。 这是故意设计成一次性的，语义更简单、不易误用。
  • await() 和 countDown() 都是线程安全的吗？ 是的。countDown() 通过 AQS 的 CAS 保证原子性；await() 通过 AQS 的队列机制保证阻塞和唤醒正确。
  • countDown() 多调用几次会怎样？ count 已为 0 时再调用 countDown() 是无操作（state 不会变成负数），不会抛异常。
  • await() 响应中断吗？ 是的，await() 会响应中断并抛出 InterruptedException，应该正确处理而不是吞掉。

小结

记忆口诀：CountDownLatch = "倒计时闹钟"。构造时设定几点钟（count）， 任何线程都可以拨快时间（countDown），到零时所有等待者（await）同时醒来，且闹钟不能重拨。

核心要点：countDown() 必须放 finally；生产代码用带超时的 await(timeout, unit)； 需要重复使用的屏障请改用 CyclicBarrier。

与其他知识点的联系

  • AQS 共享锁模式：本文第 2 节讲的 tryAcquireShared / tryReleaseShared 是 AQS 共享模式的典型实现，与 ReentrantLock（独占模式）、Semaphore（同样是共享模式，但 state 双向增减）、ReentrantReadWriteLock 的读锁共用同一套 AQS 队列机制。理解了 CountDownLatch 的 tryReleaseShared 归零后广播唤醒，再看 Semaphore 的 release 逻辑会发现两者代码结构几乎一样，只是语义不同。
  • 与 CyclicBarrier 的本质差异：本文第 7 节的对比表已经列出关键区别，但更深一层看：CountDownLatch 是"等待者与计数者分离"（任意线程都能 countDown，等待者被动），而 CyclicBarrier 是"参与者身份统一"（每个参与线程既是等待者也是计数者），这是理解下一节 CyclicBarrier 为什么能"自动重置"的关键前提——因为它的语义是"大家互等"而不是"有人倒计时"。
  • 与线程池 awaitTermination 的联系：本文两个示例都配合 ExecutorService 使用，而 ExecutorService.awaitTermination() 内部也是"等待某个条件发生"的模式，只是 JDK 没有直接暴露一个 CountDownLatch 给你操作，而是把等待逻辑封装在了线程池内部——本质思路相通，都是"多线程协作完成后再往下走"。
  • happens-before 与内存可见性体系：本文第 6 节强调的"countDown() happens-before await() 返回"，与 volatile 写读、synchronized 解锁加锁、线程 start()/join() 共同构成 JMM 的 happens-before 规则集合，是 Java 并发大厦的地基——面试中常要求横向列出这些规则并举例。
  • 分布式协调工具的单机雏形：ZooKeeper 的 Barrier / DoubleBarrier recipe 就是把 CountDownLatch/CyclicBarrier 的思想搬到跨进程场景：本地用 AQS 的 CAS+队列协调线程，分布式场景用 ZK 的临时节点+Watcher 协调进程，问题模型是同构的。
  • 面试高频对比："CountDownLatch 和 join() 的区别"本文已给出，但更深追问是"为什么 join 不能实现发令枪模式"——因为 join 只能等一个线程"结束"，而 CountDownLatch 可以在线程运行过程中的任意时刻 countDown，不要求线程终止，这是两者适用场景分裂的根本原因。
