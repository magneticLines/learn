07. CyclicBarrier 循环屏障

让一组线程在同一个检查点集合等待，全部就位后再一起放行——还能无限复用。

为什么需要 CyclicBarrier？

想象一场接力赛：只有全部队员都到达交接区，裁判才会鸣枪，所有人同时出发进入下一棒。 这就是 屏障（Barrier） 语义——多个线程分头干活，在某个"检查点"处互等， 全员到齐之后再一起继续。

用 Thread.join() 或 CountDownLatch 也能实现"等齐再走"， 但它们都是一次性的：用完即毁，下一轮需要重新创建对象。 CyclicBarrier 的"Cyclic"正是为了解决这个问题——屏障可以循环复用， 多轮迭代计算（如并行模拟、矩阵分块运算）无需反复构造同步器。

CountDownLatch vs CyclicBarrier
CountDownLatch：1 个协调者倒计数，N 个等待者被动等待；一次性，不可重置。
CyclicBarrier：N 个参与者互等，大家地位平等；可复用，每轮自动重置。

核心 API 一览

Java — CyclicBarrier 构造与使用
// 构造1：指定参与者数量
CyclicBarrier barrier = new CyclicBarrier(4);

// 构造2：指定参与者数量 + 屏障动作（最后一个到达的线程执行）
CyclicBarrier barrier2 = new CyclicBarrier(4,
    () -> System.out.println("所有线程就绪，开始下一阶段"));

// 参与者调用 await() 表示"我到了"
try {
    barrier.await();            // 阻塞直到全部线程到达
    barrier.await(5, TimeUnit.SECONDS);  // 带超时版本
} catch (BrokenBarrierException e) {
    // 屏障已损坏，必须处理！
}

// 查询状态
barrier.isBroken();   // 是否已损坏
barrier.getNumberWaiting();  // 当前等待线程数
barrier.reset();      // 强制重置（危险！会触发 BrokenBarrierException）

内部原理：Generation + ReentrantLock

JDK 源码中，CyclicBarrier 用一个 ReentrantLock + Condition 实现等待/唤醒，用一个内部类 Generation 表示"当前这一轮"。 每次屏障被突破或重置，都会创建新的 Generation 实例， 这就是"Cyclic"得以成立的关键——旧一轮的等待者不会干扰新一轮。

CyclicBarrier
├── lock: ReentrantLock <-- 保护所有状态
├── trip: Condition <-- 等待/唤醒队列
├── parties: int <-- 总参与者数
├── count: int <-- 当前轮还差多少人到达
├── generation: Generation <-- 当前代（含 broken 标志）
└── barrierCommand: Runnable <-- 可选的屏障动作

当第 N 个线程调用 await()，count 减为 0： 先执行 barrierCommand（如果有），然后调用 nextGeneration() 创建新的 Generation 并 signalAll() 唤醒全部等待者， 屏障自动为下一轮复位。

JDK 源码简化版 — dowait 核心逻辑
private int dowait(boolean timed, long nanos)
        throws InterruptedException, BrokenBarrierException {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        final Generation g = generation;
        if (g.broken) throw new BrokenBarrierException();

        int index = --count;
        if (index == 0) {  // 最后一个到达
            if (barrierCommand != null) barrierCommand.run();
            nextGeneration();  // 重置 count、创建新 Generation、signalAll
            return 0;
        }
        // 不是最后一个：在 Condition 上等待
        for (;;) {
            if (!timed) trip.await();
            else if (nanos > 0L) nanos = trip.awaitNanos(nanos);
            if (g.broken) throw new BrokenBarrierException();
            if (g != generation) return index;  // 新一代，正常返回
            if (timed && nanos <= 0L) { breakBarrier(); throw new TimeoutException(); }
        }
    } finally {
        lock.unlock();
    }
}

典型用法：并行迭代计算

以并行粒子模拟为例：每轮迭代让 N 个线程各自更新自己负责的粒子， 全部更新完毕后统一合并结果，再进入下一轮。

ParticleSimulation.java — 多轮迭代场景
public class ParticleSimulation {

    private final int THREADS = 4;
    private final double[] positions = new double[1000];  // 粒子位置
    private final AtomicReference<double[]> snapshot = new AtomicReference();

    private final CyclicBarrier barrier = new CyclicBarrier(THREADS,
        // 屏障动作：最后一个线程到达后执行，此时所有线程都停在 await()，无竞争
        () -> snapshot.set(positions.clone())
    );

    class Worker implements Runnable {
        private final int from, to;
        Worker(int from, int to) { this.from = from; this.to = to; }

        @Override
        public void run() {
            try {
                for (int round = 0; round < 100; round++) {
                    updateParticles(from, to);    // 第一步：各自更新分片

                    barrier.await();              // 等待全部线程完成更新
                    // ↑ 屏障动作（快照）在此处由最后一个到达的线程执行

                    renderParticles(snapshot.get(), from, to);  // 第二步：基于快照渲染

                    barrier.await();              // 等待全部线程完成渲染，进入下一轮
                }
            } catch (InterruptedException | BrokenBarrierException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
屏障动作的执行时机：barrierCommand 由最后一个到达的线程在持有锁时同步执行， 此时其余线程都挂起在 Condition.await()，不存在并发竞争。 因此屏障动作里可以安全地合并/汇总各线程的中间结果，无需额外同步。

并发陷阱与反例

陷阱 1：忽略 BrokenBarrierException

只要任意一个参与者超时、被中断或抛出异常，屏障就会进入损坏（broken）状态， 随后所有正在等待或将来调用 await() 的线程都会收到 BrokenBarrierException。 如果你把它 catch 后静默吞掉，程序会在屏障损坏后继续以不一致的状态执行。

反例 — 静默吞异常
try {
    barrier.await(2, TimeUnit.SECONDS);
} catch (Exception e) {
    // 危险：屏障已损坏，后续逻辑拿到的数据是不完整的
    // 什么都不做 → 继续向下执行 → 数据一致性破坏
}
mergeResults();  // 此时某些分片可能根本没完成！
正例 — 正确处理屏障损坏
try {
    barrier.await(2, TimeUnit.SECONDS);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    return;  // 中断：退出任务
} catch (BrokenBarrierException e) {
    // 屏障已损坏（超时/其他线程中断导致）
    cancelTask();  // 回滚/取消/上报错误，绝不继续执行
    return;
} catch (TimeoutException e) {
    // 屏障超时（此线程超时后屏障已损坏，其他线程将收到 BrokenBarrierException）
    cancelTask();
    return;
}

陷阱 2：屏障动作中抛出异常

如果 barrierCommand 抛出了未捕获的异常，屏障同样会进入损坏状态， 所有等待线程收到 BrokenBarrierException。 务必在屏障动作内部处理异常，不要让它向上传播。

反例 vs 正例 — 屏障动作异常处理
// 反例：屏障动作抛出异常 → 损坏屏障 → 所有等待线程报错
CyclicBarrier bad = new CyclicBarrier(4,
    () -> mergeResults());  // mergeResults 可能抛 NPE !

// 正例：在动作内部捕获，屏障状态始终健康
CyclicBarrier good = new CyclicBarrier(4, () -> {
    try {
        mergeResults();
    } catch (Exception e) {
        mergeError.set(e);  // 记录错误，不向外抛
    }
});

陷阱 3：参与者数量不匹配

如果实际调用 await() 的线程数少于构造时声明的 parties， 屏障会永远等下去（死锁）。常见场景：

  • 线程池复用时某线程意外终止，新任务提交前屏障就已经"缺人"
  • 动态跳过某些分支导致部分线程根本不调用 await()
死锁陷阱：务必保证每轮迭代中 parties 个线程都必然调用 await()。 如果某线程有可能不调用（如提前 return），要在 finally 块中处理，或使用带超时的 await。

陷阱 4：误用 reset()

手动调用 barrier.reset() 会使当前等待的线程全部收到 BrokenBarrierException。 这不是"优雅重置"，而是一次强制中断——只应用于明确知道需要放弃当前轮并重新开始的场景， 绝不能在正常迭代中使用。

CyclicBarrier vs CountDownLatch vs Phaser

特性 CountDownLatch CyclicBarrier Phaser
可复用 否（一次性） 是（自动重置） 是（多阶段）
等待角色 等待者 ≠ 计数者 参与者平等互等 参与者平等互等
参与者数量 构造后不变 构造后不变 可动态注册/注销
屏障动作 无 有（可选 Runnable） 有（可继承覆写）
适用场景 一次性启动/等待完成 固定线程数多轮迭代 动态参与者/多阶段流水

实战：并行数据导入 + 汇总校验

4 个线程各自导入 1/4 的数据行，全部导入完毕后由屏障动作触发一次完整性校验， 校验通过后再一起进入提交阶段。

ParallelImporter.java
public class ParallelImporter {

    private static final int N = 4;
    private final AtomicBoolean valid = new AtomicBoolean(false);

    // 屏障动作：所有线程完成导入后，做一次全局完整性校验
    private final CyclicBarrier barrier = new CyclicBarrier(N, () -> {
        try {
            boolean ok = validateIntegrity();
            valid.set(ok);
        } catch (Exception e) {
            valid.set(false);  // 异常不向外抛，避免损坏屏障
        }
    });

    public void run(List<Row> rows) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(N);
        int chunk = rows.size() / N;

        for (int i = 0; i < N; i++) {
            final List<Row> slice = rows.subList(i * chunk,
                i == N - 1 ? rows.size() : (i + 1) * chunk);
            pool.submit(() -> {
                try {
                    importSlice(slice);   // 阶段 1：导入分片
                    barrier.await();    // 等待全部完成 → 触发校验

                    if (valid.get()) commitSlice(slice);  // 阶段 2：校验通过再提交
                    else rollbackSlice(slice);
                } catch (InterruptedException | BrokenBarrierException e) {
                    Thread.currentThread().interrupt();
                    rollbackSlice(slice);
                }
            });
        }
        pool.shutdown();
        pool.awaitTermination(30, TimeUnit.SECONDS);
    }
}

面试高频考点

  • CyclicBarrier 和 CountDownLatch 的核心区别？
    CountDownLatch 是一次性的，计数只减不增，且等待者不参与倒计数； CyclicBarrier 可复用，参与者地位平等，全员到达后自动重置。
  • BrokenBarrierException 什么时候会触发？
    任意等待线程被中断、超时，或屏障动作抛出异常，屏障即进入 broken 状态， 当前及后续 await() 全部抛出此异常。
  • 屏障动作由哪个线程执行？
    最后一个调用 await() 的线程，在持有内部锁时同步执行，执行完毕后才唤醒其余线程。
  • 如果某线程不调用 await() 会怎样？
    屏障永远等不到 parties 个到达者，所有等待线程无限阻塞——典型死锁。 用带超时的 await 或 finally 块保证一定会调用。
  • CyclicBarrier 底层用了什么锁？
    ReentrantLock + Condition，不是 AQS 的直接子类（而 CountDownLatch 直接继承 AQS）。

小结

CyclicBarrier 的核心价值在于：让固定数量的线程在同一检查点互相等待， 并支持多轮循环复用，非常适合"分阶段并行计算"场景。 关键设计点：

  • 用 Generation 对象隔离每一轮，使自动重置安全可靠
  • 屏障动作在所有线程停止后、唤醒前执行，天然无竞争
  • 屏障损坏具有传染性——任何参与者出问题，全部参与者都需要响应
  • 参与者数量固定是最大限制；若需动态增减，升级到 Phaser

与其他知识点的联系

  • 为什么不是 AQS 子类：本文第 8 节已指出 CyclicBarrier 底层用 ReentrantLock + Condition 而非直接继承 AQS，这与 CountDownLatch/Semaphore 都基于 AQS 共享模式形成鲜明对比——原因在于 CyclicBarrier 的语义（N 个参与者互等 + 可选屏障动作 + 自动重置为新一代）比"简单的计数增减"复杂得多，用 Lock+Condition 手写状态机比硬套 AQS 的 tryAcquireShared 更清晰，这是一个"何时该用 AQS，何时该自己撑起同步逻辑"的设计取舍案例。
  • 与 Fork/Join 框架的关系：本文"并行迭代计算"场景（分片计算→屏障同步→下一轮）与 Fork/Join 框架解决的问题高度相似，但思路不同：Fork/Join 用递归分治+工作窃取自动调度任务，CyclicBarrier 需要你手动控制固定数量的线程和轮次。如果分阶段计算的每一轮都是同构任务且轮数不固定，通常 Fork/Join 或 Phaser 更合适；如果是"固定线程数、固定阶段结构"的场景，CyclicBarrier 更直观。
  • Generation 机制与 CAS 版本号思想的呼应：本文第 3 节的 Generation 对象每轮创建新实例来隔离新旧等待者，这与 AQS 内部用 state 版本变化区分"哪一轮"的思路类似，也类似 AtomicStampedReference 用版本号解决 ABA 问题的思路——本质都是"给状态打版本戳，避免新旧数据混淆"这一通用套路的不同实现。
  • BrokenBarrierException 与异常传播设计：屏障损坏后所有参与者都会收到异常，这种"一人失败、全员感知"的设计与 CompletableFuture 链式调用中异常沿链传播、下游全部走 exceptionally 分支的思路是同构的——都是"分布式/多线程协作中，局部失败必须能够被全局感知，不能被静默吞掉"这一原则的具体体现。
  • ZooKeeper DoubleBarrier 的分布式对应：ZooKeeper 官方 recipe 中的 DoubleBarrier（进入屏障+离开屏障）就是把本文的循环屏障思想搬到跨进程集群场景：本地用 Generation+Condition 协调线程重置，分布式场景用临时子节点计数+Watcher 通知，都是"固定数量参与者互等后统一放行"这一模型的不同实现层级。
  • 面试高频对比："CyclicBarrier 和 CountDownLatch 分别适合什么场景"本文第 8 节已给出直接答案，但更本质的追问是"为什么 CyclicBarrier 能重复使用而 CountDownLatch 不能"——根源在于 CountDownLatch 只有"计数减到 0"这一个不可逆的终态，而 CyclicBarrier 通过 Generation 把"一轮"的生命周期显式建模出来，天然支持轮转。
