Part II 构建模块 · 容器

01. 同步容器及其问题

Vector、Hashtable 与 Collections.synchronizedXxx —— 粗粒度锁的设计哲学、线程安全的假象，以及复合操作的致命陷阱。

1. 问题从哪里来：普通容器为何不安全

Java 早期的 ArrayList、HashMap 等集合类，内部状态（数组、链表节点）没有任何同步保护。多线程并发修改时，典型危害包括：

  • 数据丢失：两个线程同时 add，都读到相同的 size，写入同一个槽位，后者覆盖前者。
  • 无限循环：JDK 7 及以下 HashMap 在并发扩容时链表成环，导致 get 永不退出（CPU 飙升经典案例）。
  • 数组越界 / NPE：一个线程正在扩容，另一个线程读到旧引用，越界访问。
根本原因：可见性 + 原子性双重缺失。读-改-写不是一个原子操作，且没有 happens-before 保证，多核下各自的缓存行会产生竞态。

2. 同步容器的设计思路：给每个方法加锁

JDK 1.0 时代的解法非常直接：把每个公开方法都用 synchronized 修饰，以 this（对象本身）作为内置锁。

同步容器家族
├── Vector ← ArrayList 的同步版，add/get/remove 全加 synchronized
├── Hashtable ← HashMap 的同步版，同样方法级锁
└── Collections.synchronized*
    ├── synchronizedList(list)
    ├── synchronizedMap(map)
    └── synchronizedSet(set)

2.1 Vector 的锁实现

Vector 的核心方法在 JDK 源码中大致如下（简化）：

Vector.java（JDK 源码简化）Java
public class Vector<E> extends AbstractList<E> {

    protected Object[] elementData;
    protected int elementCount;

    // 每个方法都加 synchronized —— 锁是 this 本身
    public synchronized boolean add(E e) {
        ensureCapacityHelper(elementCount + 1);
        elementData[elementCount++] = e;
        return true;
    }

    public synchronized E get(int index) {
        if (index >= elementCount)
            throw new ArrayIndexOutOfBoundsException(index);
        return elementData(index);
    }

    public synchronized int size() {
        return elementCount;
    }
}

2.2 Collections.synchronizedList 的包装思路

Collections.synchronizedList 采用装饰器模式：把原有 List 包一层，每个方法委托给内部 List，同时加锁。

Collections.java（源码简化）Java
static class SynchronizedList<E> implements List<E> {
    final List<E> list;
    final Object mutex;   // 锁对象，默认 = this

    SynchronizedList(List<E> list) {
        this.list = list;
        mutex = this;
    }

    public E get(int index) {
        synchronized (mutex) { return list.get(index); }
    }

    public boolean add(E e) {
        synchronized (mutex) { return list.add(e); }
    }

    // ... 其余方法同理 ...

    // ⚠️ iterator() 没有加锁！调用方需要自己同步
    public Iterator<E> iterator() {
        return list.iterator();  // 未加锁！
    }
}

3. 同步容器真正保证了什么

同步容器通过方法级锁，保证了以下两件事：

  • 单个方法的原子性：一次 get 或一次 add 不会被其他线程打断。
  • 可见性：synchronized 退出时会将工作内存刷回主内存，进入时会重新从主内存读取，满足 happens-before。
类比：把超市货架（容器）换成一间有保安的储藏室——每次只允许一个人进入拿货或放货，单次操作安全了。但如果你的业务逻辑需要"先看一眼有没有货，没有才补货"，两步之间保安不会帮你站着等，这就是后面要讲的陷阱。

4. 核心陷阱：复合操作不是原子的

同步容器只保护单次方法调用。当业务逻辑需要多个方法调用组成一个逻辑单元时，锁会在两次调用之间释放，另一个线程可以趁虚而入。

陷阱一：先检查后执行（Check-Then-Act）

❌ 反例：看似安全，实则有竞态Java
Vector<String> list = new Vector<>();

// 线程 A 和线程 B 同时执行 getLast()
public String getLast(Vector<String> list) {
    int lastIndex = list.size() - 1;  // ① 调用 size()，锁获取后立即释放
    // ⚠️ 此处锁已释放！线程 B 可能调用 deleteLast() 删掉最后一个元素
    return list.get(lastIndex);      // ② 再次获取锁，但 lastIndex 可能已越界！
}

public void deleteLast(Vector<String> list) {
    int lastIndex = list.size() - 1;
    list.remove(lastIndex);           // 删掉最后一个
}
// 结果：getLast 抛出 ArrayIndexOutOfBoundsException！
✅ 正例：在客户端手动加锁，持有锁跨越整个复合操作Java
public String getLast(Vector<String> list) {
    synchronized (list) {             // 锁住 list 对象本身（与 Vector 内置锁相同）
        int lastIndex = list.size() - 1;
        return list.get(lastIndex);    // size() 和 get() 在同一个锁块内，原子
    }
}

public void deleteLast(Vector<String> list) {
    synchronized (list) {
        int lastIndex = list.size() - 1;
        list.remove(lastIndex);
    }
}
关键原则：客户端加锁必须使用与同步容器内部相同的锁对象。Vector 的内置锁是 this（即 Vector 实例本身），所以写 synchronized (list) 才能与 Vector 内部锁互斥。若用其他对象作锁，两个锁之间没有关系，复合操作仍然不安全。

陷阱二：迭代时被并发修改（ConcurrentModificationException）

同步容器的迭代器是快速失败（fail-fast）的：迭代期间若容器被修改（modCount 变化），下一次 next() 就会抛出 ConcurrentModificationException。

❌ 反例：迭代被并发修改打断Java
List<String> syncList = Collections.synchronizedList(new ArrayList<>());

// 线程 A：迭代
for (String item : syncList) {      // iterator() 未加锁
    process(item);                    // 线程 B 在这里 add/remove → CME ！
}
✅ 正例：手动同步整个迭代块Java
synchronized (syncList) {           // 整个迭代持有锁，阻止并发修改
    for (String item : syncList) {
        process(item);
    }
}
// 代价：迭代期间其他线程无法 add/get，吞吐量下降
隐蔽的迭代：toString()、hashCode()、equals()、containsAll()、addAll() 等方法在内部都会触发迭代。把同步容器放进日志语句 log.debug("list={}", list) 都可能因此抛 CME，这是高并发系统中非常隐蔽的 bug。

陷阱三：若不存在则添加（put-if-absent）

❌ 反例：Hashtable 也救不了你Java
Hashtable<String, Connection> cache = new Hashtable<>();

public Connection getConnection(String key) {
    if (!cache.containsKey(key)) {   // ① 锁获取、检查、释放
        // ⚠️ 此处锁已释放！两个线程可能同时到达 ② 并各自建立连接
        cache.put(key, createConnection(key));  // ② 建立多余连接，资源泄漏
    }
    return cache.get(key);
}
✅ 正例：客户端加锁保护整个复合操作Java
public Connection getConnection(String key) {
    synchronized (cache) {
        if (!cache.containsKey(key)) {
            cache.put(key, createConnection(key));
        }
        return cache.get(key);
    }
}
// 更好的方案：直接换用 ConcurrentHashMap.computeIfAbsent()（下一节）

5. 性能瓶颈：粗粒度锁的代价

同步容器的锁粒度极粗——一把锁管整个容器。这意味着：

  • 读读互斥：两个线程同时读 get() 也要串行，因为 get 也是 synchronized。
  • 锁竞争升级：高并发下大量线程阻塞排队，上下文切换开销远超业务本身。
  • 迭代期间全部阻塞：若对迭代加外部锁，整个迭代过程中其他读写线程全部等待。
场景同步容器行为影响
10 线程并发 get() 全部串行，一次只有一个线程读 读吞吐量 = 单线程
1 线程写 + 9 线程读 写时 9 线程全部阻塞 读线程长时间饥饿
迭代 10 万条记录 外部加锁后，期间所有 add/get 排队 明显延迟其他操作

6. 何时该用，何时该换

场景推荐方案原因
单线程或线程封闭 ArrayList / HashMap 无竞争，零开销
低并发、简单加锁即可 Collections.synchronizedXxx 最简单，改造成本低
高并发读多写少 ConcurrentHashMap / CopyOnWriteArrayList 分段锁 / 读无锁，吞吐量数量级提升
需要原子复合操作 ConcurrentHashMap.computeIfAbsent 等 内置原子复合操作，无需客户端加锁
发布-订阅、生产-消费 BlockingQueue 专门为生产者-消费者设计，内置阻塞语义
实战建议：Vector 和 Hashtable 在新代码中几乎不应出现，它们是 Java 1.0 的遗留设计。Collections.synchronizedXxx 偶尔用于遗留代码改造（给现有集合加锁），但要牢记：必须把所有复合操作包进同步块，迭代也不例外。

7. 面试高频考点

Q1：Vector 是线程安全的，为什么还会出现 ArrayIndexOutOfBoundsException？

因为 size() 和 get() 是两次独立加锁，中间另一个线程可能 remove() 了元素。方法级别线程安全 ≠ 复合操作线程安全。

Q2：Collections.synchronizedList 的 iterator() 为什么不加锁？

迭代器可能要遍历整个集合，若自动加锁则持有时间不可控，会极大损害并发性。JDK 把这个决策权交给调用方，文档明确要求："使用返回的 iterator 时，用户必须手动在 syncList 上同步。"

Q3：客户端加锁时锁对象用错了会怎样？

❌ 常见错误：锁对象不一致Java
List<String> list = Collections.synchronizedList(new ArrayList<>());
private final Object myLock = new Object();

public void addIfAbsent(String s) {
    synchronized (myLock) {          // ❌ myLock ≠ list（SynchronizedList 内部的 mutex）
        if (!list.contains(s))
            list.add(s);              // 与其他线程直接调用 list.add() 之间没有互斥
    }
}

两把锁互不干扰，复合操作的原子性依然没有保证。

Q4：Hashtable 和 ConcurrentHashMap 的本质区别是什么？

Hashtable 用一把锁锁住整张表；ConcurrentHashMap（JDK 8+）用 CAS + 对每个桶头节点加锁，并发度从 1 提升到接近桶数量，详见下一节。

小结

  • 同步容器通过方法级 synchronized 解决了单次操作的可见性与原子性问题。
  • 但复合操作（check-then-act、迭代、put-if-absent）依然需要客户端手动加锁，且锁对象必须与容器内部锁一致。
  • 一把粗粒度锁导致读读串行，高并发下性能很差。
  • 新代码应优先选择 java.util.concurrent 包中的并发容器，它们既提供原子复合操作，又有更好的吞吐量。
