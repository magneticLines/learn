03. CopyOnWriteArrayList

写时复制（Copy-On-Write）—— 以空间换时间的无锁读并发容器，彻底解决「读写互斥」导致的性能瓶颈。

为什么需要 CopyOnWriteArrayList？

并发场景下使用普通 ArrayList 会崩，而 Collections.synchronizedList 虽然线程安全， 却把所有读操作也加了锁——读写完全串行，高并发读的场景下性能极差。

生活类比：图书馆里有一本热门参考书，所有人来查阅都要等前一个人翻完才能进馆——这就是 synchronizedList 的做法。 但如果每次有人要修改这本书时，馆员先悄悄复印一份新版本改好，再原子替换上架，那么其他读者始终拿着旧版本自由翻阅，无需等待——这就是 Copy-On-Write。

核心矛盾：在「读多写少」场景中，读操作为写操作的锁等待付出了不必要的代价。 CopyOnWriteArrayList 用「每次写都复制一份新数组」彻底解耦了读与写。

内部结构一览

JDK 源码（简化）核心字段只有两个：

CopyOnWriteArrayList.java（JDK 源码简化）
public class CopyOnWriteArrayList<E> implements List<E> {

    // 写操作专用锁，读操作完全不需要它
    final transient ReentrantLock lock = new ReentrantLock();

    // volatile 保证写完替换后，所有读线程立即可见新数组引用
    private volatile Object[] array;

    // 读：直接拿数组快照，无锁
    public E get(int index) {
        return elementAt(getArray(), index);  // 无锁！
    }

    // 写：加锁 → 复制 → 修改 → 替换引用
    public boolean add(E e) {
        synchronized (lock) {
            Object[] es = getArray();
            int len = es.length;
            // 关键：复制出新数组，长度 +1
            Object[] newEs = Arrays.copyOf(es, len + 1);
            newEs[len] = e;
            // volatile 写，发布新数组引用给所有读线程
            setArray(newEs);
            return true;
        }
    }
}
写操作流程：
┌─ 加锁（ReentrantLock）
│ ├─ 读取当前 array 引用（快照）
│ ├─ Arrays.copyOf → 新数组
│ ├─ 在新数组上修改
│ └─ setArray(newArray) ← volatile 写，刷新主内存
└─ 解锁

读操作流程：
└─ 直接读 volatile array 引用 → 无锁访问元素

为什么读操作不加锁也线程安全？

关键在于 Java 内存模型（JMM）对 volatile 的 happens-before 保证：

  • 写线程对 array 字段的 volatile 写，happens-before 所有后续读线程对 array 字段的读。
  • 读线程拿到的是一个完整的、不可变的数组快照——这个数组一旦被设置到 array 字段，就再也不会被修改（写操作总是复制出新数组再替换）。
  • 因此读线程即使读的是"旧版本"数组，那也是一个合法的完整状态，不存在半初始化的情况。
类比 Git：每次 commit 都产生一个不可变的 snapshot，旧 snapshot 永远合法。 读线程就像 checkout 某个历史 commit，始终拿到完整且一致的状态。

实战：事件监听器列表

CopyOnWriteArrayList 最经典的用途是监听器/观察者列表： 注册/注销（写）远少于触发通知（读遍历），且遍历时不希望被其他线程的写操作打断。

EventBus.java — 监听器列表的并发安全实现
public class EventBus {

    // 监听器列表：读多（每次发事件都遍历）写少（注册/注销）
    private final CopyOnWriteArrayList<EventListener> listeners
            = new CopyOnWriteArrayList<>();

    public void register(EventListener l) {
        listeners.addIfAbsent(l);  // 防重复注册，COW 原子操作
    }

    public void unregister(EventListener l) {
        listeners.remove(l);
    }

    public void publish(Event event) {
        // 遍历时无锁，且不会抛 ConcurrentModificationException
        // 此时其他线程注册新监听器不影响本次遍历（快照隔离）
        for (EventListener l : listeners) {
            l.onEvent(event);
        }
    }
}
为什么不抛 ConcurrentModificationException？
普通 ArrayList 的 iterator 有一个 modCount 检查，一旦遍历中有写操作就抛异常。 CopyOnWriteArrayList 的 iterator 拿的是创建时刻的数组快照引用，写操作修改的是全新数组，两者互不干扰，因此永远不抛此异常。

并发陷阱与常见误区

陷阱一：迭代器只是快照，不反映实时状态

反例 — 依赖迭代器做实时判断
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
list.add("A");

// 线程 1：拿到 iterator（快照：["A"]）
Iterator<String> it = list.iterator();

// 线程 2：此刻 add("B")，list 内部已替换为新数组 ["A","B"]
list.add("B");

// 线程 1 继续迭代：只能看到 "A"，看不到 "B" ← 弱一致性！
while (it.hasNext()) {
    System.out.println(it.next());  // 只打印 "A"
}
弱一致性（Weakly Consistent）是 COW 迭代器的设计决策，不是 bug。 如果你的业务需要迭代时看到最新数据，COW 不适合，应考虑加锁或换用其他方案。

陷阱二：先检查后执行（Check-Then-Act）复合操作不原子

反例 vs 正例 — 条件添加
// ❌ 反例：两步操作之间可能被其他线程插入
if (!list.contains(element)) {
    list.add(element);     // 可能重复添加！
}

// ✅ 正例：使用原子的 addIfAbsent
list.addIfAbsent(element);  // 内部加锁，contains + add 原子完成

陷阱三：高频写场景内存压力极大

每次 add / remove 都会复制整个数组。如果列表有 10 万个元素，每秒写 1000 次， 就要每秒创建 1000 个 10 万元素的数组——GC 压力巨大。

写频繁时的替代方案：
  • 需要并发读写且写多读多 → 用 ConcurrentHashMap 或加锁的 ArrayList
  • 需要队列语义 → 用 ConcurrentLinkedQueue / BlockingQueue
  • 需要跳表 → 用 ConcurrentSkipListSet

陷阱四：迭代器不支持 remove()

反例 — 通过迭代器删除
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (someCondition(it.next())) {
        it.remove();  // ❌ 抛 UnsupportedOperationException！
    }
}

// ✅ 正例：用 removeIf（原子批量操作，JDK 8+）
list.removeIf(element -> someCondition(element));

与其他并发 List 方案对比

方案 读性能 写性能 迭代安全 适用场景
ArrayList 最快 最快 ❌ CME 单线程
synchronizedList 慢（全锁） 慢（全锁） 需手动加锁 读写均衡、低并发
CopyOnWriteArrayList 极快（无锁） 慢（复制） ✅ 快照隔离 读多写少、监听器列表
ConcurrentLinkedQueue 快 快（CAS） ✅ 弱一致 FIFO 队列语义

使用决策树

是否需要 List 语义（索引访问）？
├─ 否 → 考虑 Queue / Set / Map 系列
└─ 是
    ├─ 单线程？→ ArrayList
    └─ 多线程？
        ├─ 读多写少，遍历频繁？→ CopyOnWriteArrayList ✓
        ├─ 写频繁，列表较大？→ Collections.synchronizedList + 手动锁
        └─ 高并发写且需要有序？→ ConcurrentSkipListSet（Set 语义）

面试高频考点

  • COW 如何保证读线程看到完整数据？ — volatile 写保证 happens-before；新数组在发布前已完全初始化；读线程拿的是不可变快照。
  • 为什么迭代不抛 ConcurrentModificationException？ — iterator 持有创建时刻的数组引用快照，写操作对新数组操作，两者独立，无 modCount 检查。
  • COW 的弱点是什么？ — 写开销大（O(n) 复制）；内存占用翻倍；迭代器只是快照，实时性弱。
  • CopyOnWriteArrayList 和 CopyOnWriteArraySet 的关系？ — CopyOnWriteArraySet 内部直接持有一个 CopyOnWriteArrayList， add 委托给 addIfAbsent 实现去重。
  • remove 操作的时间复杂度？ — O(n)，需要遍历找到元素位置，再复制跳过该位置的新数组。

源码关键点速记

三句话记住 COW 核心：
1. 读不加锁：volatile 读数组引用，直接访问元素，零竞争。
2. 写加锁复制：ReentrantLock 保证写-写互斥，Arrays.copyOf 产生新数组，修改后 volatile 写替换引用。
3. 迭代器持快照：创建时绑定当前数组引用，遍历期间外部任何修改都不影响它，永不抛 CME，但也看不到新数据。
