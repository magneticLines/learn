02. ConcurrentHashMap
分段锁 → CAS + synchronized

从 JDK 1.5 的 16 段锁，到 JDK 8 彻底重构为 CAS + synchronized per-bucket，深入理解并发 Map 的设计演进与实现细节。

1. 问题根源：同步容器的性能瓶颈

上一节介绍的 Hashtable 和 Collections.synchronizedMap 都用一把全局锁保护整个 Map。这意味着无论读写操作发生在哪个 key，所有线程都要排队，并发度恒为 1。

想象一栋公寓楼：Hashtable 相当于整栋楼只有一把大门钥匙——任何时刻只有一个人能进出，哪怕他们分别住在 1 楼和 18 楼，完全不会互相干扰。锁的粒度远大于冲突的粒度，这就是性能问题的本质。

核心问题：Map 内部数据分布在多个 bucket（桶），不同 bucket 的读写天然无冲突。全局锁把本可并行的操作串行化了。

2. JDK 7 的方案：Segment 分段锁

JDK 7 的 ConcurrentHashMap 将内部结构分为 N 个 Segment（默认 16 个），每个 Segment 相当于一个小型 Hashtable，持有自己的 ReentrantLock。不同 Segment 的操作互不阻塞。

ConcurrentHashMap (JDK 7) ├── Segment[0] ← ReentrantLock + HashEntry[] ├── Segment[1] ← ReentrantLock + HashEntry[] ├── ... └── Segment[15] ← ReentrantLock + HashEntry[]

写操作流程：先用 key 的高位 hash 定位到某个 Segment，再对该 Segment 加锁，然后在 Segment 内部的 HashEntry 数组中进行操作。最多有 16 个线程可以同时写不同的 Segment。

JDK 7 ConcurrentHashMap — put 简化逻辑
public V put(K key, V value) {
    // 第一步：通过高位 hash 定位 Segment，最多 16 个
    Segment<K,V> s = segmentFor(hash(key));
    // 第二步：只锁当前 Segment，其余 15 个 Segment 不受影响
    return s.put(key, hash(key), value, false);
}

JDK 7 的问题

  • Segment 数量在初始化时固定，不能随运行时负载动态调整（扩容也只扩 Segment 内部的数组）
  • 每个 Segment 本质是一个 ReentrantLock，即使不竞争也要维护锁对象，内存占用较大
  • 跨 Segment 的聚合操作（如 size()）需要锁住全部 Segment，开销大
  • 结构复杂，两级数组（Segment[] + HashEntry[]）使代码难以维护

3. JDK 8 的重构：CAS + synchronized per-bucket

JDK 8 彻底抛弃了 Segment，回归单层 Node[] 数组（类似普通 HashMap），但用更细粒度的锁替代分段锁：

  • 读操作：完全无锁，通过 volatile 保证可见性
  • 写操作（bucket 为空）：用 CAS 原子插入第一个节点，无需加锁
  • 写操作（bucket 非空）：只锁这个 bucket 的头节点（synchronized(头节点)），其他 bucket 完全不受影响
  • 并发度：理论上等于 bucket 数量（默认初始 16，可扩容至 N）
ConcurrentHashMap (JDK 8) Node[] table (volatile) ├── bucket[0] → Node(k,v) → Node → ... ← 锁头节点 ├── bucket[1] → null ← CAS 插入 ├── bucket[2] → TreeBin(红黑树) ← 锁 TreeBin └── ...

关键源码拆解：put 方法核心逻辑

JDK 8 ConcurrentHashMap.put — 简化注释版
final V putVal(K key, V val, boolean onlyIfAbsent) {
    int hash = spread(key.hashCode());
    for (Node[] tab = table;;) {
        Node f; int n, i, fh;

        if (tab == null) {
            // 懒初始化：第一次 put 时才创建数组，用 CAS 保证只初始化一次
            tab = initTable();

        } else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // bucket 为空：CAS 直接写入，不加锁
            // CAS 失败说明有并发竞争，循环重试
            if (casTabAt(tab, i, null, new Node(hash, key, val)))
                break;

        } else if ((fh = f.hash) == MOVED) {
            // 当前 bucket 是 ForwardingNode，说明正在扩容迁移
            // 当前线程加入协助扩容，而不是阻塞等待
            tab = helpTransfer(tab, f);

        } else {
            // bucket 已有节点：只锁这一个 bucket 的头节点
            synchronized (f) {
                // 再次验证头节点未被替换（double-check）
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {
                        // 链表：遍历找相同 key 或追加到尾部
                        traverseLinkedList(...);
                    } else if (f instanceof TreeBin) {
                        // 红黑树：走树的插入逻辑
                        treeInsert(...);
                    }
                }
            }
            // 链表长度 >= 8 且数组长度 >= 64 时转为红黑树
            treeifyBin(tab, i);
            break;
        }
    }
    addCount(1L, binCount); // 用 LongAdder 思路统计 size
    return null;
}
为什么重新引入 synchronized 而不用 ReentrantLock？
JDK 8 对 synchronized 的偏向锁、轻量级锁做了大量优化。无竞争时偏向锁几乎零开销；轻度竞争时自旋升级，性能与 ReentrantLock 相当；而 synchronized 不需要额外创建锁对象，节省内存。Segment 时代用 ReentrantLock 是因为当时 synchronized 还未优化到位。

4. volatile 与 Unsafe：可见性的基石

JDK 8 的 ConcurrentHashMap 中，Node[] table 被声明为 volatile，但这只能保证数组引用的可见性，不能保证数组元素的可见性。因此读写数组元素必须用 Unsafe 的 volatile 语义方法：

tabAt / casTabAt / setTabAt — 绕过 JMM 直接操作内存
// 读：getObjectVolatile 保证读到最新值（相当于 volatile 读）
static final <K,V> Node<K,V> tabAt(Node<K,V>[] tab, int i) {
    return (Node<K,V>) U.getObjectVolatile(tab, ((long)i << ASHIFT) + ABASE);
}

// 写：CAS，期望值为 null 时才插入（compareAndSwapObject）
static final <K,V> boolean casTabAt(Node<K,V>[] tab, int i,
        Node<K,V> c, Node<K,V> v) {
    return U.compareAndSwapObject(tab, ((long)i << ASHIFT) + ABASE, c, v);
}

同理，Node 的 val 和 next 字段都声明为 volatile，使得无锁的 get 操作能读到最新写入的值。

5. size() 如何并发安全地统计？

全局加锁统计太慢。JDK 8 借鉴了 LongAdder 的思路：用一个 baseCount 加上多个 CounterCell[] 分散计数，减少 CAS 竞争。

addCount — LongAdder 风格的分布式计数
private final void addCount(long x, int check) {
    // 1. 尝试 CAS 更新 baseCount（低竞争时直接成功）
    if (!casBaseCount(b, b + x)) {
        // 2. CAS 失败说明竞争激烈，随机选一个 CounterCell 累加
        // 效果类似 LongAdder：分散竞争，sum() 时汇总所有 cell
        fullAddCount(x, false);
    }
}

// size() = baseCount + sum(counterCells)
public int size() {
    long n = sumCount();  // 汇总，但不保证强一致
    return ((n < Integer.MIN_VALUE) ? Integer.MIN_VALUE :
            (n > Integer.MAX_VALUE) ? Integer.MAX_VALUE : (int)n);
}
注意：size() 返回的是一个估算值，并发场景下不保证精确。如果需要精确计数，应改用 mappingCount()（返回 long）并接受其"最终一致"语义，或配合外部同步。

6. 并发扩容：多线程协作迁移

JDK 8 的扩容非常精妙：不是一个线程独占迁移整个数组，而是将数组分成多个迁移任务段（每段默认 16 个 bucket），多个写线程在调用 put 时，遇到 ForwardingNode（hash == MOVED）就会调用 helpTransfer 主动参与迁移，迁移完成后再完成自己的写操作。

扩容流程（table 容量 16 → 32） ① 创建 nextTable（新数组，容量×2） ② 将 table 分成若干段，每段 ≥16 个 bucket ③ 线程 A 领取 bucket[0..15] 迁移任务 ④ 迁移完成的 bucket 置为 ForwardingNode（哨兵） ⑤ 线程 B 写 bucket[5]，发现是 ForwardingNode → 调用 helpTransfer，领取剩余迁移段 → 迁移完成后再完成自己的 put ⑥ 全部迁移完成，table = nextTable

8. 并发陷阱与反例

陷阱一：复合操作不是原子的

ConcurrentHashMap 保证的是单次操作的原子性，"先检查后操作"的复合操作仍然存在竞态条件。

反例 — check-then-act 竞态
// 两个线程同时执行，都看到 key 不存在，都执行 put → 后者覆盖前者
if (!map.containsKey("counter")) {
    map.put("counter", 1);  // 危险！两次独立的原子操作
}

// 正例：用原子的 putIfAbsent / computeIfAbsent
map.putIfAbsent("counter", 1);

// 或用 compute 实现原子的"读-改-写"
map.compute("counter", (k, v) -> v == null ? 1 : v + 1);

陷阱二：迭代器是弱一致的

ConcurrentHashMap 的迭代器不会抛 ConcurrentModificationException，但它只保证弱一致性：迭代过程中发生的修改，可能被看到也可能看不到，不会崩溃但也不保证完整性。

弱一致迭代 — 不会崩溃，但可能遗漏新插入的元素
// 安全：不抛异常，但迭代开始后插入的 key 可能不被遍历到
for (Map.Entry<String,Integer> entry : map.entrySet()) {
    // 此时其他线程可能在 put 或 remove
    process(entry);
}

// 如果需要对迭代的快照做精确处理，先复制一份
Map<String,Integer> snapshot = new HashMap<>(map);
// 此后对 snapshot 的迭代不受并发修改影响

陷阱三：不允许 null key / null value

与 HashMap 不同，ConcurrentHashMap 禁止 null。原因：在并发场景下，get(key) 返回 null 有歧义——到底是 key 不存在，还是 value 本身就是 null？若允许 null，还需要用 containsKey 做二次确认，这又是一次非原子的复合操作。

null 歧义问题演示
// HashMap 中可以这样写，但在并发场景下有歧义
V v = map.get("key");
if (v == null) {
    // 是 key 不存在？还是 value == null？
    // 此时另一个线程可能已经 put(key, nonNullValue) 了！
    if (map.containsKey("key")) {  // 二次检查，仍然是竞态
        handleNullValue();
    }
}

// ConcurrentHashMap 直接禁止 null，逼迫调用者用 getOrDefault
V v = concMap.getOrDefault("key", defaultValue);  // 语义清晰

陷阱四：误用 putIfAbsent 导致资源泄漏

反例 vs 正例 — computeIfAbsent 更安全
// 反例：如果 value 构造代价很高，putIfAbsent 会浪费
// 所有竞争线程都先创建了 ExpensiveObject，但只有一个被使用
Cache cache = new Cache();  // 高代价对象先创建
map.putIfAbsent("key", cache);  // 可能白创建

// 正例：computeIfAbsent 只在 key 不存在时才执行 lambda
// lambda 在持有 bucket 锁期间执行，保证只创建一次
map.computeIfAbsent("key", k -> new Cache(k));
面试高频陷阱：computeIfAbsent 的 lambda 在持有 bucket 锁期间执行，不要在 lambda 里对同一个 ConcurrentHashMap 进行递归操作，否则会产生死锁（JDK 8 已知问题，JDK 9+ 部分改善但仍需注意）。

9. 常用原子操作 API 速查

JDK 8+ 原子复合操作
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// 1. 不存在时才插入，返回旧值（null 表示插入成功）
map.putIfAbsent("a", 1);

// 2. 不存在时用 lambda 计算 value，高代价初始化首选
map.computeIfAbsent("b", k -> expensiveLoad(k));

// 3. 存在时更新，不存在时忽略
map.computeIfPresent("a", (k, v) -> v + 1);

// 4. 原子读-改-写（无论存不存在）
map.compute("c", (k, v) -> v == null ? 1 : v + 1);

// 5. merge：key 不存在时设置初始值，存在时用 BiFunction 合并
map.merge("d", 1, Integer::sum);  // 等价于 wordCount++

// 6. 条件替换（CAS 语义）
map.replace("a", 1, 2);  // 仅当 value == 1 时替换为 2

10. 面试要点总结

  • JDK 7 vs JDK 8 最大区别：锁粒度从 Segment 级别细化到单个 bucket 头节点，且读操作完全无锁。
  • 为什么不用 volatile 数组元素：Java 规范只保证数组引用的 volatile 语义，数组元素需要用 Unsafe.getObjectVolatile 才能获得 volatile 语义。
  • 为什么重新用 synchronized 而非 ReentrantLock：JDK 6+ 的偏向锁/轻量级锁优化使 synchronized 在低竞争下近乎免费，且省去了 ReentrantLock 对象的内存开销。
  • size() 为何不精确：分布式计数（LongAdder 思路），读 size 时不加全局锁，汇总瞬间各线程 cell 的值可能仍在变化。
  • null 禁止的根本原因：消除"值不存在"和"值为 null"的歧义，强制使用 getOrDefault 等语义清晰的 API。
  • 扩容为何不阻塞：ForwardingNode 哨兵 + 多线程协作迁移，写线程遇到扩容时主动帮助迁移而非等待。
选型建议：绝大多数并发读写场景首选 ConcurrentHashMap。若读远多于写（如配置缓存），可考虑下一节的 CopyOnWriteArrayList 思路；若需要有序遍历，可使用 ConcurrentSkipListMap（跳表，支持 NavigableMap 接口）。
