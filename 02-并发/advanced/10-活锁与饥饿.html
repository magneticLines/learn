Part IV 高级主题 · 活跃性与性能

10. 活锁与饥饿

死锁让线程永远阻塞，活锁让线程永远忙碌却毫无进展，饥饿让某些线程永远等不到 CPU。三者都是活跃性故障，但成因和解法截然不同。

1. 活跃性问题全景

并发程序的正确性不仅要求"做到正确结果"，还要求"最终能做到"，后者称为活跃性（Liveness）。 违反活跃性的三种经典故障：

故障线程状态是否占用 CPU能否自行解除
死锁BLOCKED / WAITING否否（需外力）
活锁RUNNABLE是（持续消耗）有可能（引入随机抖动后）
饥饿RUNNABLE 但极少被调度极少否（需调整优先级或公平策略）
类比：死锁 = 两辆车在单行道正面相遇，谁都不退；活锁 = 两人在走廊相向而行，互相礼让却同向侧步，永远让不开；饥饿 = 食堂窗口总被高个子挡住，矮个子始终插不进去。

2. 活锁（Livelock）

2.1 本质：过度礼让导致的无进展循环

活锁的根源是每个线程都在"为对方让路"，却恰好以相同的节奏让到同一个方向，导致所有线程持续运行却无任何实质性进展。 与死锁不同，活锁中的线程是 RUNNABLE 状态，从外部看 CPU 使用率可能很高，但系统吞吐量为零。

2.2 反例：消息重试的活锁

消息处理系统中，两个 worker 共享一个消息队列。当某条消息处理失败时，各自将其放回队头重试——如果两个 worker 始终以相同节奏碰到同一条"毒消息"，就会陷入活锁。

LivelockDemo.java — 反例活锁
// 反例：两个线程在"对方退让"的循环中永远无法前进
class Worker implements Runnable {
    private final String name;
    private Worker partner;             // 协作对象
    private volatile boolean active = true;

    public void run() {
        while (active) {
            // 只要对方也是 active，就"礼让"，把 active 改为 false 再改回 true
            if (partner.isActive()) {
                System.out.println(name + " 礼让，先暂停");
                active = false;  // 让出
                sleep(1);
                active = true;   // 恢复——但此时对方也已恢复！
            } else {
                doWork();        // 对方退让时才干活，但永远等不到这一刻
                active = false;
            }
        }
    }
}

// 两个 Worker 同时启动 → 永远互相礼让，doWork() 永远不执行
陷阱：活锁在测试环境下极难复现——两个线程的调度时序必须高度对称才能触发。一旦出现 CPU 打满但无进展的现象，务必用 jstack 检查是否有大量 RUNNABLE 线程在重复执行同一段代码。

2.3 正例：引入随机退避（Randomized Backoff）

解决活锁的核心手段是打破对称性——让每个线程在退让时等待一段随机时长，使它们不再以相同节奏重试，从而错开冲突窗口。以太网的 CSMA/CD 协议正是用这个思路解决信号碰撞问题的。

RetryWithBackoff.java — 正例随机退避
private static final Random RNG = new Random();
private static final int MAX_RETRIES = 5;

public boolean trySendMessage(Message msg) {
    int attempt = 0;
    while (attempt < MAX_RETRIES) {
        if (channel.trySend(msg)) return true;

        // 关键：退避时长加入随机量，打破两端的时序对称
        long backoff = (1L << attempt)        // 指数基础：1, 2, 4, 8, 16 ms
                     + RNG.nextInt(10);    // 0~9 ms 随机抖动（jitter）
        TimeUnit.MILLISECONDS.sleep(backoff);
        attempt++;
    }
    return false;  // 超过最大重试次数，告知调用方
}
指数退避 + 随机抖动（Exponential Backoff with Jitter）是分布式系统中的标准模式。Java 生态中 Spring Retry、Resilience4j 的 RetryRegistry 都内置了此策略。

2.4 活锁的另一个经典场景：CAS 自旋

无锁算法中的 CAS 自旋在极高竞争下也可能退化为活锁——每个线程都在不停地 CAS 失败并重试，CPU 被白白消耗。 LongAdder 正是为了解决 AtomicLong 在高竞争下的这一问题而设计：将单个热点拆分为多个 Cell，从根本上降低竞争概率，而非靠退避"硬撑"。

CAS 自旋退化对比概念示意
// 反例：高竞争下 AtomicLong.incrementAndGet() 内部自旋
// 所有线程同时 CAS(old, old+1) → 只有一个成功 → 其余全部重试
// 极端情况：N 个线程只有 1 个有效推进，活锁特征明显

// 正例：LongAdder 分散热点
LongAdder counter = new LongAdder();
counter.increment();  // 写入各自绑定的 Cell，sum() 时才汇总
// 竞争概率从 O(N) 降至 O(1)，彻底避免高竞争活锁

3. 饥饿（Starvation）

3.1 本质：资源分配的不公平性

饥饿指某个线程长期无法获得所需资源（CPU 时间、锁、信号量槽位等），导致其任务迟迟无法推进。 造成饥饿的根本原因是资源分配策略存在偏向性，优先级高的线程或特定线程反复抢先，低优先级线程始终排不上队。

3.2 常见成因

饥饿的根源 ├── 线程优先级滥用：高优先级线程将 CPU 时间占尽 ├── 非公平锁实现：synchronized / 非公平 ReentrantLock 允许插队 ├── 持锁时间过长：某线程长期持有锁，其他线程长时间阻塞 ├── 无界任务队列 + 有界线程池：老任务未完成，新任务永远排队 └── wait/notify 的惊群：notifyAll 后优势线程反复抢占，弱势线程永远 wait

3.3 反例：优先级倒置（Priority Inversion）

优先级倒置是饥饿的一个特殊变体：低优先级线程持有高优先级线程需要的锁，而中等优先级线程持续占用 CPU， 导致低优先级线程无法释放锁，高优先级线程实际上以"低于中等优先级"的速度推进。 1997 年火星探路者任务的实时系统曾因此反复重启，是业界最著名的案例。

PriorityInversion.java — 反例优先级倒置
// 低优先级线程 L 持有锁 mutex，尚未释放
Thread low = new Thread(() -> {
    synchronized (mutex) {
        heavyWork();  // 长时间持锁
    }
});
low.setPriority(Thread.MIN_PRIORITY);

// 高优先级线程 H 等待同一把锁
Thread high = new Thread(() -> {
    synchronized (mutex) { criticalTask(); }
});
high.setPriority(Thread.MAX_PRIORITY);

// 中等优先级线程 M，不需要锁，疯狂占 CPU
Thread medium = new Thread(() -> {
    while (true) { cpuBoundTask(); }  // 抢走 CPU，low 无法完成 heavyWork()
});
medium.setPriority(Thread.NORM_PRIORITY);

// 结果：high 优先级最高，却被 medium 间接阻塞，实际以低于 medium 的速度运行
Java 优先级警告：Java 的线程优先级（1–10）仅是对底层 OS 的建议，JVM 并不保证其语义。在 Linux 上 Thread.setPriority() 通常被忽略（需 root 权限才能真正调整 nice 值），在 Windows 上才有一定效果。不要依赖线程优先级来设计并发逻辑。

3.4 反例：非公平锁导致的饥饿

UnfairLockStarvation.java — 反例非公平锁
// synchronized 和 new ReentrantLock(false) 都是非公平锁
// 当锁释放瞬间恰好有新线程 T_new 到来，T_new 可以"插队"直接获得锁
// 已在队列中等待的 T_old 可能反复被插队，长期得不到锁 → 饥饿

ReentrantLock unfair = new ReentrantLock();   // 默认 false，非公平
ReentrantLock fair   = new ReentrantLock(true);  // 公平锁，按等待顺序授予

// 非公平锁吞吐量更高（减少上下文切换），公平锁保证无饥饿
// 如无特殊需求，默认选非公平；对等待时间敏感的场景选公平锁

3.5 正例：用公平锁 + 合理超时消除饥饿

FairLockWithTimeout.java — 正例公平锁 + 超时
private final ReentrantLock lock = new ReentrantLock(true);  // 公平

public void process() {
    boolean acquired = false;
    try {
        // tryLock 带超时：即使公平队列排队，也不会无限等待
        acquired = lock.tryLock(500, TimeUnit.MILLISECONDS);
        if (!acquired) {
            // 超时后执行降级逻辑，而不是无限阻塞
            fallback();
            return;
        }
        doCriticalWork();
    } finally {
        if (acquired) lock.unlock();  // 只有成功加锁才解锁
    }
}

3.6 正例：用 Semaphore 实现公平的资源限流

数据库连接池、线程池等资源有限的场景下，用带公平标志的 Semaphore 可以保证"先等先得"，避免某些线程长期无法获得连接。

FairSemaphore.java — 正例公平信号量
// 公平 Semaphore：等待最久的线程优先获得许可
private final Semaphore pool = new Semaphore(10, true);

public Connection acquireConnection() {
    pool.acquire();            // 按到达顺序排队，消除饥饿
    return getFromPool();
}

public void releaseConnection(Connection conn) {
    returnToPool(conn);
    pool.release();
}

4. 线上诊断：jstack + 监控

4.1 用 jstack 识别活锁

活锁的线程是 RUNNABLE，jstack 不会像死锁那样打出 "Found one Java-level deadlock" 提示。 需要间隔 2–3 秒抓两次快照，对比同一线程的栈帧——若栈帧相同但线程在运行，高度疑似活锁。

Shell活锁诊断
# 抓两次快照，对比 RUNNABLE 线程的栈帧
jstack <pid> > snap1.txt
sleep 3
jstack <pid> > snap2.txt
diff snap1.txt snap2.txt
# 若大量 RUNNABLE 线程栈帧几乎不变 → 活锁嫌疑

4.2 用 jstack 识别饥饿

饥饿线程通常长期处于 BLOCKED 或 WAITING 状态。在快照中搜索 "waiting to lock"， 若同一线程在多次快照中始终处于等待同一把锁，且其他线程轮番持有该锁，即为饥饿。

Shell饥饿诊断
# 在 jstack 输出中查找长期阻塞的线程
grep -A 5 "BLOCKED" snap1.txt | grep "waiting to lock"
# 结合 JMX / Micrometer 监控 lock.waitTime 直方图
# 若 P99 等待时间远大于 P50，说明存在严重的不公平性

6. 实践清单

  • 重试逻辑必须加随机抖动，避免多个实例同频碰撞形成活锁。
  • CAS 自旋循环若重试次数超过阈值（如 64 次），应主动 Thread.yield() 或升级为阻塞策略。
  • 不要用 Thread.setPriority() 做业务逻辑调度，优先级在跨 OS 下语义不一致。
  • 对等待时间敏感的资源（连接池、令牌桶），使用 new ReentrantLock(true) 或 new Semaphore(n, true)。
  • 持锁时间尽量短：锁内只做必要操作，IO、计算应移到锁外，减少其他线程的等待窗口。
  • 线上部署 ThreadMXBean 监控，定期采集 getThreadCpuTime 与 getBlockedTime，建立基线告警。
面试高频考点：活锁和死锁的区别是什么？——死锁线程不消耗 CPU，处于等待状态；活锁线程消耗 CPU，处于运行状态，但没有实质进展。活锁的解法核心是打破对称性（随机退避）；死锁的解法核心是破坏循环等待（锁顺序）。
