05. CAS 与原子类 atomic

从 CPU 指令到 Java API — 读懂无锁并发的核心引擎

1. 锁的代价 — 为什么要无锁？

传统的 synchronized 和 ReentrantLock 都依赖操作系统的互斥量（mutex）。 当线程拿不到锁时，会从 用户态切换到内核态 进行上下文切换，代价极高（通常数百纳秒到数微秒）。 在竞争不激烈或临界区极短的场景下，这种切换的开销甚至远超临界区本身的工作量。

无锁（Lock-Free） 的思路是：不阻塞线程，而是让线程乐观地假设"当前没有人竞争"， 直接去改数据；如果失败了，就重试。实现这一思路的硬件原语就是 CAS（Compare-And-Swap）。

2. CAS 是什么 — 硬件级的原子三元操作

CAS 的语义可以用一句话描述："如果内存地址 V 的当前值等于期望值 A，就把它原子地改为新值 B，否则什么都不做，并告知调用者是否成功。"

伪代码 — CAS 语义
// 这三步在 CPU 层面是一条不可分割的指令（x86: LOCK CMPXCHG）
boolean compareAndSwap(Address V, int expected, int newValue) {
    if (V.read() == expected) {
        V.write(newValue);
        return true;
    }
    return false;
}

x86 通过 LOCK CMPXCHG 指令实现 CAS。LOCK 前缀会在执行期间锁住 CPU 缓存行（不是总线）， 确保该操作在多核间的原子性，代价远低于操作系统级锁。

与数据库乐观锁类比： CAS 就像数据库的"版本号乐观锁"——先读版本号（expected），更新时带上版本号做条件， 若版本不匹配说明有人抢先修改，当前操作放弃并重试。
线程读取 V 的当前值 = A
  │
  ▼
执行业务，计算出新值 B
  │
  ▼
CAS(V, A, B) ──── V 还是 A ──▶ 成功，V = B ✓
              └── V 已被改 ──▶ 失败，重新读取 V，再试

3. JVM 如何把 CAS 暴露给 Java — sun.misc.Unsafe

Java 没有直接操作 CPU 指令的能力，靠的是 sun.misc.Unsafe 这个"后门"类。 它提供了 compareAndSwapInt、compareAndSwapLong、compareAndSwapObject 等 native 方法，JIT 会将它们内联为单条 LOCK CMPXCHG 指令。

你不应该直接用 Unsafe——它要求传入对象的字段偏移量（offset），操作错误会直接导致 JVM 崩溃。 Java 官方在 java.util.concurrent.atomic 包里封装了所有原子类，这才是日常使用的 API。

5. AtomicInteger 核心方法与原理

AtomicInteger — 源码级解读（JDK 17 简化）
public class AtomicInteger extends Number {

    // Unsafe 单例，通过反射获取
    private static final Unsafe U = Unsafe.getUnsafe();
    // VALUE 是 value 字段在对象内存布局中的偏移量（编译期确定）
    private static final long VALUE
        = U.objectFieldOffset(AtomicInteger.class, "value");

    // volatile 保证可见性；CAS 保证原子性
    private volatile int value;

    // 原子地 +1 并返回旧值（等价于 i++）
    public final int getAndIncrement() {
        return U.getAndAddInt(this, VALUE, 1);
    }

    // compareAndSet：暴露给用户的 CAS 接口
    public final boolean compareAndSet(int expectedValue, int newValue) {
        return U.compareAndSetInt(this, VALUE, expectedValue, newValue);
    }

    // JDK 9+ 新增：支持内存语义参数（plain / opaque / release / acquire / volatile）
    public final int getAndUpdate(IntUnaryOperator updateFn) {
        int prev = get(), next = 0;
        for (boolean haveNext = false;;) {
            if (!haveNext)
                next = updateFn.applyAsInt(prev);
            if (weakCompareAndSetVolatile(prev, next))
                return prev;  // CAS 成功，返回旧值
            haveNext = (prev == (prev = get()));
        }
    }
}
volatile + CAS 的分工： volatile 保证其他线程对 value 的写操作立即可见（解决可见性）； CAS 保证"读—比较—写"三步的原子性（解决原子性）。两者缺一不可。

6. 实战示例 — 无锁计数器与无锁栈

6.1 反例：volatile 计数器（原子性不足）

BadCounter.java — 并发下结果不正确
public class BadCounter {
    // volatile 只保证可见性，不保证 ++ 的原子性
    private volatile int count = 0;

    public void increment() {
        count++;  // 实为 read → +1 → write，三步非原子
    }
}

// 10 个线程各执行 10000 次 increment()，最终 count 通常 < 100000

6.2 正例：AtomicInteger 计数器

AtomicCounter.java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // 单条 CAS 指令，原子且无锁
    }

    public int get() {
        return count.get();
    }
}

6.3 使用 compareAndSet 实现自定义原子操作

AtomicMax.java — 原子地更新最大值
import java.util.concurrent.atomic.AtomicInteger;

/**
 * 无锁地维护一个"已见到的最大值"。
 * 直接 synchronized 当然也行，但这里展示 CAS 自旋的惯用写法。
 */
public class AtomicMax {
    private final AtomicInteger max = new AtomicInteger(Integer.MIN_VALUE);

    public void update(int value) {
        int cur;
        // 自旋：只要 value 更大且 CAS 失败就重试
        do {
            cur = max.get();
            if (value <= cur) return;  // 无需更新，提前退出
        } while (!max.compareAndSet(cur, value));
    }

    public int get() { return max.get(); }
}

6.4 无锁栈（Treiber Stack）

这是 CAS 最经典的数据结构应用：用 AtomicReference 原子替换栈顶指针， 实现线程安全的栈，完全不用锁。

LockFreeStack.java — Treiber Stack
import java.util.concurrent.atomic.AtomicReference;

public class LockFreeStack<T> {

    private static class Node<T> {
        final T item;
        Node<T> next;
        Node(T item) { this.item = item; }
    }

    private final AtomicReference<Node<T>> top = new AtomicReference<>();

    public void push(T item) {
        Node<T> newNode = new Node<>(item);
        do {
            newNode.next = top.get();
            // 原子地把 top 从 oldTop 换成 newNode
        } while (!top.compareAndSet(newNode.next, newNode));
    }

    public T pop() {
        Node<T> oldTop;
        do {
            oldTop = top.get();
            if (oldTop == null) return null;  // 栈空
        } while (!top.compareAndSet(oldTop, oldTop.next));
        return oldTop.item;
    }
}

7. 高竞争场景：LongAdder 为什么比 AtomicLong 快

AtomicLong 在极高并发下，所有线程都抢同一个 value 字段的 CAS， 失败了就自旋重试，竞争越激烈、CPU 资源浪费越多。

LongAdder（JDK 8，Doug Lea 设计）采用 分段累加 策略： 内部维护一个 base 值和一个 Cell[] 数组，无竞争时更新 base， 有竞争时将线程哈希到不同的 Cell，最终求和时累加所有 Cell。

AtomicLong 结构： LongAdder 结构：
┌──────────┐ ┌────────┬──────────────────────┐
│ value │◀──所有线程竞争 │ base │ Cell[0..N-1] │
└──────────┘ └────────┴──────────────────────┘
                                     ▲ ▲
                                    无竞争时 竞争时各线程
                                               哈希到不同 Cell
                                  sum() = base + Cell[0] + Cell[1] + …
选型原则： 需要精确读取当前值（get() 有意义）→ 用 AtomicLong； 只需统计累加（最终求和）且并发极高 → 用 LongAdder。 LongAdder 没有 compareAndSet，不能做 CAS 条件更新。
LongAdder 用法示例
import java.util.concurrent.atomic.LongAdder;

LongAdder requestCount = new LongAdder();

// 在每次请求处理线程中：
requestCount.increment();

// 统计上报时（允许少量误差）：
long total = requestCount.sum();

// 重置并获取：
long snapshot = requestCount.sumThenReset();

8. 并发陷阱与高频考点

陷阱 1：CAS 自旋浪费 CPU（Busy Waiting）

CAS 失败后会立即重试，在竞争极其激烈的场景下，线程不断循环消耗 CPU 但不产出任何结果， 称为 自旋浪费。症状是 CPU 使用率飙高但吞吐量不增反降。

解决方案：限制自旋次数（超过后降级为阻塞锁），或改用 LongAdder 减少竞争集中度。 JDK 内部的 LockSupport.parkNanos 就是在自旋一定次数后挂起线程的策略。

陷阱 2：复合操作的原子性错觉

错误示例 — 两次原子操作合起来不是原子的
AtomicInteger counter = new AtomicInteger(0);

// ❌ 错误：get() 和 set() 之间有窗口，其他线程可以插入
if (counter.get() < 10) {
    counter.incrementAndGet();  // 可能超过 10
}

// ✅ 正确：用 CAS 做"check-then-act"的原子版本
int cur;
do {
    cur = counter.get();
    if (cur >= 10) break;
} while (!counter.compareAndSet(cur, cur + 1));

陷阱 3：AtomicReference 的对象可变性问题

AtomicReference 保证的是引用替换的原子性，不保证引用指向的对象本身是不可变的。 如果多个线程在 CAS 成功后继续修改同一个对象内部状态，仍然需要额外同步。

最佳实践：配合 AtomicReference 使用的对象应当是不可变（immutable）的， 即字段全部 final，修改时整体替换引用而非修改字段。

陷阱 4：ABA 问题

CAS 只比较值是否相等，无法感知"值从 A 变成 B 又变回 A"的过程。 在某些场景（如链表节点复用）这会导致数据结构损坏。 这是下一节 06. ABA 问题与 AtomicStampedReference 的核心内容，此处先埋个眼。

9. CAS vs synchronized — 不是替代，是互补

维度CAS / 原子类synchronized / Lock
实现层用户态，单条 CPU 指令内核态互斥量（悲观锁）
线程等待自旋（忙等）阻塞（让出 CPU）
低竞争性能极快（无切换）较快（JDK 偏向锁优化后也快）
高竞争性能自旋浪费 CPU，可能更差阻塞队列，有序等待
适用临界区极短（单个字段操作）任意长度
死锁风险无有（需要规范加锁顺序）
ABA 问题有（需 Stamped 解决）无
经验法则： 临界区只涉及 1~2 个变量的原子更新 → 首选原子类； 临界区涉及多个变量的一致性、或需要条件等待 → 用 Lock / synchronized。

10. 面试高频考点速查

  • CAS 是什么？ — Compare-And-Swap，硬件级原子三元操作，x86 对应 LOCK CMPXCHG 指令。
  • AtomicInteger 如何保证线程安全？ — volatile 保证可见性，CAS（通过 Unsafe）保证原子性。
  • CAS 的三大缺陷？ — ① 自旋浪费 CPU；② 只能保证单个变量原子性；③ ABA 问题。
  • LongAdder 为什么比 AtomicLong 快？ — 分段 Cell 数组减少竞争，但 sum() 非精确实时值。
  • Unsafe 是什么？ — JVM 提供的底层 native API，原子类的底层支撑，不应在业务代码中直接使用。
  • compareAndSet 和 weakCompareAndSet 的区别？ — weakCompareAndSet 可能伪失败（spurious failure），但在某些平台开销更低；compareAndSet 有完整的 volatile 内存语义。
