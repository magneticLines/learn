04. Condition 条件变量

把"等待某个条件成立"的语义从锁中分离出来——精准唤醒、多条件队列、彻底告别 notifyAll 的惊群效应。

1. 为什么需要条件变量

并发程序中有大量"生产者/消费者"式协作：消费者在队列为空时必须等待，生产者放入数据后再通知消费者。 这种"等待-通知"机制本质上需要两个能力：

  • 原子地释放锁 + 进入等待（否则通知可能在等待开始前就发出，永远丢失）
  • 被唤醒后重新持有锁再继续执行

synchronized 配套的 Object.wait/notify 也能做到，但有一个致命缺陷： 一把锁只有一个等待集合。多条件场景下（如：队列空 vs 队列满）只能用 notifyAll， 把所有等待线程全部唤醒，大多数线程检查条件后又立刻重新睡眠——这就是经典的惊群（Spurious Wakeup）放大版。

核心设计动机：一把 Lock 可以派生出多个 Condition，每个 Condition 维护独立的等待队列。 生产者只唤醒「消费者条件」，消费者只唤醒「生产者条件」，精准、零浪费。

2. Condition 与 Object.wait/notify 对照

维度 Object.wait/notify Condition.await/signal
等待队列数量 每个对象只有 1 个 一把锁可有任意多个
精准唤醒 不支持（notify 随机，notifyAll 全唤醒） 支持（signal 只唤醒该条件队列头部）
超时等待 wait(long ms)，精度毫秒 await(long, TimeUnit)，精度纳秒
不响应中断的等待 无 awaitUninterruptibly()
返回值（是否超时） 无 awaitUntil(Date) 返回 boolean
使用前提 必须持有对象的 monitor 必须持有关联的 Lock

3. Condition API 全景

java.util.concurrent.locks.Condition
├── await() — 等待，响应中断
├── awaitUninterruptibly() — 等待，不响应中断
├── await(long, TimeUnit) — 限时等待，返回剩余时间 > 0 表示正常唤醒
├── awaitNanos(long) — 纳秒精度限时等待
├── awaitUntil(Date) — 等待到某个绝对时刻
├── signal() — 唤醒该条件队列中等待最久的线程
└── signalAll() — 唤醒该条件队列中所有线程
必须在持有锁的情况下调用 await/signal，否则抛出 IllegalMonitorStateException——这与 wait/notify 要求持有 monitor 完全类比。

4. 经典场景：有界阻塞队列

用两个独立 Condition 分别管理"队列非空"和"队列未满"两种条件，是 Condition 最典型的教科书示例（ArrayBlockingQueue 源码就是这么实现的）。

BoundedBlockingQueue.javaJava
public class BoundedBlockingQueue<E> {

    private final ReentrantLock lock  = new ReentrantLock();
    // 两个独立条件队列：精准分工，互不打扰
    private final Condition      notFull  = lock.newCondition();
    private final Condition      notEmpty = lock.newCondition();

    private final Object[] items;
    private int head, tail, count;

    public BoundedBlockingQueue(int capacity) {
        items = new Object[capacity];
    }

    /** 生产者：队列满则等待 */
    public void put(E e) throws InterruptedException {
        lock.lock();
        try {
            // 必须用 while，不能用 if！防止虚假唤醒（spurious wakeup）
            while (count == items.length)
                notFull.await();       // 释放锁，挂起，等待 notFull 信号
            items[tail] = e;
            if (++tail == items.length) tail = 0;
            ++count;
            // 只唤醒等待"非空"的消费者，不惊扰其它生产者
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    /** 消费者：队列空则等待 */
    @SuppressWarnings("unchecked")
    public E take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await();       // 释放锁，挂起，等待 notEmpty 信号
            E x = (E) items[head];
            items[head] = null;
            if (++head == items.length) head = 0;
            --count;
            // 只唤醒等待"未满"的生产者
            notFull.signal();
            return x;
        } finally {
            lock.unlock();
        }
    }
}
为什么 signal 而不是 signalAll？
每个条件队列的语义已足够精确：放入数据后只需唤醒一个消费者（它取走一个元素就够了）； 反之亦然。此处用 signal() 性能更优。 但如果同一条件队列上存在多种不同的等待逻辑（比如有人等"队列元素 > 5"，有人等"队列元素 > 1"）， 则必须用 signalAll()，否则可能唤醒了错误的线程而正确的线程永远沉睡。

5. 内部机制：AQS 条件队列

ReentrantLock.newCondition() 返回的是 AbstractQueuedSynchronizer.ConditionObject。 理解它的内部结构，才能真正看清 await/signal 在做什么。

AQS 内部有两条队列：

同步队列（Sync Queue） — 双向链表，存放"竞争锁失败、正在排队"的线程
条件队列（Condition Queue） — 单向链表，存放"已持有锁、调用 await 主动让出"的线程

调用 await() 时：
  1. 将当前线程封装为 Node，加入 条件队列 尾部
  2. 完全释放锁（支持重入，计数归零）
  3. park 挂起线程

调用 signal() 时：
  1. 将条件队列头节点转移到 同步队列
  2. unpark 唤醒该线程（它将重新竞争锁，获得锁后从 await 返回）
await 释放的是"全部重入层次"：若当前线程持有了 3 层重入，await 会把计数从 3 归零， signal 后重新获得锁时再恢复到 3。这与 Object.wait 的语义完全一致。

6. 并发陷阱与反例

陷阱 1：用 if 代替 while 检查条件（虚假唤醒）

JVM 规范允许线程在没有收到 signal 的情况下从 await 自行返回（虚假唤醒），必须用循环重新检查条件。

反例 — if 检查条件（危险）Java
// 危险！虚假唤醒后 count 仍为 0，直接 take 会取到 null
if (count == 0)
    notEmpty.await();
// ↑ 虚假唤醒后直接跑到这里，count 还是 0
E x = (E) items[head];  // 取到 null，NPE 或数据错乱
正例 — while 循环（安全）Java
// 正确：每次被唤醒后重新检查，不满足就继续等
while (count == 0)
    notEmpty.await();

陷阱 2：在锁外调用 signal/await

反例 — 锁外 signalJava
// 忘记加锁，直接 signal → IllegalMonitorStateException
notEmpty.signal();  // 抛异常！

// 正例：signal 必须在 lock/unlock 区间内
lock.lock();
try {
    notEmpty.signal();
} finally { lock.unlock(); }

陷阱 3：signal 用错条件对象——通知了错误队列

反例 — 生产者通知了 notFull 而不是 notEmptyJava
// 生产者放完数据，却通知"队列未满"条件——等待取数据的消费者永远沉睡
notFull.signal();   // BUG：应该是 notEmpty.signal()
多个 Condition 对象的命名务必清晰。建议命名为 notFull/notEmpty， 而不是 cond1/cond2，字面语义直接对应"满足了什么条件"。

陷阱 4：await 后忘记处理中断

await() 会响应线程中断，抛出 InterruptedException。 若方法签名不声明该异常，必须显式捕获处理，不能直接吞掉。

反例 — 吞掉中断Java
try {
    while (count == 0)
        notEmpty.await();
} catch (InterruptedException e) {
    // 吞掉！中断状态丢失，调用方永远感知不到取消信号
}
正例 — 恢复中断状态Java
try {
    while (count == 0)
        notEmpty.await();
} catch (InterruptedException e) {
    // 方案 A：重新设置中断标志，让上层感知
    Thread.currentThread().interrupt();
    break;
    // 方案 B：若不想响应中断，改用 awaitUninterruptibly()
}

7. 进阶：限时等待与超时检测

生产环境中常见"最多等待 N 毫秒，超时返回失败"的需求。 await(long, TimeUnit) 返回 true 表示在超时前被正常 signal 唤醒， false 表示超时。

限时 take — 超时返回 nullJava
@SuppressWarnings("unchecked")
public E poll(long timeout, TimeUnit unit)
        throws InterruptedException {
    long nanos = unit.toNanos(timeout);
    lock.lock();
    try {
        while (count == 0) {
            // awaitNanos 返回剩余纳秒，<= 0 表示已超时
            if ((nanos = notEmpty.awaitNanos(nanos)) <= 0)
                return null;
        }
        E x = (E) items[head];
        items[head] = null;
        if (++head == items.length) head = 0;
        --count;
        notFull.signal();
        return x;
    } finally {
        lock.unlock();
    }
}
注意将 awaitNanos 的返回值赋回 nanos， 每轮循环用剩余时间继续等待——这样可以正确处理虚假唤醒叠加超时的情况。

8. JDK 中的实际应用

类使用的 Condition用途

9. 选择建议：何时用 Condition，何时用 wait/notify

  • 只有一种等待条件，且不需要超时、公平、可中断——synchronized + wait/notifyAll 足够，简单直白
  • 多种等待条件（如生产者/消费者双端都需要阻塞）——Condition 精准唤醒，性能显著优于 notifyAll
  • 需要超时精度到纳秒——awaitNanos
  • 需要公平锁语义——new ReentrantLock(true) + Condition，synchronized 无法做到
  • 需要 awaitUninterruptibly——某些底层资源获取场景，不允许响应中断
面试高频考点
① "Condition 与 Object.wait/notify 的区别"——重点说多条件队列 + 精准唤醒
② "为什么 await 必须用 while 而非 if"——虚假唤醒（spurious wakeup）
③ "ArrayBlockingQueue 如何实现阻塞"——两个 Condition + signal 而非 signalAll
④ "signal 和 signalAll 如何选择"——单一等待语义用 signal，混合等待语义用 signalAll

小结

Condition 是显式锁体系对"等待-通知"机制的全面升级： 一把锁多个条件队列消灭了 notifyAll 惊群， 精准 signal 让线程协作更高效， 丰富的 await 变体满足了超时、不可中断等生产需求。 理解它的内部机制（AQS 条件队列 → 同步队列的节点转移）是真正掌握 Java 并发基础设施的关键一步。
