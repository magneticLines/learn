11. 锁竞争与可伸缩性

理解锁竞争如何成为并发程序的性能天花板，以及三条系统性降低竞争的路径。

1. 为什么锁竞争是可伸缩性的天花板

加入更多 CPU 核心、增大线程池，程序吞吐量本应线性增长。 然而实际上，增加到某个数量之后，吞吐量反而下降甚至变负增长。 根本原因是 Amdahl 定律：

Amdahl 定律
设程序中必须串行执行的比例为 F（0 ≤ F ≤ 1），使用 N 个处理器时的最大加速比为：

Speedup ≤ 1 / (F + (1-F)/N)

当 N → ∞，加速比趋近 1/F。 哪怕只有 10% 的代码必须串行（比如持有全局锁），10 核机器的加速上限就是 5.3×，而非 10×。

锁的作用是把一段代码"强制串行化"。 每当多个线程争抢同一把锁，后来者必须阻塞等待——这些 CPU 时间被白白浪费。 竞争越激烈，被串行化的代码占比越高，可伸缩性越差。

线程数 × 锁竞争激烈度
  ↓
大量线程阻塞 → CPU 上下文切换暴增
  ↓
有效并行度 ≈ 锁的持有者数量（通常 = 1）
  ↓
吞吐量瓶颈，加核无效

2. 锁竞争的两个来源

线程请求一把锁时，有两种结果：

  • 无竞争（uncontended）：锁空闲，JVM 使用偏向锁 / 轻量级 CAS 完成，开销极小。
  • 有竞争（contended）：锁被占用，线程进入内核态挂起（OS 调度），开销是前者的几十到几百倍。

竞争激烈程度由两个因子共同决定：

因子含义类比
请求频率单位时间内多少线程尝试获取锁超市收银台每分钟来多少顾客
持有时长每次加锁后持有多久才释放每位顾客结账需要多长时间

降低竞争的三条路：①减少持有时长、②降低请求频率、③使用非独占同步（读写锁 / 无锁结构）。

3. 减少锁持有时长

"锁的作用域"越小，其他线程等待的时间就越短。 常见做法是把不需要同步的计算移到锁外。

反例：把整个方法体都锁住

BadCache.java反例
// 问题：正则预编译和日志调用根本不涉及共享状态，
// 却被包在锁里，无谓地拉长了持有时长
public synchronized String processAndCache(String key, String input) {
    Pattern p = Pattern.compile("\\d+");          // 纯计算，无需锁
    String result = p.matcher(input).replaceAll("");
    log.info("processed: {}", result);              // I/O，持有锁风险极大
    cache.put(key, result);                        // 这行才真正需要同步
    return result;
}

正例：缩小锁的作用域

GoodCache.java正例
public String processAndCache(String key, String input) {
    // 不需要锁的纯计算，先做完
    Pattern p = Pattern.compile("\\d+");
    String result = p.matcher(input).replaceAll("");
    log.info("processed: {}", result);

    // 锁只保护真正的共享状态写入，作用域极小
    synchronized (this) {
        cache.put(key, result);
    }
    return result;
}
陷阱：不要为了"缩小锁范围"而拆出多个 synchronized 块
如果两个操作对外必须是原子的（例如"检查再写入"），拆开反而引入竞态条件。 缩小作用域的前提是：被移出的代码与被保护的共享状态无关。

4. 降低锁请求频率：锁分解与锁分段

一把粗粒度的锁（如单个 this 对象锁）把不相关的数据捆绑在一起—— 访问字段 A 和访问字段 B 互相阻塞，但它们之间根本不存在数据依赖。 解法是锁分解（Lock Splitting）：每组独立状态使用各自的锁。

反例：一把锁保护所有字段

UserService.java反例 — 锁分解前
public class UserService {
    // 两组完全独立的状态，却共享同一把 this 锁
    private final Map<Long, User> users = new HashMap<>();
    private final Map<Long, Order> orders = new HashMap<>();

    public synchronized void addUser(User u) {  users.put(u.getId(), u); }
    public synchronized void addOrder(Order o) { orders.put(o.getId(), o); }
    // addUser 和 addOrder 互相阻塞，毫无必要
}

正例：锁分解

UserService.java正例 — 锁分解后
public class UserService {
    private final Object userLock  = new Object();
    private final Object orderLock = new Object();
    private final Map<Long, User>  users  = new HashMap<>();
    private final Map<Long, Order> orders = new HashMap<>();

    public void addUser(User u) {
        synchronized (userLock) { users.put(u.getId(), u); }
    }
    public void addOrder(Order o) {
        synchronized (orderLock) { orders.put(o.getId(), o); }
    }
    // 两组操作现在完全并行，互不阻塞
}

锁分段（Lock Striping）

锁分段是锁分解的极端形式：把一个大数据结构（如 HashMap）切成 N 段，每段独立加锁。 ConcurrentHashMap 在 Java 7 默认使用 16 个 Segment（继承自 ReentrantLock）， Java 8 改为 CAS + 每个桶头节点作为锁，粒度更细。

StripedMap.java（概念演示）锁分段原理
public class StripedMap<K, V> {
    private static final int STRIPES = 16;
    private final Object[] locks;
    private final List<Map<K, V>> maps;

    public StripedMap() {
        locks = new Object[STRIPES];
        maps  = new ArrayList<>();
        for (int i = 0; i < STRIPES; i++) {
            locks[i] = new Object();
            maps.add(new HashMap<>());
        }
    }

    private int stripe(Object key) {
        // 将 key 哈希映射到 [0, STRIPES) — 关键：分散均匀
        return Math.abs(key.hashCode() % STRIPES);
    }

    public void put(K key, V value) {
        int s = stripe(key);
        synchronized (locks[s]) {
            maps.get(s).put(key, value);
        }
    }
    // get 同理，只锁对应 stripe
}
锁分段的代价：跨段操作变复杂
若需要对整个 Map 做原子 size() 或 clear()， 必须按固定顺序依次获取全部 16 把锁，否则会引发死锁或看到不一致的中间状态。 ConcurrentHashMap.size() 返回的是近似值，正是这个代价的体现。

5. 非独占同步：ReadWriteLock

读操作天然可以并行——多个线程同时读一份不变的数据不会有问题。 synchronized 和 ReentrantLock 都是独占锁，不区分读写，浪费了读并行的机会。 ReentrantReadWriteLock 将锁拆分为：

  • 读锁（共享）：多个读线程可同时持有，只要没有写线程。
  • 写锁（独占）：只有一个写线程可持有，此时所有读线程阻塞。
ReadWriteCache.java读写锁正例
public class ReadWriteCache<K, V> {
    private final Map<K, V> map = new HashMap<>();
    private final ReentrantReadWriteLock rwl
        = new ReentrantReadWriteLock();
    private final Lock r = rwl.readLock();
    private final Lock w = rwl.writeLock();

    public V get(K key) {
        r.lock();
        try {
            return map.get(key);           // 多读线程可并行到这里
        } finally { r.unlock(); }
    }

    public void put(K key, V value) {
        w.lock();
        try {
            map.put(key, value);           // 写时独占，阻塞所有读
        } finally { w.unlock(); }
    }
}
读写锁适合"读多写少"场景
若写操作频繁，读锁几乎总被写锁抢先阻塞，此时读写锁比普通锁开销更大（内部维护状态更复杂）。 建议仅在读比例超过 80% 时引入。

6. 无锁路径：Atomic 与 CAS

最彻底的"避免竞争"是根本不用锁。 Java 通过 java.util.concurrent.atomic 包提供基于 CPU 原语 CAS（Compare-And-Swap） 的无锁变量。

Counter.javasynchronized vs AtomicLong
// 方式一：synchronized — 每次 ++ 都要独占锁，竞争激烈时慢
private long count = 0;
public synchronized void increment() { count++; }

// 方式二：AtomicLong — CAS 自旋，无内核态切换
private final AtomicLong count = new AtomicLong(0);
public void increment() { count.incrementAndGet(); }

// 方式三：高并发计数用 LongAdder（Java 8+）
// 内部分 Cell 数组，各线程写各自的 Cell，最终 sum() 汇总
// 竞争越激烈，LongAdder 优势越大
private final LongAdder count = new LongAdder();
public void increment() { count.increment(); }
LongAdder vs AtomicLong 选哪个？
AtomicLong 每次操作返回准确的当前值，适合需要实时精确读数的场景（如序列号生成器）。 LongAdder 把写分散到多个 Cell，sum() 有轻微延迟，适合高并发统计计数器（如 QPS、请求总数）。

7. 避免热点字段

锁分解与分段能降低整体竞争，但如果有一个字段被所有操作都访问，它就成了热点（Hot Field）， 无论分解得多细，整体吞吐量仍会被该字段的竞争拖死。

典型热点：size 字段

假设你自己实现了 StripedMap，每次 put/remove 后都更新一个 synchronized 的 size 字段。 这把"全局 size 锁"立刻成为瓶颈——16 个分段的并行写全部争抢同一个 size。

反例 — size 热点反例
private volatile int size = 0;

public void put(K key, V value) {
    int s = stripe(key);
    synchronized (locks[s]) {
        maps.get(s).put(key, value);
    }
    synchronized (this) { size++; }  // ← 所有线程争这把全局锁，分段白搭
}
正例 — 用 LongAdder 消除 size 热点正例
private final LongAdder size = new LongAdder();

public void put(K key, V value) {
    int s = stripe(key);
    synchronized (locks[s]) {
        maps.get(s).put(key, value);
    }
    size.increment();  // 无锁竞争，各线程写各自的 Cell
}

public long size() { return size.sum(); }  // 近似值，读写分离

8. 线程本地化：ThreadLocal 与减少共享

竞争的根本来源是共享可变状态。 如果能让某些对象只属于一个线程，竞争就从根本上消失。 ThreadLocal 是最直接的机制。

ThreadLocal 正例线程本地化
// SimpleDateFormat 是线程不安全的，共享一个会有竞争
// 用 ThreadLocal 给每个线程一个独立实例，零竞争
private static final ThreadLocal<SimpleDateFormat> DATE_FORMAT =
    ThreadLocal.withInitial(() ->
        new SimpleDateFormat("yyyy-MM-dd HH:mm:ss"));

public String format(Date date) {
    // 当前线程独享，无需加锁
    return DATE_FORMAT.get().format(date);
}
ThreadLocal 泄漏陷阱（线程池场景）
线程池中的线程生命周期极长，ThreadLocal 中存储的对象若不手动 remove()， 会一直留在线程的 ThreadLocalMap 中，既占内存又可能被下一个任务"脏读"。 使用后务必在 finally 块调用 threadLocal.remove()。

9. 三条路径对比

策略 典型手段 适用场景 主要代价
减少持有时长 缩小 synchronized 块；锁外做计算 通用，首选改造方向 注意不能破坏原子性
降低请求频率 锁分解、锁分段、ThreadLocal 独立数据分组、多核高并发 跨段原子操作变复杂
非独占 / 无锁 ReadWriteLock、Atomic*、LongAdder 读多写少；简单计数器 读写锁开销高于读少场景
