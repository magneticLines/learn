08. Java 内存模型 JMM 与 happens-before

理解多线程可见性、有序性问题的根源，掌握 JMM 给出的"承诺清单"——happens-before 规则，从此不再靠猜测写并发代码。

1. 从硬件现实出发：为什么需要内存模型

直觉告诉我们："我在线程 A 写了一个值，线程 B 应该能读到最新的值。"但现代硬件刻意打破了这个直觉，原因是性能。

三层"破坏者"：

  • CPU 缓存（L1/L2/L3）：每个核心有私有缓存，写入先落在缓存里，何时同步到内存取决于缓存一致性协议（MESI）的时机。
  • 写缓冲区（Store Buffer）：写操作先进队列，CPU 继续执行后续指令，写缓冲区异步刷入缓存。这让写操作对其他核心暂时不可见。
  • 指令重排序：编译器（javac / JIT）和 CPU 都会在保证单线程语义的前提下乱序执行，两线程之间的"执行顺序"因此可能与源码不同。
类比：把内存想象成一块大白板，每个 CPU 核心盯着自己桌上的草稿纸（缓存）写写画画，偶尔才把结果抄到白板上。线程间共享的是白板，但每个线程却只看自己的草稿纸。JMM 就是规定"什么情况下必须把草稿抄到白板、什么情况下必须去白板读"的规则手册。

三大并发问题

问题根源典型现象
可见性CPU 缓存 / 写缓冲区线程 A 修改了 flag，线程 B 永远看到旧值
有序性指令重排序双重检查锁（DCL）在 JDK 1.4 之前的经典 bug
原子性操作非原子i++ 在多线程下丢失更新

JMM 主要解决的是可见性和有序性；原子性依赖 synchronized、CAS 等手段。

2. JMM 是什么——一份"最小承诺"契约

Java Memory Model（JMM）定义在 JSR-133（Java 5 正式落地），规定了：

  • 哪些操作具有可见性保证（一个线程的写何时对另一个线程的读可见）
  • 哪些操作具有有序性保证（禁止跨某些边界的指令重排）

JMM 不是描述硬件如何工作；它是一个抽象规范，告诉 JVM 实现者：在任何平台（x86、ARM、RISC-V……）上都必须提供这些保证，不足的地方用内存屏障（Memory Barrier）补足。

源码顺序（程序员看到的） │ ▼ 编译器重排序 ←── JMM 在此插入禁止规则 │ ▼ 指令级重排序 ←── JMM 要求 JVM 插入 CPU 屏障 │ ▼ 内存系统重排序（写缓冲区、缓存） │ ▼ 最终执行效果（其他线程观察到的）

JMM 的核心工具就是 happens-before 关系。

3. happens-before 精解

3.1 定义

如果操作 A happens-before 操作 B，则 JMM 保证：

  • A 的所有写入结果对 B 可见（不会读到 A 之前的旧值）
  • A 在执行顺序上被保证在 B 之前（禁止跨越 A→B 边界的重排）
注意：happens-before 描述的是"可见性与顺序的逻辑承诺"，不是说 A 的机器指令一定在 B 之前执行。JVM 可以自由重排，只要最终效果符合 happens-before 承诺即可。

3.2 八条天然 happens-before 规则（JSR-133）

#规则名含义
1程序顺序规则同一线程内，前面的操作 hb 后面的操作（单线程语义保证）
2监视器锁规则unlock 操作 hb 后续对同一锁的 lock 操作
3volatile 变量规则volatile 写 hb 后续对同一变量的 volatile 读
4线程启动规则Thread.start() hb 该线程内的任何操作
5线程终止规则线程内所有操作 hb 其他线程检测到该线程终止（join() 返回 / isAlive() 为 false）
6线程中断规则interrupt() hb 被中断线程检测到中断的操作
7对象终结规则构造函数结束 hb finalize() 开始
8传递性规则若 A hb B 且 B hb C，则 A hb C

这八条是"免费"的；只要你的代码触发了其中一条，可见性和有序性就由 JVM 保证，不需要额外手段。

4. volatile 与 JMM 内存屏障

volatile 是最轻量的同步原语，它触发规则 3（volatile 变量规则）。JVM 在 volatile 写/读前后各插入两道内存屏障：

volatile 写 ┌───────────────────────────────────┐ │ StoreStore 屏障 ← 写之前 │ 禁止上方普通写与本次 volatile 写重排 │ ★ volatile write │ │ StoreLoad 屏障 ← 写之后 │ 禁止本次 volatile 写与下方 volatile 读重排 └───────────────────────────────────┘ volatile 读 ┌───────────────────────────────────┐ │ LoadLoad 屏障 ← 读之后 │ 禁止本次 volatile 读与下方普通读重排 │ ★ volatile read │ │ LoadStore 屏障 ← 读之后 │ 禁止本次 volatile 读与下方普通写重排 └───────────────────────────────────┘

实战：用 volatile 实现可见性

StopThread.java反例 → 正例
// ❌ 反例：没有 volatile，JIT 可能将 stopRequested 缓存在寄存器
//    导致子线程永远看不到 main 线程的修改，陷入死循环
public class StopThread {
    private static boolean stopRequested = false;

    public static void main(String[] args) throws InterruptedException {
        Thread bg = new Thread(() -> {
            int i = 0;
            while (!stopRequested)  // JIT 可能提升为 if(!stopRequested) while(true)
                i++;
        });
        bg.start();
        Thread.sleep(1000);
        stopRequested = true;  // main 线程的写对 bg 线程不可见！
    }
}

// ✅ 正例：加上 volatile，触发 volatile 变量规则
//    main 的写 hb bg 的读，可见性有保证
private static volatile boolean stopRequested = false;
陷阱：volatile 不保证原子性。volatile int count; count++ 仍然是非线程安全的，因为 count++ 是"读-改-写"三步，volatile 只保证读到最新值，不保证中间步骤的原子性。需要原子性请用 AtomicInteger 或 synchronized。

5. 经典案例：双重检查锁（DCL）与有序性

这是 JMM 有序性保证最著名的教科书案例，曾让无数 Java 高手踩坑。

对象创建的三步分解

Singleton instance = new Singleton(); ────────────────────────────────────── 字节码层面拆分为三步： 1. 分配内存 memory = allocate() 2. 初始化对象 <init>(memory) 3. 赋值引用 instance = memory CPU/JIT 允许将步骤 2 和 3 重排为： 1. 分配内存 3. 赋值引用（instance 非 null，但对象未初始化！） 2. 初始化对象
Singleton.javaDCL 反例 → 正例
// ❌ 反例：JDK 1.4 及以下，instance 赋值可能在初始化之前（步骤 3 先于 2）
//    线程 B 可能拿到一个半初始化的对象
public class Singleton {
    private static Singleton instance;  // ← 没有 volatile

    public static Singleton getInstance() {
        if (instance == null) {                   // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {           // 第二次检查（有锁）
                    instance = new Singleton(); // ← 赋值可能先于初始化
                }
            }
        }
        return instance;  // 线程 B 看到非 null 但字段未初始化的对象
    }
}

// ✅ 正例（JDK 5+）：volatile 写 hb 后续 volatile 读，
//    且 volatile 写前的 StoreStore 屏障禁止 new 的步骤 2 与步骤 3 重排
public class Singleton {
    private static volatile Singleton instance;  // ← 加 volatile

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

// ✅✅ 更推荐：静态内部类（Initialization-on-demand Holder）
//    利用类加载机制天然保证线程安全，无需 volatile，无锁开销
public class Singleton {
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}

6. synchronized 与 happens-before

synchronized 触发规则 2（监视器锁规则）：unlock hb lock。这意味着：

  • 退出同步块时，JVM 会刷新所有在同步块内的写操作到主内存（相当于 StoreLoad 屏障）
  • 进入同步块时，JVM 会使本地缓存失效，强制从主内存读取（LoadLoad 屏障）
Counter.javasynchronized hb 示意
public class Counter {
    private int count = 0;

    // 线程 A 调用 increment 并退出同步块（unlock）
    // → unlock hb 后续对同一锁的 lock
    // → 线程 B 调用 get 进入同步块（lock）一定能看到 count 的最新值
    public synchronized void increment() { count++; }
    public synchronized int  get()       { return count; }
}

// 💡 关键点：必须是同一把锁！
//    如果 increment 用 lockA，get 用 lockB，规则 2 不适用，可见性无保证

7. final 字段的特殊保证

JMM 对 final 字段有额外保证：构造函数结束 hb 其他线程读取 final 字段（规则 7 的延伸）。只要对象引用不在构造函数中"逃逸"，其他线程就能看到正确初始化的 final 字段——即使没有同步。

ImmutablePoint.javafinal 字段安全发布
public final class ImmutablePoint {
    private final int x, y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
        // ← 构造函数结束后，JMM 保证 x、y 对所有线程可见
        // ← 不需要 volatile 或 synchronized
    }

    public int getX() { return x; }
    public int getY() { return y; }
}

// ❌ 陷阱：在构造函数内将 this 发布出去（this 逃逸），破坏 final 保证
public BadPoint(EventSource src) {
    src.register(this);  // ← this 逃逸，其他线程可能在构造完成前看到未初始化的 final 字段
    this.x = 10;
}

8. 利用传递性推导 happens-before 链

规则 8（传递性）是隐藏的"乘法器"。通过 happens-before 链，一个操作可以跨越多个线程、多个变量建立可见性保证。

HBChain.javahappens-before 传递性示例
volatile int v = 0;
int a = 0;  // 普通变量

// 线程 A：
a = 1;           // 操作 1
v = 1;           // 操作 2（volatile 写）

// 线程 B：
int r1 = v;      // 操作 3（volatile 读）
int r2 = a;      // 操作 4（普通读）

// happens-before 链推导：
// 操作1 hb 操作2  ← 程序顺序规则（同线程）
// 操作2 hb 操作3  ← volatile 变量规则
// 操作3 hb 操作4  ← 程序顺序规则（同线程）
// ∴ 操作1 hb 操作4  ← 传递性！
// 结论：线程 B 读到 v==1 时，一定能读到 a==1（即使 a 是普通变量）
实用技巧：当你无法确定某两个操作之间是否有 happens-before 时，沿着"程序顺序 → 锁/volatile/线程启动等规则 → 传递性"逐步推导，能找到则有保证，找不到链则无保证，必须加同步。

9. 常见并发陷阱与 JMM 视角分析

陷阱JMM 根因正确做法
共享 boolean 标志位不加 volatile 无 hb 关系，JIT 可能将读提升（hoisting）导致死循环 加 volatile
DCL 不加 volatile（JDK 5 以前或遗留代码） new 对象的步骤 2/3 重排，导致读到半初始化对象 加 volatile 或改用静态内部类
volatile long/double 漏写 32 位 JVM 上 long/double 读写非原子（两次 32 位操作） 64 位 JVM 已保证原子，但加 volatile 更稳妥
用不同锁保护同一变量 规则 2 要求同一锁，不同锁之间无 hb 确保保护同一共享状态的所有操作使用同一把锁
在构造函数中 this 逃逸 破坏 final 字段的初始化保证 构造完成后再发布引用（工厂方法模式）
认为 volatile 能替代锁 volatile 只保证可见性和有序性，不保证复合操作原子性 复合操作（check-then-act、read-modify-write）需要锁或 CAS

10. JMM 在 JDK 并发工具中的体现

理解 JMM 后，JDK 并发包的设计思路就豁然开朗：

  • AQS（AbstractQueuedSynchronizer）：内部维护一个 volatile int state，所有锁/同步器的获取释放都通过读写 state 建立 hb 关系，把可见性传递给业务代码。
  • ConcurrentHashMap（JDK 8）：桶数组用 volatile 读写，put/get 通过 volatile 建立 hb，无需全表加锁。
  • AtomicInteger：底层 volatile int value + CAS，volatile 保证可见性，CAS 保证原子性。
  • CountDownLatch / Semaphore：基于 AQS，countDown() 的写 hb await() 返回后的读，所以 await 之后能看到 countDown 之前的所有写入。
  • ThreadLocal：每线程独立副本，完全规避共享，根本上消除了 hb 问题。
面试高频："为什么 AQS 要用 volatile state？"——答案就在规则 3：volatile 写 hb 后续 volatile 读，释放锁的线程对 state 的 volatile 写，建立了 hb 后续获取锁线程对 state 的 volatile 读，从而将临界区内的所有写操作的可见性传递过去。

11. 知识点总结

Java 内存模型（JMM） ├── 解决问题 │ ├── 可见性：缓存/写缓冲区导致写操作对其他线程不可见 │ └── 有序性：编译器/CPU 重排序导致多线程执行顺序不可预期 │ ├── 核心工具：happens-before │ ├── 8 条天然规则（程序顺序、监视器锁、volatile、线程启动/终止…） │ └── 传递性：A hb B，B hb C → A hb C（链式推导） │ ├── 主要实现机制 │ ├── volatile → StoreStore / StoreLoad / LoadLoad / LoadStore 屏障 │ └── synchronized → lock 前清空缓存，unlock 后刷写主内存 │ └── 实践原则 ├── 共享变量的读写必须有 hb 关系覆盖 ├── volatile 解决可见性/有序性，锁解决原子性+可见性 └── 推导不出 hb 链 → 必须加同步，不要靠运气
