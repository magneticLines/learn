Part IV 高级主题 · 显式锁

02. Lock / ReentrantLock vs synchronized

理解显式锁的设计动机，掌握它在哪些场景下比内置锁更强大，以及如何安全地使用它。

为什么会有 ReentrantLock？

synchronized 是 Java 最早的内置锁，依赖 JVM 字节码指令 monitorenter / monitorexit 实现。它足够简单，但也因此缺少灵活性——一旦线程在等待锁，什么都做不了：不能中断等待、不能设超时、不能非阻塞地试探加锁。

2004 年 Java 5 发布，java.util.concurrent.locks 包随 JSR-166 一同入库。Doug Lea 设计了 Lock 接口与 ReentrantLock 实现，把"加锁"这件事从 JVM 指令层提升到了 Java API 层，从此可以用代码完全控制锁的行为。

可重入（Reentrant）是什么意思？
同一个线程可以对同一把锁多次加锁而不会死锁。synchronized 天然可重入；ReentrantLock 名字里就带着这个承诺，内部维护一个 持有计数，每次 lock() +1，每次 unlock() -1，降到 0 才真正释放。

Lock 接口核心 API

Lock // 接口
├── lock() // 阻塞直到获得锁
├── lockInterruptibly() // 可被 interrupt 打断
├── tryLock() // 立即返回 boolean，不阻塞
├── tryLock(time, unit) // 限时等待
├── unlock() // 释放锁（必须手动调用！）
└── newCondition() // 返回绑定的 Condition 对象

反例 → 问题 → 正例

反例：忘记在 finally 中释放锁

Java — 错误写法
ReentrantLock lock = new ReentrantLock();

void badMethod() {
    lock.lock();
    // ❌ 如果这里抛出异常，unlock() 永远不会执行
    doSomethingRisky();
    lock.unlock();   // 线程一旦崩溃，其他线程永久阻塞
}
陷阱：锁泄漏（Lock Leak）
与 synchronized 不同，JVM 不会帮你自动释放 ReentrantLock。一旦 unlock() 没被调用，所有等待该锁的线程将永久挂起，进程无法恢复。

正例：lock-try-finally 惯用模式

Java — 正确写法
ReentrantLock lock = new ReentrantLock();

void goodMethod() {
    lock.lock();
    try {
        // ✅ 无论正常返回还是抛出异常，finally 必然执行
        doSomethingRisky();
    } finally {
        lock.unlock();
    }
}

规则：lock() 必须在 try 块之外调用，unlock() 必须在 finally 块中调用。 这是使用显式锁唯一被广泛接受的惯用法。

synchronized 做不到的三件事

1. 可中断的等待（lockInterruptibly）

场景：用户点了"取消"按钮，希望中止正在等锁的线程。

Java — lockInterruptibly 示例
void cancellableTask(ReentrantLock lock) throws InterruptedException {
    // 若等待期间线程被 interrupt()，直接抛出 InterruptedException
    lock.lockInterruptibly();
    try {
        doWork();
    } finally {
        lock.unlock();
    }
}

使用 synchronized 时，interrupt() 对一个正在等待 monitor 的线程完全无效，它会继续阻塞直到获得锁。

2. 限时尝试加锁（tryLock with timeout）

场景：数据库连接池，最多等 200ms，超时就降级或报错，避免线程堆积。

Java — tryLock 限时示例
boolean acquired = lock.tryLock(200, TimeUnit.MILLISECONDS);
if (!acquired) {
    // 未获得锁，执行降级逻辑
    fallback();
    return;
}
try {
    doWork();
} finally {
    lock.unlock();
}

tryLock()（无参数版本）更激进：立即返回，拿到返回 true，拿不到返回 false，完全非阻塞。

3. 公平锁（Fair Lock）

默认情况下，synchronized 和 new ReentrantLock() 都是非公平锁——新来的线程可以"插队"抢锁，吞吐量高，但可能导致某些线程饥饿（starvation）。

Java — 公平锁与非公平锁
// 非公平锁（默认）：吞吐量高，可能饥饿
ReentrantLock unfair = new ReentrantLock();

// 公平锁：FIFO 队列，不会饥饿，但吞吐量明显下降
ReentrantLock fair = new ReentrantLock(true);
公平锁的代价：每次加锁都要维护 CLH 队列（AbstractQueuedSynchronizer 内部的虚拟双向队列），吞吐量通常比非公平锁低 5～10 倍。只有在"等待时间公平"比"总吞吐量"更重要的场景（如打印队列、公平调度）才值得开启。

Condition：比 wait/notify 更精确的等待

synchronized 配套的等待机制是 Object.wait() / notify() / notifyAll()，所有等待线程共用同一个 monitor 队列，notify() 只能随机唤醒一个，无法指定。

ReentrantLock.newCondition() 可以创建多个等待队列，精确控制唤醒哪批线程。经典应用是有界缓冲区（生产者-消费者）：

Java — 有界缓冲区（Condition 精确唤醒）
class BoundedBuffer<T> {
    private final ReentrantLock lock   = new ReentrantLock();
    // 两个独立的等待集：生产者等"有空位"，消费者等"有数据"
    private final Condition      notFull  = lock.newCondition();
    private final Condition      notEmpty = lock.newCondition();
    private final Object[]       items;
    private int head, tail, count;

    BoundedBuffer(int cap) { items = new Object[cap]; }

    public void put(T x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length)
                notFull.await();   // 队列满，生产者挂起
            items[tail] = x;
            if (++tail == items.length) tail = 0;
            ++count;
            notEmpty.signal();     // 精确唤醒一个等待数据的消费者
        } finally {
            lock.unlock();
        }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await();   // 队列空，消费者挂起
            T x = (T) items[head];
            if (++head == items.length) head = 0;
            --count;
            notFull.signal();      // 精确唤醒一个等待空位的生产者
            return x;
        } finally {
            lock.unlock();
        }
    }
}

若用 synchronized + notifyAll() 实现同样逻辑，每次唤醒所有等待线程，大量线程白白竞争后再次挂起，浪费上下文切换开销。Condition.signal() 只唤醒对应等待集中的一个线程，精确高效。

底层原理：AQS（AbstractQueuedSynchronizer）

ReentrantLock 的所有功能都建立在 AbstractQueuedSynchronizer（AQS）之上。AQS 的核心是一个 volatile int state 变量和一条 CLH 变体双向等待队列。

加锁流程（非公平锁）
├── CAS(state, 0, 1) 成功 ──▶ 设置 exclusiveOwnerThread = 当前线程，直接返回
├── CAS 失败（锁被持有）
│ ├── 若 exclusiveOwnerThread == 当前线程 ──▶ state++（可重入）
│ └── 否则 ──▶ 封装成 Node 入队，LockSupport.park() 挂起
释放锁流程
└── state-- ，降到 0 时 ──▶ LockSupport.unpark(队头后继节点的线程)

LockSupport.park/unpark 底层对应 POSIX pthread_mutex + pthread_cond（Linux）或 Windows 的 SetEvent/WaitForSingleObject，比 JVM monitor 的 wait/notify 更轻量，也不需要先持有 monitor。

全面对比：ReentrantLock vs synchronized

特性 synchronized ReentrantLock
可重入 ✅ 自动 ✅ 自动（持有计数）
加锁/解锁方式 JVM 自动（字节码） 手动 lock() / unlock()
可中断等待 ❌ ✅ lockInterruptibly()
限时尝试 ❌ ✅ tryLock(time, unit)
非阻塞尝试 ❌ ✅ tryLock()
公平锁 ❌ ✅ new ReentrantLock(true)
多条件变量 ❌（单一等待集） ✅ newCondition() 任意多个
锁状态查询 ❌ ✅ isLocked() / getQueueLength()
性能（Java 6+） 偏向锁/轻量锁优化后与 ReentrantLock 接近 高竞争场景略优
代码安全性 高（JVM 保证释放） 低（需要 try-finally 纪律）

常见并发陷阱汇总

陷阱 1：在 lock() 之前的代码抛出异常

Java — 另一个常见错误
// ❌ 如果 prepare() 抛异常，lock 还没调用，unlock() 会抛 IllegalMonitorStateException
try {
    prepare();
    lock.lock();   // lock() 必须在 try 块外！
    doWork();
} finally {
    lock.unlock();   // 若 lock() 从未执行，这里会抛异常
}

// ✅ 正确：先 lock()，再 try
prepare();
lock.lock();
try {
    doWork();
} finally {
    lock.unlock();
}

陷阱 2：在 Condition.await() 之后不用 while 循环检查条件

Java — 虚假唤醒（Spurious Wakeup）
// ❌ if 只检查一次，虚假唤醒后直接往下走，可能读到脏数据
if (count == 0) notEmpty.await();

// ✅ while 循环：每次被唤醒都重新检查条件
while (count == 0) notEmpty.await();

操作系统层面存在虚假唤醒（spurious wakeup）现象——线程可能在没有任何 signal() 的情况下被唤醒。Java 规范不保证消除这一现象，因此 await() 的条件判断必须用 while，永远不要用 if。

陷阱 3：tryLock() 获取失败后仍调用 unlock()

Java — tryLock 与 unlock 配对
boolean acquired = lock.tryLock();
try {
    if (!acquired) return;   // ✅ 未获锁直接返回，不进入 try 体
    doWork();
} finally {
    if (acquired) lock.unlock();  // ✅ 只有成功获锁才解锁
}

什么时候选 ReentrantLock，什么时候用 synchronized？

实践建议
优先选 synchronized：代码简洁，JVM 会自动处理异常情况下的锁释放，现代 JVM（Java 6+）经过偏向锁、轻量锁优化后性能已与 ReentrantLock 相当。

遇到以下场景时才切换到 ReentrantLock：
  • 需要可中断的锁等待（响应取消操作）
  • 需要超时机制防止死锁或线程堆积
  • 需要非阻塞的 tryLock 实现无锁降级
  • 需要公平锁防止饥饿
  • 需要多个 Condition 精确唤醒不同等待集
  • 需要通过 isLocked() 等方法查询锁状态（用于监控/调试）

面试高频考点

  • ReentrantLock 和 synchronized 的区别？ — 从可中断、超时、公平、多 Condition 四个维度回答，底层一个是 AQS，一个是 JVM monitor。
  • 为什么 unlock() 要放在 finally 中？ — 显式锁 JVM 不自动释放，异常时若不 finally 会死锁。
  • 公平锁为何吞吐量低？ — 每次都要把新线程加入 FIFO 队列，无法利用 CPU 缓存热线程，上下文切换频繁。
  • Condition.await() 为何要用 while？ — 防止虚假唤醒（OS 层 spurious wakeup）导致条件未满足就继续执行。
  • AQS 的核心数据结构？ — volatile int state + CLH 变体双向队列 + LockSupport.park/unpark。
  • tryLock() 能避免死锁吗？ — 能在一定程度上避免：若两个线程互相持有对方需要的锁，tryLock 超时后可以回退并重试，打破死锁的"持有并等待"条件。
