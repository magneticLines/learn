06. ABA 问题与 AtomicStampedReference

CAS 看起来比较变了，但有时候"没变"只是一种幻觉——ABA 问题正是无锁编程最隐蔽的陷阱之一。

1. CAS 的隐患：它只比较值，不关心历史

上一节讲到，CAS（Compare-And-Swap）的语义是：如果当前值等于期望值，就原子地把它改成新值。 这听起来很完美，但它有一个根本局限——它只比较"现在是什么"，不知道"中间发生了什么"。

试想一个场景：线程 T1 读到变量值是 A，正准备用 CAS 把它改成 C。 就在这个间隙，线程 T2 把它从 A 改成了 B，然后又改回了 A。 T1 醒来，发现值还是 A，CAS 成功执行——但它完全不知道中间那次"B"的存在。

T1: 读到 A ──────────────────────────── CAS(A → C) 成功 ✓（但它不知道中间发生了什么） T2: A → B ── B → A ↑ 这段历史对 T1 完全透明

这就是 ABA 问题：值经历了 A → B → A 的变化，但 CAS 误以为"什么都没变"。 在许多场景下这无关紧要，但在某些数据结构（尤其是链表、栈的无锁实现）中， 这个"幻觉"会导致严重的内存错误。

2. 一个经典崩溃场景：无锁栈

无锁栈使用 CAS 来做 pop 操作。设栈顶链为 A → B → C，T1 和 T2 并发 pop：

危险场景（ABA 导致链表断裂）
① T1 读栈顶 head = A，准备 CAS(A → B)，但被调度器暂停
② T2 连续 pop 出 A、B；再 push A（A 被重用，next 指针已变成 null 或其他节点）
③ T1 恢复，CAS(A → B) 成功——但此时 A.next 已不再是 B，而是 null 或悬空指针
④ 节点 C 从栈中消失，内存泄漏或访问非法地址

3. 反例：仅用 AtomicReference 实现无锁栈

BadLockFreeStack.java — 存在 ABA 问题
public class BadLockFreeStack<T> {

    private static class Node<T> {
        T value;
        Node<T> next;
        Node(T v) { this.value = v; }
    }

    // 只保存引用，没有版本信息——ABA 风险所在
    private final AtomicReference<Node<T>> head = new AtomicReference<>();

    public void push(T val) {
        Node<T> newNode = new Node<>(val);
        while (true) {
            Node<T> cur = head.get();
            newNode.next = cur;
            if (head.compareAndSet(cur, newNode)) return;
        }
    }

    public T pop() {
        while (true) {
            Node<T> cur = head.get();
            if (cur == null) return null;
            // ❌ 如果 cur 是被重用的节点，cur.next 已不可信
            if (head.compareAndSet(cur, cur.next)) return cur.value;
        }
    }
}

问题根源：AtomicReference 的 CAS 用的是对象引用地址（即 ==）， 如果同一个节点对象被 pop 后重新 push，地址完全一样，CAS 无法感知节点的 next 已经改变。

4. 本质解法：给值打上"版本戳"

解决 ABA 的核心思路是：把"值有没有变"转化为"(值, 版本号) 对有没有变"。 即使值从 A 变回 A，版本号也会单调递增，使得旧的期望对 (A, v1) 无法匹配新的实际对 (A, v3)。

旧方案：CAS 比较 value A → B → A → CAS(A) 成功（错误） 新方案：CAS 比较 (value, stamp) (A,1) → (B,2) → (A,3) → CAS((A,1)) 失败（正确）

Java 提供了两个类来实现这个思路：

  • AtomicStampedReference<V>：stamp 是 int 版本号，每次修改手动递增
  • AtomicMarkableReference<V>：mark 是 boolean，只能标记"是否被修改过"，不能区分多次修改
为什么 AtomicMarkableReference 不够用？
它只有 true/false 两个状态。A → B → A 后 mark 依然是 true，无法区分第一次和第二次出现的 A。 只适用于"标记删除"等一次性场景（如跳表的逻辑删除节点）。

5. AtomicStampedReference 核心 API

AtomicStampedReference — 关键方法签名
// 构造：初始值 + 初始版本号
new AtomicStampedReference<>(initialRef, initialStamp);

// 读取当前引用
V getReference();

// 读取当前版本号
int getStamp();

// 同时读取引用和版本号（将 stamp 写入 int[] 数组中）
V get(int[] stampHolder);

// 核心：四元 CAS，同时比较引用和版本号，同时更新二者
boolean compareAndSet(
    V    expectedRef,   // 期望引用
    V    newRef,        // 新引用
    int  expectedStamp, // 期望版本号
    int  newStamp        // 新版本号
);

// 无条件设置（不走 CAS，直接覆盖）
void set(V newRef, int newStamp);
底层实现：AtomicStampedReference 内部有一个私有静态类 Pair<T>， 将 (reference, stamp) 封装为单个不可变对象， 再用 Unsafe.compareAndSwapObject 对这个 Pair 对象做原子替换。 本质上仍是单次 CAS，只是比较的粒度从"单个值"变成了"(值+版本)的包装对象"。

6. 正例：演示 ABA 问题被正确阻断

ABADemo.java
import java.util.concurrent.atomic.AtomicStampedReference;

public class ABADemo {

    public static void main(String[] args) throws InterruptedException {

        // 初始值 "A"，版本号 1
        AtomicStampedReference<String> ref =
            new AtomicStampedReference<>("A", 1);

        // T1：先记录快照，再睡眠，等待 T2 制造 ABA
        Thread t1 = new Thread(() -> {
            int[] stampHolder = new int[1];
            String snapshot = ref.get(stampHolder);
            int    snapshotStamp = stampHolder[0];

            System.out.println("T1 读到: " + snapshot + ", stamp=" + snapshotStamp);

            try { Thread.sleep(500); }
            catch (InterruptedException e) { Thread.currentThread().interrupt(); }

            // T2 已完成 A→B→A，此时版本号是 3，而 T1 快照版本是 1
            boolean ok = ref.compareAndSet(snapshot, "C", snapshotStamp, snapshotStamp + 1);
            System.out.println("T1 CAS 结果: " + ok);
            // 打印: T1 CAS 结果: false  ← ABA 被正确阻断！
        });

        // T2：制造 A → B → A，版本号从 1 递增到 3
        Thread t2 = new Thread(() -> {
            // A → B（stamp: 1 → 2）
            ref.compareAndSet("A", "B", 1, 2);
            System.out.println("T2: A→B, stamp=2");

            // B → A（stamp: 2 → 3）
            ref.compareAndSet("B", "A", 2, 3);
            System.out.println("T2: B→A, stamp=3");
        });

        t1.start();
        try { Thread.sleep(100); } catch (InterruptedException e) {}
        t2.start();
        t1.join();
        t2.join();
    }
}

关键点：T1 读取时版本号是 1，T2 操作后版本号变为 3， 尽管引用值都是 "A"，但 compareAndSet("A", "C", 1, 2) 中 期望 stamp=1 与实际 stamp=3 不匹配，CAS 返回 false，ABA 被阻断。

7. 正例：用 AtomicStampedReference 修复无锁栈

SafeLockFreeStack.java — 版本号防 ABA
import java.util.concurrent.atomic.AtomicStampedReference;

public class SafeLockFreeStack<T> {

    private static class Node<T> {
        final T       value;
        Node<T>       next;
        Node(T v) { this.value = v; }
    }

    // 用 AtomicStampedReference 封装栈顶，stamp 作为操作计数器
    private final AtomicStampedReference<Node<T>> head =
        new AtomicStampedReference<>(null, 0);

    public void push(T val) {
        Node<T> newNode = new Node<>(val);
        int[] stampHolder = new int[1];
        while (true) {
            Node<T> cur = head.get(stampHolder);
            newNode.next = cur;
            // 每次操作 stamp +1，即使节点被重用，版本号也不同
            if (head.compareAndSet(cur, newNode, stampHolder[0], stampHolder[0] + 1))
                return;
        }
    }

    public T pop() {
        int[] stampHolder = new int[1];
        while (true) {
            Node<T> cur = head.get(stampHolder);
            if (cur == null) return null;
            // 同时比较节点引用和版本号，缺一不可
            if (head.compareAndSet(cur, cur.next, stampHolder[0], stampHolder[0] + 1))
                return cur.value;
        }
    }
}

8. 三种方案对比

方案 能否防 ABA 版本粒度 适用场景
AtomicReference ❌ 不能 无版本信息 不涉及 ABA 的无锁操作
AtomicMarkableReference 部分能（一次性） boolean（只有两态） 逻辑删除标记（跳表、链表节点软删除）
AtomicStampedReference ✅ 完全能 int（可无限递增） 需要区分多次修改的无锁数据结构

9. 性能代价与使用建议

AtomicStampedReference 内部每次 CAS 都会创建一个新的 Pair 对象， 在高并发场景下会带来明显的 GC 压力。它的吞吐量比 AtomicInteger 低得多。

常见陷阱：int stamp 溢出
int 最大值约 21 亿，理论上很难溢出；但如果用 AtomicInteger 的 getAndIncrement() 作为 stamp 来源，需要考虑正负边界。 实践中通常没问题，但要避免用 System.currentTimeMillis() 作为 stamp， 因为时钟回拨会导致版本号倒退。
何时真的需要 AtomicStampedReference？
实际工程中，ABA 问题多见于以下场景：
① 无锁链表 / 无锁栈：节点被 pop 后放入对象池，再 push 回来
② 内存池 / 对象池：同一块内存地址被反复分配/回收
③ 数据库乐观锁（类比）：version 字段防止并发更新覆盖
如果不涉及节点复用，普通 AtomicReference 已经足够。
Java 9+ 的替代方案
java.util.concurrent 内部实现（如 ConcurrentLinkedQueue）倾向于 用"对象身份"（每次 new 一个新节点）而非版本号来规避 ABA， 用内存换并发正确性，比维护 stamp 更简单。 在写自定义无锁结构时，这是值得优先考虑的方向。

10. 知识连接

无锁并发全景 ├── CAS 原语（上节：Unsafe.compareAndSwapXxx） │ └── 局限：只比较值，不记录历史 ├── ABA 问题（本节） │ ├── AtomicStampedReference — int 版本号 │ └── AtomicMarkableReference — boolean 标记 ├── AQS（下节） │ └── 用 volatile state + CAS 构建锁/同步器 └── 内存模型 └── volatile 保证可见性，CAS 保证原子性

小结

  • ABA 问题：值经历 A→B→A 后，CAS 误以为无事发生，可能引发链表断裂等严重 bug
  • 根本原因：CAS 只比较"当前值"，不关心"中间历史"
  • 解决思路：给每次修改打上单调递增的版本戳，(值, stamp) 二元组替代裸值比较
  • AtomicStampedReference 通过内部 Pair<T> 对象的原子替换实现四元 CAS
  • 代价是 GC 压力；如果能做到"每次 new 新节点"，可以完全规避 ABA 而不引入 stamp
