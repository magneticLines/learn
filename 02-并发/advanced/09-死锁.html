09. 死锁的成因与避免

理解死锁的四个必要条件，掌握锁顺序、超时获取、资源图检测等系统性规避策略。

什么是死锁

死锁（Deadlock） 是并发程序中最令人头疼的活跃性故障：两个或多个线程相互持有对方所需的锁，永远卡住等待，程序表面上"活着"却什么也做不了。

用一个生活类比理解：两辆车在单行道对头相遇，A 说"你先倒"，B 说"你先倒"，双方都不动，道路永久堵死。

死锁与 死循环 的区别：死循环会持续消耗 CPU；死锁线程会进入 BLOCKED 状态，CPU 几乎为零，但业务请求全部挂起。监控时注意区分。

Coffman 四个必要条件

1971 年 Coffman 等人证明：死锁的出现当且仅当以下四个条件同时满足。破坏任意一个，死锁就不会发生。

条件含义Java 中的体现
互斥（Mutual Exclusion） 资源同一时刻只能被一个线程持有 synchronized、ReentrantLock
占有并等待（Hold & Wait） 线程持有至少一个资源，同时等待另一个 先锁 A，再试图锁 B
不可剥夺（No Preemption） 锁不能被强制夺走，只能由持有者主动释放 synchronized 无法被抢占
循环等待（Circular Wait） 线程形成一个等待环：T1 等 T2，T2 等 T1 两线程以相反顺序获取同两把锁
面试高频考点：能否背出四个条件只是起点，面试官更想听到"破坏哪个条件最实际，怎么破坏"。

反例：经典转账死锁

下面是教科书级别的死锁：两个账户相互转账，两个线程以相反顺序加锁。

DeadlockDemo.java — 反例
public class Account {
    private String id;
    private double balance;

    public synchronized void transfer(Account target, double amount) {
        // 先锁 this，再锁 target — 顺序由调用方决定，不一致时必死锁
        synchronized (target) {
            if (this.balance >= amount) {
                this.balance -= amount;
                target.balance += amount;
            }
        }
    }
}

// 线程 1：accountA.transfer(accountB, 100)
// 线程 2：accountB.transfer(accountA, 100)
//
// 执行时序：
//   T1 获得 lockA，等待 lockB
//   T2 获得 lockB，等待 lockA
//   → 循环等待，永久阻塞
线程状态图
T1 ──持有──▶ lockA ──等待──▶ lockB ◀──持有── T2 │ T1 ◀──等待── lockA ◀──持有── T2 ──等待──┘
    循环等待 → 死锁

策略一：固定锁顺序（破坏循环等待）

最简单、开销最小的方案：所有线程按照相同的全局顺序获取锁，循环等待便不可能形成。

对于"锁任意两个对象"的场景，可以用 System.identityHashCode() 给对象排序，让所有线程都先锁编号小的那个。

OrderedLockTransfer.java — 正例
public class BankService {

    // 用于打破极罕见的 hashCode 相等情形（需要第三把"仲裁锁"）
    private static final Object TIE_LOCK = new Object();

    public void transfer(Account from, Account to, double amount) {
        int fromHash = System.identityHashCode(from);
        int toHash   = System.identityHashCode(to);

        if (fromHash < toHash) {
            // 始终先锁 hashCode 小的
            synchronized (from) {
                synchronized (to) { doTransfer(from, to, amount); }
            }
        } else if (fromHash > toHash) {
            synchronized (to) {
                synchronized (from) { doTransfer(from, to, amount); }
            }
        } else {
            // 极罕见：hashCode 相同，用全局仲裁锁确保同一时刻只有一个线程执行
            synchronized (TIE_LOCK) {
                synchronized (from) {
                    synchronized (to) { doTransfer(from, to, amount); }
                }
            }
        }
    }

    private void doTransfer(Account from, Account to, double amount) {
        if (from.getBalance() >= amount) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}
如果对象有天然的全局唯一 ID（如数据库主键），直接按 ID 排序比 identityHashCode 更可靠，不存在 hash 碰撞问题。

策略二：使用 tryLock 超时（破坏不可剥夺）

ReentrantLock.tryLock(timeout) 允许线程在等待超时后主动放弃已持有的锁并重试，从而打破"不可剥夺"条件。

TryLockTransfer.java — 正例
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class AccountV2 {
    private final ReentrantLock lock = new ReentrantLock();
    private double balance;

    public boolean transfer(AccountV2 target, double amount)
            throws InterruptedException {

        long deadline = System.nanoTime() + TimeUnit.MILLISECONDS.toNanos(200);

        while (true) {
            if (this.lock.tryLock()) {
                try {
                    if (target.lock.tryLock()) {
                        try {
                            // 两把锁都拿到了，安全执行
                            if (this.balance >= amount) {
                                this.balance   -= amount;
                                target.balance += amount;
                            }
                            return true;
                        } finally {
                            target.lock.unlock();
                        }
                    }
                } finally {
                    this.lock.unlock();  // 拿不到 target 的锁时主动释放 this 的锁
                }
            }

            if (System.nanoTime() > deadline) {
                return false;  // 超时，返回失败，由调用方决定重试或报错
            }

            // 随机退避，减少活锁（两个线程同时重试的概率）
            Thread.sleep((long)(Math.random() * 10));
        }
    }
}
陷阱：tryLock 可能引入活锁。 两个线程同时放弃并以相同节奏重试，永远拿不齐两把锁。解决方案：加入随机退避时间（Jitter），打破节奏同步。

策略三：开放调用（减少锁的持有范围）

很多死锁发生在持有锁的情况下调用外部方法，而外部方法内部也会获取锁。这种嵌套调用极难分析。 开放调用（Open Call） 指在调用外部方法时不持有任何锁——缩小锁的临界区，只在真正需要保护共享状态时才加锁。

OpenCall.java — 反例 vs 正例
// ===== 反例：持有锁调用外部方法 =====
class Taxi {
    private Point location;
    private final Dispatcher dispatcher;

    // 持有 this 锁时调用 dispatcher.notifyAvailable(this)
    // dispatcher 内部也有锁 → 嵌套加锁，容易死锁
    public synchronized void setLocation(Point p) {
        this.location = p;
        if (atDestination())
            dispatcher.notifyAvailable(this);  // ← 危险：持锁调用外部
    }
}

// ===== 正例：先读出状态，释放锁后再调用外部 =====
class TaxiV2 {
    private Point location;
    private final Dispatcher dispatcher;

    public void setLocation(Point p) {
        boolean arrived;
        synchronized (this) {
            this.location = p;
            arrived = atDestination();
        }  // ← 先释放锁

        if (arrived)
            dispatcher.notifyAvailable(this);  // ← 锁外调用外部方法，安全
    }
}

策略四：死锁检测与诊断

对于已经发生的死锁（或怀疑正在发生），有以下诊断手段：

jstack 线程转储

这是最常用的生产环境死锁诊断工具，JDK 自带，无需额外依赖。

命令行
# 获取 JVM 进程 PID
jps -l

# 输出线程转储，自动标记死锁
jstack <pid>

# 典型输出片段：
# Found one Java-level deadlock:
# "Thread-1":
#   waiting to lock monitor 0x... (object 0x..., Account@7852e922)
#   which is held by "Thread-0"
# "Thread-0":
#   waiting to lock monitor 0x... (object 0x..., Account@4e25154f)
#   which is held by "Thread-1"

代码中主动检测

ThreadMXBean 可在程序内部定时扫描死锁，适合接入监控告警系统。

DeadlockDetector.java
import java.lang.management.ManagementFactory;
import java.lang.management.ThreadMXBean;

public class DeadlockDetector {

    public static boolean hasDeadlock() {
        ThreadMXBean bean = ManagementFactory.getThreadMXBean();
        long[] ids = bean.findDeadlockedThreads();  // 包含 ReentrantLock
        if (ids != null) {
            ThreadInfo[] infos = bean.getThreadInfo(ids, true, true);
            for (ThreadInfo info : infos)
                System.err.println("死锁线程: " + info.getThreadName());
            return true;
        }
        return false;
    }
}

更多常见死锁模式

协作对象之间的死锁

当两个对象各自有锁，并且彼此持有对方的引用相互调用时，就形成了协作对象死锁——这种死锁在代码评审时极难发现，因为每个类单独看都是正确的。

规则：如果一个加锁方法调用了另一个对象的加锁方法，要高度警觉。 考虑是否可以改为开放调用，或重新设计数据流向。

数据库事务死锁

数据库锁与 Java 锁可以交织：Java 线程持有 Java 锁等待数据库行锁，另一个事务持有行锁等待 Java 锁被释放后才能提交。这类死锁跨越两个锁系统，极难排查。

对策：不要在持有 Java 锁的情况下执行数据库操作，先释放 Java 锁，再操作数据库。

线程池诱发的死锁

单线程池中，任务 A 提交了任务 B 并等待 B 的结果，但线程池已满，B 永远无法执行——这是"资源耗尽型死锁"，不是锁导致的，但现象完全一样。

ThreadPoolDeadlock.java — 反例
ExecutorService pool = Executors.newSingleThreadExecutor();

pool.submit(() -> {
    // 任务 A：提交任务 B 并阻塞等待
    Future<String> futureB = pool.submit(() -> "B的结果");
    // ↑ 单线程池满，B 永远等不到执行机会
    return futureB.get();  // A 永久阻塞 → 死锁
});

解决：为相互依赖的任务使用不同的线程池，或改用异步回调（CompletableFuture）避免 get() 阻塞。

策略总结对比

策略破坏的条件适用场景代价
固定锁顺序 循环等待 锁的集合固定，可枚举排序 低，无额外开销
tryLock + 退避 不可剥夺 锁集合动态、无法预先排序 中，需处理重试逻辑
开放调用 减少持锁范围 持锁调用外部方法的场景 低，需重构临界区
无锁化（CAS/不可变） 互斥（根本消除） 高并发读多写少 设计复杂度较高
jstack / ThreadMXBean —（检测而非预防） 生产排查、监控告警 诊断工具，无运行代价

面试要点与常见陷阱

  • 四个条件背了没用：面试官会追问"如何破坏第 N 个条件，给出代码示例"，要能说清楚。
  • synchronized 无法中断：等待 synchronized 的线程无法响应中断，tryLock(timeout) 是唯一可以给等待设置超时的方案。
  • 死锁 vs 活锁：死锁线程完全停止；活锁线程不断尝试但都失败，CPU 占用高。下一节会讲。
  • 不要在循环里无条件 tryLock：没有退避的自旋重试在争用时会造成 CPU 爆满。
  • 数据库 + Java 锁混用：这是最难排查的死锁类型，务必规避持 Java 锁时做数据库 I/O。
  • 线程转储是第一现场：生产出现疑似死锁时，先 jstack 保存现场，再重启服务，顺序不能反。
