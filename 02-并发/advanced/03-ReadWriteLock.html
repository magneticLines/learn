03. ReentrantReadWriteLock 读写锁

允许多个读者并发访问，同时保证写者独占——用"读多写少"的业务特征换吞吐量。

为什么需要读写锁

互斥锁（synchronized / ReentrantLock）的语义是：任一时刻只有一个线程持锁。 这对"写"操作是必要的——写会改变共享状态，必须排他。 但对"读"操作来说，多个线程同时读不会产生数据竞争，互斥锁把本可并发的读操作全部串行化了，白白浪费了 CPU。

读写锁把锁拆成两把：读锁（共享锁） 和 写锁（排他锁）， 只要没有写者，所有读者可以同时持有读锁；一旦有写者，读者和其他写者都必须等待。

典型场景：缓存、配置中心、数据字典——读 QPS 远大于写 QPS。 把互斥锁换成读写锁，读并发数可直接等于 CPU 核心数，吞吐量成倍提升。

三条核心规则

当前持锁状态 新请求：读锁 新请求：写锁
无锁 允许 允许
有线程持读锁（可多个） 允许（共享） 阻塞（必须等所有读者释放）
有线程持写锁（仅一个） 阻塞 阻塞

记忆口诀：读读不互斥，读写、写写互斥。

AQS 内部实现原理

ReentrantReadWriteLock 基于 AbstractQueuedSynchronizer（AQS）实现， 它用一个 int state（32 位）同时编码读锁和写锁的持有计数：

state（32 bit） ├── 高 16 位 → 共享读锁持有线程数（读锁计数） └── 低 16 位 → 独占写锁重入次数（写锁计数）

读锁加锁：state + (1 << 16)，CAS 更新高 16 位； 写锁加锁：state + 1，CAS 更新低 16 位，且要求高 16 位为 0（没有读者）。

这就是为什么读写锁最多支持 65535 个并发读线程或 65535 次写重入——超出会抛 Error。实践中不会触达此限制。

API 速览

JavaReentrantReadWriteLock 结构
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();

Lock readLock  = rwLock.readLock();   // 共享锁
Lock writeLock = rwLock.writeLock();  // 排他锁

// 读锁操作（与 ReentrantLock 接口一致）
readLock.lock();
readLock.unlock();
readLock.tryLock();
readLock.tryLock(100, TimeUnit.MILLISECONDS);

// 写锁额外能力：支持降级为读锁（见后文）
writeLock.lock();
writeLock.unlock();

// 监控
rwLock.getReadLockCount();    // 当前读锁持有数
rwLock.isWriteLocked();        // 写锁是否被持有
rwLock.getWriteHoldCount();    // 当前线程写锁重入次数

正例：线程安全的本地缓存

这是读写锁最经典的用法——用读锁保护批量读操作，用写锁保护更新操作。

JavaReadWriteCache.java
public class ReadWriteCache<K, V> {

    private final Map<K, V> cache = new HashMap<>();
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock r = rwLock.readLock();
    private final Lock w = rwLock.writeLock();

    public V get(K key) {
        r.lock();                  // 多个读线程可同时进入
        try {
            return cache.get(key);
        } finally {
            r.unlock();
        }
    }

    public void put(K key, V value) {
        w.lock();                  // 排他：读者和写者全部阻塞
        try {
            cache.put(key, value);
        } finally {
            w.unlock();            // 务必在 finally 里解锁
        }
    }

    /** 批量预热：一次写锁保护整个批量操作，比逐条 put 更高效 */
    public void putAll(Map<K, V> data) {
        w.lock();
        try {
            cache.putAll(data);
        } finally {
            w.unlock();
        }
    }
}

锁降级：写锁 → 读锁

锁降级指：持有写锁的线程，先获取读锁，再释放写锁，从而"降级"到读锁状态。 这样可以保证：写完数据后，当前线程能立刻以读锁读到自己刚写的值， 同时不让其他写线程在这个窗口内修改数据，避免"写完立刻被别人覆盖再读"的竞态。

Java锁降级模板（官方推荐写法）
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock r = rwLock.readLock();
Lock w = rwLock.writeLock();
volatile boolean cacheValid = false;
Object data;

void processCachedData() {
    r.lock();
    if (!cacheValid) {
        // 必须先释放读锁才能申请写锁（不能锁升级！）
        r.unlock();
        w.lock();
        try {
            if (!cacheValid) {  // 二次检查，防止重复刷新
                data = loadFromDB();
                cacheValid = true;
            }
            // ① 持有写锁期间先获取读锁 —— 降级开始
            r.lock();
        } finally {
            // ② 释放写锁，此时仍持有读锁
            w.unlock();
        }
    }
    try {
        use(data);             // ③ 以读锁保护后续使用
    } finally {
        r.unlock();           // ④ 最终释放读锁
    }
}
降级的关键顺序：获取读锁 → 释放写锁 → 使用数据 → 释放读锁。 先释放写锁再获取读锁则不算降级，中间有窗口期可被其他写线程抢占。

并发陷阱与反例

陷阱 1：试图"锁升级"——死锁

ReentrantReadWriteLock 不支持锁升级（读锁 → 写锁）。 如果持有读锁的线程尝试再获取写锁，写锁必须等所有读锁释放，而自己又是读者之一，永远等不到——死锁。

Java — 反例锁升级导致死锁
r.lock();           // 持有读锁
try {
    // ... 读操作 ...
    w.lock();       // ❌ 死锁！写锁等待读锁全部释放，但自己就是读者
    try {
        // ... 永远到不了这里 ...
    } finally { w.unlock(); }
} finally { r.unlock(); }

正确做法：先完整释放读锁，再重新获取写锁（中间有窗口期，需二次检查状态）。

陷阱 2：读锁内忘记 finally——锁泄漏

读锁和写锁都必须在 finally 块中释放。 读锁泄漏会导致后续所有写操作永久阻塞，表现为系统写功能"卡死"。

Java — 反例 vs 正例锁泄漏
// ❌ 反例：异常时读锁永远不释放
r.lock();
V result = cache.get(key);  // 若这里抛异常……
r.unlock();                    // ← 这行永远不会执行！

// ✅ 正例：finally 保证一定释放
r.lock();
try {
    return cache.get(key);
} finally {
    r.unlock();
}

陷阱 3：写者饥饿（非公平模式）

默认非公平模式下，读锁是"连续可插队"的：只要当前有读者，新来的读者可以不排队直接进入。 如果读请求持续不断，写者可能长时间抢不到写锁，造成写者饥饿。

解决方案：构造时传入 true 开启公平模式，或改用 JDK 8+ 的 StampedLock（乐观读，写者优先）。

Java公平模式
// 公平模式：所有线程按 FIFO 排队，写者不会饿死
// 代价：读读操作也要排队，吞吐量低于非公平模式
ReentrantReadWriteLock fairLock = new ReentrantReadWriteLock(true);

陷阱 4：读锁不支持 Condition

readLock().newCondition() 会直接抛出 UnsupportedOperationException。 Condition 只能配合写锁（排他锁）使用，原因是共享锁语义下无法安全地实现等待/唤醒队列。 如果需要在读写锁中使用条件等待，只能在写锁侧创建 Condition。

进阶示例：带 LoadingCache 语义的安全缓存

实际业务中，缓存未命中时需要从数据库加载，同时要防止缓存击穿（同一 key 同时大量请求打到 DB）。 下面结合读写锁和二次检查实现这一目标：

JavaLoadingReadWriteCache.java
public class LoadingReadWriteCache<K, V> {

    private final Map<K, V> cache = new HashMap<>();
    private final ReentrantReadWriteLock rwLock
            = new ReentrantReadWriteLock();
    private final Function<K, V> loader;  // DB 加载函数

    public LoadingReadWriteCache(Function<K, V> loader) {
        this.loader = loader;
    }

    public V get(K key) {
        // 第一次：读锁快速检查
        rwLock.readLock().lock();
        try {
            V val = cache.get(key);
            if (val != null) return val;  // 命中，直接返回
        } finally {
            rwLock.readLock().unlock();
        }

        // 未命中：升级为写锁（先完整释放读锁）
        rwLock.writeLock().lock();
        try {
            // 第二次检查：防止多个线程同时到达写锁时重复加载
            V val = cache.get(key);
            if (val == null) {
                val = loader.apply(key);  // 只有真正未命中才调 DB
                cache.put(key, val);
            }
            return val;
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
注意：loader.apply(key) 在写锁内执行，如果 DB 调用耗时很长，写锁会长时间阻塞所有读操作。 生产环境通常改用 ConcurrentHashMap.computeIfAbsent 或 Guava LoadingCache， 它们以更细粒度的分段锁规避此问题。读写锁版本适合 DB 较快、并发量不极端的场景。

面试高频考点

  • 读写锁比互斥锁性能一定更好吗？不一定。写操作占比高时，读写锁因维护状态开销反而更慢；读多写少时才有优势。
  • ReentrantReadWriteLock 能锁升级吗？不能，会死锁。只支持锁降级（写→读）。
  • state 高低 16 位设计的好处？用单个 CAS 原子操作同时读写两个计数，避免额外同步开销。
  • 为什么读锁不支持 Condition？共享锁可被多个线程同时持有，Condition.await() 要求"原子性释放锁并入队等待"，共享语义下此语义无法正确定义。
  • StampedLock 和 ReentrantReadWriteLock 怎么选？StampedLock 无重入、无 Condition、API 复杂，但乐观读不阻塞写者，适合极高读并发；日常业务首选 RRWL。

与其他知识点的联系

  • AQS 体系：本节讲的「高 16 位读锁计数 + 低 16 位写锁计数」正是 ReentrantReadWriteLock 对 AQS state 字段的定制解释方式；对比上一节 ReentrantLock 直接把整个 state 当作重入计数，可以看出 AQS 的通用之处——不同子类只是重写 tryAcquire/tryAcquireShared 等方法，赋予 state 不同的业务含义。
  • JMM / happens-before：写锁释放时对共享数据的修改，能被之后获取读锁的线程看到，这依赖的底层机制和 ReentrantLock 一样——AQS 的 state 是 volatile 的，写锁 unlock() 里的 state 写入建立了 happens-before 关系。锁降级模板里"先获取读锁再释放写锁"的顺序，本质上就是要保证这条可见性链条不断裂。
  • CAS 与原子类：读锁的加锁不是简单加锁计数，而是用 CAS 对高 16 位做原子递增；多个读线程并发进入时依赖的正是 CAS 的「比较并交换」语义，避免了给整个 state 加互斥锁再修改的开销，这也是读写锁能做到"读读不互斥"又不破坏计数一致性的关键。
  • ConcurrentHashMap：本节最后提到生产环境更常用 ConcurrentHashMap.computeIfAbsent 或 Guava LoadingCache 而非读写锁包 HashMap，原因是 ConcurrentHashMap 用分段（CAS + synchronized 锁桶）的方式把锁粒度降到了单个哈希桶级别，比"整个 Map 共用一把读写锁"的并发度更细，这是理解"何时该用读写锁、何时该换更细粒度并发容器"的分界线。
  • Spring 场景应用：Spring 应用里本地缓存刷新（如配置中心客户端缓存最新配置）常用读写锁：查询走读锁保证高并发读性能，配置变更走写锁并结合"锁降级"模式——写完新配置后立刻用读锁读出来返回给调用方，避免写锁释放和读锁获取之间的窗口期被其他更新请求抢先修改。
  • 分布式锁对比：Redisson 提供了 RReadWriteLock，语义上模仿本节的读写锁模型，但受制于网络延迟和 Redis 单点特性，"读读并发"的吞吐提升不如 JVM 内置读写锁明显；而锁降级、非公平写者饥饿这些细节问题在分布式场景下更难精确控制，通常只在业务能容忍一定不一致的场景才用分布式读写锁。
