07. AQS 抽象队列同步器

ReentrantLock、Semaphore、CountDownLatch 背后共享的骨架——一个用 CAS + 等待队列解决「谁能持有资源」问题的通用框架。

1. 先问"为什么"：synchronized 解决不了什么？

synchronized 是 JVM 内置锁，使用简单，但有几个根本局限：

  • 不可中断：线程阻塞在 monitorenter 后无法响应 Thread.interrupt()
  • 不可超时：无法"等 300ms 没拿到就放弃"
  • 只支持独占：无法实现"多个读线程并发，写线程独占"的读写锁语义
  • 不可查询：无法知道当前有多少线程在等待

Doug Lea 在 JDK 1.5 引入 java.util.concurrent 时面临一个设计选择： 为每种同步工具单独造轮子，还是抽象出一个统一骨架？ 他选择了后者，这个骨架就是 AbstractQueuedSynchronizer（AQS）。

设计哲学：把"谁来排队等待"和"如何挂起/唤醒线程"这两件通用的苦差事放进 AQS， 让子类只需回答一个问题：当前状态允许我获取资源吗？

2. AQS 的核心数据结构

AQS 只有两个关键字段（加上一条双向链表），却能支撑几乎所有同步需求：

AbstractQueuedSynchronizer ├── volatile int state <-- 同步状态，含义由子类定义 ├── Node head <-- CLH 队列头（哨兵节点） ├── Node tail <-- CLH 队列尾 └── Node（双向链表节点） ├── int waitStatus <-- CANCELLED/SIGNAL/CONDITION/PROPAGATE/0 ├── Node prev / next ├── Thread thread <-- 被阻塞的线程 └── Node nextWaiter <-- Condition 队列链接

state 的含义由子类决定

同步工具state 的语义
ReentrantLock0 = 未锁；>0 = 重入次数
Semaphore剩余许可数
CountDownLatch剩余计数，归零后全部释放
ReentrantReadWriteLock高 16 位 = 读持有数；低 16 位 = 写重入数

CLH 变体等待队列

AQS 使用的等待队列是 CLH（Craig-Landin-Hagersten）锁队列的变体。 原始 CLH 用于自旋锁（每个节点轮询前驱的状态），AQS 将其改造为阻塞队列： 节点在获取失败后会调用 LockSupport.park() 真正挂起，不再空转消耗 CPU。

为什么用双向链表而非单向？取消（CANCELLED）时需要从队列中安全摘除节点， 必须访问前驱，所以在 CLH 原版单向链表的基础上补了 prev 指针。

3. 独占模式：acquire / release 核心流程

以 ReentrantLock.lock() 为例，完整路径如下：

AQS 独占获取骨架（伪代码 + 注释）
// ① 模板方法：子类无需覆盖，AQS 负责排队逻辑
public final void acquire(int arg) {
    if (!tryAcquire(arg)              // ② 先让子类尝试一次（CAS 改 state）
        && acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
            // ③ 失败 → 入队 → 自旋/park，直到被前驱唤醒且 tryAcquire 成功
        selfInterrupt();
}

// acquireQueued 核心自旋（简化）
final boolean acquireQueued(Node node, int arg) {
    for (;;) {
        Node p = node.predecessor();
        if (p == head && tryAcquire(arg)) {
            // 只有前驱是 head 时才有资格竞争，避免"惊群"
            setHead(node);
            p.next = null;
            return false;
        }
        if (shouldParkAfterFailedAcquire(p, node))
            parkAndCheckInterrupt(); // LockSupport.park(this)
    }
}

release() 的路径更简单：CAS 修改 state，成功后唤醒队列中第一个有效后继节点（LockSupport.unpark(s.thread)），让它重新参与竞争。

关键设计：为什么只有"前驱是 head"才能 tryAcquire？
这保证了 FIFO 顺序（非公平锁在入队前允许插队一次，但一旦入队就严格排队）， 同时避免了所有等待线程同时被唤醒后的"惊群效应"。

4. 子类只需实现 5 个钩子方法

AQS 用模板方法模式把"通用骨架"和"业务判断"分离，子类只需覆盖下面（最多）5 个方法：

方法用途独占/共享
tryAcquire(int)尝试独占获取，返回 true 表示成功独占
tryRelease(int)尝试独占释放，返回 true 表示完全释放独占
tryAcquireShared(int)尝试共享获取，返回值 ≥ 0 表示成功共享
tryReleaseShared(int)尝试共享释放，返回 true 表示需要唤醒后继共享
isHeldExclusively()当前线程是否独占持有（Condition 需要）独占

实战：用 AQS 实现一个不可重入互斥锁

Mutex.java — 最小可运行示例
class Mutex implements Lock {

    // 内部 Sync 继承 AQS，state: 0=未锁, 1=已锁
    private static final class Sync extends AbstractQueuedSynchronizer {

        @Override
        protected boolean tryAcquire(int acquires) {
            // CAS 0→1，成功则记录持有线程（支持 isHeldExclusively）
            if (compareAndSetState(0, 1)) {
                setExclusiveOwnerThread(Thread.currentThread());
                return true;
            }
            return false;
        }

        @Override
        protected boolean tryRelease(int releases) {
            // 不校验持有者：生产代码应加 assert isHeldExclusively()
            setExclusiveOwnerThread(null);
            setState(0);   // 普通写即可；unlock 后无竞争读
            return true;
        }

        @Override
        protected boolean isHeldExclusively() {
            return getState() == 1;
        }

        Condition newCondition() {
            return new ConditionObject(); // AQS 内部类，免费获得
        }
    }

    private final Sync sync = new Sync();

    @Override public void lock() { sync.acquire(1); }
    @Override public void unlock() { sync.release(1); }
    @Override public boolean tryLock() { return sync.tryAcquire(1); }
    @Override public Condition newCondition() { return sync.newCondition(); }
    // lockInterruptibly / tryLock(timeout) 省略，委托给 sync.acquireInterruptibly
}

仅 40 行代码，就获得了：可中断获取、带超时获取、Condition 支持、队列监控 API——这正是 AQS 骨架的价值所在。

5. 共享模式：Semaphore / CountDownLatch 的实现原理

共享模式与独占模式的核心区别：tryAcquireShared 成功后， 如果队列中下一个等待节点也是共享模式，AQS 会传播唤醒（doReleaseShared）， 而不是只唤醒一个——这正是 CountDownLatch.await() 归零后所有等待线程同时放行的原理。

Semaphore 的 tryAcquireShared（非公平版，JDK 源码节选）
@Override
protected int tryAcquireShared(int acquires) {
    for (;;) {
        int available = getState();
        int remaining = available - acquires;
        // remaining < 0 → 返回负数，AQS 知道需要入队
        // remaining ≥ 0 且 CAS 成功 → 返回剩余许可数（≥ 0），AQS 知道可能要传播
        if (remaining < 0
            || compareAndSetState(available, remaining))
            return remaining;
    }
}

6. ConditionObject：AQS 内建的等待/通知机制

每个 ConditionObject 维护一个独立的条件队列（单向链表）， 与 AQS 的同步队列（CLH）分开存储，两者通过节点转移来协作：

  • await()：将当前线程节点从同步队列移到条件队列，然后释放锁并 park
  • signal()：将条件队列头节点迁移回同步队列尾部，等待重新竞争锁
  • signalAll()：将条件队列中所有节点批量迁移回同步队列
并发陷阱：虚假唤醒（Spurious Wakeup）
即使没有调用 signal()，await() 也可能在某些平台上意外返回（POSIX pthread 规范允许）。 所以 必须在循环中检查条件，而不是 if：
反例 vs 正例：虚假唤醒
// ❌ 反例：用 if 检查，虚假唤醒后直接往下走
lock.lock();
try {
    if (queue.isEmpty())
        notEmpty.await();   // 虚假唤醒后跳过检查
    process(queue.poll());   // 可能 NPE！
} finally { lock.unlock(); }

// ✅ 正例：while 循环，确保条件真正满足
lock.lock();
try {
    while (queue.isEmpty())
        notEmpty.await();   // 醒来后重新检查
    process(queue.poll());
} finally { lock.unlock(); }

7. 公平锁 vs 非公平锁：一行代码的区别

ReentrantLock 的公平/非公平仅体现在 tryAcquire 中的一个判断：

ReentrantLock.FairSync vs NonfairSync（简化）
// 公平锁：只有队列为空或当前线程是队头，才 CAS
protected boolean tryAcquire(int acquires) {
    if (!hasQueuedPredecessors()          // ← 公平锁多了这一行
        && compareAndSetState(0, acquires)) {
        setExclusiveOwnerThread(Thread.currentThread());
        return true;
    }
    return false;
}

// 非公平锁：直接抢，入队前插队一次
protected boolean tryAcquire(int acquires) {
    if (compareAndSetState(0, acquires)) { // 直接 CAS，不检查队列
        setExclusiveOwnerThread(Thread.currentThread());
        return true;
    }
    return false;
}
公平锁非公平锁（默认）
吞吐量较低（每次都入队）高（减少 park/unpark 切换）
饥饿风险无（严格 FIFO）有（队列中线程可能被反复插队）
适用场景响应时间要求严格吞吐优先的通用场景

8. 使用 AQS 的常见陷阱

陷阱 1：tryRelease 没有完全释放就返回 true

对于可重入锁，tryRelease 应当在 state 降为 0 后才返回 true， 否则 AQS 会提前唤醒等待线程，破坏互斥语义。

ReentrantLock.tryRelease 正确写法
@Override
protected boolean tryRelease(int releases) {
    int c = getState() - releases;
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    boolean free = c == 0;  // state 归零才算真正释放
    if (free) setExclusiveOwnerThread(null);
    setState(c);
    return free;  // 只有 true 时 AQS 才唤醒后继
}

陷阱 2：在 tryAcquire 中执行阻塞操作

tryAcquire / tryRelease 必须是非阻塞、无锁的。 如果在其中调用 Thread.sleep()、I/O 或另一个锁，会导致持锁线程卡死， 所有等待者永久阻塞——这是 AQS 框架层面无法自动检测的死锁。

陷阱 3：忘记在 finally 中 unlock

反例 vs 正例：unlock 必须在 finally
// ❌ 反例：业务代码抛异常后锁永远不释放
lock.lock();
doSomethingRisky();   // 可能抛 RuntimeException
lock.unlock();          // 永远不会执行！

// ✅ 正例
lock.lock();
try {
    doSomethingRisky();
} finally {
    lock.unlock();   // 无论如何都释放
}

9. AQS 家族全景

AbstractQueuedSynchronizer ├── ReentrantLock.Sync 独占，可重入，公平/非公平切换 ├── ReentrantReadWriteLock.Sync 独占写 + 共享读，state 位分割 ├── Semaphore.Sync 共享，state = 许可数 ├── CountDownLatch.Sync 共享，state = 计数，归零广播 ├── ThreadPoolExecutor.Worker 独占，防止运行中工作线程被中断 └── AbstractQueuedLongSynchronizer 64 位 state 变体（不常用）

10. 面试高频考点

  • AQS 的 state 是什么类型？为什么用 volatile？——volatile int，保证可见性；写 state 用 CAS（compareAndSetState），读 state 直接读，符合 happens-before。
  • CLH 队列为什么是双向链表？——节点取消时需要向前查找有效前驱；单向链表无法 O(1) 找到前驱。
  • 非公平锁为什么比公平锁吞吐高？——新来线程直接 CAS 抢锁，避免了"park 当前线程 → unpark 队头线程 → 线程上下文切换"这一开销。
  • AQS 如何实现可中断获取？——acquireInterruptibly 在 park 返回后检查中断标志，若已中断则抛 InterruptedException 而非继续循环。
  • Condition.await() 与 Object.wait() 的核心区别？——Condition 与 Lock 绑定，支持一个锁对应多个条件队列；Object.wait 只能有一个 Monitor 等待集合。
一句话总结：AQS 用一个 volatile int state + 一条 CLH 变体双向队列， 将"排队等待"和"挂起/唤醒"的通用逻辑封装为骨架，让子类只需用 CAS 回答"能否获取资源"， 从而以极小的代码量支撑起整个 java.util.concurrent 锁体系。
