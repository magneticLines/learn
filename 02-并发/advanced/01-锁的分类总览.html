锁的分类总览

「锁」不是一种东西,而是一组从不同维度切分的设计选择。先建立这张地图,再去看每把具体的锁就有了坐标。

📖 来源:《Java 并发编程实战》 第 13 章「显式锁」 · 综合整理

一、为什么要「分类」而不是「背名字」

面试里一连串「公平锁、可重入锁、自旋锁、读写锁……」听着像并列的几十种锁,其实它们是同一把锁在不同维度上的属性。比如 ReentrantLock 同时是「悲观锁 + 可重入锁 + 独占锁 + 可公平可非公平」。所以正确的理解方式是抓维度,而不是记名字。下面四个维度回答四个不同的问题。

二、维度①:遇到冲突怎么办 —— 悲观 vs 乐观

这是从冲突处理策略切分的,也是最根本的一刀。

悲观锁乐观锁
假设一定会冲突,先上锁再操作大概率不冲突,先操作提交时再校验
代表synchronized、ReentrantLockCAS、atomic 原子类、数据库版本号
适合写多、冲突频繁读多写少、冲突少
代价加解锁开销、可能阻塞冲突多时大量重试,空耗 CPU
Java · 乐观锁的底层:CAS
// AtomicInteger.incrementAndGet() 内部就是「比较并交换 + 失败重试」
do {
    int old = value;
    int next = old + 1;
} while (!compareAndSet(old, next)); // 没人改过才写成功,否则重来

三、维度②:谁先拿到 —— 公平 vs 非公平

从获取顺序切分。公平锁严格 FIFO(先申请先得);非公平锁允许「插队」。

  • synchronized 只能非公平。
  • ReentrantLock 可选:new ReentrantLock(true) 公平,默认非公平。
为什么默认非公平? 刚释放锁的线程往往还在 CPU 上「热着」,让它直接再抢成功,可以省掉「唤醒队列里下一个线程」的上下文切换开销,吞吐更高。代价是排队线程可能被反复插队而饥饿。

四、维度③:能否重复进入 —— 可重入 vs 不可重入

从同一线程能否重复获取自己已持有的锁切分。

Java · 可重入避免自锁死
synchronized void a() { b(); } // 已持有 this 锁
synchronized void b() { } // 同一线程再次进入 this 锁——可重入才不会卡死

synchronized 和 ReentrantLock(名字里 Reentrant 就是「可重入」)都可重入。不可重入锁现实中极少见,需要自己实现。

五、维度④:能否共享 —— 独占 vs 共享

从能否被多个线程同时持有切分。独占锁同一时刻只一个线程持有;共享锁可多线程同时持有。最典型的应用是读写锁:

锁类型规则
读锁(共享)共享锁多个读线程可同时持有
写锁(排他)独占锁写时不允许任何其他读/写
ReentrantReadWriteLock 的「读读不互斥、读写/写写互斥」,非常适合读多写少的缓存类场景,比一把独占锁并发度高得多。

六、专门说说 synchronized 的「锁升级」

这其实是另一条维度——JVM 内部的实现优化,不是用户能选的。按竞争激烈程度单向升级:

无锁 ⟶ 偏向锁 ⟶ 轻量级锁(自旋) ⟶ 重量级锁(OS 互斥量)

其中自旋锁(失败不挂起、忙循环重试)正是轻量级锁的手段——用 CPU 空转换取省掉上下文切换。细节见 synchronized 一节。

七、一张总表收口

维度一端另一端判断标准
处理策略悲观(synchronized)乐观(CAS)是否假设冲突会发生
获取顺序公平非公平(默认)是否先来先得
可重入性可重入不可重入同线程能否重复获取
共享模式独占共享(读锁)能否多线程同时持有
JVM 实现偏向 → 轻量 → 重量(锁升级)针对 synchronized 的优化
常见误区:把这些当成「几十种并列的锁」去背。记住——任何一把具体的锁,都是这几个维度上各选一个值的组合。

小结

  • 锁按四个维度切分:处理策略、获取顺序、可重入性、共享模式。
  • 乐观锁靠 CAS 实现,适合读多写少;悲观锁靠阻塞,适合写多冲突频繁。
  • 非公平是默认,因为能省上下文切换;代价是可能饥饿。
  • 读写锁 = 读共享 + 写独占,是「读多写少」的利器。
  • 锁升级是 JVM 对 synchronized 的内部优化,不是用户选项。

与其他知识点的联系

  • JMM / happens-before:不管是悲观锁还是乐观锁,「锁」最终要解决的都是可见性和有序性问题。加锁/解锁本身会建立 happens-before 关系——释放锁时把工作内存的修改刷回主内存,获取锁时清空本地缓存重新读,这正是锁「既能互斥又能保证可见性」的底层原因,而不仅仅是「挡住别的线程」。
  • CAS 与原子类:乐观锁这一维度的具体实现就是 CAS,详见「CAS 与原子类」一节。理解了「悲观 vs 乐观」这个维度,再看 AtomicInteger 的 compareAndSet 循环就不再是孤立的 API,而是乐观锁思路的一个具体落地。
  • AQS 体系:公平/非公平这个维度在实现上就是 AQS 内部 CLH 队列「是否允许插队」的一个开关——非公平锁会在入队前先抢一次,公平锁则严格检查队列里是否已有等待者。ReentrantLock、Semaphore、CountDownLatch 这些看似不同的工具,底层共享的正是 AQS 这套「state + 队列」的骨架。
  • ReentrantReadWriteLock:本节讲的「独占 vs 共享」维度,最典型的落地就是读写锁——读锁是共享锁,写锁是独占锁,两者共用一个 state(高 16 位记读、低 16 位记写)。这也是为什么读写锁只有一把锁对象,却能表现出两种完全不同的加锁行为。
  • 分布式锁对比:本节的四个维度同样可以套到 Redis/ZooKeeper 分布式锁上去理解——比如 Redisson 的可重入锁用 hash 结构记录「线程标识 + 重入次数」,本质是把 JVM 内置锁「可重入」这个维度搬到了跨进程场景;而公平性在分布式锁里往往需要额外维护一个队列结构才能模拟,不像 AQS 里那样天然存在。
  • 面试高频考点:面试官常问「说说你知道的锁」,如果按名字罗列会显得零散;按本节的四个维度组织答案(策略/顺序/重入性/共享性),再举一两个具体锁作为交叉点(如 ReentrantLock = 悲观+可选公平+可重入+独占),会显得更有条理,也更容易追问下去而不露怯。
