CSS · 盒模型

box-sizing

一个属性，决定了你写的 width 到底算不算 padding 和 border。它是现代 CSS 工程的第一行配置。

一、它解决什么问题

上一节我们发现一个反直觉的事：标准盒模型下，width: 200px + padding: 20px 的盒子实际占 240px。你想要的是「这个盒子就是 200px 宽，padding 往里挤」，浏览器给你的却是「200px 是内容区，padding 往外撑」。

box-sizing 就是用来切换这两种解释方式的开关。它的本质是回答一个问题：你写的 width，量的是哪一层的宽度？

二、两个取值

取值width 量到哪实际宽度公式
content-box
（默认）		只量 contentwidth + padding + border
border-box量到 border 外缘width（已含 padding + border）

在 border-box 下，padding 和 border 是从 width 里「抠出来」的，而不是往外加的。content 区会自动缩小以容纳它们：

CSS
.box {
  box-sizing: border-box;
  width: 200px;
  padding: 20px;
  border: 5px solid;
}
/* 盒子实际宽度就是 200px（一分不多） */
/* content 区被压缩成 200 - 40 - 10 = 150px */

四、为什么 border-box 更好用

三个理由，越往后越关键：

  1. 所见即所得：写 width: 300px 盒子就是 300px，不用拿计算器减来减去。
  2. 响应式布局更稳：百分比宽度（如 width: 33.33%）配像素 padding 不会溢出换行——这是栅格系统的命脉。
  3. 改动局部不破坏整体：给一个卡片加 padding，它不会把布局撑大、挤掉旁边的兄弟元素。维护时心智负担大幅下降。
历史插曲：IE6 的「怪异模式（quirks mode）」其实就是按 border-box 算的。当年大家骂 IE 不遵守标准，结果多年后 W3C 发现 border-box 反而更符合人的直觉，于是把它做成了 box-sizing 的可选值。某种意义上，IE 的「bug」赢了。

五、全局最佳实践的来历

几乎所有现代 CSS 项目、所有 UI 框架（Bootstrap、Tailwind 等）开头都有这么一段：

CSS · 最佳实践
/* 推荐写法：可继承版本，便于第三方组件局部覆盖 */
html {
  box-sizing: border-box;
}
*, *::before, *::after {
  box-sizing: inherit;
}

注意这里用 inherit 而不是直接 * { box-sizing: border-box }。原因是：如果某个第三方组件（比如某个 jQuery 插件）依赖 content-box，它只需在自己根节点改一次，整棵子树就跟着变——比硬编码到 * 上更有弹性。当然，简单项目里直接 * { box-sizing: border-box }（本页模板就是这么写的）也完全够用。

六、易混淆点

box-sizing 不影响 margin。无论哪种模式，margin 永远在盒子外侧、永远不算进 width。border-box 只「吸收」padding 和 border，不碰 margin。想做「margin 也算进去」的整体盒子，那是另一个概念（要用 flex/grid 的 gap 或 calc）。

小结

  • box-sizing 决定 width 量到哪层：content-box 只量内容，border-box 量到边框外缘。
  • border-box 让 width 所见即所得，响应式不溢出，维护更省心。
  • 项目开头一律 border-box（用 inherit 版更优雅），是行业默认。
  • 它不影响 margin——margin 始终在盒子之外。

与其他知识点的联系

  • 盒模型：box-sizing 不是新增了盒子的层级，而是改变了"width 这个数字量到哪一层"的解释方式——没有上一节"四层同心矩形"的模型，就无法理解 border-box 到底把 padding/border 从哪里"抠"出来。
  • overflow 与百分比布局：栅格系统里 width: 33.33% 配合像素 padding 之所以能不溢出换行，正是靠 border-box 让百分比宽度真正等于占位宽度；如果用 content-box，多列布局几乎必然因为 padding/border 叠加而溢出，这是响应式布局能落地的前提条件之一。
  • JS 获取/设置尺寸：当元素是 border-box 时，设置 element.style.width = '200px' 与用 offsetWidth 读到的值是一致的；而 content-box 下二者含义不同，写响应式计算逻辑（如根据容器宽度动态设置子元素宽度）时必须先确认盒模型类型，否则数值会出现不易察觉的偏差。
  • React/Vue 组件样式：几乎所有 UI 框架和 CSS 重置方案都在全局样式最前面写 * { box-sizing: border-box }，这意味着组件库作者可以放心地用 width: 100% 加 padding 而不担心溢出——组件化开发依赖这条全局默认约定，一旦某个第三方组件没有遵守，就会出现"个别组件布局跑偏"的诡异 bug。
  • 性能优化：box-sizing 本身是一次性声明，不参与运行时高频计算，但选对它可以避免开发阶段大量"调宽度、发现溢出、再改 calc()"的反复调试——从工程效率角度看，统一 border-box 减少了后续因盒模型误解产生的返工成本。
