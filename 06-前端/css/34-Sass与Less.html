Sass / Less 预处理器（进阶）

原生 CSS 很长一段时间没有变量、没有函数、没有模块。预处理器用一门「编译成 CSS」的超集语言补上这些缺口，把样式表当成真正的程序来写。

一、问题：原生 CSS 的历史局限

在 CSS 自定义属性（变量）2017 年才普及之前，原生 CSS 没有任何「编程」能力：颜色、间距等值只能到处硬编码；想复用一组样式只能复制粘贴；大型项目想拆文件，@import 还会额外发起 HTTP 请求。Sass（2006）和 Less（2009）就是为补这些缺口而生：它们是 CSS 的超集，写起来像 CSS，但经构建工具编译成普通 CSS 再交给浏览器。

关键认知：浏览器不认识 Sass / Less。它们是「编译时」语言，所有变量、嵌套、循环都在打包阶段被算成静态 CSS。运行时浏览器看到的，仍是平平无奇的 CSS。

二、核心能力一览（以 Sass 的 .scss 语法为例）

1. 变量（Variables）

SCSS
$primary: #60a5fa;
$gap: 16px;
.btn { background: $primary; padding: $gap; }

2. 嵌套（Nesting）——以及它的陷阱

SCSS：嵌套 + 父选择器 &
.card {
  padding: $gap;
  .title { font-weight: 700; }
  &:hover { transform: translateY(-2px); }
}
嵌套陷阱：嵌套层级会被原样翻译成后代选择器。嵌 4 层就会编译出 .a .b .c .d，特异性飙升、难以覆盖——这正是上一组 BEM 章节里批判的「深层嵌套」问题。嵌套深度建议不超过 3 层，能用 & 拼 BEM 类名（如 &__title）就别真的层层缩进。

3. Mixin 与 @include（可复用样式块）

Mixin 类似「函数」，把一组声明打包，可带参数，在任意地方 @include 复用。

SCSS
@mixin flex-center($dir: row) {
  display: flex;
  flex-direction: $dir;
  align-items: center;
  justify-content: center;
}
.modal { @include flex-center(column); }

4. 函数与运算

SCSS
.box {
  width: $gap * 4;        // 编译为 64px
  background: darken($primary, 10%); // 内置颜色函数
}

5. Partials 与模块化：@import → @use

把样式拆成以下划线开头的partial 文件（如 _buttons.scss），它们不会单独编译，而是被主文件引入合并成一个 CSS。早期用 @import，但它会把所有变量/mixin 倒进全局命名空间，容易冲突。现代 Sass 推荐用 @use：它带命名空间、只加载一次，更接近真正的模块系统。

SCSS：@use 模块化
// main.scss
@use 'colors';   // 引入 _colors.scss
.btn { color: colors.$primary; } // 带命名空间访问
@import 在 Sass 中已被官方标记为弃用，新项目一律用 @use / @forward。注意别和 CSS 原生的 @import（运行时引外部样式表）搞混——它们是两回事。

三、Sass vs Less

维度Sass (SCSS)Less
变量符号$primary@primary
实现语言原 Ruby，现 Dart SassJavaScript
编程能力更强（循环、条件、模块系统 @use）较弱
语法SCSS（兼容 CSS）/ 缩进式 Sass近似 CSS
生态主流、社区最大较小，部分老项目（如旧版 Bootstrap/Ant Design）在用

结论：新项目若要用预处理器，优先 Sass（SCSS 语法），生态和能力都更强。Less 主要存在于历史项目里。

四、与原生 CSS 变量的关系：编译时 vs 运行时

这是面试高频考点。Sass 变量（$x）和 CSS 自定义属性（--x）看着像，本质完全不同：

Sass 变量 $xCSS 变量 --x
生效时机编译时，打包后就被替换成固定值运行时，浏览器中实时解析
能否被 JS 改不能，编译后已消失能，element.style.setProperty
动态主题切换做不到（值已写死）能（改 --x 全页面级联更新）
媒体查询内变值不能能
实战常用「组合拳」：用 Sass 管理嵌套、mixin、文件组织等静态结构，同时用 CSS 变量承载需要运行时切换的值（如暗色主题）。两者不冲突，各司其职。

五、现在还要不要学？

原生 CSS 已经补上了变量（--x），并且 CSS 原生嵌套（Nesting）也已在主流浏览器落地。预处理器最核心的两个卖点正被原生蚕食。但 Sass 的 mixin、函数、循环、@use 模块系统仍是原生 CSS 没有的。所以：

  • 新项目：很多团队转向「原生 CSS 变量 + 原生嵌套 + PostCSS」，或干脆用 Tailwind / CSS Modules，不再依赖预处理器。
  • 但仍值得了解：海量存量项目、组件库（Bootstrap 等）用 Sass 写成，看懂、改动它是基本功；mixin/循环在某些场景仍最省事。
一句话定位：预处理器是「原生 CSS 能力不足年代的补丁」。随着原生进化，它从「必学」降级为「按需了解」，但理解它能帮你看懂工程化的来龙去脉。
