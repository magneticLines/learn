优先级与层叠规则

当多条 CSS 规则同时作用于同一元素时，浏览器用一套严格的层叠算法裁定最终胜者——搞懂它，才能真正掌控样式。

1. 什么是层叠（Cascade）

CSS 全称 Cascading Style Sheets，"Cascading"（层叠）就是其核心机制。现实场景中，同一个元素的同一个属性可能同时被多条规则声明，例如：

  • 浏览器内置的默认样式（user-agent stylesheet）
  • 用户在操作系统或无障碍工具里设置的自定义样式
  • 开发者自己写的样式文件，可能来自多个 CSS 文件、内联 <style>、第三方库

层叠算法的本质是一个多维排序：把所有竞争的声明按规则排出优先队列，排名最高的那条生效。这个设计解决了"多来源样式冲突"的根本问题，让样式可组合、可覆盖，同时对用户保留一定控制权（accessibility 场景中 !important 用户样式优先级最高，正是对此的体现）。

与其他语言的类比：层叠就像 Java 里的方法重载解析——编译器（浏览器）根据一套固定规则从候选集中选出最匹配的那一个，而不是随机选取。

层叠判定的完整顺序

浏览器按以下维度从高到低依次筛选，前一维度已分出胜负则不继续往下比：

排序维度说明
① 来源 & 重要性（Origin & Importance）决定规则来自哪一层"权威"，!important 会反转来源顺序
② 层叠层（@layer）CSS Cascade Level 5 新增，显式控制层序
③ 特异性（Specificity）选择器有多"精确"，用三元组 (a,b,c) 量化
④ 出现顺序（Source Order）特异性相同时，文档中靠后的声明覆盖靠前的

来源与重要性的完整排序（从低到高）

优先级来源
1（最低）浏览器默认样式（user-agent normal）
2用户自定义样式（user normal）
3开发者样式（author normal）——即我们写的 CSS
4开发者 !important（author important）
5用户 !important（user important）
6（最高）浏览器 !important（user-agent important，极少见）
为什么 user !important 比 author !important 还高？
这是无障碍设计的考量：视力障碍用户可能通过操作系统强制放大字体，如果开发者的 !important 能压过用户设置，将严重影响可访问性。规范刻意把用户的 !important 放在最顶端，保护用户的控制权。

2. 特异性（Specificity）计算规则

特异性是层叠算法中最常打交道的维度。它用一个三元组 (a, b, c) 来量化一个选择器的"精确程度"：

位计数对象示例
aID 选择器数量#nav → a=1
b类选择器、属性选择器、伪类数量.btn:hover → b=2
c元素选择器、伪元素数量p::before → c=2

比较时从左到右逐位比大小，a 先比，a 相同再比 b，b 相同再比 c。不存在"进位"——100 个类选择器 (0,100,0) 也无法超过 1 个 ID 选择器 (1,0,0)。

内联样式的特异性

写在 HTML 元素 style 属性里的声明，特异性比任何选择器都高，通常表示为第四位 (1,0,0,0) 或单独说明"内联样式高于所有选择器"。唯一能覆盖内联样式的，是 !important（或更高层级的来源）。

不计入特异性的情况

  • *（通配符）和组合符（>、+、~、空格）：特异性为 (0,0,0)
  • :not() 本身不计分，但其括号内的参数正常计入
  • :where() 始终为 (0,0,0)，专为"不想影响特异性"的场景设计
  • :is() 和 :has()：取括号内参数列表中特异性最高的那个
特异性计算示例
/* :not() 示例：(0,1,0) — :not 本身 0，.active 贡献 b=1 */
:not(.active) { color: red; }

/* :where() 示例：(0,1,0) — :where 贡献 0，.btn 贡献 b=1 */
:where(.nav) .btn { color: blue; }

/* :is() 示例：(1,0,0) — :is 取内部最高特异性 #main */
:is(#main, .sidebar) p { color: green; }

3. 特异性计算练习表

从左到右难度递增，建议先自己算再对答案：

选择器特异性 (a,b,c)说明
p(0,0,1)1 个元素选择器
.btn(0,1,0)1 个类选择器
#nav(1,0,0)1 个 ID 选择器
ul li a(0,0,3)3 个元素选择器
a:hover(0,1,1)1 元素 + 1 伪类
input[type="text"](0,1,1)1 元素 + 1 属性选择器
#nav .item a(1,1,1)1 ID + 1 类 + 1 元素
:where(.x) .y(0,1,0):where 贡献 0，.y 贡献 b=1
h2::before(0,0,2)1 元素 + 1 伪元素
#app .list > li:first-child(1,2,1)1 ID + 1 类 + 1 伪类 + 1 元素；> 不计
常见陷阱：特异性不进位。无论写多少个类选择器，(0,255,0) 也无法超过 (1,0,0)。这与十进制进位完全不同——它是逐位独立比较的有序元组。

4. !important——威力与危害

它做了什么

!important 把一条声明从"作者普通层"提升到"作者重要层"，在来源与重要性维度直接跨越了所有普通声明，包括内联样式。

反例 → 问题 → 正例

反例：用 !important 修复样式冲突
反例 — 用 !important 打补丁
/* 某天发现按钮颜色不对，直接加 !important 修复 */
.btn { color: blue !important; }

/* 一周后，另一个组件需要红色按钮，于是 */
.modal .btn { color: red !important; }

/* 再一周，紧急修复又加一层 */
#checkout .modal .btn { color: green !important; }

问题：一旦引入 !important，唯一的覆盖手段是更高来源的 !important 或更高特异性的 !important。代码库很快变成"important 军备竞赛"，逻辑难以追踪，维护成本指数级上升。

正例：根治冲突的根本是理清选择器层次
正例 — 用更精确的选择器或 @layer 解决
/* 基础样式 */
.btn { color: blue; }

/* modal 内需要不同颜色，提升特异性即可，不需要 !important */
.modal .btn { color: red; }

/* 或使用 BEM，用语义而非嵌套提升特异性 */
.btn--danger { color: red; }

极少数合理场景

  • 覆盖第三方库的内联样式：某些老旧插件直接写 style="display:none"，你无法修改其源码，此时 display: block !important 是唯一出路
  • 原子/工具类 CSS（如 Tailwind）：工具类本身特意使用 !important（或高特异性），确保 utility 永远生效，这是框架层面的设计决策，不是随意滥用
  • 无障碍强制覆盖：用户样式表需要强制应用高对比度，!important 是规范授权的手段

5. 同特异性时——后写覆盖（Source Order）

当两条规则的来源相同、层叠层相同、特异性相同时，CSS 规范规定文档中出现在后面的声明覆盖前面的。这条规则简单但非常实用：它是你在同一个 .css 文件里"重置"样式最自然的方式。

同特异性后写覆盖
/* 两条规则特异性完全相同：(0,1,0) */
.later-demo { color: red; }   /* 先写，被覆盖 */
.later-demo { color: green; } /* 后写，最终生效 */

实际渲染效果：

这段文字应该是绿色（后写规则生效）
与多文件加载顺序的关系：如果样式来自多个 <link> 标签，后引入的文件中的规则视为"后写"。这也是为什么重置样式（normalize.css）要放在自定义样式之前引入——让你的样式可以安全覆盖重置样式。

6. @layer 层叠层——终结特异性军备竞赛

CSS Cascade Level 5 引入的 @layer 允许开发者显式声明层序，层序优先级高于特异性——即使低层里的选择器特异性更高，高层里的低特异性规则也能胜出。

为什么需要它

在大型项目中，你往往引入第三方 UI 库（特异性较高），同时又有自己的业务样式。传统做法只能用更高特异性甚至 !important 来覆盖。@layer 让你在架构层面声明"业务样式一定在库样式之上"，彻底绕开特异性比拼。

@layer 基本用法
/* 第一步：声明层序（越靠后优先级越高）*/
@layer reset, base, components, utilities;

/* 第二步：往各层放规则 */
@layer base {
  button { background: gray; }
}

/* components 层优先级高于 base 层，
   即使 button 特异性 (0,0,1) 低于 .primary-btn (0,1,0)，
   components 层的 button 规则也会赢过 base 层的 .primary-btn */
@layer components {
  button { background: #3b82f6; }
}

实际渲染演示（s3theme 层声明顺序晚于 s3base 层，即使元素只有一个类）：

这段文字颜色来自 s3theme 层（蓝色），而非 s3base 层（灰色）
层外（unlayered）样式的优先级：没有放入任何 @layer 的普通样式，优先级高于所有层内样式。这确保了旧代码迁移的向后兼容性。
@layer 与第三方库的实践模式
/* 把第三方库放入低优先级层，自己的样式不放入任何层（最高优先级）*/
@layer third-party;

@layer third-party {
  @import url("some-ui-library.css");
}

/* 自己的样式，无需 !important 就能轻松覆盖库样式 */
.my-btn { background: red; }

7. 综合 Demo：同一元素多规则竞争

下面的元素同时被 ID 规则、类规则（两条，后者本应覆盖前者）共同声明颜色，最终由层叠算法裁定胜者：

三条规则竞争同一属性
/* 规则 A：class，特异性 (0,1,0) */
.s3-final-cls { color: #f87171; }  /* 红色 */

/* 规则 B：同特异性 (0,1,0)，后写，本应覆盖 A */
.s3-final-cls { color: #fbbf24; }  /* 黄色 */

/* 规则 C：ID，特异性 (1,0,0)，远高于 AB */
#s3-final    { color: #a78bfa; }  /* 紫色 ← 胜者 */

最终颜色：紫色（ID 规则胜出，特异性 1,0,0 > 0,1,0）
裁定过程：来源相同（都是作者样式）→ 无 @layer → 比特异性：ID (1,0,0) > class (0,1,0) → C 胜出，颜色为紫色

可以在浏览器开发者工具 → Elements → Computed/Styles 中看到被覆盖的规则上有删除线，这正是层叠算法实时运行的可视化结果。

8. 陷阱与面试高频考点

陷阱 1：特异性不进位
(0,10,0) 无法超过 (1,0,0)。哪怕你堆叠 999 个类选择器，也打不过 1 个 ID 选择器。面试常考：.a.b.c.d.e.f.g.h.i.j vs #x，答案是 #x 赢。
陷阱 2：内联样式 vs !important
内联样式的特异性是 (1,0,0,0)，高于任何选择器，但低于 !important。即：!important 可以覆盖内联样式，这是面试高频考点。
陷阱 3：:not() 的特异性计算
:not(p) 的特异性等同于 p（(0,0,1)），不是 0。:not(#id) 的特异性等同于 #id（(1,0,0)）。:not 本身不计分，但内部参数照常计分。
陷阱 4：link 标签顺序即层叠顺序
多个 <link rel="stylesheet"> 引入的文件，后引入的优先级更高（同特异性情况下）。曾有人因为打包工具改变了 CSS 文件的引入顺序，导致生产环境样式突然"失效"。
陷阱 5：!important 反模式的滚雪球效应
第一个 !important 往往是整个代码库样式混乱的起点。一旦引入，下一个想覆盖它的人只能再加一个 !important，形成无法终止的升级链。Code Review 中见到 !important 要优先质疑其必要性。
面试考点汇总
  • 写出 div p.foo > a:hover 的特异性（答：(0,2,3)）
  • !important 能覆盖内联样式吗？（答：能）
  • 100 个类能打败 1 个 ID 吗？（答：不能）
  • :where() 和 :is() 的特异性区别？（:where 始终为 0，:is 取参数最大值）
  • @layer 是什么，解决什么问题？
  • 用户的 !important 和开发者的 !important 谁更高？（答：用户更高，无障碍设计）

浏览器实现视角

浏览器在构建 CSSOM（CSS Object Model）时，会为每条规则计算特异性并存储。样式计算阶段（Style Recalculation）为每个 DOM 节点枚举匹配的规则，按层叠算法排序，取最高优先级的声明作为 computed value。开发者工具的 Computed 面板展示的正是这一计算结果，Styles 面板的删除线则标记了被覆盖的声明——调试样式问题时这两个面板是最直接的工具。

9. 与其他知识点的联系

  • 选择器：特异性完全由选择器的构成决定——ID、类、伪类、元素各自贡献 (a,b,c) 三元组的一位，理解选择器分类是计算特异性的前提，两节内容本质是"定位方式"与"定位强度"的关系。
  • 继承：层叠算法只解决"多条规则同时命中同一元素"的冲突，而继承解决的是"没有任何规则命中"时的兜底取值——两者是级联体系里互补的两级：先看有没有直接声明并裁决优先级，没有才轮到继承上场。
  • CSS 变量：var() 的取值同样受层叠影响——同名自定义属性在不同选择器中重复定义时，一样按特异性和源顺序决出胜者；但它的"无效计算值"陷阱（invalid at computed-value time）是层叠算法之外的额外一层规则，学完本节的层叠裁决顺序，才能理解为什么变量失效时不会回退到同属性的上一条声明。
  • JS 动态样式：通过 element.style.xxx 设置的内联样式，特异性高于绝大多数选择器（但仍低于 !important），这正是本节"内联样式的特异性"一节的内容；理解这一点才能明白为什么有时候 JS 设置的样式"怎么用 CSS 都覆盖不掉"，必须用 !important 或改回用 JS 修改。
  • React 组件样式方案：CSS-in-JS、CSS Modules 通过生成唯一类名规避了传统项目中常见的"特异性军备竞赛"，本质上是用命名空间隔离代替了 !important 和 @layer 的裁决；理解层叠算法的痛点，才能理解这些工具链方案要解决的根本问题是什么。
  • 性能优化：浏览器为每条规则预先计算特异性并缓存，样式重算阶段要为每个 DOM 节点排序匹配到的所有规则——规则数量越多、层叠层越复杂，样式计算成本越高，这是大型项目里"CSS 越堆越多、页面越跑越慢"的一个隐性原因。
