CSS · 盒模型

盒模型

页面上每一个元素，本质上都是一个矩形盒子。理解盒模型，就理解了 CSS 布局的基本单位。

一、为什么是「盒子」

浏览器在渲染时，会把每一个 HTML 元素（无论是 <div> 还是一行 <span> 文字）都抽象成一个矩形盒子。这是浏览器布局引擎（layout engine，如 Blink、WebKit）的底层假设：整个页面就是一堆嵌套的、互相影响的盒子。你写的所有 width、padding、margin，都是在描述「这个盒子长什么样、占多大地方、和别人隔多远」。

之所以这样设计，是因为矩形是最容易计算的几何形状。布局引擎只需要算出每个盒子的位置和尺寸（一对坐标 + 宽高），就能把整个页面摆放出来。如果允许任意形状，碰撞检测、换行、对齐的计算复杂度会爆炸。这就是「CSS 万物皆盒」的来历。

二、盒子的四层结构

从内到外，一个盒子由四层同心矩形构成：

层属性作用类比
content 内容width / height真正放文字、图片的区域画框里的画
padding 内边距padding内容与边框之间的留白画与画框木条之间的卡纸
border 边框border盒子的边界线画框的木条
margin 外边距margin盒子与其他盒子的距离两幅画挂墙上的间隔

关键区别：padding 在 border 内侧，背景色会延伸到 padding；margin 在 border 外侧，永远透明，只负责「推开别人」。这个内外之分是后面理解 box-sizing 和 margin 折叠的前提。

四、标准盒模型如何计算总宽度

这是面试和踩坑的高频点。在标准盒模型（content-box，浏览器默认）下，你设置的 width 只代表 content 的宽度，padding 和 border 是额外往外加的：

CSS
.box {
  width: 200px;
  padding: 20px;
  border: 5px solid;
}
/* 盒子在页面上真正占据的宽度： */
/* content 200 + 左右 padding 40 + 左右 border 10 = 250px */

用一个公式记住：

盒子实际宽度 = width + 左右 padding + 左右 border
（margin 不算在「盒子尺寸」里，它是盒子之外的间隔，但会影响盒子占的总空间）

五、最容易踩的坑

反例：两个盒子各设 width: 50%，想并排放满父容器。一旦你给它们加了 padding: 20px，总宽度就变成 50% + 40px，两个加起来超过 100%，第二个被挤到下一行。

问题根源：标准盒模型下 width 不含 padding/border，你以为的「50% 就是一半」其实在加内边距后变胖了。百分比宽度 + 像素内边距混用时尤其隐蔽。

正例：把盒模型切到 box-sizing: border-box，让 width 包含 padding 和 border，50% 就真的是 50%。这正是下一节的主角。

六、height 的特殊性

顺带提醒：width 默认会撑满父容器（块级元素），但 height 默认由内容撑开（auto）。给块级元素写 height: 100% 常常不生效，因为百分比高度需要父元素有明确高度才有参照。这是初学者第二常见的盒模型困惑，记住「宽度看父级、高度看内容」这个默认行为差异。

小结

  • 页面万物皆盒：content → padding → border → margin 四层同心矩形。
  • padding 在边框内（吃背景色），margin 在边框外（永远透明，负责推开别人）。
  • 标准盒模型下 width 只是 content 宽，实际宽 = width + padding + border。
  • 百分比宽度 + 像素 padding 混用必踩坑，解药是下一节的 border-box。

与其他知识点的联系

  • box-sizing：盒模型只定义了四层结构和"标准算法"（width 只算 content），下一节 box-sizing 是在这个基础上提供了第二种算法（border-box），本质是同一套四层结构、两种不同的宽度计算方式，必须先理解本节的四层才能理解切换算法的意义。
  • margin 折叠：margin 是盒模型四层里唯一会"折叠"的一层——padding、border 永不折叠，正是因为它们在 border 内侧参与占位计算，而 margin 在盒子之外，浏览器才有空间对它做合并优化。理解 margin 在盒子结构中的位置，才能理解为什么折叠只发生在这一层。
  • JS DOM 尺寸获取：element.offsetWidth（含 padding+border）、clientWidth（含 padding 不含 border）、getBoundingClientRect().width 这几个 API 返回值互不相同，根源就是它们分别对应盒模型的不同层——不理解四层结构，就无法正确选择该用哪个 API 取尺寸。
  • flex/grid 布局：flex 和 grid 在分配子项空间时，仍然要先计算每个子项的盒模型尺寸（content+padding+border），再考虑 gap、flex-basis 等布局属性——盒模型是所有布局算法的地基，flex/grid 只是在这层地基之上叠加了新的空间分配规则。
  • 性能优化：修改 width、padding、border 这些盒模型属性会触发回流（reflow）——浏览器要重新计算受影响元素及其后续兄弟的几何位置，这是"少用 JS 频繁改布局属性、优先用 transform 做动画"这条性能建议的根本原因。
