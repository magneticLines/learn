CSS · 响应式

媒体查询 @media

同一套 HTML，让样式根据屏幕大小、设备类型、用户偏好自动切换——这是响应式设计的开关。

一、为什么需要媒体查询

2007 年 iPhone 出现之前，网页基本只在桌面浏览器跑，固定 960px 宽布局够用。今天一个页面要同时面对 360px 的手机、768px 的平板、2560px 的显示器。如果只能写一套固定样式，要么手机上字小到看不清、横向滚动；要么桌面上内容挤在中间一条。

媒体查询（media query）解决的就是「同一份文档，按运行环境给不同样式」。它的本质是一个条件块：只有当浏览器环境满足条件时，里面的 CSS 才生效。这和编程里的 if 是一个思路，只不过判断的对象是「视口宽度、设备方向、用户系统偏好」这些环境变量。

理解关键：媒体查询不改变 HTML 结构，只切换样式。响应式 = 一套结构 + 多套适配样式。这区别于「为手机单独做一个 m.xxx.com 站点」的老做法。

二、基本语法

@media 语法骨架
@media screen and (min-width: 768px) {
  .sidebar {
    display: block; /* 仅当视口 ≥768px 时显示侧栏 */
  }
}

拆开看：@media 是关键字；screen 是媒体类型（media type）；and 连接条件；(min-width: 768px) 是媒体特性（media feature）。整体读作「在屏幕设备上，且视口宽度不小于 768px 时」。

三、媒体类型：screen / print / all

类型含义典型用途
all所有设备（默认）省略类型时即 all
screen屏幕（手机/平板/电脑）绝大多数响应式
print打印 / 打印预览打印时隐藏导航、去掉背景色省墨
打印样式：常见实战
@media print {
  .nav, .footer { display: none; } /* 打印不需要导航 */
  body { color: #000; background: #fff; }
}
早期还有 handheld、tv 等类型，现已基本废弃——因为「手机也是 screen」。今天 99% 的响应式只用 screen，甚至直接省略类型。

四、媒体特性：真正的判断条件

1. 尺寸特性 min-width / max-width

最常用。min-width: 768px 表示「视口 ≥ 768px 时成立」，max-width: 767px 表示「≤ 767px 时成立」。注意是视口宽度（viewport），不是物理屏幕宽度。

2. orientation 方向

横竖屏适配
@media (orientation: landscape) { /* 横屏：宽>高 */
  .gallery { grid-template-columns: repeat(4, 1fr); }
}

3. prefers-color-scheme 深色模式

这是「用户偏好特性」，读取的是操作系统的深色/浅色设置。无需 JS 即可自动跟随系统主题。

跟随系统深色模式
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0f1117;
    --fg: #e2e8f0;
  }
}

4. prefers-reduced-motion 减少动画

无障碍（a11y）特性。部分用户（前庭功能障碍者）开启系统「减少动态效果」后，应关闭大幅动画，避免眩晕。这是负责任的前端必须考虑的。

尊重减少动画偏好
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

五、逻辑组合：and / 逗号 / not

运算符含义例子
and逻辑「且」，多条件同时成立(min-width: 768px) and (max-width: 1024px)
,（逗号）逻辑「或」，任一成立即可(max-width: 600px), (orientation: portrait)
not取反整条查询not all and (min-width: 768px)
陷阱：媒体查询里没有 or 关键字，「或」用逗号表达。另外 not 是对整条查询取反，不能写 (not min-width: ...) 这种局部取反。

区间查询：只在某个范围生效

仅平板（768px ~ 1023px）
@media (min-width: 768px) and (max-width: 1023px) {
  /* 只有平板尺寸命中 */
}

/* CSS Media Queries Level 4 新区间语法（现代浏览器） */
@media (768px <= width < 1024px) { }

六、断点设计：怎么选 768、1024 这些数？

断点（breakpoint）是「样式发生切换的临界宽度」。新手常照搬某个框架的固定值（如 Bootstrap 576/768/992/1200），但更正确的思路是：让内容决定断点。当你把窗口慢慢拉宽，看到布局开始难看（一行字太长、卡片太挤）的那个点，就是该加断点的地方。

设备档位常用断点参考
手机< 640px
平板≥ 768px
笔记本≥ 1024px
大屏≥ 1280px / 1536px
这些数字只是起点，不是教条。一篇正文阅读页可能只需要一个「≥ 65ch 就居中限宽」的断点，不必硬套四档。

七、为什么 min-width 配合「移动优先」更好

有两种写法方向：

  • 桌面优先（desktop-first）：先写桌面样式，用 max-width 逐级覆盖到小屏。
  • 移动优先（mobile-first）：先写手机样式作为基础，用 min-width 逐级增强到大屏。

推荐移动优先（min-width），原因：

  1. 基础样式最简单：手机布局通常是单列，作为默认值改动最少。
  2. 渐进增强：从简单加到复杂，符合「小屏拿到的 CSS 也最少」，对弱网手机友好。
  3. 覆盖关系清晰：min-width 断点天然按从小到大叠加，不易出现 max-width 多个区间互相打架。

八、容器查询 @container：媒体查询的「升级补丁」

媒体查询有个根本局限：它只能问「视口多宽」，不能问「我所在的这个容器多宽」。但一个卡片组件可能既出现在窄侧栏里，又出现在宽主区里——它真正该响应的是父容器宽度，而不是整个屏幕。

@container 容器查询
/* 1. 声明某元素为查询容器 */
.card-list { container-type: inline-size; }

/* 2. 根据容器宽度（而非视口）切换卡片布局 */
@container (min-width: 400px) {
  .card { display: flex; }
}
对比@media@container
判断对象视口 / 设备最近的查询容器
组件可复用性差（依赖全局视口）强（自适应所在位置）
适用场景整页布局、主题可复用组件内部
2023 年起主流浏览器已支持 @container。趋势是：整页骨架用 @media，组件内部用 @container，两者互补而非替代。

九、实时演示：缩放浏览器窗口看变化

拖动浏览器窗口宽度跨过 600px，方框颜色和文字会切换——这就是 @media 在起作用。

十、常见陷阱清单

  • 忘了 viewport meta 标签：手机会用 980px 虚拟视口渲染，媒体查询全部失效。必须在 <head> 加 <meta name="viewport" content="width=device-width, initial-scale=1">（下一节详述）。
  • min/max 边界重叠：max-width:768px 和 min-width:768px 在正好 768px 时同时成立，按源码顺序后者覆盖前者，易出诡异 bug。区间用 max-width:767.98px 错开。
  • 桌面优先写一堆 max-width 互相覆盖：维护噩梦，优先 min-width 移动优先。
  • 用 JS 判断屏幕宽度做布局：能用 CSS 解决的别上 JS，CSS 媒体查询由浏览器原生处理，性能与可靠性都更好。
