CSS · 响应式

相对单位

px 写死的尺寸无法适应不同屏幕、不同字号偏好。相对单位让布局「会呼吸」——这是响应式的基石。

一、绝对单位 vs 相对单位

绝对单位（主要是 px）是固定的：写 16px 就永远是 16 个 CSS 像素，不随任何东西变化。它直观、可控，但「死」。

相对单位的值取决于「相对谁」：相对父元素字号、相对根字号、相对视口大小……它们让一个值能根据上下文自动伸缩。响应式设计追求的就是「设一次，处处自适应」，所以相对单位是主角。

注意：CSS 的 px 不等于物理像素。在高分屏（Retina）上，1 CSS px 可能对应 2×2 或 3×3 个物理像素，由设备像素比（DPR）决定。CSS px 是「逻辑像素」，浏览器已帮你抹平了物理差异。

二、em：相对父元素的 font-size

1em = 当前元素继承到的 font-size。如果父元素字号是 20px，子元素里 1em = 20px，1.5em = 30px。

em 基本用法
.card { font-size: 20px; }
.card .title {
  font-size: 1.5em;  /* = 20 × 1.5 = 30px */
  padding: 0.5em;     /* = 30 × 0.5 = 15px（注意基准是本元素的字号30）*/
}
关键细节：em 用在 font-size 上时基准是父元素字号；用在 padding/margin/width 等其他属性上时，基准是本元素自己计算后的 font-size。这个区别是 em 难懂的根源。

em 的「复利」陷阱（反例→问题→正例）

反例：嵌套 em 字号失控
ul { font-size: 1.2em; }
/* 三层嵌套 ul，每层都 ×1.2 */
/* 第1层 1.2em，第2层 1.44em，第3层 1.728em —— 越深越大 */

问题：em 相对父级，多层同名嵌套时层层相乘（复利效应），字号像滚雪球一样失控。导航菜单、嵌套列表最容易踩这个坑。

正例：用 rem 切断继承链
ul { font-size: 1.2rem; }
/* rem 永远相对根字号，不管嵌套多深都是 1.2rem，不累积 */

三、rem：相对根元素的 font-size

rem = root em，永远相对 <html> 根元素的 font-size（默认 16px）。它去掉了 em 的「父级相对」特性，因此没有复利问题，是设字号、间距最稳的相对单位。

rem 的全局缩放威力
html { font-size: 16px; } /* 1rem = 16px */

.btn { padding: 0.75rem 1.5rem; font-size: 1rem; }

/* 只改一处根字号，全站按 rem 写的尺寸整体放大 */
@media (min-width: 1280px) {
  html { font-size: 18px; } /* 所有 rem 同步变大 */
}
无障碍要点：不要给 html 写死 font-size: 16px（用 px 锁死会让浏览器「调整默认字号」设置失效）。更友好的做法是不设或用百分比，让 rem 跟随用户偏好。这也是为什么很多团队推荐「字号一律用 rem」。

四、%：相对父元素的对应属性

百分比的基准随属性变化：width: 50% 相对父元素宽度；height: 50% 相对父元素高度；font-size: 120% 相对父元素字号；line-height: 150% 相对自身字号。它最常用于流式宽度布局。

陷阱：height: 100% 要求父元素有明确高度，否则百分比无参照、归零失效。这是新手「高度撑不开」的经典原因。

五、视口单位 vw / vh / vmin / vmax

视口（viewport）= 浏览器可视区域。视口单位直接相对它，1 单位 = 视口的 1%。

单位基准含义
vw视口宽度的 1%100vw = 整个视口宽
vh视口高度的 1%100vh = 整个视口高
vminvw 和 vh 中的较小者保证元素不超出任一方向
vmaxvw 和 vh 中的较大者按长边缩放
视口单位典型用途
.hero { height: 100vh; }        /* 首屏满高 banner */
.full-bleed { width: 100vw; }   /* 通栏背景 */
.square { width: 50vmin; height: 50vmin; } /* 始终方形且不溢出 */
陷阱：width: 100vw 会把纵向滚动条算进去，导致比实际内容区宽一点，出现横向滚动条。需要满宽时优先用 width: 100%。

六、ch 与 ex：相对字体度量

  • ch = 当前字体下「0」字符的宽度。最经典用法：max-width: 65ch 限制正文行宽——排版学公认每行 45～75 字符最易读。
  • ex = 当前字体小写「x」的高度，较少用，偶尔做上下标微调。
用 ch 控制最佳阅读行宽
article { max-width: 65ch; } /* 行宽随字号自适应，永远好读 */

七、新视口单位 dvh / svh / lvh：手机地址栏难题

移动端浏览器有个老大难：滚动时地址栏会自动收起/展开，可视高度随之变化。但传统 100vh 取的是「地址栏收起后的大视口高度」——结果页面首屏底部会被地址栏盖住一截，或出现一段空白。

单位含义
lvhlarge viewport height：地址栏收起时的大高度
svhsmall viewport height：地址栏展开时的小高度
dvhdynamic viewport height：随地址栏状态动态变化的当前高度
用 dvh 让首屏真正占满
.full-screen {
  height: 100vh;   /* 旧浏览器兜底 */
  height: 100dvh;  /* 现代浏览器：随地址栏自适应，不被遮挡 */
}
小心 100dvh 用在频繁滚动的高度上会导致布局抖动（地址栏一动高度就变）。固定首屏用 dvh 没问题，长内容容器谨慎使用。

八、横向对比：何时用哪个

单位相对谁推荐场景
px固定边框 1px、阴影等不需缩放的细节
rem根字号字号、间距（首选，可全局缩放）
em父字号组件内部「随字号联动」的间距、图标
%父对应属性流式宽度、栅格列宽
vw/vh视口首屏高度、通栏、流体字号配 clamp
vmin/vmax视口长/短边保持比例不溢出的方形/图形
ch字符宽正文限宽 max-width: 65ch
dvh/svh/lvh动态视口移动端满屏，规避地址栏问题
一句话经验：字号和间距用 rem；宽度用 % 或 fr；首屏高度用 dvh；行宽用 ch；需要随字号联动的局部用 em。下一节会看到，rem + vw 还能用 clamp() 组合出「流体排版」。
