BEM 命名规范

CSS 没有作用域、没有命名空间，类名是全局的。BEM 用一套朴素的命名约定，把「自律」变成「规则」，从根上治住命名冲突和选择器特异性失控。

一、问题：CSS 全局命名空间的两个老毛病

原生 CSS 的所有类名都活在同一个全局空间里。这意味着两个开发者、两个组件，只要起了同名的 .title，就会互相覆盖——而且覆盖谁取决于谁后加载、谁特异性高，结果难以预测。为了「保证我的样式只作用在我的范围内」，很多人会写成深层嵌套选择器：

反例：靠嵌套选择器「圈地」
/* 想让卡片内的标题不影响别处，于是层层限定 */
.sidebar .card .header .title { font-size: 18px; }
.sidebar .card.featured .header .title { color: gold; }

这会带来什么问题？

  • 特异性失控：四层后代选择器的特异性是 (0,4,0)，将来想覆盖它，要么写得更长，要么动用 !important——特异性军备竞赛就此开始。
  • 结构耦合：选择器写死了 HTML 嵌套关系，一旦把 .title 从 .header 里挪出来，样式立刻失效。CSS 和 DOM 结构被焊死。
  • 不可复用：这个 .title 只在「sidebar 里的 card 里」生效，换个地方就废了。

二、BEM 是什么：Block / Element / Modifier

BEM 是 Yandex 提出的命名方法论，把界面拆成三类概念，用固定的分隔符拼成类名：

部分含义分隔符例子
Block独立的、可复用的功能块（组件本身）—card
ElementBlock 的组成部分，离开 Block 没有意义__ 双下划线card__title
ModifierBlock 或 Element 的变体/状态-- 双连字符card--featured

读法很直观：card__title 就是「card 这个块的 title 元素」，card--featured 是「featured 状态的 card」。一看类名就知道它属于谁、是什么角色。

为什么用 __ 和 -- 这种「双字符」？因为 Block 名本身可能含单连字符（如 search-form）。用双字符做分隔，才能和命名里的连字符区分开：search-form__input--disabled 一眼可拆。

三、正例：把上面的反例改成 BEM

正例：BEM 扁平选择器
.card { padding: 16px; border: 1px solid #2d3148; }
.card__title { font-size: 18px; }
.card--featured { border-color: gold; }
.card--featured .card__title { color: gold; }

对应的 HTML：

HTML
<article class="card card--featured">
  <h3 class="card__title">特色卡片</h3>
  <p class="card__body">正文内容……</p>
</article>

问题怎么解决的？大多数选择器都是单个类，特异性恒为 (0,1,0)，平等好覆盖；类名自带命名空间，card__title 全局唯一，不会撞车；CSS 不再依赖 DOM 嵌套，把元素挪到哪它都认 card__title 这个类。

四、BEM 与「低特异性扁平选择器」的关系

BEM 的真正价值不在分隔符长得怪，而在于它逼你用单类选择器。当每个规则的特异性都一样时，层叠就退化成最简单的「后者覆盖前者」——也就是按源码顺序决定，这正是 CSS 最容易预测的状态。这与 ITCSS、SMACSS 等架构强调的「让特异性保持平坦」是同一思路。

记住对比：嵌套选择器用「位置」限定作用域（脆弱、特异性高）；BEM用「命名」限定作用域（健壮、特异性平坦）。这是从「依赖结构」到「依赖约定」的范式转变。

真实组件外观（BEM 命名后）

card · 特色卡片

类名：card card--featured / card__title / card__body。边框与标题色由 --featured 修饰符控制。

五、优缺点与替代方案

维度评价
可读性类名自解释，看 HTML 就能猜出结构关系
可维护特异性平坦、无嵌套耦合，改样式不怕连锁反应
缺点类名又长又啰嗦（nav__item--active）；HTML 里堆满长类名；需要团队纪律统一执行

其它命名/架构流派：SMACSS（按 Base/Layout/Module/State 分类，状态用 is-active 前缀）、OOCSS（结构与皮肤分离、可复用对象）、ITCSS（按特异性分层组织文件）。更现代的方案则直接从工具层面消灭命名冲突——CSS Modules（编译期生成局部 hash 类名）和 Tailwind（原子化功能类，干脆不给组件起名），后续章节会讲。BEM 仍是「原生 CSS / 无构建工具」场景下最稳的手写约定。

常见陷阱：① 不要写「Element 套 Element」的类名，如 card__header__title——BEM 只有一层 Element，应写成 card__title，扁平即可。② Modifier 不能单独用：card--featured 必须和 card 一起出现在元素上，它只是「附加状态」。

与其他知识点的联系

  • will-change 与性能优化：「动画时加 will-change，动画结束后移除」用的是 JS 临时切换状态，如果换成纯 CSS 驱动，很自然会落到 BEM 的状态 Modifier 上（如 .card--animating 触发 will-change: transform），用类名表达生命周期状态是两者共通的思路。
  • border-radius / 渐变 / filter：「圆角胶囊按钮」「渐变文字」「毛玻璃卡片」这些视觉效果往往需要组合好几行 CSS 属性才能实现，如果每个组件都重新拼一遍，样式会散落各处难以维护；BEM 的价值就是把这类固定组合封装成一个 Modifier 类，改一处即可全局生效。
  • 文字溢出：「标题单行省略」的三件套写法几乎是通用的，适合抽成一个可复用的工具类而不是重复内联——这引出了 BEM 之外的另一种思路：像 Tailwind 那样直接用原子类描述效果，而不是给每个业务组件单独起 Element 名，两种方法论各有取舍。
  • CSS 重置（下一节）：CSS 重置解决的是「浏览器默认样式不一致」，BEM 解决的是「团队协作时类名会冲突、选择器会失控」——两者是同一个「让 CSS 更可控」目标下的不同层面：一个管起点，一个管组织方式，实际项目里通常一起使用。
  • CSS Modules / Tailwind（后续预告）：BEM 靠命名约定人为避免冲突，本质是「社会共识」而非技术强制；CSS Modules 在编译期生成哈希类名从根本上消除冲突，Tailwind 则干脆不给业务组件起名——理解 BEM 的动机，才能理解这些更现代方案分别「继承」了它的哪部分思路、又「替代」了哪部分。
