CSS Modules / Tailwind 思路（进阶）

BEM 靠纪律手动避免命名冲突，预处理器没解决作用域。现代方案干脆从工具层面下手：要么自动生成局部类名，要么干脆不给组件起名。

一、根本问题：CSS 没有作用域

前几章反复提到一个老毛病——CSS 类名是全局的。两个组件都写 .title 就会互相污染。解决之路其实是一条演进线：

  • BEM：靠命名约定「假装」有作用域（人力维护，会出错）。
  • CSS Modules：让构建工具自动生成唯一类名，真正实现局部作用域。
  • CSS-in-JS：把样式写进 JS，运行时/编译期生成带 hash 的类名。
  • Tailwind：换个思路——不为组件起名，用海量预定义的「原子类」直接拼。

二、CSS Modules：编译期 hash 局部作用域

你照常写 .title，但文件名带 .module.css。构建工具会把它编译成全局唯一的 hash 类名（如 Card_title__a1b2c），并在 JS 里给你一个映射对象。这样每个组件的 .title 互不干扰。

Card.module.css
.card { padding: 16px; }
.title { font-weight: 700; }
Card.jsx
import styles from './Card.module.css';
// styles.title === "Card_title__a1b2c"（编译期生成）
<div className={styles.card}>
  <h3 className={styles.title}>标题</h3>
</div>

好处：天然组件化，再也不怕重名；保留了写普通 CSS 的手感；最终产物仍是静态 CSS，无运行时开销。代价：依赖构建工具；调试时 DevTools 里看到的是 hash 名，需要 sourcemap 辅助。

但局部作用域不是「全有或全无」——有时你确实需要写全局样式（如重置、第三方库覆盖、全局主题类）。CSS Modules 用 :global() 开一个口子：被它包住的选择器不会被 hash，保持原名输出。

Card.module.css：局部与全局混用
.card { padding: 16px; }          /* 局部 → Card_card__a1b2c */
:global(.is-active) { color: #2563eb; }   /* 全局 → 原样输出 .is-active */
/* 也可写 :global { ... } 块，让整段都走全局 */
配套还有 :local()（默认行为，可显式标注），以及组合复用的 composes。记住一句：CSS Modules 默认局部，:global() 是你主动声明「这条要全局」的逃生舱——别误以为所有样式都被强制锁进组件里。

CSS-in-JS（顺带一提）

如 styled-components、Emotion，把样式作为 JS 字符串/对象写在组件内，库在运行时（或编译期）生成 hash 类名并注入。优点是样式与组件强绑定、能用 JS 逻辑动态算样式；缺点是可能带运行时开销，且增加 bundle 体积。近年趋势偏向「零运行时」的方案（如 Vanilla Extract、Linaria）。

三、Tailwind：原子化 / Utility-First

Tailwind 提供一大批单一职责的功能类（utility class）：p-4=padding、flex=display:flex、text-blue-500=蓝色文字。你不再写自定义 CSS，而是在 HTML 里组合这些原子类拼出外观。

Tailwind：直接在标签上拼原子类
<div class="p-4 rounded-lg border flex items-center gap-3">
  <h3 class="font-bold text-lg">标题</h3>
</div>

核心思想：作用域问题被「绕过」了——原子类全局共享是有意为之的，p-4 到处都该是同一个 padding，不存在污染一说。设计 token（间距、颜色刻度）被收敛进一套配置，团队风格自动统一。生产构建时还会剔除未使用的类（purge/JIT），最终 CSS 体积很小。

原子化的争议

支持方观点反对方观点
不用为类名绞尽脑汁、不用在 HTML 与 CSS 文件间反复横跳HTML 里类名一长串，「脏」、可读性差
删组件时不会留下「孤儿 CSS」，样式随标签走本质是把样式写回了「行内」，违背关注点分离
设计系统约束强、风格一致、产物体积小需要记大量类名缩写，有学习成本
重复的类组合可用组件（React/Vue）或 @apply 收敛多处复用同一组合时仍可能重复
"关注点分离"之争：传统观点认为 HTML（结构）和 CSS（样式）该分开。Tailwind 反驳说——在组件化时代，结构与样式本就同生共死、一起复用，强行分文件反而增加心智负担。这是理念之争，没有绝对对错。

四、方案对比与选型

方案作用域开销典型场景
BEM（手写约定）靠命名（人力）零无构建工具、传统多页站点
Sass/Less不解决编译期需变量/mixin 的存量项目
CSS Modules编译期局部 hash近乎零组件化项目、想保留写 CSS 手感
CSS-in-JS运行时/编译期 hash可能有运行时开销需高度动态样式的 React 项目
Tailwind原子类全局共享（无需作用域）编译期 purge快速开发、强设计系统约束

选型建议

  • 纯静态页 / 不上构建工具：BEM + 最小重置足够。
  • 组件化但想写「正常 CSS」：CSS Modules 是平衡之选。
  • 追求开发速度、团队认同原子化：Tailwind，目前社区热度极高。
  • 样式高度依赖运行时数据：考虑 CSS-in-JS（优先零运行时方案）。
这些方案可以组合：很多团队用「Tailwind 管原子样式 + CSS 变量管主题 + 少量 CSS Modules 处理复杂局部」。没有银弹，理解每种方案「解决什么问题、代价是什么」，比记住哪个最火更重要。
陷阱：别因为「某框架很火」就盲目引入。Tailwind 对小项目可能是过度工程，CSS-in-JS 的运行时开销在性能敏感场景要谨慎评估。先想清楚你的项目规模、团队习惯和真正痛点，再选工具。

与其他知识点的联系

  • 与 BEM 及 Sass 嵌套：这三者其实是同一条"如何避免类名冲突"演进线上的三个阶段——BEM 靠命名纪律、Sass 只是让命名字符串拼接更省力、CSS Modules 才是第一个真正把作用域问题交给工具解决的方案，理解这条脉络比记住某个工具的语法更重要。
  • 与 React 组件化：CSS Modules 的 hash 类名机制天然契合 React"一个组件一个文件"的组织方式——每个组件自带一份局部作用域的样式表，组件删除时样式也随之删除，不会像全局 CSS 那样留下孤儿规则，这也是 Tailwind 反对"结构与样式分离"的论据来源。
  • 与浏览器渲染性能：CSS Modules 和 Tailwind 最终产物都是普通静态 CSS，运行时零开销；而 CSS-in-JS 方案在运行时动态生成/注入样式规则，会触发额外的样式计算甚至强制重排，这正是"零运行时 CSS-in-JS"（Vanilla Extract 等）近年受欢迎的性能动机。
  • 与构建工具（Webpack/Vite）：CSS Modules 的 hash 类名生成、Tailwind 的 JIT 按需编译和 purge 未使用类，都不是浏览器行为，而是构建阶段由 loader/插件完成的——这解释了为什么这些方案离不开现代构建工具链，纯静态页面项目很难直接享受它们的好处。
  • 与响应式设计 / 设计系统：Tailwind 的间距、颜色刻度本质是把设计系统的 token 编码进配置文件，强制团队复用同一套数值而不是各自硬编码——这与用 CSS 变量集中管理主题色是同一种"约束换一致性"的思路，只是落地方式不同。
