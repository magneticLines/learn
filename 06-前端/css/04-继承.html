CSS · 基础概念

继承

子元素自动获得祖先元素某些属性值的机制——理解它，才能写出真正简洁、可维护的样式。

什么是 inheritance（继承）

CSS 的 继承 是指：当一个元素没有为某个属性声明值时，浏览器会沿 DOM 树向上查找，把离它最近的祖先元素该属性的计算值（computed value）作为自己的值使用。

举个直觉类比：公司统一规定「员工手册字体用宋体」，每个员工（子元素）不用自己再声明字体——他们继承了公司（父元素）的规定。只有少数岗位（如工程图纸部门）需要自己特别指定别的字体。

为什么需要继承？

想象没有继承会怎样：你要给一个页面里的每个 <p>、<span>、<li>、<td>……全部单独写 color: #333; font-family: sans-serif;，不仅冗余，而且一旦设计改色，要改几百处。

有了继承，只需在 body 写一次，全站文字默认统一。这正是 CSS 设计者的意图：用最少的声明覆盖最大的范围，特殊情况再局部覆盖。

浏览器原理视角： 浏览器在构建渲染树（Render Tree）时，对每个元素的每个属性都要确定一个「使用值（used value）」。继承就是 CSS 级联算法在「没有任何声明匹配」这一步的兜底规则：可继承属性回溯父节点，不可继承属性回退到属性规范的初始值（initial value）。

哪些属性默认继承

默认继承的属性集中在文本、字体、列表、光标等与「内容排版」强相关的属性上。它们有一个共同特征：传递给后代元素通常仍然合理，不会破坏布局。

分类 属性 说明
字体 font-family font-size font-weight font-style font-variant font 字体家族、大小、粗细、斜体等，子元素保持一致的字型体验
文本 color line-height letter-spacing word-spacing text-align text-indent text-transform white-space direction 文字颜色、行高、字距等排版属性
列表 list-style list-style-type list-style-position list-style-image 列表项符号，嵌套列表共用父级样式
可见性 / 指针 visibility cursor visibility: hidden 会让子元素一起隐藏（与 display: none 不同，子元素可用 visibility: visible 重新显示）
其他 quotes orphans widows border-collapse（表格）caption-side 引号、孤行控制、表格边框合并等
记忆技巧： 凡是「写在纸上的字的样子」相关的属性，基本都继承。颜色、字体、行高、对齐——这些属性让整篇文章风格统一，继承非常合理。

哪些属性不继承

盒模型、布局、背景、边框类属性默认不继承。它们描述的是元素自身的「形状和位置」，如果传递给后代，会产生灾难性的布局错乱。

分类 属性 为何不继承
盒模型 width height min/max-width min/max-height margin padding border 子元素若继承父级的 width: 500px，每层嵌套都是 500px，完全错乱
背景 background background-color background-image 等 背景属于元素自身画布，默认 transparent 让父级背景透过来，视觉上像继承但不是
布局 display position top right bottom left float clear z-index overflow 每个元素的布局角色独立，子元素不应自动脱离文档流
Flex / Grid flex grid align-items justify-content 等 这些是容器属性，只对直接子项生效，无需继承
变换 / 动画 transform transition animation opacity 视觉效果独立于元素，不应级联给后代
轮廓 / 阴影 outline box-shadow 装饰性效果属于元素自身

为什么这样设计——直觉推导

CSS 规范对「哪些属性可继承」有一个简洁的判断标准：如果把这个属性的值逐字传递给每一个后代元素，结果仍然合理，那就让它继承。

  • color: red 传给子元素？合理——整段文字都是红色，视觉统一。
  • font-size: 16px 传给子元素？合理——统一的字号，除非局部需要调整。
  • width: 600px 传给子元素？不合理——一个宽 600px 的 div 里嵌套的每个 span、p、div 都变成 600px，完全破坏布局。
  • border: 1px solid black 传给子元素？不合理——每个嵌套元素都出现边框，页面变成无数个黑框叠套。
  • background-color: blue 传给子元素？不合理——子元素默认 transparent 正好让父级背景透出，若继承则子元素会覆盖掉父级背景，反而更难用。
面试考点： "background 为什么不继承，但看起来子元素好像有父级的背景色？"——答：子元素默认 background-color: transparent（初始值），透明让父级背景色"透"过来，视觉上像继承，但本质是透明叠加，不是继承。可以用 background-color: red 盖掉，而真正的继承是无法被覆盖的（除非重新声明）。

真实 Demo：继承在 DOM 树中的传递

下面的父容器设置了 color、font-family、line-height，子元素和孙元素没有任何声明，却自动获得了这些样式：

父元素 — color: #a78bfa; font-family: Georgia; line-height: 2
这段文字的颜色和字体来自父元素的声明。
子元素 — 无任何 color / font 声明
子元素自动继承了紫色 + Georgia 字体 + 2 倍行高。
孙元素 — 无任何声明
孙元素同样继承，颜色、字体、行高与祖先一致。
CSS父级统一声明，子孙自动继承
.parent {
  color: #a78bfa;        /* 子孙都继承这个颜色 */
  font-family: Georgia, serif;
  line-height: 2;
}

/* .child 和 .grandchild 不需要任何声明 */
/* 它们的 color / font-family / line-height 均来自继承 */

对比一下，不继承的属性（border、padding、background）就不会传递：

父元素 — width:180px; border:3px solid red; padding:12px
子元素 — 没有继承 width/border/padding，自适应宽度，无边框

四个继承控制关键字

CSS 提供了四个通用属性值关键字，可用于任意属性，精确控制继承行为。这是面试和实战的高频考点。

关键字 含义 典型用途
inherit 强制继承父元素的计算值，无论该属性是否默认继承 让 <a> 继承父级颜色；让 border-color 跟随 color
initial 重置为 CSS 规范定义的属性初始值（注意：不是浏览器默认样式） 消除所有样式的「归零」操作；display: initial = inline（不是 block！）
unset 若属性默认继承 → 等同 inherit；若不继承 → 等同 initial 重置自定义属性而不破坏继承链；配合 all: unset 清除元素所有样式
revert 回退到浏览器 UA 样式表（或用户样式表）的值，而非规范初始值 撤销作者样式，恢复浏览器默认外观；比 initial 更"人性化"

inherit vs initial vs unset vs revert 的核心区别

以 display 为例：浏览器默认 <div> 是 block，但规范 initial value 是 inline。

  • display: inherit → 取父元素的 display 值（父是 flex 则子也 flex）
  • display: initial → inline（规范初始值，不是 block！）
  • display: unset → display 不默认继承，所以等于 initial → inline
  • display: revert → 浏览器默认值 → block（div 的 UA 样式）
陷阱： initial 不等于「浏览器默认」。color: initial 得到的是 black（规范初始值），而 display: initial 得到的是 inline，不是 block。要恢复浏览器默认样式，用 revert，不用 initial。

代码示例：inherit 强制继承 border-color

CSS用 inherit 让 border 跟随文字颜色
.button {
  color: #3b82f6;
  border: 1px solid;
  border-color: inherit;  /* border-color 不继承，但 inherit 强制它用父的 color */
}

/* 修改 color 时，border-color 自动跟着变，不用两处同步修改 */
.button.danger {
  color: #ef4444;  /* border 自动变红，无需额外声明 */
}

代码示例：all: unset 清除元素所有样式

CSSall 属性配合 unset/revert 做样式重置
/* 清除按钮的所有浏览器默认样式 */
button {
  all: unset;   /* 继承属性→inherit，非继承属性→initial */
}

/* 更推荐：revert 保留浏览器焦点/无障碍行为 */
button {
  all: revert;
}

父元素 — color: #f59e0b; border: 2px solid #78716c
父级文字是橙色，边框是灰色
color: inherit; border-color: inherit
继承父的橙色文字 + 橙色边框（border-color 被 inherit 强制继承）

color 回到规范初始值（黑色），border-color 回到初始值（通常也是黑色）

用继承减少重复代码：反例 → 问题 → 正例

反例：每个元素重复声明字体和颜色

CSS — 反例冗余、难维护
h1 { color: #e2e8f0; font-family: "PingFang SC", sans-serif; }
h2 { color: #e2e8f0; font-family: "PingFang SC", sans-serif; }
p  { color: #e2e8f0; font-family: "PingFang SC", sans-serif; }
li { color: #e2e8f0; font-family: "PingFang SC", sans-serif; }
td { color: #e2e8f0; font-family: "PingFang SC", sans-serif; }
/* ... 还有几十个元素 */

问题：

  • 严重冗余——同样的两行写了几十遍。
  • 难以维护——设计改色时要逐个修改，遗漏一处就出 bug。
  • 优先级污染——这些低选择器特异性的声明散布各处，覆盖时逻辑混乱。

正例：父级统一声明 + 局部覆盖

CSS — 正例一处声明，全局生效
/* 根元素统一设定全站文字基础样式 */
body {
  color: #e2e8f0;
  font-family: "PingFang SC", "Microsoft YaHei", sans-serif;
  font-size: 16px;
  line-height: 1.8;
}

/* h1/h2/p/li/td 全部自动继承，不用重复声明 */

/* 只在需要差异化的地方覆盖 */
h1 { color: #f1f5f9; font-size: 2rem; }
.muted { color: #64748b; }   /* 局部灰色文字 */
.code  { font-family: Consolas, monospace; }
最佳实践： 在 body（或 :root）声明全站文字基调，在主题容器（如 .article、.sidebar）设置局部文字风格，只在叶子元素处覆盖个例。遵循「越靠近根越通用，越靠近叶越特殊」的原则。

陷阱 & 面试考点

陷阱 1：<a> 链接颜色不继承父级 color

你在父元素上设了 color: white，但 <a> 仍然显示浏览器默认的蓝色。原因：浏览器 UA 样式表用具体属性值（如 color: -webkit-link）显式声明了 <a> 的颜色，这个声明的来源优先级高于继承值。

本质： 继承值在 CSS 级联中处于最低优先级——它只在「没有任何来源的声明」时才生效。UA 样式表已经有声明，所以继承值被覆盖。修复方式：显式声明 a { color: inherit; }，强制 inherit 关键字让链接取父级颜色。
CSS修复 a 链接颜色不继承的问题
/* 错误期望：a 会继承父级的 color */
.nav {
  color: white;   /* a 仍然是蓝色，因为 UA 样式表胜出 */
}

/* 正确修复 */
.nav a {
  color: inherit;   /* 显式 inherit，优先级高于 UA 声明 */
}

陷阱 2：表单元素 input / button 默认不继承字体

<input>、<button>、<textarea>、<select> 的 font 属性在 UA 样式表中被显式设置为 system-ui 或操作系统 UI 字体，不会继承 body 的字体设置。这是一个常见的跨浏览器兼容问题。

CSS修复表单元素字体不继承
/* CSS Reset 的标配写法 */
input, button, textarea, select {
  font: inherit;   /* 强制继承父级字体，统一全站排版 */
}

陷阱 3：继承的是「计算值」—— line-height 的继承陷阱

这是最容易出错的细节。line-height 继承的是计算值，而不是声明值。这导致了百分比与数字值的继承行为完全不同：

CSSline-height 继承：数字 vs 百分比
/* 方式 A：用纯数字（推荐） */
body {
  font-size: 16px;
  line-height: 1.5;   /* 子元素继承 1.5（比例因子）*/
}
/* 子元素 font-size: 24px → line-height = 24 * 1.5 = 36px ✓ 正确缩放 */

/* 方式 B：用百分比（陷阱！） */
body {
  font-size: 16px;
  line-height: 150%;  /* 计算值 = 16 * 150% = 24px，子元素继承 24px（固定值！）*/
}
/* 子元素 font-size: 24px → line-height 仍然是 24px → 行高 = 字号，文字拥挤 ✗ */
面试高频： "line-height 用百分比和数字有什么区别？" — 核心在于：百分比在父元素处就被换算成 px（计算值），子元素继承这个 px 值，不再随子元素字号缩放；数字则以比例因子的形式继承，子元素拿到后再乘自己的 font-size。永远用纯数字设置全局 line-height。

陷阱 4：visibility: hidden 的继承与 display: none 的区别

visibility 是可继承属性。父元素 visibility: hidden 时，子元素默认也隐藏（继承）。但子元素可以显式声明 visibility: visible 来重新显示自己——这是继承可以被覆盖的体现。

而 display: none 不是继承——它直接把元素从渲染树中移除，子元素无法通过任何 CSS 让自己显示出来。

CSSvisibility 继承可被子元素覆盖，display: none 不行
.parent { visibility: hidden; }
.child  { visibility: visible; }  /* 可以！子元素重新可见，父元素占位但不可见 */

.parent2 { display: none; }
.child2  { display: block; }  /* 无效！父元素已不在渲染树中，子元素无法显示 */

a 链接颜色为何不继承父级？ UA 样式表有显式声明，优先级高于继承值；用 color: inherit 修复
line-height 用数字 vs 百分比的区别？ 数字继承比例因子（推荐），百分比继承计算后的 px 值（陷阱）

visibility 和 display: none 的继承区别？ visibility 可被子元素覆盖；display: none 移出渲染树，子元素无法覆盖
