z-index 与层叠上下文

"为什么我把 z-index 调成 9999 了，弹层还是被挡住？"——答案藏在"层叠上下文"里。这是 CSS 最反直觉、面试最爱问的概念之一。

z-index 不是万能的

很多人以为 z-index 是个全局的"图层高度"，数字越大就越靠前。错。z-index 有两个前提：

  • 只对"定位元素"生效：元素必须设了 position 为 relative/absolute/fixed/sticky（或者是 flex/grid 的子项），z-index 才有意义。给一个 position: static 的元素写 z-index，毫无作用。
  • 比较只在同一个"层叠上下文"内进行：z-index 的大小较量，仅限于同一个层叠上下文里的兄弟之间。跨上下文时，子元素的 z-index 再大也跳不出父级的"楼层"。

什么是层叠上下文（stacking context）

层叠上下文可以理解成一栋独立的"楼"。每栋楼内部有自己的楼层编号（z-index），但楼与楼之间的高低，由楼本身在城市里的位置决定，而不是看楼里某一层有多高。

一个元素一旦创建了层叠上下文，它的所有后代就都被"关进"这栋楼里，后代的 z-index 无论多大，都只在这栋楼内部排序，绝不可能越过这栋楼去和外面的元素比高低。

哪些情况会创建层叠上下文

条件说明
根元素 <html>天生是顶层层叠上下文
position 非 static + z-index 非 auto最经典的创建方式
position: fixed / sticky无需 z-index 就会创建
opacity 小于 1哪怕 0.99 也会创建！极易踩坑
transform / filter / perspective 非 none动画类属性常意外创建
will-change 指定了上述属性性能优化时易触发
flex/grid 子项 + z-index 非 auto无需 position
isolation: isolate专门用来"主动"创建，解决问题的利器

经典陷阱：父级压死子级

这是必考题。看下面的结构，凭直觉 .popup 的 z-index 高达 9999，应该盖住一切，但实际它被 z-index 仅为 2 的 .other 挡住了：

反例：为什么 9999 压不过 2？
.parent { position:relative; z-index:1; } /* 楼A，1层 */
  .popup  { position:absolute; z-index:9999; } /* 楼A里的9999层 */

.other  { position:relative; z-index:2; } /* 楼B，2层 */

问题出在哪

.parent（position+z-index:1）创建了一栋楼 A，整栋楼的"海拔"是 1。.other 是另一栋楼 B，海拔 2。.popup 的 9999 只是楼 A 内部的楼层，再高也跑不出楼 A 这个海拔 1 的整体。楼 B 海拔 2 > 楼 A 海拔 1，所以楼 B 整个压在楼 A 上面，.popup 自然被挡。

.popup
z-index:9999
（在楼A内）
.other z-index:2
却盖住了9999

正例：怎么修

不要继续往 .popup 上堆数字——那是治标。要么调整两栋"楼"本身的 z-index（把 .parent 提到比 .other 高），要么把 popup 移出受限的层叠上下文（如用 portal 渲染到 body 下）：

正例：在楼层级别解决
.parent { position:relative; z-index:10; } /* 抬高整栋楼A */
.other  { position:relative; z-index:2; }
/* 现在楼A(10) > 楼B(2)，popup 才真正在上面 */

opacity 创建上下文的隐蔽陷阱

最容易栽的坑：你给一个父元素加了 opacity: 0.95 做个淡淡的半透明效果，结果它内部本来正常显示的弹层突然被别的元素挡住了。因为 opacity < 1 悄悄创建了层叠上下文，把弹层"关进了楼里"。transform 做动画时同理。遇到"加了透明度/动画后层级乱了"，先想到这点。

层叠顺序规则（同一上下文内）

在同一个层叠上下文里，没写 z-index 时，元素从下到上的堆叠顺序大致是：

  1. 背景和边框（上下文的根）
  2. 负 z-index 的定位元素
  3. 普通流的 block 块
  4. 浮动元素 float
  5. 普通流的 inline 元素
  6. z-index: auto / 0 的定位元素
  7. 正 z-index 的定位元素（越大越上）

注意一个细节：z-index 相同（或都为 auto）时，后写的 DOM 元素盖在先写的上面。所以有时不用 z-index，只靠调整 HTML 顺序就能解决遮挡。

实战建议：① 不要无脑写 9999，先用浏览器 DevTools 查清各元素属于哪个层叠上下文；② 想让某个区块的内部层级"自成一体、不被外界干扰"，主动给它加 isolation: isolate；③ 全局弹层（Modal、Toast）最佳实践是用 portal 渲染到 body 末尾，从根本上避开嵌套上下文的束缚。
