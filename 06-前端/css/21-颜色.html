颜色

从 #fff 到 oklch，搞懂每种颜色语法背后「人眼怎么感知」与「机器怎么存储」的取舍，调色才不会全靠瞎猜。

一、颜色的本质：屏幕只认 RGB

无论你写 red、#ff0000 还是 hsl(0 100% 50%)，浏览器最终都要算出三个数：红、绿、蓝三个子像素各自亮多少。屏幕上每个像素由 R/G/B 三个发光点组成，三者按比例混合就是加色法（与颜料的减色法相反，所以红+绿=黄而不是脏色）。所以各种颜色语法只是「给人用的不同坐标系」，最后都换算成同一组 RGB 值。理解这点，才能明白为什么 HSL 更好调色——它换了个对人更友好的坐标系。

二、颜色关键字

最直白的写法，CSS 内置了 140 多个命名颜色。适合 demo 和快速调试，但生产代码里几乎不用——名字记不全、不成体系、也无法微调。

关键字
.box { color: tomato; background: rebeccapurple; }
/* transparent 是特殊关键字 = rgba(0,0,0,0) */
/* currentColor 也是关键字，见后文 */
冷知识：rebeccapurple（#663399）是为纪念 CSS 之父 Eric Meyer 早逝的女儿 Rebecca 而加入标准的，是唯一一个有故事的命名颜色。

三、十六进制 hex：#rgb / #rrggbb / #rrggbbaa

hex 是 RGB 的十六进制简写，每两位（00~ff，即 0~255）表示一个通道。它是最常见的写法，因为设计软件吐出来的就是 hex。

写法含义示例
#rgb3 位简写，每位自动翻倍#f30 = #ff3300
#rrggbb标准 6 位#ff3300
#rgba4 位简写，含透明度#f308
#rrggbbaa8 位，最后两位是 alpha#ff330080（50% 透明）
陷阱：#f30 的简写规则是「每位重复」，不是「补 0」。#f30 = #ff3300，绝不是 #f03000。同理 8 位的 alpha：80 是 128/255 ≈ 50%，很多人误以为 80 就是 80%。

四、rgb() / rgba()：新旧两套语法

用十进制描述三通道，可读性比 hex 好，也方便用 calc() 或 CSS 变量动态算。注意现代语法的变化：

旧语法 vs 新语法
/* 旧语法：逗号分隔，rgb 和 rgba 是两个函数 */
color: rgb(255, 51, 0);
color: rgba(255, 51, 0, 0.5);

/* 新语法（CSS Color 4）：空格分隔，斜杠加 alpha */
/* rgb() 一个函数就够了，rgba 成了别名 */
color: rgb(255 51 0);
color: rgb(255 51 0 / 50%);
/* 通道也能用百分比：rgb(100% 20% 0% / .5) */
为什么改成斜杠？因为新版颜色函数（包括 hsl/oklch）统一了「前面是颜色坐标，斜杠后面是 alpha」的格式。一致的语法更易记，解析器也更好处理。现代浏览器全支持新语法，放心用。

五、hsl() / hsla()：为什么它更适合调色

RGB 的痛点是「不符合直觉」：想让红色「亮一点」，你得同时调三个数，根本算不清。HSL 换了人脑熟悉的坐标系——色相（Hue）、饱和度（Saturation）、亮度（Lightness）：

  • H：0~360 的色环角度。0=红、120=绿、240=蓝，想换色系只改这一个数。
  • S：0%（灰）~100%（最鲜艳）。
  • L：0%（黑）~100%（白），50% 是纯色。想要 hover 变亮，L 加 10% 就行。

只改 H（色相 0→300），S/L 不变：

只改 L（亮度 25%→85%），H/S 不变——这就是一套天然的色阶：

实战价值：用 CSS 变量存 --h、--s，按钮的默认/hover/active 只改亮度即可生成整套状态色，配色系统瞬间统一。这是 HSL 相比 hex/rgb 最大的工程优势。

六、opacity vs alpha 通道：看似一样，差别很大

两者都能做「半透明」，但作用对象完全不同，这是高频面试坑：

opacity: 0.5color: rgb(.. / 50%)
作用对象整个元素（含文字、边框、子元素）仅这一个颜色属性
子元素一起变透明，无法独立不受影响
层叠上下文opacity<1 会创建新的层叠上下文不会
反例 → 问题 → 正例
/* 反例：想让卡片背景半透明，却把文字也整透明了 */
.card { background: #fff; opacity: 0.5; }
/* 问题：里面的标题、正文一起变淡，看不清 */

/* 正例：只让背景色带 alpha，文字保持不透明 */
.card { background: rgb(255 255 255 / 50%); }

七、currentColor：颜色界的「变量」

currentColor 表示「当前元素 color 的计算值」。它让边框、SVG 图标、阴影自动跟随文字颜色，一处改全部变，是写组件的利器。

currentColor 联动
.btn {
  color: #60a5fa;
  border: 2px solid currentColor; /* 边框=文字色 */
}
.btn svg { fill: currentColor; } /* 图标也跟着变 */
.btn:hover { color: #f87171; } /* 只改一处，边框+图标同步 */
文字、边框、图标 都用 currentColor 联动

八、新色彩空间 oklch 简介

HSL 虽直观，却有个隐藏缺陷：它的「亮度」不符合人眼真实感知——同样 L=50%，黄色看起来比蓝色亮得多。这导致用 HSL 生成的色阶明暗不均。oklch（CSS Color 4 引入）基于 OKLab 感知均匀色彩空间，解决了这个问题：

  • L：感知亮度（0~1），人眼觉得「一样亮」的颜色 L 就一样。
  • C：彩度（chroma），约 0~0.4。
  • H：色相角（0~360），同 HSL。
oklch
color: oklch(0.7 0.15 250);
color: oklch(0.7 0.15 250 / 50%); /* alpha 同样用斜杠 */
/* 改 H 做配色，L/C 不变 → 明暗、鲜艳度真正一致 */
趋势：Tailwind v4、现代设计系统正在转向 oklch，因为它能表达更广的 P3 广色域（更鲜艳），且渐变过渡不会出现 HSL 那种「中间发灰」的脏色。日常项目用 hsl 足够；做严肃的配色系统时，oklch 值得了解。

与其他知识点的联系

  • 关联 z-index 与层叠上下文：本节 opacity vs alpha 通道的对比小节里提到"opacity < 1 会创建层叠上下文"，这正是层叠上下文一节表格里的一条触发规则——两节合起来才能完整解释"为什么给卡片加了半透明背景，内部弹层却被意外遮挡"这个高频问题。
  • 关联相对单位与 clamp：颜色本身不是尺寸，但 HSL 里用 CSS 变量存 --h/--s/--l 再组合出状态色阶的做法，和用 rem 变量+媒体查询整体缩放尺寸的思路是同一种工程模式——都是"把可变部分抽成变量，让一处修改驱动全局联动"。
  • 关联媒体查询（prefers-color-scheme）：媒体查询一节介绍的 prefers-color-scheme 特性依赖的正是本节的颜色变量体系——深色模式通常通过在 @media (prefers-color-scheme: dark) 里重新赋值一组 HSL/oklch 颜色变量实现，不改动任何结构或类名，两节结合起来就是完整的主题切换方案。
  • 关联 Flexbox / Grid：currentColor 让边框、SVG 图标跟随文字颜色联动，这类"一处改动、多处响应"的技巧在 Flex/Grid 布局的组件（如按钮内图标+文字并排）中非常常见，理解 currentColor 能减少很多重复写颜色值的冗余代码。
  • 关联 React 组件布局：React 组件库的主题系统基本都建立在本节的颜色语法之上——用 HSL 或 oklch 定义一套基础色板，再通过调整 L（亮度）或 C（彩度）派生出 hover/active/disabled 等状态色，组件只需引用语义变量名（如 --color-primary-hover），不需要每个组件各自硬编码颜色值。
