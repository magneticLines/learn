will-change 与性能优化（进阶）

读懂浏览器渲染管线，理解为什么动画只该动 transform/opacity，以及 will-change 这把"双刃剑"何时该出鞘。

浏览器渲染管线（像素流水线）

从你改一行 CSS 到屏幕上像素更新，浏览器要走一条流水线。理解每个环节的成本，是所有动画性能优化的根基。

  • Style：计算每个元素最终生效的属性值。
  • Layout（重排/回流）：计算每个元素的几何（位置、尺寸）。最贵——改一个元素的宽度可能引发整页元素重新排布。
  • Paint（重绘）：把元素画成像素（颜色、阴影、文字）。较贵。
  • Composite（合成）：把各个已画好的图层按层级"贴"到一起。最便宜，可在 GPU 上做，且不影响其他图层。
关键洞察：越靠后的环节越便宜，且能跳过前面的环节。如果你的改动只触发 Composite，就跳过了 Layout 和 Paint，这正是丝滑 60fps 动画的秘密。

reflow vs repaint vs composite

改动的属性触发到哪一步成本
width height top left margin font-sizeLayout → Paint → Composite最高（重排）
color background box-shadow border-radius visibilityPaint → Composite中（重绘）
transform opacity filter(部分)仅 Composite最低（合成）

每秒要画 60 帧意味着每帧只有约 16.7ms 预算。在这点时间里跑完整条流水线很容易超时丢帧（卡顿）；只跑 Composite 则游刃有余。

反例 → 问题 → 正例

反例：用 left 做位移动画

反例 CSS
.box       { position: relative; left: 0; transition: left 0.3s; }
.box:hover { left: 200px; }

问题：动画的每一帧都改变 left，每帧都触发 Layout + Paint + Composite 整条流水线。元素一多、设备一弱，立刻丢帧卡顿。

正例：用 transform 做位移动画

正例 CSS
.box       { transform: translateX(0); transition: transform 0.3s; }
.box:hover { transform: translateX(200px); }

结果：每帧只做 Composite，浏览器把元素放进独立 GPU 图层，靠图层位移矩阵运算完成动画，跳过 Layout 与 Paint，丝滑省电。

悬停区域看 transform 位移。打开 DevTools 的 Rendering 面板可观察到它不触发 Layout 闪烁。

will-change：提前提升合成层

把元素提升为独立 GPU 图层是有成本的（创建图层、分配显存）。浏览器通常在动画开始的那一刻才临时提升，这一瞬间可能有微小卡顿。will-change 是给浏览器的提示："这个属性即将变化，请提前准备好图层"，从而消除起步时的抖动。

CSS
.card { will-change: transform; }   /* 告知：transform 将变化 */
.card { will-change: transform, opacity; }

will-change 的代价与正确用法

滥用代价：每个被提升的图层都占用显存/内存。给很多元素、或全局 * { will-change: transform } 这样滥用，会耗尽显存，反而拖慢整个页面，得不偿失。它是"提示"不是"命令"，过度使用还会让浏览器忽略它。
  • 该用：确实会频繁/即将动画、且当前观察到起步卡顿的少量关键元素。
  • 不该用：静态元素、大量元素、当成"性能万能药"无脑加。
  • 最佳实践：动画前用 JS 加上 will-change，动画结束后移除（设回 auto），让浏览器回收图层。不要在样式表里长期挂着。
JS — 用完即弃
el.addEventListener('mouseenter', () => {
  el.style.willChange = 'transform';  // 即将动画，提前准备
});
el.addEventListener('transitionend', () => {
  el.style.willChange = 'auto';     // 动画结束，释放图层
});
对比旧 hack：过去常用 transform: translateZ(0) 或 backface-visibility: hidden 强制提层。will-change 是为此设计的语义化、可被浏览器智能调度的正式方案，应优先使用。

其他性能优化手段

contain：隔离渲染范围

contain: layout paint 告诉浏览器"这个元素的内部布局/绘制不会影响外部"，从而把重排/重绘限制在子树内，避免牵连整页。适合独立的卡片、列表项、弹窗等组件根节点。content-visibility: auto 更进一步，可跳过视口外内容的渲染。

避免布局抖动（layout thrashing）

反例：循环里"读一次几何 → 写一次样式 → 再读 → 再写"。读取 offsetWidth/getBoundingClientRect 会强制浏览器同步刷新 Layout，紧接着的写又让它失效，下次读再次强制重排——读写交错反复触发重排，性能雪崩。
正例：先批量读，再批量写。把所有 DOM 读取放一起、所有写入放一起，让浏览器只在帧末统一重排一次。这就是 FastDOM 等库的核心思想。

排查工具

Chrome DevTools → Performance 面板录制可看到 Layout / Paint 的耗时火焰图；Rendering 面板的 "Paint flashing" 高亮重绘区域、"Layer borders" 显示合成图层。优化动画时先测量再优化，别凭感觉加 will-change。

与其他知识点的联系

  • animation（上一节）：本节的渲染管线原理正是上一节「关键帧里尽量只改 transform/opacity」这条建议的底层解释——只有理解了 Layout/Paint/Composite 三个阶段的成本差异，才知道为什么同样是动画，改 left 和改 transform 的流畅度天差地别。
  • filter：filter 一节提到「非 none 的 filter 会触发合成层」，这里的合成层就是本节讲的 Composite 阶段产物；will-change: filter 同样能提前为滤镜动画准备图层，用法和 will-change: transform 是同一套机制的不同属性。
  • border-radius / 背景属性：圆角、背景色等只触发 Paint（重绘）的属性，比触发 Layout 的属性开销小得多——这解释了为什么静态圆角随便用不心疼性能，但如果拿 border-radius 做逐帧动画，仍然不如 transform: scale() 配合遮罩省资源。
  • 文字溢出 / 字体排版：动态改变 font-size 或触发 -webkit-line-clamp 重新计算行数，都会引发 Layout 级别的重排——这是本节「越靠前的环节越贵」原则的具体案例，也说明了为什么文字排版相关的变化不适合做高频动画。
  • BEM / CSS 方法论：will-change 用完即弃的最佳实践，本质上是把「性能优化」当成一种临时状态而非静态样式来管理，这和 BEM 里用 Modifier 类表示「临时状态」（如 .card--animating）思路一致，都是用显式的状态标记控制样式的生命周期。
