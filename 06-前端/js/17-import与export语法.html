import / export 语法

一个文件想用另一个文件里的函数，靠的是模块。export 把东西"挂出去"，import 把东西"拿进来"。这套 ES Module 语法是现代前端工程的骨架，理解它的两种导出方式与"实时绑定"的本质，能避开一大批团队协作和构建优化上的坑。

它解决了什么问题

在模块系统出现之前，浏览器里多个 <script> 共享同一个全局作用域：A 文件声明的 config，B 文件能直接读甚至覆盖，命名冲突防不胜防——这就是全局污染。而且文件之间的依赖关系全靠 <script> 的书写顺序硬撑，少写一个或顺序错了就报 undefined。模块要解决三件事：① 每个文件是独立作用域，不主动 export 的东西外部看不见；② 依赖关系显式声明在代码里，由引擎自动按需加载；③ 把功能拆成小块，方便复用和单独测试。ES6（2015）把 ESModule 定为语言标准，从此 import / export 是关键字，而非函数调用。

命名导出（named export）

命名导出可以在一个模块里有任意多个，每个都带名字。两种写法等价：导出时直接加 export，或在文件末尾用 export { ... } 集中声明。后者通常更清晰——读者在文件底部就能看到"这个模块对外提供了什么"。

math.js —— 命名导出的两种写法
// 写法一：声明处直接 export
export const PI = 3.14159;
export function add(a, b) { return a + b; }

// 写法二：先正常声明，文件末尾集中导出（推荐，一眼看清出口）
const E = 2.71828;
function sub(a, b) { return a - b; }
export { E, sub };

对应的命名导入

命名导入必须用花括号，且名字要和导出的完全一致（这就是"命名"的含义）。本质上 import { add } 是一种结构化匹配，靠名字对接，不是靠位置。

app.js —— 命名导入与重命名
// 名字必须和导出一致，花括号不能省
import { PI, add } from './math.js';

// 用 as 重命名，解决本地命名冲突
import { add as sum } from './math.js';

// 用 * as 把整个模块收进一个命名空间对象
import * as math from './math.js';
console.log(math.add(1, 2), math.PI);  // 3 3.14159

默认导出（default export）

每个模块至多只能有一个默认导出，它代表这个模块"最主要的那个东西"——常见于一个文件就导出一个 React 组件、一个类、一个配置对象的场景。导入默认导出不用花括号，而且名字可以随便起，因为默认导出没有名字，导入方自己给它命名。

logger.js / 使用方
// logger.js：一个模块只暴露这一个主角
export default function log(msg) {
  console.log('[LOG] ' + msg);
}

// 导入方：无花括号，名字随意（这里叫 logger，叫 abc 也合法）
import logger from './logger.js';
logger('hello');

混合导入

一个模块可以同时有一个默认导出和多个命名导出。导入时把它们写在一条语句里，默认的在前（无花括号），命名的在后（花括号内）。

同时拿默认导出和命名导出
// React 就是典型：默认导出 React 本体 + 命名导出各种 Hook
import React, { useState, useEffect } from 'react';

再导出聚合（export ... from）

大型项目常建一个 index.js 做"统一出口"（barrel file）：把分散在多个文件里的导出集中转发出去，使用方只需从一个路径导入。export ... from 是 import 后立刻 export 的语法糖，但它不会把这些绑定引入当前模块作用域，纯粹是转发。

components/index.js —— 聚合出口
// 转发命名导出
export { Button } from './Button.js';
export { Input, Select } from './Form.js';

// 把另一个模块的所有命名导出一次性转发
export * from './icons.js';

// 使用方只认一个路径，内部怎么拆文件都不影响调用方
// import { Button, Input } from './components/index.js';

导入提升 + 只读实时绑定

ESModule 的 import 有两个底层特性，是它和"复制粘贴一段代码"的本质区别：

  • 提升（hoisting）：所有 import 在模块执行前就被解析并提到顶部。所以即使把 import 写在文件末尾，前面的代码也能用——但为可读性应始终写在顶部。
  • 只读的实时绑定（live binding）：导入的不是值的拷贝，而是指向原模块变量的引用。原模块里的值变了，导入方读到的也跟着变；同时导入方不能给它重新赋值。
实时绑定：值会变，但你不能改它
// counter.js
export let count = 0;
export function inc() { count++; }

// app.js
import { count, inc } from './counter.js';
console.log(count);  // 0
inc();
console.log(count);  // 1 —— 实时反映原模块的变化（这是绑定，不是拷贝）

count = 99;      // ❌ TypeError: Assignment to constant variable（导入是只读的）

在 HTML 中使用模块

浏览器原生支持 ESModule，但必须给 <script> 加上 type="module"。模块脚本默认延迟执行（defer）、运行在严格模式下，且有自己的顶层作用域（不污染 window）。

index.html
<script type="module" src="./app.js"></script>

<!-- 也可以内联 -->
<script type="module">
  import { add } from './math.js';
  console.log(add(2, 3));
</script>

重点对比：named vs default

什么时候用哪种，是日常最常纠结的问题。核心差异看这张表：

维度命名导出 named默认导出 default
每个模块数量任意多个至多 1 个
导入语法带花括号 { }不带花括号
导入名是否须同名必须同名（可用 as 改）可任意命名
导出语法export const/function 或 export { }export default ...
Tree-shaking友好，未用到的命名导出可被摇掉较弱，整体导出难以按需裁剪
适用场景工具函数集、多个相关 API模块只有一个主角（组件/类）
团队取舍：很多团队（如 Airbnb 风格、部分大厂规范）倾向于多用命名导出、慎用默认导出。原因正是下面要讲的命名一致性与 tree-shaking 友好度。

常见陷阱

  • 默认导出的导入名可任意 → 团队命名不一致。同一个组件，张三 import Btn、李四 import Button，全局搜索和重构都变难。命名导出靠强制同名天然避免了这个问题。
  • 浏览器原生 ESM 路径必须带扩展名。import x from './math' 在浏览器里会 404，必须写 './math.js'。Webpack/Vite 等打包器会帮你补全，所以本地写惯了省略扩展名，上线到原生 ESM 环境就翻车。
  • 导入的绑定是只读的，不能重新赋值。如上例 count = 99 直接报错。需要修改状态，应导出一个修改函数（如 inc()），而不是直接改导入变量。
  • 不能在条件语句里写静态 import。import 必须在模块顶层，因为它在编译期就要确定依赖图，不能放进 if / 函数里。要按条件加载请用动态 import()，它返回 Promise，可写在任意位置。
  • 混淆 export default 与 export { }。export default foo 导出的是"默认绑定"，导入不带花括号；export { foo } 是命名导出，导入必须带花括号。两者对应的导入写法不能互换。
静态 import vs 动态 import()
// ❌ 非法：import 不能放进 if
// if (cond) { import x from './a.js'; }

// ✅ 按需加载用动态 import()，返回 Promise
if (cond) {
  const mod = await import('./heavy.js');
  mod.run();  // 命名导出从返回对象上取；默认导出在 mod.default
}
易错点：动态 import() 拿到的对象里，命名导出是直接属性（mod.run），而默认导出挂在 mod.default 上——很多人忘了 .default 而拿到 undefined。
小结：① 模块给每个文件独立作用域，解决全局污染、依赖管理、复用三大问题；② 命名导出可多个、导入须同名带花括号、tree-shaking 友好；③ 默认导出至多一个、导入名任意不带花括号；④ 二者可混合，export ... from 做聚合出口；⑤ import 会提升、是只读的实时绑定（引用而非拷贝）；⑥ 浏览器原生 ESM 要 type="module" 且路径带扩展名，按条件加载用动态 import()。
