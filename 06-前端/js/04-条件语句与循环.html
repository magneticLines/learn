条件语句与循环

程序只会两件事：根据条件走不同的路（分支），把同一段逻辑重复执行（循环）。这两样是所有控制流的骨架——看懂它们的细节差异，才能避开穿透、遍历错对象这类经典坑。

它解决了什么问题

代码默认从上到下逐行执行。但真实世界充满"如果……就……""对每一个……都……"的逻辑：登录成功才跳转，购物车里每件商品都要算价。条件语句让程序能在运行时选择执行哪一段，循环让程序能重复执行而不必复制粘贴。JS 还为不同遍历场景提供了多套循环（for / for...of / for...in），各有适用对象——用错了不是报错，而是悄悄给出错误结果，这正是初学者踩坑最多的地方。

if / else if / else

最基础的分支。条件表达式会被转成布尔值（真值/假值判断），假值有 false、0、""、null、undefined、NaN 这几个，其余都是真值。分支从上往下匹配，命中一个就跳出整个结构。

多分支：注意顺序从上到下匹配
const score = 82;
if (score >= 90) {
  console.log("优秀");
} else if (score >= 60) {
  console.log("及格");  // 命中这里，后面不再判断
} else {
  console.log("不及格");
}
陷阱：分支区间要"从严到宽"排列。若把 score >= 60 写在 score >= 90 前面，90 分也会先命中"及格"——因为 90 同样满足 >= 60，而 if/else 命中即止。

switch：等值多分支

当你拿同一个值去比对多个固定取值时，switch 比一长串 else if 更清晰。它用 === 严格相等来匹配 case（不做类型转换），命中后从该 case 开始一直执行到遇见 break 或结尾——这就是"穿透"。

反例：漏写 break 导致穿透
const day = 1;
switch (day) {
  case 1:
    console.log("周一");   // 没写 break
  case 2:
    console.log("周二");   // 会一路穿透到这里！输出 周一 周二
    break;
  default:
    console.log("其他");
}
正例：每个 case 收尾加 break；故意穿透可合并 case
switch (day) {
  case 6:
  case 7:
    console.log("周末");   // 6 和 7 共用逻辑：有意利用穿透
    break;
  default:
    console.log("工作日");  // default 不必放最后，但习惯放最后
}
对比 if/else：switch 只能做等值（===）匹配，适合"枚举值分发"；区间判断（>= 90）、复杂布尔条件还得用 if/else。另外 case 用的是严格相等，case "1" 匹配不了数字 1。

三种基础循环：for / while / do-while

三者能力等价，区别在什么时候判断条件、循环次数是否已知。

循环判断时机典型场景
for先判断，初始化/条件/步进集中在头部次数已知，如遍历索引
while先判断，再执行（可能一次都不执行）次数未知，靠条件停
do-while先执行一次，再判断至少执行一次，如菜单循环
三种写法对照
for (let i = 0; i < 3; i++) {
  console.log(i);           // 0 1 2
}

let n = 3;
while (n > 0) {      // 条件为假就一次都不进
  console.log(n--);
}

let m = 0;
do {
  console.log("至少跑一次");  // 即使 m 不满足条件也已执行一次
} while (m > 0);

for...of vs for...in：遍历的两种姿势

这是最容易混淆、也最爱考的一对。一句话记牢：for...of 遍历可迭代对象的"值"，for...in 遍历对象的"可枚举键名"（且会顺着原型链往上找）。

维度for...offor...in
遍历内容元素值键名（字符串）
适用对象数组、字符串、Map、Set 等可迭代对象普通对象
是否含原型链属性否是（含继承的可枚举属性）
遍历数组✅ 推荐，拿到的是值⚠️ 拿到字符串索引，且可能混入额外属性
顺序保证按迭代器顺序（数组即下标序）不保证严格顺序
for...of 拿值，for...in 拿键
const arr = ["a", "b", "c"];

for (const val of arr) {
  console.log(val);          // "a" "b" "c"  ——拿到值
}

for (const key in arr) {
  console.log(key);          // "0" "1" "2" ——是字符串索引，不是数字
}
反例：用 for...in 遍历数组，原型属性混了进来
const list = [10, 20];
Array.prototype.extra = "脏数据";  // 某个库往原型上挂了东西

for (const k in list) {
  console.log(k);            // "0" "1" "extra" ——继承属性也被遍历到了！
}

// 正例：遍历数组用 for...of（拿值）或 forEach，干净且语义清晰
for (const v of list) console.log(v);  // 10 20
遍历对象的正确姿势：想遍历普通对象自身的键，更推荐 Object.keys(obj) / Object.entries(obj) 配合 for...of，它们只返回对象自身的可枚举属性，天然避开原型链；若坚持用 for...in，记得加 obj.hasOwnProperty(key) 过滤。

break、continue 与 label

break 跳出整个循环，continue 跳过本次迭代、进入下一轮。默认它们只作用于最近的一层循环。想一次跳出多层嵌套循环时，可以用 label（标签）给外层循环命名，再 break 标签名。

continue 跳过、break 跳出、label 跳出多层
for (let i = 0; i < 5; i++) {
  if (i === 2) continue;   // 跳过 2，继续下一轮
  if (i === 4) break;      // 到 4 直接结束循环
  console.log(i);             // 0 1 3
}

outer: for (let x = 0; x < 3; x++) {
  for (let y = 0; y < 3; y++) {
    if (x + y === 2) break outer;  // 一次跳出两层循环
    console.log(x, y);
  }
}
label 慎用：带标签的跳转很像 goto，多了会让控制流难追。多数情况下，把内层循环抽成一个函数、用 return 提前退出，比 label 更易读。

陷阱：循环里 let 还是 var（呼应闭包）

当循环体里有异步回调或闭包引用循环变量时，var 和 let 的差别会暴露无遗。var 在整个循环只有一个变量，所有回调共享它；let 在每轮迭代都新建一个绑定，回调各自捕获自己的副本。

反例 → 正例：循环变量被闭包共享
// 反例：var 只有一个 i，循环结束时它已是 3
for (var i = 0; i < 3; i++)
  setTimeout(() => console.log(i));   // 3 3 3

// 正例：let 每轮独立绑定，闭包捕获各自的副本
for (let j = 0; j < 3; j++)
  setTimeout(() => console.log(j));   // 0 1 2
结论：循环计数器一律用 let。这也呼应了"变量与数据类型"一章里 let 块级作用域的设计——ES6 正是为了根治这类闭包坑才引入它。
小结：① if/else 分支从上到下命中即止，区间判断要从严到宽；② switch 用 === 等值匹配，每个 case 记得 break，故意穿透可合并 case；③ for 次数已知、while 先判断、do-while 至少一次；④ 遍历数组首选 for...of 或数组方法，for...in 拿的是键名且含原型链、专给对象用；⑤ 循环计数器一律 let，避开闭包共享坑。

与其他知识点的联系

  • 类型转换与判断：if 的条件判断本质是调用了那一篇讲的 Boolean() 隐式转换，falsy 六个值决定了分支走向；switch 用 === 严格相等匹配 case，不做类型转换，这也是它比 == 更安全、更常被推荐的原因之一。
  • 作用域与闭包：本篇"循环 + let vs var"这个陷阱只是入口，下一篇会从"块级作用域每轮迭代新建绑定"的原理层面彻底讲清楚为什么 let 能让每个 setTimeout 闭包捕获到独立的循环变量——这里先看到现象，下一篇才理解本质。
  • 数组与常用方法：for...of/for...in 是手写遍历的通用工具，而 map/filter/forEach 是专为数组场景设计的声明式替代品——理解了本篇"遍历数组该用什么循环"的取舍，才能明白为什么实战中更推荐数组方法而不是裸写 for 循环。
  • React Hooks 闭包陷阱：本篇讲的 var/let 在循环中被闭包捕获的问题，和 useEffect/useCallback 里"依赖数组缺项导致拿到过期变量"是同一类 bug 的不同表现形式——都是"函数捕获的是变量引用还是当时的值"这个核心问题。
  • TypeScript 类型系统：switch 配合 TS 的联合类型能做到"穷尽性检查"（exhaustiveness check）：如果 case 没有覆盖所有可能的类型分支，TS 会在编译期报错，这是本篇 switch 穿透陷阱之外，TS 给等值分支语句带来的额外安全网。
