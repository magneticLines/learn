正则表达式基础

正则表达式是一种用"模式"来描述字符串的微型语言。当你需要校验输入、查找内容、批量替换或从文本里提取片段时，它能用一行表达式干掉几十行 if/indexOf 的活儿——前提是你得读得懂这门"天书"。

它解决了什么问题

设想你要判断"这个字符串是不是合法手机号"。不用正则，你得逐个字符检查长度、是不是数字、开头是不是 1……写一大坨循环。正则把这件事浓缩成一个模式：/^1\d{10}$/——"以 1 开头，后面跟 10 个数字，到此结束"。它的本质是用声明式的方式描述字符串的结构特征，把"怎么一步步检查"交给引擎，你只负责说清"长什么样"。校验、查找、替换、提取这四类文本任务，正则都是首选工具。

创建正则：字面量 vs 构造函数

两种写法，引擎最终生成的是同一种 RegExp 对象，区别在于模式是写死的还是动态拼出来的。

两种创建方式
// ① 字面量：模式写死在斜杠之间，引擎在编译阶段就解析，性能最佳
const re1 = /\d+/g;          // 斜杠后面的 g 是 flag

// ② 构造函数：模式是字符串，适合运行时动态拼接
const key = "abc";
const re2 = new RegExp(key + "\\d+", "g"); // 注意 \\d 双重转义
字面量是"常量"，反复执行同一个 /.../ 字面量，引擎可复用同一对象；构造函数适合模式来自变量（如用户输入的搜索词）的场景。能用字面量就用字面量。

常用元字符与语法

正则的"字母表"分成几类：匹配某类字符的字符类、控制次数的量词、卡位置的边界、组合用的分组与字符集。下表是高频核心。

分类语法含义
字符类\d / \D数字 [0-9] / 非数字
字符类\w / \W单词字符 [A-Za-z0-9_] / 非单词字符
字符类\s / \S空白符（空格/制表/换行）/ 非空白
字符类.除换行外的任意单个字符
量词*0 次或多次
量词+1 次或多次
量词?0 次或 1 次（可选）
量词{n} / {n,} / {n,m}恰好 n 次 / 至少 n 次 / n 到 m 次
边界^ / $字符串（或行）开头 / 结尾
边界\b单词边界（单词与非单词之间）
分组(...)分组并捕获
择一a|b匹配 a 或 b
字符集[abc] / [a-z] / [^...]集合内任一 / 范围 / 取反（非集合内）
转义\让元字符变回普通字符，如 \. 匹配真正的点

标志（flags）：模式的开关

flag 写在字面量末尾的斜杠后（如 /.../gi），或作为 RegExp 的第二个参数。它们调整匹配的全局行为。

flag名称作用
gglobal 全局找出所有匹配，而非第一个就停
iignoreCase忽略大小写
mmultiline 多行让 ^ $ 匹配每一行的首尾，而非整串首尾
sdotAll让 . 也能匹配换行符
uunicode正确处理 emoji 等 4 字节字符与 \u{...}

匹配方法：正则的 vs 字符串的

API 分两边：正则对象上挂 test / exec，字符串对象上挂 match / matchAll / replace / split / search。常见的选择是——只想知道"匹配不匹配"用 test；想拿到所有结果用 matchAll；要改内容用 replace。

核心方法速览
/\d+/.test("a1b");             // true —— 只返回布尔，最轻量

// match：不带 g 返回首个匹配+捕获组信息；带 g 只返回所有匹配文本数组
"a1b22".match(/\d+/g);       // ["1", "22"]

// matchAll：返回迭代器，每项含捕获组、index，处理多匹配的首选
[..."a1b22".matchAll(/\d+/g)];

// replace：$1 引用第 1 个捕获组，也可传回调函数动态计算替换值
"2024-06".replace(/(\d+)-(\d+)/, "$2/$1"); // "06/2024"
"a1b2".replace(/\d/g, d => d * 2);  // "a2b4" 回调

"a, b ,c".split(/\s*,\s*/);     // ["a","b","c"] 按模式切分
"a1b".search(/\d/);            // 1 —— 返回首个匹配的下标

分组、命名捕获与懒惰匹配

(...) 既分组又捕获（可在结果或 $1 中取出）；(?:...) 是非捕获组，只分组不记账，性能更好；(?<name>...) 是命名捕获组，用名字取值比记 $1$2 可读得多。量词默认贪婪（尽量多吃），在量词后加 ? 变懒惰（尽量少吃）。

分组与贪婪/懒惰
// 命名捕获组：从日期里取出年月日，用语义名取值
const m = "2024-06-23".match(/(?<y>\d{4})-(?<mo>\d{2})-(?<d>\d{2})/);
m.groups.y;   // "2024"

// 贪婪：.* 会一路吃到最后一个引号，结果是整段
'<a><b>'.match(/<.*>/)[0];   // "<a><b>"
// 懒惰：.*? 吃到第一个 > 就收手
'<a><b>'.match(/<.*?>/)[0];  // "<a>"

// 前瞻 (?=...)：匹配"后面跟着 元 的数字"，但不消费 元 本身
"售价99元".match(/\d+(?=元)/)[0];  // "99"

三个实战例子

校验 / 提取 / 替换
// ① 手机号粗校验：1 开头 + 第二位 3-9 + 共 11 位
const isPhone = /^1[3-9]\d{9}$/.test("13800138000"); // true

// ② 邮箱粗校验（仅演示，真实校验远比这复杂）
const isEmail = /^[\w.-]+@[\w-]+\.[\w.]+$/.test("a@b.com");

// ③ 提取所有数字 + 千分位替换
"价1200和3400".match(/\d+/g);     // ["1200","3400"]
"1234567".replace(/\B(?=(\d{3})+$)/g, ","); // "1,234,567"

常见陷阱

陷阱一（高频考点）：带 g 的正则有 lastIndex 状态。同一个 g 正则对象重复调用 test/exec 时，引擎会从上次匹配结束的位置继续，导致结果"跳变"。
反例：g 正则的 lastIndex 副作用
const re = /\d/g;
re.test("a1");   // true，匹配后 lastIndex 变成 2
re.test("a1");   // false！从下标 2 开始找，已无字符 —— 同输入结果却不同

// 正例：校验场景别加 g；或每次用新字面量；或手动重置 lastIndex
/\d/.test("a1");   // 永远 true
陷阱二：贪婪吃太多。默认量词会尽可能多匹配，提取 HTML 标签、引号内容时常常一口吞掉整段。解决办法是在量词后加 ? 改用懒惰匹配（见上文 .*? 例子）。
陷阱三：构造函数里要双重转义。字符串中 \ 本身是转义符，所以正则的 \d 在 new RegExp("\\d") 里必须写成 \\d，否则 \d 会先被字符串解析掉。
陷阱四：可读性与灾难性回溯（ReDoS）。复杂正则像天书，半年后自己都看不懂——拆分或加注释。更危险的是嵌套量词（如 (a+)+$）遇到特定输入会引发指数级回溯，让 CPU 卡死，成为安全漏洞。对用户输入跑的正则务必警惕。
陷阱五：别用正则解析 HTML。HTML 是嵌套结构，正则是"正则语言"，理论上无法可靠匹配任意嵌套——解析 HTML/XML 请用 DOMParser 等专门的解析器，正则只适合做简单的局部提取。
小结：① 正则是描述字符串模式的微型语言，擅长校验/查找/替换/提取；② 字面量优先，动态拼接才用 new RegExp（记得双重转义）；③ 牢记字符类 \d\w\s、量词 *+?{}、边界 ^$\b 和 flag gims；④ 用 matchAll 取所有匹配，replace 配 $1 或回调改内容；⑤ 警惕 g 正则的 lastIndex 状态、贪婪匹配、ReDoS，且不要拿正则解析 HTML。
