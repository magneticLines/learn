模块系统概览（ESM / CJS）

同样是 import 和 require，背后却是两套设计哲学完全不同的模块系统。搞懂它们的来龙去脉，才能理解为什么有时候 require 不能用、为什么打包工具能做 tree-shaking、为什么混用会报奇怪的错。

它解决了什么问题

最早的 JS 根本没有模块。所有脚本都往同一个全局作用域里塞变量，多个 <script> 之间靠全局变量通信。这带来两个致命问题：命名冲突（两个库都定义了 $）和依赖关系不明（你必须手动保证 <script> 的加载顺序正确，否则后面的脚本拿不到前面的变量）。模块系统要解决的核心问题就是：每个文件有自己独立的作用域，并能明确声明"我依赖谁、我导出什么"。这样依赖关系变成了代码里可分析的结构，而不是隐藏在加载顺序里的潜规则。

模块化演进简史

理解今天的 ESM 与 CJS，要顺着这条历史线看——每一步都是在补前一步的坑。

  • 无模块时代：靠全局变量 + IIFE（立即执行函数）造"命名空间"，把内部实现藏进闭包，只往全局挂一个对象，缓解冲突。
  • CommonJS（2009，为 Node 而生）：用 require() 同步加载、module.exports 导出。同步读文件在服务端没问题（本地磁盘快），但搬不到浏览器（网络请求不能同步阻塞）。
  • AMD（RequireJS）：为浏览器设计的异步方案，用回调声明依赖。语法繁琐，是过渡产物，今天基本退场。
  • ES Modules（ES6 / 2015）：终于写进语言标准，import / export 同时被浏览器和 Node 原生支持，是当下唯一推荐的统一方案。
为什么是"语言级"很关键？CJS 的 require 本质只是 Node 实现的一个普通函数；而 ESM 的 import 是语法关键字，引擎在解析阶段就能看懂模块结构，不必先运行代码。这个差别是后面一切对比的根源。

语法对照

CommonJS：运行时的函数调用
// math.cjs —— 导出就是给一个普通对象赋值
function add(a, b) { return a + b; }
module.exports = { add };

// main.cjs —— require 是函数，可以写在任何地方
const { add } = require('./math.cjs');
console.log(add(1, 2));  // 3
ESM：编译期就确定的静态声明
// math.mjs
export function add(a, b) { return a + b; }

// main.mjs —— import 必须在顶层，引擎解析时就读取这些声明
import { add } from './math.mjs';
console.log(add(1, 2));  // 3

陷阱一：CJS 是值拷贝，ESM 是实时绑定

这是两套系统最本质、也最容易出 bug 的差异。CJS 的 require 拿到的是导出那一刻的值快照；之后源模块内部即使改了变量，导入方也看不到。ESM 的 import 拿到的是一个只读的实时绑定，源模块改了，导入方读到的就是新值。

反例：CJS 拿到的是旧值
// counter.cjs
let count = 0;
setTimeout(() => { count++; }, 100);
module.exports = { count };  // 此刻拷贝了 0

// main.cjs：100ms 后读到的仍是 0（拷贝的快照不会变）
const c = require('./counter.cjs');
setTimeout(() => console.log(c.count), 200);  // 0
正例：ESM 实时反映源模块的变化
// counter.mjs
export let count = 0;
setTimeout(() => { count++; }, 100);

// main.mjs：200ms 后读到 1（import 是活的绑定，但你不能给它赋值）
import { count } from './counter.mjs';
setTimeout(() => console.log(count), 200);  // 1

陷阱二：循环依赖的不同表现

A 引 B、B 又引 A，两套系统都不会死循环（都靠"模块缓存"打破），但拿到的中间状态不同。CJS 遇到循环时，返回的是被引模块当前已执行到的那部分 exports——可能是个残缺的对象，访问还没赋值的属性会得到 undefined。ESM 因为是实时绑定，只要你在真正用到变量时它已被初始化，就能拿到正确的值；引擎对未初始化绑定的访问会直接抛 ReferenceError（暴露得更早）。

实践建议：无论哪套系统，循环依赖都是设计气味。最好的修法不是研究它的求值顺序，而是把共享的部分抽到第三个模块，让 A、B 都依赖它，从而打破环。

ESM 的静态结构与 tree-shaking

为什么 ESM 能"摇树"（删掉没用到的导出），CJS 不能？因为 ESM 的 import / export 必须写在顶层、名字写死、不能放进 if 里。于是打包工具不运行代码、只静态分析就能画出完整的依赖图，判断哪个导出从没被引用，安全地删掉。CJS 的 require 是运行时函数，参数甚至可以是变量拼出来的字符串（require('./' + name)），工具无法在编译期确定到底加载了什么，自然不敢删。

静态结构让 tree-shaking 成为可能
// utils.mjs 导出两个函数
export function used() { /* ... */ }
export function unused() { /* ... */ }

// app.mjs 只用 used，打包后 unused 会被剔除
import { used } from './utils.mjs';
used();

动态 import()：按需加载与代码分割

静态 import 不能写在条件里，但有时确实需要"用到才加载"（比如点击按钮才打开的弹窗组件）。import() 是个函数形式的动态导入，返回一个 Promise，解析出模块对象。打包工具看到它会自动把这个模块切成单独的 chunk，实现首屏不加载、按需拉取——这就是路由懒加载的底层机制。

动态导入实现按需加载
button.addEventListener('click', async () => {
  // 点击时才真正发请求加载这个模块
  const { openDialog } = await import('./dialog.mjs');
  openDialog();
});

Node 里如何区分两套系统

Node 同时支持两套，靠文件后缀和 package.json 判断：

  • .mjs 文件永远当 ESM；.cjs 文件永远当 CJS。
  • 普通 .js 文件看最近的 package.json：有 "type": "module" 就按 ESM 解析，否则（或没写）按 CJS。
package.json 决定 .js 的归属
{
  "name": "my-app",
  "type": "module"  // 此后 .js 都按 ESM 解析
}
互操作陷阱：ESM 里可以 import 一个 CJS 模块（Node 会把它的 module.exports 包装成 default 导出）；但反过来在 CJS 里不能直接 require 一个 ESM 模块（require 是同步的，ESM 是异步的，对不上），只能用动态 import()。混用时这是最常见的"为什么 require 报错"的原因。

浏览器原生 ESM

现代浏览器直接支持 ESM，只要给 <script> 加上 type="module"。它和普通脚本有几个关键区别：

  • 默认 defer：不阻塞 HTML 解析，等文档解析完按顺序执行。
  • 自带模块作用域：顶层声明不再污染全局 window。
  • 默认严格模式：不用写 'use strict'。
  • 同一模块只会被下载并执行一次，多处引用共享同一实例。
浏览器原生模块脚本
<script type="module">
  import { add } from './math.mjs';
  console.log(add(1, 2));
</script>

陷阱三：顶层 await 与打包工具的转换

ESM 因为是异步加载的，允许在模块顶层直接写 await（不必包在 async 函数里）——CJS 做不到这点。但要注意：顶层 await 会让依赖它的模块也"等"它执行完，用不好会拖慢整个加载链。

另外，你日常写 import 但项目最终跑的可能并不是原生 ESM。Webpack / Vite 这类打包工具会在中间做转换：开发期 Vite 利用浏览器原生 ESM 实现秒级热更新，生产构建时则把成百上千个模块打包、做 tree-shaking、必要时降级成兼容旧浏览器的格式。所以"我写的 ESM"和"浏览器最终执行的代码"往往不是一回事，理解这层转换能帮你看懂构建产物和很多诡异报错。

小结：① 模块化是为隔离作用域 + 明确依赖而生；演进是 无模块 → CJS → AMD → ESM；② ESM 是编译期静态、实时绑定、可 tree-shaking，CJS 是运行时动态、值拷贝、可条件加载；③ import() 返回 Promise，是懒加载/代码分割的基础；④ Node 靠 .mjs/.cjs 与 "type": "module" 区分，CJS 不能直接 require ESM；⑤ 浏览器 <script type="module"> 默认 defer + 严格模式 + 独立作用域；⑥ 你写的 import 经 Webpack/Vite 转换后才真正运行。

与其他知识点的联系

  • import/export 语法（上一篇）：上一篇专注"怎么写"ESM，本篇把它放进"ESM vs CJS"的历史与设计对比里，解释了上一篇提到的"实时绑定"到底是相对于什么而言的——正是相对于 CJS 的值拷贝语义。两篇顺序反过来读，会发现很多"为什么"缺了参照物。
  • Node.js 生态与 require：本篇是理解 Node.js 后端代码（大量遗留 CommonJS 项目）与现代前端代码（ESM）之间"看起来都是 import 却不完全一样"这种割裂感的关键。搞懂 .mjs/.cjs/package.json 的 type 字段，才能看懂各种 Node 报错（如 "require() of ES Module ... not supported"）。
  • DOM 选取元素（下一篇）：模块系统关心的是"代码怎么组织与加载"，而从下一篇开始的 DOM 操作系列关心的是"运行时怎么操作页面"——二者看似主题跳跃，实际上是前端开发的两条主线：一条管代码结构，一条管页面交互，本篇正好是承上启下的分界点。
  • 为什么 React 不用原生 DOM 操作：ESM 的静态结构支撑了 tree-shaking，这也是 React 生态能把体积做小的原因之一——按需 import 单个组件而不是整个库。同时，路由懒加载常用的动态 import() 正是 React 生态里 React.lazy(() => import('./Page')) 的底层机制，这条线直接连到了组件化框架的工程实践。
  • 打包工具与生产部署：本篇末尾强调"你写的 import 经 Webpack/Vite 转换后才真正运行"，这解释了为什么本项目部署到 GitHub Pages 这类纯静态托管时，浏览器原生 ESM（不经打包）也能跑——只要路径带扩展名、用 type="module"，配置正确的原生 ESM 完全可以省去打包步骤，这也是很多轻量学习项目、Demo 页面的常见选择。
