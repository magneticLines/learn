运算符与表达式

运算符是把数据"算出结果"的最小语法单元。看似都是中学就学过的加减乘除，但 JS 的逻辑运算符会"返回操作数而不是布尔值"，自增有前后置之分，浮点数还会算错——这些细节正是写出可预测代码、也是面试官最爱追问的地方。

它解决了什么问题

程序的本质是对数据做变换：把两个数加起来、判断谁大谁小、根据条件选一个值。运算符就是描述这些变换的"动词"，运算符连同操作数组成的、能算出一个值的代码片段就叫表达式（expression）。理解运算符的关键不在于记住符号，而在于搞清楚三件事：它接收什么类型、返回什么类型、和别的运算符放一起时谁先算。JS 在这三点上都有不少"反直觉"的设计，本节就把它们逐个讲透。

算术运算符

基本的 + - * / 不必多说，重点是三个容易被忽略的：

幂、取余与除法
console.log(2 ** 10);   // 1024，** 是幂运算（ES2016），等价于 Math.pow(2, 10)
console.log(7 % 3);     // 1，% 是取余（余数符号跟随被除数）
console.log(-7 % 3);    // -1（注意！不是 2，这点和数学的"模"不同）
console.log(5 / 2);     // 2.5，JS 没有整数除法，结果一律是浮点
陷阱：+ 的双重身份。只要有一个操作数是字符串，+ 就变成字符串拼接而非加法：1 + "2" 得到 "12"，但 1 - "2" 却是 -1（- 没有拼接含义，会把 "2" 转成数字）。这是类型转换那一节的经典坑，写代码时对用户输入务必先显式 Number()。

自增 / 自减：前置与后置的区别

i++ 和 ++i 都会让 i 加一，差别在于整个表达式的返回值：后置 i++ 返回加之前的旧值，前置 ++i 返回加之后的新值。单独写一行时没区别，一旦把它嵌进别的表达式里就会出分歧。

反例 → 问题 → 正例
let i = 5;
console.log(i++);  // 5（先返回旧值，i 之后才变 6）
console.log(i);    // 6

let j = 5;
console.log(++j);  // 6（先自增，再返回新值）

// 反例：把后置自增塞进表达式，结果常常算错
let n = 1;
let sum = n++ + n;  // 1 + 2 = 3，可读性极差，容易看错
// 正例：自增单独成行，意图一目了然
n++;
const total = n + n;

比较运算符

比较运算符返回布尔值。核心建议只有一条：永远用 === / !==（严格相等），不要用 == / !=（宽松相等）。== 会在比较前做隐式类型转换，规则繁琐且反直觉，是无数 bug 的温床。

表达式结果说明
1 === 1true类型与值都相同
1 === "1"false严格相等不转换类型
1 == "1"true ⚠️宽松相等把字符串转成数字
0 == falsetrue ⚠️false 被转成 0
null == undefinedtrue== 下二者相等（=== 则为 false）
NaN === NaNfalse ⚠️NaN 不等于任何值，判断要用 Number.isNaN()
对比记忆：=== 像 Java 比较基本类型，类型不同直接 false；== 则会"先想办法把两边凑成同类型再比"。后者唯一可接受的用法是 x == null——它能一次性判断 null 和 undefined，是社区默许的简写。

逻辑运算符与短路求值（重点）

这是本节最容易被误解、也最高频的考点。很多人以为 && 和 || 返回 true / false，错。它们返回的是某一个操作数本身，并且采用短路求值：能提前确定结果时，后面的操作数根本不会执行。

  • A && B：A 为假值就直接返回 A（不看 B）；否则返回 B。"全真才真，遇假即停"。
  • A || B：A 为真值就直接返回 A（不看 B）；否则返回 B。"遇真即停，全假才返回末尾"。
  • !A：这个才真正返回布尔值，把 A 转成布尔后取反。
什么是"假值"（falsy）？JS 中只有 8 个：false、0、-0、0n、""、null、undefined、NaN。除此之外（包括 "0"、[]、{}）全是真值。短路求值的"真假"判断，靠的就是这套规则。
短路返回的是操作数，不是布尔
console.log(0 && "hi");     // 0   （左边是假值，直接返回 0，不看右边）
console.log(3 && "hi");     // "hi"（左边为真，返回右边操作数）
console.log("" || "默认");    // "默认"（左边假值，返回右边）
console.log("已有" || "默认");  // "已有"（左边真值，短路，右边不执行）

实战用法一：&& 做"安全访问"

因为短路，a && a.b 能在 a 为 null / undefined 时跳过 .b，避免 Cannot read property of null 报错。这是可选链 ?. 出现前的标准写法。

用短路做防护
const user = null;
// 反例：直接访问会抛错
// console.log(user.name);  // TypeError: Cannot read properties of null

// 正例（旧写法）：user 为假值时短路，整体返回 null，不会去读 .name
console.log(user && user.name);  // null

// 现代写法：可选链 ?.，语义更清晰，结果是 undefined
console.log(user?.name);     // undefined

实战用法二：|| 给默认值——以及它的坑

x || 默认值 是给变量兜底的经典写法。但它有个致命问题：它对"假值"一视同仁，会把合法的 0、""、false 也当成"没值"而替换掉。这正是 ??（空值合并，ES2020）要解决的痛点。

反例 → 问题 → 正例：|| vs ??
function setVolume(v) {
  // 反例：用户传 0（静音）是合法的，却被 || 当成假值替换成 50
  const vol = v || 50;
  return vol;
}
console.log(setVolume(0));   // 50 ❌  本应是 0！

function setVolume2(v) {
  // 正例：?? 只在左边是 null / undefined 时才用默认值，0 / "" / false 会保留
  const vol = v ?? 50;
  return vol;
}
console.log(setVolume2(0));      // 0 ✅
console.log(setVolume2(undefined)); // 50 ✅  真正没传时才兜底
选用口诀："只要不是 null/undefined 就保留" → 用 ??；"任何假值都当没值" → 用 ||。绝大多数给默认值的场景，?? 才是你想要的。另注意：?? 不能和 &&/|| 直接混写，必须加括号，否则语法报错。

三元运算符 ?:

条件 ? 值A : 值B 是 JS 里唯一接收三个操作数的运算符，本质是 if/else 的表达式版——它能算出一个值，可以直接赋值或嵌进 JSX。简单分支用它很优雅，但不要嵌套多层，否则可读性崩溃，该用 if/else 就老实用。

三元：能返回值的分支
const age = 19;
const label = age >= 18 ? "成年" : "未成年";  // "成年"

// 反例：多层嵌套，几乎没人能一眼读懂
const grade = s >= 90 ? "A" : s >= 80 ? "B" : s >= 60 ? "C" : "D";

赋值与复合赋值

= 是赋值。复合赋值如 x += 1 是 x = x + 1 的简写，所有算术/位运算符都有对应版本：+= -= *= /= %= **= 等。ES2021 还补上了三个逻辑赋值运算符，它们同样遵循短路：只在需要时才赋值。

写法等价于赋值时机
x &&= yx && (x = y)x 为真值时才赋 y
x ||= yx || (x = y)x 为假值时才赋 y
x ??= yx ?? (x = y)x 为 null/undefined 时才赋 y
??= 优雅地补默认值
const config = { timeout: 0 };
config.timeout ??= 3000;  // timeout 是 0（非 null/undefined），保留不变 → 0
config.retries ??= 3;     // retries 原本不存在（undefined），赋值 → 3
console.log(config);           // { timeout: 0, retries: 3 }

位运算简述

位运算（& 与、| 或、^ 异或、~ 取反、<< 左移、>> 右移）直接操作数字的二进制位。日常前端用得少，但在权限标志位、性能敏感的算法、颜色计算里偶尔出现。需要知道的一个"小技巧反例"：

位运算的常见把戏（了解即可）
console.log(5 & 1);    // 1，用 n & 1 判断奇偶（奇数末位为 1）
console.log(5 ^ 5);    // 0，相同数异或为 0
console.log(~~3.7);    // 3，~~ 曾被用来取整，但可读性差，现在请直接用 Math.trunc()
陷阱：位运算会先把操作数转成 32 位有符号整数，超过约 21 亿的数会溢出出错。所以别用位运算处理大数，取整也优先用语义明确的 Math.floor / Math.trunc，而不是 ~~ 或 | 0 这类"炫技"写法。

运算符优先级

一行里有多个运算符时，谁先算由优先级决定（同级再看结合性，多数从左到右）。下表只列最常用的几条，从高到低：

优先级运算符例子
高() 分组、. 成员、?.最先求值
↑! ++ -- 一元!a
│** 幂（右结合）2 ** 3 ** 2 是 512
│* / %先乘除
│+ -后加减
│< > <= >=关系比较
│=== !== == !=相等比较
│&& 高于 ||a || b && c 即 a || (b && c)
│?: 三元很靠后
低= += 等赋值（右结合）a = b = 1
别死记，用括号。能记住"乘除先于加减、&& 先于 ||、比较先于逻辑"就够了。任何让你需要停下来想"这俩谁先算"的表达式，直接加 () 把意图写明白——这比依赖记忆更可靠，也对 review 你代码的同事更友好。

陷阱：浮点数运算不精确

这是所有用 IEEE 754 双精度浮点的语言（包括 Java 的 double）的通病，不是 JS 独有。0.1 和 0.2 在二进制里都是无限循环小数，存进有限位的浮点数时被截断，相加就产生了微小误差。

经典面试题：0.1 + 0.2
console.log(0.1 + 0.2);            // 0.30000000000000004，不是 0.3！
console.log(0.1 + 0.2 === 0.3);  // false ❌

// 正例 1：比较时允许一个极小误差（机器精度）
const eq = Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON;
console.log(eq);                     // true ✅

// 正例 2：金额等场景，用整数（分）运算，或用 toFixed 控制展示
console.log((0.1 + 0.2).toFixed(2));  // "0.30"（字符串，仅用于显示）
联系底层：这和 Java 里 System.out.println(0.1 + 0.2) 同样输出 0.30000000000000004 是一回事。涉及钱的计算，Java 用 BigDecimal，JS 则常用"以分为单位的整数运算"或 decimal.js 之类的库——根源都是浮点数无法精确表示十进制小数。
小结：① ** 幂、% 取余（符号随被除数）、JS 除法永远是浮点；② 自增后置返回旧值、前置返回新值，嵌进表达式易错，建议单独成行；③ 比较只用 ===；④ 核心：&&/|| 短路且返回操作数而非布尔，a && a.b 做防护、x || 默认 给兜底；⑤ || 会误杀合法的 0/""，给默认值优先 ??（含 ??=）；⑥ 优先级别死记，拿不准就加括号；⑦ 0.1 + 0.2 !== 0.3 是浮点通病，比较用 Number.EPSILON，金额用整数。
