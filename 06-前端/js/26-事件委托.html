事件委托

与其给一百个子元素各绑一个监听器，不如把监听器交给它们共同的父亲——这就是事件委托。它是事件冒泡最优雅的实战应用，也是高性能、动态列表的标配套路。

它解决了什么问题

想象一个有 1000 个 <li> 的列表，你想给每个 <li> 加点击事件。最直觉的写法是循环遍历、逐个 addEventListener。这会带来两个真实痛点：① 性能与内存——1000 个监听器意味着浏览器要维护 1000 份回调引用，绑定本身也要遍历 1000 次 DOM；② 动态元素失效——如果之后用 JS 又往列表里 append 了新的 <li>，这些新元素根本没被绑过，点了没反应，你必须记得每次新增后都重新绑定。

事件委托借助上一篇讲的事件冒泡：子元素上触发的事件会一路向上冒泡到父元素。所以我们只在父元素上绑一个监听器，等事件冒泡上来，再通过 event.target 判断"到底是哪个子元素被点了"。一个监听管一片，新增的子元素天然也在父元素之下，冒泡照样能捕获——两个问题一并解决。

核心机制：target 与 currentTarget

理解委托的关键是分清事件对象上的两个属性，它们在冒泡过程中含义完全不同：

属性含义在委托中的角色
event.target真正触发事件的最深层元素（用户实际点到的那个）用它来识别"点了哪个子元素"
event.currentTarget当前正在执行回调的元素，即绑监听器的那个父元素始终等于父元素，也等于 this
一句话区分：currentTarget 是"监听器贴在谁身上"（固定不变），target 是"事件源头在哪"（随每次点击变化）。委托的全部逻辑就是从 target 反推出我们关心的那个子元素。

反例：给每个 li 单独绑

能用，但每个元素一份监听器，且对动态新增的元素无效。

反例：逐个绑定
const items = document.querySelectorAll('#list li');
items.forEach((li) => {
  li.addEventListener('click', () => {
    console.log('点击了', li.textContent);
  });
});

// 痛点：① N 个 li 就有 N 个监听器；
// ② 之后 list.appendChild(newLi) 新增的 li 没绑过，点了没反应

正例：委托给父元素 ul

只在 <ul> 上绑一次。冒泡上来后用 event.target.closest('li') 精准定位——closest 会从 target 自身开始向上查找最近的匹配 li，这样即使用户点到的是 <li> 内部的 <span>、<b> 等子节点，也能正确归位到那一行。

正例：一个监听管一片
const list = document.getElementById('list');

list.addEventListener('click', (e) => {
  // closest 从 target 向上找最近的 li，点到 li 内部的子节点也能命中
  const li = e.target.closest('li');

  // 没命中（如点到了 ul 的空白内边距），或命中的 li 不在本列表里 —— 直接忽略
  if (!li || !list.contains(li)) return;

  console.log('点击了', li.textContent);
});

// 之后动态新增的 li 自动生效，无需重新绑定 ↓
list.insertAdjacentHTML('beforeend', '<li>新来的</li>');
为什么用 closest 而不是 e.target === li？因为 target 是用户实际点到的最深节点。如果 <li><span>文字</span></li>，点到文字时 target 是那个 <span>，直接比较就会漏判。closest 帮你向上"归位"到那一行。

陷阱一：target 可能是更深的子节点

这是委托第一大坑，承接上面。不要假设 target 就是你想要的那一层。两种稳妥的定位方式：

  • e.target.closest('li')：向上找最近的祖先（含自身），适合子元素结构复杂的情况，最常用。
  • e.target.matches('li') 或 e.target.tagName === 'LI'：只判断自身是否匹配，适合子元素结构扁平、确定不会点到内部节点时。注意 tagName 返回的是大写。
两种判断方式
list.addEventListener('click', (e) => {
  // 方式 A：向上归位（推荐）
  const li = e.target.closest('li');

  // 方式 B：只看 target 自身（结构扁平时够用）
  if (e.target.matches('li')) { /* ... */ }
  if (e.target.tagName === 'LI') { /* tagName 是大写！*/ }
});

陷阱二：不冒泡的事件不能直接委托

委托的引擎是冒泡，所以不冒泡的事件就委托不了。最典型的是 focus / blur——它们只在目标元素上触发、不向上冒泡，绑在父元素上根本收不到。解决办法是用它们会冒泡的兄弟版本：focusin / focusout。

事件是否冒泡委托时改用
focus否focusin
blur否focusout
click / input / change是可直接委托
mouseenter / mouseleave否用 mouseover / mouseout 替代

陷阱三：委托范围与 stopPropagation 的冲突

还有两个容易踩的点：

  • 范围过大导致误判：如果把监听器绑到 document 或很高层的容器上，closest('li') 可能匹配到页面里别处的、不属于本列表的 li。务必把委托范围收窄到真正的父容器，并配合 list.contains(li) 二次校验。
  • 子元素调用了 stopPropagation：委托靠冒泡，而 e.stopPropagation() 会掐断冒泡链。如果某个子元素（或第三方组件）在内部 click 里阻止了冒泡，事件就到不了你委托的父元素，委托回调静默失效。排查"委托怎么不触发"时，先去看链路上有没有人偷偷 stopPropagation。
考点：委托 vs 直接绑定怎么选？子元素多、会动态增删（列表、表格、菜单）→ 用委托。元素少且固定、或需要精确控制单个元素的多种事件 → 直接绑更清晰。委托不是银弹：它增加了一层 target 判断逻辑，元素很少时反而是过度设计。
小结：① 委托 = 把监听器绑在共同父元素上，靠冒泡接收子元素事件；② 用 event.target 找事件源、currentTarget 是绑定者，二者别混；③ 定位子元素优先 e.target.closest('选择器')，能正确处理点到内部子节点的情况；④ 好处是省内存、对动态新增元素自动生效；⑤ 三大坑：target 可能是更深节点、focus/blur 不冒泡（改用 focusin/focusout）、范围过大误判 + 子元素 stopPropagation 会让委托失效。

与其他知识点的联系

  • 事件冒泡与捕获（上一篇）：本篇是上一篇冒泡机制最直接的实战应用——没有冒泡，委托这套"绑一个监听管一片"的模式根本无法成立，陷阱二（focus/blur 不冒泡）、陷阱三（stopPropagation 掐断冒泡链）都是上一篇结论在委托场景下的直接后果。
  • addEventListener 与事件对象：委托全靠 event.target 和 event.currentTarget 这对属性区分"谁被点了"和"监听绑在谁身上"，这正是第一篇里专门强调过的高频考点。
  • 常用事件类型速查（下一篇）：速查表会具体列出哪些事件可委托、哪些不冒泡需换用 focusin/focusout，是本篇"陷阱二"表格的延伸和补全。
  • React 的事件委托与合成事件：React 并不像本篇的手写委托那样把监听绑在业务容器上，而是在应用根节点统一挂一个监听器，所有组件的 onClick 等实际上都由这一个原生监听分发触发——这是委托思想在框架层面的规模化应用，且传给回调的是包装过的 SyntheticEvent 而非原生 event。
