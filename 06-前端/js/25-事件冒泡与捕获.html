事件冒泡与捕获

当你点一个按钮，浏览器并不是"只把事件给按钮"那么简单——它会让事件沿着 DOM 树走一个完整的来回。理解这条传播路径，是写对事件委托、关闭弹层、阻止冒泡的前提，也是面试里绕不开的考点。

它解决了什么问题

设想一个嵌套结构：div 里包着 ul，ul 里包着很多 li。当你点中某个 li，从几何上看你同时也点在了它的父 ul 和祖先 div 上。那么到底是谁该响应这次点击？如果每个元素都想知道"我或我的孩子被点了"，浏览器必须有一套明确规则把事件依次派发给这条链上的每个节点。这套规则就是事件流（event flow）。它让你既能在精确的目标元素上监听，也能在祖先元素上"统一兜底"——后者正是事件委托的基础。

历史上 Netscape 主张事件应该"从外向内"传播（捕获），微软主张"从内向外"传播（冒泡），两家吵不拢。最终 W3C 标准把两者都保留，规定一次事件分成三个阶段依次走完，由开发者选择在哪个阶段响应。

事件流的三个阶段

一次事件从触发到结束，会经历严格有序的三个阶段：

  1. 捕获阶段（Capturing）：从最顶层的 window 开始，沿 DOM 树向下逐层经过 document → html → body → … 一路走到目标元素的父节点。
  2. 目标阶段（Target）：到达真正被操作的那个元素（event.target）。
  3. 冒泡阶段（Bubbling）：再从目标元素向上逐层回到 window，像水里的气泡往上浮。

假设结构是 window > div#outer > div#inner，你点击了 #inner，事件的完整传播路径如下：

一次点击在嵌套元素中的传播顺序
         ↓ 捕获阶段（向下）        ↑ 冒泡阶段（向上）
  window ───┐                        ┌─── window
            ↓                        ↑
  document ─┐                        ┌─ document
            ↓                        ↑
  div#outer ┐                        ┌ div#outer
            ↓                        ↑
          div#inner  ←── 目标阶段 ──→  div#inner
             （event.target）

// 实际触发顺序（每个元素若同时绑了捕获和冒泡监听）：
// window(捕获) → outer(捕获) → inner(目标) → outer(冒泡) → window(冒泡)

如何选择在哪个阶段监听

addEventListener 的第三个参数控制监听器注册在哪个阶段。它既可以是布尔值，也可以是配置对象：

第三参数 / capture 选项
// 默认：第三参省略或 false —— 在【冒泡阶段】触发
el.addEventListener("click", handler);
el.addEventListener("click", handler, false);

// 传 true，或 { capture: true } —— 改为在【捕获阶段】触发
el.addEventListener("click", handler, true);
el.addEventListener("click", handler, { capture: true });
第三参数含义触发阶段
省略 / false默认值冒泡阶段
true简写形式捕获阶段
{ capture: true }配置对象写法捕获阶段
绝大多数业务代码都用默认的冒泡阶段。捕获阶段较少用，典型场景是想"抢在所有子元素处理之前先拦一道"，比如全局埋点、统一权限校验。

例子：父子都绑 click，观察打印顺序

下面给同一对嵌套元素分别注册冒泡监听和捕获监听，对比点击 #inner 时控制台的输出顺序——这是把三阶段理论"看见"的最直接方式。

冒泡 vs 捕获的执行顺序
// HTML: <div id="outer"><div id="inner">点我</div></div>
const outer = document.getElementById("outer");
const inner = document.getElementById("inner");

// 默认冒泡监听
outer.addEventListener("click", () => console.log("outer 冒泡"));
inner.addEventListener("click", () => console.log("inner 冒泡"));

// 捕获监听（第三参 true）
outer.addEventListener("click", () => console.log("outer 捕获"), true);
inner.addEventListener("click", () => console.log("inner 捕获"), true);

// 点击 inner，输出顺序：
// outer 捕获   ← 捕获阶段，从外向内
// inner 捕获   ← 到达目标（目标阶段两个监听按注册顺序执行）
// inner 冒泡
// outer 冒泡   ← 冒泡阶段，从内向外
记忆口诀：先捕获后冒泡，捕获从外向内、冒泡从内向外。这个顺序记反是常见考点。注意在目标元素自身上，捕获和冒泡的概念已不分先后，两个监听仅按注册顺序执行。

阻止传播：stopPropagation 与 stopImmediatePropagation

有时你不希望事件继续往下/往上传。事件对象提供了两个方法：

方法作用
e.stopPropagation()阻止事件继续向后续节点传播，但当前元素上其余的同类监听仍会执行。
e.stopImmediatePropagation()不仅阻止继续传播，连同当前元素上后面注册的其他监听一起拦掉。
两者的区别
inner.addEventListener("click", (e) => {
  e.stopPropagation();          // outer 收不到了，但下面这个监听 2 仍执行
  console.log("监听 1");
});
inner.addEventListener("click", () => console.log("监听 2"));
// 点击输出：监听 1、监听 2（outer 不再触发）

// 换成 stopImmediatePropagation()，则"监听 2"也会被拦掉，只输出"监听 1"

注意区分另一个常被混淆的方法 e.preventDefault()：它阻止的是浏览器默认行为（如链接跳转、表单提交），与事件传播是两件互不相干的事，互不影响。

target vs currentTarget

这是事件对象里最容易搞混的一对属性，尤其在事件委托里至关重要：

属性含义传播中是否变化
e.target事件最初的触发源（真正被点的那个元素）整条链上始终不变
e.currentTarget当前正在执行监听的那个元素（即绑监听的元素）随传播逐层变化
同一次点击，两个属性的差异
// 点击 inner，但监听绑在 outer 上
outer.addEventListener("click", (e) => {
  console.log(e.target.id);         // "inner" —— 真正被点的
  console.log(e.currentTarget.id);  // "outer" —— 正在处理监听的
  console.log(this === e.currentTarget);  // true（普通函数里 this 即 currentTarget）
});
为什么事件委托靠的是 target？因为监听绑在父元素（currentTarget 恒为父），而你真正想知道"用户点的是哪个子项"——那个信息只在 target 里。下一节会专门展开。

常见陷阱

陷阱一：滥用 stopPropagation 破坏其他逻辑

很多人一遇到"点击穿透"就随手 stopPropagation()，但这会掐断整条冒泡链。后果是任何依赖冒泡的逻辑都失效：基于 document 的事件委托收不到事件、"点击空白处关闭弹层/下拉菜单"的全局监听失灵、第三方埋点统计丢数据。能用判断 e.target 解决的，就别动传播链。

反例 → 问题：弹层内部 onClick 里写了 e.stopPropagation() 来"防止冒泡关闭弹层"，结果整个页面挂在 document 上的全局快捷键、关闭其他面板的逻辑全部失效。正例：在 document 的关闭逻辑里用 if (!panel.contains(e.target)) 判断点击是否在弹层外，而不是阻止冒泡。

陷阱二：捕获与冒泡顺序记反

记住物理直觉：捕获是"信号从天线塔（window）往下传到收音机（target）"，冒泡是"气泡从水底（target）往上浮到水面（window）"。先有捕获，后有冒泡。

陷阱三：有些事件根本不冒泡

不是所有事件都会冒泡。最典型的是 focus / blur / mouseenter / mouseleave——它们的 event.bubbles 为 false，无法用在父元素上做事件委托。对应地，标准提供了会冒泡的孪生事件 focusin / focusout（以及用 mouseover / mouseout 替代 enter/leave）。

focus 不冒泡，focusin 才冒泡
// 想在表单容器上统一监听"任意输入框获得焦点"
form.addEventListener("focus", handler);    // ❌ 收不到，focus 不冒泡
form.addEventListener("focusin", handler);  // ✅ focusin 会冒泡，能委托

// 补救手段：focus/blur 也可在捕获阶段监听（虽不冒泡，但走捕获）
form.addEventListener("focus", handler, true);  // ✅ 第三参 true，捕获阶段也能拿到
小结：① 一次事件依次经历捕获（外→内）→ 目标 → 冒泡（内→外）三阶段；② addEventListener 第三参 true / {capture:true} 在捕获阶段触发，默认 false 在冒泡阶段；③ stopPropagation() 拦后续节点、stopImmediatePropagation() 连本元素其他监听一起拦，二者都与 preventDefault() 无关；④ target 是触发源（不变），currentTarget 是当前监听元素（随传播变）；⑤ 别乱用 stopPropagation 破坏委托/关弹层，focus/blur 不冒泡要换 focusin/focusout。

与其他知识点的联系

  • addEventListener 与事件对象（上一篇）：上一篇只简单提了 preventDefault/stopPropagation，本篇把它们放进完整的捕获 → 目标 → 冒泡三阶段模型里，并补上了 options.capture 具体控制的是哪个阶段。
  • 事件委托（下一篇）：委托的全部原理就是"冒泡"——把监听绑在共同父元素上，靠本篇讲的冒泡阶段把子元素事件送上来，再用 event.target 反推是哪个子元素被点了。本篇结尾的"陷阱三：不冒泡的事件"（focus/blur）直接决定了下一篇委托能不能用在这些事件上。
  • 常用事件类型速查：速查表里 mouseenter/leave 不冒泡、mouseover/out 冒泡的差异，本质就是本篇讲的"事件流"——是否会经过冒泡阶段，是速查表能否用于事件委托的分水岭。
  • React 的合成事件与统一委托：React 并不会给每个元素单独挂原生监听，而是利用本篇讲的原生事件冒泡机制，在根节点统一挂一个监听器，事件冒泡上来后由 React 自己的调度系统分发给各组件的 onClick 等处理函数——这本质上是框架级的"事件委托"，建立在浏览器原生冒泡之上。
