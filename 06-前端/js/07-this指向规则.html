this 的指向规则

this 是 JS 里最让人摸不着头脑的关键字。很多人以为它指向"函数自己"或"定义它的对象"，结果一调用就翻车。记住一句话就能解开大半谜团：this 由"函数怎么被调用"决定，而不是它写在哪里。

它解决了什么问题

同一段方法逻辑，往往希望能作用在不同的对象上。比如 greet() 想打印"谁在打招呼"，如果把对象名写死在函数里，这个函数就只能服务一个对象。this 提供了一个运行时才确定的"当前调用主体"占位符：函数被谁调用，this 就指向谁，于是一份代码可以被无数对象复用。

代价是：this 的值不在定义时锁定，而在调用那一刻才绑定。这正是初学者踩坑的根源——你看着函数定义在某个对象里，就想当然认为 this 是它，可一旦换种方式调用，指向就变了。判断 this 的唯一可靠方法，是去看调用点（call-site）那一行代码长什么样。

规则一：默认绑定（独立调用）

当一个函数被直接调用（前面没有任何对象、没有 new、没有 call/apply/bind），就走默认绑定。非严格模式下 this 指向全局对象（浏览器里是 window），严格模式下指向 undefined。

独立调用：看前面有没有"点"
function show() { console.log(this); }

show();           // 非严格：window；严格模式（"use strict"）：undefined

function strict() { "use strict"; console.log(this); }
strict();         // undefined —— 严格模式不再"兜底"成全局对象
为什么严格模式要改成 undefined？非严格模式"兜底"成 window 会掩盖错误：你以为方法挂在某对象上，结果 this.xxx = 1 悄悄污染了全局。严格模式让它直接报错，bug 暴露得更早。ES 模块（<script type="module">）默认就是严格模式。

规则二：隐式绑定（obj.fn()）

当函数作为某个对象的方法被调用——即调用点是 obj.fn() 形式——this 指向点号前面那个对象。注意只看"调用时紧挨着的那一层"，多层属性链只认最后一个点前面的对象。

this 指向"点号前面"的对象
const user = {
  name: "Shea",
  greet() { console.log(this.name); }
};
user.greet();     // "Shea" —— 调用点是 user.greet()，this = user

// 多层链：只认最后一个点前面的对象
const app = { user };
app.user.greet(); // "Shea" —— this 仍是 user，不是 app

隐式丢失是这条规则最大的坑：一旦把方法从对象里"拿出来"（赋值给变量、当作回调传递），调用点就不再是 obj.fn() 而是裸调用，this 立刻退回默认绑定。

反例 → 问题：隐式丢失
const fn = user.greet;  // 只取出了函数本身，没带 user
fn();                    // undefined（严格）/ 报错 —— 调用点是 fn()，this 丢了

setTimeout(user.greet, 100); // 同理：传进去的是裸函数，回调时 this 已不是 user

规则三：显式绑定（call / apply / bind）

当你想强制指定 this，用 call、apply 或 bind。三者都把第一个参数作为 this，区别在于"怎么传后续参数"和"是否立即执行"。

方法传参方式是否立即执行典型用途
fn.call(obj, a, b)参数逐个列出是临时借用方法
fn.apply(obj, [a, b])参数打包成数组是参数本就是数组时
fn.bind(obj, a)逐个列出（可预填）否，返回新函数回调/事件里固定 this
call / apply / bind 三连
function intro(city, job) {
  console.log(`${this.name} @ ${city}, ${job}`);
}
const me = { name: "Shea" };

intro.call(me, "上海", "前端");   // 参数逐个：Shea @ 上海, 前端
intro.apply(me, ["上海", "前端"]); // 参数装数组，结果相同

const bound = intro.bind(me, "上海"); // 不执行，返回固定了 this 和首参的新函数
bound("前端");                       // 之后随时调用：Shea @ 上海, 前端
记忆法：apply 的 a 联想 array（数组传参）；bind 是"绑定"，绑完不跑，只交还一个新函数。bind 正是解决"隐式丢失"的常用手段：setTimeout(user.greet.bind(user), 100)。

规则四：new 绑定（构造调用）

用 new 调用函数时，引擎会新建一个空对象，把它作为 this，执行构造逻辑后默认返回这个新对象。所以构造函数里的 this 永远指向那个即将被创建出来的实例。

new 让 this 指向新实例
function Person(name) {
  // new 会偷偷做：this = {}; ...; return this;
  this.name = name;
}
const p = new Person("Shea");
console.log(p.name); // "Shea" —— this 指向新建的 p

箭头函数：没有自己的 this

箭头函数是个特例，上面四条规则对它统统失效。它没有自己的 this，也不受 call/apply/bind 改变；它的 this 取自定义它时所在的词法作用域——即外层普通函数的 this。换句话说，箭头函数的 this 在写下它的那一刻就由外层环境决定了。

这恰好解决了"隐式丢失"和"回调里 this 跑偏"的经典痛点：在对象方法里写回调，用箭头函数就能稳稳保住外层的 this。

正例：箭头函数在回调里保住 this
const timer = {
  count: 0,
  start() {
    // 反例：普通函数回调，this 退回默认绑定，this.count 是 undefined
    // setInterval(function () { this.count++; }, 1000);

    // 正例：箭头函数继承 start 的 this（即 timer）
    setInterval(() => { this.count++; }, 1000);
  }
};
timer.start(); // count 正常每秒 +1
反过来的坑：箭头函数不适合做对象方法。因为它捕获的是定义时外层的 this，而对象字面量本身不构成函数作用域，外层往往是模块顶层或 window，于是 this.name 拿不到对象属性。
反例：用箭头函数当对象方法
const user = {
  name: "Shea",
  greet: () => console.log(this.name)  // this 是外层（非 user！）
};
user.greet(); // undefined —— 即使写成 user.greet()，箭头函数也不认 user

绑定优先级

当多条规则可能同时适用时，按下表优先级判定（数字越小越优先）。先看是不是箭头函数；不是再看 new；再看有没有显式绑定；然后看是不是 obj.fn()；都不是才走默认绑定。

优先级规则判断依据（看调用点）this 指向
0（凌驾一切）箭头函数函数本身是 => 定义定义时外层词法的 this
1new 绑定new Fn()新建的实例
2显式绑定fn.call/apply/bind(obj)传入的 obj
3隐式绑定obj.fn()点号前的 obj
4（兜底）默认绑定裸调用 fn()非严格 window / 严格 undefined
判断口诀：是箭头吗？→ 看外层。有 new 吗？→ 新对象。有 call/apply/bind 吗？→ 指定对象。有点调用 obj.fn() 吗？→ 点前对象。都没有？→ 默认（严格 undefined）。
小结：① this 看调用点，不看定义位置；② 四条规则按优先级：new > 显式 call/apply/bind > 隐式 obj.fn() > 默认（严格下为 undefined）；③ 把方法取出来或当回调传会触发"隐式丢失"，bind 可固定 this；④ 箭头函数没有自己的 this，继承外层词法作用域——适合回调保 this，不适合做对象方法。
