对象与属性操作

对象是 JS 里组织数据的核心结构——一组键值对。读写、增删、遍历、复制这些日常操作背后藏着几条关键规则：键到底是什么类型、遍历会不会"漏看"或"多看"、复制是真复制还是共享引用。把这些理清楚，才不会被"改了 A 怎么 B 也变了"这类幽灵 bug 困住。

它解决了什么问题

程序里的数据很少是孤零零的单个值，更多时候是"一坨相关的属性"——一个用户有名字、年龄、邮箱。如果用三个独立变量去存，传参、返回、组织都很笨拙。对象把这些相关数据打包成一个整体，用键（key）给每个值起名字，用值（value）存内容。它本质上是一张哈希表：通过键快速定位值，这也是为什么对象查找属性几乎是常数时间。理解对象，就是理解 JS 里"结构化数据"的最基本载体——后面学的 JSON、类、甚至模块导出，底层都是对象。

创建对象：字面量

最常用的方式是对象字面量（花括号）。键名是字符串（写的时候可省略引号，除非含特殊字符），值可以是任意类型，包括函数。

对象字面量
const user = {
  name: "Alice",
  age: 28,
  "e-mail": "a@x.com",   // 含特殊字符的键必须加引号
  greet() {                  // 方法简写，等价于 greet: function() {}
    return "hi";
  }
};

读写属性：点 . vs 方括号 []

两种访问方式各有适用场景。点语法简洁，但键名必须是写死的合法标识符；方括号接受任意字符串表达式，因此键名在运行时才确定（动态键名）时只能用它。

点 vs 方括号
user.name;              // "Alice" —— 点语法，键写死
user["e-mail"];         // 含 "-" 的键，点语法写不出来，只能用 []

// 动态键名：键存在变量里，运行时才知道是哪个
const field = "age";
user[field];            // 28 —— 等价于 user["age"]
user.field;             // undefined —— 这是在找名为 "field" 的键，不是 "age"！

// 写入同理
user.city = "NYC";     // 新增属性
user["age"] = 29;      // 修改已有属性
陷阱：访问不存在的属性返回 undefined，不会报错。这把"拼错键名"和"键确实不存在"两种情况混在了一起，常常让 bug 推迟暴露。后续在 undefined 上再取属性（如 user.addr.city）才会抛 TypeError。可选链 ?. 就是为缓解这个问题而生。

计算属性名：创建时就用变量当键

方括号不仅能读写，还能直接出现在对象字面量里，让键名由表达式动态生成——这叫计算属性名（ES6）。在按接口字段拼装对象时极其常用。

计算属性名 {[key]: v}
const key = "status";
const val = "active";

const obj = { [key]: val };   // { status: "active" }

// 没有方括号的话，键就是字面量 "key" 这个字符串
const wrong = { key: val }; // { key: "active" }

属性简写与方法简写

当变量名正好是想要的键名时，可以省略冒号和值，这是 ES6 的属性简写——构造返回对象、解构再组装时大量出现。方法也有对应的简写（前面字面量例子里已用过）。

简写语法
const name = "Bob", age = 30;

// 简写：{ name: name, age: age } 的等价写法
const p = { name, age };     // { name: "Bob", age: 30 }

删除属性：delete

delete 会真正把键从对象上移除（不只是设成 undefined），之后 in 判断会返回 false。注意它只删自身属性，碰不到原型链。

delete vs 赋值 undefined
const o = { a: 1, b: 2 };

delete o.a;            // 键 a 被彻底删除
"a" in o;            // false

o.b = undefined;   // 键 b 还在，只是值变成 undefined
"b" in o;            // true —— 这就是两者的区别

判断属性存在：in vs hasOwnProperty

这是高频考点。两者都能判断属性是否存在，区别在查不查原型链：in 会顺着原型链一路往上找；hasOwnProperty 只看对象自身。

in 含原型链，hasOwnProperty 只看自身
const o = { a: 1 };

"a" in o;                      // true，自身有
"toString" in o;               // true！继承自 Object.prototype

o.hasOwnProperty("a");         // true
o.hasOwnProperty("toString");  // false，不是自身的

// 现代推荐写法，避免对象自身覆盖了 hasOwnProperty 方法
Object.hasOwn(o, "a");           // true（ES2022）

遍历对象

对象不是可迭代对象（不能直接 for...of），遍历靠以下几种方式。Object.keys/values/entries 返回的是自身可枚举属性，干净可控；for...in 则会连原型链上的可枚举属性一起遍历，是个常见坑。

方式返回 / 行为是否含原型链
Object.keys(o)自身可枚举键的数组否
Object.values(o)自身可枚举值的数组否
Object.entries(o)[键, 值] 对的数组否
for...in逐个遍历可枚举键是 ⚠️
三种 Object 方法 + for...in
const o = { a: 1, b: 2 };

Object.keys(o);     // ["a", "b"]
Object.values(o);   // [1, 2]
Object.entries(o);  // [["a", 1], ["b", 2]]

// 配合解构遍历，最常用的写法
for (const [k, v] of Object.entries(o)) {
  console.log(k, v);   // a 1 / b 2
}

// for...in 会爬原型链，若需排除要手动过滤
for (const k in o) {
  if (!Object.hasOwn(o, k)) continue;  // 跳过继承来的属性
  console.log(k);
}

合并与复制：Object.assign 与展开 {...obj}

两种方式都做浅拷贝：把源对象自身可枚举属性拷到目标。Object.assign 会修改第一个参数（目标对象）；展开语法 {...obj} 更简洁，总是产生新对象，是现代首选。

合并与浅拷贝
const base = { a: 1, b: 2 };
const patch = { b: 99, c: 3 };

// 展开：后面的覆盖前面的同名键
const merged = { ...base, ...patch }; // { a:1, b:99, c:3 }

// Object.assign(目标, ...源)：返回并改写目标
Object.assign(base, patch);          // base 被改成 { a:1, b:99, c:3 }

// 纯复制：用空对象当目标，避免污染源
const copy = { ...base };
浅拷贝陷阱：嵌套对象仍共享引用。展开和 assign 只复制第一层；如果某个值本身是对象，复制的只是它的引用地址，新旧对象指向同一个嵌套对象。
浅拷贝只拷一层
const o1 = { info: { n: 1 } };
const o2 = { ...o1 };          // 浅拷贝

o2.info.n = 99;
console.log(o1.info.n);   // 99！o1 也被改了，因为 info 是同一个对象

// 需要深拷贝时（结构可序列化）：
const deep = structuredClone(o1);  // 现代内置 API，递归复制

冻结对象：Object.freeze

Object.freeze(o) 让对象不可增删改属性。在非严格模式下，违规修改会被静默忽略；严格模式下抛错。它同样是浅冻结——只冻第一层，嵌套对象仍可改。

Object.freeze 浅冻结
const cfg = Object.freeze({ port: 8080, db: { host: "x" } });

cfg.port = 9090;     // 静默失败（严格模式下抛 TypeError）
cfg.db.host = "y";  // 成功！嵌套对象没被冻

Object.isFrozen(cfg);  // true

Object.fromEntries：entries 的逆操作

它把 [键, 值] 对的列表（数组或 Map）还原成对象，正好和 Object.entries 相反。两者搭配，可以对对象做"转成数组 → 用数组方法处理 → 转回对象"的链式变换。

对对象做过滤/映射的经典套路
const scores = { a: 90, b: 40, c: 75 };

// 只保留及格（>= 60）的项
const passed = Object.fromEntries(
  Object.entries(scores).filter(([k, v]) => v >= 60)
);
console.log(passed);  // { a: 90, c: 75 }

陷阱合集：对象键的真相

一个常被忽略的事实：普通对象的键最终都会被转成字符串（唯一例外是 Symbol）。所以用数字、甚至对象当键，都会经历一次 String() 转换。

键被悄悄转成字符串
const o = {};
o[1] = "num";
o["1"] = "str";   // 覆盖了上一行！因为键 1 === "1"
console.log(o);          // { "1": "str" } —— 只有一个键

// 用对象当键更离谱：都会变成 "[object Object]"
const k1 = {}, k2 = {};
o[k1] = "A";
o[k2] = "B";      // 又覆盖了！k1、k2 都转成同一字符串
对比：正是因为普通对象的键被限制为字符串/Symbol，且无法保证插入顺序对所有键生效，Map 才应运而生——它允许任意类型当键（包括对象、函数），并严格保留插入顺序。键不是字符串、需要频繁增删、或要统计大小时，优先考虑 Map（下一节展开）。
小结：① 动态键名、含特殊字符的键用方括号 []，字面量里用计算属性名 {[k]: v}；② 访问不存在的属性返回 undefined 不报错，注意拼错键名；③ in 查原型链、hasOwnProperty/Object.hasOwn 只查自身，for...in 同样会爬原型链；④ 遍历首选 Object.keys/values/entries；⑤ 展开 {...o} 和 Object.assign 都是浅拷贝，嵌套对象共享引用，Object.freeze 也是浅冻结；⑥ 普通对象的键最终都是字符串，需要任意类型键就用 Map。

与其他知识点的联系

  • 条件语句与循环：本篇讲 for...in 会爬原型链，正呼应条件语句与循环那一篇早先提到的"用 for...in 遍历数组会混入继承属性"的坑——两处坑同源：for...in 从设计上就是给对象的可枚举属性（含继承）用的，用来遍历数组或想要"只看自身属性"都是用错场景。
  • 解构赋值与展开运算符：本篇的 {...base, ...patch} 浅拷贝合并和上一篇讲的展开运算符是同一套语法在对象上的应用；对象解构（上上篇）按属性名取值，本篇的点/方括号访问是它更基础的底层操作，两者互为"批量读"与"单个读写"的关系。
  • 数组与常用方法：本篇"浅拷贝陷阱：嵌套对象仍共享引用"和上一篇数组的 [...arr] 浅拷贝踩的是完全相同的坑——展开语法无论用在数组还是对象上，都只复制第一层，深层引用类型的字段依旧共享内存；Object.entries 与数组方法 filter/map 链式配合，是对对象做批量处理的标准套路。
  • React Hooks 与不可变数据：React 更新对象类型的 state 时必须遵循本篇"浅拷贝"的写法（setState({ ...state, name: "new" })），直接修改 state.name = "new" 不会触发重渲染——这正是本篇"展开只拷一层、必须生成新引用"的规则在 React 场景里的具体应用，嵌套字段更新时还需要逐层展开才能让 React 侦测到变化。
  • TypeScript 类型系统：本篇的计算属性名和动态键在 TS 里对应索引签名（{ [key: string]: number }）或映射类型（Mapped Types）；Object.freeze 的浅冻结特性在 TS 里可以用 Readonly<T>（同样只作用于第一层）在类型层面提前约束，编译期就能拦住对只读属性的误改。
