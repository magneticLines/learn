Class 语法

ES6 的 class 让 JS 的面向对象写法终于"长得像"主流语言。但它本质只是一层语法糖——底层仍是原型和构造函数。看懂这层糖衣，既能写出整洁的类，也不会被那些"反直觉"的行为绊倒。

它解决了什么问题

ES6 之前，要在 JS 里"造类"，得手写构造函数 + 给 prototype 挂方法 + 用 Object.create 串联原型链这一整套样板，啰嗦且容易写错（比如忘了修复 constructor 指向）。class 把这套约定俗成的写法收编成统一语法，可读性大幅提升，继承也只需一个 extends。但要牢记：它没有引入新的对象模型，跑起来仍是原型那一套（原型链的细节留到下一篇），所以理解原型才能真正吃透 class 的种种行为。

最小的类：constructor 与方法

constructor 是构造器，new 时自动调用，负责初始化实例属性（每个对象一份，存在实例自身上）。而普通方法被定义在原型上，所有实例共享同一份函数——这正是把方法放原型的意义：省内存。

定义并使用一个类
class Point {
  constructor(x, y) {
    this.x = x;      // 实例属性：存在每个对象自己身上
    this.y = y;
  }                  // 注意：方法之间不加逗号！
  distance() {          // 这个方法挂在 Point.prototype 上，所有实例共享
    return Math.hypot(this.x, this.y);
  }
}

const p = new Point(3, 4);
console.log(p.distance());  // 5
忘记 new 会直接报错。普通构造函数若漏写 new，this 会指向全局对象（或 undefined），悄悄出错；而 class 强制要求 new，直接调 Point(3,4) 会抛 TypeError: Class constructor Point cannot be invoked without 'new'——这其实是好事，错误暴露得早。

getter / setter 访问器

用 get / set 可以把"方法调用"伪装成"属性读写"，常用于计算属性或在赋值时做校验。访问时不加括号，像普通属性一样用。

访问器属性
class Temperature {
  constructor(c) { this.celsius = c; }
  get fahrenheit() { return this.celsius * 1.8 + 32; }
  set fahrenheit(f) { this.celsius = (f - 32) / 1.8; }
}

const t = new Temperature(25);
console.log(t.fahrenheit);  // 77   —— 像读属性，实际调了 getter
t.fahrenheit = 212;        // 像写属性，实际调了 setter
console.log(t.celsius);     // 100

static 静态属性与方法

static 成员属于类本身，不属于实例。常用于工具方法、工厂方法或常量。最典型的就是 Math.hypot、Array.from 这类——你从不需要 new Math()。静态方法里的 this 指向类本身，不是实例。

静态成员属于类，不属于实例
class Circle {
  static PI = 3.14159;          // 静态属性
  constructor(r) { this.r = r; }
  static fromDiameter(d) {     // 静态工厂方法
    return new Circle(d / 2);
  }
}

console.log(Circle.PI);             // 3.14159 —— 用类名访问
const c = Circle.fromDiameter(10);  // 半径 5 的圆
console.log(c.PI);                // undefined —— 实例上没有静态属性

类字段与 # 私有字段

ES2022 标准化了类字段（class fields）：可以直接在类体里写 count = 0，等价于在构造器里 this.count = 0，更简洁。在字段名前加 # 就是真正的私有字段——外部访问会语法报错，而不是像旧时代用 _name 那样靠"君子约定"。

#name 是语言级私有，外部访问直接报错
class Counter {
  count = 0;          // 公有类字段
  #secret = 42;      // 私有字段，必须先在类体里声明
  inc() { this.count++; }
  reveal() { return this.#secret; }  // 类内部才能访问
}

const c = new Counter();
c.inc();
console.log(c.count);    // 1
console.log(c.reveal()); // 42
// console.log(c.#secret);  // ❌ SyntaxError：私有字段无法在类外访问
对比旧约定：过去常用下划线前缀 this._secret 表示"请别碰我"，但它仍是公有的，外部照样能读写。# 才是引擎层面强制的私有，连 Object.keys 也看不到它。

extends 继承与 super

extends 一行搞定原型链的串联。子类构造器里有两条铁律：① 用 this 之前必须先调 super()（它负责执行父类构造器、初始化父类那部分的 this）；② 重写的方法里用 super.method() 调父类同名方法。

继承：super() 在前，super.method() 复用父类逻辑
class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} 发出声音`; }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);          // 必须先调，否则用 this 会 ReferenceError
    this.breed = breed;
  }
  speak() {
    return super.speak() + `：汪汪（${this.breed}）`;  // 复用父类
  }
}

const d = new Dog("旺财", "柴犬");
console.log(d.speak());  // 旺财 发出声音：汪汪（柴犬）
面试高频："为什么 super() 必须在 this 之前？"因为子类的 this 实际上是由父类构造器"创建并加工"出来的。在 super() 返回之前，this 尚未初始化，处于类似 TDZ 的状态，提前读写就抛 ReferenceError: Must call super constructor ... before accessing 'this'。

instanceof：判断原型链归属

instanceof 检查"某个构造函数的 prototype 是否出现在对象的原型链上"。继承关系下，子类实例对父类也返回 true。

instanceof 沿原型链向上查
console.log(d instanceof Dog);     // true
console.log(d instanceof Animal);  // true —— Dog 继承自 Animal
console.log(d instanceof Object);  // true —— 链顶是 Object

陷阱：class 不会被提升

函数声明会整体提升，先调用后定义也没问题；但 class 不会提升——确切说它存在 TDZ，定义之前访问就报错。所以类必须先定义、后使用。

函数能提前用，class 不能
sayHi();              // ✅ ok —— 函数声明被提升
function sayHi() { console.log("hi"); }

new Foo();             // ❌ ReferenceError: Cannot access 'Foo' before initialization
class Foo {}

陷阱：箭头函数定义方法不在原型上

用类字段 + 箭头函数写方法（method = () => {}）时，它不在原型上，而是每个实例一份。好处是 this 自动绑定实例（常用于 React 事件回调）；代价是更费内存，且无法用 super。这是一个权衡，不是"更好的写法"。

原型方法 vs 实例上的箭头函数字段
class Widget {
  render() {}            // 在 Widget.prototype 上，所有实例共享一份
  handleClick = () => {};  // 每个实例都新建一份，this 绑死实例
}

const a = new Widget(), b = new Widget();
console.log(a.render === b.render);            // true  —— 共享
console.log(a.handleClick === b.handleClick);  // false —— 各一份

对比：class 写法 vs 旧的构造函数 + prototype

class 只是把下面这套手写样板包装起来，两者几乎等价。看懂这组对照，就理解了"语法糖"四个字的分量——class 没变魔术，只是让你少敲字、少出错（原型链的完整机制下一篇细讲）。

等价关系：右边是左边脱掉糖衣的样子
// ── ES6 class 写法 ──
class Point {
  constructor(x) { this.x = x; }
  show() { return this.x; }
}

// ── 等价的旧写法 ──
function Point(x) { this.x = x; }      // 构造函数 = constructor
Point.prototype.show = function () { // 方法手动挂到 prototype
  return this.x;
};
维度class 写法构造函数 + prototype
本质完全相同：方法都在 prototype，靠原型链查找
提升不提升（TDZ）函数声明会提升
严格模式类体内默认开启需手动 "use strict"
必须 new是，否则报错否，漏写会悄悄出错
继承extends + super手动设置原型 + 调父构造器
可读性高，结构集中低，方法散落各处
小结：① class 是语法糖，底层仍是原型 + 构造函数；② 方法在原型上共享，实例属性各一份；③ get/set 把方法伪装成属性，static 属于类本身，# 是真私有；④ extends 继承时 super() 必须先于 this，方法里用 super.method()；⑤ 牢记五个坑：class 不提升、类体默认严格模式、方法间不加逗号、漏 new 报错、箭头函数字段每实例一份不在原型上。

与其他知识点的联系

  • 原型链与继承（下一篇）：本篇反复强调"class 只是语法糖"，下一篇会把这层糖完全剥开，讲清楚 extends 底层其实就是手动拼接 Dog.prototype.__proto__ = Animal.prototype。不看那篇，你只是记住了 class 的用法，不算真正理解它。
  • JSON 序列化（上一篇）：toJSON() 方法就定义在类体里，和 constructor、普通方法写在同一个地方——这是类字段/方法思维在"如何呈现自己"这个具体问题上的延伸应用。
  • React 组件与 this 绑定：本篇"箭头函数字段不在原型上"这个陷阱，正是 React 类组件写事件回调时的标准手法（handleClick = () => {}），目的是让 this 自动绑定实例，避免手写 this.handleClick = this.handleClick.bind(this)。理解了原型共享方法 vs 实例独立字段的取舍，才懂这行代码解决的到底是什么问题。
  • 为什么 React 不直接用原生 DOM 操作：React 组件本质上也是"类"（或函数）+ 状态的组合，它把 UI 更新抽象成"状态变了，重新描述界面"，而不是像原生 DOM API 那样手动一步步 createElement/appendChild。class 语法提供的封装（数据 + 行为绑一起）正是组件化思想的语言基础。
  • Node.js 模块系统：一个类通常就是一个模块的默认导出（export default class Foo {}），模块负责"文件级"的封装边界，类负责"对象级"的封装边界，两层封装配合，才有了现代前端项目里"一个文件一个组件/类"的组织方式。
