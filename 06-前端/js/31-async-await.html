async / await

Promise 把回调地狱拉平成了链式调用，但 .then() 套来套去依然不够直观。async / await 是建立在 Promise 之上的语法糖——它让异步代码读起来、写起来都像同步代码，是现代 JS 处理异步的首选写法。

它解决了什么问题

上一篇我们用 Promise 解决了回调地狱，但 .then() 链式调用有两个不爽：① 取出的值被困在回调函数里，要靠层层 .then 往下传；② 控制流（条件、循环、try/catch）和 .then 拼起来很别扭。async / await（ES2017）用同步的书写形式包装了 Promise：await 把"等待结果"这件事变成一行赋值语句，错误处理直接用熟悉的 try/catch。它没有引入新的异步机制，底层仍然是 Promise——只是换了件更顺眼的外衣。

同一段逻辑：Promise 链 vs async/await
// Promise 链：值困在回调里，层层缩进
function getUser() {
  return fetch('/api/user')
    .then(res => res.json())
    .then(user => fetch(`/api/orders/` + user.id))
    .then(res => res.json());
}

// async/await：像同步代码一样自上而下读
async function getUser() {
  const res = await fetch('/api/user');
  const user = await res.json();
  const orders = await fetch(`/api/orders/` + user.id);
  return await orders.json();
}

async 函数总是返回 Promise

给函数加上 async 关键字，无论你 return 什么，外部拿到的永远是一个 Promise。返回普通值时会被自动包成 Promise.resolve(值)；函数体里抛出异常时则变成一个 rejected 的 Promise。这正是 async/await 能和 Promise 无缝互通的根基——它们本就是同一套东西。

返回值与异常都被 Promise 包裹
async function f() { return 42; }
f().then(v => console.log(v));   // 42（42 被包成 Promise.resolve(42)）

async function g() { throw new Error('boom'); }
g().catch(e => console.log(e.message));  // boom（抛错 → rejected Promise）

await：暂停并取出结果

await 后面跟一个 Promise，它会暂停当前 async 函数的执行，等这个 Promise 完成（fulfilled）后取出其结果值，再继续往下走。注意"暂停"只是暂停了这个函数，主线程并没有被阻塞——引擎会去执行其它任务，等结果好了再回到这里。

规则说明
只能在 async 函数内用普通函数里写 await 是语法错误
取出的是结果值await p 的值是 p resolve 出的值，不是 Promise
await 非 Promise 值会被自动包装，等价于 await Promise.resolve(值)
Promise 被 rejectawait 处会"抛出"该错误，可被 try/catch 捕获
顶层 await仅在 ES 模块（<script type="module"> 或 .mjs）顶层可用

错误处理：用 try / catch

这是 async/await 比 .then().catch() 更自然的地方。被 await 的 Promise 一旦 reject，错误会在 await 那一行"抛出"，于是你可以用同步代码里最熟悉的 try/catch 来兜住它——同步异常和异步异常被统一成了一种处理方式。

try/catch 同时兜住同步与异步错误
async function load() {
  try {
    const res = await fetch('/api/data');
    if (!res.ok) throw new Error('HTTP ' + res.status);
    return await res.json();
  } catch (err) {
    // 网络失败、4xx/5xx、JSON 解析失败 都会落到这里
    console.error('加载失败：', err);
    return null;
  } finally {
    console.log('无论成败都执行（关 loading 等）');
  }
}
关于 Fetch API：下一站会专门讲 fetch。这里先记住一个坑——fetch 只在网络层面失败时才 reject，HTTP 404/500 这类响应它认为是"成功收到了"，res.ok 为 false 但 Promise 不会 reject，所以要手动判断并 throw。

并发要点：别让独立任务排队

这是用 async/await 最容易写慢的地方。连续的 await 是串行的：第一个等完才发起第二个。如果两个任务互不依赖，这种写法白白浪费时间。正确做法是先把 Promise 都发出去，再用 Promise.all 一起 await，让它们并发执行。

反例：串行 await（总耗时 = 之和）
async function slow() {
  const user = await fetchUser();     // 等 1s
  const posts = await fetchPosts();   // 再等 1s（其实它根本不依赖 user）
  return { user, posts };           // 总共 ≈ 2s ❌
}
正例：Promise.all 并发（总耗时 = 最大值）
async function fast() {
  // 两个请求几乎同时发出，再一起等
  const [user, posts] = await Promise.all([
    fetchUser(),
    fetchPosts(),
  ]);
  return { user, posts };           // 总共 ≈ 1s ✅
}
选 all 还是 allSettled？Promise.all 有一个失败就整体 reject（适合"全要成功"）；Promise.allSettled 等所有都结束、各自带成功/失败状态（适合"能拿多少算多少"）。

循环中的 await：当心串行与失效

在 for 循环里写 await 是串行的——这有时是你要的（需要顺序执行），但常常是无意中写慢了。更隐蔽的坑是 forEach：它的回调是普通函数，里面的 await 根本拦不住外层，forEach 会立刻返回，循环"看起来在等"实际没等。

forEach 里的 await 是无效的
// ❌ forEach 不会等待，"全部完成"会先于任何一个打印
ids.forEach(async id => {
  const data = await fetch(id);
  console.log(data);
});
console.log('全部完成');  // 这行会最先执行！

// ✅ 需要顺序：用 for...of，await 真正生效
for (const id of ids) {
  const data = await fetch(id);  // 一个接一个
  console.log(data);
}

// ✅ 不要求顺序、想要快：map 出 Promise 数组再 all
const results = await Promise.all(ids.map(id => fetch(id)));

for await...of：遍历异步迭代

当数据源是异步可迭代对象（如分页流、可读流），for await...of 会在每次迭代自动 await 取出下一项，让你像遍历普通数组一样处理一段段陆续到达的数据。

for await...of 逐块处理异步数据流
async function *pages() {           // 异步生成器：每页是一个 Promise
  yield await fetchPage(1);
  yield await fetchPage(2);
}

for await (const page of pages()) {  // 每轮自动等待下一页
  console.log(page.length);
}

常见陷阱速查

陷阱后果 / 说明
忘记写 await拿到的是 Promise 对象而非结果值，后续 data.xxx 全是 undefined
循环里 await默认串行，独立任务会拖慢；该并发时用 Promise.all
forEach 里 await完全无效，外层不会等待（用 for...of 或 map + all）
await 非 Promise会被包装成 resolved Promise，多走一个微任务 tick
漏 await 被 try/catchreturn fetchX() 不加 await，错误逃出 try（除非直接 return）
顶层 await 在普通脚本报语法错误，仅 ES 模块顶层可用
未处理的 rejectedasync 抛错变 rejected Promise，无人 catch 会触发未处理拒绝警告
最隐蔽的一个：忘记 await。例如 const data = fetchUser(); 漏了 await，data 是个 Promise，data.name 自然是 undefined，且不会报错——非常难排查。开 ESLint 的 require-await / no-floating-promises 规则能帮你提前发现。
小结：① async/await 是 Promise 的语法糖，async 函数总是返回 Promise；② await 暂停函数、取出结果值，只能在 async 内用（ESM 顶层例外）；③ 错误处理用 try/catch，比 .catch 更自然；④ 独立任务别串行 await，用 Promise.all 并发；⑤ forEach 里的 await 无效，顺序遍历用 for...of、异步流用 for await...of。
