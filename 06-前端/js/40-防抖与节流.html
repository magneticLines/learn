防抖与节流

防抖（debounce）和节流（throttle）是前端性能优化的两把手术刀。它们都用来"压制"高频触发的事件，让昂贵的逻辑别被疯狂调用——但压制的策略截然不同。搞清两者的区别与适用场景，是性能优化的基本功，也是面试常客。

它解决了什么问题

浏览器里有一批事件天生高频触发：滚动 scroll、窗口缩放 resize、鼠标移动 mousemove、输入框 input……（呼应第 27、36 篇的事件机制）。这些事件在用户操作的几秒里能触发几十上百次。如果每次回调都做昂贵的活——发网络请求、操作 DOM 引发重排（reflow）、复杂计算——主线程会被瞬间塞满，页面卡顿、掉帧甚至假死。

问题的本质是：事件触发的频率，远高于我们真正需要执行逻辑的频率。用户在搜索框连打十个字，我们并不需要为每个字符都发一次请求；用户疯狂滚动页面，我们也不必每一像素都重新计算布局。防抖和节流就是两种"把触发频率削下来"的通用手段。

防抖 debounce：等"安静"了才执行

防抖的策略是：把短时间内的连续触发合并成一次，只有当事件停止触发、安静了指定的时间，才执行最后一次。每来一次新触发，就把之前排好的执行计划取消、重新计时。

生活类比：电梯关门。电梯准备关门（计时），但只要有人进来（新触发），就重新等待几秒；直到没人再进来、安静了，门才真正关上（执行）。再比如游戏里"长按蓄力，松手才放招"。

典型场景：① 搜索框联想——用户停止输入后才发请求，避免每个字符一次请求；② 窗口 resize 完成后再重新计算布局；③ 表单的实时校验——停止输入再校验，体验更顺。它们的共同点是：只关心"最终状态"，中间过程可以丢弃。

实现的核心是闭包（呼应第 06 篇）：用一个闭包变量保存定时器 id，每次调用先 clearTimeout 掉上一个计划，再 setTimeout 重新排一个。只有最后一次的定时器能"活到"触发的那一刻。

debounce 基础实现
function debounce(fn, delay) {
  let timer = null;           // 闭包变量：保存定时器 id，跨多次调用复用
  return function (...args) {   // 用普通函数，保留调用时的 this 与参数
    clearTimeout(timer);       // 每次新触发，先取消上一个排好的计划
    timer = setTimeout(() => {
      fn.apply(this, args);   // 箭头函数继承外层 this，再透传 args
    }, delay);
  };
}

// 用法：停止输入 300ms 后才真正发请求
const onSearch = debounce((e) => sendRequest(e.target.value), 300);
input.addEventListener('input', onSearch);

防抖的 leading（立即执行）选项

上面的版本是尾部执行（trailing）：安静后才跑。有时我们希望第一次触发立刻执行（如点赞按钮立即响应），之后的连续触发再被压制——这叫 leading（前缘）防抖。常用一个 immediate 开关来切换。

带 immediate 的 debounce
function debounce(fn, delay, immediate = false) {
  let timer = null;
  return function (...args) {
    const callNow = immediate && !timer;  // 没有计时中且要求立即，则本次先执行
    clearTimeout(timer);
    timer = setTimeout(() => {
      timer = null;                  // 冷却结束，恢复"可立即执行"状态
      if (!immediate) fn.apply(this, args);
    }, delay);
    if (callNow) fn.apply(this, args);
  };
}

节流 throttle：固定频率执行

节流的策略是：在固定的时间间隔内，最多执行一次。无论事件触发多频繁，每隔 N 毫秒才放行一次——保证执行有稳定的节奏。

生活类比：水龙头限流，开多大都只流那么多；或游戏技能的 CD（冷却），放完技能要等冷却好才能再放，期间狂点也没用。

典型场景：① 滚动加载 / 滚动监听——每隔一段固定时间判断一次是否到底；② 按钮防连点——限制提交频率；③ mousemove 跟随——固定节奏更新坐标，画面平滑又不卡。它们的共同点是：过程中需要持续有反馈，但不必每次触发都跑。

实现方式一：时间戳（leading，首次立即）

时间戳版 throttle
function throttle(fn, interval) {
  let last = 0;             // 闭包变量：记录上次执行的时间戳
  return function (...args) {
    const now = Date.now();
    if (now - last >= interval) {  // 距上次已超过间隔，才放行
      last = now;
      fn.apply(this, args);
    }
  };
}

实现方式二：定时器（trailing，尾部补一次）

时间戳版首次立即执行、但最后一次可能丢失；定时器版则首次延迟、能保证尾部那次。想首尾都执行，可把两者结合（lodash 的 throttle 就提供 leading / trailing 两个选项）。

定时器版 throttle
function throttle(fn, interval) {
  let timer = null;
  return function (...args) {
    if (timer) return;          // 冷却中，直接忽略本次触发
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;             // 执行完清空，开启下一轮冷却
    }, interval);
  };
}

核心区别对比

维度防抖 debounce节流 throttle
触发时机停止触发、安静 N 毫秒后每隔固定 N 毫秒到点就执行
执行频率连续触发只执行一次（最后一次）固定频率，匀速执行多次
关注点只要"最终结果"过程中要"持续反馈"
典型场景搜索联想、resize、表单校验滚动加载、按钮防连点、mousemove
类比电梯关门 / 蓄力放招水龙头限流 / 技能 CD
一句话口诀：防抖看"最后一下"，节流看"固定频率"。 拿不准时问自己——只关心用户操作完的最终态，用防抖；操作过程中需要稳定地持续反馈，用节流。

实现里的陷阱

  • this 与参数传递：返回的包装函数务必用普通函数（不是箭头函数），这样它才能拿到事件触发时正确的 this（如绑定的 DOM 元素）；再用 fn.apply(this, args) 把 this 和参数完整透传给原函数。若用箭头函数包装，this 会被锁死成定义时的外层，事件回调里就拿不到 DOM 了。
  • 闭包保存状态：timer / last 必须是闭包里的变量，由返回的函数跨多次调用共享。若误把它写进函数体内部，每次调用都重置，防抖节流就彻底失效了。
  • 记得清理定时器：组件卸载、元素移除时，要 clearTimeout 掉挂起的定时器并移除事件监听，否则可能在已销毁的上下文里执行、引发报错或内存泄漏（呼应事件解绑那篇）。
  • leading / trailing 取舍：防抖默认尾部执行，加 immediate 可前缘立即；节流时间戳版偏首、定时器版偏尾。要"首次响应快又不丢最后一次"，得把两套机制组合，这正是用 lodash 现成实现的理由。
生产建议：真实项目优先用 lodash.debounce / lodash.throttle，它们处理好了 leading、trailing、maxWait、cancel() 等边界。手写实现主要用于理解原理和面试——能讲清闭包、this、定时器三件事，基本就过关了。
小结：① 高频事件（scroll/resize/mousemove/input）直接绑昂贵逻辑会卡顿，防抖与节流都是"削频率"的手段；② 防抖＝等安静了执行最后一次（电梯门），适合搜索联想、resize；③ 节流＝固定间隔执行一次（技能 CD），适合滚动、防连点、mousemove；④ 口诀：防抖看"最后一下"，节流看"固定频率"；⑤ 实现三要点——普通函数 + apply 透传 this/args、闭包存定时器、记得清理。
