类型转换与判断（typeof / == vs ===）

JS 是弱类型语言，类型会在你不注意时悄悄变身。理解何时显式转、何时隐式转，以及如何正确判断类型，是绕开 "1" + 1 这类经典陷阱、写出可预测代码的关键一课。

它解决了什么问题

JS 设计之初为了"对新手友好"，允许不同类型的值直接做运算和比较——它会自动帮你转换。听起来贴心，实际埋下了大量隐患："5" - 1 得 4，"5" + 1 却得 "51"，[] == ![] 居然是 true。这套隐式转换规则被称为强制类型转换（coercion）。我们要做的不是去背所有诡异结果，而是理解转换发生的时机与方向，并用显式转换 + === 把不确定性挡在门外。

显式转换：你主动发起的

显式转换是意图清晰的写法，读代码的人一眼就知道你在做什么。三个全局函数对应三种目标类型。

Number / String / Boolean
Number("42")        // 42
Number("42px")      // NaN —— 整个字符串必须是合法数字，否则失败
Number("")          // 0  —— 空字符串转成 0（坑！）
Number(true)        // 1
Number(null)        // 0
Number(undefined)   // NaN  —— 注意 null 和 undefined 结果不同

String(42)          // "42"
String(null)        // "null"（而非空串）

Boolean(0)          // false
Boolean("hello")    // true

parseInt / parseFloat 与 Number() 的区别在于：它们"能读多少读多少"，遇到非数字字符就停下，而不是直接返回 NaN。解析 CSS 的 "16px" 时尤其常用。

parseInt vs Number
parseInt("42px")     // 42   —— 读到 px 停下
parseInt("px42")     // NaN  —— 开头就不是数字
parseInt("0xFF", 16) // 255  —— 第二个参数是进制，建议永远显式传！
parseFloat("3.14m") // 3.14
陷阱：parseInt 不传进制。老引擎里 parseInt("08") 可能被当成八进制解析出 0。养成习惯第二个参数永远写 10，避免被历史行为坑到。

隐式转换：JS 偷偷帮你转

隐式转换的核心要记住两条规则，几乎能覆盖所有场景：

  • + 运算符是双重身份：只要有一个操作数是字符串，就变成字符串拼接，把另一边也转成字符串。
  • - * / % 没有拼接含义，一律把两边转成数字再运算。
+ 的陷阱：拼接 vs 相加
1 + 2            // 3   —— 都是数字，正常相加
"1" + 2          // "12" —— 有字符串，2 被转成 "2" 拼接
1 + "2" + 3      // "123" —— 从左到右：1+"2"="12"，再 "12"+3="123"
1 + 2 + "3"      // "33" —— 先 1+2=3，再 3+"3"="33"（顺序很关键！）

"5" - 1          // 4   —— 减法没有拼接含义，"5" 转成数字 5
"5" * "2"        // 10  —— 两个字符串都转数字
true + true      // 2   —— true 转成 1
面试高频：问你 [] + {} 和 {} + [] 的结果。前者是 "[object Object]"（数组和对象都先转成字符串再拼接）。记住"对象转字符串会调用 toString()"这条主线，比死记结果更靠谱。

falsy 值：布尔上下文里的"假"

在 if、&&、||、三元等需要布尔值的地方，JS 会隐式调用 Boolean()。结果为 false 的值称为 falsy，它们的数量固定就这 6 个，背下来即可，其余一切（包括 "0"、[]、{}）都是 truthy。

falsy 值说明
false布尔假本身
0 / -0 / 0n数字零（含 bigint 零）
""空字符串（注意 " " 有空格的是 truthy）
null空值
undefined未定义
NaN非数字
陷阱：空数组和空对象是 truthy。if ([]) 和 if ({}) 都会进入分支。想判断数组是否为空，要用 arr.length === 0，不能直接 if (!arr)。

== 与 ===：是否做类型转换

这是 JS 最经典的取舍。===（严格相等）不做任何类型转换：类型不同直接判 false。==（宽松相等）会先把两边转成同一类型再比较，规则繁琐且反直觉。

== 的主要转换规则（简化版）：

  • 类型相同 → 直接按 === 比较。
  • 一边 number 一边 string → 把字符串转成数字。
  • 有一边是 boolean → 先把 boolean 转成数字（true→1，false→0）。
  • 一边是对象，另一边是原始值 → 把对象转成原始值（调 valueOf / toString）。
  • null == undefined → true，且它俩不会再转换成别的类型。
== 把人坑哭的结果
1 == "1"           // true  —— "1" 转成数字 1
0 == false         // true  —— false 转成 0
0 == ""            // true  —— "" 转成 0
null == undefined  // true  —— 特例，二者宽松相等
null == 0           // false —— null 不会转成数字！别想当然
NaN == NaN           // false —— NaN 不等于任何值，包括自己

// 全部换成 === ，类型不同直接 false，符合直觉：
1 === "1"          // false
0 === false        // false
null === undefined // false（类型就不同）
烧脑名题：[] == ![] 为什么是 true？拆开看：![] 先算——空数组是 truthy，取反得 false。于是变成 [] == false；false 转成 0，[] 转成原始值是空串 ""，"" 再转数字是 0，0 == 0 成立。一连串隐式转换叠加，结果完全脱离直觉——这正是要远离 == 的最好理由。

typeof / instanceof / Array.isArray

三种判断手段适用场景不同，选错就会踩坑。

方式擅长短板
typeof判断原始类型和函数（返回字符串）typeof null 是 "object"；数组、对象、null 都返回 "object"，无法区分
instanceof判断对象是否属于某个构造函数/类的实例（看原型链）对原始值无效；跨 iframe / 跨 window 时原型链不同会误判
Array.isArray专门、可靠地判断数组只能判数组
各取所长
typeof 42            // "number"
typeof "hi"          // "string"
typeof function(){} // "function"
typeof []            // "object"  —— 区分不出数组！
typeof null          // "object"  —— 历史 bug

[] instanceof Array   // true，但跨 iframe 会失效
Array.isArray([])     // true  —— 判数组的标准答案
对比小结：判原始类型用 typeof；判 null 直接 x === null；判数组用 Array.isArray；判自定义类实例用 instanceof。想精确拿到"内部类型标签"，还可以用 Object.prototype.toString.call(x)，对 [] 返回 "[object Array]"，对 null 返回 "[object Null]"。

NaN 的判断：别用 ==

NaN 是 JS 里唯一不等于自身的值，所以 x === NaN 永远是 false，根本判不出来。早年的全局 isNaN() 又会先做隐式转换，把 "abc" 也判成 true，并不可靠。正解是 ES6 的 Number.isNaN()，它不转换类型，只对真正的 NaN 返回 true。

判断 NaN 的正确姿势
NaN === NaN            // false —— 判不出来

isNaN("abc")          // true  —— 反例！"abc" 先转成 NaN 才判，误伤
isNaN(NaN)             // true

Number.isNaN("abc")   // false —— 正例！不转换，"abc" 不是 NaN
Number.isNaN(NaN)      // true  —— 只认真正的 NaN
小结：① 显式转换用 Number/String/Boolean，意图清晰；parseInt 记得传进制；② + 遇字符串变拼接，- * / 一律转数字；③ falsy 只有 false/0/""/null/undefined/NaN 六个，空数组空对象是 truthy；④ 永远用 ===，== 的隐式转换会制造 [] == ![] 这类怪结果；⑤ 判类型分场景：typeof 判原始、Array.isArray 判数组、instanceof 判实例、x === null 判 null；⑥ 判 NaN 必须用 Number.isNaN。
