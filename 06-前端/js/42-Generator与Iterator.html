Generator 与 Iterator

为什么数组能 for...of、能展开 ...、能解构，而普通对象不行？答案藏在一套"迭代协议"里。Generator 则是 JS 给你的一把"可暂停的函数"瑞士军刀——它既是迭代器的最佳生产工具，也是 async/await 的前身。这两者，是把前面所学串成一条线的收官知识。

它解决了什么问题

ES6 之前，遍历不同集合的方式各不相同：数组用 for 索引、对象用 for...in、类数组（如 arguments、DOM 节点列表）还得手动转换。JS 缺一套统一的"如何逐个取值"的约定。ES6 用两个协议把这件事标准化了：迭代器协议（Iterator）规定"怎么取下一个"，可迭代协议（Iterable）规定"怎么拿到一个迭代器"。一旦遵守，对象就能被 for...of、展开运算符、解构、Array.from 等所有"消费方"统一处理——这正是数组、字符串、Map、Set 天生可迭代的原因（呼应 04 / 08 篇）。而 Generator，则让你不用手写这套协议，写几行就能造出一个迭代器。

迭代器协议（Iterator）：怎么取下一个

一个对象只要有一个 next() 方法，且每次调用返回形如 { value, done } 的对象，它就是迭代器。value 是本次产出的值，done 是布尔值，表示"是否已经取完"。它本质就是一个带状态的指针：每调一次 next()，指针前移一格。

手写一个最朴素的迭代器
const it = {
  cur: 0,
  next() {
    return this.cur < 3
      ? { value: this.cur++, done: false }
      : { value: undefined, done: true };
  }
};
console.log(it.next());  // { value: 0, done: false }
console.log(it.next());  // { value: 1, done: false }

可迭代协议（Iterable）：怎么拿到迭代器

光有迭代器还不够。for...of 拿到的是一个集合，它怎么知道该向谁要迭代器？答案是：集合要实现一个名为 [Symbol.iterator] 的方法（用 Symbol 当键，避免和普通属性冲突——呼应 09 篇 Symbol），调用它返回一个迭代器。实现了 [Symbol.iterator]() 的对象，就叫"可迭代对象"。这就是数组、字符串、Map、Set 能被 for...of 遍历的全部秘密——它们的原型上都内置了这个方法。

让普通对象变得可迭代
const range = {
  from: 1,
  to: 3,
  // 计算属性名：键是 Symbol.iterator，值是一个返回迭代器的函数
  [Symbol.iterator]() {
    let cur = this.from, last = this.to;
    return {
      next() {
        return cur <= last
          ? { value: cur++, done: false }
          : { value: undefined, done: true };
      }
    };
  }
};

for (const n of range) console.log(n);  // 1 2 3
console.log([...range]);            // [1, 2, 3] —— 展开运算符也认这套协议
const [a, b] = range;           // 解构同理：a=1, b=2
两个协议的分工：Iterable 回答"找谁要"（[Symbol.iterator]()），Iterator 回答"怎么取"（next()）。for...of 内部做的事就是：先调 [Symbol.iterator]() 拿到迭代器，再循环调 next() 直到 done: true。把这条链记牢，所有"可迭代"现象都不再神秘。

Generator 函数：可暂停的函数

手写迭代器又是闭包又是状态，啰嗦易错。Generator 把这套活儿全包了。它的写法是 function*（function 后加星号）配 yield 关键字。最反直觉、也最关键的一点：调用 Generator 函数并不执行函数体，而是立即返回一个生成器对象。函数体的代码要等你调 next() 才一步步执行，执行到 yield 就暂停并把 yield 后的值作为 value 吐出来，下次 next() 再从暂停处恢复。

这正是它与普通函数的本质区别：普通函数一旦调用就一口气跑到 return，中途无法停；Generator 则像一台可以反复"暂停 / 续播"的播放器，把函数的执行权交还给调用者。

function* 与 yield 的暂停-恢复
function* gen() {
  console.log('开始');
  yield 1;            // 暂停点 ①，吐出 1
  console.log('中间');
  yield 2;            // 暂停点 ②，吐出 2
  return 99;          // 结束，done 变 true
}

const g = gen();     // 注意：此时"开始"还没打印！函数体尚未执行
console.log(g.next()); // 打印"开始"，再返回 { value: 1, done: false }
console.log(g.next()); // 打印"中间"，再返回 { value: 2, done: false }
console.log(g.next()); // { value: 99, done: true } —— return 的值出现在这里

生成器对象同时满足两个协议：它有 next()（是迭代器），它的 [Symbol.iterator]() 又返回自身（是可迭代对象）。所以可以直接 for...of 它，或用展开运算符消费——再也不用手写那一堆状态代码了。

进阶用法：next(arg) 传值、yield* 委托

yield 不只是单向吐值，它还能双向通信：next(arg) 传入的实参，会成为上一个 yield 表达式的返回值。这让调用者能反过来"喂数据"给暂停中的函数，是 Generator 实现异步流程控制的核心机制。

next 传值 与 yield* 委托
function* dialog() {
  const name = yield '你叫什么？';  // yield 的"值"由下次 next(arg) 注入
  yield `你好，${name}`;
}
const d = dialog();
d.next();            // 第一次 next 启动，无参数；返回 '你叫什么？'
d.next('小明');      // '小明' 成为上面 yield 的返回值 → name

// yield* 委托：把遍历权转交给另一个可迭代对象
function* inner() { yield 1; yield 2; }
function* outer() {
  yield* inner();   // 等价于把 inner 产出的值逐个 yield 出来
  yield 3;
}
console.log([...outer()]);  // [1, 2, 3]

典型应用：惰性求值与无限序列

因为 Generator 只在 next() 时才算下一个值，它天生适合惰性求值——可以表达"无限"的序列而不会卡死，要多少取多少。自增 ID 生成器就是经典例子。

无限自增 ID 生成器
function* idMaker() {
  let id = 1;
  while (true) yield id++;  // 无限循环不会卡死：每次 next 只走一步
}
const ids = idMaker();
console.log(ids.next().value);  // 1
console.log(ids.next().value);  // 2 …… 按需取值

其他常见场景：自定义可迭代（用 Generator 实现 [Symbol.iterator]，比手写迭代器干净得多）、状态机（每个 yield 是一个状态）、以及历史上的异步流程控制——在 async/await 出现前，社区用 Generator + Promise（如 co 库）模拟"同步写法的异步代码"，yield 一个 Promise、等它 resolve 后再 next 回来。这正是 async/await 的思想原型（呼应 31 篇）：async 函数本质就是被引擎自动驱动的 Generator。

用 Generator 简化自定义可迭代
const range = {
  from: 1, to: 3,
  *[Symbol.iterator]() {   // 直接用 generator 方法，替代前面那一大坨
    for (let i = this.from; i <= this.to; i++) yield i;
  }
};
console.log([...range]);  // [1, 2, 3]

四个高频陷阱

陷阱说明
不调 next() 就不执行调用 gen() 只是拿到生成器对象，函数体一行都没跑。误以为"调用即执行"会导致副作用（如 console.log）迟迟不发生。
for...of 不消费 return 值for...of 在 done: true 那一轮就停止，不会把 return 的值纳入遍历。所以 [...gen()] 里看不到 return 99 的 99，只有各个 yield 的值。
Symbol.iterator 写错必须是 [Symbol.iterator]（带方括号的计算属性，且是 Symbol），写成字符串 'Symbol.iterator' 或拼成 Symbol.iterater 都会让对象不可迭代，for...of 直接报 is not iterable。
生成器只能遍历一次生成器对象是一次性的迭代器：跑完后状态停在 done: true，再 for...of 它将得到空结果。要重复遍历，应每次重新调用 Generator 函数拿新对象（这也是 [Symbol.iterator]() 要返回新迭代器的原因）。
易错对比：把"可迭代对象"和"迭代器"搞混。数组是可迭代的但本身不是迭代器（它没有 next()，要先 arr[Symbol.iterator]() 才拿到迭代器）；而生成器对象两者皆是。区分清楚，才能理解为什么数组能反复 for...of、而生成器只能遍历一次。
小结（JS 进阶篇收官）：① Iterator = 有 next() 返回 {value, done} 的指针；Iterable = 实现 [Symbol.iterator]() 返回迭代器，于是能被 for...of / 展开 / 解构统一消费；② 数组、字符串、Map、Set 天生可迭代，普通对象不是；③ Generator（function* + yield）是"可暂停的函数"，调用返回的生成器对象既是迭代器又可迭代，是造迭代器的利器；④ next(arg) 双向传值、yield* 委托，支撑了惰性序列、状态机，并孕育了 async/await。至此 JS 进阶篇收官——闭包给了它记忆、迭代协议给了它统一的遍历语言、Promise 与 Generator 共同点亮了异步，三者交汇于此，正是 JS 语言设计之美的缩影。
