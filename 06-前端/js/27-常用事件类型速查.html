常用事件类型速查

事件是浏览器与你的代码对话的语言——用户点了、敲了、滚了、改了，浏览器都会"喊一声"。事件名繁多，但常用的就那么几类。这一篇把它们按场景归类，重点讲清几对最容易混淆的"双胞胎"。

它解决了什么问题

网页是事件驱动的：代码不会主动轮询"用户有没有点按钮"，而是注册回调、等浏览器通知。这套机制（addEventListener + 事件对象）让我们能响应用户的一举一动。问题在于事件名实在太多，而且不少长得像、行为却不同——比如 mouseenter 和 mouseover、input 和 change、DOMContentLoaded 和 load。选错事件，要么回调触发太频繁拖垮性能，要么该触发时不触发。这篇速查就是帮你按场景快速选对事件，并避开几个经典坑。

鼠标事件

鼠标交互最常用。注意 event.button（0 左键 / 1 中键 / 2 右键）和 clientX/clientY（视口坐标）等属性都挂在事件对象上。

事件触发时机冒泡
click按下并在同一元素松开（单击）是
dblclick双击是
mousedown按下鼠标键的瞬间是
mouseup松开鼠标键的瞬间是
mousemove鼠标在元素上移动（高频）是
mouseenter指针进入元素否
mouseleave指针离开元素否
mouseover指针进入元素或其子元素是
mouseout指针离开元素或其子元素是

重点对比：mouseenter/leave vs mouseover/out

这是最容易踩的坑。mouseenter/mouseleave 不冒泡，只关心指针有没有进出元素自身的边界——指针在内部的子元素间移动，它不会重复触发。而 mouseover/mouseout 会冒泡，指针每进出一个子元素都会触发一次。做"悬浮高亮整块区域"时，mouseenter 几乎总是更省心的选择。

同一个带子元素的盒子，两种事件的行为天差地别
// box 里有若干子元素 <span>...</span>
const box = document.querySelector('.box');

// mouseenter：进入 box 只触发 1 次，在内部子元素间移动不再触发
box.addEventListener('mouseenter', () => console.log('enter'));

// mouseover：进入 box、再进入每个子 span 都会触发 —— 可能噼里啪啦响很多次
box.addEventListener('mouseover', (e) => console.log('over', e.target));
为什么有这种区别？因为 mouseover/mouseout 配合冒泡，正是事件委托（见上一篇）的基础——你想知道指针落在哪个子元素上，就需要它冒泡上来。而 mouseenter/mouseleave 是为"只关心自己整体进出"的场景专门设计的，所以刻意不冒泡。

键盘事件

键盘事件绑在能获得焦点的元素上（如 input），或绑在 document 上做全局快捷键。核心是用 event.key 读按键名。

事件触发时机常用属性
keydown按键按下（长按会重复触发）e.key e.code
keyup按键松开e.key e.code
keypress ⚠️已废弃，不要再用—
读按键名，做快捷键
document.addEventListener('keydown', (e) => {
  if (e.key === 'Enter') submit();          // e.key 是语义化键名："Enter" "Escape" "a"
  if (e.ctrlKey && e.key === 's') {
    e.preventDefault();                  // 拦掉浏览器默认的"保存网页"
    save();
  }
});
陷阱：event.keyCode 已废弃。老代码里常见 e.keyCode === 13 判回车，这种数字魔法值既不可读、又被标准淘汰。一律改用 e.key（语义键名，受输入法/布局影响）或 e.code（物理按键位，如 "KeyA"，不受布局影响）。

表单事件

表单交互的核心矛盾是"什么时候算用户改完了"，于是有了 input 和 change 两种粒度。

事件触发时机冒泡
input值每次变化就触发（实时）是
change值改变且失去焦点后才触发是
submit表单提交（绑在 <form> 上）是
focus元素获得焦点否
blur元素失去焦点否
focusin获得焦点（冒泡版）是
focusout失去焦点（冒泡版）是

重点对比：input vs change

input 在用户每敲一个字符时就触发，适合做实时搜索、字数统计、实时校验。change 要等到值真的变了并且失焦才触发，适合"用户改完了再统一处理"的场景。

实时 vs 完成态
const inp = document.querySelector('input');

inp.addEventListener('input', (e) => {
  console.log('每个字符都触发', e.target.value);  // 实时搜索建议
});

inp.addEventListener('change', (e) => {
  console.log('失焦后才触发一次', e.target.value);  // 提交校验
});
陷阱：change 在不同控件上时机不同。文本框是"失焦才触发"；但 <select> 下拉、checkbox/radio 是一选中就立刻触发，因为它们没有"边输入边改"的过程。所以监听复选框、单选框、下拉框的选择，用 change 反而比 input 更合适也更直观。
focus/blur 不冒泡，所以无法用事件委托统一监听一堆输入框的聚焦。需要委托时改用冒泡版 focusin/focusout——这正是它们存在的理由。

窗口 / 文档事件

页面生命周期与窗口变化相关的事件，通常绑在 window 或 document 上。

事件触发时机绑定对象
DOMContentLoadedHTML 解析完、DOM 树构建好（不等图片/样式）document
load页面全部资源（图片、CSS、iframe）都加载完window
resize窗口尺寸变化（高频）window
scroll页面或元素滚动（高频）window / 元素
beforeunload页面即将关闭（可弹"未保存"提示）window

重点对比：DOMContentLoaded vs load

两者都是"页面准备好了"，但准备的程度不同。DOMContentLoaded 在DOM 解析完毕时就触发——此时你能安全地操作元素、绑事件，但图片可能还在下载。load 要等到所有外部资源（含图片）都下载完才触发，明显更晚。绝大多数初始化代码（找元素、绑事件）应该用 DOMContentLoaded，触发更早、用户感知更快；只有当你确实依赖图片实际尺寸等资源信息时，才用 load。

初始化代码该挂在哪
// ✅ 推荐：DOM 就绪即可操作元素，不必等图片，初始化更早
document.addEventListener('DOMContentLoaded', () => {
  init();   // 此时 querySelector 一定能拿到元素
});

// 只有依赖图片真实宽高、或需要所有资源就位时，才等 load
window.addEventListener('load', () => {
  measureImages();
});
顺带一提：把 <script> 放在 <body> 末尾，或给它加 defer 属性，脚本执行时 DOM 已就绪，很多时候连 DOMContentLoaded 都不必再包一层。

剪贴板 / 触摸事件（简述）

移动端和复制粘贴场景偶尔会用到，了解即可。

  • 剪贴板：copy / cut / paste，事件对象上的 e.clipboardData 可读写剪贴板内容（如拦截粘贴、自定义复制文本）。
  • 触摸：touchstart / touchmove / touchend，e.touches 是当前所有触点列表，可做多指手势。
  • 统一指针：pointerdown / pointermove / pointerup 把鼠标、触摸、手写笔统一成一套 API，新项目优先考虑。

陷阱：高频事件务必配合防抖/节流

scroll、mousemove、resize、input 这类事件在短时间内可能触发几十上百次。如果回调里做了重计算、重渲染或发请求，页面会明显卡顿。标准做法是用防抖（debounce）或节流（throttle）压低触发频率（详见第 40 篇）。

滚动监听不加节流的代价
// 反例：每次滚动都同步算布局，滚动越快卡得越狠
window.addEventListener('scroll', () => heavyLayoutCalc());

// 正例：节流后每 100ms 最多算一次（throttle 见第 40 篇）
window.addEventListener('scroll', throttle(heavyLayoutCalc, 100));
小结：① 鼠标 mouseenter/leave 不冒泡、只看自身进出，mouseover/out 冒泡、子元素也触发；② 键盘用 e.key 取键名，keyCode 已废弃；③ 表单 input 实时、change 失焦才触发（但选择类控件一选就触发），focus/blur 不冒泡、要委托用 focusin/focusout；④ DOMContentLoaded（DOM 就绪，更早）vs load（含图片等资源全加载完）；⑤ scroll/mousemove/resize 等高频事件务必配防抖节流。

与其他知识点的联系

  • 事件冒泡与捕获、事件委托：本篇每一处"冒泡"列都是前两篇结论的具体落点——正因为 mouseover/out、input/change、focusin/focusout 会冒泡，它们才能被委托到父元素统一监听；而 mouseenter/leave、focus/blur 不冒泡，就是委托篇里"陷阱二"点名的反例。
  • addEventListener 与事件对象：本篇表格里的 e.key、e.button、e.clipboardData、e.touches 都是第一篇讲的"事件对象"上挂的具体成员，速查表相当于把那篇的通用概念铺开到每种事件的实际取值上。
  • 实战场景——表单校验：选 input 还是 change 直接决定校验体验：实时校验（如密码强度提示）用 input，提交前的最终校验用 change 或 submit，选错事件是表单校验最常见的返工原因。
  • 实战场景——移动端触摸：touchstart/move/end 和统一指针事件 pointerdown/move/up 是移动端手势、拖拽组件的基础；新项目直接用 pointer 系列可以少写一套鼠标/触摸兼容代码。
