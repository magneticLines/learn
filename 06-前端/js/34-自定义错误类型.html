自定义错误类型

内置的 Error 只能告诉你"出错了"，却说不清"是哪一类错、出在哪个字段、该返回什么状态码"。自定义错误类型让你给异常贴上精确的标签，把"模糊的报错"变成"可被代码分类处理的信号"。

它解决了什么问题

设想一个表单提交流程：可能因为字段校验失败，也可能因为网络请求 404，还可能因为代码本身的逻辑 bug。如果全都抛一个笼统的 Error("出错了")，catch 里就只能靠字符串匹配 err.message 来猜是哪种情况——脆弱、易碎、无法携带上下文。

自定义错误类型解决三件事：① 用 instanceof 精确区分错误种类，让 catch 分类处理；② 在错误对象上携带额外上下文（如 statusCode、出错的字段名）；③ 保留完整的调用堆栈，方便定位。这正是后端框架（Spring 的 BusinessException、Java 的受检异常体系）里"按异常类型分门别类捕获"思路在前端的对应物。

内置错误类型一览

JS 早就内置了一套错误类型，它们都继承自 Error。引擎在不同场景下会自动抛出对应类型——认识它们，是设计自己错误体系的前提。

类型何时被抛出例子
Error所有错误的基类，也可直接 throw new Error()throw new Error("自定义信息")
TypeError值的类型不对，或对 null/undefined 取属性null.foo / (123)()
RangeError数值超出合法范围new Array(-1)
SyntaxError代码或 JSON 解析时语法非法JSON.parse("{bad}")
ReferenceError访问未声明的变量（含 TDZ）console.log(notDefined)
每个 Error 实例都带三个核心属性：name（错误类型名，用于显示）、message（你传入的描述）、stack（调用堆栈字符串，由引擎自动生成）。自定义错误的目标，就是在保留这三者的前提下扩展出更多语义。

用 class extends Error 创建

ES6 的 class（见第 15 篇）让继承内置类型变得自然。核心模板只有几行，但每一行都有讲究：

自定义错误的标准写法
class ValidationError extends Error {
  constructor(message, field) {
    super(message);            // 必须！把 message 交给父类，否则 this.message 为空
    this.name = "ValidationError";  // 覆盖默认的 "Error"，让打印/日志显示真实类型
    this.field = field;     // 附加上下文：出错的字段名
  }
}

const e = new ValidationError("邮箱格式不对", "email");
console.log(e.name, e.message, e.field); // ValidationError 邮箱格式不对 email
console.log(e instanceof Error);          // true，仍是 Error，可被通用兜底捕获

为什么这么写：super(message) 调用父类 Error 的构造器，由它去设置 message 并生成 stack；this.name 默认是 "Error"，不改的话日志里看不出是哪种错；自定义属性（这里的 field）则把"出错时的现场信息"随错误一起带走。

catch 里按类型分类处理

有了类型，catch 就能像后端的多 catch 块一样，用 instanceof 把不同错误引向不同的处理逻辑——校验错误提示用户，网络错误重试，其余的当作未知 bug 上报。

ValidationError 与 HttpError 的分类处理
class HttpError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.name = "HttpError";
    this.statusCode = statusCode; // 携带 HTTP 状态码
  }
}

function submit(email) {
  if (!email.includes("@"))
    throw new ValidationError("邮箱格式不对", "email");
  throw new HttpError("用户不存在", 404);
}

try {
  submit("bad-email");
} catch (err) {
  if (err instanceof ValidationError)
    showFieldError(err.field, err.message); // 精准高亮出错字段
  else if (err instanceof HttpError && err.statusCode === 404)
    toast("资源不存在");
  else
    throw err;  // 不认识的错误，重新抛出，别吞掉
}
设计技巧：可以先建一个 AppError 基类继承 Error，再让 ValidationError、HttpError 都继承 AppError。这样 catch 里用 err instanceof AppError 就能一刀切分"我们自己抛的业务错误"和"未预料的系统错误"。

陷阱一：转译环境下 instanceof 失效

这是最隐蔽的坑。当代码用 Babel 等工具转译到 ES5（target 较低）时，class extends Error 内部依赖的原型链可能被破坏：new ValidationError() 出来的实例 instanceof ValidationError 竟然返回 false，分类处理全部失灵。

根因是 ES5 里 Error 构造器调用时会返回一个全新对象，丢掉了子类的原型。修复方法是在构造器里手动把原型链接回来：

修复转译丢失的原型链
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    // new.target.prototype 指向"实际被 new 的那个类"的原型
    // 兼容子类再被继承的情况，比写死 ValidationError.prototype 更稳
    Object.setPrototypeOf(this, new.target.prototype);
    this.name = "ValidationError";
    this.field = field;
  }
}
注意：只有当编译目标低于 ES2015、且未启用相应 helper 时才会触发。如果你的构建目标本就是现代浏览器（target: "es2017" 及以上），原生 class 不会有这个问题，可不必加。但写库代码不确定使用方环境时，加上这行更保险。

陷阱二：忘记 super 与 stack

  • 忘记 super(message)：构造器里如果先访问 this 会直接报错（子类必须先调 super 才有 this）；即便侥幸不报错，message 也会丢失、stack 不完整。第一行永远先 super(message)。
  • 关于 stack：stack 由引擎在 Error 构造时捕获，记录"错误诞生的位置"。所以一定要 throw new XxxError()，而不是复用一个老的错误对象——否则堆栈指向的是创建处，不是出错处。

对比：抛字符串 / 普通对象 vs 抛 Error 子类

抛出方式能 instanceof 分类有 stack 堆栈建议
throw "出错了"不能（是 string）无禁止
throw { code: 1 }不能（普通对象）无不推荐
throw new Error(...)仅能判断是 Error有可用
throw new XxxError(...)能精确分类有推荐

抛字符串或裸对象的最大代价是丢失堆栈：出了问题在控制台只看到一行字，定位不到是哪行代码抛的。而 Error 子类既保留堆栈，又能被 instanceof 分类，还能携带上下文——没有理由不用它。

小结：① 自定义错误 = class XxxError extends Error，构造器里第一行 super(message)，再设 this.name 和自定义属性；② 价值在于 instanceof 分类 + 携带上下文（statusCode/field）+ 保留 stack；③ 转译到 ES5 时可能 instanceof 失效，用 Object.setPrototypeOf(this, new.target.prototype) 修；④ 永远抛 Error 子类，别抛字符串或裸对象，否则丢堆栈、没法分类。

与其他知识点的联系

  • 关联 class 与继承：自定义错误类型完全建立在 ES6 class extends 语法之上，super(message) 的调用顺序、原型链的继承关系，都是 class 继承的标准套路——理解这里的坑，也就理解了继承本身的坑（比如子类构造器必须先调 super 才能用 this）。
  • 关联 try/catch/finally：自定义错误类型的价值只有在 catch 块里才能兑现——没有分类，catch 只能对着一个模糊的 err.message 猜；有了 instanceof 分类，catch 才能像后端多 catch 块一样精确分流处理。两者是"定义类型"与"消费类型"的一体两面。
  • 关联 Promise 与异步错误处理：无论是 async/await 里的 try/catch，还是 Promise.prototype.catch，捕获到的错误对象同样可以用 instanceof 做类型判断——自定义错误类型不局限于同步代码，异步链路里的错误分类同样受益。
  • 关联 JSON 序列化：把错误信息传给日志系统或前端展示时，常需要 JSON.stringify；但 Error 实例的 message、stack 是不可枚举属性，直接序列化会得到空对象 {}，必须手动挑出字段（如 { name: err.name, message: err.message, field: err.field }）才能正确传输，这是很多人踩过的坑。
  • 关联 React 错误边界（Error Boundary）：在 React 生态里，自定义错误类型可以配合 ErrorBoundary 做更精细的兜底——根据捕获到的错误类型（如是否是 ValidationError）决定展示不同的降级 UI，而不是统一的"出错了"页面。
