Intersection Observer

「元素是否进入了视口？」这个看似简单的问题，过去要靠监听滚动事件硬算，代价高得离谱。IntersectionObserver 把这件事交给浏览器异步处理——图片懒加载、无限滚动、曝光埋点，从此都有了优雅的标准答案。

它解决了什么问题

过去判断一个元素是否出现在视口里，标准做法是监听 scroll 事件，在回调里对每个目标元素调用 getBoundingClientRect() 算出它相对视口的位置。这套方案有三宗罪：① 高频触发——滚动一下能触发几十上百次回调；② 强制同步布局（reflow）——getBoundingClientRect() 会逼浏览器立刻重新计算布局，在滚动这种高频场景里极易掉帧卡顿；③ 全在主线程上跑，和渲染抢资源。

IntersectionObserver 换了个思路：你只需告诉浏览器「我关心这些元素和视口的相交关系」，剩下的由浏览器在底层异步地观察，相交状态发生变化时才回调通知你。它不阻塞主线程、不强制 reflow，天生高效。这正是「声明式」对「命令式」的胜利——你描述意图，浏览器决定何时、如何高效地完成。

基本用法

三步走：创建观察者（传入回调和配置）、用 observe() 注册要观察的元素、在回调里处理相交变化。

创建并启动观察
const observer = new IntersectionObserver((entries, obs) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      // entry.target 是当前正在相交的那个元素
      console.log('进入视口', entry.target);
    }
  });
}, {
  root: null,          // null = 相对浏览器视口
  rootMargin: '0px',  // 视口边界的外扩/内缩
  threshold: 0         // 相交比例达到多少时触发回调
});

const el = document.querySelector('.target');
observer.observe(el);      // 开始观察
// observer.unobserve(el);  停止观察某个元素
// observer.disconnect();   停止观察全部元素

回调收到的 entry 有哪些信息

回调的第一个参数是 entries 数组（注意：只包含相交状态发生变化的元素，不是全部被观察元素）。每个 entry 携带这些常用字段：

字段含义
isIntersecting布尔值，是否正在与 root 相交（最常用的判断依据）
intersectionRatio相交比例 0~1，元素露出多少（用于精细控制，如曝光面积埋点）
target触发本次变化的目标元素（多元素共用一个 observer 时靠它区分）
boundingClientRect目标元素的位置尺寸（浏览器已算好，无需自己 reflow）
intersectionRect实际相交区域的矩形
time变化发生的时间戳

三个配置项详解

root：作为参照的「视口」。默认 null 即浏览器视口；也可传一个可滚动的祖先元素，用于「容器内滚动」的场景。

rootMargin：给 root 的边界加一圈外边距（语法同 CSS margin，如 '100px 0px'）。正值提前触发——元素还没真正进视口、离边界 100px 时就回调，常用于懒加载提前加载；负值则延后触发。

threshold：相交比例阈值。0 表示一露头（哪怕 1 像素）就触发；1 表示完全进入才触发；也可传数组 [0, 0.5, 1]，在露出 0%、50%、100% 这几个节点各触发一次。

实战：图片懒加载

这是 IntersectionObserver 最经典的应用。把真实地址放在 data-src 里，元素进视口才赋给 src 触发真正的网络请求；加载后立刻 unobserve 该元素，避免重复触发、也减少观察负担。

HTML：用 data-src 占位
<img class="lazy" data-src="/photos/real.jpg" alt="" />
JS：进视口才加载，加载后停止观察
const imgs = document.querySelectorAll('img.lazy');

const io = new IntersectionObserver((entries, obs) => {
  entries.forEach((entry) => {
    if (!entry.isIntersecting) return;
    const img = entry.target;
    img.src = img.dataset.src;  // 此刻才发起真正的请求
    obs.unobserve(img);           // 加载过一次就不必再观察了
  });
}, { rootMargin: '200px' });  // 提前 200px 预加载，体验更顺滑

imgs.forEach((img) => io.observe(img));
同样的套路稍作变形即可做无限滚动：在列表末尾放一个「哨兵」空 div，观察它；哨兵进视口就加载下一页数据。还能做滚动动画（进视口添加动画 class）和曝光埋点（用 threshold 配合，元素露出 50% 才算一次有效曝光）。

对比：scroll + 节流 vs IntersectionObserver

反例：scroll 事件 + 节流 + getBoundingClientRect
// 即使加了节流，仍有成本：每次回调都要遍历元素、强制同步布局
window.addEventListener('scroll', throttle(() => {
  imgs.forEach((img) => {
    const rect = img.getBoundingClientRect(); // 触发 reflow！
    if (rect.top < innerHeight) img.src = img.dataset.src;
  });
}, 200));

陷阱与注意点

  • 回调是异步触发的，不保证实时。它在浏览器空闲时批量上报，所以别拿它做需要像素级实时同步的逻辑（如「跟手」动画）——那种场景仍需 requestAnimationFrame。
  • 初始就在视口内的元素也会触发一次回调。observe() 后浏览器会立刻做一次初始检测，因此首屏可见的元素会马上收到 isIntersecting: true，这通常正是你想要的。
  • 触发一次后记得 unobserve。懒加载、动画这类「只需触发一次」的场景，不解绑会导致元素反复进出视口时重复执行。
  • 看清 threshold 的含义。threshold: 0 是「露 1 像素就触发」，不是「完全离开」；要等完全可见用 1。初学者常把它和 rootMargin 混淆——前者管「露出多少比例」，后者管「边界扩缩多少距离」。
  • 兼容性：现代浏览器（含移动端）早已全面支持，可放心使用；如需兼容远古环境可引入官方 polyfill。
小结：① 用它替代「scroll + getBoundingClientRect」判断元素是否进视口，省掉高频触发与强制 reflow；② 三步：new IntersectionObserver(cb, options) → observe() → 回调里读 entry.isIntersecting；③ options 三件套：root 参照系、rootMargin 提前/延后、threshold 相交比例；④ 懒加载/无限滚动/曝光埋点是经典场景，记得用完 unobserve；⑤ 回调异步、初始会触发一次，按需求处理。
