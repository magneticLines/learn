修改元素内容与属性

选到元素之后，真正的"动起来"是改它的内容和属性。这里有三个改内容的 API（textContent / innerHTML / innerText）和两套改属性的体系（attribute vs property），选错不只是风格问题——轻则性能崩塌、事件丢失，重则被注入恶意脚本。

它解决了什么问题

页面不是一次渲染完就静止的。用户点了按钮、接口返回了数据、表单校验失败需要提示——这些都要求 JS 在运行时改变已有元素的显示文字、HTML 结构和状态。浏览器为此提供了一组 DOM 接口。难点在于：同一件事往往有多个 API 能做到，但它们在安全性、性能、是否解析 HTML上差异巨大。本节就是帮你建立"什么场景该用哪个"的肌肉记忆，避开几个会被反复踩的坑。

改内容：三个 API 的取舍

textContent 把内容当纯文本处理，赋什么字符串就原样显示什么，不解析标签；innerHTML 把字符串当 HTML 解析，能渲染出真正的子元素；innerText 看起来像 textContent，但它受 CSS 影响（隐藏的元素读不到）、读取时会触发回流（reflow），性能最差。

特性textContentinnerHTMLinnerText
是否解析 HTML否，纯文本是，渲染标签否，纯文本
读取隐藏元素能读到读到完整 HTML读不到（受 CSS 影响）
性能最快需解析，较慢触发回流，最慢
XSS 风险无有 ⚠️无
典型用途显示纯文字插入结构化 HTML取"用户肉眼可见"的文本
选择口诀：只显示文字一律用 textContent（最快最安全）；确实要插入 HTML 片段才用 innerHTML，且内容必须可信；innerText 几乎只在"想拿到屏幕上实际可见文本"这种少数场景才用。
三者行为对比
const el = document.querySelector('#box');

// textContent：标签被当成文本，原样显示出 <b> 这几个字符
el.textContent = '<b>加粗?</b>';   // 页面显示：<b>加粗?</b>

// innerHTML：标签被解析，真的渲染成加粗
el.innerHTML = '<b>加粗?</b>';     // 页面显示：加粗?（粗体）

重点警示：innerHTML 的 XSS 风险

这是前端最常见的安全漏洞之一。只要你把用户能控制的输入拼进 innerHTML，攻击者就能塞进 <script> 或带 onerror 的标签，让恶意代码在别人的浏览器里执行（窃取 Cookie、冒充操作）。这叫 XSS（跨站脚本攻击）。

反例：拼接用户输入到 innerHTML
// 假设这是从输入框 / URL / 接口拿到的"用户内容"
const userInput = '<img src=x onerror="alert(document.cookie)">';

// ❌ 致命：onerror 里的脚本会立即执行，Cookie 被弹出/外泄
el.innerHTML = '欢迎，' + userInput;
为什么 <script> 直接写进 innerHTML 反而不执行，而 onerror 却能？浏览器规范规定：通过 innerHTML 插入的 <script> 不会被执行——这让很多人误以为 innerHTML 安全。但带事件处理器的标签（如 <img onerror>、<svg onload>）会照常触发，照样能执行任意代码。所以"看不到 script 标签"绝不等于安全。
正例：用 textContent 修复
// ✅ textContent 永远当纯文本，标签字符被转义显示，绝不执行
el.textContent = '欢迎，' + userInput;
// 页面老老实实显示：欢迎，<img src=x onerror="...">

// 若确实需要渲染富文本，必须先用 DOMPurify 等库消毒
el.innerHTML = DOMPurify.sanitize(userInput);

改属性：attribute vs property 两套体系

这是新手最容易混淆的概念。HTML attribute 是你写在标签里的那些特性（id="x" value="abc"），它们是字符串，反映的是元素的初始状态；DOM property 是 JS 元素对象上的属性（el.id el.value），它们是类型化的值（布尔、数字等），反映元素的当前状态。浏览器解析 HTML 时会把 attribute 映射成 property，但二者之后并不总是同步。

维度HTML attributeDOM property
访问方式getAttribute / setAttribute直接 el.id el.value
值类型永远是字符串类型化（boolean / number…）
反映的状态初始值（HTML 里写的）当前值（运行时实际状态）
检测/删除hasAttribute / removeAttribute赋 null 或空，因属性而异
attribute API 一览
el.getAttribute('data-id');          // 读，返回字符串（没有则返回 null）
el.setAttribute('title', '提示');     // 写
el.hasAttribute('disabled');        // 是否存在，返回布尔
el.removeAttribute('disabled');     // 删除整个特性
常用 property：value / checked / disabled / href / src
const input = document.querySelector('input');
input.value = 'hello';        // 文本框当前内容
input.checked = true;          // 复选框选中状态，是布尔不是字符串
input.disabled = true;         // 禁用，比 setAttribute('disabled','') 更直观

const a = document.querySelector('a');
console.log(a.getAttribute('href'));  // '/page'        —— 原样字符串
console.log(a.href);                  // 'https://站点/page' —— property 自动补全为绝对 URL
对比记忆：大多数日常操作用 property 更省事（类型对、可读性好）。但像 data-*、aria-*、SVG 属性、或某些没有对应 property 的自定义特性，就得用 setAttribute/getAttribute。

陷阱：attribute 与 property 不总同步

最经典的例子是 <input> 的 value。用户在框里打字、或你用 JS 改了 el.value（property），元素的当前值变了；但 getAttribute('value') 拿到的仍是 HTML 里写的初始值，它不会跟着变。理解这点，才能解释"为什么读 attribute 拿到的是旧值"。

value：property 是当前值，attribute 是初始值
// HTML: <input id="t" value="初始">
const t = document.querySelector('#t');

t.value = '用户改的';           // 改 property（当前值）
console.log(t.value);               // '用户改的'  ← 当前状态
console.log(t.getAttribute('value')); // '初始'      ← attribute 没变！

// 想把表单"重置回初始值"时，这个区别就有意义了

陷阱：innerHTML += 会重建并丢失事件

el.innerHTML += '...' 看起来是"追加"，实际是把现有 HTML 序列化成字符串、拼接、再整段重新解析。这意味着容器里所有旧子节点都会被销毁重建——之前用 addEventListener 绑在它们身上的事件监听全部失效，性能也很差。

反例 → 问题 → 正例
// 反例：按钮先绑了点击事件
box.querySelector('button').addEventListener('click', fn);
box.innerHTML += '<span>新内容</span>';
// 问题：旧 button 被重建成"新的"，click 监听没了，点击无反应

// 正例：用 append / appendChild 真正追加，不动已有节点
const span = document.createElement('span');
span.textContent = '新内容';
box.append(span);   // 已有 button 及其事件原封不动

data-* 自定义属性与 dataset

当你想在元素上挂一点自定义数据（比如列表项的 ID），规范做法是用 data-* 属性，再通过 el.dataset 读写。命名规则：HTML 里的中划线写法 data-user-id，到 dataset 里自动变成小驼峰 userId。注意：dataset 读出来永远是字符串，数字要自己转。

dataset 读写自定义数据
// HTML: <li data-user-id="42" data-role="admin">Tom</li>
const li = document.querySelector('li');

console.log(li.dataset.userId);  // '42'  ← data-user-id → userId（小驼峰，且是字符串）
console.log(Number(li.dataset.userId));  // 42  ← 要数字得手动转

li.dataset.role = 'guest';      // 写，等价 setAttribute('data-role','guest')
小结：① 显示纯文字优先 textContent（最快最安全），要插 HTML 才用 innerHTML，innerText 慢且受 CSS 影响；② 把用户输入拼进 innerHTML 会导致 XSS——改用 textContent 或先 DOMPurify.sanitize；③ attribute 是字符串/初始值（getAttribute），property 是类型化/当前值（el.value），input.value 改后 attribute 不变；④ innerHTML += 会重建子节点、丢失已绑事件，追加请用 append；⑤ 自定义数据用 data-* + dataset（小驼峰、值为字符串）。

与其他知识点的联系

  • DOM 选取元素（上一篇）：本篇所有操作的前提都是"先选到元素"，二者是严格的先后关系。选取阶段的判空习惯在这里同样重要：对 null 调用 .textContent = ... 会直接抛出运行时错误。
  • 创建/插入/删除节点（下一篇）：本篇讲的是"改已有元素"，下一篇讲的是"造新元素、挂到树上"，二者共享同一套安全隐患——innerHTML 拼接用户输入或用户内容都有 XSS 风险，innerHTML += 都会重建并丢失事件监听。这不是两个孤立的坑，而是同一条"HTML 字符串解析"机制在不同操作上的表现。
  • JSON 序列化与反序列化： dataset 读出来的值永远是字符串，这和 JSON.parse 的角色类似——都是"把外部/DOM 世界里的文本还原成 JS 里可用的类型"，只是 dataset 需要你手动 Number() 转换，没有像 JSON 那样内置的类型推断。
  • 为什么 React 不用原生 DOM 操作：本篇讲的 attribute/property 不同步、innerHTML += 丢事件这些坑，本质都源于"手动同步 UI 状态和 DOM 状态"这件事很容易出错。React 用"状态驱动视图"的模型（改 state，React 自动 diff 出该改哪些 DOM），把"什么时候该用 property 还是 attribute"、"怎么保证事件不丢"这类细节都交给框架统一处理，你只需要声明 UI 应该长什么样。
  • classList 操作（后续章节）：本篇修改的是"内容"和"通用属性"，而 class 属性有专门的 classList API（比用 setAttribute('class', ...) 更安全好用），后续章节会展开这套专属工具，是本篇 attribute/property 框架下的一个特化分支。
