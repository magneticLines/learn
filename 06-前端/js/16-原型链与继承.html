原型链与继承

别的语言用"类"实现继承，JS 偏偏用"原型"。理解原型链，你就看穿了 class 的真面目、new 的本质，以及为什么 typeof null 那种历史坑会一直存在——它们都长在同一棵原型树上。

它解决了什么问题

设想你有一万个数组，每个都要能调用 push / map / filter。如果每个数组对象自己存一份这些方法，内存会被同样的函数撑爆。JS 的答案是：方法只存一份，放在一个"公共原型对象"上，所有实例共享它。当你在实例上找不到某个属性时，引擎就顺着一条隐藏的链条往上找，找到为止。这条链就是原型链（prototype chain），它既解决了方法复用（省内存），又顺势实现了继承——子类的实例能用到父类原型上的方法。所以 JS 的继承不是"拷贝"，而是"委托查找"：自己没有，就问上级要。

核心：每个对象都有 [[Prototype]]

每个对象内部都有一个隐藏槽位 [[Prototype]]，指向它的原型对象（另一个对象，或 null）。这个槽位不能直接写，但可以通过 Object.getPrototypeOf(obj) 读取，或通过非标准但被广泛实现的 obj.__proto__ 访问。

读取一个对象的原型
const obj = { name: "shea" };

Object.getPrototypeOf(obj) === Object.prototype;  // true，字面量对象的原型就是 Object.prototype
obj.__proto__ === Object.prototype;           // true，等价写法（但 __proto__ 是历史遗留，生产代码用 getPrototypeOf）

Object.getPrototypeOf(Object.prototype);          // null —— 原型链的尽头

属性查找：沿原型链向上，直到 null

这是整篇文章最该刻进脑子里的一句话：读取 obj.x 时，引擎先看 obj 自身有没有 x；没有就去 obj 的原型找；还没有再去原型的原型……一直找到 null 为止，全程没找到就返回 undefined。继承的本质，就是这个查找过程。

查找过程演示
const animal = { eat() { return "咀嚼"; } };

// 让 dog 的原型指向 animal —— dog 自己没有 eat，但能"继承"到
const dog = Object.create(animal);
dog.bark = () => "汪";

dog.bark();  // "汪" —— 自身找到
dog.eat();   // "咀嚼" —— 自身没有，沿链找到 animal 上的 eat
dog.fly;      // undefined —— 整条链都没有
只读不写：原型链只影响"读取"。给 dog.eat = ... 赋值时，会在 dog 自身创建一个 eat 属性（叫"属性遮蔽 / shadowing"），不会修改 animal。写入永远落在自身对象上。

函数的 prototype 与实例的 __proto__

这里是 JS 初学者最大的混淆点，务必分清两个名字相似但完全不同的东西：

  • prototype：只有函数才有的一个普通属性。它指向一个对象，这个对象将成为"用该函数 new 出来的实例"的原型。
  • __proto__（即 [[Prototype]]）：每个对象都有，指向自己的原型。

二者由 new 关键字串起来，铁律是：实例.__proto__ === 构造函数.prototype。

new 到底做了什么
function Person(name) { this.name = name; }
Person.prototype.hello = function() { return "hi " + this.name; };

const p = new Person("shea");

// new 内部大致等价于：
//   1. 创建空对象 obj
//   2. obj.__proto__ = Person.prototype   ← 关键：建立原型链
//   3. 以 obj 为 this 执行 Person(name)
//   4. 返回 obj（除非构造函数返回了别的对象）

p.__proto__ === Person.prototype;   // true
p.hello();                            // "hi shea"，hello 在原型上，被 p 继承
p instanceof Person;                 // true，instanceof 就是检查原型链上有没有 Person.prototype

constructor 指回构造函数

每个函数的 prototype 对象上，默认都有一个 constructor 属性，指回函数自己。所以实例可以通过原型链拿到 constructor，"反查"自己是被谁造出来的。

constructor 的指向
Person.prototype.constructor === Person;  // true
p.constructor === Person;                  // true（p 自身没有，沿原型链找到）
陷阱：如果你用 Person.prototype = { hello() {...} } 整体替换原型对象，会丢掉默认的 constructor，导致 p.constructor 错误地指向 Object。手动改原型时记得补一句 Person.prototype.constructor = Person。

把链条画出来

对一个 new Person() 出来的实例 p，完整的原型链是这样一条单向链表：

原型链文字图
p
  │ .__proto__
  ▼
Person.prototype       // 这里有 hello、constructor
  │ .__proto__
  ▼
Object.prototype       // 这里有 toString、hasOwnProperty、valueOf...
  │ .__proto__
  ▼
null                   // 尽头，查找结束

这就是为什么任何对象都能调用 toString() / hasOwnProperty()——它们其实定义在链条顶端的 Object.prototype 上，被所有对象共享。数组的链条则多一层：arr → Array.prototype → Object.prototype → null，所以数组既能用 map（来自 Array.prototype），又能用 toString（来自 Object.prototype）。

旧式继承：构造函数 + prototype

ES6 之前，要让 Dog 继承 Animal，靠的是手工拼接两条原型链。核心两步：① 在子构造函数里用 Animal.call(this, ...) 借调父类构造函数，继承实例属性；② 把子类的 prototype 接到父类 prototype 上，继承原型方法。

寄生组合继承（最优雅的旧式写法）
function Animal(name) { this.name = name; }
Animal.prototype.eat = function() { return this.name + " 在吃"; };

function Dog(name, color) {
  Animal.call(this, name);   // ① 借父类构造函数，继承实例属性 name
  this.color = color;
}

// ② 用 Object.create 建一个原型指向 Animal.prototype 的空对象，避免直接 new Animal() 的副作用
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;   // 别忘了修回 constructor
Dog.prototype.bark = function() { return "汪"; };

const d = new Dog("旺财", "黄");
d.eat();   // "旺财 在吃" —— 沿链找到 Animal.prototype.eat
d.bark();  // "汪"

为什么不直接 Dog.prototype = new Animal()？因为那样会立刻执行一次 Animal 构造函数（可能有副作用），还会把 name 等实例属性塞进原型里被所有 Dog 共享。Object.create 只搭桥、不执行构造函数，这就是"寄生组合"比"组合继承"更优的原因。

ES6 class 只是语法糖

呼应上一篇——class / extends 并没有引入新的继承机制，它底层仍然是原型链，只是把上面那堆手工拼接包装成了清爽的语法。下面两段代码运行时几乎等价：

class 写法（与上方寄生组合等价）
class Animal {
  constructor(name) { this.name = name; }
  eat() { return this.name + " 在吃"; }  // 方法自动挂到 Animal.prototype
}
class Dog extends Animal {
  constructor(name, color) {
    super(name);          // 等价于 Animal.call(this, name)
    this.color = color;
  }
  bark() { return "汪"; }
}

// extends 在底层就帮你做了：Dog.prototype.__proto__ = Animal.prototype
Object.getPrototypeOf(Dog.prototype) === Animal.prototype;  // true
理解了这一点，class 就不再神秘：它就是原型链的"皮肤"。所以面试问"JS 有真正的类吗"，答案是：没有，class 是基于原型的语法糖。

Object.create：直接指定原型

Object.create(proto) 创建一个新对象，并把它的 [[Prototype]] 设为传入的 proto。这是不经过构造函数、直接控制原型链的最纯粹方式。

Object.create 的用法
const base = { greet() { return "hi"; } };

const o = Object.create(base);      // o.__proto__ === base
o.greet();                          // "hi"，继承自 base

const pure = Object.create(null);  // 没有任何原型的"纯净字典"，连 toString 都没有

陷阱与高频考点

坑说明
prototype vs __proto__最大的坑。prototype 是函数的属性；__proto__ 是每个对象的原型指针。记牢公式 实例.__proto__ === 构造函数.prototype。
原型上放引用类型放在原型上的数组/对象会被所有实例共享，一个实例改动会影响全部。引用类型应放在构造函数里用 this.xxx 定义。
改原型影响全部实例原型是共享的，运行时给 Foo.prototype 加方法，所有已存在的实例立刻能用到——既强大也危险。
for...in 会遍历原型链它会枚举到原型上的可枚举属性。只想遍历自有属性请配合 hasOwnProperty，或改用 Object.keys()。
替换原型丢 constructor整体赋值 Foo.prototype = {...} 会丢失默认 constructor，需手动补回。
反例：原型上放引用类型 → 实例共享
function Cart() {}
Cart.prototype.items = [];   // ❌ 数组在原型上，被共享

const a = new Cart(), b = new Cart();
a.items.push("苹果");
b.items;   // ["苹果"] —— b 莫名其妙也有了！因为 a、b 共用原型上的同一个数组

// 正例：引用类型放在构造函数里，每个实例独立
function Cart2() { this.items = []; }
hasOwnProperty 区分自有与继承
const p = new Person("shea");

p.hasOwnProperty("name");    // true，name 是自身属性
p.hasOwnProperty("hello");   // false，hello 来自原型，不是自有
"hello" in p;                  // true，in 会查整条原型链 —— 注意和 hasOwnProperty 的区别

对比：原型继承 vs 类式继承

角度JS 原型继承Java/C++ 类式继承
继承的对象对象继承对象（实例委托给另一个对象）类继承类，对象由类"铸造"出来
本质运行时委托查找，找不到就问上级编译期确定的蓝图复制，结构固定
动态性极强：运行时可随时改原型，影响所有实例弱：类定义编译后基本固定
方法存储共享在原型对象上，只一份方法属于类，逻辑上也共享

思维差异在于：Java 程序员想的是"先有类这个模板，再造对象"；JS 的原型思维是"对象可以直接以另一个对象为模板，没有类这个中间层"。class 语法让 JS 看起来像前者，但骨子里仍是后者。看穿这层，才算真正懂了 JS 的面向对象。

小结：① 每个对象有 [[Prototype]]，用 Object.getPrototypeOf 读取；② 属性查找沿原型链向上直到 null，这就是继承的本质；③ 铁律 实例.__proto__ === 构造函数.prototype，别和函数的 prototype 混淆；④ 链条 实例 → 构造函数.prototype → Object.prototype → null；⑤ class/extends 是原型链的语法糖；⑥ 引用类型别放原型、for...in 会遍历原型链、用 hasOwnProperty 区分自有属性。

与其他知识点的联系

  • Class 语法（上一篇）：这两篇是互相解释的关系——上一篇教你"怎么用" class，本篇揭示"底层是什么"。凡是上一篇里"反直觉"的行为（漏 new 报错、方法共享、super() 顺序），根源都在这条原型链上，两篇合起来才是完整认知。
  • import/export 与模块系统（下两篇）：原型链解决的是"对象级"的代码复用（方法共享），模块系统解决的是"文件级"的代码复用（导出/导入）。二者是同一个"复用与组织代码"母题在不同粒度上的答案，学完原型链再看模块系统，能更清楚地分辨"复用什么"和"怎么隔离"是两层不同的问题。
  • React 虚拟 DOM：React 元素、Fiber 节点内部大量使用原型继承来复用内部方法（比如 Component 的生命周期方法挂在原型上）。更本质的联系是：原型链的"委托查找"是一种运行时机制，而 React 恰恰选择不依赖这种运行时可变性去操作真实 DOM——真实 DOM 节点本身也是一整套基于原型继承的复杂对象（HTMLElement → Element → Node → EventTarget → Object），直接操作和查询这些节点代价很高，这也是 React 用虚拟 DOM 做 diff、只在必要时才触碰真实原型链上那些昂贵方法（如布局相关属性）的动机之一。
  • Java 的类继承对比：本篇专门用一张表对比了 JS 原型继承与 Java/C++ 类式继承。这个对比不是随手一提，而是理解"JS 到底有没有类"这个面试高频问题的关键——多语言背景的学习者尤其容易把 Java 的"编译期蓝图"思维套到 JS 上，导致对 class 的很多行为感到意外。
  • Node.js 模块系统：CommonJS 的 module.exports 本质也是一个普通对象，多个文件 require 同一个模块时拿到的是同一个对象引用（模块被缓存），这和原型链"多个实例共享同一个原型对象"是同一种"共享引用而非拷贝"的设计思路，理解了后者能更快看透前者。
