JS · 浏览器 API

localStorage / sessionStorage / Cookie

浏览器是无状态的——刷新一下页面，内存里的变量全没了。要让数据"活过"刷新甚至关闭，就得把它写进浏览器的持久化存储。这三种机制各有定位，选错了不仅别扭，还可能埋下安全隐患。

它解决了什么问题

HTTP 协议本身是无状态的：服务器处理完一个请求就"忘记"了你，下个请求它认不出你是谁。而 JS 里的变量随页面刷新一起销毁。于是出现了两个需求：① 服务器要在多次请求间认出同一个用户（登录态、购物车）；② 前端要把数据缓存在本地（用户偏好、草稿、token）。Cookie 最早解决了前者（它会自动随请求发回服务器）；但 Cookie 容量小、且每次请求都带着跑很浪费带宽，于是 HTML5 又补上了 localStorage 和 sessionStorage，专门做纯前端、不参与网络传输的存储。理解三者的关键，就是抓住"是否随请求自动发送"和"活多久"这两条线。

三者一览对比

localStorage：永久的本地仓库

API 极其简单，核心就四个方法。注意键和值都只能是字符串——这是最常踩的坑。它的数据永久保存，关浏览器、重启电脑都还在，且同源页面互相共享（同一个域名下所有标签页读到的是同一份）。

localStorage 四个核心方法
localStorage.setItem("theme", "dark");   // 写入（值会被转成字符串）
localStorage.getItem("theme");          // 读取 → "dark"；键不存在返回 null
localStorage.removeItem("theme");       // 删除单个键
localStorage.clear();                    // 清空当前源的所有键

存对象：必须 JSON.stringify / parse

既然只能存字符串，那存数组或对象怎么办？答案是先序列化成 JSON 字符串再存，读出来再解析回对象——这正好呼应第 14 篇 JSON 的知识点。如果你直接 setItem("user", obj)，存进去的会是没用的 "[object Object]"。

对象的存取套路
const user = { name: "Tom", age: 18 };

// 写：序列化成字符串
localStorage.setItem("user", JSON.stringify(user));

// 读：解析回对象（注意 getItem 可能返回 null，要兜底）
const raw = localStorage.getItem("user");
const obj = raw ? JSON.parse(raw) : null;

// 反例：直接存对象 → 存进去变成无用的字符串
localStorage.setItem("bad", user);
localStorage.getItem("bad");  // "[object Object]"  ❌

sessionStorage：标签页级的临时态

API 和 localStorage 一模一样（同样 setItem/getItem/...），唯一的区别是生命周期：数据只属于当前这个标签页的会话，关闭标签页就清空。而且它不在标签页之间共享——即使两个标签页打开的是同一个网址，各自的 sessionStorage 也互不可见。适合存"只在这一次流程里有意义"的数据，比如多步表单的临时草稿。

一个易混点：刷新页面（F5）属于同一个会话，sessionStorage 会保留；但关闭再重新打开标签页则是新会话，数据清空。"复制标签页"会拷贝一份当前的 sessionStorage 给新标签页，但此后两者各自独立。

Cookie：会随请求"长腿"跑的数据

Cookie 是三者里最古老、也最特别的：它每次向同域服务器发请求时都会自动带上。这正是它能做登录认证的原因——服务器在响应里 Set-Cookie 一个 session id，浏览器存下来，之后每个请求都自动带回去，服务器就认出你了。代价是它容量很小（约 4KB），且 JS 操作它的 API 非常原始——只能拼字符串。

document.cookie 的读写
// 写：一次赋值只设置/更新一条；属性用分号拼接
document.cookie =
  "token=abc123; max-age=3600; path=/; Secure; SameSite=Lax";

// 读：拿到的是一整个字符串，所有 cookie 用 "; " 连在一起
document.cookie;  // "token=abc123; theme=dark"  → 要自己 split 解析

// 删：把过期时间设成过去（或 max-age=0）
document.cookie = "token=; max-age=0; path=/";

常见属性：expires/max-age 控制过期（不设则为会话 cookie，关浏览器即失效）；path、domain 控制作用范围；Secure 表示只在 HTTPS 下发送；HttpOnly 表示 JS 读不到（见下文）；SameSite 控制跨站发送策略。

陷阱与考点

  • storage 只能存字符串：数字、布尔会被 String() 隐式转换，对象变成 "[object Object]"。存复杂数据务必 JSON.stringify。
  • 存满会抛错：超过配额时 setItem 抛 QuotaExceededError，应该 try/catch 兜底，别假设写入一定成功。
  • Cookie 拖慢请求：因为每个同域请求都自动带上 Cookie，往里塞大量数据会让所有请求的体积都变大，浪费带宽。纯前端数据应放 storage，别滥用 Cookie。
  • HttpOnly 的 Cookie JS 读不到：document.cookie 看不到带 HttpOnly 的项。这是防 XSS 的关键——即使页面被注入恶意脚本，也偷不走 session id。所以认证 token 一般由服务端设为 HttpOnly。
  • SameSite 与 CSRF：SameSite=Lax/Strict 让 Cookie 在跨站请求时不自动发送，从而防御 CSRF（攻击者无法借你的浏览器悄悄带上你的登录 Cookie）。现代浏览器默认 Lax。
  • 隐私/无痕模式：storage 通常可用，但关闭窗口后数据被清；某些环境配额极小甚至写入抛错，不要把它当成可靠存储。

额外能力：storage 事件跨标签页通信

当一个标签页修改了 localStorage，同源的其它标签页会收到 storage 事件（修改的那个标签页自己不会收到）。这给了我们一种轻量的"多标签页同步"手段——比如一处退出登录，其它标签页自动感知并跟着登出。

监听其它标签页的 storage 变化
window.addEventListener("storage", (e) => {
  // e.key 变化的键，e.oldValue 旧值，e.newValue 新值
  if (e.key === "token" && e.newValue === null) {
    console.log("另一个标签页登出了，本页同步登出");
  }
});
注意 sessionStorage 不参与跨标签页通信——它本就不在标签页间共享，所以也不会触发其它标签页的 storage 事件。只有 localStorage 的变化才会广播给同源的其它标签页。
小结：① 抓两条主线——"是否随请求自动发送"和"活多久"；② localStorage 永久、同源共享，sessionStorage 仅当前标签页、关闭即清，二者 API 相同且只存字符串（对象用 JSON.stringify/parse）；③ Cookie 容量小、每次请求自动携带，专做服务端认证，HttpOnly 防 XSS、SameSite 防 CSRF；④ storage 写入可能抛 QuotaExceededError，且 localStorage 的变化能通过 storage 事件跨标签页通信。
