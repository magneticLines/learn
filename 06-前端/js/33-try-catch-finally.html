JS · 错误处理

try / catch / finally

程序运行时总会遇到意外：网络断了、数据格式不对、调了不存在的方法。错误处理决定了你的代码是优雅地"软着陆"，还是直接崩溃白屏。try/catch/finally 就是 JS 给你的安全网——但它有个致命的边界：管不了异步。

它解决了什么问题

没有错误处理时，一旦某行代码抛出异常，整个调用栈会立即中断，后续逻辑全部不执行，浏览器控制台报一个红错，用户看到的可能是半截渲染的页面或彻底卡死。错误处理的本质，是把"会失败的危险操作"用安全网圈起来：失败时不让它炸穿整个程序，而是接住它、记录它、给出降级方案，让程序继续优雅地活着。这就是工程上说的"健壮性"（robustness）——不是保证不出错，而是出错了也能可控地失败。

三块结构：try / catch / finally

try 块圈起受监控的代码；一旦它内部任何一处抛错，控制权立刻跳进 catch，错误对象作为参数传入；finally 则无论成功还是失败都会执行，是清理资源的最佳位置。

三块各司其职
function parseConfig(raw) {
  console.log("开始解析");
  try {
    const cfg = JSON.parse(raw);  // 若 raw 非法 JSON，这里抛 SyntaxError
    return cfg.name;
  } catch (err) {
    console.error("解析失败，用默认值兜底：", err.message);
    return "default";  // 降级：不让调用方崩溃
  } finally {
    console.log("解析结束（无论成败都打印）");
  }
}
执行顺序要点：即使 try 或 catch 里有 return，finally 也会先执行完再真正返回。所以 finally 永远跑得到，这正是它适合做"关闭文件 / 释放锁 / 隐藏 loading"这类收尾工作的原因。

throw：主动抛出错误

除了语言自动抛的错，你也能用 throw 主动制造一个错误，强行中断当前流程并把控制权交给最近的 catch。语法上 throw 后面可以跟任意值——字符串、数字、对象都行，但强烈推荐抛 Error 对象。

抛字符串 vs 抛 Error
// 反例：抛裸字符串，丢失了堆栈信息，排错困难
throw "年龄不能为负";

// 正例：抛 Error 对象，自带 name / message / stack
function setAge(age) {
  if (age < 0) {
    throw new Error("年龄不能为负，收到：" + age);
  }
}

为什么推荐 Error？因为它携带三个关键属性，让排错有据可查：

属性含义示例
name错误类型名"TypeError" / "Error"
message人类可读的描述构造时传入的字符串
stack抛错时的调用栈快照定位到具体文件与行号
陷阱：抛裸值后果很重。如果你 throw "出错了"，下游 catch (e) 拿到的 e 是个字符串，e.message 是 undefined，e.stack 也没有——监控系统（如 Sentry）几乎无法定位问题来源。统一抛 Error（或它的子类，见下一篇自定义错误类型）是团队协作的底线规范。

ES2019：可省略 catch 参数

有时你只关心"出没出错"，不关心错误细节。从 ES2019 起，catch 后的括号可以整个省掉。

optional catch binding
// 旧写法：即使不用 e 也得声明
try { localStorage.setItem("k", v); } catch (e) { return null; }

// ES2019：不需要错误对象时直接省略括号，更干净
try { localStorage.setItem("k", v); } catch { return null; }

最大的坑：try/catch 抓不住异步错误

这是初学者翻车率最高的地方，也是面试必考。try/catch 是同步机制：它只能捕获在 try 块同步执行期间抛出的错误。而 setTimeout、事件回调、Promise 里的代码是稍后才执行的——那时 try/catch 早已退出，调用栈完全不同，错误自然"逃逸"出去了。

反例：catch 形同虚设
try {
  setTimeout(() => {
    throw new Error("异步炸了");  // 1 秒后才执行
  }, 1000);
} catch (e) {
  // 永远不会进来！try 块同步部分早已执行完毕退出
  console.log("我抓不到", e);
}
// 结果：1 秒后抛出未捕获错误，程序崩在回调里

修复方式取决于异步的形态。对 Promise 用链式 .catch()；对 async 函数，则可以用 await 把异步"拉回"到同步流里，try/catch 就重新生效了（呼应上一篇 async / await）。

正例：两种修复
// 方式一：Promise 链式 catch
fetch("/api")
  .then(r => r.json())
  .catch(err => console.error("请求失败", err));

// 方式二：async/await + try/catch（await 让错误回到同步可捕获位置）
async function load() {
  try {
    const r = await fetch("/api");  // await 抛出的错能被下面 catch 接住
    return await r.json();
  } catch (err) {
    console.error("请求失败", err);
  }
}

陷阱：finally 里的 return 会覆盖一切

finally 如果出现 return（或 throw），它会覆盖掉 try/catch 里已经准备好的返回值或异常。这往往是无意中埋下的 bug，调试时极其反直觉。

finally 的 return 偷梁换柱
function test() {
  try {
    return 1;      // 本想返回 1
  } finally {
    return 2;      // 却被 finally 改成 2，连抛出的异常也会被它"吞掉"
  }
}
console.log(test());  // 2
规范建议：finally 里只做无副作用的清理（关连接、清定时器、复位状态），绝不要在 finally 里写 return 或 throw，否则会无声地丢失正常结果或真实异常，是排查噩梦。

三种错误处理方式对比

方式适用场景能否抓异步特点
try/catch同步代码 / await 表达式仅配合 await就近捕获，可读性好
.catch()Promise 链式调用能（限本链）链式优雅，但易漏接
全局兜底谁都没接住的"漏网之鱼"能最后防线，只宜上报日志

全局兜底是最后一道防线：同步未捕获错误触发 window.onerror，未处理的 Promise rejection 触发 unhandledrejection 事件。它们不该替代局部处理，而是用来上报监控、避免页面彻底白屏。

全局兜底（上报用）
window.onerror = (msg, url, line, col, err) => {
  report("同步未捕获", err);
};
window.addEventListener("unhandledrejection", e => {
  report("未处理的 Promise", e.reason);
});

更多常见陷阱

  • 吞掉错误：catch (e) {} 空着什么都不做，错误被默默吃掉，线上出问题时毫无线索——这是最危险的"假装没事"。至少要 console.error 或上报。
  • 捕获范围过大：把一大段代码全包进一个 try，出错时分不清是哪一步崩的。应只圈住真正可能失败的危险操作。
  • 用错误控制正常流程：把 throw/catch 当 if/else 用（比如靠抛错跳出循环）。异常机制开销大且语义混乱，正常分支该用普通控制流。
小结：① try 监控、catch (err) 捕获、finally 收尾，三者无论成败 finally 必跑；② throw 优先抛 Error 对象，才有 message/stack 可排错；③ try/catch 抓不住异步回调里的错，要用 .catch() 或 async + await + try/catch；④ 别在 finally 里 return（会覆盖结果），别空 catch 吞错，别把范围圈太大；⑤ window.onerror / unhandledrejection 是全局最后防线，只做上报。
