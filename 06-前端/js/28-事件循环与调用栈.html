事件循环与调用栈

JS 只有一根线程，却能同时处理定时器、网络请求和用户点击而不卡死——靠的就是事件循环这套调度机制。它是整个异步篇的地基：看懂它，你就能预测任何一段代码的执行顺序。

它解决了什么问题

JS 诞生时的使命是在浏览器里操作 DOM。如果它是多线程的，就会出现"线程 A 正在删除一个节点、线程 B 同时往这个节点里写内容"的竞态——DOM 是共享状态，多线程会让一切失控。所以 JS 被设计成单线程：同一时刻只做一件事，结果可预测，也不需要锁。

但单线程有个致命问题：一个耗时操作（比如等服务器返回数据）会把唯一的线程堵死，期间页面完全无响应。解法不是加线程，而是把耗时操作交给别人去等，等好了再回来通知我——这个"别人"是浏览器（宿主环境），这个"回来通知"的调度机制就是事件循环（Event Loop）。

调用栈：函数执行的舞台

JS 引擎用一个调用栈（Call Stack）来追踪"现在执行到哪了"。它是后进先出（LIFO）的：调用一个函数就压栈（push），函数返回就弹栈（pop）。栈顶永远是当前正在执行的那个函数。

调用栈的压栈与弹栈
function multiply(a, b) { return a * b; }
function square(n) { return multiply(n, n); }

square(5);
// 栈的变化（自底向上）：
// 1. square 压栈
// 2. square 内部调用 multiply，multiply 压栈（栈顶）
// 3. multiply 返回 25，弹栈
// 4. square 返回 25，弹栈 —— 栈空
栈溢出（Stack Overflow）：栈空间有限。无限递归（忘了写终止条件）会不停压栈直到撑爆，抛出 RangeError: Maximum call stack size exceeded。这正是著名问答网站的名字由来。

三个角色：栈、Web APIs、任务队列

同步代码直接进调用栈，一行行执行完就弹出。遇到异步任务（定时器、网络请求、DOM 事件监听）时，JS 引擎自己不等待——它把任务交给宿主环境的 Web APIs（浏览器或 Node 提供的，运行在引擎之外），然后立刻继续执行后面的同步代码。

当 Web APIs 那边的事情办完了（定时到点、数据回来、按钮被点），对应的回调函数不会插队直接执行，而是被放进一个任务队列（Queue）排队等候。

事件循环就是一个永不停歇的循环：它反复检查"调用栈空了吗？"。只要栈一空，就从队列里取出排在最前的回调，压入栈中执行。这就是为什么异步回调总是在所有同步代码之后才跑。

宏任务 vs 微任务（核心）

队列其实不止一个。回调分两类，优先级不同，这是事件循环最容易考、也最容易答错的地方。

对比项宏任务 macrotask微任务 microtask
典型来源setTimeout、setInterval、I/O、UI 事件Promise.then/catch/finally、queueMicrotask、MutationObserver
所在队列宏任务队列微任务队列
执行时机每轮循环只取一个来执行该轮里全部清空才罢休
优先级低高（紧跟当前宏任务）

事件循环的精确流程是：执行一个宏任务 → 清空所有微任务（包括清空过程中新产生的微任务）→ （需要时）渲染一帧 → 再取下一个宏任务。换句话说，微任务永远在下一个宏任务之前跑完。

陷阱：微任务能"插队"且会自我增殖。如果一个微任务里又不断产生新的微任务，事件循环会一直清理微任务队列、迟迟轮不到下一个宏任务，甚至饿死渲染（页面卡住）。

经典输出顺序题，逐步拆解

这道题几乎是异步面试的"投名状"。先盖住答案，自己推一遍。

同步 → 微任务 → 宏任务
console.log('1 同步');

setTimeout(() => console.log('2 setTimeout 宏任务'), 0);

Promise.resolve()
  .then(() => console.log('3 then 微任务'));

console.log('4 同步');

// 输出顺序：1 同步 → 4 同步 → 3 then 微任务 → 2 setTimeout 宏任务

逐步分析：

  1. 整段脚本本身就是第一个宏任务。同步代码先入栈：打印 1 同步。
  2. setTimeout 的回调交给 Web APIs 计时，到点后进宏任务队列排队。
  3. Promise.then 的回调进微任务队列排队。
  4. 继续同步执行：打印 4 同步。此刻栈空，本宏任务结束。
  5. 清空微任务队列：执行 then，打印 3 then 微任务。
  6. 微任务清空后，取下一个宏任务：执行 setTimeout 回调，打印 2 setTimeout 宏任务。

await 的本质也是微任务

async/await 是 Promise 的语法糖。关键认知：遇到 await 时，函数会"暂停"并立即把控制权交还给调用者；await 后面的代码相当于被包进了 .then 回调，即一个微任务。

await 之后的代码是微任务
async function foo() {
  console.log('A 同步');
  await Promise.resolve();
  console.log('C 微任务（await 之后）');
}
foo();
console.log('B 同步');
// 输出：A 同步 → B 同步 → C 微任务（await 之后）

常见陷阱与考点

  • setTimeout(fn, 0) 不是"立即执行"：它只是把回调尽快放进宏任务队列，仍要等当前同步代码跑完、微任务清空、且前面的宏任务排完才轮到。延迟 0 是"最少 0 毫秒"，不是马上。
  • 微任务永远先于下一个宏任务：哪怕 setTimeout 写在 Promise.then 前面，then 也先执行。
  • 长任务阻塞渲染：浏览器渲染发生在宏任务之间。一个跑了几百毫秒的同步循环会霸占线程，期间页面无法重绘、无法响应点击——表现为"卡死"。重活应切片或丢给 Web Worker。
  • 定时器不精确：到点了若栈不空，回调只能继续排队，实际触发会晚于设定时间。
小结：① JS 单线程是为安全操作 DOM 而生，靠事件循环实现"非阻塞"；② 调用栈 LIFO，递归无底会栈溢出；③ 异步任务交给 Web APIs，完成后回调进队列，栈空才被取出；④ 顺序铁律：同步代码 → 清空所有微任务 → 取一个宏任务，循环往复；⑤ Promise.then / await 之后是微任务，setTimeout 是宏任务，setTimeout(fn,0) 也得排队。

与其他知识点的联系

  • DOM 事件回调也走这套调度：前四篇讲的 click、input 等事件回调，触发时机同样是"Web APIs 监听到用户操作 → 回调进宏任务队列 → 栈空后被事件循环取出执行"，和 setTimeout 走的是同一条调度路径，只是触发源不同。
  • 回调函数与回调地狱（下一篇）：本篇讲的"异步任务交给 Web APIs、完成后回调进队列"正是回调机制存在的底层原因——下一篇会具体展开"回调"这个概念，以及多层异步依赖时如何演变成回调地狱。
  • Promise 与 async/await：本篇"await 的本质也是微任务"这一节，直接是 Promise、async/await 两篇的地基——.then() 回调、await 之后的代码都被推进微任务队列，理解本篇的宏任务/微任务优先级，才能推算出 Promise 链和 async 函数的真实执行顺序。
  • 长任务与 Web Worker：本篇提到重活会阻塞渲染、应切片或丢给 Web Worker，这是理解"为什么要用 fetch 而不是同步阻塞式请求"（见 Fetch API 一篇）的前置认知——单线程模型下，任何耗时操作都必须异步化才不卡界面。
