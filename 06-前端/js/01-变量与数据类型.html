变量与数据类型（var / let / const）

变量是程序记忆数据的盒子，数据类型决定盒子里能装什么、怎么比较。这两件事看似简单，却藏着 JS 最经典的几个坑——理解它们，是写出可预测代码的地基。

它解决了什么问题

早年的 JS 只有 var，它有两个让人头疼的特性：函数级作用域（而非块级）和变量提升。这导致循环里的闭包、if 块里声明的变量行为反直觉，bug 频出。ES6（2015）引入 let / const，用块级作用域和暂时性死区把这些坑堵上了。所以现代代码的铁律是：默认用 const，需要重新赋值才用 let，永远不用 var。

陷阱一：var 的提升与函数作用域

var 声明会被"提升"到函数顶部，但赋值留在原地。于是变量在声明前访问不报错，只是值为 undefined——这往往掩盖了真正的逻辑错误。

反例：var 提升导致的诡异结果
console.log(a);  // undefined（不是报错！变量已提升但未赋值）
var a = 1;

// 换成 let，提前访问会直接抛 ReferenceError —— 错误暴露得更早，更好
console.log(b);  // ReferenceError: Cannot access 'b' before initialization
let b = 2;

陷阱二：循环里的 var 闭包

这是面试高频题。var 在整个循环里只有一个变量，三次回调共享它；循环结束时它早已是 3。let 则在每次迭代都新建一个绑定，闭包各自捕获自己的副本。

经典闭包陷阱
// 反例：全部打印 3
for (var i = 0; i < 3; i++)
  setTimeout(() => console.log(i));  // 3 3 3

// 正例：let 每轮独立绑定，打印 0 1 2
for (let j = 0; j < 3; j++)
  setTimeout(() => console.log(j));  // 0 1 2

陷阱三：const 不是"不可变"

const 锁的是变量与值的绑定关系，不是值的内容。对于对象/数组，你不能重新赋值整个变量，但可以修改它内部的属性。要真正冻结内容得用 Object.freeze()。

const 锁绑定，不锁内容
const arr = [1, 2];
arr.push(3);      // ✅ 合法，arr 现在是 [1,2,3]
arr = [];          // ❌ TypeError: Assignment to constant variable

数据类型：7 种原始类型 + 对象

JS 的值分两大阵营。原始类型（primitive）是不可变的、按值存储；对象类型按引用存储。

类型说明typeof 结果
string字符串"string"
number整数与浮点统一为双精度"number"
booleantrue / false"boolean"
undefined声明未赋值"undefined"
null显式的"空""object" ⚠️ 历史 bug
symbol唯一标识符（ES6）"symbol"
bigint任意精度大整数（ES2020）"bigint"
object对象/数组/函数等"object"（函数为 "function"）
经典考点：typeof null === "object"。这是 JS 诞生之初的设计失误，因为兼容性原因永远无法修复。判断 null 要直接用 value === null，不要依赖 typeof。

值类型 vs 引用类型

原始类型赋值是拷贝值本身，互不影响；对象赋值拷贝的是指向堆内存的引用（地址），两个变量指向同一个对象。这是后面理解"深浅拷贝""函数传参""=== 比较对象"的根基。

值 vs 引用
let x = 10, y = x;   // y 拷贝了 10 这个值
y = 20;
console.log(x);          // 10，x 不受影响

const o1 = { n: 1 }, o2 = o1;  // o2 拷贝的是引用
o2.n = 99;
console.log(o1.n);       // 99，o1 和 o2 是同一个对象
小结：① 现代写法默认 const，需要改值才 let，忘掉 var；② let/const 是块级作用域 + TDZ，让错误尽早暴露；③ const 锁绑定不锁内容；④ 7 种原始类型按值存，对象按引用存，typeof null 是历史坑。

与其他知识点的联系

  • 关联知识点：HTML 表单取值（html/26-input类型.html）：input.value 拿到的永远是 string 类型，即使 type="number" 的输入框也一样——浏览器只是限制了输入内容，不会自动帮你转型。这意味着 const age = input.value + 1 会做字符串拼接而不是加法，必须先用 Number() 之类的显式转换，才能落回本文讲的 number 原始类型上做运算。
  • 关联知识点：dataset 自定义属性（html/38-data自定义属性.html）：通过 element.dataset.xxx 读到的值同样全是字符串，哪怕你在 HTML 里写的是 data-count="3"。这和 input.value 是同一个坑：DOM 属性系统天生只认字符串，凡是要参与数值比较或运算，都要先按本文的类型体系转换成 number 或 boolean，否则很容易踩到后面要讲的 == 隐式转换的雷。
  • 关联知识点：类型转换与判断（js/02-类型转换与判断.html）：本文只讲了"有哪些类型、怎么存"，还没讲类型之间怎么互相转换和比较——那正是下一篇的主题。比如 const arr = [1,2] 之后如果写 arr == "1,2"，会触发数组到字符串的隐式转换；typeof null === "object" 这个历史坑，也会在讨论 == 和 === 的取舍时再次出现，理解本文的值类型/引用类型是看懂那些转换规则的前提。
  • 关联知识点：TypeScript 的静态类型系统：本文强调的"变量类型可以随时变、typeof null 是坑"，本质上是 JS 动态类型的代价——类型错误只能等代码跑到那一行才暴露。TypeScript 在此基础上加了一层编译期的静态类型检查，比如给变量标注 let count: number 后，再赋值字符串会直接报编译错误，而不用等到运行时才发现。学完 JS 的动态类型体系，再看 TS 就能明白它到底"多做了什么"。
  • 关联知识点：React 的 useState 类型推断：const [count, setCount] = useState(0) 里，初始值 0 的类型（本文讲的 number）会被 React/TS 用来推断 setCount 只能接受 number 类型的更新，这就是本文"变量绑定的是某种数据类型"这个概念在框架层面的延伸。如果初始值给成 null 或 undefined，还会牵扯到本文提过的 typeof null === "object" 这类特例，写状态类型时要格外小心。
  • 关联知识点：陷阱二里的闭包与 let 块级绑定：本文"陷阱二"讲的每次循环 let 都新建一个绑定，是理解 React 组件里 useEffect/事件回调闭包陷阱的基础——组件重新渲染时，每次渲染捕获的 state 变量其实也是"各自独立的绑定"，和 for (let i...) 每轮新建绑定是同一套作用域机制，只是把"循环"换成了"渲染"。
