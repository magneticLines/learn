URL 与 History API

地址栏不只是给人看的字符串，它是一份结构化的数据，也是浏览器历史栈的指针。掌握 location、URL、URLSearchParams 和 History API，你就能优雅地解析地址、传参，乃至在不刷新页面的前提下操纵历史——这正是现代单页应用前端路由的根基。

它解决了什么问题

早年要从地址里取一个查询参数，前端只能手写字符串切割：split("?") 再 split("&") 再 split("=")，还得自己 decodeURIComponent 解码，遇到值里带 = 或重复 key 就漏洞百出。而要在页面间切换又必然触发整页刷新、白屏、重新下载资源。浏览器后来给出两组标准答案：URL / URLSearchParams 把地址解析成可靠的结构化对象；History API（pushState / replaceState）让 JS 能改写地址栏 URL 和历史栈却不刷新页面。后者直接催生了 React Router、Vue Router 这类前端路由——一个文档、零刷新、却有完整的前进后退体验。

location 对象：解剖当前地址

window.location 既能读取当前 URL 的各个部分，也能赋值来跳转。把它的属性记牢，等于把一条 URL 拆成了零件。

属性含义示例（https://a.com:8080/p/x?id=3#top）
href完整 URL，可读可写"https://a.com:8080/p/x?id=3#top"
protocol协议（含冒号）"https:"
host主机名 + 端口"a.com:8080"
hostname仅主机名"a.com"
port端口（默认端口时为空串）"8080"
pathname路径"/p/x"
search查询字符串（含 ?）"?id=3"
hash锚点（含 #）"#top"
跳转的三种方式
location.assign("/login");   // 跳转，且在历史栈留下一条记录（可后退）
location.replace("/login");  // 跳转，但替换当前记录 —— 后退回不到原页面
location.reload();           // 重新加载当前页

// 直接给 href 赋值，等价于 assign（会刷新页面！）
location.href = "https://example.com";
assign vs replace 的关键差异在历史栈：assign 压入一条新记录，用户能按"后退"回来；replace 顶替当前记录，常用于登录跳转后不让用户退回登录页这类场景。

URL 对象：把字符串解析成零件

new URL(str) 不依赖当前页面，能解析任意一段 URL 字符串，属性和 location 一一对应。还能传第二个参数作为 base，方便拼接相对路径。

new URL 解析与拼接
const u = new URL("https://a.com/p/x?id=3&tag=js#top");
console.log(u.hostname);  // "a.com"
console.log(u.pathname);  // "/p/x"
console.log(u.search);    // "?id=3&tag=js"

// 第二参数为 base，解析相对路径
const rel = new URL("../api/list", "https://a.com/p/x");
console.log(rel.href);     // "https://a.com/api/list"

URLSearchParams：优雅处理查询字符串

这是解析 ?key=value&... 的标准工具。手写切割迟早会栽在编码、重复 key、空值上；URLSearchParams 帮你全部处理好。它既能独立 new，也可直接通过 url.searchParams 拿到。

get / set / append / getAll / toString / 遍历
// 注意：传入的字符串可带或不带开头的 ?，都能解析
const params = new URLSearchParams(location.search);

params.get("id");          // 取第一个值；不存在返回 null
params.getAll("tag");      // 取同名多值，返回数组 ["js","css"]
params.has("id");          // true / false
params.set("page", 2);      // 有则覆盖，无则新增（唯一）
params.append("tag", "vue"); // 追加一个同名 key，不覆盖
params.delete("page");

// toString 会自动做 URL 编码，再拼回地址
console.log(params.toString());  // "id=3&tag=js&tag=css&tag=vue"

// 可直接遍历（实现了迭代器）
for (const [k, v] of params) {
  console.log(k, v);
}

// 用对象快速构造（适合发请求拼参数）
const q = new URLSearchParams({ name: "张三", age: 18 });
console.log(q.toString());  // "name=%E5%BC%A0%E4%B8%89&age=18"（中文自动编码）
陷阱：location.search 带着开头的 ?。但 URLSearchParams 足够聪明，传 "?id=3" 或 "id=3" 都能正确解析。真正别犯的错是手写字符串切割——它处理不好中文编码和重复 key，能用 URLSearchParams 就别自己切。

History API：操纵历史栈

浏览器为每个标签页维护一个历史栈，前进/后退就是在栈里移动指针。window.history 提供了导航和改写历史的能力。

导航方法
history.back();      // 后退一步，等价于点击浏览器的 ←
history.forward();   // 前进一步
history.go(-2);     // 跳跃：负数后退、正数前进、go(0) 刷新当前页
history.length;     // 当前历史栈的记录条数

核心：pushState 与 replaceState

这两个方法是 SPA 路由的命脉。它们能在不刷新页面的前提下改变地址栏 URL 并操作历史栈。签名是 pushState(state, title, url)：state 是与该记录关联的任意可序列化数据（后退时能在 popstate 里取回），title 几乎所有浏览器都忽略（传空串即可），url 是新地址（必须同源）。

不刷新页面改写地址
// push：压入新记录，地址栏变为 /list?page=2，页面不刷新
history.pushState({ page: 2 }, "", "/list?page=2");

// replace：替换当前记录，不新增历史 —— 适合"修正"当前 URL
history.replaceState({ page: 1 }, "", "/list?page=1");

// 监听前进/后退 —— 注意：pushState 自身不会触发 popstate！
window.addEventListener("popstate", (e) => {
  // e.state 就是当初 pushState 传入的那个对象
  console.log("用户前进或后退了", e.state);
  render(location.pathname);  // 根据新 URL 重新渲染对应视图
});

hash 路由 vs history 路由

SPA 前端路由有两种实现流派。早期靠 location.hash（地址里带 #），因为改 hash 不刷新页面且兼容性极好；现代多用 History API，URL 更干净，但需要服务端配合。

维度hash 路由history 路由
URL 形态/app#/list（带 #）/list（干净）
变化触发的事件hashchangepopstate
改写 URL 的方式location.hash = ...pushState / replaceState
是否刷新页面否否
服务端是否需配置不需要（# 后内容不发给服务器）需要（刷新时要把所有路径回退到 index.html）
SEO / 美观较差较好
history 路由的部署坑：用户在 /list 直接刷新时，浏览器会真的向服务器请求 /list，而服务器上并没有这个文件，于是 404。解决办法是让服务器把所有未匹配的路径都回退（fallback）到 index.html，再由前端路由接管。hash 路由没这问题——# 后面的内容浏览器根本不会发给服务器。

pushState vs replaceState 对比

维度pushStatereplaceState
对历史栈的影响压入新记录，history.length +1替换当前记录，长度不变
能否后退到原页面能不能
是否刷新页面否否
是否触发 popstate否（仅用户前进/后退才触发）否
典型场景跳转到新视图、翻页修正当前 URL、登录后替换、过滤条件变化
为什么 pushState 不触发 popstate？因为 popstate 的语义是"历史栈指针移动了"，而 pushState 是你主动新增记录——你自己调用的代码当然知道该渲染什么，无需事件再通知一次。只有用户点浏览器前进/后退（指针被动移动）时，框架才需要 popstate 来感知并重渲染。所以前端路由库通常自己包一层：调 pushState 后手动触发渲染，同时监听 popstate 处理用户操作。
小结：① location 把地址拆成 protocol/host/pathname/search/hash 等零件，search 含 ?、hash 含 #；② 跳转用 assign（留历史）/replace（不留历史）/reload，给 href 赋值会刷新页面；③ 解析任意 URL 用 new URL，处理查询参数一律用 URLSearchParams，别手写切割；④ History API 的 pushState/replaceState 能不刷新页面改地址栏，是 SPA 路由根基；⑤ pushState 自身不触发 popstate，只有用户前进后退才触发，且 URL 必须同源；⑥ hash 路由靠 hashchange、零服务端配置，history 路由更干净但刷新需服务端 fallback 到 index.html。
