addEventListener 与事件对象

页面要"活"起来，靠的是对用户操作做出反应——点击、输入、滚动。事件机制就是浏览器把这些操作通知给你的代码的方式。掌握 addEventListener 和那个传进回调的 event 对象，是写交互的入场券。

它解决了什么问题

最早绑定事件只能写 el.onclick = fn，一个元素一个事件只能挂一个处理函数，第二次赋值会无声地覆盖第一个。多个模块想监听同一个按钮的点击？互相踩脚。addEventListener 就是为此而生：它把"注册监听"做成可叠加、可移除、可配置的标准接口，同一个元素同一种事件能挂任意多个回调，按注册顺序依次触发，彼此互不干扰。这正是组件化、插件化时代的刚需。

三种绑定方式对比

历史上有三代写法，能力一代比一代强。现代代码统一用 addEventListener，前两种只需认得、能读懂老代码即可。

方式写法能挂几个可移除问题
HTML 内联<button onclick="f()">1 个难结构与行为耦合，污染全局
DOM 属性el.onclick = fn1 个置 null重复赋值会覆盖，无法叠加
addEventListenerel.addEventListener('click', fn)多个可（同引用）无明显短板，标准做法
覆盖陷阱：el.onclick = fnA; el.onclick = fnB; 最终只剩 fnB，fnA 被悄无声息地丢掉了。而两次 addEventListener 注册 fnA、fnB，点击时两个都会执行。

addEventListener 的三个参数

签名是 addEventListener(type, handler, options)：type 是事件名（不带 on 前缀，如 'click'）；handler 是回调，会收到一个 事件对象；第三个参数控制监听行为。

基本用法：同一元素叠加多个监听
const btn = document.querySelector('#save');

// 两个回调都会执行，按注册顺序：先 logClick 后 doSave
btn.addEventListener('click', logClick);
btn.addEventListener('click', doSave);

function logClick(event) {
  console.log(event.type);   // 'click'，事件名
}

事件对象 event：浏览器塞给你的"案发现场"

每次事件触发，浏览器都会自动构造一个 event 对象作为第一个参数传进回调，里面记录了这次事件的全部上下文：类型、目标元素、鼠标坐标、按了哪个键……最关键的是下面几个成员。

target vs currentTarget（事件委托的基础）

这是高频考点，二者极易混淆：event.target 是真正被点中的那个元素（事件最初从哪儿发出）；event.currentTarget 是当前正在执行监听的那个元素（即你 addEventListener 绑在谁身上）。当事件冒泡时，target 不变，currentTarget 随冒泡层级变化。下一篇讲冒泡时，正是靠 target 才能把监听挂在父元素却知道点的是哪个子元素。

target 与 currentTarget 的区别
// HTML: <ul id="list"><li>A</li><li>B</li></ul>
const list = document.querySelector('#list');

list.addEventListener('click', function (e) {
  console.log(e.target);         // 被点中的 <li>（A 或 B）
  console.log(e.currentTarget);  // 始终是 <ul#list>，监听绑在它身上
});

preventDefault：阻止浏览器的默认动作

很多元素有"出厂自带"的行为：点链接会跳转、提交按钮会刷新页面、右键会弹菜单。event.preventDefault() 就是按住浏览器的手，让它别做这件默认的事，方便你用 JS 接管（如表单前端校验、SPA 路由）。

拦截表单提交，改由 JS 处理
form.addEventListener('submit', function (e) {
  e.preventDefault();   // 阻止页面刷新跳转
  // 这里走 ajax 提交、做校验……
});
preventDefault() 与 return false 的区别：在 addEventListener 的回调里 return false 毫无作用（只对老式内联 onclick="return false" 有效，那种写法等同于同时 preventDefault + stopPropagation）。现代代码请明确调用 e.preventDefault()，意图清晰、行为可预测。

stopPropagation：点到为止，不再往上冒泡

event.stopPropagation() 阻止事件继续向父元素传播。注意它和 preventDefault 是两件正交的事：前者管"事件还往不往上传"，后者管"浏览器默认动作做不做"。冒泡的完整机制下一篇详讲，这里先建立印象。

options：第三个参数的三个开关

第三个参数可以是布尔值（仅设置 capture），更推荐传对象，能配置三个常用选项。

选项含义典型场景
capturetrue 时在捕获阶段触发，而非默认的冒泡阶段需要在子元素之前先拦截
oncetrue 时触发一次后自动移除监听初始化、只弹一次的提示
passive向浏览器承诺不会调 preventDefault，可提前优化滚动touchmove、wheel 滚动
options 用法
// once：欢迎提示只触发一次，之后监听自动消失
btn.addEventListener('click', showWelcome, { once: true });

// passive：承诺不阻止滚动，浏览器无需等回调即可流畅滚动
window.addEventListener('wheel', onWheel, { passive: true });
passive 陷阱：一旦标了 passive: true，回调里再调 preventDefault() 无效，控制台还会警告。所以只在确实不需要拦截默认行为（如纯滚动监听）时才开它——它换来的是滚动不被 JS 卡住的丝滑体验。

removeEventListener：必须是同一个函数引用

移除监听用 removeEventListener(type, handler, options)，但有个铁律：传入的 handler 必须和注册时是同一个函数引用。因为浏览器内部是按引用来匹配并删除的。这意味着匿名函数永远无法被移除——你根本拿不到那个引用。

反例 → 问题 → 正例
// ❌ 反例：匿名函数，无法移除（两次写的是两个不同的函数对象）
btn.addEventListener('click', () => save());
btn.removeEventListener('click', () => save()); // 删不掉！

// ❌ 同理：bind 每次都返回新函数，引用对不上
btn.addEventListener('click', obj.save.bind(obj));

// ✅ 正例：先存成具名引用，注册和移除用同一个
const handler = () => save();
btn.addEventListener('click', handler);
btn.removeEventListener('click', handler); // ✅ 成功移除
不想自己管移除？用 { once: true } 让它触发一次后自动清理，或在现代环境用 AbortController 的 signal 批量取消一组监听——都能绕开"必须留住引用"的麻烦。
小结：① 现代统一用 addEventListener，可叠加多个、可移除、可配置，告别 onclick= 的覆盖陷阱；② 回调收到 event：target 是被点中的元素，currentTarget 是绑监听的元素，二者之别是事件委托的根基；③ preventDefault() 阻止默认行为，stopPropagation() 阻止冒泡，二者正交；④ options 三开关：capture/once/passive，passive 下调 preventDefault 无效；⑤ removeEventListener 必须传同一个函数引用，匿名函数与 bind 后的函数都删不掉。
