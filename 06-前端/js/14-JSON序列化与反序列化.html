JSON 序列化与反序列化

内存里的对象无法直接"塞进"网络请求或存进 localStorage——它们只认字符串。JSON 就是那座桥：把对象拍扁成文本（序列化），再从文本还原回对象（反序列化）。看似两个简单的 API，背后却藏着一堆会悄悄丢数据的坑。

它解决了什么问题

程序运行时，数据是内存里带着引用、原型、方法的对象；可一旦要跨越边界——发给后端、存进磁盘、写入数据库——这些边界只认得最朴素的东西：一串字节/文本。把内存结构转成可传输、可存储的文本，叫序列化；反过来叫反序列化。

JSON（JavaScript Object Notation）就是为此而生的一种文本数据交换格式。它的语法源自 JS 对象字面量，但它语言无关——Java、Python、Go 都能解析同一段 JSON。正因为简单、人类可读、几乎所有语言原生支持，它取代了笨重的 XML，成了今天 Web API 的事实标准。JS 内置了 JSON.stringify 和 JSON.parse 两个静态方法来完成这两个方向的转换。

JSON.stringify：对象 → 字符串

把一个 JS 值序列化成 JSON 字符串。它有三个参数，第一个是要序列化的值，后两个常被忽略却很有用。

三个参数：value / replacer / space
const user = { name: "Alice", age: 30, pwd: "123456" };

// 1. 最常用：直接转字符串
JSON.stringify(user);
// '{"name":"Alice","age":30,"pwd":"123456"}'

// 2. replacer 为数组 → 白名单，只保留这些键
JSON.stringify(user, ["name", "age"]);
// '{"name":"Alice","age":30}'  —— pwd 被过滤掉了

// 2'. replacer 为函数 → 每个键值都过一遍，返回 undefined 则丢弃该键
JSON.stringify(user, (key, val) =>
  key === "pwd" ? undefined : val);

// 3. space 缩进 → 美化输出，便于人读/打日志
JSON.stringify(user, null, 2);
// {
//   "name": "Alice",
//   "age": 30,
//   "pwd": "123456"
// }

JSON.parse：字符串 → 对象

把 JSON 字符串还原成 JS 值。第二个参数 reviver 是个函数，在每个键值还原时被调用，让你有机会改写值——最典型的用途是把字符串还原成 Date 对象（因为日期序列化后变成了字符串，parse 不会自动认出它）。

reviver：还原 Date
const json = '{"name":"Alice","born":"2000-01-01T00:00:00.000Z"}';

// 不用 reviver：born 还是字符串
JSON.parse(json).born;        // "2000-01-01T..." （string）

// 用 reviver 把指定键转回 Date 对象
const obj = JSON.parse(json, (key, val) =>
  key === "born" ? new Date(val) : val);
obj.born instanceof Date;      // true ✅

典型用途

  • 网络传输：fetch 发请求时 body: JSON.stringify(data)，收响应时 res.json() 内部就是 JSON.parse。
  • 本地存储：localStorage 只能存字符串，所以存对象前先 stringify，取出后再 parse。
  • 简易深拷贝：JSON.parse(JSON.stringify(obj))——先拍成字符串再解析回来，新对象与原对象不共享任何引用。一行搞定，但局限很大（见下文）。
一行深拷贝（有坑）
const original = { a: 1, nested: { b: 2 } };
const copy = JSON.parse(JSON.stringify(original));
copy.nested.b = 99;
console.log(original.nested.b);   // 2，深层也被独立拷贝了 ✅

陷阱一：会被悄悄丢弃或改变的值

这是最危险的坑，因为它不报错，只是默默改了你的数据。序列化遵循一套固定规则，凡是 JSON 表达不了的东西，要么被忽略，要么被替换。

原始值序列化后说明
undefined对象属性被删除；数组里变 nullJSON 没有 undefined 概念
函数 / 方法同上，被忽略JSON 只描述数据，不含行为
Symbol同上，被忽略无法表示
NaN / Infinity变成 nullJSON 数字不含这些特殊值
Date变成 ISO 字符串parse 不会自动还原，需 reviver
BigInt直接抛 TypeError无法被序列化
Map / Set变成 {}（空对象）非普通对象，内容丢失
循环引用直接抛 TypeError无限嵌套，无法表达
亲眼看看这些坑
JSON.stringify({ a: undefined, b: function(){}, c: 1 });
// '{"c":1}'  —— a、b 凭空消失

JSON.stringify([undefined, NaN, Infinity]);
// '[null,null,null]'  —— 数组里统统变 null

JSON.stringify({ big: 10n });
// ❌ TypeError: Do not know how to serialize a BigInt

const o = {}; o.self = o;   // 循环引用
JSON.stringify(o);
// ❌ TypeError: Converting circular structure to JSON
面试高频："用 JSON.parse(JSON.stringify()) 做深拷贝有什么问题？"标准答案就是上表——undefined/函数/Symbol 丢失、Date 退化为字符串、NaN/Infinity 变 null、Map/Set 变空对象、BigInt 和循环引用直接报错。所以它只适合"纯数据、无特殊类型"的简单对象。

陷阱二：JSON 不是 JS 对象字面量

JSON 长得像 JS 对象，但它是一套更严格的语法。手写或拼接 JSON 字符串时，下面这些"JS 里能用"的写法在 JSON 里全是非法的，JSON.parse 会直接抛 SyntaxError。

这些都是非法 JSON
// ❌ 字符串必须用双引号，不能用单引号
{ 'name': 'Alice' }

// ❌ 键必须加双引号
{ name: "Alice" }

// ❌ 不允许尾随逗号
{ "a": 1, }

// ❌ 不允许任何注释
{ "a": 1 /* 注释 */ }

// ✅ 合法 JSON：双引号 + 引号键 + 无尾逗号 + 无注释
{ "name": "Alice", "age": 30 }
记忆口诀：双引号、键带引号、尾逗号禁止、注释禁止。所以永远不要手拼 JSON 字符串，交给 JSON.stringify 去生成才不会出错。

用 toJSON 自定义序列化

如果一个对象定义了 toJSON() 方法，JSON.stringify 会优先调用它，用它的返回值去序列化。这给了你"如何把自己拍扁"的完全控制权。Date 能输出 ISO 字符串，正是因为它内置了 toJSON。

自定义对象的序列化形态
class Money {
  constructor(cents) { this.cents = cents; }
  // 序列化时输出"元"而非内部存的"分"
  toJSON() { return (this.cents / 100).toFixed(2); }
}
JSON.stringify({ price: new Money(12345) });
// '{"price":"123.45"}'

对比：三种深拷贝方案

方案优点缺点
JSON.parse(JSON.stringify())一行，无需依赖丢失/改变特殊类型，循环引用与 BigInt 报错，性能一般
structuredClone(obj)原生（现代浏览器/Node 17+），支持 Date / Map / Set / 循环引用仍不支持函数、Symbol、DOM 节点；老环境无此 API
手写递归拷贝完全可控，可处理任意自定义逻辑要自己处理循环引用、各种类型，易写错
structuredClone 是今天做结构化深拷贝的首选——它能正确处理 JSON 法搞不定的 Date、Map、Set 和循环引用。但 JSON 法还有一个 structuredClone 没有的"副作用"用途：顺手剔除函数与 undefined 属性。深浅拷贝的完整原理与手写实现，留到第 39 篇《深浅拷贝》详讲。
小结：① JSON 是语言无关的文本交换格式，stringify 把对象拍成字符串、parse 还原回来；② stringify 的 replacer 可过滤、space 可美化，parse 的 reviver 可改写（如还原 Date）；③ 序列化会悄悄丢 undefined/函数/Symbol、把 NaN/Infinity 变 null、Date 退化为字符串，遇到 BigInt 和循环引用直接报错；④ JSON 语法严格：双引号、键带引号、禁尾逗号与注释，别手拼；⑤ 深拷贝优先 structuredClone，JSON 法只适合纯数据简单对象。

与其他知识点的联系

  • Map 与 Set（上一篇）：这也是为什么 JSON.stringify 会把 Map/Set 序列化成空对象 {}——JSON 规范里没有"集合"这种结构，只认识普通对象和数组。想正确传输 Map，得先手动转成数组（[...map]）或对象，接收端再转回来。
  • Class 语法 / toJSON（下一篇）：本篇提到的 toJSON() 钩子，正是给自定义类一个"体面地被序列化"的机会。如果你的类内部字段（比如私有字段 #cents）不适合直接暴露，就该在类里定义 toJSON，而不是让引擎去猜怎么拍平这个实例。
  • 网络请求与 fetch：前端发请求 body: JSON.stringify(data)、后端解析 JSON.parse（或 Java 里 Jackson/Gson 做同样的事），本质是两端约定好用 JSON 这套"最大公约数"格式互相理解对方内存里的数据结构。理解序列化规则的坑，才能解释"我明明传了 undefined，后端为什么收不到这个字段"。
  • Node.js 模块与 package.json：package.json 本身就是一份 JSON 文件，Node 用 JSON.parse 读取它来决定项目的模块类型、依赖版本等。可见 JSON 不只是网络传输格式，也是各类工具链读写配置的通用容器。
  • 深浅拷贝（第 39 篇）：JSON.parse(JSON.stringify(obj)) 是理解"为什么需要专门的深拷贝方案"的最佳入口——正是因为它在处理特殊类型上的种种局限（本篇陷阱表），才有了后续 structuredClone 和手写递归拷贝的用武之地。
