JS · 浏览器 API

定时器 setTimeout / setInterval

定时器让你"过一会儿"或"每隔一段时间"执行代码。它看似只是两个简单函数，背后却牵动着事件循环、内存泄漏、this 指向等一连串话题——用得对是利器，用错了就是动画卡顿和内存泄漏的源头。

它解决了什么问题

JS 是单线程的，主线程被一段同步代码占住时，什么都干不了。但很多需求天生是"延迟"或"周期"的：3 秒后弹个提示、每秒刷新一次倒计时、轮询服务器状态。定时器就是浏览器（不是 JS 语言本身，而是宿主环境提供的 Web API）给我们的"闹钟"——你登记一个回调和一个延迟，浏览器在后台计时，时间到了就把回调放进任务队列，等主线程空闲时再执行。注意最后这句：定时器不会打断正在运行的代码，它只负责"到点排队"。这一点是理解后面所有陷阱的钥匙。

setTimeout：延迟执行一次

setTimeout(fn, delay, ...args) 在 delay 毫秒后执行一次 fn，多余的参数会原样传给 fn。它返回一个数字 id，把 id 交给 clearTimeout(id) 就能在回调触发前取消它。

延迟执行与取消
const id = setTimeout((name, n) => {
  console.log(`你好 ${name}，数字是 ${n}`);
}, 1000, "小明", 42);  // 1 秒后输出：你好 小明，数字是 42

// 反悔了，在它触发前取消（超过 1 秒就来不及了）
clearTimeout(id);
为什么要把参数写在 delay 后面？历史上 IE9 之前不支持这种透传写法，老代码常用闭包 setTimeout(() => fn(name, n), 1000) 代替。现代环境两种都行，箭头函数包一层更直观，也更常见。

setInterval：周期重复执行

setInterval(fn, delay) 每隔 delay 毫秒重复执行 fn，同样返回 id，用 clearInterval(id) 停止。它适合"固定节奏"的任务，比如倒计时显示。

每秒倒计时，归零后停止
let count = 3;
const timer = setInterval(() => {
  console.log(count);
  if (count-- === 0) clearInterval(timer);  // 不清就会永远跑下去
}, 1000);

与事件循环的关系：delay 不是"精确时间"

这是最容易误解的一点（呼应第 28 篇事件循环）。delay 表示的是"最早多久后把回调放进宏任务队列"，而不是"多久后一定执行"。如果到点时主线程还忙着跑同步代码，回调只能在队列里干等。所以定时器的实际触发时间总是 ≥ delay，从不会更早。

setTimeout(fn, 0) 也得排队等
console.log("1 同步");
setTimeout(() => console.log("4 宏任务"), 0);
Promise.resolve().then(() => console.log("3 微任务"));
console.log("2 同步");
// 输出顺序：1 同步 → 2 同步 → 3 微任务 → 4 宏任务
// delay=0 也要等当前同步代码 + 所有微任务跑完才轮到它
为什么 0 也排在微任务后面？因为定时器回调属于宏任务，而每个宏任务结束后会先清空整个微任务队列（Promise 的 then、queueMicrotask 等）才进入下一个宏任务。所以 setTimeout(fn, 0) 常被当作"把任务推迟到当前轮次之后"的手段，但它永远跑在微任务之后。

陷阱一：setInterval 会堆积，递归 setTimeout 更稳

setInterval 是"不管上一次跑完没有，到点就排队"。如果回调本身耗时超过了间隔（比如间隔 1 秒，但一次请求要 1.5 秒），任务就会堆积、重叠，甚至队列里挤满还没执行的回调，导致它们一执行完就立刻接着执行下一个，间隔形同虚设。

更稳妥的做法是用递归 setTimeout：每次回调执行完毕后再安排下一次。这样能保证两次执行之间始终有完整的 delay 间隔，永不重叠。

对比setInterval递归 setTimeout
计时基准从上次"排队"算起，固定节拍从上次"执行完"算起
回调超时会堆积/重叠，间隔失真不会重叠，间隔有保证
动态调整间隔难（要 clear 重设）容易（下次 delay 随时改）
异步任务（如轮询）不推荐推荐
正例：递归 setTimeout 自我调度
let timer;
function poll() {
  // 等这次的异步活儿真正干完，再安排下一次
  fetchData().finally(() => {
    timer = setTimeout(poll, 3000);  // 两次请求之间稳定隔 3 秒
  });
}
poll();
// 停止：clearTimeout(timer)（注意停的是最近一次登记的 id）

陷阱二：忘记 clear 导致内存泄漏与重复执行

定时器只要没被 clear，它引用的回调（以及回调闭包捕获的变量、DOM 节点）就无法被垃圾回收。在 SPA 框架里这尤其致命：组件已经卸载、DOM 已经移除，定时器还在每秒触发，操作一个不存在的元素——轻则报错，重则内存越积越多。铁律：在哪里 set，就要在对应的清理时机 clear。

组件卸载时务必清理（以 React 为例）
useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id);  // 卸载时清理，缺了这行就泄漏
}, []);
常见症状：页面切走又切回后，倒计时跳得越来越快——这往往是每次进入页面都 setInterval 一个新定时器，旧的却没清，多个定时器叠加触发。看到"越来越快"几乎可以断定是定时器泄漏。

陷阱三：回调里的 this 指向

定时器回调由宿主环境调用，不是作为某个对象的方法调用。所以传普通函数进去时，this 会指向全局对象（严格模式下是 undefined），而不是你以为的那个对象。解决办法是用箭头函数（它没有自己的 this，沿用外层）或 bind 绑死。

反例 → 正例
const timer = {
  sec: 0,
  start() {
    // ❌ 普通函数：this 不是 timer，this.sec 是 undefined → NaN
    setInterval(function () { this.sec++; }, 1000);

    // ✅ 箭头函数：this 沿用 start 的 this，正是 timer
    setInterval(() => { this.sec++; }, 1000);
  },
};

陷阱四：最小延迟与后台标签页节流

定时器的 delay 不能无限小。HTML 规范规定：当定时器嵌套层数超过 5 层后，最小延迟被钳制为 4ms。也就是说你写 setTimeout(fn, 0) 在深层嵌套里实际至少是 4ms。此外，浏览器为了省电，会对后台（非激活）标签页的定时器大幅节流——通常降到 1 秒一次甚至更慢。所以千万别用定时器做精确计时（比如做秒表），正确做法是用 Date.now() 记录真实时间戳，定时器只负责触发"该刷新显示了"，差值由时间戳算。

经典考点："setTimeout(fn, 1000) 一定 1 秒后执行吗？"答案是不一定，至少 1 秒，可能更晚——受主线程繁忙、最小延迟钳制、后台节流三重影响。能答出这三点，说明你真懂事件循环。

动画别用定时器：requestAnimationFrame

做动画时（比如让元素平滑移动），用 setInterval(fn, 16) 是个常见但糟糕的选择：它的节拍和屏幕刷新不同步，会丢帧或撕裂；后台标签页还在白白计算。浏览器为此提供了 requestAnimationFrame(fn)——它把回调安排在下一次屏幕重绘之前执行，自动跟随屏幕刷新率（60Hz 屏约 16.7ms 一次，120Hz 屏更快），页面切到后台时自动暂停。结果就是更平滑、更省电。

requestAnimationFrame 驱动动画
function step(timestamp) {  // 浏览器自动传入高精度时间戳
  moveBox(timestamp);
  requestAnimationFrame(step);  // 递归预约下一帧，跟随刷新率
}
const raf = requestAnimationFrame(step);
// 停止：cancelAnimationFrame(raf)
记忆口诀：延迟一次用 setTimeout，固定节奏用 setInterval（但轮询/超时任务改用递归 setTimeout），画动画用 requestAnimationFrame。
小结：① setTimeout 延迟执行一次、setInterval 周期执行，都返回 id，用 clearTimeout/clearInterval 取消；② delay 是"最早入队时间"不是精确时间，setTimeout(fn,0) 也要等同步代码和微任务跑完；③ setInterval 回调超时会堆积，轮询类任务改用递归 setTimeout；④ 组件卸载等场景务必 clear，否则泄漏 + 重复执行；⑤ 回调里的 this 用箭头函数或 bind 锁住；⑥ 受最小延迟（4ms）和后台节流影响，别拿它做精确计时；⑦ 动画首选 requestAnimationFrame。
