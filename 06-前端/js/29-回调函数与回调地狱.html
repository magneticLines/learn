回调函数与回调地狱

回调函数是 JS 异步世界的第一块基石——"事情做完了再叫我"。它简单、直接，却在多个异步操作互相依赖时退化成层层嵌套的"金字塔"。理解回调的好与坏，才能明白后面 Promise、async/await 究竟在解决什么。

它解决了什么问题

承接上一篇事件循环：JS 是单线程的，遇到耗时操作（定时器、网络请求、读文件）如果同步等待，整个线程就被卡死，页面假死、什么都点不动。所以这些操作必须异步——"先把任务交出去，我接着干别的，等结果好了你再通知我"。问题来了：结果到来时，怎么知道"接下来该执行哪段逻辑"？答案就是回调函数——你提前把"结果到了之后要做的事"写成一个函数交给它，等异步完成时由它来调用（call back）。回调把"未来某一刻才能执行的代码"打包起来，交给事件循环在合适的时机调度。

什么是回调函数

回调函数的本质很朴素：把一个函数作为参数传给另一个函数，待某件事完成后再被调用。在 JS 里函数是"一等公民"（可以像值一样传递），这才让回调成为可能。

函数作为参数传入
// greet 不关心"打完招呼后做什么"，把决定权交给调用者
function greet(name, callback) {
  console.log("你好, " + name);
  callback();  // 在合适的时机回调
}

greet("小明", function() {
  console.log("招呼打完了");
});

同步回调 vs 异步回调

同样是"把函数传进去被调用"，回调其实分两种，初学者常把它们混为一谈，导致对执行顺序的判断出错。

  • 同步回调：在当前函数执行过程中立即、原地被调用，不脱离当前调用栈。如 arr.map(fn)、arr.forEach(fn)、arr.sort(fn)——它们在 map 返回前就把回调全部跑完。
  • 异步回调：被登记后稍后才执行，要等当前同步代码跑完、事件循环把它从任务队列取出。如 setTimeout、DOM 事件监听、fetch 等网络请求。
同步回调 vs 异步回调的执行顺序
console.log("1 开始");

// 同步回调：map 内部立刻执行，紧贴当前流程
[1, 2].map((n) => console.log("2 同步 map " + n));

// 异步回调：被登记进任务队列，等同步代码跑完才执行
setTimeout(() => console.log("4 异步 setTimeout"), 0);

console.log("3 结束");
// 输出顺序：1 开始 → 2 同步 map 1 → 2 同步 map 2 → 3 结束 → 4 异步 setTimeout
关键区别：同步回调不脱离当前调用栈，执行顺序可预测；异步回调要交给事件循环调度，晚于所有同步代码。回调地狱专指异步回调的嵌套问题——同步回调不会有这种麻烦。

回调地狱（Callback Hell）

当多个异步操作前后依赖（第二步需要第一步的结果，第三步又依赖第二步……）时，由于异步结果只能在回调里拿到，你被迫把下一步写进上一步的回调里。一层套一层，代码向右缩进成一座"金字塔"，这就是回调地狱。

反例：4 层嵌套的回调金字塔
// 场景：登录 → 拿用户资料 → 拿订单列表 → 拿第一个订单详情
login("小明", function (err, user) {
  if (err) return handle(err);
  getProfile(user.id, function (err, profile) {
    if (err) return handle(err);
    getOrders(profile, function (err, orders) {
      if (err) return handle(err);
      getDetail(orders[0], function (err, detail) {
        if (err) return handle(err);
        console.log(detail);  // 终于拿到了，缩进已经到天边
      });
    });
  });
});

回调地狱的三宗罪：① 难读——逻辑顺序（先登录后取详情）和代码缩进方向一致，但视觉上像一团乱麻，结尾一排 }); 难以对齐；② 难维护——想在中间插一步、或调整顺序，要大段挪动嵌套层；③ 错误处理重复且散落——每一层都得写一遍 if (err) return ...，漏写一处就静默吞掉错误。

Node 风格回调约定：错误优先

注意上面每个回调的第一个参数都是 err。这是 Node.js 社区的事实标准——错误优先回调（error-first callback）：回调签名固定为 callback(err, data)。成功时 err 为 null、数据在后面的参数里；失败时 err 是一个 Error 对象。

错误优先约定
const fs = require("fs");

fs.readFile("a.txt", "utf8", function (err, data) {
  if (err) {          // 永远先判断 err
    console.error("读取失败", err);
    return;              // 别忘了 return，否则会继续往下执行
  }
  console.log(data);      // 走到这里说明成功
});
为什么要约定？异步回调里的错误没法用 return 把错误抛回调用者（调用 readFile 那行早就执行完了），也没法被外层 try/catch 捕获。错误只能"随结果一起送进回调"，于是社区统一约定放在第一个参数，调用方一眼就知道去哪儿查错。

陷阱清单

陷阱说明 / 后果
try/catch 抓不到try 包住 setTimeout，回调里抛的错逃出了 try 块（执行时调用栈早已不同），catch 形同虚设。
this 丢失把对象方法当回调传出去（如 setTimeout(obj.fn)），调用时 this 不再指向 obj。需用箭头函数或 .bind(obj) 固定。
回调被多次触发同一个回调被错误地调用了两次（如重复绑定事件、重试逻辑没去重），导致逻辑执行两遍。可加 called 标志位防护。
错误处理散落每层都写 if (err) return，重复且容易漏；漏一处就静默失败，极难排查。
忘记 return判断 err 后没 return，错误分支和成功分支同时跑，引发二次报错。
陷阱演示：try/catch 抓不到异步错误 + this 丢失
// ① try/catch 抓不到异步回调里的错误
try {
  setTimeout(() => {
    throw new Error("炸了");  // 此时 try 块早已执行完毕
  }, 0);
} catch (e) {
  console.log("抓到了吗？");   // 不会执行，错误变成未捕获异常
}

// ② this 丢失
const timer = {
  id: 42,
  start() {
    setTimeout(function () {
      console.log(this.id);  // undefined！this 不再是 timer
    }, 0);
    // 修复：用箭头函数继承外层 this —— () => console.log(this.id)
  }
};
timer.start();

引出解决方案

回调地狱的根源在于：异步结果只能"困"在回调内部，无法像同步代码那样用返回值往外传、用 try/catch 统一兜错。下一篇的 Promise 用"链式 .then()"把金字塔拍扁成一条直线，并提供统一的 .catch() 收口错误；再下一篇的 async/await 更进一步，让异步代码"长得像同步"，连 try/catch 都能正常用了。它们要解决的，正是本篇暴露的全部痛点。

小结：① 回调 = 把函数作为参数传入，待某事完成后被调用；② 同步回调（map）原地执行、顺序可预测，异步回调（setTimeout/事件/请求）晚于同步代码、由事件循环调度；③ 异步必须用回调，因为单线程不能同步等待否则阻塞；④ 多个异步操作前后依赖时层层嵌套形成回调地狱——难读、难维护、错误处理散落；⑤ Node 约定错误优先 callback(err, data)；⑥ 牢记陷阱：try/catch 抓不到异步错误、this 丢失、回调重复触发、忘写 return；⑦ 这些痛点正是 Promise 与 async/await 要解决的。
