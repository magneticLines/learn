作用域与闭包

作用域决定一个变量"在哪里能被看见"，闭包则让函数把它出生时的环境"随身携带"。这两个概念是 JS 模块化、私有数据、异步回调能成立的底层支撑——理解它们，你就理解了 JS 函数为什么"有记忆"。

它解决了什么问题

如果所有变量都是全局的，任意一处代码都能读写它，大型程序会变成一锅"谁都能改的乱炖"：命名冲突、意外覆盖、无法追踪谁动了数据。作用域把变量的可见范围限制在一块代码内，实现了"隔离"；而闭包更进一步，让函数能在外层环境已经结束后，仍然安全地访问那块被它"锁住"的变量——这正是实现数据私有化（无 private 关键字时的替代方案）、状态记忆（计数器、缓存）、回调保留上下文（异步、事件处理）的基础。可以说，闭包是 JS 把"函数当一等公民"这件事用到极致的产物。

作用域的三种种类

作用域就是变量的"有效范围"。JS 里有三层：

种类边界说明
全局作用域整个程序任何地方都能访问；浏览器里挂在 window 上（仅 var / 函数声明）
函数作用域一个函数体 {}var 和函数内声明的变量只在函数内可见
块级作用域一对 {}（if / for 等）ES6 新增，let / const 才享有，var 没有
块级作用域：let 被 {} 关住，var 漏出来
if (true) {
  var a = 1;      // 函数作用域，块外仍可见
  let b = 2;      // 块级作用域，只在这对 {} 内有效
}
console.log(a);      // 1
console.log(b);      // ReferenceError: b is not defined

词法作用域：由"写在哪"决定，不是"在哪调用"

这是最容易被误解的一点。JS 采用词法作用域（也叫静态作用域）：一个函数能访问哪些外层变量，在代码书写的那一刻就已经定死了，由它嵌套的物理位置决定，与它被谁、在哪里调用毫无关系。

关键考点：value 看的是定义处，不是调用处
const value = "全局";

function inner() {
  console.log(value);   // 它定义在全局，就只认全局的 value
}

function outer() {
  const value = "outer 内";
  inner();             // 虽然在 outer 里调用，但 inner 不看这里
}

outer();                // 打印 "全局"，不是 "outer 内"
对比记忆：与之相对的是动态作用域（如 Bash），变量按调用栈查找。JS 的变量查找是词法的（看定义位置），但 this 的绑定却是动态的（看调用方式）——这两个机制独立，初学者常把它们混为一谈，下一篇会专门讲 this。

作用域链：变量是怎么被找到的

当代码访问一个变量，引擎会从当前作用域开始，沿着词法嵌套关系一层层向外查找，直到全局；找到第一个同名变量就停下，全程找不到才报 ReferenceError。这条由内到外串起来的查找路径，就是作用域链。

作用域链：逐层向外查找
const g = "g";
function a() {
  const x = "x";
  function b() {
    const y = "y";
    console.log(y, x, g);  // y 在本层；x 向上一层找；g 一直找到全局
  }
  b();
}
a();                       // "y" "x" "g"
陷阱：作用域链只向外，不向内。外层函数访问不到内层函数的变量。另外查找是逐层就近的——内层有同名变量会"遮蔽"外层（shadowing），别误以为改的是外层那个。

闭包到底是什么

一句话定义：闭包 = 函数 + 它定义时所捕获的外层变量环境。当一个内层函数引用了外层函数的变量，并且这个内层函数被传递到外层函数之外（作为返回值、回调、事件处理器等），即使外层函数早已执行完毕，那些被引用的变量也不会被销毁——它们被这个函数"锁住"了。

最小闭包：count 活过了 makeCounter 的执行
function makeCounter() {
  let count = 0;          // 本该随函数结束而消失的局部变量
  return function () {
    count++;               // 内层函数引用了它 —— 形成闭包
    return count;
  };
}

const c = makeCounter();   // makeCounter 执行完，但 count 没被回收
console.log(c());        // 1
console.log(c());        // 2 —— count 被记住了

为什么内层函数能"记住"外层变量？

原理在于引用计数与可达性。函数被创建时，引擎会给它挂上一个隐藏的内部引用（规范里叫 [[Environment]]），指向它出生时所处的词法环境。只要这个函数对象还"活着"（被变量 c 持有），它指向的那个环境就仍然可达，垃圾回收器便不会清理其中的 count。所以闭包不是"拷贝了一份变量的值"，而是持有了对那块变量内存的活引用——这也是下面几个陷阱的根源。

闭包的典型应用

① 数据私有化 / 模块模式

JS 早期没有 private，闭包就是实现真正私有状态的标准手段：把变量关在函数里，只暴露受控的方法。

模块模式：balance 外部无法直接访问
function createAccount() {
  let balance = 0;                 // 私有，外界拿不到
  return {
    deposit(n) { balance += n; },
    getBalance() { return balance; }
  };
}
const acc = createAccount();
acc.deposit(100);
console.log(acc.getBalance());  // 100
console.log(acc.balance);       // undefined —— 碰不到私有变量

② 函数柯里化

柯里化是"把多参数函数拆成一串单参数函数"，每一层用闭包记住已经传入的参数。

柯里化：每层闭包锁住一个参数
function add(a) {
  return function (b) {     // 闭包记住了 a
    return a + b;
  };
}
const add5 = add(5);        // 固化 a = 5
console.log(add5(3));      // 8
console.log(add5(10));     // 15

③ 防抖 / 节流的引子

防抖、节流需要在多次调用之间"记住"上一次的定时器或时间戳——这个跨调用的私有状态，正是靠闭包保存的。

防抖：timer 被闭包跨次调用保留
function debounce(fn, delay) {
  let timer;                     // 闭包变量，每次调用都访问同一个
  return function (...args) {
    clearTimeout(timer);     // 用上一次留下的 timer
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

陷阱一：循环 + 闭包（呼应 var / let）

这是闭包最经典的坑，也是面试高频题。根因是闭包持有的是变量的引用，不是当时的值。var 在整个循环里只有一个 i，三个回调闭包共享它；等异步回调真正执行时，循环早结束，i 已是 3。

反例：三个闭包共享同一个 i
// 反例：全部打印 3
for (var i = 0; i < 3; i++)
  setTimeout(() => console.log(i));  // 3 3 3

问题出在"只有一个 i，且回调延迟读取它"。let 在每轮迭代都新建一个块级绑定，每个闭包捕获各自独立的副本，问题自然消失。

正例：let 每轮独立绑定
// 正例：打印 0 1 2
for (let j = 0; j < 3; j++)
  setTimeout(() => console.log(j));  // 0 1 2

// ES6 前的老办法：用 IIFE 手动给每轮造一个独立作用域
for (var k = 0; k < 3; k++)
  (function (k) {
    setTimeout(() => console.log(k));  // 0 1 2，每次传入当前值的拷贝
  })(k);

陷阱二：闭包共享同一变量

同一个外层函数里定义的多个内层函数，共享同一份闭包变量。一个改了，另一个看到的就是改后的值——这有时是特性，有时是 bug。

两个函数共享同一个 n
function make() {
  let n = 0;
  return {
    inc() { n++; },
    get() { return n; }   // 与 inc 共享同一个 n
  };
}
const m = make();
m.inc(); m.inc();
console.log(m.get());      // 2

陷阱三：闭包的内存占用与泄漏

闭包让变量"活得更久"是把双刃剑：只要闭包函数还被引用，它捕获的整个环境就无法被回收。如果不小心捕获了大对象（如整个 DOM 节点、大数组），又把这个闭包长期挂在全局或事件监听上，内存就会被白白占用，形成内存泄漏。

反例 → 正例：及时释放不再需要的闭包
// 反例：bigData 被回调闭包长期持有，无法回收
function leak() {
  const bigData = new Array(1000000).fill("x");
  window.onresize = function () {
    console.log(bigData.length);   // 只用了 length，却锁死整个数组
  };
}

// 正例：只捕获真正需要的最小数据；用完解绑回调
function noLeak() {
  const len = new Array(1000000).fill("x").length;  // 大数组用完即可被回收
  const handler = () => console.log(len);
  window.addEventListener("resize", handler);
  return () => window.removeEventListener("resize", handler);  // 提供销毁手段
}
面试高频追问："闭包会导致内存泄漏吗？"标准答案是——闭包本身不是泄漏，不再使用却仍被引用才是泄漏。规避之道：闭包只捕获最小必要数据、组件卸载/元素移除时主动解绑事件监听、不需要时把持有闭包的变量置为 null。
小结：① 作用域分全局/函数/块级，let/const 才有块级作用域；② JS 是词法作用域，能访问哪些变量由"写在哪"决定，不看"在哪调用"；③ 变量沿作用域链由内向外查找，就近遮蔽；④ 闭包 = 函数 + 它捕获的外层环境，本质是持有了变量的活引用，所以函数"有记忆"；⑤ 它撑起了数据私有化、柯里化、防抖节流；⑥ 三大坑：循环 + 闭包共享 var（用 let 解）、多函数共享同一变量、捕获大对象不释放导致泄漏。
