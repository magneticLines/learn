深拷贝与浅拷贝

"我只是复制了一下数据，怎么改副本连原对象也变了？"——这是几乎每个 JS 开发者都踩过的坑。它的根源不在拷贝本身，而在于 JS 的值类型与引用类型之分。理解这一点，深浅拷贝的所有行为就都顺理成章了。

它解决了什么问题

回到 01 篇讲过的根基：原始类型按值存储，对象类型按引用存储。当你写 const b = a 而 a 是对象时，拷贝的不是对象内容，而是指向同一块堆内存的地址——a 和 b 是同一个对象的两个名字。于是改 b 就是改 a。

大多数时候我们真正想要的是一份独立的副本：修改它不会牵连原对象。这就是拷贝要解决的问题。而"拷贝到第几层"的差异，分出了浅拷贝与深拷贝。

问题的源头：赋值只拷引用
const a = { name: "Tom" };
const b = a;            // b 拿到的是同一个地址，不是新对象
b.name = "Jerry";
console.log(a.name);     // "Jerry" —— a 也跟着变了
console.log(a === b);     // true —— 它们就是同一个对象

浅拷贝：只复制第一层

浅拷贝会新建一个对象，并把原对象第一层的属性逐个拷过去。对于第一层的原始值（数字、字符串），拷的是值，互不影响；但对于第一层里嵌套的对象，拷的依然是引用——副本和原对象的嵌套对象还是同一个。

常见的浅拷贝写法
const src = { a: 1, nested: { b: 2 } };

// ① Object.assign：把源对象的可枚举属性拷到目标
const c1 = Object.assign({}, src);
// ② 展开运算符（最常用，本质同 Object.assign）
const c2 = { ...src };

// 数组的浅拷贝：slice / concat / 展开
const arr = [1, [2]];
const a1 = arr.slice();
const a2 = [].concat(arr);
const a3 = [...arr];
陷阱：改副本的嵌套属性，原对象也跟着变。这是浅拷贝最容易让人翻车的地方——你以为已经"复制独立"了，结果嵌套层还是共享的。
反例：浅拷贝的嵌套对象仍共享
const src = { a: 1, nested: { b: 2 } };
const copy = { ...src };

copy.a = 100;            // 第一层原始值：安全，互不影响
console.log(src.a);        // 1 ✅

copy.nested.b = 99;     // 嵌套对象：拷的是引用！
console.log(src.nested.b); // 99 ❌ —— 原对象被改了
console.log(copy.nested === src.nested); // true

深拷贝：递归复制每一层

深拷贝会递归地把对象的每一层都重新创建一遍，得到一棵与原对象结构相同、但内存完全独立的新树。改副本的任何一层，原对象都不受影响。代价是：层数越深、对象越大，开销越高。

四种深拷贝方案对比

方案优点缺点 / 坑
JSON.parse(JSON.stringify(o))一行搞定，无需依赖，适合纯数据丢函数、undefined、Symbol；NaN/Infinity 变 null；Date 变字符串；BigInt 直接抛错；遇循环引用抛错
structuredClone(o)浏览器/Node 原生；支持循环引用、Date/RegExp/Map/Set/ArrayBuffer 等不能拷函数（抛 DataCloneError）；老环境不支持（需较新版本）
手写递归完全可控，可按需处理特殊类型要自己处理数组/循环引用/Date/RegExp，写全不易
lodash cloneDeep(o)久经考验，几乎覆盖所有类型，最省心引入第三方依赖，包体积增加
JSON 大法的多重丢失（高频考点）。它是用"序列化成字符串再解析回来"实现的，而 JSON 格式本身就表达不了函数、undefined、Symbol、BigInt（呼应 14 篇 JSON 序列化），这些信息在序列化时就被丢掉或报错了。
JSON 深拷贝会丢失什么
const o = {
  fn: () => 1,     // 函数 → 丢失
  u: undefined,    // undefined → 丢失
  s: Symbol(),     // Symbol → 丢失
  n: NaN,          // NaN → 变成 null
  d: new Date(),   // Date → 变成字符串
};
const copy = JSON.parse(JSON.stringify(o));
console.log(copy);  // { n: null, d: "2026-06-23T..." } —— fn/u/s 全没了

// 循环引用直接抛错：
const x = {}; x.self = x;
JSON.stringify(x); // TypeError: Converting circular structure to JSON
首选 structuredClone。它是现代标准方案，结构化克隆算法天然支持循环引用和大量内置类型，绝大多数场景比 JSON 法更安全。唯一记住的限制：不能克隆函数。
structuredClone：原生且支持循环引用
const x = { d: new Date(), list: [1, 2] };
x.self = x;             // 循环引用
const copy = structuredClone(x);
console.log(copy.d instanceof Date);  // true，Date 被正确克隆
console.log(copy.self === copy);      // true，循环引用被保留

手写一个递归深拷贝

面试常要求手写。核心三点：① 原始值直接返回（递归出口）；② 用 WeakMap 记录"已拷过的对象"，解决循环引用（否则无限递归导致栈溢出）；③ 区分数组与对象，特殊类型（Date/RegExp）单独处理。下面是抓住主干的简化版：

简化版递归深拷贝
function deepClone(obj, cache = new WeakMap()) {
  // 出口：原始值（含 null）直接返回，按值拷贝
  if (obj === null || typeof obj !== "object") return obj;

  // 特殊对象类型：另起炉灶，不能当普通对象遍历
  if (obj instanceof Date) return new Date(obj);
  if (obj instanceof RegExp) return new RegExp(obj);

  // 命中缓存：说明出现循环引用，返回已建好的副本，终止递归
  if (cache.has(obj)) return cache.get(obj);

  // 区分数组与普通对象
  const result = Array.isArray(obj) ? [] : {};
  cache.set(obj, result);  // 先登记，再递归 —— 顺序很关键

  for (const key in obj) {
    if (Object.hasOwn(obj, key))
      result[key] = deepClone(obj[key], cache); // 递归
  }
  return result;
}
为什么用 WeakMap 而不是 Map？键是原对象，WeakMap 对键是弱引用，拷贝结束后原对象若无其他引用可被垃圾回收，不会因为缓存而内存泄漏。"先 cache.set 再递归"也是关键：必须在进入子属性之前登记，否则循环引用时仍会无限递归。

什么时候其实只需要浅拷贝

深拷贝不是越深越好——它有实打实的性能成本。很多场景浅拷贝就够了，盲目深拷贝是过度设计：

  • 对象很扁平（只有一层原始值），浅拷贝即完全独立。
  • 不可变数据更新模式（React/Redux）：用 { ...state, count: state.count + 1 } 只替换变化的那层引用，未变的子树故意共享以复用、便于 === 快速比较。这里恰恰不要深拷贝。
  • 明确知道不会去改嵌套对象时，浅拷贝省内存省时间。
循环引用 → 栈溢出。任何不做缓存的递归深拷贝，碰到 x.self = x 都会无限递归直到 RangeError: Maximum call stack size exceeded。这也是 JSON 法和手写法的共同雷区——前者抛错，后者爆栈。
小结：① 根源是引用类型赋值只拷地址（呼应 01 篇）；② 浅拷贝（{...o}/Object.assign/数组 slice）只复制第一层，嵌套对象仍共享，是最常见的坑；③ 深拷贝方案优先 structuredClone（不拷函数），纯数据可用 JSON 法但会多重丢失，复杂场景用 lodash cloneDeep 或手写（WeakMap 防循环引用）；④ 别盲目深拷贝，扁平对象和不可变更新用浅拷贝更划算。

与其他知识点的联系

  • 关联 JSON 序列化：JSON.parse(JSON.stringify(o)) 深拷贝法的所有局限（丢函数、undefined、Symbol、NaN 变 null）本质上就是 JSON 序列化本身的局限——把两篇内容对照着看，才能真正理解"为什么会丢"而不是死记"会丢什么"。
  • 关联 localStorage 存对象：存进 localStorage 前的 JSON.stringify、取出后的 JSON.parse，其实就是一次隐式的"深拷贝"——从内存对象到字符串再解析回内存对象，读出来的和原对象已经是完全独立的两份数据，这也是理解 storage 数据不会和原对象共享引用的关键。
  • 关联 React 的不可变数据更新模式：React/Redux 里 { ...state, count: state.count + 1 } 这种写法故意只做浅拷贝，让未变化的子树继续共享引用，从而用 === 做快速比较（决定是否跳过重渲染）——这是"什么时候不该深拷贝"的最佳实战案例，直接呼应虚拟 DOM diff 靠引用比较提效的思路（React 篇第 2 节）。
  • 关联自定义错误类型：手写深拷贝遇到循环引用不做处理会栈溢出抛出 RangeError——这提示我们无论写什么递归函数，都要提前想清楚"递归出口在哪、会不会有极端输入让它失控"，这和自定义错误类型里"用类型分类处理不同异常"的思路一样，都是写健壮代码的基本功。
  • 关联 WeakMap 与内存管理：手写深拷贝用 WeakMap 而不是 Map 缓存已拷贝对象，是"用弱引用避免额外持有对象导致内存泄漏"这一模式的典型应用，和定时器/事件监听要及时清理（第 36 篇）背后的内存管理思想是一致的。
