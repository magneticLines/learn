可选链 ?. 与空值合并 ??

这是 ES2020 带来的一对"安全取值"利器。它们专门对付前端日常最烦人的两件事：深层取值时中间断了链子报错，以及给"可能没有的值"配默认值时把合法的 0 和空串误伤。理解它们，能让你的取值代码既短又稳。

它解决了什么问题

前端从后端拿到的数据常常是层层嵌套的对象，比如 res.data.user.profile.avatar。只要中间任何一层是 null 或 undefined，访问它的属性就会抛出 TypeError: Cannot read properties of undefined，让整个渲染崩掉。

过去为了防御，我们只能写一长串 && 短路：a && a.b && a.b.c，又啰嗦又容易漏。可选链 ?. 把这件事变成一个运算符——一旦左侧是 null/undefined 就短路，整个表达式直接返回 undefined，不再继续往下访问。而空值合并 ?? 则解决了它的搭档问题：当取出来的值确实"为空"时，怎么安全地给个默认值，而不会把合法的 0、'' 也当成空。

可选链 ?. 的三种形式

?. 的判断标准很明确：只看左侧是不是 null 或 undefined。是，就短路返回 undefined；不是，就照常访问。它有三种用法，分别对应属性、动态键、函数调用。

三种形式
const user = { name: "Tom", address: null };

// ① a?.b —— 访问属性
console.log(user?.address?.city);  // undefined（address 是 null，到此短路）

// ② a?.[key] —— 动态键 / 数组下标
const key = "name";
console.log(user?.[key]);          // "Tom"
console.log(user.tags?.[0]);      // undefined（tags 不存在，不报错）

// ③ fn?.() —— 调用可能不存在的方法/回调
const obj = {};
obj.onReady?.();                // 没有 onReady 也不报错，整体返回 undefined
短路是"整段"短路。在 a?.b.c.d 中，只要 a 为空，后面的 .b.c.d 全部跳过，不会因为 b 没写 ?. 而报错。所以通常只需在"可能为空的那一层"加 ?.，不必每一层都加。

对比旧写法 a && a.b && a.b.c

&& 短路虽然也能防御，但它的判断标准是"真值"——任何 falsy 值（0、''、false）都会让链条提前停下，语义不精确；而且层级越深写得越长。?. 只对 null/undefined 短路，语义更准、更短。

反例 → 正例
// 反例：&& 链冗长，且若 a.b 是 0 会被误判为"断链"
const city1 = user && user.address && user.address.city;

// 正例：可选链一行搞定，语义清晰
const city2 = user?.address?.city;

空值合并 ?? ：精准的默认值

x ?? y 的规则只有一句话：仅当 x 为 null 或 undefined 时，才取 y；否则保留 x。它和可选链是天生一对——?. 取不到时给你 undefined，?? 紧接着兜底。

?. 与 ?? 配合兜底
const avatar = res?.data?.user?.avatar ?? "/default.png";
// 链路任意一层为空 → undefined → ?? 兜底为默认头像

重点对比：?? 与 ||

这是最高频的考点和实战坑。两者都能给默认值，但触发条件完全不同：|| 对所有 falsy 值都取右侧，而 ?? 只认 null/undefined。

左侧值x || 默认x ?? 默认
undefined / null取默认取默认
0取默认 ⚠️保留 0 ✅
''（空串）取默认 ⚠️保留 '' ✅
false取默认 ⚠️保留 false ✅
其他真值保留保留
反例：|| 把合法的 0 替换掉了
let count = 0;  // 用户填的就是 0，是合法值

// 反例：0 是 falsy，被 || 当成"空"，错误地变成 10
const bad = count || 10;   // 10 ❌ 把用户的 0 弄丢了

// 正例：?? 只在 null/undefined 时兜底，0 被正确保留
const good = count ?? 10;   // 0 ✅
选择口诀："只想排除空（null/undefined）"用 ??；"想排除所有空值包括 0、空串、false"才用 ||。表单数值、计数、布尔开关这类合法值里包含 falsy 的场景，几乎都该用 ??。

陷阱与考点

陷阱一：?? 不能与 || / && 直接混用。因为它们的优先级容易让人误解，JS 规定必须用括号显式分组，否则直接是语法错误（SyntaxError）。

混用必须加括号
// 反例：SyntaxError，引擎拒绝模糊的优先级
const x = a || b ?? c;

// 正例：用括号表明意图
const y = (a || b) ?? c;

陷阱二：?. 不能用于赋值左侧。它只能读，不能写——a?.b = 1 是非法的。

?. 只读不可写
user?.address.city = "BJ";  // ❌ SyntaxError: Invalid left-hand side

陷阱三：别滥用 ?.，它会掩盖真正的逻辑错误。如果某个值按业务逻辑本就不该为空，给它加 ?. 只会让程序静默地返回 undefined 继续往下跑，把 bug 推到更远、更难排查的地方。?. 是给"合理地可能不存在"的值用的，不是用来无脑消除报错的。

对比另一个易混点：?.（可选链）和 ??（空值合并）虽然都带 ?，作用却不同——前者负责"安全地取"，后者负责"取空时兜底"。它俩常组合，但解决的是链路的不同环节。
小结：① ?. 在左侧为 null/undefined 时短路返回 undefined，三种形式 a?.b、a?.[key]、fn?.()，比 && 链更短更准；② ?? 仅对 null/undefined 兜底，是 ?. 的黄金搭档；③ 牢记 ?? vs ||——|| 会误伤 0/''/false，含合法 falsy 的场景请用 ??；④ ?? 与 ||/&& 混用要加括号，?. 不能放在赋值左侧，更不要用 ?. 掩盖本该报错的逻辑问题。
