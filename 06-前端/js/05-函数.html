函数（声明 / 表达式 / 箭头函数 / 默认参数）

函数是 JS 组织逻辑、复用代码的最小单位，更是这门语言"一等公民"思想的核心载体。声明方式、参数处理、箭头函数的特殊行为，每一处都藏着会反复咬人的坑——搞清楚它们，才能驾驭后面的闭包、回调与异步。

它解决了什么问题

没有函数，代码就只能从上到下一条道走到黑，重复逻辑得复制粘贴，改一处要改十处。函数把一段逻辑封装成可复用、可命名、可传递的单元，输入参数、输出结果，内部细节对外隐藏。JS 更进一步：函数本身就是一种值（对象的一种），能赋给变量、当参数传、被返回——这就是"一等公民"。正因如此，JS 才能用回调、高阶函数、闭包这些手法写出灵活的异步与函数式代码。但灵活的代价是细节多：同一个"函数"概念，光声明方式就有三四种，行为各不相同。

函数声明 vs 函数表达式

这是最容易混淆、也是面试最爱考的一组对比。核心区别只有一个词：提升（hoisting）。

  • 函数声明（function fn() {}）会被整体提升到作用域顶部——连同函数体一起，所以可以"先调用后声明"。
  • 函数表达式（const fn = function() {}）只提升变量名，函数体留在原地。调用必须在赋值之后，否则要么 undefined 不是函数，要么 TDZ 报错。
提升差异：先调用后定义
// 函数声明：整体提升，提前调用没问题
sayHi();              // "hi" —— 正常运行
function sayHi() { console.log("hi"); }

// 函数表达式：只提升变量名 sayBye，此刻它还是 undefined
sayBye();             // ❌ TypeError: sayBye is not a function
var sayBye = function() { console.log("bye"); };

// 若用 let/const，提前调用会落在 TDZ：ReferenceError
为什么会这样？JS 引擎执行前有个"创建阶段"：先扫描作用域，把声明登记好。函数声明被完整登记（名字 + 函数体），而 var 只登记名字并初始化为 undefined。这就是声明能提前用、表达式不能的底层原因。

箭头函数：语法与本质区别

箭头函数（ES6）不只是"写法更短"的普通函数。它在语法上省略了 function 关键字，单表达式还能省略 return 和大括号；但更重要的是它在语义上是另一种东西——它没有自己的 this、arguments、prototype，也不能用 new 调用。

箭头函数的几种写法
const double = (x) => x * 2;      // 单表达式：隐式返回，省略 return 和 {}
const square = x => x * x;        // 单个参数可省括号
const add = (a, b) => a + b;      // 多参数必须带括号
const log = () => console.log("hi");  // 无参数用空括号

// 返回对象字面量要用括号包住，否则 {} 被当成函数体！
const make = id => ({ id, ok: true });

箭头函数 vs 普通函数

特性普通函数箭头函数
自己的 this有，调用时动态决定没有，继承定义处外层的 this
arguments 对象有没有（需用剩余参数 ...args）
能否 new可以（作构造函数）不能，会抛 TypeError
prototype 属性有没有
适用场景方法、构造函数、需动态 this回调、需要固定外层 this 的场合
设计意图：箭头函数发明出来，主要就是为了解决回调里 this 丢失的老问题（以前要写 const self = this 或 .bind(this)）。它透明地"穿透"自己去用外层的 this，所以特别适合 map、setTimeout、事件回调里访问外部上下文。

陷阱：箭头函数当对象方法，this 不指向对象

正因为箭头函数没有自己的 this，把它当对象方法用时，this 不会指向调用它的对象，而是沿用定义时外层的 this（在模块/全局顶层往往是 undefined 或 window）。这是初学者最常踩的坑。

反例 → 问题 → 正例
// 反例：箭头函数做方法，this 不是 user
const user = {
  name: "Ann",
  greet: () => console.log(this.name)
};
user.greet();        // undefined —— this 指向外层，不是 user！

// 正例：方法用普通函数（或简写法），this 才指向 user
const user2 = {
  name: "Ann",
  greet() { console.log(this.name); }
};
user2.greet();       // "Ann" ✅
记住对应关系：对象方法、原型方法、构造函数 —— 用普通函数；传给别人当回调、需要锁住外层 this —— 用箭头函数。this 的完整规则下一篇详讲，这里只需记住"箭头函数没有自己的 this"。

默认参数

ES6 之前，给参数设默认值要写 x = x || 10 这种丑陋且有 bug 的写法（传 0 或 "" 会被误判为缺省）。默认参数语法直接在形参后写 = 默认值，只在实参为 undefined 时才生效——传 null、0 都不会触发默认值。

默认参数从左到右求值
function greet(name = "Guest", greeting = "Hi") {
  return `${greeting}, ${name}`;
}
greet();                 // "Hi, Guest" —— 两个都用默认值
greet(undefined, "Hey");  // "Hey, Guest" —— 显式 undefined 才触发默认
greet(null);             // "Hi, null" —— null 不触发默认！

// 陷阱：默认值从左到右求值，后面的参数可引用前面的
function box(w, h = w) { return w * h; }
box(5);                // 25 —— h 默认取 w 的值
// 但反过来 box(w = h, h) 会报错：用到了还没初始化的 h（TDZ）

剩余参数 ...args

剩余参数（rest parameter）把多余的实参收集成一个真正的数组，写在形参末尾。它是箭头函数获取"全部参数"的唯一手段，也是 arguments 的现代替代品。注意别和"展开运算符"混淆——长得一样，方向相反：剩余是收集，展开是拆开。

剩余参数收集成数组
function sum(...nums) {      // nums 是真数组 [1,2,3]
  return nums.reduce((a, b) => a + b, 0);
}
sum(1, 2, 3);          // 6

// 与展开运算符的对比（同样是 ...，但语义相反）
const arr = [1, 2, 3];
sum(...arr);             // 展开：把数组拆成 1,2,3 传进去 => 6

arguments 对象

arguments 是普通函数内部自动存在的一个特殊对象，包含调用时传入的全部实参。它在剩余参数出现之前是获取动态参数的唯一方式，但有两个坑：它不是真数组（是"类数组"，没有 map/reduce 等方法），且箭头函数里根本没有它。

arguments 不是真数组，箭头函数也没有它
function old() {
  console.log(arguments.length);   // 能取长度
  // arguments.map(...)  ❌ 报错：它没有数组方法
  const real = Array.from(arguments);  // 需先转成真数组
  return real;
}

// 箭头函数没有自己的 arguments，会向外层找（这里报错或拿到外层的）
const bad = () => arguments.length;  // ❌ ReferenceError（顶层）
// 正确做法：箭头函数一律用剩余参数
const good = (...args) => args.length;   // ✅
建议：新代码一律用剩余参数 ...args 取代 arguments——它是真数组、语义清晰、箭头函数也能用。arguments 现在主要是读老代码时才会遇到。

函数是"一等公民"

这是 JS 函数式编程的根基：函数和数字、字符串一样是值，可以赋给变量、放进数组、当参数传给别的函数（回调），也可以作为返回值返回（这就引出了下一篇的闭包）。能"接收函数或返回函数"的函数，叫高阶函数——map、filter、setTimeout 都是。

函数作参数与返回值
// 函数当参数传入（回调）—— 高阶函数 map
const nums = [1, 2, 3];
const doubled = nums.map(n => n * 2);   // [2, 4, 6]

// 函数当返回值 —— 工厂函数，返回一个新函数
function multiplier(factor) {
  return n => n * factor;          // 记住了 factor —— 闭包雏形
}
const triple = multiplier(3);
triple(10);                          // 30
小结：① 函数声明整体提升可先用后定义，函数表达式只提升变量名、不能提前调用；② 箭头函数没有自己的 this / arguments / prototype，不能 new，适合回调、不适合做对象方法；③ 默认参数只在实参为 undefined 时生效、从左到右求值；④ 剩余参数 ...args 是真数组，全面取代类数组的 arguments；⑤ 函数是一等公民，可传可返，由此衍生高阶函数与闭包。

与其他知识点的联系

  • 作用域与闭包：本篇末尾"函数是一等公民、可以返回函数"是下一篇闭包的直接铺垫——multiplier(factor) 返回的内层函数记住了 factor，这个"记住"的机制就是闭包，下一篇会深入讲它为什么能在外层函数执行完后依然有效。
  • this 指向规则：本篇只提前预告了"箭头函数没有自己的 this、不适合做对象方法"这一条结论，完整的四条绑定规则（默认、隐式、显式、new）和优先级判定要看下一篇——这里先记住现象，下一篇才理解"this 由调用方式决定"的完整逻辑。
  • 解构赋值与展开运算符：本篇的剩余参数 ...args 和展开运算符 ...arr 用的是同一个 ... 符号，但语义相反（收集 vs 展开）——放在函数形参末尾是剩余参数，放在函数调用处或数组/对象字面量里是展开，这组对比在后面那一篇会被系统化讲清。
  • React Hooks 闭包陷阱：useCallback/useMemo 本质就是本篇"函数是一等公民、可以被传递"这个特性在组件里的应用；而组件每次渲染都会重新创建一份新函数，正是因为函数表达式（const fn = () => {}）本身不会像函数声明那样被"记住"和复用，这也解释了为什么需要 useCallback 来手动缓存函数引用。
  • TypeScript 类型系统：TS 里函数类型的写法（如 (a: number, b?: number) => number）直接对应本篇的默认参数、可选参数；箭头函数因为没有自己的 this，在 TS 中也不需要单独声明 this 参数类型，而普通函数作为对象方法时常需要显式标注 this 的类型以获得类型检查。
