JS · DOM 操作

创建 / 插入 / 删除节点

页面不是写死的——按钮点一下要冒出一行、列表要动态渲染、弹窗要随用随删。这一切都靠 JS 动态地"造节点、塞进去、再拔出来"。学会增删改 DOM，页面才真正活了起来；但 DOM 操作也是性能的重灾区，理解它何时变慢同样关键。

它解决了什么问题

HTML 写好后是一棵静态的 DOM 树。可真实应用里，数据是变化的：聊天来了新消息、表格要加一行、过滤后要删掉不匹配的项。我们需要在运行时程序化地修改这棵树——这就是节点的增、删、改。它的本质是：先用 JS 在内存里造出节点对象，再把它挂接到树的某个位置；浏览器一旦发现树变了，就会重新计算布局并绘制。理解"造节点"和"挂接"是两步，是后面所有性能优化的前提。

创建节点：先在内存里造好

新节点诞生时游离在文档之外，不在页面上，直到你把它插进 DOM 树。常用两个 API：

  • document.createElement(tag)——创建元素节点，如 <li>、<div>
  • document.createTextNode(text)——创建纯文本节点（天然防 XSS，因为它不解析 HTML）
造一个 li，但它还没出现在页面上
const li = document.createElement("li");
li.className = "item";
li.append(document.createTextNode("牛奶"));
// 此刻 li 只活在内存里，页面看不到它——必须插进 DOM 树才显示

插入节点：把它挂到树上

插入 API 有"老牌"和"现代"两套，新代码优先用现代方法（更灵活、更直观）。

方法作用特点
parent.appendChild(node)插到父元素末尾老牌；只能插 1 个节点，返回被插入的节点
parent.append(...nodes)插到父元素末尾现代；可插多个、可直接插字符串、无返回值
parent.prepend(...nodes)插到父元素开头现代；同样支持多个与字符串
node.before(...nodes)插到自己前面（成为兄弟）现代
node.after(...nodes)插到自己后面（成为兄弟）现代
parent.insertBefore(new, ref)插到 ref 节点前面老牌；ref 为 null 时等价于末尾追加
append vs appendChild
const ul = document.querySelector("ul");

// 老牌：一次只能一个节点，字符串会报错
ul.appendChild(li);

// 现代：一次插多个，还能混入字符串（自动转文本节点）
ul.append(li2, "纯文本也行", li3);
ul.prepend(li0);     // 插到最前面
li2.after(li2b);    // 作为 li2 的下一个兄弟插入
插字符串时的安全差异：append("<b>hi</b>") 会把它当作纯文本，页面上原样显示 <b>hi</b>，不解析标签——这点和 innerHTML 截然不同，因此 append 插用户输入是安全的。

insertAdjacent 系列：精确定位插入

当你想相对某个元素的"四个方位"插入时，insertAdjacentHTML / insertAdjacentElement 最顺手。第一个参数是方位：

  • "beforebegin"——元素自身前面
  • "afterbegin"——元素内部、第一个子节点之前
  • "beforeend"——元素内部、最后一个子节点之后（最常用）
  • "afterend"——元素自身后面
四方位插入
box.insertAdjacentHTML("beforeend", "<p>追加到末尾</p>");
box.insertAdjacentElement("afterbegin", node);
// HTML 版会解析标签；和 innerHTML+= 不同，它不会重建已有子节点
陷阱：insertAdjacentHTML 会解析 HTML 字符串，若拼入了用户输入就有 XSS 风险，和 innerHTML 同样危险。需要插入用户内容时用 insertAdjacentElement + textContent，或直接 createElement。

删除、替换与克隆

方法作用说明
node.remove()删除自己现代写法，直接调用，无需拿到父节点
parent.removeChild(node)从父节点删除子节点老牌；必须先有父节点引用
parent.replaceChild(new, old)用新节点替换旧节点老牌
node.replaceWith(...nodes)把自己换成若干节点现代；支持字符串
node.cloneNode(deep)克隆节点deep=true 连同子树一起复制，false 只复制自身
删除与克隆
li.remove();                 // 现代：简洁直接
ul.removeChild(li);       // 老牌：需要父节点

const copy = tpl.cloneNode(true);  // 深克隆整棵子树，适合做模板复用
ul.append(copy);
陷阱：cloneNode 不复制事件监听器（通过 addEventListener 绑定的）。克隆出的节点是"哑巴"，需要重新绑事件。内联 onclick 属性倒是会被复制，但本就不推荐用内联。

最关键的陷阱：节点会被"移动"而非"复制"

把一个已在文档中的节点再次插入别处，它不会变成两个——而是从原位置被剪切走，挪到新位置。因为一个 DOM 节点在树里只能有一个父亲。想要"复制一份"必须显式 cloneNode。

append 是移动，不是复制
listB.append(itemInA);
// itemInA 现在只存在于 listB 中，listA 里的它消失了——这是移动语义

// 想两边都有？先克隆再插
listB.append(itemInA.cloneNode(true));

性能重点：回流、重绘与批量插入

每次修改 DOM 树的几何结构，浏览器可能触发回流（reflow）——重新计算每个元素的位置和大小，接着重绘（repaint）。回流是昂贵操作。所以在循环里逐个 append 是性能反例：每次插入都可能让浏览器重新算一遍布局。

反例：循环里逐个插入
const ul = document.querySelector("ul");
for (let i = 0; i < 1000; i++) {
  const li = document.createElement("li");
  li.textContent = i;
  ul.append(li);  // ❌ 每轮都动了已在文档中的树，可能反复触发回流
}

正解是DocumentFragment——一个轻量的"内存容器"。它本身不属于任何文档、不参与渲染，你可以往里随便塞节点，最后一次性把整个 fragment 插进 DOM。这样只触发一次回流。插入后 fragment 会被"掏空"，只留下它的子节点进入文档。

正例：DocumentFragment 批量插入
const frag = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const li = document.createElement("li");
  li.textContent = i;
  frag.append(li);  // ✅ 塞进内存容器，不碰真实 DOM
}
ul.append(frag);   // 只此一次真正操作 DOM —— 一次回流搞定
另一种思路：拼字符串一次性 innerHTML。把 1000 行拼成一个 HTML 字符串再赋给 innerHTML，也只触发一次回流，且代码更短。代价是：会销毁并重建容器内所有旧节点（旧节点上的事件监听全丢失），且需自己防 XSS。DocumentFragment 不动旧节点、可保留节点引用，二者各有取舍。

innerHTML 拼接 vs createElement

维度innerHTML 拼接createElement
代码量少，一行字符串多，逐个属性设置
安全性解析 HTML，易 XSS天然安全（不解析标签）
事件绑定需事后查询再绑，且 += 会重建旧节点丢监听造好就能直接绑事件
保留节点引用难（节点被重建）易（直接持有变量）
切忌用 innerHTML += 追加。它会把容器现有内容序列化成字符串、再整段重新解析，等于销毁重建所有子节点：性能差、丢失事件监听、丢失表单输入状态。要追加请用 insertAdjacentHTML("beforeend", ...) 或 append。

节点删除后，事件监听需要清理吗

用 remove() 删掉节点后，若没有其它引用指向它，节点和它上面通过 addEventListener 绑的监听器会一起被垃圾回收，通常无需手动清理。但若你在别处（如某个数组、闭包）还存着这个节点的引用，它就无法回收，监听器也活着——这就是内存泄漏。一个稳妥做法是用事件委托：把监听绑在父容器上，子节点增删都不影响那一个监听器。

小结：① 创建用 createElement / createTextNode，新节点先游离在内存；② 插入优先现代 append/prepend/before/after（可多个、可字符串、无返回值），老牌 appendChild/insertBefore 一次一个；③ 删除用现代 remove()，替换用 replaceWith，复制用 cloneNode(true)（不带事件）；④ append 已有节点是移动不是复制；⑤ 批量插入用 DocumentFragment 把回流压到一次，远离循环里逐个 append 和 innerHTML +=。

与其他知识点的联系

  • 修改元素内容与属性（上一篇）：两篇合起来构成完整的 DOM 增删改能力——上一篇改"已存在元素"的内容和属性，本篇负责元素的"生老病死"（创建、插入、删除）。innerHTML += 会重建节点丢事件这个坑，在两篇里都作为反例出现，因为它同时触发了"内容替换"和"节点重建"两个动作。
  • 为什么 React 不用原生 DOM 操作（虚拟 DOM 的核心动机）：本篇是理解虚拟 DOM 价值的最佳对照——"循环里逐个 append 会反复触发回流，要用 DocumentFragment 批量插入"这个优化技巧，正是 React 内部做的事：先在内存里（虚拟 DOM，类似一个巨大的 fragment）算好整棵新树，diff 出最小变更集，再一次性提交给真实 DOM，从而把回流次数降到最低。手写代码要自己记住用 fragment，React 把这个优化做成了框架默认行为。
  • classList 操作（下一篇）：创建节点后常见的下一步就是给它设置初始 class 状态（如新插入的列表项要不要带 active），下一篇的 classList API 正好承接本篇"节点造好了，接下来怎么打扮它"的需求。
  • 原型链与继承： cloneNode 不复制事件监听器这个陷阱，根源在于事件监听器存储在浏览器内部的一个隐藏关联表里，而不是节点对象的可枚举属性或原型方法——这提醒我们"原型链能查到的方法"和"实例私有的运行时状态（如绑定的监听器）"是两个不同维度，克隆只复制了后者对应的属性和结构，不会魔法般地复制前者建立的行为关联。
  • Node.js 与内存管理：本篇末尾"节点删除后事件监听需要清理吗"这一节，本质是 JS 垃圾回收机制在 DOM 场景下的应用——只要没有其他引用指向被移除的节点，它和绑在其上的监听器就能被回收。这和 Node.js 后端代码里"闭包持有大对象导致内存泄漏"是同一套引用计数/可达性分析原理，只是发生场景从服务端换成了浏览器页面。
