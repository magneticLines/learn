Fetch API 与 HTTP 请求

前端的核心工作之一就是和服务器对话——拿数据、提交表单、上传文件。fetch 是浏览器内置的现代请求工具，它返回 Promise，天然适配上两篇学过的 async/await。但它有一个让无数新人栽跟头的"不报错"陷阱，本篇会重点拆解。

它解决了什么问题

在 fetch 出现之前，发请求只能用 XMLHttpRequest（XHR）——一个 1999 年遗留下来的 API。它基于事件回调，写法繁琐：要 new 对象、注册 onreadystatechange、判断 readyState、手动拼参数，一个简单的 GET 也要十几行。更糟的是它和 Promise 体系格格不入，没法直接 await，容易堆出"回调地狱"。

fetch（2015 年随 ES6 时代进入浏览器）用一句话解决了这些痛点：它返回 Promise。这意味着你可以用上两篇学的 async/await 把异步请求写得像同步代码一样线性、好读。它把"发请求"这件事抽象成了 fetch(url, options) 这一个函数，并用 Request / Response 两个标准对象描述请求和响应。

基本 GET：两步 await

这是初学者最容易混淆的地方。fetch 拿数据要 await 两次，因为它把"响应到达"和"响应体解析完成"拆成了两个异步阶段：

  • 第一次 await：等服务器返回响应头（HTTP status、headers）。此时 body 可能还在路上（想象一个几百 MB 的下载），所以 fetch 一拿到响应头就 resolve，给你一个 Response 对象。
  • 第二次 await：调 res.json() 才开始读取并解析整个响应体。它本身又是异步的，所以又返回一个 Promise。
基本 GET 请求
async function getUser(id) {
  // 第 1 次 await：拿到 Response 对象（只代表响应头已到）
  const res = await fetch(`/api/users/${id}`);
  // 第 2 次 await：把响应体按 JSON 解析（这一步才真正读完整个 body）
  const data = await res.json();
  return data;
}
陷阱：忘记第二个 await。新手常写成 const data = await fetch(url) 然后直接 data.name，结果发现拿到的是 Response 对象而不是数据——因为还没解析 body。Response 不是你的 JSON，它只是一个"信封"，.json() 才是拆信。

Response 对象：信封上的信息

第一次 await 拿到的 Response 上有这些常用成员：

成员类型说明
res.okbooleanstatus 在 200~299 之间为 true，否则 false。判断成功就看它
res.statusnumberHTTP 状态码，如 200、404、500
res.statusTextstring状态描述，如 "Not Found"
res.headersHeaders响应头对象，用 res.headers.get("Content-Type") 读取
res.json()Promise把 body 解析为 JS 对象
res.text()Promise把 body 读为纯文本字符串（HTML、CSV 等）
res.blob()Promise把 body 读为二进制 Blob（图片、文件下载）
陷阱：body 只能读一次。响应体是一个"流"（stream），res.json() / res.text() / res.blob() 三个里只能调一个、且只能调一次。读完流就枯竭了，再调第二个会抛 TypeError: body stream already read。要复用得先存下解析结果，或用 res.clone() 复制一份。

最重要的陷阱：fetch 不会因为 404/500 而 reject

这是 fetch 最反直觉、坑了最多人的设计。fetch 只在网络层面失败时才 reject——比如断网、DNS 解析失败、CORS 被拦截、请求被中断。只要服务器成功返回了一个响应，哪怕是 404 或 500，fetch 都认为"请求这件事成功了"，Promise 照常 resolve。

它的逻辑是：HTTP 404/500 是一次成功完成的 HTTP 交互，只不过服务器告诉你"资源不存在"或"我崩了"——这属于业务结果，不是请求本身的失败。所以判断业务成功与否，必须你自己检查 res.ok。

反例：以为 catch 能接住 404
try {
  const res = await fetch("/api/not-exist");  // 服务器返回 404
  const data = await res.json();      // 404 不抛错，代码继续往下走！
  render(data);                          // data 可能是错误页 HTML，解析直接出乱子
} catch (e) {
  // 永远进不来——因为 404 根本没让 fetch reject
  console.error("出错了", e);
}
正例：手动检查 res.ok 并主动抛错
async function getJSON(url) {
  const res = await fetch(url);
  // 把"HTTP 错误状态"显式转成异常，交给上层 catch 处理
  if (!res.ok) {
    throw new Error(`HTTP ${res.status} ${res.statusText}`);
  }
  return res.json();
}
记忆口诀：fetch 的 reject 只管"网络通不通"，不管"服务器答了什么"。所以每个 fetch 后面几乎都该跟一句 if (!res.ok) throw ...，这是写 fetch 的肌肉记忆。

发送 POST：写数据

GET 只传 URL，POST 要带请求体。三件事缺一不可：用 method 指定动词、用 headers 声明体的格式、用 body 放序列化后的数据。

POST JSON 数据
async function createUser(user) {
  const res = await fetch("/api/users", {
    method: "POST",
    // 必须声明 Content-Type，否则服务器不知道按 JSON 解析
    headers: { "Content-Type": "application/json" },
    // body 必须是字符串：对象要先 JSON.stringify
    body: JSON.stringify(user),
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}
陷阱：POST 忘设 Content-Type。不写这个头时，浏览器默认按 text/plain 发送，许多后端框架（如 Spring 的 @RequestBody）会因为 Content-Type 不匹配而拒绝解析，报 415 Unsupported Media Type。另一个常见错是把对象直接塞进 body 而忘了 JSON.stringify——那样会被转成无意义的 "[object Object]" 字符串。

取消请求与超时：AbortController

fetch 自身没有超时参数，也没有 .cancel()。要中断一个进行中的请求，得借助 AbortController：它产生一个 signal 传给 fetch，调用 controller.abort() 就会让那次 fetch 立刻 reject（抛 AbortError）。配合 setTimeout 即可实现超时。

带超时的 fetch
async function fetchWithTimeout(url, ms = 5000) {
  const ctrl = new AbortController();
  const timer = setTimeout(() => ctrl.abort(), ms);
  try {
    // 把 signal 交给 fetch；超时后 abort() 触发它 reject
    return await fetch(url, { signal: ctrl.signal });
  } finally {
    clearTimeout(timer);  // 别忘了清理定时器，避免内存泄漏
  }
}

CORS 跨域：浏览器的安全闸门

当你的页面（如 https://a.com）向另一个源（不同协议/域名/端口，如 https://api.b.com）发请求时，浏览器会启动 同源策略 的检查，这就是 CORS（跨域资源共享）。

关键认知：CORS 是浏览器拦的，不是服务器拒的。服务器其实可能已经正常处理并返回了响应，但如果响应头里没有 Access-Control-Allow-Origin 允许你的源，浏览器会在 JS 拿到结果前把它拦下，让 fetch reject。所以 CORS 报错无法靠改前端代码解决，必须后端配置允许的源——这是新人常误以为"我代码写错了"的典型陷阱。

对比：XHR / fetch / axios 怎么选

维度XMLHttpRequestfetchaxios（第三方）
异步模型事件回调Promise，可 awaitPromise，可 await
JSON 处理手动 JSON.parse手动 res.json()自动解析与序列化
HTTP 错误状态手动判断 status不 reject，需查 ok4xx/5xx 自动抛错
超时内置 timeout需 AbortController内置 timeout
拦截器无无有（统一加 token、错误处理）
体积/依赖内置内置需安装 npm 包
怎么选：新项目里 fetch 已足够，零依赖、标准化。但当项目需要统一注入鉴权头、全局错误处理、自动重试时，axios 的拦截器和"4xx/5xx 自动抛错"会省下大量样板代码——这正是它至今流行的原因。XHR 现在基本只在维护老代码或需要上传进度事件时才碰。
小结：① fetch 返回 Promise，配 async/await 写最顺；② 拿数据要两次 await——先 Response 再 res.json()；③ 最大的坑：404/500 不会 reject，必须手动 if (!res.ok) throw；④ POST 三件套：method + Content-Type 头 + JSON.stringify 的 body；⑤ body 只能读一次；⑥ 取消/超时用 AbortController；⑦ CORS 是浏览器拦的、要后端配；⑧ 需要拦截器和自动抛错就上 axios。

与其他知识点的联系

  • async/await（上一篇）：本篇"两次 await"的写法直接沿用上一篇的语法，上一篇错误处理小节提前点出的"fetch 只在网络层面失败时才 reject"，本篇用整整一节详细展开并给出了修复方案。
  • Promise：fetch 本身就返回一个 Promise，res.json()/res.text()/res.blob() 各自又返回新 Promise——这是 Promise 篇讲的"链式返回新 Promise"在真实 API 上的体现。Promise.all 并发多个 fetch、AbortController 配合超时，都是 Promise 篇静态方法的实战延伸。
  • try/catch/finally（下一篇）：本篇"正例：手动检查 res.ok 并主动抛错"的写法，把 HTTP 错误状态转成异常后交给 try/catch 兜底，是下一篇要系统讲解的错误处理机制的具体应用场景之一。
  • Fetch 与 Axios 的取舍：本篇对比表提到 axios 的拦截器和自动抛错，但更根本的区别是：axios 底层基于 XMLHttpRequest 而非 fetch（浏览器端），它自动做 JSON 序列化/反序列化，内置 timeout 和取消令牌，还有请求/响应拦截器可以统一注入 token、做全局错误处理——这些都是 fetch 需要手写 AbortController、手动 JSON.stringify/res.json()、且完全没有拦截器机制才能达到的效果。选型时"要不要拦截器和自动化"是关键决策点。
