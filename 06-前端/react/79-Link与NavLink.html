Link 与 NavLink：SPA 里正确的"跳转"方式

路由表建好了，怎么让用户点击切换页面？直觉是用 <a href>——但在 SPA 里，这会把你辛苦搭的单页应用"打回原形"。正确答案是 <Link>，而需要高亮当前页时用 <NavLink>。

为什么不能直接用 <a href>

原生 <a href="/about"> 的默认行为是：向服务器请求 /about，整页刷新。这会重新下载并执行所有 JS，重建整个 React 应用，之前的所有内存状态（登录态、表单输入、滚动位置）全部丢失。SPA 的"不刷新、丝滑切换"优势荡然无存。

<Link> 底层其实还是渲染成一个 <a> 标签（保留可访问性、右键"在新标签打开"等能力），但它拦截了点击事件：调用 e.preventDefault() 阻止默认跳转，改用 History API 的 pushState 修改 URL，再通知 Router 重新匹配、局部更新组件。

Link 基本用法——用 to 代替 href
import { Link } from 'react-router-dom';

function Nav() {
  return (
    <nav>
      {/* 注意是 to，不是 href */}
      <Link to="/">首页</Link>
      <Link to="/about">关于</Link>
      {/* to 也可以是对象，携带 query / hash */}
      <Link to={{ pathname: '/users', search: '?sort=name' }}>用户</Link>
    </nav>
  );
}
什么时候还是该用 <a>? 跳转到外部网址（https://google.com）时，用普通 <a href>，因为那本来就要整页离开。<Link> 只负责应用内部的路由跳转。

NavLink:能感知"是否为当前页"的 Link

导航栏经常需要高亮"当前所在页"的那个链接。<NavLink> 是 <Link> 的增强版：它会判断自己的 to 是否匹配当前 URL，匹配时自动进入 active 状态。

v6 里，className 和 style 可以写成函数，接收一个带 isActive 的对象，让你按激活状态动态返回样式：

NavLink——用 isActive 回调实现高亮
import { NavLink } from 'react-router-dom';

// className 传函数：根据 isActive 返回不同的类名
<NavLink
  to="/about"
  className={({ isActive }) => (isActive ? 'link active' : 'link')}
>
  关于
</NavLink>

// style 同理，也可以传函数
<NavLink
  to="/about"
  style={({ isActive }) => ({
    color: isActive ? '#61dafb' : '#94a3b8',
    fontWeight: isActive ? 'bold' : 'normal',
  })}
>关于</NavLink>

// children 也能传函数，拿到 isActive 渲染指示器等
<NavLink to="/msg">
  {({ isActive }) => (<>消息 {isActive && '●'}</>)}
</NavLink>
v5 vs v6 的 active 写法差异(高频考点)。 v5 用 activeClassName="active" / activeStyle={...} 两个独立属性；v6 删掉了它们，统一改成给 className/style 传函数，用参数里的 isActive 自行决定。好处是逻辑集中、更灵活（可以同时切换多个类、做三元判断），但从 v5 迁移时这里必改。

v6 还多了个 isPending（配合 data router 的加载态），以及默认注入的 CSS 类名 .active（不传函数时，激活的 NavLink 会自动带上 active 类，可直接写 CSS）。

end 属性:解决父路由被"误高亮"

NavLink 默认用前缀匹配：to="/" 会匹配所有以 / 开头的路径（即所有页面），导致"首页"链接永远高亮。加 end 要求完全匹配才激活：

end——只在 URL 完全等于 to 时才 active
// 没加 end：访问 /about 时，"首页"也会因为前缀 "/" 匹配而高亮 —— 错误
<NavLink to="/">首页</NavLink>

// 加 end：只有 URL 恰好是 "/" 时才高亮 —— 正确
<NavLink to="/" end>首页</NavLink>

场景用哪个

非"点击链接"式跳转（如提交后跳转）useNavigate（下一篇）

小结

  • SPA 内部跳转永远用 <Link to>，不要用 <a href>——后者会整页刷新、丢失所有状态。
  • Link 底层仍渲染成 <a>，但拦截点击、用 pushState 改 URL、局部重渲。
  • <NavLink> 是能感知"当前页"的 Link，通过 className/style 传函数并读 isActive 实现高亮。
  • v6 移除了 v5 的 activeClassName/activeStyle，统一为函数式写法。
  • to="/" 的导航项记得加 end，否则会因前缀匹配而永久高亮。

常见陷阱

  • 误用 href 而不是 to。把原生 <a> 的写法习惯带进 <Link>,写成 <Link href="/about">——这个属性对 Link 无效(Next.js 的 Link 才用 href,react-router 用的是 to),结果链接渲染出来但点击没有任何路由效果。
  • 给 NavLink 用 v5 的 activeClassName/activeStyle 写法。从 v5 迁移或抄旧教程时,容易写出 <NavLink activeClassName="active">,但 v6 已经删除了这两个属性,统一改成 className/style 传函数读 isActive,旧写法会被静默忽略,高亮功能失效却不报错,排查起来很隐蔽。
  • 忘记给根路径的导航项加 end。NavLink 默认前缀匹配,to="/" 会匹配一切路径,导致"首页"链接无论在哪个页面都常亮,新手很容易把这当成 CSS 问题去排查,实际上是路由匹配模式的问题。
  • 用 Link 跳转外部网址。把 <Link to="https://google.com"> 用于站外链接,虽然不一定报错,但语义和行为不对——Link 是为应用内路由设计的,跳外部站点应该用原生 <a href>,让浏览器走正常的整页导航。

面试高频考点

  1. 为什么 SPA 里要用 Link 而不是原生 a 标签?—— 原生 <a href> 会触发浏览器向服务器发起整页请求,导致页面刷新、JS 重新加载执行、React 应用状态全部丢失;<Link> 拦截点击事件、用 History API 的 pushState 修改 URL,再交给 Router 局部重渲,不产生整页刷新。
  2. Link 的实现原理是什么,它最终渲染成什么?—— Link 最终仍渲染为一个 <a> 标签(保留可访问性和"新标签打开"等原生能力),但给它加了 onClick 处理:调用 preventDefault() 阻止默认跳转,然后用 pushState/replaceState 变更 URL 并通知 Router 重新匹配。
  3. NavLink 的 v5 与 v6 用法有什么区别?—— v5 用两个独立属性 activeClassName/activeStyle;v6 移除了它们,统一让 className/style/children 都可以接收一个函数,函数参数里的 isActive(以及 data router 下的 isPending)由开发者自行决定如何渲染,逻辑更集中也更灵活。
  4. NavLink 默认的匹配规则是什么,如何避免父路由被误高亮?—— NavLink 默认用前缀匹配,任何以 to 为前缀的路径都会被判定为激活,这会导致 to="/" 之类的根路径永远高亮;加上 end 属性后改为要求 URL 与 to 完全匹配才激活。

与其他知识点的联系

  • BrowserRouter 与 HashRouter(第77篇):Link 拦截点击后调用的 pushState,正是第77篇讲的 History API;理解了 BrowserRouter 底层怎么改 URL 不刷新页面,才能理解 Link 为什么能在保留 <a> 标签可访问性的同时,做到"看起来像链接、实际不整页刷新"。
  • Routes 与 Route(第78篇):Link/NavLink 触发的 URL 变化,最终要靠 Routes 重新做一次最佳匹配才能决定渲染哪个组件——Link 负责"发起导航",Routes 负责"响应导航",两者是同一次跳转里前后相接的两个阶段。
  • useNavigate(下一篇):Link 是声明式导航(写在 JSX 里、靠用户点击触发),useNavigate 是编程式导航(写在代码里、靠逻辑触发)。两者最终都会落到同一套 History API 之上,只是触发方式和使用场景不同——本篇结尾的对比表已经点出了这层选择关系。
  • 动态段与 useParams(第81篇):本篇 Link 示例里传 to={{ pathname: '/users', search: '?sort=name' }} 这种带参数的跳转,和第81篇"如何跳转到带参数的详情页"是同一件事的两种写法——理解 Link 的 to 属性怎么拼 URL,才能理解详情页跳转链接该怎么写。
