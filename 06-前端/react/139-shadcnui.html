shadcn/ui(Tailwind 生态,可复制代码)

上一篇的 Ant Design 是装进 node_modules 的依赖——你 import 一个黑盒。shadcn/ui 反其道而行:它压根不是一个 npm 包,而是一套用 CLI 把组件源码直接复制到你项目里的方案。这个理念上的差异,决定了它适合谁、不适合谁。

不是依赖,是"复制粘贴生成器"

用 npx shadcn@latest add button,它不会往 package.json 加一行依赖,而是直接在你项目里生成一个 components/ui/button.tsx 文件——组件的完整源码就躺在你的仓库里,和你自己手写的组件没有任何区别。

CLI 用法:把组件"拉"进项目而不是"装"进依赖
# 初始化配置(选样式风格、Tailwind 路径等)
npx shadcn@latest init

# 把 Button 组件的源码复制进 components/ui/button.tsx
npx shadcn@latest add button dialog

底层技术栈是:Radix UI(下一篇要讲的无样式 headless 组件,负责行为和无障碍)+ Tailwind CSS(负责样式)。shadcn/ui 做的事情就是把这两者组合好、写成一份"参考实现",然后把这份实现的源码交给你。

和传统组件库的本质区别

Ant Design(传统组件库)
安装为 node_modules 依赖
组件是黑盒,通过 props/token 定制
升级:npm update 一下就行
想改内部实现?几乎不可能,
只能等官方支持或提 issue
shadcn/ui(复制式组件库)
组件源码直接进你的仓库
定制:直接改源码,没有限制
升级:官方源码变了要手动
比对、手动同步过来
组件完全归你所有,爱怎么改怎么改
什么时候选 shadcn/ui? 项目已经用 Tailwind、想要"看起来是自己写的"高度可定制组件、不介意团队里多一份需要维护的组件源码——这些场景下 shadcn/ui 很合适。如果你要快速搭一个后台系统、组件要"齐全"(表格、树、上传、复杂表单联动都要现成的),Ant Design 这类"箱子里啥都有"的传统库效率更高。

代价:升级不是 npm update

拿到源码的另一面是:官方修了 bug、加了新特性,你不会自动收到。 传统组件库升级版本号就行;shadcn/ui 官方如果更新了 button.tsx 的实现(比如修复了一个无障碍问题),你需要自己去看 diff,手动把改动合并进你项目里已经可能被你改过的那份源码——如果你改动很大,这个合并可能很痛苦。这是"完全拥有代码"必然要付出的维护成本,团队规模小、组件改动不频繁时问题不大;组件很多、迭代频繁时要有心理准备。
生成后的组件就是普通 React 组件,可以随意改
// components/ui/button.tsx —— 这就是你项目里的一个普通文件
import * as React from 'react';
import { cva } from 'class-variance-authority'; // 管理变体样式的小工具

const buttonVariants = cva(
  'inline-flex items-center justify-center rounded-md text-sm', // 基础 Tailwind 类名
  { variants: { variant: { default: 'bg-primary text-white', outline: 'border border-input' } } }
);
// 想改按钮圆角、加个新的 variant?直接改这个文件,不用等谁发新版

小结

  • shadcn/ui 不是 npm 依赖,而是用 CLI 把组件源码复制进你的项目,基于 Radix UI(行为)+ Tailwind(样式)组合而成。
  • 好处:组件完全归你所有,可以无限制定制,不受组件库 API 设计和版本节奏的约束。
  • 代价:升级不再是 npm update,官方源码更新后需要手动对比、手动合并进你已改动过的文件。
  • 选型上和 Ant Design 是两种取向——要"开箱即用、组件齐全"选传统组件库,要"高度定制、代码完全自主"选 shadcn/ui。
  • 下一篇拆开 shadcn/ui 的底层依赖之一 Radix UI,看它作为 headless 组件到底解决了什么问题。

常见陷阱

  • 当成普通组件库来"装"——看到 npx shadcn@latest add button 就以为这是 npm install 的变种，结果找不到 node_modules 里的包。它本质是代码生成器，不是依赖，思路要先转过来。
  • 升级全靠 npm update 的惯性——传统组件库升级版本号就完事，shadcn/ui 官方更新 button.tsx 后不会自动同步到你项目，得自己去看 diff、手动合并，忘了这一点就会一直用着有已知问题的旧版本源码。
  • 改动太随意导致以后没法合并升级——组件源码归你所有不代表可以毫无节制地改，改得越深，将来对比官方新版本、合并改动就越痛苦，尤其组件多、迭代频繁的项目要提前想好维护成本。
  • 把 Radix UI 和 Tailwind 的角色搞混——shadcn/ui 本身不提供行为逻辑也不提供样式引擎，Radix UI 管无障碍和交互行为，Tailwind 管样式，shadcn/ui 只是把两者组合成一份参考实现交给你。不理解这个分层，遇到问题就不知道该去哪个环节排查。
  • 选型时忽略"组件齐全度"的差距——想快速搭后台系统、需要表格/树/复杂表单联动这类现成重组件时硬选 shadcn/ui，会发现很多东西要自己从头拼，这类场景传统组件库（如 Ant Design）效率更高。

面试高频考点

  1. shadcn/ui 和 Ant Design 这类传统组件库的本质区别是什么？—— 本质区别在于分发方式：Ant Design 是打包进 node_modules 的黑盒依赖，通过 props/token 定制，升级靠 npm update；shadcn/ui 用 CLI 把组件源码直接复制进你的仓库，组件和你自己手写的代码没有区别，定制没有限制，但升级也不再是版本号问题，而是要手动对比、合并官方源码的变化。
  2. 为什么 shadcn/ui 选择"复制源码"而不是"发 npm 包"这种设计？—— 因为它要解决的是传统组件库"黑盒不可控"的痛点：一旦组件以依赖形式存在，你能改的只有官方开放的 API 和 token，遇到没预留的定制点就只能等官方支持或提 issue。把源码直接交给使用者，代价是升级要自己维护，但换来的是没有任何定制上限——这是一种用维护成本换灵活性的取舍，而不是免费的午餐。
  3. shadcn/ui、Radix UI、Tailwind 三者是什么关系？—— 这是分层组合关系：Radix UI 是 headless 组件，只负责行为逻辑和无障碍（键盘导航、焦点管理、ARIA 属性），不带任何样式；Tailwind 负责视觉样式；shadcn/ui 把这两者组合、写成一份具体实现（比如一个带样式的 Button），再通过 CLI 把这份实现的源码交给你。三者不是竞争关系，而是各管一层。
  4. shadcn/ui 这种模式适合什么样的团队，不适合什么样的团队？—— 适合项目已经用 Tailwind、希望组件"看起来是自己写的"、需要深度定制、并且团队能接受多维护一份组件源码的场景；不适合追求开箱即用、组件要"齐全"（复杂表格、树、上传等现成重组件）、或者团队没有精力去手动跟进官方源码更新的场景——这本质是"自主可控"和"维护成本"之间的权衡。

与其他知识点的联系

  • AntDesign：本篇开篇就对比了两者的本质区别——AntDesign 是装进 node_modules 的黑盒依赖，shadcn/ui 是复制进项目的源码。这个差异直接影响性能优化的可操作性：AntDesign 只能通过官方开放的 API（比如 virtual 属性）间接触发内部优化，shadcn/ui 因为组件源码就在项目里，可以直接打开文件给内部子组件加 memo、useMemo，不需要等官方支持。
  • React.memo / useMemo / useCallback：shadcn/ui 生成的组件"和你自己手写的组件没有任何区别"——这意味着如果某个 shadcn/ui 组件在大数据量场景下出现性能问题，完全可以直接打开 components/ui/xxx.tsx，用前面几篇讲的 memo 包裹内部子组件、用 useMemo 缓存计算、用 useCallback 稳定传给子组件的回调，这是黑盒组件库（如 AntDesign）做不到的定制自由度。
  • 虚拟滚动 TanStack Virtual：如果 shadcn/ui 没有内置某个大数据表格或长列表组件的虚拟滚动实现，因为组件源码直接在项目里，可以直接在这份源码里接入 @tanstack/react-virtual 的 useVirtualizer，把虚拟滚动逻辑写进组件内部——这是"源码归你所有"这一特性在性能优化场景下最直接的体现。
  • 组件组合与嵌套：13 篇讲的组合优先于继承，在 shadcn/ui 里体现得更彻底——它甚至不提供一个"组件库"意义上的整体，而是把 Radix UI（行为）和 Tailwind（样式）组合成一份份独立的参考实现，每份实现职责单一、互不依赖，这正是"把复杂 UI 拆成小组件再组合"这条设计哲学在生态选型层面的延伸。
  • 状态下移与内容提升模式：用 shadcn/ui 构建的 Dialog、Popover 这类组件时，同样可以应用 132 篇讲的"内容提升"——把不依赖开关状态的内容作为 children 传入，效果和用在 AntDesign 的 Modal 上完全一致；区别是 shadcn/ui 下如果这个模式不够用，还能直接改组件源码，而不必止步于结构调整这一层。
