404 页面:用 path="*" 兜底未匹配的路径

用户手滑输错 URL、点了失效的旧链接,如果所有 <Route> 都没匹配上,页面就是一片空白——比报错还糟。用通配路由 path="*"兜底,给一个友好的 404 页面。

问题:没匹配到的路径 = 空白页

React Router 会遍历 <Routes> 里的所有 <Route>,挑一个最匹配的渲染。如果一个都没匹配上,它什么也不渲染。用户访问 /typo-path 看到的是一片死寂的空白,不知道是自己输错了还是网站坏了。

解法:通配符 path="*"

* 是一个特殊的路径,匹配任意未被其他路由命中的路径。把它放在路由表里,专门捕获"漏网之鱼",渲染 404 组件。

通配路由兜底 404
<Routes>
  <Route path="/"       element={<Home />} />
  <Route path="/about"  element={<About />} />
  <Route path="/posts/:id" element={<Post />} />

  {/* ★ 以上都没命中时,兜底 404 */}
  <Route path="*" element={<NotFound />} />
</Routes>
NotFound 组件:友好提示 + 回家的路
import { Link } from 'react-router-dom';

function NotFound() {
  return (
    <div>
      <h1>404</h1>
      <p>页面走丢了</p>
      <Link to="/">返回首页</Link>
    </div>
  );
}

放置位置:v6 里顺序不重要,但习惯放最后

这是一个重要的认知升级。React Router v5 靠"从上到下第一个匹配",所以 404 必须放在 <Switch> 最底部,否则会提前拦截。

v6 改用"最佳匹配"算法——它会评估所有路由,选特异度最高的那个,path="*" 特异度最低,永远垫底。所以理论上 v6 里 path="*" 放哪都行。但为了可读性,仍强烈建议放在路由表最后,一眼就知道"这是兜底"。

v5 vs v6 匹配机制的差异
v5:<Switch> 从上往下,谁先匹配用谁——顺序敏感,/ 若不加 exact 会吃掉所有路径。
v6:<Routes> 做"最佳匹配"(rank by specificity),不再需要 exact,顺序不影响结果。这消灭了 v5 里"路由顺序写错导致 404 提前触发"的经典坑。

嵌套路由里的局部 404

path="*" 也能放进嵌套路由,做"某个模块内部的 404"。比如后台布局里,未匹配的子路径应显示带侧栏的 404,而不是整站的空白 404:

局部 404:保留布局外壳
<Route path="/app" element={<Layout />}>
  <Route path="dashboard" element={<Dashboard />} />
  {/* /app/xxx 未命中 → 带侧栏的 404,而非整站 404 */}
  <Route path="*" element={<AppNotFound />} />
</Route>
SPA 的 404 不是真正的 HTTP 404
前端路由的 404 页面,HTTP 状态码其实是 200——服务器返回了正常的 index.html,是 JS 在浏览器里判断"没匹配到路由"才显示 404 界面。这对 SEO 不友好(搜索引擎认为该页面正常存在)。若在意 SEO,需要 SSR(如 Next.js)在服务端返回真正的 404 状态码。

另外别忘了配置服务器把所有路径都回退到 index.html(nginx 的 try_files),否则刷新 /about 会直接命中服务器的真 404,连 React 都没机会跑。
* 也能取参数:在组件里 const params = useParams(),通配匹配到的路径存在 params['*'] 里。可以用它记录"用户想访问但不存在的路径",上报日志分析哪些死链最多。

常见陷阱

  • 带着 v5 的思维在 v6 里纠结通配路由的位置:v5 的 <Switch> 是"从上到下第一个匹配就用",path="*" 放错位置(比如放在最前面或没写 exact 的 / 之后)会提前拦截所有路由。v6 换成了"最佳匹配"算法,* 特异度最低永远垫底,放哪里结果都一样——但仍应放在最后,否则读代码的人会误以为顺序敏感,增加理解成本。
  • 忘记服务器端的 fallback 配置,导致刷新页面直接 404:SPA 的路由匹配完全在浏览器里的 JS 完成,如果用户直接刷新 /about 或分享这个链接给别人打开,请求会先打到服务器,如果服务器没有配置"所有路径都回退到 index.html"(如 nginx 的 try_files),会命中服务器真正的 404,连 React 代码都没机会跑起来,看到的是一片毫无样式的错误页。
  • 误以为前端的 404 页面等同于真正的 HTTP 404:即使正确显示了 <NotFound /> 组件,HTTP 响应状态码通常仍是 200(服务器认为自己正常返回了 index.html)。搜索引擎爬虫看到的是"这个页面存在且正常",对 SEO 不友好。如果业务在意这一点,得用 SSR 方案在服务端真正返回 404 状态码。
  • 嵌套路由里忘了给每个模块单独兜底,导致局部 404 变成整站空白:只在最外层 <Routes> 加了一个全局 path="*",某个带侧栏布局的模块内部路径打错时,会直接跳出布局显示整站级的 404,用户瞬间"丢失"了导航栏和上下文。应该在每个有独立布局的嵌套路由里也放一个局部的 path="*"。

面试高频考点

  1. React Router v6 里 path="*" 的匹配规则是什么,和 v5 有什么本质区别?—— v6 使用"最佳匹配"(rank by specificity)算法,会评估所有 <Route> 并挑选特异度最高的一个渲染,* 特异度最低,只有其他路由都没命中时才会生效,因此不再需要关心声明顺序。v5 的 <Switch> 则是严格从上到下第一个匹配就采用,顺序错了会导致某些路由永远匹配不到。
  2. 为什么 SPA 前端 404 页面配合 HTTP 200 状态码会有问题?怎么解决?—— 因为路由匹配发生在浏览器端的 JS 里,服务器对任何路径请求(经过 fallback 配置后)都会返回同一个 index.html,状态码天然是 200。这让搜索引擎爬虫误判页面存在,损害 SEO。解决办法是引入 SSR(如 Next.js),在服务端就能判断路由不存在并返回真正的 404 状态码。
  3. 为什么部署 SPA 时必须配置服务器的路径回退(fallback)规则?—— 因为客户端路由完全依赖 JS 加载后在浏览器里做匹配,如果用户直接访问或刷新一个非根路径(如 /about),这个请求会先发到服务器;如果服务器没有配置"找不到对应静态文件就返回 index.html",就会命中服务器自身的 404,页面直接空白报错,JS 都没机会执行。

与其他知识点的联系

  • Outlet 占位符 / 布局路由：局部 404 之所以能"保留布局外壳"，靠的就是把 path="*" 作为某个布局路由的子路由放进去——未匹配的子路径依然会先命中父级 Outlet，再由通配路由渲染兜底内容，而不是跳出整个布局显示一片空白。
  • index 路由：index 路由和通配路由分别覆盖"父路径被精确命中"与"什么都没匹配上"这两种边界情况，二者互不冲突、可以在同一个父路由下共存，合起来让所有访问路径都有明确的渲染结果，不留渲染空白。
  • Navigate 重定向：404 有时不是终点而是中转站——比如识别到某个失效的旧路径模式后，不展示"页面走丢了"，而是在 404 组件内部渲染 <Navigate to="/new-path" replace /> 悄悄迁移用户，这是通配路由和声明式重定向的常见组合。
  • Query 参数与 useSearchParams：通配路由只匹配路径部分，完全不检查 query 参数——这意味着任何带着奇怪 ?xxx=yyy 的已知路径依然会正常匹配到对应页面，只有路径本身不认识时才会落到 404，这也印证了 query 参数"不参与路由匹配、只是附加信息"的定位。
  • createBrowserRouter 与 loader：数据路由用 errorElement 统一处理 loader/action 抛出的异常，这和 path="*" 处理"路径未匹配"是两套不同的兜底机制——前者管"数据获取失败"，后者管"路由本身不存在"，实际项目里往往需要同时配置两者才能覆盖完整的异常场景。
