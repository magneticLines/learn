Axios 封装:拦截器与统一错误处理

裸用 fetch 或 axios 写请求,每次都要手写 baseURL、手动拼 Authorization 头、手动 catch 弹错误提示——十个接口写十遍一模一样的样板代码。拦截器(interceptor)把这些"每次都要做的事"收敛到一个地方,一次写好,全项目复用。

问题:裸用请求库为什么会腐化

想象一个真实项目要接入的接口:登录态要带 token、后端约定统一的错误码结构、401 要跳登录页、500 要弹 toast。如果每个组件自己发请求,这些逻辑会被复制粘贴到几十个地方:

反例:每个请求都重复处理 token 和错误
// UserProfile.jsx
const res = await fetch('https://api.example.com/user', {
  headers: { Authorization: `Bearer ${localStorage.getItem('token')}` },
});
if (res.status === 401) { location.href = '/login'; }
if (!res.ok) { toast.error('请求失败'); }
// OrderList.jsx 里几乎一模一样的代码又写一遍……
为什么这是个真问题? 不是"丑"这么简单——当后端某天把 token 的字段名从 Authorization 改成 X-Auth-Token,你得全项目搜索替换;当产品要求 401 时先弹确认框再跳转,你得改几十处。重复逻辑 = 修改成本随接口数量线性增长,这是拦截器要解决的核心问题。

request 拦截器:统一加认证头

拦截器的本质是给 axios 实例注册一个"钩子函数",每次请求发出前(request 拦截器)或每次响应回来后(response 拦截器)都会自动执行,不需要在业务代码里手动调用。

request.js:创建实例并注册 request 拦截器
import axios from 'axios';

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE,  // 环境变量,不同环境不同域名
  timeout: 10000,
});

// request 拦截器:每次请求发出前自动执行,统一挂 token
request.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;  // 必须返回 config,否则请求发不出去
}, (error) => Promise.reject(error));

response 拦截器:统一处理错误

response 拦截器接收两个回调:第一个处理成功响应(可以在这里直接"拆包",让业务代码拿到的就是 data 而不是整个 axios 响应对象),第二个处理失败(网络错误、非 2xx 状态码)。

request.js:response 拦截器统一拆包与错误处理
request.interceptors.response.use(
  (response) => response.data,  // 成功:直接返回 data,业务代码不用每次写 res.data
  (error) => {
    const status = error.response?.status;
    if (status === 401) {
      localStorage.removeItem('token');
      location.href = '/login';  // 登录态失效,统一跳登录页
    } else if (status >= 500) {
      toast.error('服务器开小差了,请稍后重试');
    } else {
      toast.error(error.response?.data?.message || '请求失败');
    }
    return Promise.reject(error);  // 继续 reject,让调用方(如 TanStack Query)也能感知失败
  }
);

export default request;
关键细节:错误处理完仍要 Promise.reject。 拦截器里弹了 toast 不代表流程结束——调用方(比如后面要讲的 TanStack Query 的 onError,或组件里的 try/catch)往往还需要知道"这次请求失败了",从而决定要不要保留 loading 态、要不要禁用重试按钮。拦截器负责"通用的副作用"(弹提示、跳转),业务代码负责"具体页面该怎么响应",两者不冲突。

对比:裸用 axios vs 封装后使用

裸用 axios
axios.get('https://api.x.com/user', {
  headers: { Authorization: `Bearer ${token}` }
}).then(res => res.data)
  .catch(err => { /* 手写弹窗 */ });
// 每个接口都要重复这套模板
封装后的 request 实例
import request from './request';

request.get('/user');
// token、baseURL、错误弹窗全部自动处理
// 业务代码只关心"要调哪个接口"

为什么这一步是给 TanStack Query 铺路

封装 axios 实例本身不解决缓存、重复请求、loading/error 状态管理这些问题——那是 TanStack Query 的职责。这里做的事情很单纯:提供一个“开箱即用、自带认证和错误处理”的请求函数,下一篇会把它作为 queryFn 传给 useQuery,两者各司其职地组合起来。

提前预告: 封装到这一步,request.get('/user') 已经可以直接当 queryFn 用。但要注意 TanStack Query 需要 queryFn 抛出错误才能进入它自己的 isError 状态——上面拦截器里保留 Promise.reject 正是为了不破坏这个机制。

小结

  • 裸用 fetch/axios 会导致 token 拼接、错误处理这类"每次都要做的事"被复制粘贴到项目各处,维护成本随接口数量线性增长。
  • axios.create() 创建一个带 baseURL、timeout 等公共配置的实例,避免硬编码域名。
  • request 拦截器在请求发出前统一注入 Authorization 头;response 拦截器统一拆包 data、统一处理 401 跳转和 5xx 提示。
  • 拦截器里处理完错误后仍要 Promise.reject(error),把"失败"这个信号继续传递给调用方,不能吞掉。
  • 封装 axios 只负责"怎么发请求",不负责缓存和状态管理,这些留给下一篇的 TanStack Query 组合使用。

常见陷阱

  • response 拦截器直接 return response.data,丢了状态信息。 拆包确实方便,但如果某个接口需要读 headers(比如分页信息放在 X-Total-Count)或者要判断 status 而不是靠 axios 自动 reject,拆包之后这些信息就没了。常见做法是给这类特殊接口单独传一个配置(如自定义 skipUnwrap),而不是所有接口都强制走同一套拆包逻辑。
  • 拦截器里忘了 return Promise.reject(error),错误被静默吞掉。 如果 response 拦截器的失败回调只顾着弹 toast,没有把 error 继续 reject 出去,这个 Promise 就会变成"resolved 但值是 undefined"——调用方的 try/catch 和 TanStack Query 的 isError 都感知不到失败,组件可能拿着 undefined 去渲染,直接白屏或报错,却完全找不到原因。
  • 多个模块共用一个 axios 实例,却各自修改它的默认配置。 比如某个模块因为要传文件,临时把公共实例的 headers['Content-Type'] 改成 multipart/form-data,这个修改是全局的、持久的——之后所有请求都被污染了。凡是"只在某次请求里生效"的配置,应该传进那次调用的 config 参数,而不是改实例的默认值。
  • 401 刷新 token 的逻辑写在拦截器里,没考虑并发请求。 如果页面同时发出 5 个请求,且 token 恰好过期,5 个请求会几乎同时收到 401,若刷新逻辑没有加锁,会触发 5 次刷新 token 的请求,其中大概率互相覆盖导致刷新失败。正确做法是维护一个"刷新中"的标记和等待队列:第一个 401 触发刷新,其余 401 先挂起等待,刷新成功后统一用新 token 重试。
  • 组件卸载后,拦截器里的副作用(如 toast.error)还在执行。 如果请求发出后组件被卸载(比如用户快速切换了页面),response 拦截器仍会在响应回来时执行,如果这段逻辑里有依赖组件状态的操作而不是纯粹的全局提示,就可能报"在已卸载组件上更新状态"的警告。拦截器本身适合放"全局无状态"的副作用,组件级别的取消应该用 AbortController 配合 useEffect 的清理函数处理,而不是依赖拦截器兜底。

面试高频考点

  1. 为什么要用 axios.create() 而不是直接用默认的 axios 对象发请求?—— 默认的 axios 对象是全局单例,它的 defaults(比如 baseURL、headers)一旦被某处代码修改,会影响项目里所有用到 axios 的地方。axios.create(config) 会返回一个独立的实例,拥有自己的一份 defaults 和拦截器链,不同业务域(比如对接不同后端服务)可以创建互不干扰的多个实例。
  2. request 拦截器和 response 拦截器的执行顺序是怎样的?—— 一次请求会先按注册顺序依次执行所有 request 拦截器,把 config 层层加工后真正发出网络请求;响应回来后再按注册顺序依次执行所有 response 拦截器。本质上 axios 内部把"发请求"这个动作也包装成了一个 Promise,拦截器就是在这个 Promise 链的头部和尾部插入 then/catch 节点,所以它遵循 Promise 链式调用的顺序规则,而不是什么特殊机制。
  3. 如何在拦截器里统一处理 401 刷新 token 并自动重试原请求?—— 核心思路是在 response 拦截器的失败回调里捕获 401,先判断是否已经在刷新(用一个模块级变量标记),如果没有则发起刷新 token 的请求,拿到新 token 后把原来失败的请求配置(error.config)里的 Authorization 头替换掉,用 request(error.config) 重新发出去并把这个 Promise 返回;如果已经在刷新中,则把当前请求推入一个等待队列,刷新完成后统一用新 token 重试队列里的所有请求。关键是要避免并发 401 导致重复刷新。
  4. axios 和 fetch 相比,封装层面解决了 fetch 的哪些痛点?—— fetch 本身不会对非 2xx 状态码 reject(需要手动判断 res.ok),没有内置的请求/响应拦截机制,没有自带的超时和取消(取消要靠 AbortController 手动接),也没有自动的 JSON 序列化/反序列化。axios 把这些都做成了内置能力,拦截器又提供了一个"集中处理横切关注点"的钩子位置,这正是本文封装能够成立的前提——如果用 fetch 硬写同样的效果,需要自己包装出等价的拦截层。

与其他知识点的联系

  • 与 useEffect 数据请求、竟态条件的关系: 封装出来的 request 实例只解决"怎么发一次干净的请求",并不会自动处理"组件卸载后响应才回来""userId 快速切换导致新旧响应互相覆盖"这类竟态问题——这些仍然需要配合 AbortController(下面第 146 篇)手动取消过期请求,或者干脆交给 TanStack Query 内置的取消机制去管。
  • 与下一篇 145(TanStack Query)的关系: 本篇解决的是"怎么发请求"这一层,还完全没有涉及缓存、loading/error 状态管理、多组件共享同一份数据去重这些问题——这些正是下一篇要引入 TanStack Query 来补上的那一层。封装好的 request 实例几乎可以原封不动地作为 queryFn 传给 useQuery,两层各司其职。
  • 与表单提交(143 篇)的关系: 143 篇 RHF 的 onSubmit 回调拿到的是校验通过、类型安全的表单数据,而这份数据下一步要做的事通常就是调用本篇封装的请求函数发出去——表单校验解决"数据合不合法",本篇解决"合法的数据怎么可靠地发出去",两者在 onSubmit(data) => request.post('/login', data) 这样的写法里自然地衔接起来。
  • 与拦截器统一处理 loading/错误提示的关系: request 拦截器和 response 拦截器承担的是"每个请求都要做的通用副作用"(挂 token、401 跳转、错误 toast),这和后面 TanStack Query 里 isLoading/isError 这类"针对某一次具体查询的状态"是两个不同粒度的关注点——拦截器管全局通用逻辑,组件/查询层管这一次调用该怎么响应,不应该把两者的职责混在一起写。
  • 与 Promise 链式调用的关系: 拦截器的执行顺序本质上遵循 Promise 链的 then/catch 规则,拦截器处理完错误后仍要 Promise.reject(error) 才能让这个"失败"信号继续沿着 Promise 链传递下去——这也是为什么下一篇 TanStack Query 能在拦截器已经处理过一轮错误之后,依然感知到 isError 并进入自己的错误状态。
