React · 列表渲染

key：给列表里每个元素一张"身份证"

列表渲染时那个总要写的 key，不是形式主义。它是 React 在更新列表时判断"谁是谁"的唯一依据。理解 key 的作用，就理解了 React 的 Diff 算法为什么需要它。

作用：帮 Diff 算法识别元素的"身份"

当列表数据变化（增、删、排序、修改），React 需要对比"旧的元素列表"和"新的元素列表"，决定哪些 DOM 复用、哪些新建、哪些删除——这就是 Diff。问题是：React 怎么知道新列表里的某个 <li> 和旧列表里的哪个是"同一个"？

如果没有额外信息，React 只能按位置（下标）逐个对比：新列表第 0 个 vs 旧列表第 0 个……这在"列表顺序变了"时会严重误判。key 就是解决方案——你给每个元素一个稳定的标识，React 就改用 key 来配对：

旧列表
key: a,b,c
→
按 key 配对
不是按位置
→
a→a b→b c→c
精准复用
本质类比：key 就像数据库表的主键，或者每个人的身份证号。没有它，React 只能靠"排队的第几位"认人——一旦有人插队或离开，后面所有人的身份都会被认错。有了 key，无论怎么增删排序，React 都能一眼认出"这个元素还是原来那个"，从而正确地复用它的 DOM 和内部 state。

规范一：稳定、唯一——用数据自身的 id

好的 key 有两个硬性要求：稳定（同一条数据的 key 在多次渲染间不变）和唯一（同一列表内不重复）。数据的业务 id 是天然的最佳选择：

✅ 用稳定的业务 id
{users.map(u =>
  <li key={u.id}>
    {u.name}
  </li>
)}
// u.id 跟着数据走，永远稳定
❌ 每次渲染都变的 key
{users.map(u =>
  <li key={Math.random()}>
    {u.name}
  </li>
)}
// 每次渲染 key 全变→全部重建
// 性能灾难，状态全丢
反例警示：key={Math.random()} 或 key={Date.now()}。这类"每次都变"的 key 是最坏的做法——每次渲染 React 都认为所有元素都是全新的，于是销毁全部旧 DOM、重建全部新 DOM，key 的优化意义完全丧失，还会导致输入框失焦、动画重置、组件 state 丢失等问题。key 的价值恰恰在于"稳定"。

规范二：key 只需在"兄弟之间"唯一

key 不需要全局唯一，只需在同一个父节点下的兄弟元素之间唯一即可。不同列表、不同层级可以有相同的 key，它们各自独立：

两个列表各自的 key=1 互不冲突
<div>
  <ul>
    {fruits.map(f => <li key={f.id}>{f.name}</li>)}
  </ul>
  <ul>
    {/* 这个列表的 key=1 和上面那个 key=1 不冲突，兄弟范围不同 */}
    {veggies.map(v => <li key={v.id}>{v.name}</li>)}
  </ul>
</div>

规范三：key 放在 map 直接返回的最外层元素上

key 要写在 map 回调直接返回的那个元素上，而不是它内部的子元素。用 Fragment 包裹时，需要用完整写法 <React.Fragment key=...>（简写 <> 不能带属性）：

需要包裹多个元素时用带 key 的 Fragment
{items.map(item => (
  // key 放在最外层的 Fragment 上，而不是里面的 dt/dd
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </React.Fragment>
))}
缺失 key 的警告。如果 map 出来的元素没写 key，控制台会报："Warning: Each child in a list should have a unique 'key' prop." 这不是致命错误，页面还能渲染，但意味着 Diff 退化成按下标对比——在列表会变动时可能引发状态错位 bug（下一篇专门讲）。看到这个警告务必补上 key，别忽略。另外：key 不是普通 prop，子组件读不到 props.key，它是 React 保留给 Diff 用的；如果子组件也需要这个值，得另传一个 prop（如 id={item.id}）。
小结
  • key 是元素的"身份证"，让 React 的 Diff 按 key 配对而非按下标，从而正确复用 DOM 与组件 state。
  • 规范：稳定 + 唯一，首选数据业务 id；严禁 Math.random()/Date.now() 这种每次都变的 key。
  • key 只需兄弟间唯一；放在 map 直接返回的最外层元素上；key 不会作为 prop 传给子组件。

常见陷阱

  • 用数组下标（index）当 key：{items.map((item, i) => <li key={i}>...</li>)} 是最常见的坑。列表顺序不变时看不出问题，但一旦发生插入、删除、排序，下标和数据的对应关系就错位了——比如在列表开头插入一条新数据，原来 index=0 的旧数据现在变成了 index=1，React 却认为"key=1 的元素内容变了"而不是"多了一个新元素"，于是错误地复用 DOM 和内部 state（典型症状：输入框的内容跑到别的行、动画错位、勾选状态跟着位置走而不是跟着数据走）。
  • 用非唯一的字段做 key，比如用 name 或某个可能重复的分类字段：如果两条数据的 name 恰好相同，会导致 key 冲突，React 只能按其中一个处理，另一个的更新可能被吞掉或渲染错误，且控制台通常只会警告"重复的 key"而不会告诉你具体是哪两条数据冲突，排查成本较高。
  • 误以为 key 变化 = 组件"更新"，实际上是"卸载重建"：如果同一位置前后渲染的元素 key 不同，React 不会尝试对比更新它们的 props，而是直接销毁旧组件（连带卸载其内部 state、触发 useEffect 清理函数）、创建全新组件。这有时会被反过来当成一个"强制重置组件状态"的技巧使用（故意换 key 让组件重新初始化），但如果是无意间导致 key 频繁变化，就会造成不必要的组件重建、动画闪烁、请求重发等问题。
  • 以为 key 加了就一定能提升性能，所有列表不管大小都要精心设计 key：key 解决的是"正确性"问题（保证 Diff 配对准确、state 不错位），不是万能的性能开关。列表本身很小或几乎不会变动顺序时，用 index 当 key 也不会出问题；真正需要重视 key 设计的，是列表会发生增删排序、且列表项内部带有 state（如输入框、勾选框、展开状态）的场景。

面试高频考点

  1. 为什么 React 列表渲染需要 key，不写会怎样？—— key 是 React 在 Diff 阶段判断"新旧列表中哪个元素对应哪个元素"的依据。不写 key 时，React 只能退化成按位置（下标）逐一比较，列表顺序发生变化时容易造成错误复用（DOM 内容对应错数据）。不写 key 本身不会报错崩溃，但控制台会给出警告，且存在潜在的状态错位风险。
  2. 为什么不推荐用数组下标作为 key？举一个具体会出问题的场景。—— 因为下标不是数据本身的属性，它会随着列表的插入、删除、排序而变化，不满足 key 需要"稳定"的要求。例如一个待办列表，每项都有一个输入框记录备注，用户在第 2 项的输入框里输入了文字；如果这时在列表最前面插入一条新数据，所有旧数据的下标全部后移一位，React 会认为"index=1 位置的内容变了"，于是把原本第 2 项输入框的 DOM（及其内部输入内容）复用给了实际上是新错位过来的那条数据，造成用户输入内容"跑"到了别的行。
  3. key 只需要满足"全局唯一"还是"局部唯一"？为什么？—— 只需要在同一父节点下的兄弟元素之间唯一。因为 React 的 Diff 算法是逐层进行的，只在同一层级、同一父节点下的子元素集合内做 key 匹配，不同父节点下的子树各自独立比较，因此不同列表、不同组件树分支中出现相同的 key 完全没有问题。
  4. key 会不会作为 prop 传给子组件？子组件内部能不能用 props.key 读取它？—— 不会。key 是 React 保留的特殊属性，专门供 Diff 算法内部使用，会在渲染前从 props 中被提取出来，子组件的 props 对象里不包含 key 字段。如果子组件业务逻辑也需要用到这个标识值，必须再显式传一个同值的普通 prop（例如同时写 key={item.id} 和 id={item.id}）。

与其他知识点的联系

  • 为什么不能用 index 作 key（下一节）：本节讲的"key 要稳定"这条规范，最典型的反例就是用数组下标——下标看似稳定，实际上会随增删排序而漂移，下一节会把这个坑展开讲清楚，两节是"原则"与"最常见违反案例"的关系。
  • 不可变更新：key 的配对依赖 Diff 算法对比新旧列表，而新旧列表本身是通过不可变更新（map/filter 返回新数组）生成的——如果直接修改原数组再渲染，React 连"新旧列表"的对比基础都没有，key 的配对机制就无从谈起，两者共同保证列表更新的正确性。
  • 组件 state 与卸载重建：key 变化会导致 React 销毁旧组件、创建新组件（连带清空其内部 state、触发 useEffect 清理），这是理解"为什么 key 错误会导致输入框内容跑到别的行"的关键——本质是 key 不稳定间接触发了不必要的组件卸载重建，而不是简单的"内容显示错了"。
  • Fragment：当 map 需要返回多个并列元素而不想多包一层 DOM 时要用 <React.Fragment key={...}>，因为 JSX 简写 <> 不支持传属性——这是 key 规范（必须放在 map 直接返回的最外层元素上）与 Fragment 语法限制交叉产生的一个实用细节。
  • useEffect 依赖与副作用清理：如果列表项组件内部有 useEffect（比如发起请求、订阅事件），key 变化触发的卸载重建会连带执行该 effect 的清理函数并在新组件挂载后重新执行 effect——这有时被有意利用（换 key 强制重置组件到初始状态），但无意间的 key 抖动会造成请求重复发起等副作用问题。
