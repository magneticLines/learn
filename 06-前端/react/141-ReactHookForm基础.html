React Hook Form 基础

用 useState 给每个输入框都做受控组件,表单小的时候很好用;但表单变成十几个字段的注册页、复杂的动态表单后,你会发现每敲一个字符,整个表单组件都重渲染一次。React Hook Form(RHF)换了一种订阅模型,专门解决这个问题。

回顾:受控表单为什么会变慢

还记得最早学 useState 时写的受控输入框吗?

纯手写受控表单:每个字段一个 useState
function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  // 字段一多,这里就要复制粘贴出十几组 useState + onChange

  return (
    <form>
      <input value={email} onChange={(e) => setEmail(e.target.value)} />
      <input value={password} onChange={(e) => setPassword(e.target.value)} />
    </form>
  );
}

这种写法叫受控组件:输入框的值完全由 React state 驱动,每次按键触发 setEmail,组件重渲染,value 属性再把新值传回输入框——这是一个"单向数据流"的闭环,好处是任何时刻 email 这个变量都能准确反映输入框当前的值,方便做实时校验、字符计数等。

代价也很直接:每敲一个字符就触发一次组件重渲染。表单只有两三个字段时无所谓,但字段一多(注册表单、订单表单动辄十几个字段),每次按键都要重新渲染整个表单组件树,加上如果表单里还嵌了别的复杂子组件,输入体验就会明显变卡。而且每加一个字段就要复制一遍 useState + onChange 的样板代码。

RHF 的思路:非受控 + 按需订阅

React Hook Form 换了一个方向:输入框不再由 React state 驱动值(非受控,值留在 DOM 节点本身),RHF 通过 ref 直接"挂"在输入框上,监听原生的 input 事件来收集值。整个过程尽量不触发 React 重渲染——只有真正需要重渲染的场景(比如某个字段出现校验错误、需要显示错误提示)才会精确地只重渲染用到那个错误信息的组件。

受控表单:每次按键 → setState → 整个表单重渲染
RHF:按键值存在 DOM 里,ref 订阅收集,不触发重渲染

基本用法:useForm + register + handleSubmit

useForm 三件套
import { useForm } from 'react-hook-form';

function LoginForm() {
  const { register, handleSubmit } = useForm(); // 一个 hook 管理整个表单的状态

  // 表单校验通过后才会被调用,拿到的 data 是所有字段值组成的对象
  const onSubmit = (data) => console.log(data); // { email: '...', password: '...' }

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      {/* register('email') 返回 { name, onChange, onBlur, ref },
          展开到 input 上就完成了"订阅"这个字段 */}
      <input {...register('email')} placeholder="邮箱" />
      <input {...register('password')} type="password" />
      <button type="submit">登录</button>
    </form>
  );
}
// 没有一个 useState,没有一个 onChange 手写逻辑,
// 打字过程中 LoginForm 组件本身几乎不会重渲染

register('email') 返回一个包含 name、onChange、onBlur、ref 的对象,用展开语法 {...register('email')} 铺到 input 上,RHF 就通过这个 ref 拿到了这个 DOM 节点,后续读值、校验都直接从 DOM 读,不需要 React state 参与。

常见误区:以为用了 RHF 就完全不能实时拿到输入值。 RHF 提供 watch('email') 来按需订阅某个字段的实时值(比如做"输入密码时实时显示强度条"),但 watch 本身会让用到它的组件重新订阅重渲染——所以只在真正需要实时值的地方用 watch,其余字段保持"非受控"才能拿到 RHF 的性能收益,不要不分场景地到处 watch。
校验规则写在哪? 这一篇只搭好了骨架,还没讲校验。register('email', { required: true }) 这种写法可以塞简单规则,但字段一多、规则一复杂(邮箱格式、密码强度、字段间联动校验)就会很凌乱。下一篇引入 Zod,把校验规则统一收进一份 schema。

小结

  • 纯手写受控表单(useState + onChange)在字段变多后,每次按键都触发整个表单重渲染,还带来大量重复样板代码。
  • React Hook Form 用非受控 + ref 订阅的方式管理表单:值留在 DOM 里,尽量不触发 React 重渲染。
  • useForm() 创建表单实例,register('字段名') 把一个输入框"接入"表单,handleSubmit(onSubmit) 包裹提交逻辑,校验通过才会调用 onSubmit。
  • watch() 可以按需订阅某字段的实时值,但会带来重渲染,不要滥用,否则失去 RHF 的性能优势。
  • 目前校验规则还比较零散,下一篇用 Zod 把校验规则统一整理成一份可复用的 schema。

常见陷阱

  • 混用受控和非受控——给 register 接管的输入框再手动写 value={xxx} + useState,两套数据源互相打架:DOM 里的真实值和 state 里的值会不同步,输入框会出现"打字卡顿""值跳回旧值"之类的怪现象。要用 RHF 就把这个字段的值交给 register 全权管理,需要实时值就用 watch,而不是自己再加一层 state。
  • 动态字段忘记用带索引的 name——渐进式增删的表单项(比如"添加联系人"列表)如果所有输入框都注册成同一个 name(比如都叫 contact),RHF 内部按 name 做键值映射,后写入的值会直接覆盖前面的,导致表单数据错乱甚至丢字段。正确做法是把索引拼进 name(如 contacts.0.email、contacts.1.email),配合 useFieldArray 管理增删。
  • 滥用 watch 导致重渲染泛滥——本文强调过 RHF 的性能优势来自"按键不重渲染",但如果在组件顶层无差别地 watch() 整个表单(不传字段名,监听所有字段),那么每次任意字段变化都会触发这个组件重渲染,等于把 RHF 又变回了受控表单的重渲染频率,性能收益全部作废。应该只 watch 真正需要实时值的具体字段,并且尽量把用到 watch 结果的 UI 拆到子组件里,缩小重渲染范围。
  • 异步获取的 defaultValues 没有及时 reset——useForm({ defaultValues }) 只在表单初始化那一刻读一次默认值。如果默认值是异步请求回来的(比如编辑页要先拉详情接口),组件首次渲染时数据还没到,表单会先用空值渲染;等数据到达后如果只是更新了外部的 state,并不会让已经"非受控"的输入框自动刷新——因为值已经交给 DOM 管理了。必须在数据到达后手动调用 reset(fetchedData),才能把新值真正写回每个输入框。

面试高频考点

  1. React Hook Form 为什么比纯 useState 受控表单性能好?—— 受控表单把输入框的值和 React state 绑在一起,每次按键都要 setState 触发组件重渲染;RHF 让输入框保持"非受控",值留在 DOM 节点里,通过 register 拿到的 ref 直接读写 DOM,并靠原生事件收集输入,绕开了 React 的渲染流程。只有真正需要响应式更新的场景(校验报错、watch 订阅)才会精确触发对应组件的重渲染,而不是整个表单重渲染。
  2. register 背后的 ref 机制是怎么工作的?—— register('email') 返回一个包含 name、onChange、onBlur、ref 的对象,展开到 input 上后,RHF 通过这个 ref 拿到真实 DOM 节点的引用,存进内部一份以字段名为键的注册表里。之后无论是读取当前值、执行校验,还是提交表单,RHF 都是直接从这个 DOM 引用上取值(element.value),而不需要经过 React state 这一层,这也是它能做到"非受控"的关键。
  3. handleSubmit 做了什么,为什么不能直接把 onSubmit 传给 form 的 onSubmit?—— handleSubmit(onSubmit) 包了一层:表单提交时先阻止默认的整页刷新行为,再遍历所有注册字段跑一遍校验规则,校验全部通过后才把收集到的字段值组装成一个对象传给你写的 onSubmit;如果校验不通过,则直接把错误信息更新到 formState.errors 里,不会调用 onSubmit。直接传原始 onSubmit 会跳过这一整套校验和数据收集逻辑。
  4. watch 和 getValues 有什么区别?—— getValues 是命令式地"读一次当前值",调用时才去 DOM 取数据,不会订阅、不会触发任何重渲染,适合在事件回调里临时取值;watch 是声明式订阅,会让调用它的组件对指定字段的变化保持响应,字段一变就重渲染,适合需要把某个字段的值实时展示在 UI 上的场景。两者的取舍本质上就是"要不要为了拿到实时值,牺牲 RHF 非受控省渲染的优势"。
