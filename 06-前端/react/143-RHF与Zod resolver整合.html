RHF + Zod resolver 整合

第 141 篇搭好了 React Hook Form 的骨架(负责状态管理、订阅、提交时机),第 142 篇用 Zod 写好了一份校验 schema(负责"数据合不合法")。这一篇把两者接起来——@hookform/resolvers 就是那根"桥",让 RHF 在校验时刻自动去问 Zod schema。

resolver:两个库之间的适配层

RHF 本身不知道 Zod 是什么,Zod 也不知道 RHF 是什么——它们是完全独立的两个库。@hookform/resolvers 提供了一个 zodResolver 函数,把 Zod schema "翻译"成 RHF 能理解的校验接口:RHF 在合适的时机(提交时、或配置了实时校验模式时)调用这个 resolver,resolver 内部跑一遍 schema.safeParse(),再把 Zod 的错误格式转换成 RHF 期望的 formState.errors 格式。

用户提交表单
→
RHF 调用 zodResolver
→
Zod schema.safeParse()
→
合法 → onSubmit(data)
→
不合法 → formState.errors
整合之后,职责边界变得非常清晰。 RHF 只管"状态怎么存、什么时候触发校验、怎么把值收集起来";Zod 只管"这份数据合不合法、错误信息是什么"。谁都不用管对方的实现细节,各自替换都很容易——比如以后想把 Zod 换成 Yup,只需要换一个 resolver,RHF 那边的代码完全不用动。

完整示例:带错误提示的登录表单

schema.ts:校验规则(复用第 142 篇的写法)
import { z } from 'zod';

export const loginSchema = z.object({
  email: z.string().email('请输入正确的邮箱格式'),
  password: z.string().min(8, '密码至少 8 位'),
});

// 类型也顺手从 schema 推导出来,表单组件的 props/state 类型直接复用它
export type LoginFormValues = z.infer<typeof loginSchema>;
LoginForm.tsx:用 zodResolver 把 schema 接进 useForm
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { loginSchema, type LoginFormValues } from './schema';

function LoginForm() {
  const {
    register,
    handleSubmit,
    formState: { errors }, // 校验失败时,错误信息会自动出现在这里
  } = useForm<LoginFormValues>({
    resolver: zodResolver(loginSchema), // ← 校验逻辑完全交给 Zod schema
  });

  const onSubmit = (data: LoginFormValues) => {
    // 走到这里,说明 data 已经通过了 loginSchema 的全部校验
    console.log('登录:', data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <div>
        <input {...register('email')} placeholder="邮箱" />
        {/* errors.email 存在时才渲染,不存在时什么都不显示 */}
        {errors.email && <p className="error">{errors.email.message}</p>}
      </div>

      <div>
        <input {...register('password')} type="password" />
        {errors.password && <p className="error">{errors.password.message}</p>}
      </div>

      <button type="submit">登录</button>
    </form>
  );
}

注意 errors.email.message 里的错误文案,正是我们在第 142 篇 schema 里写的 '请输入正确的邮箱格式'——校验规则和错误提示文案都只写在 schema 这一处,组件里不需要重复写任何校验相关的逻辑,只负责"有错误就渲染出来"。

常见坑:忘记给 useForm 传泛型 <LoginFormValues>。 不传泛型时,register('email') 里的字段名不会有类型提示和拼写检查,写错字段名(比如手滑打成 'emial')编译期不会报错,运行时才会发现表单怎么都提交不出正确的值。把 z.infer 推导出的类型传给 useForm<T>(),能让 register、errors 全程有类型检查,这也是"schema 即唯一真源"这个设计在开发体验上的直接体现。
三篇串起来看: 第 141 篇解决"表单状态怎么高效管理"(RHF 的非受控订阅模型),第 142 篇解决"校验规则怎么统一定义"(Zod schema + 类型推导),这一篇用 zodResolver 把两者粘合成一套完整方案——这也是目前 React 表单生态里最主流的组合,比自己手写 useState + 散落校验逻辑要健壮和好维护得多。

小结

  • @hookform/resolvers 里的 zodResolver 是 RHF 和 Zod 之间的适配层,把 Zod 的校验结果转换成 RHF 期望的 formState.errors 格式。
  • 整合后职责清晰:RHF 管状态和触发时机,Zod 管校验规则,互相替换都不影响对方。
  • 用法:useForm({ resolver: zodResolver(schema) }),配合 z.infer 推导出的类型作为 useForm 的泛型,获得完整的类型提示。
  • 错误提示统一从 formState.errors.字段名.message 读取,文案在 schema 里定义一次,组件里只负责"有错误就渲染"。
  • 至此表单处理这一组形成闭环:受控表单的性能问题(RHF)→ 校验规则的组织问题(Zod)→ 两者整合(resolver)。下一组进入网络请求实战,先看 Axios 的拦截器封装和统一错误处理。
