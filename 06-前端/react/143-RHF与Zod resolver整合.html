RHF + Zod resolver 整合

第 141 篇搭好了 React Hook Form 的骨架(负责状态管理、订阅、提交时机),第 142 篇用 Zod 写好了一份校验 schema(负责"数据合不合法")。这一篇把两者接起来——@hookform/resolvers 就是那根"桥",让 RHF 在校验时刻自动去问 Zod schema。

resolver:两个库之间的适配层

RHF 本身不知道 Zod 是什么,Zod 也不知道 RHF 是什么——它们是完全独立的两个库。@hookform/resolvers 提供了一个 zodResolver 函数,把 Zod schema "翻译"成 RHF 能理解的校验接口:RHF 在合适的时机(提交时、或配置了实时校验模式时)调用这个 resolver,resolver 内部跑一遍 schema.safeParse(),再把 Zod 的错误格式转换成 RHF 期望的 formState.errors 格式。

用户提交表单
→
RHF 调用 zodResolver
→
Zod schema.safeParse()
→
合法 → onSubmit(data)
→
不合法 → formState.errors
整合之后,职责边界变得非常清晰。 RHF 只管"状态怎么存、什么时候触发校验、怎么把值收集起来";Zod 只管"这份数据合不合法、错误信息是什么"。谁都不用管对方的实现细节,各自替换都很容易——比如以后想把 Zod 换成 Yup,只需要换一个 resolver,RHF 那边的代码完全不用动。

完整示例:带错误提示的登录表单

schema.ts:校验规则(复用第 142 篇的写法)
import { z } from 'zod';

export const loginSchema = z.object({
  email: z.string().email('请输入正确的邮箱格式'),
  password: z.string().min(8, '密码至少 8 位'),
});

// 类型也顺手从 schema 推导出来,表单组件的 props/state 类型直接复用它
export type LoginFormValues = z.infer<typeof loginSchema>;
LoginForm.tsx:用 zodResolver 把 schema 接进 useForm
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { loginSchema, type LoginFormValues } from './schema';

function LoginForm() {
  const {
    register,
    handleSubmit,
    formState: { errors }, // 校验失败时,错误信息会自动出现在这里
  } = useForm<LoginFormValues>({
    resolver: zodResolver(loginSchema), // ← 校验逻辑完全交给 Zod schema
  });

  const onSubmit = (data: LoginFormValues) => {
    // 走到这里,说明 data 已经通过了 loginSchema 的全部校验
    console.log('登录:', data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <div>
        <input {...register('email')} placeholder="邮箱" />
        {/* errors.email 存在时才渲染,不存在时什么都不显示 */}
        {errors.email && <p className="error">{errors.email.message}</p>}
      </div>

      <div>
        <input {...register('password')} type="password" />
        {errors.password && <p className="error">{errors.password.message}</p>}
      </div>

      <button type="submit">登录</button>
    </form>
  );
}

注意 errors.email.message 里的错误文案,正是我们在第 142 篇 schema 里写的 '请输入正确的邮箱格式'——校验规则和错误提示文案都只写在 schema 这一处,组件里不需要重复写任何校验相关的逻辑,只负责"有错误就渲染出来"。

常见坑:忘记给 useForm 传泛型 <LoginFormValues>。 不传泛型时,register('email') 里的字段名不会有类型提示和拼写检查,写错字段名(比如手滑打成 'emial')编译期不会报错,运行时才会发现表单怎么都提交不出正确的值。把 z.infer 推导出的类型传给 useForm<T>(),能让 register、errors 全程有类型检查,这也是"schema 即唯一真源"这个设计在开发体验上的直接体现。
三篇串起来看: 第 141 篇解决"表单状态怎么高效管理"(RHF 的非受控订阅模型),第 142 篇解决"校验规则怎么统一定义"(Zod schema + 类型推导),这一篇用 zodResolver 把两者粘合成一套完整方案——这也是目前 React 表单生态里最主流的组合,比自己手写 useState + 散落校验逻辑要健壮和好维护得多。

小结

  • @hookform/resolvers 里的 zodResolver 是 RHF 和 Zod 之间的适配层,把 Zod 的校验结果转换成 RHF 期望的 formState.errors 格式。
  • 整合后职责清晰:RHF 管状态和触发时机,Zod 管校验规则,互相替换都不影响对方。
  • 用法:useForm({ resolver: zodResolver(schema) }),配合 z.infer 推导出的类型作为 useForm 的泛型,获得完整的类型提示。
  • 错误提示统一从 formState.errors.字段名.message 读取,文案在 schema 里定义一次,组件里只负责"有错误就渲染"。
  • 至此表单处理这一组形成闭环:受控表单的性能问题(RHF)→ 校验规则的组织问题(Zod)→ 两者整合(resolver)。下一组进入网络请求实战,先看 Axios 的拦截器封装和统一错误处理。

常见陷阱

  • useForm 泛型和 schema 类型脱节:上面警示框提到的坑不只是"忘记传泛型"这一种——更隐蔽的是传了一个和 zodResolver(schema) 不一致的手写类型(比如字段名对不上、某个字段类型写成了 string 而 schema 里其实是 number)。TS 编译器只按你传的泛型检查,不会反过来校验 schema 和泛型是否匹配,结果就是类型检查"看起来通过"但运行时 resolver 报的错跟 IDE 提示的字段完全对不上。稳妥做法是只用 z.infer<typeof schema> 这一处推导出来的类型,不要手写第二份。
  • defaultValues 和 schema 类型不一致:useForm 的 defaultValues 也要满足 LoginFormValues 的形状,如果 schema 里某个字段是必填字符串,但 defaultValues 给了 undefined 或漏写了这个字段,TS 有时能查出来,但如果 defaultValues 类型标注得比较宽松(比如 Partial<LoginFormValues>),问题会一直拖到运行时才在 register 或校验阶段暴露出来。
  • 把 Zod 的错误结构和 RHF 的错误结构搞混:直接用 schema.safeParse() 得到的 error.issues 是 Zod 自己的格式,和 formState.errors.字段名.message 这种 RHF 格式并不是一回事。zodResolver 已经把两者的转换做掉了,组件里只需要读 formState.errors——如果习惯性地在业务代码里再手动跑一遍 safeParse 去读错误,就是重复做了 resolver 已经做好的事,还容易两套错误信息不一致。
  • 只看到"提交时校验",忽略了触发时机是可配置的:上面示例里 zodResolver 默认是在 handleSubmit 触发时才跑校验,如果没意识到这一点,容易误以为"输入的时候就会实时报错",结果发现输入框失焦、打字过程中 errors 一直是空的,以为 resolver 没生效——其实是校验时机(mode)没配对,不是 resolver 本身的问题。
  • schema 改了但没同步更新 LoginFormValues 的使用位置:因为类型是从 schema z.infer 出来的,改 schema(比如给 password 加一个 regex 校验)不会改变类型本身,很容易只改了校验规则、忘了看一眼是否也该同步调整错误文案或者上层组件对该字段的处理逻辑,造成"校验通过的规则"和"UI 提示的规则"不一致。

面试高频考点

  1. resolver 模式的本质是什么,为什么 RHF 要设计成"可插拔校验器"而不是内置一套自己的校验语法?—— resolver 本质是一个函数签名的约定:接收当前表单值,返回 { values, errors }(或等价结构),RHF 并不关心这个函数内部用什么校验库实现。这样 RHF 把"校验逻辑该长什么样"完全外包出去,只负责在合适的时机(提交、blur、change 等)去调用这个函数并消费它的返回值。好处是 RHF 不用为每个校验库(Zod、Yup、Joi……)写专门的适配代码,也不用背负一套自己维护的校验 DSL——这是典型的依赖倒置:RHF 依赖一个抽象接口(resolver 签名),具体实现(zodResolver)反过来适配这个接口。
  2. 为什么 RHF + Zod 这个组合能同时拿到类型安全和运行时校验,两者是怎么被同一份 schema 串起来的?—— 类型安全和运行时校验原本是两件不同阶段的事:TS 类型只在编译期存在,编译完就被抹掉,不能拦截"用户在表单里瞎填的数据";运行时校验能拦截真实数据,但如果单独手写,类型和校验规则容易写成两份、逐渐漂移。Zod 的关键是z.infer——它让 TS 类型直接从 schema 的运行时定义"反向推导"出来,而不是人工再写一遍 interface。这样 schema 变成唯一真源:改一处,编译期类型和运行时校验同时更新,不可能出现"类型说这是必填,但校验规则忘了加必填"的割裂。
  3. 如果不用 zodResolver,自己在 onSubmit 里手动调用 schema.safeParse() 校验,会有什么问题?—— 能工作,但会丢掉 RHF 生态里其他依赖 formState.errors 的能力:比如 errors.email 驱动的自动聚焦到第一个错误字段、逐字段错误提示、以及 mode: 'onBlur'/'onChange' 这类非提交时机的校验触发,都是建立在 RHF 内部统一走 resolver 这条路径上的。手动 safeParse 只能覆盖"提交这一刻"的校验,拿到的是 Zod 原始的 issues 数组,还需要自己再写一层转换才能塞进 formState,等于把 resolver 已经做好的适配工作重新做一遍。
  4. resolver 这一层的存在,如何体现"关注点分离"?去掉它直接耦合会带来什么后果?—— 没有 resolver 时,校验逻辑往往就地写在组件里,和状态管理代码混在一起,想换一个校验库(比如从 Zod 换成 Yup)就要把散落在各个组件里的校验代码逐个改掉。有了 resolver 这层适配,RHF 只认 resolver 函数签名,校验库的选择被隔离在 useForm({ resolver: xxxResolver(schema) }) 这一行配置里,替换成本从"改所有用到校验的组件"降到"改一行配置",这也是为什么本文开头强调"RHF 管状态,Zod 管规则,谁都不用管对方实现细节"。

与其他知识点的联系

  • 与上一篇 142(Zod schema)的关系: 142 篇写出的 loginSchema 本身不认识 RHF,resolver 正是把两者粘合起来的那层适配器——没有 resolver,schema 只能靠手动 safeParse() 调用,接进来之后 schema 才真正成为"驱动表单校验"的那一层,而不只是一份孤立的规则文档。
  • 与 RHF 基础用法(register、formState.errors)的关系: resolver 接入后,校验失败时 Zod 的 issues 会被自动转换、映射进 formState.errors.字段名.message,组件里只需要照常读 errors.email、errors.password 就能拿到对应的错误文案——底层换了校验引擎,但 register/errors 这套上层 API 完全没变,这正是 resolver 抽象层带来的好处。
  • 与受控/非受控组件的关系: RHF 走的是非受控订阅模型,register('email') 把 ref 和事件绑定挂在输入框上而不经过 state;resolver 接入的校验逻辑并不关心这一点,它只在 RHF 决定"要校验了"的那个时刻(提交、blur、change)被调用一次,拿到当前收集到的值去跑 safeParse,和输入框本身是否受控完全无关。
  • 与下一篇 144(Axios 封装)之间的关系: 本篇 onSubmit 回调里的 data 已经通过了 schema 的全部校验,是"确定合法"的数据——这正是发起网络请求的最佳时机。下一篇封装的 axios 请求函数,正是要被塞进这个校验通过之后的 onSubmit 里去调用,表单校验和网络请求由此串成一条完整的链路。
  • 与"可插拔校验器"设计模式的关系: resolver 本质是一个函数签名约定,RHF 不关心具体实现,今天用 Zod,明天想换 Yup 也只需要换一个 resolver 函数——这和后面封装 axios 时"业务代码只关心调哪个接口,不关心底层用什么发请求"是同一种关注点分离思路的重复出现。
