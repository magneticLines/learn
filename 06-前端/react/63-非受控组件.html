非受控组件:让 DOM 自己管值,用时再取

受控组件让 state 掌控一切,但有时这是"过度控制"。非受控组件把值交还给 DOM 自己保管,React 不实时跟踪,只在需要的那一刻(比如提交时)用 ref 去把值"读"出来。

定义:defaultValue 设初值 + ref 读当前值

非受控组件不绑定 value 和 onChange。它用 defaultValue 设置初始值,之后用户的输入完全由浏览器 DOM 自己记录。要拿值时,通过 ref 直接访问真实 DOM 节点的 .value。

非受控 input:提交时才读值
function NameForm() {
  // ref 是一个"抓手",指向真实的 DOM 节点
  const inputRef = useRef(null);

  function handleSubmit(e) {
    e.preventDefault();
    // 用时才去 DOM 里取当前值——不实时跟踪
    alert(inputRef.current.value);
  }

  return (
    <form onSubmit={handleSubmit}>
      // defaultValue 只设初值,后续 DOM 自己管
      <input ref={inputRef} defaultValue="Shea" />
      <button>提交</button>
    </form>
  );
}

关键区别:输入过程中不会触发任何重渲染,React 对输入框里此刻是什么值一无所知,直到你主动用 ref 去读。这更接近传统 HTML 表单的工作方式。

不要混用 value + defaultValue
同一个元素只能二选一。同时写 value 和 defaultValue,React 会警告并忽略 defaultValue。记住:value = 受控,defaultValue = 非受控,checkbox 对应是 checked vs defaultChecked。

file input:只能是非受控

有一个例外由不得你选:<input type="file"> 必须是非受控的。

文件选择框:只能用 ref 读
function Uploader() {
  const fileRef = useRef(null);

  function upload() {
    const file = fileRef.current.files[0];
    console.log(file.name);
  }

  // 不能写 value={...}:出于安全,JS 无法给 file 输入框设值
  return <input type="file" ref={fileRef} onChange={upload} />;
}
为什么 file input 不能受控?
浏览器出于安全考虑禁止用 JavaScript 设置文件输入框的值——否则网页就能伪造"用户已选择 C:/密码.txt"来窃取文件。既然值无法用代码写入,自然就无法用受控的 value 绑定,只能非受控 + ref 读取。

什么时候用非受控?

  • 简单表单,只在提交时取值:不需要实时校验/联动,用受控会写一堆 onChange 反而累赘。
  • file input:没有选择,必须非受控。
  • 集成非 React 库:某些第三方 DOM 插件(老的 jQuery 组件)自己管理 DOM,受控会打架。
  • 性能极端敏感的超大表单:避免每次按键都重渲染(实践中很少真的需要)。
本质对比:声明式 vs 命令式
受控是 React 推崇的声明式——"值等于 state,你只管声明"。非受控更偏命令式——"用的时候我主动去 DOM 里取"。React 生态整体倾向受控(它可预测、可测试),非受控是务实的特例。这也是为什么 React 官方文档把受控作为默认推荐,把非受控列为"你确实需要时的逃生舱"。

小结

  • 非受控组件:用 defaultValue(或 defaultChecked)设初值,DOM 自己保管,用 ref 在需要时读取。
  • 输入过程不触发重渲染,React 不实时跟踪值,更接近原生 HTML 表单。
  • value=受控,defaultValue=非受控,二者不可混用。
  • <input type="file"> 出于安全只能非受控。
  • 适合:简单表单、只在提交取值、集成第三方 DOM 库。受控仍是默认推荐。
