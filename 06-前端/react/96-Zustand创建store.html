Zustand 创建 store:极简的全局状态

Context 方案的痛点是没有选择性订阅、Provider 层层嵌套。Zustand 用一种极简得让人意外的方式解决了这些问题:没有 Provider、基于 Hook、state 和 action 写在同一个地方。这一篇从 create 开始,建起第一个 store。

Zustand 是什么

Zustand(德语"状态")是一个约 1KB 的极简状态管理库。它的设计哲学可以用三句话概括:

  • 无 Provider:store 是一个独立于 React 组件树的对象,不需要在根组件包一层 <Provider>。
  • 基于 Hook:用 const x = useStore(s => s.x) 读状态,和用 useState 一样自然。
  • state 与 action 同处一个 store:数据和"改数据的方法"写在一起,不像 Redux 要拆 action/reducer/dispatch。
为什么"无 Provider"是个大优点? Context 方案里,状态活在组件树里,只有 Provider 子树下的组件能访问,还会因 value 变化引发大面积重渲。Zustand 的 store 是一个独立于 React 的普通对象——它自己维护状态和订阅者列表,组件通过 Hook "订阅"它。这带来两个好处:① 任何地方(甚至组件外)都能访问 store;② 重渲染由 Zustand 精确控制,而不是 React 的 Context 传播机制。

create:一行建 store

store/useCounter.js — 最小 store
import { create } from 'zustand';

// create 接收一个函数,该函数返回 store 的初始形状
// 参数 set 是"更新状态的工具"(下一篇细讲)
export const useCounter = create((set) => ({
  // —— state:数据 ——
  count: 0,

  // —— action:改数据的方法,和 state 写在同一个对象里 ——
  increment: () => set((s) => ({ count: s.count + 1 })),
  decrement: () => set((s) => ({ count: s.count - 1 })),
  reset: () => set({ count: 0 }),
}));

create 返回的 useCounter 本身就是一个 Hook。注意命名习惯以 use 开头——它确实是自定义 Hook。组件里直接调用它就能拿到状态和方法:

组件里用:没有 Provider,直接 import 就能读
import { useCounter } from './store/useCounter';

function Counter() {
  // 直接调用 Hook,拿到整个 store(读取的细粒度写法下一篇讲)
  const count = useCounter((s) => s.count);
  const increment = useCounter((s) => s.increment);

  return <button onClick={increment}>{count}</button>;
}

// 注意:App 根组件不需要 <CounterProvider>,任何组件直接 import 即用

为什么比 Redux 轻

同一个计数器,对比一下心智负担:

· 写 reducer(switch/case)
· configureStore
· 根组件包 <Provider store>
· 组件里 useSelector + useDispatch
· dispatch(action) 才能改
概念多、文件多、样板多

· create 里写 state + action
· import 这个 Hook
· 直接调用 action 方法即可

陷阱:store 里塞服务端数据。 Zustand 轻到让人忍不住"什么都往里放"。回顾第 94 篇——API 数据交给 TanStack Query,Zustand 只放纯客户端状态(主题、UI 偏好、未提交的草稿等)。别把它当成"手动维护缓存"的工具。
命名与组织建议: 一个 store 文件对应一块业务领域,Hook 名以 use 开头(如 useCartStore、useAuthStore)。小项目一个 store 够用;大项目可以拆多个独立 store,或用切片模式(第 98 篇)组织一个大 store。

小结

  • Zustand 是约 1KB 的极简状态库:无 Provider、基于 Hook、state 与 action 同处一个 store。
  • store 是独立于 React 组件树的普通对象,自己维护状态与订阅者,因此能精确控制重渲、并可在任意位置访问。
  • create(set => ({...})) 一行建 store,返回的就是一个可直接调用的 Hook。
  • 相比 Redux 省掉了 action types、reducer、Provider、dispatch 等一大堆概念,几乎零样板。
  • 它是客户端状态库,别把 API 数据塞进来。
