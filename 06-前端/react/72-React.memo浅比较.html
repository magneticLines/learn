React.memo:用浅比较跳过无意义的重渲染

默认情况下,父组件重渲染,所有子组件无条件跟着重渲染——哪怕子组件的 props 一个字都没变。React.memo 给子组件加一道"门卫":对比新旧 props,没变就跳过这次渲染。但它的比较是浅比较,用错了不但没优化,还白搭。

默认行为:父重渲染,子必跟着渲染

Child 明明没变,却每次都跟着重渲染
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <>
      <button onClick={() => setCount(count + 1)}>+1</button>
      {/* title 是常量,从不改变 —— 但 count 变 → Parent 重渲染
          → Child 每次都重渲染,纯属浪费 */}
      <Child title="我不会变" />
    </>
  );
}

这是 React 的默认策略:它不去猜"子组件是否真的需要更新",而是无脑重跑子组件的渲染函数,再靠 diff 决定要不要动 DOM。大多数情况下这很快,不用管;但当子组件渲染开销大(长列表、复杂图表)时,这种浪费就值得优化了。

React.memo:加一道 props 门卫

用 memo 包裹,props 没变就跳过
const Child = React.memo(function Child({ title }) {
  console.log('Child 渲染');
  return <h3>{title}</h3>;
});

// 现在 count 变化 → Parent 重渲染,但 memo 发现 title 没变
// → 跳过 Child 的渲染,"Child 渲染"不再打印
父组件重渲染
→
memo 浅比较新旧 props
→
相等 → 跳过
不等 → 渲染

核心:什么是"浅比较"

memo 对 props 对象做的是浅比较(shallow compare):逐个比较 props 的每个字段,每个字段用 Object.is(约等于 ===)。它不递归深入——对于对象/数组/函数类型的字段,比的是引用,不是内容。

✅ 浅比较能判等的
title="hi"  → "hi"
count={5}   → 5
ok={true}   → true

// 原始值:值相同即相等
// memo 正确跳过
❌ 浅比较会判不等的
style={{ color:'red' }}
onClick={() => {}}
items={[1,2,3]}

// 对象/数组/函数
// 每次渲染都是新引用
// === 永远为 false
最大陷阱:memo 遇到对象/函数 props 会"静默失效"。下面这段代码,memo 完全没用——因为每次 Parent 渲染,{ color: 'red' } 和 () => {} 都是全新创建的对象/函数,引用每次都不同,浅比较永远判"变了",Child 照样每次重渲染。
memo 被内联对象/函数悄悄架空
const Child = React.memo(RawChild);

function Parent() {
  const [count, setCount] = useState(0);
  return (
    // 这两个 prop 每次渲染都是新引用 → memo 失效!
    <Child
      style={{ color: 'red' }}       // 新对象
      onClick={() => doSth()}        // 新函数
    />
  );
}
// 解法:用 useMemo 稳定对象、useCallback 稳定函数 —— 见下一篇

自定义比较函数(第二个参数)

需要更精细的控制时,可以传第二个参数,自己决定"是否相等"。注意语义和 shouldComponentUpdate 相反:返回 true 表示相等(跳过渲染)。

只关心 id 是否变化,忽略其他字段
React.memo(Child, (prev, next) => {
  // 返回 true = props 视为相等 = 跳过渲染(注意与 sCU 相反)
  return prev.id === next.id;
});
自定义比较函数别滥用:如果你在里面做深比较,深比较本身的开销可能超过它省下的渲染开销。多数时候,与其写自定义比较,不如从源头用 useMemo/useCallback 稳定 props 引用(下一篇)。

何时该用 memo(以及何时不该)

场景该用?
渲染开销大的组件(复杂图表、大列表项)值得
频繁被父组件重渲染、但自身 props 很少变值得
props 全是原始值,且父组件频繁更新值得
组件很轻量(就渲染几个字)不值得,比较成本比渲染还高
props 里必然带新对象/新函数,又没配 useMemo/useCallback无效,别装样子
不要默认给所有组件套 memo。memo 本身有成本:每次都要做浅比较、还要多存一份上次的 props。对轻量组件,这点成本可能比它省下的渲染还贵。React 官方的态度是——先别优化,用 Profiler 找到真正的性能瓶颈,再针对性地上 memo。过早优化会让代码更啰嗦、更易出 bug。(顺带一提,React 19 的编译器能自动做这类记忆化,未来可能不用手动 memo 了。)

小结

  • 默认:父渲染,子必渲染。React.memo 加一道 props 门卫,props 没变就跳过。
  • 比较是浅比较:原始值比值,对象/数组/函数比引用(不看内容)。
  • 最大陷阱:内联对象/函数 props 每次都是新引用,让 memo 静默失效。
  • 解法是用 useMemo/useCallback 稳定引用——正是下一篇的主题。
  • 别默认全套 memo,先测量再优化。
