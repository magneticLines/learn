Server Actions

表单提交的经典流程是:前端写一个 fetch(POST),后端单独写一个 API 路由接收它,两边手动对齐参数格式。Server Actions 把这层"手动搭桥"去掉——组件里直接调用一个标记了 'use server' 的函数,Next.js 自动把这次调用变成一次网络请求,函数本身却是真正跑在服务端的。

解决的问题:不用再手写一层 API

没有 Server Actions 之前,一个"新增评论"功能通常要写两块代码:一块是后端 POST /api/comments 路由,负责校验参数、写数据库;另一块是前端表单提交处理函数,负责 fetch 调这个接口、处理 loading/错误态。这两块代码本质上在描述同一件事("提交这个表单"),却要分别维护、手动保证参数格式一致。

Server Actions 让你把这两块合并成一个函数:标记 'use server' 后,这个函数的代码只在服务端执行(可以直接访问数据库、文件系统等服务端资源),但组件里可以像调用普通函数一样调用它——Next.js 在背后自动处理"这次调用要发起一次网络请求"的全部细节。

基本用法

actions.ts —— 标记为 Server Action 的函数
'use server'; // 这一行声明:这个文件里的函数只在服务端运行

export async function createComment(formData: FormData) {
  const content = formData.get('content');
  // 直接操作数据库,不需要再单独写一个 API 路由来做这件事
  await db.comment.create({ data: { content } });
}
CommentForm.tsx —— 组件里直接引用
import { createComment } from './actions';

function CommentForm() {
  return (
    {/* action 直接传入 Server Action 函数,表单提交自动触发它 */}
    <form action={createComment}>
      <textarea name="content" />
      <button type="submit">发表评论</button>
    </form>
  );
}

没有手写 onSubmit、没有手写 fetch、没有单独的 API 路由文件——表单的 action 属性直接指向这个 Server Action,提交时 Next.js 自动把表单数据序列化、发起请求、在服务端执行函数、把结果同步回来。

对比:与本系列 91 篇 React Router 的 action 处理表单

本系列 91 篇讲过 React Router 的 action——同样是"把表单提交交给框架处理,而不是手写 onSubmit + fetch"的思路。但两者有一个关键区别:

React Router 的 action(91篇)
export async function action({ request }) {
  const formData = await request.formData();
  // 这段代码依然运行在浏览器里
  // (纯客户端 SPA,没有真正的服务端)
  return fetch('/api/comments', {
    method: 'POST',
    body: formData,
  });
}

本质:统一了表单提交的"入口"和
loading/错误状态管理,但最终
还是要调用一个真实的后端 API
Next.js Server Action
'use server';
export async function createComment(formData) {
  // 这段代码真正运行在服务端
  await db.comment.create({ ... });
}

本质:函数本身就是"后端",
可以直接访问数据库、文件系统、
服务端专属的密钥等资源,
不需要再调用另一个 API
共同点与差异,一句话说清。 两者都是"表单提交交给框架统一处理"的思路,减少手写 onSubmit/fetch 的样板代码;差异在于 React Router 的 action 运行环境依然是浏览器(纯 SPA 没有服务端这一层),最终还是要调用一个真实存在的后端接口;而 Server Action 的函数本身就跑在服务端,可以直接碰数据库,不用再单独维护一层 API。
陷阱:Server Action 里不能想当然地信任前端传来的数据。 因为调用入口是浏览器发起的网络请求,恶意用户完全可以绕过表单直接构造请求调用这个函数——和写一个真实的 API 路由一样,参数校验、权限检查一个都不能少。有人误以为"函数写在组件旁边、看起来像本地调用"就等同于安全,这是常见的误区。

小结:整个 React 学习路径回顾

  • Server Actions 让表单提交/数据修改不再需要手写 API 路由 + 前端 fetch,组件里直接调用标记 'use server' 的函数,Next.js 自动处理网络层细节。
  • 与 91 篇 React Router 的 action 相比:两者都是"表单提交交给框架"的思路,但 Server Actions 真正运行在服务端,能直接访问数据库等资源,而不是仍要调用一个独立的后端接口。
  • Server Action 依然是一个可以被外部直接调用的网络入口,参数校验和权限检查一样不能省。

回顾整个系列,学习主线是:React 基础 → Hooks → 组件进阶 → 路由 → 状态管理 → 工程化 → 性能优化 → 生态实战——从最基础的组件渲染心智模型,到用 Hooks 管理副作用和状态,到组件设计模式,到用路由组织多页面应用,到 Zustand/Redux/TanStack Query 管理客户端与服务端状态,到 Vite 工程化与性能优化手段,最后落到测试和 Next.js 全栈这样的生态实战。这一路下来,已经具备了独立构建一个中大型 React 应用所需的完整知识框架。Next.js 全栈(SSR/SSG/RSC、文件系统路由、Server Actions,以及更进一步的缓存策略、Edge Runtime 等)是下一步值得深入的方向,把这套 React 知识体系真正延伸到服务端。
