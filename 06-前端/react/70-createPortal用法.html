createPortal：把子节点渲染到父 DOM 之外

通常一个组件渲染的 DOM，会乖乖待在父组件的 DOM 下面。Portal（传送门）打破了这条规则：它让你把子节点渲染到 DOM 树的任意位置，同时在 React 组件树里仍是原来的父子关系。这个"看似矛盾"的特性，正是 Modal、Tooltip 的关键。

基本用法：两个参数

createPortal(要渲染的内容, 目标 DOM 节点)
import { createPortal } from 'react-dom';

function Modal({ children }) {
  return createPortal(
    // 参数 1:要渲染的 React 节点
    <div className="modal">{children}</div>,
    // 参数 2:目标 DOM 容器(必须是已存在的真实 DOM 节点)
    document.getElementById('modal-root')
  );
}

假设 HTML 里有两个挂载点：

index.html
<div id="root"></div>       <!-- React 应用主体 -->
<div id="modal-root"></div> <!-- Portal 传送目的地,和 root 平级 -->

Modal 组件在 React 里是 <App> 深处的一个子孙，但它渲染出的 DOM 却直接出现在 #modal-root 里——脱离了 #root 这棵 DOM 子树。

React 树:App → Page → Modal
≠
DOM 树:Modal 挂到 #modal-root

核心：两棵树的分离

理解 Portal，关键是记住 React 里同时存在两棵"树"，Portal 让它们不再一致：

React 组件树（虚拟）真实 DOM 树
Modal 的位置还是 Page 的子组件被传送到 #modal-root 下
state / props 传递照旧，父组件正常传值—
Context照旧继承祖先的 Provider—
事件冒泡按 React 树冒泡不按 DOM 树
为什么 state / Context 照常工作？因为它们是 React 层面的机制，只看组件树，不看 DOM 落点。Portal 只改变最终 DOM 的挂载位置，不改变组件在 React 树中的父子关系。所以你的 Modal 依然能读到外层的主题 Context、依然收到父组件传来的 props——这正是 Portal 好用的地方:逻辑上还在原处,视觉上跑出去了。

最重要的陷阱：事件冒泡按 React 树，不按 DOM 树

这是 Portal 最反直觉、也是面试高频的点。虽然 Modal 的 DOM 被挂到了 #modal-root，但它触发的事件，仍然沿着React 组件树向上冒泡到它的 React 父组件——而不是沿着 DOM 树冒泡到 #modal-root。

点击 Portal 内的按钮,外层的 onClick 依然会触发
function Parent() {
  // 尽管 Modal 的 DOM 不在这个 div 里,
  // 点击 Modal 内的元素,这个 onClick 仍会被触发!
  return (
    <div onClick={() => console.log('父级收到点击')}>
      <Modal>       {/* DOM 被传送到 #modal-root */}
        <button>点我</button>
      </Modal>
    </div>
  );
}
为什么会这样?React 用的是合成事件系统:事件监听器统一挂在根节点上,React 自己按组件树模拟冒泡,而不是依赖浏览器原生的 DOM 冒泡。所以 Portal 里的元素虽然 DOM 上"跑出去了",在 React 眼里它的父亲还是原来那个组件,事件照样冒泡上去。

实战坑:做一个"点击外部关闭 Modal"的下拉菜单时,如果 Modal 用了 Portal,而你的关闭逻辑挂在祖先的 onClick 上,可能会误判——点 Modal 内部也会冒泡到祖先,触发意外关闭。此时应改用捕获阶段监听或判断 event.target。

需要卸载时的清理

createPortal 本身不会创建 DOM 容器——目标节点必须提前存在。如果你想动态创建容器(不写死在 HTML 里),记得在组件卸载时移除它,否则会残留孤儿节点:

动态创建 + useEffect 清理容器
function Portal({ children }) {
  const [el] = useState(() => document.createElement('div'));
  useEffect(() => {
    document.body.appendChild(el);
    return () => document.body.removeChild(el);  // 卸载时清理,避免泄漏
  }, [el]);
  return createPortal(children, el);
}

与其他知识点的联系

  • children 插槽模式(65 篇)：Portal 通常用来包裹 children——一个 Modal 组件对外仍暴露 children 插槽让使用者塞任意内容,内部再用 createPortal 把这些 children 渲染到别处,这是"组合"(children 决定内容)与"渲染位置控制"(Portal 决定挂载点)两个独立维度的叠加使用。
  • Context：本篇表格特别指出 Portal 不改变 Context 的继承关系——这呼应了更早的 Context 篇章"Context 沿组件树传递,不看 DOM 结构"的机制;Portal 恰好是一个很好的反例场景,用来验证 Context 到底是按 React 树还是 DOM 树工作的。
  • 合成事件系统 / 受控组件的 onChange：受控组件依赖 onChange 这样的合成事件在 React 树上正常冒泡触发 setState;Portal 场景下事件"按 React 树冒泡、不按 DOM 树"的特性,说明合成事件系统始终是以组件树为准的抽象层,不管 DOM 结构长什么样,这是理解 React 事件模型贯穿两个场景的同一个原理。
  • useEffect 清理副作用：动态创建 Portal 容器时用 useEffect 的清理函数移除节点,这是"副作用需要配对清理"这一通用原则(订阅要取消、定时器要清除、DOM 节点要移除)在 Portal 场景下的具体应用。
  • Portal 使用场景(下一篇)：本篇只讲"怎么用",下一篇回答"为什么非用不可"——把两篇连起来看,才能理解 Portal 不是一个可有可无的花活,而是解决 overflow: hidden 裁剪和 z-index 层叠上下文这两个真实 CSS 困境的必需手段。

小结

  • createPortal(node, container) 把 node 渲染进指定 DOM 容器,脱离父 DOM 层级。
  • 组件树关系不变:props、state、Context 全部照旧工作——只是最终 DOM 落点变了。
  • 事件按 React 组件树冒泡,不按 DOM 树——这是最容易踩的坑,做"点外部关闭"时尤其注意。
  • 目标容器必须已存在;动态创建的容器要在卸载时清理。
