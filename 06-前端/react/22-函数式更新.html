函数式更新：setState(prev => newValue)

上一节知道了 state 是快照，在同一渲染里多次 setCount(count + 1) 只会加一次。函数式更新就是为了解决这个问题——它让每次更新都能基于"最新的真实值"计算。

语法：传函数而不是传值

setState 接受两种参数：

  • 直接传值：setCount(42)——React 用 42 作为下次渲染的 state 值
  • 传函数：setCount(prev => prev + 1)——React 把当前最新值传给函数，用函数的返回值作为下次的 state 值
函数式更新语法
// 直接传值：用固定的新值替换 state
setCount(42);                // count 下次渲染变成 42
setName('Tom');             // name 下次渲染变成 'Tom'

// 函数式更新：React 保证 prev 是当前最新的 state 值
setCount(prev => prev + 1);  // 基于最新值 +1
setVisible(v => !v);        // 翻转布尔值的标准写法
setList(prev => [...prev, newItem]);  // 基于最新数组追加

// prev 只是约定命名，你也可以叫 current、old、_：
setCount(c => c + 1);       // 简写，React 社区常见风格

解决的问题：多次更新正确累加

回到上节的问题：同一事件里连续三次 setCount(count + 1) 只加了一次。函数式更新如何解决？

❌ 直接传值——只加 1
// count 快照 = 0
setCount(count + 1); // 计划→1
setCount(count + 1); // 计划→1（还是 0+1）
setCount(count + 1); // 计划→1（还是 0+1）

// 三次都在说"把 count 设为 1"
// 批处理合并：最终 count = 1
// 点一次只加了 1，不是 3
✅ 函数式更新——加 3
// React 把这三个函数排进更新队列
setCount(c => c + 1); // prev=0 → 1
setCount(c => c + 1); // prev=1 → 2
setCount(c => c + 1); // prev=2 → 3

// 每个函数拿到上一次的结果作为 prev
// 最终 count = 3 ✅
// 这就是函数式更新的价值

本质区别：传值时，React 直接覆盖；传函数时，React 把上一次的结果传给下一个函数，形成链式计算。

c=0 → c+1=1
→
c=1 → c+1=2
→
c=2 → c+1=3
→
count = 3 ✅

实际场景里的对比

场景：点一下按钮加三次（比如三连击动画）
function TripleCounter() {
  const [count, setCount] = useState(0);

  // ❌ 错误写法：三次都基于同一个快照，只 +1
  function wrongTriple() {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  }

  // ✅ 正确写法：函数式更新，每次基于前一次的结果
  function correctTriple() {
    setCount(c => c + 1);
    setCount(c => c + 1);
    setCount(c => c + 1);
    // React 把三个函数排队，依次传递结果，最终 +3
  }

  return (
    <div>
      <p>count = {count}</p>
      <button onClick={wrongTriple}>错误（只+1）</button>
      <button onClick={correctTriple}>正确（+3）</button>
    </div>
  );
}
经验法则：只要新值依赖旧值，一律用函数式
  • 数字累加/递减：setCount(c => c + 1)
  • 翻转布尔值：setVisible(v => !v)
  • 数组追加：setList(prev => [...prev, item])
  • 对象局部更新：setUser(prev => ({ ...prev, name: 'Tom' }))

只有在"设置一个与旧值完全无关的新值"时，才直接传值：setName('Tom')、setCount(0)（重置）。

切换布尔值的标准写法

在 React 社区里，切换布尔状态（如显示/隐藏、开/关）有一个约定俗成的写法，见到要认识：

切换布尔值——社区标准写法
const [open, setOpen] = useState(false);

// ✅ 函数式——基于当前值翻转，安全且简洁
<button onClick={() => setOpen(v => !v)}>切换</button>

// 也可以写成（效果相同，更短）
<button onClick={() => setOpen(o => !o)}>切换</button>

// ⚠️ 直接传值——单次调用完全没问题；若同一事件里多次调用则有竞态风险
<button onClick={() => setOpen(!open)}>切换</button>
// 多次调用时 !open 每次都是同一个快照值——函数式更新才能正确链式累积

什么时候用函数式，什么时候直接传值

场景推荐写法原因
累加、递减 setCount(c => c + 1) 新值依赖旧值，必须用函数式
翻转布尔 setOpen(v => !v) 新值依赖旧值（取反），必须用函数式
重置为固定值 setCount(0) 新值与旧值无关，直接传值即可
设置用户输入 setName(e.target.value) 新值来自事件，与旧值无关，直接传值
数组追加 setList(prev => [...prev, x]) 需要旧数组的内容，必须用函数式

常见陷阱

  • 混用直接传值和函数式，以为都"实时"：初学者常以为写了一次 setCount(c => c+1) 之后，紧跟着的 count（组件变量）就已经变成新值——实际上组件变量 count 直到下一次渲染前始终是本次渲染的快照，即便你用了函数式更新，prev 只在 React 内部链式传递，不会同步写回外部的 count 变量。
  • 在函数式更新里引入副作用：把 setCount(c => { console.log('run'); return c + 1; }) 这种带副作用的代码写进更新函数。在 React 18 严格模式下，某些场景（如 StrictMode 下的开发环境）这个函数可能被调用两次用于检测副作用，如果函数里有副作用就会被放大或产生错误的统计，更新函数应该是纯函数，只算出新值。
  • 函数式更新里直接修改 prev 本身：比如 setList(prev => { prev.push(x); return prev; })——虽然用了函数式写法，但内部对 prev 做了原地修改（push），返回的还是同一个引用，React 依然认为没有变化，页面不更新。函数式更新解决的是"取最新值"的问题，不代表内部可以违反不可变更新的规则。
  • 不必要地到处用函数式更新：如果新值完全不依赖旧值（比如 setName(e.target.value)），硬要写成 setName(prev => e.target.value) 反而让代码更难读，且没有任何实际收益——函数式更新只在"新值依赖旧值"时才有意义。

面试高频考点

  1. 为什么同一个事件处理函数里连续三次 setCount(count + 1) 只加了 1，而三次 setCount(c => c + 1) 能加 3？—— 因为 React 会把同步事件处理函数中的多次 setState 批处理（batching），在批处理结束前，事件函数体内引用的 count 始终是本次渲染时的旧快照，所以三次调用传的都是"旧值+1"这个固定结果，最终只生效一次。而函数式更新传入的是"如何计算新值"的函数，React 会把这些函数排成队列，依次把上一个函数的返回值作为下一个函数的参数执行，因此能正确累积。
  2. 函数式更新中的更新函数必须满足什么条件？—— 必须是纯函数：只根据传入的 prev 计算并返回新值，不能有副作用（网络请求、修改外部变量、原地修改 prev 本身等），因为 React 在某些情况下（如严格模式下的开发环境）可能会调用它两次来辅助检测副作用，多次调用不应该产生不同的结果。
  3. 直接传值和函数式更新在底层实现上有什么区别？—— React 内部会为每个 state 维护一个更新队列（update queue）。直接传值时，入队的是一个"替换为该值"的更新对象；传函数时，入队的是"用当前 state 调用该函数得到新 state"的更新对象。渲染前 React 会顺序遍历这个队列，依次执行每个更新，因此函数式更新天然支持链式计算，而直接传值的多次调用会互相覆盖。

与其他知识点的联系

  • state 异步更新（上一节）：函数式更新正是为了解决"state 是快照、同一渲染内 state 不会立即变化"这个问题而存在的——理解了 state 快照和批处理，才能明白为什么直接传值会互相覆盖，而传函数能链式累积。
  • 不可变更新（下一节）：函数式更新只解决"取到最新值"的问题，不代表可以在更新函数内部对 prev 做原地修改（如 prev.push(x)）。两者结合起来才是正确写法：setList(prev => [...prev, x])——用函数式拿到最新数组，再用不可变的方式基于它生成新数组。
  • useReducer：useReducer 的 reducer 函数本质上就是"广义的函数式更新"——它接收当前 state 和一个 action，返回新 state，和 setCount(c => c + 1) 的思路完全一致，只是把"如何计算新值"的逻辑从内联函数抽成了具名的 reducer，适合更复杂的状态变化场景。
  • 闭包陷阱：如果在 useEffect 或定时器里直接引用组件变量 count 来计算新值（如 setInterval(() => setCount(count + 1), 1000)），会因为闭包捕获了创建时的旧快照而导致计数不准；用函数式更新 setCount(c => c + 1) 可以完全绕开这个闭包陷阱，因为它不依赖外部变量，只依赖 React 传入的最新值。
  • Immer：Immer 的 produce(state, draft => { draft.count++ }) 也可以作为函数式更新的参数传给 setState，让"看起来像直接修改"的写法在底层生成不可变的新对象，是函数式更新和不可变更新两个理念结合的典型工具。
