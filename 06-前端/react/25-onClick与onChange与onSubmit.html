React · 事件处理

onClick / onChange / onSubmit

React 的事件处理属性都是驼峰命名，值是一个函数引用。这三个是使用频率最高的，搞懂它们基本覆盖了 80% 的交互需求。

onClick：点击交互

onClick 是最常用的事件，适用于按钮、div、任何可点击元素。处理函数接收一个 MouseEvent 类型的合成事件。

onClick 基本用法
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  // 事件处理函数：定义在组件函数体内，可访问 state 和 props
  function handleAdd() {
    setCount(count + 1);
  }

  return (
    <div>
      <p>计数：{count}</p>

      {/* 传函数引用，不加括号 */}
      <button onClick={handleAdd}>+1</button>

      {/* 也可以写内联箭头函数 */}
      <button onClick={() => setCount(0)}>重置</button>
    </div>
  );
}
常见错误：onClick={handleAdd()}
加了括号 = 在渲染时立即调用函数，返回值（通常是 undefined）才是 onClick 的值。结果：渲染时就执行了，点击时什么都不发生，且可能无限触发渲染。
正确：onClick={handleAdd}——传的是函数本身，等用户点击时 React 才调用它。

onChange：输入值变化

React 的 onChange 与 HTML 的 oninput 行为等价——每次按键后立即触发，而不是失去焦点才触发（这与 HTML 原生 onchange 不同）。

通过 e.target.value 拿到当前输入框的值。

onChange 绑定受控 input
import { useState } from 'react';

function SearchBox() {
  const [query, setQuery] = useState('');

  function handleChange(e) {
    // e.target 是触发事件的 DOM 元素，.value 是当前值
    setQuery(e.target.value);
  }

  return (
    <div>
      <input
        value={query}
        onChange={handleChange}
        placeholder="搜索..."
      />
      <p>当前输入：{query}</p>
    </div>
  );
}

select 和 textarea 的用法与 input 相同，都通过 e.target.value 获取值：

select 的 onChange
const [color, setColor] = useState('red');

<select value={color} onChange={e => setColor(e.target.value)}>
  <option value="red">红色</option>
  <option value="green">绿色</option>
  <option value="blue">蓝色</option>
</select>

onSubmit：表单提交

绑定在 <form> 上，用户按回车或点击 submit 按钮时触发。几乎每次都需要调用 e.preventDefault()，否则页面会刷新（表单的默认行为是向服务器 GET/POST）。

onSubmit 完整示例
import { useState } from 'react';

function LoginForm() {
  const [name, setName] = useState('');

  function handleSubmit(e) {
    // 必须阻止默认刷新行为，否则 React 状态全部丢失
    e.preventDefault();
    console.log('提交：', name);
    // 这里调用 API、更新状态等
  }

  return (
    {/* onSubmit 绑定在 form 上，不是按钮上 */}
    <form onSubmit={handleSubmit}>
      <input
        value={name}
        onChange={e => setName(e.target.value)}
        placeholder="用户名"
      />
      {/* type="submit" 触发 form 的 onSubmit */}
      <button type="submit">登录</button>
    </form>
  );
}

常见陷阱

  • 把 onChange 当成失去焦点才触发：受原生 HTML onchange（失焦才触发）的经验影响，容易误以为 React 的 onChange 也是这样，实际上 React 的 onChange 等价于原生的 oninput，每次按键、粘贴都会立即触发。如果需要"失焦才处理"的语义，应该用 onBlur，不要在 onChange 里做节流判断。
  • 受控组件只写了 value 没写 onChange：<input value={query} /> 若没有配套 onChange，React 会把这个输入框渲染成只读状态——用户在浏览器里根本无法输入任何字符，并且开发环境下 React 会在控制台报警告。这是"受控组件"心智没建立起来的典型信号：value 由 state 决定，就必须靠 onChange 更新 state，两者是一体的。
  • 忘记表单的 onSubmit 要绑在 <form> 上而不是按钮上：把提交逻辑写在按钮的 onClick 里，虽然多数情况也能凑效，但会丢失表单默认的"按回车提交"能力，且键盘无障碍访问、浏览器自带的表单校验（如 required）等特性都依赖于原生 submit 事件流经 <form>。
  • checkbox / radio 用 value 而不是 checked 判断状态：把复选框写成 value={done} 而非 checked={done}，会导致复选框视觉状态和实际数据脱节——checkbox/radio 的选中态由 checked 属性控制，value 只是提交给表单的值，两者是完全不同的概念。

面试高频考点

  1. React 的 onChange 和原生 DOM 的 onchange 有什么区别？—— 原生 onchange 只在元素失去焦点且值发生变化时才触发；React 出于"实时同步 state 与输入框显示值"的受控组件设计需要，把 onChange 实现为与原生 oninput 等价的行为，每次输入都会立即触发，这样才能保证 value 和用户输入保持同步渲染。
  2. 为什么 <input value={x} /> 没写 onChange 会导致输入框只读，而不是报错崩溃？—— React 把这种情况当作"受控只读组件"处理：因为 value 完全由 React 控制，每次渲染都会强制把 DOM 的实际值设回 value prop 指定的值，用户的按键输入在下一次渲染时就被"纠正"回去，视觉上表现为无法输入。这是 React 为了保证受控语义一致性做的兼容处理，同时在开发环境打印警告提醒开发者补上 onChange 或显式设置为 readOnly。
  3. 为什么表单提交建议用 onSubmit 而不是按钮的 onClick？—— onSubmit 绑定在 <form> 上，能同时响应"点击 submit 按钮"和"输入框内按回车"两种触发方式，还能触发浏览器内建的表单校验（如 required、pattern）。只监听按钮的 onClick 会丢失键盘提交路径和原生校验能力，属于对 HTML 表单语义的破坏性简化。

与其他知识点的联系

  • 合成事件与原生事件的区别：本节 onClick/onChange/onSubmit 接收到的事件对象都是上一节讲的 SyntheticEvent，接口与原生事件一致但由 React 统一委托管理——理解了合成事件的机制，才知道为什么 e.target.value、e.preventDefault() 在这些事件里能安全使用。
  • 受控组件：onChange 是受控组件的核心拼图——value 由 state 决定，state 由 onChange 里的 setState 更新，二者构成一个闭环。只讲 onChange 语法而不理解受控组件的"单向数据流"思想，很容易漏写 onChange 导致输入框只读。
  • 阻止默认行为与冒泡：onSubmit 几乎总要搭配 e.preventDefault() 阻止表单默认的整页刷新提交，这是"阻止默认行为"最典型的应用场景之一，下一节会专门展开这个知识点。
  • 传参给事件处理函数：本节的例子里事件处理函数大多不需要额外参数（如 handleAdd、handleChange），但实际业务中经常需要给处理函数额外传参（比如列表项的 id），这就需要用箭头函数包一层，是下一个知识点要解决的问题。
  • 函数式更新：onClick 里的 setCount(count + 1) 如果在同一事件里被多次调用会有批处理导致的"只加一次"问题，这时需要用函数式更新 setCount(c => c + 1)，这是事件处理函数内部更新 state 时容易被忽略的细节。
