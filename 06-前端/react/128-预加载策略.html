预加载策略

纯懒加载解决了"首屏体积过大"的问题,却引入了一个新的小烦恼:用户点击链接的那一刻才开始下载代码,期间会有一瞬间的等待和 loading 闪烁。预加载策略的思路是:在用户还没点、但很可能要点之前,提前把代码下载好,点击时几乎无感切换。

纯懒加载的副作用

用户点击"订单"链接
→
才开始下载 OrderList 的 chunk
→
下载 + 解析(有感知的等待)
→
渲染出页面

网络越差,这段"点击 → 真正看到内容"之间的等待越明显。既要享受代码分割带来的首屏优化,又不想让用户在切换页面时等,答案就是把下载时机提前——不是不分割,而是提前触发那个本来要在点击时才发生的 import()。

常见预加载时机

时机原理
鼠标 hover 链接用户把鼠标移到链接上,到真正点击通常还有几十到几百毫秒,这段时间足够提前发起 import()
路由 loader 阶段并行拉取React Router 的 loader 在导航过程中就会执行,可以和组件代码的 import() 并行,而不是等组件下载完再去请求数据
空闲时间预加载用 requestIdleCallback 在浏览器空闲时提前加载"大概率会访问"的下一步页面
视口可见时预加载用 IntersectionObserver 监测链接是否进入可视区域,进入即触发预加载

hover 预加载的代码示例

思路很简单:把 import() 单独抽出来,链接的 onMouseEnter 和真正跳转都调用同一个函数——第一次调用会真正发起下载,第二次(比如用户真的点击了)因为浏览器/打包工具的模块缓存,直接复用已经下载好的结果,几乎瞬间完成。

hover 时提前触发 import(),点击时无感知加载
// 和 lazy() 里用的是同一个 loader 函数,保证走同一份模块缓存
const loadOrderList = () => import('./pages/OrderList');
const OrderList = lazy(loadOrderList);

function NavLink() {
  return (
    <Link
      to="/orders"
      onMouseEnter={() => loadOrderList()}  // 鼠标移入就提前下载,结果被浏览器/打包器缓存
    >
      订单
    </Link>
  );
}
// 用户真正点击时,import() 直接命中缓存,Suspense fallback 几乎不会出现
为什么点两次 import() 不会重复下载? ES 模块规范保证同一个模块路径在同一个模块图里只会被求值一次,后续 import() 拿到的是同一个已缓存的 Promise/模块对象。所以"hover 时调一次、点击时（lazy 内部)又调一次"完全安全,不会有重复网络请求或重复副作用。

结合路由 loader 并行拉取

React Router 的数据路由(createBrowserRouter + loader)在开始导航的那一刻,就会同时启动"下载组件代码"和"执行 loader 拉取数据"这两件事,而不是等组件代码下载完、渲染出来之后才发起数据请求——这叫 render-as-you-fetch,比"先加载代码、代码里的 useEffect 再去发请求"的传统模式快得多。

loader 与懒加载组件并行:两件事同时开始
{
  path: '/orders/:id',
  element: <Suspense fallback={<Skeleton />}><OrderDetail /></Suspense>,
  // 导航开始的瞬间,loader 请求数据 和 OrderDetail 的 import() 同时发起
  loader: ({ params }) => fetch(`/api/orders/${params.id}`),
}
预加载太激进的代价:浪费带宽。 如果给页面上所有链接都加无差别的 hover 预加载,用户鼠标扫过一堆链接却只点了一个,其余全部白下载——对于移动网络或流量敏感场景,这是实打实的成本。常见折中:只对高频入口(导航栏、"下一步"按钮)做预加载,长尾页面维持纯懒加载。
太保守也不行:体验差。 完全不做预加载,每次导航都要经历"点击 → 下载 → loading → 渲染"的完整链路,即便代码分割已经把每个 chunk 做得很小,累积的等待感依然存在。是否预加载、预加载到什么粒度,本质是带宽成本 vs 交互流畅度之间的取舍,没有放之四海皆准的答案,需要结合真实用户的网络环境和访问路径来定。
代码分割三篇小结: React.lazy + Suspense 解决"怎么渲染懒加载组件",import() 是底层拆包机制,预加载策略解决"什么时候提前触发下载"。三者组合起来,才是一套完整的、体验和性能兼顾的代码分割方案。

小结

  • 纯懒加载会在用户点击时才开始下载代码,带来一瞬间的加载等待,预加载策略通过提前触发 import() 来缓解。
  • 常见预加载时机:鼠标 hover、路由 loader 阶段并行拉取、浏览器空闲时、元素进入可视区域。
  • hover 预加载只需把 import() 抽成独立函数,在 onMouseEnter 和 lazy() 里复用同一个引用,依靠模块缓存避免重复下载。
  • React Router 的 loader 可以和懒加载组件的下载并行进行(render-as-you-fetch),比"先下载代码再发请求"更快。
  • 预加载粒度是带宽成本和交互流畅度的权衡:全站无差别预加载浪费流量,完全不做又体验差,应聚焦在高频入口按需开启。

常见陷阱

  • 全站无差别 hover 预加载——给页面上所有链接的 onMouseEnter 都挂 import(),看起来"处处优化",实际上用户鼠标划过菜单时会连续触发一串下载请求,大部分链接根本不会被点击。这些请求会和当前页面真正需要的资源(图片、接口数据)抢带宽,尤其在弱网下反而拖慢了正在进行的交互。应该只对导航栏、"下一步"这类高确定性会被点击的入口做预加载。
  • 移动端照搬 hover 逻辑——触屏设备没有"鼠标悬停"这个中间状态,手指触摸屏幕的瞬间基本等同于点击,`onMouseEnter` 在移动端几乎不会被真正触发,或者触发时机和点击几乎同时发生,起不到"提前"的作用。照搬桌面端的 hover 预加载方案,在移动端本质上等于没做预加载,得单独接入 touchstart 或视口可见触发的方案。
  • 预加载函数和 lazy() 用了不同的引用——如果 hover 时调用的是重新写的一份 import('./pages/OrderList'),而 lazy() 里又单独写了一份看似一样的 import(),打包工具通常仍会识别为同一个模块从而复用缓存,但一旦路径写法不完全一致(比如一个带查询参数一个不带),就会被当成两个不同的模块图节点,导致重复下载。最安全的做法是像文中示例那样把 import() 抽成一个具名函数,hover 和 lazy() 严格复用同一个函数引用。
  • 只做代码预加载,不管数据请求——只对组件的 import() 做了预加载,却让页面渲染后才在 useEffect 里发起数据请求,依然是"下载完代码 → 渲染 → 再等一轮数据"的串行链路,预加载省下的时间被数据请求原地吃掉了。真正发挥效果需要像 React Router 的 loader 那样,让组件代码下载和数据拉取在导航开始的同一时刻并行发起。
  • 用预加载掩盖代码分割粒度问题——如果某个 chunk 本身体积过大,团队倾向于"反正提前预加载就行了",而不去反思是否该拆得更细。预加载只是把等待时间从"点击后"挪到"点击前",总下载量没有减少;chunk 太大依然会占用带宽、拖慢空闲时间预加载的整体效率,治标不治本。

面试高频考点

  1. hover 预加载为什么调用两次 import() 不会重复下载?—— 因为 ES 模块规范保证同一个模块路径在同一个模块图里只会被求值一次,第二次 import() 拿到的是同一个已缓存的 Promise/模块对象,前提是两次调用引用的是完全一致的模块路径(同一个具名 loader 函数)。
  2. 什么是 render-as-you-fetch,它比传统的"渲染后再 useEffect 发请求"快在哪?—— render-as-you-fetch 指在导航/渲染开始的同一时刻就并行发起数据请求,而不是等组件渲染完成后再在 useEffect 里发请求。React Router 的 loader 会和懒加载组件的 import() 同时启动,两件事并行推进,总耗时约等于两者中较慢的一个,而不是把"下载代码"和"请求数据"的耗时依次叠加。
  3. 预加载策略中,hover、路由 loader、空闲时间、视口可见这几种触发时机分别适合什么场景?—— hover 适合桌面端高确定性入口(导航链接),路由 loader 阶段并行拉取适合已经进入导航流程的下一步页面和数据,空闲时间(requestIdleCallback)适合"大概率会访问但不紧急"的资源,不抢占当前交互的资源;视口可见(IntersectionObserver)适合长列表里即将滚动到的内容,兼容移动端因为不依赖鼠标事件。
  4. 为什么不能对全站所有链接都做无差别预加载?—— 预加载本质是用"可能浪费的带宽"换"确定的交互流畅度",全站铺开会导致用户鼠标划过一堆链接却只点了一个,其余全部白下载,在移动网络或流量敏感场景是实打实的成本;应该只针对高频、高确定性会被访问的入口开启,长尾页面维持纯懒加载。

与其他知识点的联系

  • 代码分割与虚拟滚动：预加载策略解决的是"资源什么时候到达"的问题(提前触发 import()),而虚拟滚动(133 篇)解决的是"数据到手后一次渲染多少 DOM 节点"的问题;两者作用在完全不同的层——一个在资源加载层提速,一个在渲染层减负,互不冲突,组合起来才是完整的性能优化链路。
  • React.memo / useMemo / useCallback：这几个 Hook 优化的是"数据已经到达后,组件要不要重新渲染",而预加载优化的是"数据/代码到达的时机"。就算一个组件用 memo 包得再好,如果它的代码要等用户点击才开始下载,该有的加载等待感依然存在——预加载和记忆化优化的是同一条用户体验链路上的两个不同阶段。
  • render-as-you-fetch 与 useDeferredValue：文中提到 React Router 的 loader 和组件下载并行发起,这种"尽早开始,不等前一步完成"的思路,和 136 篇 useDeferredValue 让紧急更新不被非紧急渲染阻塞的思路有相通之处——都是通过重新安排"事情发生的顺序/时机"来提升体验,而不是减少总的工作量。
  • 组件组合与嵌套：13 篇讲组件应该按职责拆分,而预加载策略里"把 import() 抽成独立的具名 loader 函数,在 hover 和 lazy() 里复用同一个引用"这一实践,本质上也是一种"抽离共享逻辑,避免重复定义"的组合思想,只是应用在模块加载而不是 UI 组件上。
  • 带宽成本与记忆化开销的对称性：预加载"太激进浪费带宽、太保守体验差"这种权衡,和 useMemo/useCallback"滥用了反而增加比较开销、不用又会有多余的计算或重渲染"是同一类工程决策——任何优化手段都有隐藏成本,该不该用取决于具体场景的收益是否盖过成本,不存在无脑套用的万能方案。
