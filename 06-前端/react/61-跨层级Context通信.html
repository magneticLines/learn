Context:跨层级通信,告别 prop drilling

状态提升在两三层内很好用,但如果数据要从最顶层传到深埋七八层的组件呢?一路 props 透传太痛苦。Context 提供了一条"直达通道",让深层组件不经中间层就能拿到祖先的数据。

痛点:prop drilling(属性逐层钻透)

假设主题色 theme 存在根组件,而真正用它的是深处的 Button。中间的 Page、Panel、Toolbar 自己根本不用 theme,却被迫层层接收再传下去:

App(有 theme)
→
Page(透传)
→
Panel(透传)
→
Button(用)

中间组件被无关的 props 污染,重构时牵一发动全身。这就是 prop drilling——React 里最恼人的"样板代码"来源之一。

Context 三步走:创建、提供、消费

createContext + Provider + useContext
// ① 创建 Context(通常单独放一个文件,给个默认值)
const ThemeContext = createContext('light');

// ② 提供:在祖先用 Provider 包裹,value 就是要广播的数据
function App() {
  const [theme, setTheme] = useState('dark');
  return (
    <ThemeContext.Provider value={theme}>
      <Page />   // Page/Panel 不用再接收、传递 theme
    </ThemeContext.Provider>
  );
}

// ③ 消费:任意深度的后代直接读取,无需层层传
function Button() {
  const theme = useContext(ThemeContext);  // 直接拿到 'dark'
  return <button className={theme}>点我</button>;
}

消费者会读取最近的那个 Provider 的 value。如果组件树上没有任何 Provider 包裹,才用 createContext 时给的默认值——这个默认值主要用于单元测试和"忘记加 Provider"的兜底。

Context 也能广播函数,实现深层"子传父"

value 可以是任意值——对象、函数都行。把 state 和它的 setter 一起放进 value,深层组件就能直接触发更新:

把 { theme, toggleTheme } 一起广播
const ThemeContext = createContext(null);

function App() {
  const [theme, setTheme] = useState('dark');
  const toggle = () => setTheme(t => (t === 'dark' ? 'light' : 'dark'));
  return (
    // value 是对象:同时提供数据和修改数据的方法
    <ThemeContext.Provider value={{ theme, toggle }}>
      <DeepChild />
    </ThemeContext.Provider>
  );
}

function DeepChild() {
  const { theme, toggle } = useContext(ThemeContext);
  return <button onClick={toggle}>当前 {theme}</button>;
}

关键取舍:什么时候该用 Context?

场景推荐方案原因
父子、两三层内props / 状态提升直接、显式,数据流一眼可见
数据要穿透很多层Context避免中间层被无关 props 污染
全局性数据(主题、语言、登录用户)Context几乎所有组件都要读,天然全局
频繁变化的局部数据props / 状态提升Context 变化会重渲所有消费者,性能差
陷阱一:不要拿 Context 当"万能全局变量"
Context 解决的是"传递路径太长",不是"我懒得传 props"。滥用 Context 会让组件失去可复用性(隐式依赖某个 Provider)、让数据流变得隐蔽难追。官方明确说:先考虑状态提升,层数实在太深再上 Context。
陷阱二:Provider 的 value 每次渲染都是新对象
value={{ theme, toggle }} 每次父组件渲染都会创建新对象,导致所有 useContext 消费者都重渲染,即使 theme 没变。频繁重渲染的 Context 应该用 useMemo 包裹 value,或把"数据"和"方法"拆成两个 Context。
Context vs Redux/Zustand
Context 本身不是状态管理库,它只是"传递机制",不自带优化。搭配 useReducer 可以做轻量状态管理。但对于大型应用中高频更新的全局状态,专门的库(Redux Toolkit、Zustand、Jotai)提供了细粒度订阅,只重渲染真正用到变化数据的组件,性能优于裸 Context。

小结

  • Context 解决 prop drilling:让深层后代不经中间层直接读取祖先数据。
  • 三步:createContext 创建 → Provider value= 提供 → useContext 消费。
  • value 可以是对象/函数,一起广播 state 和 setter 即可实现深层"子传父"。
  • 取舍:两三层用状态提升,数据穿透很深或全局性数据(主题/语言/用户)才用 Context。
  • 注意 value 引用稳定性,必要时用 useMemo,避免消费者无谓重渲染。
