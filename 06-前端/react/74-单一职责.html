单一职责:一个组件只做一件事

写 React 最容易长歪的地方,是让一个组件"什么都管"——既拉数据、又算业务、又渲染一堆 UI。单一职责原则(SRP)说的是:一个组件应该只有一个"变化的理由"。做到这一点,组件才好读、好测、好复用。

什么是"一个变化的理由"

单一职责的经典定义不是"只写一个函数",而是——一个模块应该只对一类使用者/一个变化来源负责。落到组件上:如果"数据格式变了"和"UI 样式变了"这两件本该独立的事,都会逼你去改同一个组件,那这个组件就承担了多个职责。

换个提问方式最实用:"这个组件为什么会被改?" 如果答案有好几个不相干的原因(接口字段调整、设计稿改版、埋点逻辑变更……),说明它揉了多个职责。理想状态是每个原因只影响一个小组件。这与后端的 SRP 是同一个思想,只是载体从"类"换成了"组件"。

坏味道:一个组件干了三件事

❌ 数据获取 + 业务筛选 + 渲染,全塞一起
function UserDashboard() {
  const [users, setUsers] = useState([]);
  const [keyword, setKeyword] = useState('');

  // 职责 1:数据获取
  useEffect(() => { fetch('/api/users').then(...).then(setUsers); }, []);

  // 职责 2:业务逻辑(筛选 + 排序)
  const visible = users
    .filter(u => u.name.includes(keyword))
    .sort((a, b) => a.age - b.age);

  // 职责 3:一大坨 UI 渲染
  return (
    <div>
      <input value={keyword} onChange={e => setKeyword(e.target.value)} />
      {visible.map(u => ( /* 每个卡片几十行 JSX... */ ))}
    </div>
  );
}

这个组件被改的理由至少三个:接口字段变(职责1)、筛选规则变(职责2)、卡片样式变(职责3)。任何一个变动都要来动这一大坨,而且它几乎无法复用、无法单独测试。

识别信号:过大组件的坏味道

坏味道说明
行数失控一个组件几百行,读一遍都费劲(经验阈值:超过 ~150 行就该警惕)
useState 一大串五六个以上互不相关的 state,往往意味着多个关注点混在一起
难起名取名要用 "and":UserListAndFilterAndModal —— 名字里有 and 就是多职责的信号
JSX 深度嵌套缩进层层叠叠,一个 return 里塞了好几块独立 UI
数据获取 + 展示混杂useEffect 拉数据和渲染细节写在同一个组件里
改 A 却怕碰坏 B改样式提心吊胆怕影响逻辑 —— 说明关注点没分离

拆分后:各司其职

✅ 数据 / 业务抽进 Hook,UI 拆成纯展示组件
// ① 数据 + 业务逻辑 → 自定义 Hook(可复用、可单测)
function useUsers(keyword) {
  const [users, setUsers] = useState([]);
  useEffect(() => { /* fetch */ }, []);
  return users.filter(u => u.name.includes(keyword));
}

// ② 纯展示:只管一个用户卡片长什么样
function UserCard({ user }) { return /* 卡片 JSX */; }

// ③ 组装:只负责把它们串起来,自身很薄
function UserDashboard() {
  const [keyword, setKeyword] = useState('');
  const users = useUsers(keyword);
  return (
    <>
      <SearchBar value={keyword} onChange={setKeyword} />
      {users.map(u => <UserCard key={u.id} user={u} />)}
    </>
  );
}
拆分带来的直接好处:useUsers 可以脱离 UI 单独测试、也可被别的页面复用;UserCard 是纯展示组件,给什么 props 渲染什么,极易在 Storybook 里预览、极易写快照测试;UserDashboard 变得很薄,一眼看懂结构。三者各自"只有一个变化的理由"。
别矫枉过正。单一职责不等于"越小越好"。把一个 <button> 也拆成独立组件、把三行 JSX 也抽出去,只会制造一堆穿透 props 的碎片组件,反而更难读。职责的粒度是"一件有意义的事",不是"一行代码"。何时该拆、何时打住,是第 76 篇的主题。

小结

  • 单一职责 = 一个组件只有一个变化的理由;用"它为什么会被改?"来自检。
  • 典型的多职责坏味道:数据获取 + 业务逻辑 + UI 渲染混在一个大组件里。
  • 拆分方向:业务/数据逻辑抽进自定义 Hook,UI 拆成纯展示组件,顶层组件只做组装。
  • 信号:行数失控、state 一大串、名字带 and、改 A 怕碰坏 B。
  • 但别过度拆分——职责粒度是"一件有意义的事",不是一行代码。
