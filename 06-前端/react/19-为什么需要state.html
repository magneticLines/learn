为什么需要 state：普通变量的两个致命问题

在学 useState 的用法之前，先搞清楚它解决了什么问题。如果普通变量就够用，React 根本不需要发明 state 这个概念。

问题引入：用普通变量存数据

你可能会想：我在组件里 let count = 0，点击时 count++，不就能记录点击次数了吗？

试一下：

看起来合理，实际完全不工作
function Counter() {
  let count = 0;  // 普通 JS 变量

  function handleClick() {
    count++;
    // 变量确实改了，但页面没有任何变化
    // 控制台打印可以证明 count 在增加
    console.log(count);  // 1, 2, 3... 值确实变了
  }

  // 但 UI 永远显示 0，因为 React 从来没有重新渲染这个组件
  return <button onClick={handleClick}>点击了 {count} 次</button>;
}

控制台打印的 count 在不断增加，说明 JS 逻辑是对的。但页面上永远显示"点击了 0 次"。这里藏着两个致命问题。

普通变量的两个致命问题

问题一
改了 React 不知道——不触发重新渲染

React 渲染 UI 的方式是：执行组件函数，把返回的 JSX 映射到 DOM。React 不像 Vue 那样用 Proxy 监视变量的变化。你改了一个普通 JS 变量，React 完全感知不到，它不会重新执行组件函数，UI 自然不会更新。

类比：你改了一份 Word 文档，但没有保存，打印机拿到的还是旧版本。

问题二
每次渲染都重置——上次的值消失了

就算 React 某种方式触发了重新渲染，组件函数会从头执行一遍，let count = 0 这行代码会再次运行，count 被重置为 0。上一次点击积累的值，在重新渲染时全部丢失。

类比：你在一张便利贴上写数字，但每次查看时便利贴都被换成一张新的空白贴。

代码对比：普通变量 vs useState

❌ 普通变量——点击无效
function Counter() {
  // 每次渲染重置为 0
  let count = 0;

  function handleClick() {
    count++;
    // React 不知道这件事
    // 不会触发重新渲染
  }

  return (
    <button onClick={handleClick}>
      {count}  // 永远是 0
    </button>
  );
}
✅ useState——正常工作
function Counter() {
  // React 帮你在渲染之间保存值
  const [count, setCount] =
    useState(0);

  function handleClick() {
    setCount(count + 1);
    // React 知道了！安排重渲
  }

  return (
    <button onClick={handleClick}>
      {count}  // 正确更新
    </button>
  );
}

state 解决了什么

useState 一次性解决了上面两个问题：

  • 在渲染之间"记住"值：state 的值不存在组件函数的局部作用域里，而是存在 React 内部维护的一块存储区域。组件函数重新执行时，useState 会从这块区域里取出上次保存的值，而不是重新初始化。
  • 值变化时触发重新渲染：调用 setter 函数（setCount）是在告诉 React"这个组件的 state 变了，你需要重新渲染它"。React 收到这个通知，才会重新执行组件函数，生成新的 JSX，更新 DOM。

这两件事——记住值和触发重渲——都是 React 帮你管理的，你只需要用 useState 声明，用 setter 更新。

state 不是魔法：state 是 React 内部维护的一个"与组件实例绑定的存储"。当你调用 useState(0)，React 在内部记下"这个组件第 N 个 hook 的当前值是 0"。每次渲染时，React 按顺序取出对应的值交给你——这就是为什么 useState 的调用顺序必须固定，不能放在 if/循环里。如果顺序变了，React 就取错槽位了。

总结：为什么普通变量不够用

特性普通变量 let x = 0state useState(0)
在渲染之间保存值 ❌ 每次渲染重置 ✅ React 帮你保存
值变化触发重新渲染 ❌ React 不知道 ✅ setter 通知 React
适合存什么 事件处理里的临时变量、计算中间值 需要展示在 UI 上、会随时间变化的数据

常见陷阱

  • 以为"直接操作 DOM 更新数字"能替代 state：有人会想绕开重新渲染，写 document.querySelector('button').textContent = count 直接改 DOM 文本。这在极简单场景下"看起来"能用，但它彻底脱离了 React 的声明式渲染模型——React 完全不知道 DOM 被手动改过，下一次因为其他原因触发重新渲染时，React 会用它内部记录的（旧的）数据重新生成 JSX，把你手动改的 DOM 内容覆盖掉，造成显示和实际数据不一致的诡异 bug。
  • 把只在事件处理函数内部临时用的值也塞进 state：state 的核心价值是"跨渲染保留 + 变化触发渲染"，如果一个值只在某次事件处理过程中临时计算、用完就扔（比如判断"这次点击是否是双击"的时间差），不需要展示在 UI 上，也不需要触发重新渲染，用普通局部变量或 useRef 即可，滥用 state 会引入不必要的重新渲染开销。
  • 误以为"变量值变了，页面就应该跟着变"是理所当然的：这是从 Vue（基于 Proxy 自动追踪依赖）或命令式 UI 编程习惯迁移过来的人最容易踩的坑。React 的心智模型是"数据变化 → 显式通知 React → React 决定要不要重新执行组件函数"，普通变量的赋值只是内存里的一次写操作，和"通知 React"完全是两件事，React 没有任何机制去监视普通变量的读写。
  • 忽略"重新渲染会从头执行组件函数"这件事，误用 let/const 存储需要保留的数据：本篇的"问题二"讲得很清楚：函数组件每次渲染都是重新调用这个函数，函数体内所有 let x = ... 都会重新执行一遍。哪怕通过某种方式（比如强制刷新父组件）触发了重新渲染，普通变量记录的历史值也一样会被清零，因为它的生命周期只有"这一次函数调用"那么长，而 state 的生命周期是跟组件实例绑定的，能跨越多次调用持续存在。
  • 把 state 和"UI 上显示的值"划等号，忽略了它也可能只是"驱动渲染的中间数据"：state 不一定要直接显示在页面上，它可以是任何"变化时需要让组件重新渲染"的数据（比如一个决定要不要显示某个区块的布尔值、一份需要重新计算派生数据的原始列表）。判断要不要用 state 的标准是"这个值变化后是否需要 UI 跟着变"，而不是"这个值是不是直接印在屏幕上的文字"。

面试高频考点

  1. 为什么普通的 let 变量不能用来存储需要展示在 UI 上、会变化的数据？—— 两个根本原因：一是 React 不会追踪普通变量的读写，变量变了但 React 感知不到，不会触发重新渲染；二是即便触发了重新渲染（因为别的原因），函数组件本身是"每次渲染重新执行一遍"，普通变量的声明语句也会重新执行，把之前累积的值重置掉。state 通过把值存在 React 内部维护的、和组件实例绑定的存储区域里，解决了"值需要跨渲染保留"的问题；通过 setter 函数主动通知 React，解决了"变化需要触发渲染"的问题。
  2. React 怎么知道该给这次渲染中的 useState 返回哪个值？为什么 Hook 不能写在 if/循环里？—— React 内部按调用顺序而不是变量名去匹配每个 useState 对应的存储槽位（可以理解成一个和组件实例绑定的数组，第几次调用 useState 就取数组第几项）。如果把 Hook 调用包在 if/循环里，不同渲染之间 Hook 的调用次数或顺序可能不一致，会导致 React 把第二次渲染的某个 Hook 调用错配到第一次渲染时别的 Hook 的槽位上，造成状态错乱——这就是 React 官方规定"Hook 只能在组件顶层按固定顺序调用"的底层原因。
  3. 调用 setState 之后，React 具体做了什么，才让页面"看起来"更新了？—— setState（如 setCount）并不直接操作 DOM，它做的是把新值记录进 state 存储区域，并"调度"（schedule）一次重新渲染。React 随后重新执行这个组件函数，生成新的 JSX 树，再通过 diff（协调算法）和上一次的树做比较，只把真正变化的部分应用到真实 DOM 上。所以 UI 更新是"数据变化 → 重新执行组件函数 → 生成新 JSX → diff → 局部更新 DOM"这一整条链路的结果，不是数据变化本身直接引发的。

与其他知识点的联系

  • useState 基本用法（下一篇 20）：本篇讲的是"为什么需要一个能跨渲染保存值、又能触发重渲染的机制"，下一篇直接给出这个机制的具体用法（语法、命名约定、多个 state 独立管理）。先理解本篇的"两个致命问题"，再看 useState 的每一部分参数（初始值、setter）分别对应解决哪个问题，会比直接背语法理解得更深。
  • state 异步更新（21 篇）：本篇结尾提到"调用 setter 是在通知 React 重新渲染"，21 篇会进一步展开——这个"通知"不是立即生效的，而是进入队列、等本次事件处理完毕后批量处理。理解了本篇"state 存在 React 内部维护的存储区域"这个心智模型，才能理解 21 篇讲的"快照"概念——每次渲染拿到的 state 都是那个存储区域在渲染那一刻的一份只读快照。
  • Props 只读原则 / Props 默认值（17-18 篇）：props 和 state 是 React 里两类不同来源的数据——props 由外部传入、组件不能修改；state 由组件自己声明和更新。本篇讲的"普通变量为什么不够用"，本质上也回答了"为什么不能像改 props 一样直接改变量"——因为 React 的渐进渲染模型要求所有会触发 UI 变化的数据都必须经过它能感知到的通道（props 传递或 setState 调用），而不能是任意的赋值语句。
  • useEffect（本系列后续 Hooks 篇）：本篇的流程图"调用 setter → React 收到通知 → 重新执行组件函数 → UI 更新"只覆盖了渲染本身；如果状态变化后还需要执行"渲染之外"的副作用（比如发请求、订阅事件），就需要 useEffect 监听这个 state 的变化——这是本篇知识往后延伸的自然方向。
  • Java 后端的可变状态管理类比：对有 Java 背景的学习者，普通变量"改了但没人知道"这个问题，可以类比成一个字段值变了却没有触发任何监听器/观察者（Observer 模式）——React 的 state 机制本质上是内置了一套"变更必须显式通知"的框架级约束，避免了手写观察者模式的样板代码，但代价是必须遵守它规定的调用方式（如 setter、Hook 顺序）。
