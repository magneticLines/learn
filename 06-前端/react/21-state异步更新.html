state 更新是异步的：批处理机制

这是 React 初学者踩得最多的坑。调用 setCount(count + 1) 之后，count 没有立即变成新值。理解这一点，能避免大量难以排查的 bug。

核心结论：setState 是"排队"，不是"立即修改"

当你调用 setCount(newValue)，React 做的事是：把这次更新加入队列，然后安排一次重新渲染。它不会立刻改变 count 的值。

在当前这次函数执行（当前渲染）里，count 的值是固定的——就是这次渲染开始时 React 传给你的那个"快照"值。

state 是快照：每次渲染时，React 给组件一份当前 state 的"快照"。在这次渲染期间，不管你调用多少次 setter，这份快照里的值都不会变。新值要等到下一次渲染才能读到。

代码示例：调用 setter 后立刻读，读到的是旧值

setState 后立即 console.log —— 看到的是旧值
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    // 此时 count = 0（当前渲染的快照值）
    setCount(count + 1);

    // 你以为 count 现在是 1？并不是
    console.log(count);  // 打印 0，不是 1 ！

    // setCount 只是"告诉 React 下次渲染用 1"
    // 当前函数执行期间，count 始终是这次渲染的快照值 0
  }

  return <button onClick={handleClick}>点击（当前：{count}）</button>;
}

点击后，UI 上的数字会变成 1，但 console.log 打印的是 0。这不是 bug，这是 React 的设计——在下一次渲染之前，你拿到的 count 永远是本次渲染的快照。

批处理（Batching）：多次 setState 合并成一次渲染

React 18 里有一个重要的性能优化：同一个事件处理函数里的多次 setState 会被合并（批处理），只触发一次重新渲染。

这是好事（减少不必要渲染），但也解释了一个让人困惑的现象：

连续三次 setCount(count + 1) —— 结果只加了 1
function handleClick() {
  // 假设当前 count = 0

  setCount(count + 1);  // 基于快照 count=0，计划设为 1
  setCount(count + 1);  // 基于快照 count=0，计划设为 1（！不是 2）
  setCount(count + 1);  // 基于快照 count=0，计划设为 1（！不是 3）

  // 三行 setCount 都在用同一个快照值 0 计算
  // 三个"计划"都是"把 count 设为 1"
  // 批处理合并后，最终 count = 1，不是 3
}

用"外卖订单"类比：你在同一秒内下了三个单，但三单的备注都写的是"送到 1 号楼"——系统合并后只有一单，目的地是 1 号楼，不是 3 号楼。

count = 1 ❌（期望 3）

批处理的渲染时机

同一事件里多个 state 的更新，合并成一次渲染
function handleSubmit() {
  setName('Tom');      // 1. 排队
  setAge(25);          // 2. 排队
  setVisible(true);   // 3. 排队
  // 事件处理完毕后，React 批量处理，只渲染一次
  // 不是三次 setState 触发三次渲染——这就是批处理的价值
}
经典坑：为什么连续 setCount 只加了一次？
setCount(count + 1); setCount(count + 1);——两次 setState，都基于同一个旧的 count 快照，结果只加一次。这正是需要"函数式更新"的场景，下一节会讲如何用 setCount(c => c + 1) 解决这个问题。

常见陷阱

  • 把"异步"理解成"要等一段时间才更新"，而不是"要等到下一次渲染"：这里的"异步"跟 setTimeout、网络请求那种时间上的异步不是一回事。它指的是当前这次函数执行期间 state 值不会变，必须等到 React 重新执行组件函数（下一次渲染）才能看到新值——这个"等待"和时间长短无关，哪怕下一行代码立刻读取，读到的也还是旧快照。理解成"排队等下一帧"比"异步"这个词本身更准确。
  • 连续调用同一个 setter 多次，期望值线性累加，结果只加了一次：本篇讲的经典坑——setCount(count + 1) 连写三次，因为每一次里的 count 都是同一个快照值（比如都是 0），三次调用本质是三个"把 count 设为 1"的计划，批处理合并后只生效一次。这是初学者最容易在"多次点击应该加好几次"的需求里踩到的坑，根源在于混淆了"读取快照值"和"读取最新值"。
  • 在 setState 之后立刻用 console.log 或后续逻辑判断"是不是已经更新了"：因为快照在本次渲染期间不会变，任何紧跟着 setState 之后、在同一个函数调用栈里对该变量的读取，都只能读到旧值。想在"更新之后"执行某些逻辑，正确方式是用 useEffect 监听这个 state 的变化，而不是在事件处理函数里 setState 之后直接判断。
  • 误以为批处理只发生在 React 18 之前的合成事件里，在异步回调中就不会批处理：React 18 之前，批处理只在 React 自己管理的事件处理函数（比如 onClick）内部生效，在 setTimeout、Promise.then、原生事件监听器里多次 setState 会各自触发一次渲染。React 18 引入了"自动批处理"，把这个范围扩大到几乎所有场景。如果项目还跑在 React 17 及更早版本，或者用了某些绕过自动批处理的 API，behavior 会不一样，这是版本迁移时容易忽略的细节。
  • 把"快照"心智模型和"闭包陷阱"混为一谈，却不知道两者的根本原因是同一个：很多人遇到"事件处理函数里拿到的 state 总是旧值"时，第一反应是"这是闭包问题"，但更准确的说法是：这个事件处理函数是在某一次渲染期间创建的，它捕获（闭包）的正是那次渲染的 state 快照，这两个说法描述的是同一件事的不同角度——渲染快照本质上就是通过闭包实现的。

面试高频考点

  1. 为什么 setCount(count + 1) 后立即 console.log(count) 打印的是旧值？—— 因为每次渲染时 React 会把当时的 state 值作为一个只读的"快照"提供给这次渲染的组件函数（以及函数内部定义的事件处理函数）。setCount 调用只是把"下一次渲染要用的新值"加入更新队列，并不会就地修改当前渲染闭包里的 count 变量。要拿到新值，必须等 React 用新 state 重新执行组件函数——也就是下一次渲染。
  2. React 的批处理（batching）具体做了什么，为什么连续三次 setCount(count + 1) 只加了 1？—— 批处理指的是：在同一个事件处理函数执行期间，多次 setState 调用不会立刻逐一触发渲染，而是被收集进一个队列，等这个函数执行完毕后一次性处理、只重新渲染一次。因为三次调用发生在同一次渲染的闭包里，count 这个快照值始终是同一个（比如 0），三次调用实质上都是"基于 0 计算出 1，把 count 设为 1"，批处理去重合并后自然只体现一次 +1，而不是期望的 +3。
  3. 批处理是性能优化还是正确性保证？会不会有副作用？—— 首先是性能优化：避免一个事件里多次 state 更新导致多次重新渲染和多次 DOM diff，把开销收敛成一次。但它确实有副作用——如果开发者不了解"快照值在本次渲染期间恒定不变"这条规则，会写出连续 setState 却只生效一次的 bug（如上一题）。这也是为什么"函数式更新"（setCount(c => c + 1)）被推荐用在需要连续累加的场景：它让每次更新都基于队列里上一个更新的结果计算，而不是基于同一个过期快照。
  4. 怎么在 state 更新之后，"确实"执行一些依赖新值的逻辑？—— 不能指望在调用 setState 的同一行代码之后同步读取新值，因为新值要等下一次渲染才存在。正确做法是用 useEffect(() => { ... }, [count])，把依赖新值的逻辑放进 effect，React 会在组件用新的 count 完成渲染之后执行它，这时候 count 已经是更新后的值。

与其他知识点的联系

  • 函数式更新（下一篇 22）：本篇结尾的 callout 已经点明——连续 setCount(count + 1) 只加一次的坑，正是下一篇要解决的问题。函数式更新 setCount(c => c + 1) 让每次更新都基于"队列里上一步的结果"而不是"过期的渲染快照"计算，这是本篇"快照"心智模型的直接应用，不理解快照就理解不了为什么函数式写法能解决这个问题。
  • useState 基本用法（上一篇 20）：上一篇教了语法，本篇教了这个语法背后的时序陷阱——两篇合起来才是"会用 useState"的完整知识，只学上一篇会写出"看起来对、实际上有坑"的代码。
  • useEffect 与依赖数组（本系列 Hooks 篇）：本篇提到"想在 state 更新之后执行逻辑要用 useEffect"，这背后的原理和本篇讲的渲染时序完全一致——useEffect 的回调在组件用新 state 完成渲染之后才执行，这也是为什么它能读到"真正的新值"而不是某次渲染的旧快照。
  • 闭包（JavaScript 基础）：本篇"常见陷阱"里提到"快照"和"闭包陷阱"是同一件事的不同说法——事件处理函数是在某次渲染期间被创建的，它捕获的正是那次渲染时的变量值。理解 JS 闭包如何捕获外部变量，是理解 React state 快照行为的语言层面基础，这不是 React 独有的机制，而是 JS 函数作用域规则在 React 渲染模型里的自然体现。
  • useReducer（本系列后续复杂状态管理篇）：批处理机制在简单场景下（比如单个计数器）容易通过函数式更新绕过陷阱，但当一次事件里需要根据多个关联 state 计算出一致的新状态时，分散的多次 setState 调用会让"基于哪个快照计算"变得难以追踪，这正是 useReducer 把多个关联的状态更新逻辑收敛成一个纯函数、统一处理的动机之一。
