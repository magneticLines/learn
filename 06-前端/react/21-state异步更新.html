state 更新是异步的：批处理机制

这是 React 初学者踩得最多的坑。调用 setCount(count + 1) 之后，count 没有立即变成新值。理解这一点，能避免大量难以排查的 bug。

核心结论：setState 是"排队"，不是"立即修改"

当你调用 setCount(newValue)，React 做的事是：把这次更新加入队列，然后安排一次重新渲染。它不会立刻改变 count 的值。

在当前这次函数执行（当前渲染）里，count 的值是固定的——就是这次渲染开始时 React 传给你的那个"快照"值。

state 是快照：每次渲染时，React 给组件一份当前 state 的"快照"。在这次渲染期间，不管你调用多少次 setter，这份快照里的值都不会变。新值要等到下一次渲染才能读到。

代码示例：调用 setter 后立刻读，读到的是旧值

setState 后立即 console.log —— 看到的是旧值
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    // 此时 count = 0（当前渲染的快照值）
    setCount(count + 1);

    // 你以为 count 现在是 1？并不是
    console.log(count);  // 打印 0，不是 1 ！

    // setCount 只是"告诉 React 下次渲染用 1"
    // 当前函数执行期间，count 始终是这次渲染的快照值 0
  }

  return <button onClick={handleClick}>点击（当前：{count}）</button>;
}

点击后，UI 上的数字会变成 1，但 console.log 打印的是 0。这不是 bug，这是 React 的设计——在下一次渲染之前，你拿到的 count 永远是本次渲染的快照。

批处理（Batching）：多次 setState 合并成一次渲染

React 18 里有一个重要的性能优化：同一个事件处理函数里的多次 setState 会被合并（批处理），只触发一次重新渲染。

这是好事（减少不必要渲染），但也解释了一个让人困惑的现象：

连续三次 setCount(count + 1) —— 结果只加了 1
function handleClick() {
  // 假设当前 count = 0

  setCount(count + 1);  // 基于快照 count=0，计划设为 1
  setCount(count + 1);  // 基于快照 count=0，计划设为 1（！不是 2）
  setCount(count + 1);  // 基于快照 count=0，计划设为 1（！不是 3）

  // 三行 setCount 都在用同一个快照值 0 计算
  // 三个"计划"都是"把 count 设为 1"
  // 批处理合并后，最终 count = 1，不是 3
}

用"外卖订单"类比：你在同一秒内下了三个单，但三单的备注都写的是"送到 1 号楼"——系统合并后只有一单，目的地是 1 号楼，不是 3 号楼。

setCount(0+1)
+
setCount(0+1)
+
setCount(0+1)
→ 批处理 →
count = 1 ❌（期望 3）

批处理的渲染时机

同一事件里多个 state 的更新，合并成一次渲染
function handleSubmit() {
  setName('Tom');      // 1. 排队
  setAge(25);          // 2. 排队
  setVisible(true);   // 3. 排队
  // 事件处理完毕后，React 批量处理，只渲染一次
  // 不是三次 setState 触发三次渲染——这就是批处理的价值
}
经典坑：为什么连续 setCount 只加了一次？
setCount(count + 1); setCount(count + 1);——两次 setState，都基于同一个旧的 count 快照，结果只加一次。这正是需要"函数式更新"的场景，下一节会讲如何用 setCount(c => c + 1) 解决这个问题。
