React · 组件组合

children:用插槽把内容"塞"进组件

怎么写一个 Card 组件,让别人往里面放任意内容?靠一堆 props 堆参数太笨。React 有更优雅的方式:children ——组件标签之间包裹的内容,会自动作为一个特殊 prop 传进来。这就是组合优于配置的起点。

children 是什么:标签之间的内容

当你写 <Card>这里的东西</Card>,开闭标签之间的一切,React 会打包成 props.children 传给 Card。组件只要把 children 渲染出来,就成了一个"容器"。

Card:一个可以包裹任意内容的容器
// 容器组件:只负责"外壳"(边框、内边距、阴影),不关心里面是什么
function Card({ children }) {
  return <div className="card">{children}</div>;
}

// 使用:标签之间放什么,就渲染什么——高度灵活
<Card>
  <h2>标题</h2>
  <p>正文...</p>
  <button>操作</button>
</Card>

类比 HTML:<div>...</div> 本身不关心里面装什么。children 让你自定义组件也拥有这种"包裹"能力,像给内容套了个相框。

组合优于配置:为什么不用 props 堆参数?

❌ 配置式:props 越加越多
<Card
  title="标题"
  content="正文"
  showButton={true}
  buttonText="操作"
  hasIcon={true}
  iconName="star"
/>
// 需求一变就要加新 prop,
// 组件内部 if 判断越来越乱
✅ 组合式:内容自由填
<Card>
  <Icon name="star" />
  <h2>标题</h2>
  <p>正文</p>
  <button>操作</button>
</Card>
// Card 永远不用改,
// 想放什么使用者说了算
组合优于配置(composition over configuration)
配置式:组件用一大堆 props 参数去"描述"所有可能的内容,内部靠条件判断拼装,可能性越多组件越臃肿。组合式:组件只提供"骨架/外壳",把"填什么"的权力交给使用者,通过嵌套 JSX 组合。后者的组件保持简单、稳定,扩展性来自"组合"而非"改内部"——这与 Unix "小工具组合" 哲学、React 的整体设计一脉相承。

典型用途:Layout / Modal / 权限包裹

Layout:统一页面骨架,内容由页面自己填
function Layout({ children }) {
  return (
    <div>
      <Header />
      <main>{children}</main>  // 每个页面往这里塞自己的内容
      <Footer />
    </div>
  );
}

<Layout><HomePage /></Layout>

具名"插槽":用 props 传 JSX

children 是"默认插槽",只有一个位置。如果一个组件需要多个内容区(如卡片的头、身、尾),可以把 JSX 作为普通 props 传进去——JSX 本质就是一个值,能存进变量、当参数传。

多插槽:header / footer 各占一格
// 组件声明多个"槽位",每个是一段 JSX
function Panel({ header, children, footer }) {
  return (
    <div className="panel">
      <header>{header}</header>       // 具名插槽 1
      <section>{children}</section>  // 默认插槽
      <footer>{footer}</footer>       // 具名插槽 2
    </div>
  );
}

// 用时:JSX 直接作为 prop 传入(用大括号包裹)
<Panel
  header={<h2>标题</h2>}
  footer={<button>确定</button>}
>
  这里是正文(走默认 children 槽)
</Panel>
Vue 用户注意:Vue 有专门的 <slot> / 具名插槽语法。React 没有"插槽"这个语言特性——它不需要,因为 JSX 本身就是值,children 和"传 JSX 当 prop" 就自然实现了单插槽和多插槽。React 的哲学是"少发明语法,多用 JS 本身的能力"。
陷阱:children 类型不固定
children 可能是字符串、单个元素、数组(多个子节点)、甚至 undefined(没传)。别假设它一定是数组就去 .map。要遍历/操作 children 时,用官方工具 React.Children.map / React.Children.toArray,它们能安全处理各种情况。

与其他知识点的联系

  • props：children 本质上就是一个特殊的 prop,只是它的"传值方式"不是写在标签属性上,而是写在标签之间——理解了 props 是组件的输入,才能理解 children 只是 props 家族里"内容型"的那一个,而不是什么全新的机制。
  • render props(下一篇)：children 传的是"写好的 JSX"(静态结构),render props 传的是"待调用的函数"(动态数据)——这一篇结尾埋的钩子会在下一篇被扭转成对比主线:同样是"把内容/逻辑交给使用者",children 交的是内容,render props 交的是数据。
  • 组合 vs 继承(67 篇)：本篇讲的"组合优于配置"是"组合优于继承"这个更大原则在 props 设计层面的体现——67 篇会把 children 列为"组合"武器库里的第一件武器,用来对比 OOP 继承在 UI 复用场景里的力不从心。
  • 受控组件 / 非受控组件(62-64 篇)：children 解决的是"内容该由谁提供"的问题,受控/非受控解决的是"值该由谁掌管"的问题——两者是同一种设计取向("组件少管、使用者/DOM 多管")在不同维度的应用,合起来看能发现 React 组件设计里反复出现的"控制权下放"主题。
  • Portal(70-71 篇)：children 让内容在 React 组件树里"随意组合",而 Portal 让渲染结果在真实 DOM 树里"随意搬家"——两者都在打破"组件结构 = 视觉/DOM 结构"的默认假设,只是 children 动的是逻辑组合,Portal 动的是物理挂载位置。

小结

  • children 是组件开闭标签之间的内容,作为特殊 prop 自动传入,让组件成为"容器"。
  • 组合优于配置:组件只提供外壳,内容由使用者组合,组件保持简单稳定。
  • 典型用途:Card、Layout、Modal、权限包裹等"包一层"的组件。
  • 需要多个内容区时,把 JSX 作为普通 props 传入,实现"具名插槽"(React 无需专门的 slot 语法)。
  • 操作 children 用 React.Children.*,不要假设它一定是数组。
