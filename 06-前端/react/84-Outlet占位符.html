Outlet 占位符:嵌套路由的核心机制

真实应用里,页面几乎都有共享外壳——顶部导航、侧边栏、页脚。这些外壳不该在每个页面里重复写一遍。React Router 用嵌套路由 + <Outlet />解决:父路由画外壳,子路由填内容。

问题:每个页面都重复写导航栏

如果路由是"平铺"的,每个页面组件都得自己引入导航栏、侧边栏,布局代码到处复制:

❌ 平铺路由:布局在每个页面里重复
// 每个页面都要自己套一遍 Navbar / Sidebar,改一处得改 N 处
function Dashboard() {
  return (
    <div>
      <Navbar /> <Sidebar />
      <h1>仪表盘</h1>
    </div>
  );
}

解法:父路由是外壳,Outlet 是插槽

把导航栏、侧边栏提到一个父路由的布局组件里,布局组件用 <Outlet /> 标记"子路由内容渲染在这里"。<Outlet /> 就是一个占位符/插槽——React Router 会把当前匹配的子路由 element 塞进这个位置。

路由配置:嵌套的 <Route>
import { Routes, Route } from 'react-router-dom';

<Routes>
  {/* 父路由:path="/app" + 布局组件 Layout */}
  <Route path="/app" element={<Layout />}>
    {/* 子路由:path 是相对父路由的,不要写开头的 / */}
    <Route path="dashboard" element={<Dashboard />} />
    <Route path="settings" element={<Settings />} />
  </Route>
</Routes>
布局组件:用 <Outlet /> 标记子路由的坑位
import { Outlet } from 'react-router-dom';

function Layout() {
  return (
    <div className="app">
      <Navbar />          {/* 共享:所有子页面都有 */}
      <Sidebar />
      <main>
        <Outlet />     {/* ★ 子路由内容渲染在这里 */}
      </main>
    </div>
  );
}

访问 /app/dashboard 时,React Router 匹配到父路由 Layout 和子路由 Dashboard,渲染结果是"Layout 的外壳里,Outlet 位置放着 Dashboard"。切到 /app/settings,外壳不动,只有 Outlet 里的内容从 Dashboard 换成 Settings。

Layout(外壳)
⊃
Navbar + Sidebar
+
<Outlet/> → Dashboard

类比:Vue 的插槽、母版页、组件的 children

如果你写过 Vue,<Outlet /> 就相当于 <router-view />;如果写过 ASP.NET,它像"母版页里的 ContentPlaceHolder"。

更贴近 React 本身的类比:<Outlet /> 的作用就像组件的 props.children,只不过"填进来的内容"不是父组件手动传的,而是由当前 URL 决定的——路由匹配到哪个子路由,就把哪个组件填进 Outlet。

Outlet 和 children 的本质区别
props.children 是父组件在 JSX 里静态写死的;<Outlet /> 的内容是路由系统根据 URL 动态注入的。这也是为什么嵌套路由能做到"外壳保持挂载、只换中间内容"——切换子路由时,Layout 组件本身不卸载不重新挂载,Outlet 里的内容才变。这对保留侧边栏滚动位置、避免导航栏闪烁非常重要。

多层嵌套:Outlet 可以套 Outlet

子路由本身也可以是一个带 <Outlet /> 的布局,形成多级嵌套。比如"设置"页面内部还有"个人资料 / 安全 / 通知"三个 tab:

两层嵌套:Settings 内部又是一层布局
<Route path="/app" element={<Layout />}>
  <Route path="settings" element={<SettingsLayout />}>
    <Route path="profile"  element={<Profile />} />
    <Route path="security" element={<Security />} />
  </Route>
</Route>

// URL /app/settings/security 的渲染层级:
// Layout > Outlet > SettingsLayout > Outlet > Security

常见陷阱

陷阱后果 / 修复
忘记在布局组件里放 <Outlet />子路由能匹配到,但页面上什么都不显示——因为没坑位。这是最常见的"路由对了页面空白"原因。
子路由 path 写成 /dashboard(带斜杠开头)会被当成绝对路径,脱离父路由。应写相对路径 dashboard。
访问父路径 /app 本身页面空白父路径精确匹配时没有子路由渲染,需要配 index 路由(见后续章节)。
useOutletContext:给所有子路由传共享数据
Outlet 支持传 context:<Outlet context={sharedData} />,子路由用 const data = useOutletContext() 读取。这比层层传 props 或专门开个 Context Provider 更轻量,适合"布局层持有、子页面共享"的数据(如当前登录用户)。

与其他知识点的联系

  • 布局路由（下一篇）：本文的父路由都带了 path="/app"，子路由 URL 因此多了前缀；布局路由是同一套 Outlet 机制的变体——父路由不写 path，只负责提供外壳，子路由的 URL 保持干净。理解了 Outlet 是"渲染插槽"，就知道布局路由改变的只是"要不要往 URL 里加前缀"，插槽机制本身完全不变。
  • index 路由：Outlet 解决了"子路由内容渲染在哪"，但没回答"父路径被精确访问时 Outlet 里该放什么"——这正是 index 路由补上的空白。两者是同一个 Outlet 插槽在不同 URL 精确度下的两种填充方式。
  • 认证守卫 PrivateRoute：守卫组件本质上就是一个"逻辑布局路由"——已登录时渲染 <Outlet /> 放行子路由，未登录时渲染 <Navigate> 拦截。不理解 Outlet 是"按条件决定要不要渲染的插槽"，就很难看懂守卫组件为什么可以这样写。
  • 404 页面：通配路由 path="*" 也能作为某个布局的子路由存在，这样未匹配的子路径依然会经过父级的 Outlet 渲染出"带外壳的局部 404"，而不是跳出整个布局显示一片空白——这依赖的正是 Outlet 对子路由内容的统一承载能力。
  • createBrowserRouter 与 loader：数据路由把路由配置从 JSX 挪到了对象数组，但嵌套结构（父路由 children 里放子路由）和 Outlet 渲染子内容的机制完全没变——loader 只是给每一层路由额外挂了个"渲染前取数"的钩子，路由树的组织方式仍然是 Outlet 那一套。
