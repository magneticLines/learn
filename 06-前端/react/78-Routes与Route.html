Routes 与 Route：声明式定义你的路由表

包好了 Router 容器，接下来要告诉它"哪个 URL 渲染哪个组件"——这就是 <Routes> 和 <Route> 的职责。它是整个路由系统的核心，v6 相比 v5 在这里做了最大的 API 改动。

核心写法:Routes 包着一组 Route

路由表是声明式的：你不写 if-else 判断当前 URL，而是像写配置一样列出所有"路径 → 组件"的映射，Router 自己去匹配。

App.jsx——一张典型的路由表
import { Routes, Route } from 'react-router-dom';

function App() {
  return (
    <Routes>                                    {/* 匹配器：只会渲染其中一个 Route */}
      <Route path="/" element={<Home />} />
      <Route path="/about" element={<About />} />
      <Route path="/users" element={<UserList />} />
      <Route path="*" element={<NotFound />} />  {/* * 兜底：都不匹配时的 404 */}
    </Routes>
  );
}

<Routes> 是"选择器"——它会在内部所有 <Route> 中挑出最匹配当前 URL 的那一个渲染，其余全部不渲染。<Route> 是一条规则：path 是要匹配的路径，element 是命中后渲染的内容。

v6 的关键变化:element 属性(vs v5 的 component)

如果你看过老教程或老代码，会发现 v5 的写法完全不同。这是面试和踩坑的高频点：

v5(旧)
<Switch>
  <Route
    exact
    path="/"
    component={Home}
  />
  <Route
    path="/user/:id"
    render={props => ...}
  />
</Switch>
v6(现行)
<Routes>
  <Route
    path="/"
    element={<Home />}
  />
  <Route
    path="/user/:id"
    element={<User />}
  />
</Routes>
项v5v6
容器<Switch><Routes>
指定组件component={Home} / render={fn}element={<Home />}
精确匹配需要 exact默认精确，无需 exact
传 props需 render 回调直接 element={<Home foo={x} />}
为什么从 component={Home} 改成 element={<Home />}? v5 传的是"组件类型"（一个函数引用），Router 内部帮你 <Home /> 实例化，导致你想传自定义 props 时必须绕道 render 回调，很别扭。v6 直接让你传已经写好的 JSX 元素，谁需要传 props 就直接写在标签上——更符合 React "一切都是元素"的直觉，也更灵活。这是 v6 设计上的一次简化。

匹配规则:最佳匹配,而非从上到下

这是 v6 相比 v5 最容易被忽略、也最重要的一个变化。

v5: <Switch> 从上到下逐个试，第一个匹配的就用——所以顺序极其重要，具体路径必须写在模糊路径前面，否则被"抢先"匹配。

v6: <Routes> 会给所有 Route 打分，选出最精确（最佳）的那一个，与书写顺序无关。

v6:顺序无所谓,静态段永远赢过动态段
<Routes>
  {/* 就算把动态段写在前面…… */}
  <Route path="/user/:id" element={<UserDetail />} />
  {/* …访问 /user/new 时，v6 仍会精准命中这一条（静态 > 动态） */}
  <Route path="/user/new" element={<CreateUser />} />
</Routes>
// 在 v5 里，/user/new 会被 /user/:id 抢先匹配到 UserDetail —— 经典 bug

打分的大致优先级：静态段（/user/new） > 动态段（/user/:id） > 通配符（*）。所以 path="*" 的 404 兜底路由，写在哪儿都能正确地"最后才被选中"。

常见误区:以为 v6 还要靠顺序。 从 v5 迁移过来的人常保留"具体路由写前面"的习惯，虽然在 v6 里这么写也不会错，但要理解这已经不是必需的了。真正的坑反过来：在 v6 里指望通过调整顺序改变匹配结果——不会生效，因为它算的是最佳匹配分数。

path 的写法要点

  • v6 默认精确且大小写不敏感，不再需要 exact。
  • 末尾斜杠一般会被忽略：/about 和 /about/ 视为同一条。
  • path="*" 表示"匹配任意剩余路径"，是 404 兜底的标准写法。
  • 嵌套路由（子路由 + <Outlet>）是另一大主题，后面单独讲。

小结

  • <Routes> 是选择器，从内部所有 <Route> 中挑一个渲染；<Route path element> 是一条"路径→组件"规则。
  • v6 用 element={<Comp />} 取代 v5 的 component/render，传 props 更直接。
  • v6 用 <Routes> 取代 <Switch>，默认精确匹配，不再需要 exact。
  • 最重要：v6 是最佳匹配（按精确度打分），与书写顺序无关；静态段优先于动态段优先于通配符。
  • path="*" 做 404 兜底。

与其他知识点的联系

  • BrowserRouter 与 HashRouter(上一篇):Router 容器负责"改 URL 不刷新页面",Routes/Route 负责"URL 变了该渲染谁"——前者是地基,后者是地基上的规则表,必须先包好 Router 才能使用 Routes,这也是为什么两篇顺序相连。
  • Link 与 NavLink(下一篇):Routes/Route 定义的是"路径 → 组件"的静态映射表,而 Link/NavLink 是触发这张映射表被重新匹配的方式之一——点击 Link 改变 URL,Routes 立刻根据新 URL 重新选出最佳匹配的 Route。
  • 动态段与 useParams(第81篇):本篇提到的 path="/user/:id" 这类动态段写法,正是第81篇 useParams 的数据来源——理解 Routes 如何用最佳匹配算法给动态段和静态段打分,才能理解为什么 /user/new 不会被 /user/:id 意外抢先匹配。
  • 何时拆分组件(第76篇):路由表本身是一种"按页面职责"组织组件的方式——每个 <Route> 对应的 element 组件,天然满足"不同变化理由对应不同组件"的拆分原则,只是这里的拆分粒度由 URL 结构决定。
