单向数据流

React 规定数据只能从父组件流向子组件。这个约束不是限制，而是让组件树的状态变化变得可预测、可追踪的核心设计。

什么是单向数据流

在 React 组件树里，数据的传递方向只有一个：从父组件到子组件，通过 props 向下流动。子组件可以读取父组件传来的数据，但不能直接修改它。

这和"双向绑定"（如 Vue 的 v-model、Angular 的 [(ngModel)]）形成对比：双向绑定允许子组件直接改变绑定的值，而单向数据流要求子组件通过"通知父组件"的方式来触发变更。

数据流向示意
Parent(state)
──props──▶
Child
数据向下流（props）
Parent(setState)
◀──callback──
Child(event)
事件向上传（回调函数）

为什么要单向：双向的代价

想象一个有 20 个组件的页面，每个组件都能直接修改其他组件的状态。当某个数据显示异常时，你需要在 20 个组件里排查"是哪个组件在什么时机改了这个值"——这几乎不可能。

单向数据流强制规定：一份状态只有一个"主人"（声明它的组件），只有主人才能修改它，其他组件只能读。这样当数据出了问题，你只需要看一个地方。

双向数据流单向数据流（React）
状态修改来源任何组件都可能修改只有状态的"主人"才能修改
Debug 难度难以追踪状态从哪里被改了变化路径清晰，易追踪
代码复杂度简单场景代码少需要写回调函数，稍多
适合场景表单等局部简单双向绑定复杂应用状态管理
水流类比：数据像山上的水，只能从山上（父组件）往山下（子组件）流。如果山下的村民想改变水源，他们不能爬上山自己动水管（那样会乱，不同的村民可能同时乱动），而是给山上的管理员打电话（调用父组件传下来的回调函数），由管理员统一决定怎么调整。这样整条水流永远只有一个"控制中心"，出了问题知道找谁。

代码示例：父持有状态，子只读与回调

单向数据流：完整的父子通信模式
// 父组件：状态的"主人"，只有它能修改 count
function Parent() {
  const [count, setCount] = useState(0);

  // 把"状态值"和"修改状态的能力"一起传给子组件
  // 子组件无法绕过这个回调直接改 count
  return (
    <div>
      <h2>父组件管理的计数：{count}</h2>
      <Counter
        count={count}
        onIncrement={() => setCount(c => c + 1)}
        onReset={() => setCount(0)}
      />
    </div>
  );
}

// 子组件：只能"读" count，想改必须调父组件给的回调
// 这个组件完全不知道 count 从哪来，也不关心父组件如何管理它
function Counter({ count, onIncrement, onReset }) {
  return (
    <div>
      <span>当前值：{count}</span>
      {/* 触发事件 → 调回调 → 父组件改状态 → React 重渲染 → 新 count 通过 props 流回来 */}
      <button onClick={onIncrement}>+1</button>
      <button onClick={onReset}>重置</button>
    </div>
  );
}

注意整个数据变化的完整循环：

  1. 用户点击按钮 → 触发 onClick
  2. 调用父组件传来的 onIncrement 回调
  3. 父组件里执行 setCount，状态更新
  4. React 触发重新渲染，Counter 收到新的 count props
  5. 界面更新

整个过程数据流方向始终是单向的：向下流（props），变更通知向上传（回调）。

状态提升：共享状态的正确姿势

如果两个兄弟组件需要共享同一份数据，React 的做法是状态提升（Lifting State Up）：把状态提升到它们共同的父组件，再分别通过 props 向下传递。

原则：状态应该放在"需要它的所有组件的最近公共祖先"里。这是单向数据流下管理共享状态的核心原则，也是理解 Redux、Zustand 等状态管理库存在意义的前提——当"公共祖先"变成了整个应用的根，就需要专门的状态管理工具来替代层层 props 传递。
常见陷阱：Props 穿透（Prop Drilling）
当组件层级很深时，你可能需要把某个状态从顶层一路通过 props 传到第 N 层的子组件，中间每一层都要接收再转发这个 prop，即使它们自己根本不用。这叫"Props 穿透"或"Prop Drilling"。

这不是单向数据流的问题，而是状态位置选择不当或组件层级过深的信号。解决方案是 React.createContext（Context API）或外部状态管理库，它们让你跳过中间层直接传递数据——但数据流方向仍然是单向的。
