本地 state 与全局 state：状态应该放在哪里

很多人一学 React 就急着装 Redux/Zustand，把所有状态都塞进全局仓库。其实绝大多数状态应该是本地的。这一篇讲清楚一个贯穿始终的原则：把状态放在离使用它最近的地方，只有真正跨组件共享才提升为全局。

先问一句：这个状态谁在用？

决定状态放哪儿，唯一要问的问题是：有多少个组件需要读或写它？

  • 只有一个组件用 → 放这个组件的 useState（本地 state）。
  • 几个相邻组件用 → 提升到它们最近的公共父组件，用 props 往下传（状态提升,lifting state up）。
  • 相隔很远的一堆组件都要用 → 才考虑 Context 或外部状态库（全局 state）。

这是一条渐进的路径：默认本地，需要共享才往上抬，抬到实在传不动了才全局化。不要一上来就跳到最后一步。

原则：状态放在离使用它最近的地方

这条原则叫 colocation（就近放置）。为什么这么设计?因为状态的作用域越小,你要理解它、修改它、调试它就越容易——你不用担心"改这个值会不会影响到不认识的组件"。

一个搜索框:状态只有它自己用,就该是本地的
function SearchBox() {
  // query 只有这个组件用来渲染 input,没有别人关心它
  // 放本地 state 天经地义,提到全局纯属污染
  const [query, setQuery] = useState('');

  return <input value={query}
    onChange={e => setQuery(e.target.value)} />;
}

需要共享时,先想"状态提升",而不是"全局化"

如果两个兄弟组件要共享同一份数据(比如一个筛选器和一个列表),标准做法是把 state 抬到它们的公共父组件,而不是塞进全局:

状态提升:兄弟组件共享,抬到公共父即可
function ProductPage() {
  // filter 被 FilterBar 和 ProductList 两个子组件共享
  // 抬到它们最近的公共父 ProductPage,用 props 下发
  const [filter, setFilter] = useState('all');

  return (
    <>
      <FilterBar value={filter} onChange={setFilter} />
      <ProductList filter={filter} />
    </>
  );
}
什么时候提升"传不动"了? 当你为了把一个值送到深层子组件,不得不让中间七八层组件逐层转发它根本不用的 props——这叫 props drilling(逐层钻孔)。这种痛感就是"该上 Context / 状态库"的真正信号。别的时候都不该急。

过早全局化的坏处

把本该本地的状态提到全局,不是"更灵活",而是给自己埋雷:

✅ 本地 state
组件卸载,状态自动销毁
作用域清晰,一眼看到谁改它
改动只影响自己
天然支持"多个实例各自独立"
调试时状态就在组件旁边
❌ 过早全局化
组件卸载了,脏数据还留在仓库
谁都能改,追 bug 像大海捞针
牵一发动全身,重渲染范围失控
两个实例共享一份 state,互相干扰
状态和使用它的组件天各一方
最典型的过早全局化:表单输入框的值。 新手常把每个 input 的当前值都放全局仓库,理由是"以后可能要用"。结果是:仓库塞满了只有一个组件关心的临时值、每敲一个字母就触发全局更新、组件卸载后残留脏数据。表单输入是本地 state 的教科书级场景——放 useState 就好。

一份"状态放哪里"的判断表

状态类型典型例子推荐位置
组件私有的 UI 状态输入框内容、下拉是否展开、hover本地 useState
相邻组件共享筛选条件 + 列表、表单 + 预览提升到公共父
全应用共享的 UI 状态主题、语言、登录用户、侧栏开合Context / 状态库
来自服务端的数据用户列表、订单详情数据请求库(下一篇细讲)
能从别的 state 算出来过滤后的列表、总价、是否全选不存!渲染时直接算(派生)
省事技巧:能派生就别存。 如果一个值可以由现有 state/props 计算得到(比如"过滤后的列表""购物车总价"),就在渲染时算,不要单独 useState 存一份。多存一份就多一处要手动同步,极易出现"两份数据对不上"的 bug。这和"状态放哪儿"是同一个思路:让状态尽可能少、尽可能局部。

小结

  • 决定状态位置的唯一问题:有多少组件需要它。
  • 路径是渐进的:本地 useState → 状态提升到公共父 → 出现严重 props drilling 才上 Context/状态库。
  • 就近原则(colocation):状态放在离使用它最近的地方,作用域越小越好懂、好调、好维护。
  • 过早全局化的代价:脏数据残留、改动范围失控、多实例互相干扰、状态与组件分离。
  • 表单输入、开关这类私有 UI 状态,几乎永远是本地的;能派生的值别单独存。
  • 下一篇会揭示一个更关键的区分:全局状态里还分服务端状态和客户端状态,它们该用完全不同的工具。

常见陷阱

  • 状态提升过度,提到比实际需要更高的层级:两个兄弟组件共享状态时,有人图省事直接把状态提到根组件甚至塞进全局 Context,而不是提到它们真正的公共父组件。这会导致本该局部的重渲染范围扩大到整棵树,而且离使用的地方越远,理解"这个值为什么会变"就越难。应该提到能覆盖所有使用者的最近公共父组件,而不是随便找个更高的祖先。
  • 把所有 state 都塞进 Context,导致无关组件被拖累重渲染:文中提到全应用共享的 UI 状态才该上 Context,但新手常见的做法是"怕以后要共享,先放 Context 保险"。Context 的 value 只要引用变化,所有订阅它的组件都会重渲染,不管它们实际用到的是哪个字段——把开关状态、表单临时值这类局部状态也塞进同一个 Context,会让整棵子树无差别地跟着抖动。
  • 该派生的值单独存成 state,导致两份数据脱节:比如已经有原始列表和筛选条件,还单独 useState 存一份"过滤后的列表",每次筛选条件变化都要记得手动同步这份派生状态。一旦某个更新路径忘了同步,就会出现"界面显示的过滤结果和真实筛选条件不一致"的 bug。能在渲染时算出来的值,不应该单独存。
  • 表单输入这类典型本地状态被过早全局化:理由通常是"以后可能要在别处用到这个值",结果是全局仓库里塞满了只有一个输入框关心的临时值,每敲一个字符都触发一次全局更新,组件卸载后这些值还残留在仓库里变成脏数据。表单输入几乎永远该是本地 useState。

面试高频考点

  1. 状态提升(lifting state up)的原则是什么,提升到什么位置才对?—— 当多个组件需要共享或同步某个状态时,把状态从各自的组件里删除,统一放到它们的最近公共父组件,再通过 props 把值和更新函数往下传。原则是"提升到刚好能覆盖所有使用者的那一层",不要图省事提得比实际需要更高,否则会扩大重渲染范围、增加理解成本。
  2. 什么是 props drilling,它和"该不该上 Context"是什么关系?—— props drilling 指为了把一个值传到很深的子组件,中间一层层组件被迫转发它们自己根本不使用的 props。这是判断"该不该引入 Context 或状态库"的真实信号:只有当状态提升导致了严重的逐层转发,才值得引入全局方案；仅仅两三层的传递不算 props drilling,不必急着上 Context。
  3. 为什么"能派生的值不要存成 state"?举例说明。—— 因为多存一份数据就多一处需要手动同步的地方,一旦某次更新忘记同步派生值,就会出现两份数据不一致的 bug。比如"过滤后的列表"完全可以由"原始列表 + 筛选条件"在渲染时计算得出,没必要单独用 useState 存一份,直接在渲染函数体里算出来即可,天然保持同步。
  4. 为什么不应该把所有共享状态都无差别地放进同一个全局 Context?—— 因为 Context 的更新粒度是"整个 value 对象",只要 Provider 传下去的 value 引用发生变化,所有订阅了这个 Context 的组件都会重渲染,即使它们只用到了 value 里的某一个字段。如果把频繁变化的状态和很少变化的状态混在同一个 Context 里,会导致大量无关组件被无谓地拖累重渲染,这也是后续要学的"拆分 Context / 换用支持选择器的状态库"要解决的问题。

与其他知识点的联系

  • useLoaderData 与 useActionData(第 92 篇):数据路由把"页面级数据"整体搬到了路由层,这其实是"状态放在离使用它最近的地方"这条原则的一个特例延伸——只不过这里"最近的地方"从组件变成了路由,因为数据的归属天然和 URL 绑定,不是某一个组件私有的。
  • 服务端状态与客户端状态(第 94 篇):本篇讲的是"状态该放多局部",下一篇讲的是"全局状态里还要按数据来源再分一次类"——两者是递进关系:先判断要不要全局化,全局化之后还要再判断这份数据是服务端来的还是纯客户端的,分别匹配 TanStack Query 或 Zustand。
  • Context 与 useReducer 轻量方案(第 95 篇):本篇末尾提到的"props drilling 严重时才上 Context"正是第 95 篇的起点——Context+useReducer 就是"提升到公共父都不够用了"之后的下一步方案,两篇讲的是同一条渐进路径上前后相邻的两级。
  • Zustand 创建 store(第 96 篇):Zustand 的 selector 机制(只订阅用到的字段)可以看作是"就近原则"在全局状态库层面的落地——即便状态已经全局化,组件依然只应该"消费"自己关心的那一小块,而不是订阅整个 store,这和"状态放在离使用它最近的地方"是同一种设计直觉。
  • 能派生就别存,与 Zustand 的 get():本篇强调"能从别的 state 算出来的值不要单独存",这条原则在 Zustand 里同样适用——第 97 篇的 get() 就是在 action 内部做只读计算,而不是把计算结果另存一份 state,道理完全一致。
