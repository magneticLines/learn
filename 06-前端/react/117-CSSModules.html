CSS Modules（.module.css）

两个组件都写了 .button,谁的样式生效?普通 CSS 是全局的,后加载的规则会覆盖先加载的,团队协作时类名冲突几乎不可避免。CSS Modules 让构建工具在打包时自动把类名改成独一无二的字符串,天然获得"模块作用域",这是从纯手写 CSS 迈向可维护样式方案的第一步。

问题:CSS 天生没有作用域

JS 有 import/模块作用域,变量名冲突很少见;但 CSS 选择器一旦写出来,就对整个页面生效。想象一下:

Card.css
.title {
  font-size: 20px;
  color: #333;
}
Sidebar.css
.title {
  font-size: 12px;
  color: #999;
}

两个文件都在全局作用域里定义了 .title,谁在 HTML 里最后被引入,谁的规则就会覆盖前者(CSS 层叠规则)。项目越大、组件越多,这种"不知道哪来的样式突然生效"的问题就越常见——这也是为什么早期前端团队要发明 BEM(.card__title--large)这类命名规范来"人肉避免冲突"。

BEM 的局限: BEM 能缓解冲突,但要求团队严格自律地手写长命名,写错、漏写没有任何工具能帮你检查,本质上是靠约定而非机制解决问题。CSS Modules 把这件事交给构建工具,从"靠自觉"变成"靠不可能冲突"。

解决方案:构建时把类名哈希化

CSS Modules 的核心思路很简单:文件命名为 xxx.module.css,构建工具(webpack、Vite 都内置支持)看到这个后缀,就不再把里面的类名当全局类处理,而是在打包阶段把每个类名替换成一个包含文件路径哈希的唯一字符串,例如 .button 会变成 .Button_button__a1b2c。

Button.module.css
.button {...}
→
构建工具编译
→
.Button_button__a1b2c {...}

因为最终生成的类名带着文件级哈希,不同文件里即使都写 .button,编译后也是两个完全不同的字符串,物理上不可能冲突。这就是"模块作用域"——作用域被限定在了这一个 CSS 文件对应的模块里。

用法:import 后当对象取值

在 JS/JSX 里,.module.css 文件被当成一个模块 import 进来,得到的是一个"类名映射对象"——键是你写的原始类名,值是编译后的哈希类名。

Button.module.css
.button {
  padding: 8px 16px;
  border-radius: 6px;
}
.primary {
  background: #61dafb;
}
Button.jsx
import styles from './Button.module.css';

function Button({ primary, children }) {
  // styles.button === 'Button_button__a1b2c' (编译后的真实类名)
  const cls = primary
    ? `${styles.button} ${styles.primary}`
    : styles.button;

  return <button className={cls}>{children}</button>
}
陷阱:类名里有中划线怎么取? 如果你在 CSS 里写了 .btn-primary,在 JS 对象里不能用 styles.btn-primary(会被解析成"减法运算"),必须用方括号语法:styles['btn-primary']。这也是为什么社区更推荐 CSS Modules 里的类名用驼峰命名(.btnPrimary),可以直接 styles.btnPrimary,少一层心智负担。

composes:组合复用类名

普通 CSS 想复用一组样式,常见做法是让一个元素同时挂两个类名(class="button primary")。CSS Modules 提供了 composes,让一个类在 CSS 文件内部"继承"另一个类的所有声明,JS 里只需要引用一个类名:

composes:在 CSS 层面组合,而不是在 JSX 里拼字符串
.base {
  padding: 8px 16px;
  border-radius: 6px;
}
.primary {
  composes: base;  /* 先拿到 base 的所有样式 */
  background: #61dafb;
}
/* JSX 里只需要 className={styles.primary},编译后自动带上 base 的类 */
composes 还能跨文件组合:composes: base from './base.module.css';,把公共样式抽到一个文件里被多个组件复用,效果上有点像 CSS 版的"继承",但注意它只能用于纯类选择器,不支持组合到嵌套选择器或伪类上。

逃生舱::global()

有时候确实需要写一个全局样式(比如覆盖第三方组件库的类名,或是设置 body 的全局样式),CSS Modules 提供 :global() 包裹,告诉构建工具"这部分不要哈希化":

:global() 跳出模块作用域
:global(.ant-btn) {
  border-radius: 8px; /* 覆盖第三方库的全局类,原样保留类名 */
}

与普通 CSS 对比

普通 CSSCSS Modules
作用域全局,容易冲突文件级,编译后类名唯一
命名负担要靠 BEM 等约定人肉避免冲突随便起名,构建工具保证唯一
动态样式切换 class 或写内联 style同左,本质仍是普通 CSS
调试DevTools 里看到的就是你写的类名看到的是哈希后的类名,需要 source map 或 devtool 插件辅助
学习成本无额外概念需理解 import 取值、composes、:global()
放在样式方案的坐标系里: CSS Modules 解决的只是"作用域"这一个问题——它没有改变你还是在写 CSS,选择器、层叠、盒模型这些老朋友都还在,只是类名不会撞车了。下一篇会看 Tailwind 这种完全不同的思路:不写自定义类,直接在 HTML 里拼工具类。

小结

  • 普通 CSS 类名是全局的,大项目里靠 BEM 等命名约定人肉避免冲突,容易出错。
  • CSS Modules 让 .module.css 文件里的类名在构建时被哈希化,天然获得文件级作用域,不可能冲突。
  • 用法是 import styles from './x.module.css',像访问对象属性一样取真实类名:styles.button;中划线类名要用 styles['btn-primary']。
  • composes 可以在 CSS 层面组合复用类,:global() 是跳出作用域写全局样式的逃生舱。
  • 它本质仍是"写 CSS",只是加了一层编译期的类名唯一化处理,学习成本低,是从零散 CSS 迁移到工程化样式方案的常见第一步。
