Strict Mode 双重调用行为

"我明明只点了一次按钮,为什么控制台打印了两条日志?React 是不是有 bug?"——这是几乎每个初学者都问过的问题。答案是:这是故意的。开发模式下 <StrictMode> 会把组件函数体、部分 Hook、以及 useEffect 故意调用两次,目的是提前替你暴露不纯的渲染逻辑和没清理的副作用。

为什么要故意调用两次

React 的核心假设是:组件函数必须是纯函数——同样的 props/state 输入,渲染函数应该产出同样的结果,不能在渲染过程中"顺手"做一些有副作用的事(比如修改外部变量、发请求、写 DOM)。这个假设是 React 能实现并发渲染(可以暂停、丢弃、重新开始一次渲染)的基础:如果渲染函数不纯,被打断重来时就会出问题。

但"你的组件是不是纯的"光靠肉眼审查很难发现,React 团队的解法很直接:故意重复调用一次,如果两次结果不一致或者产生了看得见的副作用,你就会立刻发现问题——而不是等到上线后并发特性触发才踩坑。

开发模式挂载组件
→
渲染函数执行第一次
→
立即再执行第二次(结果丢弃,只留第二次)
→
正常显示 UI

被双重调用的三类东西

被双调用的内容说明
组件函数体整个函数重新执行一遍,包括函数体里的 console.log、临时变量计算等
部分 Hook(如 useState 的初始化函数、useReducer)传给它们的初始化函数会被调用两次,用来发现初始化逻辑里的副作用
useEffect(挂载 + 清理)挂载时:执行 → 清理 → 再执行一次;模拟"组件卸载又重新挂载"的场景
StrictMode 下 useEffect 的真实执行顺序
useEffect(() => {
  console.log('订阅了 socket');
  const socket = connect();
  return () => {
    console.log('清理了 socket');
    socket.disconnect();
  };
}, []);

// 开发模式 + StrictMode 下,挂载时控制台依次打印:
// "订阅了 socket"  → 第一次执行 effect
// "清理了 socket"  → 立即调用清理函数
// "订阅了 socket"  → 再执行一次 effect(这次真正留下)
这是在帮你提前测试"卸载重挂载"的场景。 React 未来的并发特性(比如离屏渲染、Activity 之类的能力)会出现"组件被暂时卸载、之后又恢复挂载,内部状态还要保留"的情况。如果你的 useEffect 写了清理函数却漏了什么(比如订阅了却没退订、定时器没清除),在 StrictMode 下"挂载→卸载→再挂载"的过程会立刻让 bug 现形,而不是留到生产环境某个复杂场景才炸。
常见误解:"React 是不是有 bug,渲染了两次?" 不是 bug,而是故意的开发环境行为,并且只发生在开发模式 + 包裹了 <StrictMode> 的场景下——生产构建(npm run build 之后的产物)不会双重调用,用户实际看到的行为是正常的单次渲染。判断"我的 effect 到底触发了几次"永远要以生产环境为准,不要在开发环境数日志条数来下结论。
典型的踩坑写法:计数器在 StrictMode 下"跳两下"
let idCounter = 0;  // ✗ 渲染函数外的可变变量,在渲染过程中被修改——不纯

function Item() {
  idCounter++;          // StrictMode 下这行执行两次,idCounter 会多加 1
  return <div>{idCounter}</div>;
}
// 正确做法:用 useRef 或 useId 生成稳定 id,不要在渲染函数体里直接修改外部变量
和前两篇的联系: Profiler 里看到的"渲染两次"如果是开发环境下且用了 StrictMode,属于正常现象,不要误判为性能问题去优化它;真正该关注的是生产环境下、由三种真实重渲染原因(自身 state、父组件、Context)触发的多余渲染。

小结

  • StrictMode 只在开发模式下故意把组件函数体、部分 Hook 初始化、useEffect 调用两次,目的是提前暴露不纯的渲染逻辑和没清理干净的副作用。
  • 这是为将来的并发特性(暂停/丢弃渲染、卸载重挂载保留状态)做准备——不纯的组件在这些特性下会出问题。
  • 生产构建不会双重调用,用户看到的是正常的单次渲染,不要用开发环境的日志条数判断生产行为。
  • 常见踩坑:在渲染函数体里直接修改外部可变变量、副作用没写清理函数——这些在 StrictMode 下会很快暴露。
  • 理解重渲染的三篇到此结束;接下来进入"代码分割"模块,从 React.lazy + Suspense 开始。

常见陷阱

  • 清理函数缺失导致"重复订阅"——StrictMode 下 useEffect 挂载时会走"执行→清理→再执行"三步。如果你只写了 effect 的正向逻辑(比如 socket.connect()、addEventListener),却没写对应的清理函数去 disconnect 或 removeEventListener,第一次的连接/监听就会永久留存,第二次又建一份——最终页面上出现两个 socket 连接、事件回调被触发两次。原理是清理函数就是 React 用来"撤销上一次副作用"的钩子,没有它,双重调用就只剩"重复执行"没有"先清空",数据自然错乱。
  • 把双调用当成 bug 去"修复"——看到控制台打印两条日志,第一反应是去 GitHub 提 issue 或者到处加 if 判断"防止重复执行"。这类"修复"往往是在渲染函数或 effect 里加全局标志位拦截第二次调用,结果反而掩盖了真正的不纯逻辑,以后并发渲染场景下照样会炸。正确反应是:先确认是不是开发模式 + StrictMode,如果是,说明是故意的健康检查,该做的是修好 effect 的清理逻辑,而不是消灭双调用现象本身。
  • 拿开发环境的调用次数去衡量生产性能——因为只有开发模式 + <StrictMode> 才会双调用,生产构建永远是单次执行,如果照着开发环境日志数量去做"性能优化"(比如为了"减少一次渲染"给组件加莫名其妙的缓存),很可能是在优化一个生产环境根本不存在的问题,白费功夫还可能引入新 bug。判断性能要么关掉 StrictMode 在开发环境测,要么直接以 npm run build 后的产物为准。
  • 渲染函数体里直接改外部可变变量——像 idCounter++ 这种写法,在 StrictMode 下会因为函数体被执行两次而"多加一次",看起来像计数器跳着走。根源是渲染函数被要求是纯函数,不能在渲染过程中修改渲染函数外部的状态;双调用只是把这个"不纯"的后果提前放大到开发阶段让你看见,而不是它本身引入了新问题。正确做法是用 useRef/useId 或者把状态挪到 useState 里,避免在渲染期间产生看得见的副作用。
  • 组件初始化里做请求或订阅当成"只跑一次"——把发请求、建立订阅之类的操作直接写在组件函数体或 useState 的初始化函数里,以为它只会跑一次,结果 StrictMode 下这些初始化函数也会被调用两次,导致重复发起请求。这类逻辑本应该放进 useEffect 并配好清理函数,而不是塞进渲染阶段——渲染阶段只应该是"计算这一次要显示什么",副作用永远要交给 effect 管理生命周期。

面试高频考点

  1. StrictMode 为什么要把组件函数体和 effect 调用两次?—— 为了在开发阶段提前暴露不纯的渲染逻辑和没清理干净的副作用。React 的并发渲染要求组件函数是纯函数(同样输入产出同样输出、渲染过程中不能有可见副作用),因为渲染可能被打断、丢弃、重新开始。双调用是低成本地模拟这种"重来一次"的场景,让开发者在写代码时就发现问题,而不是等并发特性真正上线后才踩坑。
  2. StrictMode 下 useEffect 挂载时的执行顺序是什么?—— 先执行一次 effect,立即调用其返回的清理函数,再执行一次 effect(这次才是真正留下的)。也就是"执行→清理→再执行",本质是模拟"组件卸载后又重新挂载"的场景,用来检验清理函数是否把副作用完全撤销干净。
  3. 生产环境下组件还会被双重调用吗?—— 不会。双重调用只发生在开发模式且组件被 <StrictMode> 包裹的情况下,生产构建(npm run build 之后)不会有这个行为,用户实际看到的是正常的单次渲染。因此不能拿开发环境的日志条数或调用次数去判断生产环境的性能或行为。
  4. 如果发现组件在 StrictMode 下表现异常(比如状态多算了一次),该怎么排查?—— 先检查是否在渲染函数体内直接修改了外部可变变量,或者在 useState/useReducer 的初始化函数、组件函数体里做了本该放进 useEffect 的副作用(发请求、订阅等)。再检查已有的 useEffect 是否缺少清理函数,导致挂载→卸载→再挂载的过程中副作用被重复叠加而没有撤销。定位到具体是哪一类"不纯"操作,而不是去屏蔽双调用本身。
