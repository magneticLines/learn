Strict Mode 双重调用行为

"我明明只点了一次按钮,为什么控制台打印了两条日志?React 是不是有 bug?"——这是几乎每个初学者都问过的问题。答案是:这是故意的。开发模式下 <StrictMode> 会把组件函数体、部分 Hook、以及 useEffect 故意调用两次,目的是提前替你暴露不纯的渲染逻辑和没清理的副作用。

为什么要故意调用两次

React 的核心假设是:组件函数必须是纯函数——同样的 props/state 输入,渲染函数应该产出同样的结果,不能在渲染过程中"顺手"做一些有副作用的事(比如修改外部变量、发请求、写 DOM)。这个假设是 React 能实现并发渲染(可以暂停、丢弃、重新开始一次渲染)的基础:如果渲染函数不纯,被打断重来时就会出问题。

但"你的组件是不是纯的"光靠肉眼审查很难发现,React 团队的解法很直接:故意重复调用一次,如果两次结果不一致或者产生了看得见的副作用,你就会立刻发现问题——而不是等到上线后并发特性触发才踩坑。

开发模式挂载组件
→
渲染函数执行第一次
→
立即再执行第二次(结果丢弃,只留第二次)
→
正常显示 UI

被双重调用的三类东西

被双调用的内容说明
组件函数体整个函数重新执行一遍,包括函数体里的 console.log、临时变量计算等
部分 Hook(如 useState 的初始化函数、useReducer)传给它们的初始化函数会被调用两次,用来发现初始化逻辑里的副作用
useEffect(挂载 + 清理)挂载时:执行 → 清理 → 再执行一次;模拟"组件卸载又重新挂载"的场景
StrictMode 下 useEffect 的真实执行顺序
useEffect(() => {
  console.log('订阅了 socket');
  const socket = connect();
  return () => {
    console.log('清理了 socket');
    socket.disconnect();
  };
}, []);

// 开发模式 + StrictMode 下,挂载时控制台依次打印:
// "订阅了 socket"  → 第一次执行 effect
// "清理了 socket"  → 立即调用清理函数
// "订阅了 socket"  → 再执行一次 effect(这次真正留下)
这是在帮你提前测试"卸载重挂载"的场景。 React 未来的并发特性(比如离屏渲染、Activity 之类的能力)会出现"组件被暂时卸载、之后又恢复挂载,内部状态还要保留"的情况。如果你的 useEffect 写了清理函数却漏了什么(比如订阅了却没退订、定时器没清除),在 StrictMode 下"挂载→卸载→再挂载"的过程会立刻让 bug 现形,而不是留到生产环境某个复杂场景才炸。
常见误解:"React 是不是有 bug,渲染了两次?" 不是 bug,而是故意的开发环境行为,并且只发生在开发模式 + 包裹了 <StrictMode> 的场景下——生产构建(npm run build 之后的产物)不会双重调用,用户实际看到的行为是正常的单次渲染。判断"我的 effect 到底触发了几次"永远要以生产环境为准,不要在开发环境数日志条数来下结论。
典型的踩坑写法:计数器在 StrictMode 下"跳两下"
let idCounter = 0;  // ✗ 渲染函数外的可变变量,在渲染过程中被修改——不纯

function Item() {
  idCounter++;          // StrictMode 下这行执行两次,idCounter 会多加 1
  return <div>{idCounter}</div>;
}
// 正确做法:用 useRef 或 useId 生成稳定 id,不要在渲染函数体里直接修改外部变量
和前两篇的联系: Profiler 里看到的"渲染两次"如果是开发环境下且用了 StrictMode,属于正常现象,不要误判为性能问题去优化它;真正该关注的是生产环境下、由三种真实重渲染原因(自身 state、父组件、Context)触发的多余渲染。

小结

  • StrictMode 只在开发模式下故意把组件函数体、部分 Hook 初始化、useEffect 调用两次,目的是提前暴露不纯的渲染逻辑和没清理干净的副作用。
  • 这是为将来的并发特性(暂停/丢弃渲染、卸载重挂载保留状态)做准备——不纯的组件在这些特性下会出问题。
  • 生产构建不会双重调用,用户看到的是正常的单次渲染,不要用开发环境的日志条数判断生产行为。
  • 常见踩坑:在渲染函数体里直接修改外部可变变量、副作用没写清理函数——这些在 StrictMode 下会很快暴露。
