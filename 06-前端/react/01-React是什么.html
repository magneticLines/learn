React 是什么：UI = f(state)

学 React 最大的坑不是 API，而是没建立正确的心智模型。这一节聚焦一件事：React 为什么存在，以及它的核心设计哲学是什么。

它解决了什么问题：jQuery 时代的痛苦

在 React 出现之前（2013 年），前端主流是 jQuery。jQuery 的操作方式是命令式的：你告诉浏览器"找到这个按钮 → 监听点击 → 找到那个 DOM 节点 → 改它的文本 → 再找另一个节点 → 加个 class"。

这在页面逻辑简单时没有问题。但当业务复杂起来——登录状态影响导航栏、购物车数量影响角标、消息数量影响标题——你发现自己在多处手动同步同一份数据。改了数据，忘了更新某个 DOM，bug 就出现了，而且你根本不知道是哪个地方漏掉了。

Facebook Messenger 的真实案例：Messenger 的"未读消息数"角标总是显示错误——有时显示有新消息但打开后没有，有时消息来了角标没变。原因就是有多处 jQuery 代码在独立地操作同一个"未读数"DOM，它们之间的状态不一致。这个 bug 最终促使 Facebook 的工程师决心从根本上解决"UI 与数据同步"的问题，React 由此诞生。

问题的根源是：UI 的状态和数据的状态是两套独立的东西，靠程序员手工保持一致。只要程序员一疏忽，它们就会漂移。React 要做的是：彻底消灭"手动同步"这件事。

核心公式：UI = f(state)

React 的核心思想用一个公式概括：

UI = f( state )

UI 是状态的函数。你的组件就是一个函数：给它一个状态（state），它返回对应的界面描述（UI）。状态变了，React 重新调用这个函数，得到新的界面描述，再把差异更新到 DOM。

这意味着什么？你永远不需要主动去操作 DOM。你只需要管理好状态——数据是什么，UI 就长什么样，这个对应关系由你在组件里描述一次，React 来负责把它变成真实的 DOM 操作。

Excel 类比：你在 A1 单元格写了数据，在 B1 单元格写了公式 =A1*2。只要 A1 一变，B1 自动更新——你不需要去手动改 B1。React 和你的关系就是这样：你写好"UI 和 state 的对应关系"（就像写公式），当 state 改变，React 自动帮你"重算" UI。你从来不用说"去找到那个格子，把它的值改成多少"。

React 只做 UI 层：它是库不是框架

这一点很多初学者会混淆，需要明确说清楚：

  • React 只负责 UI——把状态渲染成界面、处理用户交互事件。它不管网络请求、不管路由跳转、不管全局状态管理，这些全靠额外的库来补充（React Router 管路由、TanStack Query 管请求、Zustand 管全局状态）。
  • React 是库，不是框架——它不规定项目文件夹怎么组织、用什么打包工具、怎么发 HTTP 请求。自由度很高，但也意味着你要做更多选择。这就是为什么 Next.js 存在——它是在 React 上加了规范和约定的框架。
  • React 可以渐进接入——不需要整个项目都改成 React，可以在页面的某个 div 里挂载一个 React 应用。Facebook 当年就是这样在老页面里局部引入的。
常见误区：把 React 当 jQuery 用
表现：在 useEffect 里写 document.getElementById('xxx').innerText = '新内容'，或者通过 ref 直接读写 DOM 来"更新界面"。

这完全抛弃了 React 的设计哲学——你相当于在 Excel 里不写公式，反而在 B1 发生变化后手动去改 B1 的值。

正确做法：把需要变化的东西放进 state，让 React 来驱动 DOM。你只管状态，React 管 DOM。

为什么这个设计是对的

你可能会问：每次 state 变化都重新"描述"一遍 UI，效率不低吗？这正是 React 设计精妙之处——它引入了虚拟 DOM和Reconciliation（协调）机制，在内存中做 diff，只把真正变化的部分更新到真实 DOM。这是下一节的主题。

在这一节，你只需要记住核心：UI 是 state 的因变量。你不"更新 UI"，你"更新 state"，UI 随之自动变化。

旧思维（jQuery）新思维（React）
找到 DOM 节点，手动改它改变 state，React 自动更新 DOM
UI 和数据是两套东西，手动同步UI 是数据的函数，天然同步
关注"怎么做"（How）关注"是什么"（What）
操作步骤分散在各处，难以追踪状态在一处管理，变化路径清晰
