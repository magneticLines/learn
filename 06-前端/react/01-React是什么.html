React 是什么：UI = f(state)

学 React 最大的坑不是 API，而是没建立正确的心智模型。这一节聚焦一件事：React 为什么存在，以及它的核心设计哲学是什么。

它解决了什么问题：jQuery 时代的痛苦

在 React 出现之前（2013 年），前端主流是 jQuery。jQuery 的操作方式是命令式的：你告诉浏览器"找到这个按钮 → 监听点击 → 找到那个 DOM 节点 → 改它的文本 → 再找另一个节点 → 加个 class"。

这在页面逻辑简单时没有问题。但当业务复杂起来——登录状态影响导航栏、购物车数量影响角标、消息数量影响标题——你发现自己在多处手动同步同一份数据。改了数据，忘了更新某个 DOM，bug 就出现了，而且你根本不知道是哪个地方漏掉了。

Facebook Messenger 的真实案例：Messenger 的"未读消息数"角标总是显示错误——有时显示有新消息但打开后没有，有时消息来了角标没变。原因就是有多处 jQuery 代码在独立地操作同一个"未读数"DOM，它们之间的状态不一致。这个 bug 最终促使 Facebook 的工程师决心从根本上解决"UI 与数据同步"的问题，React 由此诞生。

问题的根源是：UI 的状态和数据的状态是两套独立的东西，靠程序员手工保持一致。只要程序员一疏忽，它们就会漂移。React 要做的是：彻底消灭"手动同步"这件事。

核心公式：UI = f(state)

React 的核心思想用一个公式概括：

UI = f( state )

UI 是状态的函数。你的组件就是一个函数：给它一个状态（state），它返回对应的界面描述（UI）。状态变了，React 重新调用这个函数，得到新的界面描述，再把差异更新到 DOM。

这意味着什么？你永远不需要主动去操作 DOM。你只需要管理好状态——数据是什么，UI 就长什么样，这个对应关系由你在组件里描述一次，React 来负责把它变成真实的 DOM 操作。

Excel 类比：你在 A1 单元格写了数据，在 B1 单元格写了公式 =A1*2。只要 A1 一变，B1 自动更新——你不需要去手动改 B1。React 和你的关系就是这样：你写好"UI 和 state 的对应关系"（就像写公式），当 state 改变，React 自动帮你"重算" UI。你从来不用说"去找到那个格子，把它的值改成多少"。

React 只做 UI 层：它是库不是框架

这一点很多初学者会混淆，需要明确说清楚：

  • React 只负责 UI——把状态渲染成界面、处理用户交互事件。它不管网络请求、不管路由跳转、不管全局状态管理，这些全靠额外的库来补充（React Router 管路由、TanStack Query 管请求、Zustand 管全局状态）。
  • React 是库，不是框架——它不规定项目文件夹怎么组织、用什么打包工具、怎么发 HTTP 请求。自由度很高，但也意味着你要做更多选择。这就是为什么 Next.js 存在——它是在 React 上加了规范和约定的框架。
  • React 可以渐进接入——不需要整个项目都改成 React，可以在页面的某个 div 里挂载一个 React 应用。Facebook 当年就是这样在老页面里局部引入的。
常见误区：把 React 当 jQuery 用
表现：在 useEffect 里写 document.getElementById('xxx').innerText = '新内容'，或者通过 ref 直接读写 DOM 来"更新界面"。

这完全抛弃了 React 的设计哲学——你相当于在 Excel 里不写公式，反而在 B1 发生变化后手动去改 B1 的值。

正确做法：把需要变化的东西放进 state，让 React 来驱动 DOM。你只管状态，React 管 DOM。

为什么这个设计是对的

你可能会问：每次 state 变化都重新"描述"一遍 UI，效率不低吗？这正是 React 设计精妙之处——它引入了虚拟 DOM和Reconciliation（协调）机制，在内存中做 diff，只把真正变化的部分更新到真实 DOM。这是下一节的主题。

在这一节，你只需要记住核心：UI 是 state 的因变量。你不"更新 UI"，你"更新 state"，UI 随之自动变化。

旧思维（jQuery）新思维（React）
找到 DOM 节点，手动改它改变 state，React 自动更新 DOM
UI 和数据是两套东西，手动同步UI 是数据的函数，天然同步
关注"怎么做"（How）关注"是什么"（What）
操作步骤分散在各处，难以追踪状态在一处管理，变化路径清晰

常见陷阱

  • 把"响应式"和"自动同步"混为一谈：初学者常以为 React 会"监听"所有变量的变化，只要变量变了 UI 就会更新。实际上 React 只认state（通过 useState/useReducer 声明的变量）。你直接改一个普通变量 let count = 1; count = 2，UI 完全不会动——因为 React 根本不知道这个变量存在，它没有触发重新渲染的机制。
  • 误以为"UI = f(state)"意味着组件函数只在 state 变化时执行一次：实际上父组件重渲染会导致子组件函数也重新执行（除非用 memo 优化），"函数"这个类比容易让人忽略"谁触发了调用"这一层。
  • 该用 state 的地方用了 ref 或普通变量：表现是：改了值，但界面死活不更新。根源往往是把需要驱动 UI 的数据存成了 ref.current 或模块级变量——这类值的变化不会触发 React 重新调用组件函数，因为它们不在"state"这个能触发渲染的通道里。
  • 觉得 React 会自动管理网络请求、路由、全局状态：React 只解决 UI = f(state) 这一层，其余全靠生态库补充。初学者常常以为装了 React 就自带这些能力，结果发现连发个请求都要自己配 fetch 或引入额外库。

面试高频考点

  1. React 是框架还是库？为什么？—— 是库。它只负责"状态到 UI"的渲染这一层，不规定路由、请求、项目结构等，这些都靠额外的库（React Router、TanStack Query 等）组合而成；框架（如 Next.js、Angular）通常会替你做好这些约定和整合。
  2. 为什么 React 不推荐直接操作 DOM？—— 因为 React 的心智模型是"状态驱动视图"，直接操作 DOM 会让真实 DOM 的状态和 React 内部维护的虚拟 DOM 树产生不一致，下一次 React 重新渲染时可能会覆盖你手动做的改动，导致行为难以预测。正确方式是把要变化的内容放进 state，交给 React 去决定如何更新 DOM。
  3. UI = f(state) 这个公式具体怎么理解？—— 组件是一个纯函数（理想情况下），输入是 state/props，输出是描述 UI 结构的虚拟 DOM。state 变化时 React 重新调用这个函数得到新的输出，而不是让你手工修改上一次的输出，这样保证了"同样的状态永远产生同样的 UI"。

与其他知识点的联系

  • 关联虚拟 DOM 与 Reconciliation（下一节）："UI = f(state)"这个公式说的是设计哲学，但没解释"每次都重新描述 UI 会不会很慢"——这正是下一节要解决的问题：React 用虚拟 DOM 在内存里做 diff，把"声明式重新描述"和"高效更新真实 DOM"这对矛盾调和起来。没有虚拟 DOM，UI=f(state) 只是一个理论上优雅但实践上会卡死的想法。
  • 关联 JS 的函数与闭包：把组件理解成"输入 state/props、输出 UI 描述的函数"，直接借用了 JS 里"函数是一等公民"和"纯函数"的概念——组件内部用到的变量、事件处理函数所形成的闭包，也是后面 useState/useEffect 等 Hook 能正常工作的基础（Hook 本质上依赖闭包记住每次渲染时的状态）。
  • 关联事件循环与异步：state 更新不是立刻同步生效的——多次 setState 调用会被 React 批处理（batching），这和 JS 事件循环里"微任务/宏任务分批执行"的思路相通：不是"改一次就立刻跑一次渲染"，而是"攒一批变化，找合适的时机统一处理"。
  • 关联防抖与节流：如果不理解"只有 state 变化才会触发重渲染"，很容易在高频事件（如 mousemove、scroll）里直接无节制地 setState，造成大量无意义的重渲染——这时候需要用防抖节流（JS 篇第 40 节）控制 setState 的调用频率，本质上和优化原生 DOM 操作频率是同一个问题。
  • 关联 ref 与命令式 API：本节反复强调"别直接操作 DOM，该用 state"，但 React 也保留了 ref 这个"逃生舱口"用于少数确实需要命令式操作的场景（如聚焦输入框、读取元素尺寸）——理解"什么时候该用 state、什么时候该用 ref"，是判断一段 React 代码是否"地道"的关键分界线。
