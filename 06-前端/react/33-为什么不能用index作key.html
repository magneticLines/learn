为什么不能用数组 index 作 key

{list.map((item, index) => <li key={index}>...)} 是新手最常见的偷懒写法。列表不变时它没问题，可一旦发生增删或排序，就会出现输入框内容串位、勾选状态错乱这类诡异 bug。这一篇用"反例 → 问题 → 正例"讲透它。

问题根源：index 描述的是"位置"，不是"身份"

回顾上一篇：key 应该是元素的身份证——跟着数据本身走。而 index 是元素在数组里的排位——它跟着"位置"走。当列表顺序变化时，同一条数据的 index 会变，同一个 index 也会指向不同的数据。用 index 作 key，等于告诉 React"第几位就是谁"，这恰恰是 key 要避免的按位置认人。

数据 A B C
index 0 1 2
→
删除 A
→
数据 B C
index 0 1
→
B 的 key 从 1 变成 0
React 认错人

反例：在列表头部删除一项

假设一个待办列表，每项旁边有个非受控的输入框（DOM 自己维护输入内容）。用 index 作 key，删掉第一项会发生什么：

❌ 反例：用 index 作 key
{todos.map((todo, index) => (
  <li key={index}>            // ← 用位置当身份，隐患所在
    <span>{todo.text}</span>
    <input defaultValue="" />  // 用户在这里输入的内容由 DOM 维护
  </li>
))}

问题：状态"留在了位置上"，跟着错位

列表 [A, B, C]，用户在 A、B、C 的输入框分别打了 "a"、"b"、"c"。现在删除 A，数据变成 [B, C]：

删除前index/key输入框内容
A0a
B1b
C2c
删除后index/keyReact 怎么想结果
B0key=0 还在，复用原 0 号 DOMB 显示，输入框却是 "a" ❌
C1key=1 还在，复用原 1 号 DOMC 显示，输入框却是 "b" ❌
—2key=2 消失，删除 2 号 DOM带 "c" 的输入框被删 ❌
为什么错得这么离谱？React 按 key 配对：删除后仍有 key=0、key=1，React 认为"0 号和 1 号元素还在，只是内容变了"，于是复用了它们的 DOM 节点（包括输入框里的内容），只更新了文本 span。结果文本从 A→B、B→C 变了，但输入框还是原位置的旧内容。用户看到的是：删了第一行，剩下行的文字对了，但每个输入框里的内容都串到了上一位。同样的错位也会发生在受控组件的 state、组件内部 state、CSS 动画上。

正例：用数据自己的 id 作 key

✅ 正例：key 跟着数据身份走
{todos.map(todo => (
  <li key={todo.id}>          // ← 身份跟着数据，删 A 后 B/C 的 key 不变
    <span>{todo.text}</span>
    <input defaultValue="" />
  </li>
))}

删除 A 后，B 的 key 仍是 b、C 的 key 仍是 c。React 发现 key=a 消失，只删除 A 对应的那个 DOM 节点，B、C 的节点（连同输入框内容）原封不动地保留。这才是符合直觉的行为。

数据没有 id 怎么办？优先在数据源头补上稳定 id（后端返回主键，或前端 crypto.randomUUID() 在创建时生成一次并存进数据里，而不是在 render 里现生成）。若字段组合能唯一标识，也可用 key={`${item.name}-${item.date}`}。关键是 key 必须绑定到"数据身份"上，且创建后不再变。

那么 index 什么时候能安全地当 key？

index 并非绝对禁用。同时满足以下三个条件时，用 index 作 key 是安全的：

  • 列表是静态的——渲染后不增、不删、不排序（比如一次性渲染的导航菜单、页脚链接）。
  • 列表项没有内部状态——没有输入框、勾选、展开折叠等由元素自身维护的状态。
  • 数据里确实没有可用的唯一 id——纯展示型的字符串数组等。

只要将来可能"从中间增删"或"排序"，或者列表项带状态，就必须用稳定 id。判断不准时，默认用 id 更安全——用 index 的收益只是省一点点事，代价却可能是难排查的错位 bug。

面试高频结论："能不能用 index 作 key？"标准答案是——看列表是否会变动。静态、无状态、无唯一 id 时可以；只要会增删/排序或列表项含状态，用 index 会导致 React 按位置错误复用 DOM，引发状态错位。根因是 index 表达"位置"而非"身份"，与 key 的设计目的相悖。
小结
  • index 描述位置不是身份；增删/排序后 index 与数据的对应关系错乱，React 按 key 复用 DOM 就会认错人。
  • 典型症状：删除/插入/排序后，输入框内容、勾选状态、组件 state 串位到相邻项。
  • 正解：用数据自身稳定唯一的 id；仅当列表静态、无状态、无 id 时 index 才安全。

与其他知识点的联系

  • key 的作用与规范（上一节）：本节是上一节"key 必须稳定"这条规范的具体反例展开——index 看起来满足"唯一"，却不满足"稳定"，唯一性不能替代稳定性，这正是初学者最容易混淆的两个要求。
  • 不可变更新：示例里"删除 A 后 B/C 的 key 不变"这个正确行为，前提是删除操作用的是 filter 返回新数组而不是原地 splice——不可变更新保证了新数组里 B、C 对应的仍是原来那个数据对象（只是位置变了），key 才能稳定地跟着 id 走。
  • 受控组件与非受控组件：本节的反例特意用了非受控输入框（defaultValue，由 DOM 自己维护内容）来演示 index 作 key 的错位问题——如果换成受控组件（value 由 state 管理），错位现象会以另一种形式出现：state 数组里的值和 UI 显示对不上，同样是 key 配对错误导致的，只是表现形式不同。
  • 组件 state 与卸载重建：index 作 key 导致的"复用了不该复用的 DOM"，本质上是 React 的 Diff 算法认为这两个组件是"同一个"从而保留了其内部 state（包括非受控 DOM 状态）——理解 key 变化会触发组件卸载重建、key 不变则会强制复用，才能理解为什么这里的错位不是渲染 bug，而是"认错身份"导致的状态保留错误。
  • useState 的挂载位置：React 内部把每个组件实例的 state 与 Fiber 树中的位置（受 key 影响的身份）绑定，而不是与"数据内容"绑定——这也解释了为什么哪怕 span 里显示的文字换成了正确的新数据，输入框内部状态却还停留在旧数据上：文字是每次渲染重新读 props 算出来的，而输入框的状态是绑定在"这个 DOM/组件实例"上、跟着错误复用的身份走的。
