错误边界：让一个组件崩溃不拖垮整个应用

React 16 起，只要一个组件在渲染时抛出未捕获的错误，整棵组件树都会被卸载，用户看到白屏。错误边界（Error Boundary）就是用来"围住"这类错误、只让局部降级、其余部分照常运行的机制。

问题：默认行为是"整树崩溃"

这是 React 16 的一个有意为之的设计决定：如果一个渲染错误没有被任何边界捕获，React 会把整个根组件树卸载掉。原因是——一个损坏的 UI 往往比没有 UI 更危险（比如支付金额显示错乱），所以 React 宁可白屏，也不愿意保留一个可能错误的界面。

但整个应用白屏体验太糟。错误边界让你把"崩溃"的影响范围限制在一个子树里：某个图表组件挂了，只把图表区域换成"加载失败"，页面其余部分不受影响。

为什么错误边界必须是 class 组件

这是初学者最容易困惑的一点：都 2020 年代了，Hooks 一统天下，为什么错误边界还得写成 class？因为错误边界依赖两个生命周期方法，而 React 至今没有为它们提供 Hooks 等价物：

静态方法 · 渲染 fallback
static getDerivedStateFromError(error)
实例方法 · 上报副作用
componentDidCatch(error, info)
一个标准的错误边界组件
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  // ① 捕获错误、更新 state → 决定"渲染成什么"
  //    必须是纯函数、必须 static：React 在"渲染阶段"调用它，
  //    此阶段不允许有副作用（不能上报日志、不能 setState 外的操作）
  static getDerivedStateFromError(error) {
    return { hasError: true };  // 返回新 state，触发 fallback 渲染
  }

  // ② 处理副作用 → 在"提交阶段"调用，可以安全地上报日志
  componentDidCatch(error, info) {
    // info.componentStack 是出错组件的调用栈，排查利器
    reportToSentry(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return <h2>这部分内容加载失败了</h2>;
    }
    return this.props.children;  // 没出错就正常渲染子树
  }
}
两个方法的分工，对应 React 的两个阶段。React 一次更新分为"渲染阶段"（计算要渲染什么，可被打断、可能重跑，因此必须纯净）和"提交阶段"（把结果写入 DOM，此时才允许副作用）。getDerivedStateFromError 属于渲染阶段所以是纯的静态方法；componentDidCatch 属于提交阶段所以能安全上报日志。理解了这个分工，就不会把上报日志写错地方。

怎么用：把它当成一个包裹组件

按业务粒度放置边界，缩小崩溃影响面
function App() {
  return (
    <Layout>
      {/* 侧边栏挂了不影响主内容 */}
      <ErrorBoundary><Sidebar /></ErrorBoundary>

      {/* 图表区独立兜底，崩了只降级这一块 */}
      <ErrorBoundary><Dashboard /></ErrorBoundary>
    </Layout>
  );
}

边界是就近捕获的：错误会冒泡到组件树上离它最近的那个错误边界。放得越靠近叶子，崩溃影响面越小，但样板代码越多；放得越靠近根，兜底越省事，但一崩就崩一大片。常见做法是路由级 + 关键 widget 级两层兜底。

陷阱：错误边界捕获不了什么

这是面试与实战里最关键的知识点。错误边界只捕获子组件在渲染期间抛出的错误，以下四类它一律捕获不到：

捕获不到的场景原因 & 正确做法
事件处理函数里的错误
onClick={() => { throw ... }}		 事件回调不在渲染流程中，React 不接管。用普通 try/catch 自己处理。
异步代码
setTimeout / fetch().then		 回调执行时早已脱离渲染栈。用 .catch() 或 try/catch（配合 async/await）。
服务端渲染 (SSR) 错误边界是客户端渲染机制，SSR 阶段不生效。
边界组件自己抛的错 边界不能捕获自身错误，只能捕获子树。需要上层再包一个边界。
最常见的错误认知："我用了错误边界，按钮点击报错应该被兜住"。不会。因为它是onClick里的错误，属于事件处理器。要想让边界捕获事件里的错误，一个技巧是：在 catch 里 setState 把错误存起来，然后在 render 中重新抛出——这样错误就回到了渲染流程，边界才能接住。
让事件里的错误"回流"到渲染阶段
const [error, setError] = useState(null);
if (error) throw error;  // 渲染期抛出 → 边界能捕获

const handleClick = async () => {
  try { await doSomething(); }
  catch (e) { setError(e); }  // 存进 state，下次渲染再抛
};

与其他知识点的联系

  • 组合 vs 继承(上一篇)：错误边界必须写成 class 组件、extends React.Component,乍看像是打破了上一篇"用组合不用继承"的原则,但上一篇已经区分过——这是"继承框架 API"以获取 getDerivedStateFromError 等平台能力,不是"业务组件之间用继承复用",两者不矛盾,反而是对那条区分的实际验证。
  • 渲染阶段 vs 提交阶段(useEffect 的心智模型)：getDerivedStateFromError(纯函数、渲染阶段)与 componentDidCatch(允许副作用、提交阶段)的分工,和函数组件里"渲染函数体必须纯、副作用只能放进 useEffect"是同一套 React 更新模型的两种表现形式——只是错误边界用类组件的生命周期方法表达,函数组件用 Hooks 表达。
  • 受控组件的"事件里出错"陷阱：本篇"事件处理函数里的错误捕获不到"这条陷阱,和受控组件 onChange 里的逻辑本质上处于同一类代码位置(事件回调),提醒我们:表单校验逻辑如果写得不小心抛出异常,错误边界是兜不住的,必须自己 try/catch。
  • react-error-boundary 库(下一篇)：本篇手写的 class 版本暴露的三个痛点(样板重复、fallback 拿不到 error、没有重试)正是下一篇要解决的问题——不理解这里 getDerivedStateFromError/componentDidCatch 的原始分工,就看不出下一篇的库到底"封装"了什么。

小结

  • 错误边界把渲染错误的影响面从"整树崩溃"缩小到"局部降级"。
  • 必须写成 class 组件，因为依赖 getDerivedStateFromError（纯函数、渲染阶段、返回 fallback state）和 componentDidCatch（副作用、提交阶段、上报日志），Hooks 目前无等价物。
  • 只捕获子组件渲染期的错误；事件处理、异步、SSR、边界自身的错误一律捕获不到。
