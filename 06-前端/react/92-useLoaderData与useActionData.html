useLoaderData / useActionData:数据路由的整体心智

loader 取的数据、action 返回的结果,组件怎么拿?靠这三个 hook:useLoaderData、useActionData、useNavigation。这一篇把前面几节串成完整的数据路由心智模型。

useLoaderData:读取 loader 返回的数据

loader 在渲染前跑完、返回数据,组件里用 useLoaderData() 同步读到——没有 loading 判断,因为数据一定已经就绪(这正是渲染前取数的好处)。

useLoaderData:拿数据,天然无 loading
import { useLoaderData } from 'react-router-dom';

function UserPage() {
  // loader 已经跑完,这里直接同步拿到数据
  const user = useLoaderData();
  // 不需要 if (!user) return <Spinner/>,渲染时一定有数据
  return <h1>{user.name}</h1>;
}

useActionData:读取 action 返回的结果

action 除了写数据 + redirect,还能返回值(如校验错误)。组件用 useActionData() 读取——最典型的用途是表单校验错误回显:action 校验不通过时 return 错误对象,组件把它显示在表单上。

action 返回校验错误 → useActionData 回显
// action:校验失败不 redirect,而是 return 错误
async function action({ request }) {
  const data = await request.formData();
  const email = data.get('email');
  if (!email.includes('@')) {
    return { error: '邮箱格式不正确' };  // 返回给组件
  }
  await subscribe(email);
  return redirect('/done');
}

function Subscribe() {
  const result = useActionData();  // 拿 action 的返回值
  return (
    <Form method="post">
      <input name="email" />
      {result?.error && <p className="err">{result.error}</p>}
      <button>订阅</button>
    </Form>
  );
}

useNavigation:全局加载态

导航切换、loader 取数、action 提交这些异步过程中,useNavigation() 告诉你当前处于什么状态,用它做"提交中禁用按钮""页面切换进度条"。

useNavigation:提交中禁用按钮
import { useNavigation } from 'react-router-dom';

function NewPost() {
  const navigation = useNavigation();
  // state: 'idle' | 'submitting' | 'loading'
  const isSubmitting = navigation.state === 'submitting';
  return (
    <Form method="post">
      <input name="title" />
      {/* 提交中禁用,防止重复提交,无需自己维护 loading state */}
      <button disabled={isSubmitting}>
        {isSubmitting ? '发布中...' : '发布'}
      </button>
    </Form>
  );
}

整体心智模型:一图串起数据路由

导航到路由
→
loader 取数
→
组件 useLoaderData 拿数据渲染
<Form> 提交
→
action 写数据
→
loader 自动重跑
→
页面数据刷新

整个过程中,useNavigation 提供加载态,useActionData 承接 action 的返回(如错误),errorElement 兜底 loader/action 抛出的异常。数据的读、写、加载态、错误——全部由路由体系统一编排。

Hook / 概念职责
loader渲染前读数据
useLoaderData()组件读取 loader 的返回
action处理写操作(表单提交)
useActionData()组件读取 action 的返回(如校验错误)
useNavigation()全局加载 / 提交态
useRouteError() + errorElement捕获 loader/action 的异常
心智升级:从"组件驱动数据"到"路由驱动数据"
传统 React:组件是数据的中心,useState/useEffect 在组件里管理请求、loading、错误。数据路由把这些职责上移到路由层——路由知道每个 URL 需要什么数据(loader)、如何变更数据(action)、变更后如何刷新(自动 revalidate)。组件退回到"纯展示"的本分,只用几个 hook 读取已经准备好的数据。这与 Remix、Next.js App Router 的设计一脉相承,是现代 React 数据管理的主流方向。
什么时候用数据路由,什么时候用 React Query?
数据路由擅长"页面级、与 URL 强相关"的数据(进入某页就该有的数据)。React Query 擅长"缓存、去重、后台刷新、乐观更新"等精细的服务端状态管理。两者不互斥,大型项目常结合使用。小到中型项目,数据路由自带的 loader/action + 自动 revalidate 往往已经够用,能显著减少样板代码。
别在数据路由里继续用 useEffect 取数
切到 createBrowserRouter 后,如果还习惯性地在组件里写 useEffect + fetch,就白白丢掉了 loader 的所有好处(渲染前取数、自动 revalidate、错误统一处理),还退回了瀑布加载。迁移时要有意识地把请求从组件搬进 loader。

常见陷阱

  • 在数据路由下还习惯性写 useEffect + fetch:切到 createBrowserRouter 之后,如果继续在组件里用 useEffect 取数,等于把数据获取从"渲染前"退回"渲染后",重新引入了瀑布式加载(先渲染骨架 → 再发请求 → 再渲染数据),白白丢掉了 loader 天然自带的"进入页面时数据已就绪"的好处。
  • 误以为 useLoaderData 需要处理 loading 状态:很多人写 const user = useLoaderData() 后习惯性加一个 if (!user) return <Spinner />,这其实是多余且可能有害的判断——loader 在组件渲染之前已经跑完,数据在渲染时必然存在(除非 loader 本身抛错,那是 errorElement 该管的事)。这种冗余判断反而掩盖了数据结构可能有问题的真实错误。
  • 把 useActionData 当成"随时可读的最新表单结果":useActionData 返回的是"最近一次这个路由的 action 执行结果",如果页面首次加载还没提交过表单,它是 undefined;如果用户导航离开又导航回来,之前的 action 结果可能还残留展示,容易让人误判"这是不是最新提交的反馈"。使用前要结合 useNavigation().state 判断当前是否正在提交。
  • 混淆 loader 报错和 action 返回校验错误两种"错误"的处理方式:loader/action 内部 throw 的异常应该被 errorElement 兜底捕获,走的是"页面级异常"路径；而表单校验失败这类"可预期的业务错误"应该用 return { error: ... },交给 useActionData 在原表单里回显。两者搞反的话,校验失败会直接跳到错误边界整页报错,或者程序性异常被当成表单提示悄悄吞掉。

面试高频考点

  1. useLoaderData 读到的数据为什么不需要处理 loading 状态?—— 因为 loader 是在路由对应的组件渲染之前执行并等待完成的,React Router 保证组件真正渲染时,loader 的返回值已经就位。这与传统 useEffect 取数(先渲染组件,再发请求,请求期间要手动维护 loading state)是根本不同的时序,数据路由把"等待"这一步从组件渲染阶段挪到了导航阶段。
  2. useLoaderData/useActionData 这套模型和组件里手写 useEffect + fetch 相比,解决了什么问题?—— 主要解决三类问题:一是避免瀑布式加载(渲染前就能拿到数据,而不是渲染完再发请求);二是统一错误处理(loader/action 抛出的异常自动被 errorElement 捕获,不用每个组件各写一套 try/catch);三是自动重新验证(action 提交后相关 loader 自动重跑,数据自动刷新,不用手动管理"刷新" 逻辑)。
  3. action 返回校验错误和直接 throw 异常有什么区别,该怎么选?—— return 一个错误对象是"预期内的业务失败"(如表单校验不通过),会被 useActionData 接住,在原表单页面就地回显,用户可以立即修正重试;throw 异常是"非预期的程序性错误"(如网络中断、服务器 500),会被路由的 errorElement 捕获,通常展示为一个独立的错误页面。用错会导致体验错位——业务校验错误本该在表单里提示,却整页跳错误页。
  4. useNavigation 解决了什么问题,为什么不用自己维护一个 isSubmitting 的 useState?—— 因为提交状态、导航状态在数据路由模型里是路由体系统一管理的,useNavigation().state 直接反映"当前是否在提交/加载中",不需要在每个表单组件里手写一份 isSubmitting 状态并操心什么时候 setTrue/setFalse,也避免了手写逻辑遗漏边界情况(如提交失败后忘记复位)。

与其他知识点的联系

  • 本地 state 与全局 state(第 93 篇):数据路由把"数据获取"这一职责从组件搬到了路由层,这本质上是在回答"这份数据到底该由谁持有"的问题——loader 取到的数据不属于任何一个组件的本地 state,而是路由层统一持有、通过 useLoaderData 分发下去,天然规避了"要不要把请求结果提升到父组件"的纠结。
  • 服务端状态与客户端状态(第 94 篇):loader/action 管理的数据本质上就是服务端状态的一种落地方式——它同样面临"数据会过期""提交后要刷新"的问题,只是数据路由用"自动 revalidate"取代了 TanStack Query 的"缓存失效策略"。两者是解决同一类问题的不同工具,选型时要想清楚项目是否已经用了 React Router 的数据路由能力。
  • Context 与 useReducer 轻量方案(第 95 篇):无论是数据路由还是 Context+useReducer,核心思路都是把"状态怎么变"的逻辑从组件里挪出去、集中管理——reducer 集中处理 action,loader/action 集中处理数据读写,组件都只负责"读结果、发指令",不再关心过程细节。
  • Zustand 系列(第 96-99 篇):如果项目里的全局状态既有纯客户端偏好(该用 Zustand),又有强依赖 URL 的页面级数据(该用数据路由),两者会同时存在于一个应用中——理解各自的职责边界,才不会出现"该用 loader 取的数据硬塞进 Zustand,或者反过来"的架构混乱。
  • action 返回值与状态更新的心智一致性: useActionData 读取 action 的返回值,和 Zustand 的 set 更新 store、useReducer 的 dispatch 触发 reducer,本质上都是"触发一次变更 → 组件订阅到最新结果"的单向数据流模式,只是变更的触发方式和承载位置不同。
