React.lazy + Suspense 路由懒加载

用户只想看首页,却要先下载"设置页""订单详情页""后台管理页"全部代码打成的一个巨大 bundle——这就是没做代码分割的典型症状。路由级代码分割让每个路由只在真正被访问时才加载自己的 JS,首屏体积大幅缩小。React.lazy + Suspense 是实现它最基础的方式。

为什么需要路由级代码分割

不做分割时,打包工具(Vite/webpack)默认会把所有 import 到的模块打进一个(或很少几个) JS 文件。应用越大,这个文件越臃肿,而用户打开首页时,浏览器必须先把整个应用的代码下载、解析、执行完,才能看到第一屏内容——即使这一屏根本用不到"订单详情页"的代码。

不分割
main.js 包含所有路由代码
首屏必须等全部下载完
用户可能永远不会访问的页面
也要跟着首屏一起下载
路由级分割
每个路由是独立的 chunk 文件
首屏只加载当前路由需要的代码
访问其他路由时才按需下载
首屏体积、加载时间显著下降

React.lazy 的基本用法

React.lazy(loader) 接收一个返回 import() Promise 的函数,返回一个"懒加载组件"。这个组件第一次被渲染时才会真正触发 import() 去下载对应代码;下载完成前需要用 <Suspense> 包裹并提供 fallback,告诉 React "加载过程中显示什么"。

结合 createBrowserRouter 做路由级懒加载
import { lazy, Suspense } from 'react';
import { createBrowserRouter } from 'react-router-dom';

// 只有真正导航到 /orders 时,OrderList 对应的 chunk 才会被下载
const OrderList = lazy(() => import('./pages/OrderList'));
const Settings = lazy(() => import('./pages/Settings'));

const router = createBrowserRouter([
  {
    path: '/orders',
    element: (
      <Suspense fallback={<PageSkeleton />}>
        <OrderList />
      </Suspense>
    ),
  },
  { path: '/settings', element: (<Suspense fallback={<PageSkeleton />}><Settings /></Suspense>) },
]);

为什么 lazy 组件必须是 default export

lazy(loader) 要求 loader() 返回的模块对象里必须有 default 字段,因为 lazy 内部实现是直接取 module.default 作为要渲染的组件。如果页面组件用的是具名导出(export function Settings() {}),需要手动包一层转换。

具名导出需要手动转成 default export
// Settings.jsx 里如果写的是: export function Settings() {...}
const Settings = lazy(() =>
  import('./pages/Settings').then((module) => ({ default: module.Settings }))
);
常见踩坑: 页面组件用了具名导出,却直接 lazy(() => import('./pages/Settings')),运行时会报"组件是 undefined"这类错误——因为 module.default 根本不存在。约定路由页面组件统一用 export default,能省掉这层转换代码,也是社区的通行做法。
Suspense 边界放在哪一层? 可以整个应用只放一个顶层 Suspense,也可以像上面例子一样给每个路由单独包一层。后者的好处是:切换路由时只有对应内容区显示 loading,页头、侧边栏等公共布局不受影响,体验更好——这也是为什么很多项目会把 Suspense 放在布局组件里而不是应用最外层。
和路由篇的联系: createBrowserRouter 定义的 element 只是一段 JSX,懒加载组件放进去和放普通组件没有本质区别——真正起作用的是 lazy 让这段 JSX 对应的代码被单独打包,配合 Suspense 处理"代码还没下载完"这段真空期的 UI。

小结

  • 路由级代码分割让首屏只加载当前路由需要的 JS,避免把整个应用打成一个体积巨大的 bundle。
  • React.lazy(() => import('./Page')) 定义懒加载组件,渲染时才真正下载代码;必须配合 Suspense fallback 处理加载中的 UI。
  • 懒加载组件的模块必须有 default 导出,具名导出需要手动包一层 .then() 转换。
  • Suspense 边界放的位置决定了 loading 影响的范围,通常放在路由内容区而不是整个应用外层。
  • 下一篇讲 import() 本身的原理——它不是 React 专有语法,而是 ES 标准动态导入。

常见陷阱

  • 忘记包 Suspense:直接把 lazy 组件当普通组件用,不套 <Suspense>,渲染时会抛出"A component suspended while responding to synchronous input"之类的报错。根本原因是懒加载组件在代码没下载完时会 throw 一个 Promise,React 靠最近的 Suspense 边界捕获这个 Promise 并显示 fallback——没有边界,这个"异常"就一直往上抛,谁都接不住。
  • 具名导出直接传给 lazy:页面组件写的是 export function Settings() {},却直接 lazy(() => import('./pages/Settings')),运行时报"组件是 undefined"。因为 lazy 内部固定去取 module.default,具名导出的模块里根本没有 default 字段。养成路由页面统一 export default 的习惯,能避免这个坑,也省掉手动 .then() 转换那层代码。
  • fallback 缺失或太简陋导致白屏闪烁:Suspense 的 fallback 属性是可选的,但不写就等于加载期间什么都不渲染,用户会看到一闪而过的白屏,体验比不分割还差。fallback 应该给一个和目标页面尺寸、布局接近的骨架屏(如例子里的 PageSkeleton),而不是随手写个"加载中...",否则加载完成后内容会突然"跳一下"。
  • 顶层单一 Suspense 导致布局跟着一起消失:如果把唯一的 Suspense 放在应用最外层,切换任何路由时,连页头、侧边栏这些公共布局都会被 fallback 替换掉,整页闪烁。应该像文中示例那样把 Suspense 下沉到每个路由的内容区,让 loading 状态只影响真正在等待的那部分 UI。
  • 没有错误边界,chunk 加载失败时整个应用崩溃:懒加载的本质是运行时再发一次网络请求下载 JS 文件,网络波动、CDN 失效都可能导致这次 import() 直接 reject。Suspense 只处理"还在加载"这一种状态,不处理"加载失败";如果外层没有 ErrorBoundary,这个 reject 会变成未捕获异常,把整个应用炸挂。生产环境给每个路由懒加载都配一层错误边界,失败时给出"重试"按钮,才是完整的方案。

面试高频考点

  1. React.lazy 是怎么让代码分割生效的?—— lazy 本身不做打包,它只是把 import() 包装成一个特殊组件:首次渲染时才调用 loader() 触发 import(),而 import() 是 Vite/webpack 识别的"动态导入"语法,打包工具会据此把目标模块拆成独立 chunk。真正的分割动作发生在构建阶段,lazy 只是在运行时控制"什么时候去下载这个 chunk"。
  2. 为什么 lazy 组件渲染时会抛出 Promise,而不是直接返回 loading UI?—— 这是 Suspense 机制的设计:组件渲染函数发现依赖的资源没准备好,就 throw 一个 pending 的 Promise,React 捕获后暂停渲染这棵子树,展示最近的 Suspense fallback,并在 Promise resolve 后重新渲染。这样"异步加载"和"渲染逻辑"解耦——组件本身不需要写任何 loading 状态的判断代码。
  3. 具名导出为什么不能直接传给 lazy?—— lazy(loader) 内部约定 loader() 返回的模块对象必须有 default 字段,取的是 module.default;具名导出的模块没有这个字段,拿到的就是 undefined,渲染时自然报错。解决办法是让页面组件用 export default,或者手动 .then(module => ({ default: module.XXX })) 包一层。
  4. 多个路由各自的 Suspense 和一个全局 Suspense 有什么区别?—— 全局 Suspense 会让任意子路由的加载状态影响整个应用(包括页头、侧边栏等公共布局);路由级 Suspense 把加载边界收窄到内容区本身,切换路由时只有对应区域展示 fallback,公共布局保持稳定,是更常见的实践。
