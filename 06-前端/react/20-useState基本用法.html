useState 基本用法

知道了为什么需要 state，现在来学 useState 的语法和惯用法。这是 React 里使用频率最高的 Hook，值得把每个细节都搞清楚。

语法解析：三个部分

const [count, setCount] = useState(0) 这一行包含三个独立的概念，拆开看：

useState 语法全解析
import { useState } from 'react';

function Counter() {
  //  ↓ 解构赋值，拆出数组第一项和第二项
  const [count,    setCount]  =  useState(0);
  //       ↑             ↑              ↑
  //   当前状态值     setter 函数      初始值
  //  （这次渲染的值） （调用它来更新）  （只在第一次渲染时用）

  return (
    <div>
      <p>当前计数：{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(0)}>重置</button>
    </div>
  );
}
部分含义说明
count 当前状态值 这次渲染时 React 从内部存储取出来的值，是个普通 JS 变量，只读
setCount setter 函数 调用它来通知 React 更新，会触发组件重新渲染
useState(0) 初始值 只在组件第一次挂载时使用，后续渲染全部忽略这个参数

命名约定：[xxx, setXxx]

React 官方和整个社区都遵循这个约定：state 变量和它的 setter 配对命名，setter 以 set 开头。这是约定俗成，不是语法要求（因为本质是数组解构，你可以叫任何名字），但违反约定会让代码难读。

✅ 遵循约定
const [name, setName] = useState('');
const [age, setAge] = useState(0);
const [open, setOpen] = useState(false);
const [items, setItems] = useState([]);
❌ 让人困惑
const [a, b] = useState('');
const [data, update] = useState(0);
const [x, changeX] = useState(false);
// 见名不知意，团队协作噩梦

多个 state：各自独立，互不影响

一个组件可以有任意多个 useState，每一个都是独立的存储槽。更新其中一个，不会影响其他的。

三个独立 state：name / age / visible
function UserForm() {
  // 三个 useState 各自管理一块数据，完全独立
  const [name,    setName]    = useState('');       // 字符串
  const [age,     setAge]     = useState(18);      // 数字
  const [visible, setVisible] = useState(false);   // 布尔

  // 改 name 不会让 age 或 visible 重置，React 只更新对应的槽

  return (
    <div>
      <input
        value={name}
        onChange={e => setName(e.target.value)}
        placeholder="姓名"
      />
      <input
        type="number"
        value={age}
        onChange={e => setAge(Number(e.target.value))}
      />
      <button onClick={() => setVisible(!visible)}>
        {visible ? '隐藏预览' : '显示预览'}
      </button>
      {visible && <p>姓名：{name}，年龄：{age}</p>}
    </div>
  );
}

初始值的类型：可以是任何 JS 值

useState 的初始值没有类型限制，任何合法的 JS 值都可以：

初始值类型举例
useState(0)                    // 数字
useState('')                   // 空字符串
useState(false)                // 布尔
useState(null)                 // null（常用于"还没加载"的状态）
useState([])                   // 空数组
useState({ name: '', age: 0 }) // 对象

// 初始值也可以是函数（懒初始化，适合初始值计算开销大时）
useState(() => JSON.parse(localStorage.getItem('data')))
// 传函数而非值：只有第一次渲染时才调用这个函数
高频陷阱：useState(props.value)
初始值只在组件第一次挂载时生效，后续渲染（包括 props 更新触发的重渲）React 直接忽略初始值参数。所以 useState(props.value) 是一个非常常见的错误写法——你以为 props 更新时 state 会跟着变，实际上它永远保持第一次传入的值。

如果你需要把 props 同步到 state，应该用 useEffect 或直接重新设计数据流，而不是把 props 塞进 useState 的初始值。

与其他知识点的联系

  • 为什么需要 state（上一篇 19）：上一篇讲了普通变量的两个致命问题（不触发重渲、每次渲染重置），本篇的三个部分正好一一对应着解决方案——useState(初始值) 里的初始值只在第一次挂载生效，解决"要不要重新初始化"的问题；count 是 React 帮你跨渲染保存的值；setCount 是主动通知 React 重新渲染的入口。脱离上一篇的问题背景，本篇的语法会显得像是"要背的规则"，而不是"解决具体问题的设计"。
  • state 异步更新（下一篇 21）：本篇只讲了 useState 的静态语法，没有涉及"调用 setter 之后 count 到底什么时候变"这个问题——这正是下一篇要讲的批处理机制。学完本篇如果直接照搬 setCount(count + 1) 连写三次，会撞上下一篇讲的经典坑（只加了一次），所以这两篇要连起来学，不能只学半截。
  • useReducer（本系列后续复杂状态管理篇）：本篇演示的是"一个组件有多个独立 useState"的场景（name/age/visible 各自管理）。当这些状态之间开始有联动逻辑（比如修改 A 要同时影响 B 的合法性），继续用多个孤立的 useState 会让更新逻辑分散在各处难以维护，这时候通常需要升级到 useReducer，把所有更新逻辑集中到一个 reducer 函数里管理。
  • useEffect（本系列 Hooks 篇）：本篇 callout 提到的"useState(props.value) 陷阱"给出的正确方向就是用 useEffect 监听 props 变化后手动 setState 同步——这是 state 和 props 之间"何时该同步、怎么同步"这一类问题的标准解法，也是理解 useEffect 依赖数组机制的一个天然切入口。
  • Hook 调用顺序规则（19 篇 callout 已提及）：本篇"多个 state 各自独立"的例子里，三个 useState 按固定顺序调用，React 靠这个顺序而不是变量名去匹配每个 Hook 对应的存储槽位——这也解释了为什么不能把某个 useState 调用放进 if 判断里：一旦某次渲染跳过了它，后面所有 Hook 的槽位匹配都会错位。
