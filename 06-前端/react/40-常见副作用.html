React · useEffect

三大常见副作用及其清理

数据获取、事件订阅、定时器——这是 useEffect 最典型的三类用途。每一类都遵循同一个模板：setup 建立 + return 清理。但在写之前，先问一句官方灵魂拷问："你可能根本不需要 Effect。"

① 数据获取

fetch + 竞态处理 + 清理
function Profile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    let ignore = false;
    fetch(`/api/user/${userId}`)
      .then(r => r.json())
      .then(d => { if (!ignore) setUser(d); });

    return () => { ignore = true; }; // 防竞态:见上一篇
  }, [userId]); // userId 变了重新拉取
}

② 事件订阅

addEventListener 必须配 remove
useEffect(() => {
  function onResize() { setWidth(window.innerWidth); }
  window.addEventListener('resize', onResize);

  // 清理:传入同一个函数引用才能正确移除
  return () => window.removeEventListener('resize', onResize);
}, []); // 空数组:只订阅一次
WebSocket / 第三方库订阅同理：socket.on(...) → socket.off(...)，observable.subscribe() → subscription.unsubscribe()。凡是"订阅"就必然有"退订"，成对写。

③ 定时器

setInterval 必须配 clearInterval
useEffect(() => {
  const id = setInterval(() => {
    setSeconds(s => s + 1); // 函数式更新:避免闭包陷阱
  }, 1000);
  return () => clearInterval(id); // 卸载/重跑前清掉
}, []);

重要:你可能不需要 Effect

官方专门写了一整章 "You Might Not Need an Effect"。很多人把 effect 当万金油，其实以下场景不该用 effect：

❌ 用 effect 派生数据
const [full, setFull] = useState('');
useEffect(() => {
  setFull(first + ' ' + last);
}, [first, last]);
// 多一次渲染，纯属浪费
✅ 渲染时直接算
// 能从现有 state 算出来的，
// 直接在渲染期计算即可
const full = first + ' ' + last;
// 不需要 state，不需要 effect
  • 能由现有 props/state 算出的值 → 渲染时直接算，别放 state + effect。
  • 响应用户事件做的事（提交表单、点击发请求）→ 放事件处理器，不是 effect。effect 是给"因渲染而必须同步外部系统"的场景。
  • 昂贵计算 → 用 useMemo，不是 effect。
  • props 变了要重置 state → 优先用 key 让组件重新挂载。
判断口诀："这段代码是因为组件显示在屏幕上而必须运行，还是因为用户做了某个操作而运行？" 前者用 effect，后者用事件处理器。混淆这两者是 effect 滥用的根源，会导致多余的渲染、难追的时序 bug。
数据获取的现实建议。裸 useEffect + fetch 能用，但要手动处理竞态、缓存、loading、错误、重试……真实项目通常用 React Query / SWR / RTK Query 这类库，或框架内置的数据加载（如 Next.js / React Router 的 loader）。它们把 effect 里那些繁琐的边界情况都封装好了。理解 effect 版是为了懂原理，生产里别重复造轮子。
