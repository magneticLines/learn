Props 默认值：defaultProps 与解构默认参数

组件是给别人用的——你没法保证使用者每次都会传所有 props。当某个 prop 缺失时，子组件拿到 undefined。默认值就是在"缺货"时自动备货的机制，让组件在不完整输入下依然能正常工作。

为什么需要默认值：undefined 的危害

当父组件忘记传（或者有意不传）某个 prop 时，子组件访问到的是 undefined。不处理的话有两种后果：

没有默认值时的两种灾难
function Avatar({ size, src, alt }) {
  return <img width={size} height={size} src={src} alt={alt} />;
}

// 父组件只传了 src，没传 size 和 alt
<Avatar src="/avatar.png" />

// 结果渲染出：<img width="undefined" height="undefined" src="/avatar.png" alt="undefined">
// 灾难一：width="undefined" 不是数字，图片尺寸完全混乱
// 灾难二：alt="undefined" 出现在页面上，对屏幕阅读器用户极不友好

如果 prop 缺失时还调用了它的方法（比如 src.split('/')），那就直接报错："Cannot read properties of undefined"——页面崩溃。

方式一：解构默认值（推荐，现代写法）

在参数解构时直接用 = 指定默认值，这是标准的 ES6 语法：

解构默认值——声明与默认值在同一处，一目了然
// size 没传时用 40，src 没传时用占位图，alt 没传时用空字符串
function Avatar({
  size = 40,                        // 数字默认值
  src = '/default-avatar.png',    // 字符串默认值
  alt = '',                         // 空字符串也是合法默认值
}) {
  return <img width={size} height={size} src={src} alt={alt} />;
}

// 各种使用场景
<Avatar />                              // size=40, src='/default-avatar.png', alt=''
<Avatar size={80} />                  // size=80（覆盖默认值），其余用默认
<Avatar src="/shea.png" alt="Shea" />  // src、alt 覆盖，size 用默认 40

方式二：defaultProps（旧写法，了解即可）

React 早期的默认值机制——在组件函数外部设置一个 defaultProps 对象属性：

defaultProps——旧写法，React 18+ 已标记为过时
function Avatar({ size, src, alt }) {
  return <img width={size} height={size} src={src} alt={alt} />;
}

// 默认值放在函数外面，和函数定义是分离的
// 看函数时不知道有默认值，需要滚到文件底部才能看到
Avatar.defaultProps = {
  size: 40,
  src: '/default-avatar.png',
  alt: '',
};

// React 18 控制台会警告：
// Warning: Avatar: Support for defaultProps will be removed from
// function components in a future major release.

两种方式对比

维度 解构默认值（推荐） defaultProps（旧写法）
声明位置 函数参数处，默认值在定义旁边 函数体外部，通常在文件末尾
可读性 函数签名即文档，一眼看到所有 props 和默认值 需要翻到文件底部才能看到默认值
TypeScript 支持 配合类型标注，类型推断完美 类型定义复杂，不够优雅
React 版本状态 推荐，永远有效 React 18+ 已标记为 deprecated，未来版本将移除
适用场景 所有场景 维护旧代码时会看到，新代码不应使用

一个容易犯的错：null 不会触发默认值

null vs undefined——默认值只对 undefined 生效
function Avatar({ src = '/default.png' }) {
  // src 的值取决于父组件传了什么
  return <img src={src} />;
}

// 不传 → undefined → 用默认值 '/default.png' ✅
<Avatar />

// 传 undefined → 同上，用默认值 ✅（显式传 undefined 和不传等价）
<Avatar src={undefined} />

// 传 null → 不用默认值！null 是有意义的值，表示"明确设为空" ❌
// src 会是 null，<img src={null}> 不显示图片，也不用默认图
<Avatar src={null} />
null 和 undefined 的语义差异：undefined 表示"没有传"，null 表示"明确告诉你：这里没有值"。ES6 解构默认值只在 undefined 时触发，null 被视为合法的值传入。如果你的业务中可能出现 null，要在组件内部用 src ?? '/default.png' 做兜底（空值合并运算符对 null 和 undefined 都生效）。
本节核心：① 父组件未传的 prop 在子组件中是 undefined，不处理会导致渲染异常；② 推荐在解构参数处设默认值，声明与默认值在同一位置，可读性最好；③ defaultProps 是旧写法，React 18+ 已废弃，看旧代码会遇到，新代码用解构默认值；④ 默认值只对 undefined 生效，null 不会触发默认值，这是 JavaScript 语言层面的设计。

常见陷阱

  • 以为 null 会触发默认值，结果拿到的还是 null：本篇讲过的核心坑——解构默认值只在右值严格等于 undefined 时生效，null 被 JS 当作"合法的、有意义的值"直接放行。如果后端接口习惯用 null 表示"暂无数据"，直接套解构默认值不会生效，必须额外用 ??（空值合并运算符）兜底：const value = props.x ?? defaultX，它对 null 和 undefined 都生效。
  • 把 defaultProps 用在函数组件上，遇到 React 未来版本直接失效：React 官方已经在文档和控制台警告里明确表示，函数组件的 defaultProps 支持会在未来主版本移除（class 组件的 defaultProps 不受影响，还会继续支持）。新代码一律用解构默认值，看到旧代码里的 defaultProps 要留意这是历史遗留，迁移时机成熟应替换掉。
  • 默认值是对象/数组时，每次渲染都创建新的默认对象：比如 function List({ items = [] })，虽然效果上没问题，但如果这个默认值被传进某个用 useEffect 依赖它、或者用 React.memo 做浅比较优化的地方，每次渲染生成的新数组引用会导致依赖判断"总是变了"，从而让优化失效。这种情况需要把默认值提到组件外部做成一个稳定的常量引用。
  • 给必传的 prop 设默认值，掩盖了"忘记传参"的真实 bug：默认值的本意是"这个字段本来就允许缺省"，如果给一个业务上必须传的字段（比如用户 ID）也设了默认值，父组件忘记传时不会报错也不会有明显异常，只会在运行到很后面才发现数据不对，排查成本反而更高。可选字段才该有默认值，必传字段应该让缺失暴露出来（比如用 TypeScript 标记为必填，或在开发环境加校验）。
  • 解构默认值只作用于"这次调用没传该字段"，不会对已解构出的值做二次校验：比如 { size = 40 }，如果父组件传了 size={0}，0 是合法数字（不是 undefined），默认值不会介入，最终拿到的 size 就是 0——如果业务上 0 是无意义的尺寸，这不是默认值机制的锅，需要在组件内部自己做业务校验。

面试高频考点

  1. 解构默认值和 defaultProps 在触发时机和实现层级上有什么本质区别？—— 解构默认值是纯 JavaScript 语言特性，在函数调用、参数绑定的那一刻由 JS 引擎完成判断和赋值，不依赖任何框架；defaultProps 是 React 在渲染组件之前，由协调（reconciliation）逻辑读取组件对象上的静态属性，主动把缺失字段合并进 props 对象里再传给组件函数。前者更"轻"、和组件定义耦合更紧，后者曾经的优势是能配合 PropTypes 做类型+默认值集中声明，但代价是默认值和组件定义分离在两处。
  2. 为什么默认值判断用的是 undefined 而不是"falsy 值"（如 0、''、false）？—— 因为 falsy 值本身经常是合法的业务输入（比如数量传 0、开关传 false），如果默认值机制把这些也当作"没传"来覆盖，会导致明确传入的合法值被意外替换成默认值，这是比"忘记设默认值"更隐蔽的 bug。只用严格等于 undefined 判断，才能准确表达"确实没传这个参数"的语义。
  3. ??（空值合并运算符）和解构默认值可以互相替代吗？—— 不完全能。解构默认值只能在函数参数解构、数组解构、对象解构这几个语法位置使用，且只对 undefined 生效；?? 是一个通用表达式运算符，可以在任意位置使用（不局限于解构），且同时覆盖 null 和 undefined。当你需要"null 也要兜底"时只能用 ??，反过来单纯做参数默认值，解构默认值写法更简洁、更贴近函数签名即文档的可读性诉求。

与其他知识点的联系

  • children prop（上一篇 16）：children 本质也是一个普通 prop，一样可以设默认值（比如 function Empty({ children = '暂无数据' })），本篇讲的"解构默认值只对 undefined 生效"这条规则对 children 同样适用——不传内容时 children 是 undefined，会触发默认值；显式传 null 则不会。
  • Props 只读原则（下一篇 18）：默认值解决的是"没传值时怎么办"，只读原则解决的是"传进来的值能不能改"，两者合起来构成了 props 机制的完整边界——组件只能读 props（包括默认值兜底之后的最终值），不能反向修改它，这也是为什么默认值必须在"进入组件之前"就确定好，而不是组件内部动态调整。
  • 为什么需要 state（19 篇）：props 默认值解决的是"静态兜底"问题——值从外部传入时的缺省处理；state 解决的是"动态变化"问题——组件内部随时间演变的数据。两者不要混淆：如果发现自己想给一个 prop"设置默认值之后还想再改它"，往往说明这个数据其实该用 useState 内部管理，而不是一直依赖 props 默认值。
  • TypeScript 类型系统（本系列工程化篇）：本篇提到解构默认值配合 TypeScript 类型标注能获得完美的类型推断——比如 size = 40 会让 TS 自动推断 size 是 number 类型（即便调用方没传也不会是 undefined），这也是它相比 defaultProps 在类型层面更优的原因之一，类型系统和默认值机制在这里是互相增益的。
  • useState 初始值（20 篇）：Props 默认值和 useState(初始值) 有个容易混淆但完全不同的地方——props 默认值在每次调用（每次渲染）时，只要没传就会重新生效；而 useState 的初始值只在组件第一次挂载时使用一次，之后即便参数变了也不会重新生效。20 篇里提到的 useState(props.value) 陷阱，本质上就是把这两种"默认值"的生效时机搞混了。
