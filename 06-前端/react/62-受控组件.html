React · 受控组件

受控组件:让 state 成为表单的唯一真相

原生 <input> 自己会记住用户输入的值。但在 React 里我们更推荐把这份"记忆"交给 state——输入框显示什么、值是多少,全由 state 说了算。这就是受控组件。

定义:value 由 state 提供,onChange 把变化写回 state

受控组件的判断标准很简单:表单元素的 value 属性绑定到某个 state,同时用 onChange 监听输入、把新值 setState 回去。React 完全掌控这个值。

受控 input 的标准写法
function NameForm() {
  // state 是这个输入框的"唯一真相"
  const [name, setName] = useState('');

  return (
    <input
      value={name}                          // 显示什么由 state 决定
      onChange={e => setName(e.target.value)}  // 输入时把新值写回 state
    />
  );
}

数据闭环:一个完整的循环

受控组件形成了一个严丝合缝的闭环——用户敲键盘不会"直接"改屏幕,而是先经过 React:

用户输入
→
onChange 触发
→
setState
→
重新渲染
→
value 更新到界面

注意:屏幕上你看到的字符,严格来说不是"你刚敲进去的",而是 React 重新渲染后从 state 里读出来又渲染上去的。正常情况下二者一致,你感觉不到差别。但这个机制正是受控的威力所在。

单一数据源(single source of truth)
受控组件的核心思想:一个值的"真相"只存在一处——React state。DOM 里的显示只是这份真相的"投影"。这与状态提升的"单一数据源"是同一个哲学:数据只有一个权威副本,UI 永远跟着它走。因为如此,你随时可以从代码里读到、改到当前值,而不用去查询 DOM。

为什么推荐受控?state 即拿即用

因为值实时同步在 state 里,你可以随手做任何"边输入边处理"的事,无需等到提交或去读 DOM:

受控让"实时能力"信手拈来
const [email, setEmail] = useState('');

// ① 实时校验:state 变化立刻能算出是否合法
const isValid = email.includes('@');

// ② 格式化:输入时强制转大写、去空格
const onChange = e => setEmail(e.target.value.trim());

return (
  <div>
    <input value={email} onChange={onChange} />
    // ③ 禁用提交:根据实时值控制按钮
    <button disabled={!isValid}>提交</button>
    {!isValid && <span>邮箱格式不对</span>}
  </div>
);

实时校验、输入格式化、按钮联动、字数统计、联动其他字段……这些都是受控组件"顺手就能做"的。因为值一直在 state 里,你不需要额外去"抓取"。

下拉、勾选、多行:都是同一套路

select / checkbox / textarea 全部受控
// select:value 绑 state
<select value={city} onChange={e => setCity(e.target.value)}> ... </select>

// checkbox:用 checked(不是 value),读 e.target.checked
<input type="checkbox" checked={agreed}
  onChange={e => setAgreed(e.target.checked)} />

// textarea:React 里也用 value(而非在标签内写文本)
<textarea value={bio} onChange={e => setBio(e.target.value)} />
最经典陷阱:只写 value 不写 onChange,输入框"打不进字"
<input value={name} /> 而没有 onChange,React 会把值锁死在 state 的初始值上——你敲键盘,state 不变,重渲染后又把老值写回去,看起来就像"输入框失灵"。控制台还会警告:You provided a `value` prop without an `onChange` handler。要么补上 onChange 变成受控,要么改用 defaultValue 变成非受控(下一篇)。
陷阱:value={undefined / null} 会被当成非受控
如果 state 初始值是 undefined,React 会认为这是非受控组件;等你后来给它赋了字符串,又变成受控,触发"从非受控切换到受控"的警告。受控组件的初始值一定要给确定的值(如空字符串 '')。

小结

  • 受控组件:value(或 checkbox 的 checked)由 state 提供,onChange 把变化写回 state。
  • React state 是唯一真相,DOM 显示只是它的投影,形成"输入→onChange→setState→重渲染"闭环。
  • 优势:实时校验、格式化、字段联动、按钮禁用等能力"顺手就能做",值即拿即用。
  • select/textarea 用 value,checkbox 用 checked,套路一致。
  • 坑:只写 value 不写 onChange 会"打不进字";初始值别用 undefined/null。
