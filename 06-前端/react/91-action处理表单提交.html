React · 路由

action:用路由处理表单提交(写数据)

loader 负责"读",action 负责"写"。数据路由用 <Form> 组件 + 路由的 action 函数,把表单提交也纳入路由体系——回归 HTML 原生的表单模型,却又是纯前端、无刷新的。

回归本源:HTML 表单本就是"提交到某处"

原生 HTML 里,<form action="/x" method="post"> 提交会把表单数据发到 action 指定的地址。React Router 的数据路由借用了这个心智:<Form> 组件提交时,不刷新页面,而是调用当前路由的 action 函数。

<Form> + action:声明式写数据
import { Form, redirect } from 'react-router-dom';

// 组件:用 <Form> 代替 <form>,不写 onSubmit
function NewPost() {
  return (
    <Form method="post">
      <input name="title" />
      <textarea name="body" />
      <button type="submit">发布</button>
    </Form>
  );
}

// 路由配置:action 接收提交,写数据
const router = createBrowserRouter([
  {
    path: '/posts/new',
    element: <NewPost />,
    action: async ({ request }) => {
      // action 拿到原生 Request 对象
      const formData = await request.formData();
      const title = formData.get('title');  // 按 input 的 name 取
      await createPost({ title, body: formData.get('body') });
      // 成功后重定向(在 action 里用 redirect,不用 Navigate)
      return redirect('/posts');
    },
  },
]);

关键机制:action 后 loader 自动重跑

这是 action 最省心的地方。写操作完成后,React Router 会自动重新执行相关路由的 loader,页面数据自动刷新。你不需要手动 setState 或重新 fetch——增删改后列表自己就更新了。这个"写完自动revalidate"的闭环,是数据路由相比手写 fetch 的最大体验提升。

为什么鼓励回到"非受控"表单?
受控组件(每个 input 绑 state)是 React 的常规做法,但对纯提交型表单,它带来大量样板:每个字段一个 state、一个 onChange。<Form> 走的是浏览器原生的 FormData 路线——input 只需一个 name,提交时统一收集。这不仅代码更少,还天然支持渐进增强(即使 JS 没加载,原生 form 也能提交)。React Router 在这里刻意向"平台原生能力"靠拢。
useFetcher:不导航的提交
<Form> 提交默认会触发导航/revalidate。如果只想在原地提交(如"点赞"按钮、行内编辑),用 useFetcher() 拿到 fetcher.Form 提交——它调用 action 但不改变 URL、不触发页面导航,适合局部交互。
陷阱:input 忘写 name
action 靠 formData.get(name) 取值,如果 input 没有 name 属性,FormData 里根本不会收集它,取出来是 null。这和原生 HTML 表单一致——没 name 的字段不参与提交。从受控表单迁移过来时最容易漏,因为受控表单不依赖 name。
