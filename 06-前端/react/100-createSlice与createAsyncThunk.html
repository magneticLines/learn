createSlice 与 createAsyncThunk：告别 Redux 样板代码

传统 Redux 要手写 action type 常量、action creator、switch reducer、深拷贝更新——一个功能改三四个文件。Redux Toolkit（RTK）是官方现在唯一推荐的写法，一个 createSlice 把这些全包了。

传统 Redux 的痛：样板代码淹没逻辑

先看传统 Redux 写一个计数器要多少代码。真正的业务逻辑（加一、减一）只有两行，其余全是"仪式"。

传统 Redux：三段式样板
// 1) action type 常量——纯粹为了避免拼写错误
const INCREMENT = 'counter/increment';
const DECREMENT = 'counter/decrement';

// 2) action creator——手写一个返回 { type, payload } 的函数
const increment = () => ({ type: INCREMENT });
const decrement = () => ({ type: DECREMENT });

// 3) reducer——switch，且必须手动返回新对象（不可变更新）
function reducer(state = { value: 0 }, action) {
  switch (action.type) {
    case INCREMENT: return { ...state, value: state.value + 1 };
    case DECREMENT: return { ...state, value: state.value - 1 };
    default: return state;
  }
}
为什么这是问题？不只是"字多"。type 常量分散在多个文件、容易写错；每个 reducer 都要记得 ...state 展开否则漏字段；异步请求还得额外配 redux-thunk / redux-saga。RTK 就是为了把这些全部消化掉而生的官方方案。

createSlice：一个函数搞定 action + reducer

createSlice 让你只写"发生了什么状态变化"，它自动帮你生成对应的 action type 和 action creator。上面那一大坨等价于：

counterSlice.js
import { createSlice } from '@reduxjs/toolkit';

const counterSlice = createSlice({
  name: 'counter',              // action type 前缀：counter/increment
  initialState: { value: 0 },
  reducers: {
    // 关键：这里可以"直接改"state，因为 RTK 内置 Immer
    increment(state) { state.value += 1; },
    decrement(state) { state.value -= 1; },
    // action.payload 就是 dispatch 时传的参数
    incrementBy(state, action) { state.value += action.payload; },
  },
});

// createSlice 自动生成同名 action creator，导出即可用
export const { increment, decrement, incrementBy } = counterSlice.actions;
export default counterSlice.reducer;
为什么能"直接改"？——Immer 的魔法
RTK 内部用 Immer 把 state 包成一个 Proxy（草稿 draft）。你写 state.value += 1 时，改的是草稿；Immer 记录你的每一步操作，最后基于原 state 生成一个全新的不可变对象返回。所以你享受了"可变写法"的直觉，实际得到的仍是不可变更新——引用变了，React/Redux 才能检测到变化。
Immer 陷阱：不要"既改又返回"。reducer 里要么直接改 state.x = ...（不 return），要么 return 一个全新对象——两者只能选一个。如果你既改了草稿又 return 别的东西，Immer 会报错。此外，如果你要整体替换 state（比如 return action.payload），那就走 return 分支，不要再碰 state。

createAsyncThunk：处理异步的标准姿势

请求接口是异步的，reducer 必须是纯同步函数，所以异步逻辑要放在 thunk 里。createAsyncThunk 帮你把一次异步请求自动拆成三个生命周期 action：

dispatch(fetchUser())
→
pending
（请求发出）
→
fulfilled
（成功，带 payload）
/
rejected
（失败，带 error）
userSlice.js —— 异步获取用户
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// 第一个参数是 action type 前缀，第二个是 async 函数（payloadCreator）
export const fetchUser = createAsyncThunk(
  'user/fetch',
  async (userId, thunkAPI) => {
    const res = await fetch(`/api/users/${userId}`);
    if (!res.ok) // 主动失败时用 rejectWithValue 携带自定义错误
      return thunkAPI.rejectWithValue('加载失败');
    return await res.json();   // return 的值 → fulfilled 的 payload
  }
);

const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, loading: false, error: null },
  reducers: {},
  // extraReducers：响应「外部」action（thunk 的三个生命周期）
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending, (s) => { s.loading = true; s.error = null; })
      .addCase(fetchUser.fulfilled, (s, a) => { s.loading = false; s.data = a.payload; })
      .addCase(fetchUser.rejected, (s, a) => { s.loading = false; s.error = a.payload; });
  },
});

reducers vs extraReducers 的区别

reducersextraReducers
处理谁本 slice 自己定义的 action外部 action（thunk、其他 slice）
是否生成 action是，自动生成 action creator否，只响应已存在的 action
典型用途同步的本地状态变更异步 thunk 的 pending/fulfilled/rejected

面试高频：①"RTK 里为什么能直接修改 state？"——因为内置 Immer，reducer 收到的是 draft Proxy，最终生成新对象。②"createAsyncThunk 有哪三个状态？"——pending / fulfilled / rejected，在 extraReducers 里用 addCase 分别响应。③"reducers 和 extraReducers 区别？"——前者定义并处理自己的 action，后者只响应外部 action。

与其他知识点的联系

  • Redux 核心概念（reducer、action）：createSlice 不是引入了新概念，而是把传统 Redux 里"action type、action creator、reducer"这三样东西自动化生成。理解传统 Redux 的心智模型（纯函数 reducer 接收 state 和 action，返回新 state）仍然是必要的——createSlice 只是替你写了这层"仪式代码"，底层运行机制完全没变，dispatch 一个 action、reducer 响应它计算新状态的流程一模一样。
  • Immer 与不可变数据：本文反复强调的"看似直接修改，实际生成新对象"正是 Immer 这个库的核心能力。Immer 用 Proxy 拦截你对 draft 对象的读写操作，记录下所有"变更意图"，最后一次性基于原始 state 生成一个新对象——这是为什么 state.value += 1 这种看起来违反 Redux"状态不可变"原则的写法，实际结果依然完全符合不可变更新的要求。
  • 异步流程与 loading/error 状态管理：createAsyncThunk 把"发请求→等待→成功/失败"这套异步生命周期标准化成三个 action（pending/fulfilled/rejected），这和普通组件里用 useState 手动维护 loading、error 状态是同一件事的两种做法——区别在于 RTK 把这套状态管理逻辑从组件里抽出来，放进了全局可复用的 slice 里，避免每个用到这个请求的组件都要重复写一遍 loading/error 判断。
  • configureStore（101）：createSlice 产出的 reducer 函数最终要交给 configureStore 组装成一个完整的 store 才能生效——本文只是"定义状态变化规则"，101 篇讲的是"把这些规则组装起来并接入 React 应用"，两者是紧密的上下游关系，缺一个都无法运行。
