BrowserRouter 与 HashRouter：SPA 路由的两种底层实现

React 本身只负责渲染 UI，不管 URL。要让"点击链接切换页面而不刷新整站"这件事发生，需要 React Router。它的第一块拼图，就是选一个"路由容器"包裹整个应用——而容器有两种：BrowserRouter 和 HashRouter。

先搞清楚：SPA 路由到底是什么

传统多页应用（MPA）里，每个 URL 对应服务器上的一个页面文件。你点链接 → 浏览器向服务器发请求 → 服务器返回一整张新 HTML → 整页刷新（白屏一下，重新加载所有 JS/CSS）。

单页应用（SPA）只有一张 HTML，所有"页面"都是同一份 JS 在客户端动态渲染出来的。所谓"路由"，本质上是：在不向服务器发请求、不刷新页面的前提下，改变地址栏的 URL，并根据新 URL 渲染不同的组件。

点击链接
→
JS 拦截,改 URL
→
匹配路由表
→
渲染对应组件

关键词是"拦截"和"不刷新"。要做到这两点，浏览器提供了两套原生能力，对应 Router 的两种实现。

BrowserRouter:基于 History API

BrowserRouter 底层用的是浏览器的 History API（history.pushState() / history.replaceState()）。这套 API 能在不刷新页面的前提下修改地址栏的完整路径，并往浏览器历史栈里压入记录。

History API——SPA 路由的浏览器底座
// 这就是 React Router 内部帮你做的事（简化版）
// 修改 URL 为 /about，但不发请求、不刷新页面
history.pushState(null, '', '/about');

// 浏览器前进/后退时触发，Router 监听它来重新匹配路由
window.addEventListener('popstate', () => {
  // 重新根据 location.pathname 渲染组件
});

URL 长这样：https://site.com/user/42——干净、专业，和普通网站一模一样。这是它最大的优点。

陷阱:BrowserRouter 需要服务端配合。 当用户在 /user/42 页面直接刷新（或收藏后直接访问）时，浏览器会真的向服务器请求 /user/42 这个路径。但服务器上根本没有这个文件，会返回 404。

解决办法：配置服务器把所有路径都回退到 index.html（Nginx 的 try_files $uri /index.html，或 historyApiFallback）。让前端 JS 接手后，再根据 URL 渲染。这就是为什么用 BrowserRouter 部署后经常"首页能看,刷新子页面就 404"。

HashRouter:基于 URL 锚点(#)

HashRouter 用的是 URL 里 # 后面的部分（hash / 锚点）。浏览器有个天然特性：改变 hash 不会触发页面请求（它本来是用来页内跳转到锚点的），并且能通过 hashchange 事件监听变化。

Hash 模式的 URL 形态
// BrowserRouter
https://site.com/user/42

// HashRouter —— 真实路径永远是根，# 后面的才是"前端路由"
https://site.com/#/user/42
//            ↑ 服务器只看到 "/"，永远返回 index.html

因为 # 后面的内容永远不会发给服务器，服务器眼里所有请求都是根路径 /，只需返回 index.html 即可。所以 HashRouter 不需要任何服务端配置，刷新永不 404。代价是 URL 里多了个丑丑的 #，且对 SEO 不友好。

根组件包裹:两种都只是换个容器

用法上两者完全一致，只是把最外层容器换个名字。通常在应用入口处包裹整个 App：

main.jsx / index.js——路由容器包裹根组件
import { BrowserRouter } from 'react-router-dom';
// import { HashRouter } from 'react-router-dom';  // 换成这个即为 hash 模式

ReactDOM.createRoot(document.getElementById('root')).render(
  <BrowserRouter>       {/* 必须包在最外层，路由 hook 才能拿到 context */}
    <App />
  </BrowserRouter>
);
面试高频陷阱:为什么必须包在最外层? Router 本质是一个 Context Provider，它把当前 location、navigate 等信息通过 Context 往下传。<Routes>、<Link>、useNavigate 等都是它的消费者。如果组件不在 Router 内部，就拿不到这个 context，会直接报错 "useXxx() may be used only in the context of a <Router> component"。这也是新手最常见的报错来源之一。

如何取舍

维度BrowserRouterHashRouter
底层History API（pushState）URL hash（#）
URL 形态/user/42（干净）/#/user/42（带 #）
服务端配置必须回退到 index.html不需要任何配置
刷新子页面配置错就 404永远正常
SEO友好（可配 SSR）不友好（爬虫常忽略 #）
适用场景正式产品、有服务端控制权静态托管、纯内部后台、无法改服务器配置（如挂在别人子路径下）
实践建议:优先 BrowserRouter。 现代部署（Vercel / Netlify / Nginx）配置回退非常简单，得到的干净 URL 更专业、更利于 SEO 和分享。只有在无法控制服务器路由（例如把应用部署到 GitHub Pages 的子路径、或嵌进某个不受控的静态环境）时，才退而选择 HashRouter。

还有第三种 MemoryRouter——路由状态只存在内存里，完全不碰 URL，主要用于测试环境和 React Native。

小结

  • SPA 路由的本质：用 JS 拦截导航，改 URL 但不刷新页面，再根据 URL 渲染组件。
  • BrowserRouter 用 History API，URL 干净，但刷新子页面需要服务端把所有路径回退到 index.html。
  • HashRouter 用 URL hash，# 后内容不发给服务器，无需服务端配置、刷新永不 404，代价是 URL 带 # 且 SEO 差。
  • 两者都是 Context Provider，必须包在应用最外层，否则路由 hook / 组件报错。
  • 默认选 BrowserRouter，无服务器控制权时才用 HashRouter。

与其他知识点的联系

  • Context 机制:本篇提到的"必须包在最外层"陷阱,本质是 React Context 的传递规则——Router 是 Provider,useNavigate/useParams 等路由 hook 都是消费者,不在 Provider 包裹范围内就拿不到 context,这与其他基于 Context 的能力(如主题、国际化)报错原理完全一致。
  • 何时拆分组件(第76篇):路由本身就是一种天然的组件拆分依据——不同路径对应不同页面组件,恰好命中"职责不同"的拆分信号。选定 BrowserRouter/HashRouter 之后,才谈得上用路由表去组织这些被拆分出来的页面组件。
  • useNavigate / Link(第79–80篇):本篇解释的 History API 和 URL hash 机制,正是后面 Link 点击拦截、useNavigate 编程式跳转的底层实现原理——理解了 pushState 不刷新页面的本质,才能理解为什么这些 API 能做到"看起来像跳转、实际没有整页刷新"。
