分页 vs 无限滚动

上一篇的虚拟滚动解决的是"数据已经在手里,怎么少渲染 DOM"。但更前置的问题是:数据本身就不该一次性全部拉下来——几万条数据一次性请求,接口慢、流量大,用户也用不上。分页和无限滚动是两种"分批要数据"的方案,权衡点完全不同。

共同目标:避免一次性加载全部数据

不管是分页还是无限滚动,后端接口的本质都是按页/按游标返回一小段数据(比如 GET /items?page=2&size=20)。区别只在于:前端用什么方式决定"什么时候请求下一段数据"、以及"怎么把它们呈现给用户"。

传统分页

把数据切成一页一页,页面上显示页码,用户点击页码或"下一页"按钮切换。

分页:每次只请求当前页,替换而非追加数据
function ProductPage() {
  const [page, setPage] = useState(1);
  const { data } = useQuery({
    queryKey: ['products', page],  // page 变化就是新的缓存 key,天然对应"翻页 = 换一页新数据"
    queryFn: () => fetchProducts(page),  // 每页数据独立,不和其他页拼接
  });

  return (
    <>
      {data?.items.map(item => <Row key={item.id} {...item} />)}
      <Pagination current={page} total={data?.totalPages} onChange={setPage} />
    </>
  );
}

优点:页码明确、可以直接跳转到第 N 页;URL 里带上页码就能被搜索引擎收录、被用户收藏和分享(SEO 友好);每页数据独立,内存占用可控,不会无限增长。

缺点:翻页会打断阅读体验——用户看到第 3 页底部,点"下一页"整个列表被替换,原来看到哪条容易丢失上下文;对于"随手往下刷"的浏览类场景(社交动态、图片流),分页的操作成本显得繁琐。

无限滚动

用户滚动到列表底部时,自动追加加载下一批数据,拼接在已有列表后面,视觉上是一个不断变长的连续列表。

无限滚动:IntersectionObserver 检测触底,追加下一页
function Feed() {
  const { data, fetchNextPage, hasNextPage } = useInfiniteQuery({
    queryKey: ['feed'],
    queryFn: ({ pageParam }) => fetchFeed(pageParam),
    getNextPageParam: (lastPage) => lastPage.nextCursor,
  });

  const bottomRef = useRef(null);

  useEffect(() => {
    // IntersectionObserver:监听"哨兵元素"是否进入视口,进入即代表滚动到了列表底部
    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting && hasNextPage) fetchNextPage();
    });
    if (bottomRef.current) observer.observe(bottomRef.current);
    return () => observer.disconnect();
  }, [hasNextPage]);

  return (
    <>
      {data?.pages.flatMap(p => p.items).map(item => <Row key={item.id} {...item} />)}
      <div ref={bottomRef} />  {/* 哨兵元素,进入视口就触发加载下一页 */}
    </>
  );
}
和 TanStack Query 的联系: 无限滚动天然对应 useInfiniteQuery——它专门管理"多页数据拼接成一个连续列表"这件事:每页数据存在 data.pages 数组里,getNextPageParam 告诉它怎么从上一页结果推算下一页的游标/页码,fetchNextPage 负责触发加载并把新数据追加进去。手写"翻页替换数据"反而更适合普通的 useQuery(参见前面 useQuery 相关篇目),因为分页不需要拼接历史数据。

优点:体验连续,适合"随手往下刷"的浏览场景,不需要用户主动做翻页操作。

缺点:难以定位到某个具体位置(没有页码可跳转);返回列表位置的处理复杂——用户点开第 200 条的详情页再返回,如果列表状态没有被正确保留(已加载的页数、滚动位置),要么重新从头加载、要么要专门做"滚动位置恢复"和"已加载页数缓存";不利于 SEO,因为大部分内容要靠 JS 滚动触发才会加载,搜索引擎爬虫不一定会执行滚动交互。

对比总结

维度传统分页无限滚动
定位/跳转支持,页码明确不支持,只能顺序加载
浏览体验翻页有割裂感连续、沉浸感强
返回列表状态保留简单,回到对应页码即可复杂,需要缓存已加载数据和滚动位置
SEO / 可分享友好,URL 带页码不友好,内容靠交互懒加载
典型场景后台管理系统、订单列表、搜索结果社交动态、图片瀑布流、内容推荐流
配套 APIuseQuery(每页独立缓存)useInfiniteQuery(多页拼接)
怎么选: 问自己一个问题——用户是"要精确定位/反复查找某条数据"(订单、工单、后台管理),还是"漫无目的地随便刷"(社交流、推荐流)?前者选分页,定位能力和状态可恢复性更重要;后者选无限滚动,连续体验优先。不要仅凭"无限滚动看起来更现代"就无脑套用到所有列表上。
容易被忽视的陷阱: 无限滚动如果不做"返回时恢复滚动位置和已加载页数",用户从详情页返回列表会看到列表被重置回第一页、之前刷到的内容全部消失,体验比分页还差。工程上通常需要把已加载的页数和滚动位置存在路由状态或全局缓存里,在列表重新挂载时用来恢复——这也是无限滚动"隐藏成本"比分页高的地方。

小结

  • 分页和无限滚动都是为了避免一次性把全部数据拉到前端,区别在于"何时请求下一批"以及"新数据是替换还是追加"。
  • 传统分页:页码明确、可跳转、SEO 友好、状态易恢复,但翻页体验割裂,适合需要精确定位的场景(后台管理、订单列表)。
  • 无限滚动:用 IntersectionObserver 检测哨兵元素是否进入视口来触发加载下一页,体验连续,适合随手浏览的场景(社交流、推荐流),但难以定位、返回列表位置的恢复逻辑更复杂,对 SEO 不友好。
  • 分页天然对应 useQuery(每页数据独立、互相替换);无限滚动天然对应 useInfiniteQuery(多页数据拼接进 data.pages)。
  • 选择依据是产品场景而非技术偏好:能不能精确定位、要不要 SEO、返回列表要不要保留状态,这些产品需求决定了该用哪种方案。
  • 虚拟滚动、分页、无限滚动三者是互补关系:分页/无限滚动控制"一次向后端要多少数据",虚拟滚动控制"拿到手的数据一次渲染多少 DOM"——大数据量列表往往需要两者结合。
