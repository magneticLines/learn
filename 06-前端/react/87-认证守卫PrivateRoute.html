认证守卫:未登录就拦截并跳转登录页

后台页面必须登录才能看。React Router 本身没有"守卫"这个概念——它只负责匹配 URL 渲染组件。所谓"路由守卫",本质是一个检查登录态、决定放行还是重定向的普通组件。

心智:守卫就是一个包装组件

不同于 Vue Router 的 beforeEach 全局钩子,React Router v6 的守卫是声明式的——你写一个组件,它内部判断:登录了就渲染真正的页面,没登录就渲染一个 <Navigate> 重定向。

RequireAuth
→
已登录 → <Outlet/>
/
未登录 → <Navigate to="/login"/>

v6 推荐写法:守卫作为布局路由包裹 Outlet

最优雅的做法是把守卫做成一个布局路由(上一章的无 path 路由):它内部判断登录态,已登录渲染 <Outlet /> 放行子路由,未登录渲染 <Navigate />。一层守卫保护一整组页面。

RequireAuth:包裹 Outlet 的守卫组件
import { Navigate, Outlet, useLocation } from 'react-router-dom';

function RequireAuth() {
  const user = useAuth();         // 从 Context 读当前登录用户
  const location = useLocation(); // 记住用户想去的地方

  if (!user) {
    // 未登录:重定向到登录页,并把来源路径塞进 state
    // replace 避免在历史里留下"被拦截的页面"
    return <Navigate to="/login" state={{ from: location }} replace />;
  }
  return <Outlet />;  // 已登录:放行,渲染子路由
}
路由配置:守卫包一组受保护页面
<Routes>
  <Route path="/login" element={<Login />} />

  {/* 守卫作为父路由,里面所有页面都需要登录 */}
  <Route element={<RequireAuth />}>
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/settings"  element={<Settings />} />
  </Route>
</Routes>

保存来源路径,登录后跳回

好的体验是:用户想访问 /settings,被拦到登录页,登录成功后自动回到 /settings,而不是丢到首页。诀窍在守卫里通过 state={{ from: location }} 把来源路径带过去,登录页读出来用它跳回。

Login 页:登录成功后读取 from 跳回
function Login() {
  const navigate = useNavigate();
  const location = useLocation();
  // 拿到守卫塞进来的来源;没有就默认回首页
  const from = location.state?.from?.pathname || '/';

  async function handleLogin() {
    await doLogin();
    // replace:登录页不该留在历史里,否则返回键又回登录页
    navigate(from, { replace: true });
  }
  // ...
}

另一种写法:守卫包裹单个 element

如果只想保护一两个页面,不必单开布局路由,可以让守卫接收 children:

接收 children 的守卫,包在 element 上
function RequireAuth({ children }) {
  const user = useAuth();
  const location = useLocation();
  if (!user) return <Navigate to="/login" state={{ from: location }} replace />;
  return children;
}

// 用法:直接包在需要保护的 element 外面
<Route path="/settings" element={
  <RequireAuth><Settings /></RequireAuth>
} />
写法适用
包裹 <Outlet />(布局路由)保护一整组页面,只写一次,推荐
接收 children零散保护个别页面,或每个页面权限不同
前端守卫只是体验,不是安全边界
路由守卫拦的是"渲染",不是"数据"。任何人都能改前端代码绕过它。真正的权限校验必须在后端做——每个受保护接口都要独立验证 token。前端守卫的意义是:未登录用户不该看到空白报错页,而应被友好地引导去登录。别把它当成安全机制。
登录态"加载中"的处理
如果登录态需要异步获取(如刷新页面后从后端校验 token),守卫里 user 初始可能是 undefined(未知)。别把"未知"当"未登录"直接踢到登录页,否则每次刷新都闪一下登录页。应区分三态:loading 显示占位、已登录 放行、未登录 才重定向。
