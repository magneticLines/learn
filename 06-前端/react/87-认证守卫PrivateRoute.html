认证守卫:未登录就拦截并跳转登录页

后台页面必须登录才能看。React Router 本身没有"守卫"这个概念——它只负责匹配 URL 渲染组件。所谓"路由守卫",本质是一个检查登录态、决定放行还是重定向的普通组件。

心智:守卫就是一个包装组件

不同于 Vue Router 的 beforeEach 全局钩子,React Router v6 的守卫是声明式的——你写一个组件,它内部判断:登录了就渲染真正的页面,没登录就渲染一个 <Navigate> 重定向。

RequireAuth
→
已登录 → <Outlet/>
/
未登录 → <Navigate to="/login"/>

v6 推荐写法:守卫作为布局路由包裹 Outlet

最优雅的做法是把守卫做成一个布局路由(上一章的无 path 路由):它内部判断登录态,已登录渲染 <Outlet /> 放行子路由,未登录渲染 <Navigate />。一层守卫保护一整组页面。

RequireAuth:包裹 Outlet 的守卫组件
import { Navigate, Outlet, useLocation } from 'react-router-dom';

function RequireAuth() {
  const user = useAuth();         // 从 Context 读当前登录用户
  const location = useLocation(); // 记住用户想去的地方

  if (!user) {
    // 未登录:重定向到登录页,并把来源路径塞进 state
    // replace 避免在历史里留下"被拦截的页面"
    return <Navigate to="/login" state={{ from: location }} replace />;
  }
  return <Outlet />;  // 已登录:放行,渲染子路由
}
路由配置:守卫包一组受保护页面
<Routes>
  <Route path="/login" element={<Login />} />

  {/* 守卫作为父路由,里面所有页面都需要登录 */}
  <Route element={<RequireAuth />}>
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/settings"  element={<Settings />} />
  </Route>
</Routes>

保存来源路径,登录后跳回

好的体验是:用户想访问 /settings,被拦到登录页,登录成功后自动回到 /settings,而不是丢到首页。诀窍在守卫里通过 state={{ from: location }} 把来源路径带过去,登录页读出来用它跳回。

Login 页:登录成功后读取 from 跳回
function Login() {
  const navigate = useNavigate();
  const location = useLocation();
  // 拿到守卫塞进来的来源;没有就默认回首页
  const from = location.state?.from?.pathname || '/';

  async function handleLogin() {
    await doLogin();
    // replace:登录页不该留在历史里,否则返回键又回登录页
    navigate(from, { replace: true });
  }
  // ...
}

另一种写法:守卫包裹单个 element

如果只想保护一两个页面,不必单开布局路由,可以让守卫接收 children:

接收 children 的守卫,包在 element 上
function RequireAuth({ children }) {
  const user = useAuth();
  const location = useLocation();
  if (!user) return <Navigate to="/login" state={{ from: location }} replace />;
  return children;
}

// 用法:直接包在需要保护的 element 外面
<Route path="/settings" element={
  <RequireAuth><Settings /></RequireAuth>
} />
写法适用
包裹 <Outlet />(布局路由)保护一整组页面,只写一次,推荐
接收 children零散保护个别页面,或每个页面权限不同
前端守卫只是体验,不是安全边界
路由守卫拦的是"渲染",不是"数据"。任何人都能改前端代码绕过它。真正的权限校验必须在后端做——每个受保护接口都要独立验证 token。前端守卫的意义是:未登录用户不该看到空白报错页,而应被友好地引导去登录。别把它当成安全机制。
登录态"加载中"的处理
如果登录态需要异步获取(如刷新页面后从后端校验 token),守卫里 user 初始可能是 undefined(未知)。别把"未知"当"未登录"直接踢到登录页,否则每次刷新都闪一下登录页。应区分三态:loading 显示占位、已登录 放行、未登录 才重定向。

常见陷阱

  • 认证态闪烁(flash of login page):刷新页面后登录态要靠异步请求(校验 token / 拉用户信息)才能确定,如果守卫把"还没拿到结果"的初始值(如 undefined)当成"未登录",会先把用户踢到登录页,等请求回来才发现其实登录了,又跳回来——页面闪一下。必须把状态拆成三态(loading / 已登录 / 未登录),loading 时渲染占位而不是直接下判断。
  • 把前端守卫当成安全边界:守卫只是控制"渲染哪个组件",本质上是纯前端逻辑,任何人打开控制台改改代码或直接调接口都能绕过去。真正的权限校验必须在后端每个接口里做,前端守卫的价值只是体验层面——不让未登录用户看到裸露的报错或空白页。
  • 忘记 replace,导致返回键死循环:<Navigate to="/login" /> 默认是 push,如果不加 replace,历史栈会变成 /settings, /login,用户点返回又回到 /settings,又被拦到 /login,形成死循环。守卫场景下的重定向几乎总该加 replace。
  • 异步校验期间的竞态:如果登录校验是异步的(比如请求校验 token 有效性),用户可能在校验完成前已经点击了别的链接或提交了表单。如果校验结果回来时不检查"当前还是不是那个页面/请求",可能会用过期的结果错误地重定向或放行,需要配合 AbortController 或"请求 id 比对"处理。
  • 忘记保存来源路径,登录后跳回首页:很多人图省事,守卫里直接 <Navigate to="/login" />,登录成功后固定跳首页,用户被迫从头点回原来想去的页面。正确做法是把 location 存进 state.from,登录页读出来跳回去。

面试高频考点

  1. React Router 有没有内置的"路由守卫"API?—— 没有。v6 完全是声明式匹配 URL 渲染组件,没有 Vue Router 那种 beforeEach 全局钩子概念。"守卫"是开发者自己用一个判断登录态、决定渲染 <Outlet /> 还是 <Navigate /> 的普通组件实现出来的模式,不是框架特性。
  2. 为什么守卫不能只是"未登录就 return null"?—— 因为那样用户会看到一片空白,不知道发生了什么,也没有引导。应该渲染 <Navigate> 主动跳转到登录页,给出明确反馈,而不是静默地什么都不显示。
  3. 包裹 <Outlet /> 和接收 children 两种守卫写法有什么区别,该怎么选?—— 包裹 Outlet 的写法是把守卫做成父路由,一次性保护一整组子路由,适合"一大片页面统一要求登录"的场景,代码只写一次;接收 children 的写法要在每个受保护页面外面手动包一层,适合零散保护、或不同页面需要不同权限判断的场景。
  4. 为什么守卫里要区分登录态的 loading / 已登录 / 未登录三态,而不是用一个布尔值?—— 因为登录态往往要靠异步请求确认(刷新页面后重新校验 token)。如果只用布尔值,初始值必须是 true 或 false 中的一个,而这两个值都不代表真实情况,会导致误判——把"还不知道"当成"未登录"从而错误跳转,造成页面闪烁。三态能让 UI 在不确定时显示占位,等结果确定后才做出正确判断。

与其他知识点的联系

  • 布局路由 / Outlet 占位符：推荐写法里的 RequireAuth 本质就是一个"逻辑布局路由"——它不提供视觉外壳，只做鉴权判断，判断通过后渲染 <Outlet /> 放行子路由。不理解布局路由和 Outlet 的组合关系，就很难看懂为什么守卫可以像普通父路由一样包裹一组页面。
  • 路由 state 传参：守卫拦截时用 state={{ from: location }} 把来源路径带给登录页，这是路由 state 最典型的实战场景——数据不需要出现在 URL 上，只是登录成功后"跳回原处"的临时线索，刷新页面丢了也无妨（用户重新走一遍登录流程即可）。
  • Navigate 重定向：守卫的拦截动作就是一次 <Navigate to="/login" replace />，本文强调的"必须加 replace 避免返回键死循环"正是 Navigate 重定向那篇讲的核心陷阱——两者结合起来才能保证被拦截的页面不会赖在浏览器历史里。
  • 404 页面：守卫和 404 都是"路由渲染前的额外判断"，区别在于判断维度不同——404 判断"URL 是否存在对应路由"，守卫判断"用户是否有权限看这个已存在的路由"。一个页面理论上可能同时经过两层检查：先匹配到路由，再被守卫拦下。
  • createBrowserRouter 与 loader：数据路由下，鉴权判断也可以放进 loader 里做——loader 发现未登录时 throw redirect('/login')，效果等同于组件里渲染 <Navigate>，但判断时机提前到了"渲染前"，避免了组件先渲染一瞬又被重定向的闪烁问题。
