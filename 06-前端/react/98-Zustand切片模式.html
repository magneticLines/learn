Zustand 切片模式:把大 store 拆成 slice

随着应用变大,一个 store 里塞进认证、购物车、UI 偏好……几百行挤在一个 create 里,谁也读不下去。切片模式(slices pattern)让你把大 store 按业务领域拆成多个独立文件,再组合成一个 store——模块化,又保留了"单一 store"的便利。

问题:单文件 store 会膨胀

回顾一下:一个 create(set => ({...})) 里,state 和 action 全写在一起。领域一多,这个对象就会变成几百行的巨物,不同业务(登录、购物车、通知)的代码搅在一起,难维护、难协作。

为什么不干脆建多个独立 store? 也可以——不同领域完全无关时,拆成 useAuthStore、useCartStore 几个独立 store 是最简单的做法。切片模式解决的是另一种需求:你想保留一个 store(比如 slice 之间需要互相调用、或想统一持久化),但又想让代码按领域分文件组织。两条路都对,按耦合度选。

核心思路:slice 是"接收 set/get,返回一部分 state"的函数

一个 slice 就是把原来 create 回调里的一段抽出来,单独写成一个函数。它签名和 create 的回调完全一样 (set, get) => ({...}),只不过只返回自己负责的那部分:

slices/authSlice.js — 每个 slice 一个文件
// slice 是一个函数:签名和 create 的回调一样 (set, get) => ({...})
// 只返回自己这块 state + action
export const createAuthSlice = (set) => ({
  user: null,
  login: (user) => set({ user }),
  logout: () => set({ user: null }),
});
slices/cartSlice.js
export const createCartSlice = (set, get) => ({
  items: [],
  addItem: (item) => set((s) => ({ items: [...s.items, item] })),
  clear: () => set({ items: [] }),

  // slice 之间可通过 get() 跨领域访问:结账时读 user
  checkout: () => {
    const user = get().user;  // 访问 authSlice 的 user,因为它们同在一个 store
    if (!user) return alert('请先登录');
    // ...提交订单
  },
});

组合:在 create 里把 slice 展开合并

store/index.js — 组合所有 slice 成一个 store
import { create } from 'zustand';
import { createAuthSlice } from './slices/authSlice';
import { createCartSlice } from './slices/cartSlice';

export const useStore = create((set, get) => ({
  // 把 set/get 透传给每个 slice,再把返回的对象展开合并到一起
  ...createAuthSlice(set, get),
  ...createCartSlice(set, get),
}));

// 组件里用起来毫无区别:还是一个 store,一个选择器
// const user  = useStore(s => s.user);   // 来自 authSlice
// const items = useStore(s => s.items);  // 来自 cartSlice

关键点:所有 slice 共享同一个 set / get。因为它们最终被展开进同一个对象,get() 拿到的是合并后的完整 state——所以 cartSlice 的 checkout 才能读到 authSlice 的 user。

陷阱:slice 之间的字段名不能撞车。 因为最终是展开(...)合并到一个扁平对象里,如果两个 slice 都有一个叫 loading 的字段,后展开的会覆盖前面的,悄无声息地出 bug。约定:要么字段名带领域前缀(cartLoading),要么保证全局唯一。

set({ ... }) 的浅合并也是全局的——它合并的是整个 store 的顶层,不是"只在本 slice 内"。心里要清楚这是一个扁平的大对象。

什么时候用切片,什么时候用多 store

场景推荐
领域之间完全无关,互不调用多个独立 store
slice 之间要互相读/调切片模式(共享 set/get)
想统一持久化 / 统一中间件切片模式(一个 store 挂一次中间件)
只是嫌一个文件太长切片模式,纯为分文件
TypeScript 提示: 用 TS 时给组合 store 定义一个总类型 type Store = AuthSlice & CartSlice(交叉类型),每个 slice 用 StateCreator<Store, [], [], AuthSlice> 标注。这样 slice 内部 get() 能拿到整个 store 的类型提示,跨 slice 访问也有类型安全。
另外:切片模式和 Redux Toolkit 的 createSlice 名字像但不是一回事——RTK 的 slice 会自动生成 action/reducer,是下一模块的内容。

小结

  • 切片模式把一个大 store 按业务领域拆成多个 slice 文件,再在 create 里展开合并成单一 store。
  • slice 就是一个 (set, get) => ({...}) 函数,只返回自己负责的那块 state 与 action。
  • 所有 slice 共享同一个 set/get,因此 get() 能读到合并后的完整状态,slice 之间可跨领域调用。
  • 陷阱:最终是扁平合并,字段名不能重复,否则后者覆盖前者;set 的合并是全局顶层。
  • 领域彻底无关就用多个独立 store;要互相访问、统一持久化/中间件、或只是想分文件,就用切片模式。

与其他知识点的联系

  • Zustand 创建 store(第 96 篇)与读取更新(第 97 篇):切片模式不是新概念,而是把前两篇学到的 create((set, get) => ({...})) 这一个函数,拆成多个签名一致的小函数再组合起来——slice 本身就是"一段 create 回调"的抽出,理解前两篇的 set/get 用法是读懂本篇的前提。
  • 本地 state 与全局 state(第 93 篇):切片模式解决的问题和状态提升要解决的问题在本质上相通——都是"如何组织不断增长的状态,让代码保持可维护"。状态提升是组件树维度的组织方式,切片模式是同一个 store 内部按业务领域的组织方式。
  • Context 与 useReducer 轻量方案(第 95 篇):本篇提示"切片模式和 Redux Toolkit 的 createSlice 名字像但不是一回事",而 Context+useReducer 的 reducer 集中管理逻辑的思路,恰恰是 Redux 的 reducer 模式在 React 内置能力里的简化版——三者(useReducer、Zustand slice、Redux Toolkit slice)都是"把状态变更逻辑按某种边界拆分"的不同实现,值得放在一起对比理解命名相似但语义不同的地方。
  • 服务端状态与客户端状态(第 94 篇):切片模式让 store 结构复杂化的门槛更低,也更容易让人"顺手"把服务端数据也塞进某个 slice——这时更要提醒自己回顾第 94 篇的边界:即便切片再方便,服务端数据依然不该进 Zustand,该拆的是客户端状态本身的业务领域,不是把数据来源也混进来。
  • Zustand 持久化 persist(第 99 篇):本篇提到"想统一持久化"是选择切片模式而非多个独立 store 的一个理由——因为 persist 中间件是包在整个 store 外层的,只有当所有领域数据都在同一个 store 里,才能用一次 persist 配置统一处理,这也是下一篇要讲的内容。
