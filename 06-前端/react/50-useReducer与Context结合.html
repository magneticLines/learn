React · useReducer

useReducer + Context：轻量级全局状态

把两个工具拼起来会产生化学反应：useReducer 负责"如何管理复杂状态"，Context 负责"把状态送到任意深处"。二者组合，就能不引入任何第三方库，实现一套可用的全局状态管理。

思路：state 和 dispatch 都下发下去

在顶层用 useReducer 拿到 [state, dispatch]，然后通过 Context 把它们广播给整棵树。任意后代既能读 state，也能 dispatch action 来更新——完全绕开 prop drilling。

顶层 useReducer
→
Context 广播 state + dispatch
→
任意深层组件读/派发

推荐做法：拆成两个 Context

回顾第 47 篇的优化——把 state 和 dispatch 分成两个 Context。dispatch 引用永远稳定，所以"只派发不读取"的组件（比如一个"添加"按钮）不会因为 state 变化而重渲。这是这个模式性能上的关键。

store.jsx —— 定义 Provider 和便捷 Hook
import { createContext, useContext, useReducer } from 'react';

const StateCtx = createContext(null);
const DispatchCtx = createContext(null);

function reducer(state, action) {
  switch (action.type) {
    case 'add':    return { ...state, todos: [...state.todos, action.todo] };
    case 'clear':  return { ...state, todos: [] };
    default:       return state;
  }
}

export function StoreProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, { todos: [] });
  return (
    // dispatch 在外层：它引用稳定，包住它的这层不随 state 变
    <DispatchCtx.Provider value={dispatch}>
      <StateCtx.Provider value={state}>{children}</StateCtx.Provider>
    </DispatchCtx.Provider>
  );
}

// 封装成自定义 Hook：使用处更干净，还能校验有没有套 Provider
export const useTodos = () => useContext(StateCtx);
export const useTodosDispatch = () => useContext(DispatchCtx);

使用：读的组件和派发的组件互不拖累

消费端
// 只读列表：state 变才重渲
function TodoList() {
  const { todos } = useTodos();
  return todos.map(t => <li key={t.id}>{t.text}</li>);
}

// 只派发：dispatch 稳定，这个按钮永不因 todos 变化重渲
function AddButton() {
  const dispatch = useTodosDispatch();
  return <button onClick={() => dispatch({ type: 'add', todo })}>加一条</button>;
}
把 useContext 封装成 useTodos() 这样的自定义 Hook，好处有三：使用处不用 import 具体 Context、可以在 Hook 内加"没套 Provider 就报错"的校验、将来换实现（比如换成 Zustand）时使用方代码不用动。这是社区非常常见的封装。

与专门状态库的边界

这套组合能扛住相当多的场景，但它不等于 Redux/Zustand。核心差距在于：Context 没有 selector——state 一变，所有订阅 StateCtx 的组件都会重渲，无法"只订阅我关心的那个字段"。

useReducer + ContextZustand / Redux
额外依赖无(React 内置)需装库
细粒度订阅(selector)没有有
大 state 频繁更新易触发大面积重渲只更新订阅了的组件
DevTools/中间件无有
适合规模中小型 / 局部子树中大型全局
什么时候该升级到专门状态库？当出现这些信号：state 树很大且更新频繁、明显因 Context 重渲导致卡顿、需要跨路由持久化/时间旅行调试、多个团队协作需要规范的数据流。在此之前，useReducer + Context 通常够用，且没有额外依赖和学习成本。不要一上来就 Redux。

小结

  • 组合模式：顶层 useReducer + Context 广播，实现零依赖的轻量全局状态。
  • 拆成 State Context 和 Dispatch Context，让"只派发"的组件免于 state 重渲。
  • 封装成 useXxx() 自定义 Hook，隐藏 Context 细节、便于校验和替换实现。
  • 边界：没有 selector、大 state 频繁更新会大面积重渲——到那个规模再考虑 Zustand/Redux。

常见陷阱

  • 只用一个 Context 同时装 state 和 dispatch：写成 <StoreCtx.Provider value={{ state, dispatch }}>，即便 dispatch 引用稳定，只要 state 变了，整个 value 对象就是新引用，所有消费者（包括只想 dispatch 的组件）全部跟着重渲。拆成两个 Context 才能真正发挥 dispatch 引用稳定的优势。
  • value 里包一层新对象，抵消了拆分的效果：即使拆了两个 Context，如果还写 <StateCtx.Provider value={{ state }}> 多包一层对象，这层包装对象本身每次渲染都是新引用，等于白拆。应该直接把 state/dispatch 本身作为 value，不要再包一层。
  • 忘记给自定义 Hook 做"是否在 Provider 内"的校验：useContext 拿到默认值 null 后没有判断就直接用，导致组件树忘记套 StoreProvider 时，报错发生在很深的调用栈里，定位起来比在 Hook 入口直接 throw new Error('必须在 StoreProvider 内使用') 麻烦得多。
  • 把这套组合当成万能替代品，滥用到大型全局状态：state 树一大、更新一频繁，没有 selector 的 Context 会导致大面积无谓重渲，反而比引入一个专门状态库的心智负担更高。这个模式的甜蜜区是中小型、局部子树的状态共享，不是整个大型应用的状态管理方案。

面试高频考点

  1. 为什么要把 state 和 dispatch 拆成两个 Context？—— dispatch 由 useReducer 保证引用永远稳定，不随 state 变化而改变；把它单独放一个 Context，"只派发不读取"的组件（比如一个添加按钮）就不会因为 state 变化被连带重渲。若混在一起，state 变化会导致 value 整体是新引用，所有消费者无差别重渲。
  2. 这套组合和 Redux 最本质的差距是什么？—— 缺少 selector 机制。Redux/Zustand 允许组件只订阅 store 里自己关心的那一小块数据，其他字段变化不会引起该组件重渲；而 Context 的订阅粒度是整个 value，只要引用变就通知所有消费者，无法做细粒度订阅。这是决定"什么时候该升级到专门状态库"的核心依据。
  3. 把 useContext 封装成自定义 Hook（如 useTodos）有什么工程价值？—— 三点：调用处不必知道具体 Context 对象、可以在 Hook 内部加"没套 Provider 就报错"的运行时校验、未来更换底层实现（比如从 Context 换成 Zustand）时，只需改 Hook 内部实现，所有调用方代码不用动，起到隔离变化的作用。
  4. 什么信号提示该从 useReducer + Context 升级到 Redux/Zustand？—— state 树体量大且更新频繁、Profiler 已经证实由 Context 重渲引起卡顿、需要跨路由持久化或时间旅行调试、多团队协作需要统一规范的数据流约定。在这些信号出现之前强行引入额外状态库，属于过度设计。

与其他知识点的联系

  • 引用稳定性问题（第 51 篇）：这套组合能拆成 State/Dispatch 两个 Context，根本原因是 useReducer 返回的 dispatch 天然引用稳定——这正是下一篇要系统讲的"引用稳定性"概念在这里的具体应用。理解了引用稳定性，才能明白为什么"只派发"的组件能免于 state 变化引起的重渲。
  • useMemo 缓存计算（第 52 篇）：如果 Context 的 value 不是像本篇这样直接传 state/dispatch，而是需要传一个额外派生的对象，就该用 useMemo 稳定这个对象的引用，避免每次渲染都造出新对象让所有消费者跟着重渲——这是第 61 篇跨层级 Context 通信里也会强调的同一个坑。
  • 抽离复用逻辑（第 56 篇）：本篇把 useContext(StateCtx) 封装成 useTodos() 自定义 Hook，思路和第 56 篇的 useFetch/useLocalStorage 完全一致——都是把一段固定的状态访问逻辑打包，让调用处更干净、还能加运行时校验。区别是这里复用的是"访问方式"而不是"独立状态"。
  • 跨层级 Context 通信（第 61 篇）：本篇是第 61 篇 Context 基础用法的进阶实战——不仅广播只读数据，还把修改数据的能力（dispatch）一起广播下去，相当于把第 59 篇"子传父回调"的模式从"父子之间"扩展到了"祖先与任意深度后代之间"。
  • 兄弟组件状态提升（第 60 篇）：可以把 useReducer + Context 理解成"状态提升"思路的终极形态——第 60 篇把 state 提升到最近的共同父组件，本篇则相当于把这个"共同父组件"换成了一个可以被任意深度访问的 Context Provider，解决的是提升后依然要一层层 props 透传的问题。
  • 过度优化的坏处（第 54 篇）：本篇结尾强调"没到规模别急着上 Redux"，这和第 54 篇"先测量再优化"的原则是同一种工程态度——不要为想象中的问题引入额外复杂度，无论是手动 memo 还是引入状态管理库，都应该等到有实际信号才升级。
