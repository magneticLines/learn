Context + useReducer:不装库的轻量全局状态

确认了某份客户端状态需要跨很多组件共享,第一反应不必是"装 Zustand"。React 自带的 useReducer + Context 就能拼出一个够用的轻量全局状态,零依赖。这一篇讲怎么拼、适用多大规模,以及它的性能边界在哪——什么时候该换 Zustand。

思路:useReducer 管逻辑,Context 管分发

把前面学过的两个 Hook 组合起来,各管一摊:

  • useReducer:集中管理"状态怎么变"——所有更新逻辑收敛到一个 reducer 函数,组件只负责 dispatch 一个动作。
  • Context:把 state 和 dispatch 广播给整棵子树,任意深度的组件都能拿到,不用逐层传 props。
useReducer
→

完整实现:一个主题 + 用户的全局 store

store.jsx — reducer + Context + Provider + 自定义 Hook
// ① reducer:所有状态变更逻辑集中在这里,纯函数好测
function reducer(state, action) {
  switch (action.type) {
    case 'toggleTheme':
      return { ...state, theme: state.theme === 'light' ? 'dark' : 'light' };
    case 'login':
      return { ...state, user: action.payload };
    default: return state;
  }
}

// ② 两个 Context:state 和 dispatch 分开(下面讲为什么要拆)
const StateCtx = createContext(null);
const DispatchCtx = createContext(null);

// ③ Provider:把 state / dispatch 广播下去
export function StoreProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, { theme: 'light', user: null });
  return (
    <DispatchCtx.Provider value={dispatch}>
      <StateCtx.Provider value={state}>
        {children}
      </StateCtx.Provider>
    </DispatchCtx.Provider>
  );
}

// ④ 自定义 Hook:消费方不直接碰 Context,读起来更干净
export const useStoreState = () => useContext(StateCtx);
export const useDispatch = () => useContext(DispatchCtx);
组件里消费:dispatch 动作即可
function ThemeToggle() {
  const { theme } = useStoreState();
  const dispatch = useDispatch();
  return <button onClick={() => dispatch({ type: 'toggleTheme' })}>
    {theme}
  </button>;
}
为什么把 state 和 dispatch 拆成两个 Context? 因为 dispatch 的引用永远不变。只需要派发动作、不读取状态的组件(比如一个纯"新增"按钮),订阅 DispatchCtx 后就永远不会因为 state 变化而重渲染。这是 Context 方案里最实用的一个性能技巧。

适用规模:什么时候够用

这套方案在下列情况下完全够用,别急着上第三方库:

  • 共享的状态不大、更新不频繁(主题、语言、登录用户、权限)。
  • 消费该状态的组件数量有限。
  • 团队不想引入额外依赖,喜欢"只用 React 自带能力"。

性能边界:Context 的致命短板

Context 有一个绕不过去的机制:只要 Provider 的 value 变了,所有 useContext 订阅它的组件都会重渲染——无论它实际用到的是 value 里的哪个字段。

Context 无法做"细粒度订阅"。 假设 store 里有 { theme, user, cart },只改了 cart,那么只用 theme 的组件也会跟着重渲染——因为它订阅的是整个 value 对象,而 value 引用变了。状态一大、更新一频繁,页面上一堆无关组件被反复重渲,性能就崩了。
问题演示:只改 cart,订阅 theme 的组件也白白重渲
// value = { theme, user, cart } 是一个对象
// dispatch 改了 cart → reducer 返回新对象 → value 引用变
// 于是所有 useContext(StateCtx) 的组件全部重渲:

ThemeToggle    // 只用 theme,却被 cart 变化连累重渲 ❌

CartIcon       // 用 cart,重渲是应该的 ✅

// Context 没有"我只关心 state.cart"这种选择性订阅能力

该换 Zustand 的信号

  • store 里字段越来越多,且更新频繁(如输入、拖拽、实时数据)。
  • 你开始为了避免无关重渲,拆出一大堆细分 Context,Provider 套了好多层,代码变丑。
  • 你需要在组件外部(如普通函数、事件回调)读写状态——Context 只能在组件内 useContext。

Zustand 天生支持选择器(selector):组件可以声明"我只关心 cart",cart 不变就不重渲——这正是 Context 缺的那块。下一篇开始进入 Zustand。

对比记忆: Context 是 React 的依赖注入 / 广播机制,它本职是"把值传下去",不是为高频状态管理设计的。把它当全局状态库用没问题,但要清楚它没有选择性订阅。Zustand/Redux 这类库的核心价值之一,就是补上"细粒度订阅、精确重渲"这一环。

小结

  • useReducer 集中更新逻辑,Context 负责广播,两者拼出零依赖的轻量全局状态。
  • 把 state 与 dispatch 拆成两个 Context,只派发的组件可免于状态变化引起的重渲。
  • 适用于小而稳定的共享状态(主题、语言、登录态),此时不必引第三方库。
  • 性能边界:Context 的 value 一变,所有订阅者全部重渲,没有细粒度订阅——状态大且更新频繁时会拖垮性能。
  • 当出现频繁更新、Provider 层层嵌套、需要在组件外读写状态时,就该换支持选择器的 Zustand。

常见陷阱

  • 把 state 和 dispatch 塞进同一个 Context,白白扩大重渲染范围:如果不像文中那样拆成 StateCtx 和 DispatchCtx 两个 Context,那么只需要 dispatch 动作、不关心当前状态的组件(比如一个纯粹的"新增"按钮)也会因为 state 变化被强制重渲染,即使它根本没读取任何 state 字段。
  • 误以为拆开 Context 就能解决所有粒度问题:即便拆出了 StateCtx,只要 store 里字段变多(比如 { theme, user, cart }),任何一个字段变化都会让整个 state 对象引用改变,所有订阅 StateCtx 的组件依然会无差别重渲染——拆 dispatch 只解决了"纯派发型组件"的问题,没解决"只关心某个字段的读取型组件"的问题。这正是 Context 缺乏细粒度订阅(selector)能力的体现。
  • 在 reducer 里直接发起异步请求或调用副作用:reducer 应该是纯函数,只根据 (state, action) 计算出新的 state,不应该在里面直接 fetch 或操作 DOM。正确做法是在组件或专门的地方先完成异步操作,拿到结果后再 dispatch 一个描述"发生了什么"的动作(如 dispatch({ type: 'fetchSuccess', payload })),reducer 只管同步地根据这个动作算出新状态。
  • 状态和更新逻辑越做越大却不愿意迁移,硬撑着用 Context:文中提到的信号——字段变多、更新频繁、Provider 层层嵌套、需要在组件外读写状态——出现后仍坚持用 Context+useReducer 硬扛,会导致代码里出现大量为了"减少重渲染范围"而拆出来的细分 Context,可读性和维护成本急剧上升,这时其实早就该换成支持选择器的 Zustand。

面试高频考点

  1. 为什么要把 state 和 dispatch 拆成两个独立的 Context?—— 因为 dispatch 函数的引用在组件生命周期内是恒定不变的,而 state 每次更新都会产生新的引用。如果两者合并在同一个 Context 里,任何 state 变化都会连带让"只需要 dispatch、不读取 state"的组件重渲染。拆开后,这类组件订阅 DispatchCtx 即可完全避免因 state 变化引发的重渲染。
  2. Context 方案相比 Zustand/Redux,最大的性能短板是什么?—— 缺乏细粒度订阅(selector)能力。Context 的更新单位是整个 value 对象——只要 Provider 传下去的 value 引用变了,所有 useContext 订阅它的组件都会重渲染,不管它们实际读取的是 value 里的哪个字段。而 Zustand 等库支持"组件只声明自己关心的那部分状态",该部分不变就不重渲染,从根本上避免无关重渲染。
  3. 什么时候该从 Context+useReducer 换成 Zustand 这类状态库?—— 几个典型信号:store 字段越来越多且更新频繁(如实时数据、拖拽);为了避免无关重渲染,不得不拆出一大堆细分 Context,导致 Provider 层层嵌套、代码变得难以维护;需要在组件外部(普通函数、事件回调里,而非通过 Hook)读写全局状态,而 Context 只能在组件内部通过 useContext 访问。
  4. 为什么不该在 reducer 函数里直接执行异步操作(比如 fetch)?—— reducer 的设计契约是"给定同样的 state 和 action,永远返回同样的新 state"(纯函数),这样才能被可预测地测试、被 React 在某些场景下安全地多次调用(如严格模式下的重复调用检测)。异步操作有不确定的完成时间和结果,混入 reducer 会破坏这个契约。正确模式是在组件或副作用里完成异步逻辑,再用 dispatch 一个同步描述结果的 action 交给 reducer 处理。

与其他知识点的联系

  • 本地 state 与全局 state(第 93 篇):Context+useReducer 正是第 93 篇路径图上的终点——"本地 useState → 状态提升到公共父 → props drilling 严重才上 Context/状态库"。本篇实现的就是这条路径最后一步的具体做法。
  • 服务端状态与客户端状态(第 94 篇):本篇反复强调"这套方案管的是客户端状态",这个边界正是上一篇划定的——服务端数据不该塞进这里,否则又要重新手写缓存、失效、竞态处理这一整套服务端状态的难题。
  • Zustand 创建 store(第 96 篇):本篇末尾指出的 Context 性能边界(无选择性订阅、需要在组件外读写状态),恰好就是 Zustand 要解决的问题——下一篇会看到 Zustand 用 selector 实现了 Context 做不到的"只有用到的字段变化才重渲染"。
  • Zustand 读取与更新状态(第 97 篇):本篇拆开 state/dispatch 两个 Context 来"手动优化重渲染范围",这种手工优化在 Zustand 里被 selector 机制原生取代——第 97 篇里 useStore(s => s.x) 的写法,本质上是把本篇要靠拆 Context 才能做到的效果,变成了框架自带的默认行为。
  • reducer 的纯函数约束与 Zustand 的 set/get:本篇强调 reducer 必须是纯函数、异步操作要挪到组件外,这条约束在 Zustand 里同样存在——action 函数可以直接调用 set/get 而不必套一层 reducer,但"不该在更新状态的同步逻辑里掺入不确定的异步过程"这一设计原则是相通的。
