CSS transition / keyframes 在 React 中的使用

React 不需要专门的动画库才能做动画——大多数简单交互(悬浮变色、展开收起、淡入淡出)靠state 控制 className触发 CSS transition 就够了。这一篇讲最朴素的做法,也埋下一个后面会反复遇到的坑:条件渲染的元素没法播放退出动画。

最朴素的方式:state 切换 className 触发 transition

CSS transition 的原理是:当一个可过渡的 CSS 属性(如 opacity、transform)的值发生变化时,浏览器自动在两个值之间插值、平滑过渡,而不是瞬间跳变。React 要做的只是让某个 CSS 类在合适的时机被加上或去掉,浏览器接管剩下的插值计算。

点击展开/收起面板,用 state 控制 className
function Accordion() {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen(!open)}>切换</button>
      <div className={`panel ${open ? 'panel-open' : ''}`}>
        面板内容
      </div>
    </div
  );
}
CSS:声明"过渡什么属性、用多久、什么缓动曲线"
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.3s ease;  /* 声明:max-height 变化时,0.3秒内平滑过渡 */
}
.panel-open {
  max-height: 300px;  /* className 一变,浏览器自动从 0 插值到 300px */
}
核心心智模型: React 只负责"在正确的时机切换 className"这一件事,动画本身完全交给浏览器的 CSS 引擎去插值计算。这和 React 声明式的哲学是一致的——你描述"状态对应什么样子",不用命令式地写"从 0px 每帧加多少像素"。

keyframes:多阶段的循环/入场动画

transition 只能在两个状态之间过渡;如果动画有多个阶段(比如"淡入的同时从下往上滑入"),或者需要无限循环(加载中的旋转图标),就需要 @keyframes 定义完整的动画序列。

@keyframes:定义多阶段动画,配合 animation 使用
@keyframes slideUpFade {
  from { opacity: 0; transform: translateY(12px); }
  to   { opacity: 1; transform: translateY(0); }
}
.toast {
  animation: slideUpFade 0.25s ease-out;  /* 元素一出现就自动播放这段动画,不需要 JS 触发 */
}
@keyframes spin {
  to { transform: rotate(360deg); }
}
.spinner { animation: spin 1s linear infinite; }  /* infinite:无限循环 */
transition
两个状态之间的平滑过渡
需要触发条件(属性值变化)
适合:悬浮、展开收起、
颜色/位置切换
@keyframes + animation
定义多阶段的完整动画序列
元素一出现就能自动播放
适合:循环动画(loading)、
入场动画、复杂多步骤动画

陷阱:条件渲染的元素没有退出动画

入场动画(元素刚出现时播放)很好实现,因为元素从"不存在"变成"存在于 DOM 中",@keyframes 会自动播放。但退出动画完全是另一回事——试试这个看似合理的写法:

反例:条件渲染不会播放退出动画
function Toast({ show }) {
  return show && <div className="toast toast-exit">提交成功</div>;
}
// show 从 true 变成 false 的瞬间,React 直接把这个 div 从 DOM 里移除
// 不管 CSS 里给 .toast-exit 写了多绚丽的 transition,都来不及播放——
// 因为播放动画需要"元素在 DOM 中停留一段时间",而它已经被卸载了
为什么会这样? React 的条件渲染 {show && <div />} 是瞬间的——show 一变成 false,React 在下一次渲染直接把这个元素从真实 DOM 树里摘除,浏览器根本没有机会启动 transition 的插值过程(CSS 过渡需要元素在两次样式计算之间都存在于文档中)。这是初学者做"弹窗关闭动画""列表项删除动画"时最常踩的坑:改了半天 CSS,发现退出动画根本没有播放过。
朴素的临时解法(仅作对比,不推荐长期用): 用 setTimeout 延迟卸载——先把 className 切换成"退出中"的样式播放动画,等动画时长过去了再真正把 state 设成不渲染。这样能工作,但代码里会散落大量和动画时长强耦合的定时器,难以维护、容易和真实 CSS 时长不同步。下一篇要介绍的 Framer Motion 的 AnimatePresence 就是为了系统性解决这个问题而生的——它会自动检测退出动画结束的时机,不需要手写定时器。

小结

  • React 做简单动画不需要额外的库:用 state 控制 className 触发 transition,浏览器负责插值计算,React 只负责"什么时候切换类名"。
  • transition 适合两个状态间的平滑过渡;@keyframes + animation 适合多阶段序列或需要无限循环的动画(如 loading 图标)。
  • 核心陷阱:条件渲染 {show && <div/>} 会瞬间把元素从 DOM 移除,退出动画没有时间播放,这是做弹窗/列表删除动画时最常见的翻车点。
  • 朴素的 setTimeout 延迟卸载能临时绕过这个问题,但代码和动画时长强耦合、难维护。
  • 下一篇 Framer Motion 引入声明式动画写法,之后的 AnimatePresence 专门解决这里留下的"退出动画播放不出来"的问题。

与其他知识点的联系

  • 与下一篇 Framer Motion 的关系: 本篇是动画方案的起点——纯 CSS 能做的事 Framer Motion 也能做,但一旦动画需要和 JS 计算的动态值、手势交互深度联动,纯 CSS 就会变得笨拙(比如要用 CSS 变量硬凑动态值)。这正是引入 JS 动画库的动机,而不是"CSS 不够用就换新工具"的盲目升级。
  • 与 React 条件渲染的关系: 纯 CSS 动画依赖"元素在 DOM 中停留一段时间"才能播放 transition,但组件被条件渲染卸载时是瞬间移除,CSS 引擎根本没有插值的窗口。这是本篇留下的核心陷阱,也是下下篇 AnimatePresence 要系统性解决的问题的"原生版本"。
  • 与 useState 触发重渲染的关系: className 的切换本质上是 state 变化触发组件重渲染,React 只负责"渲染出正确的 class 字符串",动画插值完全交给浏览器——这和 React 声明式的核心哲学(描述状态对应的样子,而不是命令式地控制过程)是一致的。
  • 与列表渲染/key 的关系: 如果给列表项做入场动画(比如新增一项时淡入),依赖的仍是 React 对列表项的挂载判断,而挂载判断本身依赖 key 是否稳定——这个坑会在后面 AnimatePresence 篇的列表删除场景里进一步放大。
