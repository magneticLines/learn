Radix UI 无样式 Headless 组件

上一篇说 shadcn/ui 的底层依赖是 Radix UI——这一篇拆开来看:Radix UI 到底是什么?为什么一个"不管样式"的组件库反而是行业标准级的存在?答案在于它解决的是最难、最容易被低估的那部分:交互和无障碍。

Headless UI:只管"怎么用",不管"长什么样"

试着自己实现一个 Dropdown(下拉菜单):点击触发按钮打开菜单,点击菜单外部关闭,按 Esc 关闭,按方向键在菜单项间移动焦点,按 Enter 选中,菜单打开时焦点要"困"在菜单内(不能 Tab 到背景元素上),屏幕阅读器要正确播报"这是一个菜单""当前选中第 2 项,共 5 项"。这些全做对,比看起来难得多。

Headless(无头)组件的思路是:把这些"正确的交互行为 + 无障碍属性"封装成组件逻辑,但完全不带任何样式——长什么样、用什么颜色、什么动画,统统留给你用 CSS(或 Tailwind)决定。Radix UI 就是这类库的代表:它导出的 Dialog、DropdownMenu、Popover 等组件,渲染出来默认是"裸的",没有边框没有阴影,但键盘导航、焦点管理、ARIA 属性全部是对的。

Radix 组件
=
正确的交互行为 + ARIA
+
你自己的样式(CSS / Tailwind)

基本用法:Radix Dialog

Dialog:交互和无障碍是内置的,样式是你自己写的
import * as Dialog from '@radix-ui/react-dialog';

function ConfirmDialog() {
  return (
    <Dialog.Root>
      <Dialog.Trigger className="btn">删除</Dialog.Trigger>

      {/* Portal:把弹窗渲染到 body 下,避免被父级 overflow:hidden 裁剪 */}
      <Dialog.Portal>
        {/* Overlay:背景遮罩,Radix 自动处理点击遮罩关闭 */}
        <Dialog.Overlay className="overlay" />
        {/* Content:焦点自动困在这里,Esc 自动关闭,全是内置行为 */}
        <Dialog.Content className="content">
          <Dialog.Title>确认删除?</Dialog.Title>
          <Dialog.Description>此操作不可撤销</Dialog.Description>
          <Dialog.Close className="btn">取消</Dialog.Close>
        </Dialog.Content>
      </Dialog.Portal>
    </Dialog.Root>
  );
}
// className 全是自己起的名字,配什么样式完全自己决定;
// 但打开/关闭、焦点管理、Esc 关闭、ARIA 属性,一行 JS 都不用写
组合式 API 是 Radix 的另一个设计亮点。 Dialog.Root/Trigger/Content/Title 这种拆成多个子组件、通过 Context 内部通信的写法,你可以只用需要的部分、自由排列结构,而不是被一个大而全的 <Modal title="..." visible={...}> 单组件 API 锁死。这也是很多现代组件库(包括 shadcn/ui 生成的组件)采用的设计模式。

和上一篇的联系:它是 shadcn/ui 的地基

回顾第 139 篇——shadcn/ui 生成的组件源码里,样式部分是 Tailwind 类名,而行为部分往往就是包了一层 Radix UI 组件。也就是说 shadcn/ui 并没有重新发明"怎么做一个正确的 Dialog",它复用了 Radix 已经做对的交互逻辑,自己只负责"贴上一套好看的默认样式,并把这份组合好的代码交给你"。理解了 Radix,就理解了 shadcn/ui 组件源码里那些看起来陌生的 Root/Trigger/Content 结构从哪来的。

常见误区:以为 Radix UI 是"又一个组件库",可以像 antd 一样直接拿来当 UI 用。 它默认渲染出来几乎没有样式(可能就是一个没有任何视觉修饰的 div),直接用会显得"很丑"或者布局错乱——这是设计使然,不是 bug。它的定位是行为层,永远需要你自己补一层样式(手写 CSS、Tailwind,或者直接用 shadcn/ui 帮你补好的那一层)。

小结

  • Headless(无样式)组件只负责正确的交互行为和无障碍支持(键盘导航、焦点管理、ARIA),样式完全交给使用者。
  • 解决的核心问题:一个正确的 Dropdown/Dialog 涉及大量细节(焦点陷阱、Esc 关闭、屏幕阅读器播报),自己重新实现代价很高,容易漏掉边界情况。
  • Radix UI 采用组合式 API(Root/Trigger/Content 等子组件 + Context 通信),比传统"大而全单组件"API 更灵活。
  • Radix UI 是 shadcn/ui 的底层依赖之一:shadcn/ui 复用 Radix 已经做对的交互逻辑,只负责补上一套 Tailwind 样式并把源码交给你。
