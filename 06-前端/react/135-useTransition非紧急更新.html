useTransition:非紧急更新

输入框里打字,大列表跟着过滤,结果每敲一个字都卡一下——按键的回显都跟着掉帧。问题不在"计算慢",而在 React 把"更新输入框"和"重新渲染大列表"当成同等紧急的两件事,谁先谁后全看代码顺序。useTransition 让你告诉 React:这次更新不急,先紧着别的。

为什么会有"更新优先级"这回事

React 18 之前,渲染是不可中断的:一旦开始渲染一棵组件树,必须做完才能处理下一个事件,哪怕这次渲染毫无用户能立刻感知到的价值(比如渲染的是几万行列表里此刻根本不在视口的部分)。这就是同步渲染模型——所有更新一视同仁,谁先触发谁先跑完。

但真实场景里,更新天生就有轻重缓急之分:

  • 紧急更新:用户直接交互触发、需要立即看到反馈的——打字时输入框里的字符回显、点击按钮时按钮的按下态。慢一帧用户就能感觉到"卡"。
  • 非紧急更新(过渡更新,transition):是"结果",不是"交互"本身——根据刚才输入的关键字重新过滤一个大列表、切换 tab 后渲染新页面内容。哪怕晚个几十毫秒才出现,用户体验也几乎无损,但绝不能因为它,让紧急更新卡住。

React 18 引入的并发渲染(Concurrent Rendering)解决的正是这个问题:渲染变成可中断的——React 把渲染工作拆成一个个小单元,每做完一小块就检查"有没有更紧急的事要处理"(比如一次新的按键事件),有的话就暂停当前这次低优先级渲染,先去处理紧急更新,忙完再回来接着渲染或者直接扔掉重新开始。这就是"可中断渲染"——不是取消任务,而是有能力随时让路给更重要的事,做完之后再继续。

useTransition 就是这套机制暴露给开发者的接口:把某次 setState 标记为"过渡",React 就知道这次渲染可以被打断、可以延后。

按下按键
→
紧急更新:输入框立即回显
→
过渡更新:后台渲染过滤后的列表
→
列表准备好后才替换上屏

反例:一次 setState 卡住整个输入框

先看没有 useTransition 时会发生什么——过滤一万条数据的列表,和更新输入框的 setState 写在同一个事件处理函数里,两者优先级相同,浏览器必须先把过滤计算和列表渲染做完,才能让输入框的字符显示出来。

反例:同步更新,大列表拖慢输入框
function SearchList({ items }) {
  const [query, setQuery] = useState('');
  const filtered = items.filter((i) => i.name.includes(query)); // 一万条数据的过滤+渲染很重

  function handleChange(e) {
    setQuery(e.target.value); // 这一次 setState 同时触发输入框回显 + 列表重渲染,两者绑在一起
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      <List items={filtered} />  {/* 每敲一个字都要重新渲染这个重列表,导致输入卡顿 */}
    </>
  );
}
感受一下卡顿的本质: 用户按下一个键,浏览器要做的事情是"更新 query state → 重新渲染 input → 重新渲染 List(过滤 + diff 一万个节点)"。这些工作全在一次不可中断的渲染里完成,输入框的新字符必须等列表渲染完才画到屏幕上——按键越快,积压的渲染任务越多,输入体验就越"粘"。

正例:用 startTransition 把非紧急部分标记出来

useTransition 返回一个 isPending 布尔值和一个 startTransition 函数。把"不那么紧急"的那次 setState 包进 startTransition,React 就知道:这次更新造成的渲染可以被打断,如果输入框又来了新的按键(紧急更新),优先处理按键,过渡渲染要么让路要么直接被丢弃重新算。

正例:startTransition 包裹过滤逻辑,输入框始终流畅
function SearchList({ items }) {
  const [query, setQuery] = useState('');       // 紧急:直接决定输入框显示什么
  const [filterQuery, setFilterQuery] = useState(''); // 非紧急:驱动大列表过滤
  const [isPending, startTransition] = useTransition();

  const filtered = items.filter((i) => i.name.includes(filterQuery));

  function handleChange(e) {
    const value = e.target.value;
    setQuery(value);                 // ① 立即更新,输入框始终跟手
    startTransition(() => {
      setFilterQuery(value);          // ② 标记为过渡:列表重渲染可被打断/延后
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending && <span>列表更新中…</span>}
      <List items={filtered} />
    </>
  );
}

关键在于把一次操作拆成了两个 state:query 驱动输入框,永远同步、优先;filterQuery 驱动大列表,用 startTransition 包裹后就是"低优先级"的。打字再快,输入框都不会卡,列表会"稍微滞后一点"才跟上,但整体感觉依然流畅,而不是每敲一个字都卡顿一下。

isPending 的作用: 过渡更新是"延后执行"而不是"立即完成",这中间有个真实的时间差。isPending 告诉你"过渡渲染还没完成",可以用来显示一个轻量的 loading 提示(比如列表半透明、显示"更新中…"),避免用户以为过滤没生效。

易踩的坑

坑一:startTransition 里不能直接用事件里的同步值做副作用判断。 startTransition 的回调必须是同步函数,里面只能放 setState,不能放 await 或者依赖当时 DOM 状态的逻辑——过渡任务可能被打断重新执行,副作用重复跑会出问题。

坑二:以为 startTransition 能让"计算"本身变快。 它解决的是调度顺序问题,而不是算法性能问题——如果 filter 本身极其低效(比如 O(n²)),包一层 startTransition 只是让它不阻塞输入框,总耗时并不会变短,该做的性能优化(虚拟列表、useMemo、Web Worker)一个都不能少。

坑三:紧急更新和过渡更新混在一次 setState 里。 上面反例的问题根源就是"一个 state 同时服务两个用途"。要用好 useTransition,通常得像正例那样拆成两份 state,一份紧急、一份可延后。

和已学内容的联系

useTransition 只是"标记优先级",它本身不产生任何新数据、也不做请求。它经常和本系列前面讲过的东西配合:大列表场景常和"虚拟列表/分页"一起用来减少单次渲染的开销;和 useState 的关系是"给 setState 打了个优先级标签";如果过滤逻辑本身涉及数据请求(比如服务端搜索),那就要交给 useQuery 之类的方案处理加载态,而不是 useTransition 该管的事。

小结

  • React 18 的并发渲染让渲染变成可中断的:渲染到一半如果来了更紧急的更新,React 可以先处理紧急的,再回头继续或重新做低优先级渲染。
  • useTransition 返回 [isPending, startTransition],把一次 setState 包进 startTransition 就是告诉 React"这次更新不紧急,可以被打断/延后"。
  • 典型用法是把一次用户操作拆成两个 state:一个紧急(如输入框回显),一个包进 startTransition(如驱动大列表重渲染)。
  • isPending 用来展示"过渡还没完成"的轻量加载态。
  • 它优化的是调度顺序,不是算法本身的性能,该有的性能优化(memo、虚拟列表)依然要做。
