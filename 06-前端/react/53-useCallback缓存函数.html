useCallback：缓存函数引用

上一篇的 useMemo 缓存的是"计算结果"，这一篇的 useCallback 缓存的是"函数本身"。它专门解决第 51 篇提到的痛点：每次渲染都新建函数、破坏 memo 子组件的问题。

用法：useCallback(fn, deps)

useCallback 接收一个函数和依赖数组，返回一个"被缓存的函数引用"。只要依赖不变，多次渲染返回的都是同一个函数；依赖变了才返回新函数。

缓存传给 memo 子组件的回调
const Child = React.memo(Child);  // 只在 props 变时重渲

function Parent() {
  const [n, setN] = useState(0);

  // 依赖为空 → handleClick 引用永远稳定
  // 即使 Parent 因 n 变化重渲，Child 拿到的还是同一个函数 → 不重渲
  const handleClick = useCallback(() => {
    doSomething();
  }, []);

  return (
    <div>
      <button onClick={() => setN(n + 1)}>{n}</button>
      <Child onClick={handleClick} />  // 稳定引用，memo 生效
    </div>
  );
}
关键前提：useCallback 只有配合 React.memo 才有意义。如果 Child 没被 memo 包裹，它本来就会跟着父组件重渲——你缓存不缓存 handleClick 都一样，useCallback 纯属白加开销。缓存函数引用是为了让"引用比较"通过，而只有 memo（或 effect 依赖）才会去做这个比较。

它其实是 useMemo 的语法糖

useCallback 和 useMemo 本质是一回事——前者缓存"函数"，后者缓存"函数的返回值"。下面两行完全等价：

useCallback(fn, deps) === useMemo(() => fn, deps)
// 缓存函数 fn 本身
const a = useCallback(fn, deps);

// useMemo 缓存"返回 fn 的那个函数"的结果 = fn
const b = useMemo(() => fn, deps);

// a 和 b 行为完全一致。useCallback 只是为"缓存函数"这个高频需求
// 提供的便捷写法——省得每次写 () => fn 这层包装
useMemouseCallback
缓存的是计算的结果函数本身
你传入一个"算值"的函数要缓存的那个函数
典型场景昂贵计算、稳定对象传给 memo 子组件的回调
等价关系useCallback(fn, d) === useMemo(() => fn, d)

依赖数组的坑：闭包捕获旧值

useCallback 缓存的函数会闭包捕获创建它时的 props/state。如果依赖数组漏写了函数内用到的变量，函数会一直用创建时的"旧值"，产生难查的 bug。

❌ 漏依赖 → 用到旧 count
const inc = useCallback(() => {
  setCount(count + 1)
}, [])  // 漏了 count
// 永远读到初始的 count=0，
// 点多少次都是 setCount(1)
✅ 函数式更新免依赖
const inc = useCallback(() => {
  setCount(c => c + 1)  // 用回调
}, [])  // 不再依赖 count
// setter 稳定，逻辑正确
// 且引用保持稳定
技巧：更新依赖旧状态时，用函数式更新 setCount(c => c + 1) 就能把 count 从依赖里去掉，既保证逻辑正确、又让 useCallback 依赖为空、引用永远稳定。这是 useCallback + 函数式更新的经典搭配（呼应第 22 篇函数式更新）。
同样不要滥用。给每个函数无脑套 useCallback 是反模式：没有 memo 子组件消费时它毫无收益，还增加代码噪音、依赖数组维护负担和一点点开销。判断标准很简单——这个函数会不会被传给 memo 组件、或进某个依赖数组？不会就别包。第 54 篇专讲这种过度优化的代价。

小结

  • useCallback(fn, deps) 缓存函数引用，依赖不变则复用同一个函数。
  • 只有配合 React.memo（或作为 effect/其它 Hook 依赖）时才有意义，否则白加。
  • 本质等价 useMemo(() => fn, deps)，是缓存函数的语法糖。
  • 依赖要写全，防闭包捕获旧值；更新依赖旧态时用函数式更新去掉依赖。不要滥用。

常见陷阱

  • 没有 memo 子组件也无脑用 useCallback：给每个事件处理函数都套一层 useCallback，但组件树里没有任何 React.memo 消费这个引用、也没进任何依赖数组。这时候缓存函数没有任何收益，反而多了依赖比较和闭包维护的开销，纯属"看起来在优化"的心理安慰。
  • 依赖数组漏写，闭包里用的是旧的 state/props：比如 useCallback(() => setCount(count + 1), []) 漏了 count，缓存的函数永远闭包捕获创建时的初始值，无论点多少次结果都一样。这是 useCallback 最容易踩、也最难查的坑，因为报错现场是"逻辑不对"而不是崩溃。
  • 以为把依赖数组清空就等于"性能更好"：为了让引用绝对稳定，故意把用到的变量从依赖数组里删掉（哄骗 ESLint），代价是函数内部读到的永远是过期数据。正确解法是用函数式更新（setCount(c => c + 1)）真正消除对旧值的依赖，而不是隐瞒依赖。
  • 忽视 useCallback 本身也有维护成本：多一个 Hook 调用、多一份依赖数组要跟着代码演进同步维护，团队协作时容易有人改了函数体却忘了同步依赖数组，造成隐蔽 bug。只有明确"这个函数要传给 memo 组件或进依赖数组"时才值得引入这份成本。
  • 混淆"函数没有重新创建"和"函数内部逻辑没有变化"：useCallback 保证的是引用稳定性，不代表函数执行的结果不会变——依赖变化后返回的是携带新闭包的新函数，行为完全可能不同。把它和"记忆化结果"（如 useMemo 缓存返回值）搞混，会误判某些场景该不该用。

面试高频考点

  1. useCallback 到底解决什么问题？为什么普通函数定义不够？—— JS 里每次渲染时组件函数体重新执行，其中定义的箭头函数/普通函数都是"新对象"，即使代码逻辑完全一样，引用也不相等。这会导致依赖引用比较（React.memo 的浅比较、useEffect 依赖比较）永远判定"变了"，从而使 memo 失效或 effect 重复触发。useCallback 让同一段逻辑跨渲染保持同一个函数引用，解决的正是这个"意外的引用不稳定"问题。
  2. useCallback(fn, deps) 和 useMemo(() => fn, deps) 是什么关系？—— 完全等价，useCallback 是 useMemo 针对"缓存函数"这个高频场景的语法糖：useMemo 缓存的是"计算函数的返回值"，如果这个返回值恰好是另一个函数，效果就和 useCallback一样。二者共享同一套底层缓存机制（依赖比较、缓存槎位）。
  3. useCallback 什么时候是"白加"，完全没有性能收益？—— 当这个函数引用没有被传给任何 React.memo 包裹的子组件、也没有作为其它 Hook（如 useEffect/useMemo）的依赖时。此时不管函数引用是否稳定，都不会影响任何比较结果或触发额外渲染，缓存的成本（依赖比较、内存占用）纯属浪费。
  4. 为什么函数式更新（setState(prev => ...)）能帮 useCallback 减少依赖？—— 普通写法 setCount(count + 1) 需要在闭包里读取当前的 count，因此必须把 count 放进依赖数组，一旦 count 变化，缓存的函数就要重新生成。而 setCount(c => c + 1) 让 React 在更新时把"当前最新值"作为参数传入回调，函数体内不再需要闭包捕获外部的 count，依赖数组可以安全清空，函数引用能保持真正的稳定。

与其他知识点的联系

  • useMemo 缓存计算（第 52 篇）：本篇已经证明 useCallback(fn, deps) 就是 useMemo(() => fn, deps) 的语法糖，两者共享同一套"依赖不变则复用缓存"的机制——学 useCallback 本质是在学怎么用 useMemo 缓存"函数"这一种特殊值。
  • 引用稳定性问题（第 51 篇）：本篇解决的正是第 51 篇提出的"破坏点一"——父组件每次渲染都新建函数字面量，导致 React.memo 子组件的 props 比较永远判定"变了"。useCallback 是那篇留下的悬念的具体解法。
  • 过度优化的坏处（第 54 篇）：本篇陷阱里提到"没有 memo 子组件也无脑用 useCallback"是白加开销，下一篇会把这个道理系统化——预防性缓存的三大代价（可读性、缓存开销、依赖维护负担）同样适用于 useCallback。
  • useReducer 与 Context 结合（第 50 篇）：那一篇里 dispatch 天然引用稳定、不需要额外 useCallback 包装，是因为 useReducer 内部已经保证了这一点——对比本篇需要手动 useCallback 才能稳定普通函数，能更清楚看出"哪些引用天生稳定、哪些需要手动稳定"的区别。
  • 子传父回调（第 59 篇）：第 59 篇讲父组件把函数当 props 传给子组件实现"子传父"，如果这个子组件被 memo 包裹，父组件传下去的回调函数就需要用本篇的 useCallback 稳定引用，否则回调模式和性能优化会互相拖累。
