React DevTools Profiler 使用

"感觉这个列表卡"、"我猜是这个组件重渲染太多"——凭感觉优化,十有八九是在瞎猜。过早优化是万恶之源,而且没测量的优化经常优化错地方,反而让代码更复杂却没有效果。React DevTools 的 Profiler 面板能精确告诉你:谁渲染了、渲染了多久、为什么渲染。这一篇讲它的核心用法。

为什么必须先测量,再优化

上一篇讲过:父组件重渲染,子组件默认全部跟着重渲染。这个现象普遍存在,但不代表它都是问题——如果子组件本身很轻(渲染耗时 0.1ms),重渲染 100 次也感觉不到卡顿;真正值得优化的是渲染耗时长 + 触发频繁的组件。不测量就动手加 memo/useMemo,常见后果是:

凭感觉优化
给一堆组件包上 memo
额外的比较成本抵消收益
代码变复杂、难维护
真正的瓶颈可能根本没碰到
先用 Profiler 测量
录制一次真实交互
定位耗时最长/渲染最频繁的组件
只对症下药优化这几个
用 Profiler 验证优化是否生效

Profiler 面板的核心用法

打开浏览器 React DevTools 扩展,切到 Profiler 标签(和 Components 标签并列)。基本流程只有三步:

点击录制●
→
在页面上操作(比如打字、点击筛选)
→
停止录制,分析结果

两种视图

视图看什么
火焰图(Flamegraph)按组件树的层级展示,方块越宽代表渲染耗时越长;灰色方块代表这次提交(commit)没有重渲染该组件
排序图(Ranked)把当前这次提交里重渲染过的组件按耗时从长到短排列,一眼看出谁最慢
实操建议: 录制期间尽量只做一个操作(比如只点一次筛选按钮),这样一次录制里的每个"提交"(commit)含义清晰,不会把多次交互的耗时混在一起分析。面板顶部可以左右切换查看每一次提交(每次 state/props 变化触发的一整轮渲染)。

为什么渲染(Why did this render?)

选中火焰图里的某个组件方块,右侧面板会显示这次渲染的原因——是因为 props 变化、state 变化,还是 hooks 变化,甚至会具体列出哪个 prop 变了、新旧值分别是什么。这个信息比"耗时多少"更重要:耗时短的组件即使渲染频繁也不用管,但"props 没变却渲染了"往往就是可以用 memo 优化的信号。

用 Profiler API 编程方式记录渲染信息(可选,配合面板使用)
import { Profiler } from 'react';

function onRender(id, phase, actualDuration) {
  // id: Profiler 树的标识;phase: "mount" 或 "update"
  // actualDuration: 这次渲染实际耗费的毫秒数
  console.log(id, phase, actualDuration);
}

<Profiler id="ProductList" onRender={onRender}>
  <ProductList />
</Profiler>
常见误区:只看总耗时,不看"为什么渲染"。 有人看到火焰图里某个组件方块很宽,就直接给它加 memo,结果发现完全没用——因为它本来就是因为自身 state 变了才渲染的,memo 只能阻止"props 没变但父重渲染导致的无效渲染",管不了自身状态触发的渲染。先看渲染原因,再决定用哪种优化手段,不要看到"耗时长"就无脑加 memo。
和上一篇的联系: 上一篇讲的三种重渲染原因,在 Profiler 里都能对应看到——"props 变化"对应父组件重渲染导致的传递、"state 变化"对应自身状态、还有 Context 变化会在"为什么渲染"里显示为 hooks 变化。Profiler 就是把抽象的"重渲染原因"变成可以肉眼验证的具体数据。

小结

  • 优化前一定先测量——过早优化不仅浪费精力,还可能让代码更复杂却没有实际收益。
  • Profiler 面板:录制一次交互 → 火焰图(层级+耗时)或排序图(按耗时排序)→ 定位问题组件。
  • 选中组件后查看"为什么渲染",比只看耗时更关键——它告诉你该用 memo、useMemo 还是压根不用优化。
  • 常见误区:看到耗时长就无脑加 memo,却不管这次渲染其实是自身 state 变化触发的、加了也没用。
