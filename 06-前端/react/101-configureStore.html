configureStore：一行创建、开箱即用的 store

写好 slice 之后，需要把它们组装成一个 store——整个应用唯一的状态容器。RTK 的 configureStore 取代了传统的 createStore，把中间件、DevTools 这些配置都做了合理默认。

createStore 的历史包袱

传统 Redux 用 createStore，想用异步和调试工具，得手动拼中间件和 enhancer，代码晦涩且容易抄错：

传统 createStore：手动拼装
import { createStore, applyMiddleware, combineReducers } from 'redux';
import thunk from 'redux-thunk';
import { composeWithDevTools } from 'redux-devtools-extension';

const rootReducer = combineReducers({ counter, user });
const store = createStore(
  rootReducer,
  composeWithDevTools(applyMiddleware(thunk))  // 这行几乎没人记得住
);

configureStore：合理默认，一步到位

app/store.js
import { configureStore } from '@reduxjs/toolkit';
import counterReducer from '../features/counterSlice';
import userReducer from '../features/userSlice';

export const store = configureStore({
  // reducer 字段是个对象，key 就是 state 树里的字段名
  // configureStore 内部自动调用 combineReducers 帮你合并
  reducer: {
    counter: counterReducer,   // → state.counter
    user: userReducer,        // → state.user
  },
});

// TS 项目常导出这两个类型，供 useSelector / useDispatch 使用
// export type RootState = ReturnType<typeof store.getState>;
// export type AppDispatch = typeof store.dispatch;
configureStore 默认帮你做了什么？
  • 自动 combineReducers 合并 reducer 对象
  • 默认装好 redux-thunk（所以 createAsyncThunk 直接能用）
  • 开发环境默认接入 Redux DevTools，无需手动配 compose
  • 加了两个开发期检查中间件：不可变性检查（防止你手滑改了 state）和 序列化检查（防止往 state 里塞 Promise、函数等不可序列化的值）

用 Provider 把 store 注入 React

store 建好后，用 react-redux 的 <Provider> 包在应用最外层。它底层用 React Context 把 store 向下传递，任何后代组件都能通过 hook 访问，无需层层传 props。

main.jsx —— 应用入口
import { Provider } from 'react-redux';
import { store } from './app/store';

ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>       // 把 store 通过 Context 提供给全树
    <App />
  </Provider>
);

自定义中间件（进阶）

需要加自定义中间件时，用 getDefaultMiddleware 拿到默认列表再拼接，避免把 thunk、检查中间件覆盖掉：

在默认中间件基础上追加
configureStore({
  reducer: rootReducer,
  middleware: (getDefaultMiddleware) =>
    getDefaultMiddleware().concat(logger),  // 保留默认，追加 logger
  devTools: import.meta.env.DEV,  // 仅开发环境开 DevTools
});
面试点与对比：①"configureStore 相比 createStore 好在哪？"——内置 thunk、DevTools、开发期检查，且用 reducer 对象自动 combineReducers。②"Provider 的原理？"——基于 React Context 下发 store，避免 props 逐层透传。③一个应用只应有一个 store（单一数据源），多个 slice 挂在同一个 store 的不同字段下，不要建多个 store。

与其他知识点的联系

  • createSlice 组合多个 slice reducer（100）：本文的 reducer 字段对象（{ counter: counterReducer, user: userReducer }）里的每一个值，正是 100 篇 createSlice 导出的 .reducer。configureStore 内部自动调用 combineReducers 把它们合并成一棵完整的 state 树——这解释了为什么 useSelector(state => state.counter.value) 里的 counter 这一层，刚好对应 reducer 对象里的 key 名。
  • Redux DevTools：传统 createStore 要手动引入 redux-devtools-extension 并用 composeWithDevTools 包裹 enhancer，而 configureStore 默认在开发环境自动接好 DevTools。这不只是省代码，更重要的是降低了新手接触 Redux 调试工具的门槛——不知道怎么手动配置 compose 的人，用 configureStore 也能立刻用上时间旅行调试。
  • 中间件（redux-thunk 默认内置）：本文强调 configureStore 默认装好了 redux-thunk，这正是 100 篇 createAsyncThunk 能直接工作的前提——thunk 中间件的作用是让 dispatch 除了能接收普通 action 对象外，还能接收一个函数（thunk），由中间件负责执行它、并在内部继续 dispatch 真正的 action。如果没有这个中间件，dispatch(fetchUser()) 会直接报错，因为原生 Redux 的 dispatch 只认 action 对象。
  • Context API 与 Provider：本文提到 <Provider store={store}> 底层基于 React Context，这和 03 篇讲的"Props 穿透"问题遥相呼应——Redux 选择用 Context 把 store 一次性注入到组件树顶层，避免手动把 store 通过 props 一层层往下传，这也是 Context API 最典型、最广泛的实际应用场景之一。
