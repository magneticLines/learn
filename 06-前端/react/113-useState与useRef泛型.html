useState / useRef 泛型

useState(0) 不用写泛型,TS 自己就能推断出 number。但初始值是 null、或状态本身是"要么是 User、要么是 null"这种联合类型时,推断就不够用了——必须手动传泛型告诉 TS 这个状态"将来可能是什么"。useRef 同理,而且它还有两种截然不同的用法,类型行为也不一样。

useState:什么时候推断够用,什么时候不够

TS 给泛型函数标注类型有两种方式:让它从参数自动推断,或者你显式传。useState 的类型签名大致是 function useState<T>(initial: T): [T, Dispatch<SetStateAction<T>>]——推断出的 T 就是初始值的类型。

推断够用
// T 被推断为 number,足够
const [count, setCount] = useState(0);

// T 被推断为 string
const [name, setName] = useState('');

// T 被推断为 boolean
const [open, setOpen] = useState(false);
推断不够,需要显式泛型
// 不传泛型:T 被推断成 null,
// 之后 setUser(someUser) 会报类型错误
const [user, setUser] = useState(null);

// 显式传泛型:T 是 User | null
const [user, setUser] =
  useState<User | null>(null);

典型场景是初始值是 null/undefined,但状态之后会被赋成别的类型——比如"异步请求前是 null,请求完是拿到的数据"。这时候不显式传泛型,TS 会把类型锁死在 null,后续赋值全部报错。

常见场景:异步数据的 useState 泛型
interface User { id: number; name: string; }

function Profile() {
  // 请求完成前是 null,完成后是 User —— 必须显式给联合类型
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetchUser().then(setUser);  // 合法:User 属于 User | null
  }, []);

  // user 用之前必须判空 —— 这正是联合类型带来的强制安全检查
  if (!user) return <p>加载中...</p>;
  return <p>{user.name}</p>;
}
陷阱: 只写 useState(null) 不加泛型,TS 把类型推断为 null(不是 any,也不是"以后你想赋什么都行")。之后任何 setUser(someObject) 都会报 "不能将类型 User 分配给类型 null"。很多初学者一看这个报错就懵——根因就是没有显式传联合类型泛型。

useRef:两种截然不同的用法

useRef 名字里没有体现出的一点是,它其实有两种完全不同的使用场景,类型层面的表现也不一样:

场景一:DOM 引用
const inputRef =
  useRef<HTMLInputElement>(null);

// .current 类型是
// HTMLInputElement | null
// 只读:不能手动赋值 .current
<input ref={inputRef} />

inputRef.current?.focus();
场景二:可变值容器
const timerRef = useRef<number>(0);

// .current 类型是 number(可写)
// 存"跨渲染不触发重渲染"的值
timerRef.current = window
  .setInterval(tick, 1000);

clearInterval(timerRef.current);

关键区别在于初始值传的是不是 null,这决定了 TS 给 .current 推断出的类型可不可写:

写法.current 类型能否手动赋值 .current典型用途
useRef<HTMLInputElement>(null)HTMLInputElement | null不能(React 内部管理,只读)绑定给 JSX 的 ref 属性,拿 DOM 节点
useRef<number>(0)number能存定时器 ID、上一次的值等"可变但不触发渲染"的数据
为什么会有这个区别? useRef 的类型签名其实是重载的:传了 null 作初始值时,TS 认为这是"绑定 DOM 节点"的用法,返回一个 RefObject<T>,current 是只读的(因为 DOM 节点应该由 React 挂载/卸载时自动写入,不该被你手动改);不传 null(比如传 0)时,返回的是 MutableRefObject<T>,current 可读可写,专门用于你自己维护的可变数据。
DOM ref 常见泛型对照(不同标签对应不同的 Element 类型)
const inputRef  = useRef<HTMLInputElement>(null);   // <input ref={inputRef} />
const divRef    = useRef<HTMLDivElement>(null);     // <div ref={divRef} />
const buttonRef = useRef<HTMLButtonElement>(null); // <button ref={buttonRef} />
常犯错: useRef<HTMLInputElement>() 不传参数——TS 会报错要求必须传初始值,写 useRef<HTMLInputElement>(null) 才对。另外访问 DOM ref 的 .current 时,由于类型是 HTMLInputElement | null,直接 inputRef.current.focus() 会被 TS 拦下——要用可选链 inputRef.current?.focus(),这不是啰嗦,而是提醒你"组件还没挂载完成时这个值确实是 null"这个真实存在的场景。

和本系列前面知识点的联系

useState 泛型本质就是"泛型函数调用时显式传类型参数",跟 TS 泛型基础篇讲过的 identity<T>(x: T) 是同一套机制;useRef 的只读/可写差异,则是"函数重载"在实际库类型定义里的一个典型应用——同一个函数名,根据传参不同返回不同形状的类型。

小结

  • useState 的泛型能从初始值自动推断;初始值是 null/undefined 但状态之后会变成别的类型时,必须显式传联合类型泛型,如 useState<User | null>(null)。
  • 不显式传泛型直接写 useState(null),类型会被锁死为 null,后续任何赋值都会报类型错误。
  • useRef 有两种用法:传 null 初始值(DOM 引用,.current 只读)、传非 null 初始值(可变值容器,.current 可写)。
  • DOM ref 的泛型要匹配具体标签对应的 Element 类型,如 HTMLInputElement、HTMLDivElement。
  • 访问 DOM ref 的 .current 前记得判空(可选链),因为组件挂载前它确实是 null。
