乐观更新：先改 UI，再等服务器

点赞、切换开关这类高频操作，如果每次都等服务器返回再更新 UI，会有明显延迟。乐观更新（Optimistic Update）反过来：假设一定成功，先立刻改界面，请求失败再回滚。

悲观 vs 乐观

悲观更新（默认）
点击 → loading
→ 等服务器返回
→ 才更新 UI
安全，但有等待感
乐观更新
点击 → 立即改 UI
→ 后台发请求
→ 成功：保持
   失败：回滚 + 提示
体验丝滑，逻辑更复杂
生活类比：发微信消息时，气泡立刻出现在对话里（乐观），如果发送失败才显示红色感叹号让你重发。没人愿意点发送后盯着转圈等服务器确认才看到自己的消息。

三步走：onMutate / onError / onSettled

乐观更新的骨架就是 mutation 的三个回调协同工作：

onMutate
先改缓存 + 存快照
→
onError
用快照回滚
→
onSettled
invalidate 兜底同步
乐观切换 todo 完成状态
const qc = useQueryClient();

const mutation = useMutation({
  mutationFn: toggleTodo,   // (todo) => PATCH /todos/:id

  // ① onMutate：请求发出「前」执行——这里做乐观改动
  onMutate: async (newTodo) => {
    // 取消进行中的重取，避免它用旧数据覆盖我们的乐观更新
    await qc.cancelQueries({ queryKey: ['todos'] });
    // 存快照，供失败时回滚
    const prev = qc.getQueryData(['todos']);
    // 直接把缓存改成「预期结果」——UI 立刻更新
    qc.setQueryData(['todos'], (old) =>
      old.map(t => t.id === newTodo.id ? newTodo : t)
    );
    return { prev };   // 返回的对象成为 context，传给 onError/onSettled
  },

  // ② onError：失败时用快照回滚，撤销乐观改动
  onError: (err, newTodo, context) => {
    qc.setQueryData(['todos'], context.prev);  // 还原
    toast.error('操作失败，已撤销');
  },

  // ③ onSettled：无论成败都重新拉一次，确保和服务器最终一致
  onSettled: () => {
    qc.invalidateQueries({ queryKey: ['todos'] });
  },
});
为什么每一步都必要？
  • cancelQueries：不取消的话，一个恰好在途的后台重取可能带着旧数据返回，把你的乐观更新覆盖掉
  • 快照 prev：失败回滚必须知道"改之前长什么样"，所以要先存
  • onSettled invalidate：即便成功，服务器返回的真实数据可能和你猜的略有出入（比如后端生成的 id、时间戳），最后同步一次才权威

代价与取舍：不是所有场景都该用

维度说明
复杂度要写快照、回滚、取消、兜底，比普通 mutation 代码量翻几倍
回滚闪烁失败率高的操作，用户会频繁看到"改了又弹回去"，反而更糟
结果可预测性只适合"结果基本能预判"的操作（点赞 +1、勾选完成）；若结果依赖服务器计算（如下单后的最终价格），别乐观
什么时候用、什么时候别用：点赞、收藏、开关、拖拽排序——高频、低失败率、结果可预测，适合乐观更新。支付、下单、涉及金额或强一致性、失败代价高的操作——老老实实悲观更新，让用户看到明确的加载和结果。别为了"炫技"给关键流程加乐观更新。
面试点：①"乐观更新怎么实现？"——onMutate 里 cancelQueries + 存快照 + setQueryData 改缓存，onError 用快照回滚，onSettled invalidate 兜底。②"为什么要 cancelQueries？"——防止在途请求用旧数据覆盖乐观更新。③"什么场景不该用？"——高失败率或结果不可预测（支付、强一致）的操作。

与其他知识点的联系

  • 关联缓存与失效策略：onMutate 里用的 setQueryData 正是上一章提到的"直接改缓存、不发请求"那条路径，乐观更新可以看作缓存操作的一种高级组合用法，而不是全新的独立技术。
  • 关联 useMutation 的三个回调：onMutate/onError/onSettled 都是上上章定义好的 mutation 生命周期钩子，本节只是给它们填入了"先改、错了再回滚、最后兜底同步"这套具体逻辑，是对已学内容的复用而非新概念。
  • 关联 RTK Query：RTK Query 也有类似的乐观更新机制（updateQueryData + 手动 patch + 失败回滚），思路完全一致，只是 API 名字不同，说明"先改 UI 再等服务器"是数据获取库的通用模式而非 TanStack Query 专属。
  • 关联 useState 本地状态：如果没有服务端状态管理库，开发者也能用 useState 手写"乐观改 UI、失败再 setState 回退"的逻辑，但要自己处理竞态和取消，理解这点能体会 TanStack Query 帮你规避了哪些坑（尤其是 cancelQueries 这一步）。
  • 关联工程取舍原则：本节"高频低失败率才用乐观更新"的判断标准，和后面项目工程化部分反复强调的"不要过度设计"是一致的方法论——技术选型要看场景收益，不是新特性就无脑上。
