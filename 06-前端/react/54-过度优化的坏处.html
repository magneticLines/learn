过度优化的坏处：memo 不是越多越好

学完 useMemo / useCallback / memo，很多人会走向另一个极端：给所有东西都套上缓存，"以防万一"。这一篇讲清楚——预防性优化往往比不优化更糟。

Donald Knuth 的老话

"过早优化是万恶之源（premature optimization is the root of all evil）。"这句软件工程界的名言在 React 里同样成立。useMemo/useCallback 不是"性能增强剂"，它们是有明确成本的权衡工具，只在特定场景下才划算。

代价一：可读性下降

本来一行朴素代码，套上缓存后变成三行、多一层缩进、多一个依赖数组。当满屏都是 useMemo/useCallback 时，真正重要的逻辑被淹没在缓存样板里，代码更难读、更难改。

✅ 朴素、清晰
const total =
  price * count

const onClick = () =>
  addToCart(id)
❌ 无谓的缓存包装
const total = useMemo(
  () => price * count,
  [price, count]
)
const onClick = useCallback(
  () => addToCart(id),
  [id]
)  // 没 memo 子组件，全白包

代价二：缓存本身有开销

useMemo/useCallback 不是免费快照。每次渲染它们都要：保存依赖数组、逐项比较新旧依赖、保留上次结果占内存。对便宜的计算，这些开销比重新计算还大——你以为在优化，其实在拖慢。

代价三：依赖数组的维护负担与 bug 风险

每个缓存都拖着一个依赖数组。代码演进时，函数里加了个新变量却忘了加进依赖——就出现"读到旧值"的隐蔽 bug。缓存越多，这种维护点越多，出错面越大。可以说，每个 memo 都是一处需要持续维护的债务。

最讽刺的情况：为了让 memo 子组件不重渲，你给回调加了 useCallback；但传给它的另一个 prop 是个内联对象 style={{...}}，引用照样每次都变——memo 还是失效。结果：优化没生效，却平添了 useCallback 的开销和复杂度。优化要么做全，要么别做。

正确姿势：先测量，再优化

唯一靠谱的优化流程是"用数据驱动"，而不是凭感觉：

  1. 先写朴素代码——不加任何缓存，把功能做对。
  2. 发现卡顿——实际交互中感觉到掉帧、输入延迟。
  3. 用 Profiler 定位——React DevTools 的 Profiler 面板录制一次交互，看哪个组件渲染耗时最长、渲染了多少次。
  4. 只优化那个瓶颈——针对性加 memo/useMemo/useCallback。
  5. 再测量确认——验证优化确实生效，而不是自我安慰。
React DevTools Profiler 怎么用：安装浏览器扩展 → 打开 Profiler 标签 → 点录制 → 操作一次 → 停止。火焰图里越宽的条 = 渲染越慢的组件；开启 "Highlight updates" 还能在页面上高亮每次重渲的组件，一眼看出哪里在"无谓地闪"。定位到真瓶颈，再动手。

好消息：React 19 Compiler 让手动 memo 成为历史

React 19 引入的 React Compiler（编译期自动记忆化）能在构建时分析代码，自动为组件和值插入等价于 memo/useMemo/useCallback 的优化——由编译器精确判断"哪里真的需要缓存"，比人手动加更准、更全，还不污染源码。

这对你意味着什么：在启用了 React Compiler 的项目里，绝大多数手写的 useMemo/useCallback 都可以删掉——编译器会接管。趋势很明确：手动记忆化正在成为"编译器的活",不是"程序员的活"。但在没上 Compiler 的存量项目里，理解并按需手动优化仍是必备技能。无论哪种，"先测量再优化"的原则永不过时。

一句话决策

问自己结论
有没有实际测到性能问题？没有 → 别加
这个值/函数会被 memo 组件或依赖数组比较吗？不会 → 别加
计算真的昂贵（Profiler 证实）吗？不是 → 别加
项目启用了 React Compiler 吗？是 → 交给它，几乎不用手写

小结

  • 预防性 memo 三大代价：可读性下降、缓存开销、依赖维护/bug 风险。
  • 缓存对便宜计算是净亏损；优化"做一半"（还有别的不稳定 prop）等于没做。
  • 正确流程：朴素代码 → 发现卡顿 → Profiler 定位 → 只优化瓶颈 → 再测量。
  • React 19 Compiler 会自动记忆化，手动 memo 正逐渐被编译器取代——但"先测量"永远对。
