过度优化的坏处：memo 不是越多越好

学完 useMemo / useCallback / memo，很多人会走向另一个极端：给所有东西都套上缓存，"以防万一"。这一篇讲清楚——预防性优化往往比不优化更糟。

Donald Knuth 的老话

"过早优化是万恶之源（premature optimization is the root of all evil）。"这句软件工程界的名言在 React 里同样成立。useMemo/useCallback 不是"性能增强剂"，它们是有明确成本的权衡工具，只在特定场景下才划算。

代价一：可读性下降

本来一行朴素代码，套上缓存后变成三行、多一层缩进、多一个依赖数组。当满屏都是 useMemo/useCallback 时，真正重要的逻辑被淹没在缓存样板里，代码更难读、更难改。

✅ 朴素、清晰
const total =
  price * count

const onClick = () =>
  addToCart(id)
❌ 无谓的缓存包装
const total = useMemo(
  () => price * count,
  [price, count]
)
const onClick = useCallback(
  () => addToCart(id),
  [id]
)  // 没 memo 子组件，全白包

代价二：缓存本身有开销

useMemo/useCallback 不是免费快照。每次渲染它们都要：保存依赖数组、逐项比较新旧依赖、保留上次结果占内存。对便宜的计算，这些开销比重新计算还大——你以为在优化，其实在拖慢。

代价三：依赖数组的维护负担与 bug 风险

每个缓存都拖着一个依赖数组。代码演进时，函数里加了个新变量却忘了加进依赖——就出现"读到旧值"的隐蔽 bug。缓存越多，这种维护点越多，出错面越大。可以说，每个 memo 都是一处需要持续维护的债务。

最讽刺的情况：为了让 memo 子组件不重渲，你给回调加了 useCallback；但传给它的另一个 prop 是个内联对象 style={{...}}，引用照样每次都变——memo 还是失效。结果：优化没生效，却平添了 useCallback 的开销和复杂度。优化要么做全，要么别做。

正确姿势：先测量，再优化

唯一靠谱的优化流程是"用数据驱动"，而不是凭感觉：

  1. 先写朴素代码——不加任何缓存，把功能做对。
  2. 发现卡顿——实际交互中感觉到掉帧、输入延迟。
  3. 用 Profiler 定位——React DevTools 的 Profiler 面板录制一次交互，看哪个组件渲染耗时最长、渲染了多少次。
  4. 只优化那个瓶颈——针对性加 memo/useMemo/useCallback。
  5. 再测量确认——验证优化确实生效，而不是自我安慰。
React DevTools Profiler 怎么用：安装浏览器扩展 → 打开 Profiler 标签 → 点录制 → 操作一次 → 停止。火焰图里越宽的条 = 渲染越慢的组件；开启 "Highlight updates" 还能在页面上高亮每次重渲的组件，一眼看出哪里在"无谓地闪"。定位到真瓶颈，再动手。

好消息：React 19 Compiler 让手动 memo 成为历史

React 19 引入的 React Compiler（编译期自动记忆化）能在构建时分析代码，自动为组件和值插入等价于 memo/useMemo/useCallback 的优化——由编译器精确判断"哪里真的需要缓存"，比人手动加更准、更全，还不污染源码。

这对你意味着什么：在启用了 React Compiler 的项目里，绝大多数手写的 useMemo/useCallback 都可以删掉——编译器会接管。趋势很明确：手动记忆化正在成为"编译器的活",不是"程序员的活"。但在没上 Compiler 的存量项目里，理解并按需手动优化仍是必备技能。无论哪种，"先测量再优化"的原则永不过时。

计算真的昂贵（Profiler 证实）吗？不是 → 别加
项目启用了 React Compiler 吗？是 → 交给它，几乎不用手写

小结

  • 预防性 memo 三大代价：可读性下降、缓存开销、依赖维护/bug 风险。
  • 缓存对便宜计算是净亏损；优化"做一半"（还有别的不稳定 prop）等于没做。
  • 正确流程：朴素代码 → 发现卡顿 → Profiler 定位 → 只优化瓶颈 → 再测量。
  • React 19 Compiler 会自动记忆化，手动 memo 正逐渐被编译器取代——但"先测量"永远对。

常见陷阱

  • 给未 memo 的子组件传 useCallback：如果子组件本身没用 memo 包裹，父组件重渲时子组件一定会重渲，useCallback 完全不起作用，只是白白多了一次依赖比较的开销。很多人误以为"加了 useCallback 就能防止子组件重渲"，其实它只是保证函数引用稳定，真正阻止重渲的是 memo。
  • 依赖数组"半缓存"：useCallback 包住了回调，但同时传给子组件的另一个 prop 是内联对象/数组（如 style={{}}），引用每次渲染都变，memo 照样判定 props 变化而重渲。优化没生效，代码却复杂了——典型的"做了一半等于没做"。
  • 把 useMemo 当"缓存开关"滥用：对一个简单的加法或字符串拼接套 useMemo，计算成本远低于依赖比较和占内存的成本，属于负优化。判断是否值得缓存，应该看计算本身是否昂贵（如遍历大数组、复杂正则），而不是"反正加了总没错"。
  • 没测量就下结论：凭感觉觉得"这里应该会卡"就加 memo，往往加错地方——真正的瓶颈可能在别的组件。跳过 Profiler 直接优化，是"过早优化"最常见的表现形式。
  • 过度依赖 React Compiler 后忽视基本原理：即使用了 React 19 Compiler 自动记忆化，也不代表可以完全不理解 memo 的原理——遇到 Compiler 无法安全优化的场景（如带副作用的表达式），仍需要手动介入，不理解原理会看不懂 Compiler 为什么"没生效"。

面试高频考点

  1. 为什么给所有组件都加 memo 不是好习惯？—— memo 本身有比较 props 的开销（浅比较所有 prop），如果组件本来就渲染很快、props 经常变化，这层比较反而是净增负担；同时大量 memo 会让代码充满样板，降低可读性和可维护性。优化应该基于 Profiler 测量出的真实瓶颈，而不是全面覆盖。
  2. useCallback 和 useMemo 分别解决什么问题？为什么"用了就一定变快"是误解？—— useMemo 缓存计算结果，useCallback 缓存函数引用本身（等价于 useMemo(() => fn, deps)）。它们只在"结果会被 memo 组件的 props 比较""或被其它 effect 依赖数组比较"时才有意义；如果没人消费这个稳定引用，缓存本身的比较和存储开销可能比重新计算还贵。
  3. 如何科学地判断一个组件是否需要优化？—— 用 React DevTools Profiler 录制真实交互，观察火焰图里渲染耗时长、渲染次数多的组件；开启 "Highlight updates" 直观看到无谓的重渲染。只有测量到确实的性能问题，才针对瓶颈组件精准施加 memo/useMemo/useCallback，然后再次测量验证效果。
  4. React 19 的 Compiler 是如何替代手动 memo 的？—— Compiler 在构建期静态分析组件代码的数据流，自动判断哪些值/函数在渲染间保持不变、哪些子组件可以跳过重渲，并插入等价于 memo/useMemo/useCallback 的优化代码，判断粒度比人工更精细、覆盖更全，且不污染源码可读性。

与其他知识点的联系

  • useMemo 缓存计算（第 52 篇）与 useCallback 缓存函数（第 53 篇）：本篇是前两篇"什么时候该用"的反面收尾——那两篇分别讲了怎么用这两个 Hook，本篇讲的是它们"不该用"的三大代价（可读性、缓存开销、依赖维护负担），完整闭环才是"该不该用"的全貌。
  • 引用稳定性问题（第 51 篇）：本篇强调的"优化要么做全、要么别做"（例子里 useCallback 稳定了回调，但内联 style={{}} 依然让 memo 失效）根源就是第 51 篇讲的引用比较机制——只稳定一半引用，另一半依然会让整个 props 浅比较判定"变了"。
  • 抽离复用逻辑（第 56 篇）：本篇陷阱里提到"没测量就下结论"，同样的态度也适用于要不要抽自定义 Hook——第 56 篇的判断标准是"确实重复或逻辑复杂"，而不是"看起来像该抽"，两篇都在强调"基于证据决策，而不是凭直觉"。
  • useReducer 与 Context 结合（第 50 篇）：那一篇结尾"什么信号提示该升级到 Redux/Zustand"和本篇"什么信号提示该加 memo"是同一种工程判断逻辑——都要求先有实际的性能/规模信号，再引入额外复杂度，不能提前预防性优化。
