Suspense 数据加载

第 126 篇讲过 Suspense 配合 React.lazy 做代码分割——组件代码还没下载完,先显示 fallback。React 18 把这套"没就绪就显示 fallback,就绪了自动切换"的能力,从"等代码"扩展到了"等数据"。前提是数据获取方案得专门支持它,普通的 useEffect 请求做不到。

先回顾:Suspense 到底在等什么

第 126 篇里,Suspense 等的是 React.lazy(() => import('./Chart')) 返回的模块什么时候下载完。React 18 把这个能力泛化了:Suspense 真正等待的不是"代码"或"数据"这种具体东西,而是一个抛出来的 Promise——只要某个组件在渲染过程中"还没准备好",就可以抛出一个 Promise,离它最近的 Suspense 边界会捕获这个 Promise,显示 fallback,等 Promise resolve 后重新渲染这个组件。

代码分割只是这套机制的第一个应用场景:React.lazy 内部帮你做了"组件还没加载好就抛出加载中的 Promise"这件事。数据加载是第二个应用场景:一个支持 Suspense 的数据获取库,在数据还没到达时,同样"抛出一个还未 resolve 的 Promise",React 用完全相同的机制处理它。

组件渲染时数据未就绪
→
组件抛出一个 Promise
→
最近的 Suspense 边界捕获
→
显示 fallback
→
Promise resolve → 自动重渲染,显示真实内容

命令式 vs 声明式:两种加载态写法的对比

先看最熟悉的写法——手动用 isLoading 控制展示什么,这是命令式的:每个用到数据的组件都要自己知道"现在是不是在加载中"。

命令式:手写 isLoading
function Profile() {
  const { data, isLoading } = useQuery(...);
  if (isLoading) return <Spinner />;
  return <ProfileCard user={data} />;
}
// 每一层用到异步数据的组件
// 都要写一遍这个判断逻辑,
// 加载态和展示逻辑耦合在一起
声明式:交给 Suspense
function Profile() {
  const { data } = useSuspenseQuery(...);
  return <ProfileCard user={data} />; // 走到这里数据一定已就绪
}

<Suspense fallback={<Spinner />}>
  <Profile />
</Suspense>
// 加载态提到外层统一声明,
// 组件本身只管"数据到了怎么展示"
声明式的好处不止是少写几行 if。 当页面里有多个异步数据源(比如用户信息 + 订单列表 + 推荐商品)时,命令式写法要么每个组件各自 loading(界面一块一块地"跳出来"),要么手动合并多个 isLoading 判断;用 Suspense 可以直接嵌套多个边界,精确控制"哪几块内容一起等、哪几块可以各自独立展示",布局更可控,代码也不用手动合并加载状态。

示例:用支持 Suspense 的数据方案

拿 TanStack Query(本系列前面讲 useQuery 用的那个库)举例,它提供了 useSuspenseQuery——用法和 useQuery 几乎一样,区别是数据未就绪时它会抛出 Promise 而不是返回 isLoading: true:

useSuspenseQuery:数据没到就让 Suspense 接管
import { useSuspenseQuery } from '@tanstack/react-query';
import { Suspense } from 'react';

function UserProfile({ userId }) {
  // 没有 isLoading 字段了——渲染到这一行时,数据必然已经就绪
  const { data } = useSuspenseQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  });
  return <h2>{data.name}</h2>;
}

function Page() {
  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userId={1} />  {/* 数据没到 → 抛 Promise → 显示 Spinner;到了 → 自动切回这里 */}
    </Suspense>
  );
}

另一个常见落地场景是 Next.js 的异步 Server Component——组件函数本身就是 async function Page() { const data = await fetchData(); ... },框架在渲染层面把"await 未完成"接到了 Suspense 机制上,同样是"没数据显示 fallback,数据到了自动替换"的声明式效果,写法上比手写 useSuspenseQuery 更直接,但这属于服务端渲染范畴,机制上是同一套思想的另一种实现。

陷阱:不是所有数据获取方式都能配合 Suspense

最容易踩的坑: 普通的 useState + useEffect 数据获取(useEffect(() => { fetch(...).then(setData) }, []))完全不支持 Suspense。它的加载过程发生在渲染完成之后的副作用里,不会在渲染阶段抛出 Promise,React 根本感知不到"这个组件在等数据"。把它包在 Suspense 里没有任何效果——该显示 fallback 的时候不会显示,组件依然是先渲染出空/undefined 状态,再等 effect 跑完手动 setState 触发二次渲染。

要接入 Suspense,数据获取逻辑必须发生在渲染阶段并且遵循"没准备好就 throw 一个 Promise"的约定,这通常意味着要依赖专门支持它的库(TanStack Query 的 useSuspenseQuery、Relay、或者框架自带的机制),不建议自己手写这套底层逻辑——处理竞态、缓存、去重的细节很容易出错。

和已学内容的联系

把这几篇串起来看:第 126 篇讲 Suspense + React.lazy 解决"代码还没下载好";这一篇讲同一套 Suspense 机制被数据获取库借用,解决"数据还没到"；而前面两篇 useTransition/useDeferredValue 解决的是另一件事——"渲染优先级"。三者可以组合使用:比如切换 tab 触发新的 useSuspenseQuery 请求时,用 startTransition 包裹这次切换,React 会保持旧内容展示而不是立刻跳出 fallback,直到新数据就绪才切换,这就是 React 官方文档里"用 Suspense 避免每次切换都闪烁 loading"的组合用法。

小结

  • Suspense 的底层机制是"捕获组件渲染时抛出的 Promise",第 126 篇的代码分割和这一篇的数据加载是同一套机制的两个应用场景。
  • 声明式加载态把"要不要显示 loading"从每个组件里剥离出来,统一交给外层 Suspense 边界声明,尤其在多个异步数据源共存时更清晰。
  • 要用上 Suspense 数据加载,必须用专门支持它的方案(useSuspenseQuery、Relay、异步 Server Component 等),不能靠普通的 useState + useEffect。
  • 普通 useEffect 请求的加载发生在渲染之后的副作用里,不会抛出 Promise,包进 Suspense 也不会生效——这是最常见的误用。
  • 可以和 useTransition 组合,让 tab 切换等场景在新数据未就绪前保留旧内容,避免频繁闪烁 fallback。
