AbortController 取消请求

搜索框每敲一个字就发一次请求,网络有快有慢——如果不取消"过时"的请求,先发出的请求可能后回来,把新结果覆盖成旧结果,界面显示的内容和用户输入的关键字对不上。AbortController 就是浏览器原生提供的"喊停"机制。

竞态问题:为什么必须主动取消请求

设想一个搜索框,用户依次输入 "r" → "re" → "react",每次输入都触发一次搜索请求。三个请求几乎同时发出,但网络延迟不可控:

发 "r" 请求
→
耗时 800ms(网络慢)
发 "react" 请求
→
耗时 100ms(先回来)

结果是:"react" 的搜索结果先渲染到屏幕上,紧接着 "r" 的搜索结果姗姗来迟,把界面覆盖回错误的旧数据——用户看到的是和当前输入框内容完全对不上的搜索结果。这就是经典的竞态条件(race condition)。

面试高频考点。 这类问题不止发生在搜索框——任何"参数变化时重新发请求"的场景(切换 tab、翻页、快速点击筛选条件)都可能踩坑。解决思路只有两种:①主动取消过时的请求;②给每次请求打标记,回来时比对"是不是最新一次"再决定要不要用这个结果。AbortController 用的是第一种思路。

AbortController 原理与用法

AbortController 是浏览器/Node 原生 API,原理很直接:创建一个"控制器",它的 signal 属性可以传给 fetch(或支持的库),调用控制器的 abort() 就会让这个请求提前中止并抛出错误。

原生用法:每次发请求前先取消上一个
let controller;  // 记住上一次的控制器

async function search(keyword) {
  if (controller) {
    controller.abort();  // 上一次请求还没回来?直接喊停
  }
  controller = new AbortController();

  try {
    const res = await fetch(`/api/search?q=${keyword}`, {
      signal: controller.signal,  // 把信号交给 fetch
    });
    return await res.json();
  } catch (err) {
    if (err.name === 'AbortError') return;  // 被自己取消的,不算真错误,静默忽略
    throw err;
  }
}

axios 同样支持 signal 选项(现代版本推荐用这个,而不是旧的 CancelToken),写法和原生 fetch 一致:axios.get(url, { signal: controller.signal })。

TanStack Query:signal 自动注入,大多数场景不用手写

如果每个用到"取消请求"的地方都要手动维护一个 controller 变量,代码会变得很啰嗦。好消息是:回顾前面 104/145 篇讲过的 useQuery,它的 queryFn 会自动收到第二个参数,里面包含一个现成的 signal,而且 TanStack Query 会在下面两种情况自动调用 abort():

  • 组件卸载——用户切换到别的页面,不再需要这个数据了
  • 依赖的 queryKey 变化——比如上面搜索关键字变了,旧请求会被自动取消,不需要手写"取消上一个"的逻辑
useQuery 的 queryFn 自动拿到 signal
function useSearch(keyword) {
  return useQuery({
    queryKey: ['search', keyword],
    queryFn: ({ signal }) =>       // ← 第二个参数里的 signal 是 TanStack Query 自动生成的
      request.get('/search', { params: { q: keyword }, signal }),
    enabled: keyword.length > 0,
  });
}
// keyword 从 "r" 变成 "re" 时,"r" 对应的请求会被 TanStack Query 自动 abort
// 不需要自己维护 controller 变量、不需要手动比对"是不是最新请求"
为什么这里“自动取消”比自己手写更可靠? 手写方案容易漏掉一种情况——组件在请求进行中被卸载(比如用户搜到一半直接跳走了页面),如果不取消,请求成功后的回调仍会尝试 setState,导致"在已卸载组件上调用 setState"的警告(这正好呼应本系列 Hooks 篇讲过的组件生命周期与副作用清理——useEffect 的 cleanup 函数就是用来做这类清理的)。TanStack Query 把"请求参数变化"和"组件卸载"两种取消时机都内置处理好了,不容易漏。

对比:手写 useEffect + AbortController vs 交给 TanStack Query

手写 useEffect 管理取消
useEffect(() => {
  const controller = new AbortController();
  fetchSearch(keyword, controller.signal)
    .then(setResult)
    .catch(err => {
      if (err.name !== 'AbortError') throw err;
    });
  return () => controller.abort();
  // cleanup:keyword 变化或组件卸载时执行
}, [keyword]);
// 还要自己管 loading/error 状态
交给 TanStack Query
const { data, isLoading } = useQuery({
  queryKey: ['search', keyword],
  queryFn: ({ signal }) =>
    request.get('/search', { params: { q: keyword }, signal }),
});
// 取消、loading、error、缓存全部内置
// 代码量少了一半,还不容易漏边界情况

什么时候还需要自己手动用 AbortController

不是所有取消场景都能靠 TanStack Query 兜底。比如一个和 queryKey 无关的独立操作——用户点了"开始导出大文件"按钮,又手动点了"取消导出",这种由用户主动触发的取消动作(而不是"参数变了自动废弃旧请求"),依然要自己创建 AbortController 并在按钮点击时调用 abort()。

判断标准: 如果取消的原因是"请求参数变了/组件卸载了",交给 TanStack Query 的自动机制;如果取消的原因是"用户主动点了取消按钮",需要自己持有 AbortController 实例,在事件处理函数里调用 abort()。

小结

  • 快速连续触发的请求存在竞态问题:网络慢的旧请求可能晚于新请求返回,把界面覆盖成过时数据。
  • AbortController 是浏览器原生的"请求喊停"机制:controller.signal 传给 fetch/axios,调用 controller.abort() 即可中止请求。
  • TanStack Query 的 queryFn 第二个参数会自动提供 signal,并在 queryKey 变化或组件卸载时自动调用 abort,多数场景不需要手写取消逻辑。
  • 自动取消机制呼应了 Hooks 篇讲过的副作用清理:请求进行中组件卸载时若不取消,容易触发"在已卸载组件上 setState"的警告。
  • 由用户主动触发的取消(比如点击"取消上传"按钮)仍需自己手动创建和调用 AbortController,TanStack Query 的自动机制只覆盖"参数变化/组件卸载"这两种场景。
