抽离复用逻辑：把重复的 stateful 逻辑变成 Hook

上一篇讲了自定义 Hook 是什么。这一篇用两个最经典的例子——useFetch 和 useLocalStorage——演示怎么把散落在各个组件里的重复逻辑抽出来，以及一个关键性质：每次调用状态都是独立的。

动机：从"复制粘贴的逻辑"开始

假设多个页面都要"进入时拉数据、显示 loading、处理错误"。如果每个组件都写一遍 useState(loading) + useState(data) + useEffect(fetch)，就是三份几乎一样的样板——改一处得改三处，还容易漏。这正是抽 Hook 的信号。

例子一：useFetch —— 把请求逻辑打包

useFetch.js —— 复用"拉数据"这套状态逻辑
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let ignore = false;       // 防竞态：url 变了旧请求别写 state
    setLoading(true);
    fetch(url)
      .then(r => r.json())
      .then(d => { if (!ignore) { setData(d); setLoading(false); } })
      .catch(e => { if (!ignore) { setError(e); setLoading(false); } });

    return () => { ignore = true; };  // 清理，避免过期写入
  }, [url]);              // url 变就重新拉

  return { data, loading, error };  // 返回对象，调用处按需解构
}
调用处：三行搞定，样板全没了
function UserProfile({ id }) {
  const { data, loading, error } = useFetch(`/api/users/${id}`);
  if (loading) return <Spinner />;
  if (error) return <Error msg={error.message} />;
  return <h1>{data.name}</h1>;
}

例子二：useLocalStorage —— 状态与本地存储同步

让一个 state 自动读写 localStorage——刷新页面还在。逻辑抽成 Hook，用起来就像加强版 useState。

useLocalStorage.js
function useLocalStorage(key, initial) {
  // 惰性初始化：只在首次挂载时读一次 localStorage（读磁盘不便宜）
  const [value, setValue] = useState(() => {
    const saved = localStorage.getItem(key);
    return saved ? JSON.parse(saved) : initial;
  });

  // value 变就写回，保持同步
  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];  // 模仿 useState 的返回形状，用起来无缝
}

// 用起来跟 useState 一模一样，但会持久化
const [theme, setTheme] = useLocalStorage('theme', 'light');

关键性质：每次调用，状态都独立

这是最容易误解的点。自定义 Hook 不共享状态——它只共享逻辑。两个组件各自调用 useFetch，会得到两份完全独立的 data/loading/error，互不影响。

A 调 useFetch
→
A 自己的 state
B 调 useFetch
→
B 自己的 state（互不影响）
为什么独立？因为 Hook 里的 useState/useEffect 是在调用它的那个组件的上下文里执行的。React 按"哪个组件、第几个 Hook"来存 state。useFetch 只是把 useState 的调用"转包"进去，state 依然挂在各自的组件上。类比：一个函数里声明的局部变量，每次调用都是新的一份——Hook 的 state 也是这个道理。
想共享状态怎么办？如果确实要多个组件共用同一份数据，自定义 Hook 做不到——那是 Context 或状态库的活（回顾第 50 篇）。别指望"两个组件调同一个 Hook 就能同步 state"，这是新手常见误解。Hook 复用逻辑，Context 共享状态。

小结

  • 发现多处重复的 useState+useEffect 组合，就抽成 useXxx 自定义 Hook。
  • useFetch/useLocalStorage 是最经典的两个例子：把请求/持久化逻辑打包复用。
  • 返回形状自定：模仿 useState 返回数组、或返回对象供解构。
  • 核心性质：每次调用状态独立——Hook 共享的是逻辑，不是状态；共享状态用 Context。

常见陷阱

  • 误以为调用同一个 Hook 会共享状态：这是最典型的误解——两个组件各自调用 useFetch(url)，会得到两份完全独立的 data/loading，改一个不会影响另一个。想要真正共享同一份数据，必须用 Context 或状态库，Hook 本身无法做到"跨组件同步"。
  • useFetch 里忘记处理竞态（race condition）：如果 url 快速切换（比如搜索框联想），旧请求可能比新请求慢返回，若不加 ignore 标志或 AbortController 判断，旧请求的结果会覆盖新请求的结果，界面显示错误数据。这也是为什么示例代码里一定要在 useEffect 的清理函数里置位 ignore。
  • useLocalStorage 每次渲染都读 localStorage：如果不用 useState 的惰性初始化写法（传函数而不是直接读取的结果），每次组件渲染都会重新执行 localStorage.getItem，虽然只在初始化那次真正生效，但代码语义上容易被误改成每次渲染都读，带来不必要的同步 I/O 开销。
  • 把不该抽的逻辑也强行抽成 Hook：只在一个地方用、且逻辑简单的 useState+useEffect 组合，抽成 Hook 反而多了一层间接，读代码时要多跳一次文件。抽离的信号应该是"确实有重复"或"逻辑复杂到污染组件主体"，不是"看起来像 Hook 就该抽"。
  • 返回值形状设计不一致：有的 Hook 返回数组（模仿 useState），有的返回对象，团队里如果没有统一约定，调用处经常要翻实现才知道怎么解构，增加认知负担。

面试高频考点

  1. 两个组件调用同一个自定义 Hook，它们的 state 是否共享？为什么？—— 不共享。Hook 内部的 useState/useEffect 实际运行在调用它的那个组件的渲染上下文里，React 按"组件 + 第几个 Hook 调用"来存取 state。自定义 Hook 只是把这些调用"打包转发"，state 依然各自挂在调用者身上，类似普通函数里的局部变量每次调用都是新的一份。
  2. useFetch 类 Hook 为什么要在 useEffect 里做清理（cleanup）？不做会有什么问题？—— 如果 url 在请求未完成时又变化，会触发新的请求，但旧请求的 .then 回调仍会在稍后执行。不加清理判断（如 ignore 标志），旧请求返回时会用过期数据覆盖新请求已经设置好的 state，造成界面显示与当前输入不一致的竞态 bug。
  3. 什么时候应该把逻辑抽成自定义 Hook，什么时候不需要？—— 判断标准是"重复"和"复杂度"：多个组件出现几乎相同的 state+effect 组合，或单个组件里这段逻辑本身足够复杂、影响组件主体可读性，就该抽。如果逻辑简单且只用一次，抽出来反而增加间接层，得不偿失。

与其他知识点的联系

  • useReducer 与 Context 结合（第 50 篇）：本篇陷阱里提到"想共享状态用 Context"，第 50 篇的 useTodos()/useTodosDispatch() 正是这种共享状态的自定义 Hook 封装——表面看都是"useXxx() 返回一份数据"，但第 50 篇内部走的是 useContext，多个组件调用会拿到同一份数据；本篇的 useFetch 内部是 useState，各自独立，这正是区分两者的关键。
  • useMemo 缓存计算（第 52 篇）与 useCallback 缓存函数（第 53 篇）：自定义 Hook 内部同样会用到这两个性能 Hook——比如 useFetch 如果要对返回的 data 做进一步派生计算，或者要把某个方法传给 memo 子组件，一样要考虑引用稳定性问题，抽 Hook 并不能绕开这些考量，只是把它们封装到了一处。
  • 过度优化的坏处（第 54 篇）：本篇陷阱"把不该抽的逻辑也强行抽成 Hook"和第 54 篇的核心思想同源——都是"没有实际收益却引入了额外的间接层/开销"，抽 Hook 和加 memo 都要基于"确实有重复/确实测到瓶颈"的证据，而不是看起来该做就做。
  • 父传子 Props（第 58 篇）：自定义 Hook 的返回值最终往往还是要通过 props 或直接在组件内使用，理解 Hook 返回的数据/函数如何被消费，离不开对 props 传递机制的理解——尤其是 Hook 返回对象/数组时的引用稳定性，会直接影响子组件是否被无谓重渲染。
  • 跨层级 Context 通信（第 61 篇）：本篇明确划清"Hook 复用逻辑，Context 共享状态"的边界——如果需求从"多处需要同样的处理逻辑"变成"多处需要读写同一份数据"，就该从自定义 Hook 切换到 Context，这是判断该用哪种方案的关键分界线。
