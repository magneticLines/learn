memo + useCallback + useMemo:三者缺一不可

上一篇结尾留了个坑:React.memo 遇到对象/函数 props 会失效。这一篇讲怎么补上——用 useCallback 稳定函数引用、useMemo 稳定对象引用。关键在于理解它们是一套组合:只用 memo 不稳定 props,或只稳定 props 不用 memo,都等于没做。

反例:只用了 memo

Child 用了 memo,但依然每次都渲染
const Child = React.memo(({ onClick }) => {
  console.log('Child 渲染');
  return <button onClick={onClick}>点我</button>;
});

function Parent() {
  const [count, setCount] = useState(0);
  // ❌ 每次 Parent 渲染,handleClick 都是全新函数(新引用)
  const handleClick = () => console.log('click');
  return (
    <>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <Child onClick={handleClick} />  {/* 新引用 → memo 失效 */}
    </>
  );
}

问题:为什么函数每次都是"新的"

组件每次渲染,函数体都会重新执行一遍。const handleClick = () => {...} 这行,每次执行都创建一个全新的函数对象。函数功能一模一样,但内存地址不同。memo 做浅比较时,prevProps.onClick === nextProps.onClick 永远是 false——所以 memo 判定"props 变了",老老实实重渲染。

正例:useCallback 稳定函数引用

useCallback 让函数在依赖不变时保持同一个引用
function Parent() {
  const [count, setCount] = useState(0);

  // ✅ 依赖数组为空 → 每次渲染返回同一个函数引用
  const handleClick = useCallback(() => {
    console.log('click');
  }, []);  // 依赖变了才生成新函数

  return (
    <>
      <button onClick={() => setCount(count + 1)}>+1</button>
      {/* 现在 onClick 引用稳定 → memo 生效 → Child 不再重渲染 */}
      <Child onClick={handleClick} />
    </>
  );
}

对象 props 用 useMemo

函数用 useCallback,对象/数组用 useMemo——本质是同一件事:让引用在依赖不变时保持稳定。(其实 useCallback(fn, deps) 就等价于 useMemo(() => fn, deps),前者只是给函数场景的语法糖。)

对象/数组 props 也要稳定引用
// ❌ 每次新对象
<Child config={{ theme: 'dark' }} />

// ✅ useMemo 稳定引用
const config = useMemo(() => ({ theme: 'dark' }), []);
<Child config={config} />

为什么"缺一不可"

你做了什么结果
只 memo,不稳定 props失效:props 引用每次变,memo 每次判"变了"
只 useCallback/useMemo,子组件没 memo失效:子组件没门卫,父渲染照样跟着渲染
memo + useCallback + useMemo 全上生效:引用稳定 + 有门卫 → 真正跳过重渲染
部分 props 稳定,还有一个是内联对象失效:浅比较只要一个字段不等就整体判"变"
最隐蔽的坑:一颗老鼠屎。你把 onClick 和 config 都稳定了,却漏了一个 style={{...}} 内联对象。浅比较只要任意一个字段引用不等,就判定整个 props 变化——memo 前功尽弃。所以要么全部 props 稳定,要么别指望 memo。这也是为什么"随手加 memo"经常没效果:配套工作没做全。

别过度使用:成本 vs 收益

useCallback / useMemo 不是免费的。它们自己要保存依赖数组、每次渲染要比较依赖、还要缓存上次的值——这些都有成本。如果子组件没被 memo 包裹,或子组件本身很轻量,那么"稳定引用"这件事纯属白做,反而增加了内存和比较开销、让代码更难读。

正确心态:useCallback/useMemo 主要是为了配合 memo 子组件或作为其他 Hook 的依赖项而存在的。没有这两个理由,就别提前包。React 19 的编译器目标就是自动完成这类记忆化,把开发者从手动 memo 的心智负担里解放出来。

小结

  • 函数/对象 props 每次渲染都是新引用,会让 React.memo 静默失效。
  • useCallback 稳定函数引用,useMemo 稳定对象/数组引用。
  • 三者是一套组合:memo(门卫)+ useCallback/useMemo(稳定引用),缺一不可;且所有对象/函数 props 都要稳定,漏一个就整体失效。
  • 别为了优化而优化——没有 memo 子组件或依赖项需求时,包裹 useCallback/useMemo 是净亏损。

常见陷阱

  • useCallback 依赖数组写空,但闭包里用了会变的变量。为了"稳定引用"图省事,把依赖数组直接写成 [],结果函数内部引用的 count 永远是第一次渲染时的值(经典的闭包陷阱)。稳定引用不能靠瞒报依赖换来,该有的依赖必须老实写全,真正想要"引用稳定又拿到最新值"应该用函数式更新或 useRef。
  • 以为 useCallback 本身能防止函数重新创建。实际上 useCallback 每次渲染时函数字面量还是会被重新求值一遍,只是 React 内部把新函数丢弃、返回缓存的旧引用——它省的是"引用变化",不是"函数体重新执行"的开销。
  • 只包了 onClick,漏了 style/config 等内联对象。浅比较是"全部字段都要相等",漏一个内联对象或数组,memo 照样整体判定为变化。要么把所有传给 memo 子组件的非原始类型 props 都稳定,要么这次优化就是无效功。
  • 给不会被 memo 包裹的子组件也套 useCallback。子组件不是 React.memo 包出来的,父组件重渲染时它本来就会重渲染,这时 useCallback 只增加了依赖比较和缓存的开销,没有任何收益。
  • 把 useMemo 当成"缓存计算结果"的万能钥匙,用在廉价计算上。比如 useMemo(() => a + b, [a, b]),加法本身比 useMemo 的比较、缓存开销还便宜,包一层纯属负优化。

面试高频考点

  1. 为什么 React.memo 配合内联箭头函数会失效?—— 因为 JSX 里 onClick={() => ...} 每次渲染都会创建一个新的函数对象,函数地址不同,memo 的 Object.is 浅比较结果为 false,判定 props 变化,照常重渲染。
  2. useCallback(fn, deps) 和 useMemo(() => fn, deps) 有什么关系?—— 前者是后者的语法糖,两者本质都是"依赖不变则返回同一个缓存值"的记忆化机制,区别只是 useCallback 缓存的是函数本身,useMemo 缓存的是函数的返回值。
  3. useCallback 的依赖数组写错(比如漏依赖)会导致什么问题?—— 会形成闭包陷阱:函数体内引用的外部变量停留在创建时的值,后续状态更新不会反映到这个函数里,表现为"点击没反应"或"用的是旧数据"这类隐蔽 bug。
  4. 什么时候不该用 useCallback/useMemo?—— 子组件没有被 memo 包裹、依赖的计算本身很廉价、或者结果没有被其他 Hook 当作依赖项时,包一层记忆化只有比较和缓存的成本、没有收益,属于过度优化。

与其他知识点的联系

  • React.memo 浅比较(上一篇):本篇是上一篇留下的坑的直接解法。理解浅比较"比引用不比内容"是前提,理解 useCallback/useMemo"依赖不变则返回同一引用"是解法,两篇合起来才是一套完整的记忆化优化链路。
  • 单一职责(第74篇):一个组件如果职责单一、props 简单,需要稳定的引用往往就一两个,配套工作量小、容易做全;职责混杂的大组件里对象/函数 props 一多,漏配一个就整体失效——这也是"拆分得好的组件更容易被性能优化"的又一个例证。
  • 容器组件 vs 展示组件(第75篇):useCallback/useMemo 通常出现在"容器"一侧(或替代容器的自定义 Hook 里),用来给下层的纯展示组件传递稳定的 props;展示组件本身不需要关心引用稳定性,它只管拿到 props 后怎么渲染。
  • 何时拆分组件(第76篇):该篇强调"性能驱动的拆分要用 Profiler 验证",这条原则同样适用于 useCallback/useMemo——不要凭直觉到处包一层,而要先确认子组件确实被 memo 包裹、确实存在可观测的重渲染浪费,再决定要不要引入这层记忆化。
