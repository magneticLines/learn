传参给事件处理函数

列表里的每一行都有个"删除"按钮，点击时需要告诉处理函数"删掉第几条"——这就是传参场景。直接写 onClick={handleDelete(id)} 是一个经典错误。

问题：为什么不能直接传参

onClick={handleDelete(id)} 在语法上等同于"先调用 handleDelete(id)，把返回值赋给 onClick"。这发生在渲染阶段，不是用户点击时。结果：

  • 每次渲染都会立即执行 handleDelete
  • 如果 handleDelete 调用了 setState，会触发无限渲染循环
  • 用户点击时什么都不发生（因为 onClick 的值是 undefined）

解法一：箭头函数包裹（最常用）

用箭头函数把带参数的调用包起来，onClick 的值变成了一个函数，点击时才执行：

箭头函数包裹传参
function TodoList() {
  const [todos, setTodos] = useState([
    { id: 1, text: '学 React' },
    { id: 2, text: '写项目' },
  ]);

  function handleDelete(id) {
    setTodos(todos.filter(t => t.id !== id));
  }

  return (
    <ul>
      {todos.map(todo => (
        <li key={todo.id}>
          {todo.text}
          {/* 箭头函数包裹：渲染时不执行，点击时才执行 */}
          <button onClick={() => handleDelete(todo.id)}>删除</button>
        </li>
      ))}
    </ul>
  );
}

解法二：返回函数的函数（柯里化风格）

让 handleDelete 本身接收 id，返回一个事件处理函数。这样可以直接写 onClick={handleDelete(id)}——但这里调用的是"生成处理函数"，不是"执行删除逻辑"：

柯里化：函数返回函数
// handleDelete(id) 返回的是一个新函数，这个函数才是真正的事件处理器
function handleDelete(id) {
  return function(e) {
    setTodos(todos.filter(t => t.id !== id));
  };
}

// JSX 里看起来像直接传参，实际是"生成处理函数"
<button onClick={handleDelete(todo.id)}>删除</button>

对比两种方案

箭头函数包裹
onClick={() => fn(id)}

优点：直观，随处可用
缺点：每次渲染创建新函数
     （通常无性能问题）
柯里化
onClick={fn(id)}
// fn 返回新函数

优点：JSX 看起来更简洁
缺点：同样每次渲染创建新函数，
     函数签名不直观

日常使用箭头函数包裹就够了——更直观，更常见，出错也更容易排查。

同时需要自定义参数和事件对象

有时候既要传自定义参数（如 id），又要用到事件对象（如 e.target.value）：

同时传参数和事件对象
function handleChange(id, e) {
  // id 是自定义参数，e 是事件对象
  console.log(id, e.target.value);
}

{/* 箭头函数接收事件对象 e，再转发给 handleChange */}
<input onChange={e => handleChange(item.id, e)} />
React 不传参时事件对象从哪来？
直接写 onClick={handleClick}，React 调用 handleClick(e) 时会自动把合成事件作为第一个参数传入。加了箭头函数包裹后，() => fn(id) 里的那个函数才是真正的 onClick 处理器，React 把事件传给它——箭头函数接不接收 e 都行，只要你不需要用它。

常见陷阱

  • 写成 onClick={handleDelete(id)} 却没意识到问题：这是本文开头就点出的经典错误，本质是把"函数调用表达式"当成了"函数引用"传给 onClick。危险之处在于如果 handleDelete 内部没有调用 setState，代码可能"看起来正常"（只是永远不会真正响应点击），只有当内部触发了状态更新时才会暴露成无限渲染循环，排查起来容易走偏。
  • 在循环里为每一项都写复杂的箭头函数，误以为会有明显性能问题：确实每次渲染都会为列表里每一项创建新的箭头函数，但这在绝大多数场景下性能影响可忽略——只有在列表极大（几千项以上）且子组件用了 React.memo 依赖 props 引用相等做优化时，才需要考虑用 useCallback 或事件委托（把事件绑在父级 ul 上，用 e.target.dataset 取参数）来减少函数创建。不要在没有性能问题的场景下过早优化。
  • 柯里化写法里忘记内层要返回函数：写 function handleDelete(id) { setTodos(...) } 却按柯里化的方式调用 onClick={handleDelete(id)}，这其实退化成了本文一开始警告的"直接调用"错误——柯里化能生效的前提是外层函数必须返回一个新函数，而不是直接执行副作用逻辑。
  • 混淆"事件对象"和"自定义参数"的顺序：原生 addEventListener 回调只接收事件对象，而箭头函数包裹场景里参数顺序完全由你自己定义（比如 (id, e) => handleChange(id, e)），容易在团队协作里因为参数顺序不统一造成读错、传错的问题，建议团队内统一约定参数顺序（如事件对象放最后）。

面试高频考点

  1. 为什么 onClick={handleClick(id)} 是错的，onClick={() => handleClick(id)} 是对的？—— JSX 中 {} 内是一个表达式，会在组件每次渲染（即调用组件函数）时立即求值。handleClick(id) 这个表达式在渲染阶段就被执行，其返回值（通常是 undefined）才是真正赋给 onClick 的值，因此点击时不会有任何行为，而渲染时逻辑却已经执行了。用箭头函数包裹后，() => handleClick(id) 本身作为一个函数值赋给 onClick，渲染阶段只是"创建"了这个函数，真正调用 handleClick(id) 要等到用户点击、React 调用这个箭头函数时才发生。
  2. 不传参数时，事件对象是怎么传给处理函数的？—— React 在触发事件时，会调用 onClick 上挂的函数，并把这次的 SyntheticEvent 对象作为第一个实参传入，等价于 onClickHandler(syntheticEvent)。如果你写的是 onClick={handleClick}，handleClick 的第一个形参自然拿到事件对象；但如果写了 onClick={() => handleClick(id)}，React 传的事件对象实际是传给了那个匿名箭头函数，而箭头函数没有转发它，所以 handleClick 内部拿不到事件对象，除非显式写成 e => handleClick(id, e)。
  3. 箭头函数包裹传参和柯里化传参，本质区别是什么？—— 两者最终赋给 onClick 的都必须是一个函数引用，区别在于"由谁在何时创建这个函数"。箭头函数包裹是在 JSX 里直接创建一个闭包函数；柯里化则是调用一个"高阶函数"（handleDelete(id)），该高阶函数内部返回一个新函数作为真正的处理器。二者性能上没有本质差异（都是每次渲染创建新函数），只是代码组织风格不同。

与其他知识点的联系

  • onClick / onChange / onSubmit：本节解决的是"给这些事件处理函数额外传参"的问题，前提是先掌握了 onClick 的基本绑定方式（传函数引用而非调用结果）——如果没理解 onClick={handleAdd()} 为什么错，也就很难理解为什么 onClick={handleDelete(id)} 同样是错的。
  • 闭包：箭头函数包裹传参 () => handleDelete(todo.id) 本质是利用闭包把 todo.id 这个值"锁"在函数内部，等用户点击时再读取——这和函数式更新一节提到的闭包陷阱是同一机制的两面：这里是"善用闭包传递当前渲染的值"，那里是"闭包捕获旧值导致的坑"。
  • 不可变更新：示例中的 handleDelete 用 todos.filter(t => t.id !== id) 生成新数组而不是 splice 原地删除，这是不可变更新原则在事件处理函数里的直接应用——传参的目的（知道删哪一项）和删除的实现方式（不可变）是两个独立但常常同时出现的知识点。
  • 性能与 useCallback：每次渲染都为列表里的每一项创建新的箭头函数，这个"看似浪费"的做法关联到 React.memo + useCallback 的优化场景——只有当子组件用 memo 依赖 props 引用相等时，才值得用 useCallback 缓存这些传参函数，否则是不必要的复杂度。
  • key 的作用与规范：示例代码里 todos.map 渲染列表项时用了 key={todo.id}，这与传参逻辑用的是同一个 id——用稳定的业务 id 而不是数组下标做 key，恰好也是给事件处理函数传参时能"认对目标项"的可靠依据。
