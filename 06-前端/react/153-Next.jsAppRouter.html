Next.js App Router — SSR / SSG / RSC

本系列从头到尾用的都是 Vite + CSR(客户端渲染):浏览器下载一个几乎空的 HTML、拉取 JS bundle、执行 React 才把页面画出来。这套方案简单直接,但天生有两个短板——首屏白屏时间、搜索引擎不友好。Next.js 就是为解决这两个问题而生的全栈框架,这一篇是承上启下的收尾,讲清它的三种渲染模式。

纯客户端渲染(CSR)的天然短板

回忆一下本系列最开始学的 index.html:body 里几乎只有一个空的 <div id="root"></div>。用户请求这个页面时,服务器什么内容都还没有,必须等浏览器下载完 JS bundle、执行 React、把组件渲染成真实 DOM,页面才第一次有内容。这个过程带来两个问题:

  • 首屏白屏时间长。 网络越差、bundle 越大,用户盯着白屏的时间越长——这个等待时间完全花在"下载和执行 JS"上,而不是"展示内容"上。
  • SEO 不友好。 搜索引擎爬虫抓取到的 HTML 里几乎没有实际内容(内容要等 JS 跑完才生成),内容型网站(博客、电商、官网)如果纯 CSR,搜索引擎很难正确收录页面的真实内容。
什么场景 CSR 完全够用? 后台管理系统、需要登录的工具型应用——这些场景不需要被搜索引擎收录,用户也能接受一次性加载后长时间停留使用,本系列的技术栈选择在这类场景下依然是最优解,不是所有项目都需要 Next.js。

三种渲染模式对比

模式渲染时机适合场景
SSR(服务端渲染)每次请求时,服务器实时渲染出完整 HTML 再返回内容因用户/请求而异,且要求首屏就有数据(如个性化仪表盘)
SSG(静态生成)构建(build)时提前渲染好 HTML,之后直接当静态文件分发内容对所有用户一致、更新不频繁(如博客文章、官网落地页)
RSC(React Server Components)组件本身运行在服务端,产出结果直接嵌入页面,不下发对应 JS不需要交互、只做数据展示的组件(如列表、详情页主体)
SSR vs SSG 怎么选? 判断标准很简单:这个页面的内容是否每个用户/每次请求都不同?不同就用 SSR(比如登录后的个人主页);相同就用 SSG(比如产品介绍页),因为 SSG 提前生成好静态文件,访问速度和成本都远优于每次请求都实时渲染的 SSR。

RSC 与"服务端渲染 HTML"的本质区别

这是最容易混淆的一点:RSC 不是"把 HTML 在服务端拼好",SSR 才是那个意思。RSC 的本质是——某些组件只在服务端运行,它的代码(包括用到的第三方库)根本不会被打包进客户端 bundle。

传统 SSR
组件代码在服务端跑一遍生成 HTML
之后同一份组件代码
还要在浏览器端再跑一遍(hydration)
才能挂上事件监听、变得可交互

→ 组件的 JS 代码客户端也需要下发
RSC
标记为 Server Component 的组件
只在服务端执行一次
产出结果(不是 HTML,是一种序列化的
描述格式)直接嵌入页面

→ 这个组件的 JS 代码
   客户端 bundle 里根本不存在

举例:一个"文章详情"组件如果需要引入一个很重的 Markdown 解析库来处理内容,用 RSC 写这个组件,这个 Markdown 解析库完全不会出现在浏览器要下载的 JS 里——因为它自始至终只在服务器上跑。这是 RSC 比传统 SSR 更进一步的地方:传统 SSR 只是让首屏有内容,但该下发的 JS 一点没少;RSC 能实实在在减少客户端 bundle 体积。

陷阱:RSC 不能有交互逻辑。 因为 Server Component 的代码根本不会发到浏览器,它内部不能使用 useState/useEffect/事件处理函数这些依赖客户端运行时的东西。需要交互的部分要显式标记 'use client',拆成一个 Client Component,两者可以混用,但初学者常常忘记加这个标记导致报错,或者反过来给所有组件都无脑加 'use client',把 RSC 的优势完全抵消掉。

和本系列前面内容的关系。 之前学的 TanStack Query 是在客户端管理服务端状态的缓存和请求时机;RSC 则是把"获取数据 + 渲染"这一步直接搬到服务端,某种程度上是对客户端数据获取模式的一种补充甚至替代——具体怎么取舍,是数据要不要交互、要不要缓存在客户端反复使用来决定的。两者不是二选一,实际项目里经常同时存在。

小结

  • 纯 CSR(本系列的 Vite 技术栈)存在首屏白屏时间长、SEO 不友好两个天然短板,但对不需要 SEO 的工具型应用依然是合适的选择。
  • SSR 每次请求实时渲染,适合内容因人而异的页面;SSG 构建时预渲染成静态文件,适合内容一致、更新不频繁的页面。
  • RSC 的本质是组件只在服务端运行,其代码根本不会打包进客户端 bundle,这与"把 HTML 在服务端拼好后还要在客户端重跑一遍"的传统 SSR 有本质区别。
  • Server Component 不能用 useState/事件处理等客户端能力,需要交互的部分要显式标记 'use client' 拆成 Client Component。
  • 三种模式可以在同一个 Next.js 应用里混用,根据每个页面/组件的数据特性和交互需求分别选择。
