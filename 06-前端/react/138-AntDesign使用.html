Ant Design 使用(按需引入 / 主题定制)

一个后台管理系统要用到 Button、Table、Form、Modal、DatePicker、message 提示……几十个组件,每个都要考虑无障碍、键盘操作、边界情况。从零造这些轮子不划算——这正是成熟组件库存在的意义。这一篇讲 Ant Design 怎么按需引入、怎么用 token 系统定制主题。

为什么不自己写这些组件

一个看起来简单的 Modal,真正做好要处理:焦点陷阱(Tab 键不能跳出弹窗)、Esc 关闭、打开时锁住背景滚动、关闭后焦点归还给触发按钮、屏幕阅读器的 aria-modal……这些细节自己写很容易漏掉,而且每个项目都要重新踩一遍坑。

Ant Design(简称 antd)这类成熟组件库把这些通用交互、样式规范、国际化、无障碍支持都封装好了,你只需要关注业务逻辑本身。它本质上是拿"灵活性"换"开发效率"——你不能随心所欲改内部实现,但换来的是几十个开箱即用、经过大量项目验证的组件。

基本用法:直接引入组件
import { Button, DatePicker, message } from 'antd';

function ReportForm() {
  const handleSubmit = () => {
    message.success('提交成功'); // 全局提示,不用自己写 Toast 组件
  };
  return (
    <div>
      <DatePicker onChange={(date) => console.log(date)} />
      <Button type="primary" onClick={handleSubmit}>提交</Button>
    </div>
  );
}

按需加载:为什么现在不用配 babel-plugin-import 了

早期用 antd,如果直接 import { Button } from 'antd',打包工具会把整个 antd 库都打进产物,体积暴涨。旧方案是装 babel-plugin-import,把这行代码在编译期改写成 import Button from 'antd/es/button' 这种精确路径引入,手动实现按需加载。

antd 5.x 之后把组件目录结构改成了 ESM(ES Module)+ 天然可 tree-shaking 的形式——现代打包工具(Vite、Webpack 5+)本身就能分析 import 语句,识别出你只用了 Button 和 DatePicker,自动把没用到的组件从最终产物里"摇掉"。所以现在不再需要额外插件,直接 import { Button } from 'antd' 就是按需加载。

怎么验证按需加载生效了? 打包后看产物体积,或用 vite-bundle-visualizer / webpack-bundle-analyzer 看依赖图——只用到的组件应该单独成块,而不是整个 antd 打包进主 bundle。

主题定制:ConfigProvider + token 系统

过去改 antd 主题要么覆盖 CSS(脆弱、优先级容易冲突),要么用 Less 变量重新编译整个库。antd 5.x 引入了 token 系统:所有组件的颜色、圆角、间距都从一份 token(设计变量)派生,改 token 相当于改"设计系统的源头",不用碰 CSS。

ConfigProvider:全局注入主题 token
import { ConfigProvider } from 'antd';

function App() {
  return (
    <ConfigProvider
      theme={{
        token: {
          colorPrimary: '#61dafb',  // 全局主色,所有组件的"主色调"都会跟着变
          borderRadius: 8,          // 全局圆角
        },
      }}
    >
      <YourApp />
    </ConfigProvider>
  );
}

ConfigProvider 包在应用最外层,内部所有 antd 组件都会读取这份 token 来渲染。也支持给单个组件局部覆盖(嵌套一层 ConfigProvider),或者切换到内置的暗色算法 theme.darkAlgorithm。

常见坑:组件库样式和项目自身 CSS(如 Tailwind)冲突。 antd 组件自带一套 CSS 类和默认样式,如果项目同时用 Tailwind 的全局 reset 或工具类,容易出现"改了 className 却不生效"——本质是CSS 优先级或加载顺序的问题。常见解法:① 用 antd 的 cssVar / token 定制而不是硬覆盖 class;② Tailwind 配置里加 important 或调整 corePlugins.preflight;③ 用 antd 提供的 @ant-design/cssinjs 控制样式注入顺序,确保自定义样式在 antd 样式之后加载。遇到"明明写了样式却不生效",先检查是不是这个优先级战争。

小结

  • 成熟组件库(如 Ant Design)把无障碍、交互细节、国际化这些通用但繁琐的工作封装好,换来的是开发效率,代价是牺牲部分底层灵活性。
  • antd 5.x 组件目录天然支持 tree-shaking,直接 import { Button } from 'antd' 即可按需加载,不再需要 babel-plugin-import。
  • ConfigProvider + token 系统是新版主题定制的标准方式:改 colorPrimary、borderRadius 等 token,所有组件联动变化。
  • 组件库自带样式容易和项目自身 CSS(尤其 Tailwind)产生优先级冲突,优先用官方的定制机制而不是硬覆盖 CSS。
