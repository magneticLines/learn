React · 组件通信

父传子:Props——最基本的通信方式

组件不是孤岛,它们要互相传递数据。最基础的一种:父组件通过 props 把数据交给子组件。理解它,就理解了 React "单向数据流" 的起点。

props 是什么:父组件调用子组件时传的"参数"

把组件想成一个函数——它接收输入,返回 UI。这个输入就是 props(properties 的缩写)。父组件在 JSX 里写子组件标签时,标签上的属性就成了子组件收到的 props 对象。

父组件传,子组件收
// 子组件:声明一个参数 props,它是一个对象
function Greeting(props) {
  return <h1>你好, {props.name}!</h1>;
}

// 父组件:像给函数传参一样,在标签上写属性
function App() {
  return (
    <div>
      <Greeting name="Shea" />   // props = { name: 'Shea' }
      <Greeting name="Tom" />    // 同一组件,不同数据
    </div>
  );
}

字符串用引号,其他类型(数字、布尔、对象、数组、函数)都要用大括号 {} 包起来传:

传递非字符串:一律用大括号
<User
  name="Shea"                 // 字符串:引号
  age={25}                   // 数字:大括号
  isAdmin={true}             // 布尔:大括号
  tags={['a', 'b']}          // 数组
  profile={{ city: 'BJ' }}   // 对象(注意双大括号:外层是 JSX 表达式,内层是对象)
/>

解构写法:更简洁地接收 props

直接写 props.name 略啰嗦,实践中几乎都用解构把需要的字段拆出来,还能顺手写默认值:

解构 + 默认值(最常见的写法)
// 在参数位置直接解构,一眼看清这个组件"吃"哪些 props
function Greeting({ name, greeting = '你好' }) {
  return <h1>{greeting}, {name}!</h1>;
}

<Greeting name="Shea" />              // 你好, Shea!(greeting 用默认值)
<Greeting name="Tom" greeting="欢迎" /> // 欢迎, Tom!

核心铁律:props 是只读的

子组件绝对不能修改自己收到的 props。React 把这称为组件必须像"纯函数"一样对待 props——给同样的输入,永远返回同样的输出,不产生副作用。

❌ 反例:改 props
function Item({ item }) {
  // 直接改 props——禁止!
  item.done = true;
  return <li>{item.text}</li>;
}
✅ 正例:通知父组件去改
function Item({ item, onToggle }) {
  // 不改数据,只把"意图"报给父组件
  return (
    <li onClick={onToggle}>
      {item.text}
    </li>
  );
}

为什么这么设计? 数据的"所有权"属于创建它的组件(通常是父组件)。如果子组件能随手改 props,数据就有了多个修改源,追踪 bug 会变成噩梦——你不知道到底是谁改了它。React 强制 数据只能由拥有者修改,让数据流可预测。

单向数据流(one-way data flow)
props 只能从父流向子,像瀑布一样自上而下,不能倒流。父组件的 state 变化 → 传给子组件的 props 变化 → 子组件重新渲染。这种"单向"约束让整个应用的数据走向清晰:想知道某个值从哪来,顺着组件树往上找即可。这正是 React 可维护性的基石,也是它与"双向绑定"框架(如早期 Angular)最大的哲学差异。

props 变化会触发子组件重渲染

props 不是"传一次就固定"。当父组件重新渲染、传下来的 props 值变了,子组件会自动跟着重新渲染。这就是数据向下流动的动态过程:

父 state 变化
→
父重渲染
→
新 props 传下
→
子重渲染
常见误区:props 不是"变量赋值一次"
初学者常以为 <Child value={count} /> 只在挂载时读一次 count。其实每次父组件渲染,这行 JSX 都会重新执行,把最新的 count 传下去。子组件永远拿到的是父组件当前渲染时刻的值。

props vs state:两个最易混淆的概念

维度propsstate
来源父组件传入组件自己用 useState 创建
可变性只读,子组件不能改可变,通过 setter 修改
归属数据属于父组件数据属于组件自身
作用组件间通信(向下传)组件内部记忆

一句话记:props 是"别人给的",state 是"自己的"。给的东西不能改,自己的东西才能改。

常见陷阱

  • 误以为 props 是"传一次就固定"的静态值：新手常把 <Child value={count} /> 理解成挂载时的一次性赋值，实际上父组件每次渲染都会重新执行这行 JSX，把当前时刻最新的 count 传下去——props 是动态跟随父组件渲染的，不是快照。
  • 在子组件里直接修改 props：比如 item.done = true 这种直接改传入对象的写法，即使页面暂时"看起来"生效了（因为改的是同一个对象引用），也破坏了数据只能由拥有者修改的约定，会导致父组件不知情、下次渲染状态不一致，且 React 无法感知这次修改从而不会触发正确的重渲染。
  • 忘记非字符串类型要用大括号包裹：写 age="25" 传的是字符串 "25" 而不是数字 25，如果子组件里做数值运算（比如 age + 1），会得到字符串拼接的结果 "251" 而不是 26，这是非常隐蔽的类型 bug。
  • 对象类型 prop 用内联字面量传递：profile={{ city: 'BJ' }} 这种写法每次渲染都创建一个新对象，即使内容一样引用也不同，如果子组件被 memo 包裹，这个 prop 的浅比较永远判定"变了"，导致 memo 优化失效（可回顾第 54 篇"过度优化的坏处"）。
  • 过度解构导致丢失原始 props 引用：全部解构出来使用固然清晰，但如果后续要把"剩余的所有 props"原样转发给更底层的 DOM 元素（常见于封装通用组件），忘了用 ...rest 收集剩余属性，会导致某些自定义属性传不下去。

面试高频考点

  1. 为什么 React 规定 props 必须只读？如果子组件直接修改会有什么后果？—— 数据的所有权应该属于创建它的组件，如果子组件可以随意修改自己收到的 props，数据就有了多个可能的修改源，追踪"这个值到底被谁改了"会变得极其困难。React 要求组件像纯函数一样对待 props——相同输入始终产生相同输出，这是让数据流可预测、可调试的基础。
  2. 什么是单向数据流？它解决了什么问题？—— 单向数据流指 props 只能从父组件流向子组件，不能反向修改，数据变化路径固定为"父 state 变 → 父重渲 → 新 props 传下 → 子重渲"。它让"某个值从哪来"可以顺着组件树往上找到唯一来源，避免了双向绑定框架里"数据在哪个环节被改了"难以追踪的问题，是 React 可维护性和可测试性的基石。
  3. props 和 state 的核心区别是什么？—— props 是外部（父组件）传入的、只读的数据，归属权在父组件；state 是组件通过 useState 自己创建和管理的、可变的数据，归属权在组件自身。一句话：props 是"别人给的"，state 是"自己的"，只有自己的东西才能用 setter 修改。
  4. 子组件想修改父组件传来的数据，正确做法是什么？—— 子组件不能直接改 props，而应该通过父组件同时传入的回调函数（如 onToggle）把"修改意图"报告给父组件，由父组件在自己的作用域内调用 setState 完成真正的修改，修改后新的 state 会作为新的 props 重新传给子组件，形成闭环（下一篇"子传父：回调函数"详细讲）。

与其他知识点的联系

  • 子传父:回调函数（第 59 篇）：本篇讲"数据只能向下流"，下一篇正是解决"子组件想反馈信息给父组件"这一缺口的方案——把函数当作一种特殊的 props 传下去，子组件调用它来完成"事件向上"，两篇合起来才是完整的父子通信闭环。
  • 兄弟组件状态提升（第 60 篇）：状态提升本质就是本篇"父传子 props"和下一篇"子传父回调"的组合应用——父组件把 state 通过 props 传给一个兄弟（读），再通过回调从另一个兄弟接收修改（改），没有这两个基础模式，状态提升无从谈起。
  • 引用稳定性问题（第 51 篇）：本篇陷阱里提到"对象类型 prop 用内联字面量传递会让 memo 失效"，这正是第 51 篇要系统讲解的"每次渲染都创建新引用"问题在 props 场景下的具体表现——理解了那篇的原理，才能理解为什么 profile={{ city: 'BJ' }} 这种写法有隐藏成本。
  • useMemo 缓存计算（第 52 篇）与 useCallback 缓存函数（第 53 篇）：如果确实需要把对象或函数当 props 传给 memo 子组件，又想避免上面这个陷阱，这两个 Hook 就是标准解法——先在这里认识到"props 引用不稳定会破坏 memo"，后面两篇才是针对性的修复工具。
  • 跨层级 Context 通信（第 61 篇）：props 是"一层传一层"的显式通道，当组件层级很深、中间层被迫透传无关数据（prop drilling）时，就是本篇方式的局限所在——第 61 篇的 Context 正是为突破这个局限而存在的下一级方案。

小结

  • props 是父组件调用子组件时传入的数据,子组件通过参数(对象)接收,常用解构 + 默认值写法。
  • 字符串用引号,其余类型一律用大括号 {} 传。
  • props 只读:子组件不能修改,数据只能由拥有者(父组件)修改。
  • props 体现 React 的单向数据流——数据自上而下流动,可预测、易追踪。
  • props 变化会自动触发子组件重渲染,它是动态的,不是"传一次就固定"。
