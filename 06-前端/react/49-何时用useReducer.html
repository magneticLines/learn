React · useReducer

何时用 useReducer：与 useState 的取舍

useReducer 不是 useState 的升级替代，二者是不同场景的工具。用错了会让简单的事情变复杂，用对了能让复杂的状态逻辑清爽可控。这一篇给出判断标准。

它俩其实是一回事的两种形态

本质上 useState 就是 useReducer 的一个特例——React 内部的 useState 可以用一个"把新值原样返回"的 reducer 实现。区别只在于"更新逻辑写在哪、由谁负责"。

useState
更新逻辑散落在
各个事件处理函数里，
"新值是什么"由调用处决定
useReducer
更新逻辑集中在
reducer 一个函数里，
调用处只"描述发生了什么"

判断标准：这四种信号出现，就该考虑 reducer

  • 状态复杂：state 是有多个字段的对象/嵌套结构，而不是单个值。
  • 字段关联：改一个字段常常要连带改另几个（比如切换分页要同时重置选中项和滚动位置）。
  • 更新方式多：同一份 state 有很多种不同的更新动作（增/删/改/重置/批量……）。
  • 下一状态依赖前一状态：状态转移有明确规则，像个状态机。

反过来，如果只是一个布尔开关、一个输入框的值、一个计数——这种独立、简单的状态，useState 更直接，硬套 reducer 是过度设计。

对比：一个"多字段关联"的例子

useState：更新逻辑散落
const [items, setItems] = useState([])
const [selected, setSelected] = useState(null)
const [editing, setEditing] = useState(false)

function addItem(x) {
  setItems([...items, x])
  setSelected(x.id)     // 关联更新
  setEditing(true)      // 又一处
}
// 三次 setXxx 要记得"一起改"，
// 漏一个就状态不一致
useReducer：转移集中
function reducer(s, a) {
  switch (a.type) {
    case 'add':
      return {         // 一次算出完整新状态
        items: [...s.items, a.item],
        selected: a.item.id,
        editing: true,
      }
    // ...其他动作
  }
}
dispatch({ type: 'add', item: x })
// 一处派发，转移规则不会漏

好处一：状态转移集中，可读可控

所有"怎么从旧状态到新状态"的规则都收拢在 reducer 里，一眼看清这份状态有哪些合法变化。改 bug 时不用在十几个事件处理器里翻找"到底哪里改了这个字段"——只看 reducer 的 case 就行。

联系状态机思想：reducer 天然把状态建模成"状态 + 转移"。这和有限状态机（FSM）思路一致：当前在什么状态、收到什么事件、转移到什么新状态，规则显式且封闭。复杂交互（如多步表单、拖拽、播放器）用这个心智模型会清晰很多。

好处二：可测试性强

reducer 是纯函数——不依赖 React、不依赖组件、没有副作用。你可以脱离整个渲染环境，像测普通函数一样直接测它：给定输入 state 和 action，断言输出 state。这是 useState 那种"逻辑嵌在组件里"很难做到的。

reducer 单元测试——无需渲染组件
test('increment 让 count +1', () => {
  const next = counterReducer(
    { count: 0 },              // 输入 state
    { type: 'increment' }       // 输入 action
  );
  expect(next.count).toBe(1);  // 纯函数，输入定则输出定
});

决策速查

场景推荐
单个布尔/字符串/数字useState
几个互不相关的独立状态多个 useState
多字段、字段间有关联的状态对象useReducer
更新动作种类多、有明确转移规则useReducer
逻辑复杂、想单独测试更新逻辑useReducer
状态需在多组件共享(配合 Context)useReducer(下一篇)
实用建议：不确定就先用 useState。当你发现"一个操作要同时 setX、setY、setZ"、或者 setter 越来越多、更新逻辑开始重复时，这就是重构成 useReducer 的信号。别一开始就为"可能会复杂"而套 reducer——那是过早设计。

小结

  • useState 是 useReducer 的特例，区别在"更新逻辑写在哪"。
  • 状态复杂/字段关联/更新方式多/依赖前态时选 useReducer。
  • 好处：转移逻辑集中好维护、reducer 是纯函数可独立测试。
  • 简单独立状态别硬上 reducer；不确定先 useState，复杂了再重构。
