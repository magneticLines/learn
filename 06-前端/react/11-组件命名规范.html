组件命名规范：为什么必须 PascalCase

PascalCase 不是"最好写成这样"，而是"必须这样写"——否则 React 无法区分你的组件和 HTML 原生标签。这背后是 JSX 编译规则，理解这个规则，你就彻底明白了为什么。

PascalCase 是什么

PascalCase（帕斯卡命名法）：每个单词的首字母都大写，单词之间无分隔符。

✅ PascalCase 示例
UserProfile
NavBar
LoginForm
ShoppingCart
ProductDetailPage
其他命名风格（不用于 React 组件）
userProfile    ← camelCase（变量/函数）
nav-bar        ← kebab-case（HTML 属性）
login_form     ← snake_case（Python 风格）
BUTTON         ← SCREAMING_SNAKE（常量）

为什么强制大写开头：JSX 编译规则决定的

这是本节最核心的内容。JSX 不是真正的 HTML，它会被 Babel 编译成 React.createElement() 调用。标签首字母的大小写，决定了第一个参数是字符串还是函数引用。

JSX 编译过程——字符串 vs 函数引用
// JSX 你写的：
<div className="box">内容</div>
<UserCard name="Alice" />

// Babel 编译后：
createElement('div', { className: 'box' }, '内容')
//              ↑ 字符串 → 交给浏览器创建原生 DOM 节点

createElement(UserCard, { name: 'Alice' })
//              ↑ 函数引用 → React 调用这个函数，拿到 JSX 继续处理

关键点：Babel 看的是标签名的首字母，不是你起的名字叫什么。大写 → 在当前作用域找同名变量（你定义的组件函数）；小写 → 直接传字符串给 React，让浏览器处理 HTML。这个规则写死在 Babel 的 JSX 转换逻辑里，没有例外。

如果组件小写开头会发生什么
// 定义时小写——语法上合法，但用起来出问题
function userCard() {
  return <div>用户卡片</div>;
}

// 调用时：
<userCard />

// 编译后：
createElement('userCard', null)
// React 把它当成 HTML 标签字符串，不调用你的函数
// 浏览器不认识 <userCard>，渲染成空的自定义元素
// 页面空白，且不报"找不到组件"的错误——很难排查

正确 vs 错误命名对照表

✅ 正确❌ 错误错误原因
UserProfile userProfile 首字母小写，Babel 编译成字符串，React 当 HTML 处理
NavBar nav-bar 连字符是 HTML 属性风格，在 JS 里不是合法标识符
LoginForm login_form 下划线是 Python/snake_case 风格，和 React 生态不一致
Button BUTTON 全大写容易和常量混淆，可读性差

文件命名约定：与组件名一致

社区约定（也是 Create React App、Vite 模板的默认做法）：文件名和其中导出的组件名保持完全一致。

src/components/
  UserCard.jsx    ← 文件名 = 导出组件名
  NavBar.jsx
  LoginForm.jsx
  Button.jsx
UserCard.jsx
// 文件名 UserCard.jsx，导出名也是 UserCard
// 这样 IDE 自动导入时路径和名称都能对上
export default function UserCard({ name, avatar }) {
  return (
    <div className="user-card">
      <img src={avatar} alt={name} />
      <span>{name}</span>
    </div>
  );
}

这个约定的好处：看到 import UserCard from './UserCard'，你立刻知道去哪个文件找它；ESLint 的 react/display-name 规则也能正确识别组件名用于调试。

高频问题：组件渲染出来是空白，不报错
遇到这种情况，第一反应是检查组件名是否小写开头。症状：写了组件、传了 props，页面就是不显示，React DevTools 也找不到你的组件——十有八九是组件名首字母小写了，导致 React 把它当成 HTML 未知标签，安静地渲染成空节点。这是 React 里最"静默"的错误之一，不抛异常，只是什么都不渲染。
延伸：动态组件名必须用大写变量
如果你要动态渲染组件（比如根据 type 决定渲染哪个组件），必须把它赋给大写变量：
const Component = componentMap[type]; 然后 <Component />
不能直接写 <componentMap[type] />——JSX 里不支持表达式作为标签名，必须先提取到大写变量。
小结：① PascalCase = 每个单词首字母大写，无分隔符；② 强制大写的原因是 Babel 编译规则——大写编译成函数引用，小写编译成 HTML 字符串；③ 文件名与组件名保持一致，方便 IDE 和 ESLint；④ 静默空白 + 无报错 = 先查组件名首字母。

常见陷阱

  • 组件名小写开头导致静默空白渲染：这是本节最核心的坑——写了 function userCard() 并用 <userCard /> 调用，React 不会报"找不到组件"，因为 Babel 把它编译成了字符串 'userCard'，浏览器把它当成一个未知自定义 HTML 标签渲染成空节点。没有报错、没有警告，只有页面空白，是最难排查的一类问题。
  • 动态渲染组件时直接用小写变量或表达式做标签名：写 <componentMap[type] /> 这种形式，JSX 语法本身就不支持用表达式当标签名；即便先赋值给一个小写变量 const comp = UserCard 再写 <comp />，Babel 依然会因为首字母小写把它当字符串处理。必须赋值给大写开头的变量才能被识别为组件引用。
  • 把 HTML 原生标签名当组件名复用：给自定义组件取名 Table、Form 这类和原生标签同名（大写后）的名字本身没问题（因为大小写不同，不会冲突），但容易在阅读代码时和原生标签混淆，尤其是嵌套使用时不容易一眼看出哪层是自定义组件、哪层是原生 DOM 标签。
  • 文件名和导出组件名不一致：比如文件叫 Card.jsx 却导出 export default function UserCard()。这不会导致运行时错误，但会让 IDE 的自动导入、ESLint 的组件规则、以及团队协作时的可读性全部打折——看到文件名猜不到组件叫什么，看到 import 语句也对不上文件路径。

面试高频考点

  1. 为什么 React 组件名必须大写开头？如果写成小写会发生什么？—— 因为 JSX 编译时，Babel 只看标签首字母的大小写来决定生成的代码：大写会被当作作用域内的变量（函数/类）引用，编译成 createElement(ComponentName, ...)；小写会被当作字符串字面量，编译成 createElement('tagname', ...) 交给浏览器当原生标签处理。如果组件名小写开头，React 会把它当成未知的原生标签，安静地渲染成空节点，不会抛出任何异常。
  2. 如果要动态选择渲染哪个组件（比如根据数据类型渲染不同组件），该怎么写？—— 必须先把目标组件赋值给一个大写开头的变量（如 const Component = componentMap[type]），再用 <Component /> 渲染。JSX 不支持把表达式直接写在标签位置，也不认变量名大小写以外的其他判断依据，所以"赋值给大写变量"是唯一能让 Babel 生成函数引用调用的方式。
  3. 组件命名和文件命名的最佳实践是什么，为什么？—— 组件名用 PascalCase，文件名与其导出的组件名保持一致（如 UserCard.jsx 导出 UserCard）。这不是语法要求，而是社区约定：能让 IDE 自动导入的路径和组件名对应，让 ESLint 的 display-name 等规则正确识别，也让团队成员通过文件名就能猜到组件是什么。

与其他知识点的联系

  • 关联定义与调用组件（上一章）：上一章讲"函数就是组件"，本章补上"这个函数必须大写命名"这条硬性约束——两节合起来才是一个可用组件的完整定义，缺了命名规则，写出的函数在 JSX 里根本不会被当成组件对待。
  • 关联 JSX 编译与 createElement：本节反复提到的"Babel 编译规则"背后就是 JSX → React.createElement 的转换过程，命名规范表面是风格问题，实质是在利用这套编译期的静态规则区分"组件"和"DOM 标签"。
  • 关联项目目录结构规范：无论按 type 还是按 feature 分层组织项目，"文件名与组件名一致"这条约定都要遵守，目录解决"放在哪个文件夹"，命名规范解决"这个文件和其中的组件该叫什么"，两者共同支撑项目的可维护性。
  • 关联条件渲染与动态组件：本节"动态组件名必须赋给大写变量"的规则，会在后续写条件渲染、路由懒加载、根据类型渲染不同表单组件等场景反复用到，是那些场景能正常工作的前提条件。
  • 关联 ESLint 与团队规范：react/display-name、react/jsx-pascal-case 这类 ESLint 规则就是把本节的命名约定自动化检查，团队项目里通常会强制开启，避免命名问题在代码评审阶段才被发现。
