React · useReducer

reducer 函数模式：把"怎么改"集中起来

useState 是"直接告诉 React 新值是什么"。useReducer 换了个思路：组件只负责"描述发生了什么事"（派发 action），至于状态该怎么变，交给一个独立的 reducer 函数统一决定。

reducer 是什么：(state, action) => newState

reducer 就是一个纯函数：接收"当前状态"和"一个动作描述"，返回"新状态"。它不修改传入的 state，而是算出并返回一个全新的 state。

一个计数器 reducer
// 纯函数：相同输入必得相同输出，无副作用，不改原 state
function counterReducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + 1 };
    case 'decrement':
      return { ...state, count: state.count - 1 };
    case 'reset':
      return { count: action.payload };
    default:
      throw new Error('未知 action: ' + action.type);
  }
}
为什么叫 reducer？它和数组的 Array.prototype.reduce 同源思想：reduce 把"累加器 + 当前元素"合成"新累加器"。这里是把"当前 state + action"合成"新 state"。一连串 action 依次 reduce 下来，就得到最终状态。React 内部维护的正是这个"折叠"过程。

action 的约定

action 是一个描述"发生了什么"的普通对象。社区约定：必带一个 type 字段（字符串，标识动作类型），需要附带数据时放在 payload（或直接平铺字段）里。

action 对象长什么样
{ type: 'increment' }                       // 无数据的动作
{ type: 'reset', payload: 0 }             // 带数据的动作
{ type: 'todo/added', text: '学 reducer' }  // 数据平铺也可以

关键在于：action 只描述意图（"用户点了加一"），不包含具体怎么改 state 的逻辑。改的逻辑全在 reducer 里。这种"意图与实现分离"是 reducer 模式的灵魂。

useReducer 基本用法

useReducer(reducer, initialState) 返回 [state, dispatch]。state 是当前状态；dispatch(action) 用来派发动作——它会把当前 state 和你给的 action 喂给 reducer，用返回值作为新 state。

Counter.jsx —— 组件里使用
import { useReducer } from 'react';

function Counter() {
  // state = 当前状态; dispatch = 派发 action 的函数
  const [state, dispatch] = useReducer(counterReducer, { count: 0 });

  return (
    <div>
      <span>{state.count}</span>
      // 组件只"派发意图"，不关心 count 具体怎么算
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'reset', payload: 0 })}>reset</button>
    </div>
  );
}

与 Redux 同源

如果你了解 Redux，会发现 useReducer 就是 Redux 核心思想的"内置迷你版"：单向数据流、纯函数 reducer、通过 dispatch action 更新状态。区别是 useReducer 的状态只活在组件内（不是全局单一 store），也没有中间件、DevTools 时间旅行等生态。

reducer 必须是纯函数：不能在里面发请求、改 DOM、写 localStorage、直接 mutate 传入的 state。原因和 useState 一样——React 可能在开发模式(严格模式)下故意调用两次来帮你发现副作用；不纯的 reducer 会产生不可预测的行为。副作用应放在事件处理或 useEffect 里。

小结

  • reducer 是纯函数 (state, action) => newState，把"状态如何变化"集中到一处。
  • action 是"描述发生了什么"的对象，约定带 type，数据放 payload。
  • useReducer(reducer, init) 返回 [state, dispatch]；组件只派发意图，不写更新逻辑。
  • 思想与 Redux 同源（单向流 + 纯 reducer + dispatch），是其内置轻量版。

常见陷阱

  • 在 reducer 里直接 mutate state：写成 state.count++; return state; 而不是返回新对象。React 用引用比较判断是否要重渲——你改的还是同一个对象引用，React 会认为"state 没变"，页面不更新，还破坏了严格模式下"重复调用 reducer 检测副作用"的假设。
  • 在 reducer 里搞副作用：在 case 里发请求、写 localStorage、操作 DOM。reducer 必须是纯函数，React（尤其严格模式下）可能多次调用同一个 reducer 来帮你揪出不纯的逻辑，副作用被多次触发就会出诡异 bug。正确做法是把副作用放到 dispatch 之后的 useEffect 或事件处理函数里。
  • 忘记写 default 分支，或者 default 里啥都不干却没 return：漏掉 default: return state;，遇到未知 action 时函数末尾隐式 return undefined，state 直接变成 undefined，后续读取字段全部报错。
  • action.type 拼错却没有报错：dispatch({ type: 'incremnt' }) 这种拼写错误，如果 reducer 的 default 只是安静地 return state，界面会"看起来什么都没发生"，调试起来毫无线索。养成 default 里 throw 或至少 console.warn 的习惯能提前暴露问题。
  • action 里塞了过多逻辑判断，寄望组件端先算好"新值"再传：比如 dispatch({ type: 'set', value: state.count + 1 })，这样"怎么变"的逻辑又跑回组件里了，违背了 reducer 模式"集中决策"的初衷，也让 action 不再是单纯描述"发生了什么"。

面试高频考点

  1. 为什么 reducer 必须是纯函数？—— 因为 React（尤其是严格模式）可能出于检测副作用的目的多次调用同一个 reducer；纯函数保证"相同输入一定产出相同输出、不产生副作用"，这样重复调用不会破坏状态一致性，也让 reducer 可以脱离组件单独做单元测试。
  2. action 一定要有 type 字段吗？为什么社区都这么约定？—— 不是语法强制，是约定。type 是 reducer 里 switch 分支判断的依据，统一用字符串常量便于查找引用、避免拼写不一致导致的静默失败，也方便未来接入 Redux DevTools 之类工具做日志和时间旅行调试。
  3. reducer 模式和面向对象里的命令模式（Command Pattern）有什么相似之处？—— 两者思路一致：把"要做什么"封装成一个独立的数据对象（action / command），和"怎么执行"的逻辑（reducer / handler）解耦。调用方只需要构造描述意图的对象并转交出去，不需要知道具体如何实现，这样修改执行逻辑时不影响调用方代码。
  4. 如果 reducer 返回了和传入 state 全等（===）的引用，会发生什么？—— React 会认为状态没有变化，跳过这次重渲染（这也是为什么"某个 case 不需要改变状态"时可以直接 return state）。这依赖于 reducer 返回新对象而非 mutate 原对象的约定——如果你 mutate 后返回同一个引用，即便数据变了，React 也会误判为"没变"。

与其他知识点的联系

  • 避免 Context 滥用（上一篇）：上一篇"优化二：拆 state 与 dispatch"用到的正是本篇的 useReducer——dispatch 引用永远稳定，把它单独放进一个 Context，只触发更新、不读数据的组件就不会因为 state 变化而重渲。这也是 useReducer 和 Context 组合、手写一个轻量全局状态方案的经典模式。
  • useState：本篇开篇就点明了 useState 和 useReducer 的思路差异——useState 是"直接告诉 React 新值"，useReducer 是"描述发生了什么，交给 reducer 决定怎么变"。当组件里有多个 useState 互相关联、更新逻辑分散在各个事件处理器里时，正是切换到 useReducer 的信号——它把"怎么改"从组件里抽出来，集中到一个纯函数里。
  • useEffect 依赖数组：本篇提到 dispatch 函数引用稳定、不必放进依赖数组，这和 useState 的 setter 是稳定值的道理完全一致——依赖数组篇讲的"哪些值算响应式值、哪些是稳定值不用列"的判断标准，同样适用于 useReducer 返回的 dispatch。
  • createContext 与 Provider：reducer 模式"意图与实现分离"的思想——组件只 dispatch 描述意图，具体怎么改交给 reducer——和 Context 的"传输管道"定位可以很好地配合：Provider 内部用 useReducer 管理状态，再把 state 和 dispatch 分别通过 Context 下发，子组件既不知道数据怎么存的，也不知道更新逻辑怎么写的，只管读数据、派发动作。
