index 路由:父路径精确匹配时的默认子路由

访问 /app(而不是 /app/dashboard)时,Outlet 里该显示什么?如果没配,页面就是空的。index 路由正是回答这个问题:它是父路由"精确命中时"的默认内容。

问题:访问父路径,Outlet 里空空如也

回顾上一篇的结构,父路由 /app 下有 dashboard、settings 两个子路由。当 URL 是 /app/dashboard 时一切正常,但当用户直接访问 /app——父路由匹配上了(渲染 Layout 外壳),可没有任何子路由匹配,Outlet 里没东西,页面只剩一个空壳。

访问 /app
→
Layout 匹配
→
Outlet 空白(无子路由匹配)

解法:用 index 路由指定默认内容

<Route index element={...} /> 声明一个 index 路由——它没有 path,而是用 index 属性标记自己是"父路径精确匹配时的默认子路由"。

index 路由:填补父路径精确匹配的空白
<Route path="/app" element={<Layout />}>
  {/* ★ index:URL 恰好是 /app 时,渲染这个 */}
  <Route index element={<Home />} />
  <Route path="dashboard" element={<Dashboard />} />
  <Route path="settings"  element={<Settings />} />
</Route>

现在访问 /app,Layout 外壳里的 Outlet 会渲染 Home;访问 /app/dashboard,Outlet 渲染 Dashboard。index 路由天然共享父路由的布局。

为什么叫 "index"?
这个名字借鉴了 Web 服务器的 index.html——访问一个目录(如 /docs/)时,服务器默认返回该目录下的 index.html。React Router 的 index 路由是同样的心智:访问父路由这个"目录"时,默认展示的那个"首页"。它对应的正是父路径本身,不多一层路径段。

index 路由 vs path="" 空路径

你可能见过 <Route path="" element={...} />,它在很多情况下效果和 index 路由一样,但两者语义不同,存在细微差异:

index 路由
<Route index
  element={<Home/>} />

// 语义:我是默认子路由
// 只匹配父路径精确命中
// 不能再有子路由/嵌套 path
空 path 路由
<Route path=""
  element={<Layout/>}>
  ...子路由
</Route>

// 语义:我是一段"空"路径
// 可以带自己的子路由继续嵌套
indexpath=""
能否有子路由不能(它是叶子)能(可继续嵌套)
能否有自己的 index—能
典型用途父路径的默认展示页无路径的中间布局分组
常见错误
1. 给 index 路由再写 path:<Route index path="home" /> 是非法的——index 和 path 互斥,index 本就意味着"无路径的默认项"。
2. 一个父路由下配多个 index:只能有一个,多个会冲突。
3. 误以为 index 会匹配所有未命中的子路径:不会。index 只在父路径精确命中时生效;捕获未匹配路径要用通配 path="*"(见 404 章节)。

典型用法:根路由的首页

网站根首页就是根布局的 index 路由
<Route path="/" element={<RootLayout />}>
  <Route index element={<HomePage />} />   {/* 访问 / 时的首页 */}
  <Route path="about" element={<About />} />
</Route>

相比"访问 / 时用 Navigate 重定向到 /home",index 路由更干净:URL 保持 /,首页直接显示,不产生额外的跳转。

常见陷阱

  • 给 index 路由同时写 path。<Route index path="home" /> 是非法组合——index 的语义本身就是"无路径的默认子路由",和 path 互斥,这个组合会直接报错或行为异常,而不是"两者取一个生效"。
  • 一个父路由下配了多个 index。index 路由的作用是"父路径精确命中时渲染谁",这个答案必须唯一;配置多个会导致路由冲突,通常会报错或只有其中一个生效,排查时容易误以为是别的路由配置问题。
  • 误以为 index 路由能兜底所有未匹配的子路径。index 只在父路径被精确命中(不多不少)时生效,访问一个既不匹配 index 也不匹配任何子路由 path 的路径时,index 不会顶上——需要用通配符 path="*" 单独处理未匹配情况(404)。
  • 给 index 路由挂子路由。index 路由是路由树上的叶子节点,不能再嵌套 <Route>;如果需要"这一层既是默认展示又要继续往下分支",应该用 path="" 的空路径路由代替,它可以带自己的子路由并继续嵌套。

面试高频考点

  1. index 路由解决了什么问题?—— 解决"访问父路径本身(不多任何路径段)时 Outlet 里渲染什么"的问题;没有 index 路由时,父路由匹配上只渲染布局外壳,子路由都没命中,Outlet 一片空白。
  2. index 路由和 path="" 的空路径路由有什么区别?—— index 路由语义是"我是这一层的默认子路由",是路由树的叶子节点,不能再有自己的子路由,也不能再配 index;path="" 是一段空的路径段,本质仍是普通路由,可以继续嵌套子路由、甚至有自己的 index。
  3. 为什么 index 路由的命名借鉴自 index.html?—— 传统 Web 服务器访问一个目录路径(如 /docs/)时会默认返回该目录下的 index.html;React Router 的 index 路由是同一心智模型的迁移——访问父路由这个"目录",默认展示它的"首页"内容。
  4. 用 index 路由展示首页,相比用 Navigate 重定向到 /home 有什么优势?—— index 路由不产生跳转,URL 始终保持父路径本身(比如 /),用户看到的地址栏更干净,也不会多一次历史记录条目;而 Navigate 重定向会实际改变 URL,产生一次额外的导航跳转。

与其他知识点的联系

  • Outlet 占位符：index 路由填补的正是"父路由的 Outlet 在精确匹配时该渲染什么"这个问题——没有 Outlet 机制，index 路由无处安放；反过来说，只要父组件里有 Outlet，index 路由就能自然地把内容渲染进去，两者是同一套插槽机制的一体两面。
  • 布局路由：布局路由（无 path 的父路由）本身没有"父路径"概念，因此不能配 index 路由——index 只适用于带 path 的父路由。这个限制反过来帮助区分"什么时候该用带 path 的父路由 + index"和"什么时候该用无 path 的布局路由"。
  • Navigate 重定向：文中对比过"用 index 路由展示首页"和"用 Navigate 重定向到 /home"两种方案——前者 URL 不变、无额外跳转，后者会产生一次真实的历史记录。这组对比也提醒我们：能用 index 路由解决的"默认页面"需求，优先别用 Navigate，减少不必要的重定向。
  • 404 页面：index 路由和通配路由 path="*" 覆盖的是路由匹配的两个极端——index 管"父路径被精确命中"，通配符管"什么都没匹配上"。两者都不参与"部分匹配"的中间情况，合起来才能让一个父路由下的所有访问情形都有对应的渲染结果。
  • createBrowserRouter 与 loader：数据路由的配置对象里同样支持 index: true 字段，且 index 路由可以单独挂自己的 loader——这意味着"父路径的默认页面需要独立取数"这个需求在数据路由模式下依然能被 index 路由自然承载，写法上只是从 JSX 属性换成了对象字段。
