React · useRef

forwardRef：把 ref 转发给内部 DOM

父组件想拿到自定义子组件内部某个 DOM 节点（比如给封装好的 <MyInput> 调 focus），直接传 ref 是拿不到的——函数组件默认不接收 ref。forwardRef 就是解决这个"ref 穿透一层组件"的问题。

为什么函数组件默认收不到 ref

ref 和 key 一样，是 React 的保留属性，不会作为普通 prop 传进组件。你在子组件里 props.ref 是拿不到的。这是有意设计：ref 指向"实例/DOM 节点"，而函数组件没有实例，React 不知道该把 ref 指向它内部的哪个节点——只能由组件作者显式声明要转发给谁。

反例:直接传 ref 给函数组件,拿不到
function MyInput(props) {
  return <input {...props} />;
}

function Form() {
  const ref = useRef(null);
  // ❌ ref 不会传进 MyInput，ref.current 永远是 null
  // 旧版本还会报警告:Function components cannot be given refs
  return <MyInput ref={ref} />;
}

用 forwardRef 转发

forwardRef 让 ref 落到内部 input 上
import { forwardRef } from 'react';

// 第二个参数就是父组件传来的 ref
const MyInput = forwardRef(function MyInput(props, ref) {
  // 把 ref 挂到真正想暴露的那个 DOM 节点上
  return <input ref={ref} {...props} />;
});

function Form() {
  const ref = useRef(null);
  // ✅ 现在 ref.current 就是内部那个真实的 input DOM
  return <MyInput ref={ref} onClick={() => ref.current.focus()} />;
}

useImperativeHandle:只暴露想给的方法

有时你不想把整个 DOM 节点交出去（父组件能乱调 innerHTML），只想暴露受控的几个方法。useImperativeHandle 让你自定义 ref 暴露出去的内容——形成一个"命令式接口"。

只暴露 focus 和 scrollToTop
const FancyInput = forwardRef((props, ref) => {
  const inputRef = useRef(null);

  // 父组件的 ref.current 只会有这两个方法，拿不到真实 DOM
  useImperativeHandle(ref, () => ({
    focus() { inputRef.current.focus(); },
    scrollToTop() { inputRef.current.scrollIntoView(); },
  }), []);

  return <input ref={inputRef} />;
});
命令式接口要克制使用。React 本质是声明式的，useImperativeHandle 是又一个"逃生舱"。绝大多数父子通信应该靠 props / 回调，只有 focus、scroll、play 这类"命令"才值得用它。滥用会让数据流变得难以追踪。

React 19：ref 可以直接当 prop 用

React 19 起，函数组件可以直接把 ref 作为普通 prop 接收，不再需要 forwardRef 包裹。forwardRef 会在未来版本被逐步废弃。

React 18 及以前
const MyInput = forwardRef(
  (props, ref) =>
    <input ref={ref} {...props} />
);
React 19+
// ref 就是普通 prop 了
function MyInput({ ref, ...props }) {
  return <input ref={ref}
    {...props} />;
}
为什么 React 19 能简化？历史上 forwardRef 存在，是因为早期 ref 被特殊对待、不进 props。React 19 重新设计后取消了这层特殊性，ref 变成普通 prop，API 更直观、少一层包裹、TS 类型也更简单。这是 React 团队"消除历史包袱"的一步。学习时理解 forwardRef 的原理仍有价值——大量现存代码库和第三方组件库还在用它，短期内不会消失。

常见陷阱

  • 以为给函数组件传 ref 就能像传普通 prop 一样在组件里通过 props.ref 拿到：这是最根本的误解。React 会把 ref（和 key）当作保留属性特殊处理，从 JSX 元素上摘出来，不会塞进传给组件的 props 对象。所以在 React 18 及以前，函数组件内部读 props.ref 永远是 undefined，必须用 forwardRef 包一层，通过它的第二个参数才能拿到。
  • 忘记把转发进来的 ref 真正挂到目标 DOM 节点上：写了 forwardRef((props, ref) => {...}) 却在返回的 JSX 里漏掉 ref={ref}，这样虽然组件"支持"接收 ref 了，但 ref.current 依然是 null，因为没有任何 DOM 元素真正持有这个 ref。forwardRef 只是打通了"接收"的通道，"挂载到哪个元素"仍需要手动指定。
  • 用 useImperativeHandle 时忘了依赖数组，导致每次渲染都重新生成暴露对象：useImperativeHandle(ref, createHandle, deps) 的第三个参数决定了暴露对象是否要在每次渲染重新创建。忘记传或传错依赖，会导致父组件每次拿到的都是一个新对象引用，如果父组件依赖这个引用做 memo 比较或者放进另一个 effect 的依赖数组，会引发不必要的重复执行。
  • 混淆 forwardRef 转发"整个 DOM 节点"和 useImperativeHandle 暴露"受限接口"的场景：直接 ref={ref} 转发会把整个真实 DOM 节点交给父组件，父组件可以随意调用任何原生方法甚至直接改内容/结构，破坏封装；如果只想让父组件调用几个受控的方法（如 focus、scrollToTop），应该用 useImperativeHandle 自定义暴露的内容，而不是简单转发整个节点。

面试高频考点

  1. 为什么函数组件不能像普通 prop 一样接收 ref，必须用 forwardRef？—— 因为 React 把 ref 和 key 视为框架层面的保留属性，会在创建元素时从传入的属性里摘出来单独处理，不会作为普通 prop 传进组件函数收到的 props 对象里。这是有意设计：ref 语义上指向"某个实例或 DOM 节点"，而函数组件本身没有实例，React 不能替开发者猜测该把 ref 指向组件内部的哪个节点，只能通过 forwardRef 显式声明"这个 ref 该转发给谁"。
  2. forwardRef 和 useImperativeHandle 分别解决什么问题，二者关系是什么？—— forwardRef 解决"如何让 ref 穿透一层组件，落到内部某个节点或实例上"的问题，是转发的通道。useImperativeHandle 则在 forwardRef 拿到 ref 之后，进一步自定义"父组件通过这个 ref 到底能访问到什么"，可以只暴露几个受控方法而不是整个真实 DOM 节点。二者常搭配使用：forwardRef 打通通道，useImperativeHandle 收窄暴露范围，形成一个命令式的、可控的接口。
  3. React 19 之后 forwardRef 还有必要学吗？和之前的写法有什么区别？—— 有必要，因为大量现存代码库和第三方组件库仍在使用 forwardRef，短期内不会消失，理解其原理是读懂旧代码的前提。区别在于：React 19 之前 ref 是保留属性、不进 props，必须用 forwardRef 包裹组件才能接收并转发；React 19 取消了这层特殊性，ref 可以像普通 prop 一样在函数组件的参数里直接解构出来（如 function MyInput({ ref, ...props })），不再需要额外包一层，API 更直观、TS 类型也更简单。

与其他知识点的联系

  • useRef 访问 DOM：forwardRef 解决的问题建立在"ref 可以挂到 DOM 元素上"这个基础之上——如果不理解 <input ref={inputRef}> 会让 inputRef.current 指向真实 DOM 节点，就无法理解 forwardRef 转发的第二个参数 ref 最终要挂到哪里去。forwardRef 本质是把"谁能拿到这个 ref"这件事从"只能是直接渲染该元素的那层"扩展到了"任意外部调用者"。
  • useRef 存储可变值：forwardRef 组件内部常见写法是先用 useRef 拿到内部 DOM 的引用（如 inputRef），再通过 useImperativeHandle 把基于这个 ref 封装出的方法暴露给外部——内部这层 ref 本身依然遵循"跨渲染保持同一引用、读写不触发渲染"的普通 useRef 规则，forwardRef 只是多加了一层"决定暴露什么给外部"的包装。
  • createContext 与 Provider（下一篇）：forwardRef 和 Context 都是 React 提供的"跨层级传递东西"的机制，但传递的内容和方向不同：ref 传递的是"对某个实例/DOM 的命令式引用"，通常自上而下由父组件持有、指向子组件内部；Context 传递的是"数据"，向下广播给所有订阅的后代。两者常被初学者混淆，但适用场景完全不同——需要调用子组件方法用 ref，需要跨层共享数据用 Context。
  • Hooks 只在函数组件中调用：forwardRef 包裹的函数、以及内部调用的 useImperativeHandle，仍然要遵守"只在函数组件或自定义 Hook 里调用 Hook"的规则——forwardRef 返回的组件本质上还是一个函数组件，只是多接收了一个 ref 参数，Hooks 的调用规则对它没有任何特殊豁免。
