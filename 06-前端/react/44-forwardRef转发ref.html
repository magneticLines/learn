React · useRef

forwardRef：把 ref 转发给内部 DOM

父组件想拿到自定义子组件内部某个 DOM 节点（比如给封装好的 <MyInput> 调 focus），直接传 ref 是拿不到的——函数组件默认不接收 ref。forwardRef 就是解决这个"ref 穿透一层组件"的问题。

为什么函数组件默认收不到 ref

ref 和 key 一样，是 React 的保留属性，不会作为普通 prop 传进组件。你在子组件里 props.ref 是拿不到的。这是有意设计：ref 指向"实例/DOM 节点"，而函数组件没有实例，React 不知道该把 ref 指向它内部的哪个节点——只能由组件作者显式声明要转发给谁。

反例:直接传 ref 给函数组件,拿不到
function MyInput(props) {
  return <input {...props} />;
}

function Form() {
  const ref = useRef(null);
  // ❌ ref 不会传进 MyInput，ref.current 永远是 null
  // 旧版本还会报警告:Function components cannot be given refs
  return <MyInput ref={ref} />;
}

用 forwardRef 转发

forwardRef 让 ref 落到内部 input 上
import { forwardRef } from 'react';

// 第二个参数就是父组件传来的 ref
const MyInput = forwardRef(function MyInput(props, ref) {
  // 把 ref 挂到真正想暴露的那个 DOM 节点上
  return <input ref={ref} {...props} />;
});

function Form() {
  const ref = useRef(null);
  // ✅ 现在 ref.current 就是内部那个真实的 input DOM
  return <MyInput ref={ref} onClick={() => ref.current.focus()} />;
}
Form 的 ref
→
forwardRef(MyInput)
→
内部 <input> DOM

useImperativeHandle:只暴露想给的方法

有时你不想把整个 DOM 节点交出去（父组件能乱调 innerHTML），只想暴露受控的几个方法。useImperativeHandle 让你自定义 ref 暴露出去的内容——形成一个"命令式接口"。

只暴露 focus 和 scrollToTop
const FancyInput = forwardRef((props, ref) => {
  const inputRef = useRef(null);

  // 父组件的 ref.current 只会有这两个方法，拿不到真实 DOM
  useImperativeHandle(ref, () => ({
    focus() { inputRef.current.focus(); },
    scrollToTop() { inputRef.current.scrollIntoView(); },
  }), []);

  return <input ref={inputRef} />;
});
命令式接口要克制使用。React 本质是声明式的，useImperativeHandle 是又一个"逃生舱"。绝大多数父子通信应该靠 props / 回调，只有 focus、scroll、play 这类"命令"才值得用它。滥用会让数据流变得难以追踪。

React 19：ref 可以直接当 prop 用

React 19 起，函数组件可以直接把 ref 作为普通 prop 接收，不再需要 forwardRef 包裹。forwardRef 会在未来版本被逐步废弃。

React 18 及以前
const MyInput = forwardRef(
  (props, ref) =>
    <input ref={ref} {...props} />
);
React 19+
// ref 就是普通 prop 了
function MyInput({ ref, ...props }) {
  return <input ref={ref}
    {...props} />;
}
为什么 React 19 能简化？历史上 forwardRef 存在，是因为早期 ref 被特殊对待、不进 props。React 19 重新设计后取消了这层特殊性，ref 变成普通 prop，API 更直观、少一层包裹、TS 类型也更简单。这是 React 团队"消除历史包袱"的一步。学习时理解 forwardRef 的原理仍有价值——大量现存代码库和第三方组件库还在用它，短期内不会消失。
