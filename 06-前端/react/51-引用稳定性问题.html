引用稳定性：每次渲染都是"新"的函数和对象

要理解 useMemo / useCallback 为什么存在，得先看清一个事实：组件每次渲染，函数体里写的对象字面量、数组字面量、箭头函数，都会被重新创建成一个全新的引用。这个"新引用"会悄悄破坏很多优化。

根源：渲染就是重新执行函数

函数组件每次渲染，就是把整个函数体从头执行一遍。里面每一处 {...}、[...]、() => {} 都是一条"创建新值"的语句——执行到就新建一个。所以哪怕内容一模一样，两次渲染拿到的也是两个不同的引用。

内容相同，引用不同
function Component() {
  // 每次渲染，这三行都重新执行，产生全新引用
  const obj = { a: 1 };          // 新对象
  const arr = [1, 2];            // 新数组
  const fn = () => {};          // 新函数

  // 上一次渲染的 obj !== 这一次的 obj，即使 { a: 1 } 完全相同
}
回顾第 23 篇：React 用 Object.is（即 ===）做引用比较，不看内容。{a:1} === {a:1} 永远是 false，因为它们是内存里两个不同的对象。基本类型（数字、字符串、布尔）没有这个问题——1 === 1 永远 true。问题只出在对象、数组、函数这些引用类型上。

破坏点一：React.memo 失效

React.memo 让子组件在"props 没变"时跳过重渲。但它判断"没变"也是用引用比较。父组件每次渲染都传一个新函数/新对象当 prop，memo 每次都认为"props 变了"，于是白白包了 memo 却从不生效。

新函数 prop 让 memo 形同虚设
const Child = React.memo(function Child({ onClick }) {
  console.log('Child 渲染');
  return <button onClick={onClick}>点我</button>;
});

function Parent() {
  const [n, setN] = useState(0);

  // ❌ 每次 Parent 重渲都生成新函数 → Child 的 onClick prop 每次都"变"
  //    → React.memo 判定 props 变了 → Child 每次都重渲，memo 没用
  return (
    <div>
      <button onClick={() => setN(n + 1)}>{n}</button>
      <Child onClick={() => doSomething()} />  // 新函数字面量
    </div>
  );
}

破坏点二：useEffect 依赖每次都"变"

useEffect 的依赖数组也是引用比较。如果把每次都新建的对象/函数放进依赖，React 每次都判定"依赖变了"，effect 每次渲染都重新执行——可能变成无限请求、无限订阅。

不稳定依赖导致 effect 反复触发
function Search({ keyword }) {
  // 每次渲染都是新对象
  const options = { q: keyword, limit: 10 };

  useEffect(() => {
    fetchData(options);
  }, [options]);  // ❌ options 引用每次都变 → 每次渲染都发请求
}

解法：把引用"钉住"

这两个问题的根都是同一个：需要一个跨渲染保持稳定的引用。React 给了两个专门工具来"记住上次的引用、只在依赖变化时才换新"：

要缓存的东西工具作用
一个函数useCallback(fn, deps)依赖不变则复用同一个函数引用
一个计算结果(对象/数组/值)useMemo(() => v, deps)依赖不变则复用同一个结果
每次渲染新引用
→
useMemo / useCallback
→
依赖不变 = 引用稳定
不是所有新引用都要修。只有当这个引用被别的东西比较时才要紧——比如传给 memo 子组件、进 useEffect/useMemo 依赖、当 Context value。如果只是组件内部用一下、不参与任何比较，新建引用完全无所谓，不用管。后两篇会反复强调这个"只在必要时才缓存"的原则。
面试常问："为什么需要 useCallback？" 别答"为了性能"——那太笼统。要答到根上："因为函数组件每次渲染都会重建函数字面量，产生新引用；当这个函数作为 memo 子组件的 prop 或 useEffect 依赖时，新引用会导致本可避免的重渲/重执行。useCallback 缓存函数引用，让依赖不变时复用同一个函数。"

小结

  • 函数组件每次渲染都重新创建内部的对象/数组/函数，引用每次都不同。
  • React 到处用引用比较（Object.is）：memo 的 props、effect 的依赖、Context value。
  • 不稳定引用会让 memo 失效、让 useEffect 反复触发。
  • 解法是用 useMemo/useCallback 稳定引用——但只在引用真的被比较时才需要。

与其他知识点的联系

  • useMemo 缓存计算（第 52 篇）与 useCallback 缓存函数（第 53 篇）：这两篇是本篇提出的问题的直接答案——useMemo 稳定对象/数组/计算结果的引用，useCallback 稳定函数的引用，本质都是"给 React 一个跨渲染保持同一个引用的容器"，用来对付本篇讲的"每次渲染都是新引用"的根源问题。
  • useReducer 与 Context 结合（第 50 篇）：那一篇能把 dispatch 单独拆出一个 Context、让"只派发"的组件免于重渲，靠的正是 useReducer 保证 dispatch 天然引用稳定这一点——本篇讲的原理是理解那个优化为什么有效的前提。
  • 过度优化的坏处（第 54 篇）：本篇结尾提醒"不是所有新引用都要修"，只有被比较时才要紧——这正是第 54 篇要重点展开的话题：滥用 useMemo/useCallback 去"修"那些根本没人比较的引用，缓存本身的开销反而比问题更大。
  • 跨层级 Context 通信（第 61 篇）：Context 的 value 同样受引用比较的影响——value={{ theme, toggle }} 这种内联对象每次渲染都是新引用，会让所有消费者无谓重渲染，这是本篇"引用比较无处不在"原则在 Context 场景下的具体体现。
  • 父传子 Props（第 58 篇）：第 58 篇提到的"对象类型 prop 用内联字面量传递会让 memo 失效"，其根源就是本篇讲的引用不稳定——理解了渲染即重新执行函数体，才能理解为什么 profile={{ city: 'BJ' }} 这种写法每次都产生新对象。
