数据路由:createBrowserRouter + loader 在渲染前取数

React Router v6.4 引入了"数据路由"这套全新模式。核心思想:把数据获取从组件里挪到路由配置里,让数据在组件渲染之前就加载好,消灭"先渲染空壳、再发请求、再填充"的瀑布式加载。

老问题:请求写在组件里的"渲染后取数"瀑布

传统写法把数据请求放进 useEffect:组件先渲染出加载中的空壳,挂载后才发请求,数据回来再重渲。这有两个代价:一是额外一轮渲染,二是父子组件各自取数会串成瀑布(父加载完才渲染子,子才开始加载)。

❌ 传统:fetch-on-render,先渲染再请求
function UserPage() {
  const [user, setUser] = useState(null);
  useEffect(() => {
    // 组件已经渲染出来了,现在才开始请求
    fetch('/api/user').then(r => r.json()).then(setUser);
  }, []);
  if (!user) return <Spinner />; // 空壳阶段
  return <Profile user={user} />;
}

新模式:createBrowserRouter + loader

数据路由用 createBrowserRouter 以配置对象(而非 JSX)定义路由,每个路由可挂一个 loader 函数。导航到该路由时,React Router 会先执行 loader、等数据就绪,再渲染组件——组件拿到的第一帧就有数据。

配置式路由 + loader
import { createBrowserRouter, RouterProvider } from 'react-router-dom';

const router = createBrowserRouter([
  {
    path: '/users/:id',
    element: <UserPage />,
    // loader:渲染前执行,返回值给组件用
    loader: async ({ params }) => {
      // params 来自路由,这里拿到 :id
      const res = await fetch(`/api/users/${params.id}`);
      return res.json();  // 返回的数据 → useLoaderData() 读取
    },
  },
]);

// 用 RouterProvider 挂载,取代 <BrowserRouter>+<Routes>
ReactDOM.createRoot(el).render(<RouterProvider router={router} />);

核心转变:数据是路由的一部分,不是组件的副作用
传统模式下,"这个页面需要什么数据"藏在组件的 useEffect 里,路由系统一无所知。数据路由把数据需求提升到路由配置层——路由系统知道每个 URL 需要哪些数据,于是能在导航时并行地、提前地把所有层级的数据一起加载(而不是父子瀑布)。这也是 Remix / Next.js App Router 等现代框架的共同思想:数据获取与路由绑定。

解决父子瀑布:嵌套 loader 并行执行

嵌套路由的多个 loader 会并行触发,而不是等父 loader 完成才开始子 loader。这从根本上消除了传统写法里"父组件加载→渲染→子组件挂载→再加载"的串行瀑布。

defer:大数据别拖住整页

如果 loader 里有一个慢查询,会拖住整个页面渲染。defer 让你把慢的部分"延迟"——先渲染页面框架和快数据,慢数据后到,配合 <Await> 局部展示加载态。

defer:快数据先渲染,慢数据流式补上
import { defer, Await } from 'react-router-dom';

// loader:快的 await 等好,慢的不 await,直接交给 defer
async function loader() {
  return defer({
    user: await fetchUser(),       // 快:阻塞等待
    comments: fetchComments(),    // 慢:传 Promise,不 await
  });
}

// 组件:用 Await + Suspense 局部等慢数据
<Suspense fallback={<Spinner />}>
  <Await resolve={data.comments}>
    {comments => <Comments list={comments} />}
  </Await>
</Suspense>
两套 API 别混用
数据路由(loader/action)只在用 createBrowserRouter + RouterProvider 时生效。如果你还在用 <BrowserRouter> + <Routes>(声明式路由树),loader 不会被调用,useLoaderData 会报错。二者是并存的两套模式,要用数据 API 就必须整体切到 createBrowserRouter。

另外注意:较新版本里 defer 已可省略(loader 直接返回含 Promise 的对象也支持),但 defer 写法兼容性更好、语义更清晰,面试和存量项目里仍常见。
和 React Query 的关系:loader 解决"何时取数",但不做缓存、重试、后台刷新。生产项目常把 loader 与 React Query 结合——loader 里 queryClient.ensureQueryData() 预取,组件里 useQuery 复用缓存,两全其美。

与其他知识点的联系

  • Outlet 占位符 / 布局路由 / index 路由：数据路由只是把路由树的写法从 JSX 挪到了配置对象（children 数组），嵌套结构、Outlet 渲染子内容、index 路由填补精确匹配空白这些概念全部原样保留——理解声明式路由树，是看懂数据路由配置对象的前提，不是另一套要重新学的知识。
  • 认证守卫 PrivateRoute：数据路由下鉴权判断可以提前到 loader 里做——throw redirect('/login')，比组件里渲染 <Navigate> 更早介入，能避免"组件先渲染一瞬又被踢走"的闪烁，这是 loader"渲染前执行"这一特性最直接的应用。
  • 404 页面：loader 抛出的异常由 errorElement 统一兜底，这和通配路由 path="*" 处理的是两类不同的"找不到"——一个是"数据请求失败/资源不存在"，一个是"URL 本身没有匹配的路由"，两者常常需要同时配置才能覆盖真实项目里的异常情况。
  • Query 参数与 useSearchParams：loader 函数拿到的 request 是原生 Request 对象，用 new URL(request.url).searchParams 就能读到 query 参数——这让"根据分页/筛选条件请求数据"的逻辑可以直接写在 loader 里，比在组件里读 useSearchParams 再触发 useEffect 取数更早、更省一轮渲染。
  • action 处理表单提交（下一篇）：loader 和 action 是数据路由的一体两面——loader 负责路由激活时的"读"，action 负责表单提交时的"写"，两者共享同一套路由配置对象，也共享"在组件渲染之外单独处理数据"的设计思想。
