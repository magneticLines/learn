useMemo：缓存昂贵计算的结果

组件每次渲染都会把函数体重跑一遍。如果里面有一段很耗时的计算，而它的输入根本没变，每次都重算就是浪费。useMemo 让这段计算的结果被"记住"，输入不变时直接复用。

用法：useMemo(() => 计算, 依赖)

useMemo 接收两个参数：一个"计算函数"和一个"依赖数组"。它执行计算函数、缓存返回值；下次渲染时，只要依赖没变，就跳过计算、直接返回上次缓存的结果。

缓存一个昂贵计算
function ProductList({ products, keyword }) {
  // 假设过滤+排序上万条数据很慢
  const visible = useMemo(() => {
    return products
      .filter(p => p.name.includes(keyword))
      .sort((a, b) => b.score - a.score);
  }, [products, keyword]);
  // 只有 products 或 keyword 变了才重算；
  // 别的 state 变化引起的重渲会直接复用上次结果

  return visible.map(p => <Item key={p.id} {...p} />);
}

第二个用途：稳定引用（承接上一篇）

useMemo 不只缓存"昂贵计算"，也常用来稳定一个对象/数组的引用——即使这个对象计算不贵，只要它要被当 memo 子组件的 prop、effect 依赖、或 Context value，就需要它跨渲染保持同一个引用。

用 useMemo 稳定要传下去的对象
// 这个对象计算不贵，但要进 effect 依赖 → 需要稳定引用
const config = useMemo(
  () => ({ q: keyword, limit: 10 }),
  [keyword]  // keyword 不变时，config 是同一个引用
);

useEffect(() => { fetchData(config); }, [config]);  // 不再每次触发

什么才算"昂贵"？

这是最容易误判的地方。绝大多数计算其实很便宜，用 useMemo 反而得不偿失。真正值得 memo 的，通常是：

  • 对大数组（成千上万条）做 filter / sort / map / reduce。
  • 复杂的数据结构转换、递归、构建 index。
  • 可测量到的、在慢设备上会造成卡顿的计算。
怎么判断"昂贵"？测。别拍脑袋。用 console.time('calc') … console.timeEnd('calc') 包住计算，或用 React DevTools Profiler 看渲染耗时。经验值：如果一次计算 < 1ms，加 useMemo 几乎没有收益——比较依赖、维护缓存本身也有开销，可能比重算还慢。

缓存本身有成本，不要滥用

useMemo 不是免费的。它要：保存上次的依赖数组和结果（占内存）、每次渲染比较依赖、多一层函数调用。对便宜的计算，这些开销 > 省下的计算——净亏损，还让代码更难读。

❌ 过度使用
// 计算就是个加法，
// memo 的开销比算它还大
const sum = useMemo(
  () => a + b,
  [a, b]
)
// 直接写 const sum = a + b
✅ 值得使用
// 对 5 万条做过滤排序
const rows = useMemo(
  () => bigList
    .filter(...).sort(...),
  [bigList, filter]
)
// 明显耗时，缓存划算
核心原则：先测量，再优化，不要预防性 memo。不要"以防万一"给每个计算都套 useMemo。正确顺序是：先写朴素代码 → 发现卡顿 → 用 Profiler 定位到具体是哪个计算慢 → 只给它加 useMemo。为想象中的性能问题优化，是典型的过早优化，代价是可读性和维护成本（第 54 篇详谈）。
依赖数组要诚实：凡是计算里用到的、会变的值（props、state），都要写进依赖，否则会读到"过期的旧值"，产生难查的 bug。用 eslint-plugin-react-hooks 的 exhaustive-deps 规则帮你检查——别手动删依赖来"骗"它闭嘴。

小结

  • useMemo(() => v, deps) 缓存计算结果，依赖不变则复用，跳过重算。
  • 两个用途：缓存昂贵计算 + 稳定对象/数组引用（供 memo/effect/Context 用）。
  • "昂贵"要靠测量判断（<1ms 基本不值得）；缓存本身有内存和比较开销。
  • 不要预防性滥用——先测量再优化，依赖数组必须完整诚实。

常见陷阱

  • 给便宜的计算无脑套 useMemo：像 a + b 这种加法也包一层 useMemo，缓存本身要保存依赖数组、做比较、多一层函数调用，这些开销比直接重算还高，属于负优化，还让代码可读性变差。判断标准是"测量后确实慢"，不是"看起来是个计算就该缓存"。
  • 依赖数组漏写，读到过期值：计算函数里用到的 props/state 没有全部放进依赖数组，导致依赖变化后 useMemo 仍然返回旧的缓存结果，产生"数据明明变了、界面却没更新"的诡异 bug。应该始终依赖 eslint-plugin-react-hooks 的 exhaustive-deps 规则，而不是手动"精简"依赖来图省事。
  • 误以为 useMemo 能保证结果绝对不重算：useMemo 只是给 React 的一个"性能提示"，不是语义上的强保证——在某些情况下（如 React 未来的特性、内存压力）React 可能会丢弃缓存重新计算。不能把它当成必然只执行一次的记忆化，函数体不该包含"只能跑一次"才对的副作用逻辑。
  • 把 useMemo 当成解决所有重渲染问题的银弹：只稳定了计算结果，却没意识到问题根源其实是"这个子组件本不该重渲染"——真正该做的可能是提升/下沉 state、拆分组件，而不是到处撒 useMemo 去掩盖架构上的重渲染问题。
  • 用 useMemo 做数据获取或订阅副作用：把 fetch、事件监听塞进 useMemo 的计算函数里。useMemo 的语义是"纯计算+缓存"，不是副作用管理，副作用应该放进 useEffect；混用会导致渲染阶段产生副作用，破坏 React 严格模式下"渲染必须是纯粹的"这一假设。

面试高频考点

  1. useMemo 解决的核心问题是什么？—— 组件每次渲染都会把函数体整体重新执行一遍；如果里面有输入没变但计算耗时的逻辑，重复计算就是纯浪费。useMemo 让"计算函数 + 依赖数组"配对，依赖不变时跳过计算直接复用上次结果，用空间(缓存)换时间(重算)。
  2. 什么时候用 useMemo 才划算？如何判断？—— 不能拍脑袋，要测量：用 console.time/timeEnd 或 React DevTools Profiler 量出该计算的实际耗时，经验值是低于 1ms 基本不值得缓存，因为比较依赖数组、维护缓存本身也有开销。真正值得的场景通常是大数组的 filter/sort/reduce、复杂数据结构转换等可测量到的耗时操作。
  3. useMemo 缓存对象引用和缓存计算结果，是同一件事吗？—— 本质上是同一个机制的两种应用：缓存计算结果针对的是"避免重复的昂贵运算"；缓存对象/数组引用针对的是"让引用保持稳定"，供 React.memo 的 props 比较、useEffect 的依赖比较、Context 的 value 比较使用，即使这个对象的计算本身并不贵。两种用途都是靠"依赖不变则返回同一个缓存值"实现的。
  4. useMemo 的依赖数组漏写会造成什么后果，和 useEffect 漏写依赖的坑一致吗？—— 一致，两者都是闭包问题：计算/副作用函数在创建时捕获了当时的变量值，如果依赖数组没有把用到的、会变化的值写全，函数会一直用创建时的"旧值"，导致数据更新了但界面/逻辑仍反映旧状态。用 exhaustive-deps ESLint 规则可以在编译期而不是运行时暴露这个问题。

与其他知识点的联系

  • 引用稳定性问题（第 51 篇）：本篇的"第二个用途"直接承接第 51 篇的痛点——即使计算不贵，只要结果要进 memo 的 props 比较或 useEffect 依赖，就需要稳定引用。useMemo 是那篇提出问题后给出的第一个工具。
  • useCallback 缓存函数（第 53 篇）：下一篇会揭示 useCallback(fn, deps) 本质就是 useMemo(() => fn, deps) 的语法糖——两者共享同一套依赖比较、缓存槽位机制，区别只是缓存的东西是"函数"还是"函数的返回值"。
  • 过度优化的坏处（第 54 篇）：本篇反复强调"先测量、别预防性 memo"，这正是第 54 篇要系统展开的话题——给便宜计算套 useMemo 属于典型的过早优化，缓存本身的依赖比较和内存开销可能比重算还贵。
  • useReducer 与 Context 结合（第 50 篇）：那一篇提到如果 Context 的 value 需要在 state/dispatch 之外再包一层派生对象，就该用 useMemo 稳定这个对象的引用——否则会像"多包一层新对象"的陷阱那样抵消拆分 Context 的效果，是本篇"稳定引用"用途的一处典型场景。
  • 抽离复用逻辑（第 56 篇）：useMemo 缓存计算结果、useFetch 之类的自定义 Hook 缓存的是整套"状态+副作用"逻辑，两者都是"把重复或耗时的东西打包，让调用处更简洁"，但对象不同：一个缓存值，一个复用逻辑流程。
