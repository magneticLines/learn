Hooks 第一规则：只在顶层调用

不要在条件语句、循环、嵌套函数里调用 Hook。这不是"最佳实践"式的建议，而是硬性规则——违反它，React 的状态就会错位，产生诡异到难以调试的 bug。理解它必须从"React 靠什么记住每个 Hook 的状态"讲起。

规则本身：只在函数最顶层调用

所谓"顶层"，指函数组件（或自定义 Hook）的函数体最外层，不被任何控制流包裹。下面几种情况都是违规：

四种违规写法
function Bad({ show, items }) {
  // ❌ 1. 条件里调用
  if (show) {
    const [x, setX] = useState(0);
  }

  // ❌ 2. 循环里调用
  for (const item of items) {
    const ref = useRef();
  }

  // ❌ 3. 提前 return 之后调用（本质也是条件）
  if (!show) return null;
  const [y, setY] = useState(0); // show 为真才会执行到这里

  // ❌ 4. 嵌套函数/回调里调用
  setTimeout(() => { useEffect(...); }, 100);
}

底层原因：React 靠"调用顺序"匹配状态，不靠名字

这是全篇最关键的一点。很多人以为 React 是用变量名（比如 x、y）来记住每个 useState 的值，其实不是——React 根本看不到你的变量名。它维护的是一个按调用顺序排列的链表（数组）。

每次组件渲染，Hook 按代码从上到下依次执行；React 用一个内部指针，第 1 个调用的 Hook 拿链表第 1 个槽位，第 2 个拿第 2 个槽位……顺序即身份。

React 内部机制的简化模型（伪代码）
// React 内部维护的状态链表（挂在 fiber 节点上）
hooks = [ stateA, stateB, stateC ];
cursor = 0;  // 每次渲染前归零

function useState(initial) {
  const slot = hooks[cursor];  // 按当前指针位置取槽位
  cursor++;                     // 指针后移，给下一个 Hook 用
  return slot;
}

类比：React 是个只认座位号、不认脸的点名系统。第一次上课，学生依次坐 1、2、3 号座位。老师记住"1 号座位是张三的状态"。只要每次上课顺序不变，点名永远正确。

条件调用为什么会错位

假设组件里有两个 Hook，第二个被 if 包住。首次渲染 show=true，两个都执行；某次 show 变成 false，第一个 Hook 被跳过，后面所有 Hook 的槽位集体前移一位，状态全部错配。

顺序错乱的具体过程
function Form({ show }) {
  if (show) {
    const [name, setName] = useState(''); // Hook #0（show 为真时）
  }
  const [age, setAge] = useState(18);     // 希望它永远是 Hook #1
}

// 首次渲染 show=true：
//   cursor 0 → name 状态；cursor 1 → age 状态。正确。
// 重渲染 show=false：
//   name 的 useState 被跳过
//   age 变成第一个执行的 Hook，拿走了 cursor 0（原本是 name 的槽位！）
//   age 读到了 name 的 '' —— 状态张冠李戴
React 会直接报错。实际开发中你会看到红字：React has detected a change in the order of Hooks called by Form，并附上一张"上次渲染 vs 本次渲染"的 Hook 顺序对照表。看到这个错误，第一反应就是检查：是不是把某个 Hook 写进了 if / for / return 之后。

正确做法：把条件挪进 Hook 内部

规则不是"不能有条件逻辑"，而是"Hook 的调用不能被条件包住"。做法是：Hook 无条件调用，把条件放到它的参数或返回值处理里。

❌ 条件包住 Hook
if (show) {
  const [x] = useState(0);
}
✅ Hook 在顶层，条件在内部
// 总是调用，用值来控制行为
const [x, setX] = useState(0);
// 需要时才用 x

useEffect(() => {
  if (!show) return; // 条件放 effect 里
  doSomething();
}, [show]);

如果确实需要"列表里每项一个状态"，正确解法不是循环里调 Hook，而是把每一项抽成子组件，让每个子组件各自持有自己的 Hook——子组件的 Hook 顺序在它自己内部是稳定的。

记忆口诀：Hook 调用要像"仪式"——每次渲染都以完全相同的顺序、相同的次数念一遍。React 靠这个"仪式的稳定性"给每个 Hook 分配持久的记忆位置。破坏顺序，就破坏了记忆。
为什么 React 团队接受"顺序绑定"这个约束？另一条路是给每个 Hook 一个显式 key（像 useState('name')），但那会让 API 变啰嗦、还要处理重名。用"调用顺序"作为隐式身份，代价是这条规则，换来的是极简的 API——useState(0) 就够了。同时配套 eslint-plugin-react-hooks 在编译期自动揪出违规，把约束的心智负担转嫁给工具，这是很划算的工程权衡。

常见陷阱

  • 把 Hook 藏在"看起来是顶层"的三元表达式或 && 短路里：show && useEffect(...) 本质上和 if 一样是条件调用，容易被忽略，因为它看起来只是"一行表达式"。原理上，React 并不理解你的语法结构，它只是按调用发生的顺序把值塞进链表——只要某次渲染这行代码没执行，游标就会错位。
  • 提前 return 之后还有 Hook：很多人写"参数不合法就提前返回"的防御式代码，却把校验放在了后面几个 Hook 之前。只要 return 依赖的条件在某次渲染变化，return 之后的 Hook 就会时而执行、时而不执行，等价于条件调用。正确做法是把所有 Hook 都放在任何 return 之前。
  • 误以为"顺序错乱"只在开发环境报错，生产环境会静默出错更危险：React 在开发和生产都会做数量校验（Hook 数量对不上直接抛错），但如果两次渲染 Hook 数量相同、只是顺序/身份错位（比如两个 useState 互换），React 检测不出来，只会安静地把状态串位，这是最难排查的情况，因为没有任何报错提示。
  • 在循环里想"给列表每一项一个状态"：直觉上会写 items.map(item => useState(...))，但列表长度一变，后面所有 Hook 的槽位全部错位。正确做法是把每一项抽成子组件，让状态归属到每个子组件自己的 Fiber 上，而不是挤在父组件的一条链表里。

面试高频考点

  1. 为什么 Hook 不能写在 if 语句里？—— 因为 React 用"调用顺序"而不是变量名来匹配每个 Hook 对应的状态槽位。Hook 状态实际存放在一条挂在组件 Fiber 上的链表里，每次渲染都从头按顺序遍历。条件语句会导致某次渲染跳过某个 Hook，链表指针后移的次数变化，后续所有 Hook 都会读到"别人的"状态。
  2. 如果两次渲染 Hook 的调用顺序不同，React 是怎么发现并报错的？—— React 在渲染时会记录本次调用的 Hook 类型序列，和上一次渲染时记录的序列做对比；如果类型或数量不匹配，就抛出 "React has detected a change in the order of Hooks" 错误。但如果两次调用的 Hook 类型正好相同、只是逻辑归属不同（比如两个都是 useState），React 无法识别语义错误，只会静默错位。
  3. 为什么不能用"给每个 Hook 一个唯一 key"的方式代替顺序匹配？—— 技术上可行，但会让 API 变得啰嗦（每次调用都要传 key，还要处理重名冲突），偏离了 Hook "像普通函数调用一样简单"的设计目标。React 团队选择用顺序作隐式身份，把复杂度换成了一条硬性规则，并配合 eslint-plugin-react-hooks 在编译期兜底。

与其他知识点的联系

  • useState 基础用法：本节讲的"链表按调用顺序分配槽位"正是 useState 能在多次渲染间"记住"上次的值的底层机制——之前学 useState 时只需要知道"它能记住状态"，这一节把"怎么记住"的秘密揭开了，也解释了为什么 useState 不需要你手动传一个 key 来标识状态。
  • 自定义 Hook：自定义 Hook 内部调用的 useState/useEffect 等，同样要遵守顶层调用规则，且它们的槽位是嵌在调用它们的组件的同一条链表里、按调用顺序排列的——这意味着自定义 Hook 本身也不能在条件里被调用，否则会打乱它内部及外部所有 Hook 的顺序。
  • 为什么不能用 index 作 key（上一节）：本节和上一节都在讲"React 靠某种隐式顺序/身份来维持状态的正确对应"，只是维度不同——key 错位对应的是"列表里哪个元素是哪个"，Hook 顺序错位对应的是"这个组件内部哪个状态是哪个"，本质都是"身份匹配依赖稳定的顺序/标识，一旦被打乱就会状态串位"。
  • useEffect 依赖数组：useEffect 同样依赖调用顺序被记录在这条链表里，它的依赖数组对比（决定是否重新执行 effect）也是基于"这次渲染的第 N 个 useEffect"和"上次渲染的第 N 个 useEffect"做对比——如果 Hook 顺序错位，effect 的依赖对比也会读到错的历史依赖值，引发副作用执行时机的诡异 bug。
  • 列表项状态管理（把每一项抽成子组件）：本节提到"想给列表每项一个状态，应该抽成子组件而不是在循环里调 Hook"，这和 key 一节讲的"列表项内部带 state 时 key 设计要格外谨慎"是同一类场景的两个侧面——子组件拆分解决了"Hook 顺序不能变"的约束，而稳定的 key 则保证这些子组件在列表变动时不会被错误复用或销毁重建。
