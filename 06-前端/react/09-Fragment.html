React · JSX

Fragment：透明的包裹容器

JSX 必须有一个根节点——但用 <div> 包裹往往会产生多余的 DOM 节点，破坏布局。Fragment 就是为了解决这个矛盾而生的：满足"一个根"的规则，同时不在真实 DOM 里留下任何痕迹。

为什么需要 Fragment

JSX 规定每个组件只能返回一个根元素（因为编译后是函数调用，函数只能有一个返回值）。最直观的解法是用 <div> 包裹——但这会带来一个隐患：多余的 DOM 节点破坏 CSS 布局。

最典型的场景是 Flex/Grid 布局。父容器是 display: flex，直接子元素才是 flex 子项。如果组件在外面多套了一个 <div>，这个 div 就成了 flex 子项，组件内部的元素不再是直接子项——布局就乱了。

❌ div 包裹破坏 flex 布局
/* 父容器 flex，期望 3 列 */
<div style={{ display: 'flex' }}>
  <ColA />
  <ColB />
  <ColC />
</div>

/* ColA 内部用 div 包裹 */
function ColA() {
  return (
    <div>          {/* ← 这个多余的 div */}
      <h2>A标题</h2>
      <p>A内容</p>
    </div>
  );
  /* flex 子项是 div，不是 h2/p */
}
✅ Fragment 不产生多余节点
/* 父容器 flex，期望 3 列 */
<div style={{ display: 'flex' }}>
  <ColA />
  <ColB />
  <ColC />
</div>

/* ColA 用 Fragment 包裹 */
function ColA() {
  return (
    <>           {/* Fragment，不产生 DOM */}
      <h2>A标题</h2>
      <p>A内容</p>
    </>
  );
  /* h2 和 p 直接是 flex 的子项 */
}

两种写法

Fragment 有两种写法，各有适用场景：

Fragment 的两种写法
// 写法一：简写语法 <></>（最常用，99% 的情况用这个）
function MyComponent() {
  return (
    <>
      <h1>标题</h1>
      <p>内容</p>
    </>
  );
}

// 写法二：完整写法 <React.Fragment></React.Fragment>
// 唯一优势：可以传 key 属性（列表渲染时需要）
import React from 'react';

function MyComponent() {
  return (
    <React.Fragment>
      <h1>标题</h1>
      <p>内容</p>
    </React.Fragment>
  );
}

DOM 结构对比：div 包裹 vs Fragment

用 div 包裹 vs Fragment，真实 DOM 输出对比
// 用 div 包裹时，真实 DOM 是：
<div id="root">
  <div>          <!-- 多余的 div，来自组件内部 -->
    <h1>标题</h1>
    <p>内容</p>
  </div>
</div>

// 用 Fragment 时，真实 DOM 是：
<div id="root">
  <h1>标题</h1>   <!-- 直接在 root 下，没有多余包裹 -->
  <p>内容</p>
</div>

// Fragment 在真实 DOM 里完全不存在，它只活在虚拟 DOM 层。

需要 key 的场景：列表渲染中的 Fragment

什么时候必须用完整写法？

在 .map() 里用 Fragment 时，React 要求每个列表项有 key 属性（帮助 React 识别哪个项变了）。简写 <> 不支持传属性，必须用完整写法 <React.Fragment key={...}> 才能加 key。
❌ 简写无法加 key
items.map(item => (
  <>             {/* 简写不支持属性 */}
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </>
))
// 报错：key 属性无法加到 <>
✅ 完整写法可以加 key
items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </React.Fragment>
))
// key 成功传入，React 能追踪列表变化

实际场景：表格行里的 Fragment

Fragment 有一个"救命"场景——表格。HTML 的 <table> 结构有严格语义要求：<tbody> 的直接子元素必须是 <tr>，<tr> 的直接子元素必须是 <td>/<th>。如果中间插入 <div>，浏览器会忽略非法结构，表格彻底乱掉。Fragment 是唯一正确的解法：

表格列组件——Fragment 是唯一正确选择
// 需求：把"姓名 + 年龄"两列封装成一个组件

// ❌ 用 div 包裹：浏览器把 div 当无效标签忽略，td 位置错乱
function PersonCols({ name, age }) {
  return (
    <div>              // ← div 在 tr 里是非法的，表格结构崩溃
      <td>{name}</td>
      <td>{age}</td>
    </div>
  );
}

// ✅ 用 Fragment：td 直接作为 tr 的子元素，表格结构完整
function PersonCols({ name, age }) {
  return (
    <>                 // Fragment 不产生 DOM 节点
      <td>{name}</td>  // td 直接是 tr 的子元素 ✅
      <td>{age}</td>
    </>
  );
}

// 使用方式
function Table({ people }) {
  return (
    <table>
      <tbody>
        {people.map(p => (
          <tr key={p.id}>
            <PersonCols name={p.name} age={p.age} />
          </tr>
        ))}
      </tbody>
    </table>
  );
}
记忆口诀：<></> 是"隐形容器"——满足 JSX 规则，但不污染 DOM。需要 key 时换成 <React.Fragment key={...}>。碰到 CSS 布局因为多余 div 乱掉、或 HTML 语义结构（table/dl/select）不允许 div 的地方，Fragment 是第一反应。
