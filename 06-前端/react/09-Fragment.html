React · JSX

Fragment：透明的包裹容器

JSX 必须有一个根节点——但用 <div> 包裹往往会产生多余的 DOM 节点，破坏布局。Fragment 就是为了解决这个矛盾而生的：满足"一个根"的规则，同时不在真实 DOM 里留下任何痕迹。

为什么需要 Fragment

JSX 规定每个组件只能返回一个根元素（因为编译后是函数调用，函数只能有一个返回值）。最直观的解法是用 <div> 包裹——但这会带来一个隐患：多余的 DOM 节点破坏 CSS 布局。

最典型的场景是 Flex/Grid 布局。父容器是 display: flex，直接子元素才是 flex 子项。如果组件在外面多套了一个 <div>，这个 div 就成了 flex 子项，组件内部的元素不再是直接子项——布局就乱了。

❌ div 包裹破坏 flex 布局
/* 父容器 flex，期望 3 列 */
<div style={{ display: 'flex' }}>
  <ColA />
  <ColB />
  <ColC />
</div>

/* ColA 内部用 div 包裹 */
function ColA() {
  return (
    <div>          {/* ← 这个多余的 div */}
      <h2>A标题</h2>
      <p>A内容</p>
    </div>
  );
  /* flex 子项是 div，不是 h2/p */
}
✅ Fragment 不产生多余节点
/* 父容器 flex，期望 3 列 */
<div style={{ display: 'flex' }}>
  <ColA />
  <ColB />
  <ColC />
</div>

/* ColA 用 Fragment 包裹 */
function ColA() {
  return (
    <>           {/* Fragment，不产生 DOM */}
      <h2>A标题</h2>
      <p>A内容</p>
    </>
  );
  /* h2 和 p 直接是 flex 的子项 */
}

两种写法

Fragment 有两种写法，各有适用场景：

Fragment 的两种写法
// 写法一：简写语法 <></>（最常用，99% 的情况用这个）
function MyComponent() {
  return (
    <>
      <h1>标题</h1>
      <p>内容</p>
    </>
  );
}

// 写法二：完整写法 <React.Fragment></React.Fragment>
// 唯一优势：可以传 key 属性（列表渲染时需要）
import React from 'react';

function MyComponent() {
  return (
    <React.Fragment>
      <h1>标题</h1>
      <p>内容</p>
    </React.Fragment>
  );
}

DOM 结构对比：div 包裹 vs Fragment

用 div 包裹 vs Fragment，真实 DOM 输出对比
// 用 div 包裹时，真实 DOM 是：
<div id="root">
  <div>          <!-- 多余的 div，来自组件内部 -->
    <h1>标题</h1>
    <p>内容</p>
  </div>
</div>

// 用 Fragment 时，真实 DOM 是：
<div id="root">
  <h1>标题</h1>   <!-- 直接在 root 下，没有多余包裹 -->
  <p>内容</p>
</div>

// Fragment 在真实 DOM 里完全不存在，它只活在虚拟 DOM 层。

需要 key 的场景：列表渲染中的 Fragment

什么时候必须用完整写法？

在 .map() 里用 Fragment 时，React 要求每个列表项有 key 属性（帮助 React 识别哪个项变了）。简写 <> 不支持传属性，必须用完整写法 <React.Fragment key={...}> 才能加 key。
❌ 简写无法加 key
items.map(item => (
  <>             {/* 简写不支持属性 */}
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </>
))
// 报错：key 属性无法加到 <>
✅ 完整写法可以加 key
items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </React.Fragment>
))
// key 成功传入，React 能追踪列表变化

实际场景：表格行里的 Fragment

Fragment 有一个"救命"场景——表格。HTML 的 <table> 结构有严格语义要求：<tbody> 的直接子元素必须是 <tr>，<tr> 的直接子元素必须是 <td>/<th>。如果中间插入 <div>，浏览器会忽略非法结构，表格彻底乱掉。Fragment 是唯一正确的解法：

表格列组件——Fragment 是唯一正确选择
// 需求：把"姓名 + 年龄"两列封装成一个组件

// ❌ 用 div 包裹：浏览器把 div 当无效标签忽略，td 位置错乱
function PersonCols({ name, age }) {
  return (
    <div>              // ← div 在 tr 里是非法的，表格结构崩溃
      <td>{name}</td>
      <td>{age}</td>
    </div>
  );
}

// ✅ 用 Fragment：td 直接作为 tr 的子元素，表格结构完整
function PersonCols({ name, age }) {
  return (
    <>                 // Fragment 不产生 DOM 节点
      <td>{name}</td>  // td 直接是 tr 的子元素 ✅
      <td>{age}</td>
    </>
  );
}

// 使用方式
function Table({ people }) {
  return (
    <table>
      <tbody>
        {people.map(p => (
          <tr key={p.id}>
            <PersonCols name={p.name} age={p.age} />
          </tr>
        ))}
      </tbody>
    </table>
  );
}
记忆口诀：<></> 是"隐形容器"——满足 JSX 规则，但不污染 DOM。需要 key 时换成 <React.Fragment key={...}>。碰到 CSS 布局因为多余 div 乱掉、或 HTML 语义结构（table/dl/select）不允许 div 的地方，Fragment 是第一反应。

常见陷阱

  • 在 .map() 里用简写 Fragment 却想加 key：写 <> 时尝试塞 key={item.id}，React 会直接报错或警告，因为简写语法根本不支持任何属性。列表渲染中需要 key 的场景必须换成完整的 <React.Fragment key={...}>，这是唯一的例外场景。
  • 以为 Fragment 能像 div 一样加 className 或 style：Fragment 除了 key 之外不接受任何属性，因为它在设计上就是"完全透明、不产生任何 DOM 痕迹"的占位符。如果需要一个可以设置样式的包裹容器，本质上你需要的是一个真实的 div，不是 Fragment。
  • 在有严格子元素语义的 HTML 结构里用 div 而不是 Fragment：比如在 <tr> 里用组件包一层 div 返回 <td>，浏览器会认为这个 div 是非法内容并尝试"修复"DOM 结构，导致表格渲染完全错乱，而且这种错误在 React 层面不会报错，只有打开浏览器 DevTools 检查真实 DOM 才能发现。
  • 忽视 Fragment 对 CSS 选择器的隐藏影响：如果依赖 :first-child、:nth-child 等基于 DOM 结构的选择器，中间插入的 div 包裹会改变父子关系层级，从而让这些选择器命中错误的元素。换成 Fragment 后，子元素直接暴露给外层，选择器行为才符合预期。

面试高频考点

  1. 为什么 React 需要 Fragment，而不是永远用 div 包一层？—— JSX 语法要求组件必须返回单一根节点，但用 div 包裹会在真实 DOM 里多产生一层节点，这层多余的 div 可能破坏依赖直接父子关系的 CSS（如 flex/grid 布局）或 HTML 语义结构（如 table/dl/select）。Fragment 满足"单一根节点"的语法要求，同时在渲染到真实 DOM 时不留下任何痕迹。
  2. Fragment 的两种写法有什么区别，什么时候必须用完整写法？—— 简写 <></> 更简洁，覆盖绝大多数场景；完整写法 <React.Fragment> 唯一的额外能力是可以传 key 属性。在 .map() 渲染列表且外层不需要额外容器时（比如返回多个 <td>），必须用完整写法配合 key，让 React 能正确追踪每一项。
  3. Fragment 在虚拟 DOM 和真实 DOM 里分别是什么？—— Fragment 在虚拟 DOM 层是一个真实存在的节点类型（type 是 Symbol(react.fragment)），React 需要它来组织子节点、支持 key；但在渲染到真实 DOM 时，React 会跳过 Fragment 本身，只把它的子节点挂载到父容器上，因此真实 DOM 里完全看不到 Fragment 的任何标签或注释。

与其他知识点的联系

  • JSX 单一根节点限制（05）：Fragment 存在的唯一理由就是为了满足 05 篇讲的"组件必须返回单一根节点"这条规则，同时避免用 <div> 包裹带来的副作用。可以说 Fragment 是"规则"和"实际需求"之间的一个巧妙折中方案——语法上妥协，DOM 输出上不妥协。
  • key 属性与列表渲染（06）：本文提到列表渲染中 Fragment 需要用完整写法才能加 key，这和 06 篇讲的 .map() 列表渲染是同一套机制——React 依赖 key 在 Diff 阶段追踪每一项，无论子元素是普通标签还是 Fragment 包裹的多个元素，key 的作用完全一致，只是简写语法 <> 出于简洁性砍掉了传属性的能力，因此需要退回完整写法。
  • 虚拟 DOM 与 Diff 算法（08）：Fragment 在虚拟 DOM 树里确实占据一个节点位置（帮助 React 组织结构、定位 key），但在提交到真实 DOM 时会被"拆包"，只留下它的子节点。这个"虚拟层存在、真实层消失"的特性，正好体现了虚拟 DOM 作为一层抽象、不必和真实 DOM 一一对应的设计自由度。
