ESLint + eslint-plugin-react

useEffect 依赖数组漏写了一个变量,代码照样能跑、没有任何报错,直到某天线上出现"数据没更新"的诡异 bug,排查半天才发现是闭包里用了旧值。这类问题运行时不报错、review 时也未必能一眼看出,而静态检查工具能在你刚写完这行代码的时候就标红提醒——这正是 ESLint 存在的意义。

静态检查解决的问题:把 bug 拦在写代码阶段

软件开发里发现问题的时机越晚,修复成本越高:写代码时发现 → 几乎零成本;code review 时发现 → 多一轮沟通成本;测试阶段发现 → 要重新定位、修复、回归测试;线上发现 → 可能已经影响用户。ESLint 的价值在于把一部分本来要等到运行时甚至线上才暴露的问题,提前到编辑器里就标红。

它是一个静态分析工具——不运行你的代码,而是把代码解析成语法树(AST),按预设的规则检查每一处写法是否有问题。比如"变量声明了但没用过"、"switch 语句忘了 break"、"用了没定义的变量",这些都不需要真的跑起来就能发现。

和 TypeScript 类型检查的分工不同: TypeScript 主要管"类型对不对"(传参类型错了、访问了不存在的属性);ESLint 管的是代码风格和潜在逻辑问题(未使用变量、可能的空值访问模式、React Hooks 用法是否规范等),两者互补,大项目通常两个都会开启。

eslint-plugin-react / eslint-plugin-react-hooks

ESLint 核心只理解通用 JS 语法,不懂 React 的"游戏规则"(比如 Hooks 必须在顶层调用、组件必须大写开头)。这些 React 特有的检查靠插件扩展进来:

  • eslint-plugin-react:检查 JSX/组件层面的问题,比如 key 缺失、prop-types 校验、无效的 JSX 写法。
  • eslint-plugin-react-hooks:专门检查 Hooks 用法是否符合规则,是这两个插件里实用性最高、也最容易帮你避坑的一个。

react-hooks/exhaustive-deps:最值得关注的一条规则

这条规则检查 useEffect/useCallback/useMemo 的依赖数组,是否完整包含了函数体内用到的所有外部变量。依赖数组遗漏是 React 开发里最高频的 bug 来源之一——遗漏的变量会在闭包里"定格"在 effect 首次创建时的值,导致后续更新不生效。

反例:依赖数组遗漏 count,eslint 会标红提醒
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const timer = setInterval(() => {
      console.log(count); // 闭包里的 count 永远是 effect 创建时的 0
    }, 1000);
    return () => clearInterval(timer);
  }, []); // ← eslint 报错:count 被用到了但没写进依赖数组
}
陷阱:不要"消音"这条规则来图省事。 很多人被这条规则烦到之后,会在依赖数组上加 // eslint-disable-next-line react-hooks/exhaustive-deps 直接屏蔽掉。这等于把"编辑器帮你发现的 bug"重新变回"运行时才会暴露的 bug"。正确做法通常是:要么把用到的变量加进依赖数组,要么用 useRef/函数式更新等方式避免依赖闭包里的旧值,而不是压制警告。这也是面试高频问题:"exhaustive-deps 报错了但你觉得不需要加依赖,该怎么办?"——考察的就是对闭包和 effect 生命周期的理解。

配置文件结构

现代 ESLint(9.x+)使用 flat config(eslint.config.js),核心结构是一个数组,每一项配置一部分文件的规则:

eslint.config.js(简化示例)
import react from 'eslint-plugin-react';
import reactHooks from 'eslint-plugin-react-hooks';

export default [
  {
    files: ['**/*.{js,jsx}'],
    plugins: { react, 'react-hooks': reactHooks },
    rules: {
      'react-hooks/rules-of-hooks': 'error',      // Hooks 必须顶层调用,违反直接报错
      'react-hooks/exhaustive-deps': 'warn',     // 依赖数组遗漏,给警告
      'react/prop-types': 'off',              // 用 TS 的话不需要这条
    },
  },
];

每条规则可以设置为 "off"(关闭)、"warn"(警告,不阻断)、"error"(报错,常用来配合 CI 阻断合并)。react-hooks/rules-of-hooks 检查的是"Hooks 只能在组件顶层/自定义 Hook 里调用,不能出现在条件语句、循环、普通函数里"这条硬性规则,几乎所有项目都应该设为 error。

为什么 rules-of-hooks 要设成 error 而不是 warn? 违反这条规则不是"风格问题",而是会让 React 内部维护 Hook 状态的链表直接对不上,导致状态错乱这种硬故障。风格类问题(如缩进、引号)适合用 warn 或干脆交给 Prettier;会导致运行时错误的规则应该设成 error,配合 CI 强制拦截。

小结

  • ESLint 是静态分析工具,不运行代码,靠解析语法树在写代码阶段就发现潜在 bug 和风格问题,比等到运行时或 review 时发现成本低得多。
  • 它和 TypeScript 分工不同:TS 管类型对不对,ESLint 管代码风格和潜在逻辑问题,两者通常一起用。
  • React 特有的检查靠 eslint-plugin-react(JSX/组件层面)和 eslint-plugin-react-hooks(Hooks 用法)两个插件补充。
  • react-hooks/exhaustive-deps 检查依赖数组是否遗漏,是最高频的 bug 来源之一;遇到报错应该修正依赖或改用 useRef 等方式,而不是直接屏蔽警告。
  • 规则严重级别分 off/warn/error:会导致运行时硬故障的规则(如 rules-of-hooks)应设为 error 并接入 CI 拦截。
