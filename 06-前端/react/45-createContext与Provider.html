createContext 与 Provider：跨层级传数据

当一份数据（主题、当前用户、语言）需要被很多层深处的组件用到时，一层层用 props 往下传会让中间那些"根本不关心这个数据"的组件被迫当搬运工。Context 就是为解决这个问题而生的。

它解决的痛点：prop drilling（逐层透传）

假设 App 里有当前用户 user，最深处的 Avatar 要用它。中间隔着 Layout、Header、Nav——这三层自己压根不用 user，却必须声明 user prop 再往下传。这就是 prop drilling（属性钻透）。

App(user)
→
Layout
→
Header
→
Nav
→
Avatar(用 user)

中间三层被迫透传，坏处：① 代码噪音，每层多写一份 prop；② 想改数据结构，一整条链都要动；③ 中间组件被"不属于它的东西"污染，复用性变差。Context 让数据"直接空投"到需要它的组件，跳过中间层。

心智模型：Context 像给组件树接了一根"隐形电线"。Provider 是插座（供电），任意后代组件用 useContext 就能"接电"，不用中间组件帮忙拉线。

第一步：createContext 创建一个 Context 对象

createContext(defaultValue) 返回一个对象，里面带着 Provider 组件。通常单独放一个文件，方便多处 import。

ThemeContext.js —— 创建 Context
import { createContext } from 'react';

// 参数是"默认值"：仅当组件外面没有对应 Provider 时才生效
// 这里给一个合理默认，也方便编辑器推断类型
export const ThemeContext = createContext('light');

第二步：Provider 提供 value

用 <ThemeContext.Provider value={...}> 包住子树，value 就是要下发的数据。被它包裹的所有后代，无论多深，都能读到这个 value。

App.jsx —— 在顶层提供 value
import { useState } from 'react';
import { ThemeContext } from './ThemeContext';

function App() {
  const [theme, setTheme] = useState('dark');

  return (
    // value 会下发给整棵子树；这里传的是当前 theme 状态
    <ThemeContext.Provider value={theme}>
      <Layout />          // 中间层不用再接收 theme prop
    </ThemeContext.Provider>
  );
}
React 19 起可以直接把 Context 对象当组件用：<ThemeContext value={theme}>，不必再写 .Provider。旧写法仍然有效，本系列示例沿用 .Provider 以兼容。

关键行为：value 变化 → 消费者重渲

当 Provider 的 value 变化（React 用 Object.is 比较新旧 value），所有读取该 Context 的后代组件都会重新渲染——即使它们的 props 没变、即使 memo 也拦不住（Context 的更新会穿透 memo）。

setTheme('light')
→
Provider value 变了
→
所有 useContext 消费者重渲
易踩的坑：value 传对象字面量。写 value={{ theme, setTheme }} 时，每次 App 重渲都会生成一个全新对象，Object.is 判定"变了"，于是所有消费者跟着重渲——哪怕 theme 根本没动。第 47 篇专门讲怎么用 useMemo 或拆 Context 来治它。

小结

  • Context 解决 prop drilling：让深层组件跳过中间层直接拿数据。
  • 三件套：createContext(default) 创建 → <Ctx.Provider value> 提供 → 后代 useContext 消费（下一篇）。
  • value 变化会触发全部消费者重渲，且穿透 memo——这既是它的能力，也是性能陷阱的根源。
  • Context 适合"少变、全局、多处读"的数据（主题/用户/语言），不是用来替代所有状态管理的。
