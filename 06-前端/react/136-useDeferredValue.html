React · 并发特性

useDeferredValue

上一篇 useTransition 包的是"触发更新的那个动作"——你得改代码,拆出一个新 state、手动调用 startTransition。但如果这个大列表组件是别人写的,你拿到的只是一个 query 值,改不了它内部怎么 setState,怎么办?useDeferredValue 换了个角度:不包"动作",直接包"值"。

和 useTransition 是同一个问题的两个入口

两者解决的都是"紧急更新不该被非紧急渲染拖慢"这个并发渲染问题,区别在于你手里有什么、能改什么:

  • useTransition:你能控制"触发更新的代码"——在事件处理函数里,把某次 setState 包进 startTransition,声明"这次更新不紧急"。适合自己写 state 更新逻辑的场景。
  • useDeferredValue:你只能拿到"一个已经存在的值"(可能来自 props、也可能是外部 state,你无法改变它产生的方式),给它生成一个"滞后的版本",下游用滞后版本渲染。适合值的来源不受你控制、或者只是想给某个下游渲染"减速"的场景。

本质上,useDeferredValue 内部也是靠过渡更新实现的——`deferredValue` 从旧值变成新值这件事,本身就是被 React 标记为可中断、可被更紧急任务打断的低优先级更新,原理和 useTransition 完全一致,只是暴露的 API 形态不同。

示例:滞后的搜索关键字驱动昂贵列表

场景不变,还是搜索过滤大列表,但这次不拆 state,直接对 query 求一个滞后值,渲染列表时用这个滞后值:

useDeferredValue:值滞后于最新输入,下游用滞后值渲染
function SearchList({ items }) {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query); // 滞后版本:落后于最新的 query,不阻塞输入

  const isStale = query !== deferredQuery; // 两者不一致 = 列表还没跟上最新输入

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      {isStale && <span style={{ opacity: 0.5 }}>更新中…</span>}
      {/* ExpensiveList 内部按 deferredQuery 过滤/渲染,和 query 完全解耦 */}
      <ExpensiveList query={deferredQuery} items={items} />
    </>
  );
}

// ExpensiveList 甚至可以是第三方组件,只要它拿到的是 deferredQuery 而不是 query 就行——
// 因为 React 内部知道 deferredQuery 的更新是低优先级的,渲染它可以被打断
直觉类比: query 是"此刻的真相",deferredQuery 是"过一会儿才追上来的影子"。打字快的时候,影子会明显落后;一旦你停下来不再输入,React 会趁着空闲把影子"追平"到最新值,触发一次(可被打断的)渲染。isStale(即 query !== deferredQuery)就是用来判断"影子有没有追上",可以用它显示一个轻微的过渡态提示。

useTransition vs useDeferredValue:怎么选

维度useTransitionuseDeferredValue
包裹对象触发更新的动作(一次 setState 调用)已存在的值(state、props 都行)
适用场景自己写事件处理逻辑,能主动拆分 state值来自外部/props,或不想改上游的 setState 逻辑
是否新增 state通常要新增一个 state 承载非紧急部分不需要,直接对现有值取滞后版本
加载态标志isPending(React 提供)需要自己用 value !== deferredValue 比较得出
典型例子tab 切换时明确知道"切换动作"发生的时机包一个第三方/子组件收到的 prop,不方便动它的更新逻辑

都不是防抖(debounce)

很多人第一反应是"这不就是防抖吗",但两者的机制完全不同,搞混是这两个 Hook 最常见的认知陷阱:

防抖 debounce
固定延迟等待:不管系统忙不忙,
都要等满 300ms 没有新输入才触发。
即使设备很空闲、渲染毫秒级完成,
用户也必须白等这 300ms。
是"时间驱动"的延迟策略,
和渲染是否卡顿完全无关。
useTransition / useDeferredValue
没有固定延迟:如果设备空闲、
渲染很快,几乎立刻就更新完成,
用户感觉不到任何延迟。
只有当"更紧急的任务"确实在抢占时,
才会把这次更新往后让一让。
是"优先级驱动"的调度策略,
目的是保护交互流畅度,不是省计算量。
关键区别记一句话就够了: 防抖是"不管忙不忙,都固定等一段时间";并发特性是"不忙就立刻做,忙的时候才把它挤到后面"。如果目的是减少请求次数(比如搜索接口调用频率),那该用防抖/节流;如果目的是让 UI 交互在计算量大时依然流畅,才是 useTransition/useDeferredValue 的用武之地——两者甚至可以叠加使用,互不冲突。

易踩的坑

坑一:期待 useDeferredValue 能减少函数调用次数。 它不会减少 ExpensiveList 被要求渲染的次数,只是让这些渲染变成"可被打断的低优先级任务"——如果渲染函数本身很慢,该优化(memo、虚拟列表)还是得做。

坑二:忘记处理 isStale 态,导致用户以为卡住了。 滞后值和最新值不一致的这段时间里,如果不给任何视觉反馈,用户会觉得"输入了但列表没反应",体验反而变差,一定要配合 value !== deferredValue 展示过渡提示。

坑三:对基本类型之外的复杂对象取 deferredValue。 如果传入的是每次渲染都新建的对象/数组字面量,由于引用不同,React 依然会认为"值变了",滞后机制的效果会打折扣,最好像 useMemo 那样保证引用稳定,或者只对基本类型使用。

小结

  • useDeferredValue 和 useTransition 解决同一个并发渲染问题,区别是包裹对象不同:一个包"值",一个包"触发更新的动作"。
  • 典型用法:const deferredX = useDeferredValue(x),下游渲染用 deferredX,用 x !== deferredX 判断是否"还没追上"。
  • 特别适合值来自外部/props、或不方便改上游 setState 逻辑的场景。
  • 它和防抖本质不同:防抖是固定时间等待,它是"不忙就立刻更新,忙才延后"的优先级调度,不减少计算量本身。
  • 不能替代已有的性能优化手段(memo、虚拟列表),它解决的是调度顺序,不是算法效率。
