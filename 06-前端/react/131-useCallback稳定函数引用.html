useCallback 稳定函数引用

第 129 篇留了一个坑没填:传给 memo 子组件的回调函数,只要是在父组件渲染函数体内直接写的箭头函数,每次渲染都是全新引用,浅比较必然判定"变了",memo 直接失效。useCallback 就是专门解决这个问题的。

问题根源:函数组件每次渲染都创建全新的函数

这其实是第 123 篇讲过的重渲染机制的自然结果:函数组件每次渲染都是重新执行一遍函数体。函数体里写的 const handleClick = () => {...} 本质上是一条普通的 JS 语句——每执行一次函数体,就会在内存里创建一个新的函数对象。哪怕两次创建出来的函数逻辑完全相同,它们在内存里也是两个不同的对象,=== 比较为 false。

验证:每次渲染函数引用都不同
function Demo() {
  const handleClick = () => console.log('click');
  // 第一次渲染:handleClick 是函数对象 A
  // 触发重渲染后:handleClick 是全新的函数对象 B
  // A === B 永远是 false,即便两者代码逐字符相同
  // ...
}

会带来什么实际问题

单纯"每次都是新函数"本身不算问题——真正出问题的是这个新函数被用在依赖比较的地方:

  1. 传给 memo 包裹的子组件:子组件的 props 里有这个函数,浅比较必然判定"变了",子组件被迫重渲染,memo 白加了。
  2. 作为其他 Hook 的依赖项:比如 useEffect(() => { ... }, [callback]),只要 callback 每次渲染都是新引用,这个 effect 就会在每次渲染后都重新执行——即便你的本意是"只在真正需要时才重新执行"。

反例:memo 子组件因为内联回调函数而失效

反例:onSelect 每次都是新函数,ExpensiveList(memo)照样重渲染
const ExpensiveList = memo(function ExpensiveList({ items, onSelect }) {
  console.log('ExpensiveList 渲染了');
  return /* ...渲染 items,点击项调用 onSelect... */;
});

function App() {
  const [count, setCount] = useState(0);
  const [items] = useState([/* ... */]);

  // 每次 App 渲染,这个箭头函数都是新的引用
  const handleSelect = (id) => console.log('选中', id);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {/* items 引用没变,但 onSelect 每次都是新函数 → memo 浅比较照样判定"变了" */}
      <ExpensiveList items={items} onSelect={handleSelect} />
    </>
  );
}

正例:useCallback 缓存函数引用

useCallback(fn, deps) 的行为和 useMemo 完全一致:deps 不变就返回上一次那个函数对象(同一个引用),不会创建新的;deps 变了才创建新函数并更新缓存。

正例:useCallback 让 handleSelect 引用保持稳定
function App() {
  const [count, setCount] = useState(0);
  const [items] = useState([/* ... */]);

  // 依赖数组是 []:handleSelect 只在组件首次渲染时创建一次,之后永远复用同一个引用
  const handleSelect = useCallback((id) => {
    console.log('选中', id);
  }, []);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {/* items 和 onSelect 引用都稳定 → ExpensiveList(memo)真正跳过重渲染 */}
      <ExpensiveList items={items} onSelect={handleSelect} />
    </>
  );
}
闭包陷阱: useCallback 缓存的是"函数对象",不是"最新逻辑"。如果依赖数组漏写了函数内部用到的某个 state/prop,这个函数在后续渲染中会一直"记着"创建它那一刻的旧值——这是闭包陈旧(stale closure)问题,和 useEffect 依赖漏写导致的 bug 是同一类根因。规则很简单:函数体内用到的所有会变化的变量,都要放进依赖数组,不能为了"少重新创建"就故意漏写。

和 useMemo 的关系:本质是同一个东西

useCallback(fn, deps) 是 useMemo 的一个语法糖,等价于用 useMemo 缓存"返回这个函数"这件事本身:

useCallback 等价于 useMemo(() => fn, deps)
// 这两行完全等价
const handleSelect = useCallback((id) => doSomething(id), [doSomething]);
const handleSelect = useMemo(() => (id) => doSomething(id), [doSomething]);

区别只是: useMemo 缓存的是调用计算函数后得到的结果(比如一个排序后的数组);useCallback 缓存的是函数本身,不会去"调用"它——它只是把你传入的那个函数,在依赖不变时原样返回同一个引用。

useMemouseCallback
缓存的是什么函数执行后的返回值函数本身(不执行,只保留引用)
典型场景排序/过滤后的数组、聚合统计结果传给 memo 子组件的回调、useEffect 依赖里的函数
写法useMemo(() => expensiveCalc(a, b), [a, b])useCallback((x) => doSomething(x, a), [a])
什么时候值得用 useCallback: 只有当这个函数确实会被传给 memo 组件、或用作其他 Hook 的依赖项时,稳定引用才有意义。如果一个函数只是普通事件处理器,组件本身没被 memo 包裹、也没被其他 Hook 依赖,那么加不加 useCallback 对性能毫无影响——徒增代码复杂度。这一点和上一篇 useMemo 的陷阱是同一类问题:到处滥用 useMemo/useCallback,不仅不会让应用更快,反而增加了心智负担和额外的内存/比较开销,这是初学者(甚至有经验的开发者)都容易踩的坑。

小结

  • 函数组件每次重渲染都会创建全新的函数对象,即便逻辑完全相同,引用也不同——这是导致 memo 失效、useEffect 依赖误触发的根源。
  • useCallback(fn, deps) 在依赖不变时返回同一个函数引用,配合传给 memo 子组件的回调可以让 memo 真正生效。
  • 本质上 useCallback(fn, deps) 等价于 useMemo(() => fn, deps)——区别只是 useMemo 缓存"调用结果",useCallback 缓存"函数本身"。
  • 陷阱一:依赖数组漏写函数内部引用的变量会导致闭包陈旧(拿到旧值),和 useEffect 依赖问题同根同源。
  • 陷阱二:只有函数会被 memo 组件消费、或作为其他 Hook 依赖时,useCallback 才有意义;普通事件处理器加了纯属浪费,到处滥用反而增加复杂度。
