组件 Props 类型定义(interface / type)

JS 组件的 props 是"运行到哪算哪"——传错了、少传了,只有跑起来才知道。TypeScript 给 props 加上类型契约:调用方少传必填项、传错类型,写代码时编辑器就标红,不用等到浏览器报错。这一篇讲清 props 类型怎么写、interface 和 type 怎么选、children 怎么标注。

最基本的写法:定义一个 Props 类型

组件本质是一个函数,props 是它的唯一参数。给这个参数标注类型,就是"组件 Props 类型定义"的全部含义:

用 interface 定义 Props,函数参数处解构
interface UserCardProps {
  name: string;
  age: number;
  isVip?: boolean;        // ? 表示可选属性
}

function UserCard({ name, age, isVip = false }: UserCardProps) {
  // isVip 没传时,TS 知道它是 boolean,不是 undefined —— 因为给了默认值
  return <div>{name} {age}{isVip && '⭐'}</div>;
}

// 用的时候:少传 age,编辑器立刻标红,不用跑起来才发现
// <UserCard name="Tom" />  ❌ Property 'age' is missing

这跟本系列前面讲过的"函数参数类型标注"是同一套语法——组件只是一个参数是 props、返回值是 JSX 的函数,并没有什么特殊魔法。

interface 与 type:怎么选

两者在定义 props 形状上90% 场景等价,但设计目的不同:interface 是"描述一个对象的形状,并允许后续扩展";type 是"给任意类型起一个别名",能力更广(可以表示联合类型、元组、映射类型等)。

interface
interface ButtonProps {
  label: string;
}
// 可以声明合并 / 被 extends 扩展
interface ButtonProps {
  disabled?: boolean;
}
interface IconButtonProps extends ButtonProps {
  icon: string;
}
type
type ButtonProps = {
  label: string;
};
// 能表达 interface 做不到的联合类型
type Size = 'sm' | 'md' | 'lg';
type ButtonProps = {
  label: string;
  size: Size;
};
怎么选? 大部分团队的实际约定很简单:props 形状用 interface,涉及联合类型 / 工具类型组合用 type。理由是 interface 支持"声明合并"和 extends,更贴近"组件契约会被扩展"的场景;而按钮尺寸 'sm' | 'md' | 'lg' 这种联合类型只有 type 能直接表达。二者选一个团队统一风格远比纠结"哪个更好"重要。

children 的类型:ReactNode

凡是组件要接收"子元素"(即 JSX 里标签中间的内容),children 属性的类型几乎总是 React.ReactNode——它是 React 里"任何可以被渲染的东西"的类型,涵盖 string、number、JSX.Element、null、undefined、数组等。

children: React.ReactNode
interface CardProps {
  title: string;
  children: React.ReactNode;   // 字符串、数字、JSX、数组、null 都合法
}

function Card({ title, children }: CardProps) {
  return (
    <div>
      <h3>{title}</h3>
      {children}
    </div
  );
}

// <Card title="标题">这里随便写文字、图标、别的组件都行</Card>
陷阱: 别用 JSX.Element 当 children 的类型——它只覆盖"单个 JSX 元素",排除了字符串、数字、数组、null 这些同样合法的 children,会导致 <Card>纯文本</Card> 这种常见写法直接报类型错误。ReactNode 才是"能塞进 JSX 里的任何东西"的正确类型。二者的区别在后面《FC 类型与 ReactNode / ReactElement》一篇会详细展开。

可选属性与默认值:两种互补的写法

可选属性(?:)和默认值(解构时 = xxx)经常一起出现,但作用不同:? 是告诉 TS"调用方可以不传",默认值是告诉 JS"不传时用什么兜底"。

可选属性 + 解构默认值:组合使用
interface AlertProps {
  message: string;
  type?: 'info' | 'warning' | 'error';  // 不传时允许 undefined
}

function Alert({ message, type = 'info' }: AlertProps) {
  // 函数体内 type 的类型已经收窄为 'info' | 'warning' | 'error'(不再含 undefined)
  return <div className={`alert-${type}`}>{message}</div>;
}
只写 ? 不给默认值会怎样? 函数体内该属性的类型会保留 | undefined,你需要自己处理"没传"的情况(比如 type ?? 'info')。而解构时给了默认值,TS 会自动把 undefined 从类型里排除掉——这也是为什么给默认值的可选 props,不需要在函数体里再写一遍空值判断。

事件类 / 回调类 props 先留个坑

props 里经常有 onClick、onChange 这类回调函数,它们的类型怎么写、跟原生 DOM 事件类型有什么关系,下一篇《事件类型 React.ChangeEvent / MouseEvent》会专门讲,这里先按下不表。

和函数组件的关系: 不管组件内部用不用 React.FC(下下篇会讲它的争议),props 类型定义本身都独立于组件的声明方式——先把 interface XxxProps 写清楚,再决定函数怎么标注返回值,是更稳妥的顺序。

小结

  • 组件本质是"参数为 props、返回 JSX 的函数",props 类型定义就是给这个参数标类型,写法和普通函数参数标注一致。
  • 形状简单、需要被 extends 扩展用 interface;涉及联合类型 / 工具类型组合用 type,团队统一风格比纠结哪个"更好"更重要。
  • children 应标注为 React.ReactNode,而不是 JSX.Element——后者排除了字符串、数字、数组等同样合法的 children。
  • ? 表示"调用方可以不传",默认值负责"不传时用什么兜底";两者搭配后函数体内该属性类型会自动排除 undefined。
  • 回调类 props(如 onClick)的具体事件类型,留到下一篇结合 ChangeEvent / MouseEvent 一起讲。

与其他知识点的联系

  • 组件的返回值约束: props 类型定义约束的是组件"接收什么",而《组件返回值约束》约束的是组件"能产出什么"——两者合起来才是一个组件完整的类型契约:输入合法才能进函数体,输出合法才能被父组件渲染。理解 props 类型时想清楚"这是函数参数",理解返回值时同样要记住"组件只是一个普通函数",是同一套心智模型的两个方向。
  • FC 类型与 ReactNode / ReactElement: 本篇结尾已经留了钩子——props 类型定义本身独立于组件用什么方式声明(React.FC 还是普通函数)。下一步要理清 React.FC 为什么不推荐,以及 children: ReactNode 这个约定和 ReactElement 的区别,这两篇实际上是同一个"给组件加类型契约"话题的连续两步。
  • useState / useRef 泛型: props 类型是"描述一个对象形状"的泛型应用场景之一,而 useState<T>()、useRef<T>() 是"给 Hook 的返回值标注类型"的泛型应用场景——都是 TypeScript 泛型在 React 里的具体落地,写法上有共通的心智模型:先想清楚"这里流动的数据是什么类型"。
  • CSS Modules 的类型: 如果项目同时用了 CSS Modules 和 TypeScript,import styles from './x.module.css' 拿到的对象本身也需要类型声明(靠 *.module.css.d.ts 或插件生成),这和给 props 写 interface 是同一类"给外部输入的数据加类型契约"的工程习惯,只是作用对象从组件参数变成了 CSS 类名映射对象。
  • ESLint 与 TypeScript 的分工: props 类型少传、传错类型这类问题由 TypeScript 在编译期拦截,而 ESLint(尤其是 eslint-plugin-react 的 prop-types 规则)历史上是给纯 JS 项目做同样事情的替代方案——用了 TypeScript 后这条规则通常要关掉,因为职责已经被类型系统接管,这也是后面《ESLint 与 eslint-plugin-react》一篇会提到的分工边界。
