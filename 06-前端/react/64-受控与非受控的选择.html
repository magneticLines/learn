一条铁律:拿不准就用受控
React 官方和社区的默认立场是优先受控。原因:受控让状态可预测、可测试、易于加功能;非受控省下的代码在需求变复杂时(突然要加实时校验)往往要推倒重来。只有在明确"就是个简单表单""要接第三方 DOM 库""必须 file input"时,才选非受控。

混用陷阱:从非受控切换到受控的警告

这是新手最常撞的错误。同一个 input,在生命周期里时而受控、时而非受控,React 会在控制台报出经典警告:

Warning: A component is changing an uncontrolled input to be controlled. This is likely caused by the value changing from undefined to a defined value.
❌ 反例:初值 undefined
// name 初始是 undefined
const [name, setName] = useState();

// value={undefined} → 被当成非受控
// 用户输入后 setName('x')
// value 变成字符串 → 切成受控
// React 报警告!
<input value={name}
  onChange={e =>
    setName(e.target.value)} />
✅ 正例:初值给确定值
// 初值给空字符串,自始至终受控
const [name, setName] =
  useState('');

<input value={name}
  onChange={e =>
    setName(e.target.value)} />

为什么会这样?

React 在首次渲染时就根据 value 是不是 undefined/null 决定这个 input 是受控还是非受控,并且要求整个生命周期不能变。初值是 undefined,React 判定"非受控";等异步数据回来把值填成字符串,又变"受控"——身份切换,于是报警。

常见触发场景:异步数据 + 直接绑定
const [user, setUser] = useState(null);  // 初始 null
useEffect(() => { fetchUser().then(setUser); }, []);

// ❌ user 为 null 时 user?.name 是 undefined → 非受控;数据回来 → 受控
<input value={user?.name} onChange=... />

// ✅ 兜底成空字符串,永远受控
<input value={user?.name ?? ''} onChange=... />
修复口诀:受控 input 的 value 永远不能是 undefined/null。文本框兜底成 '',数字框想清空也用 '' 而非 0,checkbox 的 checked 兜底成 false。一个 ?? '' 就能消除 90% 的这类警告。

反向警告也存在

相反方向——从受控变非受控(value 从字符串变回 undefined)也会报 "changing a controlled input to be uncontrolled"。根因相同:身份在生命周期中变了。解法也相同:保证 value 类型稳定。

小结

  • 受控:实时校验/联动首选;非受控:简单表单、只提交时取值、file input、集成 DOM 库。
  • 铁律:拿不准就用受控——可预测、可测试、易扩展,是官方默认推荐。
  • 混用陷阱:同一 input 在受控/非受控间切换会报警告,根因是 value 在 undefined 与确定值间跳变。
  • 修复:受控 input 的 value 永远给确定值,用 ?? '' 兜底,别让它出现 undefined/null。
