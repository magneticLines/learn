依赖数组：控制 effect 何时重跑

useEffect 的第二个参数——依赖数组，是整个 Hooks 里最容易踩坑的地方。它决定 effect "什么时候重新执行"，而闭包捕获和依赖不全导致的过期值 bug，是 React 开发者共同的痛点。这篇务必吃透。

三种形态

写法执行时机典型用途
useEffect(fn)
（无数组）		每次渲染后都执行几乎不用，通常是写漏了
useEffect(fn, [])
（空数组）		仅挂载时执行一次一次性初始化、订阅
useEffect(fn, [a, b])挂载时 + a 或 b 变化后最常用，随依赖同步
三种形态对照
// ① 无依赖数组:每次渲染后都跑（含每次 setState 触发的重渲染）
useEffect(() => { console.log('每次渲染'); });

// ② 空数组:只在挂载后跑一次，卸载时执行清理
useEffect(() => { console.log('只在挂载'); }, []);

// ③ 有依赖:count 变化才跑（React 用 Object.is 逐个比较依赖）
useEffect(() => { console.log('count 变了', count); }, [count]);

React 每次渲染后，把本次依赖数组和上次的逐项 Object.is 比较；只要有一项不同，就重跑 effect（先跑上次的清理函数）。空数组永远"和上次相同"，所以只跑一次。

核心陷阱:闭包捕获了旧值

这是重灾区中的重灾区。effect 的回调是一个闭包，它捕获的是创建它的那次渲染里的 state / props。如果依赖数组不让它重建，它就永远看到那次的旧值。

经典 bug:定时器里的 count 永远是 0
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      // ❌ 这里的 count 被"钉死"在首次渲染的 0
      setCount(count + 1);  // 永远是 0 + 1 = 1
    }, 1000);
    return () => clearInterval(id);
  }, []); // 空数组:effect 只建一次，闭包锁死初始 count
  // 现象:显示 1 后就不动了，不是 1、2、3…
}

为什么？[] 让 effect 只在挂载时创建那一次，此时闭包里的 count 是 0。之后 count 变了，但那个定时器回调还引用着老闭包里的 0，所以永远 setCount(1)。

修复一:用函数式更新，摆脱对 count 的依赖

最佳解:setCount(c => c + 1)
useEffect(() => {
  const id = setInterval(() => {
    // ✅ 不读闭包里的 count，让 React 传入"最新的值"
    setCount(c => c + 1);
  }, 1000);
  return () => clearInterval(id);
}, []); // 现在空数组是对的:effect 不再依赖 count

修复二:老实把 count 加进依赖

也可行:count 变就重建定时器
useEffect(() => {
  const id = setInterval(() => setCount(count + 1), 1000);
  return () => clearInterval(id);
}, [count]); // count 每变一次就清旧定时器、建新的（闭包刷新）
// 能工作，但每秒重建定时器，不如"修复一"优雅
永远不要"为了消警告而说谎"。最坏的做法是：effect 里用了 count，却写 [] 骗过 lint。依赖数组必须诚实反映 effect 里用到的所有响应式值。如果诚实填写后发现"重跑太频繁"，那是在提示你——effect 的逻辑该重构（比如改用函数式更新、把值移出、拆分 effect），而不是删依赖。

依赖数组的心智模型

官方推荐的思路：不要问"我想让 effect 什么时候跑"，而要问"这个 effect 用到了哪些值"。把用到的响应式值全列进去，effect 自然会在这些值任何一个变化时同步。依赖数组不是"触发条件配置"，而是"这段代码依赖了什么"的如实声明。

什么算"响应式值"？组件内部会随渲染变化的东西：props、state、以及由它们计算出的变量或函数。反之，useState 返回的 setter、useRef 的 ref 对象、模块级常量是稳定的，不必也不该放进依赖（React 保证 setter 引用不变）。
StrictMode 会让 effect 在开发时跑两遍（挂载 → 卸载清理 → 再挂载）。这不是 bug，是 React 故意帮你检验"清理函数写对了没"。如果你的 effect 在跑两遍时出问题，说明清理逻辑有缺陷——正好该在开发期暴露（下一篇讲清理函数）。
面试高频。"useEffect 里的定时器为什么读到旧 state？" 标准答案要点：① effect 回调是闭包，捕获创建它那次渲染的值；② 空依赖数组让它只创建一次，值被锁死；③ 解法是函数式更新 setCount(c => c+1) 或用 useRef 存最新值。能把"闭包 + 渲染快照"讲清楚，基本就过关了。

与其他知识点的联系

  • 清理函数（下一篇）：依赖数组不仅决定"何时重跑"，也决定"何时先清理"——依赖变化时 React 会先执行上一次 effect 的清理函数，再用新依赖重新执行 setup。理解依赖数组是理解清理函数触发时机的前提，两者本质是同一个机制的两面。
  • useRef 存储可变值：本篇定时器 bug 的两种修复方式之一是函数式更新，另一种更通用的方式是用 useRef 存"永远最新的值"，让闭包通过 ref.current 绕开依赖数组的限制。这是因为 ref 对象引用不变、但内容可以随时更新，闭包读它读到的永远是最新写入的值，不受"闭包锁定创建时刻的值"这条规则约束。
  • Hooks 只在函数组件中调用：本篇反复强调 effect 回调是"渲染之后才执行的闭包"，这正好呼应了 Hooks 规则里"事件处理器、setTimeout 回调不是渲染期"的说法——effect 回调同样不在渲染期执行，所以它内部不能再调用其他 Hook，只能读之前渲染阶段已经创建好的闭包变量。
  • useReducer 的 dispatch 稳定性：本篇提到 setState 的 setter 是"稳定"的、不必放进依赖数组。同样的稳定性也适用于 useReducer 返回的 dispatch 函数——这也是为什么后面 Context 相关文章会把 dispatch 单独拆到一个 Context 里，因为它的引用永远不变，天然适合当作"不触发额外更新"的依赖或传递对象。
