声明式 vs 命令式

这是学习 React 必须建立的最根本的思维转变。命令式和声明式不只是两种不同的写法，而是两种完全不同的"与程序对话的方式"。

命令式：告诉计算机"怎么做"

命令式编程的核心是步骤。你把每一步操作写清楚，告诉计算机"先做什么，再做什么，然后做什么"。计算机照单全收，按顺序执行。

jQuery 就是典型的命令式：找到那个按钮，给它加监听器，用户点了之后，找到那个 <span>，读取它当前的文本，转成数字，加一，再写回去，再判断是否要显示警告……每一步都要你亲自交代。

声明式：告诉计算机"是什么"

声明式编程的核心是目标状态的描述。你不说"怎么做"，你说"结果应该是什么样的"，然后让运行时（React、数据库、CSS 引擎）自己去想办法实现它。

React 是声明式的：你描述"当 count 等于某个值时，界面应该长什么样"，仅此而已。你不说"去找那个 DOM 节点"，不说"改它的文本"，不说"判断是否要加 class"——这些都是 React 的事。

导航类比：命令式就像给司机逐步导航——"出门左转，走 200 米，在红绿灯右转，再走 500 米，到路口左转……"；声明式就像告诉导航软件目的地——"去浦东机场 T2 航站楼"，然后坐好，导航自己规划路线、实时调整。你只描述想去哪，不描述怎么走。

代码对比：同一个计数器，两种写法

用一个最简单的计数器来感受区别：点击按钮，数字加一，超过 5 显示警告。

命令式（jQuery）
// 每次点击，手动找节点、手动改内容
// 你在描述"操作步骤"
$('#btn').click(function() {
  const n = parseInt(
    $('#num').text()
  );
  $('#num').text(n + 1);

  // 警告逻辑：手动判断 + 手动操作
  if (n + 1 > 5) {
    $('#warn').show();
  } else {
    $('#warn').hide();
  }
});
声明式（React）
// 只描述"状态→UI的对应关系"
// 没有一行是"操作 DOM"的指令
function Counter() {
  const [count, setCount] =
    useState(0);

  return (
    <div>
      <p>{count}</p>
      {count > 5 &&
        <p>警告！</p>}
      <button onClick={() =>
        setCount(c => c + 1)}>
        +1
      </button>
    </div>
  );
}

React 版本里，没有"找 DOM"、没有"改 DOM"、没有"show/hide"。你只描述了：

  • 当前数字是 {count}（状态到 UI 的映射）
  • 当 count 大于 5 时，有一个警告段落（条件渲染的声明）
  • 点击按钮时 count 加一（状态变更，而非 DOM 变更）

这个"状态→UI 的对应关系"只写了一次，从此无论 count 怎么变，React 都知道界面应该怎么更新。

为什么声明式的优势随复杂度放大

当状态只有一个 count，命令式也还好。但当业务变复杂——登录状态、购物车数量、用户权限、消息未读数……同时影响界面上十几处地方——命令式代码就变成了噩梦：你要为每一种状态变化，写一套"找节点、改节点"的操作流程，稍有遗漏就是 bug。

声明式的优势是：你只描述一次"状态→UI 的完整映射"。状态不管怎么变、不管从哪里变，React 都用这同一份描述来计算界面，永远不会有"某个地方忘了更新"的问题。

命令式声明式（React）
核心关注每一步怎么操作（How）目标状态是什么（What）
简单场景代码更直接，容易理解相对有些"绕"
复杂场景步骤爆炸，极难维护仍然只描述一份映射，线性增长
Bug 来源某处操作被遗漏，UI 和数据不一致状态本身出错，UI 忠实反映状态
测试难度需要模拟 DOM 操作序列只需给定状态，断言渲染结果
常见误区：在 React 里用命令式思维
表现一：写了一堆 useEffect，里面是 document.querySelector(...).style.display = 'none'——这是在 React 里写 jQuery。

表现二：用 ref 读取 DOM 节点的值来"同步状态"（比如 inputRef.current.value），而不是用受控组件的 value + onChange。

根本原因：没有完成从"操作 DOM"到"描述状态"的思维转变。每次想到要"改某个元素"，停下来问自己：我应该改哪个 state？状态变了，React 会自动改那个元素。
判断你是否真正理解了声明式：当界面需要变化时，你脑子里第一个反应是"哪个状态变了"而不是"哪个 DOM 节点要改"——那么你已经建立了正确的 React 思维。
