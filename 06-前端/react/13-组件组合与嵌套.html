React · 函数组件

组件组合：像搭积木一样构建 UI

React 的核心哲学不是"写更复杂的组件"，而是"把复杂 UI 拆成简单组件，再组合起来"。这和搭积木一样——你先有各种形状的小块，再把它们拼成你想要的东西。

组合优先于继承：React 的设计哲学

传统面向对象编程里，复用 UI 逻辑的方式是继承：class Button extends BaseComponent。但 React 官方文档明确说：不推荐用继承来复用组件。

为什么？因为继承是"is-a"关系（Button 是一种 BaseComponent），而 UI 复用更多是"has-a"关系（UserCard 包含一个 Avatar）。继承一层套一层，改动牵一发动全身；组合是把独立的小零件拼在一起，每个零件只管自己，改动互不影响。

React 官方原话："If you want to reuse non-UI functionality between components, we suggest extracting it into a separate JavaScript module. The components may import it and use that function, object, or class, without extending it." — 复用逻辑用提取模块或自定义 Hook，不用继承。

三层结构示例：原子 → 分子 → 页面

真实项目里，组件天然形成层级——小组件被大组件复用，大组件被页面复用。这和化学里的"原子 → 分子 → 物质"一样，因此业界称为 Atomic Design（原子设计）：

三层组件结构——每层只关心自己的职责
// ① 原子层：最小的可复用单元，只做一件事
//    Avatar 只负责显示头像，不关心谁在用它
function Avatar({ src, name }) {
  return <img src={src} alt={name} className="avatar" />;
}

// ② 分子层：由原子组合而成，代表一个独立的 UI 概念
//    UserCard 组合 Avatar + 文字信息，但不关心列表逻辑
function UserCard({ user }) {
  return (
    <div className="user-card">
      {/* 复用原子组件 Avatar——改 Avatar 样式，所有用到它的地方都更新 */}
      <Avatar src={user.avatar} name={user.name} />
      <div className="user-info">
        <h3>{user.name}</h3>
        <p>{user.bio}</p>
      </div>
    </div>
  );
}

// ③ 页面层：组合多个分子，处理数据和布局
//    UserList 只管循环渲染，不关心每个卡片长什么样
function UserList({ users }) {
  return (
    <section className="user-list">
      {users.map(u => (
        // key 放在最外层——UserCard 不需要知道自己有 key
        <UserCard key={u.id} user={u} />
      ))}
    </section>
  );
}

这种分层结构映射到文件系统：

src/
  components/
    Avatar.jsx        ← 原子，被多处复用
    UserCard.jsx     ← 分子，复用 Avatar
  pages/
    UserList.jsx     ← 页面，复用 UserCard

何时应该拆分组件

没有绝对规则，但有四个明确信号说明该拆了：

信号为什么要拆
代码超过 100 行 单个组件太长，一眼看不清整体结构；拆开后每个文件职责单一，可读性大幅提升
某段 JSX 在多处重复 重复 = 抽象的信号；提取成组件后，改一处生效所有位置，消除"改一个漏一个"的 bug
逻辑职责明显不同 列表和列表项是两种职责：列表管循环，列表项管展示；混在一起各自的逻辑都难看清
某部分有独立的 state 独立状态 = 独立生命周期；有自己 state 的 UI 单元天然应该是独立组件，方便隔离更新

不要在组件内部定义组件

这是最高频且最隐蔽的 bug 来源之一。在函数组件的函数体里定义另一个组件，看起来是封装，实际上会造成严重的性能和状态问题：

❌ 错误：组件内部定义组件
function Parent() {
  // 每次 Parent 渲染，
  // Child 都是一个全新的函数！
  // 函数引用变了，React 销毁
  // 旧 Child 重建新 Child
  function Child() {
    const [n, setN] = useState(0);
    return <button onClick={() =>
      setN(n+1)}>{n}</button>;
  }

  return <Child />;
  // 问题：点击按钮后父组件
  // 触发渲染，Child 状态归零！
}
✅ 正确：提到外部定义
// Child 定义在 Parent 外部
// 函数引用在组件生命周期内
// 保持稳定，React 不会重建
function Child() {
  const [n, setN] = useState(0);
  return <button onClick={() =>
    setN(n+1)}>{n}</button>;
}

function Parent() {
  // 每次渲染，Child 引用不变
  // React 识别为同一组件，
  // 状态正常保留
  return <Child />;
}
为什么内部定义会出问题？
React 通过函数引用是否相同来判断前后两次渲染里的组件是不是"同一个"。如果在父组件里定义子组件，每次父组件渲染，子组件的函数引用都是全新的对象（function Child() {} 每次都创建新函数）。React 认为"这是一个全新的组件类型"，就会卸载旧的、挂载新的，子组件所有状态（useState、useRef）全部丢失。症状：输入框每次失焦都清空、计数器点击后归零、动画每次重置。
过度拆分的代价：文件多不等于更好
把 3 行 JSX 也提取成单独文件，你每次阅读代码都要在多个文件间跳来跳去才能理解一个功能，认知成本反而更高。拆分的目的是"让代码更清晰"，不是"让文件更多"。判断标准：拆完之后，每个组件能不能单独讲清楚它做什么？能的话值得拆；不能的话说明拆过头了。
小结：① React 用组合代替继承——把复杂 UI 拆成小组件，再组合；② 三层结构（原子 → 分子 → 页面）是实际项目的常见模式；③ 四个拆分信号：超 100 行、重复 JSX、职责不同、独立 state；④ 绝对不要在组件内部定义组件——每次渲染新函数引用，React 销毁重建，状态丢失；⑤ 拆分是为了清晰，不是文件数量多。

常见陷阱

  • 组件内部定义组件：在函数体里写 function Child() {}，每次父组件渲染都创建一个新的函数引用。React 判断组件是否"同一个"靠的是函数引用而不是名字，引用变了就认为是全新组件类型，于是卸载旧的、挂载新的，useState、useRef 里的状态全部清空。避免方法很简单：所有组件定义都提到模块顶层，函数体里只写调用（<Child />），不写定义。
  • 滥用继承代替组合：写惯了 OOP 的人容易想用 class Button extends BaseComponent 去复用样式或逻辑，结果是继承链越深越难改——改父类怕影响所有子类，加新子类又要理清一层层的方法重写关系。React 组件之间大多是"has-a"关系（UserCard 里有一个 Avatar），不是"is-a"关系，所以该用组合把小组件拼进来，复用非 UI 逻辑则提取成普通函数或自定义 Hook，不要建继承树。
  • 列表渲染时 key 放错位置或漏写：示例里 key={u.id} 必须放在 map 直接返回的最外层元素（UserCard）上，而不是塞进 UserCard 内部某个子节点。React 靠 key 在两次渲染之间对齐列表项、决定哪些该复用哪些该重建；key 放错层级或者用数组下标当 key，数据顺序一变就会导致状态错位、输入框内容对应到别的行这种诡异 bug。
  • 拆分粒度失控（过度拆分或该拆不拆）：把几行 JSX 也提取成独立文件，读代码变成在文件间反复跳转；反过来一个组件塞进几百行掺杂列表逻辑和展示逻辑，改一处要通读全文件才敢下手。判断标准要回到"职责"：像 UserList 只管循环、UserCard 只管展示单条数据，这种职责边界清晰才该拆成独立组件，不是数行数或数文件数。
  • 三层结构里跨层直接耦合：Atomic Design 的价值在于每层只依赖下一层、不关心上一层是谁在用它——Avatar 不该知道自己被 UserCard 用，更不该知道最终在 UserList 里。一旦原子组件里写了"如果是列表场景就怎样"之类的特判逻辑，等于把上层的知识泄漏进了下层，下次换个使用场景就要重新改这个本该通用的原子组件。

面试高频考点

  1. 为什么 React 推荐组合而不是继承？—— 因为 UI 复用大多是"has-a"关系而不是"is-a"关系：UserCard 包含一个 Avatar，不是"UserCard 是一种 Avatar"。继承会把组件耦合成一条链，改动会沿着继承链向下传播，牵一发动全身；组合是把独立的小组件拼装起来，每个组件只管自己的职责，互相替换和改动不会影响别的组件。非 UI 逻辑的复用应该提取成模块或自定义 Hook，而不是放进基类里靠继承共享。
  2. 为什么不能在函数组件内部定义另一个组件？—— React 用函数引用判断两次渲染中的组件是否是同一个。在父组件函数体里写 function Child(){}，每次父组件重渲染都会重新执行这段代码，产生一个全新的函数对象，即使代码内容完全一样，引用也不相等。React 因此认为这是"全新的组件类型"，卸载旧实例、挂载新实例，导致子组件的所有内部状态被重置。正确做法是把子组件定义放到模块顶层，让它的函数引用在整个应用生命周期内保持稳定。
  3. 什么时候应该把一段 JSX 拆成独立组件？—— 没有绝对行数规则，但有几个信号：代码长到难以一眼看清结构、同一段 JSX 在多处重复出现、某部分逻辑职责明显和周围不同（比如列表循环和列表项展示）、某部分拥有独立的 state。核心判断标准是"拆出来的组件能不能单独说清楚它是干什么的"，能就值得拆，拆分是为了让代码更清晰，不是为了让文件数量变多。

与其他知识点的联系

  • React.memo：本篇讲的"职责单一、避免跨层耦合"的组合原则,直接决定了后面 129 篇 memo 能不能生效——如果组件拆分合理,某个子组件的 props 大多数时候确实不变,memo 的浅比较才有意义；如果组件职责混乱、什么状态都揉在一起，props 几乎每次渲染都真的会变，memo 加了也是白加。
  • 状态下移与内容提升模式：132 篇的"状态下移"本质上是本篇"某部分有独立的 state 就该拆成独立组件"这条拆分信号的延伸——不仅要拆组件，还要把 state 放到真正需要它的那一层；"内容提升"用到的 children 作为 props 传入，也是组合思想的进一步应用：把不相关的子树从外层传进来，而不是写死在内部。
  • useMemo / useCallback：本篇"不要在组件内部定义组件"这条陷阱，和 130、131 篇讲的"内联对象/函数导致引用每次渲染都变化"是同一类根因——组件体内定义组件会导致函数引用不稳定，组件体内定义内联对象/函数会导致 props 引用不稳定，两者都源于"函数组件每次渲染都重新执行函数体"这一底层机制。
  • 虚拟滚动 TanStack Virtual：本篇三层结构示例里的 UserList 组件只管循环渲染、UserCard 只管展示单条数据——这种职责拆分是虚拟滚动能够顺利接入的前提：133 篇的 useVirtualizer 需要清晰地知道"每一项该怎么渲染"，如果列表项和列表本身的逻辑没有拆开，改造成虚拟滚动会更困难。
  • AntDesign 与 shadcn/ui：138、139 篇讲的两种组件库思路，本质上都是"组合优先于继承"这条设计哲学在生态层面的延伸——不管是安装 AntDesign 的黑盒组件再组合使用，还是把 shadcn/ui 的源码复制进项目再自由拼装，都是"拿现成的小组件拼出复杂 UI"，而不是通过继承某个基类来复用功能。
