RTK Query：把接口请求变成声明式数据层

用 createAsyncThunk 手写请求，每个接口都要写 loading/error/data、缓存、防重复请求……重复劳动惊人。RTK Query 是 RTK 内置的数据获取与缓存方案，你只声明"有哪些接口"，它自动生成 hooks、管理缓存、处理失效与重取。

手写 thunk 的问题

上一章的 createAsyncThunk 能用，但每加一个接口，你都要重复：定义 thunk → 在 slice 写 pending/fulfilled/rejected → 组件里 useEffect 触发 → 手动判断是否已加载避免重复请求 → 组件卸载再挂载时又从头 loading……这些都是"服务端状态"的通用需求，不该反复手写。

关键认知：服务端状态 ≠ 客户端状态。接口数据是服务器的"缓存副本"，有新鲜度、需要失效和重取；而计数器、表单、弹窗开关是纯客户端状态。RTK Query 专门服务前者，让你不用把请求生命周期硬塞进普通 reducer。

createApi：声明一个 API 服务

核心是 createApi：指定基础请求方式（baseQuery）和一组 endpoints（每个接口一条），RTK Query 就为每个 endpoint 自动生成一个 React hook。

services/api.js
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

export const api = createApi({
  reducerPath: 'api',          // 缓存挂在 state.api 下
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  tagTypes: ['Post'],       // 用于缓存失效的标签（下一章展开）
  endpoints: (builder) => ({
    // query = 读取（GET）；下方 mutation = 修改
    getPosts: builder.query({
      query: () => '/posts',        // 返回请求路径
      providesTags: ['Post'],       // 这份数据带 Post 标签
    }),
    getPostById: builder.query({
      query: (id) => `/posts/${id}`,
    }),
  }),
});

// 自动生成 hook：命名 = use + Endpoint 名 + Query/Mutation
export const { useGetPostsQuery, useGetPostByIdQuery } = api;

接入 store

store.js —— 挂 reducer 和 middleware
configureStore({
  reducer: { [api.reducerPath]: api.reducer },
  // 必须加 api.middleware——缓存、失效、轮询全靠它
  middleware: (gdm) => gdm().concat(api.middleware),
});

组件里直接用生成的 hook

PostList.jsx
import { useGetPostsQuery } from './services/api';

function PostList() {
  // 一行拿到 data / loading / error，全自动
  const { data, isLoading, isError } = useGetPostsQuery();

  if (isLoading) return <p>加载中…</p>;
  if (isError) return <p>出错了</p>;
  return <ul>{data.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}
自动化都发生了什么？
  • 自动缓存：同一个 endpoint + 同一参数只请求一次，多个组件共享同一份缓存
  • 自动去重：两个组件同时挂载都调 useGetPostsQuery()，只发一次网络请求
  • 引用计数：没有组件再用某份数据时，一段时间后自动清理缓存
  • 自动重取：可配置窗口聚焦、网络恢复时自动刷新
  • 失效驱动：mutation 修改数据后，用 tag 让相关缓存失效并自动重取

对比：手写 thunk vs RTK Query

能力手写 createAsyncThunkRTK Query
loading/error 状态每个接口手写 3 个 casehook 自动返回
缓存去重自己判断"是否已加载"内置，按参数缓存
缓存失效手动 dispatch 重新拉tag 标签自动失效重取
后台刷新/轮询自己写定时器配置项开启
适合场景复杂副作用编排标准 CRUD 数据获取
面试点与取舍：①RTK Query 是 RTK 自带的，如果项目已用 Redux，它是"零额外依赖"的服务端状态方案。②它和后面要讲的 TanStack Query 定位几乎一样，区别是 RTK Query 深度绑定 Redux 生态、更适合已有 Redux 的项目；TanStack Query 更轻量、框架无关。③query 用于读、mutation 用于写，这是 RTK Query 的基本二分。
