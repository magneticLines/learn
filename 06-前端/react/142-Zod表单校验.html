Zod 表单校验

上一篇提到,校验规则散落在各个 register('email', { required: true, pattern: ... }) 里,字段一多就很乱,还容易和后端接口的校验规则重复写两遍。Zod 的做法是:把所有校验规则收进一份 schema(模式定义),一处定义,处处复用。

为什么校验规则不该散落在 onChange 里

最朴素的写法是每个字段自己在 onChange 里写 if 判断:

反例:校验逻辑散落、难维护、无法复用
function handleEmailChange(e) {
  const value = e.target.value;
  if (!value.includes('@')) {
    setEmailError('邮箱格式不对'); // 这条规则只在这一个组件里生效
  }
  // 密码强度、手机号格式……每个字段重复这套模式
  // 而且后端接口很可能要重新写一遍一模一样的校验
}

问题很明显:规则和 UI 代码绑死,不可复用(同样的邮箱格式规则,后端接口校验又要抄一遍)、不可维护(规则改了要满代码库找 onChange)、不好组合(字段之间有依赖关系,比如"确认密码必须等于密码"时更难表达)。

Zod 的思路是把这些规则声明成一份独立的schema(可以理解成"数据的形状说明书"),schema 和 UI 完全解耦——它只描述"一份合法数据长什么样",不关心这份数据来自表单输入框还是 HTTP 请求体。

基本用法:z.object 定义 schema

schema.ts:用 Zod 声明式描述数据形状
import { z } from 'zod';

export const loginSchema = z.object({
  email: z.string().email('请输入正确的邮箱格式'),
  password: z
    .string()
    .min(8, '密码至少 8 位')
    .regex(/[0-9]/, '密码必须包含数字'),
  age: z.number().min(18, '必须年满 18 岁').optional(), // 可选字段
});

// 直接调用 schema 就能拿到校验结果,不依赖任何表单库
// success 为 false 时,error 里有详细的逐字段错误信息
const result = loginSchema.safeParse({ email: 'abc', password: '123' });

注意这里的关键:loginSchema 是一个普通 JS 对象,和 React、和任何表单库都没有关系。你可以在浏览器端用它校验表单,也可以原封不动地在 Node.js 后端接口里用同一份 schema 校验请求体——这就是"一份规则,处处复用"。

杀手级特性:z.infer 自动推导 TypeScript 类型

写 TypeScript 项目时,通常要为一份数据手写一遍 interface/type,再另外写一遍校验逻辑,两份代码很容易改一个忘改另一个,产生"类型说这个字段是必填,但校验逻辑里忘了加"的不一致。Zod 用 z.infer 解决了这个重复:

z.infer:从 schema 直接拿到对应的 TS 类型
import { z } from 'zod';

const loginSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
});

// 不用手写 interface LoginForm { email: string; password: string },
// TS 类型直接从 schema "推导"出来,schema 改了类型自动跟着改
type LoginForm = z.infer<typeof loginSchema>;
// 等价于 type LoginForm = { email: string; password: string }
这解决了什么真实痛点? 没有 Zod 时,你要么"类型和校验各写一遍、手动保持同步"(容易漂移),要么"干脆不写类型、校验逻辑里用 any"(失去类型安全)。Zod 让 schema 成为唯一真源(single source of truth)——改一处,类型和校验规则同时更新,不可能出现两者不一致的情况。
常用 Zod 方法作用
z.string().min(n) / .max(n)字符串长度范围
z.string().email()邮箱格式
z.number().min(n)数字范围
z.string().optional()字段可选
z.object({...}).refine(fn, msg)跨字段联合校验(如两次密码一致)
schema.safeParse(data)不抛异常,返回 { success, data/error }
常见误区:用 parse() 而不是 safeParse()。 schema.parse(data) 校验失败时会抛出异常,如果不包 try/catch 会直接让程序崩溃;表单场景几乎总该用 safeParse(),它返回一个包含 success 布尔值的对象,校验失败也不会抛异常,更适合"预期内会失败"的用户输入场景。

小结

  • 把校验规则写成一份独立的 Zod schema,而不是分散在各个字段的 onChange 里手写 if,换来可维护、可复用(前后端共用一份规则)。
  • Zod 的 z.object() + 各种链式方法(.string().email().min() 等)声明式地描述"合法数据长什么样"。
  • z.infer<typeof schema> 能直接从 schema 推导出 TypeScript 类型,类型和校验规则永远保持一致,不用维护两份。
  • 表单场景优先用 safeParse() 而不是 parse(),避免校验失败时抛异常导致程序崩溃。
  • 目前 schema 还是独立于表单库存在的,下一篇用 @hookform/resolvers 把它和 React Hook Form 接起来,变成真正驱动表单校验的那一层。

常见陷阱

  • schema 和手写 interface 各写一遍,两边慢慢漂移。 明明已经有了 loginSchema,却还习惯性地手写一个 interface LoginForm,后面改了 schema 忘了同步改 interface,类型和真实校验规则就悄悄脱节了——这正是 z.infer 想消灭的问题,却因为没用上而重新出现。
  • 该用 z.infer 的地方还是手写类型。 团队里如果没形成"改字段先改 schema"的习惯,很容易在写 props 类型、写函数参数类型时图快手写一遍,结果又制造出第二份"真源",违背了 schema 作为唯一真源的初衷。
  • 混淆 optional() 和"允许为空"。 .optional()只表示这个字段可以不存在(即值为 undefined),但如果表单没填,输入框传上来的往往是空字符串 '' 而不是 undefined——这种情况 optional()并不会放过空字符串,需要额外配合空值转换或 .min(0) 之类的规则,而不是想当然认为"可选"就等于"随便填"。
  • 用 parse() 代替 safeParse()。 表单输入本质上是"预期内会出错"的数据源,parse() 校验失败直接抛异常,一旦忘了包 try/catch 就会让整个组件崩溃;文中已经强调过,表单场景应默认选 safeParse(),只有在你能确定数据一定合法(比如校验自己代码里的常量)时才考虑 parse()。
  • 用 refine() 做跨字段校验时,忘了这条错误信息挂在哪个字段上。 比如"确认密码要等于密码"这种规则写在 z.object({...}).refine(fn, msg) 上时,默认这条错误是挂在整个对象上,而不是自动挂到 confirmPassword 这个具体字段——如果不特别处理,UI 上就不知道该在哪个输入框下面显示这条错误。

面试高频考点

  1. Zod 的 schema 和 TypeScript 的 interface 是什么关系?能互相替代吗?—— 不能简单互相替代,而是"schema 生成 interface"的单向关系。interface 只是编译期的类型标注,编译成 JS 后就完全消失,运行时没有任何东西可以拿来做校验;Zod 的 schema 是一个运行时存在的 JS 对象(z.object({...})),既能在运行时调用 safeParse() 真正执行校验逻辑,又能通过 z.infer<typeof schema> 在编译期"反向"生成对应的 interface。所以更准确的说法是:schema 是唯一真源,interface 是从它派生出来的产物。
  2. 为什么说 TypeScript 的类型检查不能替代 Zod 之类的运行时校验?—— 因为 TS 的类型系统只在编译阶段起作用,编译成 JS 之后所有类型标注都被抹掉,不会留下任何运行时可执行的检查代码。而表单输入、HTTP 请求体这些数据都来自"TS 类型系统管不到"的外部世界(用户可以在输入框里输入任何字符,后端也可能收到不符合约定的请求体),TS 只能保证"你自己代码里写的逻辑是类型安全的",却无法保证"外部传进来的数据真的符合这个类型"。这正是运行时校验(Zod)存在的意义——用实际执行的代码去检查数据,而不是仅凭编译期的假设。
  3. 为什么表单场景更适合 schema-first 而不是每个字段单独写校验规则?—— 因为表单里字段之间经常有依赖关系(比如两次密码要一致),也经常需要和后端共用同一套规则。如果规则散落在各个字段的 onChange 里,字段间依赖表达起来很别扭,规则本身也无法被别处复用。schema-first 把所有规则收进一份独立于 UI 的对象里,规则可以被表单库调用,也可以原封不动地丢给后端接口做请求体校验,同一份 schema 在两端各自 safeParse() 一次,不用维护两份逻辑。
  4. parse() 和 safeParse() 的本质区别是什么,该怎么选?—— 二者背后的校验逻辑完全一样,区别只在"失败时怎么把结果交出来"。parse() 失败时用抛异常的方式传递错误,调用者必须用 try/catch 兜底,适合"校验失败属于程序 bug、理论上不该发生"的场景;safeParse() 失败时用返回值({ success: false, error })传递错误,不会打断程序执行,适合"校验失败是预期内的正常情况"的场景,比如用户表单输入——这也是为什么表单校验几乎总是选 safeParse()。

与其他知识点的联系

  • 与下一篇 143(RHF + Zod resolver 整合)的关系: 本篇写的 loginSchema 目前是一个完全独立的 JS 对象,跟任何表单库都没有绑定关系,只能靠手动调用 safeParse() 才能用起来。下一篇会通过 @hookform/resolvers 里的 zodResolver 把它"翻译"成 React Hook Form 认识的校验接口,schema 才真正从"一份规则说明书"变成"驱动表单实时报错的那一层"。
  • 与 React Hook Form 基础(受控/非受控组件)的关系: Zod 本身根本不关心数据是从受控组件的 value/onChange 收集来的,还是从 RHF 的非受控 register 订阅收集来的——它只认最终拿到的那份 JS 对象长什么样。这也是 schema-first 校验能同时兼容多种表单实现方式的原因:校验规则和"表单怎么收集数据"这两件事被彻底解耦了。
  • 与 TypeScript 类型系统的关系(z.infer): 普通的 interface 只是编译期标注,编译成 JS 后完全消失,不会在运行时拦截任何数据;而 Zod schema 是运行时真实存在的对象,z.infer<typeof schema> 让类型直接从这个运行时定义"反向推导"出来,把类型安全和运行时校验统一到同一份真源上,不用再手写第二份 interface。
  • 与后端接口校验复用的关系: 因为 schema 是纯粹的 JS/TS 对象,不依赖浏览器环境,同一份 loginSchema 可以原封不动地丢进 Node.js 后端,对请求体调用一次 safeParse()。前端校验解决用户体验(即时反馈),后端校验解决安全性(不能信任任何前端传来的数据),两边共用同一份规则,不用维护两套容易漂移的校验逻辑。
  • 与 REST/GraphQL 请求体校验的关系: 144、145 篇会讲到的请求封装和 TanStack Query,最终发出去的请求体、收到的响应体,其形状也完全可以用 Zod schema 描述——校验思路是一致的:数据从"不受信任的外部来源"进入系统的边界处,都值得用 schema 过一遍,而不是走到深层业务逻辑里才发现字段缺失或类型不对。
