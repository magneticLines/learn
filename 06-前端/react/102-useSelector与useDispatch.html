useSelector 与 useDispatch：读状态、发动作

store 建好之后，组件靠两个 hook 和它打交道：useSelector 从 store 读取需要的那部分状态，useDispatch 派发 action 触发状态更新。读写分离，一目了然。

useDispatch：派发 action

useDispatch() 返回 store 的 dispatch 函数。把 slice 里导出的 action creator 调用一下，传进 dispatch 即可：

派发同步与异步 action
import { useDispatch } from 'react-redux';
import { increment, incrementBy } from './counterSlice';
import { fetchUser } from './userSlice';

function Toolbar() {
  const dispatch = useDispatch();
  return (
    <>
      {/* action creator 调用后返回 action 对象，交给 dispatch */}
      <button onClick={() => dispatch(increment())}>+1</button>
      <button onClick={() => dispatch(incrementBy(5))}>+5</button>
      {/* 异步 thunk 也是 dispatch，用法完全一致 */}
      <button onClick={() => dispatch(fetchUser(42))}>加载用户</button>
    </>
  );
}
dispatch 引用在组件生命周期内是稳定的，所以放进 useEffect / useCallback 的依赖数组里是安全的，不会导致重复执行。

useSelector：读取状态 + 订阅重渲

useSelector(selectorFn) 接收一个"选择器"函数，从整个 state 里挑出组件关心的那一小块。它做了两件事：取值 + 订阅——只要 store 一变化，就重新运行选择器，比较结果是否变了，变了才重渲组件。

读取状态
import { useSelector } from 'react-redux';

function CounterValue() {
  // state 是整个 store 树；只取 counter.value 这一个原始值
  const count = useSelector((state) => state.counter.value);
  return <h1>{count}</h1>;
}
重渲机制：默认用 === 严格相等比较。useSelector 拿本次选择器的返回值和上次比。取原始值（number / string / boolean）时天然安全——值相等就相等。危险的是取对象或数组：如果选择器每次返回一个新对象，=== 永远为 false，组件每次 store 变化都会重渲，哪怕内容根本没变。

选择器优化：别返回新对象

❌ 每次返回新对象
// 每次都是新数组/对象字面量
// === 恒为 false，无脑重渲
const { a, b } = useSelector(
  s => ({ a: s.x, b: s.y })
);
✅ 分开选，取原始值
// 各取原始值，=== 稳定
const a = useSelector(s => s.x);
const b = useSelector(s => s.y);

如果确实要一次取多个字段并组合成对象，两种正规解法：

方案一：改用浅比较；方案二：用 reselect 记忆化
// 方案一：告诉 useSelector 用「浅比较」而非 ===
import { shallowEqual } from 'react-redux';
const { a, b } = useSelector(
  (s) => ({ a: s.x, b: s.y }),
  shallowEqual   // 第二参数：比较函数，逐字段浅比较
);

// 方案二：reselect 的 createSelector——输入不变则复用缓存结果
import { createSelector } from '@reduxjs/toolkit';
const selectDoneTodos = createSelector(
  (s) => s.todos,                    // 输入选择器
  (todos) => todos.filter(t => t.done)  // todos 没变就返回上次的过滤结果
);
最高频性能坑：在 useSelector 里写 .filter() / .map() / { ... } 直接返回。这些每次都产出新引用，导致组件在任何 store 更新时都重渲。要么把派生计算搬进 createSelector 记忆化，要么拆成多个只取原始值的 useSelector。

完整数据流

用户点击
→
dispatch(action)
→
reducer 算出新 state
→
useSelector 重新求值
→
值变 → 组件重渲
面试点：①"useSelector 靠什么决定要不要重渲？"——默认对选择器返回值做 === 比较，变了才重渲。②"为什么选择器不能返回新对象？"——引用每次都不同，=== 永远 false，无谓重渲；解法是 shallowEqual 或 createSelector。③useSelector 用了 useSyncExternalStore 订阅外部 store，与 React 并发渲染兼容。
