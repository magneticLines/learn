CSS-in-JS 概览(styled-components / emotion)

组件的逻辑、结构(JSX)都写在同一个 .jsx 文件里,唯独样式非要拆到单独的 .css 文件?CSS-in-JS 的想法是:把样式也搬进 JS,和组件写在一起,天然获得作用域,还能直接用 JS 变量、props 做动态样式。这条路线一度非常流行,但这几年风向变了,这一篇讲清楚为什么。

核心思路:样式和组件耦合在一起

styled-components 是这条路线最具代表性的库,写法上用模板字符串定义一段 CSS,返回一个自带这段样式的组件:

styled-components 基本用法
import styled from 'styled-components';

// styled.button`...` 返回一个新的按钮组件,自带下面这段样式
const Button = styled.button`
  padding: 8px 16px;
  border-radius: 6px;
  background: ${(props) => (props.primary ? '#61dafb' : '#334155')};
`;

function App() {
  // primary 是普通 prop,直接决定了 CSS 里 background 的值
  return <Button primary>确认</Button>;
}

这里的关键能力是:${...} 里可以塞任意 JS 表达式,直接读组件的 props 来决定样式值。CSS Modules 和 Tailwind 想做"props 驱动的动态样式",通常要靠切换 className(比如 primary ? styles.primary : styles.default);CSS-in-JS 则是直接在 CSS 声明里写 JS 逻辑,少了一层"类名和样式值的映射"。

作用域怎么来的? styled-components 在运行时(或构建时,取决于配置)给每个 styled.xxx 生成一个唯一的 className 并注入到页面的 <style> 标签里,原理上和 CSS Modules 的"哈希类名"异曲同工,只是发生的时机和实现方式不同。

运行时开销与 SSR 复杂度的争议

问题出在"怎么把样式变成实际生效的 CSS"这一步。styled-components 早期主要靠运行时实现:组件渲染时才计算出最终样式字符串、动态插入 <style> 标签。这带来两个实打实的成本:

运行时开销
每次渲染都要:
1. 执行模板字符串里的 JS 逻辑
2. 生成/查找对应的 className
3. 必要时插入新的 style 标签
→ 比"直接应用一个静态 className"多了不少计算
SSR 复杂度
服务端渲染时:
1. 要在服务端收集所有用到的样式
2. 手动拼进 HTML 的 <head>
3. 客户端 hydrate 时要能对上,
   否则出现 hydration mismatch
→ 需要额外的框架集成 / 插件
这不是"用不好"的问题,是架构性的取舍。 CSS Modules 和 Tailwind 的样式在构建时就变成了静态 CSS 文件,浏览器直接当普通 CSS 加载,没有运行时计算;CSS-in-JS 早期方案把这部分工作挪到了运行时,换来了"能用 JS 变量写动态样式"的灵活性,但代价是额外的 CPU 开销和 SSR 集成复杂度,在 Next.js 这类 SSR 框架里尤其明显。

为什么社区近几年转向 Tailwind / CSS Modules

结合前两篇的内容看会更清楚这个趋势:

  • 动态样式的需求被弱化了。 Tailwind 的条件 className(primary ? 'bg-sky-500' : 'bg-slate-700')已经能覆盖大部分"根据 props 切样式"的场景,不一定非要在 CSS 声明里嵌 JS 逻辑。
  • 构建时方案更简单可靠。 Tailwind、CSS Modules 都是在构建阶段把样式变成静态 CSS,没有运行时开销,SSR 场景下天然没有"服务端收集样式"这类额外步骤。
  • 部分 CSS-in-JS 库开始转向零运行时。 一些新方案(如 vanilla-extract、部分 emotion 配置)尝试在构建时把样式提取成静态 CSS,吸取 CSS-in-JS 的作用域优点、丢掉运行时开销,某种程度上是向 CSS Modules 的思路靠拢。

面试高频问法:"CSS-in-JS 解决了什么问题,又带来了什么问题?" 标准答案结构是:解决了作用域隔离和props 驱动的动态样式两个问题,代价是(纯运行时方案下)运行时性能开销和 SSR 集成复杂度。能说出这个"解决了 X,代价是 Y"的对仗结构,比死记硬背某个库的 API 更能体现理解深度。
回到样式方案的坐标系: 至此 CSS 方案三篇讲完——CSS Modules 解决作用域、保留手写 CSS 的心智模型;Tailwind 用原子工具类消灭"起名字"的负担;CSS-in-JS 把样式和组件逻辑彻底耦合,换取更强的动态能力,但运行时方案的性能和 SSR 成本让它在近几年的新项目里不再是默认选择。三者没有绝对的对错,选择取决于团队习惯和项目的 SSR 需求。

小结

  • CSS-in-JS(以 styled-components 为代表)把样式定义直接写在 JS 文件里,和组件耦合,天然作用域隔离,还能用 ${props => ...} 直接读 props 做动态样式。
  • 作用域实现原理和 CSS Modules 类似(生成唯一 className),区别在于发生的时机——运行时方案是组件渲染时才计算。
  • 运行时方案的代价是:每次渲染都有额外 JS 计算和 <style> 标签插入开销,SSR 场景下还要额外收集/注入样式,否则容易出现 hydration mismatch。
  • 近几年社区更倾向 Tailwind / CSS Modules,原因是它们在构建时就产出静态 CSS,没有运行时开销,而条件 className 已经能覆盖大多数动态样式场景。
  • 部分新一代 CSS-in-JS 方案(如零运行时方案)在尝试吸收两边的优点,但主流趋势仍是"构建时静态化"。

与其他知识点的联系

  • CSS Modules: 两者的作用域隔离原理相似(都是生成唯一 className),但发生的时机完全不同——CSS Modules 在构建时把类名哈希化,产出的是普通静态 CSS 文件;CSS-in-JS(运行时方案)要等组件渲染时才计算出样式并动态插入 <style> 标签,这个时机差异正是本篇反复强调的运行时开销和 SSR 复杂度的根源。
  • Tailwind CSS 工具类: 本篇提到"Tailwind 的条件 className 已经能覆盖大部分动态样式场景"——CSS-in-JS 靠 ${props => ...} 直接在 CSS 声明里写 JS 表达式,Tailwind 靠三元表达式切换一整组工具类,两者是解决同一类需求("样式随 props 变化")的不同路径,前者更灵活但有运行时代价,后者更受限但零成本。
  • useEffect 与副作用: styled-components 运行时方案里"组件渲染时动态插入 <style> 标签"本质上是一种发生在渲染阶段的副作用操作,这和 useEffect 强调的"副作用应该在渲染之外单独处理"的设计理念形成有趣的对照——只是这类插入操作被库在渲染函数内部隐式完成,普通业务代码里手写类似逻辑通常就应该用 useEffect。
  • Next.js App Router / SSR: 本篇提到 CSS-in-JS 运行时方案在 SSR 场景下需要"服务端收集样式再拼进 HTML,否则出现 hydration mismatch"——这是后面学习 Next.js SSR/SSG 时会再次遇到的概念,hydration 不匹配的坑不止出现在样式上,也是 SSR 框架里的通用陷阱。
  • ESLint 与 Prettier 的分工: 本篇总结"解决了 X,代价是 Y"的对仗式回答方式,同样适用于理解下一组要学的工程化工具——ESLint 解决代码质量问题、代价是需要维护规则集;Prettier 解决格式统一问题、代价是失去逐行控制权。这种"技术选型必然是权衡而非免费午餐"的思考框架,在样式方案和工具链选型上是通用的。
