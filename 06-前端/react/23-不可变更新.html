不可变更新：为什么不能直接修改对象和数组

state 里存的是数字或字符串，更新很直观。但存的是对象或数组时，直接修改内容会让 React 检测不到变化——UI 不更新，这是 React 开发里最高频的 bug 之一。

根本原因：React 用引用比较检测变化

React 判断 state 是否变化，用的是 Object.is(oldState, newState)——本质就是 ===，引用比较，不是深度比较内容。

对于对象和数组，=== 比较的是内存地址，不是内容：

引用比较：地址相同 = React 认为没变化
const obj = { name: 'Shea' };
const before = obj;  // 保存修改前的引用

// 直接修改属性——引用（内存地址）没有变
obj.name = 'Tom';
console.log(before === obj);  // true —— React 认为："没变，不渲染"

// 创建新对象——引用变了
const newObj = { ...obj, name: 'Tom' };
console.log(obj === newObj);  // false —— React 认为："变了，重新渲染"

类比：React 是一个只看"快递单号"的检查员，不拆包看内容。你把同一个包裹的内容换掉，单号没变，检查员说"没有新包裹，不处理"。

黄金法则：永远不用修改方法，要用返回新值的方法

❌ 修改原数据（禁止）
// 数组
list.push(x)
list.pop()
list.splice(0, 1)
list.sort()
list.reverse()
list[0] = newVal

// 对象
obj.name = 'Tom'
delete obj.key
✅ 返回新数据（推荐）
// 数组
[...list, x]
list.slice(0, -1)
list.filter(...)
[...list].sort()
list.map(...)

// 对象
{ ...obj, name: 'Tom' }
Object.fromEntries(
  Object.entries(obj)
    .filter(([k]) => k !== 'key')
)

对象更新：展开运算符

修改对象的某个属性，标准做法是展开旧对象，然后覆盖需要改的字段：

对象 state 的正确更新方式
const [user, setUser] = useState({
  name: 'Shea',
  age: 25,
  city: 'Beijing'
});

// ❌ 错误：直接修改，引用不变，React 不重渲染
function wrongUpdate() {
  user.name = 'Tom';   // 改的是同一个对象
  setUser(user);       // 传进去同一个引用，React：没变化
}

// ✅ 正确：展开旧对象 + 覆盖要改的属性 = 全新对象
function updateName(newName) {
  setUser({
    ...user,          // 先复制所有旧属性
    name: newName     // 然后覆盖 name，其他属性保持不变
  });
}

// 用函数式更新（当新值依赖旧值时更安全）
function updateAge() {
  setUser(prev => ({ ...prev, age: prev.age + 1 }));
}

数组操作：三种常用场景

数组 state 的三种操作——添加、删除、修改
const [list, setList] = useState([
  { id: 1, text: '学习 React',    done: false },
  { id: 2, text: '理解不可变更新', done: false },
]);

// ① 添加：展开旧数组 + 新元素
function addItem(text) {
  setList([
    ...list,                          // 旧数组的所有元素
    { id: Date.now(), text, done: false }  // 追加新元素
  ]);
}

// ② 删除：filter 过滤掉目标项，返回新数组
function removeItem(targetId) {
  setList(list.filter(item => item.id !== targetId));
  // filter 总是返回新数组，引用必然不同
}

// ③ 修改某项：map 遍历，命中的替换，其余原样返回
function toggleDone(targetId) {
  setList(list.map(item =>
    item.id === targetId
      ? { ...item, done: !item.done }  // 命中：返回新对象（翻转 done）
      : item                            // 未命中：原样返回
  ));
  // map 总是返回新数组，React 检测到引用变化，触发重渲
}
最高频 bug：list.push(x); setList(list)
这是 React state 里最常见的错误。push 直接修改了原数组，然后 setList(list) 传入同一个引用，React 检测到"引用没变"，判断没有更新，UI 不重新渲染。

症状：你明明 push 了一条数据，列表没有更新，但 console.log(list) 打印出来数据是对的——这就是这个 bug 的典型现象。

修复：setList([...list, x]) 或 setList(prev => [...prev, x])。

方法速查：禁止 vs 推荐

操作❌ 禁止（修改原值）✅ 推荐（返回新值）
末尾添加 list.push(x) [...list, x]
开头添加 list.unshift(x) [x, ...list]
删除末尾 list.pop() list.slice(0, -1)
按条件删除 list.splice(idx, 1) list.filter(i => i.id !== id)
修改某项 list[idx].x = v list.map((item, i) => i === idx ? newItem : item)
排序 list.sort() [...list].sort()（先复制再排）
修改属性 obj.name = 'x' { ...obj, name: 'x' }
为什么 React 选择引用比较而不是深度比较？
深度比较（递归对比对象所有字段）开销太大——每次渲染都对整个 state 树做深度比较，性能不可接受。引用比较是 O(1) 的，瞬间完成。代价就是你需要在更新时创建新对象——这看起来"浪费"，但 JS 引擎对短生命周期对象的 GC 做了大量优化，实践中性能完全可以接受。如果觉得手写展开太繁琐，可以用 Immer 库，它让你写"看起来像直接修改"的代码，但底层帮你生成新对象。
