Hooks 第二规则：只在函数组件或自定义 Hook 中调用

Hook 不能在普通 JS 函数、事件处理器、class 组件里调用。它只在两个地方合法：React 函数组件和自定义 Hook。理解这一条，要弄清"Hook 的状态到底存在哪里"。

两个合法的调用位置

只有这两处能调 Hook
// ✅ 1. React 函数组件（首字母大写，返回 JSX）
function Counter() {
  const [n, setN] = useState(0);
  return <button onClick={() => setN(n + 1)}>{n}</button>;
}

// ✅ 2. 自定义 Hook（函数名以 use 开头）
function useWindowWidth() {
  const [w, setW] = useState(window.innerWidth);
  // 自定义 Hook 是"能调用其他 Hook 的普通函数"
  return w;
}

四种违规位置

这些地方都不能调 Hook
// ❌ 1. 普通函数（名字不以 use 开头，也不是组件）
function calcTotal(items) {
  const [t] = useState(0);  // 违规
}

// ❌ 2. 事件处理器里
function App() {
  function handleClick() {
    const ref = useRef();  // 违规：点击时才跑，不是渲染期
  }
}

// ❌ 3. class 组件里
class Old extends React.Component {
  render() {
    const [n] = useState(0);  // 违规：class 有自己的 this.state
  }
}

// ❌ 4. useEffect / useMemo 等的回调里
useEffect(() => {
  const [x] = useState(0);  // 违规：回调不是渲染期执行
}, []);

底层原因：Hook 状态挂在组件的 Fiber 上

上一篇讲过，Hook 的状态存在一条链表里。这条链表挂在当前正在渲染的组件对应的 Fiber 节点上。React 内部有个"当前渲染组件"的指针（dispatcher），只有在渲染某个函数组件的过程中，这个指针才指向有效目标。

React 开始渲染 Counter
→
dispatcher 指向 Counter 的 Fiber
→
useState 能找到存储位置

如果你在事件处理器、普通函数或 setTimeout 回调里调 Hook，那时候没有任何组件正在渲染，dispatcher 指向空（或指向上一个渲染的组件），Hook 找不到自己的"归属"，React 会抛出：Invalid hook call. Hooks can only be called inside of the body of a function component。

这条规则和第一条是同源的。第一条（只在顶层）保证"顺序稳定"，第二条（只在组件/自定义 Hook 里）保证"有一个 Fiber 承载状态、且在渲染期执行"。两条一起，才能让"按顺序读写链表"这套机制成立。

命名约定 use 前缀为什么重要

React 和 ESLint 靠函数名判断"这是不是一个 Hook"。只有 use 开头的函数，才被允许内部调用其他 Hook；也只有这样，lint 工具才知道该对它套用两条规则。这是约定驱动的静态分析——名字本身携带了语义。

❌ 普通函数里想用状态
function fetchAndCount() {
  const [n, setN] = useState(0);
  // 违规
}
✅ 改名成自定义 Hook
function useFetchCount() {
  const [n, setN] = useState(0);
  // 合法：use 开头，
  // 且必须被组件/其他 Hook 调用
  return n;
}
Invalid hook call 的三大真实原因（官方文档明确列出）：① 违反了这两条规则；② React 和 React DOM 版本不匹配；③ 项目里装了多份 React 副本（Hook 内部依赖模块级单例，多份 React 会让 dispatcher 对不上）。生产中最隐蔽的是第三种，常见于 npm link 本地调试组件库时。
和 class 组件对比。class 用 this.state / this.setState，状态天然绑在实例 this 上，所以没有"调用顺序"问题。Hook 抛弃了 this，改用"渲染期 + 顺序"来定位状态——代价就是这两条规则。好处是逻辑复用从"高阶组件 / render props 的嵌套地狱"变成了扁平的自定义 Hook 组合。

常见陷阱

  • 在事件处理器里调用 Hook：新手容易觉得"事件处理器也是组件里的函数，应该能调"，但事件处理器是渲染之后、由浏览器事件触发执行的，那时 dispatcher 早已不指向当前组件，Hook 找不到归属的 Fiber。判断标准很简单：只有"组件函数体本身执行的那一刻"才是合法窗口，回调、定时器、Promise.then 里都不算。
  • 把工具函数误认为自定义 Hook，只因为它内部用到了状态：自定义 Hook 的合法性不来自"用了 useState"，而来自命名以 use 开头 + 被组件或另一个 Hook 调用这两个约定同时满足。只把函数改名成 useXxx 却仍在普通函数里调用它（比如在 calcTotal 内部调 useXxx），本质问题没解决，只是骗过了 lint 的命名检测。
  • npm link 本地调试组件库时遇到 "Invalid hook call" 却怀疑自己写错了代码：这类报错很多时候和两条规则无关，而是项目里存在两份 React 副本（比如库自己 node_modules 里的 React 和主项目的 React 不是同一份），dispatcher 单例对不上。排查顺序应该是：先看是否真的违规调用，再查 npm ls react 是否有多份。
  • 在 class 组件里"抄"函数组件的 Hook 写法：老项目迁移时偶尔会有人在 class 的 render 方法里直接写 useState，这是双重错误——class 组件没有 dispatcher 指向的渲染上下文，且它本身有自己的状态机制 this.state，两套状态模型不能混用。

面试高频考点

  1. 为什么 Hook 不能在普通函数或事件处理器里调用？—— Hook 的状态存在当前渲染组件对应的 Fiber 节点上，React 内部通过一个 dispatcher 指针记录"现在正在渲染哪个组件"。事件处理器、setTimeout 回调等都是在渲染完成之后才执行的，那时 dispatcher 不指向任何有效的渲染上下文，Hook 无法定位该把状态存到哪里，因此 React 直接抛出 Invalid hook call。
  2. 自定义 Hook 的 use 前缀只是命名约定，还是有实际约束力？—— 既是约定也有实际后果。React 本身在运行时不会校验函数名，但 ESLint 的 rules-of-hooks 依赖这个前缀来判断"这个函数内部调用其他 Hook 是否合法"；同时约定统一之后，其他开发者和工具（包括 React DevTools）都能一眼识别出这是一个可复用的状态逻辑单元，而不是普通函数。
  3. 遇到 Invalid hook call 报错，排查思路是什么？—— 官方给出三个方向：① 检查是否违反了"只在顶层"或"只在组件/自定义 Hook 里"调用的规则；② 检查 react 和 react-dom 版本是否一致；③ 检查项目里是否存在多份 React（常见于 monorepo 或 npm link 本地组件库场景），因为 Hook 依赖模块级单例的 dispatcher，多份 React 会导致组件用的 dispatcher 和实际渲染用的不是同一个。

与其他知识点的联系

  • Hooks 只在顶层调用（上一篇）：这两条规则同源于同一套底层机制——链表按调用顺序存取状态，且这个链表挂在当前渲染组件的 Fiber 上。顶层规则保证"顺序稳定"，本篇的规则保证"有一个正在渲染的 Fiber 可以挂靠"。少了任何一条，"按顺序读写链表"这套设计都无法成立。
  • useEffect 依赖数组、清理函数：本篇提到"useEffect 的回调里不能再调用 Hook"，因为回调是异步执行、不在渲染期。理解了这一点，再看后面 useEffect 依赖数组和清理函数时会更清楚：effect 回调本身就是一段"渲染之外"执行的代码，这也是它不能直接调用 Hook、却可以安全读写 ref.current 的原因。
  • 自定义 Hook 与 useRef 系列：自定义 Hook 的合法性来自"use 前缀 + 被组件或另一个 Hook 调用"这两个约定，后面学到的 useRef、useReducer 等都遵循同样的规则——它们也只能在函数组件或自定义 Hook 内部调用，背后原因完全一致：都要依赖 dispatcher 指向当前正在渲染的组件。
  • forwardRef 与命令式代码：本篇强调"事件处理器、setTimeout 回调不是渲染期"，这个"渲染期 vs 非渲染期"的区分，后面在 forwardRef、useImperativeHandle 讨论"什么时候可以安全操作 DOM"时会再次出现——原则相通：声明式渲染逻辑和命令式的事件/回调逻辑是两个不同的执行阶段，各自有各自的规则。
