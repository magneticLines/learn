FC 类型与 ReactNode / ReactElement

早期 TS + React 教程几乎都教const Foo: React.FC<Props> = (props) => {...},但社区现在的主流建议是不要用 React.FC。这不是"过时的写法一定要淘汰"的空话,而是它确实带来了几个实际问题。搞懂这些问题,顺便把 ReactNode 和 ReactElement 这对经常混淆的类型也理清楚。

React.FC 曾经的样子

React.FC 写法(不推荐,但要认识)
interface GreetingProps {
  name: string;
}

const Greeting: React.FC<GreetingProps> = ({ name }) => {
  return <p>你好,{name}</p>;
};

React.FC<P> 本质是一个函数类型:接收 P(外加隐式的 children),返回 ReactElement | null。它看起来方便——"这是个函数组件"一目了然——但代价比想象中大。

React.FC 的三个实际问题

问题一:隐式注入 children,类型不诚实

用 FC:children 被偷偷加上
const Greeting: React.FC<GreetingProps>
  = ({ name, children }) => { ... };

// GreetingProps 里根本没定义 children,
// 但下面这样用 TS 不会报错:
<Greeting name="Tom">
  意外能传 children
</Greeting>
不用 FC:props 类型即真相
function Greeting({ name }: GreetingProps) {
  return <p>你好,{name}</p>;
}

// 没在 GreetingProps 里声明 children,
// 传了就会正确报类型错误
<Greeting name="Tom">
  ❌ 类型错误,符合预期
</Greeting>
为什么这是个真问题? 组件明明不接受 children,却因为用了 React.FC 而"看起来能传"——这跟上一篇强调的"props 类型是契约"的原则相悖:一个不该有子元素的组件(比如 <Avatar /> 这种自闭合组件),用 FC 定义反而掩盖了误用,少了一层本该有的类型检查。React 18 的类型定义已经把 FC 里默认自带的 children 去掉了,但历史遗留项目和网上大量旧教程仍在用老写法,这点要知道。

问题二:泛型组件写起来别扭

如果组件本身要用泛型(比如一个通用的 List<T> 组件,渲染任意类型的列表项),React.FC 的语法没法直接配合箭头函数的泛型写法,写法要么很绕、要么直接不支持,而普通函数声明天然支持泛型:

泛型组件:普通函数声明天然支持
interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
}

// 普通函数声明,泛型 T 写在函数名后面,很自然
function List<T>({ items, renderItem }: ListProps<T>) {
  return <ul>{items.map((item, i) => <li key={i}>{renderItem(item)}</li>)}</ul>;
}

// 调用时 T 被自动推断成 string,不用手动传
// <List items={['a', 'b']} renderItem={(s) => s.toUpperCase()} />

问题三:返回值类型收得过窄

React.FC 要求返回值必须是 ReactElement | null,但 React 18 之后函数组件其实也可以合法返回字符串、数字、数组、甚至 undefined(某些场景),用 FC 反而会在这些合法写法上报错,普通函数声明配合"不显式标注返回值类型"更宽松、更贴近实际运行行为。

现在的推荐写法: 直接写普通函数,只标注 props 参数的类型,不显式标注返回值类型(交给 TS 推断),需要泛型直接享受函数天然支持泛型的便利。 
function Greeting({ name }: GreetingProps) {
  return <p>你好,{name}</p>;
}

ReactNode 与 ReactElement:超集与子集的关系

这两个类型经常被混用,但范围完全不同——ReactElement 是 ReactNode 的一个子集。

类型含义典型使用场景
ReactElement调用 <Foo /> 或 React.createElement() 产生的单个具体对象,有明确的 type/props 结构函数组件显式标注返回值类型(可选);高阶组件里"我确定收到的一定是个元素"
ReactNode"任何能被 React 渲染出来的东西"的超集,包含 ReactElement、字符串、数字、布尔、null、undefined、以及它们的数组children 属性的类型(上一篇讲过);任何"渲染什么都行"的 props
为什么 children 用 ReactNode 而不是 ReactElement
interface WrongProps {
  children: React.ReactElement;  // 过窄:只接受"单个元素"
}
// <Wrong>纯文本</Wrong>        ❌ 报错,字符串不是 ReactElement
// <Wrong>{null}</Wrong>        ❌ 报错

interface RightProps {
  children: React.ReactNode;     // 正确:覆盖所有合法的 JSX 子内容
}
// <Right>纯文本</Right>         ✅
// <Right>{condition && <Icon />}</Right>  ✅(条件为假时是 false,也在 ReactNode 内)
面试高频问法: "ReactNode 和 ReactElement 的区别是什么、什么时候用哪个"——记住一句话即可:ReactElement 是"一个具体的元素对象",ReactNode 是"能塞进 JSX 里的任何值"。凡是不确定内容形态、只要求"可渲染"的场景(children、render props 的返回值)用 ReactNode;需要精确操作元素结构(比如 cloneElement 克隆并修改 props)才用 ReactElement。

小结

  • React.FC 会隐式给 props 加上未声明的 children、不方便写泛型组件、返回值类型收得过窄,社区现在更推荐直接用普通函数 + 显式标注 props 类型。
  • 推荐写法:function Foo(props: FooProps) { ... },不必显式标注返回值类型,交给 TS 推断。
  • ReactElement 是"调用组件/createElement 产生的单个具体对象",范围窄;ReactElement 只是 ReactNode 的一部分。
  • ReactNode 是"任何能被 React 渲染"的超集,涵盖字符串、数字、布尔、null、数组、ReactElement,children 属性应该用它。
  • 需要精确操作元素结构(如 cloneElement)才用 ReactElement;只是"渲染点什么"用 ReactNode。
