缓存与失效策略：staleTime、gcTime 与 invalidate

TanStack Query 的核心价值就是缓存。理解它的缓存生命周期——什么时候用缓存、什么时候后台刷新、什么时候丢弃——才能既省请求又不给用户看陈旧数据。

queryKey：缓存的主键

每份缓存都由 queryKey 唯一标识。key 相同 = 同一份缓存；key 不同 = 各自独立。key 是数组，可以嵌套对象，TanStack Query 会做确定性序列化（字段顺序不影响相等判断）。

queryKey 的层级设计
['todos']                          // 全部 todos
['todos', { status: 'done' }]      // 已完成的（不同筛选 = 不同缓存）
['todo', id]                      // 单条详情

// 失效时可按前缀「模糊」匹配：下面会让上面前两条一起失效
queryClient.invalidateQueries({ queryKey: ['todos'] });

staleTime vs gcTime：两条不同的时间线

这是最容易混淆的两个概念。它们管的是缓存生命的不同阶段：

staleTime（新鲜期）
数据「多久内算新鲜」
默认 0（立即变陈旧）

新鲜期内：完全用缓存，
   不发请求
过了：数据仍显示，但
   触发时机会后台重取
gcTime（垃圾回收）
无组件使用后，缓存
「多久后被删除」
默认 5 分钟

期间再用到 → 秒出旧数据
到期删除 → 下次重新 loading
（旧名 cacheTime）
用图书馆类比：staleTime 是"这本书多久内不用去核对最新版"；gcTime 是"没人借的书在书架上留多久才下架"。一份数据可以已经陈旧（stale）但仍在缓存里——它会被展示给你（避免白屏），同时后台去取新版。
按数据特性设置 staleTime
useQuery({
  queryKey: ['config'],
  queryFn: fetchConfig,
  // 配置类数据几乎不变：设长一点，减少无谓请求
  staleTime: 1000 * 60 * 5,   // 5 分钟内不重取
  gcTime: 1000 * 60 * 30,     // 无人使用后 30 分钟才清理
});

触发后台重取的时机

当数据已 stale（过了 staleTime），以下事件会触发自动后台重取（可逐项关闭）：

选项默认含义
refetchOnMounttrue组件重新挂载时（数据 stale 才会真发）
refetchOnWindowFocustrue浏览器标签重新聚焦时——切回页面看到最新
refetchOnReconnecttrue断网重连后
refetchIntervalfalse轮询：设成毫秒数则定时重取
面试高频误区："staleTime=0 是不是每次都重新请求、缓存没用？"——不是。staleTime=0 表示数据立即变陈旧，但只有在上面那些"触发时机"发生时才后台重取；两次触发之间，缓存照样直接返回旧数据（不闪 loading）。所以缓存一直在起作用，staleTime 只控制"何时认为需要刷新"。

主动失效：invalidateQueries

写操作后要立刻让相关缓存失效。invalidateQueries 做两件事：把匹配的缓存标记为 stale，并对当前正在使用的 query 立即重取。

数据被 mutation 改动
→
invalidateQueries({ queryKey })
→
匹配缓存标记 stale
→
活跃 query 立即重取
invalidate 的几种粒度
// 精确失效一条
qc.invalidateQueries({ queryKey: ['todo', id] });

// 前缀失效：所有以 ['todos'] 开头的缓存（含各种筛选）
qc.invalidateQueries({ queryKey: ['todos'] });

// 手动 refetch：不改 stale 状态，直接重取（较少用）
const { refetch } = useQuery(...);
refetch();

invalidate vs setQueryData

invalidateQueries
标记失效 → 重新请求
拿到服务器权威数据
代价：一次网络往返
setQueryData
直接把新数据写进缓存
无需请求，立即更新
用于乐观更新（下一章）
面试点：①"staleTime 和 gcTime 区别？"——staleTime 控制"多久算新鲜、何时后台刷新"，gcTime 控制"无人用的缓存多久被删"。②"staleTime=0 缓存还有用吗？"——有，两次触发之间仍返回缓存，只是数据被视为可刷新。③"怎么让列表在新增后更新？"——mutation 的 onSuccess 里 invalidateQueries 对应 key。
