第三方库类型 @types/*

npm install lodash 之后在 TS 项目里 import _ from 'lodash',编辑器却提示"找不到模块的类型声明"——这是因为 lodash 本身是纯 JS 写的,不带类型信息。TS 生态用一套叫 DefinitelyTyped 的社区仓库,把成千上万个 JS 库的类型补全,你只需要 npm install -D @types/lodash。这一篇讲清这套机制怎么运作、什么时候要装、什么时候要自己写。

为什么会有"库本身没类型"这回事

TS 是 JS 的超集,但绝大多数 npm 包诞生时就是纯 JS,早于 TS 普及,或者作者压根没打算支持 TS。TS 编译器要检查你调用这些库的代码对不对,就必须知道这些库导出的函数/对象长什么样——这些"形状描述"就是类型声明文件(.d.ts)。

TS 项目里能获得类型提示和检查

类型声明的来源分三种情况,决定了你要不要额外装东西:

情况说明你需要做什么
库自带类型作者用 TS 写的库,或手写了 .d.ts 一起发布,package.json 里有 "types" 字段什么都不用做,npm install 后直接有类型提示(如 axios、zustand)
社区维护类型库本身没类型,但 DefinitelyTyped 社区补了一份npm install -D @types/库名
完全没有类型库太小众,没人补;或私有内部库自己写 declare module 声明,或退而求其次标为 any

DefinitelyTyped 与 @types/* 的关系

DefinitelyTyped 是 GitHub 上一个巨大的社区仓库,专门收录"给没有自带类型的 JS 库写的类型声明"。每个库对应仓库里一个文件夹,发布到 npm 时统一用 @types/ 前缀命名——这也是为什么装 lodash 的类型要装的是 @types/lodash 而不是别的名字。

典型安装场景
# 库本身,项目运行时依赖
npm install lodash

# 类型声明,只在开发/编译时用,所以是 -D(devDependencies)
npm install -D @types/lodash
为什么类型包要装到 devDependencies? .d.ts 文件只在编译期被 TS 读取用来做类型检查,不会被打进最终的 JS 产物、也不会在浏览器/Node 运行时被加载。所以它跟 webpack、eslint 一样属于"开发工具链的一部分",装到 -D 是约定俗成的规范。

怎么判断一个库要不要装 @types

不用死记硬背,两步就能确认:

  1. 先直接 import 试试——如果编辑器不报"找不到类型声明"的错,说明库自带类型(或者是纯类型安全的,比如只导出字符串常量),不用装。
  2. 报错了,去 npm 搜 @types/库名 看有没有对应的包;大部分主流库都有(React 本身的类型就是 @types/react,而不是 React 自带的)。
一个反直觉的例子:React 自己的类型也是 @types。 很多人以为 React.FC、React.ChangeEvent 这些类型是 React 库自带的,其实早期(现在部分仍然)是靠 @types/react 单独维护提供的——这也是为什么升级 React 大版本时,有时候光升级 react 不够,还得同步升级 @types/react 到匹配的版本,否则新 API 用不了或者类型对不上。

declare module:自己动手写类型声明

如果一个库既没自带类型、DefinitelyTyped 上也没人维护(常见于小众库、公司内部私有包、或者只导入个 .svg/.png 这种非 JS 资源),TS 会直接报错"找不到模块"。这时候可以自己写一份最简类型声明,让 TS "闭嘴"或提供基础提示。

src/types/some-lib.d.ts:手写声明文件
// 最简单粗暴:告诉 TS 这个模块存在,但内容类型是 any
declare module 'some-untyped-lib';

// 更负责任的写法:补上实际用到的函数签名
declare module 'some-untyped-lib' {
  export function doSomething(input: string): number;
}
常见用途:让 TS 认识非 JS 资源导入(如 SVG)
// Vite/CRA 项目常见,放在 src/vite-env.d.ts 或 src/types 下
declare module '*.svg' {
  const src: string;
  export default src;
}
// 有了这个声明,import logo from './logo.svg' 才不会报"找不到模块"
陷阱:类型包版本与库版本不匹配。 @types/* 是社区独立维护、独立发版的,版本号不一定和库本身同步。常见坑:升级了 react-router-dom 到某个大版本,但 @types/react-router-dom 还是旧版本(甚至新版本库自带类型后,旧的 @types 包已废弃却还留在 package.json 里),导致类型报错跟实际 API 对不上、或者出现"类型说这个方法不存在,但文档说存在"的诡异现象。排查这类问题第一步:检查 @types/* 包版本是否跟得上库本身版本,库若已自带类型就该把对应的 @types 包卸载掉。

和本系列前面知识点的联系

这一篇是"类型系统"这条主线里相对独立但很实用的一环——前面几篇讲的都是怎么给自己写的组件/状态标类型,而 @types/* 解决的是怎么让类型检查覆盖到别人写的代码。项目里但凡引入新的第三方库,第一反应应该是"这库类型全不全",而不是等 TS 报错才后知后觉。

小结

  • 大多数纯 JS 编写的 npm 包本身没有类型声明,TS 需要额外的 .d.ts 文件才能做类型检查。
  • DefinitelyTyped 是社区维护的类型声明仓库,发布时统一用 @types/库名 命名,装到 devDependencies。
  • 判断要不要装:先直接 import,编辑器报"找不到类型声明"再去搜对应的 @types 包。
  • 连 @types 都没有的库(或非 JS 资源如 SVG),用 declare module 自己写最简声明。
  • 常见陷阱是 @types/* 版本和库本身版本不同步,升级库之后类型报错要先检查类型包版本,库自带类型后要记得卸载多余的 @types 包。
