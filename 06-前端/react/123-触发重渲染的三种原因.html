触发重渲染的三种原因

做性能优化之前,得先知道什么会触发重渲染——否则 memo、useMemo、useCallback 用起来就是"抄药方",不知道到底在防什么。重渲染只有三个触发源:自身 state 变化、父组件重渲染、Context 值变化。这一篇把它们讲透,尤其是最反直觉的第二种。

原因一:自身 state 变化

这是最好理解的一种。组件调用 useState / useReducer 得到的 setter 被调用,并且新值和旧值不同(用 Object.is 比较),React 就会安排这个组件重新执行函数体。

自身 state 变化触发重渲染
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(c => c + 1)}>
      {count}
    </button>
  );
}
// count 从 0 变成 1,组件函数体重新执行一遍

原因二:父组件重渲染(即使 props 没变)

这是初学者最容易踩的认知误区:父组件重渲染时,默认情况下所有子组件都会跟着重渲染——不管子组件收到的 props 有没有变化。React 的默认行为是"不做比较、全部重新调用一遍函数",这是一个刻意的设计取舍。

父重渲染,子 props 没变但依然重渲染
function Child({ name }) {
  console.log('Child 渲染了');  // 每次都会打印,即便 name 一直是 "Tom"
  return <div>{name}</div>;
}

function Parent() {
  const [count, setCount] = useState(0);
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {/* name 永远是 "Tom",但每次点按钮 Child 都会重渲染 */}
      <Child name="Tom" />
    </>
  );
}
为什么 React 要这样设计? 因为判断"props 是否真的变了"本身有成本(对象/数组做浅比较也要花时间),React 团队的默认取舍是:渲染函数本身应该便宜、纯粹,与其默认做比较拖慢每一次渲染,不如默认"全部重新调用",把"要不要跳过"的决定权交给开发者——这正是后面 React.memo 要解决的问题。也就是说,子组件重渲染不代表 DOM 会更新:重渲染只是重新执行函数、生成新的虚拟 DOM,React 还会做 diff,真正的 DOM 操作只发生在 diff 出差异的地方。重渲染有成本(函数执行、diff),但通常远小于真实 DOM 操作的成本。

原因三:Context 值变化

用了 useContext 订阅某个 Context 的组件,一旦该 Context 的 value 变化,所有消费者都会重渲染——不管这个组件在树里离 Provider 有多远,也不管中间隔了多少层没有订阅 Context 的组件(中间层不会重渲染,只有真正消费的组件会)。

Context 值变化,所有消费者一起重渲染
const ThemeContext = createContext('light');

function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={theme}>
      {/* theme 一变,树里所有 useContext(ThemeContext) 的组件都重渲染 */}
      <Toolbar />
    </ThemeContext.Provider>
  );
}
面试高频考点: "Context 变化会导致整棵树重渲染吗?"——不准确。只有真正调用了 useContext 消费该 Context 的组件才会重渲染,没订阅的子树不受影响。但常见的坑是:很多人把大量不相关的状态塞进同一个 Context 的 value 对象,导致任何一个字段变化,所有消费者(哪怕只用到其中一个字段)全部重渲染——这也是后面要讲的"拆分 Context"要解决的问题。

三种原因的对比

触发源影响范围能否绕过
自身 state 变化只有该组件(及其子树,受原因二影响)合并 state、用 useReducer 减少更新次数
父组件重渲染默认所有子组件,不管 props 是否变化React.memo 做浅比较后跳过
Context 值变化所有 useContext 该 Context 的消费者拆分 Context、用状态管理库替代
为什么先讲这一篇? 后面三篇要学的 React.memo、useMemo、useCallback 分别对应解决原因二(阻断父传子的无效重渲染)和它的配套问题(props 里的对象/函数每次都是新引用,导致 memo 失效)。不先搞清楚"为什么会重渲染",学这些 API 只会变成死记硬背"什么时候该加"。

小结

  • 重渲染只有三个触发源:自身 state 变化、父组件重渲染、Context 值变化。
  • 父组件重渲染时,子组件默认全部跟着重渲染,不做 props 是否变化的比较——这是 React 为了保持渲染函数简单而做的默认取舍。
  • 重渲染≠DOM 更新:重渲染只是重新执行函数生成新虚拟 DOM,真正的 DOM 操作由 diff 之后决定。
  • Context 变化只会影响真正订阅它的消费者,不是整棵树;但把无关状态塞进同一个 Context 会放大这个影响范围。
