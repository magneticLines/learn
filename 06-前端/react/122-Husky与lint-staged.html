Husky + lint-staged

ESLint 和 Prettier 配好了,但如果没人强制要求,总会有人图快跳过检查直接 git commit,不规范的代码就这样溜进了仓库,等 CI 跑起来才报错——这时候已经浪费了一次 CI 排队时间,甚至可能是别人 pull 代码之后才发现。Husky + lint-staged 要解决的问题就是:让检查在代码离开本地之前就强制发生,而不是靠自觉。

为什么要在 commit 前就跑 lint,而不是等 CI

把检查往前挪一步,本质上是上一篇"发现问题的时机越晚、成本越高"这条原则的延续:

写代码
→
git commit(拦截点)
→
git push
→
CI 跑 lint
→
失败,推回重来

如果只靠 CI 兜底,流程是"写完 → 提交 → 推送 → 排队等 CI → CI 报错 → 回头改 → 再推一次"——中间有网络延迟、CI 排队时间,等结果通常要几分钟到十几分钟。如果在 git commit 这一步就拦截,问题在几秒钟内就能反馈给开发者,不用等 CI,也不会把不规范的代码推上远程仓库让同事 pull 到本地。

不是替代 CI,是加一道更早的关卡。 commit 前的检查和 CI 里的检查通常是同一套规则,但 CI 依然要保留——因为本地钩子可以被 --no-verify 跳过,CI 是不能被绕开的最后一道防线。两者是"尽量早发现"和"确保一定会被发现"的配合关系,不是二选一。

Husky:让 git hooks 变得可配置、可共享

Git 本身早就自带 git hooks 机制——在 .git/hooks/ 目录下放一个叫 pre-commit 的可执行脚本,每次 git commit 时 Git 会自动先跑这个脚本,脚本失败(非零退出码)就会中断提交。这个机制原生就有,但有个致命问题:.git 目录不会被提交到仓库里,你写的 hook 脚本只在你自己电脑上生效,没法分享给团队。

Husky 解决的正是这个"钩子没法共享"的问题:它把 hook 脚本放进项目里一个会被提交的目录(如 .husky/),再通过 npm install 时的钩子自动把它们链接到 Git 的钩子机制上。这样团队里任何人 clone 项目、装完依赖,pre-commit 检查就自动生效,不需要每个人手动配置。

.husky/pre-commit
npx lint-staged
理解 Husky 的本质: 它不是发明了新机制,而是把 Git 原生的 hooks 功能工程化——变成"配置文件形式、可提交进仓库、装依赖时自动生效"。这也是不少工具库的共同套路:不重新造轮子,而是把平台原生但难用/难共享的能力包一层,变得对团队友好。

lint-staged:只检查暂存区的改动文件

有了 pre-commit 钩子,下一个问题是:钩子里该跑什么? 最直接的想法是对整个项目跑一遍 eslint .,但这在大项目里是灾难——几千个文件全量检查,每次 commit 都要等半分钟以上,而你这次提交可能只改了一个文件。

lint-staged 解决的就是这个效率问题:它只拿 git 暂存区(staged)里、本次 commit 真正会提交的那些文件去跑检查,而不是全仓库扫描。改了 1 个文件就只检查这 1 个文件,速度和改动量成正比,不会因为项目变大而拖慢每一次提交。

package.json 里的 lint-staged 配置
{
  "lint-staged": {
    "*.{js,jsx}": [
      "eslint --fix",    // 先自动修复能修的问题
      "prettier --write" // 再统一格式化
    ],
    "*.{css,md}": [
      "prettier --write"
    ]
  }
}

执行完这些命令后,lint-staged 还会自动把修复后的文件重新加入暂存区,保证最终提交的是修复过的版本,而不是修复前的旧内容。

陷阱:eslint --fix 不代表"检查通过"。 --fix 只能自动修复格式类、少量简单的规则违反(比如缺少分号),但像 react-hooks/exhaustive-deps 这种需要人判断的逻辑问题,--fix 是无法自动修的——命令仍会以非零状态码退出,阻止 commit,这是符合预期的行为。不要误以为配了 --fix 就能让所有检查自动通过,该手动修的问题还是需要人介入。

整体流程串起来

1. git commit
触发 .husky/pre-commit 脚本
2. npx lint-staged
读取 package.json 的配置
3. 只处理暂存区文件
跑 eslint --fix / prettier --write
4. 全部通过
重新暂存修复结果,commit 成功;否则中断
回到代码质量三篇的坐标系: ESLint 负责发现代码质量问题,Prettier 负责统一格式,Husky + lint-staged 把这两者强制接入 git commit 流程,让"写完代码就检查"从依赖个人自觉变成机制上不可跳过(至少不能无意跳过)。三者配合 CI 里的同一套规则,构成了"本地秒级反馈 + 远程兜底保证"的完整质量防线。

小结

  • 在 git commit 前跑 lint,能把问题反馈时间从"CI 排队几分钟"缩短到"几秒钟",还能避免不规范代码被推上远程仓库影响他人。
  • Git 原生就有 hooks 机制,但 .git/hooks/ 不会被提交,没法团队共享;Husky 把钩子脚本放进可提交的 .husky/ 目录,装依赖时自动生效。
  • lint-staged 只对 git 暂存区里本次要提交的文件跑检查,而不是全仓库扫描,速度只和改动量相关,不会随项目变大而变慢。
  • eslint --fix / prettier --write 能自动修复格式类问题,但逻辑类问题仍需人工修复,命令失败时 commit 会被正常中断。
  • 本地钩子可以被 --no-verify 跳过,所以它不能替代 CI,而是作为 CI 之前更早的一道反馈更快的关卡,两者配合使用。
