React Testing Library 组件测试

Vitest 解决了"怎么跑测试",但组件测试还有个更本质的问题:测什么?是断言组件内部某个 state 变成了什么值,还是断言用户点击按钮后屏幕上真的出现了新文字?React Testing Library(RTL)的答案很鲜明:只测用户能看到、能操作的东西。

核心哲学:像用户一样测试你的组件

RTL 的口号是 "The more your tests resemble the way your software is used, the more confidence they can give you"(测试越像真实使用方式,给你的信心就越大)。落到 API 设计上,它刻意不提供访问组件内部 state、直接调用组件方法的能力——你拿到的只是渲染后的真实 DOM,查询方式也优先是用户能感知的文本、角色(role)、标签,而不是内部实现细节(如 class 名、组件变量名)。

类比: 测试一个"登录表单"组件,QA 工程师不会打开 React DevTools 看 isLoading 这个 state 是不是 true——他们只会看屏幕上是不是出现了"登录中..."的文字、按钮是不是变灰不能点。RTL 强迫你的测试代码也这样做。

基本用法:render + screen + userEvent

三个核心 API:render 把组件挂载到一个虚拟 DOM(jsdom)里;screen 提供一系列 getBy* 查询方法,在渲染结果里找元素;userEvent 模拟真实的用户操作(点击、输入),比直接触发 DOM 事件更贴近真实浏览器行为。

Counter.tsx —— 被测试组件
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>当前计数:{count}</p>
      <button onClick={() => setCount(c => c + 1)}>加一</button>
    </div>
  );
}
Counter.test.tsx —— 像用户一样点击并断言可见文本
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect } from 'vitest';
import { Counter } from './Counter';

describe('Counter', () => {
  it('点击加一按钮后,计数文本更新', async () => {
    const user = userEvent.setup();
    render(<Counter />);

    // 用"角色 + 可访问名"找按钮,而不是 class 或 id
    const button = screen.getByRole('button', { name: '加一' });
    await user.click(button);

    // 断言屏幕上真实可见的文本,而不是组件内部 count 变量
    expect(screen.getByText('当前计数:1')).toBeInTheDocument();
  });
});
为什么用 getByRole 而不是 getByTestId? role(按钮、链接、输入框等)是浏览器和屏幕阅读器都能识别的语义,用它查询顺带验证了你的 HTML 语义是否正确、是否具备基本的无障碍访问能力。getByTestId 需要在源码里手写 data-testid,属于"专为测试开的后门",官方文档把它列为优先级最低的查询方式,只在实在没有语义化选择时才用。

对比:测实现细节 vs 测用户可见行为

测实现细节(不推荐)
// 用 Enzyme 风格直接读内部 state
const wrapper = shallow(<Counter />);
wrapper.find('button').simulate('click');
expect(wrapper.state('count')).toBe(1);

问题:重构组件内部实现
(比如把 useState 换成 useReducer)
测试就跟着挂掉,即使用户看到的
结果完全没变。
测用户可见行为(推荐)
const user = userEvent.setup();
render(<Counter />);
await user.click(screen.getByRole('button', { name: '加一' }));
expect(screen.getByText('当前计数:1')).toBeInTheDocument();

无论内部用 useState 还是
useReducer 实现,只要用户看到
的行为不变,测试就一直通过。
面试高频问题:"为什么 RTL 不推荐测 state?" 答案的核心是测试的价值在于给"重构"托底。如果测试和实现细节绑死,每次内部重构(哪怕外部行为完全没变)都要跟着改测试,测试反而成了重构的阻力,大家最终会选择不写测试。只锚定"用户能感知的行为",测试才能长期稳定地起到"看门狗"的作用。

常用查询方法优先级

方法说明
getByRole最推荐,按钮/输入框/标题等语义角色,兼顾无障碍
getByLabelText表单控件,按关联的 <label> 文本查找
getByText按可见文本内容查找,适合非交互元素
getByTestId兜底方案,需手写 data-testid,优先级最低
find 与 query 前缀。 getBy* 找不到会立刻抛错,适合断言"这个元素现在必须在";queryBy* 找不到返回 null,适合断言"这个元素不应该存在";findBy* 是异步版本,内部自带等待重试,适合测"点击后异步出现"的元素(比如接口返回后才渲染的内容)。

小结

  • RTL 的核心哲学是"像用户一样测试组件":只通过屏幕上可见的文本/角色查询元素、只模拟真实的用户交互,不触碰组件内部实现。
  • render 挂载组件,screen.getByRole/getByText 查询元素,userEvent 模拟点击输入等交互。
  • 查询方法有优先级:getByRole 最优先(兼顾无障碍语义),getByTestId 是兜底方案。
  • "测实现细节"(如直接读 state)会让测试和内部重构强耦合,重构就挂;"测用户可见行为"能让测试在重构后依然稳定,这才是测试该起的作用。
  • getBy* 找不到立刻报错,queryBy* 找不到返回 null,findBy* 是带等待的异步版本,分别对应不同的断言场景。
