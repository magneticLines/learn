虚拟滚动 TanStack Virtual

上一组讲的 memo/useMemo/状态下移,解决的是"重渲染要不要发生"的问题;而虚拟滚动解决的是更前置的问题——DOM 节点数量本身。哪怕一个列表项完全不会重渲染,几千个真实 DOM 节点常驻页面本身就是负担。

问题:全量渲染长列表的代价

假设要渲染一个 5000 条数据的列表,最直接的写法是 items.map(item => <Row key={item.id} {...item} />)——这会一次性创建 5000 个真实 DOM 节点。代价体现在三个地方:

  • 首屏渲染慢:浏览器要创建、布局(layout)、绘制(paint)这几千个节点,即便用户当下只能看到其中十几条。
  • 内存占用高:每个 DOM 节点在浏览器内部都有对应的渲染树结构,节点越多占用越大。
  • 滚动卡顿:节点太多会拖慢浏览器的重排(reflow)和事件派发,滚动时容易掉帧。
关键认知: 这不是 React 的问题,是浏览器 DOM 本身的物理限制——不管用什么框架,一次性塞几千个真实 DOM 节点都会慢。memo 只能保证"没变的行不重新渲染",但没变的行依然是已经存在于 DOM 里的真实节点,不会减少节点总数。

核心原理:只渲染可视区域 + 缓冲区

虚拟滚动(virtual scrolling / windowing)的思路是:用户任意时刻只能看到视口(viewport)里的一小部分内容,那就只为这一小部分创建真实 DOM,列表其余部分完全不渲染。具体做法:

  1. 用一个高度等于"完整列表总高度"的占位容器撑开滚动条,让浏览器认为这里有一个完整的长列表可以滚动。
  2. 只在这个占位容器内部,渲染当前可视区域及其上下少量缓冲区(overscan)内的条目,给它们设置正确的 transform 或 top 定位,让它们看起来"排列"在正确的位置上。
  3. 监听滚动事件,根据新的滚动位置重新计算"应该渲染哪些条目",动态替换实际渲染的那一小批 DOM 节点。
占位容器(总高度 = 5000 × 行高)
条目 0~1200(不渲染)
|
条目 1201~1220(真实渲染,可视+缓冲)
|
条目 1221~5000(不渲染)
为什么缓冲区(overscan)很重要: 如果只精确渲染"刚好在视口内"的条目,快速滚动时新条目还没渲染出来就已经进入视口,会出现短暂的白屏/空白。多渲染视口上下各几行作为缓冲,滚动时提前把内容准备好,视觉上更平滑。这是"渲染范围"和"流畅度"之间的取舍——缓冲区越大越流畅,但也意味着更多的 DOM 节点。

TanStack Virtual 基本用法

手写虚拟滚动要处理滚动监听、位置计算、边界情况(变高行、动态尺寸测量)等大量细节,通常直接用现成的库。@tanstack/react-virtual 的核心 API 是 useVirtualizer:

useVirtualizer 基本用法
import { useVirtualizer } from '@tanstack/react-virtual';

function BigList({ items }) {  // items 有 5000 条
  const parentRef = useRef(null);  // 有固定高度、可滚动的容器

  const virtualizer = useVirtualizer({
    count: items.length,            // 列表总条数(用来计算占位容器总高度)
    getScrollElement: () => parentRef.current,  // 谁负责滚动
    estimateSize: () => 48,          // 每行预估高度(px),用来算总高度和初始位置
    overscan: 5,                 // 视口外多渲染 5 行作为缓冲
  });

  return (
    <div ref={parentRef} style={{ height: 600, overflow: 'auto' }}>
      {/* 占位容器:高度 = 总条数 × 预估行高,撑出正确的滚动条长度 */}
      <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
        {virtualizer.getVirtualItems().map((virtualRow) => (
          <div
            key={virtualRow.key}
            style={{
              position: 'absolute', top: 0, left: 0, width: '100%',
              height: virtualRow.size,
              transform: `translateY(${virtualRow.start}px)`, // 用 transform 定位到正确位置
            }}
          >
            {items[virtualRow.index].text}
          </div>
        ))}
      </div>
    </div>
  );
}

关键点:getTotalSize() 撑出占位容器的完整高度,让滚动条表现得像列表真的有 5000 条;getVirtualItems() 只返回当前应该渲染的那一小批条目(通常十几到几十个),每一项带有 start(应该定位到的位置)和 size(高度)。

为什么用 transform: translateY 而不是 top: transform 的位移不会触发浏览器的重排(reflow),只走合成层(compositing),性能开销更低;而修改 top 会导致浏览器重新计算布局。这是虚拟滚动库的通用实现细节,也是 CSS 动画性能优化的常见考点。

什么时候需要虚拟滚动

场景是否需要
列表条数几十条以内,固定展示不需要,直接渲染即可,虚拟滚动反而引入复杂度
聊天记录、日志流、大表格,条数可能上千甚至上万需要——DOM 节点数量本身就是瓶颈
每行内容复杂、渲染开销大,且条数中等可以先试 memo,条数继续增长再上虚拟滚动
需要 Ctrl+F 浏览器原生查找列表内容要谨慎——虚拟滚动下不可见内容不在 DOM 里,原生查找找不到,需要额外处理搜索
常见陷阱: 虚拟滚动要求容器有确定的高度才能计算总高度和可视范围,如果列表所在容器高度是"自适应内容撑开"的,会和虚拟滚动的假设冲突,需要显式给容器一个固定或可计算的高度。另外,行高不固定(内容长度不一)时要用 estimateSize 给出预估值,并结合库提供的动态测量能力(如 measureElement)做校正,否则会出现滚动跳动。

小结

  • 长列表全量渲染会创建大量真实 DOM 节点,导致首屏慢、内存高、滚动卡顿——这是浏览器 DOM 的物理限制,不是 React 本身的问题,memo 也解决不了节点数量本身。
  • 虚拟滚动的核心原理:用一个撑出完整高度的占位容器模拟"列表滚动条",只在可视区域及缓冲区(overscan)内渲染少量真实条目,滚动时动态替换渲染内容。
  • 缓冲区大小是"流畅度"和"渲染节点数"之间的权衡,过小滚动会白屏,过大失去虚拟滚动的意义。
  • @tanstack/react-virtual 的 useVirtualizer 提供 getTotalSize()(占位总高度)和 getVirtualItems()(当前应渲染的条目),配合 transform: translateY 定位(避免触发重排)。
  • 不是所有列表都需要虚拟滚动:几十条的固定列表直接渲染即可;条数上千、且渲染开销明显的场景才值得引入,同时要注意固定容器高度、行高预估、浏览器原生查找失效等实际约束。
  • 下一篇对比"分页"和"无限滚动"两种应对大数据量列表的产品/技术方案,和虚拟滚动是互补关系——分页/无限滚动控制"一次拉多少数据",虚拟滚动控制"拉到手的数据里一次渲染多少"。

与其他知识点的联系

  • React.memo：文中已经明确指出两者的分工——memo 只能保证"没变的行不重新渲染"，但没变的行依然是已经存在于 DOM 里的真实节点，不会减少节点总数。虚拟滚动解决的是节点数量本身的问题，是比 memo 更前置的一层优化，两者可以叠加：虚拟滚动先把实际渲染的行数压到十几到几十行，剩下的这些行再用 memo 避免无关的重渲染。
  • useMemo：130 篇的 useMemo 缓存的是"计算结果"（比如对大数组排序），本篇的虚拟滚动缓存/复用的是"DOM 节点"——两者都是应对大数据集的手段，但作用对象不同：useMemo 省 CPU 计算时间，虚拟滚动省 DOM 创建和内存占用。实际项目里，一个几千条的列表往往需要先用 useMemo 缓存排序/过滤结果，再用虚拟滚动只渲染可视区域。
  • 状态下移与内容提升模式：132 篇讲的是"重渲染要不要发生"这个更早期的问题，本篇讲的是"哪怕不重渲染，几千个 DOM 节点常驻本身就是负担"这个更底层的问题——两者是同一条性能优化链路上的不同阶段：先通过组件组合减少不必要的重渲染，再通过虚拟滚动减少实际存在的节点数量，缺一个都不够。
  • useCallback：虚拟滚动场景下，每一行通常需要绑定点击等事件回调，如果这些回调在父组件渲染函数体内内联创建，即便虚拟滚动已经把渲染的行数压缩到十几行，这些回调依然会在每次父组件渲染时被重新创建——配合 useCallback 稳定这些回调的引用，能进一步减少不必要的开销，尤其是当行内组件本身也用 memo 包裹时。
  • AntDesign 与 shadcn/ui：138 篇提到 AntDesign 的 Table、Select 等组件面对大数据集时需要开启内置的虚拟滚动选项（比如 virtual 属性）才能维持流畅——这正是本篇原理的直接应用；而 139 篇提到 shadcn/ui 因为组件源码就在项目里，如果官方没有内置虚拟滚动，可以直接在组件源码里接入 @tanstack/react-virtual，不需要等待官方支持这个特性。
