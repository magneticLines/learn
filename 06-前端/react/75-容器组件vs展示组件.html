容器组件 vs 展示组件:数据与 UI 的分工

这是 React 早期最有影响力的组件设计模式(Dan Abramov 2015 年提出)。核心思想很简单:把"怎么拿数据、怎么处理逻辑"和"长什么样"拆成两类组件。它今天依然是好设计的底层直觉——只是实现方式,在 Hooks 时代变了。

两类组件的分工

容器组件 Container
关心:事情怎么运作
· 拿数据(fetch/store)
· 处理业务逻辑
· 管理 state
· 传数据给展示组件
· 几乎没有自己的 DOM
· 难复用(和业务绑定)
展示组件 Presentational
关心:东西长什么样
· 只从 props 拿数据
· 通过回调向上通知
· 无副作用、无数据请求
· 一堆 DOM 和样式
· 高度可复用
· 好测、好在 Storybook 预览
Container:拿数据+逻辑
→ props →
Presentational:纯渲染
← 回调 ←

经典写法(类组件时代)

UserListContainer 管数据,UserList 只管画
// 容器:负责数据和逻辑,不关心样式
class UserListContainer extends Component {
  state = { users: [] };
  componentDidMount() {
    fetch('/api/users').then(...).then(users => this.setState({ users }));
  }
  render() {
    // 只把数据和回调透传下去,自己不画 UI
    return <UserList users={this.state.users} />;
  }
}

// 展示:纯函数式,给什么画什么,零逻辑
function UserList({ users }) {
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

好处:UserList 不知道数据从哪来,给它 mock 数据也能渲染,拿去别处复用、写测试都很方便。逻辑的变化只影响容器,样式的变化只影响展示组件——这正是上一篇"单一职责"的一种具体落地。

Hooks 时代的演变:容器"消失"了

Dan Abramov 后来自己更新了立场:自定义 Hook 出现后,不必再为了分离逻辑而硬造一个容器组件了。过去容器组件的价值,是把"逻辑"从"UI"里剥出来;而 Hook 能更轻量地做到同一件事——把逻辑抽进 Hook,UI 组件直接调用。

逻辑进 Hook,不再需要单独的容器组件
// 逻辑抽进自定义 Hook —— 取代了"容器组件"的角色
function useUsers() {
  const [users, setUsers] = useState([]);
  useEffect(() => { fetch('/api/users').then(...).then(setUsers); }, []);
  return users;
}

// 一个组件搞定:调 Hook 拿数据,直接渲染。逻辑与 UI 仍分离,
// 但分离发生在"Hook vs 组件",而不是"容器 vs 展示"两个组件之间
function UserList() {
  const users = useUsers();
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
本质没变,载体变了。"分离数据逻辑与呈现"这个思想永远正确;但实现它的最佳工具从"两个组件(容器/展示)"演进成了"Hook + 组件"。Hook 的好处是:不额外增加组件层级、逻辑可组合(一个组件里用多个 Hook)、复用更自然。所以现代 React 里你很少再刻意命名 XxxContainer,取而代之的是 useXxx。

对比:容器/展示模式 vs 自定义 Hook

维度容器/展示组件自定义 Hook + 组件
分离的是什么两个组件逻辑(Hook)与呈现(组件)
组件层级多一层容器包裹不增加层级
逻辑复用靠高阶组件/复制容器,较笨重直接调用同一个 Hook,自然
组合多份逻辑嵌套多个容器,层层包裹一个组件里并列用多个 Hook
现状思想仍有价值,写法已过时现代主流做法
展示组件的价值今天依然成立。演变主要是"容器"这一侧被 Hook 取代了;而"纯展示组件(只吃 props、无副作用)"依然是好东西——它可复用、可预览、可测试。所以现代写法常是:自定义 Hook 管逻辑 + 纯展示组件管 UI + 一个薄薄的页面组件把两者接起来。

小结

  • 经典模式:容器组件管数据与逻辑、展示组件只吃 props 画 UI,通过 props/回调连接。
  • 底层思想是"分离数据逻辑与呈现",这一直正确。
  • Hooks 时代,"容器组件"这一侧被自定义 Hook取代:不增加层级、逻辑更好复用与组合。
  • 现代实践:自定义 Hook(逻辑)+ 纯展示组件(UI)+ 薄页面组件(组装)。
  • 下一篇收尾:到底何时该拆分组件,以及过度拆分的坏处。

常见陷阱

  • 在 Hooks 时代仍然刻意造一个"容器组件"包一层。看到教程还在讲 Container/Presentational,就机械地给每个页面都造 XxxContainer,徒增一层组件嵌套。逻辑该抽进自定义 Hook,而不是抽进另一个组件。
  • 把"展示组件"写成了隐性容器。展示组件本该零副作用、只吃 props,但很容易在里面偷偷塞一个 useEffect 或直接 fetch,导致它不再能脱离业务场景独立复用、独立测试,丧失了展示组件的核心价值。
  • 混淆"自定义 Hook 替代容器组件"与"完全不需要分离逻辑"。Hook 取代的只是"容器组件"这个载体,分离逻辑与呈现的思想没有变——如果把 fetch、状态管理全写进展示型组件内部,又倒退回了"什么都管"的大组件。
  • 用 render props 或高阶组件(HOC)硬套容器模式,层层包裹。这是 Hook 出现之前复用容器逻辑的老办法,嵌套多个 HOC 会导致组件树多层包裹、调试时 React DevTools 里全是 WithXxx(WithYyy(Component)),可读性和调试体验都变差,现代代码里应优先用 Hook 组合替代。

面试高频考点

  1. 容器组件和展示组件的核心区别是什么?—— 容器组件关心"数据怎么来、逻辑怎么跑",几乎没有自己的 DOM;展示组件只关心"长什么样",只从 props 拿数据、通过回调向上通知,不产生副作用,因此高度可复用、易测试。
  2. Hooks 出现后,为什么很多项目不再写 XxxContainer 组件了?—— 因为自定义 Hook 能以更轻量的方式完成"分离逻辑与呈现"这件事:不需要多包一层组件、不增加组件树层级,并且多个逻辑可以在同一个组件里并列组合使用(多个 Hook),而多个容器只能嵌套包裹。
  3. 展示组件的哪些特性使它易于测试和复用?—— 无副作用、无数据请求、纯粹由 props 决定输出,这意味着给定相同 props 一定渲染相同结果(类似纯函数),可以脱离真实业务数据用 mock props 渲染,天然适合快照测试和 Storybook 预览。
  4. 如果一个团队的老代码里还有大量 Container 组件,要不要立刻重构成 Hook?—— 不必为了"赶时髦"强行重构,思想上两者等价;通常在自然接触到该模块、需要新增逻辑复用或修复 bug 时顺手迁移即可,盲目大规模重写收益有限且风险不小。

与其他知识点的联系

  • 单一职责(第74篇):容器/展示模式是单一职责原则在组件层面最经典的实例——"数据与逻辑"和"呈现"是两个天然独立的变化理由,把它们拆成两类载体(或本篇提到的 Hook + 组件),就是把抽象原则落到了具体结构上。
  • React.memo(第72篇):展示组件只吃 props、无副作用,是最适合套 React.memo 的对象——它是否需要重渲染完全由 props 决定,不存在内部状态或副作用干扰判断。容器组件(或调用 Hook 的组装层)通常不需要 memo,因为它本身就是变化的起点。
  • 何时拆分组件(下一篇):本篇讲的是"按职责怎么拆"(逻辑 vs 呈现),下一篇进一步讨论"到底要不要拆、拆到什么颗粒度",两篇合起来才是组件拆分的完整方法论——有方向,也有边界。
  • 自定义 Hook:本篇的核心结论——Hooks 时代"容器组件"被自定义 Hook 取代——直接呼应了单一职责篇里的拆分示例(useUsers)。理解这层演变,才能明白为什么现代 React 代码里几乎看不到 XxxContainer 这种命名了。
