虚拟 DOM 与 Reconciliation

上一节说了"UI 是 state 的函数"——每次 state 变化 React 都重新描述 UI。但真实 DOM 操作开销很大，React 怎么解决这个矛盾的？这一节讲清楚其中的机制。

为什么需要虚拟 DOM：真实 DOM 操作的代价

浏览器的真实 DOM 操作非常昂贵。每次你修改一个 DOM 节点，浏览器可能需要：

  • 重排（Reflow）：重新计算元素的几何位置和尺寸，整棵渲染树都可能受影响
  • 重绘（Repaint）：把新的布局结果绘制到屏幕上

这两步是浏览器里最耗性能的操作之一。如果你有一个复杂页面，每次状态变化都把整个页面的 DOM 销毁重建，帧率会直接掉到个位数，用户体验灾难。

但声明式写法天然要求"每次都重新描述整个 UI"——这和"不能全量更新 DOM"之间存在矛盾。虚拟 DOM 就是解决这个矛盾的桥梁。

虚拟 DOM 是什么

虚拟 DOM（Virtual DOM）是内存中的 JavaScript 对象树，是真实 DOM 的轻量级副本。它用普通 JS 对象来描述 DOM 结构，比如：

一个简单组件对应的虚拟 DOM 结构
// 你写的 JSX：
<div className="card">
  <h2>React</h2>
  <p>{count}</p>
</div>

// React 在内存里维护的虚拟 DOM（简化版）：
// 只是普通 JS 对象，创建/比较它们比操作真实 DOM 快得多
{
  type: 'div',
  props: { className: 'card' },
  children: [
    { type: 'h2', props: {}, children: 'React' },
    { type: 'p',  props: {}, children: 42 }
  ]
}

创建和比较这样的 JS 对象，成本远低于直接操作真实 DOM 节点。这就是虚拟 DOM 存在的意义：在内存里做"全量描述"，在真实 DOM 里做"最小化更新"。

Reconciliation（协调）：完整流程

当你调用 setState 或某个 Hook 触发状态更新时，React 会走完这整个流程：

1
触发重新渲染：state 或 props 发生变化，React 标记该组件需要重新渲染
↓
2
重新调用组件函数：React 用新的 state/props 重新调用你写的组件函数，得到新的 JSX 返回值
↓
3
生成新虚拟 DOM 树：React 把新的 JSX 转换成新的虚拟 DOM 对象树（存在内存中）
↓
4
Diff：与旧树对比：React 把新虚拟 DOM 树与上次渲染保存的旧虚拟 DOM 树逐节点比较，找出哪些节点发生了变化
↓
5
最小化更新真实 DOM：只把 diff 发现的差异部分（patch）同步到真实 DOM，没变的节点原封不动

这个"新虚拟树和旧虚拟树做差异比较、只更新差异到真实 DOM"的整个过程，就叫 Reconciliation（协调）。

草稿类比：你要修改一份 10 页的报告。聪明的做法不是把整份报告重新打印，而是：先在草稿纸上写好新版本（虚拟 DOM），然后和原版逐段对比（diff），只把改动了的段落重新打印覆盖（最小化真实 DOM 更新）。React 做的正是这件事——在内存里折腾，对真实 DOM 动最少的手。

Diff 算法的关键规则

对比两棵树的时间复杂度理论上是 O(n³)，React 通过两个假设把它降到 O(n)：

  • 不同类型的节点产生不同的树：如果一个 <div> 变成了 <span>，React 不会尝试复用，直接销毁整棵子树重建
  • key 属性标识列表中的稳定元素：渲染列表时，React 用 key 来判断哪个元素是"同一个"，避免因为顺序变化而错误地销毁/重建节点
常见误解："虚拟 DOM 让 React 更快"
这个说法流传很广，但并不准确。虚拟 DOM 本身引入了额外的开销：每次渲染要创建新的对象树，还要做 diff 计算。

虚拟 DOM 的真正价值是：让你可以用声明式写法（描述 UI 应该是什么），同时保持合理的性能。如果你直接手写精确的原生 DOM 操作，理论上会比 React 更快——但随着业务复杂，这样的代码会越来越难以维护和正确。

React 在"开发体验/可维护性"与"运行性能"之间找到了很好的平衡点，这才是它的真正价值。

React 18 的 Fiber：Reconciliation 的进化

React 16 引入了 Fiber 架构，把原来同步、不可中断的 Reconciliation 变成了可以暂停、恢复、优先级调度的异步过程。React 18 的 Concurrent Mode 就建立在 Fiber 之上。

现阶段你只需要知道：React 的协调过程不是同步阻塞主线程的，它可以把高优先级的更新（用户输入）插队到低优先级的更新（数据加载）之前执行，保证界面响应不卡顿。这是虚拟 DOM + Fiber 带来的更深层价值。
