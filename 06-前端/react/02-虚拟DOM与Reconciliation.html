虚拟 DOM 与 Reconciliation

上一节说了"UI 是 state 的函数"——每次 state 变化 React 都重新描述 UI。但真实 DOM 操作开销很大，React 怎么解决这个矛盾的？这一节讲清楚其中的机制。

为什么需要虚拟 DOM：真实 DOM 操作的代价

浏览器的真实 DOM 操作非常昂贵。每次你修改一个 DOM 节点，浏览器可能需要：

  • 重排（Reflow）：重新计算元素的几何位置和尺寸，整棵渲染树都可能受影响
  • 重绘（Repaint）：把新的布局结果绘制到屏幕上

这两步是浏览器里最耗性能的操作之一。如果你有一个复杂页面，每次状态变化都把整个页面的 DOM 销毁重建，帧率会直接掉到个位数，用户体验灾难。

但声明式写法天然要求"每次都重新描述整个 UI"——这和"不能全量更新 DOM"之间存在矛盾。虚拟 DOM 就是解决这个矛盾的桥梁。

虚拟 DOM 是什么

虚拟 DOM（Virtual DOM）是内存中的 JavaScript 对象树，是真实 DOM 的轻量级副本。它用普通 JS 对象来描述 DOM 结构，比如：

一个简单组件对应的虚拟 DOM 结构
// 你写的 JSX：
<div className="card">
  <h2>React</h2>
  <p>{count}</p>
</div>

// React 在内存里维护的虚拟 DOM（简化版）：
// 只是普通 JS 对象，创建/比较它们比操作真实 DOM 快得多
{
  type: 'div',
  props: { className: 'card' },
  children: [
    { type: 'h2', props: {}, children: 'React' },
    { type: 'p',  props: {}, children: 42 }
  ]
}

创建和比较这样的 JS 对象，成本远低于直接操作真实 DOM 节点。这就是虚拟 DOM 存在的意义：在内存里做"全量描述"，在真实 DOM 里做"最小化更新"。

Reconciliation（协调）：完整流程

当你调用 setState 或某个 Hook 触发状态更新时，React 会走完这整个流程：

1
触发重新渲染：state 或 props 发生变化，React 标记该组件需要重新渲染
↓
2
重新调用组件函数：React 用新的 state/props 重新调用你写的组件函数，得到新的 JSX 返回值
↓
3
生成新虚拟 DOM 树：React 把新的 JSX 转换成新的虚拟 DOM 对象树（存在内存中）
↓
4
Diff：与旧树对比：React 把新虚拟 DOM 树与上次渲染保存的旧虚拟 DOM 树逐节点比较，找出哪些节点发生了变化
↓
5
最小化更新真实 DOM：只把 diff 发现的差异部分（patch）同步到真实 DOM，没变的节点原封不动

这个"新虚拟树和旧虚拟树做差异比较、只更新差异到真实 DOM"的整个过程，就叫 Reconciliation（协调）。

草稿类比：你要修改一份 10 页的报告。聪明的做法不是把整份报告重新打印，而是：先在草稿纸上写好新版本（虚拟 DOM），然后和原版逐段对比（diff），只把改动了的段落重新打印覆盖（最小化真实 DOM 更新）。React 做的正是这件事——在内存里折腾，对真实 DOM 动最少的手。

Diff 算法的关键规则

对比两棵树的时间复杂度理论上是 O(n³)，React 通过两个假设把它降到 O(n)：

  • 不同类型的节点产生不同的树：如果一个 <div> 变成了 <span>，React 不会尝试复用，直接销毁整棵子树重建
  • key 属性标识列表中的稳定元素：渲染列表时，React 用 key 来判断哪个元素是"同一个"，避免因为顺序变化而错误地销毁/重建节点
常见误解："虚拟 DOM 让 React 更快"
这个说法流传很广，但并不准确。虚拟 DOM 本身引入了额外的开销：每次渲染要创建新的对象树，还要做 diff 计算。

虚拟 DOM 的真正价值是：让你可以用声明式写法（描述 UI 应该是什么），同时保持合理的性能。如果你直接手写精确的原生 DOM 操作，理论上会比 React 更快——但随着业务复杂，这样的代码会越来越难以维护和正确。

React 在"开发体验/可维护性"与"运行性能"之间找到了很好的平衡点，这才是它的真正价值。

React 18 的 Fiber：Reconciliation 的进化

React 16 引入了 Fiber 架构，把原来同步、不可中断的 Reconciliation 变成了可以暂停、恢复、优先级调度的异步过程。React 18 的 Concurrent Mode 就建立在 Fiber 之上。

现阶段你只需要知道：React 的协调过程不是同步阻塞主线程的，它可以把高优先级的更新（用户输入）插队到低优先级的更新（数据加载）之前执行，保证界面响应不卡顿。这是虚拟 DOM + Fiber 带来的更深层价值。

常见陷阱

  • 以为虚拟 DOM 一定比原生操作快：这是最广泛的误解。虚拟 DOM 本身有创建对象树和 diff 的开销，如果你精确知道要改哪个节点，手写原生 DOM 操作永远更快。虚拟 DOM 换来的是"声明式写法 + 可维护性"，不是"绝对性能"，二者不能混为一谈。
  • 误以为 diff 会跨层级找可复用节点：React 的 diff 只在同一层级比较，不会把 A 节点挪到别的层级去复用（哪怕内容完全一样）。如果你把一段 JSX 从父组件的这个位置移到另一个嵌套层级，React 会认为是"旧的销毁、新的创建"，而不是"移动"。
  • 列表渲染不加 key，或者用数组下标当 key：没有 key 时 React 默认按位置比对，插入/删除中间项会导致后面所有节点被错误复用，出现状态错乱（比如输入框内容跑到别的行）。用 index 当 key 本质上和不加 key 是一样的坑，因为顺序一变，index 对应的"身份"也变了。
  • 把 Reconciliation 和"重新渲染真实 DOM"划等号：很多人以为 setState 之后马上会有一次真实 DOM 操作。实际上"重新调用组件函数得到新虚拟树"和"把差异同步到真实 DOM"是两个阶段，中间还夹着 diff；理解这个分层，才能理解后面 Fiber、并发渲染这些机制。

面试高频考点

  1. 虚拟 DOM 一定让 React 变快吗？—— 不一定。虚拟 DOM 引入了额外的创建和比较开销，它的价值在于让开发者可以用声明式方式描述 UI，同时把"最小化真实 DOM 操作"这件事自动化，在开发体验和性能之间取得平衡，而不是单纯地比手写 DOM 操作更快。
  2. React 的 diff 算法为什么能把复杂度从 O(n³) 降到 O(n)？—— 基于两个启发式假设：不同类型的节点直接认为是不同的树（不尝试跨类型复用，整棵子树重建），以及同层级节点通过 key 判断身份而不是位置。这两个假设放弃了理论上最优的比较结果，换来了线性时间复杂度，适合绝大多数真实 UI 场景。
  3. 为什么列表渲染要用稳定的 key，而不是数组 index？—— key 是 React 用来判断"这是不是同一个元素"的唯一依据。用 index 时，一旦列表顺序变化（插入、删除、排序），同一个 index 对应的元素身份就变了，导致 React 错误复用 DOM 节点和内部 state（比如输入框的值残留在错误的行上）。用数据自身的唯一标识（如 id）才能让 key 和元素的逻辑身份一一对应。
  4. Fiber 架构解决了 Reconciliation 的什么问题？—— 在 Fiber 之前，Reconciliation 是同步、不可中断的一整块计算，如果树很大，会长时间阻塞主线程导致掉帧。Fiber 把渲染工作拆成可中断、可恢复、带优先级的单元，让高优先级更新（比如用户输入）可以打断低优先级更新，从而在复杂应用中保持界面响应流畅。

与其他知识点的联系

  • 关联 React 是什么（上一节）：上一节说"UI = f(state)"，本节补上了"重新描述 UI 为什么不会很慢"的答案——虚拟 DOM 和 diff 是让声明式哲学在真实性能约束下站得住脚的工程实现，两节合起来才是完整的心智模型：一个讲"为什么这么设计"，一个讲"怎么做到高效"。
  • 关联真实 DOM 的重排与重绘：本节反复出现的"reflow/repaint 代价高"，直接呼应 JS 篇里 IntersectionObserver（第 38 节）反对"scroll + getBoundingClientRect"的理由——getBoundingClientRect() 会强制触发同步布局，这和"直接操作真实 DOM 成本高"是同一个浏览器渲染管线层面的问题，只是分别体现在"判断可见性"和"更新界面"两个场景。
  • 关联深拷贝与浅拷贝、引用比较：React 的 diff 能又快又对，很大程度上依赖"不可变数据 + 引用比较"这套组合——没变的子树保持同一个引用，React 靠浅层 === 比较就能快速跳过整片子树的 diff，这正好呼应 JS 篇讲"不可变数据更新模式故意只做浅拷贝"（第 39 节）的设计动机。
  • 关联 key 与迭代协议：列表渲染要用稳定 key 而非 index，本质是给"哪个虚拟节点对应哪个真实节点"一个可靠身份标识；这和 Generator/Iterator 篇里"生成器对象只能遍历一次、每次调用产出新身份"的思路不同维度但同样在强调——顺序和位置不能替代"身份"本身，这是理解很多 bug（如列表状态错乱、迭代器复用出错）的共同钥匙。
  • 关联事件循环与 Fiber 的可中断渲染：Fiber 把渲染拆成可中断、带优先级的单元，思路上和 JS 事件循环"用宏任务/微任务分批调度任务，不让某一段代码长时间独占主线程"是相通的——都是"把一块大任务拆小，见缝插针地执行，保证关键交互不被阻塞"这一浏览器性能优化母题的具体实现。
