项目目录结构规范

Vite 脚手架给你的只有一个 src/App.tsx,剩下的全靠自己组织。三个月后项目从 5 个组件涨到 80 个,当初"先放着,以后再整理"的想法会变成"没人敢动这个项目"。目录结构不是审美问题,而是团队协作成本和长期可维护性的问题——它决定了新人接手时要花多久才能找到"改这个功能该动哪个文件"。

两种主流分层方式

组织目录本质是回答一个问题:"文件该按什么维度归类?" 常见的两种答案是"按文件类型分"和"按业务功能分"。

方式一:按类型分层(type-based)

按类型分层:components / hooks / pages / services / utils
src/
├── components/     // 所有可复用组件平铺或按子目录放这里
│   ├── Button.tsx
│   ├── UserCard.tsx
│   └── OrderTable.tsx
├── hooks/          // 所有自定义 Hook
│   ├── useAuth.ts
│   └── useOrders.ts
├── pages/          // 路由页面组件
│   ├── UserProfilePage.tsx
│   └── OrderListPage.tsx
├── services/       // 接口请求封装
│   ├── userApi.ts
│   └── orderApi.ts
├── utils/          // 纯工具函数
└── types/          // TS 类型定义

优点是新人一眼就知道"所有 Hook 都在 hooks 里",查找同类文件很方便;缺点是做一个功能要在五六个目录间来回跳——改"订单"这个业务,你得同时打开 components/OrderTable.tsx、hooks/useOrders.ts、pages/OrderListPage.tsx、services/orderApi.ts,项目一大,这种"横向切"会让人晕。

方式二:按功能/领域分层(feature-based)

按功能分层:每个 feature 内聚自己的一整套文件
src/
├── features/
│   ├── user/
│   │   ├── UserProfilePage.tsx
│   │   ├── UserCard.tsx
│   │   ├── useAuth.ts
│   │   ├── userApi.ts
│   │   └── types.ts
│   └── order/
│       ├── OrderListPage.tsx
│       ├── OrderTable.tsx
│       ├── useOrders.ts
│       └── orderApi.ts
├── components/     // 真正跨功能复用的通用组件,如 Button、Modal
├── hooks/          // 真正跨功能复用的通用 Hook
└── lib/            // 第三方库的初始化封装,如 axios 实例

优点是改一个业务,基本只在一个目录里活动,内聚性强,删除一个功能模块也很干净(直接删整个 feature 目录);缺点是"什么算通用组件、什么算功能私有"需要团队有共识,划分不清会出现同样的东西在多个 feature 里重复实现。

按类型分层(type-based)
适合:小型项目、组件库
优点:同类文件好找
缺点:改一个功能要跨多目录
心智模型:「这是什么」
按功能分层(feature-based)
适合:中大型业务项目
优点:业务内聚,改动局部化
缺点:通用/私有边界需要约定
心智模型:「这是干什么用的」
怎么选? 项目小(几个页面、团队 1-2 人)用 type-based 足够,不必过度设计;项目会持续长大、多人协作,优先 feature-based,把"跨功能复用"的部分单独抽到顶层 components/、hooks/、lib/。很多真实项目是混合体:业务用 feature 分,基础设施用 type 分——这也是本节目录树里两者共存的原因。

一个中等规模项目的目录树示例

混合分层:业务按 feature,基础设施按 type
src/
├── app/              // 应用级配置:路由表、全局 Provider、入口布局
│   ├── router.tsx
│   └── providers.tsx
├── features/         // 一个子目录 = 一块业务,自成一体
│   ├── auth/
│   └── order/
├── components/       // 纯 UI、无业务语义的通用组件(Button、Modal、Table)
├── hooks/            // 跨功能复用的通用 Hook(useDebounce、useLocalStorage)
├── lib/              // 第三方库封装(axios 实例、query client)
├── utils/            // 纯函数工具(格式化、校验)
├── types/            // 跨模块共享的全局类型
├── assets/           // 图片、字体等静态资源
├── App.tsx
└── main.tsx
陷阱:目录越随意,项目越难维护。 最常见的失败模式不是"选错了分层方式",而是压根没有约定——今天顺手把新组件扔进 components/,明天又建个 common/ 放差不多的东西,后天干脆堆在 pages/ 旁边。几周后,同一类文件散落在四五个地方,新人不知道该往哪儿放,也不知道该去哪儿找。目录结构一旦定下来,就该写进团队约定(哪怕只是 README 里几行字),而不是"看心情"。
一个判断"该不该放进 components/ 通用目录"的简单标准: 问自己"这个组件如果被另一个完全不相关的 feature 拿去用,需要改代码吗?"——不需要改就是真通用,该放 components/;需要改(比如硬编码了"订单"相关的文案或类型),说明它其实是 features/order/ 的私有产物,先别急着往外提。

小结

  • 按类型分层(components/hooks/pages/services)适合小项目,同类文件好找,但改一个功能要跨多目录。
  • 按功能分层(features/xxx)适合会持续长大的中大型项目,业务内聚、改动局部化,但需要团队约定"通用 vs 私有"的边界。
  • 真实项目常是混合体:业务代码按 feature 分,跨功能的组件/Hook/工具按 type 分放在顶层。
  • 判断一个组件是否该放进通用目录:换一个不相关的业务场景用,要不要改代码——不用改才是真通用。
  • 目录结构的最大风险不是"选错方案",而是"没有约定",导致同类文件散落各处,新人无从下手。
  • 约定一旦定下就该写下来(哪怕只是 README 几行字),而不是靠口口相传或"看心情"维持。

与其他知识点的联系

  • 关联 Vite 创建 React 项目（上一章）：脚手架只给了 src/App.tsx 这一个文件，本章讨论的目录规范正是"脚手架结束之后，项目该往哪个方向长"的答案，是上一章遗留问题的延续。
  • 关联环境变量 import.meta.env（下一章）：按功能分层时，lib/ 目录常用来存放"读取环境变量后初始化好的实例"（如按 VITE_API_BASE_URL 配置好 baseURL 的 axios 实例），目录结构和环境变量配置是同一个工程化体系里配合工作的两部分。
  • 关联 useQuery/useMutation/RTK Query 的组织位置：本章 feature-based 分层里，每个 features/xxx/ 目录一般会放该业务自己的 query hook 封装（如 useOrders.ts 内部包一层 useQuery），这是"数据获取逻辑"和"目录结构"两个知识点在真实项目里的交汇点。
  • 关联组件命名规范：无论按 type 还是按 feature 分层，"文件名与组件名一致"这条约定都要遵守，目录解决"放哪儿"，命名规范解决"叫什么"，两者共同构成项目的可维护性基础。
  • 关联团队协作与代码评审：本章强调的"约定要写下来"，本质上是在为后续的 code review、新人 onboarding 降低沟通成本，这类工程规范的价值往往要到项目变大、多人协作时才会显现。
