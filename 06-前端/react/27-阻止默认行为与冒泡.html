阻止默认行为与冒泡

事件系统有两个独立的概念容易混淆：默认行为是浏览器对特定元素内置的动作；冒泡是事件从子元素向上传播到父元素的机制。两者需要不同的 API 来阻止。

阻止默认行为：e.preventDefault()

某些 HTML 元素有浏览器内置的默认行为，不阻止它们会干扰 React 应用：

最常见的三种需要阻止默认行为的场景
// ① form 提交 → 默认刷新页面（最高频）
function handleSubmit(e) {
  e.preventDefault();  // 阻止页面跳转/刷新
  // 这里用 JS 处理数据，不让浏览器自己提交
}
<form onSubmit={handleSubmit}>...</form>

// ② a 标签点击 → 默认跳转页面
function handleLinkClick(e) {
  e.preventDefault();  // 阻止跳转，改用 JS 路由
  router.push('/new-page');
}
<a href="/new-page" onClick={handleLinkClick}>...</a>

// ③ 右键菜单 contextmenu → 默认弹出系统菜单
function handleRightClick(e) {
  e.preventDefault();
  showCustomMenu(e.clientX, e.clientY);
}

事件冒泡：事件如何向上传播

点击一个元素时，事件不只在它自己身上触发，还会向上传播给所有祖先元素。这叫冒泡。

<button> 被点击，触发 onClick
↑ 冒泡
<div> 的 onClick 也被触发
↑ 冒泡
<body> 的 onClick 也被触发
冒泡问题：点按钮触发了父元素的事件
function Modal() {
  function handleOverlayClick() {
    closeModal();  // 点击遮罩关闭弹窗
  }

  function handleContentClick() {
    // 点击弹窗内容区，本意是什么都不做
    // 但事件会冒泡到遮罩层，触发 closeModal !
  }

  return (
    {/* 点击遮罩关闭 */}
    <div className="overlay" onClick={handleOverlayClick}>
      {/* 点弹窗内容也会冒泡到遮罩层 → 弹窗意外关闭！ */}
      <div className="content" onClick={handleContentClick}>
        弹窗内容
      </div>
    </div>
  );
}

阻止冒泡：e.stopPropagation()

在内容区阻止事件向上传播
function handleContentClick(e) {
  // 阻止点击内容区的事件冒泡到遮罩层
  e.stopPropagation();
}

<div className="content" onClick={handleContentClick}>
  弹窗内容（点击不会关闭弹窗了）
</div>

两者的区别与组合

方法阻止的是什么典型场景
e.preventDefault() 浏览器内置的默认行为 form 提交刷新、a 标签跳转
e.stopPropagation() 事件向父元素冒泡 Modal 内容区、列表行内操作按钮
两者一起用 默认行为 + 冒泡 列表行点击行（导航）+ 行内删除按钮（不导航不冒泡）
同时阻止默认行为和冒泡
function FileList() {
  function handleRowClick(file) {
    openFile(file);  // 点击行 → 打开文件
  }

  function handleDeleteClick(e, file) {
    e.stopPropagation();  // 不让删除事件冒泡到行，避免同时打开文件
    deleteFile(file);
  }

  return (
    <ul>
      {files.map(file => (
        <li key={file.id} onClick={() => handleRowClick(file)}>
          {file.name}
          <button onClick={e => handleDeleteClick(e, file)}>删除</button>
        </li>
      ))}
    </ul>
  );
}
不要滥用 stopPropagation
阻止冒泡会让事件分析工具（如埋点）漏采。如果父组件依赖事件冒泡来感知子元素的交互，stopPropagation 会悄悄破坏这条链路。优先用条件判断解决问题，只在确实需要隔离事件流时才用 stopPropagation。
onClickCapture：捕获阶段监听
React 也支持在捕获阶段（从上往下）监听事件，只需把 on 后面加 Capture：onClickCapture。这在调试冒泡问题时有用，日常开发几乎不需要。

常见陷阱

  • 把 preventDefault() 和 stopPropagation() 用错场景：两者解决的是完全不同的问题——preventDefault 阻止的是浏览器对该元素的内置行为（跳转、刷新、右键菜单），stopPropagation 阻止的是事件向父元素传播。表单提交刷新页面用 stopPropagation 是无效的（因为默认行为和冒泡是独立的两条线），必须用 preventDefault。混用会导致"改了却没生效"的困惑。
  • 忘记 onChange 之类的非冒泡事件不受 stopPropagation 影响：并不是所有事件都会冒泡，比如 onMouseEnter/onMouseLeave 本身设计上就不冒泡，在这些事件里调用 stopPropagation 没有任何意义，说明对"这个事件是否冒泡"的判断错了，而不是 stopPropagation 本身失效。
  • 滥用 stopPropagation 破坏了依赖冒泡的功能：很多 UI 库（下拉菜单、弹层、埋点系统）依赖在 document 上监听冒泡上来的点击事件来判断"点击了外部区域"从而关闭浮层。如果子组件里滥用了 stopPropagation，会让这类"点击外部关闭"逻辑失效，且这种 bug 往往要跨组件排查才能发现，定位成本很高。
  • 以为 return false 能像老式 jQuery 那样阻止默认行为：在 React 的事件处理函数里 return false 没有任何特殊含义，不会阻止表单提交或链接跳转，这是从 jQuery/原生 onclick="..." attribute 时代带来的错误直觉，必须显式调用 e.preventDefault()。
  • 在异步回调里调用 preventDefault 却为时已晚：比如在 onSubmit 里先 await 一个请求，再调用 e.preventDefault()，此时浏览器可能已经开始执行默认的表单提交/页面跳转了。preventDefault 必须在事件处理函数同步执行的阶段、事件还没有完成默认动作之前调用。

面试高频考点

  1. e.preventDefault() 和 e.stopPropagation() 分别阻止的是什么，能否举例说明必须同时使用的场景？—— preventDefault 阻止浏览器对当前元素的默认动作（如表单提交刷新、a 标签跳转），stopPropagation 阻止事件对象继续向父元素冒泡触发它们的监听器，两者互不影响、可以独立使用也可以同时使用。典型的同时使用场景：列表行整体可点击跳转（父级 onClick 处理导航），行内还有一个"删除"按钮，点击删除既不希望触发父级的导航（需要 stopPropagation 阻止冒泡到父级），本身如果是 <a> 或 <button type="submit"> 还可能需要 preventDefault 阻止其自身默认行为。
  2. React 的事件冒泡顺序和浏览器原生事件冒泡顺序一致吗？—— 一致，都是从触发元素开始逐层向上传播到祖先元素（先捕获阶段从外到内，再冒泡阶段从内到外，日常 onClick 监听的是冒泡阶段）。区别在于 React 内部把所有事件统一委托到根节点后再模拟分发，但对开发者暴露出来的语义和原生冒泡完全一致，写代码时可以直接套用原生事件冒泡的心智模型。
  3. 为什么"点击弹窗遮罩关闭弹窗，点击弹窗内容不关闭"这个需求必须用 stopPropagation 而不是在遮罩的处理函数里判断 e.target？—— 其实两种方案都能实现：用 e.target === e.currentTarget 判断"点击的确实是遮罩本身而不是子元素"是另一种常见做法，不依赖 stopPropagation，也不会破坏内容区事件继续冒泡给更外层的能力（比如埋点统计）。stopPropagation 的方案更直接但会截断该事件后续的冒泡路径，两种方案的取舍本质是"事件流该不该被中断"的设计决策，面试可以对比说出优劣。
  4. onClickCapture 和 onClick 的区别是什么，什么场景需要用捕获阶段？—— onClick 监听的是事件冒泡阶段（从子到父），onClickCapture 监听的是捕获阶段（从父到子，比冒泡阶段更早触发）。当需要在事件真正到达目标元素之前就介入处理（比如全局的权限拦截、埋点前置校验，或者要确保某个外层逻辑一定先于内层子组件的 onClick 执行）时，会用 Capture 版本。

与其他知识点的联系

  • 合成事件与原生事件的区别：preventDefault、stopPropagation 都是 SyntheticEvent 对象上的方法，接口与原生事件完全一致，但实际的分发是在 React 委托到根节点的事件系统内完成的——理解合成事件的委托机制，才能理解为什么"React 的冒泡顺序和原生一致"这个结论并不是巧合，而是 React 刻意在抽象层之上还原了原生冒泡语义。
  • onSubmit（表单提交）：preventDefault 最常用的场景就是表单的 onSubmit——不阻止默认行为，浏览器会尝试真正发起页面级的 GET/POST 请求，导致整页刷新、React 应用状态全部丢失，这也是"几乎每次 onSubmit 都要 preventDefault"这条经验法则的由来。
  • 传参给事件处理函数：上一节例子里给事件处理函数传参时常常需要同时保留事件对象（如 (e, file) => handleDeleteClick(e, file)），这里的 e 正是用来调用 stopPropagation 的，两个知识点在"行内操作按钮"这种场景里天然要结合使用。
  • 条件渲染与短路运算符：阻止冒泡常用于弹窗/下拉菜单"点遮罩关闭、点内容不关闭"的场景，而这类组件本身往往依赖条件渲染（{open && }）来控制显示与隐藏，二者共同构成完整的弹层交互逻辑。
  • useEffect 里的原生事件监听：依赖冒泡实现的"点击外部关闭"逻辑，通常是在 useEffect 里给 document 绑定原生 click 监听实现的，如果内部组件滥用了 stopPropagation 截断了合成事件的冒泡，并不会影响这种绑在 document 上的原生监听（因为原生事件冒泡链路和 React 合成事件冒泡链路是分别独立模拟的），这也是排查"点击外部关闭失效"问题时容易搞混的一个点。
