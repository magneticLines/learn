useEffect vs useLayoutEffect：绘制前还是绘制后

两者 API 一模一样，区别只在执行时机：useEffect 在浏览器绘制之后异步执行，useLayoutEffect 在绘制之前同步执行。这个几毫秒的差别，决定了会不会出现视觉闪烁。

关键差别：useLayoutEffect 卡在"DOM 已改、但屏幕还没画"这个窗口里同步跑，会阻塞绘制——它跑完（可能还触发一次重渲染）浏览器才画。useEffect 则等画完了才跑，不阻塞。

为什么需要 useLayoutEffect:避免闪烁

典型场景：你需要先读取 DOM 布局（比如 tooltip 的高度、元素位置），再据此调整位置。如果用 useEffect，流程是：先以错误位置画一帧 → effect 跑 → 修正 → 再画一帧。用户会看到 tooltip"闪一下跳到正确位置"。

tooltip 定位:必须用 useLayoutEffect
function Tooltip() {
  const ref = useRef(null);
  const [height, setHeight] = useState(0);

  // ✅ 绘制前同步读高度并 setState:用户看不到"先错后对"
  useLayoutEffect(() => {
    const { height } = ref.current.getBoundingClientRect();
    setHeight(height); // 触发的重渲染也在同一次绘制前完成
  }, []);
  // 若这里换成 useEffect:会先画一帧错的，再跳位 → 肉眼可见闪烁

  return <div ref={ref} style={{ top: -height }}>...</div>;
}

性能权衡:为什么默认用 useEffect

useLayoutEffect 同步阻塞绘制。如果你在里面做耗时操作，用户会觉得界面"卡住"——点了没反应，等一下才出来。useEffect 让浏览器先把画面画出来（用户立刻有反馈），副作用在后台补做，体验更流畅。

决策规则：默认用 useEffect；只有当"不同步执行就会看到视觉闪烁"时，才换 useLayoutEffect。换句话说，useLayoutEffect 几乎只用于"读取布局尺寸/位置并据此调整 UI"这一类需求。绝大多数副作用（请求、订阅、日志、改 title）都不该阻塞绘制。
SSR 陷阱：useLayoutEffect 在服务端渲染时会报警告——因为服务端没有真实 DOM、也不"绘制"，layout effect 无从执行。React 会提示 useLayoutEffect does nothing on the server。解决办法：确认这段逻辑是否真的需要在服务端跑（通常不需要），或用 useEffect 替代 / 条件降级。这是 Next.js 等 SSR 框架里的常见告警。

和浏览器渲染管线的联系

浏览器一帧内大致是：JS 执行 → 样式计算 → 布局(Layout/Reflow) → 绘制(Paint)。useLayoutEffect 之所以能"读到准确布局"，是因为 React 已经把 DOM 改完、浏览器也已算好布局，但还没 Paint。此时 getBoundingClientRect 拿到的是最新尺寸。理解这条管线，就理解了为什么叫 "Layout" Effect——它在布局已定、绘制未发生的缝隙里工作。

class 组件对照。useLayoutEffect 对应 class 里的 componentDidMount / componentDidUpdate（它们也是同步、绘制前执行）；useEffect 没有完全等价的 class 生命周期，它是 Hooks 特有的"异步、不阻塞"模型。从 class 迁移代码时，若原来在 componentDidMount 里读 DOM 尺寸，迁移后要用 useLayoutEffect 才能保持无闪烁；纯请求类逻辑则迁到 useEffect 更好。
