自定义 Hook 命名规范：为什么必须以 use 开头

自定义 Hook 是 React 里复用"带状态的逻辑"的方式。它看起来就是个普通函数，但有一条铁律：名字必须以 use 开头。这不是约定俗成的建议，而是让整套 Hook 机制正常工作的前提。

自定义 Hook 的本质：能调用其它 Hook 的函数

自定义 Hook 就是一个普通 JavaScript 函数，特别之处只有一点：它内部调用了其它 Hook（useState、useEffect、useContext……甚至别的自定义 Hook）。它把一段"用到 Hook 的逻辑"打包，让多个组件复用。

一个最简单的自定义 Hook
// use 开头 + 内部调用了 useState → 它就是个自定义 Hook
function useToggle(initial = false) {
  const [on, setOn] = useState(initial);
  const toggle = () => setOn(v => !v);
  return [on, toggle];  // 返回什么、什么形状，完全你自己定
}

// 用起来和内置 Hook 一样
function Modal() {
  const [open, toggle] = useToggle();
  // ...
}
它复用的是"逻辑"，不是"UI"。组件复用的是渲染输出；自定义 Hook 复用的是有状态的行为——每个组件调用它，都会拿到一份独立的 state（下一篇细讲）。这弥补了 class 时代 HOC / render props 那些笨重的复用方案。

核心规则：必须以 use 开头

React 靠函数名是否以 use 开头来识别"这是一个 Hook"。这个命名不是给人看的礼貌用词，而是给 工具链（ESLint、编译器、React 本身）看的信号。

原因一：ESLint 靠命名执行 Hook 规则

eslint-plugin-react-hooks 的两条规则——"只在顶层调用 Hook""只在组件/Hook 里调用 Hook"——就是靠 use 前缀来判断哪些是 Hook 调用的。名字对了，ESLint 才能帮你检查有没有把 Hook 写进 if、循环、普通函数里。

✅ use 开头
function useCounter() {
  const [n, setN] =
    useState(0)  // ✓ ESLint 检查
  return n
}
// 违反 Hook 规则会报错
❌ 不以 use 开头
function getCounter() {
  const [n, setN] =
    useState(0)  // ESLint 看不出
}                // 这是 Hook！
// 规则失效，if 里调用也不报错
// → 潜在的顺序错乱 bug 溜过去

原因二：给读代码的人正确的心智预期

看到 use 前缀，开发者立刻知道"这个函数内部可能有 state/effect，受 Hook 规则约束，不能条件调用"。看到普通函数名，就假设它是纯工具函数、可以随便调用。名字传递的是"契约"。

反过来也要遵守：不是 Hook 的函数，不要用 use 开头。一个纯计算工具函数如果叫 useFormat()，ESLint 会误以为它是 Hook，反而对它施加 Hook 规则、发出误报。use 前缀是"我内部调用了 Hook"的承诺——没调用就别用。

命名建议：use + 语义

Hook 名表达的语义
useToggle一个可切换的布尔状态
useFetch(url)按 url 拉取数据的完整逻辑
useLocalStorage(key)与 localStorage 同步的状态
useDebounce(value, ms)防抖后的值
useWindowSize当前窗口尺寸（随 resize 更新）

好名字应描述"它给你什么/做什么"，而不是内部实现。命名清晰的 Hook，调用处一眼就懂，不用翻实现。

什么时候该抽自定义 Hook？当你发现两个以上组件里出现几乎一样的 useState + useEffect 组合逻辑时——比如都在 mount 时订阅、卸载时取消。把这段逻辑搬进一个 useXxx 函数即可。注意：即使只在一个组件用，把复杂逻辑抽成命名良好的 Hook，也能让组件本身更清爽。

小结

  • 自定义 Hook 就是内部调用了其它 Hook 的普通函数，复用的是"有状态的逻辑"。
  • 名字必须以 use 开头：ESLint / 编译器靠这个前缀识别 Hook、执行 Hook 规则。
  • 反之，非 Hook 的普通函数不要用 use 开头，避免误报。
  • 命名描述"能力/产物"（useFetch/useToggle），让调用处一目了然。

常见陷阱

  • 普通工具函数误用 use 前缀：一个不调用任何 Hook 的纯函数（比如格式化日期）如果命名为 useFormatDate，ESLint 会把它当作 Hook，对它施加"只能在组件顶层调用"的规则，导致你在普通函数里调用它时收到莫名其妙的报错。use 前缀是承诺而不是装饰。
  • 真正的 Hook 没用 use 开头：函数内部调用了 useState，但函数名是 getCounter 之类普通命名。ESLint 无法识别它是 Hook，于是在 if/循环里条件调用它时不会报错，Bug 悄悄潜伏——直到某次条件变化导致 Hook 调用顺序错乱，出现状态错位的诡异问题才暴露。
  • 以为命名只是"风格问题"：初学者常觉得 use 前缀无关紧要，改不改都行。但它是 React 工具链（ESLint 插件、React Compiler、未来的静态分析）识别 Hook 的唯一依据——命名错了，所有基于名字做检查/优化的机制都会失效或误判。
  • 忘记自定义 Hook 也要遵守 Hook 规则：既然它内部调用了其它 Hook，那么它本身也受"只在顶层调用"的约束——不能在自定义 Hook 内部的 if/循环里条件调用 useState，这跟在组件里犯的错误是同一类问题。

面试高频考点

  1. 为什么自定义 Hook 必须以 use 开头？这只是约定还是有强制作用？—— 不只是约定。eslint-plugin-react-hooks 靠这个前缀静态识别哪些函数调用属于 Hook 调用，进而检查"只在顶层调用""只在组件/Hook 内调用"两条规则。命名错误会让 lint 规则失效，埋下 Hook 调用顺序错乱的隐患；同时它也是给阅读者的契约，告诉别人"这个函数内部可能有状态，受 Hook 规则约束"。
  2. 一个函数内部调用了 useContext，但没有以 use 命名，会有什么后果？—— ESLint 无法识别它是 Hook，因此不会检查它是否被在条件语句、循环、普通事件回调里调用。如果它被写进了 if 分支，一旦条件变化，Hook 的调用顺序在多次渲染间不一致，React 内部按"调用顺序"关联 state 的机制会错位，导致读到错误的 state 或抛出错误，且这类问题不会在编写时被工具提前拦截。
  3. 自定义 Hook 和普通的工具函数（utility function）如何界定？—— 界定标准只有一个：函数内部是否调用了其它 Hook（useState/useEffect/useContext 或别的自定义 Hook）。调用了就必须以 use 命名并受 Hook 规则约束；没调用的纯计算函数应该用普通函数命名，避免让 ESLint 错误地把它当 Hook 检查。
