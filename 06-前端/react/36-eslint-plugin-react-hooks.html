eslint-plugin-react-hooks：把规则交给工具

前两篇的规则纯靠人记很容易漏。React 官方出了这个 ESLint 插件，在你写代码时就静态检查两条规则，还额外提供了 exhaustive-deps 帮你补全依赖数组。它是 React 项目的标配。

插件提供的两条规则

规则作用默认级别
rules-of-hooks强制前两篇的两条规则：只在顶层、只在组件/自定义 Hook 中调用error（报错）
exhaustive-deps检查 useEffect/useMemo/useCallback 的依赖数组是否完整warn（警告）

注意默认级别：违反 Hook 规则是硬伤，直接 error；依赖数组是启发式建议，可能有误报，所以只是 warn。

如何配置（Flat Config，ESLint 9+）

eslint.config.js（推荐 recommended 预设）
import reactHooks from 'eslint-plugin-react-hooks';

export default [
  // 直接用官方推荐预设，自动开启两条规则
  reactHooks.configs['recommended-latest'],

  // 或手动配置，方便调级别
  {
    plugins: { 'react-hooks': reactHooks },
    rules: {
      'react-hooks/rules-of-hooks': 'error',
      'react-hooks/exhaustive-deps': 'warn',
    },
  },
];

老项目的 .eslintrc（旧格式）里则是 "extends": ["plugin:react-hooks/recommended"]。Vite / CRA 的 React 模板通常已内置。

exhaustive-deps 在做什么

它分析你 effect 回调里用到的所有响应式值（props、state、以及基于它们派生的变量、函数），检查这些值是否都出现在依赖数组里。缺了就警告，并提供自动修复。

典型警告与修复
function Chat({ roomId }) {
  useEffect(() => {
    const conn = connect(roomId);  // 用到了 roomId
    return () => conn.close();
  }, []); // ⚠️ 警告：缺少依赖 'roomId'
  // 自动修复 → [roomId]
  // 修复后：roomId 变化时会重连，行为才正确
}
exhaustive-deps 不只是"洁癖"，它在防真实 bug。缺失的依赖会让 effect 捕获到过期的闭包值（下一篇 useEffect 依赖数组会详讲）——effect 里读到的 roomId 永远是首次渲染那个，用户切了房间却还连着旧房间。这个 bug 极隐蔽，lint 能在编码期就拦下。

为什么它有争议

争议几乎全在 exhaustive-deps（rules-of-hooks 没人质疑）。焦点是：

  • 有时你"故意"只想在挂载时跑一次，但用到了某个 prop，插件坚持要你加进依赖，加了又会导致重复执行。这时容易被"骗"着去写不该有的依赖，或滥用 // eslint-disable-next-line。
  • 函数依赖的连锁反应：effect 依赖一个函数，函数每次渲染重建，导致 effect 每次都跑——正确解法是 useCallback 或把函数挪进 effect，但新手常直接禁用规则了事。
  • 批评者认为它"暴露了 useEffect 心智模型的复杂性"，把本该由框架处理的东西推给了开发者。
❌ 用禁用注释掩盖问题
useEffect(() => {
  doThing(dep);
// eslint-disable-next-line
}, []); // 骗过 lint，
        // 但 dep 变了不会重跑
✅ 正视警告，改结构
useEffect(() => {
  doThing(dep);
}, [dep]);
// 若不想因 dep 重跑，
// 说明依赖设计有问题，
// 该重构而非禁用
禁用规则前先问自己一句："我真的确定这个依赖不需要吗？" 官方立场很明确——90% 情况下警告是对的，你只是暂时没看出 bug。eslint-disable 应当是极罕见的例外，且必须写清理由。React 团队后续在实验性的 React Compiler 里进一步弱化了手动管理依赖的需求。
联系工程实践：把规则设为 error 意味着 CI 会因违规而红。这正是 lint 的价值——把"团队约定"变成"机器强制"，避免 code review 时反复口头提醒。Hook 规则天然适合 lint，因为它们是语法层面可静态判定的（顺序、位置、命名）。
