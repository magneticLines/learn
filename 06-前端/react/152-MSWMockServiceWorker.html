MSW(Mock Service Worker)

组件测试里经常要测"发请求拿数据后渲染列表"这种场景,但测试环境里没有真实后端。MSW 不是又一个"mock 掉 fetch 函数"的工具,而是在网络层拦截请求——业务代码完全不知道自己被 mock 了。

直接 mock fetch/axios 的痛点

最朴素的做法是在测试里直接替换 global.fetch 或 vi.mock('axios'),让它返回一个假的 Promise。这样做能跑通,但有两个明显问题:

  • 和实现细节耦合。 测试代码必须知道组件内部到底用的是 fetch 还是 axios、调用参数长什么样,一旦把请求库从 axios 换成 fetch,所有 mock 代码全部要重写——这正是上一篇强调要避免的"测实现细节"。
  • 测不出真实的请求流程。 直接 mock 函数返回值,跳过了"发出真实的 HTTP 请求 → 走请求拦截器/鉴权头 → 解析响应"这整条链路,一些真实环境才会暴露的问题(比如请求头缺失、URL 拼错)在测试里根本发现不了。

MSW 的原理:在网络层拦截,业务代码无感知

MSW 的关键设计是:它不去改你的业务代码,而是在更底层——浏览器里通过 Service Worker、Node 环境(测试运行时)里通过拦截原生请求模块——把匹配到的请求截住,返回你定义的假数据。对组件而言,它调用的还是同一个 fetch('/api/users'),跟连真实后端时代码完全一样,只是这次请求半路被 MSW"接住"了。

组件调用 fetch('/api/users')
→
MSW 在网络层拦截
→
按 handler 规则返回假数据
→
组件收到"看似真实"的响应
类比: 直接 mock 函数,相当于把电话本身换成一个"自动答录机",打电话的人(组件)得知道自己打的不是真电话。MSW 更像是在电话公司的机房里把这通电话转接到了一个模拟对端——打电话的人全程无感知,拨号、等待、听到声音的整个体验和真的打通了一模一样。

基本用法:定义 handler

核心概念是 handler——描述"匹配到什么请求,返回什么数据"的规则。测试启动前注册好 handler,之后组件里的请求就会被自动接管。

handlers.ts —— 定义接口的假数据
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.get('/api/users', () => {
    // 拦截到这个 GET 请求,直接返回假数据,不会真的发出网络请求
    return HttpResponse.json([
      { id: 1, name: '张三' },
      { id: 2, name: '李四' },
    ]);
  }),
];
setupTests.ts —— 在测试环境里启动拦截服务
import { setupServer } from 'msw/node';
import { handlers } from './handlers';

export const server = setupServer(...handlers);

beforeAll(() => server.listen());   // 测试开始前打开拦截
afterEach(() => server.resetHandlers()); // 每个测试后重置,避免互相污染
afterAll(() => server.close());       // 测试全部结束后关闭

配好之后,组件测试里可以正常用 RTL 的 findByText 等异步查询,等待"请求返回后渲染出来的用户名"出现——整个过程和真实调用后端没有区别,唯一不同的是数据是假的。

常见坑:忘记 resetHandlers。 如果某个测试里用 server.use(...) 临时覆盖了某个 handler(比如模拟接口报错),没有在测试后重置,这个覆盖会泄漏到下一个测试,导致看起来毫不相关的测试莫名其妙失败。养成"每个测试后自动 reset"的习惯能省很多排查时间。

顺带的好处:开发环境也能复用

因为 MSW 拦截的是真实的网络层,同一套 handler 定义不仅能用在测试里,也能在开发环境启动一个 mock server,在后端接口还没写完时,前端就能拿着约定好的假数据把页面和交互先联调起来。这是很多"直接 mock 函数"方案做不到的额外收益——测试代码和开发联调代码共享同一份 mock 数据定义,不用维护两套。

和 TanStack Query 的关系。 本系列在数据获取篇讲过 TanStack Query 管理服务端状态的缓存、重试、失效策略。测试这类组件时,MSW 正好补上"服务端响应"这一环——TanStack Query 该怎么发请求就怎么发,MSW 在半路接住,两者组合能测出接近真实的数据流转全貌,而不需要真的连一个后端。

小结

  • 直接 mock fetch/axios 函数会和实现细节耦合(换请求库测试就要重写),也测不出真实请求链路上的问题。
  • MSW 在网络层拦截请求,业务代码发出的还是真实调用,完全无感知,测试更接近真实使用场景。
  • 核心概念是 handler:定义"匹配什么请求,返回什么假数据",用 setupServer 在测试里启动拦截。
  • 常见陷阱:忘记 afterEach 里 resetHandlers(),导致某个测试临时覆盖的 handler 污染后续测试。
  • 同一套 handler 定义还能复用到开发环境做联调 mock,不需要为测试和开发分别维护两套假数据。
