路由 state 传参：不写进 URL 的"隐式"数据

有时你想在跳转时带点数据过去，但又不想暴露在 URL 里——比如"从哪个页面跳来的""列表项的完整对象，省一次请求"。navigate 的 state 选项就是干这个的：数据藏在 History 记录里，用 useLocation 取出。

为什么需要第三种传参方式

前两篇讲了 params（/users/:id）和 query（?page=2），它们都显示在 URL 上。但有些数据不适合放 URL：太长（一整个对象）、不想让用户看到/篡改、或纯粹是"临时上下文"（从哪来的）。路由 state 让数据随跳转传递但不出现在地址栏。

发送方:navigate 第二参数传 state
import { useNavigate } from 'react-router-dom';

function UserRow({ user }) {
  const navigate = useNavigate();

  function openDetail() {
    // URL 只带 id，但把整个 user 对象和来源藏进 state
    navigate(`/users/${user.id}`, {
      state: { user, from: '/users' }
    });
  }
  return <tr onClick={openDetail}>...</tr>;
}

<Link> 也能传：<Link to="/users/42" state={{ from: '/users' }}>。

接收方:useLocation 读 state

目标页用 useLocation().state 取出
import { useLocation, useParams } from 'react-router-dom';

function UserDetail() {
  const { id } = useParams();
  const location = useLocation();
  // state 可能为 null（比如直接输入 URL 进来的），一定要兜底
  const user = location.state?.user;

  // 有 state 就直接用（省一次请求）；没有就用 id 兜底去拉
  if (!user) {
    // fetch(`/api/users/${id}`) ...
  }
  return <h1>{user?.name}</h1>;
}

最大的陷阱:刷新页面 state 就没了! state 存在浏览器 History 记录里（history.state），跟着当前这条历史项走。用户刷新、直接输入 URL、从外部链接打开时，location.state 会是 null。所以:

① 读 state 必须用可选链 location.state?.xxx 并做兜底（比如用 URL 里的 id 重新请求）。
② 绝不能把 state 当作页面唯一的数据来源——它只是"锦上添花"的优化（省一次请求），不是"雪中送炭"的依赖。

state 能存什么?为什么不能存函数? state 会被浏览器结构化克隆后存进 History，所以只能放可序列化的数据（对象、数组、字符串、数字等），不能放函数、DOM 节点、类实例等。而且它有大小限制（各浏览器不同，通常几 MB），别塞超大数据。本质上它和 params/query 一样是"传参"，只是载体是内存中的 History 记录而非 URL 文本。

典型适用场景

  • 记录来源页: 传 { from: location.pathname }，详情页的"返回"按钮据此 navigate(from)；鉴权重定向后登录完跳回原页也用这招。
  • 列表 → 详情省请求: 列表已有完整对象，跳详情时带过去，首屏直接渲染，同时保留 id 兜底重新拉取。
  • 跨页临时数据: 多步表单里上一步的数据、提示 toast 的内容等一次性、不需要出现在 URL 的信息。
  • 保存/恢复滚动位置等 UI 上下文。

小结

  • 路由 state 通过 navigate(path, { state })（或 <Link state>）传递不显示在 URL 的隐式数据，用 useLocation().state 读取。
  • 它存在 History 记录里，刷新/直接访问会变 null——必须用 ?. 兜底，绝不能作为唯一数据源。
  • 只能传可序列化数据（不能是函数），有大小限制。
  • 三者取舍: 定位资源用 params，可分享的视图状态用 query，不想暴露、不需持久、可带复杂对象的临时数据用 state。

与其他知识点的联系

  • Query 参数与 useSearchParams：query 和 state 是同一个问题（跳转时带数据）的两种答案，取舍标准恰好相反——想让数据可分享、刷新不丢就用 query；想让数据不出现在地址栏、能传复杂对象就用 state。写代码前先想清楚"这份数据该不该被用户看见/收藏"，就知道该用哪个。
  • Outlet 占位符（下一篇）：state 常用来在跳转前"预置"数据，而嵌套路由里更推荐用 <Outlet context={...}> + useOutletContext() 在布局层和子路由间共享数据——前者是"一次性带过去"，后者是"父路由持续提供给所有子页面"，场景不同但目的类似，都是绕开 URL 传数据。
  • 认证守卫 PrivateRoute：本文開头提到的"记录来源页跳回"正是认证守卫的核心技巧——守卫里 <Navigate to="/login" state={{ from: location }} /> 把被拦截的路径存进 state，登录页读出来跳回去，这是 state 传参在真实项目里最常见的落地场景。
  • Navigate 重定向：<Navigate> 组件本身就支持 state 属性，和 navigate(path, { state }) 是同一套机制的两种触发方式（声明式 vs 命令式）——理解了 useNavigate 传 state，直接就能读懂 Navigate 组件的 state 用法，两者语义完全一致。
  • action 处理表单提交：action 完成后用 redirect() 跳转，这是纯服务端风格的重定向，不经过组件也就没有 state 传参的概念——如果提交后想带点"提示信息"给下一页，只能靠 loader 重新查询数据或额外的状态管理，这也是数据路由模式和传统 state 传参思路的一个分界点。
