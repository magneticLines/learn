动态段与 useParams：一条路由服务无数详情页

用户详情、文章详情、商品详情……每个 id 一个页面，你不可能给每个 id 都写一条 Route。动态路由段 :id 让一条规则匹配所有 id，再用 useParams 读出具体是哪个。这是路由参数最核心的用法。

问题:详情页的 id 是无穷多的

你有 /users/1、/users/2、/users/9527……它们用同一个 UserDetail 组件渲染，只是数据不同。总不能写几万条 Route。解决办法是在 path 里用 动态段（用 : 开头的占位符）：

用 :id 声明动态段——一条规则匹配所有 id
<Routes>
  <Route path="/users" element={<UserList />} />
  {/* :id 是占位符，匹配这一段的任意值 */}
  <Route path="/users/:id" element={<UserDetail />} />
</Routes>
// /users/1、/users/42、/users/abc 全部命中 UserDetail

:id 里的 id 是参数名，你随便起（:userId、:slug 都行），后面用这个名字取值。一条 path 里可以有多个动态段，如 /posts/:postId/comments/:commentId。

useParams:读出动态段的实际值

在被渲染的组件里，用 useParams() 拿到一个对象，键就是你在 path 里定义的参数名：

UserDetail.jsx——典型详情页
import { useParams } from 'react-router-dom';
import { useEffect, useState } from 'react';

function UserDetail() {
  // 键名 id 对应 path 里的 :id
  const { id } = useParams();
  const [user, setUser] = useState(null);

  // id 变了要重新拉数据，所以把 id 放进依赖数组
  useEffect(() => {
    fetch(`/api/users/${id}`)
      .then(r => r.json())
      .then(setUser);
  }, [id]);

  if (!user) return <p>加载中...</p>;
  return <h1>{user.name}</h1>;
}

头号陷阱:useParams 取出的值永远是字符串! URL 本质是文本，所以 /users/42 里的 id 是字符串 '42'，不是数字 42。这会引发一连串隐蔽 bug：

id === 42 → false（'42' 严格不等于 42）
id + 1 → '421'（字符串拼接，不是 43）

要当数字用，先转换：const idNum = Number(id) 或 parseInt(id, 10)。这是面试和实战里都极高频的坑。

多个动态段

一条 path 可以有多个参数
// 路由定义
<Route path="/posts/:postId/comments/:commentId" element={<Comment />} />

// 组件里一次性解构出来
const { postId, commentId } = useParams();
// URL /posts/7/comments/99 → { postId: '7', commentId: '99' }
用 id 做 key 强制重新挂载。 从 /users/1 跳到 /users/2 时，React 复用同一个 UserDetail 实例，只是 id 变了——所以必须靠 useEffect 的依赖数组 [id] 触发重新取数。如果组件内部状态复杂、想彻底"重来一遍"，可以给它加 key={id}（如 <UserDetail key={id} />），id 变化时 React 会卸载旧实例、挂载新实例，所有内部 state 重置。这是处理详情页切换的一个实用技巧。

如何跳转到带参数的详情页

拼接 URL 即可，Link 和 navigate 都行
// 声明式：列表里每一项
<Link to={`/users/${user.id}`}>{user.name}</Link>

// 编程式
navigate(`/users/${user.id}`);

小结

  • 动态段 :id 让一条 Route 匹配无穷多个具体路径，是详情页的标准做法。
  • 参数名自定义（:id/:slug），组件里用 useParams() 按同名键读取，支持多个动态段。
  • useParams 的值永远是字符串，要当数字用必须 Number()/parseInt()——严格相等和加法都会出坑。
  • id 变化时用 useEffect([id]) 重新取数；状态复杂时可用 key={id} 强制重新挂载。
  • 跳转时直接把 id 拼进 URL 字符串（to={`/users/${id}`}）。

与其他知识点的联系

  • Routes 与 Route(第78篇):动态段 :id 正是在第78篇讲的 path 里声明的,而"静态段优先于动态段"的最佳匹配规则,直接决定了 /users/:id 和 /users/new 这类路由能否正确共存——本篇是第78篇匹配规则的一次具体应用。
  • Link 与 useNavigate(第79–80篇):本篇"如何跳转到带参数的详情页"一节,分别用 Link 的 to 和 navigate() 拼接带 id 的 URL 字符串——这是这两个 API 最常见的实战用法,把前两篇的抽象跳转能力落到了"详情页"这个具体场景上。
  • useEffect 依赖数组(Hooks 系列):本篇强调 id 变化要放进 useEffect 的依赖数组才能触发重新取数,这与前面 useCallback/useMemo 篇讲的"依赖数组决定何时重新计算"是同一套机制——依赖数组写漏,这里的后果是"切换详情页数据不更新",本质和那边的闭包陷阱同源。
  • key 强制重新挂载:本篇提到的 key={id} 技巧,利用的是 React 用 key 判断元素身份的机制——key 变化时 React 认为这是"全新元素"而非"同一元素更新",从而卸载旧实例、挂载新实例,这个原理在列表渲染的 key 用法里同样适用,只是这里用在了单个详情页组件上。
