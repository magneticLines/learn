Vitest 单元测试基础

写了 150 篇组件和 Hook,却从没写过一个测试——这是很多学习者的真实状态。测试不是"锦上添花",而是让你敢重构、敢升级依赖的底气。Vitest 是 Vite 生态原生的测试框架,这一篇从"为什么选它"讲到最基本的断言写法。

为什么 Vite 项目选 Vitest,而不是 Jest

Jest 是 React 生态最老牌的测试框架,但它诞生于 CRA(Create React App)时代,底层用的是自己的转换管线(Babel)。如果你的项目已经用 Vite 构建(本系列从工程化篇开始就默认这个前提),引入 Jest 意味着要再搭一套独立的编译配置——单独配置 babel-jest 或 ts-jest 来处理 TypeScript/JSX,还要手动对齐路径别名、CSS Modules 等 Vite already 处理好的东西。两套编译管线并存,配置经常"测试环境和真实构建不一致"。

Vitest 的核心思路是:直接复用 Vite 的配置和转换插件。你在 vite.config.ts 里配好的路径别名、TS 支持、JSX 转换,测试时原样生效,不用再写一份。加上它用 esbuild/SWC 做转换、原生支持 ESM,冷启动和监听模式的速度都明显快于 Jest 的 Node 转换管线。

Jest
诞生早,生态最成熟
需要额外配置 babel/ts-jest
CJS 优先,ESM 支持较别扭
和 Vite 是两套转换管线
在非 Vite 项目(如 Node 后端)更常见
Vitest
复用 Vite 配置,零额外编译配置
原生 ESM、原生 TS/JSX
基于 esbuild,启动和热更快
API 与 Jest 高度兼容,几乎无缝迁移
Vite 项目(含 React/Vue)首选
API 兼容意味着什么? Vitest 的 describe / it / expect / vi.fn() 等 API 基本是 Jest 的镜像(vi 对应 Jest 的 jest 对象)。这意味着你之前学的、网上搜到的大量 Jest 用法和断言写法可以直接照搬,几乎没有额外学习成本,这也是它能快速取代 Jest 成为 Vite 生态默认选择的重要原因。

安装与最小配置

安装依赖
npm install -D vitest

大部分 Vite 项目不需要额外配置就能跑起来,因为 Vitest 会自动读取项目根目录的 vite.config.ts。如果想让配置更清晰,也可以在其中加一个 test 字段:

vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig({
  test: {
    globals: true,       // 开启后 describe/it/expect 不用逐个 import
    environment: 'jsdom', // 模拟浏览器 DOM 环境,测组件时必需
  },
});

最基本的写法:测一个纯函数

测试的核心结构永远是三层:describe(描述一组相关测试,可嵌套)、it(描述一个具体场景,也可写 test)、expect(断言实际结果符合预期)。先从最简单的纯函数开始建立直觉:

sum.ts —— 被测试的函数
export function sum(a: number, b: number) {
  return a + b;
}
sum.test.ts —— 测试文件
import { describe, it, expect } from 'vitest';
import { sum } from './sum';

describe('sum', () => {
  it('两个正数相加', () => {
    expect(sum(1, 2)).toBe(3);
  });

  it('负数相加', () => {
    expect(sum(-1, -1)).toBe(-2);
  });
});

跑起来只需要在 package.json 加一条脚本 "test": "vitest",默认进入监听模式——改动文件就自动重跑相关测试,这也是它比 CI 里一次性跑完的模式更适合日常开发的地方。

常用断言速查

断言用途
toBe(x)全等比较(基本类型,或同一引用)
toEqual(x)深度比较(对象/数组内容相同即可)
toBeTruthy() / toBeFalsy()断言真值/假值
toContain(x)数组/字符串包含某项
toThrow()断言函数调用会抛出异常
toHaveBeenCalledWith(x)配合 vi.fn() 断言 mock 函数被特定参数调用过
toBe vs toEqual 是高频踩坑点。 toBe 用 Object.is 做比较,两个内容相同但不是同一引用的对象/数组用 toBe 会失败;必须用 toEqual 做深度比较。反过来,比较基本类型(数字、字符串、布尔)时习惯上仍用 toBe,更直接地表达"值相等"的语义。
只是开始。 这一篇只测了纯函数,还没涉及 React 组件本身——组件测试要处理渲染、用户交互、异步更新,规则不太一样。下一篇的 React Testing Library 就是在 Vitest 这套跑分和断言体系之上,专门解决"怎么测组件"的问题。

小结

  • Vitest 是 Vite 生态原生的测试框架,直接复用 vite.config.ts 里已有的编译配置,不需要像 Jest 那样额外配 babel/ts-jest。
  • 它基于 esbuild,启动和监听模式的速度明显快于 Jest 的 Node 转换管线。
  • API 与 Jest 高度兼容(describe/it/expect,vi 对应 jest),迁移成本极低,网上大量 Jest 教程可以直接套用。
  • 测组件需要 environment: 'jsdom' 模拟浏览器 DOM,测纯函数则不需要。
  • toBe 是全等比较,toEqual 是深度比较——对象/数组内容比较务必用后者。

常见陷阱

  • 忘记配置 environment: 'jsdom'，一测组件就报错：Vitest 默认的测试环境是 node，没有 document、window 这些浏览器 API。测纯函数（如本篇的 sum）没问题，但只要涉及渲染组件、操作 DOM，就会报"document is not defined"。必须在 vite.config.ts 里显式加 environment: 'jsdom'（或用 // @vitest-environment jsdom 注释单独指定某个测试文件）。
  • 忘记 globals: true 却不 import，导致 describe is not defined：不开启 globals 选项时，describe/it/expect 都需要从 'vitest' 显式 import，这是很多从 Jest 迁移过来的人第一次运行就报错的原因——Jest 默认全局注入，Vitest 默认不注入。
  • toBe 和 toEqual 混用，测试"假通过"或"真失败"：本篇提到的坑值得再强调一遍：toBe 底层用 Object.is，比较对象/数组时比的是引用地址，两个内容完全一样但分别 new 出来的对象用 toBe 永远失败；反过来如果该用 toBe（比较基本类型）却用了 toEqual，测试意图会变得含糊，未来出现类型错误（比如把数字错写成字符串）时 toEqual 可能不会捕捉到。
  • 异步测试忘记 await，测试提前"假成功"结束：如果被测函数是异步的（返回 Promise，或者组件里有 setTimeout/网络请求触发的状态更新），测试函数体内如果没有 await 断言语句就直接 return，Vitest 会在异步操作完成前就判定这个 it 通过了——因为它根本没等。正确做法是 it('...', async () => { await expect(fn()).resolves.toBe(x); }) 或搭配 waitFor（组件测试库提供）。
  • 用 vi.fn() mock 后忘记在每个测试间重置调用记录：vi.fn() 创建的 mock 函数会持续累积调用记录（mock.calls），如果在多个 it 之间复用同一个 mock 实例却不清理，前一个测试的调用次数会污染后一个测试的断言。应该在 beforeEach 里调用 vi.clearAllMocks() 或每次重新创建 mock。

面试高频考点

  1. Vitest 为什么比 Jest 更适合 Vite 项目？原理是什么？—— 核心在于转换管线是否复用。Jest 需要独立的转换器（Babel/ts-jest）来处理 TS/JSX，和 Vite 构建时用的 esbuild/SWC 是两套完全不同的编译逻辑，容易出现"测试环境能跑、生产构建报错"或反过来的不一致。Vitest 直接吃 Vite 的 vite.config.ts，路径别名、JSX 转换、CSS Modules 全部原样复用，本质上是把"用什么跑测试"和"用什么跑生产"统一成了同一套工具链。
  2. toBe 和 toEqual 底层分别用什么比较逻辑？—— toBe 基于 Object.is（近似严格相等 ===，但能正确处理 NaN 和 ±0 的边界情况），比较对象/数组时看的是引用是否相同。toEqual 做递归的结构比较（deep equality），逐个字段比较值，不关心是不是同一个引用。这也是为什么测基本类型习惯用 toBe（表达"值语义"），测对象/数组必须用 toEqual（表达"结构语义"）。
  3. 为什么测试要用 mock（比如 vi.fn()），而不是直接调用真实依赖？—— 单元测试的核心诉求是"隔离被测单元、结果可复现"。如果测试直接依赖真实的网络请求、数据库、定时器，测试结果会受外部环境影响（网络波动、时间不确定），而且慢、不稳定。mock 把这些外部依赖替换成可控的假实现，你能精确断言"这个函数是否被调用、被什么参数调用了几次"，同时让测试运行速度和确定性都大幅提升。
  4. 监听模式（watch mode）相比 CI 里一次性运行，解决了什么问题？—— 监听模式在文件改动时只重新运行受影响的测试，反馈速度接近实时，适合开发过程中频繁试错；CI 环境需要的是一次性、确定性的全量运行结果用于门禁判断。两者不是相互替代关系，而是同一套测试代码在不同场景下的两种运行模式——这也是为什么 package.json 里通常会分别配 "test": "vitest"（本地监听）和 "test:ci": "vitest run"（CI 单次运行）。

与其他知识点的联系

  • React Testing Library（下一篇 151）：Vitest 提供的是"运行测试、做断言"这套跑分基座，但它本身不知道怎么渲染 React 组件、怎么模拟用户点击。RTL 是搭在 Vitest 之上的组件测试工具，两者不是竞争关系——学完这一篇的 describe/it/expect 三层结构和 jsdom 环境配置，下一篇才能专注讲"怎么测组件"本身，而不用重复讲测试框架基础。
  • MSW（152）：组件测试经常涉及"发请求拿数据后渲染"的场景,而 Vitest 本身不管网络层的事。vi.fn() 能 mock 单个函数调用,但更贴近真实使用场景的是在网络层拦截——这正是 MSW 要解决的问题,它和 Vitest 组合使用,而不是互相替代。
  • useState / state 异步更新（19-21篇）：测试组件时最常踩的坑是"状态更新是异步的",如果测试代码里 setCount 之后立刻断言,和本系列 21 篇讲的"state 是快照"是同一个原理——测试也要等 React 完成重新渲染后才能断言到新值,这也是为什么组件测试库要提供 waitFor/findBy* 这类异步查询 API。
  • Vite 工程化（本系列构建工具篇）：Vitest 之所以"零额外配置"就能跑起来,根源在于本系列前面讲过的 Vite 插件体系和 vite.config.ts 统一配置思路——理解了 Vite 怎么处理路径别名、JSX 转换,才能理解 Vitest 为什么能直接复用这套东西而不需要像 Jest 那样另起一套编译管线。
  • TypeScript 类型系统：本篇代码示例用 TS 写测试文件,断言函数（如 toBe）的参数类型会根据被测函数的返回值类型做检查,这也是"测试即文档"的一种体现——类型错误在写测试的阶段就能被 IDE 提前捕捉到,不用等运行时才发现参数类型不对。
