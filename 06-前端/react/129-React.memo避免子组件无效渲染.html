React.memo 避免子组件无效渲染

第 123 篇讲过触发重渲染的三种原因之一是"父组件重渲染,子组件跟着重渲染"——哪怕子组件收到的 props 其实一模一样。React.memo 就是为这个场景生的:它让组件在 props 没变时跳过重渲染,直接复用上一次的渲染结果。

原理:对 props 做浅比较

memo 包裹一个组件后,React 在父组件重渲染、准备渲染这个子组件之前,会先拿这次的 props 和上一次的 props 逐个字段做浅比较(默认用 Object.is,和 useMemo/useState 判断依赖是否变化的比较方式一致)。如果每个字段都相等,React 直接跳过这次渲染,复用上次的渲染输出(虚拟 DOM);只要有一个字段不同,就正常重新渲染。

浅比较是什么意思? 只比较第一层。{ a: 1 } === { a: 1 } 是 false(两个不同的对象引用),但 1 === 1 是 true。所以数字、字符串、布尔值这类基本类型作为 props 传给 memo 组件很安全;对象、数组、函数作为 props 传,只要引用变了(哪怕内容一样),浅比较也会判定"变了"。这是下面陷阱的根源。

反例:父组件重渲染拖累无关的子组件

没有 memo 时,父组件任何一次重渲染,默认会级联渲染所有子组件——不管子组件的 props 有没有变化。

反例:count 变化,ExpensiveList 明明没变化也跟着重渲染
function ExpensiveList({ items }) {
  console.log('ExpensiveList 渲染了');  // 每次父组件渲染都会打印
  return (
    <ul>
      {items.map((item) => <li key={item.id}>{item.text}</li>)}
    </ul>
  );
}

function App() {
  const [count, setCount] = useState(0);
  const [items] = useState([{ id: 1, text: '学习 React' }]);  // 从不变化
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {/* count 变化只影响按钮文字,items 从没变过,但每次点击 ExpensiveList 都重渲染一遍 */}
      <ExpensiveList items={items} />
    </>
  );
}

问题:点击按钮只是想让 count 变化、按钮文字更新,但 ExpensiveList 的 items 从头到尾没变,却因为"父组件重渲染 → 默认级联渲染所有子组件"这条规则被迫跟着重渲染。如果 ExpensiveList 渲染成本高(列表很长、内部有复杂计算),这就是纯粹的性能浪费。

正例:用 memo 包裹后跳过无效渲染

正例:memo 包裹,props 没变就跳过
const ExpensiveList = memo(function ExpensiveList({ items }) {
  console.log('ExpensiveList 渲染了');  // 现在只在 items 真正变化时打印
  return (
    <ul>
      {items.map((item) => <li key={item.id}>{item.text}</li>)}
    </ul>
  );
});

// App 组件不用改,count 变化时 React 发现 items 引用没变 → 跳过 ExpensiveList 的重渲染
为什么这次生效了? 上面例子里 items 是用 useState 存的、从未 setItems,所以它的引用在多次渲染间保持不变。memo 浅比较发现 items 前后引用相同 → 跳过重渲染。这也说明 memo 本身没有魔法,它依赖"引用是否变化"这个朴素规则生效。

陷阱:内联对象 / 数组 / 函数让 memo 形同虚设

初学者最常踩的坑: 如果传给 memo 组件的 props 是在父组件渲染函数体内现场创建的对象、数组或函数字面量,那么每次父组件渲染都会生成一个全新的引用——即便内容看起来一样,浅比较也会判定"变了",memo 完全不起作用。
反例:内联对象 / 内联函数让 memo 失效
function App() {
  const [count, setCount] = useState(0);
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {/* style={{ color: 'red' }} 每次渲染都是新对象,即便内容永远一样 */}
      <ExpensiveList style={{ color: 'red' }} onSelect={(id) => console.log(id)} />
      {/* ↑ style 和 onSelect 都是内联字面量,每次都是新引用,memo 浅比较必然判定"变了" */}
    </>
  );
}

解决办法留给后面两篇:对象/数组用 useMemo 缓存引用,函数用 useCallback 缓存引用,让它们在依赖不变时保持同一个引用,memo 才能真正生效。

自定义比较函数

默认的浅比较对多数场景够用,但如果某个 prop 是深层对象、你确定它的深层内容才是判断依据,可以传第二个参数自定义比较逻辑:

memo 第二个参数:自定义 props 比较函数
const UserCard = memo(
  function UserCard({ user }) { /* ... */ },
  (prevProps, nextProps) => prevProps.user.id === nextProps.user.id
  // 返回 true 表示"相等,跳过渲染";只要 user.id 没变就复用,哪怕 user 对象引用变了
);
返回值方向和 shouldComponentUpdate 相反: 自定义比较函数返回 true 表示 props 相等,跳过渲染;返回 false 才会重新渲染。class 组件的 shouldComponentUpdate 是反过来的(返回 true 表示要更新),容易记混,是面试和代码审查里的高频坑。

什么时候值得用 memo

场景是否值得
组件渲染成本高(大列表、复杂计算、深层子树)值得——跳过一次渲染能省下真实的开销
组件经常因为"父组件重渲染但 props 不变"而被迫重渲染值得
组件本身很简单(几个 div、几行文本)不值得——浅比较本身也有成本,简单组件重渲染更便宜
props 几乎每次都真的会变不值得——memo 的比较总是失败,白白多一层开销
常见误区: 不要看到"性能优化"三个字就把所有组件都套上 memo。浅比较本身不是免费的,对一个本来渲染开销就很小的组件加 memo,多做的比较工作可能比省下的渲染工作还多,纯属"为了优化而优化"。先用 React DevTools Profiler 找到真正慢的组件,再有针对性地加 memo,而不是无脑全局套用。

小结

  • React.memo 包裹组件后,React 会在重渲染前对新旧 props 做浅比较(Object.is),相同则跳过渲染、复用上次结果。
  • 它解决的是第 123 篇提到的"父组件重渲染,子组件被迫级联重渲染"的问题,但只在子组件 props 实际没变时才生效。
  • 浅比较只比第一层:内联对象、数组、函数字面量每次渲染都是新引用,会让 memo 直接失效,这是最常见的踩坑点。
  • 第二个参数可传自定义比较函数,返回 true 表示"相等、跳过渲染"——方向和 shouldComponentUpdate 相反,别记混。
  • 只对渲染成本真正高、且 props 确实经常"看起来变了但其实没变"的组件加 memo;简单组件或 props 每次都真变的组件加了反而更亏。
