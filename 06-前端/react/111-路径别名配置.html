路径别名配置

import Button from '../../../../components/Button'——这种"数路径深度"式的 import,是每个中大型前端项目的通病。文件挪个目录,一堆相对路径全断;新建文件时也很容易多写或少写一个 ../。路径别名(如 @/)把这个问题彻底解决,但它需要两处配置同时生效才算真正配好,这也是新手最容易踩的坑。

为什么需要别名:相对路径地狱

典型的相对路径地狱
// src/features/order/pages/OrderDetailPage.tsx
import { Button } from '../../../components/Button';
import { formatDate } from '../../../utils/date';
import { useAuth } from '../../../../hooks/useAuth';
// 这个文件如果被挪到别的目录层级,上面三行全部要手动改

用别名之后,同样的 import 变成和文件在哪一层完全无关的绝对路径:

用 @/ 别名后
import { Button } from '@/components/Button';
import { formatDate } from '@/utils/date';
import { useAuth } from '@/hooks/useAuth';
// 无论这个文件本身在哪一层,这三行永远不用改
约定俗成: @/ 通常指向 src/ 根目录,这不是 Vite 或 React 的硬性规定,只是社区约定(Vue CLI、Nuxt 等生态也用同样的符号)。你也可以配置成 ~/ 或其他符号,团队统一即可。

两处配置,缺一不可

这是本篇的核心陷阱来源:别名要同时在两个完全独立的系统里配置,因为它们各自负责不同的职责,谁也替代不了谁。

vite.config.ts
resolve.alias
+
tsconfig.json
compilerOptions.paths

第一处:vite.config.ts 里的 resolve.alias

这一处决定运行/构建时,Vite 把 @/xxx 实际解析成哪个真实文件路径。没有这一步,浏览器/构建工具根本不知道 @ 是什么,会直接报"模块找不到"。

vite.config.ts:配置运行时的别名解析
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'node:path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),  // @ 指向 src 目录的绝对文件系统路径
    },
  },
});

第二处:tsconfig.json 里的 paths

这一处决定编辑器(TypeScript 语言服务)怎么理解 @/xxx——包括类型检查、"跳转到定义"、自动补全。TypeScript 编译器和 Vite 是两套完全独立的工具链,Vite 的配置它读不到,必须单独告诉 TS 一遍。

tsconfig.json:配置编辑器/类型检查的别名解析
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
陷阱:只配了一处的两种典型症状。

只配 tsconfig,没配 vite.config: 编辑器里代码提示正常、也不报红,你以为万事大吉;结果 npm run dev 一跑就报 "Failed to resolve import"——因为 TS 只负责类型检查,真正解析模块路径跑起来的是 Vite,它压根不知道 @ 是什么。

只配 vite.config,没配 tsconfig: 反过来,代码能正常跑起来,但编辑器里每一行 @/xxx 的 import 都飘红报错、补全失效、"跳转到定义"用不了——因为 TS 语言服务读的是 tsconfig,不知道 Vite 私底下做了别名转换。
联系到之前学的内容: 这种"同一个概念要在两套独立工具链里各配一份"的情况在前端工程化里很常见——就像上一篇的环境变量,Vite 的 VITE_ 前缀规则也是 Vite 自己的一套约定,和 Node 的 process.env 是两套体系。记住一个原则能少踩很多坑:凡是"编译/打包工具"和"类型检查工具"分离的项目,涉及路径解析的配置往往要两边都改。

验证配置是否生效

现象说明
编辑器不报红 + npm run dev 正常跑两处都配好了
编辑器不报红,但 npm run dev 报 "Failed to resolve import"漏配 vite.config.ts 的 resolve.alias
npm run dev 正常,但编辑器里 @/xxx 全部飘红漏配 tsconfig.json 的 paths

小结

  • 路径别名(如 @/)把相对路径的"数层级"问题变成和文件位置无关的绝对路径,文件挪动不再需要改一堆 import。
  • @/ 通常约定指向 src/ 根目录,符号本身可自定义,团队统一即可。
  • 别名要配两处:vite.config.ts 的 resolve.alias(决定运行/构建时怎么解析)、tsconfig.json 的 compilerOptions.paths(决定编辑器类型检查和跳转)。
  • 两套系统相互独立,谁也读不到对方的配置,少配一处就会出现"编辑器正常但跑不起来"或"跑得起来但编辑器全飘红"两种典型症状之一。
  • 这种"编译工具链 + 类型检查工具链分别配置"的模式,在前端工程化里很常见,遇到路径解析类问题先检查是不是漏了其中一边。
