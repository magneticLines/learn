render props:把"怎么渲染"交给使用者

children 让你传"静态内容"。但如果一个组件想共享自己的内部数据(比如鼠标位置、加载状态),又不替使用者决定怎么画,该怎么办?render props 给出答案:把一个返回 JSX 的函数当作 prop 传进去。

核心:props 值是一个"渲染函数"

普通 prop 传的是数据;render prop 传的是函数。组件在内部调用这个函数,把自己的状态作为参数喂给它,函数返回的 JSX 就是最终要渲染的东西。渲染的"决定权"交回给使用者,组件只负责"提供数据"。

经典例子:追踪鼠标位置的可复用组件
// 这个组件只做一件事:追踪鼠标,不决定怎么显示
function MouseTracker({ render }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      // 关键:调用 render 函数,把内部状态 pos 交出去
      {render(pos)}
    </div>
  );
}

// 使用者决定"拿到 pos 后怎么画"——同样的数据,不同的呈现
<MouseTracker render={pos => <p>坐标:{pos.x}, {pos.y}</p>} />
<MouseTracker render={pos => <Cat x={pos.x} y={pos.y} />} />

它和 children 的区别:静态 vs 动态
children 是"预先写好的 JSX",组件只是把它放到某个位置——组件不能给它传数据。render props 传的是"一个待调用的函数",组件能把内部状态作为参数喂进去。一句话:children 共享"结构",render props 共享"逻辑/数据"。

变体:children as a function(children 作为函数)

render 只是个约定的 prop 名。既然 children 也是 prop,那它同样可以是一个函数,这就是更常见的写法——children as function:

把函数写在标签之间,组件调用 children
function MouseTracker({ children }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  // children 是函数,像调用 render 一样调用它
  return <div onMouseMove={...}>{children(pos)}</div>;
}

// 使用:标签之间放一个函数(注意是 {pos => ...},不是 JSX)
<MouseTracker>
  {pos => <p>坐标 {pos.x}, {pos.y}</p>}
</MouseTracker>

它解决的问题:逻辑复用

render props 在 Hooks 之前,是解决"多个组件共享同一段有状态逻辑"的主力方案之一(另一个是高阶组件 HOC)。鼠标追踪、数据订阅、鼠标悬停状态……这些逻辑写一次,靠 render props 让任意 UI 复用。

陷阱:嵌套地狱(callback hell 的 JSX 版)
多个 render props 组件层层嵌套时,会形成深层缩进的"金字塔",可读性极差:
<A>{a => <B>{b => <C>{c => ...}</C>}</B>}</A>
这正是 Hooks 想要解决的痛点之一。

和自定义 Hook 的关系:Hook 出现后它退居二线

render props(旧)
<MouseTracker>
  {pos => <Cat {...pos}/>}
</MouseTracker>
// 复用逻辑要嵌套组件
// 多个叠加就是金字塔
自定义 Hook(新)
function App() {
  const pos = useMouse();
  // 一行拿到逻辑,扁平
  return <Cat {...pos} />;
}
// 无嵌套,可任意组合多个

2019 年 Hooks 出现后,逻辑复用这件事被自定义 Hook 干得又扁平又干净——它把"有状态逻辑"从"渲染结构"里彻底剥离,不再需要靠嵌套组件传递。所以如今:

  • 纯逻辑复用(不涉及具体 DOM 结构):优先用自定义 Hook。
  • render props 仍有一席之地:当复用的东西天然和"渲染结构/DOM 节点"绑定时,比如某些图表、虚拟列表、拖拽库(它们要控制一个具体的容器元素),render props / children as function 依然常见。
面试高频:被问"render props / HOC / 自定义 Hook 的关系"时,标准回答:三者都是逻辑复用的手段;render props 和 HOC 是 Hooks 之前的方案,各有嵌套/包裹的缺点;Hooks 出现后成为逻辑复用的首选,更扁平、更直观,但 render props 在"逻辑与渲染结构强绑定"的场景仍不可替代。

小结

  • render props:把一个返回 JSX 的函数作为 prop 传入,组件调用它并把内部状态当参数喂进去。
  • 与 children 的区别:children 共享"静态结构",render props 共享"动态数据/逻辑"。
  • 变体 children as function:把函数直接写在标签之间,更常见。
  • 主要用于逻辑复用;缺点是多层嵌套会形成"金字塔"。
  • Hooks 出现后,纯逻辑复用优先用自定义 Hook;render props 仍适用于逻辑与 DOM 结构强绑定的场景。
