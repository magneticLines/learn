Props 解构：更简洁的接收方式

上一节每次访问 props 都要写 props.xxx，就像每次叫人都要说全名+姓氏。解构让你把常用的 props 在入口就"拆包"成独立变量，后续直接用变量名——代码干净一倍。

为什么要解构：消灭重复的 props. 前缀

想象一个卡片组件有 5 个 props，不解构时函数体里每处用到都要带前缀：

不解构——props. 噪声严重
function ProductCard(props) {
  return (
    <div>
      {/* 每一行都在重复 props.，真正的信息被淹没了 */}
      <h2>{props.name}</h2>
      <p>{props.price} 元</p>
      <p>{props.description}</p>
      {props.inStock && <span>有货</span>}
      <button onClick={props.onBuy}>购买</button>
    </div>
  );
}

解构后，函数体里只剩真正有意义的变量名，阅读时眼睛不需要"跳过" props.：

解构后——干净，变量名即含义
function ProductCard({ name, price, description, inStock, onBuy }) {
  return (
    <div>
      {/* 直接用变量名，没有噪声，代码意图一目了然 */}
      <h2>{name}</h2>
      <p>{price} 元</p>
      <p>{description}</p>
      {inStock && <span>有货</span>}
      <button onClick={onBuy}>购买</button>
    </div>
  );
}

方式一：参数位置解构（推荐写法）

直接在函数参数位置写解构——这是 React 社区的标准写法，几乎所有项目都这么做。

参数位置解构——最常用
// 大括号 { } 放在参数位置，等号后面的 props 对象被"就地拆开"
function Avatar({ src, size, alt }) {
  return <img src={src} width={size} height={size} alt={alt} />;
}

// 这和以下写法完全等价，只是 JavaScript 的解构语法而已：
// function Avatar(props) {
//   const { src, size, alt } = props;
//   ...同样的 JSX
// }
本质是 ES6 解构，不是 React 特有的：参数位置解构是标准 JavaScript 特性。React 社区只是约定把它用在这里，你在普通函数里也能这么写：function greet({ name, age }) {...}。

方式二：函数体内解构（也可以，但不常用）

在函数体第一行把 props 对象解构，效果完全一样，只是把动作推迟到了函数体内。

函数体内解构
function Avatar(props) {
  // 需要在使用前手动解构一下
  const { src, size, alt } = props;

  return <img src={src} width={size} height={size} alt={alt} />;
}

// 什么时候用这种写法？
// 当你需要先对 props 整体做某些操作，再解构时：
function UserCard(props) {
  // 比如需要把 props 传给某个工具函数
  logRenderProps(props);
  const { name, age } = props;
  // ...
}

用 ...rest 收集剩余 Props（透传场景）

封装组件时，有些 props 是给自己用的，其余的需要"透传"给底层元素。用展开运算符 ...rest 可以把未命名的 props 全部收进一个对象：

...rest 收集剩余 props 并透传
// 封装一个带 loading 状态的按钮
// loading 是我们自己用的，其余 props（如 onClick, disabled, type）透传给原生 button
function LoadingButton({ loading, label, ...rest }) {
  // rest = { onClick: fn, disabled: false, type: 'submit', ... }
  return (
    // 把 rest 全部展开传给原生 button，避免逐个转发
    <button {...rest} disabled={loading || rest.disabled}>
      {loading ? '加载中...' : label}
    </button>
  );
}

// 使用：onClick 和 type 会通过 ...rest 透传给 button
<LoadingButton
  loading={isSubmitting}
  label="提交"
  onClick={handleSubmit}
  type="submit"
/>
...rest 透传到原生元素要注意：如果 rest 里含有浏览器不认识的自定义属性（比如 isAdmin、onBuy），透传到原生 <div>、<button> 会产生浏览器警告。解决方案：先把自定义属性解构出来，只把原生 HTML 属性放进 rest。

两种写法完整对比

props.xxx 写法
function Card(props) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{props.desc}</p>
      <button
        onClick={props.onClick}
      >
        {props.btnText}
      </button>
    </div>
  );
}
解构写法（推荐）
function Card({
  title,
  desc,
  onClick,
  btnText
}) {
  return (
    <div>
      <h2>{title}</h2>
      <p>{desc}</p>
      <button onClick={onClick}>
        {btnText}
      </button>
    </div>
  );
}

两段代码输出完全一样。解构写法的优势：函数签名即文档，一眼看出这个组件需要哪些 props；函数体里没有噪声前缀，代码更易读。

本节核心：① 解构是 ES6 特性，React 只是习惯用它来接收 props；② 推荐在参数位置解构，函数签名即 props 文档；③ ...rest 收集剩余 props，适合封装透传场景；④ 两种写法完全等价，选解构是为了可读性，不是语法要求。
