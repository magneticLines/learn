Portal 使用场景:逃离父级的 CSS 牢笼

上一篇讲了 Portal 怎么用。这一篇回答那个更实际的问题:为什么 Modal、Tooltip、Toast 非用 Portal 不可?答案是两个 CSS 特性——overflow: hidden 的裁剪,和 z-index 的层叠上下文——会把浮层"关"在父容器里。

陷阱一:被 overflow: hidden 裁剪

假设 Tooltip 直接渲染在触发它的按钮旁边,而这个按钮外层有一个 overflow: hidden 的滚动容器(卡片、表格单元格很常见):

❌ 不用 Portal
.card {
  overflow: hidden;
  /* 卡片外的内容
     被无情裁掉 */
}

// Tooltip 是 card 的
// 子元素 → 超出卡片
// 的部分看不见了
✅ 用 Portal
createPortal(
  <Tooltip/>,
  document.body
)

// Tooltip 挂到 body
// 不再受 card 的
// overflow 约束
// 完整显示

overflow: hidden 会裁剪掉溢出父容器边界的所有子内容。浮层天生要"溢出"(下拉菜单要伸到卡片外、Tooltip 要飘在元素上方),一旦父级有这个属性,浮层就被切掉一半。Portal 把浮层挂到 body,让它彻底脱离这个裁剪容器。

陷阱二:z-index 的层叠上下文

更隐蔽的坑。你给 Modal 设了 z-index: 9999,信心满满以为它一定在最上层——结果还是被某个 z-index: 10 的元素盖住了。原因是层叠上下文(stacking context)。

关键认知:z-index 只在同一个层叠上下文内比较。当父元素创建了一个新的层叠上下文(比如设置了 position + z-index、transform、opacity < 1、filter 等),它内部所有子元素的 z-index 都被"封印"在这个上下文里。子元素的 z-index: 9999 只在父上下文内部有意义;一旦父上下文本身的层级低,子元素再高也翻不过身。

类比:公司内部你是 9999 号高管,但你所在的公司在整个集团里排名靠后,你再高也压不过排名靠前公司的普通员工。
典型的"z-index 失灵"现场
<div style={{ transform: 'translateZ(0)' }}>  // 创建了新层叠上下文!
  <Modal style={{ zIndex: 9999 }} />  // 9999 被封印在这个 div 里
</div>
<header style={{ zIndex: 10 }} />       // 反而盖住了 Modal!

// 用 Portal 挂到 body 顶层,Modal 就跟 header 处于同一上下文
// 9999 > 10,正常覆盖
createPortal(<Modal/>, document.body)

实战:一个完整的 Modal

Modal.jsx —— Portal + 遮罩 + Esc 关闭
import { createPortal } from 'react-dom';

function Modal({ open, onClose, children }) {
  useEffect(() => {
    const onKey = (e) => e.key === 'Escape' && onClose();
    window.addEventListener('keydown', onKey);
    return () => window.removeEventListener('keydown', onKey);
  }, [onClose]);

  if (!open) return null;

  return createPortal(
    // 点遮罩关闭;点内容区 stopPropagation 阻止穿透
    <div className="overlay" onClick={onClose}>
      <div className="dialog" onClick={(e) => e.stopPropagation()}>
        {children}
      </div>
    </div>,
    document.getElementById('modal-root')
  );
}
呼应上一篇的事件冒泡陷阱。这个 Modal 里 stopPropagation 拦的是 React 合成事件的冒泡。但如果你在祖先组件上挂了"点外部关闭"的逻辑,由于 Portal 事件按 React 树冒泡,点 Modal 内部也会冒到祖先——所以要么把关闭逻辑限定在 overlay 层(如上),要么用 event.target 判断是否点在浮层外。

什么时候用 / 不用 Portal

组件用 Portal?原因
Modal / Dialog是需覆盖全屏、逃离父级 overflow 与层叠上下文
Tooltip / Popover是要飘在元素外、避免被裁剪、精确定位
Toast / 全局通知是固定屏幕角落,与页面结构解耦
Dropdown 下拉菜单视情况在滚动容器/表格里易被裁剪时用
普通页面内容否没有溢出/层级问题,徒增复杂度

小结

  • Portal 存在的根本理由:让浮层逃离父级的 CSS 约束——overflow: hidden 的裁剪 与 z-index 层叠上下文的封印。
  • z-index: 9999 不管用时,先怀疑祖先是否创建了新的层叠上下文(transform / opacity / filter / position+z-index)。
  • Modal、Tooltip、Toast 是 Portal 的经典用武之地;普通内容别乱用。
  • 用了 Portal 记得回忆事件按 React 树冒泡的特性,处理好"点击外部关闭"。
