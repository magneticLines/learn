状态下移 / 内容提升模式

前三篇都是"事后补救"——先有重渲染问题,再用 memo/useMemo/useCallback 堵漏洞。这一篇讲一种更根本的思路:通过组件结构设计,从源头上让不该重渲染的部分根本不会被卷入重渲染,完全不用记忆化 API。

回顾:重渲染为什么会"波及"子树

第 123 篇讲过,某个组件的 state 变化会触发它自身重渲染,并且默认级联渲染它渲染出的所有子组件。如果这个 state 被放在树的高层(比如整个页面的根组件),那么每次它变化,整棵子树都要跟着走一遍渲染流程——哪怕子树里 99% 的部分和这个 state 毫无关系。

手段一:状态下移(state colocation)

思路很直接:state 应该放在离"使用它的 UI"最近的地方,而不是图方便都堆在顶层组件。如果只有页面里一个小小的输入框用到某个 state,就不要把它放在管理整个页面的根组件上。

反例:输入框的临时状态放在了顶层 Page 组件
function Page() {
  const [searchText, setSearchText] = useState('');  // 只有搜索框自己关心这个状态

  return (
    <div>
      <Header />
      <input value={searchText} onChange={(e) => setSearchText(e.target.value)} />
      {/* searchText 每变化一次,Sidebar、ProductGrid、Footer 全部跟着重渲染 */}
      <Sidebar />
      <ProductGrid />
      <Footer />
    </div>
  );
}
正例:把 searchText 下移到 SearchBox 内部
function SearchBox() {
  const [searchText, setSearchText] = useState('');  // 状态只属于这一个组件
  return <input value={searchText} onChange={(e) => setSearchText(e.target.value)} />;
}

function Page() {
  return (
    <div>
      <Header />
      <SearchBox />  {/* 输入只会让 SearchBox 自己重渲染,不影响其他兄弟组件 */}
      <Sidebar />
      <ProductGrid />
      <Footer />
    </div>
  );
}

Page 不再持有这个 state,自然也不会因为它变化而重渲染——重渲染的范围被"下移"到了真正需要它的那个小组件里,连 memo 都不需要用。

什么时候不能下移: 如果多个兄弟组件都需要读取或响应同一个 state(比如搜索框的值要同时影响 ProductGrid 的过滤结果),那这个 state 就必须"提升"到能同时管住它们的公共父组件——这是状态提升(lifting state up)的经典场景,和状态下移是同一个原则的两个方向:state 应该放在刚好能覆盖所有需要它的组件、又不多不少的那一层。

手段二:内容提升 / children 作为 props

有时候某个 state 就是必须待在高层(比如一个弹窗的开关状态放在 Page 上,因为 Page 要决定弹窗渲不渲染)。这种情况下状态下移用不上,但还有另一招:把不依赖这个 state 的子树,作为 children(或别的 prop)从更外层传进来。

反例:isOpen 变化,ExpensiveChart 作为内部子组件被迫跟着重渲染
function Page() {
  const [isOpen, setIsOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setIsOpen(true)}>打开</button>
      <Modal isOpen={isOpen}>
        {/* ExpensiveChart 写在 Modal 内部,Modal 一重渲染它就跟着重渲染,即便弹窗还没打开 */}
        <ExpensiveChart />
      </Modal>
    </div>
  );
}

这里 ExpensiveChart 是作为 Modal 的 JSX 子节点直接写在 Page 内部的,但因为它在书写结构上"属于" Modal,很多人会误以为把它拆成独立组件、甚至加 memo 才能解决问题。其实 React 有一个更直接的天然规则:

关键原理: <Modal><ExpensiveChart /></Modal> 在编译后等价于 Modal({ children: <ExpensiveChart /> })——ExpensiveChart 这个 JSX 元素对象,是在 Page 组件的渲染函数里创建的,不是在 Modal 内部创建的。只要 Page 本身没有重渲染(比如 isOpen 的变化被下移到了 Modal 自己内部管理),那么每次 Modal 重渲染时,它收到的 children prop 都是同一个已经创建好的元素引用,React 发现这个子树的输入没变,天然跳过对它的重渲染——不需要 memo,不需要浅比较,是 React 元素创建机制本身决定的。
正例:children 作为 props,ExpensiveChart 天然不受 isOpen 影响
function Modal({ isOpen, children }) {
  if (!isOpen) return null;
  return <div className="modal">{children}</div>;
}

function Page() {
  const [isOpen, setIsOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setIsOpen(true)}>打开</button>
      {/* ExpensiveChart 元素对象在这里创建;isOpen 变化只让 Page 和 Modal 重渲染,
         这个已创建好的元素作为 children 原样传入,不会被重新创建、也不会被重渲染 */}
      <Modal isOpen={isOpen}>
        <ExpensiveChart />
      </Modal>
    </div>
  );
}
容易误解的地方: 上面两段代码的 JSX 结构看起来一模一样——都是把 <ExpensiveChart /> 写在 <Modal> 标签内部。真正的区别在于 Page 组件本身有没有因为 isOpen 变化而重渲染:如果 isOpen 状态在 Page 里,Page 每次重渲染都会重新执行渲染函数、重新创建 <ExpensiveChart /> 这个元素,children 引用照样会变,提升不了任何效果。这个模式真正生效的前提,是持有 state 的组件和渲染 children 的组件是分开的两层,children 从更上层、不因这个 state 变化的组件传入。

三种手段的对比

memo / useMemo / useCallback
事后补救:先接受组件树结构,
用记忆化 API 在渲染前拦截。
需要额外心智负担(依赖数组、
浅比较的边界条件)。
状态下移 / 内容提升
事前设计:通过调整 state 的
归属层级和组件组合方式,
让不相关的子树天然不会被
卷入重渲染,零额外 API。
优先级建议: 遇到"重渲染范围太大"的问题,先看能不能通过状态下移或内容提升这种组件组合方式解决——它们零成本、零心智负担,而且是架构层面的根治;实在没法通过结构调整解决(比如子组件确实需要接收会变化的、开销大的 props),再考虑 memo/useMemo/useCallback 这套记忆化工具。这也是 React 官方文档推荐的顺序:先组合,后记忆化。

小结

  • 重渲染默认会级联到子树,如果触发它的 state 放在树的高层,会波及大量和这个 state 无关的组件。
  • 状态下移:把 state 放到真正使用它的最小组件里,让重渲染范围自然收窄,而不是图方便堆在顶层。
  • 内容提升 / children 作为 props:JSX 子节点在书写结构上"属于"内层组件,但它的元素对象其实是在外层创建的;只要持有 state 的组件和创建这个子节点的组件是分开的两层,子节点作为 children 传入时引用不变,天然跳过重渲染,不需要 memo。
  • 这个模式生效的关键前提是"持有 state 的组件本身没有因为这个 state 而重新创建 children"——JSX 结构长得一样不代表效果一样。
  • 优先用组件组合的方式(状态下移、内容提升)解决重渲染范围问题,这比到处加 memo/useMemo/useCallback 更简洁、也更不容易出错;记忆化工具留给结构调整解决不了的场景。

常见陷阱

  • 状态放得太高的惯性——很多人写组件时习惯"状态先放在最外层再说",觉得这样以后万一别的组件要用会方便。这其实是过度设计:state 提前放在高层不会带来任何好处,只会让这个 state 变化时白白拖着一整棵无关子树重渲染。正确做法是反过来——先把 state 放在离使用它最近的组件里,等真的出现"多个兄弟组件都要用"的需求时,再往上提升,而不是提前占位。
  • 下移了状态却没搬对内容——把 searchText 这类 state 下移到 SearchBox 内部,却忘了检查 Page 里还有没有其他"看起来和这个 state 无关但其实被拖下水"的逻辑(比如 Page 上还留着一段依赖 searchText 做副作用的 useEffect)。状态下移不是简单地剪切 useState 那一行代码,而是要把所有只服务于这个 state 的逻辑和 UI 一起搬过去,否则优化没有真正生效,Page 该重渲染还是会重渲染。
  • 误以为 memo/useMemo 能达到同样效果——看到 ExpensiveChart 重渲染,第一反应是给它包一层 memo,或者用 useMemo 缓存它的元素。这样确实能"堵住"问题,但属于文中说的"事后补救",还得维护依赖数组、操心浅比较的边界条件。如果本质原因是组件结构没设计好(state 和不相关的子树耦合在同一层),那更省心的办法是用状态下移或内容提升从结构上解决,根本不需要引入记忆化 API 的心智负担。
  • 只改 JSX 写法,没搬状态归属——照抄"children 作为 props"的写法,把 <ExpensiveChart /> 写在 <Modal> 标签里,但 isOpen 这个 state 依然留在同一个组件里管理。文中已经点破:这个模式生效的前提是持有 state 的组件和创建 children 的组件必须是分开的两层。如果 Page 自己既管 isOpen 又渲染 Modal 和 ExpensiveChart,那 isOpen 变化时 Page 照样会重新执行、重新创建 ExpensiveChart 元素,children 引用照样变,JSX 结构写得再像也没用。
  • 矫枉过正,把所有状态都往下移——理解了状态下移的好处后,容易走到另一个极端:凡是 state 都尽量往最底层的组件塞。但如果几个兄弟组件本来就需要共享和响应同一份数据(比如搜索框的值要联动过滤商品列表),硬把它下移到某个子组件里,只会导致其他组件拿不到这份数据,还得靠更别扭的方式(比如全局状态、事件总线)绕回来。判断标准始终是"这个 state 到底被谁需要",覆盖到刚好够用的那一层就好,不多不少。

面试高频考点

  1. 为什么把 children 作为 props 传入能减少重渲染?—— 因为 JSX 子节点在编译后本质上是 Modal({ children: <ExpensiveChart /> }) 这样的函数调用,<ExpensiveChart /> 这个元素对象是在外层(创建它的那次渲染)生成的,不是在 Modal 内部生成的。只要外层组件本身没有因为某个 state 而重新执行渲染函数,这个元素对象的引用就不会变,Modal 重渲染时拿到的 children 还是同一个引用,React 据此判断这部分子树的输入没变而跳过渲染——这是 React 元素创建机制的天然结果,不依赖 memo 或任何浅比较逻辑。
  2. 状态下移和状态提升是不是矛盾的两个技巧?—— 不矛盾,它们是同一个原则的两个方向:state 应该放在"刚好覆盖所有需要它的组件"的那一层。只有一个组件用就下移到那个组件内部;多个兄弟组件都要共享就提升到能同时管住它们的公共父组件。判断依据始终是这个 state 实际的使用范围,不是固定往上或往下移动。
  3. 状态下移和内容提升,分别适用于什么场景?两者能否互相替代?—— 状态下移解决的是"某个 state 本身可以彻底不放在高层"的场景(比如输入框自己的临时值);内容提升解决的是"state 必须留在高层(比如弹窗开关要由外层决定渲不渲染),但受它影响的子树里有一部分内容其实不依赖这个 state"的场景。两者不能互相替代,但可以配合:先看能不能下移,不能下移的部分再考虑把无关内容通过 children 提升出去。
  4. 为什么说状态下移/内容提升比 memo 等记忆化手段更"根本"?—— memo/useMemo/useCallback 是在组件树结构不变的前提下,靠额外的比较逻辑在渲染前"拦截"不必要的渲染,本质是事后补救,需要维护依赖数组、留意浅比较失效的边界条件。而状态下移和内容提升是调整 state 的归属层级和组件组合方式,让不相关的子树从设计上就不会被卷入重渲染的触发路径,零运行时比较成本,也不会有依赖数组写错导致的隐蔽 bug,这也是 React 官方推荐"先组合、后记忆化"的原因。

与其他知识点的联系

  • React.memo / useMemo / useCallback：文中"三种手段的对比"已经把关系讲清楚——memo 系列是"事后补救"，需要记忆化 API 和依赖数组；状态下移/内容提升是"事前设计"，零心智负担。但两者不是互斥的：结构调整解决不了的场景（子组件确实需要接收会变化的、开销大的 props）依然要靠 memo/useMemo/useCallback 兜底，React 官方推荐的顺序是"先组合，后记忆化"。
  • 组件组合与嵌套：13 篇讲的"某部分有独立的 state 就该拆成独立组件"这条拆分信号，正是本篇"状态下移"的前置基础——本篇的 SearchBox 之所以能把 searchText 下移进去，前提是它本来就该被拆成一个独立组件；"内容提升"用到的 children 作为 props，也是组合思想的延伸应用。
  • 虚拟滚动 TanStack Virtual：下一篇 133 讲的虚拟滚动和本篇的定位一脉相承——本篇解决的是"重渲染要不要发生"，虚拟滚动解决的是更前置的"DOM 节点数量本身"；两者都是从架构/机制层面减少渲染成本，而不是像 memo 那样靠运行时比较去拦截，是同一种"从设计上根治"思路在不同问题域的应用。
  • useDeferredValue：本篇的状态下移和内容提升是通过调整组件结构减少不必要的重渲染，136 篇的 useDeferredValue 则是在重渲染确实要发生的前提下，把它标记为可被打断的低优先级更新——前者是"不让它发生"，后者是"让它发生但不阻塞更紧急的交互"，两者可以在同一个应用里配合使用，分别处理"结构能解决的部分"和"结构解决不了、只能靠调度让步的部分"。
  • AntDesign 与 shadcn/ui：138、139 篇提到 AntDesign 是黑盒依赖、shadcn/ui 是可直接编辑的源码——面对第三方组件库（比如 AntDesign 的 Modal）时，本篇的"内容提升"模式格外有用：只要把不依赖某个 state 的内容作为 children 传给这些黑盒组件，就能天然避免重渲染，而不需要等官方支持某个记忆化选项，这是黑盒场景下少数几种"不用改组件内部实现"就能生效的优化手段。
