React · useEffect

副作用：渲染之外的世界

React 组件的本职工作是根据 state / props 计算出 JSX——一个纯函数。但真实应用还要发请求、订阅事件、操作 DOM、设定时器。这些"渲染之外、会影响外部世界"的操作就是副作用，它们必须放进 useEffect。

什么是"纯"，什么是"副作用"

纯函数：相同输入永远得到相同输出，且不改变外部任何东西。React 要求组件的渲染过程（函数体本身 + 返回的 JSX）是纯的。

副作用：任何"和渲染出 UI 无关、却对外部世界产生影响"的操作。

✅ 渲染（纯计算）
function App({ name }) {
  const greeting =
    'Hi, ' + name;   // 纯计算
  return <h1>{greeting}</h1>;
}
// 只依赖输入，只产出 JSX
⚠️ 副作用（影响外部）
fetch('/api/user')      // 网络请求
document.title = name   // 改 DOM
localStorage.setItem()  // 存储
setInterval(...)        // 定时器
socket.subscribe(...)   // 订阅

为什么渲染期间不能有副作用

这是理解 useEffect 的关键。React 的渲染不保证只跑一次，也不保证会提交：

  • 并发特性下，React 可能渲染到一半放弃，重新渲染。若你在渲染期发了请求，就发了两次、甚至发了个"最后没用上的"请求。
  • 严格模式（StrictMode）开发时会故意渲染两次，帮你暴露不纯的代码。
  • 渲染期直接改 DOM，会和 React 自己的 DOM 更新打架。
反例:在渲染期直接执行副作用
function Bad({ userId }) {
  // ❌ 渲染期直接请求：每次渲染都发，且渲染可能被中断/重复
  fetch('/api/user/' + userId).then(...);

  // ❌ 渲染期直接改 DOM：绕过了 React，且时机不对
  document.title = 'User ' + userId;

  return <div>...</div>;
}
渲染必须"无副作用地算出 UI"。把渲染想成"拍一张 UI 的照片"——照相不应该改变被拍的世界。副作用是"照完相后要做的事"，得推迟到照片拍好、贴到屏幕上之后再做。

useEffect：把副作用推迟到渲染之后

useEffect 的作用就是：告诉 React "这段代码不是渲染的一部分，请你在把 DOM 更新到屏幕之后再执行它"。

正确写法:副作用放进 useEffect
function Good({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // 这里的代码在"渲染完成、屏幕已更新"之后执行
    fetch('/api/user/' + userId)
      .then(r => r.json())
      .then(setUser);

    document.title = 'User ' + userId; // 改 DOM 也放这里
  }, [userId]); // 依赖数组:userId 变了才重新执行（下一篇详讲）

  return <div>{user?.name}</div>;
}

useEffect 的执行时机

组件函数执行(渲染)
→
React 提交 DOM 到屏幕
→
浏览器绘制
→
运行 effect 回调

关键点：effect 在渲染完成、浏览器已经画出画面之后异步执行，所以它不会阻塞浏览器绘制，用户先看到界面，副作用随后跑。（若确实需要在绘制前同步执行，用 useLayoutEffect，见第 41 篇。）

副作用应"同步"于外部系统，而非"响应事件"。官方给的直觉：如果一件事是"用户点击按钮才做"（比如提交表单），那属于事件处理器，不是 effect；如果一件事是"只要组件显示在屏幕上，就必须让某个外部系统保持某种状态"（比如连着聊天服务器），那才是 effect。这个区分是判断"要不要用 useEffect"的核心标尺。
联系函数式编程。"纯函数 + 把副作用隔离到边界"是 FP 的核心思想（Haskell 的 IO Monad 也是同一哲学）。React 把这套搬到 UI：渲染是纯的（可预测、可缓存、可中断），副作用被 useEffect 圈到一个明确的、受控的边界里。理解了这层，就理解了 React 为什么这么设计。

常见陷阱

  • 把"事件触发的操作"也塞进 useEffect：比如点击提交按钮后发请求，直接写进一个监听某个 state 变化的 effect 里。这样一来，凡是这个 state 因为任何原因发生变化（不只是用户点击），请求都会被重新触发。判断标准是问自己：这件事是"用户做了某个动作才该发生"，还是"只要组件显示着、状态是这样，就该一直保持同步"——前者用事件处理器，后者才用 effect。
  • 在渲染期直接调用 setState 导致无限循环：如果把 setUser(...) 直接写在组件函数体里（不包在 effect 里），会触发重渲染，重渲染又执行到这行代码，再次 setState——陷入死循环。这是"渲染必须是纯计算"这条规则被打破后最典型的报错表现。
  • 误以为 StrictMode 下 effect 跑两次是 bug：开发环境下 React 故意把组件渲染两次、effect 也执行两次（挂载→卸载→再挂载），目的是主动暴露"没写清理函数"或"不是幂等"的副作用。看到这个现象不该关掉 StrictMode 掩盖，而应该检查副作用是否正确处理了重复执行的情况。
  • 把"读取外部值"误判为副作用而滥用 useEffect：例如根据 props 计算一个派生值，本该直接在渲染期用一行表达式算出来，却写成 useEffect(() => setDerived(compute(props)), [props])。这不仅多了一次渲染的延迟，还违背了"渲染应该是纯计算"的原则——纯粹的计算不是副作用，不需要 effect。

面试高频考点

  1. 为什么 React 组件的渲染过程要求是"纯"的？—— 因为 React 不保证渲染只执行一次、也不保证一定会被提交到屏幕:并发特性下可能渲染到一半被打断重来,StrictMode 下会故意重复渲染来暴露问题。如果渲染期直接执行有副作用的代码(发请求、改 DOM),这些操作会被不受控地重复或提前执行,产生难以预测的 bug。纯计算不管重复执行多少次结果都一样,才能被 React 安全地中断、重试、复用。
  2. 什么样的操作应该放进 useEffect，什么样的不需要？—— 判断标准不是"是否复杂"，而是"是否影响渲染之外的世界"。发请求、订阅事件、操作 DOM、设置定时器都会影响外部系统，必须放进 effect，让它们在渲染提交到屏幕之后才执行。而单纯基于 props/state 计算出的派生值属于纯计算，应该直接在渲染期算出，不需要额外用 effect 包一层再 setState。
  3. useEffect 的回调什么时候实际执行，为什么这么设计？—— effect 在组件渲染完成、React 把 DOM 更新提交到屏幕、浏览器完成绘制之后才异步执行,不会阻塞首次绘制。这样设计是为了让用户尽快看到界面,副作用(往往涉及网络、DOM 操作等有延迟或开销的事)被推迟到不影响视觉响应速度的时间点。如果需要在绘制前同步读取或修改 DOM 布局(避免闪烁),才用 useLayoutEffect。
