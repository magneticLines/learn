Vite 创建 React 项目

几年前学 React,第一件事是 create-react-app;现在打开任何一个新项目的 README,几乎清一色是 npm create vite@latest。CRA 已经被官方标记为不再维护,原因不是"过时了",而是它的架构设计在今天的项目规模下越来越吃力。理解 Vite 为什么快,能帮你理解现代前端工具链的一条主线:能不做的事就不做,能延后做的事就延后做。

CRA 的问题:为什么冷启动慢、改一行代码也要等

CRA 底层是 Webpack。Webpack 的工作模式是:开发服务器启动前,先把整个项目(包括 node_modules 里几百上千个模块)按照依赖关系打包成一个或几个 bundle,再启动服务器。项目越大,这个"打包"耗时越长——这就是大家吐槽的"改一行代码,页面刷新要等好几秒"。

扫描全部模块
→
打包成 bundle
→
启动开发服务器
→
才能看到页面(CRA/Webpack)

Vite 的思路:利用浏览器原生 ESM,按需编译

Vite 的核心洞察是:现代浏览器原生支持 ES Module(import/export),不需要打包器帮你"拼"成一个文件才能跑。于是 Vite 开发模式下根本不打包整个项目——它启动一个服务器,浏览器请求哪个模块,它才编译哪个模块,按需给。这就是"冷启动几乎瞬间完成"的原因:不用等全量打包,先把首屏用到的几个文件编译好就能显示。

启动服务器(几乎无耗时)
→
浏览器按需请求模块
→
Vite 实时编译返回(Vite)

那 node_modules 里那些不是 ESM 格式(比如 CommonJS)、或者被拆成几百个小文件的第三方依赖(如 lodash-es)怎么办?这些如果也逐个按原生 import 请求,会发出海量网络请求,反而更慢。Vite 用 esbuild(Go 写的打包器,比 JS 写的打包器快一到两个数量级)预先把这些依赖预构建成少数几个 ESM 文件,缓存起来——这一步只在依赖变化时重新做一次,不是每次改代码都做。

一句话记住 Vite 快在哪: 开发时"你的代码"按需编译(不打包,原生 ESM 直接跑),"第三方依赖"用 esbuild 提前预构建一次(避免几百个请求)。生产构建(vite build)仍然用 Rollup 打包,因为生产环境要的是体积小、请求少,和开发时要的"启动快"目标不同。
常见误区: "Vite 生产环境也不打包吗?"——不是的。npm run dev 走的是上面说的按需编译模式,追求开发体验;npm run build 走的是 Rollup 打包模式,追求最终产物的体积和加载性能。两套引擎、两个目标,别混为一谈。

创建项目

官方推荐用 npm create vite@latest(等价于 npm init vite@latest),它会拉起一个交互式脚手架,或者直接用参数一步到位:

终端:创建一个 React + TypeScript 项目
# 交互式,会依次问项目名、框架、变体
npm create vite@latest

# 或者直接指定模板,一步到位
npm create vite@latest my-app -- --template react-ts

cd my-app
npm install
npm run dev

生成的目录结构大致是这样:

Vite + React + TS 模板的目录结构
my-app/
├── public/              // 静态资源,原样拷贝到构建产物,不经过编译
├── src/
│   ├── main.tsx         // 应用入口,ReactDOM.createRoot 挂载 App
│   ├── App.tsx          // 根组件
│   ├── App.css
│   └── index.css
├── index.html          // 入口 HTML,注意在根目录,不在 public 里!
├── vite.config.ts      // Vite 配置文件
├── tsconfig.json
└── package.json
陷阱:index.html 的位置。 传统项目里 index.html 是"静态资源",放在 public/;Vite 里它是构建入口,必须放在项目根目录。Vite 会解析这个 HTML 里的 <script type="module" src="/src/main.tsx"> 来找到应用的起点——这也是为什么 Vite 项目的 index.html 里能直接写一行 <script> 引用 .tsx 文件,而不用像老项目那样先编译好再引用。

vite.config.ts 基本结构

vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

// defineConfig 只是提供类型提示,本身不做额外的事
export default defineConfig({
  plugins: [react()],  // 让 Vite 认识 JSX、支持 Fast Refresh(组件级热更新)
  server: {
    port: 5173,       // 开发服务器端口,默认就是 5173
  },
});

@vitejs/plugin-react 这个插件负责把 JSX 转成浏览器能跑的 JS,并接入 Fast Refresh——组件代码改动后只替换这一个组件、保留其他组件的 state,而不是整页刷新。这也是"HMR(热模块替换)体验好"的关键,和"冷启动快"是两件事:冷启动快靠不打包,HMR 快靠精确到模块级的替换 + 不用重新打包整个依赖图。

接下来: 项目创建好了,下一篇讲一个同样重要但容易被忽视的问题——目录该怎么组织。脚手架只给了最基础的 src/App.tsx,项目稍微一大,文件往哪儿放、按什么规则分层,直接决定了半年后这个项目是"越改越顺手"还是"越改越乱"。

小结

  • CRA 基于 Webpack,开发前要把整个项目打包,项目越大冷启动/HMR 越慢,官方已不再维护。
  • Vite 开发模式利用浏览器原生 ESM,不打包业务代码,浏览器请求哪个模块就编译哪个,冷启动几乎瞬间完成。
  • 第三方依赖用 esbuild 提前预构建成少量 ESM 文件,避免海量网络请求,这一步只在依赖变化时重做。
  • 生产构建(vite build)换用 Rollup 打包,目标是体积小、请求少,和开发时"启动快"的目标不同,别混淆。
  • npm create vite@latest my-app -- --template react-ts 一步创建项目;index.html 在根目录而非 public/,是构建入口而非静态资源。
  • @vitejs/plugin-react 负责 JSX 转换与 Fast Refresh(组件级热更新,保留 state)。
