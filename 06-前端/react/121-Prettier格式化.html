Prettier 格式化

Code review 里最没营养的争论是什么?"这里该用单引号还是双引号""缩进两个空格还是四个""这行要不要加分号"。这些问题没有对错,只有一致与否,却能耗掉大量精力吵架。Prettier 的哲学很直接:这类没有争议价值的格式问题,交给工具自动决定,人不用再讨论。

Prettier 与 ESLint 的分工

上一篇讲到 ESLint 管"代码质量和潜在错误"(未使用变量、Hooks 用法违规、依赖数组遗漏);Prettier 管的是纯粹的排版格式——空格、换行、引号风格、行宽、逗号位置。两者看起来都在"检查代码",但检查的维度完全不同:

ESLint 管:代码质量
- 变量声明了却没用
- useEffect 依赖数组遗漏
- Hooks 用在了条件语句里
- 用了 == 而不是 ===
→ 这些问题会导致 bug,不能忽略
Prettier 管:纯格式
- 用单引号还是双引号
- 每行最多多少字符换行
- 对象属性后要不要加尾逗号
- 缩进用几个空格
→ 这些问题没有对错,只要团队统一
判断一条规则该归谁管的简单方法: 问自己"这个问题会不会导致代码行为出错?" 会,归 ESLint;纯粹是"看着舒服与否"的问题,归 Prettier。行业经验是:格式类问题一旦交给 ESLint 管理,规则数量会爆炸式增长(要覆盖引号、分号、换行、缩进等无数种排列组合),而 Prettier 用一套预设好的格式规则直接把这类问题连讨论的空间都拿掉,效率高得多。

为什么要用 eslint-config-prettier

问题在于:ESLint 的核心规则集里,历史上也包含了一些格式类规则(比如 indent、quotes、semi)。如果这些规则和 Prettier 的判定标准不一致,就会出现两个工具互相打架的情况——ESLint 说这里要用单引号,Prettier 自动格式化时又给你改成双引号,保存一次代码,规则报错换了个位置,反复横跳。

ESLint: 要求单引号
vs
Prettier: 自动改成双引号
eslint-config-prettier
→
关掉 ESLint 里和 Prettier 冲突的格式规则

eslint-config-prettier 本身不新增任何规则,它做的唯一一件事是:把 ESLint 配置里所有会和 Prettier 冲突的格式化规则关掉,让 ESLint 只专心管代码质量,格式的事完全交给 Prettier 一个工具决定,消除两者的"地盘之争"。

eslint.config.js:让 eslint-config-prettier 放在配置数组最后
import js from '@eslint/js';
import reactHooks from 'eslint-plugin-react-hooks';
import prettierConfig from 'eslint-config-prettier';

export default [
  js.configs.recommended,
  { plugins: { 'react-hooks': reactHooks }, rules: { /* ... */ } },
  prettierConfig, // ← 必须放最后,才能"覆盖"前面配置里冲突的格式规则
];
陷阱:顺序放错等于没生效。 eslint-config-prettier 的原理是"覆盖前面的规则设置",所以必须放在配置数组的最后一项。如果放在中间,后面的规则集又会把格式规则重新打开,冲突照样存在。另外注意它叫 eslint-config-prettier,和另一个包 eslint-plugin-prettier(把 Prettier 格式差异当成 ESLint 报错抛出来)是两回事,常被搞混——多数团队只需要前者,没必要让 ESLint 和 Prettier 报同一个问题两次。

保存时自动格式化

Prettier 装好只是第一步,真正提升效率的是保存文件时自动跑一遍格式化,写代码时完全不用手动调格式,保存那一刻自动变整齐。以 VS Code 为例:

.vscode/settings.json
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true
}

把这份配置提交到仓库里(而不是只放在个人的全局设置),能保证团队每个人保存代码时都用同一份格式规则,配合 .prettierrc 定义具体规则(比如是否要分号、单双引号、行宽),彻底终结格式类的 code review 争论。

文件作用
.prettierrc定义格式规则,如 semi、singleQuote、printWidth
.prettierignore指定不需要格式化的文件(如构建产物、第三方代码)
.vscode/settings.json配置编辑器保存时自动格式化,提交到仓库让团队共用
eslint-config-prettier关闭 ESLint 里和 Prettier 冲突的格式规则
放在质量工具链的坐标系: ESLint 管"会不会出 bug",Prettier 管"看起来整不整齐",两者是互补而非竞争关系。下一篇会看这两个工具如何在 git commit 之前自动跑起来——靠 Husky 挂 git hooks、lint-staged 只检查改动过的文件,把质量把关从"人记得跑"变成"提交时强制跑"。

小结

  • Prettier 与 ESLint 的分工:Prettier 管纯格式(引号、缩进、换行等无对错之分的风格问题),ESLint 管代码质量和潜在错误。
  • ESLint 核心规则集里历史遗留的格式类规则会和 Prettier 冲突,需要用 eslint-config-prettier 关掉它们,让两个工具不再互相打架。
  • eslint-config-prettier 必须放在 flat config 数组的最后一项才能生效,且注意不要和 eslint-plugin-prettier 搞混。
  • 配合编辑器的"保存时自动格式化"(formatOnSave),写代码时不用管格式,保存那一刻自动整理好。
  • 把编辑器格式化配置(如 .vscode/settings.json)和 .prettierrc 一起提交到仓库,保证团队协作时格式标准统一,终结无意义的格式类 review 争论。

与其他知识点的联系

  • ESLint 与 eslint-plugin-react: 上一篇讲的静态分析能发现"未使用变量""依赖数组遗漏"这类会导致 bug 的问题,本篇的 Prettier 管的是没有对错、纯粹靠约定统一的格式问题——两者的分工边界要清楚:如果一个问题的答案是"看团队喜好",就不该让 ESLint 去管,应该交给 Prettier 一次性解决,避免规则数量爆炸。
  • Husky 与 lint-staged: 下一篇要解决的问题是"没人强制执行,规范就形同虚设"——Prettier 配好了 .prettierrc 和 formatOnSave,但如果开发者用的编辑器没装插件、或者忘了保存前格式化,不规范的代码依然会被提交。Husky + lint-staged 就是把 Prettier(和 ESLint)接入 git commit 流程,变成机制上很难跳过的强制检查。
  • eslint-plugin-react-hooks 的 exhaustive-deps: 本篇强调"格式类问题没有对错",但要注意区分——exhaustive-deps 报的是逻辑问题(依赖遗漏会导致闭包捕获旧值),不是格式问题,prettier --write 这类格式化命令解决不了它,必须靠开发者手动修正依赖或调整代码结构,这是判断"什么该归 ESLint、什么该归 Prettier"的实际案例。
  • Vite 创建 React 项目与路径别名配置: 项目搭建阶段配置的 .vscode/settings.json、tsconfig.json 里的路径别名等,都属于"提交到仓库、让团队共用一套约定"的工程化实践,和本篇强调的"格式化配置要提交而不是只放个人设置"是同一种思路——把本该团队统一的东西,从"靠自觉"变成"靠配置文件"。
  • CSS Modules / Tailwind 的类名书写: Prettier 对 JSX 里过长的 className 字符串(尤其是 Tailwind 风格)也会做自动换行处理,这也是社区专门出现 prettier-plugin-tailwindcss 插件(自动按官方推荐顺序排列工具类)的原因——格式化的边界会随着项目用的 CSS 方案不同而扩展到新的场景。
