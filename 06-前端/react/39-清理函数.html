React · useEffect

清理函数：善后与防内存泄漏

effect 建立的东西（订阅、定时器、连接）在不需要时必须拆掉，否则会残留、重复、泄漏。useEffect 的回调可以 return 一个函数，React 会在合适的时机调用它做清理。理解"何时清理"是写对 effect 的关键。

语法:return 一个函数

effect 返回清理函数
useEffect(() => {
  const conn = createConnection(roomId);
  conn.connect();       // 建立(setup)

  return () => {       // 返回的就是清理函数(cleanup)
    conn.disconnect();   // 拆除，和 setup 一一对应
  };
}, [roomId]);

清理函数何时执行

两个时机，记牢：

  1. 下一次 effect 重新执行之前——依赖变了，React 先跑上一次的清理，再跑新的 setup。
  2. 组件卸载时——最后一次清理，彻底善后。
setup(roomId=A)
→
cleanup(A)
→
setup(roomId=B)
→
cleanup(B) 卸载

关键理解：每一次 setup 都配一次 cleanup，成对出现。清理函数捕获的是"它对应那次 setup"的闭包值——所以上面清理的 conn 正是那次建立的连接，不会错拆别人的。

不清理会怎样:重复与泄漏

反例:定时器不清理,越积越多
useEffect(() => {
  setInterval(() => tick(), 1000);
  // ❌ 没有 return 清理
}, [something]);
// something 每变一次就新开一个定时器，旧的还在跑
// 3 次变化后 → 3 个定时器同时 tick，且组件卸载后仍在跑
// 这就是内存泄漏 + 逻辑错乱
❌ 泄漏的订阅
useEffect(() => {
  window.addEventListener(
    'resize', onResize);
  // 没移除，卸载后监听器
  // 仍持有组件闭包 → 泄漏
}, []);
✅ 配对清理
useEffect(() => {
  window.addEventListener(
    'resize', onResize);
  return () =>
    window.removeEventListener(
      'resize', onResize);
}, []);

取消请求:避免竞态与"卸载后 setState"

数据请求是清理的经典场景。请求是异步的，可能在组件已卸载、或用户已切换参数后才返回，导致：① 对卸载的组件调 setState；② 竞态——旧请求比新请求晚返回，覆盖了正确结果。

用 ignore 标志或 AbortController 处理竞态
useEffect(() => {
  let ignore = false;  // 本次 effect 专属的标志

  fetch('/api/user/' + id)
    .then(r => r.json())
    .then(data => {
      // 只有"没被清理"的那次请求才允许写入
      if (!ignore) setUser(data);
    });

  return () => {
    ignore = true;  // 清理时把旧请求的结果"作废"
  };
}, [id]);
// id 从 1 快速切到 2:cleanup 把 id=1 的 ignore 置 true，
// 即便 id=1 的响应后到，也不会覆盖 id=2 的结果
更现代的做法用 AbortController：const ctrl = new AbortController(); fetch(url, {signal: ctrl.signal})，清理里 ctrl.abort()，直接取消底层请求。但 ignore 标志更通用（对付任何 Promise），两者常一起用。
StrictMode 会立刻检验你的清理。开发模式下 React 对每个 effect 走一遍 setup → cleanup → setup。如果清理写得不对（比如没解除订阅），你会立刻看到"订阅了两次"的现象。这正是它的价值——把只有在生产里偶发的泄漏 bug，提前到本地必现。不要通过关掉 StrictMode 来"解决"这个问题。
心智模型:effect 描述"如何同步到外部系统"，清理描述"如何断开"。不要把 effect 想成"挂载时做 X、卸载时做 Y"的生命周期钩子（那是 class 组件的旧思维）。正确的想法是：每次渲染，effect 都在回答"当前这个状态下，外部系统该处于什么状态"，清理则负责在切换前先回到干净状态。这样即使依赖频繁变化，setup/cleanup 成对执行也永远能保持外部系统同步。

常见陷阱

  • 清理函数里读到的是"过期的"闭包值，而不是最新值：清理函数是在它对应的那次 setup 创建时被捕获的闭包，读到的变量永远是那次渲染时的值，不会自动更新到最新。这本身是正确行为（也是清理能精确对应到该次连接的原因），但新手常误以为清理函数能拿到"当前"最新的 state，写出逻辑错误的清理代码。
  • 只在组件卸载时才想到要清理，忘了"依赖变化时也会清理一次"：很多人把清理函数简单理解为"componentWillUnmount"，只考虑卸载场景。但实际上只要依赖数组里的值变化，React 就会先跑上一次的清理、再跑新的 setup——如果清理逻辑写得不完整（比如只清理了一半的副作用），依赖频繁变化时会不断残留没清理干净的部分。
  • 异步请求不清理，导致竞态覆盖新数据：请求是异步的，参数快速切换时，旧请求可能比新请求更晚返回。如果没有用 ignore 标志或 AbortController 让清理时"作废"旧请求的结果，旧数据会在新数据之后到达并覆盖它，界面显示错误的内容，而且这个 bug 只在网络慢或参数切换快时才复现，很难排查。
  • 用 StrictMode 下"跑两次"的现象反推清理函数没问题：开发环境故意 setup→cleanup→setup 一遍是用来检验清理是否正确的，不是清理逻辑本身的一部分。如果发现某个功能"订阅了两次"，说明清理没有正确对称地拆除 setup 建立的东西，应该去修复清理逻辑，而不是关掉 StrictMode 让问题在本地隐藏、留到生产环境暴露。

面试高频考点

  1. 清理函数具体在什么时机被调用？—— 两个时机：① 依赖数组中的值发生变化、effect 要重新执行之前，React 先调用上一次 effect 返回的清理函数，再执行新一轮的 setup；② 组件卸载时，调用最后一次的清理函数做最终善后。可以理解为"每一次 setup 都精确配对一次 cleanup"，成对出现，从不脱节。
  2. 为什么清理函数总能拿到和它对应的那次 setup 一致的变量，不会错拆？—— 因为清理函数是在某一次 effect 回调执行时通过闭包捕获的，它引用的是那次渲染时的局部变量（比如那次创建的 conn 连接对象），不是共享的可变状态。即使依赖变化触发了新的 setup 和新的闭包，旧的清理函数依然精确指向旧的那个实例，不会因为组件后续状态改变而"看错"要清理的对象。
  3. 不写清理函数会导致什么问题？举例说明。—— 会导致资源重复创建和内存泄漏。例如在 effect 里 setInterval 却不清理，依赖每变化一次就新开一个定时器，旧的仍在跑，最终多个定时器同时触发，且组件卸载后定时器仍在后台运行，持有对已销毁组件的闭包引用，造成内存泄漏和逻辑错乱。同理，事件监听、WebSocket 连接、订阅等都需要成对清理。

与其他知识点的联系

  • useEffect 依赖数组（上一篇）：清理函数的触发时机完全由依赖数组决定——依赖数组不变，清理只在卸载时跑一次；依赖数组里的值一变，清理会在每次重新 setup 之前先跑一遍。可以说依赖数组回答"effect 何时重跑"，清理函数回答"重跑或卸载之前该做什么善后"，两者是同一机制配套的两半。
  • 常见副作用（下一篇）：本篇讲的"取消请求""移除监听""清定时器"三种清理场景，正是下一篇要系统整理的三大副作用类型（数据获取、事件订阅、定时器）各自对应的清理动作。可以把本篇当作"清理的通用原理"，下一篇是"套用到具体场景的模板"。
  • useRef 存储可变值：清理函数里常用的 ignore 标志，本质上是一个"本次 effect 专属"的可变变量，作用和 useRef 存储可变值有相似之处——都是不触发渲染、纯粹用来"记账"的状态。区别在于 ignore 是每次 effect 调用时局部声明的普通变量，靠闭包捕获；而 useRef 提供的是跨渲染保持同一个引用的容器，两者适用场景不同但思路相通。
  • Hooks 只在函数组件中调用：清理函数本身是在 effect 回调创建时通过闭包捕获的，而 effect 回调又是"渲染之后执行"的代码——这解释了为什么清理函数内部同样不能调用其他 Hook：它执行的时机和事件处理器一样，都脱离了"当前正在渲染某个组件"的窗口。
