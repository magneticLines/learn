React · JSX

className · style · htmlFor：JSX 与 HTML 的属性差异

从 HTML 切换到 JSX，第一个让人迷惑的就是属性名变了：class 变成了 className，for 变成了 htmlFor，onclick 变成了 onClick……这些改变不是随意为之，背后有一套统一的逻辑。

为什么属性名和 HTML 不同

根本原因：JSX 属性最终是 JS 对象的键名（key）。

回忆一下 JSX 编译后的产物：<div className="card"> 编译成 React.createElement('div', { className: 'card' })。属性对象 { className: 'card' } 就是普通的 JS 对象——所以属性名必须是合法的 JS 标识符，遵守 JS 的命名规则：

  • 不能用 JS 保留字（class、for 都是 JS 关键字）
  • 约定驼峰命名（JS 对象属性习惯驼峰，而 HTML 属性用连字符）

5 个核心差异（用表格记忆）

规则 HTML 写法 JSX 写法 原因
CSS 类名 class="card" className="card" class 是 JS 保留字（ES6 的 class 关键字），直接用会冲突
label 绑定 for="email" htmlFor="email" for 是 JS 保留字（for 循环），加 html 前缀区分
事件处理 onclick="fn()" onClick={fn} JS 对象键名约定驼峰；值是函数引用，不是字符串
自闭合标签 <img>（省略斜杠） <img />（必须闭合） JSX 遵循 XML 语法，所有标签必须显式关闭
内联样式 style="color:red"（字符串） style={{ color: 'red' }}（对象） JSX 属性值是 JS，样式用对象表达；CSS 属性名也驼峰化（font-size → fontSize）

style 属性的双层花括号是什么意思？

这是初学者最常疑惑的地方。style={{ color: 'red' }} 里有两层 {{ }}：

  • 外层 { }：JSX 表达式插槽（告诉 JSX "这里放 JS 值"）
  • 内层 { }：JS 对象字面量（样式对象本身）

所以 style={{ color: 'red', fontSize: 16 }} 本质上是 style={一个JS对象}，和 const s = { color: 'red' }; style={s} 完全等价。

综合代码示例：5 个差异一次看完

包含全部 5 个差异的 JSX 示例
<div
  className="form-card"           // ← class → className（JS 保留字）
  style={{                       // ← style 是对象，不是字符串
    backgroundColor: '#fff',  // ← background-color → 驼峰
    padding: 16,               // ← 数字默认单位 px
    fontSize: '14px'            // ← font-size → 驼峰
  }}
>
  <label htmlFor="email">邮箱</label>     // ← for → htmlFor（JS 保留字）

  <input
    id="email"
    type="email"
    onChange={handleChange}      // ← onclick → onClick（驼峰事件）
  />                               // ← 自闭合（JSX XML 要求）

  <img
    src="logo.png"
    alt="logo"
  />                               // ← img 也必须自闭合

  <button onClick={handleSubmit}>提交</button>
</div>

style 对象中的 CSS 属性名转换规则

CSS 里用连字符的属性，在 JSX style 对象里一律改成驼峰命名：

CSS 属性JSX style 对象键名
background-colorbackgroundColor
font-sizefontSize
border-radiusborderRadius
z-indexzIndex
text-aligntextAlign
flex-directionflexDirection

数字值默认单位是 px：fontSize: 16 等同于 font-size: 16px。如果需要其他单位，写字符串：fontSize: '1.2em'。

记忆技巧：两条规则，覆盖所有情况

规则一：HTML 属性用连字符的 → JSX 里改驼峰。
font-size → fontSize，background-color → backgroundColor

规则二：HTML 属性是 JS 保留字的 → 换名或加前缀。
class（JS 关键字）→ className，for（JS 关键字）→ htmlFor
常见错误：初学者经常在 JSX 里写 class="..."（HTML 习惯），React 会给出警告（Warning: Invalid DOM property 'class'. Did you mean 'className'?），但组件仍然能渲染——只是样式可能不生效。养成习惯，切换到 JSX 就想 className。

常见陷阱

  • style 写成字符串：把 HTML 习惯带过来写 style="color: red"，React 会直接报错，因为 style 属性必须是一个 JS 对象，不接受字符串。忘了这一点的人经常卡在"为什么样式属性完全不生效甚至报类型错误"上。
  • style 对象里用连字符写属性名：写成 { 'font-size': 16 } 而不是 { fontSize: 16 }。因为 style 对象的键最终是 JS 对象属性，遵守驼峰约定；用连字符的字符串键虽然语法上不报错，但浏览器无法识别这个 CSS 属性，样式悄悄失效。
  • 数字长度值忘记单位，或者不该加单位时加了：fontSize: 16 默认解释成 16px，这点没问题；但像 lineHeight、zIndex、opacity 这些无单位属性，如果错误地写成字符串加单位（如 '1.5px'）反而会导致样式失效或行为异常。
  • onClick 后面加括号直接调用：写成 onClick={handleClick()} 而不是 onClick={handleClick}，会导致组件一渲染就立刻执行这个函数（因为花括号里是 JS 表达式，handleClick() 的求值结果才是 onClick 拿到的值），而不是等点击时才执行。这是初学者调试"函数为什么自动执行了"时最常见的根源。
  • 自定义 DOM 属性直接搬 HTML 写法：比如给 <div> 加 data-testid 之类的属性通常没问题（data-*/aria-* 是例外，保留原样），但如果照搬其他非标准 HTML 属性名，React 会给出 "Unknown prop" 警告，容易和 className/htmlFor 这类必须转换的属性搞混。

面试高频考点

  1. 为什么 JSX 里用 className 而不是 class？—— 因为 JSX 属性最终会变成 React.createElement 调用里 props 对象的键名，而 class 是 JS 的保留关键字（ES6 class 语法），不能合法地用作这个位置的标识符，所以 React 选择了 className 作为替代名。
  2. JSX 里 style 属性为什么必须是对象，双花括号是什么意思？—— 外层花括号是 JSX 里"这里插入一个 JS 表达式"的语法，内层花括号是这个表达式本身——一个对象字面量。写 style={{ color: 'red' }} 本质上和先声明 const s = { color: 'red' } 再写 style={s} 完全等价，只是内联省略了变量。这样设计是为了让样式可以用 JS 变量动态计算，而不是被限制在字符串拼接里。
  3. onClick={fn} 和 onClick={fn()} 有什么区别？—— {fn} 传递的是函数引用，交给 React 在事件触发时才调用；{fn()} 是在渲染这一刻立即调用 fn，并把它的返回值（通常是 undefined）作为 onClick 的值。这是 JSX 花括号"里面是表达式，立即求值"这一规则的直接体现，也是最容易踩的坑之一。

与其他知识点的联系

  • DOM 属性 vs React props 的本质区别：本文所有差异的根源都是"JSX 属性最终是 JS 对象的键"，这意味着 React 组件收到的从来不是 DOM 属性字符串，而是一份 JS 对象（props）。这也是为什么 style 能传对象、onClick 能传函数引用——DOM 原生属性只能是字符串，而 props 可以是任意 JS 值，能力上完全不是一个量级的东西。
  • JSX 编译原理（08）：本文表格里每一行差异，都能在 08 篇讲的编译产物里找到直接对应：className="card" 编译后就是 props 对象里的 { className: 'card' } 这一个键值对，命名规则完全遵循 JS 对象属性名的语法约束（不能是保留字、约定驼峰）。两篇结合看，"为什么要用 className"就不再是死记的规则，而是编译过程的必然结果。
  • HTML/JS 命名规范差异：HTML 属性历史上大量使用连字符命名（font-size、background-color），这是因为 HTML 属性本身只是纯文本；而 JS 生态（包括对象属性、CSS-in-JS）统一用驼峰命名，因为连字符在 JS 里会被解析成减号运算符。JSX 站在 JS 这一侧，所以必须做一次命名风格的转换，这不是 React 独有的怪癖，而是任何"把 CSS/HTML 概念搬进 JS 对象"的方案都要面对的问题。
