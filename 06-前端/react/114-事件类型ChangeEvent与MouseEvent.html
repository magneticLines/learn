事件类型 React.ChangeEvent / MouseEvent

写 onChange={(e) => ...} 时,e 是什么类型?TS 项目里如果不显式标注,把这个函数单独抽出去写就会立刻报错——因为 React 的事件不是浏览器原生事件,而是一层包装过的合成事件(SyntheticEvent),必须用 React 自己的类型。这一篇讲清这套类型体系怎么用。

为什么不能直接用原生 DOM 的 Event 类型

浏览器原生事件系统里,input 元素的 change 事件类型是 Event,e.target 的类型只是笼统的 EventTarget——根本没有 .value 属性。React 为了跨浏览器行为一致(所有事件用统一冒泡机制、统一属性名)和性能优化(事件池,已在 React 17 移除但类型体系保留至今),对原生事件做了一层包装,这就是 SyntheticEvent。

原生 DOM 类型(不能用)
function handleChange(e: Event) {
  // ❌ 报错:Event 上没有 target.value
  console.log(e.target.value);
}
React 合成事件类型(正确)
function handleChange(
  e: React.ChangeEvent<HTMLInputElement>
) {
  // ✅ target 被精确标注为 HTMLInputElement
  console.log(e.target.value);
}
命名规律: React 的事件类型基本是React.XxxEvent<T> 的形式——Xxx 对应事件种类(Change / Mouse / Keyboard / Form / Focus...),泛型 T 是触发事件的 DOM 元素类型,决定了 e.target/e.currentTarget 的具体类型。这跟上一篇 useRef<HTMLInputElement> 用的是同一批 DOM 类型,是一套体系。

常见事件类型速查

场景类型写法常用属性
输入框内容变化React.ChangeEvent<HTMLInputElement>e.target.value
下拉选择变化React.ChangeEvent<HTMLSelectElement>e.target.value
多行文本框变化React.ChangeEvent<HTMLTextAreaElement>e.target.value
按钮点击React.MouseEvent<HTMLButtonElement>e.clientX,e.currentTarget
表单提交React.FormEvent<HTMLFormElement>e.preventDefault()
键盘按键React.KeyboardEvent<HTMLInputElement>e.key,e.code
受控输入框:onChange 类型标注完整示例
function SearchInput() {
  const [keyword, setKeyword] = useState('');

  // 泛型里的 HTMLInputElement 让 e.target.value 有正确的 string 类型
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setKeyword(e.target.value);
  };

  return <input value={keyword} onChange={handleChange} />;
}
内联写法不用手动标类型。 如果直接写 <input onChange={(e) => setKeyword(e.target.value)} />,TS 能从 onChange 属性的期望类型反向推断出 e 的类型,不需要手写 React.ChangeEvent<...>。只有把处理函数单独抽成一个具名函数时,才必须显式标注参数类型——这也是为什么很多人一开始没遇到这个问题,一重构就报错。

currentTarget 与 target 的区别

e.target 是实际触发事件的元素(比如点击了按钮里的图标,target 就是图标),e.currentTarget 是绑定事件处理函数的那个元素(即使事件冒泡上来,currentTarget 也稳定指向按钮本身)。这个区别在事件委托、冒泡场景里很关键。

MouseEvent:currentTarget 类型更精确
function handleClick(e: React.MouseEvent<HTMLButtonElement>) {
  // currentTarget 的类型精确到 HTMLButtonElement(泛型给的类型)
  e.currentTarget.disabled = true;

  // target 的类型是笼统的 EventTarget,想拿具体属性还得手动断言
  // e.target.disabled // ❌ 类型上不保证 target 就是 button 本身
}
陷阱: 泛型参数 T 只精确了 currentTarget,并不会让 e.target 也变成同一个类型——因为 target 理论上可以是被绑定元素的任意后代,类型层面 React 没法保证具体是谁。需要用 target 上的具体属性时,要么改成事件委托到确定的元素并断言,要么直接改用 currentTarget。

事件处理函数类型的两种定义方式

除了给参数标类型,也可以直接给整个处理函数标注 React 提供的处理函数类型,两者等价,后者更简洁:

标注参数类型
const handleClick = (
  e: React.MouseEvent<HTMLButtonElement>
) => {
  console.log(e.clientX);
};
标注函数整体类型
const handleClick:
  React.MouseEventHandler<HTMLButtonElement>
  = (e) => {
  console.log(e.clientX);
};
和 Props 类型定义的联系: 上一篇讲 props 类型时留了个坑——onClick、onChange 这类回调 props 到底怎么标类型?答案就是用这里的事件处理函数类型,比如 interface ButtonProps { onClick: React.MouseEventHandler<HTMLButtonElement> },组件内部再按需解构使用。

小结

  • React 事件是包装过的 SyntheticEvent,不是浏览器原生 Event,不能直接用原生 DOM 事件类型标注。
  • 事件类型统一形如 React.XxxEvent<T>,泛型 T 是触发/绑定事件的 DOM 元素类型,决定 target/currentTarget 的具体形状。
  • 内联写事件处理函数时 TS 能自动反向推断参数类型;抽成具名函数后必须显式标注,这是最容易踩坑的地方。
  • target 是实际触发元素,类型宽泛;currentTarget 是绑定事件的元素,类型精确到泛型参数 T。
  • props 里的回调可以用 React.MouseEventHandler<T> 等处理函数类型整体标注,和分开标参数类型等价。
