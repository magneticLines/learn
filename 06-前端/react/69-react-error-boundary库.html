react-error-boundary：函数式时代的错误边界

上一篇我们手写了 class 版错误边界。它能用，但每次要重复样板、还不好"重置状态重试"。react-error-boundary 这个由 React 团队成员维护的小库，把这些痛点封装掉，让函数式组件也能优雅地兜错、重试。

手写 class 的三个痛点

  • 样板重复：每个项目都要抄一遍 getDerivedStateFromError + componentDidCatch。
  • fallback 拿不到 error 对象：手写版通常只显示固定文案，想展示具体错误信息还得自己往 state 里塞。
  • 没有"重试"能力：错误发生后 hasError 一直是 true，用户没法点一下"重试"回到正常状态——需要自己实现重置逻辑。

核心 API：ErrorBoundary 组件

它底层仍是一个 class 组件（因为 Hooks 无法实现错误边界，这点无法绕过），但对外暴露的是干净的 props 接口。最常用的是 FallbackComponent：

FallbackComponent 能拿到 error 和 resetErrorBoundary
import { ErrorBoundary } from 'react-error-boundary';

// fallback 是一个普通函数组件，自动收到两个 prop
function Fallback({ error, resetErrorBoundary }) {
  return (
    <div role="alert">
      <p>出错了:</p>
      <pre>{error.message}</pre>
      {/* 点一下清空错误状态、重新渲染子树 */}
      <button onClick={resetErrorBoundary}>重试</button>
    </div>
  );
}

function App() {
  return (
    <ErrorBoundary
      FallbackComponent={Fallback}
      onReset={() => refetchData()}   // 重置时顺带重新拉数据
      onError={(err, info) => log(err, info)}  // 等价 componentDidCatch
    >
      <Profile />
    </ErrorBoundary>
  );
}
三种指定 fallback 的方式：FallbackComponent={Comp}（传组件、最常用）、fallbackRender={({error}) => <div/>}（传渲染函数、内联方便）、fallback={<div/>}（传静态元素、拿不到 error）。三选一，不要同时给。

重置 & 重试：onReset + resetKeys

"重试"是这个库相对手写版最大的增值。有两种触发重置的方式：

手动重置
// fallback 里调用
resetErrorBoundary()
  ↓
清空错误 state
触发 onReset 回调
重新渲染子树
自动重置
<ErrorBoundary
  resetKeys={[userId]}
>
// userId 变化时
// 边界自动重置
// 适合"换个参数重来"

resetKeys 是一个数组，只要其中任一值变化，边界就自动清除错误重新渲染。典型场景：错误由某个 id 引起，用户切换到另一个 id 时，应该自动重试而不是继续显示旧错误。

useErrorBoundary / useErrorHandler：手动抛错

回忆上一篇的陷阱：错误边界捕获不到事件处理和异步里的错误。这个库提供 Hook，让你在这些场景里手动把错误"交给"最近的边界：

把异步错误转交给错误边界
import { useErrorBoundary } from 'react-error-boundary';

function UserProfile() {
  const { showBoundary } = useErrorBoundary();

  const load = async () => {
    try {
      await fetchUser();
    } catch (e) {
      showBoundary(e);  // 交给最近的 ErrorBoundary 显示 fallback
    }
  };
  // 效果等价于上一篇"存进 state 再抛"的技巧，但由库封装好了
}
注意版本 API 名字变过。老版本叫 useErrorHandler()（返回一个 handleError 函数），新版本改成 useErrorBoundary()（返回 { showBoundary, resetBoundary }）。文档、旧代码里两种都能见到，行为一致，注意对齐你项目里的版本。

对比：手写 vs 库

维度手写 class 边界react-error-boundary
底层实现class 组件仍是 class（无法绕过），但对外是 props 接口
fallback 拿 error要自己塞进 state自动作为 prop 传入
重试/重置手动实现resetErrorBoundary / resetKeys 开箱即用
异步/事件错误手动"存 state 再抛"useErrorBoundary().showBoundary
依赖零依赖约 2KB，可忽略

与其他知识点的联系

  • 错误边界(上一篇)：本篇反复强调"库没有魔法,底层仍是 class 组件"——如果没先理解上一篇 getDerivedStateFromError/componentDidCatch 的原始分工和"只能捕获渲染期错误"的边界,就无法理解这个库到底在封装什么、useErrorBoundary 又是怎么把异步/事件错误"转交"回渲染流程的。
  • 自定义 Hook：useErrorBoundary/useErrorHandler 是自定义 Hook 思路在错误处理领域的应用——它把"手动 try/catch 再 setState 再抛出"这一套繁琐操作封装成一行 showBoundary(e) 调用,这正是自定义 Hook 的核心价值:把重复的有状态逻辑抽出来复用。
  • 受控组件的实时状态思想：resetKeys 自动重置的机制,本质是"某个依赖值变化就触发某种行为"的响应式思路,这和受控组件里"state 变化就重渲染"是同一种声明式编程哲学——你只需要声明"当 userId 变化时应该怎样",不需要手动命令式地清理错误状态。
  • 组合 vs 继承(67 篇)：这个库把错误边界的"类组件实现细节"包裹在干净的 props 接口(FallbackComponent/onReset/onError)之后,让使用者以组合的方式使用它(像用 children 包裹任意子树),而不需要关心它内部是不是继承了 React.Component——这是"组合优于继承"在库设计层面的体现:继承细节被封装,暴露出来的是组合友好的接口。

小结

  • 库没有魔法：底层还是 class 错误边界，只是把样板、fallback 传参、重置重试都封装成了干净的 API。
  • <ErrorBoundary> 用 FallbackComponent 显示降级 UI，用 onError 上报，用 onReset + resetKeys 支持重试。
  • useErrorBoundary().showBoundary(e) 解决了原生边界"捕获不到异步/事件错误"的痛点。
  • 结论：新项目里想用错误边界，直接上这个库，别再手抄 class 样板。
