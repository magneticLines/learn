Query 参数与 useSearchParams：URL 里的可选筛选条件

列表页的分页、排序、搜索关键词、筛选标签……这些"可选、可组合、非必需"的参数不适合放进路径，而应放在 URL 的 ? 后面。v6 用 useSearchParams 读写它们，API 就是浏览器原生的 URLSearchParams。

Query 参数长什么样

URL 中 ? 之后、# 之前的部分是 query string（查询字符串），由 key=value 用 & 连接：

一个带 query 的 URL
https://shop.com/products?category=book&sort=price&page=2
//                          └──────────── query string ────────────┘
// category=book, sort=price, page=2 —— 三个筛选/分页条件

它天生适合表达"同一个页面的不同视图状态"：还是那个商品列表页，只是筛选、排序、翻页不同。把这些状态放进 URL 的好处是——可分享、可收藏、可前进后退、刷新不丢。

useSearchParams:读写 query

它的设计刻意模仿了 useState：返回一个 [值, setter] 数组。区别是"值"是一个 URLSearchParams 实例，setter 会把新参数写回 URL并触发重渲。

读取 query 参数
import { useSearchParams } from 'react-router-dom';

function ProductList() {
  const [searchParams, setSearchParams] = useSearchParams();

  // .get() 读单个值；不存在时返回 null（不是 undefined）
  const category = searchParams.get('category');   // 'book' 或 null
  // 常配合 ?? 给默认值。注意：和 params 一样，取出来是字符串
  const page = Number(searchParams.get('page') ?? '1');
  // .getAll() 读同名多值：?tag=a&tag=b → ['a', 'b']
  const tags = searchParams.getAll('tag');

  return <div>...</div>;
}
写入 query 参数——写回即改 URL 并重渲
// ① 传对象：整体替换所有 query（简单直接）
setSearchParams({ category: 'book', page: '2' });
// URL 变成 ?category=book&page=2

// ② 传函数：基于旧参数增量修改（保留其他参数时用它，最安全）
function goToPage(n) {
  setSearchParams(prev => {
    prev.set('page', String(n));  // 只改 page，category/sort 原样保留
    return prev;
  });
}
陷阱一:传对象会覆盖掉所有其他参数。 setSearchParams({ page: '2' }) 会把 URL 变成只剩 ?page=2，之前的 category、sort 全没了。想在保留其他参数的前提下改一个，用函数式写法 prev => { prev.set('page', ...); return prev; }。这和列表分页里"翻页但保留筛选条件"的常见需求直接相关。

陷阱二:值必须是字符串。 和 useParams 一样，query 里没有数字类型，写入要 String(n)，读取要 Number(...)。
为什么是 URLSearchParams 而不是普通对象? searchParams 是浏览器原生的 URLSearchParams 实例，不是一个 {}。所以你用的是 .get()/.getAll()/.has()/.set() 这些方法，不能直接 searchParams.category 或解构 const { category } = searchParams——那样拿不到值。React Router 复用这个原生标准，好处是行为可预测、和后端/其他工具统一，且天然支持同名多值（getAll）。想一次性转成普通对象可以用 Object.fromEntries(searchParams)（但会丢掉同名多值）。

动态段 vs Query 参数:何时用哪个

这是设计路由时的核心判断，也是常见面试题。一句话：决定"是哪个资源"的用动态段（路径），描述"怎么看这个资源"的用 query。

动态段 :id（路径）Query 参数 ?k=v
语义标识"哪一个资源"修饰"资源的呈现方式"
是否必需必需（缺了就是另一条路由）可选、可缺省
是否有序/可组合顺序固定、位置绑定无序、可任意组合多个
典型例子/users/42、/posts/abc?page=2&sort=price&q=手机
读取useParamsuseSearchParams
速记: 详情页看哪个用户 → /users/:id（动态段）；列表页第几页、怎么排、搜什么 → ?page=2&sort=name&q=xxx（query）。两者常常同时出现，如 /users/42/orders?status=paid&page=3。

典型场景:可分享的搜索/筛选页

把搜索状态放进 URL——刷新不丢、能分享
function SearchPage() {
  const [params, setParams] = useSearchParams();
  const q = params.get('q') ?? '';

  return (
    <input
      value={q}
      {/* 受控输入直接把值同步进 URL：state 的唯一来源就是 URL */}
      onChange={e => setParams(e.target.value ? { q: e.target.value } : {})}
    />
  );
}
// 好处：把 ?q=手机 这个链接发给别人，对方打开就是同样的搜索结果

小结

  • Query 参数（?k=v&k2=v2）表达"同一页面的不同视图状态"：分页、排序、搜索、筛选。
  • useSearchParams 仿 useState，返回 [searchParams, setSearchParams]；值是 URLSearchParams 实例，用 .get()/.getAll() 读，不能直接点属性。
  • 写入传对象会整体覆盖；要保留其他参数用函数式 prev => { prev.set(...); return prev; }。
  • 值都是字符串，读写时按需 Number()/String()。
  • 取舍: "是哪个资源"用动态段，"怎么呈现资源"用 query；两者常同时出现。

与其他知识点的联系

  • 动态段与 useParams：两者是"传参三件套"里最先出现的一对——动态段负责"哪个资源"，query 负责"怎么呈现"，同一个页面常常同时用到，比如 /users/:id/orders?status=paid&page=2。理解它们的分工，才能在设计路由时不把该用 query 的东西硬塞进路径。
  • 路由 state 传参（下一篇）：query 和 state 都是"辅助数据"，但可见性完全相反——query 写进 URL，可分享、刷新不丢；state 藏在 History 里，刷新即丢。选型时要问自己：这份数据用户愿不愿意在地址栏看到、别人分享链接时要不要带上，答案决定用 query 还是 state。
  • createBrowserRouter 与 loader：数据路由的 loader 函数会拿到 request 对象，其中 new URL(request.url).searchParams 就是服务端等价物——在 loader 里读 query 参数来决定请求哪页数据，本质上和组件里 useSearchParams().get() 是同一套 query 语义，只是读取的位置从组件搬到了路由配置层。
  • Navigate 重定向：如果要在重定向时保留原有的筛选/分页状态（比如未登录用户带着 ?page=3 被拦截后又放回来），需要手动把 location.search 拼接到 to 里——这是 query 参数和守卫重定向结合时最容易被忽略的细节。
  • 404 页面：通配路由 path="*" 只关心路径部分是否匹配，完全不管 query 参数——用户带着任意 ?foo=bar 访问一个不存在的路径段，依然会命中 404，这也说明 query 参数从设计上就是"路由匹配之外的附加信息"，不参与路由的匹配逻辑本身。
