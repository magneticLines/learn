JSX 编译原理：它实际上是函数调用

JSX 最终变成什么？是真实 DOM 吗？是字符串模板吗？都不是——它是普通的 JS 函数调用，返回一个普通的 JS 对象。理解这一点，是看懂 React 一切行为的钥匙。

编译产物：React.createElement()

Babel 把每一个 JSX 标签都翻译成 React.createElement() 调用，签名是：

React.createElement 的签名
React.createElement(
  type,      // 标签类型：字符串（'div'）或组件（MyComponent）
  props,     // 属性对象：{ className: 'card', onClick: fn } 或 null
  ...children // 子节点：字符串、数字、另一个 createElement 调用...
)

它返回的不是真实 DOM 元素，而是一个描述 UI 结构的普通 JS 对象——即"虚拟 DOM 节点"（Virtual DOM Node）。

并排对比：你写的 JSX ↔ Babel 编译后的 JS

你写的 JSX
const el = (
  <div className="card">
    <h1>标题</h1>
    <p>正文内容</p>
  </div>
);
Babel 编译后的 JS
const el = React.createElement(
  'div',
  { className: 'card' },
  React.createElement(
    'h1', null, '标题'
  ),
  React.createElement(
    'p', null, '正文内容'
  )
);

注意看：每一层嵌套都变成了一层函数调用的嵌套。JSX 只是让你以更直观的方式写出这堆括号。

createElement 返回什么：虚拟 DOM 对象

React.createElement 的实际返回值（控制台可以 console.log 看到）
// 这行 JSX：
const el = <h1 className="title">Hello</h1>;

// el 实际上是这个 JS 对象：
{
  $$typeof: Symbol(react.element),  // React 内部标记，用于安全校验
  type: 'h1',                        // 标签类型
  props: {
    className: 'title',
    children: 'Hello'                // 子节点被合并进 props.children
  },
  key: null,                          // 列表渲染用的 key
  ref: null                           // ref 引用
}

// 这个对象是轻量的 JS 对象，不是真实 DOM——
// React 拿到它之后，才决定如何操作真实 DOM。
为什么要先产生虚拟 DOM 对象，而不是直接操作真实 DOM？
因为操作真实 DOM 代价昂贵（触发浏览器重排/重绘）。React 先在内存里比对新旧虚拟 DOM 对象（Diff 算法），找出最小变化，再批量更新真实 DOM——这就是 React 高效的核心秘密。

这解释了什么：JSX 为什么能像值一样使用

理解了"JSX 是函数调用，返回 JS 值"，下面这些行为就不奇怪了：

JSX 就是普通 JS 值——可以做一切值能做的事
// ✅ 赋值给变量（因为 createElement 有返回值）
const header = <h1>标题</h1>;

// ✅ 放进数组
const items = [<li>A</li>, <li>B</li>];

// ✅ 作为函数参数（props 就是这样传的）
function render(element) {
  // element 就是个普通对象，可以检查它的属性
  console.log(element.type);
}
render(<h1>Hello</h1>);  // 打印 'h1'

// ✅ 三元运算符（条件选择不同 JSX）
const ui = isAdmin ? <AdminPanel /> : <UserPanel />;

// ✅ 函数返回 JSX——这正是 React 组件的本质
function MyComponent() {
  return <div>我是组件</div>;  // return 一个 JS 对象
}

React 17+ 的变化：新 JSX 转换

React 17 之前，文件顶部必须写 import React from 'react'——因为 JSX 编译后要用 React.createElement，React 对象必须在作用域内。

React 17+ 引入了新的 JSX 转换，Babel 自动从 react/jsx-runtime 导入辅助函数，不再需要手动 import React。编译产物从：
React.createElement('div', null, 'Hello')
变为：
_jsx('div', { children: 'Hello' })

效果相同，只是语法略有差异，且 _jsx 由编译工具链自动注入。现在创建新项目，CRA、Vite 模板都默认用新转换——所以你不会再看到 "必须 import React" 的报错了。

为什么了解编译原理很重要

不只是为了"知道底层"这种虚荣心——了解编译产物能让你从 JS 逻辑角度推导所有 JSX 限制，而不是死记规则：

  • 为什么必须有一个根节点？因为函数只能 return 一个值，两个平级标签 = 两个 createElement 调用 = 两个返回值 = 语法错误
  • 为什么标签必须闭合？因为 Babel 解析 JSX 用的是 XML parser，XML 要求所有标签关闭
  • 为什么属性名用驼峰？因为属性变成了 JS 对象的键名，JS 对象键名约定驼峰
  • 为什么 class 变 className？因为 class 是 JS 保留字，不能作为对象键名直接使用
  • 为什么花括号里不能放 if？因为函数参数位置需要表达式（有值），if 语句没有值
终极记忆法：脑子里把每一段 JSX 替换成对应的 React.createElement(...)——如果替换后是合法 JS，那 JSX 就合法；如果替换后是非法 JS，那 JSX 就报错。这条规则解释了 JSX 的一切。

常见陷阱

  • 以为 JSX 是模板语言（类似 Vue 的 template 或 Handlebars）：JSX 没有独立的运行时解析过程，它在构建时就被 Babel 编译成普通函数调用。这意味着 JSX 里能用的"逻辑"完全由 JS 表达式的能力决定，而不是模板引擎提供了哪些指令（没有 v-if、没有 {{#each}}，只有 JS 本身的三元、&&、.map()）。
  • 在花括号里写 if/for 语句：因为 createElement 的参数位置要求是表达式（有返回值），而 if/for 是语句，没有值。这是编译原理直接决定的限制，硬记"JSX 花括号里不能写 if"不如理解"参数位置只能放表达式"来得深刻。
  • 忘记 JSX 必须有唯一根节点的真正原因，靠死记硬背：很多人记住"必须包一层"但不知道为什么，遇到需要多根节点却不能加 div 的场景（如 flex 布局、表格行）时想不到用 Fragment。根因是函数只能 return 一个值，多个平级标签编译后是多个 createElement 调用，语法上不允许一个函数同时返回多个值。
  • 混淆 React 17 前后的 JSX 转换差异：在教程或旧项目里看到必须 import React from 'react'，然后照抄到新项目里，遇到"新写法明明没 import React 却能用 JSX"就困惑。根源是新版 JSX transform（react/jsx-runtime）不再需要 React 在作用域内，这是编译工具链层面的变化，和 React 本身的 API 没关系。

面试高频考点

  1. JSX 本质上是什么？浏览器能直接执行 JSX 吗？—— JSX 是 React.createElement()（或新转换里的 _jsx()）调用的语法糖，浏览器不能直接执行 JSX，必须先经过 Babel 等工具编译成合法的 JS 才能运行。理解这一点后，JSX 里的所有限制（根节点唯一、标签必须闭合、属性驼峰）都能从"它就是函数调用"这个事实推导出来。
  2. 为什么组件必须返回单一根节点？如果要返回多个并列元素怎么办？—— 因为 JS 函数只能 return 一个值，多个并列的 JSX 标签编译后会变成多个独立的 createElement 调用，无法同时作为一个 return 语句的值。解决办法是用 <>...</> 或 <React.Fragment> 包裹，它们编译后只产生一个虚拟节点，但不会在真实 DOM 里留下多余标签。
  3. React 17 的新 JSX 转换解决了什么问题？—— 旧转换要求每个用 JSX 的文件手动 import React，即使代码里没有直接用到 React 这个标识符，只是为了让编译出的 React.createElement 能找到 React。新转换让 Babel 自动从 react/jsx-runtime 引入辅助函数（如 _jsx），减少了这个隐式依赖，也为未来 React 内部实现的调整留出了空间。

与其他知识点的联系

  • Babel 与前端工具链：本文讲的整套编译过程离不开 Babel（或 Vite/esbuild 内置的类似转换器）。JSX 从来不是浏览器能直接理解的语法，必须经过构建工具在开发/打包阶段转换成合法 JS——这也解释了为什么直接在浏览器里写 <div>Hello</div> 而不经过打包会报语法错误。
  • 虚拟 DOM 对象结构：本文展示的 { $$typeof, type, props, key, ref } 结构就是"虚拟 DOM"这个概念在代码层面的真实样子。它不是什么抽象概念，而是 createElement() 返回的一个普通对象——React 之后用这些普通对象在内存里做 Diff 比较，找出最小变更集再操作真实 DOM，这是 React 高性能渲染的基础。
  • JSX 语法规则（05）与 JSX 属性差异（07）：05、07 两篇讲的所有"规则"和"命名差异"，都是本文编译原理的表现形式。比如根节点唯一（函数只有一个返回值）、className 替代 class（JS 对象键名不能是保留字）——本文是这些规则共同的"第一性原理"，建议在学完 05/06/07 后回来看本文，会有一种"原来如此"的贯通感。
