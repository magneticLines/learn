JSX 编译原理：它实际上是函数调用

JSX 最终变成什么？是真实 DOM 吗？是字符串模板吗？都不是——它是普通的 JS 函数调用，返回一个普通的 JS 对象。理解这一点，是看懂 React 一切行为的钥匙。

编译产物：React.createElement()

Babel 把每一个 JSX 标签都翻译成 React.createElement() 调用，签名是：

React.createElement 的签名
React.createElement(
  type,      // 标签类型：字符串（'div'）或组件（MyComponent）
  props,     // 属性对象：{ className: 'card', onClick: fn } 或 null
  ...children // 子节点：字符串、数字、另一个 createElement 调用...
)

它返回的不是真实 DOM 元素，而是一个描述 UI 结构的普通 JS 对象——即"虚拟 DOM 节点"（Virtual DOM Node）。

并排对比：你写的 JSX ↔ Babel 编译后的 JS

你写的 JSX
const el = (
  <div className="card">
    <h1>标题</h1>
    <p>正文内容</p>
  </div>
);
Babel 编译后的 JS
const el = React.createElement(
  'div',
  { className: 'card' },
  React.createElement(
    'h1', null, '标题'
  ),
  React.createElement(
    'p', null, '正文内容'
  )
);

注意看：每一层嵌套都变成了一层函数调用的嵌套。JSX 只是让你以更直观的方式写出这堆括号。

createElement 返回什么：虚拟 DOM 对象

React.createElement 的实际返回值（控制台可以 console.log 看到）
// 这行 JSX：
const el = <h1 className="title">Hello</h1>;

// el 实际上是这个 JS 对象：
{
  $$typeof: Symbol(react.element),  // React 内部标记，用于安全校验
  type: 'h1',                        // 标签类型
  props: {
    className: 'title',
    children: 'Hello'                // 子节点被合并进 props.children
  },
  key: null,                          // 列表渲染用的 key
  ref: null                           // ref 引用
}

// 这个对象是轻量的 JS 对象，不是真实 DOM——
// React 拿到它之后，才决定如何操作真实 DOM。
为什么要先产生虚拟 DOM 对象，而不是直接操作真实 DOM？
因为操作真实 DOM 代价昂贵（触发浏览器重排/重绘）。React 先在内存里比对新旧虚拟 DOM 对象（Diff 算法），找出最小变化，再批量更新真实 DOM——这就是 React 高效的核心秘密。

这解释了什么：JSX 为什么能像值一样使用

理解了"JSX 是函数调用，返回 JS 值"，下面这些行为就不奇怪了：

JSX 就是普通 JS 值——可以做一切值能做的事
// ✅ 赋值给变量（因为 createElement 有返回值）
const header = <h1>标题</h1>;

// ✅ 放进数组
const items = [<li>A</li>, <li>B</li>];

// ✅ 作为函数参数（props 就是这样传的）
function render(element) {
  // element 就是个普通对象，可以检查它的属性
  console.log(element.type);
}
render(<h1>Hello</h1>);  // 打印 'h1'

// ✅ 三元运算符（条件选择不同 JSX）
const ui = isAdmin ? <AdminPanel /> : <UserPanel />;

// ✅ 函数返回 JSX——这正是 React 组件的本质
function MyComponent() {
  return <div>我是组件</div>;  // return 一个 JS 对象
}

React 17+ 的变化：新 JSX 转换

React 17 之前，文件顶部必须写 import React from 'react'——因为 JSX 编译后要用 React.createElement，React 对象必须在作用域内。

React 17+ 引入了新的 JSX 转换，Babel 自动从 react/jsx-runtime 导入辅助函数，不再需要手动 import React。编译产物从：
React.createElement('div', null, 'Hello')
变为：
_jsx('div', { children: 'Hello' })

效果相同，只是语法略有差异，且 _jsx 由编译工具链自动注入。现在创建新项目，CRA、Vite 模板都默认用新转换——所以你不会再看到 "必须 import React" 的报错了。

为什么了解编译原理很重要

不只是为了"知道底层"这种虚荣心——了解编译产物能让你从 JS 逻辑角度推导所有 JSX 限制，而不是死记规则：

  • 为什么必须有一个根节点？因为函数只能 return 一个值，两个平级标签 = 两个 createElement 调用 = 两个返回值 = 语法错误
  • 为什么标签必须闭合？因为 Babel 解析 JSX 用的是 XML parser，XML 要求所有标签关闭
  • 为什么属性名用驼峰？因为属性变成了 JS 对象的键名，JS 对象键名约定驼峰
  • 为什么 class 变 className？因为 class 是 JS 保留字，不能作为对象键名直接使用
  • 为什么花括号里不能放 if？因为函数参数位置需要表达式（有值），if 语句没有值
终极记忆法：脑子里把每一段 JSX 替换成对应的 React.createElement(...)——如果替换后是合法 JS，那 JSX 就合法；如果替换后是非法 JS，那 JSX 就报错。这条规则解释了 JSX 的一切。
