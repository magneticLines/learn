避免 Context 滥用：性能陷阱与取舍

Context 用起来太方便了，方便到很多人把它当"全局状态银弹"，把所有状态一股脑塞进一个大 Context。结果就是：动一个字段，整棵树跟着重渲。这一篇讲清楚 Context 的性能代价和正确用法。

核心陷阱：value 一变，所有消费者全重渲

回顾第 45 篇的关键行为：Provider 的 value 一旦变化，订阅这个 Context 的每一个组件都会重渲，而且穿透 React.memo。当你把频繁变化的状态放进 Context，就等于给整片子树装了个"重渲炸弹"。

大 Context: {user, theme, cart, ...}
→
改 cart 一项
→
只用 theme 的组件也被迫重渲
最隐蔽的坑：value 是内联对象。即使状态没变，只要 Provider 的父组件重渲，value={{ user, setUser }} 就生成新对象，Object.is 判"变了"，全体消费者陪跑。这是 Context 性能问题里最常见的一种。

Context 不是状态管理库

Context 只是一个"传输管道"，它本身不管理状态、不做局部订阅、不做选择性更新。它擅长的是"传"，不是"管"。

能力Context专门状态库(Redux/Zustand)
跨层级传值擅长擅长
细粒度订阅(只订阅用到的字段)不支持支持(selector)
避免无关重渲需手动拆分内建
中间件/时间旅行/持久化无有
心智负担低中~高

经验法则：少变、全局、读多写少的数据（主题、语言、登录用户、路由）用 Context 很合适；频繁变化、局部性强、需要细粒度订阅的数据（表单输入、列表过滤、复杂交互状态），Context 会带来性能问题，考虑局部 state 或专门状态库。

优化一：按变化频率拆分 Context

把"经常一起变"和"几乎不变"的数据分到不同 Context。让频繁更新只波及真正关心它的那部分组件。

❌ 一个大 Context
<AppCtx value={{
  user,      // 几乎不变
  theme,     // 偶尔变
  mousePos,  // 每帧都变!
}}>
// mousePos 一动，
// 全体消费者重渲
✅ 按频率拆分
<UserCtx value={user}>
 <ThemeCtx value={theme}>
  <MouseCtx value={mousePos}>
    ...
// mousePos 变只影响
// 订阅 MouseCtx 的组件

优化二：拆 state 与 dispatch

一个常见模式：把"会变的 state"和"永不变的更新函数（dispatch/setter）"分成两个 Context。只需要触发更新、不读数据的组件（比如一个"加购"按钮），订阅 dispatch Context 就永远不会因为 state 变化而重渲。

state 与 dispatch 分离
const StateContext = createContext(null);
const DispatchContext = createContext(null);

function Provider({ children }) {
  const [state, dispatch] = useReducer(reducer, init);
  return (
    // dispatch 引用永远稳定 → 只用 dispatch 的组件不会因 state 变化重渲
    <DispatchContext.Provider value={dispatch}>
      <StateContext.Provider value={state}>
        {children}
      </StateContext.Provider>
    </DispatchContext.Provider>
  );
}

优化三：用 useMemo 稳定 value

当 value 必须是对象时，用 useMemo 缓存它，只有依赖真正变化才生成新对象，避免"父组件重渲导致 value 引用变化"的假更新。

useMemo 缓存 value
// ❌ 每次渲染都是新对象
<Ctx.Provider value={{ user, setUser }}>

// ✅ 只有 user 变才生成新对象（setUser 引用本来就稳定）
const value = useMemo(
  () => ({ user, setUser }),
  [user]
);
<Ctx.Provider value={value}>
面试高频："Context 会导致什么性能问题、怎么优化？" 标准答案就是这三招——拆分 Context（按变化频率）、拆 state/dispatch、useMemo 稳定 value。再补一句："Context 没有 selector 机制做不到细粒度订阅，重更新场景应考虑 Zustand/Jotai 等。" React 19 编译器能自动 memo 部分场景，但拆分 Context 的架构决策仍需人来做。

小结

  • Context 的更新是"全体消费者重渲"，且穿透 memo——频繁变化的数据放进去会拖垮性能。
  • Context 是传输管道不是状态管理库，没有细粒度订阅能力。
  • 三大优化：按频率拆分 Context、拆 state 与 dispatch、useMemo 稳定 value。
  • 选型：少变全局数据用 Context；重更新、需 selector 的场景用专门状态库。

与其他知识点的联系

  • createContext 与 Provider：本篇所有优化都建立在第 45 篇讲的那条关键行为之上——value 变化会让全部消费者重渲、且穿透 memo。不理解这条底层行为，就无法理解为什么"拆分 Context""拆 state/dispatch""useMemo 稳定 value"这三招分别针对的是什么问题：它们本质上都是在想办法减少"value 被判定为变化"的频率和范围。
  • useEffect 依赖数组与 useMemo：本篇"优化三：用 useMemo 稳定 value"和依赖数组篇讲的"对象每次渲染都是新引用"是同一个坑的另一种表现形式，解法也相通——依赖数组场景里 useMemo/useCallback 用来稳定传入 effect 或子组件的引用，这里则用来稳定传入 Provider value 的引用，思路完全一致。
  • reducer 函数模式（下一篇）：本篇"优化二：拆 state 与 dispatch"直接用到了下一篇才正式讲解的 useReducer——因为 dispatch 函数引用天生稳定（跨渲染不变），把它单独放进一个 Context，就能让"只需要触发更新、不需要读数据"的组件完全不受 state 变化影响。这是 useReducer 和 Context 组合使用的典型模式，也是全局状态管理常见的手写实现思路。
  • useRef 存储可变值：本篇强调"细粒度订阅、避免无关重渲"的诉求，本质上和 useRef 系列讲的"不影响 UI 的数据不该触发渲染"是同一种性能意识的不同应用场景——Context 的问题在于它没有内置手段区分"哪些消费者真正关心这次变化"，只能靠手动拆分或引入专门状态库来弥补这个缺陷。
