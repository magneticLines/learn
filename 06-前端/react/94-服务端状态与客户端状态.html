服务端状态 vs 客户端状态:一个被忽视的关键区分

上一篇讲了"哪些状态该全局"。但在全局状态里,还藏着一个决定性的分野:数据是从服务器来的,还是纯前端 UI 的? 把这两类混为一谈,是 React 应用里最普遍、最隐蔽的架构错误。

两种状态,本质完全不同

先看清楚它们各自是什么:

服务端状态(Server State)
来源:后端 API / 数据库
特点:
· 你不是唯一的拥有者
· 别人可能同时在改它
· 会"过期",需要重新拉
· 有 loading / error 态
· 需要缓存、重试、失效
例子:用户列表、订单、文章
客户端状态(Client State)
来源:用户在浏览器里的操作
特点:
· 你完全拥有它
· 只有本应用会改
· 永远"最新",不会过期
· 同步的,读写即时
· 不需要缓存/重试
例子:主题、语言、弹窗开合

关键差异在于"谁拥有这份数据的真相"。服务端状态的真相在数据库里,你手上的只是一份可能已经过时的副本(缓存);客户端状态的真相就在浏览器内存里,你说了算。

为什么区分它们如此重要

因为服务端状态天生带着一堆客户端状态根本不需要的难题:

  • 缓存:同一份数据别每个组件都请求一次。
  • 过期/失效(staleness):数据什么时候该重新拉?窗口重新聚焦时?每隔 30 秒?
  • 加载与错误态:请求中要转圈,失败要提示、要能重试。
  • 竞态(race condition):先发的请求后返回,别用旧结果覆盖新结果。
  • 后台同步:悄悄刷新,保持数据新鲜。

如果你用 useState + useEffect 手撸这一切,会写出大量重复且极易出错的样板代码。这正是 TanStack Query(React Query)、SWR 这类"数据请求库"存在的意义——它们专治服务端状态。

手撸服务端状态:样板多、坑多(反例)
function UserList() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
    // 缺:缓存?去重?窗口聚焦刷新?竞态处理?重试?
    // 全都得自己补,而且每个页面重复一遍
  }, []);
  // ...三个 state 手动拼装 UI
}
用 TanStack Query:缓存/重试/失效全内置(正例)
function UserList() {
  // 一行搞定:自动缓存、去重、后台刷新、重试、竞态处理
  const { data, isLoading, error } = useQuery({
    queryKey: ['users'],          // 缓存的键,相同 key 复用同一份数据
    queryFn: () => fetch('/api/users').then(r => r.json()),
  });

  if (isLoading) return <Spinner />;
  if (error) return <Error />;
  return <List data={data} />;
}

最常见的错误:把 API 数据塞进 Zustand / Redux。 很多人把接口返回的用户列表、订单直接存进全局仓库,然后手动写"什么时候刷新、失效、去重"。结果就是把状态库当成了一个残废的缓存来用——功能不全,还得手动维护过期逻辑。API 数据交给数据请求库,状态库只放纯客户端状态。
那 Zustand / Redux 到底还管什么? 引入 TanStack Query 后,你会发现全局状态库要管的东西少了一大半——因为原来塞进去的大部分其实是服务端状态。剩下真正属于客户端状态库的,通常只有:主题、语言、登录态、UI 偏好、跨页面的临时表单草稿、购物车(未提交时)等。所以本系列后面讲的 Zustand,定位就是轻量的客户端全局状态,而不是"什么都往里塞的大仓库"。

小结

  • 全局状态先分两类:服务端状态(API 数据,有真正的远端真相)和客户端状态(纯 UI,浏览器里的真相)。
  • 服务端状态天生要处理缓存、过期、加载/错误、竞态、后台同步——手撸 useEffect 又累又易错。
  • 服务端状态用 TanStack Query / SWR;客户端状态用 useState / Context / Zustand。
  • 把 API 数据塞进 Zustand/Redux 是高频反模式:等于用状态库当残废缓存。
  • 正确分工后,客户端状态库要管的东西会少很多——这也解释了为什么 Zustand 这种"轻量库"就够用。
