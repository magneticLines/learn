服务端状态 vs 客户端状态:一个被忽视的关键区分

上一篇讲了"哪些状态该全局"。但在全局状态里,还藏着一个决定性的分野:数据是从服务器来的,还是纯前端 UI 的? 把这两类混为一谈,是 React 应用里最普遍、最隐蔽的架构错误。

两种状态,本质完全不同

先看清楚它们各自是什么:

服务端状态(Server State)
来源:后端 API / 数据库
特点:
· 你不是唯一的拥有者
· 别人可能同时在改它
· 会"过期",需要重新拉
· 有 loading / error 态
· 需要缓存、重试、失效
例子:用户列表、订单、文章
客户端状态(Client State)
来源:用户在浏览器里的操作
特点:
· 你完全拥有它
· 只有本应用会改
· 永远"最新",不会过期
· 同步的,读写即时
· 不需要缓存/重试
例子:主题、语言、弹窗开合

关键差异在于"谁拥有这份数据的真相"。服务端状态的真相在数据库里,你手上的只是一份可能已经过时的副本(缓存);客户端状态的真相就在浏览器内存里,你说了算。

为什么区分它们如此重要

因为服务端状态天生带着一堆客户端状态根本不需要的难题:

  • 缓存:同一份数据别每个组件都请求一次。
  • 过期/失效(staleness):数据什么时候该重新拉?窗口重新聚焦时?每隔 30 秒?
  • 加载与错误态:请求中要转圈,失败要提示、要能重试。
  • 竞态(race condition):先发的请求后返回,别用旧结果覆盖新结果。
  • 后台同步:悄悄刷新,保持数据新鲜。

如果你用 useState + useEffect 手撸这一切,会写出大量重复且极易出错的样板代码。这正是 TanStack Query(React Query)、SWR 这类"数据请求库"存在的意义——它们专治服务端状态。

手撸服务端状态:样板多、坑多(反例)
function UserList() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
    // 缺:缓存?去重?窗口聚焦刷新?竞态处理?重试?
    // 全都得自己补,而且每个页面重复一遍
  }, []);
  // ...三个 state 手动拼装 UI
}
用 TanStack Query:缓存/重试/失效全内置(正例)
function UserList() {
  // 一行搞定:自动缓存、去重、后台刷新、重试、竞态处理
  const { data, isLoading, error } = useQuery({
    queryKey: ['users'],          // 缓存的键,相同 key 复用同一份数据
    queryFn: () => fetch('/api/users').then(r => r.json()),
  });

  if (isLoading) return <Spinner />;
  if (error) return <Error />;
  return <List data={data} />;
}

最常见的错误:把 API 数据塞进 Zustand / Redux。 很多人把接口返回的用户列表、订单直接存进全局仓库,然后手动写"什么时候刷新、失效、去重"。结果就是把状态库当成了一个残废的缓存来用——功能不全,还得手动维护过期逻辑。API 数据交给数据请求库,状态库只放纯客户端状态。
那 Zustand / Redux 到底还管什么? 引入 TanStack Query 后,你会发现全局状态库要管的东西少了一大半——因为原来塞进去的大部分其实是服务端状态。剩下真正属于客户端状态库的,通常只有:主题、语言、登录态、UI 偏好、跨页面的临时表单草稿、购物车(未提交时)等。所以本系列后面讲的 Zustand,定位就是轻量的客户端全局状态,而不是"什么都往里塞的大仓库"。

小结

  • 全局状态先分两类:服务端状态(API 数据,有真正的远端真相)和客户端状态(纯 UI,浏览器里的真相)。
  • 服务端状态天生要处理缓存、过期、加载/错误、竞态、后台同步——手撸 useEffect 又累又易错。
  • 服务端状态用 TanStack Query / SWR;客户端状态用 useState / Context / Zustand。
  • 把 API 数据塞进 Zustand/Redux 是高频反模式:等于用状态库当残废缓存。
  • 正确分工后,客户端状态库要管的东西会少很多——这也解释了为什么 Zustand 这种"轻量库"就够用。
  • 下一篇:如果不想引第三方库,能不能用 Context + useReducer 自建一个轻量全局状态?

常见陷阱

  • 把 API 返回的数据直接存进 Redux/Zustand,当成"一份状态"管理:这是文中提到的最典型反模式。服务端数据一旦进了全局仓库,就得自己手写"什么时候算过期""要不要重新拉""多个组件同时请求同一份数据要不要去重"这些逻辑,等于把状态库当成一个功能残缺的缓存来用,还得不断给它"打补丁"。
  • 手写 useEffect + fetch 时忘记处理竞态(race condition):比如搜索框每次输入都发一次请求,如果慢的那次请求碰巧比快的晚返回,结果就会用旧数据覆盖新数据,界面显示的是过时的搜索结果。React Query 这类库内置了"只用最新一次请求结果"的机制,手撸时必须自己用请求 id 比对或 AbortController 处理。
  • 只处理了 loading,漏了 error 态或重试逻辑:手写服务端状态时,很多人只写了 data 和 loading 两个 state,请求失败后页面卡在 loading 或者直接空白,用户不知道发生了什么、也没法重试。服务端状态天然会失败(网络问题、服务端错误),错误处理不是可选项。
  • 没意识到服务端状态会"过期",一直展示陈旧数据:客户端状态永远是最新的(你自己改的),但服务端状态代表的是数据库在某个时间点的快照,别的用户或别的操作可能已经改变了它。如果页面长时间停留不刷新、也不做失效处理,用户看到的可能是几分钟前甚至更久之前的过时数据而不自知。

面试高频考点

  1. 服务端状态和客户端状态的本质区别是什么?—— 核心在于"数据的真相在哪"。服务端状态的真相存在后端数据库里,前端拿到的只是一份可能过期的缓存副本,别的客户端或操作可能同时在修改它;客户端状态的真相就在浏览器内存里,只有当前应用会修改它,永远是最新的、同步可读写的,不需要缓存和失效机制。
  2. 为什么不建议把服务端数据直接放进 Redux/Zustand 这类客户端状态库?—— 因为服务端数据天生需要缓存、去重、过期失效、loading/error 态、竞态处理、后台刷新这一整套机制,而客户端状态库压根不管这些,全部要开发者手写补齐,结果就是重新发明一个功能不全的缓存系统,还容易出现多组件重复请求、数据不同步等问题。这些问题应该交给专门的数据请求库(TanStack Query、SWR)处理。
  3. TanStack Query(React Query)这类库具体解决了手写 useEffect+fetch 的哪些问题?—— 主要是:按 queryKey 自动缓存和去重(同一份数据多个组件用不会重复请求)、自动处理竞态(只采用最新请求的结果)、内置 loading/error 状态、失效和重新拉取策略(如窗口重新聚焦时刷新)、失败自动重试。这些都是手写代码里最容易被忽略或写错的部分。
  4. 引入数据请求库之后,原本的客户端状态库(如 Zustand)还剩下管什么?—— 剩下的是真正的纯客户端 UI 状态:主题、语言、登录态标记、UI 偏好、跨页面临时表单草稿、未提交的购物车等——这些数据的真相本来就在浏览器里,不涉及"缓存服务端数据"的问题。分工清楚后,客户端状态库要管理的状态量通常会显著减少。

与其他知识点的联系

  • 本地 state 与全局 state(第 93 篇):上一篇解决的是"该不该全局化",本篇是对"全局"这个大类做的二次拆分——服务端状态和客户端状态本质上不该用同一种思路管理,这也是为什么"全局状态"不能一概而论地都塞进一个状态库。
  • useLoaderData 与 useActionData(第 92 篇):数据路由里的 loader/action 本质上就是在手工实现服务端状态需要的那些能力(取数、刷新、错误处理),只是没有 TanStack Query 那样的精细缓存策略。理解了服务端状态的难点,才能明白为什么 React Router 要专门设计一套"渲染前取数、写后自动重跑"的机制。
  • Context 与 useReducer 轻量方案(第 95 篇):本篇明确划定了 Context/useReducer 的适用范围——它们只该用来管理客户端状态,如果把服务端数据也塞进 Context,会同时背上"手写缓存逻辑"和"Context 无选择性订阅导致大面积重渲染"两重负担。
  • Zustand 系列(第 96-99 篇):本篇是理解"Zustand 该管什么"的前提——Zustand 定位为轻量的客户端状态库,正是因为服务端状态已经被 TanStack Query 分走了大半江山,后续 Zustand 的所有讲解(创建、读写、切片、持久化)都建立在"store 里只放纯客户端数据"这个边界之上。
  • persist 持久化(第 99 篇): persist 中间件解决的是"客户端状态如何跨会话保留",这个问题在服务端状态那边完全不存在——服务端数据的"真相"本来就在数据库里,刷新页面重新拉取即可,不需要也不该用 localStorage 硬存一份可能过期的副本。
