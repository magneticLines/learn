React · 列表渲染

用 array.map() 把数据变成一列元素

界面上的列表——商品、评论、待办——背后都是一个数组。React 里没有 v-for 这样的模板指令，渲染列表就用你已经会的 JS：map()。理解"数据数组 → 元素数组"这个映射，是 React 列表渲染的全部核心。

核心：JSX 能直接渲染"元素数组"

上一篇讲过，JSX 的 {} 里可以放数组，React 会把数组摊平、逐个渲染。所以只要能把"数据数组"变成"JSX 元素数组"，塞进 {} 就能渲染出列表。而把数组一对一转换成新数组，正是 map() 的天职：

数据数组
[{id,name}, ...]
→
.map(转换)
→
元素数组
[<li/>, <li/>, ...]
→
{} 渲染
标准列表渲染
function UserList({ users }) {
  return (
    <ul>
      {/* map 把每个 user 对象映射成一个 
  •  元素 */}
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
      {/* 结果：users.map 返回 [
  • , 
  • , ...]，React 逐个渲染 */}
    </ul>
  );
}
    • 注意箭头函数的括号。上面用 user => ( ... ) 圆括号包裹 JSX，是"隐式返回"——箭头函数直接返回括号里的表达式。如果你写成 user => { ... } 花括号，那就是函数体，必须显式 return，否则返回 undefined，列表全空。这是初学者常踩的坑：花括号里忘了 return。

    为什么用 map 而不是 forEach？

    这是理解的关键，也是高频面试点。map 和 forEach 都能遍历数组，但本质不同：

    ✅ map —— 返回新数组
    {users.map(u =>
  <li key={u.id}>{u.name}</li>
)}
// map 收集每次返回值，
// 组成一个新数组交给 JSX
    ❌ forEach —— 返回 undefined
    {users.forEach(u =>
  <li key={u.id}>{u.name}</li>
)}
// forEach 没有返回值(undefined)
// JSX 里啥都渲染不出来

    根本区别：map 会把每次回调的返回值收集成一个新数组并返回；forEach 只是"执行动作"，返回值恒为 undefined。JSX 需要的是"一个元素数组"这个值——回顾第 28 篇，{} 里必须是能求出值的表达式。map(...) 求值为数组，能渲染；forEach(...) 求值为 undefined，什么都不显示（undefined 不渲染）。

    联系函数式编程：这背后是"声明式 vs 命令式"的思维差异。forEach 是命令式的——"对每一项，去做某事"（副作用）。map 是声明式的——"这个数据数组，等于那个元素数组"（转换/映射）。React 整体是声明式的（UI = f(state)），列表渲染自然选 map：你描述"数据长什么样就映射成什么样的 UI"，而不是"一步步命令怎么把元素塞进 DOM"。同理，其他返回数组的方法如 filter（先筛后渲）也能直接用在 JSX 里。

    常见组合：filter + map

    因为它们都返回新数组，可以链式组合——先过滤再映射，是实战中最常见的写法：

    只渲染已完成的待办
    {todos
  .filter(t => t.done)          // 先筛掉未完成的，得到子数组
  .map(t => <li key={t.id}>{t.text}</li>)}  // 再映射成元素
    你已经注意到每个 <li> 上都有 key 了。这不是可选的——用 map 渲染列表时，每个元素必须有一个稳定唯一的 key，否则 React 会警告，且可能出现状态错位的 bug。key 是列表渲染真正的重点，下面两篇（第 32、33 篇）专门讲它的原理和陷阱。
    小结
    • 列表渲染 = 把数据数组用 map 转成元素数组，再放进 JSX 的 {} 里，React 自动摊平逐个渲染。
    • 必须用 map 不用 forEach：map 返回新数组（有值可渲染），forEach 返回 undefined（不渲染）。这是声明式思维的体现。
    • 用花括号函数体时别忘了 return；filter + map 可链式组合；每个元素都要有 key。
