useMemo 缓存昂贵计算

函数组件每次重渲染都会完整重新执行一遍函数体——这是第 123 篇讲过的重渲染机制的直接推论。这意味着组件内任何一段计算,不管多贵,只要组件重渲染就会重跑一次。useMemo 让你把"计算结果"缓存下来,依赖不变就直接复用,不用重新算。

原理:依赖不变则跳过重新计算

useMemo(calcFn, deps) 在组件首次渲染时执行 calcFn 拿到结果并缓存;之后每次重渲染,React 会用和 memo 浅比较相同的方式(Object.is)比较这次的 deps 数组和上次的 deps 数组——逐项比较,如果每一项都相等,就直接返回上次缓存的结果,calcFn 根本不会被重新调用;只要有一项变了,才重新执行 calcFn 算出新结果并更新缓存。

→

→
相同:复用缓存结果

→
比较新旧 deps
→
不同:重新执行计算

什么时候真正需要:计算量大 + 每次渲染都会重跑

典型场景:对一个较大的数组做排序、过滤、聚合统计,而这个数组本身不常变化,但组件会因为其他无关状态(比如输入框的值、某个开关)频繁重渲染。

反例:每次输入都重新排序一遍大数组
function ProductList({ products }) {  // products 有几千条
  const [keyword, setKeyword] = useState('');

  // 每敲一个字符,组件就因 keyword 变化而重渲染一次
  // 而这行排序逻辑,不管 keyword 变没变,每次渲染都会重新跑一遍完整排序
  const sorted = [...products].sort((a, b) => a.price - b.price);

  return (
    <>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />
      {sorted.filter(p => p.name.includes(keyword)).map(p => <Item key={p.id} {...p} />)}
    </>
  );
}

问题:排序是 O(n log n),products 根本没变,但只因为用户敲了一个字符导致 keyword 变化、组件重渲染,这几千条数据的排序就要白白重跑一遍——每次按键都可能感觉到明显卡顿。

正例:useMemo 缓存排序结果,keyword 变化不触发重新排序
function ProductList({ products }) {
  const [keyword, setKeyword] = useState('');

  // 只依赖 products;products 没变时,keyword 怎么变都不会重新排序
  const sorted = useMemo(
    () => [...products].sort((a, b) => a.price - b.price),
    [products]
  );

  // filter 开销小,且必须依赖 keyword,不用额外缓存
  const filtered = sorted.filter((p) => p.name.includes(keyword));

  return (
    <>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />
      {filtered.map(p => <Item key={p.id} {...p} />)}
    </>
  );
}
注意这里的取舍: 排序开销大、依赖(products)不常变 → 值得缓存;过滤(filter)开销小、且每次输入都必须重新算 → 不用缓存,直接算更简单。不是把所有中间计算都塞进 useMemo,而是只缓存"贵、且依赖变化频率明显低于组件渲染频率"的那部分。

陷阱:给开销很小的计算加 useMemo,反而更亏

useMemo 本身不是免费的。 它需要:①在内存里保存上一次的 deps 数组和计算结果;②每次渲染都执行一次 deps 的逐项比较。如果被缓存的计算本身只是 a + b 或者 list.length 这种开销微乎其微的操作,那么"比较 deps 是否变化"的成本可能比"直接重新算一遍"还高——加了 useMemo 反而更慢、还多占了一份内存,纯粹是负优化。不是所有计算都值得缓存,这是 React 官方文档明确提醒过的高频面试考点。
不该用 useMemo
const total = items.length;
const label = `共 ${count} 项`;
const isEmpty = list.length === 0;
// 这些计算本身比"比较 deps"还便宜
该用 useMemo
const sorted = useMemo(
  () => bigArray.slice().sort(cmp),
  [bigArray]
);
// 排序/过滤/聚合大数据集,
// 且依赖变化频率明显低于渲染频率

useMemo 缓存对象引用,配合 memo 使用

回到上一篇的陷阱:传给 memo 子组件的对象/数组 prop,如果在渲染函数体内现场创建就会每次都是新引用。useMemo 除了缓存"计算结果",也常被用来缓存"对象引用本身",让下游的 memo 真正生效:

用 useMemo 稳定对象引用,配合 memo 组件
function App() {
  const [count, setCount] = useState(0);

  // style 对象只在依赖不变时保持同一个引用,ExpensiveList(memo)才能命中浅比较
  const style = useMemo(() => ({ color: 'red' }), []);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ExpensiveList style={style} />
    </>
  );
}

小结

  • 函数组件每次重渲染都会重新执行整个函数体,useMemo(calcFn, deps) 让 deps 不变时直接复用上次的计算结果,跳过重新计算。
  • 依赖比较方式和 memo 一致:逐项 Object.is 比较,任意一项变化就重新执行 calcFn。
  • 真正值得用的场景:计算开销大(排序/过滤/聚合大数据集)、且依赖变化频率明显低于组件渲染频率。
  • 陷阱:给开销微小的计算加 useMemo,比较 deps 的成本可能比直接重算还高,是负优化而非优化——不是所有计算都值得缓存。
  • useMemo 也常用来稳定对象/数组类型 prop 的引用,配合下游的 memo 组件才能真正跳过渲染。
  • 下一篇讲 useCallback——用同样的思路解决"函数类型"的引用稳定问题,和 useMemo 是近亲。

与其他知识点的联系

  • React.memo：本篇结尾提到的"用 useMemo 稳定对象引用配合 memo 使用"是两者最直接的联系——129 篇的陷阱是内联对象每次渲染都是新引用导致 memo 浅比较失效，useMemo 把这个对象缓存起来，依赖不变时返回同一个引用，memo 才能真正跳过渲染。两者缺一不可：memo 负责"比较并跳过"，useMemo 负责"让被比较的东西保持稳定"。
  • useCallback：下一篇 useCallback 本质上是 useMemo 的语法糖，等价于 useMemo(() => fn, deps)——区别只是 useMemo 缓存"调用计算函数后的返回值"（比如排序后的数组），useCallback 缓存"函数本身"而不去调用它。理解了 useMemo 的依赖比较机制（逐项 Object.is），useCallback 几乎不需要额外学习成本。
  • 状态下移与内容提升模式：132 篇提供了一种不依赖 useMemo 的替代思路——如果一个昂贵计算的触发源（比如某个 state）能通过状态下移放到更小范围的组件里，可能根本不需要缓存计算结果，因为无关的父组件不会重渲染，昂贵计算也就不会被无意义地重新触发。useMemo 是"事后缓存"，状态下移是"事前避免"。
  • useDeferredValue：136 篇的 useDeferredValue 解决的是"优先级调度"问题（紧急更新不被非紧急渲染拖慢），和 useMemo 解决的"避免重复计算"是完全不同的轴——两者可以组合：先用 useDeferredValue 让输入框保持流畅、把耗时渲染滞后，再用 useMemo 缓存基于滞后值的计算结果，避免滞后值没变时的重复运算。
  • 虚拟滚动 TanStack Virtual：本篇的排序/过滤例子处理的是"几千条数据要不要重新计算"，133 篇的虚拟滚动处理的是"几千条数据要不要一次性渲染成 DOM"——两者都是应对大数据集的性能手段，但作用层次不同：useMemo 省的是 CPU 计算时间，虚拟滚动省的是 DOM 节点创建和内存占用，实际项目里常常同时用到。
