布局路由:用无 path 的父路由承载共享外壳

上一篇的父路由带了 path="/app",所有子路由都被强制加上 /app 前缀。但很多时候,我们只想共享布局,不想改变 URL 结构——这就要用"无 path 的布局路由(layout route)"。

问题:共享布局却不想加 URL 前缀

设想一个后台:/dashboard、/orders、/users 都用同一套"顶栏 + 侧边栏"布局。如果按上一篇写 path="/app" 包起来,URL 就会变成 /app/dashboard——但产品要求 URL 就是 /dashboard,没有前缀。

解法:父路由只写 element,不写 path

React Router 允许 <Route> 只提供 element(布局),不提供 path。这种路由叫布局路由(layout route)——它不参与 URL 匹配,纯粹用来包裹一组子路由、提供共享外壳。

无 path 的布局路由:共享外壳但不影响 URL
<Routes>
  {/* 布局路由:没有 path,只提供 element */}
  <Route element={<MainLayout />}>
    {/* 子路由 path 直接是绝对路径的样子,无前缀 */}
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/orders"    element={<Orders />} />
  </Route>

  {/* 这个登录页不在上面的布局里,没有顶栏侧栏 */}
  <Route path="/login" element={<Login />} />
</Routes>

访问 /dashboard,URL 干净没前缀,但页面被 MainLayout 包着(顶栏 + 侧栏 + Outlet)。访问 /login,因为它不在布局路由内,渲染的是一个光秃秃的登录页。

核心价值:同一 URL 层级,不同布局分组

布局路由最大的用处是按布局把页面分组,而组的边界不体现在 URL 上。典型场景:后台页面共享侧栏布局,登录/注册页共享居中卡片布局,两组的 URL 都在根级。

多套布局:登录组 vs 后台组
<Routes>
  {/* 后台布局组 */}
  <Route element={<DashboardLayout />}>
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/users"     element={<Users />} />
  </Route>

  {/* 认证布局组:居中卡片,无侧栏 */}
  <Route element={<AuthLayout />}>
    <Route path="/login"    element={<Login />} />
    <Route path="/register" element={<Register />} />
  </Route>
</Routes>
/dashboard /users
→
DashboardLayout(侧栏)
/login /register
→
AuthLayout(居中卡片)

带 path vs 无 path 的父路由

带 path 的父路由无 path 的布局路由
写法<Route path="/app" element={...}><Route element={...}>
对 URL 的影响子路由加前缀 /app/xxx无前缀,URL 保持子路由原样
能否直接访问父路径能(/app,配 index 可显示内容)无父路径可访问
适用URL 就该带命名空间(如 /admin/*)只想共享布局、不动 URL 结构
布局路由也能承载守卫逻辑
布局路由不一定非要渲染视觉外壳。你可以写一个只做鉴权判断、内部返回 <Outlet /> 的"逻辑布局路由",用它把一组需要登录的页面包起来(下一节的认证守卫正是这么用的)。一个组件同时充当"布局 + 守卫"是很常见的组合。
别忘了 Outlet
布局路由的 element 组件里必须有 <Outlet />,否则子路由无处渲染,页面空白。这是从"带 path"改成"无 path"时最容易漏掉的一步——因为结构变了,人容易只盯着 path 改。

常见陷阱

  • 布局路由的 element 组件里忘了写 Outlet。从"带 path 的父路由"改成"无 path 的布局路由"时,人容易只关注路径去掉了没有,却忘了子路由渲染的位置仍然要靠 <Outlet /> 占位——结果访问子路径时布局壳子出来了,内容却是空白。
  • 以为无 path 的父路由自己也能被"访问"。带 path 的父路由(如 path="/app")本身是一段可匹配的 URL,配合 index 路由能直接展示内容;而无 path 的布局路由根本不参与 URL 匹配,没有对应的"父路径"可以访问,只能通过子路由的路径间接进入。
  • 把不需要共享布局的页面也塞进同一个布局路由组。比如把 /login 也放进带侧边栏的 DashboardLayout 包裹范围里,导致登录页也顶着一个后台侧栏,布局路由分组的边界应该严格按"UI 外壳是否相同"来划分,不是按功能模块随意归类。
  • 混淆"布局路由"与"嵌套路由"两个概念。嵌套路由描述的是父子路由的层级关系(子路由的内容渲染进父路由的 Outlet),布局路由只是"父路由不写 path"这一种特殊写法——嵌套路由不一定是布局路由(父路由可以带 path),布局路由一定是嵌套路由的一种用法。

面试高频考点

  1. 什么是布局路由(layout route)?它和普通嵌套路由的区别是什么?—— 布局路由是指父级 <Route> 只提供 element 不提供 path,它不参与 URL 匹配,纯粹用来给一组子路由提供共享的 UI 外壳;普通嵌套路由的父级带 path,会给子路由的 URL 加上前缀。
  2. 为什么布局路由不写 path 也能正常工作?React Router 是怎么匹配的?—— React Router 匹配路由时是按树形结构逐层匹配 path,没有 path 的节点视为直接"透传"匹配结果,不影响 URL 拼接,只负责在匹配成功后提供渲染外壳(内部通过 Outlet 渲染实际匹配到的子路由)。
  3. 如何用布局路由让不同分组的页面使用不同的公共外壳?—— 建多个不带 path 的 <Route element={...}>,各自包裹一组子路由(比如 DashboardLayout 包裹后台页面组,AuthLayout 包裹登录注册组),URL 层级不受影响,但渲染时会分别套上各自的布局组件。
  4. 布局路由除了做视觉外壳,还能做什么?—— 可以承载与 UI 无关的逻辑,比如鉴权守卫:组件内部做权限判断,通过则返回 <Outlet />,不通过则重定向,从而把"布局"与"守卫"合二为一,统一控制一组子路由的访问权限。

与其他知识点的联系

  • Outlet 占位符：布局路由只是"父路由不写 path"这一种特殊写法，它能生效的根本原因仍然是 Outlet 机制——父组件里必须有 <Outlet /> 才能承载子路由内容，这是本文反复强调"别忘了 Outlet"的原因，也是理解布局路由的前提知识。
  • index 路由：带 path 的父路由可以配 index 路由来填补"精确访问父路径"的空白，但无 path 的布局路由天生没有"父路径"这个概念，也就不存在给它配 index 的场景——这是判断"该用带 path 的父路由还是布局路由"时的一个隐性依据。
  • 认证守卫 PrivateRoute：本文提到"布局路由也能承载守卫逻辑"，下一节的 RequireAuth 正是把布局路由的"共享外壳"用途换成了"共享鉴权判断"——同一种路由结构，用途从视觉布局延伸到了逻辑控制，理解这一点能少走很多"守卫为什么要这样组织路由"的弯路。
  • Navigate 重定向：登录组和后台组这类多套布局的场景里，常常需要在某个布局路由的顶层做"默认跳转到某个子路由"，这时会用到 <Navigate> 配合 index 路由，三者组合起来才能覆盖"进入即跳转"的完整需求。
  • createBrowserRouter 与 loader：数据路由的配置对象里，布局路由表现为一个没有 path 字段、只有 element 和 children 的节点——写法从 JSX 变成对象，但"无 path 只提供外壳"的语义完全一致，说明布局路由是贯穿声明式路由和数据路由两套 API 的通用概念。
