Props 只读原则：为什么不能修改 props

这是 React 最重要的约束之一，也是整个数据模型的基石。初学者常常困惑"为什么我不能改 props"——理解了背后的设计逻辑，这条规则就会从"限制"变成"保护"。

原则：子组件绝对不能修改 props

无论是直接赋值、还是修改对象/数组的属性——只要是父组件传来的，子组件一律不能修改。React 文档原文是：

"All React components must act like pure functions with respect to their props."
——所有 React 组件对 props 的处理必须像纯函数一样：给定相同的 props，永远返回相同的输出，并且不修改输入。

为什么不能改：三个层面的原因

原因一：UI 不会更新

React 的更新机制依赖状态变化触发重新渲染。子组件直接改 props，只是改了 JavaScript 对象里的某个值，React 完全不知道这件事发生了——不触发重新渲染，UI 就不更新。你以为改了，页面上什么都没变。

原因二：数据所有权在父组件

类比现实：你租了一套公寓，可以在里面住，但不能拆墙改结构——房子的所有权不是你的，你没有权力做结构性改动。props 也一样，它是父组件的数据，父组件把它"出借"给子组件展示，子组件只有"使用权"，没有"修改权"。

原因三：共享 props 导致数据混乱

父组件可能把同一份数据传给多个子组件。如果某个子组件悄悄修改了它，其他子组件拿到的就是被污染的数据——而且这种 bug 极难追踪，因为修改发生在"别处"。

共享数据被意外修改——灾难场景
function App() {
  const user = { name: 'Shea', role: 'user' };

  // 把同一个 user 对象传给两个子组件
  return (
    <>
      <Header user={user} />
      <Sidebar user={user} />  // 和 Header 拿到的是同一个对象引用
    </>
  );
}

// 如果 Sidebar 偷偷改了 user.role
function Sidebar({ user }) {
  // ❌ 修改了 props 里的对象属性——user 是引用，改的是同一个对象
  user.role = 'admin';
  // Header 里的 user.role 也变成了 'admin'！它不知情，也不会重新渲染
  return <nav>{user.role}</nav>;
}

错误写法 vs 正确写法

❌ 错误：直接改 props
function Counter({ count }) {
  function handleClick() {
    // count 是 props，直接赋值：
    // 1. 只改了本地变量副本
    // 2. React 不感知，不重渲染
    // 3. 点按钮，UI 毫无反应
    count = count + 1;
  }
  return (
    <button onClick={handleClick}>
      {count}
    </button>
  );
}
✅ 正确：回调给父组件
function Counter({
  count,
  onIncrement  // 父组件传来的回调
}) {
  return (
    // 点击时调回调，通知父组件
    // 父组件修改 state → 重渲染
    // 新 count 作为 props 传来
    <button onClick={onIncrement}>
      {count}
    </button>
  );
}

想改数据怎么办：通过回调"申请修改"

子组件修改数据的正确姿势是：调父组件传来的回调函数，让数据的主人来决定怎么改。父组件改了自己的 state 后，React 重新渲染，新数据作为 props 传回来——闭环完成。

完整的"子改数据"正确流程
// ── 父组件：数据的主人，掌握修改权 ──
function App() {
  // 数据存在父组件的 state 里，子组件无权直接改
  const [count, setCount] = useState(0);

  // 修改数据的函数——父组件自己定义，然后作为 prop 传给子组件
  const handleIncrement = () => setCount(count + 1);

  return (
    <Counter
      count={count}             // 数据传给子组件展示
      onIncrement={handleIncrement}  // 修改方法传给子组件使用
    />
  );
}

// ── 子组件：只有展示权和"申请修改"权 ──
function Counter({ count, onIncrement }) {
  return (
    <div>
      <p>当前值：{count}</p>
      // 点击时"申请"父组件修改——自己不改，让父组件改
      <button onClick={onIncrement}>+1</button>
    </div>
  );
}

联系单向数据流：Props 只读是它的实现保障

单向数据流的完整闭环：

数据从父流向子（通过 props）→ 子组件展示数据 → 用户交互触发事件 → 子组件调回调函数（向上通知）→ 父组件修改 state → React 重新渲染 → 新数据通过 props 再次流向子组件。

数据永远向下流，事件永远向上传。Props 只读是让"数据向下流"这个方向不被破坏的约束。一旦子组件能随意改 props，数据流就从"单向"变成了"乱向"——组件树里任意一处都能改任意数据，追踪 bug 的成本指数级增长。
JavaScript 层面的注意：React 没有在运行时冻结 props 对象（严格模式下也没有），所以你技术上"可以"改——不会报错。但这是违反契约的行为，会带来难以追踪的 bug。TypeScript 把 props 标记为只读类型（Readonly<Props>），在编译阶段就能捕获这类错误——这是用 TypeScript 写 React 的好处之一。

快速判断：这种场景该用什么？

场景正确做法
子组件想展示 props 数据 直接读 props，渲染出来
子组件想修改一个数字/字符串 父组件传回调，子组件调回调
子组件想修改 props 里的对象属性 同上，绝对不能直接 props.user.name = '...'
子组件需要一个只给自己用的变量 用 useState 或普通的局部变量（不要碰 props）
子组件需要在 props 基础上加工数据 派生一个新变量：const fullName = firstName + ' ' + lastName;
本节核心：① props 只读是 React 最核心的约束，不是建议；② 子组件修改 props 会导致 UI 不更新、共享数据混乱，且 React 无法感知；③ 想修改数据：让父组件通过回调函数来改，数据的主人只有父组件；④ props 只读是单向数据流的实现基础——数据向下，事件向上，方向明确，追踪容易；⑤ 如果只是子组件内部需要的临时变量，用 state 或局部变量，不要动 props。
