短路渲染 {cond && <X/>}：优雅，但有个致命陷阱

"有未读消息时才显示小红点"——这种"满足条件才渲染，否则啥都不显示"的场景，用 && 短路最简洁。但如果条件写成一个数字，可能会把一个刺眼的 0 显示在页面上。

原理：&& 返回的是"值"，不是布尔

很多人以为 a && b 一定返回 true/false——错。JS 的 && 是短路求值，它返回的是其中一个操作数本身：

  • 如果左边是 falsy（假值），直接返回左边，右边根本不执行。
  • 如果左边是 truthy（真值），返回右边。
&& 返回的是操作数本身
true  && 'hi'   // → 'hi'  （左真，返回右边）
false && 'hi'   // → false （左假，直接返回左边，右边不执行）
0     && 'hi'   // → 0     （0 是 falsy，返回 0 本身！）
''    && 'hi'   // → ''    （空串也是 falsy，返回 ''）

结合上一篇讲的"React 不渲染 false"这个特性，短路渲染就成立了：

短路渲染的标准写法
function Inbox({ hasUnread }) {
  return (
    <div>
      {/* hasUnread 为 true → 渲染红点；为 false → React 不渲染 false */}
      {hasUnread && <span className="dot" />}
    </div>
  );
}

致命陷阱：数字 0 会被渲染出来！

这是面试和实战里都极高频的坑。假设你想"购物车有商品时显示列表"，很自然地写：

❌ count 为 0 时，页面上会出现一个刺眼的 0
function Cart({ items }) {
  const count = items.length;
  return (
    <div>
      {/* count = 0 时：0 &&  → 求值为 0 */}
      {/* React 会把数字 0 当作文本渲染出来！ */}
      {count && <List items={items} />}
    </div>
  );
}
为什么会显示 0？关键在于 0 是 falsy，所以 0 && <List/> 短路，整个表达式求值为 0 这个数字。而 React 的规则是：false/null/undefined 不渲染，但数字 0 是有效内容，会被当文本渲染。于是页面上凭空多出一个 "0"。

同理，如果左边是空字符串 ''，虽然渲染出来看不见（空文本节点），但逻辑上同样是"该隐藏时没隐藏"的隐患。NaN 更糟，会渲染出 "NaN"。

如何避免：把左边变成真正的布尔值

❌ 直接用数字/长度
{count && <List/>}
// count=0 → 渲染出 "0"

{list.length && <List/>}
// 空数组 length=0 → "0"
✅ 显式转布尔 / 用比较
{count > 0 && <List/>}
// 左边是 true/false，安全

{!!count && <List/>}
// !! 强制转布尔

{count ? <List/> : null}
// 三元，falsy 时给 null

三条修复思路，本质都是保证 && 左边是布尔值：

  • count > 0 && ...：用比较运算符，结果天然是布尔。最推荐，语义也最清晰（"数量大于 0 才显示"）。
  • !!count && ...：双感叹号把任意值转成布尔。简洁但语义弱一点。
  • count ? <List/> : null：改用三元，假分支显式给 null（null 不渲染）。当"假的时候要渲染别的东西"时这个最合适。
联系：这个坑的根源是 JS 的 falsy 值。JS 里 0、''、NaN、null、undefined、false 都是 falsy。&& 遇到 falsy 左值会原样返回它，而这几个里只有 0/''/NaN 会被 React 当"内容"处理（null/undefined/false 不渲染）。所以真正危险的是可能等于 0 的数字和可能为空串的字符串。养成习惯：短路渲染的左边，只放布尔表达式。
小结
  • && 返回操作数本身，不是布尔；配合"React 不渲染 false"实现"满足才渲染"。
  • 最大陷阱：左边是数字 0（如 count、arr.length）时，页面会渲染出一个 "0"。
  • 修复：让左边变成真布尔——首选 count > 0 && ...，或 !!x && ...，或改用三元 x ? ... : null。
