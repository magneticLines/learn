组合 vs 继承:React 的官方立场

从 Java/OOP 过来的同学,习惯用继承复用代码:抽个基类,子类 extends 它。但 React 官方明确表态:用组合,不用继承。这一篇讲清为什么,以及用组合怎么优雅地实现你想用继承达到的一切。

官方原话:组合能覆盖所有场景

React 官方文档明确写道:"在 Facebook,我们在成千上万个组件里使用 React,还没有发现任何需要用继承来构建组件层级的场景。" React 推荐用组合(composition)——把小组件拼装成大组件,而不是用继承(inheritance)——让组件去 extends 另一个组件。

为什么不用继承?

面向对象里继承常被诟病"耦合太紧":子类紧紧依赖父类的实现细节,父类一改子类就崩,这就是经典的脆弱基类问题。而 UI 的复用需求,几乎都是"把 A 放进 B 里"这种包含关系,而不是"A 是一种 B"的is-a 关系。

❌ 继承的思路(别这么做)
// 想复用 Dialog 的能力
class AlertDialog
  extends Dialog {
  // 依赖 Dialog 内部实现
  // Dialog 一改,这里就可能坏
  // 层级深了极难维护
}
✅ 组合的思路(推荐)
// 把 Dialog 当容器,内容传进去
function AlertDialog() {
  return (
    <Dialog title="警告">
      <p>确定删除?</p>
      <button>确定</button>
    </Dialog>
  );
}

问题在于:UI 组件之间几乎不存在真正的"是一种"关系。AlertDialog 不是"一种更具体的 Dialog 类型",它只是"一个用了 Dialog 外壳的东西"。用 children(见 65 篇)把内容塞进 Dialog,比继承清晰得多,也不会被父类实现细节绑架。

场景一:特例化组件(用组合代替"子类")

想要一个"更具体版本"的组件——OOP 里会写子类。React 里:写个新组件,内部渲染通用组件,把特定的 props 填好即可。这叫特例化(specialization)。

WelcomeDialog 是 Dialog 的"特例",但不是子类
// 通用组件:提供外壳 + 插槽
function Dialog({ title, children }) {
  return (
    <div className="dialog">
      <h1>{title}</h1>
      {children}
    </div>
  );
}

// "特例":通过组合(渲染 Dialog + 填 props),而非继承
function WelcomeDialog() {
  return (
    <Dialog title="欢迎">
      <p>感谢使用本产品!</p>
    </Dialog>
  );
}

场景二:复用逻辑(用 Hook,不用继承基类)

OOP 里常把公共逻辑放进基类让子类继承。React 里,共享逻辑用自定义 Hook(见 66 篇结尾):把有状态逻辑抽成 useXxx,任何组件调用即可,完全不需要继承关系。

逻辑复用靠 Hook,而不是基类
// 公共逻辑抽成 Hook,谁需要谁调用
function useOnlineStatus() { /* 订阅网络状态 */ }

function Avatar() {
  const online = useOnlineStatus();  // 一行复用,无继承
  return <img className={online ? 'on' : 'off'} />;
}

"组合优于继承" 不是 React 独创
这是软件设计的普遍原则(《设计模式》GoF 早就提出 "favor composition over inheritance"),Go 语言干脆取消了继承只保留组合。React 只是把它贯彻得非常彻底。继承带来的紧耦合、脆弱基类、层级僵化,在快速变化的 UI 里代价尤其高——组合的灵活拼装天然更适合 UI。
唯一"例外":类组件时代的 Component 基类
有人会说"class 组件不是 extends React.Component 吗,那不是继承?"——那是继承 React 框架提供的基类(为了拿到 setState 等能力),属于"继承框架 API"。而 React 反对的是在你自己的业务组件之间用继承来复用。何况函数组件 + Hooks 时代,连这个基类都不需要了。

常见陷阱

  • 把"特例化组件"误写成继承：从 OOP 转过来的人容易本能地想写 class WelcomeDialog extends Dialog，但函数组件根本没有这种继承机制。正确做法永远是"新组件内部渲染通用组件、把特定 props 填好"，靠组合而不是类继承实现"特化版"。
  • 误以为 class 组件 extends React.Component 就是在践行继承复用：这是继承 React 框架提供的基类以获取 setState/生命周期等能力，属于"继承框架 API"，跟"业务组件之间要不要用继承复用逻辑"是两件不同的事——React 反对的是后者。函数组件 + Hooks 时代，连前者也不需要了。
  • 用继承的思维去"包一层"而不是组合：比如想给 Dialog 加一个通用的边框样式，容易想着"写个基类 Dialog，再派生出带边框的版本"，实际上应该用组合——让"带边框"的组件把 children 渲染进一个通用的 Dialog 里，或者用 props 传入样式差异，而不是建立继承链。
  • 共享逻辑时求助"基类方法"而不是自定义 Hook：想在多个组件间共享"获取在线状态"这类逻辑,如果还在想"要不要抽个基类让组件继承",说明还没转过来 React 的思维——正确答案永远是抽成 useXxx 自定义 Hook,任何函数组件调用即可,无需任何继承关系。
  • 过度使用 children 插槽导致组件"看不出接口"：为了践行组合而无限制地把所有内容都塞进 children，会让组件本身失去清晰的 props 接口约束，调用者不看内部实现就不知道该传什么进去。组合不代表放弃类型约束，具名插槽（用具体命名的 props 承载 JSX）往往比笼统的 children 更清晰。

面试高频考点

  1. React 官方为什么不推荐用继承来复用组件？—— 因为 UI 组件之间几乎不存在真正的"is-a"（是一种）关系，绝大多数复用需求本质是"has-a / contains"（包含关系）——比如 AlertDialog 不是"一种特殊的 Dialog 类型"，而是"用了 Dialog 外壳的东西"。继承还会造成经典的脆弱基类问题：父类实现细节一变，所有子类都可能遭连带影响，耦合度高、维护成本大，组合能覆盖所有这些场景且更灵活。
  2. 如何用组合实现 OOP 里"子类特化父类行为"的效果？—— 写一个新组件，内部渲染通用组件（如 Dialog），把这个特化版本需要的 title、children 等具体内容通过 props 填好，对外只暴露这个新组件。这被称为"特例化（specialization）"：没有继承链，没有 extends，只是把通用组件当作可配置的容器来使用。
  3. 组合和继承在"共享逻辑"这件事上分别怎么做？哪个是 React 推荐的？—— OOP 继承会把公共逻辑放进基类，子类通过继承拿到这些方法；React 推荐把有状态逻辑抽成自定义 Hook（如 useOnlineStatus），任何函数组件直接调用即可获得同样的能力，不需要建立任何继承关系，也不会有基类被多个子类耦合的问题。
  4. "组合优于继承"是 React 独创的原则吗？—— 不是。这是软件设计领域的普遍原则，最早由 GoF 在《设计模式》里提出（"favor composition over inheritance"），Go 语言甚至直接取消了类继承只保留组合。React 只是把这一原则在 UI 场景里贯彻得非常彻底——因为 UI 需求变化快，继承的紧耦合和层级僵化在这种场景下代价格外高。

与其他知识点的联系

  • children 插槽模式(65 篇)：本篇"全景对照表"里把 children 列为"把内容放进外壳"这一复用需求的组合方案——65 篇讲的"组合优于配置"只是局部结论,本篇把它扩展成"组合优于继承"这个更高层的原则,children 是这个原则在 props 设计上最直接的落地。
  • render props 模式(66 篇)：对照表把 render props 列为"共享数据给渲染"对应 OOP "受保护成员"的组合方案——如果没先理解 render props 如何让组件把内部 state 交给使用者决定渲染什么,这里"共享数据"这一行就只是个抽象标签。
  • 自定义 Hook：本篇把"共享有状态逻辑"明确判给自定义 Hook,而不是继承基类——这是对 66 篇结尾"Hook 出现后 render props 退居二线"的呼应和收口,也解释了为什么现代 React 代码里很少再看到刻意设计的继承层级。
  • 错误边界(下一篇)：错误边界是本篇原则的一个例外确认——它恰恰必须写成 class 组件、继承 React.Component,但这属于"继承框架 API 以获取平台能力"(callout 里已经区分过),不是"业务组件之间用继承复用",两者并不矛盾,反而更凸显 React 反对的是后者。

小结

  • React 官方立场:用组合,不用继承——组合能覆盖所有 UI 复用场景。
  • UI 复用大多是"包含关系",不是"is-a 关系",继承的紧耦合、脆弱基类问题在 UI 里代价更高。
  • 想要"特化版组件":写新组件内部渲染通用组件 + 填好 props(特例化),而非 extends。
  • 复用内容用 children/具名插槽,复用数据用 render props,复用逻辑用自定义 Hook。
  • "组合优于继承"是通用软件原则,React 把它贯彻得最彻底。
