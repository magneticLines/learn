Zustand 持久化:persist 与水合

刷新页面,主题偏好、登录态、购物车全没了?persist 中间件把 store 自动同步到 localStorage,刷新后还在。但持久化引入了两个新概念:该存哪些字段(partialize)、状态怎么从存储恢复(水合 hydration),还有升级后的版本迁移。这一篇把坑一次讲清。

基础用法:用 persist 包一层

persist 是一个中间件——它包在你原本传给 create 的函数外面,自动做两件事:state 变化时写入存储、页面加载时从存储读回来。

persist 包裹:自动读写 localStorage
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

export const useSettings = create(
  persist(
    (set) => ({            // ← 原来的 store 定义,原样放进来
      theme: 'light',
      lang: 'zh',
      toggleTheme: () => set((s) => ({ theme: s.theme === 'light' ? 'dark' : 'light' })),
    }),
    {
      name: 'app-settings',  // localStorage 的键名,必填且要唯一
      // storage 默认是 localStorage;换 sessionStorage 见下
    }
  )
);

就这样,theme 一变,localStorage['app-settings'] 自动更新;下次打开页面,store 自动用存储里的值初始化。组件用法完全不变。

partialize:只持久化部分字段

默认 persist 会把整个 state 对象(除了函数)都存进去。但很多字段不该持久化——比如临时 UI 状态、或从服务端来的、每次都该重新拉的数据。partialize 让你精确挑选要存哪些:

partialize:白名单式挑选要持久化的字段
persist(
  (set) => ({
    theme: 'light',
    lang: 'zh',
    isSidebarOpen: false,   // 临时 UI 状态,不想持久化
    // ...
  }),
  {
    name: 'app-settings',
    // 只把 theme、lang 存进 localStorage,isSidebarOpen 不存
    partialize: (state) => ({ theme: state.theme, lang: state.lang }),
  }
)
为什么要 partialize? ① 存储空间有限,别把无关数据塞满 localStorage;② 有些状态持久化反而是错的——比如"弹窗是否打开""当前加载中",刷新后本就该重置。原则:只持久化用户真正希望"跨会话保留"的偏好(主题、语言、草稿),临时/派生/服务端状态一律不存。

水合(hydration):状态从存储恢复的过程

水合指的是"页面加载时,把 localStorage 里的值读出来、填回 store"这个过程。理解它能避开两个经典 bug。

页面加载
→
用初始默认值渲染
→
读 localStorage
→
填回 store(水合完成)

陷阱一:SSR / SSG 的水合不匹配(hydration mismatch)

Next.js 等 SSR 场景高频报错。 服务端没有 localStorage,只能用默认值渲染 HTML(如 theme: 'light');浏览器端读出 localStorage 里的 'dark' 再渲染——两次结果不一致,React 报 hydration mismatch。

解法:用 persist 提供的 onRehydrateStorage 或 useStore.persist.hasHydrated() 判断是否水合完成,水合前先渲染默认/占位 UI,水合后再显示真实值。纯客户端(CSR)应用一般不用担心这个问题。
等待水合完成再渲染依赖持久化值的 UI
function useHydrated() {
  const [hydrated, setHydrated] = useState(false);
  useEffect(() => setHydrated(useSettings.persist.hasHydrated()), []);
  return hydrated;
}
// 组件:if (!useHydrated()) return <Skeleton />;

版本迁移:store 结构变了怎么办

问题:用户浏览器里存的是旧结构的数据,你发新版把字段改了(比如 lang: 'zh' 改成 locale: 'zh-CN')。直接水合旧数据会导致结构对不上、出错。version + migrate 用来平滑升级:

version + migrate:把旧存储结构迁移到新结构
persist(
  (set) => ({ locale: 'zh-CN' }),
  {
    name: 'app-settings',
    version: 2,   // 当前版本;存储里记录的版本 < 它时触发 migrate
    migrate: (persisted, fromVersion) => {
      if (fromVersion === 1) {
        // v1 用的是 lang,把它转成 v2 的 locale
        return { locale: persisted.lang === 'zh' ? 'zh-CN' : 'en-US' };
      }
      return persisted;
    },
  }
)
不写 migrate 会怎样? 当 version 提高但没提供 migrate,persist 会丢弃旧的持久化数据、回退到初始默认值——不会崩,但用户偏好丢了。所以只要改了持久化字段的结构,就应该提升 version 并写 migrate。

常用配置速查

配置项作用
name存储键名,必填,须唯一
storage换存储介质,如 createJSONStorage(() => sessionStorage)
partialize挑选要持久化的字段(白名单)
version + migrate结构升级时迁移旧数据
onRehydrateStorage水合前后的回调,可处理 loading / 错误
skipHydration关闭自动水合,手动调 rehydrate()(SSR 常用)
回到全局主线: persist 让 Zustand 特别适合放"跨会话的客户端偏好"(主题、语言、未提交草稿)。但别忘了第 94 篇——服务端数据不该靠 persist 硬存,那是 TanStack Query 缓存 + 失效策略的活。持久化只是给客户端状态加一层"记住我"。

小结

  • persist 是中间件,包在 create 的函数外,自动把 state 同步到存储(默认 localStorage),刷新后恢复。
  • name 必填且唯一;可通过 createJSONStorage 换成 sessionStorage 等。
  • partialize 只持久化真正需要跨会话保留的字段,临时/派生/服务端状态别存。
  • 水合是"从存储读回填入 store"的过程:SSR 下要处理 hydration mismatch(用 hasHydrated/占位 UI),CSR 一般无忧。
  • 改了持久化结构就提升 version 并写 migrate,否则旧数据被丢弃、用户偏好丢失。
  • 持久化服务的是客户端状态;服务端数据仍交给 TanStack Query。至此 Zustand 篇结束,下一模块进入 Redux Toolkit 的 createSlice / createAsyncThunk。

与其他知识点的联系

  • Zustand 切片模式(第 98 篇):本篇开头提到"想统一持久化"是选切片模式的理由之一——因为 persist 中间件包裹的是整个 store,如果领域数据分散在多个独立 store 里,就得给每个 store 分别配置持久化,统一到一个 store 里再切片,才能一次性搞定持久化。
  • 服务端状态与客户端状态(第 94 篇):本篇反复回扣的边界——"服务端数据不该靠 persist 硬存"——直接继承自第 94 篇的结论。persist 解决的是客户端状态"跨会话保留"的需求,而服务端数据的正确做法是重新拉取或走 TanStack Query 的缓存策略,两者的"过期"含义完全不同。
  • partialize 与"能派生就别存"(第 93 篇): partialize 挑选要持久化的字段,和第 93 篇"能从别的 state 算出来的值不要单独存"背后是同一种"精简状态"的直觉——只是应用场景不同:一个是决定"要不要存这份 state",一个是决定"这份已经存在的 state 要不要写进 localStorage"。
  • useLoaderData 与 useActionData(第 92 篇):水合(hydration)这个概念和数据路由的"渲染前取数"有相似的时序问题——两者都存在"数据尚未就绪时该渲染什么"的思考,数据路由靠"loader 跑完才渲染"规避了这个问题,而 persist 的客户端水合天然有一个"先渲染默认值,再被存储值覆盖"的短暂窗口,这也是 SSR 场景下 hydration mismatch 的根源。
  • Zustand 创建 store(第 96 篇): persist 是一个中间件,包裹的正是第 96 篇里 create 接收的那个 (set) => ({...}) 函数——理解了 store 最基础的创建方式,才能看懂 persist(fn, options) 只是在原有创建逻辑外面包了一层"自动读写存储"的能力,并没有改变 store 本身的写法。
