环境变量 import.meta.env

接口地址在开发环境是 localhost:3000,生产环境是 api.example.com——这种"随环境变化的配置"不该写死在代码里,而应该放进环境变量。Vite 有一套专门的机制来管理它们,核心规则只有一条,但这条规则背后是一个关乎安全的重要原因。

.env 文件:按环境分层配置

Vite 约定了一套 .env 文件命名规则,会根据当前运行模式自动加载对应文件:

项目根目录下的 .env 文件族
.env                 // 所有环境都会加载(通用默认值)
.env.local           // 所有环境都加载,但不提交到 git(本地私密配置)
.env.development     // 仅 npm run dev(mode = development)时加载
.env.production      // 仅 npm run build(mode = production)时加载
.env.development 示例
VITE_API_BASE_URL=http://localhost:3000
VITE_ENABLE_MOCK=true
.env.production 示例
VITE_API_BASE_URL=https://api.example.com
VITE_ENABLE_MOCK=false

同名变量后加载的文件会覆盖先加载的(.env.local 优先级高于 .env.development),这样团队成员可以在自己本地的 .env.local(通常加进 .gitignore)里覆盖个别配置,而不影响别人。

为什么必须加 VITE_ 前缀:一条安全规则

如果你在 .env 里写一个不带前缀的变量,比如 DB_PASSWORD=xxx,在代码里通过 import.meta.env.DB_PASSWORD 访问,你会发现拿到的是 undefined。这不是 bug,是故意设计的安全限制。

为什么要这样限制? Vite 项目里的 .env 文件经常和后端共用同一个仓库,里面难免会混入数据库密码、第三方 API 密钥等只该服务端知道的敏感信息。如果 Vite 把 .env 里所有变量都无脑打进浏览器能拿到的 bundle,任何人打开浏览器控制台或者查看 JS 源码,都能直接看到这些密钥。只暴露带 VITE_ 前缀的变量是一道"白名单闸门":你必须显式地在变量名前加上前缀,相当于主动声明"这个值我确认可以给到浏览器",从而避免"不小心把服务端密钥打进前端"的事故。
代码里读取环境变量
const baseURL = import.meta.env.VITE_API_BASE_URL;

fetch(`${baseURL}/users`);
// 开发环境请求 http://localhost:3000/users
// 生产环境请求 https://api.example.com/users
// 代码完全不用写 if (开发环境) ... else ...

内置变量:MODE / DEV / PROD

除了自定义的 VITE_xxx,Vite 还内置了几个开箱即用的变量,不需要在 .env 里声明:

变量含义
import.meta.env.MODE当前模式字符串,如 "development"、"production"
import.meta.env.DEV布尔值,是否为开发模式(npm run dev)
import.meta.env.PROD布尔值,是否为生产模式(npm run build)
import.meta.env.BASE_URL应用部署的基础路径,对应 vite.config.ts 的 base 配置
用 DEV 做只在开发环境生效的逻辑
if (import.meta.env.DEV) {
  console.log('[debug] 当前请求参数:', params);
}
陷阱:DEV/PROD 判断会被 tree-shaking 优化掉。 Vite 生产构建时,import.meta.env.PROD 会被静态替换成字面量 true/false,像 if (import.meta.env.DEV) { ... } 这种分支在生产构建里会被打包工具直接删掉——这意味着调试代码不会被打进生产包,可以放心用它包裹 debug 日志,不用手动删。

对比 Node.js 的 process.env

写过 Node 后端或者用过 Webpack 的话,会更熟悉 process.env.NODE_ENV 这种写法。两者概念相似,但机制完全不同:

Node.js process.env
Node 运行时的全局对象
值来自操作系统进程环境
在 Webpack 项目里靠 DefinePlugin
在编译期做字符串替换才能用在浏览器代码里
本身是 Node 概念,浏览器原生不存在
Vite import.meta.env
ESM 标准里 import.meta 的扩展字段
Vite 在编译期读取 .env 文件后
静态注入,浏览器代码原生可用
自带 VITE_ 前缀白名单机制
天生只暴露显式允许的变量
为什么 Vite 不直接用 process.env? process 是 Node.js 的全局对象,浏览器环境里根本不存在。Vite 是面向浏览器原生 ESM 的工具,import.meta 才是 ES 模块规范里定义的、浏览器和 Node 都支持的标准语法,import.meta.env 是 Vite 在这个标准接口上做的扩展。用它而不是 process.env,也是 Vite"贴近浏览器原生能力"这一设计哲学的延续(参考上一篇讲的原生 ESM 开发模式)。

小结

  • Vite 按 .env / .env.local / .env.development / .env.production 分层加载环境变量,后加载的覆盖同名的先加载的。
  • 只有以 VITE_ 为前缀的变量才会暴露给客户端代码,这是一道安全白名单,防止服务端密钥被意外打进前端 bundle。
  • 用 import.meta.env.VITE_XXX 读取自定义变量,不需要在代码里写环境判断分支。
  • 内置变量 MODE/DEV/PROD 开箱即用;DEV/PROD 在生产构建时会被静态替换并触发死代码消除,调试代码不会混进生产包。
  • import.meta.env 是 ESM 标准 import.meta 的扩展,和 Node.js 的 process.env 是两个不同世界的东西,不要混用概念。
  • .env.local 一般要加入 .gitignore,用来放开发者本地私有的覆盖配置,不提交到仓库。
