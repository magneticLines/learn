与 TanStack Query 结合

上一篇封装好了 axios 实例,很多初学者会疑惑:"有了 axios,为什么还要 TanStack Query?" 答案是两者根本不是同一层的东西——axios 管"怎么把请求发出去",TanStack Query 管"请求回来的数据怎么被组件消费、缓存、失效"。这一篇把两者接起来。

分工:axios 和 TanStack Query 各管什么

把请求这件事拆成两半来看会清楚很多:

axios(封装的 request 实例)
负责"怎么发"这一层:
- 拼 baseURL、加请求头
- 序列化参数、超时控制
- 统一错误提示、401 跳转
它不知道"这个请求的结果
要不要缓存""多个组件用同
一个数据要不要发两次请求"
TanStack Query
负责"结果怎么用"这一层:
- 按 queryKey 缓存结果
- 自动去重、后台重新验证
- loading / error / success 状态
- 失效(invalidate)后自动重新拉取
它不关心请求具体怎么发,
只要给它一个返回 Promise 的
函数(queryFn)
类比: axios 像是"快递员"——负责把包裹(请求)送到、把回信(响应)带回来;TanStack Query 像是"收发室的登记系统"——记录哪些包裹已经收过(缓存)、什么时候该重新催一次(失效重取)、同一份包裹有没有被重复登记(去重)。快递员和登记系统不是互相替代的关系,而是配合工作。

queryFn 里调用封装好的 axios 实例

回顾第 104、105 篇讲过的 useQuery 基本用法——它需要一个 queryFn,返回一个 resolve 出数据的 Promise。把 144 篇封装的 request 实例传进去,几乎不需要额外代码:

api/user.js:把 axios 调用包成语义化的函数
import request from './request';

// 每个接口对应一个函数,命名即文档,方便复用和 mock 测试
export function getUserProfile(userId) {
  return request.get(`/users/${userId}`);  // 拦截器已经拆包,这里直接拿到 data
}
UserProfile.jsx:useQuery 里调用
function UserProfile({ userId }) {
  const { data, isLoading, isError } = useQuery({
    queryKey: ['user', userId],       // 依赖 userId,变了就重新请求
    queryFn: () => getUserProfile(userId),  // 调用封装好的 axios 函数
  });

  if (isLoading) return <Spinner />;
  if (isError) return <ErrorTip />;  // 拦截器弹了 toast,这里再决定页面级降级 UI
  return <div>{data.name}</div>;
}

注意这里没有手写 useState 存 loading/error、没有手写 useEffect 发请求、也没有手写"组件重新挂载会不会重复请求"的逻辑——这些正是 TanStack Query 在 axios 之上补的那一层。

结合 useMutation 做写操作

回顾第 106 篇讲过的 useMutation 和缓存失效策略,同样的分工模式适用于写操作:

更新用户资料后让缓存失效,触发重新拉取
function updateUserProfile(userId, payload) {
  return request.patch(`/users/${userId}`, payload);
}

const queryClient = useQueryClient();
const mutation = useMutation({
  mutationFn: (payload) => updateUserProfile(userId, payload),
  onSuccess: () => {
    // 写成功后让 ['user', userId] 这条缓存失效,useQuery 自动重新拉最新数据
    queryClient.invalidateQueries({ queryKey: ['user', userId] });
  },
});
常见误区:把 axios 拦截器的错误处理和 TanStack Query 的 onError 搞混。 拦截器负责通用的副作用(所有请求都要弹 toast、都要处理 401),而 useMutation 的 onError 负责这一次操作特有的反应(比如"删除失败,把列表项的删除按钮恢复可点击状态")。两者不是二选一,而是叠加:拦截器先跑,再轮到组件级的 onError。

为什么不能反过来,把缓存逻辑写进 axios 拦截器

有人会想:能不能在 axios 的 response 拦截器里自己维护一个 Map 做缓存,省得引入 TanStack Query?理论上可以,但会重新发明一大堆轮子:

能力手写缓存要解决TanStack Query 内置
多组件共享同一份数据不重复发请求要手动做请求去重按 queryKey 自动去重
数据过期后自动重新拉取要手写定时器/焦点监听staleTime/refetchOnWindowFocus 内置
写操作后让相关数据失效要手动清理缓存 MapinvalidateQueries 一行搞定
loading/error 状态管理要手写 state 机isLoading/isError 开箱即用
回到分层视角: axios 拦截器应该保持"薄"——只做认证和错误提示这种和具体接口无关的通用逻辑。缓存、失效、状态管理这些"数据生命周期管理"的活,交给专门为此设计的 TanStack Query,不要在两层之间职责混淆。

小结

  • axios(或封装后的 request 实例)负责"怎么发请求";TanStack Query 负责"请求结果怎么被缓存、去重、失效"——两者是分层协作关系,不是互相替代。
  • queryFn 里直接调用封装好的 axios 请求函数即可,TanStack Query 不关心底层用什么发请求。
  • 写操作用 useMutation 包裹 axios 调用,onSuccess 里用 invalidateQueries 让相关的 queryKey 失效,触发自动重新拉取——这是回顾第 106 篇缓存失效策略在真实请求场景下的落地。
  • axios 拦截器处理"通用"副作用(认证、401 跳转、错误提示),useMutation 的 onError 处理"具体这次操作"的反应,两者叠加而非二选一。
  • 不要在 axios 拦截器里手写缓存逻辑,那是在重新发明 TanStack Query 已经做好的轮子。
