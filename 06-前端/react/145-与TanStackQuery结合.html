与 TanStack Query 结合

上一篇封装好了 axios 实例,很多初学者会疑惑:"有了 axios,为什么还要 TanStack Query?" 答案是两者根本不是同一层的东西——axios 管"怎么把请求发出去",TanStack Query 管"请求回来的数据怎么被组件消费、缓存、失效"。这一篇把两者接起来。

分工:axios 和 TanStack Query 各管什么

把请求这件事拆成两半来看会清楚很多:

axios(封装的 request 实例)
负责"怎么发"这一层:
- 拼 baseURL、加请求头
- 序列化参数、超时控制
- 统一错误提示、401 跳转
它不知道"这个请求的结果
要不要缓存""多个组件用同
一个数据要不要发两次请求"
TanStack Query
负责"结果怎么用"这一层:
- 按 queryKey 缓存结果
- 自动去重、后台重新验证
- loading / error / success 状态
- 失效(invalidate)后自动重新拉取
它不关心请求具体怎么发,
只要给它一个返回 Promise 的
函数(queryFn)
类比: axios 像是"快递员"——负责把包裹(请求)送到、把回信(响应)带回来;TanStack Query 像是"收发室的登记系统"——记录哪些包裹已经收过(缓存)、什么时候该重新催一次(失效重取)、同一份包裹有没有被重复登记(去重)。快递员和登记系统不是互相替代的关系,而是配合工作。

queryFn 里调用封装好的 axios 实例

回顾第 104、105 篇讲过的 useQuery 基本用法——它需要一个 queryFn,返回一个 resolve 出数据的 Promise。把 144 篇封装的 request 实例传进去,几乎不需要额外代码:

api/user.js:把 axios 调用包成语义化的函数
import request from './request';

// 每个接口对应一个函数,命名即文档,方便复用和 mock 测试
export function getUserProfile(userId) {
  return request.get(`/users/${userId}`);  // 拦截器已经拆包,这里直接拿到 data
}
UserProfile.jsx:useQuery 里调用
function UserProfile({ userId }) {
  const { data, isLoading, isError } = useQuery({
    queryKey: ['user', userId],       // 依赖 userId,变了就重新请求
    queryFn: () => getUserProfile(userId),  // 调用封装好的 axios 函数
  });

  if (isLoading) return <Spinner />;
  if (isError) return <ErrorTip />;  // 拦截器弹了 toast,这里再决定页面级降级 UI
  return <div>{data.name}</div>;
}

注意这里没有手写 useState 存 loading/error、没有手写 useEffect 发请求、也没有手写"组件重新挂载会不会重复请求"的逻辑——这些正是 TanStack Query 在 axios 之上补的那一层。

结合 useMutation 做写操作

回顾第 106 篇讲过的 useMutation 和缓存失效策略,同样的分工模式适用于写操作:

更新用户资料后让缓存失效,触发重新拉取
function updateUserProfile(userId, payload) {
  return request.patch(`/users/${userId}`, payload);
}

const queryClient = useQueryClient();
const mutation = useMutation({
  mutationFn: (payload) => updateUserProfile(userId, payload),
  onSuccess: () => {
    // 写成功后让 ['user', userId] 这条缓存失效,useQuery 自动重新拉最新数据
    queryClient.invalidateQueries({ queryKey: ['user', userId] });
  },
});
常见误区:把 axios 拦截器的错误处理和 TanStack Query 的 onError 搞混。 拦截器负责通用的副作用(所有请求都要弹 toast、都要处理 401),而 useMutation 的 onError 负责这一次操作特有的反应(比如"删除失败,把列表项的删除按钮恢复可点击状态")。两者不是二选一,而是叠加:拦截器先跑,再轮到组件级的 onError。

为什么不能反过来,把缓存逻辑写进 axios 拦截器

有人会想:能不能在 axios 的 response 拦截器里自己维护一个 Map 做缓存,省得引入 TanStack Query?理论上可以,但会重新发明一大堆轮子:

能力手写缓存要解决TanStack Query 内置
多组件共享同一份数据不重复发请求要手动做请求去重按 queryKey 自动去重
数据过期后自动重新拉取要手写定时器/焦点监听staleTime/refetchOnWindowFocus 内置
写操作后让相关数据失效要手动清理缓存 MapinvalidateQueries 一行搞定
loading/error 状态管理要手写 state 机isLoading/isError 开箱即用
回到分层视角: axios 拦截器应该保持"薄"——只做认证和错误提示这种和具体接口无关的通用逻辑。缓存、失效、状态管理这些"数据生命周期管理"的活,交给专门为此设计的 TanStack Query,不要在两层之间职责混淆。

小结

  • axios(或封装后的 request 实例)负责"怎么发请求";TanStack Query 负责"请求结果怎么被缓存、去重、失效"——两者是分层协作关系,不是互相替代。
  • queryFn 里直接调用封装好的 axios 请求函数即可,TanStack Query 不关心底层用什么发请求。
  • 写操作用 useMutation 包裹 axios 调用,onSuccess 里用 invalidateQueries 让相关的 queryKey 失效,触发自动重新拉取——这是回顾第 106 篇缓存失效策略在真实请求场景下的落地。
  • axios 拦截器处理"通用"副作用(认证、401 跳转、错误提示),useMutation 的 onError 处理"具体这次操作"的反应,两者叠加而非二选一。
  • 不要在 axios 拦截器里手写缓存逻辑,那是在重新发明 TanStack Query 已经做好的轮子。

常见陷阱

  • queryKey 漏掉依赖参数: 上面例子里 queryKey: ['user', userId] 特意把 userId 放进数组,如果偷懒写成 ['user'],那么用户 A 的资料页切到用户 B 时,TanStack Query 会认为"这是同一条缓存",直接把 A 的数据渲染出来,直到重新请求才刷新——本质是把 queryKey 当成了"接口名字"而不是"缓存的身份标识",凡是 queryFn 里用到的参数,都要出现在 queryKey 里。
  • 在 queryFn 里把错误 catch 掉: 如果在 queryFn 内部写 try/catch 并且 catch 里返回了一个"看起来正常"的值(比如 null 或空数组),TanStack Query 会认为这次请求成功了,isError 永远是 false,页面上该展示的 ErrorTip 也就永远不会出现。queryFn 必须让错误继续往外抛(或者不做 catch),让 TanStack Query 自己捕获并进入 error 状态。
  • mutation 成功后忘记 invalidateQueries: 92 行的例子里 onSuccess 主动调了 invalidateQueries({ queryKey: ['user', userId] })。如果漏了这一步,useMutation 确实把数据改到了后端,但页面上 useQuery 缓存的还是旧数据,用户会看到"明明保存成功了,页面却没变"——这是把"写操作成功"和"读缓存刷新"当成了同一件事,实际上它们是两条独立的链路,需要手动用 invalidateQueries 连起来。
  • 把拦截器已经处理过的错误在 queryFn 里再处理一遍: 144 篇封装的拦截器已经统一弹了错误提示、处理了 401,如果又在 queryFn 里对同一个错误再 catch 一次弹一次 toast,用户会看到重复的错误提示。记住表格里的分层原则:拦截器管"通用副作用",queryFn/组件层只管"这次调用要不要额外反应",不要在两层做重复的事。
  • 混淆 staleTime 和"缓存被清空": 很多人以为数据"过期"(超过 staleTime)之后缓存就没了、下次请求要重新走一遍 loading。实际上过期只是标记为"可能不新鲜了",TanStack Query 依然会先把旧缓存显示出来,同时在后台悄悄重新请求(后台重新验证),等新数据回来再替换——如果不理解这一点,容易误以为 staleTime 设置无效或者去手写多余的强制刷新逻辑。

面试高频考点

  1. TanStack Query 解决了什么问题,为什么不能直接用 axios + useState/useEffect?—— axios 只负责发请求,useState/useEffect 手写方案要自己维护 loading/error 状态、自己判断要不要重新请求、自己做多组件共享同一份数据时的去重,还要自己处理"写操作后怎么让别处的数据也刷新"。TanStack Query 把这些"数据在组件生命周期里如何被消费"的通用逻辑抽成了一层,本质是把大家在 useEffect 里重复写的那套请求状态机标准化了。
  2. 为什么说 TanStack Query 管理的是"服务端状态",而不是普通的客户端状态?—— 普通状态(比如输入框的值、弹窗开关)只属于当前组件,由前端完全掌控,什么时候变、怎么变都是本地决定的。服务端状态的真相在后端数据库里,前端拿到的只是某个时间点的"快照",随时可能被别的用户、别的标签页改掉,还需要跨组件共享同一份快照。所以服务端状态天然需要"缓存 + 过期 + 重新验证 + 失效"这套机制,这正是 useState 天生不具备、而 TanStack Query 专门解决的。
  3. queryKey 和缓存之间是什么关系,为什么它的设计这么重要?—— TanStack Query 内部用 queryKey 序列化后的字符串作为缓存的唯一索引,可以理解成一个以 queryKey 为键的 Map:同样的 key 命中同一份缓存(自动去重、共享数据),key 里包含的参数变了就等于换了一个新的缓存条目(自动重新请求)。所以 queryKey 本质上是在告诉 TanStack Query"这份数据由哪些因素决定",漏写依赖参数就是让缓存的身份标识和数据实际内容不一致,这也是它被追问细节最多的地方。
  4. invalidateQueries 的失效机制是怎么工作的?—— 调用 invalidateQueries({ queryKey: [...] }) 并不会立刻重新发请求,它做的是把匹配到的缓存条目标记为"stale(过期)"。如果这个 queryKey 当前正被某个挂载的组件通过 useQuery 订阅着,TanStack Query 会自动触发一次重新请求;如果没有组件在订阅,它只是被标记,等下次有组件用到这个 key 时再重新拉取。这也是为什么"失效"和"重新请求"要分开理解——失效是缓存层面的状态变化,重新请求是订阅关系触发的结果。

与其他知识点的联系

  • 与上一篇 144(Axios 封装)的关系: queryFn 里调用的正是 144 篇封装好的 request 实例(或包装成语义化函数后的调用),两者是清晰的分层关系——axios 负责"怎么把请求发出去"(baseURL、认证头、错误提示),TanStack Query 负责"请求结果在组件生命周期里怎么被缓存和消费",谁都不替代谁,谁也不该越界去做对方的事。
  • 与手动用 useEffect 管理请求状态的对比: 没有 TanStack Query 时,每个需要发请求的组件都要自己手写 useState 存 data/loading/error,自己在 useEffect 里判断依赖数组、自己处理组件卸载后状态更新报错的问题。TanStack Query 把这套重复的手写状态机标准化成了 isLoading/isError/data 声明式返回值,本质是用"配置一个 queryKey + queryFn"替代了"手写一整套请求生命周期管理代码"。
  • 与下一篇 146(AbortController)的关系: TanStack Query 内部已经集成了自动取消过期请求的机制——比如 queryKey 变化导致旧请求还没返回就要作废,或者组件卸载时正在进行的请求应该被取消,它会自动用 AbortController 处理好这些细节。下一篇要手写的 AbortController 逻辑,本质上就是 TanStack Query 在背后已经帮你封装掉的那部分能力,先理解手写版再回头看会更清楚它到底省了什么。
  • 与组件卸载、竟态条件的关系: 144 篇提到裸用请求时,组件卸载后响应才回来可能引发"在已卸载组件上更新状态"的问题,也可能出现新旧请求响应顺序颠倒导致的竟态问题。TanStack Query 通过按 queryKey 管理请求生命周期,天然规避了这两类问题——不需要像手写方案那样自己在 useEffect 清理函数里判断"这次响应是不是已经过期了"。
  • 与表单提交、写操作的关系(143 篇): useMutation 包裹的正是提交表单后要调用的写请求,onSuccess 里的 invalidateQueries 让相关的读缓存失效并自动重新拉取——这把"表单校验通过后发起请求"(143 篇)和"请求结果怎么反映到界面上"这两个环节最终串成了一条完整链路:校验 → 发请求 → 缓存与状态自动更新。
