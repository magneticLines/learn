React · TanStack Query

useQuery：服务端状态的专用工具

TanStack Query（旧名 React Query）不是又一个像 Redux 那样的全局状态库，它只做一件事：管理"服务端状态"。useQuery 是它的核心 hook——声明一个查询，剩下的缓存、加载态、后台刷新全自动。

先想清楚：服务端状态为什么该用它，而非 Redux

很多人把接口数据塞进 Redux，然后手动管理 loading、缓存、失效——这其实是在用"客户端状态工具"处理"服务端状态"，费力不讨好。二者本质不同：

客户端状态服务端状态
拥有者前端自己（弹窗开关、表单）后端服务器（前端只是缓存副本）
是否会过期不会会——别人可能改了数据
需要什么读写、派生缓存、失效、重取、去重、后台刷新
合适工具useState / Zustand / ReduxTanStack Query / RTK Query
一句话分工：客户端状态用 Redux/Zustand，服务端状态用 TanStack Query。两者可以并存——一个应用里，弹窗开关放 Zustand，用户列表放 useQuery，各司其职。

基础用法：queryKey + queryFn

先在应用根部用 QueryClientProvider 提供一个 client（类似 Redux 的 Provider），组件里就能用 useQuery：

main.jsx —— 提供 QueryClient
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient();  // 缓存都存在这个 client 里

root.render(
  <QueryClientProvider client={queryClient}>
    <App />
  </QueryClientProvider>
);
Todos.jsx —— useQuery 三要素
import { useQuery } from '@tanstack/react-query';

function Todos() {
  const { data, isLoading, error, isFetching } = useQuery({
    // queryKey：这份数据的唯一标识，也是缓存的 key
    queryKey: ['todos'],
    // queryFn：返回 Promise 的取数函数，必须 throw 或 reject 才算失败
    queryFn: async () => {
      const res = await fetch('/api/todos');
      if (!res.ok) throw new Error('请求失败'); // fetch 不会因 4xx 自动 reject，要手动抛
      return res.json();
    },
  });

  if (isLoading) return <p>首次加载中…</p>;
  if (error) return <p>{error.message}</p>;
  return (
    <ul>
      {/* isFetching：后台正在刷新（区别于首次 isLoading） */}
      {isFetching && <span>刷新中…</span>}
      {data.map(t => <li key={t.id}>{t.title}</li>)}
    </ul>
  );
}

isLoading vs isFetching 别搞混

isLoading
缓存里「还没有数据」
且正在首次请求
→ 显示骨架屏/加载中
isFetching
任何时候在发请求
（含后台刷新已有数据）
→ 显示小的「刷新中」提示
为什么 useQuery 体验这么好？——"陈旧但可用"（stale-while-revalidate）。当你带着已缓存的数据重新进入页面，useQuery 会立即返回旧数据让你先看到内容（不闪 loading），同时在后台悄悄发请求拿新数据，回来后无缝替换。这就是 isLoading（无数据）和 isFetching（有数据也在刷）要分开的原因。

动态 queryKey：参数变化自动重取

queryKey 是数组，把依赖的参数放进去。参数一变，TanStack Query 就把它当成"另一份数据"，自动重新请求并单独缓存：

按 id 查询详情
function TodoDetail({ id }) {
  const { data } = useQuery({
    queryKey: ['todo', id],   // id 变 → key 变 → 自动重取，各 id 独立缓存
    queryFn: () => fetchTodo(id),
    enabled: id != null,    // enabled=false 时暂不请求（等条件满足）
  });
  // ...
}
面试点：①"为什么服务端状态不用 Redux？"——Redux 不解决缓存/失效/去重/后台刷新，用它得全部手写；TanStack Query 内建这些。②"queryKey 的作用？"——既是缓存键，也是重取触发器，参数放进去实现"依赖变则重取"。③"isLoading 和 isFetching 区别？"——前者是"无数据的首次加载"，后者是"任何进行中的请求（含后台刷新）"。
