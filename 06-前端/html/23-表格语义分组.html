表格语义分组：thead / tbody / tfoot

上一节我们用 table / tr / th / td 拼出了表格骨架。但只有骨架不够——浏览器、打印机、屏幕阅读器都想知道：哪一行是表头？哪些是数据？哪一行是合计？这就是语义分组要解决的问题。

一、三个分组：表头、主体、表尾

一张完整的表格在语义上分成三块，对应三个标签。它们都是 <table> 的直接子元素， 内部再放 <tr>：

  • <thead>——表头组，放列标题那一行（通常是一排 <th>）
  • <tbody>——主体组，放真正的数据行，可以有多个
  • <tfoot>——表尾组，放汇总/合计行（如"总计""平均值"）
一张语义完整的表格
<table>
  <thead>
    <tr><th>月份</th><th>收入</th></tr>
  </thead>
  <tbody>
    <tr><td>一月</td><td>1200</td></tr>
    <tr><td>二月</td><td>1500</td></tr>
  </tbody>
  <tfoot>
    <tr><td>合计</td><td>2700</td></tr>
  </tfoot>
</table>
注意：分组只改变语义，不改变默认外观。加不加 thead/tbody/tfoot， 表格肉眼看上去一样。它们的价值藏在打印、滚动、无障碍、CSS 选择器这些"看不见"的地方——下面逐个讲。

二、为什么要分？四个实用价值

1. 打印长表格时 thead / tfoot 每页重复

这是最直观的好处。一张几百行的财报表打印成多页时，浏览器会在每一页顶部自动重复 thead，底部重复 tfoot。 翻到第 5 页你照样知道每列是什么意思，不用回第 1 页对照。如果你不分组，全堆在一坨 <tr> 里， 第 2 页开始就是一堆没有表头的裸数据。

2. 可以对 tbody 单独滚动

常见需求："表头固定，数据区滚动"。有了 tbody 这个独立容器，配合 CSS （给 tbody 设 display:block; overflow-y:auto; max-height，或用 position:sticky 固定 thead）， 实现起来才有抓手。没有分组就没有这个可定位的"主体区"。

3. 屏幕阅读器理解结构

读屏软件依赖语义判断"这是表头还是数据"。thead 里的 <th> 会被当成列标题， 阅读数据单元格时自动播报"列名：值"（如"收入：1200"），盲人用户才能听懂表格。无语义分组时， 它只能把一堆单元格平铺念出来，体验极差。这也是无障碍（a11y）合规的硬要求。

4. CSS / JS 有了精确的定位钩子

想"只给数据行加斑马纹，不影响表头"，写 tbody tr:nth-child(even) 就行； 想用 JS 只遍历数据行，table.tBodies[0].rows 直接拿到主体行。分组让选择器表达力大大增强。

三、高频大坑：tbody 是隐式存在的

这是表格里最容易翻车、面试也爱考的点：就算你不写 <tbody>，浏览器解析时也会自动给你补一个。 HTML 解析器看到 <table> 下直接是 <tr>，会悄悄把这些 <tr> 包进一个隐式的 <tbody> 里。

你写的 HTML
<table>
  <tr><td>A</td></tr>
</table>
浏览器实际构建出的 DOM（自动补了 tbody）
<table>
  <tbody>          <!-- 👻 你没写，但它在 -->
    <tr><td>A</td></tr>
  </tbody>
</table>

它会怎么坑你

JS 选择器踩坑
// ❌ 选不到！因为 tr 的父级是 tbody，不是 table
const rows = document.querySelectorAll('table > tr')
console.log(rows.length)  // 0 —— 一行都没选到

// ✅ 正确：考虑到隐式 tbody
document.querySelectorAll('table tr')           // 后代选择器，OK
document.querySelectorAll('table > tbody > tr')  // 显式写出 tbody，OK
同理也坑 CSS：你写 table > tr { ... }（子选择器）会完全不生效， 因为 DOM 里 tr 的直接父级是 tbody 而不是 table。换成后代选择器 table tr 或 table > tbody > tr 才对。
记住一句话：在 DOM 层面，table 和 tr 之间永远隔着一个 tbody（除非那行属于 thead/tfoot）。
反向利用：正因为隐式 tbody 一定存在，table.querySelector('tbody') 永远能拿到东西， table.tBodies 也总是非空。写斑马纹直接 tbody tr:nth-child(even)， 天然就只命中数据行、漏掉表头——这恰恰是隐式 tbody 帮了忙。

四、tfoot 的位置：写哪都渲染在底部

早期 HTML4 强制要求 tfoot 必须写在 tbody 之前。 原因很现实：浏览器流式解析，想在收到全部数据行之前就知道表尾长什么样（方便分页时预留底部空间）。 所以源码顺序是 thead → tfoot → tbody，但渲染时 tfoot 依然画在最底下。

HTML5 放宽了：两种写法都合法，渲染结果相同
<!-- 写法 A（HTML5 推荐，符合直觉）：tfoot 放最后 -->
<table>
  <thead>...</thead>
  <tbody>...</tbody>
  <tfoot>...</tfoot>   <!-- 渲染在底部 -->
</table>

<!-- 写法 B（HTML4 老式）：tfoot 放 tbody 前面 -->
<table>
  <thead>...</thead>
  <tfoot>...</tfoot>   <!-- 写这，但还是渲染在底部 -->
  <tbody>...</tbody>
</table>
常见误区：很多人以为"tfoot 必须放在最后，否则合计行会跑到中间去"。 错——位置由标签语义决定，不由源码顺序决定。无论 tfoot 写在哪， 渲染永远在表格底部。HTML5 时代直接用直觉的写法 A 即可，不必再像 HTML4 那样别扭地提前写。

五、colgroup 与 col：对整列做配置

前面的分组（thead/tbody/tfoot）是按行切的。但有时你想按列统一设置—— 比如"第一列宽 120px、背景灰一点"。这就是 <colgroup> 和 <col> 的用武之地。 它们放在 <table> 开头、thead 之前，每个 <col> 对应一列。

用 col 定义列宽和列样式
<table>
  <colgroup>
    <col style="width:120px; background:#13151f">  <!-- 第1列 -->
    <col span="2" style="width:80px">          <!-- 第2、3列 -->
  </colgroup>
  <thead>...</thead>
  <tbody>...</tbody>
</table>

<col> 是空元素（无内容、无闭合标签），span 属性表示"这一个 col 横跨几列"。 它免去了"给每行的第 N 个单元格逐一设宽"的重复劳动——定义一次，整列生效。

col 上的 CSS 有限制：出于浏览器渲染机制，<col> 上只有少数属性会生效， 主要是 width、background、border、visibility。 像 color、font-size、padding 这类作用在文字/盒子内部的属性对 col 无效—— 因为单元格不是 col 的子元素，它们只是"逻辑上同列"。要改这些，还得用 td:first-child 之类的选择器。

六、综合：搭配 col + thead/tbody/tfoot 的完整结构

一张"列配置 + 行分组"齐全的表格
<table>
  <!-- 列级配置：首列加宽并高亮 -->
  <colgroup>
    <col style="width:120px; background:#13151f">
    <col>
  </colgroup>

  <!-- 表头组 -->
  <thead>
    <tr><th>季度</th><th>销售额</th></tr>
  </thead>

  <!-- 主体组：真实数据 -->
  <tbody>
    <tr><td>Q1</td><td>3000</td></tr>
    <tr><td>Q2</td><td>4200</td></tr>
  </tbody>

  <!-- 表尾组：汇总行 -->
  <tfoot>
    <tr><td>合计</td><td>7200</td></tr>
  </tfoot>
</table>

这就是一张"工业级"表格该有的样子：列宽用 colgroup 统一控制，行按语义分成头/体/尾， 打印、滚动、读屏全都照顾到了。日常写表格养成这个习惯，比裸堆 <tr> 强得多。

七、陷阱速查

误区真相 / 后果正确做法
不写 tbody 就以为 DOM 里没有它 浏览器会自动补隐式 tbody，它真实存在于 DOM 选择器要考虑这一层，别用 table > tr
table > tr 选不到行，以为标签写错了 tr 的父级是 tbody，子选择器断在这 用 table tr 或 table > tbody > tr
以为 tfoot 必须写在最后才会在底部 位置由语义决定，写哪都渲染在底部 HTML5 直接放最后，符合直觉即可
不用 thead，全堆 <tr> 打印多页时后续页没有表头，读屏也念不出列名 列标题行一律包进 thead
在 <col> 上写 color/padding 期望生效 col 只认 width/background/border 等少数属性 文字类样式用 td:nth-child(n) 选择器

八、与 Java 的类比

表格概念Java / 后端类比
thead / tbody / tfoot 三段分组 一份报表的表头区 / 明细区 / 汇总区——比如 JasperReports 的 columnHeader / detail / summary 三个 band，结构与职责一一对应
隐式 tbody（不写也存在） 框架自动补全的默认值——像 Spring Boot 没配 DataSource 也会有自动配置的默认实例，你"没写"不等于"没有"，绕不过它
colgroup / col 列级配置 对一整列做统一配置——类似给某字段加 @Column(length=120)，一处声明、整列生效，而不是每行单独设
tfoot 源码顺序≠渲染顺序 声明顺序与执行/呈现顺序解耦——如 Bean 的定义顺序不决定初始化顺序（由依赖关系决定），别用书写位置去推断最终结果
语义分组带来无障碍/打印能力 给数据打上语义标签换取通用工具支持——如实体加 JPA 注解后，ORM、校验、序列化框架都能"看懂"它并自动处理
本节的核心就两件事：用 thead/tbody/tfoot 表达表格的语义结构， 以及牢记隐式 tbody 这个看不见但真实存在的层。前者让浏览器、打印机、读屏软件都能正确理解你的表格； 后者是 CSS 子选择器和 JS 遍历最常见的翻车点。下一节我们看怎么让单元格跨行跨列——合并单元格。
