HTML · 表单

input 类型大全：type 不只是样式

同一个 <input> 标签，靠 type 属性变身二十多种控件。type 不是装饰——它决定了移动端唤起哪种键盘、浏览器做什么内置校验、值如何被序列化。全用 type=text 是新手最常见的"能用就行"陷阱。

一、核心心智模型：type 是数据类型声明

后端工程师可以这样理解：HTML 的 <input> 就像一个空的输入槽， 而 type 相当于给这个槽声明数据类型。 你在 Java 里写 String email 还是 int age，编译器会据此做不同的处理； 在 HTML 里写 type="email" 还是 type="number"，浏览器也会据此做三件事：

  • 渲染对应控件：日期类弹出日历，颜色类弹出取色盘，范围类渲染滑块。
  • 触发内置校验：email 检查是否含 @，number 拒绝非数字。
  • 在移动端唤起匹配的虚拟键盘（下面单独讲，这是最被低估的价值）。
一个标签，几十种形态：除了 <textarea>、<select> 这些独立标签， 绝大多数表单控件都是 <input> + 不同 type 的组合。 记住 type 列表，你就掌握了表单控件的大半。

二、文本类：text / password / email / tel / url / search / number

这一类都是单行文本输入框，外观相近，但语义和行为差别很大。

文本类 input
<!-- 普通文本，无任何校验 -->
<input type="text" name="nickname">

<!-- 密码，输入内容显示为圆点 -->
<input type="password" name="pwd">

<!-- 邮箱，校验格式 + 移动端 @ 键盘 -->
<input type="email" name="email">

<!-- 电话，移动端唤起拨号键盘（不校验格式，号码格式各国不同）-->
<input type="tel" name="phone">

<!-- 网址，校验 URL 格式 -->
<input type="url" name="website">

<!-- 搜索框，部分浏览器带清除小叉 -->
<input type="search" name="q">

<!-- 数字，只接受数字，可配 min/max/step -->
<input type="number" name="age" min="0" max="120" step="1">
陷阱：number 不适合电话/银行卡号/验证码。 这些"看起来是数字"的东西其实是数字串——首位 0 会被吞掉、超长会触发数字格式化、上下箭头毫无意义。 电话用 type="tel"，验证码用 type="text" + inputmode="numeric" 更合适。 number 只留给真正参与计算的数量、年龄、价格。

三、为什么不能全用 text：移动端键盘

这是 type 语义价值里最容易被 PC 端开发者忽略的一点。 桌面键盘只有一种，所以全用 text 在电脑上"看不出区别"； 但手机的虚拟键盘是根据 type 动态切换的：

type移动端唤起的键盘用户体验差异
text标准全键盘输数字要切到符号页，麻烦
email带 @ 和 . 的键盘输邮箱不用切页找 @
tel九宫格拨号键盘大数字按键，输号码极快
number纯数字键盘只有 0-9，按错率低
url带 / 和 .com 的键盘输网址更顺手
search回车键变成"搜索"语义化的提交按钮
一句话经验：哪怕你不在乎 PC 端，光是"手机用户输入更顺手"这一条， 就足以让你为每个字段挑对 type。这是几乎零成本就能拿到的体验提升。 如果某 type 不合适但又想控制键盘，可以单独用 inputmode 属性（如 inputmode="decimal"）。

四、选择类：checkbox（多选）/ radio（单选）

这两个最特殊：它们靠 name 属性分组，靠 checked 表示选中状态。 理解"name 分组机制"是用对它们的关键。

radio：同 name 互斥单选
<!-- 三个 radio 共用 name="gender"，浏览器视为一组，只能选一个 -->
<input type="radio" name="gender" value="male" checked> 男
<input type="radio" name="gender" value="female"> 女
<input type="radio" name="gender" value="other"> 其他
<!-- 提交时只发送一个键值对：gender=male -->
checkbox：同 name 可多选
<!-- 共用 name="hobby[]"，可同时勾选多个 -->
<input type="checkbox" name="hobby" value="read" checked> 阅读
<input type="checkbox" name="hobby" value="code" checked> 写代码
<input type="checkbox" name="hobby" value="game"> 游戏
<!-- 提交时发送多个同名键值对：hobby=read&hobby=code -->

<!-- 单个 checkbox 也常用作"同意条款"开关 -->
<input type="checkbox" name="agree" required> 我已阅读并同意
陷阱：radio 必须共享同一个 name 才能互斥。 如果三个 radio 各写了不同的 name，它们就成了三个独立的"开关"，可以同时选中—— 互斥效果完全失效。这是初学者排查"radio 怎么能多选"问题的头号原因。
关于提交的值：checkbox / radio 提交的是 value 属性的值，不是显示文字。 没写 value 时浏览器默认发 on，几乎没用。所以务必给每个选项写明确的 value。 未勾选的 checkbox 根本不出现在提交数据里（不是发 false，而是这个键直接不存在）。

五、日期时间类：date / time / datetime-local / month / week

这些 type 会渲染原生日历/时间选择器，免去手写 datepicker 组件的成本。

日期时间类 input
<input type="date">            <!-- 选日期：2026-06-23 -->
<input type="time">            <!-- 选时间：14:30 -->
<input type="datetime-local">  <!-- 日期+时间（本地时区，无时区信息）-->
<input type="month">           <!-- 选年月：2026-06 -->
<input type="week">            <!-- 选第几周：2026-W25 -->
注意时区坑：datetime-local 的值不带时区（名字里的 local 就是提示）。 它提交的是用户本地墙上时间字符串，后端必须自己约定时区或额外传时区信息， 否则跨时区时会算错。日期类的提交格式都是 ISO 8601 字符串，和 Java 的 LocalDate / LocalDateTime 解析格式天然契合。

六、特殊类：file / color / range / hidden / 按钮 / image

特殊用途 input
<!-- 文件上传：accept 限制类型，multiple 允许多选 -->
<input type="file" accept="image/*" multiple>

<!-- 颜色选择器，弹出取色盘，值为 #rrggbb -->
<input type="color" value="#fb923c">

<!-- 滑块，靠 min/max/step 定义范围（不显示数值，需自己显示）-->
<input type="range" min="0" max="100" step="5">

<!-- 隐藏字段：不显示，但会随表单提交（常存 id/token）-->
<input type="hidden" name="userId" value="1001">

<!-- 三种按钮 -->
<input type="submit" value="提交">  <!-- 提交表单 -->
<input type="reset" value="重置">   <!-- 清空表单 -->
<input type="button" value="自定义"> <!-- 无默认行为，配 JS 用 -->

<!-- 图片提交按钮：点击会提交 + 上传点击坐标 x,y -->
<input type="image" src="submit.png" alt="提交">
陷阱：file 多选忘了 multiple。 只写 type="file" 用户每次只能选一个文件。要支持多选必须显式加 multiple， 并且后端接收时也要按数组/多文件处理。 accept 只是"过滤建议"（让选择框默认显示符合类型的文件），不是安全校验—— 用户仍能绕过选别的，服务端必须二次校验文件类型。
按钮的现代写法：<input type="submit"> 仍然有效， 但实际项目更常用独立的 <button> 标签，因为它能放图标、富文本，灵活得多。 注意 <button> 在表单内默认 type 就是 submit， 不想提交时要显式写 type="button"，这也是常见的"点一下按钮表单就刷新了"的坑。

七、通用属性：所有 input 都能用

属性作用
value初始值 / 提交的值（文本类是输入内容，选择类是选项值）
placeholder占位提示文字，输入后消失，不是默认值
name提交时的键名，没有 name 的字段不会被提交
required必填，空值时阻止提交并提示
readonly只读，不能改但会随表单提交
disabled禁用，变灰、不能改、且值不会提交
autocomplete控制浏览器自动填充（如 autocomplete="email" / "off"）
checked仅 checkbox/radio：初始是否选中
陷阱：disabled 的值不提交，readonly 会提交。 这是两者最容易混淆又最影响结果的区别。 如果你想让字段"用户改不了但仍要把值传给后端"，必须用 readonly； 用 disabled 的话后端收不到这个字段，常导致"明明界面有值，后端却拿到 null"。
placeholder ≠ 默认值：placeholder 只是灰色提示，用户不输入时提交的是空字符串。 要设默认值用 value。把 placeholder 当默认值用是常见误解。

十、与 Java 的类比

input 概念Java 类比
type 选择 字段的数据类型声明——就像 String / int / LocalDate，决定了如何解析、校验、序列化
内置校验（email / url / number / required） Bean Validation 注解——@Email / @URL / @NotNull / @Min，声明式地约束输入
radio 同 name 互斥单选 枚举单选——一个字段只能取 enum 中的一个值（Gender gender）
checkbox 同 name 多选 Set<Hobby> 多选集合——可以同时持有多个值
hidden 隐藏字段 DTO 里随请求传递但 UI 不展示的字段（如 id、version、CSRF token）
前端校验 + 后端校验 前端校验是"友好提示"，后端校验才是安全底线——正如永远不能信任客户端传来的数据，service 层必须再校一遍
核心收获：把 type 当成"给输入框声明数据类型"， 你就会自然地为每个字段挑对类型——拿到的是免费的移动端键盘优化、免费的内置校验、清晰的语义。 全用 text 等于把所有字段都声明成 Object：能塞东西进去，但丢掉了类型系统给你的一切好处。

与其他知识点的联系

  • 关联知识点：表单校验属性（29 号文件）：本文讲的是每种 type 自带的"隐式校验"（email 查 @、number 拒非数字）， 而 required / pattern / minlength / maxlength 是叠加在 type 之上的"显式校验"。 两者不是替代关系而是配合关系：type="email" 只保证格式像邮箱，真要求必填还得再加 required； type="text" 想限制成手机号格式，就要靠 pattern 补上 type 本身给不了的校验能力。
  • 关联知识点：React 受控组件的 value / onChange：不同 type 在 React 里"受控"的写法并不统一—— text / email / number 用 value + onChange(e.target.value)； checkbox 必须用 checked（不是 value）+ onChange(e.target.checked)， 因为它的语义是"是否选中"而不是"文本值"；radio 同一组共享一个 state 变量，靠比较 value 是否等于当前 state 来决定哪个被选中； type="file" 则完全不能被 React 受控（浏览器出于安全禁止脚本设置 value），只能用 ref 读取 files。 不了解这些差异，照抄 value+onChange 的模板去写 checkbox，会立刻报 React 的受控组件警告。
  • 关联知识点：可访问性（label 关联与屏幕阅读器播报）：type 不仅影响视觉控件，也决定屏幕阅读器怎么"读"这个字段—— type="email" 会被读作"电子邮件，可编辑文本"，type="radio" 会额外播报"单选按钮，第几项共几项"， type="checkbox" 播报"复选框，已选中/未选中"。但这一切播报的前提是 <label> 通过 for/id 或包裹方式正确关联到该 input——如果只写了 placeholder 没写 label，屏幕阅读器读到的只是"可编辑文本"，用户根本不知道这个框是干什么的， 这是比"陷阱速查表"里那些陷阱更容易被忽视、但对真实用户影响更大的问题。
  • 关联知识点：JS 数据类型体系（01 号文件 变量与数据类型）：不管 type 声明成什么， input.value 在 JS 里读出来永远是字符串——这是 DOM API 的固定行为，和 HTML 里 type="number" 带来的"看起来是数字"的直觉正好相反。<input type="number" value="18">，用 JS 读 input.value 拿到的是字符串 "18" 而不是数字 18，直接拿去做加法会得到字符串拼接的结果（比如 "18" + 1 是 "181"）； 必须像 01 号文件里讲的那样显式用 Number(input.value) 或 parseInt 做类型转换后才能参与数值计算。 唯一的例外是 input.valueAsNumber（type 为 number/range/date 等时可用），它才会直接返回数字/Date 类型。
  • 关联知识点：表单提交与序列化（25 号文件 form 基础与提交）：本文提到的"checkbox 未勾选时键不出现在提交数据里"、 "disabled 的值不提交"，都是 25 号文件讲的 FormData 序列化规则在不同 type 上的具体体现—— 理解 type 决定值的存在与否（不是存在但为假），才能解释为什么后端偶尔会收到"缺字段"而不是"字段值为 false"的请求体。
