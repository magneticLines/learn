HTML · 表单

form 基础与提交方式

表单是前端把数据交给后端的最原始入口。作为后端工程师，你早已熟悉 @RequestParam 和 Content-Type，这一节反过来看：浏览器是怎么把用户输入打包成一个 HTTP 请求发出去的。

一、form 的两个核心属性：action 与 method

一个 <form> 标签本质上是一个"HTTP 请求的声明"。两个属性决定了这个请求往哪发、用什么方式发：

最小可用表单
<form action="/api/login" method="post">
  <input type="text" name="username">
  <input type="password" name="password">
  <button type="submit">登录</button>
</form>
  • action：请求的目标 URL。等价于后端那条路由的路径（/api/login）。不写 action 时默认提交到当前页面 URL。
  • method：HTTP 方法，只有两个合法值 get / post（大小写不敏感）。注意：原生 form 不支持 PUT / DELETE，这是 HTML 规范的历史限制，想用只能靠 JS。
对照后端：action + method 合起来就锁定了后端的一个 handler。 action="/api/login" method="post" 命中的就是 Spring 里那个 @PostMapping("/api/login") 的方法。前端这边只是声明"我要打哪个门"。

二、GET vs POST：数据放哪、能干什么

这是后端最熟悉的话题，但前端侧有个直观的区别：同一个表单，method 改一个词，请求长得完全不一样。

method="get"：参数拼进 URL 查询串

GET 提交后浏览器地址栏变成
// 表单 action="/search"，填了 keyword=java, page=2
GET /search?keyword=java&page=2 HTTP/1.1
Host: example.com
// 没有请求体，参数全在 URL 的 ? 后面

method="post"：参数放进请求体

POST 提交的请求长这样
POST /search HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

keyword=java&page=2
// 参数在请求体里，URL 干净
维度GETPOST
参数位置URL 查询串 ?k=v请求体 (body)
长度限制有（受浏览器/服务器 URL 长度限制，约 2KB～8KB）无实际限制
幂等性幂等（多次请求结果一致）不幂等（每次可能产生新数据）
可缓存 / 可收藏可以（URL 含全部参数）不可以
浏览器后退/刷新直接重发，无提示弹"确认重新提交表单"
典型用途查询、筛选、分页（读操作）登录、注册、下单（写操作）
陷阱：用 GET 提交敏感信息。GET 把参数全暴露在 URL 上——会被浏览器历史记录、 服务器 access log、代理日志、Referer 头统统记下来。密码、token 这类数据 绝不能用 GET 提交。这和后端"敏感参数不要拼在 URL 里打日志"是同一条铁律。
幂等不是"安全"。GET 幂等指"重复请求不改变服务器状态"，所以浏览器敢随便重发、缓存、预取。 一旦你写了个 GET /deleteUser?id=5 这种"用 GET 干写操作"的接口， 搜索引擎爬虫一爬，用户全删没了——这正是 GET 必须保持只读的根本原因。

三、name 属性：决定提交的字段名

表单提交的本质是把控件打包成一串 key=value。这个 key 就是控件的 name 属性， 而不是 id，也不是变量名。

name 决定提交的 key
<input type="text" name="userName" value="shea">
// 提交时产生：userName=shea

<!-- ❌ 没有 name：这个控件根本不会被提交！-->
<input type="text" id="email" value="a@b.com">
// id 只用于 JS/CSS 定位，跟提交无关
高频坑：没有 name 的控件不会被提交。后端拿不到值、调了半天 form， 结果是某个 <input> 只写了 id 忘了 name。 记住：id 是给前端（JS/CSS）用的，name 才是给 HTTP 提交用的。 被 disabled 的控件同样不会提交（但 readonly 会提交）。
对照后端：name 就是 @RequestParam("userName") 里那个名字。 前端 name="userName" 和后端 @RequestParam("userName") 必须对上，否则后端绑定不到—— 跟 JSON 字段名要和 DTO 属性名对应是一个道理。

四、enctype：请求体怎么编码

当 method 是 POST 时，enctype 决定浏览器用什么格式打包请求体—— 它最终体现为 HTTP 的 Content-Type 请求头。三个取值：

enctype含义用途
application/x-www-form-urlencoded 默认值。字段拼成 k1=v1&k2=v2，特殊字符 URL 编码 普通文本表单
multipart/form-data 把每个字段切成独立的 part，用分隔符隔开，可携带二进制 文件上传必须
text/plain 纯文本，几乎不做编码 调试用，生产基本不用

为什么文件上传必须 multipart/form-data

默认的 urlencoded 把所有字段拼成一行字符串，无法表达二进制文件内容 （文件里全是任意字节，没法塞进 k=v 这种纯文本格式）。 multipart/form-data 把每个字段切成带边界（boundary）的"段"， 每段有自己的头和原始字节，文件才能完整传过去。

文件上传表单 + 对应的请求体
<form action="/upload" method="post"
      enctype="multipart/form-data">
  <input type="text" name="title">
  <input type="file" name="avatar">
  <button>上传</button>
</form>

// 请求体（boundary 是浏览器自动生成的分隔符）
Content-Type: multipart/form-data; boundary=----abc123

------abc123
Content-Disposition: form-data; name="title"

我的头像
------abc123
Content-Disposition: form-data; name="avatar"; filename="a.png"
Content-Type: image/png

‹原始二进制字节›
------abc123--
最常见的上传失败：放了 <input type="file"> 却忘了在 form 上写 enctype="multipart/form-data"。结果浏览器只把文件名字符串发过去， 文件内容丢失，后端 MultipartFile 拿到的是空的。 一上传就报错或文件为空，先检查这个 enctype。
对照后端：enctype 就是 HTTP 的 Content-Type。Spring 里 multipart/form-data 对应 MultipartFile 参数， urlencoded 对应普通 @RequestParam。后端的 MultipartResolver 正是按 Content-Type 决定要不要走文件解析逻辑。

五、怎么触发提交

  • submit 按钮：<button>（默认 type 就是 submit）或 <input type="submit">。
  • 回车键：当焦点在 form 内的单行文本框里时，按回车会自动触发提交——这是浏览器内置行为，很多人没意识到。
  • JS 调用：form.submit() 程序化提交（注意它不会触发表单校验和 submit 事件）。
陷阱：button 默认 type 是 submit。在 form 里写 <button>点我</button>，点击会触发表单提交导致页面刷新。 如果这个按钮只是想绑个 JS 事件、不想提交，必须显式写 type="button"。 这是新手"点个按钮页面就莫名其妙刷新了"的头号原因。

六、原生提交会刷新页面，现代做无刷新提交

原生表单提交是一次整页跳转：浏览器发请求、丢弃当前页面、加载响应回来的新页面。 这是 form 诞生时（多页应用时代）的默认行为。现代 SPA 不希望页面刷新， 做法是 拦截默认提交 + 用 fetch 自己发请求。

无刷新提交：preventDefault + fetch
const form = document.querySelector('#loginForm')

form.addEventListener('submit', async (e) => {
  e.preventDefault()   // 关键：阻止浏览器默认的整页跳转

  // FormData 自动收集 form 内所有带 name 的控件
  const data = new FormData(form)

  const res = await fetch('/api/login', {
    method: 'POST',
    body: data   // 传 FormData，浏览器自动设好 multipart 的 Content-Type
  })
  const json = await res.json()
  console.log(json)
})
为什么是 e.preventDefault()：submit 事件的默认动作就是"发请求 + 整页跳转"。 调 preventDefault() 取消这个默认动作，页面就不刷新了，剩下的请求由 fetch 全权接管。 这和你给 <a> 标签 preventDefault 阻止跳转是同一个机制。

七、FormData API：收集表单数据

FormData 是浏览器提供的、专门用来打包表单数据的对象。 new FormData(formElement) 会自动遍历这个 form 内所有带 name 的控件， 把 name=value 全收集进来——包括文件。

FormData 的常用操作
const fd = new FormData(form)

fd.get('username')        // 取单个字段值
fd.getAll('hobby')        // 同名多值（如多选框）取数组
fd.append('extra', '1')   // 手动追加字段
fd.has('avatar')         // 判断是否存在

// 遍历查看收集到了什么
for (const [k, v] of fd.entries()) {
  console.log(k, v)
}

// 转成普通对象（注意：文件字段会变成 File 对象）
const obj = Object.fromEntries(fd)
对照后端：FormData 携带文件时，对应后端的 MultipartFile； 携带普通字段时对应 @RequestParam。前端 new FormData(form) 这一步， 干的就是浏览器原生提交时"自动收集控件"那件事——只不过现在交给你的 JS 手动掌控。
用 fetch + FormData 时不要手动设 Content-Type。FormData 作为 body 时， 浏览器会自动生成带正确 boundary 的 multipart/form-data 头。 你一旦手动 headers: {'Content-Type': 'multipart/form-data'}， 反而漏掉了 boundary，后端解析直接失败。这是 fetch 上传的经典坑。

八、陷阱速查

陷阱后果正确做法
控件没写 name 该字段不会被提交，后端拿不到 所有要提交的控件都加 name（别和 id 搞混）
文件上传忘了 multipart/form-data 只传文件名，内容丢失 有 type="file" 必加 enctype="multipart/form-data"
用 GET 提交密码/token 敏感信息暴露在 URL、日志、历史记录 写操作和敏感数据一律用 POST
提交按钮放在 <form> 外面 点击不触发提交 按钮放 form 内，或用 form="formId" 属性关联
form 内 <button> 没写 type 默认 submit，点击导致页面刷新 非提交按钮显式写 type="button"
无刷新提交忘了 preventDefault() fetch 发了，页面同时也整页跳转了 submit 监听里第一行就 e.preventDefault()

九、与 Java 的类比

前端 form 概念Java / Spring 类比
action + method @PostMapping("/login") / @GetMapping("/search")——定位哪个 handler、用哪种 HTTP 方法
控件的 name 属性 @RequestParam("userName") 里的参数名——前后端必须对得上才能绑定
enctype HTTP 的 Content-Type——决定后端 MultipartResolver 走不走文件解析
multipart/form-data + type="file" MultipartFile 参数——专门接收上传文件
FormData 对象 请求体的封装——普通字段 → @RequestParam，文件字段 → MultipartFile
GET 幂等 / POST 不幂等 REST 语义里 GET 只读、POST 改数据——和后端接口设计原则完全一致
一句话串起来：表单提交就是浏览器替你拼了一个 HTTP 请求。 action 是 URL，method 是动词，name 是参数 key，enctype 是 Content-Type，控件值是参数 value。 你在后端写 @RequestParam 接到的每个值，源头都是某个带 name 的表单控件—— 理解了这条链路，前后端的数据流就彻底打通了。
