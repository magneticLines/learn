HTML · 语义化

页面结构标签：header / nav / main / aside / footer

过去整个页面是一堆 <div class="header"> 堆出来的，机器读不懂哪块是导航、哪块是正文。HTML5 给出了一套有语义的"地标"标签，让页面骨架既给人看，也给机器看。

一、什么是地标(landmark)标签

HTML5 新增的 header / nav / main / aside / footer 这一组标签，统称为地标(landmark)标签。它们本身的样式表现和 <div> 完全一样 （都是块级、没有默认外观），区别只在语义——它们明确告诉浏览器、搜索引擎、屏幕阅读器 "这一块在页面里扮演什么角色"。

"地标"这个词怎么来的？就像你在陌生城市靠"火车站、市政厅、大教堂"这些地标快速定位， 屏幕阅读器用户也靠这些标签在页面里跳转——一个快捷键就能跳到 main 正文、跳到 nav 导航， 不用从头听到尾。无语义的 div 不是地标，提供不了这种导航能力。

每个地标标签还自带一个对应的 ARIA landmark role，辅助技术正是读取这些 role 来构建"地标列表"的。 用语义标签 = 自动获得这些 role，不用手写 role="..."。

标签语义角色对应 ARIA role
<header>页眉 / 引导性内容banner（仅页面级时）
<nav>导航区navigation
<main>主内容main
<aside>侧边 / 补充内容complementary
<footer>页脚contentinfo（仅页面级时）

二、逐个拆解

<header> —— 页眉/引导性内容

header 放的是一块内容的"引导部分"：通常是站点 Logo、标题、搜索框，或者文章的标题和作者信息。 关键点：它不是全页唯一的。一个页面可以有页面级的 header（站点头）， 每篇 <article> 内部也可以有自己的 header（文章标题区）。

语义会随位置变化：只有当 header 是页面顶层（不嵌在 article/section/main/aside 内）时， 它才映射为 banner role（"全站页眉"）。嵌在文章里的 header 只是局部标题区，不是 banner。

<nav> —— 导航

nav 用来包主要的导航性链接组：主菜单、面包屑、分页器、目录。 注意"主要"和"导航性"两个限定——不是页面上所有链接都要塞进 nav。 正文里的一个外链、页脚里零散的版权链接，都不属于 nav。

<main> —— 主内容（全页唯一）

main 包裹页面的核心内容，也就是"去掉头、导航、侧栏、脚之后，这个页面真正想表达的东西"。 它有一条硬性规则：每个页面只能有一个可见的 main。正因为唯一，屏幕阅读器才能提供 "跳到主内容"(skip to main content)的快捷功能——用户按一下就跳过了每页都重复的导航，直奔正文。

为什么"唯一"是它的核心价值？因为唯一才能成为可靠的跳转目标。如果一页有好几个 main， "跳到主内容"就失去了确定的落点。这和"全站不能有两个首页"是一个道理——锚点必须唯一才有意义。

<aside> —— 侧边/补充内容

aside 放的是与主内容相关、但抽走也不影响主内容完整性的东西： 侧边栏、相关文章推荐、广告、术语解释、引用注释。判断标准：把它删掉，正文还读得通吗？读得通就适合放 aside。

<footer> —— 页脚/收尾信息

footer 放一块内容的收尾信息：版权、备案号、联系方式、相关链接，或文章末尾的作者署名、发布日期。 和 header 一样，它也不是全页唯一——页面有页面级 footer，文章也能有自己的 footer。 只有页面顶层的 footer 才映射为 contentinfo role。

三、完整页面骨架

一个语义化的页面骨架
<body>
  <!-- 页面级页眉：站点头，映射 banner -->
  <header>
    <h1>我的博客</h1>
    <!-- 主导航放在 header 里很常见 -->
    <nav>
      <a href="/">首页</a>
      <a href="/about">关于</a>
    </nav>
  </header>

  <!-- 全页唯一的主内容 -->
  <main>
    <article>
      <!-- 文章内部也能有自己的 header -->
      <header>
        <h2>为什么需要语义化</h2>
        <p>作者：Shea · 2026-06-23</p>
      </header>
      <p>正文内容……</p>
      <!-- 文章内部也能有自己的 footer -->
      <footer>标签：HTML、语义化</footer>
    </article>

    <!-- 与正文相关的补充内容 -->
    <aside>
      <h3>相关文章</h3>
      <ul><li>……</li></ul>
    </aside>
  </main>

  <!-- 页面级页脚：映射 contentinfo -->
  <footer>
    <p>© 2026 我的博客</p>
  </footer>
</body>

注意上面出现了两个 header、两个 footer（页面级各一个 + 文章内各一个），但只有一个 main——这正是地标标签的典型用法。

四、结构可视化

把上面的骨架画成色块，布局关系一目了然：

典型博客页面布局
<header> 站点头Logo / 标题
<nav> 主导航首页 · 关于 · 归档
<main> 主内容（全页唯一）<article> 正文，内部含自己的 header / footer
<aside>侧栏 / 相关
<footer> 站点脚版权 / 备案 / 联系方式
header（页眉） nav（导航） main（主内容） aside（补充） footer（页脚）
布局 ≠ 语义：图里把 aside 画在右边，只是常见视觉习惯。 地标标签不规定任何位置，aside 放左放右、nav 放顶放底，都靠 CSS 决定。 标签管的是"这块是什么"，CSS 管的是"这块画在哪"——两件事别混。

五、常见陷阱

陷阱问题正确做法
一个页面写多个 <main> 违反唯一性，"跳到主内容"失去确定落点，辅助技术行为混乱 每页只留一个可见 main
把正文里、页脚里的零散链接全塞进 <nav> nav 应只含"主要导航链接组"；滥用会让"导航列表"充满噪音 只包主菜单、面包屑、分页等导航性链接
以为全页只能有一个 header / footer 误解语义，文章/区块内部该有的局部页眉页脚被省略 页面级一个 + 每个 article/section 内可各有一个
继续用 <div class="header"> 代替语义标签 机器读不懂角色，丢失 landmark role，可访问性和 SEO 双输 能用语义标签就别用 div
把不相关内容硬塞进 <aside> aside 要求"与主内容相关"，无关内容放进去语义错误 只放删掉也不影响正文的补充信息
最高频的两个面试点：① main 全页唯一；② header/footer 可以多个、且语义随嵌套位置变化（顶层才是 banner/contentinfo）。这两条记牢，基本就答对了。

六、与 Java 的类比

地标标签概念Java / 后端类比
地标标签明确职责（header/nav/main/aside/footer） 分层架构里 Controller / Service / DAO 各司其职——一看类名就知道这层干什么，不用读实现
main 全页唯一 应用单一入口 public static void main()——一个程序只有一个明确的启动点
嵌套的局部 header / footer 每个模块/聚合根也有自己的边界与"门面"——局部模块在大系统里仍维持自己的封装
语义随嵌套位置变化（顶层 header = banner） 同一个 Bean 在不同 @Scope / 上下文里行为不同——角色由它所处的容器决定
用语义标签代替 <div class="..."> 用类型/注解表达意图，而非靠命名约定——编译器和工具能读懂，靠字符串约定则全凭自觉
一句话总结：地标标签干的事，和后端用分层、用类型系统表达职责是同一种思想—— 把"靠人脑记住的约定"变成"机器能直接读懂的结构"。结构清晰了，人和机器都受益。
