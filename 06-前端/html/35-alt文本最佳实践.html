alt 文本最佳实践

同一张图片，alt 写得好坏，决定了盲人用户能不能看懂你的页面。它不是"填上就行"的占位符，而是一段需要根据图片角色精心设计的替代文本。这是可访问性里最基础、却最容易写错的一环。

一、alt 到底在为谁服务：三大作用

很多人以为 alt 只是"图片裂了时显示的字"。其实它有三个完全不同的受众， 理解这三者，才能明白为什么 alt 不能随便写。

作用受众 / 场景说明
屏幕阅读器朗读 视障用户 读屏软件（NVDA / VoiceOver / JAWS）遇到 <img> 时，会朗读 alt 内容。这是 alt 最重要、最被忽视的作用。
加载失败显示 所有用户 图片 404、网络慢、用户关图省流量时，浏览器在原位置显示 alt 文本，保证信息不丢失。
图片 SEO 搜索引擎 爬虫"看不见"图片像素，只能靠 alt 理解图片内容，影响图片搜索排名。
关键认知：alt 的全称是 alternative text（替代文本）—— 它要替代的是这张图在页面里承担的功能/信息，而不是描述这张图"长什么样"。 这个区别是写好 alt 的总纲，后面所有规则都从它推导出来。

二、核心方法：按"图片角色"决定 alt 怎么写

写 alt 没有放之四海的模板。同一张图，放在不同位置、承担不同角色，alt 应该完全不同。 正确的思路不是"描述这张图"，而是先问自己：

决策第一问：如果把这张图整个删掉，用户会丢失什么？ 把"丢失的那部分"用文字补回来，就是这张图的 alt。如果删掉它用户什么都不会丢失（纯装饰），那 alt 就该是空的。

常见的图片角色有五类，逐一拆解：

① 信息图：描述传达的信息，而不是外观罗列

图片本身在传递信息（一张产品图、一张说明图）。alt 要写出这张图想让用户知道的事， 而不是机械罗列画面里有什么颜色、什么形状。

信息图：好 vs 坏
<!-- 场景：电商页一张红色连衣裙的产品图 -->

// ❌ 坏：罗列外观，啰嗦且没抓住重点
<img src="dress.jpg"
     alt="一张图片，里面有一个穿着红色衣服的女模特站在白色背景前">

// ✅ 好：传达用户真正关心的信息
<img src="dress.jpg"
     alt="红色 A 字版收腰中长连衣裙，V 领短袖">

坏例子里"一张图片""女模特""白色背景"对买衣服的用户毫无价值；好例子直接说清版型、领型、长度—— 这才是用户（和搜索引擎）真正需要的信息。

② 功能图：描述功能 / 目的，而不是图像

当图片被包在 <a> 或 <button> 里、本身是个可点击控件时， 用户点它是为了触发某个动作。此时 alt 要写"点了会发生什么"，而不是描述图标长相。

功能图：图标按钮 / 图片链接
// ❌ 坏：描述图标外观，用户听不懂这能干嘛
<button>
  <img src="magnifier.svg" alt="放大镜图标">
</button>

// ✅ 好：描述这个按钮的功能
<button>
  <img src="magnifier.svg" alt="搜索">
</button>

// ✅ 好：logo 链接，描述去哪而非长相
<a href="/">
  <img src="logo.svg" alt="返回首页">
</a>
陷阱：读屏用户听到"放大镜图标 按钮"，根本不知道按下去会发生什么。 他需要的是动作意图——"搜索 按钮"。功能图的 alt = 它的功能名，不是它的图案名。

③ 装饰图：alt=""（空，让辅助技术跳过）

纯装饰的图（分隔线、背景花纹、纯氛围插画、和旁边文字重复的配图）不承载任何信息。 它的正确 alt 是空字符串 alt=""。

装饰图：必须是空 alt，不是不写 alt
// ✅ 好：空 alt，读屏软件会完全跳过这张图
<img src="divider.png" alt="">

// ❌ 坏：漏写 alt 属性
//    读屏软件会去读 src 文件名 "divider.png"，制造噪音
<img src="divider.png">

// ❌ 坏：给装饰图编了个没用的描述
<img src="divider.png" alt="装饰性分隔线">
「空 alt」≠「不写 alt」，这是高频考点！
· alt=""：主动声明"这张图没有信息"，读屏软件会跳过它——这是正确做法。
· 完全不写 alt：读屏软件无法判断意图，多数会去朗读图片的 src 文件名（如 "IMG underscore 2024 dot jpg"），变成噪音。
所以：装饰图要写 alt=""，而不是省略它。

④ 复杂图（图表 / 数据）：简短 alt + 周边长描述

数据图表、流程图、信息量很大的示意图，一句 alt 装不下。 做法是：alt 写一句概括（图表说明了什么趋势/结论），完整数据放在图片旁边的可见文字或表格里。

复杂图：短 alt + 长描述分工
// ❌ 坏：把所有数据塞进 alt，读屏用户被一长串数字淹没
<img src="sales.png"
     alt="1月100万2月120万3月90万4月150万5月...">

// ✅ 好：alt 给结论，详细数据用 figcaption / 表格承载
<figure>
  <img src="sales.png"
       alt="2024 上半年销售额柱状图，整体上升，5 月达峰值">
  <figcaption>详见下方数据表</figcaption>
</figure>

<!-- 真正的数据用语义化 table 给出，对所有人都更友好 -->
<table> ... 月份 / 销售额 ... </table>
为什么这样分工：屏幕阅读器是线性朗读的，长 alt 没法像看图那样跳跃浏览。 把精确数据放进 <table>，读屏用户能逐行导航、对比单元格——比塞进 alt 体验好得多， 而且这份数据对正常用户、搜索引擎同样有价值。

⑤ 图片即文字（logo 含文字、文字截图）：alt 写出文字

如果图片里本身就有文字（带文字的 logo、印着 slogan 的 banner、一段文字的截图）， alt 就把图里的文字原样写出来——因为对用户而言，这张图承载的信息就是那些字。

图片即文字
// ✅ 好：logo 含品牌名文字，alt 写出文字
<img src="brand-logo.png" alt="Supernova">

// ✅ 好：促销 banner 上印着活动文案
<img src="promo.jpg"
     alt="618 大促 全场 5 折起 6 月 18 日截止">

// ❌ 坏：描述外观，丢掉了图里最关键的文字信息
<img src="promo.jpg" alt="一张红色的促销海报">
一句话总结角色决策：先判断图片角色 → 信息图写信息、功能图写功能、装饰图写空、复杂图短 alt 配长描述、文字图写出文字。 永远从"用户会丢失什么"出发，而不是"这张图长什么样"。

三、好 alt vs 坏 alt 速查

场景坏 alt好 alt
团队合影（关于页） "图片" "五人产品团队在办公室合影"
购物车图标按钮 "购物车图标" "查看购物车（3 件）"
文章封面（标题旁已有大标题） "文章配图" alt=""（与标题重复，作装饰）
错误提示里的红色叹号 "红色感叹号图标" "错误"
用户头像（评论区） "头像" "张三 的头像"
带文字的 logo "公司 logo" 写出 logo 里的品牌文字
纯背景花纹 "背景图案" alt=""

四、冗余措辞：别写"图片""image of""一张…的照片"

读屏软件在朗读 alt 前，已经先报出了"图像"这个角色（它知道这是个 <img>）。 所以你再在 alt 里写"图片""一张照片""image of…"，用户会听到双倍的"图像，图像里有…"，纯属噪音。

去掉冗余前缀
// ❌ 坏：读屏会读成 "图像，一张猫的照片"（重复）
<img src="cat.jpg" alt="一张橘猫趴在窗台上的照片">

// ✅ 好：直接说内容
<img src="cat.jpg" alt="橘猫趴在窗台上">
黑名单措辞：图片 / 图像 / 照片 / 截图 / image of / picture of / photo of / graphic of。 一律删掉，直接进入实际内容。例外：当"它是张照片还是插画"本身是有意义的信息时（如"梵高《星空》油画"），才保留。

五、alt 写多长：简洁优先，约 125 字符内

alt 应当简洁但完整。业界经验值是控制在 125 个字符以内（约一两句话）。 这个数字源于部分老旧读屏软件会在约 125 字符处截断 alt，更重要的是：太长的 alt 朗读起来令人疲惫， 用户没法像扫一眼图那样快速跳过。

  • 装的下一句就别用两句：alt 是替代信息，不是图片说明文学。
  • 需要更长？说明它不该全塞进 alt：用 <figcaption> 或周边正文承载长描述（见第二节复杂图）。
  • 不需要句号也行：alt 通常是短语而非完整句子，自然朗读即可。
自检小技巧：把你写的 alt 念出来，想象你在打电话跟一个看不见屏幕的朋友描述这一处—— 你会自然地说"搜索按钮""红色连衣裙"，而不会说"一张里面有放大镜图案的图片"。说人话，就对了。

六、常见错误速查

错误后果正确做法
装饰图省略 alt 属性 读屏朗读文件名，制造噪音 写 alt="" 主动声明无信息
功能图描述图标外观 用户不知道点了会怎样 写功能/动作（"搜索""返回首页"）
把整张图表数据塞进 alt 读屏被长串数字淹没 短 alt 给结论 + 表格/正文给数据
alt 以"图片/照片"开头 与读屏自带的"图像"角色重复 删掉冗余前缀，直接说内容
把 src 文件名当 alt "IMG_2024.jpg" 毫无意义 按角色重新写有意义的替代文本
用关键词堆砌 alt 做 SEO 体验差，且会被搜索引擎判作弊 写真实、准确的描述，SEO 自然受益

七、与 Java 的类比

alt 的写法哲学，其实和你写 Java 接口/方法时的"表达意图"思想高度一致—— 都在强调对外暴露"意图/契约"，而不是暴露"实现/外观"。

alt 写法原则Java 设计类比
alt 描述意图/信息，而非图片外观 方法名表达它做什么（findActiveUsers()），而非内部怎么实现（怎么遍历、怎么过滤）
装饰图主动写 alt="" 显式声明"无信息" Optional.empty() 显式表达"确实没有值"，而不是返回 null 让调用方猜
按图片角色（信息/功能/装饰）决定 alt 按职责设计接口——同一份数据，对外暴露什么取决于它在这个上下文承担的角色
功能图 alt 写功能名而非图案 面向接口编程：调用方只关心契约（能做什么），不关心实现细节（用什么图标/什么类）
复杂图：短 alt + 结构化长描述（table） 方法返回简明结果 + 详细数据用结构化 DTO/集合承载，而非把一切塞进一个字符串
本质上，写好 alt 和写好 Java 接口是同一种思维： 站在使用者的角度，暴露"对他有用的意图"，隐藏"对他无关的实现/外观"。 读屏用户是你图片的"调用方"，而 alt 就是你为他设计的 API 契约—— 设计得好，他无需"看见"实现，也能正确理解和使用。
