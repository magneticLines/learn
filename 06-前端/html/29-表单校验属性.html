表单内置校验属性

浏览器自带一套"不写一行 JS 就能拦住非法输入"的校验机制。但作为后端工程师，你要牢记一条铁律：这些校验只是体验糖衣，永远拦不住坏人——真正的防线在服务端。

一、为什么浏览器要内置校验

在内置校验出现前，"邮箱格式对不对、必填项填没填、密码够不够长"全靠 JS 手写。 每个表单都重复造轮子，且交互体验参差不齐。于是 HTML5 把最常见的校验规则做成了标签属性： 加一个 required、一个 type="email"，浏览器就会在用户点击提交时自动拦截、弹出原生气泡提示。

它的定位：内置校验解决的是"即时反馈"——让用户在还没发请求前就知道哪里填错了， 省掉一次往返。它不是安全机制，这一点贯穿整篇，后面会反复强调。

二、核心校验属性一览

常用校验属性
<!-- required：必填，留空会被拦截 -->
<input name="username" required>

<!-- minlength / maxlength：字符串长度区间 -->
<input name="pwd" type="password" minlength="8" maxlength="32">

<!-- min / max / step：数值范围与步长（对 number/date/range 生效）-->
<input type="number" name="age" min="18" max="120" step="1">

<!-- pattern：用正则约束格式（这里要求 11 位手机号）-->
<input name="phone" pattern="1[3-9]\d{9}" title="请输入 11 位手机号">

<!-- type=email / url：浏览器内置格式校验 -->
<input type="email" name="mail">
<input type="url" name="site">
属性作用适用控件
required非空校验，留空即非法几乎所有输入控件
minlength / maxlength字符数下限 / 上限text、password、textarea 等
min / max数值 / 日期的最小 / 最大值number、range、date、time
step合法步长（如只能填偶数、0.5 的倍数）number、range、date
pattern用正则约束文本格式text、tel、search、url、password、email
type="email"内置邮箱格式校验—
type="url"内置 URL 格式校验—
maxlength 和其他属性的微妙区别：maxlength 是硬性限制—— 浏览器根本不让你输入超过上限的字符，所以它"不会触发校验失败"（因为压根输不进去）。 而 minlength、pattern 等是提交时才校验，可以正常输入但提交会被拦。

三、原生校验是怎么工作的

当用户点击 submit 按钮（或 JS 调用 form.requestSubmit()）时，浏览器会触发原生校验：
⚠️ 注意：直接调用 form.submit() 会绕过校验、也不触发 submit 事件；要走原生校验流程必须用 requestSubmit()。

  • 从上到下逐个检查每个带约束的控件；
  • 遇到第一个不合法的控件，阻止表单提交，把焦点移到它身上；
  • 在该控件旁弹出一个原生气泡提示（Constraint Validation 的内置文案，如"请填写此字段"）。
一个会被自动拦截的表单
<form>
  <input type="email" name="mail" required>
  <!-- 留空或邮箱格式错误，点提交会被气泡拦下 -->
  <button>提交</button>
</form>

气泡提示的文案默认跟随浏览器语言，不可直接用 CSS 完全定制（只能通过 JS 改文字，见第六节）。 给控件加 title 属性时，pattern 校验失败的气泡会把 title 内容附带显示出来—— 所以 pattern 几乎总应该配一个 title 说明格式要求。

四、:valid / :invalid 做视觉反馈

光靠提交时的气泡还不够友好。CSS 提供了 :valid 和 :invalid 伪类， 让你实时给合法 / 非法的控件上不同样式（绿框 / 红框）：

实时边框反馈
/* 合法：绿色边框 */
input:valid { border-color: #4ade80; }

/* 非法：红色边框 */
input:invalid { border-color: #f87171; }

/* 进阶：只在用户碰过的字段上显示红框，避免一进页面满屏红 */
input:user-invalid { border-color: #f87171; }
体验陷阱与解法：:invalid 从页面加载那一刻就生效，会导致还没填的必填项一进来就标红，很难看。 现代浏览器提供了 :user-invalid（曾用名 :-moz-ui-invalid），只在用户交互过该字段后才判定，体验好得多。

五、关闭原生校验：novalidate

有时你想完全用 JS 接管校验（比如要统一展示错误样式、做联动校验），可以在 form 上加 novalidate， 关掉浏览器的自动拦截和气泡。注意：约束属性依然存在，只是不再自动触发—— 你仍可以用 JS 的 Constraint Validation API 主动读取校验结果。

关掉原生气泡，自己接管
<form novalidate>
  <input name="mail" type="email" required>
  <button>提交</button>
</form>

六、自定义校验：Constraint Validation API

内置规则覆盖不了所有场景（如"两次密码必须一致""用户名是否已被占用"）。 这时用 JS 的 Constraint Validation API。三个核心成员：

成员作用
el.validity一个对象，按维度告诉你为什么不合法（如 valueMissing、tooShort、patternMismatch、typeMismatch）；全合法时 validity.valid === true
el.checkValidity()返回布尔：当前是否合法（不合法会触发 invalid 事件，但不弹气泡）
el.setCustomValidity(msg)设置自定义错误信息：传非空字符串 → 该控件变为非法且气泡显示此文案；传空字符串 → 清除，恢复合法
两次密码一致性校验
const pwd = document.querySelector('#pwd')
const confirm = document.querySelector('#confirm')

confirm.addEventListener('input', () => {
  if (confirm.value !== pwd.value) {
    confirm.setCustomValidity('两次密码不一致')  // 标记为非法
  } else {
    confirm.setCustomValidity('')            // 关键：传空串清除，否则永远 invalid
  }
})
最容易踩的坑：调用 setCustomValidity('某错误') 后，如果不在条件恢复时调用 setCustomValidity('') 清空， 那个控件会被永久标记为非法——哪怕用户改对了也提交不了，而且毫无提示，极难排查。 记住：设了自定义错误，就必须配套清除逻辑。

七、铁律：前端校验永远不能替代后端校验

这是整篇最重要的一节。前端的所有校验——required、pattern、type="email"、 JS 自定义校验——全部运行在用户的浏览器里，全部可以被绕过。它们只是体验优化，绝不是安全防线。

攻击者（或仅仅是好奇的用户）有无数种方式跳过前端校验：

  • 打开开发者工具，直接删掉 required 属性、改掉 pattern；
  • 给 form 加上 novalidate，原生校验瞬间失效；
  • 根本不用浏览器——用 curl / Postman / 脚本直接对你的接口发请求，前端那套 HTML 压根不参与。
绕过前端校验，直接打接口
# 前端写了 age 必须 18-120，但攻击者直接发：
curl -X POST https://api.example.com/register \
  -H 'Content-Type: application/json' \
  -d '{"username":"","age":-999,"mail":"not-an-email"}'

# 浏览器里的 required / min / max / type=email 全都没机会执行

作为后端工程师，你应该把这件事内化成肌肉记忆：到达服务端的每一个字段都是不可信的， 必须在后端用 Bean Validation（@NotNull / @Size / @Pattern / @Email 等）重新完整校验一遍。 前端校验和后端校验不是二选一，而是纵深防御的两层：前端管体验，后端管安全。

类比：前端校验就像超市入口的"请勿携带宠物"告示牌——提醒有素质的顾客，但拦不住真要硬闯的人。 后端校验才是真正的保安。告示牌可以撤掉（绕过），保安不能没有。

八、陷阱速查

陷阱后果正确做法
只做前端校验，后端不校验 安全漏洞：脏数据 / 攻击数据直接入库 后端必须用 Bean Validation 重新校验
pattern 正则没写完整匹配 浏览器会自动给 pattern 包上 ^(?:...)$，但很多人误以为是"包含匹配"，写错规则放过非法值 记住 pattern 是整串完整匹配，照此写正则
对 hidden / disabled 字段加 required 校验不生效——隐藏和禁用控件被排除在校验之外，且 disabled 字段不会随表单提交 需要校验就别 disabled；隐藏可用 readonly 或 JS 校验
setCustomValidity 后忘记传空串清除 控件被永久标记 invalid，改对了也提交不了 条件满足时务必 setCustomValidity('')
用 :invalid 而非 :user-invalid 页面一加载所有空必填项就标红，体验差 用 :user-invalid，或交互后才加样式类

九、与 Java 的类比

前端校验概念Java / 后端类比
required Bean Validation 的 @NotNull / @NotBlank
minlength / maxlength @Size(min=..., max=...)
min / max @Min / @Max / @Range
pattern @Pattern(regexp = "...")
type="email" @Email
前端校验可被绕过 永远不信任客户端输入——所有外部输入到了 Controller 都要 @Valid 重新校验
前端校验 + 后端校验（双重） 纵深防御（Defense in Depth）：多层校验，任何一层都不假设上一层可靠
Constraint Validation API（JS 自定义） 自定义 ConstraintValidator / @Constraint 注解，处理内置规则覆盖不了的业务校验
一句话收尾：前端校验是给用户的礼貌，后端校验是给系统的保险。 礼貌可以缺席（被绕过），保险绝不能少。两者各司其职，组成纵深防御—— 这和你在后端"DTO 校验 + Service 业务校验 + 数据库约束"层层设防的思路，本质完全一致。

与其他知识点的联系

  • 关联知识点：input 类型（26 号文档）不同 type 自带不同的默认校验行为——type="email" 会自动做邮箱格式校验，type="url" 会自动做 URL 格式校验，type="number" 会配合 min/max/step 做数值范围与步长校验；本文讲的 pattern、required 等属性正是叠加在这些类型之上，二者是"类型决定基础校验规则，属性决定具体约束值"的分层关系。
  • 关联知识点：React 受控组件在 React 里表单值由 state 驱动，原生的气泡校验和 :invalid 伪类容易和受控渲染打架（比如 state 更新触发重渲染，气泡消失）；实践中常见做法是保留 required/pattern 等属性做基础的语义化提示，但把真正复杂的校验逻辑（跨字段联动、异步校验）交给 onChange/onBlur 里手写的 JS 判断，这和本文第六节 Constraint Validation API 的思路是一脉相承的，只是触发时机换成了 React 的事件回调。
  • 关联知识点：zod / yup 等 JS 校验库这类库把校验规则从"HTML 属性"搬到了"JS 对象/schema"里描述，本质是把本文的 required/minlength/pattern 等声明式约束改写成可编程、可复用、可跨表单共享的 schema；优势是能表达"两次密码一致""用户名唯一"这类内置属性做不到的复杂规则（对应本文第六节提到的场景），代价是失去了浏览器原生的即时气泡反馈，需要自己接管错误展示。
  • 关联知识点：可访问性（ARIA）原生气泡提示本身对屏幕阅读器不够友好，尤其是关掉 novalidate 自己接管校验之后，必须手动给非法控件加 aria-invalid="true"，并用 aria-describedby 把错误提示文本关联到输入框，这样屏幕阅读器才能像看到红框的用户一样感知到"这个字段有问题、问题是什么"；这也是第五节"关闭原生校验后要自己接管"里被省略的一块重要拼图。
  • 关联知识点：JS 的 checkValidity() / ValidityState / setCustomValidity()本文第六节已经出现的这套 API 是本节其他关联点的技术基础——无论是 React 里手写校验逻辑，还是给 zod 校验失败的字段补一个原生的非法状态，最终都可以通过 setCustomValidity() 把结果"接回"浏览器的 Constraint Validation 体系，从而复用原生的 :invalid 样式钩子和表单提交拦截机制，而不必完全推倒重造一套 UI 反馈。
