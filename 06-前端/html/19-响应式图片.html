响应式图片：srcset / picture

一张 2000px 宽的桌面大图，手机其实只需要 400px。如果不加处理，用户在 4G 下白白下载了几倍的流量，首屏还慢。响应式图片就是让浏览器"按需取图"——同一个 <img>，不同设备拿到不同的文件。

一、先理解问题：为什么一张图不够用

传统写法 <img src="photo.jpg"> 只有一个文件地址，所有设备拿到的都是同一张图。 但真实世界里，设备千差万别，这带来两个独立的问题：

问题表现本质
手机下载桌面大图 显示框只有 375px 宽，却下载了 2000px 的文件，流量和解码全浪费 尺寸不匹配：图太大
高分屏看着模糊 Retina / 安卓旗舰是 DPR=2 甚至 3，1 个 CSS 像素对应 2×2 个物理像素，普通图被拉伸发虚 清晰度不够：图太小
关键概念 DPR（Device Pixel Ratio，设备像素比）：CSS 里写 width: 300px， 在 DPR=2 的屏幕上实际由 600 个物理像素来绘制。所以要"看起来清晰"， 图片源文件的真实宽度要 ≥ CSS 宽度 × DPR。这就是为什么高分屏需要更大的图。

注意这两个问题方向相反：一个嫌图太大（费流量），一个嫌图太小（不清晰）。 响应式图片的目标，就是让浏览器在"省流量"和"够清晰"之间，根据当前设备自己算出最优解。

二、srcset：给浏览器一个"候选清单"

srcset 的思路很简单：你别只给一个文件，给我一个候选列表， 并标注每个候选的"规格"，剩下的"挑哪个"交给浏览器。它有两种标注规格的方式。

方式一：密度描述符 1x / 2x（解决高分屏）

按 DPR 切换，适合固定显示尺寸的图（如图标、头像）
<img src="avatar.jpg"
     srcset="avatar.jpg 1x,
             avatar@2x.jpg 2x,
             avatar@3x.jpg 3x"
     alt="用户头像">

2x 的意思是"这张图适合 DPR=2 的屏幕"。普通屏取 1x， Retina 取 2x，浏览器自动判断。注意 src 仍要保留，作为不认识 srcset 的老浏览器的兜底。

方式二：宽度描述符 400w + sizes（解决变宽布局）

密度描述符有个局限：它假设图片的显示宽度是固定的。但响应式布局里， 图片宽度本身就随屏幕变化（手机占满屏、桌面占半屏）。这时要用宽度描述符。

srcset 标真实宽度，sizes 告诉浏览器"我会显示多宽"
<img
  srcset="photo-400.jpg 400w,
          photo-800.jpg 800w,
          photo-1200.jpg 1200w"
  sizes="(max-width: 600px) 100vw,
         (max-width: 1000px) 50vw,
         400px"
  src="photo-800.jpg"
  alt="风景照">

这两个属性是配合工作的，缺一不可：

  • 400w 里的 w 是文件的真实像素宽度（photo-400.jpg 本身就是 400px 宽）。
  • sizes 用类似媒体查询的语法，告诉浏览器这张图在不同视口下会被渲染成多宽。
浏览器是怎么挑的？以屏幕宽 375px、DPR=2 为例： sizes 命中第一条 100vw，即显示宽度 = 375px； 乘以 DPR = 需要 750px 的图；候选里 800w 最接近且不小于它，于是下载 photo-800.jpg。 整个决策在 HTML 解析阶段就完成了，比 CSS 和 JS 更早，所以能尽早开始下载。
为什么不能让浏览器自己量宽度？因为图片下载发生在"预加载扫描"阶段， 那时 CSS 还没解析、布局还没算出来，浏览器根本不知道图最终多宽。 sizes 就是你提前用文字把布局意图告诉它，让它能在布局完成前就做出选择。 所以 srcset（宽度描述符）不写 sizes 是典型错误——浏览器只能假设 100vw，往往取了过大的图。

三、picture：你来决定，不让浏览器挑

srcset 解决的是"同一张图的不同尺寸"——内容相同，只是大小/清晰度不同。 但有时你需要的是完全不同的图，这时就轮到 <picture> 了。它有两大用途。

用途一：艺术指导（Art Direction）——不同断点用不同构图

一张宽幅风景照在手机竖屏里会变得又扁又小，主体看不清。更好的做法是： 桌面用宽幅原图，手机换成裁剪后突出主体的竖图。这不是缩放能解决的，是"换一张图"。

用 media 条件切换不同裁剪的图
<picture>
  <!-- 窄屏：竖向裁剪，突出主体 -->
  <source media="(max-width: 600px)"
          srcset="hero-portrait.jpg">
  <!-- 宽屏：横向宽幅原图 -->
  <source media="(min-width: 601px)"
          srcset="hero-wide.jpg">
  <!-- 兜底 + alt 写在这里 -->
  <img src="hero-wide.jpg" alt="产品首图">
</picture>

用途二：格式降级——优先用更省的新格式

AVIF、WebP 比 JPG 体积小很多，但老浏览器不认。<picture> 可以让浏览器 从上往下找第一个自己支持的格式，自动降级，不需要你写 JS 判断。

AVIF → WebP → JPG 逐级降级
<picture>
  <source type="image/avif" srcset="photo.avif">
  <source type="image/webp" srcset="photo.webp">
  <!-- 所有 source 都不支持时，回退到这个 img -->
  <img src="photo.jpg" alt="风景照">
</picture>
source 是"短路匹配"，顺序就是优先级。浏览器从第一个 <source> 开始检查 type / media 是否满足，命中第一个就停下，后面的全部忽略。 所以必须从最优（AVIF）排到兜底（JPG）——这和 Java 异常处理里"子类 catch 必须写在父类前面"是同一个道理：顺序错了，后面的分支永远轮不到。
三个必须记住的硬规则：
1. <picture> 里必须有一个 <img> 作为最终兜底，否则什么都不显示——<source> 自己不会渲染任何东西，它只是"给 img 出主意"。
2. alt 和 src 永远写在 <img> 上，不要写在 <picture> 或 <source> 上（写了也无效）。
3. <source> 用 srcset 属性指定地址，没有 src。

四、分工：srcset 和 picture 各管一摊

新手最大的困惑是"到底用哪个"。记住一句话： srcset 解决"同一张图、不同尺寸"，picture 解决"不同的图、不同的格式"。

需求用哪个谁来决策
同图，按 DPR 给高清版 srcset + 2x 浏览器自动挑
同图，按显示宽度给合适尺寸 srcset + w + sizes 浏览器自动挑
不同断点用不同构图（艺术指导） <picture> + media 你用条件指定
优先新格式、自动降级 <picture> + type 你排优先级
它们可以叠加用：<source> 里照样能写 srcset + sizes。 比如"WebP 格式 + 多尺寸 + 按视口选"可以一起上。底层逻辑是： picture 先帮你选出"哪一组候选"（哪个格式/哪个断点）， 选中的那组里再由 srcset/sizes 选"具体哪个文件"。一层选组，一层选文件。

五、常见陷阱速查

陷阱后果正确做法
用宽度描述符 w 却不写 sizes 浏览器只能假设 100vw，几乎总取最大图，省流量失败 宽度描述符必须配 sizes
把 alt 写在 <picture> 上 无障碍读屏拿不到替代文本，等于没写 alt 只写在内部的 <img>
<source> 顺序写反（JPG 在 AVIF 前） 命中第一个就停，永远用不上更优的格式 从最优到兜底排列
<picture> 里忘了写 <img> 所有 source 没命中时整个图区空白 必须保留一个 <img> 兜底
w 写成"显示宽度"而非"文件真实宽度" 浏览器计算依据错误，挑错文件 w = 文件本身的像素宽度

六、与 Java / 后端的类比

这套机制的设计思想，和后端工程师天天打交道的几个模式高度同构：

响应式图片概念Java / 后端类比
<picture> 多个 <source> 逐个尝试、命中即停 责任链 / 策略降级：一串 handler 顺序尝试，第一个能处理的接管，类似 try { avif } catch { webp } catch { jpg } 或 Spring 的 ViewResolver 链
按设备能力（格式支持 / 屏幕）选不同图 内容协商：HTTP Accept 头让客户端声明能接受的类型，服务端按优先级返回最合适的表示（同一资源，多种 representation）
srcset + sizes 多分辨率候选自动选 CDN 多分辨率适配：一份资源预生成多档规格，边缘节点按请求参数（设备/DPR）回不同档位，本质是空间换带宽
source 顺序即优先级，子类在前 catch 块顺序：具体异常必须写在父类异常之前，否则永远进不去——匹配规则都是"从上到下，命中即止"
浏览器在解析阶段提前决策、尽早下载 预测/预取优化：在拿到完整上下文前用声明式提示提前调度 I/O，类似数据库的预读、JVM 的预测分支
一句话收尾：srcset 是"让浏览器自动算"，picture 是"让你显式声明"。 前者把决策权交给运行时（像策略模式按上下文自动选实现）， 后者把决策权握在自己手里（像配置文件里写死的优先级链）。 两者解决的都是同一个根本问题——同一份内容，针对不同环境给出最优表示， 这正是后端"内容协商"思想在前端的镜像。
