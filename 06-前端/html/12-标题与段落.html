标题与段落：h1~h6 与 p

很多人把 <h1>~<h6> 当成"字号 1 到 6 号"来用，这是最普遍的误解。它们不是字体大小控件，而是文档的语义大纲层级——就像 Java 代码里的包结构和类的访问层级，反映的是结构关系，不是外观。

一、标题不是字号，是大纲层级

HTML 提供六级标题 <h1> 到 <h6>。初学者最容易踩的坑是 "想要小一点的字就用 h3，想要大一点就用 h1"—— 把语义标签当成了样式开关。

标题的真正职责是声明这段内容在文档结构中的层级。 浏览器、搜索引擎爬虫、屏幕阅读器都会根据 h1~h6 的嵌套关系， 在脑子里（或算法里）构建出一棵文档大纲树。字号大小只是浏览器给的默认样式， 完全可以用 CSS 改掉，但层级语义改不掉。

核心心智模型：把标题想象成 Word/Markdown 里的"一级标题、二级标题"。 你选 ## 还是 ###，是在决定它在目录里的缩进位置， 而不是在调字号。HTML 的 h1~h6 完全一样。
语义 vs 样式：两个职责要分开
<!-- ❌ 错：为了字号小就用 h4，破坏了层级语义 -->
<h4>这其实是页面主标题，只是我嫌 h1 太大</h4>

<!-- ✅ 对：语义用 h1，外观交给 CSS -->
<h1 class="page-title">页面主标题</h1>
/* CSS 里随便调小：.page-title { font-size: 1rem; } */

二、标题层级的三条规则

把标题用对，记住三条规则就够了：

规则说明为什么
一个页面只有一个 h1 h1 是整个页面的主题，唯一 爬虫和阅读器靠它判断"这页讲什么"，多个 h1 等于主题不明
层级连续，不跳级 h1 下面接 h2，不能直接跳到 h4 跳级会让大纲树出现"断层"，结构关系丢失
按语义选级别，不按视觉 "它是不是上一节的子话题"决定级别，跟字号无关 字号用 CSS 控制，标签只表达从属关系
跳级是常见错误
<!-- ❌ 错：从 h2 直接跳到 h4，中间断了一层 -->
<h1>JVM 内存模型</h1>
  <h2>堆内存</h2>
    <h4>新生代</h4>  <!-- 跳过了 h3 -->

<!-- ✅ 对：逐级递进 -->
<h1>JVM 内存模型</h1>
  <h2>堆内存</h2>
    <h3>新生代</h3>

三、可视化：标题构成的大纲树

下面这段标题结构，浏览器和爬虫会在内部解析成一棵带缩进的大纲树—— 这正是屏幕阅读器用户按 H 键能逐级跳转的依据：

一份合理的标题大纲 → 自动生成的文档结构
h1 Java 并发编程
└─ h2 线程基础
   └─ h3 线程的生命周期
   └─ h3 创建线程的方式
└─ h2 锁机制
   └─ h3 synchronized
      └─ h4 偏向锁 / 轻量级锁 / 重量级锁

缩进就是嵌套深度。h3 之所以"属于"上面的 h2， 不是因为它写在下面、字小一点，而是因为级别数字更大—— 这套规则让机器无需理解内容，仅凭标签就能还原结构。

四、为什么对 SEO 和无障碍这么重要

标题层级不是"洁癖"，它直接影响两类真实用户：

角色怎么用标题
搜索引擎爬虫 把 h1 当作页面主题的最强信号，按 h2/h3 理解内容层次，影响排名与摘要展示
屏幕阅读器（视障用户） 可生成"标题列表"导航，用户按 H / 数字键在各级标题间跳转，像看目录一样快速定位
浏览器 / 阅读模式 / 插件 自动提取大纲、生成 TOC（目录）、做"回到顶部 / 跳到某节"
换位思考：视障用户不会从头听到尾，他们靠标题"跳读"。 如果你把主标题用 h4、把无关大字也用 h1， 他们的导航目录就会一团乱——相当于你给同事一份没有目录、章节号全乱的文档。

五、段落 p：文本的基本单位

<p>（paragraph）是承载正文文本的块级元素（block-level）。 块级的含义是：它默认独占一行，前后自带外边距，宽度撑满父容器—— 就像 Word 里敲一个回车另起一段。

段落是结构化的，不是"换行凑出来的"
<p>这是第一段。浏览器会把连续的空白、换行都折叠成一个空格，
所以源码里怎么排版，不影响段落内的显示。</p>

<p>这是第二段，自然和上一段之间有间距。</p>

注意 <p> 内部的多个空格和换行会被浏览器折叠成一个空格。 想真正分段，就用多个 <p>，而不是在一个段落里硬敲回车或加 <br>。

六、陷阱一：用 <br> 制造段落间距

最常见的错误：为了让两段文字之间有空隙，连敲两个 <br>。 这在视觉上"看起来对了"，但语义上是错的—— <br> 表示"段内强制换行"（如诗歌、地址），不表示"新段落"。

间距应该来自结构 + CSS，而不是空标签
<!-- ❌ 错：用 br 凑间距，机器看不出这是两段 -->
第一段内容<br><br>
第二段内容

<!-- ✅ 对：两个语义段落，间距交给 CSS margin -->
<p>第一段内容</p>
<p>第二段内容</p>
/* CSS: p { margin-bottom: 1em; } 控制段间距 */
本质区别：<br> 是"内容里就该换行"（如一首诗的每一句）， 属于内容本身；段落间距是排版表现，属于样式，归 CSS 管。 用空 <br> 凑间距，等于把样式硬塞进结构， 和 Java 里把魔法数字硬编码进业务逻辑一个味道。

七、陷阱二：p 里不能放块级元素

<p> 在 HTML 规范里只允许包含短语内容（phrasing content，即行内元素和文本）， 不允许嵌套块级元素如 <div>、<ul>、<h2>。 一旦你这么写，浏览器会静默地提前闭合这个 <p>， 结果 DOM 结构和你写的完全不一样。

浏览器会偷偷改写你的 DOM
<!-- 你写的 -->
<p>
  文字开头
  <div>里面的盒子</div>
  文字结尾
</p>

<!-- 浏览器实际解析成（p 在 div 前被强制闭合）-->
<p>文字开头</p>
<div>里面的盒子</div>
文字结尾
<p></p>  <!-- 多出一个空 p -->
为什么会"自动闭合"？HTML 解析器有一套容错规则， 遇到不该出现在 <p> 里的块级标签，它假定你忘了写 </p>， 就替你补上。这就是为什么你的 CSS 选择器有时"莫名失效"—— 用浏览器 DevTools 看 Elements 面板里的真实 DOM， 往往和源码不一样。需要包裹块级内容时，外层用 <div> 而不是 <p>。

八、陷阱三：多个 h1 / 标题当大字用

反模式后果正确做法
一个页面写多个 h1 主题不明，SEO 权重分散，阅读器目录混乱 全页一个 h1，其余用 h2+
为了字大用 h1 / 为了字小用 h5 大纲树和实际内容层级不符 选级别看语义，字号用 font-size
用 <br><br> 分段 无段落语义，间距不可维护 <p> + margin
在 <p> 里塞 <div> p 被强制闭合，DOM 错乱 外层改用 <div> / <section>
面试高频：（1）h1~h6 的作用是什么、为什么不能跳级； （2）<p> 能否嵌套 <div>（不能，会被自动闭合）； （3）block 与 inline 元素的区别； （4）为什么不用 <br> 排版。这几道几乎是语义化 HTML 的必考题。

九、与 Java 的类比

HTML 概念Java 类比
标题层级 h1>h2>h3 包结构层级 com.x.y.z——通过层级表达模块归属，而不是随便堆名字
层级不能跳级（h2→h4） 分层架构不能越层调用（Controller 不能直接打 Mapper，要经 Service）
按语义选标签，不按外观 方法/类命名按职责语义化（calcTax() 而非 doStuff2()）
一个页面只有一个 h1 一个 .java 文件只有一个 public 顶级类——主题唯一、入口明确
样式与结构分离（CSS 管字号） 关注点分离：业务逻辑与配置分开，魔法值抽成常量/配置文件
<p> 不能嵌块级（语法约束） 编译期/规范约束（如 final 不可重赋值、接口方法签名限制）——违反就被"纠正"或报错
一句话总结：HTML 标签选的是语义，不是外观。 就像写 Java 时你按职责分包、分层、命名，是为了让人和工具读懂结构； 写 HTML 时按语义选 h1~h6 和 p， 是为了让浏览器、爬虫、屏幕阅读器读懂你的文档结构。外观，永远交给 CSS。

与其他知识点的联系

  • 行内文本标签（下一节 strong/em）：标题解决的是"块级结构该分几层"，下一节的 strong/em 解决的是"一段文字里哪个词更重要"——两者是同一套"语义优先于样式"心法在不同粒度上的应用，学完标题再看行内标签会更容易理解为什么不能拿 b/i 图省事。
  • 列表与表格语义分组：标题构建的是纵向的文档大纲树，列表（ul/ol）和表格的 thead/tbody 则是在标题划出的某个章节内部，进一步表达并列关系或行列关系——大纲树的每个叶子节点下面，往往就挂着这些更细粒度的结构。
  • CSS 的 :nth-child 与选择器：本文强调字号只是默认样式，可以被 CSS 完全覆盖，实际项目里几乎一定会写 h1~h6 的自定义样式（字号、行高、上下间距），这正是本文档演示用的 .hero h1、h2 选择器所做的事——语义标签负责"是什么"，选择器负责"长什么样"。
  • 可访问性（accessibility）：屏幕阅读器按 H 键跳转标题、生成标题列表导航，这个能力完全依赖标题层级的正确性和连续性，是本文"陷阱一二三"背后最直接的现实后果，也是 aria-* 属性、role 等无障碍知识点的天然延伸方向。
  • SEO 与结构化数据：搜索引擎依赖 h1 判断页面主题、依赖 h2/h3 理解内容层次，这与后续可能学到的 <meta> 描述标签、Open Graph、结构化数据（JSON-LD）共同构成"让机器读懂页面"的完整拼图，标题层级是其中最基础、成本最低的一环。
  • React JSX 中的标题使用：在 React 组件里封装 <Heading level={2}> 这类组件时，本质上就是把"选对语义级别"这件事做成可复用的抽象——组件内部根据 level 动态渲染 h1~h6，但依然要遵守本文"不跳级、全局唯一 h1"的规则，语义约束不会因为换了框架就消失。
