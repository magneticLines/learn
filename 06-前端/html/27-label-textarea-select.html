HTML · 表单

label / textarea / select

表单不只有 input。label 是无障碍的灵魂，textarea 和 select 则各有自己"反直觉"的取值方式——它们的值不在你以为的那个属性里。这三个标签的坑，几乎每个新手都踩过。

一、label：给控件配一个"标签"

<label> 本身不是输入控件，它的唯一职责是把一段说明文字和某个表单控件绑定起来。 绑定后它能带来三个实打实的好处，下面会一一拆开。先看两种写法：

label 的两种关联写法
<!-- 写法① for + id：label 的 for 指向控件的 id -->
<label for="username">用户名</label>
<input id="username" name="username">

<!-- 写法② 包裹：把控件直接放进 label 里，无需 id -->
<label>
  用户名
  <input name="username">
</label>

两种写法等价，但 for/id 写法更灵活——label 和控件可以分处不同的布局容器（比如左列放 label、右列放 input）， 而包裹写法要求两者在 DOM 上嵌套，结构被绑死。实际项目里 for/id 用得更多。

label 到底解决了什么问题

  • 点击 label 自动聚焦/触发控件：点"用户名"三个字，光标就跳进输入框；对 checkbox / radio 更关键——文字和小方框都能点，不用精准戳那个几像素的圆点。
  • 扩大点击热区：手机上手指粗，一个 12px 的 radio 很难点中。关联 label 后，整行文字都成了点击区，可用性大幅提升。
  • 屏幕阅读器朗读（无障碍核心）：盲人用户用读屏软件 Tab 到输入框时，软件会朗读"用户名，编辑框"。没有 label，它只会念"编辑框"——用户根本不知道这里要填什么。
为什么无障碍这么重要？它不只是"照顾少数人"。很多国家/企业（尤其政府、金融、大厂）有法律或合规要求（如 WCAG、美国 Section 508）， 表单缺 label 会直接审计不通过。而且关联了 label 的控件，自动化测试和表单填充工具也更容易定位——这是有连带收益的基础工程素养。
陷阱：label 没真正关联上。最常见的两种"假关联"：① 只写了 <label>用户名</label> 却没有 for， 也没包裹控件——它只是一段普通文字，无障碍价值为零；② 写了 for="username" 但控件的 id 拼错或根本没设 id，for 找不到目标，同样失效。 判断标准很简单：点一下文字，光标会不会跳进输入框。不跳，就是没关联上。

二、textarea：多行文本框

<input> 是单行的，要输入大段文字（评论、备注、地址）就得用 <textarea>。 它最容易让从 input 过来的人栽跟头的地方是：初始值不写在 value 属性里，而是写在标签的内容（开闭标签之间）。

textarea 的值放在标签内容里
<!-- ✅ 正确：默认文本写在开闭标签之间 -->
<textarea name="bio" rows="4" cols="40">这是默认内容</textarea>

<!-- ❌ 错误：textarea 没有 value 属性，这样写默认文本不会显示 -->
<textarea name="bio" value="这是默认内容"></textarea>
陷阱：把 input 的习惯带过来用 value 属性。浏览器会直接忽略 textarea 上的 value，框里一片空白，你还以为是数据没传过来。 根本原因：textarea 要承载带换行的多行文本，HTML 属性值表达换行很别扭，所以设计成"内容即值"。记住一句话：input 用属性，textarea 用内容。

常用属性：

  • rows / cols：可见的行数和列数（字符宽度），决定初始尺寸。注意它们只是初始建议值，用户通常能拖动改变大小。
  • CSS 的 resize：控制用户能否拖拽缩放。resize: none 禁止拖动，vertical 只允许上下拉，both（默认）随意拖。实际项目里常用 CSS 控宽高、用 resize: vertical 防止用户把框拉变形破坏布局。
用 CSS 控制尺寸更可靠
<textarea name="comment" style="width:100%; height:120px; resize:vertical"></textarea>
小细节：textarea 标签内容里的换行和空格会被原样保留。所以为了让代码好看而把 <textarea> 内容缩进、换行书写时， 那些缩进空格也会进到框里变成默认值。要默认空，就紧贴着写：<textarea></textarea>，中间不留任何字符。

三、select + option：下拉选择

<select> 是下拉框，里面用 <option> 列出每个选项。它的取值规则是本节最重要的一点： select 提交/读取的值，是被选中那个 option 的 value 属性，而不是用户看到的那行文字。

value 与显示文本是两回事
<select name="city">
  <!-- value 是提交的值，标签内文字是给人看的 -->
  <option value="bj">北京</option>
  <!-- selected：页面加载时默认就选中这一项 -->
  <option value="sh" selected>上海</option>
  <option value="gz">广州</option>
</select>

用户选了"北京"，但表单实际提交的是 city=bj。这种"显示值 / 实际值"分离的设计非常有用： 前端展示对人友好的中文，后端拿到的是稳定的、与数据库主键或枚举对齐的短码。改个显示文案不影响后端逻辑，做国际化时换文字也不动 value。

陷阱：忘了给 option 设 value。如果 option 没写 value 属性，浏览器会退而用标签内的文本当值提交。 于是后端收到的是 city=北京 这样的中文——一旦你改了显示文案、或要做多语言，后端匹配逻辑就全崩了。规范做法：每个 option 都显式写 value。

selected：默认选中项

给某个 option 加 selected，页面打开时它就是默认选项。不写的话，单选 select 默认选中第一个 option—— 这常导致"用户没动它，却莫名其妙提交了第一项"。常见做法是放一个引导性的占位项：

占位项：强制用户主动选择
<select name="city" required>
  <!-- value 为空 + disabled + selected：显示提示，但本身不可被选作答案 -->
  <option value="" disabled selected>请选择城市</option>
  <option value="bj">北京</option>
</select>

optgroup：把选项分组

选项很多时，用 <optgroup> 按类别分组，下拉里会出现带标题的分隔块，更易浏览。注意 optgroup 的 label 只是分组标题，不能被选中，也不会提交。

optgroup 分组
<select name="city">
  <optgroup label="华北">
    <option value="bj">北京</option>
    <option value="tj">天津</option>
  </optgroup>
  <optgroup label="华东">
    <option value="sh">上海</option>
  </optgroup>
</select>

multiple：多选下拉

给 select 加 multiple，它会变成一个可滚动的多选列表框（按住 Ctrl/Cmd 或 Shift 多选）。 多选时取值不再是单个字符串，而是一组被选中 option 的 value，这给提交带来了新坑。

多选 select 的 name 约定
<!-- 多个值需要后端能接住一个数组 -->
<select name="tags[]" multiple>
  <option value="java">Java</option>
  <option value="go">Go</option>
  <option value="rust">Rust</option>
</select>
陷阱：多选 select 的提交需要特殊处理。选了 Java 和 Go，浏览器会发出两条同名字段 tags=java&tags=go。 部分后端框架默认只取第一个，导致明明多选了，后端却只收到一个值。两种解法：① 像 PHP/某些框架那样把 name 写成 tags[]，框架会自动聚合成数组； ② 后端按"同名多值"的方式读取（如 Spring 用 List<String> 或 String[] 接收）。这是前后端必须对齐的约定。

四、三个标签的取值方式对比

控件当前值从哪里来默认值怎么设
input value 属性 value="默认"
textarea 标签的内容（开闭标签之间的文本） <textarea>默认</textarea>
select 被选中 option 的 value（不是显示文本） 给某个 option 加 selected
把这张表记牢，就避开了本节最高频的三个坑：textarea 别用 value、select 取的是 option 的 value、option 一定要显式写 value。

五、常见错误速查

错误后果正确做法
label 没有 for 也没包裹控件 点文字光标不跳；读屏软件念不出字段名，无障碍失效 用 for=id 关联，或把控件包进 label
for 的值和控件 id 对不上 关联无效，等于没写 确保 for 与 id 完全一致
textarea 用 value 属性设默认值 属性被忽略，框里空白 把默认文本写在开闭标签之间
option 不写 value 提交的是显示文本（如中文），改文案就崩 每个 option 都显式写 value
单选 select 没设默认/占位项 默认选中第一项，用户没动也会提交它 放 disabled selected 的空占位项
多选 select 后端只收到一个值 多选数据丢失 name 用 [] 或后端按多值/数组接收

六、与 Java 的类比

HTML 概念Java / 后端类比
label 关联控件（文字↔字段） 字段上的元信息注解，如 @Schema(description=...) 或 i18n 资源里的字段显示名——把"给人看的文案"和"程序里的字段"绑定起来
select + option 下拉 枚举类型 enum 做下拉：option 的 value ≈ enum.name()，显示文本 ≈ 枚举的 getLabel()
option 的 value vs 显示文本分离 枚举的常量名（稳定、入库）与展示名（可随 i18n 变）分离——后端只认 value，文案随便改
optgroup 分组 枚举按类别归类，如订单状态分"进行中 / 已完成 / 已取消"几个大类
textarea 多行大文本 数据库的 TEXT / CLOB 大字段，对应实体里的长 String（区别于定长的 VARCHAR）
multiple select 提交多值 后端用 List<String> / String[] 接收一组同名参数，对应数据库的多对多关联
一条贯穿始终的思想：把"展示"和"数据"解耦。label 让文案独立于控件、option 的 value 让实际值独立于显示文本—— 这和后端用枚举常量名入库、用 i18n 资源做展示，是同一个设计哲学：稳定的标识符给机器，可变的文案给人。
