合并单元格：colspan / rowspan

表格本质是一张严格的网格。合并单元格不是"把格子变大"，而是"让一个格子吞掉旁边的格子"——被吞的位置必须把 td 删掉。这一删一占，是写复杂表格唯一容易翻车的地方。

一、先建立网格思维

在动手合并之前，必须先把表格在脑子里想象成一张规则的二维网格。 一个 3 行 × 3 列 的表格，就是 9 个格子，每个格子都有自己的"坐标" (行, 列)。普通情况下，每个 <td> 严格对应一个格子：

普通表格：每个 td 占一个格子，行列数严格对齐
(0,0)
(0,1)
(0,2)
(1,0)
(1,1)
(1,2)
(2,0)
(2,1)
(2,2)
核心心智模型：合并单元格 = 让某个格子占据多个网格位置。 它向右占叫 colspan（跨列），向下占叫 rowspan（跨行）。 一旦某个格子被别人占了，那个位置上原本的 <td> 就必须删掉—— 因为那个网格坐标已经"有人了"。

二、colspan：横向合并 N 列

在某个 <td> 或 <th> 上写 colspan="2"， 意思是"这个单元格横向占 2 列宽"。它会把自己右边那个格子吃掉，所以这一行要少写一个 td。

colspan：第一行合并前两列
<table>
  <tr>
    <!-- 占 2 列，所以本行只写 2 个 td 就够 3 列宽 -->
    <td colspan="2">A（跨 2 列）</td>
    <td>B</td>
  </tr>
  <tr>
    <!-- 普通行：老老实实写满 3 个 td -->
    <td>C</td>
    <td>D</td>
    <td>E</td>
  </tr>
</table>

实际渲染效果（橙色格子是被合并的）：

colspan="2" 让 A 横跨两列
A（跨 2 列） B
C D E
第一行只有 2 个 td，却撑满了 3 列——因为 A 自己算 2 列。 如果你这里手滑写了 3 个 td（A 带 colspan="2" + B + 多写一个）， 第一行就会变成 4 列宽，表格立刻错位。合并几列，本行就要少写几个 td。

三、rowspan：纵向合并 N 行

rowspan="2" 表示"这个单元格纵向占 2 行高"。它会向下吃掉下一行对应列的格子， 所以下一行要少写那一个 td。这是比 colspan 更容易出错的地方—— 因为要删的 td 不在当前行，而在下面的行里。

rowspan：第一列合并前两行
<table>
  <tr>
    <!-- 这个格子向下占 2 行 -->
    <td rowspan="2">A（跨 2 行）</td>
    <td>B</td>
  </tr>
  <tr>
    <!-- 第一列被 A 占了，本行不写第一列的 td -->
    <td>C</td>
  </tr>
</table>

实际渲染效果：

rowspan="2" 让 A 纵跨两行
A（跨 2 行） B
C
第二行只写了一个 td（C），却正好对齐到第二列—— 因为第一列的位置被上一行的 A 用 rowspan 占走了。 新手最常见的错误：在第二行又写了第一列的 td，结果那一行被挤成了 3 列，A 旁边凭空多出一格。

四、核心坑：合并后必须删除被占的 td

这是整节最重要的一点，单独拎出来讲。浏览器渲染表格的逻辑是： 它按你写的 td 顺序，从左到右、从上到下往网格里"填空"， 遇到已经被 colspan/rowspan 占据的坐标就跳过。

所以你心里要时刻维护一张"哪些坐标已被占"的地图。 被占的坐标对应的 td 绝对不能再写，否则它会被挤到下一个空位，导致整行多出一格、表格右侧"长出来"。

rowspan="2" + colspan="2" 时的网格占位（橙=合并源，红=被吃掉、不能写 td）
A 占
A 占
(0,2)
被 A 吃
被 A 吃
(1,2)
(2,0)
(2,1)
(2,2)
A 同时跨 2 行 2 列：注意第二行少写了 2 个 td
<tr>
  <td rowspan="2" colspan="2">A</td>  <!-- 占 (0,0)(0,1)(1,0)(1,1) 四格 -->
  <td>B</td>                  <!-- (0,2) -->
</tr>
<tr>
  <!-- (1,0)(1,1) 已被 A 吃掉，本行只剩 (1,2) 一个空位 -->

六、陷阱速查

陷阱后果正确做法
合并后没删被占的 td 表格右侧"长出"多余列，整体错位 合并几格，对应行就少写几个 td
rowspan 时在下一行重复写了那列 td 下一行被挤偏，对不齐 被纵向占据的行，跳过那一列不写 td
colspan / rowspan 同时用时数错格子 占位计算混乱，行列总数对不上 在纸上/脑中画网格，逐格标记"已占"
用合并硬凑复杂页面布局 结构僵硬、难维护、响应式失效 布局用 flex / grid，table 只放真正的二维数据
过度合并破坏无障碍 屏幕阅读器读不清行列对应关系 合并克制；配合 scope / headers 标注表头关系
怎么自查有没有删错 td？ 最快的办法：数一数每一行"实际占用的列数"（每个 td 的 colspan 相加， 再加上从上面 rowspan 落下来的列数），每一行的总列数必须完全相等。 不相等，就是哪里多写或少写了 td。

七、与 Java 的类比

表格 / 合并概念Java 类比
网格占位思维（每格有坐标） 二维数组 int[row][col] 的索引映射——每个 td 对应一组确定的下标
colspan / rowspan 跨格聚合 SQL 的 GROUP BY 把多行聚成一行；类似把多个网格坐标归到一个值
删除被占据的 td 往二维数组写值时避免索引重复——一个坐标只能被赋值一次，否则覆盖/越界
每行列数必须相等 矩阵的"行列对齐"约束；越界或维度不匹配直接 ArrayIndexOutOfBoundsException
浏览器按顺序填空、跳过已占坐标 遍历 boolean[][] occupied 标记位，遇到 true 就跳过，找下一个空位写入
本质上，写合并表格就是在脑子里跑一个二维填充算法： 维护一张"已占用坐标"的地图，按行列顺序填 td，遇到被占的坐标跳过。 把它当成"往二维数组里安全写值、不让索引冲突"来理解，那些错位 bug 就都说得通了。

与其他知识点的联系

  • 表格语义分组（上一节）：合并单元格常常发生在 thead 内（如多级表头），也会跨越 tbody 里的多行——本文陷阱速查提到的"用 scope/headers 标注表头关系"，正是因为合并之后原本"第 N 列对应第 N 个 th"的简单对应关系被打破了，必须显式声明，才能让上一节讲的读屏"列名：值"播报机制继续正常工作。
  • 表格基础结构：colspan/rowspan 本质上是在更早的"表格基础结构"章节建立的 td/tr 二维网格模型上做的扩展操作——没有先建立"每个 td 对应一个网格坐标"的心智模型，合并单元格的规则会显得毫无道理，这也是本文第一节要专门先讲网格思维的原因。
  • CSS Grid/Flex 布局：本文陷阱速查明确提醒"布局用 flex/grid，table 只放真正的二维数据"——这是因为早期 Web 开发滥用表格布局（table-based layout）做页面排版是反模式，CSS Grid 的 grid-column/grid-row 概念和 colspan/rowspan 的"跨列跨行"思路相似，但语义完全不同：前者管布局，后者管数据关系。
  • 可访问性（accessibility）：合并单元格是表格无障碍支持里最脆弱的一环，本文提到的 scope/headers 属性是专门为解决"合并后行列对应关系模糊"设计的机制，过度合并、又不补充这些属性，是表格无障碍审计里最常见的失分点。
  • React 中的表格渲染：在 React 里用 .map() 动态渲染表格行时，如果数据源里有需要合并单元格的场景（比如多级分类的报表），必须在渲染逻辑里手动计算"这一行该跳过哪几个位置的 td"，这正是本文"维护一张已占用坐标的地图"思路在组件化开发中的直接体现，处理不好一样会出现行列错位的 bug。
