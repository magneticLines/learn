HTML · 表单

fieldset / datalist / output：三个被低估的表单增强元素

它们不是表单的"主角"（input / select 才是），但各自解决了一个真实的痛点：把控件分组、给输入框做自动补全、把计算结果展示出来。会用它们，能少写一堆 div 和 JS。

一、fieldset：把相关控件"圈"成一组

一个稍微复杂的表单（注册、下单）里，字段往往分属不同主题：一组是"账号信息"，一组是"收货地址"。 如果全平铺在一起，用户和屏幕阅读器都难以理解结构。<fieldset> 就是用来 在语义层面把相关控件圈成一组的容器，<legend> 则是这一组的标题。

fieldset + legend 分组
<fieldset>
  <legend>收货地址</legend>  <!-- 必须是第一个子元素 -->

  <label>收件人 <input name="name"></label>
  <label>详细地址 <input name="addr"></label>
</fieldset>

浏览器默认会给 fieldset 画一个边框，legend 嵌在边框上方——这就是你常见的 "带标题的分组框"。它的价值不只是好看：屏幕阅读器读到组内某个 input 时，会一并念出 legend 的文字，让"看不见"的用户也知道"我正在填收货地址那一组"。

杀手锏：disabled 一键禁用整组

给 <fieldset> 加上 disabled，组内所有控件会全部变灰、不可交互， 不需要你给每个 input 都挂一遍 disabled。配合 JS 切换，可以做"未勾选同意条款前， 整组表单不可填"这类效果。

一行禁用整组
<!-- 加 disabled，里面所有 input/select/button 全部禁用 -->
<fieldset disabled>
  <legend>发票信息（暂不可用）</legend>
  <input name="taxNo">
</fieldset>
陷阱 ①：被 disabled 的 fieldset 内部控件，它们的值 不会随表单提交（这是 disabled 控件的统一规则，不只是 fieldset）。 如果你只是想"不让用户改、但仍要提交值"，应该用 readonly（注意 readonly 只对文本类 input 有效， fieldset 没有 readonly 属性）。
陷阱 ②：<legend> 必须是 fieldset 的第一个子元素。 放到中间或末尾，浏览器不会把它当作分组标题渲染（边框上方那个标题会消失），可访问性关联也会失效。

二、datalist：给 input 加自动补全候选

<datalist> 解决的问题是：我想给用户一些候选项作为提示，但又不想限制他只能选这几个。 它本身不显示任何东西，而是通过 input 的 list 属性，按 id 关联到某个 datalist，浏览器就会在用户聚焦/输入时弹出候选下拉。

input list 关联 datalist
<!-- input 用 list 指向 datalist 的 id -->
<input name="city" list="cityOptions" placeholder="输入城市">

<datalist id="cityOptions">
  <option value="北京">
  <option value="上海">
  <option value="广州">
</datalist>

注意 datalist 里的 <option> 是自闭合写法（值放在 value 属性里），跟 select 里 <option>文本</option> 的写法不一样。

datalist vs select：到底差在哪

这是最容易混淆的点。两者都给"一组候选项"，但本质完全不同：select 是 "封闭选择"——用户只能从给定项里挑一个，不能输入别的；datalist 是 "开放建议"——候选项只是提示，用户完全可以无视它们，自己敲任意内容。

维度selectdatalist
能否自由输入不能，只能选能，候选只是建议
取值约束值一定在选项集合内值可以是任意字符串
典型场景性别、省份、订单状态等枚举搜索框历史、城市/标签提示等开放输入
本质载体独立控件，自带交互给某个 input 挂的补全数据源
是否需要选中有默认值/必选不强制使用任何候选
陷阱 ③：别把 datalist 当 select 用。它不强制用户从候选里选—— 用户随便敲个不在列表里的值也会被接受并提交。如果你的字段必须是固定枚举（比如订单状态只能是几个值）， 用 datalist 后端就可能收到非法值。这种需要"约束取值"的场景请用 select， 或者在 datalist 之外再加服务端校验。

三、output：把计算结果展示出来

<output> 是一个语义化的结果展示元素，专门用来显示 "由表单内其他控件计算出来的结果"——比如根据数量和单价算出的总价、根据 range 滑块当前值显示的百分比。 它的 for 属性可以列出"参与这次计算的控件 id"（空格分隔），表达一种语义关联， 方便屏幕阅读器理解"这个结果是从哪些输入算出来的"。

range + output（for 关联参与计算的控件）
<input type="range" id="vol" min="0" max="100" value="50"
       oninput="out.value = this.value">

<!-- for 指向参与计算的控件 id；它本身不会自动算 -->
<output id="out" for="vol">50</output>
陷阱 ④：output 不会自动计算任何东西。 for 只是"声明关联"的语义标记，浏览器不会因为你写了 for="vol" 就自动把 vol 的值填进来。 真正的赋值仍然要靠 JS（监听 input/change 事件后写 output.value = ...）。 把它想成"一个有语义的 span"更准确——区别是它属于表单元素、可被 form.elements 访问。
总价实时计算：JS 负责算，output 负责展示
<input type="number" id="qty" value="1"> × ¥
<input type="number" id="price" value="99">
= ¥<output id="total" for="qty price"></output>

<script>
  const calc = () =>
    total.value = (qty.value * price.value);
  qty.oninput = price.oninput = calc; calc();
</script>

四、陷阱速查

误区真相 / 正确做法
fieldset disabled 后值还会提交 不会。被禁用的控件值不进表单提交；想保留值用 readonly 思路或隐藏字段
legend 放哪都行 必须是 fieldset 的第一个子元素，否则不渲染为分组标题
把 datalist 当 select，以为能限制取值 不强制选择，用户可输任意值。要约束枚举用 select + 服务端校验
datalist 的 option 写成 <option>文本</option> 用 <option value="..."> 自闭合写法，值放在 value 属性
output 写了 for 就会自动算 不会算，for 只是语义关联；赋值仍靠 JS 写 output.value

五、与 Java 的类比

这三个元素的"职责分工"，在后端建模里都有对应的影子。用熟悉的概念去类比，更容易记住它们各自的定位：

HTML 元素 / 概念Java / 后端类比
fieldset（把相关字段分组） DTO 里的嵌套对象（如 OrderDTO 内含 AddressDTO），或 Bean Validation 的分组校验 Group——把一组字段当一个逻辑单元处理
fieldset disabled（一键禁用整组） 对一个嵌套对象整体跳过处理 / 不参与序列化（如某分组校验组不触发、字段标 @JsonIgnore）
legend（分组标题） 分组对象上的语义标签 / 注释，告诉读者"这一坨是什么"，类似分组 Group 接口的命名
datalist（开放式补全建议） 一个自动补全 / suggest 接口（如 /api/cities/suggest?q=北）——返回的是建议项，不约束最终取值，校验仍在服务端兜底
select（封闭枚举选择） 后端的 enum——取值被限定在固定集合内，传非法值直接报错
output（展示派生结果） 实体里的计算属性 / 派生字段（如 getTotal() { return qty * price; }）——值由其它字段算出，不独立存储
一句话记忆：fieldset 管"结构分组"，datalist 管"开放建议"， output 管"派生展示"。它们都是用更精确的语义，替代你本来要堆的 div + class + JS—— 本质上和后端"用类型和注解表达意图"是同一种追求。
