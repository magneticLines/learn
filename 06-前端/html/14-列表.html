列表：ul / ol / dl

列表不只是"前面带个圆点"。选 ul、ol 还是 dl，本质是在向浏览器和屏幕阅读器声明"这组数据之间是什么关系"——这正是语义化 HTML 的核心。

一、三种列表，三种语义

HTML 提供三种列表，区别不在外观，而在"这组项之间是什么关系"：

标签含义顺序重要吗典型场景
<ul>无序列表（unordered）不重要功能点、标签、导航菜单
<ol>有序列表（ordered）重要操作步骤、排行榜、目录
<dl>定义列表（description）成对术语-解释、元数据、FAQ
判断依据是"顺序有没有意义"：如果把项打乱顺序后意思不变（如一袋水果），用 ul； 如果打乱就错了（如"先打鸡蛋，再下锅"），用 ol。这和"是否带数字编号"无关—— 编号是 CSS 决定的外观，语义才是选标签的依据。

二、ul：无序列表

ul + li
<ul>
  <li>支持深色模式</li>
  <li>离线可用</li>
  <li>多语言切换</li>
</ul>
浏览器渲染效果
  • 支持深色模式
  • 离线可用
  • 多语言切换

这三个功能谁先谁后都无所谓，所以用 ul。默认每项前面是个实心圆点（disc）， 但这只是浏览器的默认样式，不要因为"想要圆点"才用 ul，也不要因为"不想要圆点"就不用。

三、ol：有序列表

当顺序本身携带信息时（步骤、排名、流程），必须用 ol。它默认带数字编号。

ol + li
<ol>
  <li>克隆仓库</li>
  <li>安装依赖</li>
  <li>启动服务</li>
</ol>
浏览器渲染效果
  1. 克隆仓库
  2. 安装依赖
  3. 启动服务

ol 的常用属性

属性作用示例
start指定起始序号<ol start="5"> 从 5 开始
reversed倒序编号<ol reversed> 3、2、1
type编号样式1 / A / a / I / i
属性组合 + li 的 value
<!-- 从 5 开始、大写罗马数字 -->
<ol start="5" type="I">
  <li>第五章</li>          <!-- V -->
  <li value="10">第十章</li> <!-- 跳到 X -->
  <li>第十一章</li>        <!-- XI，接着 value 往后算 -->
</ol>
浏览器渲染效果
  1. 第五章
  2. 第十章
  3. 第十一章
start / reversed / value 是语义属性，不是样式：它们改变的是"这个项实际代表第几"， 会被屏幕阅读器念出来，也影响打乱重排时的正确性。而 type 偏外观—— 现代写法更推荐用 CSS 的 list-style-type 控制编号样式，把 HTML 留给语义。

四、dl：定义列表

dl 用于"成对"的内容：<dt>（term，术语/名称）+ <dd>（description，描述）。 它是唯一能表达"键-值关系"的列表。

dl + dt + dd
<dl>
  <dt>HTML</dt>
  <dd>超文本标记语言，描述网页结构</dd>

  <dt>CSS</dt>
  <dd>层叠样式表，控制网页外观</dd>
</dl>
浏览器渲染效果
HTML
超文本标记语言，描述网页结构
CSS
层叠样式表，控制网页外观
dt 和 dd 可以多对多：一个 dt 后可跟多个 dd（一个术语多条解释）， 多个 dt 也能共享一个 dd（多个同义词指向同一解释）。 所以 dl 不是简单的"一行一对"，而是表达"一组名称与一组描述的关联"。

五、结构规则：谁能放在谁里面

列表的父子结构是有硬性规定的，浏览器和校验器都会检查：

容器直接子元素只能是
<ul> / <ol><li>（以及 script/template）
<dl><dt> / <dd>（可用 div 包裹成对）
<li> / <dd>任意流内容（段落、图片、甚至嵌套列表）
✅ 正确 vs ❌ 错误
<!-- ❌ 错误：ul 里直接放非 li 元素 -->
<ul>
  <div>这不合法</div>
  <p>这也不合法</p>
</ul>

<!-- ✅ 正确：内容包进 li，li 里想放什么都行 -->
<ul>
  <li><div>合法</div></li>
  <li><p>段落也行</p></li>
</ul>
为什么这条规则严格？无障碍技术（屏幕阅读器）依赖这个结构来报"列表共 3 项，当前第 1 项"。 如果 ul 里混进了游离的 div，列表的项数统计就乱了，盲人用户拿到的信息是错的。

六、列表可以嵌套

嵌套列表用来表达层级关系（目录、多级菜单、组织架构）。 关键：子列表要放在父级 li 内部，而不是两个 li 之间。

嵌套：子 ul 在父 li 内
<ul>
  <li>前端
    <ul>
      <li>HTML</li>
      <li>CSS</li>
    </ul>
  </li>
  <li>后端</li>
</ul>
浏览器渲染效果
  • 前端
    • HTML
    • CSS
  • 后端

七、行业惯例：导航菜单用 nav > ul > li > a

几乎所有网站的导航栏都是这个结构。看起来是一排横向链接，HTML 却用 ul 包起来—— 因为导航在语义上就是"一组并列的链接"。

标准导航结构
<nav>
  <ul>
    <li><a href="/">首页</a></li>
    <li><a href="/docs">文档</a></li>
    <li><a href="/about">关于</a></li>
  </ul>
</nav>
为什么不直接一排 <a> 就完了？用 ul 包起来后，屏幕阅读器会播报 "导航，列表，共 3 项"，让视障用户知道这里有几个入口、自己在第几个。横向排列是 CSS 的事 （display:flex 或 list-style:none 去掉圆点），语义和样式分离——这是行业默认做法。

八、列表的默认样式与 list-style

"圆点""数字"这些项目符号是浏览器默认 CSS 给的，不是 HTML 标签自带的能力。 控制它们的属性是 list-style（及其子属性）：

CSS 属性作用
list-style-type符号样式：disc / circle / decimal / none …
list-style-position符号在内还是外（inside / outside）
list-style-image用图片当符号
去掉导航的圆点（最常见用法）
/* 导航 ul 通常要去掉默认圆点和缩进 */
nav ul {
  list-style: none;
  margin: 0; padding: 0;
}
记住这条分工："是不是列表"由 HTML 标签决定（语义），"长什么样"由 CSS 决定（外观）。 去掉圆点不会让它"不是列表"——它在无障碍树里依然是列表，屏幕阅读器照样播报项数。

九、常见陷阱速查

陷阱问题正确做法
用 ul 只为了让内容缩进 语义错误，无障碍树里凭空多出"列表 N 项" 缩进是排版，用 CSS margin / padding
ul / ol 里直接放 div、p 非法结构，项数统计错乱 所有内容必须包在 li 里
用 dl 做不相关的两列布局 dt/dd 没有"术语-解释"关系时是语义滥用 纯排版用 grid/flex；表格数据用 table
有顺序的步骤用了 ul 丢失"先后"语义，读者/AT 不知是流程 步骤、排名一律用 ol
嵌套时把子 ul 放在两个 li 之间 非法，层级关系丢失 子列表放进父 li 内部

十、与 Java 的类比

后端工程师可以把三种列表对应到熟悉的集合类型——选哪个的逻辑和"选哪个集合"几乎一样：先问"这组数据的关系是什么"。

HTML 列表Java 集合类比共同点
<ul> 无序列表 Set（顺序不重要）/ 不依赖索引遍历的集合 一组同级元素，顺序不携带语义
<ol> 有序列表 有序的 List / LinkedList 位置（索引）本身就是信息，乱序即出错
<dl> 定义列表 Map<K, V>（键值对） dt 是 key，dd 是 value，表达关联
嵌套列表 树形结构 / List<List<T>> / DOM 树本身 用父子层级表达从属关系
li 只能进 ul/ol 泛型约束 List<Item> 只能装 Item 容器对子元素类型有强制约束
本质上，选 ul / ol / dl 和在 Java 里选 List / LinkedList / Map 是同一种思维：先看清数据之间的关系（并列？有序？键值？层级？），再选能准确表达这种关系的结构。 区别只是后端选错了顶多性能差点，前端选错了会让屏幕阅读器给出错误信息——语义化的代价更直接。
