HTML · 可访问性

ARIA 基础：role / aria-label / aria-hidden

ARIA 是给屏幕阅读器等辅助技术"补话"用的属性集。但它有一条铁律：能用原生语义 HTML 就别碰 ARIA。用错了，比不用更糟。

一、ARIA 是什么：给辅助技术补语义

ARIA（Accessible Rich Internet Applications）是一套 HTML 属性，作用是 向辅助技术（屏幕阅读器、语音控制等）补充语义信息。 它不改变页面外观、也不改变任何行为，只影响"无障碍树（Accessibility Tree）"—— 那是浏览器在 DOM 之外额外维护的、专门喂给辅助技术的一棵树。

打个比方：一个明眼用户看到一个 🔍 图标就知道"这是搜索按钮"，因为图标本身传递了视觉语义。 但屏幕阅读器读不懂图标，它只能读文本和语义标签。ARIA 就是补上这层"用眼睛能看出来、但代码里没写出来"的信息。

ARIA 影响的是无障碍树，不是视觉
// 浏览器维护两套结构：
DOM 树            → 渲染成像素，给眼睛看
Accessibility 树  → 抽取语义，给屏幕阅读器读

// ARIA 属性只往 Accessibility 树里写信息，
// CSS 改的是 DOM 渲染，两者互不干扰。

二、ARIA 第一规则：能用原生 HTML 就别用 ARIA

这是整篇最重要的一条，请刻进脑子里。 W3C 官方《ARIA 使用规范》第一条就是： "如果一个原生 HTML 元素已经具备你想要的语义和行为，那就用它，不要换成另一个元素再用 ARIA 把语义补回来。" ARIA 是补充，不是替代。

最经典的反例就是用 <div> 模拟按钮：

反例 → 用 div + role 硬凑一个按钮
<!-- ❌ 表面上"无障碍"，实际处处是坑 -->
<div role="button" tabindex="0" onclick="submit()">提交</div>

这个 div 为了"像个按钮"，你得手动补齐一堆原生 <button> 免费就有的东西：

原生 <button> 自带div role="button" 要手动补
可被键盘 Tab 聚焦得加 tabindex="0"
Enter / Space 触发点击得自己监听 keydown 判断按键
被屏幕阅读器读作"按钮"得加 role="button"
disabled 状态、聚焦样式得自己写 aria-disabled + CSS
表单内自动提交没有，得自己实现
正例 → 一行原生标签全搞定
<!-- ✅ 键盘、焦点、语义、状态全部免费 -->
<button onclick="submit()">提交</button>
记忆口诀：<button> 优于 <div role="button">， <nav> 优于 <div role="navigation">， <a href> 优于 <span role="link">。 每当你准备写 role= 时，先问自己一句：有没有现成的原生标签？

三、role：声明元素的角色

role 告诉辅助技术"这个元素扮演什么角色"。它主要用在 没有原生语义的元素（如 div / span）上， 或者用来表达 HTML 没有对应标签的复杂控件（如 tablist、dialog、alert）。

role 的合理使用场景
<!-- HTML 没有原生 tab 标签，这里 role 是必要的 -->
<div role="tablist">
  <button role="tab" aria-selected="true">概览</button>
  <button role="tab" aria-selected="false">详情</button>
</div>
陷阱：role 不要和原生语义冲突。 像 <ul role="navigation"> 会让阅读器把列表语义换掉， <h2 role="button"> 更是直接抹掉了"标题"语义。 给一个元素加 role，等于覆盖它原本的语义——别给本来就有正确语义的标签乱加 role。

四、aria-label：给没有可见文本的元素起名

很多控件没有可见文字，比如只有一个图标的按钮。明眼人看图标就懂， 但屏幕阅读器读不出来——这时用 aria-label 提供一个"无障碍名称"。

图标按钮：必须给个名字
<!-- ❌ 阅读器只能读出"按钮"，用户不知道点了干嘛 -->
<button>🔍</button>

<!-- ✅ 阅读器读出"搜索，按钮" -->
<button aria-label="搜索">🔍</button>
有可见文本时不要再加 aria-label。 aria-label 会覆盖元素内的可见文字。 给 <button aria-label="提交">保存</button> 加错标签， 屏幕阅读器读"提交"、屏幕显示"保存"，语音控制用户说"点保存"反而点不动——可见文本和无障碍名称必须一致。

aria-labelledby / aria-describedby：引用别的元素

如果名称或描述已经写在页面别处，不必重复文本，直接用 id 引用： aria-labelledby 指定"标签"，aria-describedby 指定"补充描述"。

引用现成文本作为标签和描述
<h2 id="billing">账单地址</h2>

<input aria-labelledby="billing"
       aria-describedby="hint">
<p id="hint">请填写发票邮寄地址</p>
优先级要点：aria-labelledby > aria-label > 元素自身文本/ <label>。能用原生 <label for> 关联表单控件时，优先用它——又是第一规则。

五、aria-hidden="true"：对辅助技术隐藏

aria-hidden="true" 把元素从无障碍树里摘掉——屏幕阅读器会完全跳过它， 但视觉上仍然可见。专门用于纯装饰、对理解内容没帮助的东西， 比如装饰性图标、重复出现的视觉符号。

装饰性图标：藏起来避免噪音
<!-- 文字已表达含义，图标只是装饰，读出来反而啰嗦 -->
<button>
  <span aria-hidden="true">➕</span> 新增联系人
</button>
最危险的陷阱：千万别在可交互元素上加 aria-hidden。 如果你给一个能 Tab 聚焦的元素（按钮、链接、输入框）加了 aria-hidden="true"， 会出现一种诡异的"幽灵焦点"：键盘用户能 Tab 到它，屏幕阅读器却读不出任何内容—— 用户彻底懵了。要彻底移除，应同时去掉可聚焦性（如 tabindex="-1" 或 disabled）。

六、常用状态属性速览

ARIA 还有一类"状态属性"，用来表达会动态变化的状态。关键是： 加了状态属性，就必须在交互时用 JS 同步更新它，否则就是在撒谎。

属性含义典型场景
aria-expanded展开 / 折叠状态下拉菜单、折叠面板的触发按钮
aria-current当前项导航里高亮的当前页 aria-current="page"
aria-selected是否被选中tab 标签、列表选项
aria-disabled禁用（仍可聚焦）非 button 元素的禁用态
aria-live动态区域，内容变化时朗读表单错误提示、消息通知 toast
aria-expanded 必须随交互同步
<button aria-expanded="false" aria-controls="menu">菜单</button>

// 点击切换面板时，别忘了同步状态：
btn.setAttribute('aria-expanded', 'true')
陷阱：加了 role 却不补配套。 比如声明了 role="tab" 却不处理 aria-selected 切换、不实现左右箭头键切换 tab， 阅读器用户会被告知"这是 tab"，结果一操作发现啥都不响应。 用 ARIA 模拟一个控件，就要把它的状态管理 + 键盘交互全部补齐，做半套等于挖坑。

七、滥用比不用更糟：常见错误速查

错误做法后果正确做法
用 div role="button" 替代 <button> 键盘 / 焦点 / 状态全要手补，漏一个就有障碍 直接用原生 <button>
给可聚焦元素加 aria-hidden="true" 键盘能 Tab 到，阅读器读不到（幽灵焦点） 同时移除可聚焦性，或干脆不藏
有可见文本还加 aria-label 覆盖可见文字，语音控制点不动 无可见文本时才用 aria-label
给已有语义的标签乱加 role 覆盖掉原生语义，误导用户 只在 div/span 或自定义控件上用 role
加了 role / 状态属性却不更新 状态长期不准，等于向用户撒谎 交互时用 JS 同步状态 + 补键盘操作
核心心智：ARIA 不是"加了就更无障碍"的勋章。 一个错误的 role 比没有 role 危害更大——没有时阅读器至少按原生语义如实播报， 错误时它会用错误信息误导用户。所以"少即是多"，第一规则永远是优先原生。

八、与 Java 的类比

ARIA 概念Java 类比
ARIA 补充语义信息（不改行为） 泛型 / 注解给编译器和框架补充类型与元信息，不改运行时核心逻辑
第一规则：优先原生 HTML 优先用标准库（如 List、Optional），别自己造轮子再补一堆约定
role 声明元素角色 接口 / 契约声明——告诉调用方"我扮演这个角色，按这套约定来用我"
加了 role 必须补全配套状态与交互 implements 一个接口就必须实现全部方法，否则编译不过 / 运行抛异常
滥用 ARIA / 错误的 role 反而误导 过度配置、错误注解（如乱标 @Transactional）让框架行为出乎意料，比不写更难排查
本质上 ARIA 和 Java 注解是同一种思路：在不改变主体的前提下，附加一层供"外部读者"消费的元信息。 Java 注解的读者是编译器、Spring、JPA；ARIA 的读者是浏览器无障碍树和屏幕阅读器。 两者都遵循同一条工程美德——能让平台/语言原生表达的语义，就别用附加层去模拟。
