HTML · 链接与媒体

图片 <img>

一个看起来只有两个属性的标签，却牵扯到无障碍、SEO、布局稳定性、首屏性能。写好 <img>，是前端性能优化和可访问性的第一道门槛，也是面试高频区。

一、最小可用：src 和 alt

<img> 是空元素（void element，没有闭合标签、没有内容），靠属性工作。 它真正的"必填项"有两个：src（图片地址）和 alt（替代文本）。 很多人以为 alt 可选——规范上它必须存在，只是某些场景值可以是空字符串。

最小写法
<img src="/avatar.png" alt="张三的头像">

src 指向资源，浏览器发起一次 GET 请求把它下载回来渲染。 alt 是当图片"不在场"时的文本替身——它不是注释，而是图片的语义内容。 下面单独讲透 alt，因为它是这一节最容易被忽视、又最容易被考的点。

二、alt：替代文本不是可有可无的

把 alt 理解成"图片用文字怎么说"。它在三种时刻发挥作用，对应三类受众：

时刻受众 / 用途意义
图片加载失败（网络断、路径错、被拦截） 普通用户 浏览器在原位显示 alt 文本，用户仍知道这里"本该有什么"
用户用屏幕阅读器（视障人士） 无障碍 / a11y 读屏软件念出 alt，图片对盲人才"可被感知"
搜索引擎爬取索引 SEO 爬虫看不懂像素，靠 alt 理解图片内容，影响图片搜索排名

三种场景，三种写法

关键认知：alt 该写什么，取决于图片在页面里扮演什么角色，而不是图片画了什么。 同一张图，做内容图和做装饰图，alt 写法完全不同。

按角色决定 alt
<!-- ① 内容图：描述"内容"，传达信息本身 -->
<img src="chart.png"
     alt="2025 年营收增长 32%，Q4 达到峰值">

<!-- ② 功能图：图是按钮/链接，描述"功能"而非外观 -->
<a href="/cart">
  <img src="cart-icon.svg" alt="查看购物车">
</a>

<!-- ③ 装饰图：纯装饰、无信息量，用空 alt -->
<img src="divider.png" alt="">
功能图的陷阱：如果 <img> 被包在链接/按钮里，alt 要描述点了会发生什么（"查看购物车"）， 而不是图标长什么样（"一个购物车图标"）。读屏用户关心的是动作，不是图案。

为什么装饰图要 alt="" 而不是直接不写

这是最反直觉、也最能区分新手老手的点：

写法屏幕阅读器行为结果
alt=""（空字符串） 明确知道"这图无信息"，直接跳过 ✅ 干净，不打扰用户
完全不写 alt 属性 不知道是疏忽还是无意义，可能念出文件名（如 "divider underscore 2 dot png"） ❌ 噪音，体验糟糕
记住这个区分：alt="" 是一句主动声明——"我确认这张图没有信息，请忽略它"。 而省略 alt 是沉默，机器无法判断你是故意的还是忘了，只能猜，往往猜错。 后面"与 Java 的类比"会把它对应到"主动声明无内容防御性编程"。

三、width / height：不是为了缩放，是为了防抖动

很多人不写 width/height，觉得"反正 CSS 会控制大小"。 但这两个属性最重要的作用不是设尺寸，是提前告诉浏览器图片的宽高比，预留出空间。 不写它，就会引发 CLS。

什么是 CLS（Cumulative Layout Shift）

浏览器解析 HTML 是流式的。遇到 <img> 时，图片还没下载完， 浏览器不知道它多高，只能先按高度 0 排版，把后面的文字往上挤。 等图片下载完拿到真实尺寸，再"撑开"占位——后面所有内容突然往下跳一截。 这个肉眼可见的跳动就是 Layout Shift（布局偏移），累计起来就是 CLS。

同一页面：不写尺寸 vs 写了尺寸（图片下载完成的瞬间）
❌ 没写 width/height
图片突然撑开
↑ 文字被瞬间往下推，用户可能正点错按钮
✅ 写了 width/height
早已预留位置
↑ 加载前就占好位，图片填进来，零偏移

CLS 是 Google Core Web Vitals 三大指标之一（另两个是 LCP、INP）， 直接计入搜索排名。一个 CLS 高的页面，不仅体验差（用户点击瞬间内容跳走、点错链接）， SEO 也会被扣分。

正确写法：HTML 给尺寸，CSS 做响应式

诀窍：HTML 属性写图片的原始像素宽高（浏览器据此算出宽高比预留空间）， CSS 用 max-width:100%; height:auto 让它在容器内自适应缩放。 现代浏览器会从 HTML 的 width/height 推断出 aspect-ratio，缩放时也不会塌成 0 高度。

黄金组合
<!-- HTML：写真实像素尺寸，浏览器据此预留空间 -->
<img src="hero.jpg" alt="产品主图"
     width="800" height="450">

/* CSS：让它在小屏不溢出，同时保持比例 */
img {
  max-width: 100%;   /* 不超过容器宽度 */
  height: auto;     /* 高度按比例自动算，不变形 */
}
常见误区：只写 width="800" 不写 height，或者只在 CSS 里设 width 不设 height:auto—— 前者算不出宽高比，CLS 照样发生；后者会让图片被拉伸变形。两个属性要成对出现。

四、loading="lazy"：原生懒加载

页面有十几张图，但用户首屏只看得到一两张。把看不到的图也立刻下载， 既浪费带宽，又抢占首屏图的加载资源。loading="lazy" 让浏览器 等图片快滚动进可视区时才去下载它，原生支持，不需要 JS。

懒加载：非首屏图才加
<!-- 首屏图：不要加 lazy，要尽快加载 -->
<img src="hero.jpg" alt="首屏大图"
     width="1200" height="600">

<!-- 页面下方、需滚动才看到的图：加 lazy -->
<img src="section-3.jpg" alt="第三屏配图"
     width="800" height="450"
     loading="lazy">
高频面试陷阱：千万别给首屏图（尤其 LCP 元素）加 loading="lazy"。 lazy 会让浏览器延后发起请求，首屏主图变成"慢半拍才出现"，直接拖垮 LCP（Largest Contentful Paint）。 lazy 是给"折叠线以下"的图用的，首屏图反而应该用 fetchpriority="high" 抢优先级。

五、decoding="async"

图片下载完还要解码（把 JPG/PNG 的压缩数据还原成像素位图）。 解码是 CPU 活，默认可能在主线程同步进行，卡住渲染。 decoding="async" 告诉浏览器把解码放到主线程之外异步做， 解码完再绘制，避免大图解码时页面卡顿。

异步解码，常和 lazy 一起用
<img src="photo.jpg" alt="风景照"
     width="1000" height="667"
     loading="lazy"
     decoding="async">
loading 管的是什么时候下载，decoding 管的是下载后怎么解码—— 两个不同阶段，互不冲突，非首屏图常常两个一起加。

六、图片格式怎么选

格式选错，要么图糊、要么文件大到拖慢加载。先记住一条主线： 能用矢量就用 SVG，照片优先 AVIF/WebP，需要兼容兜底再回退 JPG/PNG。

格式类型透明动画特点 / 适用
AVIF 位图（有损/无损） ✅ ✅ 压缩率最强，同画质比 WebP 还小，照片首选；老浏览器支持稍弱，需回退
WebP 位图（有损/无损） ✅ ✅ 比 JPG/PNG 小 25%~35%，支持度好，照片/通用图的主力
SVG 矢量 ✅ ✅(CSS/SMIL) 无限缩放不糊、文件极小，图标/logo/简单插画必用；不适合照片
PNG 位图（无损） ✅ ❌ 无损 + 透明，适合截图、需要透明的位图；照片用它会很大
JPG 位图（有损） ❌ ❌ 照片经典格式，压缩率高但不支持透明；现在多被 WebP/AVIF 替代
GIF 位图（无损） ❌ ✅ 仅 256 色、动画文件大，已过时；动图改用 WebP/AVIF 或视频更优
选型口诀：图标/logo → SVG；照片 → AVIF 优先、WebP 兜底； 需要无损透明截图 → PNG；动图别用 GIF，用 WebP/AVIF 或 <video>。 多格式降级回退怎么写，下一节 <picture> 会讲。

七、img 是"行内替换元素"

<img> 的盒子类型很特殊：它是 inline replaced element（行内替换元素）。 "行内"意味着它和文字一样排在一行里、不独占整行； "替换"意味着它的内容来自外部资源（图片），不是 HTML 本身——浏览器用图片"替换"掉这个标签。

特性普通行内元素（如 span）img（行内替换元素）
能否设 width/height❌ 无效✅ 有效
是否随文字排在一行✅✅
内容来源HTML 内部文本外部资源（图片）
底部空隙之谜：img 默认按 vertical-align: baseline 对齐， 会在图片下方留出一条诡异的小空白（给字母下伸部如 g、p 预留的空间）。 常见解法：给 img 设 display: block 或 vertical-align: middle/bottom。 这是因为它"行内"的身份还在，按文字基线排版。

八、常见陷阱速查

陷阱后果正确做法
不写 alt，或写成 alt="图片" / alt="image" 读屏念出无意义内容，SEO 拿不到信息，失败时只剩破图 内容图描述内容，功能图描述功能，装饰图 alt=""
不写 width/height 图片加载完撑开页面，CLS 抖动，扣 SEO 分 HTML 写真实尺寸 + CSS max-width:100%;height:auto
给首屏图 / LCP 图加 loading="lazy" 主图延迟下载，LCP 变差，首屏更慢 首屏图不加 lazy，必要时加 fetchpriority="high"
所有图都用原图、不压缩 几 MB 的大图拖垮加载，浪费带宽 压缩 + 选对格式（AVIF/WebP），按显示尺寸出图
照片用 PNG、图标用 JPG 照片 PNG 体积巨大；图标 JPG 边缘模糊、不能透明 照片用有损（AVIF/WebP/JPG），图标用 SVG

九、与 Java 的类比

img 概念Java 类比
alt=""（主动声明"无内容"） 显式返回 Collections.emptyList() / Optional.empty() 而不是 null—— 主动告诉调用方"确实没有"，让对方安全跳过，防 NPE。 省略 alt 就像返回 null，调用方分不清是"无值"还是"忘了赋值"
loading="lazy" 懒加载 Spring 的 @Lazy / Hibernate 延迟加载—— 用到时才初始化，不在启动/首屏时一次性把所有资源都拉起来
格式降级回退（AVIF → WebP → JPG） 策略模式 / 责任链——按浏览器能力选择不同实现， 都失败再回退到通用兜底方案（下一节 <picture> 就是声明式的策略选择）
decoding="async" 异步解码 把耗 CPU 的工作丢进线程池异步处理（CompletableFuture）， 不阻塞主线程，类似不让 EDT / UI 线程做重活
width/height 预留空间防 CLS 提前 new ArrayList<>(expectedSize) 预分配容量—— 知道最终大小就先占好，避免运行时反复扩容（数组拷贝）的抖动
<img> 的精髓是：用声明式属性，把"语义（alt）、布局（width/height）、加载策略（lazy/decoding）" 都提前告诉浏览器，让它能并行优化。这和后端"提前声明意图，框架据此优化执行" （如预分配、懒加载、异步线程池）是同一种工程思维。

与其他知识点的联系

  • 路径（上一节）：src 属性的值完全遵循上一节讲的相对/绝对路径规则，图片"加载失败"这个 alt 要处理的场景，很大一部分现实原因就是路径写错——理解了路径的基准点，才能真正明白 alt 兜底机制在防什么。
  • 响应式图片 srcset/picture（下一节）：本文只讲了单张图的 src，下一节的 srcset/<picture> 本质上是把"一个 src"扩展成"一组候选图片"，让浏览器根据屏幕密度、视口宽度自己选——width/height 防 CLS、loading="lazy" 懒加载这些属性在响应式图片里依然要写，只是要多考虑"选中的是哪张图"这个变量。
  • 视频与 iframe（后续章节）：<video>、<iframe> 同样面临"内容还没加载完，该占多大空间"的问题，CLS 防抖动的思路（提前声明宽高比）会在这些标签上重复出现，是同一类"外部资源嵌入"标签的通用性能考量。
  • CSS 的 object-fit 与背景图：本文提到的 max-width:100%;height:auto 只解决等比缩放，如果要图片"裁切填满容器不变形"，还需要 CSS 的 object-fit: cover；而纯装饰性的图（对应本文 alt="" 的场景）很多时候干脆不用 <img>，直接用 CSS background-image 实现，因为背景图天生就不需要 alt 语义。
  • Core Web Vitals 与整体性能优化：本文的 CLS、LCP 只是 Core Web Vitals 的一部分，图片优化（格式选择、懒加载、尺寸预留）通常是性能优化里收益最高的一步，后续学到资源预加载（<link rel="preload">）、字体优化时，会用到和这里完全一致的"提前声明，让浏览器提前规划"的思路。
  • React 中的图片处理：在 React 里写 <img> 时，alt、width/height 依然是必须手写的原生属性，Next.js 的 <Image> 组件更是把本文讲的"防 CLS、懒加载、格式选择"这些最佳实践直接做成了强制默认行为——理解本文的原理，才能明白这些框架组件到底在替你做什么。
