HTML · 表格

表格基础结构：table / tr / th / td

表格是 HTML 里语义最强、也最容易被误用的元素。用对了，它是展示二维数据的利器；用错了（拿来做布局），它就是 20 年前最臭名昭著的反模式。这一节先把骨架立稳。

一、四个核心标签：先认清角色

一个表格由四个基础标签搭起来，它们的关系是层层嵌套的： table 装下所有内容，里面是若干 tr（行），每个 tr 里再放 th 或 td（单元格）。

四个基础标签的职责
标签全称职责
tabletable表格容器，包裹整张表
trtable row一行，横向的一条记录
thtable header表头单元格，有语义（默认加粗居中）
tdtable data数据单元格，存放普通数据
记忆锚点：tr 是 row（行），td 是 data（数据），th 是 header（表头）。 全是英文首字母缩写，认出全称就再也记不混。

二、表格是"按行组织"的（先行后列）

这是新手最容易栽的地方：HTML 表格没有"列"这个标签。 你不是先定义几列、再往列里塞数据，而是一行一行地从上往下写， 每行里依次摆放单元格，单元格的位置（第几个）自动决定它属于第几列。

一个最小可用的表格
<table>
  <!-- 第一行：表头 -->
  <tr>
    <th>姓名</th>
    <th>部门</th>
  </tr>
  <!-- 第二行：数据 -->
  <tr>
    <td>张三</td>
    <td>研发</td>
  </tr>
</table>

上面渲染出来就是下面这张表。注意第一行用 th，所以它默认加粗——这不是样式，是语义带来的视觉表现：

员工列表（上面代码的渲染效果）
姓名部门
张三研发
李四市场
类比 Java：表格像一个 List<Row>，你只能 add 一行一行的 Row， 每个 Row 里按顺序放字段。没有"先建列再填值"的写法——和你 for 循环遍历 ResultSet 时 一行行 next() 是同一个思维。

三、caption：给表格一个标题

caption 是表格的标题，必须紧跟在 <table> 后面、作为第一个子元素。 很多人省略它，觉得"反正旁边有个 h2 写了标题"——但对屏幕阅读器用户来说， caption 是表格内部唯一可靠的"这张表是干嘛的"说明。

caption 必须是 table 的第一个子元素
<table>
  <caption>2026 年第一季度销售额（单位：万元）</caption>
  <tr> ... </tr>
</table>
无障碍提醒：当屏幕阅读器进入一张表时，会先读出 caption， 用户才知道"我现在在哪张表里"。没有 caption，盲人用户面对一堆孤立的单元格会完全失去上下文。 数据表请务必加 caption。

四、scope：告诉屏幕阅读器表头管哪个方向

th 表示"我是表头"，但它管的是一整列还是一整行？ 人眼一看就懂（顶部那行管列，最左那列管行），但屏幕阅读器看不出来—— 必须用 scope 属性显式声明。

scope 的两个取值
取值含义用在哪
scope="col"这个表头管理它所在的整列顶部那行的表头
scope="row"这个表头管理它所在的整行最左侧那列的表头
行表头 + 列表头：典型的二维表
<table>
  <caption>各城市季度销量</caption>
  <tr>
    <td></td>                  <!-- 左上角通常留空 -->
    <th scope="col">Q1</th>
    <th scope="col">Q2</th>
  </tr>
  <tr>
    <th scope="row">北京</th>
    <td>120</td>
    <td>150</td>
  </tr>
</table>

渲染效果（注意最左列也是加粗的 th，因为它是行表头）：

各城市季度销量（单位：万件）
Q1Q2
北京120150
上海98110
为什么 scope 重要：当屏幕阅读器读到"150"这个单元格时， 有了 scope 它能补一句"上海 Q2：150"——把孤立的数字和它对应的行表头、列表头关联起来。 没有 scope，用户只能听到一串没有上下文的数字。

五、最重要的原则：表格只用于二维数据，绝不用于布局

这是整节最关键的一条。表格的语义是"这是一组结构化的、有行列关系的数据"。 它存在的全部理由，就是表达数据之间的二维关系（这一行、这一列代表什么）。

反模式：用 table 做页面布局
2000 年代 CSS 还不成熟时，前端流行用嵌套的 <table> 来排版整个页面—— 左边一栏导航、右边一栏内容，全靠表格单元格拼出来。现在这是明确禁止的反模式。

为什么 table 布局是反模式

table 布局的四宗罪
问题具体后果
语义错误屏幕阅读器把导航栏当成"数据表"来读，盲人用户彻底懵掉
无法响应式表格列宽僵硬，难以在手机上重排成单列布局
渲染慢浏览器要等整个表格下载完才能计算列宽，无法增量渲染
代码臃肿嵌套五六层 table 才排出一个页面，难维护、难修改

正确做法：页面布局用 CSS（Flexbox、Grid）。 布局是"视觉表现"，应该交给 CSS；table 是"数据语义"，只描述内容是什么。 这正是 HTML / CSS 分离的核心思想——结构与样式解耦。

一句话判断：问自己"这些内容去掉边框后，还是不是一张二维数据表？" 如果是销量表、成绩单、对照表——用 table。 如果只是想把几个区块并排放——那是布局，用 CSS Grid / Flex。

六、常见陷阱速查

表格新手高频错误
陷阱后果正确做法
用 table 做页面布局 语义错乱、无障碍崩坏、难响应式 布局用 CSS Grid / Flexbox
把 th 当"加粗的 td"用 普通数据被当成表头，语义错误 只有真正的表头才用 th，加粗要靠 CSS
省略 caption 屏幕阅读器用户不知道这是什么表 数据表都加 caption
th 不写 scope 表头与单元格关联丢失，读屏体验差 列表头加 scope="col"，行表头加 scope="row"
各行单元格数量不一致 表格错位，列对不齐 每行 td/th 数量保持一致（合并单元格除外）

七、与 Java 的类比

HTML 表格概念对照 Java
表格概念Java 类比
table 整张表 二维数组 String[][]，或一个 JDBC 的 ResultSet——本质都是行列结构的数据集
tr 一行 结果集里的一条记录 Row，resultSet.next() 拿到的那一行
th 表头 列名 / 元数据（ResultSetMetaData.getColumnName()）——描述这一列是什么，不是数据本身
td 数据单元格 记录里某个字段的值，resultSet.getString("name") 取出的内容
scope 关联表头 字段值与列名的映射关系（列索引 → 列名），让单元格知道自己属于哪一列
"按语义用表格，不拿来布局" "用对数据结构"——别拿 Map 当 List 用，类型/语义匹配才好维护
后端里你绝不会把一个"用户列表"硬塞进一个本该存配置的 Properties 对象—— 因为语义不对。前端同理：table 是为"二维数据"这种语义准备的， 拿它去排版页面，就像拿数组去当对象用，能跑，但是错的设计。 选对的元素表达对的语义，是写好 HTML 的底层素养。
