HTML · 文本与内容

引用与代码块：blockquote / q / cite / pre / code

这一组标签的共同主题是"原样保留与语义标注"。引用要告诉机器"这段话是别人说的"，代码块要告诉浏览器"别动我的空格和换行"。理解它们，绕不开 HTML 一个根本机制——空白折叠。

一、blockquote：块级引用

<blockquote> 表示一段独立成块的长引文——别人书里的一段话、 一篇文章里摘出来的段落。它是块级元素，浏览器默认会给它左右缩进。 引文的来源 URL 写在 cite 属性里。

blockquote 长引文，cite 属性放来源链接
<blockquote cite="https://www.w3.org/TR/html52/">
  <p>HTML 是一种用来描述网页结构的标记语言，
     而不是用来控制外观的语言。</p>
  <footer>——出自 <cite>HTML 5.2 规范</cite>
  </footer>
</blockquote>
浏览器渲染效果

HTML 是一种用来描述网页结构的标记语言，而不是用来控制外观的语言。

——出自 HTML 5.2 规范
注意：cite 属性的 URL 不会显示在页面上。它只是给搜索引擎、 辅助技术和爬虫的机器可读元数据。想让读者看到来源，得自己在 <footer> 里写出来。

二、q：行内短引用

<q>（quote）用于夹在句子中间的短引文。它最大的特点是： 浏览器会自动加上引号，你自己不要再手动打引号，否则会出现两层引号。

q 行内引用，引号由浏览器自动生成
<p>鲁迅说过 <q>其实地上本没有路</q>，
   走的人多了便成了路。</p>

<!-- ❌ 错误：自己又加了引号，会变成 “「...」” 双层 -->
<p>他说 “<q>你好</q>”</p>
浏览器渲染效果（引号是 q 自动加的）

鲁迅说过 其实地上本没有路，走的人多了便成了路。

怎么选 blockquote 还是 q？看引文是否独立成段。 单独占一块、需要换行的长引文用 blockquote（块级）； 嵌在正文句子里的短引文用 q（行内）。这和 <div> 与 <span> 的块/行内区分是一个道理。

三、cite：标注作品名称

<cite> 这个标签的名字极具迷惑性。它不是"引用来源"的意思， 而是用来标注一部作品的名称——书名、文章名、电影名、歌曲名、软件名等。 浏览器默认把它渲染成斜体。

cite 标注作品名（书名 / 文章名）
<p>这个观点最早出现在 <cite>设计模式</cite> 一书中。</p>
高频陷阱：很多人把 <cite> 当成"引用某人说的话的出处"， 于是写 <cite>张三</cite> 来标人名。这是错的—— 人名、机构名不属于"作品"，不该用 cite。 HTML 规范明确：cite 标注的是创作物的标题。人名用普通文本或 <footer> 即可。

四、空白折叠：理解 pre 的前提

要讲清楚 <pre>，必须先讲 HTML 一条贯穿始终的规则： 空白折叠（whitespace collapsing）。 在普通 HTML 里，连续的空格、Tab、换行，都会被合并成一个空格。

源码里排得整整齐齐
<p>Hello          World
这一行换行了
这一行也换行了</p>
浏览器渲染：多个空格 + 换行 → 全压成一个空格

Hello World 这一行换行了 这一行也换行了

为什么浏览器要这么设计？因为 HTML 的定位是"描述结构"， 排版（行宽、换行位置）应该由浏览器根据窗口大小动态决定。 如果源码里的每个空格和换行都照单全收，那作者写代码时为了可读性加的缩进、 断行，全都会跑到页面上去——这显然不是想要的。 所以默认行为是：把源码中的"格式空白"当作可忽略的排版噪音，统一折叠。

类比：这就像 Java 源码里的缩进和空行——它们只是给人看的， 编译器（javac）会忽略它们，int a=1; 和缩进对齐的写法编译出的字节码完全一样。 HTML 的空白折叠，本质上是同一种"格式归格式，内容归内容"的分离思想。

五、pre：预格式化文本（空白折叠的例外）

<pre>（preformatted）是空白折叠规则的唯一例外。 它告诉浏览器：这段文本里的每一个空格、Tab、换行都要原样保留， 并且默认用等宽字体渲染。这正是展示代码、ASCII 图、诗歌缩进等场景需要的。

pre 原样保留空格与换行
<pre>名称      价格    库存
苹果      3.5     120
香蕉      1.2     88</pre>
浏览器渲染：对齐和换行都保住了
名称      价格    库存
苹果      3.5     120
香蕉      1.2     88
陷阱：<pre> 对源码缩进非常敏感。 既然它保留所有空白，那你写在标签里的换行和缩进就会照搬到页面上。 所以 <pre> 的内容通常要"顶着行首写"，别跟着外层 HTML 缩进，否则页面上会多出一堆前导空格。

六、code 与 pre>code：行内代码与代码块

<code> 表示一段计算机代码，默认等宽字体。 它本身是行内元素，且不改变空白折叠规则—— 单独用 <code> 只适合句子中间提一个变量名、方法名这种行内代码。

行内代码：code 单用
<p>调用 <code>list.add(x)</code> 把元素追加到末尾。</p>

展示多行代码块时，正确做法是 <pre> 套 <code>： <pre> 负责"保留换行和缩进"，<code> 负责"语义上这是代码"。 两者各司其职，是语义最准确的写法。

代码块：pre > code 组合（语义最准确）
<pre><code>public static void main(String[] args) {
    System.out.println("Hello");
}</code></pre>
为什么不能只用 <code> 放多行代码？ 因为 <code> 不改变空白折叠——多行代码里的换行和缩进会被压成一个空格， 整段代码挤成一行。必须靠外层 <pre> 来保住格式。 记法：pre 管"形"（保留排版），code 管"义"（标注是代码）。

七、实体转义：代码块里的 < > & 必须转义

这是写代码块时最容易踩的坑。假设你想在页面上展示一段 HTML 代码， 比如 <div>，如果直接把 <div> 写进 <pre> 里—— 浏览器会把它当成真正的 div 标签去解析，而不是当文本显示出来。 结果就是你的代码"消失"了，或页面结构错乱。

解决办法是把这些HTML 特殊字符转义成实体（entity）。 浏览器看到实体，会显示对应的字面字符，而不会当成标签语法：

字符实体写法为什么必须转义
<&lt;否则被当成标签开始
>&gt;否则被当成标签结束
&&amp;& 是实体的起始符，本身也要转义
要在页面上显示 <div>，源码里得写转义
<!-- ❌ 错误：浏览器把它当真标签解析，页面上看不到 -->
<code><div></code>

<!-- ✅ 正确：转义后才能原样显示出 <div> 这几个字 -->
<code>&lt;div&gt;</code>
顺便说一个细节：这整个学习页面里，每一段示例代码（包括你正在看的这些） 都做了转义——所有的 < 在源码里其实写的是 &lt;。 这也是所有代码高亮组件、Markdown 渲染器在底层必做的一步。

用 cite 标注人名/出处 语义错误，cite 只用于作品标题 人名用普通文本，URL 放 cite 属性

九、与 Java 的类比

HTML 概念Java 类比
pre 保留原始格式（空格 / 换行） 文本块 text block（三引号 """）——原样保留多行字符串里的换行与缩进，不用再拼 \n
空白折叠（默认压缩多余空白） 普通字符串拼接里你得手动控制空白；javac 忽略源码缩进，和 HTML"格式归格式"是同一思路
实体转义 &lt; &gt; &amp; StringEscapeUtils.escapeHtml4()（Apache Commons）——把特殊字符转成安全的实体，防止被当语法解析
blockquote 的 cite 属性标注来源 Javadoc 里的 @see / {@link} 指向引用来源——机器可读的"出处指针"，不直接出现在正文逻辑里
cite 标注作品名 Javadoc @author / 注解里的元信息——标注"这是什么"，属于描述性元数据而非内容本身
一句话总结：这一组标签的核心是"分清形与义"。 pre 解决"形"（保留排版），转义解决"安全显示"， 而 blockquote / q / cite / code 解决"义"（告诉机器这段内容是什么）。 HTML 的语义化，和 Java 用类型、注解、文本块把意图显式表达出来，是相通的工程审美。
