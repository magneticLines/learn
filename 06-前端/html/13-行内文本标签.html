行内文本语义标签

同样是"加粗"，用 strong 还是 b？同样是"斜体"，用 em 还是 i？看起来一样，但屏幕阅读器、搜索引擎、未来的你读到的是两回事。这一节讲的核心只有一句话：标签选的是语义，不是样式。

一、核心心法：语义 vs 样式

HTML 标签的本职工作是描述内容"是什么"，而不是"长什么样"。 "长什么样"是 CSS 的活儿。一个 <strong> 默认显示为加粗，但加粗只是浏览器顺手给的 默认样式——它真正的含义是"这段内容很重要"。

为什么这个区别要紧？因为内容会被很多"读者"消费，不只是人眼：

消费者它怎么用语义
屏幕阅读器读到 strong 会加重语气、提高音量；读到 b 就平铺直叙
搜索引擎用语义判断页面结构、关键词权重
浏览器/插件阅读模式、翻译、自动生成大纲都依赖语义
未来维护者看到 time 就知道这是时间，而不是一段普通文字
一句话立场：能用语义标签就别用 span，能表达"重要/强调"就别只为了加粗斜体用 b/i。 样式有 CSS 兜底，语义一旦丢了就永远丢了。

二、strong vs b：重要 ≠ 加粗

两者默认都是加粗，视觉上一模一样。区别全在语义：

标签语义什么时候用
<strong>重要性 / 严重性 / 紧迫性，阅读器会重读警告、关键提示、不能忽略的内容
<b>仅视觉加粗，不含任何重要性关键词、产品名、文章导语中需吸引眼球但并不"更重要"的词
语义对比
<!-- ✅ strong：这是一句必须重视的警告 -->
<p>操作前请<strong>务必备份数据</strong>，删除不可恢复。</p>

<!-- ✅ b：只是想让关键词显眼，不含"更重要" -->
<p>本文使用 <b>Spring Boot</b> 搭建后端。</p>
判断口诀：把内容读出来，如果你会不自觉地加重语气，那就是 strong； 如果只是想让它"看起来粗一点"，那是样式需求，可以用 b，更推荐直接用 CSS font-weight。

三、em vs i：强调语气 ≠ 斜体

和 strong/b 是同一套逻辑。两者默认都是斜体，语义不同：

标签语义什么时候用
<em>强调语气（emphasis），会改变句子重音口语里你会重读的那个词
<i>仅斜体，表示"另一种声调/类别"书名、术语、外来词、思想活动、专有名
语气会随 em 的位置改变
<!-- em 落在不同词上，整句意思就变了 -->
<p>我没说<em>他</em>偷了钱。</p> <!-- 暗示是别人偷的 -->
<p>我没说他<em>偷</em>了钱。</p> <!-- 暗示是借/拿，不是偷 -->

<!-- ✅ i：书名、术语，用斜体表达"另一类文字"，不是强调 -->
<p>他在读 <i>深入理解 Java 虚拟机</i>。</p>

四、其余语义标签速览

下面这些都自带"看一眼就懂"的视觉效果，但每个都有明确语义，别只当成样式开关用。

mark / del / ins —— 高亮与变更

mark 高亮、del/ins 表达版本变更
<!-- mark：因当前上下文而被高亮的部分（如搜索命中词）-->
<p>搜索结果：Java <mark>并发</mark> 编程</p>

<!-- del + ins：不只是划线，表示"内容被删改"的语义，可配 datetime -->
<p>价格：<del>￥99</del> <ins>￥69</ins></p>
别拿 del 当删除线样式用：del/ins 表示"文档内容被删改"的版本语义， 阅读器会读成"已删除/已插入"。如果你只是想要一条横线（比如装饰），用 CSS text-decoration: line-through， 或语义上用 <s>（表示"不再准确/不再相关"，但不是被编辑删除）。

sub / sup —— 下标与上标

化学式、数学幂、脚注引用
<p>水是 H<sub>2</sub>O，能量公式 E = mc<sup>2</sup>。</p>

small —— 附属说明

免责声明、版权、旁注
<p>立即购买 <small>（最终解释权归商家所有）</small></p>

small 的语义是"附属细则/旁注"，不是"把字变小"。要单纯改字号请用 CSS font-size。

abbr —— 缩写（配 title 全称）

鼠标悬停显示全称
<p>前端三件套之一是 <abbr title="Cascading Style Sheets">CSS</abbr>。</p>

time —— 机器可读时间

datetime 属性给机器看，标签内文字给人看
<!-- 人看到"2026年6月23日"，机器读 datetime 里的标准格式 -->
<time datetime="2026-06-23">2026年6月23日</time>

<!-- 带时区的精确时刻 -->
<time datetime="2026-06-23T14:30:00+08:00">下午两点半</time>
time 是 SEO 和日历应用的好朋友：搜索引擎能识别文章发布日期，浏览器/插件能把它加进日历。 格式遵循 ISO 8601（和后端 LocalDateTime 序列化的格式同源）。

code / kbd —— 代码与键盘按键

行内代码片段 与 用户该按的键
<!-- code：一小段代码、命令、变量名 -->
<p>调用 <code>list.add(item)</code> 添加元素。</p>

<!-- kbd：提示用户敲哪些键 -->
<p>按 <kbd>Ctrl</kbd> + <kbd>S</kbd> 保存。</p>

五、span：没有语义的兜底容器

<span> 是无语义的行内容器——它什么都不表示，存在的唯一理由是： 当你需要给一小段文字单独挂样式或当 JS 钩子，而又找不到任何合适的语义标签时， 用 span 包起来。

span 的正当用途：纯样式/JS 锚点
<!-- 给一段文字单独上色，没有语义可言，span 合适 -->
<p>余额 <span class="text-red">-￥50</span></p>

<!-- 给 JS 一个抓取点 -->
<span id="countdown">60</span> 秒后跳转
陷阱：span 包一切。很多人习惯性地 <span class="bold">、<span class="code">， 其实那些场景早有语义标签（strong、code）。先问自己"这段内容是什么"， 有语义标签就用语义标签；确实只是"一块需要单独控制的文本"，才轮到 span。

七、常见陷阱速查

错误问题正确做法
用 b/i 代替 strong/em 丢失"重要/强调"语义，阅读器不重读，SEO 拿不到信号 表达重要性用 strong，表达语气用 em
用 span 包一切再挂 class 该有语义的地方变成"哑标签"，可访问性和可维护性双输 先找语义标签，找不到才用 span
把 del 当纯删除线样式用 del 带"内容被编辑删除"的版本语义，会被误读 纯装饰用 CSS line-through；"不再准确"用 <s>
abbr 不写 title 缩写没有全称提示，语义价值大打折扣 始终用 title 给出完整写法
time 只写文字、漏 datetime 机器读不到标准时间，等于普通文本 补上 ISO 8601 格式的 datetime
为"把字变小/变粗"而选 small/strong 用语义标签解决样式问题，语义被滥用 纯样式需求交给 CSS，标签只管语义

八、与 Java 的类比

"语义 vs 样式"这件事，后端工程师其实再熟悉不过——它就是元信息 vs 实现的分离。

HTML 概念Java 类比
语义标签（strong/time/code） 注解元信息（@Deprecated/@NotNull）——给框架、工具、阅读者额外的"含义"，而非运行逻辑本身
strong（有重要性语义）vs b（仅加粗） 有业务含义的方法名 chargeUserAccount() vs 纯工具方法 format()——前者承载意图，后者只是手段
span（无语义兜底容器） Object / 裸 Map<String,Object>——什么都能装，但什么也没表达，能用具体类型就别用它
time datetime（机器可读 + 人可读双轨） 实体里 LocalDateTime 字段 + @JsonFormat 显示格式——一份给机器算，一份给人看
del/ins（带版本变更语义） 审计字段 / @Version 乐观锁——不只是记录值，更记录"被改过"这件事
样式交给 CSS、语义交给标签 关注点分离（SoC）：业务逻辑、展示、配置各归各位，别揉成一坨
本质上，选对行内标签和你给字段挑对类型、给方法起对名字是同一种素养： 让代码（或文档）自己说清楚"它是什么"。 样式可以随时换皮，语义却是内容的骨架——骨架对了，可访问性、SEO、可维护性都跟着对。

与其他知识点的联系

  • 标题与段落（上一节）：标题与段落解决的是"块级结构"的语义划分，行内标签解决的是块内部"这个词/这句话是什么"的更细粒度语义——两者是同一套"内容语义优先"思路在不同粒度上的应用。
  • 可访问性（accessibility）：屏幕阅读器对 strong、em、time、abbr 有专门的朗读策略（重读、全称提示），而 b/i/span 只是纯视觉标签，阅读器会原样跳过——这正是为什么本节反复强调"语义标签优先"：它不是代码洁癖，而是决定了残障用户能不能获取到和视觉用户同等的信息。
  • SEO：搜索引擎爬虫会给 strong/em 包裹的关键词额外权重，也会解析 time datetime 作为结构化的发布时间信号；如果全用 b/i/span 代替，等于把这些排名信号亲手抹掉了。
  • CSS 样式覆盖：浏览器给 strong/b 加粗、em/i 斜体，都只是 User Agent 默认样式，实际项目里常被 CSS reset 清零或用 font-weight/font-style 重新定义——默认视觉会变，但标签背后的语义（重要/强调/删改）不会因为你改了 CSS 而消失，这也是为什么不能靠"看起来像什么"来选标签。
  • JSX 语法：React 里这些标签原样保留（<strong>、<time dateTime="...">），唯一的坑是 HTML 的 datetime 属性在 JSX 里要写成驼峰 dateTime——这也提醒你，语义标签和它承载的属性是一体的，迁移到 JSX 时属性名会变，但语义含义不变。
  • span 与 React 组件设计：span 作为"无语义兜底容器"的定位，和 React 里常见的 <div>/<span> 包裹纯样式内容是同一个道理——如果发现自己在组件里大量用 span 挂 class 而没有任何语义，通常说明还没找到合适的语义元素，或者这段内容本就该拆成一个有名字的子组件。
