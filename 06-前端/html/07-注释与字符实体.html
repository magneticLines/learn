注释与字符实体

两个小知识点，但坑点不少。注释被很多人当"死代码仓库"，字符实体被很多人写错或完全不知道。

一、HTML 注释

1.1 语法

注释语法
<!-- 这是一条注释，浏览器不会渲染它 -->

<!-- 多行注释：
     第二行
     第三行
-->

<p>正常内容</p>
<!-- <p>这段被注释掉，不显示</p> -->
<p>正常内容</p>

1.2 注释的正确用途

注释应该解释为什么，而不是是什么。代码本身应该能说明它做了什么。

好的注释 vs 坏的注释
<!-- ❌ 废话注释：重复了代码已经说明的事 -->
<!-- 导航栏 -->
<nav>...</nav>

<!-- ✅ 有价值的注释：解释了非显而易见的约束 -->
<!-- 移动端导航：只在 max-width:768px 时由 JS 切换 .open -->
<nav id="mobile-nav">...</nav>

<!-- ✅ 标记待办：TODO 让团队知道这里需要处理 -->
<!-- TODO: 后端接口就绪后替换为动态数据 -->
<ul id="news-list">...</ul>

1.3 注释的陷阱

注释内容对用户可见：HTML 注释不会在页面上显示，但用户打开"查看源代码"就能看到。 绝对不要在注释里写密码、API Key、内部系统路径、TODO 中的业务逻辑细节。 源码是公开的，注释也是。

二、字符实体

2.1 为什么需要字符实体

HTML 里有些字符有特殊含义：< 开启标签，> 关闭标签，& 开启实体。 如果你在页面上想显示这些字符本身，直接写会被浏览器解析成语法，必须用字符实体来转义。

不转义 vs 转义
<!-- ❌ 直接写 < 会被解析为标签开始，页面显示异常 -->
<p>1 < 2 > 0</p>   ← 浏览器可能把 < 2 > 当成未知标签

<!-- ✅ 用字符实体 -->
<p>1 &lt; 2 &gt; 0</p>  ← 页面显示：1 < 2 > 0

2.2 字符实体语法

字符实体有两种写法，都以 & 开头、; 结尾：

写法格式示例
命名实体 &名称; &lt; → <
数字实体（十进制） &#数字; &#60; → <
数字实体（十六进制） &#x十六进制; &#x3C; → <

命名实体可读性更好，优先用命名实体。数字实体用于没有命名版本的特殊符号。

2.3 必须记住的实体

显示字符命名实体何时必须用
<&lt;文本内容里显示小于号（不在标签里）
>&gt;文本内容里显示大于号（技术上可不转，但推荐）
&&amp;文本里显示 & 符号，必须转义
"&quot;属性值内部出现双引号时
'&apos;属性值内部出现单引号时（HTML5 新增）
（不换行空格）&nbsp;需要空格但不希望此处换行
最容易漏掉的：&amp;
显示"AT&T"要写 AT&amp;T，写成 AT&T 会让浏览器把 &T 当成未知实体， 部分浏览器会静默忽略，有些浏览器会显示原文，行为不一致。正确写法始终是转义。

2.4 常用符号实体速查

符号实体用途
 （不换行空格）&nbsp;防止两个词之间换行（如"100 km"）
©&copy;版权符号
®&reg;注册商标
™&trade;商标符号
—&mdash;破折号（em dash）
–&ndash;连接号（en dash，如日期范围）
×&times;乘号（区别于字母 x）
÷&divide;除号
→&rarr;右箭头
←&larr;左箭头
…&hellip;省略号（一个字符，不是三个点）
«»&laquo; &raquo;书名号/法文引号

2.5 现代写法：直接用 UTF-8

在 HTML 文件声明了 <meta charset="UTF-8"> 之后， 绝大多数特殊字符（©、→、×、中文…）可以直接在源码里写，不必用实体。 唯一必须转义的是对 HTML 解析器有特殊含义的字符：<、>、&，以及属性值内的引号。

UTF-8 时代的推荐写法
<!-- ✅ 直接写 UTF-8 字符，更直观 -->
<p>版权所有 © 2026 SuperNova Inc.</p>
<p>请求流程：客户端 → 网关 → 服务</p>

<!-- 以下三个仍然必须用实体 -->
<p>泛型语法：List&lt;String&gt;</p>  ← < >
<p>公司名：AT&amp;T</p>           ← &
结论：现代项目只需记住三个必须转义的字符—— &lt;（<）、&gt;（>）、&amp;（&）。 其余特殊字符直接输入 UTF-8 即可，实体写法留给没有中文输入法或历史遗留代码的场景。

2.6 &nbsp; 的常见误用

不要用 &nbsp; 来制造缩进或间距
很多初学者用多个 &nbsp;&nbsp;&nbsp; 来实现视觉缩进—— 这和用 <br> 制造段落间距一样，是用内容标记做排版的错误做法。 间距和缩进应该用 CSS 的 padding、margin、text-indent 实现。

&nbsp; 唯一正确的语义用途：防止特定两个词之间断行 （如 100&nbsp;km、Mr.&nbsp;Smith）。

三、与 Java 的类比

HTML 概念Java 类比
HTML 注释 <!-- --> Java 注释 // ... / /* ... */：源码可见，编译/渲染后消失
注释不写进生产代码 Java 中不把密码写注释里，不把注释掉的代码提交——同一个原则
字符实体转义 &lt; StringEscapeUtils.escapeHtml4("<") → &lt;，后端返回 HTML 内容时必须做的转义
不转义 < 直接输出到页面 SQL 不用参数化查询直接拼字符串 → XSS 漏洞的根源和 SQL 注入同一类问题
安全视角：字符实体转义不只是"让字符正确显示"， 它本质上是防止 XSS（跨站脚本攻击）的第一道防线。 如果后端把用户输入的 <script>alert(1)</script> 不经转义直接插入 HTML， 浏览器会执行它。这是 OWASP Top 10 常客。前后端都必须做这层转义。
