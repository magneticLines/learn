HTML · 表单

button 类型：submit / reset / button

同一个 <button>，写不写 type 行为天差地别。表单里漏写 type 导致页面莫名其妙刷新、数据丢失，是初学者排查半天才发现的高频坑。

一、button 的三种 type

<button> 有且只有三种 type，决定了它被点击时浏览器要不要"动表单":

三种 type 的行为
<!-- ① submit：提交表单（这是默认值！）-->
<button type="submit">提交</button>

<!-- ② reset：把表单所有字段重置为初始值 -->
<button type="reset">重置</button>

<!-- ③ button：无任何默认行为，纯靠 JS 绑事件 -->
<button type="button">点我</button>
type点击后浏览器默认做什么典型用途
submit触发表单校验 → 提交表单 → 页面跳转/刷新"登录""保存""下单"按钮
reset把表单内所有控件还原成 HTML 里的初始值"清空""恢复默认"（实际很少用，体验差）
button什么都不做，行为完全交给 JS切换、计算、弹窗等纯交互按钮
核心记忆点：<button> 的 type 默认值是 submit，不是 button。 这一个反直觉的默认值，是后面所有坑的根源。

二、最大的坑：form 里的 button 不写 type 会误提交

设想你在一个 <form> 里放了一个"切换密码可见"的小按钮，纯前端交互，本不该提交表单。 但你没写 type：

反例：一点就刷新整个页面
<form action="/login">
  <input type="password" id="pwd">

  <!-- ❌ 没写 type → 默认 submit → 点击直接提交表单 -->
  <button onclick="toggle()">👁 显示密码</button>

  <button type="submit">登录</button>
</form>

用户点"显示密码",toggle() 确实跑了,但浏览器同时把整个表单提交了—— 页面跳转到 /login 或者刷新,用户填的内容全没了。更隐蔽的是:很多人用单测/手测时 JS 函数明明执行了,就是想不通页面为什么会跳走。

正例：明确写 type="button"
  <!-- ✅ 纯交互按钮一律写 type="button" -->
  <button type="button" onclick="toggle()">👁 显示密码</button>
铁律：只要 <button> 写在 <form> 里、又不是用来提交的, 必须显式加 type="button"。React/Vue 项目里这个坑同样存在——表单里的图标按钮忘记加 type,一点就触发 onSubmit。
为什么这样设计? <button> 诞生时就是为表单服务的,默认值定为 submit 是为了"最常见的场景少写一个属性"。 但今天 <button> 早已成为通用交互元素,这个历史默认值就成了反直觉的陷阱。HTML 不能改默认值(会破坏存量页面),只能靠你记住。

三、button vs input：能不能放 HTML 内容

提交按钮其实有两种写法:<button type="submit"> 和 <input type="submit">。 最大区别是能放什么内容:

button 可包含 HTML，input 只能纯文本
<!-- button：内容是子元素，可以塞图标、加粗、多行 -->
<button type="submit">
  <svg>...</svg> <strong>保存</strong>
</button>

<!-- input：自闭合，文字只能写在 value 里，纯字符串 -->
<input type="submit" value="保存">
对比项<button><input type="submit">
内容子元素,可含图标/标签/HTML只有 value 属性,纯文本
默认 typesubmit(易踩坑)由 type 决定,无歧义
样式灵活度高(可对内部元素单独设样式)低
现代推荐优先用它基本被取代
现代开发几乎一律用 <button>,因为要放图标 + 文字、loading 旋转图标等富内容。 <input type="submit"> 只在极简纯文本按钮里偶尔出现。

四、button 的 form 相关属性

一个表单可能有多个提交按钮,各自提交到不同地址、用不同方法。 <button> 上的这几个 form* 属性可以覆盖 <form> 自身的设置:

同一表单，两个按钮提交到不同地方
<form action="/save" method="post">
  <input name="title" required>

  <!-- 正常提交：走 form 的 /save + post -->
  <button type="submit">发布</button>

  <!-- 存草稿：覆盖成 /draft，并跳过校验 -->
  <button type="submit"
          formaction="/draft"
          formmethod="post"
          formnovalidate>存草稿</button>
</form>
属性作用
formaction覆盖 form 的 action,改变提交地址
formmethod覆盖 form 的 method(GET/POST)
formnovalidate这次提交跳过 HTML 表单校验(如 required 不再拦截)
formenctype / formtarget覆盖编码方式 / 提交目标窗口
formnovalidate 特别实用:"存草稿"时用户可能只填了一半,你不希望 required 拦着他, 加上它就能绕过校验直接提交。这些属性只在 type="submit" 的按钮上有意义。

五、可访问性：为什么别用 <div onclick> 当按钮

很多人为了好控制样式,用 <div onclick="..."> 或 <span> 模拟按钮。 视觉上一样,但对键盘用户和屏幕阅读器是灾难:

反例 vs 正例
<!-- ❌ div 假按钮：不能 Tab 聚焦、回车/空格无效、屏幕阅读器读不出"按钮" -->
<div class="btn" onclick="submit()">提交</div>

<!-- ✅ 真 button：天生可聚焦、回车/空格触发、被识别为 button 角色 -->
<button type="button" onclick="submit()">提交</button>

用真 <button>,浏览器免费送你这些:

能力<button><div onclick>
Tab 键可聚焦✅ 默认可❌ 需手动加 tabindex
回车 / 空格触发✅ 默认❌ 需手动监听 keydown
屏幕阅读器识别为"按钮"✅ 自带 button role❌ 读成普通文本
disabled 禁用✅ 原生支持❌ 不存在该属性
语义优先:能用 <button> 就别用 <div> 模拟。 要"长得不像按钮",改 CSS 就行(去掉边框背景),功能和无障碍一个都不会丢。 用 <div> 你得用 tabindex + role="button" + 键盘事件手动补全一堆东西,费力还容易漏。

六、disabled 按钮的样式与行为

给 <button> 加 disabled,浏览器会:不可点击、不触发任何事件、不能聚焦、不参与表单提交。

disabled 的行为
<button type="submit" disabled>提交中…</button>

/* 浏览器默认会把它变灰，也可自定义： */
button[disabled] {
  opacity: 0.5;
  cursor: not-allowed;
}
两个 disabled 陷阱:
1. disabled 的表单控件不会随表单提交——它的值在后端收不到。想让值提交但不可改,用 readonly(仅对 input/textarea)而非 disabled。
2. disabled 按钮不触发任何鼠标/键盘事件,所以无法给它绑 tooltip 之类的 hover 提示。需要提示时常改用"视觉禁用 + JS 拦截点击"而非真 disabled。

七、常见错误速查

错误后果正确做法
form 内交互按钮不写 type 点击意外提交表单,页面刷新/跳转,数据丢失 纯交互一律写 type="button"
用 <div onclick> 模拟按钮 键盘不可用、屏幕阅读器读不出,无障碍崩塌 用真 <button>,样式靠 CSS
把要提交的字段设 disabled 该字段的值不会发到后端 需提交但不可改用 readonly
想放图标却用 <input type="submit"> value 只能纯文本,塞不进 HTML 改用 <button>

八、与 Java 的类比

button 概念Java 类比
type="submit" 是默认值,易踩坑 方法的默认参数/隐式行为——没显式指定就走默认,默认值不直观时就成了 bug 来源(如忘记 StandardCharsets.UTF_8 走了平台默认编码)
显式写 type="button" 消除歧义 显式传参、显式声明,不依赖隐式默认——"显式优于隐式"
用 <button> 而非 Object 式的 <div> 用对接口/具体类型而非到处 Object——选语义正确的类型,编译器和运行时才能帮你
可访问性是浏览器与用户的契约 面向接口编程的契约——遵守 button 这个"接口约定",键盘/屏幕阅读器这些"调用方"才能正确协作
formaction 覆盖 form 设置 子类/局部覆盖父级配置(方法重写、局部 @Transactional 覆盖默认传播)
一句话总结:<button> 的坑都源于一个反直觉的默认值 type="submit"。 应对之道和后端一样——不依赖隐式默认,选语义正确的类型,遵守平台契约。 显式 type="button"、用真 <button> 元素,就能避开 90% 的问题。

与其他知识点的联系

  • 关联知识点：表单校验属性（required/pattern）： 浏览器内建校验只在 type="submit" 的点击触发提交时才会跑一遍——这也是为什么本文反复强调"纯交互按钮必须写 type="button"": 如果误把它当成 submit,不仅会意外提交表单,还会意外触发一次校验拦截,把用户搞得一头雾水； 而 formnovalidate 则是反过来,专门给"存草稿"这类需要跳过校验的 submit 按钮用的开关。
  • 关联知识点：为什么需要语义化： 本文第五节"别用 <div onclick> 当按钮"就是语义化最直接的例子——<button> 这个标签本身就是"可交互控件"的语义声明, 浏览器、屏幕阅读器、搜索引擎都能据此正确理解元素用途；换成 <div> 后语义信息全部丢失,只剩下视觉外观,这正是语义化标签存在的核心价值。
  • 关联知识点：tabindex 与键盘焦点： 表格里提到 <button> 默认可 Tab 聚焦、回车/空格可触发,而 <div> 必须手动加 tabindex="0" 才能进入 Tab 顺序, 还要自己监听 keydown 模拟"回车/空格=点击"的行为。这说明 tabindex 只是补救手段,不如直接用原生 <button> 一步到位。
  • 关联知识点：React 中的事件处理与 e.preventDefault()： 在 React 表单里,提交按钮触发的 onSubmit 事件默认还是会带着"提交并刷新页面"的浏览器原生行为, 所以几乎每个 React 表单的 onSubmit 处理函数里第一行都是 e.preventDefault()——这本质上是在用 JS 手动拦截 type="submit" 按钮的默认动作, 和本文"不想提交就显式写 type="button""是同一个问题的两种解法：一个是从源头不让它成为 submit,一个是允许它 submit 但半路截住。
  • 关联知识点：CSS 伪类 :disabled / :hover： 第六节展示了 button[disabled] 的样式写法,这和 CSS 伪类选择器是同一套机制——disabled 是 HTML 属性, 但可以直接作为属性选择器参与样式匹配；同理"视觉禁用"常用 :hover/:active 之类伪类去补充真 disabled 做不到的 hover 提示效果。
