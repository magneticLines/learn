资源预加载：preload / prefetch

浏览器加载资源有自己的"优先级算法"，但它不是你肚子里的蛔虫。当它判断错了、或来不及发现关键资源时，你可以主动给它一个"提前量"——这就是预加载提示要解决的问题。

一、先理解问题：浏览器为什么需要"提示"

浏览器拿到 HTML 后从上到下解析，发现一个资源引用（CSS、JS、图片、字体）才会去下载它。 同时浏览器内部有一套优先级算法：CSS 和首屏 JS 优先级高，图片低， 而且会限制同域名的并发连接数（HTTP/1.1 下通常 6 个），排队下载。

但这套算法有两个天然盲区，会让关键资源"被发现得太晚"：

关键资源"被发现得太晚"的两种典型场景
// 场景①：字体藏在 CSS 里，要等 CSS 下载并解析完才被发现
HTML → 下载 style.css → 解析 CSS → 发现 @font-face → 才开始下载字体
// 字体在这条"瀑布链"的末端，首屏文字会先用兜底字体再闪一下（FOIT/FOUT）

// 场景②：首屏大图由 JS 动态插入，HTML 里根本没有它
HTML → 下载 app.js → 执行 JS → 创建 <img> → 才开始下载图片
// 这张 LCP 大图被压在 JS 后面，首屏渲染被拖慢

本质问题是：浏览器只能"解析到哪发现到哪"，无法预知后面或动态产生的关键资源。 预加载提示就是开发者用 <link> 主动告诉浏览器： "这个资源很重要 / 等会儿要用，请你提前去拉。"

四个提示，按"做的事情多少"排个序：
dns-prefetch（只解析 DNS）→ preconnect（连接也建好）→ preload（当前页关键资源，现在就下）→ prefetch（下个页面的资源，闲了再下）。
前两个是"建立连接"，后两个是"下载内容"。

二、preload：高优先级，当前页马上要用

<link rel="preload"> 告诉浏览器："这个资源本页就要用，请用高优先级、 立刻开始下载，先别管你解析到没解析到。"它只下载不执行——下载好放进缓存， 等真正解析到引用时直接命中，省掉等待。

preload 必须配 as 属性
<!-- 预加载首屏 JS：as 告诉浏览器这是脚本，好定优先级和 Accept 头 -->
<link rel="preload" href="app.js" as="script">

<!-- 预加载首屏关键 CSS -->
<link rel="preload" href="critical.css" as="style">

<!-- 预加载 LCP 首屏大图 -->
<link rel="preload" href="hero.webp" as="image">

<!-- 预加载字体：注意必须加 crossorigin（原因见下文） -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

as 属性为什么是必须的

浏览器光看 href 不一定知道这是什么类型，as 解决三件事： ① 按正确的优先级排队下载；② 发送正确的 Accept 请求头； ③ 做正确的缓存匹配，让后面真正的引用能命中这份预载。 不写 as，preload 基本失效，浏览器会按最低优先级处理甚至忽略。

字体 preload 必须加 crossorigin——这是最高频的坑。字体请求遵循 CORS 机制， 即使是同源字体，@font-face 发出的请求始终是匿名跨域模式。 如果你 preload 时不加 crossorigin，预载的请求和真正的字体请求"模式不一致"， 缓存对不上，于是字体被下载两次——预载白费，还多耗一倍带宽。

三、prefetch：低优先级，预测下一步

<link rel="prefetch"> 是另一个维度的事：它针对的不是当前页，而是 用户"很可能"接下来要访问的资源。浏览器会用最低优先级、 趁网络空闲时悄悄把它下载到缓存，等用户真跳过去时秒开。

prefetch 预测下一跳
<!-- 用户在商品列表页，大概率会点进详情页 → 提前拉详情页的 JS chunk -->
<link rel="prefetch" href="product-detail.chunk.js" as="script">

<!-- 也可以 prefetch 整个下一页文档（next-page navigation）-->
<link rel="prefetch" href="/checkout">
一句话区分 preload 和 prefetch：
preload = "现在这个页面马上要用，赶紧下"（高优先级，确定性需求）
prefetch = "下个页面可能要用，闲了再下"（最低优先级，赌一个概率）
名字像，干的事却完全不同——一个抢首屏速度，一个赌下一跳。

四、preconnect / dns-prefetch：提前建连接

访问一个跨域资源（CDN、第三方 API、Google Fonts），浏览器在真正发请求前要先做几步握手： DNS 解析 → TCP 三次握手 → TLS 协商。这几步加起来可能上百毫秒，且是串行的。 这两个提示就是把这段"准备工作"提前做掉。

preconnect vs dns-prefetch
<!-- preconnect：DNS + TCP + TLS 全建好，资源一引用就直接发请求 -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- dns-prefetch：只解析 DNS，成本极低，作为老浏览器的兜底 -->
<link rel="dns-prefetch" href="https://fonts.gstatic.com">
为什么这俩常常一起写？preconnect 效果好但建连接有开销， 浏览器只会保留少数几个空闲连接，过期会被回收。dns-prefetch 成本极低、兼容性更好， 放在 preconnect 后面作为降级兜底（不支持 preconnect 的浏览器至少把 DNS 解了）。 建连接也要 crossorigin 是因为字体等资源走匿名 CORS 连接，不加会建一条用不上的连接。

五、四者对比表

提示做什么优先级时机典型用途
preload 下载资源内容（不执行） 高 立即，当前页 首屏字体 / 关键 CSS·JS / LCP 大图
prefetch 下载资源内容（不执行） 最低 网络空闲时 下个页面的 JS chunk / 文档
preconnect DNS + TCP + TLS 建连接 中 立即 即将用到的跨域源（CDN / 字体源）
dns-prefetch 只做 DNS 解析 低 立即 跨域源兜底 / 不太确定要不要用的域
记忆锚点：竖着看 = "做的事递增"——dns-prefetch（解析名字）⊂ preconnect（连上）⊂ preload（连上且下内容）。 而 prefetch 跳出当前页这个维度，是唯一面向"下一个页面"的提示。

六、陷阱与面试考点

陷阱后果正确做法
preload 了资源但本页根本没用到 浪费带宽抢占高优先级，控制台报警告
was preloaded but not used		 只 preload 确定本页会用的关键资源，用完即删
preload 字体不写 crossorigin 预载请求与真实字体请求模式不符，字体下两遍 字体 preload 永远带 crossorigin
什么都 preload，preload 一大堆 高优先级队列被挤爆，真正的首屏资源反被推后，首屏更慢 preload 是稀缺资源，一般只给少数几个最关键的
preload 不写 as 优先级判断失败、缓存匹配失败，预载几乎无效 preload 必须配 as
把"下一页资源"用 preload 抢了当前页首屏的高优先级带宽 下一页资源用 prefetch（最低优先级）
核心原则：预加载是"借未来换现在"，但带宽是零和的。 你给 A 提高优先级，本质就是把 B 的优先级压低。preload 用对了首屏快几百毫秒， 用错了（preload 一堆非关键资源）反而拖慢首屏——这正是面试爱问"preload 越多越好吗"的考点。 答案是否定的：preload 的价值在于精准纠正浏览器的优先级误判，不是无脑提速。

七、与 Java 的类比

这些"预加载"思想，后端工程师其实早就在用——都是用空闲资源提前做准备工作， 把首次请求的等待时间摊销掉。本质完全相通。

前端预加载概念Java / 后端类比
preconnect（提前建好连接待命） 数据库连接池预热（HikariCP 的 minimumIdle）——启动时先建好连接， 首个请求来时不用现场三次握手
preload（关键资源提前拉到缓存） 缓存预热（启动时主动跑 @Cacheable 把热点数据灌进 Redis）， 避免首个用户撞上缓存击穿
prefetch（空闲时预测性加载下一步） MyBatis / JPA 的预读 / 批量抓取，或消息队列的 prefetch 预取， 趁空闲先把"可能要用的"拉过来
dns-prefetch（提前解析域名） 服务启动时提前解析下游服务地址 / 预热 DNS 缓存， 避免首次远程调用卡在域名解析上
"用未来换现在"但带宽零和 JVM 预热（warmup）：启动后跑几轮假请求触发 JIT 编译， 用启动期的空闲 CPU 换取上线后第一批真实请求的低延迟
一句话收束：无论 preload 还是连接池预热，做的都是同一件事—— 把"按需现拉"的串行等待，提前到空闲时并行准备好。 区别只在于战场不同：一个在浏览器抢首屏渲染，一个在 JVM 抢首请求延迟。 理解了一边，另一边的设计动机你就秒懂了。

八、与其他知识点的联系

  • 与 script 加载策略（上一节）：preload 解决"资源被发现得太晚"，defer/async 解决"JS 下载执行会不会阻塞解析"——两者常常组合使用：先用 preload 把关键 JS 提前拉到缓存，再用 defer 控制它的执行时机，分别管住"下载"和"执行"这两个独立的阶段。
  • 与 Core Web Vitals（LCP/FCP）：本节反复提到的"LCP 大图"直接对应 Google 定义的核心性能指标之一——Largest Contentful Paint，preload 首屏大图是压缩 LCP 时间最直接的手段之一，这也是本节内容在真实性能优化考核（Lighthouse 评分）中最常被检查的一项。
  • 与字体加载与 FOIT/FOUT：本节场景①提到的"字体藏在 CSS 里被发现得晚"，正是字体章节会讲的 FOIT（不可见文本闪烁）/ FOUT（无样式文本闪烁）问题的根源之一——preload 字体是缓解这个视觉抖动的常见手段，但必须配合 font-display 属性一起用才完整。
  • 与 React/Vue 路由懒加载：本节 prefetch 的"预测下一跳"思路，正是前端路由懒加载（React.lazy + import()）背后常见的性能优化配套手段——框架的路由预取（Next.js 的 Link 组件默认 prefetch）本质就是在恰当时机插入本节讲的 <link rel="prefetch">。
  • 与 HTTP/2 与 CDN：preconnect 提前完成的 DNS/TCP/TLS 三步握手，在 HTTP/2 多路复用和 CDN 边缘节点场景下依然必要——即使协议层已经优化了单连接内的并发，跨域到新主机时这几步握手成本并未消失，理解这一点才能判断什么场景下 preconnect 仍然值得用。
