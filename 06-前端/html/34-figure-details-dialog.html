HTML · 语义化

figure / details / dialog

过去要写一堆 JS + CSS 才能搞定的"带说明的图""可折叠面板""模态框"，现在三个原生标签就能拿下。这是平台能力增强（Platform over Library）的最好缩影。

一、一句话先抓住主线

这三个标签放在一节里讲，不是因为它们功能相似，而是因为它们体现了同一个趋势： 曾经必须靠 JavaScript 库实现的 UI 组件，浏览器现在用原生标签直接提供了。

标签解决什么从前的做法
figure + figcaption带说明的独立内容（图/图表/代码/引用）<div class="figure"> + 手写样式
details + summary可折叠展开（手风琴 / FAQ）JS 监听点击 + 切 class + 改 max-height
dialog对话框 / 模态框JS 库 + 遮罩 + 焦点陷阱一大坨代码
类比后端：这就像 JDK 把过去要自己写或引第三方库的功能（如 java.util.concurrent、 HttpClient、var）逐步"内建"进语言和标准库。能力下沉到平台，应用层就不必造轮子。

二、figure / figcaption：带说明的独立内容

figure 表示一段自包含、可被独立引用的内容。关键词是"独立"—— 把它从正文里抽走、单独放到附录或侧栏，文章读起来依然通顺。figcaption 是它的标题或说明文字。

figure 不只用于图片
<!-- 图片 + 说明 -->
<figure>
  <img src="chart.png" alt="2025 营收趋势">
  <figcaption>图 1：2025 年季度营收，单位万元</figcaption>
</figure>

<!-- 也可以包代码、引用、图表 -->
<figure>
  <pre><code>System.out.println("hi");</code></pre>
  <figcaption>代码 1：Hello World</figcaption>
</figure>
真实渲染效果
陷阱：不要把所有图片都裹进 figure。 一个纯装饰的 banner、一个内联的小 icon、一张和正文紧密绑定无法抽离的配图——这些用普通 <img> 就好。 figure 的语义是"独立可引用单元"（论文里能写"见图 3"的那种）。滥用会让语义失真，对屏幕阅读器反而是噪音。
figure ≠ figcaption 必须有。figcaption 是可选的； 只想要"独立内容块"的语义而不需要标题时，可以只写 <figure>。一个 figure 最多一个 figcaption。

三、details / summary：零 JS 的折叠面板

details 是一个原生可折叠组件。里面第一个 summary 是始终可见、可点击的标题； 其余内容默认隐藏，点击 summary 即展开/收起。不需要任何 JavaScript，浏览器内置了全部交互。

FAQ 手风琴，纯 HTML
<details>
  <summary>什么是 details 标签？</summary>
  <p>浏览器原生的折叠组件，点击标题即可展开。</p>
</details>

<!-- open 属性：默认展开 -->
<details open>
  <summary>这一项默认是打开的</summary>
  <p>因为加了 open。</p>
</details>
真实交互演示（点点试试，纯 HTML 无 JS）
details 怎么做到不用 JS 就能折叠？
展开/收起的状态由浏览器维护：点击 summary 会切换 details 的 open 属性， 内容区随之显示或隐藏。键盘可聚焦、回车可触发、屏幕阅读器会读出"已折叠/已展开"—— 这些无障碍能力全是免费送的。
怎么让它默认展开？
在标签上加 open 属性。它是一个布尔属性，存在即为真。
这一项我加了 open，所以默认就是开的
注意看：页面一加载它就是展开状态，不需要先点一下。
分组互斥手风琴。给同一组 details 加相同的 name 属性， 它们就会变成"同一时间只能展开一个"的单选式手风琴（打开一个，其它自动收起）——这也是纯 HTML，无需 JS。
陷阱一：默认是折叠的。不写 open 时内容初始隐藏。 如果有内容你希望用户一进来就看到，别忘了加 open，否则它会被藏起来。
陷阱二：SEO 与搜索。折叠内容仍在 DOM 里、能被搜索引擎索引，但用户用 Ctrl+F 页内搜索时， 旧浏览器不会自动展开命中折叠区的内容（新版 Chrome 已支持自动展开）。关键内容不要无脑塞进折叠区。

四、dialog：原生对话框 / 模态框

dialog 是浏览器原生的对话框元素。它默认隐藏，通过 JS 的两个方法打开，区别很关键：

方法行为背景遮罩焦点陷阱
show()非模态：像浮层，页面其它部分仍可交互无无
showModal()模态：阻断背景，必须先处理它有 ::backdrop有（Tab 锁在框内）
close(val?)关闭对话框，可传返回值给 returnValue——
模态框 + returnValue
<dialog id="confirm">
  <p>确定要删除吗？</p>
  <!-- form method="dialog"：提交即关闭，并把按钮 value 写入 returnValue -->
  <form method="dialog">
    <button value="cancel">取消</button>
    <button value="ok">删除</button>
  </form>
</dialog>

<script>
  const dlg = document.getElementById('confirm')
  dlg.showModal()              // 模态打开，背景被遮罩 + 焦点陷在框内
  dlg.addEventListener('close', () => {
    console.log(dlg.returnValue)  // 'ok' 或 'cancel'
  })
</script>
真实交互演示（点按钮打开模态框）
returnValue：（还没操作）

原生模态框

注意：背景被 ::backdrop 遮住了，Tab 键只在框内循环（焦点陷阱），按 Esc 可直接关闭——这些都是浏览器免费给的。
showModal 帮你做了多少脏活？背景遮罩、点击遮罩/按 Esc 关闭、把焦点移进对话框、 Tab 时焦点锁在框内（焦点陷阱）、关闭后焦点还原到触发元素、对屏幕阅读器宣告为 dialog 角色…… 这些"无障碍模态框"的细节，过去用库（如 a11y-dialog）才能做对，现在一个 showModal() 全包了。
陷阱：show() vs showModal() 别用错。 show() 是非模态的——没有遮罩、没有焦点陷阱、Esc 也不关闭，背景照样能点。 真正要"逼用户先处理这个弹窗"的确认框、登录框，必须用 showModal()。 用错成 show() 会让模态框形同虚设。
陷阱：兼容性与 polyfill。dialog 在 2022 年才被所有主流浏览器支持（Safari 较晚）。 要兼容老浏览器需引入 dialog-polyfill。另外，直接用 CSS 写 dialog { display:none } 可能 覆盖原生的显示逻辑——应通过 dialog:not([open]) 或交给浏览器管理。

五、共同主题：平台能力增强

把这三个放一起看，会发现一条清晰的演进线：UI 组件正在从"库的责任"变成"平台的责任"。 折叠面板、模态框这类需求太普遍，每个项目各写一遍既浪费又容易出无障碍 bug，于是浏览器把它们标准化、内建化。

维度过去（库时代）现在（原生标签）
代码量JS + CSS 几十上百行几个标签 / 一行 API
无障碍得自己做对（极易漏）浏览器内建保证
体积引入第三方库，增加 bundle零依赖
一致性各家实现各不相同遵循同一标准

六、与 Java 的类比

前端概念Java / 后端类比
原生标签替代手写组件（details/dialog） 用标准库（java.util.concurrent、HttpClient）而不是自己造轮子或引第三方——能力沉到平台，应用层更瘦
details 声明式折叠、open 控制状态 配置优于编码（Convention over Configuration）——加个属性/注解就改变行为，不必写命令式逻辑
showModal() 自动管理焦点陷阱、遮罩、Esc 框架封装的横切关注点（AOP）——焦点/遮罩好比事务、日志、鉴权，由底层统一织入，业务代码不必关心
figure 表达"独立可引用单元"的语义 给数据加明确的领域类型/注解（如 @Entity），让结构自带含义，而非一律用 Map<String,Object>
能力下沉到浏览器平台的趋势 JDK 不断把常用能力内建（var、record、虚拟线程）——平台越强，上层越简单
本质上，这一节讲的是同一个工程哲学：把通用、易错、重复的能力下沉到平台层，由平台保证正确性。 前端把它做成原生标签，后端把它做成标准库和框架。学会优先用平台已提供的能力，就是"不造轮子"的成熟体现。

与其他知识点的联系

  • 关联知识点：article / section / div 的语义分层：上一节讲的是"内容结构"的语义化（哪里是文章、哪里是独立章节），这一节讲的是"交互组件"的语义化（哪里是可折叠面板、哪里是对话框）。两者是同一条语义化主线在不同层面的延伸：figure 恰好是 section 内部常见的子结构——一段独立可引用的图/代码块，配合上一节的 article 使用，能让一篇技术文章从大到小都具备正确的语义骨架。
  • 关联知识点：details 的 toggle 事件：本节演示里 details 靠浏览器内置逻辑切换 open 属性，但如果要在展开/收起时执行自定义逻辑（比如懒加载内容、上报埋点），就需要监听 details 上的 toggle 事件——它和 dialog 的 close 事件是同一种设计思路：原生标签把"状态切换"这件事做完后，通过事件把结果通知给 JS，而不是要求 JS 主动去驱动状态切换。
  • 关联知识点：dialog.showModal() / close() 与命令式 DOM API：演示代码里 showModal() 打开、监听 close 事件读取 returnValue，本质上是一套"命令式方法 + 事件回调"的 API 风格，和 details 声明式的 open 属性形成对比——前者用于"需要一次性触发、有明确生命周期"的场景（弹窗打开关闭），后者用于"始终存在、只是显示态切换"的场景（面板折叠）。理解这个差异有助于判断一个交互组件该设计成属性驱动还是方法驱动。
  • 关联知识点：可访问性——原生语义控件 vs div 模拟：本节反复强调的"零 JS 折叠""焦点陷阱免费送"，核心原因是浏览器把键盘可聚焦（Tab/Enter 触发 summary）、ARIA 角色宣告（dialog 被读屏软件识别为对话框）、焦点管理（showModal 后自动把焦点移入、关闭后还原到触发元素）这些无障碍细节都内建实现了。如果改用 div + onclick 手写同样的折叠面板或模态框，这些行为要一条条自己补齐，任何一条漏掉都是可访问性 bug——这正是"平台能力下沉"这条主线在无障碍层面的具体体现。
  • 关联知识点：React 中原生标签 vs 自封装组件的取舍：在 React 项目里遇到折叠面板或弹窗需求时，优先考虑直接用 <details>/<dialog> 包一层薄组件，而不是从零造一个纯 div 实现——原生标签自带的键盘操作、焦点管理、屏幕阅读器语义可以直接复用，React 组件只需负责把 open 属性、toggle/close 事件桥接成受控的 state 和 props。只有当设计要求的视觉/动效原生标签无法满足（比如自定义打开动画）时，才值得退回到 div 自己实现，并承担把无障碍细节补全的成本。
