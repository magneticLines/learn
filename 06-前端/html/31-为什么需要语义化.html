为什么需要语义化

同样的视觉效果，用 <h1> 还是 <div class="title">，肉眼看不出区别——但对浏览器、搜索引擎、屏幕阅读器和接手代码的同事来说，是两个完全不同的世界。

一、核心：HTML 标签是"内容的说明书"

很多人把 HTML 标签当成视觉容器来用——"我想要个大标题，那就找个能变大变粗的标签"。 这是个根本性的误解。HTML 的设计初衷不是描述长什么样（那是 CSS 的活）， 而是描述这块内容是什么。

换句话说，标签是写给"看不见样式的读者"看的一份语义说明书： 这是标题、这是导航、这是一篇文章、这是一个按钮。 样式可以被关掉、被覆盖、被换肤，但内容的"含义"应该是稳定的、机器可读的。

一句话抓住本质：语义化 = 用正确的标签描述内容的含义， 而不是用标签去凑外观。外观交给 CSS，含义交给 HTML，这是关注点分离（separation of concerns）在 Web 层的体现。

二、一个标签，四类读者

考虑这两段代码。它们配上同样的 CSS 后，在屏幕上长得一模一样：

视觉相同，含义天差地别
<!-- ① 只是一个被 CSS 撑大的 div -->
<div class="title">2026 年度财报</div>

<!-- ② 一个有语义的一级标题 -->
<h1>2026 年度财报</h1>

人眼看不出差别，但 HTML 文档其实同时有四类读者，它们各自只关心标签的含义，根本不在乎像素：

🌐
浏览器
靠标签构建无障碍树、提供默认行为。<h1> 会进入文档大纲、可被快捷键跳转；<div> 什么都不是。
🔍
搜索引擎爬虫（SEO）
靠标签判断哪段是页面主旨。爬虫给 <h1> 高权重，<div class="title"> 在它眼里只是普通文本块。
🦮
屏幕阅读器（无障碍）
视障用户靠"按标题跳转""列出所有导航"来导航。没有语义标签，整页就是一坨无法跳读的文字流。
👩‍💻
开发者（可维护）
接手的人读 <nav><article> 一眼懂结构；读满屏 <div class="xxx"> 得猜每个 div 到底是干嘛的。
关键认知：CSS 能让任何标签长成任何样子，所以"视觉一致"从来不能证明"语义正确"。 用 <div> 拼出来的页面在普通用户屏幕上可以完美，却对另外三类读者是彻底失明的。

三、反模式：div 汤（divitis）

当开发者把所有东西都用 <div> + class 堆出来，就形成了所谓的 div 汤（div soup），也叫 divitis（"div 炎"）。 下面是同一个页面的两种写法对比：

✗ div 汤：全是 div，零语义
<div class="header">
  <div class="nav">
    <div class="link">首页</div>
  </div>
</div>
<div class="main">
  <div class="post">
    <div class="title">文章标题</div>
    <div class="body">正文…</div>
  </div>
</div>
<div class="footer">…</div>
✓ 语义化：标签自带含义
<header>
  <nav>
    <a href="/">首页</a>
  </nav>
</header>
<main>
  <article>
    <h1>文章标题</h1>
    <p>正文…</p>
  </article>
</main>
<footer>…</footer>

两边渲染出来可以完全一样。但右边的每个标签都在自报家门："我是导航""我是主内容""我是一篇独立文章"。 左边则要求读者（无论是机器还是人）去读 class 名字猜含义——而 class 名是给 CSS 用的、任意的、不被任何机器理解的字符串。

陷阱：很多人以为"我 class 起名规范，叫 .nav .article 不就有语义了？" 错。class 对浏览器、爬虫、屏幕阅读器毫无意义，它们只认标签名。 <div class="nav"> 和 <nav> 在机器眼里差着十万八千里。

四、语义化对 SEO 的实质影响

搜索引擎爬虫拿到的是纯 HTML，没有你的设计稿。它要回答一个问题："这个页面到底在讲什么，该排在什么关键词下？" 它唯一的线索就是标签结构：

爬虫看到它如何理解
<h1>页面最核心的主题，权重最高（一页通常只该有一个）
<h2> <h3>子主题、层级结构，构成页面大纲（outline）
<article>一块可独立成篇的内容，适合被收录、被聚合
<nav>导航区，帮助爬虫发现站内其他页面
<a href>链接关系，是 PageRank 等算法的基础
满屏 <div>一团没有主次的纯文本，爬虫只能自己猜，权重分配模糊

所以语义化不是"锦上添花的规范"，而是直接影响搜索排名的可见性问题： 一个用 <h1> 标出主题、用标题层级组织内容的页面，爬虫能精准理解； 一个 div 汤页面，爬虫只能降级处理，关键词命中和排名都会吃亏。

五、语义化对无障碍的意义

视障用户用屏幕阅读器（screen reader）上网。它把页面"读"出来。 明眼人可以一眼扫到导航、跳过广告直奔正文——视障用户做不到，他们依赖语义标签提供的导航能力：

屏幕阅读器的常用快捷导航（以含义为单位跳转）
按 H        → 在所有"标题"之间跳转
按 1/2/3   → 跳到下一个 h1 / h2 / h3
按 D        → 在"地标区域"（nav/main/footer…）之间跳转
列出表单    → 枚举所有 <button> <input> <label>

这些功能全部建立在语义标签之上。如果整页都是 <div>，屏幕阅读器找不到任何标题、任何地标， 视障用户只能从第一个字符开始一行行听到尾——这相当于把网站的"目录"和"路标"全拆了。

额外好处：用 <button> 而不是 <div onclick>， 你白嫖了一大堆默认能力：键盘可聚焦、回车/空格可触发、屏幕阅读器会念"按钮"、自带 disabled 语义。 用 div 模拟按钮，这些全要手写 tabindex、role、键盘监听才能补齐——费力还容易漏。

六、语义化让代码自解释、易维护

这一条对你这个后端工程师可能最有共鸣。看到 <article><section><nav>， 不用读 CSS、不用问原作者，结构一目了然。标签本身就是文档。

反过来，div 汤把"这块是什么"的信息全藏进了 class 名和 CSS 文件里。 半年后改版时，你面对一百个 <div>，每个都得去 CSS 里反查它的角色—— 这和后端代码里满屏 Map<String, Object>、不知道里面装的是什么字段，是同一种痛苦。

自解释代码（self-documenting code）的思想是相通的：好的标签 / 好的命名能消除大量注释和文档。 <nav> 不需要 <!-- 这是导航 --> 来解释，就像 calculateMonthlyInterest() 不需要 // 计算月利息 来解释。让"结构"自己说话，比写注释更不容易过期。

七、与 Java 的类比

你在后端早就实践过同样的原则，只是没意识到它和语义化是一回事：

前端：语义化后端：Java 里对应的实践
用 <article> <nav> 等语义标签，让结构自带含义 用 @Entity @Service @Controller 标注角色，而不是一个裸 class 加一行注释说"这其实是个 Service"
标签自解释，不需要 <!-- 注释 --> 说明这块是啥 好的类名 / 方法名（OrderRepository、refund()）减少注释——命名即文档
divitis：到处用 <div>，含义全丢失 到处用 Object / Map<String,Object> 传数据，不肯建模成 DTO / 实体——类型和含义全靠猜
class 名只给 CSS 用，机器不理解 注释只给人看，编译器 / 框架不理解——靠注释维持的"约定"会悄悄失效
语义标签让爬虫 / 屏幕阅读器能自动处理 注解让 Spring / JPA 等框架能自动装配、自动建表——含义被机器读取后产生行为
一以贯之的原则：无论前端后端，都是"让含义对机器可读、对人自解释"。 语义标签之于 HTML，正如强类型与注解之于 Java——你不会愿意接手一个全是 Map<String,Object> 的后端， 同理也不该交付一个全是 <div> 的前端。

与其他知识点的联系

  • 关联知识点：页面结构标签（32号）：本篇讲的是"为什么"——语义化的动机和原理；32号讲的是"怎么用"——<header><nav><main><footer> 这些具体的页面级结构标签该怎么搭配使用。没有本篇打底，32号很容易被当成"又一批要背的标签"，而不是"解决 div 汤问题的工具"。
  • 关联知识点：article/section/div 的取舍（33号）：本篇第三节展示了 div 汤和语义化两种写法的对比，但没细讲 <article> 和 <section> 具体怎么区分、什么时候还是该用 <div>。33号会把"含义要匹配标签"这条原则落到"独立可分发内容用 article，主题分组用 section，纯样式容器才用 div"这种可操作的判断标准上。
  • 关联知识点：figure/details/dialog 等语义标签（34号）：本篇的四类读者模型（浏览器、爬虫、屏幕阅读器、开发者）同样适用于 <figure>（图文关联）、<details>（原生可折叠、无需 JS）、<dialog>（原生模态框、自带焦点管理）——这些标签之所以该被优先选用，正是因为它们比"div + JS 模拟"多提供了本篇第五节说的"白嫖的默认能力"。
  • 关联知识点：无障碍与 ARIA：本篇第五节提到屏幕阅读器靠语义标签做地标导航，这只是无障碍工作的起点。当原生标签确实无法表达某种语义时（比如自定义组件），才需要 role / aria-* 属性去补充说明——但补 ARIA 永远是语义化标签用不了时的"补救方案"，不是替代方案，这也是 WAI-ARIA 规范里"第一条规则"的由来。
  • 关联知识点：CSS 选择器与样式解耦：本篇第三节的陷阱提到 class="nav" 对机器毫无意义——但反过来看，语义化也让 CSS 选择器可以直接写 nav a、article h2 而不必给每个元素都发明一个 class 名。结构自解释之后，class 就能只承担"样式变体"这一件事，而不必同时承担"描述这是什么"的职责，这正是关注点分离在 CSS 命名上的延伸。
  • 关联知识点：React 组件设计：组件化开发容易让人误以为"反正是 JS 在管结构，标签随便用 div 包一层就行"——但 React 最终渲染出的仍然是真实 DOM，四类读者（尤其爬虫和屏幕阅读器）看到的还是最终产出的标签。<Nav /> 组件如果内部渲染成一堆 <div>，语义照样是丢失的；好的 React 组件应该像好的 Java 类一样，命名描述"角色"（组件名对开发者），渡染产物描述"含义"（DOM 标签对浏览器/爬虫/屏幕阅读器），两者缺一不可。
