script 加载策略：async / defer

同一段 JS，放对位置、加对属性，首屏可以快几百毫秒；放错了，整个页面会卡在那里等它下载。这是性能优化里最高频的一个考点。

一、先理解问题：JS 为什么会阻塞页面

浏览器解析 HTML 是从上到下逐行进行的，边解析边构建 DOM 树。 当它遇到一个 <script> 标签时，默认行为是：

默认 script 的阻塞行为
// 浏览器遇到普通 script 时：
1. 暂停 HTML 解析（DOM 构建停止）
2. 下载 JS 文件（网络往返，可能几百毫秒）
3. 执行 JS
4. 执行完毕，才继续解析后面的 HTML

这就是为什么把 <script> 放在 <head> 里会拖慢首屏—— 用户盯着白屏，等 JS 下载执行完，后面的内容才开始渲染。

为什么必须阻塞？因为 JS 可能用 document.write() 修改正在解析的文档， 或者操作还没构建出来的 DOM 节点。浏览器无法预知，只能保守地"停下来等 JS 跑完"。 这是历史设计决定的安全默认行为，async / defer 就是用来打破它的。

二、三种加载方式对比

三种写法
<!-- ① 普通：下载 + 执行都阻塞 HTML 解析 -->
<script src="app.js"></script>

<!-- ② async：下载不阻塞，下载完立即执行（执行时阻塞）-->
<script src="app.js" async></script>

<!-- ③ defer：下载不阻塞，等 HTML 解析完才执行 -->
<script src="app.js" defer></script>

三、核心差异表

执行时机下载完立即下载完立即HTML 解析完后
执行是否阻塞解析是是否
多个脚本执行顺序按书写顺序不保证（谁先下完谁先跑）按书写顺序
DOM 是否就绪不一定不一定一定就绪
对 inline 脚本有效—无效无效
两个关键区别记牢：
1. 执行顺序：defer 保证按书写顺序执行，async 不保证（哪个先下载完哪个先跑）。
2. DOM 就绪：defer 执行时整个 DOM 已构建完成，可以安全操作任意元素；async 不一定。

四、async 和 defer 只对外部脚本有效

注意有 src 才生效
<!-- ✅ 有效：有 src 的外部脚本 -->
<script src="app.js" defer></script>

<!-- ❌ 无效：内联脚本上的 async/defer 会被忽略 -->
<script defer>
  console.log('我会立即执行，defer 没用')
</script>

五、那"放在 body 底部"算什么

在 async / defer 普及前，最经典的做法是把 <script> 放在 </body> 前面：

传统做法：脚本置底
<body>
  <h1>页面内容</h1>
  <!-- ... 所有内容 ... -->

  <!-- 放最后：解析到这里 DOM 已基本就绪 -->
  <script src="app.js"></script>
</body>

置底和 defer 效果接近（DOM 都已就绪），但 defer 更优： defer 的下载在解析阶段就并行开始了，而置底脚本要等解析到那一行才开始下载。 现代推荐：放 <head> 里加 defer，下载更早，DOM 又保证就绪。

六、怎么选：决策指南

场景选择原因
主应用脚本、依赖 DOM、有依赖顺序 defer 保证顺序 + DOM 就绪，绝大多数场景的默认选择
独立的第三方脚本（统计、广告） async 互不依赖，越早执行越好，顺序无所谓
必须在 DOM 解析前运行（如反闪烁主题脚本） 普通（放 head） 少数需要阻塞的特例
ES Module 脚本 type="module" 默认就有 defer 行为，无需额外加
一句话经验：拿不准就用 defer。它兼顾"不阻塞首屏"和"DOM 就绪 + 顺序可控"， 是 90% 场景的正确答案。async 留给那些"独立、无依赖、谁都不等"的第三方脚本。

type="module" 的隐藏特性

ES Module 默认 defer
<!-- module 脚本天然延迟执行，等价于自带 defer -->
<script type="module" src="main.js"></script>

<!-- 想让 module 变成 async 行为，显式加 async -->
<script type="module" src="analytics.js" async></script>

七、常见错误速查

错误后果正确做法
把大 JS 放 head 里不加任何属性 首屏白屏时间增加，FCP/LCP 变差 加 defer，或放 body 底部
有依赖关系的脚本用 async 执行顺序乱，依赖未就绪报错 有顺序依赖必须用 defer
async 脚本里操作 DOM DOM 可能还没构建，querySelector 返回 null 用 defer，或监听 DOMContentLoaded
给内联脚本加 defer 期望延迟 无效，内联脚本立即执行 内联代码包进 DOMContentLoaded 回调

八、与 Java 的类比

script 概念Java / 并发类比
普通 script（同步阻塞） 主线程里直接调用阻塞方法（如同步 socket.read()）——整个流程停在这等
async（并行下载，乱序执行） 提交多个 CompletableFuture 不指定顺序——谁先完成谁先回调
defer（并行下载，顺序执行） 多个任务并发执行，但用 join() 在某个屏障点按提交顺序汇总结果
DOM 就绪后才执行（defer） Spring 的 @PostConstruct / ApplicationReadyEvent——等容器初始化完成再跑逻辑
下载与解析并行 I/O 与 CPU 计算重叠（异步 I/O），不让 CPU 空等网络
本质上，async 和 defer 解决的是同一类问题： 把"必须串行等待"变成"能并行的并行，该有序的有序"。 这和后端用线程池、异步 I/O 把阻塞调用并行化，是完全相同的性能优化思想—— 区别只在于一个发生在浏览器主线程，一个发生在 JVM 线程。
