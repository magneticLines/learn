iframe 内嵌：在页面里嵌一个独立文档

地图、视频、第三方支付页、客服组件——这些"页面里的页面"靠的都是 iframe。它强大，但也是 Web 安全里风险最集中的标签之一。用好它的关键，是搞懂沙箱和跨域通信。

一、iframe 是什么：一个嵌进来的独立浏览器上下文

<iframe>（inline frame）会在当前页面里开一个完整独立的文档窗口， 加载另一个 URL。它不是"把别人的 HTML 拷过来拼在一起"，而是浏览器为它单独创建一套环境： 独立的 DOM、独立的 window / document 对象、独立的 JS 执行上下文、独立的 cookie/storage 边界。

一个直觉类比：iframe 像在你的程序里起了一个独立进程，而不是引入一个类库。 它有自己的内存空间和生命周期，跟宿主页面之间隔着一道墙——这道墙既是它的隔离保护，也是它跨域通信麻烦的根源。
最小可用的 iframe
<iframe
  src="https://www.openstreetmap.org/export/embed.html"
  width="600" height="400"
  title="门店位置地图">
</iframe>

典型用途：嵌地图、嵌 YouTube/B站视频、嵌第三方支付收银台、嵌客服/评论组件、后台系统嵌入旧子系统。

二、核心属性

属性作用说明
src要加载的文档 URL核心属性；也可用 srcdoc 直接写一段 HTML
width / height尺寸建议配 CSS 控制，避免布局抖动
title无障碍标题必填——屏幕阅读器靠它说明"这个框是什么"
loadinglazy 懒加载滚动到视口附近才加载，省首屏流量
allow权限策略（Permissions Policy）白名单放行摄像头/麦克风/全屏/支付等能力
referrerpolicy控制发给子页面的 Referer如 no-referrer、origin，避免泄露完整 URL
sandbox安全沙箱本页重点，见第三节
属性配齐的写法
<iframe
  src="https://www.youtube.com/embed/xxxx"
  title="产品介绍视频"          <!-- 无障碍 -->
  width="560" height="315"
  loading="lazy"          <!-- 进视口再加载 -->
  allow="fullscreen; picture-in-picture"  <!-- 放行全屏 -->
  referrerpolicy="no-referrer">
</iframe>
缺 title 是最常见的无障碍缺陷。屏幕阅读器读到一个没有 title 的 iframe 时， 只能念出 "frame" 或一段 URL，视障用户完全不知道这块内容是什么。这是无障碍审计（如 axe、Lighthouse）必报的一项。

三、安全核心：sandbox 沙箱

嵌入第三方页面意味着你在自己的站点里运行了别人的代码。如果不加约束，被嵌的页面可以： 弹窗、自动跳转你的顶层页面、读你的表单、用脚本搞鬼。sandbox 就是给这个不可信文档套一层"牢笼"。

sandbox 的设计哲学是默认全部禁止，按需逐项放开——这是典型的"白名单"安全模型， 比"默认放开、出事再封"安全得多。只要写上 sandbox（哪怕值为空），子页面就被锁到最严： 不能跑脚本、不能提交表单、不能弹窗、被强制当成"无源"（origin 为 null，访问不了任何同源资源/cookie）。

从最严到按需放开
<!-- 空 sandbox = 最严：脚本/表单/弹窗/同源 全禁 -->
<iframe src="untrusted.html" sandbox title="不可信内容"></iframe>

<!-- 按需放权：允许跑脚本 + 提交表单，但仍禁弹窗、禁同源 -->
<iframe src="widget.html"
  sandbox="allow-scripts allow-forms"
  title="第三方组件"></iframe>

常用权限项

token放开了什么
allow-scripts允许运行 JavaScript
allow-same-origin保留子页面的原始 origin（可访问自己域的 cookie/storage）
allow-forms允许提交表单
allow-popups允许 window.open / target="_blank" 弹窗
allow-top-navigation允许子页面跳转顶层页面（默认禁，防钓鱼劫持）
头号陷阱：同时给 allow-scripts + allow-same-origin ≈ 没沙箱。
两者一起放开后，子页面既能跑脚本、又拥有真实 origin，那它就能用 JS 拿到 window.parent、移除自己身上的 sandbox 属性、再刷新——沙箱被它自己拆掉了。 所以这个组合只能用在你完全信任、本就同源的内容上；对真正的第三方，二者不可兼得。

四、同源策略与跨域通信：postMessage

浏览器的同源策略（Same-Origin Policy）规定：协议、域名、端口完全相同才算同源。 跨域时，父页面不能直接读写 iframe 内部的 DOM 和变量，反之亦然—— iframe.contentWindow.document 会直接抛跨域错误。这是浏览器最基础的安全隔离。

为什么必须隔离？设想你登录了网银，又在另一个标签打开了恶意站点。 如果没有同源策略，恶意站点的脚本就能读你网银页面的 DOM、偷你的余额和 cookie。 同源策略是浏览器"谁能碰谁的数据"的边界线。

跨域又确实需要通信怎么办？官方答案是 postMessage——一个受控的、基于消息传递的跨文档通信通道。 它不让你直接访问对方内存，而是让双方互发消息、各自校验后再处理，是典型的"消息传递"而非"共享内存"模型。

父页面 → iframe 发消息
const frame = document.querySelector('iframe')

// 第二个参数 targetOrigin：只发给这个源，绝不要用 '*'
frame.contentWindow.postMessage(
  { type: 'pay', amount: 99 },
  'https://pay.example.com'
)
iframe 内：接收并校验来源
window.addEventListener('message', (e) => {
  // 关键：必须校验 origin，否则任何站点都能给你发消息
  if (e.origin !== 'https://shop.example.com') return

  console.log('收到父页面消息：', e.data)
  // 回消息用 e.source（对方的 window 引用）
  e.source.postMessage({ type: 'done' }, e.origin)
})
postMessage 的两条铁律：
1. 发送时 targetOrigin 写明确的源，不要图省事写 '*'——否则消息可能被你不期望的页面截获。
2. 接收时一定校验 event.origin（和必要时 event.source）——这是身份鉴权，跳过它等于谁都能命令你。

五、安全风险：点击劫持 Clickjacking

既然别人能 iframe 你的页面，反过来想：恶意站点把你的页面用 iframe 嵌进去，调成透明、盖在一张诱饵图上面， 用户以为点的是"领取奖品"按钮，实际点中的是被透明覆盖的"确认转账"按钮。这就是点击劫持（Clickjacking）。

防御思路不是在 HTML 里写，而是由服务器在响应头里声明"我这个页面允不允许被别人 iframe"。 两种手段：

HTTP 响应头（服务端设置）
# 老办法：X-Frame-Options（兼容性好，能力弱）
X-Frame-Options: DENY            # 任何人都不许 iframe 我
X-Frame-Options: SAMEORIGIN      # 只允许同源页面 iframe 我

# 新办法：CSP frame-ancestors（更灵活，推荐）
Content-Security-Policy: frame-ancestors 'none'
Content-Security-Policy: frame-ancestors 'self' https://trusted.com
方向别搞反：sandbox 是你限制"你嵌进来的别人"； X-Frame-Options / frame-ancestors 是你限制"谁能嵌你"。 一个管入站，一个管出站，是两套独立的防护。现代项目优先用 CSP 的 frame-ancestors， 它能精确列白名单，且是 X-Frame-Options 的官方继任者。

六、性能开销与陷阱速查

每个 iframe 都是一个独立的文档上下文：单独的渲染树、单独的 JS 引擎堆栈、单独的网络请求。 它的成本远高于一个普通 <div>——多个 iframe 会显著拖慢页面、抢占内存。能不用就不用，要用就上 loading="lazy"。

陷阱后果正确做法
allow-scripts + allow-same-origin 给第三方 沙箱被子页面自行拆除，等于裸奔 对不可信内容二者只能取其一
缺 title 屏幕阅读器无法说明该框内容 每个 iframe 都写有意义的 title
缺 loading="lazy" 首屏外的 iframe 也立刻加载，拖慢首屏 非首屏 iframe 一律加 lazy
postMessage 不校验 origin 任意站点都能给你发指令，被注入 收消息先比对 event.origin
页面没设 frame-ancestors 可能被恶意站点 iframe 实施点击劫持 服务端设响应头限制可嵌入方

七、与 Java 的类比

iframe 概念Java / 后端类比
sandbox 按需放权 安全沙箱 / SecurityManager 的权限白名单——默认拒绝，逐项 grant
postMessage 跨域通信 跨服务的消息传递（MQ）——不共享内存，只收发消息，各自反序列化校验后处理
同源策略（Same-Origin Policy） 服务间的鉴权边界——跨域调用必须带凭证、过网关，不能直接读对方内存
独立文档上下文 独立进程：单独的地址空间与生命周期，比同进程调用开销大
校验 event.origin 消费 MQ 消息前校验来源/签名，等同于接口鉴权，绝不无条件信任 payload
frame-ancestors 限制被嵌 服务端的来源白名单 / CORS——由被调用方决定"谁可以调我"
本质上 iframe 的安全模型，和你后端做微服务隔离是同一套思路： 默认不信任、最小权限、消息而非共享、来源必校验。 把它当成"嵌在页面里的一个独立、不可信的远程服务"来对待，所有设计取舍就都顺理成章了。

与其他知识点的联系

  • 音频与视频（上一节）：嵌入 YouTube/B 站视频时用的正是 <iframe>，而不是原生 <video>——本文第二节的 loading="lazy"、allow="fullscreen" 直接呼应上一节视频标签的懒加载和全屏需求，两者常常一起出现在同一段嵌入代码里。
  • 图片 img 的性能与占位：iframe 的 width/height 建议配 CSS 控制以避免布局抖动，原理和图片一节讲的 CLS 防抖动完全一致，只是 iframe 的加载成本（独立文档上下文）远高于图片，抖动和首屏拖慢的影响也更严重。
  • 表格基础结构（下一节）：本文是"链接与媒体"模块的收尾，下一节转入"表格"这个全新话题——两者共享的是同一份关于语义化标签的心法：iframe 必须有 title 才能被屏幕阅读器理解，表格也必须有 caption/th 才能被正确朗读，都是"给机器一份结构化说明"的思路延续。
  • CSS 与响应式布局：iframe 内部是完全独立的渲染环境，外部 CSS 无法穿透进去设置内部样式（这也是它安全隔离的副作用），常见的"响应式 iframe"技巧（如用 padding-top 百分比模拟宽高比容器）就是绕开这个限制的经典 CSS hack。
  • JavaScript 跨域通信与网络安全：本文的 postMessage、同源策略、点击劫持防御，是前端安全里最容易在面试和实战中被问到的知识点，也是后续学习 CORS、CSRF、CSP 等 Web 安全主题的直接前置知识——理解了 iframe 的隔离边界，才能理解为什么跨域请求需要额外的鉴权机制。
