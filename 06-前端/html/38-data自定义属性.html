HTML · data-* 自定义属性

data-* 自定义属性与 dataset API

你想在一个按钮上"挂"一点数据给 JS 用，又不想往全局变量里塞、也不想发明一个浏览器看不懂的属性。data-* 就是 HTML 官方为这件事准备的标准口袋——元素自带、JS 随手可取。

一、问题：把数据"挂"在元素上

后端渲染或前端逻辑里，经常需要让一个 DOM 元素携带一点只有 JS 关心的数据： 这一行代表哪个用户？这个 tab 当前是开还是关？点击时该调哪个接口？

早年大家乱来：要么塞进 id（id="user-123" 再字符串切割）， 要么自创属性（<li userid="123">）， 要么用 className 编码状态。这些做法的问题是——不合法、不语义、解析麻烦。 HTML5 给了官方答案：data-* 自定义属性。

在元素上存自定义数据
<li
  data-user-id="123"
  data-role="admin"
  data-active="true">
  张三
</li>

只要属性名以 data- 开头，它就是合法的 HTML，浏览器不会报错、不会渲染、不影响样式默认行为， 并且自动暴露给 JS 的 dataset 接口。这就是"标准方式"的含义。

二、命名规则与 camelCase 转换

HTML 属性名大小写不敏感，所以约定用小写 + 连字符（kebab-case）。 而 JS 里属性名不能带连字符（obj.user-id 会被当成减法）， 于是 dataset 访问时浏览器自动把 kebab-case 转成 camelCase。

HTML 写法dataset 访问
data-iddataset.id
data-user-iddataset.userId
data-max-retry-countdataset.maxRetryCount
data-roledataset.role

转换规则很机械：去掉 data- 前缀，每个连字符后面的字母变大写，连字符删掉。

命名陷阱：属性名里别用大写或下划线。 data-userId 会被浏览器小写化成 data-userid，再转回来是 dataset.userid（不是你期望的 userId）； data-user_id 则原样保留下划线，变成 dataset.user_id，破坏了 camelCase 约定。 永远用全小写 + 连字符。

三、JS 读写：dataset API

element.dataset 是一个 DOMStringMap， 像普通对象一样读、写、删，改动会实时反映回 DOM 属性。

dataset 读 / 写 / 删
const li = document.querySelector('li')

// 读：取出来永远是 string
li.dataset.userId       // "123"  （注意是字符串！）
li.dataset.role         // "admin"

// 写：会同步生成/更新 data-* 属性
li.dataset.active = 'false'   // DOM 上变成 data-active="false"
li.dataset.status = 'open'    // 新增 data-status="open"

// 删：两种方式
delete li.dataset.status            // 移除 data-status
li.removeAttribute('data-active') // 等价

dataset vs getAttribute

dataset 是语法糖，底层还是属性。下面两种写法等价， 但 getAttribute 要写完整的带连字符的属性名：

两种等价写法
// 推荐：dataset（自动 camelCase）
li.dataset.userId

// 等价：getAttribute（写原始属性名）
li.getAttribute('data-user-id')

// 写 / 删 同理
li.setAttribute('data-user-id', '456')
li.removeAttribute('data-user-id')
选哪个？日常用 dataset，代码更短、更语义。 只有当属性名是运行时拼出来的变量（比如 'data-' + key）时， getAttribute 才更方便——因为 dataset 没法用变量动态取连字符形式的 key。

四、值永远是字符串

这是初学者最容易踩的坑：HTML 属性本质就是文本， 所以从 dataset 取出来的一律是 string，没有数字、布尔、对象之分。

类型陷阱
// <div data-count="5" data-active="false">

el.dataset.count          // "5"      不是 5
el.dataset.count + 1      // "51"     字符串拼接！
Number(el.dataset.count) + 1 // 6        要显式转换

el.dataset.active         // "false"  注意：非空字符串
if (el.dataset.active) {} // 永远为 true！"false" 是真值

// 布尔要这样判断
if (el.dataset.active === 'true') {}
面试高频：if (el.dataset.active) 当属性值是 "false" 时仍然成立—— 因为非空字符串都是 truthy。布尔语义的 data-* 必须用 === 'true' 显式比较， 或者干脆用"属性存不存在"来表达（'active' in el.dataset）。

五、CSS 也能用 data-*

data-* 不只服务 JS，CSS 也能读它——这让"用属性驱动样式"成为可能， 比写一堆状态 class 更干净。两个工具：属性选择器和 attr() 函数。

属性选择器做状态样式

用 data-state 切换样式
/* JS 只改 data-state，CSS 自动响应 */
.panel[data-state="closed"] {
  height: 0;
  overflow: hidden;
}
.panel[data-state="open"] {
  height: auto;
}
配套的 JS：只切一个属性
panel.dataset.state = 'open'   // 样式立即跟着变
panel.dataset.state = 'closed'

attr() 取值显示在内容里

用 attr() 把 data 值渲染成伪元素内容
/* <span data-tooltip="保存"> */
.btn[data-tooltip]:hover::after {
  content: attr(data-tooltip);  /* 显示"保存" */
}
设计思想：让 data-* 成为单一数据源——JS 只负责改属性， CSS 负责"看到属性长什么样就渲染成什么样"。 这正是声明式 UI 的雏形：状态驱动视图，而不是手动操作每个样式。

六、典型用途

1. 给 JS 传递配置

组件初始化参数写在元素上
<div class="carousel" data-interval="3000" data-loop="true"></div>

// JS 读配置，不必硬编码
const interval = Number(el.dataset.interval)

2. 事件委托时识别元素

把监听挂在父容器上，靠子元素的 data-* 判断点了谁—— 这是处理动态列表的标准模式，避免给每一项都绑监听。

事件委托 + data-id
list.addEventListener('click', (e) => {
  const item = e.target.closest('[data-id]')
  if (!item) return
  deleteUser(item.dataset.id)   // 拿到被点项的 id
})

3. 给元素打状态标记

data-loading、data-selected、data-state 等， 配合 CSS 属性选择器，把"状态"从一堆 class 里解放出来，语义更清晰。

七、陷阱速查

陷阱后果正确做法
存敏感数据（token、密钥、价格） 用户 F12 直接看到、改了重放 敏感数据只放后端校验，前端永不存
把大量数据塞 data-*（整个 JSON 列表） HTML 臃肿、解析慢、反复 JSON.parse 用 JS 对象 / Map 在内存里管理
命名用大写或下划线 camelCase 映射错乱，取不到值 全小写 + 连字符
忘记取出来是 string "5"+1="51"、"false" 判真 Number() 转、=== 'true' 比
滥用 data-* 代替语义属性 失去无障碍 / 表单原生能力 有 href/value/aria-* 就用原生的
安全这条最致命：data-* 是明文写在 HTML 里的， 用户可见、可改、可伪造。它只是"方便 JS 取的临时数据口袋"， 绝不是存储或鉴权机制。任何"前端说了算"的安全逻辑都能被绕过。
什么时候别用 data-*：如果一个值已经有原生属性表达（链接用 href、 输入值用 value、无障碍状态用 aria-expanded），就用原生的—— 它们有内置语义和浏览器行为。data-* 是"没有合适原生属性时"的补充，不是万能替代品。

八、与 Java 的类比

data-* 概念Java 类比
给元素挂自定义数据（data-*） 给对象附加元数据 / 用 Map<String,Object> 扩展字段——不动主结构，临时挂载
dataset 取值都是 String HTTP 请求参数 / Properties 配置取出来都是 String，要 Integer.parseInt 再用
kebab-case ↔ camelCase 自动转换 Jackson 的 PropertyNamingStrategy 把 user_name 映射到 userName 字段
CSS 属性选择器响应 data 状态 观察者模式 / 状态驱动渲染——状态变了，视图自动跟着变，不手动刷
不在 data-* 存敏感数据 不把数据库密码 / 密钥写进前端或日志——可信边界永远在服务端
一句话本质：data-* 是 HTML 给"元素 + 一小撮 JS 私有数据"提供的标准、合法、零污染的绑定方式。 它替代了往 id / class / 全局变量里硬塞数据的脏做法—— 就像后端用扩展字段而非乱改表结构，用配置而非硬编码。 记住三件事：取出来是 String、命名全小写连字符、永远别存敏感数据。

与其他知识点的联系

  • 关联知识点：JS 变量与数据类型（01-变量与数据类型.html）： dataset 取出来的值永远是 string，这不是 dataset 自己的怪癖， 而是 JS 弱类型 + HTML 属性本质是文本这两件事叠加的结果。 和 Number()、String() 这些强制类型转换函数是同一套底层逻辑—— 本文第四节的 "5"+1="51" 陷阱，本质就是变量类型没搞清楚就直接用。
  • 关联知识点：React 的 props 与状态驱动视图： React 里组件间传数据用 props，不用 data-*，因为 props 是类型化的 JS 值， 直接活在组件树里，不需要"写进 DOM 再读出来"这一圈转换。 但 data-* 没有消失——写自定义元素（Web Components）、 对接不受 React 管理的第三方库（如 jQuery 插件）， 或给 E2E 测试（Cypress/Playwright）留 data-testid 选择器时， 仍然要用 data-*，因为这些场景需要脱离 JS 内存、落地成 DOM 上的可查询标记。
  • 关联知识点：CSS 属性选择器 [data-xxx]（本文第五节）： 这不是额外补充，而是 data-* 设计上就双向可用的证明—— 同一个属性，JS 用 dataset 读写，CSS 用属性选择器和 attr() 读。 这正是本文强调的"单一数据源"思想：状态只存一份（DOM 属性）， JS 和 CSS 各自订阅它，而不是 JS 改一个变量、CSS 再手动同步一个 class。
  • 关联知识点：为什么需要语义化（31-为什么需要语义化.html）： data-* 和语义化标签解决的是两个不同层面的问题，容易被误当成互相替代。 语义化标签（<nav>、<article>）告诉浏览器/屏幕阅读器 "这块内容是什么角色"；data-* 只是"挂一点 JS 私有数据"，浏览器不会赋予它任何语义。 本文第六节提到的"滥用 data-* 代替语义属性"陷阱，本质就是把数据载体误用成了语义载体—— 该用 aria-expanded 表达状态语义时，绝不能用 data-expanded 替代， 因为屏幕阅读器只认前者。
  • 关联知识点：Jackson 命名策略（后端 JSON 序列化）： 本文第八节已经提到 kebab-case ↔ camelCase 的转换类比 Jackson 的 PropertyNamingStrategy， 这里再往深一层看：两者都在解决"两套命名习惯的边界系统如何互通"的问题—— HTML 属性名大小写不敏感只能用 kebab-case，JS 变量名不能带连字符只能用 camelCase， dataset 就是这两套规则之间的一层自动翻译，和后端 JSON 字段名与 Java 字段名之间的映射是同一种"边界适配"设计。
