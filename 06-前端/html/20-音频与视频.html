HTML · 链接与媒体

音频 audio 与视频 video

原生媒体标签让浏览器内置了播放器，不用 Flash、不用第三方库。但"为什么我的视频不自动播放"是新手第一天就会撞上的坑——根子在浏览器的自动播放策略上。

一、基本用法

<audio> 和 <video> 用法几乎一样，区别只是一个出声、一个出画面。 最简单的写法是直接给 src，但更推荐用 <source> 子标签（后面讲多格式兼容时会解释为什么）。

最小可用示例
<!-- 音频：加 controls 才会显示播放条 -->
<audio src="bgm.mp3" controls></audio>

<!-- 视频：通常还要给宽高和封面 -->
<video src="demo.mp4" controls width="640"></video>
不加 controls 会怎样？媒体元素默认不显示任何控件， 页面上看起来像一片空白（音频甚至完全不可见）。除非你打算用 JS 自己造播放按钮，否则记得加 controls。

二、核心属性一览

属性作用说明
controls显示浏览器自带的播放/进度/音量控件布尔属性，写了就生效
autoplay加载后尝试自动播放带声音时几乎都会被浏览器拦截（见第四节）
loop播放结束后自动从头循环背景视频、循环 BGM 常用
muted初始静音是 autoplay 能生效的关键
preload控制预加载策略取值 none / metadata / auto
poster视频未播放时显示的封面图仅 <video> 有
playsinline移动端内联播放（不强制全屏）iOS 背景视频必备

preload：告诉浏览器"要不要提前下载"

preload 是一个对带宽很敏感的属性。它本质上是你给浏览器的一个建议 （浏览器可以不完全听，但通常会尊重）：

取值含义适用场景
none先不下载，等用户点播放再说页面里有很多视频、或大概率不会被播放
metadata只下载元数据（时长、尺寸、首帧）想显示时长但不想浪费流量，常用默认折中
auto浏览器可以放心预下载整个文件用户极可能播放，如落地页主视频
类比懒加载：preload="none" 就像 JPA 的懒加载（LAZY）—— 用到才去查；preload="auto" 像急加载（EAGER）——先全拿回来。 metadata 则像只查个 count 和主键，不拉整行 blob。选错会导致首屏白白消耗几 MB 流量。
poster + preload 组合
<video
  src="intro.mp4"
  controls
  poster="intro-cover.jpg"   <!-- 没播放时显示这张封面 -->
  preload="metadata">     <!-- 只拉元数据，省流量 -->
</video>

三、多格式兼容：多个 source

不同浏览器支持的视频编码不完全一样（历史上有专利、授权的纠葛）。mp4（H.264）兼容性最好， webm 在开源浏览器上体积更小。解决办法：在元素里并列写多个 <source>， 浏览器从上到下挑第一个它能播的，挑到就停。

多 source + 降级文本
<video controls width="640" poster="cover.jpg">
  <!-- 浏览器按顺序尝试，选中第一个支持的就不看后面了 -->
  <source src="demo.webm" type="video/webm">
  <source src="demo.mp4"  type="video/mp4">

  <!-- 都不支持时（很老的浏览器）显示这段降级内容 -->
  你的浏览器不支持视频播放，<a href="demo.mp4">点此下载</a>。
</video>
type 属性别省：写上 type="video/webm" 后，浏览器不用先下载就能判断自己支不支持， 直接跳过不能播的格式，省一次无用请求。把兼容性最好的 mp4 放最后兜底是常见策略。
这套机制本质就是后端常说的内容协商（content negotiation）：客户端声明自己的能力， 服务端/资源提供多个候选，双方匹配出一个都能接受的格式——和 HTTP 的 Accept 头按优先级协商响应类型一个道理。

四、自动播放策略（高频坑，务必理解）

几乎每个新手都会问："我加了 autoplay，为什么视频还是不动？" 答案是：现代浏览器禁止"带声音"的媒体自动播放。这不是 bug，是有意为之的规则。

为什么浏览器要禁止

想想没有这条规则的年代：你打开一个网页，突然一段广告视频带着巨大音量轰你一脸——这在地铁、办公室里非常尴尬。同时自动播放还会偷偷消耗移动流量。所以浏览器厂商统一约定：

核心规则：只有满足下列任一条件，autoplay 才会被允许：
1. 媒体是静音的（muted），或本身没有音轨；
2. 用户已经与页面发生过交互（点过、按过键），表明这是用户想看的内容。
否则浏览器直接忽略 autoplay，视频停在第一帧。
反例 → 正例
<!-- ❌ 反例：只写 autoplay，带声音 → 被浏览器拦截，不播 -->
<video src="hero.mp4" autoplay loop></video>

<!-- ✅ 正例：加 muted，autoplay 才生效 -->
<video src="hero.mp4" autoplay muted loop playsinline></video>

这就是为什么所有官网首屏的背景视频都是静音的——它们想自动循环播放， 唯一的合法途径就是 muted。如果你确实需要声音，正确做法是：默认静音自动播放， 再放一个"点击取消静音"的按钮，让用户的点击来解锁声音。

记忆口诀：autoplay 想生效，muted 不能少。 想要有声音，等用户来点。

五、字幕 track（无障碍 + 国际化）

<track> 给视频挂字幕文件（.vtt 格式）。它不只是"方便"， 更是无障碍（accessibility）的硬要求——听障用户、静音环境（如刷手机不开声音）都依赖字幕。 很多内容合规标准也强制要求视频带字幕。

多语言字幕
<video src="lesson.mp4" controls>
  <!-- kind=subtitles：翻译给听得见的人看 -->
  <track kind="subtitles" src="zh.vtt" srclang="zh" label="中文" default>
  <track kind="subtitles" src="en.vtt" srclang="en" label="English">

  <!-- kind=captions：含音效描述，给听障用户 -->
  <track kind="captions" src="zh-cc.vtt" srclang="zh" label="中文(含音效)">
</video>
kind 取值区别
subtitles对白翻译，假设用户听得到声音，只是看不懂语言
captions对白 + 音效/说话人标注，给听不到声音的用户
descriptions画面内容的语音描述，给视障用户
类比 i18n：多个 <track srclang> 就像 Spring 里按 locale 加载不同的 messages_zh.properties / messages_en.properties—— 同一份内容、多语言资源、按用户偏好选择。srclang 就是那个 locale 键。

六、用 JS 控制播放

媒体元素在 JS 里是一个 HTMLMediaElement 对象，提供了一套方法和属性。这里只做简单提及：

常用 JS 控制 API
const v = document.querySelector('video')

v.play()          // 播放（返回 Promise，被拦截会 reject）
v.pause()         // 暂停
v.currentTime = 30 // 跳到第 30 秒
v.volume = 0.5     // 音量 0~1
v.muted = false   // 取消静音（常在用户点击回调里做）

// play() 是 Promise：自动播放被拦时能捕获到
v.play().catch(() => {
  // 浏览器拒绝了自动播放，这里降级处理（如显示播放按钮）
})
记住 play() 返回 Promise：当你想 JS 自动播放但浏览器拦截时， 会抛 NotAllowedError。一定要 .catch()，否则控制台会报未捕获的异常，且用户看不到任何提示。

七、常见陷阱速查

陷阱后果正确做法
autoplay 不加 muted 被浏览器拦截，视频卡在首帧 自动播放必须配 muted
用 video 当背景，忘了 playsinline iOS 上自动跳成全屏播放 背景视频用 muted autoplay loop playsinline
大视频不设 preload 默认可能预下载几 MB，拖慢首屏、烧流量 显式 preload="metadata" 或 none
视频不挂 <track> 字幕 听障/静音用户无法理解内容，不符合无障碍规范 至少提供一条 captions
只给一种格式且不放降级文本 个别浏览器无法播放且无任何提示 多 <source> + 末尾降级内容
JS play() 不 catch 自动播放被拒时静默失败、控制台报错 play().catch() 做降级

八、与 Java 的类比

媒体概念Java / 后端类比
多个 <source> 按序降级 内容协商：HTTP Accept 头按优先级匹配，挑出双方都支持的格式
preload="none" JPA 懒加载 LAZY——用到才查，不提前拉数据
preload="auto" JPA 急加载 EAGER / 缓存预热——提前把数据拿回来
preload="metadata" 只查投影字段（SELECT id, duration），不拉整行大字段
多语言 <track srclang> i18n 国际化：按 locale 加载 messages_zh/en.properties
自动播放策略（需用户交互解锁） 受保护操作需权限/授权才能执行——没有合法凭证就拒绝
play() 返回 Promise 异步方法返回 CompletableFuture，失败要处理异常分支
一条主线：HTML 的媒体元素把"能力声明 + 资源候选 + 按需加载 + 多语言"这些后端早就熟悉的工程思想， 搬到了浏览器里。理解了 内容协商、懒加载、i18n、权限控制， 再看 source / preload / track / 自动播放策略，就全是老朋友换了层皮。
