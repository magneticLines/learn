tabindex 与键盘焦点

不是所有人都用鼠标。视障用户靠屏幕阅读器 + 键盘，行动不便者可能只用键盘甚至开关设备，重度用户也偏爱 Tab 飞快穿梭。一个页面能不能"只用键盘走完全程"，是可访问性的硬指标——而 tabindex 是其中最容易用错的一环。

一、为什么键盘导航是底线，不是加分项

鼠标点击靠的是"看见 + 指向"，但很多用户做不到这两点：

  • 视障用户：看不见鼠标在哪，只能让屏幕阅读器逐个朗读，用 Tab / Shift+Tab 在交互元素间跳。
  • 行动不便用户：无法精准操控鼠标，依赖键盘或单一开关设备顺序遍历。
  • 高效用户：填表、操作后台时不愿离开键盘，Tab 比挪鼠标快得多。

所谓"焦点（focus）"，就是当前接收键盘输入的那个元素——同一时刻只有一个。按 Tab 就是把焦点从一个可聚焦元素移到下一个，移动的先后叫做Tab 顺序（tab order）。

类比：焦点就像演播室里那只"亮着红灯的话筒"——同一时间只有一个在收音，Tab 就是把红灯切给下一支话筒。你的页面设计，决定了红灯按什么顺序流转。

二、哪些元素"天生"可聚焦

浏览器默认只让交互元素进入 Tab 顺序——因为只有它们需要接收键盘操作：

元素条件说明
<a>必须有 href没有 href 的 <a> 只是普通文本，不可聚焦
<button>未 disableddisabled 元素会被移出 Tab 顺序
<input> / <textarea>未 disabledtype="hidden" 例外，不可聚焦
<select>未 disabled下拉选择
带 contenteditable 的元素—可编辑区域天然可聚焦
关键陷阱：<div>、<span>、<p> 这些非交互元素默认不可聚焦。如果你用 <div onclick> 自制了一个"按钮"，键盘用户根本 Tab 不到它，也无法回车触发——它对一半用户是不存在的。这正是 tabindex 要登场的地方（但更好的答案往往是：直接用 <button>）。

三、tabindex 的三种值

tabindex 用来显式控制元素的可聚焦性和 Tab 顺序。它只有三类取值，含义截然不同：

三种取值速览
<!-- ① tabindex="0"：加入自然 Tab 顺序（让本不可聚焦的元素可聚焦）-->
<div tabindex="0" role="button">自定义按钮</div>

<!-- ② tabindex="-1"：JS 可 focus()，但不在 Tab 顺序里 -->
<div id="dialog" tabindex="-1">模态框</div>

<!-- ③ tabindex="正数"：强制指定顺序 —— 反模式，慎用！-->
<input tabindex="3">

① tabindex="0" —— 把元素"接入"自然顺序

它的作用是：让一个本来不可聚焦的元素（如 div）变得可以 Tab 到，并且插入到它在 DOM 中应在的位置——也就是顺着源码顺序排，不打乱别的元素。这是给自定义控件补可访问性的标准做法。

记忆点：0 不代表"排第 0 个"，而代表"按文档流的自然顺序排队"。它是最常用、最安全的值。

② tabindex="-1" —— 可被 JS 聚焦，但 Tab 跳过它

负数（约定俗成写 -1）表示：用户按 Tab 到不了这个元素，但代码里可以用 element.focus() 主动把焦点送进去。典型用途：

  • 模态框容器：打开时用 JS 把焦点移进去，但它本身不该出现在正常 Tab 流里。
  • skip link 的目标：<main id="content" tabindex="-1">，让"跳到主内容"链接能把焦点真正落到主区域。
  • 表单校验失败时，用 JS 聚焦到第一个错误项（哪怕它不是天生可聚焦的容器）。

③ tabindex="正数" —— 强制顺序，反模式

正数会强行抢到 Tab 顺序的最前面：浏览器先按 1, 2, 3... 升序遍历所有正数 tabindex 元素，之后才轮到所有 tabindex="0" 和天生可聚焦的元素（按 DOM 顺序）。

正数 tabindex 如何打乱顺序
<!-- 视觉上 A 在最前，但 Tab 顺序会是：B → C → A -->
<input placeholder="A 姓名">              <!-- 无 tabindex，最后才到 -->
<input placeholder="B 邮箱" tabindex="1">  <!-- 先 -->
<input placeholder="C 电话" tabindex="2">  <!-- 后 -->
为什么是反模式？
  • 视觉顺序与 Tab 顺序脱节：用户看到 A 在最上，按 Tab 却跳到 B，体验错乱。
  • 维护噩梦：一旦用了正数，整页凡是需要进入这段顺序的元素都得标正数，否则统统被挤到最后。新增一个字段就要重排所有数字。
  • 等于硬编码全局优先级：和在代码里写满 magic number 一样，牵一发动全身。
正确做法：调整 DOM 顺序本身，让源码顺序 = 视觉顺序 = Tab 顺序。需要可聚焦就用 0，永远不要用正数。

四、焦点必须可见：:focus 与 :focus-visible

键盘用户唯一的"光标"就是焦点环（focus ring）——那圈高亮轮廓。看不见它，键盘用户就彻底迷路：不知道现在选中了哪个、回车会触发什么。

最常见的可访问性事故
/* ❌ 灾难：把焦点环干掉却不给替代，键盘用户瞎了 */
button:focus { outline: none; }

很多人嫌默认焦点环丑就 outline: none。问题在于：鼠标点击时确实不需要焦点环（点哪知道哪），但键盘 Tab 时极度需要。一刀切去掉，就牺牲了所有键盘用户。

:focus-visible 正是为此而生：浏览器智能判断——鼠标点击时不显示焦点环，键盘聚焦时才显示。这样既能美化又不伤可访问性：

正确做法：自定义且只在键盘时显示
/* 鼠标点击：不显示难看的环 */
button:focus:not(:focus-visible) { outline: none; }

/* 键盘聚焦：给一个清晰、自定义的焦点环 */
button:focus-visible {
  outline: 2px solid #fb923c;
  outline-offset: 2px;
}
底线规则：可以替换焦点样式（换颜色、加边框、加阴影），但绝不能无替代地移除它。WCAG 明确要求焦点必须可见。记住口诀：去掉 outline 就必须补一个等效可见的提示。

五、焦点管理：模态框与焦点陷阱

打开一个模态框（弹窗）时，如果不管焦点，焦点还停在背后页面的某个按钮上——键盘用户按 Tab 会跑到被遮住的背景元素里，完全失控。正确的焦点管理需要三步：

  • 打开时：把焦点移入弹窗（聚焦到弹窗容器或第一个可操作元素）。
  • 打开期间：把焦点困在弹窗内——Tab 到最后一个元素后回到第一个，形成循环，这叫焦点陷阱（focus trap）。
  • 关闭时：把焦点还原到打开弹窗前的那个触发元素，用户回到原来的位置。
焦点管理的骨架（打开/还原配对）
let lastFocused;

function openDialog() {
  lastFocused = document.activeElement;   // 1. 记下当前焦点
  dialog.hidden = false;
  dialog.focus();                       // 2. 焦点移入（dialog 需 tabindex="-1"）
}

function closeDialog() {
  dialog.hidden = true;
  lastFocused?.focus();                  // 3. 还原到打开前的元素
}
和 try-finally 是同一种纪律：openDialog 里"记录 + 移入"和 closeDialog 里"还原"必须成对出现——就像 Java 里 lock() 必须配 finally { unlock() }、资源 acquire 必须配 release。漏掉还原那一步，焦点就"泄漏"了，用户体验上等价于资源泄漏。

六、skip link：让键盘用户跳过重复导航

每个页面顶部往往有一大堆导航链接。键盘用户每次进入页面都要 Tab 几十下才能到正文，非常痛苦。skip link（跳过链接）是放在页面最前的一个隐藏链接，平时不可见，一旦被 Tab 聚焦就显形，回车即可直达主内容。

skip link 标准实现
<!-- 放在 body 第一个，指向主内容 -->
<a href="#main" class="skip-link">跳到主内容</a>
<!-- ... 一长串导航 ... -->
<main id="main" tabindex="-1"> ... </main>
平时藏起来，聚焦时出现
.skip-link {
  position: absolute;
  left: -9999px;          /* 移出可视区，但仍可被 Tab 到 */
}
.skip-link:focus {
  left: 8px; top: 8px;    /* 一聚焦就跳回可见处 */
}
为什么 <main> 要加 tabindex="-1"？因为 <main> 默认不可聚焦，单靠 href="#main" 只会滚动页面，焦点仍卡在原处，键盘用户的 Tab 起点没变。加上 -1，点击 skip link 后焦点才真正落进主内容，后续 Tab 从这里开始。注意：别用 display:none 隐藏 skip link，那样它会被移出 Tab 顺序，等于没做。

七、陷阱速查表

陷阱后果正确做法
使用 tabindex="1" 等正数 Tab 顺序与视觉脱节，难维护，挤压其他元素 调整 DOM 顺序，需要可聚焦只用 0
outline: none 且无替代 键盘用户看不到焦点在哪，完全迷路 用 :focus-visible 提供自定义焦点环
<div onclick> 当按钮，不加 tabindex / 键盘事件 键盘 Tab 不到、回车/空格触发不了 优先用 <button>；非用 div 则加 tabindex="0" + role + keydown
模态框打开后不移焦点 焦点留在背景，Tab 跑到被遮挡元素 打开移入、做 focus trap、关闭还原
删除/隐藏当前聚焦的元素 焦点"丢失"，回退到 <body>，Tab 重新从头开始 删除前把焦点主动转移到合理的相邻元素
用 display:none 藏 skip link 被移出 Tab 顺序，键盘永远聚焦不到 用 position:absolute; left:-9999px 视觉隐藏
自测方法：合上鼠标，纯用 Tab / Shift+Tab / Enter / Esc 把你的页面核心流程走一遍。如果你能顺利完成、且始终看得见焦点在哪，键盘可访问性就基本过关了。这一步成本极低，却能抓出绝大多数问题。

八、与 Java 的类比

前端概念Java 类比
自然 Tab 顺序（DOM 顺序遍历） 责任链 / 管道按注册顺序依次传递——处理者的先后由声明顺序决定，清晰可预测
tabindex="正数" 强制优先级（反模式） 在代码里硬编码全局优先级数字（magic number）——加一个新节点就要重排所有人，牵一发动全身
焦点管理：移入 + 还原成对出现 try-finally / lock()`+`finally{unlock()}——资源 acquire 必须配 release，漏掉就泄漏
"可聚焦"是元素对键盘用户的契约 实现某个接口就承诺履行其方法约定——声明可交互（role/button）就必须真正可用键盘操作
tabindex="-1"：不进默认流程，但可被显式调用 非 public 方法 / 不自动装配的 Bean——不参与默认遍历，但代码可主动调用它
一句话收束：键盘可访问性的核心，是让"看不见的顺序"（Tab 顺序、焦点流转）和"看得见的结构"（视觉布局、DOM）保持一致。这和后端追求"代码执行顺序 = 业务意图、控制流清晰可推理"是同一种工程审美——不要让隐式行为偏离显式声明。

与其他知识点的联系

  • 关联知识点：可访问性整体——tabindex 只是键盘可访问性拼图的一块，它解决"能不能 Tab 到"，但 role/aria-* 解决"屏幕阅读器怎么念"、焦点环解决"看不看得见"。三者缺一都不算合格：一个 div 加了 tabindex="0" 却没配 role 和 keydown 事件，Tab 能到但回车没反应，屏幕阅读器也不知道念它是"按钮"——这也是本文反复强调"优先用 <button>"的原因。
  • 关联知识点：button 类型（原生可聚焦元素）——<button>、有 href 的 <a>、<input> 这些交互元素天生就在 Tab 顺序里，浏览器已经免费帮你实现了聚焦、键盘触发（Enter/Space）、语义朗读，根本不需要手动加 tabindex。tabindex="0" 存在的意义恰恰是"补课"——给 div/span 这类天生没有键盘能力的元素强行接入 Tab 顺序，但补的只是"能不能聚焦"，键盘事件和语义还得自己写，远不如直接用原生元素省心可靠。
  • 关联知识点：dialog 的焦点陷阱（figure-details-dialog）——本文第五节讲的"移入 → 困住 → 还原"三步，正是 <dialog> 一类弹窗必须实现的焦点管理。弹窗容器要有 tabindex="-1" 才能被 focus() 主动聚焦（它本身不是交互元素，天生不可聚焦）；打开期间要做 focus trap，让 Tab 在弹窗内部循环，不能漏到背景里被遮挡的元素上；关闭时要把焦点还原到触发按钮，否则用户会"凭空"回到页面顶部，完全迷失方向感。
  • 关联知识点：React 中的焦点管理（useRef + focus）——原生 DOM 里 element.focus() 要先拿到元素引用，React 里同理，只是引用的获取方式变成了 useRef()：用 const dialogRef = useRef(null) 拿到 DOM 节点，在 useEffect 里对打开状态做响应式调用 dialogRef.current?.focus()。本文骨架代码里的 lastFocused = document.activeElement 到 React 里同样成立，只是"记录上一个焦点"和"还原焦点"要放进 effect 的清理函数或状态变化分支里，思路完全一致——React 只是换了触发时机的写法，没有改变焦点管理这套底层规则。
  • 关联知识点：CSS :focus-visible——本文第四节讲的 :focus-visible 解决的是"什么时候该显示焦点环"：鼠标点击不用显示（点哪知道哪），键盘 Tab 必须显示（否则用户瞎走）。它和 tabindex 是配套关系——tabindex="0" 决定一个元素"能不能"被 Tab 到，:focus-visible 决定它被 Tab 到之后"看不看得见"。如果只加了 tabindex="0" 却没配可见的焦点样式（或者被全局 outline: none 误杀），键盘用户依然会迷路——两者必须同时具备才算真正可用。
