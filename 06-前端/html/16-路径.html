路径：相对路径 vs 绝对路径

"图片不显示"、"链接 404"，十有八九是路径写错了。搞懂路径只需要回答一个问题：这个路径是相对于"谁"来计算的？

一、核心问题：路径相对于"谁"

浏览器看到 <img src="logo.png"> 时，必须把 logo.png 拼成一个完整的 URL 才能去下载。 问题是：拼到哪个目录前面？答案是当前 HTML 文件所在的目录——这就是一切的基准点。

记住这一句：相对路径永远相对于"当前文件所在的那个目录"， 不是相对于网站首页、不是相对于项目根、更不是相对于你写代码时的工作目录。 后面所有规则都是这一句的推论。

路径分两大类：

  • 绝对路径：给出完整定位，不依赖"当前在哪"。如 https://cdn.com/a.png、/img/a.png。
  • 相对路径：从"当前文件所在目录"出发，一步步走过去。如 a.png、../img/a.png。

二、一棵目录树（后面都用它举例）

项目结构（部署后网站根 = my-site/）
my-site/            ← 部署后这里是网站根目录 /
├── index.html
├── assets/
│   ├── logo.png
│   └── style.css
├── blog/
│   ├── post.html   ← 假设我们正在 这个 文件里写路径
│   ├── draft.html
│   └── img/
│       └── cover.png
└── about/
    └── team.html

"当前文件" = blog/post.html，所以它的"当前目录" = blog/。 下面所有相对路径都从 blog/ 这个点出发。

三、相对路径：从当前目录出发走

先理解两个最重要的符号：

  • . 表示当前目录（就是你站着的地方）。
  • .. 表示上一级目录（往回退一格）。
在 blog/post.html 里，各种相对路径指向哪
<!-- 同级：blog/ 下的 draft.html -->
<a href="draft.html">草稿</a>
<!-- 等价写法，./ 显式表示"当前目录" -->
<a href="./draft.html">草稿</a>

<!-- 子级：进入 blog/img/ 再取 cover.png -->
<img src="img/cover.png">

<!-- 上一级：.. 退到 my-site/，再进 assets/ -->
<img src="../assets/logo.png">

<!-- 上一级再换目录：退到 my-site/，进 about/ -->
<a href="../about/team.html">团队</a>

把 .. 想成"按返回键"：每个 .. 往上跳一层。 ../assets/logo.png 读作"退一层（到 my-site）→ 进 assets → 取 logo.png"。

./ 几乎可以省略：./draft.html 和 draft.html 完全等价。 显式写 ./ 的真正价值在 JS 模块导入（如 import x from './x.js'）—— 那里 ./ 不能省，省了会被当成 node_modules 包名去找。HTML 里可省。

四、绝对路径的三种形态

三种"绝对"，依赖程度逐级降低
<!-- ① 完整 URL：带协议，最绝对，跨站点也能用 -->
<img src="https://cdn.example.com/logo.png">

<!-- ② 协议相对：省略 http/https，跟随当前页面的协议 -->
<img src="//cdn.example.com/logo.png">

<!-- ③ 根相对：/ 开头，相对"网站根域名"，不带域名 -->
<img src="/assets/logo.png">
形态例子相对于谁典型用途
完整 URLhttps://cdn.com/a.png谁都不依赖，全宇宙唯一引用外部 CDN、第三方资源
协议相对//cdn.com/a.png当前页面的协议（http/https）历史方案，现已少用（建议直接写 https）
根相对/assets/a.png网站根域名 https://你的站/站内资源，最常用的"站内绝对路径"
别被名字骗了：根相对路径 /assets/logo.png 名字带"相对"， 但它不从当前文件目录出发——它从网站根出发。 无论你在 blog/post.html 还是 about/team.html 里写它， 都解析成 https://你的站/assets/logo.png，结果完全一样。这正是它的优势。

五、根相对 / 开头 vs 相对 ../ 的取舍

要从 blog/post.html 引用 assets/logo.png，两种写法都对：

同一个目标，两种思路
<!-- 相对：从当前目录一步步爬 -->
<img src="../assets/logo.png">

<!-- 根相对：直接从网站根定位 -->
<img src="/assets/logo.png">
对比维度相对路径 ../根相对 /
基准点当前文件所在目录网站根目录
移动当前文件后容易失效（层级变了，.. 数量就错了）不受影响（基准是固定的根）
整个站换部署子目录不受影响（内部相对关系没变）容易失效（根变了，详见陷阱）
本地 file:// 直接打开正常工作常出错（/ 指向磁盘根）
可移植性（拷到别处）整个文件夹一起拷就 OK依赖部署在域名根
经验法则： 站点稳定部署在域名根（如 example.com/）→ 用 / 开头，深层页面引用全站公共资源时尤其省心。 需要本地双击打开、或会部署到子目录（如 example.com/myapp/）、或文件夹要整体搬运 → 用相对 ../ 更稳。

六、<base href> 会改写所有相对路径的基准

默认情况下相对路径的基准是"当前文档 URL"。但 <head> 里放一个 <base href="...">，就能强行改写这个基准——页面里所有相对路径 都改从 base 指定的地址算起。

base 改变后续所有相对 URL 的解析起点
<head>
  <base href="https://cdn.example.com/v2/">
</head>
<body>
  <!-- 实际请求 https://cdn.example.com/v2/logo.png -->
  <img src="logo.png">
</body>
base 是把双刃剑：它影响整个页面所有相对链接（包括锚点 #section 也会被拼到 base 上！）， 一处设置全局生效，很容易引发"莫名其妙全跳到别处"的诡异 bug。 SPA 框架（Vue/React Router）有时靠它定 base path，但手写页面里慎用—— 问题排查时它常是第一个被忽略的元凶。

七、陷阱速查

陷阱 1：移动文件后 ../ 失效

你把 blog/post.html 移到 blog/2024/post.html，层级深了一层。 原来的 ../assets/logo.png（退 1 层）现在退到的是 blog/ 而不是 my-site/， 需要改成 ../../assets/logo.png（退 2 层）。相对路径与文件位置强绑定，挪动即失效。

陷阱 2：本地 file:// 下 / 指向磁盘根

用浏览器双击打开 HTML 文件时（地址栏是 file:///D:/my-site/blog/post.html）， 根相对路径 /assets/logo.png 会被解析成 file:///D:/assets/logo.png—— 指向磁盘 D 盘的根，而不是你的项目根！所以本地直接打开时 / 开头的路径几乎必坏。 解决：用本地静态服务器（如 npx serve、VS Code Live Server）跑，让它有真正的"网站根"。

陷阱 3：Linux 服务器大小写敏感

Windows / macOS 文件系统默认大小写不敏感，Logo.png 和 logo.png 在本地都能打开， 所以本地一切正常。但部署到 Linux 服务器后大小写敏感， src="Logo.PNG" 找不到 logo.png，线上直接 404。 统一全小写是最省事的约定。这是"本地好好的，一上线就挂"的经典原因。

陷阱 4：空格和特殊字符要编码

URL 里空格必须编码
<!-- ❌ 空格会截断或解析异常 -->
<img src="my photo.png">

<!-- ✅ 空格编码成 %20 -->
<img src="my%20photo.png">

更省心的做法：文件名别用空格和中文，用连字符 my-photo.png。 URL 里 空格 → %20、# → %23 等都需要百分号编码，否则浏览器会误读。

八、与 Java 的类比

路径这套"相对 vs 绝对"、"从哪个基准出发"的思路，后端工程师其实早就天天在用—— classpath 和文件系统路径就是同一回事。

前端路径概念Java / JVM 类比
相对路径 img/cover.png（相对当前文件目录） classpath 相对资源 getClass().getResource("cover.png")——相对当前 .class 所在包目录
绝对路径 https://... / 文件全路径 文件系统绝对路径 new File("/var/data/a.png") 或 D:\\...——不依赖工作目录
../（退到上级目录再引用） 包之间的相对引用，从 com.app.web 回退到 com.app 下找别的类
根相对 /assets/a.png（从网站根） classpath:/config.yml 的 / 根——从 classpath 根开始找，不管当前在哪个包
<base href> 改写所有相对基准 设置 working directory（user.dir）后，所有相对 File 路径基准随之改变
本地 file:// 下 / 指向磁盘根 new File("/a") 在不同 OS / 不同工作目录下解析不同——同样是"基准没定清楚"踩的坑
本质上，前端的路径和 Java 的 classpath 解决的是同一个问题： 给一个资源命名，并约定好"从哪个起点去找它"。 相对路径换来可移植性（整个文件夹/jar 包搬走仍然对），绝对路径换来确定性（基准固定不会错）。 选哪种，永远取决于你更怕"被移动"还是更怕"基准不一致"——这个权衡前后端完全一致。

与其他知识点的联系

  • 超链接 a / 图片 img（后续章节）：路径本身不是一个独立的知识点，而是 <a href>、<img src>、<link href>、<script src> 等所有"引用外部资源"的标签共用的底层规则——学完这一节，后面遇到任何带 src/href 属性的标签，都可以直接套用"相对于当前文件目录"这套心智模型。
  • 响应式图片 srcset（下一模块）：后面讲 <img srcset> 时会一次性提供多张图片的路径供浏览器按屏幕密度选择，每一个候选路径依然遵循本文的相对/绝对路径规则，只是从"一个 src"变成"一组候选路径"，路径解析的基准点完全不变。
  • iframe 的 src 与跨域：<iframe src="..."> 如果填的是完整 URL（本文第四节的"完整 URL"形态），加载的就是另一个完全独立的源，这也是浏览器同源策略、跨域限制会介入的地方——路径的写法直接决定了这是"站内资源"还是"跨站请求"。
  • CSS 中的 url() 与背景图路径：CSS 文件里 background: url(../img/bg.png) 的路径基准是"该 CSS 文件所在目录"，而不是引用它的 HTML 文件所在目录——这是新手极易踩的坑，本质上和本文"路径相对于谁"的核心问题完全一致，只是基准点换成了 CSS 文件。
  • React/前端工程化中的模块路径：本文提到 ./ 在 JS 模块导入里不能省略（否则被当成包名去 node_modules 找），这在 React 项目写 import Logo from './assets/logo.png' 时会天天遇到；而打包工具（webpack/vite）处理这类路径时又引入了"编译时路径解析"这一新的基准维度，是本文规则在前端工程化场景下的延伸。
  • SEO 与规范化 URL：站内资源到底用根相对路径还是完整 URL，也会影响搜索引擎抓取和 <link rel="canonical"> 规范化链接的写法——路径的确定性和一致性是搜索引擎正确收录页面资源的前提之一。
